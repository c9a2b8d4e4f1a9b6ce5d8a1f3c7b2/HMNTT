# ERC20 approve race condition

## Summary
The report states that there is a bug in the ERC20 contract that could potentially be exploited by attackers. This bug is caused by a race condition in the approve function, which allows for a frontrun. This means that if Alice approves Bob for 10 tokens in one transaction, but then changes the approval to 5 tokens in another transaction, Bob can use a higher gas fee to execute a third transaction and spend the 10 tokens before the second transaction is processed. To mitigate this issue, it is recommended to use the increaseAllowance() and decreaseAllowance() functions in addition to the approve function, as shown in the Openzeppelin version of ERC20 tokens. More information on how to implement this can be found in the provided links. Overall, this bug is considered to have a medium severity and has been acknowledged by the team.

## Details
**Severity**: Medium

**Status**: Acknowledged

**Description**

The ERC20 contract has a race condition in its approve function, which can be exploited by attackers via a frontrun. For example, let’s say Alice approves Bob for 10 tokens in transaction 1, but then changes the approve to 5 tokens in transaction 2. Bob can exploit this by spending the 10 tokens in a transaction 3 and executing it before transaction 2 by paying a higher gas fee.
