# Initialization functions can be front-run

## Summary
A bug was identified in the Hermez.sol smart contract, which is used for data validation. This bug allows an attacker to front-run the initialization functions of Hermez, HermezAuctionProtocol, and WithdrawalDelayer, allowing them to incorrectly initialize the contracts. This is due to the use of the delegatecall proxy pattern, which prevents the contracts from being initialized with a constructor.

Exploit Scenario: Bob deploys the Hermez contracts. Eve front-runs the HermezAuctionProtocol's initialization and sets her own address for the donation address. As a result, she receives the tokens sent as a donation.

Recommendations for short-term solutions include using a factory pattern to prevent front-running of the initialization, or ensuring the deployment scripts are robust in case of a front-running attack. It is also recommended to carefully review the Solidity documentation and the pitfalls of using delegatecall proxy pattern.

## Description  
Hermez, HermezAuctionProtocol, and WithdrawalDelayer have initialization functions that can be front-run, allowing an attacker to incorrectly initialize the contracts. Due to the use of the delegatecall proxy pattern, Hermez, HermezAuctionProtocol, and WithdrawalDelayer cannot be initialized with a constructor, and have initializer functions:

```solidity
function initializeHermez (
    address[] memory _verifiers,
    uint256[] memory _verifiersParams,
    address _withdrawVerifier,
    address _hermezAuctionContract,
    address _tokenHEZ,
    uint8 _forgeL1L2BatchTimeout,
    uint256 _feeAddToken,
    address _poseidon2Elements,
    address _poseidon3Elements,
    address _poseidon4Elements,
    address _hermezGovernanceDAOAddress,
    address _safetyAddress,
    uint64 _withdrawalDelay,
    address _withdrawDelayerContract
) external initializer {
    // set state variables
    _initializeVerifiers(_verifiers, _verifiersParams);
    withdrawVerifier = VerifierWithdrawInterface(_withdrawVerifier);
    hermezAuctionContract = AuctionInterface(_hermezAuctionContract);
    tokenHEZ = _tokenHEZ;
    forgeL1L2BatchTimeout = _forgeL1L2BatchTimeout;
    feeAddToken = _feeAddToken;
    
    // set default state variables
    lastIdx = _RESERVED_IDX;
    // lastL1L2Batch = 0 --> first batch forced to be L1Batch
    // nextL1ToForgeQueue = 0 --> First queue will be forged
    nextL1FillingQueue = 1;
    // stateRootMap[0] = 0 --> genesis batch will have root = 0
    tokenList.push(address(0)); // Token 0 is ETH

    // initialize libs
    _initializeHelpers(
        _poseidon2Elements,
        _poseidon3Elements,
        _poseidon4Elements
    );
    _initializeWithdraw(
        _hermezGovernanceDAOAddress,
        _safetyAddress,
        _withdrawalDelay,
        _withdrawDelayerContract
    );
}
```

## Exploit Scenario  
Bob deploys the Hermez contracts. Eve front-runs the HermezAuctionProtocol’s initialization and sets her own address for the donation address. As a result, she receives the tokens sent as a donation.

## Recommendations  
Short term, either:
- Use a factory pattern that will prevent front-running of the initialization, or
- Ensure the deployment scripts are robust in case of a front-running attack.

Carefully review the [Solidity documentation](https://docs.soliditylang.org/en/v0.8.0/) especially the Warnings section. Carefully review the pitfalls of using the delegatecall proxy pattern.
