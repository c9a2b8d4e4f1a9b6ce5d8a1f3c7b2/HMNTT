# ESMS use of sanitized user_amount & user_id values ✓ Fixed

## Summary
This bug report is about an issue with the Signer service in the gtc-request-signer code. A problem arises when the user inputs are not properly preserved and passed down in the function. This issue is caused by the fact that the values are sanitized, but the original user inputs are being used in the code. As a result, if a float amount is passed for `user_amount`, all checks will pass, however the final amount will be slightly different that what it is intended. This issue was fixed by using the sanitized integer value in the code flow. To avoid this issue in the future, it is recommended to use the sanitized value for the rest of the code flow after the sanity check.

## Details
#### Resolution



Fixed in <https://github.com/nopslip/gtc-request-signer/pull/4/> , by using the sanitized integer value in the code flow.


#### Description


In the Signer service, values are properly checked, however the checked values are not preserved and the user input is passed down in the function.


The values are sanitized here:


**code/gtc-request-signer-main-5eb22e882e28e6f3192b80f237f7a3bcd15b1ee9/app.py:L98-L108**



```
try:
    int(user\_id)
except ValueError:
    gtc\_sig\_app.logger.error('Invalid user\_id received!')
    return Response('{"message":"ESMS error"}', status=400, mimetype='application/json')
## make sure it's an int
try:
    int(user\_amount)
except ValueError:
    gtc\_sig\_app.logger.error('Invalid user\_amount received!')
    return Response('{"message":"ESMS error"}', status=400, mimetype='application/json')

```
But the original user inputs are being used here:


**code/gtc-request-signer-main-5eb22e882e28e6f3192b80f237f7a3bcd15b1ee9/app.py:L110-L113**



```
try:
    leaf = proofs[str(user\_id)]['leaf']
    proof = proofs[str(user\_id)]['proof']
    leaf\_bytes = Web3.toBytes(hexstr=leaf)

```
**code/gtc-request-signer-main-5eb22e882e28e6f3192b80f237f7a3bcd15b1ee9/app.py:L128-L131**



```
## this is a bit of hack to avoid bug in old web3 on frontend
## this means that user\_amount is not converted back to wei before tx is broadcast! 
user\_amount\_in\_eth = Web3.fromWei(user\_amount, 'ether')


```
#### Examples


if a float amount is passed for `user_amount`, all checks will pass, however the final amount will be slightly different that what it is intended:



```
>>> print(str(Web3.fromWei(123456789012345, 'ether')))
0.000123456789012345
>>> print(str(Web3.fromWei(123456789012345.123, 'ether')))
0.000123456789012345125

```
#### Recommendation


After the sanity check, use the sanitized value for the rest of the code flow.
