# Missing TransactionPayload Type Validation

## Summary
The bug report discusses an issue with the TransactionPayload struct in the Aptos blockchain system. This struct is responsible for containing the payload and its type. Currently, it only supports four types of payloads: WriteSet, Scripts, ModuleBundle, and EntryFunction. However, there is a problem with the assumption that the payload can only be of type EntryFunction. This assumption is not validated and can cause issues with internal functions that rely on it. To fix this, the function payload EntryFunction deserialization should be removed so that the transaction can be of any type. Additionally, the type should be validated against EntryFunction in the momentum_safe::register function. This bug has been fixed in version 6f30e62.

## Details
## TransactionPayload Struct

The `TransactionPayload` struct contains the payload and its type. Aptos supports payloads of type:

- **WriteSet**
- **Scripts**
- **ModuleBundle**
- **EntryFunction**

## RUST

```rust
/// Different kinds of transactions.
pub enum TransactionPayload {
    /// A system maintenance transaction.
    WriteSet(WriteSetPayload),
    /// A transaction that executes code.
    Script(Script),
    /// A transaction that publishes code.
    Module(Module),
    /// A transaction that executes an existing entry function published on-chain.
    EntryFunction(EntryFunction),
}
```

The payload in the `TransactionPayload` struct can be any transaction type, not just `EntryFunction`. This assumption should be validated. For example, many internal functions such as register payload validation assume the layout of the `TransactionPayload` is an `EntryFunction`.

## Remediation

Remove the function payload `EntryFunction` deserialization so that the transaction can be of any type. 

In addition, in `momentum_safe::register`, validate the `type_uleb128` against `EntryFunction`.

## Patch

Fixed in `6f30e62`.
