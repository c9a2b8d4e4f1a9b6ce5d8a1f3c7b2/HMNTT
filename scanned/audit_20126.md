# [M-03] _wethWithdrawTo is vulnerable re-entrancy

## Summary
This bug report is about a vulnerability that could be exploited in the function withdrawBorrowETH in the code. The function invokes _wethWithdrawTo and later _checkMinReserve, however, the check of reserve is not necessary here as function _wethWithdrawTo already does that after transferring the ether. This could lead to re-entrancy attacks and the MIN_RESERVE could be bypassed. 

The recommended mitigation steps are to use a re-entrancy guard on all main action functions (e.g. deposit, withdraw, borrow, repay, etc). Re-entrancy guards are designed to protect against re-entrancy attacks, which are a type of attack where an attacker attempts to repeatedly call a function in a contract and take advantage of the fact that the same function can be called multiple times before the contract state is updated. The guard works by ensuring that only one execution of a function is allowed at a time.

## Impact
function withdrawBorrowETH invokes _wethWithdrawTo and later _checkMinReserve, however, the check of reserve is not necessary here, as function _wethWithdrawTo also does that after transferring the ether. However, this reserve check might be bypassed as TransferHelper._wethWithdrawTo uses a low level call that is vulnerable to re-entrancy attacks. As this MIN_RESERVE sounds like an important value, you should consider preventing re-entrancy attacks here.
