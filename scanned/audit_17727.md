# [M-27] RubiconMarket.feeTo set to zero-address can DoS buy function

## Summary
A bug was found in the `RubiconMarket` contract, where the `feeTo` storage variable can be set to any arbitrary address, including a zero-address. This can cause the `buy` function calls with a `_offer.buy_gem` ERC20 token to revert when attempting to transfer fees to the zero-address. To mitigate this issue, the code should be modified to check if `feeTo` is set to a non-zero address before transferring fees. The bug was found through manual review of the code at the following link: https://github.com/code-423n4/2022-05-rubicon/blob/8c312a63a91193c6a192a9aab44ff980fbfd7741/contracts/RubiconMarket.sol#L297-L300.

## Vulnerability details

## Impact

In the `RubiconMarket` contract, the `feeTo` storage variable stores the recipient of taker trade fees (i.e. `buy` function). `feeTo` can be set to any arbitrary address in `setFeeTo(address newFeeTo)`. As there is no zero-address validation, the owner can set a zero-address as the `feeTo` address.

All subsequent `buy` function calls with a `_offer.buy_gem` ERC20 token reverting on zero-address transfers (e.g. `USDC`), will revert.

## Recommended mitigation steps
