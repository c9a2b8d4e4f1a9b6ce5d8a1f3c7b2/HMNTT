# Self-XSS in dApp Browser

## Summary
The client has implemented a recommendation to address a self-XSS issue in the `dapp-browser-tab` component. However, it may not prevent all scripting attacks and the client suggests using established libraries like DOMPurify and js-xss. The issue allows an attacker to access a user's local/session storage by tricking them into pasting a specific code into the URL input field. The client suggests expanding a feature that allows users to share links and potentially transform the self-XSS into a more dangerous reflected XSS vulnerability. This could compromise users' private information stored in the wallet's localStorage. The recommendation is to validate the URL beginning with `https` instead of just checking if it contains the string.

## Details
**Update**
The recommendation has been implemented. However, the specific implementation may not prevent all scripting attacks. The client would do well to consider using well-established libraries such as [DOMPurify](https://github.com/cure53/DOMPurify) and [js-xss](https://github.com/leizongmin/js-xss).

Marked as "Fixed" by the client. Addressed in: `063f993`. The client provided the following explanation: Have implemented global URL santization function

**Description:** The `dapp-browser-tab` component is vulnerable to a self-XSS issue, allowing an attacker to obtain all information in the browser’s local/session storage as long as they could trick a user into pasting something like `js:document.write(window.localStorage.hashconnectData)//https` (where `js` is `javascript`) into the URL input field under the DAPPS tab. This works because `loadCustomUrl()` only validates that the the URL passed by the user contains the string https without validating it is actually in the beginning of the supplied string.

We endeavored to elevate the self-XSS issue into a reflected XSS vulnerability but were unable to identify any supporting functionality to chain the attack. However, the HashPack wallet does offer a feature that enables browser navigation to dApps. If this functionality were expanded to allow users to share links that automatically open a dApp within the HashPack wallet, the self-XSS could potentially transform into a reflected XSS vulnerability. This would require fewer prerequisites to exploit and pose a greater risk of compromising users.

Typically, self-XSS is considered a low-severity issue. However, since HashPack is a non-custodial wallet that stores users' private information in encrypted form within the localStorage, even minor XSS vulnerabilities pose a significant threat to the integrity of users' private keys. Consequently, this could result in more severe consequences and compromise the security of user accounts.

**Recommendation:** Validate that the URL begins with `https` instead of checking whether it is part of the string.
