# GetBlobHeader is not rate limited allowing for DA Node DoS vector

## Summary
This report highlights a medium risk vulnerability in the `node.GetBlobHeader` request. The issue is that this request is not limited in any way and involves multiple resource-intensive operations, making it vulnerable to exploitation by unauthenticated attackers. The recommended solution is to implement global and origin-based rate limiting and to create a cache for the requested values to prevent excessive resource usage.

## Details
## Medium Risk Vulnerability Report

## Severity
**Medium Risk**

## Context
`node/grpc/server.go#L268`

## Description
The `node.GetBlobHeader` request is not rate limited in any way and contains multiple resource-intensive operations:

- **(s *Server) getBlobHeader**: retrieves blob headers from the node store database.
- **GetBlobHeaderHash()**: creates Keccak256 hashes (called multiple times in the request flow).
- **(s *Server) rebuildMerkleTree()**: reads batch headers from the node store db and builds a merkle tree of the batch header hashes.
- **GenerateProof()**: generates a merkle proof of the blob headers' inclusion.

There is no origin-based rate limiting and no caching of the requested values. This allows for infinite resource grieving of the DA node by unauthenticated adversaries.

## Recommendation
Implement global and origin-based rate limiting. When adding global rate limiting, be careful that it cannot be triggered by a malicious peer to prevent valid requests from being serviced. Implement a Blob-Header cache so that the hashing, database lookups, merkle tree, and merkle proof generation do not have to happen on each request.
