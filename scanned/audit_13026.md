# Centralization risks 

## Summary
The report discusses a vulnerability in the EnglishAuctionCollateralLiquidator smart contract. The vulnerability allows the admin to create a malicious bundle and use it to manipulate the auction process, potentially causing financial loss to bidders. This can happen in two scenarios - when the admin creates a malicious bundle and allows it using the `setCollateralWrapper` function, or when the admin approves a malicious wrapper. To fix this issue, the admin role has been removed and only 5 predefined collateral wrappers are allowed via the constructor. Additionally, it is recommended to use a multi-signature wallet for the admin role to reduce the risk. The issue has been resolved in the latest commit.

## Details
## EnglishAuctionCollateralLiquidator Vulnerability Analysis

## Context
Source: `EnglishAuctionCollateralLiquidator.sol#L612-L620`

## Description
The admin can allow new collateral wrappers using the `setCollateralWrapper` function, which could bring centralization risk.

### Scenario 1: Malicious Bundle Creation
1. Admin wants to prohibit the auction of collateral **X** with token ID **Y**.
2. Admin creates a malicious bundle and allows it using the `setCollateralWrapper` function.
3. This malicious bundle always returns collateral **X** with token ID **Y** when calling the `enumerate` function. On calling `unwrap`, it does nothing.
4. Admin creates an NFT using this malicious bundle and calls the `liquidate` function.
5. The `enumerate` function is called, which sets:
   - `underlyingCollateralToken` as **X**
   - `underlyingCollateralTokenIds` as **Y**

   ```solidity
   if (_collateralWrappers[collateralToken]) {
       /* Get underlying collateral token and underlying collateral token IDs */
       (underlyingCollateralToken, underlyingCollateralTokenIds) = ICollateralWrapper(collateralToken).enumerate(
           collateralTokenId,
           collateralWrapperContext
       );
   }
   ```

6. This means the auction will start on token ID **Y** of **X**, even though the contract does not have this token.
7. Bidding will start, and the bidder's funds will be locked.
8. The claim will fail since the contract does not have the NFT.
9. The bidder's funds become stuck, and if token ID **Y** actually needs to be placed for auction, it will also fail since there is already an auction in place.

### Scenario 2: Malicious Wrapper Approval
1. Admin simply approves a malicious wrapper.
2. Admin mints himself a new bundle token with NFTs **X**, **Y**, **Z** without actually obtaining these NFTs.
3. Admin borrows and provides the bundle token from step 2.
4. The borrow passes, and the admin benefits from the extra tokens.

### Another Instance
If a bot only calls `withdrawCollateral` on `ExternalCollateralLiquidator.sol` and `liquidateCollateral` is never called, this will mean that collateral will be taken without paying any proceeds.

## Recommendation
- The `_admin` could be a multi-signature wallet so that instead of relying on a single owner, multiple parties must approve an operation. This can help reduce the risk.
- As communicated by the product team, `ExternalCollateralLiquidator` is just for testing.

## MetaStreet Resolution
Resolved in commit `17388ef`.

## Cantina Resolution
Fixed. The admin has been removed, and only 5 predefined collateral wrappers are allowed via the constructor.
