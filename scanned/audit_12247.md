# Lack of vetoer can lead to 51% attack

## Summary
The bug report is about the lack of vetoer in VeloGovernor, which can lead to Velodrome losing their veto power unintentionally and open to 51% attack. A 51% attack refers to a malicious actor taking control of more than 50% of the computing power of a network, allowing them to control the network and manipulate transactions. This can be done by changing the governor in Voter contract or by bypassing the tokens whitelist. 

The recommendation was to add vetoer with two-step validation and add the function to execute a veto. The code for the veto function was provided as an example. Velodrome acknowledged the importance of the veto power and decided to use the _cancel function provided in Governor. They also decided to add the ability to set/change a vetoer.

Spearbit mentioned additional considerations during the migration from V1 to V2, such as the time when it would be very cheap to attack the Governor without a vetoer. They also listed out a set of attacks, such as whitelisting privileged pairs, raising the quorum, setting the governor to an EOA, and creating a managed lock. 

Velodrome fixed the bug in commit 64fe60 and Spearbit verified it.

## Security Assessment Report

## Severity
**Medium Risk**

## Context
- VeloGovernor.sol
- EpochGovernor.sol

## Description
The veto power is an important functionality in a governance system to protect against malicious proposals. However, there is a lack of a vetoer in VeloGovernor, which might unintentionally lead to Velodrome losing their veto power and becoming vulnerable to a 51% attack. 

In a 51% attack, a malicious actor can change the governor in the Voter contract or bypass the tokens whitelist, adding new gauges with malicious tokens.

## Recommendation
It is recommended to add a vetoer with two-step validation and implement a function to execute a veto. Below is an example of a veto function that should be adapted in the code:

```solidity
function veto(bytes32 _proposalId) external {
    // Ensure the caller is the vetoer
    require(msg.sender == vetoer, "Only vetoer");
    ProposalState status = state(proposalId);
    // Ensure the proposal has not been executed
    require(status != ProposalState.Canceled && status != ProposalState.Expired && status != ProposalState.Executed, "Proposal not active");
    
    // Get the pointer to the proposal
    Proposal storage proposal = proposals[_proposalId];
    // Update the proposal as vetoed
    proposal.vetoed = true;
    emit ProposalVetoed(_proposalId);
}
```

## Velodrome
Acknowledged, given the ability of this to severely disrupt normal operations of the protocol. The simplest way to implement this would be to use the `_cancel` function provided in Governor. We will also add the ability to set/change a vetoer, which will most likely be set to `emergencyCouncil`. Feedback will be solicited around this.

**Are there any recommendations regarding parameters that should be set?** e.g., the appropriate quorum fraction?

## Spearbit
Additional considerations, especially during migration from V1 to V2, indicate that there should be a time in which it will be very cheap to attack the Governor without a vetoer. The first attack could involve raising the quorum, which may force the V1 price to raise as it becomes more urgent to migrate to V2. This could enable new tokens that are malicious or privileged, aimed at obtaining emissions from V2 in perpetuity and establishing a head start that makes it impossible for others to catch up.

### Set of attacks:
- Whitelist privileged pair (to steal emissions)
- Raise quorum to make it harder/impossible to catch up
- Set governor to an EOA / hijack it
- Create a managed lock and refuse to create it for others
