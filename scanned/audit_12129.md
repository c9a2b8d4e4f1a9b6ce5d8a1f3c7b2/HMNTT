# [M-03] Centralization Risk in Multiple Places

## Summary
This bug report discusses issues with the code in two files, `PhiDaily.sol` and `BaseObject.sol`, which are part of the `DailyMaterial` project. The report states that the `setEmissionLogic` and `setMaterialObject` functions in `PhiDaily.sol` are not properly validating the addresses being set, allowing the admin to set any address instead of just contracts. Similarly, the `setTreasuryAddress`, `setMaxClaimed`, `setRoyalityFee`, and `setSecondaryRoyalityFee` functions in `BaseObject.sol` also do not have proper validation and can result in loss of funds for users. 

The affected code can be found in the `src/PhiDaily.sol` and `src/utils/BaseObject.sol` files. The report recommends adding proper address validation and upper-bound checks for the fees setter functions, despite the fact that they can only be called by the owner. The team has acknowledged the issue and plans to implement a timelock and multisig system to address the problem.

## Details
**Severity**

Medium Risk

**Description**

For `setEmissionLogic` and `setMaterialObject` in `PhiDaily.sol`, it's documented that `emissionLogic` and `materialObject` should be contracts but in reality, the admin can set any address.

With `setTreasuryAddress`, `setMaxClaimed`, `setRoyalityFee` and `setSecondaryRoyalityFee` in `BaseObject.sol` owner can set not validated values which can result in loss of funds for users.
