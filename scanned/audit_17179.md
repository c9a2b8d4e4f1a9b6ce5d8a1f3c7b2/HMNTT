# Lack of input validation

## Summary
The OptimisticGovernor contract has a general lack of input validation which could lead to a non-functional module instance and potential loss of funds. Specifically, the setUp function does not validate that the _finder and _rules arguments are non-zero and non-empty respectively. Additionally, the setLiveness and setRules functions have the same issue. The setUp and setBond functions allow a zero bond which the OptimisticOracle will accept, but it is difficult to discern if this is a legitimate value or an oversight of the validation logic. 

To avoid errors and unexpected system behavior, input validation should be implemented in the form of require statements. Where zero-values are acceptable, inline documentation should be left to avoid ambiguity. This bug has since been fixed in four pull requests and the commits associated with them.

## Details
The [`OptimisticGovernor`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol) contract has a general lack of input validation. For instance, the [`setUp`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L87) function does not validate that the [`_finder`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L90) argument is non-zero, which can lead to a non-functional module instance. Nor does `setUp` validate that the [`_rules`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L94) argument is non-empty, which could lead to a loss of funds if optimistic governance proposals are the sole way to manage an Avatar.


In the same function, `_liveness` is checked to be greater than zero, in line with the Optimistic Oracle’s [lower bound](https://github.com/UMAprotocol/protocol/blob/f258069fff4080a1b1ad26b3efcb1fe628e3f1a7/packages/core/contracts/oracle/implementation/OptimisticOracle.sol#L675) requirement. However, `setUp` does not check that `_liveness` is less than 5200 weeks, which is the Optimistic Oracle’s [upper bound](https://github.com/UMAprotocol/protocol/blob/f258069fff4080a1b1ad26b3efcb1fe628e3f1a7/packages/core/contracts/oracle/implementation/OptimisticOracle.sol#L674) requirement.


The [`setLiveness`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L148) and [`setRules`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L138) functions have the same sort of lacking input validation as `setUp` does.


Both the `setUp` and [`setBond`](https://github.com/UMAprotocol/protocol/blob/fca8e24275e928f7ddf660b5651eb93b87f70afb/packages/core/contracts/zodiac/OptimisticGovernor.sol#L119) functions allow a zero bond, which the `OptimisticOracle` will accept. However, because of the overall lack of input validation, it becomes difficult to discern if a zero bond is a legitimately acceptable value or simply an oversight of the validation logic.


To avoid errors and unexpected system behavior, consider implementing require statements to validate all user-controlled input. Where zero-values are acceptable, consider leaving inline documentation to that effect to avoid ambiguity.
