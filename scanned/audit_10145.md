# Signature proof not including a domain separator

## Summary
This bug report discusses a potential risk in two functions, purchaseAllowed() in ERC20DistributionNative.sol and ERC20Distribution.sol. The risk is that if a contract does not use a domainSeparator when verifying signatures, it could be vulnerable to cross-domain attacks. This means that an attacker could use a valid signature from one context (such as a different chain or address) to perform unauthorized actions in a different context. The recommendation is to follow the definition of domainSeparator in eip-712 to construct a secure proof. This issue has been resolved in a recent commit by adding chain.id and contract address information to the function hashForKYC(). However, it should be noted that a full eip-712 specification is not necessary according to the project's requirements.

## Details
**Description**

In ERC20DistributionNative.sol & ERC20Distribution.sol - Function purchaseAllowed () posses the following risk: If a contract does not use a domainSeparator when verifying signatures: bytes calldata proof, it becomes susceptible to cross-domain attacks. An attacker could potentially take a valid signature from one context (chain, address, ...) and use it to perform unauthorized actions in a different context.

**Recommendation**

See Definition of domainSeparator in eip-712 and follow it to construct a secure proof.

Resolved.
