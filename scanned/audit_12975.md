# Updating Message Service Does Not Emit Event ✓ Fixed

## Summary
This bug report concerns the Linea team's token bridge project. The function `setMessageService` allows the owner to update the message service address, but it does not emit any event reflecting the change. This means that if the owner is compromised, they can add a malicious message service to the project without anyone knowing. As a result, users can be exploited and their funds put at risk. To prevent this, the Linea team has implemented recommendations in pull request 69, with the final commit hash as `1fdd5cfc51c421ad9aaf8b2fd2b3e2ed86ffa898`, which suggests emitting an event reflecting the update from the old message service to the new one. This will help to ensure that users are aware of any changes to the message service and that off-chain monitoring tools can be used to alert users if any malicious activity is detected.

## Details
#### Resolution



The recommendations are implemented by the Linea team in the pull request [69](https://github.com/Consensys/linea-token-bridge/pull/69) with the final commit hash as `1fdd5cfc51c421ad9aaf8b2fd2b3e2ed86ffa898`


#### Description


The function `setMessageService` allows the owner to update the message service address. However, it does not emit any event reflecting the change. As a result, in case the owner gets compromised, it can silently add a malicious message service, exploiting users’ funds. Since, there was no event emitted, off-chain monitoring tools wouldn’t be able to trigger alarms and users would continue using rogue message service until and unless tracked manually.


#### Examples


**contracts/TokenBridge.sol:L237-L240**



```
function setMessageService(address \_messageService) public onlyOwner {
 messageService = IMessageService(\_messageService);
}

```
#### Recommendation


Consider emitting an event reflecting the update from the old message service to the new one.
