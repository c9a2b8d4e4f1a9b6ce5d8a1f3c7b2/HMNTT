# Hardcoded Operator Limit Logic

## Summary
This bug report discusses an issue with a contract that sets a limit on the number of operators that can be added. The current approach is not suitable for upgradeable contracts and may lead to mistakes. If the contract is upgraded and the limit is not changed, it will still restrict the number of operators that can be added. Additionally, the contract only supports one operator, which is hardcoded in the code. 

To address this issue, it is recommended to use a storage variable or constant to set the limit on operators during contract initialization. This will allow for more flexibility and prevent accidental mistakes. The contract should also consider supporting dynamic operator deposits in future versions instead of using a hardcoded index.

## Details
#### Description


The contract defines some hardcoded limits which is not the right approach for upgradeable contracts and opens doors for accidental mistakes, if not handled with care.


The operators for the current version are limited to 1. If the auditee team decides to open the system to work with more operators but fails to change the limit while upgrading, the upgraded contract will have no effect, and will still disallow any more operators to be added.

#### Recommendation


A better approach could be to constrain the limit of operators that can be added with a storage variable or constant, provided at the time of contract initialization. The contract should also consider supporting dynamic operator deposits for future versions instead of the default hardcoded index.
