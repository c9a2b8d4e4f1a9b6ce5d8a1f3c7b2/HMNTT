# Accounting available flow as available liquidity will lead to underestimate utilization

## Summary
The bug report is about an issue in the MiniPoolPiReserveInterestRateStrategy code, which is used to calculate available liquidity for mini pools. The problem is that the code counts available flow from the main lending pool as available liquidity, which leads to underestimating the utilization of the mini pool. This, in turn, affects the liquidity rate and the borrow rate for the mini pool. The recommendation is to remove the available flow from the available liquidity calculation, which would keep the utilization of the mini pool at 100% while bootstrapping liquidity, but would attract lenders to supply to the mini pool. The bug has been fixed in the latest commit.

## Details
## Severity: Medium Risk

## Context
`MiniPoolPiReserveInterestRateStrategy.sol#L106-L108`

## Description
When computing available liquidity for minipools, available flow from the main lending pool is counted as available liquidity.

### Code Reference
```solidity
// MiniPoolPiReserveInterestRateStrategy.sol#L91-L112:
function getAvailableLiquidity(address asset, address aToken)
    public
    view
    override
    returns (uint256 availableLiquidity, address underlying, uint256 currentFlow)
{
    (,, bool isTranched) = IAERC6909(aToken).getIdForUnderlying(asset);
    if (isTranched) {
        IFlowLimiter flowLimiter =
            IFlowLimiter(IMiniPoolAddressesProvider(_addressProvider).getFlowLimiter());
        underlying = IAToken(asset).UNDERLYING_ASSET_ADDRESS();
        address minipool = IAERC6909(aToken).getMinipoolAddress();
        currentFlow = flowLimiter.currentFlow(underlying, minipool);
        availableLiquidity = IERC20(asset).balanceOf(aToken) // <<<
            + IAToken(asset).convertToShares(flowLimiter.getFlowLimit(underlying, minipool)) // <<<
            - IAToken(asset).convertToShares(currentFlow); // <<<
    } else {
        availableLiquidity = IERC20(asset).balanceOf(aToken);
    }
}
```

However, this leads to underestimating minipool utilization, and in turn:

- If there is no "flow", liquidity rate is underestimated. Indeed, the liquidity rate is derived from the borrow rate by applying the utilization factor because borrow interest is shared proportionally to all available liquidity. However, in this case, "available flow" does not earn from this interest accrual, so there is no need to include it.

### Code Reference
```solidity
// BasePiReserveRateStrategy.sol#L297-L306:
function getLiquidityRate(
    uint256 currentVariableBorrowRate,
    uint256 utilizationRate,
    uint256 reserveFactor
) internal pure returns (uint256) {
    return currentVariableBorrowRate.rayMul(utilizationRate).percentMul(
        PercentageMath.PERCENTAGE_FACTOR - reserveFactor
    );
}
```

- If there is "flow" (i.e., minipool is borrowing from the main), the borrow rate is overestimated. In this case, a minimum liquidity rate is forced on the minipool so it can eventually repay its debt. The borrow rate is scaled by a factor of `1/utilization` to provide the liquidity rate to lenders. However, as stated above, "available flow" does not earn interest, causing overcharging of minipool borrowers.

### Code Reference
```solidity
// MiniPoolPiReserveInterestRateStrategy.sol#197-L223:
if (currentFlow != 0) {
    //@audit minliquidity rate formula
    ...
    // `&& utilizationRate != 0` to avoid 0 division. It's safe since the minipool flow is

```

## Recommendation
Please consider removing available flow from the available liquidity calculation:

### Code Reference
```solidity
// MiniPoolPiReserveInterestRateStrategy.sol#L91-L112:
function getAvailableLiquidity(address asset, address aToken)
    public
    view
    override
    returns (uint256 availableLiquidity, address underlying, uint256 currentFlow)
{
    (,, bool isTranched) = IAERC6909(aToken).getIdForUnderlying(asset);
    if (isTranched) {
        IFlowLimiter flowLimiter =
            IFlowLimiter(IMiniPoolAddressesProvider(_addressProvider).getFlowLimiter());
        underlying = IAToken(asset).UNDERLYING_ASSET_ADDRESS();
        address minipool = IAERC6909(aToken).getMinipoolAddress();
        currentFlow = flowLimiter.currentFlow(underlying, minipool);
        // Remove available flow from calculation
        availableLiquidity = IERC20(asset).balanceOf(aToken)
            + IAToken(asset).convertToShares(flowLimiter.getFlowLimit(underlying, minipool))
            - IAToken(asset).convertToShares(currentFlow);
    } else {
        availableLiquidity = IERC20(asset).balanceOf(aToken);
    }
}
```

Please note that this change would maintain the minipool utilization at 100% while bootstrapping liquidity (no organic liquidity provided to minipool yet). However, the liquidity rate would rise alongside the borrow rate, attracting lenders to supply to the minipool.
