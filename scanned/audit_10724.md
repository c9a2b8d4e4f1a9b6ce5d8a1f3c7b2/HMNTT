# Governance can backdoor new Safes via malicious upgrade

## Summary
This bug report is about the AddressProvider and the SafeDeployer contracts. The AddressProvider is responsible for setting the addresses for the SafeDeployer to use when deploying and configuring new accounts and sub-accounts. If the governance address of the AddressProvider is malicious, it could manipulate the addresses to deploy faulty or backdoored safes, which could be exploited by a backdoor installed by the governance. This would have a high impact on users, so the risk is set to medium.

The recommendation is to either hardcode the safe factory and singleton addresses into the SafeDeployer contract, or to allow users to pass them in as parameters, avoiding the AddressProvider as the source of truth. This would also improve the gas usage of the safe creation method by avoiding the expensive external call and storage access.

Brahma and Spearbit both acknowledged this recommendation, with Brahma noting that the singleton should be upgradable in case new bugs are discovered, and the severity for governance abuse and impact should be on low likelihood.

## Description
The SafeDeployer contract is responsible for deploying and configuring new "top-level" console accounts and sub-accounts. These are initialized via the authorized safe factory address with the authorized reference singleton (the implementation contract for Safe multi-signature wallets). These addresses upon which the SafeDeployer relies can be set by the governance address of the AddressProvider.

Malicious governance could manipulate these addresses to have new accounts deployed with an invalid, faulty, or even backdoored safe, meaning users could start unsuspectingly using their accounts and one day unknowingly be exploited by a backdoor that governance previously installed. This could be obfuscated by making the malicious configuration part of a sandwich attack whereby deploying transactions are wrapped with two transactions that set and unset the malicious variants. This would prevent users from detecting this by simply querying and verifying the address configured in the AddressProvider.

The likelihood of this happening is low, but if it occurs, the impact for the users is high. Therefore, we've set this to medium risk.

## Recommendation
This should be fixed by either hardcoding the safe factory and singleton addresses into the SafeDeployer contract or allowing users to pass them in as parameters, avoiding the provider as the core source of truth. This not only resolves this issue but will also improve the gas usage of the safe creation method as the addresses will be available locally (either via calldata parameter or hardcoded as a constant), meaning the expensive external call and storage access will be avoided.

## Feedback
**Brahma:** Acknowledged, within the trust scope of governance. Need singleton to be upgradable in case new bugs are discovered. Also, severity for governance abuse and impact should be of low likelihood.

**Spearbit:** Acknowledged.
