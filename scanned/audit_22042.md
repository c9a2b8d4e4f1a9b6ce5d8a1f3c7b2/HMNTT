## Summary
This bug report is about a known race condition in the ERC20 standard, which is a set of rules that govern how generic token contracts are created. The race condition is related to the approve and transferFrom functions, which give a third party permission to spend tokens. If the third party sees the transaction containing the call before it has been mined, they can call transferFrom to transfer the previous value and still receive the authorization to transfer the new value. 

For example, if Alice calls approve(Bob, 500) to give Bob permission to spend 500 tokens, but then changes her mind and calls approve(Bob, 1000), Bob can call transferFrom(Alice, X, 500) before Alice's transaction is mined. If Bob's transaction is mined first, Bob will be able to transfer 1500 tokens, which was not Alice's intention.

Unfortunately, there is no straightforward solution to this issue. One possible solution is to forbid a call to approve if all the previous tokens are not spent by adding a require to approve. Another possible solution is to use a temporal mutex, which would prevent a call to approve during a limited time. However, both of these solutions may result in unexpected behavior for a third party. 

Overall, this is a flaw in the ERC20 design that cannot be easily fixed without modifications to the standard.

## Description
The ERC20 standard contains a known race condition on the `approve` function, making possible the theft of tokens. The ERC20 standard describes how to create generic token contracts. Among others, an ERC20 contract has to define these two functions:

- `transferFrom(from, to, value)`
- `approve(spender, value)`

The goal of these functions is to grant permission to a third party to spend tokens. Once the function `approve(spender, value)` has been called by a user, `spender` can spend up to `value` tokens of the user by calling `transferFrom(user, to, value)`.

This schema is vulnerable to a race condition when the user calls `approve` a second time on an already allowed spender. If the spender sees the transaction containing the call before it has been mined, they can call `transferFrom` to transfer the previous value and still receive the authorization to transfer the new value.

## Exploit Scenario
1. Alice calls `approve(Bob, 500)`. This allows Bob to spend 500 tokens.
2. Alice changes her mind and calls `approve(Bob, 1000)`. This changes the number of tokens that Bob can spend to 1000.
3. Bob sees the transaction and calls `transferFrom(Alice, X, 500)` before it has been mined.

If Bob’s transaction is mined before Alice’s, 500 tokens will be transferred by Bob. Once Alice’s transaction is mined, Bob can call `transferFrom(Alice, X, 1000)`. As a result, Bob has transferred 1500 tokens even though this was not Alice’s intention.

## Recommendation
While this issue is known and can have a severe impact, there is no straightforward solution. 

One possible solution is to forbid a call to `approve` if all the previous tokens are not spent by adding a `require` to `approve`. This solution prevents the race condition but it may result in unexpected behavior for a third party.

```solidity
require(_approvals[msg.sender][guy] == 0)
```

Another possible solution is the use of a temporal mutex. Once `transferFrom` has been called for a user, it needs to prevent a call to `approve` during a limited time. The user can then verify if someone transferred the tokens. However, this solution adds complexity and may also result in unexpected behavior for a third party.

This issue is a flaw in the ERC20 design. It cannot be fixed easily without modifications to the standard.
