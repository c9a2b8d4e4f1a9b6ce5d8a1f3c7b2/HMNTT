# A malicious transaction object can trigger a Node DoS

## Summary
This bug report describes a high-risk issue in the Berachain code, specifically in the HashStaticBytes() function in the ssz/hasher.go file. This function takes in any type whose underlying type is a byte array of specific sizes. However, a recent change in the Berachain code added the ability to accept arrays of any size, including 0-length arrays, which can cause the program to crash. This bug was discovered when fuzzing the transaction processing flows and has been fixed in the latest commit. The recommendation is to not modify the commonBytesLengths interface and to define a separate type for variable length byte arrays. This issue has been verified and fixed in the Spearbit code.

## Details
## Severity: High Risk

## Context
[github.com/itsdevbear/ssz/hasher.go#L132](https://github.com/itsdevbear/ssz/hasher.go#L132)

## Description
Berachain uses forked versions of various SSZ libraries including [this one](https://github.com/itsdevbear/ssz), which is forked from [karalabe/ssz](https://github.com/karalabe/ssz). This library implements `HashStaticBytes()`:

```go
func HashStaticBytes[T commonBytesLengths](h *Hasher, blob *T) {
    // The code below should have used `blob[:]`, alas Go's generics compiler
    // is missing that (i.e. a bug): https://github.com/golang/go/issues/51740
    h.hashBytes(unsafe.Slice(&(*blob)[0], len(*blob)))
}
```

This function takes in any type whose underlying type is that of `commonBytesLengths`. This interface is defined in the upstream repository as any type whose underlying type is a byte array of the following sizes:

```go
type commonBytesLengths interface {
    // fork | nonce | address | verkle-stem | hash | pubkey | committee | signature | bloom
    ~[4]byte | ~[8]byte | ~[20]byte | ~[31]byte | ~[32]byte | ~[48]byte | ~[64]byte | ~[96]byte | ~[256]byte
}
```

An issue was introduced when a change to this interface was made in the Berachain-used fork that added the variable byte array size `~[]byte`:

```go
type commonBytesLengths interface {
    // fork | address | verkle-stem | hash | pubkey | committee | signature | bloom | blob & tx
    ~[4]byte | ~[20]byte | ~[31]byte | ~[32]byte | ~[48]byte | ~[64]byte | ~[96]byte | ~[256]byte | ~[131072]byte | ~[]byte
}
```

This addition allows for any function that accepts this interface type to accept an array of any size, including a 0-length array, introducing a panic: `runtime error: index out of range [0] with length 0` when the 0 indexing of the blob object is done. This is exposed in the dependencies of transaction processing in Berachain due to the way transactions are deserialized when they are hashed. This allows for an attacker to create a malicious transaction that can crash the beacon-kit process depending on how its execution client sanitizes these transactions.

This is triggerable from an untrusted buffer when transactions are hashed in `/mod/engine-primitives/pkg/engine-primitives/transactions.go` and `/mod/engine-primitives/pkg/engine-primitives/transactions_bartio.go` and was discovered when fuzzing Berachain's transaction processing flows.

## Recommendation
Do not modify the `commonBytesLengths` interface definition to include variable length byte arrays. If a variable length array is needed, then define another byte length type.

**Berachain:** Fixed in commit `8a041731`.  
**Spearbit:** Fix verified.
