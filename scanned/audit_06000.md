# Validator cutting board weights can disconnect from onchain voting outcomes 

## Summary
This bug report discusses a function called queueNewCuttingBoard in the Infrared contract. This function is used to queue new cutting board weights for an Infrared validator based on offchain data and analysis. However, there is a risk of incorrect representation of the onchain voting outcomes, which could lead to losses for some BGT receivers. The likelihood of this bug occurring is low, but it is recommended to implement a new contract to establish a trustless connection between the onchain voting outcomes and the process of queuing new cutting board weights. This could also include a view method for anyone to verify the currently queued weights against the voting outcomes.

## Details
## Function: queueNewCuttingBoard

## Context
`Infrared.sol#L826-L833`

### Function Signature
```solidity
function queueNewCuttingBoard(
    bytes calldata _pubkey,
    uint64 _startBlock,
    IBeraChef.Weight[] calldata _weights
) external onlyKeeper 
```

### Description
`Infrared.queueNewCuttingBoard()` queues new cutting board `_weights` (BGT receiver address & share percentage) for an Infrared validator with `_pubkey` that become active at `_startBlock`. This method, which is only accessible via Infrared's Keeper role, is a crucial component of the protocol that facilitates the representation of the IRED onchain voting distribution outcomes as actual cutting board weights for each Infrared validator.

However, the provided `_weights` data and `_startBlock` in this method must be fully trusted since it relies on offchain data/analysis by the Keeper based on the onchain voting outcomes but lacks any onchain connection or verification.

### Impact
**High**: Any failure to correctly represent the onchain voting outcomes as cutting board weights could lead to losses for specific BGT receivers, in addition to reputational damage due to deviation from the voting outcomes.

### Likelihood
**Low**: The Keeper role is expected to be trusted, and the relevant offchain code is assumed to be tested and reviewed.

### Recommendation
Consider implementing a new contract that establishes a trustless connection between the onchain voting outcomes and the process of queuing new cutting board weights. If the involved computations are too expensive concerning gas fees, a view method could be implemented to allow anyone to verify the currently queued weights against the voting outcomes.
