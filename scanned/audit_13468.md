# [M-02]

## Summary
This bug report is submitted by Koolex and is related to the liquid staking manager (Syndicate owner) calling the deRegisterKnots function to deregister knots. If any of those knots is inactive, the transaction will revert, which means in order to deregister an inactive knot, one should call the updateCollateralizedSlotOwnersAccruedETH function since it calls the internal _deRegisterKnots. 

The expected behaviour is for deregistering knots by Syndicate owner to succeed, especially if the knot is inactive. The cause of this behaviour is a circular dependency between deRegisterKnots and _updateCollateralizedSlotOwnersLiabilitySnapshot functions. The deRegisterKnots function always calls the internal _updateCollateralizedSlotOwnersLiabilitySnapshot and the latter one calls the _deRegisterKnot in case the knot is inactive. Therefore, when the knot is deregistered the first time, the second time it reverts. 

The property violated is that the isNoLongerPartOfSyndicate is not set to true since the function reverts. This bug was detected by the rule deregisterInactiveKnotShouldSucceed(). The assessment of the bug is medium/high severity as the logic will not revert when deregistering an inactive knot.

## Details
### Summary

Deregistering inactive knot via calling deRegisterKnots directly by the Syndicate owner always reverts.

### Description

**Actual Behaviour & Impact**
The liquid staking manager (Syndicate owner) calls `deRegisterKnots` function to deregister knots. However, if any of those knots is inactive, the transaction will revert. This means in order to deregister an inactive knot, one should call `updateCollateralizedSlotOwnersAccruedETH` function since it calls internal `_deRegisterKnots`.

**Expected Behaviour**
Deregistering knots by Syndicate owner should succeed especially if the knot is inactive.

**Cause**
There is a circular dependency between deRegisterKnots and `_updateCollateralizedSlotOwnersLiabilitySnapshot` functions which leads to this behaviour.

`deRegisterKnots` always calls internally `_updateCollateralizedSlotOwnersLiabilitySnapshot`, and `_updateCollateralizedSlotOwnersLiabilitySnapshot` calls `_deRegisterKnot` in case the knot is inactive. So when the knot is deregistered the first time, the second time it reverts.
