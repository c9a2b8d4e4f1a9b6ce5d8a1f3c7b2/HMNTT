# Rounding directions

## Summary
This bug report discusses a medium risk issue in the code for a financial application. The bug is located in two specific sections of the code and involves rounding up calculations for totalDebtInETH. The report recommends making sure all operations round up when calculating this value. It also suggests rounding up the amountScaled in another section of the code. Additionally, the report mentions a need to round up a variable or check for a specific condition in two other sections of the code. The bug has been fixed in a recent commit, except for one section which has been acknowledged.

## Details
## Severity: Medium Risk

## Context
- `GenericLogic.sol#L236-L242`
- `MiniPoolGenericLogic.sol#L202-L209`

## Description/Recommendation
- **GenericLogic.sol#L236-L242**, **MiniPoolGenericLogic.sol#L202-L209**: Make sure all operations round-up when calculating `totalDebtInETH`.
- **AToken.sol#L184-L185**: Best to round-up `amountScaled`.
- **ValidationLogic.sol#L205-L212**, **MiniPoolValidationLogic.sol#L208-L214**: It would be best to round up `vars.amountOfCollateralNeededETH` or instead check:
  
  ```
  vars.userBorrowBalanceETH + validateParams.amountInETH <= (vars.userCollateralBalanceETH).percentMul(vars.currentLtv)
  ```
