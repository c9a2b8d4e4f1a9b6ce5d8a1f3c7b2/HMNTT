# [M-06] totalUnderlyingMinusSponsored() may revert on underflow and malfunction the contract

## Summary
This bug report is about a vulnerability in the Vault smart contract of the Sandclock project. The vulnerability is in the `totalUnderlyingMinusSponsored()` function, which can revert on underflow when `sponsorAmount > totalUnderlying()`. This is possible and has been considered elsewhere in the contract. A proof of concept is provided, which involves a sponsor calling the `sponsor()` function and sending 10,000 USDT, then the `NonUSTStrategy.sol#doHardWork()` swaps USDT for UST, with the resulting `pendingDeposits` being 9,996 and `totalUnderlying()` being 9,996. When Alice tries to call `deposit()`, the transaction will revert due to an underflow in `totalUnderlyingMinusSponsored()`. The recommendation is to change the function to the following: 

```solidity
function totalUnderlyingMinusSponsored() public view returns (uint256) {
    uint256 _totalUnderlying = totalUnderlying();
    if (totalSponsored > _totalUnderlying) {
        return 0;
    }
    return _totalUnderlying - totalSponsored;
}
```

### PoC

- Underlying token = USDT
- Swap Fee = 0.04%

1. Sponsor call `sponsor()` and send 10,000 USDT

- totalSponsored = 10,000

2. `NonUSTStrategy.sol#doHardWork()` swapped USDT for UST

- pendingDeposits = 9,996
- totalUnderlying() = 9,996

3. Alice tries to call `deposit()`, the tx will revet due to underflow in `totalUnderlyingMinusSponsored()`.

### Recommendation

Change to:

```solidity
function totalUnderlyingMinusSponsored() public view returns (uint256) {
    uint256 _totalUnderlying = totalUnderlying();
    if (totalSponsored > _totalUnderlying) {
        return 0;
    }
    return _totalUnderlying - totalSponsored;
}
```
