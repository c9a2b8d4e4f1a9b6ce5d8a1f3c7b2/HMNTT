# Be careful with small transactions

## Summary
This bug report is about the token calculation at line 90 of HackerGold.sol, which is used in the EtherCamp Virtual Accelerator. The issue is that when a call is made to the function `createHKG` with a `msg.value` less than 0.001 ether, the result of the integer division used in the calculation is truncated to an integer value, meaning that the tokens created are zero. This means that any amount of ether after the 3rd decimal place is ignored, and if 0.12345678 ether is sent to `createHKG`, only 24.6 HKG will be created and assigned to the `msg.sender`. 

To fix the issue, the line should be changed to: `uint tokens = msg.value * getPrice() / 1000000000000000;`. This yields 24.691 HKG for the previous example, which is better, but still ignores some small amounts. It is also recommended to warn users against sending amounts smaller than 0.001 ether and to use checked operations for math operations. EtherCamp fixed this problem in a commit.

## Details
Token calculation [at line 90 of HackerGold.sol](https://github.com/ether-camp/virtual-accelerator/blob/2529ffe5efd5294b44f1bc89dc9a4721a7b16409/contracts/HackerGold.sol#L90) uses an integer division by 1,000,000,000,000,000, [which truncates the result to an integer value](http://solidity.readthedocs.io/en/develop/types.html#integers). Any call to the function `createHKG` with a `msg.value` less than 0.001 ether will be lost, resulting in a 0 token balance. Note that the value of 0.0005 ether in theory corresponds to 0.1 HKG, which is a valid representation (less than 3 decimals), but in this case it is considered zero HKG.


Moreover, any amount of ether after the 3rd decimal place will be ignored. If 0.12345678 ether is sent to `createHKG`, only 24.6 HKG will be created and assigned to the `msg.sender`.


Consider changing that line to:



```
uint tokens = msg.value * getPrice() / 1000000000000000;
```

This yields 24.691 HKG for the previous example, which is better, but still ignores some small amounts. Consider warning users against sending amounts smaller than 0.001 ether.


Also, consider changing the math operations to checked operations, as mentioned in the previous point.


EDIT: EtherCamp fixed this problem [in this commit](https://github.com/ether-camp/virtual-accelerator/commit/99d2940f91db20a074c5c20cb7ddf53fda255e59)
