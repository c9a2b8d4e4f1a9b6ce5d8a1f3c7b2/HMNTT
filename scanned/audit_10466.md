# Superform protocol doesn 't support vaults if their underlying asset is a fee-on-transfer type token 

## Summary
The Vault Protocol allows users to add their vaults to the protocol, but it was noticed that the protocol does not support tokens that deduct fees from transferred amounts. This means that users cannot deposit in vaults with these types of tokens. The report recommends updating the function to support these tokens.

## Description
The protocol allows any vault owner to add their vaults to the protocol by wrapping them to one of the approved form implementations. These vaults can have any type of underlying assets, including fee-on-transfer token types, which deduct a fee from the transferred amount. As a result, the final balance of the receiver would be less than the sent amount by the amount of the deducted fees.

It was noticed that the protocol doesn't support such types of tokens. Users can't deposit in vaults with fee-on-transfer tokens due to this check made in the `ERC4626FormImplementation._processDirectDeposit` function:

```solidity
vars.assetDifference = IERC20(vars.asset).balanceOf(address(this)) - vars.balanceBefore;
// @dev the difference in vault tokens, ready to be deposited, is compared with the amount inscribed in the
// superform data
if (vars.assetDifference < singleVaultData_.amount) {
    revert Error.DIRECT_DEPOSIT_INVALID_DATA();
}
```

Where `vars.assetDifference` will always be less than `singleVaultData_.amount` due to the deducted fees (for fee-on-transfer underlying vault tokens).

## Recommendation
Update the function to support fee-on-transfer tokens.
