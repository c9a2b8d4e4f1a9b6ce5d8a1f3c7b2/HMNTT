### Title
Missing Account Existence Validation in `getAccountTokenAllowances()` Returns HTTP 200 for Non-Existent Accounts

### Summary
`getAccountTokenAllowances()` in `rest/controllers/tokenAllowanceController.js` calls `EntityService.getEncodedId()` to resolve a numeric account ID but never calls `EntityService.isValidAccount()` to verify the account exists in the database. For any syntactically valid but non-existent numeric account ID (e.g., `0.0.99999999`), the endpoint returns HTTP 200 with `{"allowances": [], "links": {"next": null}}` instead of HTTP 404, contradicting both the OpenAPI specification and the behavior of the comparable `getTokenRelationships()` endpoint.

### Finding Description
**Exact code path:**

- `rest/controllers/tokenAllowanceController.js`, `getAccountTokenAllowances()`, lines 68–81:
  - Line 69: `const accountId = await EntityService.getEncodedId(...)` — resolves the ID
  - No subsequent `EntityService.isValidAccount(accountId)` call exists
  - Line 72: proceeds directly to `TokenAllowanceService.getAccountTokenAllowances(query)`, which returns an empty result set for a non-existent owner

**Root cause:**

`EntityService.getEncodedId()` (`rest/service/entityService.js`, lines 118–137) has a critical split in behavior:
- For **numeric/shard.realm.num IDs** (line 122–123): returns `entityId.getEncodedId()` — a **pure in-memory computation** with no database lookup, no existence check.
- For **EVM addresses** (line 124): calls `getEntityIdFromEvmAddress()` which queries the DB and throws `NotFoundError` if absent.
- For **aliases** (line 126): calls `getAccountIdFromAlias()` which queries the DB and throws `NotFoundError` if absent.

So for numeric IDs, `getEncodedId()` never touches the database and always succeeds for any syntactically valid number.

**Contrast with `getTokenRelationships()`** (`rest/controllers/tokenController.js`, lines 66–92):
```js
// Lines 67-71 — the guard that tokenAllowanceController is missing:
const accountId = await EntityService.getEncodedId(...);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) {
  throw new NotFoundError();
}
```

**Test spec confirmation** (`rest/__tests__/specs/accounts/{id}/allowances/tokens/account-not-found.json`, lines 1–14):
```json
{
  "url": "/api/v1/accounts/1001/allowances/tokens",
  "responseStatus": 200,
  "responseJson": { "allowances": [], "links": { "next": null } }
}
```
This is the **documented, tested, confirmed** behavior — HTTP 200 for a non-existent account queried by numeric ID.

The OpenAPI spec (`rest/api/v1/openapi.yml`, lines 334–337) declares that this endpoint SHOULD return 404, making this a spec violation.

`cryptoAllowanceController.js` (`getAccountCryptoAllowances`, lines 76–98) has the **same missing check**, broadening the affected surface.

### Impact Explanation
Any client application that uses this endpoint to determine whether an account exists and has token allowances will receive a false-positive HTTP 200 with an empty list for any non-existent account ID. Downstream systems that gate logic on "does this account have allowances?" (e.g., pre-flight checks before submitting an allowance-dependent transaction, wallet UIs displaying account state, compliance tools auditing allowance grants) will silently treat phantom accounts as real accounts with zero allowances. This is an information-integrity violation: the API asserts a false state about the ledger. Severity is medium — no direct fund movement is possible through the mirror node (read-only), but the misleading response can corrupt application-layer decision logic.

### Likelihood Explanation
No authentication or special privilege is required. Any unauthenticated HTTP client can trigger this with a single GET request using any syntactically valid but non-existent numeric account ID. The behavior is deterministic and 100% reproducible. The test suite explicitly encodes this as expected behavior, meaning it has persisted through code review cycles. Likelihood is high.

### Recommendation
Add the same `isValidAccount` guard used in `getTokenRelationships()` immediately after `getEncodedId()` in both `getAccountTokenAllowances()` and `getAccountCryptoAllowances()`:

```js
// In tokenAllowanceController.js, after line 69:
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) {
  throw new NotFoundError();
}
```

Update the corresponding test spec `account-not-found.json` to expect `responseStatus: 404` for numeric non-existent accounts, consistent with the alias case already tested in the same file (lines 15–27).
