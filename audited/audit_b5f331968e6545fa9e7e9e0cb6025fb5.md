### Title
GraphQL `findByAlias` Returns Entities with `deleted = NULL` That Should Be Treated as Absent

### Summary
The GraphQL `EntityRepository.findByAlias()` uses the SQL predicate `deleted is not true`, which in PostgreSQL's three-valued logic evaluates to TRUE for both `deleted = FALSE` and `deleted = NULL`. This causes the GraphQL endpoint to return entities whose deletion status is NULL — a state that all other mirror-node modules (rest-java, importer) explicitly treat as absent by using `deleted <> true`. An unprivileged user can trigger this by submitting a well-formed GraphQL alias query, receiving an entity record that should not be visible.

### Finding Description
**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 13:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
```

**Root cause — SQL NULL semantics divergence:**

| Module | Query predicate | Returns `deleted = NULL`? |
|---|---|---|
| GraphQL (`EntityRepository`) | `deleted is not true` | **YES** |
| Importer (`EntityRepository`) | `deleted <> true` | No |
| Rest-Java (`EntityRepository`) | `deleted <> true` | No |

In PostgreSQL, `NULL IS NOT TRUE` → `TRUE`, so the GraphQL query matches rows where `deleted` is NULL. `NULL <> TRUE` → `NULL` (unknown), so the importer/rest-java queries correctly exclude those rows.

**Why `deleted = NULL` exists:** A DB migration explicitly made the column nullable: *"allow nullable on entity deleted as transaction cannot make this assumption on updates"* (`V1.39.1__upsert_support.sql`). During upsert operations where a transaction does not carry an explicit delete flag, the column is left NULL. This is a documented, reachable DB state.

**Authoritative expected behavior:** The rest-java integration test (`EntityRepositoryTest.findByAlias`, lines 23–28) explicitly creates an entity with `deleted = null` and asserts `isEmpty()` — i.e., it must not be returned. The GraphQL `EntityRepositoryTest.findByAlias` (lines 22–26) tests only the `deleted = false` happy path and has no coverage for the NULL case, leaving the bug undetected.

**Call chain:**
1. `AccountController.account()` (line 48) → `EntityService.getByAliasAndType(alias, ACCOUNT)`
2. `EntityServiceImpl.getByAliasAndType()` (line 30) → `entityRepository.findByAlias(decodeBase32(alias))`
3. `EntityRepository.findByAlias()` (line 13–14) — executes `deleted is not true`, returns the NULL-deleted entity
4. Result is mapped and returned to the caller with no further deletion check

### Impact Explanation
Any entity row in the `entity` table whose `deleted` column is NULL and which has a known alias will be returned by the GraphQL `/graphql/alpha` endpoint. This causes the mirror node to export an account record that all other subsystems (REST API, importer) treat as absent. Downstream consumers relying on the GraphQL API receive stale or transitional entity state that should not be observable, violating the mirror node's data-consistency contract.

### Likelihood Explanation
No authentication or authorization is required to query the GraphQL endpoint — it is publicly accessible. The attacker only needs a valid Base32-encoded alias, which is derivable from on-chain data or by observing network traffic. The `deleted = NULL` state is a normal intermediate state produced by the importer's upsert logic and can persist for any entity that has not had an explicit delete transaction processed. The condition is repeatable and deterministic.

### Recommendation
Change the GraphQL `EntityRepository` query predicates to match the rest-java/importer behavior:

```java
// EntityRepository.java (graphql module)
@Query(value = "select * from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

Additionally, add a test case to `EntityRepositoryTest.findByAlias()` (and `findByEvmAddress()`) that persists an entity with `deleted = null` and asserts `isEmpty()`, mirroring the existing rest-java test coverage.

### Proof of Concept
**Precondition:** An entity row exists in the `entity` table with a non-null `alias` and `deleted = NULL` (e.g., created by an upsert that did not carry a delete flag).

**Steps:**
1. Obtain the Base32 encoding of the entity's alias bytes.
2. Submit the following GraphQL query to the unauthenticated endpoint (`POST /graphql/alpha`):
```graphql
query {
  account(input: { alias: "<BASE32_ALIAS>" }) {
    alias
    deleted
    entityId { shard realm num }
  }
}
```
3. **Expected (correct) result:** `null` — entity is absent.
