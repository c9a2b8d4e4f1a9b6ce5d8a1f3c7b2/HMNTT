### Title
Unauthenticated Self-Referential Request Amplification DoS via `GET /health/readiness`

### Summary
`checkNetworkStatus()` in `rosetta/app/middleware/health.go` registers a readiness check that makes two outbound HTTP calls back to the **same** server process (`/network/list` then `/network/status`) on every invocation. Because `/health/readiness` has no application-level rate limiting or concurrency cap, an unprivileged attacker who can reach the service port directly can flood this endpoint, causing a 3× request amplification loop that exhausts goroutines and memory, ultimately starving the server's ability to accept new connections.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `checkNetworkStatus()`, lines 80–101:

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
    serverUrl := fmt.Sprintf("http://localhost:%d", port)   // same server, same port
    cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
    rosettaClient := client.NewAPIClient(cfg)

    return func(ctx context.Context) (checkErr error) {
        networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, ...)   // call 1 → /network/list
        ...
        _, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, ...)            // call 2 → /network/status
        ...
    }
}
```

This closure is registered as a `health.Config` check with a **10-second timeout** and `SkipOnErr: false` (lines 44–49). The `hellofresh/health-go` library does **not** cache results; it re-executes every registered check on every incoming request to `/health/readiness`.

**Root cause:** The health check calls back into the same `net/http` server that is serving the health request. There is no rate limit, no concurrency semaphore, and no result cache at the application layer. Each external `GET /health/readiness` generates two additional internal HTTP requests to the same process, creating a 3× amplification.

**Why existing mitigations are insufficient:**

The Traefik middleware chain (`inFlightReq: 5`, `rateLimit: average: 10`) defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 has two fatal gaps:
1. `global.middleware` defaults to `false` (line 95), so the Traefik `Middleware` CRD is never created unless explicitly enabled.
2. Even when enabled, the middleware applies only to ingress-routed paths. The ingress rules (lines 119–133) expose only `/rosetta/account`, `/rosetta/block`, etc. — `/health/readiness` is **not** in the ingress path list and is therefore never filtered by Traefik.

The Go HTTP server itself (`rosetta/main.go` lines 220–227) has no `http.Server`-level concurrency limit and no middleware that rate-limits the health path.

**Amplification mechanics under attack:**

- Attacker sends *N* req/s to `GET /health/readiness`.
- Each request spawns goroutines for both the `postgresql` check and the `network` check (hellofresh/health-go runs checks concurrently).
- The `network` check goroutine opens two new TCP connections back to `localhost:5700` and waits up to 10 s for responses.
- Those two internal requests are themselves served by the same goroutine pool, each potentially triggering further DB queries.
- At steady state with a 10 s timeout: **30 × N goroutines** are live simultaneously.
- At N = 500 req/s: ~15,000 goroutines, each with a growing stack, consuming hundreds of MB of RAM.
- When the Go runtime is saturated, `net.(*TCPListener).Accept()` is not called fast enough; the OS TCP accept backlog (default 128 on Linux) fills and new SYN packets are silently dropped, denying service to legitimate API callers.

### Impact Explanation

The Rosetta API is the primary programmatic interface for exchange integrations and custodians interacting with the Hedera/Hiero network. A sustained DoS against this endpoint would prevent balance queries, block fetches, and transaction construction for all clients. Because the health endpoint is on the same port (5700) as the API, exhausting the server's connection capacity affects all API paths simultaneously, not just health checks. This meets the "non-network-based DoS" classification because the amplification is internal (loopback), not dependent on network bandwidth.

### Likelihood Explanation

The precondition is direct reachability of port 5700 on the pod or service, bypassing the ingress. This is realistic in three common scenarios:
1. The Kubernetes `Service` is of type `LoadBalancer` or `NodePort` (common for public blockchain APIs).
2. An attacker has compromised any pod in the same namespace/cluster (lateral movement).
3. A misconfigured network policy exposes the service port.

No credentials, tokens, or special headers are required. The attack is trivially scriptable (`ab -n 100000 -c 200 http://<target>:5700/health/readiness`) and fully repeatable. The 10-second timeout window means even a modest 100 req/s flood sustains ~3,000 goroutines continuously.

### Recommendation

1. **Break the self-reference**: Move the network readiness check to call a dedicated internal probe endpoint that does **not** re-enter the same HTTP server, or use a direct in-process function call to the `NetworkAPIService` instead of an outbound HTTP client.
2. **Add a result cache**: Wrap the check closure with a short-lived cache (e.g., 5–10 s TTL) so repeated requests within the window return the cached result without spawning new goroutines or connections.
3. **Add a concurrency semaphore**: Gate the health handler with a `semaphore` (e.g., `golang.org/x/sync/semaphore`) limiting simultaneous in-flight health checks to a small number (e.g., 3).
4. **Expose health on a separate port**: Serve `/health/*` on a dedicated internal-only port (not 5700) so it is never reachable from the public ingress or external load balancer.
5. **Enable and extend the Traefik middleware**: Set `global.middleware: true` and add `/health/readiness` to the rate-limited ingress paths, or add an equivalent `rateLimit` middleware at the application layer.

### Proof of Concept

**Preconditions:** Direct TCP access to the Rosetta service on port 5700 (e.g., from within the cluster or via a LoadBalancer IP).

```bash
# Step 1 – confirm the endpoint is reachable and triggers internal calls
curl -s http://<ROSETTA_HOST>:5700/health/readiness

# Step 2 – flood with concurrent requests (no auth required)
ab -n 500000 -c 300 http://<ROSETTA_HOST>:5700/health/readiness &

# Step 3 – observe goroutine explosion via metrics (same port, no auth)
watch -n1 'curl -s http://<ROSETTA_HOST>:5700/metrics | grep rosetta_request_inflight'

# Step 4 – confirm legitimate API requests are dropped
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://<ROSETTA_HOST>:5700/network/list \
  -H "Content-Type: application/json" \
  -d '{}'
# Expected result: connection timeout or connection refused
```

**Expected result:** Within seconds of step 2, step 4 begins returning connection timeouts as the server's accept queue fills. The `rosetta_request_inflight` metric climbs unboundedly. Stopping the flood allows the server to recover (goroutines drain after the 10 s timeout), confirming the causal link.
