### Title
Soft-Deleted Hook Records with `adminKey` Exposed via Missing `deleted=false` Filter in `findByOwnerIdAndHookIdIn`

### Summary
`HookRepository.findByOwnerIdAndHookIdIn` contains no `deleted=false` predicate, so Spring Data JPA returns soft-deleted `Hook` rows alongside live ones. Because `AbstractHook.adminKey` carries no `@JsonIgnore` annotation and `HookMapper` auto-maps all matching fields to the REST model, the raw cryptographic key bytes of a deleted hook are serialised into the public API response. Any unauthenticated caller who can supply a valid `ownerId`/`hookId` pair can retrieve key material that was supposed to be inaccessible after deletion.

### Finding Description

**Exact code path**

`HooksController.getHooks` → `HookServiceImpl.getHooks` → `HookRepository.findByOwnerIdAndHookIdIn` → `HookMapper.map` → JSON response.

`HookRepository` (rest-java): [1](#0-0) 

The derived-query method name contains only `ByOwnerIdAndHookIdIn`; Spring Data generates `WHERE owner_id = ? AND hook_id IN (?)` with no `AND deleted = false` clause.

`HookServiceImpl` calls this method unconditionally when equal-filter hook IDs are present: [2](#0-1) 

`AbstractHook` declares both the `deleted` flag and the `adminKey` byte array: [3](#0-2) 

`adminKey` is annotated only with `@ToString.Exclude` — there is no `@JsonIgnore` and no explicit `@Mapping(ignore=true)` in `HookMapper`: [4](#0-3) 

MapStruct therefore auto-maps `adminKey` into the REST model, and Jackson serialises it into the HTTP response.

**Contrast with storage layer** — `HookStorageRepository` methods correctly append `AndDeletedIsFalse`: [5](#0-4) 

The same omission exists in `findByOwnerIdAndHookIdBetween` (line 15 of `HookRepository`), but the question scope is `findByOwnerIdAndHookIdIn`.

**Root cause / failed assumption**: The developer assumed that hooks are never logically deleted while still present in the `hook` table, or that the `deleted` flag would be filtered elsewhere. Neither assumption holds; the table uses soft-delete semantics (`deleted` Boolean column, `@UpsertColumn` with default coalesce), and no post-query filter exists in the service or controller.

### Impact Explanation
An attacker obtains the raw `adminKey` bytes of a hook that the owner has deleted. `adminKey` is a cryptographic key used to authorise administrative operations on the hook. Exposure of this material after deletion violates the security invariant that deletion revokes access. Severity is **High**: the leaked data is cryptographic key material, not merely metadata.

### Likelihood Explanation
The endpoint `/api/v1/accounts/{ownerId}/hooks?hookId={id}` is a public read API with no authentication gate visible in the controller. `ownerId` values are Hedera account IDs (enumerable from the ledger); `hookId` values are sequential longs. An attacker can enumerate both with a modest number of HTTP requests. The exploit is fully repeatable and requires no special privileges.

### Recommendation
1. Rename the repository method to include the deleted filter:
   ```java
   List<Hook> findByOwnerIdAndHookIdInAndDeletedFalse(
       long ownerId, Collection<Long> hookIds, Pageable pageable);
   ```
   Apply the same fix to `findByOwnerIdAndHookIdBetween`.
2. As defence-in-depth, add `@JsonIgnore` (or `@Mapping(target="adminKey", ignore=true)` in `HookMapper`) so `adminKey` is never serialised even if a query regression occurs.
3. Add a repository test asserting that deleted hooks are excluded from both query methods.

### Proof of Concept

**Preconditions**: A hook with `ownerId=1000`, `hookId=42` exists in the database with `deleted=true` and a non-null `adminKey`.

**Steps**:
```
GET /api/v1/accounts/1000/hooks?hookId=42
```

**Expected (correct) result**: `{"hooks": []}` — deleted hook not returned.

**Actual result** (current code):
```json
{
  "hooks": [{
    "hookId": 42,
    "adminKey": "<base64-encoded cryptographic key bytes>",
    "deleted": true,
    ...
  }]
}
```

The `adminKey` bytes of the deleted hook are returned to any caller, with no authentication required.
