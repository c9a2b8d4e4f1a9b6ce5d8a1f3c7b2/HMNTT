### Title
Unauthenticated DB Connection Pool Exhaustion via Unique EVM Address Lookups on Account Routes

### Summary
The REST API account sub-routes (e.g., `/api/v1/accounts/:id/nfts`) accept EVM addresses as path parameters with no rate limiting. For each unique EVM address, `EntityService.getEncodedId()` unconditionally issues a live database query via `getEntityIdFromEvmAddress()`. An unprivileged attacker flooding the endpoint with distinct random EVM addresses can saturate the default 10-connection DB pool, causing all subsequent legitimate requests to queue and time out.

### Finding Description

**Code path:**

1. `rest/routes/accountRoute.js` line 15 registers the route with no rate-limiting middleware:
   ```
   router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
   ```
   The middleware stack in `rest/server.js` (lines 67–99) contains only `authHandler` (optional basic-auth), metrics, and an opt-in Redis response cache — no request-rate limiter.

2. `rest/controllers/accountController.js` line 91 calls `getEncodedId` unconditionally:
   ```js
   const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
   ```

3. `rest/service/entityService.js` lines 118–124: when the input is a valid EVM address string, `getEncodedId` branches to `getEntityIdFromEvmAddress`:
   ```js
   return entityId.evmAddress === null
     ? entityId.getEncodedId()
     : await this.getEntityIdFromEvmAddress(entityId, requireResult);
   ```

4. `rest/service/entityService.js` lines 90–104: `getEntityIdFromEvmAddress` executes a live DB query with no caching:
   ```js
   const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
     [Buffer.from(entityId.evmAddress, 'hex')]);
   ```
   where `entityFromEvmAddressQuery` is:
   ```sql
   SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
   ```

**Why the existing cache does not help:**
`rest/entityId.js` lines 301–332 (`parseCached`) caches only the *parsed `EntityId` object* (the string→`{shard,realm,num,evmAddress}` mapping). It does **not** cache the result of the DB lookup. For every unique EVM address, `getEntityIdFromEvmAddress` always hits the database.

**DB pool size:**
`docs/configuration.md` line 556 documents the default:
```
hiero.mirror.rest.db.pool.maxConnections = 10
```
`rest/dbpool.js` line 14 applies this as `max: config.db.pool.maxConnections`.

### Impact Explanation
With only 10 default DB connections and no rate limiting, an attacker sending ≥10 concurrent requests with distinct EVM addresses ties up all connections. New legitimate requests queue and wait up to `connectionTimeout` (default 20 000 ms) before failing. This constitutes a full REST API denial-of-service for all users of the mirror node's account endpoints, with no economic cost to the attacker.

### Likelihood Explanation
No authentication is required. The attack requires only an HTTP client capable of sending concurrent requests with randomized 40-hex-character path segments. It is trivially scriptable, repeatable, and requires no special knowledge of the network state. The attacker does not need to know any valid account addresses.

### Recommendation
1. **Add a rate limiter** (e.g., `express-rate-limit` or `express-slow-down`) as middleware on account sub-routes or globally in `rest/server.js`, keyed by IP address.
2. **Cache EVM address DB lookup results** inside `getEntityIdFromEvmAddress` using the existing `entityId` LRU cache (already configured with `maxSize: 100000`, `maxAge: 1800s`), so repeated or concurrent lookups for the same address do not each consume a DB connection.
3. **Increase the default pool size** or configure a connection timeout that rejects queued requests quickly rather than holding them for 20 seconds.

### Proof of Concept
```bash
# Send 50 concurrent requests with unique random EVM addresses
for i in $(seq 1 50); do
  ADDR=$(openssl rand -hex 20)
  curl -s "http://<mirror-node>/api/v1/accounts/0x${ADDR}/nfts" &
done
wait
# Legitimate requests now queue/timeout; DB pool is saturated
```

Each request triggers `EntityService.getEncodedId()` → `getEntityIdFromEvmAddress()` → `SELECT id FROM entity WHERE evm_address = $1`, consuming one DB connection for the duration of the query. With the default pool of 10, 10 concurrent such requests exhaust all connections.
