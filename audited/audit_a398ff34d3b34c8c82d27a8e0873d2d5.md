### Title
Missing `deleted` Check for Numeric Encoded Entity IDs in `getEncodedId()`

### Summary
`getEncodedId()` in `rest/service/entityService.js` applies a `deleted` filter only when resolving by EVM address or alias, but returns the encoded ID directly—without any database lookup—when the input is a numeric encoded ID string. Any unprivileged user can supply a numeric encoded ID for a deleted entity and receive a valid encoded ID back, bypassing the deletion guard entirely. Downstream API handlers that rely on `getEncodedId()` as the sole validation gate will then operate on deleted-entity IDs.

### Finding Description

**Code path:**

`rest/service/entityService.js`, lines 118–137:

```js
async getEncodedId(entityIdString, ...) {
  if (EntityId.isValidEntityId(entityIdString)) {          // line 120
    const entityId = EntityId.parseString(entityIdString, {paramName});
    return entityId.evmAddress === null
      ? entityId.getEncodedId()                            // line 123 — NO DB call
      : await this.getEntityIdFromEvmAddress(entityId, requireResult);
  } else if (AccountAlias.isValid(entityIdString)) {
    return await this.getAccountIdFromAlias(...);          // DB call with deleted check
  }
}
```

**Root cause:** `encodedEntityIdRegex = /^-?\d{1,19}$/` in `rest/entityId.js` (line 38) matches any plain numeric string. `isValidEntityId()` (line 134) returns `true` for it. `parseString()` routes it through `parseFromEncodedId()` (line 325), which always returns `evmAddress = null` (line 211). The ternary at line 122–124 therefore takes the left branch and returns the locally-computed encoded ID with zero database interaction.

**Contrast with other paths:**

| Input type | DB query | `deleted` check |
|---|---|---|
| EVM address | `entityFromEvmAddressQuery` | `WHERE deleted <> true` ✓ |
| Alias | `entityFromAliasQuery` | `WHERE coalesce(deleted, false) <> true` ✓ |
| Numeric / `shard.realm.num` | **None** | **None** ✗ |

**Test confirmation** (`rest/__tests__/service/entityService.test.js`, line 161–163):
```js
test('EntityService.getEncodedId - No match', async () => {
  // entity NOT loaded into DB — yet resolves successfully
  await expect(EntityService.getEncodedId(defaultInputEntity[0].num + ''))
    .resolves.toBe(defaultExpectedEntity.id);
});
```
The test is explicitly named "No match" and the entity is never inserted, yet the call succeeds. This is the authoritative proof that no existence or deletion check is performed for numeric inputs.

### Impact Explanation
Every REST API controller that calls `entityService.getEncodedId()` (`accountController.js`, `contractController.js`, `cryptoAllowanceController.js`, `tokenAllowanceController.js`, `tokenController.js`, `balances.js`, `tokens.js`, `topicmessage.js`, `transactions.js`, `schedules.js`) receives an encoded ID for a deleted entity without any warning. If those downstream handlers do not independently re-check `deleted` (and they are designed to trust `getEncodedId()` as the validation layer), they will query and return records for deleted entities. This breaks the invariant that deleted entities are hidden from the API, leaking data that should be inaccessible.

### Likelihood Explanation
No privileges are required. The attacker only needs to know (or enumerate) the numeric encoded ID of a deleted entity—a trivially guessable integer. The attack is stateless, repeatable, and requires a single HTTP request. The `encodedEntityIdRegex` accepts any 1–19 digit integer, so the attack surface is the entire numeric ID space.

### Recommendation
Add a database existence-and-deletion check for the numeric/`shard.realm.num` branch, consistent with the other two branches. Introduce a query analogous to `entityExistenceQuery` but with a `deleted` filter:

```sql
SELECT type FROM entity WHERE id = $1 AND coalesce(deleted, false) <> true
```

Call it inside `getEncodedId()` when `evmAddress === null`, and throw `NotFoundError` (when `requireResult` is true) or return `null` if the entity is absent or deleted. Alternatively, consolidate all three branches to always perform a DB lookup so the deletion invariant is uniformly enforced.
