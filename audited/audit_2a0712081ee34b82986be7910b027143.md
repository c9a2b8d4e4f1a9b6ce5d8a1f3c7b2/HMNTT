### Title
Degenerate Equal-Range on `spender.id` + `token.id` Produces Duplicate Rows via UNION ALL in `getAccountTokenAllowances`

### Summary
When an unprivileged caller supplies `spender.id=gte:X&spender.id=lte:X&token.id=gte:Y&token.id=lte:Y` (identical lower and upper bounds for both parameters), the multi-union query builder in `baseController.js` generates three subqueries whose lower and upper parts have overlapping `WHERE` clauses. Because the subqueries are joined with `UNION ALL` (no deduplication), the row `(spender=X, token_id=Y)` is returned twice in the same response. No privilege is required and the request passes all existing validation.

### Finding Description

**Parsing — `rest/controllers/bound.js` lines 48–68**

`spender.id=gte:X` sets `primary.lower = {operator: gte, value: X}` and `spender.id=lte:X` sets `primary.upper = {operator: lte, value: X}`. Likewise `token.id=gte:Y` / `token.id=lte:Y` set `secondary.lower` / `secondary.upper`. No check is made that `lower.value != upper.value`. [1](#0-0) 

**Validation — `rest/controllers/baseController.js` lines 56–123**

All four guards pass for this input:
- `validateBoundsRange`: primary has a bound but no equal → no throw.
- `validateSecondaryBound`: primary is non-empty → no throw.
- `validateLowerBounds`: `primary.lower.operator` is `gte` (not `gt`) → no throw.
- `validateUpperBounds`: `primary.upper.operator` is `lte` (not `lt`) → no throw. [2](#0-1) 

**Filter generation — `rest/controllers/baseController.js` lines 131–183**

With both primary and secondary having bounds, and `primary.hasLower() && secondary.hasLower()` being true:

- `getLowerFilters` → `[{spender = X}, {token_id >= Y}]`
- `getInnerFilters` → `[{spender > X}, {spender < X}]` ← impossible, always 0 rows
- `getUpperFilters` → `[{spender = X}, {token_id <= Y}]` [3](#0-2) [4](#0-3) [5](#0-4) 

**Query assembly — `rest/service/tokenAllowanceService.js` lines 63–74**

All three non-empty filter arrays produce three subqueries joined with `UNION ALL`:

```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ...)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ...)  -- always empty
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ...)
ORDER BY spender ASC, token_id ASC LIMIT $2
```

params: `[ownerAccountId, limit, X, Y, X, X, X, Y]`

The row `(spender=X, token_id=Y)` satisfies `token_id >= Y` (subquery 1) **and** `token_id <= Y` (subquery 3). `UNION ALL` does not deduplicate, so it appears twice. [6](#0-5) 

No deduplication is applied in the controller before the response is sent: [7](#0-6) 

### Impact Explanation
The REST endpoint returns a JSON array containing the same allowance entry twice. Any consumer (smart contract, dApp, or off-chain service) that iterates the `allowances` array and counts entries or sums `amount_granted` values will compute an inflated figure — double the actual granted amount for the targeted `(spender, token_id)` pair. In the context of the Hedera mirror node, this is the authoritative read path for token allowance state; incorrect counts can cause smart contracts to believe a spender has more allowance than actually exists, or to approve/reject operations based on a wrong tally. No funds are directly drained, but the integrity of allowance accounting is broken for the queried account.

### Likelihood Explanation
The attack requires zero privileges — any HTTP client can craft the request. The parameters are standard query-string values accepted by the public API. The condition is trivially reproducible: supply identical numeric values for `gte` and `lte` on both `spender.id` and `token.id`. The existing test suite covers the case `spender gte and lte, token gte and lte` with *different* values and confirms the three-subquery path is taken, but no test covers the degenerate equal-value case. [8](#0-7) 

### Recommendation
Add a degenerate-range check in `validateBounds` (or inside `Bound.parse` / `validateBoundsRange`) that throws `InvalidArgumentError` when `primary.lower.value === primary.upper.value` (and likewise for secondary). Alternatively, when the lower and upper values are equal, collapse the range into a single `eq` filter before building the subqueries, which would route the request through the single-subquery path and eliminate the overlap. A targeted fix in `getUpperFilters` / `getLowerFilters` to use strict inequalities (`>` / `<`) at the boundary when both bounds share the same value would also prevent the overlap.

### Proof of Concept

**Precondition**: Account `0.0.1001` has a token allowance row with `spender=0.0.2001` and `token_id=0.0.10001`.

**Request**:
```
GET /api/v1/accounts/1001/allowances/tokens?spender.id=gte:2001&spender.id=lte:2001&token.id=gte:10001&token.id=lte:10001
```

**Generated SQL** (simplified):
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ORDER BY spender ASC, token_id ASC LIMIT $2)
ORDER BY spender ASC, token_id ASC LIMIT $2
-- params: [1001, 25, 2001, 10001, 2001, 2001, 2001, 10001]
```

**Result**: The `allowances` array in the JSON response contains the entry for `(spender=0.0.2001, token_id=0.0.10001)` **twice**. A smart contract or application summing `amount_granted` across the returned entries will compute `2 × actual_amount` for that allowance.
