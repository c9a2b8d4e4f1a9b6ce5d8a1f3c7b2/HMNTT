### Title
`parseInt` Scientific-Notation Truncation Corrupts `api_request_size_bytes` Histogram in `metricsMiddleware`

### Summary
In `metricsMiddleware`, the `content-length` header is parsed with `parseInt(..., 10)`, which silently truncates scientific-notation strings at the decimal point. An unprivileged attacker can send `Content-Length: 1.7976931348623157e+308`, causing `parseInt` to return `1` instead of the astronomically large value, recording a near-zero observation in the `api_request_size_bytes` histogram and permanently skewing its statistical output.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback of `metricsMiddleware`.

```js
// line 209
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** `parseInt(string, 10)` parses only leading valid base-10 characters and stops at the first non-digit. For the string `'1.7976931348623157e+308'`, it reads `'1'`, then stops at `'.'`, returning `1`. The `|| 0` guard only catches `NaN` (i.e., completely non-numeric strings); it does not catch the case where `parseInt` returns a small but truthy integer derived from a malformed large value.

**Exploit flow:**
1. Attacker sends any HTTP request to any instrumented route with the header `Content-Length: 1.7976931348623157e+308`.
2. Express/Node.js does not reject this header at the framework level — it is passed through as a raw string in `req.headers['content-length']`.
3. `parseInt('1.7976931348623157e+308', 10)` returns `1`.
4. `requestSizeHistogram.record(1, labels)` is called, recording a 1-byte observation.
5. Repeated at scale, the histogram's sum, mean, and bucket distributions are corrupted with artificially low values.

**Why existing checks fail:** The `?? '0'` nullish-coalescing guard handles a missing header. The `|| 0` guard handles `NaN`. Neither handles the case where `parseInt` returns a small positive integer from a truncated scientific-notation string.

### Impact Explanation
The `api_request_size_bytes` histogram is used for capacity planning, anomaly detection, and SLA monitoring. Injecting a flood of `1`-byte observations while claiming enormous bodies causes the histogram's statistical outputs (mean, percentiles, bucket counts) to diverge from reality. Operators relying on this metric for alerting or autoscaling decisions receive misleading data. No user data is read or modified; the impact is confined to observability infrastructure (griefing / metric poisoning).

### Likelihood Explanation
No authentication, special role, or network position is required. Any client that can reach the REST API can send an arbitrary `Content-Length` header. The attack is trivially scriptable, repeatable at high frequency, and leaves no obvious application-layer error (the request is processed normally; only the metric is wrong). Detection requires cross-correlating actual body sizes against histogram values, which is non-trivial.

### Recommendation
Replace `parseInt` with a safe numeric conversion that handles scientific notation correctly and rejects non-integer values:

```js
// Option 1: use Number() and validate range
const rawCL = req.headers['content-length'];
const contentLength = rawCL !== undefined ? Math.max(0, Math.trunc(Number(rawCL))) || 0 : 0;
requestSizeHistogram.record(Number.isFinite(contentLength) ? contentLength : 0, labels);

// Option 2: reject non-integer strings entirely
const rawCL = req.headers['content-length'];
const contentLength = /^\d+$/.test(rawCL ?? '') ? parseInt(rawCL, 10) : 0;
requestSizeHistogram.record(contentLength, labels);
```

Option 2 (strict integer regex) is preferred: it matches the HTTP/1.1 RFC 7230 requirement that `Content-Length` be a non-negative decimal integer, and silently drops any malformed value.
