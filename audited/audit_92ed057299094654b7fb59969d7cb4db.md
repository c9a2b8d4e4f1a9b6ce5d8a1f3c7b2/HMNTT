### Title
Log Injection via Unsanitized `X-Forwarded-For` / `X-Real-IP` Headers in `TracingMiddleware`

### Summary
`getClientIpAddress()` reads the `X-Forwarded-For` or `X-Real-IP` header verbatim and returns it without any sanitization. The value is embedded directly into a log message string via `fmt.Sprintf` and written to the log stream via `log.Info()`. Because logrus `TextFormatter` does not escape embedded newlines in message strings, an unauthenticated external attacker can inject arbitrary fake log lines into the log stream, enabling SIEM/monitoring evasion that allows real attacks to go undetected.

### Finding Description
**Code path:**

- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–75: reads `X-Real-IP` then `X-Forwarded-For` with `r.Header.Get(...)` and returns the raw string with zero validation or sanitization.
- `TracingMiddleware`, lines 52–59: embeds the raw return value into `message` via `fmt.Sprintf("%s %s %s (%d) in %s", clientIpAddress, ...)` — `%s` does not escape newlines — then calls `log.Info(message)`.
- `rosetta/main.go`, lines 43–52: logrus is configured with `log.TextFormatter{DisableColors: true, FullTimestamp: true}`. In text/logfmt mode, logrus does **not** escape newline characters embedded in the `msg` field; they are written literally to stdout.

**Root cause:** The code assumes the `X-Forwarded-For` / `X-Real-IP` header contains a valid IP address string. It never validates format, strips control characters, or escapes the value before logging. The failed assumption is that HTTP header values are safe for direct inclusion in structured log output.

**Exploit flow:**
1. Attacker sends any HTTP request (no authentication required) to the Rosetta API with a crafted header:
   ```
   X-Forwarded-For: 1.2.3.4\ntime="2025-01-01T00:00:00Z" level=info msg="GET /network/status (200) in 1ms"
   ```
2. `getClientIpAddress()` returns the string containing the embedded `\n`.
3. `fmt.Sprintf` produces a message string with a literal newline mid-string.
4. `log.Info(message)` writes it to stdout; the embedded newline causes everything after it to appear as a separate, syntactically valid log line.
5. The forged line is indistinguishable from a real logrus entry in the log stream.

**Why existing checks fail:** There are none. The test file (`trace_test.go`, lines 52–58) only tests well-formed IP values like `"10.0.0.100"` — no test exercises a newline-containing header. No IP format validation, no `strings.ReplaceAll`, no `regexp` sanitization, no logrus field-based logging (which would have escaped the value) is present anywhere in the path.

### Impact Explanation
An attacker can forge arbitrary log entries at will — including fake successful requests, fake health-check passes, or fake error suppressions — causing SIEM/alerting systems to build incorrect baselines or suppress real alerts. If an attacker is conducting a coordinated attack to degrade or shut down mirror-node processing nodes (e.g., resource exhaustion, malformed transaction flooding), log injection allows them to mask the attack traffic in the log stream, preventing operators and automated monitoring from detecting and responding to the attack before it reaches the threshold of shutting down 30%+ of network processing nodes.

### Likelihood Explanation
The Rosetta API is a public-facing HTTP endpoint requiring no authentication for data queries. Any external user can send a crafted `X-Forwarded-For` header. The attack requires only a single HTTP request with a crafted header — no special tooling, no credentials, no prior access. It is trivially repeatable and automatable. The only prerequisite is network reachability to the Rosetta port (default 5700).

### Recommendation
1. **Validate the IP address format** in `getClientIpAddress()` using `net.ParseIP()` and return only the validated IP string (or a safe fallback like `"unknown"`).
2. **Strip control characters** (at minimum `\n`, `\r`, `\t`) from any header-derived value before logging.
3. **Use structured logrus fields** instead of a pre-formatted string: `log.WithFields(log.Fields{"client_ip": clientIpAddress, "method": ..., "path": ...}).Info("request")` — logrus will then quote/escape field values containing special characters.

### Proof of Concept
```bash
# Target: Rosetta API on localhost:5700
# Inject a fake log line via X-Forwarded-For

curl -s http://localhost:5700/network/list \
  -H 'Content-Type: application/json' \
  -H $'X-Forwarded-For: 1.2.3.4\ntime="2025-01-01T00:00:00Z" level=info msg="GET /network/status (200) in 500\xc2\xb5s"' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
```

**Expected log output (two lines, second is forged):**
```
time="2025-01-01T00:00:00Z" level=info msg="1.2.3.4
time="2025-01-01T00:00:00Z" level=info msg="GET /network/status (200) in 500µs"
```

The second line is entirely attacker-controlled and is syntactically valid logfmt, indistinguishable from a real entry in automated log processing pipelines. [1](#0-0) [2](#0-1) [3](#0-2)

func getClientIpAddress(r *http.Request) string {
	ipAddress := r.Header.Get(xRealIpHeader)

	if len(ipAddress) == 0 {
		ipAddress = r.Header.Get(xForwardedForHeader)
	}

	if len(ipAddress) == 0 {
		ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
	}

	return ipAddress
}
