### Title
Missing Length Constraint on Base32 Alias Input Enables Memory Exhaustion via `getByAliasAndType()`

### Summary
The `alias` field in the GraphQL `AccountInput` type is validated only by a character-set `@Pattern` directive, with no upper bound on string length. An unauthenticated attacker can submit an arbitrarily long valid Base32 string that passes the pattern check, causing `decodeBase32()` to allocate a proportionally large byte array in the JVM heap, which is then forwarded as a parameter to the native SQL query `findByAlias()`. Repeated requests can exhaust JVM heap memory and degrade or crash the service.

### Finding Description
**Code path:**

1. `graphql/src/main/resources/graphql/account.graphqls`, line 90:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   The only validation is a character-set regex. There is no `@Size` or equivalent length constraint.

2. `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, line 48:
   ```java
   return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
   ```
   The raw (potentially enormous) string is passed directly to the service.

3. `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 30:
   ```java
   return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   ```

4. `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 83–85:
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   No length check whatsoever. Apache Commons Codec `Base32.decode()` will allocate a byte array of approximately `(inputLength * 5) / 8` bytes.

5. `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 13–14:
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```
   The full byte array is serialized and sent to PostgreSQL as a query parameter.

**Root cause:** The `@Pattern` directive enforces only that characters belong to the Base32 alphabet `[A-Z2-7]`. A string of one million such characters is perfectly valid per this check. No `@Size(max=...)` or equivalent constraint exists anywhere in the input validation chain.

### Impact Explanation
An unauthenticated attacker sending repeated GraphQL queries with a multi-megabyte alias string (e.g., `"AAAA..."` × 1,000,000 characters → ~625 KB byte array per request) can:
- Force large heap allocations in the JVM on every request, leading to GC pressure and eventual `OutOfMemoryError`.
- Force the JDBC driver to serialize and transmit large byte arrays to PostgreSQL, consuming network and DB resources.
- Cause service unavailability (DoS) for all users of the GraphQL endpoint, including legitimate transaction entity resolution.

Severity: **High** — unauthenticated, remotely exploitable, repeatable, no rate-limiting or size guard in the code path.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required to query accounts by alias. The exploit requires only a standard HTTP client sending a crafted GraphQL query. The `@Pattern` check is trivially satisfied by any string of uppercase letters and digits 2–7. The attack is fully repeatable and can be automated.

### Recommendation
1. Add a `@Size` (or equivalent) constraint on the `alias` field in the GraphQL schema, e.g.:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max: 128)
   ```
   (Hedera account aliases are at most a serialized Ed25519/ECDSA public key, which is at most ~44 Base32 characters for a 32-byte key, or ~56 for a 35-byte ECDSA key.)

2. Add an explicit length guard in `decodeBase32()` or at the start of `getByAliasAndType()`:
   ```java
   if (alias != null && alias.length() > 128) {
       throw new IllegalArgumentException("Alias exceeds maximum allowed length");
   }
   ```

3. Consider adding a global GraphQL request size limit (e.g., via Spring Boot's `spring.graphql.websocket.connection-init-timeout` or a custom `WebGraphQlInterceptor`) to bound total request payload size.

### Proof of Concept
```bash
# Generate a 1,000,000-character valid Base32 alias (all 'A's pass ^[A-Z2-7]+$)
ALIAS=$(python3 -c "print('A' * 1000000)")

# Send the GraphQL query
curl -s -X POST http://<mirror-node-graphql-host>/graphql \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"
```

Repeat this request in a loop from multiple clients. Each request forces `decodeBase32()` to allocate a ~625 KB byte array and the JDBC driver to transmit it to PostgreSQL. Under sustained load, JVM heap exhaustion and service degradation will occur. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
