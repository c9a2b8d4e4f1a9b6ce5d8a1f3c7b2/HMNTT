### Title
Unbounded Subscription Creation in `PollingTopicListener.listen()` Enables Resource Exhaustion via Subscription Flood

### Summary
`PollingTopicListener.listen()` creates a new indefinitely-polling `Flux` per call with no per-client, per-channel, or global subscription cap. The only tracking mechanism, `subscriberCount` in `TopicMessageServiceImpl`, is wired exclusively to a metrics gauge and is never enforced as a limit. An unprivileged attacker can open arbitrarily many gRPC streams, each spawning a long-lived polling loop that issues a database query every 500 ms, exhausting the DB connection pool, the `boundedElastic` scheduler queue, and JVM heap.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–48) accepts every inbound gRPC stream with no admission control and immediately calls `topicMessageService.subscribeTopic()`. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (lines 88–91) increments `subscriberCount` on subscribe and decrements it on `doFinally`, but the counter is registered only as a Micrometer `Gauge` (lines 52–55) — it is never compared against a maximum. [2](#0-1) [3](#0-2) 

`incomingMessages()` (line 120) calls `topicListener.listen(newFilter)` for every active subscription. [4](#0-3) 

`PollingTopicListener.listen()` (lines 34–49) allocates a fresh `PollingContext` and schedules `RepeatSpec.times(Long.MAX_VALUE)` on `Schedulers.boundedElastic()` — an infinite polling loop that fires a DB query every `interval` (default 500 ms) for the lifetime of the subscription. [5](#0-4) 

**Root cause:** The failed assumption is that gRPC transport-level flow control or OS-level TCP limits will bound the number of concurrent subscriptions. In practice, HTTP/2 multiplexes many streams over a single TCP connection, and nothing in the application layer caps how many `listen()` invocations can be active simultaneously.

**Why the cancel handler is insufficient:** `setOnCancelHandler(disposable::dispose)` (line 51–53 of `ConsensusController`) only fires on a well-formed gRPC `CANCEL` frame. An attacker who simply drops the TCP connection or holds it open indefinitely bypasses this path; the server-side subscription remains alive until the transport layer detects the dead connection, which can take minutes under default TCP keepalive settings. [6](#0-5) 

### Impact Explanation

Each active subscription issues one `topicMessageRepository.findByFilter()` call every 500 ms. [7](#0-6) 

With N concurrent subscriptions the server executes 2N DB queries per second (one from the listener, one from the historical retriever path). A modest flood of 200 subscriptions saturates a typical JDBC pool (default size ~10–20), causing all legitimate queries to queue or time out. Concurrently, `Schedulers.boundedElastic()` has a default task-queue cap of 100,000; filling it causes `RejectedExecutionException` for all new scheduled work server-wide. Each `PollingContext` also retains heap references for the duration of the subscription, enabling gradual memory exhaustion. The net effect is full denial of service for all legitimate subscribers.

### Likelihood Explanation

No authentication or special privilege is required — the gRPC endpoint is publicly reachable. A single attacker machine can open hundreds of HTTP/2 streams over a small number of TCP connections using any standard gRPC client library. Subscriptions with no `endTime` and no `limit` never self-terminate, so the attacker need only open streams and hold them open. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Enforce a global subscription cap**: Compare `subscriberCount` against a configurable maximum inside `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when the limit is exceeded.
2. **Add per-IP / per-channel subscription limits**: Implement a gRPC `ServerInterceptor` that tracks active stream counts per remote address and rejects new streams above a threshold.
3. **Set aggressive gRPC keepalive and max-connection-age**: Configure `GRPC_ARG_MAX_CONNECTION_AGE_MS` and `GRPC_ARG_KEEPALIVE_TIMEOUT_MS` so dead connections are detected and cleaned up quickly, shrinking the window for subscription leaks.
4. **Expose `subscriberCount` as an enforcement gate**, not just a metric, by adding a `maxSubscribers` property to `ListenerProperties` or `GrpcProperties`.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import timestamp_pb2, consensus_topic_id_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

def open_subscription():
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=consensus_topic_id_pb2.ConsensusTopicID(topicNum=1),
        # No consensusEndTime → subscription never self-terminates
        # No limit → subscription never self-terminates
    )
    for _ in stub.subscribeTopic(req):
        pass  # consume nothing; keep stream open

threads = [threading.Thread(target=open_subscription) for _ in range(500)]
for t in threads:
    t.start()
# 500 concurrent subscriptions → 1000 DB queries/sec, scheduler queue fills,
# legitimate subscribers receive UNAVAILABLE or timeout errors.
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
