### Title
Incomplete Pagination in `fetchMirrorNodeNetwork()` Enables MITM Node-Set Truncation

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` parses the `Links.Next` pagination field from the mirror node REST response but never follows it. A network-adjacent attacker who can intercept the unencrypted HTTP connection (the documented default is `http://mirror-rest:5551`) can serve a crafted first-page response containing only attacker-chosen nodes plus a plausible `Links.Next` pointer. The pinger accepts this truncated map as complete, and all subsequent transactions are routed exclusively to the attacker-selected consensus nodes for the lifetime of the process.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–132.

The `nodesEnvelope` struct (lines 17–22) declares `Links.Next *string` and it is populated by `json.NewDecoder(resp.Body).Decode(&payload)` at line 96. After decoding, the function iterates only over `payload.Nodes` (lines 102–125) and returns at line 131 — `payload.Links.Next` is read into memory and then silently discarded. There is no loop, no recursive call, and no check for a non-nil `Next` pointer.

**Root cause / failed assumption:** The implementation assumes the mirror node will always return the complete node set in a single page. The REST API is a paginated endpoint; real deployments with many nodes will naturally split results across pages, and an attacker can force this split artificially.

**Exploit flow:**
1. Attacker positions themselves to intercept HTTP traffic between the pinger pod and the mirror REST service (same Kubernetes namespace, ARP/DNS poisoning, rogue sidecar, etc.). The documented default URL `http://mirror-rest:5551` (config.go line 37) uses plain HTTP with no TLS.
2. Attacker intercepts `GET /api/v1/network/nodes`.
3. Attacker replies with a valid JSON body containing only the nodes they wish to bias toward (e.g., a single node `0.0.3`) and a syntactically correct `"links": {"next": "/api/v1/network/nodes?limit=25&node.id=gt:0.0.3"}`.
4. `fetchMirrorNodeNetwork()` decodes the response, finds `len(network) == 1 > 0`, passes the guard at line 127, and returns the one-entry map with `retry=false, err=nil`.
5. `buildNetworkFromMirrorNodes()` (line 54) accepts this as success and returns it.
6. `hiero.ClientForNetwork(netmap)` (sdk_client.go line 22) builds the SDK client with only that one node.
7. Every `submitWithRetry` call for the entire process lifetime routes all transactions to that single node.

**Why existing checks are insufficient:**
- The only guard is `len(network) == 0` (line 127–129). An attacker trivially bypasses this by including at least one valid node entry.
- There is no TLS enforcement, no certificate pinning, and no response integrity check.
- The retry loop in `buildNetworkFromMirrorNodes()` (lines 52–69) only retries on HTTP errors or 5xx/429 status codes — a 200 OK with a truncated body is accepted unconditionally.

### Impact Explanation
All pinger-originated transactions are concentrated on attacker-chosen consensus nodes. This distorts the transaction distribution recorded in the mirror node history, which is the exact data the pinger is designed to exercise. An attacker can make specific nodes appear to handle disproportionate load, suppress activity on other nodes, or route all traffic through a node they monitor. While no funds are directly stolen (the operator key is not exposed), the integrity of the mirror node's transaction history is compromised for the duration of the pinger's uptime. Severity: **Medium** — meaningful protocol-level manipulation without direct financial loss.

### Likelihood Explanation
The attack requires a network-adjacent position, which is realistic in containerized deployments where the mirror REST service is accessed over plain HTTP within a cluster. Kubernetes namespace co-tenancy, compromised ingress/service-mesh components, or DNS spoofing within the cluster are all practical vectors. The attack is repeatable: it only needs to succeed once at startup (or after a pinger restart) since the node map is built once and never refreshed. No credentials or application-level privileges are required.

### Recommendation
1. **Follow pagination:** After decoding each page, check `payload.Links.Next`; if non-nil and non-empty, construct the next absolute URL and loop until `Next` is nil. This is the primary fix.
2. **Enforce HTTPS:** Reject `mirrorRest` URLs that do not use the `https` scheme, or configure the `http.Client` with a strict TLS policy. The documented default `http://mirror-rest:5551` should be changed to `https://`.
3. **Minimum node count:** After collecting all pages, reject the result if the number of distinct nodes is below a configurable threshold (e.g., `minNodes`), making single-node truncation attacks fail the guard.
4. **Periodic refresh:** Rebuild the network map on a schedule so a one-time MITM attack does not persist indefinitely.

# 1. Stand up a fake mirror REST server that returns a truncated node list:
cat > fake_mirror.py << 'EOF'
from http.server import HTTPServer, BaseHTTPRequestHandler
import json

class H(BaseHTTPRequestHandler):
    def do_GET(self):
        body = json.dumps({
            "nodes": [{
                "node_account_id": "0.0.3",
                "service_endpoints": [{"ip_address_v4": "34.94.106.61", "port": 50211}],
                "grpc_proxy_endpoint": None
            }],
            "links": {"next": "/api/v1/network/nodes?limit=25&node.id=gt:0.0.3"}
        }).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 5551), H).serve_forever()
EOF
python3 fake_mirror.py &

# 2. Run the pinger pointed at the fake server:
HIERO_MIRROR_PINGER_NETWORK=other \
HIERO_MIRROR_PINGER_REST=http://127.0.0.1:5551 \
HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2 \
HIERO_MIRROR_PINGER_OPERATOR_KEY=<key> \
HIERO_MIRROR_PINGER_TO_ACCOUNT_ID=0.0.98 \
  ./pinger

# 3. Observe: pinger starts successfully (no error), and all transactions
#    are submitted exclusively to node 0.0.3 (34.94.106.61:50211),
#    even though the real network has many more nodes.
#    The real nodes (0.0.4, 0.0.5, ...) receive zero transactions.
