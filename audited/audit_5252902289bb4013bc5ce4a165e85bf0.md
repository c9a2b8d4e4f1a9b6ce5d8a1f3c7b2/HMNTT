### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent NFT Transfer History Requests

### Summary
`getNftTransferHistoryRequest()` in `rest/tokens.js` issues three sequential `pool.queryQuietly()` calls per request with no per-IP or per-user rate limiting enforced at the application layer. An unprivileged attacker can flood `GET /api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` with concurrent requests to exhaust the finite pg connection pool, causing legitimate requests to queue and time out.

### Finding Description
**Code path:** `rest/tokens.js`, `getNftTransferHistoryRequest()`, lines 997–1056.

Each invocation makes three sequential DB round-trips:
- Line 1022: `pool.queryQuietly(timestampQuery, timestampParams)` — timestamp lookup across `nft`, `nft_history`, `entity`
- Line 1032: `pool.queryQuietly(query, params)` — full transaction detail fetch
- Line 1047: `NftService.getNft(tokenId, serialNumber)` — NFT existence check

The pool is initialized in `rest/dbpool.js` (lines 7–16) with `max: config.db.pool.maxConnections` — a single finite integer. The `rest/server.js` middleware stack (lines 68–144) contains: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — **no rate-limiting middleware is present**. The `authHandler` enforces optional authentication but applies no request-rate cap to unauthenticated callers. The throttle/rate-limit code found in the codebase (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) belongs exclusively to the `web3` Java module and has no equivalent in the Node.js REST service.

**Root cause:** The REST API trusts that infrastructure-level controls (reverse proxy, pgbouncer) will absorb floods, but no such guarantee is enforced in code. The `statement_timeout` pool option bounds individual query duration but does not limit how many concurrent connections are held simultaneously.

**Exploit flow:**
1. Attacker identifies any valid `tokenId`/`serialNumber` pair (publicly enumerable via `/api/v1/tokens/{id}/nfts`).
2. Attacker sends N concurrent HTTP requests to `GET /api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` where N ≥ `maxConnections`.
3. Each request holds a pool connection for the duration of up to three sequential queries.
4. Pool slots are exhausted; subsequent legitimate requests block waiting for `connectionTimeoutMillis` ms, then fail with a connection timeout error.

### Impact Explanation
All REST API endpoints sharing the same `global.pool` instance are affected — not just the NFT endpoint. Legitimate users querying transaction history, account data, or any other endpoint backed by `pool.queryQuietly` will receive errors or timeouts for the duration of the attack. The mirror node's ability to serve gossiped transaction data is fully suppressed for all clients during pool exhaustion.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. The endpoint is publicly reachable. A single attacker with a modest number of concurrent HTTP connections (matching or exceeding `maxConnections`, defaulting to a small integer in typical pg pool configs) can trigger this. The attack is trivially repeatable and scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in parallel). The attacker does not need to know valid NFT data — invalid `tokenId`/`serialNumber` pairs still consume a pool connection for the first query before returning empty rows.

### Recommendation
1. **Add application-level rate limiting** to the REST server: integrate `express-rate-limit` (or equivalent) as middleware in `rest/server.js` before route handlers, keyed on `req.ip`, with a per-window cap on requests to `/tokens/:tokenId/nfts/:serialNumber/transactions`.
2. **Enforce per-IP connection limits** at the pgbouncer layer (`max_user_client_connections` is already set for `mirror_rest` in `values.yaml` but this is infrastructure-level and not enforced in code).
3. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) around `pool.queryQuietly` calls to cap the number of in-flight DB queries globally or per-endpoint.
4. Consider **response caching** for this endpoint (already supported by the cache middleware infrastructure) to serve repeated identical requests without hitting the DB.

  const tokenId = getAndValidateTokenIdRequestPathParam(req);
  const serialNumber = getAndValidateSerialNumberRequestPathParam(req);

  const filters = utils.buildAndValidateFilters(
    req.query,
    acceptedNftTransferHistoryParameters,
    validateTokenQueryFilter
  );

  const response = {
    transactions: [],
    links: {
      next: null,
    },
  };
  res.locals[responseDataLabel] = response;

  // Get transaction consensus timestamps
  const {
    query: timestampQuery,
    params: timestampParams,
    limit,
    order,
  } = extractSqlFromNftTransferHistoryRequest(tokenId, serialNumber, filters);
  const {rows} = await pool.queryQuietly(timestampQuery, timestampParams);
  if (rows.length === 0) {
    return;
  }

  // Get nft transfer related transaction details
  const jsonbPathQueryVars = utils.JSONStringify({token_id: tokenId, serial_number: serialNumber});
  const timestamps = rows.map((r) => r.timestamp);
  const query = `${nftTransactionHistoryDetailsQuery} ${order}`;
  const params = [tokenId, serialNumber, jsonbPathQueryVars, timestamps];
  const {rows: transactions} = await pool.queryQuietly(query, params);

  response.transactions = transactions.map(formatNftTransactionHistoryRow);
  const anchorTimestamp = last(response.transactions)?.consensus_timestamp ?? 0;
  response.links.next = utils.getPaginationLink(
    req,
    response.transactions.length !== limit,
    {
      [filterKeys.TIMESTAMP]: anchorTimestamp,
    },
    order
  );

  if (response.transactions.length > 0) {
    // check if nft exists
    const nft = await NftService.getNft(tokenId, serialNumber);
    if (nft?.createdTimestamp == null) {
      // set 206 partial response
      res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
      logger.debug(`getNftTransferHistory returning partial content`);
    }
  }

};

function parseDbPoolConfig() {
  const {pool} = getConfig().db;
  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
    }
    pool[configKey] = parsed;
  });
}
