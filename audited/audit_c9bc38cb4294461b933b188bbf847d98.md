### Title
Gas Token Non-Restoration via Dead Try-Catch in `constructServiceParameters()` Enables Gas Bucket Depletion DoS

### Summary
In `ContractController.call()`, the `try-catch` block in `constructServiceParameters()` wraps a simple getter (`request.getData()`) that can never throw, making it dead code. The actual `hexToBytes(data)` call at line 80 sits outside this catch and throws `IllegalArgumentException` — not `InvalidParametersException` — on malformed input. Since `restore()` is only called for `InvalidParametersException`, gas tokens consumed by `throttle()` are permanently lost for any such failure, enabling an unprivileged attacker to continuously deplete the `gasLimitBucket` and deny service to legitimate callers.

### Finding Description

**Code path:**

`ContractController.java`, `call()`, lines 38–51: [1](#0-0) 

The only restore path is the `catch (InvalidParametersException e)` block at line 46. Any other exception propagates without restoring tokens.

**Root cause — dead try-catch:**

`constructServiceParameters()`, lines 67–72: [2](#0-1) 

The `try` block wraps `request.getData()` — a plain Lombok getter that is incapable of throwing. The developer's intent (evidenced by the error message "contains invalid odd length characters") was to catch exceptions from `hexToBytes(data)`, but that call is at line 80, **outside** the try-catch: [3](#0-2) 

**`hexToBytes()` throws `IllegalArgumentException`, not `InvalidParametersException`:** [4](#0-3) 

`org.apache.commons.codec.binary.Hex.decodeHex()` raises `DecoderException` for odd-length hex strings; `hexToBytes()` re-wraps it as `IllegalArgumentException`. This is not caught in `call()`, so `restore()` is never invoked.

**`restore()` only refunds `gasLimitBucket`, never `rateLimitBucket`:** [5](#0-4) 

**`throttle()` consumes both buckets before `constructServiceParameters()` is reached:** [6](#0-5) 

**Exploit trigger — `@Hex` on `data` does not enforce even-length:**

The `data` field carries `@Hex` with no length constraints: [7](#0-6) 

The developer's own dead-catch error message ("odd length characters") is direct evidence that odd-length strings were a known concern that `@Hex` does not prevent. An odd-length hex string such as `0x123` passes `@Valid`, passes `throttle()`, passes `validateContractMaxGasLimit()`, then causes `hexToBytes()` to throw `IllegalArgumentException` — consuming gas tokens permanently.

### Impact Explanation

Each malicious request with `gas` near `maxGasLimit` consumes `scaleGas(gas)` tokens from `gasLimitBucket` with no restoration. With default settings (`gasPerSecond = 7,500,000,000`, `requestsPerSecond = 500`, `GAS_SCALE_FACTOR = 10,000`): [8](#0-7) 

- Bucket capacity: `scaleGas(7,500,000,000)` = 750,000 tokens
- Attacker rate: 500 req/s × `scaleGas(15,000,000)` = 500 × 1,500 = 750,000 tokens/s

The attacker can continuously drain the gas bucket at exactly the refill rate, holding it at zero and causing every legitimate contract call to receive `ThrottleException("Gas per second rate limit exceeded")`. This is a complete, sustained DoS on the `/api/v1/contracts/call` endpoint with no economic cost to the attacker.

### Likelihood Explanation

- **No privileges required**: any HTTP client can POST to `/api/v1/contracts/call`.
- **Trivially repeatable**: a single script sending `{"data":"0x123","gas":15000000}` in a loop at 500 req/s is sufficient.
- **No detection barrier**: the requests look syntactically valid and pass Spring validation; they only fail deep inside `constructServiceParameters()`.
- **Sustainable**: the attacker's own rate-limit tokens refill at 500/s, matching their send rate, so the attack can run indefinitely.

### Recommendation

1. **Move `hexToBytes()` inside the try-catch** (or wrap the entire `constructServiceParameters()` body):

```java
// ContractController.java – constructServiceParameters()
try {
    // ... address parsing ...
    return ContractExecutionParameters.builder()
        ...
        .callData(hexToBytes(data))   // ← move inside try
        ...
        .build();
} catch (InvalidParametersException e) {
    throw e;
} catch (Exception e) {
    throw new InvalidParametersException("data field contains invalid characters: " + e.getMessage());
}
```

2. **Restore both buckets** in `ThrottleManagerImpl.restore()` — currently only `gasLimitBucket` is refunded; `rateLimitBucket` should also be restored for validation failures: [5](#0-4) 

3. **Add even-length enforcement to `@Hex` validator** so odd-length strings are rejected at the `@Valid` boundary before any tokens are consumed.

### Proof of Concept

```bash
# Deplete gasLimitBucket with odd-length data that passes @Valid but fails hexToBytes()
while true; do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x123","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}' &
done
# After ~1 second, all legitimate callers receive:
# HTTP 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

**Preconditions**: no authentication, no special role, public endpoint.
**Trigger**: `data` field with odd-length hex (e.g., `0x123`) passes `@Hex` and `@Valid`, passes `throttle()` (consuming gas tokens), then `hexToBytes("0x123")` throws `IllegalArgumentException` outside the dead try-catch, bypassing `restore()`.
**Result**: gas tokens permanently consumed, bucket depleted, legitimate users throttled.

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
