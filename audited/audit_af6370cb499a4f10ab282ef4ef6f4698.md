### Title
Unauthenticated Metrics Endpoint Flood Causes Event-Loop Starvation via Unbounded Concurrent `exporter.collect()` Calls

### Summary
When `hiero.mirror.rest.metrics.config.authentication` is set to `false`, the `/swagger/metrics/` endpoint is fully open to unauthenticated callers. There is no rate limiting, no concurrency guard, and no request cap on this path. Every inbound HTTP request immediately invokes `exporter.collect()`, which schedules async observable callbacks that mutate shared closure state and queue microtasks on Node.js's single event loop. A flood of concurrent requests saturates the loop, starving legitimate fund-transfer query handlers.

### Finding Description
**File:** `rest/middleware/metricsHandler.js`
**Function:** `metricsMiddleware()` (returned by `metricsHandler()`)
**Lines:** 137–164

**Authentication bypass (lines 137–144):**
```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {   // ← returns true with zero credential check
    return true;
  }
  ...
};
```
When `authentication: false`, every caller is unconditionally admitted.

**Unbounded `exporter.collect()` (lines 156–164):**
```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) { ... }          // passes when auth disabled
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));
  });
}
```
No rate limit, no in-flight counter, no semaphore. Each request fires a new `exporter.collect()` Promise.

**Shared mutable state in observable callbacks (lines 108–123):**
```js
let previousCpuUsage = process.cpuUsage();
let previousHrTime   = process.hrtime.bigint();

meter.createObservableGauge('nodejs_process_cpu_usage_percentage', ...)
  .addCallback((result) => {
    const currentCpuUsage = process.cpuUsage();
    const currentHrTime   = process.hrtime.bigint();
    ...
    previousCpuUsage = currentCpuUsage;   // ← mutates shared closure
    previousHrTime   = currentHrTime;     // ← mutates shared closure
    result.observe(...);
  });
```
Every concurrent `collect()` call schedules this callback. Because Node.js is single-threaded but cooperative, many `collect()` Promises can be in flight simultaneously; their callbacks interleave across microtask ticks, each reading and overwriting `previousCpuUsage`/`previousHrTime` before the prior call has finished, corrupting CPU-usage deltas and generating continuous microtask pressure.

**No rate limiting anywhere on the metrics path** — confirmed by grep across `rest/**`: no `express-rate-limit`, `helmet`, or equivalent middleware is registered. `server.js` lines 88–92 register the metrics middleware globally with no guard.

### Impact Explanation
Node.js runs a single event loop. Thousands of concurrent `exporter.collect()` Promises each enqueue microtasks (observable callbacks + serialization). When the microtask queue is continuously replenished faster than it drains, I/O callbacks for legitimate API requests (account balance queries, transaction lookups, fund-transfer history) are deferred indefinitely. The REST API becomes unresponsive for the duration of the flood. Because this mirror node is the query layer for Hedera fund-transfer verification, sustained unavailability prevents clients from confirming or disputing transfers.

### Likelihood Explanation
Precondition — `authentication: false` — is a single-line config change that operators make for convenience in internal or staging deployments, and is explicitly documented as a supported option (`hiero.mirror.rest.metrics.config.authentication`, default `true`). Once disabled, the attack requires only an HTTP client capable of sending high-concurrency GET requests to a known, fixed path (`/swagger/metrics/`). No credentials, tokens, or special knowledge are needed. The attack is trivially repeatable and scriptable (e.g., `wrk`, `ab`, `hey`).

### Recommendation
1. **Add a concurrency gate** — track in-flight `collect()` calls with an atomic counter; return HTTP 429 if a collection is already in progress:
   ```js
   let collecting = false;
   if (collecting) return res.status(429).send('Too Many Requests');
   collecting = true;
   exporter.collect().then(...).finally(() => { collecting = false; });
   ```
2. **Add per-IP rate limiting** on the metrics path using `express-rate-limit` (already absent from `package.json` — add it).
3. **Never allow `authentication: false` in production** — enforce this in config validation (`config.js`) by throwing `InvalidConfigError` when `metrics.enabled && !metrics.config.authentication`.
4. **Protect shared observable state** — move `previousCpuUsage`/`previousHrTime` updates inside a guard that only runs when no other collection is active, or use a single scheduled interval instead of per-request callbacks.

### Proof of Concept
**Precondition:** Set `HIERO_MIRROR_REST_METRICS_CONFIG_AUTHENTICATION=false` and start the REST server.

```bash
# Install a load tool
npm install -g autocannon

# Flood the metrics endpoint with 500 concurrent connections for 30 seconds
autocannon -c 500 -d 30 http://localhost:5551/swagger/metrics/

# In a separate terminal, observe that legitimate API requests time out:
curl -w "\nTotal: %{time_total}s\n" http://localhost:5551/api/v1/transactions
# Expected: response time climbs to tens of seconds or connection refused
```

**Expected result:** During the flood, `api/v1/transactions` and other fund-transfer query endpoints become unresponsive because the Node.js event loop is saturated processing concurrent `exporter.collect()` microtasks. After the flood stops, service recovers — confirming the DoS is directly caused by the unbounded metrics collection path.
