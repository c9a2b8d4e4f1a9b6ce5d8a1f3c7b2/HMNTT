### Title
Global Shared Rate Limiter Without Per-Client Isolation Enables Single-Attacker DoS of `/api/v1/contracts/call`

### Summary
`ContractController.call()` delegates all throttling to a single process-wide `rateLimitBucket` (default 500 tokens/second) via `ThrottleManagerImpl`. Because no per-IP or per-client bucket exists anywhere in the application, a single unauthenticated attacker sustaining exactly 500 req/s permanently drains the shared bucket, causing every other caller to receive HTTP 429 for as long as the attack continues.

### Finding Description
**Exact code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` with no client identity information: [1](#0-0) 

`ThrottleManagerImpl.throttle()` (lines 37–42) checks only the single global `rateLimitBucket` — one token consumed per request, regardless of source: [2](#0-1) 

`ThrottleConfiguration.rateLimitBucket()` creates exactly one application-scoped `Bucket` with capacity = `requestsPerSecond` (default 500), refilled greedily at 500 tokens/second: [3](#0-2) [4](#0-3) 

**Root cause / failed assumption:** The design assumes the 500 RPS budget will be shared fairly across many clients. There is no mechanism to enforce that assumption. The `ThrottleManager` interface accepts only a `ContractCallRequest` — it has no access to `HttpServletRequest` or any client identity: [5](#0-4) 

**Why the `RequestProperties` filter system does not help:** `RequestProperties` can match on request body fields (`BLOCK`, `GAS`, `TO`, `VALUE`, `DATA`) but has no `FilterField` for source IP. It cannot be configured to apply per-client sub-limits: [6](#0-5) 

**Why infrastructure-level config does not close the gap:** `charts/hedera-mirror-web3/values.yaml` sets `maxRatePerEndpoint: 250` at the GCP gateway, which is a per-pod throughput cap, not a per-source-IP limit. `sessionAffinity: CLIENT_IP` is a load-balancing stickiness setting, not a rate-limiting control: [7](#0-6) 

### Impact Explanation
A sustained 500 req/s flood from a single source exhausts the shared token bucket. Every concurrent legitimate caller receives `HTTP 429 Too Many Requests` for the entire duration of the attack. The mirror node's EVM simulation API (`/api/v1/contracts/call`) becomes completely unavailable to all other users — dApps, wallets, and tooling that rely on `eth_call` / `eth_estimateGas` equivalents are fully blocked. The attack requires no authentication, no special payload, and no amplification.

### Likelihood Explanation
The attack is trivially reproducible with any HTTP load tool (`wrk`, `hey`, `ab`, a simple Python `asyncio` loop). The attacker needs only a single machine with sufficient outbound bandwidth to sustain 500 small POST requests per second — well within reach of any commodity VPS. The attack is repeatable indefinitely and self-sustaining: as long as the attacker matches the refill rate, the bucket never recovers for other callers.

### Recommendation
1. **Per-IP token buckets:** Introduce a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` / `X-Real-IP` / `RemoteAddr`) inside `ThrottleManagerImpl`, allocating each IP a sub-bucket (e.g., 10–50 req/s). The global bucket remains as a hard ceiling.
2. **Pass `HttpServletRequest` into `ThrottleManager`:** Extend the `ThrottleManager.throttle()` signature to accept the servlet request so the IP is available for per-client accounting.
3. **Infrastructure layer:** Configure a per-source-IP rate limit at the GCP Cloud Armor / Ingress level as a defence-in-depth measure independent of application logic.

```bash
# Minimal valid payload
cat > payload.json <<'EOF'
{"to":"0x0000000000000000000000000000000000000001","data":"0x","gas":21000,"estimate":false}
EOF

# Attacker terminal: saturate the shared bucket at exactly 500 req/s
wrk -t4 -c50 -d60s -s <(cat <<'LUA'
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = io.open("payload.json"):read("*a")
LUA
) http://<mirror-node-host>/api/v1/contracts/call

# Victim terminal (concurrent, different source IP or same host):
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","data":"0x","gas":21000,"estimate":false}'
# Expected result while attack is running: 429
```
