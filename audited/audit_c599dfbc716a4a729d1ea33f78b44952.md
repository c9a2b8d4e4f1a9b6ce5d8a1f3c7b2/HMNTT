### Title
Unauthenticated EVM Address Lookup Causes Unbounded Database Queries Leading to Service Degradation

### Summary
Any unprivileged external user can supply a syntactically valid but non-existent EVM address as `ownerId` in a `getHooks` request. This causes `EntityServiceImpl.lookup()` to unconditionally issue a `findByEvmAddress()` database query on every request, receive no result, and throw `EntityNotFoundException`. With no rate limiting or negative-result caching at the service layer, an attacker can trivially flood the database with useless queries, degrading service for legitimate users.

### Finding Description
**Code path:**

- `HookServiceImpl.getHooks()` ( [1](#0-0) ) unconditionally calls `entityService.lookup(request.getOwnerId())` before any other guard.
- `EntityServiceImpl.lookup()` performs a switch on the parameter type: [2](#0-1) 
  - For `EntityIdEvmAddressParameter`, it calls `entityRepository.findByEvmAddress(p.evmAddress())` — a live DB query — and if the result is empty, throws `EntityNotFoundException`.
- `EntityIdEvmAddressParameter` only validates the **format** of the supplied address via regex: [3](#0-2) 
  - It does **not** validate existence. Any 40-hex-character string passes.

**Root cause:** The failed assumption is that callers will supply EVM addresses that exist in the database. There is no rate limiting, no negative-result caching, and no authentication requirement visible in the service layer. Every request with a non-existent EVM address unconditionally hits the database.

### Impact Explanation
An attacker sending a high volume of requests with random (but format-valid) EVM addresses causes a proportional number of `findByEvmAddress()` DB queries, all returning empty. This wastes DB connection pool slots, CPU, and I/O, degrading response times for legitimate users. The EVM address space (2¹⁶⁰) is effectively infinite, so the attacker never exhausts unique inputs and cannot be blocked by deduplication alone.

### Likelihood Explanation
Exploitation requires zero privileges — only the ability to send HTTP requests. The attack is trivially scriptable: generate random 40-hex strings, prefix with `0x`, and send in a loop. No special knowledge of the system is needed. The attack is repeatable indefinitely and can be parallelized across multiple source IPs to bypass simple IP-based rate limiting at the network edge.

### Recommendation
1. **Negative-result caching:** Cache `EntityNotFoundException` results for EVM address lookups (e.g., via a short-TTL cache like Caffeine/Redis) so repeated queries for the same non-existent address do not hit the database.
2. **Application-level rate limiting:** Apply per-IP or per-client rate limiting at the controller/filter layer for endpoints that trigger entity lookups.
3. **Fail-fast on format:** If the EVM address does not match a known prefix pattern for the configured shard/realm, reject early before issuing any DB query.
4. **Circuit breaker:** Apply a circuit breaker around `entityRepository.findByEvmAddress()` to shed load under sustained abuse.

### Proof of Concept
```bash
# Generate and send 10,000 requests with random non-existent EVM addresses
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/hooks?ownerId=0x${ADDR}" &
done
wait
```

Each request causes `EntityServiceImpl.lookup()` to call `entityRepository.findByEvmAddress()` against the database, receive an empty result, and throw `EntityNotFoundException` — with no caching or rate limiting preventing the flood. [4](#0-3)
