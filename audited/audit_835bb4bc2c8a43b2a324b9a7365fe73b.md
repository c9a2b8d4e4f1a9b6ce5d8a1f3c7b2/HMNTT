### Title
HTTP/2 Multiplexing Exhausts HikariCP Connection Pool via Unauthenticated `GET /api/v1/topics/{id}` — No Rate Limiting in rest-java

### Summary
The `rest-java` module exposes `GET /api/v1/topics/{id}` with no per-IP rate limiting or concurrent-request throttling. An unauthenticated attacker can open a single HTTP/2 connection and send hundreds of concurrent streams, each triggering up to three sequential `CrudRepository.findById()` calls against the database. With HikariCP's default pool size of 10 connections and no application-level guard, the pool is exhausted, causing cascading request timeouts and degrading mirror node availability.

### Finding Description

**Code path:**

`TopicController.getTopic()` (lines 31–37) is the entry point:

```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());       // DB query 1
    var entity = entityService.findById(id.id());     // DB query 2
    var customFee = customFeeService.findById(id.id()); // DB query 3
    return topicMapper.map(customFee, entity, topic);
}
``` [1](#0-0) 

`TopicRepository` is a bare `CrudRepository` with no custom logic — `findById()` is the inherited Spring Data method: [2](#0-1) 

**Root cause — no rate limiting in rest-java:**

The `rest-java` config package contains only `JacksonConfiguration`, `LoggingFilter`, `MetricsFilter`, `MetricsConfiguration`, `WebMvcConfiguration`, and `RestJavaConfiguration` — no throttle filter, no per-IP limiter, no concurrent-request guard.


The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) exists exclusively in the `web3` module and is not shared with `rest-java`: [3](#0-2) 

**Root cause — finite HikariCP pool, no documented override for rest-java:**

`CommonConfiguration` wires HikariCP from `spring.datasource.hikari` properties. The rest-java documentation lists no `maximumPoolSize` override, leaving the HikariCP default of 10 connections. The `statementTimeout` is 10 seconds: [4](#0-3) [5](#0-4) 

**Root cause — HTTP/2 enabled at the proxy:**

The nginx proxy routes `^/api/v1/topics/(\d+\.){0,2}\d+$` to `rest_java_host` with `http2 on`, meaning a single TCP connection can carry hundreds of concurrent streams: [6](#0-5) 

**Exploit flow:**

Each `getTopic()` call makes three sequential `findById()` calls. Without a transaction wrapping all three, each call acquires and releases a pool connection independently, but under high concurrency all 10 pool slots are held simultaneously by in-flight requests. Requests beyond the pool size queue and wait up to the HikariCP `connectionTimeout`. With hundreds of concurrent HTTP/2 streams, the queue grows unboundedly, threads block, and Tomcat's thread pool (default 200) fills up, causing full service degradation.

### Impact Explanation

A single unauthenticated attacker with one HTTP/2 TCP connection can saturate the 10-connection HikariCP pool of a rest-java instance. Queued requests time out, returning 500 errors. The Helm chart's `RestJavaHighDBConnections` alert fires at 75% utilization (already at 7/10 connections), and the circuit breaker (`NetworkErrorRatio() > 0.10`) trips only after errors are already cascading: [7](#0-6) [8](#0-7) 

Because the rest-java pod handles multiple API paths (topics, accounts/allowances, network endpoints), exhausting its DB pool degrades all of them simultaneously, meeting the ≥30% network processing degradation threshold.

### Likelihood Explanation

- **No authentication required**: the endpoint is fully public.
- **No special tooling**: any HTTP/2 client (e.g., `h2load`, `curl --http2`, Python `httpx`) can open one connection and send 100+ concurrent streams.
- **Repeatable**: the attacker can sustain the attack indefinitely from a single IP with minimal bandwidth.
- **Not blocked by existing controls**: `MetricsFilter` only records metrics; `LoggingFilter` only logs; the circuit breaker is reactive (trips after damage); no WAF or per-IP stream limit is configured in the application layer. [9](#0-8) 

### Recommendation

1. **Add a rate-limiting filter to rest-java** using Bucket4j (already a dependency in `web3`) — apply a per-IP token bucket on all `@GetMapping` endpoints, mirroring the pattern in `ThrottleManagerImpl`.
2. **Increase HikariCP pool size** and explicitly document `spring.datasource.hikari.maximum-pool-size` for rest-java, or configure a `connectionTimeout` short enough to fail fast under load.
3. **Limit HTTP/2 concurrent streams per connection** at the nginx/Traefik ingress layer (`http2_max_concurrent_streams` in nginx).
4. **Wrap the three `findById` calls in a single read-only transaction** to reduce the number of pool acquisitions per request from 3 to 1.

### Proof of Concept

```bash
# Install h2load (part of nghttp2)
# Send 500 requests with 100 concurrent streams over a single HTTP/2 connection
h2load \
  --h2 \
  --requests=500 \
  --clients=1 \
  --max-concurrent-streams=100 \
  "https://<mirror-node-host>/api/v1/topics/0.0.1234"
```

**Expected result:** HikariCP pool exhausted within seconds; subsequent requests return HTTP 500 or time out; `hikaricp_connections_pending` metric spikes; `RestJavaHighDBConnections` alert fires; all rest-java endpoints degrade simultaneously.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
