### Title
Unbounded Request Body Parsing on `/mempool/transaction` Enables Memory Exhaustion via Oversized JSON Payloads

### Summary
The `MempoolTransaction` handler in `rosetta/app/services/mempool_service.go` immediately discards its `*types.MempoolTransactionRequest` parameter and returns `ErrNotImplemented`, but the `rosetta-sdk-go` framework fully reads and JSON-decodes the request body into that struct before the handler is ever called. No `http.MaxBytesReader` or equivalent body-size cap is applied anywhere in the middleware chain, so any unauthenticated caller can force the server to allocate arbitrarily large heap objects on every request, causing GC pressure that degrades responsiveness for all concurrent users.

### Finding Description

**Handler code** — `rosetta/app/services/mempool_service.go` lines 30–35:
```go
func (m *mempoolAPIService) MempoolTransaction(
    _ context.Context,
    _ *types.MempoolTransactionRequest,   // parameter is discarded
) (*types.MempoolTransactionResponse, *types.Error) {
    return nil, errors.ErrNotImplemented
}
```
The blank identifier `_` confirms the parsed request is never used. Despite this, the `rosetta-sdk-go` `MempoolAPIController` (an external SDK dependency) reads the full HTTP body via `json.NewDecoder(r.Body).Decode(&request)` and constructs a `types.MempoolTransactionRequest` before dispatching to this handler.

**No body-size guard** — `rosetta/main.go` lines 220–227:
```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```
`http.MaxBytesReader` is never called. A `grep` across the entire rosetta module for `MaxBytesReader`, `LimitReader`, or `ReadLimit` returns zero matches. The middleware chain (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware`) adds no body-size enforcement either.

**Partial mitigation — `ReadTimeout`**: The default `ReadTimeout` is 5 seconds (`5000000000` ns per `docs/configuration.md` line 664). On a 1 Gbps link an attacker can deliver ~625 MB within that window; on a 100 Mbps link still ~62 MB per request. The Go JSON decoder allocates heap proportional to the decoded object graph, so even a few hundred MB of deeply nested JSON can trigger multi-second GC stop-the-world pauses.

**Optional Traefik middleware** — `charts/hedera-mirror-rosetta/values.yaml` lines 151–166 defines `inFlightReq: amount: 5` and `rateLimit: average: 10`, but this is only activated when `global.middleware: true`, which defaults to `false` (line 95). A default deployment has no rate limiting.

### Impact Explanation
Every POST to `/mempool/transaction` with a large body forces a full heap allocation and subsequent GC cycle on the server, even though the result is immediately discarded. An attacker sending a sustained stream of ~50–100 MB payloads (well within the 5-second `ReadTimeout`) can keep the Go GC under continuous pressure, increasing tail latency and potentially causing request timeouts for all other API consumers. Because the endpoint requires no authentication and is publicly reachable, the attack surface is the entire internet. The impact is availability degradation (griefing), not data exfiltration.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of the Rosetta API spec (publicly documented). No credentials, tokens, or special network position are needed. The exploit is trivially repeatable in a loop. The only natural throttle is the 5-second `ReadTimeout`, which still permits large per-request allocations. Any operator who has not deployed the optional Traefik middleware (the default) is fully exposed.

### Recommendation
1. Wrap the request body with `http.MaxBytesReader` in a middleware applied before the SDK controller dispatches, e.g.:
   ```go
   func maxBodyMiddleware(next http.Handler, limit int64) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, limit)
           next.ServeHTTP(w, r)
       })
   }
   ```
   A limit of 4–16 KB is sufficient for any valid Rosetta request.
2. Add this middleware to the chain in `rosetta/main.go` before `server.CorsMiddleware`.
3. Enable the Traefik `inFlightReq` and `rateLimit` middleware by default (`global.middleware: true`) or document it as a required hardening step.

### Proof of Concept
```bash
# Generate a ~50 MB JSON payload with a large string field
python3 -c "
import json, sys
payload = {'transaction_identifier': {'hash': 'A' * 50_000_000}}
sys.stdout.write(json.dumps(payload))
" > big_payload.json

# Send to the endpoint (repeat in a loop to sustain pressure)
while true; do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/mempool/transaction \
       -H 'Content-Type: application/json' \
       --data-binary @big_payload.json &
done
# Monitor server GC pressure: watch -n1 'curl -s http://<rosetta-host>:5700/metrics | grep go_gc'
```
