### Title
Unbounded Multi-Connection Exhaustion of Shared `boundedElastic` Scheduler via Unauthenticated `getNodes()` gRPC Streaming

### Summary
The `getNodes()` handler in `NetworkController` accepts unauthenticated gRPC streaming requests with no global connection limit and no global concurrency cap. The only guard — `maxConcurrentCallsPerConnection = 5` — is trivially bypassed by opening multiple connections. Each concurrent stream drives blocking database calls through the JVM-wide shared `Schedulers.boundedElastic()` instance, which is also used by the topic-message listener and retriever subsystems. An attacker flooding the server with many connections saturates the shared thread pool, causing service-wide degradation.

### Finding Description

**Entry point** — `NetworkController.getNodes()`, lines 33–43: [1](#0-0) 

No authentication check, no rate-limit guard, and no per-subscriber concurrency counter exist anywhere in the call path. The `Mono.fromCallable(() -> toFilter(request)).flatMapMany(networkService::getNodes).subscribe(...)` chain is created fresh for every inbound RPC with no back-pressure or admission control.

**Only mitigation — per-connection cap** — `GrpcConfiguration`, line 33: [2](#0-1) 

`maxConcurrentCallsPerConnection = 5` (from `NettyProperties` default) limits concurrent RPCs *per TCP connection*, not globally. No `maxConnections` or equivalent is configured anywhere in the gRPC module. [3](#0-2) 

**Scheduler usage** — `NetworkServiceImpl.getNodes()`, lines 68–72: [4](#0-3) 

`Schedulers.boundedElastic()` returns the JVM-wide singleton bounded-elastic scheduler (default cap: `10 × availableProcessors` threads, queue: 100 000 tasks). The `repeatWhen` scheduler fires the next `page()` call on a `boundedElastic` thread. `page()` executes a synchronous, blocking `transactionOperations.execute()` DB query, holding that thread for the full query duration. [5](#0-4) 

**Shared scheduler consumers** — the same `Schedulers.boundedElastic()` singleton is used by:
- `PollingTopicListener` (line 31) [6](#0-5) 
- `SharedPollingTopicListener` (line 37) [7](#0-6) 
- `SharedTopicListener.listen()` (line 25) [8](#0-7) 
- `PollingTopicMessageRetriever` (line 41) [9](#0-8) 

Saturating the pool starves all of these subsystems simultaneously.

**`limit = 0` prolongs each stream** — the proto definition allows `limit = 0` to mean "return all nodes": [10](#0-9) 

Combined with `pageDelay = 250 ms` and `pageSize = 10` (defaults), a single stream with `limit = 0` issues repeated DB queries every ~250 ms indefinitely until the address book is exhausted, keeping a `boundedElastic` thread occupied on each page fetch. [11](#0-10) 

### Impact Explanation
Once the `boundedElastic` thread pool is saturated, all components sharing it — topic-message polling listeners, the retriever, and new `getNodes()` requests — queue behind the 100 000-task limit or are rejected. Topic-message streaming (the primary product feature) stalls for all legitimate subscribers. The gRPC service becomes effectively unavailable without any crash or restart, and the condition persists as long as the attacker maintains connections.

### Likelihood Explanation
The attack requires no credentials, no special tooling, and no knowledge beyond the public proto definition (`network_service.proto`). Any gRPC client library can open dozens of HTTP/2 connections and issue 5 concurrent streaming calls per connection. On a typical 4-core deployment the `boundedElastic` pool has 40 threads; 8 connections × 5 streams = 40 concurrent page-fetching tasks is sufficient to saturate it. The attack is repeatable and automatable.

### Recommendation
1. **Add a global concurrent-stream counter** in `NetworkController` (e.g., `AtomicInteger activeStreams`) and reject new subscriptions above a configurable threshold, returning `RESOURCE_EXHAUSTED`.
2. **Dedicate a separate, size-limited scheduler** for `NetworkServiceImpl` rather than sharing the JVM-wide `Schedulers.boundedElastic()` with topic listeners.
3. **Configure a server-level connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / a custom connection-count interceptor to bound total inbound connections.
4. **Enforce a maximum `limit` value** in `AddressBookFilter` (currently unbounded at 0) to cap stream duration.
5. Consider adding a gRPC-level rate-limit interceptor analogous to the `ThrottleConfiguration` used in the web3 module.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import network_service_pb2, network_service_pb2_grpc
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc-host:5600"
CONNECTIONS = 20   # 20 connections × 5 streams = 100 concurrent streams
STREAMS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    threads = []
    for _ in range(STREAMS_PER_CONN):
        req = network_service_pb2.AddressBookQuery(
            file_id=basic_types_pb2.FileID(shardNum=0, realmNum=0, fileNum=102),
            limit=0   # stream all nodes, never self-terminate early
        )
        t = threading.Thread(target=lambda: list(stub.getNodes(req)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for w in workers: w.start()
# Observe: legitimate topic-message subscriptions stall; getNodes latency spikes
```
