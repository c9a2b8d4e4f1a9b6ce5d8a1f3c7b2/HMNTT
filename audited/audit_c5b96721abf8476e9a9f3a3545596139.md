### Title
Unbounded Concurrent Subscriptions Enable Heap Exhaustion via Per-Poll Full Materialization in PollingTopicListener

### Summary
`PollingTopicListener.poll()` issues a database query capped at `maxPageSize=5000` per poll cycle per subscription, but `TopicMessageRepositoryCustomImpl.findByFilter()` materializes the entire result set into a `List` in heap before returning a `Stream`. Because there is no global limit on concurrent gRPC subscriptions and no authentication is required, an unprivileged attacker can open many connections and subscriptions simultaneously, causing the JVM to allocate gigabytes of heap across concurrent poll cycles and triggering `OutOfMemoryError`.

### Finding Description

**Code path 1 — poll() caps page size but does not limit subscriptions:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 54–61:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // capped at 5000
...
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
When no client limit is set, `pageSize` is always `maxPageSize` (default 5000). The filter passed to the repository has `limit=5000`.

**Code path 2 — full heap materialization per poll:**

`grpc/src/main/java/org/hiero/mirror/grpc/repository/TopicMessageRepositoryCustomImpl.java`, line 60:
```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
```
`getResultList()` fetches all rows (up to 5000) into a `java.util.List` in heap before wrapping as a `Stream`. This is not a lazy/cursor-based stream. Each poll cycle for each subscription allocates a full `List<TopicMessage>` of up to 5000 objects.

**Root cause:** The only per-subscription guard is `maxPageSize=5000` (a throughput cap, not a memory guard). There is no global limit on the number of concurrent subscriptions, no authentication on the gRPC endpoint, and `maxConcurrentCallsPerConnection=5` is enforced per TCP connection only — an attacker opens many connections.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection=5` (`NettyProperties`, line 14) is per-connection. An attacker opens N connections → N×5 concurrent subscriptions.
- `TopicMessageFilter.isValidStartTime()` (line 49–50) only requires `startTime <= now()`, so `startTime=0` (epoch) is accepted.
- No authentication or rate limiting exists on the gRPC `subscribeTopic` RPC (the rate-limiting infrastructure in `ThrottleConfiguration` applies only to the web3/REST layer).
- The `subscriberCount` gauge in `TopicMessageServiceImpl` (line 48) is metrics-only; no ceiling is enforced.

### Impact Explanation
Each concurrent subscription in POLL mode allocates up to 5000 `TopicMessage` objects per poll cycle (every 500 ms by default). A `TopicMessage` carries a message body up to 6 KB. At 5000 messages × 6 KB = ~30 MB per subscription per poll cycle. With 200 connections × 5 subscriptions = 1000 concurrent subscriptions, peak heap demand reaches ~30 GB per poll wave. This triggers `OutOfMemoryError`, crashing the JVM and taking the mirror node offline. Because the mirror node is the only path for clients to receive HCS topic messages, this constitutes a complete denial of service for topic subscriptions.

### Likelihood Explanation
No credentials are required. A single attacker machine can open hundreds of TCP connections to port 5600 and issue `subscribeTopic` RPCs with `startTime=0` and no limit. The attack is repeatable: after a restart the attacker reconnects. The gRPC protocol allows HTTP/2 multiplexing, making it cheap to open many streams. The POLL listener type must be configured (`hiero.mirror.grpc.listener.type=POLL`), but it is a documented, supported option and the vulnerability also partially applies to the retriever path (`PollingTopicMessageRetriever` uses the same `getResultList()` materialization for historical messages regardless of listener type).

### Recommendation
1. **Enforce a global concurrent-subscription ceiling** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum, returning `RESOURCE_EXHAUSTED`.
2. **Replace `getResultList()` with a true cursor/streaming query** in `TopicMessageRepositoryCustomImpl.findByFilter()` to avoid full heap materialization. Use `ScrollableResults` with `FORWARD_ONLY` / `READ_ONLY` hints, or switch to R2DBC for reactive back-pressure.
3. **Add a global connection limit** via `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a server-wide `maxConcurrentCalls` (not just per-connection).
4. **Add rate limiting at the gRPC interceptor layer** (e.g., bucket4j or Resilience4j) analogous to the existing `ThrottleConfiguration` in the web3 module.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_pb2

TARGET = "mirror-node-host:5600"
TOPIC_ID = "0.0.12345"   # any high-volume topic
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5  # maxConcurrentCallsPerConnection default

def subscribe(stub):
    query = mirror_pb2.ConsensusTopicQuery(
        topicID=...,          # set to TOPIC_ID
        consensusStartTime=0, # epoch — very old startTime, accepted by isValidStartTime()
        # limit intentionally omitted → no limit → pageSize = maxPageSize = 5000
    )
    for _ in stub.subscribeTopic(query):
        pass  # consume slowly or not at all to keep subscription alive

threads = []
for _ in range(NUM_CONNECTIONS):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(target=subscribe, args=(stub,))
        t.start()
        threads.append(t)

# 200 connections × 5 calls = 1000 concurrent subscriptions
# Each poll cycle (every 500ms): 1000 × 5000 TopicMessage objects materialized into heap
# → OOM / mirror node crash
