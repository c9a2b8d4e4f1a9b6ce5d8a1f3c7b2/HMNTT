### Title
Malicious Block Node Can Exploit `BlockAssembler.append()` to Cause Repeated Error Loop, Disrupting Block Streaming

### Summary
A block node operator can intentionally send a `BLOCK_HEADER` item while a previous block's items are still pending in `BlockAssembler`, or send a non-`BLOCK_HEADER` item as the very first response, triggering a `BlockStreamException`. This increments the per-node error counter and, after `maxSubscribeAttempts` (default: 3) consecutive failures, marks the node inactive. Because `getNode()` force-readmits the only available inactive node, a single malicious node can trap the importer in a permanent disruption loop with no ability to advance block processing.

### Finding Description

**Root cause and failed assumption:**

The `append()` check correctly validates protocol ordering but assumes the remote block node is honest. There is no authentication or integrity check on the content of `SubscribeStreamResponse` messages. A malicious node can send any sequence of `BlockItem` types it chooses.

**Exploit flow:**

The attacker operates a block node that the importer is configured to connect to (via static config or auto-discovery, since `autoDiscoveryEnabled = true` by default in `BlockProperties`).

On each connection attempt, the malicious node immediately sends a `BlockItemSet` whose first item is **not** a `BLOCK_HEADER` (e.g., a `SIGNED_TRANSACTION`). This triggers the second branch of `append()` before `errors.set(0)` at line 157 is ever reached (that reset only fires after a successful response is fully processed). The exception propagates to `onError()`, incrementing `errors`.

- Connection 1 → `errors = 1`
- Connection 2 → `errors = 2`
- Connection 3 → `errors = 3` → `active = false`, `errors = 0`, `readmitTime = now + 1 min`

When `getNode()` is called next and the malicious node is the only one, the force-readmit path (`node.tryReadmit(true)`) re-activates it unconditionally, and the cycle repeats indefinitely. The importer never advances past the failing block number.

**Why existing checks are insufficient:**

- `onError()` is designed to handle transient failures, not adversarial ones. The force-readmit fallback in `getNode()` (lines 122–128) was intended to prevent permanent stalls when all nodes are temporarily unavailable, but it also prevents the importer from ever giving up on a malicious node.
- `errors.set(0)` (line 157) only resets the counter after a successful response, so an attacker who never sends a valid first item can accumulate errors across connections without interference.
- There is no rate-limiting, circuit-breaking, or permanent blacklisting of a node that repeatedly sends malformed data.
- `StreamProperties.readmitDelay` (default 1 minute) provides only a temporary pause, not a permanent exclusion.

### Impact Explanation

If the malicious node is the sole configured block node, the importer is trapped in a tight loop: connect → malformed response → error → inactive → force-readmit → repeat. Block processing halts entirely for the duration of the attack. Even with multiple nodes, if the attacker controls all configured nodes, the same outcome applies. The impact is classified as griefing/availability disruption with no direct economic damage to network users, but it can cause the mirror node to fall behind on block ingestion, degrading data freshness for downstream consumers.

### Likelihood Explanation

The precondition is that the attacker's node must be in the importer's node list. With `autoDiscoveryEnabled = true` (default in `BlockProperties`), nodes can be added dynamically. A block node operator — even one with no special network privileges — who is included in the discovery set can execute this attack trivially by modifying their node software to return a malformed first `BlockItemSet`. The attack is fully repeatable, requires no cryptographic material, and survives the 1-minute readmit delay because the force-readmit path re-enables the node unconditionally.

### Recommendation

1. **Permanent blacklisting after repeated malformed-data errors**: Distinguish between transient connectivity errors and protocol-violation errors (`BlockStreamException` thrown from `append()`). Protocol violations should use a separate, non-resetting counter with exponential backoff or permanent exclusion rather than the same `maxSubscribeAttempts` counter used for network timeouts.
2. **Remove unconditional force-readmit**: The `node.tryReadmit(true)` call in `getNode()` should not bypass the readmit delay for nodes that were deactivated due to protocol violations. Only nodes deactivated for connectivity reasons should be force-readmitted.
3. **Authenticated block node connections**: Require mutual TLS or a signed challenge-response so that only authorized block nodes can serve streams to the importer.
4. **Separate error counters**: Track `protocolErrors` separately from `connectivityErrors` so that a node sending malformed data is penalized more severely than one that simply times out.

### Proof of Concept

**Preconditions:**
- Attacker controls a block node reachable by the importer (via static config or auto-discovery).
- The importer has only this one node configured, or the attacker controls all configured nodes.

**Steps:**

1. Stand up a gRPC server implementing `BlockStreamSubscribeService.subscribeBlockStream`.
2. For every incoming `SubscribeStreamRequest`, immediately respond with a single `SubscribeStreamResponse` containing a `BlockItemSet` whose first (and only) `BlockItem` has `ItemCase = SIGNED_TRANSACTION` (any non-`BLOCK_HEADER` type).
3. Observe the importer log: `BlockStreamException: Incorrect first block item case SIGNED_TRANSACTION` is thrown.
4. `onError()` increments `errors`. After 3 such connections, the node is logged as inactive.
5. On the next `doGet()` call, `getNode()` finds no active nodes, force-readmits the malicious node via `tryReadmit(true)`, and returns it.
6. Steps 2–5 repeat indefinitely. The importer's block counter does not advance; block streaming is disrupted for as long as the attacker's node remains in the node list.
