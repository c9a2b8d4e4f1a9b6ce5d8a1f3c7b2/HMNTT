### Title
Unauthenticated Pagination Amplification via `limit=1` in `/accounts/:id/rewards` Enabling DB Connection Exhaustion

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` accepts `limit=1` with no minimum limit enforcement and no per-IP rate limiting. An unprivileged attacker can follow each auto-generated pagination link sequentially, causing two DB queries per request (account validation + reward fetch), multiplying total DB load by up to the default-limit factor (25×) compared to a normal traversal.

### Finding Description
**Code path:**
- Route registration: `rest/routes/accountRoute.js` line 16 — `router.getExt(getPath('rewards'), AccountController.listStakingRewardsByAccountId)` — no auth middleware.
- Handler: `rest/controllers/accountController.js` lines 170–203.
  - Line 171: `EntityService.getEncodedId(...)` — DB query 1 (entity lookup).
  - Line 172: `EntityService.isValidAccount(accountId)` — DB query 2 (account validation).
  - Lines 180–185: `StakingRewardTransferService.getRewards(...)` — DB query 3 (reward fetch with `LIMIT $2`).
  - Lines 194–200: If `rewards.length === query.limit`, a `next` link is emitted containing the last timestamp as a cursor.
- `extractStakingRewardsQuery` (lines 110–162): at line 124–126, `limit = filter.value` is assigned directly from the parsed filter with **no minimum value check**.
- `buildAndValidateFilters` caps the limit at a configured maximum but does **not** enforce a minimum above 1.

**Root cause:** No minimum limit is enforced for the `limit` query parameter on this endpoint, and no per-IP or per-session rate limiting is applied. With `limit=1`, every single staking reward record generates a new pagination link, turning a single full traversal into N sequential HTTP requests each consuming 3 DB queries.

### Impact Explanation
For an account with N staking rewards (accounts staking daily can accumulate hundreds to thousands of records), an attacker using `limit=1` forces N×3 DB queries instead of ⌈N/25⌉×3 with the default limit — a 25× amplification. Sustained parallel attacks across multiple account IDs can exhaust the DB connection pool (configured via `rest/config.js` `db.pool.maxConnections`), causing connection timeouts and service degradation for all legitimate users. No economic loss occurs, but availability is impacted.

### Likelihood Explanation
The endpoint is fully public (no authentication, no API key). The pagination `next` link is machine-readable JSON, making automated traversal trivial with a simple loop. Any external actor can execute this with a basic HTTP client. The attack is repeatable and scalable across multiple concurrent clients.

### Recommendation
1. Enforce a minimum `limit` value (e.g., ≥ 1 is already the floor, but raise it to ≥ 10 or ≥ 25) in `extractStakingRewardsQuery` or in `buildAndValidateFilters` for this endpoint.
2. Apply per-IP rate limiting middleware to the `/accounts/:id/rewards` route (and all paginated endpoints).
3. Consider caching responses for identical query parameters to reduce repeated DB hits.

### Proof of Concept
```bash
# Step 1: Initial request with limit=1
curl "https://<mirror-node>/api/v1/accounts/0.0.1001/rewards?limit=1"
# Response includes: "links": {"next": "/api/v1/accounts/0.0.1001/rewards?order=desc&limit=1&timestamp=lt:1234567890.000033333"}

# Step 2: Automate following all pagination links
NEXT="/api/v1/accounts/0.0.1001/rewards?limit=1"
while [ "$NEXT" != "null" ]; do
  RESPONSE=$(curl -s "https://<mirror-node>$NEXT")
  NEXT=$(echo $RESPONSE | jq -r '.links.next')
done
# Each iteration fires 3 DB queries; N rewards = N iterations = 3N DB queries
# Run this in parallel across many accounts/clients to exhaust DB pool
```

```javascript
  listStakingRewardsByAccountId = async (req, res) => {
    const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
    const isValidAccount = await EntityService.isValidAccount(accountId);
    if (!isValidAccount) {
      throw new NotFoundError();
    }
    const filters = utils.buildAndValidateFilters(req.query, acceptedRewardsParameters);
    const query = this.extractStakingRewardsQuery(filters);
    // insert account id at $1, and limit (at $2)
    query.params.unshift(accountId, query.limit);
    const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(
      query.order,
      query.limit,
      query.conditions,
      query.params
    );
    const rewards = stakingRewardsTransfers.map((reward) => new StakingRewardTransferViewModel(reward));
    const response = {
      rewards,
      links: {
        next: null,
      },
    };

    if (response.rewards.length === query.limit) {
      const lastRow = last(response.rewards);
      const lastValue = {
        [filterKeys.TIMESTAMP]: lastRow.timestamp,
      };
      response.links.next = utils.getPaginationLink(req, false, lastValue, query.order);
    }

    res.locals[responseDataLabel] = response;
  };
```

```javascript
  async getRewards(order, limit, conditions, initParams) {
    const {query, params} = this.getRewardsQuery(order, limit, conditions, initParams);
    const rows = await super.getRows(query, params);
    return rows.map((srt) => new StakingRewardTransfer(srt));
  }

  getRewardsQuery(order, limit, conditions, params) {
    const query = [
      StakingRewardTransferService.listStakingRewardsByAccountIdQuery,
      conditions.length > 0 ? `and ${conditions.join(' and ')}` : '', // "and" since we already have "where account_id = $1" at the end of the above line
      super.getOrderByQuery(
        OrderSpec.from(StakingRewardTransfer.getFullName(StakingRewardTransfer.CONSENSUS_TIMESTAMP), order)
      ),
      super.getLimitQuery(2), // limit is specified in $2 (not necessarily a limit *of* 2)
    ].join('\n');

    return {query, params};
  }
```
