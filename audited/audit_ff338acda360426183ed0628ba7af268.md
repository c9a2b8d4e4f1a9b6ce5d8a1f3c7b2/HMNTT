### Title
Unauthenticated Repeated Calls to `/api/v1/network/supply` Cause Unbounded DB Aggregation with No Caching or Rate Limiting

### Summary
The `GET /api/v1/network/supply` endpoint is publicly accessible with no authentication, rate limiting, or response caching in the `rest-java` module. Every call with no `timestamp` parameter triggers `entityRepository.getSupply()`, which executes a non-trivial PostgreSQL aggregation query joining the `entity` table against 7 account ranges via `unnest(string_to_array(...))` on every invocation. A sustained moderate request rate from any unauthenticated client can drive DB CPU well above the 24-hour baseline.

### Finding Description

**Code path:**

1. `NetworkController.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 132–137: the endpoint is a plain `@GetMapping` with no authentication annotation, no rate-limit guard, and no cache annotation.

2. `NetworkServiceImpl.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, lines 59–89: when `timestamp.isEmpty()` (the no-parameter case), it unconditionally calls `entityRepository.getSupply(lowerBounds, upperBounds)` on every invocation. No `@Cacheable` or any other memoization is present anywhere in the `rest-java` service or repository layer (confirmed by exhaustive grep for `@Cacheable`, `@CacheEvict`, `CacheConfig` — zero matches in `rest-java/**/*.java`).

3. `EntityRepository.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, lines 19–29: the native query is:
   ```sql
   select cast(coalesce(sum(e.balance), 0) as bigint) as unreleased_supply,
          cast(coalesce(max(e.balance_timestamp), 0) as bigint) as consensus_timestamp
   from entity e
   join unnest(
           cast(string_to_array(:lowerBounds, ',') as bigint[]),
           cast(string_to_array(:upperBounds, ',') as bigint[])
        ) as ranges(min_val, max_val)
     on e.id between ranges.min_val and ranges.max_val
   ```
   This materialises 7 range pairs (from `NetworkProperties.java` lines 22–29: accounts 2, 42, 44–71, 73–87, 99–100, 200–349, 400–750), performs a join against the `entity` table, and computes `sum(balance)` and `max(balance_timestamp)` — a full aggregation pass — on every single HTTP request.

4. **No rate limiting in `rest-java`**: The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` classes exist only in the `web3` module. The `rest-java` module has no equivalent. The `HIGH_VOLUME_THROTTLE` parameter visible in `NetworkController` applies only to the POST `/fees` endpoint. No per-IP, per-endpoint, or global request-rate guard exists for `/api/v1/network/supply`.

**Root cause:** The design assumes the query is cheap enough to run on every request, but provides no caching layer and no rate-limiting layer to bound the aggregate DB load from concurrent or rapid callers.

### Impact Explanation

Each request forces PostgreSQL to: parse two comma-separated strings, cast them to `bigint[]`, unnest 7 pairs, join the `entity` table (which on a production mirror node contains millions of rows, though the ranges cover system accounts), and compute two aggregate functions. Even with index scans on `entity.id`, the repeated query planning, function evaluation, and aggregation overhead accumulates linearly with request rate. Because the result is never cached, 100–500 concurrent or rapid sequential requests per second — well within what a single attacker machine can sustain — can push DB CPU 30%+ above a low-to-moderate baseline, degrading query latency for all other mirror-node consumers (balance lookups, transaction queries, etc.).

### Likelihood Explanation

- **Precondition**: none — the endpoint is public, unauthenticated, and documented in the OpenAPI spec (`rest/api/v1/openapi.yml` lines 1010–1028).
- **Attacker capability**: any HTTP client; a single `ab`, `wrk`, or `curl` loop suffices.
- **Repeatability**: indefinite — no server-side counter, token bucket, or circuit breaker exists in `rest-java` for this path.
- **Detection difficulty**: the requests are individually valid and indistinguishable from legitimate monitoring traffic (the project's own `rest/monitoring/network_tests.js` polls this endpoint).

### Recommendation

1. **Add response caching**: annotate `NetworkServiceImpl.getSupply()` (or the repository method) with Spring's `@Cacheable` with a short TTL (e.g., 5–15 seconds). The supply value changes only at consensus-balance-update frequency, so staleness is acceptable.
2. **Add rate limiting to `rest-java`**: implement a `bucket4j`-based filter (mirroring the `web3` `ThrottleConfiguration`) applied globally or per-endpoint to the `rest-java` Spring context.
3. **Consider a materialized/pre-computed supply value**: schedule a background job to compute and store the current supply, and serve it directly from a single-row cache table, eliminating the per-request aggregation entirely.

### Proof of Concept

```bash
# No authentication, no parameters — triggers entityRepository.getSupply() on every call
# Run from any machine with network access to the mirror node REST-Java port

wrk -t8 -c200 -d60s http://<mirror-node-host>/api/v1/network/supply

# Alternatively with curl in a tight loop:
while true; do
  curl -s http://<mirror-node-host>/api/v1/network/supply -o /dev/null &
done

# Monitor DB CPU on the PostgreSQL host:
# watch -n1 "psql -c \"SELECT query, calls, total_exec_time FROM pg_stat_statements WHERE query LIKE '%unreleased_supply%' ORDER BY total_exec_time DESC LIMIT 5;\""
```

Sustained execution will show `calls` incrementing rapidly for the `getSupply` query and `total_exec_time` growing proportionally, with DB CPU rising above the 24-hour baseline by 30%+ under moderate concurrency.
