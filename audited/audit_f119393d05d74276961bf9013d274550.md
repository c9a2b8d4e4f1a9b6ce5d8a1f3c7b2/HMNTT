### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses DB Page Size Cap, Enabling Unbounded Query DoS

### Summary
An unprivileged gRPC client can supply `limit = Long.MAX_VALUE` (or any value > `Integer.MAX_VALUE`) in a `ConsensusTopicQuery`. The unchecked `(int)` narrowing cast in `PollingTopicListener.poll()` wraps the value to `-1`, which then causes `hasLimit()` to return `false` in the repository layer, removing the `setMaxResults` guard entirely and allowing the database to return every matching row on every poll cycle.

### Finding Description

**Entry point — no upper-bound validation:**
`ConsensusController.toFilter()` copies the proto `uint64 limit` field directly into `TopicMessageFilter` with no ceiling check. [1](#0-0) 

`TopicMessageFilter.limit` carries only `@Min(0)` — there is no `@Max` constraint, so `Long.MAX_VALUE` passes bean validation. [2](#0-1) 

**Overflow site — `PollingTopicListener.poll()` lines 54-57:**
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())   // ← narrowing cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```
With `filter.getLimit() = Long.MAX_VALUE` and `context.getCount() = 0`:
- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- `(int) 0x7FFFFFFFFFFFFFFF = 0xFFFFFFFF = -1` (lower 32 bits, two's-complement)
- `Math.min(-1, 5000) = -1`
- `newFilter` is built with `limit = -1` [3](#0-2) 

**Guard bypass — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51-53:**
```java
if (filter.hasLimit()) {          // hasLimit() = (limit > 0) = (-1 > 0) = false
    typedQuery.setMaxResults(...); // ← never reached
}
```
`hasLimit()` returns `false` for any negative limit, so `setMaxResults` is never called and the JPA query fetches **all** rows matching the topic/timestamp predicate. [4](#0-3) [5](#0-4) 

The `maxPageSize` default of 5000 is completely bypassed. [6](#0-5) 

### Impact Explanation
Every poll cycle (default every 500 ms) issues an unbounded `SELECT *` against the `topic_message` table for the chosen topic. On a busy topic with millions of rows, this exhausts database memory, connection pool resources, and JVM heap on the mirror node. A single persistent gRPC subscription is sufficient to degrade or crash the service for all other subscribers, constituting a non-network-based DoS against the gRPC mirror node component.

### Likelihood Explanation
The gRPC endpoint is publicly reachable with no authentication. The `ConsensusTopicQuery` proto field `uint64 limit` accepts the full 64-bit range; setting it to `Long.MAX_VALUE` requires only a standard gRPC client (e.g., `grpcurl`). No special privileges, tokens, or knowledge of internal state are required. The attack is trivially repeatable and persistent for the lifetime of the subscription. [7](#0-6) 

### Recommendation
1. **Add an upper-bound constraint** on `TopicMessageFilter.limit` (e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate ceiling such as `1_000_000L`) so the bean-validation layer rejects oversized values before they reach the listener.
2. **Replace the unsafe narrowing cast** in `PollingTopicListener.poll()` with a saturating clamp:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Validate in the controller** that `query.getLimit()` does not exceed a configured maximum before building the filter. [3](#0-2) 

### Proof of Concept
```bash
# Prerequisites: grpcurl installed, mirror node gRPC on localhost:5600,
# topic 0.0.1234 exists with at least one message.

grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
**Expected (vulnerable) behaviour:** The mirror node issues a `SELECT * FROM topic_message WHERE topic_id = 1234 AND consensus_timestamp >= 0 ORDER BY consensus_timestamp ASC` with **no** `LIMIT` clause on every 500 ms poll, returning the entire table partition for that topic. Database CPU and memory spike; other subscribers experience degraded latency or connection timeouts.

**Root cause confirmation:** Add a breakpoint or log at `PollingTopicListener.java:57` — `pageSize` will be `-1` and `newFilter.getLimit()` will be `-1`, confirming the overflow and the subsequent bypass of `setMaxResults`.
