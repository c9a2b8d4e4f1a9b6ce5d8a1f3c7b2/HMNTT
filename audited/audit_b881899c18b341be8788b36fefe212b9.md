### Title
Integer Overflow in `getAdjustedLowerRangeValue()` Bypasses Range Guard, Enabling Unbounded NFT_ALLOWANCE Table Scan

### Summary
`Bound.getAdjustedLowerRangeValue()` increments the raw `long` value by 1 when the operator is `GT`, with no overflow guard. An unprivileged caller can supply a valid Hedera entity ID that encodes to `Long.MAX_VALUE` (e.g. `511.65535.274877906943`) with operator `gt`, causing the increment to silently wrap to `Long.MIN_VALUE`. The subsequent range-validity check `adjustedLower > adjustedUpper` evaluates to `false` (Long.MIN_VALUE is not greater than Long.MAX_VALUE), so no exception is thrown and the corrupted lower bound is forwarded to the database query, effectively removing the secondary account-ID filter and forcing a full per-account scan of `NFT_ALLOWANCE`.

### Finding Description

**Exact code path**

`Bound.java`, lines 86–97: [1](#0-0) 

```java
public long getAdjustedLowerRangeValue() {
    if (this.lower == null) { return 0; }
    long lowerBound = this.lower.value();
    if (this.lower.operator() == RangeOperator.GT) {
        lowerBound++;          // ← no overflow check
    }
    return lowerBound;
}
```

**Root cause – EntityId encoding allows Long.MAX_VALUE**

`EntityId.encode()` packs shard (10 bits), realm (16 bits), and num (38 bits) into a signed `long`. [2](#0-1) 

`Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF` decomposes to shard=511, realm=65535, num=274877906943 — all within their respective bit-width limits — so `encode()` accepts it without error. A user can therefore supply the URL parameter `account.id=gt:511.65535.274877906943`.

**Why the guard fails**

The `Bound` constructor calls `getAdjustedLowerRangeValue()` and then checks: [3](#0-2) 

```java
long adjustedLower = getAdjustedLowerRangeValue(); // Long.MIN_VALUE after overflow
long adjustedUpper = adjustUpperBound();            // Long.MAX_VALUE (no upper supplied)
if (primarySortField && adjustedLower > adjustedUpper) { throw ... }
```

`Long.MIN_VALUE > Long.MAX_VALUE` is `false`, so the guard is silently bypassed.

**Exploit flow**

1. Attacker sends: `GET /api/v1/accounts/{id}/allowances/nfts?account.id=gt:511.65535.274877906943`
2. `EntityIdRangeParameter.valueOf()` parses this into `(GT, Long.MAX_VALUE)`.
3. `Bound` constructor calls `getAdjustedLowerRangeValue()` → `Long.MAX_VALUE + 1` → `Long.MIN_VALUE`.
4. Range guard passes (Long.MIN_VALUE ≤ Long.MAX_VALUE).
5. `NftAllowanceRepositoryCustomImpl.findAll()` builds: `WHERE owner = :id AND approved_for_all = true AND spender >= Long.MIN_VALUE` — the spender predicate matches every row. [4](#0-3) 

**Service-layer checks are insufficient**

`NftAllowanceServiceImpl.checkOwnerSpenderParamValidity()` calls `verifyRangeId()` (checks for NE, duplicate operators, EQ+range mix) but never re-validates the adjusted numeric value for overflow. [5](#0-4) 

### Impact Explanation
Every request with this crafted parameter causes the database to scan all `NFT_ALLOWANCE` rows for the target account (ignoring the intended spender/owner range filter). With a large account, this is a significantly wider scan than intended. Repeated at high frequency by any unauthenticated caller, this constitutes a griefing/DoS vector against the database tier. The result set is bounded by `limit` (MAX_LIMIT), so data exfiltration beyond what is normally visible is minimal, but the server-side cost is disproportionate to the apparent request.

### Likelihood Explanation
No authentication or special privilege is required. The crafted entity ID (`511.65535.274877906943`) is syntactically valid and passes all input parsing. The attack is trivially repeatable with a single HTTP GET request and requires no knowledge of internal state. Any user of the public REST API can trigger it.

### Recommendation
1. **Guard against overflow in `getAdjustedLowerRangeValue()`** (and the identical pattern in `getInclusiveRangeValue()`):
   ```java
   if (this.lower.operator() == RangeOperator.GT) {
       if (lowerBound == Long.MAX_VALUE) {
           throw new IllegalArgumentException(
               "GT lower bound overflow for " + parameterName);
       }
       lowerBound++;
   }
   ```
2. Apply the same fix to `adjustUpperBound()` for the `LT` decrement path (symmetric risk with `Long.MIN_VALUE`).
3. Alternatively, cap the maximum accepted entity ID numeric value to a known-safe ceiling (e.g. the maximum valid encoded ID) before the `Bound` is constructed.

### Proof of Concept

```
# Step 1 – identify a valid account on the network (any existing account ID works)
ACCOUNT=0.0.1234

# Step 2 – send the crafted request
curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/allowances/nfts?account.id=gt:511.65535.274877906943"

# Expected (vulnerable) behaviour:
#   HTTP 200, returns up to MAX_LIMIT allowances for the account
#   with NO lower-bound filtering on spender/owner — identical to
#   omitting the account.id parameter entirely.
#
# Expected (fixed) behaviour:
#   HTTP 400 Bad Request – "GT lower bound overflow for account.id"
```

Repeating this request in a tight loop against a high-volume account will cause sustained full-index scans on `NFT_ALLOWANCE`, degrading database performance for all users.

    private static long encode(long shard, long realm, long num) {
        if (shard > SHARD_MASK || shard < 0 || realm > REALM_MASK || realm < 0 || num > NUM_MASK || num < 0) {
            throw new InvalidEntityException("Invalid entity ID: " + shard + "." + realm + "." + num);
        }

        if (shard == 0 && realm == 0) {
            return num;
        }

        return (num & NUM_MASK) | (realm & REALM_MASK) << NUM_BITS | (shard & SHARD_MASK) << (REALM_BITS + NUM_BITS);
    }

    private void checkOwnerSpenderParamValidity(Bound ownerOrSpenderParams, Bound tokenParams) {
        if (ownerOrSpenderParams.isEmpty() && !tokenParams.isEmpty()) {
            throw new IllegalArgumentException("token.id parameter must have account.id present");
        }

        verifyRangeId(ownerOrSpenderParams);
        verifyRangeId(tokenParams);

        if (!ownerOrSpenderParams.hasLowerAndUpper()
                && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
            throw new IllegalArgumentException("Invalid range provided for %s".formatted(Constants.TOKEN_ID));
        }

        if (tokenParams.getCardinality(RangeOperator.LT, RangeOperator.LTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.LTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an lte or eq account.id parameter");
        }
        if (tokenParams.getCardinality(RangeOperator.GT, RangeOperator.GTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.GTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an gte or eq account.id parameter");
        }
    }

    private void verifyRangeId(Bound ids) {
        ids.verifyUnsupported(RangeOperator.NE);
        ids.verifySingleOccurrence();
        ids.verifyEqualOrRange();
