### Title
Unbounded Reactive Subscription Resource Exhaustion via Unauthenticated `subscribeTopic` with `limit=0` and Past `startTime`

### Summary
Any unauthenticated external caller can send a `ConsensusTopicQuery` with `limit=0` and `consensusStartTime` set to the distant past (e.g., epoch), causing `subscribeTopic()` to create a reactive `Flux` that never terminates. Because there is no global subscriber cap and the per-connection call limit is trivially bypassed by opening multiple connections, an attacker can exhaust thread pool, database connection pool, and memory resources, denying service to legitimate users.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` without authentication and passes it directly to `toFilter()`: [1](#0-0) 

`toFilter()` (lines 55–73) sets `limit = query.getLimit()`. When the client sends `limit=0`, `TopicMessageFilter.hasLimit()` returns `false`: [2](#0-1) 

The only `startTime` validation is `isValidStartTime()`, which accepts any timestamp **at or before now** — including epoch (year 1970): [3](#0-2) 

In `TopicMessageServiceImpl.subscribeTopic()`, the `take(limit)` operator is only applied when `hasLimit()` is true, and `pastEndTime()` returns `Flux.never()` when no `endTime` is set: [4](#0-3) [5](#0-4) 

The resulting flux is `historical.concatWith(Flux.merge(safetyCheck, live).takeUntilOther(Flux.never()))` — it never completes unless the client disconnects. The `subscriberCount` field is a metrics gauge only; it enforces no cap: [6](#0-5) 

The only server-side connection throttle is `maxConcurrentCallsPerConnection = 5`, which is trivially bypassed by opening multiple TCP connections: [7](#0-6) [8](#0-7) 

**Root cause:** The protocol explicitly documents `limit=0` as "receive indefinitely" and `consensusStartTime` in the past as valid. The server implements this faithfully but provides no global bound on concurrent indefinite subscriptions and no per-IP rate limit on the gRPC endpoint. [9](#0-8) 

### Impact Explanation

Each open subscription holds:
- A reactive scheduler thread slot (bounded elastic pool)
- A periodic database polling connection (R2DBC pool, default `statementTimeout=10s` per poll cycle, but the subscription itself is indefinite)
- A Redis listener slot (when `listener.type=REDIS`)
- Memory for the `TopicContext` and message buffers (up to `maxBufferSize=16384` messages)

With N connections × 5 calls each, an attacker can saturate the thread pool and DB connection pool, causing legitimate subscribers to time out or be rejected. The `GrpcHighDBConnections` alert threshold is 75% utilization — an attacker can trigger this and beyond. [10](#0-9) 

### Likelihood Explanation

No authentication, no API key, no IP-based rate limit, and no global subscriber cap are required. The gRPC port (default 5600) is publicly reachable. The attack requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK). It is repeatable and scriptable. The attacker needs only a valid topic ID (which is public on-chain data) or can exploit `checkTopicExists=false` deployments without even that.

### Recommendation

1. **Enforce a global maximum concurrent subscriber count** — reject new subscriptions with `RESOURCE_EXHAUSTED` when the count exceeds a configurable threshold.
2. **Enforce a per-IP concurrent subscription limit** using the remote address from `ServerCallStreamObserver`.
3. **Enforce a maximum subscription duration** (e.g., `maxConnectionAge` on the Netty server builder) so indefinite subscriptions are forcibly terminated after a configurable wall-clock timeout.
4. **Add gRPC-level rate limiting** on new `subscribeTopic` calls per source IP, analogous to the `ThrottleConfiguration` used in the web3 module.
5. **Consider requiring `limit > 0` or `endTime` to be set** for subscriptions that start in the distant past, since replaying the full topic history with no termination condition is the highest-risk pattern.

### Proof of Concept

```bash
# Install grpcurl, then:
grpcurl -plaintext \
  -proto consensus_service.proto \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0, "nanos": 0},
    "limit": 0
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Stream never terminates. Repeat in a loop across multiple connections:
for i in $(seq 1 200); do
  grpcurl -plaintext ... &
done
# Each connection holds 5 concurrent calls = 1000 open subscriptions,
# exhausting the DB connection pool and thread pool.
```
