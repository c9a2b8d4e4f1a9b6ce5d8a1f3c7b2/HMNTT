### Title
Unauthenticated GraphQL `account` Query Exposes System Account Balance and Key via Long-Zero EVM Address

### Summary
The GraphQL `account(input: AccountInput!)` resolver accepts an `evmAddress` that encodes a Hedera entity number in long-zero form (first 12 bytes all zero, last 8 bytes = entity num). When such an address is supplied, `EntityServiceImpl.getByEvmAddressAndType` decodes it and calls `entityRepository.findById(entityNum)` with no restriction on system account numbers. Any unprivileged external caller can therefore retrieve the `balance` and `key` fields of system accounts such as 0.0.98 (fee collection) by supplying `evmAddress: "0x0000000000000000000000000000000000000062"`.

### Finding Description

**Code path:**

`AccountController.account()` (line 51–54) passes the raw `evmAddress` string to `EntityServiceImpl.getByEvmAddressAndType`: [1](#0-0) 

Inside `EntityServiceImpl.getByEvmAddressAndType` (lines 34–41), the address is decoded and the ByteBuffer is inspected: [2](#0-1) 

When the first 4 bytes (shard) and next 8 bytes (realm) are both zero, the method calls `entityRepository.findById(buffer.getLong())` — a plain Spring Data `CrudRepository.findById()` — with the entity number extracted from the last 8 bytes. The only post-fetch filter is `e.getType() == type` (must be `ACCOUNT`), which system accounts satisfy. [3](#0-2) 

**Root cause:** The GraphQL `EntityRepository` and `EntityServiceImpl` apply no system-account number guard. The `web3` module does have such a guard (`AccountReadableKVState` checks `systemAccounts.contains(key)` and `ContractCallContext.isBalanceCallSafe()`), but that protection is entirely absent in the `graphql` module. [4](#0-3) 

**No authentication exists** on the GraphQL endpoint — the only configuration files found are `GraphQlConfiguration.java` (query complexity/depth limits) and `LoggingFilter.java`; no `SecurityConfiguration` exists in the graphql module. [5](#0-4) 

### Impact Explanation
An attacker can read the current `balance` (fee reserve level) and `key` (admin key structure) of any Hedera system account — including the fee collection account (0.0.98), treasury (0.0.2), staking reward account (0.0.800), and others — without any credentials. This exposes operational intelligence about fee reserve levels and the structure of privileged admin keys, which is outside the design parameters of the public mirror node API.

### Likelihood Explanation
Exploitation requires zero privileges, zero credentials, and a single HTTP POST to the public GraphQL endpoint. The long-zero EVM address encoding is documented behavior (used throughout the codebase and in tests). Any external party aware of Hedera's EVM address encoding convention can perform this immediately and repeatably.

### Recommendation
In `EntityServiceImpl.getByEvmAddressAndType` (and analogously in `getByIdAndType`), add a system-account blocklist or range check before calling `entityRepository.findById`. Mirror the guard already present in the `web3` module:

```java
// After extracting entityNum from the long-zero address:
if (entityNum > 0 && entityNum <= SYSTEM_ACCOUNT_BOUNDARY) {
    return Optional.empty(); // refuse to expose system accounts
}
return entityRepository.findById(entityNum).filter(e -> e.getType() == type);
```

Alternatively, add a `@PreAuthorize` or Spring Security rule restricting system-account queries to authenticated/privileged callers.

### Proof of Concept

```bash
# 0x0000000000000000000000000000000000000062 = decimal 98 = account 0.0.98 (fee collection)
curl -X POST http://<mirror-node-host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "{ account(input: { evmAddress: \"0x0000000000000000000000000000000000000062\" }) { balance key entityId { shard realm num } } }"
  }'
```

Expected result: HTTP 200 with `balance` (fee reserve in tinybars) and `key` (admin key map) for account 0.0.98 returned to an unauthenticated caller.

To target other system accounts, substitute the last byte(s):
- `0x0000000000000000000000000000000000000002` → 0.0.2 (treasury)
- `0x0000000000000000000000000000000000000320` → 0.0.800 (staking reward)

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
