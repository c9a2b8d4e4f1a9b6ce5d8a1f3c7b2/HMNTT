### Title
Unbounded Cache Exhaustion via Unique Query Document Flooding in GraphQL Preparsed Document Cache

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed on the raw query string with a hard cap of 1000 entries and a 1-hour TTL. Because the GraphQL module has no rate limiting, no authentication, and no query normalization, any unauthenticated external user can trivially generate 1000+ syntactically distinct but semantically equivalent query strings (e.g., by varying whitespace or inline comments) to continuously evict legitimate cached entries, forcing repeated expensive parse-and-validate cycles on every subsequent request from real users.

### Finding Description

**Exact code path:**

`CachedPreparsedDocumentProvider` ( [1](#0-0) ) builds an `AsyncCache<String, PreparsedDocumentEntry>` from the `CacheProperties` default string `"expireAfterWrite=1h,maximumSize=1000,recordStats"` ( [2](#0-1) ). The cache key is `executionInput.getQuery()` — the **raw, un-normalized query string** ( [3](#0-2) ).

**Root cause:** Two query strings that are semantically identical but textually different (e.g., differing only in whitespace, field aliases, or inline comments) are treated as distinct cache keys. There is no query normalization step before the cache lookup.

**Failed assumption:** The design assumes clients will reuse the same query string across requests (as is typical with persisted queries or static client code). An adversary is not bound by this assumption.

**Exploit flow:**
1. Attacker generates 1001 unique but valid query strings targeting the `account` field, each differing by a single character (e.g., appending `# 1`, `# 2`, … `# 1001` as inline comments, or varying whitespace). All remain within the parser limits (`maxCharacters=10000`, `maxTokens=1000`). ( [4](#0-3) )
2. Each request hits the GraphQL endpoint (`POST /graphql/alpha`) with no authentication required — the graphql module has **no** `ThrottleConfiguration`, `ThrottleManager`, or Spring Security filter. ()
3. After 1000 unique queries, the Caffeine cache is full. The 1001st entry triggers Caffeine's size-based eviction, displacing a legitimate cached entry.
4. By continuously streaming new unique queries, the attacker keeps the cache in permanent churn: every legitimate repeated query from real users misses the cache and triggers a full parse-and-validate cycle.

**Why existing checks are insufficient:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` ( [5](#0-4) ) run **after** parsing; they do not prevent cache pollution.
- Parser limits (`maxCharacters`, `maxTokens`) bound the cost of a single parse but do not prevent an attacker from sending many bounded-cost requests. ( [4](#0-3) )
- The `LoggingFilter` only logs requests; it enforces nothing. ( [6](#0-5) )
- The `ThrottleConfiguration`/`ThrottleManager` exist only in the `web3` module, not in `graphql`. ()
- The `hiero.mirror.graphql.cache.query` property has no documented mechanism to increase `maximumSize` as a defense, and doing so only raises the bar for the attacker proportionally. ( [7](#0-6) )

### Impact Explanation
Every cache miss forces a full GraphQL parse-and-validate cycle. Under sustained attack, **all** requests from legitimate users will miss the cache, increasing CPU consumption and latency proportionally to request volume. Because the GraphQL API is publicly accessible with no authentication or rate limiting, this is a sustained, low-cost degradation attack. The bounded parser limits prevent a single request from being catastrophically expensive, but the aggregate effect across all concurrent legitimate users is meaningful latency regression and CPU exhaustion under load.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only an HTTP client capable of sending POST requests. Generating 1000 unique valid query strings is trivial (a one-line loop appending an incrementing comment). The attack is fully repeatable and stateless — the attacker does not need to maintain any session. The GraphQL endpoint is publicly documented at `/graphql/alpha`. ( [8](#0-7) )

### Recommendation
1. **Add rate limiting to the GraphQL module** — port or replicate the `bucket4j`-based `ThrottleConfiguration` from the `web3` module to the `graphql` module, enforcing a global requests-per-second cap and optionally a per-IP cap.
2. **Normalize query strings before cache lookup** — strip comments and normalize whitespace before using the query as a cache key in `CachedPreparsedDocumentProvider.getDocumentAsync`. This collapses semantically equivalent queries to a single cache entry and eliminates the trivial comment-variation attack vector. ( [9](#0-8) )
3. **Consider persisted queries** — require clients to register query documents by hash and only accept the hash at runtime, making the cache key space finite and controlled.

### Proof of Concept
```bash
# Fill the cache with 1001 unique query strings (each differs by comment)
for i in $(seq 1 1001); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"# $i\\n{ account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }\"}" \
    > /dev/null
done

# Now send a "legitimate" repeated query — it will miss the cache because
# its entry was evicted, forcing a fresh parse-and-validate cycle every time.
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }"}'
```
Repeat the loop continuously to maintain permanent cache churn. Observable via the `recordStats` Caffeine metrics: cache hit rate will drop to near 0% for legitimate queries while the attacker's loop runs.

    private final AsyncCache<String, PreparsedDocumentEntry> cache;

    CachedPreparsedDocumentProvider(CacheProperties properties) {
        cache = Caffeine.from(properties.getQuery()).buildAsync();
    }

    @Override
    public CompletableFuture<PreparsedDocumentEntry> getDocumentAsync(
            ExecutionInput executionInput, Function<ExecutionInput, PreparsedDocumentEntry> parseAndValidateFunction) {
        return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
    }
