### Title
Unbounded In-Memory Accumulation via `memory=true` Opcode Tracing Causes OOM DoS

### Summary
The `/api/v1/contracts/results/{txHash}/opcodes?memory=true` endpoint, when enabled, captures the full EVM memory snapshot at every single opcode step with no size cap on either the per-opcode memory payload or the total accumulated opcode list. A single unauthenticated request replaying a gas-heavy historical transaction can force the JVM to allocate tens of gigabytes of heap, causing an OutOfMemoryError that crashes the mirror-node web3 service and halts its ability to serve any requests.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` accepts `memory=true` from any caller with no authentication: [1](#0-0) 

The only guards are a gzip header check and a global rate-limit token consume: [2](#0-1) 

The rate limit defaults to **1 request per second** globally — it does not prevent a single request from being catastrophic: [3](#0-2) 

For every opcode executed during EVM replay, `tracePostExecution()` calls `captureMemory()`: [4](#0-3) 

`captureMemory()` reads the **entire EVM memory** unconditionally — no size cap, no truncation: [5](#0-4) 

Each captured snapshot is appended to an unbounded `ArrayList<Opcode>`: [6](#0-5) 

**Root cause:** `captureMemory()` has no guard on `frame.memoryWordSize()` and `OpcodeContext.addOpcodes()` has no bound on the list length. The gzip requirement only compresses the HTTP response body; it does not prevent the JVM heap from being filled during construction of the `List<Opcode>` object graph.

**Exploit math:** With the default `maxGasLimit = 15_000_000`: [7](#0-6) 

EVM memory expansion to 32 MB (32,768 words) costs ~2.1 M gas, leaving ~13 M gas for opcodes. At 3 gas/opcode (MSTORE), ~4.3 M opcode steps are possible. Each step captures 32,768 × ~200 bytes (Java `String` object overhead) ≈ 6.5 MB. Total heap demand: **~28 TB** — far beyond any realistic JVM heap, guaranteeing OOM. Even at 1/1000th scale (a few thousand opcodes with a few MB of memory), the heap pressure is gigabytes per request.

### Impact Explanation

An OOM in the web3 service JVM kills the process entirely, halting all contract call, estimate-gas, and opcode endpoints. This is a complete denial of service for the mirror node's EVM API surface. Because the endpoint re-executes a historical transaction (not a new one), the attacker only needs to identify any previously confirmed transaction that consumed significant EVM memory — these are publicly visible on-chain. The impact is rated **Critical** because a single HTTP GET request from an unprivileged user can crash the node.

### Likelihood Explanation

The endpoint is disabled by default (`enabled = false`), which limits exposure to deployments that have explicitly opted in. However, any operator who enables the opcode tracer for debugging or tooling purposes is immediately exposed. The attack requires no credentials, no special knowledge beyond a transaction hash, and is trivially repeatable. The attacker does not need to submit a new transaction; replaying any existing memory-heavy transaction suffices.

### Recommendation

1. **Add a hard cap on `memoryWordSize`** inside `captureMemory()`: if `frame.memoryWordSize()` exceeds a configurable threshold (e.g., 1,024 words / 32 KB), either truncate the snapshot or return an error.
2. **Add a hard cap on total opcode count** in `OpcodeContext.addOpcodes()`: reject or truncate once the list exceeds a configurable maximum (e.g., 100,000 entries).
3. **Add a configurable maximum response size** and abort the replay if the estimated serialized size would exceed it.
4. **Apply a per-request gas budget** specifically for opcode tracing that is lower than `maxGasLimit`, reducing the maximum possible memory × opcode product.

### Proof of Concept

```
# 1. Identify or submit a transaction that allocates large EVM memory
#    (e.g., a contract that runs MSTORE in a loop, consuming ~10M gas)
TX_HASH=0x<hash_of_memory_heavy_transaction>

# 2. Send the opcode trace request with memory=true
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true&stack=false&storage=false"

# 3. Observe: the mirror-node web3 JVM process throws OutOfMemoryError
#    and crashes, making all subsequent requests return connection refused.
```

The server-side OOM occurs during `captureMemory()` accumulation inside the EVM replay loop, before any bytes are written to the HTTP response, so the gzip compression layer provides no protection.

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
