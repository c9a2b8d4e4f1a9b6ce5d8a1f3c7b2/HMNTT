### Title
Unauthenticated Historical Replay DoS via Unbounded Throttled Polling Loop in `PollingTopicMessageRetriever`

### Summary
Any unauthenticated external user can open a `subscribeTopic` gRPC stream with `consensusStartTime` set to epoch (or any arbitrarily old timestamp) and no `limit`, causing `PollingTopicMessageRetriever.poll()` to return exactly `maxPageSize` messages on every iteration. Because `isComplete()` in throttled mode only returns `true` when the last page is smaller than `maxPageSize`, the condition is never satisfied while historical data fills each page, and the repeat loop runs with `numRepeats = Long.MAX_VALUE` until all historical messages are drained. The configured 60-second timeout never fires because items are emitted every 2 seconds (well within the timeout window). Multiple concurrent connections amplify this into a sustained database and gRPC resource exhaustion attack.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `PollingTopicMessageRetriever.retrieve(filter, throttled=true)`

**No authentication or authorization gate exists anywhere in this path.** `ConsensusController` accepts any `ConsensusTopicQuery` from any caller without credentials.

`TopicMessageFilter.isValidStartTime()` only enforces `startTime <= DomainUtils.now()`, so `startTime = 0` (Unix epoch) is accepted:

```java
// TopicMessageFilter.java lines 48-51
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```

In `retrieve()`, throttled mode sets `numRepeats = Long.MAX_VALUE` and `frequency = 2s` (default), then repeats while `!context.isComplete()`:

```java
// PollingTopicMessageRetriever.java lines 51-55
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))
```

`isComplete()` in throttled mode:

```java
// PollingTopicMessageRetriever.java lines 121-129
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;
}
```

`pageSize` is reset to 0 at the start of each `poll()` call (line 73: `context.getPageSize().set(0L)`) and incremented by `onNext()` for each received message. When a topic has ≥ `maxPageSize` (default 1000) historical messages remaining, every poll fills the page exactly: `pageSize = 1000`. The condition `1000 < 1000` is `false`, and with no limit set `limitHit` is also `false`, so `isComplete()` always returns `false`.

The `.timeout(retrieverProperties.getTimeout(), scheduler)` (default 60 s) is a **per-element** timeout in Reactor — it fires only if no item is emitted within the window. Since items are emitted in bursts every 2 seconds (the polling frequency), the timeout never triggers.

`TopicMessageServiceImpl` tracks `subscriberCount` as a metric only; there is no enforcement of a maximum subscriber count for HCS topics (unlike the contract-log design which specifies `maxActiveSubscriptions`).

### Impact Explanation

An attacker targeting a high-volume topic (e.g., the fee-schedule topic 0.0.111 on mainnet, which accumulates years of history) forces the mirror node to:
- Issue a DB query every ~2 seconds, each returning 1000 rows, for the entire duration of the historical replay
- Stream all results over the gRPC connection continuously

With N concurrent connections doing this, the DB connection pool is saturated, query latency spikes for all other users, and gRPC thread/memory resources are exhausted. The subscription then transitions to live mode (no `endTime` set → `pastEndTime()` returns `Flux.never()`) and holds the connection open indefinitely, compounding resource consumption. This is a practical denial-of-service against the mirror node's gRPC API and its backing database.

### Likelihood Explanation

The exploit requires zero privileges, zero credentials, and only a standard gRPC client (e.g., `grpcurl`). The protobuf interface is public and documented. The attacker only needs to know a topic ID with significant history — the fee-schedule topic ID is well-known and publicly documented. The attack is trivially repeatable and scriptable, and can be amplified by opening many concurrent connections from a single host or a botnet.

### Recommendation

1. **Enforce a maximum concurrent subscriber count** in `TopicMessageServiceImpl.subscribeTopic()`, rejecting new subscriptions when the count exceeds a configurable threshold (analogous to `maxActiveSubscriptions` in the contract-log design).
2. **Cap the historical replay window**: reject or truncate `startTime` values older than a configurable maximum lookback (e.g., 7 days), or require a non-zero `limit` when `startTime` is in the distant past.
3. **Apply a hard wall-clock timeout** on the entire subscription stream (not just per-element), so that long-running historical replays are forcibly terminated regardless of emission rate.
4. **Add per-IP or per-client connection rate limiting** at the gRPC ingress layer (e.g., via Traefik middleware already present in the Helm chart).
5. In `isComplete()`, consider also returning `true` when `total` exceeds a configurable absolute cap, independent of page fullness.

### Proof of Concept

```bash
# Install grpcurl, then run against a mirror node with a high-volume topic:
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 111},
    "consensusStartTime": {"seconds": 0, "nanos": 0}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Steps:**
1. Send the above request with no `limit` and `consensusStartTime` at epoch 0.
2. `TopicMessageFilter.isValidStartTime()` passes (0 ≤ now).
3. `PollingContext` is created with `throttled=true`, `numRepeats=Long.MAX_VALUE`, `maxPageSize=1000`.
4. Each `poll()` call queries the DB for 1000 messages starting from the oldest available timestamp.
5. `isComplete()` evaluates `1000 < 1000 || false` → `false` after every poll.
6. The loop repeats every ~2 seconds, streaming 1000 messages per iteration.
7. Open 50+ concurrent connections to exhaust the DB connection pool and degrade service for all other users.
