### Title
Null Decimals Exported for Deleted Tokens in `getTokenRelationships` Due to Missing Fallback in `getTokenAccounts`

### Summary
When an account has an active association (`associated = true`) with a token that has been deleted from the `token` table and is not present in the in-memory LRU cache, `getTokenAccounts` never sets `row.decimals`, causing `TokenRelationshipViewModel` to export `decimals: null`. This is an incorrect representation because the token had a concrete decimal value when it existed, and any unprivileged user can trigger this condition by querying the `/accounts/{id}/tokens` endpoint for such an account.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js` → `getTokenRelationships()` (line 74–75) calls `TokenService.getTokenAccounts(query)` and maps each result directly to `new TokenRelationshipViewModel(token)`.

`rest/service/tokenService.js` → `getTokenAccounts()` (lines 96–115):

```js
// tokenRelationshipsQuery does NOT select decimals from token_account
// so row.decimals is undefined after the DB fetch
const rows = await super.getRows(sqlQuery, params);
const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
const cachedTokens = await this.getCachedTokens(tokenIds);
return rows.map((row) => {
  const cachedToken = cachedTokens.get(row.token_id);
  if (cachedToken) {                          // ← only branch that sets decimals
    row.decimals = cachedToken.decimals;
    ...
  }
  // if cachedToken is undefined, row.decimals is never assigned
  return new TokenAccount(row);
});
```

`getCachedTokens()` (lines 136–161) queries `select decimals, ... from token where token_id = any ($1)`. If the token has been deleted from the `token` table and is not in the LRU cache, the row is absent from the result, `cachedTokens.get(row.token_id)` returns `undefined`, the `if (cachedToken)` branch is skipped, and `row.decimals` remains `undefined`.

`rest/viewmodel/tokenRelationshipViewModel.js` line 20:
```js
this.decimals = tokenRelationship.decimals;  // propagates undefined → null in JSON
```

**Root cause:** The `tokenRelationshipsQuery` intentionally omits `decimals` from the `token_account` table (it is not stored there), relying entirely on the cache/token-table lookup. There is no fallback when that lookup yields nothing, so the field silently becomes `null` in the API response.

**Failed assumption:** The code assumes every token referenced in `token_account` will always be resolvable via the cache or the `token` table. This assumption breaks for deleted tokens whose `token_account` rows survive with `associated = true`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any API consumer (wallet, exchange, DeFi protocol) that reads `decimals` from `/accounts/{id}/tokens` and receives `null` for a deleted-but-still-associated token will either crash, display incorrect balances, or silently compute wrong amounts (e.g., treating a 6-decimal token as having 0 decimals). The mirror node is the authoritative read layer; exporting `null` for a field that had a known, immutable value is a data-integrity violation that propagates to all downstream consumers.

### Likelihood Explanation
Token deletion is a normal Hedera network operation. Any account that held an association with a subsequently deleted token is permanently in this state. An unprivileged attacker needs only to:
1. Know (or discover by enumeration) an account ID that was associated with a deleted token.
2. Issue a plain HTTP GET to `/api/v1/accounts/{id}/tokens`.

No authentication, no special privileges, no rate-limit bypass required. The condition is permanent and repeatable.

### Recommendation
In `getTokenAccounts`, add an explicit fallback when `cachedToken` is absent. Options:

1. **Preserve last-known decimals in `token_account`**: Store `decimals` in the `token_account` table at association time and include it in `tokenRelationshipsQuery`. This is the most robust fix because it survives token deletion.
2. **Soft-delete guard**: If `cachedToken` is `undefined` after the lookup, set `row.decimals` to a sentinel (e.g., `0`) and log a warning, rather than leaving it `undefined`.
3. **Filter deleted tokens**: Extend `tokenRelationshipsQuery` to join against the `token` table and exclude rows where the token no longer exists, so they are never returned to the caller.

### Proof of Concept
1. Create token T with `decimals = 6` on a Hedera testnet.
2. Associate account A with token T.
3. Delete token T (admin key required, but this is a normal network operation, not an attacker action).
4. As an unprivileged user, send:
   ```
   GET /api/v1/accounts/{A}/tokens
   ```
5. Observe the response contains:
   ```json
   { "token_id": "0.0.T", "decimals": null, ... }
   ```
   instead of `"decimals": 6`. [4](#0-3) [5](#0-4)

```javascript
  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
```
