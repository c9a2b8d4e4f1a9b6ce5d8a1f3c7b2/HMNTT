### Title
Global Opcode Rate-Limit Bucket Allows Single Unprivileged User to Monopolize the Entire 1 RPS Budget

### Summary
The `opcodeRateLimitBucket` is a single JVM-wide singleton `Bucket` with a hard-coded default capacity of 1 token per second and no per-IP or per-user partitioning. Any unauthenticated caller who sends one request per second to `GET /api/v1/contracts/results/{id}/opcodes` will continuously consume the only available token, causing every concurrent or subsequent request from any other user to receive HTTP 429 for as long as the attacker keeps the loop running.

### Finding Description
**Exact code path:**

- `ThrottleProperties.java` line 29: `private long opcodeRequestsPerSecond = 1;` — the default global budget is 1 token/second. [1](#0-0) 

- `ThrottleConfiguration.java` lines 47–55: `opcodeRateLimitBucket()` is a Spring `@Bean` (singleton scope) that builds one shared `Bucket` with `capacity(rateLimit)` and `refillGreedy(rateLimit, Duration.ofSeconds(1))`. No `SynchronizationStrategy` is set (unlike `gasLimitBucket`), and — critically — no per-caller partitioning exists. [2](#0-1) 

- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` against that single shared bucket. If the token is gone, a `ThrottleException` is thrown immediately. [3](#0-2) 

- `OpcodesController.java` line 61: every incoming request hits `throttleManager.throttleOpcodeRequest()` before any processing. [4](#0-3) 

**Root cause / failed assumption:** The design assumes the 1 RPS global ceiling is sufficient to protect the endpoint, but it conflates *system-wide* throughput protection with *per-user* fairness. With `capacity=1` and `refillGreedy(1, 1s)`, the bucket holds exactly one token at any moment. A single caller who times requests at ≥1-second intervals will always find a token available and will never be throttled, while every other caller racing for the same token will be rejected.

**Why existing checks fail:** There is no IP-based, session-based, or API-key-based sub-bucket. The `request` filter list in `ThrottleProperties` applies only to the general `ContractCallRequest` path (`throttle()`), not to `throttleOpcodeRequest()`. [5](#0-4) 

### Impact Explanation
The opcodes endpoint re-executes EVM transactions and is intentionally restricted to 1 RPS because it is computationally expensive. A single attacker holding that 1 RPS slot indefinitely means **zero legitimate users** can reach the endpoint. This is a complete, sustained denial-of-service of a high-value debugging/auditing feature, achievable with a trivial HTTP client loop and no credentials.

### Likelihood Explanation
Preconditions: none — the endpoint is publicly reachable, no authentication is required, and the attacker needs only a standard HTTP client. The attack is trivially scriptable (`while true; do curl ...; sleep 1; done`), requires no special knowledge, and is fully repeatable. The 1-second cadence is easy to maintain with any scripting language or tool.

### Recommendation
Replace the single global bucket with per-caller (IP or API-key) buckets, for example using Bucket4j's `ProxyManager` keyed on the remote address extracted from `HttpServletRequest`. Alternatively, raise `opcodeRequestsPerSecond` to a value that allows fair sharing (e.g., N × expected-concurrent-users) while adding a per-IP sub-limit so no single caller can consume more than their fair share. At minimum, add IP-level rate limiting at the reverse-proxy/ingress layer in front of this endpoint.

### Proof of Concept
```bash
# Attacker terminal – runs indefinitely, consuming every refilled token
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node-host>/api/v1/contracts/results/<txHash>/opcodes"
  sleep 1
done
# Expected: attacker always receives 200

# Victim terminal – any concurrent request loses the race
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/<txHash>/opcodes"
# Expected: HTTP 429 "Requests per second rate limit exceeded"
```

The attacker loop consumes the single token the instant it refills every second. Because `tryConsume(1)` is non-blocking and the bucket holds at most 1 token, any request that arrives even milliseconds after the attacker's will find the bucket empty and receive 429. [2](#0-1) [1](#0-0)

    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
