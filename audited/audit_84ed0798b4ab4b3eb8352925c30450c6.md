### Title
Unauthenticated `/metrics` Endpoint Exposes `status_code` Label Dimension of `requestDurationHistogram`, Enabling Construction Endpoint Error Enumeration

### Summary
The Rosetta service's `/metrics` endpoint is registered via `promhttp.Handler()` with zero authentication, unlike the REST API's equivalent which enforces Basic Auth. Any unprivileged external user can `GET /metrics` and read the `hiero_mirror_rosetta_request_duration` histogram's `status_code` label dimension, which records the HTTP status code for every route including all `/construction/*` endpoints. This allows systematic enumeration of which inputs trigger which error codes, mapping out validation logic without any credentials.

### Finding Description
**Exact code path:**

In `rosetta/app/middleware/metrics.go`, `Routes()` registers the `/metrics` endpoint directly with `promhttp.Handler().ServeHTTP` — no authentication wrapper of any kind:

```go
// Lines 64-73
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,
            promhttp.Handler().ServeHTTP,  // bare handler, no auth
        },
    }
}
```

The `requestDurationHistogram` is declared with `status_code` as a label:

```go
// Lines 28-32
requestDurationHistogram = prometheus.NewHistogramVec(prometheus.HistogramOpts{
    Name:    "hiero_mirror_rosetta_request_duration",
    Buckets: []float64{.1, .25, .5, 1, 2.5, 5},
    Help:    "Time (in seconds) spent serving HTTP requests.",
}, []string{"method", "route", "status_code", "ws"})
```

`MetricsMiddleware` wraps the entire router (including all construction API controllers) and records `status_code` per route:

```go
// rosetta/main.go lines 217-219
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

The router includes `constructionAPIController` (both online and offline modes), so every `/construction/preprocess`, `/construction/metadata`, `/construction/payloads`, `/construction/combine`, etc. call is instrumented with its `status_code`.

**Root cause:** The failed assumption is that `/metrics` is an internal-only scrape endpoint. No authentication middleware exists anywhere in the rosetta server chain (no IP allowlist, no Basic Auth, no token check). The REST API (`rest/middleware/metricsHandler.js`) correctly implements an `authenticate()` guard that returns 401 when `config.metrics.config.authentication` is true — the Rosetta service has no equivalent.

**Exploit flow:**
1. Attacker sends crafted requests to `/construction/payloads` with varying inputs (invalid account IDs, malformed operations, wrong network identifiers, etc.)
2. Each response status code (200, 400, 422, 500) is recorded in `hiero_mirror_rosetta_request_duration{route="/construction/payloads", status_code="422"}` etc.
3. Attacker immediately `GET /metrics` (no credentials needed) and reads the histogram label set
4. By correlating which inputs produced which `status_code` values, the attacker maps the full validation decision tree of the construction pipeline

### Impact Explanation
An attacker learns the exact error-code boundaries of construction endpoint validation logic — which inputs are rejected at which layer (asserter, service, constructor), and which produce 5xx vs 4xx. This is reconnaissance that directly informs crafting of malformed-but-accepted construction requests. While no funds are directly at risk from the metrics read itself, the information materially reduces the effort required to find exploitable edge cases in transaction construction logic (e.g., inputs that bypass validation but produce unintended transaction semantics).

### Likelihood Explanation
Exploitation requires only network access to the Rosetta port (default 5700) and a single unauthenticated HTTP GET. No credentials, no special tooling, no timing constraints. The Rosetta port is exposed via the nginx proxy (`location /rosetta/`) in the default docker-compose configuration, making it reachable from the internet in typical deployments. Repeatability is unlimited — the attacker can poll `/metrics` continuously.

### Recommendation
Wrap the `promhttp.Handler()` in an authentication middleware before registering it, mirroring the pattern used in `rest/middleware/metricsHandler.js`:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,
            basicAuthMiddleware(promhttp.Handler()).ServeHTTP,
        },
    }
}
```

Alternatively, move the `/metrics` endpoint to a separate, non-public port (e.g., a dedicated admin/scrape port not exposed through the nginx proxy), which is the standard Prometheus operator pattern. At minimum, add an IP allowlist restricting access to known Prometheus scraper IPs.

### Proof of Concept
```bash
# Step 1: Trigger construction endpoint with invalid input to generate a non-200 status_code label
curl -s -X POST http://<rosetta-host>:5700/construction/preprocess \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hiero","network":"mainnet"},"operations":[],"metadata":{}}'

# Step 2: Read metrics with no credentials — succeeds with HTTP 200
curl -s http://<rosetta-host>:5700/metrics | grep 'hiero_mirror_rosetta_request_duration'

# Expected output includes lines like:
# hiero_mirror_rosetta_request_duration_bucket{application="rosetta",method="POST",route="/construction/preprocess",status_code="400",ws="false",le="0.1"} 1
# hiero_mirror_rosetta_request_duration_bucket{application="rosetta",method="POST",route="/construction/preprocess",status_code="422",ws="false",le="0.1"} 3
# ...
# By varying inputs and re-reading /metrics, the full status_code label space per route is enumerated.
