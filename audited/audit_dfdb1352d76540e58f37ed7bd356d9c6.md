### Title
Partial Node Exclusion via Port == 0 in Mirror REST Response Allows Silent Reduction of Pinger's Network View

### Summary
In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()` silently skips any `serviceEndpoint` where `Port == 0`. The only guard against manipulation is a check that the resulting `network` map is non-empty. An attacker who can control the mirror REST response (e.g., via HTTP MITM) can return valid `nodeEntry` records for ≥30% of nodes with all their `serviceEndpoints` having `Port == 0`, causing those nodes to be silently dropped from the pinger's network map while the function returns success with the remaining nodes.

### Finding Description
**Exact code path:** `pinger/mirror_node_client.go`, lines 113–124 and 127–129.

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← silent skip
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
// ...
if len(network) == 0 {               // ← only total-exclusion guard
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

**Root cause:** The function applies a per-endpoint filter (`Port == 0` → skip) but has no per-node or aggregate threshold check. If a node's every endpoint is skipped, that node is silently absent from the returned map. The sole safety net (`len(network) == 0`) only fires when every node across the entire response is excluded.

**Exploit flow:**
1. Attacker intercepts the plain-HTTP request to the mirror REST API (configured via `HIERO_MIRROR_PINGER_REST`, example value `http://mirror-rest:5551`).
2. Attacker returns a crafted JSON response where ≥30% of `nodeEntry` records have all `serviceEndpoints` with `"port": 0` (but valid `domain_name`/`ip_address_v4` and a valid `node_account_id`).
3. `fetchMirrorNodeNetwork` skips those endpoints, those nodes are never added to `network`.
4. `len(network) > 0` (remaining nodes are present), so no error is returned.
5. `hiero.ClientForNetwork(netmap)` is called with the reduced map; the pinger only routes transactions to the surviving nodes.

**Why existing checks fail:** The `len(network) == 0` guard at line 127 is the only protection. It is bypassed as long as at least one node survives, regardless of how many were silently dropped. [1](#0-0) 

### Impact Explanation
The pinger's Hiero SDK client is built exclusively from the returned `network` map. Nodes absent from that map receive no transactions from the pinger. For a deployment using `network=other` (the only mode that calls this function), an attacker can blind the pinger to ≥30% of consensus nodes: those nodes appear healthy to the pinger (no errors, no alerts) while actually being unmonitored. This satisfies the "≥30% of network processing nodes silently excluded from monitoring without shutting down the network" impact category. [2](#0-1) 

### Likelihood Explanation
The precondition is controlling the mirror REST response. The configured URL uses plain HTTP (`http://mirror-node:5551` per the example), making the response trivially interceptable by any on-path attacker (ARP spoofing, rogue DHCP, compromised network device, or DNS poisoning). No credentials or privileged access to the pinger process are required. The attack is repeatable on every pinger restart or network refresh cycle. [3](#0-2) 

### Recommendation
1. **Enforce HTTPS** for `mirrorRest` and validate the TLS certificate to eliminate MITM as an attack vector.
2. **Add a minimum-node threshold check**: after building the `network` map, compare the number of distinct `AccountID` values added against the total number of valid `nodeEntry` records in the response. If the ratio of included nodes falls below a configurable threshold (e.g., 80%), return an error and retry.
3. **Log and alert on skipped nodes**: emit a warning for every `nodeEntry` whose every endpoint was skipped, so operators can detect manipulation.

### Proof of Concept
Assume a 10-node network. Attacker intercepts the HTTP GET to `/api/v1/network/nodes` and returns:

```json
{
  "nodes": [
    {"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"node3.example.com","port":0}]},
    {"node_account_id":"0.0.4","service_endpoints":[{"domain_name":"node4.example.com","port":0}]},
    {"node_account_id":"0.0.5","service_endpoints":[{"domain_name":"node5.example.com","port":0}]},
    {"node_account_id":"0.0.6","service_endpoints":[{"domain_name":"node6.example.com","port":50211}]},
    ...7 more legitimate nodes...
  ]
}
```

Nodes `0.0.3`, `0.0.4`, `0.0.5` (30%) have `port: 0` → all their endpoints are skipped → they are absent from the returned map. `len(network) > 0` → no error. The pinger starts and never sends transactions to those three nodes. [4](#0-3)
