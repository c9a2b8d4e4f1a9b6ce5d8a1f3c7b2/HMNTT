### Title
Unbounded Historical State Scan DoS via Multiple `slot` Parameters with Timestamp Filter in `getContractStateById()`

### Summary
`GET /contracts/:contractId/state` accepts an arbitrary number of `slot=` query parameters (bounded only by the generic `maxRepeatedQueryParameters` config, not by any slot-specific limit) combined with a `timestamp=lte:T` filter. When a timestamp is present, `extractContractStateByIdQuery()` routes the request through `contractStateTimestampQuery`, which executes a `DISTINCT ON (slot) … ORDER BY slot, consensus_timestamp DESC` scan over the append-only `contract_state_change` table. This sort order does not align with the primary key `(consensus_timestamp, contract_id, slot)`, forcing a full sequential scan and sort for every request. Any unauthenticated caller can trigger this repeatedly to exhaust database CPU/IO.

### Finding Description

**Code path:**

1. `rest/routes/contractRoute.js` line 15 — `router.getExt('/:contractId/state', ContractController.getContractStateById)` — no authentication middleware.

2. `rest/controllers/contractController.js` lines 964–994 — `getContractStateById()` calls `extractContractIdAndFiltersFromValidatedRequest()` → `utils.buildAndValidateFilters()`.

3. `rest/utils.js` lines 1234–1259 — `buildFilters()` checks `isRepeatedQueryParameterValidLength(values)` against `config.query.maxRepeatedQueryParameters`. This is the **only** guard on slot count. It is a generic, configurable limit (default observed in tests as ≥ 4; production defaults are higher), not a slot-specific or timestamp-path-specific limit.

4. `rest/controllers/contractController.js` lines 894–956 — `extractContractStateByIdQuery()`:
   - Line 921: every `slot` filter is pushed into `slotFilters` with no count check beyond what `buildFilters` already passed.
   - Line 933: `utils.formatSlot(slotFilter.value, !timestampPresent)` — when `timestampPresent=true`, `leftPad=false`, triggering the timestamp-mode conversion path for every slot.
   - Lines 934–935: all `eq`-operator slots are accumulated into `slotInValues`.
   - Lines 946–948: a single `slot IN ($1, $2, …, $N)` condition is appended.
   - Line 954: `timestamp: true` is returned, routing to the expensive query.

5. `rest/service/contractService.js` lines 256–263 — when `timestamp=true`, `contractStateTimestampQuery` is used instead of `contractStateQuery`.

6. `rest/service/contractService.js` lines 111–122 — `contractStateTimestampQuery`:
```sql
SELECT DISTINCT ON (slot)
  contract_id, slot, evm_address,
  coalesce(value_written, value_read) as value,
  consensus_timestamp as modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
[WHERE contract_id = X AND slot IN (...) AND consensus_timestamp <= T]
ORDER BY slot, consensus_timestamp DESC
LIMIT $N
```

**Root cause:** The `ORDER BY slot, consensus_timestamp DESC` required by `DISTINCT ON (slot)` is incompatible with the primary key index `(consensus_timestamp, contract_id, slot)`. PostgreSQL cannot satisfy this sort using the index and must perform a full heap scan + sort of all matching rows before applying `LIMIT`. The `LIMIT` clause does not reduce scan cost because `DISTINCT ON` requires the full sorted set to be materialized first. The `contract_state_change` table is append-only and grows without bound.

**Why the existing check fails:** `isRepeatedQueryParameterValidLength` is a generic guard. Even at a conservative default of 25, an attacker can send 25 slot values with `timestamp=lte:99999999999` (far-future timestamp = all rows), forcing the database to scan and sort the entire `contract_state_change` history for that contract for every request.

### Impact Explanation
A single request with `maxRepeatedQueryParameters` slot values and a far-future timestamp forces a full sequential scan + sort of `contract_state_change` for the targeted contract. For a high-activity contract (millions of state-change rows, realistic for DeFi protocols), this query can take seconds to minutes of CPU and I/O. Sending a modest flood of such requests (e.g., 10–50 concurrent) can saturate the database connection pool and degrade or deny service for all mirror-node API consumers. The `contract_state_change` table is never pruned by default, so impact worsens over time as the chain grows.

### Likelihood Explanation
No authentication, API key, or rate limiting is required. The attack requires only knowledge of any valid `contractId` (trivially discoverable via `/api/v1/contracts`). The exploit is a single HTTP GET request, easily scripted, and repeatable at high frequency. The attacker needs no privileged access and no on-chain funds.

### Recommendation
1. **Add a slot-specific limit for the timestamp path:** In `extractContractStateByIdQuery()`, when `timestampPresent` is true and `slotInValues.length > 0`, enforce a hard cap (e.g., 5–10) independent of `maxRepeatedQueryParameters`.
2. **Add a composite index** on `contract_state_change (contract_id, slot, consensus_timestamp DESC)` to allow the `DISTINCT ON` query to use an index scan instead of a full sort.
3. **Apply rate limiting** per IP/client on the `/contracts/:contractId/state` endpoint, especially when a `timestamp` filter is present.
4. **Consider a query timeout** at the database layer for this endpoint.
