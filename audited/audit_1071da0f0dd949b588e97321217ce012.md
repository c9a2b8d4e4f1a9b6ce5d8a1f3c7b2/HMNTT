### Title
Unauthenticated DB Connection Exhaustion via Entity-Lookup Bypass and UNION ALL Query Amplification in `getAccountTokenAllowances`

### Summary
An unprivileged attacker can craft requests using a `shard.realm.num` account ID format to bypass the entity existence DB lookup entirely, then supply spender_id and token_id range filters that force the service to build a three-part `UNION ALL` query, each branch carrying its own `LIMIT` clause. With no rate limiting present in the controller layer, high-concurrency flooding of this endpoint exhausts the shared DB connection pool, rendering the mirror node API unavailable.

### Finding Description
**Entity lookup bypass** — In `rest/service/entityService.js`, `getEncodedId()` (lines 120–123) checks `EntityId.isValidEntityId()` first; if the input is a `shard.realm.num` string and `evmAddress === null`, it calls `entityId.getEncodedId()` and returns immediately with **zero DB round-trips**. Alias and EVM-address inputs each require a DB query; `shard.realm.num` does not. This means an attacker using `0.0.12345` skips the entity-existence query entirely, reducing per-request latency and allowing higher sustained request throughput.

**Three-part UNION ALL amplification** — In `rest/service/tokenAllowanceService.js`, `getQuery()` (lines 63–74) builds up to three subqueries (`lower`, `inner`, `upper`) joined with `UNION ALL`. Each subquery receives its own `LIMIT $2` clause (line 58: `super.getLimitQuery(2)`), and the outer wrapper adds another `ORDER BY … LIMIT $2`. Triggering all three parts (by supplying both `spender_id` and `token_id` as range bounds) causes the DB to execute three full scans of `token_allowance`, each bounded only by the maximum limit, before merging results. This multiplies DB work by ~3× per request.

**No rate limiting in the controller** — `getAccountTokenAllowances` (lines 68–81 of `tokenAllowanceController.js`) calls `buildAndValidateFilters` for input validation and then directly invokes the service. No per-IP, per-account, or global rate-limiting middleware is applied at this layer. The `acceptedTokenAllowanceParameters` set (lines 84–89) permits `limit` as a free parameter, and the limit value is passed through to the query without a visible server-side cap enforced in the controller.

**Combined exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0.0.1/allowances/tokens?spender.id=gte:0.0.1&spender.id=lte:0.0.9999&token.id=gte:0.0.1&token.id=lte:0.0.9999&limit=100` (or whatever the configured max).
2. `getEncodedId("0.0.1")` returns immediately — no DB query.
3. `extractTokenMultiUnionQuery` produces non-empty `lower`, `inner`, and `upper` filter sets.
4. `getQuery` emits a three-branch `UNION ALL` SQL statement, each branch with `LIMIT $2`.
5. DB executes three sequential scans; connection is held for the full duration.
6. Attacker floods with hundreds of concurrent identical requests; each holds a DB connection for the duration of three scans.
7. Connection pool saturates; all subsequent requests queue or fail.

### Impact Explanation
Exhausting the shared PostgreSQL connection pool makes the mirror node REST API entirely unresponsive. All downstream consumers (wallets, explorers, dApps) lose the ability to query balances, transactions, and allowances. The mirror node is a critical read infrastructure component; its unavailability degrades the usability of the network for all participants. The impact is mirror node API unavailability — not consensus-layer transaction confirmation, which operates independently — but the practical effect on end-user and application access is severe.

### Likelihood Explanation
No authentication or API key is required. The `shard.realm.num` format is publicly documented and trivially constructed. The three-part UNION ALL is triggered by standard, documented query parameters. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain hundreds of concurrent requests. The attack is fully repeatable and requires no special knowledge beyond reading the public API documentation.

### Recommendation
1. **Rate limiting**: Apply per-IP (and optionally per-account-ID) rate limiting middleware before the controller, e.g., using `express-rate-limit` or an API gateway policy.
2. **Connection pool guard**: Set a `statement_timeout` and `lock_timeout` in the DB pool config to bound how long any single query can hold a connection.
3. **Entity existence check**: For `shard.realm.num` inputs, consider performing a lightweight existence check (the existing `entityExistenceQuery` in `entityService.js` line 28) rather than skipping it entirely, so that requests for non-existent accounts are rejected cheaply before reaching the heavy UNION ALL path.
4. **Max limit enforcement**: Enforce a hard server-side cap on the `limit` parameter (e.g., 25–50) to reduce per-query DB work regardless of what the client supplies.

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${TokenAllowance.OWNER} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...TokenAllowanceService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery([], params, accountIdCondition, limitClause, orderClause);
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }
