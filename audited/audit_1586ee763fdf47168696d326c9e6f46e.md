### Title
Unauthenticated Sequential DB Query Amplification Enabling Connection Pool Exhaustion in `getBalances()`

### Summary
The `getBalances()` handler in `rest/balances.js` issues up to three sequential `pool.queryQuietly` calls per request when both a `timestamp` and `account.publickey` filter are supplied. The REST Node.js service has no per-IP or global request rate limiting for unauthenticated users, allowing any external caller to flood the endpoint with concurrent requests, each holding a pool connection across three sequential round-trips, exhausting the finite connection pool and causing service unavailability for all users.

### Finding Description

**Code path** — `rest/balances.js`, `getBalances()`, lines 113–153:

```
if (tsQuery) {
  const tsQueryResult = await getTsQuery(tsQuery, tsParams);        // DB call 1 (line 114)
  ...
  const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit); // DB call 2 (line 119)
  ...
  [sqlQuery, tsParams] = await getBalancesQuery(...);
}
...
const result = await pool.queryQuietly(pgSqlQuery, sqlParams);      // DB call 3 (line 153)
```

- **DB call 1**: `getTsQuery` → `getAccountBalanceTimestampRange` → `pool.queryQuietly` (line 189 of `balances.js`)
- **DB call 2**: `getAccountIdsByPublicKey` → `pool.queryQuietly` (line 66 of `balances.js`); triggered only when `account.publickey` is present
- **DB call 3**: main balance query, always executed (line 153)

Each `pool.queryQuietly` call acquires a connection from the shared `pg` pool (`rest/dbpool.js`, `max: config.db.pool.maxConnections`), executes the query, then releases it. Because the three calls are sequential `await`s, a single request occupies a pool connection for three consecutive round-trips, tripling the effective connection-hold time compared to a single-query endpoint.

**Root cause**: No rate-limiting middleware exists for unauthenticated callers on the REST Node.js service. A grep across `rest/**/*.js` for `rateLimiter`, `rateLimit`, `throttle`, and `express-rate` returns only one hit in a test utility file — none in production middleware. The `authHandler` middleware (`rest/middleware/authHandler.js`) only sets a custom *response-size* limit for authenticated users; it does not throttle request rate for anonymous callers.

**Exploit flow**:
1. Attacker sends `C` concurrent HTTP requests (where `C` ≥ `maxConnections`):
   ```
   GET /api/v1/balances?timestamp=lte:9999999999&account.publickey=<valid_64_hex>&limit=100
   ```
2. Each request enters the `if (tsQuery)` branch and begins three sequential `pool.queryQuietly` calls.
3. With `C` concurrent requests each holding a connection for the duration of three sequential queries, the pool is saturated.
4. Subsequent legitimate requests block waiting for `connectionTimeoutMillis` and then fail, returning errors to real users.
5. Attacker repeats continuously; `statement_timeout` releases connections eventually, but the attacker re-floods before the pool recovers.

### Impact Explanation
When the pool is exhausted, all endpoints sharing the same `global.pool` instance return connection-timeout errors. The service becomes effectively unavailable to legitimate users for the duration of the attack. Because Node.js is single-threaded and the pool is global, this affects the entire REST API, not just `/api/v1/balances`. The `statement_timeout` (`rest/dbpool.js` line 15) bounds individual query duration but does not prevent the attacker from continuously re-saturating the pool faster than connections are released.

### Likelihood Explanation
No authentication, no CAPTCHA, no IP-based rate limit, and no concurrency cap is required. Any internet-accessible deployment is reachable. The required parameters (`timestamp`, `account.publickey`) are documented public API features. A valid public key can be any 64-hex-character string that passes `isValidPublicKeyQuery`; even a key that matches zero accounts still triggers all three DB calls (the second call returns `null` only after executing the query). The attack is trivially scriptable with `curl`, `ab`, or any HTTP load tool.

### Recommendation
1. **Add per-IP rate limiting** to the REST Express server (e.g., `express-rate-limit`) for all unauthenticated callers, applied globally in `rest/server.js` before route handlers.
2. **Add a global concurrency cap** on in-flight DB-bound requests (e.g., a semaphore or queue) so that pool exhaustion cannot be triggered regardless of request rate.
3. **Reduce the number of sequential round-trips**: merge the `getAccountBalanceTimestampRange` lookup and the `getAccountIdsByPublicKey` lookup into a single query or a parallel `Promise.all`, so each request holds a connection for a shorter total window.
4. **Lower `maxConnections`** is not a fix — it only lowers the threshold at which exhaustion occurs.

const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
