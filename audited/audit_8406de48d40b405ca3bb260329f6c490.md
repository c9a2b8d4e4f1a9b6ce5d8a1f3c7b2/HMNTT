### Title
Long Overflow in `getInclusiveValue()` Causes Semantic Bypass: `GT:Long.MAX_VALUE` Returns All Nodes Instead of Empty Set

### Summary
`NumberRangeParameter.getInclusiveValue()` performs unchecked arithmetic (`value + 1`) on a user-supplied `long`. When an unprivileged user supplies `node.id=GT:9223372036854775807` (Long.MAX_VALUE), the addition overflows to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` clamps `lowerBound` back to `0`, so the guard `lowerBound > upperBound` never fires, and the repository query executes with `lowerBound=0, upperBound=Long.MAX_VALUE` — returning every node in the table instead of the semantically correct empty result.

### Finding Description

**Overflow site:** [1](#0-0) 

`getInclusiveValue()` with `operator == GT` returns `value + 1`. Java `long` arithmetic wraps silently; `Long.MAX_VALUE + 1 == Long.MIN_VALUE`.

**Input validation — no upper-bound check:** [2](#0-1) 

`getNumberValue` only rejects negative values. `Long.MAX_VALUE` (9223372036854775807) passes cleanly.

**Consumption in `getNetworkNodes`:** [3](#0-2) 

Step-by-step:
1. `lowerBound` initialised to `0L`.
2. For `GT:9223372036854775807`, `hasLowerBound()` is true → `lowerBound = Math.max(0L, Long.MIN_VALUE)` = **`0L`** (overflow value is negative, so `Math.max` clamps it to 0).
3. `upperBound` stays at `Long.MAX_VALUE`.
4. Guard `lowerBound > upperBound` → `0 > Long.MAX_VALUE` → **false** → no exception.
5. Query fires with `lowerBound=0, upperBound=Long.MAX_VALUE` — a full-table scan returning all nodes.

The semantically correct result for "nodes with id strictly greater than Long.MAX_VALUE" is an **empty set**. The overflow silently converts that into "all nodes."

### Impact Explanation
- **Data exposure:** All network node records are returned to the caller. While network node data is generally public, the caller bypasses the intended filter and receives a result set they should not receive (an empty one).
- **Resource exhaustion / griefing:** Any unauthenticated client can repeatedly issue this single crafted request to force a full-table scan on the `address_book_entry` / node tables, consuming database CPU and I/O with no rate-limiting benefit from the filter predicate. This maps directly to the declared scope: *"griefing with no economic damage."*
- Severity: **Medium** (DoS/griefing vector; no privilege required; no data confidentiality breach beyond public data).

### Likelihood Explanation
- Requires zero privileges — any HTTP client can reach the `/api/v1/network/nodes` endpoint.
- The payload is a single, trivially constructed query parameter: `?node.id=gt:9223372036854775807`.
- Fully repeatable and scriptable; no authentication, no rate-limit bypass needed beyond what the API already allows.

### Recommendation
Add an explicit upper-bound check in `getNumberValue` or in `getInclusiveValue`:

```java
// Option A: reject Long.MAX_VALUE for GT/LT operators before arithmetic
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        if (value == Long.MAX_VALUE) {
            throw new IllegalArgumentException("node.id GT Long.MAX_VALUE produces an empty range");
        }
        return value + 1;
    } else if (operator == RangeOperator.LT) {
        if (value == 0) {
            throw new IllegalArgumentException("node.id LT 0 produces an empty range");
        }
        return value - 1;
    }
    return value;
}
```

Or use `Math.addExact` / `Math.subtractExact` which throw `ArithmeticException` on overflow, then map that to a 400 Bad Request.

Additionally, add a domain-level maximum for `node.id` (e.g., reject values above a known maximum node count) in `getNumberValue`.

### Proof of Concept

```
GET /api/v1/network/nodes?node.id=gt:9223372036854775807
```

Expected: HTTP 200 with empty `nodes` array (no node id can exceed Long.MAX_VALUE).  
Actual: HTTP 200 with **all** network nodes returned, because:

1. `Long.parseLong("9223372036854775807")` → `Long.MAX_VALUE` ✓ (passes `< 0` check)
2. `getInclusiveValue()`: `Long.MAX_VALUE + 1` → `Long.MIN_VALUE` (silent overflow)
3. `Math.max(0L, Long.MIN_VALUE)` → `0L` (lowerBound reset to 0)
4. `0L > Long.MAX_VALUE` → `false` (guard bypassed)
5. Repository called with `lowerBound=0, upperBound=Long.MAX_VALUE` → full table returned

Repeat the request in a tight loop to trigger repeated full-table scans for griefing/DoS.
