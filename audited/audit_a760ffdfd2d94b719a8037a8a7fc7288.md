### Title
N+1 Database Query DoS via Multiple CREATE2 EVM Address `contract.id` Filters in `getContracts()`

### Summary
An unauthenticated attacker can supply multiple `contract.id` query parameters containing CREATE2 EVM addresses to `GET /api/v1/contracts`. For each such filter, `extractSqlFromContractFilters()` sequentially `await`s a separate database lookup via `computeContractIdFromString()` → `getContractIdByEvmAddress()`, creating N independent DB queries per request. By sending many concurrent requests each with the maximum allowed repeated parameters, an attacker can exhaust the database connection pool and cause denial of service.

### Finding Description

**Exact code path:**

`getContracts()` in `rest/controllers/contractController.js` (line 745–771) calls `extractSqlFromContractFilters(filters)` at line 750.

Inside `extractSqlFromContractFilters()` (lines 83–144), a `for` loop iterates over every filter:

```js
// contractController.js lines 102–113
for (const filter of filters) {
  switch (filter.key) {
    case filterKeys.CONTRACT_ID:
      const contractIdValue = await ContractService.computeContractIdFromString(filter.value);
      ...
``` [1](#0-0) 

Each `await` is sequential — not batched. For each `CONTRACT_ID` filter value, `computeContractIdFromString()` is called:

```js
// contractService.js lines 468–476
async computeContractIdFromString(contractIdValue) {
  const contractIdParts = EntityId.computeContractIdPartsFromContractIdValue(contractIdValue);
  if (contractIdParts.hasOwnProperty('create2_evm_address')) {
    return this.getContractIdByEvmAddress(contractIdParts);  // DB query here
  }
  return EntityId.parseString(contractIdValue, ...).getEncodedId();
}
``` [2](#0-1) 

`getContractIdByEvmAddress()` issues a real database query for every invocation:

```js
// contractService.js lines 452–466
async getContractIdByEvmAddress(evmAddressFilter) {
  const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
    Buffer.from(create2EvmAddress, 'hex'),
  ]);
  ...
}
``` [3](#0-2) 

**Root cause:** The loop in `extractSqlFromContractFilters()` does not batch EVM address lookups; it resolves them one-at-a-time with sequential `await`, producing exactly N DB queries for N EVM address filters before the main contract query even runs.

**Why the existing check is insufficient:** `buildFilters()` in `utils.js` enforces `maxRepeatedQueryParameters` per query key: [4](#0-3) 

This caps N per request, but does not prevent an attacker from sending many concurrent requests. With the default limit (configurable, tested at values like 4–100), each request still triggers N sequential DB queries, and concurrent requests multiply the load linearly. No authentication, no per-IP rate limiting, and no caching of EVM address lookups are present in this code path.

### Impact Explanation
Each attacker request with N CREATE2 EVM address filters causes N sequential DB queries before the main query. With concurrent requests, the database connection pool is rapidly saturated. Since `getContractIdByEvmAddress()` queries the `entity` table by `evm_address` (not a unique index per the comment at line 460), each query may be a full or partial table scan under load. This can render the mirror node REST API unresponsive for all users — a complete availability failure. [5](#0-4) 

### Likelihood Explanation
The endpoint is public and requires zero authentication. The exploit requires only a standard HTTP client. The attacker needs no special knowledge beyond the public API documentation showing `contract.id` accepts EVM addresses. The attack is trivially repeatable and scriptable. Any valid 20-byte hex string is accepted as a CREATE2 EVM address input — the attacker does not need to know real contract addresses since a `NotFoundError` is thrown only *after* the DB query completes. [6](#0-5) 

### Recommendation
1. **Batch the EVM address lookups**: Replace the sequential `await` loop with a single batched query that resolves all CREATE2 EVM addresses in one SQL `WHERE evm_address = ANY($1)` call before building the final filter.
2. **Add a hard cap** on the number of `contract.id` filters accepted per request (separate from `maxRepeatedQueryParameters`, which is configurable and may be set high).
3. **Cache EVM address → entity ID resolutions** with a short TTL (e.g., using an in-process LRU cache) to prevent repeated DB hits for the same address across requests.
4. **Apply rate limiting** at the API gateway or middleware layer for this endpoint.

### Proof of Concept

```bash
# Generate N distinct fake CREATE2 EVM addresses and send as repeated contract.id filters
# Each address triggers one DB query in getContractIdByEvmAddress()

BASE_URL="https://<mirror-node-host>/api/v1/contracts"

# Build query string with maxRepeatedQueryParameters EVM addresses
PARAMS=""
for i in $(seq 1 25); do
  ADDR=$(printf '%040x' $RANDOM$RANDOM$RANDOM)
  PARAMS="${PARAMS}&contract.id=0x${ADDR}"
done

# Send many concurrent requests
for i in $(seq 1 200); do
  curl -s "${BASE_URL}?${PARAMS}" &
done
wait
```

Each of the 200 concurrent requests triggers 25 sequential DB queries (5,000 total DB queries) before any main query executes. Scale `seq 1 200` and the number of addresses to exhaust the connection pool. No credentials required.
