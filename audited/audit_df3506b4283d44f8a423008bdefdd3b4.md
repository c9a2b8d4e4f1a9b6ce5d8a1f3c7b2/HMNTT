### Title
Unauthenticated Thread-Exhaustion via Timeout-Duration EVM Calls on `/api/v1/contracts/call`

### Summary
Any unauthenticated caller can POST to `/api/v1/contracts/call` with maximum gas and a storage-heavy call target, causing the serving thread to be held synchronously for up to the full `requestTimeout` (10 000 ms default). Because the timeout is enforced only at Hibernate SQL-statement boundaries and not at the HTTP/thread level, an attacker sending ~20 such requests per second can saturate a typical Tomcat thread pool (200 threads), causing all other users' requests to queue and eventually time out.

### Finding Description

**Entry point – no authentication:**
`ContractController.call()` is a plain `@PostMapping` with no security annotation. [1](#0-0) 

**Timeout is SQL-boundary-only:**
`HibernateConfiguration.statementInspector()` checks elapsed time only immediately before each Hibernate SQL statement executes. Between SQL statements — including during pure EVM opcode execution — the thread is never interrupted. [2](#0-1) 

`ContractCallContext.startTime` is set at context creation and is the sole reference point for the elapsed-time check. [3](#0-2) 

**Execution is fully synchronous:**
`ContractExecutionService.processCall()` runs inside `ContractCallContext.run()` on the calling thread with no async handoff. The thread is held until EVM execution completes or the timeout fires at the next SQL boundary. [4](#0-3) 

**Throttle does not bound per-request duration:**
`ThrottleManagerImpl.throttle()` enforces a global RPS bucket (500 req/s) and a global gas bucket (7.5 B gas/s). Both are checked once, up-front, before execution begins. Neither limits how long a single accepted request may hold its thread. [5](#0-4) 

**Exploit flow:**
1. Attacker identifies (or deploys) a contract with many distinct storage slots.
2. Attacker sends `POST /api/v1/contracts/call` with `gas=15000000` (the maximum) and calldata that triggers a function reading many cold storage slots (SLOAD, 2 100 gas each → up to ~7 142 cold SLOADs per call).
3. Each SLOAD causes a DB query; each query passes through the `StatementInspector`. With ~1–2 ms per query, total wall-clock time approaches 10 s.
4. The request terminates only when the timeout fires at the next SQL boundary (≥10 000 ms elapsed) or gas is exhausted — whichever comes first.
5. The attacker sends ~20 such requests per second (well within the 500 RPS and gas-per-second limits). After 10 s, 200 threads are simultaneously occupied.
6. Tomcat's default thread pool (200 threads) is saturated; all subsequent legitimate requests queue and eventually time out.

**Why existing checks fail:**
- The gas throttle (`gasPerSecond = 7.5 B`, `maxGasLimit = 15 M`) allows 500 max-gas requests per second — far more than the ~20/s needed to saturate the thread pool.
- The RPS throttle (500/s) is also far above the attack threshold.
- There is no per-IP rate limit, no per-request wall-clock deadline at the HTTP/servlet layer, and no async processing. [6](#0-5) [7](#0-6) 

### Impact Explanation
Thread-pool exhaustion causes a complete denial of service for all users of the web3 module. Legitimate `eth_call` and `eth_estimateGas` requests queue behind the attacker's long-running requests and time out with HTTP 503. No economic damage occurs to any on-chain user; the impact is purely availability-based griefing of the mirror-node API.

### Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond knowing the endpoint exists. Any contract with many storage slots (common on mainnet/testnet) serves as the target. The attacker needs only ~20 HTTP requests per second — trivially achievable from a single machine. The attack is repeatable indefinitely as long as the gas and RPS buckets are not exhausted, which they will not be at 20 req/s.

### Recommendation
1. **Add a hard per-request wall-clock deadline at the servlet/filter layer** (e.g., a Spring `HandlerInterceptor` or Tomcat connector `connectionTimeout`/`asyncTimeout`) that interrupts the thread or returns 503 after `requestTimeout` ms, independent of SQL boundaries.
2. **Reduce `requestTimeout`** to a value that reflects realistic expected latency (e.g., 2–3 s) rather than 10 s.
3. **Add per-source-IP rate limiting** (e.g., via a servlet filter or API gateway) so a single attacker cannot consume the global RPS budget.
4. **Consider moving EVM execution to a bounded thread pool** with a `Future.get(timeout)` pattern so the calling thread is not held for the full duration.

### Proof of Concept
```bash
# Step 1: identify a contract with many storage slots on the target network
TARGET_CONTRACT="0x000000000000000000000000000000000000XXXX"

# Step 2: craft calldata for a function that reads many cold storage slots
# (e.g., a loop reading 7000 distinct slots)
CALLDATA="0x<encoded_function_selector_for_storage_heavy_function>"

# Step 3: send ~20 concurrent requests per second for 15+ seconds
for i in $(seq 1 300); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"$TARGET_CONTRACT\",\"gas\":15000000,\"data\":\"$CALLDATA\"}" &
  # 300 requests / 15 seconds ≈ 20 req/s
  [ $((i % 20)) -eq 0 ] && sleep 1
done

# Expected result: after ~10 seconds, legitimate requests to the same endpoint
# receive HTTP 503 Service Unavailable or hang until their own timeout fires.
```

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
    }
