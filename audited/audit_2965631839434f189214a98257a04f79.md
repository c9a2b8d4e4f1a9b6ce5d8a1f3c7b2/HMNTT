### Title
Unauthenticated DoS via Unbounded File Data Aggregation in `getContractResultsByTimestamp`

### Summary
Any unauthenticated external user can send `GET /api/v1/contracts/:contractId/results/:consensusTimestamp` for a transaction whose matched Ethereum record has a `callDataId` referencing a large on-chain file. This unconditionally triggers `FileDataService.getLatestFileDataContents()`, which executes a `string_agg`-based SQL query that concatenates every historical chunk of that file into a single in-memory buffer. Because Hedera is a public ledger, the required identifiers are freely discoverable, and no rate-limiting or size guard exists on this path.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 18 routes `GET /:contractId/results/:consensusTimestamp` to `ContractController.getContractResultsByTimestamp`.

Inside `getContractResultsByTimestamp` (`rest/controllers/contractController.js` lines 1002–1042):

```js
// lines 1016-1024
const [contractResults, ethTransactions, recordFile, contractLogs, contractStateChanges] =
  await this.getDetailedContractResults(contractDetails, contractId);   // 5 parallel DB queries

const ethTransaction = ethTransactions[0];

let fileData = null;
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {             // guard
  fileData = await FileDataService.getLatestFileDataContents(          // 6th DB query
    ethTransaction.callDataId, {whereQuery: []});
}
```

**The guard — `isValidUserFileId`** (`rest/utils.js` lines 445–447):

```js
const isValidUserFileId = (val) => {
  return !isNil(val) && val !== '' && EntityId.parse(val).num > 1000;
};
```

This only rejects system-entity file IDs (≤ 1000). It imposes **no size cap, no rate limit, and no authentication requirement**.

**The query — `getLatestFileDataContents`** (`rest/service/fileDataService.js` lines 18–41, 84–87):

```js
static latestFileContentsQuery = `with latest_create as (
    select max(consensus_timestamp) ...
    from file_data where entity_id = $1 ...
  )
  select
    max(...), min(...),
    string_agg(file_data, '' order by consensus_timestamp) as file_data
  from file_data fd
  join latest_create l on fd.consensus_timestamp >= l.consensus_timestamp
  where fd.entity_id = $1 ...`;
```

`string_agg` concatenates **all** file-data chunks since the last create/update into a single PostgreSQL result row, which is then loaded entirely into the Node.js heap. For Ethereum call-data files that were uploaded in many `FileAppend` chunks, this can be megabytes per request.

**Root cause / failed assumption:** The code assumes that `callDataId` will always reference a small file, and that the endpoint is not a meaningful amplification vector. Neither assumption is enforced in code.

### Impact Explanation

Each request to a qualifying endpoint causes:
1. Five parallel DB queries (already present for every call).
2. One additional `string_agg` aggregation query that scans and concatenates the entire file history for the referenced file entity.
3. The full concatenated byte array is allocated in the Node.js heap before being serialised into the HTTP response.

An attacker who identifies a single `(contractId, consensusTimestamp)` pair whose Ethereum transaction references a large call-data file can flood the endpoint. The result is elevated DB CPU/IO, heap pressure in the REST service, and potential OOM-kill of the Node.js process — a non-network-based DoS affecting the REST API tier without any privileged access.

### Likelihood Explanation

Hedera is a public ledger. All consensus timestamps, contract IDs, and Ethereum transaction metadata (including `call_data_id`) are publicly visible via the mirror node itself or third-party explorers. An attacker needs only to:
1. Query `/api/v1/contracts/results` to enumerate transactions.
2. Identify one with a non-null `call_data_id` referencing a large file.
3. Replay that single URL at high frequency.

No credentials, no special network position, and no prior knowledge beyond public chain data are required. The attack is trivially scriptable and repeatable.

### Recommendation

1. **Add a file-size pre-check**: Before calling `getLatestFileDataContents`, query only the file's size (e.g., via a `SELECT sum(length(file_data))` or a stored metadata column) and reject or truncate if it exceeds a configurable threshold (e.g., 128 KB).
2. **Cache the result**: The file content for a given `callDataId` is immutable once the Ethereum transaction is finalised. Cache the resolved bytes (keyed by `callDataId`) so repeated requests do not re-execute the aggregation query.
3. **Apply per-IP rate limiting** specifically to the `/:contractId/results/:consensusTimestamp` endpoint, independent of the authenticated-user limit mechanism.
4. **Stream / paginate** the file-data response rather than loading the entire `string_agg` result into heap.
