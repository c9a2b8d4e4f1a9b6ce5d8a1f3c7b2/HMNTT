### Title
Unbounded `ne` Timestamp Conditions in `getContractResultsById` Enable Non-Authenticated DoS via Expensive DB Scans

### Summary
The `GET /contracts/:contractId/results` endpoint accepts an arbitrary number of `timestamp=ne:X` query parameters (bounded only by `maxRepeatedQueryParameters`). When a `contractId` is present, `optimizeTimestampFilters` is bypassed entirely, so each `ne` value is emitted as a separate `AND cr.consensus_timestamp != $N` SQL condition with no deduplication and no mandatory range constraint. An unauthenticated attacker can force a full-table scan over all results for a popular contract, repeated at will.

### Finding Description

**Code path:**

`getContractResultsById` (line 856) calls `extractContractIdAndFiltersFromValidatedRequest` → `buildAndValidateFilters`, which accepts repeated `timestamp` values up to `config.query.maxRepeatedQueryParameters`. [1](#0-0) 

Each `timestamp` filter is collected into `timestampFilters` at line 461–463. [2](#0-1) 

At line 504–505, when `contractId` is defined the code takes the `else` branch and skips `optimizeTimestampFilters` entirely, passing the raw `timestampFilters` straight through: [3](#0-2) 

`optimizeTimestampFilters` (only called when `contractId === undefined`) is the only place that runs `parseTimestampFilters`, which uses a `Set` to deduplicate `ne` values: [4](#0-3) [5](#0-4) 

For the `contractId` path, each raw filter (including every `ne`) is fed to `updateConditionsAndParamsWithInValues`. Because the operator is not `eq`, each one is appended as a distinct SQL condition: [6](#0-5) 

This produces a query of the form:
```sql
WHERE cr.contract_id = $1
  AND cr.consensus_timestamp != $2
  AND cr.consensus_timestamp != $3
  ...
  AND cr.consensus_timestamp != $N
  AND cr.transaction_nonce = 0
LIMIT $N+1
```

With no `gte`/`lte` range, the planner must scan every row for that contract and evaluate N inequality predicates per row.

**The only guard** is `isRepeatedQueryParameterValidLength` in `buildFilters`, which rejects requests only when the count of values for a single key exceeds `config.query.maxRepeatedQueryParameters`: [7](#0-6) 

The test confirms this default is well above 4 (the test artificially lowers it to 4 to trigger the error), meaning an attacker can supply tens to hundreds of `ne` conditions before being rejected. [8](#0-7) 

### Impact Explanation
A popular contract (e.g., a DEX or stablecoin) can accumulate millions of `contract_result` rows. A request with the maximum allowed `ne` conditions and no range bounds forces a sequential scan of all rows for that contract, evaluating N inequality predicates per row, before the `LIMIT` is satisfied. Repeated at high frequency from a single client (no auth required), this can saturate DB CPU and I/O, degrading or denying service for all users of the mirror node. This is a non-network-based DoS (DB resource exhaustion), matching the stated severity class.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is public. The attacker needs only to know (or guess) a contract ID with a large result set, which is trivially discoverable from the public API. The request is cheap to construct and can be automated. The `maxRepeatedQueryParameters` limit does reduce the per-request cost, but does not prevent sustained bombardment.

### Recommendation
1. **Enforce a mandatory timestamp range** for the `contractId` path, just as `optimizeTimestampFilters` does for the no-contractId path. Reject requests that supply only `ne` operators with no `gte`/`lte` bounds.
2. **Cap `ne` operator count independently** (e.g., max 3–5 `ne` values) rather than relying on the generic `maxRepeatedQueryParameters` limit.
3. **Deduplicate `ne` values** for the `contractId` path by routing through `parseTimestampFilters` (with its `Set`-based deduplication) before building SQL conditions, mirroring the behavior of the no-contractId path.

### Proof of Concept
```bash
# Replace 0.0.XXXXX with a contract known to have many results
BASE="https://<mirror-node>/api/v1/contracts/0.0.XXXXX/results"

# Build a request with maxRepeatedQueryParameters ne values and no range
NE_PARAMS=$(python3 -c "
import random
params = '&'.join(f'timestamp=ne:{random.randint(1,9999999999999999999)}' for _ in range(25))
print(params)
")

# Fire repeatedly
while true; do
  curl -s "$BASE?$NE_PARAMS" > /dev/null &
done
```

Each request generates a SQL query with 25 separate `AND cr.consensus_timestamp != $i` conditions and no range bound, forcing a full scan of all rows for the target contract. Sustained parallel requests exhaust DB CPU.
