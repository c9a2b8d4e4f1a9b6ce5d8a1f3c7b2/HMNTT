### Title
Unauthenticated gRPC Subscription DoS via Non-TOPIC Entity ID Exhaustion

### Summary
The `topicExists()` method in `TopicMessageServiceImpl` performs an unconditional database lookup for every `subscribeTopic` call, then creates a new `IllegalArgumentException` (with full JVM stack trace) when the resolved entity is not of type `TOPIC`. Because the gRPC endpoint has no per-IP or global request rate limiting, an unprivileged attacker can drive sustained DB query load and exception-creation CPU overhead by repeatedly submitting subscriptions with valid account or contract entity IDs.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` (line 87) unconditionally calls `topicExists(filter)` before any message streaming begins:

```java
return topicExists(filter)
        .thenMany(flux...);
```

`topicExists()` (lines 94–106):
```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))   // DB hit every call
            .switchIfEmpty(...)
            .filter(e -> e.getType() == EntityType.TOPIC)                 // type check
            .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic"))); // exception + stack trace
}
```

**Root cause / failed assumption:** The design assumes callers supply a valid TOPIC entity ID. There is no pre-validation of entity type before the DB call, no caching of negative results visible in this call path, and no rate limiting on the gRPC layer.

**Why existing checks are insufficient:**
- `TopicMessageFilter` bean validation (`@NotNull`, `@Min`) only enforces structural constraints on the filter fields — it does not validate that the `topicId` resolves to a TOPIC entity. [1](#0-0) 
- `hiero.mirror.grpc.netty.maxConcurrentCallsPerConnection = 5` limits concurrent streams *per connection*, not the total request rate across many connections. An attacker opens N connections and cycles through them. [2](#0-1) 
- Rate limiting (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module for contract calls — it is entirely absent from the `grpc` module. [3](#0-2) 
- `checkTopicExists = true` (default) means the entity-not-found path also throws, but the wrong-type path always hits the DB first regardless of this flag. [4](#0-3) 

### Impact Explanation
Each malicious request causes: (1) a synchronous `entityRepository.findById()` database query, (2) construction of a `new IllegalArgumentException` including JVM stack-trace capture (expensive), and (3) a `log.info("Subscribing to topic: {}", filter)` log write. At high request rates across multiple connections this translates to: elevated DB connection pool pressure, CPU saturation from stack-trace generation, and log-sink I/O amplification — all of which degrade service for legitimate subscribers. [5](#0-4) 

### Likelihood Explanation
Preconditions are minimal: no authentication is required to call the gRPC `subscribeTopic` endpoint; account and contract entity IDs are publicly enumerable from the mirror node's own REST API. The attacker needs only a gRPC client library and a list of known non-TOPIC entity IDs. The attack is trivially scriptable, repeatable indefinitely, and requires no special privileges or on-chain resources.

### Recommendation
1. **Add application-level rate limiting to the gRPC module** analogous to the `ThrottleManagerImpl` / `Bucket4j` pattern already used in `web3`. Apply a per-IP or global cap on `subscribeTopic` calls per second.
2. **Cache entity-type lookups** so that repeated requests for the same non-TOPIC entity ID do not hit the database. The `entityCacheSize = 50_000` property already exists in `GrpcProperties` but is not wired to `topicExists()`. [6](#0-5) 
3. **Avoid allocating exception objects with stack traces** for the wrong-entity-type case; use a pre-allocated singleton error signal or a lightweight sentinel instead of `new IllegalArgumentException(...)` inside the reactive pipeline. [7](#0-6) 

### Proof of Concept
```
# Prerequisites: grpcurl installed; mirror node gRPC port 5600 reachable
# 1. Enumerate a known account entity ID (e.g. 0.0.98 — treasury account)
# 2. Repeatedly invoke subscribeTopic with that ID as topicId

while true; do
  grpcurl -plaintext -d '{
    "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 98},
    "consensusStartTime": {"seconds": 0}
  }' <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
```
Each iteration: triggers `entityRepository.findById(98)` → entity found (ACCOUNT type) → `IllegalArgumentException` created with stack trace → gRPC INVALID_ARGUMENT returned. Running hundreds of concurrent goroutines/threads drives sustained DB and CPU load with zero on-chain cost to the attacker.

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
