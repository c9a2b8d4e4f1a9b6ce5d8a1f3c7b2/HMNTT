### Title
Cache Key Bypass via Non-Normalized `req.originalUrl` in `cacheKeyGenerator()`

### Summary
`cacheKeyGenerator()` in `rest/middleware/responseCacheHandler.js` hashes the raw `req.originalUrl` string directly with MD5, without any URL normalization. Because `req.originalUrl` is Express's unmodified original request URL, an unprivileged attacker can craft semantically identical requests with different query parameter orderings, URL-encoded characters, or mixed-case parameter keys to produce distinct MD5 hashes, causing perpetual cache misses and forcing full database processing for every variant. The codebase itself acknowledges this gap in a TODO comment referencing a future normalizer integration.

### Finding Description
**Exact code location:** `rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, lines 151–153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

The comment immediately above (lines 141–149) explicitly states:
> *"path?query - In the future, this will utilize Edwin's request normalizer (9113)."*

**Root cause:** `req.originalUrl` in Express is the raw, unmodified URL string as received from the TCP socket. It is never rewritten by any middleware in the pipeline. The `requestQueryParser` (registered at `server.js` line 60) only populates `req.query` with lowercased keys — it does not touch `req.originalUrl`. The `normalizeRequestQueryParams` function in `rest/middleware/requestNormalizer.js` (which sorts parameters, adds defaults, and collapses duplicates) is never called in the middleware chain before `responseCacheCheckHandler` (registered at `server.js` line 97), and is not invoked inside `cacheKeyGenerator()` at all.

**Exploit flow:**

The following URL pairs are semantically identical to the API but produce different MD5 hashes:

| Variant A | Variant B |
|---|---|
| `?limit=3&account.id=gte:0.0.18` | `?account.id=gte:0.0.18&limit=3` |
| `?account.id=gte:0.0.18` | `?account.id=gte%3A0.0.18` |
| `?LIMIT=3&account.id=gte:0.0.18` | `?limit=3&account.id=gte:0.0.18` |

Each variant misses the cache, hits `responseCacheUpdateHandler`, and triggers a full PostgreSQL query. An attacker can enumerate N! orderings of N parameters, plus URL-encoding permutations, to generate an unbounded number of distinct cache keys for a single logical query.

**Why existing checks fail:**
- `requestQueryParser` (`requestHandler.js` lines 38–68): Lowercases `req.query` keys and canonicalizes `order`/`result` values, but operates only on the parsed query object — `req.originalUrl` is never modified.
- `normalizeRequestQueryParams` (`requestNormalizer.js` lines 35–59): Correctly sorts parameters and adds defaults, but is not wired into the cache key path. It is only used in tests.
- No rate limiting or deduplication is applied to cache key generation.

### Impact Explanation
Every cache miss forces a full backend database query. With N query parameters, an attacker can generate N! parameter-order permutations plus URL-encoding variants, each producing a unique cache key. For a typical endpoint with 4–5 parameters (e.g., `/api/v1/accounts` with `account.id`, `limit`, `order`, `balance`, `timestamp`), this yields 120+ distinct cache-bypassing URLs per endpoint. Sustained at moderate request rates (e.g., 100 req/s across variants), this can saturate the PostgreSQL connection pool and drive CPU above the 30% threshold specified in the threat model, constituting a denial-of-service against the node's processing capacity.

### Likelihood Explanation
No authentication or privilege is required — the REST API is publicly accessible. The attack requires only knowledge of valid query parameters (documented in the public OpenAPI spec) and the ability to reorder or URL-encode them. It is trivially scriptable, repeatable, and requires no special tooling beyond `curl` or a simple HTTP client loop. The codebase's own TODO comment confirms the developers are aware of the missing normalization, indicating it is a known open gap.

### Recommendation
Integrate `normalizeRequestQueryParams` from `rest/middleware/requestNormalizer.js` into `cacheKeyGenerator()` so the cache key is derived from the normalized URL rather than `req.originalUrl`:

```js
import {normalizeRequestQueryParams} from './requestNormalizer.js';

const cacheKeyGenerator = (req) => {
  const normalizedUrl = normalizeRequestQueryParams(req.openapi?.openApiRoute, req.path, req.query);
  return crypto.createHash('md5').update(normalizedUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

This resolves parameter-order and default-value variance. Additionally:
- Decode and re-encode `req.path` and query values canonically before hashing to eliminate URL-encoding variants.
- Ensure `req.query` key lowercasing (already done by `requestQueryParser`) feeds into the normalized URL.

### Proof of Concept

```bash
# Both requests are semantically identical but produce different cache keys

# Request 1 - populates cache with key = MD5("/api/v1/accounts?limit=3&account.id=gte:0.0.18")
curl "https://<node>/api/v1/accounts?limit=3&account.id=gte:0.0.18"

# Request 2 - CACHE MISS, key = MD5("/api/v1/accounts?account.id=gte:0.0.18&limit=3")
curl "https://<node>/api/v1/accounts?account.id=gte:0.0.18&limit=3"

# Request 3 - CACHE MISS, key = MD5("/api/v1/accounts?account.id=gte%3A0.0.18&limit=3")
curl "https://<node>/api/v1/accounts?account.id=gte%3A0.0.18&limit=3"

# Automated flood: generate all permutations of 4 parameters
python3 -c "
import itertools, requests
params = [('limit','3'),('account.id','gte:0.0.18'),('order','asc'),('balance','true')]
for perm in itertools.permutations(params):
    qs = '&'.join(f'{k}={v}' for k,v in perm)
    requests.get(f'https://<node>/api/v1/accounts?{qs}')
"
# Generates 24 unique cache keys for 1 logical query, each triggering a full DB query
```
