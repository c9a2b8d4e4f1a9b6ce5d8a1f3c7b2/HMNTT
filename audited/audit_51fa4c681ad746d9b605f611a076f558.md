### Title
Silent Zero-Balance Return for Non-Existent Alias-Based Accounts in `RetrieveBalanceAtBlock`

### Summary
`RetrieveBalanceAtBlock` in `rosetta/app/persistence/account.go` explicitly returns a zero-balance success response (no error) when an alias-based account does not exist in the database. Any unprivileged caller querying the Rosetta `/account/balance` endpoint with a fabricated or non-existent alias receives an HTTP 200 with zero HBAR balance, indistinguishable from a real account with zero balance.

### Finding Description
**Exact code path:**

In `rosetta/app/persistence/account.go`, `RetrieveBalanceAtBlock` (lines 161–223):

```go
entity, err := ar.getCryptoEntity(ctx, accountId, consensusEnd)
if err != nil {
    return nil, entityIdString, nil, err
}

if entity == nil && accountId.HasAlias() {
    return types.AmountSlice{&types.HbarAmount{}}, entityIdString, nil, nil  // line 173
}
```

When `getCryptoEntity` finds no matching entity (returns `nil, nil`), and the queried `accountId` has an alias, the function returns `types.AmountSlice{&types.HbarAmount{}}` — a slice containing a zero-value HBAR amount — with a `nil` error.

**Root cause:** The failed assumption is that returning zero balance with no error is a safe sentinel for "alias not yet on-chain." Instead, it is semantically identical to a real account with zero balance.

**Caller propagation:**

In `rosetta/app/services/account_service.go`, `AccountBalance` (lines 61–78):
```go
balances, accountIdString, publicKey, rErr := a.accountRepo.RetrieveBalanceAtBlock(ctx, accountId, block.ConsensusEndNanos)
if rErr != nil {
    return nil, rErr
}
// ... builds and returns AccountBalanceResponse with balances
```

Since `rErr` is `nil`, the service constructs a full `AccountBalanceResponse` with the zero balance and returns HTTP 200. The only observable difference from a real zero-balance account is that `metadata["account_id"]` is absent (because `entityIdString` is empty), but this is not an error signal to API consumers.

**Why existing checks fail:**

The check `if accountId.HasAlias() && accountIdString != ""` at line 67 of `account_service.go` only controls metadata population — it does not gate the response or signal non-existence to the caller.

### Impact Explanation
Any system consuming the Rosetta `/account/balance` API (exchanges, wallets, monitoring tools, block explorers) will receive a successful response with zero balance for completely fabricated alias addresses. These systems cannot distinguish "account exists with zero balance" from "account does not exist." This breaks account-existence checks, can suppress alerts for missing accounts, and can cause incorrect state assumptions in downstream reconciliation or deposit-detection logic. Severity is medium: no funds are moved, but data integrity of the balance API is compromised for all alias-based queries.

### Likelihood Explanation
Exploitation requires zero privileges. Any HTTP client can call the public Rosetta `/account/balance` endpoint with a hex-encoded public key as the address (the alias format). The attacker does not need any account, key, or network access beyond the ability to send an HTTP request. The attack is trivially repeatable and requires no special knowledge — any random 32-byte or 64-byte hex string that parses as a valid public key alias will trigger the path.

### Recommendation
Replace the silent zero-balance return with an explicit `ErrAccountNotFound` error when `entity == nil && accountId.HasAlias()`:

```go
if entity == nil && accountId.HasAlias() {
    return nil, entityIdString, nil, hErrors.ErrAccountNotFound
}
```

This aligns with the behavior of `GetAccountId` (lines 151–153 of `account.go`), which already returns `hErrors.ErrAccountNotFound` when an alias-based account is not found. The balance endpoint should be consistent.

### Proof of Concept
1. Generate any valid ECDSA secp256k1 or ED25519 public key and encode it as a hex alias (e.g., `0x3a21<33-byte-compressed-pubkey>`).
2. Ensure no account with that alias exists on the network.
3. Send a POST to `/account/balance`:
   ```json
   {
     "network_identifier": { "blockchain": "Hedera", "network": "mainnet" },
     "account_identifier": { "address": "0x3a21<fabricated-pubkey-bytes>" }
   }
   ```
4. Observe HTTP 200 response:
   ```json
   {
     "block_identifier": { ... },
     "balances": [{ "value": "0", "currency": { "symbol": "HBAR", "decimals": 8 } }]
   }
   ```
5. No error is returned. The caller cannot determine whether the account exists or not. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
