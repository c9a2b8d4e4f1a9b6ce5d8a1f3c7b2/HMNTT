### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Timestamp Range on `GET /api/v1/transactions`

### Summary
The `GET /api/v1/transactions` route in `rest/server.js` accepts arbitrary timestamp ranges from any unauthenticated caller with no rate limiting. Because `bindTimestampRange` defaults to `false`, the timestamp range enforcement in `rest/timestampRange.js` is bypassed entirely, allowing queries spanning all historical data to be submitted. With a default pool of only 10 DB connections and a 20-second `statement_timeout`, an attacker sending 10 concurrent wide-range requests can hold every connection open for the full timeout window, exhausting the pool and denying service to all other API consumers.

### Finding Description

**Route registration — no rate limiting middleware:**

`rest/server.js` line 132 registers the route directly:
```js
app.getExt(`${apiPrefix}/transactions`, transactions.getTransactions);
``` [1](#0-0) 

The middleware stack applied before this route is: `cors`, `compression`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`. None of these impose per-IP or per-user request rate limits. [2](#0-1) 

No `express-rate-limit` or equivalent package is present anywhere in the `rest/` Node.js service.

**`bindTimestampRange` is `false` by default:**

`rest/timestampRange.js` lines 19–22 short-circuit all range enforcement when the flag is off:
```js
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};   // ← user-supplied range returned unchanged
  }
``` [3](#0-2) 

The documented default for `hiero.mirror.rest.query.bindTimestampRange` is `false`: [4](#0-3) 

Even when enabled, `maxTransactionsTimestampRange` is 60 days — still a very wide scan window. [5](#0-4) 

**Query execution path:**

`getTransactions` → `doGetTransactions` → `getTransactionTimestamps` → `pool.queryQuietly(query, params)`. The user-supplied timestamp range flows directly into the SQL query with no server-side clamping when `bindTimestampRange=false`. [6](#0-5) [7](#0-6) 

**Pool configuration — small size, long timeout:**

`rest/dbpool.js` configures the pool with defaults from config:
```js
max: config.db.pool.maxConnections,          // default: 10
statement_timeout: config.db.pool.statementTimeout,  // default: 20000 ms
``` [8](#0-7) [9](#0-8) 

Each wide-range query holds a connection for up to 20 seconds before PostgreSQL kills it. With 10 connections in the pool, 10 concurrent attacker requests saturate the pool entirely.

### Impact Explanation

While the pool is saturated, every legitimate API request that requires a DB connection blocks for up to `connectionTimeout` (default 20 000 ms) and then fails. This affects all endpoints sharing the same pool — accounts, balances, tokens, schedules, and transactions. The result is a complete, sustained denial of service for all API consumers with no authentication required. Severity: **High** (availability impact, no integrity/confidentiality loss, but full service disruption achievable by a single attacker).

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only a standard HTTP client. The attacker needs to maintain ~10 concurrent connections — trivially achievable with `curl --parallel`, `ab`, `wrk`, or any scripting language. The attack is repeatable indefinitely: each 20-second saturation window can be immediately followed by another batch. There is no IP-based blocking, no CAPTCHA, and no circuit breaker in the REST service itself.

### Recommendation

1. **Enable `bindTimestampRange: true` in production** and keep `maxTransactionsTimestampRange` at a value that limits query cost (e.g., 7 days instead of 60 days).
2. **Add per-IP rate limiting** to the Express application (e.g., `express-rate-limit`) applied globally before route handlers in `rest/server.js`.
3. **Increase pool size** or add a concurrency semaphore so that a single client IP cannot hold all connections simultaneously.
4. **Reduce `statementTimeout`** for the `mirror_rest` DB user to a value that reflects realistic query expectations (e.g., 5 000 ms), limiting the window each connection is held.

### Proof of Concept

```bash
# Exhaust the 10-connection pool with 10 concurrent wide-range requests
# (no authentication required)
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=gte:0&timestamp=lte=9999999999&limit=100" &
done
wait

# While the above runs, all other API requests will block/fail:
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: hangs for ~20s then returns a 503/connection error
```

Repeat the first block every ~18 seconds to maintain continuous pool exhaustion.
