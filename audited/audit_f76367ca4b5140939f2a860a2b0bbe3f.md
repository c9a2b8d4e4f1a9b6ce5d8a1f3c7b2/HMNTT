### Title
`isValidAccount()` Returns `true` for Deleted Accounts Due to Missing `deleted` Filter in `entityExistenceQuery`

### Summary
`EntityService.entityExistenceQuery` queries the `entity` table by ID alone, with no filter on the `deleted` column. As a result, `isValidAccount()` returns `true` for any entity that exists in the database — including those marked as deleted. Both `tokenController.getTokenRelationships` and `accountController.listStakingRewardsByAccountId` rely on this check as their sole access gate, so an unprivileged caller can retrieve token-relationship and staking-reward history for deleted accounts.

### Finding Description
**Exact code location:**

`rest/service/entityService.js`, lines 28–30 (`entityExistenceQuery`) and lines 60–63 (`isValidAccount`):

```js
// line 28-30
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;   // ← no deleted filter

// line 60-63
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);   // true for deleted entities
}
```

**Contrast with sibling queries** that correctly exclude deleted entities:

```js
// line 17-20  (alias lookup)
where coalesce(${Entity.DELETED}, false) <> true  and ${Entity.ALIAS} = $1

// line 22-25  (EVM address lookup)
where ${Entity.DELETED} <> true  and ${Entity.EVM_ADDRESS} = $1
```

**Call sites that rely on `isValidAccount` as the sole gate:**

- `rest/controllers/tokenController.js` lines 68–71: `getTokenRelationships` — queries token associations after this check.
- `rest/controllers/accountController.js` lines 172–175: `listStakingRewardsByAccountId` — queries staking rewards after this check.

**Root cause:** The developer assumed that "entity exists in DB" ≡ "entity is valid/active." That assumption is false; deleted entities remain in the `entity` table with `deleted = true`.

**Failed assumption in tests:** The integration test suite (`rest/__tests__/service/entityService.test.js`, lines 148–158) only tests a non-deleted entity returning `true` and a missing entity returning `false`. There is no test for a deleted entity — the case that should return `false` but currently returns `true`.

### Impact Explanation
An unprivileged external user can enumerate deleted account IDs (they are sequential integers) and issue:

- `GET /accounts/{deleted_id}/tokens` → receives full token-relationship history for a deleted account.
- `GET /accounts/{deleted_id}/rewards` → receives full staking-reward history for a deleted account.

This is inconsistent with the system's own access-control intent: alias and EVM-address lookups both refuse to resolve deleted entities, so the same data is inaccessible via those paths. The numeric-ID path bypasses that intent. The impact is unauthorized read access to historical financial data (token holdings, staking rewards) of accounts whose owners deleted them with an expectation of inaccessibility.

### Likelihood Explanation
No authentication or special privilege is required. Account IDs are sequential integers, trivially enumerable. Any HTTP client can reproduce this. The only precondition is knowing (or guessing) the numeric ID of a deleted account, which is trivial given the sequential nature of Hedera entity IDs and the public availability of mirror-node data.

### Recommendation
Add a `deleted` filter to `entityExistenceQuery`, consistent with the other two queries in the same class:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1
                                 and coalesce(${Entity.DELETED}, false) <> true`;
```

Additionally, add an integration test for `isValidAccount` with a deleted entity asserting the result is `false`.

### Proof of Concept
**Preconditions:**
1. Mirror node REST API is running.
2. Account with numeric ID `12345` exists in the `entity` table with `deleted = true`.

**Steps:**
```
GET /api/v1/accounts/0.0.12345/tokens
GET /api/v1/accounts/0.0.12345/rewards
```

**Expected (correct) behavior:** `404 Not Found`

**Actual behavior:** `200 OK` with token-relationship or staking-reward data for the deleted account, because `isValidAccount(accountId)` queries `entityExistenceQuery` (no `deleted` filter), finds the row, and returns `true`, allowing the downstream service queries to execute.
