### Title
`ConstructionParse` Returns Unverified Account Identifiers as Signers When `signed=true`

### Summary
`ConstructionParse()` in `rosetta/app/services/construction_service.go` populates `AccountIdentifierSigners` from accounts derived purely from the transaction body (sender accounts with negative balances) when `request.Signed = true`, without performing any cryptographic signature verification. An unprivileged attacker can craft a transaction naming any victim account as a sender, submit it to the endpoint with `signed=true`, and receive a response listing the victim as a confirmed signer — a claim the service has no basis to make.

### Finding Description
**Code path:**

In `ConstructionParse()` (lines 183–215), the flow is:
1. The transaction is decoded from the caller-supplied hex string via `unmarshallTransactionFromHexString` — no signature check occurs here.
2. `c.transactionHandler.Parse(ctx, transaction)` is called (line 198), which dispatches to either `cryptoTransferTransactionConstructor.Parse` or `cryptoCreateTransactionConstructor.Parse`.
3. For a `TransferTransaction`, `cryptoTransferTransactionConstructor.Parse` (lines 54–88 of `crypto_transfer_transaction_constructor.go`) identifies "signers" as any account with a negative HBAR balance in the transfer map — i.e., senders — derived entirely from the transaction body bytes, with zero signature verification.
4. Back in `ConstructionParse`, lines 203–208:
```go
signers := make([]*rTypes.AccountIdentifier, 0, len(accounts))
if request.Signed {
    for _, account := range accounts {
        signers = append(signers, account.ToRosetta())
    }
}
```
The `request.Signed` flag is caller-controlled. There is no check that the transaction bytes actually contain a valid signature from any of the returned accounts.

**Root cause:** The function conflates "accounts that are expected to sign" (derived from the transaction body) with "accounts that have actually signed" (which would require verifying the `sigPairs` in the `SignedTransaction` protobuf). The `signed` flag is trusted blindly from the caller.

**Contrast with `ConstructionCombine`:** That endpoint (lines 54–97) does perform real Ed25519 verification at line 82:
```go
if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {
    return nil, errors.ErrInvalidSignatureVerification
}
```
`ConstructionParse` has no equivalent check.

### Impact Explanation
Any downstream system (exchange, custodian, relay, or Rosetta client middleware) that calls `/construction/parse` with `signed=true` and uses the returned `account_identifier_signers` as evidence that those accounts have authorized the transaction is deceived. The attacker can produce a response claiming account `0.0.12345` (the victim) has signed a transfer of funds, without ever possessing the victim's private key. This can be used to bypass off-chain authorization gates, trigger conditional logic, or produce fraudulent audit trails. The Rosetta specification's intent for `signed=true` is that the returned signers have actually signed; this implementation violates that contract.

### Likelihood Explanation
The attack requires no credentials, no privileged access, and no on-chain interaction. The endpoint is unauthenticated by design (Rosetta is a public API). The attacker only needs to:
- Know a victim account ID (publicly visible on-chain)
- Construct a valid `TransferTransaction` protobuf with the victim as sender
- Send one HTTP POST to `/construction/parse`

This is trivially repeatable and automatable.

### Recommendation
When `request.Signed = true`, `ConstructionParse` must verify that the transaction bytes contain at least one valid signature from each account in `AccountIdentifierSigners` before returning them. Specifically:

1. Extract `sigPairs` from the `SignedTransaction.sig_map` in the decoded transaction.
2. For each account returned as a signer, verify that a valid Ed25519 signature over `BodyBytes` exists in `sig_map` for a key associated with that account (mirroring the logic already present in `ConstructionCombine` at lines 72–87).
3. If no valid signature is found for a claimed signer, either omit that account from `AccountIdentifierSigners` or return an error.

Alternatively, add explicit documentation and a runtime guard that `signed=true` is only accepted for transactions that have passed through `ConstructionCombine`, though cryptographic verification is the only reliable enforcement.

### Proof of Concept
1. Obtain any valid victim account ID, e.g. `0.0.9999`.
2. Construct a `TransferTransaction` protobuf with:
   - `AccountID` of `0.0.9999` sending `-100` tinybars to attacker account `0.0.1234`
   - A valid `TransactionID` (any payer, any timestamp)
   - No `sig_map` entries (unsigned)
3. Serialize and hex-encode the transaction bytes.
4. POST to `/construction/parse`:
```json
{
  "network_identifier": { ... },
  "signed": true,
  "transaction": "<hex_encoded_unsigned_tx>"
}
```
5. Observe the response:
```json
{
  "operations": [...],
  "account_identifier_signers": [
    { "address": "0.0.9999" }
  ]
}
```
The victim account `0.0.9999` is returned as a confirmed signer despite having provided no signature. The service accepted `signed=true` from the attacker and returned the victim's identity as an authorizer based solely on the attacker-controlled transaction body.
