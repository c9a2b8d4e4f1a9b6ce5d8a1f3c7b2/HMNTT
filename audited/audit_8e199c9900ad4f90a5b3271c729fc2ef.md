### Title
OR-Query Multi-Row Collision Enables Targeted DoS on EVM Address Lookups via Unprivileged Account Creation

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` issues a native SQL query with `OR` across two columns (`evm_address` and `alias`) with no `LIMIT` clause and no database-level uniqueness constraint on either column. An unprivileged Hedera user can create an account whose 20-byte `alias` equals an existing entity's `evm_address`, causing the query to return multiple rows. Spring Data JPA's `Optional<Entity>` return type then throws `IncorrectResultSizeDataAccessException`, permanently disrupting all web3 lookups for that address until the colliding entity is deleted.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 39–49:

```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
@Query(value = """
        select *
        from entity
        where (evm_address = ?1 or alias = ?1) and deleted is not true
        """, nativeQuery = true)
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

**Root cause — three compounding failures:**

1. **No uniqueness constraint on either column.** The database schema adds only non-unique indexes on `evm_address` and `alias`:
   - `V1.58.6__ethereum_nonce.sql`: `create index if not exists entity__evm_address on entity (evm_address) where evm_address is not null;`
   - `V1.50.3__account_alias.sql`: `create index if not exists entity__alias on entity (alias) where alias is not null;`
   No `UNIQUE` constraint exists on either column, and no cross-column uniqueness is enforced.

2. **`OR` query with no `LIMIT` returns all matching rows.** The query has no `LIMIT 1` and no tie-breaking `ORDER BY`. When two entities match — one via `evm_address = ?1`, another via `alias = ?1` — the query returns two rows. Spring Data JPA's `getSingleResult()` path for `Optional<Entity>` throws `IncorrectResultSizeDataAccessException`.

3. **`Utility.aliasToEvmAddress` identity for 20-byte inputs.** In `importer/src/main/java/org/hiero/mirror/importer/util/Utility.java`, lines 71–73:
   ```java
   if (alias.length == DomainUtils.EVM_ADDRESS_LENGTH) {
       return alias;
   }
   ```
   A 20-byte alias is returned unchanged as the `evm_address`. So when an attacker creates Account B with `alias = X` (20 bytes), `CryptoCreateTransactionHandler` sets both `alias = X` and `evm_address = X` on Account B. The OR query then matches Account A (via `evm_address = X`) and Account B (via `evm_address = X` OR `alias = X`) — two distinct rows.

**Exploit flow:**

1. Attacker observes target Account A's `evm_address = X` (public information via mirror node REST API).
2. Attacker submits a `CryptoCreate` transaction with `alias = X` (20-byte EVM address). The network stores Account B with `alias = X` and `evm_address = X`. No database constraint prevents this.
3. Any subsequent call to `findByEvmAddressOrAliasAndDeletedIsFalse(X)` — triggered by `CommonEntityAccessor.get(Bytes alias, ...)` at line 49 — executes the OR query, gets two rows, and throws `IncorrectResultSizeDataAccessException`.
4. Because `unless = "#result == null"` only suppresses caching on null results, and the method throws before returning, nothing is cached. Every subsequent call re-executes the query and re-throws. The disruption is permanent until Account B is deleted.

### Impact Explanation

All web3 API calls that resolve Account A by its EVM address via the alias path (`CommonEntityAccessor.get(Bytes, Optional<Long>)`) will throw an uncaught `IncorrectResultSizeDataAccessException`. This renders Account A's EVM address unresolvable through the mirror node web3 endpoint for the duration Account B exists. The attacker can target any account whose `evm_address` is known (all of them, via public APIs). No funds are at risk; the impact is service disruption (griefing) for specific addresses.

### Likelihood Explanation

Any Hedera user can submit a `CryptoCreate` transaction with a 20-byte EVM address alias — this is a standard, documented operation. The only cost is the Hedera transaction fee (fractions of a cent). The target's `evm_address` is publicly readable from the mirror node REST API. The attack is repeatable: if Account B is deleted, the attacker can create Account C with the same alias. No privileged access is required.

### Recommendation

1. **Add `LIMIT 1` with deterministic ordering** to the query to prevent `IncorrectResultSizeDataAccessException`:
   ```sql
   select * from entity
   where (evm_address = ?1 or alias = ?1) and deleted is not true
   order by id asc
   limit 1
   ```
2. **Separate the lookup paths**: use `evm_address`-only lookup when the input is exactly 20 bytes (matching `findByEvmAddressAndDeletedIsFalse`), and `alias`-only lookup otherwise. The combined OR query is only needed when the caller cannot distinguish the two cases.
3. **Add a partial unique index** on `evm_address` to prevent duplicate EVM addresses across active entities: `CREATE UNIQUE INDEX entity__evm_address_unique ON entity (evm_address) WHERE evm_address IS NOT NULL AND deleted IS NOT TRUE;`
4. **Fix the cache key**: `@spelHelper.hashCode(#alias)` uses `Arrays.hashCode()`, a 32-bit hash susceptible to collisions. Use a collision-resistant key (e.g., `Arrays.toString(#alias)` or a hex encoding).
