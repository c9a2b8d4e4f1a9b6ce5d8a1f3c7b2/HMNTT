### Title
Unauthenticated, Rate-Limit-Free DB Query Flooding via `account(evmAddress)` GraphQL Endpoint

### Summary
The GraphQL `account()` query with `evmAddress` input is accessible to any unauthenticated user with no per-request or per-IP rate limiting at either the application or infrastructure layer. Every syntactically valid `evmAddress` query bypasses all existing guards and triggers a live database lookup, allowing an attacker to exhaust the DB connection pool and cause service unavailability through sustained flooding.

### Finding Description

**Exact code path:**

`AccountController.account()` receives an `evmAddress` string, passes it to `EntityServiceImpl.getByEvmAddressAndType()`, which unconditionally calls either `entityRepository.findById()` or `entityRepository.findByEvmAddress()` — both are live DB queries with no result caching. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — failed assumption:** The only guards in `GraphQlConfiguration` are query complexity (max 200) and depth (max 10) instrumentation, plus parser-level character/token limits. [4](#0-3) 

These limits prevent a single *large* or *deeply nested* query, but they do not limit the *rate* of simple, valid queries. A minimal `account(input:{evmAddress:"0x..."})` query is well within all complexity and depth budgets and can be sent at arbitrary frequency.

**No application-level rate limiting exists in the `graphql` module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter lives exclusively in the `web3` module and is wired only to `ContractCallRequest` processing — it is never invoked for GraphQL requests. [5](#0-4) 

**No infrastructure-level rate limiting exists for the GraphQL service.** The Helm chart for `hedera-mirror-graphql` defines no Traefik/ingress rate-limit middleware. By contrast, the `hedera-mirror-rosetta` chart explicitly configures `rateLimit: average: 10` per host. [6](#0-5) 

**The `CachedPreparsedDocumentProvider` caches only the parsed query *document*, not query results.** Every request with a different `evmAddress` variable value hits the DB. [7](#0-6) 

**The `@Pattern` directive on `evmAddress` uses `OnValidationErrorStrategy.RETURN_NULL`.** For input fields this means a pattern-failing value is silently nulled out rather than hard-rejected at the HTTP layer, so the attacker simply sends pattern-conforming addresses (trivially generated) and every request reaches the DB. [8](#0-7) [9](#0-8) 

### Impact Explanation

An attacker flooding the endpoint with valid `evmAddress` queries exhausts the HikariCP connection pool (the Helm chart itself monitors `hikaricp_connections_active / hikaricp_connections_max > 0.75` as a critical alert). Once the pool is saturated, all GraphQL queries — including legitimate ones — queue or fail. This constitutes full service unavailability for the GraphQL API. Because the mirror node is a public read API, no account or credentials are required to trigger this. [10](#0-9) 

### Likelihood Explanation

Preconditions: network reachability to the GraphQL endpoint (publicly exposed). No credentials, no privileged access, no prior knowledge of valid addresses needed — the attacker can cycle through the long-zero EVM address space (`0x000000000000000000000000<8-byte-entity-id>`) to guarantee each request hits the `findById` path, or send random 20-byte hex strings to hit `findByEvmAddress`. A single machine with a persistent HTTP/2 connection can sustain thousands of requests per second. The attack is repeatable and requires no special tooling beyond `curl` or `wrk`.

### Recommendation

1. **Add application-level rate limiting to the `graphql` module** using the same Bucket4j pattern already present in `web3/ThrottleConfiguration` — a global requests-per-second bucket and optionally a per-IP bucket via a servlet filter or Spring Security filter chain.
2. **Add infrastructure-level rate limiting** to the `hedera-mirror-graphql` Helm chart (Traefik `rateLimit` middleware), mirroring what `hedera-mirror-rosetta` already does.
3. **Add result-level caching** (e.g., Caffeine with a short TTL) in `EntityServiceImpl` for `getByEvmAddressAndType` to reduce DB pressure from repeated identical lookups.

### Proof of Concept

```bash
# Generate 10,000 valid evmAddress queries at maximum rate
# No authentication required
for i in $(seq 1 10000); do
  ADDR=$(printf "0x%024d%016x" 0 $i)
  curl -s -X POST https://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"$ADDR\\\"}){entityId{num}} }\"}" &
done
wait
# Result: HikariCP connection pool exhausted; subsequent legitimate queries
# receive connection timeout errors or HTTP 500 responses.
```

Each request is within query complexity budget (complexity = 2, depth = 2), passes the `@Pattern` regex, and triggers a live `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` or `SELECT * FROM entity WHERE id = ?` against the database.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);

        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
