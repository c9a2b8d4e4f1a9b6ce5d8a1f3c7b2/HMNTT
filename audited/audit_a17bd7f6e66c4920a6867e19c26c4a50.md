### Title
Malicious Block Node Can Cause Intermittent Ingestion Stalls via Deliberate `serverStatus()` Response Timing Near Deadline

### Summary
`BlockNode.getBlockRange()` applies a 400 ms gRPC deadline to `serverStatus()` calls but catches all deadline-exceeded exceptions silently, returning `EMPTY_BLOCK_RANGE` without ever calling `onError()`. Because `onError()` is the sole mechanism that marks a node inactive, a block node operator who deliberately delays responses to the boundary of `responseTimeout` can cause intermittent `EMPTY_BLOCK_RANGE` returns with zero penalty, making the node permanently "active-but-useless" and causing repeated ingestion stalls when it is the highest-priority or only configured node.

### Finding Description

**Exact code path:**

`StreamProperties.java:35` — default `responseTimeout = Duration.ofMillis(400)`, minimum enforced at 100 ms.

`BlockNode.java:99-112` — `getBlockRange()`:
```java
public Range<Long> getBlockRange() {
    try {
        final var blockNodeService = BlockNodeServiceGrpc.newBlockingStub(channel)
                .withDeadlineAfter(streamProperties.getResponseTimeout());   // 400 ms deadline
        final var response = blockNodeService.serverStatus(SERVER_STATUS_REQUEST);
        ...
    } catch (Exception ex) {
        log.error("Failed to get server status for {}", this, ex);
        return EMPTY_BLOCK_RANGE;   // silent fallback, no onError() call
    }
}
```

`BlockNode.java:196-206` — `onError()` (the only path to `active = false`) is called exclusively from `streamBlocks()`, never from `getBlockRange()`.

`BlockNodeSubscriber.java:133-145` — `hasBlock()` calls `getBlockRange()` and returns `false` when the range is empty.

`BlockNodeSubscriber.java:108-131` — `getNode()` iterates all active nodes calling `hasBlock()`. If every active node returns `false`, it falls through to inactive nodes (also calling `hasBlock()`), and if still none, throws `BlockStreamException("No block node can provide block …")`.

**Root cause:** The failed assumption is that a `serverStatus()` timeout is a transient network event rather than a deliberate, repeatable attack. Because `onError()` is never invoked on a `getBlockRange()` failure, the node's `active` flag and error counter are untouched. The node remains permanently eligible for selection, consuming up to `responseTimeout` ms of wall-clock time on every `getNode()` invocation while always returning `EMPTY_BLOCK_RANGE`.

**Why existing checks fail:** The `maxSubscribeAttempts` / `readmitDelay` circuit-breaker only fires inside `streamBlocks()`. A node that never reaches the streaming phase (because it always times out at the status check) is invisible to the circuit-breaker entirely.

### Impact Explanation

- **Single malicious node (only configured node):** Every call to `getNode()` waits the full 400 ms for the status timeout, then throws `BlockStreamException`. The importer cannot ingest any block; ingestion stalls indefinitely.
- **Malicious node among multiple nodes:** The malicious node is tried first (if it has the highest priority or appears first in config order). Each `getNode()` call wastes up to 400 ms before falling through to a healthy node. At one block per ingestion cycle, this adds a sustained 400 ms latency floor per block, degrading throughput.
- **Intermittent variant:** If the response arrives at exactly the deadline boundary, behavior is non-deterministic under system load. Sometimes the node appears to have the block (response arrives just before deadline), sometimes not (just after). This produces unpredictable node selection and sporadic `BlockStreamException` throws that are indistinguishable from legitimate node failures.

Severity matches the stated scope: griefing / availability degradation with no direct economic damage to network users.

### Likelihood Explanation

A block node operator needs to be either statically configured in the importer's config file or registered in the `registered_node` table (auto-discovery path). In a permissioned deployment this requires admin action; in a semi-open deployment where node registration is accessible to participants, any registered node operator can execute this attack. The timing manipulation is trivial: the server-side gRPC handler simply sleeps for `responseTimeout - ε` ms before responding. No cryptographic material, no special network position, and no exploit of a memory-safety bug is required. The attack is fully repeatable and requires no ongoing effort once the delay is configured.

### Recommendation

1. **Call `onError()` (or a dedicated status-check penalty) when `getBlockRange()` fails.** A node that consistently fails `serverStatus()` should accumulate error counts and eventually be marked inactive, just as streaming failures do.
2. **Separate the status-check failure counter from the streaming failure counter** if desired, but ensure both paths feed into the circuit-breaker.
3. **Increase the default `responseTimeout` margin** or make the deadline adaptive so that a response arriving at exactly the boundary does not produce non-deterministic behavior under load.
4. **Add a maximum consecutive `getBlockRange()` failure threshold** after which the node is temporarily excluded from `getNode()` selection, independent of `streamBlocks()` errors.

### Proof of Concept

**Preconditions:**
- Attacker controls a block node that is configured (statically or via auto-discovery) in the importer.
- `responseTimeout` is at its default of 400 ms.

**Steps:**

1. On the malicious block node, implement `serverStatus()` to sleep for 395–400 ms before returning a valid `ServerStatusResponse`.
2. Start the importer with only this block node configured (or with it as the highest-priority node).
3. Observe that every call to `BlockNodeSubscriber.doGet()` → `getNode()` → `hasBlock()` → `BlockNode.getBlockRange()` either:
   - Times out (response arrives after deadline) → returns `EMPTY_BLOCK_RANGE` → `hasBlock()` returns `false`
   - Succeeds intermittently under load → `hasBlock()` returns `true` → streaming begins but can be aborted similarly
4. With a single node: `getNode()` throws `BlockStreamException("No block node can provide block N")` on every cycle. Ingestion stalls.
5. Confirm via logs: repeated `"Failed to get server status for BlockNode(…)"` errors, but `node.isActive()` remains `true` and the error counter in `BlockNode.errors` stays at 0 — the circuit-breaker is never triggered.
6. With multiple nodes: measure per-block ingestion latency; observe a consistent ~400 ms overhead attributable to the malicious node's status timeout on each cycle.
