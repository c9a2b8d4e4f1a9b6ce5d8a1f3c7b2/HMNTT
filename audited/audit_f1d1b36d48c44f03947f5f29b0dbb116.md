### Title
Unbounded Per-IP gRPC Connection Exhaustion Leading to `applicationTaskExecutor` Thread Pool DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures `NettyServerBuilder` with only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) but applies no limit on the number of connections accepted from a single IP address. An unauthenticated attacker can open an unbounded number of TCP/HTTP2 connections from a single IP, each carrying up to 5 concurrent gRPC calls, exhausting the shared `applicationTaskExecutor` thread pool and rendering the gRPC API unavailable to all legitimate clients.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:

```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` exposes `maxConnectionsPerIp(int)` and `maxConnections(int)` methods that are never called. The only restriction applied is `maxConcurrentCallsPerConnection(5)`, which is a per-connection call-concurrency cap, not a per-source-IP or global connection count cap. A `grep` across all Java sources confirms zero calls to `maxConnectionsPerIp`, `maxConnections`, `maxConnectionAge`, or `maxConnectionIdle`.

**Why existing checks fail:** `maxConcurrentCallsPerConnection = 5` limits how many simultaneous RPC calls can be in-flight on a single HTTP/2 connection. It does not prevent an attacker from opening 10,000 separate connections. Each of those connections is independently entitled to 5 concurrent calls, yielding up to 50,000 concurrent tasks submitted to the single shared `applicationTaskExecutor`. The Traefik ingress middleware (`charts/hedera-mirror-grpc/values.yaml` lines 156–161) only configures a circuit breaker and retry policy — no connection-rate or connection-count limiting per source IP. The nginx proxy in `docker-compose.yml` (lines 226–229) similarly has no `limit_conn` directive for the gRPC upstream. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The `applicationTaskExecutor` is the single shared Spring Boot thread pool that dispatches all gRPC call handling (topic subscription streaming, address book queries, reflection). Exhausting it causes every subsequent legitimate gRPC call to queue indefinitely or be rejected, making the mirror node's gRPC API completely unavailable. Downstream clients (wallets, dApps, SDKs) that rely on `subscribeTopic` or `getNodes` lose service. Because gRPC connections are long-lived HTTP/2 streams, the attacker's connections persist without sending any data, holding thread-pool slots open continuously with minimal bandwidth cost. [4](#0-3) 

### Likelihood Explanation

No authentication or API key is required to open a gRPC connection to port 5600. The attack requires only a standard gRPC client library and a single machine with enough file descriptors (ulimit). Opening thousands of HTTP/2 connections is trivially scriptable. The absence of any ingress-level connection-count rate limiting (confirmed in both the Helm chart and docker-compose configurations) means there is no compensating control between the attacker and the vulnerable `NettyServerBuilder`. The attack is repeatable and persistent as long as the attacker holds connections open. [5](#0-4) [6](#0-5) 

### Recommendation

1. **Add `maxConnections` and/or `maxConnectionsPerIp`** to `NettyServerBuilder` inside `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionsPerIp(nettyProperties.getMaxConnectionsPerIp());
   ```
   Add the corresponding field to `NettyProperties` with a safe default (e.g., 10–50).

2. **Add `maxConnectionAge` and `maxConnectionIdle`** to reclaim connections from idle or long-lived clients:
   ```java
   serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
   ```

3. **Add ingress-level rate limiting** in the Traefik middleware (e.g., `InFlightReq` or `RateLimit`) to cap connections per source IP before they reach the application. [7](#0-6) [8](#0-7) 

### Proof of Concept

```python
import grpc
import threading

TARGET = "grpc-mirror-node-host:5600"
NUM_CONNECTIONS = 5000

channels = []
stubs = []

def open_connection(_):
    ch = grpc.insecure_channel(TARGET)
    channels.append(ch)
    # Initiate a streaming call to hold the connection open
    # (subscribeTopic with a valid topic ID keeps the stream alive)
    # Each connection consumes executor threads up to maxConcurrentCallsPerConnection=5

threads = [threading.Thread(target=open_connection, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# At this point, 5000 connections × 5 concurrent calls = up to 25,000 tasks
# queued in applicationTaskExecutor. Legitimate clients receive RESOURCE_EXHAUSTED
# or indefinite hangs on new calls.
print(f"Opened {len(channels)} connections. Server applicationTaskExecutor is saturated.")
```

Preconditions: network access to port 5600, no authentication required.
Trigger: run the script.
Result: `applicationTaskExecutor` saturated; all new legitimate gRPC calls time out or are rejected.
