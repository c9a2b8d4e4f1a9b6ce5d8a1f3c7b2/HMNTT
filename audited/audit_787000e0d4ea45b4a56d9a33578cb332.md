### Title
`Optional.empty()` Cached by `findRuntimeBytecode()` Due to Insufficient `unless` Condition Enables Persistent Missing Bytecode via Race on Contract Deployment

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`, which does not exclude `Optional.empty()` from being cached — only a literal `null` return is excluded. Because contract entity creation and `runtime_bytecode` insertion are two separate database operations, an unprivileged user who queries during that window receives a cached `Optional.empty()` that persists for up to 1 hour (`expireAfterAccess=1h`), causing the mirror node to permanently serve missing bytecode for a legitimately deployed contract until cache expiry.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's `@Cacheable` `unless` SpEL expression `"#result == null"` only suppresses caching when the method returns a literal Java `null`. When the DB row is absent or `runtime_bytecode` is `NULL`, the JPA/native query returns `Optional.empty()` — a non-null object — so the condition evaluates to `false` and the empty Optional **is stored in the cache**.

**Two-phase write window:** `SidecarContractMigration.migrate()` (`importer/src/main/java/org/hiero/mirror/importer/migration/SidecarContractMigration.java`, lines 44–48) performs an `INSERT … ON CONFLICT DO UPDATE SET runtime_bytecode = …` as a separate batch operation after the contract entity row already exists. There is a measurable window between the entity row being committed and the `runtime_bytecode` column being populated.

**Cache TTL:** `CacheProperties.java` line 22 sets `contract = "expireAfterAccess=1h,maximumSize=1000,recordStats"`. A cached `Optional.empty()` survives for 1 hour after last access. If the attacker periodically re-queries the same contract ID, the entry is refreshed and the stale empty result persists indefinitely.

**Downstream effect:** `ContractBytecodeReadableKVState.readFromDataSource()` (`web3/src/main/java/org/hiero/mirror/web3/state/keyvalue/ContractBytecodeReadableKVState.java`, lines 42–46) maps the result directly — an empty Optional produces `null`, which the EVM layer treats as a non-existent contract.

### Impact Explanation
Any `eth_getCode` call, EVM simulation, or contract-call that relies on bytecode lookup for the targeted contract will receive an empty/zero response for up to 1 hour (or longer with access-based refresh). This constitutes an incorrect/missing record exported by the mirror node for a legitimately deployed contract. Downstream integrations (block explorers, DeFi protocols, contract verifiers) will incorrectly treat the contract as non-existent or undeployed.

### Likelihood Explanation
No privileges are required. Contract deployments on Hedera are public; the contract ID is visible in the transaction record immediately after submission. The deployment-to-sidecar-write window is on the order of seconds (one record file interval), which is easily targeted by automated tooling. The attack is repeatable for every new contract deployment and requires only a standard JSON-RPC `eth_getCode` or equivalent call.

### Recommendation
Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a populated `Optional<byte[]>` is stored in the cache, and a missing-bytecode result always triggers a fresh DB query on the next call.

Alternatively, switch the cache to `expireAfterWrite` with a short TTL (e.g., 1–2 seconds, consistent with the entity/token caches) so that any transiently cached empty result expires quickly.

### Proof of Concept
1. Monitor the Hedera network (or a local mirror node test environment) for a `ContractCreate` transaction.
2. Immediately after the transaction reaches consensus (contract entity row committed, before the sidecar record file is processed), issue:
   ```
   eth_getCode <contractAddress> latest
   ```
   via the mirror node's JSON-RPC endpoint.
3. Observe that the mirror node queries the DB, finds no `runtime_bytecode`, and returns `0x` (empty).
4. Wait for the importer to process the sidecar record and populate `runtime_bytecode` in the DB.
5. Re-issue `eth_getCode` — the mirror node still returns `0x` because `Optional.empty()` is served from the Caffeine cache.
6. Confirm the DB now contains the correct bytecode (`SELECT runtime_bytecode FROM contract WHERE id = <id>`).
