### Title
Unauthenticated gRPC `subscribeTopic()` Endpoint Allows Distributed Subscription Flooding, Rendering IP-Based Rate Limiting Ineffective

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` requires no authentication and imposes no global connection or subscriber limit. An attacker controlling distributed clients across many source IPs can open an unbounded number of long-lived streaming subscriptions, exhausting server-side resources (reactive threads, database connections, memory) while making any IP-based rate limiting at the infrastructure layer entirely ineffective, since each attacking IP contributes only a small fraction of the total load.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, `subscribeTopic()`, lines 43â€“53:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

```

No authentication token, API key, or identity check is performed before the subscription is established.

**Root cause â€” failed assumptions:**

1. **No authentication**: The only production `ServerInterceptor` found (`grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`) is a test-only class that only sets an endpoint context label; there is no production interceptor enforcing identity or credentials.

2. **`maxConcurrentCallsPerConnection = 5` is per-connection, not global**: `GrpcConfiguration.java` line 33 calls `serverBuilder.maxConcurrentCallsPerConnection(5)`. This limits calls *per TCP connection* but places no cap on the total number of simultaneous connections or the aggregate subscriber count across all connections.

3. **`subscriberCount` is a metric, not a guard**: `TopicMessageServiceImpl.java` lines 89â€“90 increment/decrement an `AtomicLong subscriberCount` gauge, but this value is only exported to Micrometer for observability â€” it is never compared against a maximum to reject new subscriptions.

4. **No IP-based rate limiting in the gRPC layer**: The `ThrottleConfiguration` / `Bucket4j` rate limiting found in the codebase exists only in the `web3` module; there is no equivalent in the `grpc` module.

**Exploit flow:**

- Attacker deploys N clients across N distinct source IPs (cloud VMs, residential proxies, or a botnet).
- Each client calls `subscribeTopic` with `limit = 0` (no limit, streams indefinitely per the proto definition) and a valid `topicID`.
- Each subscription opens a reactive `Flux` pipeline backed by a `TopicListener` (Redis or polling) and a database retriever, consuming a thread from the bounded elastic scheduler, a database connection from the HikariCP pool, and heap memory for the `TopicContext` and message buffers.
- Because each IP contributes only 1â€“5 subscriptions, any IP-based block list or rate limiter at the load balancer/ingress layer would need to enumerate every attacker IP individually â€” operationally infeasible during an active distributed attack.
- The server has no mechanism to distinguish this traffic from legitimate subscribers.

### Impact Explanation
Each open subscription holds resources for its lifetime: a Reactor scheduler thread slot, a HikariCP database connection (the Helm chart alert fires at 75% pool utilization), and heap for the `TopicContext` buffer. With enough distributed subscribers, the database connection pool is exhausted (alerting threshold documented in `charts/hedera-mirror-grpc/values.yaml` line 214), the bounded elastic thread pool saturates, and legitimate subscribers receive errors or timeouts. The impact is service degradation or denial of service for all users of the HCS subscribe API, with no economic cost to the attacker.

### Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond the publicly documented gRPC endpoint and protobuf schema (documented in `docs/grpc/README.md` with a working `grpcurl` example). Distributed execution across many IPs is achievable with commodity cloud infrastructure or free-tier VMs. The subscription is long-lived by design (`limit = 0` streams indefinitely), so each attacker connection persists and consumes resources continuously. The attack is fully repeatable and requires no coordination beyond launching clients.

### Recommendation
1. **Add a global maximum subscriber count**: In `TopicMessageServiceImpl.subscribeTopic()`, compare `subscriberCount.get()` against a configurable ceiling before accepting a new subscription; return `RESOURCE_EXHAUSTED` if exceeded.
2. **Enforce per-IP connection limits**: Extend `GrpcConfiguration` to extract the remote peer address via `io.grpc.Grpc.TRANSPORT_ATTR_REMOTE_ADDR` in a `ServerInterceptor` and apply a per-IP token-bucket rate limiter (e.g., Bucket4j, already used in the `web3` module).
3. **Add optional API-key authentication**: Implement a `ServerInterceptor` that reads a metadata header (e.g., `x-api-key`) and rejects unauthenticated calls or applies stricter limits to them, consistent with the REST API's `authHandler` pattern.
4. **Set `maxConnectionAge`**: Configure `NettyServerBuilder.maxConnectionAge()` in `GrpcConfiguration` to recycle long-lived connections and prevent indefinite resource hold.
