### Title
`incrementByteArray()` Overflow on MAX_KEY_BYTES Causes Full Key-Space Scan Instead of Empty Result

### Summary
When an unprivileged user supplies `key=gt:0xFFFF...FFFF` (32 bytes of `0xFF`), `SlotRangeParameter.valueOf()` calls `incrementByteArray()` on the maximum possible key value. The Tuweni `MutableBytes32.increment()` wraps around to all-zeros (`0x0000...0000`). The subsequent comparison in `hookStorageChangeRequest()` silently discards this overflowed value, leaving `lowerBound` at `MIN_KEY_BYTES` (also all-zeros), so the query scans the entire key space and returns all storage entries instead of an empty result.

### Finding Description

**Exact code path:**

1. **`SlotRangeParameter.valueOf()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/parameter/SlotRangeParameter.java`, lines 35–45:
   - Input `gt:0xFFFF...FFFF` is parsed; `operator = GT`, `hex = "FFFF...FFFF"` (64 chars, 32 bytes of `0xFF`).
   - `operator.toInclusive()` converts `GT` → `GTE` (stored on the record).
   - `getInclusiveValue(GT, hex)` is called; because `operator == RangeOperator.GT`, it calls `incrementByteArray(bytes)`.

2. **`BytesUtil.incrementByteArray()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/util/BytesUtil.java`, line 14:
   ```java
   return MutableBytes32.wrap(bytes).increment().toArrayUnsafe();
   ```
   `MutableBytes32.increment()` on `0xFFFF...FFFF` (32 bytes) wraps around to `0x0000...0000` — standard fixed-width unsigned overflow in the Tuweni library.

3. **`hookStorageChangeRequest()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, lines 167–183:
   ```java
   var lowerBound = MIN_KEY_BYTES;   // all zeros
   var upperBound = MAX_KEY_BYTES;   // all 0xFF
   ...
   } else if (Arrays.compareUnsigned(value, lowerBound) > 0) {
       lowerBound = value;
   }
   ```
   After overflow, `value = 0x0000...0000` and `lowerBound = MIN_KEY_BYTES = 0x0000...0000`. The comparison `compareUnsigned(zeros, zeros) > 0` evaluates to `false`, so `lowerBound` is **never updated**. The request is built with `keyLowerBound = MIN_KEY_BYTES` and `keyUpperBound = MAX_KEY_BYTES` — the full key space.

**Root cause:** No overflow/saturation guard exists in `incrementByteArray()` or in the comparison logic. The code assumes `incrementByteArray(MAX_KEY_BYTES)` produces a value strictly greater than `MIN_KEY_BYTES`, but it produces an equal value, silently defeating the filter.

### Impact Explanation
A `gt:MAX_KEY` query should return zero results (no key can exceed the maximum). Instead, the API returns up to `limit` storage entries spanning the entire key space for the targeted hook. An attacker can enumerate all contract storage slots for any hook ID without knowing valid key ranges, bypassing the intended key-range filter. This is an unauthenticated information-disclosure issue affecting every hook's storage endpoint.

### Likelihood Explanation
No authentication or special privilege is required — the endpoint is a plain `GET` request. The payload is a single, trivially constructed query parameter. The attack is deterministic and repeatable with zero cost. Any user who can reach the API can exploit it.

### Recommendation
Add an overflow guard in `getInclusiveValue()` inside `SlotRangeParameter`:

```java
private static byte[] getInclusiveValue(RangeOperator operator, String hexValue) throws DecoderException {
    byte[] bytes = Hex.decodeHex(hexValue);

    if (operator == RangeOperator.GT) {
        // If bytes == MAX_KEY_BYTES, incrementing wraps to zero — return a sentinel
        // that signals "no valid lower bound exists" (i.e., empty result).
        if (Arrays.equals(bytes, MAX_KEY_BYTES)) {
            throw new IllegalArgumentException(
                "gt: value exceeds maximum key; no results possible");
        }
        return incrementByteArray(bytes);
    } else if (operator == RangeOperator.LT) {
        if (Arrays.equals(bytes, MIN_KEY_BYTES)) {
            throw new IllegalArgumentException(
                "lt: value is below minimum key; no results possible");
        }
        return decrementByteArray(bytes);
    }
    return bytes;
}
```

Alternatively, after `incrementByteArray`, assert the result is strictly greater than the input; if not, reject the parameter or return an empty response immediately.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF
```

**Expected:** `[]` (empty storage list — no key can be greater than `0xFF...FF`)

**Actual:** Returns up to `limit` storage entries covering the full key range `[0x00...00, 0xFF...FF]`, disclosing all storage slots for hook `1`.

**Step-by-step:**
1. `SlotRangeParameter.valueOf("gt:0xFFFF...FFFF")` parses successfully (pattern allows 64 hex chars).
2. `incrementByteArray(0xFF...FF)` → `MutableBytes32.increment()` wraps to `0x00...00`.
3. Record stored: `operator=GTE, value=0x00...00`.
4. In `hookStorageChangeRequest()`: `compareUnsigned(0x00...00, MIN_KEY_BYTES) > 0` → `false`; `lowerBound` unchanged.
5. `HookStorageRequest` built with `keyLowerBound=MIN_KEY_BYTES`, `keyUpperBound=MAX_KEY_BYTES`.
6. Service query returns all storage entries for the hook.
