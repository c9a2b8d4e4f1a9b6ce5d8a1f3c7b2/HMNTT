### Title
Unprivileged Attacker Can Open Traefik Circuit Breaker via inFlightReq-Generated 503s, Causing Global Rosetta Node Denial of Service

### Summary
The Traefik middleware chain for the Rosetta node places `circuitBreaker` as the outermost wrapper around `inFlightReq`. When `inFlightReq` rejects excess concurrent requests with HTTP 503, the circuit breaker's `ResponseCodeRatio(500, 600, 0, 600)` counter increments for those 503s. Because the circuit breaker is a global state machine, any unprivileged attacker sending more than 5 concurrent requests from a single IP can push the 5xx ratio above 0.25 and open the circuit breaker, denying service to all users.

### Finding Description

**Exact code locations:**

`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166: [1](#0-0) 

`charts/hedera-mirror-rosetta/templates/middleware.yaml`, lines 11–15 — the Helm template renders these as a Traefik `chain` middleware in listed order: [2](#0-1) 

**Middleware chain order (outermost → innermost → backend):**
```
circuitBreaker → inFlightReq → rateLimit → retry → stripPrefix → rosetta backend
```

**Root cause:** In a Traefik `chain`, the first-listed middleware is the outermost handler. The `circuitBreaker` wraps the entire remaining chain. Traefik's circuit breaker instruments the `next` handler by wrapping the `http.ResponseWriter` with a recorder; whatever status code the next handler writes — including a 503 emitted by `inFlightReq` — is recorded and fed into `ResponseCodeRatio(500, 600, 0, 600)`.

**Failed assumption:** The design assumes `inFlightReq`'s 503 rejections are invisible to the circuit breaker. They are not — `inFlightReq` is inside the circuit breaker's measurement boundary.

**Exploit flow:**

1. Attacker opens ≥10 concurrent HTTP connections to any Rosetta ingress path (e.g., `POST /rosetta/network/list`). No authentication is required.
2. `inFlightReq` allows the first 5 concurrent requests per source IP (`amount: 5`, `ipStrategy.depth: 1`) and immediately returns HTTP 503 for the remaining 5+.
3. The circuit breaker's recorder captures all responses. With 5 successes and 5 503s: `ResponseCodeRatio = 5/10 = 0.50 > 0.25`.
4. The circuit breaker transitions to **open** state.
5. All subsequent requests from **all users** receive HTTP 503 from the circuit breaker until the recovery window elapses (Traefik default: 10 seconds), after which the attacker can immediately re-trigger it.

**Additional amplifier — `retry` middleware:** With `attempts: 3`, each backend-level 5xx causes 3 backend requests, all counted as 5xx by the circuit breaker. An attacker who can cause even a single backend error gets a 3× amplification of the 5xx count. [3](#0-2) 

**Why existing checks fail:**

- `inFlightReq` (`amount: 5`, per-IP): This is the mechanism that *generates* the 503s. It does not prevent them from being observed by the circuit breaker.
- `rateLimit` (`average: 10`, per `requestHost`): Limits sustained throughput globally but does not prevent a burst of >5 simultaneous connections, which is all that is needed. The attacker only needs one burst, not a sustained rate.
- No authentication on the Rosetta API: `rosetta/app/middleware/metrics.go` and `rosetta/main.go` show no auth middleware is applied to any Rosetta route. [4](#0-3) [5](#0-4) 

### Impact Explanation
When the circuit breaker opens, Traefik returns HTTP 503 to **all** clients for the entire rosetta service — not just the attacker. The Rosetta API is used by Coinbase and exchange integrators for blockchain data queries; a sustained circuit-breaker open state means complete service unavailability. The attacker can sustain the outage indefinitely by re-triggering the burst every ~10 seconds (Traefik's default recovery window), requiring no more than a single IP and a simple HTTP client. This is a complete, externally-triggerable denial of service with no privilege requirement.

### Likelihood Explanation
The precondition is zero: any internet-accessible Rosetta deployment is vulnerable. Sending 10 concurrent HTTP POST requests is trivially achievable with `curl --parallel`, `ab`, `wrk`, or a simple script. The attack is repeatable, stateless, and requires no knowledge of the application beyond the public endpoint paths listed in the ingress configuration. The `rateLimit` of 10 req/s per host does not block this because the attacker only needs a single burst of concurrent connections, not a sustained rate. [6](#0-5) 

### Recommendation

1. **Move `circuitBreaker` inside `inFlightReq`** in the chain order so that `inFlightReq`-generated 503s are never observed by the circuit breaker:
   ```yaml
   middleware:
     - inFlightReq:
         amount: 5
         sourceCriterion:
           ipStrategy:
             depth: 1
     - circuitBreaker:
         expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
     - rateLimit: ...
     - retry: ...
     - stripPrefix: ...
   ```
2. **Add per-IP rate limiting** (not just per-host) so a single IP cannot exhaust the inFlightReq budget repeatedly.
3. **Scope `ResponseCodeRatio` to backend-only errors** by ensuring the circuit breaker only wraps the backend service, not other protective middlewares.
4. **Increase `inFlightReq.amount`** or add a `burst` parameter to `rateLimit` to reduce the probability of legitimate traffic triggering the condition.

### Proof of Concept

```bash
# Target: any publicly accessible Rosetta ingress host
HOST="https://<rosetta-host>"

# Send 10 concurrent requests (5 will pass inFlightReq, 5 will get 503)
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST "$HOST/rosetta/network/list" \
    -H "Content-Type: application/json" \
    -d '{"metadata":{}}' &
done
wait

# Expected: ~5 responses with 200/500, ~5 responses with 503 (from inFlightReq)
# After this burst, the circuit breaker is open.

# Verify circuit breaker is open — all subsequent requests return 503:
curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST "$HOST/rosetta/network/list" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{}}'
# Expected output: 503

# Re-trigger every 9 seconds to sustain the outage indefinitely:
while true; do
  for i in $(seq 1 10); do
    curl -s -o /dev/null \
      -X POST "$HOST/rosetta/network/list" \
      -H "Content-Type: application/json" \
      -d '{"metadata":{}}' &
  done
  wait
  sleep 9
done
```
