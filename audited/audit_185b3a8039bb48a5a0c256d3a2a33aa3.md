### Title
Thread-Exhaustion DoS via Uncircuit-Broken `findRuntimeBytecode()` During DB Network Partition

### Summary
`ContractRepository.findRuntimeBytecode()` has no application-level circuit breaker and the web3 Helm chart is the only service missing a Traefik circuit-breaker middleware. During a DB network partition, an unprivileged attacker sending requests with unique contract IDs at a rate exceeding `threads / statementTimeout` exhausts the Tomcat thread pool, rendering the web3 API completely unavailable to all users.

### Finding Description

**Exact code path:**

`findRuntimeBytecode()` is defined at: [1](#0-0) 

The `@Cacheable` annotation uses `unless = "#result == null"`. Because the return type is `Optional<byte[]>`, the method never returns a bare `null` — it returns either `Optional.of(bytes)` or `Optional.empty()`. Both are non-null, so **all results are cached**. However, an attacker using a different `contractId` per request guarantees a cache miss on every call, since the cache has `maximumSize=1000` and `expireAfterAccess=60m`: [2](#0-1) 

The cache spec default is `expireAfterAccess=60m,maximumSize=1000,recordStats`: [3](#0-2) 

Every cache miss causes a synchronous JDBC query. The web3 DB user has `statementTimeout=3000ms` (PostgreSQL-level): [4](#0-3) 

The `StatementInspector` in `HibernateConfiguration` only aborts a query if the **overall request** has already exceeded `requestTimeout=10000ms` — it does not prevent the thread from blocking during the DB call itself: [5](#0-4) 

**No circuit breaker exists at any layer for web3.** Every other service (graphql, grpc, rest-java, rest, rosetta) has a Traefik `circuitBreaker` middleware. The web3 chart has only `retry`: [6](#0-5) 

Compare with graphql, which has a circuit breaker: [7](#0-6) 

There is no Resilience4j or equivalent application-level circuit breaker on `findRuntimeBytecode()` or anywhere in the web3 repository layer.

**Exploit flow:**

The global rate limiter allows up to 500 requests/second through: [8](#0-7) 

The rate limiter check occurs inside the Tomcat thread (after thread allocation), so throttled requests release quickly, but requests that pass the limiter and hit a DB cache-miss block their thread for up to 3000ms. At only **67 RPS** of cache-miss requests (200 threads ÷ 3s), all Tomcat threads are saturated. The attacker only needs to sustain this rate — well below the 500 RPS cap — to maintain the DoS.

### Impact Explanation

During a DB network partition, the web3 `/api/v1/contracts/call` endpoint becomes completely unavailable. All Tomcat threads are held by requests waiting on DB connections that will never respond within the partition window. Legitimate contract call and gas estimation requests are rejected or queue-dropped. This is a full availability failure for the web3 API, which is the primary EVM-compatible interface of the mirror node.

### Likelihood Explanation

No authentication is required. The attacker needs only HTTP access to the public `/api/v1/contracts/call` endpoint. The attack requires a concurrent DB network partition (e.g., firewall rule, cloud network event, or attacker-induced TCP RST injection against the DB). The attacker does not need to know valid contract IDs — any sequence of unique numeric IDs suffices. The attack is repeatable and self-sustaining: once threads are exhausted, the service cannot recover until the partition ends or the process is restarted. The required request rate (67 RPS) is trivially achievable from a single machine.

### Recommendation

1. **Add a Traefik circuit-breaker middleware to the web3 Helm chart**, matching the pattern used by all other services (e.g., `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`).
2. **Add a Resilience4j `@CircuitBreaker` on `findRuntimeBytecode()`** (or on the `ContractBytecodeReadableKVState.readFromDataSource()` caller) to open the circuit after a threshold of DB timeouts and return a fallback immediately.
3. **Configure HikariCP `connectionTimeout`** to a low value (e.g., 1000ms) so threads are not held waiting for pool acquisition during a partition.
4. **Consider caching negative results with a short TTL** (e.g., 5s) to reduce DB pressure from repeated lookups of non-existent contracts, even outside partition scenarios.

### Proof of Concept

**Preconditions:** DB network partition is active (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the web3 pod, or a network policy change).

**Steps:**

```bash
# Generate unique contract IDs to guarantee cache misses
# Send at 100 RPS (well above the 67 RPS exhaustion threshold)
for i in $(seq 1 10000); do
  curl -s -X POST http://<web3-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"0x\",\"to\":\"0x$(printf '%040x' $i)\",\"gas\":50000}" &
  # Rate: ~100 concurrent requests per second
  [ $((i % 100)) -eq 0 ] && sleep 1
done
```

**Expected result:** After ~2 seconds, all Tomcat threads are blocked. Subsequent legitimate requests receive connection refused or HTTP 503. The service remains unavailable for the duration of the partition. Monitoring will show `hikaricp_connections_pending` saturated and `tomcat_threads_busy_threads` at maximum. [9](#0-8)

```java
    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
```
