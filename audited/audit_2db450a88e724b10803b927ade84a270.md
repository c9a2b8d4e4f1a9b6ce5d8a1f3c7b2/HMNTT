### Title
Thundering Herd via Unguarded `cache.get()` Loader Invocation in `doGetTransactions()`

### Summary
`Cache.get()` in `rest/cache.js` has no in-flight request deduplication. When Redis is unavailable (`this.ready === false`) every concurrent request bypasses the cache entirely and calls the loader directly. Even when Redis is available, concurrent requests that all miss the same keys simultaneously each invoke `loader(missingKeys)` independently before any can populate the cache. An unprivileged attacker can exploit this by flooding `GET /transactions` with concurrent requests during a Redis outage or cold-cache window, causing a proportional burst of `getTransactionsDetails()` DB queries that exhausts the connection pool.

### Finding Description

**Code path:**

`rest/transactions.js` line 703–705 — the loader is defined inline and passed to `cache.get()`: [1](#0-0) 

`rest/cache.js` lines 95–140 — the `Cache.get()` implementation: [2](#0-1) 

**Root cause — two distinct failure modes, both in `Cache.get()`:**

**Mode 1 (Redis down / partition):** Lines 99–101:
```js
if (!this.ready) {
  return loader(keys);   // ← no dedup, no rate-limit, no circuit-break
}
``` [3](#0-2) 

Every concurrent request that arrives while `this.ready === false` calls `loader(keys)` independently. There is no promise map, no mutex, no semaphore, and no in-flight tracking. `N` concurrent HTTP requests produce exactly `N` concurrent `getTransactionsDetails()` DB queries.

**Mode 2 (cold cache / simultaneous miss):** Lines 112–113:
```js
if (missingKeys.length > 0) {
  const missing = await loader(missingKeys);
``` [4](#0-3) 

When Redis is available but the keys are not yet populated (e.g., after a Redis flush, restart, or first request for a timestamp range), all concurrent requests that reach the `mgetBuffer` check before any of them completes the `mset` write-back will all find the same keys missing and all call `loader()` simultaneously. The `mset` write-back on line 130 is fire-and-forget and does not prevent the race: [5](#0-4) 

**Exploit flow:**

1. Attacker sends `N` concurrent `GET /api/v1/transactions?timestamp=X` requests (identical or overlapping timestamp ranges).
2. If Redis is down: all `N` requests hit line 100 and each independently call `getTransactionsDetails()`.
3. If Redis is up but cold: all `N` requests complete `mgetBuffer` before any completes `mset`; all find `missingKeys.length > 0` and each independently call `getTransactionsDetails()`.
4. Each `getTransactionsDetails()` call issues a multi-CTE SQL query against the DB (joining `transaction`, `crypto_transfer`, `token_transfer`): [6](#0-5) 
5. `N` simultaneous heavy queries exhaust the DB connection pool; subsequent legitimate requests queue or fail.

### Impact Explanation

DB connection pool exhaustion causes all in-flight and queued DB queries to fail or time out, resulting in a full API denial-of-service. The `getTransactionsDetails()` query is a multi-table CTE join — it is not a trivial query. Even a modest `N` (e.g., 50–100 concurrent requests) can saturate a typical PostgreSQL connection pool. Because the `GET /transactions` endpoint is public and unauthenticated, no credentials are required.

### Likelihood Explanation

The attack requires no authentication, no special knowledge, and no exploit tooling beyond a standard HTTP load generator (e.g., `ab`, `wrk`, `hey`). The Redis-down scenario is the most severe: any Redis restart, network partition, or sentinel failover sets `this.ready = false` and opens the window. The cold-cache scenario is always exploitable on startup or after a `FLUSHALL`. Both conditions are realistic in production. The attack is repeatable and automatable.

### Recommendation

Implement **in-flight request coalescing** (also called "single-flight" or "request deduplication") in `Cache.get()`:

1. Maintain a `Map<string, Promise>` of in-flight loader calls keyed by the cache key (or a stable hash of the key set).
2. When a cache miss is detected, check the in-flight map first. If a promise already exists for that key, await it instead of calling `loader()` again.
3. Only call `loader()` once per unique key set; store the resulting promise in the map; remove it after resolution.
4. For the `!this.ready` fallback path, apply the same coalescing or add a concurrency limiter (e.g., a semaphore with a configurable max-concurrent-DB-queries cap) to prevent unbounded fan-out to the DB during Redis outages.

### Proof of Concept

  async get(keys, loader, keyMapper = (k) => (k ? k.toString() : k)) {
    if (isEmpty(keys)) {
      return [];
    }
    if (!this.ready) {
      return loader(keys);
    }

    const buffers =
      (await this.redis
        .mgetBuffer(map(keys, keyMapper))
        .catch((err) => logger.warn(`Redis error during mget: ${err.message}`))) || new Array(keys.length);
    const values = buffers.map((t) => JSONParse(t));

    let i = 0;
    const missingKeys = keys.filter(() => isNil(values[i++]));

    if (missingKeys.length > 0) {
      const missing = await loader(missingKeys);
      const newValues = [];
      let j = 0;

      missing.forEach((value) => {
        // Update missing values in Redis array
        for (; j < values.length; j++) {
          if (isNil(values[j])) {
            values[j] = value;
            newValues.push(keyMapper(keys[j]));
            newValues.push(JSONStringify(value));
            break;
          }
        }
      });

      if (newValues.length > 0) {
        this.redis.mset(newValues).catch((err) => logger.warn(`Redis error during mset: ${err.message}`));
      }
    }

    if (logger.isDebugEnabled()) {
      const count = keys.length - missingKeys.length;
      logger.debug(`Redis returned ${count} of ${keys.length} keys`);
    }

    return values;
  }
