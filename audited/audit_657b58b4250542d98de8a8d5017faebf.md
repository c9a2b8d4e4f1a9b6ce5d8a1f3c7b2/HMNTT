### Title
Unauthenticated Redis Write Enables Forged `runningHash` Injection to All gRPC Subscribers

### Summary
`RedisTopicListener.subscribe()` deserializes `TopicMessage` objects from a Redis pub/sub channel and forwards them directly to all downstream gRPC subscribers with no integrity verification of any field, including `runningHash`. An attacker who gains write access to the Redis channel (e.g., via an exposed, unauthenticated Redis port) can publish a crafted message with an arbitrary `runningHash`, which is then delivered verbatim to every connected gRPC client as a legitimate Hashgraph consensus response.

### Finding Description

**Code path:**

`RedisTopicListener.subscribe()` (lines 68–80) subscribes to a Redis channel and emits deserialized `TopicMessage` objects: [1](#0-0) 

The deserialized message flows through `CompositeTopicListener.listen()`, which applies only a `topicId` and `consensusTimestamp` filter — no `runningHash` check: [2](#0-1) 

`TopicMessageServiceImpl.subscribeTopic()` applies only a `sequenceNumber > last.getSequenceNumber()` deduplication filter — no `runningHash` check: [3](#0-2) 

`ConsensusController.toResponse()` directly maps `t.getRunningHash()` into the gRPC `ConsensusTopicResponse` with no validation: [4](#0-3) 

**Root cause:** The system implicitly trusts all data arriving on the Redis pub/sub channel. There is no cross-check of the `runningHash` (or any other field) against the authoritative database (`topic_message` table) before forwarding to gRPC clients.

**Serialization format:** Messages are encoded in MessagePack via `Jackson2JsonRedisSerializer<TopicMessage>` with a custom `ObjectMapper`: [5](#0-4) 

The `TopicMessage` domain object exposes `runningHash` as a plain `byte[]` field with no immutability or signing: [6](#0-5) 

### Impact Explanation

The `runningHash` field is the SHA-384 cumulative hash that proves the integrity of the entire HCS message sequence. It is the primary cryptographic anchor clients use to verify that Hashgraph history has not been tampered with. Injecting a forged `runningHash` via Redis causes every active gRPC subscriber to receive a falsified consensus history. Clients that rely on this value for audit, compliance, or chain-of-custody verification will silently accept fabricated data. The `ConsensusTopicResponse.runningHash` field is defined in the protobuf as "The running hash (SHA384) of every message" — its integrity is the protocol guarantee. [7](#0-6) 

### Likelihood Explanation

The Helm chart generates a random 40-character Redis password by default, which is a meaningful barrier in a correctly deployed Kubernetes environment: [8](#0-7) 

However, the attack is realistic in the following scenarios:
- Redis deployed without authentication (misconfigured `SPRING_DATA_REDIS_PASSWORD` left empty, or `requirepass` not set)
- Redis port exposed outside the cluster (misconfigured `NetworkPolicy`, cloud security group, or `LoadBalancer` service)
- Internal network attacker (compromised pod in the same namespace)
- The test configuration explicitly uses `ALLOW_EMPTY_PASSWORD=yes`, indicating this is a known operational mode [9](#0-8) 

Once Redis write access is obtained, the attack is trivially repeatable and affects all concurrent gRPC subscribers simultaneously.

### Recommendation

1. **Cross-validate against the database:** Before forwarding a live Redis message to gRPC clients, verify the `runningHash` (and `sequenceNumber`) against the `topic_message` table. This is already done for historical messages via `TopicMessageRetriever`; apply the same check to live messages.
2. **Sign Redis messages:** Have the importer HMAC-sign each published `TopicMessage` using a shared secret. Verify the signature in `RedisTopicListener.subscribe()` before emitting the message downstream.
3. **Enforce Redis authentication and network isolation:** Ensure `requirepass` is always set in production and that Redis is not reachable from outside the trusted service mesh.
4. **Reject messages with null/empty `runningHash`:** Add a minimum sanity filter in `subscribe()` to drop messages with a missing or zero-length `runningHash`.

### Proof of Concept

**Preconditions:** Redis is reachable on port 6379 with no password (or attacker has obtained the password).

**Steps:**

1. Serialize a crafted `TopicMessage` in MessagePack format matching the schema used by `Jackson2JsonRedisSerializer` (fields: `@type`, `consensus_timestamp`, `running_hash`, `sequence_number`, `topic_id`, etc.) with `running_hash` set to an arbitrary 48-byte value (e.g., all `0xFF`).

2. Publish to the target topic channel:
   ```
   redis-cli PUBLISH topic.<target_topic_id> <msgpack_encoded_payload>
   ```

3. Any gRPC client subscribed via `subscribeTopic` on that topic will immediately receive a `ConsensusTopicResponse` with the forged `runningHash` value, indistinguishable from a legitimate consensus message.

4. The sequence number filter in `TopicMessageServiceImpl` (line 76) can be satisfied by choosing a `sequenceNumber` greater than the last seen value, ensuring the message is not dropped. [1](#0-0) [4](#0-3)

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
