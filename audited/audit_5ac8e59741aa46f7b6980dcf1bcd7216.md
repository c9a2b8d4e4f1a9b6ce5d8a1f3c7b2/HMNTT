### Title
Timestamp Range Validation Bypass in `getOneAccount()` Enables DB-Level DoS via Unbounded Timestamp Scans

### Summary
`getOneAccount()` in `rest/accounts.js` calls `parseTimestampFilters` with `validateRange = false` and `allowOpenRange = true`, explicitly bypassing the configured `maxTimestampRange` (default 7 days) that protects every other endpoint. An unauthenticated attacker can supply a single-sided or arbitrarily large timestamp filter (e.g., `timestamp=gte:0`) that is passed unmodified to a complex multi-table FULL OUTER JOIN query in `doGetTransactions()`, causing expensive DB-level work on every request.

### Finding Description

**Exact code path:**

`getOneAccount()` at line 413 of `rest/accounts.js`:
```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [1](#0-0) 

The six arguments map to: `filterRequired=false`, `allowNe=true`, `allowOpenRange=true`, `strictCheckOverride=false`, `validateRange=false`.

**Root cause — the range-size guard is dead code for this call:**

In `utils.js`, the only place the `maxTimestampRangeNs` (7 days) is enforced is inside the `if (validateRange)` block:
```js
if (validateRange) {
  const {maxTimestampRange, maxTimestampRangeNs} = config.query;
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(`Timestamp range ... must be positive and within ${maxTimestampRange}`);
  }
}
``` [2](#0-1) 

Because `validateRange = false` is hardcoded in the `getOneAccount()` call, this block is never reached. The same call also passes `allowOpenRange = true`, so a single `timestamp=gte:0` (no upper bound) is accepted without error.

**Why `bindTimestampRange` does not save you:**

`bindTimestampRange` (default `false` per docs) is the only other guard that could cap the range. When it is `false`, `bindTimestampRange()` in `timestampRange.js` returns the range unchanged:
```js
if (!queryConfig.bindTimestampRange) {
  return {range};
}
``` [3](#0-2) 

Even when enabled, it caps to `maxTransactionsTimestampRangeNs` (default 60 days), not the 7-day `maxTimestampRange`. The default deployment has `bindTimestampRange=false`. [4](#0-3) 

**The expensive query path:**

The unbounded `timestampRange` is forwarded directly to `doGetTransactions()`:
```js
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)
  : emptyTransactionsPromise;
``` [5](#0-4) 

`doGetTransactions` → `getTransactionTimestamps` → `getTransactionTimestampsQuery`, which generates a query with a FULL OUTER JOIN across `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction` tables, each sub-queried over the full unbounded timestamp range: [6](#0-5) 

Although each sub-query carries a `LIMIT`, the PostgreSQL planner must evaluate the FULL OUTER JOIN across potentially years of partitioned data before applying the outer LIMIT. With no lower bound on `consensus_timestamp`, the planner cannot prune partitions and may resort to sequential scans across all time partitions.

Additionally, `getAccountBalanceTimestampRange` is called with the unbounded timestamp query before the transaction query, adding a second DB round-trip: [7](#0-6) 

### Impact Explanation

An attacker can hold DB connections and CPU for the duration of each expensive query. Because Node.js is single-threaded for I/O scheduling and the DB connection pool is finite, a small number of concurrent requests (e.g., 10–20) with `timestamp=gte:0` targeting a high-activity account (e.g., `0.0.2`, the treasury) can saturate the DB connection pool and degrade or block all other API responses, including gossip-related transaction queries. The `processRow()` and `formatTransactionRows()` CPU cost is bounded by `limit` (max 100), so the primary amplification is at the DB layer.

### Likelihood Explanation

- **No authentication required.** Any external user can call `GET /api/v1/accounts/{id}?timestamp=gte:0`.
- **Trivially repeatable.** A single `curl` loop is sufficient; no special tooling needed.
- **Default configuration is vulnerable.** `bindTimestampRange=false` is the documented default.
- **High-value targets exist.** Accounts like `0.0.2` (treasury) have transactions spanning the entire chain history, maximizing scan cost.

### Recommendation

1. **Remove `validateRange = false`** from the `parseTimestampFilters` call in `getOneAccount()`. Change line 413 to pass `true` as the last argument so the 7-day `maxTimestampRange` is enforced:
   ```js
   const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, true);
   ```
2. **Enable `bindTimestampRange=true`** in production deployments as a defense-in-depth measure, which caps the effective range to `maxTransactionsTimestampRangeNs`.
3. **Add a rate-limit or request timeout** at the reverse-proxy layer for the `/api/v1/accounts/:id` endpoint to bound the blast radius of any single client.

### Proof of Concept

```bash
# Step 1: Open-ended range from genesis — no upper bound, no auth required
curl "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=gte:0"

# Step 2: Explicit multi-year closed range (bypasses the 7-day check)
curl "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=gte:0&timestamp=lte:9999999999"

# Step 3: Flood with concurrent requests to exhaust DB pool
for i in $(seq 1 20); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=gte:0" &
done
wait
```

Expected result: each request triggers the full multi-table FULL OUTER JOIN scan over all time partitions; concurrent requests exhaust the DB connection pool; legitimate API calls (including gossip transaction queries) time out or queue indefinitely.
