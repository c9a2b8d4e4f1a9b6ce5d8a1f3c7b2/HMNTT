### Title
PgBouncer Strips `statement_timeout` Startup Parameter, Enabling Connection Pool Exhaustion via Hung Queries During Network Partition

### Summary
`getScheduleById()` in `rest/schedules.js` issues a database query via `pool.queryQuietly()` with no per-query timeout enforcement. Although `statement_timeout` is configured in the pg Pool startup parameters (`dbpool.js:15`), PgBouncer is explicitly configured to strip this parameter via `ignore_startup_parameters` (`values.yaml:361`), so PostgreSQL never enforces it. During a network partition or sustained database unresponsiveness, each in-flight query holds a pool connection indefinitely, and with a default pool size of 10, only 10 concurrent requests are needed to exhaust the global pool and deny service to all API endpoints.

### Finding Description
**Exact code path:**

`rest/schedules.js:115–120` — `getScheduleById()` calls `pool.queryQuietly(getScheduleByIdQuery, scheduleId)` with no `preQueryHint`, so execution falls into the `!preQueryHint` branch at `rest/utils.js:1519–1520`:

```js
if (!preQueryHint) {
  result = await this.query(query, params);   // hangs indefinitely if DB is unreachable
}
```

The `finally` block at `rest/utils.js:1540–1544` only releases `client` when it was explicitly acquired (the `preQueryHint` path). For the `this.query()` path, node-postgres internally holds a pool connection for the duration of the query. If the query never resolves, the connection is never returned to the pool.

**Root cause — `statement_timeout` silently neutralized by PgBouncer:**

`rest/dbpool.js:15` sets `statement_timeout` as a pool-level startup parameter (default 20 000 ms per `docs/configuration.md:557`):

```js
statement_timeout: config.db.pool.statementTimeout,
```

In node-postgres, this value is sent to the server as a startup parameter when each connection is established. However, `charts/hedera-mirror/values.yaml:361` configures PgBouncer to silently discard it:

```yaml
ignore_startup_parameters: extra_float_digits,options,statement_timeout
```

PgBouncer accepts the connection without error but never forwards `statement_timeout` to PostgreSQL. The `mirror_rest` user has no explicit `pool_mode: session` override (`values.yaml:371–373`), so it inherits the default `pool_mode: transaction` (`values.yaml:364`), which further prevents any session-level setting from persisting across transactions even if it were somehow delivered.

`connectionTimeoutMillis` (`dbpool.js:13`) only governs how long a caller waits to *acquire* a free slot from the pool — it does not bound how long an already-running query executes. No HTTP-level server timeout is present in the REST API server code to abort the handler.

**Failed assumption:** The codebase assumes `statement_timeout` in the pool config reaches PostgreSQL and will terminate long-running queries. PgBouncer's `ignore_startup_parameters` silently breaks this assumption with no error or log entry visible to the application.

### Impact Explanation
With `max: config.db.pool.maxConnections` defaulting to 10 (`docs/configuration.md:556`), only 10 concurrent hung requests are required to hold every connection in the global `pool` object (`dbpool.js:36`). All other REST API endpoints that share this same global pool — accounts, transactions, tokens, topic messages, etc. — will queue waiting for a free connection until `connectionTimeoutMillis` (20 s) elapses, at which point they return a 500 error. The service is effectively offline for all users for the duration of the partition. Because the pool is a single global object (`global.pool`), there is no isolation between endpoints.

**Severity: High** — complete REST API denial of service achievable with a small number of unauthenticated requests, requiring only an external network condition (partition or sustained DB latency) as a co-trigger.

### Likelihood Explanation
No authentication or rate-limiting is required to call `GET /api/v1/schedules/:scheduleId`. Any valid schedule ID (or even an invalid one that still reaches the DB query) suffices. Network partitions between application pods and the database are a realistic operational event in cloud-hosted Kubernetes deployments (node drain, PgBouncer restart, network policy change, cloud provider blip). An attacker who monitors for elevated DB latency (observable via response-time degradation) can opportunistically flood the endpoint with 10–20 concurrent requests to lock the pool for the partition's duration. The attack is trivially repeatable with a simple script and requires no credentials.

### Recommendation
1. **Issue `SET statement_timeout` per-query, not as a startup parameter.** Modify `queryQuietly` in `rest/utils.js` to explicitly run `SET LOCAL statement_timeout = <ms>` inside a transaction wrapping every query, bypassing PgBouncer's startup-parameter stripping. This is the only reliable way to enforce the timeout through a PgBouncer transaction-mode proxy.
2. **Remove `statement_timeout` from `ignore_startup_parameters`** in `charts/hedera-mirror/values.yaml:361` if the intent is to honour it, and switch `mirror_rest` to `pool_mode: session` so session-level settings persist.
3. **Add an HTTP-level request timeout** in the Express server so that handlers that are blocked on a DB call are aborted and their connections released after a bounded wall-clock time.
4. **Enforce a per-endpoint concurrency limit** (e.g., via a semaphore or connection-pool reservation) so a single endpoint cannot monopolise all pool slots.

### Proof of Concept
**Preconditions:** PgBouncer deployed with the Helm chart defaults (`ignore_startup_parameters` includes `statement_timeout`); `mirror_rest` user in transaction pool mode; default pool size of 10.

**Steps:**

1. Simulate a network partition between the REST API pod and PgBouncer/PostgreSQL (e.g., apply a Kubernetes `NetworkPolicy` blocking egress on port 5432, or use `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the app node).
2. Send 10 concurrent requests from an unauthenticated client:
   ```bash
   for i in $(seq 1 10); do
     curl -s "http://<mirror-node-rest>/api/v1/schedules/0.0.1234" &
   done
   wait
   ```
3. While those 10 requests are in-flight (hanging at `pool.queryQuietly` in `schedules.js:120`), send a request to any other endpoint:
   ```bash
   curl -v "http://<mirror-node-rest>/api/v1/transactions"
   ```
4. **Observed result:** The transactions request hangs for 20 seconds (the `connectionTimeoutMillis` wait) then returns a 500/503 error. All 10 pool connections are held by the schedule queries with no timeout firing, because PostgreSQL never received the `statement_timeout` startup parameter that PgBouncer stripped.
5. Restore network connectivity; all 10 hung queries resolve and the pool recovers — confirming the root cause is the missing per-query timeout, not a permanent failure. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
