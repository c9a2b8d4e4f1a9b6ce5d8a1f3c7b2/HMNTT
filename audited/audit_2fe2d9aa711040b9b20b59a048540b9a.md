### Title
Silent Sort-Order Override via Case-Variant Duplicate `order` Parameter

### Summary
The `requestQueryParser` function in `rest/middleware/requestHandler.js` intentionally merges case-variant duplicate query parameters (e.g., `order=asc&ORDER=desc`) into an array under the lowercased key. When this array reaches `parseLimitAndOrderParams()` in `rest/utils.js`, the strict equality check against the string constants `'asc'`/`'desc'` silently fails, causing the function to fall back to the endpoint's default sort order rather than the user-specified one. This silently corrupts pagination cursors, causing boundary records to be skipped or duplicated.

### Finding Description

**Step 1 — `merge()` produces an array for case-variant `order` params**

`rest/middleware/requestHandler.js` lines 39–51 and 56–66: [1](#0-0) 

When `order=asc&ORDER=desc` is received, `qs.parse` (case-sensitive) yields `{order: 'asc', ORDER: 'desc'}`. The loop lowercases both keys to `order`; the second entry triggers `merge('asc', 'desc')` → `['asc', 'desc']`. The result stored in `req.query.order` is the array `['asc', 'desc']`.

This is explicitly confirmed by the existing test suite: [2](#0-1) 

**Step 2 — `parseLimitAndOrderParams()` silently ignores the array**

`rest/utils.js` lines 668–672: [3](#0-2) 

`value` is `['asc', 'desc']`. The strict equality `value === 'asc'` and `value === 'desc'` both evaluate to `false`. No error is thrown; `order` silently stays as `defaultOrder`. There is no array-unwrapping logic for `order` (contrast with `getLimitParamValue`, which does handle array `limit` by taking the last element — confirmed by the `limit array` test at lines 419–428 of `utils.test.js`). [4](#0-3) 

**Step 3 — Downstream use of the wrong `order`**

Callers such as `getAccounts` use the returned `order` directly in SQL `ORDER BY` clauses and in pagination link generation: [5](#0-4) 

If the user requested `asc` but the default is `desc`, the SQL query runs in descending order while the pagination cursor is also computed for descending order, causing the client to miss records that exist only in the ascending range.

### Impact Explanation
Any API endpoint that calls `parseLimitAndOrderParams` (accounts, tokens, schedules, etc.) is affected. An attacker can force the server to use the default sort order regardless of what the user requested. For paginated result sets this means the `next` link is computed against the wrong boundary, causing records to be silently omitted from subsequent pages. Severity is **Medium**: data integrity of paginated responses is compromised without any authentication requirement.

### Likelihood Explanation
Exploitation requires only an HTTP GET request with two case-variant `order` parameters — no authentication, no special privileges, no rate-limit bypass needed. It is trivially repeatable from any HTTP client (browser, curl, script). The `requestHandler.test.js` test suite already documents and expects this exact array-producing behavior, confirming the attack surface is stable and not an edge case.

### Recommendation
In `parseLimitAndOrderParams`, normalize the `order` value before the equality check:

```js
const raw = req.query[constants.filterKeys.ORDER];
const value = Array.isArray(raw) ? raw[raw.length - 1] : raw;
if (value === constants.orderFilterValues.ASC || value === constants.orderFilterValues.DESC) {
  order = value;
}
```

Alternatively, reject requests where `order` is an array (return HTTP 400), consistent with how other single-value parameters are treated. A corresponding unit test for array `order` input should be added to `utils.test.js` alongside the existing `limit array` test.

### Proof of Concept

```
GET /api/v1/accounts?order=asc&ORDER=desc HTTP/1.1
Host: <mirror-node>
```

1. `requestQueryParser` merges the two params → `req.query.order = ['asc', 'desc']`
2. `parseLimitAndOrderParams` strict-equality check fails → `order = 'asc'` (the ASC default for `/accounts`)
   — In this specific case the default happens to be ASC, so the effect is masked.
   Try an endpoint whose default is DESC (e.g., `/api/v1/transactions`):

```
GET /api/v1/transactions?order=asc&ORDER=desc HTTP/1.1
```

3. User intends ascending order; `parseLimitAndOrderParams` falls back to `desc` (the default).
4. Response is in descending order; the `next` pagination link encodes a `lt:` cursor instead of `gt:`, causing the next page to walk backward and skip the ascending range of records the user requested.
