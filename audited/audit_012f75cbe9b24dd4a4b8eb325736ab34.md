### Title
Unauthenticated Cache-Bypass DoS via Unique Timestamp Parameters Exhausting DB Connection Pool on `GET /api/v1/transactions`

### Summary
The `GET /api/v1/transactions` endpoint in `rest/transactions.js` performs an unconditional database query (`getTransactionTimestamps`) on every request before any Redis caching is consulted. Because the response-level cache is keyed on the full URL and is disabled by default, an unauthenticated attacker can flood the endpoint with unique `timestamp` parameter values, generating a cache miss and a fresh DB query for every request. With a default pool ceiling of 10 connections and no rate limiting in the REST API layer, this exhausts the connection pool and renders the mirror node unable to serve transaction confirmation queries.

### Finding Description

**Code path:**

`rest/server.js:132` registers the route:
```
app.getExt(`${apiPrefix}/transactions`, transactions.getTransactions);
``` [1](#0-0) 

`getTransactions` (line 671) calls `doGetTransactions`, which calls `getTransactionTimestamps`: [2](#0-1) 

Inside `getTransactionTimestamps`, line 483 issues an **unconditional** `pool.queryQuietly` call — there is no cache check before this DB hit: [3](#0-2) 

Only *after* this first DB round-trip does the code consult the Redis row-level cache for transaction details: [4](#0-3) 

The row-level cache key is `transaction:<consensus_timestamp>` — it caches individual rows, not the timestamp-lookup query itself. Every request with a novel timestamp range still hits the DB for the first query.

**Response-level cache is disabled by default and bypassable:**

`server.js` only enables the response cache when both `config.cache.response.enabled` AND `config.redis.enabled` are true: [5](#0-4) 

The default for `cache.response.enabled` is `false`: [6](#0-5) 

Even when enabled, the cache key is `MD5(req.originalUrl)`: [7](#0-6) 

Every unique `?timestamp=gt:X` value produces a unique URL → unique MD5 → guaranteed cache miss.

**Connection pool is tiny with no rate limiting:**

Default `maxConnections = 10`, `statementTimeout = 20000 ms`, `connectionTimeoutMillis = 20000 ms`: [8](#0-7) 

Pool is initialized directly from these values: [9](#0-8) 

**No rate limiting exists in the REST API layer.** The throttle mechanism (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` Java module and is not applied to the Node.js REST API. The `authHandler` middleware only governs response-size limits for authenticated users, not request rate.

### Impact Explanation

With 10 pool connections and a 20-second statement timeout, an attacker sustaining just 10–15 concurrent requests with unique timestamp values will hold all pool slots. Legitimate callers receive `connectionTimeoutMillis` errors after 20 seconds. The mirror node REST API becomes unable to answer any `GET /api/v1/transactions` queries, so clients and wallets cannot confirm transaction finality through the mirror node. Other endpoints sharing the same pool (accounts, balances, tokens) are collaterally degraded. The actual Hedera consensus network is unaffected (the mirror node is read-only), but the mirror node's role as the canonical transaction-confirmation API is fully disrupted.

### Likelihood Explanation

No authentication, no API key, no CAPTCHA, and no IP-based rate limit is required. The attack is trivially scriptable: generate sequential nanosecond timestamps and fire concurrent HTTP requests. A single attacker with a modest connection rate (10–20 req/s) is sufficient to saturate the pool. The attack is repeatable indefinitely and requires no special knowledge of the system beyond the public API documentation.

### Recommendation

1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit` per IP) before the route handlers, specifically for expensive list endpoints.
2. **Enable and enforce the response cache by default** (`cache.response.enabled = true`) so repeated or near-identical queries are served from Redis without touching the DB.
3. **Cache the timestamp-lookup query result** in `getTransactionTimestamps`, not just the row-detail results, so the first DB query is also shielded.
4. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer is already referenced in Helm values) in front of the REST API DB user to absorb burst load without exhausting the pool.
5. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) inside `getTransactionTimestamps` to cap simultaneous in-flight DB queries regardless of HTTP concurrency.

### Proof of Concept

```bash
# Generate 50 concurrent requests each with a unique nanosecond timestamp
# No authentication required
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gt:156577920${i}.711927001" &
done
wait
# All 10 pool connections are held; subsequent legitimate requests time out
# with "Error: timeout exceeded when trying to connect" after 20 seconds
```

Observe in mirror node logs: repeated `error event emitted on pg pool` or `DbError: timeout exceeded` for all subsequent requests, confirming pool exhaustion.
