### Title
Unauthenticated Arbitrary Transaction Opcode Trace Exposure via `getContractOpcodes()`

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint in `OpcodesController` performs no authentication or ownership verification before re-executing any historical transaction on the EVM and returning its full opcode trace (stack, memory, storage). Any unprivileged external user who knows or can enumerate a valid `TransactionIdParameter` (format `0.0.X-seconds-nanos`) can trigger a full EVM replay of any other user's transaction and receive the complete execution trace, including sensitive in-memory data.

### Finding Description
**Code path:**

`OpcodesController.getContractOpcodes()` — [1](#0-0) 

The only guards present are:
1. `properties.isEnabled()` — a feature flag (default `false`), not an access control mechanism. [2](#0-1) 
2. `validateAcceptEncodingHeader()` — enforces `Accept-Encoding: gzip`, not identity. [3](#0-2) 
3. `throttleManager.throttleOpcodeRequest()` — rate-limits requests, does not authenticate or authorize. [4](#0-3) 

There is **no Spring Security configuration** in the web3 module — a grep for `SecurityFilterChain`, `@PreAuthorize`, `@Secured`, `hasRole`, `hasAuthority` returns zero matches across all web3 Java sources.

In `OpcodeServiceImpl.buildCallServiceParameters()`, when a `TransactionIdParameter` is received, the `payerAccountId` field is used purely as a database lookup key — it is never compared against the identity of the HTTP caller: [5](#0-4) 

**Root cause:** The failed assumption is that the `payerAccountId` embedded in the transaction ID path parameter constitutes an ownership proof. It does not — it is attacker-supplied and only used to locate the transaction record in the database.

**Exploit flow:**
1. Attacker observes any on-chain transaction ID (publicly visible on any block explorer or mirror node REST API), e.g., `0.0.1234-1700000000-000000001`.
2. Attacker sends: `GET /api/v1/contracts/results/0.0.1234-1700000000-000000001/opcodes?stack=true&memory=true&storage=true` with header `Accept-Encoding: gzip`.
3. `OpcodeServiceImpl` looks up the transaction, reconstructs `ContractDebugParameters` from historical data, and passes it to `contractDebugService.processOpcodeCall()`.
4. The full EVM re-execution trace — including every stack frame, memory slot, and storage read/write — is returned to the attacker.

### Impact Explanation
- **Information disclosure:** Full opcode traces expose every stack value, memory word, and storage slot touched during execution. Contracts that process sensitive data (private inputs, encrypted values, authorization secrets passed as calldata) leak that data to any observer.
- **Resource exhaustion:** EVM re-execution is computationally expensive (noted in the controller's own Javadoc: *"may take a significant amount of time"*). An attacker can submit a high volume of requests for gas-heavy historical transactions, consuming server CPU/memory up to the throttle limit — and the throttle only limits rate, not total concurrent load per attacker.
- **Severity: High** — combines unauthorized data exposure of arbitrary users' transaction internals with a resource-abuse vector, with zero privilege required.

### Likelihood Explanation
- Transaction IDs are fully public on Hedera's network and mirror node REST API (`/api/v1/transactions`). No secret knowledge is required to enumerate valid IDs.
- The only barrier is the `enabled` feature flag. When operators enable this feature (its intended production use), the attack surface is immediately open to the entire internet.
- The attack is trivially repeatable and scriptable. A single HTTP client with gzip support is sufficient.

### Recommendation
1. **Add authentication and authorization** to the opcodes endpoint. At minimum, require the caller to prove ownership of the `payerAccountId` (e.g., via a signed challenge or API key tied to the account). Alternatively, restrict the endpoint to authenticated operators/admins only via Spring Security (`SecurityFilterChain` with `requestMatchers("/api/v1/contracts/results/*/opcodes").authenticated()`).
2. **Enforce per-caller rate limiting** (not just global throttling) to prevent resource exhaustion even for authorized users.
3. **Scope the feature flag** so that enabling it does not implicitly open the endpoint to all unauthenticated callers.

### Proof of Concept
**Preconditions:** Feature flag `hiero.mirror.web3.opcode.tracer.enabled=true` (operator-enabled). Attacker has no credentials.

**Steps:**
```bash
# 1. Enumerate any contract transaction ID from the public mirror node REST API
TX_ID=$(curl -s "https://<mirror-node>/api/v1/transactions?transactiontype=CONTRACTCALL&limit=1" \
  | jq -r '.transactions[0].transaction_id')

# 2. Convert to path format: replace @ with - (0.0.X@seconds.nanos -> 0.0.X-seconds-nanos)
TX_PATH=$(echo "$TX_ID" | sed 's/@/-/' | sed 's/\./\-/3')

# 3. Request full opcode trace with stack + memory + storage as an unprivileged user
curl -s -H "Accept-Encoding: gzip" \
  "https://<web3-node>/api/v1/contracts/results/${TX_PATH}/opcodes?stack=true&memory=true&storage=true" \
  | gunzip
```

**Result:** Full opcode trace of another user's transaction is returned, including all stack values, memory contents, and storage accesses — with no authentication required.

    @GetMapping(value = "/{transactionIdOrHash}/opcodes")
    OpcodesResponse getContractOpcodes(
            @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
            @RequestParam(required = false, defaultValue = "true") boolean stack,
            @RequestParam(required = false, defaultValue = "false") boolean memory,
            @RequestParam(required = false, defaultValue = "false") boolean storage,
            @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
        if (properties.isEnabled()) {
            validateAcceptEncodingHeader(acceptEncoding);
            throttleManager.throttleOpcodeRequest();

            final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
            return opcodeService.processOpcodeCall(request);
        }

        throw new ResponseStatusException(HttpStatus.NOT_FOUND);
    }
