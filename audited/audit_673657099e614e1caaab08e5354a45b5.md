### Title
Unauthenticated Pagination Loop Causes Unbounded DB Load Amplification via UNION ALL Queries with No Rate Limiting

### Summary
The `getNftsByAccountId` handler in `rest/controllers/accountController.js` accepts any EVM address, resolves it to an account ID, and executes up to three-part `UNION ALL` SQL queries per request with no application-level rate limiting. An unprivileged attacker can follow the `next` pagination link in a tight loop against a high-NFT account, generating up to 3× the DB query load per HTTP request with no throttle mechanism to stop them, causing sustained database connection pool saturation and degraded service for all users.

### Finding Description

**Code path:**

`rest/controllers/accountController.js` lines 90–103 — `getNftsByAccountId`: [1](#0-0) 

No account existence pre-check is performed (contrast with `listStakingRewardsByAccountId` at line 172 which calls `EntityService.isValidAccount`). The handler immediately calls `NftService.getNfts(query)`.

`rest/service/nftService.js` lines 84–126 — `getQuery`: [2](#0-1) 

When pagination bounds are fully populated (lower + inner + upper all non-empty), the method emits a three-part `UNION ALL` query — each sub-query independently scanned with its own `LIMIT $2`. This is triggered on every follow-up pagination request.

**How pagination activates the three-part query:**

`rest/controllers/baseController.js` `getPaginationLink` (line 195) encodes the last row's `token_id` and `serial_number` into the `next` URL. When the client follows that link, `extractNftMultiUnionQuery` populates all three of `lower`, `inner`, and `upper` bounds, causing `getQuery` to emit the full three-subquery `UNION ALL` form: [3](#0-2) 

**No rate limiting in the REST Node.js API:**

A `grep` for `rateLimit|rateLimiter|throttle` across all `rest/**/*.js` files returns only a single hit in a test utility file. The throttle mechanisms found (`web3/src/main/java/.../ThrottleManagerImpl.java`) are scoped exclusively to the web3 Java API, not the REST Node.js API. The `rest/routes/accountRoute.js` registers the handler with no middleware guard: [4](#0-3) 

**Limit cap exists but is insufficient:**

The limit is capped at max=100 (default config): [5](#0-4) 

This bounds each sub-query to 100 rows, but does not limit the *rate* of incoming requests. With three sub-queries per HTTP request and no per-IP throttle, an attacker multiplies DB query volume by 3× per request at arbitrary request rates.

**Root cause:** The failed assumption is that infrastructure-level rate limiting (e.g., a reverse proxy) will always be present in front of the REST API. The application itself has zero rate-limiting defense for this endpoint.

### Impact Explanation

Each pagination follow-up request issues up to 3 independent DB sub-queries (each with `account_id = $1` and `LIMIT 100`). Without rate limiting, an attacker sending N requests/second causes up to 3N DB queries/second. At sufficient concurrency this exhausts the DB connection pool (`rest/dbpool.js` uses a fixed pool), causing query queuing and timeout errors for all concurrent users. The impact is service degradation (increased latency, 503s) for all API consumers — a griefing/DoS with no economic precondition for the attacker.

### Likelihood Explanation

- **Zero privilege required**: any HTTP client can call the endpoint.
- **Trivially scriptable**: the `next` link is returned in the JSON response body; a simple loop suffices.
- **Amplifier is always active on pagination**: any account with >100 NFTs produces a `next` link, activating the three-part UNION ALL on every subsequent page.
- **No observable signal to the attacker**: no 429 responses, no backoff hints.
- **Realistic target accounts exist**: on mainnet, accounts holding large NFT collections (gaming, marketplace) are publicly known.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST Node.js API (e.g., `express-rate-limit`) applied globally or specifically to the `/accounts/:id/nfts` route before the handler.
2. **Add a DB statement timeout** for the REST Node.js pool (analogous to `statementTimeout` already present in the REST Java and web3 APIs) to bound worst-case query duration.
3. **Add an account existence check** before executing the NFT query (as done in `listStakingRewardsByAccountId`) to avoid wasting DB resources on non-existent EVM addresses.
4. **Consider caching** the EVM-address-to-account-ID resolution result to eliminate the extra DB lookup on every request.

### Proof of Concept

```bash
# 1. Identify a mainnet account with many NFTs (e.g., 0x<known_nft_whale_evm_address>)
TARGET="0x<evm_address>"
BASE="https://<mirror-node-host>/api/v1"

# 2. Fetch first page
NEXT=$(curl -s "$BASE/accounts/$TARGET/nfts?limit=100" | jq -r '.links.next')

# 3. Follow next links in a tight loop (no auth, no rate limit)
while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  # Each request triggers up to 3 UNION ALL sub-queries
  NEXT=$(curl -s "$BASE$NEXT" | jq -r '.links.next') &
done
# Repeat with many parallel workers to saturate the DB connection pool
```

Each iteration of the loop, when run with high concurrency (e.g., 50–200 parallel workers), issues up to 3 DB sub-queries per HTTP request with no server-side throttle, causing sustained DB load and degraded response times for all other API users.

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
