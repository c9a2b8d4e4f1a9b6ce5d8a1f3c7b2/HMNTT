### Title
Cache Null-Miss for Non-Existent Contract Storage Slots Causes Unbounded Repeated DB Queries (DoS)

### Summary
`ContractStateServiceImpl.findStorage()` only populates `contractStateCache` with a value when the DB returns a result for a slot. For non-existent slots, nothing is ever written to `contractStateCache`, so every subsequent call for that slot returns a cache miss (`null`) and unconditionally re-invokes `findStorageBatch()`, which issues a live DB query. An unprivileged user can exploit the public `/api/v1/contracts/call` endpoint to repeatedly trigger lookups of non-existent storage slots, causing every request to bypass the cache and amplify database load.

### Finding Description

**Exact code path:**

In `ContractStateServiceImpl.findStorage()` (lines 58–70):

```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}

return findStorageBatch(contractId, key);  // always called for non-existent slots
``` [1](#0-0) 

In `findStorageBatch()` (lines 103–114), values are only written to `contractStateCache` for slots that the DB actually returns:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
byte[] cachedValue = null;

for (final var contractSlotValue : contractSlotValues) {
    contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);  // only for found slots
    ...
}
``` [2](#0-1) 

**Root cause:** When a slot does not exist in the DB, `contractSlotValues` is empty, the loop body never executes, and no sentinel is stored in `contractStateCache` for that key. The next call to `findStorage()` for the same slot finds `null` in the cache (indistinguishable from a first-time miss) and calls `findStorageBatch()` again, issuing another DB query. This repeats indefinitely.

**Amplification:** `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` (line 90) accumulates non-existent slot keys in the per-contract slots cache (default `maximumSize=1500`). Each subsequent `findStorageBatch()` call collects all accumulated slot keys and issues a single batch DB query containing all of them — so filling the per-contract cache with 1500 non-existent keys means every future miss for any of those keys triggers a batch query with up to 1500 non-existent slot lookups. [3](#0-2) 

### Impact Explanation
Every `eth_call` that reads an uninitialized storage slot bypasses the cache entirely and hits the database. With the per-contract slots cache holding up to 1500 non-existent keys (`slotsPerContract` default `maximumSize=1500`), a single `findStorage()` call for any of those keys issues a batch query scanning all 1500 non-existent slots. The `contractState` cache has `expireAfterWrite=2s` for real values, but non-existent slots are never written, so there is no expiry — the miss is permanent until the per-contract slots cache entry itself expires (default `expireAfterAccess=5m`). This allows sustained, amplified DB load from a small number of API requests. [4](#0-3) 

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint requires no authentication. [5](#0-4) 

The global rate limiter allows up to 500 RPS and 1.5B gas/second by default. [6](#0-5) 

A single `eth_call` can read many storage slots within the gas budget. An attacker needs only to: (1) identify or deploy any contract, (2) craft calldata that reads from uninitialized slots (e.g., reading slot keys that were never written), and (3) repeat at the allowed rate. No special privileges, tokens, or on-chain transactions are required. The attack is trivially scriptable and repeatable.

### Recommendation
Store a sentinel value (e.g., `EMPTY_VALUE`) in `contractStateCache` for slots confirmed absent from the DB, and handle that sentinel in `findStorage()` to return `Optional.empty()` without re-querying. Specifically:

1. After `findStorageBatch()` returns, for each slot key in `cachedSlots` that was **not** present in `contractSlotValues`, call `contractStateCache.put(generateCacheKey(contractId, slotKey), EMPTY_VALUE)`.
2. In `findStorage()`, add a branch: if `cachedValue == EMPTY_VALUE`, return `Optional.empty()` immediately without calling `findStorageBatch()`.

This ensures that a confirmed DB miss is cached and served from memory on subsequent calls, eliminating the repeated DB queries.

### Proof of Concept

**Preconditions:**
- A deployed contract `C` at address `0xADDR` with no storage written to slot `0xDEAD...BEEF`.
- `enableBatchContractSlotCaching = true` (default).

**Steps:**

1. Send repeated POST requests to `/api/v1/contracts/call`:
   ```json
   {
     "to": "0xADDR",
     "data": "<calldata that triggers SLOAD of slot 0xDEAD...BEEF>",
     "gas": 50000
   }
   ```
2. Observe that each request causes a call to `ContractStateServiceImpl.findStorage()` for slot `0xDEAD...BEEF`.
3. Confirm via DB query logs that `findStorageBatch` (or `findStorage` fallback) issues a DB query on **every** request, not just the first.
4. To amplify: first send 1500 requests each reading a different non-existent slot for the same contract. This fills the per-contract slots cache. Then send any request reading any of those slots — each triggers a batch DB query scanning all 1500 non-existent slots.
5. At 500 RPS (the default global limit), this sustains ~500 batch DB queries/second, each scanning up to 1500 non-existent slot keys.
