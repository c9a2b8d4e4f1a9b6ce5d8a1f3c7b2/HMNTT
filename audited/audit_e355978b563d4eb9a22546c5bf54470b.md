### Title
Unauthenticated, Uncached, Unlimited Concurrent DB Queries via `/api/v1/network/stake` Enable Resource Exhaustion

### Summary
The `GET /api/v1/network/stake` endpoint is publicly accessible with no authentication, no rate limiting, and no query-result caching. Every request causes `NetworkServiceImpl.getLatestNetworkStake()` to execute a native SQL correlated-subquery against the database. An unprivileged attacker can exploit HTTP/2 stream multiplexing to fire hundreds of concurrent requests over a single TCP connection, each independently hitting the database, with no mechanism to coalesce or throttle them.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 127â€“130) calls `networkService.getLatestNetworkStake()` with no guards: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52â€“56) unconditionally delegates to the repository: [2](#0-1) 

`NetworkStakeRepository.findLatest()` (lines 12â€“19) executes a native SQL query with a correlated subquery (`select max(consensus_timestamp) from network_stake`) on every invocation â€” no `@Cacheable` annotation is present: [3](#0-2) 

**Root cause â€” failed assumption:** The design assumes either (a) a caching layer absorbs repeated identical reads, or (b) an upstream rate limiter throttles request volume. Neither exists for this endpoint in the `rest-java` module. The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module for contract calls: [4](#0-3) 

By contrast, the analogous `grpc` module's `NodeStakeRepository` explicitly applies `@Cacheable` to its equivalent read: [5](#0-4) 

No such protection exists in `rest-java`'s `NetworkStakeRepository`.

**Exploit flow:**
1. Attacker opens one HTTP/2 connection to the mirror node's REST-Java endpoint.
2. Sends N concurrent GET streams to `/api/v1/network/stake` (HTTP/2 allows up to the server's `SETTINGS_MAX_CONCURRENT_STREAMS`, typically 100â€“250 by default in Spring Boot / Netty).
3. Each stream independently reaches `findLatest()`, which issues a separate DB query with a correlated subquery.
4. No cache hit is possible (no `@Cacheable`); no request is rejected (no rate limiter); no authentication is checked.
5. Repeating this in a tight loop from one or more clients sustains elevated DB CPU.

### Impact Explanation
The `network_stake` table is queried with a correlated `MAX()` subquery on every request. Under concurrent load this translates directly to proportional DB CPU and I/O. Because the data changes only once per staking period (roughly daily), the absence of caching means every request is a redundant full-table scan for the max timestamp. A single attacker with an HTTP/2 client can sustain hundreds of simultaneous in-flight queries, plausibly exceeding a 30% DB CPU increase over baseline on a normally-loaded mirror node. The impact is availability degradation of the database, which is shared with other mirror node services.

### Likelihood Explanation
The endpoint is publicly documented in the OpenAPI spec with no authentication requirement: [6](#0-5) 

No credentials, API keys, or tokens are needed. HTTP/2 multiplexing is supported by all modern HTTP clients and browsers. The attack requires only a single IP and a standard HTTP/2 library (e.g., `h2load`, `curl --http2`, Python `httpx`). It is trivially repeatable and scriptable.

### Recommendation
Apply one or more of the following:

1. **Add `@Cacheable`** to `NetworkStakeRepository.findLatest()` in `rest-java` with a TTL matching the staking period (e.g., 1 hour). This is the pattern already used in the `grpc` module's `NodeStakeRepository`.
2. **Add a rate limiter** to the `rest-java` module analogous to `ThrottleConfiguration` in `web3`, applied at the servlet filter or controller level for `/api/v1/network/stake`.
3. **Set HTTP/2 `SETTINGS_MAX_CONCURRENT_STREAMS`** to a low value (e.g., 10â€“25) at the server level to limit per-connection parallelism.
4. Combine (1) and (3) as the minimum viable fix: caching eliminates the DB amplification, and stream limits reduce per-connection burst capacity.

### Proof of Concept

```bash
# Prerequisites: h2load (nghttp2) or any HTTP/2 client
# Target: mirror node REST-Java endpoint

# Send 500 concurrent requests over 10 HTTP/2 connections
h2load -n 500 -c 10 -m 50 \
  https://<mirror-node-host>/api/v1/network/stake

# Observe DB CPU on the PostgreSQL host during the run:
# Expected: significant CPU spike from repeated correlated subquery
# SELECT * FROM network_stake WHERE consensus_timestamp = (SELECT MAX(...))
# fired 500 times with zero cache hits

# Alternatively with curl (single connection, sequential but scriptable in parallel):
for i in $(seq 1 200); do
  curl -s --http2 https://<mirror-node-host>/api/v1/network/stake &
done
wait
```

No authentication token, API key, or privileged access is required. The attack is repeatable indefinitely.
