### Title
Multi-Operation Document Bypasses Complexity/Depth Guards, Wasting Parse and Validation Resources

### Summary
The `graphQlCustomizer()` method installs `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation`, but both instrumentations only analyze the **single executed operation**, not every operation definition present in the document. Because `ParserOptions` sets no limit on the number of named operations per document, an unprivileged caller can submit a document containing many high-complexity operations within the 10,000-character / 1,000-token budget, name only one for execution, and force the parser and validator to process all of them while the complexity/depth guards fire only once.

### Finding Description
**Code locations:**

- `GraphQlConfiguration.java` lines 31–38 — parser limits (`maxCharacters(10000)`, `maxTokens(1000)`); no `maxOperationCount` is set.
- `GraphQlConfiguration.java` lines 42–48 — `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are wired as execution-phase instrumentations; they run only against the operation selected by `operationName`.
- `GraphQLInterceptor.java` lines 28–29 — `parser.parseDocument(request.getDocument())` is called unconditionally and without caching on every request, iterating all `OperationDefinition` nodes.
- `CachedPreparsedDocumentProvider.java` line 26 — caches by raw query string; any character variation (e.g., a different literal value) produces a cache miss, forcing a full re-parse and re-validation of all operations.

**Root cause:** `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` are invoked during the *execution* phase for the selected operation only. The *parse* and *validate* phases, however, process the entire document — every named operation definition. No `ParserOptions` knob limits how many operation definitions a document may contain, so the character/token budget is the only constraint on how many operations an attacker can pack in.

**Exploit flow:**
1. Attacker crafts a document with N named operations, each individually at or near the complexity ceiling (200), staying within 10,000 characters and 1,000 tokens.
2. POST to `/graphql/alpha` with `operationName` set to one of the N operations.
3. `GraphQLInterceptor.extractEndpointFromQuery()` calls `parser.parseDocument()` on the full document — all N operations are parsed, uncached.
4. The graphql-java pipeline parses and validates all N operations (cache miss on first or varied request).
5. `MaxQueryComplexityInstrumentation` fires only for the single selected operation; the other N-1 operations are never complexity-checked.
6. Repeat with slight query variations to defeat the preparsed-document cache.

**Why existing checks are insufficient:**
- Character/token limits bound total document size but do not bound the number of distinct operation definitions.
- Complexity and depth instrumentations are execution-phase only; they do not cover parse or validation cost of non-executed operations.
- The preparsed-document cache is bypassed by any query variation (e.g., changing a numeric literal), so parse+validate cost is paid again.
- `GraphQLInterceptor` always re-parses independently of the cache.

### Impact Explanation
An attacker can multiply the effective parse and validation work per HTTP request by a factor proportional to the number of operations that fit within the budget (roughly 50–150 short operations within 10,000 chars / 1,000 tokens). This amplifies CPU consumption on the GraphQL server without triggering any rate-limiting guard tied to query complexity or depth. The impact is resource exhaustion / griefing; no data is leaked and no economic damage to network participants occurs.

### Likelihood Explanation
No authentication or special privilege is required — the GraphQL endpoint is publicly accessible. The attack is trivially scriptable: a single crafted document template with N named operations can be sent in a tight loop, varying one literal per request to defeat the cache. The technique is well-known in the GraphQL security community and requires no specialized tooling.

### Recommendation
1. **Limit operation count at parse time.** Add a custom `NoUnusedOperations`-style validation rule or, preferably, reject documents containing more than one operation definition before parsing proceeds. In graphql-java this can be done with a custom `Instrumentation` or a pre-execution `WebGraphQlInterceptor` that counts `OperationDefinition` nodes and returns an error if the count exceeds 1 (or a small configured maximum).
2. **Apply complexity/depth checks to all operations, not just the executed one.** Run `MaxQueryComplexityInstrumentation` logic over every operation in the document during the validation phase, not only the selected one.
3. **Cache the `GraphQLInterceptor` parse result.** `GraphQLInterceptor.extractEndpointFromQuery()` re-parses the document on every request independently of the `PreparsedDocumentProvider` cache. Share the cached `Document` object or extract the endpoint name from the already-parsed document available later in the pipeline.

### Proof of Concept
```bash
# Build a document with 50 named operations, each near complexity limit
QUERY='query op1{account(input:{entityId:{shard:0,realm:0,num:1}}){balance alias{shard realm num}}}
query op2{account(input:{entityId:{shard:0,realm:0,num:2}}){balance alias{shard realm num}}}
... (repeat to op50, varying num each time) ...'

# Execute only op1; parser/validator processes all 50
curl -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"$QUERY\", \"operationName\": \"op1\"}"

# Vary a literal on each iteration to defeat the preparsed-document cache
for i in $(seq 1 1000); do
  QUERY="query op1{account(input:{entityId:{shard:0,realm:0,num:$i}}){balance}} \
         query op2{account(input:{entityId:{shard:0,realm:0,num:$((i+1))}}){balance}} \
         ... (N operations) ..."
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\", \"operationName\": \"op1\"}" &
done
```
