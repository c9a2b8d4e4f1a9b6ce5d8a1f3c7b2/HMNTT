### Title
New Transaction Created on Every Retry After Successful Execute — Receipt-Timeout Causes Multi-Transfer Per Tick

### Summary
`submitWithRetry()` constructs a brand-new `TransferTransaction` on every loop iteration. If `Execute(client)` succeeds (transaction committed on-chain) but `resp.GetReceipt(client)` subsequently times out or errors — which a network-layer attacker can force by delaying receipt responses — the code treats the attempt as failed and retries with a completely new transaction ID. With the default `maxRetries=10`, this allows up to 11 separate fund transfers per tick instead of one.

### Finding Description
**File:** `pinger/transfer.go`, `submitWithRetry()`, lines 23–57.

```
for i := 1; i <= attempts; i++ {          // attempts = maxRetries + 1 (default 11)
    ...
    cryptoTransfer := hiero.NewTransferTransaction().   // NEW tx, NEW tx ID each iteration
        AddHbarTransfer(...)
        AddHbarTransfer(...)

    resp, err := cryptoTransfer.Execute(client)        // tx submitted & executed on-chain
    if err == nil {
        receipt, rerr := resp.GetReceipt(client)       // polls consensus nodes for receipt
        if rerr == nil {
            return nil
        }
        err = rerr   // GetReceipt failed → falls through to retry
    }
    // backoff, then loop: new tx built and submitted
}
```

**Root cause:** `Execute` and `GetReceipt` are two distinct network operations. `Execute` submits the signed transaction to a consensus node; the transaction is processed by the network independently of whether the client ever receives the receipt. `GetReceipt` merely polls for confirmation. The code conflates "receipt fetch failed" with "transaction did not execute," which is false.

**Failed assumption:** The code assumes that if `GetReceipt` returns an error, the transfer did not occur and it is safe to retry with a new transaction. This assumption is wrong — the transfer may already be finalized on-chain.

**Exploit flow:**
1. Attacker with MITM position between the pinger pod and Hedera consensus nodes selectively delays or drops TCP segments carrying the receipt gRPC response, while allowing the `Execute` gRPC call to pass through normally.
2. `Execute(client)` completes successfully; the `CryptoTransfer` transaction is accepted and executed by the network (funds moved).
3. `resp.GetReceipt(client)` blocks waiting for the receipt response, which the attacker is suppressing. The SDK's internal receipt-polling timeout fires; `GetReceipt` returns a timeout/error.
4. `err = rerr` is set; the loop continues to iteration `i+1`.
5. A new `hiero.NewTransferTransaction()` is built with a fresh transaction ID and submitted via `Execute` — a second, independent fund transfer.
6. The attacker repeats step 3 for each subsequent receipt poll.
7. All `maxRetries+1` iterations execute successfully on-chain; only the final receipt (or none) is allowed through.

Hedera's transaction deduplication is keyed on transaction ID. Because each retry generates a new transaction (new ID), deduplication provides no protection. [1](#0-0) [2](#0-1) 

### Impact Explanation
With default configuration (`maxRetries=10`, `amountTinybar=10000`, `interval=1s`), the operator account is drained at up to 11× the intended rate per tick. Over time this multiplies total fund loss by `maxRetries+1`. The destination account (`toAccountID`, default `0.0.98` — the Hedera fee collection account) receives all transfers, so the operator's balance is depleted. If the pinger is used as a liveness/monitoring tool with a funded operational account, the attacker can exhaust that account's balance, causing the pinger to fail and potentially triggering false infrastructure alerts. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The attacker requires a MITM position on the network path between the pinger and the Hedera consensus nodes. In Kubernetes/cloud deployments this is achievable via: a compromised CNI plugin or sidecar, a malicious cloud-provider network component, BGP route hijacking toward Hedera node IPs, or a compromised service mesh. The attack is selective (only receipt responses need to be dropped, not the Execute calls), making it stealthy and repeatable on every tick. No application-layer credentials are required.

### Recommendation
1. **Track whether `Execute` succeeded** and, if so, do not retry with a new transaction. Instead, retry only the `GetReceipt` call for the same `resp.TransactionID`:

```go
resp, err := cryptoTransfer.Execute(client)
if err == nil {
    // Execute succeeded — transaction is on-chain. Retry only the receipt fetch.
    for j := 0; j < receiptRetries; j++ {
        receipt, rerr := resp.GetReceipt(client)
        if rerr == nil {
            log.Printf("transfer success ...")
            return nil
        }
        // backoff and retry receipt only
    }
    return fmt.Errorf("receipt fetch failed after execute succeeded: %w", lastReceiptErr)
}
// Only retry Execute if Execute itself failed (tx never submitted)
```

2. Alternatively, set a fixed `TransactionID` before the outer loop and reuse it across all retries, so that even if a duplicate `Execute` is sent, Hedera's deduplication rejects it.

3. Separate the retry budget for `Execute` failures from the retry budget for `GetReceipt` failures. [5](#0-4) 

### Proof of Concept
**Preconditions:** Attacker has MITM capability on the network path between the pinger and Hedera consensus nodes (e.g., via iptables on a compromised node in the same Kubernetes cluster).

**Steps:**
1. Deploy the pinger with default config (`maxRetries=10`, `amountTinybar=10000`, `interval=1s`).
2. On the MITM host, install a filter that:
   - Allows outbound gRPC traffic to Hedera consensus nodes (port 50211) — `Execute` calls pass through.
   - Drops/delays inbound gRPC responses that carry `TransactionGetReceipt` responses (identifiable by protobuf message type or by timing: receipt responses arrive ~3–5 s after submission).
3. Observe that per tick, `submitWithRetry` logs `attempt N/11 failed: context deadline exceeded` (or similar) for each of the first 10 iterations, then either succeeds on the last or exhausts all attempts.
4. Query the Hedera network (e.g., via mirror node REST API) for the operator account's transaction history: observe up to 11 `CryptoTransfer` transactions with distinct IDs all executed within the same tick window.
5. Confirm operator balance decreases by up to `11 × amountTinybar` per tick instead of `1 × amountTinybar`. [6](#0-5) [7](#0-6)

```go
func submitWithRetry(ctx context.Context, client *hiero.Client, cfg config) error {
	toID, err := hiero.AccountIDFromString(cfg.toAccountID)
	if err != nil {
		return fmt.Errorf("invalid destination id: %w", err)
	}

	var lastErr error
	attempts := cfg.maxRetries + 1

	for i := 1; i <= attempts; i++ {
		if err := ctx.Err(); err != nil {
			return err
		}

		start := time.Now()
		cryptoTransfer := hiero.NewTransferTransaction().
			AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
			AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))

		resp, err := cryptoTransfer.Execute(client)
		if err == nil {
			receipt, rerr := resp.GetReceipt(client)
			if rerr == nil {
				log.Printf("transfer success: status=%s txID=%s elapsed=%s",
					receipt.Status.String(), resp.TransactionID.String(), time.Since(start))
				return nil
			}
			err = rerr
		}

		lastErr = err
		log.Printf("attempt %d/%d failed: %v", i, attempts, err)

		if i < attempts {
			sleep := backoff(cfg.baseBackoff, i)
			timer := time.NewTimer(sleep)
			select {
			case <-ctx.Done():
				timer.Stop()
				return ctx.Err()
			case <-timer.C:
			}
		}
	}

	return fmt.Errorf("all attempts failed: %w", lastErr)
```
