### Title
Repeated Synchronous `unzipSync()` Decompression on Every Cache Hit via `accept-encoding: identity`

### Summary
When the response cache stores a compressed body (`cachedResponse.compressed === true`), any request that causes `negotiate()` to return `false` triggers `cachedResponse.getUncompressedBody()`, which calls `unzipSync()` synchronously on every invocation with no memoization. An unprivileged attacker can send `accept-encoding: identity` (or any non-gzip encoding) to force this path on every cache hit, causing repeated synchronous CPU-bound decompression that blocks the Node.js event loop. The REST service has no rate limiting middleware, making this trivially repeatable at high frequency.

### Finding Description

**Exact code path:**

In `rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler` (lines 71–72): [1](#0-0) 

calls `negotiate()` (lines 121–135): [2](#0-1) 

`negotiate()` returns `false` when `cachedResponse.compressed` is `true` but `negotiator.encoding(['gzip'])` does not return `'gzip'` — which happens whenever the client sends `accept-encoding: identity`, `accept-encoding: deflate`, or `accept-encoding: *;q=0`. When `negotiate()` returns `false`, line 72 calls `cachedResponse.getUncompressedBody()`.

In `rest/model/cachedApiResponse.js`, `getUncompressedBody()` (lines 29–31): [3](#0-2) 

calls `unzipSync()` **synchronously on every invocation** with no memoization of the decompressed result. The decompressed body is never stored back; it is recomputed from scratch on every cache hit.

**Root cause / failed assumption:** The design assumes that most clients will accept gzip (the common browser/client default), so the decompression path is treated as a rare fallback. In reality, any client can force this path by explicitly advertising a non-gzip encoding preference. The `CachedApiResponse` object stores only the compressed form and recomputes the uncompressed form on demand without caching it.

**Why the cache key does not help:** The cache key is generated from the URL only (line 151–153), explicitly excluding `accept-encoding`: [4](#0-3) 

So the same compressed `CachedApiResponse` object is returned for all clients regardless of their encoding preference, and every non-gzip client triggers `unzipSync()` against it.

**No rate limiting in the REST service:** `rest/server.js` registers no rate-limiting middleware: [5](#0-4) 

The only throttling in the repository is in the `web3` Java service, which is a separate component and does not protect the REST Node.js service.

### Impact Explanation

`unzipSync()` is a **synchronous** call that blocks the Node.js event loop for the duration of decompression. For large cached responses (e.g., paginated account/transaction lists), this can be tens to hundreds of milliseconds of blocked CPU per request. Because Node.js is single-threaded, a sustained flood of requests with `accept-encoding: identity` to any popular cached endpoint will starve all other concurrent requests of CPU time, degrading or denying service to legitimate users. The impact scales with response body size and request rate, and is amplified by the fact that the cache is working as intended (high hit rate), meaning the attacker never triggers database queries — only the decompression bottleneck.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only a single non-standard HTTP header (`accept-encoding: identity`). It is fully reproducible with any HTTP client (curl, wrk, ab). The attacker does not need to know internal details — any publicly documented cacheable endpoint suffices. The attack is repeatable indefinitely and can be automated trivially. The `accept-encoding: identity` value is a valid, RFC-compliant header value, so it will not be blocked by standard WAF rules that only filter malformed requests.

### Recommendation

1. **Memoize the decompressed body** in `CachedApiResponse.getUncompressedBody()`: store the result of `unzipSync()` in a private field on first call and return the cached value on subsequent calls.
2. **Alternatively**, decompress once at cache-store time in `responseCacheUpdateHandler` and store both the compressed and uncompressed forms in the cached object, so `getUncompressedBody()` becomes a simple field access.
3. **Add rate limiting** to the REST Node.js service (e.g., `express-rate-limit`) to bound the number of requests per IP per second, independent of the decompression fix.

### Proof of Concept

**Preconditions:**
- Response caching and Redis are enabled (`config.cache.response.enabled = true`, `config.redis.enabled = true`)
- Response compression is enabled (`config.cache.response.compress = true`)
- At least one cacheable endpoint has been warmed up (e.g., `/api/v1/accounts/0.0.12345/rewards`)

**Steps:**

1. Warm up the cache with a normal request (so the response is stored compressed):
   ```
   curl -H "accept-encoding: gzip" http://mirror-node/api/v1/accounts/0.0.12345/rewards
   ```

2. Flood the server with requests that force decompression on every cache hit:
   ```
   wrk -t8 -c200 -d60s \
     -H "accept-encoding: identity" \
     http://mirror-node/api/v1/accounts/0.0.12345/rewards
   ```

3. **Observed result:** CPU usage on the mirror-node REST process spikes due to repeated synchronous `unzipSync()` calls. Latency for all other concurrent requests increases proportionally. No database queries are issued (pure cache-hit path), confirming the bottleneck is decompression alone.

4. **Contrast:** Running the same load with `accept-encoding: gzip` serves the pre-compressed buffer directly (line 22 of `cachedApiResponse.js`) with no decompression, resulting in significantly lower CPU usage. [6](#0-5)

```javascript
const negotiate = (cachedResponse, req, res) => {
  res.setHeader(VARY_HEADER, 'accept-encoding');

  if (cachedResponse.compressed) {
    const negotiator = new Negotiator(req);
    if (negotiator.encoding([GZIP_ENCODING]) === GZIP_ENCODING) {
      res.setHeader(CONTENT_ENCODING_HEADER, GZIP_ENCODING);
      res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getLength());
      return true;
    }
  }

  res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getUncompressedLength());
  return false;
};
```
