### Title
Unauthenticated DB Connection Exhaustion via All-Zeros EVM Address in GraphQL `getByEvmAddressAndType()`

### Summary
An unprivileged external user can send a flood of GraphQL `account` queries with `evmAddress: "0x0000000000000000000000000000000000000000"`, which passes schema validation, triggers the `ByteBuffer` zero-prefix check in `EntityServiceImpl.getByEvmAddressAndType()`, and causes `entityRepository.findById(0L)` to be called on every request. The GraphQL module has no request-rate limiting, so a sustained flood can exhaust the database connection pool and render the mirror node's GraphQL API unavailable.

### Finding Description

**Code path:**

`AccountController.account()` (line 51–54) receives the `evmAddress` field and calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)`. [1](#0-0) 

The schema declares `evmAddress: String @Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")`. The string `0x0000000000000000000000000000000000000000` (40 hex zeros with optional `0x` prefix) fully satisfies this pattern. [2](#0-1) 

Inside `EntityServiceImpl.getByEvmAddressAndType()`:

```
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);   // → 20 zero bytes
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {     // true for all-zeros
    return entityRepository.findById(buffer.getLong())   // getLong() on remaining 8 zero bytes → findById(0L)
           .filter(e -> e.getType() == type);
}
``` [3](#0-2) 

`decodeEvmAddress` strips the `0x` prefix and hex-decodes, producing 20 zero bytes with no error. [4](#0-3) 

`entityRepository.findById(0L)` is a Spring Data `CrudRepository` call that issues `SELECT * FROM entity WHERE id = 0` on every invocation. Entity ID 0 never exists in Hedera, so it always returns empty — but the DB round-trip still occurs. [5](#0-4) 

**Why existing checks are insufficient:**

`GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`. These limit the structural complexity of a single query document, not the number of independent HTTP requests per second. [6](#0-5) 

The bucket4j rate-limiting (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is wired exclusively to `ContractCallRequest` processing. It is entirely absent from the `graphql` module. [7](#0-6) 

No authentication is required to reach the GraphQL endpoint.

### Impact Explanation
A sustained flood of cheap, structurally valid GraphQL requests each forces one synchronous JDBC call. Once the connection pool is saturated, all subsequent queries queue or fail, making the mirror node's GraphQL API completely unresponsive. The mirror node is a read-only service and does not participate in Hedera consensus, so this does not cause a network-wide transaction halt; the correct impact classification is **mirror node GraphQL DoS / availability loss**.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single, trivially constructed query string. It is repeatable indefinitely from a single host or botnet. Any attacker with HTTP access to the GraphQL endpoint can execute it immediately.

### Recommendation
1. **Add request-rate limiting to the GraphQL module.** Port the existing bucket4j pattern from `web3/ThrottleConfiguration` into the `graphql` module, applying a per-IP or global requests-per-second cap at the servlet filter or Spring Security layer.
2. **Reject the all-zeros EVM address early.** In `getByEvmAddressAndType()` (or in a schema directive), explicitly reject `0x0000000000000000000000000000000000000000` before any DB call, since entity ID 0 is not a valid Hedera entity.
3. **Set a DB connection pool timeout and maximum pool size** appropriate for the expected load, so exhaustion degrades gracefully rather than hanging threads.

### Proof of Concept

```bash
# Send 10 000 concurrent GraphQL requests with the all-zeros EVM address
seq 1 10000 | xargs -P 200 -I{} curl -s -X POST \
  -H "Content-Type: application/json" \
  -d '{"query":"{ account(input:{evmAddress:\"0x0000000000000000000000000000000000000000\"}){memo} }"}' \
  http://<mirror-node-graphql-host>/graphql
```

Each request reaches `EntityServiceImpl.getByEvmAddressAndType()`, passes the `ByteBuffer` zero check, and issues `SELECT * FROM entity WHERE id = 0`. With no rate limiting, the DB connection pool exhausts and subsequent legitimate queries time out or are rejected.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
