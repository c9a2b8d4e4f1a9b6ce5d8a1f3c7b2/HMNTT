### Title
Unbounded Log Amplification via Unauthenticated Requests to `TracingMiddleware` with Rate Limiting Disabled by Default

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` unconditionally emits a `log.Info` entry for every non-internal HTTP request with no application-level rate limiting or log throttling. The only rate-limiting defense is an opt-in Traefik middleware chain that is **disabled by default** (`global.middleware: false`). An unauthenticated attacker can flood the Rosetta API with high-volume requests, exhausting disk space and creating a window during which Hashgraph history tampering events go unlogged.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 43–61 — specifically line 58:

```go
// line 47: path is fully user-controlled (includes query string)
path := request.URL.RequestURI()
// ...
// line 52-53: message built from user-controlled path
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))

// line 55-59: every non-internal path unconditionally logs at Info level
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)   // ← no throttle, no dedup, no size cap
}
```

`main.go` lines 217–219 confirm `TracingMiddleware` wraps every request at the application level with no rate-limiting layer beneath it:

```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

**Root cause — failed assumption:** The design assumes a Traefik rate-limiting middleware chain will always be active in front of the application. This assumption is false: `charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false` as the default, and the Traefik `Middleware` CRD is only rendered when `global.middleware` is `true` (template guard at `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3: `{{ if and .Values.global.middleware .Values.middleware }}`). In the default deployment, no rate limiting exists at any layer.

**Secondary amplifier:** `path` is taken from `request.URL.RequestURI()` (line 47), which includes the full query string. An attacker can craft requests with kilobyte-scale query strings, multiplying per-entry log size. Additionally, `getClientIpAddress` (lines 63–74) blindly trusts `X-Real-IP` and `X-Forwarded-For` headers, allowing IP spoofing that defeats any manual IP-based blocking and corrupts forensic log data.

### Impact Explanation

Disk exhaustion on the mirror node host causes the logging subsystem to fail. Once disk is full:
- All subsequent `log.Info` / `log.Error` calls silently drop, including any security-relevant events (DB errors, unexpected state transitions, authentication failures).
- The node continues serving stale or manipulated block/transaction data with no audit trail.
- This directly enables the "undetected history tampering" scenario: an attacker who can influence the mirror node's data source (e.g., via a compromised upstream) can act during the logging blackout with no forensic record.

Severity: **High** — availability of the audit log is a prerequisite for detecting integrity violations in Hashgraph history.

### Likelihood Explanation

- **No authentication required**: all Rosetta Data API endpoints (`/block`, `/block/transaction`, `/account/balance`, etc.) are publicly accessible by the Rosetta specification.
- **Default configuration is vulnerable**: `global.middleware: false` means any standard Helm install is unprotected.
- **Trivially repeatable**: a single machine with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain tens of thousands of requests per second against a Go HTTP server.
- **IP spoofing via headers**: the `X-Forwarded-For` trust (lines 64–68) means even naive IP-rotation bypasses manual blocking.

### Recommendation

1. **Application-level rate limiting (primary fix):** Add a token-bucket or sliding-window rate limiter directly in the Go middleware chain, before `TracingMiddleware`, so it is enforced regardless of infrastructure configuration. Libraries such as `golang.org/x/time/rate` or `github.com/ulule/limiter` integrate cleanly as `http.Handler` wrappers.

2. **Log throttling:** Introduce per-IP or global log-rate capping inside `TracingMiddleware` (e.g., emit at most N log lines per second; drop or count-aggregate the rest).

3. **Change the default:** Set `global.middleware: true` in `charts/hedera-mirror-rosetta/values.yaml` so the Traefik `inFlightReq`/`rateLimit` chain is active out of the box.

4. **Validate `X-Forwarded-For`:** Only trust proxy headers when the request originates from a known trusted proxy CIDR; otherwise fall back to `RemoteAddr`.

5. **Log rotation with size caps:** Ensure the logging backend (logrus output) is configured with log rotation and maximum disk usage limits as a defense-in-depth measure.

### Proof of Concept

**Preconditions:**
- Default Helm deployment (`global.middleware: false`)
- Rosetta API port reachable (default: 5700)

**Steps:**

```bash
# 1. Confirm no rate limiting is active (expect immediate 200 responses at high rate)
hey -n 100000 -c 200 -m POST \
  -H "Content-Type: application/json" \
  -H "X-Forwarded-For: 1.2.3.4" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}' \
  http://<rosetta-host>:5700/block

# 2. Amplify per-entry log size with a long query string on any endpoint
hey -n 100000 -c 200 \
  "http://<rosetta-host>:5700/block?$(python3 -c 'print("A"*8000)')"

# 3. Monitor disk usage on the node
watch -n1 df -h /var/log

# Expected result: /var/log (or wherever logrus writes) fills to 100%;
# subsequent log.Info / log.Error calls in TracingMiddleware are silently dropped;
# security-relevant events during this window produce no audit record.
```

**Verification:** After disk exhaustion, trigger a known-loggable event (e.g., a DB connection error) and confirm no log entry appears, demonstrating the audit blackout window.
