### Title
Partial (206) Responses Cached and Validated as 304 Not Modified, Causing Indefinite Stale Partial Data

### Summary
The `responseCacheUpdateHandler` caches HTTP 206 (Partial Content) responses because `httpStatusCodes.isSuccess` includes the full 2xx range (200–299). When a subsequent request arrives with an `If-None-Match` header matching the ETag of a cached 206 response, `responseCacheCheckHandler` unconditionally returns 304 without checking whether the cached entry was partial. Any client that received a 206 during a network partition can be told indefinitely that their incomplete data is "not modified."

### Finding Description

**Step 1 — 206 responses are cached.**

In `rest/constants.js` line 159, `isSuccess` is defined as:
```js
isSuccess: (code) => code >= 200 && code < 300,
```
206 satisfies this predicate. In `rest/middleware/responseCacheHandler.js` line 95, the cache-update guard is:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```
A 206 response with a non-empty body and a positive TTL is therefore written to Redis with `statusCode = 206` (line 114–115).

**Step 2 — ETag matching ignores the cached status code.**

In `responseCacheCheckHandler` (lines 52–54):
```js
const conditionalHeader = req.get(CONDITIONAL_HEADER);
const clientCached = conditionalHeader && conditionalHeader === cachedResponse.headers[ETAG_HEADER];
const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
```
The only condition for returning 304 is that the `If-None-Match` value equals the cached ETag. There is no check that `cachedResponse.statusCode !== 206`. A cached 206 entry is treated identically to a cached 200 entry for ETag validation purposes.

**Step 3 — 206 is emitted during partition.**

Controllers explicitly set 206 when data is incomplete, e.g. in `contractController.js` lines 1026–1029:
```js
if (isNil(contractResults[0].callResult)) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
```
and in `tokens.js` lines 1048–1051:
```js
if (nft?.createdTimestamp == null) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
```

**Exploit flow:**
1. During a network partition the mirror node returns a 206 with body `{"call_result":"0x",...}` and an ETag (e.g. `W/"abc123"`).
2. `responseCacheUpdateHandler` writes this partial body + ETag to Redis with the configured TTL.
3. Any client (including an unprivileged external user) that observed the 206 response and its ETag sends a follow-up request: `GET /api/v1/contracts/{id}/results/{ts}` with `If-None-Match: W/"abc123"`.
4. `responseCacheCheckHandler` finds the Redis entry, sees the ETag matches, and returns `304 Not Modified` — without ever re-querying the database.
5. The client's HTTP cache treats the partial body as authoritative and will not re-fetch it until the ETag changes or the client's own cache expires.

### Impact Explanation

Clients (wallets, indexers, dApps) that cached the 206 partial response will permanently use incomplete blockchain data for the affected resource. Fields such as `call_result`, `gas_used`, `state_changes`, or NFT `createdTimestamp` will remain null/missing. Because the server actively confirms the data as "not modified," clients have no signal to retry. This can corrupt downstream state (e.g., incorrect gas accounting, missing NFT provenance) for the lifetime of the client's cache, which may be unbounded.

### Likelihood Explanation

No privilege is required. The attacker only needs to have received a 206 response (or observe one via a shared proxy/CDN) and replay its ETag. Network partitions in distributed ledger mirror nodes are a documented operational condition (the codebase explicitly handles them with `PARTIAL_CONTENT`). The window during which a 206 is cached is bounded by the Redis TTL, but once a client has the ETag the 304 loop persists in the client's own cache indefinitely. The attack is trivially repeatable with a single HTTP header.

### Recommendation

1. **Do not cache 206 responses.** Change the guard in `responseCacheUpdateHandler` (`rest/middleware/responseCacheHandler.js` line 95) to exclude partial content:
   ```js
   if (responseBody && responseCacheKey &&
       (isUnmodified || (httpStatusCodes.isSuccess(res.statusCode) &&
        res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code))) {
   ```
2. **Alternatively, refuse 304 for cached 206 entries.** In `responseCacheCheckHandler` (line 53), add a guard:
   ```js
   const clientCached = conditionalHeader &&
     conditionalHeader === cachedResponse.headers[ETAG_HEADER] &&
     cachedResponse.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code;
   ```
   Option 1 is preferred because it prevents partial data from polluting the cache at all.

### Proof of Concept

```bash
# 1. During a partition, fetch the partial resource and capture the ETag
ETAG=$(curl -si https://mirror-node/api/v1/contracts/0.0.5001/results/167654.000123456 \
  | grep -i '^etag:' | awk '{print $2}' | tr -d '\r')
# Response: HTTP/1.1 206 Partial Content  ETag: W/"abc123"

# 2. Confirm the 206 body is cached (wait a moment, then fetch without If-None-Match)
curl -si https://mirror-node/api/v1/contracts/0.0.5001/results/167654.000123456
# Response: HTTP/1.1 206 Partial Content  (served from Redis)

# 3. Send If-None-Match with the 206 ETag — server returns 304
curl -si https://mirror-node/api/v1/contracts/0.0.5001/results/167654.000123456 \
  -H "If-None-Match: $ETAG"
# Response: HTTP/1.1 304 Not Modified
# Client now treats its partial body as authoritative indefinitely.
```
