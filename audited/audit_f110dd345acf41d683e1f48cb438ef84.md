### Title
Unbounded Per-IP TCP Connections Allow Bypass of `maxConcurrentCallsPerConnection` Cap Leading to Resource Exhaustion

### Summary
The gRPC server enforces a maximum of 5 concurrent calls per connection via `NettyProperties.maxConcurrentCallsPerConnection`, but imposes no limit on the number of TCP connections accepted from a single IP address or in total. An unauthenticated attacker can open an arbitrary number of connections and multiply the effective concurrency by 5 per connection, exhausting the JVM thread pool, file descriptors, and database connection pool.

### Finding Description
**Code path:**

- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java` line 14: `maxConcurrentCallsPerConnection = 5`
- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java` lines 28–35: the `ServerBuilderCustomizer` bean applies only two settings to the `NettyServerBuilder`: `executor` and `maxConcurrentCallsPerConnection`. [1](#0-0) [2](#0-1) 

No call to `maxConnectionAge`, `maxConnectionIdle`, `maxInboundConnections`, or any per-IP connection throttle is present anywhere in the gRPC module.

**Root cause:** The design assumes `maxConcurrentCallsPerConnection` is a global concurrency cap, but it is only a *per-connection* cap. Without a bound on the number of connections, the effective global concurrency is `N_connections × 5`, where `N_connections` is unbounded.

**Why existing checks fail:**

1. The GCP `backendPolicy.maxRatePerEndpoint: 250` in `charts/hedera-mirror-grpc/values.yaml` is only active when `gateway.gcp.enabled: true` (an opt-in infrastructure path), limits *request rate* not *open connections*, and is bypassed entirely when the service is accessed directly or via Traefik ingress. [3](#0-2) 

2. The Traefik middleware configures only a circuit breaker and retry — no `InFlightReq`, `RateLimit`, or IP-based connection cap. [4](#0-3) 

3. Prometheus alerts (`GrpcHighFileDescriptors`, `GrpcHighCPU`) fire *after* exhaustion occurs — they are observability, not prevention. [5](#0-4) 

### Impact Explanation
Each open gRPC streaming call (`subscribeToTopic`) holds a Reactor subscription, a database polling loop, and a thread from the shared executor. With 2,000 connections × 5 streams = 10,000 concurrent calls, the JVM thread pool saturates, the HikariCP database pool (default bounded) is exhausted, and the OS file-descriptor limit is hit. Legitimate subscribers receive `RESOURCE_EXHAUSTED` or connection timeouts. The service becomes effectively unavailable — a complete denial of service against the public gRPC API.

### Likelihood Explanation
No authentication is required to open a TCP connection to port 5600. A single commodity machine can sustain thousands of simultaneous TCP connections with minimal CPU. The attack is trivially scriptable with any gRPC client library (e.g., `grpc-go`, `grpcurl` in a loop). It is repeatable and persistent as long as the attacker holds connections open. The absence of any application-level or infrastructure-level per-IP connection limit makes this straightforward to execute.

### Recommendation
Apply multiple layers of defense:

1. **Application layer** — Add `maxConnectionAge` and `maxConnectionIdle` to `GrpcConfiguration` to recycle connections and prevent indefinite hold:
   ```java
   serverBuilder.maxConnectionAge(30, TimeUnit.MINUTES);
   serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
   ```
   Add a `maxConcurrentCallsPerConnection` field to `NettyProperties` alongside a new `maxConnections` field and call `serverBuilder.maxInboundConnections(...)` (available in grpc-java ≥ 1.60).

2. **Expose `maxConcurrentCallsPerConnection` as a tunable** — The current default of 5 is very low; document and expose it so operators can tune it alongside a connection cap. [6](#0-5) 

3. **Infrastructure layer** — Enable a Traefik `InFlightReq` middleware with a per-source-IP limit on the gRPC ingress, or enforce the GCP `maxRatePerEndpoint` unconditionally.

4. **Network layer** — Add a Kubernetes `NetworkPolicy` to restrict which sources can reach port 5600, and configure `iptables`/cloud firewall rules to cap simultaneous connections per source IP.

### Proof of Concept
```bash
# Requires grpc-go or any gRPC client; run from a single machine
for i in $(seq 1 2000); do
  # Each invocation opens a new TCP connection and holds 5 streaming calls
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <grpc-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```
With 2,000 background processes each holding one stream (or a single client library opening 2,000 connections with 5 streams each), the server's thread pool and DB connection pool are exhausted. Subsequent legitimate `subscribeTopic` calls return `RESOURCE_EXHAUSTED` or time out. The `GrpcHighFileDescriptors` and `GrpcHighCPU` Prometheus alerts fire, but the service is already unavailable.
