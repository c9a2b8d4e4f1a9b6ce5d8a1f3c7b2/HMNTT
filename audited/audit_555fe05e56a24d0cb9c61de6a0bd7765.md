### Title
Unbounded HTTP Method String Used as Prometheus Label Enables Cardinality-Based Memory Exhaustion (DoS)

### Summary
`MetricsMiddleware()` in `rosetta/app/middleware/metrics.go` delegates instrumentation to `github.com/weaveworks/common/middleware.Instrument`, which uses the raw `r.Method` value directly as a Prometheus label across all four metric vectors. Because Go's `net/http` server accepts any valid HTTP token as a method (up to the 1 MB `MaxHeaderBytes` default), an unprivileged attacker can send requests with arbitrarily many distinct, long method strings, each creating a new Prometheus time series. With no method sanitization, length cap, or rate limiting, this causes unbounded memory growth that degrades or crashes the server for all users.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

The `weaveworks/common/middleware.Instrument.Wrap()` extracts `r.Method` verbatim and passes it as the `"method"` label to every metric vector on each request. No normalization, truncation, or allowlist check is applied.

**Root cause / failed assumption:** The code assumes the HTTP method will always be one of a small, bounded set of standard verbs (GET, POST, etc.). In reality, Go's `net/http` server accepts any RFC 7230-compliant token as a method. The server in `rosetta/main.go` (lines 220–227) sets no `MaxHeaderBytes` override (defaulting to 1 MB) and performs no method validation before the metrics middleware runs.

**Middleware chain order** (`rosetta/main.go`, lines 217–219):
```
corsMiddleware → tracingMiddleware → MetricsMiddleware → router
```
`MetricsMiddleware` runs before the router, so even requests that the router ultimately rejects still create Prometheus time series. The `InflightRequests` gauge is incremented at the start of the request, before any routing decision.

**Exploit flow:**
1. Attacker sends HTTP requests with unique, long method tokens (e.g., `AAAA...1`, `AAAA...2`, …).
2. Go's `net/http` server accepts each (valid token, within 1 MB header limit).
3. `Instrument.Wrap()` calls `requestInflightGauge.With(prometheus.Labels{"method": "AAAA...N", "route": ...})` and similarly for the three histogram vectors.
4. Each unique method string creates new time series in all four metric vectors. `requestDurationHistogram` alone has 6 buckets + `_sum` + `_count` = 8 series per unique label combination.
5. Prometheus client_golang stores all time series in memory indefinitely (no eviction).
6. Memory grows without bound; the process is eventually OOM-killed.

**Existing checks reviewed and shown insufficient:**
- `ReadHeaderTimeout` / `ReadTimeout` (lines 224–226): limit connection duration, not method length or request rate.
- No `MaxHeaderBytes` set: defaults to 1,048,576 bytes — still allows ~1 MB method strings.
- No allowlist or length check on `r.Method` anywhere in the rosetta middleware stack.
- No rate limiting visible in the server setup.

### Impact Explanation
Each unique method string permanently allocates new Prometheus time series across four metric vectors. With 10,000-character method strings and thousands of unique values, memory pressure grows rapidly. The Prometheus registry holds strong references to all label combinations; there is no TTL or eviction. Sustained attack leads to OOM termination of the rosetta process, causing a complete denial of service for all legitimate users of the mirror node's Rosetta API. The impact is server-wide, not per-user.

### Likelihood Explanation
No authentication or privilege is required — the Rosetta API is an externally accessible HTTP service. The attack requires only a basic HTTP client capable of sending custom method strings (e.g., `curl`, `netcat`, or a simple script). It is trivially repeatable and automatable. A single attacker with a modest script can exhaust memory within minutes by cycling through unique method strings at high request rates.

### Recommendation
1. **Normalize the method label before recording metrics.** Replace `r.Method` with a sanitized value:
   ```go
   func sanitizeMethod(m string) string {
       allowed := map[string]bool{"GET":true,"POST":true,"PUT":true,"DELETE":true,"PATCH":true,"HEAD":true,"OPTIONS":true}
       if allowed[strings.ToUpper(m)] {
           return strings.ToUpper(m)
       }
       return "UNKNOWN"
   }
   ```
   Apply this before passing to `Instrument` (e.g., wrap `r.Method` via a custom `http.Handler` that rewrites `r.Method` before delegating).

2. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g., 8 KB) to limit the size of the request line and headers.

3. **Add rate limiting** (e.g., `golang.org/x/time/rate`) per IP before the metrics middleware.

### Proof of Concept
```bash
#!/bin/bash
# Send 50,000 requests each with a unique 500-char method string
# No authentication required; target is the public Rosetta API port
for i in $(seq 1 50000); do
  METHOD=$(python3 -c "print('A'*500 + str($i))")
  printf "${METHOD} /network/list HTTP/1.1\r\nHost: target:5700\r\nContent-Type: application/json\r\nContent-Length: 2\r\n\r\n{}" | nc -q1 target 5700 &
done
wait
# Each request creates new Prometheus time series across 4 metric vectors.
# Monitor RSS of the rosetta process: it will grow unboundedly until OOM.
```
