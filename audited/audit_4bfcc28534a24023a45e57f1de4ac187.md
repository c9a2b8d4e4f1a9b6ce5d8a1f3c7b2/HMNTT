### Title
Bracket-Notation Query Parameter Bypasses Type Guard, Injecting Plain Object into String-Expecting Downstream Code

### Summary
The `requestQueryParser()` function configures `qs` with `depth: 1`, which permits one level of bracket-notation nesting (e.g., `?order[b]=1`). The `strictDepth: true` option only throws when depth is *exceeded* (depth > 1), so depth-1 payloads pass through silently as plain objects. `lowerCaseQueryValue` explicitly returns non-strings unchanged, so the object reaches downstream filter-validation code that unconditionally calls `.split(':')` on the value, throwing an unhandled `TypeError`.

### Finding Description

**Code path:**

`rest/middleware/requestHandler.js` — `queryOptions` sets `depth: 1` and `strictDepth: true`: [1](#0-0) 

`qs.parse('order[b]=1', queryOptions)` returns `{ order: { b: '1' } }` — depth 1 is within the allowed limit, so `strictDepth` does not throw.

`canonicalizeValue` dispatches to `lowerCaseQueryValue` for the `order` key: [2](#0-1) 

`lowerCaseQueryValue` has an explicit type guard that returns non-strings unchanged: [3](#0-2) 

The plain object `{ b: '1' }` is stored as `req.query.order`.

Downstream, `paramValidityChecks` (called from `filterValidityChecks` → `validateFilters` → `buildAndValidateFilters`) does: [4](#0-3) 

`{ b: '1' }.split(':')` throws `TypeError: opAndVal.split is not a function`. This propagates out of the async route handler (e.g., `getAccounts`, `getContractResults`) as an unhandled promise rejection. [5](#0-4) 

**Root cause:** The code assumes `qs` with `depth: 1` + `strictDepth: true` prevents nested objects from reaching value-processing code. This assumption is wrong: `strictDepth` only rejects depth > 1; depth-1 bracket notation is fully parsed into a plain object.

### Impact Explanation

Every API endpoint that calls `buildAndValidateFilters` with `order` or `result` in its accepted parameter set will throw an unhandled `TypeError` when the crafted query string is supplied. In Node.js ≥ 15 (default `--unhandled-rejections=throw`), repeated unhandled promise rejections crash the process. Even in earlier versions, every such request returns an uncontrolled 500 rather than a clean `400 Bad Request`, and the error bypasses the normal `InvalidArgumentError` path entirely.

### Likelihood Explanation

No authentication or special privilege is required. The payload is a single, trivially crafted HTTP query string (`?order[b]=1`). It is repeatable at will, requires no knowledge of internal state, and works against any endpoint that accepts an `order` or `result` parameter. Any automated scanner or curious user could discover it.

### Recommendation

1. **Reject nested objects at the parser boundary.** After `qs.parse`, iterate the result and throw (or return a 400) for any value that is a plain object (i.e., `typeof value === 'object' && !Array.isArray(value)`).
2. **Alternatively, set `depth: 0`** in `queryOptions`. This prevents qs from ever producing nested objects; bracket-notation keys are then treated as literal strings and will fail normal key validation.
3. **Harden `lowerCaseQueryValue` and `paramValidityChecks`** to guard against non-string, non-number inputs explicitly, so that even if an object leaks through, it produces a controlled `InvalidArgumentError` rather than an unhandled `TypeError`.

### Proof of Concept

```
# Against any endpoint that accepts ?order=
GET /api/v1/transactions?order[x]=asc HTTP/1.1
Host: <mirror-node-host>

# qs parses → { order: { x: 'asc' } }
# lowerCaseQueryValue returns { x: 'asc' } unchanged
# buildAndValidateFilters → paramValidityChecks → { x: 'asc' }.split(':') → TypeError
# Result: unhandled promise rejection / 500 error
```

Reproducible in isolation:
```js
import {requestQueryParser} from './rest/middleware/requestHandler.js';
const result = requestQueryParser('order[x]=asc');
console.log(result); // { order: { x: 'asc' } }  ← plain object, not a string
```
