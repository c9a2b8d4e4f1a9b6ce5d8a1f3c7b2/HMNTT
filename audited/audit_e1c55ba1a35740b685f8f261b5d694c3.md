### Title
Missing Back-Pressure and Rate-Limiting in `TopicServiceImpl.findById()` Enables Retry-Amplified Feedback Loop During DB Partition

### Summary
`TopicServiceImpl.findById()` performs a synchronous, blocking JPA repository call with no concurrency limit, no circuit breaker, and no application-level rate limiting. The Traefik ingress for `rest-java` is configured with a `retry` middleware (3 attempts, 100ms interval) but lacks the `inFlightReq` and `rateLimit` middlewares present in comparable services (e.g., `hedera-mirror-rosetta`). During a database network partition, an unprivileged attacker flooding `GET /api/v1/topics/{id}` causes thread-pool exhaustion that is amplified 3× by Traefik retries before the circuit breaker can open, creating a feedback loop that actively prevents the partition from healing.

### Finding Description

**Exact code path:**

`TopicServiceImpl.findById()` (line 20) issues a blocking synchronous call to `topicRepository.findById()` — a standard Spring Data JPA call backed by HikariCP: [1](#0-0) 

This is invoked directly from `TopicController.getTopic()` with no concurrency guard: [2](#0-1) 

**Root cause — missing middleware controls:**

The `rest-java` Traefik middleware chain is configured with only `circuitBreaker` and `retry`: [3](#0-2) 

Compare this to `hedera-mirror-rosetta`, which additionally configures `inFlightReq` (max 5 concurrent per IP) and `rateLimit` (10 req/s per host): [4](#0-3) 

There is also no application-level throttling in `rest-java` (unlike `web3`, which uses bucket4j `ThrottleManagerImpl`): [5](#0-4) 

**Exploit flow:**

1. Attacker (no credentials required) sends a high-rate flood of `GET /api/v1/topics/{id}` requests.
2. During a DB network partition, each `findById()` call blocks a JVM thread waiting for HikariCP to time out acquiring a connection.
3. Traefik's `retry` middleware retries each failed request **3 times** before the circuit breaker observes the error, multiplying in-flight DB connection attempts by 3×.
4. The circuit breaker threshold (`NetworkErrorRatio() > 0.10`) is only evaluated after errors accumulate; the retry loop feeds errors back into the circuit breaker's window, delaying its opening.
5. With no `inFlightReq` cap, the JVM thread pool and HikariCP connection pool are exhausted. New health-check connections and DB reconnection attempts cannot acquire threads or pool slots, preventing the partition from healing.

**Why existing checks fail:**

- The `circuitBreaker` fires reactively after 10% network error ratio, but the `retry` (3 attempts) runs first, tripling the error rate seen by the circuit breaker window and delaying its trigger.
- HikariCP's bounded pool provides a ceiling on DB connections but not on blocked JVM threads waiting for pool acquisition — those threads pile up until the servlet container's thread pool is exhausted.
- There is no per-IP or global request rate limit on the `rest-java` ingress path for `/api/v1/topics/{id}`. [3](#0-2) 

### Impact Explanation

An unauthenticated attacker can exhaust the JVM thread pool and HikariCP connection pool of the `rest-java` service during any DB connectivity degradation. This prevents the service from processing any requests (including health checks and DB reconnection probes), extending the effective outage duration beyond the actual partition window. The `retry` amplification means even a moderate request rate (e.g., 100 req/s) produces 300 DB connection attempts per second, saturating a typical pool of 10–20 connections almost instantly. Severity: **High** (availability impact, no authentication required, exploitable during any transient DB instability).

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the public API path `/api/v1/topics/{id}`, and no sophisticated tooling — a simple `ab` or `curl` loop suffices. DB partitions (even brief ones) are common in cloud environments. The absence of `inFlightReq` and `rateLimit` in the `rest-java` middleware (unlike `rosetta`) is a configuration gap that any attacker can exploit opportunistically or deliberately (by first inducing DB load to trigger a partition). Repeatability is high; the feedback loop re-engages on every partition event.

### Recommendation

1. **Add `inFlightReq` and `rateLimit` to the `rest-java` Traefik middleware** in `charts/hedera-mirror-rest-java/values.yaml`, mirroring the `rosetta` configuration:
   ```yaml
   middleware:
     - circuitBreaker:
         expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
     - inFlightReq:
         amount: 50          # tune to thread pool size
         sourceCriterion:
           ipStrategy:
             depth: 1
     - rateLimit:
         average: 100        # tune to expected traffic
         sourceCriterion:
           requestHost: true
     - retry:
         attempts: 3
         initialInterval: 100ms
   ```
2. **Add application-level back-pressure** in `TopicServiceImpl` or at the controller layer using a semaphore or bucket4j rate limiter (as done in `web3`'s `ThrottleManagerImpl`).
3. **Configure HikariCP `connectionTimeout`** to a short value (e.g., 2–5 s) so blocked threads fail fast and release, rather than holding for the full pool timeout.
4. **Move `retry` after `circuitBreaker`** in the middleware chain so the circuit breaker can open before retries amplify load.

### Proof of Concept

**Preconditions:** `rest-java` service running; DB connectivity degraded (e.g., firewall rule blocking port 5432, or DB under heavy load causing connection timeouts).

**Steps:**

```bash
# 1. Simulate DB partition (on the DB host or via network policy)
#    e.g., iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# 2. Flood the topic endpoint from an unprivileged client (no auth needed)
ab -n 100000 -c 500 https://<mirror-node-host>/api/v1/topics/0.0.1

# 3. Observe: JVM thread pool exhausts (all threads blocked in HikariCP.getConnection())
#    Traefik retries each request 3x, tripling effective DB connection attempts
#    Circuit breaker opens only after ~10% error ratio accumulates across retried requests
#    Health check endpoint /actuator/health/liveness becomes unresponsive
#    Even after DB partition heals, service cannot recover until thread pool drains

# 4. Verify: curl https://<mirror-node-host>/actuator/health/liveness
#    Returns 503 or times out, confirming thread pool exhaustion
```

**Result:** The service remains unavailable for a period significantly longer than the actual DB partition, because the retry-amplified thread exhaustion prevents reconnection probes from succeeding.
