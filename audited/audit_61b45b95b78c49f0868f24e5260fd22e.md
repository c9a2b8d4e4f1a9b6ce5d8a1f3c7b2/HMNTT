### Title
Operator Private Key Exposed as Plain-Text Environment Variable with Hardcoded Insecure Default

### Summary
The pinger's operator private key is read from the `HIERO_MIRROR_PINGER_OPERATOR_KEY` environment variable as a plain string and stored unprotected in the `config` struct. A hardcoded default key is baked into the source code. Any attacker who can read the process environment — via `/proc/<pid>/environ`, a shared-PID-namespace sidecar, or a path-traversal in a co-located web app — obtains the raw private key and can sign and submit arbitrary Hiero transactions that are cryptographically indistinguishable from legitimate pinger transactions.

### Finding Description
**Code path:**

- `pinger/config.go` line 40: the key is read with `os.Getenv` via `envOr()` and stored as a plain `string` field in the `config` struct. A hardcoded default (`302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137`) is compiled into the binary and used when the env var is absent.
- `pinger/sdk_client.go` lines 40–45: `hiero.PrivateKeyFromString(cfg.operatorKey)` parses the raw string into a signing key, which is then registered with `client.SetOperator(opID, opKey)`.

**Root cause:** The private key lives its entire lifetime as a plain, unprotected string in the process environment and in Go heap memory. There is no use of a secrets manager, encrypted secret store, HSM, or even a memory-locked buffer. `os.Getenv` leaves the value in the process's environment block, which is readable at `/proc/<pid>/environ` by any process sharing the same UID or PID namespace.

**Failed assumption:** The design assumes the environment is a trust boundary. In containerised deployments (Kubernetes, Docker Compose) this assumption frequently fails: a co-located web app with a path-traversal bug, a debug sidecar with `shareProcessNamespace: true`, or a container escape all give an attacker read access to `/proc/1/environ` or `/proc/<pinger_pid>/environ` without root on the host.

### Impact Explanation
Once the attacker holds the raw private key they can:
1. Sign and submit `CryptoTransfer` transactions from the operator account to any destination — draining the account entirely.
2. Submit any other transaction type (token operations, contract calls, etc.) that the operator account is authorised for.
3. All such transactions carry a valid cryptographic signature; the Hiero network and mirror node cannot distinguish them from legitimate pinger traffic.

The hardcoded default key compounds the impact: any deployment that omits the env var uses a publicly known key, making exploitation trivial with zero environment-read capability.

### Likelihood Explanation
- Kubernetes `shareProcessNamespace: true` is a common debugging configuration and exposes `/proc/<pid>/environ` to every container in the pod.
- Path-traversal or SSRF vulnerabilities in a co-located service (e.g., a REST API in the same pod) can read `/proc/1/environ` without any privilege escalation.
- The hardcoded default means deployments that forget to set the env var are immediately exploitable by anyone who knows the source code (it is public).
- The attacker needs no special tooling beyond an HTTP client or shell access to a co-located process.

### Recommendation
1. **Never pass private keys as environment variables.** Use a secrets manager (Vault, AWS Secrets Manager, Kubernetes `Secret` mounted as a file with `0400` permissions) and read the key from a file path, not an env var.
2. **Remove the hardcoded default key** from `config.go` line 40. Fail fast with a clear error if the key is not supplied.
3. If env vars must be used, unset the variable immediately after reading it (`os.Unsetenv("HIERO_MIRROR_PINGER_OPERATOR_KEY")`) to reduce the window of exposure in `/proc/<pid>/environ`.
4. Consider using a hardware-backed key or a signing service so the raw private key never exists in process memory.

### Proof of Concept
**Preconditions:** Attacker has read access to the pinger container's filesystem or PID namespace (e.g., via a path-traversal in a co-located service, or `kubectl exec` into a debug sidecar with `shareProcessNamespace: true`).

```
# Step 1 – extract the key from the process environment
$ cat /proc/$(pgrep pinger)/environ | tr '\0' '\n' | grep OPERATOR_KEY
HIERO_MIRROR_PINGER_OPERATOR_KEY=302e0201003005...

# Step 2 – use the Hiero SDK (or any compatible tool) to sign a transfer
# targeting an attacker-controlled account
$ cat > exploit.go << 'EOF'
package main
import (
    hiero "github.com/hiero-ledger/hiero-sdk-go/v2/sdk"
    "fmt"
)
func main() {
    key, _ := hiero.PrivateKeyFromString("<extracted_key>")
    client, _ := hiero.ClientForName("mainnet")
    opID, _ := hiero.AccountIDFromString("<operator_id>")
    client.SetOperator(opID, key)
    // Transfer entire balance to attacker account
    tx, _ := hiero.NewTransferTransaction().
        AddHbarTransfer(opID, hiero.HbarFrom(-10000, hiero.HbarUnits.Tinybar)).
        AddHbarTransfer(<attacker_account>, hiero.HbarFrom(10000, hiero.HbarUnits.Tinybar)).
        Execute(client)
    fmt.Println(tx)
}
EOF
$ go run exploit.go
# Transaction accepted by network; signature is valid and indistinguishable
# from a legitimate pinger transaction.
```
