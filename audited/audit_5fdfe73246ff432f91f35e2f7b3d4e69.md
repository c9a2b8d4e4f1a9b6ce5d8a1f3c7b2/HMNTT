### Title
Unbounded Alias Processing in `resolveAccountAliases()` Enables Resource Exhaustion via DB Query Amplification and Unbounded Memory Allocation

### Summary
The `resolveAccountAliases()` function in `ConstructionMetadata` processes a caller-controlled, comma-separated list of account aliases with no upper bound on count and no deduplication. An unprivileged attacker who knows a single valid alias can repeat it thousands of times in one request, triggering one synchronous database query per entry and causing `strings.Join()` to allocate a proportionally large result string. The combination of unbounded DB queries and unbounded memory allocation can increase node resource consumption well above 30% with no authentication required.

### Finding Description

**Exact code path:**

`ConstructionMetadata()` (line 159) delegates to `resolveAccountAliases()` (lines 557–589):

```go
// line 575-588
var accountMap []string
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    if err != nil {
        return emptyResult, errors.ErrInvalidAccount
    }

    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)   // DB query per alias
    if rErr != nil {
        return emptyResult, rErr
    }
    accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
}

return strings.Join(accountMap, ","), nilErr   // single large allocation
``` [1](#0-0) 

**Root cause — two compounding failures:**

1. **No alias count limit.** The loop at line 575 iterates over every comma-separated token in the attacker-supplied string with no guard on `len(accountMap)`.
2. **No deduplication.** The same alias token is processed independently on every occurrence, so a single known-valid alias repeated N times produces N database queries and N `accountMap` entries.

**`GetAccountId()` issues a real DB query for every alias-based account ID:**

```go
// account.go lines 142-158
if !accountId.HasAlias() {
    return accountId, nil   // numeric IDs skip DB; alias-based IDs do not
}
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()
// ... executes selectCurrentCryptoEntityByAlias against the DB
``` [2](#0-1) 

**Exploit flow:**

1. Attacker discovers one valid alias (e.g., from public Hedera mirror REST API or prior transactions).
2. Attacker crafts a POST to `/construction/metadata` with `Options["account_aliases"]` = `"<valid_alias>,<valid_alias>,...,<valid_alias>"` repeated N times (e.g., N = 50,000).
3. `resolveAccountAliases()` iterates all N tokens, issuing N synchronous DB queries via `GetAccountId()`.
4. `accountMap` grows to N entries; `strings.Join()` at line 588 allocates a single contiguous string of N × ~82 bytes (alias ~70 chars + `:` + account ID ~10 chars).
5. This string is stored in `metadata[metadataKeyAccountMap]` and serialised into the HTTP response.

**Why existing checks are insufficient:**

- The only validation per alias is format parsing (line 576) and successful DB resolution (line 581). Neither limits count.
- The application server (`main.go` lines 220–227) sets `ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout` but applies **no `http.MaxBytesReader`** body-size cap.
- The Traefik ingress middleware (`values.yaml` lines 152–161) limits in-flight requests to 5 per IP and rate to 10 req/s per host, but these are Kubernetes-deployment-only controls that: (a) are bypassed if the service is accessed directly; (b) do not prevent a single large request from containing tens of thousands of aliases; (c) allow 5 concurrent requests each carrying 50,000 aliases = 250,000 DB queries simultaneously. [3](#0-2) [4](#0-3) 

### Impact Explanation

**Database exhaustion:** N = 50,000 aliases per request × 5 concurrent requests = 250,000 synchronous DB queries, saturating the connection pool and starving all other operations.

**Memory pressure:** Each `accountMap` entry is ~82 bytes; `strings.Join` allocates a single contiguous string. At N = 50,000 per request × 5 concurrent = ~20 MB of live string data. If the baseline RSS is ~50–100 MB (typical for this Go service), this represents a 20–40% increase sustained for the duration of the requests. Repeated at 10 req/s (Traefik rate limit), the GC cannot reclaim fast enough to prevent heap growth above 30%.

**Severity: High.** No authentication is required. The `/construction/metadata` endpoint is publicly reachable. The attack is amplified by the DB query fan-out, not just the string allocation.

### Likelihood Explanation

A valid alias is trivially discoverable: any account that has ever submitted a transaction with an alias is visible via the public mirror node REST API. The attacker needs only one such alias. The attack requires a single HTTP POST with a large body (a 50,000-entry comma-separated string of a ~70-char alias is ~3.5 MB, well within default HTTP client limits and below common reverse-proxy body limits). It is repeatable at the Traefik rate limit of 10 req/s, making sustained exhaustion straightforward. No special tooling beyond `curl` or a simple script is needed.

### Recommendation

1. **Enforce an alias count limit** in `resolveAccountAliases()` before the loop:
   ```go
   const maxAccountAliases = 10  // or whatever the protocol requires
   aliases := strings.Split(accountAliases, ",")
   if len(aliases) > maxAccountAliases {
       return emptyResult, errors.ErrInvalidOptions
   }
   ```
2. **Deduplicate aliases** before processing to prevent amplification from repeated tokens.
3. **Apply `http.MaxBytesReader`** in `main.go` to cap request body size at the application level, independent of any ingress configuration.
4. **Validate that `optionKeyAccountAliases` originates from `ConstructionPreprocess`** (which only ever sets a single alias at line 314), or document and enforce that only one alias is accepted. [5](#0-4)
