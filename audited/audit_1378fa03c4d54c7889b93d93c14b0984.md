### Title
Hex `content-length` Header Causes `requestSizeHistogram` to Record 0 via `parseInt` Radix-10 Truncation

### Summary
In `metricsMiddleware()`, the `content-length` header is parsed with `parseInt(value, 10)`. When an attacker supplies a hex-formatted value such as `0x7FFFFFFF`, JavaScript's `parseInt` with radix 10 stops parsing at the `x` character and returns `0`. The subsequent `|| 0` fallback silently accepts this `0`, causing `requestSizeHistogram` to record a 0-byte request regardless of the actual declared body size. Any unprivileged external user can trigger this with a single crafted HTTP header.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, line 209:

```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** `parseInt('0x7FFFFFFF', 10)` — with an explicit radix of `10` — parses only the leading `'0'` (a valid base-10 digit) and stops at `'x'` (not valid in base 10), returning integer `0`. This is standard, specified ECMAScript behavior. The `|| 0` guard was intended to coerce `NaN` (from completely non-numeric strings) to `0`, but it also silently accepts the `0` produced by the hex truncation, making the two cases indistinguishable.

**Exploit flow:**
1. Attacker sends any HTTP request to any transaction-history endpoint (e.g., `GET /api/v1/transactions`) with the header `content-length: 0x7FFFFFFF`.
2. Express/Node.js passes the raw header string through to `req.headers['content-length']`.
3. `parseInt('0x7FFFFFFF', 10)` → `0`.
4. `0 || 0` → `0`.
5. `requestSizeHistogram.record(0, labels)` is called — a declared ~2 GB request is logged as 0 bytes.

**Why existing checks fail:** The `|| 0` fallback is the only guard and it is the mechanism that masks the bug rather than catching it. There is no validation that the header value is a well-formed decimal integer before parsing.

### Impact Explanation
The `requestSizeHistogram` (metric `api_request_size_bytes`) is the primary signal operators use to detect anomalously large inbound payloads, scraping activity, or abuse of bulk transaction-history endpoints. By forcing every such request to appear as 0 bytes in metrics, an attacker can:
- Conduct sustained high-volume or large-body scraping of transaction history without triggering any size-based alert or dashboard anomaly.
- Skew capacity-planning and SLO data, causing operators to under-provision defenses.
- Render post-incident forensic analysis of request sizes unreliable.

No authentication or privilege is required. The impact is monitoring/observability blindness for request size across all instrumented routes.

### Likelihood Explanation
The precondition is zero: any HTTP client can set an arbitrary `content-length` header. The trigger is a single header value change. The behavior is deterministic and repeatable across every request. Automated scanners or a simple `curl -H 'content-length: 0x7FFFFFFF'` suffice. No rate limiting, authentication, or server-side validation prevents this.

### Recommendation
Replace the `parseInt(..., 10)` call with a strict decimal-only parse and explicit range validation before recording:

```javascript
const rawLen = req.headers['content-length'];
const parsed = rawLen !== undefined ? Number(rawLen) : 0;
const requestBytes = Number.isInteger(parsed) && parsed >= 0 ? parsed : 0;
requestSizeHistogram.record(requestBytes, labels);
```

`Number('0x7FFFFFFF')` returns `2147483647` (correctly), but more importantly a strict `/^\d+$/` guard or `Number.isInteger` + range check will reject non-decimal strings entirely, preventing both the silent-zero and potential negative-value cases.

### Proof of Concept
```bash
# Against any transaction-history endpoint on a running mirror-node REST service

  -H 'content-length: 0x7FFFFFFF' \
  http://<mirror-node-host>/api/v1/transactions

# Then scrape Prometheus metrics
curl -s http://<mirror-node-host>/api/v1/metrics/ | grep api_request_size_bytes

# Expected (correct): bucket counts reflect a ~2 GB declared body
# Actual (vulnerable): the request is counted in the 0-byte bucket,
#   i.e., api_request_size_bytes_bucket{le="0",...} increments by 1
#   while all higher buckets remain unchanged.
```
