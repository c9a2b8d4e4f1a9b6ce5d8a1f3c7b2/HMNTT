### Title
Inverted Timestamp Range Bypasses Inversion Guard, Silently Returns Empty History in `getHookStorageChange()`

### Summary
`Bound.of()` constructs a `Bound` with `primarySortField = false`, which disables the only inversion guard in the `Bound` constructor. An unprivileged user can supply two `timestamp` parameters forming an inverted range (lower > upper), which passes through all validation and is forwarded verbatim to the SQL `BETWEEN` clause, causing it to return zero rows. This allows any caller to falsely assert that no Hashgraph history exists for a given hook storage key in any time period.

### Finding Description

**Step 1 — Controller accepts two timestamp params with no ordering check.**

In `HooksController.getHookStorage()`, the `timestamps` array (max size 2, enforced by `@Size(max = 2)`) is passed to `Bound.of()`: [1](#0-0) [2](#0-1) 

**Step 2 — `Bound.of()` hardcodes `primarySortField = false`, disabling the inversion guard.**

`Bound.of()` always passes `false` as the second argument to the `Bound` constructor: [3](#0-2) 

**Step 3 — The inversion guard in the `Bound` constructor is gated on `primarySortField`.**

The only place where `adjustedLower > adjustedUpper` is checked and rejected is: [4](#0-3) 

Because `primarySortField` is `false` for all timestamp `Bound` objects, this guard is **never reached**.

**Step 4 — The inverted bounds are forwarded directly to SQL.**

In `HookServiceImpl.getHookStorageChange()`, the bounds are extracted and passed to the repository without any additional validation: [5](#0-4) 

**Step 5 — SQL `BETWEEN` with inverted bounds returns zero rows.**

Both repository queries use:
```sql
and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [6](#0-5) [7](#0-6) 

In PostgreSQL, `x BETWEEN a AND b` is equivalent to `x >= a AND x <= b`. When `a > b`, this is always false — zero rows are returned regardless of actual data.

### Impact Explanation
Any unprivileged caller can query any hook's storage history with an inverted timestamp range and receive an empty result. This allows a caller to falsely assert that no storage changes occurred for a given hook key during any time window, effectively suppressing visible Hashgraph history through a crafted read query. While this is a read-only API, the integrity of historical state queries is the core security property of a mirror node — returning provably incorrect empty results for valid historical periods directly undermines that guarantee.

### Likelihood Explanation
The exploit requires zero privileges: the endpoint is a public `GET` request. The inverted range is trivially constructed with two standard `timestamp` query parameters. It is fully repeatable and deterministic. No race conditions, authentication, or special knowledge of internal state is required.

### Recommendation
In `Bound.of()`, change `primarySortField` from `false` to `true`, or add an explicit post-construction inversion check unconditionally:

```java
// In Bound.of() — line 181
return new Bound(timestamp, true, parameterName, field);
```

Alternatively, add an unconditional guard in the `Bound` constructor (removing the `primarySortField &&` condition):

```java
// Bound.java lines 58-60
if (adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
```

This ensures that any inverted timestamp range supplied by a user is rejected with a `400 Bad Request` before reaching the repository layer.

### Proof of Concept

```
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:9999999999&timestamp=lte:1
```

**Preconditions:** Hook `1` owned by account `0.0.123` exists and has storage change records in the database.

**Trigger:** The two `timestamp` parameters produce `timestampLowerBound = 9999999999000000000` (nanoseconds) and `timestampUpperBound = 1000000000` (nanoseconds). Since lower > upper, the SQL `BETWEEN` clause matches no rows.

**Result:** The API returns `{"storage": [], ...}` — an empty list — despite real history existing, with HTTP 200 OK and no error indication.
