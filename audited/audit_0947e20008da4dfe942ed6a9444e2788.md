### Title
Contract Filter Bypass via Entity-Less Recipient Address Causes Unconditional Opcode Logging

### Summary
In `MirrorOperationActionTracer.tracePostExecution()`, the contract filter (`contractFilterCheck()`) is only evaluated when `commonEntityAccessor.get()` returns a non-empty `Optional`. When the recipient address has no corresponding entity in the entity store (e.g., native EVM precompiles at 0x1–0x9, or Hedera system contracts not yet persisted), `recipientNum` is empty, the short-circuit `&&` skips `contractFilterCheck()` entirely, and every opcode in that call is unconditionally logged. An unprivileged user can repeatedly trigger this to amplify log volume beyond operator-configured filter intent.

### Finding Description

**Exact code path:**

`MirrorOperationActionTracer.tracePostExecution()` — lines 59–86:

```java
final var recipientAddress = frame.getRecipientAddress();
final var recipientNum = recipientAddress != null
        ? commonEntityAccessor.get(recipientAddress, Optional.empty())
        : Optional.empty();                                          // ← empty for precompile/unknown addresses

if (recipientNum.isPresent()                                        // ← false → short-circuits
        && traceProperties.contractFilterCheck(
                CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
    return;                                                          // ← never reached
}

log.info("type={} operation={}...", ...);                           // ← always executes
```

**Root cause:** The guard condition uses `recipientNum.isPresent() && contractFilterCheck(...)`. When `commonEntityAccessor.get()` returns `Optional.empty()` — which it does for any address not in the entity store — the `&&` short-circuits and `contractFilterCheck()` is never invoked. The code then falls through unconditionally to `log.info()`.

**`contractFilterCheck()` in `TraceProperties.java` lines 33–35:**
```java
public boolean contractFilterCheck(String contract) {
    return !getContract().isEmpty() && !getContract().contains(contract);
}
```
This is designed to suppress logging for contracts not in the configured set. It is never reached when the entity lookup fails.

**`CommonEntityAccessor.get()` lines 29–36:** For a long-zero address (e.g., `0x0000000000000000000000000000000000000001`), it calls `getEntityByMirrorAddressAndTimestamp()` → `entityRepository.findByIdAndDeletedIsFalse(entityId)`. Native EVM precompiles (entity IDs 1–9) and unregistered system contract addresses will return `Optional.empty()` if not persisted in the mirror node's entity table.

**Failed assumption:** The code assumes that any address worth filtering will always have a resolvable entity. This is false for native precompiles and any system contract address absent from the entity store.

**No test covers this gap:** The existing test suite (`MirrorOperationActionTracerTest`) tests `contractFilterMismatch` (entity present, wrong contract) and `contractFilter` (entity present, correct contract), but has no test for the case where `recipientNum` is empty with a non-empty contract filter configured.

### Impact Explanation

When an operator enables tracing with a contract filter (e.g., `hiero.mirror.web3.evm.trace.enabled=true`, `contract=["0x000...dead"]`), the intent is to limit log output to specific contracts. An attacker bypasses this by directing calls to any address with no entity in the store. Every opcode executed in such a call frame is logged at INFO level. A high-frequency attacker can flood the logging subsystem, consuming disk I/O, storage, and log aggregation pipeline capacity. This is a griefing vector with no economic damage to network users but potential denial-of-observability for operators relying on filtered traces.

### Likelihood Explanation

Preconditions are partially operator-controlled (tracing must be enabled — it defaults to `false`), but once enabled in any production/debug deployment, the attack requires zero privileges: any user who can submit an `eth_call` or `eth_estimateGas` request targeting a precompile address (e.g., `0x0000000000000000000000000000000000000001`) triggers the bypass. The attack is trivially repeatable in a loop. Native EVM precompile addresses (0x1–0x9) are universally valid targets since they are never stored as entities in the mirror node's entity table.

### Recommendation

Change the filter logic to treat an unresolvable entity as a reason to **suppress** logging (fail-closed) when a contract filter is configured:

```java
// Current (fail-open when entity not found):
if (recipientNum.isPresent()
        && traceProperties.contractFilterCheck(...)) {
    return;
}

// Fixed (fail-closed when entity not found and filter is active):
if (!traceProperties.getContract().isEmpty()) {
    if (recipientNum.isEmpty()) {
        return; // unknown address: suppress when filter is active
    }
    if (traceProperties.contractFilterCheck(
            CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
        return;
    }
}
```

Alternatively, fall back to matching the raw recipient address hex against the contract filter set when no entity is found, so operators can explicitly allowlist precompile addresses if desired.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service running with `hiero.mirror.web3.evm.trace.enabled=true`
- `hiero.mirror.web3.evm.trace.contract` set to any non-empty address set (e.g., `["0000000000000000000000000000000000000064"]`)
- Native EVM precompile `0x0000000000000000000000000000000000000001` (ECREC) has no entity row in the mirror node entity table

**Steps:**
1. Submit repeated `eth_call` requests to the web3 JSON-RPC endpoint targeting `0x0000000000000000000000000000000000000001` with valid ECREC input data.
2. Observe that despite the contract filter being configured to only log address `0x64`, every opcode from the precompile call frame is emitted at INFO level.
3. `commonEntityAccessor.get(Address.fromHexString("0x1"), Optional.empty())` returns `Optional.empty()` → `recipientNum.isPresent()` is `false` → `contractFilterCheck()` is skipped → `log.info(...)` executes unconditionally.
4. Repeat at high frequency to amplify log volume arbitrarily.
