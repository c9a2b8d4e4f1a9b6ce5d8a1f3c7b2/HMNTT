### Title
Cache-Poisoning Slot Amplification: Unprivileged Inflation of Batch DB Query Size via `findStorageBatch` Slot Registration

### Summary
An unauthenticated attacker can permanently register up to 1,500 arbitrary slot keys into the per-contract `slotsPerContract` Caffeine cache by issuing `eth_call` requests that trigger SLOAD opcodes for distinct storage slots of a target contract. Once seeded, every subsequent `findStorageBatch` invocation for that contract — which recurs every ≥2 seconds as the `contractState` cache expires — issues a DB batch query carrying all 1,500 registered keys instead of the single key actually needed. This constitutes a sustained, low-effort amplification of database load that persists for the full 5-minute `expireAfterAccess` window and is trivially repeatable.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 59–70) checks the `contractState` cache (TTL: `expireAfterWrite=2s`). On a miss it calls `findStorageBatch()` (line 85–122).

Inside `findStorageBatch()`:

```
Line 86-87: per-contract CaffeineCache obtained/created from outer contractSlotsCache
Line 90:    contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)   ← registers attacker key
Line 91:    cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet()  ← ALL keys
Line 103:   contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)  ← full batch
``` [1](#0-0) 

**Root cause — failed assumption:** The design assumes the per-contract slot registry only grows with *legitimate* repeated accesses to the same contract. It does not bound or validate who may register a key. `putIfAbsent` unconditionally writes the caller-supplied key into the per-contract cache before the batch query is built. The batch query then blindly uses the entire key-set snapshot, so an attacker-inflated registry directly inflates every subsequent DB query.

**Cache configuration:**
- `contractSlots` (outer, contractId → per-contract cache): `expireAfterAccess=5m, maximumSize=3000`
- `slotsPerContract` (inner, per-contract, slotKey → EMPTY_VALUE): `expireAfterAccess=5m, maximumSize=1500`
- `contractState` (slot values): `expireAfterWrite=2s, maximumSize=25000` [2](#0-1) 

**Why `expireAfterAccess` does not self-heal:** Caffeine's `asMap().keySet()` iteration (line 91) does **not** count as an access for `expireAfterAccess` purposes — only explicit `Cache.get()` / `Cache.put()` calls do. The only key whose timer is reset on each legitimate request is the one key passed to `putIfAbsent` (line 90). The 1,499 attacker-seeded keys are only iterated, so their timers run independently from the moment of seeding. They expire 5 minutes after their last explicit access, not after each batch iteration. [3](#0-2) 

**Why the `contractState` cache does not protect:** Its `expireAfterWrite=2s` TTL means that within any 2-second window the batch result is served from cache. But every ≥2 seconds any new request for the target contract misses the `contractState` cache and re-enters `findStorageBatch`, re-issuing the full 1,500-key batch query. [4](#0-3) 

### Impact Explanation

Each `findStorageBatch` call with 1,500 keys issues a single SQL query with a 1,500-element `IN` clause against the `contract_state` table. Compared to the baseline single-key lookup, this query:
- transfers ~1,500× more key data to the DB
- forces the DB to evaluate 1,500 index lookups per call
- returns up to 1,500 rows per call

For any contract receiving sustained traffic (e.g., a popular DeFi contract), the DB receives a 1,500-key batch query every ≥2 seconds per concurrent request thread. With the global 500 RPS limit and a popular contract handling even 50 RPS, this translates to 50 × 1,500 = 75,000 slot lookups per second instead of 50, a >1,000× amplification of DB read load for that contract's queries — far exceeding the 30% threshold. The attack degrades service for all users sharing the DB connection pool.

### Likelihood Explanation

- **No authentication required.** The `/api/v1/contracts/call` endpoint is public.
- **Seeding cost is trivial.** 1,500 requests at the 500 RPS global limit takes 3 seconds. Each request need only call a contract function that reads a mapping/array slot at an attacker-controlled index (e.g., `balanceOf(address)` on any ERC-20).
- **Maintenance cost is low.** To keep the seeded keys alive past 5 minutes, the attacker re-seeds every ~4.5 minutes (5 requests/second sustained, well within rate limits).
- **No per-IP rate limiting exists.** The `rateLimitBucket` is a single global token bucket; there is no per-source throttle.
- **Fully repeatable and automatable** with a simple script. [5](#0-4) [6](#0-5) 

### Recommendation

1. **Cap per-request batch growth:** Before issuing the batch query, limit `cachedSlots` to a configurable maximum (e.g., 50–100 most-recently-accessed keys), discarding the oldest entries rather than using the entire registry.
2. **Separate attacker-controlled keys from trusted keys:** Only add a slot key to the per-contract registry after it has been successfully resolved from the DB at least once (i.e., move `putIfAbsent` to *after* the batch result is processed, and only for keys that returned a value).
3. **Add per-contract slot registration rate limiting:** Reject or ignore `putIfAbsent` if the per-contract cache already holds more than a threshold of keys that were registered within the last N seconds from a single source.
4. **Add per-IP / per-caller rate limiting** at the HTTP layer to prevent a single source from consuming the full global RPS budget during the seeding phase.

### Proof of Concept

**Preconditions:** Target contract `0xTARGET` is deployed and receives legitimate traffic. The mirror node web3 API is accessible. The contract exposes any function that reads `mapping(address => uint256)` storage (e.g., ERC-20 `balanceOf`).

**Step 1 — Seed 1,500 unique slot keys (takes ~3 seconds):**
```bash
for i in $(seq 1 1500); do
  ADDR=$(printf "0x%040x" $i)   # unique address per iteration
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0xTARGET\",\"data\":\"0x70a08231000000000000000000000000${ADDR:2}\",\"gas\":50000}" &
done
wait
```
Each call triggers `SLOAD` for a distinct mapping slot → `findStorageBatch` registers each unique `ByteBuffer` key via `putIfAbsent`.

**Step 2 — Wait 2 seconds** for the `contractState` cache to expire.

**Step 3 — Issue any legitimate request for `0xTARGET`:**
```bash
curl -X POST http://mirror-node/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xTARGET","data":"0x70a08231000000000000000000000000LEGITIMATE_ADDR","gas":50000}'
```
Observe (via DB slow-query log or metrics) that `findStorageBatch` issues a query with 1,500 keys in the `IN` clause instead of 1.

**Step 4 — Repeat Step 3 every 2+ seconds** to confirm the amplified batch query is issued continuously for the full 5-minute window.

**Step 5 — Re-seed every ~4.5 minutes** to maintain the attack indefinitely.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
