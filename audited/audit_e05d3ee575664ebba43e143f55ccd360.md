### Title
Unauthenticated Full-Table-Scan DoS on `GET /contracts/results/logs` via Missing Timestamp Guard When `bindTimestampRange=false`

### Summary
When `bindTimestampRange` is `false` (the documented default), calling `GET /api/v1/contracts/results/logs` with no query parameters causes `optimizeTimestampFilters()` to return an empty filter set, which propagates through `extractContractLogsMultiUnionQuery()` to produce a SQL query against `contract_log` with **no WHERE clause at all**. An unprivileged attacker can flood this endpoint with ~10 concurrent requests to exhaust the DB connection pool (default max 10), blocking all other queries including transaction confirmation.

### Finding Description

**Exact code path:**

`getContractLogs()` (line 820) accepts the request with no filters: [1](#0-0) 

It calls `extractContractLogsMultiUnionQuery(filters)` with `contractId=undefined`. Inside, since `contractId === undefined` and no `transactionHash`, it calls `optimizeTimestampFilters` with an empty array: [2](#0-1) 

`optimizeTimestampFilters([])` calls `parseTimestampFilters` with `filterRequired=false`, which returns `{range: null, eqValues: [], neValues: []}`: [3](#0-2) 

`range?.isEmpty()` on `null` is `undefined` (falsy) — no early return. Since `eqValues.length === 0`, it calls `bindTimestampRange(null, order)`: [4](#0-3) 

With `bindTimestampRange=false` (the default per documentation), this immediately returns `{range: null}`. Back in `optimizeTimestampFilters`, `optimizedRange?.begin` and `optimizedRange?.end` are both falsy — **zero timestamp filters are pushed**. The function returns `{filters: []}`.

The resulting query object has `lower: []`, `inner: []`, `upper: []`, `conditions: []`. `getContractLogsQuery()` detects all three filter arrays are empty and builds a query with **no WHERE clause**: [5](#0-4) 

The generated SQL is:
```sql
with entity as (select evm_address, id from entity)
select cl.bloom, cl.contract_id, cl.consensus_timestamp, ...
from contract_log cl
left join entity e on id = contract_id
order by cl.consensus_timestamp desc, cl.index desc
limit $1
```

This is confirmed by the integration test that passes `conditions: []` and gets a full-table result: [6](#0-5) 

**Root cause:** `optimizeTimestampFilters` has no guard that rejects or substitutes a bounded range when the input is empty and `bindTimestampRange=false`. The assumption that operators will always provide a timestamp filter is not enforced at the API layer for this endpoint.

**Default config confirmation:** [7](#0-6) 

### Impact Explanation
On a production mirror node, `contract_log` contains hundreds of millions of rows. A query with no WHERE clause and `ORDER BY consensus_timestamp DESC, index DESC LIMIT 25` forces PostgreSQL to perform a full sequential scan (or a full index scan of the entire `(consensus_timestamp, index)` index) to return the first 25 rows. Each such query holds a DB connection for up to the 20-second statement timeout. With a default pool of 10 connections, 10 concurrent requests saturate the pool entirely. All other REST API queries — including those that feed transaction confirmation status checks — queue indefinitely or fail. This constitutes a complete REST API denial of service, which at scale prevents clients from confirming new transactions. [8](#0-7) 

### Likelihood Explanation
The attack requires only unauthenticated HTTP GET requests with no parameters — no credentials, no special knowledge, no rate-limit bypass. The endpoint is publicly documented. An attacker needs to send 10 concurrent requests and repeat them every ~20 seconds to maintain the DoS. This is trivially scriptable with `curl` or any HTTP client. The default `bindTimestampRange=false` means most deployments not using the `values-v2.yml` Helm chart are vulnerable out of the box. [9](#0-8) 

### Recommendation

1. **Enforce a mandatory timestamp filter** for `GET /contracts/results/logs` when no `contractId` is present. Add a check in `getContractLogs()` analogous to `checkTimestampsForTopics` that rejects requests with no timestamp filter when `bindTimestampRange=false`.

2. **Change the default** of `bindTimestampRange` to `true`, or make `optimizeTimestampFilters` always apply a bounded range when the input is empty (regardless of the config flag), so that no unbounded query can reach the DB.

3. **Add a guard in `optimizeTimestampFilters`**: if `range` is `null` and `eqValues` is empty after `parseTimestampFilters`, return `{filters: [], skip: true}` to signal the caller to short-circuit with an empty result or a 400 error.

### Proof of Concept

```bash
# Send 10 concurrent unbounded requests to exhaust the DB connection pool
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/contracts/results/logs" &
done
wait

# All subsequent requests will queue/timeout while the above hold all 10 connections
curl -v "http://<mirror-node-host>:5551/api/v1/contracts/results/logs"
# Expected: connection timeout or 503 after pool exhaustion
```

**Verification of no-WHERE-clause query:** Call `GET /api/v1/contracts/results/logs` with no parameters on a node with `bindTimestampRange=false` (default) and enable PostgreSQL `log_min_duration_statement=0`. The logged query will show no `WHERE` clause and a full sequential scan on `contract_log`.

```javascript
const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
```
