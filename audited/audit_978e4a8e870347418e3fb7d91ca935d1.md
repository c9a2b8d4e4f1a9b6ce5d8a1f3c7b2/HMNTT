### Title
Unbounded Orphaned Chunk Accumulation in `topic_message` Table via Incomplete Multi-Chunk HCS Messages

### Summary
`ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()` unconditionally persists every successful chunk transaction into the `topic_message` table with no tracking of whether a multi-chunk message set is ever completed. An attacker can repeatedly submit `chunkNum=1, chunkTotal=2` transactions and never send chunk 2, causing the `topic_message` table to accumulate unbounded orphaned first-chunk rows for a target topic. Because the retention job is disabled by default and no orphan-specific cleanup exists, these rows persist indefinitely and degrade query performance for all mirror node consumers of that topic.

### Finding Description

**Exact code path:**

`importer/src/main/java/org/hiero/mirror/importer/parser/record/transactionhandler/ConsensusSubmitMessageTransactionHandler.java`, `doUpdateTransaction()`, lines 52–93.

The only guard at line 53 is:
```java
if (!entityProperties.getPersist().isTopics() || !recordItem.isSuccessful()) {
    return;
}
```
This rejects failed transactions but accepts every *successful* chunk transaction regardless of whether the full multi-chunk set will ever be completed.

Lines 76–85 then blindly persist `chunkNum` and `chunkTotal` into a new `TopicMessage` row:
```java
if (transactionBody.hasChunkInfo()) {
    ConsensusMessageChunkInfo chunkInfo = transactionBody.getChunkInfo();
    topicMessage.setChunkNum(chunkInfo.getNumber());
    topicMessage.setChunkTotal(chunkInfo.getTotal());
    if (chunkInfo.hasInitialTransactionID()) {
        topicMessage.setInitialTransactionId(...);
    }
}
```
There is no in-memory state machine, no pending-chunk registry, no expiry timer, and no post-insert cleanup job that targets rows where `chunk_num < chunk_total` and no sibling rows with the same `initial_transaction_id` ever arrive.

**Root cause:** The handler treats each chunk as a self-contained, complete record. The assumption that "the application layer will reassemble chunks" is never enforced or cleaned up at the mirror node level.

**Database schema confirms unbounded growth:**
`topic_message` is partitioned by `consensus_timestamp` and indexed on `(topic_id, consensus_timestamp)` and `(topic_id, sequence_number)`. Orphaned chunk-1 rows are valid rows in these indexes; every query for the target topic must scan through them.

**Retention job is disabled by default:**
`RetentionProperties.enabled = false` (line 23 of `RetentionProperties.java`). Even when enabled, `TopicMessageRepository.prune()` deletes by `consensusTimestamp <= ?1` — it prunes *all* old rows by age, not orphaned chunks specifically. With the default 90-day retention period, orphaned rows accumulate for months.

### Impact Explanation

- The `topic_message` table for the targeted topic grows without bound until the (disabled-by-default) retention job ages the rows out after 90 days.
- Every REST API or gRPC query for that topic (`/api/v1/topics/{id}/messages`, `subscribeTopic`) must scan through all orphaned rows in the relevant time partitions, increasing I/O and query latency proportionally to the number of orphaned entries.
- The `topic_message_lookup` table also accumulates corresponding lookup entries via `TopicMessageLookupEntityListener.onTopicMessage()`, compounding the bloat.
- Severity: **Medium** — no funds are at risk, but sustained griefing can make a high-traffic topic's message feed unusably slow for all consumers.

### Likelihood Explanation

- Any Hedera account can create a topic without a `submitKey`, making it open for anyone to submit messages. The attacker creates their own topic and targets it.
- `CONSENSUSSUBMITMESSAGE` costs approximately **1,890,400 tinycents** (~$0.001 USD at typical HBAR prices, confirmed by `FeeEstimationServiceTest` line 75). Submitting 10,000 orphaned chunks costs roughly $10.
- The Hedera network enforces per-account TPS throttling, but an attacker can use multiple accounts or spread submissions over time.
- The attack is fully repeatable and requires no special privileges — only a funded Hedera account.
- The mirror node has no rate-limiting or anomaly detection for incomplete chunk sets.

### Recommendation

1. **Add an orphaned-chunk cleanup job**: Periodically query for rows where `chunk_num IS NOT NULL AND chunk_num < chunk_total` and no sibling row with the same `initial_transaction_id` and `chunk_num = chunk_total` exists within a configurable timeout window (e.g., 24 hours), then delete them.
2. **Track pending chunk sets**: Maintain an in-memory or DB-backed map keyed by `initial_transaction_id` that records expected vs. received chunk numbers. Emit a warning and schedule cleanup when a set is not completed within a timeout.
3. **Enable retention by default** or document clearly that leaving it disabled allows indefinite accumulation of all `topic_message` rows including orphaned chunks.
4. **Add a DB-level partial index** or a separate `topic_message_incomplete` staging table so that incomplete chunk sets do not pollute the main query path until they are confirmed complete.

### Proof of Concept

1. Create a Hedera account with sufficient HBAR balance (e.g., 10 HBAR).
2. Create a topic with no `submitKey` (open topic).
3. In a loop, submit 10,000 `ConsensusSubmitMessage` transactions to that topic, each with `chunkInfo.number=1, chunkInfo.total=2` and a unique `initialTransactionID`. Never submit the corresponding chunk 2.
4. Wait for the mirror node to ingest the record stream.
5. Query `SELECT COUNT(*) FROM topic_message WHERE topic_id = <id> AND chunk_num = 1 AND chunk_total = 2` — observe 10,000 orphaned rows.
6. Benchmark `GET /api/v1/topics/<id>/messages` before and after — observe increased latency proportional to orphaned row count.
7. Confirm `RetentionProperties.enabled` is `false` (default) — rows persist for 90 days with no cleanup.
