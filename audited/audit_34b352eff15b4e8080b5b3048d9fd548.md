### Title
Unauthenticated `/health/readiness` Endpoint Allows Connection Pool Exhaustion via Concurrent PostgreSQL Health Checks

### Summary
The `/health/readiness` endpoint is publicly accessible with no rate limiting or authentication. Each request triggers a live PostgreSQL ping via a separate, unbounded `sql.DB` pool created by `postgres.New()`. Under concurrent flood, this pool can exhaust PostgreSQL's server-side `max_connections`, denying connections to the application's legitimate GORM pool.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `NewHealthController()`, lines 37–43:

```go
readinessChecks := []health.Config{
    {
        Name:      "postgresql",
        Timeout:   time.Second * 10,
        SkipOnErr: false,
        Check:     postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
    },
```

`Routes()` at lines 63–78 registers this handler with no middleware, no authentication, and no rate limiting.

**Root cause — two compounding failures:**

1. `postgres.New(postgres.Config{DSN: ...})` (from `github.com/hellofresh/health-go/v4/checks/postgres`) internally calls `sql.Open()` to create a **separate** `*sql.DB` instance. This pool is entirely independent of the application's GORM pool (configured in `rosetta/app/db/db.go` lines 31–33). Critically, no `SetMaxOpenConns()` is ever called on this health-check pool, so it defaults to **0 (unlimited)**.

2. The `hellofresh/health-go` v4 library does not cache check results by default. Every HTTP request to `/health/readiness` triggers a fresh `db.PingContext(ctx)` call with a 10-second deadline.

**Exploit flow:**

- Attacker sends N concurrent `GET /health/readiness` requests.
- Each request causes the health-check `sql.DB` to acquire a connection from PostgreSQL and call `PingContext`.
- If PostgreSQL is slow or under load, each connection is held for up to 10 seconds (the configured `Timeout`).
- Even under normal conditions, with sufficient concurrency the unbounded pool opens N simultaneous connections.
- PostgreSQL enforces a global `max_connections = 200` (per `rosetta/container/postgresql.conf` line 4 and Helm chart `values.yaml` line 210).
- Once that limit is reached, the application's GORM pool (`rosetta/app/db/db.go` line 33, `SetMaxOpenConns(100)`) cannot acquire new connections, causing all application database queries to fail.

**Why existing checks are insufficient:**

- No rate limiting exists anywhere in the middleware chain for health endpoints (confirmed by `Routes()` returning bare `HandlerFunc` with no wrapping middleware).
- No authentication is required.
- The health-check `sql.DB` has no `MaxOpenConns` constraint.
- The 10-second `Timeout` is a per-check deadline, not a global concurrency guard.

### Impact Explanation

An unprivileged external attacker can cause a complete denial of service for all database-backed API endpoints by flooding `/health/readiness`. PostgreSQL's `max_connections` (200) is a hard server-side limit shared across all pools. Once exhausted, the application's GORM pool cannot open new connections, causing every request requiring a database query to fail. The health-check pool's connections are held for up to 10 seconds per request under degraded conditions, making the attack sustainable with modest concurrency (~200 concurrent requests). This matches the "griefing" classification: no funds are at risk, but service availability is fully disrupted.

### Likelihood Explanation

The endpoint requires no credentials and is reachable by any network-adjacent or internet-facing client. The attack requires only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`) with ~200 concurrent connections. It is trivially repeatable and requires no special knowledge of the application. The only precondition is network access to port 5700 (default Rosetta port).

### Recommendation

1. **Add rate limiting** to the `/health/readiness` handler (e.g., via `golang.org/x/time/rate` or a middleware like `github.com/ulule/limiter`).
2. **Bound the health-check connection pool**: After `postgres.New()` returns its check function, wrap it to reuse the application's existing GORM `*sql.DB` (call `db.PingContext(ctx)` on the shared pool) rather than creating a separate unbounded pool.
3. **Restrict external access** to `/health/liveness` and `/health/readiness` at the network/ingress layer (these endpoints are intended for Kubernetes probes, not public consumers).
4. **Set `MaxOpenConns`** on the health-check `sql.DB` to a small value (e.g., 2) if a separate pool must be kept.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wil/wrk) or equivalent
# Target: Rosetta API default port 5700

wrk -t 50 -c 200 -d 30s http://<rosetta-host>:5700/health/readiness

# Simultaneously, observe PostgreSQL connection count:
psql -c "SELECT count(*) FROM pg_stat_activity;"
# Expected: count approaches max_connections (200)

# Observe application API failures:
curl http://<rosetta-host>:5700/network/list
# Expected: 503 / database connection error once pool is exhausted
