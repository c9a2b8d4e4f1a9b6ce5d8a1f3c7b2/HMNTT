### Title
Unauthenticated Resource Exhaustion via Unbounded Historical Balance Queries in `AccountBalance()`

### Summary
The `AccountBalance()` endpoint in `rosetta/app/services/account_service.go` accepts arbitrary account IDs and historical block identifiers from any unauthenticated user with no application-level rate limiting or result caching. Each request triggers 4 distinct database queries against large tables (`entity`, `entity_history`, `account_balance`, `crypto_transfer`). An attacker flooding the endpoint with many distinct account-block pairs causes a proportional surge in unique DB queries, exhausting database connection pool resources and increasing node resource consumption well beyond 30%.

### Finding Description

**Code path:**

`AccountBalance()` at `rosetta/app/services/account_service.go` lines 41–79 accepts any `AccountBalanceRequest` without authentication or rate limiting. It calls `a.accountRepo.RetrieveBalanceAtBlock(ctx, accountId, block.ConsensusEndNanos)` at line 61.

`RetrieveBalanceAtBlock()` at `rosetta/app/persistence/account.go` lines 161–223 executes **4 sequential DB queries** per call:
1. `getCryptoEntity()` (line 167) — UNION query across `entity` + `entity_history`
2. `getLatestBalanceSnapshot()` (line 192) — first queries `mirror_node_time_partitions` (`selectPreviousPartitionLowerBound`), then a correlated subquery on `account_balance` (`latestBalanceBeforeConsensus`)
3. `getBalanceChange()` (line 201) — aggregate `SUM` over `crypto_transfer` for the time range

**Root cause:** No caching exists in `AccountAPIService` (the `block_service.go` has a cache but `account_service.go` does not). No authentication or rate limiting is applied at the application layer. The middleware stack in `rosetta/main.go` lines 217–219 only wraps `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`.

**Why existing checks fail:**

The Traefik middleware chain (circuit breaker, `inFlightReq: 5`, `rateLimit: average 10`) defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 is **disabled by default**: the template at `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3 requires `{{ if and .Values.global.middleware .Values.middleware -}}`, and `global.middleware` defaults to `false` at line 95. Even when enabled, the rate limit uses `sourceCriterion: requestHost: true` (line 159–160), meaning it limits by hostname rather than client IP — trivially bypassed by distributing requests across multiple source hosts or using direct IP access.

### Impact Explanation

Each `/account/balance` POST with a unique `(accountId, blockIdentifier)` pair bypasses any DB-level query cache and executes 4 expensive queries involving partition lookups, correlated subqueries, and aggregate scans over `crypto_transfer`. With N concurrent connections sending M distinct pairs each, the DB receives 4×N×M unique queries. On a production mirror node with millions of accounts and thousands of historical blocks, an attacker can trivially generate tens of thousands of distinct pairs. This exhausts the DB connection pool, increases CPU and I/O by well over 30%, and degrades or denies service to legitimate users.

### Likelihood Explanation

The endpoint is publicly reachable with no authentication. The Rosetta API spec requires supporting historical balance queries, so the functionality cannot simply be removed. An attacker needs only a list of valid account IDs (publicly available from the mirror node's own `/api/v1/accounts` REST endpoint) and a list of valid block hashes/indices (from `/api/v1/blocks`). The attack is fully scriptable, repeatable, and requires no special privileges or credentials.

### Recommendation

1. **Add application-level rate limiting** in the rosetta Go server (e.g., `golang.org/x/time/rate` per-IP token bucket) before the request reaches `AccountBalance()`.
2. **Enable the Traefik middleware by default** by changing `global.middleware: false` to `true` in `charts/hedera-mirror-rosetta/values.yaml` and switch `sourceCriterion` from `requestHost` to `ipStrategy` for the `rateLimit` middleware.
3. **Add a result cache** in `AccountAPIService` keyed on `(accountId, consensusEndNanos)` with a short TTL (e.g., 30 seconds), similar to the existing cache in `block_service.go`.
4. **Limit `inFlightReq`** at the application layer, not just infrastructure, to bound concurrent DB connections consumed by this endpoint.

### Proof of Concept

```bash
# 1. Collect valid account IDs from the public mirror REST API
curl https://<mirror-node>/api/v1/accounts?limit=1000 | jq '.[].account' > accounts.txt

# 2. Collect valid block indices
curl https://<mirror-node>/api/v1/blocks?limit=1000 | jq '.[].number' > blocks.txt

# 3. Flood /account/balance with distinct (account, block) pairs using parallel requests
while read account; do
  while read block; do
    curl -s -X POST https://<rosetta-node>/account/balance \
      -H 'Content-Type: application/json' \
      -d "{
        \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
        \"account_identifier\": {\"address\": \"$account\"},
        \"block_identifier\": {\"index\": $block}
      }" &
  done < blocks.txt
done < accounts.txt
wait

# Result: DB query rate spikes proportionally; each unique (account, block) pair
# triggers 4 distinct queries with no cache hits, exhausting DB connections
# and increasing node resource consumption by >30%.
```
