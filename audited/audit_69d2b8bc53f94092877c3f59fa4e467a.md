### Title
Unauthenticated `getNodes()` gRPC Endpoint Lacks Rate Limiting, Enabling DB Connection Pool Exhaustion via Uncached Timestamp Queries

### Summary
The `getNodes()` endpoint in `NetworkController.java` is publicly accessible with no authentication, no per-IP rate limiting, and no application-level throttle. Every invocation unconditionally executes two uncached SQL `MAX()` aggregate queries against the database (`addressBookRepository.findLatestTimestamp()` and `nodeStakeRepository.findLatestTimestamp()`). An attacker opening many parallel gRPC connections and flooding the endpoint can exhaust the database connection pool, starving all other mirror node operations including gossip-related data processing.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any unauthenticated request and immediately delegates to `networkService.getNodes()`: [1](#0-0) 

`NetworkServiceImpl.getNodes()` (lines 61–64) then executes two uncached database queries on every single call: [2](#0-1) 

The underlying queries are raw SQL aggregates with no result caching:

- `AddressBookRepository.findLatestTimestamp()`: `select max(start_consensus_timestamp) from address_book where file_id = ?` [3](#0-2) 

- `NodeStakeRepository.findLatestTimestamp()`: `select max(consensus_timestamp) from node_stake` [4](#0-3) 

Note: `findAllStakeByConsensusTimestamp` is `@Cacheable`, but `findLatestTimestamp()` is **not**, so it hits the DB on every request. [5](#0-4) 

**Root cause:** The `getNodes()` handler has no authentication gate, no per-IP or global request rate limiter, and no circuit breaker. The two timestamp queries are not cached, so each request unconditionally consumes at least one database connection for the duration of both queries.

**Why existing checks fail:**

1. `maxConcurrentCallsPerConnection = 5` (default in `NettyProperties`) only limits concurrency *per TCP connection*. An attacker opens N connections and gets 5N concurrent in-flight requests. [6](#0-5) 

2. The GCP gateway `maxRatePerEndpoint: 250` is an optional infrastructure-layer Helm value, not enforced at the application level, and is absent in non-GCP or bare-metal deployments. [7](#0-6) 

3. No `ServerInterceptor`, Spring Security filter, or throttle bean exists in the gRPC module for this endpoint (unlike the `web3` module which has `ThrottleManagerImpl` with `Bucket4j`). [8](#0-7) 

### Impact Explanation
HikariCP's default pool size is 10 connections. With N attacker connections × 5 concurrent calls each, the pool is exhausted in milliseconds. All other database-dependent operations — including ingestion of gossip transactions, topic message retrieval, and address book updates — block waiting for a connection. This constitutes a full denial-of-service against the mirror node's core data pipeline. Severity: **High**.

### Likelihood Explanation
The endpoint is publicly documented and trivially invocable with `grpcurl` (no credentials, no token, no API key). The attacker needs only a standard gRPC client and a network path to port 5600. The attack is fully repeatable, requires no special knowledge, and can be automated with a simple loop. Deployments without the GCP gateway (e.g., self-hosted, testnet, or development nodes) have zero infrastructure-level mitigation.

### Recommendation
1. **Cache `findLatestTimestamp()` results** in both `AddressBookRepository` and `NodeStakeRepository` using `@Cacheable` with a short TTL (e.g., 5–10 seconds), matching the pattern already used for `findAllStakeByConsensusTimestamp`.
2. **Add application-level rate limiting** to the gRPC module using a `ServerInterceptor` backed by Bucket4j (mirroring `ThrottleManagerImpl` in the `web3` module), enforcing a global and/or per-IP request rate cap on `getNodes()`.
3. **Enforce `maxRatePerEndpoint`** at the infrastructure layer unconditionally, not as an optional Helm value.
4. **Set an explicit, conservative HikariCP pool size** in the gRPC module's configuration and document it as a tuning parameter.

### Proof of Concept
```bash
# Open 20 parallel gRPC connections, each sending 5 concurrent getNodes() calls (100 in-flight DB queries)
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes &
  done
done
wait
# Observe: subsequent legitimate gRPC calls and gossip ingestion hang or time out
# DB logs show connection pool exhaustion / queue timeout errors
```
