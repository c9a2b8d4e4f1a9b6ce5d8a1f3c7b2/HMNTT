### Title
Unbounded `CaffeineCacheManager` Internal Registry Leak via Distinct `contractId` Queries Enables Unauthenticated Memory Exhaustion DoS

### Summary
In `ContractStateServiceImpl.findStorageBatch()`, every distinct `contractId` causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to register a new named `CaffeineCache` in the `CaffeineCacheManager`'s internal `ConcurrentHashMap`. This internal registry has no size limit and is never pruned, even when the outer `contractSlotsCache` (bounded to 3,000 entries) evicts the corresponding entry. An unauthenticated attacker can exhaust JVM heap memory by issuing public JSON-RPC calls with millions of distinct contract addresses, causing an OOM crash and complete service denial.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, `findStorageBatch()`, lines 86–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
    contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two independent bounds that do not compose:**

The outer cache `contractSlotsCache` is configured via `CACHE_MANAGER_CONTRACT_SLOTS` with `expireAfterAccess=5m,maximumSize=3000`. This bounds the number of live `contractId → sub-cache` mappings held in the outer Caffeine cache.

However, the value loader calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Spring's `CaffeineCacheManager.getCache()` stores every created cache in its own internal `ConcurrentHashMap<String, Cache>` (the `cacheMap` field). This map has **no eviction policy and no size limit**. It is a plain `ConcurrentHashMap` that accumulates entries for the lifetime of the JVM process.

The `cacheManagerSlotsPerContract` bean is declared without `setCacheNames()`, meaning it operates in dynamic mode — it creates and registers a new named cache for every unique string key passed to `getCache()`:

```java
// EvmConfiguration.java lines 107–112
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;
}
```

The `maximumSize=1500` in `slotsPerContract` config applies only to the number of *slot-key entries within each sub-cache*, not to the number of sub-caches registered in the manager.

**Lifecycle mismatch — the core bug:**

When the outer `contractSlotsCache` evicts an entry for `contractId_X` (due to its `maximumSize=3000` limit), the sub-cache object for `contractId_X` is removed from the outer cache's data structure. But `cacheManagerSlotsPerContract`'s internal `cacheMap` still holds a strong reference to that sub-cache under the key `contractId_X.toString()`. It is never removed. The sub-cache object — a full `BoundedLocalCache` with internal arrays, scheduler references, and metadata — remains in heap indefinitely.

**Exploit flow:**

1. Attacker sends `eth_getStorageAt(address_N, slot, "latest")` for N = 1, 2, 3, … with distinct addresses.
2. Each call reaches `ContractStorageReadableKVState.readFromDataSource()` → `contractStateService.findStorage()` → `findStorageBatch()`.
3. For each new `contractId`, the outer cache misses and the value loader fires `cacheManagerSlotsPerContract.getCache(contractId.toString())`.
4. A new `CaffeineCache` is created and inserted into `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap`.
5. After 3,001 distinct contractIds, the outer cache begins evicting old entries. The evicted sub-caches remain in `cacheManagerSlotsPerContract.cacheMap`.
6. After N total distinct contractIds, `cacheManagerSlotsPerContract.cacheMap` holds N entries. Memory grows without bound.

**Why existing checks are insufficient:**

- The outer cache `maximumSize=3000` only limits the outer cache's live entries; it has no effect on `cacheManagerSlotsPerContract`'s internal registry.
- The `expireAfterAccess=5m` on the outer cache causes eviction of old entries but does not trigger any cleanup in `cacheManagerSlotsPerContract`.
- The rate limit of `requestsPerSecond=500` slows the attack but does not prevent it; at 500 RPS, an attacker creates 1.8 million new sub-caches per hour.
- `enableBatchContractSlotCaching=true` is the default, so the vulnerable path is active by default.
- No authentication is required for `eth_getStorageAt` or `eth_call`.

### Impact Explanation

Each `CaffeineCache` / `BoundedLocalCache` object carries non-trivial heap overhead (internal segment arrays, timer wheel, reference queues, etc.) — conservatively 2–10 KB per instance even when empty. At 1 million sub-caches this is 2–10 GB of heap. The JVM will throw `OutOfMemoryError`, crashing the web3 service. This prevents all contract call processing, including gossip-related transaction simulation, effectively constituting a complete denial of service. The attack is permanent until the process is restarted, and can be repeated immediately after restart.

### Likelihood Explanation

The attack requires only the ability to send unauthenticated HTTP JSON-RPC requests — no private keys, no tokens, no on-chain transactions. The attacker needs only a list of distinct Ethereum-format addresses (trivially generated). The rate limit of 500 RPS is a throttle, not a barrier; a single attacker machine can sustain this rate. The attack is fully repeatable and automatable with a simple script. The `enableBatchContractSlotCaching` flag is `true` by default, so no special configuration is needed to trigger the vulnerable path.

### Recommendation

1. **Bound the sub-cache registry directly.** Replace the `CaffeineCacheManager` for `slotsPerContract` with a custom `LoadingCache<EntityId, Cache>` (a Caffeine cache of caches) that has its own `maximumSize` matching the outer cache's limit (e.g., 3,000). This ensures sub-cache objects are evicted from the registry when the outer cache evicts them.

2. **Register an eviction listener on the outer cache** that calls `cacheManagerSlotsPerContract`'s internal map removal (or a custom invalidation method) when a `contractId` entry is evicted, so the sub-cache is dereferenced from the manager.

3. **Alternatively**, eliminate the two-layer design entirely: use a single `Cache<SimpleKey(contractId, slotKey), byte[]>` with a global `maximumSize` covering all contracts and slots, removing the per-contract sub-cache indirection.

4. **Short-term mitigation**: set `hiero.mirror.web3.cache.enableBatchContractSlotCaching=false` to disable the vulnerable code path until a fix is deployed.

### Proof of Concept

```python
import requests, itertools, threading

URL = "http://<mirror-node-host>:8545"

def make_address(n):
    return "0x" + hex(n)[2:].zfill(40)

def attack(start, count):
    for i in range(start, start + count):
        addr = make_address(i)
        payload = {
            "jsonrpc": "2.0", "method": "eth_getStorageAt",
            "params": [addr, "0x0", "latest"], "id": i
        }
        try:
            requests.post(URL, json=payload, timeout=5)
        except:
            pass

# Launch 10 threads, each querying 100,000 distinct addresses
threads = [threading.Thread(target=attack, args=(i*100000, 100000)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()
# After ~1M requests, cacheManagerSlotsPerContract.cacheMap holds ~1M CaffeineCache objects
# Monitor JVM heap: watch for OOM / service crash
```

Each iteration registers a new `CaffeineCache` in `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap`. After sufficient iterations the JVM heap is exhausted and the service crashes with `OutOfMemoryError`.
