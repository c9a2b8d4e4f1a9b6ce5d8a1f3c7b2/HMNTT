### Title
GraphQL Alias Batching Bypasses Per-Request Account Limit, Enabling Bulk Staking Reward Enumeration

### Summary
The GraphQL configuration in `GraphQlConfiguration.java` sets a `maxTokens(1000)` parser limit and a `MaxQueryComplexityInstrumentation(200)` execution limit, but neither prevents an unauthenticated attacker from using GraphQL field aliases to batch approximately 52 separate `account` queries — each resolving `pendingReward` — within a single HTTP request. No alias count limit (`MaxAliasesInstrumentation`) is configured, and the default complexity calculator assigns a flat weight of 1 per field regardless of whether it is an expensive root resolver or a cheap leaf, making the complexity ceiling ineffective against alias-based batching.

### Finding Description

**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `static {}` block (lines 31–39) and `graphQlCustomizer` bean (lines 42–49).

**Root cause:**
The static initializer sets `maxTokens(1000)` on both default and operation parser options:
```java
Consumer<Builder> consumer =
    b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
```
The `graphQlCustomizer` bean adds only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`:
```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
```
No `MaxAliasesInstrumentation` is registered anywhere in the GraphQL module (confirmed: zero matches for alias-limiting instrumentation across all `graphql/**/*.java` files).

**Exploit flow using GraphQL aliases:**
GraphQL allows a client to alias the same root field multiple times with different arguments in one document:
```graphql
{
  a1: account(input: {entityId: {num: 1}}) { pendingReward }
  a2: account(input: {entityId: {num: 2}}) { pendingReward }
  a3: account(input: {entityId: {num: 3}}) { pendingReward }
  ...
}
```
Token count per alias entry (`a1 : account ( input : { entityId : { num : 1 } } ) { pendingReward }`) is approximately 19 tokens. With 2 tokens for the outer braces, the 1000-token ceiling allows `(1000 − 2) / 19 ≈ 52` aliased account lookups per request. Complexity per alias is 2 (one for the `account` root field, one for `pendingReward`), so 52 aliases consume only 104 complexity units — well under the 200 limit. Depth remains 2, far under the limit of 10.

**Why existing checks fail:**
- `maxTokens(1000)` limits document size but still permits ~52 aliases.
- `MaxQueryComplexityInstrumentation(200)` uses the default flat-weight calculator (1 per field), which does not distinguish between a root resolver that issues a database query and a trivial leaf field. The ceiling of 200 is never approached.
- `MaxQueryDepthInstrumentation(10)` is irrelevant; aliases are breadth, not depth.
- No application-level rate limiting exists in the `graphql` module (the `ThrottleConfiguration`/`ThrottleManagerImpl` classes are in the `web3` module only). The Helm chart's `inFlightReq: amount: 5` is an infrastructure-level concurrent-connection cap, not a per-request alias cap.
- No authentication is required to reach `/graphql/alpha`.

### Impact Explanation
Each aliased `account` field triggers an independent database query against the `entity` and `entity_stake` tables to resolve `pendingReward`. An attacker can enumerate staking reward balances for ~52 accounts per HTTP request with no credentials, multiplied across as many concurrent requests as the infrastructure allows (the Helm chart permits 5 in-flight per IP, giving 260 account lookups per round-trip per IP). This enables systematic bulk harvesting of `pendingReward` values across the entire account space, and simultaneously amplifies database load by a factor of ~52 per request compared to the intended single-account design of the `account(input: AccountInput!): Account` query.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only standard GraphQL knowledge. The alias technique is well-documented and trivially scriptable. The endpoint is publicly reachable at `/graphql/alpha`. Repeatability is limited only by network bandwidth and the infrastructure-level in-flight cap, both of which are easily worked around by distributing requests across IPs or time.

### Recommendation
1. **Add `MaxAliasesInstrumentation`** (available in `graphql-java` ≥ 20) with a limit of 1 or a small number (e.g., 5) to the `ChainedInstrumentation` in `graphQlCustomizer`.
2. **Use a weighted complexity calculator** that assigns a higher cost (e.g., 10–20) to root-level resolver fields (`account`) versus leaf scalar fields, so the complexity ceiling of 200 becomes a meaningful constraint against alias batching.
3. **Add application-level rate limiting** to the `graphql` module (analogous to `ThrottleConfiguration` in `web3`) keyed per IP or per authenticated identity.

### Proof of Concept
```bash
# Build a query with 52 aliased account lookups, each requesting pendingReward
python3 -c "
aliases = '\n'.join(
    f'  a{i}: account(input: {{entityId: {{num: {i}}}}}) {{ pendingReward }}'
    for i in range(1, 53)
)
print('{' + aliases + '}')
" > query.graphql

curl -s -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"$(cat query.graphql | tr '\n' ' ')\"}" | jq .
```
The response will contain `pendingReward` values for up to 52 distinct accounts in a single unauthenticated request, with no error from the parser or complexity instrumentation.
