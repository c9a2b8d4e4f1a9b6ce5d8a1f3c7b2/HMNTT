### Title
Unbounded Goroutine Accumulation via Unauthenticated `/health/readiness` Endpoint Causes Memory Exhaustion DoS

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is publicly accessible with no rate limiting or concurrency control at the application level. Each request spawns goroutines that hold for up to 10 seconds (two checks running concurrently, each with a 10-second timeout), and the DB connection pool can be exhausted simultaneously. The Traefik middleware that could mitigate this is disabled by default and does not cover the `/health/readiness` path even when enabled.

### Finding Description

**Exact code path:**

In `rosetta/app/middleware/health.go`, `Routes()` registers the readiness handler with no concurrency guard: [1](#0-0) 

Two checks are registered, each with a 10-second timeout: [2](#0-1) 

The `hellofresh/health-go/v4` library runs all checks concurrently (each in its own goroutine), so per incoming request the library spawns 2 check goroutines, each capable of blocking for up to 10 seconds. The handler goroutine itself also blocks waiting for results.

The HTTP server is configured with `WriteTimeout: 10,000,000,000 ns` (10 seconds): [3](#0-2) [4](#0-3) 

This means goroutines are held for up to 10 seconds per request before the server forcibly closes the connection. With no concurrency limit, N simultaneous requests produce 3N goroutines (1 handler + 2 check goroutines each).

**Root cause — failed assumption:** The code assumes an external rate-limiting layer (Traefik) will protect this endpoint. However:

1. `global.middleware` defaults to `false`, so the Traefik middleware chain is **disabled by default**: [5](#0-4) 

2. Even when enabled, the ingress only covers `/rosetta/*` paths — `/health/readiness` is **not in the ingress path list** and therefore never passes through the `inFlightReq` (limit: 5) or `rateLimit` (average: 10) middleware: [6](#0-5) [7](#0-6) 

**DB connection pool exhaustion (secondary impact):** The postgresql check opens a real DB connection. The pool is capped at 100 open connections: [8](#0-7) 

With >100 concurrent readiness requests, goroutines queue waiting for a DB connection, extending their lifetime beyond 10 seconds and amplifying goroutine accumulation.

### Impact Explanation
An attacker sending a sustained flood of GET `/health/readiness` requests causes:
1. **Memory exhaustion** — unbounded goroutine stack growth (Go has no built-in goroutine limit; each goroutine starts at ~8 KB and grows). At 1,000 req/s, ~10,000 goroutines accumulate within 10 seconds (~80 MB minimum, growing with stack expansion).
2. **DB connection pool starvation** — all 100 DB connections consumed, blocking legitimate API requests that also need DB access.
3. **Process crash or OOM kill** — the Kubernetes pod is OOM-killed, causing a service outage for the Rosetta API, which is a critical infrastructure component for Hedera/Hiero blockchain operations.

### Likelihood Explanation
- **No authentication required** — any internet-accessible client can reach port 5700 directly or via any non-Traefik ingress.
- **Default configuration is vulnerable** — `global.middleware: false` means most deployments lack the only intended mitigation.
- **Trivially reproducible** — a simple `ab -n 100000 -c 5000 http://target:5700/health/readiness` or equivalent is sufficient.
- **Repeatability** — the attack can be sustained indefinitely; each wave of requests restarts the goroutine accumulation cycle.

### Recommendation
1. **Application-level concurrency limit**: Add a semaphore or `sync.Mutex`-guarded counter in the readiness handler to cap concurrent health checks (e.g., max 10 in-flight).
2. **Shared/cached health result**: Cache the readiness result for 1–5 seconds so concurrent requests return the cached value without spawning new check goroutines.
3. **Fix ingress path coverage**: Add `/health/readiness` and `/health/liveness` to the ingress path list so the Traefik `inFlightReq` and `rateLimit` middleware applies to them.
4. **Enable middleware by default**: Change `global.middleware` default to `true` or document the security risk of leaving it disabled.

### Proof of Concept
```bash
# Prerequisites: rosetta running on port 5700, default config (global.middleware=false)
# Step 1: Send a burst of 10,000 concurrent readiness requests
ab -n 10000 -c 2000 http://<rosetta-host>:5700/health/readiness

# Step 2: Observe goroutine count via pprof (if enabled) or /metrics
curl http://<rosetta-host>:5700/metrics | grep inflight

# Step 3: Monitor memory
# kubectl top pod <rosetta-pod> -n <namespace>
# Expected: memory grows rapidly; pod is OOM-killed or becomes unresponsive
# DB connections exhausted: legitimate /network/status calls begin returning errors

# Sustained attack (keeps goroutines accumulating across 10s windows):
while true; do
  ab -n 5000 -c 1000 http://<rosetta-host>:5700/health/readiness &
  sleep 5
done
```
