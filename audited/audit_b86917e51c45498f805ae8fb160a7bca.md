### Title
Unbounded Alias-Resolution DB Queries in `getAccountTokenAllowances` Enable Unauthenticated DoS via Database Saturation

### Summary
`TokenAllowanceController.getAccountTokenAllowances()` calls `EntityService.getEncodedId()` on every inbound request with no in-service caching. When the path parameter is an account alias or EVM address, this unconditionally issues a live database query. Because the REST service has no application-level rate limiting and the optional Redis response cache is trivially bypassed by varying query parameters, an unauthenticated attacker can flood the database with identical alias-resolution queries, exhausting the connection pool and degrading or blocking the importer's ability to persist incoming gossip transactions.

### Finding Description

**Exact code path:**

`tokenAllowanceController.js` line 69 calls `EntityService.getEncodedId()` unconditionally on every request: [1](#0-0) 

`EntityService.getEncodedId()` (lines 118–137) has two branches that hit the database — alias resolution and EVM-address resolution — with **zero caching**: [2](#0-1) 

The alias branch calls `getAccountFromAlias()`, which executes `entityFromAliasQuery` directly against the `entity` table on every invocation: [3](#0-2) [4](#0-3) 

**Why the response cache does not protect this path:**

The Redis response cache (`responseCacheCheckHandler`) is gated on both `config.cache.response.enabled` **and** `config.redis.enabled`: [5](#0-4) [6](#0-5) 

Even when both flags are enabled, the cache key is the MD5 of `req.originalUrl`: [7](#0-6) 

An attacker trivially bypasses this by cycling any accepted query parameter (e.g., `?limit=25`, `?limit=24`, `?order=desc`, `?spender.id=0.0.X`) while keeping the same alias in the path. Each distinct URL is a cache miss, and the default TTL is only 1 second: [8](#0-7) 

**No application-level rate limiting exists** in the REST JS service. A `grep` across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, or `throttle` returns zero matches. The Traefik middleware YAML is optional infrastructure configuration, not enforced at the application layer: [9](#0-8) 

**Contrast with the Java importer**, which correctly wraps all alias lookups in a `Cache` and uses `cacheLookup()` to avoid repeated DB hits: [10](#0-9) 

The JS REST `EntityService` has no equivalent mechanism.

### Impact Explanation
The shared PostgreSQL instance serves both the REST API and the importer. Flooding the alias-resolution query saturates the database connection pool and query throughput. The importer, which must write every gossip transaction to the database, competes for the same resources. Under sustained attack, importer write latency increases, gossip transactions queue up, and the mirror node's view of network state falls behind or stalls entirely. This is a denial-of-service against the mirror node's core function of reflecting the Hedera network's transaction history. Severity: **High** — complete loss of mirror node liveness is achievable without any credentials.

### Likelihood Explanation
The attack requires no authentication, no special knowledge beyond a valid account alias (publicly discoverable via the same API), and no sophisticated tooling — a simple concurrent HTTP flood with rotating query parameters suffices. The endpoint is publicly reachable. The absence of application-level rate limiting means there is no server-side throttle to overcome. The attack is fully repeatable and automatable.

### Recommendation
1. **Add in-process caching to `EntityService.getEncodedId()`**: Cache alias→encodedId and evmAddress→encodedId mappings with a short TTL (e.g., 5–30 seconds) using an LRU cache (e.g., `lru-cache`). This mirrors the pattern already used in the Java `EntityIdServiceImpl`.
2. **Add application-level rate limiting**: Integrate `express-rate-limit` (or equivalent) scoped to the `/accounts/:id/allowances/*` route family, keyed by IP or authenticated identity.
3. **Make the Redis response cache non-optional** for production deployments, and document that disabling it removes a critical DoS mitigation.
4. **Normalize query parameters** before generating cache keys so that semantically equivalent requests (e.g., `?limit=25` vs default) share a cache entry.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly accessible; a valid account alias `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ` exists; Redis response cache is disabled (default) or bypassed.

**Steps:**
```bash
# Generate 10,000 concurrent requests with rotating query params to bypass URL cache
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ/allowances/tokens?limit=$((RANDOM % 25 + 1))" &
done
wait
```

**What happens:**
1. Each request reaches `getAccountTokenAllowances()`.
2. `EntityService.getEncodedId("AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ")` is called — `AccountAlias.isValid()` returns `true`.
3. `getAccountIdFromAlias()` → `getAccountFromAlias()` → `entityFromAliasQuery` fires against the DB.
4. Varying `limit` values produce distinct `req.originalUrl` values → all cache misses.
5. The DB connection pool is exhausted; importer write queries time out; gossip transaction persistence stalls.
