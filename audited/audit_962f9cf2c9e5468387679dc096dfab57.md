### Title
DB Connection Pool Starvation via Unauthenticated Concurrent Requests to `getTransactionDetailsFromTransactionId()`

### Summary
The REST API exposes `/api/v1/transactions/:transactionId` with no per-IP rate limiting or in-flight request cap. Each request triggers `getTransactionDetailsFromTransactionId()`, which issues a parameterized query against the `transaction` table. With the default DB pool capped at 10 connections and a 20-second statement timeout, a single unauthenticated attacker sending ≥10 concurrent requests can monopolize the entire pool, starving all other API consumers until their connection-acquisition attempts time out.

### Finding Description
**Code path:**
- `rest/server.js` line 133: `app.getExt(\`${apiPrefix}/transactions/:transactionIdOrHash\`, transactions.getTransactionsByIdOrHash)` — no rate-limit middleware applied before this route.
- `rest/service/transactionService.js` lines 64–72: `getTransactionDetailsFromTransactionId()` calls `this.getTransactionDetails(...)` → `super.getRows(query, params)` → `this.pool().queryQuietly(query, params)`, issuing one DB query per request.
- `rest/dbpool.js` lines 7–16: pool is configured with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20,000 ms**).

**Root cause — missing per-IP throttle:**
The Express middleware stack in `rest/server.js` (lines 68–144) contains: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. None of these enforce a per-source-IP request rate or in-flight concurrency cap.

The `authHandler` (`rest/middleware/authHandler.js`) only sets a custom *response result limit* for authenticated users via `httpContext`; it does not reject or throttle unauthenticated callers.

The Traefik ingress middleware for the REST API (`charts/hedera-mirror-rest/values.yaml` lines 134–139) is:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 10
      initialInterval: 100ms
```
There is **no** `rateLimit` or `inFlightReq` entry — in direct contrast to the Rosetta API (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160) which explicitly configures both:
```yaml
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true
```

The `retry: attempts: 10` Traefik middleware actually amplifies the attack: each attacker request that times out is retried up to 10 times, multiplying pool pressure by up to 10×.

**Why the pool is the bottleneck:**
With `maxConnections = 10` and `statementTimeout = 20 s`, an attacker holding 10 concurrent connections blocks the pool for up to 20 seconds per wave. Legitimate requests queue for a connection and fail after `connectionTimeoutMillis = 20,000 ms`, returning errors to all other consumers.

### Impact Explanation
All REST API endpoints share the same global `pool` object (`rest/service/baseService.js` line 96–98). Pool exhaustion caused by the transaction endpoint degrades every other endpoint (accounts, tokens, balances, contracts, etc.) simultaneously. Affected users receive connection-timeout errors (HTTP 500/503) for the duration of the attack. No funds are at risk, but API availability — the primary function of the mirror node REST service — is fully disrupted. This matches the "griefing / availability impact with no economic damage" severity classification.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. Any internet-accessible deployment is reachable. The attacker needs only a list of valid `transactionId` values (trivially obtained from the public ledger via the same API) and a tool capable of issuing concurrent HTTP requests (e.g., `ab`, `wrk`, `curl` in parallel). The attack is repeatable indefinitely and requires no special knowledge beyond the public API spec.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` Traefik middleware to the REST chart** (mirroring the Rosetta configuration) to enforce per-source-IP concurrency and rate caps at the ingress layer.
2. **Add an application-level concurrency limiter** (e.g., `express-rate-limit` or a semaphore) specifically for DB-bound routes, so pool exhaustion cannot be triggered even if Traefik is misconfigured or bypassed.
3. **Reduce `statementTimeout`** for the transaction lookup query to a value well below 20 s (e.g., 3–5 s) to shorten the window each connection is held under attack.
4. **Remove or cap the Traefik `retry` count** for this service, as retrying timed-out DB queries amplifies pool pressure rather than recovering from transient errors.

### Proof of Concept
```bash
# 1. Collect 20 valid transactionIds from the public API
IDS=$(curl -s "https://<mirror-node>/api/v1/transactions?limit=20" \
  | jq -r '.transactions[].transaction_id')

# 2. Fire 20 concurrent requests (exceeds default pool of 10)
echo "$IDS" | xargs -P 20 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node>/api/v1/transactions/{}"

# 3. While step 2 runs, observe that legitimate requests to other endpoints
#    (e.g., /api/v1/accounts/0.0.1) return 500 or hang until connectionTimeout (20 s).
#    With retry:attempts=10, each of the 20 attacker requests generates up to 10
#    DB connection attempts, sustaining pool exhaustion across multiple waves.
```
