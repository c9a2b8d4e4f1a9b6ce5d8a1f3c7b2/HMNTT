### Title
Unauthenticated Cache-Miss Thundering Herd via Inline-Literal Query Flooding Exhausts DB Connection Pool

### Summary
The `CachedPreparsedDocumentProvider` keys its Caffeine cache on the raw GraphQL query string. Because inline literal values (e.g., `{num: 1}`, `{num: 2}`) produce distinct cache keys, an unprivileged attacker can trivially fill the 1000-entry cache and then flood the service with novel unique queries, each triggering a concurrent parse-and-DB-execute cycle. The GraphQL module has no rate limiting (unlike the web3 module), so all concurrent requests reach the HikariCP connection pool simultaneously, exhausting it and partitioning the service from its database.

### Finding Description

**Cache key is the raw query string:**

`CachedPreparsedDocumentProvider.java` line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

`executionInput.getQuery()` is the verbatim query string. The queries:
- `{ account(input: {entityId: {num: 1}}) { id } }`
- `{ account(input: {entityId: {num: 2}}) { id } }`

are two distinct cache keys. Variables would share a key; inline literals do not.

**Cache configuration — maximumSize=1000, expireAfterWrite=1h:** [2](#0-1) 

**No rate limiting on the GraphQL module.** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) lives exclusively in the `web3` module. The `graphql` module has no equivalent — no `requestsPerSecond` cap, no IP-based throttle, no authentication gate. [3](#0-2) 

**Endpoint is publicly accessible** at `/graphql/alpha` with no authentication: [4](#0-3) 

**DB connection pool** is HikariCP configured via `spring.datasource.hikari` with no explicit `maximumPoolSize` override in the graphql module (HikariCP default: 10 connections): [5](#0-4) 

**Every cache miss triggers both parsing and a DB query.** After `parseAndValidateFunction.apply(executionInput)` returns the parsed AST, GraphQL-Java proceeds to execute the query, which calls `EntityRepository.findById()` — a synchronous JDBC call consuming a HikariCP connection: [6](#0-5) 

**Exploit flow:**

1. **Phase 1 — cache fill:** Send 1000 requests with `num: 1` through `num: 1000`. Cache is now at capacity (`maximumSize=1000`).
2. **Phase 2 — thundering herd:** Concurrently send N requests with `num: 1001` through `num: 1000+N`. Every request is a cache miss. Each miss triggers:
   - GraphQL parse + validate (CPU)
   - `EntityRepository.findById()` (DB connection from HikariCP pool)
3. With N >> pool size (default 10), all connections are consumed. Legitimate requests queue indefinitely until `connectionTimeout` fires, returning errors.
4. The attack is repeatable continuously — new `num` values are unbounded.

**Existing mitigations are insufficient:**

- `maxCharacters=10000`, `maxTokens=1000` (parser limits) prevent oversized individual queries but do not limit concurrency or request rate. [7](#0-6) 
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` limit per-query cost but not request volume. [8](#0-7) 
- The `account` query schema requires no authentication: [9](#0-8) 

### Impact Explanation
An unauthenticated attacker can exhaust the HikariCP connection pool, causing all subsequent GraphQL requests to fail with connection-timeout errors. This constitutes a full denial-of-service for the GraphQL API and, if the pool is shared, for any other component using the same datasource. The DB statement timeout of 10 seconds means each connection is held for up to 10 seconds per attack request, amplifying the pool exhaustion window. [10](#0-9) 

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a basic HTTP client capable of sending concurrent POST requests. The `num` field accepts any `Long ≥ 0`, providing ~9.2×10¹⁸ unique cache keys. The attack is trivially scriptable, repeatable every hour (after `expireAfterWrite` expiry) or continuously by cycling through new `num` values. No special knowledge of the system is required beyond the publicly documented GraphQL schema.

### Recommendation

1. **Add rate limiting to the GraphQL module** mirroring the web3 `ThrottleConfiguration` — implement a `requestsPerSecond` cap (e.g., 100 RPS) using Bucket4j or Spring's built-in mechanisms.
2. **Normalize the cache key** to strip inline literal values and use only the structural query template (or require clients to use variables, which naturally share a single cache key per query shape).
3. **Set an explicit `maximumPoolSize`** on the HikariCP pool for the graphql datasource appropriate to expected concurrency.
4. **Add a concurrency/connection semaphore** at the controller layer to bound simultaneous in-flight DB requests.

### Proof of Concept

```bash
# Phase 1: fill the cache with 1000 unique entries
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {entityId: {num: $i}}) { id } }\"}" &
done
wait

# Phase 2: thundering herd — 500 concurrent cache-miss requests
for i in $(seq 1001 1500); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {entityId: {num: $i}}) { id } }\"}" &
done
wait

# Observe: legitimate requests now time out or receive connection errors
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: {entityId: {num: 2}}) { id } }"}'
# Expected: connection timeout / HikariCP pool exhaustion error
```
