### Title
Integer Overflow in `poll()` Causes Unbounded Database Query via Negative Page Size

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` unsafely narrows a `long` to `int`. An unprivileged user can supply a `limit` value greater than `Integer.MAX_VALUE` (e.g., `2147483648`), which passes the `@Min(0)` validation but overflows to a negative `int` after the cast. This negative value propagates as the `limit` on the internal `newFilter`, causing `hasLimit()` to return `false` in `findByFilter()`, which then executes a JPA query with no `LIMIT` clause, loading all topic messages into memory.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` maps the raw gRPC `uint64` field directly to `TopicMessageFilter.limit` (a Java `long`): [1](#0-0) 

The only validation on `limit` is `@Min(0)`, which accepts any non-negative `long`, including values > `Integer.MAX_VALUE`: [2](#0-1) 

In `poll()`, the remaining limit is computed with an unsafe narrowing cast: [3](#0-2) 

With `filter.getLimit() = 2147483648L` and `total = 0`:
- `(int)(2147483648L - 0L)` = `(int)(2147483648L)` = **`-2147483648`** (Integer.MIN_VALUE)
- `Math.min(-2147483648, maxPageSize)` = **`-2147483648`**

This negative `pageSize` is set as the `limit` on `newFilter`: [4](#0-3) 

In `findByFilter()`, the guard `filter.hasLimit()` evaluates `limit > 0`: [5](#0-4) 

Since `-2147483648 > 0` is `false`, `setMaxResults()` is never called: [6](#0-5) 

The JPA query then executes with **no LIMIT clause**, and `getResultList()` loads the entire result set into memory: [7](#0-6) 

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative values; it does not bound the value to `Integer.MAX_VALUE`.
- Spring's `@Valid` is applied only at the `subscribeTopic()` service boundary. The `newFilter` built inside `poll()` bypasses this validation entirely and is passed directly to the repository.
- `isComplete()` uses the original filter's `limit` (still `2147483648`), so the polling loop does not terminate early. [8](#0-7) 

### Impact Explanation
Every poll cycle issues an unbounded `SELECT *` against the `topic_message` table for the target topic, loading all rows into the JVM heap via `getResultList()`. On a topic with millions of messages this causes heap exhaustion (OOM) or severe GC pressure, crashing or severely degrading the gRPC service for all users. This is a server-side denial-of-service reachable by any unauthenticated gRPC client.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible. The attacker only needs to send a single well-formed `ConsensusTopicQuery` with `limit` set to any value in the range `(Integer.MAX_VALUE, Long.MAX_VALUE]`. No authentication, no special role, and no prior knowledge beyond the topic ID is required. The attack is trivially repeatable and scriptable.

### Recommendation
Replace the unsafe narrowing cast with a bounded conversion. Before casting, clamp the remaining limit to `[0, Integer.MAX_VALUE]`:

```java
long remaining = filter.getLimit() - context.getTotal().get();
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

Additionally, add an upper-bound constraint on `TopicMessageFilter.limit` (e.g., `@Max(Long.MAX_VALUE)` is not useful, but a practical cap such as the configured `maxPageSize` or a documented maximum like `50_000_000`) to prevent excessively large values from entering the system at all.

### Proof of Concept

1. Identify any valid topic ID on the mirror node (e.g., topic `0.0.1234`).
2. Send a gRPC `subscribeTopic` request:
   ```
   ConsensusTopicQuery {
     topicID: { topicNum: 1234 }
     consensusStartTime: { seconds: 0 }
     limit: 2147483648   // Integer.MAX_VALUE + 1
   }
   ```
3. Observe in server logs/metrics that `poll()` issues a query with no LIMIT clause.
4. On a topic with a large message history, the server heap usage spikes as `getResultList()` materializes all rows; repeated requests exhaust heap and trigger OOM or severe GC pauses.
5. Confirm: set `limit = 2147483647` (Integer.MAX_VALUE) — this does NOT overflow and the query correctly uses `LIMIT 2147483647`. Set `limit = 2147483648` — overflow occurs and no LIMIT is applied.
