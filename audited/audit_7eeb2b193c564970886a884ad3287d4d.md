### Title
Unauthenticated Resource Exhaustion via Unbounded Entity Table Scan in `/api/v1/balances`

### Summary
An unauthenticated attacker can send repeated requests to `/api/v1/balances?account.balance=gte:0&limit=100` (no timestamp parameter), triggering the no-timestamp code path in `getBalances()`. This path builds a CTE that scans the entire `entity` table with no internal limit, then executes a correlated `token_account` subquery for each of the up to 100 returned rows. With no rate limiting present, concurrent requests exhaust the small DB connection pool (default: 10), holding each connection for up to the full `statement_timeout` (20 seconds), making the REST API unavailable to legitimate users.

### Finding Description
**Code path:** `rest/balances.js`, `getBalances()`, lines 133–149.

When no `timestamp` query parameter is present, the else-branch executes:

```javascript
// rest/balances.js lines 139-149
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances
  from entity_balance ab
  ${whereClause}
  order by ab.account_id ${order}
  ${limitQuery}`;
```

The `entity_balance` CTE contains **no LIMIT clause** — it selects all rows from `entity` where `type in ('ACCOUNT', 'CONTRACT')`. The `whereClause` (e.g., `where ab.balance>=0`) and `limitQuery` (`limit 100`) are applied only to the **outer** query. The correlated subquery `getTokenAccountBalanceSubQuery` (lines 308–318) then executes once per returned row, querying `token_account` for each account.

**Root cause / failed assumption:** The code assumes PostgreSQL will push the LIMIT down through the CTE and terminate the scan early. While PostgreSQL 16 can inline non-recursive, non-volatile CTEs, the planner is not guaranteed to do so when a `balance >= 0` filter (matching all rows) is present alongside an ORDER BY, and the correlated subquery adds significant per-row cost regardless. Even if the CTE is inlined, 100 correlated subqueries each scanning `token_account` on a mainnet-scale database (millions of accounts, many with dozens of token associations) generate substantial I/O.

**Exploit flow:**
1. Attacker sends `GET /api/v1/balances?account.balance=gte:0&limit=100` — no authentication required.
2. `parseBalanceQueryParam` accepts `gte:0` as valid, producing `ab.balance>=?` with value `0`.
3. `parseLimitAndOrderParams` caps limit at `responseLimit.max = 100` (default).
4. No timestamp → else-branch → full entity table CTE + 100 correlated subqueries.
5. Query runs until `statement_timeout = 20000ms` kills it, holding one DB connection the entire time.
6. Attacker repeats with 10+ concurrent connections, exhausting `maxConnections = 10` (default pool).
7. All subsequent legitimate requests queue or fail with connection timeout.

### Impact Explanation
The REST API mirror node becomes unavailable for the duration of the attack. Legitimate users cannot query account balances, transaction history, or other endpoints that share the same DB connection pool. **This does not affect Hedera network consensus** — the mirror node is a read-only service and does not participate in transaction confirmation. The severity classification of "total network shutdown" is incorrect; the actual impact is REST API denial-of-service.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and a single HTTP client capable of sending concurrent GET requests. The endpoint is publicly documented in the OpenAPI spec. No rate limiting exists in the REST API codebase. The attack is trivially repeatable and automatable. On mainnet-scale databases (millions of entity rows, large `token_account` tables), each query is likely to consume the full 20-second `statement_timeout`.

### Recommendation
1. **Add a LIMIT to the CTE** or restructure the query to apply the LIMIT before the correlated subquery executes, ensuring the entity table scan terminates early regardless of planner decisions.
2. **Add rate limiting** at the HTTP layer (e.g., per-IP request rate limiting via middleware) for the `/api/v1/balances` endpoint.
3. **Reduce `statement_timeout`** for the REST DB user on expensive read-only endpoints, or add a per-endpoint query timeout.
4. **Increase `maxConnections`** or add a request queue with a maximum wait time so connection exhaustion does not cause hard failures.
5. Consider requiring at least one selective filter (e.g., `account.id` equality) when no timestamp is provided, to prevent unbounded scans.

### Proof of Concept
```bash
# Send 15 concurrent requests to exhaust the default 10-connection pool
for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/balances?account.balance=gte:0&limit=100" &
done
wait

# Legitimate request now times out or receives connection error
curl -v "https://<mirror-node-host>/api/v1/balances?account.id=0.0.98"
```

On a mainnet-scale database, each of the 15 background requests will hold a DB connection for up to 20 seconds while scanning the full entity table and executing 100 correlated `token_account` subqueries. const getTokenAccountBalanceSubQuery = (order) => {

    select json_agg(json_build_object('token_id', token_id, 'balance', balance))

      select token_id, balance
      from token_account ta
