### Title
Gas Throttle Bypass via `findStorageBatch()` DB Query Amplification in `ContractStateServiceImpl`

### Summary
The gas-per-second throttle in `ThrottleManagerImpl` measures declared gas limit, not actual database query count. Because `findStorageBatch()` issues one DB query per `findStorage()` call containing **all** previously accumulated slot keys for that contract (O(N²) amplification), and because `gasLimitRefundPercent=100` restores unused gas after successful calls, an unprivileged attacker can sustain maximum-rate storage-heavy `eth_call` requests that saturate the shared PostgreSQL database while the gas bucket continuously refills. The throttle is per-node in-memory (bucket4j), so the shared DB receives load from every mirror-node instance simultaneously.

### Finding Description

**Code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → EVM SLOAD → `ContractStateServiceImpl.findStorage()` → `findStorageBatch()`

**Root cause — `findStorageBatch()` O(N²) DB amplification:** [1](#0-0) 

Every call to `findStorage()` for a slot not in `contractStateCache`:
1. Adds the new slot key to `contractSlotsCache` (per-contract, `expireAfterAccess=5m, maximumSize=1500`)
2. Reads **all** currently cached slot keys for that contract
3. Issues a single `findStorageBatch()` DB query with that entire growing key list

So for a contract with N unique slots accessed sequentially, the DB receives queries of size 1, 2, 3, … N — total O(N²) row lookups.

**Root cause — gas throttle does not account for DB query count:** [2](#0-1) 

The throttle consumes `scaleGas(request.getGas())` tokens upfront, then: [3](#0-2) 

With `gasLimitRefundPercent=100` (default), a successful call restores `min(gasRemaining, gasLimit)`. If a call uses 1 M gas out of 15 M, 14 M is returned. Net bucket drain ≈ actual gas used, not the declared limit. The bucket is only truly exhausted when calls consume the full 15 M gas.

**Exploit flow:**

An attacker deploys (or targets an existing) contract with ≥ 7,142 storage slots. Each `eth_call` with `gas=15,000,000` reads 7,142 unique cold slots (2,100 gas each ≈ 15 M gas total). This exhausts the gas bucket at exactly 100 req/s (1.5 B gas/s ÷ 15 M = 100). Each request triggers 7,142 `findStorage()` → `findStorageBatch()` calls with growing batch sizes (1 … 7,142 slots), producing O(N²) ≈ 25 M slot-row lookups per request, or **2.5 billion DB row lookups per second** at 100 req/s.

The `contractStateCache` TTL is only 2 seconds: [4](#0-3) 

After expiry, all slots are cold again and the full batch query sequence repeats. The `slotsPerContract` cache max is 1,500: [5](#0-4) 

When the 1,500-slot limit is exceeded, eviction triggers the `isKeyEvictedFromCache=true` fallback path: [6](#0-5) 

This falls back to individual `findStorage()` queries, further amplifying DB load.

**Why existing checks fail:**

- `requestsPerSecond=500` (default) is per-node, in-memory; 100 req/s is well within it.
- `gasPerSecond=1,500,000,000` (documented default) limits gas/s, not DB queries/s.
- The gas bucket is per-node (bucket4j `SYNCHRONIZED` strategy is intra-process only): [7](#0-6) 

With N mirror-node replicas behind a load balancer, the shared DB receives N × 100 req/s of heavy batch queries simultaneously.

### Impact Explanation
The shared PostgreSQL database is the single point of failure. Sustained O(N²) batch queries from `findStorageBatch()` at 100 req/s per node (× number of replicas) can saturate DB connection pools and I/O, causing query timeouts across all mirror-node services (REST API, importer, web3). This meets the ≥30% network processing degradation threshold because the DB is shared infrastructure. The `db.statementTimeout=3000ms` default means queries time out and cascade into errors across all endpoints. [8](#0-7) 

### Likelihood Explanation
No authentication is required for `eth_call`. The attacker needs only a publicly accessible mirror-node endpoint and a contract with many storage slots (trivially deployable on Hedera testnet/mainnet). The attack is repeatable indefinitely — the gas bucket refills every second, and the 2-second `contractStateCache` TTL ensures slots go cold again. A single attacker with a modest HTTP client can sustain 100 req/s.

### Recommendation
1. **Rate-limit DB queries, not just gas**: Track the number of `findStorage()` calls per request and enforce a per-request cap (e.g., 1,000 SLOADs).
2. **Cap `findStorageBatch()` batch size**: Limit `cachedSlots` to a fixed maximum (e.g., 100) per invocation to bound the O(N²) amplification.
3. **Distribute the gas throttle**: Use a distributed rate limiter (e.g., Redis-backed bucket4j) so the gas budget is shared across all replicas, not per-node.
4. **Increase `contractStateCache` TTL** or add a per-request slot-value cache to prevent repeated cold-slot DB hits within a single EVM execution.
5. **Add DB query count as a throttle dimension** alongside gas.

### Proof of Concept
```
# 1. Deploy a Solidity contract with 7142 storage slots (slot[0]..slot[7141])
# 2. Craft eth_call calldata that reads all 7142 slots via SLOAD in a loop
# 3. Send 100 concurrent requests/second:

for i in $(seq 1 100); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "<contract_address>",
      "gas": 15000000,
      "data": "<calldata_reading_7142_unique_slots>"
    }' &
done
# Repeat every second. Each request:
# - Consumes ~15M gas (7142 cold SLOADs × 2100 gas)
# - Triggers 7142 findStorageBatch() DB queries with sizes 1..7142
# - After 2s contractStateCache TTL, all slots go cold again
# - DB receives ~2.5B row lookups/second per node
# Observable: DB CPU/IO saturation, query timeouts, 503s on all mirror-node endpoints
```

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
