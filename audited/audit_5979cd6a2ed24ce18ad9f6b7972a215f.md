### Title
Pool Exhaustion DoS via Missing Client-Side Socket Timeout During Network Partition

### Summary
The REST API's `getNft()` function acquires a database connection from a pool capped at 10 slots (`max: 10`) and relies solely on a server-side `statement_timeout` (20000ms) to bound query duration. During a network partition between the API server and PostgreSQL, the server-side timeout cannot be delivered to the client, causing each acquired pool connection to hang indefinitely. An unprivileged attacker sending just 10 concurrent requests can exhaust all pool slots and maintain a complete denial-of-service for the duration of the partition with a single burst.

### Finding Description
**Code path:**

`rest/service/nftService.js` line 39 calls `pool.queryQuietly(NftService.nftByIdQuery, ...)` with no `preQueryHint` argument.

In `rest/utils.js` lines 1518–1520, when `preQueryHint` is falsy, execution falls into:
```js
result = await this.query(query, params);
```
This uses `pg-pool`'s managed `pool.query()`, which internally acquires a client, runs the query, and releases it. There is no client-side socket timeout, no `Promise.race()` with a deadline, and no application-level abort signal.

**Pool configuration** (`rest/dbpool.js` lines 7–16):
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20000ms
```

`statement_timeout` is a PostgreSQL server-side parameter. It is sent to the server at session start and enforced by the server process. During a network partition, the server cannot transmit the cancellation signal back to the client. The client-side TCP socket has no read deadline configured, so `pool.query()` awaits a response that never arrives.

`connectionTimeoutMillis` controls only how long a *new* caller waits for a free slot in the pool — it does not bound how long an already-acquired connection may remain occupied.

**Exploit flow:**
1. A network partition occurs between the Node.js REST API and PostgreSQL (e.g., firewall rule, cloud AZ split, DB failover window).
2. Attacker sends 10 concurrent HTTP requests to any endpoint invoking `getNft()` (e.g., `GET /api/v1/tokens/{tokenId}/nfts/{serialNumber}`).
3. Each request acquires one of the 10 pool slots and calls `pool.query()`. The TCP write succeeds (data enters the kernel buffer), but no response ever arrives.
4. All 10 pool slots are now occupied by hanging promises. The pool is exhausted.
5. Every subsequent legitimate request calls `pool.query()`, waits up to `connectionTimeoutMillis` (20 s) for a free slot, receives a pool timeout error, and returns HTTP 500 to the user.
6. The 10 attacker connections never time out on the client side; they remain held for the entire duration of the partition (potentially hours, until OS TCP keepalive fires — default ~2 hours on Linux).

**Why existing checks fail:**
- `statement_timeout = 20000` — server-side only; unreachable during partition.
- `connectionTimeoutMillis = 20000` — governs slot-wait time for *new* callers, not query execution duration for *current* holders.
- No application-level `AbortController`, `Promise.race`, or socket-level `setTimeout` wraps `pool.query()`.
- The REST API middleware (`rest/middleware/`) contains no rate limiting or concurrency cap for NFT endpoints. The throttling found (`web3/ThrottleManagerImpl`) applies only to the separate Web3 Java API.
- The Traefik `inFlightReq` / `rateLimit` middleware in the Helm chart is optional infrastructure (`{{ if and .Values.global.middleware .Values.middleware }}`) and is not a code-level guarantee.

### Impact Explanation
During any network partition — a realistic operational event — the entire REST API becomes unavailable to all users. Every API endpoint that touches the database (not just NFT endpoints) is affected because the shared global pool is exhausted. The attacker achieves a complete, sustained DoS with no authentication, no special knowledge, and no ongoing effort beyond the initial 10 requests. Recovery requires either the partition to heal or the Node.js process to be restarted.

### Likelihood Explanation
Network partitions occur in production environments (cloud AZ failures, DB failovers, misconfigured firewalls). An attacker monitoring API latency can detect degradation and immediately send 10 concurrent requests. No credentials, tokens, or elevated access are required — the endpoint is public. The attack rate (10 requests, once) is orders of magnitude below any conventional rate-limiting threshold. The attacker does not need to sustain traffic; the 10 connections self-maintain the DoS state.

### Recommendation
1. **Add a client-side socket/query timeout** in the `pg` client configuration: set `query_timeout` (node-postgres client option) to a value shorter than `connectionTimeoutMillis` (e.g., 15000ms). This bounds how long any single `pool.query()` call can block, regardless of server reachability.
2. **Set `idleTimeoutMillis`** on the pool so idle connections are recycled and stale connections during a partition are detected sooner.
3. **Wrap `pool.query()` with a `Promise.race`** against a deadline in `queryQuietly`, so hung queries are aborted at the application layer.
4. **Enforce a per-IP or global concurrency limit** at the REST API layer (e.g., express-rate-limit or Traefik `inFlightReq` as a mandatory deployment requirement) to cap simultaneous in-flight DB-bound requests.
5. **Configure OS-level TCP keepalive** on the database connection sockets to detect dead connections within seconds rather than hours.
