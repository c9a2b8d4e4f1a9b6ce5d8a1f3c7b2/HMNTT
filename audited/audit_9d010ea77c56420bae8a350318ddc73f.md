### Title
Unbounded Per-Subscriber DB Query Storm in `PollingTopicListener` Enables Partition-Amplified DoS

### Summary
When `hiero.mirror.grpc.listener.type=POLL` is configured, `PollingTopicListener.listen()` creates an independent `PollingContext` and polling loop per subscriber, each issuing its own `topicMessageRepository.findByFilter()` query every 500 ms. There is no global subscription cap enforced anywhere in the gRPC stack, and `maxConcurrentCallsPerConnection=5` limits only calls per connection — not the number of connections. An unauthenticated attacker who opens thousands of connections can therefore generate a sustained, unbounded query storm that exhausts the HikariCP connection pool and prevents the database from recovering from a network partition.

### Finding Description

**Exact code path:**

`CompositeTopicListener.listen()` delegates to `PollingTopicListener` when `type=POLL`: [1](#0-0) 

`PollingTopicListener.listen()` allocates a fresh `PollingContext` — with its own `AtomicReference<TopicMessage> last` — for every subscriber call, then starts an independent repeat-poll loop: [2](#0-1) 

Each loop calls `topicMessageRepository.findByFilter()` independently: [3](#0-2) 

The `PollingContext` inner class confirms the per-subscriber state: [4](#0-3) 

**No global subscription cap:** `TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge — it is never compared against a maximum before accepting a new subscription: [5](#0-4) [6](#0-5) 

**Per-connection limit does not bound total subscriptions:** `maxConcurrentCallsPerConnection=5` is enforced per TCP connection only: [7](#0-6) [8](#0-7) 

There is no limit on the number of connections, so an attacker with N connections gets 5N concurrent subscriptions.

**`Schedulers.boundedElastic()` is insufficient:** The scheduler (default: `10 × CPU cores` threads, queue depth 100,000) throttles *concurrent* thread execution but does not cap the number of enqueued poll tasks. With 1,000 subscribers polling every 500 ms, 2,000 tasks/second are enqueued. During a 30-second partition where each query blocks until `db.statementTimeout=10 s`, the bounded thread pool holds up to `10 × cores` DB connections open simultaneously — easily exhausting a typical HikariCP pool — while tens of thousands of tasks queue behind them. On partition recovery, all queued tasks fire in rapid succession. [9](#0-8) 

**No authentication:** `ConsensusController.subscribeTopic()` performs no credential check; any network-reachable client can subscribe: [10](#0-9) 

**Root cause / failed assumption:** The design assumes `POLL` mode is used only with a small, controlled number of subscribers (the documentation notes `SHARED_POLL` as the scalable alternative). No enforcement of that assumption exists in code.

### Impact Explanation

With `type=POLL` active, an attacker can hold the database connection pool at saturation for the entire duration of a partition and prevent it from draining on recovery. Every legitimate query — including those from the importer and REST API — is blocked. The `db.statementTimeout=10000 ms` means each hung query holds a HikariCP slot for up to 10 seconds; with a pool of 10–50 connections and hundreds of concurrent polling threads, the pool is fully exhausted. This constitutes a complete denial of service against the mirror node's database tier, triggered and sustained entirely by an unauthenticated external actor.

### Likelihood Explanation

The attack requires only a gRPC client library (e.g., `grpcurl`, any HCS SDK) and the ability to open many TCP connections — no credentials, no special knowledge, no privileged access. The `POLL` listener type is a documented, supported configuration option. The attacker does not need to cause the partition; they only need to be subscribed when one occurs naturally (or they can combine this with a separate network-layer disruption). The attack is fully repeatable and scriptable.

### Recommendation

1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount.get()` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP or per-connection subscription limit** at the Netty/gRPC interceptor layer, independent of `maxConcurrentCallsPerConnection`.
3. **Apply backpressure-aware circuit breaking** in `PollingTopicListener`: if the DB connection pool utilization exceeds a threshold (already monitored via the `GrpcHighDBConnections` Prometheus rule), pause or reject new poll scheduling rather than queuing indefinitely.
4. **Prefer `SHARED_POLL` or `REDIS`** (the default) in production; add a startup warning or documentation note that `POLL` mode is unsafe for public-facing deployments without an upstream connection limit.
5. **Set a bounded queue depth** on the `boundedElastic` scheduler used for polling, so that task accumulation during a partition is capped and excess tasks are dropped rather than queued.

### Proof of Concept

```bash
# Step 1: Configure the mirror node with type=POLL (non-default but documented)
# hiero.mirror.grpc.listener.type=POLL

# Step 2: Open 200 connections × 5 subscriptions each = 1,000 independent polling loops
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}}' \
      <mirror-node>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 3: Induce or wait for a DB network partition
# (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the mirror node host)

# Step 4: Observe:
#   - HikariCP active connections saturate to pool max
#   - All 1,000 polling loops queue tasks on boundedElastic
#   - On partition recovery, queued tasks fire simultaneously
#   - DB remains unresponsive to legitimate queries for the duration of the backlog drain
#   - hiero.mirror.grpc.subscribers gauge shows 1,000 with no enforcement action taken
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    @Data
    private class PollingContext {

        private final TopicMessageFilter filter;
        private final AtomicLong count = new AtomicLong(0L);
        private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();

        void onNext(TopicMessage topicMessage) {
            last.set(topicMessage);
            count.incrementAndGet();
        }

        private @Nullable TopicMessage getLast() {
            return last.get();
