### Title
Unbounded Request Body Size Allows Overflow Bucket Saturation in `requestBytesHistogram`

### Summary
`MetricsMiddleware()` in `rosetta/app/middleware/metrics.go` registers a Prometheus histogram with a maximum explicit bucket of `50*1024` bytes but applies no `http.MaxBytesReader` or equivalent body-size cap anywhere in the middleware chain. Any unauthenticated caller can POST arbitrarily large bodies, causing every such observation to land in the implicit `+Inf` overflow bucket, inflating `_sum` and corrupting percentile/average calculations used for capacity planning.

### Finding Description
**Code path:**

- `rosetta/app/middleware/metrics.go` line 20: `sizeBuckets = []float64{512, 1024, 10 * 1024, 25 * 1024, 50 * 1024}` — the highest explicit bucket is 51 200 bytes.
- Lines 22–26: `requestBytesHistogram` is created with only these buckets; no `NativeHistogramBucketFactor` or upper-bound clamp is set.
- Lines 76–84: `MetricsMiddleware` delegates to `weaveworks/common/middleware.Instrument`, which wraps `req.Body` in a `countingReadCloser`. Bytes are tallied as the downstream handler reads them and the total is `Observe()`d into `requestBytesHistogram` after the request completes.

**Middleware chain** (`rosetta/main.go` lines 217–219):
```
corsMiddleware → tracingMiddleware → metricsMiddleware → router
```
No stage in this chain calls `http.MaxBytesReader`. The grep across all `rosetta/**/*.go` confirms zero occurrences of `MaxBytesReader`, `LimitReader`, or any equivalent.

**Root cause / failed assumption:** The code assumes that legitimate Rosetta JSON payloads are small (well under 50 KB) and that the histogram buckets therefore cover the realistic range. It never enforces this assumption at the transport layer, so an attacker trivially violates it.

**Why existing checks are insufficient:**
- `ReadTimeout: 5 s` (default, `rosetta/main.go` line 225) only bounds wall-clock time, not bytes. On a 10 Mbps link a single connection can deliver ~6 MB in 5 s — two orders of magnitude above the 50 KB threshold.
- `ReadHeaderTimeout: 3 s` applies only to headers, not the body.
- The Rosetta SDK's `json.NewDecoder(r.Body).Decode()` reads the full body before returning an error for oversized/malformed JSON, so the `countingReadCloser` accumulates the full byte count before the handler rejects the request.

### Impact Explanation
Every request body exceeding 51 200 bytes is recorded in the `+Inf` bucket. This:
1. Inflates `hiero_mirror_rosetta_request_bytes_sum` without a corresponding increase in any named bucket, making `rate(…_sum)/rate(…_count)` (the average shown in the Grafana dashboard at `charts/hedera-mirror-common/dashboards/hedera-mirror-rosetta.json` line 1028) arbitrarily large.
2. Renders histogram-based percentile estimates (e.g., `histogram_quantile`) unreliable for all operators scraping the same Prometheus instance.
3. Misleads capacity-planning decisions (buffer sizing, rate-limit thresholds) for every operator of the mirror node.

Severity is low-to-medium: no funds are at risk, no data is corrupted, but observability infrastructure is permanently polluted until the process restarts or the metric is reset.

### Likelihood Explanation
Preconditions: none beyond network reachability of port 5700 (the default Rosetta port). The Rosetta API is intentionally public-facing. A single attacker with a modest script can sustain the attack indefinitely within the `ReadTimeout` window per connection, and can open many parallel connections. No authentication, no API key, no rate-limit is enforced at the application layer.

### Recommendation
1. **Enforce a body-size cap** immediately after the router is constructed and before `MetricsMiddleware` wraps it, using `http.MaxBytesReader` (e.g., 64 KB):
   ```go
   // rosetta/main.go, before MetricsMiddleware wraps router
   limitedRouter := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
       r.Body = http.MaxBytesReader(w, r.Body, 64*1024)
       router.ServeHTTP(w, r)
   })
   metricsMiddleware := middleware.MetricsMiddleware(limitedRouter)
   ```
2. **Extend `sizeBuckets`** to include a bucket at or above the enforced limit so that near-limit requests are still bucketed rather than overflowing.
3. Optionally add a rate-limiter (e.g., `golang.org/x/time/rate`) per source IP to bound the volume of oversized-body attempts.

### Proof of Concept
```bash
# Generate a 100 KB payload (well above the 50*1024 bucket ceiling)
python3 -c "import json,sys; sys.stdout.write(json.dumps({'network_identifier':{'blockchain':'Hedera','network':'mainnet'},'_pad':'A'*102400}))" > big.json

# Send at high volume (adjust --parallel and -n as needed)
ab -n 10000 -c 50 -p big.json -T application/json http://<rosetta-host>:5700/network/status

# Verify overflow bucket saturation
curl -s http://<rosetta-host>:5700/metrics | grep 'rosetta_request_bytes'
# Expected: hiero_mirror_rosetta_request_bytes_bucket{...,le="+Inf"} >> le="51200" bucket
# and _sum value far exceeds count * 51200, confirming overflow pollution
```
