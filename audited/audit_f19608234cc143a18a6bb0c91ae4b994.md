### Title
Unbounded gRPC Connection Multiplexing Enables Outbound Bandwidth Exhaustion via Unauthenticated Topic Subscriptions

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection(5)` on the Netty server, leaving no limit on the total number of inbound connections, no connection lifetime enforcement, and no per-IP or global bandwidth controls. Because `ConsensusService.subscribeTopic` requires no authentication and accepts indefinite subscriptions (`limit=0`, no `consensusEndTime`), an unprivileged attacker can open an unbounded number of connections — each carrying 5 concurrent streams — causing the server to replicate the full message stream to every subscriber and saturate outbound network I/O.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default: 5
};
```

`NettyProperties.java` line 14: `private int maxConcurrentCallsPerConnection = 5;`

The following `NettyServerBuilder` controls are **absent**:
- `maxConnectionAge` / `maxConnectionIdle` — connections live indefinitely
- `flowControlWindow` — default HTTP/2 window (65535 bytes) is used
- Any per-IP or global connection count limit
- Any authentication interceptor (the only `@GlobalServerInterceptor` in the grpc module is `GrpcInterceptor` in the test tree, which only sets an `EndpointContext` thread-local)

**Root cause:** `maxConcurrentCallsPerConnection` limits *streams per connection*, not the number of connections. There is no ceiling on how many TCP connections a single client IP may establish.

**Exploit flow:**
1. Attacker identifies a high-throughput HCS topic (publicly visible on-chain).
2. Attacker opens *N* TCP connections to port 5600 (no connection limit enforced).
3. On each connection, attacker issues 5 `subscribeTopic` calls with `limit=0` and no `consensusEndTime` → indefinite streaming.
4. `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) creates a live `Flux` per subscriber with no subscriber-count cap; the `subscriberCount` gauge is metrics-only.
5. The server serialises and transmits a full copy of every `ConsensusTopicResponse` (up to ~6 KiB per message per the proto comment) to each of the *N×5* open streams simultaneously.
6. Outbound bandwidth scales as: `N_connections × 5 × message_rate × message_size`.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection=5` is per-connection; opening 100 connections yields 500 concurrent streams.
- Infrastructure-level controls (`maxRatePerEndpoint: 250` in the GCP backend policy, Traefik middleware) are gated on `gateway.gcp.enabled: true` / `global.middleware: true` and are absent in bare-metal or non-GCP deployments.
- The web3 `ThrottleConfiguration` (bucket4j) is scoped to the `web3` module and has no effect on the `grpc` module.
- No `ServerInterceptor` in the grpc application path enforces any rate or connection limit.

### Impact Explanation
Every additional attacker connection causes the server to duplicate its entire outbound message stream. On a topic receiving the maximum 6 KiB messages at even a modest rate (e.g., 100 msg/s), 20 attacker connections × 5 streams = 100 streams × 6 KiB × 100 msg/s = ~60 MB/s of additional outbound traffic, easily exceeding a 30% increase over baseline on a lightly loaded node. The attack is amplified by the fact that the server must also serialise each protobuf response independently per stream, adding CPU overhead alongside the network saturation.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or any HTTP/2 library), zero credentials, and knowledge of a valid topic ID (all topic IDs are public). It is repeatable, scriptable, and can be sustained indefinitely because connections are never forcibly closed. A single commodity machine with a moderate uplink can open hundreds of TCP connections.

### Recommendation
In `grpcServerConfigurer()`, add the following controls to `NettyServerBuilder`:

```java
serverBuilder.maxConnectionAge(grpcProperties.getNetty().getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(grpcProperties.getNetty().getMaxConnectionIdle(), TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(grpcProperties.getNetty().getMaxConnectionAgeGrace(), TimeUnit.SECONDS);
```

Additionally:
- Add a `ServerInterceptor` that tracks and enforces a per-IP connection/stream count, rejecting excess calls with `Status.RESOURCE_EXHAUSTED`.
- Expose `maxConnections` as a configurable `NettyProperties` field and call `serverBuilder.maxConnectionAge(...)`.
- Require the infrastructure-level rate limiting (GCP backend policy / Traefik) to be enabled by default in all deployment profiles, not opt-in.
- Consider requiring a subscription `limit` or `consensusEndTime` to prevent truly indefinite streams.

### Proof of Concept
```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Replace TOPIC_NUM with a known high-throughput topic (e.g., 0.0.1234)

for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1234}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 50 connections × 5 streams each (if using a multiplexed client) = 250 concurrent streams
# Monitor server outbound bandwidth: watch -n1 'cat /proc/net/dev | grep eth0'
# Observe network I/O increase proportional to connection count with no server-side rejection
```
