### Title
Unbounded Concurrent gRPC Subscriptions Cause Database Query Storm via PollingTopicListener

### Summary
`PollingTopicListener.listen()` hardcodes `RepeatSpec.times(Long.MAX_VALUE)` with a fixed polling interval, issuing one `topicMessageRepository.findByFilter()` database query per active subscription per interval tick. No rate limiting, authentication, or concurrent-subscription cap exists on the gRPC `subscribeTopic` endpoint, so any unauthenticated attacker can open an arbitrary number of concurrent subscriptions to the fee-schedule topic, multiplying DB query load linearly and starving legitimate fee-update consumers.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43ŌĆō53) accepts any unauthenticated gRPC call and immediately delegates to `TopicMessageServiceImpl.subscribeTopic()`, which calls `topicListener.listen(newFilter)` (line 120). When the listener type is `POLL`, this resolves to `PollingTopicListener.listen()`.

Inside `listen()` (lines 38ŌĆō43):
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ŌåÉ hardcoded, per-subscription
                .jitter(0.1)
                .withFixedDelay(interval)              // ŌåÉ default 500 ms, min 50 ms
                .withScheduler(scheduler))
```

Every active subscription independently calls `poll()` (line 61):
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`findByFilter()` executes a full JPA criteria query against the `topic_message` table (lines 33ŌĆō61 of `TopicMessageRepositoryCustomImpl`), including a native `SET LOCAL random_page_cost = 0` hint that modifies the planner for every call.

**Failed assumptions / missing controls:**

- `TopicMessageServiceImpl` only tracks `subscriberCount` as a Micrometer gauge (lines 52ŌĆō55, 89ŌĆō90) ŌĆö it is never compared against a maximum; no rejection path exists.
- `ConsensusController.subscribeTopic()` has no authentication check, no per-IP rate limit, and no global subscription cap.
- `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection, but an attacker trivially opens many connections.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module; it is entirely absent from the `grpc` module.
- The sole `GrpcInterceptor` (test package) only sets `EndpointContext`; it performs no throttling.

**Exploit flow:**

1. Attacker opens `C` TCP connections to the gRPC port (no credentials required).
2. On each connection, attacker issues up to 5 concurrent `subscribeTopic` calls targeting the fee-schedule topic (e.g., topic `0.0.111`), with no `limit` and no `endTime` (so the subscription never self-terminates).
3. Each subscription enters `PollingTopicListener.listen()` and begins issuing `findByFilter()` every `interval` ms (default 500 ms; configurable down to 50 ms).
4. Total DB query rate = `C ├Ś 5 / interval_seconds`. With `C = 200` connections and the default 500 ms interval: **2,000 queries/second** against the `topic_message` table.
5. The `boundedElastic` scheduler queues tasks up to 100,000 entries, so the reactor pipeline does not self-limit.

### Impact Explanation

The fee-schedule topic carries network-wide transaction fee updates. A sustained query storm against `topic_message` saturates the PostgreSQL connection pool and I/O bandwidth, causing:
- Legitimate fee-update subscribers to receive delayed or dropped messages, causing clients to operate on stale fee schedules.
- Cascading latency across all other gRPC and REST consumers sharing the same database.
- Potential complete unavailability of the mirror node gRPC service (DoS).

Severity: **High** ŌĆö directly impacts fee-schedule propagation integrity and service availability with no privilege required.

### Likelihood Explanation

The attack requires only a standard gRPC client library (e.g., `grpcurl`, Hedera SDK) and network access to port 5600. No account, token, or credential is needed. The attack is fully scriptable, repeatable, and can be sustained indefinitely since each subscription loop runs `Long.MAX_VALUE` iterations. A single commodity machine with 200 concurrent TCP connections is sufficient to produce a meaningful query storm.

### Recommendation

1. **Enforce a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP / per-connection subscription rate limiting** via a gRPC `ServerInterceptor` using a token-bucket (analogous to `ThrottleConfiguration` in the `web3` module).
3. **Require subscriptions without `endTime` or `limit` to carry an authenticated identity**, or enforce a server-side maximum subscription duration.
4. **Prefer `SharedPollingTopicListener`** (or `REDIS` mode) when multiple subscribers target the same topic ŌĆö it issues a single shared poll regardless of subscriber count, eliminating the per-subscriber query multiplication.
5. **Set a minimum `interval` floor** in `ListenerProperties` that is operationally safe under maximum expected subscriber load, and document it as a security parameter.
