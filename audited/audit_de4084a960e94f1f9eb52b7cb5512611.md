### Title
Alias Cache Exhaustion via 1-Second TTL Enables Sustained DB Query Amplification Without Privileges

### Summary
The `CACHE_NAME_ALIAS` sub-cache inside `cacheManagerEntity()` is configured with `expireAfterWrite=1s,maximumSize=10000`. Because the only request-level protection is a single global (non-per-IP) rate-limit bucket of 500 req/s, an unprivileged attacker can continuously send `eth_call` requests that each resolve a distinct alias value, guaranteeing a DB lookup on every request. The 1-second TTL ensures that even previously-seen aliases re-miss the cache every second, sustaining the amplified DB query rate indefinitely.

### Finding Description

**Exact code path**

`CacheProperties.java` line 19 defines the shared entity cache spec:
```
expireAfterWrite=1s,maximumSize=10000,recordStats
``` [1](#0-0) 

`EvmConfiguration.cacheManagerEntity()` applies this spec to `CACHE_NAME_ALIAS` (and two other sub-caches) via a single shared `CaffeineCacheManager`: [2](#0-1) 

`EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse()` is the DB method backed by this cache. Its key is `@spelHelper.hashCode(#alias)` and its negative-result guard is `unless = "#result == null"`: [3](#0-2) 

`CommonEntityAccessor.get(Bytes alias, ...)` is the call site that routes non-historical alias lookups to this cached method: [4](#0-3) 

**Root cause and failed assumption**

The design assumes the cache will absorb repeated lookups for the same alias. This holds only when the same alias is queried more than once within the same 1-second window. An attacker who rotates through ≥500 distinct alias values per second ensures every single request is a cache miss, because:

1. Each distinct alias produces a unique Caffeine key (`hashCode(alias)`).
2. `unless = "#result == null"` does **not** suppress caching of `Optional.empty()` (an empty Optional is not `null`), so negative results are cached — but only for 1 second. After expiry the same alias causes another DB hit.
3. With 500 distinct aliases per second the attacker never needs to wait for expiry; every request is a first-time lookup.

**Exploit flow**

1. Attacker identifies any deployed contract that internally calls an alias-resolving precompile (e.g. `isValidAlias`, `getEvmAddressAlias`, `getHederaAccountNumAlias` on the HRC-632 contract, or any token precompile that accepts an alias address as a parameter).
2. Attacker sends `POST /api/v1/contracts/call` at the maximum allowed rate (default 500 req/s) with the `data` field encoding calls to that contract, each using a different 20-byte or public-key alias as the argument.
3. Each request reaches `AliasesReadableKVState.readFromDataSource()` → `CommonEntityAccessor.get(Bytes, ...)` → `EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse()`.
4. Because every alias is distinct, the Caffeine cache is always cold for that key → a full DB query (`SELECT * FROM entity WHERE (evm_address = ? OR alias = ?) AND deleted IS NOT TRUE`) is issued for every request.
5. After 1 second the attacker may recycle the same 500 aliases; they have all expired, so the next cycle again produces 500 DB hits.

**Why existing checks are insufficient**

The only active guard is the global `rateLimitBucket` (default 500 req/s) and the global `gasLimitBucket`: [5](#0-4) [6](#0-5) 

Both buckets are **global singletons** — there is no per-IP, per-account, or per-alias rate limiting anywhere in the codebase. A single attacker can legally consume the entire 500 req/s budget, leaving zero capacity for legitimate users while simultaneously driving 500 uncached DB queries per second.

### Impact Explanation

Each attacker request that misses the alias cache issues a native SQL query against the `entity` table with an `OR` predicate on two indexed columns (`evm_address`, `alias`). At 500 such queries/second from a single unauthenticated source, and given that a typical mirror-node deployment handles on the order of 1,000–2,000 DB queries/second across all subsystems, the additional load easily exceeds the 30% threshold. The attack also starves legitimate `eth_call` traffic because the global rate-limit bucket is fully consumed.

### Likelihood Explanation

- **No credentials required** — the `/api/v1/contracts/call` endpoint is publicly accessible.
- **Trivially scriptable** — a simple loop sending `eth_call` to any alias-aware precompile with a rotating 20-byte address is sufficient.
- **Repeatable and sustained** — the 1-second TTL guarantees the attack remains effective indefinitely without any state on the attacker side.
- **No on-chain cost** — `eth_call` is a read-only simulation; the attacker pays nothing.

### Recommendation

1. **Increase the alias cache TTL** — `expireAfterWrite=1s` is far too short for alias-to-entity mappings that change infrequently. A TTL of 30–60 seconds would reduce the DB hit rate by 30–60× for repeated aliases.
2. **Add per-IP (or per-source) rate limiting** — the current single global bucket allows one client to monopolise the entire request budget. A per-IP sub-bucket (e.g. 50 req/s per IP) would cap the per-attacker DB query rate.
3. **Cache negative results with a separate, longer TTL** — non-existent aliases are currently cached for only 1 second. A dedicated negative-result cache with a longer TTL (e.g. 10s) would prevent repeated DB hits for the same non-existent alias.
4. **Consider request-level alias-count limiting** — reject or throttle requests whose calldata encodes more than N distinct alias arguments per second from the same source.

### Proof of Concept

```bash
# Pre-condition: HRC632 contract deployed at $CONTRACT_ADDR on the mirror node
# Tool: any HTTP client capable of 500 req/s (e.g. wrk, hey, or a simple loop)

ENDPOINT="http://<mirror-node>/api/v1/contracts/call"

# isValidAlias(address) selector = 0x5a4f9a8e (example)
# Rotate through 10,000 distinct 20-byte addresses as the alias argument

for i in $(seq 1 10000); do
  ALIAS=$(printf '%040x' $i)   # distinct alias per request
  DATA="0x5a4f9a8e000000000000000000000000${ALIAS}"
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT_ADDR\",\"data\":\"$DATA\",\"gas\":50000}" &
done
wait

# Each request misses CACHE_NAME_ALIAS → issues SELECT against entity table.
# Repeat every second to exploit the 1s TTL expiry.
# Monitor DB query rate: expect >30% increase over baseline within 5 seconds.
```
