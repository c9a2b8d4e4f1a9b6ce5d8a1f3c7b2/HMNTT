### Title
Unbounded Concurrent `subscribeTopic()` Streams Exhaust Shared `Schedulers.boundedElastic()` Thread Pool via `PollingTopicMessageRetriever`

### Summary
`PollingTopicMessageRetriever.retrieve()` creates a shared `Schedulers.boundedElastic()` instance and uses it to schedule repeat polling tasks. When `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE` with no enforced subscriber limit anywhere in the call chain. An unprivileged attacker can open arbitrarily many concurrent `subscribeTopic()` gRPC streams, each driving blocking DB polls on the shared scheduler, saturating the thread pool and degrading or halting all scheduler-dependent processing for legitimate users.

### Finding Description

**Code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`

- Lines 34–42: A single `Schedulers.boundedElastic()` instance is created at construction time and shared across **all** concurrent subscriptions:
  ```java
  scheduler = Schedulers.boundedElastic();
  ```
- Lines 94–108 (`PollingContext` constructor): When `throttled=true`, `numRepeats` is unconditionally set to `Long.MAX_VALUE`:
  ```java
  if (throttled) {
      numRepeats = Long.MAX_VALUE;
      frequency = retrieverProperties.getPollingFrequency();  // default 2s
      maxPageSize = retrieverProperties.getMaxPageSize();
  }
  ```
- Lines 51–55: The `repeatWhen` operator uses `.withScheduler(scheduler)`, meaning when each 2-second delay expires, the re-subscription (and thus the call to `poll()`) is dispatched **on a thread from the shared boundedElastic pool**:
  ```java
  .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
          .jitter(0.1)
          .withFixedDelay(context.getFrequency())
          .withScheduler(scheduler))
  ```
- Line 78: `poll()` calls `topicMessageRepository.findByFilter(newFilter)` — a **blocking JDBC call** — which holds the scheduler thread for the duration of the DB query.

**`TopicMessageServiceImpl.subscribeTopic()`** (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 59–92):
- Line 63: Every new subscriber immediately triggers `retrieve(filter, true)` (throttled=true → `numRepeats=Long.MAX_VALUE`).
- Lines 48–55: `subscriberCount` is tracked as a gauge metric but **never enforced** — there is no `if (subscriberCount > MAX) reject` guard anywhere.
- Line 70: The `safetyCheck` also uses `subscribeOn(Schedulers.boundedElastic())`, adding further pressure on the same pool.

**Root cause:** The shared `Schedulers.boundedElastic()` pool (default: `10 × availableProcessors` threads, e.g. 80 threads on an 8-core host) is used to dispatch blocking DB poll tasks for every concurrent subscriber. No subscriber limit is enforced. An attacker opening N concurrent streams causes N simultaneous blocking DB calls on the shared pool. Once all threads are occupied, new tasks queue (up to 100,000 by default); when the queue fills, tasks are rejected with `RejectedExecutionException`, breaking all scheduler-dependent operations including timeouts and safety checks.

**Why the timeout does not save it:** The `.timeout(Duration.ofSeconds(60), scheduler)` (line 59) itself relies on the same saturated scheduler to fire the timeout signal. If the scheduler is exhausted, timeout tasks are also queued/delayed, meaning subscriptions can outlive their 60-second window, compounding the exhaustion.

### Impact Explanation
The shared `Schedulers.boundedElastic()` is the single thread pool backing all polling, timeout signaling, and safety-check subscriptions for the entire gRPC node process. Saturating it causes:
- All active legitimate subscribers to stop receiving messages (polls queue indefinitely).
- Timeout signals delayed or never fired, keeping zombie subscriptions alive longer.
- The safety-check gap-recovery path (`subscribeOn(Schedulers.boundedElastic())`) also stalls.
- Effective shutdown of the node's topic-message processing capability without touching consensus — matching the stated scope of "≥30% of network processing nodes" impact.

### Likelihood Explanation
- **No authentication required**: The gRPC `subscribeTopic` endpoint is publicly accessible; any client with network reach can call it.
- **No rate limiting or subscriber cap**: `subscriberCount` is a metric only; no enforcement exists.
- **Low resource cost per attacker stream**: Each stream is a single gRPC streaming RPC with a minimal filter (e.g., `startTime=0`, no `endTime`, no `limit`). The attacker does not need to send any data after the initial request.
- **Repeatable and automatable**: A simple loop opening gRPC streams is sufficient. On an 8-core host, ~80–100 concurrent streams saturate the pool; on smaller deployments fewer are needed.
- **Sustained attack**: The attacker continuously opens new streams as old ones time out (every 60s), maintaining saturation indefinitely.

### Recommendation
1. **Enforce a maximum subscriber count**: In `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount` against a configurable maximum and return an error (e.g., `RESOURCE_EXHAUSTED` gRPC status) before proceeding.
2. **Isolate the polling scheduler**: Give `PollingTopicMessageRetriever` its own bounded scheduler with a fixed thread cap (e.g., `Schedulers.newBoundedElastic(maxThreads, maxQueue, "polling")`) separate from the scheduler used for timeouts and safety checks, so saturation in one does not cascade.
3. **Make blocking DB calls non-blocking or wrap with `subscribeOn` on a dedicated pool**: Ensure `topicMessageRepository.findByFilter()` does not hold a shared scheduler thread; use a dedicated DB-query thread pool.
4. **Add per-IP or per-client connection rate limiting** at the Netty/gRPC layer via `NettyProperties` or an interceptor.
5. **Cap `numRepeats` for throttled mode**: Even for live subscriptions, a finite repeat count with re-subscription logic is safer than `Long.MAX_VALUE`.

        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));
