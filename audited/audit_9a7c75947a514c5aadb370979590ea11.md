### Title
Unbounded Concurrent Subscriptions via `subscribeTopic()` Enable Resource Exhaustion DoS

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no global or per-client cap on concurrent subscriptions. The only server-side guard, `maxConcurrentCallsPerConnection = 5`, is scoped per TCP connection and is trivially bypassed by opening multiple connections. An unprivileged attacker can flood the server with indefinitely-held subscriptions, exhausting the `boundedElastic` scheduler thread pool, the database connection pool, and heap memory.

### Finding Description

**Code path and root cause:**

`subscriberCount` at [1](#0-0)  is a pure metric gauge — it is incremented/decremented for observability only and is never checked against any maximum before accepting a new subscription. [2](#0-1) 

The only structural limit is `maxConcurrentCallsPerConnection`, defaulting to 5, applied at the Netty layer: [3](#0-2) [4](#0-3) 

This is a **per-connection** limit. An attacker opening `C` connections can hold `5 × C` concurrent subscriptions simultaneously.

**Per-subscription resource consumption:**

Each call to `subscribeTopic()` unconditionally allocates:
1. A `TopicContext` object with `AtomicLong`, `AtomicReference`, `Stopwatch`. [5](#0-4) 
2. A historical retrieval flux that issues DB queries. [6](#0-5) 
3. A **safety-check** flux that schedules work on `Schedulers.boundedElastic()` — a bounded thread pool (default: 10× CPU cores, max ~200 k queued tasks). [7](#0-6) 
4. A live listener subscription via `topicListener.listen()`. [8](#0-7) 

**Indefinite lifetime when no `endTime` is set:**

When `filter.getEndTime() == null`, `pastEndTime()` returns `Flux.never()`, so the subscription never self-terminates. [9](#0-8)  `isComplete()` also returns `false` unconditionally in this case. [10](#0-9) 

**Filter validation does not restrict subscription count:**

`TopicMessageFilter` validation only enforces `@Min(0)` on `limit`, `@NotNull` on `topicId`, and temporal ordering of `startTime`/`endTime`. There is no field that limits how many times a client may call `subscribeTopic()`. [11](#0-10) 

### Impact Explanation

An attacker opening `N` TCP connections (each with 5 calls, no `endTime`, valid `topicId`) holds `5N` indefinitely-live subscriptions. Each subscription pins:
- A `boundedElastic` worker for the safety-check path
- A DB connection slot for historical retrieval polling
- Heap for `TopicContext` and filter copies

Once the `boundedElastic` pool and/or the DB connection pool are saturated, legitimate subscribers receive errors or indefinite delays. This constitutes a denial-of-service against the gRPC network layer with no funds at direct risk, matching the stated severity scope (Layer 0/1/2 network DoS).

### Likelihood Explanation

No authentication is required to call `subscribeTopic()` — any network-reachable client can issue gRPC `SubscribeTopic` RPCs. The only friction is the per-connection cap of 5, which is bypassed by opening more connections. This is trivially scriptable with any gRPC client library. The attack is repeatable and requires no special knowledge beyond a valid topic ID (which is public on-chain data).

### Recommendation

1. **Enforce a global subscription cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription in `subscribeTopic()`, and return `RESOURCE_EXHAUSTED` status if exceeded.
2. **Enforce a per-IP/per-client subscription cap**: Track active subscriptions by remote address and reject excess requests.
3. **Require `endTime` or `limit`**: Reject or auto-cap subscriptions that specify neither, preventing indefinitely-held streams.
4. **Add connection-level rate limiting**: Complement `maxConcurrentCallsPerConnection` with a global `maxConnections` setting on the `NettyServerBuilder`.

### Proof of Concept

```python
# Pseudocode — repeat with any gRPC client
import grpc, threading

stubs = []
for i in range(200):                          # 200 connections
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(channel)
    stubs.append(stub)

def flood(stub):
    for _ in range(5):                        # 5 calls per connection = 1000 total
        query = ConsensusTopicQuery(
            topicID=TopicID(topicNum=1),      # any valid topic
            # no consensusEndTime → indefinite subscription
        )
        threading.Thread(
            target=lambda: list(stub.subscribeTopic(query))
        ).start()

for stub in stubs:
    flood(stub)

# Result: 1000 indefinitely-held subscriptions
# boundedElastic pool saturated → safety-check tasks queue unboundedly
# DB connection pool exhausted → legitimate queries time out
# Server becomes unresponsive to new subscribers
```

public class TopicMessageFilter {

    private static final SecureRandom RANDOM = new SecureRandom();

    private Long endTime;

    @Min(0)
    private long limit;

    @Min(0)
    @NotNull
    @Builder.Default
    private long startTime = DomainUtils.now();

    @Builder.Default
    private String subscriberId = RandomStringUtils.random(8, 0, 0, true, true, null, RANDOM);

    @NotNull
    private EntityId topicId;

    public boolean hasLimit() {
        return limit > 0;
    }

    @AssertTrue(message = "End time must be after start time")
    public boolean isValidEndTime() {
        return endTime == null || endTime > startTime;
    }

    @AssertTrue(message = "Start time must be before the current time")
    public boolean isValidStartTime() {
        return startTime <= DomainUtils.now();
    }
