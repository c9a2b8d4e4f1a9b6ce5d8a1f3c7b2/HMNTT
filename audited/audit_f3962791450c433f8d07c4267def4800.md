### Title
Shared `boundedElastic` Scheduler Between `repeatWhen` and `timeout` Enables Zombie Subscription Persistence via Thread Starvation During Network Partition

### Summary
`PollingTopicMessageRetriever` (a singleton Spring bean) creates a single `Schedulers.boundedElastic()` instance that is shared between the `repeatWhen` operator (which executes blocking JDBC calls on scheduler threads) and the `timeout` operator (which schedules its cancellation task on the same scheduler). During a network partition, an unprivileged attacker who opens enough concurrent gRPC subscriptions can exhaust all scheduler threads with hung JDBC calls, preventing the `timeout` task from being dispatched, and keeping subscriptions alive indefinitely beyond `retrieverProperties.getTimeout()`.

### Finding Description

**Exact code path**: `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`

**Root cause — single scheduler, two competing roles**:

```java
// Line 41 — one scheduler instance for the entire singleton bean
scheduler = Schedulers.boundedElastic();

// Lines 51-62 — both operators share it
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(...)
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))          // ← role 1: re-subscription thread
        ...
        .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
        .timeout(retrieverProperties.getTimeout(), scheduler)  // ← role 2: timeout task
```

`RepeatSpec.withScheduler(scheduler)` causes Reactor to re-subscribe to `Flux.defer(() -> poll(context))` on a `boundedElastic` worker thread after each delay. `poll()` calls `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 78), which is a synchronous JDBC stream that runs to completion on that same worker thread. During a network partition the JDBC socket blocks until the OS-level TCP timeout fires (often 2–10 minutes when no explicit `socketTimeout` is set on the datasource), holding the worker thread for the entire duration.

`timeout(retrieverProperties.getTimeout(), scheduler)` (default 60 s) schedules a `TimeoutTask` on the **same** `boundedElastic` scheduler. `BoundedElasticScheduler` has a hard cap of `10 × Runtime.availableProcessors()` threads (e.g. 40 on a 4-core host). If all 40 threads are blocked in JDBC calls, the `TimeoutTask` is queued but never dispatched until a thread is freed — which only happens when the OS TCP timeout fires, not when the configured 60 s elapses.

`retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` (line 58) placed **inside** the `timeout` boundary means every JDBC failure is retried indefinitely, continuously re-occupying scheduler threads after each backoff.

**Why existing checks fail**:
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14) limits calls per TCP connection, not total connections. An attacker opens ⌈N/5⌉ connections to achieve N concurrent subscriptions.
- No global gRPC connection limit or per-IP rate limit exists in the gRPC module (the `ThrottleConfiguration` / `ThrottleManagerImpl` is web3-only).
- The `timeout` operator is the sole termination mechanism for zombie subscriptions, and it is the mechanism being defeated.

### Impact Explanation

Zombie subscriptions hold:
- A `boundedElastic` worker thread (blocked in JDBC) for the duration of the OS TCP timeout
- A HikariCP database connection per active poll (the pool alert fires at 75 % utilisation per the Helm alert rule)
- gRPC stream state and associated heap objects

With all scheduler threads blocked, **no** subscription managed by this singleton retriever can time out, including legitimate ones. The service degrades to a state where new subscriptions queue indefinitely and the gRPC pod becomes unresponsive. This is a full denial-of-service against the historical-message retrieval path, achievable without any credentials.

### Likelihood Explanation

Preconditions:
1. A network partition between the mirror-node gRPC pod and the database (can be caused externally, e.g. firewall rule, or coincide with a real outage).
2. The attacker can open gRPC connections to port 5600 (publicly exposed per the Helm service definition).
3. No JDBC `socketTimeout` is configured (common default).

On a 4-core pod (40 `boundedElastic` threads), the attacker needs 8 TCP connections × 5 calls each = 40 concurrent `subscribeTopic` RPCs. This is trivially achievable with any gRPC client library in a single script. The attack is repeatable: as soon as threads free (OS TCP timeout), the attacker re-subscribes. No authentication, no special knowledge, no timing precision required.

### Recommendation

1. **Separate schedulers**: Use `Schedulers.newBoundedElastic(...)` with a dedicated instance for `timeout` so its tasks are never queued behind blocking JDBC work:
   ```java
   private final Scheduler pollScheduler  = Schedulers.boundedElastic();
   private final Scheduler timerScheduler = Schedulers.newSingle("retriever-timer");
   // ...
   .repeatWhen(...withScheduler(pollScheduler))
   .timeout(retrieverProperties.getTimeout(), timerScheduler)
   ```
2. **Move blocking I/O off the repeat scheduler**: Wrap `poll()` with `.subscribeOn(pollScheduler)` and use a non-blocking or virtual-thread executor for JDBC, so scheduler threads are not held during network I/O.
3. **Set JDBC socket timeout**: Configure `socketTimeout` (PostgreSQL JDBC) or equivalent to a value shorter than `retrieverProperties.getTimeout()` so threads are released promptly during a partition.
4. **Add a global gRPC connection/subscription limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / `maxConcurrentCallsPerConnection` at the server level, or an IP-level rate limiter.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
