### Title
Connection Pool Exhaustion via Unauthenticated Concurrent Requests Starves `/network/status` DB Access

### Summary
The Rosetta API's shared `database/sql` connection pool, configured with `MaxOpenConnections: 100` and no application-level rate limiting or connection-acquisition timeout, can be fully exhausted by an unprivileged attacker sending concurrent requests to any DB-backed endpoint. Once the pool is exhausted, the internal readiness check's `/network/status` call blocks indefinitely waiting for a connection, causing the health probe to time out, the pod to be marked not-ready, and the node to be removed from the load balancer — a de-facto network partition.

### Finding Description

**Pool configuration** is defined as a plain struct with no validation or minimum-connection-acquisition-timeout field: [1](#0-0) 

Defaults (from documentation) are `MaxIdleConnections: 20`, `MaxOpenConnections: 100`, `MaxLifetime: 30` minutes. These are applied directly to the `database/sql` pool with no connection-wait timeout: [2](#0-1) 

Go's `database/sql` semantics: when `MaxOpenConnections` is reached, any call to acquire a connection **blocks the goroutine indefinitely** until one is returned — unless the request context is cancelled. The only cancellation boundary is the HTTP `WriteTimeout` (default 10 s): [3](#0-2) 

**`/network/status`** makes three sequential DB round-trips per request (genesis block, latest block, address book entries), each acquiring and releasing a pool connection: [4](#0-3) 

The readiness health check calls `/network/status` via HTTP (not a direct DB ping), so it competes for the same pool: [5](#0-4) 

The readiness check has a 10-second timeout: [6](#0-5) 

**There is no application-level rate limiting or in-flight request cap** inside the Rosetta Go process. The Traefik middleware (`inFlightReq: amount: 5` per IP, `rateLimit: average: 10` per host) is defined in the Helm chart but is **opt-in** (`global.middleware: false` by default) and is per-IP, so a distributed attacker bypasses it entirely: [7](#0-6) 

**Exploit flow:**
1. Attacker opens 100 concurrent HTTP connections to any DB-backed endpoint (e.g., `/block`, `/account/balance`, `/network/status` itself).
2. Each goroutine acquires a pool connection; with `MaxOpenConnections: 100`, the pool is saturated.
3. The `statementTimeout` is 20 s server-side; the HTTP `WriteTimeout` is 10 s client-side — connections are held for up to 10 s per wave.
4. The Kubernetes readiness probe calls `/health/readiness`, which internally calls `/network/status`, which blocks waiting for a pool connection.
5. After 10 s the readiness check times out; after `failureThreshold: 5` consecutive failures (50 s), Kubernetes removes the pod from the Service endpoints.
6. Repeating the flood in waves maintains the exhaustion continuously.

### Impact Explanation
A sustained pool-exhaustion flood causes the Kubernetes readiness probe to fail repeatedly, eventually removing the Rosetta pod from the load balancer. From the perspective of any Coinbase Rosetta client (exchange, indexer, construction workflow), the node becomes unreachable — a network partition. Because all DB-backed endpoints share the same pool with no priority queue, there is no way for health-check traffic to preempt attacker traffic. Severity: **High** (availability impact on a consensus-critical API surface).

### Likelihood Explanation
The attack requires only the ability to send unauthenticated HTTP POST requests to the public Rosetta port (5700). No credentials, no special knowledge of the chain state, and no exploit of memory-safety bugs are needed. A single attacker with ~100 persistent HTTP connections (trivially achievable with `curl --parallel` or any load-testing tool) is sufficient when Traefik middleware is absent. With distributed sources, even the per-IP `inFlightReq` limit is bypassed. The attack is fully repeatable and automatable.

### Recommendation
1. **Add a connection-acquisition timeout** via `db.SetConnMaxIdleTime` and by passing a deadline context to every `GetDbWithContext` call, so blocked goroutines fail fast instead of piling up.
2. **Enforce an application-level concurrency limit** (e.g., a semaphore or `golang.org/x/net/netutil.LimitListener`) inside the Go process, independent of the ingress layer.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) and change `sourceCriterion` on `rateLimit` from `requestHost` to `ipStrategy` so per-IP limits apply.
4. **Separate the health-check DB path** from the request-serving pool (a dedicated single-connection health-check client), so readiness probes cannot be starved by request traffic.

### Proof of Concept
```bash
# Exhaust the pool with 110 concurrent /network/status requests
# (replace NETWORK_IDENTIFIER with the actual value)
seq 110 | xargs -P110 -I{} curl -s -X POST http://<rosetta-host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}' \
  -o /dev/null &

# In parallel, observe the readiness probe failing
watch -n1 'curl -s http://<rosetta-host>:5700/health/readiness'
# Expected: after pool saturation, /health/readiness returns 503 / times out
# After 5 consecutive failures (50 s), kubectl get endpoints shows pod removed
```

func (n *networkAPIService) NetworkStatus(
	ctx context.Context,
	_ *rTypes.NetworkRequest,
) (*rTypes.NetworkStatusResponse, *rTypes.Error) {
	if !n.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	genesisBlock, err := n.RetrieveGenesis(ctx)
	if err != nil {
		return nil, err
	}

	currentBlock, err := n.RetrieveLatest(ctx)
	if err != nil {
		return nil, err
	}

	peers, err := n.addressBookEntryRepo.Entries(ctx)
	if err != nil {
		return nil, err
	}

	return &rTypes.NetworkStatusResponse{
		CurrentBlockIdentifier: currentBlock.GetRosettaBlockIdentifier(),
		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}
```

func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
```
