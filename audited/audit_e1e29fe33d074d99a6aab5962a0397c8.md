### Title
Unauthenticated DB Connection Pool Exhaustion via Arbitrarily Short Base32 Alias in GraphQL `getByAliasAndType()`

### Summary
The `alias` field in the GraphQL `AccountInput` is validated only for character set (`^[A-Z2-7]+$`), with no minimum decoded-byte-length check. A 2-character Base32 string such as `AA` passes validation and decodes to a 1-byte array, which is forwarded directly to a native SQL query against the `entity` table. Because the GraphQL module has no rate limiting, an unauthenticated attacker can flood this endpoint to exhaust the database connection pool and deny service to all consumers of the mirror node.

### Finding Description
**Code path:**

- `graphql/src/main/resources/graphql/account.graphqls` line 90: `alias: String @Pattern(regexp: "^[A-Z2-7]+$")` ŌĆö validates only that every character belongs to the Base32 alphabet; no minimum length is enforced.
- `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java` lines 83ŌĆō85: `decodeBase32` calls `BASE32.decode(base32)` unconditionally and returns the raw byte array with no length check.
- `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java` line 30: `entityRepository.findByAlias(decodeBase32(alias))` passes the decoded bytes directly to the repository.
- `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java` line 13ŌĆō14: executes `select * from entity where alias = ?1 and deleted is not true` as a native query regardless of the byte array length.

**Root cause:** The schema-level `@Pattern` directive only enforces the Base32 alphabet, not a minimum decoded length. Apache Commons Codec's `Base32.decode("AA")` returns `[0x00]` (1 byte), because 2 Base32 characters encode 10 bits, of which the first 8 form a valid byte. There is no post-decode length guard anywhere in the call chain. Every such request acquires a JDBC connection, executes a full table scan on the `alias` column, and releases the connection only after the query completes.

**Why existing checks fail:**
- `@Pattern(regexp: "^[A-Z2-7]+$")` ŌĆö passes for `AA`, `AB`, `BA`, etc.
- `validateOneOf(alias, entityId, evmAddress, id)` ŌĆö only ensures exactly one field is set; does not constrain alias length.
- `MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)` ŌĆö these are GraphQL structural limits, not per-field value limits.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter exists only in the `web3` module; there is no equivalent in the `graphql` module.

### Impact Explanation
Each crafted request forces a database round-trip. With a sufficiently high request rate (easily achievable from a single host or a small botnet), the HikariCP connection pool is saturated. Once exhausted, all GraphQL queries ŌĆö including legitimate account, token, and contract lookups ŌĆö queue indefinitely or fail with connection-timeout errors. Because the REST and importer services may share the same PostgreSQL instance, pool pressure can spill over and degrade those services as well. The mirror node becomes effectively unavailable for read queries, breaking downstream integrations (wallets, explorers, dApps) that depend on it.

### Likelihood Explanation
No authentication, API key, or session token is required to reach the GraphQL endpoint (`/graphql/alpha`). The malicious payload is a standard GraphQL POST request with a 2-character alias string ŌĆö trivially scriptable with `curl` or any HTTP library. The attack is stateless and repeatable at line-rate. A single attacker with modest bandwidth can sustain the flood indefinitely. The absence of any per-IP or global rate limiter in the `graphql` module makes mitigation entirely dependent on external infrastructure (load balancer, WAF), which may not be present in all deployments.

### Recommendation
1. **Enforce a minimum decoded length** in `decodeBase32` or in `getByAliasAndType`: real Hedera aliases are serialized protobuf `Key` messages (ED25519 = 35 bytes, ECDSA = 35 bytes minimum). Reject any decoded byte array shorter than, e.g., 32 bytes before issuing a DB query.
2. **Tighten the schema pattern** to require a minimum Base32 string length that corresponds to the minimum valid alias size (e.g., `^[A-Z2-7]{56,}$` for a 35-byte alias).
3. **Add rate limiting to the GraphQL module** analogous to `ThrottleConfiguration` in `web3`, keyed per source IP.
4. **Return `Optional.empty()` early** when the decoded alias length is outside the expected range, avoiding any DB interaction.

  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: { alias: \"AA\" }) { id } }"}'
# Returns: {"data":{"account":null}}  ŌĆö query executed, connection consumed

# Flood to exhaust connection pool (adjust -n / -c to taste)

# /tmp/payload.json contains the same single-line JSON above

# After pool exhaustion, legitimate queries begin timing out:

  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: { entityId: { num: 3 } }) { id } }"}'
