### Title
Synchronized DB Polling Burst via Unjittered `delaySubscription` in `PollingTopicListener.listen()`

### Summary
`PollingTopicListener.listen()` applies a fixed, unjittered `delaySubscription(interval, scheduler)` to every new subscriber. When an unprivileged attacker opens many gRPC connections simultaneously, all subscriptions fire their first database query at exactly `T + interval`, creating a thundering-herd burst. The developers partially addressed this problem by adding `jitter(0.1)` to `repeatWhen` for subsequent polls, but left the initial delay completely unjittered, meaning the first burst is always perfectly synchronized.

### Finding Description

**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–43.

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // line 39 — fixed delay, NO jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                             // jitter only on repeat, not on first fire
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

**Root cause:** `delaySubscription(interval, scheduler)` uses the same fixed `Duration interval` (default 500 ms, minimum 50 ms per `@DurationMin`) for every subscriber. There is no randomization on this initial delay. The `jitter(0.1)` on `RepeatSpec` only spreads out the *subsequent* poll cycles; it has no effect on the first DB hit.

**Exploit flow:**
1. Attacker opens N simultaneous TCP connections to the public gRPC port (5600).
2. Each connection issues up to `maxConcurrentCallsPerConnection = 5` `subscribeTopic` calls (no authentication required — the only `ServerInterceptor` present, `GrpcInterceptor`, only sets an `EndpointContext` and passes the call through).
3. Each call invokes `PollingTopicListener.listen()`, which schedules a first poll at `now + interval`.
4. Because all N×5 subscriptions were created at approximately the same instant, all N×5 `Flux.defer(() -> poll(context))` publishers fire at approximately the same instant.
5. Each `poll()` call executes `topicMessageRepository.findByFilter(newFilter)` with `pageSize` up to `maxPageSize = 5000` rows.
6. N×5 simultaneous full-page DB queries arrive at the database at the same moment.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` limits calls *per connection*, not total connections. An attacker opens as many connections as needed.
- No global subscription count cap exists anywhere in the gRPC stack.
- No per-IP or per-client rate limiting exists for gRPC (the `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiting is scoped exclusively to the `web3` module).
- The `Schedulers.boundedElastic()` scheduler queues tasks but does not reject them; it will eventually execute all N×5 queries. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
When `ListenerType.POLL` is active, every active subscription is an independent cold publisher that issues its own DB query on every tick. A synchronized burst of N×5 queries, each fetching up to 5,000 rows, can saturate the database connection pool and exhaust I/O capacity, causing query timeouts and cascading failures for all legitimate subscribers. Because the mirror node gRPC service is a public infrastructure component (ingress enabled, no authentication), this constitutes a non-network-based DoS against the database tier. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no prior access. The gRPC endpoint is publicly reachable. A single attacker machine can open hundreds of TCP connections and issue thousands of `subscribeTopic` RPCs in milliseconds using any standard gRPC client or `grpcurl`. The attack is trivially repeatable: after each `interval` cycle the burst re-synchronizes because the fixed `withFixedDelay(interval)` on `repeatWhen` re-aligns all subscribers that started together. The only prerequisite is that the operator has configured `type: POLL`; this is a documented, supported configuration option. [7](#0-6) [8](#0-7) 

### Recommendation
1. **Add jitter to the initial delay.** Replace the fixed `delaySubscription(interval, scheduler)` with a randomized delay in the range `[0, interval]`:
   ```java
   Duration jitteredDelay = Duration.ofMillis(
       (long)(Math.random() * interval.toMillis()));
   .delaySubscription(jitteredDelay, scheduler)
   ```
2. **Cap total concurrent POLL subscriptions.** Introduce a global `AtomicInteger` counter in `PollingTopicListener` and reject (with `RESOURCE_EXHAUSTED`) subscriptions that exceed a configurable maximum.
3. **Add gRPC-level rate limiting.** Implement a `ServerInterceptor` that enforces per-IP or global subscription rate limits, analogous to the `ThrottleManagerImpl` used in the web3 module.
4. **Prefer `SHARED_POLL` or `REDIS` over `POLL`.** `SharedPollingTopicListener` uses a single shared `Flux` with `.share()`, so N subscribers produce exactly one DB query per tick regardless of subscriber count. [9](#0-8) 

### Proof of Concept
```bash
# Open 200 simultaneous subscribeTopic streams at the same instant
# (requires grpcurl and a topic that exists on the target)
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```
All 200 streams (and up to 1,000 if spread across 200 connections × 5 calls each) will schedule their first `poll()` at `T + 500ms`. At that moment, 200–1,000 simultaneous `SELECT` queries each requesting up to 5,000 rows hit the database. Monitor DB connection pool exhaustion and query latency to confirm the burst.

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
