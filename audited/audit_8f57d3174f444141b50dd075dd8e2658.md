### Title
Unbounded Redis Channel Subscription Creation via Unique TopicId Flooding in `subscribeTopic()`

### Summary
An unprivileged external user can open gRPC subscriptions to many distinct valid `topicId` values simultaneously. Because `RedisTopicListener.getSharedListener()` creates one Redis channel subscription per unique topic and stores it in an unbounded `ConcurrentHashMap`, an attacker can grow this map without limit, exhausting Redis pub/sub channel capacity and server-side memory on the gRPC node. The per-connection call limit of 5 is trivially bypassed by opening many TCP connections.

### Finding Description

**Code path:**

`subscribeTopic()` (line 59â€“92, `TopicMessageServiceImpl.java`) calls `topicListener.listen(newFilter)` via `incomingMessages()` (line 120). This reaches `SharedTopicListener.listen()` (line 21â€“26, `SharedTopicListener.java`), which calls `getSharedListener(filter)` on `RedisTopicListener`.

`RedisTopicListener.getSharedListener()` (lines 59â€“62):
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`getTopic()` (lines 64â€“66) produces a channel name `topic.{id}` per unique topicId:
```java
private Topic getTopic(TopicMessageFilter filter) {
    return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
}
```

`subscribe()` (lines 68â€“80) calls `container.flatMapMany(r -> r.receive(...))` â€” adding a new pub/sub channel subscription to the single `ReactiveRedisMessageListenerContainer` for every unique topic. The `topicMessages` `ConcurrentHashMap` (line 43) is never bounded.

**Root cause:** There is no cap on the number of unique topics that can be subscribed to, no rate limiting on `subscribeTopic()`, and no global limit on total active subscriptions. The `computeIfAbsent` pattern means each unique `topicId` permanently occupies a slot in the map and a Redis channel subscription until all subscribers disconnect.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14): limits calls *per TCP connection* only. An attacker opens N connections Ă— 5 calls = 5N concurrent subscriptions across N connections â€” no global ceiling.
- `checkTopicExists = true` (`GrpcProperties`, line 19): requires a valid topic entity. On mainnet/testnet, millions of valid topic IDs exist, so this is not a meaningful barrier.
- `subscriberCount` gauge (line 48): monitoring only, enforces no limit.
- The `.share()` on the returned `Flux` (line 79) only deduplicates subscribers to the *same* topic; different topics each get their own Redis subscription.
- `unsubscribe()` (lines 82â€“85) only fires on cancel/complete â€” while the attacker holds connections open, entries remain.

### Impact Explanation
Each unique topic subscription adds a channel to the single `ReactiveRedisMessageListenerContainer`. Redis tracks all subscribed channels server-side; at scale this consumes Redis memory and CPU for subscription management. The `topicMessages` map on the gRPC server grows without bound, consuming heap. If Redis hits its `client-output-buffer-limit` for the pub/sub connection or the gRPC JVM runs out of heap, the `RedisTopicListener` fails. Because the container is a singleton (`.cache()`, line 55), its failure propagates to *all* active subscribers on that node â€” a single gRPC node can be taken down, and if multiple nodes share the same Redis instance, the impact widens.

### Likelihood Explanation
No authentication is required to call the gRPC `subscribeTopic` endpoint. The attacker needs only a list of valid topic IDs (publicly available via the mirror node REST API) and a gRPC client. Opening thousands of connections with 5 subscriptions each to distinct topics is trivially scriptable. The attack is repeatable and requires no special privileges or insider knowledge.

### Recommendation
1. **Global subscription cap:** Add an `AtomicLong` total-subscription counter with a configurable maximum; reject new subscriptions when the limit is reached.
2. **Per-IP / per-client rate limiting:** Enforce a maximum number of concurrent subscriptions per source IP at the gRPC interceptor layer.
3. **Bound the `topicMessages` map:** Use a size-limited cache (e.g., Caffeine with eviction) instead of an unbounded `ConcurrentHashMap`.
4. **Cap unique active Redis channels:** Reject `getSharedListener()` calls when `topicMessages.size()` exceeds a configured threshold.
5. **Connection-level limits:** Configure Netty's `maxConnectionsPerIp` in addition to `maxConcurrentCallsPerConnection`.

### Proof of Concept
```python
import grpc, threading, time
# Assume proto stubs are available
from mirror_node_pb2_grpc import ConsensusServiceStub
from mirror_node_pb2 import ConsensusTopicQuery

TARGET = "mirror.mainnet.example.com:5600"
# Collect N valid topic IDs from REST API: GET /api/v1/topics?limit=1000
TOPIC_IDS = list(range(1, 5001))  # 5000 distinct valid topic IDs

def flood(topic_id):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)
    req = ConsensusTopicQuery()
    req.topicID.topicNum = topic_id
    try:
        for _ in stub.subscribeTopic(req):
            pass  # hold connection open
    except Exception:
        pass

threads = [threading.Thread(target=flood, args=(t,)) for t in TOPIC_IDS]
for t in threads:
    t.start()
time.sleep(300)  # hold all subscriptions open for 5 minutes
```
Each thread subscribes to a distinct topic, causing `RedisTopicListener` to register 5000 separate Redis channel subscriptions and populate the `topicMessages` map with 5000 entries. Monitor Redis with `CLIENT INFO` / `PUBSUB CHANNELS` and gRPC node heap usage to observe exhaustion.
