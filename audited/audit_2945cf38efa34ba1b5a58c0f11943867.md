### Title
Unbounded Indefinite gRPC Topic Subscription Enables Resource Exhaustion and Persistent Message Surveillance

### Summary
The `subscribeTopic()` method in `ConsensusController` accepts unauthenticated gRPC connections with no maximum subscription duration, no per-client connection limit, and no server-side timeout. When a client omits `consensusEndTime` and `limit`, the server-side reactive pipeline is wired to `Flux.never()` for its termination signal, causing the subscription to persist indefinitely. An attacker can open many connections and saturate server resources while passively observing all messages on any topic.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) converts the incoming `ConsensusTopicQuery` into a `TopicMessageFilter` via `toFilter()` (lines 55–73) and delegates to `TopicMessageServiceImpl.subscribeTopic()`. [1](#0-0) 

In `toFilter()`, `endTime` is only set if `query.hasConsensusEndTime()` is true, and `limit` defaults to 0 — both are entirely optional with no server-enforced defaults. [2](#0-1) 

In `TopicMessageFilter`, `endTime` is a nullable `Long` with no `@NotNull` constraint and no maximum-duration validation. `limit` has only a `@Min(0)` constraint, meaning 0 (unlimited) is fully valid. [3](#0-2) 

In `TopicMessageServiceImpl.subscribeTopic()`, the termination signal `pastEndTime()` explicitly returns `Flux.never()` when `endTime == null`, and the `flux.take(limit)` branch is skipped when `limit == 0`. The subscription therefore never terminates server-side. [4](#0-3) [5](#0-4) 

Each live subscription registers a `topicListener.listen()` call (Redis or polling), a periodic DB retriever, a `TopicContext` object, and increments `subscriberCount`. These resources are only released on client disconnect. [6](#0-5) 

**Root cause:** The server assumes clients will self-terminate (via `endTime` or `limit`), but enforces neither. There is no server-side maximum subscription duration, no authentication, and no per-IP connection limit.

**Why existing checks fail:**

The only server-side connection control is `maxConcurrentCallsPerConnection = 5`, which limits calls per TCP connection but places no cap on the total number of connections or connections per IP. [7](#0-6) [8](#0-7) 

The only `ServerInterceptor` in the gRPC module is in the test source tree and only sets an endpoint-tracking context — no authentication or rate-limiting interceptor exists in production code. [9](#0-8) 

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to the gRPC service.

The `retriever.timeout = 60s` is a between-message idle timeout, not a subscription duration limit; it does not fire while messages are flowing. [10](#0-9) 

### Impact Explanation

Each indefinite subscription holds: a Reactor subscription chain, a Redis listener slot (or a DB polling thread), a `TopicContext` heap object, and a DB connection lease during historical retrieval. An attacker opening hundreds of connections × 5 calls each can exhaust the thread pool, DB connection pool (max 800 connections per the Helm values), and Redis listener capacity, causing legitimate subscribers to be rejected or starved. Additionally, since there is no topic-level authorization, any attacker can subscribe to any topic and passively receive every message published to it for an unlimited duration — a persistent confidentiality breach for topics intended to be private.

### Likelihood Explanation

No special privileges, credentials, or network position are required. Any client with TCP access to port 5600 can exploit this. The attack is trivially scriptable: open N connections, send a `subscribeTopic` request with no `endTime` and `limit=0` on each. It is repeatable and persistent. The only external barrier is network-level firewall rules, which are deployment-specific and not enforced by the application itself.

### Recommendation

1. **Enforce a server-side maximum subscription duration**: Add a configurable `maxSubscriptionDuration` property to `GrpcProperties` and apply `flux.timeout(maxSubscriptionDuration)` or `flux.take(maxSubscriptionDuration)` in `TopicMessageServiceImpl.subscribeTopic()` when the client does not supply an `endTime`.
2. **Enforce a global concurrent-subscriber cap**: Reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
3. **Add per-IP connection limiting**: Configure Netty's `maxConnectionsPerIp` or add a gRPC `ServerInterceptor` that tracks and limits active calls per remote address.
4. **Add authentication/authorization**: Require a token or API key via a `ServerInterceptor` before allowing subscriptions, especially for sensitive topics.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc-host:5600"
TOPIC_NUM = 1234  # any valid topic

channels = []
for i in range(200):  # open 200 connections
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM)
        # no consensusEndTime, no limit → indefinite subscription
    )
    # Open 5 concurrent calls per connection (maxConcurrentCallsPerConnection limit)
    for _ in range(5):
        stub.subscribeTopic(query)  # non-blocking streaming call
    channels.append(ch)

# 200 connections × 5 calls = 1000 indefinite subscriptions
# Server resources (threads, DB connections, Redis slots) are now saturated
# All topic messages are received by the attacker indefinitely
```

    private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
        final var filter = TopicMessageFilter.builder().limit(query.getLimit());

        if (query.hasTopicID()) {
            filter.topicId(EntityId.of(query.getTopicID()));
        }

        if (query.hasConsensusStartTime()) {
            long startTime = convertTimestamp(query.getConsensusStartTime());
            filter.startTime(startTime);
        }

        if (query.hasConsensusEndTime()) {
            long endTime = convertTimestamp(query.getConsensusEndTime());
            filter.endTime(endTime);
        }

        return filter.build();
    }
