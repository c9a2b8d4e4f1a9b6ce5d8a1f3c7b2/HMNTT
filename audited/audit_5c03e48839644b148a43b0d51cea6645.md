### Title
Unbounded `ItemizedTransferSlice.Scan()` JSON Unmarshal Before Response Truncation Enables CPU Amplification via `/block` Endpoint

### Summary
In `FindBetween()`, GORM calls `ItemizedTransferSlice.Scan()` → `json.Unmarshal()` for every row returned across all batch iterations, regardless of how many transactions will ultimately be returned to the caller. The `maxTransactionsInBlock` guard in `Block()` truncates the response only **after** `FindBetween()` has already fetched and unmarshaled every transaction in the block. An unauthenticated caller can repeatedly request a block containing many transactions with large `itemized_transfer` JSONB payloads, forcing the server to perform the full unmarshal work on every request while returning only a fraction of the data.

### Finding Description

**Exact code path:**

1. `rosetta/app/services/block_service.go` lines 56–66: `Block()` calls `FindBetween()` unconditionally, then checks `maxTransactionsInBlock` only after the call returns: [1](#0-0) 

2. `rosetta/app/persistence/transaction.go` lines 124–147: `FindBetween()` loops in batches of `batchSize = 2000`, issuing repeated DB queries until all rows in the timestamp range are fetched: [2](#0-1) 

3. For every row returned by every batch query, GORM invokes `ItemizedTransferSlice.Scan()` on the `itemized_transfer` JSONB column: [3](#0-2) 

**Root cause:** `Scan()` calls `json.Unmarshal(bytes, &result)` on the raw JSONB bytes with no size guard. The response-level cap (`maxTransactionsInBlock`) is applied in `Block()` after `FindBetween()` has already completed all DB round-trips and all unmarshal operations. There is no query-level `LIMIT` tied to `maxTransactionsInBlock`, no per-request rate limit, and no cap on the byte size of `itemized_transfer` per row.

**Failed assumption:** The design assumes that `maxTransactionsInBlock` bounds server-side work. It does not — it only bounds the response payload.

### Impact Explanation
Every `/block` request for a block with N transactions causes N calls to `json.Unmarshal()` on potentially large JSONB arrays, plus N×(batch iterations) DB round-trips. If a block contains, say, 10 000 transactions each with a 10 KB `itemized_transfer` array, a single request forces ~100 MB of JSON parsing. Because the endpoint is unauthenticated and the work is proportional to block content rather than response size, an attacker can sustain elevated CPU and memory pressure with a small number of concurrent requests targeting the same large block. This satisfies the ≥30% resource-increase threshold without brute-force volume.

### Likelihood Explanation
The Rosetta `/block` endpoint requires no credentials. On Hedera mainnet, blocks with hundreds to thousands of transactions exist naturally (token-transfer-heavy blocks, DeFi activity). An attacker needs only to identify such a block (trivially done by scanning block metadata) and issue repeated `POST /block` requests. No special privileges, no on-chain action, and no exploit tooling beyond a standard HTTP client are required. The attack is fully repeatable and stateless.

### Recommendation
1. **Apply the transaction limit at the query layer**: pass `maxTransactionsInBlock` as a `LIMIT` into `selectTransactionsInTimestampRangeOrdered` (or as a `.Limit()` call on the GORM query) so the DB never returns more rows than the response will use.
2. **Cap `itemized_transfer` byte size at the DB level**: add a `pg_column_size` guard or a `CHECK` constraint, or truncate the JSONB in the SQL projection.
3. **Add per-IP or per-endpoint rate limiting** at the HTTP layer so that repeated requests to the same block are throttled.
4. **Short-circuit the batch loop** once the accumulated transaction count reaches `maxTransactionsInBlock`.

### Proof of Concept
1. Identify a Hedera block (by index or hash) known to contain a large number of transactions with token transfers (these produce large `itemized_transfer` arrays).
2. Send repeated unauthenticated `POST /block` requests:
   ```
   POST /block
   {"network_identifier": {...}, "block_identifier": {"index": <large_block>}}
   ```
3. Observe via server metrics (CPU, memory, DB query duration) that each request triggers full batch iteration over all transactions in the block and a corresponding spike in `json.Unmarshal` CPU time, while the response contains only `maxTransactionsInBlock` entries.
4. Sustain 5–10 concurrent such requests to push CPU utilization ≥30% above baseline without any authentication or on-chain action.

func (i *ItemizedTransferSlice) Scan(value any) error {
	bytes, ok := value.([]byte)
	if !ok {
		return errors.New(fmt.Sprint("Failed to unmarshal JSONB value", value))
	}

	result := ItemizedTransferSlice{}
	err := json.Unmarshal(bytes, &result)
	*i = result
	return err
}
