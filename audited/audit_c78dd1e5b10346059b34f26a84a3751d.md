### Title
Unauthenticated Resource Exhaustion via Maximum-Complexity Crypto Allowance Queries with No Rate Limiting

### Summary
The `GET /api/v1/accounts/:idOrAliasOrEvmAddress/allowances/crypto` endpoint accepts up to 100 repeated `spender.id=eq:X` parameters per request (the configured `maxRepeatedQueryParameters` default) and a `limit` of up to 100, generating a large `IN (...)` SQL clause against the `crypto_allowance` table. No IP-based or per-client rate limiting exists in the Node.js REST API layer, allowing any unauthenticated external user to flood the endpoint with maximum-complexity queries across distinct account IDs, sustaining elevated database load without triggering any brute-force detection.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 17 registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/controllers/cryptoAllowanceController.js` `getAccountCryptoAllowances` (lines 76–98) calls `buildAndValidateFilters` then `extractCryptoAllowancesQuery`, which collects all `spender.id=eq:X` values into `spenderInValues` and passes them to the service: [2](#0-1) [3](#0-2) 

`rest/service/cryptoAllowanceService.js` `getAccountAllowancesQuery` (lines 19–30) builds the final SQL as `SELECT * FROM crypto_allowance WHERE owner=$1 AND spender IN ($2,...,$N) AND amount > 0 ORDER BY spender DESC LIMIT $M`, executing it directly against the database: [4](#0-3) 

**Root cause — failed assumption:** The system assumes that the `maxRepeatedQueryParameters` cap (default **100**) is sufficient to bound per-request cost, and that external infrastructure (e.g., Traefik) will enforce rate limits. However:

1. The REST API server (`rest/server.js`) applies no IP-based or per-client rate-limiting middleware to the accounts route — only logging, optional auth (for custom response limits), metrics, and optional Redis cache: [5](#0-4) 

2. The `qs` parser enforces `arrayLimit: config.query.maxRepeatedQueryParameters` (100) with `throwOnLimitExceeded: true`, meaning exactly 100 `spender.id` values per request are accepted and processed: [6](#0-5) 

3. The throttling code (`ThrottleManagerImpl`, `ThrottleConfiguration`) applies exclusively to the **web3 Java** service, not the Node.js REST API: [7](#0-6) 

4. The `maxRepeatedQueryParameters` default of 100 is documented and confirmed: [8](#0-7) 

**Exploit flow:**

An attacker sends a continuous stream of requests such as:
```
GET /api/v1/accounts/0.0.1/allowances/crypto?spender.id=eq:1&spender.id=eq:2&...&spender.id=eq:100&limit=100
GET /api/v1/accounts/0.0.2/allowances/crypto?spender.id=eq:1&spender.id=eq:2&...&spender.id=eq:100&limit=100
GET /api/v1/accounts/0.0.3/allowances/crypto?...
```

Each request: (a) passes all validation, (b) generates a `SELECT * FROM crypto_allowance WHERE owner=$1 AND spender IN ($2,...,$101) AND amount > 0 ORDER BY spender DESC LIMIT $102` query, (c) is distinct (different account ID) so Redis response caching does not help, and (d) is never blocked because there is no rate limiter.

### Impact Explanation
Each request forces a non-trivial indexed scan on the `crypto_allowance` table with a 100-element `IN` clause and a `LIMIT 100` result set. The database connection pool is bounded at 10 connections by default (`db.pool.maxConnections: 10`), so a sustained flood of concurrent maximum-complexity requests saturates the pool and increases query queue depth, CPU, and I/O on the database node. This can increase processing node resource consumption well above 30% compared to baseline, degrading response times for all other API consumers. No authentication, no special account state, and no prior knowledge beyond valid account IDs (which are enumerable) is required. [9](#0-8) 

### Likelihood Explanation
Any external user with network access to the REST API can execute this attack. Valid account IDs are publicly enumerable via the same API. The attack requires no credentials, no special tooling beyond a standard HTTP client, and is trivially scriptable. It is repeatable indefinitely because no detection or blocking mechanism exists at the application layer. The only mitigation would be external infrastructure (e.g., Traefik rate limiting), which is not guaranteed to be deployed in all configurations and is not enforced by the application itself.

### Recommendation
1. **Add application-layer rate limiting** to the Node.js REST API (e.g., `express-rate-limit` keyed by IP) for the `/accounts/*/allowances/crypto` route and other high-cost endpoints.
2. **Reduce `maxRepeatedQueryParameters`** for `spender.id` specifically on this endpoint (e.g., cap at 10–25 instead of 100), or add a separate per-endpoint cap for `IN`-clause parameters.
3. **Add a query cost guard**: reject requests where `spender.id` count × `limit` exceeds a configurable threshold.
4. **Ensure Traefik (or equivalent) rate limiting** is mandatory in all deployment configurations, not optional.

### Proof of Concept
```bash
# Construct a URL with 100 spender.id=eq:X parameters and limit=100
PARAMS=$(python3 -c "print('&'.join([f'spender.id=eq:{i}' for i in range(1,101)]))")

# Flood the endpoint with distinct account IDs (no rate limiting will stop this)
for ACCT in $(seq 1 10000); do
  curl -s "http://<mirror-node>:5551/api/v1/accounts/0.0.${ACCT}/allowances/crypto?${PARAMS}&limit=100" &
done
wait
```

Each concurrent request generates a `SELECT * FROM crypto_allowance WHERE owner=$1 AND spender IN ($2,...,$101) AND amount > 0 ORDER BY spender DESC LIMIT $102` query. With the default pool of 10 DB connections, this saturates the pool and causes measurable resource consumption increase on both the Node.js process and the PostgreSQL database, with no blocking or detection triggered.
