### Title
Unauthenticated Timestamp-Triggered Double DB Query Enables DB Load Amplification on `GET /tokens/:tokenId/balances`

### Summary
Any unauthenticated caller can supply a `timestamp` query parameter to `GET /tokens/:tokenId/balances`, unconditionally forcing `extractSqlFromTokenBalancesRequest()` to execute an extra `account_balance` lookup via `balances.getAccountBalanceTimestampRange()` before the main `token_balance` query. Because the REST API has no rate-limiting middleware, an attacker can sustain a high request rate and double the database query load compared to requests without a timestamp filter, degrading service for all users.

### Finding Description

**Exact code path:**

In `rest/tokens.js` lines 638–666, whenever `tsConditions.length > 0` (i.e., any `timestamp` query parameter is present), the function unconditionally awaits `balances.getAccountBalanceTimestampRange()`:

```js
// rest/tokens.js:638-640
if (tsConditions.length) {
  const tsQuery = tsConditions.join(' and ');
  const {lower, upper} = await balances.getAccountBalanceTimestampRange(tsQuery, tsParams);
``` [1](#0-0) 

`getAccountBalanceTimestampRange()` in `rest/balances.js` lines 167–196 issues a real DB query against the `account_balance` table every time it is called — there is no caching:

```js
// rest/balances.js:182-189
const query = `
  select consensus_timestamp
  from account_balance
  where ${condition}
  order by consensus_timestamp desc
  limit 1`;
const {rows} = await pool.queryQuietly(query, params);
``` [2](#0-1) 

After that returns, the main `token_balance` query is also executed in `getTokenBalances()`:

```js
// rest/tokens.js:724
const {rows} = await pool.queryQuietly(query, params);
``` [3](#0-2) 

**Root cause:** The timestamp range resolution is performed eagerly and unconditionally on every request with no result caching, no request-level deduplication, and no rate limiting.

**Why existing checks fail:**

- The `limit` parameter is capped at `responseLimit.max` (default 100) by `getLimitParamValue()`, which only bounds result set size, not request frequency. [4](#0-3) 

- The `authHandler` middleware only sets a custom `limit` for authenticated users; it does not block or throttle unauthenticated requests. [5](#0-4) 

- `rest/server.js` registers no rate-limiting middleware for the REST API. The throttle infrastructure found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belongs exclusively to the `web3` Spring Boot module and does not apply here. [6](#0-5) 

### Impact Explanation

An attacker sustaining N requests/second with `?timestamp=lte:<any_valid_ts>` causes 2N database queries per second instead of N, doubling DB CPU and I/O load. Because the `account_balance` table is large and partitioned by month, even the "lightweight" range-lookup query is non-trivial under load. Sustained amplification degrades query latency for all users of the mirror node, potentially causing timeouts and service unavailability. Severity is medium (griefing/DoS, no on-chain economic damage).

### Likelihood Explanation

Preconditions: none — no account, no token, no authentication required. The attacker needs only a valid (or even invalid) token ID and any timestamp value. The attack is trivially scriptable with `curl` or any HTTP load tool. It is repeatable indefinitely and requires no special knowledge beyond reading the public API documentation.

### Recommendation

1. **Add rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied globally or specifically to expensive endpoints like `/tokens/:tokenId/balances`.
2. **Cache the timestamp range result**: since `getAccountBalanceTimestampRange` is deterministic for a given `(tsQuery, tsParams)` tuple and balance snapshots are ingested periodically (not per-request), a short-lived in-process or Redis cache (e.g., TTL = 15s) would eliminate the extra DB round-trip for repeated or concurrent identical timestamp queries.
3. **Require authentication** for high-cost query patterns (timestamp-filtered balance queries), or enforce stricter per-IP connection limits at the infrastructure layer (nginx/HAProxy).

### Proof of Concept

```bash
# No authentication required. Flood with timestamp-filtered requests.
TOKEN_ID="0.0.1234"
TIMESTAMP="lte:9999999999.999999999"

# Each request triggers 2 DB queries instead of 1
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/tokens/${TOKEN_ID}/balances?timestamp=${TIMESTAMP}&limit=100" &
done
wait
```

Each concurrent request causes:
1. `SELECT consensus_timestamp FROM account_balance WHERE account_id=$1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3 ORDER BY consensus_timestamp DESC LIMIT 1` (extra query, `rest/balances.js:182`)
2. `SELECT DISTINCT ON (ti.account_id) ... FROM token_balance AS ti WHERE ... LIMIT 100` (main query, `rest/tokens.js:649`)
