### Title
Unbounded Historical Timestamp Scan DoS on Hook Storage Change Endpoint

### Summary
An unauthenticated user can supply only an upper-bound timestamp (e.g., `timestamp=lte:9223372036.854775807`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing `Bound.of()` to produce `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`. This forces the `hook_storage_change` query to scan every time partition in the table for the given hook, with no lower bound to enable partition pruning, enabling a sustained denial-of-service via repeated requests.

### Finding Description

**Exact code path:**

`HooksController.java` line 186 calls `Bound.of(timestamps, TIMESTAMP, ...)`: [1](#0-0) 

`Bound.of()` in `Bound.java` lines 169–182 passes `primarySortField=false`, so the range-validity check (`adjustedLower > adjustedUpper`) is **skipped**: [2](#0-1) 

When only an upper-bound timestamp is supplied (no lower bound), `lower` remains `null`. `getAdjustedLowerRangeValue()` returns `0` unconditionally: [3](#0-2) 

`adjustUpperBound()` returns `Long.MAX_VALUE` when `upper.operator() == LTE` and `upper.value() == Long.MAX_VALUE`: [4](#0-3) 

`HookServiceImpl.getHookStorageChange()` lines 98–99 passes these directly to the repository: [5](#0-4) 

The repository query uses `consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound` with `DISTINCT ON (key)`: [6](#0-5) 

**Root cause:** `Bound.of()` is called with `primarySortField=false`, disabling the lower/upper range validation. When only an upper bound is provided, `lower` is `null` and `getAdjustedLowerRangeValue()` silently returns `0`. No separate validation enforces a minimum lower bound or requires both bounds to be present for the timestamp parameter on this endpoint.

**Why `timestamp=lte:9223372036.854775807` works:** `TimestampParameter` accepts up to 17 digits for seconds and up to 9 digits for nanoseconds. `9223372036.854775807` parses as `9223372036 * 1_000_000_000 + 854775807 = Long.MAX_VALUE` in nanoseconds — exactly within the regex limit: [7](#0-6) 

**Why the `DISTINCT ON` scan is expensive:** The `hook_storage_change` table is partitioned by `consensus_timestamp`. With `BETWEEN 0 AND Long.MAX_VALUE`, partition pruning is entirely ineffective — every partition must be scanned. The `DISTINCT ON (key) ORDER BY key, consensus_timestamp DESC` merge across all partitions cannot be satisfied by a single index range scan; PostgreSQL must gather and merge results from all partitions before applying `DISTINCT ON`. [8](#0-7) 

### Impact Explanation

Any unauthenticated caller can force the mirror node to execute a full cross-partition historical scan of `hook_storage_change` for any `(owner_id, hook_id)` pair. The `limit` cap of 100 rows limits the response size but does **not** limit the amount of data the database engine must read to satisfy `DISTINCT ON (key)` — the DB must still scan all matching rows across all partitions before it can return the top-N. Repeated requests from one or more clients can saturate database I/O, degrade query performance for all other users, and potentially exhaust connection pool resources. The hook storage change table records gossip-prevention storage mutations, so sustained scanning also reveals the existence and volume of historical state changes for any hook.

### Likelihood Explanation

The endpoint requires no authentication or authorization. The exploit requires a single HTTP GET request with one crafted query parameter. It is trivially repeatable in a loop. Any external user who can reach the REST API can execute this attack. The only precondition is knowing a valid `(ownerId, hookId)` pair, which is discoverable via the public `GET /api/v1/accounts/{ownerId}/hooks` endpoint.

### Recommendation

1. **Require a lower bound when a timestamp upper bound is supplied** in `hookStorageChangeRequest()`: if only an upper bound is present, either reject the request or default the lower bound to a recent window (e.g., `upper - maxWindowNs`).
2. **Enforce a maximum timestamp range window** for historical queries on `hook_storage_change`, similar to how `bindTimestampRange` works in the Node.js REST layer.
3. **Add range validation in `Bound.of()`** (or in the controller) even when `primarySortField=false`, specifically for the timestamp parameter on this endpoint.
4. **Add authentication/authorization** to the hook storage endpoint so only privileged callers can query historical state.

### Proof of Concept

```
# Step 1: Discover a valid (ownerId, hookId) pair (public endpoint)
GET /api/v1/accounts/0.0.123/hooks

# Step 2: Trigger full historical scan with no lower bound
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:9223372036.854775807

# Step 3: Repeat in a loop to sustain DB load
while true; do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:9223372036.854775807" > /dev/null
done
```

**Observed effect:** Each request causes the database to execute `SELECT DISTINCT ON (key) ... FROM hook_storage_change WHERE owner_id=123 AND hook_id=1 AND consensus_timestamp BETWEEN 0 AND 9223372036854775807` scanning all time partitions, with no partition pruning possible.
