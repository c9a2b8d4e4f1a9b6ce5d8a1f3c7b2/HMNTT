### Title
Unauthenticated Forced `entity`+`entity_history` UNION ALL via Guaranteed-Empty Balance Snapshot Timestamp

### Summary
An unprivileged attacker can send `GET /accounts/:id?timestamp=gte:1&timestamp=lte:1` to `getOneAccount()`. Because no balance snapshot exists at nanosecond 1 of the Unix epoch, `getAccountBalanceTimestampRange()` always returns `{}`, which unconditionally sets `accountBalanceQuery.forceUnionEntityHistory = true`. This forces every subsequent DB query for that request to use a `UNION ALL` across `entity`+`entity_history` and `entity_stake`+`entity_stake_history` instead of the cheaper single-table path, consuming more DB resources than a normal request. The trigger is deterministic, requires no privileges, and is repeatable at arbitrary frequency.

### Finding Description

**Step 1 — Input passes all validation.**

`acceptedSingleAccountParameters` includes `TIMESTAMP`. [1](#0-0) 

`isValidTimestampParam` accepts any 1–10 digit integer, so `1` is valid. [2](#0-1) 

**Step 2 — Range validation is intentionally disabled for this endpoint.**

`parseTimestampFilters` is called with `validateRange = false` (6th argument), so the `maxTimestampRangeNs` guard that would reject tiny or huge ranges is never reached. [3](#0-2) [4](#0-3) 

For `gte:1&lte:1`, `difference = 1n`, `range = Range(1n, 1n, '[]')`. `range.isEmpty()` is `false` (a single-point range is not empty), so no `NotFoundError` is thrown.

**Step 3 — `getOptimizedTimestampRange` produces a valid but ancient window.**

`lowerBound = 1n`, `upperBound = 1n`. `effectiveUpperBound = min(1n, nowInNs) = 1n`. `optimalLowerBound = getFirstDayOfMonth(1n, -1)` resolves to a negative nanosecond value (December 1969), so `lowerBound` stays `1n`. The function returns `{lowerBound: 1n, upperBound: 1n, neParams: []}`. [5](#0-4) 

**Step 4 — DB query for treasury balance snapshot returns 0 rows.**

```sql
SELECT consensus_timestamp FROM account_balance
WHERE account_id = <treasury> AND consensus_timestamp >= 1 AND consensus_timestamp <= 1
ORDER BY consensus_timestamp DESC LIMIT 1
```
No balance snapshot exists at nanosecond 1 of the epoch. `rows.length === 0` → returns `{}`. [6](#0-5) 

**Step 5 — `forceUnionEntityHistory` is set unconditionally.**

Back in `getOneAccount`, `upper` is `undefined` (destructuring `{}`), so the `else` branch fires: [7](#0-6) 

**Step 6 — Two UNION ALL queries are forced.**

`needHistoricalBalanceInfo = true` because `forceUnionEntityHistory = true`. The entity table becomes:
```sql
(SELECT * FROM entity AS e WHERE ... UNION ALL SELECT * FROM entity_history AS e WHERE ... ORDER BY timestamp_range DESC LIMIT 1)
``` [8](#0-7) 

Since `isHistorical = true` (because `timestampFilters.length > 0`), `getEntityStakeQuery` also returns a UNION ALL over `entity_stake` + `entity_stake_history`: [9](#0-8) 

A normal request (no timestamp filter) queries only `entity` and `entity_stake` directly.

### Impact Explanation

Each crafted request forces:
1. One extra DB round-trip to `account_balance` (guaranteed to return empty).
2. Two extra index scans (`entity_history`, `entity_stake_history`) via UNION ALL instead of the single-table path.

On a production node where `entity_history` accumulates millions of rows over time, the per-request overhead is measurably higher. An attacker flooding the endpoint at high concurrency degrades DB connection pool availability and query throughput for all users. No authentication or special account state is required; any valid account ID works.

### Likelihood Explanation

The trigger is a single HTTP GET with two query parameters. It is deterministic — the same two parameters always produce the expensive path because no balance snapshot will ever exist at nanosecond 1. The REST API has no built-in per-IP rate limiting; the optional Traefik middleware is not enforced by default. [10](#0-9) 

Any external user who can reach the public mirror node REST API can exploit this repeatedly.

### Recommendation

1. **Enforce a minimum timestamp floor in `getOneAccount`**: reject timestamp values below a reasonable epoch (e.g., Hedera mainnet genesis ~`1569514800000000000` ns). This prevents the guaranteed-empty balance snapshot path from being triggered by design.
2. **Enable `validateRange` for the single-account endpoint**: change the `parseTimestampFilters` call at line 413 to pass `validateRange = true`, so a 1-nanosecond range is rejected with a 400 error.
3. **Add per-IP rate limiting** to the REST API at the application layer (not just as optional Traefik middleware).
4. **Short-circuit early**: if `getOptimizedTimestampRange` returns an `upperBound` that is far in the past (e.g., before Hedera genesis), return a 400 or 404 immediately rather than issuing a DB query that is guaranteed to return empty.

### Proof of Concept

```bash
# Any valid account ID works; 0.0.2 (treasury) is always present
curl "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=gte:1&timestamp=lte:1"

# Expected: 200 OK with balance from entity_history UNION ALL path
# (or 404 if account not found in that range, but the expensive DB path is still taken)

# Flood to amplify DB load:
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=gte:1&timestamp=lte:1" &
done
wait
```

Each request issues the treasury `account_balance` probe query (returns empty), then executes the `entity UNION ALL entity_history` + `entity_stake UNION ALL entity_stake_history` path, consuming roughly 2–3× the DB resources of a normal single-account lookup.

const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
};
```
