### Title
Zero-pageSize Unbounded Database Query in `PollingTopicListener.poll()` When Delivered Count Reaches Filter Limit

### Summary
In `PollingTopicListener.poll()`, when the number of messages delivered equals `filter.getLimit()`, the computed `limit` becomes zero. `Math.min(0, maxPageSize)` yields `pageSize = 0`, which is then set as the new filter's `limit`. Because `TopicMessageFilter.hasLimit()` returns `false` for `limit == 0`, the repository's `findByFilter()` never calls `setMaxResults()`, issuing an unbounded SQL query that loads all matching rows into memory — bypassing the `maxPageSize` design constraint entirely.

### Finding Description

**Exact code path:**

`PollingTopicListener.poll()` (lines 54–61): [1](#0-0) 

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

When `context.getCount().get() == filter.getLimit()` (i.e., the listener has delivered exactly N messages for a user-supplied limit of N), `limit = 0` and `pageSize = Math.min(0, 5000) = 0`.

`TopicMessageFilter.hasLimit()` (lines 39–41): [2](#0-1) 

```java
public boolean hasLimit() {
    return limit > 0;
}
```

`limit = 0` → `hasLimit() = false`.

`TopicMessageRepositoryCustomImpl.findByFilter()` (lines 51–53): [3](#0-2) 

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

Because `hasLimit()` is `false`, `setMaxResults()` is never called. The JPA `TypedQuery` executes with no `LIMIT` clause, returning every matching row from `startTime` onward for the given topic.

**Root cause:** The code assumes `limit` computed from `filter.getLimit() - context.getCount().get()` will always be positive when `filter.hasLimit()` is true. It does not guard against the zero case. `Math.min(0, maxPageSize) = 0` silently produces a filter value that `hasLimit()` treats as "no limit."

**Exploit flow:**

1. User subscribes via gRPC with any positive `limit = N`.
2. `TopicMessageServiceImpl.incomingMessages()` calls `topicListener.listen(newFilter)` where `newFilter.limit = N - already_delivered`. [4](#0-3) 
3. `PollingTopicListener` polls repeatedly via `repeatWhen(RepeatSpec.times(Long.MAX_VALUE)...)`. [5](#0-4) 
4. After N messages are delivered, `context.getCount().get() == N`. The outer `flux.take(N)` in `TopicMessageServiceImpl` sends a cancel signal upstream. [6](#0-5) 
5. Due to the bounded-elastic scheduler and the fixed polling interval (default 500 ms), the next `poll()` invocation may already be scheduled or executing before the cancel signal propagates through the operator chain.
6. That `poll()` call computes `pageSize = 0`, builds `newFilter` with `limit = 0`, and calls `findByFilter(newFilter)` — which issues an unbounded SQL query. [7](#0-6) 

### Impact Explanation

The unbounded JPA query loads all `topic_message` rows from `startTime` onward for the subscribed topic into the JVM heap. For a high-volume topic this can be millions of rows, causing:
- **Memory exhaustion** on the mirror node gRPC server (OOM).
- **Database CPU/IO spike** from a full table scan with no LIMIT.
- **Denial of service** for other subscribers sharing the same JVM and database connection pool.

The user does not receive more data than their limit (the outer `take(N)` cancels after the first element of the unbounded result), so this is a server-side resource exhaustion issue, not a data exfiltration issue. Severity: **Medium** (DoS, no privilege escalation or data leak).

### Likelihood Explanation

- Requires the server to be configured with `listenerProperties.type = POLL` (non-default; default is `REDIS`). [8](#0-7) 
- Any authenticated (or unauthenticated, depending on gRPC auth config) subscriber can trigger this by simply setting any positive `limit` and waiting for messages to arrive.
- The race window is bounded by the polling interval (≥50 ms, default 500 ms) — small but reproducible under load or with a crafted subscription.
- Repeatability: an attacker can open many concurrent subscriptions with small limits to amplify the effect.

### Recommendation

Add an explicit guard in `poll()` to return an empty Flux when the computed limit is zero or negative, preventing the zero-limit filter from reaching the repository:

```java
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessageFilter filter = context.getFilter();
    TopicMessage last = context.getLast();
    int limit = filter.hasLimit()
            ? (int) (filter.getLimit() - context.getCount().get())
            : Integer.MAX_VALUE;
    if (limit <= 0) {
        return Flux.empty(); // limit exhausted; cancellation is in-flight
    }
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
    ...
}
```

Alternatively, enforce `pageSize >= 1` before building `newFilter`, or add a `@Min(1)` guard on the computed page size before passing it to the repository.

### Proof of Concept

**Preconditions:** Mirror node configured with `hiero.mirror.grpc.listener.type=POLL`.

**Steps:**

1. Subscribe to any topic via gRPC `subscribeTopic` with `limit = 1` and `startTime` set to a time before an existing message.
2. The `PollingTopicListener` delivers 1 message; `context.getCount().get()` becomes 1.
3. The outer `take(1)` sends a cancel signal, but the `repeatWhen` scheduler (bounded elastic, 500 ms interval) may already have queued the next `poll()` invocation.
4. The next `poll()` computes `limit = 1 - 1 = 0`, `pageSize = Math.min(0, 5000) = 0`.
5. `newFilter.limit = 0` → `newFilter.hasLimit() = false`.
6. `findByFilter(newFilter)` executes `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with **no LIMIT clause**.
7. Observe in database slow-query logs or JVM heap metrics that a full-scan query was issued and all matching rows were loaded.

Repeat with many concurrent subscriptions to amplify resource exhaustion.
