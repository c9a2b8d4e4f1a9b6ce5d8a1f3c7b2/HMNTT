### Title
Missing Genesis Block Cache in `FindByIndex` Enables Repeated Correlated Subquery Execution

### Summary
`FindByIndex(ctx, 0)` always executes the `selectRecordBlockByIndex` SQL query — including its correlated subquery for `index+1` — on every invocation, even though the identical data is already cached in `br.genesisBlock` and served without any DB round-trip by `RetrieveGenesis`. Any unauthenticated caller can exploit this asymmetry to drive sustained, unnecessary database load by repeatedly requesting the genesis block by index.

### Finding Description
**Code path:**
- `rosetta/app/persistence/block.go`, `FindByIndex()` lines 171–181 → `findBlockByIndex()` lines 210–224
- `selectRecordBlockByIndex` constant, lines 75–85

`FindByIndex` accepts any non-negative `int64` index from an unauthenticated HTTP caller (via `RetrieveBlock` in `base_service.go` line 82–83). After calling `initGenesisRecordFile` (which is a no-op after first run due to `sync.Once`), it unconditionally delegates to `findBlockByIndex`, which executes:

```sql
select consensus_start,
       coalesce((
         select consensus_start-1
         from record_file
         where index = @index + 1::bigint   -- correlated subquery fires every call
       ), consensus_end) as consensus_end,
       ...
from record_file
where index = @index
```

For `index=0` this is two indexed lookups per request (`WHERE index = 0` + `WHERE index = 1`).

**Root cause / failed assumption:** The code assumes `findBlockByIndex` is only called for non-genesis blocks, or that the DB cost is negligible. Neither is enforced. `RetrieveGenesis` (lines 183–188) proves the genesis result is already available in memory (`br.genesisBlock`), but `FindByIndex` never checks whether `index == br.genesisBlock.Index` before issuing the query.

**Why existing checks fail:**
- `initGenesisRecordFile` caches the genesis *record* but `findBlockByIndex` ignores that cache entirely.
- Traefik rate limiting (`average: 10` per `requestHost`, lines 157–160 of `values.yaml`) is keyed on the HTTP `Host` header — trivially bypassed by rotating headers or using distributed clients. It is also absent in bare-metal / non-Helm deployments.
- No application-level rate limiting or result caching exists in the Go code.

### Impact Explanation
Each `POST /block` with `{"block_identifier": {"index": 0}}` causes two synchronous DB index scans. Under sustained flood (easily achievable from multiple IPs or with varied `Host` headers), this generates a continuous stream of DB queries for data that is already resident in process memory. On a busy mirror node where the DB is shared with other services, this can measurably increase DB CPU and I/O, degrading block-query latency for all consumers. The scope matches the stated threat model: increasing node resource consumption without brute-force credential attacks.

### Likelihood Explanation
The Rosetta API is a public, unauthenticated read interface by design (Coinbase Rosetta spec). No login, API key, or privileged access is required. The genesis block index (typically `0` or a small integer) is publicly known from `RetrieveGenesis` or network-status responses. A single attacker with a basic HTTP client and multiple source IPs or rotating `Host` headers can sustain the attack indefinitely.

### Recommendation
In `findBlockByIndex`, short-circuit to the cached genesis block when the requested index matches:

```go
func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
    if index < br.genesisBlock.Index {
        return nil, hErrors.ErrBlockNotFound
    }
    // Return cached genesis without a DB round-trip
    if index == br.genesisBlock.Index {
        return br.genesisBlock.ToBlock(br.genesisBlock), nil
    }
    // ... existing DB query for non-genesis blocks
}
```

This mirrors the behaviour already implemented in `RetrieveGenesis` (lines 183–188) and eliminates the unnecessary correlated subquery for the most frequently targeted index value.
