### Title
Mirror-Node Empty-Endpoint Response Bypasses Retry Logic and Terminates Pinger Process

### Summary
In `fetchMirrorNodeNetwork()`, when a mirror node response contains nodes whose every `service_endpoint` has both `domain_name` and `ip_address_v4` as empty strings, all endpoints are silently filtered and the function returns `retry=false` alongside an error. This `retry=false` signal causes `buildNetworkFromMirrorNodes` to break out of its retry loop immediately — regardless of the configured `mirrorNodeClientMaxRetries` value — and propagates the error to `newClient`, which causes `main()` to call `log.Fatalf`, terminating the pinger process entirely.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–129:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {
        continue          // silently skipped
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

If every endpoint has `domain_name=""` and `ip_address_v4=""`, the inner `continue` fires for every entry and `network` remains empty.

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
    //              ^^^^^ retry=false
}
```

**Retry bypass:**

In `buildNetworkFromMirrorNodes` (lines 52–69):

```go
network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
...
if !retry || attempt == attempts {
    break   // exits immediately on retry=false
}
```

Because `retry=false`, the loop breaks on the very first attempt, ignoring `cfg.mirrorNodeClientMaxRetries` (default: 10).

**Process termination:**

`sdk_client.go` line 19–21: `buildNetworkFromMirrorNodes` error propagates to `newClient`, which returns it. `main.go` line 43: `log.Fatalf("client error: %v", err)` — the process exits.

**Root cause / failed assumption:** The code assumes that "no usable endpoints found" is a permanent, non-retryable condition. In reality it can be caused by a single crafted or injected HTTP response, and the `retry=false` signal permanently forfeits all configured retry attempts.

### Impact Explanation

The pinger is the liveness/readiness probe component for the mirror node deployment. It writes `/tmp/alive` (heartbeat) and `/tmp/ready` (readiness marker). A successful attack at startup prevents the process from ever reaching the ticker loop, meaning:
- `/tmp/ready` is never written → readiness probe fails → container never enters service
- In a crash-loop scenario (container restart policy), the attacker can sustain the DoS by serving the crafted response on every startup attempt
- All periodic HBAR transfer pings cease, breaking the monitoring/alerting chain for the network

### Likelihood Explanation

**Preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` must be set (required for custom/private deployments)
2. The attacker must be able to inject a response for the mirror REST URL

The documented default example URL in `config.go` line 37 is `http://mirror-rest:5551` — plain HTTP with no TLS. On a shared network segment (e.g., a Kubernetes cluster where the mirror-rest service is reachable over HTTP), an attacker with pod-level access or the ability to perform ARP/DNS poisoning can MITM the HTTP connection and inject the crafted JSON. No authentication or special privilege is required to serve a valid HTTP 200 response with the crafted body. The attack is fully repeatable on every container restart.

### Recommendation

1. **Fix the retry signal:** Change the `len(network) == 0` return to `return nil, true, fmt.Errorf(...)` so that an empty-endpoint response is treated as a transient, retryable condition rather than a permanent failure.
2. **Enforce HTTPS:** Reject or warn when `cfg.mirrorRest` uses `http://` instead of `https://`, or enforce TLS verification in the `http.Client`.
3. **Minimum-node threshold:** Require at least N usable endpoints (e.g., 1/3 of returned nodes) before accepting the response as valid, making partial-poisoning attacks harder.
4. **Response validation:** Cross-check the returned node account IDs against a locally configured allowlist or a known-good set before replacing the active network map.

### Proof of Concept

1. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>:5551`.
2. Serve the following JSON at `GET /api/v1/network/nodes`:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"domain_name": "", "ip_address_v4": "", "port": 50211}
      ]
    }
  ],
  "links": {"next": null}
}
```
3. Start the pinger. Observe:
   - `fetchMirrorNodeNetwork` returns `retry=false, err="no usable service_endpoints found"`
   - `buildNetworkFromMirrorNodes` breaks on attempt 1/11 (no retries)
   - `main()` prints `client error: attempt 1/11: no usable service_endpoints found from ...` and exits via `log.Fatalf`
   - `/tmp/ready` is never created; the readiness probe permanently fails
