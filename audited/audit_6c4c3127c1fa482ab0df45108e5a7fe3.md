### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables Unauthenticated Database Exhaustion via Full-Scan `DISTINCT ON` Query

### Summary
An unauthenticated external user can trigger `getHookStorageChange()` by supplying a single timestamp parameter (e.g., `timestamp=gte:0`) with no key filters, causing `findByKeyBetweenAndTimestampBetween()` to execute a `DISTINCT ON (key)` full-range scan over the entire `hook_storage_change` table partition for a given `owner_id`/`hook_id`. No timestamp range width limit, no rate limiting, and no query timeout are enforced, making this trivially repeatable and concurrently exploitable.

### Finding Description

**Entry point** — `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` in `HooksController.java`: [1](#0-0) 

The controller accepts up to 2 timestamp parameters (`@Size(max = 2)`) and up to 100 key parameters, but imposes **no constraint on the width of the timestamp range**. When no keys are supplied, `lowerBound` and `upperBound` default to the full 32-byte key space: [2](#0-1) 

`Bound.of()` is called with `primarySortField = false`, which bypasses the only range-inversion guard in `Bound`'s constructor: [3](#0-2) 

When no upper timestamp is provided, `adjustUpperBound()` returns `Long.MAX_VALUE`: [4](#0-3) 

In `getHookStorageChange()`, when `requestHasKeys = false` (no key parameters supplied), the code unconditionally calls `findByKeyBetweenAndTimestampBetween()` with the full key range and the attacker-controlled timestamp range: [5](#0-4) 

The underlying native query uses `DISTINCT ON (key)` without a covering index hint, requiring PostgreSQL to sort and deduplicate **all rows** matching the WHERE clause before applying the `LIMIT`: [6](#0-5) 

The `limit` cap of 100 (`MAX_LIMIT`) only restricts returned rows — it does not prevent the database from scanning millions of qualifying rows to compute the `DISTINCT ON` result set. [7](#0-6) 

**Root cause**: The `Bound` class has no maximum range width enforcement for the timestamp dimension, and the service layer has no guard that short-circuits or rejects requests with unbounded timestamp ranges when `requestHasKeys = false`.

### Impact Explanation

An attacker targeting a hook with a large history of storage changes can force repeated full-partition scans (`DISTINCT ON` over all rows for a given `owner_id`/`hook_id`). Concurrent requests from multiple source IPs exhaust database CPU, I/O, and connection pool resources. Because the mirror node's REST-Java service is the read path for network state queries, sustained exhaustion degrades or halts the ability of clients and validators to confirm transaction state — matching the stated critical scope of "network not being able to confirm new transactions."

### Likelihood Explanation

The endpoint is publicly accessible with no authentication. The attacker only needs any valid `ownerId` and `hookId` (both discoverable via other public API calls). The exploit requires a single HTTP GET request with one query parameter (`timestamp=gte:0`). It is trivially scriptable, repeatable at high frequency, and requires no privileged access, special tokens, or on-chain transactions.

### Recommendation

1. **Enforce a maximum timestamp range width**: In `HookServiceImpl.getHookStorageChange()` or in `Bound.of()`, reject or cap requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 24 hours in nanoseconds).
2. **Require a timestamp lower bound**: Reject requests where only an upper bound is supplied, or where the lower bound resolves to `0` (epoch), forcing callers to provide a meaningful window.
3. **Add rate limiting**: Apply per-IP or per-account rate limiting on the `/hooks/{hookId}/storage` endpoint.
4. **Add a query statement timeout**: Configure a PostgreSQL `statement_timeout` for the read replica used by this service to bound worst-case query duration.

### Proof of Concept

```
# Step 1: Discover a valid ownerId and hookId via the public hooks listing
GET /api/v1/accounts/0.0.1234/hooks

# Step 2: Send the exhaustion request — no keys, unbounded timestamp range
GET /api/v1/accounts/0.0.1234/hooks/5678/storage?timestamp=gte:0&limit=100

# Step 3: Repeat concurrently (e.g., 50 parallel connections) to exhaust DB resources
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/5678/storage?timestamp=gte:0&limit=100" &
done
wait
```

With `timestamp=gte:0` and no key parameters: `requestHasKeys = false`, `timestampLowerBound = 0`, `timestampUpperBound = Long.MAX_VALUE`, `keyLowerBound = 0x000...000`, `keyUpperBound = 0xFFF...FFF` — the most expensive possible query path is triggered on every request.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
