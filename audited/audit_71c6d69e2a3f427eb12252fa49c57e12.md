### Title
Plaintext gRPC Channel by Default: No TLS Enforced at Application or Ingress Layer Enables Network-Level MITM

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only an executor and connection limit — no SSL context or `useTransportSecurity()` call is ever made. The default Helm deployment compounds this by shipping with `ingress.tls.enabled: false` and the service backend explicitly annotated as `h2c` (HTTP/2 cleartext). Any attacker with network-path access between a client and the server can intercept and tamper with all gRPC traffic in transit.

### Finding Description
**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 27–35, function `grpcServerConfigurer()`.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

No call to `serverBuilder.useTransportSecurity(...)` or any SSL/TLS context is present. `NettyProperties` (the only properties class wired in) exposes only `maxConcurrentCallsPerConnection` — there is no TLS property at all. A grep across the entire codebase for `useTransportSecurity`, `sslContext`, `SslContext`, or `spring.grpc.server.ssl` returns zero hits in the server-side code.

The Helm chart default values reinforce the plaintext posture:
- `charts/hedera-mirror-grpc/values.yaml` line 142–144: `ingress.tls.enabled: false`
- Line 329: `traefik.ingress.kubernetes.io/service.serversscheme: h2c` — explicitly declares the backend as cleartext HTTP/2

The ingress template at `charts/hedera-mirror-grpc/templates/ingress.yaml` lines 35–42 only emits a TLS stanza when `ingress.tls.enabled` is true, which it is not by default. The global gateway is also disabled by default (`global.gateway.enabled: false`).

**Root cause:** The `NettyServerBuilder` customizer is the sole application-level hook for server configuration. Because it never calls `useTransportSecurity()` and no Spring gRPC TLS properties are set, the Netty server starts in plaintext mode. The default deployment configuration provides no compensating TLS termination at the ingress layer either.

**Exploit flow:**
1. Attacker positions themselves on the network path between a gRPC client and the mirror node (e.g., BGP hijack, ISP-level interception, compromised cloud routing, or same-segment ARP poisoning).
2. Client connects to port 5600 (or via the Traefik ingress) using standard gRPC over HTTP/2 cleartext.
3. Attacker intercepts the TCP stream. Because there is no TLS, the HTTP/2 frames are fully readable.
4. Attacker reads streaming `ConsensusService` topic message responses (HCS message content, sequence numbers, consensus timestamps) or `NetworkService` address book responses.
5. Attacker injects modified frames — e.g., altered topic message content, forged sequence numbers, or a poisoned address book pointing clients to attacker-controlled consensus nodes.

**Why existing checks fail:** The ingress TLS block is conditional and off by default. There is no application-level enforcement that would reject plaintext connections or redirect to TLS. No mutual TLS or channel-level authentication exists.

### Impact Explanation
The gRPC API streams HCS topic messages (which may carry application-layer sensitive data) and the network address book (node IP addresses and certificates). A MITM can:
- **Read** all topic message content in transit (confidentiality breach).
- **Inject** forged topic messages with arbitrary content, sequence numbers, or timestamps, causing downstream applications to act on false data.
- **Replace** the network address book with attacker-controlled node endpoints, redirecting SDK clients to malicious consensus nodes for future transaction submission.

Severity is **High** for deployments using default Helm values, as the entire channel is unprotected end-to-end.

### Likelihood Explanation
The precondition is network-path access, not just an internet connection. However:
- Cloud-hosted deployments share infrastructure with other tenants; cloud-internal routing attacks are documented.
- BGP hijacking of public IP ranges is a known, repeatable technique used against blockchain infrastructure specifically.
- Any operator who deploys with default Helm values (the documented starting point) is exposed without any additional action required.
- The `h2c` service annotation makes the plaintext posture an explicit, documented choice rather than an oversight, increasing the chance it persists in production.

Likelihood is **Medium-High** for internet-facing deployments.

### Recommendation
1. **Application layer (preferred):** In `grpcServerConfigurer()`, call `serverBuilder.useTransportSecurity(certChain, privateKey)` with a provisioned certificate, or configure `spring.grpc.server.ssl.*` properties and wire them into the customizer via `NettyProperties`.
2. **Add a `tlsEnabled` flag to `NettyProperties`** so operators can explicitly opt into plaintext only for internal/loopback deployments.
3. **Helm chart:** Change `ingress.tls.enabled` default to `true` and require `secretName` to be set; add a validation note in `values.yaml`.
4. **Defense in depth:** Enable mutual TLS (`serverBuilder.clientAuth(ClientAuth.REQUIRE)`) for any deployment where clients are known services rather than anonymous subscribers.

### Proof of Concept
**Preconditions:** Default Helm deployment, attacker has network-path access (e.g., same Kubernetes node, compromised router, or BGP-level).

```bash
# 1. Confirm plaintext — connect directly to the gRPC port
grpcurl -plaintext <mirror-node-host>:5600 list
# Returns service list without TLS negotiation, confirming h2c

# 2. Capture traffic with tcpdump (attacker on network path)
tcpdump -i eth0 -w grpc.pcap port 5600

# 3. Open capture in Wireshark — HTTP/2 frames are fully decoded,
#    topic message payloads visible in plaintext

# 4. MITM injection using mitmproxy with HTTP/2 support:
mitmproxy --mode transparent --ssl-insecure -p 5600
# Intercept and modify ConsensusService SubscribeTopic responses,
# replacing message contents or injecting synthetic messages before
# forwarding to the legitimate client.
```
