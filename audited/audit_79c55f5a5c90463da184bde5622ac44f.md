### Title
Unauthenticated Three-Part UNION ALL Query Exhausts DB Connection Pool via Token Allowance Endpoint

### Summary
An unauthenticated external user can craft a single request to `/api/v1/accounts/{id}/allowances/tokens` with both `spender.id` and `token.id` range bounds, causing `extractTokenMultiUnionQuery()` and `getQuery()` to build a three-part `UNION ALL` query — each part with its own `ORDER BY` and `LIMIT` clause — that performs wide-range index scans on the `token_allowance` table. Because the REST API has no per-IP rate limiting and the default DB connection pool is only 10 connections, as few as 10 concurrent such requests exhaust the pool for up to 20 seconds each, blocking all other queries and degrading the mirror node REST API for all users.

### Finding Description

**Exact code path:**

`rest/controllers/tokenAllowanceController.js` lines 68–71 — `getAccountTokenAllowances` calls `buildAndValidateFilters` then `extractTokenMultiUnionQuery`: [1](#0-0) 

`extractTokenMultiUnionQuery` (lines 22–60) parses `spender.id` and `token.id` range filters with no restriction on the numeric range of the bounds: [2](#0-1) 

`tokenAllowanceService.getQuery()` (lines 63–75) builds up to three subqueries, each with its own `ORDER BY` and `LIMIT $2` clause, joined by `UNION ALL`: [3](#0-2) 

**Root cause:** The `limit` parameter is correctly capped at `responseLimit.max` (default 100) by `getLimitParamValue`: [4](#0-3) 

However, there is **no bound on the numeric range** of `spender.id` or `token.id` values. An attacker supplying `spender.id=gte:1&spender.id=lte:9999999999&token.id=gte:1&token.id=lte:9999999999` causes the inner subquery to be `spender > 1 AND spender < 9999999999`, scanning the full spender range for the given owner. The three-part UNION ALL is generated whenever both primary and secondary bounds are present (lower + inner + upper all non-empty), as confirmed by the test fixture: [5](#0-4) 

**No rate limiting exists** in the REST API middleware stack — only `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response cache are registered: [6](#0-5) 

**DB pool defaults** are 10 max connections and a 20-second statement timeout: [7](#0-6) [8](#0-7) 

### Impact Explanation

With a pool of 10 connections and a 20-second statement timeout, an attacker needs only 10 concurrent requests to hold all DB connections for up to 20 seconds. During this window, every other REST API request that requires a DB connection queues or fails. Because the mirror node REST API is the primary data-serving layer, this degrades or halts data availability for all consumers. The three-part UNION ALL with wide range bounds is more expensive than a simple query because PostgreSQL must plan and execute three separate subqueries plus an outer merge sort, amplifying per-request DB load. This maps to the stated severity: degradation of ≥30% of mirror node processing capacity without requiring privileged access.

### Likelihood Explanation

No authentication or rate limiting is required. Any external user who can reach the REST API endpoint can trigger this. The attack is trivially repeatable with a simple script sending 10 concurrent HTTP requests in a loop. The attacker does not need to know valid account IDs — any account ID (including non-existent ones) will still execute the query path (the query simply returns zero rows after the full plan/scan cycle). The attack is stateless and requires no prior knowledge of the system.

### Recommendation

1. **Add application-level rate limiting** (e.g., `express-rate-limit`) to the REST API middleware stack, keyed by IP address, for all endpoints.
2. **Bound the numeric range** of `spender.id` and `token.id` range queries — reject or clamp requests where `upper - lower` exceeds a configurable maximum (similar to `maxTimestampRange` used for timestamp parameters).
3. **Increase the DB connection pool** size or add a query queue with a maximum wait time to fail fast rather than holding connections.
4. **Consider query result caching** (Redis cache is already present but optional) for this endpoint to absorb repeated identical queries.

### Proof of Concept

```bash
# Send 10 concurrent requests to exhaust the DB connection pool
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1/allowances/tokens?\
spender.id=gte:1&spender.id=lte:9999999999&\
token.id=gte:1&token.id=lte:9999999999&\
limit=100" &
done
wait
```

Each request triggers the three-part UNION ALL:
```sql
(select * from token_allowance where owner=$1 and amount>0 and spender=$3 and token_id>=$4
 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender>$5 and spender<$6
 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender=$7 and token_id<=$8
 order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
-- params: [1, 100, 1, 1, 1, 9999999999, 9999999999, 9999999999]
```

With 10 concurrent requests holding all pool connections for up to 20 seconds each, all subsequent REST API requests that require DB access will queue or return errors, degrading the mirror node for all users.
