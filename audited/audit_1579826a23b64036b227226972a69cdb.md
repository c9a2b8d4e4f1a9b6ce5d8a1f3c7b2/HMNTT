### Title
HTTP 206 "Partial Mirror Node" Responses Incorrectly Cached and Served to All Subsequent Users

### Summary
`responseCacheUpdateHandler()` uses `httpStatusCodes.isSuccess(res.statusCode)` to decide whether to cache a response. Because `isSuccess` is defined as `code >= 200 && code < 300`, it returns `true` for status code 206 (`PARTIAL_CONTENT` — meaning the mirror node has incomplete records). Once a 206 response is cached, every subsequent request for the same URL is served the stale, partial payload until the TTL expires, even after the mirror node has fully caught up.

### Finding Description

**Exact code location:**

`rest/middleware/responseCacheHandler.js`, `responseCacheUpdateHandler()`, line 95: [1](#0-0) 

```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```

`httpStatusCodes.isSuccess` is defined in `rest/constants.js` line 159: [2](#0-1) 

```js
isSuccess: (code) => code >= 200 && code < 300,
```

`PARTIAL_CONTENT` is defined as: [3](#0-2) 

```js
PARTIAL_CONTENT: new StatusCode(206, 'Partial mirror node'),
```

This is **not** the standard HTTP range-request 206 — it is a domain-specific code meaning the mirror node does not yet have complete data for the requested resource (e.g., a transaction or contract log page is missing records because the node is lagging).

**Root cause:** The guard `httpStatusCodes.isSuccess(res.statusCode)` was intended to cache only fully successful responses, but the range `[200, 300)` inadvertently includes 206. A 206 response represents a *transient degraded state* of the mirror node, not a complete, authoritative answer.

**Cache storage path (lines 114–116):** [4](#0-3) 

```js
const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
```

The 206 status code is stored verbatim in `CachedApiResponse`. When `responseCacheCheckHandler` later serves the cached entry, it replays the stored `statusCode` (206) and the partial body to every client: [5](#0-4) 

```js
const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
```

**Existing checks and why they fail:**

- The `isUnmodified` branch (304) is handled separately and correctly normalised to 200 before caching — but 206 is not given the same treatment.
- The `ttl > 0` guard (line 97) only prevents caching when `max-age=0`; it does not filter on status code.
- There is no explicit exclusion of 206 anywhere in the caching pipeline. [6](#0-5) 

### Impact Explanation

Any paginated or time-sensitive endpoint (transactions, contract logs, token balances, etc.) that returns 206 during a mirror node lag window will have its incomplete payload frozen in Redis for the full `max-age` TTL. All users querying the same URL during that window receive missing records and a misleading 206 status, even after the mirror node has fully caught up. This directly causes **data integrity loss** for consumers relying on the mirror node for complete ledger history, and it is persistent (not self-healing until TTL expiry).

### Likelihood Explanation

Mirror nodes routinely lag behind the consensus network during high-throughput periods, node restarts, or network upgrades — all conditions that naturally produce 206 responses. No authentication, special headers, or elevated privileges are required. Any anonymous HTTP client that happens to be the first to query a given URL during such a lag window will trigger the poisoning. The attack is repeatable: an adversary can monitor for 206 responses and immediately re-query to confirm the cache has been poisoned, then wait for the TTL to expire and repeat.

### Recommendation

Restrict caching to exactly `200 OK` (and `304` as already handled). Replace the broad `isSuccess` check with an explicit allowlist:

```js
// Before (vulnerable)
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {

// After (fixed)
const isCacheable = res.statusCode === httpStatusCodes.OK.code;
if (responseBody && responseCacheKey && (isUnmodified || isCacheable)) {
```

Alternatively, add an explicit exclusion:

```js
const isSuccess = httpStatusCodes.isSuccess(res.statusCode) &&
                  res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code;
```

### Proof of Concept

1. Stand up the mirror node REST service with Redis caching enabled (`config.redis.enabled = true`).
2. Arrange for a target endpoint (e.g., `GET /api/v1/transactions?limit=25`) to return HTTP 206 — this occurs naturally when the mirror node is behind the network, or can be reproduced in tests by stubbing `res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code` in the response pipeline.
3. Issue the first request while the node is in the partial state:
   ```
   curl -i http://localhost:5551/api/v1/transactions?limit=25
   # → HTTP/1.1 206 Partial mirror node  (partial body, missing records)
   ```
4. Confirm the entry is now in Redis (TTL > 0).
5. Allow the mirror node to fully catch up (it would now return 200 with complete data for the same URL).
6. Issue a second request from a different client:
   ```
   curl -i http://localhost:5551/api/v1/transactions?limit=25
   # → HTTP/1.1 206 Partial mirror node  (stale partial body served from cache)
   ```
   The complete records are never returned until the Redis TTL expires, demonstrating that the partial response has been permanently cached and served to all users.

```javascript
const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
```
