### Title
Gas Estimation Binary Search Bypasses Per-Request Gas Throttle, Enabling CPU Exhaustion DoS

### Summary
The gas throttle is consumed exactly once per request at the controller level (for `request.getGas()`), but `BinaryGasEstimator.search()` runs up to `maxGasEstimateRetriesCount=20` additional full EVM executions with `estimate=true`, which explicitly skips both throttle consumption and gas restoration. Simultaneously, `requestTimeout` is enforced only at Hibernate SQL statement boundaries via a `StatementInspector`, meaning CPU-bound EVM iterations that hit cached state never trigger the timeout. An unprivileged attacker can therefore cause up to 21Ã— the throttle-accounted EVM work per request.

Note on the "always reverts" premise in the question: a contract that reverts unconditionally fails the initial `callContract(params, context)` call in `estimateGas` (which uses `estimate=false` and calls `validateResult`), throwing `MirrorEvmTransactionException` before the binary search starts. The exploitable variant is a contract that **succeeds with full gas (15 M) but reverts at any lower gas**, which drives all 20 binary search iterations.

### Finding Description

**Step 1 â€“ Single throttle deduction at controller entry**

`ContractController.call()` calls `throttleManager.throttle(request)` once, consuming `scaleGas(request.getGas())` tokens from the shared `gasLimitBucket`. [1](#0-0) 

**Step 2 â€“ Initial call uses `estimate=false`, validates result**

`estimateGas` calls `callContract(params, context)` â†’ `doProcessCall(params, params.getGas(), false)`. Because `estimate=false`, `validateResult` is invoked; a revert here throws and aborts before the binary search. For a contract that succeeds at full gas, execution continues. [2](#0-1) 

**Step 3 â€“ Binary search calls `doProcessCall` with `estimate=true` â€” no throttle, no restore**

`binaryGasEstimator.search()` loops up to 20 times, each time calling `doProcessCall(params, gas, true)`. The `finally` block in `doProcessCall` is gated on `!estimate`, so neither `restoreGasToBucket` nor any throttle consumption occurs for any of these iterations. [3](#0-2) [4](#0-3) 

**Step 4 â€“ `requestTimeout` only fires at Hibernate SQL boundaries**

`HibernateConfiguration.statementInspector()` checks elapsed time only when Hibernate is about to execute a SQL statement. If binary search iterations hit cached EVM state (contract bytecode, account data already in the read cache), no SQL is issued and the timeout never fires regardless of wall-clock time. [5](#0-4) 

**Step 5 â€“ `scaleGas` floors small values to zero**

`ThrottleProperties.scaleGas()` returns `0` for any gas â‰¤ 10 000, meaning requests with gas in that range consume nothing from the bucket at all. [6](#0-5) 

**Root cause**: The gas throttle model assumes one EVM execution per request. For `estimate=true` requests the actual execution count is `1 + min(20, iterations_to_converge)`, but the throttle only accounts for 1.

### Impact Explanation

With default settings (`requestsPerSecond=500`, `maxGasLimit=15 000 000`, `maxGasEstimateRetriesCount=20`), each estimate request that drives the full 20-iteration binary search performs 21 EVM executions while consuming only 1 slot in the gas bucket. An attacker sending 500 concurrent estimate requests per second (within the RPS limit) forces the server to execute up to 10 500 EVM runs per second instead of the intended 500. Each run at 15 M gas is a substantial CPU workload. Combined with the timeout gap for cache-warm iterations, this can saturate all available server threads, causing legitimate requests to queue indefinitely or be rejected.

### Likelihood Explanation

No authentication or privileged access is required. The attacker only needs:
1. A deployed contract (or any existing contract address) that succeeds at full gas but reverts below a threshold â€” trivially constructable.
2. The ability to send HTTP POST requests to `/api/v1/contracts/call` with `estimate: true`.

The attack is fully repeatable, scriptable, and requires no on-chain funds since the mirror node simulates execution without submitting transactions.

### Recommendation

1. **Account for binary search iterations in the gas throttle**: Before entering `binaryGasEstimator.search()`, pre-consume `maxGasEstimateRetriesCount Ã— scaleGas(params.getGas())` additional tokens from the gas bucket, or consume one token per iteration inside the loop.
2. **Enforce `requestTimeout` on wall-clock time, not only at SQL boundaries**: Add an explicit deadline check at the top of each `BinaryGasEstimator` loop iteration (e.g., check `ContractCallContext.get().getStartTime()` against the configured timeout).
3. **Cap the effective gas for estimate requests**: Apply a lower `maxGasLimit` for `ETH_ESTIMATE_GAS` call type, or introduce a separate `estimateGasPerSecond` throttle bucket.
4. **Reduce `maxGasEstimateRetriesCount`**: The default of 20 is generous; 10â€“12 iterations are sufficient for convergence across the full 21 000â€“15 000 000 gas range (logâ‚‚(15 000 000/21 000) â‰ˆ 9.5).

### Proof of Concept

```solidity
// Deploy once; succeeds at full gas, reverts below threshold
contract GasHog {
    function work() external {
        uint256 g = gasleft();
        require(g > 14_000_000, "not enough gas");
        // burn gas
        for (uint i; i < 100_000; i++) {}
    }
}
```

```python
import asyncio, aiohttp, json

URL = "http://<mirror-node>/api/v1/contracts/call"
PAYLOAD = {
    "to": "<GasHog address>",
    "data": "0x<work() selector>",
    "gas": 15000000,
    "estimate": True
}

async def flood(session):
    while True:
        await session.post(URL, json=PAYLOAD)

async def main():
    async with aiohttp.ClientSession() as s:
        await asyncio.gather(*[flood(s) for _ in range(500)])

asyncio.run(main())
```

Each request triggers 1 initial EVM run (succeeds) + up to 20 binary search runs (all revert because `gasleft() < 14 000 000` at mid-point gas values), totalling 21 EVM executions while consuming only 1 gas-bucket slot. Server CPU saturates before `requestTimeout` fires because the binary search iterations hit the warm read cache and issue no SQL.
