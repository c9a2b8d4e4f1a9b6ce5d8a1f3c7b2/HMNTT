### Title
Unbounded Concurrent Memory Buffering via ShallowEtagHeaderFilter with No Rate Limiting in rest-java

### Summary
The `etagFilter()` bean in `RestJavaConfiguration` registers a `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no concurrent-request throttle in the rest-java module. `ShallowEtagHeaderFilter` wraps every response in a `ContentCachingResponseWrapper` that accumulates the full serialized response body in a heap byte array before computing the MD5 ETag. Because the rest-java service has no per-IP or global rate limiter (unlike the web3 module which has `ThrottleConfiguration`), an unauthenticated attacker can open many simultaneous connections, each holding a full response buffer in the JVM heap until the filter finishes, exhausting available heap and crashing the service.

### Finding Description
**Exact code location:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
``` [1](#0-0) 

**Root cause:** Spring's `ShallowEtagHeaderFilter` intercepts the servlet response and replaces the output stream with a `ContentCachingResponseWrapper`. Every byte written by the controller is accumulated in an in-memory byte array. Only after the entire response is written does the filter compute the MD5 hash, set the `ETag` header, and flush the buffer to the real socket. This means the full serialized JSON body of every in-flight request is simultaneously resident in heap memory.

**No rate limiting in rest-java:** The web3 module has an explicit `ThrottleConfiguration` with `rateLimitBucket` / `gasLimitBucket` beans. The rest-java module has no equivalent — only a `LoggingFilter` and a `MetricsFilter` are registered. [2](#0-1) [3](#0-2) 

**Pagination limits exist but are insufficient:** `Constants.MAX_LIMIT = 100` caps list endpoints at 100 items per response. [4](#0-3) 

However, pagination limits bound individual response size, not the number of simultaneous in-flight requests. With N concurrent requests each buffering R bytes, total heap consumption is N × R. At `limit=100` with moderately rich objects (network nodes with service endpoints, hook storage slots, airdrop records), individual responses can reach hundreds of KB. With hundreds of concurrent connections — trivially achievable with tools like `ab`, `wrk`, or `hey` — total buffered memory exceeds the 2 GB container limit. [5](#0-4) 

**Endpoints with larger per-item payloads:** `/api/v1/network/fees` returns the full fee schedule (all transaction types, all sub-types) with no pagination at all. [6](#0-5) 

### Impact Explanation
The rest-java service is the authoritative source for network node information, fee schedules, exchange rates, token airdrops, NFT allowances, and hook storage. An `OutOfMemoryError` in the JVM causes the Spring Boot process to crash. Kubernetes will restart the pod, but a sustained flood keeps it in a crash loop. The Kubernetes HPA is disabled by default (`hpa.enabled: false`). Loss of this service means clients cannot query network topology, fee data, or account state, directly impairing the ability of wallets and SDKs to construct and submit transactions to the Hedera network. [7](#0-6) 

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. Any internet-accessible deployment is reachable. The attack requires only a standard HTTP load-generation tool (e.g., `wrk -t 50 -c 500 -d 60s https://host/api/v1/network/fees`). The `/api/v1/network/fees` endpoint is particularly attractive because it is unpaginated and returns a large, stable payload. The attack is repeatable and requires no special knowledge of the application.

### Recommendation
1. **Add a rate-limiting filter to rest-java** analogous to the `ThrottleConfiguration` in the web3 module — either a global requests-per-second bucket or a per-IP bucket using Bucket4j or a servlet filter.
2. **Replace `ShallowEtagHeaderFilter` with a streaming ETag approach** (e.g., compute the ETag from a hash of the DB query result set before serialization, or use a `WeakEtagHeaderFilter` backed by a cache key rather than full body buffering).
3. **Set `server.tomcat.max-connections` and `server.tomcat.threads.max`** in the Spring Boot configuration to bound the number of simultaneous in-flight requests.
4. **Paginate `/api/v1/network/fees`** or add a response-size cap.
5. **Enable HPA** so the service can scale horizontally under load.

### Proof of Concept
```bash
# Step 1: Confirm the ETag filter is active (no auth required)
curl -I https://<host>/api/v1/network/fees
# Response includes: ETag: "..."

# Step 2: Flood with concurrent requests to exhaust heap
wrk -t 50 -c 500 -d 120s \
    -H "Cache-Control: no-cache" \
    https://<host>/api/v1/network/fees

# Step 3: Observe JVM heap exhaustion
# Monitor via: kubectl top pod -l app.kubernetes.io/component=rest-java
# Or watch for OOMKilled / CrashLoopBackOff in pod status:
kubectl get pods -w -l app.kubernetes.io/component=rest-java

# Expected result: pod restarts with OOMKilled reason within 60-120 seconds
# of sustained 500-connection flood, as each connection holds a full
# fee-schedule response buffer in the ShallowEtagHeaderFilter's
# ContentCachingResponseWrapper until the MD5 is computed.
```
