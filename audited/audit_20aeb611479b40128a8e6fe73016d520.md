### Title
Missing Request Body Size Limit Enables Memory Exhaustion DoS in Rosetta API Server

### Summary
The `Http` struct in `rosetta/app/config/types.go` defines only time-based timeout fields with no maximum request body size field, and the `http.Server` constructed in `rosetta/main.go` applies no `http.MaxBytesReader` or equivalent body-size cap anywhere in the middleware chain. An unprivileged remote attacker can open many concurrent connections and stream large POST bodies, exhausting server memory, crashing the process, and dropping all active network connections.

### Finding Description
**Code path:**

`rosetta/app/config/types.go` lines 64–69 — the `Http` struct:
```go
type Http struct {
    IdleTimeout       time.Duration `yaml:"idleTimeout"`
    ReadTimeout       time.Duration `yaml:"readTimeout"`
    ReadHeaderTimeout time.Duration `yaml:"readHeaderTimeout"`
    WriteTimeout      time.Duration `yaml:"writeTimeout"`
}
```
No `MaxRequestBodySize` or equivalent field exists.

`rosetta/main.go` lines 220–227 — server construction:
```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```
Only the four timeout values are set. `http.Server` has no built-in body-size field; the standard Go mitigation is `http.MaxBytesReader` applied per-request or in middleware.

**Middleware chain** (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware`): a full grep for `MaxBytesReader`, `LimitReader`, `maxBodySize`, and `MaxBody` across all `rosetta/**/*.go` returns **zero matches**. None of the three middleware layers apply any body-size restriction.

**Root cause / failed assumption:** The design assumes that `ReadTimeout` (default 5 s) is sufficient to bound memory consumption. It is not: `ReadTimeout` is a wall-clock limit, not a byte limit. On a 1 Gbps link an attacker can deliver ~625 MB within the 5-second window per connection. Go's `net/http` does not buffer bodies automatically, but the Rosetta SDK handlers call `json.NewDecoder(r.Body).Decode(...)`, which reads the entire body into memory before returning. With N concurrent connections the in-process allocation is N × body_size.

### Impact Explanation
An attacker who exhausts the process heap triggers an OOM condition. The Go runtime either panics or the OS OOM-killer terminates the process. Either outcome drops every active TCP connection the Rosetta node holds, making the node unreachable to all peers and clients until it is restarted. Because the Rosetta API is the external interface for block/construction/network queries, this constitutes a complete, externally-induced service partition. Severity: **High** (full availability loss, no authentication required, no rate limiting present).

### Likelihood Explanation
The attack requires only the ability to open TCP connections to port 5700 (the default Rosetta port). No credentials, tokens, or prior knowledge of internal state are needed. The exploit is trivially scriptable with `curl` or any HTTP client capable of streaming a large body. It is repeatable: after the process restarts, the same attack can be re-launched immediately. In a public or semi-public deployment the port is reachable by any internet host.

### Recommendation
1. **Add a `MaxRequestBodySize` field to `Http`** in `rosetta/app/config/types.go` with a sensible default (e.g., 1 MB, well above any legitimate Rosetta JSON payload).
2. **Apply `http.MaxBytesReader` in a middleware** inserted before the router, e.g.:
   ```go
   func BodyLimitMiddleware(maxBytes int64, next http.Handler) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
           next.ServeHTTP(w, r)
       })
   }
   ```
   Wire it into the chain in `rosetta/main.go` before `metricsMiddleware`.
3. Optionally add a connection-count limit via `golang.org/x/net/netutil.LimitListener` to bound concurrent connections independently of body size.

### Proof of Concept
```bash
# Generate a 500 MB payload
dd if=/dev/zero bs=1M count=500 | tr '\0' 'A' > /tmp/bigbody.bin

# Open 20 concurrent POST requests to any Rosetta endpoint
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/network/list \
       -H "Content-Type: application/json" \
       --data-binary @/tmp/bigbody.bin &
done
wait
```
Each goroutine in the server attempts to JSON-decode the body, allocating ~500 MB per connection. With 20 concurrent connections the process attempts to allocate ~10 GB, triggering OOM and crashing the server. All existing connections are dropped.
