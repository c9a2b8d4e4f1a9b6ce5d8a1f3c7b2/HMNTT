### Title
Unauthenticated gRPC Topic Subscription Flood Causes Unbounded DB Query and Log Amplification

### Summary
An unprivileged external user can repeatedly subscribe to the gRPC `ConsensusService` with a non-existent topic ID, triggering a database lookup and a `log.warn()` call on every attempt with no rate limiting or connection throttling in place. Because `Status.NOT_FOUND` is not filtered by the monitor's `shouldRetry` logic (only `INVALID_ARGUMENT` is excluded), internal subscribers also retry indefinitely. A sustained flood of such requests can amplify DB query load and logging overhead well beyond 30% of baseline.

### Finding Description

**Code path:**

1. External client calls `ConsensusController` → `TopicMessageServiceImpl.subscribeTopic()`.
2. `topicExists()` at [1](#0-0)  calls `entityRepository.findById(topicId.getId())` — a live DB query per subscription attempt.
3. When the topic does not exist and `checkTopicExists=true` (the default), it emits `Mono.error(new EntityNotFoundException(topicId))`. [2](#0-1) 
4. `EntityNotFoundException` extends `MirrorNodeException` via `super(String.format(MESSAGE, entityId))`. [3](#0-2) 
5. `ProtoUtil.toStatusRuntimeException()` matches the `EntityNotFoundException` branch and calls `clientError()`, which executes `log.warn("Client error {}: {}", ...)` and returns `Status.NOT_FOUND`. [4](#0-3) 
6. The `GrpcSubscriber.shouldRetry()` only suppresses retries for `INVALID_ARGUMENT`; `NOT_FOUND` passes through, causing indefinite retries with exponential backoff that resets. [5](#0-4) 

**Root cause:** No rate limiting, no per-IP connection cap, and no negative-result caching exist in the gRPC module. [6](#0-5)  Every subscription attempt with a non-existent topic unconditionally hits the database and emits a log line.

**Note on constructor:** The question references `MirrorNodeException(Throwable)`, but `EntityNotFoundException` actually invokes `MirrorNodeException(String)`. [7](#0-6)  The attack path is otherwise accurate.

### Impact Explanation
Each malicious subscription attempt costs: one synchronous DB read (`entityRepository.findById`), one `log.warn()` call, and gRPC stream setup/teardown. With no throttle, a single attacker sending hundreds of concurrent streams can saturate DB connection pool slots and I/O threads. The `subscriberCount` gauge increments and decrements per attempt but provides no back-pressure. [8](#0-7)  Sustained at scale this constitutes a resource-exhaustion DoS exceeding the 30% overhead threshold.

### Likelihood Explanation
The gRPC `ConsensusService` endpoint is publicly reachable with no authentication. Any client with network access can open gRPC streams in a tight loop. The `maxAttempts` for the internal monitor subscriber is `Long.MAX_VALUE` with a max backoff of only 8 seconds, meaning even internal retry storms are possible. [9](#0-8)  External attackers need no credentials and no special tooling — standard gRPC clients suffice.

### Recommendation
1. Add per-IP or per-connection rate limiting at the Netty/gRPC interceptor layer (e.g., a `ServerInterceptor` with a token-bucket limiter).
2. Cache negative entity-lookup results (topic not found) with a short TTL to avoid a DB hit on every retry.
3. In `shouldRetry`, also exclude `NOT_FOUND` from retry to prevent the monitor from amplifying the problem. [5](#0-4) 
4. Enforce a maximum concurrent-subscriber cap in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge as a guard. [10](#0-9) 

### Proof of Concept
```
# Requires grpcurl or any gRPC client
# Replace <mirror-node-host>:5600 with the target

for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 999999999}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

Each concurrent stream causes `entityRepository.findById` to execute and `log.warn` to fire. Observe DB connection pool exhaustion and log throughput spike in server metrics. Repeat in a loop to sustain the load above the 30% baseline threshold.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
