### Title
Deleted Entity Resolution Bypass via Long-Zero EVM Address in `EntityIdServiceImpl.findByEvmAddress()`

### Summary
In `EntityIdServiceImpl.findByEvmAddress()`, `DomainUtils.fromEvmAddress()` is called first and short-circuits the `Optional.or()` chain for any long-zero EVM address, returning an `EntityId` directly from the encoded entity number without any database lookup. This completely bypasses the `deleted <> true` guard in `EntityRepository.findByEvmAddress()`, allowing a deleted entity to be resolved and associated with incoming transactions in the mirror node's database.

### Finding Description
**Code path:**

`EntityIdServiceImpl.java` lines 169–178:
```java
private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
    var id = Optional.ofNullable(DomainUtils.fromEvmAddress(evmAddress))
            .or(() -> entityRepository.findByEvmAddress(evmAddress).map(EntityId::of));
    ...
    return id;
}
```

`DomainUtils.java` lines 285–297:
```java
public static EntityId fromEvmAddress(byte[] evmAddress) {
    if (isLongZeroAddress(evmAddress)) {
        final var num = Longs.fromByteArray(Arrays.copyOfRange(evmAddress, 12, 20));
        return EntityId.of(commonProperties.getShard(), commonProperties.getRealm(), num);
    }
    return null;
}
```

`isLongZeroAddress()` returns `true` for any 20-byte address whose first 12 bytes are all zeros (`MIRROR_PREFIX = new byte[12]`).

**Root cause:** The `Optional.or()` short-circuit means the DB branch — `entityRepository.findByEvmAddress(evmAddress)` with its `deleted <> true` SQL predicate — is **never reached** when the input is a long-zero address. `DomainUtils.fromEvmAddress()` returns a non-null `EntityId` purely from arithmetic on the address bytes, with no liveness check.

**Exploit flow:**
1. Entity `0.0.65535` exists and is subsequently deleted (marked `deleted = true` in the `entity` table).
2. Attacker submits a Hedera transaction (e.g., a `ContractCall` or `CryptoTransfer`) referencing the EVM address `0x000000000000000000000000000000000000FFFF` (long-zero encoding of entity 65535).
3. The mirror node importer receives the record stream item and calls `EntityIdServiceImpl.lookup(ContractID)` → `findByEvmAddress(bytes, ...)`.
4. `DomainUtils.fromEvmAddress()` detects the long-zero pattern, extracts `num = 65535`, and returns `EntityId.of(0, 0, 65535)` immediately.
5. The `Optional.or()` branch is never evaluated; `entityRepository.findByEvmAddress()` is never called; the `deleted <> true` SQL check is never executed.
6. The importer records the transaction as associated with the deleted entity `0.0.65535`.

**Why existing checks fail:** The `deleted <> true` guard exists only inside the SQL query in `EntityRepository.findByEvmAddress()` (line 19–20 of `EntityRepository.java`). It is structurally unreachable for long-zero addresses because the calling code in `EntityIdServiceImpl` uses `Optional.ofNullable(...).or(...)`, and the first operand is non-null for all long-zero inputs.

### Impact Explanation
The mirror node incorrectly resolves and persists associations between live transactions and deleted entities. Downstream consumers of the mirror node REST API and web3 API receive stale/incorrect entity data — e.g., a deleted contract or account appears as a valid participant in recent transactions. This corrupts the mirror node's indexed state and can mislead applications that rely on it for balance queries, contract state, or transaction history. Severity is medium: the Hedera consensus layer is unaffected, but mirror node data integrity is compromised for any deleted entity reachable via a long-zero address.

### Likelihood Explanation
Any unprivileged user can submit a transaction to the Hedera network referencing an arbitrary long-zero EVM address. No special permissions, keys, or privileged access are required. The attacker only needs to know (or guess) the numeric ID of a deleted entity, which is trivially discoverable via the mirror node's own REST API. The attack is repeatable and cheap (a single transaction per target entity).

### Recommendation
Add a deletion check after resolving a long-zero address. The fix should be applied in `EntityIdServiceImpl.findByEvmAddress()`:

```java
private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
    Optional<EntityId> id;
    EntityId longZero = DomainUtils.fromEvmAddress(evmAddress);
    if (longZero != null) {
        // Verify the entity exists and is not deleted before trusting the long-zero decode
        id = entityRepository.findByEvmAddress(evmAddress)
                .map(EntityId::of)
                .or(() -> Optional.of(longZero)); // fallback only if no DB row (pre-genesis system entities)
    } else {
        id = entityRepository.findByEvmAddress(evmAddress).map(EntityId::of);
    }
    ...
}
```

Alternatively, add a dedicated `existsByIdAndNotDeleted(long id)` query and call it before returning the long-zero-decoded `EntityId`.

### Proof of Concept
1. Create entity `0.0.65535` on a Hedera testnet (e.g., deploy a contract).
2. Delete the entity (submit a `ContractDelete` transaction).
3. Confirm `deleted = true` in the mirror node DB: `SELECT deleted FROM entity WHERE id = <encoded_id>`.
4. Submit any transaction referencing `ContractID` with `evm_address = 0x000000000000000000000000000000000000FFFF`.
5. Observe the mirror node importer processes the transaction and calls `EntityIdServiceImpl.findByEvmAddress()`.
