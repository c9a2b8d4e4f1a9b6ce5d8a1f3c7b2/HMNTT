### Title
Unauthenticated DB Connection Pool Exhaustion via Deterministic Three-Part UNION Query on `/accounts/:id/nfts`

### Summary
An unprivileged external user can craft a single URL with four query parameters (`token.id=gte:X`, `token.id=lte:Y`, `serialnumber=gte:A`, `serialnumber=lte:B`) that deterministically forces `extractNftMultiUnionQuery()` to produce a three-sub-query `UNION ALL` — the most expensive query this endpoint can generate. Because the REST API has no application-level rate limiting and the default DB connection pool is capped at 10 connections, flooding the endpoint with concurrent such requests exhausts the pool and renders the mirror node REST API completely unresponsive.

### Finding Description

**Trigger path — three-part UNION is deterministically forced:**

In `rest/controllers/accountController.js` lines 34–82, `extractNftMultiUnionQuery()` parses filters and delegates to three helpers in `rest/controllers/baseController.js`:

- `getLowerFilters()` (line 131): returns non-empty when `primary.hasLower() && secondary.hasLower()` — satisfied by `token.id=gte:X` + `serialnumber=gte:A`.
- `getInnerFilters()` (line 154): returns non-empty when `primary.hasBound() && secondary.hasBound()` — satisfied by the same combination.
- `getUpperFilters()` (line 176): returns non-empty when `primary.hasUpper() && secondary.hasUpper()` — satisfied by `token.id=lte:Y` + `serialnumber=lte:B`.

`validateBounds()` (line 56) does not reject this combination; it only rejects `gt` (not `gte`) with a secondary lower bound, and `lt` (not `lte`) with a secondary upper bound. Using `gte`/`lte` on both dimensions passes all validation.

In `rest/service/nftService.js` lines 93–122, `getQuery()` then builds:

```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(...));
// → 3 sub-queries joined with UNION ALL
```

Each sub-query performs a full `SELECT … FROM nft LEFT JOIN entity … WHERE account_id = $1 … ORDER BY token_id, serial_number LIMIT $2`. With `limit=100` (the maximum), the outer query re-sorts and re-limits across all three result sets.

**No application-level rate limiting:**

`rest/server.js` registers: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — no rate-limiting middleware exists. The Traefik `inFlightReq`/`rateLimit` middleware is infrastructure-optional (gated on `{{ if and .Values.global.middleware .Values.middleware }}`). The throttling in `web3/ThrottleManagerImpl.java` applies only to the separate web3 Java service.

**DB pool is the bottleneck:**

`rest/dbpool.js` lines 7–16 configure the pool with defaults from configuration:
- `max: config.db.pool.maxConnections` → **10 connections** (default)
- `statement_timeout: config.db.pool.statementTimeout` → **20 000 ms**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` → **20 000 ms**

With 10 connections and a 20-second statement timeout, an attacker maintaining ≥10 concurrent slow three-part UNION requests saturates the pool. All subsequent requests queue for up to 20 seconds before failing, making the service appear fully down.

### Impact Explanation

The mirror node REST API becomes completely unresponsive to all users for the duration of the attack. Every endpoint that touches the DB (accounts, transactions, tokens, NFTs) is affected because they share the same 10-connection pool. The attack requires no authentication, no special account, and no prior knowledge beyond the public API documentation, which explicitly describes the `gte`/`lte` filter combinations.

### Likelihood Explanation

The attack is trivially reproducible: a single `curl` loop or any HTTP load tool (e.g., `ab`, `wrk`, `hey`) with 10–20 concurrent workers and the crafted URL is sufficient. No credentials, tokens, or privileged access are needed. The endpoint is publicly documented in `rest/api/v1/openapi.yml` lines 104–144. The filter combination that triggers the three-part UNION is explicitly described as a supported use case. Any motivated attacker can sustain the attack indefinitely at negligible cost.

### Recommendation

1. **Add application-level rate limiting** in `rest/server.js` (e.g., `express-rate-limit`) scoped per IP, before route handlers, independent of infrastructure middleware.
2. **Add a per-IP in-flight request cap** to prevent a single source from holding all pool connections.
3. **Increase `maxConnections`** or use a connection-queue timeout shorter than 20 seconds so queued requests fail fast rather than piling up.
4. **Add a query-complexity budget**: reject requests where both `token.id` and `serialnumber` carry range bounds simultaneously, or require authentication for the three-part UNION path.
5. **Reduce `statementTimeout`** for this endpoint to a value well below 20 seconds (e.g., 5 seconds) to limit how long each connection is held.

### Proof of Concept

```bash
# Craft the maximum-complexity request (three-part UNION, max limit)
URL="http://<mirror-node>/api/v1/accounts/0.0.2/nfts?\
token.id=gte:1&token.id=lte:9999999999&\
serialnumber=gte:1&serialnumber=lte:9999999999&\
limit=100"

# Flood with 20 concurrent workers (only 10 needed to saturate the pool)
hey -n 100000 -c 20 "$URL"

# Verify service is unresponsive to normal requests during the attack
curl -s "http://<mirror-node>/api/v1/accounts/0.0.2/nfts" \
  --max-time 5  # will time out
```

**What happens internally:**
1. Each request reaches `extractNftMultiUnionQuery()` → all three of `lower`, `inner`, `upper` are non-empty.
2. `NftService.getQuery()` emits a three-sub-query `UNION ALL` SQL statement.
3. The query acquires one of the 10 pool connections for up to 20 seconds.
4. With 10+ concurrent requests, the pool is exhausted; all other API requests queue and eventually time out. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

```javascript
  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
```
