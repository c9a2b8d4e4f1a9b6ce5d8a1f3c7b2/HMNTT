### Title
Unauthenticated DoS via Uncached Full Address Book DB Scan on Every `/network/status` Request

### Summary
`NetworkStatus()` in `rosetta/app/services/network_service.go` calls `addressBookEntryRepo.Entries(ctx)` on every request with no result caching at the service layer. The underlying `Entries()` implementation in `rosetta/app/persistence/address_book_entry.go` executes a multi-table JOIN query against the database on every invocation. Because the Traefik rate-limiting middleware is disabled by default, an unprivileged attacker can flood this public endpoint to exhaust DB resources and degrade performance for all API consumers.

### Finding Description
**Exact code path:**

`rosetta/app/services/network_service.go`, `NetworkStatus()`, line 77:
```go
peers, err := n.addressBookEntryRepo.Entries(ctx)
```
This calls the concrete implementation in `rosetta/app/persistence/address_book_entry.go`, `Entries()`, lines 57–83, which executes on every call:
```go
db.Raw(latestNodeServiceEndpoints, sql.Named("file_id", fileId)).Scan(&nodes)
```
The SQL query (`latestNodeServiceEndpoints`, lines 18–30) is a multi-table JOIN with a correlated subquery:
```sql
select ... from (
  select max(start_consensus_timestamp) from address_book where file_id = @file_id
) current
join address_book_entry abe on abe.consensus_timestamp = current.max
left join node n on n.node_id = abe.node_id
left join address_book_service_endpoint abse on ...
group by ...
```
This runs up to twice per request (once for file 101, once for file 102 as fallback). There is **no in-memory cache, no TTL, and no deduplication** at the service or persistence layer in the Rosetta Go codebase.

**Root cause / failed assumption:** The design assumes either (a) the endpoint is called infrequently, or (b) an external rate limiter is always active. Neither assumption is enforced in code.

**Why existing checks fail:**
The Traefik middleware chain (`inFlightReq: amount: 5` per IP, `rateLimit: average: 10` per requestHost) is defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166, but the middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3) gates on `{{ if and .Values.global.middleware .Values.middleware }}`. The global flag defaults to `false` (line 95: `middleware: false`), so **the entire middleware chain is disabled by default**. There is no application-level throttle, concurrency limit, or connection-pool guard specific to this endpoint in the Go server.

### Impact Explanation
An attacker sending a sustained stream of concurrent POST requests to `/network/status` causes repeated expensive DB scans. Each request holds a DB connection for the duration of the multi-table JOIN. Under sufficient concurrency, the DB connection pool is exhausted, query latency rises for all other API consumers (block queries, account queries, etc.), and the service becomes effectively unavailable. This is a griefing/availability impact with no economic damage to network participants, consistent with the Medium classification.

### Likelihood Explanation
The endpoint is fully public and unauthenticated — no API key, session, or credential is required. The Rosetta spec requires POST with a minimal JSON body (`{"network_identifier": {...}}`), which any HTTP client can produce. The default deployment has no rate limiting active. A single attacker with a modest number of concurrent connections (e.g., 50–100) is sufficient to saturate a typical DB connection pool. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add a service-layer cache** for the result of `Entries()` with a short TTL (e.g., 30–60 seconds). The address book changes rarely; a stale-while-revalidate pattern is appropriate.
2. **Enable the Traefik middleware by default** by changing `global.middleware` default to `true`, or enforce per-IP rate limiting unconditionally.
3. **Add application-level concurrency limiting** (e.g., a semaphore or `golang.org/x/time/rate` token bucket) inside `NetworkStatus()` or as HTTP middleware in `rosetta/main.go`, independent of the ingress layer.

### Proof of Concept
```bash
# Send 100 concurrent /network/status requests with no authentication
for i in $(seq 1 100); do
  curl -s -X POST http://<rosetta-host>/network/status \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' &
done
wait
# Each request triggers Entries() -> two potential multi-table JOIN queries
# DB connection pool exhaustion causes latency spikes / errors for all other consumers
```
Preconditions: Rosetta running in online mode, default Helm deployment (middleware disabled). No credentials needed.
