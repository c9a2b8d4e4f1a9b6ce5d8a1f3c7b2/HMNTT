### Title
Query String Bypass of `internalPaths` Debug-Level Suppression in `TracingMiddleware`

### Summary
`TracingMiddleware` uses `request.URL.RequestURI()` to obtain the request path, which includes any query string. The `internalPaths` map contains only bare paths (e.g., `/health/readiness`). Any request to `/health/readiness?anything=1` produces a path string that does not match the map key, causing the request to be logged at `Info` level instead of `Debug`, polluting the operational/transaction log with health-probe noise that any unauthenticated external user can inject at will.

### Finding Description
**File:** `rosetta/app/middleware/trace.go`, function `TracingMiddleware()`, lines 47 and 55.

```
path := request.URL.RequestURI()   // line 47 — returns "/health/readiness?block=999"
...
if internalPaths[path] {           // line 55 — map key is "/health/readiness" → miss
    log.Debug(message)
} else {
    log.Info(message)              // executed instead
}
```

`net/http`'s `RequestURI()` returns the raw, unmodified request target including the query string. The `internalPaths` map (line 19) is populated with the bare constants `livenessPath = "/health/liveness"`, `readinessPath = "/health/readiness"`, and `metricsPath`. A lookup of `/health/readiness?block=999` against a key `/health/readiness` returns `false`, so the `else` branch fires and the entry is written to the `Info`-level log.

The Go HTTP router routes on the path component only, so the underlying health handler still responds 200 OK — the request is fully processed and logged at `Info`.

The existing test suite (trace_test.go line 49) tests a *different* path (`/metricss`) to confirm Info-level logging for non-internal paths, but no test covers an internal path with a query string appended, leaving this bypass undetected.

### Impact Explanation
The `Info`-level log is the operational record of Rosetta API transactions. An attacker can continuously inject entries of the form:

```
<attacker-IP> GET /health/readiness?block=999 (200) in Xms
```

at arbitrary volume. This:
- Pollutes the transaction history log, making forensic analysis and anomaly detection harder.
- Can be used to obscure real transaction events by burying them in health-probe noise.
- Allows an attacker to insert misleading path strings (e.g., `?block=<number>`) that resemble transaction-related parameters, confusing log consumers or SIEM rules that parse the path field.

No funds are directly stolen, but the integrity and reliability of the audit/transaction log is degraded — matching the "Reorganizing transaction history without direct theft of funds" severity category.

### Likelihood Explanation
- **No authentication required.** Health endpoints are intentionally publicly reachable.
- **Zero preconditions.** Any HTTP client can append a query string.
- **Trivially repeatable.** A simple loop or a single `curl` command suffices.
- **No rate limiting** is visible in the middleware stack for health paths.

### Recommendation
Replace `request.URL.RequestURI()` with `request.URL.Path` for the `internalPaths` lookup, while keeping `RequestURI()` only for the log message if the full URI is desired:

```go
path    := request.URL.RequestURI()   // keep for logging
rawPath := request.URL.Path           // use for map lookup only

if internalPaths[rawPath] {
    log.Debug(message)
} else {
    log.Info(message)
}
```

`URL.Path` contains only the decoded path component, never the query string, so no query-string variant can bypass the check.

### Proof of Concept
```bash
# Normal readiness probe — logged at Debug, invisible in Info log
curl -s http://<rosetta-host>/health/readiness

# Bypass — logged at Info, appears in transaction log
curl -s "http://<rosetta-host>/health/readiness?block=999"

# Flood the Info log with fake entries
for i in $(seq 1 10000); do
  curl -s "http://<rosetta-host>/health/readiness?x=$i" &
done
```

Observe that the second and third commands produce `level=info` entries in the log while the first produces only `level=debug` (or nothing, if the log level is Info). The `internalPaths` check is confirmed bypassed at: [1](#0-0) 

with the path constants defined at: [2](#0-1)
