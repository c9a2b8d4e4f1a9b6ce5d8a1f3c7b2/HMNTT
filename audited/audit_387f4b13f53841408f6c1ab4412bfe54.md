### Title
In-Memory Gas Rate Limiter Bypass via Multi-Instance Horizontal Scaling

### Summary
The `gasLimitBucket` in `ThrottleConfiguration.gasLimitBucket()` is a JVM-local Bucket4j instance with no distributed backend. In any horizontally-scaled deployment with N web3 service instances, each instance independently enforces the full `gasPerSecond` budget, allowing an unprivileged attacker to drive N× the intended aggregate gas execution per second by saturating each instance through a standard load balancer.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 34–45 construct the bucket:
```java
return Bucket.builder()
    .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
    .addLimit(limit)
    .build();
```
`SynchronizationStrategy.SYNCHRONIZED` is a Bucket4j local strategy that only provides `synchronized`-block thread safety within a single JVM. It does not use any distributed store (Redis, Hazelcast, etc.). The bucket is a plain in-memory object scoped to one Spring application context.

`ThrottleManagerImpl.java` line 40 consumes from this local bucket on every request:
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
```

**Root cause / failed assumption:** The design assumes a single instance enforces a global gas budget. In a multi-instance deployment, each instance holds an independent bucket initialized to the full `gasPerSecond` capacity (`7_500_000_000` gas/sec default, scaled to `750_000` tokens). There is no shared state, no cross-instance token deduction, and no coordination layer.

**Exploit flow:**
1. Attacker observes the service is deployed behind a load balancer with N replicas (standard Kubernetes/cloud deployment).
2. Attacker sends a continuous stream of requests with `gas = 15_000_000` (the configured `maxGasLimit`) to the load balancer endpoint — no credentials required.
3. The load balancer distributes requests round-robin across N instances.
4. Each instance independently allows up to `7_500_000_000` gas/sec. Instance 1 processes its full budget; Instance 2 processes its full budget; etc.
5. Total gas executed per second = N × 7,500,000,000 — the intended cap is exceeded by `(N-1)/1 × 100%`.

**Why existing checks fail:**
- `SynchronizationStrategy.SYNCHRONIZED` (line 42) only prevents race conditions between threads within one JVM — it is explicitly documented by Bucket4j as a local strategy.
- The `rateLimitBucket` (line 25–32) and `opcodeRateLimitBucket` (lines 47–55) share the same architectural flaw.
- There is no IP-based or session-based sticky routing that would confine a single attacker to one instance.
- The `scaleGas` division by `10_000` (ThrottleProperties line 46) is a Bucket4j capacity workaround, not a security control.

### Impact Explanation
With 2 instances (the minimum for any HA deployment), aggregate EVM execution is 2× the intended limit — a 100% increase, far exceeding the 30% threshold. With 3 instances it is 200% over. EVM execution is CPU- and memory-intensive; sustained over-limit execution degrades database query throughput, increases JVM GC pressure, and can exhaust thread pools across all instances simultaneously. The impact scales linearly with replica count, which operators increase precisely under load — making the attack self-amplifying.

### Likelihood Explanation
No authentication, API key, or special network access is required. The `/api/v1/contracts/call` endpoint is public. The attacker needs only the load balancer's public hostname and knowledge that the service scales horizontally (observable from response latency variance or public deployment documentation). The attack is repeatable indefinitely, requires no exploit tooling beyond `curl` or any HTTP client, and is not detectable as "brute force" since each individual request is within per-instance limits and returns HTTP 200.

### Recommendation
Replace the local Bucket4j bucket with a distributed rate-limiting backend:

1. **Bucket4j + Redis/Hazelcast**: Use `Bucket4j`'s `ProxyManager` with a Redis or Hazelcast backend so all instances share a single token pool. This is a drop-in replacement requiring only configuration changes in `ThrottleConfiguration.gasLimitBucket()`.
2. **API Gateway-level throttling**: Enforce gas-equivalent request throttling at the ingress/gateway layer (e.g., Kong, Envoy, AWS API Gateway) before requests reach individual instances.
3. **Sticky sessions by client IP**: As a partial mitigation only, configure the load balancer to route by client IP so one attacker is confined to one instance — but this does not prevent distributed attacks from multiple IPs.

The recommended fix is option 1: replace `Bucket.builder().withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)` with a Bucket4j `ProxyManager` backed by a shared store.

### Proof of Concept
```bash
# Assume 2 web3 instances behind load balancer at https://web3.example.com
# gasPerSecond default = 7,500,000,000; maxGasLimit = 15,000,000

# Terminal 1 — saturate instance A's budget
while true; do
  curl -s -X POST https://web3.example.com/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}' &
done

# Terminal 2 — saturate instance B's budget simultaneously
while true; do
  curl -s -X POST https://web3.example.com/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}' &
done

# Result: both instances process up to 7.5B gas/sec each = 15B gas/sec total
# = 100% above the intended 7.5B/sec cap, with zero throttle errors returned
# to the attacker since each instance sees only its own local bucket state.
