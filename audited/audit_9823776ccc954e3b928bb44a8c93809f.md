### Title
Missing Inverted Range Validation for `serial_number` Bound Silently Returns Zero NFT Airdrop Records

### Summary
The `Bound` constructor in `Bound.java` only validates that `adjustedLower > adjustedUpper` (inverted range) when `primarySortField == true`. The `serialNumbers` bound is always constructed with `primarySortField=false` in `TokenAirdropsController`, so an inverted range like `serial_number=gte:100&serial_number=lte:1` passes construction without error. The resulting impossible SQL condition (`serial_number >= 100 AND serial_number <= 1`) silently returns zero rows instead of rejecting the request, causing all matching NFT airdrop records to be absent from the API response.

### Finding Description

**Exact code path:**

In `TokenAirdropsController.java` line 103, the `serialNumbers` bound is constructed with `primarySortField=false`:
```java
.serialNumbers(new Bound(serialNumbers, false, SERIAL_NUMBER, TOKEN_AIRDROP.SERIAL_NUMBER))
```

In `Bound.java` lines 56–60, the inversion check is gated on `primarySortField`:
```java
long adjustedLower = getAdjustedLowerRangeValue();
long adjustedUpper = adjustUpperBound();
if (primarySortField && adjustedLower > adjustedUpper) {   // <-- never true for serialNumbers
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
```

Because `primarySortField` is `false` for `serialNumbers`, the `IllegalArgumentException` is never thrown for an inverted serial number range.

**Downstream effect in `JooqRepository.java` lines 32–35:**
```java
private Condition getCondition(Bound bound) {
    var field = bound.getField();
    return getCondition(bound.getLower(), field).and(getCondition(bound.getUpper(), field));
}
```
For `serial_number=gte:100&serial_number=lte:1`, this emits `serial_number >= 100 AND serial_number <= 1` — a logically impossible predicate that matches no rows.

**When `serialNumbers` is the only filter** (`getBounds()` in `TokenAirdropRequest.java` line 68 returns `List.of(serialNumbers)`), `getBoundConditions` with a single-element list falls into the `bounds.size() == 1` branch (line 73–74 of `JooqRepository.java`) and directly calls `getCondition(primary)`, producing the impossible condition and returning zero rows.

**Contrast with `entityIds`** (line 97 of the controller), which is constructed with `primarySortField=true` and therefore does get the inversion check.

### Impact Explanation
Any pending NFT airdrop records that exist in the database for the queried account are completely absent from the API response when an inverted `serial_number` range is supplied. The API returns HTTP 200 with an empty `airdrops` array and no indication of an error, making it indistinguishable from a legitimate "no results" response. Users or integrations relying on this endpoint to enumerate pending NFT airdrops will silently receive incomplete data, potentially causing missed claim actions or incorrect state assumptions.

### Likelihood Explanation
No authentication or special privilege is required. Any HTTP client can send the crafted request to the public mirror node REST API. The parameters are standard query string values accepted by the `/pending` and `/outstanding` endpoints. The attack is trivially repeatable and requires no knowledge of the target account's actual airdrop data — the inverted range universally suppresses all results regardless of what records exist.

### Recommendation
Remove the `primarySortField` gate from the inversion check in `Bound.java`. The validation should apply unconditionally whenever both a lower and upper bound are present:

```java
// Current (broken):
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}

// Fixed:
if (adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}
```

This ensures that any caller — regardless of whether the bound is a primary or secondary sort field — receives a `400 Bad Request` for an inverted range instead of a silent empty result.

### Proof of Concept

**Precondition:** Account `0.0.12345` has at least one pending NFT airdrop with a serial number between 1 and 50.

**Step 1 — Confirm records exist (valid query):**
```
GET /api/v1/accounts/0.0.12345/airdrops/pending?serial_number=gte:1&serial_number=lte:50
```
Expected: returns one or more airdrop records.

**Step 2 — Trigger the bug (inverted range):**
```
GET /api/v1/accounts/0.0.12345/airdrops/pending?serial_number=gte:50&serial_number=lte:1
```
Expected (correct): HTTP 400 with "Invalid range provided for serial_number".
Actual: HTTP 200 with `{"airdrops": [], "links": {"next": null}}` — all records silently omitted.
