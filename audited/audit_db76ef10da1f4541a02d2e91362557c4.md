### Title
Unauthenticated Repeated Calls to `GET /api/v1/network/supply?timestamp=lte:<now>` Trigger Unbounded `DISTINCT ON` Table Scans, Enabling DB Resource Exhaustion

### Summary
The `getSupply()` method in `NetworkServiceImpl` contains no rate limiting, no caching, and no lower-bound enforcement when a user supplies only an upper-bound timestamp. Any unauthenticated caller can repeatedly invoke `GET /api/v1/network/supply?timestamp=lte:<current_nanosecond>`, causing `getFirstDayOfMonth(maxTimestamp, -1)` to set the effective lower bound to the first day of the prior month and triggering a `DISTINCT ON (account_id)` sequential scan across up to two months of `account_balance` rows per request. With no throttle on this endpoint, an attacker can sustain this load to increase DB I/O well above the 30% threshold.

### Finding Description

**Exact code path:**

`NetworkController.java` line 132–137 accepts the request with no authentication or rate-limiting annotation: [1](#0-0) 

`NetworkServiceImpl.java` lines 66–81 branch on whether a timestamp is present. When only `lte:<T>` is supplied, `timestamp.getAdjustedLowerRangeValue()` returns `0` (no lower bound set), and `getFirstDayOfMonth(maxTimestamp, -1)` computes the first day of the prior calendar month: [2](#0-1) 

`getFirstDayOfMonth` with `monthOffset = -1` always produces a timestamp ~1–2 months before `maxTimestamp`: [3](#0-2) 

`AccountBalanceRepository.getSupplyHistory` then executes a `DISTINCT ON (ab.account_id)` with `ORDER BY ab.account_id ASC, ab.consensus_timestamp DESC` over the full `[first_day_of_prior_month, maxTimestamp]` range — potentially millions of rows: [4](#0-3) 

**Root cause:** The only guard is `minTimestamp > maxTimestamp` (line 73–75), which is never triggered when only an upper bound is provided. There is no rate limiter, no `@Cacheable`, and no per-IP throttle on this endpoint in the `rest-java` module. The throttle infrastructure found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` module's contract-call endpoints and is not applied here. [5](#0-4) 

### Impact Explanation
Each request with `timestamp=lte:<now>` forces a `DISTINCT ON` scan over all `account_balance` rows written in the prior ~30 days. On a production Hedera mirror node, the `account_balance` table contains hundreds of millions of rows. A sustained flood of such requests (even at modest concurrency — e.g., 10–20 req/s from a single host) can saturate PostgreSQL I/O bandwidth, inflate shared-buffer eviction, and degrade all other queries sharing the same DB instance. This directly satisfies the 30%+ resource-consumption threshold without brute-force volume, because each individual query is already expensive.

### Likelihood Explanation
The endpoint is publicly documented in the OpenAPI spec with no authentication requirement. The exploit requires zero privileges, zero tokens, and only knowledge of the current Unix nanosecond timestamp (trivially obtained). It is fully repeatable and scriptable. A single attacker with a standard HTTP client can sustain the attack indefinitely.

### Recommendation
1. **Add a rate limiter** to the `rest-java` `NetworkController` for the `/supply` endpoint (e.g., via a Spring `HandlerInterceptor` or a Bucket4j filter analogous to the `web3` throttle).
2. **Cache the timestamp-parameterized response**: apply `@Cacheable` with a short TTL (e.g., 15 seconds) keyed on the rounded/truncated timestamp to collapse repeated identical or near-identical queries.
3. **Enforce a maximum lookback window**: reject or clamp requests where `maxTimestamp - minTimestamp` exceeds a configurable threshold (e.g., 7 days) rather than always defaulting to the prior month.
4. **Require a lower-bound timestamp** when an upper-bound is provided, preventing the implicit `0 → getFirstDayOfMonth` expansion.

### Proof of Concept

```bash
# Get current nanosecond timestamp (Linux)
NOW_NS=$(date +%s%N)

# Single expensive request
curl "https://<mirror-node>/api/v1/network/supply?timestamp=lte:$(echo $NOW_NS | sed 's/.\{9\}$/.\0/')"

# Sustained attack (10 concurrent workers, no auth required)
seq 1 10 | xargs -P10 -I{} bash -c '
  while true; do
    NOW_NS=$(date +%s%N)
    SEC=${NOW_NS:0:10}
    NS=${NOW_NS:10}
    curl -s "https://<mirror-node>/api/v1/network/supply?timestamp=lte:${SEC}.${NS}" > /dev/null
  done
'
```

Each request causes PostgreSQL to execute:
```sql
SELECT DISTINCT ON (ab.account_id) ab.balance, ab.consensus_timestamp
FROM account_balance ab
JOIN unnest(...) AS ranges(min_val, max_val)
  ON ab.account_id BETWEEN ranges.min_val AND ranges.max_val
WHERE ab.consensus_timestamp BETWEEN <first_day_of_prior_month> AND <now>
ORDER BY ab.account_id ASC, ab.consensus_timestamp DESC;
```
with `lowerTimestamp` set to the first day of the prior month and `upperTimestamp` set to the current time — scanning the maximum possible recent data window on every call.

    private long getFirstDayOfMonth(long timestamp, int monthOffset) {
        final var instant = Instant.ofEpochSecond(0, timestamp);
        final var dateTime = instant.atZone(ZoneOffset.UTC);
        final var firstDay = dateTime.plusMonths(monthOffset).withDayOfMonth(1);

        return firstDay.toLocalDate().atStartOfDay(ZoneOffset.UTC).toEpochSecond() * DomainUtils.NANOS_PER_SECOND;
    }
