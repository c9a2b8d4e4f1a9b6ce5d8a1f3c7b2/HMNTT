### Title
Global-Only Rate Limiting in `throttle()` Enables Distributed Exhaustion (DDoS) with No Per-Client Mitigation

### Summary
`ThrottleManagerImpl.throttle()` enforces only application-wide singleton token buckets (`rateLimitBucket`, `gasLimitBucket`) with no per-IP, per-session, or per-identity tracking. Any unauthenticated caller can participate in a distributed attack where many source IPs collectively exhaust the global quota, denying service to all legitimate users while each individual attacker node stays well below any detectable threshold.

### Finding Description
**Exact code path:**

`ContractController.call()` (line 40) invokes `throttleManager.throttle(request)` passing only the request body — no client identity is extracted or forwarded. [1](#0-0) 

Inside `throttle()`, the only guards are two global singleton bucket checks: [2](#0-1) 

Both buckets are application-scoped Spring beans instantiated once for the entire JVM: [3](#0-2) [4](#0-3) 

**Root cause:** The `throttle()` method signature accepts a `ContractCallRequest` but never reads any client-identifying field (source IP, API key, session token). The `ContractCallRequest` viewmodel carries no such field, and no Spring Security configuration exists in the web3 module (confirmed: `Security*.java` returns no results under `web3/src/main/java/org/hiero/mirror/web3/config/`). The optional `request[]` filter rules in `ThrottleProperties` match on request *content* fields (DATA, GAS, TO, BLOCK, VALUE) — not on caller identity. [5](#0-4) 

**Why existing checks fail:** The global `rateLimitBucket` defaults to 500 RPS and `gasLimitBucket` to 7.5 billion gas/sec (scaled). These limits are shared across every caller. A botnet of N nodes each sending R requests/sec exhausts the bucket when N×R > 500. With `maxGasLimit = 15,000,000` per request, only ~500 max-gas requests/sec saturate the gas bucket — achievable by a botnet of 50 nodes each sending 10 RPS. Each individual node is indistinguishable from a normal user. [6](#0-5) 

### Impact Explanation
Once either global bucket is exhausted, every subsequent request — including those from legitimate users — receives a `ThrottleException` (HTTP 429-equivalent). The service is effectively unavailable for the duration of the attack. Because the gas bucket is consumed proportionally to the `gas` field in the request body (which the attacker controls up to `maxGasLimit = 15,000,000`), an attacker can amplify gas-bucket exhaustion by always submitting max-gas requests, requiring far fewer source IPs to saturate the gas limit than the request-rate limit. [7](#0-6) 

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint requires no authentication. Any unprivileged actor with access to a modest botnet (50–100 nodes) or a cloud provider with multiple egress IPs can execute this attack. The attack is repeatable indefinitely since the buckets refill every second, meaning the attacker only needs to sustain the rate — not accumulate tokens. No special knowledge of the system internals is required beyond knowing the public API endpoint. [8](#0-7) 

### Recommendation
1. **Per-IP rate limiting**: Introduce a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `HttpServletRequest.getRemoteAddr()` or a trusted `X-Forwarded-For` header) so each source IP has its own token bucket, preventing one IP cluster from starving others.
2. **Pass client identity into `throttle()`**: Extend the `ThrottleManager.throttle()` interface to accept a client key, enabling per-client enforcement alongside the existing global cap.
3. **Deploy an upstream WAF/API gateway** with per-IP rate limiting as a defense-in-depth layer before requests reach the application.
4. **Cap the gas amplification vector**: Consider capping the gas consumed from the bucket at a lower value regardless of the requested `gas` field, or require proof-of-work/authentication for high-gas requests.

### Proof of Concept
```bash
# Attacker controls 50 IPs (e.g., cloud VMs or proxies)
# Each sends 15 requests/sec with max gas = 15,000,000
# Collective: 50 * 15 = 750 RPS > 500 RPS global limit
# Gas consumed per second: 50 * 15 * 15,000,000 = 11,250,000,000 > 7,500,000,000 gas/sec limit

# From each attacker node (different source IP):
for i in $(seq 1 15); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"0xdeadbeef","to":"0x0000000000000000000000000000000000000001","gas":15000000}' &
done

# Legitimate user on a separate IP simultaneously:
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# Result: ThrottleException — "Gas per second rate limit exceeded." or
#         "Requests per second rate limit exceeded"
# Legitimate user is denied despite sending a single low-gas request.
``` [9](#0-8)

            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
