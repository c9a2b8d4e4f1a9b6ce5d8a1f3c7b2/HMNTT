### Title
Unauthenticated Unbounded Enumeration of Crypto Allowances via Pagination — No Rate Limiting on REST API

### Summary
The `/api/v1/accounts/{id}/allowances/crypto` endpoint accepts `limit=1` from any unauthenticated caller and returns a `next` pagination cursor when results equal the requested limit. The REST API middleware stack contains no IP-based rate limiting, meaning an attacker can iterate through every crypto allowance (spender identity + authorized HBAR amount) for any target account one record at a time, indefinitely, without being throttled or blocked.

### Finding Description

**Route registration** — `rest/routes/accountRoute.js:17`: [1](#0-0) 

**Handler** — `rest/controllers/cryptoAllowanceController.js:76-98`: [2](#0-1) 

The handler accepts `limit`, `order`, and `spender.id` from `acceptedCryptoAllowanceParameters` (line 101). When `response.allowances.length === limit`, it emits a `next` link keyed on the last spender ID: [3](#0-2) 

`limit=1` is a valid value — `getLimitParamValue` only caps values *above* `responseLimit.max`; values at or below it pass through unchanged: [4](#0-3) 

**Middleware stack** — `rest/server.js:67-98`: [5](#0-4) 

The only "access control" middleware is `authHandler`, which is entirely optional — if no `Authorization` header is present it returns immediately without blocking the request: [6](#0-5) 

Its sole purpose is to *elevate* the response limit for known users; it provides no access gate for unauthenticated callers: [7](#0-6) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` that does exist in the codebase lives exclusively in the `web3` module and is never wired into the REST API: [8](#0-7) 

**Root cause**: The REST API was designed as a public read-only mirror with no IP-level rate limiting. The `authHandler` was added to allow privileged users a *higher* limit, but this left the lower bound (including `limit=1`) completely unguarded for anonymous callers.

### Impact Explanation
An attacker can fully enumerate every spender account that holds a crypto allowance against any target account, along with the exact authorized HBAR amounts and grant timestamps. This reveals the complete financial authorization graph of any account. Beyond information disclosure, issuing thousands of `limit=1` requests in rapid succession generates one database query per request against the `crypto_allowance` table, enabling a low-cost resource-exhaustion attack against the mirror node's PostgreSQL connection pool (`db.pool.maxConnections`).

### Likelihood Explanation
No special privileges, credentials, or network position are required. The endpoint is publicly reachable. The `next` link is machine-readable JSON, making full automation trivial with a simple loop. Any account with a large number of allowances (e.g., a DeFi protocol owner) is a high-value target. The attack is repeatable and stateless.

### Recommendation
1. Add an IP-based rate-limiting middleware (e.g., `express-rate-limit`) to the REST API server before the route handlers, applied globally or specifically to the `/allowances/*` sub-paths.
2. Enforce a minimum allowed `limit` value of, for example, 10 for unauthenticated callers, so that full enumeration requires proportionally fewer requests and each request carries more DB cost relative to network cost.
3. Consider requiring authentication (or an API key) for allowance enumeration endpoints if the data is considered sensitive in the deployment context.

### Proof of Concept
```bash
# Step 1 – first page, one record
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/allowances/crypto?limit=1&order=asc"
# Response contains:
# { "allowances": [{ "spender": "0.0.2001", "amount": 1000, ... }],
#   "links": { "next": "/api/v1/accounts/0.0.1001/allowances/crypto?limit=1&order=asc&spender.id=gt:0.0.2001" } }

# Step 2 – follow next link
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/allowances/crypto?limit=1&order=asc&spender.id=gt:0.0.2001"
# ... repeat until "next": null

# Automated loop (bash)
NEXT="/api/v1/accounts/0.0.1001/allowances/crypto?limit=1&order=asc"
while [ "$NEXT" != "null" ]; do
  RESP=$(curl -s "https://<mirror-node>${NEXT}")
  echo "$RESP" | jq '.allowances[]'
  NEXT=$(echo "$RESP" | jq -r '.links.next')
done
```
No credentials are supplied at any step; every request returns HTTP 200 with one allowance record and a cursor to the next.

    const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
    const filters = utils.buildAndValidateFilters(req.query, acceptedCryptoAllowanceParameters);

      links: {
        next: null,
      },
    };

      const lastValues = {
        [filterKeys.SPENDER_ID]: lastRow.spender,
      };
      response.links.next = utils.getPaginationLink(req, false, lastValues, order);
    }

    res.locals[responseDataLabel] = response;
  };

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
