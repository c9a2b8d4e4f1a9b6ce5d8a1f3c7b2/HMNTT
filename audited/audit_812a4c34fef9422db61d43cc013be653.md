### Title
Unbounded `hex.DecodeString` Allocation in `FindByHashInBlock` Enables Memory-Exhaustion DoS

### Summary
`FindByHashInBlock()` in `rosetta/app/persistence/transaction.go` passes the caller-supplied `hashStr` directly to `hex.DecodeString` with no length guard, allocating a byte slice proportional to the input size. Because the HTTP server applies no body-size limit and the Traefik rate-limiting middleware is disabled by default, an unauthenticated attacker can send concurrent requests with arbitrarily large `transaction_identifier.hash` values to exhaust process memory and crash the node.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go` line 89 passes `request.TransactionIdentifier.Hash` (a raw JSON string field) directly to `base_service.FindByHashInBlock`, which delegates to `transactionRepository.FindByHashInBlock`.

At `rosetta/app/persistence/transaction.go` line 180:
```go
transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
```
`tools.SafeRemoveHexPrefix` (in `rosetta/app/tools/hex.go` lines 18-23) only strips the `0x` prefix — it performs no length check. `hex.DecodeString` then allocates a `[]byte` of `len(hashStr)/2` bytes. There is no maximum-length check anywhere in the call chain before this allocation.

**Root cause / failed assumption:**
The code assumes callers supply a hash of reasonable length (e.g., 96 hex chars for a 48-byte SHA-384). No invariant enforces this.

**Why existing checks are insufficient:**

1. **HTTP `ReadTimeout` (5 s default, `main.go` line 225):** At 1 Gbps, 5 seconds allows ~625 MB of body data. `hex.DecodeString` would allocate ~312 MB per request on top of the ~625 MB already held by the JSON decoder — ~937 MB per request.

2. **No `http.MaxBytesReader`:** `main.go` lines 220-227 configure only timeout fields on `http.Server`; no `http.MaxBytesReader` wraps the request body anywhere in the middleware chain (`middleware/metrics.go`, `middleware/trace.go`).

3. **Traefik middleware disabled by default:** `charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false`. The middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3) requires `and .Values.global.middleware .Values.middleware` to be true before the `inFlightReq` (5 concurrent/IP) and `rateLimit` (10 req/s) rules are deployed. In the default configuration they are absent.

### Impact Explanation
An attacker can exhaust the Go process's heap by sending a small number of concurrent POST requests to the public `/block/transaction` endpoint, each carrying a multi-megabyte `transaction_identifier.hash` string. Memory exhaustion causes an OOM kill or severe GC pressure, making the Rosetta API unavailable. Because the Rosetta mirror node is a read-only query interface used by exchanges and wallets to track HBAR balances, its unavailability directly disrupts asset custody and transfer operations.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. No special knowledge of the network state is required — any valid-looking (but oversized) hex string triggers the allocation. The attack is trivially repeatable with a single `curl` loop or any HTTP load tool. The only natural friction is the 5-second `ReadTimeout`, which still permits very large payloads on fast links and does not prevent concurrent connections.

### Recommendation
Apply all three layers of defense:

1. **Input length guard in `FindByHashInBlock`** — reject `hashStr` whose stripped length exceeds the maximum expected hash size (e.g., 96 hex chars for SHA-384):
   ```go
   stripped := tools.SafeRemoveHexPrefix(hashStr)
   if len(stripped) > maxTransactionHashHexLen {
       return nil, hErrors.ErrInvalidTransactionIdentifier
   }
   transactionHash, err := hex.DecodeString(stripped)
   ```

2. **HTTP body-size limit** — wrap the request body in `main.go` before routing:
   ```go
   handler = http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
       r.Body = http.MaxBytesReader(w, r.Body, maxBodyBytes)
       next.ServeHTTP(w, r)
   })
   ```

3. **Enable Traefik middleware by default** — set `global.middleware: true` in the Helm chart defaults so `inFlightReq` and `rateLimit` are active out of the box.

### Proof of Concept

```bash
# Generate a 10 MB hex string (valid hex, wrong length → allocation before rejection)
LARGE_HASH=$(python3 -c "print('aa' * 5_000_000)")

# Send 20 concurrent requests to exhaust memory
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"block_identifier\": {\"index\": 1, \"hash\": \"0xaabbcc\"},
      \"transaction_identifier\": {\"hash\": \"0x${LARGE_HASH}\"}
    }" &
done
wait
# Expected result: process OOM-killed or severe latency spike on the Rosetta node
```

Each request causes the JSON decoder to hold ~10 MB and `hex.DecodeString` to allocate an additional ~5 MB before returning `ErrInvalidTransactionIdentifier`. Twenty concurrent requests consume ~300 MB of heap in the allocation phase alone; scaling the string size or concurrency exhausts available memory.
