### Title
Unauthenticated Pagination Loop Exhausts DB Connection Pool via `generateNextLink()`

### Summary
`generateNextLink()` in `rest/controllers/blockController.js` unconditionally emits a `next` pagination link whenever a full page of results is returned. The REST API has no application-level rate limiting, and the default DB connection pool is only 10 connections. An unprivileged attacker issuing concurrent tight-loop pagination requests can saturate the pool, causing connection timeouts for all legitimate users across mirror node instances.

### Finding Description
**Exact code path:**

`generateNextLink()` (lines 90–99) returns a next link whenever `blocks.length === filters.limit`:
```js
generateNextLink = (req, blocks, filters) => {
  return blocks.length
    ? utils.getPaginationLink(
        req,
        blocks.length !== filters.limit,   // isEnd=false when full page returned
        {[filterKeys.BLOCK_NUMBER]: last(blocks).index},
        filters.order
      )
    : null;
};
```
Each follow-up request calls `getBlocks` → `RecordFileService.getBlocks()` → `super.getRows(query, params)`, which acquires a connection from the `pg` pool configured in `rest/dbpool.js`:
```js
const poolConfig = {
  max: config.db.pool.maxConnections,   // default: 10
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default: 20000ms
  ...
};
```
The documented default is `maxConnections = 10` (`hiero.mirror.rest.db.pool.maxConnections`).

**Middleware stack** (`rest/server.js` lines 68–144): `requestLogger` → `authHandler` → optional `metricsHandler` → optional `responseCacheCheckHandler` → routes → `responseHandler` → `handleError`. There is **no rate-limiting middleware** in this chain. A `grep` across all `rest/**/*.js` for `rateLimit|throttle` returns zero hits in production code.

The Traefik-level rate limit (`charts/hedera-mirror-rest/templates/middleware.yaml`) is gated on `{{ if and .Values.global.middleware .Values.middleware }}` — it is **opt-in** and not enforced at the application layer.

The throttle system found in `web3/src/main/java/.../ThrottleManagerImpl.java` applies only to the Java `web3` module, not to the Node.js REST service.

**Root cause:** `generateNextLink` assumes clients paginate at a reasonable rate. No server-side enforcement exists to prevent a client from following the emitted `next` link in a tight loop.

### Impact Explanation
With a default pool of 10 connections and a `statementTimeout` of 20 seconds, an attacker opening ~10–20 concurrent HTTP connections and immediately following each `next` link keeps all pool slots occupied. Legitimate requests queue and eventually time out with a 500 error. Because all REST API endpoints share the same pool, this degrades the entire mirror node REST service — not just the `/blocks` endpoint — satisfying the ≥30% processing degradation threshold. The attack is amplified by setting `limit=1` (maximizing the number of DB round-trips per unit time) or by running multiple attacker processes.

### Likelihood Explanation
No authentication is required. The `next` link is self-describing and machine-readable from the JSON response body. Any script that loops `while (next) { fetch(next) }` with concurrency ≥ 10 is sufficient. The attack is repeatable indefinitely, requires no special knowledge, and is trivially automatable from a single IP or distributed across IPs to bypass any upstream IP-based controls.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP rate-limit middleware (e.g., `express-rate-limit`) to `rest/server.js` before the route handlers, covering all paginated endpoints.
2. **Concurrency cap**: Enforce a maximum number of in-flight DB queries per source IP using a semaphore or middleware (e.g., `express-slow-down`).
3. **Reduce default pool size exposure**: Ensure `connectionTimeoutMillis` is set low enough to fail fast and free the queue, rather than holding slots for 20 seconds.
4. **Enable Traefik middleware by default**: Set `global.middleware: true` and configure `inFlightReq` and `rateLimit` in the REST Helm chart values as defaults, not opt-ins.
5. **Minimum page size floor**: Consider enforcing a minimum `limit` value to prevent `limit=1` amplification.

### Proof of Concept
```bash
# Step 1: Get first page and extract next link
NEXT=$(curl -s "http://<mirror-node>/api/v1/blocks?limit=1&order=asc" \
  | jq -r '.links.next')

# Step 2: Open 15 concurrent tight-loop clients
for i in $(seq 1 15); do
  (while true; do
    NEXT=$(curl -s "http://<mirror-node>${NEXT}" | jq -r '.links.next')
    [ "$NEXT" = "null" ] && break
  done) &
done

# Step 3: Observe legitimate requests timing out
curl -v "http://<mirror-node>/api/v1/blocks"
# Expected: 500 or connection timeout after pool exhaustion (~10 concurrent slots consumed)
```
