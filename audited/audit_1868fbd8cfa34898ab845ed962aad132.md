### Title
Misplaced `sync.Once` in `initGenesisRecordFile` Allows Concurrent Goroutines to Exhaust DB Connections via Repeated `selectGenesis` Execution

### Summary
In `rosetta/app/persistence/block.go`, the `initGenesisRecordFile()` function uses `sync.Once` only to guard the *write* to `br.genesisBlock`, but the expensive `selectGenesis` correlated-subquery is executed *before* the `sync.Once` block and is protected by nothing but an unsynchronized field read. Any number of concurrent goroutines can simultaneously pass the unprotected check at line 244 and each independently execute the expensive query, exhausting the database connection pool and CPU before a single result is ever stored.

### Finding Description

**Exact code path** — `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263:

```
243: func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
244:     if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {   // ← unsynchronized read
245:         return nil
246:     }
247:
248:     db, cancel := br.dbClient.GetDbWithContext(ctx)                      // ← N goroutines reach here
249:     defer cancel()
250:
251:     var rb recordBlock
252:     if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {  // ← N queries fired
253:         ...
255:     }
256:
257:     br.once.Do(func() {                                                  // ← only write is guarded
258:         br.genesisBlock = rb
259:     })
260: ...
263: }
```

**Root cause:** The `sync.Once` is placed *after* the expensive DB query (line 257), so it only serialises the assignment. The early-exit guard at line 244 reads `br.genesisBlock.ConsensusStart` without any mutex or atomic operation. Before genesis is initialised, every concurrent goroutine reads the sentinel value `-1`, passes the check, acquires a DB connection, and fires the full `selectGenesis` query (which itself contains two correlated subqueries). The `sync.Once` never prevents duplicate queries — it only prevents duplicate writes.

**There is also a Go data race:** line 244 reads `br.genesisBlock.ConsensusStart` while line 258 (inside `once.Do`) writes the entire `br.genesisBlock` struct, with no synchronisation between them. Under Go's memory model this is undefined behaviour.

**Exploit flow:**
1. Attacker sends N concurrent HTTP requests to any public Rosetta endpoint (e.g. `POST /rosetta/block`, `POST /rosetta/network/status`) immediately after service startup or during any period where genesis has not yet been cached.
2. Go's HTTP server dispatches each request to its own goroutine; all N goroutines call `initGenesisRecordFile()`.
3. All N goroutines read `br.genesisBlock.ConsensusStart == -1` at line 244 (no lock, no atomic), all pass the guard.
4. All N goroutines call `GetDbWithContext`, each acquiring a connection from the pool, and each execute the `selectGenesis` query with its two correlated subqueries against `record_file` and `account_balance`.
5. DB connection pool is exhausted; subsequent legitimate requests receive connection-pool errors or time out. CPU spikes from N simultaneous correlated subqueries.
6. `sync.Once` fires exactly once (line 257), storing one result; the other N-1 query results are discarded — but the damage (connection exhaustion, CPU) is already done.

**Why existing checks are insufficient:**
- The `sync.Once` at line 257 guards only the struct assignment, not the query.
- The early-exit at line 244 is an unsynchronised read — it cannot prevent concurrent goroutines from all observing the unset sentinel simultaneously.
- There is no rate-limiting, concurrency cap, or mutex around the query path.
- The Rosetta API has no authentication (confirmed in `rosetta/main.go` lines 111–119 and ingress config in `charts/hedera-mirror-rosetta/values.yaml` lines 119–133). [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
An unauthenticated attacker can exhaust the database connection pool and spike CPU by sending a burst of concurrent requests to any block/network/account endpoint during the startup window (or any window where genesis is not yet cached). This renders the Rosetta node unavailable to all other users. Because the `selectGenesis` query joins `record_file` with `account_balance` and uses two correlated subqueries, each execution is significantly more expensive than a simple lookup. The attack is repeatable: if the process is restarted (e.g. by a liveness probe failure caused by the same DoS), the window reopens. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
- The Rosetta API is publicly exposed via ingress with no authentication requirement.
- The attack requires only a standard HTTP client capable of sending concurrent POST requests — no credentials, no special knowledge.
- The vulnerable window exists at every process startup and persists until the first successful genesis query completes and `once.Do` fires.
- Kubernetes liveness/readiness probes can inadvertently extend or reopen the window if the DoS causes probe failures and pod restarts.
- The attack is trivially scriptable and repeatable. [6](#0-5) [7](#0-6) 

### Recommendation
Move the entire initialisation — query and assignment — inside `sync.Once`, and add an atomic or mutex-protected post-check so that after `once.Do` completes, callers that raced can still detect failure:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id",
            br.treasuryEntityId.EncodedId)).First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", rb.Index)
    })
    if initErr != nil {
        br.once = sync.Once{} // reset so next caller retries
        return initErr
    }
    if br.genesisBlock.ConsensusStart == genesisConsensusStartUnset {
        return hErrors.ErrNodeIsStarting
    }
    return nil
}
```

This ensures exactly one goroutine ever executes the expensive query, regardless of how many concurrent callers arrive before genesis is cached. [8](#0-7) 

### Proof of Concept

**Preconditions:** Rosetta node is starting up (genesis not yet cached); the `/rosetta/block` endpoint is reachable.

```bash
# Send 200 concurrent POST requests to /rosetta/block during startup
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>/rosetta/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1}}' &
done
wait
```

**Expected result (vulnerable):** All 200 goroutines pass the unsynchronised check at line 244, each acquires a DB connection and fires `selectGenesis`. The DB connection pool is exhausted; subsequent requests return errors or time out. DB CPU spikes to near 100%.

**Expected result (fixed):** Only one goroutine executes `selectGenesis`; all others block inside `sync.Once` and return immediately after the single query completes. [9](#0-8)
