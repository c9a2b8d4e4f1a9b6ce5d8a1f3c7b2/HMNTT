### Title
Log Injection via Unsanitized `X-Forwarded-For` / `X-Real-IP` Headers in `TracingMiddleware`

### Summary
`getClientIpAddress()` returns the raw, unvalidated value of the `X-Forwarded-For` or `X-Real-IP` header directly into the log message constructed by `TracingMiddleware`. Because no sanitization of newline characters, carriage returns, or ANSI escape sequences is performed before the value is passed to `log.Info()`, any unauthenticated HTTP client can inject arbitrary content into the log stream. This allows an attacker to forge, overwrite, or suppress log entries for `/construction/submit` requests, defeating automated alerting and audit trails.

### Finding Description
**Code path:**

- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74: reads `X-Real-IP` then `X-Forwarded-For` verbatim, with no length cap, character-class check, or control-character stripping.
- `TracingMiddleware()`, lines 52–53: interpolates the raw return value directly into the log message string via `fmt.Sprintf`.
- Lines 55–59: for any path not in `internalPaths` (which covers only `/health/liveness`, `/health/readiness`, `/metrics`), the message is emitted at `log.Info` level — `/construction/submit` always falls into this branch.

```go
// trace.go lines 52-58 — unsanitized clientIpAddress flows straight to log
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))
...
log.Info(message)   // no sanitization before emission
```

**Root cause:** The implicit assumption is that `X-Forwarded-For` / `X-Real-IP` contain only a valid IP address. The code never validates this. Logrus's `log.Info(string)` emits the string as-is; it performs no escaping of embedded newlines or ANSI sequences.

**Exploit flow:**

1. Attacker sends a `POST /construction/submit` request with a crafted header:
   ```
   X-Forwarded-For: 10.0.0.1\ntime="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 1ms"
   ```
2. `getClientIpAddress()` returns the entire string including the embedded `\n` and fake log line.
3. `fmt.Sprintf` embeds it at the start of `message`.
4. `log.Info(message)` writes two lines to the log sink: the injected fake entry and the real (now-mangled) entry.
5. A SIEM or log parser that splits on newlines sees a plausible, benign-looking entry and may discard or misattribute the remainder.

For terminal-based viewers, a carriage-return payload (`\r192.168.1.1`) causes the real IP to be overwritten on-screen, hiding the attacker's address entirely.

**Existing checks reviewed:** The test suite (`trace_test.go` lines 26–85) only exercises well-formed IP strings (`10.0.0.100`, `192.0.2.1`). There are no tests for control characters, ANSI sequences, or multi-line values. No middleware, no HTTP framework layer, and no logrus hook performs sanitization.

### Impact Explanation
An attacker submitting a tampered Hedera transaction via `/construction/submit` can simultaneously inject a forged "success from a benign IP" log entry and corrupt or suppress the real entry. Automated SIEM rules keyed on IP address, path, or status code will fire on the fake entry (or not fire at all if the real entry is truncated). This directly undermines the audit trail for the highest-risk Rosetta endpoint — the one that broadcasts signed transactions to the network — making forensic investigation and real-time alerting unreliable.

### Likelihood Explanation
The precondition is trivially met: any HTTP client reachable by the Rosetta node can set arbitrary headers. No authentication, no API key, and no network privilege is required. The attack is repeatable on every request and leaves no reliable trace in the log. Standard HTTP libraries (curl, Python `requests`, Go `net/http`) all allow setting `X-Forwarded-For` freely. The only environmental constraint is that the attacker must be able to reach the Rosetta port, which is the same requirement as submitting a transaction at all.

### Recommendation
Sanitize the IP string returned by `getClientIpAddress()` before it enters the log pipeline:

1. **Validate format**: after reading the header, parse the value with `net.ParseIP()` (or take only the first comma-separated token for `X-Forwarded-For` and parse that). Return `"invalid"` or the `RemoteAddr` fallback if parsing fails.
2. **Strip control characters**: at minimum, replace or remove `\r`, `\n`, `\x1b`, and other non-printable bytes before constructing the log message.
3. **Use structured logging fields** instead of a pre-formatted string, so logrus can safely encode each field independently:
   ```go
   log.WithFields(log.Fields{
       "ip":     clientIpAddress,
       "method": request.Method,
       "path":   path,
       "status": tracingResponseWriter.statusCode,
       "ms":     time.Since(start).Milliseconds(),
   }).Info("request")
   ```
   Structured (JSON) output encodes embedded newlines as `\n` literals, neutralising the injection.

### Proof of Concept
```bash
# Inject a fake benign log entry while submitting a transaction
curl -s -X POST http://<rosetta-host>/construction/submit \
  -H 'Content-Type: application/json' \
  -H $'X-Forwarded-For: 10.0.0.1\ntime="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 2ms"' \
  -d '{"network_identifier":{...},"signed_transaction":"<tampered_tx>"}'
```

**Expected log output (two lines emitted by a single request):**
```
time="..." level=info msg="10.0.0.1
time="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 2ms"
 POST /construction/submit (200) in 3ms"
```

A SIEM rule matching `level=info` + `/construction/submit` + `(200)` will match the injected fake line first; the real (malformed) line may be dropped as unparseable, effectively hiding the tampered submission. [1](#0-0) [2](#0-1) [3](#0-2)

func getClientIpAddress(r *http.Request) string {
	ipAddress := r.Header.Get(xRealIpHeader)

	if len(ipAddress) == 0 {
		ipAddress = r.Header.Get(xForwardedForHeader)
	}

	if len(ipAddress) == 0 {
		ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
	}

	return ipAddress
