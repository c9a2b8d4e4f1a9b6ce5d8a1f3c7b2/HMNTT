### Title
Unauthenticated Full-Range Key Scan DoS via Unrestricted Hook Storage Query Endpoint

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint, when called without any `key` parameter, defaults to a full key range scan (`0x00...00` to `0xFF...FF`) against the `hook_storage` table for the given `(ownerId, hookId)` pair. No rate limiting exists in the `rest-java` module, allowing any unauthenticated caller to issue a sustained flood of full-range index scans, degrading database availability for all other queries.

### Finding Description

**Code path:**

`HooksController.hookStorageChangeRequest()` initializes bounds to the full range when no `key` params are supplied: [1](#0-0) 

These defaults flow directly into `HookServiceImpl.getHookStorage()`, which, when `keys.isEmpty()`, unconditionally calls the repository with the full range: [2](#0-1) 

The repository method issues a `WHERE owner_id = ? AND hook_id = ? AND key BETWEEN ? AND ? AND deleted = false` query with `LIMIT`: [3](#0-2) 

**Root cause / failed assumption:** The design assumes callers will supply narrow key filters. When they do not, the full index range for the `(owner_id, hook_id)` partition is scanned. The `LIMIT` clause only caps returned rows; the database must still traverse all index entries (including logically-deleted ones) until it accumulates `limit` live rows. With many deleted entries, this scan can be arbitrarily expensive.

**Why existing checks are insufficient:**

- `@Max(MAX_LIMIT)` on `limit` caps the result set size but does not bound the index scan work.
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `keys` only limits the number of key parameters; an empty list is explicitly allowed and triggers the full-range path.
- The throttle/rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) exists only in the `web3` module, not in `rest-java`. [4](#0-3) 

No equivalent rate-limiting bean or filter is present in the `rest-java` module:


### Impact Explanation
An attacker can saturate the database connection pool and I/O bandwidth by issuing a high-frequency stream of unauthenticated requests to the storage endpoint with no key filter. Each request forces a full index range scan over all rows for the targeted `(owner_id, hook_id)` pair. This degrades query latency for all other users of the mirror node REST API. The impact is griefing/availability degradation with no direct economic loss to network participants, consistent with the Medium severity classification.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The exploit requires only a valid `ownerId` and `hookId` (both enumerable from other public endpoints) and a simple HTTP client capable of issuing repeated GET requests. No special privileges, tokens, or on-chain transactions are required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a bucket4j or similar rate-limiting filter in the `rest-java` module, mirroring the pattern already used in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`.
2. **Require a key filter or narrow the default range**: Reject requests that omit `key` entirely, or require at least one `key` parameter to prevent unbounded scans.
3. **Add a query timeout**: Configure a statement timeout at the JDBC/datasource level so runaway scans are killed before they exhaust DB resources.
4. **Index optimization**: Ensure a partial index on `(owner_id, hook_id, key) WHERE deleted = false` exists so the `deleted = false` filter is applied at the index level, reducing scan cost.

### Proof of Concept
```bash
# Enumerate a valid ownerId and hookId from the public hooks listing
OWNER=0.0.12345
HOOK_ID=1

# Issue a sustained flood of full-range scans with no key filter
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/$OWNER/hooks/$HOOK_ID/storage" &
done
```

Each request triggers `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(ownerId, hookId, 0x00...00, 0xFF...FF, page)` with no server-side rate check, causing repeated full-range index scans and progressive degradation of database response times for all concurrent users.
