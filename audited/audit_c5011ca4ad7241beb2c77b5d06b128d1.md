### Title
`inFlightCounter` Permanent Inflation via Aborted Connections Due to Missing `res.on('close')` Handler

### Summary
In `rest/middleware/metricsHandler.js`, `metricsMiddleware()` increments `inFlightCounter` on every incoming request but only decrements it inside a `res.on('finish')` listener. In Node.js, the `finish` event fires only when the response is fully flushed to the OS; if a client closes the TCP connection before the server finishes sending the response, `finish` never fires and the counter is never decremented. Any unprivileged external user can repeatedly trigger this by opening connections and immediately dropping them, permanently inflating `api_all_request_in_processing_total` until the process is restarted.

### Finding Description
**Exact code path:**

- `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, line 169: `inFlightCounter.add(1)` — incremented unconditionally for every non-metrics request.
- Lines 185–186: `res.on('finish', () => { inFlightCounter.add(-1); … })` — the **only** place the counter is decremented.
- No `res.on('close', …)` handler exists anywhere in the JS codebase (confirmed by exhaustive grep).

**Root cause:** The code assumes every request lifecycle ends with a `finish` event. This assumption is wrong. Node.js HTTP semantics define two distinct terminal events on `ServerResponse`:
- `finish` — emitted when all response data has been handed to the OS (i.e., the server successfully completed the response).
- `close` — emitted when the underlying socket is destroyed before `response.end()` was called or able to flush (i.e., the client disconnected early).

When a client aborts the connection mid-flight, only `close` fires; `finish` is never emitted. Because the decrement lives exclusively in the `finish` handler, `inFlightCounter` is incremented at line 169 and never decremented.

**Why existing checks are insufficient:** There are no fallback cleanup paths, no timeout-based decrement, no `req.on('close')` guard, and no `res.on('close')` handler anywhere in the middleware chain.

### Impact Explanation
Each aborted request permanently adds +1 to the `api_all_request_in_processing_total` UpDownCounter exported via Prometheus. The value is monotonically increasing and resets only on process restart. The Grafana dashboard (`charts/hedera-mirror-common/dashboards/hedera-mirror-rest.json`) visualises this metric. Operators and automated alerting systems relying on this gauge to assess REST API load will observe a continuously growing "in-flight" count that bears no relation to actual concurrency. This corrupts capacity planning, triggers false high-load alerts, and can cause operators to incorrectly throttle or block inbound connections — including those used for transaction submission paths that feed the gossip layer. The metric is permanently wrong for the lifetime of the process.

### Likelihood Explanation
No authentication, rate limiting, or special network position is required. Any HTTP client on the internet can send a GET request to any REST API endpoint and immediately close the TCP connection (e.g., `curl --max-time 0`, a raw socket send-then-RST, or a scripted loop). The attack is trivially repeatable, stateless, and requires no knowledge of the application beyond the public API surface. A single attacker with a modest script can inflate the counter by thousands per second.

### Recommendation
Replace the single `finish` listener with a one-shot handler that fires on whichever terminal event arrives first:

```js
let decremented = false;
const decrement = () => {
  if (!decremented) {
    decremented = true;
    inFlightCounter.add(-1);
  }
};
res.on('finish', () => {
  decrement();
  // … record other metrics …
});
res.on('close', decrement);
```

Alternatively, use the `res.on('close')` event exclusively, since Node.js guarantees `close` is always emitted after `finish` as well (in Node ≥ 18 `close` is emitted after `finish` for normal completions too), making it a safe single-event decrement point. The `once` helper from Node's `events` module can also be used to ensure idempotency.
