### Title
Integer Narrowing Cast Overflow in `poll()` Bypasses DB Query Page Size Limit, Enabling Unbounded Result Set Fetch

### Summary
In `PollingTopicMessageRetriever.poll()`, a user-supplied `limit` value greater than `Integer.MAX_VALUE` (e.g., `2147483648L`) is silently narrowed to a negative `int` via an unsafe cast. This negative value causes `hasLimit()` on the derived `newFilter` to return `false`, which suppresses the `setMaxResults()` call in the repository layer, issuing a SQL query with no `LIMIT` clause that loads the entire topic message table partition into JVM heap memory on every poll cycle.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` passes the raw proto `uint64 limit` directly to the filter builder with no upper-bound check: [1](#0-0) 

`TopicMessageFilter` only enforces `@Min(0)` — no `@Max` constraint — so any value in `(Integer.MAX_VALUE, Long.MAX_VALUE]` passes Bean Validation: [2](#0-1) 

Inside `poll()`, the narrowing cast `(int)(filter.getLimit() - context.getTotal().get())` overflows for any `limit > Integer.MAX_VALUE`: [3](#0-2) 

Concrete arithmetic:
- `limit = 2147483648L` (Integer.MAX_VALUE + 1)
- `(int)(2147483648L - 0L)` = `(int)0x80000000` = `-2147483648`
- `Math.min(-2147483648, 1000)` = `-2147483648`
- `newFilter.limit` = `-2147483648L`

The derived `newFilter` is built via the Lombok builder's `build()` — not a Spring-managed method — so `@Validated` / Bean Validation is **not** re-triggered: [4](#0-3) 

In the repository, `hasLimit()` evaluates `-2147483648 > 0` → `false`, so `setMaxResults` is never called and the JPA query carries no SQL `LIMIT` clause: [5](#0-4) 

`getResultList()` then materialises the entire result set into heap memory before returning: [6](#0-5) 

The overflow **persists on every subsequent poll** because `(int)(2147483648L - N)` remains negative for any `N` that fits in a `long` but whose lower 32 bits keep the sign bit set.

### Impact Explanation

Every poll cycle issues an unbounded `SELECT … FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT`, loading the full topic history into JVM heap. For a high-volume topic this causes repeated GC pressure and eventual `OutOfMemoryError`, crashing the gRPC service for all subscribers. The service-level `flux.take(filter.getLimit())` guard in `TopicMessageServiceImpl` limits what is *emitted* to the client but does not prevent the unbounded DB fetch that already occurred: [7](#0-6) 

The `isComplete()` throttled-mode check (`pageSize.get() < maxPageSize`) will also never terminate the loop because the unbounded query returns more rows than `maxPageSize`, so polling continues indefinitely with unbounded queries: [8](#0-7) 

### Likelihood Explanation

The proto field is `uint64`: [9](#0-8) 

Any unauthenticated client with network access to port 5600 can send `limit = 2147483648` using `grpcurl` or any protobuf client. No credentials, no special role, no prior knowledge of the system beyond the public proto definition is required. The attack is trivially repeatable and can be parallelised across multiple connections (up to `maxConcurrentCallsPerConnection = 5` per connection by default, but multiple connections are allowed).

### Recommendation

1. **Add `@Max(Integer.MAX_VALUE)` to `TopicMessageFilter.limit`** (or a domain-appropriate maximum) to reject oversized values at the validation boundary before they reach `poll()`.
2. **Replace the unsafe narrowing cast** with a bounds-checked conversion:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
3. **Add a defensive guard in `findByFilter`**: if `filter.getLimit() <= 0` and `hasLimit()` is false, apply a hard server-side cap via `setMaxResults(serverMaxPageSize)` to prevent unbounded queries regardless of upstream logic.

### Proof of Concept

```bash
# Send limit = 2147483648 (Integer.MAX_VALUE + 1) to a topic with many messages
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110},
       "consensusStartTime": {"seconds": 0},
       "limit": 2147483648}' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected (vulnerable) behaviour:**
- `poll()` computes `(int)(2147483648L - 0L)` = `-2147483648`
- `newFilter.limit` = `-2147483648`; `hasLimit()` = `false`
- `findByFilter` issues `SELECT … FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with **no LIMIT clause**
- `getResultList()` loads the entire topic history into heap
- On a topic with millions of messages the gRPC JVM process exhausts heap and crashes, denying service to all subscribers
