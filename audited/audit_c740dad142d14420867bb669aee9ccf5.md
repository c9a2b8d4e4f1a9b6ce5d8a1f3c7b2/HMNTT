### Title
Global Gas Bucket Exhaustion via Concurrent Max-Gas Requests (Unauthenticated Griefing DoS)

### Summary
`ThrottleManagerImpl` maintains a single global `gasLimitBucket` shared across all users with no per-user or per-IP partitioning. Gas tokens are consumed atomically at request admission time in `ContractController.call()` but are only restored after EVM execution completes inside `ContractCallService.doProcessCall()`'s `finally` block. An unprivileged attacker can flood the service with concurrent max-gas requests, exhausting the bucket during the execution window and causing all other users' calls to fail with `GAS_PER_SECOND_LIMIT_EXCEEDED` for the duration of that window.

### Finding Description

**Full call path:**

1. `ContractController.call()` (line 40) calls `throttleManager.throttle(request)`, which immediately deducts `scaleGas(request.getGas())` tokens from `gasLimitBucket`: [1](#0-0) 

2. Execution proceeds through `contractExecutionService.processCall()` → `callContract()` → `doProcessCall()`. Gas is only restored in the `finally` block of `doProcessCall()`: [2](#0-1) 

3. `restoreGasToBucket()` calls `throttleManager.restore()`, which adds tokens back: [3](#0-2) 

**Default capacity math:**

| Parameter | Raw value | Scaled (÷10,000) |
|---|---|---|
| `gasPerSecond` | 7,500,000,000 | **750,000 tokens** (bucket capacity) |
| `maxGasLimit` | 15,000,000 | **1,500 tokens** per max-gas request |
| `requestsPerSecond` | 500 | 500 tokens (rate bucket) |

To exhaust the gas bucket: `750,000 / 1,500 = 500` concurrent requests at max gas — exactly equal to the global rate limit. [4](#0-3) [5](#0-4) 

**Root cause — timing gap with no per-user guard:**

Both `rateLimitBucket` and `gasLimitBucket` are global singletons with no per-IP or per-user partitioning: [6](#0-5) 

The `throttle()` call and the `restore()` call are not atomic with respect to execution. During the entire EVM execution window the tokens are absent from the bucket. An attacker who fires 500 concurrent requests at `gas=15_000_000` consumes all 750,000 scaled tokens simultaneously. Any new request arriving while those 500 are executing hits:

```
gasLimitBucket.tryConsume(1500) → false → throw GAS_PER_SECOND_LIMIT_EXCEEDED
``` [7](#0-6) 

**Why the restore mechanism does not prevent this:**

The restore is designed to return *unused* gas after execution. For a lightweight read call using 21,000 gas out of a 15,000,000 limit, `min(14,979,000, 15,000,000)` = 14,979,000 gas is restored — but only *after* execution finishes. During the execution window (which can span tens to hundreds of milliseconds for complex contracts), the bucket remains at zero. The attacker can pipeline new waves of 500 requests every second (when both buckets refill) to maintain near-continuous exhaustion. [3](#0-2) 

### Impact Explanation

All legitimate users sharing the same mirror-node instance are denied service for the duration of each exhaustion window. Because the attacker's actual gas consumption per request is negligible (e.g., 21,000 gas used vs. 15,000,000 declared), the attacker sustains the attack at near-zero cost while causing 100% rejection of other users' `eth_call` and `eth_estimateGas` requests with `GAS_PER_SECOND_LIMIT_EXCEEDED`. The attack is repeatable every second as both buckets refill greedily.

### Likelihood Explanation

No authentication, API key, or IP-based quota is required. Any HTTP client capable of issuing 500 concurrent POST requests to `/api/v1/contracts/call` with `"gas": 15000000` can trigger this. Standard HTTP load-testing tools (e.g., `wrk`, `hey`, `ab`) suffice. The attacker needs no on-chain funds or privileged access.

### Recommendation

1. **Per-source rate limiting**: Introduce per-IP (or per-API-key) sub-buckets so a single source cannot consume the entire global gas budget.
2. **Admission-time gas cap**: Throttle against a conservative per-request gas ceiling rather than the user-supplied `gas` field, or defer the full gas deduction until actual gas used is known and only pre-deduct a smaller reservation.
3. **Concurrent-request limit per source**: Reject requests from a source that already has N in-flight executions.
4. **Reduce `gasLimitRefundPercent` default**: Lowering it below 100 increases the cost of the attack by making each request permanently consume more of the budget.

### Proof of Concept

```bash
# Exhaust the gasLimitBucket with 500 concurrent max-gas requests
# Requires: curl, GNU parallel or similar

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000167","gas":15000000,"data":"0x"}' &
done
wait

# Immediately after (within the execution window), a legitimate user's request fails:
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000167","gas":21000,"data":"0x"}'
# Response: 429 / "Gas per second rate limit exceeded."
```

Repeat the 500-request wave every ~1 second to maintain continuous exhaustion. No credentials or on-chain state required.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }

    private static final long GAS_SCALE_FACTOR = 10_000L;

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }

    Bucket gasLimitBucket() {

        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
