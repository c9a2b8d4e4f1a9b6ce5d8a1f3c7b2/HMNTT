### Title
Unauthenticated Plain-HTTP Mirror Node Response Allows Network Map Injection in `buildNetworkFromMirrorNodes()`

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` issues an unauthenticated plain-HTTP GET to whatever URL is stored in `cfg.mirrorRest` and unconditionally trusts the JSON response to build the gRPC network map. No TLS is enforced, no response signature or integrity check exists, and the resulting `map[string]hiero.AccountID` is passed directly to `hiero.ClientForNetwork()`. A network-level attacker who can intercept or spoof the HTTP response can redirect every subsequent gRPC transaction to attacker-controlled endpoints and return fabricated receipts, causing the pinger to misrepresent Hashgraph history.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` (lines 36–72) calls `fetchMirrorNodeNetwork()` (lines 74–132).

Inside `fetchMirrorNodeNetwork()`:

```go
// line 46 – plain http.Client, no TLS config, no cert pinning
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}

// line 79 – raw GET, URL comes verbatim from cfg.mirrorRest (e.g. http://mirror-rest:5551)
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
...
resp, err := httpClient.Do(req)

// lines 95-98 – response decoded with zero authentication
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }

// lines 100-124 – every entry in the attacker-supplied payload is inserted into the network map
network := make(map[string]hiero.AccountID)
for _, n := range payload.Nodes {
    ...
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

`pinger/sdk_client.go` line 22 then passes this map directly:
```go
client = hiero.ClientForNetwork(netmap)
```

**Root cause:** The code assumes the HTTP channel is trusted. There is no TLS requirement, no HMAC/signature on the response, no comparison against a known-good node list, and no IP/domain allowlist for `service_endpoints`.

**Existing checks and why they are insufficient:**

- `n.NodeAccountID == ""` check (line 103): only skips blank IDs; attacker supplies valid-looking IDs like `0.0.3`.
- `hiero.AccountIDFromString` (line 107): validates format only, not that the ID belongs to a real consensus node.
- `host == "" || ep.Port == 0` check (line 118): only skips blank entries; attacker supplies real-looking host/port pairs pointing to their own server.
- HTTP status check (lines 90-93): only validates the HTTP layer, not the payload authenticity.

None of these checks verify that the returned nodes are legitimate Hedera consensus nodes.

### Impact Explanation

When `network=other` is configured (the only mode that calls this function), the pinger's entire gRPC node set is derived from this single unauthenticated HTTP response. An attacker who injects a `nodesEnvelope` containing only rogue entries causes:

1. All `CryptoTransfer` transactions to be submitted to attacker-controlled gRPC servers instead of real consensus nodes.
2. The attacker's server returns fabricated `TransactionReceipt` responses (e.g., `SUCCESS`) for transactions that were never submitted to the real Hashgraph network.
3. The pinger logs and metrics report successful transfers that never occurred on-chain, directly misrepresenting Hashgraph history.
4. The operator's private key signs real transactions that are silently dropped or replayed at the attacker's discretion.

The actual ledger state is unaffected, but the monitoring layer's view of it is completely under attacker control.

### Likelihood Explanation

**Preconditions:**
- `HIERO_MIRROR_PINGER_NETWORK=other` must be set (non-default but documented use case for private/custom networks).
- `HIERO_MIRROR_PINGER_REST` must use `http://` (the documented example in `config.go` line 37 explicitly shows `http://mirror-rest:5551`).
- Attacker needs a network-level position to intercept or spoof the HTTP response.

**Feasibility for an unprivileged attacker:**
- If the mirror REST URL resolves to an internal hostname (e.g., Kubernetes service), the attacker needs pod-level network access — elevated but not application-privileged.
- If the URL is an external HTTP endpoint (fully valid per the code), a network-path attacker (DNS poisoning, BGP hijack, same-LAN ARP spoofing) can intercept with no application privileges whatsoever.
- The attack is repeatable: `buildNetworkFromMirrorNodes()` is called once at startup; a single poisoned response permanently corrupts the client for the lifetime of the process.
- No cryptographic material is required from the attacker side.

### Recommendation

1. **Enforce HTTPS:** Reject `cfg.mirrorRest` URLs that do not use the `https` scheme at config validation time (`pinger/config.go`, `loadConfig()`).
2. **TLS certificate pinning or CA restriction:** Configure the `http.Client` with a custom `tls.Config` that pins the expected CA or leaf certificate for the mirror node.
3. **Response integrity:** Compare returned `node_account_id` values against a configurable allowlist of known-legitimate consensus node account IDs before inserting into the network map.
4. **Endpoint allowlist:** Reject `service_endpoints` whose IP/domain does not fall within a configured allowlist of trusted address ranges.
5. **Periodic re-validation:** If the network map must be refreshed at runtime, re-apply all the above checks on each refresh.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- Attacker has network-level access to intercept traffic to `mirror-rest:5551` (e.g., DNS poisoning to redirect `mirror-rest` to attacker IP, or ARP spoofing on the same segment).

**Steps:**

1. Attacker stands up a rogue HTTP server on their controlled IP, port 5551, and a rogue gRPC server on port 50211.

2. Rogue HTTP server responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "<attacker-ip>", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

3. Pinger starts, calls `buildNetworkFromMirrorNodes()`, receives the rogue payload, and builds `netmap = {"<attacker-ip>:50211": AccountID{0,0,3}}`.

4. `hiero.ClientForNetwork(netmap)` configures the SDK to send all gRPC calls to `<attacker-ip>:50211`.

5. Rogue gRPC server receives `CryptoTransfer` requests and responds with `TransactionReceipt{status: SUCCESS, ...}`.

6. Pinger logs successful transfers; monitoring dashboards show healthy Hashgraph activity. No real transactions were ever submitted to the network. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
