### Title
Integer Overflow in `poll()` Causes Unbounded Database Query via Negative pageSize

### Summary
In `PollingTopicMessageRetriever.poll()`, the remaining-limit calculation unsafely narrows a `long` to `int` via `(int)(filter.getLimit() - context.getTotal().get())`. Any unprivileged gRPC subscriber that sets `limit > Integer.MAX_VALUE` (permitted by the proto `uint64` field and the `@Min(0)`-only validation) causes this cast to produce a negative `int`. `Math.min()` then selects that negative value as `pageSize`, which is stored as the `limit` in the derived filter. Because `TopicMessageFilter.hasLimit()` returns `false` for any value `<= 0`, `TopicMessageRepositoryCustomImpl.findByFilter()` never calls `setMaxResults()`, issuing an unbounded SQL query that returns every message in the topic on every poll cycle.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` maps the proto `uint64 limit` directly to `TopicMessageFilter.limit` (a `long`) with no upper-bound check:
```java
// ConsensusController.java line 56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

`TopicMessageFilter.limit` carries only `@Min(0)` — no `@Max`:
```java
// TopicMessageFilter.java line 25-26
@Min(0)
private long limit;
```

Inside `PollingTopicMessageRetriever.poll()` (lines 68–71):
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

With `filter.getLimit() = 3_000_000_000L` and `total = 0`:
- `3_000_000_000L - 0L = 3_000_000_000L`
- `(int) 3_000_000_000L = -1_294_967_296` (two's-complement overflow)
- `Math.min(-1_294_967_296, 1000) = -1_294_967_296`

The derived filter is built with this negative value (line 75):
```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
```

In `TopicMessageRepositoryCustomImpl.findByFilter()` (lines 51–53):
```java
if (filter.hasLimit()) {          // hasLimit() → limit > 0 → FALSE for -1_294_967_296
    typedQuery.setMaxResults(...); // NEVER CALLED
}
```

`setMaxResults()` is skipped, so Hibernate issues a `SELECT … WHERE topic_id = ? AND consensus_timestamp >= ?` with **no LIMIT clause**, returning every row for the topic.

The outer `flux.take(filter.getLimit())` in `TopicMessageServiceImpl` (line 84) uses the original `long` limit (3 billion), which is not a practical guard.

The identical pattern exists in `PollingTopicListener.poll()` (lines 54–59), making both polling paths vulnerable.

### Impact Explanation

Every poll cycle (default: every 2 s for throttled, every 20 ms for unthrottled) issues a full-table-scan query against the `topic_message` table for the targeted topic. A single malicious subscriber can:
- Saturate database I/O and connection pool, degrading service for all subscribers (DoS).
- Force the JVM to materialise the entire result set in memory (`getResultList()` at line 60), causing heap pressure or OOM.
- Receive all historical messages for a topic regardless of the intended per-page cap, bypassing the rate-limiting design of the retriever.

Severity: **High** — resource exhaustion / availability impact on a shared infrastructure component.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible to any client. The proto field is `uint64`, so any standard gRPC client can trivially set `limit = 2147483648` (Integer.MAX_VALUE + 1). No authentication or privilege is required. The attack is repeatable and can be scripted. The only prerequisite is knowledge of a valid topic ID, which is public information on a mirror node.

### Recommendation

1. **Clamp the limit before the cast.** Replace the unsafe narrowing cast with a bounded conversion:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an upper-bound validator** on `TopicMessageFilter.limit`:
   ```java
   @Max(Integer.MAX_VALUE)   // or a domain-specific cap
   private long limit;
   ```
3. **Apply the same fix to `PollingTopicListener.poll()`** (line 54–55), which contains the identical pattern.
4. **Defensive guard in `findByFilter()`**: treat any non-positive limit as "no query" or throw, rather than silently removing the LIMIT clause.

### Proof of Concept

```
# Prerequisites: gRPC client (e.g. grpcurl), valid topicId on the mirror node

grpcurl -plaintext \
  -d '{
    "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 2147483648
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected (vulnerable) behaviour:**
- `(int)(2147483648L - 0L)` = `-2147483648`
- `Math.min(-2147483648, 1000)` = `-2147483648`
- `hasLimit()` returns `false` → no `LIMIT` in SQL
- Every poll cycle executes `SELECT * FROM topic_message WHERE topic_id = 1 AND consensus_timestamp >= 0 ORDER BY consensus_timestamp` with no row cap, returning the full table partition for that topic.
