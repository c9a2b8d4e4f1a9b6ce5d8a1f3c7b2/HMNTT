### Title
Unauthenticated Connection Pool Exhaustion via Public Key Query Dedicated Client Path in `getAccounts()`

### Summary
In `rest/accounts.js`, the `getAccounts()` function unconditionally checks out a dedicated pg client from the pool whenever `account.publickey` is present in the request, holding that client for the full duration of a potentially expensive join query. With no rate limiting on the REST API and a default pool size of 10, an unprivileged attacker can exhaust the connection pool with as few as 10 concurrent requests, causing all subsequent requests requiring a dedicated client to fail or queue indefinitely. The `statement_timeout` mitigation is rendered ineffective by the pgbouncer configuration that explicitly ignores it.

### Finding Description

**Exact code path:**

In `getAccounts()`, line 366 of `rest/accounts.js`:
```js
const preQueryHint = pubKeyQuery.query !== '' && constants.zeroRandomPageCostQueryHint;
const result = await pool.queryQuietly(query, params, preQueryHint);
``` [1](#0-0) 

`pubKeyQuery.query` is non-empty whenever `account.publickey` is supplied (any valid hex key passes `parsePublicKeyQueryParam`). `constants.zeroRandomPageCostQueryHint` is the string `'set local random_page_cost = 0'`. [2](#0-1) 

In `queryQuietly()` (`rest/utils.js`, lines 1519–1527), when `preQueryHint` is truthy:
```js
client = await this.connect();          // checks out dedicated client from pool
client.on('error', clientErrorCallback);
await client.query(`begin; ${preQueryHint}`);
result = await client.query(query, params);  // holds client for full query duration
await client.query('commit');
``` [3](#0-2) 

The client is only released in the `finally` block after the query completes or errors. The pool has a default `max` of **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). [4](#0-3) [5](#0-4) 

**Why `statement_timeout` does not mitigate this:**

The pool config sets `statement_timeout: config.db.pool.statementTimeout` (default 20,000 ms) as a startup parameter sent to pgbouncer. [6](#0-5) 

However, the production pgbouncer configuration explicitly lists `statement_timeout` in `ignore_startup_parameters`:
```yaml
ignore_startup_parameters: extra_float_digits,options,statement_timeout
``` [7](#0-6) 

pgbouncer strips this parameter before forwarding to PostgreSQL, so the PostgreSQL server never enforces the timeout. No global `statement_timeout` is set in the PostgreSQL server config either. Queries can run for arbitrarily long durations.

**Combined filter amplification:**

When both `account.balance=gte:0` and `account.publickey=<key>` are supplied, `getEntityBalanceQuery()` includes both conditions in the `whereCondition` AND constructs a CTE (`with latest_token_balance as (...)`) joined against the entity table and `entity_stake`. The `account.balance=gte:0` predicate matches every account, forcing a full scan with the join, maximizing query cost. [8](#0-7) [9](#0-8) 

**No rate limiting on the REST API:**

The throttling infrastructure found in the codebase applies only to the web3 Java API. There is no rate limiting middleware for the Node.js REST API's `/api/v1/accounts` endpoint.

### Impact Explanation

With a default pool of 10 connections, an attacker sending 10 concurrent `GET /api/v1/accounts?account.publickey=<valid_key>&account.balance=gte:0` requests exhausts the pool. All subsequent requests that trigger the dedicated-client path (any public-key query) will block waiting for a connection until `connectionTimeoutMillis` (20 s) elapses, at which point they fail. Legitimate users receive connection timeout errors for the duration of the attack. Because the attack is trivially repeatable (re-send 10 requests as soon as the previous batch times out), the REST API can be kept in a degraded state continuously. This constitutes a denial-of-service against the mirror node REST API.

### Likelihood Explanation

The attack requires zero authentication, zero account privileges, and only a valid public key (any 64/66/88-character hex string). The attacker needs only 10 concurrent HTTP connections. The request is a standard GET with no body. The attack is fully automatable with any HTTP load tool (curl, wrk, ab). The absence of rate limiting means there is no server-side defense to overcome.

### Recommendation

1. **Apply rate limiting** to `/api/v1/accounts` at the middleware level (e.g., express-rate-limit keyed by IP), specifically for requests containing `account.publickey`.
2. **Enforce `statement_timeout` at the PostgreSQL role level** (`ALTER ROLE mirror_api SET statement_timeout = '20s'`) so it cannot be bypassed by pgbouncer's `ignore_startup_parameters`.
3. **Remove `statement_timeout` from pgbouncer's `ignore_startup_parameters`** or apply it via a `SET` command inside the transaction rather than as a startup parameter.
4. **Increase the pool size** or add a semaphore to cap the number of concurrent dedicated-client checkouts for public-key queries.
5. Consider **not using a dedicated client** for the `SET LOCAL` hint; instead, issue `SET LOCAL` as the first statement of a regular pooled query using pgbouncer's transaction mode, which already scopes session variables to the transaction.

### Proof of Concept

```bash
# Obtain any valid ED25519 public key (64 hex chars)
PUBKEY="2b60955bcbf0cf5e9ea880b52e5b63f664b08edf6ed15e301049517438d61864"

# Exhaust the pool (default max=10) with concurrent expensive requests
for i in $(seq 1 12); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts?account.publickey=${PUBKEY}&account.balance=gte:0&limit=100" &
done
wait

# All subsequent public-key queries now fail with connection timeout errors
curl -v "http://<mirror-node-host>:5551/api/v1/accounts?account.publickey=${PUBKEY}"
# Expected: connection timeout / 500 error after connectionTimeoutMillis (20s)

# Repeat every ~20s to maintain continuous DoS
```
