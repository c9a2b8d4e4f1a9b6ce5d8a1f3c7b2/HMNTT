### Title
Thread Pool Exhaustion via Shared Resource Starvation in `EntityServiceImpl` — No Bulkhead Isolation Between Entity Lookup Paths

### Summary
`EntityServiceImpl` in the GraphQL module exposes three entity lookup methods (`getByIdAndType`, `getByAliasAndType`, `getByEvmAddressAndType`) that all execute synchronously on the shared Tomcat thread pool with no bulkhead isolation between them. An unprivileged attacker flooding `getByEvmAddressAndType` from multiple source IPs can exhaust the shared thread pool, causing all three lookup paths to become unavailable simultaneously — even though only one database query type is slow or under load.

### Finding Description

**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 19–41, all three methods.

**Root cause:**
The service is a plain synchronous Spring bean with no `@Bulkhead`, no Resilience4j configuration, and no per-method thread pool partitioning. The build file (`graphql/build.gradle.kts`, line 28) confirms `spring-boot-starter-web` — blocking Tomcat, not reactive WebFlux. Every incoming GraphQL request occupies a Tomcat thread for the full duration of the database call.

All three methods delegate to the same `EntityRepository` instance:
- `getByIdAndType` → `entityRepository.findById(...)` (primary key lookup, fast)
- `getByAliasAndType` → `entityRepository.findByAlias(...)` (secondary index)
- `getByEvmAddressAndType` → `entityRepository.findByEvmAddress(...)` (secondary index, potentially slower for non-ID-encoded addresses)

There is no isolation: a thread blocked in `findByEvmAddress` occupies the same pool slot as one that would serve `findById`.

**Exploit flow:**
1. Attacker sends concurrent `evmAddress`-based GraphQL queries from N source IPs.
2. Each request enters `getByEvmAddressAndType`, hits the `findByEvmAddress` DB path (line 40), and blocks a Tomcat thread for up to the configured `statementTimeout` of 10,000 ms (per `docs/graphql/README.md`).
3. With enough concurrent requests across IPs, all Tomcat threads (default 200) are occupied.
4. New requests for `getByIdAndType` and `getByAliasAndType` — which would complete in milliseconds — queue behind the saturated pool and time out.
5. The entire GraphQL endpoint becomes unresponsive.

**Why existing checks fail:**

The only per-request rate limit is the Traefik `inFlightReq: amount: 5` per source IP (`charts/hedera-mirror-graphql/values.yaml`, lines 138–142). This is:
- Per-IP, not global — 40 source IPs × 5 concurrent = 200 threads saturated
- Applied at the ingress layer, not the application layer — direct access bypasses it entirely
- Not differentiated by query type — all five slots per IP can target the slow path

GraphQL-level protections (`MaxQueryComplexityInstrumentation(200)`, `MaxQueryDepthInstrumentation(10)`, `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` lines 43–45) limit query structure, not concurrency or thread occupancy.

No `@Bulkhead`, `@RateLimiter`, or Resilience4j annotation exists anywhere in `graphql/src/main/java/` (confirmed by grep).

### Impact Explanation
Complete denial of service for all entity lookup operations in the GraphQL API. A targeted flood of `evmAddress` queries starves `entityId`- and `alias`-based lookups that are otherwise healthy and fast. The blast radius is the entire `EntityService` surface, not just the attacked path. Since the GraphQL API is publicly accessible without authentication (no auth layer visible in the codebase), this is reachable by any external party.

### Likelihood Explanation
Requires no credentials, no special knowledge of the system internals, and no exploit tooling beyond a standard HTTP load generator (e.g., `wrk`, `hey`, `ab`). A distributed attack using 40 IPs — trivially achievable via cloud VMs or a small botnet — saturates the default Tomcat thread pool. The attack is repeatable and can be sustained indefinitely. The 10-second statement timeout means each wave of requests holds threads for the maximum duration before releasing, making recovery slow.

### Recommendation
1. **Introduce per-path bulkhead isolation** using Resilience4j `@Bulkhead(name="evmAddress", type=THREADPOOL)` on `getByEvmAddressAndType`, with a separate named thread pool, so slow EVM address lookups cannot consume threads reserved for ID/alias lookups.
2. **Add a global concurrent-request limit** at the application level (e.g., via a `WebMvcConfigurer` or Servlet filter) independent of the Traefik ingress, so direct access is also protected.
3. **Apply per-query-type rate limiting** at the GraphQL resolver level, not only at the ingress.
4. **Consider migrating to a reactive stack** (`spring-boot-starter-webflux`) so DB calls do not hold OS threads, reducing the blast radius of slow queries.

### Proof of Concept

**Preconditions:** GraphQL endpoint reachable at `http://<host>/graphql/alpha`. Attacker controls 40+ source IPs (or Traefik middleware is not deployed).

**Step 1 — Craft a slow-path query:**
```graphql
{ account(input: { evmAddress: "0xaabbccddaabbccddaabbccddaabbccddaabbccdd" }) { balance } }
```
This routes to `getByEvmAddressAndType` → `findByEvmAddress` (line 40 of `EntityServiceImpl.java`), the non-ID-encoded path.

**Step 2 — Flood from multiple IPs:**
```bash
# Run from 40 different source IPs, 5 concurrent each
for ip in $(seq 1 40); do
  wrk -t5 -c5 -d60s -s graphql_evmaddr.lua http://<host>/graphql/alpha &
done
```

**Step 3 — Observe collateral degradation:**
```bash
# From a separate client — fast path, should return in <10ms
curl -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'
# Result: connection timeout or HTTP 503 — thread pool exhausted
```

**Expected result:** The `entityId`-based query (which uses `findById`, a primary key lookup) fails with a timeout despite the database being healthy for that query type, because all Tomcat threads are occupied by the `evmAddress` flood. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```java
public class EntityServiceImpl implements EntityService {

    private final EntityRepository entityRepository;

    @Override
    public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
        return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
        return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
```
