### Title
Unbounded Gap-Recovery Database Scan via Safety-Check Path in `missingMessages()`

### Summary
Any unprivileged user who subscribes to a topic without an `endTime` will unconditionally trigger the safety-check path in `missingMessages()` after one second. The resulting `gapFilter` inherits the original filter's `null` `endTime` and zero `limit`, causing `topicMessageRetriever.retrieve(gapFilter, false)` to issue an unbounded historical scan of the entire topic from the user-supplied `startTime` with no throttling. An attacker can open many such subscriptions simultaneously to exhaust database resources.

### Finding Description

**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`

**Safety-check setup (lines 67–70):**
```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [1](#0-0) 

**`isComplete()` always returns `false` when `endTime == null` (lines 203–205):**
```java
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;
    }
``` [2](#0-1) 

This means for any subscription without an `endTime`, the guard `!topicContext.isComplete()` is always `true`, so the safety check fires unconditionally after 1 second.

**`missingMessages()` gap-filter construction (lines 142–149):**
```java
if (current == null) {
    long startTime = last != null
            ? last.getConsensusTimestamp() + 1
            : topicContext.getFilter().getStartTime();
    var gapFilter =
            topicContext.getFilter().toBuilder().startTime(startTime).build();
    ...
    return topicMessageRetriever.retrieve(gapFilter, false);
}
``` [3](#0-2) 

`toBuilder().startTime(startTime).build()` copies all other fields from the original filter — including `endTime` (which is `null`) and `limit` (which is `0`). No upper bound is added. The call uses `throttled=false`, bypassing any retriever-level throttling.

**`TopicMessageFilter` allows `endTime = null` and `limit = 0` (lines 23–26, 39–41):** [4](#0-3) [5](#0-4) 

**`startTime` validation only requires `startTime <= now()` (lines 48–51):** [6](#0-5) 

An attacker can set `startTime` to any past timestamp (e.g., network genesis), making the scan cover the entire topic history.

**Root cause:** The `current == null` branch in `missingMessages()` builds `gapFilter` by inheriting the caller-controlled filter without enforcing any `endTime` or `limit` upper bound, and the `isComplete()` guard never blocks this path when `endTime` is absent.

### Impact Explanation
Each subscription without an `endTime` triggers one unbounded `SELECT` against the topic message table after 1 second, scanning from an arbitrarily old `startTime` to the present with no row limit and no throttling (`throttled=false`). An attacker opening N concurrent subscriptions causes N simultaneous full-history scans, exhausting database I/O, connection pool, and memory. This is a denial-of-service against the mirror node's database with no authentication required.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication. The exploit requires only a standard gRPC client call with `startTime` set to an old epoch and no `endTime`. It is trivially repeatable and scriptable. The 1-second delay before triggering is not a meaningful barrier.

### Recommendation
In the `current == null` branch of `missingMessages()`, enforce an upper bound on the gap filter before issuing the retrieval:

1. **Always set `endTime`** on `gapFilter` to `DomainUtils.now()` (or the original filter's `endTime` if non-null) so the scan is bounded to the present.
2. **Enforce a maximum `limit`** on the gap filter (e.g., cap at a configurable `maxGapFillMessages`).
3. Consider making the safety check conditional on `endTime != null` in the original filter, or add a separate bounded "catch-up" query with explicit limits.

Example fix for the gap filter construction:
```java
long upperBound = topicContext.getFilter().getEndTime() != null
        ? topicContext.getFilter().getEndTime()
        : DomainUtils.now();
var gapFilter = topicContext.getFilter().toBuilder()
        .startTime(startTime)
        .endTime(upperBound)
        .limit(maxGapFillMessages)
        .build();
```

### Proof of Concept
1. Identify any valid topic ID on the mirror node.
2. Open a gRPC stream to `subscribeTopic` with:
   - `topicId` = valid topic
   - `startTime` = 0 (or network genesis nanoseconds)
   - `endTime` = not set
   - `limit` = not set
3. Wait 1 second without sending any messages.
4. Observe in server logs: `"Safety check triggering gap recovery query with filter ..."` with no `endTime` in the filter.
5. Observe the database executing a full-history scan of the topic message table.
6. Repeat step 2 in a loop with many concurrent connections to amplify the load.
