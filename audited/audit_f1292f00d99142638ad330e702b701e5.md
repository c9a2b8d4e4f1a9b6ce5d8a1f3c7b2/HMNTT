### Title
Cache TTL Asymmetry Enables Sustained Low-Rate Amplified Batch DB Query Attack

### Summary
The `contractStateCache` (`expireAfterWrite=2s`) expires 150× faster than the per-contract slot-key accumulator (`slotsPerContract`, `expireAfterAccess=5m`). Because every `contractStateCache` miss unconditionally triggers `findStorageBatch()` using the full set of accumulated slot keys, an attacker who has pre-seeded the slot-key accumulator can force a large-IN-clause DB query (`WHERE slot IN (up to 1500 slots)`) with a single request every 2 seconds — well below any rate-limit threshold.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` first. On any miss it calls `findStorageBatch()` (lines 85–122). [1](#0-0) 

Inside `findStorageBatch()`, the per-contract slot-key cache (`contractSlotsCache`) is read in full and every key it contains is passed to the DB: [2](#0-1) 

**Root cause — TTL asymmetry:**

| Cache | Config | Effective lifetime |
|---|---|---|
| `contractStateCache` (values) | `expireAfterWrite=2s` | 2 seconds |
| `slotsPerContract` (slot-key accumulator) | `expireAfterAccess=5m` | up to 5 minutes, reset on every access | [3](#0-2) 

Slot keys accumulate in the per-contract cache for up to 5 minutes (max 1 500 entries per contract, `maximumSize=1500`). The corresponding values in `contractStateCache` expire every 2 seconds. There is no mechanism that clears accumulated slot keys when their values expire. Therefore, after every 2-second window, the full accumulated key set is re-queried against the DB.

**Failed assumption:** The design assumes that `contractStateCache` hits will be the common case and that `findStorageBatch()` will only be called infrequently. In reality, the 2-second TTL guarantees a cache miss at least every 2 seconds for any active slot, and each miss re-issues the entire accumulated batch.

**Exploit flow:**

1. **Seed phase (one-time):** Attacker sends 1 500 requests, each for a distinct slot of a target contract. Each request adds a new key to the per-contract `slotsPerContract` cache via `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` (line 90). The `slotsPerContract` TTL resets on every access, so keys remain alive as long as the attacker keeps touching them. [4](#0-3) 

2. **Sustained attack:** Every ≥2 seconds the attacker sends a single request for any slot of that contract. `contractStateCache` is expired, so `findStorageBatch()` is called. It reads all 1 500 accumulated keys and issues: `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (1500 entries)`. [5](#0-4) 

3. **Amplification:** 1 HTTP request → 1 large-IN-clause DB query with 1 500 parameters, repeated every 2 seconds indefinitely.

**Existing checks reviewed and shown insufficient:**

- `requestsPerSecond=500` throttle: The attack requires only 0.5 req/s (1 request per 2 s), far below the limit.
- `maximumSize=1500` on `slotsPerContract`: Caps the batch at 1 500 slots but does not prevent the attack; it only bounds the maximum query size.
- `maximumSize=3000` on `contractSlots` (contractId → per-contract cache): Allows up to 3 000 contracts to be targeted simultaneously, multiplying the effect. [6](#0-5) 

### Impact Explanation

Each triggered `findStorageBatch()` call issues a parameterized SQL query with up to 1 500 bind parameters against the `contract_state` table. On a production node with a large `contract_state` table, repeated 1 500-slot IN-clause queries every 2 seconds per targeted contract constitute a sustained, amplified DB load increase achievable with a negligible HTTP request rate. An attacker targeting multiple contracts (up to 3 000 simultaneously) can multiply this effect proportionally. The DB `statementTimeout` is 3 000 ms, meaning each query can hold a DB connection for up to 3 seconds — longer than the 2-second re-trigger interval, enabling connection pool exhaustion as a secondary effect. [7](#0-6) 

### Likelihood Explanation

The attack requires no authentication, no special tooling, and no precise timing — waiting any interval ≥2 seconds between requests is sufficient. The seed phase (1 500 requests) can be spread over minutes to avoid detection. The sustained phase requires only 1 request every 2 seconds per targeted contract. Any public-facing deployment of the web3 API is exposed. The attack is fully repeatable and self-sustaining because each access resets the `expireAfterAccess` TTL on the slot-key accumulator.

### Recommendation

1. **Align TTLs:** Set `slotsPerContract` TTL ≤ `contractState` TTL (e.g., both at 2 s), so slot keys are evicted when their values expire and the batch cannot grow unboundedly between value refreshes.
2. **Cap batch size per request:** In `findStorageBatch()`, limit the number of slot keys passed to `findStorageBatch()` to a small constant (e.g., 50–100) regardless of how many keys are in the accumulator.
3. **Rate-limit by DB query cost:** Track the number of slots included in batch queries per source IP or per time window and throttle accordingly, independent of the HTTP request rate.
4. **Separate accumulator lifetime from access pattern:** Use `expireAfterWrite` (not `expireAfterAccess`) on `slotsPerContract` so that attacker-driven accesses cannot indefinitely extend the accumulator lifetime.

### Proof of Concept

```
# Step 1: Seed 1500 distinct slot keys for contract 0x<TARGET>
for i in $(seq 1 1500); do
  SLOT=$(printf '%064x' $i)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x<TARGET>\",\"data\":\"0x<SLOAD_SLOT_${SLOT}>\"}" &
done
wait

# Step 2: Every 2 seconds, send a single request — triggers findStorageBatch()
# with all 1500 accumulated slot keys on each invocation
while true; do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x<TARGET>","data":"0x<ANY_SLOT_READ>"}'
  sleep 2
done

# Observable effect: DB slow-query log shows repeated
# "SELECT slot, value FROM contract_state WHERE contract_id=? AND slot IN (?,?,?,...)"
# with 1500 parameters firing every ~2 seconds from a single client IP.
```

        if (!cacheProperties.isEnableBatchContractSlotCaching()) {

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
