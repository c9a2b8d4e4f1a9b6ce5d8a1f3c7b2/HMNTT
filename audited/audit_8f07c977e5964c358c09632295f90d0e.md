### Title
BlockAssembler Sublist Explosion: Unbounded `pending` List Growth via Single-Item `BlockItemSet` Messages

### Summary
`BlockNode.BlockAssembler` accumulates incoming `BlockItemSet` messages as sublists in `pending` (`List<List<BlockItem>>`). The only guard — `pendingCount > maxBlockItems` — counts total `BlockItem` objects but never limits the number of sublists. A block node can send up to 800,000 single-item `BlockItemSet` messages, filling `pending` with 800,000 single-element `ArrayList` objects, then forcing 800,000 individual `addAll()` calls on `EndOfBlock`, causing measurable memory fragmentation and CPU overhead well above 30% compared to a legitimate block sending the same items in a handful of large messages.

### Finding Description

**Exact code path:**

`BlockNode.java` — inner class `BlockAssembler`:

- `pending` field: [1](#0-0) 
- `append()` — the only guard checks `pendingCount` (total items), never the number of sublists: [2](#0-1) 
- `onEndOfBlock()` — iterates every sublist calling `addAll` once per sublist: [3](#0-2) 

`StreamProperties.java` — default limit is 800,000 items: [4](#0-3) 

**Root cause:** `append()` calls `pending.add(blockItems)` unconditionally and only checks `pendingCount += blockItems.size()` against `maxBlockItems`. There is no check on `pending.size()` (the number of sublists). Sending N messages each containing exactly 1 `BlockItem` produces N sublists while `pendingCount` reaches N — right at the limit without ever exceeding it.

**Exploit flow:**
1. Attacker controls a block node the mirror subscribes to (see Likelihood).
2. For each block, attacker sends exactly `maxBlockItems` (800,000) `SubscribeStreamResponse.BLOCK_ITEMS` messages, each carrying a `BlockItemSet` with exactly 1 `BlockItem`.
3. Each call to `onBlockItemSet` → `append` adds one single-element `ArrayList` to `pending`; `pendingCount` increments by 1. The guard `pendingCount > 800_000` is never triggered.
4. Attacker sends `END_OF_BLOCK`. `onEndOfBlock` enters the `else` branch (`pending.size() == 800,000 > 1`), allocates a new `ArrayList`, and calls `block.addAll(items)` 800,000 times, each adding 1 element.

**Why checks fail:**
- `maxBlockItems` (line 293) guards total item count, not sublist count. [5](#0-4) 
- `maxStreamResponseSize` (36 MB) limits individual gRPC message size, not message count. [6](#0-5) 
- There is no cap on `pending.size()` anywhere in `append` or `onBlockItemSet`. [7](#0-6) 

### Impact Explanation

**Memory fragmentation:** Each Java `ArrayList` carrying 1 element costs ~52 bytes of object overhead (header + `elementData` array + `size` field). 800,000 such objects = ~41 MB of pure overhead on top of the actual `BlockItem` data, plus equivalent GC pressure. A legitimate sender batching the same 800,000 items into a few large `BlockItemSet` messages produces only a handful of sublists.

**CPU overhead:** The `onEndOfBlock` assembly loop executes 800,000 `addAll` calls (each adding 1 element) instead of, e.g., 10 calls adding 80,000 elements each. The per-call overhead of `addAll` (bounds check, array copy, size update) multiplied by 800,000 is substantially higher than the same total work done in bulk. Additionally, the gRPC layer must deserialize, frame, and dispatch 800,000 separate messages instead of a small number of large ones — this alone can easily account for >30% additional CPU time on the importer thread.

**Severity:** Medium. The attack degrades the mirror node's block ingestion throughput and increases heap pressure, potentially causing GC pauses or OOM under sustained attack, without crashing the process outright.

### Likelihood Explanation

**Precondition — acting as a block node:** The mirror node connects to block nodes discovered via two paths: (a) static configuration (`blockProperties.getNodes()`), and (b) auto-discovery from `registeredNodeRepository` when `autoDiscoveryEnabled = true` (the default). [8](#0-7)  Auto-discovery reads nodes registered on-chain as `BLOCK_NODE` type with tier-1 APIs. [9](#0-8)  Registering a node on the Hedera network requires staking but is not restricted to a privileged administrator of the mirror node itself — any network participant with sufficient stake can register a block node endpoint. Once registered and discovered, the mirror node will subscribe to it and process its responses without further authentication of the stream content.

**Repeatability:** The attack can be repeated on every block indefinitely. The `errors` counter and `readmitDelay` only trigger if the connection throws an exception; sending well-formed but maximally fragmented messages never increments `errors`. [10](#0-9) 

### Recommendation

Add a cap on the number of sublists in `pending` inside `append()`, independent of the item count:

```java
// In StreamProperties.java — add a new property:
@Min(1)
private int maxBlockItemSets = 1000; // tune to expected legitimate max

// In BlockAssembler.append():
pending.add(blockItems);
pendingCount += blockItems.size();
if (pending.size() > streamProperties.getMaxBlockItemSets()) {
    throw new BlockStreamException(String.format(
        "Too many BlockItemSet messages in a pending block: received %d, limit %d",
        pending.size(), streamProperties.getMaxBlockItemSets()));
}
if (pendingCount > streamProperties.getMaxBlockItems()) { ... }
```

This bounds both the number of sublists and the total item count, eliminating the fragmentation vector while preserving the existing item-count guard.

### Proof of Concept

**Preconditions:**
- Attacker operates a gRPC server implementing `BlockStreamSubscribeService` and has it registered/configured as a block node for the target mirror node.

**Steps:**
1. When the mirror node calls `subscribeBlockStream(startBlockNumber=N)`, respond with:
   - 1 `BLOCK_ITEMS` message: `BlockItemSet` containing 1 `BlockItem` with a valid `BLOCK_HEADER` for block N.
   - 799,999 additional `BLOCK_ITEMS` messages: each a `BlockItemSet` with 1 `BlockItem` of any non-header type (e.g., `EVENT_HEADER`).
   - 1 `END_OF_BLOCK` message: `BlockEnd(blockNumber=N)`.
2. Observe on the mirror node:
   - `pending.size()` reaches 800,000 before `onEndOfBlock` is called.
   - `onEndOfBlock` executes the `else` branch and calls `addAll` 800,000 times.
   - Heap profiler shows ~41 MB of `ArrayList` overhead; CPU profiler shows the assembly loop consuming disproportionate time vs. a baseline run sending the same 800,000 items in 10 large `BlockItemSet` messages.
3. Repeat for every subsequent block to sustain the elevated resource consumption.
