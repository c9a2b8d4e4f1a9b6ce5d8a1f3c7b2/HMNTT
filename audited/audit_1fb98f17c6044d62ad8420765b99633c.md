### Title
3x Database Read Amplification via UNION ALL Subquery Limit Multiplication in `getQuery()`

### Summary
In `rest/service/tokenAllowanceService.js`, the `getQuery()` function constructs up to three independent subqueries each carrying the same user-controlled `LIMIT $2` clause. When all three partitions (lower, inner, upper) are non-empty, the database must scan up to `3 × max_limit` rows from `token_allowance` before the outer `LIMIT` is applied. Any unprivileged user can trigger this with a single crafted request, multiplying per-request DB read work by up to 3x compared to a simple query.

### Finding Description

**Exact code path:**

`rest/service/tokenAllowanceService.js`, `getQuery()`, lines 54–78:

```js
const params = [ownerAccountId, limit];          // limit = $2
const limitClause = super.getLimitQuery(2);       // → "limit $2"

const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

Each call to `getSubQuery()` receives the **same** `limitClause` (`limit $2`), and the outer UNION ALL wrapper also appends the **same** `limitClause`. The resulting SQL when all three partitions are populated is:

```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4
 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6
 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8
 ORDER BY spender ASC, token_id ASC LIMIT $2)
ORDER BY spender ASC, token_id ASC
LIMIT $2
```

With `$2 = 100` (the enforced maximum), the database independently executes three full scans each bounded at 100 rows, materializing up to 300 rows before the outer `LIMIT 100` discards 200 of them.

**Root cause / failed assumption:** The design assumes each subquery's `LIMIT $2` is a cheap early-termination guard. The failed assumption is that the three subqueries are not independent scans — they are, and each pays the full `LIMIT $2` cost before the merge step.

**Triggering the 3-subquery branch:** `extractTokenMultiUnionQuery()` in `rest/controllers/tokenAllowanceController.js` lines 22–59 populates `lower`, `inner`, and `upper` when the user supplies a **closed range on `spender.id` AND a closed range on `token.id`**. The test case "spender closed range and token closed range" in `rest/__tests__/service/tokenAllowanceService.test.js` lines 197–233 confirms this is a supported, documented code path.

**Why existing checks are insufficient:** `formatComparator` in `rest/utils.js` lines 544–553 correctly caps `limit` at `responseLimit.max` (default 100). This check prevents a single subquery from reading more than 100 rows, but it does **nothing** to prevent the 3× amplification: the cap is applied once to `$2`, and all three subqueries share that same capped value.

### Impact Explanation

A single HTTP request causes the database to read up to 300 rows instead of the expected 100, a 3× (200%) increase in per-request DB I/O. For a mirror-node deployment where the REST layer is DB-bound, this directly translates to proportionally higher CPU (query planning × 3), memory (result buffer × 3), and I/O (index scans × 3) per request. The 30% resource-consumption threshold is exceeded by a wide margin. Because the `token_allowance` table can be large in production (one row per owner/spender/token triple across all accounts), the scans are not trivially cheap.

### Likelihood Explanation

The exploit requires zero privileges — it is a standard public REST API call. The attacker needs only to know the API accepts `spender.id` and `token.id` range filters (documented in the OpenAPI spec). The crafted request is a single HTTP GET with four query parameters. It is trivially repeatable in a loop without any authentication, rate-limit bypass, or brute-force enumeration. Any automated client or script can sustain this continuously.

### Recommendation

Apply the limit to the **outer** query only, and use a fixed internal cap (e.g., `LIMIT $2` only on the outer wrapper) or replace the per-subquery limit with a larger internal sentinel (e.g., `LIMIT $2 * 3` or no per-subquery limit when a UNION ALL merge is used). Alternatively, track total rows fetched across subqueries and short-circuit once the outer limit is reached. A rate-limit or cost-budget per request at the middleware layer would also mitigate amplification.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1/allowances/tokens
    ?spender.id=gt:0.0.1
    &spender.id=lt:0.0.9999999
    &token.id=gte:0.0.1
    &token.id=lte:0.0.9999999
    &limit=100
```

This triggers all three partitions (lower: `spender=1, token_id>=1`; inner: `spender>1 AND spender<9999999`; upper: `spender=9999999, token_id<=9999999`) and sets `$2=100`. The database executes three independent index scans each reading up to 100 rows (300 total) before the outer `LIMIT 100` is applied. Repeat this request in a tight loop from a single unauthenticated client to sustain elevated DB resource consumption.
