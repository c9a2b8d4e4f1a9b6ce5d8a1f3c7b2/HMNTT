### Title
Unbounded `node.id` Array Input Enables Database DoS via Unconstrained `any(:nodeIds)` Clause

### Summary
The `GET /api/v1/network/nodes` endpoint accepts an unlimited number of `node.id` query parameters, which are collected into an unbounded `Long[]` array and passed directly to a PostgreSQL `any(:nodeIds)` clause inside a complex multi-CTE query. No size constraint exists on the `nodeIds` list in `NetworkNodeRequest`, unlike the analogous `RegisteredNodesRequest` which explicitly caps its list at 2. An unauthenticated attacker can repeatedly send requests with hundreds of `node.id` values to exhaust database CPU and cause denial-of-service.

### Finding Description

**Exact code path:**

`NetworkNodeRequest.nodeIds` is declared as a plain `List<NumberRangeParameter>` with no `@Size` annotation: [1](#0-0) 

By contrast, the analogous field in `RegisteredNodesRequest` is explicitly bounded: [2](#0-1) 

`RequestParameterArgumentResolver.validateAndAddParameter()` passes all multi-value parameters through without any count check — it only rejects multiple values for non-collection fields: [3](#0-2) 

`NetworkServiceImpl.getNetworkNodes()` iterates every element and builds a `HashSet<Long>` with no size guard, then converts it to a `Long[]` array: [4](#0-3) 

That array is passed directly to the repository, which embeds it in a `any(:nodeIds)` predicate inside a query containing three CTEs, three joins, and a correlated subquery per row: [5](#0-4) 

**Root cause:** The `@Size` constraint present on `RegisteredNodesRequest.registeredNodeIds` was never applied to `NetworkNodeRequest.nodeIds`. The `binder.validate()` call in the resolver enforces Jakarta validation annotations, but since none exist on `nodeIds`, nothing is enforced. [6](#0-5) 

The `limit` parameter only caps the number of rows returned, not the size of the input filter array: [7](#0-6) 

### Impact Explanation
Each request with a large `node.id` array forces PostgreSQL to evaluate `abe.node_id = any(:nodeIds)` for every row in `address_book_entry`, while simultaneously executing three CTE subqueries and a correlated `jsonb_agg` subquery per result row. Tomcat's default `max-parameter-count` is 10,000, allowing an attacker to submit thousands of distinct node IDs per request. Repeated concurrent requests can saturate database CPU and memory, making the mirror node unavailable to all users. No authentication is required.

### Likelihood Explanation
The endpoint is public and unauthenticated. The exploit requires only a standard HTTP client capable of sending repeated GET requests with many query parameters — no special knowledge, credentials, or protocol access is needed. The attack is trivially scriptable and repeatable. The absence of rate limiting on the `/api/v1/network/nodes` endpoint (unlike the fee-estimation endpoint which has `HIGH_VOLUME_THROTTLE`) removes the primary mitigation that would otherwise reduce feasibility. [8](#0-7) 

### Recommendation
Add `@Size(max = 2)` (or a similarly small bound matching the API contract) to `NetworkNodeRequest.nodeIds`, mirroring the existing constraint on `RegisteredNodesRequest.registeredNodeIds`:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 2)
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, consider adding rate limiting to the `/api/v1/network/nodes` endpoint consistent with other high-volume endpoints.

### Proof of Concept
```bash
# Build a query string with 500 distinct node.id equality parameters
PARAMS=$(python3 -c "print('&'.join(f'node.id={i}' for i in range(500)))")

# Send repeated concurrent requests to exhaust DB resources
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/network/nodes?$PARAMS" &
done
wait
```

Each request causes PostgreSQL to evaluate `abe.node_id = any(ARRAY[0,1,2,...,499])` inside the multi-CTE query. With 50 concurrent requests, database CPU spikes and legitimate requests begin timing out.
