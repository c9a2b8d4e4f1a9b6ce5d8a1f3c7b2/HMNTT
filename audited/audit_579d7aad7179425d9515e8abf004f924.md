### Title
Unauthenticated Full Entity Table Scan via `getBalances()` Enables DB Connection Pool Exhaustion

### Summary
The `getBalances()` handler in `rest/balances.js` takes a code path when no `timestamp` filter is provided that materializes a CTE scanning the entire `entity` table, then executes a correlated `token_account` subquery for each returned row. Because no authentication is required, any external user can trigger this expensive query path. With the default DB connection pool capped at 10 connections and a 20-second statement timeout, a small number of concurrent unauthenticated requests is sufficient to saturate the pool and deny service to all other users.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 133–149, when no `timestamp` query parameter is present, `getBalances()` takes the `else` branch:

```javascript
// rest/balances.js lines 133-149
const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
const whereClause = conditions && `where ${conditions}`;
const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances
  from entity_balance ab
  ${whereClause}
  order by ab.account_id ${order}
  ${limitQuery}`;
```

When no filters are supplied, `conditions` is an empty string, so `whereClause` evaluates to `''` (falsy short-circuit at `conditions && ...`). The CTE therefore reads **every row** from the `entity` table with no predicate beyond `type in ('ACCOUNT', 'CONTRACT')`.

The correlated subquery injected per row is built by `getTokenAccountBalanceSubQuery()` at lines 308–318:

```javascript
// rest/balances.js lines 308-318
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
```

This subquery executes once per row emitted by the outer `SELECT` (up to `LIMIT` rows, default 25, max 100). However, the CTE itself is a PostgreSQL optimization fence — it is materialized before the outer `LIMIT` is applied, meaning the full `entity` table scan occurs regardless of the requested limit.

**Root cause and failed assumption:**

The code assumes that the `LIMIT` clause on the outer query is sufficient to bound query cost. It is not: the CTE is evaluated in full first, and the correlated subquery still runs for each of the up to 100 returned rows. On a production network with millions of entity rows, each unfiltered request performs a full sequential scan of `entity` plus up to 100 index lookups into `token_account`.

**Why existing checks are insufficient:**

- `statementTimeout` is 20,000 ms (20 seconds) per `rest/dbpool.js` line 15 and the documented default. This bounds individual query duration but does not prevent concurrent requests from holding all pool slots simultaneously.
- `maxConnections` defaults to **10** (documented in `docs/configuration.md` line 556, configured in `rest/dbpool.js` line 14). Ten concurrent slow queries saturate the pool entirely.
- The Traefik `inFlightReq` / `rateLimit` middleware is **disabled by default**: `charts/hedera-mirror-rest/values.yaml` line 89 sets `global.middleware: false`, so no per-IP rate limiting or in-flight request cap is applied in a default deployment.
- No authentication or API key is required to call `/api/v1/balances`.

### Impact Explanation

With the pool at 10 connections and each unfiltered request potentially holding a connection for up to 20 seconds, an attacker maintaining just 10 concurrent requests continuously saturates the pool. All other API endpoints sharing the same pool (`/api/v1/accounts`, `/api/v1/transactions`, etc.) begin returning connection-timeout errors or queuing indefinitely. This constitutes a full availability denial for all users of the mirror node REST API with no economic cost to the attacker and no privileged access required. Severity is Medium (griefing / availability degradation, no fund loss or data corruption).

### Likelihood Explanation

The attack requires only an HTTP client capable of sending concurrent requests — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. No credentials, tokens, or special network position are needed. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml` lines 391–419). The attack is repeatable and stateless: the attacker simply keeps 10+ connections open in a loop. Distributed amplification (multiple source IPs) defeats even the optional per-IP Traefik middleware if it were enabled.

### Recommendation

1. **Require at least one selective filter** before executing the no-timestamp path. If `accountQuery`, `pubKeyQuery`, and `balanceQuery` are all empty, return a 400 error requiring the caller to supply a narrowing parameter (e.g., `account.id` or `account.balance`).
2. **Inline the CTE or rewrite as a direct query** so PostgreSQL can push the `LIMIT` into the scan and avoid materializing the full `entity` table.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) with `inFlightReq` and `rateLimit` configured for the REST service, mirroring the Rosetta configuration in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166.
4. **Reduce `statementTimeout`** for the no-timestamp balance path, or add a query-level timeout hint.
5. **Increase `maxConnections`** or deploy a connection pooler (e.g., PgBouncer) to decouple HTTP concurrency from DB connection count.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent unfiltered requests
# No credentials required; runs from any machine with network access to the mirror node

for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/balances" \
       -o /dev/null &
done
wait

# Legitimate request now times out or receives a connection pool error:
curl -v "https://<mirror-node-host>/api/v1/transactions?limit=1"
# Expected: delayed response or 503/connection refused
```

Each background `curl` triggers the `entity_balance` CTE full-table scan path at `rest/balances.js` lines 139–149, holding a DB connection for the duration of the scan (potentially several seconds on a large dataset).
