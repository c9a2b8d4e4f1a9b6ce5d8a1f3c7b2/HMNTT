### Title
Unauthenticated Resource Exhaustion via `getBalances()` with Broad Balance Filter and Token Aggregation

### Summary
The `getBalances()` handler in `rest/balances.js` accepts `account.balance=gte:0`, `limit=100`, and a `timestamp` filter from any unauthenticated caller. Each such request triggers a multi-stage PostgreSQL query including a correlated subquery that fetches up to 50 token balances per account (up to 5,000 token-balance rows per request). The Node.js REST server has no application-level rate limiting, so concurrent flooding of this endpoint causes sustained DB connection exhaustion, high Node.js heap pressure from large result-set serialization, and degraded availability of the mirror node service.

### Finding Description

**Exact code path:**

`rest/balances.js`, `getBalances()`, lines 83–156.

When a `timestamp` query parameter is present, execution follows:
1. `getTsQuery()` → `getAccountBalanceTimestampRange()` — issues a preliminary DB query (lines 167–197).
2. `getBalancesQuery()` (lines 199–218) — builds a `SELECT DISTINCT ON (account_id)` query against `account_balance` with a correlated subquery via `getTokenBalanceSubQuery()` (lines 294–306):

```js
select json_agg(json_build_object('token_id', token_id, 'balance', balance))
from (
  select distinct on (token_id) token_id, balance
  from token_balance tb
  where tb.account_id = ab.account_id
    and ${consensusTsQuery}
  order by token_id ${order}, consensus_timestamp desc
  limit ${tokenBalanceLimit.multipleAccounts}   // = 50
) as account_token_balance
```

This correlated subquery executes once per row in the outer result set (up to 100 rows when `limit=100`), yielding up to 5,000 `token_balance` lookups per request.

**Root cause — failed assumption:**

The code assumes infrastructure-level controls (e.g., Traefik middleware in Helm charts) will throttle abusive callers. The Node.js REST server itself has **no application-level rate limiting** on the `/api/v1/balances` route. Confirmed by inspecting `rest/server.js` (lines 100–129): no `express-rate-limit` or equivalent middleware is registered for this route. The `authHandler` middleware (lines 85–86 of `server.js`) only sets a custom limit for authenticated users; unauthenticated requests pass through without restriction.

The `account.balance=gte:0` filter is accepted by `parseBalanceQueryParam()` (`rest/utils.js`, line 628–635) and produces a valid SQL condition `ab.balance >= 0`, which matches every account. The limit is capped at `responseLimit.max = 100` (confirmed in `rest/__tests__/config.test.js` line 324), so `limit=100` is the maximum permitted and is fully accepted.

The project's own documentation acknowledges this: `docs/rest/README.md` line 81–82 states: *"The `account.balance` query parameter on `/api/v1/accounts` and `/api/v1/balances` has performance issues since depending upon the parameter value it can scan a large number of rows."*

**Exploit flow:**

1. Attacker sends `GET /api/v1/balances?account.balance=gte:0&limit=100&timestamp=<recent_valid_ts>` with no credentials.
2. `getBalances()` issues 2 DB queries: a timestamp-range lookup, then a `DISTINCT ON` query with 100 correlated token-balance subqueries.
3. Attacker sends N such requests concurrently (e.g., 50–200 simultaneous connections).
4. The DB connection pool is saturated; pending requests queue in Node.js heap holding large in-flight result buffers.
5. `formatBalancesResult()` (lines 16–56) maps over 100 rows, calling `utils.parseTokenBalances(row.token_balances)` for each, serializing up to 5,000 token objects into JSON per response.
6. Node.js event loop stalls under heap pressure and JSON serialization load; the process becomes unresponsive to legitimate requests.

### Impact Explanation

The mirror node REST service becomes unavailable or severely degraded for all users. Because the mirror node is the primary read interface for the Hedera network (balance queries, transaction history, token data), sustained unavailability constitutes a meaningful denial of service against network data consumers. The attack does not require brute force — a single attacker with modest bandwidth can sustain the load. The severity aligns with the stated classification: degradation of ≥30% of mirror node processing capacity without shutting down the underlying consensus network.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and no special knowledge beyond the public API documentation. The endpoint is publicly accessible. The `account.balance=gte:0` pattern is a documented, valid query parameter. An attacker needs only a script sending concurrent HTTP GET requests. The attack is repeatable and sustainable indefinitely. No CAPTCHA, API key, or IP-based application-level throttle exists in the Node.js server code to prevent it.

### Recommendation

1. **Add application-level rate limiting** to the `/api/v1/balances` route in `rest/server.js` using `express-rate-limit` or equivalent, keyed by IP address, independent of infrastructure-layer controls.
2. **Restrict or require authentication for broad `account.balance` filters**: reject or throttle requests where `account.balance` uses `gte:0` or `lte:<large_number>` without an accompanying narrow `account.id` filter.
3. **Add a DB statement timeout** for the REST Node.js pool (analogous to `web3`'s `statementTimeout: 3000`) to bound the maximum duration of any single query.
4. **Consider deprecating or gating the timestamp path** for `/api/v1/balances` as the documentation already recommends using `/api/v1/accounts/{id}/tokens` instead.

### Proof of Concept

```bash
# Step 1: Find a valid recent timestamp (public endpoint, no auth)
TS=$(curl -s 'https://<mirror-node>/api/v1/transactions?limit=1' \
  | jq -r '.transactions[0].consensus_timestamp')

# Step 2: Flood with maximum-cost requests (no credentials required)
for i in $(seq 1 100); do
  curl -s "https://<mirror-node>/api/v1/balances?account.balance=gte:0&limit=100&timestamp=${TS}" \
    -o /dev/null &
done
wait

# Observe: mirror node REST API response times degrade significantly or
# returns 503/timeout errors for legitimate requests during the flood.
# Node.js process heap and CPU spike; DB connection pool exhausted.
```

**Preconditions:** None. No account, API key, or special network access required.
**Trigger:** Concurrent unauthenticated GET requests to `/api/v1/balances` with the parameters above.
**Result:** Mirror node REST service degradation / denial of service.
