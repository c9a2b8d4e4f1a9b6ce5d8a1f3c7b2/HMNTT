### Title
Unbounded Unthrottled Gap-Recovery Query via Safety Check When `last == null`

### Summary
The one-shot safety check in `TopicMessageServiceImpl.subscribeTopic()` fires `missingMessages(topicContext, null)` after one second. When no message has been delivered yet (`last == null`), `missingMessages()` builds a `gapFilter` that inherits the original user-supplied `startTime`, `endTime`, and `limit` verbatim, then calls `topicMessageRetriever.retrieve(gapFilter, false)` — the `false` flag bypasses throttling. Any unprivileged subscriber can craft a filter with an arbitrarily old `startTime`, no `endTime`, and no `limit`, causing an unthrottled full-history scan to run concurrently with the already-running historical retrieval, leading to memory exhaustion.

### Finding Description

**Exact code path:**

`subscribeTopic()` launches the safety check unconditionally: [1](#0-0) 

After one second it calls `missingMessages(topicContext, null)`. Inside that method, the `current == null` branch: [2](#0-1) 

When `last == null` (no message has been emitted yet), `startTime` falls back to `topicContext.getFilter().getStartTime()` — the raw user-supplied value. The `gapFilter` is built with `.toBuilder().startTime(startTime).build()`, which copies every other field from the original filter unchanged:

- `endTime` → `null` if the user omitted it (no upper bound)
- `limit` → `0` if the user omitted it (no row cap)

The retriever is then called with `throttled = false`, bypassing any rate-limiting applied to the normal historical retrieval at line 63 (`retrieve(filter, true)`).

**Root cause / failed assumption:** The safety check assumes that by the time it fires, at least one message will have been delivered (`last != null`), so the `gapFilter` will be bounded by `last.getConsensusTimestamp() + 1`. When the historical retrieval is slow (e.g., DB under load, partial network partition), this assumption is violated and the fallback path issues an unbounded, unthrottled query.

**Why existing checks are insufficient:**

- `isValidStartTime()` only requires `startTime <= now()` — any past timestamp, including near-epoch, is accepted. [3](#0-2) 
- `isComplete()` only returns `true` when `endTime` is set and has elapsed — a no-`endTime` subscription is never complete. [4](#0-3) 
- The duplicate-sequence filter at lines 74-77 filters emitted *items* but does not prevent the DB query from executing and loading all rows into memory. [5](#0-4) 
- `GrpcProperties` has no per-subscriber query size cap or maximum result-set limit. [6](#0-5) 

### Impact Explanation
An attacker opens many concurrent subscriptions with `startTime` set to the earliest possible timestamp, no `endTime`, and no `limit`. Each subscription triggers two simultaneous unbounded DB scans — the throttled historical one and the unthrottled safety-check one — after just one second of artificial delay (e.g., by subscribing during any period of DB slowness or by simply relying on the 1-second window before the first message arrives on a high-latency topic). The result is heap exhaustion on the mirror-node gRPC process, causing an OOM crash and denial of service for all subscribers.

### Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. The gRPC `subscribeTopic` endpoint is publicly reachable. The attacker only needs to know a valid topic ID (trivially discoverable from the public ledger) and to subscribe before the first message arrives within the 1-second window — a reliable condition for any low-traffic or newly created topic, or during any period of DB latency. The attack is repeatable and scriptable.

### Recommendation
1. **Cap the safety-check query**: When building `gapFilter` in the `current == null` branch, always set an explicit `endTime` (e.g., `DomainUtils.now()`) and a maximum `limit` (e.g., a configurable `maxSafetyCheckMessages` property), regardless of what the original filter contains.
2. **Throttle the safety-check retrieval**: Pass `true` (throttled) to `retrieve()` in the safety-check path, consistent with the historical retrieval.
3. **Enforce a server-side maximum result-set size** in `GrpcProperties` and apply it inside `TopicMessageRetriever` implementations so no single query can return an unbounded number of rows.
4. **Rate-limit concurrent subscriptions per client** to reduce the blast radius of a scripted attack.

### Proof of Concept
1. Identify any valid topic ID on the network (e.g., topic `0.0.1234`).
2. Open a gRPC `subscribeTopic` stream with:
   - `topicID = 0.0.1234`
   - `consensusStartTime = 0` (epoch, or the earliest valid nanosecond timestamp)
   - no `consensusEndTime`
   - no `limit`
3. Ensure the topic has low message traffic so no message is delivered within the first second (or introduce artificial latency by subscribing during a DB-slow period).
4. After ~1 second, the safety check fires. Because `last == null`, `missingMessages()` builds a `gapFilter` with `startTime = 0`, `endTime = null`, `limit = 0` and calls `retrieve(gapFilter, false)`.
5. The retriever issues a full-table scan of all topic messages from epoch with no row cap and no throttle, concurrently with the already-running historical scan.
6. Repeat with hundreds of concurrent connections to exhaust heap and crash the gRPC service.
