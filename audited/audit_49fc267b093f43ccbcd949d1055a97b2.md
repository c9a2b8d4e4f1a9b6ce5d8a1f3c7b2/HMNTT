### Title
Slowloris / Slow-Read Connection Exhaustion on `GET /blocks` Due to Missing HTTP Server Write Timeout and Connection Limits

### Summary
The Node.js HTTP server in `rest/server.js` is started with `app.listen()` but no `keepAliveTimeout`, `headersTimeout`, or write/response-flush timeout is configured on the returned server object. Combined with the absence of any rate-limiting or connection-count middleware, an unprivileged attacker can open many `GET /blocks` connections and read responses at near-zero speed (slow-read / Slowloris variant), holding each socket open indefinitely while the server buffers the full JSON payload waiting for the TCP window to drain, exhausting file descriptors and connection slots across mirror-node instances.

### Finding Description

**Code path:**

`rest/server.js` lines 150–166 — the HTTP server is created and started:

```js
const server = app.listen(port, '0.0.0.0', (err) => { ... });
createTerminus(server, { ... });
```

No properties are set on `server` after `app.listen()` returns. Specifically absent:
- `server.keepAliveTimeout` (Node.js default: 5 000 ms — only fires *after* a response is fully flushed, not during a slow read)
- `server.headersTimeout` (Node.js default: 60 000 ms — only guards slow *request* headers, not slow response reads)
- Any socket write/drain timeout

`rest/routes/blockRoute.js` line 12:
```js
router.getExt('/', BlockController.getBlocks);
```

`rest/controllers/blockController.js` lines 101–112 — `getBlocks` builds a complete in-memory JSON object and stores it in `res.locals`:
```js
res.locals[responseDataLabel] = {
  blocks: blocks.map((model) => new BlockViewModel(model)),
  links: { next: this.generateNextLink(req, blocks, formattedFilters) },
};
```

The `responseHandler` middleware then calls `res.json(...)`, which serialises the entire payload and hands it to Node.js's TCP send buffer. If the client's TCP receive window is near zero, Node.js cannot drain the write buffer; the socket stays open and the event-loop callback for that write is never resolved. The connection is held open until the OS-level socket timeout (hours on most Linux defaults).

**No rate-limiting or connection-count middleware is registered** anywhere in `rest/server.js` or `rest/middleware/` (confirmed by directory listing and grep for `rateLimit`, `express-rate-limit`, `maxConnections`).

**Root cause:** The server object returned by `app.listen()` is never configured with a write/response timeout, and no middleware enforces per-IP connection limits or request rates.

### Impact Explanation

Each held connection consumes a file descriptor, a Node.js socket object, and the buffered response payload in memory. With a default or configured `limit` parameter, each `GET /blocks` response can be several kilobytes to tens of kilobytes of JSON. An attacker opening thousands of slow-read connections across multiple mirror-node replicas can exhaust the OS file-descriptor limit (`ulimit -n`, typically 65 535) and Node.js heap, causing new legitimate connections to be refused with `ECONNREFUSED` or silently dropped. Because the mirror-node REST API is the primary read interface for block data, exhausting ≥30 % of deployed instances degrades network observability and downstream integrations (wallets, explorers, dApps) without requiring any authentication or special privileges.

### Likelihood Explanation

The attack requires only a standard TCP socket library (e.g., Python `socket`, `hping3`, or `slowhttptest --slowread`). No credentials, API keys, or knowledge of internal state are needed. The attacker needs to sustain ~thousands of concurrent slow-read connections per node, which is achievable from a single host with a modest number of open file descriptors or from a small botnet. The attack is repeatable and silent — it generates no error-level log entries, only normal access-log lines. The absence of any reverse-proxy timeout enforcement in the codebase means the protection gap exists at the application layer regardless of deployment topology unless an operator independently configures an ingress controller.

### Recommendation

1. **Set server-level timeouts immediately after `app.listen()`** in `rest/server.js`:
   ```js
   server.keepAliveTimeout = 65_000;   // ms — must exceed any LB idle timeout
   server.headersTimeout   = 66_000;   // ms — must be > keepAliveTimeout
   // Node.js has no built-in write timeout; use a socket-level approach:
   server.on('connection', (socket) => {
     socket.setTimeout(30_000);        // fires if socket is idle in either direction
     socket.on('timeout', () => socket.destroy());
   });
   ```
2. **Add per-IP rate limiting** via `express-rate-limit` before the block route.
3. **Set an explicit connection limit** (`server.maxConnections = N`) or enforce it at the ingress/load-balancer layer.
4. **Deploy behind a reverse proxy** (nginx/Envoy) with `send_timeout` / `proxy_read_timeout` configured to terminate stalled response flushes.

### Proof of Concept

```python
import socket, time, threading

TARGET = ('mirror-node-host', 5551)
PATH   = b'GET /api/v1/blocks HTTP/1.1\r\nHost: mirror-node-host\r\nConnection: keep-alive\r\n\r\n'

def slow_read():
    s = socket.socket()
    s.connect(TARGET)
    # Advertise a 1-byte receive window via SO_RCVBUF to force slow drain
    s.setsockopt(socket.SOL_SOCKET, socket.SO_RCVBUF, 1)
    s.send(PATH)
    # Read one byte every 60 seconds — connection stays open indefinitely
    while True:
        s.recv(1)
        time.sleep(60)

threads = [threading.Thread(target=slow_read, daemon=True) for _ in range(5000)]
for t in threads:
    t.start()
# After saturation, legitimate GET /blocks requests receive ECONNREFUSED
```

Running this against ≥30 % of mirror-node replicas simultaneously causes those instances to stop accepting new connections, degrading block-data availability across the network.
