### Title
Unbounded gRPC Subscription Duration Enables Resource-Exhaustion DoS via Indefinite Stream Accumulation

### Summary
The `listen()` method in `TopicListener` and its callers impose no server-side maximum subscription duration. When a client subscribes without an `endTime`, `TopicMessageServiceImpl.pastEndTime()` explicitly returns `Flux.never()`, making the stream immortal. Because the per-connection stream cap (`maxConcurrentCallsPerConnection = 5`) is not a global limit, an attacker opens arbitrarily many connections and fills each with 5 perpetual streams, exhausting the server's thread pool, DB connection pool, and memory.

### Finding Description

**Code path and root cause:**

`TopicMessageFilter.endTime` is declared nullable with no maximum-value constraint: [1](#0-0) 

In `TopicMessageServiceImpl.subscribeTopic()`, the termination signal `pastEndTime()` is only wired in when `endTime != null`. When it is null, the method explicitly returns `Flux.never()` — a stream that emits nothing and never completes — so `takeUntilOther` never fires: [2](#0-1) 

The `takeWhile` and `take(limit)` guards are also conditional on the client supplying values: [3](#0-2) 

The underlying `PollingTopicListener.listen()` schedules `RepeatSpec.times(Long.MAX_VALUE)` — effectively infinite DB polling every 500 ms — on a `boundedElastic` scheduler: [4](#0-3) 

The only server-side stream cap is `maxConcurrentCallsPerConnection = 5`, which is **per-connection**, not global. No `maxInboundConnections`, `maxConnectionAge`, or global stream counter is configured: [5](#0-4) [6](#0-5) 

**Exploit flow:**

1. Attacker opens N TCP connections to port 5600.
2. On each connection, attacker sends 5 `subscribeTopic` RPCs with a valid `topicId`, `startTime = now`, no `endTime`, no `limit`.
3. Each RPC passes all `@Validated` constraints (no `endTime` is explicitly allowed).
4. `pastEndTime()` returns `Flux.never()` for each; the reactive pipeline never terminates.
5. `PollingTopicListener.listen()` begins polling the DB every 500 ms per stream, indefinitely.
6. With N connections, 5N streams are open simultaneously, each holding a `boundedElastic` thread slot and issuing periodic DB queries.
7. `boundedElastic` default cap is `10 × CPU cores` (e.g., 80 threads on an 8-core node). With 16 connections the thread pool saturates.
8. DB connection pool exhaustion follows; legitimate subscribers' queries queue or time out.

### Impact Explanation

Every open stream without `endTime` permanently occupies a scheduler thread and issues a DB query every 500 ms. Thread pool exhaustion (`boundedElastic`) causes new subscriptions to queue indefinitely. DB connection pool exhaustion causes query timeouts for all other gRPC and REST consumers. The result is a complete denial of service to all new topic subscribers and degraded service to existing ones, matching the "≥30% of network processing nodes" severity tier described in the question scope.

### Likelihood Explanation

No authentication or rate-limiting is required. Any client that can reach port 5600 and knows a valid `topicId` (publicly observable on-chain) can execute this attack. The attack is trivially scriptable: open connections in a loop, send subscribe RPCs with no `endTime`. It is repeatable and persistent — streams survive until the attacker's TCP connections are dropped. A single attacker machine with modest bandwidth can open hundreds of connections.

### Recommendation

1. **Enforce a server-side maximum subscription duration.** Add a `@Max` or custom `@AssertTrue` constraint to `TopicMessageFilter` capping `endTime - startTime` (e.g., 1 hour). Reject filters where `endTime == null` and `limit == 0` simultaneously, or impose a default `endTime = startTime + maxDuration` when neither is supplied.
2. **Add a global concurrent-subscription limit.** Gate `subscribeTopic()` on `subscriberCount` and return `RESOURCE_EXHAUSTED` when a configurable ceiling is reached.
3. **Configure Netty connection-level limits.** Add `maxConnectionAge`, `maxConnectionIdle`, and `maxInboundConnections` to `GrpcConfiguration` to bound the total number of open connections and their lifetimes.
4. **Apply per-IP rate limiting** at the gRPC interceptor layer to prevent a single source from opening many connections.

### Proof of Concept

```python
import grpc
import threading
# proto: hiero/mirror/api/proto/ConsensusService.proto

def open_stream(stub, topic_id):
    req = consensus_pb2.ConsensusTopicQuery(
        topicID=topic_id,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=int(time.time())),
        # no consensusEndTime, no limit
    )
    for _ in stub.subscribeTopic(req):
        pass  # stream never completes; blocks thread

threads = []
for conn_idx in range(20):           # 20 connections
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_grpc.ConsensusServiceStub(channel)
    for stream_idx in range(5):      # 5 streams per connection = 100 total
        t = threading.Thread(target=open_stream, args=(stub, valid_topic_id))
        t.daemon = True
        t.start()
        threads.append(t)

# After ~seconds: new subscribeTopic calls from legitimate clients
# receive UNAVAILABLE or hang indefinitely as thread pool is saturated.
```
