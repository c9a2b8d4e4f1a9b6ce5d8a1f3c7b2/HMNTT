### Title
Unauthenticated Thundering-Herd DoS via Uncached, Unthrottled Triple-DB-Query in `getTopic()`

### Summary
`getTopic()` in `TopicController` issues three independent synchronous DB queries per request with no server-side caching, no request deduplication, and no rate limiting in the `rest-java` module. An unprivileged attacker flooding the endpoint with concurrent requests for the same topic ID causes every request to independently hit the database three times, enabling sustained CPU and DB I/O amplification well above 30%.

### Finding Description

**Exact code path:**

`getTopic()` at [1](#0-0)  issues three sequential, independent DB calls:
- `topicService.findById(id.id())` → `TopicServiceImpl` → `topicRepository.findById()` [2](#0-1) 
- `entityService.findById(id.id())` → `EntityServiceImpl` → `entityRepository.findById()` [3](#0-2) 
- `customFeeService.findById(id.id())` → `CustomFeeServiceImpl` → `customFeeRepository.findById()` [4](#0-3) 

**Root cause — no server-side caching in `rest-java`:**

A grep for `@Cacheable` across all `rest-java` service/repository classes returns zero matches. Contrast this with the `grpc` module's `EntityRepository`, which carries `@Cacheable` with a 24-hour Caffeine cache. [5](#0-4)  The `rest-java` equivalents have no such annotation.

**No rate limiting in `rest-java`:**

The only servlet filters registered in `rest-java` are `LoggingFilter` [6](#0-5)  and `MetricsFilter` [7](#0-6)  — neither enforces any rate limit. The Bucket4j-based `ThrottleConfiguration` and `ThrottleManagerImpl` exist only in the `web3` module. [8](#0-7) 

**The only partial mitigation — HTTP `Cache-Control` response header:**

The test confirms the response carries `Cache-Control: public, max-age=5`. [9](#0-8)  This is a *client/proxy* hint only. It does not prevent the application server from processing every request that reaches it directly (e.g., when no CDN/reverse-proxy is deployed, or when an attacker bypasses it by varying `Cache-Control: no-cache` request headers or hitting the origin directly).

**Exploit flow:**

1. Attacker identifies a valid topic ID (public blockchain data, no secret required).
2. Attacker opens N concurrent HTTP connections and sends `GET /api/v1/topics/{id}` simultaneously.
3. Each request independently executes 3 DB queries (topic table, entity table, custom_fee table).
4. N concurrent requests → 3N concurrent DB queries with no coalescing, no cache hit, no throttle rejection.
5. DB connection pool saturates; CPU spikes from query parsing, plan execution, and result serialization.

### Impact Explanation

Each request multiplies DB load by a factor of 3. With no rate limiting or server-side cache, a modest flood (e.g., 500 concurrent requests/second) produces 1,500 DB queries/second against what would otherwise be 0 (if cached) or 500 (if single-query). This easily exceeds the 30% resource increase threshold. Sustained attack degrades response times for all other endpoints sharing the same DB connection pool and PostgreSQL instance.

### Likelihood Explanation

- **No authentication required**: the endpoint is fully public. [10](#0-9) 
- **No special knowledge needed**: topic IDs are enumerable from the public ledger.
- **Trivially repeatable**: standard HTTP load tools (`wrk`, `hey`, `ab`) suffice.
- **Bypassing the CDN cache**: sending `Cache-Control: no-cache` or `Pragma: no-cache` request headers, or targeting the origin IP directly, bypasses any upstream proxy cache.

### Recommendation

1. **Add server-side caching** on `TopicServiceImpl.findById`, `EntityServiceImpl.findById`, and `CustomFeeServiceImpl.findById` using `@Cacheable` with a short TTL (e.g., 5 seconds, matching the existing `Cache-Control` header), backed by Caffeine — mirroring the pattern already used in the `grpc` module's `EntityRepository`.
2. **Add rate limiting** to the `rest-java` module for the `/api/v1/topics/{id}` endpoint, using Bucket4j (already a dependency in `web3`) or Spring's `HandlerInterceptor`.
3. **Parallelize the three DB calls** (e.g., using `CompletableFuture`) to reduce per-request latency even before caching is added.

### Proof of Concept

```bash
# Flood the endpoint with 200 concurrent connections, 10000 total requests
wrk -t 20 -c 200 -d 30s \
    -H "Cache-Control: no-cache" \
    http://<mirror-node-host>/api/v1/topics/0.0.1234

# Observe PostgreSQL active queries during the attack:
# SELECT count(*) FROM pg_stat_activity WHERE state = 'active';
# Expected: count spikes to ~3x the concurrent connection count,
# confirming 3 independent queries per HTTP request with no deduplication.
```

public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
