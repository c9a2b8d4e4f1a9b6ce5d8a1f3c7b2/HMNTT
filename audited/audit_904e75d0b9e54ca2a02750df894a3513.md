### Title
Silent Node Exclusion via Empty `NodeAccountID` in Mirror REST Response Allows Partial Network Map Construction

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` silently skips any `nodeEntry` whose `node_account_id` field is empty (line 103–105), with no minimum-node-count guard beyond `len(network) == 0`. An attacker who can influence the mirror REST response can return a payload where ≥30% of node records carry an empty `node_account_id`, causing those nodes to be silently dropped and the pinger to operate against only a subset of the network — with no error, no log, and no operator alert.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 102–125:

```go
for _, n := range payload.Nodes {
    if n.NodeAccountID == "" {   // line 103 — silent skip, no counter, no log
        continue
    }
    ...
}

if len(network) == 0 {           // line 127 — only guard: total wipeout
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

**Root cause:** The only integrity check on the returned node list is a zero-length guard. Any partial exclusion (1 to N-1 nodes dropped) passes silently.

**Exploit flow:**
1. Attacker positions themselves to influence the mirror REST response (see Likelihood).
2. Attacker crafts a `nodesEnvelope` JSON response where ≥30% of `nodeEntry` objects have `"node_account_id": ""` (or omit the field entirely, which JSON-decodes to the zero value `""`).
3. `fetchMirrorNodeNetwork()` iterates the list; for each such entry, `n.NodeAccountID == ""` is true → `continue` → node is dropped with no log entry.
4. The remaining ≤70% of nodes form the returned `network` map.
5. `len(network) > 0` → no error is returned; `buildNetworkFromMirrorNodes()` returns the partial map.
6. `hiero.ClientForNetwork(netmap)` is called with the reduced map; the pinger only ever contacts the surviving subset.

**Why existing checks fail:**
- The `len(network) == 0` guard (line 127) only fires on total wipeout. Any partial exclusion is invisible.
- There is no minimum-node-count threshold, no comparison against an expected node count, and no logging of skipped entries.
- The `http.Client` (line 46) uses the default transport with no TLS certificate pinning, no response signature, and no HMAC/JWT authentication on the mirror REST response, so a network-positioned attacker can substitute a crafted payload.

### Impact Explanation
The pinger's purpose is to detect network-wide liveness issues. If ≥30% of consensus nodes are silently removed from its view, the pinger will never send transactions to those nodes and will never detect their failure. This satisfies the stated scope condition ("shutdown of ≥30% of network processing nodes without brute force"). In a broader deployment where this same `buildNetworkFromMirrorNodes` path is used to bootstrap an SDK client, transactions would be routed only to the surviving subset, reducing decentralization and potentially enabling targeted censorship or denial-of-service against specific nodes without triggering any alarm.

### Likelihood Explanation
**Precondition:** The attacker must be able to substitute or modify the mirror REST response. Realistic paths:
- If `cfg.mirrorRest` is an `http://` URL (no TLS), any on-path attacker (ISP, cloud provider, same LAN) can trivially MITM.
- If HTTPS is used without certificate pinning (the code uses the default `http.Transport` with no pinning), a DNS-poisoning or rogue-CA attack suffices.
- If the mirror node itself is operated by a semi-trusted third party (common in permissionless deployments), the operator of that mirror node can serve a crafted response.
- No authentication credential is required; the endpoint is a plain unauthenticated HTTP GET.

The attack is repeatable on every startup/reconnect cycle and requires no privileged access to the pinger process or the consensus network.

### Recommendation
1. **Add a minimum-node-count check:** After building `network`, verify that the count of distinct `AccountID`s meets a configurable minimum (e.g., the known network size or at least a floor like 5). Return an error if the threshold is not met.
2. **Log and count skipped entries:** Emit a warning (and optionally a metric) for every node skipped due to an empty or unparseable `NodeAccountID`.
3. **Enforce TLS with certificate pinning or a trusted CA bundle** for the mirror REST connection; reject responses from endpoints that cannot be authenticated.
4. **Optionally verify response integrity** via a signed manifest or compare the returned node list against a locally configured expected set.

### Proof of Concept
**Setup:** Run the pinger with `network=other` and `mirrorRest=http://attacker-controlled-host/`.

**Attacker serves at `GET /api/v1/network/nodes`:**
```json
{
  "nodes": [
    { "node_account_id": "",        "service_endpoints": [{"ip_address_v4":"1.2.3.4","port":50211}] },
    { "node_account_id": "",        "service_endpoints": [{"ip_address_v4":"1.2.3.5","port":50211}] },
    { "node_account_id": "",        "service_endpoints": [{"ip_address_v4":"1.2.3.6","port":50211}] },
    { "node_account_id": "0.0.4",   "service_endpoints": [{"ip_address_v4":"1.2.3.7","port":50211}] }
  ],
  "links": { "next": null }
}
```

**Result:** 3 of 4 nodes (75%) are silently dropped. `buildNetworkFromMirrorNodes()` returns `{"1.2.3.7:50211": AccountID{0,0,4}}` with no error. The pinger operates against only node `0.0.4`, never detecting failures on nodes `0.0.1`–`0.0.3`.
