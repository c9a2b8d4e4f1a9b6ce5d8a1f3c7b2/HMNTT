### Title
Unauthenticated Large IN-Clause Griefing via `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse`

### Summary
Any unauthenticated caller can supply up to 100 distinct, non-existent 32-byte `key=eq:0x…` parameters to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`. Because the only pre-query guard (`keysInRange.isEmpty()`) never fires when the default key bounds span the full byte range, all 100 keys are forwarded verbatim to the database as a 100-element `IN`-clause that always returns zero rows. Repeating this at high frequency causes sustained unnecessary index lookups and can exhaust the database connection pool, degrading service for legitimate users.

### Finding Description

**Code path:**

1. **Controller** — `HooksController.java` line 108: `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` permits up to 100 `key` query parameters. [1](#0-0) 

2. **Controller** — `hookStorageChangeRequest` (lines 165–184): every `key=eq:0x…` parameter is appended to `keyFilters` with no deduplication or existence check. Default `lowerBound = MIN_KEY_BYTES` (all `0x00`) and `upperBound = MAX_KEY_BYTES` (all `0xFF`). [2](#0-1) 

3. **Service** — `HookServiceImpl.getHookStorage` (lines 72–79): calls `request.getKeysInRange()`, which filters keys to those within `[keyLowerBound, keyUpperBound]`. Because the bounds default to `[0x00…00, 0xFF…FF]`, every valid 32-byte key passes the filter. The early-exit guard `if (keysInRange.isEmpty())` (line 74) therefore **never fires** when the attacker supplies valid-format keys. [3](#0-2) 

4. **DTO** — `HookStorageRequest.getKeysInRange()` (lines 43–52): pure byte-range comparison; no existence check. [4](#0-3) 

5. **Repository** — `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse` (lines 13–14): Spring Data JPA generates `WHERE owner_id = ? AND hook_id = ? AND key IN (?, …×100) AND deleted = false`. The database evaluates 100 index probes and returns zero rows. [5](#0-4) 

**Root cause:** The `keysInRange.isEmpty()` guard is the sole pre-query short-circuit, but it is structurally unreachable when the attacker supplies any valid 32-byte hex keys (which always fall within the full-range defaults). There is no deduplication, no existence pre-check, and no rate limiting in the visible code path.

**Why existing checks fail:**
- `@Size(max = 100)` caps the count at 100 but does not prevent 100 non-existent keys from reaching the DB. [6](#0-5) 
- `keysInRange.isEmpty()` only returns early when all supplied keys fall outside the explicit range bounds, which never happens with the default full-range bounds. [7](#0-6) 

### Impact Explanation
Each crafted request forces the database to evaluate a 100-element `IN`-clause against the `hook_storage` index, returning zero rows. Sustained high-frequency requests from one or more attackers increase DB CPU and I/O, hold connections open for the duration of each query, and can exhaust the connection pool, causing legitimate queries to queue or time out. The impact is service degradation (griefing) with no direct economic loss to network participants.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The attack requires only knowledge of any valid `ownerId`/`hookId` pair (discoverable via the hooks listing API) and the ability to send HTTP GET requests. The payload is trivially constructed (100 random hex strings), is repeatable at arbitrary frequency, and requires no special tooling — a simple shell loop or any HTTP load tool suffices.

### Recommendation
Apply one or more of the following mitigations:

1. **Reduce the effective IN-clause cap for key lookups** — use a separate, lower constant (e.g., 10–20) for the `key` parameter rather than reusing the global `MAX_REPEATED_QUERY_PARAMETERS = 100`.
2. **Deduplicate keys before the query** — convert `keyFilters` to a `Set` before passing to the repository to eliminate redundant probes.
3. **Add rate limiting** — apply per-IP or per-account request throttling at the API gateway or Spring filter layer for this endpoint.
4. **Require authentication** for key-specific lookups — restrict the `key=eq:…` filter form to authenticated callers.

### Proof of Concept

```bash
# Discover a valid ownerId/hookId via the hooks listing API (unauthenticated)
OWNER=0.0.12345
HOOK_ID=1
BASE_URL="https://<mirror-node-host>/api/v1/accounts/${OWNER}/hooks/${HOOK_ID}/storage"

# Build a query string with 100 distinct non-existent 32-byte keys
PARAMS=""
for i in $(seq 1 100); do
  KEY=$(printf "%064x" $RANDOM$RANDOM$RANDOM$RANDOM)
  PARAMS="${PARAMS}&key=eq:0x${KEY}"
done

# Send the request repeatedly to grieve the DB connection pool
while true; do
  curl -s "${BASE_URL}?${PARAMS}" -o /dev/null &
done
```

Each iteration causes the database to execute:
```sql
SELECT * FROM hook_storage
WHERE owner_id = <ownerId>
  AND hook_id = <hookId>
  AND key IN (<100 non-existent values>)
  AND deleted = false
-- Returns 0 rows after 100 index probes
```
