### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Alias Lookups in GraphQL `account` Query

### Summary
The GraphQL `account(input: AccountInput!)` endpoint accepts an `alias` field validated only by a regex pattern (`^[A-Z2-7]+$`). The Caffeine `PreparsedDocumentProvider` caches only parsed/validated query documents keyed by the raw query string — it does not cache query results. The `EntityServiceImpl.getByAliasAndType()` method has no result cache and issues a direct database call on every invocation. With no rate limiting present in the GraphQL module, an unprivileged attacker can flood the endpoint with concurrent requests carrying distinct valid alias values, exhausting the HikariCP connection pool and denying service to all users.

### Finding Description

**Cache scope — parsed documents only, not results:**

`CachedPreparsedDocumentProvider` caches by `executionInput.getQuery()` — the raw query string. [1](#0-0) 

Default cache spec is `expireAfterWrite=1h,maximumSize=1000`. [2](#0-1) 

When an attacker inlines the alias value directly in the query body (e.g., `{ account(input: { alias: "ABCDE" }) { id } }` vs `{ account(input: { alias: "ABCDF" }) { id } }`), each unique alias string produces a unique cache key → cache miss → re-parse + re-validate + resolver execution. Even when using GraphQL variables (same query string → cache hit), the resolver still executes and a DB call is made.

**No result caching in the service layer:**

`EntityServiceImpl.getByAliasAndType()` calls `entityRepository.findByAlias(decodeBase32(alias))` with no `@Cacheable` annotation or any in-memory result cache. [3](#0-2) 

**No rate limiting in the GraphQL module:**

The `ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` module. The GraphQL module has no equivalent. The design document explicitly notes "We should implement rate limiting in the load balancer" — confirming it is not implemented in the application. [4](#0-3) 

`GraphQlConfiguration` only applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — these limit structural query complexity, not request rate. [5](#0-4) 

**Input validation is insufficient:**

The `@Pattern(regexp: "^[A-Z2-7]+$")` directive on `alias` only enforces character set — it does not bound alias length meaningfully (the Jackson `maxStringLength` is 11000) and does not limit request rate. [6](#0-5) 

**Exploit flow:**

1. Attacker generates N distinct Base32 strings matching `^[A-Z2-7]+$` (trivially: `AAAA`, `AAAB`, `AAAC`, …).
2. Sends N concurrent POST requests to `/graphql/alpha` with inlined alias values.
3. Each request: document cache miss (unique query string) → parse → validate → `AccountController.account()` → `EntityServiceImpl.getByAliasAndType()` → `entityRepository.findByAlias()` → DB query.
4. HikariCP's default pool (10 connections for Spring Boot) is saturated; subsequent legitimate requests block or time out.

### Impact Explanation

Complete denial of service for the GraphQL API. All users — including legitimate ones querying by `entityId` or `evmAddress` — are blocked when the connection pool is exhausted. The `statementTimeout` of 10 seconds means each attacker connection holds a pool slot for up to 10 seconds, requiring only ~10 concurrent requests to saturate the default pool. The attack is amplified by the document cache eviction: with `maximumSize=1000`, sustained flooding with >1000 unique query strings continuously evicts cached entries, adding CPU re-parsing overhead on top of DB pressure. [7](#0-6) 

### Likelihood Explanation

No authentication is required. The alias alphabet (`A-Z`, `2-7`) provides 32 characters; a 4-character alias yields 32⁴ = 1,048,576 unique valid values — far more than needed. The attack requires only an HTTP client capable of concurrent requests (e.g., `curl`, `ab`, `wrk`). It is repeatable indefinitely and requires no special knowledge of the system beyond the public GraphQL schema.

### Recommendation

1. **Add result caching** in `EntityServiceImpl.getByAliasAndType()` using `@Cacheable` with a short TTL (e.g., 30s) to avoid redundant DB lookups for the same alias.
2. **Implement application-level rate limiting** in the GraphQL module, mirroring the `ThrottleConfiguration` / `ThrottleManagerImpl` pattern from the `web3` module, applied via a `WebGraphQlInterceptor`.
3. **Bound alias length** at the schema or validation layer (e.g., max 64 characters) to reduce the attack surface and parsing cost.
4. **Increase HikariCP pool size** or configure a connection acquisition timeout to fail fast rather than queue indefinitely under load.

### Proof of Concept

```bash
# Generate 200 unique valid alias values and fire concurrent requests
for i in $(seq 1 200); do
  ALIAS=$(python3 -c "import random, string; print(''.join(random.choices('ABCDEFGHIJKLMNOPQRSTUVWXYZ234567', k=8)))")
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { alias: \\\"${ALIAS}\\\" }) { id } }\"}" &
done
wait
# Observe: legitimate requests begin timing out or returning connection errors
# Monitor: hikaricp_connections_pending metric spikes, hikaricp_connections_active saturates
```
