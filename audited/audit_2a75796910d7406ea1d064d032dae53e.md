### Title
Unbounded Concurrent `subscribeTopic()` Subscriptions Enable Resource Exhaustion DoS

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` accepts unlimited concurrent infinite-duration subscriptions from any unauthenticated caller. The only server-side guard — `maxConcurrentCallsPerConnection=5` — is scoped per TCP connection, not globally, so an attacker opening many connections bypasses it entirely. With `limit=0` and no `endTime`, each subscription is perpetual, and the cumulative load exhausts the shared Reactor `boundedElastic` scheduler thread pool, the database connection pool, memory, and OS file descriptors, rendering the gRPC service unable to serve new requests.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) builds a reactive pipeline and subscribes immediately with no authentication, no per-IP throttle, and no global subscription cap:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        ...
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
``` [1](#0-0) 

**Infinite subscription condition:**

`TopicMessageFilter.hasLimit()` returns `false` when `limit == 0`:
```java
public boolean hasLimit() { return limit > 0; }
``` [2](#0-1) 

In `TopicMessageServiceImpl.subscribeTopic()`, when `hasLimit()` is false, `flux.take()` is never applied (line 83–85). When `endTime` is null, `pastEndTime()` returns `Flux.never()` (line 124–125), so the subscription never terminates: [3](#0-2) [4](#0-3) 

**Insufficient guard — per-connection only:**

`GrpcConfiguration` sets only `maxConcurrentCallsPerConnection` (default 5). No `maxConnections`, no `maxConnectionAge`, no IP-based rate limit is configured: [5](#0-4) [6](#0-5) 

An attacker opening N TCP connections can hold 5N concurrent infinite subscriptions.

**Shared `boundedElastic` scheduler saturation:**

`PollingTopicListener`, `SharedPollingTopicListener`, `PollingTopicMessageRetriever`, and the safety-check path in `TopicMessageServiceImpl` all call `Schedulers.boundedElastic()`, which returns a global singleton thread pool capped at `10 × CPU_count` threads. Each active polling subscription occupies a thread slot on every poll cycle. With enough subscriptions, the pool saturates and all subsequent scheduler tasks queue behind the backlog, stalling message delivery and gap-recovery queries for all subscribers. [7](#0-6) [8](#0-7) 

**Database connection pool exhaustion:**

Each polling subscription issues periodic DB queries. The HikariCP pool is finite. With enough concurrent subscriptions all polling simultaneously, the pool is exhausted and new `topicExists()` checks (required for every new subscription) block indefinitely, causing new subscription attempts to hang or fail. [9](#0-8) 

### Impact Explanation

The gRPC mirror node service becomes unable to accept or process new `subscribeTopic` calls. Existing legitimate subscribers experience stalled message delivery. DB connection exhaustion cascades to all DB-dependent operations in the service. Memory and file-descriptor exhaustion can cause JVM OOM or OS-level connection refusals. This is a full denial-of-service of the mirror node's HCS subscription API, which downstream clients and applications depend on for real-time consensus data.

### Likelihood Explanation

No authentication or API key is required. The attack requires only a network path to port 5600 and the ability to open many TCP connections — achievable from a single machine or a small botnet. The `grpcurl` tool (referenced in the project's own docs) can trivially script this. The attack is repeatable and persistent as long as the attacker holds connections open. The default `maxConcurrentCallsPerConnection=5` provides no meaningful protection since there is no connection-count ceiling.

### Recommendation

1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., 1000).
2. **Set `maxConnectionAge` and `maxConnections`** on the `NettyServerBuilder` in `GrpcConfiguration` to bound total TCP connections.
3. **Enforce a maximum subscription duration** — reject or auto-terminate subscriptions with no `endTime` after a configurable TTL, requiring clients to re-subscribe.
4. **Add per-IP rate limiting** at the load-balancer or ingress layer (e.g., Traefik middleware already partially configured in the Helm chart).
5. **Expose `subscriberCount` as a circuit-breaker metric** and return `RESOURCE_EXHAUSTED` gRPC status when the threshold is exceeded.

### Proof of Concept

```bash
# Open 200 connections, each with 5 concurrent infinite subscriptions (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Verify service is unresponsive to new legitimate subscriptions:
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: hangs or returns UNAVAILABLE / RESOURCE_EXHAUSTED
```
