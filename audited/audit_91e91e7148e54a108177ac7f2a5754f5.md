### Title
Unauthenticated Concurrent `getNodes()` Flood Exhausts Database Connection Pool via Unbounded gRPC Connections

### Summary
The `getNodes()` handler in `NetworkController.java` requires no authentication and is subject to no application-level rate limiting. An attacker can open an unbounded number of gRPC connections — each carrying up to 5 concurrent streaming calls — causing `NetworkServiceImpl.getNodes()` to hold HikariCP database connections for the full duration of paginated address book retrieval (with a 250 ms inter-page delay). Flooding enough connections exhausts the shared DB pool and denies service to all legitimate gRPC callers.

### Finding Description

**Code path:**

`NetworkController.getNodes()` (lines 33–43) immediately subscribes a Reactor pipeline with no auth check and no rate gate:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(networkService::getNodes)
        ...
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`NetworkServiceImpl.getNodes()` (lines 55–77) then executes inside `transactionOperations.execute()` — holding a JDBC connection from the HikariCP pool — and pages through the address book with a hard-coded 250 ms delay between pages on `Schedulers.boundedElastic()`:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())   // 250 ms default
                .withScheduler(Schedulers.boundedElastic()))
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
```

When `limit = 0` (the protobuf default), `take(Long.MAX_VALUE)` is used, meaning the full address book is streamed.

**Server-side connection limit:**

`GrpcConfiguration.java` (lines 28–35) configures only `maxConcurrentCallsPerConnection` (default 5 per `NettyProperties.java` line 14). No `maxConnectionAge`, no global connection count cap, and no per-IP limit is set:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

**No rate limiting in the gRPC module:**

The only `ServerInterceptor` present (`grpc/src/test/…/GrpcInterceptor.java`) is a test-only interceptor that only sets an `EndpointContext` label. The `ThrottleManagerImpl` / `ThrottleConfiguration` exist exclusively in the `web3` module and are never applied to gRPC endpoints.

**Root cause:** The server enforces 5 concurrent calls *per connection* but places no ceiling on the number of connections, no per-IP limit, and no application-level rate limit. An attacker multiplies connections to multiply concurrent DB-holding subscriptions.

### Impact Explanation

Each `getNodes()` subscription holds a HikariCP connection for the entire multi-page retrieval window (N pages × 250 ms delay each). With a typical pool of 10–20 connections, an attacker opening ~4 connections and sending 5 concurrent `getNodes(limit=0)` calls per connection (20 total) saturates the pool. Once the pool is exhausted, every subsequent gRPC call — including `subscribeTopic` and `getFeeEstimate` — blocks waiting for a connection and eventually times out, producing a complete denial of service of the gRPC API. The Prometheus alert `GrpcHighDatabaseConnections` (threshold 75%) confirms the pool is a recognized bottleneck.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond a standard gRPC client (e.g., `grpcurl`), and no privileged network position. The endpoint is publicly documented and reachable on port 5600. A single attacker machine can open hundreds of TCP connections and issue the flood. The `maxConcurrentCallsPerConnection = 5` guard is trivially bypassed by opening more connections. The optional GCP `maxRatePerEndpoint: 250` backend policy is infrastructure-level, not enforced in the application, and absent in non-GCP deployments.

### Recommendation

1. **Add a global concurrent-call limit** in `GrpcConfiguration.java` via `NettyServerBuilder.maxConcurrentCallsPerConnection` combined with a total connection cap (`maxConnectionAge`, `maxConnectionIdle`, or a custom `ServerInterceptor` tracking active calls globally).
2. **Add application-level rate limiting** in the gRPC module analogous to `ThrottleManagerImpl` in `web3` — a `GlobalServerInterceptor` using Bucket4j that rejects calls exceeding a configurable RPS threshold.
3. **Enforce a minimum non-zero `limit`** or cap the maximum page count per request so that `limit=0` does not translate to `Long.MAX_VALUE` items, bounding the per-request DB hold time.
4. **Set `maxConnectionAge` / `maxConnectionIdle`** on the Netty server builder to recycle connections and prevent indefinite connection accumulation from a single source IP.

### Proof of Concept

```bash
# Open 10 parallel connections, each sending 5 concurrent getNodes(limit=0) calls
# (50 total concurrent DB-holding subscriptions)
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done
wait

# Observe: subsequent legitimate calls hang or return UNAVAILABLE
grpcurl -plaintext \
  -d '{"file_id": {"fileNum": 102}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.NetworkService/getNodes
# Expected result: timeout or RESOURCE_EXHAUSTED / UNAVAILABLE
```
