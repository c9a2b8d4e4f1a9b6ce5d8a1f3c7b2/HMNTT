### Title
Unauthenticated Full Key-Range Scan Amplification via Concurrent Multi-OwnerID Requests on Unthrottled Hook Storage Endpoint

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint is publicly accessible with no authentication and no rate limiting in the `rest-java` module. When no `key` parameter is supplied, the controller defaults to a full key range (`0x00...00` to `0xFF...FF`), causing `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` to execute a full-range index scan per request. Because `hook_storage` is distributed by `owner_id` in Citus, concurrent requests with distinct valid `ownerId` values fan out to separate shards simultaneously, with no server-side mechanism to bound aggregate query concurrency.

### Finding Description

**Code path:**

1. `HooksController.java` lines 104–130: `getHookStorage()` accepts `ownerId` as a path variable with no authentication guard. When `keys` is empty (the default), `lowerBound = MIN_KEY_BYTES` (32×`0x00`) and `upperBound = MAX_KEY_BYTES` (32×`0xFF`) are set at lines 167–168.

2. `HookServiceImpl.java` lines 65–69: the empty-keys branch unconditionally calls `hookStorageRepository.findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(ownerId, hookId, keyLowerBound, keyUpperBound, page)`.

3. `HookStorageRepository.java` lines 16–17: Spring Data JPA generates:
   ```sql
   SELECT * FROM hook_storage
   WHERE owner_id = ? AND hook_id = ? AND key BETWEEN '\x00...' AND '\xff...'
     AND deleted = false
   ORDER BY key ASC LIMIT ?
   ```
   `deleted` is not part of the primary key `(owner_id, hook_id, key)`, so the engine must scan the full key range for the given `(owner_id, hook_id)` pair and filter `deleted = false` row-by-row until `LIMIT` rows are satisfied.

4. **Database schema** (`V2.17.1__add_hooks_support.sql` line 70): `hook_storage` is a Citus distributed table sharded by `owner_id`. Each distinct `ownerId` routes to a different worker shard, so N concurrent requests with N distinct ownerIds produce N simultaneous shard-level scans.

**Root cause:** The `rest-java` module contains zero rate-limiting infrastructure. The `ThrottleConfiguration` / `ThrottleManagerImpl` exist exclusively in the `web3` module and are never applied to `HooksController`. The only guards present are `@Max(MAX_LIMIT)` (capping rows returned at 100) and `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on key parameters — neither limits request concurrency or per-IP query rate.

### Impact Explanation
An attacker can drive sustained parallel full-range scans across all Citus worker nodes simultaneously. Each scan must traverse the entire `(owner_id, hook_id)` key space in the index, applying the `deleted = false` predicate row-by-row. With a large `hook_storage` dataset and many concurrent requests, aggregate CPU across worker nodes rises proportionally to concurrency. Because the endpoint is unauthenticated and the default request requires zero parameters beyond a valid entity ID, the attack surface is maximally broad.

### Likelihood Explanation
Exploitation requires only an HTTP client and a list of valid entity IDs (trivially enumerable from the public `/api/v1/accounts` endpoint). No credentials, tokens, or privileged access are needed. The attack is fully repeatable, scriptable, and can be sustained indefinitely. The absence of any rate limiting or connection throttling in `rest-java` means there is no server-side defense to overcome.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a per-IP (or global) request-rate bucket for the hooks storage endpoint, analogous to `ThrottleConfiguration` in `web3`.
2. **Add a partial index on `deleted`**: Create a partial index `WHERE deleted = false` on `hook_storage` to avoid full-range scans when filtering deleted rows.
3. **Require a non-trivial key range**: Reject or warn on requests where `fromKey = MIN_KEY_BYTES AND toKey = MAX_KEY_BYTES` without an explicit key parameter, or enforce a tighter default limit for full-range queries.
4. **Connection/concurrency limits**: Apply servlet-level or infrastructure-level (e.g., nginx) per-IP concurrency limits in front of `rest-java`.

### Proof of Concept

```bash
# 1. Enumerate valid ownerIds from the public accounts endpoint
# 2. Fire concurrent full-range storage scans with no key filter
for ownerId in 1000 1001 1002 1003 ... 1999; do
  curl -s "http://<mirror-node>/api/v1/accounts/${ownerId}/hooks/1/storage" &
done
wait
```

Each request triggers `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` with `fromKey=0x00...00`, `toKey=0xFF...FF` on a distinct Citus shard. With sufficient concurrency (e.g., 100–500 parallel requests), aggregate worker-node CPU rises well above the 30% threshold relative to baseline, with no server-side mechanism to reject or throttle the flood.
