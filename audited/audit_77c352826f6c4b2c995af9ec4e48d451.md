### Title
Unbounded Cache Pollution via Unauthenticated contractId Injection into `contractSlotsCache`

### Summary
Any unauthenticated caller can issue `eth_call` requests targeting 3000+ unique contract addresses, filling the `contractSlotsCache` (maximumSize=3000) with attacker-controlled entries. Once full, each new attacker-supplied contractId evicts the LRU legitimate contract's per-contract slot cache, destroying the batch-query optimization for that contract and forcing all subsequent slot reads (after the 2-second `contractStateCache` TTL) to fall back to individual DB queries. The attack is repeatable indefinitely with no special privileges.

### Finding Description

**Cache architecture:**

- `contractSlotsCache` (`CACHE_MANAGER_CONTRACT_SLOTS`): `expireAfterAccess=5m, maximumSize=3000`. Keys = `EntityId` (contractId). Values = `CaffeineCache` instances (per-contract slot key registries).
- `slotsPerContract` (`CACHE_MANAGER_SLOTS_PER_CONTRACT`): `expireAfterAccess=5m, maximumSize=1500`. Each value in `contractSlotsCache` is one of these caches, tracking which slot keys have been seen for that contract.
- `contractStateCache` (`CACHE_MANAGER_CONTRACT_STATE`): `expireAfterWrite=2s, maximumSize=25000`. Stores actual slot values; expires quickly.

**Vulnerable code path:**

In `ContractStateServiceImpl.findStorageBatch()`:

```java
// Line 86-87: unconditionally creates a new cache entry for any contractId
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

`Cache.get(key, valueLoader)` inserts a new entry if the key is absent — with zero validation that `contractId` is a real, deployed contract. There is no per-caller quota, no contractId allowlist, and no guard preventing arbitrary population of this cache.

**Exploit flow:**

1. Attacker sends 3000 `eth_call` (or `eth_estimateGas`) requests, each targeting a distinct contract address (real or synthetic). Each request reaches `findStorage` → `findStorageBatch` → line 86-87, inserting one entry into `contractSlotsCache`.
2. Cache is now at capacity (3000 entries).
3. Attacker sends request #3001 with a new contractId. Caffeine LRU-evicts the oldest legitimate contract's entry, discarding its entire per-contract slot key registry.
4. After the 2-second `contractStateCache` TTL, any `findStorage` call for the evicted contract misses `contractStateCache` (line 63), falls into `findStorageBatch`, and finds a brand-new empty per-contract slot cache (line 86-87 re-creates it). Only the single requested slot key is present; the batch query at line 103 fetches only that one slot instead of all previously seen slots.
5. Attacker repeats step 3 continuously, cycling through a pool of 3001+ contractIds to keep evicting legitimate contracts.

**Why existing checks are insufficient:**

- `isEnableBatchContractSlotCaching()` (line 59) is a global on/off flag; it does not restrict which contractIds can populate the cache.
- The global throttle (`requestsPerSecond=500` default) allows filling 3000 slots in ~6 seconds and does not prevent sustained cycling.
- There is no per-IP, per-caller, or per-contractId rate limit on cache insertion.

### Impact Explanation

The batch optimization in `findStorageBatch` is designed to amortize DB round-trips: once a contract's slot keys are registered, a single `findStorageBatch` DB query re-fetches all of them together after `contractStateCache` expires. Evicting a contract's slot cache forces N individual DB queries (one per slot) instead of one batch query for a contract with N active slots. Under sustained attack, all high-traffic contracts can be kept permanently evicted, multiplying DB query volume proportionally to the number of slots per contract and the request rate. This degrades response latency and increases DB load for all legitimate users of the mirror node web3 API.

### Likelihood Explanation

The web3 JSON-RPC endpoint is publicly accessible with no authentication. The attacker needs only HTTP access and knowledge of 3001 contract addresses (trivially enumerable from the mirror node's own REST API or from on-chain data). The attack requires no gas, no tokens, and no on-chain transactions. At the default 500 RPS throttle, the cache can be saturated in ~6 seconds and kept saturated indefinitely by cycling through a pool of contractIds at a low sustained rate (well below the throttle ceiling). The attack is fully scriptable and repeatable.

### Recommendation

1. **Validate contractId before cache insertion**: Before calling `cacheManagerSlotsPerContract.getCache(...)` in `findStorageBatch`, verify that the contractId corresponds to an existing contract entity (e.g., via a lightweight existence check against the entity repository or a separate bounded allowlist cache). Reject or skip cache population for unknown contractIds.
2. **Bound cache insertion rate per caller**: Apply per-IP or per-caller rate limiting specifically on the cache-insertion path, separate from the global RPS throttle.
3. **Use a size-bounded map of weak references**: Store per-contract slot caches as weak or soft references so that memory pressure naturally reclaims them without the LRU eviction being attacker-controllable.
4. **Reduce `contractSlotsCache` TTL or size**: Lowering `maximumSize` reduces the attack surface but does not eliminate it; the root fix is input validation.

### Proof of Concept

```python
import requests, json

MIRROR_NODE_URL = "http://<mirror-node-host>:8545"

def eth_call(contract_address):
    payload = {
        "jsonrpc": "2.0",
        "method": "eth_call",
        "params": [{"to": contract_address, "data": "0x"}, "latest"],
        "id": 1
    }
    requests.post(MIRROR_NODE_URL, json=payload)

# Step 1: Fill contractSlotsCache with 3000 unique contractIds
for i in range(1, 3001):
    # Hedera contract address format: 0x + zero-padded entity number
    addr = "0x" + hex(i)[2:].zfill(40)
    eth_call(addr)

# Step 2: Continuously evict legitimate contracts by cycling new contractIds
import time
victim_contract = "0x" + "deadbeef".zfill(40)  # a real high-traffic contract

# First, seed the victim into the cache
eth_call(victim_contract)

# Step 3: Evict victim by inserting one more unique contractId
evict_addr = "0x" + hex(3001)[2:].zfill(40)
eth_call(evict_addr)

# Step 4: Wait for contractStateCache to expire (2s), then read victim slot
time.sleep(3)
# This read now misses contractSlotsCache, re-creates an empty slot registry,
# and executes a single-slot DB query instead of a batch query.
eth_call(victim_contract)

# Step 5: Repeat steps 3-4 indefinitely to sustain the degradation
```

**Reproducible result**: After step 4, `findStorageBatch` for `victim_contract` executes `contractStateRepository.findStorageBatch` with a list containing only the single requested slot key, instead of all previously accumulated slot keys — confirmed by the logic at lines 86-103 of `ContractStateServiceImpl.java` where a freshly created (empty) per-contract cache produces a single-element `cachedSlots` list.
