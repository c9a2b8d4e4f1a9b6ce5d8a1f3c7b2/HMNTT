### Title
Cache-Eviction Griefing via Batch Amplification in `findStorageBatch()` — Unbounded Per-User Writes to Shared `contractStateCache`

### Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally writes all per-contract cached slot keys back into the global `contractStateCache` (maximumSize=25000, expireAfterWrite=2s) on every cache miss, with no per-user or per-IP quota on cache entries. An unprivileged attacker who pre-populates the per-contract slots caches (slotsPerContract, maximumSize=1500) for ~17 contracts can then trigger batch writes of ~1500 entries per request, filling all 25,000 slots in the global cache with just 17 requests every 2 seconds, continuously evicting legitimate users' cached slot values and forcing every subsequent legitimate `findStorage()` call to re-execute batch DB queries.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 63) checks `contractStateCache` for the key. On a miss (or `EMPTY_VALUE`), it calls `findStorageBatch()` (line 69).

Inside `findStorageBatch()` (lines 85–122):

```
// Line 90: unconditionally registers the new key in the per-contract slots cache
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);

// Line 91: reads ALL currently cached slot keys for this contract
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

// Line 103: batch-queries ALL of them from the DB
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);

// Lines 106–114: writes ALL results into the global contractStateCache
contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
```

**Root cause:** There is no bound on how many entries a single caller can inject into `contractStateCache` per request. Each call to `findStorageBatch()` for a contract with N slots in its per-contract slots cache writes N entries into the global shared cache. The per-contract slots cache (`slotsPerContract`, maximumSize=1500) acts as an amplifier: one HTTP request → up to 1500 `contractStateCache.put()` calls.

**Exploit flow:**

1. **Setup phase** — Attacker sends 1500 requests for each of 17 distinct contracts (25,500 total), each with a different existing slot key. This populates the `slotsPerContract` cache for each contract to its maximum of 1500 entries. The `contractSlots` cache (maximumSize=3000) holds the per-contract cache references. Slot keys are publicly enumerable from the Hedera mirror node REST API or blockchain explorers.

2. **Attack phase** — After the 2-second `contractStateCache` TTL expires, the attacker sends exactly 17 requests (one per pre-loaded contract). Each request triggers `findStorageBatch()`, which reads all 1500 cached slot keys and writes 1500 entries into `contractStateCache`. After 17 requests, all 25,500 slots in `contractStateCache` are filled with attacker-controlled entries.

3. **Sustain phase** — The attacker repeats step 2 every 2 seconds. Only 17 requests per 2-second window are needed to maintain full cache saturation — a negligible fraction of the 500 req/s global rate limit.

**Why existing checks fail:**

- **Rate limiting** (`ThrottleManagerImpl`, `rateLimitBucket`, 500 req/s global): The maintenance phase requires only 17 requests per 2 seconds (8.5 req/s), well under the 500 req/s global limit. The rate limit is a single shared bucket, not per-IP, so the attacker does not need to bypass it.
- **`maximumSize=25000`**: This is the attack surface, not a defense. Caffeine's W-TinyLFU eviction policy will evict the least-recently-used entries when the cache is full, which are legitimate users' entries that were written before the attacker's flood.
- **`expireAfterWrite=2s`**: The attacker re-floods every 2 seconds, keeping the cache perpetually saturated with attacker entries.
- **No per-user/per-IP cache quota**: `contractStateCache` is a single global `CaffeineCache` instance with no partitioning by caller identity.

### Impact Explanation

Every legitimate `findStorage()` call that misses the cache (because its entry was evicted) falls through to `findStorageBatch()`, which issues a DB query (`findStorageBatch` or `findStorage` on `ContractStateRepository`). Under sustained attack, all 25,000 cache slots are occupied by attacker entries, so every legitimate request incurs a DB round-trip. This increases database load proportionally to legitimate traffic, degrades response latency for all `eth_call`/`eth_estimateGas` users, and can cascade into DB connection pool exhaustion under high legitimate load. No funds are at risk; the impact is availability degradation (griefing).

### Likelihood Explanation

The attack requires no credentials, no on-chain transactions, and no special tooling — only the ability to send HTTP requests to the public web3 API. Existing slot keys for popular contracts (e.g., large DeFi contracts with thousands of storage slots) are trivially enumerable from the mirror node's own REST API. The maintenance cost (17 req/2s) is negligible. The attack is fully repeatable and can be sustained indefinitely from a single IP without triggering the global rate limit. A moderately sophisticated attacker can automate this with a simple script.

### Recommendation

1. **Bound per-contract contributions to `contractStateCache`**: Before writing batch results at line 109, check whether the number of entries already in `contractStateCache` attributed to this `contractId` exceeds a per-contract quota (e.g., `slotsPerContract.maximumSize`). Alternatively, namespace `contractStateCache` by `contractId` and apply a per-contract `maximumSize`.

2. **Decouple the per-contract slots cache from the global state cache**: The batch amplification arises because `findStorageBatch()` writes all N cached slot keys back into the global cache on every miss. Consider writing only the specifically requested key's result into `contractStateCache` immediately, and lazily populating other slots only when they are individually requested.

3. **Add per-IP or per-caller rate limiting on cache-filling paths**: The global `rateLimitBucket` does not prevent a single caller from monopolizing cache space. A per-IP token bucket for cache-write operations would limit the attacker's ability to fill the cache.

4. **Consider a shorter `slotsPerContract` TTL or smaller `maximumSize`**: Reducing `slotsPerContract.maximumSize` from 1500 reduces the amplification factor per request.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }
