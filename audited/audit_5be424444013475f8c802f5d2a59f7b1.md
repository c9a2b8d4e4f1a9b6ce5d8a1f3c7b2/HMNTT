### Title
Race Condition Between `computeIfAbsent` and `topicMessages.remove()` Causes Orphaned Redis Subscriptions Leading to Memory/Connection Exhaustion

### Summary
`RedisTopicListener` uses a `ConcurrentHashMap` to track one shared `Flux` per topic, but the cleanup path (`unsubscribe` → `remove`) and the allocation path (`getSharedListener` → `computeIfAbsent`) are not atomically coordinated. An unprivileged external gRPC client can exploit the resulting race window to repeatedly create orphaned Redis subscriptions that accumulate indefinitely, exhausting the Redis connection pool and JVM heap memory.

### Finding Description

**Exact code locations:**
- `getSharedListener()` — [1](#0-0) 
- `subscribe()` — [2](#0-1) 
- `unsubscribe()` — [3](#0-2) 

**Root cause:**

`topicMessages` is a `ConcurrentHashMap<String, Flux<TopicMessage>>`. [4](#0-3) 

`ConcurrentHashMap` guarantees per-operation atomicity, but the compound check-then-act sequence spanning `computeIfAbsent` (in `getSharedListener`) and `remove` (in `unsubscribe`) is **not** atomic. The `.share()` operator on the returned Flux uses Reactor's `publish().refCount()` semantics: when the subscriber count drops to 0, the upstream is cancelled and `doOnCancel` fires; when count rises from 0 to 1, a **new** upstream subscription is created. [5](#0-4) 

**Race window and exploit flow:**

```
Thread A (last subscriber cancels for "topic.X"):
  1. .share() detects refcount → 0, cancels upstream
  2. doOnCancel fires → unsubscribe("topic.X") is called
  3. [RACE WINDOW OPENS]
  4. topicMessages.remove("topic.X") executes

Thread B (new subscriber arrives for "topic.X"):
  [executes between steps 2 and 4 above]
  1. computeIfAbsent("topic.X", ...) → entry STILL present → returns OLD shared Flux
  2. New subscriber subscribes → .share() refcount 0→1 → NEW upstream Redis subscription created
  3. Thread A's remove() fires → OLD shared Flux evicted from map
  4. OLD shared Flux is now ORPHANED: has 1 active subscriber + active Redis connection, but is unreachable via the map
  5. Future subscribers for "topic.X" call computeIfAbsent → entry absent → create ANOTHER new Flux → ANOTHER Redis subscription
```

Each successful race leaves one orphaned `Flux` with a live Redis subscription that can never be cleaned up (its `doOnCancel`/`doOnComplete` will call `remove()` on a key that no longer maps to it, or that has already been replaced). [3](#0-2) 

**Why existing checks are insufficient:**

`ConcurrentHashMap` only prevents data corruption within a single operation. It provides no mutual exclusion across the `computeIfAbsent` → `remove` boundary. There is no lock, no `compute`-based atomic swap, and no version/generation counter to detect stale removals. [6](#0-5) 

### Impact Explanation

Each successful race iteration leaks one `ReactiveRedisMessageListenerContainer` subscription. Redis pub/sub connections are persistent and stateful; they are not returned to a pool on GC. Accumulating N orphaned subscriptions:
1. Exhausts the Redis connection pool, causing new legitimate subscriptions to block or fail.
2. Each orphaned `Flux` holds an `onBackpressureBuffer` with `maxBufferSize` slots — heap memory grows linearly with orphan count. [7](#0-6) 
3. The service becomes unavailable to all subscribers (DoS) without any authentication requirement.

### Likelihood Explanation

An unprivileged external user only needs the ability to open gRPC streaming connections to `subscribeTopic`, which is the public API. [8](#0-7) 

The race window is narrow (nanoseconds between `doOnCancel` firing and `remove()` executing), but:
- A single attacker script opening hundreds of concurrent connections to the same topic and cycling them rapidly will hit the window within seconds on any multi-core JVM.
- The attack is fully repeatable and requires no special knowledge beyond the topic ID.
- No rate limiting, connection cap, or subscription quota is present in the listener path.

### Recommendation

Replace the non-atomic `computeIfAbsent` / `remove` pair with a single atomic `compute` call that validates the entry being removed is still the same object (identity check), preventing a stale removal from evicting a freshly inserted entry:

```java
// In unsubscribe():
private void unsubscribe(Topic topic, Flux<TopicMessage> expected) {
    topicMessages.remove(topic.getTopic(), expected); // atomic conditional remove
    log.info("Unsubscribing from {}", topic);
}
```

Pass the specific `Flux` instance into `doOnCancel`/`doOnComplete` so the `remove(key, value)` overload is used — this is an atomic compare-and-remove that only removes the entry if the value is still the same object reference, closing the race window entirely.

Additionally, consider replacing the `ConcurrentHashMap` + `computeIfAbsent` pattern with a `compute`-based approach that atomically checks-and-inserts, or use a dedicated lock per topic key.

### Proof of Concept

```
Preconditions:
- Mirror node gRPC service running with Redis listener enabled
- One topic ID known (e.g., topic 1)

Steps:
1. Open 2 gRPC streaming connections (C1, C2) to subscribeTopic for topic 1.
   → Both share the same Flux entry in topicMessages["topic.1"].

2. Cancel C1 (the "last-but-one" subscriber).
   → C2 is still active; shared Flux stays alive. (Warm-up step.)

3. In a tight loop, repeat:
   a. Open a new connection C_new to topic 1 on Thread B.
   b. Simultaneously cancel C2 (the last subscriber) on Thread A.
   c. Thread A: doOnCancel fires → unsubscribe() called.
   d. Thread B: computeIfAbsent executes in the race window before remove().
   e. Thread B gets the OLD Flux; .share() creates a new upstream Redis subscription.
   f. Thread A's remove() evicts the entry → OLD Flux is now orphaned.
   g. Set C2 = C_new; repeat from (a).

4. After N iterations, observe:
   - Redis `CLIENT LIST` shows N+1 pub/sub connections for the same channel.
   - JVM heap grows proportionally.
   - Eventually Redis connection limit is hit; new subscriptions return errors.
   - Service is unavailable (DoS achieved without any credentials).
```

    protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
        Topic topic = getTopic(filter);
        return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
    }

    private Topic getTopic(TopicMessageFilter filter) {
        return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
    }

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }

    private void unsubscribe(Topic topic) {
        topicMessages.remove(topic.getTopic());
        log.info("Unsubscribing from {}", topic);
    }
