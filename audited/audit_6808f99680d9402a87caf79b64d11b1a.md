Looking at the `keyMapper` function and `cache.get` implementation carefully:

### Title
Cache Key Collision in `doGetTransactions`: `payer_account_id` Omitted from Redis Key Causes Cross-Payer Transaction Data Misattribution

### Summary
The `keyMapper` function in `rest/transactions.js` generates Redis cache keys using only `consensus_timestamp`, discarding `payer_account_id`. Because the `transaction` table's primary key is the composite `(consensus_timestamp, payer_account_id)`, two distinct transactions can share the same `consensus_timestamp` with different payers. When this occurs, the first payer's transaction data is stored under `transaction:{T}` and subsequently served to any request for the second payer's transaction at the same timestamp, misattributing transaction details in the history.

**Root cause â€” failed assumption:** The developer assumed `consensus_timestamp` alone uniquely identifies a transaction. It does not. The actual database primary key is the composite `(consensus_timestamp, payer_account_id)`:

```sql
alter table if exists transaction
    add constraint transaction__pk primary key (consensus_timestamp, payer_account_id);
``` [3](#0-2) 

Child/inner transactions (e.g., those spawned by smart contracts or batch operations) share the parent's `consensus_timestamp` but carry a different `payer_account_id` and `nonce`. Both rows are valid, distinct DB records at the same timestamp.

**Cache collision mechanics** (`cache.js` lines 95â€“139): [4](#0-3) 

When `payerAndTimestamps` contains `[{consensus_timestamp: T, payer_account_id: A}, {consensus_timestamp: T, payer_account_id: B}]`:
1. `mgetBuffer(["transaction:T", "transaction:T"])` â€” both miss (same key queried twice).
2. `loader(missingKeys)` correctly fetches both rows from DB using both fields.
3. The fill loop stores `transaction:T â†’ txA` then `transaction:T â†’ txB` via `mset` â€” last write wins, so `transaction:T` ends up holding `txB`.
4. On the next request for payer A at timestamp T: `mgetBuffer(["transaction:T"])` returns `txB` â€” cache hit, wrong data served.

**Why existing checks are insufficient:** The DB query in `getTransactionsDetails` correctly filters by both `payer_account_id` and `consensus_timestamp`: [5](#0-4) 
But this correctness is entirely bypassed on a cache hit. The cache layer has no awareness of `payer_account_id`.

### Impact Explanation
Any user querying `/api/v1/transactions` with an `account.id` filter that resolves to a timestamp shared with another payer's child transaction will receive the wrong transaction's details (wrong `transaction_id`, wrong `transfers`, wrong `token_transfers`, wrong fees, wrong result). This constitutes cross-user transaction history misattribution â€” a data integrity and information disclosure issue. Severity is medium: no funds are moved, but financial records shown to users are incorrect and can include another account's private transfer details.

### Likelihood Explanation
Child transactions (nonce > 0) sharing a parent's `consensus_timestamp` with a different `payer_account_id` are a normal, frequent occurrence on Hedera mainnet (smart contract calls, scheduled transactions, batch transactions). No privileges are required â€” the `/transactions` endpoint is fully public. An attacker only needs to identify such a timestamp pair (observable from the public ledger) and issue two sequential requests. The race window is the Redis TTL of the first cached entry. Fully repeatable and automatable.

### Recommendation
Include `payer_account_id` in the cache key. Replace the `keyMapper` with:

```js
const keyMapper = (key) => {
  let timestamp = key.consensus_timestamp;
  if (typeof timestamp === 'string') {
    timestamp = timestamp.replace('.', '');
  }
  return `transaction:${key.payer_account_id}:${timestamp}`;
};
```

This makes the cache key match the actual composite primary key of the `transaction` table, eliminating the collision.

### Proof of Concept

**Preconditions:**
- Redis cache is enabled and warm.
- Timestamp `T` exists in the DB with two rows: `(payer_account_id=A, consensus_timestamp=T, nonce=0)` and `(payer_account_id=B, consensus_timestamp=T, nonce=1)` (parent + child transaction).

**Steps:**
1. `GET /api/v1/transactions?account.id=B&timestamp=T` â€” cache miss; DB returns txB; stored as `transaction:T â†’ txB` in Redis.
2. `GET /api/v1/transactions?account.id=A&timestamp=T` â€” cache hit on `transaction:T`; returns txB's data (wrong payer, wrong transfers, wrong fees) to the requester expecting payer A's record.

**Result:** Payer A's transaction history at timestamp T shows payer B's transaction details.

  async get(keys, loader, keyMapper = (k) => (k ? k.toString() : k)) {
    if (isEmpty(keys)) {
      return [];
    }
    if (!this.ready) {
      return loader(keys);
    }

    const buffers =
      (await this.redis
        .mgetBuffer(map(keys, keyMapper))
        .catch((err) => logger.warn(`Redis error during mget: ${err.message}`))) || new Array(keys.length);
    const values = buffers.map((t) => JSONParse(t));

    let i = 0;
    const missingKeys = keys.filter(() => isNil(values[i++]));

    if (missingKeys.length > 0) {
      const missing = await loader(missingKeys);
      const newValues = [];
      let j = 0;

      missing.forEach((value) => {
        // Update missing values in Redis array
        for (; j < values.length; j++) {
          if (isNil(values[j])) {
            values[j] = value;
            newValues.push(keyMapper(keys[j]));
            newValues.push(JSONStringify(value));
            break;
          }
        }
      });

      if (newValues.length > 0) {
        this.redis.mset(newValues).catch((err) => logger.warn(`Redis error during mset: ${err.message}`));
      }
    }

    if (logger.isDebugEnabled()) {
      const count = keys.length - missingKeys.length;
      logger.debug(`Redis returned ${count} of ${keys.length} keys`);
    }

    return values;
