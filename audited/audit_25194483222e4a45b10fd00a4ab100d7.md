### Title
Throttle Token Leak via Uncaught `IllegalArgumentException` from `hexToBytes` in `ContractController.call()`

### Summary
In `ContractController.call()`, after `throttleManager.throttle(request)` consumes gas tokens, `constructServiceParameters()` calls `hexToBytes(data)` outside its try-catch block. `hexToBytes` throws `IllegalArgumentException` on invalid hex input, which is not caught by the `catch (InvalidParametersException e)` handler in `call()`. As a result, `throttleManager.restore(request.getGas())` is never invoked, permanently leaking throttle tokens. Any unauthenticated user can exploit this to drain the gas-limit bucket and deny service to legitimate callers.

### Finding Description

**Exact code path:**

`ContractController.java`, `call()` (lines 38–51):
```java
try {
    throttleManager.throttle(request);          // line 40 — tokens consumed
    validateContractMaxGasLimit(request);
    final var params = constructServiceParameters(request);  // line 43
    ...
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());  // line 48 — ONLY reached for InvalidParametersException
    throw e;
}
```

`ContractController.java`, `constructServiceParameters()` (lines 66–80):
```java
String data;
try {
    data = request.getData() != null ? request.getData() : HEX_PREFIX;  // line 68 — getter, NEVER throws
} catch (final Exception e) {
    throw new InvalidParametersException(...);  // line 70 — dead code path
}
// hexToBytes is called OUTSIDE the try-catch:
return ContractExecutionParameters.builder()
    ...
    .callData(hexToBytes(data))   // line 80 — CAN throw IllegalArgumentException
    ...
```

`BytesDecoder.java`, `hexToBytes()` (lines 79–89):
```java
public static byte[] hexToBytes(final String hexString) {
    ...
    try {
        return org.apache.commons.codec.binary.Hex.decodeHex(hex);
    } catch (DecoderException e) {
        throw new IllegalArgumentException(e.getMessage());  // line 87
    }
}
```

**Root cause:** The try-catch in `constructServiceParameters()` wraps only `request.getData()`, a plain getter that cannot throw. The actual conversion `hexToBytes(data)` at line 80 sits outside that block and throws `IllegalArgumentException` for any malformed hex string (odd length, non-hex characters). `IllegalArgumentException` is not a subtype of `InvalidParametersException`, so the catch in `call()` does not intercept it, and `throttleManager.restore()` is never called.

### Impact Explanation
The gas-limit bucket (`gasLimitBucket`) is a finite token reservoir managed by Bucket4j. Each call to `throttleManager.throttle(request)` deducts `scaleGas(request.getGas())` tokens. When the bucket empties, all subsequent legitimate requests receive `ThrottleException("Gas per second rate limit exceeded.")`. By repeatedly submitting requests with the maximum allowed gas value and an invalid `data` field, an attacker can drain the bucket continuously, causing a sustained denial-of-service against all users of the `/api/v1/contracts/call` endpoint without ever performing any real EVM computation.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The trigger requires only a single crafted HTTP POST with a syntactically invalid hex string in the `data` field (e.g., `"0xGG"` or `"0x1"` — odd-length). No special privileges, accounts, or on-chain state are needed. The attack is trivially scriptable, repeatable at high frequency, and requires no knowledge of the system beyond the public API schema.

### Recommendation
Move `hexToBytes(data)` inside the existing try-catch in `constructServiceParameters()` so that a `DecoderException`/`IllegalArgumentException` is translated into an `InvalidParametersException`:

```java
String data;
byte[] callData;
try {
    data = request.getData() != null ? request.getData() : HEX_PREFIX;
    callData = hexToBytes(data);   // moved inside the try block
} catch (final Exception e) {
    throw new InvalidParametersException(
        "data field '%s' contains invalid hex characters".formatted(request.getData()));
}
```

This ensures the exception type is `InvalidParametersException`, which is caught in `call()`, causing `throttleManager.restore(request.getGas())` to execute and return the consumed tokens.

### Proof of Concept

**Preconditions:** None. The endpoint is unauthenticated.

**Steps:**

1. Identify the gas-per-second limit (default or configured value).
2. Send repeated POST requests to `/api/v1/contracts/call` with maximum gas and invalid hex data:
```http
POST /api/v1/contracts/call
Content-Type: application/json

{
  "data": "0xGGGGGGGG",
  "gas": 15000000,
  "to": "0x0000000000000000000000000000000000000001"
}
```
3. Each request: (a) passes `@Valid` bean validation (the `data` field passes the `HexValidator` pattern check for `0x`-prefixed strings with non-hex chars depending on validator config, or use odd-length `"0x1"`), (b) consumes up to `scaleGas(15000000)` tokens from `gasLimitBucket`, (c) throws `IllegalArgumentException` from `hexToBytes`, (d) bypasses the `catch (InvalidParametersException)` block, (e) never restores tokens.
4. After enough requests, `gasLimitBucket` is exhausted; all legitimate callers receive HTTP 429 `Gas per second rate limit exceeded.`

**Verification:** Confirm by observing that the bucket token count monotonically decreases with each malformed request and is never replenished, while a well-formed request with the same gas value would trigger restore after a validation failure.
