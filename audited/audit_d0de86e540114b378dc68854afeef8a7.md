### Title
Timing Side-Channel in `authenticate()` via `&&` Short-Circuit Enables Two-Phase Username Enumeration on Metrics Endpoint

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` chains two `tsscmp` calls with the `&&` operator. Because JavaScript's `&&` short-circuits, the password comparison is skipped entirely when the username is wrong, creating a measurable timing difference between an incorrect username and a correct username with an incorrect password. An attacker with network access to the metrics endpoint can exploit this difference statistically to enumerate the valid username, then brute-force the password in a second phase.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `authenticate()`, line 143.

```js
// line 137-144
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

`tsscmp` internally computes HMAC-SHA256 of both operands before comparing, making each individual call constant-time. However, the `&&` operator short-circuits:

| Scenario | Calls executed | Cost |
|---|---|---|
| Wrong username | `tsscmp(name, username)` only | 1× HMAC-SHA256 |
| Correct username + wrong password | `tsscmp(name, username)` + `tsscmp(pass, password)` | 2× HMAC-SHA256 |

The second `tsscmp` call is never executed when the username is wrong, so the total wall-clock time for the authentication check is measurably shorter for an incorrect username than for a correct username paired with a wrong password. This leaks a single bit of information per request: "was the username correct?"

No rate limiting exists on the metrics endpoint — the `metricsMiddleware` function applies no throttling before calling `authenticate()`.

**Root cause / failed assumption:** The developer correctly used `tsscmp` to protect each individual string comparison, but assumed that chaining them with `&&` preserves the overall timing-safe property. It does not, because `&&` introduces a control-flow branch that depends on the secrecy of `username`.

### Impact Explanation
A successful attacker gains authenticated access to the Prometheus metrics endpoint, which exposes internal operational data: per-route request counts, latency histograms, in-flight request counts, CPU and heap usage. While this is not direct financial data, it constitutes unauthorized access to a protected internal endpoint and leaks information about API usage patterns (including which endpoints are being queried and at what rate). The "staking rewards" framing in the question overstates the direct financial impact — the metrics payload does not contain reward balances or account data — but the authentication bypass itself is a real security boundary violation.

### Likelihood Explanation
The timing signal is small (one HMAC-SHA256 ≈ 1–10 µs on modern hardware) relative to typical network jitter (1–100 ms). However:
- With statistical aggregation over hundreds of requests per candidate username, the signal becomes detectable (this is the standard approach for remote timing attacks, as demonstrated in practice against real systems).
- No rate limiting or account lockout is present on the metrics endpoint.
- The attacker only needs to enumerate a single username (the one configured for metrics access), which is a small search space.
- Once the username is known, the password brute-force is a standard dictionary/credential-stuffing attack against a single known account.

The attack is feasible for a motivated attacker with low-latency network access (e.g., same datacenter or cloud region).

### Recommendation
Always evaluate both comparisons unconditionally, regardless of the result of the first:

```js
// Constant-time: both comparisons always run
const nameMatch = tsscmp(credentials.name, username);
const passMatch = tsscmp(credentials.pass, password);
return credentials && nameMatch && passMatch;
```

Assigning both `tsscmp` results to variables before combining them with `&&` forces both HMAC computations to complete before any branching occurs, eliminating the timing side-channel. Apply the same fix to `findUser()` in `rest/middleware/authHandler.js` line 12 for consistency.

### Proof of Concept
1. Identify the metrics endpoint path from config (default: `/swagger/metrics/`).
2. Send ~500 requests with a known-wrong username (e.g., `wronguser:anypass`) and record response times.
3. Send ~500 requests with a candidate username (e.g., `admin:anypass`) and record response times.
4. Compute the mean response time for each candidate. The correct username will show a statistically higher mean (by ~1–10 µs) due to the second `tsscmp` call executing.
5. Repeat step 3–4 across a list of common usernames or usernames derived from config leaks.
6. Once the username is confirmed, perform a standard dictionary attack against the password with the known username.
7. Authenticate to the metrics endpoint and retrieve the full Prometheus metrics payload. [1](#0-0)
