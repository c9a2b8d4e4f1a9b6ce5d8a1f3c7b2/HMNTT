### Title
Unauthenticated DB Connection Pool Exhaustion via Parallel Fan-Out in `getDetailedContractResults()`

### Summary
The `GET /contracts/results/:transactionIdOrHash` endpoint is publicly accessible with no rate limiting in the REST API layer. Each request triggers `getDetailedContractResults()`, which fans out 5 parallel database queries via `Promise.all()`. With a default connection pool of only 10 connections, as few as 2–3 concurrent requests fully saturate the pool, causing all subsequent requests to queue and time out, rendering the mirror node REST API unavailable.

### Finding Description
**Exact code path:**

`rest/routes/contractRoute.js` line 21 routes `GET /results/:transactionIdOrHash` to `getContractResultsByTransactionIdOrHash`. That handler (lines 1120–1203 of `rest/controllers/contractController.js`) calls `getDetailedContractResults()` at line 1166.

`getDetailedContractResults()` (lines 1205–1220) issues exactly 5 concurrent DB queries via `Promise.all()`:
- `ContractService.getContractResultsByTimestamps`
- `TransactionService.getEthTransactionByTimestampAndPayerId`
- `RecordFileService.getRecordFileBlockDetailsFromTimestamp`
- `ContractService.getContractLogsByTimestamps`
- `ContractService.getContractStateChangesByTimestamps`

The connection pool is configured in `rest/dbpool.js` (line 14) as `max: config.db.pool.maxConnections`. The documented default for `hiero.mirror.rest.db.pool.maxConnections` is **10** (`docs/configuration.md` line 556).

**Root cause:** No rate-limiting middleware exists in the Node.js REST API for this endpoint. The throttle mechanisms found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) are exclusively in the `web3` Java module and do not apply here. The REST API has no per-IP or global request rate limiter.

**Failed assumption:** The design assumes the connection pool (default: 10) is large enough relative to concurrent request volume. With 5 connections consumed per request, only 2 concurrent requests are needed to exhaust the pool. A third concurrent request causes all 5 of its `Promise.all` branches to queue, blocking for up to `connectionTimeoutMillis` (default: 20,000 ms) before failing.

### Impact Explanation
With the pool exhausted, all REST API endpoints sharing the same pool become unresponsive — not just the contract results endpoint. New requests queue until `connectionTimeoutMillis` (20 s) elapses, returning errors. This constitutes a full denial-of-service of the mirror node REST API. Operators relying on the mirror node for transaction data, contract results, or block queries are completely blocked. Because the attack requires only a handful of concurrent HTTP requests, it is trivially sustainable with minimal attacker resources.

### Likelihood Explanation
The endpoint requires only a valid Ethereum transaction hash, which is publicly observable on-chain. No credentials, tokens, or privileged access are needed. An attacker can obtain valid hashes from any block explorer. Sending 3–5 concurrent requests in a tight loop from a single machine is sufficient to maintain pool exhaustion indefinitely. The attack is repeatable, cheap, and requires no specialized knowledge beyond basic HTTP tooling.

### Recommendation
1. **Add rate limiting to the REST API**: Apply a per-IP (and global) rate limiter middleware (e.g., `express-rate-limit`) to all `/contracts/results/*` routes, specifically capping requests to `getContractResultsByTransactionIdOrHash`.
2. **Increase the default pool size**: Raise `hiero.mirror.rest.db.pool.maxConnections` to a value that reflects realistic concurrency (e.g., 50–100), and document the relationship between pool size and the 5× fan-out multiplier.
3. **Serialize or reduce the fan-out**: Consider whether all 5 queries must be parallel, or whether sequential execution with early-exit on missing data reduces peak connection consumption.
4. **Add a concurrency limiter**: Use a semaphore or queue (e.g., `p-limit`) inside `getDetailedContractResults()` to cap the number of in-flight DB connections per request.

### Proof of Concept
```bash
# Obtain a valid Ethereum transaction hash from any public mirror node or block explorer
HASH="0xabc123...valid_eth_tx_hash"
BASE_URL="https://<mirror-node-host>/api/v1"

# Send 5 concurrent requests — enough to exhaust the default pool of 10 connections
for i in $(seq 1 5); do
  curl -s "$BASE_URL/contracts/results/$HASH" &
done
wait

# All subsequent requests will now queue and time out (HTTP 500 or connection timeout)
curl -v "$BASE_URL/contracts/results/$HASH"
# Expected: hangs for ~20 seconds then returns an error
# All other REST API endpoints also become unresponsive during this window
```
