### Title
Unauthenticated Mirror Node Response Causes Non-Retryable DoS via Empty Nodes Array

### Summary
`fetchMirrorNodeNetwork()` makes an unauthenticated HTTP GET request to the operator-configured mirror node REST endpoint and blindly trusts the JSON response. An attacker with network-level access (DNS poisoning, ARP/BGP MITM, or a compromised network path) can return `{"nodes":[]}`, which causes `len(network) == 0` at line 127, triggering a non-retryable error that propagates through `buildNetworkFromMirrorNodes()` → `newClient()` → `log.Fatalf()`, permanently terminating the pinger process.

### Finding Description

**Exact code path:**

`fetchMirrorNodeNetwork()` at [1](#0-0)  performs an unauthenticated HTTP GET with no TLS enforcement, no certificate pinning, and no response signature verification. The parsed `payload.Nodes` slice is iterated to build the `network` map.

If the attacker returns `{"nodes":[]}` (or nodes with no valid `service_endpoints`), the loop at [2](#0-1)  produces an empty `network` map.

The check at [3](#0-2)  then returns `retry=false` with the error `"no usable service_endpoints found"`.

**Why retries don't help:**

In `buildNetworkFromMirrorNodes()`, the retry loop at [4](#0-3)  breaks immediately when `retry == false`. The empty-nodes path explicitly sets `retry=false`, so all configured `mirrorNodeClientMaxRetries` are bypassed — a single poisoned response is sufficient.

**Fatal propagation:**

`newClient()` returns the error directly at [5](#0-4) , and `main()` calls `log.Fatalf` at [6](#0-5) , which calls `os.Exit(1)` — the process terminates permanently.

**Root cause:** No authentication of the mirror node HTTP response. The example URL `http://mirror-rest:5551` uses plain HTTP, and the code imposes no HTTPS requirement anywhere in `config.go` or `mirror_node_client.go`. [7](#0-6) 

### Impact Explanation

The pinger process is completely terminated. Since `newClient()` is only called once at startup, the pinger never reaches the ticker loop and never submits transactions again until manually restarted. The `/tmp/ready` readiness file is never written, so Kubernetes readiness probes also fail. [8](#0-7)  This is a complete, persistent denial of service of the pinger's core function.

### Likelihood Explanation

The attack requires network-level interposition between the pinger and the mirror node REST endpoint. This is realistic in several scenarios:

- **Plain HTTP deployment** (as shown in the documented example `http://mirror-rest:5551`): any attacker on the same L2 segment (ARP poisoning) or controlling a router on the path can intercept and replace the response.
- **DNS poisoning**: if the mirror node URL uses a hostname, an attacker who can poison the DNS resolver used by the pinger pod can redirect the request to an attacker-controlled server returning `{"nodes":[]}`.
- **Kubernetes network policy gaps**: a compromised pod in the same namespace with no network policy restrictions can perform ARP spoofing against the mirror-rest service IP.

No credentials, no privileged OS access to the pinger host, and no knowledge of operator keys are required. The attack is repeatable on every restart.

### Recommendation

1. **Enforce HTTPS with certificate verification** for the mirror node REST URL. Reject `http://` URLs at config validation time in `loadConfig()`.
2. **Treat empty-nodes as a retryable error**, not a permanent failure. Change `return nil, false, fmt.Errorf(...)` at line 128 to `return nil, true, fmt.Errorf(...)` so all configured retries are exhausted before giving up.
3. **Enforce a minimum node count** (e.g., ≥ 1) and log a warning if the count drops significantly from a previously known value.
4. **Do not use `log.Fatalf` for transient network errors** at startup; instead, retry with backoff until the context is cancelled.

### Proof of Concept

1. Stand up a rogue HTTP server that always responds with `HTTP 200` and body `{"nodes":[],"links":{"next":null}}`.
2. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<rogue-server>`.
3. Start the pinger. Observe:
   - `fetchMirrorNodeNetwork()` decodes the empty nodes array.
   - `len(network) == 0` → returns `retry=false, "no usable service_endpoints found"`.
   - `buildNetworkFromMirrorNodes()` exits the retry loop on the first attempt.
   - `newClient()` returns the error.
   - `main()` logs `client error: attempt 1/11: no usable service_endpoints found from http://<rogue-server>/api/v1/network/nodes` and calls `os.Exit(1)`.
4. For a real MITM scenario: use `arpspoof` or a rogue DNS entry to redirect the legitimate mirror node hostname to the rogue server, then restart the pinger.
