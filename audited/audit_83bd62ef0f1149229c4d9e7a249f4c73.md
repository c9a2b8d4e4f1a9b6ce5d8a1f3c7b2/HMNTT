### Title
Unbounded DB Polling Leak on Network Partition Due to Missing gRPC Keepalive and No Flux Timeout

### Summary
`PollingTopicListener.listen()` creates an infinite polling flux (`RepeatSpec.times(Long.MAX_VALUE)`) with no application-level timeout or `takeUntilOther()` operator. The sole cleanup path is `serverCallStreamObserver.setOnCancelHandler(disposable::dispose)` in `ConsensusController`, which only fires when the gRPC framework itself detects client disconnection. Because the Netty server is configured with no keepalive probes, a silently-dropped TCP connection (network partition) is never detected, so the cancel handler never fires and the polling loop runs indefinitely against the database for every dead subscriber.

### Finding Description
**Code path:**

- `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–48: the flux repeats `Long.MAX_VALUE` times with a fixed delay; there is no `timeout()`, no `takeUntilOther()`, and no idle-detection operator.
- `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, `subscribeTopic()`, lines 50–52: the only teardown hook is `serverCallStreamObserver.setOnCancelHandler(disposable::dispose)`.
- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35: the `NettyServerBuilder` customizer sets only `executor` and `maxConcurrentCallsPerConnection`; no `keepAliveTime`, `keepAliveTimeout`, or `permitKeepAliveWithoutCalls` is configured.
- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15: `NettyProperties` exposes only `maxConcurrentCallsPerConnection`; no keepalive knob exists.

**Root cause:** gRPC-Java's `ServerCallStreamObserver.setOnCancelHandler` is invoked only when the gRPC transport layer signals cancellation — either via an explicit HTTP/2 RST_STREAM from the client, or via a failed keepalive ping. With no keepalive configured, a silently-dropped TCP connection (no FIN, no RST) is invisible to the server indefinitely. The `disposable.dispose()` call is therefore never reached, the Reactor subscription stays alive, and `poll()` continues issuing `topicMessageRepository.findByFilter()` queries every 500 ms (default `interval`) with up to 5 000 rows per page.

**Why existing checks fail:** `maxConcurrentCallsPerConnection = 5` limits streams per connection, not total connections. An attacker opens many connections. The nginx `grpc_read_timeout 600s` (docker-compose proxy) only applies when nginx is in the path and resets on each received message; it is absent in direct deployments and does not bound the server-side polling loop.

### Impact Explanation
Each orphaned subscription holds a live Reactor subscription on `Schedulers.boundedElastic()`, issues a DB query every ~500 ms, and buffers up to `maxPageSize = 5 000` rows per poll cycle. With N dead subscriptions the DB connection pool is saturated, CPU rises proportionally, and heap fills with buffered `TopicMessage` objects. This constitutes a server-side resource-exhaustion DoS. No authentication is required to call `subscribeTopic`; the endpoint is publicly reachable on port 5600.

### Likelihood Explanation
Any unprivileged user with TCP access to port 5600 can trigger this. The attacker needs only a standard gRPC client (e.g., `grpcurl`) and the ability to silently drop their own outbound packets (`iptables -A OUTPUT -p tcp --dport 5600 -j DROP`). This is trivially repeatable, requires no special knowledge of the application, and scales linearly: each additional connection adds 5 more leaking subscriptions. The attack is persistent until the server process is restarted or the OS TCP stack times out the connection (default Linux `tcp_keepalive_time` is 2 hours when TCP keepalive is not enabled at the application layer).

### Recommendation
1. **Configure gRPC keepalive** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.permitKeepAliveWithoutCalls(true);
   ```
   Expose these as fields in `NettyProperties` so they are operator-configurable.

2. **Add an application-level idle timeout** in `PollingTopicListener.listen()` as a defence-in-depth measure:
   ```java
   .timeout(Duration.ofMinutes(10))   // or a configurable property
   ```
   This ensures the flux self-terminates even if the transport layer never signals cancellation.

3. **Add a global subscription cap** (total active subscriptions, not just per-connection) to bound worst-case resource consumption.

### Proof of Concept
```bash
# 1. Subscribe to any topic (no auth required)
grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# 2. Immediately drop outbound packets to simulate a network partition
#    (server sees the connection as ESTABLISHED; no RST/FIN is sent)
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# 3. Observe on the server: the polling loop continues indefinitely.
#    DB query rate stays at ~2/s (one poll per 500 ms interval) per subscription.
#    Repeat steps 1-2 with many connections to exhaust the DB connection pool.
```
