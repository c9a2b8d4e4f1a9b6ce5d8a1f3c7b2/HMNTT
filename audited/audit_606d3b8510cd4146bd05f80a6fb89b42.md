### Title
Unbounded Redis Channel Subscription Growth via Unauthenticated Topic ID Enumeration Causes Message Delivery Failures

### Summary
`RedisTopicListener.getSharedListener()` uses an unbounded `ConcurrentHashMap` keyed by topic ID with no cap on entries. Each distinct topic ID causes a new Redis channel subscription via `ReactiveRedisMessageListenerContainer`. An unprivileged attacker opening many gRPC connections and subscribing to many distinct valid topic IDs can exhaust Redis pub/sub subscription capacity, degrading or breaking message delivery for all legitimate subscribers.

### Finding Description

**Exact code path:**

`getSharedListener()` at line 61 calls `computeIfAbsent` with no size guard:
```java
// RedisTopicListener.java:59-62
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`subscribe()` at line 72 opens a new Redis channel subscription for every distinct topic key:
```java
// RedisTopicListener.java:68-80
private Flux<TopicMessage> subscribe(Topic topic) {
    return container
            .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
            ...
            .share();
}
```

The `.share()` operator deduplicates subscribers to the **same** topic, but each **distinct** topic ID gets its own `r.receive()` call — a new Redis `SUBSCRIBE` command on the shared `ReactiveRedisMessageListenerContainer` connection.

Cleanup in `unsubscribe()` (lines 82–85) only fires on `doOnCancel`/`doOnComplete`, meaning entries persist as long as any subscriber holds the connection open.

**Root cause:** No upper bound on `topicMessages` map size; no limit on the number of distinct Redis channel subscriptions the container manages; no per-client or global subscription rate limit.

**Existing checks and why they fail:**

- `checkTopicExists = true` (default, `GrpcProperties.java:19`) — requires a valid topic ID in the DB, but Hedera has millions of publicly enumerable valid topic IDs discoverable via the REST API. This is not a meaningful barrier.
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java:14`, enforced in `GrpcConfiguration.java:33`) — limits calls **per TCP connection**, not total connections. An attacker opens `N` connections × 5 calls = `5N` simultaneous distinct-topic subscriptions with no server-side ceiling.
- No IP-based rate limiting, no authentication, no subscription quota anywhere in the call chain (`ConsensusController` → `TopicMessageServiceImpl` → `RedisTopicListener`).

**Exploit flow:**

1. Attacker queries the mirror node REST API to collect a large set of valid topic IDs.
2. Attacker opens `N` gRPC connections, each issuing 5 concurrent `subscribeTopic` RPCs to distinct topic IDs.
3. Each unique topic ID triggers `computeIfAbsent` → `subscribe()` → `r.receive()` → a new Redis `SUBSCRIBE` on the shared container connection.
4. The `topicMessages` map grows to `5N` entries; the `ReactiveRedisMessageListenerContainer` manages `5N` active channel subscriptions.
5. Redis pub/sub subscription limits (default 10,000 per connection in many deployments) are approached or exceeded; the container connection becomes saturated.
6. New legitimate subscriptions fail or existing ones experience delivery stalls/drops, causing missing records.

### Impact Explanation

When the Redis subscription connection is saturated, `r.receive()` calls for new topics fail or block. Existing shared `Flux` pipelines backed by an overwhelmed container experience backpressure or silent drops. Because `TopicMessageServiceImpl` relies on the listener for live message delivery (line 120: `topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t))`), saturation directly translates to missed live messages for all subscribers on the node — matching the "missing records exported to mirror nodes" scope. The `ConcurrentHashMap` also grows without bound, consuming heap proportional to the number of active distinct-topic subscriptions.

### Likelihood Explanation

The attack requires no credentials, no on-chain transactions, and no special tooling — only a gRPC client library and a list of valid topic IDs (freely available from the public REST API). It is repeatable and scalable: each additional attacker connection multiplies the subscription count by 5. The absence of any server-side connection count limit or subscription quota makes this trivially automatable.

### Recommendation

1. **Cap the subscription map**: Enforce a configurable maximum on `topicMessages.size()` before calling `computeIfAbsent`; reject or queue new subscriptions beyond the cap.
2. **Limit total concurrent gRPC connections**: Configure Netty's `maxConnectionAge` and a global `maxConcurrentCalls` (not just per-connection) via `NettyServerBuilder`.
3. **Rate-limit subscriptions per client IP**: Apply a token-bucket or sliding-window rate limiter at the gRPC interceptor layer before the request reaches `ConsensusController`.
4. **Monitor and alert on map size**: Expose `topicMessages.size()` as a Micrometer gauge and alert when it exceeds a threshold.

### Proof of Concept

```python
import grpc, threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hedera.proto import basic_types_pb2

# Collect valid topic IDs from REST API first
VALID_TOPIC_IDS = list(range(1, 10001))  # enumerate from REST /api/v1/topics
TARGET = "mirror-node-grpc-host:5600"

def flood_subscribe(topic_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_id)
    )
    try:
        for _ in stub.subscribeTopic(query):
            pass  # keep connection open
    except Exception:
        pass

threads = [threading.Thread(target=flood_subscribe, args=(tid,)) for tid in VALID_TOPIC_IDS]
for t in threads:
    t.start()
# Each thread holds a distinct Redis channel subscription open.
# After ~2000 threads, Redis subscription saturation begins degrading delivery for legitimate subscribers.
```
