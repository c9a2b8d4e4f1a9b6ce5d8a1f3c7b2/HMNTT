### Title
Cache Stampede via Unprotected `@Cacheable` with 1-Second TTL in `findByIdAndDeletedIsFalse`

### Summary
`EntityRepository.findByIdAndDeletedIsFalse` uses Spring's `@Cacheable` without `sync=true` against a Caffeine cache configured with a 1-second `expireAfterWrite` TTL. When the TTL expires, all concurrent threads simultaneously observe a cache miss and each independently issues a DB query, exhausting the HikariCP connection pool. An unprivileged attacker can deliberately trigger this by flooding the endpoint with requests for the same `entityId` at the TTL boundary, causing DB connection pool exhaustion and cascading request timeouts on affected nodes.

### Finding Description

**Exact code path:**

`EntityRepository.java` lines 20–30 declare:
```java
@Caching(cacheable = {
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_ENTITY, unless = "#result == null"),
    ...
})
Optional<Entity> findByIdAndDeletedIsFalse(Long entityId);
```
No `sync = true` attribute is present on either `@Cacheable`.

`EvmConfiguration.java` lines 99–105 configure `CACHE_MANAGER_ENTITY` as a plain `CaffeineCacheManager` using `setCacheSpecification(cacheProperties.getEntity())`.

`CacheProperties.java` line 19 sets the spec:
```java
private static final String ENTITY_CACHE_CONFIG = "expireAfterWrite=1s,maximumSize=10000,recordStats";
```

**Root cause:** Spring's `@Cacheable` without `sync=true` performs no per-key locking on cache miss. When the 1-second TTL expires, every thread that calls `findByIdAndDeletedIsFalse` with the same `entityId` simultaneously finds the key absent and independently invokes the underlying JPA/DB query. Caffeine's `expireAfterWrite` (as opposed to `refreshAfterWrite` with an async loader) provides zero thundering-herd protection in this configuration.

**Failed assumption:** The design assumes the 1-second TTL is short enough to keep data fresh but long enough that concurrent misses are rare under normal load. Under adversarial load (500 req/s, all for the same key), the miss window at each TTL boundary is exploited by every in-flight request simultaneously.

**Exploit flow:**
1. Attacker identifies any valid `entityId` (publicly observable from chain data).
2. Attacker sends a sustained flood of `eth_call` / contract-call requests that internally trigger `findByIdAndDeletedIsFalse(entityId)` — no authentication required.
3. The global rate limiter (`ThrottleManagerImpl`, default 500 req/s) allows up to 500 requests/second through.
4. Every ~1 second the cache entry expires. At that instant, all concurrent threads (potentially hundreds, depending on thread pool size and request latency) simultaneously observe a cache miss.
5. Each thread independently acquires a HikariCP connection and issues the DB query. HikariCP's default pool size is 10; all remaining threads queue waiting for a connection.
6. With `statementTimeout=3000ms`, threads block for up to 3 seconds. During this window the next TTL boundary arrives, repeating the stampede.
7. The connection pool is continuously saturated, causing `SQLTimeoutException` / `HikariPool-1 - Connection is not available` errors, making the node unresponsive.

### Impact Explanation
DB connection pool exhaustion on the affected web3 node causes all concurrent requests to fail or time out, effectively taking that node offline for the duration of the attack. Because the TTL is only 1 second, the stampede repeats every second, making recovery impossible while the attack continues. If multiple web3 nodes share the same PostgreSQL backend (or pgpool), the amplified query load can degrade the shared DB, impacting all nodes reading from it. The `requestTimeout=10000ms` (10 s) means clients receive errors, not stale data, so there is no graceful degradation.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send HTTP POST requests to the public `/api/v1/contracts/call` endpoint. The attacker needs only one valid contract address (trivially obtained from public chain explorers). The attack is fully repeatable and automatable with a simple HTTP load generator (e.g., `wrk`, `hey`, `ab`). The 500 req/s global rate limit is the only barrier, and it is high enough to sustain the stampede. No exploit code, credentials, or insider knowledge are required.

### Recommendation
1. **Immediate fix:** Add `sync = true` to the `@Cacheable` annotation on `findByIdAndDeletedIsFalse`. Spring will then use a per-key lock so only one thread loads from DB on a miss; all others wait for the result.
   ```java
   @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_ENTITY,
              unless = "#result == null", sync = true)
   ```
   Note: `sync = true` is incompatible with `@Caching` multi-cache stacking in Spring — the dual-cache pattern (entity + system account) must be refactored to use a single synchronized cache or a programmatic `Cache.get(key, Callable)` call.
2. **Alternative:** Switch to Caffeine's `AsyncLoadingCache` / `refreshAfterWrite` so only one thread reloads while others receive the stale value.
3. **Defense-in-depth:** Lower `requestsPerSecond` or add per-IP / per-`to`-address rate limiting in `ThrottleProperties` to reduce the blast radius.

### Proof of Concept
```bash
# 1. Identify any valid contract address (e.g., from hashscan.io)
CONTRACT="0x0000000000000000000000000000000000000167"

# 2. Flood the endpoint at 500 req/s, all targeting the same contract
#    (triggers findByIdAndDeletedIsFalse for the same entityId on every call)
hey -n 10000 -c 500 -m POST \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"'"$CONTRACT"'","estimate":false}' \
  http://<web3-node>/api/v1/contracts/call

# 3. Observe: after ~1 second, DB connection pool exhaustion errors appear in logs:
#    "HikariPool-1 - Connection is not available, request timed out after 30000ms"
#    All subsequent requests return HTTP 500 / 503 until attack stops.
```
