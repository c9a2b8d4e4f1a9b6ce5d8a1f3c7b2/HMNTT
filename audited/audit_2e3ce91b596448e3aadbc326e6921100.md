### Title
DB Connection Pool Exhaustion via Concurrent Long-Running Queries Blocks Gossip Transaction Confirmation

### Summary
The Rosetta API's `GetDbWithContext` enforces a 20-second Go-level `context.WithTimeout` per query, but the default pool of 100 open connections has no application-level per-client concurrency limit. The Traefik `inFlightReq`/`rateLimit` middleware that would enforce per-IP limits is disabled by default (`global.middleware: false`). An unprivileged attacker can flood the server with 100 concurrent requests to DB-backed endpoints, holding all pool connections for up to 20 seconds each, starving legitimate gossip-confirmation DB operations of connections.

### Finding Description

**Code path:**

`rosetta/app/config/types.go:45` — `StatementTimeout int` defaults to 20 seconds. [1](#0-0) 

`rosetta/app/db/client.go:36` — every DB-backed handler calls `GetDbWithContext`, which wraps the request context with a 20-second deadline. The connection is held until the query completes or the deadline fires. [2](#0-1) 

`rosetta/app/db/db.go:31-33` — the pool is configured with `MaxOpenConnections` defaulting to 100 and no `SetConnWaitTimeout`. [3](#0-2) 

`rosetta/app/persistence/transaction.go:120-146` — `FindBetween` holds a single connection across a batched loop (2000 rows/iteration). A block with many transactions causes repeated queries on the same connection, keeping it occupied for the full 20-second window. [4](#0-3) 

**Root cause / failed assumption:** The design assumes that the Traefik middleware chain (specifically `inFlightReq: amount: 5` per source IP and `rateLimit: average: 10`) will bound concurrent DB-holding requests per client. However, `global.middleware` defaults to `false`, so the middleware template is never rendered and the Traefik `Middleware` CRD is never created. [5](#0-4) [6](#0-5) [7](#0-6) 

There is no application-level semaphore, rate limiter, or per-client connection cap anywhere in the Go server code. [8](#0-7) 

### Impact Explanation

When all 100 pool slots are occupied, Go's `database/sql` blocks new callers waiting for a free connection with no configurable wait timeout. Legitimate calls to `/block`, `/block/transaction`, and `/search/transactions` — which the Rosetta construction service uses to confirm that a submitted transaction was gossiped and included in consensus — queue indefinitely or until the 20-second context deadline fires. This produces a sustained denial-of-service against gossip-confirmation queries: operators and integrators cannot verify transaction finality for the full duration of the attack. Severity is **High** (availability impact on a critical read path; no data is modified).

### Likelihood Explanation

The attack requires no credentials, no special knowledge of the chain state, and no exploit tooling beyond a standard HTTP client. Sending 100 concurrent POST requests to `/block` with a valid (or even invalid) block identifier is trivial. Because the default Helm chart ships with `global.middleware: false`, the vast majority of deployments that follow the provided chart without explicit hardening are vulnerable. The attack is repeatable: as soon as the 20-second timeout expires and connections are returned, the attacker can immediately re-flood.

### Recommendation

1. **Enable the Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`.
2. **Add a pool wait timeout at the application layer**: Call `sqlDb.SetConnMaxIdleTime` and use a `context.WithTimeout` wrapping the pool-acquisition step, or set `db.SetMaxOpenConns` in conjunction with a bounded semaphore in `ConnectToDb` (`rosetta/app/db/db.go`).
3. **Add a Go-level concurrency semaphore**: Introduce an HTTP middleware (alongside `MetricsMiddleware` and `TracingMiddleware` in `rosetta/main.go`) that limits total in-flight DB-bound requests to a value well below `MaxOpenConnections`.
4. **Set `statement_timeout` at the PostgreSQL role level unconditionally**: The Helm chart already sets `statement_timeout = 10000` for `mirror_rosetta` in `secret-passwords.yaml`, but this is only applied during initial cluster setup. Ensure it is enforced independently of the Go-level context so the DB itself terminates runaway queries and releases server-side resources.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or xargs
# Target: default Rosetta deployment with global.middleware=false

ROSETTA_URL="http://<rosetta-host>:5700"
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'

# Craft a /block request for a block known to have many transactions
PAYLOAD=$(cat <<EOF
{
  "network_identifier": $NETWORK,
  "block_identifier": {"index": 1}
}
EOF
)

# Fire 100 concurrent requests, each will hold a DB connection for up to 20s
for i in $(seq 1 100); do
  curl -s -X POST "$ROSETTA_URL/block" \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait

# Now send a legitimate gossip-confirmation query — it will time out or error
# because all 100 pool connections are occupied
curl -X POST "$ROSETTA_URL/block/transaction" \
  -H "Content-Type: application/json" \
  -d "{\"network_identifier\":$NETWORK,\"block_identifier\":{\"index\":1},\"transaction_identifier\":{\"hash\":\"0x...\"}}"
# Expected: connection wait timeout / 500 error instead of normal response
```
