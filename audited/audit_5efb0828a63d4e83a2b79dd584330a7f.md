### Title
Unbounded Infinite Retry Loop in `subscribe()` Exploitable by Unprivileged gRPC Clients During Redis Network Partition

### Summary
`RedisTopicListener.subscribe()` uses `Retry.backoff(Long.MAX_VALUE, interval)` with no circuit breaker, meaning any unprivileged gRPC client that subscribes to a topic triggers a shared retry loop that runs indefinitely during a Redis network partition. An attacker subscribing to many distinct topic IDs creates proportionally many independent retry loops, each making repeated TCP connection attempts to Redis with no mechanism to halt when recovery is clearly not imminent.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()` (lines 68–80):

```java
return container
    .flatMapMany(r -> r.receive(...))
    ...
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .share();
``` [1](#0-0) 

The `container` Mono itself also retries with `Long.MAX_VALUE`:
```java
.retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(8)))
``` [2](#0-1) 

**Root cause:** `Long.MAX_VALUE` as the retry count is effectively infinite. There is no circuit breaker, no maximum attempt count, and no failure-rate threshold that would open a circuit and stop retrying. The only rate-limiting is the exponential backoff capped at `interval × 4` = 2 seconds (default `interval` = 500 ms). [3](#0-2) 

**Subscription scoping:** `getSharedListener()` uses `computeIfAbsent` keyed on topic name, so one retry loop exists per distinct topic ID, not per client. [4](#0-3) 

**Exploit flow:**
1. Redis network partition occurs (infrastructure event or attacker-induced Redis saturation).
2. Attacker opens N gRPC connections, each subscribing to a distinct topic ID (e.g., `topic.1`, `topic.2`, … `topic.N`). Each call to `getSharedListener()` with a new topic ID invokes `subscribe()` and inserts a new entry into `topicMessages`.
3. Each of the N independent `subscribe()` Flux pipelines enters its own `retryWhen` loop, making TCP connection attempts to Redis at up to 2 req/s per topic (initial 500 ms backoff, max 2 s).
4. No circuit breaker fires. The loops run indefinitely at N × (up to 2 attempts/s) regardless of how long the partition lasts.
5. The attacker keeps their gRPC streams open (or reconnects) to prevent `unsubscribe()` from removing entries from `topicMessages` and terminating the loops.

**Why existing checks are insufficient:** The backoff provides rate-limiting per topic but is not a circuit breaker — it never stops retrying. There is no failure-count threshold, no half-open state, and no mechanism to detect that the partition is sustained. The `share()` operator prevents per-client amplification but does not limit the number of distinct topics an attacker can subscribe to.

### Impact Explanation
Each active retry loop issues a TCP SYN to Redis every 500 ms–2 s. With N = 1000 distinct topic subscriptions (trivially achievable via scripted gRPC calls), the server generates 500–2000 TCP connection attempts per second to Redis, consuming CPU (reactor scheduler threads, connection pool management), file descriptors, and network bandwidth on both the gRPC node and the Redis host. This degrades or denies service to legitimate subscribers and can exhaust OS-level connection limits. The impact persists for the entire duration of the partition with no self-limiting behavior.

### Likelihood Explanation
The gRPC topic subscription API is publicly accessible by design (mirror node public API). No authentication or authorization is required to subscribe to a topic. An attacker needs only a gRPC client library and knowledge of the API (publicly documented). The network partition can be a pre-existing infrastructure failure; the attacker does not need to cause it — they only need to subscribe during it. The attack is trivially scriptable and repeatable.

### Recommendation
1. **Add a circuit breaker** using Reactor's `Retry.backoff(...).filter(...)` or Resilience4j's `CircuitBreaker` operator. After a configurable number of consecutive failures (e.g., 10) or a failure rate over a time window, open the circuit and stop retrying until a probe succeeds.
2. **Cap retry attempts** by replacing `Long.MAX_VALUE` with a finite bound (e.g., `Retry.backoff(20, interval).maxBackoff(...)`), then surface the terminal error to subscribers so they can reconnect on their own schedule.
3. **Rate-limit topic subscriptions per client** at the gRPC layer to prevent a single unprivileged client from creating an unbounded number of independent retry loops.
4. **Increase `maxBackoff`** significantly (e.g., 60 s) so that during a sustained partition the retry rate is negligible.

### Proof of Concept
```python
import grpc
import threading

# Publicly available mirror node gRPC stub
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
import google.protobuf.timestamp_pb2 as ts

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

def subscribe(topic_id):
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,  # set topic_id here
        consensusStartTime=ts.Timestamp(seconds=0)
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass  # keep thread alive to hold the subscription entry

# Subscribe to 1000 distinct topic IDs
threads = [threading.Thread(target=subscribe, args=(i,)) for i in range(1, 1001)]
for t in threads:
    t.start()

# With Redis partitioned, the server now runs 1000 independent retry loops
# each attempting TCP connections to Redis at up to 2/s = up to 2000 conn attempts/s
```

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
