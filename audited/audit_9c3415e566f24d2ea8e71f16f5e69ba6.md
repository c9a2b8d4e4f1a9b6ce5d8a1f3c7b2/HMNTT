### Title
Unbounded Timestamp Range in `getRecordFileBlockDetailsFromTimestampArray` Enables DB Range-Scan DoS

### Summary
`getRecordFileBlockDetailsFromTimestampArray` in `rest/service/recordFileService.js` constructs a `record_file` query bounded by `[minTimestamp, BigInt(maxTimestamp) + config.query.maxRecordFileCloseIntervalNs]` with no validation that the span of the caller-influenced timestamp array is reasonable. An unprivileged user can trigger this with a wide timestamp range via the public `/api/v1/contracts/results/logs` endpoint, causing the inner correlated subquery to perform an expensive range scan across the full history of the `record_file` table, repeated up to 100 times per request. Flooding the limited DB connection pool (default 10) with such requests saturates I/O and starves legitimate queries.

### Finding Description

**Code location:** `rest/service/recordFileService.js`, `getRecordFileBlockDetailsFromTimestampArray()`, line 101. [1](#0-0) 

The function derives `minTimestamp` and `maxTimestamp` purely from the first and last elements of the `timestamps` array: [2](#0-1) 

These become `$2` and `$3` in the SQL template: [3](#0-2) 

The inner correlated subquery runs once per element of `unnest($1::bigint[])` (up to 100 elements, the `maxRepeatedQueryParameters` ceiling), and each execution performs a range scan on `record_file.consensus_end` between `$2` and `$3`. If those bounds span the entire network history (e.g., timestamp 1 to current time), every correlated execution scans the full live range of the table.

**Call path to user-controlled input:**

`getContractLogs` in `contractService.js` collects `consensus_timestamp` values from the contract-log result rows and passes them directly: [4](#0-3) 

Those rows are produced by the contract-logs query, whose timestamp bounds come from the user's HTTP query parameters, processed in `contractController.js`: [5](#0-4) 

**Why existing checks fail:**

- `maxRepeatedQueryParameters` (default 100) caps the *count* of timestamps but not their *span*.
- `bindTimestampRange` is **disabled by default** (`false`), so `maxTransactionsTimestampRangeNs` is never enforced on the contract-logs query: [6](#0-5) 
- `maxRecordFileCloseIntervalNs` (default 10 s) adds only a negligible constant to `$3`.
- `statementTimeout` (default 20 000 ms) provides a partial backstop but does not prevent the attack from consuming all pool connections for 20 s per wave. [7](#0-6) 

### Impact Explanation
The DB connection pool defaults to 10 connections. Each malicious request holds a connection for up to 20 s (statement timeout) while the correlated subquery executes up to 100 range scans across the full `record_file` history. A single attacker sending ~10 concurrent requests continuously exhausts the pool, blocking all legitimate REST API queries. Because the mirror-node REST API is shared infrastructure for all network participants querying state, sustained pool exhaustion degrades or halts the ability of dependent services (wallets, explorers, dApps) to read network state, consistent with the ≥30% processing-node disruption classification.

### Likelihood Explanation
The endpoint `/api/v1/contracts/results/logs` requires no authentication. The exploit requires only a standard HTTP client and knowledge of two timestamp values (e.g., `timestamp=gte:0.000000001&timestamp=lte:9999999999.999999999`). It is trivially repeatable and scriptable, and no rate-limiting is applied at the application layer.

### Recommendation
1. **Enforce a maximum timestamp span** before calling `getRecordFileBlockDetailsFromTimestampArray`: reject or clamp requests where `maxTimestamp - minTimestamp` exceeds a configured threshold (e.g., `maxTimestampRangeNs`, default 7 d).
2. **Enable `bindTimestampRange`** by default, or apply it unconditionally to the contract-logs timestamp filter so the range fed into the record-file lookup is always bounded.
3. **Add per-IP or global rate limiting** on the contract-logs endpoints.
4. Consider rewriting the correlated subquery as a single range query with a `LATERAL` join to avoid N separate index scans.

### Proof of Concept
```bash
# Send 10 concurrent requests with a timestamp range spanning the full network history
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/contracts/results/logs?timestamp=gte:0.000000001&timestamp=lte:9999999999.000000000&limit=100" &
done
wait
# Repeat in a loop; legitimate API calls will time out or receive 503 within seconds
# as all 10 DB pool connections are held by the expensive correlated record_file scans.
```
