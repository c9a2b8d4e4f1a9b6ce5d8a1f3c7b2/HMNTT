### Title
Uncached, Unauthenticated GraphQL Entity Lookups with No Rate Limiting Enable Sustained DB I/O and CPU Exhaustion

### Summary
The GraphQL module's `EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` on every request with no caching and no rate limiting. Unlike the `grpc` and `web3` modules which both apply `@Cacheable` to their `EntityRepository.findById()` methods, the GraphQL `EntityRepository` has no cache annotations. The GraphQL module also has no request-rate throttle (no bucket4j, no requests-per-second limit), meaning any unauthenticated external user can drive an unbounded number of full-row DB fetches per second.

### Finding Description

**Code path:**

`AccountController.account()` (line 43) calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)`. [1](#0-0) 

`EntityServiceImpl.getByIdAndType()` (line 25) calls `entityRepository.findById(entityId.getId())` with no caching layer. [2](#0-1) 

The GraphQL `EntityRepository` extends bare `CrudRepository<Entity, Long>` with no `@Cacheable` annotation on `findById()`. [3](#0-2) 

Contrast with the `grpc` module's `EntityRepository`, which applies `@Cacheable` to `findById()`: [4](#0-3) 

And the `web3` module's `EntityRepository`, which applies `@Caching` with multiple cache managers: [5](#0-4) 

**No rate limiting in the GraphQL module:** The only protections configured are `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which limit query structure but impose no per-IP or global request-rate cap. There is no bucket4j, no `requestsPerSecond` property, and no throttle filter anywhere in the `graphql` module. [6](#0-5) 

The `Entity` domain object (`AbstractEntity`) has 29 mapped columns including variable-length byte arrays (`alias`, `key`, `evmAddress`, `delegationAddress`) and a `publicKey` string. Every `findById()` call deserializes the full row. [7](#0-6) 

**Exploit flow:**
1. Attacker identifies any valid account entity ID (publicly observable on-chain).
2. Attacker sends a tight loop of GraphQL POST requests:
   ```graphql
   { account(input: { entityId: { shard: 0, realm: 0, num: 1234 } }) { id balance memo key } }
   ```
3. Each request passes complexity/depth checks (trivially simple query), reaches `getByIdAndType()`, and issues a fresh `SELECT * FROM entity WHERE id = ?` with full row deserialization.
4. No cache absorbs repeated identical lookups; no rate limiter throttles the request stream.

### Impact Explanation
Each request causes a synchronous DB round-trip and full Hibernate entity deserialization. At sustained high request rates (easily achievable from a single host), this drives DB I/O and JVM CPU well above the 30% baseline threshold defined in the scope. The GraphQL pod's CPU alert fires at 80% sustained for 5 minutes; the DB connection utilization alert fires at 75%. [8](#0-7) 

### Likelihood Explanation
No authentication is required. The GraphQL endpoint is publicly exposed. A single attacker with a standard HTTP client can sustain hundreds of requests per second. The query is trivially valid (complexity well under 200, depth well under 10), so all existing guards pass. The attack is repeatable indefinitely.

### Recommendation
1. Add `@Cacheable` to the GraphQL `EntityRepository.findById()` (mirroring the `grpc` module pattern) with a short TTL (e.g., 1–5 seconds) to absorb repeated identical lookups.
2. Implement a request-rate limiter (bucket4j or Spring's `HandlerInterceptor`) in the GraphQL module, analogous to `ThrottleManagerImpl` in the `web3` module, enforcing a global and/or per-IP requests-per-second cap.
3. Consider adding a connection-pool-level statement timeout for the GraphQL datasource.

### Proof of Concept
```bash
# Identify any valid account ID (e.g., from public mirror REST API)
# Then flood the GraphQL endpoint:
while true; do
  curl -s -X POST https://<mirror-node-graphql>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}) { id balance memo } }"}' \
    &
done
# No credentials required. Each request hits the DB with a full SELECT * FROM entity WHERE id=2.
# Monitor DB I/O and JVM CPU — both rise proportionally with request rate, uncapped.
```
