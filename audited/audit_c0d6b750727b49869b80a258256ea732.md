### Title
Unbounded Concurrent Subscription DoS via Near-Future `endTime` Keeping Flux Pipelines Alive for 30 Seconds Each

### Summary
An unprivileged gRPC client can set `consensusEndTime` to a value just nanoseconds past the current time, which passes input validation but causes `isComplete()` to return `false` for the full `endTimeInterval` (default 30 seconds). By opening many TCP connections — each carrying up to 5 concurrent `subscribeTopic` calls — an attacker can accumulate hundreds of live Flux pipelines simultaneously, exhausting `boundedElastic` scheduler threads, heap memory, and database connection pool resources without any authentication or privilege.

### Finding Description

**Exact code path:**

`TopicMessageFilter.isValidEndTime()` only enforces `endTime > startTime`: [1](#0-0) 

There is no upper bound — `endTime = now + 1 nanosecond` passes validation.

`TopicContext.isComplete()` returns `false` until `endTime + endTimeInterval` is in the past: [2](#0-1) 

With `endTimeInterval = 30s` (the default): [3](#0-2) 

…a subscription with `endTime = now + 1ns` keeps `isComplete()` returning `false` for ~30 full seconds.

`pastEndTime()` polls `isComplete()` every `endTimeInterval`, so the `takeUntilOther` signal is never emitted during that window: [4](#0-3) 

The `safetyCheck` branch runs on `Schedulers.boundedElastic()`, consuming a thread from the bounded pool for every subscription that is not yet complete after 1 second: [5](#0-4) 

The `subscriberCount` field is a **metric gauge only** — it is never checked against any cap: [6](#0-5) 

**Why the only existing guard is insufficient:**

`maxConcurrentCallsPerConnection = 5` limits calls *per TCP connection*: [7](#0-6) [8](#0-7) 

There is no global cap on the number of TCP connections or total concurrent subscriptions. An attacker opens `N` connections × 5 calls = `5N` simultaneous live pipelines.

### Impact Explanation

Each live pipeline holds: a `TopicContext` object, a `boundedElastic` thread (via `safetyCheck`), a `topicListener` subscription (Redis or shared listener), and a pending `pastEndTime` repeat timer. With enough concurrent subscriptions the `boundedElastic` thread pool (default: `10 × CPU cores`) is saturated, blocking all other reactive work on the server. Memory pressure from hundreds of in-flight `Flux` pipelines and `TopicContext` objects can trigger GC pressure or OOM. Database connection pool exhaustion is also possible via repeated `topicMessageRetriever.retrieve()` calls from the safety-check path. The service becomes unavailable to legitimate users — a non-network DoS.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC endpoint is publicly reachable. The exploit requires only a standard gRPC client library and the ability to open multiple TCP connections, which is trivial. The 30-second window per subscription means an attacker sustaining ~10 new connections per second can maintain 1,500 concurrent pipelines indefinitely. The attack is fully repeatable and scriptable.

### Recommendation

1. **Enforce an upper bound on `endTime`**: Reject subscriptions where `endTime > now + maxAllowedFutureWindow` (e.g., 60 seconds) in `TopicMessageFilter.isValidEndTime()`.
2. **Enforce a global (or per-IP) concurrent subscription cap**: Check `subscriberCount` against a configurable maximum before creating a new pipeline in `subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when exceeded.
3. **Reduce `endTimeInterval`** or make it proportional to how far `endTime` is in the future, so near-past `endTime` subscriptions terminate quickly.
4. **Add per-source-IP connection rate limiting** at the Netty layer to slow connection-flood amplification.

### Proof of Concept

```python
import grpc
import time
from hashgraph.com.hedera.hapi.node.base import timestamp_pb2
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = ...  # any valid topic shard/realm/num

connections = []
for i in range(200):           # 200 TCP connections
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    now_ns = time.time_ns()
    end_ns = now_ns + 1        # 1 nanosecond in the future
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        consensusEndTime=timestamp_pb2.Timestamp(
            seconds=end_ns // 1_000_000_000,
            nanos=end_ns % 1_000_000_000),
    )
    # Open 5 concurrent streams per connection (maxConcurrentCallsPerConnection)
    for _ in range(5):
        stub.subscribeTopic(req)   # non-blocking streaming call
    connections.append(ch)

# 200 × 5 = 1000 live Flux pipelines, each alive for ~30 s
# boundedElastic pool exhausted; server unresponsive to legitimate clients
time.sleep(35)   # repeat to maintain pressure indefinitely
```

        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }
