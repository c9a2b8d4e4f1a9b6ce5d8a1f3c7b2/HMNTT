### Title
Unbounded Concurrent Request Logging in `TracingMiddleware` Enables Log-Mutex DoS Degrading Transaction Gossip

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` unconditionally calls `log.Info` (via logrus) for every non-internal HTTP request with no rate limiting, connection cap, or goroutine bound at the application layer. Because logrus serializes all writes through an internal `sync.Mutex`, an unprivileged attacker flooding the server with concurrent requests forces all goroutines to queue on that mutex, compounding CPU and scheduling overhead and degrading throughput for legitimate transaction-gossip endpoints (`/construction/submit`). The only rate-limiting controls that exist are Traefik-level and are **disabled by default** in the shipped Helm chart.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 43–61:

```go
func TracingMiddleware(inner http.Handler) http.Handler {
    return http.HandlerFunc(func(responseWriter http.ResponseWriter, request *http.Request) {
        start := time.Now()
        ...
        inner.ServeHTTP(tracingResponseWriter, request)
        message := fmt.Sprintf(...)
        if internalPaths[path] {
            log.Debug(message)   // /health, /metrics
        } else {
            log.Info(message)    // ALL other paths — unconditional
        }
    })
}
```

Every request to any non-internal path (including `/construction/submit`, `/block`, `/network/list`, etc.) unconditionally calls `log.Info`. Logrus's `Info` path acquires an internal `sync.Mutex` before writing to the output stream. Go's `net/http` server spawns one goroutine per accepted connection with no configured limit.

**Middleware chain** (`rosetta/main.go`, lines 217–219):
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```
`TracingMiddleware` is the outermost application handler; every request passes through it before any business logic.

**`http.Server` configuration** (`rosetta/main.go`, lines 220–227): No `MaxHeaderBytes` override, no connection limit, no goroutine pool — only optional read/write timeouts whose defaults are not set in `config/types.go`.

**Root cause:** The middleware assumes an upstream infrastructure layer will throttle requests before they reach the application. That assumption is encoded in the Helm chart's Traefik middleware chain (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166: `rateLimit: average: 10`, `inFlightReq: amount: 5`), but that entire chain is gated by `global.middleware: false` (line 95) — **off by default**. The template guard (`{{ if and .Values.global.middleware .Values.middleware }}`) means no Traefik `Middleware` CRD is ever deployed unless an operator explicitly opts in.

**Why existing checks fail:**
- Application layer: zero rate limiting, zero connection cap.
- Infrastructure layer: Traefik middleware exists in config but is disabled by default; bypassed entirely if the pod port is accessed directly (e.g., via `kubectl port-forward`, NodePort, or misconfigured LoadBalancer).

### Impact Explanation

Under a flood of concurrent requests, every goroutine calls `log.Info`, which blocks on logrus's internal mutex. With thousands of goroutines contending on a single mutex, the Go scheduler spends increasing time context-switching between blocked goroutines, and the log writer becomes a serialization bottleneck. Combined with unbounded goroutine creation and `tracingResponseWriter` buffering each response body in memory (`w.data = data`, line 38), the server experiences:

- CPU saturation from goroutine scheduling and mutex spin/wait
- Memory pressure from per-request `tracingResponseWriter` allocations
- Increased latency for all requests, including `/construction/submit` (transaction gossip)
- Potential complete unavailability of the construction endpoint, preventing transaction submission to the Hedera network

This directly impacts the protocol-level function of the Rosetta node: an attacker can prevent legitimate clients from gossiping transactions during the attack window.

### Likelihood Explanation

**Preconditions:** None. No authentication, no API key, no special network position required. Any internet-accessible Rosetta node is a valid target.

**Feasibility:** A single attacker machine with a tool like `wrk`, `hey`, or a simple Go/Python script can sustain tens of thousands of concurrent HTTP connections. The attack is stateless (any valid or invalid HTTP request triggers the log write), cheap to sustain, and immediately effective.

**Repeatability:** The attack is fully repeatable and can be sustained indefinitely. Recovery requires the flood to stop; there is no self-healing mechanism.

**Default exposure:** Because `global.middleware: false` is the shipped default, any deployment using the Helm chart without explicitly enabling Traefik middleware is fully exposed at the application level.

### Recommendation

1. **Application-level rate limiting:** Add a token-bucket or leaky-bucket rate limiter directly inside `TracingMiddleware` (or as a dedicated middleware before it) using `golang.org/x/time/rate`. This must be at the Go layer, not delegated solely to infrastructure.

2. **Limit concurrent connections:** Set `http.Server.MaxHeaderBytes` and consider wrapping the listener with `netutil.LimitListener` to cap total concurrent connections.

3. **Enable Traefik middleware by default:** Change `global.middleware: false` to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` and `rateLimit` controls are active in all deployments.

4. **Async/sampled logging:** Consider logging at `Debug` level for high-frequency paths or sampling log output (e.g., log 1-in-N requests under load) to reduce mutex contention.

### Proof of Concept

```bash
# Prerequisites: Rosetta node accessible at localhost:5700,
# global.middleware NOT enabled (default deployment)

# Install wrk (HTTP benchmarking tool)
# Then flood the construction endpoint:
wrk -t 50 -c 5000 -d 60s \
  -s post.lua \
  http://localhost:5700/network/list

# post.lua:
# wrk.method = "POST"
# wrk.body   = '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
# wrk.headers["Content-Type"] = "application/json"
```

**Expected result:** Within seconds, `log.Info` call rate saturates the logrus mutex. Server response latency for `/construction/submit` (sent from a separate legitimate client) climbs from milliseconds to seconds or times out entirely. The server process shows near-100% CPU with most goroutines blocked in `(*Logger).log` → `sync.Mutex.Lock`. Legitimate transaction gossip is effectively denied for the attack duration. 		} else {

		}
	})
}
