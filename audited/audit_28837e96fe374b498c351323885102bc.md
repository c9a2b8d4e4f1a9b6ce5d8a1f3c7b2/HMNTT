### Title
Unprivileged SLOAD-Amplified Database I/O Saturation via Global Gas Throttle Bypass on Historical Calls

### Summary
Any unauthenticated user can submit `eth_call` requests targeting a contract with many storage slots, using the maximum gas limit (15,000,000), causing the EVM to execute thousands of SLOAD operations per request. For historical block calls, every SLOAD maps directly to an uncached database query with no per-user rate limiting, allowing a single attacker to saturate database I/O and degrade service for all users.

### Finding Description

**Exact code path:**

`ContractController.call()` → `throttleManager.throttle(request)` (global gas bucket only) → `doProcessCall()` (lines 109–138) → `transactionExecutionService.execute()` → EVM executes SLOADs → `ContractStorageReadableKVState.readFromDataSource()` → for historical calls: `contractStateService.findStorageByBlockTimestamp()` → direct uncached DB query every time.

**Root cause — failed assumption:**

`doProcessCall()` relies entirely on a single global gas-per-second bucket as its only I/O guard. The failed assumption is that gas cost is proportional to database I/O cost. It is not, for two reasons:

1. **Global, not per-user throttle.** `ThrottleManagerImpl.throttle()` checks one shared `gasLimitBucket` for all callers. A single attacker can legally consume the entire gas budget, starving legitimate users and maximising DB pressure simultaneously.

2. **Historical calls are completely uncached.** `ContractStorageReadableKVState.readFromDataSource()` branches on the presence of a timestamp:
   - Non-historical path → `contractStateService.findStorage()` → Caffeine cache (`maximumSize=25000, expireAfterWrite=2s`) + batch DB query.
   - Historical path → `contractStateService.findStorageByBlockTimestamp()` → **direct `contractStateRepository.findStorageByBlockTimestamp()` call with zero caching**.

   Every cold SLOAD in a historical call is a separate DB round-trip.

**Gas arithmetic:**

- `maxGasLimit = 15,000,000` (EvmProperties line 69)
- Cold SLOAD cost (EIP-2929) = 2,100 gas → **~7,142 cold SLOADs per max-gas request**
- `gasPerSecond = 7,500,000,000` (ThrottleProperties line 26) → allows **500 max-gas requests/second**
- `requestsPerSecond = 500` (ThrottleProperties line 35) → same ceiling
- **Result: 500 × 7,142 = ~3,571,000 uncached DB reads/second** from a single attacker

`scaleGas()` divides by `GAS_SCALE_FACTOR = 10,000`, so 15M gas = 1,500 tokens consumed. The `restoreGasToBucket` logic in lines 140–152 restores only unused gas, so the effective throttle is based on actual gas used — but this does not bound the number of DB reads per gas unit.

**Why existing checks fail:**

- `validateContractMaxGasLimit()` (ContractController line 92–96) only enforces the ceiling; it does not limit DB reads.
- The gas throttle (ThrottleManagerImpl lines 37–48) is a global token bucket with no per-IP or per-user dimension.
- The `contractStateCache` (docs: `expireAfterWrite=2s, maximumSize=25000`) is bypassed entirely on historical calls and can be bypassed on current calls by targeting contracts with more than 25,000 unique slots or by rotating slot keys.
- `db.statementTimeout = 3000ms` limits individual query duration but does not limit the number of concurrent queries issued within a single EVM execution.

### Impact Explanation

A single unauthenticated attacker can drive the mirror-node PostgreSQL instance to its I/O ceiling by submitting 500 historical `eth_call` requests per second, each triggering ~7,142 uncached `contract_state` table lookups. This degrades or denies service for all other users of the Web3 API. Because the gas throttle is global, the attacker simultaneously exhausts the gas budget, throttling legitimate callers. No funds are at risk, but availability of the read-only API is fully compromised. Severity is Medium (griefing / availability impact, no economic loss).

### Likelihood Explanation

No account, API key, or payment is required. The attack requires only an HTTP client and knowledge of any deployed contract with many storage slots (the project's own `SlotContract` test fixture with 600 slots is publicly documented in `tools/k6/README.md`). The attack is trivially repeatable and scriptable. The attacker does not need to observe any response — fire-and-forget POST requests to `/api/v1/contracts/call` with `block` set to a historical value and `gas` set to 15,000,000 are sufficient.

### Recommendation

1. **Add per-IP (or per-caller) rate limiting** in addition to the global gas bucket, so a single source cannot consume the entire gas budget.
2. **Cache historical storage reads.** `findStorageByBlockTimestamp()` returns immutable data (historical state never changes); it is safe to cache aggressively. Add a dedicated cache keyed on `(contractId, slotKey, blockTimestamp)`.
3. **Introduce a per-request SLOAD (DB-read) counter** inside `doProcessCall()` or the EVM execution layer and reject or short-circuit requests that exceed a configurable threshold (e.g., 1,000 unique slot reads per call).
4. **Reduce `maxGasLimit`** or introduce a separate, lower limit for historical calls, where caching is absent.
5. **Decouple gas throttle from DB-read throttle**: track actual DB queries issued per second as a separate bucket independent of gas.

### Proof of Concept

**Preconditions:**
- A contract with ≥600 storage slots exists on the network (e.g., the documented `SlotContract` / `SlotContractCaller` at address `0x00000000000000000000000000000000008f005f`).
- No authentication is required.

**Steps:**

```bash
# Send 500 concurrent historical eth_call requests per second,
# each triggering ~7,142 cold SLOADs via heavyReadBothHalves()
# (3,000 SLOADs documented; with max gas, even more are possible)

while true; do
  for i in $(seq 1 500); do
    curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d '{
        "to":   "0x00000000000000000000000000000000008f005f",
        "data": "0x523adad6",
        "gas":  15000000,
        "block": "0x1000000"
      }' &
  done
  wait
done
```

**Result:** Each request triggers `findStorageByBlockTimestamp()` for every SLOAD with no caching. At 500 RPS × ~3,000–7,142 DB reads, the `contract_state` table is queried millions of times per second, saturating DB I/O and causing timeouts or severe latency degradation for all concurrent users.
