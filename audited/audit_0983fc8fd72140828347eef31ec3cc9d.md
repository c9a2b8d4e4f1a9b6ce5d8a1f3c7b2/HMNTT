### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Enables MITM Balance Manipulation

### Summary
`GetDsn()` in `rosetta/app/config/types.go` hardcodes `sslmode=disable` unconditionally, making it impossible to configure TLS for the PostgreSQL connection regardless of deployment environment. An attacker with L2 network access to the same segment as the Rosetta-to-DB link can ARP-spoof the connection, intercept the plaintext PostgreSQL wire protocol, and inject forged query responses. This causes `RetrieveBalanceAtBlock` to return an attacker-controlled balance, which an exchange or wallet consuming the Rosetta API would treat as authoritative.

### Finding Description

**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
"host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
``` [1](#0-0) 

The `Db` struct (lines 39–47) has no `SslMode` field at all — the value is not configurable via YAML or environment variable. [2](#0-1) 

`ConnectToDb()` passes this DSN directly to GORM/`pgx` with no override: [3](#0-2) 

`RetrieveBalanceAtBlock` issues multiple raw SQL queries over this unencrypted connection — `getLatestBalanceSnapshot` and `getBalanceChange` — and arithmetically combines their results before returning the final balance: [4](#0-3) 

**Root cause:** `sslmode=disable` is hardcoded and non-overridable. Unlike the importer component (`DBProperties.java` line 50: `SslMode sslMode = SslMode.DISABLE` which is configurable via `hiero.mirror.importer.db.sslMode`), the Rosetta component provides no mechanism to enable transport security. [5](#0-4) 

**Exploit flow:**
1. Attacker gains L2 adjacency to the network segment carrying Rosetta↔PostgreSQL traffic (e.g., compromised VM on same cloud subnet, rogue container in a flat-network pod, or physical LAN access).
2. Attacker performs ARP cache poisoning to position themselves as MITM between the Rosetta process and the PostgreSQL server.
3. Attacker forwards all traffic normally until a `RetrieveBalanceAtBlock` call is observed (identifiable by the plaintext SQL query patterns `latestBalanceBeforeConsensus` / `selectPreviousPartitionLowerBound` in the wire stream).
4. Attacker drops the legitimate PostgreSQL response and injects a crafted response conforming to the PostgreSQL wire protocol (DataRow + CommandComplete messages) with an inflated `balance` value.
5. `RetrieveBalanceAtBlock` returns `hbarAmount.Value = ab.Balance + hbarValue` where both components are attacker-controlled.

**Why existing checks fail:** There are no integrity checks on query results, no response signing, and no TLS certificate validation. The application has no way to distinguish a legitimate DB response from a forged one because the transport is unauthenticated plaintext.

### Impact Explanation
An exchange or wallet using the Rosetta `/account/balance` endpoint receives a fabricated HBAR balance. If the system credits deposits based on this balance, funds that do not exist on-chain are credited to the attacker's account. This is a direct loss-of-funds scenario. Severity is **Critical** because the manipulated value flows directly into financial accounting with no secondary on-chain verification in the Rosetta layer.

### Likelihood Explanation
The precondition — L2 network adjacency — is non-trivial but realistic in:
- Shared cloud VPC environments where multiple tenants share a subnet
- Kubernetes clusters with flat pod networking and no NetworkPolicy
- Colocation facilities with shared switching infrastructure

ARP spoofing requires no privileges on the target systems, only the ability to send Ethernet frames on the shared segment. The attack is repeatable and scriptable using standard tools (`arpspoof`, `ettercap`, `scapy`). The PostgreSQL wire protocol is fully documented, making response forgery straightforward.

### Recommendation
1. **Remove the hardcoded `sslmode=disable`** from `GetDsn()` in `rosetta/app/config/types.go`.
2. **Add a `SslMode string` field to the `Db` config struct** (mirroring the importer's `DBProperties.java` pattern) with a secure default of `require` or `verify-full`.
3. **Update `GetDsn()`** to interpolate the configurable `SslMode` value.
4. **For production deployments**, require `sslmode=verify-full` with a trusted CA certificate to prevent both eavesdropping and server impersonation.

### Proof of Concept

```bash
# 1. Gain L2 adjacency (e.g., on same subnet as rosetta-app and postgres)

# 2. ARP poison: tell rosetta-app that postgres's IP is at attacker MAC
arpspoof -i eth0 -t <rosetta-app-ip> <postgres-ip> &
arpspoof -i eth0 -t <postgres-ip> <rosetta-app-ip> &

# 3. Enable IP forwarding to stay transparent
echo 1 > /proc/sys/net/ipv4/ip_forward

# 4. Use scapy or mitmproxy with a custom PostgreSQL dissector to:
#    - Detect DataRow responses to balance queries (wire protocol type 'D')
#    - Replace the 8-byte int64 balance field with 0x7FFFFFFFFFFFFFFF (max int64)
#    - Recalculate and patch the message length field
#    - Forward the forged packet to rosetta-app

# 5. Trigger a /account/balance Rosetta API call for the target account.
#    The response will show balance = 9223372036854775807 tinybars
#    instead of the real on-chain balance.
```
