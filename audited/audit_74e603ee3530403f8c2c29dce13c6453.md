### Title
Connection Pool Exhaustion via Unbounded DB Connections When `statementTimeout <= 0`

### Summary
When `statementTimeout` is configured as `<= 0`, `GetDbWithContext()` attaches no deadline to the database context, allowing queries to run indefinitely. Because the connection pool has a hard cap (`MaxOpenConnections`), an attacker who saturates the pool with concurrent long-running requests causes all subsequent requests to block indefinitely at connection acquisition, effectively denying service to the Rosetta API node without any authentication.

### Finding Description

**Exact code path:**

`rosetta/app/db/client.go`, `GetDbWithContext()`, lines 22–29:

```go
func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
    if d.statementTimeout <= 0 {          // ← branch taken when timeout disabled
        db := d.db
        if ctx != nil {
            db = db.WithContext(ctx)       // ← only the HTTP request ctx, no deadline added
        }
        return db, noop                   // ← noop cancel; no timeout enforced
    }
    ...
    childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
    return d.db.WithContext(childCtx), cancel
}
```

**Root cause:** The branch at line 23 is the intended "no statement timeout" path. It propagates only the caller's HTTP request context. If the HTTP server's `WriteTimeout` is also zero (or unset), the request context carries no deadline at all. Go's `database/sql` pool blocks new `Acquire()` calls when `MaxOpenConnections` is reached; it does not time out by default unless a context deadline is present.

**Connection pool cap** is set in `rosetta/app/db/db.go` line 33:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)
```

**Exploit flow:**
1. Attacker identifies that `statementTimeout` is `<= 0` (default or explicit config) and `Http.WriteTimeout` is `0`.
2. Attacker opens exactly `MaxOpenConnections` concurrent HTTP requests to any persistence-backed endpoint (e.g., `/block`, `/account/balance`), each triggering a query via `GetDbWithContext`.
3. Each goroutine acquires one pool connection and holds it while the query runs (or blocks in a slow query / large table scan).
4. The pool is now fully saturated. Every subsequent request calls `db.WithContext(ctx)` and then blocks inside `database/sql`'s `(*DB).conn()` waiting for a free slot.
5. Because no timeout exists on either the DB context or the HTTP layer, these waiting goroutines never unblock. The node is effectively down for all new traffic.

**Why existing checks are insufficient:**
- `SetConnMaxLifetime` recycles idle connections, not active ones — it does not help while connections are in use.
- HTTP-layer timeouts (`Http.WriteTimeout`) are optional and default to `0` in the `Http` struct; operators who disable `statementTimeout` often also leave HTTP timeouts unconfigured.
- There is no per-IP rate limiting, concurrency cap, or semaphore in the request path.

### Impact Explanation
A fully saturated pool means 100% of new API requests on the affected node block indefinitely. In a multi-node Rosetta deployment, an attacker targeting `≥ 30%` of nodes with this technique shuts down a significant fraction of network processing capacity without brute force. The attack is silent (no errors until callers time out on their side) and persistent until the attacker drops connections.

### Likelihood Explanation
- **No authentication required**: all Rosetta endpoints are public.
- **Low resource cost for attacker**: holding `MaxOpenConnections` (typically 10–100) HTTP connections open is trivial with any HTTP client supporting keep-alive or streaming.
- **Configuration dependency**: requires `statementTimeout <= 0` AND `Http.WriteTimeout == 0`. Both are plausible in production (operators disable statement timeouts to avoid killing legitimate long-running queries; HTTP timeouts are often left at zero to avoid cutting off slow clients).
- **Repeatable**: dropping and re-opening connections immediately re-saturates the pool.

### Recommendation
1. **Enforce a minimum context deadline** in the `statementTimeout <= 0` branch — either apply a configurable `maxQueryDuration` fallback or reject the zero-timeout configuration entirely.
2. **Set a non-zero `Http.WriteTimeout`** in the default configuration so the HTTP request context always carries a deadline, which propagates into `GetDbWithContext` and cancels stalled queries.
3. **Add a connection-wait timeout** via `db.WithContext` using a short deadline specifically for pool acquisition (separate from statement execution timeout).
4. **Implement concurrency limiting** (e.g., a semaphore or middleware) to cap the number of simultaneous in-flight DB-backed requests per node.

### Proof of Concept
```bash
# Assumes statementTimeout=0 and Http.WriteTimeout=0 on target node
# MAX_CONNS = configured MaxOpenConnections (e.g., 20)

for i in $(seq 1 $MAX_CONNS); do
  # Send a request that triggers a slow/blocking DB query; keep connection open
  curl -s -X POST http://<rosetta-node>:8082/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}' \
    --max-time 3600 &   # hold open for 1 hour
done

# Now send a legitimate request — it will block indefinitely:
time curl -s -X POST http://<rosetta-node>:8082/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
# Expected: hangs until attacker drops connections
