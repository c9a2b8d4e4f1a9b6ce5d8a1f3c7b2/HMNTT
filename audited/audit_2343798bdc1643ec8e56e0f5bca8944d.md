### Title
Ghost Slot Key Accumulation in `findStorageBatch()` Causes Unbounded DB Overhead on Every Legitimate Batch Query

### Summary
`findStorageBatch()` unconditionally stores every queried slot key in `contractSlotsCache` via `putIfAbsent(wrappedKey, EMPTY_VALUE)`, but never removes keys that the DB confirms do not exist. These ghost keys persist for up to 5 minutes (refreshed on each access) and are included in every subsequent `findStorageBatch` DB call for the same contract. An unprivileged attacker can fill the per-contract slot cache to its 1,500-key maximum with non-existent keys, forcing every legitimate batch query for that contract to carry up to 1,500 phantom entries into the database indefinitely.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorageBatch()`, lines 85–122.

```
Line 90:  contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
Line 91:  final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
Line 103: contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

**Root cause — failed assumption:**

The design assumes that keys stored with `EMPTY_VALUE` are placeholders for *pending* lookups that will be resolved and promoted to real values in `contractStateCache`. However, when the DB returns no row for a key (non-existent slot), nothing is ever written to `contractStateCache` for that key, and nothing removes the ghost key from `contractSlotsCache`. The code has no post-query cleanup step that evicts keys confirmed absent by the DB.

**Exploit flow:**

1. Attacker calls any public contract-storage query endpoint (e.g., `eth_getStorageAt` or `eth_call` touching storage) for a target `contractId`, supplying fabricated slot keys that do not exist on-chain.
2. Each call reaches `findStorageBatch()`. Line 90 stores the fabricated key with `EMPTY_VALUE` in the per-contract Caffeine cache (`slotsPerContract`, `expireAfterAccess=5m, maximumSize=1500`).
3. Line 103 issues `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (...)` — the DB returns zero rows for the ghost keys.
4. Because no value is written to `contractStateCache` for ghost keys, the next call for the same ghost key finds a `contractStateCache` miss and re-enters `findStorageBatch()`, where `putIfAbsent` refreshes the Caffeine access-time (Spring's `CaffeineCache.putIfAbsent` calls `cache.get(key, loader)` internally, which counts as an access).
5. After 1,500 fabricated keys are inserted (the `maximumSize` of `slotsPerContract`), every subsequent `findStorageBatch` call for any slot of that contract — including legitimate ones — sends a batch of up to 1,500 keys to the DB.
6. The attacker re-queries each ghost key once every <5 minutes to reset `expireAfterAccess`, keeping all 1,500 ghost keys alive indefinitely.

**Why existing checks are insufficient:**

- `maximumSize=1500` caps the ghost keys per contract but does not prevent the attack; it only bounds the overhead.
- `expireAfterAccess=5m` would naturally expire ghost keys, but the attacker's periodic re-queries reset the timer via `putIfAbsent` (which triggers a Caffeine read internally).
- `contractState` cache uses `expireAfterWrite=2s`, meaning legitimate callers re-enter `findStorageBatch()` every 2 seconds, so the inflated batch is sent to the DB at high frequency.
- The global RPS throttle (500 req/s) limits the rate of ghost-key injection but does not prevent accumulation or the sustained overhead once ghost keys are established.

### Impact Explanation

For each targeted contract, every legitimate `findStorageBatch` invocation issues a SQL `IN` clause with up to 1,500 ghost keys appended to the real keys. The DB must perform index scans for all 1,500 non-existent rows on every call. Because `contractStateCache` expires every 2 seconds, this inflated query fires continuously for any active contract. Across up to 3,000 contracts (the `contractSlots` outer cache `maximumSize`), the aggregate DB overhead is significant. The impact is sustained DB CPU and I/O waste (griefing), potentially degrading query latency for all users sharing the same mirror-node instance.

### Likelihood Explanation

The attack requires no authentication, no on-chain funds, and no special knowledge beyond a valid `contractId`. The attacker needs only to send ~1,500 HTTP requests per contract to fill the ghost-key cache, then ~1,500 requests every 5 minutes to keep them alive. This is well within the reach of any external caller and is fully repeatable. The public `eth_getStorageAt` RPC endpoint is the natural trigger surface.

### Recommendation

After `findStorageBatch` receives the DB results, evict from `contractSlotsCache` any key that was sent to the DB but returned no row:

```java
final Set<ByteBuffer> resolvedKeys = contractSlotValues.stream()
    .map(v -> ByteBuffer.wrap(v.getSlot()))
    .collect(Collectors.toSet());

for (var slot : cachedSlotKeys) {
    if (!resolvedKeys.contains(slot)) {
        contractSlotsCache.evict(slot);   // remove confirmed-absent ghost keys
    }
}
```

This ensures that non-existent slot keys are removed immediately after the DB confirms their absence, preventing indefinite accumulation. Alternatively, store a negative-result sentinel in `contractStateCache` (with a short TTL) so that repeated queries for non-existent slots short-circuit at the `contractStateCache` check in `findStorage()` and never reach `findStorageBatch()` again.
