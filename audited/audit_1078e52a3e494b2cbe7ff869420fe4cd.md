### Title
Indefinite gRPC Subscription via `Long.MAX_VALUE` End Time Causes Unbounded Resource Exhaustion

### Summary
An unprivileged external user can send a `ConsensusTopicQuery` with `consensusEndTime.seconds >= 9223372035L`, causing `convertTimestamp()` to return `Long.MAX_VALUE` as the filter's `endTime`. This value is interpreted by `TopicContext.isComplete()` as a timestamp in the year ~2262, so the completion check never returns `true`, and the gRPC subscription is held open indefinitely. An attacker opening many such connections exhausts server-side resources (threads, memory, DB polling connections) without any authentication requirement.

### Finding Description

**Step 1 — Timestamp conversion produces `Long.MAX_VALUE`**

In `ConsensusController.java`, `convertTimestamp()` explicitly returns `Long.MAX_VALUE` for any timestamp with `seconds >= 9223372035L`:

```java
// ConsensusController.java lines 76-81
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
``` [1](#0-0) 

`toFilter()` passes this value directly into the filter's `endTime` field with no further clamping: [2](#0-1) 

**Step 2 — Validation passes**

`TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`. Since `Long.MAX_VALUE` is always greater than any current nanosecond timestamp, the `@AssertTrue` constraint is satisfied and the filter is accepted: [3](#0-2) 

**Step 3 — `isComplete()` never returns `true`**

`TopicContext.isComplete()` converts `endTime` to an `Instant` via `Instant.ofEpochSecond(0, Long.MAX_VALUE)`, which resolves to approximately the year 2262. Adding the 30-second `endTimeInterval` still yields a far-future instant, so `.isBefore(Instant.now())` is permanently `false`:

```java
// TopicMessageServiceImpl.java lines 203-215
boolean isComplete() {
    if (filter.getEndTime() == null) { return false; }
    if (filter.getEndTime() < startTime) { return true; }
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());   // always false with Long.MAX_VALUE
}
``` [4](#0-3) 

**Step 4 — `pastEndTime()` never signals termination**

`pastEndTime()` is the only mechanism that can terminate the live-message flux via `takeUntilOther`. It emits a completion signal only when `isComplete()` returns `true`. Since that never happens, the `RepeatSpec` predicate `!topicContext.isComplete()` is always `true`, the flux repeats indefinitely without emitting, and `takeUntilOther` never fires: [5](#0-4) 

**Step 5 — `takeWhile` on `endTime` also never terminates**

The secondary guard `takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime())` is permanently `true` because real consensus timestamps (current epoch nanoseconds, ~`1.7 × 10¹⁸`) are always less than `Long.MAX_VALUE` (`9.2 × 10¹⁸`): [6](#0-5) 

### Impact Explanation

Each malicious subscription holds a live gRPC stream open indefinitely. The stream drives:
- A `PollingTopicMessageRetriever` that polls the database on a `boundedElastic` scheduler thread at `pollingFrequency` intervals with `numRepeats = Long.MAX_VALUE`
- A `safetyCheck` flux on another `boundedElastic` thread
- An active entry in the `subscriberCount` gauge (no enforcement cap)

The only per-connection limit is `maxConcurrentCallsPerConnection = 5` (from `NettyProperties`), but an attacker can open arbitrarily many TCP connections, each carrying 5 such subscriptions. There is no global subscriber cap, no per-IP rate limit, and no authentication requirement. Thread pool exhaustion, memory pressure, and database connection saturation are all realistic outcomes, constituting a practical denial-of-service against the mirror node's gRPC service. [7](#0-6) 

### Likelihood Explanation

The exploit requires zero privileges — the gRPC `subscribeTopic` endpoint is publicly accessible. The trigger is a single protobuf field set to a value >= `9223372035` seconds. The attacker needs no knowledge of valid topic IDs (an invalid topic ID with `checkTopicExists = false` still opens the subscription). The attack is trivially scriptable, repeatable, and can be parallelized across many connections from a single host.

### Recommendation

1. **Cap `endTime` to a reasonable future horizon** inside `convertTimestamp()` or `toFilter()` — e.g., reject or clamp any `endTime` more than N hours/days in the future relative to `Instant.now()`.
2. **Fix `isComplete()` to handle `Long.MAX_VALUE` explicitly** — treat `Long.MAX_VALUE` as "no end time" (return `false` from `isComplete()` and skip `takeWhile`) or map it to `null` before storing in the filter.
3. **Enforce a global maximum concurrent subscriber count** — add a hard cap in `subscribeTopic()` that rejects new subscriptions when the limit is reached, returning `RESOURCE_EXHAUSTED`.
4. **Add per-IP or per-connection subscription rate limiting** at the Netty/gRPC interceptor layer.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Set consensusEndTime.seconds >= 9223372035 to trigger Long.MAX_VALUE path
end_ts = Timestamp(seconds=9223372035, nanos=0)

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    consensusEndTime=end_ts,
)

# Each call opens a subscription that never terminates
# Repeat in a loop across many connections to exhaust server resources
for msg in stub.subscribeTopic(query):
    pass  # stream never completes
```

Repeat across N parallel connections × 5 calls each. Monitor server thread pool and DB connection pool exhaustion.
