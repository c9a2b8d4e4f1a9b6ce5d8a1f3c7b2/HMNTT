### Title
Unauthenticated DB Connection Pool Exhaustion via Opaque EVM Address in `getTokenRelationships`

### Summary
An unprivileged attacker can supply any valid 40-hex-character opaque EVM address as the `idOrAliasOrEvmAddress` path parameter to `/api/v1/accounts/{id}/tokens`. This passes `isValidEntityId()` validation and causes `getEntityIdFromEvmAddress()` to execute a live DB query on every request, with no caching of the DB result. Because the default connection pool is only 10 connections and no rate limiting exists, as few as 10 concurrent requests fully exhaust the pool, making the REST API unresponsive to all users.

### Finding Description

**Exact code path:**

1. `getTokenRelationships` (tokenController.js:67) calls `EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS])` with no pre-validation or rate limiting.

2. `getEncodedId` (entityService.js:120) calls `EntityId.isValidEntityId(entityIdString)`. In `entityId.js:133-139`, this returns `true` for any string matching `evmAddressRegex = /^(0x)?[A-Fa-f0-9]{40}$/` — a 40-hex-char string like `71eaa748d5252be68c1185588beca495459fdba4` passes unconditionally.

3. `EntityId.parseString` calls `parseCached` (entityId.js:314-333). `parseCached` caches only the parsed `EntityId` object (keyed by the input string). For an opaque EVM address, `parseFromString` returns `[shard, realm, null, numOrEvmAddress]` (entityId.js:253), so `entityId.evmAddress` is non-null.

4. Back in `getEncodedId` (entityService.js:122-124), because `entityId.evmAddress !== null`, it always calls `await this.getEntityIdFromEvmAddress(entityId, requireResult)` — the DB lookup is **never cached**.

5. `getEntityIdFromEvmAddress` (entityService.js:91) executes:
   ```js
   const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
     [Buffer.from(entityId.evmAddress, 'hex')]);
   ```
   This issues a live SQL query (`SELECT id FROM entity WHERE deleted <> true AND evm_address = $1`) against the DB on **every single request**, regardless of whether the same address was queried before.

**Root cause:** The `parseCached` cache (entityId.js:301-304) caches only the syntactic parse result, not the DB lookup outcome. There is no second-level cache for the `getEntityIdFromEvmAddress` result. Combined with the absence of any rate-limiting middleware in `server.js` (confirmed: no `rateLimit`/`throttle` middleware registered), every HTTP request with an opaque EVM address unconditionally consumes a DB connection.

**Why existing checks fail:**
- `isValidEntityId` correctly accepts opaque EVM addresses by design — it cannot distinguish a DoS probe from a legitimate lookup.
- `parseCached` caches the EntityId object but the DB round-trip in `getEntityIdFromEvmAddress` is outside its scope.
- The `statementTimeout` of 20,000 ms (docs/configuration.md:557) means each connection is held for up to 20 seconds per query.
- The DB pool `maxConnections` defaults to **10** (docs/configuration.md:556, dbpool.js:14).

### Impact Explanation
With a pool of 10 connections and a 20-second statement timeout, an attacker needs only 10 concurrent HTTP GET requests to fully exhaust the pool. All subsequent requests from legitimate users queue and time out after `connectionTimeoutMillis` (20,000 ms, docs/configuration.md:555), rendering the REST API completely unresponsive. Since the mirror node REST API is the primary read interface for dApps and ecosystem tooling, sustained exhaustion degrades or halts ≥30% of mirror node processing capacity without any brute-force or privileged access.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and only a standard HTTP client. A single attacker with a modest script sending 10–20 concurrent requests in a loop can sustain the outage indefinitely. The endpoint is publicly reachable, the EVM address format is well-documented, and the pool size is small enough that even accidental traffic spikes could trigger the condition. Repeatability is trivial.

### Recommendation
1. **Cache the DB lookup result**: In `getEntityIdFromEvmAddress`, cache the `(evmAddress → entityId)` mapping using the existing `parseCached`-style LRU cache or a dedicated cache, so repeated lookups for the same address do not hit the DB.
2. **Add rate limiting**: Register a per-IP rate-limiting middleware (e.g., `express-rate-limit`) in `server.js` before route handlers, specifically targeting the `/accounts/:id/tokens` and similar EVM-address-accepting endpoints.
3. **Increase pool size or add a request queue cap**: Raise `maxConnections` and add a maximum in-flight request limit so pool exhaustion cannot be triggered by a small number of concurrent connections.
4. **Short-circuit on non-existent EVM addresses**: Consider a negative cache (TTL-bounded) for EVM addresses that returned no rows, to avoid repeated DB hits for the same non-existent address.

### Proof of Concept

```bash
# Generate a valid opaque EVM address (non-parsable: first 12 bytes are NOT all zeros)
EVM="71eaa748d5252be68c1185588beca495459fdba4"
TARGET="http://<mirror-node-host>:5551/api/v1/accounts/${EVM}/tokens"

# Send 20 concurrent requests in a loop to exhaust the 10-connection pool
for i in $(seq 1 20); do
  curl -s "$TARGET" &
done
wait

# Legitimate request now times out or receives a connection error:
curl -v "http://<mirror-node-host>:5551/api/v1/accounts/0.0.1234/tokens"
# Expected: hangs for 20s then connection timeout
```

Each of the 20 concurrent requests enters `getEntityIdFromEvmAddress` and blocks on `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1`, holding a pool connection for up to 20 seconds. With only 10 pool slots, all legitimate traffic is queued and times out.
