### Title
Unbounded High-Frequency Pagination Requests Exhaust DB Connection Pool in `getNftAllowances()`

### Summary
The `getNftAllowances()` endpoint in `AllowancesController` accepts `limit=1` with no per-client rate limiting in the `rest-java` module. An unprivileged attacker can issue a sustained flood of requests with `limit=1`, following each pagination `next` link, causing one DB query per request against the `nft_allowance` table. With sufficient concurrency this exhausts the finite HikariCP connection pool, degrading or denying service to legitimate users.

### Finding Description
**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, `getNftAllowances()`, line 61:
```java
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
```
`@Positive` enforces `limit >= 1`; `@Max(MAX_LIMIT)` caps at 100. `limit=1` is fully valid.

**Query path per request:** `getNftAllowances()` → `NftAllowanceServiceImpl.getNftAllowances()` → `NftAllowanceRepositoryCustomImpl.findAll()` → jOOQ `SELECT … FROM nft_allowance WHERE … LIMIT ?` executed synchronously, holding a HikariCP connection for the query duration.

**Pagination amplification:** `linkFactory.create()` (line 80) generates a `next` link encoding cursor parameters (`account.id=gte:X&token.id=gt:Y`). With `limit=1` the attacker receives one row and a new `next` link per request, enabling sequential full-table traversal one row at a time.

**No rate limiting in rest-java:** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleManager`) exists exclusively in the `web3` module. `AllowancesController` injects only `LinkFactory`, `NftAllowanceService`, and `NftAllowanceMapper` — no throttle bean. The constant `HIGH_VOLUME_THROTTLE` defined in `Constants.java` line 19 is never referenced in `AllowancesController`. No Spring Security filter, servlet filter, or interceptor applies per-IP or global rate limiting to this endpoint in the `rest-java` module.

**DB pool:** `CommonConfiguration` wires a HikariCP `DataSource` via `@ConfigurationProperties("spring.datasource.hikari")`. The rest-java Grafana dashboard (`hedera-mirror-rest-java.json`) monitors `hikaricp_connections_active/max` and fires a `RestJavaHighDBConnections` alert at 75% pool utilization, confirming the pool is the finite shared resource.

### Impact Explanation
An attacker sending N concurrent `limit=1` requests keeps N DB connections occupied simultaneously. Once the HikariCP pool is saturated, all subsequent requests — including legitimate ones — block waiting for a connection up to the connection-acquisition timeout, then fail. This constitutes a griefing denial-of-service against the mirror node's REST Java API with no economic cost to the attacker and no direct on-chain economic damage, matching the medium-severity "griefing" classification.

### Likelihood Explanation
No authentication is required. Any public internet client can reach `/api/v1/accounts/{id}/allowances/nfts`. The attack requires only a single machine issuing concurrent HTTP/1.1 or HTTP/2 requests; no botnet is needed. The pagination `next` link is machine-readable and trivially scriptable. Accounts with large NFT allowance sets (common for DeFi/marketplace operators) maximize the number of valid pages available. The attack is repeatable indefinitely.

### Recommendation
1. **Add per-IP rate limiting** to the `rest-java` module analogous to the `web3` `ThrottleManagerImpl` / `ThrottleConfiguration` pattern, applied globally via a servlet filter or Spring MVC interceptor.
2. **Enforce a minimum effective limit** — consider rejecting `limit < N` (e.g., `limit < 5`) or adding a cost-based throttle that charges more tokens for low-limit paginated requests.
3. **Set a DB statement timeout** for the rest-java datasource (e.g., `spring.datasource.hikari.connection-init-sql` or a jOOQ `ExecuteListener`) to bound the maximum connection hold time per query.
4. **Use the existing `HIGH_VOLUME_THROTTLE` constant** (defined but unused in `Constants.java` line 19) as the hook to implement endpoint-level throttling in `AllowancesController`.

### Proof of Concept
```bash
# Step 1: obtain first page (limit=1)
curl "https://<mirror-node>/api/v1/accounts/0.0.1000/allowances/nfts?limit=1&order=asc"
# Response contains: "links": {"next": "/api/v1/accounts/0.0.1000/allowances/nfts?limit=1&account.id=gte:0.0.X&token.id=gt:0.0.Y"}

# Step 2: script concurrent pagination followers (bash example)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1000/allowances/nfts?limit=1&account.id=gte:0.0.$i&token.id=gt:0.0.0" &
done
wait

# Step 3: observe HikariCP pool saturation via metrics
# GET /actuator/metrics/hikaricp.connections.active  →  value approaches hikaricp.connections.max
# Legitimate requests begin returning 503 / connection timeout errors
```
No credentials, special headers, or privileged account required.
