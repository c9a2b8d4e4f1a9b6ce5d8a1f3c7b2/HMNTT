### Title
Unbounded IN Clause via Unlimited Keys Collection in `getHookStorage()` Enables Database Query DoS

### Summary
`HookServiceImpl.getHookStorage()` passes the result of `getKeysInRange()` directly to `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse()` without any cap on the number of keys. Since neither `HookStorageRequest` nor `getKeysInRange()` enforces a maximum key count, an unprivileged caller can supply an arbitrarily large collection of keys that all satisfy the range filter, causing Spring Data JPA to emit a SQL `IN (?, ?, …)` clause with thousands of bind parameters, degrading database performance and causing API-level denial of service.

### Finding Description
**File:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, lines 72–79
**DTO:** `rest-java/src/main/java/org/hiero/mirror/restjava/dto/HookStorageRequest.java`, lines 43–51

Code path:
1. `HookStorageRequest.keys` is a `Collection<byte[]>` with no `@Size` or other cardinality constraint. [1](#0-0) 
2. `getKeysInRange()` streams `keys` and filters only by `keyLowerBound`/`keyUpperBound` comparison — it does **not** limit the resulting list size. [2](#0-1) 
3. The full, unbounded `keysInRange` list is forwarded directly to the repository: [3](#0-2) 
4. `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse()` is a Spring Data derived query that translates the list into a SQL `IN (…)` clause with one bind parameter per key — no server-side truncation occurs. [4](#0-3) 
5. The `limit` field (default 25) only controls the `Pageable` result-row cap; it has no effect on the size of the `IN` clause sent to the database. [5](#0-4) 

A codebase-wide search for `maxKeys`, `MAX_KEYS`, `keys.size`, and `keys.*limit` returned zero matches, confirming no guard exists anywhere in the call chain.

**Root cause / failed assumption:** The design assumes callers supply a small, bounded set of keys (matching the `limit` intent), but nothing enforces this. The range filter is not a cardinality filter.

### Impact Explanation
An attacker can craft a single HTTP request containing tens of thousands of keys, all within the accepted range, forcing the database to evaluate a massive `IN` clause. This can:
- Saturate database CPU/memory for the duration of the query.
- Block or slow concurrent legitimate queries sharing the same connection pool.
- Be repeated in parallel to sustain a denial-of-service condition against the mirror node REST API.

The impact is availability degradation of the mirror node read API, not a direct ledger/consensus impact (this is a read-only mirror service, so "block time" framing in the question is inapplicable, but API DoS is a concrete, real impact).

### Likelihood Explanation
No authentication is required to call the hook storage endpoint — any unprivileged external user can reach it. The attack requires only a crafted HTTP request with a large array of keys; no special knowledge, credentials, or tooling beyond a standard HTTP client is needed. It is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a hard cap on `keys` collection size** in `HookStorageRequest`, e.g., add a `@Size(max = 100)` (or a domain-appropriate constant) Bean Validation constraint on the `keys` field.
2. **Validate in the controller** before building the DTO, rejecting requests that exceed the maximum key count with HTTP 400.
3. **Optionally**, truncate `keysInRange` to `limit` entries before passing to the repository, since the result page is already bounded by `limit`.

### Proof of Concept
```
POST /api/v1/hooks/{ownerId}/storage?hookId=1
Content-Type: application/json

{
  "keyLowerBound": "0x00",
  "keyUpperBound": "0xFF...FF",
  "keys": [
    "0x0001", "0x0002", "0x0003", ... /* 50,000 entries, all within range */
  ]
}
```
Spring Data JPA emits:
```sql
SELECT * FROM hook_storage
WHERE owner_id = ? AND hook_id = ? AND key IN (?,?,?,.../* 50,000 params */)
  AND deleted IS FALSE
ORDER BY key ASC LIMIT 25;
```
The database must evaluate the full IN list regardless of the LIMIT clause, causing query-time resource exhaustion.

        if (keys.isEmpty()) {
            return List.of();
        }

        return keys.stream()
                .filter(key -> Arrays.compareUnsigned(key, keyLowerBound) >= 0
                        && Arrays.compareUnsigned(key, keyUpperBound) <= 0)
                .toList();
