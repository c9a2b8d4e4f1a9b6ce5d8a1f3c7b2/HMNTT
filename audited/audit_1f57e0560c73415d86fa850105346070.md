### Title
Unprivileged Access to Internal Contract Call Results via `internal=true` Query Parameter

### Summary
The mirror node REST API exposes a `internal` query parameter on the `/api/v1/contracts/results` and `/api/v1/contracts/{id}/results` endpoints that is accepted from any unauthenticated caller. When set to `true`, the server omits the `transaction_nonce = 0` SQL filter that normally excludes internal sub-call records, returning all internal contract execution results â€” including `call_result`, `function_parameters`, `function_result`, and `error_message` â€” to any external user without any privilege check.

### Finding Description
**Code location:** `rest/controllers/contractController.js`, lines 425, 478â€“480, 531â€“533.

`filterKeys.INTERNAL` is listed in `acceptedContractResultsParameters` (line 1341) with no special guard, so `utils.buildAndValidateFilters` accepts it from any caller. Inside `extractContractResultsByIdQuery`, the variable `internal` defaults to `false` (line 425). The filter loop at lines 478â€“480 unconditionally assigns the user-supplied boolean:

```js
case filterKeys.INTERNAL:
  internal = filter.value;   // attacker controls this
  break;
```

The only gate that separates public from internal records is:

```js
if (!internal) {
  conditions.push(`${ContractResult.getFullName(ContractResult.TRANSACTION_NONCE)} = 0`);
}
```

When `internal=true` is supplied, this `WHERE transaction_nonce = 0` clause is never appended, so the SQL query returns every row regardless of nonce â€” including all internal sub-calls (nonce > 0). There is no authentication middleware, role check, or IP restriction applied to this parameter anywhere in the controller or routing layer.

The test fixture `rest/__tests__/specs/contracts/{id}/results/internal-true.json` explicitly confirms that records with `transaction_nonce: 1` and `transaction_nonce: 2` (internal sub-calls) are returned when `internal=true` is passed, while `internal-false.json` shows only `transaction_nonce: 0` records are returned by default.

**Root cause:** The `internal` filter key is treated as a plain user-controlled boolean with no authorization gate. The design assumes only privileged callers will set it to `true`, but no enforcement of that assumption exists.

### Impact Explanation
Any anonymous HTTP client can retrieve internal contract sub-call records that are intentionally hidden from the default API response. These records contain `call_result`, `function_parameters`, `function_result`, `error_message`, `sender_id`, and `gas_consumed` for every internal call within a transaction. This leaks the full internal execution trace of smart contracts, which may include sensitive business logic, intermediate state, failed sub-calls with revealing error messages, and data that contract authors expect to remain opaque. The scope is network-wide: both the per-contract endpoint and the global `/api/v1/contracts/results` endpoint are affected.

### Likelihood Explanation
Exploitation requires zero privileges, zero special tooling, and zero prior knowledge beyond the public OpenAPI spec (which documents the `internal` parameter at `rest/api/v1/openapi.yml` lines 570 and 650). Any user who reads the API documentation or fuzzes query parameters can trigger this. It is trivially repeatable and automatable.

### Recommendation
Restrict the `internal=true` path to authenticated/privileged callers only. Options:
1. Remove `filterKeys.INTERNAL` from `acceptedContractResultsParameters` entirely if public exposure is unintended.
2. Add a middleware or in-handler check that rejects `internal=true` unless the request carries a valid operator/admin credential.
3. If the parameter must remain public, document the intentional exposure and ensure no sensitive data is present in internal sub-call records.
