### Title
Connection Pool Exhaustion via Unauthenticated Concurrent Requests to `getSchedules()` Three-Query Pattern

### Summary
The `getSchedules()` function in `rest/schedules.js` issues up to three database queries per request — one sequential `schedulesQuery` followed by two parallel queries (`entityQuery` + `signatureQuery`) via `Promise.all` — consuming up to three connections from a shared pool whose default size is only 10. With no application-level rate limiting on the REST API, an unprivileged external attacker can send as few as 3–4 concurrent requests to `/api/v1/schedules` to exhaust the entire connection pool, causing all REST API endpoints to fail with connection timeout errors for the duration of the attack.

### Finding Description
**Exact code path:**

In `rest/schedules.js`, `getSchedules()` (lines 234–282) performs:
1. Line 241: `await pool.queryQuietly(schedulesQuery, params)` — acquires one connection, holds it until the query completes.
2. Lines 261–264: `await Promise.all([pool.queryQuietly(entityQuery, entityIds), pool.queryQuietly(signatureQuery, entityIds)])` — acquires two more connections simultaneously.

Each request therefore holds up to **3 connections** concurrently during the `Promise.all` phase.

**Root cause:**

The DB pool is configured in `rest/dbpool.js` (line 14) with `max: config.db.pool.maxConnections`. The documented default (`docs/configuration.md`, line 556) is `maxConnections = 10`. With 3 connections consumed per in-flight `getSchedules()` request, only **⌊10/3⌋ = 3 concurrent requests** are needed to exhaust the pool. A fourth concurrent request will queue and wait up to `connectionTimeoutMillis` (default 20,000 ms, `dbpool.js` line 13) before failing.

**No application-level rate limiting exists.** Reviewing `rest/server.js` (lines 68–144), the middleware stack is: `express.urlencoded`, `express.json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, route handlers, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. There is no rate-limiting or in-flight-request-limiting middleware. A `grep` across all `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns zero matches in production code.

**Why existing checks fail:**

- `statement_timeout` (default 20,000 ms): Limits query duration but does not prevent pool exhaustion — connections are held for the full timeout window, sustaining the attack.
- `connectionTimeoutMillis` (default 20,000 ms): Causes queued requests to fail after 20 s, but does not reject attacker requests or protect legitimate users.
- `authHandler`: The `/schedules` endpoint requires no authentication; `authHandler` only enforces credentials when `users` config is populated, which is not the default.
- The shared `global.pool` is used by **all** REST API endpoints, so exhaustion of the pool by `/schedules` requests denies service to every other endpoint.

### Impact Explanation
Exhausting the 10-connection pool blocks all database-backed REST API endpoints (accounts, balances, contracts, tokens, transactions, schedules, topics) for the duration of the attack. Since the mirror node REST API is the primary read interface for the Hedera network's state, sustained pool exhaustion constitutes a denial-of-service against the mirror node's processing capacity. The attack degrades ≥30% of the mirror node's ability to serve network state queries without requiring any brute-force volume — only 3–4 sustained concurrent HTTP requests suffice.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no sophisticated tooling — a simple script sending 4 concurrent `GET /api/v1/schedules` requests in a loop is sufficient. The endpoint is publicly accessible. The attacker can sustain the attack indefinitely by keeping connections open (slow queries, large result sets via `limit` parameter) or by continuously re-issuing requests. The low connection pool default (10) makes this trivially reproducible from a single machine.

### Recommendation
1. **Add application-level rate limiting** to the REST API server (e.g., `express-rate-limit` or similar) before route handlers in `rest/server.js`, limiting requests per IP per second to a safe threshold.
2. **Add a per-endpoint in-flight request cap** to prevent a single endpoint from monopolizing the pool.
3. **Increase `maxConnections`** to a value that reflects realistic concurrency, or use a connection-per-request model with a queue depth limit.
4. **Separate DB pools per endpoint group** so that one endpoint class cannot starve others.
5. **Enforce infrastructure-level rate limiting** (e.g., Traefik `inFlightReq` + `rateLimit` middleware, as already done for the Rosetta API in `charts/hedera-mirror-rosetta/values.yaml` lines 152–161) for the REST API Helm chart as well.

### Proof of Concept
```bash
# Requires: curl, GNU parallel or xargs
# Send 5 concurrent long-running requests to exhaust the 10-connection pool
# (each holds 3 connections; 4 requests = 12 > 10)

for i in $(seq 1 5); do
  curl -s "https://<mirror-node-host>/api/v1/schedules?limit=100" &
done
wait

# Immediately after, attempt a legitimate request to any endpoint:
curl -v "https://<mirror-node-host>/api/v1/transactions"
# Expected result: connection timeout or 500 error due to pool exhaustion
# ("timeout exceeded when trying to connect" from pg pool)
```

The attack is sustained by keeping the 5 background `curl` processes alive (they will each hold connections for up to 20 s per the `statement_timeout`). Re-issuing the loop before the timeout expires maintains continuous pool exhaustion. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

const getSchedules = async (req, res) => {
  // extract filters from query param
  const filters = utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters);

  // get sql filter query, params, order and limit from query filters
  const {filterQuery, params, order, limit} = extractSqlFromScheduleFilters(filters);
  const schedulesQuery = getSchedulesQuery(filterQuery, order, params.length);
  const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;
  }

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)
    .join(',');
  const entityQuery = `select ${entityFields} from entity where id in (${positions}) order by id ${order}`;
  const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),
    pool.queryQuietly(signatureQuery, entityIds),
  ]);
