### Title
Unauthenticated Full Historical Scan via Unbounded `timestamp=lte:X` on `/contracts/:contractId/state`

### Summary
The `/contracts/:contractId/state` endpoint accepts a `timestamp=lte:X` query parameter that, when set to a very large nanosecond value, causes `contractStateTimestampQuery` to execute a `DISTINCT ON (slot)` scan over the entire `contract_state_change` table for that contract. Unlike other contract endpoints, this path applies no timestamp range bounding or `optimizeTimestampFilters` call, and the `LIMIT` clause is applied only after the full `DISTINCT ON` sort completes. Any unauthenticated user can trigger this repeatedly to drive sustained DB I/O and CPU load.

### Finding Description

**Exact code path:**

`getContractStateById` (controller) → `extractContractStateByIdQuery` → `getContractStateByIdAndFilters` (service) → `contractStateTimestampQuery`

**`contractStateTimestampQuery`** (`rest/service/contractService.js`, lines 111–122): [1](#0-0) 

This query uses `DISTINCT ON (slot)` with `ORDER BY slot ASC, consensus_timestamp DESC`. PostgreSQL must scan and sort **all** rows for the contract up to the timestamp before it can apply `DISTINCT ON` and only then apply `LIMIT`. The `LIMIT` clause does not reduce the internal scan cost.

**`getContractStateByIdAndFilters`** (`rest/service/contractService.js`, lines 245–267): [2](#0-1) 

When `timestamp=true`, the function switches to `contractStateTimestampQuery` with the user-supplied `consensus_timestamp <= $N` condition directly injected.

**`extractContractStateByIdQuery`** (`rest/controllers/contractController.js`, lines 894–956): [3](#0-2) 

This function processes the timestamp filter at lines 909–918 and sets `timestampPresent = true`, but **never calls `optimizeTimestampFilters()`** or `bindTimestampRange()`. Compare this to `extractContractResultsByIdQuery` which routes timestamp filters through `optimizeTimestampFilters` (lines 384–404): [4](#0-3) 

**Timestamp validation** (`rest/utils.js`, lines 151–154): [5](#0-4) 

`isValidTimestampParam` only validates format (`/^\d{1,10}$/`), accepting values up to `9999999999.999999999` — far beyond any real network timestamp. No upper-bound semantic check exists.

**Root cause:** The contract state endpoint is the only contract endpoint that (a) uses a `DISTINCT ON` query requiring a full pre-sort scan and (b) applies no timestamp range bounding, allowing an attacker to set the upper bound to the maximum valid timestamp and force a complete historical scan.

### Impact Explanation

For a contract with a large history in `contract_state_change`, a single request with `timestamp=lte:9999999999` forces PostgreSQL to:
1. Perform a full index/sequential scan of all rows for that contract
2. Sort by `(slot, consensus_timestamp DESC)` across the entire result set
3. Apply `DISTINCT ON` before returning any rows to the `LIMIT` clause

The DB `statement_timeout` is set to 20,000 ms for the REST user: [6](#0-5) 

This means each malicious request can consume up to 20 seconds of DB CPU and I/O before being killed. With a default pool of 10 max connections, an attacker sending 10 concurrent requests can hold all connections in expensive scans simultaneously, causing legitimate requests to queue or time out. This directly satisfies the ">30% resource increase" threshold through sustained, repeated requests — no brute force needed, just a handful of concurrent HTTP calls.

### Likelihood Explanation

- **No authentication required**: The endpoint is fully public.
- **No rate limiting** visible in the REST layer for this specific endpoint.
- **Trivially repeatable**: A single `curl` loop or a small script sending concurrent requests is sufficient.
- **Targeting**: An attacker can identify high-activity contracts (e.g., popular DeFi contracts with millions of state changes) from public blockchain data to maximize scan cost per request.
- **Amplification**: The `DISTINCT ON` pattern means even a `limit=1` request still triggers the full scan.

### Recommendation

1. **Apply timestamp range bounding** in `extractContractStateByIdQuery` analogous to `optimizeTimestampFilters` used by other contract endpoints. Reject or clamp `lte` timestamps that exceed `now()` or a configurable maximum range.
2. **Enforce a maximum timestamp value**: Reject timestamps beyond the current network time plus a small buffer (e.g., reject `timestamp > now() + 1s`).
3. **Consider query restructuring**: Rewrite `contractStateTimestampQuery` to use a lateral join or CTE with per-slot limits so PostgreSQL can use index scans per slot rather than a full sort.
4. **Add per-endpoint rate limiting** for the `/contracts/:contractId/state` endpoint with timestamp filters.
5. **Reduce `statement_timeout`** for this query class, or add a dedicated DB role with a tighter timeout for historical state queries.

### Proof of Concept

```bash
# Identify a high-activity contract (e.g., 0.0.1234567)
# Send 10 concurrent requests with maximum timestamp to saturate DB connections

for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234567/state?timestamp=lte:9999999999.999999999&limit=1" &
done
wait
```

Each request triggers:
```sql
SELECT DISTINCT ON (slot) contract_id, slot, evm_address,
       coalesce(value_written, value_read) AS value,
       consensus_timestamp AS modified_timestamp
FROM contract_state_change
LEFT JOIN entity ON entity.id = contract_state_change.contract_id
WHERE contract_id = <id> AND consensus_timestamp <= 9999999999999999999
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT 25
```

PostgreSQL must scan and sort the entire `contract_state_change` history for the contract before returning 25 rows. With 10 concurrent requests, all DB pool connections are held for up to 20 seconds each, blocking legitimate traffic and driving DB CPU/I/O above the 30% threshold.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};

  async extractContractStateByIdQuery(filters, contractId) {
    let limit = defaultLimit;
    let order = orderFilterValues.ASC;
    let timestampPresent = false;
    const conditions = [this.getFilterWhereCondition(ContractState.CONTRACT_ID, {operator: '=', value: contractId})];
    const slotFilters = [];

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        case filterKeys.TIMESTAMP:
          if (utils.opsMap.ne === filter.operator) {
            throw new InvalidArgumentError(`Not equals (ne) operator is not supported for ${filterKeys.TIMESTAMP}`);
          }

          if (utils.opsMap.eq === filter.operator) {
            filter.operator = utils.opsMap.lte;
          }
          conditions.push(this.getFilterWhereCondition(ContractStateChange.CONSENSUS_TIMESTAMP, filter));
          timestampPresent = true;
          break;
        case filterKeys.SLOT:
          slotFilters.push(filter);
          break;
        default:
          break;
      }
    }

    const slotInValues = [];
    for (const slotFilter of slotFilters) {
      // Left pad the slot value if no timestamp filter is present
      // If a timestamp filter is present the slot value needs additional conversion
      // because there is an inconsistency between the column slot in contract_state and contract_state_change.
      const slot = utils.formatSlot(slotFilter.value, !timestampPresent);
      if (slotFilter.operator === utils.opsMap.eq) {
        slotInValues.push(slot);
      } else {
        conditions.push(
          this.getFilterWhereCondition(ContractState.SLOT, {
            operator: slotFilter.operator,
            value: slot,
          })
        );
      }
    }

    if (slotInValues.length !== 0) {
      conditions.push(this.getFilterWhereCondition(ContractState.SLOT, {operator: 'in', value: slotInValues}));
    }

    return {
      conditions,
      order,
      limit,
      timestamp: timestampPresent,
    };
  }
