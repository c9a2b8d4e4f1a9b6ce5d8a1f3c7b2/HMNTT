### Title
Unauthenticated Bandwidth and CPU Exhaustion via Unbounded Repeated Requests to `getNftAllowances()`

### Summary
The `getNftAllowances()` endpoint in `AllowancesController.java` is publicly accessible with no authentication and no rate limiting in the `rest-java` module. Any unprivileged attacker can repeatedly issue requests with `limit=100` (the enforced maximum) and broad range filters, causing the service to execute full database queries and serialize up to 100 NFT allowances per response on every call, exhausting network bandwidth and serialization CPU.

### Finding Description
**Exact code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, `getNftAllowances()`, lines 57–83; `rest-java/src/main/java/org/hiero/mirror/restjava/common/Constants.java`, lines 34–35.

**Root cause:** The `limit` parameter is bounded only by `@Max(MAX_LIMIT)` (line 61 of `AllowancesController.java`), where `MAX_LIMIT = 100` (line 35 of `Constants.java`). This is a per-request cap, not a rate limit. There is no rate-limiting infrastructure in the `rest-java` module at all — the `ThrottleConfiguration`, `ThrottleManagerImpl`, and `bucket4j` machinery exist exclusively in the `web3` module. The `RestJavaProperties` class only configures response headers. The `HIGH_VOLUME_THROTTLE` constant defined in `Constants.java` (line 19) is referenced only in `NetworkController.java` and fee-estimation services, never in `AllowancesController`.

**Exploit flow:**
1. Attacker (no credentials needed) sends: `GET /api/v1/accounts/0.0.X/allowances/nfts?account.id=gte:0.0.1&token.id=gt:0.0.1&limit=100&order=asc`
2. `getNftAllowances()` builds a `NftAllowanceRequest` with `limit=100` and broad GTE range filters (lines 66–73).
3. `NftAllowanceServiceImpl.getNftAllowances()` executes a DB query returning up to 100 rows (line 30 of `NftAllowanceServiceImpl.java`).
4. `nftAllowanceMapper.map()` serializes all 100 records to JSON (line 76 of `AllowancesController.java`).
5. The full response is sent. Attacker immediately repeats — no throttle, no backoff, no IP-based limit.

**Why existing checks fail:** `@Max(MAX_LIMIT)` (line 61) only prevents requesting more than 100 records per call; it does not limit call frequency. `@Positive` (line 61) only prevents zero/negative limits. `@Size(max = 2)` on filter arrays (lines 60, 64) only limits filter count. None of these prevent rapid repeated calls.

### Impact Explanation
Each request forces a paginated DB scan over the `nft_allowance` table with broad range predicates, followed by full JSON serialization of up to 100 objects. An attacker running this in a tight loop from multiple IPs can saturate the service's outbound bandwidth and serialization thread pool, degrading or denying service to legitimate users. No economic damage to on-chain users occurs; the impact is griefing of the mirror node service itself, consistent with the medium scope classification.

### Likelihood Explanation
No special privileges, tokens, or on-chain state are required. The attack requires only knowledge of any valid account ID (publicly discoverable via other mirror node endpoints). It is trivially scriptable with `curl` or any HTTP client. The broad `gte:0.0.1` filter ensures maximum result set size regardless of which account is targeted. The attack is fully repeatable and stateless.

### Recommendation
1. Introduce a rate-limiting filter in the `rest-java` module analogous to the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern already present in the `web3` module, applying a per-IP (or global) requests-per-second cap to all `/api/v1/accounts/*/allowances/*` endpoints.
2. Wire the existing `HIGH_VOLUME_THROTTLE` constant (already defined in `Constants.java` line 19) into `AllowancesController` to gate high-volume query patterns.
3. Consider deploying an API gateway or reverse proxy (e.g., nginx, Envoy) with connection-rate and request-rate limits in front of the `rest-java` service as a defense-in-depth measure.

### Proof of Concept
```bash
# No credentials required. Replace 0.0.1000 with any valid account ID.
while true; do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1000/allowances/nfts?account.id=gte:0.0.1&token.id=gt:0.0.1&limit=100&order=asc" > /dev/null
done
```
Run from multiple hosts simultaneously to amplify bandwidth and CPU load. Each iteration triggers a full DB range scan and serialization of up to 100 NFT allowance records with no server-side throttle.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
