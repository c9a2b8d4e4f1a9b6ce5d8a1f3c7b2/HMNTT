### Title
Silent Treasury Account Substitution When Smart Contract Address Used as Sender in `getSenderAccountID()`

### Summary
In `TransactionExecutionService.getSenderAccountID()`, when a caller supplies a smart contract address as the `sender` parameter, the code silently replaces it with the treasury account (`systemEntity.treasuryAccount()`) instead of rejecting the request. This causes the simulated transaction to execute with the treasury account as the payer/authorizer in the `TransactionID`, granting treasury-level key authorization to an unprivileged caller's simulation. The actual Hedera consensus node would reject this with `PAYER_ACCOUNT_NOT_FOUND`, but the Mirror Node silently elevates the privilege.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/TransactionExecutionService.java`, `getSenderAccountID()`, lines 239–265:

```java
private AccountID getSenderAccountID(final CallServiceParameters params) {
    if (params.getSender().isZero() && params.getValue() == 0L) {
        return EntityIdUtils.toAccountId(systemEntity.treasuryAccount()); // default for zero sender
    }
    final var senderAddress = params.getSender();
    final var accountIDNum = getSenderAccountIDAsNum(senderAddress);

    final var account = accountReadableKVState.get(accountIDNum);
    if (account == null) {
        throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
    } else if (account.smartContract()) {
        return EntityIdUtils.toAccountId(systemEntity.treasuryAccount()); // ← SILENT SUBSTITUTION
    }
    ...
    return accountIDNum;
}
```

The returned `AccountID` is placed directly into `TransactionID.accountID` at line 167:

```java
.transactionID(TransactionID.newBuilder()
        .transactionValidStart(...)
        .accountID(getSenderAccountID(params))  // treasury injected here
        .build())
```

**Root cause:** The code at line 250–251 detects that the resolved account is a smart contract and, rather than throwing `PAYER_ACCOUNT_NOT_FOUND` (as the real Hedera services do per the comment at lines 287–288), it silently substitutes the treasury account. The comment itself acknowledges the correct behavior is to throw `PAYER_ACCOUNT_NOT_FOUND`, but the implementation diverges.

**Failed assumption:** The code assumes that substituting the treasury account is a safe fallback for smart contract senders. In reality, the treasury account's keys are treated as "active" (signed) during the simulated execution, granting it elevated authorization for Hedera precompile operations (token mint, burn, freeze, wipe, update, etc.) that require treasury-level key signatures.

**Exploit flow:**
1. Attacker identifies any deployed smart contract address on the network (trivially discoverable via Mirror Node REST API).
2. Attacker submits an `eth_call` or `eth_estimateGas` request to the Mirror Node web3 endpoint with `from` set to that smart contract address and a non-zero `value` (or any value).
3. `getSenderAccountID()` resolves the address, finds `account.smartContract() == true`, and returns `EntityIdUtils.toAccountId(systemEntity.treasuryAccount())`.
4. The `TransactionBody` is built with the treasury account as the payer/authorizer.
5. The executor runs the simulation with treasury keys considered active, bypassing key-authorization checks in Hedera precompile calls (e.g., `burnToken`, `mintToken`, `wipeToken`, `freezeToken`, `updateTokenInfo`, `deleteToken`).
6. The simulation succeeds and returns results that would only be achievable if the treasury account had actually signed the transaction.

**Why existing checks fail:**
- The zero-address guard at line 241 only catches the case where `sender.isZero() && value == 0`. A non-zero smart contract address with any value bypasses it entirely.
- There is no validation that the resolved account must be a non-contract account before proceeding.
- The test `ethCallWithValueAndSenderContractSucceeds` (ContractCallServiceTest.java, lines 514–530) explicitly asserts this path succeeds, confirming no rejection occurs.
- The `isSenderSmartContract` parameterized test (ContractCallServicePrecompileModificationTest.java, lines 820–858) further confirms that token creation simulations succeed when the sender is a smart contract, with the treasury substitution in effect.

### Impact Explanation

The treasury account in Hedera holds elevated administrative keys. By forcing the simulation to run with the treasury as the payer, an attacker can:

1. **Bypass precompile authorization checks**: Simulate token operations (mint, burn, wipe, freeze, update, delete) that require treasury or admin key signatures, obtaining accurate simulation results without holding those keys.
2. **Information disclosure**: Determine exact calldata, gas costs, and return values for privileged operations, enabling precise crafting of actual on-chain transactions if the attacker later acquires the necessary keys or finds another vulnerability.
3. **Simulation result divergence**: The Mirror Node returns a `SUCCESS` simulation result for operations that the real consensus node would reject, misleading integrators and dApps that rely on Mirror Node simulation for pre-flight checks before submitting real transactions.

The Mirror Node is a read-only simulation service, so no actual on-chain state is modified. However, the divergence from consensus node behavior and the unauthorized access to treasury-authorized simulation results constitutes a meaningful security boundary violation.

### Likelihood Explanation

- **Precondition**: Attacker needs only a publicly known smart contract address (any deployed contract). These are freely enumerable via the Mirror Node REST API (`/api/v1/contracts`).
- **No authentication required**: The web3 endpoint (`/api/v1/contracts/call`) is unauthenticated and publicly accessible.
- **Trivially repeatable**: The exploit requires a single HTTP POST with a crafted `from` field. No special tooling, keys, or on-chain transactions are needed.
- **Deterministic**: The substitution logic is unconditional — every request with a smart contract sender triggers it.

### Recommendation

Replace the silent substitution at line 250–251 with the same rejection behavior used for null accounts:

```java
} else if (account.smartContract()) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
}
```

This aligns the Mirror Node's behavior with the actual Hedera consensus node (`SolvencyPreCheck#getPayerAccount()` returns `PAYER_ACCOUNT_NOT_FOUND` for smart contract payers), as the comment at lines 287–288 already documents as the intended behavior. If smart contract senders must be supported for specific use cases (e.g., opcode tracing), add an explicit allowlist or a separate code path that does not elevate to treasury authorization.

### Proof of Concept

```bash
# 1. Discover any deployed smart contract address via Mirror Node REST API
CONTRACT_ADDR=$(curl -s "https://<mirror-node>/api/v1/contracts?limit=1" \
  | jq -r '.contracts[0].evm_address')

# 2. Submit eth_call with the smart contract address as the sender (from)
#    Targeting any token precompile operation, e.g., mintToken
curl -X POST "https://<mirror-node>/api/v1/contracts/call" \
  -H "Content-Type: application/json" \
  -d '{
    "from": "'"$CONTRACT_ADDR"'",
    "to": "0x0000000000000000000000000000000000000167",
    "data": "<mintToken calldata>",
    "value": 1,
    "gas": 300000,
    "estimate": false
  }'

# Expected (correct) behavior: HTTP 400 / PAYER_ACCOUNT_NOT_FOUND
# Actual behavior: HTTP 200 with simulation result executed under treasury authorization
# The simulation succeeds as if the treasury account signed the transaction.
```
