### Title
Global Rate Limit Exhaustion Allows Unauthenticated DoS of Mirror Node Web3 API Endpoints

### Summary
Both `rateLimitBucket` (500 req/sec) and `opcodeRateLimitBucket` (1 req/sec) are global singleton `Bucket` instances with no per-IP or per-client partitioning. Any unauthenticated external user can exhaust either bucket by sending requests at or above the configured rate, causing all other clients to receive HTTP 429 for the remainder of each one-second window. The attack requires no credentials and is trivially repeatable.

**Root cause:** `ThrottleConfiguration` creates a single application-scoped `Bucket` bean for each limit. [4](#0-3)  The `opcodeRateLimitBucket` defaults to **1 token/second** and `rateLimitBucket` defaults to **500 tokens/second**. [3](#0-2) 

`throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` against this shared global bucket — a single request per second from any source exhausts it entirely. [1](#0-0)  Similarly, `throttle()` checks the global `rateLimitBucket`. [5](#0-4) 

There is no IP-based partitioning, no authentication gate, and no per-client sub-bucket anywhere in the web3 throttle stack. [6](#0-5) 

**Exploit flow:**
1. Attacker opens two concurrent loops with no credentials.
2. Loop A: `GET /api/v1/contracts/results/{any_hash}/opcodes` with `Accept-Encoding: gzip` at ≥1 req/sec → exhausts `opcodeRateLimitBucket` (capacity 1).
3. Loop B: `POST /api/v1/contracts/call` at ≥500 req/sec → exhausts `rateLimitBucket` (capacity 500).
4. All legitimate clients receive HTTP 429 for the remainder of each 1-second window on both endpoints.
5. Buckets refill every second; attacker simply continues the flood.

**Why existing checks fail:** `tryConsume(1)` is a non-blocking check against a shared counter. It has no concept of caller identity. The `ThrottleException` handler returns 429 to everyone — attacker and legitimate user alike. [7](#0-6) 

### Impact Explanation
Legitimate users of the mirror node's `eth_call` (`/api/v1/contracts/call`) and opcode-tracing (`/api/v1/contracts/results/{id}/opcodes`) endpoints are denied service for as long as the attacker sustains the flood. **Important correction to the question's framing:** the mirror node is a read-only query service; it does not submit or confirm transactions on the Hedera consensus network. Exhausting these buckets does not affect Hedera network transaction confirmation. The actual impact is availability loss of the mirror node web3 read API — severity is **Medium** (availability of a query service), not "total network shutdown."

### Likelihood Explanation
Trivially achievable by any unauthenticated actor with a basic HTTP client. The `opcodeRateLimitBucket` requires only **1 request per second** to keep exhausted. The `rateLimitBucket` requires 500 req/sec, achievable from a single machine with concurrent connections. No special knowledge, credentials, or on-chain state is required. The attack is fully repeatable and stateless.

### Recommendation
1. **Per-IP (or per-client) rate limiting:** Replace the single global `Bucket` with a `Map<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), or use bucket4j's `ProxyManager` with a distributed cache.
2. **Raise the opcode limit or add authentication:** The default of 1 req/sec for opcodes is so low that even a single legitimate concurrent user can exhaust it; require an API key or raise the limit with per-client enforcement.
3. **Network-layer protection:** Deploy a reverse proxy (e.g., nginx, Envoy) with per-IP rate limiting in front of the web3 service as a defense-in-depth measure.

```bash
  curl -s -o /dev/null -H "Accept-Encoding: gzip" \

# Terminal 2 – exhaust rateLimitBucket (500 req/sec)
ab -n 100000 -c 50 -p body.json -T application/json \
  https://<mirror-node>/api/v1/contracts/call

# Legitimate user – observe 429 responses on both endpoints
curl -v -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/<valid_hash>/opcodes"
# Expected: HTTP/1.1 429 Too Many Requests

curl -v -X POST -H "Content-Type: application/json" -d '{"to":"0x...","gas":50000}' \
  https://<mirror-node>/api/v1/contracts/call
# Expected: HTTP/1.1 429 Too Many Requests
```

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
