### Title
`ConstructionParse` Reports False Signers: Negative-Balance Accounts Equated to Cryptographic Signers Without Signature Verification

### Summary
The `/construction/parse` endpoint, when called with `request.Signed = true`, returns `AccountIdentifierSigners` derived solely from which accounts have negative HBAR balances in the transaction body — not from which accounts actually cryptographically signed the transaction. An unprivileged attacker can craft a `TransferTransaction` hex listing any arbitrary account as the negative-balance sender, submit it to `/construction/parse?Signed=true`, and receive a response falsely asserting that account signed the transaction, with zero cryptographic evidence required.

### Finding Description
**Exact code path:**

In `rosetta/app/services/construction/crypto_transfer_transaction_constructor.go`, the `Parse` function (lines 54–88):

```go
hbarTransferMap := transferTransaction.GetHbarTransfers()   // line 68 — reads tx body only
// ...
senderMap := senderMap{}
for _, operation := range operations {
    if operation.Amount.GetValue() < 0 {                    // line 82 — negative amount = "signer"
        senderMap[operation.AccountId.String()] = operation.AccountId
    }
}
return operations, senderMap.toSenders(), nil               // line 87
```

These `accounts` are then consumed in `rosetta/app/services/construction_service.go` lines 203–208:

```go
signers := make([]*rTypes.AccountIdentifier, 0, len(accounts))
if request.Signed {
    for _, account := range accounts {
        signers = append(signers, account.ToRosetta())      // blindly trusted
    }
}
```

**Root cause:** The code conflates "account with a negative transfer in the transaction body" with "account that cryptographically signed the transaction." The `Parse` function never inspects the transaction's signature map. The only guard (line 64) checks that `TransactionID.AccountID != nil` — it says nothing about signatures.

**Exploit flow:**
1. Attacker constructs a `TransferTransaction` with victim account V as the negative-amount sender (e.g., V: -100 HBAR, attacker A: +100 HBAR). No private key for V is needed to build this transaction body.
2. Attacker serializes the transaction to hex (unsigned or with only attacker's own signature attached — the code does not check).
3. Attacker POSTs to `/construction/parse` with `transaction = <hex>` and `signed = true`.
4. Server calls `Parse`, sees V has a negative amount, adds V to `senderMap`, returns V as a signer.
5. Response: `AccountIdentifierSigners: [V]` — falsely asserting V authorized the transaction.

### Impact Explanation
Any downstream system or human operator using the Rosetta `/construction/parse` response to verify authorization before releasing funds or taking action will be deceived. The Rosetta Construction API is explicitly designed as a verification step before submission (the caller is expected to confirm signers match expectations). A false `AccountIdentifierSigners` list breaks this trust model entirely. An attacker can impersonate any account as a signer of any transfer, enabling social-engineering attacks against automated settlement systems, custodians, or exchange integrations that rely on this endpoint for pre-submission authorization checks.

### Likelihood Explanation
The attack requires no privileges, no credentials, and no on-chain interaction. Any HTTP client can craft a valid `TransferTransaction` protobuf, hex-encode it, and call the public endpoint. The Hedera SDK makes constructing such a transaction trivial. The attack is repeatable and fully automated. The only precondition is that a victim system trusts the Rosetta parse response as authoritative for signer verification.

### Recommendation
When `request.Signed = true`, the `Parse` function must verify that the transaction's embedded signature map contains valid signatures from the accounts being reported as signers. Specifically:
- Extract the signature map from the signed transaction bytes.
- For each account reported as a signer, verify that a valid Ed25519 (or other supported) signature over the transaction body bytes exists in the signature map for a key associated with that account.
- Only include an account in `AccountIdentifierSigners` if its signature is cryptographically verified.
- If signature verification is not feasible at the Rosetta layer (e.g., because account public keys require a network lookup), the endpoint should either refuse to populate `AccountIdentifierSigners` or clearly document that the field reflects transaction-body intent, not verified signatures.
