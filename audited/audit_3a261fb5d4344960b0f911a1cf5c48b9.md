### Title
Cache Key Pollution via `order` Parameter Alternation Doubles DB Load in `getBalances()`

### Summary
The response cache in `rest/middleware/responseCacheHandler.js` keys entries on the raw `req.originalUrl` (MD5 hash), meaning `?order=asc` and `?order=desc` produce two independent cache slots. Because the REST API has no rate limiting, an unprivileged attacker can alternate between the two values at will, forcing two expensive DB queries per cache-TTL window instead of one, doubling effective database load compared to a single-parameter flood.

### Finding Description
**Cache key generation** (`rest/middleware/responseCacheHandler.js`, line 151–153):
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
`req.originalUrl` includes the full query string verbatim. `/api/v1/balances?order=asc` and `/api/v1/balances?order=desc` hash to two completely different Redis keys.

**`getBalances()` query cost** (`rest/balances.js`, lines 139–149 and 207–216): both the entity-table path and the account-balance snapshot path execute multi-table queries with `DISTINCT ON`, correlated token-balance subqueries, and `ORDER BY ab.account_id {order}`. The `order` parameter is directly interpolated into the SQL `ORDER BY` clause, so the two variants are semantically distinct queries that cannot share a single cache entry.

**No rate limiting on the REST service**: a `grep` across `rest/**/*.js` for `rateLimit`/`throttle` returns only a test-utility file. The `ThrottleManager` that exists in the codebase is scoped exclusively to the `web3` Java service (`ContractController`). The Node.js REST service exposes `/api/v1/balances` with zero per-IP or per-endpoint request throttling.

**Attack flow**:
1. Attacker sends `GET /api/v1/balances?order=asc` → cache miss → DB query executed → stored under key A.
2. Attacker sends `GET /api/v1/balances?order=desc` → cache miss → DB query executed → stored under key B.
3. After the cache TTL expires (as low as 1 s per `DEFAULT_REDIS_EXPIRY`), both entries are evicted.
4. Attacker repeats steps 1–2 indefinitely, sustaining 2 DB queries per TTL window.

A single-parameter flood is limited to 1 DB query per TTL window by the cache. Alternating two values doubles that floor with no additional privilege required.

### Impact Explanation
The `/api/v1/balances` endpoint is documented as expensive and already deprecated for general use (OpenAPI description recommends alternatives). Sustained alternating requests force the PostgreSQL backend to execute two full `DISTINCT ON` + correlated-subquery scans per second (at the 1 s default TTL), or two per 10 s if a longer `max-age` is configured. Under concurrent attackers or combined with other parameter dimensions (e.g., `limit`, `account.id`), the multiplicative effect on DB CPU and I/O can degrade service for legitimate users. This is a griefing-class denial-of-service with no economic precondition.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of the public API. No authentication, tokens, or on-chain assets are needed. The two valid values for `order` (`asc`, `desc`) are documented in the OpenAPI spec. The attack is trivially scriptable, repeatable indefinitely, and undetectable at the application layer because each request is individually valid and returns HTTP 200.

### Recommendation
1. **Add per-IP rate limiting** to the Node.js REST service (e.g., `express-rate-limit` or an nginx/ingress layer) for all `/api/v1/*` endpoints, especially `/api/v1/balances`.
2. **Normalize the cache key** before hashing: sort query parameters canonically so that semantically equivalent requests (same filters, different parameter order) share a cache slot. The code comment at line 148–149 of `responseCacheHandler.js` already acknowledges this as future work ("In the future, this will utilize Edwin's request normalizer (9113)").
3. **Increase the cache TTL** for the balances endpoint to reduce the frequency at which cache misses can be forced.

### Proof of Concept
```bash
# Terminal 1 – flood order=asc
while true; do
  curl -s "https://<mirror-node>/api/v1/balances?order=asc" -o /dev/null
done

# Terminal 2 – flood order=desc
while true; do
  curl -s "https://<mirror-node>/api/v1/balances?order=desc" -o /dev/null
done
```
Observe in PostgreSQL `pg_stat_activity` or slow-query logs that two distinct `DISTINCT ON (account_id)` queries execute continuously, one per terminal, rather than the single query that a same-parameter flood would produce after the first cache warm-up.
