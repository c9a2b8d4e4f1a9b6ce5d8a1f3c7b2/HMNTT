### Title
Unbounded `index=` Filter Accumulation in `getContractActions()` Enables Resource Exhaustion via Oversized SQL Query Construction

### Summary
`getContractActions()` in `rest/controllers/contractController.js` collects every `index=` query parameter into an unbounded array with no count cap. This array is forwarded verbatim to `ContractService.getContractActions()` in `rest/service/contractService.js`, which appends one `AND index <op> $N` SQL condition and one positional parameter per filter entry. An unauthenticated caller can therefore force the server to build an arbitrarily large `params` array and SQL query string, exhausting Node.js heap memory and imposing query-parsing overhead on the database for every request.

### Finding Description

**Exact code path:**

`contractController.js` lines 1229–1241 — the loop collects every `filterKeys.INDEX` filter into `filters[]` with no upper bound:

```js
for (const filter of rawFilters) {
  ...
  } else if (filter.key === filterKeys.INDEX) {
    if (filter.operator === utils.opsMap.ne) {
      throw InvalidArgumentError.forRequestValidation(filterKeys.INDEX);
    }
    filters.push(filter);   // ← no count check
  }
}
``` [1](#0-0) 

The unbounded `filters` array is then passed directly to the service: [2](#0-1) 

`contractService.js` lines 489–509 — `getContractActions()` iterates the array and, for every entry, pushes a value into `params` and concatenates a new `AND` clause onto `whereClause`:

```js
for (const filter of filters) {
  if (filter.key === 'index') {
    params.push(filter.value);
    whereClause += `\nand ${ContractAction.INDEX}${filter.operator}$${params.length}`;
  }
}
``` [3](#0-2) 

**Root cause:** The only guard on `index=` filters is a rejection of the `ne` operator. There is no limit on how many times the parameter may appear. `acceptedContractActionsParameters` controls which keys are accepted, not how many repetitions are allowed. [4](#0-3) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/<any-valid-hash>/actions?index=gte:0&index=gte:0&...` with N repetitions (e.g., N = 50 000).
2. `buildAndValidateFilters` accepts all of them (key is in the allowed set, operator is not `ne`).
3. The controller loop pushes all N entries into `filters[]`.
4. `getContractActions()` in the service appends N entries to `params[]` and N `AND index >= $k` clauses to the query string.
5. The resulting query string and params array are proportional to N; at large N this exhausts heap and/or causes the DB to spend significant time parsing the query.

### Impact Explanation
- **Node.js heap exhaustion:** Each filter adds a string segment to `whereClause` and an element to `params`. At tens of thousands of repetitions the allocations are substantial and can trigger OOM or GC pressure, degrading or crashing the REST service for all users.
- **Database query-parsing overhead:** PostgreSQL must tokenize, parse, and plan a query with N positional parameters. While the `consensus_timestamp = $1` predicate is selective (limiting the scan itself), the planning phase is O(N) in parameter count and can hold a DB connection for an extended period, exhausting the connection pool under concurrent attack.
- **No authentication required:** The endpoint is public; any external caller can trigger this.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of any valid transaction hash or ID (both are publicly observable on-chain). It is trivially scriptable, repeatable at high frequency, and requires no credentials or special privileges. A single persistent attacker can sustain the load.

### Recommendation
1. **Cap the number of `index=` filters** in `getContractActions()` before the loop, e.g.:
   ```js
   const indexFilters = rawFilters.filter(f => f.key === filterKeys.INDEX);
   if (indexFilters.length > MAX_REPEATED_QUERY_PARAMS) {
     throw InvalidArgumentError.forRequestValidation(filterKeys.INDEX);
   }
   ```
2. **Enforce a global repeated-parameter limit** inside `buildAndValidateFilters` / `filterValidityChecks` so all endpoints benefit.
3. Consider collapsing multiple equality `index=` values into a single `IN (…)` clause rather than N separate `AND` conditions.

### Proof of Concept
```bash
# Build a URL with 10 000 repeated index filters
PARAMS=$(python3 -c "print('&'.join(['index=gte:0']*10000))")
curl -s "https://<mirror-node>/api/v1/contracts/results/0x$(python3 -c "print('a'*64)")/actions?$PARAMS"
```
Observe: the Node.js process RSS grows proportionally; the DB receives a query with 10 000+ positional parameters; under concurrent requests the connection pool is exhausted and legitimate requests begin timing out.
