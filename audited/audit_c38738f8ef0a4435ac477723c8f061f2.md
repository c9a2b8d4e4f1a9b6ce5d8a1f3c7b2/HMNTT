### Title
Missing Per-IP In-Flight Request Limiting on `/api/v1/accounts/{id}/airdrops/outstanding` Enables Connection Pool Exhaustion

### Summary
The `rest-java` service exposes the `/api/v1/accounts/{id}/airdrops/outstanding` endpoint via `TokenAirdropsController.processRequest()` with no per-IP connection or in-flight request limiting at any layer — neither in the application code nor in the Traefik middleware configuration. An unprivileged attacker can open thousands of simultaneous slow HTTP connections (Slowloris-style), exhausting Tomcat's thread/connection pool and denying service to legitimate users. This gap is confirmed by comparing the `rest-java` middleware configuration against the `rosetta` service, which explicitly configures `inFlightReq` per-IP limits.

### Finding Description

**Exact code location — controller (no throttling):**

`processRequest()` in `TokenAirdropsController.java` lines 89–114 builds a `TokenAirdropRequest`, calls `service.getAirdrops()`, maps the result, and returns — with zero rate-limiting, per-IP connection checks, or throttle guards of any kind. [1](#0-0) 

**Exact code location — missing middleware:**

The Traefik middleware chain for `rest-java` is defined in `charts/hedera-mirror-rest-java/values.yaml` lines 150–155. It contains only a `circuitBreaker` and a `retry` entry — **no `inFlightReq` and no `rateLimit`**: [2](#0-1) 

**Contrast with rosetta**, which explicitly configures per-IP in-flight request limiting and rate limiting: [3](#0-2) 

**Application-level config — no Tomcat/Netty limits set:**

No `server.tomcat.max-connections`, `server.tomcat.accept-count`, or `server.tomcat.connection-timeout` overrides exist anywhere in the `rest-java` module. The `RestJavaConfiguration.java` and `WebMvcConfiguration.java` register only an ETag filter and argument resolvers — no throttle filter. [4](#0-3) 

**Root cause:** The `rest-java` service was never given the `inFlightReq` per-IP middleware that other services received. The failed assumption is that an upstream load balancer or GCP gateway provides sufficient protection.

**Exploit flow:**
1. Attacker opens thousands of TCP connections to the `rest-java` ingress.
2. For each connection, the attacker sends the HTTP request line and headers extremely slowly (one byte every few seconds — Slowloris technique), keeping each Tomcat worker thread blocked in the header-read phase.
3. Because there is no per-IP in-flight limit, all connections are accepted.
4. Tomcat's default thread pool (200 threads) and connection queue (`acceptCount`) are exhausted.
5. Legitimate requests queue and then time out with connection refused or 503.

### Impact Explanation

Tomcat's default `maxThreads` is 200 and `acceptCount` is 100. A single attacker with ~300 persistent slow connections can saturate one `rest-java` pod entirely. With `hpa.minReplicas: 1` and HPA disabled by default (`hpa.enabled: false`), a single pod is the common deployment. Even with multiple replicas, the attacker scales connections proportionally. The `airdrops/outstanding` and `airdrops/pending` endpoints are both affected via the same `processRequest()` path. This directly maps to the stated severity: degradation of 30%+ of mirror node REST capacity without brute force. [5](#0-4) 

### Likelihood Explanation

This requires no authentication, no special knowledge, and no exploit code — only a standard Slowloris tool (e.g., `slowloris.py`, `hping3`). The endpoint is publicly routable via the ingress path `/api/v1/accounts/.../airdrops`. The attack is repeatable and automatable. The only partial mitigation is the GCP backend policy `timeoutSec: 20`, but: (a) it only applies when `gateway.gcp.enabled: true` is set, (b) it is a backend response timeout, not a client header-read timeout, and (c) the Traefik ingress path has no equivalent timeout. The `maxRatePerEndpoint: 250` is a global throughput cap, not a per-IP connection limit, and its own comment notes it requires HPA changes to take effect. [6](#0-5) 

### Recommendation

1. **Add `inFlightReq` per-IP middleware** to `charts/hedera-mirror-rest-java/values.yaml` mirroring the rosetta configuration:
   ```yaml
   middleware:
     - inFlightReq:
         amount: 10
         sourceCriterion:
           ipStrategy:
             depth: 1
     - rateLimit:
         average: 50
         burst: 100
         sourceCriterion:
           ipStrategy:
             depth: 1
     - circuitBreaker:
         expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
     - retry:
         attempts: 3
         initialInterval: 100ms
   ```
2. **Set Tomcat header read timeout** in `rest-java` application properties: `server.tomcat.connection-timeout=5000` to bound slow-header attacks independently of the proxy layer.
3. **Enable HPA** so that connection exhaustion on one pod does not take down the entire service.

### Proof of Concept

```bash
# Install slowloris (pip install slowloris) or use the Python script
# Target: the rest-java ingress host

python3 slowloris.py <mirror-node-host> \
  --port 443 \
  --ssl \
  --sleeptime 10 \
  --socket-count 500 \
  --path "/api/v1/accounts/0.0.1234/airdrops/outstanding"

# Expected result within ~60 seconds:
# - Legitimate curl requests to the same endpoint return 503 or hang indefinitely
# - Tomcat active thread count (via /actuator/metrics/tomcat.threads.busy) pegged at max
# - New connections are queued then refused
```
