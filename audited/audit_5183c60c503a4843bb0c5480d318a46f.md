### Title
Unauthenticated Repeated Full Entity Table Aggregation via `/api/v1/network/supply` Without Timestamp Parameter

### Summary
The `GET /api/v1/network/supply` endpoint in the rest-java module, when called without a `timestamp` parameter, unconditionally executes a full aggregation query over the `entity` table on every request. There is no caching, no rate limiting, and no authentication requirement on this code path, allowing any unprivileged external user to repeatedly trigger the most expensive database operation at will.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (line 132–150, `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`) accepts `timestamp` as `required = false`. When omitted, `Bound.of(null, ...)` produces an empty `Bound`.

`NetworkServiceImpl.getSupply(Bound timestamp)` (line 59–89, `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`) branches at line 66:
```java
if (timestamp.isEmpty()) {
    networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
}
```

`EntityRepository.getSupply()` (lines 19–29, `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`) executes:
```sql
select cast(coalesce(sum(e.balance), 0) as bigint) as unreleased_supply,
    cast(coalesce(max(e.balance_timestamp), 0) as bigint) as consensus_timestamp
from entity e
join unnest(
        cast(string_to_array(:lowerBounds, ',') as bigint[]),
        cast(string_to_array(:upperBounds, ',') as bigint[])
     ) as ranges(min_val, max_val)
  on e.id between ranges.min_val and ranges.max_val
```

This is a full sequential scan + aggregation (`SUM`, `MAX`) over all rows in the `entity` table matching the configured account ranges (default: 7 ranges covering accounts 2–750, `NetworkProperties.java` lines 22–29).

**Root cause and failed assumption:**

- No `@Cacheable` annotation exists anywhere in the call chain for the no-timestamp path (`EntityRepository`, `NetworkServiceImpl`, `NetworkController`).
- The rest-java module has **no rate limiting** for any endpoint. The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and applies exclusively to contract call requests.
- The `rest-java/build.gradle.kts` includes `spring-boot-starter-cache` but no cache is configured or applied to this query.
- The failed assumption is that the no-timestamp path (current live supply) would be cheap or protected; in reality it is the most expensive path and is completely unguarded.

**Why existing checks fail:**

- No Spring Security authentication guard on the endpoint.
- No `@RateLimiter`, `@Cacheable`, or any middleware throttle on `getSupply()`.
- The `statementTimeout` for rest-java DB is 10,000 ms (config line 630), meaning each query can run for up to 10 seconds before timing out — each such query holds a DB connection and consumes CPU/IO for its full duration.

### Impact Explanation

On a production Hedera mirror node, the `entity` table contains tens of millions of rows. Each unauthenticated `GET /api/v1/network/supply` (no timestamp) forces a full aggregation scan. With a connection pool of 10 (default `maxConnections`), an attacker sending concurrent requests can saturate the DB connection pool and drive sustained high CPU/IO on the database server. Because each query can run up to 10 seconds, even a modest rate of ~1 request/second from a single attacker can keep multiple DB connections busy continuously. A 30%+ increase in database resource consumption over the 24-hour baseline is achievable with a low-volume, sustained attack from a single IP with no special privileges.

### Likelihood Explanation

- **Preconditions**: None. No account, API key, or authentication required.
- **Feasibility**: A single `curl` loop or any HTTP client suffices. The endpoint is publicly documented.
- **Repeatability**: Fully repeatable and automatable. The attacker does not need to vary requests — the same `GET /api/v1/network/supply` with no parameters triggers the expensive path every time.
- **Detection difficulty**: The requests are individually legitimate (valid API calls returning correct data), making them hard to distinguish from normal traffic without rate-of-call monitoring.

### Recommendation

1. **Add caching** to `EntityRepository.getSupply()` or `NetworkServiceImpl.getSupply()` with a short TTL (e.g., 15–30 seconds), since the live supply value changes only with each consensus round. A `@Cacheable` annotation with a Caffeine or Redis-backed cache manager would eliminate repeated DB hits for the no-timestamp path.
2. **Add rate limiting** to the rest-java module for the `/api/v1/network/supply` endpoint (and ideally all endpoints), similar to the bucket4j-based throttle in the web3 module.
3. **Consider a dedicated materialized view or scheduled refresh** for the current supply value, so the no-timestamp path reads a pre-computed result rather than executing a live aggregation.

### Proof of Concept

```bash
# No authentication, no timestamp — triggers full entity table aggregation on every call
while true; do
  curl -s "https://<mirror-node-host>/api/v1/network/supply" -o /dev/null &
done
# Run from multiple terminals or use a tool like wrk/ab:
wrk -t4 -c50 -d60s "https://<mirror-node-host>/api/v1/network/supply"
```

Monitor PostgreSQL CPU and I/O on the mirror node DB during the 60-second run. Compare against the 24-hour baseline metrics. The aggregation query (`sum(e.balance)` over the entity table) will appear repeatedly in `pg_stat_activity` and `pg_stat_statements`, with measurable increases in `blks_read`, `shared_blks_hit`, and CPU time.
