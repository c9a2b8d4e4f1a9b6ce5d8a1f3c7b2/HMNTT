### Title
Unbounded Subscription via startTime=0 and limit=Long.MAX_VALUE Causes Indefinite Pipeline Stage Exhaustion

### Summary
An unprivileged external user can open a gRPC topic subscription with `startTime=0`, no `endTime`, and `limit=Long.MAX_VALUE`. This causes the historical retriever to scan every stored topic message from the beginning of time, then seamlessly transitions to a live subscription that never terminates. No server-side enforcement caps either stage, allowing an attacker to hold open arbitrarily many such subscriptions and exhaust DB connections, thread pool slots, and memory.

### Finding Description

**Validation of filter fields** (`TopicMessageFilter.java`, lines 25–51):

- `limit` carries only `@Min(0)`, so `Long.MAX_VALUE` is accepted. [1](#0-0) 
- `startTime` carries `@Min(0)` and `isValidStartTime()` checks `startTime <= DomainUtils.now()`. The value `0` (epoch) satisfies both. [2](#0-1) 
- `endTime` is entirely optional; no field forces the caller to supply one. [3](#0-2) 

**Stage 1 – historical retriever** (`TopicMessageServiceImpl.java`, line 63):

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```

With `startTime=0` and no `endTime`, the retriever issues paginated DB queries from timestamp `0` through all stored records. There is no server-side cap on how far back `startTime` may reach. [4](#0-3) 

**Stage 2 – live subscription never terminates** (`TopicMessageServiceImpl.java`, lines 72–85):

```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
    ...
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());   // take(Long.MAX_VALUE) ≈ take(∞)
}
```

`pastEndTime()` returns `Flux.never()` when `endTime == null` (line 124–125), so `takeUntilOther` never fires. `take(Long.MAX_VALUE)` in Project Reactor is functionally equivalent to no limit. The live stage therefore runs indefinitely. [5](#0-4) 

`isComplete()` also returns `false` unconditionally when `endTime == null`, keeping the safety-check polling loop alive as well. [6](#0-5) 

**No subscriber cap**: `subscriberCount` is only a metric gauge; it is never compared against a maximum to reject new subscriptions. [7](#0-6) 

**Cancel handler exists but is client-controlled**: The cancel handler (`disposable::dispose`) only fires if the client sends a gRPC cancel. An attacker simply never cancels. [8](#0-7) 

### Impact Explanation
Each malicious subscription holds open one or more DB connections (paginated historical queries), a bounded-elastic scheduler thread (safety-check), and a live-listener slot indefinitely. An attacker opening N such subscriptions can exhaust the DB connection pool, starve legitimate subscribers, and cause the gRPC service to become unresponsive. Because the service is publicly reachable with no authentication, this is a straightforward denial-of-service against the mirror node's topic subscription infrastructure.

### Likelihood Explanation
No authentication, no per-IP rate limit, no maximum subscriber count, and no server-enforced `endTime` or `limit` ceiling are present in the code. Any internet-reachable deployment is exploitable with a single gRPC client library call. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a maximum `limit`**: Add `@Max(MAX_ALLOWED_LIMIT)` to the `limit` field in `TopicMessageFilter` and reject `limit=0` (no-limit) subscriptions from unauthenticated callers, or require an explicit `endTime` when no limit is set.
2. **Enforce a maximum look-back window**: Reject `startTime` values older than a configurable threshold (e.g., 30 days) unless the caller holds a privileged role.
3. **Enforce a maximum concurrent subscriber count**: Compare `subscriberCount` against a configurable ceiling in `subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
4. **Enforce a server-side subscription timeout**: Automatically terminate subscriptions that have been open longer than a configurable maximum duration, regardless of client behaviour.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    limit=9223372036854775807  # Long.MAX_VALUE
    # no consensusEndTime set
)

# Open N connections; never cancel
for _ in range(500):
    responses = stub.subscribeTopic(query)
    # Do not iterate or cancel — server holds all pipeline stages open indefinitely
```

Each open call causes the server to (1) page through all historical records from epoch 0 and (2) hold a live listener open forever, exhausting DB connections and scheduler threads.
