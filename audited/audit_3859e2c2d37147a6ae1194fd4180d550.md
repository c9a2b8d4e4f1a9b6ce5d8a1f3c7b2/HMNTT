### Title
Cache Bypass via Non-Existent EVM Address Lookups Enabling Sustained Database Load Amplification

### Summary
`findByEvmAddressAndDeletedIsFalse()` uses `unless = "#result == null"` which, due to Spring Cache's Optional unwrapping behavior, prevents caching of empty results when a queried EVM address does not exist in the database. An unprivileged attacker can repeatedly submit contract call requests targeting non-existent EVM addresses, causing every such request to bypass the cache and execute a full database query. The existing rate limiter (500 RPS global) does not prevent this because it operates at the HTTP request level, not per-address, and the attacker can rotate through the 2^160 EVM address space indefinitely.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:

```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

**Root cause:** Spring Cache's `@Cacheable` with an `Optional` return type unwraps the Optional before evaluating SpEL expressions. When the method returns `Optional.empty()` (entity not found), `#result` evaluates to `null`, making `unless = "#result == null"` evaluate to `true`, and the result is **not stored in the cache**. Every subsequent call with the same (or any other) non-existent address hits the database directly.

**Call path:** `POST /api/v1/contracts/call` → `ContractController.call()` → `ContractExecutionService.processCall()` → `CommonEntityAccessor.get(Address, timestamp)` → `getEntityByEvmAddressTimestamp()` → `entityRepository.findByEvmAddressAndDeletedIsFalse(addressBytes)`.

The entity cache is configured as `expireAfterWrite=1s,maximumSize=10000` (see `CacheProperties.java` line 19). Even for addresses that do exist, the 1-second TTL means the cache provides minimal protection. For non-existent addresses, the cache provides **zero** protection.

**Why existing checks fail:**

1. **Rate limiter** (`ThrottleManagerImpl`, lines 37–42): The global `requestsPerSecond = 500` limit allows 500 requests/second. Each request with a unique non-existent EVM address generates one uncached DB query. At 500 RPS, this is 500 DB queries/second sustained indefinitely.

2. **Gas throttle** (`ThrottleManagerImpl`, line 40): `scaleGas()` in `ThrottleProperties` (lines 42–47) returns `0` for gas ≤ 10,000, meaning requests with minimal gas consume zero gas tokens. The attacker can send requests with gas ≤ 10,000 and bypass the gas throttle entirely while still triggering the EVM address lookup.

3. **DB statement timeout** (`statementTimeout = 3000ms`): This limits individual query duration but does not limit query volume.

### Impact Explanation
An attacker sustaining 500 RPS of contract calls to non-existent EVM addresses generates 500 uncached database queries per second against the `entity` table. This constitutes a database load amplification attack: the application's caching layer provides no protection for the miss case, so the database absorbs the full request rate. Under sustained attack, this can degrade response times for legitimate users, exhaust database connection pool capacity, and cause service degradation. Severity is Medium (griefing/DoS with no direct economic damage to users).

### Likelihood Explanation
The attack requires no credentials, no privileged access, and no special knowledge beyond the public API endpoint. EVM addresses are 20-byte values; the attacker has 2^160 unique addresses to rotate through, making address-based blocking ineffective. The attack is trivially scriptable with standard HTTP tooling. The only barrier is the 500 RPS global rate limit, which the attacker can fully saturate with cache-bypassing requests.

### Recommendation
Cache negative (empty) results explicitly using a sentinel value, or change the `unless` condition to also cache empty Optionals:

```java
// Option 1: Cache empty Optional results (remove the unless clause or change condition)
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

Spring Cache natively supports caching `Optional.empty()` when no `unless` condition excludes it. Alternatively, implement per-IP or per-address rate limiting at the API gateway layer to prevent a single client from saturating the global RPS budget with cache-miss requests.

### Proof of Concept
1. Generate N unique random 20-byte hex strings (non-existent EVM addresses).
2. Send `POST /api/v1/contracts/call` at 500 RPS with `{"to": "<random_address>", "gas": 10000, "data": "0x"}` rotating through the generated addresses.
3. Monitor database query rate (e.g., via `pg_stat_activity` or Micrometer metrics exposed by the service).
4. Observe that the database receives one query per HTTP request with no cache hits, confirming the cache bypass. The `recordStats` on the Caffeine cache will show a 0% hit rate for the `evmAddress` cache under this workload.
