### Title
Unauthenticated Client-Disconnect Context Cancellation Enables DB Query Griefing

### Summary
`GetDbWithContext()` propagates the HTTP request context directly to database queries. An unprivileged external user can trigger expensive correlated-subquery operations and then immediately close the TCP connection, causing the in-flight PostgreSQL query to be cancelled mid-execution. With no rate limiting in place, this can be repeated at high frequency to generate sustained query-cancellation overhead on the database server, degrading performance for all users.

### Finding Description
**Code path:**

In `rosetta/app/db/client.go`, `GetDbWithContext()` (lines 22–38):

- When `statementTimeout <= 0`: the raw HTTP request context (`ctx`) is attached directly to the gorm DB instance via `db.WithContext(ctx)` (line 26). [1](#0-0) 
- When `statementTimeout > 0`: a child context is created with `context.WithTimeout(ctx, statementTimeout)` (line 36), but the parent is still the HTTP request context. [2](#0-1) 

In both branches, the HTTP request context is the root. In Go's `net/http`, `r.Context()` is cancelled the moment the client closes the TCP connection. This cancellation propagates immediately to any child context, including the one attached to the gorm DB, causing PostgreSQL to receive a query-cancellation signal mid-execution.

**Expensive queries reachable by unauthenticated users:**

- `latestBalanceBeforeConsensus` — a correlated subquery inside a derived table, triggered by any `/account/balance` request. [3](#0-2) 
- `selectTransactionsInTimestampRange` — two correlated `json_agg` subqueries per row, triggered by `/block`. [4](#0-3) 
- `selectCryptoEntityByAlias` — a `UNION ALL` across `entity` and `entity_history` with a range operator, triggered by `/account/balance` with an alias. [5](#0-4) 

**Why existing checks are insufficient:**

The only middleware present is `TracingMiddleware` (logging only) and health/metrics endpoints. [6](#0-5)  There is no rate limiting, no per-IP connection throttle, and no request concurrency cap at the HTTP layer. The DB connection pool (`MaxOpenConnections`) limits concurrent DB sessions but does not prevent rapid serial or concurrent cancellation storms. [7](#0-6)  The `statementTimeout` config only bounds how long a query runs before the server cancels it; it does not prevent the client from triggering cancellations earlier and more frequently than the timeout would. [8](#0-7) 

### Impact Explanation
Each query cancellation in PostgreSQL requires the backend process to unwind its execution state, release locks, and roll back any partial work. Under a sustained flood of cancel signals, the cumulative overhead (process wakeups, lock manager churn, WAL activity from partial writes) degrades throughput for all concurrent legitimate queries. Because the Rosetta API is a public-facing read API with no authentication requirement, any internet-accessible deployment is exposed. The severity is bounded to availability/performance degradation (no data exfiltration or integrity impact), consistent with the "griefing" classification.

### Likelihood Explanation
The attack requires only the ability to send HTTP POST requests and close the connection — no credentials, no special knowledge, and no expensive resources. Standard tools (`curl`, `ab`, `wrk`, custom scripts) can trivially implement the connect-send-disconnect loop at hundreds of requests per second from a single host. The Rosetta API endpoints (`/account/balance`, `/block`) are standard, well-documented paths. The attack is fully repeatable and stateless.

### Recommendation
1. **Add an HTTP-layer rate limiter** (e.g., `golang.org/x/time/rate` or a middleware like `go-chi/httprate`) keyed by client IP, applied before requests reach service handlers.
2. **Decouple DB query lifetime from client connection lifetime**: instead of passing `r.Context()` directly, derive a server-owned context with a fixed `statementTimeout` deadline that is independent of client disconnect. The current `statementTimeout` branch already does this partially, but the parent context cancellation still propagates. Use `context.WithoutCancel(ctx)` (Go 1.21+) as the base before applying the timeout, so client disconnect does not short-circuit the server-side timeout.
3. **Enforce `statementTimeout > 0`** in configuration validation so the no-timeout branch (lines 23–29) is never reachable in production. [1](#0-0) 
4. **Limit `MaxOpenConnections`** to a value that prevents the DB from being overwhelmed by concurrent cancellation-in-progress sessions. [7](#0-6) 

### Proof of Concept
```bash
# Requires: curl, bash. Target: running Rosetta instance at localhost:5700.
# Step 1: Identify a valid network identifier (replace as needed).
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'

# Step 2: Flood /account/balance with immediate disconnects.
# Each request triggers latestBalanceBeforeConsensus + balanceChangeBetween queries.
for i in $(seq 1 500); do
  curl -s -m 0.05 -X POST http://localhost:5700/account/balance \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":$NETWORK,\"account_identifier\":{\"address\":\"0.0.2\"}}" \
    --max-time 0.05 &   # disconnect after 50ms, before query completes
done
wait

# Step 3: Observe PostgreSQL pg_stat_activity for cancelled queries:
# psql -c "SELECT count(*) FROM pg_stat_activity WHERE state='idle in transaction (aborted)';"
# Monitor DB CPU and query latency for legitimate requests during the flood.
```

The `--max-time 0.05` flag causes curl to abort the connection after 50 ms, cancelling any in-flight DB query. Running 500 concurrent instances saturates the DB with cancellation signals. Legitimate queries issued during this window will experience elevated latency or timeouts.

func TracingMiddleware(inner http.Handler) http.Handler {
	return http.HandlerFunc(func(responseWriter http.ResponseWriter, request *http.Request) {
		start := time.Now()
		clientIpAddress := getClientIpAddress(request)
		path := request.URL.RequestURI()
		tracingResponseWriter := newTracingResponseWriter(responseWriter)

		inner.ServeHTTP(tracingResponseWriter, request)

		message := fmt.Sprintf("%s %s %s (%d) in %s",
			clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))

		if internalPaths[path] {
			log.Debug(message)
		} else {
			log.Info(message)
		}
	})
}
