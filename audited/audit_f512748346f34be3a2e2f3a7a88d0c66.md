### Title
Global Rate Limit Exhaustion DoS on Opcodes Endpoint via Single Unprivileged IP

### Summary
The `opcodeRateLimitBucket` in `ThrottleConfiguration.java` is a single, globally-shared in-memory `Bucket4j` bucket with a default capacity of **1 request per second**, shared across all clients with no per-IP partitioning. Any unauthenticated attacker can continuously send requests to the opcodes endpoint, consuming all available tokens and denying all other users access to opcode traces for any transaction.

### Finding Description
**Code path:**
- `OpcodesController.java:61` — `throttleManager.throttleOpcodeRequest()` is called on every request before processing.
- `ThrottleManagerImpl.java:52–56` — `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` on a single shared bucket instance.
- `ThrottleConfiguration.java:47–55` — `opcodeRateLimitBucket` is a Spring singleton `Bucket` bean, constructed once for the entire JVM process.
- `ThrottleProperties.java:29` — Default `opcodeRequestsPerSecond = 1`.

**Root cause:** The bucket is a single JVM-level object. `Bucket.tryConsume(1)` is called without any per-client (IP, session, or user) key. There is no IP-based partitioning, no authentication requirement, and no secondary per-client limit. The failed assumption is that the global limit would be shared fairly; in reality, a single client can monopolize it entirely.

**Exploit flow:**
1. Attacker identifies any valid transaction ID or hash (publicly visible on-chain).
2. Attacker sends a tight loop of `GET /api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip` (the only other enforced check).
3. Each request consumes 1 token from the global bucket. With a 1 req/sec default, the attacker saturates the bucket continuously.
4. All concurrent legitimate users receive HTTP 429 `Too Many Requests` for every request.

**Why existing checks fail:**
- `validateAcceptEncodingHeader` (`OpcodesController.java:75–86`) only checks for `gzip` — trivially satisfied.
- `properties.isEnabled()` (`OpcodesController.java:59`) is a feature flag, not a security control.
- There is no IP-based rate limiting, no authentication, and no per-client bucket.

### Impact Explanation
The opcodes endpoint replays EVM transactions to produce full opcode traces (stack, memory, storage). Exhausting the global rate limit makes opcode traces for **all** transactions inaccessible to all users simultaneously. While the underlying transaction data is not altered, the ability to audit or inspect any transaction's execution trace is completely blocked for the duration of the attack. This constitutes a targeted availability attack on transaction auditability/history inspection, classifiable as high-severity DoS.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero on-chain interaction. The attacker only needs network access to the endpoint and knowledge of any valid transaction hash (trivially obtained from a public explorer). The attack is fully repeatable, scriptable with a single `curl` loop, and self-sustaining as long as the attacker maintains the request rate. The extremely low default limit (1 req/sec) makes exhaustion trivial even from a single thread.

### Recommendation
Replace the single global `Bucket` with a per-IP `Caffeine`-backed or `Bucket4j`-distributed cache of buckets, keyed on the client's remote IP (extracted from `HttpServletRequest.getRemoteAddr()` or a trusted `X-Forwarded-For` header). Example pattern:

```java
// Per-IP bucket cache
LoadingCache<String, Bucket> perIpBuckets = Caffeine.newBuilder()
    .expireAfterAccess(1, TimeUnit.MINUTES)
    .build(ip -> buildOpcodeBucket());

// In throttleOpcodeRequest(String clientIp):
if (!perIpBuckets.get(clientIp).tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

Additionally, consider adding a global ceiling bucket in conjunction with per-IP buckets to cap total server load.

### Proof of Concept
```bash
# Exhaust the global opcode rate limit from a single IP
# Replace TX_HASH with any valid transaction hash
TX_HASH="0xabc123..."
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes"
done
# Expected: After the first request succeeds (200), all subsequent requests
# (including from other IPs) return 429 Too Many Requests for the remainder
# of the current 1-second window, repeating indefinitely.
```
