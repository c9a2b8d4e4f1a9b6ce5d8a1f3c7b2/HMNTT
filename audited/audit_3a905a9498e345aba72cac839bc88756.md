### Title
SQL Wildcard Expansion via Boolean `true` Interpolation in `getContractStateChangesByTimestamps` Leaks Cross-Contract State Changes

### Summary
When `involvedContractIds` is an empty array (the default), the ternary at line 426 assigns the JavaScript boolean `true` to `contractIdsQuery`. This literal `true` is then string-interpolated directly into the SQL WHERE clause, producing a condition that matches every non-migration row at the queried timestamp — regardless of `contract_id`. Any unprivileged caller who queries a transaction that produced no involved/created contracts (the common case) will receive state changes belonging to all other contracts active at that timestamp.

### Finding Description

**Exact code path:** [1](#0-0) 

```js
// line 426-428
const contractIdsQuery = involvedContractIds?.length
  ? `${ContractStateChange.CONTRACT_ID} in (${involvedContractIds.join(',')})`
  : true;                          // ← JS boolean, not a SQL fragment

// line 429-433  (contractId branch)
conditions.push(
  `(${ContractStateChange.CONTRACT_ID} = $${params.length} or (${contractIdsQuery} and migration is false))`
);
// → "(contract_id = $N or (true and migration is false))"

// line 434-437  (no-contractId branch)
conditions.push(contractIdsQuery);          // pushes literal `true`
conditions.push(`${ContractStateChange.MIGRATION} is false`);
// → "where … and true and migration is false"
```

**Root cause:** `[].length === 0`, which is falsy, so the ternary always falls to `: true` for the default/empty case. JavaScript's `true` coerces to the string `"true"` during template-literal interpolation, which is a valid PostgreSQL boolean literal meaning "always match."

**Resulting SQL (contractId branch):**
```sql
WHERE consensus_timestamp = $1
  AND (contract_id = $2 OR (true AND migration IS FALSE))
```
The `OR true` arm makes the entire disjunction unconditionally true for every non-migration row, so the `contract_id = $2` filter is completely bypassed.

**Resulting SQL (no-contractId branch):**
```sql
WHERE consensus_timestamp = $1
  AND true
  AND migration IS FALSE
```
Returns every non-migration state change at that timestamp with no contract filter at all.

### Impact Explanation
An attacker receives the full set of `contract_state_change` rows for every contract that participated in the same consensus timestamp, including storage slot keys and values that may encode balances, ownership mappings, or other sensitive on-chain state. This constitutes an information-disclosure vulnerability: the fund-state view for the queried contract is polluted with foreign contract data, and the foreign contracts' state is exposed without authorization. Severity: **High** (confidentiality breach; data integrity of the returned view is also broken).

### Likelihood Explanation
No privilege is required. The trigger condition — an empty `involvedContractIds` — is the **default** and occurs for every simple contract call that does not create child contracts. An attacker needs only to:
1. Know any valid `contractId` and a `consensus_timestamp` at which that contract executed.
2. Issue a standard REST GET request to the mirror-node API endpoint that internally calls `getContractStateChangesByTimestamps`.

Both pieces of information are publicly observable on-chain. The attack is trivially repeatable and requires no special tooling.

### Recommendation
Replace the boolean fallback with a SQL fragment that is a no-op when the list is empty, or guard the condition explicitly:

```js
// Option A: omit the involvedContractIds sub-clause entirely when the list is empty
const contractIdsQuery = involvedContractIds?.length
  ? `${ContractStateChange.CONTRACT_ID} in (${involvedContractIds.join(',')})`
  : null;   // null → skip the clause

if (contractId) {
  params.push(contractId);
  const involvedPart = contractIdsQuery
    ? `(${contractIdsQuery} and migration is false)`
    : 'false';   // no involved contracts → second arm never matches
  conditions.push(`(${ContractStateChange.CONTRACT_ID} = $${params.length} or ${involvedPart})`);
} else if (contractIdsQuery) {
  conditions.push(contractIdsQuery);
  conditions.push(`${ContractStateChange.MIGRATION} is false`);
} else {
  // nothing to query — return empty or throw
}
```

Additionally, `involvedContractIds.join(',')` interpolates IDs directly into SQL without parameterization; those values should be passed as bound parameters to prevent SQL injection if the list is ever populated from user-influenced data.

### Proof of Concept

1. Identify any `consensus_timestamp` T at which at least two different contracts recorded state changes (observable via the public ledger).
2. Pick `contractId` = ID of contract A (not contract B).
3. Ensure the transaction at T did not create child contracts, so `involvedContractIds` will be `[]`.
4. Call the mirror-node REST endpoint that resolves to:
   ```
   GET /api/v1/contracts/{contractA}/results/{T}/statechanges
   ```
5. Observe that the response includes `contract_state_change` rows whose `contract_id` equals contract B — data that should never be returned for a query scoped to contract A.
6. The leaked rows expose storage slot keys/values of contract B to the caller of contract A's endpoint.
