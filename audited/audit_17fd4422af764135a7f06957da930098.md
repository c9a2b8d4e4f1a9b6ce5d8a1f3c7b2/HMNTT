### Title
Unbounded Concurrent Subscriptions Exhaust Shared `boundedElastic()` Scheduler in `PollingTopicListener` (POLL Mode DoS)

### Summary
When the gRPC listener is configured to use `POLL` mode (`hiero.mirror.grpc.listener.type=POLL`), each call to `listen()` creates an independent `PollingContext` and schedules perpetual DB polling on the global shared `Schedulers.boundedElastic()` thread pool. Because there is no authentication on the gRPC endpoint, no global connection limit, and no global subscription cap, an unauthenticated attacker can open an unbounded number of connections (each with up to 5 subscriptions per the per-connection limit) to saturate the shared thread pool, causing all poll tasks — including those for legitimate subscribers — to queue indefinitely and halting topic message delivery.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 43–53) accepts any unauthenticated request, maps it to a `TopicMessageFilter`, and calls `TopicMessageServiceImpl.subscribeTopic()`. That method calls `topicListener.listen(newFilter)` (line 120), which — when `POLL` mode is active — dispatches to `PollingTopicListener.listen()`. [1](#0-0) 

`Schedulers.boundedElastic()` is a static factory that returns the **global** shared bounded-elastic scheduler (thread cap = `10 × availableProcessors()`; task queue = 100,000). Every subscription shares this single pool. [2](#0-1) 

Each `listen()` call:
1. Allocates a new `PollingContext` (line 35).
2. Schedules an initial delay on the shared scheduler (`delaySubscription`, line 39).
3. Schedules a perpetual repeat (`RepeatSpec.times(Long.MAX_VALUE)`) on the same scheduler (lines 40–43).
4. On each tick, executes `poll()` → `Flux.fromStream(topicMessageRepository.findByFilter(...))` — a **synchronous** JDBC stream — on a scheduler thread, holding that thread for the full duration of the DB query. [3](#0-2) 

**Root cause:** No global subscription limit, no per-IP rate limit, and no connection count cap in the application layer. The only server-side guard is `maxConcurrentCallsPerConnection = 5`, which is a **per-connection** limit, not a global one. [4](#0-3) [5](#0-4) 

No `maxInboundConnections` or equivalent is set anywhere in `GrpcConfiguration`. [6](#0-5) 

The gRPC endpoint has **no authentication interceptors** — confirmed by the absence of any auth/security interceptor in the grpc Java source tree.

The same global `boundedElastic()` scheduler is also used by `TopicMessageServiceImpl`'s safety-check path, so exhaustion affects the entire service: [7](#0-6) 

### Impact Explanation

On a 4-core server the thread cap is 40. If each DB poll takes ~100 ms, the pool can sustain ~200 concurrent subscriptions at the 500 ms interval. Beyond that, poll tasks queue. Once the 100,000-task queue fills, `RejectedExecutionException` is thrown, terminating subscriptions. Between saturation and queue exhaustion, all legitimate subscribers experience exponentially growing poll delays — effectively halting real-time topic message delivery. Because the same scheduler backs the safety-check `Mono.delay` in `TopicMessageServiceImpl`, gap-recovery for all subscribers is also blocked, compounding the outage. [8](#0-7) 

### Likelihood Explanation

**Precondition:** `hiero.mirror.grpc.listener.type=POLL` must be configured (non-default; default is `REDIS`). This is a documented, supported option. Any operator who selects POLL mode is immediately exposed.

**Attacker capability:** Zero privileges required. The gRPC port (5600) is publicly reachable. A single attacker machine can open hundreds of HTTP/2 connections, each carrying 5 streams, yielding thousands of concurrent subscriptions with a trivial script (e.g., using `grpc-java` or `grpcurl` in a loop). No special knowledge beyond the public proto definition is needed. [9](#0-8) 

The attack is repeatable and requires no state — connections can be re-opened immediately if the server sheds them.

### Recommendation

1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., `hiero.mirror.grpc.maxSubscribers`).
2. **Isolate the POLL scheduler**: replace `Schedulers.boundedElastic()` with a dedicated, bounded `Schedulers.newBoundedElastic(threadCap, queueSize, "poll-listener")` so exhaustion cannot bleed into other reactive chains.
3. **Add a global connection limit** in `GrpcConfiguration` via `serverBuilder.maxConnectionIdle(...)` and `serverBuilder.maxInboundConnections(...)` (available in Netty gRPC).
4. **Add per-IP rate limiting** at the load-balancer or application level for the `subscribeTopic` RPC.

### Proof of Concept

```bash
# Requires grpcurl and a server configured with listener.type=POLL
# Open 200 connections × 5 streams = 1000 concurrent subscriptions

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":1}}' \
      -H 'content-type: application/grpc' \
      <grpc-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Monitor: legitimate subscriber now receives no messages or extreme delays
# Server logs will show boundedElastic thread pool saturation
# hiero.mirror.grpc.subscribers metric climbs without bound
```

Each background process holds an open streaming RPC. With a 4-core server (40 `boundedElastic` threads) and DB queries taking >50 ms, the pool saturates within seconds, and all poll tasks for legitimate subscribers queue indefinitely.

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
