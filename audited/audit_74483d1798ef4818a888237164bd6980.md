### Title
Unauthenticated Health Endpoint Lacks Request Deduplication, Enabling N-Fold Prometheus Query Amplification

### Summary
The `health()` method in `ImporterLagHealthIndicator` performs a full synchronous Prometheus HTTP query on every invocation with no caching, coalescing, or rate limiting. The `/actuator/health/cluster` path is exposed via ingress without authentication. An unprivileged attacker sending N concurrent requests triggers N independent Prometheus PromQL queries and N independent `parseLagByCluster()` stream pipelines, amplifying CPU, memory, and outbound network load proportionally to N.

### Finding Description
**Exact code path:**

`ImporterLagHealthIndicator.health()` (lines 48–62) unconditionally calls `prometheusClient.query(query)` (a blocking synchronous HTTP call via `SimpleClientHttpRequestFactory`) and then `evaluate()` → `parseLagByCluster()` on every invocation. There is no `@Cacheable`, no shared `CompletableFuture`/`Mono`, no `synchronized` guard, and no in-flight deduplication of any kind. [1](#0-0) 

`PrometheusApiClient.query()` issues a full blocking HTTP GET to Prometheus with a complex multi-level PromQL (rate + sum + max + join) on every call: [2](#0-1) 

The Helm ingress configuration explicitly exposes `/actuator/health/cluster` to the public internet with no authentication and no rate-limit annotation: [3](#0-2) 

**Root cause:** The `health()` method treats every HTTP request as independent work. Spring Boot's `HealthEndpoint` does not deduplicate concurrent calls to the same `HealthIndicator`; each caller gets its own execution path through `health()`.

**Why existing checks fail:** The only guard is `properties.isEnabled()` (line 49), which is a boolean flag, not a concurrency control. The exception handler (lines 58–61) only masks errors; it does not prevent amplification. There is no Spring Security configuration protecting the actuator path in the monitor module.

### Impact Explanation
Each concurrent request to `/actuator/health/cluster` causes:
1. One blocking thread occupied for the duration of the Prometheus HTTP round-trip (configurable timeout, default likely seconds).
2. One full PromQL evaluation on the Prometheus server (rate + sum + max aggregation over a 3-minute window across all importer series).
3. One JSON deserialization of the Prometheus response body.
4. One `parseLagByCluster()` stream pipeline execution.

With N=20–50 concurrent requests (trivially achievable with `curl`/`ab`/`wrk`), the monitor service exhausts its HTTP thread pool, and Prometheus receives N simultaneous expensive range queries. Because the monitor's baseline CPU is very low (it is a lightweight polling service), even N=10 concurrent health-check floods can push CPU well above the 30% amplification threshold. The Prometheus server, which is shared infrastructure, is also impacted.

### Likelihood Explanation
The attack requires only the ability to send unauthenticated HTTP GET requests to the publicly exposed ingress host on path `/actuator/health/cluster`. No credentials, tokens, or special headers are needed. The attack is trivially repeatable with any HTTP load tool and is stateless — the attacker does not need to maintain session state. The ingress host is discoverable via DNS or certificate transparency logs.

### Recommendation
1. **Add response caching** to `health()` using Spring's `@Cacheable` with a short TTL (e.g., 10–30 seconds), or store the last result in an `AtomicReference` updated by a scheduled background task (`@Scheduled`) rather than on-demand per request.
2. **Restrict the actuator endpoint** — add an ingress-level authentication annotation (e.g., Traefik `BasicAuth` middleware) or move the health endpoint to a non-public port/path.
3. **Add in-flight deduplication** — use a `Mono.cache()` or a `CompletableFuture` that is shared across concurrent callers and refreshed after completion.
4. **Apply rate limiting** at the ingress layer for `/actuator/health/*` paths.

### Proof of Concept
```bash
# Send 50 concurrent unauthenticated requests to the public health endpoint
# Replace <MONITOR_HOST> with the ingress host from values.yaml
ab -n 500 -c 50 http://<MONITOR_HOST>/actuator/health/cluster

# Observe on the monitor pod:
kubectl top pod <monitor-pod>   # CPU spikes proportionally to concurrency

# Observe on Prometheus:
# Each request generates one query matching:
# max by (cluster) (sum(rate(hiero_mirror_importer_stream_latency_seconds_sum{...}[3m])) ...)
# Visible in Prometheus /api/v1/query active queries or slow query log
```

Each of the 50 concurrent requests independently executes the full PromQL and stream pipeline, producing ~50× the baseline resource consumption with zero authentication required.

                .uri(builder -> builder.queryParam("query", encodedQuery).build())
