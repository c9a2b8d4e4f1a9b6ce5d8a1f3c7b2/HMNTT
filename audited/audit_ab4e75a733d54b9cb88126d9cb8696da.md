### Title
Unauthenticated Request Flood on `getNodes()` gRPC Endpoint Due to Absent Rate Limiting

### Summary
The `getNodes()` method in `NetworkController` accepts requests from any unauthenticated caller with no per-client or global rate limiting at the application layer. The only concurrency guard is `maxConcurrentCallsPerConnection = 5`, which limits concurrent calls per single TCP connection but imposes no cap on the number of connections or the aggregate request rate. An attacker opening many parallel gRPC connections can drive an unbounded number of simultaneous `NetworkServiceImpl.getNodes()` Flux pipelines, each triggering multiple synchronous database queries, exhausting the DB connection pool and degrading service for all users.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) immediately subscribes a new Reactor pipeline for every incoming call with no guard:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java:33-43
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)   // new DB pipeline per call
            .map(this::toNodeAddress)
            .onErrorMap(ProtoUtil::toStatusRuntimeException)
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    ...
}
```

Each `networkService::getNodes` invocation in `NetworkServiceImpl` (lines 55–77) executes at minimum three synchronous DB queries before streaming results:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java:61-66
long addressBookTimestamp = addressBookRepository.findLatestTimestamp(fileId.getId())...
long nodeStakeTimestamp   = nodeStakeRepository.findLatestTimestamp()...
var  nodeStakeMap         = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
```

followed by paginated reads via `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` (line 85–86).

**Root cause — failed assumption:**

The only server-side concurrency control is `maxConcurrentCallsPerConnection = 5` (default, `NettyProperties` line 14 / `GrpcConfiguration` line 33). This limits concurrent RPCs **per TCP connection**, not the total number of connections or the aggregate request rate. There is no per-IP connection limit, no global RPC rate limiter, and no authentication requirement.

**Why existing checks are insufficient:**

| Control | Where | Why insufficient |
|---|---|---|
| `maxConcurrentCallsPerConnection = 5` | `GrpcConfiguration` L33 | Per-connection only; attacker opens N connections → N×5 simultaneous calls |
| Traefik `circuitBreaker` + `retry` middleware | `charts/hedera-mirror-grpc/values.yaml` L157–161 | Triggers on error ratio / network errors, not on request volume; retry amplifies load |
| GCP `maxRatePerEndpoint: 250` | `values.yaml` L69 | Backend load-balancing hint for GCP Gateway, not a per-client rate limit; not enforced in non-GCP deployments |
| No throttle module in `grpc/` | (absent) | The bucket4j `ThrottleConfiguration` / `ThrottleManagerImpl` exists only in the `web3` module |
| No `@GrpcService` interceptor for rate limiting | `grpc/src/test/java/.../GrpcInterceptor.java` | Only sets endpoint context; no rate enforcement |

### Impact Explanation
Each flood request consumes a HikariCP DB connection for the duration of the paginated address-book read. With enough parallel connections the DB connection pool is exhausted (`GrpcHighDBConnections` alert threshold is 75% utilization), causing all subsequent gRPC calls — including `subscribeTopic` — to queue or fail. Because the endpoint is unauthenticated and publicly reachable on port 5600, this constitutes a practical denial-of-service against the entire gRPC service. Severity: **High** (availability impact, no authentication barrier, no remediation at runtime without infrastructure changes).

### Likelihood Explanation
The endpoint is documented and trivially invocable with `grpcurl` (no credentials, no API key). A single attacker machine can open hundreds of HTTP/2 connections (gRPC multiplexes over HTTP/2), each carrying 5 concurrent streams, producing thousands of simultaneous DB queries. The attack is repeatable, requires no special knowledge beyond the public proto definition, and is not detectable until DB saturation occurs.

### Recommendation
1. **Application-layer rate limiting**: Add a `ServerInterceptor` (analogous to the `web3` `ThrottleManagerImpl`) that enforces a global and/or per-IP request-per-second limit using bucket4j before the call reaches `NetworkController`.
2. **Connection-level cap**: Add `serverBuilder.maxConnectionsTotal(N)` or `maxConnectionAge` to `GrpcConfiguration` to bound total server-side connections.
3. **Infrastructure**: Configure Traefik `rateLimit` middleware (requests/second per source IP) in `charts/hedera-mirror-grpc/values.yaml` alongside the existing `circuitBreaker`.
4. **Response caching**: Cache the full `getNodes` response (address book changes infrequently) to avoid DB hits on repeated calls.

### Proof of Concept

```bash
# Install grpcurl; no credentials needed
# Open 200 parallel connections, each sending getNodes with limit=0 (full scan)
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done
wait
```

**Expected result**: DB connection pool saturates (observable via `hikaricp_connections_active` metric), subsequent legitimate gRPC calls return `UNAVAILABLE` or time out, and the `GrpcHighDBConnections` Prometheus alert fires. No authentication or special privilege is required.
