### Title
Unbounded gRPC Connection Exhaustion via Missing Global Connection Limit

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection(5)`, which limits RPC calls per individual TCP connection but imposes no cap on the total number of simultaneous connections. An unauthenticated attacker can open an unbounded number of TCP connections, each carrying up to 5 active streaming calls, exhausting server threads, executor queue capacity, and database connection pool resources. During a network partition, TCP connections linger without cleanup (no `maxConnectionAge` or `maxConnectionIdle` configured), compounding the exhaustion as new connections continue to arrive.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 27–35:
```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** The `NettyServerBuilder` is customized with only two settings: a shared executor and a per-connection call cap of 5. The following critical server-level controls are entirely absent:
- No `maxConnectionAge(...)` — connections live indefinitely
- No `maxConnectionIdle(...)` — idle connections are never reaped
- No `permitKeepAliveTime(...)` — no keepalive enforcement
- No global connection count ceiling (gRPC-Java's `NettyServerBuilder` has no built-in default cap on total connections)

**Failed assumption:** The design assumes that limiting calls per connection is sufficient to bound total server load. It is not — total load scales as `(number_of_connections) × 5`, and the number of connections is unbounded.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication required — public gRPC API).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` streaming RPCs (the per-connection maximum).
3. Each streaming call occupies a thread from `applicationTaskExecutor`, a slot in the shared listener buffer, and potentially a database connection via `PollingTopicMessageRetriever`.
4. During a network partition, the server cannot receive TCP FIN/RST from the attacker's side; existing connections remain in `ESTABLISHED` state and are never cleaned up (no `maxConnectionIdle` to trigger server-side teardown).
5. The attacker continues opening new connections from other IPs or after reconnecting, stacking additional resource consumption on top of the lingering zombie connections.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` only bounds calls per connection; with N connections the total is `5N`.
- No infrastructure-level connection limiting was found in the gRPC-specific configuration (the `ThrottleConfiguration` present in the codebase is scoped to the `web3` module, not `grpc`).
- No Kubernetes `NetworkPolicy` or ingress-level connection rate limiting was found applied specifically to the gRPC port.

### Impact Explanation
A successful attack causes denial of service against the gRPC mirror node. The `applicationTaskExecutor` thread pool becomes saturated, causing new legitimate RPCs to be rejected or queued indefinitely. If the executor is shared with other subsystems, broader service degradation occurs. Database connection pool exhaustion is a secondary impact when each streaming call triggers polling queries. Severity is **High** — complete unavailability of the public gRPC API with no authentication barrier.

### Likelihood Explanation
Preconditions are minimal: network reachability to port 5600 and the ability to open TCP connections. No credentials, tokens, or special protocol knowledge are required beyond standard gRPC client tooling (e.g., `grpcurl`, any gRPC stub). The attack is trivially scriptable, repeatable, and can be amplified from a single host by multiplexing connections. Network partition conditions (which worsen the impact) occur naturally in cloud environments and can also be simulated by the attacker using firewall rules to suppress RST packets. Likelihood is **High**.

### Recommendation
Apply the following additional settings inside `grpcServerConfigurer()`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveTime(nettyProperties.getPermitKeepAliveTime(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Add corresponding fields to `NettyProperties` with safe defaults (e.g., `maxConnectionAge = 300s`, `maxConnectionIdle = 60s`). Additionally, consider adding a global connection limit via a custom `ServerTransportFilter` that tracks and rejects connections above a configured ceiling, and expose `maxConnections` as a configurable property in `NettyProperties`.

### Proof of Concept
```bash
# Open 1000 connections, each with 5 concurrent subscribeTopic streaming calls
# Requires grpcurl and a valid topic ID on the target

for i in $(seq 1 1000); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <TARGET_HOST>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# To simulate network partition (prevent FIN/RST from reaching server):
# iptables -A OUTPUT -p tcp --dport 5600 -j DROP
# Then repeat the loop above — zombie connections accumulate server-side
# while new connections continue to be accepted.
```
