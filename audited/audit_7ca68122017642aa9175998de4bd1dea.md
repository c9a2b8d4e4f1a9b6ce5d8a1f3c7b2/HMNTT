### Title
Cache-Amplified Batch DB Query DoS via Non-Existent Slot Key Accumulation in `findStorageBatch()`

### Summary
An unauthenticated attacker can accumulate up to 1,500 unique non-existent slot keys in the per-contract Caffeine cache by sending one request per unique key. Because non-existent slots are never written to `contractStateCache`, every subsequent request for any of those keys bypasses the 2-second state cache and triggers a full `WHERE slot IN (…1500 keys…)` batch query against the database. At the default 500 RPS throttle, this produces a sustained 750,000-slot-lookup/second DB load from a single attacker with no privileges.

### Finding Description

**Exact code path**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` first. A cache miss (null return) unconditionally calls `findStorageBatch()`.

`findStorageBatch()` (lines 85–122):
1. Line 90 — `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally inserts the caller-supplied key into the per-contract slot-key cache.
2. Lines 91–101 — reads **all** keys currently in that per-contract cache into `cachedSlots`.
3. Line 103 — issues `findStorageBatch(contractId, cachedSlots)` — a native SQL `WHERE slot IN (:slots)` with the full list.
4. Lines 106–114 — only slots that **exist in the DB** are written into `contractStateCache`. Non-existent slots produce no cache entry.

**Root cause / failed assumption**

The design assumes that after a batch query the requested key will be stored in `contractStateCache` and future calls will hit the cache. This assumption fails for non-existent slots: `contractStateCache.get()` always returns null for them, so `findStorageBatch()` is called on every single request, each time carrying the full accumulated key set.

**Cache parameters (defaults)**

| Cache | Config |
|---|---|
| `contractState` | `expireAfterWrite=2s, maximumSize=25000` |
| `slotsPerContract` | `expireAfterAccess=5m, maximumSize=1500` |
| `contractSlots` | `expireAfterAccess=5m, maximumSize=3000` |

**Exploit flow**

*Phase 1 – accumulation (≈3 s at 500 RPS):*
Send 1,500 requests, each for a distinct non-existent slot key against the same contract. Each call hits `putIfAbsent` (line 90), growing the per-contract cache to its 1,500-entry maximum. The `expireAfterAccess=5m` policy keeps all entries alive as long as they are periodically touched.

*Phase 2 – sustained amplification:*
Send requests for any of the 1,500 non-existent keys at 500 RPS. For each request:
- `contractStateCache.get()` → null (non-existent slot, never stored).
- `findStorageBatch()` is entered.
- `cachedSlots` contains all 1,500 keys.
- One `SELECT … WHERE slot IN (…1500 entries…)` is issued to PostgreSQL.

Result: 500 req/s × 1,500 slots/query = **750,000 slot-row lookups/second** against the DB, compared to ≤500 for normal single-slot traffic — a >1,000× amplification.

**Why existing checks are insufficient**

- `ThrottleManagerImpl` (lines 37–48) enforces `requestsPerSecond=500` and `gasPerSecond` limits. Both operate at the HTTP-request level; neither bounds the number of DB rows touched per request.
- `gasLimitBucket` tracks EVM gas, which is unrelated to the number of SQL parameters in a batch query.
- The `maximumSize=1500` on `slotsPerContract` caps the batch size but does not prevent the attacker from keeping the cache saturated with adversarial keys.
- There is no per-IP, per-contract, or per-slot-key rate limit on `findStorageBatch()` invocations.

### Impact Explanation

Each attacker request causes the database to evaluate a 1,500-element `IN` clause instead of a single-row point lookup. PostgreSQL must parse, plan, and execute a significantly larger query for each call. At 500 RPS this translates to a sustained, disproportionate increase in DB CPU, I/O, and connection time — well above the 30% threshold stated in scope. Because the `contractSlotsCache` holds up to 3,000 contracts, an attacker targeting multiple contracts multiplies the effect further. The mirror node's `statementTimeout=3000ms` provides a partial backstop but does not prevent the amplification; it only caps individual query duration.

### Likelihood Explanation

No authentication, no special account, and no on-chain funds are required. The only precondition is knowledge of a valid contract ID (trivially obtained from the public mirror REST API). The attack is fully automatable with a simple HTTP client loop. The accumulation phase completes in approximately 3 seconds; the amplification phase is indefinitely sustainable. The attacker need not exceed the published rate limit at any point.

### Recommendation

1. **Cap batch size before querying**: In `findStorageBatch()`, truncate `cachedSlots` to a configurable maximum (e.g., 100) before passing it to `findStorageBatch()` in the repository. Keys beyond the cap should fall back to the single-key `findStorage()` path.
2. **Cache negative results**: After a batch query, explicitly store a sentinel value (e.g., `EMPTY_VALUE`) in `contractStateCache` for every key in `cachedSlots` that was *not* returned by the DB. This prevents non-existent slots from triggering `findStorageBatch()` on every call.
3. **Bound per-request slot-key insertion**: Reject or ignore `putIfAbsent` when the per-contract cache is already at `maximumSize`, rather than silently evicting LRU entries and allowing the attacker to keep the cache saturated with adversarial keys.
4. **Add a DB-query-level rate limit** independent of the HTTP request throttle, or expose a configurable `maxBatchSlots` property.

### Proof of Concept

```python
import requests, threading

MIRROR_NODE = "http://<mirror-node>:8545"
CONTRACT    = "0x<valid_contract_address>"

# Phase 1: fill per-contract slot cache with 1500 non-existent keys
def make_slot(i):
    return "0x" + format(i, '064x')   # 32-byte slot key, non-existent in DB

def call_slot(slot):
    requests.post(f"{MIRROR_NODE}/api/v1/contracts/call", json={
        "to": CONTRACT,
        "data": "0x" + "20" + slot[2:],   # SLOAD opcode targeting the slot
        "gas": 50000
    })

# Accumulate 1500 unique non-existent slots (≈3 s at 500 RPS)
for i in range(1500):
    call_slot(make_slot(i))

# Phase 2: sustained amplification — every request triggers a 1500-slot batch query
def flood():
    while True:
        for i in range(1500):
            call_slot(make_slot(i))

threads = [threading.Thread(target=flood) for _ in range(10)]
for t in threads: t.start()
# Observe DB CPU/query-time spike on the mirror node's PostgreSQL instance
```
