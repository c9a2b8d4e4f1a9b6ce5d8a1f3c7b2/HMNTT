### Title
Missing `@Size` Constraint on `nodeIds` Enables Unbounded HashSet Allocation in `getNetworkNodes()`

### Summary
`NetworkNodeRequest.nodeIds` has no `@Size` upper-bound constraint, unlike analogous fields in other DTOs (`RegisteredNodesRequest.registeredNodeIds` uses `@Size(max = 2)`). An unauthenticated attacker can submit up to Tomcat's default `maxParameterCount` (10,000) `node.id=eq:X` values combined with a tight GT/LT range that excludes all of them, forcing the server to allocate and populate a large `HashSet<Long>`, stream-filter it, find zero matches, and return early — all wasted work, repeatable on every request.

### Finding Description
**File:** `rest-java/src/main/java/org/hiero/mirror/restjava/dto/NetworkNodeRequest.java`, lines 36–38
**File:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, lines 106–130

`NetworkNodeRequest.nodeIds` is declared as:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();
```

There is no `@Size` annotation. Compare with `RegisteredNodesRequest`:

```java
@Size(max = 2)
private List<NumberRangeParameter> registeredNodeIds = List.of();
```

In `getNetworkNodes()`, every EQ parameter is unconditionally added to a `HashSet<Long>`:

```java
final Set<Long> nodeIds = new HashSet<>();
for (final var nodeIdParam : nodeIdParams) {
    if (nodeIdParam.operator() == RangeOperator.EQ) {
        nodeIds.add(nodeIdParam.value());
    } else if (nodeIdParam.hasLowerBound()) { ... }
    else if (nodeIdParam.hasUpperBound()) { ... }
}
```

Then, if EQ values are present, they are stream-filtered against the GT/LT range:

```java
nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
if (nodeIdArray.length == 0) {
    return List.of();
}
```

If all EQ values fall outside the range, the method returns `List.of()` — but only after allocating and populating the full `HashSet` and running the stream filter. The constant `MAX_REPEATED_QUERY_PARAMETERS = 100` is defined in `Constants.java` but is never applied to this field; the `RequestParameterArgumentResolver` only enforces `@Size` JSR-303 annotations, which are absent here.

### Impact Explanation
Any unauthenticated caller can send a crafted GET request to `/api/v1/network/nodes` with thousands of `node.id=eq:X` values (up to Tomcat's `maxParameterCount`, default 10,000) combined with a narrow range (e.g., `node.id=gt:9999999`) that excludes all EQ values. Each such request forces the server to allocate a `HashSet` of up to 10,000 `Long` objects, iterate over all of them, and stream-filter them — all discarded work. This is a CPU and heap amplification vector. Repeated at high frequency from multiple clients, it degrades service for legitimate users. Severity matches the stated scope: griefing with no direct economic damage.

### Likelihood Explanation
No authentication is required. The endpoint is public. The attack requires only a crafted HTTP GET request with repeated query parameters, trivially scriptable with any HTTP client (`curl`, `ab`, `wrk`). The `MAX_REPEATED_QUERY_PARAMETERS = 100` constant already exists in the codebase, demonstrating awareness of this class of risk — it simply was not applied here. The asymmetry between `RegisteredNodesRequest` (protected with `@Size(max = 2)`) and `NetworkNodeRequest` (unprotected) makes this an oversight rather than an intentional design choice.

### Recommendation
Add a `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` constraint to the `nodeIds` field in `NetworkNodeRequest`, consistent with how `RegisteredNodesRequest.registeredNodeIds` is bounded:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = MAX_REPEATED_QUERY_PARAMETERS)
private List<NumberRangeParameter> nodeIds = List.of();
```

Import `MAX_REPEATED_QUERY_PARAMETERS` from `Constants`. This will cause `RequestParameterArgumentResolver` to reject oversized lists via JSR-303 validation before the service layer is reached.

### Proof of Concept
```bash
# Build a query with 200 EQ node.id values all below 1,
# combined with gt:1000 so none can match.
PARAMS="node.id=gt:1000"
for i in $(seq 1 200); do
  PARAMS="${PARAMS}&node.id=eq:${i}"
done

curl -s "http://<mirror-node-host>/api/v1/network/nodes?${PARAMS}"
# Returns {"nodes":[],"links":{}} after allocating a 200-entry HashSet
# and stream-filtering it on every call.

# Flood at high rate:
ab -n 10000 -c 50 "http://<mirror-node-host>/api/v1/network/nodes?${PARAMS}"
```

Each request forces server-side allocation and filtering of the full EQ set with zero useful work performed.
