### Title
Unauthenticated HTTP Fetch of Node Topology Enables Transaction Misdirection via DNS Spoofing or MITM

### Summary
When `network=other` is configured, `buildNetworkFromMirrorNodes` fetches consensus node addresses from `cfg.mirrorRest` over plain HTTP with no response authentication. An attacker who can poison DNS or intercept the HTTP response can inject arbitrary gRPC endpoints, causing the pinger to submit all signed transactions to attacker-controlled nodes. The attacker can then suppress, delay, or selectively forward those transactions, manipulating their ordering in consensus history.

### Finding Description
**Code path:**
- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes`, lines 36–72
- `pinger/sdk_client.go`, `newClient`, lines 17–22

`buildNetworkFromMirrorNodes` constructs a URL from `cfg.mirrorRest` (set via `HIERO_MIRROR_PINGER_REST` or `-mirror-rest` flag, with no HTTPS enforcement) and issues a plain `http.Get`. The response JSON is decoded and the `host:port` pairs from `service_endpoints` are placed directly into the network map returned to `hiero.ClientForNetwork(netmap)`. No TLS is required, no response signature is checked, no allowlist of valid node IPs/domains is consulted.

The only checks present are:
- HTTP status code range (lines 90–93) — bypassed by a server the attacker controls
- JSON decode success (lines 96–98) — trivially satisfied
- Non-empty network map (lines 127–129) — trivially satisfied

Once `hiero.ClientForNetwork(netmap)` is called with poisoned endpoints, every subsequent `cryptoTransfer.Execute(client)` in `submitWithRetry` (`transfer.go` lines 33–42) sends the operator-signed transaction to the attacker's gRPC listener instead of real consensus nodes.

**Root cause:** The function assumes the HTTP response from `cfg.mirrorRest` is trustworthy. There is no mechanism to verify that the returned node addresses belong to the legitimate network.

### Impact Explanation
The attacker receives fully signed `CryptoTransfer` transactions. They can:
1. **Suppress** them entirely — the pinger's health-check transactions never reach consensus, causing monitoring to report false failures or masking real outages.
2. **Delay and replay** them — hold transactions and submit them to the real network at a chosen time, altering their position in consensus-ordered history relative to other transactions.
3. **Observe** transaction content and timing patterns of the operator account.

Because the pinger is a liveness/health monitoring tool, sustained suppression can blind operators to real network degradation.

### Likelihood Explanation
The attack is scoped to `network=other` deployments (custom/private networks). In Kubernetes or Docker Compose environments — the typical deployment target for a self-hosted mirror node stack — DNS is resolved via an internal resolver. An attacker who has compromised any workload in the same namespace/network can poison DNS for the pinger's mirror REST hostname without any privileges on the pinger pod itself. Alternatively, if `cfg.mirrorRest` uses a plain `http://` URL (no TLS), any on-path attacker (e.g., a compromised sidecar or network device) can perform a trivial MITM. No credentials or host-level access to the pinger are required.

### Recommendation
1. **Enforce HTTPS**: Reject `cfg.mirrorRest` URLs that do not use `https://` at config load time (`config.go`, `loadConfig`).
2. **Pin or validate node identity**: After fetching topology, cross-check returned node account IDs and addresses against a locally configured allowlist or a signed/pinned address book. Do not trust the HTTP response as the sole source of truth.
3. **Use TLS with certificate verification**: Configure the `http.Client` with a strict TLS config; do not allow insecure transports.
4. **Periodic re-fetch with consistency check**: If topology is re-fetched at runtime, alert and abort if the new topology differs significantly from the previously trusted one.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. On the same network, run a DNS server that resolves `mirror-rest` to an attacker-controlled IP.
3. On the attacker IP, serve a valid JSON response from `/api/v1/network/nodes` with `service_endpoints` pointing to an attacker-controlled gRPC listener (e.g., `attacker-ip:50211`, account ID `0.0.3`).
4. Start the pinger. `buildNetworkFromMirrorNodes` fetches the poisoned topology; `hiero.ClientForNetwork` builds a client pointing solely at the attacker's gRPC server.
5. The attacker's gRPC listener receives every signed `CryptoTransfer`. By never forwarding them to real consensus nodes, all pinger transactions are suppressed. By forwarding them with an arbitrary delay, their consensus ordering is shifted relative to other network activity.
