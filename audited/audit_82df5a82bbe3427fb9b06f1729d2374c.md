### Title
Unauthenticated `/metrics` Endpoint Exposes Prometheus Scrape Counter, Enabling Monitoring-Aware Attack Timing

### Summary
The Rosetta service exposes its `/metrics` endpoint to any unprivileged external caller with no authentication. The endpoint is served directly by `promhttp.Handler()` with no access control in the route registration or any middleware layer. This allows an attacker to read `promhttp_metric_handler_requests_total` and all other Prometheus metrics, revealing whether automated monitoring is active, its scrape interval, and precisely when it stops — information that can be used to time tampering of Hashgraph history during monitoring gaps.

### Finding Description
**Code path:**

In `rosetta/app/middleware/metrics.go`, the `Routes()` function registers the `/metrics` endpoint at line 64–73:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,           // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth wrapper
        },
    }
}
```

`promhttp.Handler()` is the standard Prometheus HTTP handler. It automatically exposes the `promhttp_metric_handler_requests_total` counter, which increments on every scrape of the `/metrics` endpoint itself.

**Root cause:** No authentication middleware is applied to the `/metrics` route. The middleware chain in `rosetta/main.go` (lines 217–219) is:

```
MetricsMiddleware → TracingMiddleware → CorsMiddleware
```

None of these enforce access control. `TracingMiddleware` in `rosetta/app/middleware/trace.go` (line 19) does reference `metricsPath` in `internalPaths`, but only to suppress log verbosity (debug vs. info level) — it does not gate access:

```go
var internalPaths = map[string]bool{livenessPath: true, metricsPath: true, readinessPath: true}
// ...
if internalPaths[path] {
    log.Debug(message)   // only affects log level, not access
}
```

The unit test at `rosetta/app/middleware/metrics_test.go` line 15–23 confirms the endpoint returns HTTP 200 with no credentials and explicitly asserts `promhttp_metric_handler_requests_total` is present in the response body.

**Exploit flow:**
1. Attacker sends unauthenticated `GET /metrics` repeatedly over time.
2. Reads `promhttp_metric_handler_requests_total{code="200"}` — a monotonically increasing counter of every legitimate Prometheus scrape.
3. By observing the rate of increase, attacker determines the scrape interval (e.g., every 15 seconds).
4. When the counter stops incrementing, attacker knows monitoring has been disabled or the Prometheus scraper has lost connectivity.
5. Attacker uses this window to perform tampering operations on Hashgraph history without triggering metric-based alerts.

**Why existing checks fail:** There are no existing checks. The route is registered with a bare `promhttp.Handler()` and no authentication wrapper is applied at any layer in the middleware chain or router configuration.

### Impact Explanation
An unprivileged external attacker gains precise, real-time visibility into the operational monitoring state of the Rosetta node. Beyond `promhttp_metric_handler_requests_total`, the full metrics dump also exposes `hiero_mirror_rosetta_request_duration`, `hiero_mirror_rosetta_request_bytes`, and inflight request gauges — revealing traffic patterns, endpoint usage, and system load. In the context of Hashgraph history integrity, this constitutes a reconnaissance primitive: an attacker can determine the exact window during which tampering would go undetected by automated monitoring systems.

### Likelihood Explanation
The precondition is only that the Rosetta port is reachable from the attacker's network — a common deployment scenario for a public blockchain API node. No credentials, tokens, or special headers are required. The attack is trivially repeatable with a single HTTP GET request and requires no exploitation of any other vulnerability. Any internet-facing deployment is affected by default.

### Recommendation
Wrap the `/metrics` handler with an authentication check before registering it in `Routes()`. A minimal fix in `rosetta/app/middleware/metrics.go`:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,
            requireMetricsAuth(promhttp.Handler()).ServeHTTP,
        },
    }
}

func requireMetricsAuth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // check Bearer token or Basic auth from config
        // return 401 if not authorized
        next.ServeHTTP(w, r)
    })
}
```

Alternatively, bind the `/metrics` endpoint to a separate, non-public port (e.g., localhost-only or a dedicated management port), consistent with the Prometheus security model recommendation of not exposing the metrics endpoint on the same public-facing port as the API.

### Proof of Concept
```bash
# No credentials required — returns HTTP 200 with full metrics
curl -s http://<rosetta-host>:<port>/metrics | grep promhttp_metric_handler_requests_total

# Example output:
# promhttp_metric_handler_requests_total{code="200"} 47
# promhttp_metric_handler_requests_total{code="500"} 0

# Poll every 5 seconds to determine scrape interval and detect monitoring gaps
while true; do
  COUNT=$(curl -s http://<rosetta-host>:<port>/metrics \
    | grep 'promhttp_metric_handler_requests_total{code="200"}' \
    | awk '{print $2}')
  echo "$(date): scrape_count=$COUNT"
  sleep 5
done
# When COUNT stops incrementing → monitoring is down → safe window for tampering
```
