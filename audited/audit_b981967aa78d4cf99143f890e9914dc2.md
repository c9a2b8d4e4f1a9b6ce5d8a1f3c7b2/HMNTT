### Title
Unauthenticated `/mempool` Endpoint with Unconditional HTTP 500 Enables Botnet-Driven Circuit Breaker Exhaustion

### Summary
The `/mempool` endpoint in `rosetta/app/services/mempool_service.go` unconditionally returns `ErrNotImplemented`, which the rosetta-sdk-go framework serializes as HTTP 500. No authentication is required. The Traefik middleware chain (active in production via `values-prod.yaml`) includes a `retry` middleware that amplifies each request into 4 HTTP 500 responses, and a circuit breaker that trips when 5xx responses exceed 25% of total traffic. The `inFlightReq` middleware is scoped per-IP (depth: 1), which a botnet bypasses by distributing requests across many IPs, allowing it to generate a sustained flood of HTTP 500 responses sufficient to trip the circuit breaker and cause a network partition.

### Finding Description

**Code path:**

`rosetta/app/services/mempool_service.go`, lines 22–27:
```go
func (m *mempoolAPIService) Mempool(
    _ context.Context,
    _ *types.NetworkRequest,
) (*types.MempoolResponse, *types.Error) {
    return nil, errors.ErrNotImplemented
}
```
`ErrNotImplemented` is defined in `rosetta/app/errors/errors.go` line 70 as `newError(NotImplemented, 111, false)`. The rosetta-sdk-go framework maps any returned `*types.Error` to an HTTP 500 response body. Every single call to `/mempool` or `/mempool/transaction` returns HTTP 500 — deterministically, with no conditions.

**Middleware chain** (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–163):
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1          # ← per-IP limit
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true   # ← per-hostname, NOT per-IP
  - retry:
      attempts: 3
      initialInterval: 100ms
```

**Middleware activation:** `global.middleware: false` by default, but `charts/hedera-mirror/values-prod.yaml` line 8 sets `global.middleware: true`, activating the full chain in production. The middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml`, line 3) gates on `{{ if and .Values.global.middleware .Values.middleware }}`.

**Root cause and failed assumptions:**

1. The `inFlightReq` middleware limits concurrent requests to 5 **per source IP** (`ipStrategy.depth: 1`). A botnet with N IPs can sustain 5N concurrent in-flight requests simultaneously — bypassing this per-IP cap entirely.
2. The `retry` middleware (3 attempts) means each original request that reaches the backend generates **4 HTTP 500 responses** (1 original + 3 retries). This is a 4× amplification of the 5xx ratio seen by the circuit breaker.
3. The `rateLimit` (`average: 10, requestHost: true`) limits by hostname, not by IP. While this caps total throughput at ~10 req/s regardless of botnet size, the `inFlightReq` per-IP limit is the constraint a botnet bypasses — allowing it to keep the pipeline saturated with concurrent requests even when individual IPs would be capped.
4. The circuit breaker expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` fires when 5xx responses exceed 25% of **all** responses. During low-traffic periods, even a modest flood of `/mempool` requests can push the ratio above this threshold.

**Exploit flow:**

- Attacker controls a botnet of N IPs.
- Each IP opens 5 concurrent POST requests to `/rosetta/mempool` (valid JSON body required by the asserter, but trivially constructable).
- Each request returns HTTP 500 immediately; the `retry` middleware retries it 3 more times → 4 HTTP 500 responses per original request.
- The `inFlightReq` per-IP cap (5) is bypassed: N IPs × 5 concurrent = 5N simultaneous requests.
- The circuit breaker evaluates `ResponseCodeRatio(500, 600, 0, 600)` across all traffic. If legitimate traffic is low (e.g., off-peak), the attacker's 5xx flood exceeds 25% of total responses.
- Circuit breaker trips → Traefik returns 503 for **all** `/rosetta/*` paths → complete network partition of the Rosetta API.

### Impact Explanation

When the circuit breaker trips, Traefik stops forwarding **all** requests to the Rosetta backend — not just `/mempool`. Every endpoint (`/rosetta/account`, `/rosetta/block`, `/rosetta/construction`, `/rosetta/network`, etc.) becomes unavailable with HTTP 503. Exchange integrations relying on the Rosetta API for balance queries, block data, and transaction construction are completely severed. The circuit breaker remains open until the error ratio drops below the threshold, which cannot happen while the attack continues since the backend is no longer receiving requests to generate successful responses. This is a full denial-of-service against the Rosetta API layer.

Severity: **High**. The impact is total API unavailability. The `/mempool` endpoint is publicly listed in the ingress (`/rosetta/mempool`) and requires no credentials.

### Likelihood Explanation

**Preconditions:** None beyond network access to the Rosetta ingress. No authentication, no API key, no prior knowledge beyond the public Rosetta spec. The endpoint is documented as part of the standard Rosetta API surface.

**Feasibility:** The attack requires only valid JSON conforming to the `NetworkRequest` schema (network identifier), which is publicly documented. A botnet is not strictly required — even a single IP can attempt this during low-traffic periods — but a botnet makes it reliable at any traffic level by bypassing the per-IP `inFlightReq` cap.

**Repeatability:** The circuit breaker resets after the error ratio drops, but the attacker can immediately re-trigger it. The attack is trivially scriptable and repeatable.

**Realistic attacker:** Any party motivated to disrupt exchange integrations with the Hiero network (e.g., competing actors, extortion, coordinated disruption during a market event).

### Recommendation

1. **Remove `/mempool` from the public ingress** (`charts/hedera-mirror-rosetta/values.yaml`, line 131). Since the endpoint is permanently unimplemented, it should not be publicly routable.
2. **Return HTTP 404 or 501 instead of HTTP 500** for unimplemented endpoints. The circuit breaker expression only counts `ResponseCodeRatio(500, 600, ...)`. A 4xx response would not contribute to the circuit breaker threshold. This requires either a custom error mapping in the rosetta-sdk-go controller or a Traefik error-rewriting middleware.
3. **Change the `inFlightReq` source criterion to `requestHost: true`** (matching the `rateLimit` criterion) so that the per-host concurrent request cap is consistent and cannot be bypassed by distributing across IPs.
4. **Scope the circuit breaker per-route** rather than globally, so that a permanently-failing stub endpoint cannot affect the health signal of functional endpoints.
5. **Add a Traefik `ipAllowList` or authentication middleware** in front of the Rosetta ingress if the API is intended only for known exchange partners.

### Proof of Concept

**Prerequisites:** Production deployment with `global.middleware: true` (as set in `values-prod.yaml`). Rosetta ingress accessible at `$HOST`.

```bash
# Minimal valid NetworkRequest body (public Rosetta spec)
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'

# Single-IP baseline: confirm HTTP 500
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://$HOST/rosetta/mempool \
  -H "Content-Type: application/json" \
  -d "$BODY"
# Expected: 500

# Botnet simulation: distribute across N IPs, 5 concurrent per IP
# Each node runs:
for i in $(seq 1 5); do
  curl -s -o /dev/null \
    -X POST https://$HOST/rosetta/mempool \
    -H "Content-Type: application/json" \
    -d "$BODY" &
done
wait

# After sustained flood (seconds to minutes depending on legitimate traffic volume),
# probe a legitimate endpoint:
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://$HOST/rosetta/network/list \
  -H "Content-Type: application/json" \
  -d "$BODY"
# Expected after circuit breaker trips: 503 (network partition achieved)
