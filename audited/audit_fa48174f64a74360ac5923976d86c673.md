### Title
Unauthenticated Three-Part UNION ALL Query Amplification Enables DB Connection Pool Exhaustion DoS on Token Allowances Endpoint

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` is publicly accessible with no application-level rate limiting. When an attacker supplies both `spender.id` and `token.id` closed-range bounds (e.g., `gte`+`lte` for each), `extractTokenMultiUnionQuery()` produces a three-part `UNION ALL` query via `TokenAllowanceService.getQuery()`, each sub-query carrying its own `LIMIT $2` clause. Flooding the endpoint with thousands of such requests exhausts the PostgreSQL connection pool, rendering the mirror node REST API unavailable.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:18` registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/controllers/tokenAllowanceController.js:68-81` — `getAccountTokenAllowances` calls `extractTokenMultiUnionQuery` then immediately executes the resulting query with no concurrency guard: [2](#0-1) 

`rest/controllers/tokenAllowanceController.js:22-59` — `extractTokenMultiUnionQuery` builds `lower`, `inner`, and `upper` filter sets from the user-supplied `spender.id` and `token.id` range parameters: [3](#0-2) 

`rest/service/tokenAllowanceService.js:63-74` — `getQuery` emits up to three sub-queries joined with `UNION ALL`, each with its own `LIMIT $2`: [4](#0-3) 

The resulting SQL for a closed-range request is:
```sql
(select * from token_allowance where owner=$1 and amount>0 and spender=$3 and token_id>=$4 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender>$5 and spender<$6 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender=$7 and token_id<=$8 order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
```

The inner sub-query (`spender > X and spender < Y` with a maximally wide range) forces a full index range scan over the `token_allowance` table for that owner, bounded only by `LIMIT 100`. Three such scans run per request.

**Root cause:** The REST server middleware stack (`rest/server.js:68-97`) contains no application-level rate limiting for this endpoint: [5](#0-4) 

The only throttling in the codebase (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) applies exclusively to the EVM/web3 module, not the REST API: [6](#0-5) 

**Failed assumption:** The design assumes infrastructure-level Traefik middleware (defined in `charts/hedera-mirror-rest/templates/middleware.yaml`) will enforce rate limits, but this is optional and not enforced in the application itself. An attacker bypassing the load balancer (direct node access, misconfigured ingress, or simply overwhelming the rate limiter with distributed sources) faces no application-side defense.

### Impact Explanation

Each crafted request causes three concurrent index scans on `token_allowance`. The PostgreSQL connection pool has a finite size (typically 10–20 connections for a Node.js REST service). With thousands of concurrent requests each holding a DB connection for the duration of three sequential sub-queries, the pool is exhausted. Subsequent legitimate requests queue indefinitely or time out, making the mirror node REST API completely unavailable. This affects all mirror node consumers: wallets, explorers, dApps, and any service relying on the REST API for account/allowance data.

### Likelihood Explanation

No authentication or API key is required. The endpoint is publicly documented and reachable by any internet user. The attack requires only a standard HTTP client and knowledge of the query parameter format (publicly documented in `rest/api/v1/openapi.yml`). It is trivially scriptable and repeatable. A single attacker with modest bandwidth can sustain the attack indefinitely.

### Recommendation

1. **Application-level rate limiting:** Add a per-IP rate limiter (e.g., `express-rate-limit`) as Express middleware before route handlers, specifically for the `/allowances/tokens` endpoint or globally.
2. **DB query timeout:** Set a statement timeout on the PostgreSQL connection (e.g., `SET statement_timeout = '5s'`) so runaway queries are killed rather than holding connections.
3. **Connection pool limit with queue rejection:** Configure `pg-pool` to reject (not queue) requests when the pool is exhausted, returning HTTP 503 immediately rather than stalling.
4. **Require authentication for range queries:** Require an API key or authenticated session when both `spender.id` and `token.id` range bounds are supplied simultaneously (the three-part UNION path).

### Proof of Concept

```bash
# Trigger the three-part UNION ALL path with maximum limit
ENDPOINT="https://<mirror-node>/api/v1/accounts/0.0.1/allowances/tokens"
PARAMS="spender.id=gte:1&spender.id=lte:9999999999&token.id=gte:1&token.id=lte:9999999999&limit=100"

# Flood with concurrent requests (no auth required)
for i in $(seq 1 5000); do
  curl -s "${ENDPOINT}?${PARAMS}" &
done
wait

# Result: DB connection pool exhausted; subsequent requests return 503/timeout
curl -v "${ENDPOINT}"
```

    const bounds = {
      primary: new Bound(filterKeys.SPENDER_ID, 'spender'),
      secondary: new Bound(filterKeys.TOKEN_ID, 'token_id'),
    };
    let limit = defaultLimit;
    let order = orderFilterValues.ASC;

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.SPENDER_ID:
          bounds.primary.parse(filter);
          break;
        case filterKeys.TOKEN_ID:
          bounds.secondary.parse(filter);
          break;
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        default:
          break;
      }
    }

    this.validateBounds(bounds);

    return {
      bounds,
      lower: this.getLowerFilters(bounds),
      inner: this.getInnerFilters(bounds),
      upper: this.getUpperFilters(bounds),
      order,
      ownerAccountId,
      limit,
    };
