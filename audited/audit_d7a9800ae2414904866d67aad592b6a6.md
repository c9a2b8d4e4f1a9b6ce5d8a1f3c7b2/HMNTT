### Title
Counter Exhaustion via Non-Matching Requests Bypasses Filter-Based Protections in `RequestProperties.test()`

### Summary
In `RequestProperties.test()`, the `AtomicLong counter` is incremented via `counter.getAndIncrement()` at line 46 — before any filter matching occurs. This means every request that passes the initial `rate == 0` guard consumes the configured `limit`, regardless of whether the request matches any `RequestFilter`. An unprivileged attacker can deliberately flood the endpoint with non-matching requests to exhaust the counter, causing the filter to permanently return `false` for all subsequent requests, including malicious ones the filter was designed to block.

### Finding Description

**Exact code path:**

`RequestProperties.java`, `test()`, line 46:
```java
if (rate == 0 || counter.getAndIncrement() >= limit) {
    return false;
}
``` [1](#0-0) 

The counter is incremented unconditionally before the filter-matching loop at lines 54–58:
```java
for (var filter : filters) {
    if (filter.test(contractCallRequest)) {
        return true;
    }
}
``` [2](#0-1) 

**Root cause:** The `limit` field is semantically intended to cap how many times the filter action is applied to *matching* requests. However, the implementation counts *all* requests that enter `test()`, including those that do not match any `RequestFilter`. Once `counter >= limit`, the method returns `false` unconditionally, disabling the filter for all future requests.

**Failed assumption:** The developer assumed `limit` would only be consumed by requests that match the configured filters. In reality, non-matching requests silently consume the limit.

**Exploit flow:**
1. Admin configures a `RequestProperties` entry with `action = REJECT`, a set of `filters` targeting malicious patterns (e.g., calls to a specific contract address), and a finite `limit` (e.g., `10000`).
2. Attacker sends requests that do NOT match the configured filters (e.g., calls to a benign contract address). Each such request still increments the counter.
3. After `limit` total requests (matching or not), `counter.getAndIncrement() >= limit` becomes true and `test()` returns `false` permanently.
4. The `REJECT` filter is now disabled. The attacker sends malicious requests targeting the originally blocked contract — they pass through unblocked.

**Why existing checks are insufficient:**

The global `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` in `ThrottleManagerImpl.throttle()` still apply: [3](#0-2) 

However, these only limit throughput — they do not prevent counter exhaustion over time. At 500 req/s, a `limit` of `10000` is exhausted in 20 seconds. The `limit` field defaults to `Long.MAX_VALUE`: [4](#0-3) 

So the vulnerability is only active when an operator explicitly configures a finite `limit`, but that is a documented, supported configuration option.

### Impact Explanation
When a finite `limit` is configured on a `REJECT`-action filter, an attacker can permanently disable that filter by exhausting the counter with non-matching requests. All subsequent requests — including those the filter was designed to block — bypass the protection. This undermines the operator's ability to enforce targeted request-level access controls (e.g., blocking calls to specific contracts or from specific addresses), potentially allowing abuse of the node's execution resources or exploitation of contracts that should be inaccessible.

### Likelihood Explanation
The attack requires no authentication or special privileges — only the ability to send HTTP requests to the web3 endpoint. The attacker does not need to know the exact filter configuration; they only need to send requests that don't match the filter (which is trivially achievable by varying the `to` address or `data` field). The exhaustion is bounded only by the global rate limit, making it feasible within seconds to minutes depending on the configured `limit`. The attack is repeatable: once the counter is exhausted, the filter stays disabled permanently (the counter is never reset).

### Recommendation
Move `counter.getAndIncrement()` to after the filter-matching logic, so the counter is only incremented when a request actually matches a filter:

```java
@Override
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0) {
        return false;
    }

    if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
        return false;
    }

    boolean matches = filters.isEmpty();
    for (var filter : filters) {
        if (filter.test(contractCallRequest)) {
            matches = true;
            break;
        }
    }

    if (!matches) {
        return false;
    }

    // Only consume the limit for requests that actually match
    return counter.getAndIncrement() < limit;
}
```

This ensures `limit` semantically means "apply this filter action to at most N *matching* requests," which is the intended behavior and prevents counter exhaustion via non-matching traffic.

### Proof of Concept

**Preconditions:**
- Mirror node web3 endpoint is publicly accessible.
- Operator has configured a `RequestProperties` with `action = REJECT`, `filters = [{field: TO, type: EQUALS, expression: "0xMaliciousContract"}]`, and `limit = 10000`.

**Steps:**
1. Send 10,001 POST requests to `/api/v1/contracts/call` with `"to": "0xBenignContract"` (does not match the filter). Each request passes the global rate limit check and increments the counter.
2. Verify: after 10,001 requests, send a request with `"to": "0xMaliciousContract"`. Observe that it is **not** rejected — the filter has been disabled.
3. The attacker can now freely call `0xMaliciousContract` without restriction.

**Expected (correct) behavior:** The counter should only increment when `to == 0xMaliciousContract`, so 10,001 non-matching requests should have no effect on the filter's ability to block malicious requests.
