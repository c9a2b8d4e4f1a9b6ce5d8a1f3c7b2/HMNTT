### Title
Unbounded `latest_token_balance` CTE Full Table Scan on `token_account` via Unauthenticated `GET /accounts`

### Summary
The `getEntityBalanceQuery()` function in `rest/accounts.js` unconditionally emits a `WITH latest_token_balance AS (SELECT … FROM token_account WHERE associated IS TRUE)` CTE with no row limit whenever `balance=true` (the default). Any unauthenticated caller can trigger this path, and because no application-level rate limiting exists, concurrent requests force the database to repeatedly scan the entire `token_account` table, exhausting working memory and degrading or partitioning the service.

### Finding Description

**Code path:**

`GET /accounts` → `getAccounts()` (line 335) → `getBalanceParamValue()` returns `true` by default (line 323) → `getAccountQuery(…, includeBalance=true, …)` (line 352) → `getEntityBalanceQuery()` (line 295).

Inside `getEntityBalanceQuery()`, the branch taken depends on `needHistoricalBalanceInfo` (line 164):

```js
const needHistoricalBalanceInfo = accountBalanceQuery.query || accountBalanceQuery.forceUnionEntityHistory;
```

`getAccounts()` passes `accountBalanceQuery` as `undefined` (line 354), which defaults to `{query: '', params: []}` (line 257). Both fields are falsy, so `needHistoricalBalanceInfo` is always `false` for this endpoint, and execution falls into the `else` branch at lines 183–186:

```js
queries.push(`with latest_token_balance as (
   select account_id, balance, token_id
   from token_account
   where associated is true)`);
``` [1](#0-0) 

The CTE carries **no LIMIT clause**. It is then consumed inside a correlated scalar subquery in the SELECT list (lines 187–195), which is evaluated once per row returned by the outer `entity` query. [2](#0-1) 

**Root cause / failed assumption:** The design assumes PostgreSQL will always inline (push down) the CTE and apply the `account_id = e.id` predicate before touching the heap. This holds by default in PostgreSQL ≥ 12 for non-recursive CTEs referenced once, but:
- PostgreSQL < 12 always materialises CTEs (optimization fence), causing a full sequential scan of `token_account` per query.
- PostgreSQL ≥ 12 may still materialise when the planner's cost model favours it (e.g., stale statistics, large table, many outer rows).
- No `NOT MATERIALIZED` hint is present to guarantee inlining.

When materialised, the CTE reads every row of `token_account` where `associated IS TRUE` into a temporary in-memory hash table before any per-account filtering occurs.

**Why the outer LIMIT does not help:** `parseLimitAndOrderParams` caps the outer query at 100 rows (max) or 25 (default). [3](#0-2) 
This limit applies only to the `entity` table scan; the CTE is built before the outer LIMIT is applied and is not constrained by it.

### Impact Explanation
On a production Hedera mirror node the `token_account` table contains tens of millions of rows. Each materialised CTE execution allocates a temporary result set proportional to the number of associated token-account rows. With N concurrent requests, N such allocations exist simultaneously in PostgreSQL's `work_mem` / temp-file space. At sufficient concurrency this exhausts shared memory, forces heavy temp-file I/O, or triggers OOM conditions in the database process, causing query timeouts or a full service partition for all consumers of the mirror node REST API.

### Likelihood Explanation
The preconditions are zero: no authentication, no special headers, no unusual parameters. `balance=true` is the default, so a plain `GET /api/v1/accounts` suffices. An attacker needs only an HTTP client capable of issuing concurrent requests (e.g., `ab`, `wrk`, or a simple script). No application-level rate limiting or in-flight-request cap exists in the REST application code. [4](#0-3) 
The Traefik `inFlightReq` / `rateLimit` middleware visible in the Rosetta Helm chart is absent from the REST API chart defaults and is not enforced at the application layer.

### Recommendation
1. **Add `NOT MATERIALIZED`** to the CTE to guarantee inlining on all supported PostgreSQL versions:
   ```sql
   WITH latest_token_balance AS NOT MATERIALIZED (
     SELECT account_id, balance, token_id
     FROM token_account
     WHERE associated IS TRUE
   )
   ```
2. **Add a per-account LIMIT inside the CTE** or restructure the query to push the `account_id = e.id` predicate directly into the `token_account` scan, eliminating the CTE entirely.
3. **Add application-level concurrency controls** (e.g., an `express-rate-limit` or `p-limit` middleware) to cap simultaneous in-flight DB queries per IP.
4. **Set a PostgreSQL `statement_timeout`** for the REST read-only role to bound the worst-case query duration.

### Proof of Concept
```bash
# No authentication, no special parameters — balance=true is the default
# Send 200 concurrent requests
ab -n 2000 -c 200 "https://<mirror-node-host>/api/v1/accounts"
```
Expected result: PostgreSQL `work_mem` exhaustion or heavy temp-file spill visible in `pg_stat_activity` / `pg_stat_bgwriter`; REST API begins returning 500 errors or timing out for all clients. [5](#0-4) [6](#0-5) [7](#0-6)
