### Title
Unauthenticated EVM Address `from` Filter Triggers Uncached Database Lookup Per Request, Enabling DoS via Resource Exhaustion

### Summary
Any unprivileged user can supply a `from` query parameter containing a valid EVM address to the `/contracts/results` or `/contracts/:contractId/results` endpoints. For every such request, `extractContractResultsByIdQuery()` unconditionally calls `EntityService.getEncodedId()`, which executes a live database query against the `entity` table with no result caching. By flooding the API with requests using unique EVM addresses, an attacker can exhaust the database connection pool and degrade or deny service to the mirror node.

### Finding Description

**Exact code path:**

`extractContractIdAndFiltersFromValidatedRequest()` (lines 373–382) calls `utils.buildAndValidateFilters()` which validates format but performs no DB work. The DB lookup is triggered downstream in `extractContractResultsByIdQuery()`:

`EntityService.getEncodedId()` (lines 118–124) detects an EVM address and delegates to `getEntityIdFromEvmAddress()`: [2](#0-1)

```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
``` [4](#0-3) 

**Root cause:** There is no in-process cache (memoization, LRU, etc.) for the EVM-address-to-entity-ID resolution. Every request with a valid EVM address in `from` unconditionally hits the database.

**Why existing checks are insufficient:**

1. `buildFilters()` enforces `maxRepeatedQueryParameters` to cap the number of `from` values per single request, but this only limits per-request amplification — it does not prevent high request volume. [5](#0-4) 

2. The response cache (`responseCacheHandler.js`) caches full HTTP responses keyed on the full request URL. An attacker using a different unique EVM address per request trivially bypasses it — every request is a cache miss.

3. No authentication or rate limiting is enforced at the controller or service layer for this endpoint.

### Impact Explanation
An attacker sending a high volume of requests with unique, syntactically valid EVM addresses (e.g., `0x` + 40 random hex chars) causes one uncached `SELECT` against the `entity` table per request. Under sustained load this exhausts the PostgreSQL connection pool, causing query queuing, timeouts, and eventual unavailability of the mirror node REST API. Because the mirror node is the primary read interface for dApps and network tooling, this constitutes a significant availability impact.

### Likelihood Explanation
No privileges, accounts, or special knowledge are required. The endpoint is publicly accessible. Generating unique 20-byte EVM addresses is trivial. A single attacker with modest bandwidth can sustain thousands of requests per second, each guaranteed to produce a DB round-trip. The attack is repeatable indefinitely and requires no brute-force guessing.

### Recommendation
1. **Cache EVM-address lookups**: Introduce a short-lived (e.g., 30–60 s) in-process LRU cache inside `EntityService.getEntityIdFromEvmAddress()` keyed on the hex address. Non-existent addresses should also be negatively cached to prevent repeated misses.
2. **Rate-limit by IP/client** at the API gateway or middleware layer, specifically for endpoints that trigger DB lookups per filter value.
3. **Validate existence eagerly and cheaply**: Consider rejecting `from` EVM addresses that do not match a known format before issuing a DB query, or deferring the lookup to after a cheap primary filter (e.g., timestamp range) has already narrowed the result set.

### Proof of Concept

```bash
# Generate and send requests with unique EVM addresses in a tight loop
for i in $(seq 1 10000); do
  ADDR=$(python3 -c "import secrets; print('0x' + secrets.token_hex(20))")
  curl -s "https://<mirror-node>/api/v1/contracts/results?from=${ADDR}" &
done
wait
```

Each request reaches `extractContractResultsByIdQuery()`, passes `EntityId.isValidEvmAddress()`, and triggers `EntityService.getEncodedId()` → `getEntityIdFromEvmAddress()` → one `SELECT` on the `entity` table. With sufficient concurrency the DB connection pool is saturated and legitimate queries begin timing out.
