### Title
Unauthenticated Alias Lookup Triggers Unthrottled DB Queries Enabling Non-Network DoS

### Summary
The public `/accounts/:idOrAliasOrEvmAddress/tokens` endpoint in `rest/controllers/tokenController.js` requires no authentication and triggers multiple unthrottled database queries per request when an account alias is supplied. The REST API has no application-level rate limiting, allowing an unprivileged attacker to flood the endpoint with concurrent alias-based requests, exhausting the database connection pool and causing a non-network-based DoS.

### Finding Description

**Exact code path:**

`getTokenRelationships()` at [1](#0-0)  calls `EntityService.getEncodedId()`, which — when the path parameter matches `AccountAlias.isValid()` — calls `getAccountIdFromAlias()` → `getAccountFromAlias()`, executing:

```sql
SELECT id FROM entity
WHERE COALESCE(deleted, false) <> true
  AND alias = $1
``` [2](#0-1) 

This is followed unconditionally by a second DB query via `isValidAccount()`: [3](#0-2) 

And then a third query via `TokenService.getTokenAccounts()`. [4](#0-3) 

**Root cause — two compounding issues:**

1. **No application-level rate limiting in the REST API.** The REST middleware stack contains only: `authHandler`, `httpErrorHandler`, `metricsHandler`, `openapiHandler`, `requestHandler`, `requestNormalizer`, `responseCacheHandler`, `responseHandler` — no rate limiter.

The throttle/rate-limit code found in the repo belongs exclusively to the `web3` Java module. [5](#0-4) 

2. **COALESCE prevents optimal index usage.** The database has a partial index `entity__alias ON entity (alias) WHERE alias IS NOT NULL`. [6](#0-5) 
The expression `COALESCE(deleted, false) <> true` is a non-sargable function call on `deleted`, preventing PostgreSQL from using a covering or partial index that also filters on `deleted`. Every matching row requires a heap fetch to evaluate the `deleted` column, adding per-query overhead compared to a plain `WHERE alias = $1 AND (deleted IS NULL OR deleted = false)` with a suitable partial index.

**Why the cache does not save it:** `responseCacheHandler` caches by request URL. An attacker rotating through different valid alias values (or appending varying `token.id` query parameters) produces distinct cache keys, bypassing the cache entirely on every request. [7](#0-6) 

### Impact Explanation
Each inbound alias request causes at least 3 sequential or concurrent DB queries with no per-IP or global request-rate guard at the application layer. Saturating the PostgreSQL connection pool stalls all other mirror-node REST consumers (explorers, wallets, dApps) that depend on the same database, constituting a full non-network-based DoS of the mirror node REST service.

### Likelihood Explanation
No privileges, tokens, or accounts are required — only knowledge that the endpoint accepts base32 aliases (documented in the public OpenAPI spec). The attack is trivially scriptable: generate valid base32 strings matching `accountAliasRegex` (`/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`) and send them concurrently. [8](#0-7) 
The attack is repeatable, stateless, and requires no prior knowledge of existing accounts (non-existent aliases still execute the full DB query before returning null). [9](#0-8) 

### Recommendation
1. **Add application-level rate limiting** to the REST API middleware (e.g., `express-rate-limit` or a Redis-backed limiter), scoped per source IP, applied before DB-touching handlers.
2. **Fix the COALESCE anti-pattern**: replace `COALESCE(deleted, false) <> true` with `(deleted IS NULL OR deleted = false)` and add a partial index `ON entity (alias) WHERE alias IS NOT NULL AND (deleted IS NULL OR deleted = false)` to enable index-only filtering. [2](#0-1) 
3. **Cache alias→id resolution** in-process (e.g., a short-TTL LRU map) so repeated alias lookups for the same alias do not hit the DB on every request.

### Proof of Concept

```bash
# Generate 500 concurrent requests with distinct valid base32 aliases
# (aliases need not exist in the DB — the query still executes)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/AABBCCDD${i}/tokens" &
done
wait
```

**Preconditions:** None — no account, no credentials, no prior state required.

**Trigger:** Each request with a base32 alias matching `accountAliasRegex` causes `entityFromAliasQuery` + `entityExistenceQuery` + `getTokenAccounts` to execute against the DB. [10](#0-9) 

**Result:** DB worker threads saturate; legitimate requests queue or time out; mirror node REST service becomes unavailable.
