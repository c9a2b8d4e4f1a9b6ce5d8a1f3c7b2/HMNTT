### Title
Unauthenticated Rapid Re-subscription Loop Causes Unbounded DB Query Amplification via `PollingTopicListener.listen()`

### Summary
An unprivileged external user can open a gRPC topic subscription with `limit=1`, which causes `TopicMessageRepositoryCustomImpl.findByFilter()` to skip the `TOPIC_MESSAGES_BY_ID_QUERY_HINT` query-planner hint, resulting in a potentially less efficient DB query plan. Because there is no rate limit on subscription setups in the gRPC module and `maxConcurrentCallsPerConnection` only restricts concurrent calls per single connection, an attacker can open multiple connections and rapidly re-subscribe in a tight loop after each `limit=1` subscription completes, generating a sustained flood of DB queries that can exhaust the database connection pool and deny service to legitimate subscribers.

### Finding Description

**Code path 1 — hint bypass:** [1](#0-0) 

When `filter.getLimit() == 1`, the branch at line 55 is false, so `entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate()` is never called. The hint (`set local random_page_cost = 0`) exists specifically to steer the PostgreSQL planner toward the `(topic_id, consensus_timestamp)` composite index. Without it, the planner may fall back to a primary-key scan, which is more expensive for topic-scoped range queries.

**Code path 2 — subscription lifecycle:** [2](#0-1) 

Each call to `subscribeTopic()` unconditionally:
1. Executes `topicExists()` → `entityRepository.findById()` (DB query).
2. Executes `topicMessageRetriever.retrieve(filter, true)` (one or more DB queries for historical messages).
3. Constructs a new `PollingTopicListener` flux via `incomingMessages()`.

When `filter.hasLimit()` is true and `limit=1`, line 84 applies `.take(1)`, which terminates the flux as soon as one message is emitted. The attacker's client immediately receives the stream-complete signal and can re-subscribe with zero enforced delay.

**Code path 3 — polling listener:** [3](#0-2) 

`RepeatSpec.times(Long.MAX_VALUE)` with a 500 ms interval governs the internal polling cadence *within* a single subscription. This delay does not apply between successive independent subscriptions opened by the attacker.

**Missing control — no subscription rate limit:** [4](#0-3) [5](#0-4) 

`maxConcurrentCallsPerConnection = 5` limits concurrent RPCs per TCP connection. It does not limit: (a) the rate at which sequential subscriptions are opened on the same connection after prior ones complete, or (b) the number of distinct TCP connections from the same IP. The gRPC module contains no token-bucket, leaky-bucket, or per-IP rate limiter analogous to the `ThrottleConfiguration` present in the `web3` module.

### Impact Explanation
Every re-subscription triggers at minimum two synchronous DB round-trips (entity lookup + historical retrieval) plus a `findByFilter` call that, with `limit=1`, uses a suboptimal query plan. An attacker driving N connections × 5 concurrent subscriptions, each completing in milliseconds on a topic with recent messages, can sustain thousands of DB queries per second. This exhausts the JDBC connection pool, starves legitimate subscribers, and can cause the gRPC service to become unresponsive — a non-network-based DoS.

### Likelihood Explanation
No authentication is required to open a gRPC subscription. The gRPC port (default 5600) is publicly reachable. The attack requires only a standard gRPC client (e.g., `grpcurl`, a custom Go/Java client) and knowledge of any valid topic ID. The loop can be scripted in under 20 lines of code. A single attacker machine with multiple TCP connections is sufficient; a botnet is not required.

### Recommendation
1. **Add a per-IP subscription rate limiter** in the gRPC interceptor layer (e.g., Bucket4j token bucket keyed on remote IP), analogous to `ThrottleConfiguration` in the `web3` module.
2. **Add a global concurrent-subscriber cap** enforced in `TopicMessageServiceImpl.subscribeTopic()` — reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
3. **Add a per-connection subscription-setup rate limit** (e.g., max N new subscriptions per second per connection) via a `ServerInterceptor`.
4. **Consider applying the query hint unconditionally** or restructuring the hint logic so that `limit=1` does not silently degrade query plan quality.

### Proof of Concept
```
# Prerequisites: grpcurl installed, mirror node running with POLL listener,
# topic 0.0.1234 has at least one message.

while true; do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1234},"limit":1,"consensusStartTime":{"seconds":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    > /dev/null 2>&1
done
```
Each iteration completes in milliseconds (one message received → stream ends), immediately re-subscribing. Running 5–10 parallel instances of this loop across separate TCP connections saturates the DB connection pool and causes `subscribeTopic` latency to spike for all other clients.

```java
    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```
