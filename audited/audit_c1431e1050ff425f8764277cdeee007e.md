### Title
Unbounded Concurrent Subscriptions via `subscribeTopic()` Enable Resource Exhaustion DoS

### Summary
`TopicMessageServiceImpl.subscribeTopic()` accepts an unlimited number of concurrent subscriptions from any unauthenticated caller. The `subscriberCount` field is a metrics-only gauge that is never checked against a maximum before accepting a new subscription. The sole guard — `maxConcurrentCallsPerConnection = 5` — is a per-TCP-connection limit that an attacker trivially bypasses by opening many connections, enabling unbounded resource consumption across file descriptors, memory, database connections, and Redis subscriptions.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 43-53) accepts any unauthenticated `ConsensusTopicQuery` and delegates directly to `TopicMessageServiceImpl.subscribeTopic()` with no pre-admission check. [1](#0-0) 

Inside `subscribeTopic()`, `subscriberCount` is incremented on subscribe and decremented on completion, but is **never read to gate admission**: [2](#0-1) [3](#0-2) 

It is registered only as a Micrometer `Gauge` for observability: [4](#0-3) 

The only configured server-side limit is `maxConcurrentCallsPerConnection = 5`, applied per TCP connection: [5](#0-4) [6](#0-5) 

No `maxConnections`, `maxConnectionAge`, or global stream cap is set on the `NettyServerBuilder`. No authentication interceptor exists in the production path — the only `ServerInterceptor` sets an endpoint-context label for table-usage tracking: [7](#0-6) 

Subscriptions with no `endTime` never self-terminate: `pastEndTime()` returns `Flux.never()` when `endTime` is null, keeping the stream open indefinitely: [8](#0-7) 

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (plaintext, no auth required per the public API docs). [9](#0-8) 
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams (the per-connection maximum), each with `limit=0` and no `endTime` — yielding `5N` indefinitely-running subscriptions.
3. Each subscription allocates: a `TopicContext` object (heap), a `topicListener` registration (Redis subscription or polling slot), a `Schedulers.boundedElastic()` task for the safety check, and periodic DB queries via `topicMessageRetriever`.
4. With no global cap, `N` is bounded only by OS-level file descriptors and available heap — both exhaustible.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` is per-connection, not global. An attacker with `N` connections gets `5N` streams.
- `subscriberCount` is a read-never metric counter; it imposes zero admission control.
- The GCP backend policy `maxRatePerEndpoint: 250` is an optional infrastructure component that is disabled by default (`gateway.gcp.enabled: true` only in the Helm chart, not guaranteed in all deployments) and controls request rate, not concurrent open streams. [10](#0-9) 

### Impact Explanation

Each open subscription consumes: heap (TopicContext + Reactor chain), a Redis subscription or polling thread slot, and periodic JDBC connections from the HikariCP pool. Exhausting the DB connection pool (monitored but not protected against subscriber-driven exhaustion) causes all subscriptions — including legitimate ones — to stall waiting for a connection. File descriptor exhaustion prevents new TCP connections entirely. The result is a complete denial of service for the gRPC topic subscription API, preventing any client from observing topic messages streamed from the mirror node.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no on-chain resources. The endpoint is publicly documented and reachable over plaintext. A single attacker machine can open thousands of TCP connections and multiplex 5 streams each using any gRPC client library. The attack is fully scriptable, repeatable, and leaves no authentication trail. Likelihood is high.

### Recommendation

1. **Add a global subscription cap**: Check `subscriberCount` against a configurable maximum inside `subscribeTopic()` before building the flux, and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP or per-connection subscription limit** via a `ServerInterceptor` that tracks active streams per remote address.
3. **Set `maxConnectionAge` and `maxConnections`** on the `NettyServerBuilder` in `GrpcConfiguration` to bound total TCP connections server-wide.
4. **Enforce a minimum `endTime` or maximum subscription duration** for open-ended subscriptions to prevent indefinite stream accumulation.
5. **Apply infrastructure-level rate limiting** (e.g., the GCP backend policy) as a mandatory deployment requirement, not an optional one.

### Proof of Concept

```bash
# Open 200 TCP connections, each with 5 concurrent indefinite subscriptions = 1000 total
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1000}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# Observe: subscriberCount metric climbs to 1000 with no server-side rejection.
# DB connection pool saturates; new legitimate subscribers receive errors or hang.
```
