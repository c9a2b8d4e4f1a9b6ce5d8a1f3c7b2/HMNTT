### Title
Unauthenticated `/block` Endpoint Exhausts DB Connection Pool via Concurrent Large-Block Queries, Causing ≥30% CPU Increase

### Summary
The Rosetta `/block` endpoint requires no authentication and has no application-level rate limiting. A single request for a block with many transactions triggers multiple expensive correlated-subquery DB round-trips (batched loop + per-operation account alias lookups). An attacker flooding this endpoint with concurrent requests for a known large block can saturate the `MaxOpenConnections` pool (default: 100) and drive DB CPU well above 30% of baseline.

**Root cause / failed assumption:** The design assumes a network-layer throttle will always be present. Without it, the shared GORM pool (100 connections, 20 s timeout) is the only resource bound. With a 20-second statement timeout, an attacker needs only ⌈100 / 20⌉ = 5 sustained concurrent requests to keep the pool fully occupied indefinitely.

### Impact Explanation

- **DB connection pool exhaustion:** All 100 connections held for up to 20 s each; legitimate requests queue or fail with "too many connections."
- **DB CPU spike ≥30%:** Each batch query runs two correlated subqueries per row. A block with 10 000 transactions produces 5 batch iterations × 2 subqueries × 10 000 rows = 100 000 subquery executions per request. Ten concurrent such requests drive sustained heavy sequential scans on `crypto_transfer` and `staking_reward_transfer`.
- **Cascading effect:** Other Rosetta endpoints (`/account/balance`, `/network/status`) share the same pool and become unavailable.

### Likelihood Explanation

- **No credentials required** — any internet-reachable Rosetta node is a target.
- **Block selection is trivial** — Hedera mainnet/testnet block indices are public; an attacker queries `/network/status` to find the current tip, then picks a historically dense block.
- **Low request volume needed** — 5–10 concurrent HTTP POST requests suffice to saturate the pool; achievable from a single machine with `curl --parallel` or any HTTP load tool.
- **Repeatable** — the attack can be sustained indefinitely; each 20-second timeout window is immediately refilled by the next wave of requests.

### Recommendation

1. **Add application-level concurrency limiting** in `rosetta/main.go` using `golang.org/x/net/netutil` or a semaphore middleware before the router, independent of infrastructure.
2. **Add per-IP rate limiting** in the Go HTTP stack (e.g., `golang.org/x/time/rate`) so the control is not optional.
3. **Reduce `statementTimeout`** for the `/block` path or add a shorter per-request context deadline to bound connection hold time.
4. **Consider query-level caching** for recently-fetched blocks to avoid re-executing the correlated subqueries on repeated requests for the same block.
5. **Make Traefik `inFlightReq`/`rateLimit` mandatory** (remove the `if` guard or document it as a hard deployment requirement).

### Proof of Concept

```bash
# 1. Identify a large block (many transactions) via network status
curl -s -X POST http://<rosetta-host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
# Note a block index with high transaction count from chain history

# 2. Flood /block with 20 concurrent requests for that block
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{
      "network_identifier":{"blockchain":"Hedera","network":"mainnet"},
      "block_identifier":{"index":<LARGE_BLOCK_INDEX>}
    }' &
done
wait

# 3. Observe: subsequent legitimate requests hang or return errors;
#    DB pg_stat_activity shows 100 active connections;
#    DB CPU (pg_stat_bgwriter / OS metrics) shows sustained spike ≥30%.
```
