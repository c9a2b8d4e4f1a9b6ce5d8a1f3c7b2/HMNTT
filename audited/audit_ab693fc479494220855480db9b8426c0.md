### Title
Historical Opcode Replay Uses Current Bytecode Instead of Historical Bytecode, Producing Fabricated Execution Traces

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint re-executes historical transactions on the EVM but loads contract bytecode from the `contract` table without any timestamp filter, always fetching the current/latest bytecode. When a contract's bytecode has been updated since the original transaction, the replay executes under the new code, producing an opcode trace that never actually occurred. Any unprivileged user can trigger this by calling the endpoint for a historical transaction on a contract whose bytecode has since changed.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (line 64) → `OpcodeServiceImpl.processOpcodeCall()` (line 71) → `ContractDebugService.processOpcodeCall()` (line 51) sets `ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1))` to establish a historical context, then calls `callContract()` (line 56). During EVM execution, `ContractBytecodeReadableKVState.readFromDataSource()` (lines 39–46) is invoked to load the contract's bytecode. It calls:

```java
contractRepository.findRuntimeBytecode(entityId.getId())
```

which executes:

```sql
select runtime_bytecode from contract where id = :contractId
```

**Root cause:** This query has no timestamp predicate. It always returns the current row from the `contract` table. The historical timestamp set in `ContractCallContext` is used for storage slot lookups (`findStorageByBlockTimestamp` with `consensus_timestamp <= ?3`) and entity lookups, but `ContractBytecodeReadableKVState` never reads `ContractCallContext.getTimestamp()` and never queries `contract_history`.

**Failed assumption:** The system assumes contract bytecode is immutable. In reality, the database schema explicitly supports historical bytecode: the SQL migration `V1.64.0__contract_action.sql` adds `runtime_bytecode` to both `contract` and `contract_history`. `SidecarContractMigration` uses `ON CONFLICT DO UPDATE SET runtime_bytecode = excluded.runtime_bytecode`, overwriting the current bytecode when a new version is ingested. Proxy-pattern upgrades and Hedera's native contract update mechanism both produce new bytecode entries.

**Why existing checks fail:** `ContractDebugService.processOpcodeCall()` correctly sets `ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1))` (line 51), and `ContractCallContext.getTimestamp()` correctly returns this value when `useHistorical()` is true (block ≠ LATEST). However, `ContractBytecodeReadableKVState.readFromDataSource()` never calls `ContractCallContext.get().getTimestamp()` — it bypasses the historical context entirely and calls `contractRepository.findRuntimeBytecode()` directly with no timestamp. ### Impact Explanation
The opcodes endpoint is the primary forensic tool for auditing historical EVM execution on Hedera. When a contract's bytecode has been updated, every call to this endpoint for a pre-update transaction returns a fabricated opcode trace — one that represents an execution that never occurred on-chain. The returned `opcodes` array, `return_value`, `failed` flag, and `gas` values all reflect the new bytecode's behavior, not the original. This corrupts forensic investigations, dispute resolution, and any tooling that relies on the mirror node for historical execution fidelity. The endpoint is read-only so no funds are moved, but the integrity of the historical record is completely undermined for affected contracts.

### Likelihood Explanation
The precondition — a contract with updated bytecode — is realistic. Upgradeable proxy patterns are standard practice in EVM development, and Hedera's native contract update mechanism explicitly supports bytecode replacement (evidenced by the `contract_history.runtime_bytecode` column and the `ON CONFLICT DO UPDATE` upsert in `SidecarContractMigration`). No privilege is required to trigger the issue: any user who knows a transaction ID or hash for a pre-update transaction can call the endpoint. The exploit is fully passive and repeatable. The throttle check (`throttleManager.throttleOpcodeRequest()`) limits request rate but does not prevent the incorrect result from being returned. [5](#0-4) [6](#0-5) 

### Recommendation
`ContractBytecodeReadableKVState.readFromDataSource()` must respect the historical timestamp from `ContractCallContext`. Specifically:

1. Add a timestamp-aware query to `ContractRepository` that checks `contract_history` when a historical timestamp is present:
   ```sql
   select runtime_bytecode from contract_history
   where id = :contractId and lower(timestamp_range) <= :timestamp
   order by lower(timestamp_range) desc limit 1
   ```
   Fall back to the `contract` table only when no historical timestamp is set or no history row is found.

2. In `ContractBytecodeReadableKVState.readFromDataSource()`, call `ContractCallContext.get().getTimestamp()` and pass it to the repository, mirroring the pattern used by `ContractStateRepository.findStorageByBlockTimestamp()`. [7](#0-6) [8](#0-7) 

### Proof of Concept

**Preconditions:**
- Contract `0.0.X` was deployed at time T1 with bytecode `A`.
- A transaction `TX1` called contract `0.0.X` at time T2 > T1.
- The contract's bytecode was updated to `B` at time T3 > T2 (via proxy upgrade or Hedera contract update). The `contract` table now holds bytecode `B`; `contract_history` holds bytecode `A` for the range `[T1, T3)`.

**Steps:**
1. As any unprivileged user, send:
   ```
   GET /api/v1/contracts/results/{TX1_hash}/opcodes?stack=true
   Accept-Encoding: gzip
   ```
2. The service resolves `TX1` → `consensusTimestamp = T2`, sets historical context to `T2 - 1`.
3. `ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode(X)` → returns bytecode `B` (current), not `A` (historical).
4. The EVM replays the original calldata against bytecode `B`.
5. The response contains opcodes, return value, and gas figures from bytecode `B`'s execution — a trace that never occurred on-chain.

**Verification:** Compare the returned `opcodes` sequence against the original sidecar contract actions stored for `TX1`. They will differ whenever bytecode `A ≠ B`.
