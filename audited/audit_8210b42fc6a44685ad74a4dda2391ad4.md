### Title
Unauthenticated Metrics Endpoint Flooding Causes CPU Spike and Event-Loop Saturation (DoS/Griefing)

### Summary
When `config.metrics.config.authentication` is `false`, the `metricsMiddleware` function in `rest/middleware/metricsHandler.js` allows any unauthenticated caller to trigger `exporter.collect()` and `serializer.serialize(resourceMetrics)` on every request with no rate limiting, concurrency cap, or response caching. An attacker can flood the endpoint with concurrent requests, saturating the Node.js event loop and degrading response latency for all other API consumers.

### Finding Description
**Exact code path:**

`rest/middleware/metricsHandler.js`, function `metricsMiddleware`, lines 152–165:

```js
return function metricsMiddleware(req, res, next) {
  const {pathname} = new URL(req.url, 'http://localhost');
  const normalizedPath = pathname.endsWith('/') ? pathname : `${pathname}/`;

  if (normalizedPath === metricsPath) {
    if (!authenticate(req)) {          // ← gate
      res.set('WWW-Authenticate', 'Basic realm="Metrics"');
      return res.status(401).send('Unauthorized');
    }
    return exporter.collect().then(({resourceMetrics}) => {   // ← triggered per-request
      res.set('Content-Type', 'text/plain; charset=utf-8');
      res.send(serializer.serialize(resourceMetrics));         // ← CPU-bound serialization
    });
  }
  ...
```

**Root cause — `authenticate()` short-circuits to `true` when auth is off:**

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;   // ← no credential check, no rate check, nothing
  }
  ...
};
``` [1](#0-0) 

Every request that reaches the metrics path unconditionally calls `exporter.collect()`, which fires all registered observable-gauge callbacks synchronously inside the OpenTelemetry SDK:
- `process.cpuUsage()` + `process.hrtime.bigint()` (CPU-usage gauge, lines 116–122)
- `process.memoryUsage()` × 4 (memory gauges, lines 131–133)

followed by `serializer.serialize(resourceMetrics)`, a synchronous string-building pass over all metric data. [2](#0-1) 

**No rate limiting exists anywhere in the REST service.** A grep for `rateLimit`, `throttle`, `semaphore`, or `concurrency` across `rest/**/*.js` returns zero matches. The `authHandler` middleware only sets per-user response-size limits for authenticated users; it does not gate the metrics path. [3](#0-2) 

### Impact Explanation
Node.js runs on a single event-loop thread. Each concurrent request to `/swagger/metrics/` enqueues a `collect()` Promise resolution that, when it runs, executes multiple synchronous system calls and a full serialization pass. A flood of concurrent requests fills the microtask/Promise queue, blocking the event loop from processing other pending I/O callbacks. The result is measurably increased response latency for all other REST API endpoints served by the same process. Under sustained flood conditions the process can become effectively unresponsive. No authentication credential is required; no prior relationship with the service is needed.

### Likelihood Explanation
The default value of `hiero.mirror.rest.metrics.config.authentication` is `true`, so the attack surface only opens when an operator explicitly sets it to `false`. [4](#0-3) 

However, operators may disable authentication to allow Prometheus scrapers to reach the endpoint without managing credentials, which is a common operational pattern. Once disabled, the endpoint is reachable from any network client with no preconditions. The attack is trivially repeatable with a single `ab`/`wrk`/`curl` loop and requires no special knowledge or tooling.

### Recommendation
Apply at least one of the following mitigations directly in `metricsMiddleware`:

1. **In-flight concurrency cap** — maintain a module-level counter; if a collect is already in progress, return the cached last result or a 429.
2. **Response caching with TTL** — cache the serialized output for N seconds (e.g., 15 s) and serve the cached string on subsequent requests, skipping `collect()` entirely.
3. **Rate limiting** — add a token-bucket or sliding-window rate limiter (e.g., `express-rate-limit`) scoped to the metrics path before the `metricsMiddleware` is registered.
4. **Re-enable authentication by default** and document that disabling it without a network-level access control (firewall/ingress rule) is unsafe.

### Proof of Concept
**Preconditions:**
- `config.metrics.config.authentication = false` (operator-set)
- `config.metrics.enabled = true` (default)
- Default `uriPath = '/swagger'` → metrics endpoint at `http://<host>:5551/swagger/metrics/`

**Steps:**
```bash
# Flood with 500 concurrent connections, 10 000 total requests
ab -n 10000 -c 500 http://<host>:5551/swagger/metrics/

# Simultaneously measure latency degradation on a normal API endpoint
while true; do
  curl -o /dev/null -s -w "%{time_total}\n" http://<host>:5551/api/v1/transactions?limit=1
done
```

**Expected result:** Response times for `/api/v1/transactions` increase significantly (seconds vs. milliseconds baseline) while the flood is active, demonstrating event-loop saturation caused by repeated `exporter.collect()` + `serializer.serialize()` invocations with no concurrency control.
