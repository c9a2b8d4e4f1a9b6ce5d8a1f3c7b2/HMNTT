### Title
Uncached, Unthrottled `NetworkStakeRepository.findLatest()` Enables Thread Pool Exhaustion via DB Slowdown Cascade

### Summary
The `GET /api/v1/network/stake` endpoint in the rest-java service invokes `networkStakeRepository.findLatest()` on every request with no caching, no per-IP rate limiting, and no application-level bulkhead or circuit breaker. The Traefik middleware for rest-java omits both `inFlightReq` and `rateLimit` directives (present in other services like Rosetta), and its circuit breaker only fires on network errors or 5xx responses — not on slow-but-successful queries. An unprivileged attacker flooding this endpoint can hold Tomcat threads blocked against the DB for up to 20 seconds each, exhausting the thread pool and driving JVM CPU and memory above the 30% threshold.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (no rate-limit annotation, no throttle guard): [1](#0-0) 

calls `NetworkServiceImpl.getLatestNetworkStake()` (no `@Cacheable`, no Resilience4j `@CircuitBreaker`/`@Bulkhead`): [2](#0-1) 

which calls `NetworkStakeRepository.findLatest()` — a native correlated subquery executed synchronously on every request: [3](#0-2) 

**Root cause — failed assumptions:**

1. **No caching**: The grpc module's analogous `NodeStakeRepository.findAllStakeByConsensusTimestamp()` is annotated `@Cacheable` with a dedicated `CacheManager`. [4](#0-3) 
The rest-java `NetworkStakeRepository` has no equivalent — every HTTP request hits the DB.

2. **No per-IP rate limiting or inFlightReq at the proxy**: The rest-java Traefik middleware only configures `circuitBreaker` and `retry`: [5](#0-4) 
Compare with Rosetta, which adds `inFlightReq` (5 concurrent per IP) and `rateLimit` (10 req/s per host): [6](#0-5) 

3. **Traefik circuit breaker is blind to slow 200 OK responses**: The expression `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` only trips on network-layer errors or HTTP 5xx. A DB query that takes 19 seconds and returns HTTP 200 never triggers it. [7](#0-6) 

4. **Retry middleware amplifies the attack**: With `attempts: 3`, each attacker request can generate up to 3 DB queries, tripling thread and connection consumption. [8](#0-7) 

5. **HPA is disabled**: No auto-scaling to absorb the load. [9](#0-8) 

**Exploit flow:**
- Attacker sends a sustained flood of `GET /api/v1/network/stake` requests (no authentication required, publicly documented endpoint).
- Each request acquires a Tomcat thread and a HikariCP DB connection, executing the correlated subquery.
- Under load, the DB slows (lock contention, I/O saturation, or simply connection pool pressure from concurrent queries). Each thread blocks for up to the 20-second statement timeout. [10](#0-9) 
- With Tomcat's default 200-thread pool, ~200 concurrent requests each blocking 20 seconds saturates all threads. New requests queue in the accept backlog, growing heap.
- JVM CPU spikes from thread context switching and GC pressure on the growing request queue; heap memory grows from queued request objects.

### Impact Explanation

Thread pool exhaustion causes the rest-java service to stop processing all requests (not just `/network/stake`), affecting every endpoint routed through it including `/api/v1/network/`, account allowances, airdrops, and topics. JVM CPU and memory increase well beyond 30% — the Prometheus alert thresholds themselves are set at 80% CPU and 80% memory, confirming the operational significance. The `prometheusRules` are defined but `enabled: false` by default, meaning alerts may not fire. [11](#0-10) 

### Likelihood Explanation

The endpoint is unauthenticated, publicly documented in the OpenAPI spec, and requires only standard HTTP GET requests — no special knowledge, credentials, or protocol manipulation. The attack is trivially repeatable with any HTTP load tool (e.g., `ab`, `wrk`, `hey`). The absence of per-IP rate limiting or inFlightReq means a single IP can open hundreds of concurrent connections. The 20-second statement timeout is the only bound, and it is long enough to make the attack effective with modest concurrency (~50–100 concurrent requests).

### Recommendation

1. **Add `@Cacheable`** to `NetworkServiceImpl.getLatestNetworkStake()` or `NetworkStakeRepository.findLatest()` with a short TTL (e.g., 30–60 seconds). Network stake data changes only once per staking period (~24 hours), making aggressive caching safe and effective.

2. **Add `inFlightReq` and `rateLimit` Traefik middleware** to the rest-java chart, mirroring the Rosetta configuration:
   ```yaml
   - inFlightReq:
       amount: 5
       sourceCriterion:
         ipStrategy:
           depth: 1
   - rateLimit:
       average: 20
       sourceCriterion:
         requestHost: true
   ```

3. **Enable HPA** for rest-java to allow horizontal scaling under load.

4. **Add a Resilience4j `@Bulkhead`** on `getLatestNetworkStake()` to bound the number of concurrent DB calls regardless of proxy configuration.

5. **Reduce the DB statement timeout** for the rest-java user from 20 seconds to 3–5 seconds for read-only queries on small tables like `network_stake`.

### Proof of Concept

```bash
# Flood the endpoint with 500 concurrent connections, 10000 total requests
# No authentication, no special headers required
wrk -t 50 -c 500 -d 60s http://<mirror-node-host>/api/v1/network/stake

# Observe JVM metrics via Actuator or Prometheus:
# - jvm_memory_used_bytes{application="rest-java"} increases >30%
# - process_cpu_usage{application="rest-java"} increases >30%
# - hikaricp_connections_active{application="rest-java"} saturates pool
# - http_server_requests_seconds_count with status=5xx begins appearing
#   once thread pool is exhausted and requests are rejected
```

Preconditions: Network access to the rest-java service (publicly exposed via ingress at `/api/v1/network/`). No credentials required. [12](#0-11)
