### Title
Connection Pool Exhaustion DoS via Unbounded Concurrent Requests in Rosetta API

### Summary
The `ConnectToDb()` function in `rosetta/app/db/db.go` configures a hard cap of 100 open database connections via `SetMaxOpenConns()`. The Rosetta Go application has no in-process concurrency or rate-limiting middleware, and the Traefik infrastructure-level middleware (which includes `inFlightReq: 5` per IP) is **disabled by default** (`global.middleware: false`). An unprivileged attacker can flood the API with concurrent requests, exhaust the connection pool, and cause legitimate requests to stall or time out.

### Finding Description

**Exact code path:**

`rosetta/app/db/db.go`, `ConnectToDb()`, lines 31–33:
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default: 20
sqlDb.SetConnMaxLifetime(time.Duration(dbConfig.Pool.MaxLifetime) * time.Minute) // default: 30 min
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default: 100
```

`rosetta/app/db/client.go`, `GetDbWithContext()`, lines 36–37:
```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
return d.db.WithContext(childCtx), cancel
```
`statementTimeout` defaults to **20 seconds** (docs: `hiero.mirror.rosetta.db.statementTimeout = 20`).

**Root cause:** Go's `database/sql` pool blocks callers waiting for a free connection when `MaxOpenConns` is reached. There is no application-level semaphore, queue depth limit, or per-IP concurrency guard in the Go middleware stack. The middleware chain in `main.go` (lines 217–219) is only: `MetricsMiddleware → TracingMiddleware → CorsMiddleware` — none of which limit concurrency.

**Why the Traefik guard fails:** The Helm chart defines `inFlightReq: amount: 5` per IP and `rateLimit: average: 10` per host in `charts/hedera-mirror-rosetta/values.yaml` (lines 152–160), but the middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml`, line 3) is gated on `{{ if and .Values.global.middleware .Values.middleware }}`. The `global.middleware` value defaults to `false` (line 95 of the same values file), so **the Traefik middleware is not deployed in a default installation**.

**Why `statementTimeout` is insufficient:** Each connection is held for up to 20 seconds (or up to the HTTP `WriteTimeout` of 10 seconds if the HTTP layer cancels the context first). With a pool of 100 connections and a 10-second hold time, the attacker only needs to sustain ~10 requests/second to keep the pool perpetually exhausted — trivially achievable with any HTTP load tool.

### Impact Explanation
When the pool of 100 connections is saturated, Go's `database/sql` queues all new acquisition attempts. Legitimate API callers block until either a connection is freed or the HTTP server's `WriteTimeout` (10 seconds) fires, at which point they receive a connection/timeout error. All data-serving Rosetta endpoints (`/block`, `/account`, `/search`, etc.) are affected. No economic damage occurs, but service availability is fully denied for the duration of the attack — a classic griefing DoS.

### Likelihood Explanation
The Rosetta API is a public HTTP endpoint with no authentication requirement. Any external actor with a basic HTTP client (e.g., `ab`, `wrk`, `curl` in a loop) can open 100+ concurrent long-running requests (e.g., to `/block/transaction` with a valid but slow query). No credentials, tokens, or special knowledge are required. The attack is repeatable and cheap to sustain.

### Recommendation
1. **Add an in-process concurrency limiter** in the Go application (e.g., a buffered channel semaphore or `golang.org/x/net/netutil`) that caps total in-flight DB-bound requests well below `MaxOpenConnections`.
2. **Enable the Traefik middleware by default** by setting `global.middleware: true` in the default Helm values, so `inFlightReq` and `rateLimit` are active out of the box.
3. **Set `SetConnWaitTimeout`** (via `db.SetConnMaxIdleTime` or a context deadline on pool acquisition) so that requests waiting for a connection fail fast rather than blocking indefinitely.
4. Consider reducing `MaxOpenConnections` to a value that reflects realistic concurrency needs and pair it with a matching application-level concurrency cap.

### Proof of Concept
```bash
# Exhaust the 100-connection pool (default config, no Traefik middleware)
# Send 110 concurrent requests to a DB-hitting endpoint; hold each open via a slow query path
for i in $(seq 1 110); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}' &
done
wait

# Legitimate request issued while pool is exhausted:
curl -v -X POST http://<rosetta-host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'
# Result: connection stalls for ~10 seconds then times out (HTTP WriteTimeout)
