### Title
Cache Key / Query Normalization Mismatch Enables Cache Bypass and Inconsistent Transaction Ordering via Mixed-Case Parameter Keys

### Summary
`requestQueryParser()` normalizes query-parameter keys to lowercase and merges same-logical-key duplicates into ordered arrays, but `cacheKeyGenerator()` keys the Redis cache on the raw `req.originalUrl` MD5. An unprivileged attacker can exploit this split by sending requests whose mixed-case keys produce a different `req.originalUrl` (and therefore a different cache key) while yielding a different `req.query.order` array ordering, causing a cache miss and a freshly-computed response whose transaction sort direction differs from the cached one.

### Finding Description

**Code locations**

- `rest/middleware/requestHandler.js` lines 38–69 — `requestQueryParser()`: lowercases every key; when the same logical key appears more than once (across any case variant), `merge()` appends values in URL-parameter order into a single array.
- `rest/middleware/responseCacheHandler.js` lines 151–153 — `cacheKeyGenerator()`: `crypto.createHash('md5').update(req.originalUrl).digest('hex')` — uses the raw, un-normalized URL.
- `rest/utils.js` lines 1234–1260 — `buildFilters()`: when `req.query[key]` is an array it emits one filter object per element, preserving array order.
- `rest/transactions.js` lines 387–416 — `extractSqlFromTransactionsRequest()`: iterates filters with a plain `order = value` assignment; the **last** `ORDER` filter in the array wins.
- `rest/utils.js` lines 662–675 — `parseLimitAndOrderParams()`: strict equality check (`value === 'asc' || value === 'desc'`) silently falls back to `defaultOrder` when `value` is an array, so endpoints using this helper are unaffected — but the transactions endpoint uses `extractSqlFromTransactionsRequest` via `buildFilters`, which is affected.

**Root cause**

The failed assumption is that `req.originalUrl` and `req.query` are always semantically equivalent for caching purposes. They are not: `requestQueryParser` collapses `?order=asc&Order=desc` and `?Order=desc&order=asc` to the same key (`order`) but to **different arrays** (`['asc','desc']` vs `['desc','asc']`), while `cacheKeyGenerator` produces a different MD5 for each raw URL string.

**Exploit flow — step by step**

1. `?order=asc&Order=desc` arrives.  
   - `qs.parse` → `{order:'asc', Order:'desc'}` (insertion order preserved).  
   - `requestQueryParser` loop: first entry `order` → `{order:'asc'}`; second entry `Order` → merge → `{order:['asc','desc']}`.  
   - `buildFilters` emits `[{key:'order',value:'asc'}, {key:'order',value:'desc'}]`.  
   - `extractSqlFromTransactionsRequest` last-write wins → `order = 'desc'` → SQL `ORDER BY consensus_timestamp desc`.  
   - Cache key = MD5(`/api/v1/transactions?order=asc&Order=desc`) → **K1**. Response cached under K1.

2. `?Order=desc&order=asc` arrives.  
   - `qs.parse` → `{Order:'desc', order:'asc'}`.  
   - `requestQueryParser` loop: first entry `Order` → `{order:'desc'}`; second entry `order` → merge → `{order:['desc','asc']}`.  
   - `buildFilters` emits `[{key:'order',value:'desc'}, {key:'order',value:'asc'}]`.  
   - `extractSqlFromTransactionsRequest` last-write wins → `order = 'asc'` → SQL `ORDER BY consensus_timestamp asc`.  
   - Cache key = MD5(`/api/v1/transactions?Order=desc&order=asc`) → **K2 ≠ K1** → cache miss → fresh DB query → **ASC result returned**, opposite of the DESC result stored under K1.

**Why existing checks are insufficient**

- `validateReq` / `filterValidityChecks` validate each element of an array individually; both `'asc'` and `'desc'` pass `isValidValueIgnoreCase`, so the conflicting-value array is never rejected.
- `requestNormalizer.js` (`normalizeRequestQueryParams`) is documented as called *after* error handling and is not in the cache-key generation path; it does not affect `req.originalUrl`.
- The comment in `responseCacheHandler.js` line 149 explicitly acknowledges the gap: *"In the future, this will utilize Edwin's request normalizer (9113)"* — confirming the normalizer is not yet wired into cache-key generation.

### Impact Explanation

An attacker can deterministically obtain ASC-ordered transaction history from an endpoint that has a cached DESC response (or vice versa) without any authentication. More broadly, any parameter whose backend semantics depend on array element order (e.g., `timestamp`, `block.number` — explicitly called out in `requestNormalizer.js` lines 17–22 as order-sensitive) can be similarly manipulated. The cache is also trivially exhausted: every unique mixed-case permutation of a query string occupies a separate Redis slot, multiplying cache storage consumption and forcing repeated DB hits. Severity: **Medium** (cache integrity + ordering manipulation; no direct fund theft, but transaction history presentation is corrupted for the attacker's session and cache resources are wasted).

### Likelihood Explanation

No privileges, tokens, or special network position are required. The attack is purely HTTP-layer: any client that can send GET requests can exploit it. The payload is a standard URL with mixed-case parameter names, indistinguishable from a typo. It is trivially scriptable and repeatable at high frequency. The `requestHandler.test.js` test at line 21–24 already documents and tests the merge behavior, confirming it is stable and not an edge case.

### Recommendation

1. **Normalize `req.originalUrl` before hashing** in `cacheKeyGenerator`: reconstruct the cache key from the already-normalized `req.query` object (sorted keys, sorted non-order-sensitive values, lowercased) rather than from the raw URL. Wire `normalizeRequestQueryParams` (issue 9113) into the cache-key path.
2. **Reject conflicting scalar parameters at validation time**: for parameters that are semantically scalar (`order`, `limit`, `result`), add a check in `validateReq` / `filterValidityChecks` that rejects any array value with more than one distinct element, returning HTTP 400.
3. **Short-term mitigation**: in `extractSqlFromTransactionsRequest`, replace the last-write-wins `order = value` with an explicit rejection (throw `InvalidArgumentError`) when the `ORDER` filter key appears more than once.

const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};
