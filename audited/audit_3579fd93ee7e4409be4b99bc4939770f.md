### Title
Unbounded Signature Count in `ConstructionCombine()` Enables CPU Exhaustion DoS

### Summary
`ConstructionCombine()` in `rosetta/app/services/construction_service.go` enforces only a minimum signature count (non-zero) but imposes no upper bound. An unauthenticated attacker can submit a single request containing thousands of cryptographically valid ed25519 signatures, forcing the server to execute a proportional number of `ed25519.Verify()` and `hiero.TransactionAddSignature()` calls, saturating CPU and degrading or halting the mirror node's block ingestion pipeline.

### Finding Description
**Exact location:** `rosetta/app/services/construction_service.go`, `ConstructionCombine()`, lines 58–87. [1](#0-0) 

The only guard is a lower-bound check:

```go
if len(request.Signatures) == 0 {
    return nil, errors.ErrNoSignature
}
```

No upper-bound check exists anywhere in the function or in any application-level middleware for the Rosetta Go service. [2](#0-1) 

The loop then unconditionally processes every supplied signature:

```go
for _, signature := range request.Signatures {
    // type check
    pubKey, err := hiero.PublicKeyFromBytes(signature.PublicKey.Bytes)
    // ...
    if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {
        return nil, errors.ErrInvalidSignatureVerification
    }
    _, _ = hiero.TransactionAddSignature(transaction, pubKey, signature.Bytes)
}
```

**Root cause / failed assumption:** The code assumes callers will supply only the small number of signatures actually required for a transaction (typically 1–2). There is no server-side enforcement of this assumption.

**Why the early-exit on invalid signatures does not mitigate this:** The attacker controls the signatures. They can generate an arbitrary number of valid ed25519 key pairs offline, sign the same `frozenBodyBytes` with each, and include all of them in one request. Every signature passes `ed25519.Verify()`, so the loop runs to completion for all N entries.

**Application-level rate limiting:** A search across all Rosetta Go middleware confirms no `MaxBytesReader`, signature-count cap, or per-request work limit exists in the application code. [3](#0-2) 

**Traefik middleware** (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160) provides an optional `inFlightReq: amount: 5` and `rateLimit: average: 10` guard, but: [4](#0-3) 

1. It is gated on `global.middleware` being enabled (line 3 of `middleware.yaml`), which defaults to `false`. [5](#0-4) 
2. Even when enabled, the rate limit is per-request, not per-signature. A single request with 100,000 signatures consumes one request slot while performing 100,000 crypto operations.

### Impact Explanation
Each `ed25519.Verify()` call is a non-trivial elliptic-curve operation. At 100,000 signatures per request and 5 concurrent requests (the inFlightReq ceiling, when Traefik is even enabled), the server must sustain ~500,000 concurrent ed25519 verifications. On a typical server this saturates all CPU cores for the duration of the request. The Rosetta service and the mirror node's block ingestion goroutines share the same process and OS scheduler; CPU starvation directly delays block processing. The 500%+ block-time delay threshold is reachable with a modest number of concurrent malicious requests from a single host.

### Likelihood Explanation
- **No authentication required.** The `/construction/combine` endpoint is public per the Rosetta specification.
- **No special knowledge required.** The attacker only needs to call `ConstructionPayloads` once to obtain a valid `frozenBodyBytes`, then generate N ed25519 key pairs locally (trivial with any crypto library) and sign the bytes.
- **Highly repeatable.** The attack can be scripted and re-issued continuously. Even with Traefik rate limiting active, 10 requests/second × 100,000 signatures/request = 1,000,000 ed25519 verifications/second sustained.
- **Low cost.** Generating ed25519 key pairs and signatures is cheap on the attacker's side; the asymmetry is entirely in the server's favor for the attacker.

### Recommendation
Add an explicit maximum signature count check immediately after the minimum check:

```go
const maxSignatures = 10 // or whatever the protocol maximum is

if len(request.Signatures) == 0 {
    return nil, errors.ErrNoSignature
}
if len(request.Signatures) > maxSignatures {
    return nil, errors.ErrTooManySignatures
}
```

Additionally, enforce an HTTP request body size limit at the Go `net/http` layer (e.g., `http.MaxBytesReader`) so that oversized payloads are rejected before any deserialization or cryptographic work occurs.

### Proof of Concept
1. Call `POST /construction/payloads` with a valid transfer operation to obtain `unsigned_transaction` and the `frozenBodyBytes` (the signing payload bytes).
2. Offline, generate 50,000 ed25519 key pairs and sign `frozenBodyBytes` with each private key.
3. Construct a `ConstructionCombineRequest` with `unsigned_transaction` from step 1 and all 50,000 `(public_key, signature)` pairs.
4. Send `POST /construction/combine` with this payload to the mirror node's Rosetta endpoint.
5. Observe: the server enters the loop at line 72 and executes 50,000 `ed25519.Verify()` + `hiero.TransactionAddSignature()` calls before returning. CPU usage spikes to 100% on all available cores for the duration.
6. Repeat with 5 concurrent connections (within the inFlightReq limit if Traefik is active) to sustain the saturation and measure block ingestion latency degradation exceeding 500% of baseline.
