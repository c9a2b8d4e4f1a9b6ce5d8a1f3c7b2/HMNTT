### Title
Indefinite Cache Staleness via `expireAfterAccess` Reset in `findRuntimeBytecode()` Contract Bytecode Cache

### Summary
The `findRuntimeBytecode()` method in `ContractRepository` uses a Caffeine cache configured with `expireAfterAccess=1h`. Because Caffeine's `expireAfterAccess` policy resets the expiration timer on every cache read, an unprivileged attacker who continuously issues contract-call API requests for a specific contract ID can prevent the cache entry from ever expiring. After a network partition (or any bytecode update event), the stale cached bytecode is served to all users indefinitely as long as the attacker sustains the access pattern.

### Finding Description

**Exact code path:**

`ContractRepository.java` line 16–18 declares the cached method:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

`EvmConfiguration.java` lines 67–73 wires the cache manager:
```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

`CacheProperties.java` line 22 sets the policy:
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```

**Root cause:** Caffeine's `expireAfterAccess` resets the idle-expiry clock on every cache read, not just on writes. There is no `expireAfterWrite` bound, so a continuously-accessed entry has no guaranteed maximum age. The `unless = "#result == null"` guard only prevents caching of empty results; it provides zero protection against stale non-null entries.

**Exploit flow:**
1. A contract's `runtime_bytecode` is loaded into the cache (e.g., during normal operation or just before a network partition).
2. The contract's bytecode is updated in the database (via a `ContractUpdateTransaction` / migration sidecar — confirmed possible by the codebase's own migration tests).
3. The attacker, with no credentials, repeatedly calls any public mirror-node endpoint that resolves to `findRuntimeBytecode(contractId)` (e.g., `eth_call`, `eth_estimateGas`, `eth_getCode` for that contract address).
4. Each call is a cache hit, which resets the `expireAfterAccess` 1-hour timer.
5. The cache entry never reaches its idle deadline; the DB is never re-queried.
6. All users — including legitimate ones — receive the stale (pre-update) bytecode for every EVM execution involving that contract.

### Impact Explanation
Every EVM call routed through the mirror node that targets the affected contract executes against the wrong bytecode. Depending on the contract, this can mean: incorrect return values for `eth_call` queries, wrong gas estimates, bypassed access-control logic if the update patched a vulnerability, or divergence between what the Hedera network executes and what the mirror node reports. The impact is persistent and affects all users of the mirror node, not just the attacker.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send HTTP requests to the public mirror-node JSON-RPC endpoint. The attacker needs to know the contract address (trivially available from on-chain data or block explorers). Sustaining the access pattern requires only a low-rate periodic request (e.g., one request every 50 minutes) to keep the timer from expiring. The attack is fully automatable, repeatable, and silent — it generates no errors and leaves no anomalous log entries beyond normal cache-hit traffic.

### Recommendation
Replace `expireAfterAccess` with `expireAfterWrite` for the contract bytecode cache. `expireAfterWrite` sets an absolute wall-clock deadline from the time the entry was written, regardless of how many reads occur. This guarantees a maximum staleness bound that no access pattern can circumvent:

```java
// CacheProperties.java
private String contract = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```

Additionally, consider adding a `@CacheEvict` or `@CachePut` hook in the importer pipeline that invalidates or refreshes the bytecode cache entry whenever a `ContractUpdateTransaction` or migration sidecar is processed, providing proactive invalidation rather than relying solely on TTL.

### Proof of Concept

**Preconditions:**
- Mirror node is running with default `CacheProperties` (`expireAfterAccess=1h`).
- Contract with ID `42` exists; its `runtime_bytecode` has been cached (e.g., by a prior `eth_getCode` call).
- The contract's bytecode is then updated in the database (simulating a migration or network partition recovery).

**Steps:**
```bash
# 1. Confirm stale bytecode is cached (first call populates cache)
curl -X POST http://mirror-node:8545 \
  -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["0x<contract_addr>","latest"],"id":1}'

# 2. Update runtime_bytecode in the DB directly (simulating migration):
#    UPDATE contract SET runtime_bytecode = '\xNEWBYTECODE' WHERE id = 42;

# 3. Attacker loop — resets expireAfterAccess timer every 30 minutes:
while true; do
  curl -s -X POST http://mirror-node:8545 \
    -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["0x<contract_addr>","latest"],"id":1}'
  sleep 1800
done

# 4. Legitimate user queries — always receives OLD bytecode:
curl -X POST http://mirror-node:8545 \
  -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["0x<contract_addr>","latest"],"id":1}'
# Returns: old bytecode hash, not the updated one
```

**Expected result without fix:** The legitimate user's query returns the stale bytecode indefinitely as long as the attacker loop runs.
**Expected result with fix (`expireAfterWrite`):** After at most 1 hour from the last write, the cache entry expires and the next query fetches the updated bytecode from the DB regardless of attacker access patterns.
