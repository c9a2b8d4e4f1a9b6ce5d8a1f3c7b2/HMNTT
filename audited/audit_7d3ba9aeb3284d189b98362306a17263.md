### Title
Database Connection Details Leaked via Unauthenticated Health Check Endpoint Error Messages

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is publicly accessible without authentication and uses `github.com/hellofresh/health-go/v4` to run a postgres connectivity check. When the postgres connection fails, the library serializes the raw driver error message (from `lib/pq`) into the JSON response body. These error messages contain the database hostname/IP, port, username, and database name, enabling an unprivileged attacker to enumerate internal infrastructure details.

### Finding Description
**Code path:** `rosetta/app/middleware/health.go`, `NewHealthController()`, line 42:
```go
Check: postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
```
`GetDsn()` (`rosetta/app/config/types.go`, lines 49–57) produces:
```
host=<HOST> port=<PORT> user=<USERNAME> dbname=<DBNAME> password=<PASSWORD> sslmode=disable
```
The `hellofresh/health-go/v4` library's `HandlerFunc` (registered at line 75 for `readinessPath`) runs each registered check and, on failure, places the raw `error.Error()` string into the JSON response under a `failures` map and a per-component `error` field. The `lib/pq` driver (go.mod line 15: `github.com/lib/pq v1.12.3`) produces error messages that directly embed connection parameters:
- Network failure: `dial tcp <HOST>:<PORT>: connect: connection refused`
- Auth failure: `pq: password authentication failed for user "<USERNAME>"`
- Missing DB: `pq: database "<DBNAME>" does not exist`

No sanitization or wrapping of these error strings occurs before they are serialized into the HTTP response. The routes are registered with no authentication middleware anywhere in the middleware chain.

**Root cause:** The failed assumption is that health check error messages are internal-only. The `hellofresh/health-go/v4` library is designed to expose check details for observability, but the endpoint is unauthenticated and internet-facing.

### Impact Explanation
An attacker who learns the internal database hostname/IP, port, username, and database name can:
- Directly target the database host with network-layer attacks (DoS, port scanning, lateral movement)
- Use the username for targeted credential attacks against the database
- Confirm the exact database name for SQL injection payload tuning
- Map internal network topology for further exploitation

This constitutes a meaningful information disclosure that lowers the bar for follow-on attacks capable of degrading or disabling the database backend, which would take down the rosetta node's readiness and processing capability.

### Likelihood Explanation
The `/health/readiness` endpoint requires no credentials and is reachable by any external user who can reach the rosetta service port. The postgres check fails (and thus leaks details) during any DB restart, network partition, startup phase, or misconfiguration — all common operational events. An attacker can simply poll the endpoint repeatedly at low cost. No special tooling is required beyond `curl`.

### Recommendation
1. **Sanitize error output:** Wrap the postgres check to catch errors and return a generic message (e.g., `"postgresql connectivity check failed"`) without forwarding the raw driver error to the HTTP response.
2. **Restrict the readiness endpoint:** Place the `/health/readiness` endpoint behind network-level controls (e.g., only accessible from within the cluster/load-balancer, not from the public internet).
3. **Use a custom check wrapper:**
```go
Check: func(ctx context.Context) error {
    err := postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()})(ctx)
    if err != nil {
        log.Errorf("postgresql health check failed: %v", err)
        return errors.New("postgresql connectivity check failed")
    }
    return nil
},
```

### Proof of Concept
1. Deploy the rosetta service with a postgres DSN pointing to `host=10.0.1.50 port=5432 user=mirror_user dbname=mirror_node`.
2. Simulate a DB connectivity failure (stop postgres or block the port).
3. Send: `curl -s http://<rosetta-host>:<port>/health/readiness`
4. Observe response:
```json
{
  "status": "unavailable",
  "failures": {
    "postgresql": "dial tcp 10.0.1.50:5432: connect: connection refused"
  },
  "details": {
    "postgresql": {
      "status": "unavailable",
      "error": "dial tcp 10.0.1.50:5432: connect: connection refused"
    }
  }
}
```
5. Repeat with wrong credentials to obtain username: `"pq: password authentication failed for user \"mirror_user\""`. [1](#0-0) [2](#0-1) [3](#0-2)
