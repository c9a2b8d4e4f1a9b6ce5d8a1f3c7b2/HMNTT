### Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded Database Queries via `Long.MAX_VALUE` Limit

### Summary
An unauthenticated gRPC client can send `limit = Long.MAX_VALUE` (a valid `uint64` value) in a `ConsensusTopicQuery`. This value propagates into `PollingTopicListener.poll()` where a narrowing cast `(int)(Long.MAX_VALUE - 0)` produces `-1`, causing `Math.min(-1, maxPageSize)` to return `-1`. The resulting filter with `limit = -1` bypasses the `hasLimit()` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`, issuing unbounded `SELECT *` queries against the database on every poll cycle indefinitely.

### Finding Description

**Step 1 — User input reaches the filter without an upper-bound check.**

In `ConsensusController.toFilter()`: [1](#0-0) 

`query.getLimit()` is a protobuf `uint64` field. The Java protobuf runtime returns it as a signed `long`. A client sending `Long.MAX_VALUE` (9223372036854775807) passes the only constraint on `TopicMessageFilter.limit`: [2](#0-1) 

`@Min(0)` — `Long.MAX_VALUE > 0` passes. `hasLimit()` also returns `true` since `Long.MAX_VALUE > 0`: [3](#0-2) 

**Step 2 — The limit is forwarded unchanged into the listener.**

In `TopicMessageServiceImpl.incomingMessages()`, when `topicContext.getCount()` is 0: [4](#0-3) 

`Long.MAX_VALUE - 0 = Long.MAX_VALUE` is stored in `newFilter.limit` and passed to `topicListener.listen(newFilter)`.

**Step 3 — Integer overflow in `poll()`.**

In `PollingTopicListener.poll()`: [5](#0-4) 

- `(int)(Long.MAX_VALUE - 0)` = `(int) 0x7FFFFFFFFFFFFFFF` = `-1` (Java narrowing cast truncates to low 32 bits: `0xFFFFFFFF` = `-1`)
- `Math.min(-1, maxPageSize)` = `-1`
- `filter.toBuilder().limit(-1).build()` is called. This is a direct builder call — Spring's `@Validated` proxy is **not** involved, so `@Min(0)` is **not** enforced.

**Step 4 — Negative limit bypasses the guard in the repository.**

In `TopicMessageRepositoryCustomImpl.findByFilter()`: [6](#0-5) 

`filter.hasLimit()` checks `limit > 0`. With `limit = -1`, this returns `false`. `setMaxResults()` is never called. The JPA query executes with **no LIMIT clause**, returning every matching `topic_message` row for the topic.

**Step 5 — This repeats indefinitely.**

The poll loop is: [7](#0-6) 

`RepeatSpec.times(Long.MAX_VALUE)` means the unbounded query fires on every poll interval for the lifetime of the subscription. The outer `flux.take(Long.MAX_VALUE)` in `subscribeTopic` provides no practical bound. [8](#0-7) 

### Impact Explanation

Every poll cycle issues a full table scan of `topic_message` for the given topic with no row limit. On a topic with millions of messages, each poll returns the entire dataset, consuming database CPU, I/O, and network bandwidth. A single connection held open by an attacker causes this to repeat at the configured poll interval indefinitely. Multiple concurrent connections amplify the effect linearly. This is a practical denial-of-service against the database tier with no authentication required.

### Likelihood Explanation

The gRPC endpoint is publicly accessible. The proto field `limit` is `uint64`; sending `Long.MAX_VALUE` requires only a standard gRPC client (e.g., `grpcurl`) with a single field set. No credentials, no special knowledge, and no prior access are needed. The attack is trivially repeatable and scriptable.

### Recommendation

Replace the unsafe narrowing cast with a bounds-checked conversion in `PollingTopicListener.poll()`:

```java
// Before (vulnerable):
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;

// After (safe):
int limit = filter.hasLimit()
        ? (int) Math.min(filter.getLimit() - context.getCount().get(),
                         listenerProperties.getMaxPageSize())
        : listenerProperties.getMaxPageSize();
```

The same pattern exists in `PollingTopicMessageRetriever.poll()` at line 68–70 and should be fixed identically. [9](#0-8) 

Additionally, add an upper-bound validator on `TopicMessageFilter.limit` (e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate cap), and enforce it at the controller layer before the filter is constructed.

### Proof of Concept

```bash
# Using grpcurl against a running mirror node gRPC endpoint
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected (vulnerable) behavior:**
1. Request passes `@Min(0)` validation — `Long.MAX_VALUE` is non-negative.
2. `incomingMessages()` forwards `limit = Long.MAX_VALUE` to `PollingTopicListener`.
3. `poll()` computes `(int)(Long.MAX_VALUE) = -1`; `pageSize = -1`.
4. `findByFilter` with `limit = -1`: `hasLimit()` returns `false`; no `setMaxResults()` called.
5. Full `SELECT * FROM topic_message WHERE topic_id = 1 AND consensus_timestamp >= 0 ORDER BY consensus_timestamp` executes on every poll interval.
6. Connection stays open; database is hammered indefinitely.
