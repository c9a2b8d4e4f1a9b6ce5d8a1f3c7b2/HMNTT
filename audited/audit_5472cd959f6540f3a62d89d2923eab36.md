### Title
Unauthenticated IP Spoofing via Trusted-Header Injection in `getClientIpAddress`

### Summary
`getClientIpAddress` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header without verifying it originates from a trusted proxy. Any unprivileged external client that can reach the rosetta service directly (port 5700) can inject an arbitrary `X-Real-IP` value, causing the tracing middleware to log a fabricated IP address instead of the attacker's real source address. This permanently hides the true origin of all requests in audit logs.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // line 64 — no proxy trust check

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // line 71 — real TCP IP
    }

    return ipAddress
}
```

**Root cause:** The function applies a fixed priority (`X-Real-IP` → `X-Forwarded-For` → `RemoteAddr`) with no validation that the header was set by a trusted upstream proxy. `r.Header.Get` returns whatever the HTTP client sent verbatim; Go's `net/http` server performs no stripping or validation of these headers.

**Exploit flow:**

1. The rosetta container exposes port 5700 directly to the host in `docker-compose.yml` (`ports: - 5700:5700`). An attacker reachable to that port bypasses nginx entirely.
2. The attacker sends any Rosetta API request with `X-Real-IP: 1.2.3.4` (any spoofed value).
3. `getClientIpAddress` reads `X-Real-IP` at line 64, finds it non-empty, and returns `1.2.3.4`.
4. `TracingMiddleware` (line 52–53) logs `1.2.3.4 POST /block/transaction (200) in 3ms`.
5. The attacker's real `RemoteAddr` is never evaluated (line 71 is never reached).
6. Setting `X-Forwarded-For` simultaneously is irrelevant to the outcome — `X-Real-IP` wins unconditionally — but it confirms the priority logic is fully attacker-controlled.

**Why existing checks fail:**

The nginx proxy config (`docker-compose.yml` lines 204–205) does sanitize headers for traffic routed through port 8080:
```nginx
proxy_set_header X-Real-IP      $remote_addr;           # overwrites client value
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```
However, this protection is **not enforced at the application layer**. Port 5700 is independently reachable, and even in Kubernetes the service may be reachable within the cluster without passing through the ingress. The application itself has zero trusted-proxy logic.

The test suite (`trace_test.go` lines 52–58) explicitly confirms the design accepts arbitrary header values as authoritative — it tests that a client-supplied `X-Real-IP: 10.0.0.100` is logged as-is, with no assertion that it came from a trusted source.

### Impact Explanation
Every access log entry for the rosetta service can be falsified by the requester. An attacker probing Hashgraph history (e.g., enumerating blocks, transactions, or account balances via the Rosetta API) can attribute all their requests to an innocent third-party IP. Incident response and forensic attribution become impossible. Because the spoofed IP is written at `log.Info` level (line 58) for all non-internal paths, it affects the primary operational audit trail.

### Likelihood Explanation
Exploitation requires no credentials, no special tooling, and no prior knowledge beyond knowing the service port. A single HTTP request with a crafted header is sufficient. The attack is trivially repeatable and undetectable after the fact because the only record of the request contains the fabricated IP. Port 5700 is exposed by default in the reference `docker-compose.yml`, making direct access the common case for non-Kubernetes deployments.

### Recommendation
Implement a trusted-proxy allowlist before accepting forwarded-IP headers. Only read `X-Real-IP` or `X-Forwarded-For` when `r.RemoteAddr` matches a known trusted proxy CIDR (e.g., the nginx container's subnet or a configurable list). A minimal fix:

```go
var trustedProxies = []net.IPNet{ /* configure: nginx subnet, load balancer CIDRs */ }

func getClientIpAddress(r *http.Request) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); ip != "" {
            return ip
        }
        if ip := r.Header.Get(xForwardedForHeader); ip != "" {
            // take only the last (rightmost) entry added by the trusted proxy
            parts := strings.Split(ip, ",")
            return strings.TrimSpace(parts[len(parts)-1])
        }
    }
    return remoteIP
}
```

Additionally, remove the direct host-port binding for port 5700 in production deployments so all traffic is forced through the sanitizing proxy.

### Proof of Concept
```bash
# Direct access to rosetta, bypassing nginx (port 5700 exposed by docker-compose)

  -H "Content-Type: application/json" \
  -H "X-Real-IP: 8.8.8.8" \
  -H "X-Forwarded-For: 8.8.8.9" \
  -d '{"metadata":{}}'

# Observe rosetta application log — entry will show:
# level=info msg="8.8.8.8 POST /network/list (200) in Xms"
