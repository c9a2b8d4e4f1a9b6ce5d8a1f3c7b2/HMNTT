### Title
Guaranteed-Empty DB Query via `receiver.id=lt:1` in `getOutstandingAirdrops()`

### Summary
An unprivileged attacker can send `GET /api/v1/accounts/{id}/airdrops/outstanding?receiver.id=lt:1`, which causes `Bound.adjustUpperBound()` to compute an effective upper bound of 0 for `receiver_account_id`. Since all valid entity IDs are ≥ 1, the resulting DB query (`receiver_account_id <= 0`) is structurally guaranteed to return zero rows, yet the query is still executed. The existing range-validity check fails to catch this edge case, allowing repeated griefing of DB resources with no authentication required.

### Finding Description
**Code path:**

1. `TokenAirdropsController.getOutstandingAirdrops()` (line 71) accepts `receiverIds` as `EntityIdRangeParameter[]` with no lower-bound constraint on the parsed value. [1](#0-0) 

2. `EntityIdRangeParameter.valueOf("lt:1")` parses to `(operator=LT, value=1)`. `getEntityId("1")` produces `EntityId.of(0, 0, 1)` whose encoded ID is `1`. [2](#0-1) 

3. `getInclusiveValue()` for `LT` returns `value - 1 = 0`. [3](#0-2) 

4. In `Bound` constructor, `hasUpperBound()` is true for `LT`, so `upper = param`. `adjustUpperBound()` computes `1 - 1 = 0`. `getAdjustedLowerRangeValue()` returns `0` (lower is null). [4](#0-3) 

5. The only guard is `if (primarySortField && adjustedLower > adjustedUpper)` → `0 > 0` → **false**. No exception is thrown; the Bound is accepted. [5](#0-4) 

6. The service proceeds to execute a DB query with the effective condition `receiver_account_id <= 0`, which is structurally impossible (all stored entity IDs ≥ 1). [6](#0-5) 

**Root cause:** The range-validity check uses strict inequality (`>`), so the degenerate case where `adjustedLower == adjustedUpper == 0` (a value below the minimum valid entity ID) is silently accepted.

### Impact Explanation
Every such request causes a full DB round-trip that returns zero rows. An attacker can flood the endpoint with these requests to exhaust DB connection pool capacity, increase query-processing overhead, and degrade service for legitimate users. No authentication or special privilege is required. The impact is griefing/resource exhaustion with no direct economic damage to network participants, consistent with the Medium severity classification.

### Likelihood Explanation
The exploit requires only a single crafted HTTP GET request with no credentials. The parameter format (`lt:1`) is valid and accepted by the parser without error. The attack is trivially repeatable in a tight loop from any internet-connected client, making sustained griefing highly feasible.

### Recommendation
Add a post-adjustment validation in `Bound` (or in `EntityIdRangeParameter.valueOf`) that rejects any adjusted upper bound below the minimum valid encoded entity ID (i.e., 1 for shard=0, realm=0, num=1). Concretely, after computing `adjustedUpper`, check:

```java
if (primarySortField && adjustedUpper < 1) {
    throw new IllegalArgumentException("Invalid range provided for " + parameterName);
}
```

Alternatively, enforce a minimum value of 1 inside `getEntityId()` by rejecting `num=0` for range parameters, or add a `@Min(1)` constraint on the parsed entity number before the LT/GT adjustment is applied.

### Proof of Concept
```
# No authentication required
curl "https://<mirror-node>/api/v1/accounts/0.0.100/airdrops/outstanding?receiver.id=lt:1"

# Expected (current): HTTP 200, empty airdrops array, DB query executed
# Expected (fixed):   HTTP 400, invalid range error

# Flood to exhaust DB resources:
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.100/airdrops/outstanding?receiver.id=lt:1" &
done
```

The DB receives `WHERE receiver_account_id <= 0` on every iteration, guaranteed to scan and return nothing, wasting connection and query-processing resources.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
