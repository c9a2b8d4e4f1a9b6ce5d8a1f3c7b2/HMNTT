### Title
Unbounded Concurrent Subscriptions Exhaust Shared `boundedElastic` Scheduler, Causing DoS on Historical Hashgraph Message Retrieval

### Summary
`PollingTopicMessageRetriever.retrieve()` schedules all poll executions on a single `Schedulers.boundedElastic()` instance whose thread pool is capped at `10 × CPU_cores`. Because there is no limit on the total number of concurrent gRPC subscriptions (only a per-connection cap of 5), an unauthenticated attacker can open enough connections to saturate the thread pool, starving legitimate historical retrieval requests of scheduler threads and causing them to time out or stall indefinitely.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` is a singleton Spring bean. In its constructor it creates one scheduler:

```java
// PollingTopicMessageRetriever.java line 41
scheduler = Schedulers.boundedElastic();
```

`Schedulers.boundedElastic()` returns the **global default** bounded elastic scheduler (shared JVM-wide with `PollingTopicListener`, `SharedPollingTopicListener`, and `TopicMessageServiceImpl`). Its thread cap is `10 × Runtime.availableProcessors()` (e.g., 40 threads on a 4-core host).

In `retrieve()` (lines 51–59), every subscription's repeat-delay and timeout tracking are pinned to this scheduler:

```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))          // line 55 – delay tasks run on scheduler
    ...
    .timeout(retrieverProperties.getTimeout(), scheduler)  // line 59 – timeout on same scheduler
```

When a repeat delay expires, the bounded elastic scheduler thread that fires the timer immediately executes the next `poll()` call, which performs a **synchronous** JDBC query via `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 78). This blocks the scheduler thread for the duration of the database round-trip.

**Root cause / failed assumption:**
The design assumes a bounded number of concurrent subscriptions. There is no application-level cap on total concurrent subscriptions — only `NettyProperties.maxConcurrentCallsPerConnection = 5` (per TCP connection). An attacker opening *N* connections creates up to `5N` concurrent subscriptions, each holding a bounded elastic thread during every poll cycle.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits calls per connection but imposes no ceiling on the number of connections or total subscriptions.
- `timeout = 60s` (line 59) eventually cancels stalled subscriptions, but the attacker simply reconnects; `Retry.backoff(Long.MAX_VALUE, ...)` (line 58) causes the server itself to retry indefinitely, compounding thread consumption.
- There is no IP-based rate limit, no global subscriber count gate, and no circuit breaker on the gRPC endpoint.

### Impact Explanation
When the bounded elastic thread pool is saturated, all pending poll tasks queue behind the cap. Legitimate subscribers waiting for historical Hashgraph messages receive no data until a thread frees up. If the queue depth exceeds the 100,000-task limit, tasks are rejected outright. The 60-second `timeout` fires for starved legitimate subscriptions, returning errors to real clients. Because the attacker's subscriptions with `throttled=true` set `numRepeats = Long.MAX_VALUE` (line 99), they persist indefinitely, maintaining the saturation. This constitutes a complete denial of historical topic message retrieval — directly impacting the integrity and availability of Hashgraph history access.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to open many TCP connections — no credentials, no special knowledge, no privileged access. The `subscribeTopic` RPC is publicly documented and unauthenticated. On a typical 4-core deployment, saturating 40 threads requires only 8 connections (8 × 5 = 40 subscriptions), each with a historical `startTime` set far in the past to guarantee continuous polling. This is trivially repeatable and automatable.

### Recommendation
1. **Enforce a global concurrent-subscription limit** in `TopicMessageServiceImpl.subscribeTopic()` using the existing `subscriberCount` gauge — reject new subscriptions when the count exceeds a configurable threshold.
2. **Replace `Schedulers.boundedElastic()` with a dedicated, explicitly sized scheduler** (`Schedulers.newBoundedElastic(threadCap, queueSize, "retriever-scheduler")`) so the retriever's thread pool is isolated from other components and sized deliberately.
3. **Offload blocking JDBC calls** in `poll()` away from the scheduler thread (e.g., wrap `topicMessageRepository.findByFilter` with `subscribeOn(Schedulers.boundedElastic())` separately, or use a non-blocking R2DBC repository).
4. **Add per-IP or per-client connection rate limiting** at the Netty/gRPC layer (e.g., `maxConnectionsPerIp`) to prevent a single attacker from opening unlimited connections.

### Proof of Concept

```bash
# Open 10 connections, each with 5 concurrent historical subscriptions (50 total)
# On a 4-core host this exceeds the 40-thread cap

for i in $(seq 1 10); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}, "consensusStartTime": {"seconds": 0}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: legitimate subscribeTopic calls now stall for 60s then receive DEADLINE_EXCEEDED
# The bounded elastic thread pool is saturated; poll() tasks queue and starve
```

```java
    @Override
    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
```
