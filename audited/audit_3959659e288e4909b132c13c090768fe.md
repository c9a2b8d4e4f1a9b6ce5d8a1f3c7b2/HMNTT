### Title
Unbounded `json_agg` on `transaction_signature` in `getSchedules()` Enables DB Resource Amplification

### Summary
`getSchedules()` in `rest/schedules.js` issues a secondary `signatureQuery` that uses `json_agg` to aggregate **all** signatures for every schedule returned, with no per-schedule signature cap. Because the `transaction_signature` table has no uniqueness constraint on `(entity_id, public_key_prefix)` and no row limit is applied in the aggregation, an unauthenticated attacker can repeatedly request `limit=100` schedules that each carry many signatures, forcing the database to build and transmit arbitrarily large JSON payloads on every request.

### Finding Description
**Exact code path:**

In `rest/schedules.js` lines 255–263, after fetching up to `limit` (max 100) schedule rows, the handler constructs and fires a second query:

```js
const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;
``` [1](#0-0) 

`transactionSignatureJsonAgg` is an unbounded `json_agg(json_build_object(...))` with no `LIMIT` or `FETCH FIRST N ROWS` clause:

```js
const transactionSignatureJsonAgg = `
  json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp,
    'public_key_prefix', encode(ts.public_key_prefix, 'base64'),
    'signature', encode(ts.signature, 'base64'),
    'type', ts.type
  ) order by ts.consensus_timestamp)`;
``` [2](#0-1) 

**Root cause / failed assumption:** The design assumes the number of signatures per schedule is small. The `transaction_signature` table explicitly carries **no uniqueness constraint** on `(entity_id, public_key_prefix)` — the design doc notes "There's no unique constraint/primary key since the client can potentially sign multiple times with the same key." There is therefore no database-level cap on how many rows a single `entity_id` can accumulate. [3](#0-2) 

**Exploit flow:**
1. Identify (or wait for) schedules on mainnet/testnet that have accumulated many `transaction_signature` rows (e.g., popular multi-sig schedules with hundreds of signers, or schedules that were signed repeatedly).
2. Send `GET /api/v1/schedules?limit=100` (no authentication required).
3. The handler fetches 100 schedule IDs, then fires `signatureQuery` with all 100 IDs in the `IN` clause. PostgreSQL must scan, sort, and `json_agg` every matching row across all 100 schedules — potentially hundreds of thousands of rows — into a single result set before returning it.
4. Repeat concurrently to exhaust the DB connection pool (`maxConnections` = 10 by default). [4](#0-3) 

**Why existing checks are insufficient:**

- The `limit` parameter is capped at `response.limit.max` = 100 (default). This bounds the number of *schedules* but not the number of *signatures per schedule*. [5](#0-4) 

- `db.pool.statementTimeout` = 20 000 ms is the only backstop. It terminates individual queries after 20 seconds but does not prevent the DB from consuming significant memory and CPU during that window, and 10 concurrent 20-second queries fully saturate the pool. [6](#0-5) 

- There is no rate limiting on the REST JS API (rate limiting exists only in the web3 Java service).

### Impact Explanation
An attacker can hold all 10 DB connections busy for up to 20 seconds each with heavy `json_agg` aggregations, causing legitimate requests to queue or time out (connection pool exhaustion). Each query forces the DB to materialise a potentially multi-megabyte JSON blob in memory before transmitting it over the DB–application link, consuming both DB working memory and internal network bandwidth. This constitutes a griefing/DoS against the mirror node's read API with no economic cost to the attacker beyond sending HTTP requests.

### Likelihood Explanation
No authentication is required. The attacker only needs to know schedule IDs with many signatures — these are discoverable by paginating through `/api/v1/schedules`. On a live network, popular multi-sig schedules naturally accumulate many signatures over time. The attack is trivially repeatable from a single machine and requires no special tooling.

### Recommendation
1. **Add a per-schedule signature limit in the SQL query**, e.g. use a lateral subquery or `FETCH FIRST N ROWS` inside the aggregation to cap signatures per schedule at a configurable maximum (e.g. 25–50).
2. **Alternatively**, move signature fetching to a separate paginated endpoint (as is done for other sub-resources) rather than embedding all signatures inline in the list response.
3. **Add rate limiting** to the REST JS API (analogous to the web3 throttle) to bound requests per second per IP.
4. Consider reducing `response.limit.max` for the schedules endpoint specifically, or introducing a dedicated `response.limit.max.schedules` setting.

### Proof of Concept
```bash
# 1. Discover schedules with many signatures (paginate to find high-signature ones)
curl "https://<mirror-node>/api/v1/schedules?limit=100&order=asc"

# 2. Fire concurrent amplified requests (no auth needed)
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/schedules?limit=100" &
done
wait
# Result: DB connection pool saturated; legitimate requests receive 503 or hang
# until statementTimeout (20 s) expires per connection
```
