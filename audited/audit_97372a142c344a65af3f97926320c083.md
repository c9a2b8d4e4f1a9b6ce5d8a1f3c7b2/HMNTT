### Title
Unbounded Repeated Execution of Expensive `selectGenesis` SQL Query via Unauthenticated `/network/status` Requests

### Summary
In `rosetta/app/persistence/block.go`, the `initGenesisRecordFile` function uses `sync.Once` only to write the cached genesis block, but the guard check and the expensive SQL query execution occur **outside** `sync.Once`. When the query fails (e.g., during node startup before genesis data is available), `once.Do` is never called, leaving the cache permanently unpopulated. Every subsequent unauthenticated request to `/network/status` re-executes the expensive CTE query joining `account_balance` and `record_file`, with no application-level rate limiting by default.

### Finding Description

**Exact code path:**

`NetworkStatus()` in `rosetta/app/services/network_service.go` (line 67) calls `n.RetrieveGenesis(ctx)`, which delegates to `b.blockRepo.RetrieveGenesis(ctx)` in `rosetta/app/services/base_service.go` (line 97), which calls `br.initGenesisRecordFile(ctx)` in `rosetta/app/persistence/block.go` (line 184).

The critical function is `initGenesisRecordFile` (lines 243–263):

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil  // early exit only if already cached
    }

    db, cancel := br.dbClient.GetDbWithContext(ctx)
    defer cancel()

    var rb recordBlock
    if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
        First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)  // ← returns WITHOUT calling once.Do
    }

    br.once.Do(func() {
        br.genesisBlock = rb  // ← only reached on success
    })
    ...
}
```

**Root cause — two compounding flaws:**

1. **`once.Do` is only called on success.** If `selectGenesis` returns no rows (genesis not yet in DB), the function returns `ErrNodeIsStarting` at line 254 and `once.Do` is never invoked. `br.genesisBlock.ConsensusStart` remains `genesisConsensusStartUnset` (-1), so the guard check at line 244 will never short-circuit. Every future call re-executes the expensive query.

2. **The guard check is not inside `sync.Once`.** Even when genesis data is available, N concurrent goroutines can all pass the `ConsensusStart == genesisConsensusStartUnset` check simultaneously before any of them completes the query and calls `once.Do`, causing N parallel executions of the expensive query.

**The `selectGenesis` query** (lines 49–72) is a CTE that:
- Scans `account_balance` filtered by `account_id` and ordered by `consensus_timestamp`
- Joins the result against `record_file` on `rf.consensus_end > genesis.timestamp`
- Orders by `rf.consensus_end` with `LIMIT 1`

This is a cross-table join with ordering on potentially large tables.

**Why existing checks fail:**

The Traefik-based rate limiting (`average: 10`, `inFlightReq: amount: 5`) defined in `charts/hedera-mirror-rosetta/values.yaml` (lines 157–160) is gated on `global.middleware`, which **defaults to `false`** (line 95). It is an optional infrastructure concern, not enforced at the application layer. The Go HTTP server in `rosetta/main.go` (lines 217–227) applies only metrics, tracing, and CORS middleware — no rate limiting.

### Impact Explanation
During node startup (before genesis data is ingested), an attacker can sustain a flood of POST `/network/status` requests that each trigger the `selectGenesis` CTE query. This exhausts database connection pool slots and CPU, slowing or blocking the importer from writing genesis data — creating a self-reinforcing loop that extends the DoS window indefinitely. Even after genesis is available, a burst of concurrent requests triggers the race window. The Rosetta API is used by exchanges and custodians for balance and block queries; unavailability during startup or under load constitutes a non-network DoS against critical financial infrastructure.

### Likelihood Explanation
No authentication is required. The `/network/status` endpoint is publicly reachable. The attacker needs only an HTTP client capable of sending concurrent POST requests. The vulnerability is most impactful during node startup (a predictable, observable event), but the race window also exists transiently on every fresh `blockRepository` instance. The attack is trivially repeatable and requires no special knowledge beyond the Rosetta API spec.

### Recommendation
Fix the cache-miss-on-failure problem by moving the entire check-query-set sequence inside a mutex or by using a separate `sync.Once`-guarded initialization that retries until success. Specifically:

1. Replace the bare `sync.Once` pattern with a mutex-protected check-and-set that also handles the failure case — e.g., only skip the query if `ConsensusStart != genesisConsensusStartUnset`, and use a `sync.Mutex` to serialize concurrent initializations so only one goroutine executes the query at a time.
2. Add application-level concurrency limiting (e.g., a semaphore or `golang.org/x/sync/singleflight`) so that concurrent calls to `initGenesisRecordFile` collapse into a single in-flight query.
3. Enable the Traefik middleware by default (`global.middleware: true`) or enforce rate limiting inside the Go application.

### Proof of Concept

**Preconditions:** Rosetta node in online mode, genesis data not yet in DB (node starting), no Traefik middleware enabled (default config).

**Steps:**
```bash
# Send 100 concurrent POST requests to /network/status
for i in $(seq 1 100); do
  curl -s -X POST http://<rosetta-host>:5700/network/status \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' &
done
wait
```

**Expected result:** Each of the 100 concurrent requests passes the `ConsensusStart == genesisConsensusStartUnset` guard (since genesis is not yet cached), executes the full `selectGenesis` CTE query against the DB, and returns `ErrNodeIsStarting`. The DB receives 100 simultaneous expensive queries. Repeating this loop keeps the DB saturated, delaying genesis ingestion and extending the DoS window. Once genesis data is available, a single burst still triggers N parallel query executions before `once.Do` fires.

func (n *networkAPIService) NetworkStatus(
	ctx context.Context,
	_ *rTypes.NetworkRequest,
) (*rTypes.NetworkStatusResponse, *rTypes.Error) {
	if !n.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	genesisBlock, err := n.RetrieveGenesis(ctx)
	if err != nil {
		return nil, err
	}

	currentBlock, err := n.RetrieveLatest(ctx)
	if err != nil {
		return nil, err
	}

	peers, err := n.addressBookEntryRepo.Entries(ctx)
	if err != nil {
		return nil, err
	}

	return &rTypes.NetworkStatusResponse{
		CurrentBlockIdentifier: currentBlock.GetRosettaBlockIdentifier(),
		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}
