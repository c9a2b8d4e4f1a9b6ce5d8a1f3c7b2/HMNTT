### Title
Cache-Miss Amplification via Unique Storage Slot Keys Bypasses contractState Cache, Forcing Sustained DB Query Flood

### Summary
The `contractState` Caffeine cache configured in `cacheManagerContractState()` uses a 2-second `expireAfterWrite` TTL with a key space of `(contractId, slotKey)` where `slotKey` is fully attacker-controlled. An unprivileged attacker can send `eth_call` requests targeting a new unique storage slot per request, guaranteeing a cache miss on every call and forcing a live DB query against `contract_state` for each. The global RPS throttle (500 req/s, shared across all callers) is the only guard, and it has no per-IP partitioning, allowing a single attacker to consume the entire budget with cache-miss requests.

### Finding Description

**Exact code locations:**

- Cache definition: `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 83–89 (`cacheManagerContractState()`)
- Cache spec: `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 28 — `"expireAfterWrite=2s,maximumSize=25000,recordStats"`
- Cache lookup path: `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 58–70 (`findStorage()`)
- DB query on miss: `web3/src/main/java/org/hiero/mirror/web3/repository/ContractStateRepository.java`, lines 19–21 (`findStorage()`) and lines 23–28 (`findStorageBatch()`)
- Global throttle (no per-IP): `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42

**Root cause and exploit flow:**

The cache key for `contractStateCache` is `(contractId, slotKey)`. The `slotKey` is a 32-byte value supplied indirectly by the attacker through the `data` field of the `eth_call` request — a contract's `SLOAD` opcode reads from a slot the attacker's calldata selects. By deploying or targeting a contract that reads `msg.data`-derived slots, or simply targeting any contract with a large slot space, the attacker generates a monotonically incrementing slot key per request.

In `ContractStateServiceImpl.findStorage()` (line 63), the cache is checked:
```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
```
A unique slot key always misses. The code then falls into `findStorageBatch()` (line 85–121), which:
1. Adds the new slot key to the per-contract `contractSlotsCache` (line 90)
2. Collects **all** previously seen slot keys for that contract (line 91)
3. Issues a batch DB query for all of them (line 103): `select slot, value from contract_state where contract_id = :contractId and slot in (:slots)`

This creates an O(N) DB query per request where N is the number of unique slots accumulated in the `contractSlotsCache` (capped at `maximumSize=1500` per contract, `maximumSize=3000` globally). The attacker can grow N to 1,500 before the per-contract cache evicts, meaning each new request triggers a batch query scanning up to 1,500 rows.

**Why the throttle fails:**

`ThrottleManagerImpl.throttle()` uses a single global `rateLimitBucket` (500 tokens/s) and `gasLimitBucket`. There is no per-IP, per-source, or per-session partitioning. A single attacker can legally consume all 500 req/s. At minimum gas (21,000), `scaleGas(21000) = floor(21000/10000) = 2` tokens are consumed from a 750,000-token gas bucket — negligible. The RPS limit of 500/s is the binding constraint, and it is shared globally.

### Impact Explanation

At 500 req/s with unique slot keys:
- 500 DB queries/second against `contract_state`, each potentially scanning up to 1,500 rows (batch path)
- The `contract_state` table is indexed by `(contract_id, slot)`, but 500 index lookups/second with growing batch sizes constitutes real I/O pressure
- On a system with baseline load of 100–200 req/s (normal usage), the attacker's 500 req/s of cache-miss DB queries represents a 250–500% increase in DB query rate, far exceeding the 30% threshold
- CPU elevation from query parsing, result serialization, and EVM execution overhead compounds the DB I/O impact
- The attack is sustained indefinitely: the 2-second TTL means the attacker only needs to cycle through 1,000 unique slot keys to maintain a permanent stream of cache misses

### Likelihood Explanation

**Preconditions:** None. No account, no API key, no authentication required. The `/api/v1/contracts/call` endpoint is publicly accessible.

**Attacker capability:** Any HTTP client capable of sending 500 POST requests/second. A single machine with a standard network connection can achieve this. The attacker needs only a valid contract address (any deployed contract on the network) and a way to vary the slot key (trivially done by varying calldata).

**Repeatability:** Fully deterministic and repeatable. The attack runs indefinitely as long as the attacker sends requests at the global RPS cap with unique slot keys. No state is required between requests.

### Recommendation

1. **Add per-IP rate limiting** at the `ThrottleManagerImpl` or servlet filter layer. The current global bucket must be supplemented with per-source-IP buckets to prevent a single caller from consuming the full budget.
2. **Increase the `contractState` cache TTL** from 2 seconds to a value aligned with the Hedera block time (~2–3 seconds is already borderline; consider 5–10 seconds) to reduce the frequency at which cached entries become stale and re-queryable.
3. **Cap the batch query size** in `ContractStateServiceImpl.findStorageBatch()` to a fixed maximum (e.g., 100 slots) regardless of the `contractSlotsCache` size, to bound the per-request DB work.
4. **Consider negative caching**: cache misses (slot not found in DB) should also be cached for the TTL duration to prevent repeated DB queries for non-existent slots.

### Proof of Concept

```bash
# Target: any deployed contract address on the network
CONTRACT="0x000000000000000000000000000000000000ABCD"

# Generate 500 unique eth_call requests per second, each reading a different storage slot
# The calldata encodes a SLOAD of slot N (attacker increments N each request)
for i in $(seq 1 10000); do
  SLOT=$(printf "%064x" $i)  # unique 32-byte slot key per request
  curl -s -X POST https://mirror-node/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT\",\"data\":\"0x${SLOT}\",\"gas\":21000}" &
  # Rate: ~500/s with parallel requests
done
```

**Expected result:** Each request misses the `contractState` cache (unique slot key, 2s TTL), triggers `findStorageBatch()` with a growing list of accumulated slots, and issues a live DB query against `contract_state`. Monitoring `pg_stat_activity` on the PostgreSQL instance will show a sustained flood of `select slot, value from contract_state where contract_id = ? and slot in (...)` queries. CPU and I/O metrics on the DB host will show elevation proportional to the request rate, exceeding 30% above baseline within seconds of attack initiation.     private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
```java
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
```
