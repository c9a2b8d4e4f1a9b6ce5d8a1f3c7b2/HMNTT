### Title
Data Race on `br.genesisBlock` in `initGenesisRecordFile` via Unsynchronized Pre-Check

### Summary
`initGenesisRecordFile()` in `rosetta/app/persistence/block.go` performs an unsynchronized read of `br.genesisBlock.ConsensusStart` at line 244 as a fast-path guard, while the only write to `br.genesisBlock` occurs inside `sync.Once` at line 258. Because the guard read is entirely outside any synchronization primitive, concurrent goroutines can race the write, producing a Go data race (undefined behavior) that can expose a partially-written `recordBlock` struct to callers, corrupting genesis block data returned over the public Rosetta HTTP API.

### Finding Description

**Exact code path:**

`rosetta/app/persistence/block.go`, function `initGenesisRecordFile`, lines 243–263:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    // LINE 244 — unsynchronized read of a multi-field struct field
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil                          // fast-path: no sync primitive involved
    }
    // ... DB query into local `rb` ...
    br.once.Do(func() {
        br.genesisBlock = rb               // LINE 258 — struct write, not atomic
    })
    log.Infof("...", br.genesisBlock.Index) // LINE 261
    return nil
}
```

`blockRepository` is a single shared instance created once in `NewBlockRepository` (line 127–133) and injected into every HTTP handler. Every public endpoint (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) calls `initGenesisRecordFile` on each request.

**Root cause and failed assumption:**

The developer assumed `sync.Once` makes the initialization safe under concurrency. It does not. `sync.Once.Do` only guarantees the closure runs exactly once and provides a happens-before edge between the closure's completion and the *return of subsequent `Do` calls*. It provides **no** happens-before edge between the write at line 258 and the bare read at line 244, which is outside any synchronization scope.

`recordBlock` is a five-field struct (`ConsensusStart int64`, `ConsensusEnd int64`, `Hash string`, `Index int64`, `PrevHash string`). The assignment `br.genesisBlock = rb` is compiled into multiple store instructions — it is not atomic. A concurrent reader at line 244 can observe a partially-written struct.

**Race scenario (two goroutines):**

- **G-A** enters `initGenesisRecordFile`, passes the line-244 check (sees `-1`), queries DB, enters `once.Do`, and begins writing `br.genesisBlock = rb` field-by-field. Suppose `ConsensusStart` is stored first.
- **G-B** concurrently reads `br.genesisBlock.ConsensusStart` at line 244. It now sees the new non-`-1` value (written by G-A) and returns `nil` immediately — without calling `once.Do`, so it has **no** happens-before guarantee on the remaining fields.
- G-B proceeds to call `br.genesisBlock.ToBlock(br.genesisBlock)` (line 187 in `RetrieveGenesis`) or reads `br.genesisBlock.Index` (line 203/211/235) while G-A is still writing `Hash`, `Index`, `PrevHash`.
- G-B reads a torn struct: `ConsensusStart` is correct, but `Hash` is `""`, `Index` is `0`, `PrevHash` is `""`.

This is confirmed as a data race by the Go memory model: concurrent unsynchronized read and write to the same memory location is undefined behavior regardless of architecture.

**Why existing checks are insufficient:**

`sync.Once` at line 257 only serializes the write. It does not protect the read at line 244. Any goroutine that short-circuits at line 244 after the write has *started but not finished* will observe a partially-written struct and then use it without any further synchronization. The Go race detector (`go test -race`) would flag this immediately.

### Impact Explanation

Callers of `RetrieveGenesis`, `FindByHash`, `FindByIndex`, `FindByIdentifier`, and `RetrieveLatest` can receive a `types.Block` constructed from a torn `recordBlock` where `Hash` is empty string and `Index` is `0`. In the Rosetta protocol, the genesis block hash and index are used as the canonical chain anchor; returning a zeroed hash or index causes downstream Rosetta clients and indexers to treat the chain as starting from an invalid block, breaking block-range queries and balance reconciliation. The corruption is silent — no error is returned to the caller.

### Likelihood Explanation

The race window exists on every cold start of the mirror node (i.e., every restart or first boot). An unprivileged attacker needs only to:
1. Detect that the node has restarted (e.g., by polling `/network/status` until it responds).
2. Immediately flood any Rosetta block endpoint with many concurrent HTTP requests.

No credentials, tokens, or special network position are required. The Rosetta API is designed to be publicly accessible. The race is repeatable on every restart and is more likely to trigger under load, which is exactly the condition an attacker can manufacture.

### Recommendation

Replace the unsynchronized fast-path check and `sync.Once` pattern with a `sync.RWMutex` or move the entire initialization — including the guard — inside `sync.Once`:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id",
            br.treasuryEntityId.EncodedId)).First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            // Reset once so callers can retry when DB becomes available
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: if retry-on-failure is required (current behavior when DB is unavailable), use a `sync.RWMutex` with a separate `initialized bool` flag instead of `sync.Once`, so the lock protects both the guard read and the struct write atomically.
