### Title
GraphQL Alias Amplification Enables Unbounded DB Query Multiplication via `findByEvmAddress`

### Summary
An unprivileged user can craft a single GraphQL document containing up to ~58 aliased `account(input: {evmAddress: ...})` fields, each of which independently resolves through `AccountController` → `EntityServiceImpl.getByEvmAddressAndType` → `EntityRepository.findByEvmAddress`, issuing a separate SQL query per alias. The `MaxQueryComplexityInstrumentation(200)` uses a flat weight of 1 per field and does not distinguish DB-backed root resolvers from cheap scalar fields, so the complexity budget is not exhausted before the token limit is reached. No rate limiting exists in the GraphQL module, allowing an attacker to sustain this amplification indefinitely.

### Finding Description

**Exact code path:**

`AccountController.account` (line 33–58) is the `@QueryMapping` resolver. For each aliased `account` field in a document, GraphQL-Java invokes this method independently. When `evmAddress` is supplied, it calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)` (line 51–54), which delegates to `EntityServiceImpl.getByEvmAddressAndType` (line 34–41). For any non-zero-padded EVM address, this calls `entityRepository.findByEvmAddress(evmAddressBytes)` (line 40), executing the native SQL:

```sql
select * from entity where evm_address = ?1 and deleted is not true
```

**Root cause — complexity instrumentation does not reflect DB cost:**

`GraphQlConfiguration` installs `MaxQueryComplexityInstrumentation(200)` with no custom `FieldComplexityCalculator`. The default assigns complexity 1 to every field. A minimal aliased query:

```graphql
{ a1: account(input:{evmAddress:"0x0000000000000000000000000000000000000001"}){id}
  a2: account(input:{evmAddress:"0x0000000000000000000000000000000000000002"}){id}
  ...
}
```

contributes 2 complexity per alias (`account` + `id`). At 58 aliases: 116 total — well under the 200 ceiling. The complexity check never fires.

**Binding constraint is the token limit, not complexity:**

`ParserOptions` sets `maxTokens(1000)`. Each alias consumes ~17 GraphQL tokens (LBRACE, NAME, COLON, NAME, LPAREN, NAME, COLON, LBRACE, NAME, COLON, STRING, RBRACE, RPAREN, LBRACE, NAME, RBRACE, RBRACE). With 2 outer braces: `(1000 − 2) / 17 ≈ 58` aliases per document. This is the actual ceiling.

**No rate limiting in the GraphQL module:**

The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` classes exist only in the `web3` module and apply exclusively to `ContractCallRequest`. There is no equivalent bucket-based or per-IP rate limiter wired into the GraphQL service. An attacker can send requests at arbitrary frequency.

### Impact Explanation

Each HTTP request to `/graphql` with 58 aliased `account` fields causes 58 synchronous JDBC calls against the `entity` table. With concurrent requests from a single client (or a small botnet), the DB connection pool can be saturated, increasing query latency for all users of the mirror node. This is a griefing-class denial-of-service with no authentication requirement and no economic cost to the attacker. Severity is Medium per the stated scope.

### Likelihood Explanation

The attack requires zero privileges, no API key, and no special knowledge beyond the public GraphQL schema. The exploit document is trivially generated programmatically. The amplification factor (~58×) is meaningful enough to stress a shared DB connection pool under sustained load. Any motivated actor can repeat this indefinitely.

### Recommendation

1. **Assign higher complexity weight to root-level DB-backed resolvers.** Supply a custom `FieldComplexityCalculator` to `MaxQueryComplexityInstrumentation` that assigns a weight of, e.g., 10 to the `account` root field, reducing the effective alias ceiling to ~18 before the 200 budget is exhausted.
2. **Add per-IP or global rate limiting to the GraphQL module**, mirroring the bucket4j pattern already used in `web3/ThrottleConfiguration`.
3. **Explicitly limit the number of top-level selections** in a single operation via a custom `Instrumentation` or `WebGraphQlInterceptor` that counts root-level fields before execution.

### Proof of Concept

```bash
# Build a document with 58 aliased account lookups
python3 -c "
aliases = '\n'.join(
  f'  a{i}: account(input:{{evmAddress:\"0x{i:040x}\"}}){{id}}'
  for i in range(1, 59)
)
print('{\n' + aliases + '\n}')
" > payload.graphql

# Send in a tight loop (no auth required)
while true; do
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$(cat payload.graphql | tr -d '\n' | sed 's/"/\\"/g')\"}" &
done
```

Each iteration of the loop dispatches 58 `SELECT * FROM entity WHERE evm_address = ?` queries. Under sustained concurrent load this exhausts the DB connection pool and degrades service for all users. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
