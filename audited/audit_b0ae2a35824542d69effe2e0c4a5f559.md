### Title
Unauthenticated Concurrent Requests to `/contracts/:contractId/results/:consensusTimestamp` Exhaust DB Connection Pool via Unconditional 5-Parallel-Query Execution

### Summary
The `getContractResultsByTimestamp` handler in `rest/controllers/contractController.js` unconditionally fires 5 parallel database queries via `Promise.all` inside `getDetailedContractResults()` before checking whether `callResult` is null. Because the REST API has no rate limiting and requires no authentication, an unprivileged attacker who identifies any contract result with a null `callResult` can send a flood of concurrent requests, each consuming up to 5 DB connections simultaneously, exhausting the finite pgbouncer/pool connection budget and degrading or denying service to the mirror-node REST tier.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 18 registers the route with no authentication guard: [1](#0-0) 

`getContractResultsByTimestamp` (lines 1002–1042) calls `getDetailedContractResults` at line 1016–1017 **before** the null check at line 1026: [2](#0-1) 

`getDetailedContractResults` (lines 1205–1220) issues all 5 queries in a single `Promise.all`, meaning all 5 DB connections are acquired simultaneously: [3](#0-2) 

The null check at line 1026 only sets `res.locals.statusCode = 206`; it does not short-circuit or skip any query. All 5 queries have already completed (or are in-flight) by the time this check runs.

**Root cause:** The design assumes that setting a 206 status is a lightweight "annotation" step, but the expensive work (5 parallel DB queries) is done unconditionally before it. There is no early-exit path for the null-`callResult` case.

**Why existing checks fail:**

- `server.js` applies no rate-limiting middleware to the contracts routes — only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler` (which passes unauthenticated requests through), `metricsHandler`, and `responseHandler`: [4](#0-3) 

- The DB pool is bounded: `rest/dbpool.js` sets `max: config.db.pool.maxConnections`: [5](#0-4) 

- The Helm/pgbouncer config caps `mirror_rest` at `max_user_connections: 250`: [6](#0-5) 

With 5 connections consumed per request, only ~50 concurrent requests are needed to saturate the 250-connection budget.

### Impact Explanation
When the DB connection pool is exhausted, all subsequent REST API requests that require a DB connection will block until `connectionTimeoutMillis` expires and then fail. This affects the entire REST service, not just the contract-results endpoint. Because mirror-node REST nodes are stateless and horizontally scaled, an attacker targeting 30%+ of nodes simultaneously (each with its own bounded pool) can degrade network-wide query capacity, matching the "shutdown of ≥30% of processing nodes" severity threshold. The `statement_timeout` limits per-query duration but does not prevent pool exhaustion under sustained concurrent load.

### Likelihood Explanation
- **No privileges required:** The endpoint is fully public.
- **Stable trigger:** Contract results with null `callResult` are a documented, expected state (the server explicitly handles them with 206). An attacker can enumerate such transactions via the public ledger or by querying `/api/v1/contracts/results` and filtering for partial responses.
- **Low cost:** A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple async script) can sustain 50–200 concurrent requests indefinitely.
- **Repeatable:** The same timestamp can be requested repeatedly; there is no deduplication or in-flight request coalescing.

### Recommendation
1. **Add rate limiting to the REST API** for the contracts routes (e.g., `express-rate-limit` or an ingress-level Traefik `rateLimit` middleware matching what is already used for the Rosetta service).
2. **Limit in-flight DB connections per endpoint** using a semaphore or concurrency limiter (e.g., `p-limit`) around `getDetailedContractResults`.
3. **Optionally restructure the handler** to fetch `contractResults` first (one query), check `callResult`, and only proceed with the remaining 4 queries if the result is non-null — though this changes the semantics of the 206 response.
4. **Set a lower `connectionTimeoutMillis`** and ensure pgbouncer queue limits are configured to shed load quickly rather than queue indefinitely.

### Proof of Concept
```bash
# 1. Find a contract result with null callResult (returns HTTP 206):
curl -s "https://<mirror-node>/api/v1/contracts/<contractId>/results/<timestamp>"
# Confirm response status is 206.

# 2. Flood the endpoint with 100 concurrent requests:
ab -n 10000 -c 100 \
  "https://<mirror-node>/api/v1/contracts/<contractId>/results/<timestamp>"

# 3. Observe: after pool exhaustion (~50 concurrent requests sustained),
#    all other REST API endpoints begin returning 503 / connection timeout errors.
#    Monitor with:
curl -s "https://<mirror-node>/api/v1/transactions" 
# Will start failing or timing out once the pool is saturated.
```
