### Title
Unbounded Goroutine Leak via Context-less `hiero.TransactionExecute` in `ConstructionSubmit`

### Summary
`ConstructionSubmit` explicitly discards the incoming HTTP request context and calls `hiero.TransactionExecute` with no deadline or timeout. If the Hiero SDK gRPC call to a consensus node hangs, the goroutine serving that HTTP request is blocked indefinitely. Because no application-level timeout guards the SDK call and no `SetRequestTimeout`/`SetGrpcDeadline` is configured on the `sdkClient`, repeated submissions targeting an unresponsive node accumulate leaked goroutines, degrading or denying service to legitimate callers.

### Finding Description
**Exact location:** `rosetta/app/services/construction_service.go`, `ConstructionSubmit`, lines 332–368.

```go
func (c *constructionAPIService) ConstructionSubmit(
    _ context.Context,          // ← context explicitly discarded
    request *rTypes.ConstructionSubmitRequest,
) (*rTypes.TransactionIdentifierResponse, *rTypes.Error) {
    ...
    _, err = hiero.TransactionExecute(transaction, c.sdkClient)  // ← no deadline
```

**Root cause – two compounding omissions:**

1. The `context.Context` passed by the HTTP framework (which carries the `WriteTimeout` deadline) is thrown away with `_`. Even if the HTTP server closes the TCP connection after `WriteTimeout` (default 10 s), the goroutine is not cancelled; it continues to block inside the SDK.

2. The `sdkClient` is constructed in `NewConstructionAPIService` (lines 596–645) with only `sdkClient.SetMaxAttempts(1)`. No `SetRequestTimeout`, `SetGrpcDeadline`, or equivalent is ever called on it anywhere in the rosetta Go codebase (confirmed by grep). The Hiero Go SDK's default gRPC deadline is on the order of minutes; if the target node is completely unresponsive the SDK may never return.

**Exploit flow:**

1. Attacker crafts a protobuf-encoded `CryptoTransferTransaction` (or any supported type) whose `nodeAccountID` field points to a consensus node that is partitioned, firewalled, or otherwise unresponsive. No valid signature is required to reach `TransactionExecute`; `unmarshallTransactionFromHexString` only deserialises bytes.
2. Attacker POSTs the hex-encoded bytes to `/construction/submit`.
3. The handler goroutine enters `hiero.TransactionExecute` and blocks on the gRPC dial/send.
4. After `WriteTimeout` (10 s) the HTTP connection is torn down, but the goroutine remains alive and blocked — Go's `net/http` does not cancel handler goroutines on connection close.
5. Repeating step 2 accumulates one leaked goroutine per request. Each goroutine holds its stack (initial 8 KB, growing as needed) plus any SDK-internal allocations.

**Why existing checks are insufficient:**

- `WriteTimeout = 10 s` closes the TCP socket but does **not** interrupt the goroutine.
- The Traefik `inFlightReq: amount: 5` middleware (in `charts/hedera-mirror-rosetta/values.yaml`) is optional infrastructure; it is absent in bare deployments and can be bypassed by direct access to the pod/service port. It is not enforced at the application layer.
- `SetMaxAttempts(1)` prevents SDK-level retries but does not bound the time spent on a single attempt.

### Impact Explanation
Each hung goroutine leaks memory (stack + gRPC buffers). Under sustained attack the process heap grows until OOM-killed or until the scheduler spends increasing time context-switching blocked goroutines, raising latency for all other endpoints. Because `ConstructionSubmit` is the only path to gossip transactions, blocking it constitutes a targeted denial-of-service against the gossip function of the Rosetta node. Severity: **Medium–High** (availability impact on a critical submission path; no confidentiality or integrity impact).

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. Crafting a structurally valid Hiero protobuf transaction is trivial using the open-source SDK. Targeting an unresponsive node is achievable by specifying a node account ID whose gRPC endpoint is firewalled or by operating a malicious node. The attack is repeatable and requires no privileged access. The Traefik rate limit (5 in-flight) reduces throughput but does not eliminate the leak; an attacker sustaining 5 concurrent hung requests continuously accumulates goroutines over time.

### Recommendation
1. **Propagate the request context with a deadline** into `TransactionExecute`:
   ```go
   func (c *constructionAPIService) ConstructionSubmit(
       ctx context.Context,   // accept, do not discard
       ...
   ) ... {
       ctx, cancel := context.WithTimeout(ctx, submitTimeout)
       defer cancel()
       _, err = hiero.TransactionExecuteWithContext(ctx, transaction, c.sdkClient)
   ```
2. **Set a gRPC deadline on the SDK client** at construction time:
   ```go
   sdkClient.SetRequestTimeout(30 * time.Second)
   // or
   sdkClient.SetGrpcDeadline(30 * time.Second)
   ```
3. **Enforce an application-level concurrency limit** (e.g., a semaphore or `golang.org/x/sync/semaphore`) on `ConstructionSubmit` independent of Traefik.

### Proof of Concept
```bash
# 1. Build a minimal valid CryptoTransfer protobuf targeting an unreachable node
#    (e.g., node 0.0.3 with gRPC endpoint 10.0.0.1:50211 that is firewalled).
#    Use the Hiero Go SDK to serialise and hex-encode the frozen, signed transaction.

# 2. Send concurrent requests:
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-online>:5700/construction/submit \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},
         "signed_transaction":"0x<hex-of-crafted-tx>"}' &
done
wait

# 3. Observe goroutine count via pprof:
curl http://<rosetta-online>:5700/debug/pprof/goroutine?debug=1 | grep -c "TransactionExecute"
# Each blocked goroutine appears in the stack trace; count grows with each batch.
```
