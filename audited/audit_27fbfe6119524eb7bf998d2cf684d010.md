### Title
Timestamp Range Binding Bypassed in `getContractResultsById`, Enabling Unbounded Historical Enumeration

### Summary
The `getContractResultsById` handler in `rest/controllers/contractController.js` intentionally skips the `optimizeTimestampFilters` call (which invokes `bindTimestampRange`) when a `contractId` is present. This means an unprivileged caller can supply an open-ended lower-bound timestamp filter such as `timestamp=gte:0` and, by following pagination links, enumerate the complete execution history of any contract from genesis with no temporal depth restriction. The sibling endpoint `getContractResults` (no contract ID) does enforce this bound.

### Finding Description

**Exact code path:**

In `extractContractResultsByIdQuery` (line 504–505 of `rest/controllers/contractController.js`):

```js
const {filters: optimizedTimestampFilters, next} =
  contractId === undefined
    ? await optimizeTimestampFilters(timestampFilters, order)
    : {filters: timestampFilters};   // ← raw user filters, no binding
``` [1](#0-0) 

When `contractId` is defined (i.e., every call through `getContractResultsById`), the branch short-circuits to `{filters: timestampFilters}`, returning the raw, user-supplied timestamp filters without calling `optimizeTimestampFilters`.

**What `optimizeTimestampFilters` does (and what is skipped):**

`optimizeTimestampFilters` (line 384–404) calls `bindTimestampRange`, which enforces `maxTransactionsTimestampRangeNs` — a configured maximum window size. If the user-supplied range exceeds this window, `bindTimestampRange` clips it and returns a `next` cursor. [2](#0-1) 

`bindTimestampRange` in `rest/timestampRange.js` (line 19–38) explicitly clips the range:

```js
if (boundRange.end - boundRange.begin + 1n <= maxTransactionsTimestampRangeNs) {
  return {range: boundRange};
}
// else clip to maxTransactionsTimestampRangeNs
``` [3](#0-2) 

**Why existing checks are insufficient:**

`contractResultsFilterValidityChecks` (line 273–279) only adds a special operator check for `BLOCK_NUMBER`; it performs no timestamp range validation. [4](#0-3) 

`buildAndValidateFilters` validates format (e.g., timestamp must parse as a valid nanosecond value) but does not enforce a maximum range window. A single open-ended filter like `timestamp=gte:0` passes all format checks. [5](#0-4) 

The raw `timestampFilters` array is then iterated at lines 510–518 and appended directly to the SQL `WHERE` clause via `updateConditionsAndParamsWithInValues`, producing an unbounded `cr.consensus_timestamp >= 0` predicate. [6](#0-5) 

The downstream query in `ContractService.getContractResultsByIdAndFilters` applies only a `LIMIT` clause — it does not add any timestamp floor. [7](#0-6) 

### Impact Explanation

An unprivileged external user can reconstruct the complete, ordered execution history of any contract by issuing repeated paginated requests. Each page returns up to `defaultLimit` rows; the `next` link in the response provides the cursor for the next page. There is no server-side cap on the total number of pages or the total timestamp span covered. This exposes the full `contract_result` table rows for a given contract (gas, call data, return values, sender, error messages, etc.) from timestamp 0 to present. Beyond data exposure, each such request forces a full or near-full index scan on `contract_result` for the given `contract_id`, creating a resource-exhaustion vector against the database.

### Likelihood Explanation

The exploit requires zero privileges — only knowledge of a contract ID (which is public on-chain). The request is a standard HTTP GET with a single query parameter. It is trivially repeatable and scriptable. Any actor who can reach the REST API can execute it.

### Recommendation

Apply `optimizeTimestampFilters` unconditionally, regardless of whether `contractId` is set. The current conditional was introduced to avoid the `bindTimestampRange` DB lookup for the by-ID path, but the range-clipping logic should still run. Alternatively, enforce a maximum timestamp span in `contractResultsFilterValidityChecks` for the `TIMESTAMP` key, mirroring the `maxTimestampRange` check already present in `parseTimestampFilters` when `validateRange = true`.

Change line 504–505 from:
```js
contractId === undefined ? await optimizeTimestampFilters(...) : {filters: timestampFilters}
```
to always call `optimizeTimestampFilters`, or add an explicit upper-bound injection when only a lower-bound timestamp filter is present.

### Proof of Concept

**Preconditions:** Network access to the mirror node REST API. Any valid contract ID (e.g., `0.0.5001`).

**Steps:**

1. Send:
   ```
   GET /api/v1/contracts/0.0.5001/results?timestamp=gte:0&order=asc&limit=100
   ```
2. The server returns up to 100 contract results starting from the very first recorded execution of contract `0.0.5001`.
3. The response includes a `links.next` field with a cursor timestamp.
4. Repeat with the cursor:
   ```
   GET /api/v1/contracts/0.0.5001/results?timestamp=gte:<cursor>&order=asc&limit=100
   ```
5. Continue until `links.next` is `null`. The complete execution history of the contract has been retrieved with no server-side restriction on the total historical depth.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
```

const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};
  }

  const {maxTransactionsTimestampRangeNs} = queryConfig;
  const boundRange = Range(range?.begin ?? (await getFirstTransactionTimestamp()), range?.end ?? nowInNs(), '[]');
  if (boundRange.end - boundRange.begin + 1n <= maxTransactionsTimestampRangeNs) {
    return {range: boundRange};
  }

  let next;
  if (order === orderFilterValues.DESC) {
    next = boundRange.begin = boundRange.end - maxTransactionsTimestampRangeNs + 1n;
  } else {
    next = boundRange.end = boundRange.begin + maxTransactionsTimestampRangeNs - 1n;
  }

  return {range: boundRange, next: nsToSecNs(next)};
};
```
