### Title
Unbounded Request Rate on `GET /api/v1/topics/{id}` Enables Unauthenticated Resource Exhaustion via HTTP/2 Multiplexing

### Summary
The `TopicController.getTopic()` handler and its backing `TopicServiceImpl.findById()` perform three synchronous database queries per request with no application-level rate limiting, per-IP throttling, or HTTP/2 stream-count cap. An unprivileged attacker can open multiple persistent HTTP/2 connections and multiplex hundreds of concurrent streams per connection, driving unbounded database load and exhausting the server's HikariCP connection pool without any existing control rejecting or slowing the traffic.

### Finding Description
**Code path:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java` lines 31–37: `getTopic()` calls `topicService.findById()`, `entityService.findById()`, and `customFeeService.findById()` — three DB round-trips per request, no guard.
- `rest-java/src/main/java/org/hiero/mirror/restjava/service/TopicServiceImpl.java` lines 19–21: `findById()` is a bare `topicRepository.findById(id.getId())` with no caching or throttle.

**Root cause:** The rest-java module contains zero rate-limiting infrastructure for the topics endpoint. The only servlet filters registered are `LoggingFilter` and `MetricsFilter` (observability only). The `web3` module's `ThrottleManagerImpl`/`ThrottleConfiguration` (bucket4j) is entirely absent from rest-java. `RestJavaProperties` exposes only response-header configuration — no request-rate properties exist.

**Deployment controls are also absent:** The Traefik ingress middleware (`charts/hedera-mirror-rest-java/values.yaml` lines 150–155) configures only `circuitBreaker` (error-ratio threshold) and `retry` — no `rateLimit` stanza. The circuit breaker trips only on ≥10 % network errors or ≥25 % 5xx responses, which a slow-drip flood never triggers.

**HTTP/2 amplification:** Spring Boot's embedded Tomcat defaults allow up to 200 concurrent streams per HTTP/2 connection (`maxConcurrentStreams` is not overridden anywhere in rest-java). An attacker opening *N* connections can therefore sustain *200 × N* simultaneous in-flight requests. Each request consumes one HikariCP connection for each of the three DB calls, multiplying pool pressure.

### Impact Explanation
Each `getTopic` call acquires up to three database connections from the shared HikariCP pool. Pool exhaustion causes all rest-java endpoints (not just topics) to queue or reject requests, producing cascading 503s across the service. Because the mirror node is a public read API, availability degradation directly affects all downstream consumers. Severity is medium (griefing / availability, no on-chain economic damage).

### Likelihood Explanation
No authentication or API key is required. The attack requires only a standard HTTP/2 client (e.g., `h2load`, `curl --http2`, Python `httpx`). A single commodity machine can open dozens of connections and saturate the pool. The circuit breaker will not fire because the attacker's own requests succeed (200 OK or 404), keeping the error ratio below the 10 %/25 % thresholds. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Application-level rate limiting:** Add a bucket4j `RateLimitFilter` (mirroring the web3 `ThrottleConfiguration`) to rest-java, keyed on `X-Forwarded-For` / `X-Real-IP`, applied to `/api/v1/topics/**`.
2. **HTTP/2 stream cap:** Set `server.tomcat.max-http2-concurrent-streams` (or the equivalent `Http2Protocol` attribute) to a low value (e.g., 20–50) in `application.yml` to limit per-connection parallelism.
3. **Traefik rate limiting:** Add a `rateLimit` middleware entry in `charts/hedera-mirror-rest-java/values.yaml` (e.g., `average: 100`, `burst: 50`, `sourceCriterion: ipStrategy`) alongside the existing circuit-breaker middleware.
4. **Result caching:** Cache `findById` results for a short TTL (topic metadata is immutable once created) to reduce DB pressure even if rate limiting is bypassed.

### Proof of Concept
```bash
# Install h2load (nghttp2 package)
# Replace HOST with the mirror node REST-Java endpoint

h2load \
  --h1 \                          # force HTTP/2
  -n 100000 \                     # total requests
  -c 50 \                         # concurrent connections
  -m 200 \                        # max streams per connection (Tomcat default)
  "https://HOST/api/v1/topics/0.0.1000"

# Expected result:
# - HikariCP pool exhausted within seconds
# - Subsequent requests to ANY rest-java endpoint return 503 / connection timeout
# - Traefik circuit breaker does NOT trip (all attacker requests return 200/404)
# - No authentication or special privilege required
```
