### Title
Unauthenticated Repeated All-Zero EVM Address Queries Cause Unbounded DB Connection Pool Exhaustion

### Summary
Any unauthenticated caller can send an unlimited number of GraphQL `account` queries with `evmAddress: "0x0000000000000000000000000000000000000000"`. The `ByteBuffer` zero-prefix check in `getByEvmAddressAndType()` deterministically routes this input to `entityRepository.findById(0L)` on every request. Because no rate limiting or result caching exists at the application layer, each request consumes a DB connection pool slot for a query that always returns empty, enabling a griefing/resource-exhaustion attack.

### Finding Description
**Code path:**

`AccountController.java:51-54` — any caller supplying `evmAddress` (no authentication required) reaches: [1](#0-0) 

`GraphQlUtils.java:87-98` — `decodeEvmAddress("0x0000000000000000000000000000000000000000")` strips the `0x` prefix and hex-decodes to 20 zero bytes without error: [2](#0-1) 

`EntityServiceImpl.java:34-41` — the critical routing logic: [3](#0-2) 

With 20 zero bytes in the buffer:
- `buffer.getInt()` (bytes 0–3) = `0` → condition true
- `buffer.getLong()` (bytes 4–11) = `0` → condition true
- `buffer.getLong()` (bytes 12–19) = `0` → calls `entityRepository.findById(0L)`

`EntityRepository` extends `CrudRepository<Entity, Long>`, so `findById(0L)` issues a real SQL `SELECT` against the `entity` table. No entity with `id = 0` exists in Hedera (IDs start at 1), so every call returns `Optional.empty()` after consuming a DB connection.

**Why existing checks are insufficient:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` in `GraphQlConfiguration.java:43-45` limit per-query structure, not request rate: [4](#0-3) 

- `CachedPreparsedDocumentProvider` caches the **parsed query document** (query string → AST), not execution results. Repeated identical queries still execute against the DB: [5](#0-4) 

- `LoggingFilter` only logs requests; it performs no rate limiting or blocking: [6](#0-5) 

- No authentication is enforced anywhere in the GraphQL controller layer. [7](#0-6) 

### Impact Explanation
An attacker flooding the endpoint with this single, trivially crafted query saturates the DB connection pool. Legitimate queries queue or fail with connection-timeout errors, degrading or denying service to all users. The attack requires no credentials, no knowledge of real data, and no complex query construction. The impact is service degradation (griefing) with no direct economic damage to network participants, consistent with the Medium severity classification.

### Likelihood Explanation
The precondition is zero: no account, no token, no prior knowledge of the system state. The payload is a single static GraphQL query. Any script-kiddie with `curl` or a simple loop can sustain the attack indefinitely. The deterministic nature of the all-zero address (always `findById(0)`, always empty) makes it ideal for automated flooding with no need to vary input.

### Recommendation
Apply at least one of the following mitigations:

1. **Reject the all-zero address explicitly** in `decodeEvmAddress()` or at the start of `getByEvmAddressAndType()`:
   ```java
   if (Arrays.equals(evmAddressBytes, new byte[20])) {
       return Optional.empty(); // fast-path, no DB hit
   }
   ```

2. **Cache entity lookup results** (not just parsed documents) using Spring's `@Cacheable` on `EntityServiceImpl` methods, so repeated identical inputs do not hit the DB.

3. **Add application-level rate limiting** (e.g., Spring's `RateLimiter`, Bucket4j, or a gateway-level policy) keyed by client IP or API key, to bound the number of requests per time window.

4. **Validate that `id != 0`** after the ByteBuffer extraction before issuing `findById`, since entity id=0 is never valid in Hedera.

### Proof of Concept
```bash
# No authentication required. Run in a tight loop to exhaust the DB pool.
while true; do
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: { evmAddress: \"0x0000000000000000000000000000000000000000\" }) { balance } }"}' \
    > /dev/null
done
```

Each iteration:
1. `decodeEvmAddress` decodes to 20 zero bytes — no error thrown.
2. `ByteBuffer` zero-prefix check passes → `findById(0L)` issued.
3. DB returns empty; connection released.
4. No rate limit → loop repeats indefinitely, holding DB connections and starving the pool.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
