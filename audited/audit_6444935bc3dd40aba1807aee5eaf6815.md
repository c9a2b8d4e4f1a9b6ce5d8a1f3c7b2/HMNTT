### Title
`getEntityIdFromEvmAddress()` Silently Excludes Active Entities with `deleted = NULL` Due to Missing COALESCE, Enabling Wrong-Entity Resolution

### Summary
The SQL query in `entityFromEvmAddressQuery` uses `deleted <> true` without `COALESCE`, which in SQL three-valued logic evaluates `NULL <> true` to `NULL` (not `TRUE`), silently excluding any entity whose `deleted` column is `NULL`. The sibling alias query correctly uses `coalesce(deleted, false) <> true`. If two entities share the same EVM address — one with `deleted = false` and one with `deleted = NULL` — the query returns only the `deleted = false` entity, potentially the wrong one.

### Finding Description
**Exact code location:** `rest/service/entityService.js`, lines 22–25:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

Compare with the alias query on lines 17–20, which correctly wraps `deleted` in `COALESCE`:

```js
static entityFromAliasQuery = `select ${Entity.ID}
                               from ${Entity.tableName}
                               where coalesce(${Entity.DELETED}, false) <> true
                                 and ${Entity.ALIAS} = $1`;
``` [2](#0-1) 

**Root cause:** The `deleted` column is explicitly nullable — migration `V1.39.1__upsert_support.sql` drops both the `NOT NULL` constraint and the default value: [3](#0-2) 

In PostgreSQL, `NULL <> true` yields `NULL`, which is falsy in a `WHERE` clause. So any entity row where `deleted IS NULL` is silently dropped from the result set of `entityFromEvmAddressQuery`.

**Exploit flow (two-entity variant as asked):**
1. Entity A exists with EVM address `X`, `deleted = false` (explicitly marked not-deleted via a prior update transaction).
2. Entity B exists with the same EVM address `X`, `deleted = NULL` (freshly created or never explicitly updated).
3. The `entity__evm_address` index is **non-unique** (created as a plain index, not `UNIQUE`), so both rows coexist: [4](#0-3) 
4. An external user calls any REST endpoint that resolves an EVM address (e.g., `GET /api/v1/accounts/0x<X>`), which internally calls `getEntityIdFromEvmAddress()`. [5](#0-4) 
5. The query returns Entity A's ID (wrong entity) because Entity B's `NULL <> true` evaluates to `NULL` and is excluded.

**Simpler single-entity variant (higher likelihood):** Any entity whose `deleted` column was never explicitly set (remains `NULL`) is completely invisible to EVM address lookups — `getEntityIdFromEvmAddress()` throws `NotFoundError` even though the entity is live and valid.

### Impact Explanation
- **Wrong entity resolution:** A caller receives Entity A's encoded ID when Entity B was the intended target. Downstream authorization checks, token operations, or contract calls execute against the wrong entity.
- **Denial of service / false 404:** Any valid entity with `deleted = NULL` is unreachable via EVM address, causing `NotFoundError` for legitimate users. This affects all entities that have never received an explicit delete/undelete update.
- Severity: **Medium–High**. The wrong-entity path can lead to privilege confusion; the false-404 path is a reliable DoS on EVM-address-based lookups.

### Likelihood Explanation
- The `deleted = NULL` state is the **default** for newly created entities (no default value, no NOT NULL constraint). Any entity that has never been explicitly updated to set `deleted` will have `NULL`.
- No privileged access is required: any user can submit a lookup request using a known EVM address.
- The two-entity scenario requires the database to contain two rows with the same `evm_address`, which is permitted by the schema (non-unique index). This can arise from contract migration history or the `V1.64.2__merge_contract_entity.sql` bulk insert of former `contract` rows. [6](#0-5) 

### Recommendation
Apply `COALESCE` consistently, matching the alias query pattern:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

Additionally, consider adding a `UNIQUE` constraint on `evm_address` (where not null) in the `entity` table to prevent the two-entity collision scenario at the database level.

### Proof of Concept
1. Insert two rows into the `entity` table with identical `evm_address` values: one with `deleted = false`, one with `deleted = NULL`.
2. Call `GET /api/v1/accounts/0x<evm_address>` (or any endpoint that resolves via `getEncodedId` → `getEntityIdFromEvmAddress`).
3. Observe that the response returns the ID of the `deleted = false` entity, not the `deleted = NULL` entity.
4. For the single-entity DoS: insert one row with `deleted = NULL` and the same EVM address lookup returns HTTP 404, even though the entity is active.

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
