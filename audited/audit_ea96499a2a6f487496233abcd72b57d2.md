### Title
Unauthenticated Pagination-Based DB Connection Pool Exhaustion DoS on `/accounts/{id}/nfts`

### Summary
`AccountController.getNftsByAccountId()` accepts `limit=1` from any unauthenticated caller and returns a `links.next` pagination cursor. Because the REST API has no per-IP rate limiting and no minimum-limit enforcement, an attacker can flood the endpoint with concurrent small-limit requests, exhausting the Node.js `pg` connection pool and rendering the REST API unavailable. The importer uses a separate DB user/pool, so gossip ingestion is not directly blocked, but the REST API is fully denied.

### Finding Description
**Code path:**
- `rest/routes/accountRoute.js` line 15: `router.getExt(getPath('nfts'), AccountController.getNftsByAccountId)`
- `rest/controllers/accountController.js` lines 90–103: `getNftsByAccountId` calls `utils.buildAndValidateFilters`, then `NftService.getNfts(query)` (one SQL query per HTTP request, potentially a 3-branch `UNION ALL`), then `this.getPaginationLink(...)` which emits `links.next` whenever `nfts.length === query.limit`.
- `rest/utils.js` lines 544–553 (`getLimitParamValue`): caps the limit at `responseLimit.max` (default 100) but enforces **no minimum**; `limit=1` is fully accepted.
- `rest/dbpool.js` lines 7–16: the pool is bounded by `config.db.pool.maxConnections` (default value parsed from config); there is no per-request or per-IP queue guard.
- `rest/server.js` lines 68–144: middleware stack is `cors → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler → responseHandler → handleError`. **No rate-limiting middleware is present.**

**Root cause:** The REST API layer has no request-rate limiter and no minimum-limit guard. Every HTTP request to `getNftsByAccountId` acquires a DB connection from the shared `pg` pool. An attacker who sends N concurrent requests each with `limit=1` holds N pool slots simultaneously. When N ≥ `maxConnections`, all subsequent REST requests block on pool acquisition until `connectionTimeoutMillis` expires, returning errors to all callers.

**Why existing checks fail:**
- The max-limit cap (`responseLimit.max = 100`) only prevents oversized responses; it does not prevent high request frequency.
- The `authHandler` middleware only grants a higher limit ceiling to authenticated users; it does not throttle unauthenticated ones.
- The pgbouncer config (`max_user_connections: 250` for `mirror_rest`) limits the REST user's DB connections but does not rate-limit HTTP requests; an attacker can saturate all 250 slots with concurrent HTTP connections.

### Impact Explanation
When the `pg` pool is saturated, every REST API endpoint that touches the database returns 503/timeout errors. The REST API becomes fully unavailable to all users. The importer (`mirror_importer` DB user, separate pool) continues operating, so gossip transaction ingestion is not directly interrupted; however, the public REST API — the primary interface for querying mirror-node state — is denied. Severity: **High** (availability impact on the public API surface).

### Likelihood Explanation
No authentication, no CAPTCHA, no rate limit, and no minimum limit are required. A single attacker with a modest number of concurrent HTTP connections (matching `maxConnections`) can trigger this from a single IP. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public OpenAPI spec. The pagination `links.next` cursor makes it easy to sustain the attack indefinitely across an account with many NFTs.

### Recommendation
1. **Enforce a minimum limit** (e.g., `limit >= 1` is fine, but add a server-side floor of at least 1 and document it; the real fix is rate limiting, not raising the floor).
2. **Add per-IP rate limiting** to the Express REST API (e.g., `express-rate-limit` or Traefik `rateLimit` middleware mirroring what the Rosetta service already uses).
3. **Add a global in-flight request cap** per source IP (analogous to Traefik `inFlightReq` already configured for Rosetta in `charts/hedera-mirror-rosetta/values.yaml` lines 152–156).
4. **Apply connection-pool back-pressure** by setting a short `connectionTimeoutMillis` and returning HTTP 503 early rather than queuing indefinitely.

### Proof of Concept
```bash
# Saturate the pool with concurrent limit=1 requests
ACCOUNT="0.0.1001"
BASE="https://<mirror-node>/api/v1"

for i in $(seq 1 300); do
  curl -s "$BASE/accounts/$ACCOUNT/nfts?limit=1&order=asc" &
done
wait

# All subsequent REST requests now time out or return connection errors:
curl -v "$BASE/accounts/$ACCOUNT/nfts?limit=1"
# Expected: connection timeout / 503 Service Unavailable
```

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
