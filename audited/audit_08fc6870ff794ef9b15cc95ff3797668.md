### Title
Unvalidated `X-Real-IP` Header Logged Verbatim Enables Log Injection in Rosetta Tracing Middleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Real-IP` header and returns it without any IP format validation. The raw value is interpolated directly into a log message via `fmt.Sprintf` and emitted through logrus, which does not sanitize newline characters. Any client that can reach the Rosetta service without a proxy that overwrites `X-Real-IP` (e.g., Kubernetes/direct deployments) can inject arbitrary strings — including newlines — into the access log.

### Finding Description
**Code path:**

`getClientIpAddress()` — `rosetta/app/middleware/trace.go` lines 63–74:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)   // line 64 — no validation
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 — no validation
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)  // only fallback is validated
    }
    return ipAddress
}
```

The returned string is immediately interpolated at lines 52–53:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```
and logged at line 58 via `log.Info(message)`.

**Root cause:** The function assumes `X-Real-IP` contains a valid IP address. It performs only a length check (`len(ipAddress) == 0`), not a format/validity check. `net.ParseIP()` or `net.SplitHostPort()` is never called on the header value.

**Why existing checks fail:** The only deployment that protects against this is the docker-compose nginx config (`proxy_set_header X-Real-IP $$remote_addr;` at `docker-compose.yml` line 204), which overwrites the header server-side. The Kubernetes Helm chart (`charts/hedera-mirror-rosetta/values.yaml`) configures Traefik ingress but includes no header-stripping middleware — Traefik forwards client-supplied `X-Real-IP` to the backend unchanged. Direct deployments have no protection at all.

**Exploit flow:**
1. Attacker sends: `GET /network/list HTTP/1.1` with `X-Real-IP: evil.host\nINFO 10.0.0.1 POST /construction/submit (200) in 1ms`
2. `getClientIpAddress()` returns `"evil.host\nINFO 10.0.0.1 POST /construction/submit (200) in 1ms"` verbatim.
3. logrus writes the embedded newline as-is, producing a forged log line indistinguishable from a real entry.

### Impact Explanation
- **Log injection:** Forged log lines can be inserted with arbitrary IP addresses, methods, paths, and status codes. This directly undermines any log-based alerting or IP reputation system consuming these logs.
- **Log pollution:** Even without newline injection, flooding logs with non-IP strings (hostnames, UUIDs, long strings) degrades the signal-to-noise ratio of access logs, making anomaly detection unreliable.
- **Severity:** Medium-High. The Rosetta API is a public-facing interface; no authentication is required to send requests. The impact is limited to log integrity (not direct code execution), but log integrity is the stated detection mechanism for the threat model described.

### Likelihood Explanation
Any unauthenticated HTTP client can set arbitrary request headers. The exploit requires no special privileges, no credentials, and no knowledge of the system beyond the public endpoint. It is trivially repeatable with a single `curl` command. The only mitigating factor is that docker-compose deployments are protected by nginx; Kubernetes and direct deployments are not.

### Recommendation
Validate the header value as an IP address before using it. Replace the header-reading logic with:

```go
func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        val := r.Header.Get(header)
        // Take only the first comma-separated token (X-Forwarded-For may be a list)
        if idx := strings.IndexByte(val, ','); idx != -1 {
            val = val[:idx]
        }
        val = strings.TrimSpace(val)
        if net.ParseIP(val) != nil {
            return val
        }
    }
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

Additionally, configure Traefik (or any ingress) to strip and re-set `X-Real-IP` from the actual downstream IP, preventing clients from supplying the header at all.

### Proof of Concept
```bash
# Against a Kubernetes or directly-exposed Rosetta instance:
curl -s http://<rosetta-host>:<port>/network/list \
  -H 'Content-Type: application/json' \
  -H $'X-Real-IP: 1.2.3.4\nINFO 10.0.0.1 POST /construction/submit (200) in 1ms' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# Expected result in rosetta access log:
# INFO 1.2.3.4
# INFO 10.0.0.1 POST /construction/submit (200) in 1ms   <-- forged entry
# (second line is injected, not a real request)
```
