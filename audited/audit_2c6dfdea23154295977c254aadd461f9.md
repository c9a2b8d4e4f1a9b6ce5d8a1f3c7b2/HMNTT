### Title
Unauthenticated HTTP Network-Map Fetch Enables Permanent Transfer DoS via Malicious Consensus Node Injection

### Summary
The pinger fetches its consensus-node network map exactly once at startup from an unauthenticated plain-HTTP mirror REST endpoint, with no TLS, no response integrity check, and no subsequent refresh. An attacker who can intercept that single HTTP response (e.g., via ARP spoofing or DNS poisoning within the Kubernetes cluster) injects attacker-controlled node addresses. Every subsequent `TransferTransaction` is routed to those malicious nodes, which return `TRANSACTION_EXPIRED` on every receipt query, exhausting all `cfg.maxRetries + 1` attempts each tick and logging "all attempts failed" indefinitely—blocking all legitimate transfers for the pinger's lifetime.

### Finding Description

**Exact code path:**

1. **`pinger/sdk_client.go` lines 17–22** — for `network=other` (the default in the Helm chart), `buildNetworkFromMirrorNodes()` is called **once** at startup. The returned map is passed directly to `hiero.ClientForNetwork(netmap)` and never refreshed.

2. **`pinger/mirror_node_client.go` line 46** — the HTTP client is a plain `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}` with no TLS configuration. The default Helm value (`charts/hedera-mirror-pinger/values.yaml` line 19) is `http://{{ .Release.Name }}-restjava:80`—cleartext HTTP. The JSON response is decoded and trusted without any signature or integrity verification (lines 95–131).

3. **`pinger/transfer.go` lines 23–57** — `submitWithRetry()` loops `cfg.maxRetries + 1` times (default: 11). Each iteration constructs a fresh `hiero.NewTransferTransaction()` (lines 29–31) and calls `Execute(client)` (line 33), routing to whatever nodes are in the poisoned `client` network map. If `GetReceipt()` returns an error (line 35–41)—including `TRANSACTION_EXPIRED` returned by the malicious node—`lastErr` is set and the loop continues. After all attempts, line 59 returns `fmt.Errorf("all attempts failed: %w", lastErr)`.

4. **`pinger/main.go` lines 63–68** — the error is logged and the ticker continues. The next tick calls `submitWithRetry` again with the same poisoned `client`, producing the same result indefinitely.

**Root cause and failed assumption:** The code assumes the mirror REST API response is trustworthy and that the network map is stable for the process lifetime. Neither is enforced: there is no TLS, no HMAC/signature on the response, and no periodic re-fetch that could recover from a poisoned map.

**Why existing checks are insufficient:**
- The retry loop in `submitWithRetry` retries on *all* errors indiscriminately—including `TRANSACTION_EXPIRED`—so a malicious node that consistently returns this status will exhaust every attempt.
- The context-cancellation check (`ctx.Err()`) only exits on shutdown signal, not on repeated node failures.
- The liveness heartbeat goroutine (`main.go` lines 28–39) writes `/tmp/alive` independently of transfer success, so Kubernetes never restarts the pod based on transfer failures alone.

### Impact Explanation
All `TransferTransaction` submissions fail permanently for the pinger's lifetime. The pinger appears alive (liveness probe passes) but produces zero successful transfers. Any monitoring or alerting that depends on the pinger's transfer success rate will show a complete outage. Because the network map is never refreshed, the only recovery path is a pod restart—which itself re-fetches from the same unauthenticated HTTP endpoint and can be poisoned again.

### Likelihood Explanation
The default Helm deployment uses `HIERO_MIRROR_PINGER_NETWORK: "other"` and `HIERO_MIRROR_PINGER_REST: "http://..."` (cleartext). An attacker needs only network-level access to the cluster at the moment of pinger startup—achievable via a compromised pod in the same namespace performing ARP spoofing, via DNS cache poisoning of the internal service name, or via a cloud-provider network-level interception. No credentials or privileges on the pinger pod itself are required. The attack window is the single HTTP request at startup; once poisoned, the effect is permanent until restart. The attack is repeatable across restarts.

### Recommendation
1. **Enforce TLS for the mirror REST URL**: Require `https://` and validate the server certificate; reject plain HTTP in `loadConfig()`.
2. **Periodically refresh the network map**: Re-fetch and update `client`'s network map on a schedule (e.g., every few minutes) so a poisoned startup map is self-healing.
3. **Distinguish fatal receipt errors in the retry loop**: Do not retry `TRANSACTION_EXPIRED` when the transaction was just constructed (its valid-start is `time.Now()`); treat it as a node fault and skip that node rather than exhausting all attempts.
4. **Add transfer-success liveness signal**: Gate the `/tmp/alive` write on recent transfer success, so Kubernetes restarts the pod after sustained transfer failures, limiting the DoS window.

### Proof of Concept

**Preconditions:**
- Pinger deployed with default Helm values (`network=other`, `mirrorRest=http://<svc>:80`).
- Attacker has a pod in the same Kubernetes namespace (or can poison DNS for the mirror REST service name).

**Steps:**

1. Attacker stands up a malicious HTTP server that responds to `GET /api/v1/network/nodes` with a JSON payload listing attacker-controlled gRPC endpoints as consensus nodes:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}]}
   ```

2. Attacker intercepts the pinger's DNS resolution for the mirror REST service (e.g., via a rogue DNS response or ARP spoofing) so that the startup HTTP request hits the malicious server instead.

3. Pinger starts, calls `buildNetworkFromMirrorNodes()`, receives the poisoned map, and builds `hiero.ClientForNetwork({"<attacker-ip>:50211": AccountID{0,0,3}})`.

4. Attacker's gRPC server accepts `CryptoTransfer` submissions and returns a precheck `OK` followed by a receipt with status `TRANSACTION_EXPIRED` on every `GetReceipt` call.

5. Each tick, `submitWithRetry` runs 11 attempts, all returning `TRANSACTION_EXPIRED`, then logs:
   ```
   all attempts failed: receipt for transaction ... contained error status TRANSACTION_EXPIRED
   ```

6. No legitimate transfer ever succeeds. The pinger's liveness probe continues to pass. The attack persists across all ticks until the pod is manually restarted.
