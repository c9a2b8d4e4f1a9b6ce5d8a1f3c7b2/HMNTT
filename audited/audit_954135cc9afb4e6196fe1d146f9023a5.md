### Title
Unauthenticated DB Connection Pool Exhaustion Causes Rosetta Network-Status DoS

### Summary
The Rosetta API applies no application-level rate limiting or per-client connection semaphore. Any unauthenticated caller can saturate the shared GORM connection pool (default `MaxOpenConnections=100`) with concurrent requests, blocking all subsequent DB-dependent calls including the readiness check's internal `/network/status` probe, which causes Kubernetes to mark the pod not-ready and halt traffic routing.

### Finding Description

**Code path and root cause:**

`rosetta/app/config/types.go` lines 77–81 defines the `Pool` struct with `MaxIdleConnections` (default 20) and `MaxOpenConnections` (default 100). These are applied unconditionally in `rosetta/app/db/db.go` lines 31–33:

```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)
sqlDb.SetConnMaxLifetime(time.Duration(dbConfig.Pool.MaxLifetime) * time.Minute)
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)
```

Every DB-backed endpoint (`/network/status`, `/block`, `/account/balance`, etc.) draws from this single shared pool. `rosetta/app/db/client.go` lines 36–37 wraps each query in a `context.WithTimeout` of `StatementTimeout` seconds (default 20s), meaning each connection can be held for up to 20 seconds per query.

`rosetta/app/services/network_service.go` lines 67–77 shows `/network/status` makes **three sequential DB calls** per request: `RetrieveGenesis`, `RetrieveLatest`, and `Entries`. Each call acquires a connection from the pool.

The critical cascade is in `rosetta/app/middleware/health.go` lines 80–100: the readiness check's `checkNetworkStatus` function issues an HTTP call to `localhost:{port}/network/status`, which itself needs DB connections from the same exhausted pool. If the pool is full, this HTTP call blocks/times out, the readiness check returns an error, and Kubernetes stops routing traffic to the pod.

**Why existing checks fail:**

The only rate-limiting controls are in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 (Traefik `inFlightReq: amount: 5` per IP, `rateLimit: average: 10` per host). However, `global.middleware: false` is the default, meaning this Traefik middleware is **not applied** in a default deployment. The Rosetta Go application itself contains zero rate-limiting, connection-semaphore, or per-client throttling code.

**Exploit flow:**

1. Attacker identifies the Rosetta endpoint (port 5700, publicly routable in default Helm config).
2. Attacker opens 100+ concurrent HTTP connections to any DB-backed endpoint (e.g., `POST /network/status` or `POST /block`).
3. Each request acquires a GORM pool connection for up to 20 seconds (StatementTimeout).
4. Go's `database/sql` blocks all new pool requests once `MaxOpenConnections=100` is reached.
5. The Kubernetes readiness probe calls `GET /health/readiness`, which internally calls `POST /network/status` (via `checkNetworkStatus`), which blocks waiting for a pool connection.
6. The readiness check times out (10-second timeout, line 40 of `health.go`), returns failure.
7. Kubernetes marks the pod not-ready; the load balancer stops sending traffic. Effective service outage.

### Impact Explanation

Complete denial of service for all legitimate Rosetta API consumers. The readiness-check cascade is particularly severe: pool exhaustion causes the pod to be evicted from the load balancer rotation, not just slow responses. If HPA scales up new pods, each new pod's readiness check also calls `/network/status` and can be similarly starved if the attacker maintains pressure. The Rosetta `/network/status` endpoint is the canonical liveness signal for Coinbase Rosetta integrations; its unavailability breaks all downstream block-explorer and exchange integrations.

### Likelihood Explanation

No authentication is required. The Rosetta API is publicly exposed (port 5700, ingress paths in `charts/hedera-mirror-rosetta/values.yaml` lines 126–133). Sending 100 concurrent HTTP POST requests is trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The attack is repeatable and stateless — the attacker does not need to maintain sessions or credentials. A single attacker machine with a modest network connection can sustain the required concurrency indefinitely.

### Recommendation

1. **Application-level semaphore**: Add a middleware in `rosetta/main.go` (before the metrics/tracing chain) that limits total in-flight DB-bound requests using a buffered channel or `golang.org/x/sync/semaphore`, sized to `MaxOpenConnections - N` (reserving N slots for health checks).
2. **Separate health-check DB connection**: In `rosetta/app/middleware/health.go`, the `checkNetworkStatus` readiness check should use a dedicated `database/sql` connection (outside the GORM pool) so pool exhaustion cannot affect the readiness probe.
3. **Enforce Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` and `rateLimit` controls are active in all deployments.
4. **Reduce StatementTimeout**: The default 20-second statement timeout (`rosetta/app/config/types.go` via `Db.StatementTimeout`) allows connections to be held for a long time; reducing it to 5–8 seconds shrinks the attack window.

### Proof of Concept

```bash
# Exhaust the pool with 110 concurrent /network/status requests
# (each held for up to 20s by StatementTimeout)
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

for i in $(seq 1 110); do
  curl -s -X POST http://<rosetta-host>:5700/network/status \
    -H 'Content-Type: application/json' \
    -d "$BODY" &
done

# While the above runs, observe readiness probe failure:
watch -n1 'curl -s -o /dev/null -w "%{http_code}" http://<rosetta-host>:5700/health/readiness'
# Expected: transitions from 200 to 503 within seconds

# Kubernetes will mark the pod NotReady:
kubectl get pods -l app=rosetta -w
# Expected: READY column changes from 1/1 to 0/1
```
