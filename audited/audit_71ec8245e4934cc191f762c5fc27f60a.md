### Title
Uncached EVM Address Resolution in `getAirdrops()` Causes Unconditional Double DB Round-Trip Per Request

### Summary
`TokenAirdropServiceImpl.getAirdrops()` always calls `entityService.lookup()` before querying airdrops. When the caller supplies an EVM address as the account identifier, `EntityServiceImpl.lookup()` issues a synchronous, uncached `SELECT` against the `entity` table to resolve the address to an internal ID, followed by the airdrop repository query. No caching exists on this path in the rest-java module. Any unprivileged caller can therefore double the per-request DB load simply by using the EVM address format instead of a numeric ID, without increasing request volume.

### Finding Description
**Code path:**

`TokenAirdropServiceImpl.getAirdrops()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/TokenAirdropServiceImpl.java`, lines 19–22:
```java
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    var id = entityService.lookup(request.getAccountId());   // DB hit #1 when EVM address
    return repository.findAll(request, id);                  // DB hit #2 always
}
```

`EntityServiceImpl.lookup()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/EntityServiceImpl.java`, lines 30–38:
```java
public EntityId lookup(EntityIdParameter accountId) {
    var id = switch (accountId) {
        case EntityIdNumParameter p    -> Optional.of(p.id());          // no DB query
        case EntityIdAliasParameter p  -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
        case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of); // DB query
    };
    return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
}
```

`EntityRepository.findByEvmAddress()` — `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, line 16–17:
```java
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
```

**Root cause:** The `EntityIdNumParameter` branch short-circuits with no DB access. The `EntityIdEvmAddressParameter` branch always issues a live SQL query. There is no `@Cacheable` annotation on `findByEvmAddress` in the rest-java `EntityRepository` (unlike the web3 module's `EntityRepository` which does cache). The `EntityServiceImpl` itself has no caching layer. Every EVM-address-format request therefore unconditionally pays two sequential DB round-trips.

**Failed assumption:** The design assumes that EVM address resolution is an infrequent or low-cost path. In practice, the endpoint publicly accepts EVM addresses (`EntityIdParameter.valueOf()` parses them via `EntityIdEvmAddressParameter.valueOfNullable()`), and the controller binds the path variable directly to `EntityIdParameter`, so any caller can choose this format freely.

### Impact Explanation
For every request using an EVM address, the DB receives two queries instead of one. An attacker who sends the same number of requests as a normal user but uses EVM addresses instead of numeric IDs doubles their DB footprint — a 100 % increase per request, well above the 30 % threshold — without any increase in request rate. At scale (many concurrent users or a single high-frequency client switching formats), this can saturate DB connection pools, increase query latency for all users, and degrade mirror-node availability. The `entity` table query (`evm_address` lookup) is not trivially cheap: it filters on a non-primary-key column with a `deleted <> true` predicate.

### Likelihood Explanation
The exploit requires zero privileges. Valid EVM addresses are publicly observable on-chain and via the mirror node's own APIs. The attacker needs only to substitute a hex EVM address for a numeric account ID in the URL path (e.g., `/api/v1/accounts/0x<40hexchars>/airdrops/outstanding`). No authentication, no special knowledge, and no elevated request rate are required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add caching** to `EntityServiceImpl.lookup()` for the `EntityIdEvmAddressParameter` (and `EntityIdAliasParameter`) branches, keyed on the raw bytes, with a short TTL (e.g., 30–60 s). Mirror the pattern already used in the web3 `EntityRepository` (`@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`).
2. **Alternatively**, add a `@Cacheable` annotation directly to `EntityRepository.findByEvmAddress()` in the rest-java module, consistent with the web3 module.
3. **Add per-IP or per-client rate limiting** at the API gateway or Spring filter level for the airdrop endpoints.

### Proof of Concept
**Precondition:** A valid EVM address `0xAABBCCDDEEFF...` exists in the mirror node's `entity` table (obtainable from any public block explorer or the mirror node's own `/api/v1/accounts` endpoint).

**Step 1 – Baseline (numeric ID, 1 DB query):**
```
GET /api/v1/accounts/0.0.12345/airdrops/outstanding
```
Internally: `EntityIdNumParameter` → no entity lookup → 1 DB query (airdrop table).

**Step 2 – Attack (EVM address, 2 DB queries):**
```
GET /api/v1/accounts/0xAABBCCDDEEFF000000000000000000000000AABB/airdrops/outstanding
```
Internally: `EntityIdEvmAddressParameter` → `entityRepository.findByEvmAddress()` (DB query #1 against `entity` table) → `repository.findAll()` (DB query #2 against `token_airdrop` table).

**Step 3 – Amplification without rate increase:**
Send the same N requests/second as a normal user, but all using EVM address format. DB query rate doubles from N to 2N with no change in HTTP request volume. At N = 100 req/s, the DB receives 200 queries/s instead of 100 — a sustained 100 % increase in DB load attributable solely to format choice, requiring no brute-force volume increase.
