### Title
Unauthenticated `/health/readiness` Endpoint Exhausts Shared DB Connection Pool, Starving API Queries

### Summary
The `readinessCheck()` function in `rest/health.js` issues a live `pool.query()` against the shared `pg` connection pool on every invocation. The endpoint is registered via `@godaddy/terminus` outside the Express middleware chain, so no rate-limiting or authentication applies to it. An unprivileged attacker who floods this endpoint can saturate the pool's 10 default connections, causing all legitimate API queries to queue or time out.

### Finding Description

**Code path:**

`rest/server.js` lines 159–166 register the health check via `createTerminus`, which intercepts requests at the raw Node.js `http.Server` level — before any Express middleware (auth, rate-limit, metrics) runs:

```js
createTerminus(server, {
  healthChecks: {
    '/health/readiness': health.readinessCheck,   // no middleware applied
    '/health/liveness': health.livenessCheck,
  },
  ...
});
``` [1](#0-0) 

`rest/health.js` lines 22–33: every call to `readinessCheck()` unconditionally acquires a connection from the global pool and executes a query. There is no in-flight deduplication, semaphore, or concurrency cap:

```js
const readinessCheck = async () => {
  return pool
    .query(readinessQuery)   // consumes one pool connection per concurrent call
    ...
};
``` [2](#0-1) 

`rest/dbpool.js` line 14 sets the pool ceiling to `config.db.pool.maxConnections`: [3](#0-2) 

The documented default for `hiero.mirror.rest.db.pool.maxConnections` is **10**: [4](#0-3) 

The same `global.pool` is used by every API handler (transactions, accounts, tokens, etc.). The `pg` pool queue is unbounded by default; when all 10 connections are occupied, new `pool.query()` calls from API handlers queue behind the health-check queries and wait up to `connectionTimeoutMillis` (default 20 000 ms) before failing.

**Root cause / failed assumption:** The design assumes `/health/readiness` is called only by Kubernetes probes at a low, controlled rate. It does not account for an external attacker sending concurrent requests, and it provides no mechanism to limit how many simultaneous DB connections the endpoint may consume.

### Impact Explanation

With the pool saturated, every in-flight API request that needs a DB connection stalls. Transactions, account lookups, and all other endpoints return errors or time out. Because the health endpoint itself starts failing (pool exhausted → `DbError`), Kubernetes marks the pod as not-ready and stops routing traffic to it, compounding the outage. This maps directly to the stated critical impact: network inability to confirm new transactions via the mirror REST API.

### Likelihood Explanation

The endpoint requires no credentials and is on the same port as the public API (default 5551, or port 80 behind the Kubernetes ingress). A single attacker machine can sustain thousands of HTTP/1.1 keep-alive requests per second with tools like `wrk` or `hey`. Because the readiness query (`select true from address_book limit 1`) completes in ~1 ms, the attacker must maintain roughly 10 000 req/s to keep all 10 connections busy — achievable from a single host on a fast network, and trivially achievable from a small botnet. The attack is stateless, requires no prior knowledge, and is fully repeatable.

### Recommendation

1. **Deduplicate in-flight health checks**: use a module-level promise cache so that N concurrent calls to `readinessCheck()` result in exactly one `pool.query()` until the current one resolves.
2. **Rate-limit the endpoint at the terminus/HTTP layer**: add a simple token-bucket or fixed-window counter before invoking the health check handler.
3. **Use a dedicated, single-connection health-check client** separate from the shared API pool, so health probes cannot compete with API queries for connections.
4. **Restrict network access**: if the Kubernetes deployment exposes the health port separately, place it on an internal-only port/service not reachable from the public internet.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wg/wrk)
# Target: publicly reachable mirror-node REST API host

wrk -t 10 -c 200 -d 60s http://<mirror-node-host>:<port>/health/readiness &

# In parallel, observe API degradation:
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code} %{time_total}\n" \
    "http://<mirror-node-host>:<port>/api/v1/transactions?limit=1" &
done
wait
```

Expected result: while `wrk` runs, the `/api/v1/transactions` requests return HTTP 500 (`DbError: timeout exceeded when trying to connect`) or hang for up to 20 seconds, demonstrating pool starvation caused by the unauthenticated health endpoint flood.
