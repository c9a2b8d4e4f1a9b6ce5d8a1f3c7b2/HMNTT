### Title
Unbounded Operations Array in `/construction/preprocess` Enables Resource Exhaustion Without Rate-Limit Bypass

### Summary
`ConstructionPreprocess` accepts an arbitrarily large `Operations` array with no application-level count or byte limit. `getOperationSlice` allocates and iterates every element, and `cryptoTransferTransactionConstructor.preprocess` performs additional unbounded iteration with map allocation and sum computation. A single well-crafted request with thousands of balanced `CRYPTOTRANSFER` operations can spike CPU and memory significantly above the 24-hour baseline without triggering per-request rate limits.

### Finding Description
**Code path:**

`ConstructionPreprocess` (construction_service.go:285–321) calls `getOperationSlice(request.Operations)` with no guard on `len(request.Operations)`:

```go
// construction_service.go:289
operations, rErr := c.getOperationSlice(request.Operations)
```

`getOperationSlice` (construction_service.go:432–464) allocates a slice sized to the full input and iterates every element, parsing account IDs and amounts:

```go
operationSlice := make(types.OperationSlice, 0, len(operations))
for _, operation := range operations { ... }
```

After that, `compositeTransactionConstructor.validate` (composite_transaction_constructor.go:83–104) iterates all operations again to check type homogeneity. Then `cryptoTransferTransactionConstructor.preprocess` (crypto_transfer_transaction_constructor.go:120–161) is called with `validateOperations(operations, 0, ...)` — the `size=0` argument explicitly disables the count check — and then iterates all operations a third time, building a `senderMap`, a `totalAmounts` map, and a `transfers` slice.

**Root cause:** No upper bound on `len(request.Operations)` exists anywhere in the application. The `validateOperations` helper is called with `size=0` for `CRYPTOTRANSFER`, meaning any number of operations is accepted. No `http.MaxBytesReader` is applied to the request body.

**Exploit flow:**
1. Attacker crafts a `CRYPTOTRANSFER` preprocess request with N operations (e.g., N/2 senders of −X tinybars and N/2 receivers of +X tinybars so the sum check passes).
2. Server allocates O(N) memory in `getOperationSlice`, then iterates O(N) three more times in `validate` and `preprocess`.
3. A single request with N=50,000 operations causes a large heap allocation and sustained CPU loop before returning an error or success.

**Why existing checks fail:**
- The Traefik `rateLimit: average: 10` (values.yaml:157–160) is per-host, not per-IP, and only applies when Traefik ingress is deployed. `global.middleware: false` (values.yaml:95) means the middleware CRD may not be active in all deployments.
- The `inFlightReq: amount: 5` (values.yaml:152–156) limits concurrent requests per IP to 5, but each of those 5 can carry an arbitrarily large payload.
- The Traefik `retry: attempts: 3` (values.yaml:161–163) re-sends each request up to 3 times, tripling the server-side work per client request.
- HTTP `ReadTimeout`/`WriteTimeout` are time-based, not size-based, and do not bound the number of operations parsed before the timeout fires.
- The Rosetta SDK asserter validates operation structure but imposes no maximum count.

### Impact Explanation
Each oversized request forces multiple O(N) passes over the operations array plus heap allocations proportional to N. With 5 concurrent connections allowed per IP and no application-level cap, an attacker can sustain elevated CPU and memory consumption on the Rosetta node. Because `CRYPTOTRANSFER` is the most permissive operation type (no fixed size), the attacker can maximize N while trivially satisfying the zero-sum constraint. This can push resource utilization well above the 30% threshold relative to the 24-hour baseline without sending an unusually high request rate.

### Likelihood Explanation
The endpoint is publicly reachable (no authentication required by the Rosetta spec or this implementation). The exploit requires only a valid JSON body with a large `operations` array — no credentials, no special tokens. The zero-sum constraint for `CRYPTOTRANSFER` is trivially satisfied. The attack is repeatable and scriptable. Deployments that skip or misconfigure the Traefik ingress middleware (as suggested by `global.middleware: false`) have no compensating control at all.

### Recommendation
1. **Application-level cap:** Add a configurable `MaxOperations` limit checked at the top of `getOperationSlice` (or at the start of `ConstructionPreprocess`) and return `ErrInvalidOperations` if exceeded. A reasonable default is 1,000 (matching Hedera's own transaction limits).
2. **Request body size limit:** Wrap the HTTP request body with `http.MaxBytesReader` in the server setup or in a middleware to reject oversized payloads before JSON decoding.
3. **Per-IP rate limiting:** Change the Traefik `rateLimit` `sourceCriterion` from `requestHost` to `ipStrategy` so limits apply per client IP, not per virtual host.
4. **Ensure middleware is always enabled:** Remove or invert the `global.middleware: false` default so the Traefik middleware chain is active in all deployments.

func (c *constructionAPIService) ConstructionPreprocess(
	ctx context.Context,
	request *rTypes.ConstructionPreprocessRequest,

	operations, rErr := c.getOperationSlice(request.Operations)
	if rErr != nil {
		return nil, rErr
	}

	signers, err := c.transactionHandler.Preprocess(ctx, operations)
	if err != nil {
		return nil, err
	}

	requiredPublicKeys := make([]*rTypes.AccountIdentifier, 0, len(signers))
	for _, signer := range signers {
		requiredPublicKeys = append(requiredPublicKeys, &rTypes.AccountIdentifier{Address: signer.String()})
	}

	options := make(map[string]any)
	if len(request.Metadata) != 0 {
		options = request.Metadata
	}

	options[optionKeyOperationType] = operations[0].Type

	// the first signer is always the payer account
	payer := signers[0]
	if payer.HasAlias() {
		options[optionKeyAccountAliases] = payer.String()
	}

	return &rTypes.ConstructionPreprocessResponse{
		Options:            options,
		RequiredPublicKeys: requiredPublicKeys,
	}, nil
}

func (c *constructionAPIService) getOperationSlice(operations []*rTypes.Operation) (
	types.OperationSlice,
	*rTypes.Error,
) {
	operationSlice := make(types.OperationSlice, 0, len(operations))
	for _, operation := range operations {
		var accountId types.AccountId
		if operation.Account != nil {
			var err error
			accountId, err = types.NewAccountIdFromString(operation.Account.Address, c.systemShard, c.systemRealm)
			if err != nil || accountId.IsZero() {
				return nil, errors.ErrInvalidAccount
			}
		}

		var amount types.Amount
		var rErr *rTypes.Error
		if operation.Amount != nil {
			if amount, rErr = types.NewAmount(operation.Amount); rErr != nil {
				return nil, rErr
			}
		}

		operationSlice = append(operationSlice, types.Operation{
			AccountId: accountId,
			Amount:    amount,
			Index:     operation.OperationIdentifier.Index,
			Metadata:  operation.Metadata,
			Type:      operation.Type,
		})
	}

	return operationSlice, nil

func (c *cryptoTransferTransactionConstructor) preprocess(operations types.OperationSlice) (
	[]transfer,
	[]types.AccountId,
	*rTypes.Error,
) {
	if err := validateOperations(operations, 0, c.GetOperationType(), false); err != nil {
		return nil, nil, err
	}

	senderMap := senderMap{}
	totalAmounts := make(map[string]int64)
	transfers := make([]transfer, 0, len(operations))

	for _, operation := range operations {
		accountId := operation.AccountId
		amount := operation.Amount
		if amount.GetValue() == 0 {
			return nil, nil, errors.ErrInvalidOperationsAmount
		}

		sdkAccountId, err := accountId.ToSdkAccountId()
		if err != nil {
			return nil, nil, errors.ErrInvalidAccount
		}
		transfers = append(transfers, transfer{account: sdkAccountId, amount: amount})

		if amount.GetValue() < 0 {
			senderMap[accountId.String()] = accountId
		}

		totalAmounts[amount.GetSymbol()] += amount.GetValue()
	}

	for symbol, totalAmount := range totalAmounts {
		if totalAmount != 0 {
			log.Errorf("Transfer sum for symbol %s is not 0", symbol)
			return nil, nil, errors.ErrInvalidOperationsTotalAmount
		}
	}

	return transfers, senderMap.toSenders(), nil
}

func (c *compositeTransactionConstructor) validate(operations types.OperationSlice) (
	transactionConstructorWithType,
	*rTypes.Error,
) {
	if len(operations) == 0 {
		return nil, errors.ErrEmptyOperations
	}

	operationType := operations[0].Type
	for _, operation := range operations[1:] {
		if operation.Type != operationType {
			return nil, errors.ErrMultipleOperationTypesPresent
		}
	}

	h, ok := c.constructorsByOperationType[operationType]
	if !ok {
		log.Errorf("Operation type %s is not supported", operationType)
		return nil, errors.ErrOperationTypeUnsupported
	}

	return h, nil
