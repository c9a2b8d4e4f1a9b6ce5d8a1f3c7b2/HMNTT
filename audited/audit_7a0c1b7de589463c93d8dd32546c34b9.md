### Title
Unbounded Concurrent `subscribeTopic` gRPC Calls Exhaust Shared HikariCP Connection Pool, Denying Read-Only Transaction Access

### Summary
The gRPC module enforces a per-connection concurrency limit of 5 (`maxConcurrentCallsPerConnection`) but imposes no global limit on the number of connections or total concurrent `subscribeTopic` subscriptions. Each active subscription periodically acquires a blocking JDBC connection from the shared HikariCP pool via `@Transactional(readOnly = true)` on `TopicMessageRepositoryCustomImpl.findByFilter`. An unprivileged attacker opening many connections can exhaust the pool, starving the `transactionOperationsReadOnly` bean (and all other DB operations) of connections.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic` (line 43–53) → `TopicMessageServiceImpl.subscribeTopic` (line 59–92) → `PollingTopicMessageRetriever.retrieve` (line 45–63) → `poll()` (line 65–79) → `topicMessageRepository.findByFilter()` → `TopicMessageRepositoryCustomImpl.findByFilter` (line 33–61, annotated `@Transactional(readOnly = true)`).

**Root cause — three compounding weaknesses:**

1. **Per-connection-only concurrency cap.** `NettyProperties.maxConcurrentCallsPerConnection = 5` (default) is applied per TCP connection in `GrpcConfiguration.grpcServerConfigurer` (line 33). There is no global cap on the number of connections or total concurrent subscriptions. An attacker opens `C` connections, each carrying 5 calls → `5C` concurrent subscriptions.

2. **Blocking JDBC inside a reactive pipeline.** `TopicMessageRepositoryCustomImpl.findByFilter` uses JPA `EntityManager` (blocking JDBC), not R2DBC. `PollingTopicMessageRetriever` offloads to `Schedulers.boundedElastic()` (line 41), but each poll still acquires a HikariCP connection for the duration of the `@Transactional(readOnly = true)` call. With `pollingFrequency = 2s` (default), every active subscriber holds a connection for the query duration every 2 seconds.

3. **No subscriber count enforcement.** `TopicMessageServiceImpl.subscriberCount` (line 48) is an `AtomicLong` used only as a Prometheus gauge metric (lines 52–55). There is no check that rejects or throttles new subscriptions when the count exceeds a threshold.

**Shared pool impact on `transactionOperationsReadOnly`:** The `transactionOperationsReadOnly` bean (line 21–25 of `GrpcConfiguration.java`) wraps the same `PlatformTransactionManager` backed by the same HikariCP `DataSource` (configured in `CommonConfiguration.dataSource`, line 69–95). When the pool is exhausted, any call through this bean (e.g., `NetworkServiceImpl`) blocks for up to HikariCP's `connectionTimeout` (default 30 s) then throws `SQLTransientConnectionException`, effectively denying all read-only transaction access.

### Impact Explanation

A successful attack exhausts the HikariCP connection pool shared by all gRPC DB operations. The `transactionOperationsReadOnly` bean and `@Transactional(readOnly = true)` repository calls cannot acquire connections. Legitimate clients querying submitted transaction data via `subscribeTopic` or `NetworkService` receive errors or indefinite hangs. The mirror node's ability to serve read-only transaction data is completely denied for the duration of the attack. Severity: **High** (availability impact on a public-facing, unauthenticated API endpoint).

### Likelihood Explanation

The `subscribeTopic` endpoint is publicly accessible with no authentication, no IP-based rate limiting, and no global subscriber cap. An attacker needs only a standard gRPC client (e.g., `grpcurl`) and the ability to open multiple TCP connections. The default HikariCP pool size is 10 connections; exhausting it requires only a handful of connections × 5 calls each if queries are slow (e.g., large topic history). The attack is trivially repeatable and requires no special privileges or knowledge of the system internals.

### Recommendation

1. **Enforce a global subscriber limit** in `TopicMessageServiceImpl.subscribeTopic`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` status if exceeded.
2. **Add a global `maxConcurrentCallsPerServer` limit** in `GrpcConfiguration.grpcServerConfigurer` via `NettyServerBuilder.maxConcurrentCallsPerServer(int)`, independent of per-connection limits.
3. **Separate the HikariCP pool** used by `subscribeTopic` polling from the pool used by `transactionOperationsReadOnly` / `NetworkServiceImpl`, so subscriber load cannot starve critical read-only transaction paths.
4. **Add per-IP or per-connection rate limiting** at the gRPC interceptor layer for `subscribeTopic`, analogous to the `ThrottleConfiguration` used in the web3 module.

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }

    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
