### Title
Unbounded Cache-Eviction DoS via Unique Query Flooding on Unauthenticated GraphQL Endpoint

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed on the raw query string with a hard cap of 1000 entries and no rate limiting on the GraphQL endpoint. An unauthenticated attacker can continuously submit distinct valid query strings to saturate the cache, forcing perpetual LRU eviction and repeated parse-and-validate cycles, causing sustained CPU exhaustion on the server.

### Finding Description
**Code path:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java:15` — cache spec: `expireAfterWrite=1h,maximumSize=1000,recordStats`
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java:26` — `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput))` — cache key is the **raw query string**, no normalization
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java:42-48` — `graphQlCustomizer()` wires the `PreparsedDocumentProvider` into the GraphQL engine

**Root cause:** The cache key is the verbatim query string. An attacker trivially generates unique cache keys by varying field aliases, argument values, or appending unique inline comments (e.g., `# 1`, `# 2`, …). Once the cache reaches its 1000-entry maximum, every new unique query evicts an existing entry and triggers a full `parseAndValidateFunction.apply()` call — lexing, parsing, and schema validation — on every request.

**Why existing checks fail:**
- `maxCharacters(10000)`, `maxTokens(1000)`, `maxRuleDepth(100)` (lines 33–38) bound the CPU work *per parse cycle* but impose no limit on the *rate* of parse cycles.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (lines 43–45) execute *after* parsing completes, so they do not prevent the parse overhead.
- The `LoggingFilter` (the only servlet filter in the `graphql` module) only logs; it performs no throttling.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j rate limiting exists exclusively in the `web3` module and is **not present** in the `graphql` module.
- No authentication is required to reach the `/graphql` endpoint.

### Impact Explanation
A single attacker with a modest HTTP client can maintain a continuous stream of unique-query POST requests. Each request that misses the cache forces a full parse-and-validate cycle. With no server-side rate limit, the attacker can drive CPU utilization to 100%, degrading or denying service to all legitimate users of the GraphQL API. Because the GraphQL module is a read-only mirror API, this directly impacts availability of Hiero network data queries.

### Likelihood Explanation
No privileges, credentials, or special network position are required. Generating 1001+ syntactically valid but textually distinct GraphQL queries is trivial (e.g., varying a numeric literal or appending `# N` comments). The attack is fully automatable with a simple loop in any HTTP client, is repeatable indefinitely, and requires no prior knowledge of the system beyond the public endpoint URL.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module.** Introduce a bucket4j or similar token-bucket filter (mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) that limits requests per IP per second before the GraphQL execution pipeline is entered.
2. **Normalize query strings before caching.** Strip comments and canonicalize whitespace in the cache key so that trivially mutated queries map to the same cache entry, reducing the attacker's ability to generate unique keys.
3. **Add a connection/request concurrency limit** at the servlet or Spring WebMVC layer to cap simultaneous in-flight GraphQL parse operations.
4. **Document and enforce the load-balancer rate-limiting requirement** noted in `docs/design/contract-log-subscription.md` as a hard deployment prerequisite, not an optional suggestion.

### Proof of Concept
```bash
# Fill the cache with 1000 unique queries
for i in $(seq 1 1000); do
  curl -s -X POST https://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{entityId:{shard:0,realm:0,num:$i}}){balance} }\"}" &
done
wait

# Now flood with 1001+ additional unique queries to force continuous eviction+reparse
for i in $(seq 1001 9999); do
  curl -s -X POST https://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{entityId:{shard:0,realm:0,num:$i}}){balance} }\"}" &
done
wait
# Observe CPU spike on the GraphQL service pod; each request beyond cache capacity
# triggers a full parse-and-validate cycle with no server-side throttle to stop it.
```
