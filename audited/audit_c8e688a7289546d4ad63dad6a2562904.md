### Title
Integer Overflow in `poll()` Bypasses Query Limit, Enabling Unbounded Database Reads

### Summary
In `PollingTopicMessageRetriever.poll()`, a user-supplied `limit=Long.MAX_VALUE` passes the `@Min(0)` validation constraint, then the cast `(int)(Long.MAX_VALUE - 0L)` overflows to `-1`. This `-1` is passed as the `limit` to an internally constructed `TopicMessageFilter`, where `hasLimit()` returns `false` (since `-1 > 0` is false), causing `findByFilter()` to execute with no row limit and potentially return the entire `topic_message` table.

### Finding Description

**Exact code path:** `PollingTopicMessageRetriever.java`, `poll()`, lines 68â€“75:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())  // line 69 â€” overflow here
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());       // line 71 â€” Math.min(-1, N) = -1
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build(); // line 75 â€” limit=-1
```

**Root cause:** The narrowing cast `(int)(long)` at line 69 is applied without overflow checking. `Long.MAX_VALUE` is `0x7FFFFFFFFFFFFFFF`; its lower 32 bits are `0xFFFFFFFF = -1` as a signed `int`.

**Why existing checks fail:**

1. `TopicMessageFilter.limit` is annotated `@Min(0)` â€” this only rejects negative values. `Long.MAX_VALUE` is a valid, positive `long` and passes validation.
2. `hasLimit()` at line 39 of `TopicMessageFilter.java` returns `limit > 0`. For the *external* filter with `limit=Long.MAX_VALUE`, this is `true`, so the overflow branch is taken. For the *internal* filter built with `limit=-1`, `hasLimit()` returns `false`, meaning `findByFilter()` sees no limit at all.
3. The `@Validated` annotation on `TopicMessageFilter` is enforced by Spring only on externally injected beans/method parameters. The internal `filter.toBuilder().limit(-1).build()` call is a plain Lombok builder invocation â€” Spring validation is **never triggered**, so `-1` is accepted silently.
4. `Math.min(-1, maxPageSize)` always returns `-1` regardless of the configured `maxPageSize`, completely defeating the page-size protection.

**Exploit flow:**
- External filter: `limit = Long.MAX_VALUE`, `total = 0`
- `(int)(Long.MAX_VALUE - 0L)` â†’ `-1`
- `Math.min(-1, maxPageSize)` â†’ `-1`
- Internal filter built: `limit = -1`
- `hasLimit()` on internal filter â†’ `false`
- `findByFilter()` executes with no `LIMIT` clause â†’ full table scan returned

### Impact Explanation
An unprivileged gRPC subscriber can force the mirror node to stream the entire `topic_message` table per poll cycle. This causes unbounded memory allocation in the Flux pipeline, saturates database I/O, and can exhaust heap on the mirror node â€” effectively a single-request denial of service. Because the polling loop repeats (`numRepeats = Long.MAX_VALUE` in throttled mode, line 99), the attack is sustained automatically.

### Likelihood Explanation
Any client that can open a gRPC `subscribeTopic` stream can trigger this. No authentication, elevated privilege, or special knowledge is required beyond knowing the topic ID. The `Long.MAX_VALUE` value is trivially constructable in any gRPC client. The attack is deterministic and repeatable on every poll interval.

### Recommendation
1. **Fix the cast:** Replace the unchecked narrowing cast with a safe clamp:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Cap the external limit:** Add `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate maximum (e.g., `@Max(10_000)`) to `TopicMessageFilter.limit` to prevent absurdly large values from reaching this code path.
3. **Guard the internal filter:** Before calling `findByFilter`, assert `pageSize > 0` and short-circuit to `Flux.empty()` if not.
