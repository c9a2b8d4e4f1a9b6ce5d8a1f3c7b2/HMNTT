### Title
Unbounded `data` Field Serialization in LOG Action Causes Heap Exhaustion via Repeated Large Requests

### Summary
The `throttle()` method in `ThrottleManagerImpl` passes requests through rate and gas bucket checks that are blind to payload size, then calls `log.info("{}", request)` for any request matching a LOG-action filter. Because `ContractCallRequest.data` is bounded only by the `@Hex` default `maxLength` of 1,048,576 hex characters (~512 KB of binary / ~1 MB as a Java String), an unprivileged attacker can send requests at the full allowed rate (default 500 req/s) with near-maximum `data` payloads, forcing the JVM to allocate and GC up to ~500 MB/s of log-message strings, driving heap pressure and GC overhead well above 30%.

### Finding Description

**Code path:**

1. `ThrottleManagerImpl.throttle()` — `ThrottleManagerImpl.java` lines 37–48: after `rateLimitBucket.tryConsume(1)` and `gasLimitBucket.tryConsume(scaleGas(request.getGas()))` both succeed, every configured `RequestProperties` whose `test()` returns `true` triggers `action()`.
2. `ThrottleManagerImpl.action()` — lines 66–76: the `LOG` branch executes `log.info("{}", request)`, which forces SLF4J to call `request.toString()` (Lombok `@Data`-generated, includes all fields) and build the full log string.
3. `ContractCallRequest.data` — `ContractCallRequest.java` line 29: annotated `@Hex` with **no explicit `maxLength`**, so the `HexValidator` enforces the annotation's own default of `1_048_576L` hex characters (`Hex.java` line 27). This allows a `data` string of up to 1 MB.
4. `RequestProperties.action` default — `RequestProperties.java` line 29: `private ActionType action = ActionType.LOG;`. Any operator-added filter that omits an explicit action silently defaults to LOG.

**Root cause:** Neither the rate-limit bucket (counts requests) nor the gas-limit bucket (counts gas units) accounts for the byte length of `data`. The LOG action unconditionally serializes the entire request object, including the arbitrarily large (up to 1 MB) `data` string, on every matching request.

**Failed assumption:** The code assumes that requests passing the two bucket checks are "cheap" to log. This is false when `data` is near its maximum allowed size.

### Impact Explanation

At the default rate limit of 500 req/s (`ThrottleProperties.java` line 35) with `data` at the 1 MB ceiling:

- **String allocation rate:** 500 req/s × ~1 MB/req = ~500 MB/s of short-lived `String` objects created solely for log formatting.
- **GC pressure:** The JVM's young-generation heap fills and is collected at a rate proportional to this allocation, causing frequent minor GCs and potential promotion to old-gen, triggering major/full GCs.
- **Heap memory:** Concurrent in-flight log strings and their backing `char[]`/`byte[]` arrays inflate live heap well beyond normal operating levels.
- **Threshold:** 500 MB/s of extra allocation on a typical mirror-node heap (e.g., 4–8 GB) easily exceeds a 30% increase in GC CPU time and heap occupancy, degrading throughput for all other users.

### Likelihood Explanation

- **No authentication required:** Any external caller can POST to the contract-call endpoint.
- **Precondition — LOG filter configured:** The default `request` list is empty (`ThrottleProperties.java` line 32), so the attack only activates when an operator has added at least one `RequestProperties` entry. However, the default `action` is `LOG` (`RequestProperties.java` line 29), making it easy for operators to inadvertently enable this path.
- **Staying within rate limits:** The attacker need only send ≤500 req/s with large `data` values — well within the capability of a single machine — to sustain the attack indefinitely.
- **Repeatability:** The attack is fully deterministic and repeatable with a simple HTTP client loop.

### Recommendation

1. **Truncate before logging:** Replace `log.info("{}", request)` with a version that caps `data` in the log output, e.g.:
   ```java
   case LOG -> log.info("{}", request.toSafeString(256)); // truncate data to 256 chars
   ```
2. **Add an explicit `maxLength` on `data`:** Set a tighter bound in `ContractCallRequest.java`:
   ```java
   @Hex(maxLength = 131072) // e.g., 64 KB
   private String data;
   ```
3. **Size-aware throttling:** Add a per-request payload-size bucket so that large-`data` requests consume proportionally more tokens.
4. **Async/sampled logging:** For the LOG action, use sampling (already partially supported via `rate` in `RequestProperties`) and ensure it is always set to a low value when `data` can be large.

### Proof of Concept

**Preconditions:**
- Mirror node is running with at least one `RequestProperties` entry configured (action defaults to `LOG`, filters empty → matches all requests).
- INFO logging is enabled (typical default).

**Steps:**

```bash
# Generate a ~512 KB hex payload (1,048,576 hex chars)
DATA=$(python3 -c "print('0x' + 'ab' * 524288)")

# Send at the rate limit (500 req/s) from a single client
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>:8545/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"$DATA\",\"gas\":15000000,\"to\":\"0x0000000000000000000000000000000000000001\"}" &
done
wait
```

**Expected result:** Each request passes both bucket checks (1 token consumed from rate bucket, gas tokens consumed from gas bucket), matches the LOG filter, and causes `log.info("{}", request)` to allocate and format a ~1 MB string. At 500 req/s sustained, JVM heap allocation rate spikes by ~500 MB/s, GC logs show dramatically increased collection frequency, and node throughput for legitimate requests degrades measurably within seconds.
