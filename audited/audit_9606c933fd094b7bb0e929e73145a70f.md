### Title
Missing Length Constraint on GraphQL Alias Input Enables Unbounded Base32 Decoding (DoS)

### Summary
The `alias` field in `AccountInput` is validated only for character set (`^[A-Z2-7]+$`) with no maximum length constraint. An unauthenticated attacker can submit an arbitrarily long valid Base32 string that passes all validation, causing `decodeBase32()` to allocate a proportionally large byte array, which is then forwarded as a database query parameter. Repeated requests can exhaust server heap memory and CPU, rendering the account lookup service unavailable.

### Finding Description
**Code path:**

- `graphql/src/main/resources/graphql/account.graphqls`, line 90:
  ```graphql
  alias: String @Pattern(regexp: "^[A-Z2-7]+$")
  ```
  Only `@Pattern` is applied. No `@Size`, `@Length`, or equivalent directive exists (the only custom directives defined are `@Min` and `@Pattern` per `directive.graphqls`). There is no upper bound on string length.

- `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 83–85:
  ```java
  public static byte[] decodeBase32(String base32) {
      return BASE32.decode(base32);
  }
  ```
  No length check before decoding. Apache Commons Codec `Base32.decode()` will allocate `⌊len * 5 / 8⌋` bytes for any input length.

- `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 29–31:
  ```java
  public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
      return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
  }
  ```
  The decoded byte array is passed directly to the repository query with no size guard.

**Root cause:** The `@Pattern` directive validates only that characters belong to the Base32 alphabet. It places no upper bound on the string length. The `decodeBase32` utility and `getByAliasAndType` both assume the caller has already bounded the input size, but no caller does.

**Why existing checks fail:** `@Valid` on `AccountInput` in `AccountController` (line 33) triggers Bean Validation, but the only constraint on `alias` is `@Pattern`. No `@Size(max=…)` annotation is present. No HTTP request body size limit is configured in any YAML file. No application-level length check exists anywhere in the call chain.

### Impact Explanation
An attacker sending a 10 MB Base32 string (all `A`s, valid per the regex) causes `decodeBase32` to allocate ~6.25 MB on the heap per request. With concurrent requests, heap exhaustion leads to `OutOfMemoryError` or severe GC pressure, crashing or stalling the GraphQL service. Since this service is the primary interface for account balance and alias lookups, unavailability directly prevents users from querying account state, which is fund-related. Severity: **High (DoS)**.

### Likelihood Explanation
No authentication is required to submit a GraphQL query. The exploit requires only a single HTTP POST with a crafted `alias` value. It is trivially scriptable and repeatable. Any external user can trigger it.

### Recommendation
Add a maximum length constraint on the `alias` field. Since valid Hedera aliases are at most 88 Base32 characters (for a 55-byte ECDSA compressed key), enforce this at the schema level:

1. Define a `@Size` directive in `directive.graphqls`:
   ```graphql
   directive @Size(max: Int! = 2147483647, message: String = "graphql.validation.Size.message")
   on ARGUMENT_DEFINITION | INPUT_FIELD_DEFINITION
   ```
2. Apply it in `account.graphqls`:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max: 100)
   ```
3. Add a guard in `decodeBase32` as defense-in-depth:
   ```java
   public static byte[] decodeBase32(String base32) {
       if (base32.length() > 100) throw new IllegalArgumentException("Alias too long");
       return BASE32.decode(base32);
   }
   ```

### Proof of Concept
```bash
# Generate a 5MB alias string of valid Base32 chars
ALIAS=$(python3 -c "print('A' * 5_000_000)")

curl -s -X POST http://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { shard realm num } } }\"}"
```

Repeat this request concurrently (e.g., 50 threads). Each request causes `decodeBase32` to allocate ~3.1 MB on the heap. The server will exhibit severe GC pressure or OOM, making account lookups unavailable to all users.
