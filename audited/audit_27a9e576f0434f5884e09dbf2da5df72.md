### Title
Unauthenticated `/health/readiness` Endpoint Leaks Per-Component Infrastructure Status via Named Check Enumeration

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is registered with no authentication and delegates directly to the `hellofresh/health-go/v4` library's `HandlerFunc`, which serializes a `health.Check` JSON object containing per-check names (`"postgresql"`, `"network"`), their individual statuses, and error `output` strings. Any unprivileged external user can query this endpoint to determine which specific infrastructure component is degraded and extract error messages that may reveal connection details.

### Finding Description
**Code path:**

- `rosetta/app/middleware/health.go`, `NewHealthController()` (lines 37–50): Two named checks are registered — `"postgresql"` (backed by a live DSN connection attempt) and `"network"` (backed by a localhost Rosetta API call).
- `Routes()` (lines 63–78): Both `/health/liveness` and `/health/readiness` are registered as plain `GET` routes with no authentication wrapper.
- `rosetta/main.go` (lines 217–219): The router is wrapped only in `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. No authentication middleware is applied to health routes.

**Root cause:** The `hellofresh/health-go/v4` `HandlerFunc` returns a JSON body of type `health.Check` that includes a `checks` map keyed by the exact registered check names (`"postgresql"`, `"network"`), each entry containing a `status` field and an `output` field populated with the raw error string returned by the failing check function. The failed assumption is that health endpoints are internal-only; in practice the Rosetta service is a public-facing API server with no network-level restriction on `/health/readiness`.

**Exploit flow:**
1. Attacker sends `GET /health/readiness` with no credentials.
2. If PostgreSQL is unreachable, the response body contains `"postgresql": {"status":"unavailable","output":"dial tcp <host>:<port>: connect: connection refused"}` — revealing the DB host/port.
3. If the network check fails, the response contains `"network": {"status":"unavailable","output":"..."}` with the Rosetta API error, confirming the internal port and service state.
4. If both pass, the attacker learns the full stack is healthy — useful for timing attacks.

**Why existing checks are insufficient:** `server.CorsMiddleware` only sets CORS headers; it performs no authentication. There is no middleware, route guard, or IP allowlist applied to the health routes anywhere in the rosetta server setup.

### Impact Explanation
An attacker can precisely enumerate which layer of the Rosetta infrastructure (database vs. network/consensus layer) is degraded at any moment. The `output` field from the postgres check can expose the database hostname, port, and driver-level error strings. This topology intelligence can be used to time targeted denial-of-service or exploitation attempts against the PostgreSQL instance or the layer-1 network components when they are already in a weakened state, indirectly threatening the reliability of smart contract execution that depends on this mirror node.

### Likelihood Explanation
Exploitation requires only an HTTP GET request — no credentials, no special tooling, no prior access. The Rosetta API is a public-facing service by design. An attacker can poll the endpoint continuously and cheaply to build a real-time map of infrastructure health. This is trivially repeatable and automatable.

### Recommendation
1. **Restrict the endpoint at the network level**: Place `/health/readiness` behind an internal load balancer or Kubernetes `InternalTrafficPolicy`, accessible only to the orchestration plane (kubelet, ingress health probes).
2. **Suppress per-check detail in the response**: Replace the library's default `HandlerFunc` with a custom handler that returns only a top-level `{"status":"ok"}` or `{"status":"unavailable"}` without the `checks` map, stripping the named component breakdown and error output from external responses.
3. **Add an authentication middleware** to the health routes if they must remain externally reachable, consistent with how the metrics endpoint is protected elsewhere in the codebase.
