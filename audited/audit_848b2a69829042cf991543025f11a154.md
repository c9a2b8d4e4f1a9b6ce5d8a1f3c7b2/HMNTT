### Title
Unauthenticated DoS via Unbounded Concurrent Requests to No-Timestamp `getBalances()` Path

### Summary
The `getBalances()` handler in `rest/balances.js` contains no rate limiting on the `/api/v1/balances` endpoint. When no timestamp filter is provided, the handler executes a query that performs a full scan of the `entity` table via a CTE, followed by a correlated subquery against `token_account` for each returned row. An unprivileged attacker can flood this endpoint with concurrent requests to exhaust database CPU and I/O, degrading all mirror node instances sharing the database.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 133–149, when `tsQuery` is falsy (no timestamp filter provided), the handler builds and executes:

```sql
with entity_balance as (
  select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
  from entity
  where type in ('ACCOUNT', 'CONTRACT')   -- full table scan, no index on balance
)
select ab.*, (<correlated_token_subquery>) as token_balances
from entity_balance ab
[where ab.balance >= 0]                   -- passes all rows
order by ab.account_id desc
limit 100
``` [1](#0-0) 

The correlated subquery (`getTokenAccountBalanceSubQuery`) executes once per row in the outer result:

```sql
select json_agg(...)
from (
  select token_id, balance
  from token_account ta
  where ta.account_id = ab.account_id and ta.associated is true
  order by token_id desc
  limit 50
) as account_token_balance
``` [2](#0-1) 

**Root cause — no rate limiting on the REST API:**

`server.js` registers the balances route with no rate-limiting middleware. The only middleware applied is `authHandler` (which only sets a custom limit for authenticated users and does not reject unauthenticated requests), `requestLogger`, optional metrics, and optional response cache. There is no per-IP or global request-rate limiter. [3](#0-2) 

The throttle/rate-limit infrastructure that exists in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` module's `/contracts/call` endpoint and is not applied to the REST API. [4](#0-3) 

**Why `account.balance=gte:0` is the worst-case filter:**

The `account.balance=gte:0` condition is applied as a WHERE predicate on the CTE result (`ab.balance >= 0`). Because all account balances are non-negative, this filter passes every row from the full entity table scan, ensuring the outer query always reaches the maximum `limit 100` rows and fires 100 correlated `token_account` subqueries per request. A more selective filter (e.g., `account.balance=eq:999999`) would reduce the number of correlated subqueries executed.

**Existing checks and why they are insufficient:**

| Protection | Value | Why insufficient |
|---|---|---|
| `statement_timeout` | 20 000 ms | Limits per-query duration but does not prevent many concurrent queries from holding connections for 20 s each |
| `db.pool.maxConnections` | 10 | Only the Node.js pool; PgBouncer `max_user_connections` for `mirror_rest` is 250 |
| Response `limit` max | 100 | Caps rows returned but not the cost of the full entity CTE scan |
| No rate limiter | — | Confirmed absent from all REST middleware | [5](#0-4) [6](#0-5) 

### Impact Explanation

On a production network the `entity` table contains millions of rows. Each no-timestamp `/api/v1/balances` request causes a sequential scan of that table plus up to 100 index lookups into `token_account`. With PgBouncer allowing 250 simultaneous `mirror_rest` connections and a 20-second statement timeout, an attacker can sustain 250 concurrent long-running queries. This saturates PostgreSQL CPU and I/O, causing query queuing and timeouts for all other mirror node API endpoints that share the same database, effectively degrading all mirror node instances behind that database. The mirror node is the primary read interface for the Hedera network; its degradation prevents clients from querying balances, transactions, and other state.

### Likelihood Explanation

The attack requires zero privileges — only the ability to send HTTP GET requests to a public endpoint. The endpoint is documented in the OpenAPI spec and is publicly accessible. The attack is trivially scriptable with any HTTP load tool (e.g., `ab`, `wrk`, `hey`). It is repeatable and requires no special knowledge of the network state. The attacker does not need to know any account IDs; `account.balance=gte:0` is a generic filter that works unconditionally.

### Recommendation

1. **Add a rate limiter middleware** to the REST API (e.g., `express-rate-limit`) applied globally or specifically to expensive endpoints like `/api/v1/balances`. A per-IP limit of ~10–20 req/s is a reasonable starting point.
2. **Add a `LIMIT` inside the `entity_balance` CTE** so the full entity table scan is bounded even before the outer WHERE clause is applied.
3. **Add a balance index** on `entity(balance)` where `type in ('ACCOUNT','CONTRACT')` so the `account.balance` filter can use an index scan instead of a sequential scan.
4. **Reduce `max_user_connections`** for `mirror_rest` in PgBouncer to a value consistent with the Node.js pool size (e.g., 20–30) to prevent connection amplification.
5. Consider requiring at least one selective filter (account ID or timestamp) before executing the no-timestamp path.

### Proof of Concept

```bash
# Send 250 concurrent requests with no timestamp and a maximally permissive balance filter
seq 1 250 | xargs -P 250 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/balances?account.balance=gte:0&limit=100" \
  -o /dev/null

# Repeat in a loop to sustain load
while true; do
  seq 1 250 | xargs -P 250 -I{} \
    curl -s "https://<mirror-node-host>/api/v1/balances?account.balance=gte:0&limit=100" \
    -o /dev/null
done
```

**Expected result:** PostgreSQL CPU spikes to 100%, query latency for all REST API endpoints increases dramatically, and clients begin receiving 500 errors or timeouts as the 20-second statement timeout is hit across all 250 concurrent connections.

const getTokenAccountBalanceSubQuery = (order) => {
