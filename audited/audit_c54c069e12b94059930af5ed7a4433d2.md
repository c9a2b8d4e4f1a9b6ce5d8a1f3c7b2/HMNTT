### Title
Thundering Herd via Unjittered `delaySubscription` in `PollingTopicListener.listen()` Enables DB Exhaustion

### Summary
`PollingTopicListener.listen()` applies `jitter(0.1)` only to the `repeatWhen` loop, not to the initial `delaySubscription(interval, scheduler)`. An unprivileged attacker who opens many simultaneous gRPC connections (each with up to 5 subscriptions per `maxConcurrentCallsPerConnection`) causes all initial DB poll queries to fire at the same instant (T + 500 ms), exhausting the DB connection pool and degrading or partitioning the database from the mirror node.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–48:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // line 39 — NO jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                             // line 41 — jitter only on repeat
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

**Root cause:** `delaySubscription(interval, scheduler)` schedules the first DB poll at a fixed offset (`interval`, default 500 ms) from subscription time. The `jitter(0.1)` call is scoped exclusively to the `RepeatSpec` governing subsequent polls. There is no randomisation of the initial delay.

**Failed assumption:** The design assumes subscriptions arrive spread over time, so the initial polls are naturally staggered. An attacker invalidates this by opening all connections simultaneously.

**Exploit flow:**
1. Attacker opens *N* TCP connections to port 5600 (no authentication required for `subscribeTopic`).
2. Each connection issues 5 `subscribeTopic` RPCs (the per-connection cap set by `maxConcurrentCallsPerConnection = 5` in `NettyProperties`).
3. All *5N* `PollingTopicListener.listen()` calls record subscription time ≈ T₀.
4. At T₀ + 500 ms, all *5N* `delaySubscription` timers expire simultaneously and enqueue tasks on the shared `Schedulers.boundedElastic()` instance.
5. `boundedElastic()` drains its queue as fast as threads allow (default: 10 × CPU cores threads, queue depth 100 000), issuing *5N* concurrent `topicMessageRepository.findByFilter()` calls against the database.
6. The DB connection pool is saturated; legitimate queries time out or are rejected.

**Why existing checks are insufficient:**

| Check | Why it fails |
|---|---|
| `maxConcurrentCallsPerConnection = 5` | Per-connection limit only; attacker opens *N* connections |
| `jitter(0.1)` on `repeatWhen` | Does not affect the initial `delaySubscription` |
| `Schedulers.boundedElastic()` | Queues all tasks and drains them as fast as possible — burst is delayed, not spread |
| No global subscription counter gate | `subscriberCount` in `TopicMessageServiceImpl` is a metric gauge only; it does not reject new subscriptions |

### Impact Explanation
When the server is configured with `type=POLL`, a single attacker with enough open connections can cause a synchronised burst of up to thousands of simultaneous DB queries at T₀ + 500 ms. This exhausts the JDBC/R2DBC connection pool, causing query timeouts for all other mirror-node components (REST API, importer) that share the same database, effectively partitioning the DB from the application. Severity: **High** (availability impact on the shared database tier).

### Likelihood Explanation
The attack requires no credentials — `subscribeTopic` is an unauthenticated gRPC call. Opening hundreds of TCP connections and issuing 5 RPCs each is trivial from a single host. The only prerequisite is that the operator has configured `hiero.mirror.grpc.listener.type=POLL`; this is a documented, supported mode. The attack is repeatable: after the burst subsides, the attacker can disconnect and reconnect to trigger another wave.

### Recommendation
1. **Add jitter to the initial delay:** Replace `delaySubscription(interval, scheduler)` with a randomised delay, e.g.:
   ```java
   Duration jitteredDelay = interval.plus(
       Duration.ofMillis((long)(Math.random() * interval.toMillis())));
   .delaySubscription(jitteredDelay, scheduler)
   ```
2. **Enforce a global subscription limit:** Gate `listen()` with an atomic counter and reject (with `RESOURCE_EXHAUSTED` status) subscriptions beyond a configurable maximum.
3. **Prefer `SHARED_POLL` or `REDIS`:** `SharedPollingTopicListener` issues a single shared DB poll regardless of subscriber count — it is immune to this attack. Document that `POLL` mode is unsafe for public-facing deployments.
4. **Apply DB connection pool limits:** Configure a maximum pool size and statement timeout so a burst cannot hold connections indefinitely.

### Proof of Concept
```
# Precondition: server running with hiero.mirror.grpc.listener.type=POLL

# 1. Open 200 simultaneous gRPC connections, each with 5 subscribeTopic RPCs
#    (e.g., using ghz or a custom gRPC client):
for i in $(seq 1 200); do
  ghz --insecure \
      --proto consensus_service.proto \
      --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
      --data '{"topicID":{"topicNum":1}}' \
      --concurrency 5 \
      --total 5 \
      --async \
      <mirror-node-host>:5600 &
done

# 2. All 1000 subscriptions are established at approximately T0.
# 3. At T0 + 500ms, observe 1000 simultaneous DB queries in pg_stat_activity:
#    SELECT * FROM pg_stat_activity WHERE state = 'active';
# 4. DB connection pool is exhausted; REST API queries begin timing out.
