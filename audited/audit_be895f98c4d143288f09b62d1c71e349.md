### Title
Global-Only Rate Limit Bucket Enables Single-Client DoS Starvation of Staking Reward API

### Summary
`ThrottleConfiguration.rateLimitBucket()` creates a single application-wide `Bucket` bean shared by every caller. Because no per-IP or per-account sub-limit exists anywhere in the application or infrastructure layer, a single unprivileged attacker can saturate the entire 500 RPS budget, returning HTTP 429 to all other clients — including legitimate stakers querying `pending_reward` balances via the staking-reward precompile.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) constructs one singleton `Bucket` with `capacity = requestsPerSecond` (default 500) and `refillGreedy(500, Duration.ofSeconds(1))`. [1](#0-0) 

`ThrottleManagerImpl.throttle()` (line 38) calls `rateLimitBucket.tryConsume(1)` against that single shared bean for every inbound request, with no caller identity involved. [2](#0-1) 

**Root cause / failed assumption:** The design assumes the 500-token budget will be spread across many independent clients. There is no mechanism to enforce that assumption.

**Why existing checks fail:**

- `RequestProperties` filters (the only configurable per-request logic) match on `DATA`, `FROM`, `TO`, `GAS`, `BLOCK`, `ESTIMATE`, `VALUE` — not on source IP. They are also disabled by default (`request = List.of()`). [3](#0-2) [4](#0-3) 

- `LoggingFilter` reads `request.getRemoteAddr()` for logging only — it is never fed into any throttle decision. [5](#0-4) 

- The Traefik ingress middleware is configured only for retry, not per-IP rate limiting. [6](#0-5) 

- The GCP backend policy `maxRatePerEndpoint: 250` is a per-pod cap, not a per-client cap. [7](#0-6) 

- The nginx proxy in `docker-compose.yml` has no `limit_req` directive for the `/api/v1/contracts/call` location. [8](#0-7) 

**Exploit flow:**
1. Attacker sends ≥500 POST requests/second to `POST /api/v1/contracts/call` targeting the staking-reward precompile address (e.g., `0x167` / HTS precompile), with minimal gas (≥21 000 to pass gas-bucket scaling).
2. Each request atomically consumes one token from the shared `rateLimitBucket`.
3. The bucket empties within the first millisecond of each refill window.
4. All subsequent requests from any other client receive HTTP 429 `"Requests per second rate limit exceeded"` for the remainder of that second.
5. Legitimate stakers calling the same endpoint to read `pending_reward` are denied service continuously as long as the flood persists.

### Impact Explanation
Legitimate users are unable to query `pending_reward` balances via the web3 API for the duration of the attack. While on-chain rewards are not stolen, the inability to discover pending balances can delay reward collection decisions. The affected endpoint (`/api/v1/contracts/call`) is the sole path for EVM-level staking-reward precompile queries in this service. Severity: **Medium** (availability impact, no direct fund loss).

### Likelihood Explanation
No special privilege, account, or on-chain asset is required. A single machine with a standard HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain 500 RPS trivially. The attack is repeatable indefinitely, costs the attacker nothing, and is not detectable until monitoring alerts fire. Likelihood: **High**.

### Recommendation
1. **Per-IP sub-bucket:** Maintain a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` / `X-Real-IP` with appropriate trust of the proxy). Each IP gets its own bucket with a fraction of the global limit (e.g., 10–20 RPS).
2. **Traefik/nginx rate limiting:** Add `limit_req` (nginx) or Traefik `rateLimit` middleware with `average`/`burst` per source IP at the ingress layer as a defence-in-depth measure.
3. **Expose IP as a `RequestFilter` field:** Extend `RequestFilter.FilterField` to include `REMOTE_IP` so operators can configure per-IP THROTTLE/REJECT rules without code changes.

### Proof of Concept
```bash
# Flood the staking-reward precompile endpoint at 600 RPS from a single host
# (requires wrk + Lua script or hey)
hey -n 10000 -c 600 -q 600 \
  -m POST \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000167","data":"0x<getPendingReward_selector><account_address>","gas":50000}' \
  https://<mirror-node-host>/api/v1/contracts/call

# Expected: after the first ~500 requests in each 1-second window,
# all remaining responses return HTTP 429 "Requests per second rate limit exceeded"
# Legitimate staker queries issued concurrently will also receive HTTP 429.
```

    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
