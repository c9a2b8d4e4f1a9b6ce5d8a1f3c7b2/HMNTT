### Title
Unbounded DB Reads via Non-Existent Alias in `getNftAllowances()` Enabling Griefing DoS

### Summary
`NftAllowanceServiceImpl.getNftAllowances()` unconditionally calls `entityService.lookup()` for every request, which for alias or EVM-address path parameters executes a raw SQL query against the `entity` table with no caching. Because the `rest-java` module has no rate limiting, an unprivileged attacker can flood the endpoint with syntactically valid but non-existent aliases, forcing a DB read on every call and degrading database performance for legitimate users.

### Finding Description
**Exact code path:**

`AllowancesController.getNftAllowances()` → `NftAllowanceServiceImpl.getNftAllowances()` line 28:
```java
var id = entityService.lookup(request.getAccountId());
```
`EntityServiceImpl.lookup()` (lines 30–37):
```java
case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
// ...
return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
```
`EntityRepository.findByAlias()` (line 13–14):
```java
@Query(value = "select id from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByAlias(byte[] alias);
```

**Root cause:** For `EntityIdAliasParameter` and `EntityIdEvmAddressParameter` inputs, every call issues a live SQL query. There is no `@Cacheable` annotation, no negative-result cache, and no rate limiting anywhere in the `rest-java` module (confirmed: zero matches for throttle/rate-limit/cache annotations in `rest-java/**`). The `EntityNotFoundException` is caught by `GenericControllerAdvice.notFound()` and returns HTTP 404, so the attacker receives clean feedback and the request completes without error from the attacker's perspective.

**Why existing checks fail:** The only protection is the 404 response itself. The `web3` module has a `ThrottleManagerImpl` with bucket4j rate limiting, but this is entirely absent from `rest-java`. The parameter validation in `checkOwnerSpenderParamValidity()` runs before `lookup()` but does not prevent alias-based lookups from reaching the DB.

### Impact Explanation
Each request with a non-existent alias causes one full indexed SQL read against the `entity` table. Under sustained high-volume attack (e.g., 1000 req/s with distinct random aliases), the database connection pool and I/O capacity are consumed by reads that return nothing, increasing query latency for legitimate requests. This is a griefing-class DoS: no funds are at risk, but service quality degrades proportionally to attack volume.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly documented in `openapi.yml` (`/api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/nfts`). Aliases are accepted as base32-encoded strings; generating syntactically valid but non-existent aliases requires only knowledge of the encoding format (visible from the test strings in `EntityServiceTest`). The attack is trivially scriptable with `curl` or any HTTP load tool and is fully repeatable.

### Recommendation
1. **Add a negative-result cache** in `EntityServiceImpl.lookup()` (e.g., Caffeine with a short TTL such as 5–10 s) so repeated lookups of the same non-existent alias do not hit the DB.
2. **Add rate limiting to `rest-java`** mirroring the bucket4j pattern already used in the `web3` module (`ThrottleManagerImpl`), applied at the servlet filter or controller-advice level.
3. As a defense-in-depth measure, consider returning the same 404 for both alias-not-found and numeric-not-found cases (already done) but add a configurable per-IP request rate cap.

### Proof of Concept
```bash
# Generate a syntactically valid but non-existent alias (base32 of random bytes)
ALIAS="0.0.HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA"

# Flood the endpoint - each request triggers a DB read
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/allowances/nfts" &
done
wait
# Result: 10000 concurrent SQL queries against entity table, all returning empty,
# all throwing EntityNotFoundException, all returning HTTP 404.
# DB connection pool exhaustion degrades response times for legitimate users.
```
