### Title
Unauthenticated Unbounded Pagination DoS via Absent Application-Level Rate Limiting on `/api/v1/blocks`

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts `order=asc&limit=100` from any unauthenticated caller and issues a raw SQL query against the `record_file` table with no application-level rate limiting. An attacker can follow the generated `next` pagination links in rapid succession, sustaining a continuous stream of DB queries that exhausts the database connection pool and I/O capacity. Infrastructure-level rate limiting is disabled by default in the Helm chart.

### Finding Description

**Code path:**

`blockController.js` `getBlocks()` (lines 101–112) calls `extractSqlFromBlockFilters()` then `RecordFileService.getBlocks()`: [1](#0-0) 

`extractLimitFromFilters()` (lines 57–61) contains a logic error: when `limit.value > maxLimit` it falls back to `defaultLimit` (25) instead of capping at `maxLimit` (100). For values ≤ `maxLimit` it passes the value through unchanged, so `limit=100` is accepted as-is: [2](#0-1) 

`RecordFileService.getBlocks()` (lines 149–162) interpolates `filters.limit` directly into the SQL string (not a bind parameter) and issues the query with no additional guard: [3](#0-2) 

`generateNextLink()` (lines 90–99) always emits a `next` cursor anchored to the last returned block's `index`, so following it with `order=asc` advances a window forward through the entire table: [4](#0-3) 

**Root cause — no application-level rate limiting.** The Express middleware stack (`server.js`) registers `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and an optional response-cache check, but no rate-limiting middleware for the blocks route: [5](#0-4) 

The only rate limiting present in the codebase is in the web3 Java module (`ThrottleManagerImpl`) and applies exclusively to `/contracts/call`: [6](#0-5) 

The Helm chart for the REST service has `global.middleware: false` by default, meaning the optional Traefik `rateLimit` middleware (present for Rosetta) is not deployed: [7](#0-6) 

Confirmed defaults: `responseLimit.default = 25`, `responseLimit.max = 100`: [8](#0-7) 

### Impact Explanation
Each request with `limit=100` causes the database to execute `SELECT … FROM record_file ORDER BY consensus_end ASC LIMIT 100` (or with a `WHERE index > N` cursor). While individual queries use the index and are fast, a single attacker issuing hundreds of such requests per second saturates the DB connection pool, increases I/O wait, and degrades or denies service for all other API consumers. The `record_file` table on mainnet contains tens of millions of rows; full traversal via pagination is trivially scriptable. No credentials, tokens, or special network position are required.

### Likelihood Explanation
The endpoint is publicly documented, requires zero authentication, and the pagination cursor is self-describing in the `next` link. A single HTTP client in a loop (e.g., `curl` or a trivial script) can sustain hundreds of requests per second. The attack is repeatable, requires no prior knowledge of the system, and leaves no persistent state that would block re-entry.

### Recommendation
1. **Add application-level rate limiting** to the Express REST API (e.g., `express-rate-limit` or `node-rate-limiter-flexible`) scoped per source IP, applied globally or at minimum to all list endpoints.
2. **Enable the Traefik middleware** in the Helm chart by default (`global.middleware: true`) with a `rateLimit` stanza matching the Rosetta chart's configuration.
3. **Fix the `extractLimitFromFilters` logic error**: replace `defaultLimit` with `maxLimit` in the over-limit branch so the cap is consistent:
   ```js
   return limit ? (limit.value > maxLimit ? maxLimit : limit.value) : defaultLimit;
   ```
4. **Add a DB-level statement timeout** for the REST API's PostgreSQL role (analogous to the `statementTimeout: 3000` already set for the web3 module).

### Proof of Concept
```bash
# Step 1 – initial request, forces oldest-first full-table traversal
curl -s "https://<mirror-node>/api/v1/blocks?order=asc&limit=100" | tee page1.json

# Step 2 – extract next link and loop
NEXT=$(jq -r '.links.next' page1.json)
while [ "$NEXT" != "null" ]; do
  curl -s "https://<mirror-node>$NEXT" | tee /tmp/page.json
  NEXT=$(jq -r '.links.next' /tmp/page.json)
done

# Step 3 – run N parallel instances of the above loop to saturate DB connections
for i in $(seq 1 200); do
  bash exploit.sh &
done
```
No authentication header is needed. Each iteration issues a new `SELECT` against `record_file`; 200 concurrent loops at ~10 req/s each = 2 000 DB queries/second, exhausting a typical connection pool of 20–100 connections.
