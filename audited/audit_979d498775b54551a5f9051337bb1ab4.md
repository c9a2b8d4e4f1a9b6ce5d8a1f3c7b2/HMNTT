### Title
Unauthenticated Alias-Based Request Flood Causes Database Connection Pool Exhaustion (DoS) on Crypto Allowance Endpoint

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` unconditionally issues a database query (`select id from entity where alias = $1`) for every alias-format path parameter via `EntityService.getEncodedId()`. The REST API has no rate limiting or in-flight request cap in its middleware stack, and the default connection pool is only 10 connections. An unauthenticated attacker flooding the endpoint with distinct valid-format aliases can exhaust the pool, causing connection timeout errors for all REST API consumers. The Traefik retry middleware (10 attempts) amplifies the attack by up to 10×.

### Finding Description

**Exact code path:**

`getAccountCryptoAllowances()` at line 77 calls `EntityService.getEncodedId()` unconditionally: [1](#0-0) 

Inside `getEncodedId()`, when the path parameter passes `AccountAlias.isValid()`, it calls `getAccountIdFromAlias()` → `getAccountFromAlias()`, which executes a live database query with no caching: [2](#0-1) [3](#0-2) 

**Root cause — no rate limiting in the REST middleware stack:**

The `server.js` middleware chain is: `httpContext` → `requestLogger` → `authHandler` → optional metrics → optional Redis cache check → route handlers. There is no rate-limiting or concurrency-limiting middleware: [4](#0-3) 

The `authHandler` only sets a custom response-size limit for authenticated users; it does not throttle unauthenticated requests: [5](#0-4) 

**Small default connection pool:**

The pool is capped at `maxConnections` (default **10**) with a `statementTimeout` of 20 000 ms and `connectionTimeoutMillis` of 20 000 ms: [6](#0-5) [7](#0-6) 

**Traefik middleware has no rate limit for the REST API:**

The deployed Traefik middleware for the REST chart contains only a circuit breaker and a **retry** (10 attempts), with no `rateLimit` or `inFlightReq` — unlike the Rosetta chart which has both: [8](#0-7) 

The retry directive amplifies the attack: each attacker request can generate up to 10 upstream requests.

**Response cache does not help:**

The Redis response cache is **disabled by default** (`cache.response.enabled: false`). Even when enabled, the cache key is the MD5 of `req.originalUrl`, so each distinct alias produces a unique cache miss and a fresh DB query: [9](#0-8) [10](#0-9) 

### Impact Explanation

With a pool of 10 connections and a 20-second connection timeout, an attacker sending ~50–100 concurrent requests per second with distinct valid base32 aliases will saturate the pool queue. Legitimate requests begin receiving connection-timeout errors (HTTP 500), making the entire REST API unavailable — not just the allowance endpoint — because all endpoints share the same `global.pool`. The Traefik circuit breaker trips at 25% error rate, further degrading availability.

### Likelihood Explanation

No authentication or account is required. Valid alias strings are trivially generated (any base32-encoded byte sequence). The attack is repeatable from a single host or distributed across IPs. The Traefik retry amplifier means even a modest flood (e.g., 10 req/s) becomes 100 upstream DB queries per second, which is sufficient to exhaust a 10-connection pool given any non-trivial query latency.

### Recommendation

1. **Add an in-flight request limit and per-IP rate limit to the REST Traefik middleware**, mirroring the Rosetta chart's `inFlightReq` and `rateLimit` entries.
2. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) so the pool cannot be exhausted regardless of infrastructure configuration.
3. **Cache alias→entityId resolution** in-process (the config already defines `cache.entityId.maxAge`/`maxSize` but the `EntityService` does not use it for alias lookups); a short-lived in-memory cache would eliminate the per-request DB query for repeated aliases.
4. **Increase `maxConnections`** or front the pool with PgBouncer in transaction mode (already configured for `mirror_rest` in the Citus chart) to absorb burst traffic.

### Proof of Concept

```bash
# Generate 1000 concurrent alias-based requests with distinct aliases
# (ALIAS_N are valid base32 strings, e.g. AAAQEAYEAUDAOCAJBEE...)
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/accounts/AAAQEAYEAUDAOCAJBEE${i}/allowances/crypto" &
done
wait
# Observe: legitimate requests to /api/v1/accounts/0.0.1/allowances/crypto
# begin returning HTTP 500 (connection pool exhausted) within seconds.
```
