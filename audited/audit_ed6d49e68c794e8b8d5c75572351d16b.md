### Title
Unbounded Subscription Creation via Short-Lived `endTime` Causes Sustained DB and CPU Resource Exhaustion

### Summary
Any unauthenticated caller can open an unlimited number of gRPC subscriptions by setting `endTime` to just one nanosecond past `startTime`. Each subscription immediately fires a historical database retrieval query, registers a live listener, and schedules a safety-check query after one second — then remains alive for up to 30 seconds (the `endTimeInterval`) before the `pastEndTime` poller terminates it. Because the only per-connection cap (`maxConcurrentCallsPerConnection = 5`) is not a global limit, an attacker opening many connections can sustain thousands of concurrent subscriptions, exhausting the database connection pool and CPU.

### Finding Description

**Validation gap in `TopicMessageFilter`:**
`isValidEndTime()` only requires `endTime > startTime` with no minimum gap enforced. [1](#0-0) 
A value of `endTime = startTime + 1` (one nanosecond) passes validation.

**Full pipeline launched unconditionally in `subscribeTopic()`:** [2](#0-1) 

For every subscription, regardless of how short-lived it will be:
- Line 63: `topicMessageRetriever.retrieve(filter, true)` — a DB query fires immediately.
- Lines 67–70: A safety-check `Mono.delay(1s)` fires a second DB query (`missingMessages`) one second later.
- Line 64: A live listener is registered via `topicListener.listen(...)`.

**Subscription lives for up to 30 seconds even with `endTime = now + 1ns`:**
`isComplete()` returns `true` only after `endTime + endTimeInterval` has passed: [3](#0-2) 

`endTimeInterval` defaults to 30 seconds: [4](#0-3) 

`pastEndTime()` polls on this interval, so the subscription is not torn down until the next poll fires after `endTime + 30s`: [5](#0-4) 

**`subscriberCount` is a metric only — no cap is enforced:** [6](#0-5) 

**`maxConcurrentCallsPerConnection = 5` is per-connection, not global:** [7](#0-6) 

An attacker opens N connections × 5 subscriptions each = 5N concurrent subscriptions with no server-side global limit.

### Impact Explanation

Each short-lived subscription causes at minimum two database queries (historical retrieval + safety check) and one live listener registration within its first second of life. With thousands of concurrent subscriptions cycling every ~30 seconds, the database connection pool is saturated, reactive scheduler threads are consumed, and `TopicContext` objects accumulate in heap. This degrades or denies service for legitimate subscribers without any economic cost to the attacker. Severity: **Medium** (griefing/DoS, no fund loss, but service availability is impacted).

### Likelihood Explanation

No authentication is required. The attacker needs only a gRPC client (e.g., `grpcurl` or any SDK), a valid topic ID (publicly observable on-chain), and the ability to open many TCP connections. The attack is fully scriptable, repeatable, and requires no special knowledge beyond the public API. The per-connection call limit of 5 is trivially bypassed by opening more connections.

### Recommendation

1. **Enforce a minimum `endTime` window**: Add a validation constraint requiring `endTime >= startTime + some_minimum_duration` (e.g., 5 seconds) to prevent trivially short subscriptions.
2. **Enforce a global concurrent subscription cap**: Use `subscriberCount` (already tracked) to reject new subscriptions above a configurable threshold, returning `RESOURCE_EXHAUSTED`.
3. **Add per-IP or per-client connection/subscription rate limiting** at the gRPC interceptor layer.
4. **Reduce `endTimeInterval`** or make `pastEndTime` terminate immediately when `endTime` is already in the past at subscription start, rather than waiting a full 30-second polling cycle.

### Proof of Concept

```
# Open 200 connections, each with 5 subscriptions, endTime = startTime + 1ns
# Using grpcurl or a custom gRPC client in a loop:

for i in $(seq 1 200); do
  grpcurl -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": <NOW>},
    "consensusEndTime":   {"seconds": <NOW>, "nanos": 1}
  }' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

Each of the 1000 concurrent subscriptions (200 connections × 5 calls each):
1. Immediately issues a `SELECT` against the `topic_message` table.
2. After 1 second, issues a second gap-recovery `SELECT`.
3. Holds a live listener slot for ~30 seconds before `pastEndTime` terminates it.
4. The attacker re-issues the batch every 30 seconds to maintain sustained pressure.

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```

```java
    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
```
