### Title
Unauthenticated IP Spoofing via Trusted Proxy Headers in `TracingMiddleware` Enables Audit Trail Poisoning

### Summary
`getClientIpAddress` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` and `X-Forwarded-For` headers without validating that they originate from a trusted proxy. The production Traefik deployment is explicitly configured with `--entryPoints.web.forwardedHeaders.insecure`, which passes all client-supplied forwarded headers through unmodified. Any unauthenticated external attacker can therefore inject arbitrary IP values into every access log entry, permanently corrupting the audit trail for transaction history lookups.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // line 64 — blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 — blindly trusted
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback
    }

    return ipAddress
}
```

`r.RemoteAddr` (the actual TCP peer address) is only consulted when **both** headers are absent. If either header is present, its value is returned verbatim with no validation.

**Root cause — the insecure Traefik configuration:**

`charts/hedera-mirror-common/values.yaml`, lines 635–637:

```yaml
globalArguments:  # Expose X-Forwarded-For header for tracing
  - --entryPoints.web.forwardedHeaders.insecure
  - --entryPoints.websecure.forwardedHeaders.insecure
```

`forwardedHeaders.insecure` instructs Traefik to trust forwarded headers from **any** source, including untrusted external clients. Traefik's own documentation marks this flag as a security risk. As a result, a client-supplied `X-Real-IP: <arbitrary>` header reaches the Go application unchanged.

**Exploit flow:**

1. Attacker sends `POST /block/transaction` with `X-Real-IP: 203.0.113.1`.
2. Traefik (insecure mode) forwards the header unmodified.
3. `getClientIpAddress` returns `203.0.113.1`.
4. `TracingMiddleware` logs: `203.0.113.1 POST /block/transaction (200) in 12ms`.
5. Attacker repeats with `203.0.113.2`, `203.0.113.3`, … cycling through thousands of IPs.
6. Every log line shows a different, attacker-controlled source IP; `r.RemoteAddr` (the real IP) is never logged.

**Existing checks reviewed and shown insufficient:**

The Traefik rate-limit middleware (`charts/hedera-mirror-rosetta/values.yaml`, lines 157–160) uses `sourceCriterion.requestHost: true`, meaning it throttles per `Host` header — not per source IP — and does not prevent this attack. The in-flight limit (lines 152–156) uses `ipStrategy.depth: 1` from `X-Forwarded-For`, which is itself spoofable by the same mechanism.

### Impact Explanation

Every access log entry for Rosetta endpoints (including `/block/transaction`, `/account/balance`, `/block`) can be permanently falsified. Forensic reconstruction of "who queried which transaction" becomes impossible. In a financial-audit context this directly satisfies the stated threat scope: reorganizing the observable history of transaction lookups without any fund theft. Log-based alerting (e.g., detecting a single actor repeatedly querying sensitive accounts) is also defeated.

### Likelihood Explanation

- **No authentication required** — Rosetta endpoints are public by design.
- **No special tooling** — a single `curl -H "X-Real-IP: ..."` command suffices.
- **Repeatable at scale** — the rate limit (10 req/s per host) still allows 36,000 spoofed log entries per hour from a single connection.
- **Permanent** — log entries are immutable once written; there is no mechanism to recover the true source IP after the fact.

### Recommendation

1. **Remove `forwardedHeaders.insecure`** from the Traefik entry-point configuration and replace it with an explicit trusted-IP allowlist (`forwardedHeaders.trustedIPs`) covering only the known proxy CIDR ranges.
2. **Validate the header source in `getClientIpAddress`**: only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` matches a configured set of trusted proxy addresses; otherwise fall back to `r.RemoteAddr` unconditionally.
3. **Always append `r.RemoteAddr`** to the log message as a separate field so the TCP-level peer is always recorded regardless of header content.

### Proof of Concept

```bash
# Cycle through 1000 fake source IPs against the block/transaction endpoint
for i in $(seq 1 1000); do
  curl -s -o /dev/null \
    -H "X-Real-IP: 198.51.100.$((i % 254 + 1))" \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1},
         "transaction_identifier":{"hash":"0x..."}}' \
    https://<rosetta-host>/block/transaction
done

# Resulting log lines (server side) will show 1000 different source IPs;
# the real attacker IP (r.RemoteAddr) never appears.
```
