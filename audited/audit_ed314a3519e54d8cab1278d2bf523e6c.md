### Title
Unauthenticated Resource Exhaustion via Uncached `RetrieveLatest()` DB Query on Every `/account/balance` Request Without BlockIdentifier

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` unconditionally calls `RetrieveLatest()` ŌĆö which executes a live database query on every invocation ŌĆö whenever `BlockIdentifier` is omitted from the request. The Rosetta application has no built-in rate limiting or caching for this path, and the only external protection (Traefik middleware) is optional and deployment-conditional. Any unprivileged external user can flood the endpoint with omitted-`BlockIdentifier` requests, driving repeated `SELECT ŌĆ” FROM record_file ORDER BY index DESC LIMIT 1` queries against the database and sustaining elevated CPU and DB I/O.

**Root cause:** There is no result cache, no in-process deduplication, and no application-level rate limiter anywhere in the `RetrieveLatest()` call chain. The genesis block is cached via `sync.Once`, but the latest-block query is not.

This middleware is:
1. **Optional** ŌĆö only applied when `global.middleware` is true and the Traefik CRD is deployed; direct access to the Rosetta port bypasses it entirely.
2. **Ineffective against a distributed attacker** ŌĆö `rateLimit` is keyed on `requestHost` (the Host header), not source IP, so a single attacker can consume the entire 10 req/s budget for all legitimate users.
3. **Self-amplifying** ŌĆö the `retry: attempts: 3` middleware causes Traefik to replay each failed request up to 3 times, multiplying DB load.

### Impact Explanation
Every unauthenticated POST to `/account/balance` with `block_identifier` omitted causes one synchronous `SELECT ŌĆ” FROM record_file ORDER BY index DESC LIMIT 1` database query. Under sustained flood (easily achievable with a single machine and a simple HTTP client loop), the database connection pool saturates, query latency rises, and CPU usage on both the Go process (goroutine scheduling, JSON marshalling) and the PostgreSQL instance (query execution, I/O) increases. Because the same database is shared with the mirror node importer and REST API, the impact is not isolated to the Rosetta service. A 30%+ increase in DB I/O and CPU over the 24-hour baseline is realistic with even modest request rates (hundreds per second) given the absence of any application-level throttle.

### Likelihood Explanation
No authentication, API key, or session token is required. The Rosetta `/account/balance` endpoint is publicly documented and intentionally externally accessible. The exploit requires only an HTTP client capable of sending POST requests in a loop ŌĆö no special knowledge, credentials, or protocol manipulation. The Traefik middleware is absent in many deployments (bare-metal, direct Docker, or Kubernetes without `global.middleware: true`), and even when present it is bypassable by accessing the service port directly or by distributing requests across multiple source IPs to stay under the per-IP in-flight limit while still saturating the shared per-host rate limit.

### Recommendation
1. **Application-level caching:** Cache the result of `RetrieveLatest()` with a short TTL (e.g., 1ŌĆō2 seconds, matching the block production interval) using a `sync.Mutex`-protected in-memory cache in `blockRepository`. This eliminates redundant DB queries for the common case.
2. **Application-level rate limiting:** Add a token-bucket or sliding-window rate limiter as an HTTP middleware in `rosetta/main.go`, applied before routing, to cap requests per source IP regardless of Traefik deployment.
3. **Make Traefik middleware non-optional:** Change the Helm chart default so `global.middleware` is `true` by default, and switch `rateLimit.sourceCriterion` from `requestHost` to `ipStrategy` so limits are enforced per source IP rather than globally.
4. **Remove or reduce the retry middleware:** `retry: attempts: 3` amplifies load on the backend; it should be removed or limited to network-error-only retries.
