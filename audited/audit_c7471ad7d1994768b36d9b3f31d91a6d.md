### Title
Unprivileged Historical Block Requests Bypass Cache and Saturate DB I/O via Uncached State Queries

### Summary
Any unprivileged caller can submit `eth_call` requests specifying an arbitrary historical block number. The `callContract()` method accepts any valid block without age restriction, and all historical state queries (`findStorageByBlockTimestamp`, `findActiveByIdAndTimestamp`, etc.) are explicitly uncached — unlike their current-state counterparts. The global gas/RPS throttle is calibrated against EVM gas cost, not DB I/O cost, so an attacker can sustain the maximum allowed request rate while forcing the database to execute expensive, uncached historical scans on every request, degrading service for all concurrent users.

### Finding Description

**Code path:**

`ContractCallService.callContract()` (lines 100–107) sets a lazy `blockSupplier` from the user-supplied `params.getBlock()` with no restriction on block age:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock())
        .orElseThrow(BlockNumberNotFoundException::new)));
return doProcessCall(params, params.getGas(), false);
```

`BlockType.of()` accepts any non-negative `long` (including `0` / `"earliest"`), so block 1 is as valid as the latest block.

**Root cause — no caching for historical queries:**

`ContractStateRepository.findStorage()` carries `@Cacheable`:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_CONTRACT_STATE)
Optional<byte[]> findStorage(final Long contractId, final byte[] key);
```
`ContractStateRepository.findStorageByBlockTimestamp()` has **no** `@Cacheable`:
```java
Optional<byte[]> findStorageByBlockTimestamp(long id, byte[] slot, long blockTimestamp);
```
The same pattern holds for `EntityRepository.findActiveByIdAndTimestamp()`, `findActiveByEvmAddressOrAliasAndTimestamp()`, and `AccountBalanceRepository.findHistoricalAccountBalanceUpToTimestamp()` — all historical variants are uncached.

The per-request `readCache` inside `ContractCallContext` is scoped to a single `ScopedValue` run and is never shared across requests, so there is no cross-request deduplication for historical lookups.

**Root cause — throttle measures EVM gas, not DB I/O:**

`ThrottleManagerImpl.throttle()` deducts `scaleGas(request.getGas())` tokens from the gas bucket before execution. After execution, `restoreGasToBucket()` refunds `min(gasRemaining, gasLimitToRestoreBaseline)` — with the default `gasLimitRefundPercent = 100`, the net deduction equals only `gasUsed` (EVM gas). A simple `eth_call` that reads a handful of storage slots uses ~21 000 + a few hundred gas, yet triggers multiple uncached DB queries per slot access. The throttle sees a tiny gas cost; the database sees a full table scan.

**No per-IP rate limiting:** The `rateLimitBucket` and `gasLimitBucket` are global singletons. A single attacker can consume the entire 500 RPS budget with historical requests, leaving no headroom for legitimate users.

### Impact Explanation
At 500 RPS (the default global limit), an attacker can force the database to execute up to 500 × N uncached historical queries per second (where N is the number of state accesses per call). Because historical queries scan `contract_state_change`, `entity_history`, and `account_balance` tables without cache hits, sustained attack traffic saturates DB I/O and connection pool capacity, causing latency spikes and timeouts for all concurrent users. There is no economic damage to on-chain state; the impact is purely service degradation (griefing).

### Likelihood Explanation
No authentication, no API key, and no per-IP limit are required. Any public internet client can send `POST /api/v1/contracts/call` with `"block": "0x1"` and a valid contract address. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public API specification. The only barrier is the global 500 RPS cap, which the attacker can fully occupy.

### Recommendation
1. **Cache historical state queries** — apply `@Cacheable` (with a short TTL keyed on `(contractId, slot, blockTimestamp)`) to `findStorageByBlockTimestamp` and equivalent historical repository methods, or introduce a shared per-timestamp read cache in `ContractCallContext` that survives across requests for the same block.
2. **Apply a separate, lower rate limit for historical requests** — detect `params.getBlock() != BlockType.LATEST` in `callContract()` and deduct from a dedicated historical-request bucket with a lower capacity (e.g., 50 RPS).
3. **Add per-IP rate limiting** at the ingress/filter layer so a single source cannot exhaust the global budget.
4. **Optionally restrict maximum block age** — reject requests for blocks older than a configurable threshold (e.g., 30 days) to bound the historical data surface.

### Proof of Concept
```bash
# Send 500 concurrent eth_call requests targeting block 1 (earliest),
# each reading a storage slot from a known contract.
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "block": "0x1",
      "to": "0x0000000000000000000000000000000000000167",
      "data": "0x<sload-selector>",
      "gas": 50000
    }' &
done
wait
# Observe: DB query latency increases; concurrent non-historical calls time out.
# Each request bypasses the contract_state cache and issues a full
# contract_state_change scan with consensus_timestamp <= <block-1-timestamp>.
```
