### Title
Unbounded pg-pool Queue Exhaustion via Unauthenticated Flood of `getAccountTokenAllowances()` During Network Partition

### Summary
The REST API's `/api/v1/accounts/{id}/allowances/tokens` endpoint has no rate limiting and uses a `pg` connection pool with an unbounded internal wait queue and a default maximum of 10 connections. During a network partition between the mirror node and the database, all pool connections become stuck indefinitely (TCP appears alive, server-side `statement_timeout` never fires), and any unauthenticated attacker can flood the endpoint to fill the unbounded queue, causing memory exhaustion and Node.js event-loop starvation that crashes or permanently degrades the service.

**Pool configuration** (`rest/dbpool.js:7-16`): [5](#0-4) 

The pool is configured with:
- `max: 10` (default `hiero.mirror.rest.db.pool.maxConnections`)
- `connectionTimeoutMillis: 20000` (timeout to *acquire* a slot from the pool queue)
- `statement_timeout: 20000` (PostgreSQL server-side; **never fires during a network partition** because the query never reaches the server)
- **No `maxQueue` / queue size limit** — `pg`'s internal wait queue is unbounded by default

**Root cause and failed assumption:**

The design assumes that `connectionTimeoutMillis` (20 s) bounds the damage: queued requests eventually fail. This assumption fails under two conditions that combine here:

1. During a network partition, the 10 pool connections are stuck in `this.query()` indefinitely. The TCP session appears alive (no RST), so `pg` never reclaims those connections. `statement_timeout` is server-side and never executes. The connections are held until OS-level TCP keepalive fires (default: minutes to hours).

2. Because there is **no rate limiting** on the REST API for this endpoint (the `ThrottleManagerImpl` / bucket4j throttle exists only in the `web3` module, not in `rest`), an unauthenticated attacker can issue requests at arbitrary concurrency.

With all 10 connections stuck, every new `pool.query()` call is placed into `pg`'s internal unbounded queue. Each queued call holds a live Promise, a closure, and associated V8 heap. At 1,000 req/s the attacker accumulates ~20,000 queued Promises within the 20-second timeout window. Node.js's single-threaded event loop is overwhelmed by the callback storm when timeouts eventually fire en masse, and heap memory grows until the process OOMs or becomes unresponsive.

### Impact Explanation

- **Memory exhaustion / OOM crash**: Each queued `pg` request holds a Promise chain and closure. Thousands of concurrent queued requests exhaust the Node.js heap.
- **Event-loop starvation**: When the 20-second `connectionTimeoutMillis` fires for thousands of queued requests simultaneously, the event loop is flooded with error callbacks, blocking all other processing including health checks.
- **Denial of service to all legitimate users**: The pool is shared across all REST endpoints; exhausting it blocks every API, not just token allowances.
- **Severity: High** — complete service unavailability, no authentication required, trivially repeatable.

### Likelihood Explanation

- **No authentication required**: The endpoint is fully public.
- **No rate limiting**: The `rest` module has no per-IP, per-endpoint, or global request-rate throttle. The only throttle in the codebase (`ThrottleManagerImpl`) is scoped to `web3` contract calls.
- **Network partitions are realistic**: Cloud/Kubernetes environments experience transient DB connectivity loss regularly.
- **Trivially scriptable**: A simple `ab -n 100000 -c 500` or equivalent is sufficient.
- **Repeatable**: The attacker does not need to sustain the partition; they only need to trigger it (or wait for one) and then flood.

### Recommendation

1. **Bound the pg-pool queue**: Set `pg`'s `maxQueue` (or equivalent) to a small multiple of `max` (e.g., `50`). Requests beyond the queue limit should be rejected immediately with HTTP 503.
2. **Add a REST-layer rate limiter**: Apply a per-IP or global request-rate limit (e.g., via `express-rate-limit`) to the REST API, mirroring the throttle already present in the `web3` module.
3. **Add a client-side query timeout**: Wrap `pool.query()` with a `Promise.race()` against a short timeout (e.g., 5 s) so stuck connections are detected and released client-side, independent of the server-side `statement_timeout`.
4. **Configure TCP keepalive** on the pg connections so the OS detects dead connections within seconds rather than minutes.

### Proof of Concept

**Preconditions:**
- Mirror node REST API is running and reachable.
- Network partition between mirror node and PostgreSQL is induced (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the mirror node host, or a network policy change in Kubernetes).

**Steps:**

```bash
# Step 1: Induce network partition (on mirror node host or via network policy)
iptables -I OUTPUT -p tcp --dport 5432 -j DROP

# Step 2: Flood the unauthenticated endpoint with concurrent requests
# (no credentials needed)
ab -n 500000 -c 2000 \
  'http://<mirror-node-host>:5551/api/v1/accounts/0.0.1001/allowances/tokens'

# Step 3: Observe
# - All 10 pool connections are stuck (no DB response)
# - pg internal queue grows unboundedly
# - Node.js heap climbs until OOM or process becomes unresponsive
# - Health check at /health/readiness also stops responding
# - After ~20s, mass timeout callbacks fire, spiking CPU to 100%
```

**Expected result:** The mirror node REST process crashes with an out-of-memory error or becomes permanently unresponsive, denying service to all legitimate users.
