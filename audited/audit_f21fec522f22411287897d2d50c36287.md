### Title
Unauthenticated `getNodes()` gRPC Calls Exhaust Shared HikariCP Connection Pool, Partitioning `subscribeTopic` from the Database

### Summary
The `NetworkService.getNodes()` gRPC endpoint is publicly accessible with no authentication, no per-IP rate limiting, and no per-caller connection quota. Each concurrent call acquires JDBC connections from the single shared HikariCP pool (default size 10) via `transactionOperationsReadOnly` in `NetworkServiceImpl.page()`. An attacker opening a small number of gRPC connections (each with up to 5 concurrent calls per `maxConcurrentCallsPerConnection`) can saturate the pool, causing `subscribeTopic` and all other DB-dependent operations to block or fail with connection timeout errors.

### Finding Description

**Exact code path:**

`GrpcConfiguration.transactionOperationsReadOnly()` creates a read-only `TransactionTemplate` backed by the single shared `HikariDataSource` (configured via `spring.datasource.hikari` in `CommonConfiguration`): [1](#0-0) 

This `TransactionOperations` bean is injected into `NetworkServiceImpl`: [2](#0-1) 

`NetworkServiceImpl.getNodes()` first makes **three synchronous blocking DB calls** outside any transaction (each acquires and releases a pool connection): [3](#0-2) 

Then it enters a paging loop where each iteration calls `page()`, which calls `transactionOperations.execute()` — acquiring a JDBC connection, executing the query, and releasing it: [4](#0-3) 

The `NetworkController.getNodes()` handler has **no authentication, no rate limiting, and no per-caller quota**: [5](#0-4) 

The only gRPC-level interceptor sets an endpoint context for table usage tracking — no throttling: [6](#0-5) 

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module, not in the `grpc` module.

The `maxConcurrentCallsPerConnection` is set to **5** per gRPC connection: [7](#0-6) 

This limits concurrency per connection, but there is no limit on the number of gRPC connections an attacker can open.

**Root cause:** The failed assumption is that `getNodes()` callers are few and well-behaved. The design shares one HikariCP pool (default `maximumPoolSize=10`) between `getNodes()` paging transactions and `subscribeTopic` entity-existence checks (`entityRepository.findById()`): [8](#0-7) [9](#0-8) 

### Impact Explanation

When the HikariCP pool is exhausted, any new call requiring a DB connection — including `subscribeTopic` topic-existence checks and historical message retrieval — blocks for up to `connectionTimeout` (HikariCP default: 30 seconds) then throws `SQLTransientConnectionException`. This effectively partitions the gRPC service from the database for all active and incoming HCS topic subscribers, breaking the core consensus subscription functionality. The `GrpcHighDBConnections` alert only fires after 5 minutes at >75% utilization, giving the attacker a wide window. [10](#0-9) 

### Likelihood Explanation

The attack requires zero privileges. The gRPC port (5600) is publicly exposed. An attacker needs only a standard gRPC client (e.g., `grpcurl`) and two open connections to generate 10 concurrent `getNodes()` calls (5 per connection × 2 connections), saturating the default pool of 10. The attack is trivially repeatable and scriptable. The endpoint is documented publicly: [11](#0-10) 

### Recommendation

1. **Add a global concurrent-call limit** for `getNodes()` at the gRPC server level (e.g., a semaphore or Bucket4j rate limiter in a `ServerInterceptor`, similar to `ThrottleConfiguration` in the `web3` module).
2. **Isolate connection pools**: Use a separate, size-bounded `DataSource` for `getNodes()` paging transactions so that address book queries cannot starve the pool used by `subscribeTopic`.
3. **Add per-IP rate limiting** at the gateway/load-balancer level (the existing `maxRatePerEndpoint: 250` in the GCP backend policy applies to the whole service, not per-method). [12](#0-11) 
4. **Enforce a minimum `pageDelay`** that is long enough to ensure connections are not held concurrently across many callers.

### Proof of Concept

```bash
# Open 2 gRPC connections, each with 5 concurrent getNodes calls (10 total = default pool size)
for i in $(seq 1 10); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done

# Simultaneously attempt a subscribeTopic call — it will block/fail with connection timeout
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: UNAVAILABLE or long hang (>30s) due to HikariCP pool exhaustion
```

```java
    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
```

```java
    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
```
