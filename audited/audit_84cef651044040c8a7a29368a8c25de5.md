### Title
Integer Narrowing Cast in `PollingTopicListener.poll()` Bypasses `maxPageSize` Guard, Enabling Unlimited DB Query

### Summary
In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` performs an unsafe narrowing cast from `long` to `int`. Any unprivileged user who subscribes with a `limit` value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) causes the cast to produce a negative integer. `Math.min(negativeValue, maxPageSize)` then returns the negative value, which is stored as the new filter's `limit`. Because `hasLimit()` checks `limit > 0`, the negative limit is treated as "no limit," causing `findByFilter()` to execute a database query with no `LIMIT` clause and load the entire result set into memory via `getResultList()`.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, line 56:
```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```
The proto field `uint64 limit` accepts any 64-bit value. `query.getLimit()` is passed directly to the filter with no upper-bound check.

`TopicMessageFilter` — `grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java`, lines 25–26:
```java
@Min(0)
private long limit;
```
Only a lower-bound (`@Min(0)`) constraint exists. `Long.MAX_VALUE` passes validation.

`TopicMessageServiceImpl.incomingMessages()` — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 115–118:
```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
// With filter.getLimit()=Long.MAX_VALUE and count=0: limit = Long.MAX_VALUE
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter)...
```
The adjusted filter with `limit = Long.MAX_VALUE` is forwarded to `PollingTopicListener`.

**Root cause — `PollingTopicListener.poll()`** — `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 54–59:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        //  (int)(Long.MAX_VALUE - 0L) = -1  ← narrowing overflow
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
// Math.min(-1, 5000) = -1  ← maxPageSize guard bypassed
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
// newFilter.limit = -1
```

**Why the downstream check fails — `TopicMessageRepositoryCustomImpl.findByFilter()`** — `grpc/src/main/java/org/hiero/mirror/grpc/repository/TopicMessageRepositoryCustomImpl.java`, lines 51–53:
```java
if (filter.hasLimit()) {          // hasLimit() = (-1 > 0) = false → branch skipped
    typedQuery.setMaxResults(...);
}
```
`setMaxResults` is never called. Line 60 then executes `typedQuery.getResultList().stream()`, loading the entire unbounded result set into JVM heap memory.

**Why the outer `flux.take()` does not mitigate:** `TopicMessageServiceImpl.subscribeTopic()` line 84 calls `flux.take(Long.MAX_VALUE)`, which in Project Reactor means "take up to Long.MAX_VALUE elements" — effectively no limit. The damage (full table scan + heap load) occurs inside the repository before any reactive back-pressure can act.

### Impact Explanation
When the POLL listener type is configured (`hiero.mirror.grpc.listener.type=POLL`), a single malicious subscription with `limit=Long.MAX_VALUE` causes the mirror node to execute an unbounded `SELECT` against the `topic_message` table and load the entire result set into heap memory via `getResultList()`. On a busy mainnet topic with millions of messages this can cause an OutOfMemoryError or severe GC pressure, crashing or severely degrading the gRPC service for all subscribers. The poll repeats on `RepeatSpec.times(Long.MAX_VALUE)` (line 40), so the unlimited query fires on every polling interval (default 500 ms) until the subscription is cancelled, amplifying the impact.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The proto field `uint64 limit` accepts the full 64-bit range; setting it to `Long.MAX_VALUE` requires only a standard gRPC client. No special privileges, tokens, or knowledge of internal state are needed. The attack is trivially repeatable from multiple connections simultaneously, and the default `maxConcurrentCallsPerConnection=5` does not prevent opening many connections.

### Recommendation
Replace the unsafe narrowing cast with a bounds-checked conversion before computing `pageSize`:

```java
// In PollingTopicListener.poll()
long remaining = filter.getLimit() - context.getCount().get();
int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE
          : remaining <= 0              ? 0
          : (int) remaining;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

Additionally, add an `@Max` constraint to `TopicMessageFilter.limit` (e.g., `@Max(Integer.MAX_VALUE)`) to reject oversized values at the API boundary, and apply the same fix to the identical pattern in `PollingTopicMessageRetriever.poll()` (line 68–71).

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),  # any existing topic
    limit=9223372036854775807,                     # Long.MAX_VALUE
)

# Each response triggers poll() with (int)(Long.MAX_VALUE - count) = -1
# findByFilter() runs with no LIMIT clause every 500ms
for msg in stub.subscribeTopic(query):
    pass  # server executes unbounded SELECT on every poll interval
```

**Expected result:** The mirror node executes `SELECT * FROM topic_message WHERE topic_id=1 AND consensus_timestamp >= ?` with no `LIMIT` clause on every polling interval, loading the full result set into heap memory, leading to OOM or severe service degradation.
