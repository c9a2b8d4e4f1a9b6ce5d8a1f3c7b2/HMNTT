### Title
Unauthenticated EVM Address Flood Causes Unbounded Database I/O via Uncached Negative Lookups

### Summary
The `getEntityIdFromEvmAddress()` function in `rest/service/entityService.js` issues a database query for every EVM address lookup with no negative-result caching. Because the response cache middleware explicitly excludes non-2xx (404) responses from being stored, every request for a non-existent EVM address unconditionally hits the database. An unauthenticated attacker can flood the public `/api/v1/accounts/{evmAddress}` endpoint with an unbounded stream of unique, syntactically valid 40-hex-character EVM addresses, generating sustained database index I/O and degrading service for all users.

### Finding Description
**Exact code path:**

`rest/service/entityService.js` lines 22–25 define the query:
```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where ${Entity.DELETED} <> true
      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

`getEntityIdFromEvmAddress()` at lines 90–104 executes this query on every call with no in-process or Redis caching:
```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
    [Buffer.from(entityId.evmAddress, 'hex')]);
  if (rows.length === 0) {
    if (requireResult) { throw new NotFoundError(); }
    return null;
  }
  ...
}
``` [2](#0-1) 

**Why the response cache does not help:**

`responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` line 95 only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
``` [3](#0-2) 

A 404 `NotFoundError` is not a success code, so negative results are **never cached**. Every unique non-existent EVM address causes a fresh database round-trip.

**Index lookup, not full scan — but still exploitable:**

A partial index exists on `entity(evm_address)`:
```sql
create index if not exists entity__evm_address on entity (evm_address) where evm_address is not null;
``` [4](#0-3) 

Each lookup is an index seek, not a sequential scan. However, under high concurrency with unique addresses, the cumulative index I/O (buffer pool pressure, WAL activity, connection pool exhaustion) is sufficient to saturate the database.

**No application-level rate limiting in the REST service:**

A search across all `rest/**/*.js` files finds zero rate-limiting logic. The Traefik-based rate limiting in the Helm chart defaults to `global.middleware: false` and is not enforced by default. [5](#0-4) 

The throttling that does exist (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` Java module (contract call endpoints), not the Node.js REST service. [6](#0-5) 

### Impact Explanation
An attacker can continuously degrade or deny service to the mirror node REST API for all users by exhausting the PostgreSQL connection pool and I/O bandwidth. The entity table is central to many API endpoints; saturating it affects accounts, contracts, tokens, and related lookups. Because the attack requires no authentication and no state, it is trivially automatable and can be sustained indefinitely. The 2^160 EVM address space ensures the attacker never repeats an address and the cache never helps.

### Likelihood Explanation
The attack requires zero privileges — only HTTP access to a public endpoint. Generating unique 40-hex-character strings is trivial with any scripting language. The attacker needs no knowledge of the system beyond the public API spec (`/api/v1/accounts/{idOrAliasOrEvmAddress}`). The attack is repeatable, stateless, and can be distributed across multiple source IPs to bypass any upstream IP-based rate limiting.

### Recommendation
1. **Cache negative results**: Introduce a short-lived (e.g., 5–30 second) negative-result cache (Redis or in-process) keyed on the EVM address bytes. This is the most direct fix.
2. **Cache 404 responses at the response layer**: Modify `responseCacheUpdateHandler` to also cache 404 responses for EVM address lookups with a short TTL.
3. **Enable application-level rate limiting**: Add per-IP rate limiting middleware to the Node.js REST server (e.g., `express-rate-limit`) independent of the optional Traefik middleware.
4. **Enable Traefik middleware by default**: Set `global.middleware: true` and configure `inFlightReq` and `rateLimit` in the REST Helm chart values.

### Proof of Concept
```bash
# Generate and flood with unique non-existent EVM addresses
python3 -c "
import os, requests, threading

def flood():
    while True:
        addr = os.urandom(20).hex()  # unique random 40-hex address
        requests.get(f'https://<mirror-node>/api/v1/accounts/0x{addr}',
                     timeout=5)

threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads: t.start()
for t in threads: t.join()
"
```
Each request hits `GET /api/v1/accounts/0x<unique_40_hex>`, which routes through `getEncodedId()` → `getEntityIdFromEvmAddress()` → `entityFromEvmAddressQuery` → PostgreSQL index lookup → 404 (not cached) → next request repeats. Under 200 concurrent threads, the database connection pool and I/O are saturated within seconds, causing timeouts for legitimate users.
