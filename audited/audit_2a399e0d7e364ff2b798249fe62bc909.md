### Title
Missing Entity-Type Validation in EVM Address Resolution Allows Token Entities to Be Used as Account Identifiers in Crypto Allowance Endpoint

### Summary
`EntityService.getEntityIdFromEvmAddress()` queries the `entity` table with no filter on the `type` column, so it resolves any entity — including TOKEN-type entities — to a numeric ID. `getAccountCryptoAllowances()` passes that ID directly as the `owner` SQL parameter without any subsequent type check, causing the endpoint to silently accept a token's EVM address, bind the token's entity ID as the owner condition, and execute the allowance query against it. The endpoint returns HTTP 200 with an empty allowances array instead of HTTP 404, and would expose any `crypto_allowance` rows whose `owner` column equals that token entity ID.

### Finding Description

**Exact code path:**

`rest/service/entityService.js` lines 22–25 — the SQL query used for EVM-address resolution selects only `id` with no `type` predicate:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

`getEntityIdFromEvmAddress()` (lines 90–104) returns `rows[0].id` unconditionally — no check that the matched entity is of type `ACCOUNT` or `CONTRACT`: [2](#0-1) 

`getAccountCryptoAllowances()` (line 77) calls `EntityService.getEncodedId()` and immediately uses the result as `accountId` with no downstream type validation: [3](#0-2) 

That `accountId` is placed as `$1` in `ownerCondition = "owner = $1"` and forwarded to `CryptoAllowanceService.getAccountCryptoAllowances()`: [4](#0-3) 

**Root cause / failed assumption:** The code assumes that any EVM address supplied to the `/accounts/{id}/allowances/crypto` path belongs to an account or contract. The `entity` table stores all Hedera entity types — ACCOUNT, CONTRACT, FILE, TOPIC, TOKEN, SCHEDULE — and the `evm_address` column is present and populated for TOKEN entities: [5](#0-4) 

The GraphQL layer correctly guards against this by filtering on type after the EVM-address lookup (`filter(e -> e.getType() == type)`), but the REST layer has no equivalent guard: [6](#0-5) 

### Impact Explanation

An attacker who supplies a TOKEN entity's EVM address in the path receives HTTP 200 with `{"allowances": [], "links": {"next": null}}` instead of HTTP 404. Two concrete consequences:

1. **Entity-type oracle / enumeration**: The attacker can distinguish "this EVM address belongs to a live entity" (200) from "no such entity" (404), leaking information about which EVM addresses correspond to valid on-chain entities regardless of type.
2. **Latent data exposure**: If any `crypto_allowance` row ever has a token entity ID as `owner` (e.g., due to a data-import bug or future protocol change), those rows would be returned to an unauthenticated caller. The `amount > 0` filter is the only guard, and it is a data-value filter, not an access-control filter. [7](#0-6) 

### Likelihood Explanation

Preconditions are trivially met: token EVM addresses are public on-chain data, no authentication is required, and the endpoint is unauthenticated by design. Any external user can enumerate token EVM addresses from the `/api/v1/tokens` endpoint and replay them against `/api/v1/accounts/{token_evm_address}/allowances/crypto`. The attack is repeatable and requires no special privileges.

### Recommendation

Add an entity-type check immediately after `getEncodedId()` in `getAccountCryptoAllowances()`, or add a `type in ('ACCOUNT', 'CONTRACT')` predicate to `entityFromEvmAddressQuery` (mirroring the existing filter in `accounts.js` line 145): [8](#0-7) 

Alternatively, introduce a dedicated `getAccountOrContractIdFromEvmAddress()` method that rejects non-account/contract entities with a `NotFoundError`, consistent with how the GraphQL `EntityServiceImpl.getByEvmAddressAndType()` enforces the type constraint.

### Proof of Concept

1. Query `/api/v1/tokens` to obtain a token's EVM address, e.g. `0xaabbccdd...` (40 hex chars).
2. Send: `GET /api/v1/accounts/0xaabbccdd.../allowances/crypto`
3. Internally: `EntityService.getEncodedId("0xaabbccdd...")` → `getEntityIdFromEvmAddress()` → SQL executes against `entity` table with no type filter → returns the token's numeric `id` (e.g. `1234`).
4. `extractCryptoAllowancesQuery` builds `owner = $1` with `params = [1234]`.
5. `CryptoAllowanceService.getAccountCryptoAllowances()` executes `SELECT * FROM crypto_allowance WHERE owner = 1234 AND amount > 0`.
6. Response: HTTP 200 `{"allowances": [], "links": {"next": null}}` — no 404, no error, confirming the token entity ID was accepted as a valid account identifier.

    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
