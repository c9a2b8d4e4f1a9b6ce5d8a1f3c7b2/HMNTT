### Title
Unbounded Per-Subscriber Cold-Publisher DB Polling Enables Unauthenticated Database Amplification DoS

### Summary
`PollingTopicMessageRetriever.retrieve()` is a cold Reactor publisher: every call creates an independent `PollingContext` and a `Flux.defer(() -> poll(context))` repeat loop that issues its own database queries. Because `TopicMessageServiceImpl.subscribeTopic()` calls `retrieve()` once per inbound gRPC subscription with no subscriber cap, an unprivileged attacker who opens N concurrent subscriptions to the same `topicId` drives N independent, unsynchronised DB polling loops, multiplying database query rate by N. No sharing, rate-limiting, or connection cap exists in the gRPC module to prevent this.

### Finding Description

**Exact code path**

`PollingTopicMessageRetriever.retrieve()` — `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 44–63:

```java
PollingContext context = new PollingContext(filter, throttled);   // line 50 – new per call
return Flux.defer(() -> poll(context))                            // line 51 – cold, no sharing
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())           // default 2 s (throttled)
                .withScheduler(scheduler))
        ...
        .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))  // line 58
        .timeout(retrieverProperties.getTimeout(), scheduler)             // line 59 – 60 s
```

`TopicMessageServiceImpl.subscribeTopic()` — lines 59–92 — calls `retrieve()` unconditionally for every inbound subscription:

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);  // line 63
```

The `subscriberCount` field (line 48) is wired only to a Micrometer `Gauge` (lines 52–55); it is never checked against any ceiling before accepting a new subscription.

**Root cause and failed assumption**

The design assumes subscriber volume is bounded by legitimate operational use. Unlike the live-message path (`SharedPollingTopicListener`, which calls `.share()` on a single shared Flux so all subscribers receive results from one DB poll), the historical retriever is intentionally a cold publisher — but no compensating limit on concurrent subscriptions was added. Each `retrieve()` call produces a fully independent polling loop that issues `topicMessageRepository.findByFilter(newFilter)` (line 78) on every repeat tick.

**Why existing checks fail**

| Check | Why insufficient |
|---|---|
| `subscriberCount` gauge | Observability only; no enforcement |
| `timeout(60 s)` | Limits one subscription's lifetime; attacker re-opens connections continuously |
| `Retry.backoff(Long.MAX_VALUE, 1 s)` | During a partition, each of N loops retries independently, sustaining N concurrent DB connection attempts |
| `throttled` flag | Controls page size / frequency within one subscription; does not limit subscriber count |
| Web3 `ThrottleManagerImpl` | Exists only in the `web3` module; no equivalent exists in the `grpc` module |

### Impact Explanation
With N concurrent subscriptions, the DB receives N × (1 query per `pollingFrequency` interval) = N × 0.5 queries/second (throttled, 2 s interval) for historical retrieval alone. At N = 500 that is 250 DB queries/second from a single attacker IP, sustained for up to 60 seconds per wave. During a network partition the `Retry.backoff` loop keeps each subscription alive and retrying, so the DB is hit with N concurrent reconnect/query bursts. Because `findByFilter` executes a range scan on `topic_message` keyed by `(topic_id, consensus_timestamp)`, each query is non-trivial. This can exhaust DB connection pools, degrade or deny service for legitimate users, and cause cascading failures in the importer pipeline.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge, and no prior access — only the ability to open gRPC streams to the public mirror-node endpoint. The gRPC `subscribeTopic` RPC is the primary public API of the mirror node. Tooling to open thousands of concurrent gRPC streams is trivially available (`ghz`, `grpcurl` in a loop, custom scripts). The attack is repeatable and stateless: each wave of subscriptions is independent. Real-world HCS mirror nodes are publicly reachable, making this a realistic threat.

### Recommendation
1. **Enforce a global and per-IP subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-topic subscriber limiting** so a single `topicId` cannot monopolise the pool.
3. **Share historical retrieval results** for identical filters (same `topicId` + `startTime` window) using a `Sinks.Many` or `Flux.share()` pattern analogous to `SharedPollingTopicListener`, so N subscribers with the same filter drive only one DB polling loop.
4. **Add gRPC-layer rate limiting** (e.g., via a `ServerInterceptor` using Bucket4j) mirroring the `ThrottleManagerImpl` already present in the `web3` module.

### Proof of Concept

```bash
# Open 500 concurrent subscriptions to the same topicId from a single client
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1000},"consensusStartTime":{"seconds":0}}' \
    mirror-node:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

**Expected result**: The DB receives ~250 `SELECT … FROM topic_message WHERE topic_id = 1000 AND consensus_timestamp >= … LIMIT 1000` queries per second. DB CPU and connection-pool utilisation spike proportionally to N. Legitimate subscribers experience query timeouts or connection-pool exhaustion errors. During a simulated network partition (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the mirror node host), each of the 500 `Retry.backoff` loops fires independently, sustaining 500 concurrent reconnect attempts against the DB for the full 60-second timeout window.
