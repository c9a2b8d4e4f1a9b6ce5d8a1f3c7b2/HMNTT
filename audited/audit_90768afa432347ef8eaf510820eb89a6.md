### Title
Pre-Throttle Heap Allocation via Oversized JSON `data` Field in Jackson Parser

### Summary
The `jacksonCustomizer` in `JacksonConfiguration.java` sets `maxStringLength` and `maxDocumentLength` to `ethereumMaxCallDataSize() * 2 + 1024` (~257 KB). Spring MVC deserializes the full `@RequestBody` — allocating a Java heap string of up to that size — **before** the controller invokes `throttleManager.throttle(request)`. An unprivileged attacker sending many concurrent requests with a `data` field at the maximum allowed size can force large heap allocations on every in-flight thread before any rate-limiting check fires.

### Finding Description
**Code path:**

`JacksonConfiguration.java` line 22–29: `maxSize = jumboConfig.ethereumMaxCallDataSize() * 2 + 1024` (≈ 263 168 bytes with the default 128 KiB `ethereumMaxCallDataSize`). Both `maxStringLength` and `maxDocumentLength` are set to this value.

`ContractController.java` line 38–40: Spring MVC calls `@RequestBody` deserialization (Jackson) **before** the method body executes. `throttleManager.throttle(request)` is the first line inside the method — it runs only after Jackson has already fully parsed and allocated the `data` string.

**Root cause:** The Jackson `StreamReadConstraints` limits are the only pre-throttle guard. They bound the *per-request* allocation to `maxSize` characters, but they do not prevent *concurrent* allocation across all in-flight threads. The application-level rate limiter (`rateLimitBucket`, default 500 req/s) is invoked post-parse and therefore cannot prevent the heap allocation.

**Exploit flow:**
1. Attacker opens as many concurrent TCP connections as the Tomcat thread pool allows (default 200 worker threads).
2. Each connection sends `POST /api/v1/contracts/call` with `{"data":"0x<hex string of ~263 168 chars>", ...}`.
3. Jackson reads and allocates a Java `String` of ~263 168 chars (≈ 512 KB in UTF-16 heap) per thread before the controller method is entered.
4. With 200 concurrent threads: 200 × 512 KB ≈ 100 MB of live heap just for `data` strings, plus Jackson's internal `TextBuffer` copies.
5. Throttle check fires and rejects the request, but the allocation has already occurred and GC must reclaim it under continuous load.
6. Sustained flood (thousands of requests/second from multiple source IPs) keeps GC under constant pressure, degrading throughput or causing OOM.

**Why existing checks fail:**
- `maxStringLength(maxSize)` / `maxDocumentLength(maxSize)` reject oversized input but only *after* allocating up to `maxSize` bytes — the check is a post-allocation guard, not a pre-allocation one.
- No HTTP-layer body-size limit (`server.tomcat.max-swallow-size`, `spring.servlet.multipart.max-file-size`) is configured anywhere in the repository.
- The `rateLimitBucket` (500 req/s) and `gasLimitBucket` are checked inside the controller method, after deserialization is complete.
- No servlet `Filter` performs rate-limiting or body-size rejection before the Spring dispatcher reads the body.

### Impact Explanation
Each concurrent request forces ~512 KB of heap allocation before any rejection. With the default Tomcat thread pool (200 threads) fully saturated, ~100 MB of heap is consumed by `data` strings alone, plus Jackson internal buffers. Under a sustained flood from multiple source IPs (bypassing per-IP limits), GC pressure accumulates, response latency degrades, and the JVM may reach OOM — effectively denying service to legitimate callers of the `/api/v1/contracts/call` endpoint. This maps to the stated scope: shutdown of ≥30% of network processing nodes without brute force.

### Likelihood Explanation
No authentication or API key is required to call `/api/v1/contracts/call`. The endpoint is publicly documented. An attacker needs only a standard HTTP client capable of sending concurrent requests with large bodies. The `maxSize` (~257 KB) is large enough to make each request expensive, and the absence of any HTTP-layer body-size enforcement means the full allocation occurs on every request. This is repeatable and automatable with commodity tooling (e.g., `wrk`, `ab`, `curl` in parallel).

### Recommendation
1. **Add an HTTP-layer body-size limit** before Jackson parsing, e.g. in `application.yml`:
   ```yaml
   server:
     tomcat:
       max-swallow-size: 300KB   # slightly above legitimate max
   ```
   Or configure a `OncePerRequestFilter` that rejects requests with `Content-Length` exceeding `maxSize` before the dispatcher reads the body.
2. **Move rate-limiting to a servlet filter** (ordered before Spring's `DispatcherServlet`) so the `rateLimitBucket` check fires before body deserialization.
3. **Reduce `maxStringLength`** to the actual maximum legitimate `data` field size (the hex-encoded call data, not `ethereumMaxCallDataSize * 2 + 1024`).

### Proof of Concept
```bash
# Generate a ~263 KB hex string (maxSize characters)
python3 -c "print('0x' + 'ab' * 131072)" > /tmp/bigdata.txt
DATA=$(cat /tmp/bigdata.txt)

# Send 200 concurrent requests (saturate default Tomcat thread pool)
for i in $(seq 1 200); do
  curl -s -X POST https://<target>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"$DATA\",\"to\":\"0x0000000000000000000000000000000000000168\",\"gas\":15000000,\"estimate\":false}" &
done
wait

# Repeat in a tight loop from multiple source IPs to sustain GC pressure
# Monitor JVM heap via /actuator/metrics/jvm.memory.used
```
Each iteration forces ~512 KB of heap allocation per thread before the throttle check fires, sustaining GC pressure and degrading or crashing the service under sufficient concurrency.
