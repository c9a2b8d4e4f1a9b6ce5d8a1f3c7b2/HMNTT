### Title
Unauthenticated DoS via Unrate-Limited Three-Subquery UNION ALL in `getNfts()`

### Summary
The REST API endpoint `/api/v1/accounts/{id}/nfts` invokes `NftService.getNfts()`, which can generate a three-subquery `UNION ALL` SQL statement when both lower and upper bounds are supplied for `token.id` and `serialnumber`. No per-IP rate limiting or in-flight request cap exists at the REST application or ingress layer, allowing any unauthenticated attacker to flood the shared PostgreSQL backend with concurrent, long-running compound queries, exhausting DB CPU and connection capacity across all components sharing that backend.

### Finding Description

**Exact code path:**

`rest/service/nftService.js`, `getQuery()`, lines 93–122: when `lower`, `inner`, and `upper` filter arrays are all non-empty, three subqueries are assembled and joined:

```js
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
``` [1](#0-0) 

Each subquery is a `SELECT … FROM nft LEFT JOIN entity …` with multiple filter conditions. [2](#0-1) 

The three-subquery path is triggered from `AccountController.getNftsByAccountId` → `extractNftMultiUnionQuery` → `NftService.getNfts()` with no authentication gate. [3](#0-2) 

The route is registered publicly with no middleware guard: [4](#0-3) 

**Root cause — missing rate limiting:**

`server.js` registers no rate-limiting or in-flight-request middleware for any REST route. [5](#0-4) 

The Helm chart for the REST service configures only `circuitBreaker` and `retry` Traefik middlewares — no `rateLimit` and no `inFlightReq`: [6](#0-5) 

By contrast, the Rosetta service explicitly configures both `inFlightReq: amount: 5` per IP and `rateLimit: average: 10` per host: [7](#0-6) 

A grep across `rest/**` for `rateLimit`, `rateLimiter`, `throttle`, or `inFlightReq` returns zero application-level hits.

**Why existing checks are insufficient:**

- `statement_timeout = 20000 ms` (20 s) limits individual query duration but does not prevent an attacker from maintaining a continuous stream of new queries that each consume a full connection slot for up to 20 s. [8](#0-7) 

- The default DB pool is `maxConnections: 10` per REST node instance; PgBouncer caps `mirror_rest` at `max_user_connections: 250`. With 250 concurrent 20-second queries each executing three full `nft LEFT JOIN entity` scans, the DB CPU is continuously saturated. [9](#0-8) [10](#0-9) 

- The `circuitBreaker` trips only on `NetworkErrorRatio > 0.25` or `5xx > 0.25`; CPU-saturating queries that still return 200 OK do not trip it. [11](#0-10) 

- The GCP gateway `maxRatePerEndpoint: 250` is a global cap across all clients, not a per-IP limit, so a single attacker can consume the entire budget. [12](#0-11) 

### Impact Explanation

The `nft` and `entity` tables grow proportionally with network activity. Three concurrent full-scan subqueries with a `LEFT JOIN` per request, multiplied across 250 simultaneous connections, produce sustained high CPU on the shared PostgreSQL backend. Because the importer, gRPC, and REST components all share the same database cluster, DB CPU exhaustion degrades or halts block ingestion and query serving across all mirror node components simultaneously — consistent with the ≥30% processing-node impact threshold. The REST API itself becomes unavailable to legitimate users for the duration of the attack.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only four standard query parameters (`token.id=gte:1&token.id=lte:999999999&serialnumber=gt:0&serialnumber=lt:999999999`). It is trivially scriptable with `curl` or any HTTP load tool (the repository itself ships a k6 load-test script targeting this exact endpoint). The attacker needs only a single machine with moderate outbound bandwidth. The attack is repeatable and indefinitely sustainable. [13](#0-12) 

### Recommendation

1. **Add per-IP `inFlightReq` and `rateLimit` Traefik middlewares** to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta configuration (e.g., `inFlightReq: amount: 5` per IP, `rateLimit: average: 100` per host).
2. **Add application-level throttling** in `server.js` using `express-rate-limit` or equivalent, scoped to `/api/v1/accounts/*/nfts`.
3. **Reduce `statement_timeout`** for the `mirror_rest` DB user from 20 s to a value appropriate for paginated NFT queries (e.g., 5 s).
4. **Cap `max_user_connections`** for `mirror_rest` in PgBouncer to a value that prevents full DB CPU saturation even if all slots are used.

### Proof of Concept

```bash
# Trigger the three-subquery UNION ALL path on any public mirror node
# Replace BASE_URL and ACCOUNT_ID with valid values

BASE_URL="https://<mirror-node>/api/v1"
ACCOUNT_ID="0.0.1001"

# Single request that triggers three-subquery path:
curl -s "${BASE_URL}/accounts/${ACCOUNT_ID}/nfts?token.id=gte:0.0.1&token.id=lte:0.0.9999999999&serialnumber=gt:0&serialnumber=lt:9999999999&limit=100"

# Sustained flood (no authentication required):
for i in $(seq 1 500); do
  curl -s "${BASE_URL}/accounts/${ACCOUNT_ID}/nfts?token.id=gte:0.0.1&token.id=lte:0.0.9999999999&serialnumber=gt:0&serialnumber=lt:9999999999&limit=100" &
done
wait
```

The four query parameters (`token.id=gte`, `token.id=lte`, `serialnumber=gt`, `serialnumber=lt`) populate all three of `lower`, `inner`, and `upper` in `getQuery()`, guaranteeing the `UNION ALL` branch at line 122 of `nftService.js` on every request. No account ownership, API key, or session token is required.
