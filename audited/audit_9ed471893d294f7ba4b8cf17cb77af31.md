### Title
404 Response Cache Poisoning via `Cache-Control: public, max-age=5` on `/api/v1/topics/{id}` Not-Found Responses

### Summary
The `@ModelAttribute` interceptor in `GenericControllerAdvice` unconditionally sets `Cache-Control: public, max-age=5` on the `HttpServletResponse` for all requests matching `/api/v1/topics/{id}`, including those that result in a 404. Because `TopicServiceImpl.findById()` throws `EntityNotFoundException` for missing topics and the exception handler does not clear the already-written `Cache-Control` header, the 404 response is emitted with `Cache-Control: public, max-age=5`. Any shared reverse proxy or CDN that respects this header will cache the 404 for up to 5 seconds, causing legitimate users to receive stale not-found errors for newly created topics during that window.

### Finding Description

**Code path:**

1. `TopicController.getTopic()` calls `topicService.findById(id.id())`. [1](#0-0) 

2. `TopicServiceImpl.findById()` throws `jakarta.persistence.EntityNotFoundException` when the topic row is absent. [2](#0-1) 

3. Before the controller method is invoked, Spring MVC runs the `@ModelAttribute` method in `GenericControllerAdvice`, which writes `Cache-Control: public, max-age=5` (and other path-specific headers) directly onto the `HttpServletResponse` object. [3](#0-2) 

4. The `notFound` `@ExceptionHandler` then calls `handleExceptionInternal(e, null, null, NOT_FOUND, request)`, which sets the status to 404 and writes the error body, but **does not clear** the `Cache-Control` header already committed to the `HttpServletResponse`. [4](#0-3) 

**Root cause:** The `@ModelAttribute` hook sets caching headers on the raw `HttpServletResponse` before the handler executes. Spring's `ResponseEntityExceptionHandler.handleExceptionInternal` does not strip or override headers already present on the underlying servlet response, so the `Cache-Control: public, max-age=5` header survives on the 404 reply.

**Why existing checks are insufficient:** The Node.js REST layer's `responseCacheUpdateHandler` explicitly skips non-2xx responses (`httpStatusCodes.isSuccess(res.statusCode)`), but the `/api/v1/topics/{id}` endpoint is served by the **Spring Boot rest-java** service, not the Node.js layer. There is no equivalent guard in the Spring layer; no code strips `Cache-Control` from error responses. [5](#0-4) 

The test suite confirms `Cache-Control: public, max-age=5` on successful responses but has no assertion that 404 responses carry a non-cacheable directive. [6](#0-5) 

### Impact Explanation
Any reverse proxy or CDN deployed in front of the mirror node that honours `Cache-Control: public` will cache the 404 for up to 5 seconds. Because Hedera topic IDs are sequentially assigned and publicly observable on-chain, an attacker can predict the next topic ID, pre-fetch it before the `ConsensusCreateTopic` transaction is ingested by the mirror node, and cause the resulting 404 to be cached. All users routed through the same cache node will receive the stale 404 for the duration of the TTL, breaking integrations that poll the endpoint immediately after topic creation. The impact is griefing / availability degradation with no direct economic loss; severity is Medium.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no rate-limit bypass — a single unauthenticated HTTP GET is sufficient. Topic IDs are predictable (monotonically increasing integers visible in the public ledger). The 5-second TTL limits the window per attempt, but the attack is trivially repeatable: the attacker can re-poison the cache every 5 seconds until the topic appears, extending the effective denial window indefinitely. Any deployment behind a shared CDN or reverse proxy (a common production configuration) is affected.

### Recommendation
1. **Override `Cache-Control` to `no-store` (or at minimum `private`) for all non-2xx responses.** Add a response filter or override `handleExceptionInternal` in `GenericControllerAdvice` to explicitly set `Cache-Control: no-store` before writing error responses.
2. **Alternatively**, move the `@ModelAttribute` header-setting logic so it only applies when the response status is 2xx, or apply it as a `HandlerInterceptor.afterCompletion` that checks `response.getStatus()` before writing caching headers.
3. **Short-term mitigation:** Configure the reverse proxy / CDN to never cache 4xx responses regardless of the upstream `Cache-Control` header.

### Proof of Concept

```
# Step 1 – Identify the next topic ID (e.g., 0.0.5000) from the public ledger.

# Step 2 – Before the topic is ingested by the mirror node, send:
GET /api/v1/topics/0.0.5000 HTTP/1.1
Host: <mirror-node-behind-cdn>

# Response (topic not yet in DB):
HTTP/1.1 404 Not Found
Cache-Control: public, max-age=5
Content-Type: application/problem+json
{"type":"about:blank","title":"Not Found","status":404,"detail":"Topic not found"}

# The CDN/reverse proxy caches this 404 for 5 seconds.

# Step 3 – The ConsensusCreateTopic transaction is ingested; topic 0.0.5000 now exists in the DB.

# Step 4 – Within the 5-second TTL, any user queries:
GET /api/v1/topics/0.0.5000 HTTP/1.1
Host: <mirror-node-behind-cdn>

# Response served from cache:
HTTP/1.1 404 Not Found   ← stale, topic actually exists
Cache-Control: public, max-age=<remaining TTL>

# Repeat Step 2 every 5 seconds to extend the denial window.
```
