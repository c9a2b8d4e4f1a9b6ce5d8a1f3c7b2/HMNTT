### Title
Entity Existence Bypass for Numeric AccountIds Enables Unauthenticated DB Query Flooding in `getAirdrops()`

### Summary
When a numeric `EntityIdNumParameter` is supplied, `EntityServiceImpl.lookup()` returns the parsed `EntityId` directly without any database existence check, unconditionally passing it to `TokenAirdropRepositoryCustomImpl.findAll()`. An unauthenticated attacker can flood the endpoint with arbitrary non-existent numeric account IDs, each causing a database query against the `TOKEN_AIRDROP` table, accumulating DB CPU load with no server-side guard.

### Finding Description

**Exact code path:**

In `EntityServiceImpl.lookup()`, the switch on `EntityIdParameter` short-circuits for numeric IDs:

```java
// EntityServiceImpl.java, lines 31-35
var id = switch (accountId) {
    case EntityIdNumParameter p -> Optional.of(p.id());   // ← no DB lookup
    case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
    case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
};
``` [1](#0-0) 

Alias and EVM-address paths hit the entity repository and return `Optional.empty()` for unknown IDs (causing a 404). The numeric path returns `Optional.of(p.id())` unconditionally, so any syntactically valid number (e.g. `0.0.999999999`) is accepted and forwarded.

`TokenAirdropServiceImpl.getAirdrops()` then calls `repository.findAll()` with that unvalidated ID:

```java
// TokenAirdropServiceImpl.java, lines 19-22
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}
``` [2](#0-1) 

`TokenAirdropRepositoryCustomImpl.findAll()` issues a real SQL query against `TOKEN_AIRDROP` for every call:

```java
// TokenAirdropRepositoryCustomImpl.java, lines 58-71
return dslContext
        .selectFrom(TOKEN_AIRDROP)
        .where(condition)
        .orderBy(order)
        .limit(request.getLimit())
        .fetchInto(TokenAirdrop.class);
``` [3](#0-2) 

**Root cause:** The design assumes numeric IDs are always valid/existing entities, but there is no enforcement of that assumption. The entity existence check is only performed for alias and EVM-address forms.

**Why existing checks fail:** `EntityIdNumParameter.valueOf()` only validates the format (regex match), not existence. [4](#0-3) 
No rate limiting, authentication, or existence pre-check is visible in the service or repository layer.

### Impact Explanation
Every request with a non-existent numeric account ID causes a real DB query (filtered by `sender_account_id`/`receiver_account_id` and `STATE = PENDING`). Under high concurrency, this accumulates DB CPU and I/O load on the shared PostgreSQL instance used by both the REST-Java service and the importer. If the DB becomes saturated, the importer's write throughput degrades, causing the mirror node to fall behind the consensus stream. The mirror node itself does not participate in consensus, so a true "network shutdown" is not achievable, but mirror node data staleness and service unavailability are realistic outcomes.

### Likelihood Explanation
No authentication is required. Any external user can craft requests with arbitrary numeric IDs (e.g. `GET /api/v1/accounts/0.0.999999999/airdrops/outstanding`). The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public API format. The only practical barrier is network-level rate limiting or a WAF, neither of which is present in the application code.

### Recommendation
1. **Validate entity existence for numeric IDs** in `EntityServiceImpl.lookup()`: perform an `entityRepository.findById()` check for `EntityIdNumParameter` just as alias/EVM-address paths do, and throw `EntityNotFoundException` if the entity does not exist.
2. **Apply API-level rate limiting** (e.g. Spring's `RateLimiter`, a gateway, or a WAF) on the airdrop endpoints.
3. **Add a DB-level guard**: ensure indexes exist on `sender_account_id` and `receiver_account_id` in `TOKEN_AIRDROP` so that queries for non-existent IDs are cheap index lookups rather than sequential scans.

### Proof of Concept
```bash
# Send 10,000 concurrent requests with non-existent numeric account IDs
seq 1 10000 | xargs -P 200 -I{} \
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.{}/airdrops/outstanding" \
  -o /dev/null

# Each request bypasses entity validation and issues a TOKEN_AIRDROP DB query.
# Monitor DB CPU: watch -n1 "psql -c \"SELECT count(*), state FROM pg_stat_activity GROUP BY state\""
# Observe importer lag increasing as DB is saturated.
```

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
