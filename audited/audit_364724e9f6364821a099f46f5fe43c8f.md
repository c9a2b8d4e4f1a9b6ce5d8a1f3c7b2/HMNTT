### Title
Unauthenticated Alias-Resolution DB Lookup Exhaustion via `/api/v1/balances?account.id=eq:<alias>`

### Summary
The `parseAccountIdQueryParam()` function in `rest/balances.js` unconditionally triggers a live database query via `EntityService.getAccountIdFromAlias()` for every request containing a valid-format account alias. The REST Node.js service has no per-IP or global rate limiting, and the response cache is optional (Redis-dependent). An unprivileged attacker can flood the endpoint with alias-bearing requests, causing repeated uncached DB lookups that exhaust the connection pool and degrade availability for legitimate users.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 331–332, inside `parseAccountIdQueryParam()`:

```js
if (AccountAlias.isValid(value, true) && ++evmAliasAddressCount === 1) {
  return EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false);
}
```

`AccountAlias.isValid(value, true)` (with `noShardRealm=true`) only checks the regex `/^[A-Z2-7]+$/` — any uppercase base32 string passes, including non-existent aliases. [1](#0-0) 

When the check passes, `EntityService.getAccountIdFromAlias()` is called, which calls `getAccountFromAlias()`, which executes a raw DB query against the `entity` table on every invocation — no caching, no memoization: [2](#0-1) 

```js
const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
```

The query is: [3](#0-2) 

```sql
SELECT id FROM entity WHERE coalesce(deleted, false) <> true AND alias = $1
```

**Root cause — failed assumptions:**

1. The code assumes alias resolution is cheap or infrequent. There is no in-process cache for alias lookups in the REST Node.js service (unlike the importer's `EntityIdServiceImpl` which has a `Cache`). [4](#0-3) 

2. The REST server has no rate-limiting middleware. The `grep` for `rateLimit|throttle|express-rate-limit` in `rest/**/*.js` returns only a test-file hit. The `ThrottleConfiguration` / `ThrottleManagerImpl` found in the codebase belongs exclusively to the `web3` Java service, not the REST Node.js service. [5](#0-4) 

3. The optional response cache (`responseCacheCheckHandler`) is gated on `config.cache.response.enabled && config.redis.enabled`. If Redis is not deployed, there is zero caching. Even with Redis, rotating through distinct alias strings (all syntactically valid) bypasses the cache entirely. [6](#0-5) 

### Impact Explanation

Each alias-bearing request consumes one PostgreSQL connection pool slot for the duration of the `entity` table lookup. At high request rates (hundreds/second from a single client or botnet), the pool is saturated. Legitimate requests queue or time out, causing HTTP 503 / timeout responses. No funds are at risk; the impact is pure availability degradation (griefing). Severity: Medium — matches the stated scope of "griefing with no economic damage."

### Likelihood Explanation

- **Preconditions:** None. The endpoint is public, unauthenticated, and documented in the OpenAPI spec. [7](#0-6) 
- **Attacker capability:** Any external user with HTTP access. No credentials, tokens, or on-chain assets required.
- **Repeatability:** Fully repeatable. A single `curl` loop or any HTTP load tool suffices. Rotating alias strings (e.g., `AAAAAAAAAA`, `AAAAAAAAAB`, …) defeats any response-level caching.
- **Feasibility:** High. The alias regex `/^[A-Z2-7]+$/` accepts an enormous space of valid-format strings, all of which trigger a DB lookup. [8](#0-7) 

### Recommendation

1. **Add rate limiting to the REST Node.js service** — apply `express-rate-limit` (or equivalent) globally or specifically on the `/api/v1/balances` route, keyed by IP.
2. **Cache alias-resolution results** — add an in-process LRU/TTL cache inside `EntityService.getAccountIdFromAlias()` (or `getAccountFromAlias()`) for both positive and negative (null) results, similar to the importer's `EntityIdServiceImpl` cache. [9](#0-8) 
3. **Short-circuit on non-existent aliases** — a negative result (alias not found) should be cached with a short TTL to prevent repeated DB hits for the same non-existent alias.
