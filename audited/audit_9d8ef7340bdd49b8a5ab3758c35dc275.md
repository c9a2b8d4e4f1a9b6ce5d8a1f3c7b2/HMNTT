### Title
Integer Overflow in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` Enables Unbounded DB Query via Crafted `limit` > `Integer.MAX_VALUE`

### Summary
The `limit` field in the gRPC `ConsensusTopicQuery` is a `uint64` with no server-side upper-bound cap. When a value greater than `Integer.MAX_VALUE` is supplied, the narrowing cast `(int)(filter.getLimit() - context.getCount().get())` silently overflows to a negative integer. `Math.min` then selects the negative value over `maxPageSize`, and the downstream `hasLimit()` check (`limit > 0`) returns `false` for the negative value, causing `setMaxResults` to be skipped entirely and the JPA query to execute without any row limit.

### Finding Description

**Exact code path:**

1. **Entry point** — `ConsensusController.toFilter()` copies `query.getLimit()` (a Java `long` from a protobuf `uint64`) directly into `TopicMessageFilter` with no upper-bound check:
   ```java
   final var filter = TopicMessageFilter.builder().limit(query.getLimit());
   ```
   The only constraint on `TopicMessageFilter.limit` is `@Min(0)`, which permits any positive `long`, including values > `Integer.MAX_VALUE`.

2. **Overflow site** — In `PollingTopicListener.poll()` (line 54–55) and identically in `PollingTopicMessageRetriever.poll()` (line 68–69):
   ```java
   int limit = filter.hasLimit()
           ? (int) (filter.getLimit() - context.getCount().get())
           : Integer.MAX_VALUE;
   ```
   With `filter.getLimit() = 2_147_483_648L` (= `Integer.MAX_VALUE + 1`) and `count = 0`:
   - `2_147_483_648L - 0L = 2_147_483_648L`
   - `(int) 2_147_483_648L = -2_147_483_648` (wraps to `Integer.MIN_VALUE`)

3. **`Math.min` bypass** — Line 57 / 71:
   ```java
   int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // min(-2147483648, 5000) = -2147483648
   ```
   The negative value is smaller than `maxPageSize`, so the cap is bypassed.

4. **Internal filter rebuild** — `filter.toBuilder().limit(pageSize).build()` creates a new `TopicMessageFilter` with `limit = -2_147_483_648`. The `@Validated`/`@Min(0)` annotation on `TopicMessageFilter` is a Spring Bean Validation annotation; it is **not** triggered by the Lombok builder — only by Spring AOP on `@Validated` bean method parameters. The internal `toBuilder().build()` call is not a Spring-managed invocation, so no validation fires.

5. **Unbounded query** — In `TopicMessageRepositoryCustomImpl.findByFilter()` (lines 51–53):
   ```java
   if (filter.hasLimit()) {          // hasLimit() = (limit > 0) = false for -2147483648
       typedQuery.setMaxResults(...); // NEVER CALLED
   }
   ```
   `setMaxResults` is skipped. The JPA `TypedQuery` executes with no row limit, returning every matching `topic_message` row for the given topic and start time.

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` — only blocks negative input; does not block values > `Integer.MAX_VALUE`.
- `Math.min(limit, maxPageSize)` — designed to cap large positive values; does not handle negative overflow.
- `hasLimit()` (`limit > 0`) — intended to detect "no limit set"; accidentally treats the overflowed negative as "no limit".
- `@Validated` on `TopicMessageServiceImpl.subscribeTopic` — validates the caller-supplied filter, not the internally rebuilt `newFilter`.

### Impact Explanation
An attacker can force the mirror node's gRPC service to execute a full table scan on `topic_message` for any topic with a large message history. The result set is loaded into a Java `List` (`getResultList()`) before streaming, so a sufficiently large topic causes an `OutOfMemoryError` or extreme GC pressure, crashing or severely degrading the gRPC service. Because the `PollingTopicMessageRetriever` path is used for **all** historical message retrieval (not just POLL mode), this is reachable regardless of the configured listener type (`REDIS`, `POLL`, or `SHARED_POLL`). Repeated requests amplify the impact. Service unavailability prevents clients from monitoring HCS topic messages, which is the primary function of this service.

### Likelihood Explanation
- **No authentication required** — the gRPC `subscribeTopic` endpoint is unauthenticated; any network-reachable client can send a crafted request.
- **Trivial to craft** — setting `limit = 2147483648` in a `ConsensusTopicQuery` protobuf message requires no special knowledge beyond the public proto definition.
- **Repeatable** — the attacker can send the request in a tight loop to sustain the DoS.
- **Precondition** — a topic with a non-trivial message history must exist (or the attacker can target any topic; an empty result set is harmless but the query still runs).

### Recommendation
1. **Add an upper-bound cap on `limit` at the API boundary** in `ConsensusController.toFilter()`:
   ```java
   long rawLimit = query.getLimit();
   long cappedLimit = (rawLimit == 0) ? 0 : Math.min(rawLimit, (long) Integer.MAX_VALUE);
   filter.limit(cappedLimit);
   ```
2. **Replace the unsafe narrowing cast** in both `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` with a safe conversion:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Add a `@Max(Integer.MAX_VALUE)` or a configurable max-limit constraint** to `TopicMessageFilter.limit` so the validation layer rejects oversized values before they reach the polling logic.
4. **Add a defensive guard in `findByFilter()`**: always call `setMaxResults` with a safe fallback (e.g., `maxPageSize`) when `hasLimit()` is false or when the limit is non-positive.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=<valid_topic_id>),
    limit=2147483648,  # Integer.MAX_VALUE + 1 — triggers overflow to Integer.MIN_VALUE
)

# This triggers an unbounded DB query; on a topic with large history, causes OOM/crash
for response in stub.subscribeTopic(query):
    print(response)
```

**Reproduction steps:**
1. Deploy the mirror node with `hiero.mirror.grpc.listener.type=POLL` (or any type — the `PollingTopicMessageRetriever` historical path is always active).
2. Ensure a topic exists with a large number of messages (or use any existing topic).
3. Send the above gRPC request with `limit = 2147483648`.
4. Observe: the server executes an unbounded `SELECT` on `topic_message`, loads all rows into memory, and either returns an enormous response or crashes with `OutOfMemoryError`.
