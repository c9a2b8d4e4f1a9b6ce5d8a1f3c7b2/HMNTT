### Title
Missing EVM Address Length Validation Causes `BufferUnderflowException` on Every Malformed Request

### Summary
`GraphQlUtils.decodeEvmAddress()` accepts any even-length hex string without enforcing the required 20-byte EVM address length. `EntityServiceImpl.getByEvmAddressAndType()` then wraps the arbitrarily-short byte array in a `ByteBuffer` and unconditionally calls `getInt()` (needs 4 bytes) and `getLong()` (needs 8 bytes) without checking remaining capacity, throwing an uncaught `BufferUnderflowException` that unwinds the full call stack on every such request.

### Finding Description

**Root cause — `decodeEvmAddress` performs no length check:** [1](#0-0) 

`Hex.decodeHex` only rejects odd-length strings and non-hex characters. Any even-length hex string (0, 2, 4, … bytes) is returned as-is. The `null` branch returns a 0-byte array.

**Unsafe `ByteBuffer` reads — `getByEvmAddressAndType`:** [2](#0-1) 

The three sequential reads require exactly 20 bytes:
- `buffer.getInt()` at offset 0 → needs 4 bytes
- first `buffer.getLong()` at offset 4 → needs 8 bytes (12 total)
- second `buffer.getLong()` at offset 12 → needs 8 bytes (20 total)

If the decoded array is shorter than 4 bytes, `getInt()` throws `BufferUnderflowException`. If 4–11 bytes, the first `getLong()` throws. If exactly 12 bytes with the first 12 all zero, the second `getLong()` throws. `BufferUnderflowException` extends `RuntimeException` and is never caught in this method or its callers.

**No upstream length guard exists.** The `invalidEvmAddress` test only covers null, empty, odd-length, too-long, and non-hex inputs — not short even-length strings: [3](#0-2) 

The `decodeEvmAddress` unit test explicitly validates that a 5-byte result (`"0000AaAaAa"`) is accepted: [4](#0-3) 

The `@Valid` annotation on the controller argument: [5](#0-4) 

…delegates to Bean Validation on `AccountInput`, but no `@Pattern` or `@Size` constraint on `evmAddress` is enforced that would reject short hex strings before they reach the service layer.

### Impact Explanation
Every request carrying a short even-length hex `evmAddress` (e.g., `"0000"`, `"00000000"`, `"000000000000000000000000"`) causes Java to construct a `BufferUnderflowException` including full stack-trace capture, unwind the call stack through the Spring GraphQL data-fetcher pipeline, and invoke the framework's exception-to-error-response translation. This overhead is incurred before any database I/O. Under sustained request load from a single unauthenticated client, the ratio of exception-handling CPU cycles to normal request-processing cycles is sufficient to measurably degrade throughput and increase per-request latency for legitimate users.

### Likelihood Explanation
No authentication or rate-limiting is required. The GraphQL `account` query is publicly reachable. The attacker needs only to know that the `evmAddress` field exists (visible from the GraphQL schema or any introspection query) and supply any even-length hex string shorter than 40 hex characters. The exploit is deterministic, requires no guessing, and is trivially scriptable.

### Recommendation
Add an explicit length check in `decodeEvmAddress` (or at the start of `getByEvmAddressAndType`) that rejects byte arrays whose length is not exactly 20:

```java
public static byte[] decodeEvmAddress(String evmAddress) {
    if (evmAddress == null) {
        return ArrayUtils.EMPTY_BYTE_ARRAY;
    }
    try {
        evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
        byte[] decoded = Hex.decodeHex(evmAddress);
        if (decoded.length != 20) {
            throw new IllegalArgumentException(
                "evmAddress must be exactly 20 bytes (40 hex chars)");
        }
        return decoded;
    } catch (DecoderException e) {
        throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
    }
}
```

Alternatively, add a `@Pattern(regexp = "^(0x)?[A-Fa-f0-9]{40}$")` Bean Validation constraint on the `evmAddress` field of `AccountInput` so the check is enforced at the controller boundary before the service is ever called.

### Proof of Concept
Send the following GraphQL request to the mirror-node GraphQL endpoint (no credentials required):

```graphql
query {
  account(input: { evmAddress: "0000" }) {
    entityId { shard realm num }
  }
}
```

**Expected (buggy) behaviour:** The server throws `java.nio.BufferUnderflowException` inside `getByEvmAddressAndType` at the `buffer.getInt()` call, unwinds the stack, and returns a GraphQL error response. Repeating this request in a tight loop from a single client will drive up CPU utilisation on the server due to repeated stack-trace capture and exception handling, with no database work performed.

**Variants triggering the exception at different read sites:**
- `"00000000"` (4 bytes) → `getInt()` succeeds, first `getLong()` throws
- `"000000000000000000000000"` (12 bytes, all zeros) → `getInt()` and first `getLong()` succeed, second `getLong()` throws
