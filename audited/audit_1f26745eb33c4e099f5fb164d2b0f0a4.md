### Title
Connection Pool Exhaustion via Concurrent EVM Address Lookups During DB Network Partition

### Summary
`getEncodedId()` in `rest/service/entityService.js` unconditionally calls `getEntityIdFromEvmAddress()` for every non-parsable EVM address input, which issues a live DB query via `queryQuietly()` with no result caching. During a network partition between the REST API and the database, these queries hold pool connections open indefinitely (TCP keepalive timeout >> `statement_timeout`), and with a default pool size of 10, just 10 concurrent attacker-controlled requests exhaust the pool, causing all subsequent `getEncodedId()` calls — and any other DB-dependent API path — to fail with connection timeout errors.

### Finding Description

**Exact code path:**

`getEncodedId()` at [1](#0-0)  detects a non-parsable EVM address and unconditionally awaits `getEntityIdFromEvmAddress()`.

`getEntityIdFromEvmAddress()` at [2](#0-1)  calls `this.getRows()` → `pool.queryQuietly()` with no guard.

`queryQuietly()` at [3](#0-2)  takes the `!preQueryHint` branch and calls `this.query(query, params)` — the `node-postgres` pool method that internally acquires a client, runs the query, and holds the client until the query completes or errors.

The pool is configured at [4](#0-3)  with:
- `max: config.db.pool.maxConnections` — default **10**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` — default **20 000 ms**
- `statement_timeout: config.db.pool.statementTimeout` — default **20 000 ms**

The `statement_timeout` is a **server-side** PostgreSQL parameter. During a network partition the server cannot deliver the cancellation signal to the client; the client-side `pg` driver has no independent query deadline. The connection therefore hangs until the OS TCP keepalive fires (typically minutes), not 20 s.

**Failed assumption:** The design assumes `statement_timeout` will bound connection hold time under all failure modes. It does not bound it during a network partition.

**No result caching for EVM address DB lookups:** The `parseCached()` LRU at [5](#0-4)  caches only the parsed `EntityId` struct. Because the cached object retains `evmAddress !== null`, every call to `getEncodedId()` for the same address still re-enters `getEntityIdFromEvmAddress()` and issues a fresh DB query. There is no memoization of the DB result.

**No REST API rate limiting or concurrency cap:** The throttling infrastructure found in the codebase (bucket4j, `ThrottleManagerImpl`) applies only to the `web3` Java service. [6](#0-5)  has no per-IP limit, no in-flight request counter, and no semaphore on concurrent EVM address DB lookups.

### Impact Explanation

With a pool of 10 connections, 10 concurrent requests each supplying a distinct 40-hex-char EVM address (valid per `evmAddressRegex`) will occupy all pool slots. Every subsequent request that needs any DB connection — not just EVM lookups — will block for `connectionTimeoutMillis` (20 s) and then receive a connection-timeout error. This produces **complete REST API unavailability** for the duration of the partition plus the time for TCP keepalives to reclaim the hung connections. The impact is a full denial of service against all API consumers, not just the attacker's own requests.

### Likelihood Explanation

The precondition is a network partition between the REST API pod and the database, which is a realistic transient condition in any cloud-hosted Kubernetes deployment (node drain, CNI flap, PgBouncer restart, etc.). The attacker does not need to cause the partition; they only need to detect it (e.g., by observing elevated latency on EVM-address endpoints) and then send 10 concurrent requests. No authentication is required. The attack is repeatable: once the partition resolves and connections drain, the same 10-request burst re-exhausts the pool if the partition recurs. The default `max_user_client_connections: 1000` for `mirror_rest` in PgBouncer config at [7](#0-6)  does not help because the bottleneck is the Node.js-side pool of 10, not PgBouncer.

### Recommendation

1. **Add a client-side query timeout independent of `statement_timeout`**: Use `pg`'s `query_timeout` pool option (distinct from `statement_timeout`) so the driver cancels and releases the connection after a bounded wall-clock time regardless of network state.
2. **Cache DB lookup results for EVM addresses**: Memoize the `entityId → encodedId` mapping returned by `getEntityIdFromEvmAddress()` using the existing `entityId` cache infrastructure, keyed on the hex EVM address string.
3. **Add a concurrency semaphore on EVM address DB lookups**: Limit the number of simultaneously in-flight `getEntityIdFromEvmAddress()` calls (e.g., to `maxConnections / 2`) so a burst cannot monopolize the entire pool.
4. **Increase default pool size or add per-endpoint connection budgets**: The default of 10 is extremely low for a public API; even modest legitimate traffic can exhaust it under degraded DB conditions.

### Proof of Concept

```
# 1. Confirm DB is partitioned or artificially slow (e.g., iptables drop on DB port).
# 2. Send 10 concurrent EVM-address lookup requests (each with a unique address):
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/accounts/$(openssl rand -hex 20)" &
done
wait

# 3. Immediately send an 11th request (or any DB-dependent request):
curl -v "https://<mirror-node>/api/v1/accounts/0.0.1"
# Expected result: connection timeout error after ~20 s, HTTP 500 or no response.

# 4. Observe that ALL API endpoints backed by pool.queryQuietly() are now unavailable
#    for the duration of the partition + TCP keepalive drain time.
```

The 10 concurrent requests each hit `getEncodedId()` → `getEntityIdFromEvmAddress()` → `pool.queryQuietly()` → `pool.query()`, occupying all 10 pool slots. The 11th request cannot acquire a connection and times out after `connectionTimeoutMillis`.
