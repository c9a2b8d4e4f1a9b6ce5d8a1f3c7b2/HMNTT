### Title
Unauthenticated `/health/readiness` Leaks Raw DB Error Messages as Connection-Exhaustion Oracle

### Summary
The `readinessCheck()` function in `rest/health.js` wraps raw PostgreSQL error messages directly into a `DbError` and throws it. Because the endpoint is registered via `@godaddy/terminus` (which intercepts requests before Express middleware), the Express `handleError` sanitizer that would normally suppress the raw message never runs. Any unauthenticated caller can poll the endpoint without rate-limiting and observe the distinct 503 + raw DB error body to confirm when the connection pool is fully exhausted.

### Finding Description

**Code path:**

`rest/health.js` lines 22–33:
```js
const readinessCheck = async () => {
  return pool
    .query(readinessQuery)
    .catch((err) => {
      throw new DbError(err.message);   // raw pg error forwarded verbatim
    })
    ...
};
``` [1](#0-0) 

`rest/server.js` lines 159–166 registers this under terminus with no custom `sendFailureFunction`, no authentication, and no rate-limit:
```js
createTerminus(server, {
  healthChecks: {
    '/health/readiness': health.readinessCheck,
    ...
  },
  ...
});
``` [2](#0-1) 

**Root cause — two compounding failures:**

1. `DbError(err.message)` forwards the raw `pg` library error string (e.g., `"timeout - Client checkout timed out"`, `"remaining connection slots are reserved for non-replication superuser connections"`) instead of a sanitized message. [3](#0-2) 

2. The Express `handleError` middleware that maps `DbError → 503` with the *generic* `"Service unavailable"` string (suppressing the raw message for non-client-error codes) is **never reached** for terminus-managed routes — terminus intercepts the request and serialises the thrown error directly into the response body. [4](#0-3) [5](#0-4) 

**Pool size:** the default `maxConnections` is 10, making exhaustion trivial. [6](#0-5) 

**No rate-limiting or authentication** exists on the health endpoints anywhere in the REST middleware stack. [7](#0-6) 

### Impact Explanation
An attacker obtains a reliable, zero-privilege oracle that distinguishes three precise states:
- **200** — pool healthy, DB reachable.
- **503 + raw pg timeout message** — pool exhausted / DB partitioned.
- **503 + `NotFoundError`** — DB reachable but `address_book` table empty.

The raw pg error strings also disclose internal infrastructure details (driver version strings, connection-slot counts, host names in some pg error variants). Combined with the tiny default pool (10 connections), an attacker can exhaust the pool with ≥10 concurrent slow API requests, then use the readiness endpoint to confirm exhaustion with sub-second precision before launching a follow-on attack (e.g., cache-poisoning window, replay during degraded state).

### Likelihood Explanation
- **No privilege required** — the endpoint is bound to `0.0.0.0` and publicly reachable.
- **No rate-limit** — polling at any frequency is unconstrained.
- **Pool exhaustion is cheap** — 10 concurrent requests holding connections (e.g., large `limit=` queries with `statementTimeout` of 20 s) saturate the default pool.
- The readiness endpoint itself consumes one pool slot per call, accelerating exhaustion under concurrent polling.
- Fully automatable with a simple shell loop.

### Recommendation
1. **Sanitize the error message**: replace `throw new DbError(err.message)` with `throw new DbError('Database unavailable')` so no raw pg text is forwarded.
2. **Add rate-limiting** to `/health/readiness` (e.g., via an nginx/ingress rule or an in-process token-bucket) to prevent high-frequency polling.
3. **Restrict external exposure**: health endpoints should only be reachable from the cluster/load-balancer health-check source IPs, not from the public internet.
4. **Increase pool size** or add connection-queue depth to raise the bar for exhaustion attacks.

### Proof of Concept

**Step 1 — exhaust the pool (10 connections, 20 s statement timeout):**
```bash
for i in $(seq 1 12); do
  curl -s "http://<host>:5551/api/v1/transactions?limit=100&order=asc" &
done
```

**Step 2 — poll readiness as oracle:**
```bash
while true; do
  resp=$(curl -s -o /tmp/r -w "%{http_code}" http://<host>:5551/health/readiness)
  echo "$(date +%T) HTTP $resp  body: $(cat /tmp/r)"
  sleep 0.5
done
```

**Expected output when pool is exhausted:**
```
12:00:01 HTTP 503  body: {"status":"error","details":{"error":{"message":"timeout - Client checkout timed out"}}}
```
The transition from `200` to `503` with the raw pg timeout string confirms full pool partition, giving the attacker a precise timestamp to begin follow-on exploitation.
