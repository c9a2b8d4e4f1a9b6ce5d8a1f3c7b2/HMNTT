### Title
Unbounded `CaffeineCacheManager` Named-Cache Growth in `cacheManagerSlotsPerContract` Enables Unauthenticated Heap Exhaustion (DoS)

### Summary
`cacheManagerSlotsPerContract()` in `EvmConfiguration.java` constructs a `CaffeineCacheManager` without calling `setCacheNames()`, placing it in dynamic (open) mode. `ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every distinct contract entity ID encountered at runtime, causing the manager's internal `ConcurrentHashMap<String, Cache>` to grow without bound. Because the outer `contractSlotsCache` eviction never removes entries from the inner manager's map, an unauthenticated attacker who issues requests against a large number of distinct contract addresses can exhaust JVM heap memory and crash the web3 service.

### Finding Description

**Exact code locations:**

`EvmConfiguration.java` lines 107–112 — the bean is created with no `setCacheNames()`:
```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    // ← NO setCacheNames() call → dynamic mode
    return caffeineCacheManager;
}
```

`ContractStateServiceImpl.java` lines 85–87 — a new named cache is created per distinct `contractId`:
```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause:**
Spring's `CaffeineCacheManager`, when `setCacheNames()` is not called, operates in dynamic mode: every call to `getCache(name)` with a previously unseen name allocates a new `CaffeineCache` and stores it in an internal `ConcurrentHashMap<String, Cache>`. That map has no eviction mechanism of its own.

**Why the outer cache does not bound the inner map:**
The outer `contractSlotsCache` (bean `CACHE_MANAGER_CONTRACT_SLOTS`, `maximumSize=3000`) stores `contractId → CaffeineCache` entries. When the outer cache evicts an entry (LRU), the `CaffeineCache` object is no longer referenced by the outer cache, but `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap` still holds a strong reference to it under the key `contractId.toString()`. That reference is never removed. The outer cache's bounded size therefore does **not** bound the number of named caches accumulated in the inner manager.

**Feature is enabled by default:**
`CacheProperties.java` line 30: `private boolean enableBatchContractSlotCaching = true;`
`ContractStateServiceImpl.findStorage()` line 59 only skips the batch path when this flag is `false`.

**Accumulation rate:**
Each unique contract address that reaches `findStorageBatch` adds one permanent entry to the inner manager's map. The default `slotsPerContract` spec (`expireAfterAccess=5m,maximumSize=1500`) bounds the *entries inside* each named cache, but not the number of named caches. A Caffeine cache configured with `maximumSize=1500` and time-based expiry allocates non-trivial internal structures (timer wheel, segment arrays, etc.) even when empty — on the order of tens of KB per instance.

### Impact Explanation
An attacker who drives N distinct contract IDs through the `/api/v1/contracts/call` endpoint causes N permanent `CaffeineCache` objects to accumulate in the `cacheManagerSlotsPerContract` map. At ~50 KB overhead per cache instance, 20,000 distinct contract IDs consume ~1 GB of heap. The JVM will eventually throw `OutOfMemoryError`, crashing the web3 service and denying service to all users. Because the mirror node is a read-only query service with no authentication requirement on the contract-call endpoint, this is a complete, unauthenticated availability attack.

### Likelihood Explanation
The Hedera mainnet has hundreds of thousands of deployed contracts, all of which are publicly enumerable via the mirror node REST API (`/api/v1/contracts`). An attacker can trivially enumerate contract IDs and replay them as `to` addresses in `eth_call`-style requests. The default rate limit of 500 requests/second means 20,000 distinct contract IDs can be injected in ~40 seconds. No authentication, no special privilege, and no on-chain transaction is required. The attack is fully repeatable after a service restart.

### Recommendation
1. **Bound the inner manager's named-cache map.** Replace the dynamic `CaffeineCacheManager` with a custom implementation that wraps a `ConcurrentHashMap` with a maximum-size policy (e.g., a Caffeine cache keyed by contract ID whose values are `CaffeineCache` instances), so that the map itself is subject to eviction.
2. **Alternatively, eliminate the two-level cache design.** The outer `contractSlotsCache` already maps `contractId → CaffeineCache`. The inner `cacheManagerSlotsPerContract` is redundant as a `CacheManager`; the `CaffeineCache` objects it holds could be created directly (e.g., via `Caffeine.newBuilder()...build()`) and stored only in the outer bounded cache, removing the unbounded `ConcurrentHashMap` entirely.
3. **Short-term mitigation:** Set `hiero.mirror.web3.cache.enableBatchContractSlotCaching=false` to disable the vulnerable code path until a fix is deployed.

### Proof of Concept
```bash
# 1. Enumerate contract IDs from the mirror node REST API
curl -s "https://<mirror-node>/api/v1/contracts?limit=100&order=asc" | jq -r '.contracts[].evm_address' > contracts.txt

# 2. Replay each contract address as a contract-call target
while IFS= read -r addr; do
  curl -s -X POST "https://<mirror-node>/api/v1/contracts/call" \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$addr\",\"data\":\"0x\",\"gas\":100000}" &
done < contracts.txt
wait

# 3. Repeat with the next page of contracts until heap is exhausted.
# Each unique `addr` permanently adds one CaffeineCache to
# cacheManagerSlotsPerContract's internal ConcurrentHashMap.
# Monitor with: jcmd <pid> VM.native_memory or watch heap via JMX.
```

Each iteration of the loop causes `ContractStateServiceImpl.findStorageBatch()` to call `cacheManagerSlotsPerContract.getCache(contractId.toString())` with a new name, permanently growing the manager's internal map. After sufficient iterations the JVM heap is exhausted and the service crashes with `OutOfMemoryError`.
