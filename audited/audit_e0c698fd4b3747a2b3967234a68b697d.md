### Title
Unauthenticated Request Flood on Expensive Health Endpoints Enables Resource Exhaustion DoS

### Summary
The `Routes()` function in `rosetta/app/middleware/health.go` registers `GET /health/liveness` and `GET /health/readiness` with no rate limiting, authentication, or connection throttling. The readiness handler performs a PostgreSQL connection check and two internal HTTP calls per request. An unprivileged attacker with network access to the port can flood these endpoints to exhaust the DB connection pool and goroutine budget, causing legitimate orchestrator probes to time out and triggering automated pod restarts.

### Finding Description
**Code path:**

- `rosetta/app/middleware/health.go`, `Routes()`, lines 63–78: registers both health routes with bare handler functions and no middleware wrapping.
- `rosetta/main.go`, lines 217–227: the middleware chain is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. No rate-limiting layer exists anywhere in the chain.
- `rosetta/app/config/types.go`, lines 64–69: the `Http` struct exposes only timeout fields (`IdleTimeout`, `ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`). There is no `MaxConnections`, `MaxConcurrentStreams`, or rate-limit configuration field.

**Root cause:** The server applies per-connection timeouts but imposes no bound on the number of simultaneous connections or requests per second. Go's `net/http` spawns one goroutine per accepted connection; without a ceiling, an attacker can open thousands of concurrent connections.

**Exploit flow for `/health/readiness`:**
Each request to `/health/readiness` triggers (health.go lines 37–50, `checkNetworkStatus` lines 80–100):
1. A PostgreSQL connection from the pool (`postgres.New(postgres.Config{DSN: ...})`).
2. An internal HTTP call to `/network/list`.
3. An internal HTTP call to `/network/status`.

Flooding this endpoint simultaneously exhausts the DB connection pool (`Pool.MaxOpenConnections`) and saturates the internal HTTP client. When the pool is exhausted, the readiness check itself returns an error, causing the endpoint to return HTTP 503. Kubernetes/orchestrators polling this endpoint then mark the pod unhealthy and restart it.

**Why existing checks are insufficient:**
- Timeouts (`ReadTimeout`, `WriteTimeout`) only bound individual request duration, not request rate or concurrency.
- `CorsMiddleware` and `MetricsMiddleware`/`TracingMiddleware` add no throttling.
- No IP-based or token-based access control exists on the health routes.

### Impact Explanation
If the health port is reachable by an external attacker (e.g., misconfigured `NodePort`/`LoadBalancer` service, or an attacker on the cluster network), they can continuously restart any targeted node by sustaining the flood. Restarting ≥30% of rosetta nodes simultaneously degrades the network's ability to process Rosetta API requests, meeting the stated medium-severity threshold. The readiness endpoint's DB and HTTP amplification makes the attack cheaper than a pure connection-flood.

### Likelihood Explanation
The attack requires no credentials and no special tooling — `wrk`, `ab`, or `hey` suffice. The only precondition is network reachability of the rosetta port. In cloud-native deployments where the health port is inadvertently exposed (a common misconfiguration), any internet user can trigger this. Even within a cluster, a compromised pod can execute the attack. The attack is repeatable and can be scripted to maintain continuous pressure.

### Recommendation
1. **Add a rate-limiting middleware** (e.g., `golang.org/x/time/rate` token bucket or `github.com/ulule/limiter`) applied globally or specifically to `/health/*` routes, inserted into the chain in `main.go` before the router.
2. **Restrict health endpoint exposure** at the network layer (Kubernetes `NetworkPolicy` limiting kubelet-only access to the health port).
3. **Add `MaxOpenConnections` enforcement** and a separate, lightweight liveness handler that performs no DB I/O.
4. **Set `http.Server.MaxHeaderBytes`** and consider `nethttp.LimitListener` to cap simultaneous connections.

### Proof of Concept
```bash
# Flood the readiness endpoint (replace HOST:PORT with the rosetta service address)
hey -n 100000 -c 500 -m GET http://HOST:PORT/health/readiness &

# In parallel, simulate the orchestrator probe
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" http://HOST:PORT/health/readiness
  sleep 2
done
# Expected: orchestrator probe begins returning 503 as DB pool exhausts;
# Kubernetes marks pod NotReady and triggers restart.

func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
```
