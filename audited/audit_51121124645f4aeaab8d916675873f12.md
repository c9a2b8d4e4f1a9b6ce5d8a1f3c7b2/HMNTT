### Title
Unauthenticated EVM Address Lookup Exhausts DB Connection Pool via `/balances` Endpoint

### Summary
The `getBalances()` handler in `rest/balances.js` allows any unauthenticated caller to supply an EVM address as `account.id`, which unconditionally triggers a live database query against the `entity` table via `EntityService.getEntityIdFromEvmAddress()`. There is no caching of EVM-address-to-entity-ID resolutions, no per-IP rate limiting, and the DB connection pool is finite. An attacker flooding the endpoint with concurrent requests using distinct EVM addresses can saturate the pool, causing all subsequent DB-dependent requests across the entire REST service to queue or fail.

### Finding Description

**Exact code path:**

`getBalances()` calls `parseAccountIdQueryParam()` and immediately awaits the resulting promise array: [1](#0-0) 

Inside `parseAccountIdQueryParam()`, any value that passes `EntityId.isValidEvmAddress()` is handed directly to `EntityService.getEncodedId(value, false)` — returning a live DB-bound Promise: [2](#0-1) 

`getEncodedId()` routes EVM addresses to `getEntityIdFromEvmAddress()`: [3](#0-2) 

`getEntityIdFromEvmAddress()` executes an unconditional, uncached SQL query against the `entity` table: [4](#0-3) [5](#0-4) 

The DB pool is finite, configured via `db.pool.maxConnections`: [6](#0-5) 

**Root cause:** `EntityService.getEntityIdFromEvmAddress()` has no result cache. The `quickLru` cache in `entityId.js` only caches the *parsing* of the ID string format, not the DB lookup result: [7](#0-6) 

The Redis response cache operates at the full HTTP response level and is only populated *after* the handler completes — it does not short-circuit the EVM lookup on the way in.

**Why existing checks fail:**

- The `evmAliasAddressCount === 1` guard limits each single request to one EVM lookup, but does nothing to limit concurrent requests: [2](#0-1) 

- `authHandler.js` only enforces a custom *response result limit* for authenticated users; it does not rate-limit unauthenticated callers at all: [8](#0-7) 

- `statementTimeout` caps individual query duration but does not prevent many concurrent connections from being held simultaneously: [9](#0-8) 

### Impact Explanation
When the pool is exhausted, every subsequent request across the entire REST service that needs a DB connection (all endpoints) will block waiting for `connectionTimeoutMillis` before failing with a connection error. This constitutes a full service-level denial of service — not just for `/balances` but for all API endpoints sharing the same pool. Severity: **Critical** (matches the stated impact category of network-wide transaction confirmation disruption via mirror node unavailability).

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. Any internet-accessible deployment is reachable. The attacker needs only to know the public API path (documented in the OpenAPI spec at `rest/api/v1/openapi.yml`) and generate syntactically valid 20-byte hex EVM addresses (trivially enumerable or randomly generated). The attack is fully repeatable and automatable with standard HTTP load tools (e.g., `wrk`, `ab`, `hey`). Pool sizes in typical deployments are small (tens of connections), making saturation achievable with modest concurrency.

### Recommendation

1. **Cache EVM-address-to-entity-ID resolutions** inside `EntityService.getEntityIdFromEvmAddress()` using the existing Redis `Cache` infrastructure with a short TTL (e.g., 30–60 s), so repeated or concurrent lookups for the same address hit Redis instead of Postgres.
2. **Add per-IP concurrency / rate limiting** middleware (e.g., `express-rate-limit`) applied before route handlers, specifically targeting endpoints that trigger entity lookups.
3. **Add a semaphore or queue** around EVM/alias DB lookups to bound the number of simultaneous `entity` table queries regardless of incoming request volume.
4. **Increase `statementTimeout`** awareness: ensure the pool's `connectionTimeoutMillis` is short enough that queued requests fail fast rather than piling up, reducing cascading failure.

### Proof of Concept

```bash
# Generate 200 random valid EVM addresses and flood the endpoint concurrently
for i in $(seq 1 200); do
  addr=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/balances?account.id=${addr}" &
done
wait

# Observe: subsequent legitimate requests to any endpoint begin timing out
# as the pg connection pool (default maxConnections) is saturated by
# concurrent entity table lookups, each holding a connection for the
# duration of the SELECT query.
```

function parseDbPoolConfig() {
  const {pool} = getConfig().db;
  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
    }
    pool[configKey] = parsed;
  });
}
