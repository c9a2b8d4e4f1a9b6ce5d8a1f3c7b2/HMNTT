### Title
Unbounded Inner Transaction Expansion Causes Amplified DB Load in `getTransactionsByIdOrHash`

### Summary
`getTransactionsByIdOrHash` in `rest/transactions.js` collects all entries from the `inner_transactions` bigint array stored on a batch transaction row and passes the entire array, without any size cap, to `getTransactionsByTransactionIdsSql`, which then issues a second `pool.queryQuietly` call whose SQL complexity (OR clauses + three correlated subqueries) scales linearly with the number of inner transactions. Any unprivileged caller who knows the transaction ID of a large batch transaction can repeatedly trigger this amplified query path, multiplying database load proportionally to the batch size on every request.

### Finding Description
**Exact code path:**

In `rest/transactions.js` lines 944–958, after the initial query returns rows, the handler unconditionally collects every element of every row's `inner_transactions` column:

```js
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();                                          // no length cap

if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,          // entire array forwarded
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);
  rows.push(...innerTransactionRows);
}
```

`getTransactionsByTransactionIdsSql` (lines 828–893) iterates over every `[payer, timestamp]` pair and appends one `(payer_account_id = $N and valid_start_ns = $M)` OR clause to the main `WHERE` predicate, plus one entry to the `payer_account_id IN (…)` list used by three correlated subqueries (`crypto_transfer`, `token_transfer`, `assessed_custom_fee`). With N inner transactions the emitted SQL contains N OR branches in the main scan and N payer IDs in each of the three subquery filters — four query components that all grow with N.

**Root cause / failed assumption:** The design document's pseudo-code explicitly states the inner-transaction lookup should be skipped when `scheduled=true`, and implies a bounded batch size, but the implementation omits both guards. There is no check on `innerTransactions.length` before the second database call, and the `scheduled` filter is not consulted at this branch.

**Why existing checks are insufficient:**
- `utils.buildAndValidateFilters` validates only the HTTP query parameters supplied by the caller; it has no visibility into the size of data returned from the database.
- The `!isTransactionHash` guard only prevents the path when the request uses a hash; a standard transaction-ID request always enters the branch.
- `pool.queryQuietly` has no per-query row or complexity limit visible in this code path.

### Impact Explanation
Every HTTP request to `GET /api/v1/transactions/{batchTransactionId}` for a batch transaction with N inner transactions causes the mirror-node to issue a second database query whose complexity is O(N). Because the `inner_transactions` column is `bigint[]` with no database-level size constraint (migration adds it as `bigint[] default null`), and because the Hedera network's HIP-551 batch limit is not enforced by the mirror node, a sufficiently large batch (or a moderate batch queried at high concurrency) can saturate the PostgreSQL connection pool, degrade query throughput for all other API consumers, and delay the mirror node's ability to serve recent block/transaction data — effectively a targeted read-path DoS against the mirror node REST tier.

### Likelihood Explanation
The attacker requires no credentials and no special network access. They need only:
1. The transaction ID of any existing ATOMIC_BATCH transaction with a large `inner_transactions` array (discoverable via the public `/api/v1/transactions?type=ATOMIC_BATCH` listing endpoint).
2. The ability to send repeated HTTP GET requests.

The attack is fully repeatable, requires no state, and the cost to the attacker is negligible (standard HTTP requests). The cost to the defender scales with N × request rate.

### Recommendation
1. **Cap inner transaction expansion**: Before calling `getTransactionsByTransactionIdsSql` with `innerTransactions`, enforce a hard upper bound (e.g., the network's maximum batch size, or a configurable `maxInnerTransactions` config value). Reject or truncate silently if exceeded.
2. **Honour the `scheduled` guard as designed**: The design document specifies the inner-transaction lookup should be skipped when `scheduled=true`; implement that check.
3. **Single-query approach**: Consider fetching inner transactions via a single `WHERE parent_consensus_timestamp = $batchTimestamp` query rather than expanding all `(payer, valid_start_ns)` pairs into OR clauses, which avoids unbounded query growth entirely.
4. **Response-level caching**: Cache the assembled response for batch transaction IDs to prevent repeated amplified queries for the same ID.

### Proof of Concept
```
# Step 1 – discover a large batch transaction
GET /api/v1/transactions?type=ATOMIC_BATCH&limit=1
# Note the transaction_id from the response, e.g. "0.0.1234-1700000000-000000000"

# Step 2 – repeatedly query it (e.g. with ab or wrk)
ab -n 1000 -c 50 \
  "https://<mirror-node>/api/v1/transactions/0.0.1234-1700000000-000000000"

# Each request causes:
#   Query 1: SELECT … FROM transaction WHERE payer_account_id=… AND valid_start_ns=…
#   Query 2: SELECT … FROM transaction WHERE
#              (payer_account_id=$1 AND valid_start_ns=$2) OR
#              (payer_account_id=$3 AND valid_start_ns=$4) OR … [N pairs]
#            with 3 correlated subqueries each scanning crypto_transfer,
#            token_transfer, assessed_custom_fee with N payer IDs
#
# Observed effect: DB CPU and I/O spike proportional to N × concurrency;
# other API endpoints experience elevated latency or timeouts.
```
