### Title
Unauthenticated `getNodes()` gRPC Endpoint Lacks Rate Limiting with Per-Request Uncached Database Queries Enabling Resource Exhaustion

### Summary
The `getNodes()` handler in `NetworkController` accepts requests from any unauthenticated caller with no application-level rate limiting. Every invocation unconditionally executes two uncached database queries (`addressBookRepository.findLatestTimestamp()` and `nodeStakeRepository.findLatestTimestamp()`) before any cached path is reached, plus allocates a full reactive pipeline. An attacker opening many connections and issuing rapid `limit=1` requests can exhaust the database connection pool and degrade or deny service to legitimate clients.

### Finding Description
**Code path:**

`NetworkController.java:33-43` — `getNodes()` performs no authentication check, no rate-limit check, and no request-count guard before delegating to `networkService.getNodes()`.

`NetworkServiceImpl.java:61-65` — on every call, two queries execute synchronously and without any `@Cacheable` annotation:
```java
long addressBookTimestamp = addressBookRepository
        .findLatestTimestamp(fileId.getId())          // uncached MAX() query
        .orElseThrow(...);
long nodeStakeTimestamp = nodeStakeRepository
        .findLatestTimestamp()                         // uncached MAX() query
        .orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
```
`nodeStakeRepository.findAllStakeByConsensusTimestamp()` is cached (`@Cacheable` on `NodeStakeRepository:24`), and `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` is cached (`@Cacheable` on `AddressBookEntryRepository:16`). However, the two `findLatestTimestamp()` calls are **never cached** and hit the database on every request.

After those queries, `NetworkServiceImpl.java:68-76` constructs a new `Flux.defer()` + `repeatWhen` reactive pipeline per request, consuming thread-pool and scheduler resources.

The only connection-level guard is `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java:14`, `GrpcConfiguration.java:33`), which limits concurrency **per connection** but does not bound the total number of connections or the aggregate request rate across connections.

**Root cause:** The failed assumption is that infrastructure-level controls (the optional GCP gateway `maxRatePerEndpoint: 250` in `charts/hedera-mirror-grpc/values.yaml:69`) will always be present. The application itself has no rate-limiting layer for this endpoint.

### Impact Explanation
Each `getNodes()` call consumes at least two database connections from the HikariCP pool (one per `findLatestTimestamp()` query) plus reactive scheduler threads. With a pool of typical size (10–20 connections), a sustained flood of concurrent requests from multiple connections saturates the pool, causing `getNodes()` calls from legitimate clients to queue or time out. The `hiero.mirror.grpc.db.statementTimeout = 10000 ms` means queued requests hold connections for up to 10 seconds, amplifying the exhaustion. This constitutes a denial-of-service against the address book service with no financial or authentication barrier to the attacker.

### Likelihood Explanation
The gRPC port (5600) is publicly exposed. No API key, token, or TLS client certificate is required. A single attacker machine can open hundreds of TCP connections (each allowed 5 concurrent calls) and issue sequential `getNodes(fileId=0.0.102, limit=1)` requests in a tight loop. The attack requires no special knowledge beyond the publicly documented `grpcurl` invocation in `docs/grpc/README.md:37`. The GCP gateway rate limit is a deployment option (`enabled: true` in the chart) but is not enforced by the application and may be absent in non-GCP or self-hosted deployments.

### Recommendation
1. **Add application-level rate limiting to the gRPC service.** Implement a `ServerInterceptor` using a token-bucket (e.g., Bucket4j, already used in `web3/`) keyed by remote IP that applies to all `NetworkService` RPCs.
2. **Cache `findLatestTimestamp()` results.** Add `@Cacheable` to `AddressBookRepository.findLatestTimestamp()` and `NodeStakeRepository.findLatestTimestamp()` with a short TTL (e.g., matching `cacheExpiry = 2s`) so repeated requests within the window do not hit the database.
3. **Limit total connections.** Configure `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a global `maxConcurrentCalls` (not just per-connection) to bound aggregate server load.

### Proof of Concept
```bash
# Open 50 parallel loops, each sending getNodes with limit=1 as fast as possible
for i in $(seq 1 50); do
  (while true; do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 1}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes \
      > /dev/null 2>&1
  done) &
done
# Monitor HikariCP active connections via /actuator/metrics/hikaricp.connections.active
# Observe pool saturation and increasing latency / timeouts for concurrent legitimate clients
```
