### Title
Global-Only Rate Limiting in `rateLimitBucket` Enables Distributed Token Exhaustion DoS

### Summary
`rateLimitBucket()` creates a single application-wide Bucket4j token bucket shared across all clients with no per-source-IP partitioning. An attacker controlling multiple source IPs can collectively exhaust the 500 req/s global limit, denying service to all legitimate users. Unlike the Rosetta service, the web3 Traefik middleware configuration provides no per-IP countermeasure.

### Finding Description

**Code path:**

`ThrottleConfiguration.java` lines 24–32 instantiate one singleton `Bucket` with capacity equal to `requestsPerSecond` (default 500):

```java
// ThrottleConfiguration.java:25-31
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond();
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.java` lines 37–39 consume from this single shared bucket for every inbound request regardless of origin:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

`RequestFilter.java` lines 39–46 enumerate the filterable fields (`BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE`) — **no IP/source-address field exists**, so operator-configured `RequestProperties` rules cannot target individual clients by network identity either.

**Infrastructure layer — no per-IP protection for web3:**

`charts/hedera-mirror-web3/values.yaml` lines 157–160 define the only Traefik middleware:
```yaml
middleware:
  - retry:
      attempts: 3
      initialInterval: 100ms
```
There is no `inFlightReq` with `ipStrategy` and no `rateLimit` with `sourceCriterion`. By contrast, the Rosetta service (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160) has both:
```yaml
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
- rateLimit:
    average: 10
    sourceCriterion:
      requestHost: true
```
Furthermore, the Traefik middleware template (`charts/hedera-mirror-web3/templates/middleware.yaml` line 3) only renders when `global.middleware = true`, which defaults to `false` — meaning even the retry middleware may be inactive in default deployments.

The GCP gateway `maxRatePerEndpoint: 250` (line 56) limits throughput per backend pod, not per client IP, and `sessionAffinity: CLIENT_IP` (line 58) is a routing hint, not a rate limit.

**Root cause:** The design assumes a single global capacity is sufficient to protect the service, but makes no assumption about how that capacity is distributed across clients. Any attacker who can source requests from N ≥ 500 distinct IPs can each send 1 req/s and collectively consume 100% of the bucket.

### Impact Explanation

The `/api/v1/contracts/call` endpoint (the primary target, per ingress path at line 141) executes EVM contract simulations used for token balance reads, allowance checks, gas estimation, and DeFi interactions on Hedera. Exhausting the 500 req/s global bucket causes all subsequent requests to receive HTTP 429 for the remainder of that second. Sustained attack (trivially achievable with a botnet or 500 cheap cloud VMs) produces continuous service unavailability for all legitimate users. Users relying on this endpoint for pre-flight checks before submitting financial transactions are blocked from doing so.

### Likelihood Explanation

The attack requires no authentication, no special knowledge of the system, and no on-chain funds. 500 req/s is a low threshold — a single attacker with access to a modest botnet, residential proxy network, or 500 cloud VM instances (cost: ~$5–$50/hour) can sustain the attack indefinitely. The attack is repeatable, stateless, and leaves no persistent footprint. The absence of any per-IP control at both the application layer and the Traefik ingress layer (unlike Rosetta) makes this straightforwardly exploitable.

### Recommendation

1. **Per-IP rate limiting at the application layer**: Replace the single global `Bucket` with a `CaffeineProxyManager`-backed `ProxyManager<String>` (Bucket4j's distributed/local proxy), keyed on client IP extracted from `X-Forwarded-For` / `RemoteAddr`. Each IP gets its own sub-bucket (e.g., 10–50 req/s), while the global bucket remains as a secondary ceiling.

2. **Traefik middleware parity with Rosetta**: Add `inFlightReq` with `sourceCriterion.ipStrategy` and `rateLimit` with `sourceCriterion` to `charts/hedera-mirror-web3/values.yaml`, mirroring the Rosetta configuration. Ensure `global.middleware: true` is set in production overlays.

3. **Enable the GCP gateway per-client rate limiting**: Use GCP Cloud Armor or a `BackendPolicy` with per-client IP rate limiting rather than relying solely on `maxRatePerEndpoint`.

### Proof of Concept

```bash
# Simulate 500 distinct source IPs each sending 1 req/s
# (In practice: use a botnet, residential proxies, or 500 cloud VMs)

for i in $(seq 1 500); do
  curl -s -X POST https://<target>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":21000}' \
    --interface eth0:$i &   # or use separate IPs per host
done

# After all 500 fire within the same second, a legitimate user's request:
curl -X POST https://<target>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":21000}'
# Returns: HTTP 429 Too Many Requests
# Body: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The 429 response is confirmed by `ThrottleManagerImpl.java` line 39 and the controller test at `ContractControllerTest.java` line 185.
