### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener` Exhaust Database Connection Pool

### Summary
When the mirror node gRPC service is configured with `type: POLL`, each subscriber gets its own independent polling loop that issues a raw JDBC query against the database every 500 ms (default interval). There is no cap on the number of concurrent subscriptions, no per-IP or per-user rate limit, and the subscriber count is tracked only as a metrics gauge — never enforced as a gate. An unprivileged attacker opening hundreds of concurrent gRPC subscriptions can therefore drive thousands of simultaneous `findByFilter` queries per second, exhausting the shared HikariCP connection pool and denying database access to all mirror-node operations.

### Finding Description

**Code path:**

`PollingTopicListener.listen()` [1](#0-0)  creates a **per-subscription** `PollingContext` and schedules an infinite repeat loop with `RepeatSpec.times(Long.MAX_VALUE)`. Every `interval` milliseconds (default 500 ms, minimum 50 ms enforced by `@DurationMin`) [2](#0-1)  it calls `poll(context)`.

`poll()` issues a synchronous JDBC query wrapped in `Flux.fromStream()`: [3](#0-2) 

Each call to `topicMessageRepository.findByFilter(newFilter)` acquires a connection from the shared HikariCP pool. With N concurrent subscriptions the steady-state query rate is `N / interval_seconds` queries per second — e.g., 200 subscriptions × 2 polls/s = **400 simultaneous DB queries/s**.

`maxPageSize` has only a floor constraint (`@Min(32)`) and no ceiling, defaulting to 5000 rows per poll: [4](#0-3) 

**Why existing checks fail:**

`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` and registers it as a Micrometer gauge, but **never checks it against any maximum before accepting the subscription**: [5](#0-4) 

There is no gRPC interceptor, no per-IP limit, no token bucket, and no server-side max-streams enforcement visible in the codebase. `TopicMessageFilter` validates only that `startTime` is in the past and `endTime > startTime` — no subscription-count gate: [6](#0-5) 

`SharedPollingTopicListener` would mitigate this by sharing a single poll across all subscribers via `.share()`, but that requires `type: SHARED_POLL`. Under `type: POLL` every subscriber gets its own independent loop. [7](#0-6) 

### Impact Explanation

When the HikariCP connection pool (typically 10–20 connections by default) is saturated, every component sharing that pool — historical message retrieval, entity lookups, importer writes — begins timing out or queuing indefinitely. This constitutes a full denial-of-service of the mirror node's database tier, affecting **all** clients and all topics, not just the attacker's subscriptions. Because the gRPC mirror node is a shared infrastructure component, this maps to the stated scope of "shutdown of ≥30% of network processing nodes without brute force."

### Likelihood Explanation

**Precondition:** The operator must have set `hiero.mirror.grpc.listener.type=POLL`. The default is `REDIS`, so this is not universally exploitable. However, `POLL` is a documented, supported mode and is used in environments without Redis.

**Attacker capability:** Any client that can reach the gRPC port (unauthenticated by default) can open concurrent `subscribeTopic` streams using any standard gRPC client or load-testing tool (e.g., `ghz`, `grpcurl` in a loop). No credentials, no special knowledge, and no brute force are required — just many concurrent long-lived streaming RPCs.

**Repeatability:** The attack is trivially repeatable and self-sustaining; streams stay open indefinitely (no server-side timeout enforced in the visible code), so a small number of attacker processes can maintain hundreds of subscriptions continuously.

### Recommendation

1. **Enforce a maximum concurrent subscription count** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., return `RESOURCE_EXHAUSTED` gRPC status).
2. **Add per-IP / per-client connection limits** via a gRPC `ServerInterceptor` or at the load-balancer/ingress level.
3. **Cap `maxPageSize` with `@Max`** in `ListenerProperties` to prevent a single subscription from issuing oversized queries.
4. **Prefer `SHARED_POLL` over `POLL`** in documentation and default configuration, since `SharedPollingTopicListener` amortizes DB load across all subscribers with a single shared query loop.
5. **Set a server-side gRPC `maxConnectionAge` / `keepAliveTimeout`** to reclaim idle or abusive streams.

### Proof of Concept

**Preconditions:** Mirror node running with `hiero.mirror.grpc.listener.type=POLL`; attacker has network access to the gRPC port (default 5600).

**Steps:**

```bash
# Install ghz (gRPC benchmarking tool)
# Proto: com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

# Open 300 concurrent long-lived subscriptions to topic 0.0.1234
ghz --insecure \
    --proto hedera/mirror/api/proto/consensus_service.proto \
    --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    --data '{"topicID":{"topicNum":1234},"consensusStartTime":{"seconds":0}}' \
    --concurrency 300 \
    --duration 120s \
    <mirror-node-host>:5600
```

**Expected result:**
- 300 subscriptions × 2 polls/s = **600 `findByFilter` queries/s** against the database.
- HikariCP pool (default ~10 connections) saturates within seconds.
- All other mirror node DB operations (entity lookups, historical retrieval, importer) begin timing out with `Connection is not available, request timed out after Xms`.
- Legitimate gRPC clients receive errors or stall indefinitely.
- Attack persists until connections are forcibly closed; no server-side defense terminates the streams.

```java
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
