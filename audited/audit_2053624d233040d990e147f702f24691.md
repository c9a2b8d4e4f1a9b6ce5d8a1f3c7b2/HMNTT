### Title
Unbounded Prometheus Label Cardinality via Arbitrary Path Strings in MetricsMiddleware Route Label

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route name extraction to the weaveworks `middleware.Instrument` with a `RouteMatcher`. When a request path does not match any registered route, the weaveworks library falls back to using the raw `r.URL.Path` as the `route` label value in all four Prometheus metric vectors. An unauthenticated attacker can flood the server with requests to unique, non-existent paths, creating an unbounded number of distinct label combinations in the Prometheus registry, leading to memory exhaustion and potential process crash.

### Finding Description
**Exact code location:** `rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83.

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

**Root cause:** The weaveworks `middleware.Instrument.Wrap()` calls an internal `ExtractRouteName` helper that calls `RouteMatcher.Match(r, &routeMatch)`. If the gorilla mux router (returned by `server.NewRouter()` from the coinbase rosetta SDK) does not match the incoming path to any registered route, the helper falls back to `r.URL.Path` as the route label string. This raw, attacker-controlled path is then used as the `"route"` label value when observing all four metric vectors:

- `hiero_mirror_rosetta_request_bytes` (HistogramVec, labels: `method`, `route`)
- `hiero_mirror_rosetta_request_duration` (HistogramVec, labels: `method`, `route`, `status_code`, `ws`)
- `hiero_mirror_rosetta_request_inflight` (GaugeVec, labels: `method`, `route`)
- `hiero_mirror_rosetta_response_bytes` (HistogramVec, labels: `method`, `route`)

Each unique `(method, route)` combination causes the Prometheus client library to allocate a new set of internal metric objects (one per bucket + sum + count per histogram). With 5 buckets per histogram, each unique path creates approximately 7 × 3 + 2 = 23 new time-series objects in the in-process registry.

**Why existing checks fail:** There is no path normalization, no allowlist of valid route strings, no cap on label cardinality, and no rate limiting applied before `MetricsMiddleware`. The middleware is installed at line 217 of `rosetta/main.go` as the outermost application layer (before tracing and CORS), meaning every request — including those to non-existent routes — passes through it unconditionally.

### Impact Explanation
An attacker who sends N requests each with a distinct URL path causes O(N) allocations in the Prometheus default registry. Because the registry holds these objects for the lifetime of the process and never evicts them, heap memory grows without bound. At sufficient volume, the rosetta process is OOM-killed, taking down the Rosetta API entirely. This constitutes a denial-of-service via resource exhaustion with no economic cost to the attacker and no authentication required.

### Likelihood Explanation
The Rosetta API port is publicly reachable (no network-level authentication is shown in the code). Sending HTTP requests to random paths requires no credentials, no special protocol knowledge, and no cryptographic material. A single attacker with a basic HTTP client and a loop generating unique path strings (e.g., `/x/$(uuid)`) can trigger this. The attack is trivially repeatable and scriptable, and the damage accumulates monotonically — there is no self-healing mechanism.

### Recommendation
1. **Normalize the route label before observation:** After `RouteMatcher.Match()` fails, substitute a fixed sentinel string (e.g., `"unmatched"` or `""`) instead of `r.URL.Path`. This can be done by wrapping the `RouteMatcher` in a custom adapter that returns a constant for unmatched requests.
2. **Alternatively, skip metric observation for unmatched routes:** Detect 404 responses and omit label recording entirely for those requests.
3. **Add rate limiting** at the HTTP server or ingress level to bound the request rate from any single source.
4. **Use a Prometheus registry with cardinality limits** (e.g., `prometheus.WrapRegistererWithPrefix` combined with a capping registerer) to hard-limit the number of distinct label sets.

### Proof of Concept
```bash
# Send 100,000 requests to unique non-existent paths (no auth required)
for i in $(seq 1 100000); do
  curl -s -o /dev/null -X POST "http://<rosetta-host>:<port>/attack/path/$i" &
done
wait

# Observe memory growth in the rosetta process
# Each unique path adds ~23 new Prometheus time-series objects to the in-process registry
# At scale, the process heap grows unboundedly until OOM
```

Each iteration creates a new `route="/attack/path/<i>"` label combination across all four metric vectors registered in `init()` at lines 46–52 of `rosetta/app/middleware/metrics.go`, permanently consuming additional heap memory with no eviction path.
