### Title
Unauthenticated DoS via Unbounded Entity Table Scan and Correlated Subquery in `getBalances()` No-Timestamp Path

### Summary
When `GET /api/v1/balances` is called without a `timestamp` parameter, `getBalances()` executes a query that (1) performs a full scan of the entire `entity` table via an unbounded CTE, and (2) runs `getTokenAccountBalanceSubQuery()` as a correlated subquery once per returned row (up to 100). No application-level rate limiting exists for unauthenticated users, and the default DB connection pool is only 10 connections, meaning a small number of concurrent requests from multiple IPs can saturate the pool and sustain high DB CPU load, degrading all mirror node components that share the database.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 113–149, when `tsQuery` is falsy (no timestamp filter supplied), the `else` branch executes:

```js
// rest/balances.js lines 133-149
const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')   -- NO LIMIT: full table scan
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances
  from entity_balance ab
  ${whereClause}
  order by ab.account_id ${order}
  ${limitQuery}`;                            -- limit applied only to outer result
```

`getTokenAccountBalanceSubQuery()` (lines 308–318) produces a correlated subquery referencing `ab.account_id`:

```js
// rest/balances.js lines 308-318
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(...)
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id   -- correlated: runs once per outer row
        and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}  -- 50
    ) as account_token_balance`;
};
```

**Root cause and failed assumption:**

The `entity_balance` CTE contains no `LIMIT` clause — it materializes every ACCOUNT/CONTRACT row from the `entity` table (millions of rows on mainnet) on every invocation. The outer `LIMIT` (up to 100) is applied only after the CTE is evaluated. For each of the up to 100 returned rows, PostgreSQL executes the correlated subquery against `token_account`. This is an N+1 query pattern at the database level, with N up to 100 per HTTP request.

**Why existing checks fail:**

- **Statement timeout (20,000 ms)**: Configured at `rest/dbpool.js` line 15 via `config.db.pool.statementTimeout` (default 20 s per `docs/configuration.md` line 557, enforced at DB level per `charts/hedera-mirror/templates/secret-passwords.yaml` line 124). This allows each query to run for up to 20 seconds, meaning each attacker request holds a DB connection for up to 20 seconds.
- **DB pool size (10 connections)**: Default `maxConnections: 10` (`docs/configuration.md` line 556). Only 10 concurrent requests are needed to exhaust the pool entirely, blocking all other DB operations.
- **Per-IP `inFlightReq: 5` Traefik middleware**: Only applied when `global.middleware: true` (opt-in, disabled by default per `charts/hedera-mirror-rest/values.yaml`). Even when enabled, it is per-IP and trivially bypassed with 2+ source IPs.
- **Redis response cache**: Disabled by default (`hiero.mirror.rest.cache.response.enabled: false`, `docs/configuration.md` line 549). No caching absorbs repeated identical requests.

### Impact Explanation

The shared PostgreSQL database is used by the mirror node importer (which processes Hedera network transactions), the REST API, gRPC API, and other components. Saturating the 10-connection pool or driving sustained high DB CPU load via repeated full `entity` table scans + 100 correlated `token_account` subqueries per request degrades all components. The importer's ability to process and store new transactions is directly impaired, constituting degradation of mirror node processing capacity. On a mainnet-scale deployment with millions of accounts, each no-timestamp `/api/v1/balances?limit=100` request is significantly more expensive than a typical paginated query.

### Likelihood Explanation

The attack requires zero authentication, zero special knowledge, and zero brute force. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml` line 391). The attacker needs only 2 IP addresses to bypass the per-IP in-flight limit (if even enabled) and send 10 concurrent requests to exhaust the connection pool. The attack is trivially scriptable with `curl` or any HTTP client and is fully repeatable. The no-timestamp path is the default behavior when no `timestamp` query parameter is provided, making it the most common access pattern.

### Recommendation

1. **Add a `LIMIT` to the `entity_balance` CTE** or rewrite it to push the `LIMIT` inside the CTE so the full table is never scanned: apply the pagination limit before joining with the correlated subquery.
2. **Replace the correlated subquery with a lateral join or a single aggregated join** so `token_account` is queried once for all result rows rather than once per row.
3. **Enable application-level rate limiting** (e.g., express-rate-limit) for unauthenticated requests to `/api/v1/balances`, independent of the Traefik middleware.
4. **Enable the Redis response cache** for the no-timestamp path, since it returns current state and can be cached for a short TTL (e.g., 15 seconds).
5. **Reduce `statementTimeout`** for the REST API user or add a query-level timeout specific to this endpoint.

### Proof of Concept

```bash
# From two or more IPs, run concurrently:
for i in $(seq 1 10); do
  curl -s "https://<mirror-node-host>/api/v1/balances?limit=100" &
done
wait
```

**Preconditions:** No authentication. No timestamp parameter (triggers the `entity_balance` CTE path).

**Trigger:** 10 concurrent requests from ≥2 IPs exhaust the 10-connection DB pool. Each request causes: (1) full scan of `entity` table for all ACCOUNT/CONTRACT rows, (2) up to 100 correlated subqueries against `token_account`. Each connection is held for up to 20 seconds (statement timeout).

**Result:** DB connection pool exhausted; all other mirror node components (importer, gRPC, other REST endpoints) that require DB connections are blocked or severely degraded for the duration of the attack.
