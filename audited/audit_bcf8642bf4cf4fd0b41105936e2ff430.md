### Title
Unbounded Subscription Lifetime via `Long.MAX_VALUE` endTime Causes Indefinite Resource Exhaustion

### Summary
An unprivileged gRPC client can send a `ConsensusTopicQuery` with `consensusEndTime.seconds >= 9223372035`, which `ConsensusController.convertTimestamp()` explicitly maps to `Long.MAX_VALUE`. This causes `TopicContext.isComplete()` to permanently return `false`, making `pastEndTime()` schedule a `RepeatSpec` with `Long.MAX_VALUE` iterations that never terminates, holding the subscription open indefinitely and exhausting server resources at scale.

### Finding Description

**Step 1 — Attacker-controlled `Long.MAX_VALUE` endTime:**

`ConsensusController.convertTimestamp()` at line 77–79 explicitly converts near-max timestamps to `Long.MAX_VALUE`: [1](#0-0) 

Any unauthenticated gRPC client can send `consensusEndTime.seconds = 9223372035` to trigger this path.

**Step 2 — Validation does not reject `Long.MAX_VALUE`:**

`TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`: [2](#0-1) 

Since `startTime` is current nanoseconds (far below `Long.MAX_VALUE`), validation always passes.

**Step 3 — `isComplete()` permanently returns `false`:**

`TopicContext.isComplete()` evaluates `Instant.ofEpochSecond(0, Long.MAX_VALUE)` which resolves to approximately year 2262, so `.isBefore(Instant.now())` is always `false`: [3](#0-2) 

**Step 4 — `pastEndTime()` creates a never-terminating `RepeatSpec`:**

Because `endTime != null` and `isComplete()` never returns `true`, `pastEndTime()` creates a `RepeatSpec` with `Long.MAX_VALUE` iterations and a 30-second fixed delay that runs indefinitely: [4](#0-3) 

**Step 5 — Subscription is held open indefinitely:**

`pastEndTime` is used as the termination signal via `takeUntilOther`. Since it never completes, the merged `live` and `safetyCheck` flux never terminates: [5](#0-4) 

Each open subscription holds: a `TopicContext` object, a `TopicListener` subscription, a gRPC stream slot, and a periodic scheduler task every 30 seconds (`endTimeInterval`). [6](#0-5) 

### Impact Explanation

An attacker opening N concurrent subscriptions with `endTime = Long.MAX_VALUE` causes N subscriptions to remain open indefinitely. Each subscription consumes heap memory, a gRPC stream slot, a topic listener registration, and a recurring scheduler task. There is no server-side enforcement of a maximum subscription count or maximum subscription lifetime in the shown code. At sufficient scale this exhausts heap, gRPC stream capacity, or the bounded-elastic scheduler's task queue, denying service to legitimate subscribers. The `subscriberCount` gauge tracks the leak but does not enforce any limit. [7](#0-6) 

### Likelihood Explanation

No authentication is required. The exploit requires only a standard gRPC client and a single field set to a large value. It is trivially scriptable: a loop opening hundreds of subscriptions per second is sufficient to exhaust resources on a typical deployment. The `convertTimestamp` helper even documents the `Long.MAX_VALUE` return path explicitly, making it easy to discover. The attack is repeatable across server restarts if the attacker reconnects immediately.

### Recommendation

1. **Add an upper bound on `endTime`** in `TopicMessageFilter` — e.g., reject any `endTime` more than a configurable maximum duration (e.g., 24 hours) beyond `Instant.now()`.
2. **Enforce a maximum subscription lifetime** in `TopicMessageServiceImpl` using `flux.timeout(maxDuration)` regardless of `endTime`.
3. **Enforce a per-IP or global maximum concurrent subscriber count** and return `RESOURCE_EXHAUSTED` when exceeded.
4. **Remove the explicit `Long.MAX_VALUE` return** in `convertTimestamp` or clamp it to a reasonable future bound before passing it to the filter.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hederahashgraph.api.proto.java import consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Set consensusEndTime.seconds >= 9223372035 to trigger Long.MAX_VALUE path
end_ts = Timestamp(seconds=9223372036, nanos=0)

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=...,          # any valid topic ID
    consensusEndTime=end_ts
)

# Open hundreds of subscriptions in a loop; none will ever terminate server-side
for _ in range(500):
    stub.subscribeTopic(query)  # each holds resources indefinitely
```

Each subscription passes `isValidEndTime()`, enters `pastEndTime()` with `Long.MAX_VALUE` iterations, and `isComplete()` never returns `true`, so the subscription is held open until the server runs out of resources.

        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
