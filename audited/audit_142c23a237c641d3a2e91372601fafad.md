### Title
Unauthenticated Cache Flooding via Unique Query Parameters Exhausts Redis Memory and Degrades Cache Integrity

### Summary
The `responseCacheUpdateHandler()` in `rest/middleware/responseCacheHandler.js` unconditionally writes a Redis entry for every successful response, keyed by an MD5 hash of the full `req.originalUrl` (including all query parameters) with no per-IP or per-user write rate limit. An unprivileged external attacker can flood the REST API with requests bearing unique query parameter values, generating unbounded unique cache keys that fill Redis to its `maxMemory` limit (default 250 MB), forcing eviction of legitimate cached entries and increasing database read load for all subsequent users.

### Finding Description

**Exact code path:**

`cacheKeyGenerator` hashes the raw `req.originalUrl` with no query-parameter normalization: [1](#0-0) 

Every successful response with `ttl > 0` is unconditionally written to Redis: [2](#0-1) 

The underlying `setSingle` issues a Redis `SETEX` with no guard on total key count or per-source write rate: [3](#0-2) 

**Root cause:** The cache key is derived from the full, un-normalized URL. The comment in the code explicitly acknowledges that query-parameter normalization is deferred ("In the future, this will utilize Edwin's request normalizer (9113)"): [4](#0-3) 

**No rate limiting on the REST Node.js service:** The middleware chain in `server.js` includes `authHandler` (which only sets a response-size limit for authenticated users) and no IP-based or global request-rate-limiting middleware: [5](#0-4) 

The throttle mechanisms found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belong exclusively to the `web3` Java service and do not apply to the REST Node.js API.

**Exploit flow:**
1. Attacker sends a high volume of GET requests to any cacheable endpoint (e.g., `/api/v1/accounts?account.id=gte:0.0.1`, `…gte:0.0.2`, …) with unique query parameter values.
2. Each request produces a unique `req.originalUrl`, a unique MD5 cache key, and a unique Redis `SETEX` entry.
3. Redis fills to its configured `maxMemory` (default 250 MB).
4. The `maxMemoryPolicy` (`allkeys-lfu`) begins evicting the least-frequently-used keys. Attacker's one-time keys have frequency=1, but so do any recently-added legitimate entries. Sustained flooding maintains constant memory pressure, continuously evicting low-frequency legitimate entries.
5. Subsequent legitimate users experience cache misses for evicted entries and receive responses sourced directly from the database.

**Why existing checks are insufficient:**

- `allkeys-lfu` is a partial mitigation: it preferentially evicts attacker one-time keys, but under sustained flooding it also evicts legitimate entries with low access frequency (e.g., newly cached or infrequently accessed mirror node records).
- `maxRepeatedQueryParameters: 100` limits repetition of a single parameter name but does not prevent an attacker from using distinct parameter *values* across requests. [6](#0-5) 
- There is no per-IP write budget, no maximum number of distinct cache keys, and no application-level rate limiter on the REST API.

### Impact Explanation

Redis memory exhaustion degrades the response cache's effectiveness. Evicted entries force the application to fall back to direct PostgreSQL reads, increasing database load and response latency for all users. Under a sustained attack, the cache provides no benefit, and the database connection pool (default max 10 connections) becomes the bottleneck, potentially causing query timeouts (default 20 s) for legitimate users. The default Redis memory cap of 250 MB and the absence of any rate limiting make this straightforward to trigger. [7](#0-6) 

### Likelihood Explanation

No credentials, tokens, or special network access are required. Any external user can issue HTTP GET requests to the public REST API. The attack is trivially scriptable (e.g., a simple loop incrementing a numeric query parameter). The absence of any application-level rate limiting on the Node.js REST service means the attacker is limited only by their own network bandwidth and the server's ability to process requests.

### Recommendation

1. **Normalize cache keys before hashing:** Sort and deduplicate query parameters before computing the MD5 key so that semantically equivalent requests share a cache entry and the attacker's parameter-variation strategy is neutralized.
2. **Add application-level rate limiting:** Introduce an IP-based rate-limiting middleware (e.g., `express-rate-limit`) in the Node.js REST server's middleware chain, applied before `responseCacheCheckHandler`.
3. **Bound cache write rate per IP:** Track and limit the number of distinct cache keys written per source IP within a sliding window.
4. **Consider a Redis key-count cap or namespace quota:** Prevent any single namespace or key pattern from consuming a disproportionate share of Redis memory.

### Proof of Concept

```bash
# Flood the accounts endpoint with unique query parameter values
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts?account.id=gte:0.0.$i" &
done
wait
```

Each iteration generates a unique `req.originalUrl`, a unique MD5 cache key, and a unique Redis `SETEX` write. After sufficient iterations, Redis reaches its 250 MB `maxMemory` limit and begins evicting entries under `allkeys-lfu`. Monitoring Redis with `redis-cli info stats | grep evicted_keys` will show a rising eviction count. Subsequent requests for previously cached URLs will result in cache misses and direct database reads.

/*
 * Generate the cache key to access Redis. While Accept-Encoding is specified in the API response Vary
 * header, and therefore that request header value should be used as part of the cache key, the cache
 * implementation stores the response body as the original JSON object without any encoding applied. Thus it
 * is the same regardless of the accept encoding specified, and chosen by the compression middleware.
 *
 * Current key format:
 *
 *   path?query - In the future, this will utilize Edwin's request normalizer (9113).
 */
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
