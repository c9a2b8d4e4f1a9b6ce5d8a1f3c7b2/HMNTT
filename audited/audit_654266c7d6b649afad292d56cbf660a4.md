### Title
Unauthenticated Cache Flooding of Global PreparsedDocumentProvider via Unique Query Strings

### Summary
The `CachedPreparsedDocumentProvider` maintains a single global Caffeine cache keyed by raw query string with `maximumSize=1000` and `expireAfterWrite=1h`. No application-level rate limiting exists in the GraphQL module, and the only infrastructure control (`inFlightReq: amount: 5` per IP) limits concurrency but not total request volume. An unauthenticated attacker can flood the cache with 1000 unique query strings, evicting all legitimate cached entries and forcing continuous re-parsing overhead for every subsequent legitimate request.

### Finding Description

**Cache implementation** — `CachedPreparsedDocumentProvider.java` line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The cache key is the verbatim query string from `ExecutionInput`. Any two queries that differ by even one character (e.g., a different inline literal transaction ID) are treated as distinct cache entries.

**Cache configuration** — `CacheProperties.java` line 15:
```java
private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```
The cache is global (one instance per JVM), holds at most 1000 entries, and entries live for 1 hour. Once full, Caffeine evicts entries using its W-TinyLFU policy.

**No application-level rate limiting in the GraphQL module** — `GraphQlConfiguration.java` lines 42–48 wires `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, but both run *after* the preparsed document is retrieved from (or inserted into) the cache. The `ThrottleConfiguration`/`ThrottleManagerImpl` with Bucket4j exists only in the `web3` module; there is no equivalent in the `graphql` module.

**Only infrastructure control** — `charts/hedera-mirror-graphql/values.yaml` lines 138–142:
```yaml
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
```
This Traefik middleware limits *concurrent in-flight* requests to 5 per source IP. It does not cap total request rate or total unique queries submitted over time. An attacker sending requests sequentially (5 at a time) from a single IP is not blocked.

**Exploit flow:**
1. Attacker crafts 1000+ syntactically valid GraphQL queries with different inline literal values (e.g., varying transaction hash strings). Each query is ≤10,000 characters (within `maxCharacters` limit) and ≤1000 tokens.
2. Attacker sends them sequentially (≤5 concurrent per IP) to `POST /graphql/alpha` with no authentication.
3. Each unique query string misses the cache and triggers `parseAndValidateFunction.apply(executionInput)` — full lexing, parsing, and schema validation.
4. After 1000 entries, the cache is at capacity. Each new attacker query evicts a legitimate cached entry.
5. Attacker continuously cycles through new unique queries, keeping the cache permanently polluted. Legitimate users' queries are never cached; every legitimate request incurs full re-parse cost.

### Impact Explanation
Every cache miss forces the GraphQL engine to lex, parse, and validate the query against the schema — a CPU-bound operation. Under sustained cache flooding, all legitimate queries pay this cost on every request. Combined with the `retry: attempts: 3` middleware (which retries failed requests up to 3×), each attacker request can trigger up to 3 parse cycles. At scale this degrades service availability for all users querying transaction data, with no self-healing until the attacker stops (entries expire after 1 hour).

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public GraphQL schema, and no infrastructure access. It is trivially scriptable: generate 1000+ query strings with varying inline literals and POST them sequentially. The `inFlightReq: amount: 5` per-IP limit only slows the fill rate (200 sequential batches of 5), it does not prevent it. An attacker with even a handful of IPs (e.g., residential proxies) can fill the cache in seconds. The attack is repeatable indefinitely.

### Recommendation
1. **Normalize the cache key**: Strip or ignore inline literal values before caching — use only the structural query template (field names, operation type, argument names) as the key, or require clients to use GraphQL variables (which keep the query string constant across different IDs).
2. **Add application-level rate limiting to the GraphQL module**: Implement a per-IP or per-connection token-bucket rate limiter (analogous to `ThrottleConfiguration` in `web3`) that caps total requests per second, not just concurrency.
3. **Bound cache insertion rate**: Wrap `cache.get(...)` with a per-IP or global insertion-rate guard so that cache misses (new entries) are throttled independently of cache hits.
4. **Reduce `expireAfterWrite`**: Shortening the TTL reduces the window an attacker-inserted entry occupies a slot, but does not eliminate the attack.

### Proof of Concept
```bash
# Fill the cache with 1001 unique query strings (each with a different inline hash literal)
for i in $(seq 1 1001); do
  HASH=$(printf '%064x' $i)
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ transaction(input: { hash: \\\"0x${HASH}\\\" }) { id } }\"}" &
  # Respect inFlightReq limit: batch 5 at a time
  if (( i % 5 == 0 )); then wait; fi
done
wait

# Now send a legitimate query that was previously cached — it will miss and re-parse
curl -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: { entityId: { shard: 0, realm: 0, num: 2 } }) { balance } }"}'
# Observe: response time is higher than a cached hit; repeat to confirm no caching benefit
