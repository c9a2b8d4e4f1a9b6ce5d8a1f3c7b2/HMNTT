### Title
Deleted-Entity Bypass via Crafted EVM Address Exposes Staking Reward Data Through `getByEvmAddressAndType()`

### Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` routes a crafted 20-byte EVM address (first 12 bytes zero, last 8 bytes encoding any entity ID) to `entityRepository.findById()`, which is Spring Data's `CrudRepository.findById()` and carries no `deleted is not true` filter. Both sibling lookup paths (`findByEvmAddress`, `findByAlias`) explicitly filter deleted entities, but the `findById` branch does not. Any unauthenticated caller can therefore retrieve the full `Entity` record of a deleted account, and the GraphQL framework will subsequently resolve `pendingReward` (and other sensitive fields) against that record.

### Finding Description

**Exact code path:**

`EntityServiceImpl.getByEvmAddressAndType()` â€” lines 34â€“41: [1](#0-0) 

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);

    return entityRepository.findById(buffer.getLong())         // bytes 12-19 = entity ID
```

**Root cause â€” asymmetric deleted filtering:**

The graphql-module `EntityRepository` defines explicit `deleted is not true` guards on both custom queries: [2](#0-1) 

However, `findById` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT * FROM entity WHERE id = ?` with no deleted predicate. The `findByAlias` and `findByEvmAddress` branches are safe; the `findById` branch is not.

**Exploit flow:**

1. Attacker identifies a deleted account's numeric entity ID (e.g., `12345`).
2. Attacker constructs a 20-byte hex EVM address: bytes 0â€“3 = `00000000`, bytes 4â€“11 = `0000000000000000`, bytes 12â€“19 = `0000000000003039` (big-endian encoding of `12345`).
3. Attacker sends a GraphQL query:
   ```graphql
   { account(input: { evmAddress: "0x000000000000000000000000000000000000003039" }) {
       pendingReward deleted balance key memo
   }}
   ```
4. `buffer.getInt()` reads bytes 0â€“3 â†’ `0`; `buffer.getLong()` reads bytes 4â€“11 â†’ `0`; condition is true.
5. `entityRepository.findById(12345)` returns the deleted `Entity` row.
6. The `.filter(e -> e.getType() == type)` check only validates entity type, not deletion status.
7. The GraphQL framework resolves `pendingReward` by querying `entity_stake` by entity ID (no deleted guard there either), returning the unclaimed reward value.

**Why existing checks are insufficient:**

The only post-retrieval filter is `.filter(e -> e.getType() == type)` â€” it enforces entity type but not deletion status. [3](#0-2) 

The `findByEvmAddress` path correctly applies `deleted is not true` at the SQL level: [4](#0-3) 

The `findById` path has no equivalent guard.

**`pendingReward` exposure:**

`pendingReward` is a first-class field on the `Account` GraphQL type: [5](#0-4) 

It is also declared on the `Accountable` interface: [6](#0-5) 

The underlying data lives in `entity_stake` (keyed by entity ID, no deletion guard), so the GraphQL resolver will return the stale `pendingReward` value for a deleted account. [7](#0-6) 

### Impact Explanation

An unauthenticated attacker can read the full `Entity` record of any deleted account, including `pendingReward`, `balance`, `key`, `memo`, `deleted` (confirmed `true`), and staking metadata. This is an **information disclosure** vulnerability. The attacker cannot claim or redirect the rewards through this endpoint â€” the impact is data leakage of financial state that the protocol intends to hide after account deletion. Severity is **Medium**: no funds are moved, but sensitive financial data of deleted accounts is exposed to any caller with network access to the GraphQL endpoint.

### Likelihood Explanation

Exploitation requires no authentication, no special privileges, and no on-chain interaction. The attacker only needs to know (or enumerate) a deleted account's numeric entity ID, which is public information derivable from the mirror node's own REST API or block explorers. The crafted EVM address format is trivially constructable. The attack is fully repeatable and scriptable.

### Recommendation

Add a deleted-entity guard to the `findById` branch, consistent with the other two lookup paths. The simplest fix is to add a post-retrieval filter:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
               .filter(e -> e.getType() == type && !Boolean.TRUE.equals(e.getDeleted()));
}
```

A more robust fix is to add a dedicated repository method with a SQL-level deleted guard:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

and replace the `findById` call with `findByIdAndNotDeleted`. This mirrors the pattern already used for `findByAlias` and `findByEvmAddress`. [2](#0-1) 

Note also that `getByIdAndType()` (line 25) has the identical missing deleted check and should be fixed in the same pass. [8](#0-7) 

### Proof of Concept

**Preconditions:** A deleted account with entity ID `N` that had staking rewards exists in the mirror node database.

**Steps:**

1. Encode `N` as a big-endian 8-byte hex string, e.g., for `N = 12345`: `0000000000003039`.
2. Prepend 24 zero hex characters to form a 40-character EVM address: `0x000000000000000000000000000000000000003039`.

   *(Note: the address must be exactly 40 hex chars / 20 bytes; adjust padding accordingly.)*

3. Send the following GraphQL request to the mirror node GraphQL endpoint:

```graphql
{
  account(input: { evmAddress: "0x000000000000000000000000000000000000003039" }) {
    entityId
    deleted
    balance
    pendingReward
    memo
    key
  }
}
```

4. **Expected (vulnerable) response:** The server returns the deleted account's data including `deleted: true` and a non-zero `pendingReward` value, confirming the bypass.

5. **Expected (fixed) response:** The server returns `null` or a not-found error, consistent with the behavior of `findByEvmAddress` for deleted entities.
