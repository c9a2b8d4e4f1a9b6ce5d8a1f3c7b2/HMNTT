### Title
Unauthenticated `GET /api/v1/network/stake` Exhausts HikariCP Connection Pool via Uncached Correlated Subquery

### Summary
The `getNetworkStake()` endpoint in `NetworkController` accepts unlimited unauthenticated requests with no rate limiting and no caching. Every request triggers a correlated subquery against the `network_stake` table via `NetworkStakeRepository.findLatest()`, consuming a HikariCP connection for the full query duration. Under high concurrency, an attacker can saturate the shared connection pool and degrade all other endpoints served by the same application.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 127–130) calls `networkService.getLatestNetworkStake()` unconditionally with no guards: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) directly delegates to the repository with no caching layer: [2](#0-1) 

`NetworkStakeRepository.findLatest()` (lines 12–19) executes a native correlated subquery — two sequential scans per call: [3](#0-2) 

**Root cause — three compounding failures:**

1. **No rate limiting in rest-java.** The `ThrottleManager`/bucket4j infrastructure exists exclusively in the `web3` module. The `rest-java` module has no equivalent. Searching all config classes under `rest-java/src/main/java/org/hiero/mirror/restjava/config/` (`LoggingFilter`, `MetricsFilter`, `WebMvcConfiguration`, `RestJavaConfiguration`) reveals no request-rate enforcement for any endpoint.


2. **No caching.** The grpc module's analogous `NodeStakeRepository.findAllStakeByConsensusTimestamp()` is annotated `@Cacheable`, explicitly protecting it from repeated DB hits: [4](#0-3) 
`NetworkServiceImpl.getLatestNetworkStake()` and `NetworkStakeRepository.findLatest()` carry no such annotation.

3. **Finite shared HikariCP pool.** The application uses a single HikariCP pool (confirmed by the `hikaricp_connections_active` / `hikaricp_connections_max` Grafana alert at the 75% threshold): [5](#0-4) 
The pool is configured via `spring.datasource.hikari` with no per-endpoint connection budget: [6](#0-5) 

**Why existing checks fail:** The `MetricsFilter` and `LoggingFilter` are observability-only. No servlet filter, interceptor, or Spring Security rule enforces a per-IP or global request rate on this endpoint.

### Impact Explanation

When the HikariCP pool is exhausted, every other endpoint (`/api/v1/network/nodes`, `/api/v1/network/supply`, `/api/v1/accounts`, etc.) that needs a DB connection will block waiting for one to become available, eventually timing out and returning 500 errors to legitimate users. The `network_stake` table is updated only once per staking period (~24 h), so the data returned is effectively static — yet every request re-executes the correlated subquery. The impact is full service degradation for all consumers of the mirror node REST API, not just the `/stake` endpoint.

### Likelihood Explanation

No preconditions are required: the endpoint is publicly documented in the OpenAPI spec, accepts zero parameters, and requires no authentication or API key: [7](#0-6) 
A single attacker with a modest botnet (or even a single machine using async HTTP clients) can sustain thousands of concurrent requests. The attack is trivially repeatable and requires no on-chain funds or privileged access.

### Recommendation

Apply at least two of the following mitigations:

1. **Add `@Cacheable`** to `NetworkServiceImpl.getLatestNetworkStake()` or `NetworkStakeRepository.findLatest()` with a TTL matching the staking period (e.g., 5–60 minutes). This eliminates the DB hit for the vast majority of requests.

2. **Add a global rate-limiting filter** in the `rest-java` module (mirroring the bucket4j `ThrottleConfiguration` already present in `web3`) that caps total requests per second across all endpoints, or specifically caps `/api/v1/network/stake`.

3. **Configure HikariCP `connectionTimeout`** to fail fast rather than queue indefinitely, so pool exhaustion causes quick 503s rather than cascading thread starvation.

### Proof of Concept

```bash
# Requires: wrk or similar HTTP load tool
# No authentication, no parameters needed

wrk -t 16 -c 500 -d 60s \
  "https://<mirror-node-host>/api/v1/network/stake"

# Expected result within seconds:
# - hikaricp_connections_active reaches hikaricp_connections_max
# - Concurrent requests to /api/v1/network/nodes, /api/v1/accounts, etc.
#   begin returning HTTP 500 (connection pool timeout)
# - Application logs show HikariCP "Connection is not available, request timed out"
```
