### Title
Unauthenticated Resource Exhaustion via Unbounded DB Queries on NFT Allowances Endpoint

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint in `AllowancesController` accepts any valid numeric account ID without rate limiting. For numeric IDs, `EntityServiceImpl.lookup()` skips the entity DB lookup entirely and always succeeds, causing `NftAllowanceRepositoryCustomImpl.findAll()` to execute a DB query on every request. With no rate limiting in the `rest-java` module and a circuit breaker that only triggers on error rates (not request volume), an unprivileged attacker can flood this endpoint with arbitrary numeric IDs to drive sustained DB load.

### Finding Description
**Code path:**

1. `AllowancesController.getNftAllowances()` (line 58–83) accepts `{id}` as `EntityIdParameter` with no authentication or rate-limiting guard.
2. `NftAllowanceServiceImpl.getNftAllowances()` (line 28) calls `entityService.lookup(request.getAccountId())`.
3. `EntityServiceImpl.lookup()` (line 31–32) handles `EntityIdNumParameter` with `Optional.of(p.id())` — **no DB query is issued**, and the call always succeeds for any positive numeric value. This is explicitly confirmed by the test comment: *"Valid numeric account IDs are not looked up in the entity table in support of partial mirror nodes."*
4. `NftAllowanceRepositoryCustomImpl.findAll()` (line 41–46) then unconditionally executes `SELECT * FROM nft_allowance WHERE (owner = ? AND approved_for_all = true) ORDER BY ... LIMIT ?` against the DB. For non-existent accounts this returns 0 rows but the query still executes and consumes DB resources.
5. The response is HTTP 200 with an empty allowances list — **not a 4xx/5xx error**.

**Root cause:** The `rest-java` module contains zero rate-limiting infrastructure. A grep across all `rest-java/src/main/java/**/*.java` files confirms no `bucket4j`, `Bucket`, or `ThrottleManager` usage in any controller or service handling this endpoint. The `ThrottleConfiguration` and `ThrottleManagerImpl` exist exclusively in the `web3` module. The ingress middleware (`charts/hedera-mirror-rest-java/values.yaml` lines 150–155) only configures a circuit breaker on error ratios and a retry policy — neither limits request throughput.

**Why existing checks fail:**
- The circuit breaker (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) never triggers because all responses are HTTP 200.
- The retry middleware amplifies rather than limits load.
- `@Max(MAX_LIMIT)` on the `limit` parameter caps result size but not request frequency.
- No authentication is required.

### Impact Explanation
Each request causes one indexed DB query against the `nft_allowance` table. While each individual query is cheap, an attacker issuing thousands of requests per second across many different numeric IDs (e.g., `0.0.1` through `0.0.N`) produces a sustained, unbounded stream of DB queries. This can exhaust DB connection pool capacity (monitored via `hikaricp_connections_active`), increase CPU and I/O on the DB host, and degrade response times for all other mirror node API consumers. The 30% resource consumption threshold is achievable with modest request rates given the absence of any throttle.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The endpoint is publicly routable via the ingress path regex (`/api/v1/accounts/(\d+\.){0,2}(\d+|...)/allowances/nfts`). Any attacker with HTTP access can automate requests using standard tooling (curl, wrk, ab). The attack is repeatable, stateless, and produces no error signals that would trigger automated defenses.

### Recommendation
1. Add application-level rate limiting to the `rest-java` module analogous to `ThrottleManagerImpl` in `web3` — implement a `bucket4j`-based filter on the `/api/v1/accounts/*/allowances/*` path family.
2. Add per-IP rate limiting at the ingress layer (e.g., Traefik `RateLimit` middleware) in `charts/hedera-mirror-rest-java/values.yaml`.
3. Consider returning HTTP 404 for numeric account IDs that do not exist in the entity table before executing the allowance query, eliminating the "always succeeds" bypass in `EntityServiceImpl.lookup()` for the REST API context.

### Proof of Concept
```bash
# Flood with distinct non-existent numeric account IDs, each triggering a DB query
for i in $(seq 1000000 1100000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$i/allowances/nfts" &
done
wait
# All requests return HTTP 200 with {"allowances":[],"links":{...}}
# DB query count increases proportionally; no circuit breaker or throttle activates
```

Observe `hikaricp_connections_active` and DB CPU metrics rising without any 4xx/5xx responses being generated, confirming the circuit breaker remains silent throughout.
