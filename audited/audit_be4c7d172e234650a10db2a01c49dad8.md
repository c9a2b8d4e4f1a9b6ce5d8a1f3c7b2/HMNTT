### Title
Unbounded Historical Time Window in `toFilter()` Enables Unauthenticated DB Resource Exhaustion

### Summary
`ConsensusController.toFilter()` converts user-supplied `consensusStartTime` and `consensusEndTime` into a `TopicMessageFilter` with no constraint on the size of the time window. The only validation in `TopicMessageFilter.isValidStartTime()` checks that `startTime <= now`, meaning any past timestamp (including Unix epoch) is accepted. An unauthenticated attacker can open many concurrent gRPC connections, each requesting messages from epoch to `Long.MAX_VALUE`, forcing the `PollingTopicMessageRetriever` to execute repeated paginated DB scans consuming significant CPU and I/O until the 60-second retriever timeout fires.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` at `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java` lines 55–73:

```java
if (query.hasConsensusStartTime()) {
    long startTime = convertTimestamp(query.getConsensusStartTime());
    filter.startTime(startTime);          // no lower-bound check
}
if (query.hasConsensusEndTime()) {
    long endTime = convertTimestamp(query.getConsensusEndTime());
    filter.endTime(endTime);              // no window-size check
}
```

`convertTimestamp()` (lines 76–81) only guards against overflow (`seconds >= 9223372035L`); it does not reject arbitrarily old timestamps.

The built filter is passed to `TopicMessageServiceImpl.subscribeTopic(@Valid TopicMessageFilter filter)` (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageService.java` line 12). Bean validation fires via `@Validated` on `TopicMessageServiceImpl`, but `TopicMessageFilter`'s constraints (`grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java` lines 43–51) only enforce:

- `startTime >= 0` (`@Min(0)`) — epoch (1 s → 1,000,000,000 ns) passes.
- `startTime <= DomainUtils.now()` — any past timestamp passes.
- `endTime > startTime` — `Long.MAX_VALUE > 1,000,000,000` passes.

**No constraint on `endTime - startTime` (window size) exists.**

`TopicMessageServiceImpl.subscribeTopic()` immediately calls `topicMessageRetriever.retrieve(filter, true)` (line 63), which enters `PollingTopicMessageRetriever.retrieve()`. In throttled mode (`PollingContext`, lines 94–108):

- `maxPageSize = 1000` records per query
- `pollingFrequency = 2 s`
- `numRepeats = Long.MAX_VALUE`
- `isComplete()` returns `true` only when a page returns fewer than 1000 rows (i.e., the last page of the entire historical dataset)

The retriever keeps issuing `topicMessageRepository.findByFilter(newFilter)` every 2 seconds until either the dataset is exhausted or the 60-second `retrieverProperties.getTimeout()` fires. With a dataset spanning years, this means ~30 DB queries per connection before timeout.

**No rate limiting exists in the gRPC module.** The `ThrottleConfiguration`/`ThrottleManagerImpl` classes exist only in the `web3` module. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per TCP connection but imposes no cap on the number of TCP connections from a single IP.

### Impact Explanation

Each malicious subscription with `startTime=epoch, endTime=Long.MAX_VALUE` drives ~30 sequential full-history DB scans (1000 rows each) over 60 seconds. With N concurrent connections (each allowed 5 calls), an attacker with N/5 TCP connections generates 30N DB queries simultaneously. On a production mirror node with years of HCS history, each query performs a large index range scan. Sustained attack degrades DB throughput for all legitimate users. Severity: **Medium** (griefing / availability impact, no economic loss, no data exfiltration).

### Likelihood Explanation

The gRPC endpoint is publicly accessible with no authentication. The attack requires only a standard gRPC client (e.g., `grpcurl`) and knowledge of any valid topic ID. It is trivially scriptable, repeatable, and requires no special privileges or on-chain resources.

### Recommendation

1. **Enforce a maximum lookback window**: Add a constraint to `TopicMessageFilter` (or in `toFilter()`) rejecting `startTime` older than a configurable maximum (e.g., 30 days before `now`).
2. **Enforce a maximum time window size**: Reject requests where `endTime - startTime` exceeds a configurable threshold.
3. **Add gRPC-level rate limiting**: Implement a per-IP or global subscription rate limiter in `ConsensusController` or via a gRPC interceptor, analogous to the `ThrottleManagerImpl` in the `web3` module.
4. **Cap total concurrent subscriptions**: Add a global subscriber limit check before accepting new subscriptions (the `subscriberCount` gauge already exists but is not used as a gate).

### Proof of Concept

```bash
# Using grpcurl against a running mirror node gRPC endpoint
# Set startTime to Unix epoch (seconds=1) and endTime to far future
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 1, "nanos": 0},
    "consensusEndTime": {"seconds": 9223372034, "nanos": 999999999}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# Repeat in a loop to open many concurrent connections
for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{
      "topicID": {"topicNum": 1234},
      "consensusStartTime": {"seconds": 1, "nanos": 0},
      "consensusEndTime": {"seconds": 9223372034, "nanos": 999999999}
    }' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Each connection drives ~30 DB queries over 60s; 100 connections = ~3000 DB queries
# Monitor DB CPU/I/O to observe exhaustion
```

**Validation of checks failing:**
- `isValidStartTime()`: `1_000_000_000 <= DomainUtils.now()` → `true` (passes) [1](#0-0) 
- `isValidEndTime()`: `Long.MAX_VALUE > 1_000_000_000` → `true` (passes) [2](#0-1) 
- `toFilter()` applies no window-size check [3](#0-2) 
- Retriever polls indefinitely until timeout [4](#0-3) 
- No gRPC-layer rate limiter exists [5](#0-4)
