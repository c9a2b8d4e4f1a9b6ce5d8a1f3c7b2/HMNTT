### Title
Unauthenticated HTTP Redirect Injection Allows Full Node-Map Poisoning via `buildNetworkFromMirrorNodes()`

### Summary
`buildNetworkFromMirrorNodes()` constructs a plain `http.Client` with no `CheckRedirect` policy and no TLS enforcement. A network-adjacent attacker who can intercept the plaintext HTTP connection to the mirror REST endpoint can respond with an HTTP 301/302 redirect to an attacker-controlled server. Go's default redirect-following behavior silently fetches the attacker's crafted `nodesEnvelope`, which maps every Hedera node account ID to attacker-controlled gRPC endpoints. The pinger then uses this poisoned map as its entire network, routing all subsequent transactions away from the legitimate Hedera network.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 46:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```
No `CheckRedirect` field is set. Go's `net/http` package default (`defaultCheckRedirect`) follows up to 10 redirects unconditionally, including cross-origin redirects to arbitrary hosts.

The constructed URL (`cfg.mirrorRest + "/api/v1/network/nodes"`) is sourced from the environment variable `HIERO_MIRROR_PINGER_REST`, whose documented example value is `http://mirror-rest:5551` — plaintext HTTP with no TLS.

`fetchMirrorNodeNetwork()` (lines 84–93) performs the request and then checks:
```go
if resp.StatusCode < 200 || resp.StatusCode >= 300 {
    ...
}
```
This check operates on the **final** response after all redirects have already been followed. It does not detect that a redirect occurred, does not validate the final URL's host/scheme, and does not compare it against the originally configured `cfg.mirrorRest`.

The returned `network` map is passed directly to `hiero.ClientForNetwork(netmap)` in `sdk_client.go` line 22, with no further validation of the endpoint addresses.

**Root cause:** The `http.Client` is created with only a timeout; no `CheckRedirect` to block cross-origin redirects, no `Transport` enforcing TLS, and no post-request URL origin check.

**Exploit flow:**
1. Attacker achieves a MITM position on the plaintext HTTP path between the pinger pod and the mirror REST service (e.g., ARP spoofing, DNS poisoning, rogue pod on the same Kubernetes overlay network, or a compromised ingress/proxy).
2. Attacker intercepts `GET http://mirror-rest:5551/api/v1/network/nodes`.
3. Attacker replies with `HTTP/1.1 301 Moved Permanently` + `Location: http://attacker.example.com/nodes`.
4. Go's `http.Client` silently follows the redirect and issues `GET http://attacker.example.com/nodes`.
5. Attacker's server returns a valid-looking `nodesEnvelope` JSON with all `node_account_id` values mapped to attacker-controlled `service_endpoints` (IP/port pairs pointing to attacker gRPC servers).
6. `fetchMirrorNodeNetwork()` decodes this payload, builds the `network` map, and returns it without error.
7. `hiero.ClientForNetwork(netmap)` configures the Hedera SDK client to use only attacker endpoints.
8. Every subsequent `submitWithRetry` call sends signed transactions to the attacker's gRPC servers.

### Impact Explanation
The attacker gains full control over where the pinger sends Hedera transactions. Consequences include:
- **Total transaction suppression:** Attacker's gRPC servers can drop all transactions, causing the pinger to report continuous failures — a complete monitoring/liveness blackout.
- **Transaction replay/inspection:** Attacker receives fully signed transactions (including operator key signatures) and can log, delay, or selectively replay them.
- **False confirmation:** Attacker can return synthetic success responses, masking a real network outage.

This directly matches the stated critical impact: "Network not being able to confirm new transactions (total network shutdown)."

### Likelihood Explanation
- The default and documented deployment uses plaintext HTTP (`http://mirror-rest:5551`), making passive interception trivial on any shared network segment.
- In Kubernetes, any compromised pod on the same CNI overlay can perform ARP/NDP spoofing or DNS poisoning without cluster-admin privileges.
- The attack is repeatable: the pinger calls `buildNetworkFromMirrorNodes()` at startup; if the attacker can sustain the MITM, every restart re-poisons the map.
- No authentication, no TLS, no redirect guard — the attacker needs only a MITM position, not any credential or system privilege on the target.

### Recommendation
1. **Disable redirect following entirely** by setting `CheckRedirect` to return an error:
   ```go
   httpClient := &http.Client{
       Timeout: cfg.mirrorNodeClientTimeout,
       CheckRedirect: func(req *http.Request, via []*http.Request) error {
           return http.ErrUseLastResponse
       },
   }
   ```
2. **Enforce HTTPS** for `cfg.mirrorRest` and reject plaintext URLs at config validation time.
3. **Validate the final response URL** against the configured base URL after the request completes.
4. **Pin the expected host** in a custom `http.Transport` with TLS certificate verification.
