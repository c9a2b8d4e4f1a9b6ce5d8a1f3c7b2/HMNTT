### Title
Contract Existence Enumeration via Divergent Error Responses in `getContractResultsById`

### Summary
When a caller supplies a valid EVM address as `:contractId` to `GET /api/v1/contracts/:contractId/results`, the handler calls `computeContractIdFromString`, which in turn calls `getContractIdByEvmAddress`. If the address is absent from the database, a `NotFoundError` with a descriptive message is thrown and the API returns HTTP 404. If the address exists but has no results, the API returns HTTP 200 with an empty `results` array. This asymmetry allows any unauthenticated caller to enumerate which EVM addresses correspond to deployed contracts.

### Finding Description
**Route registration** (`rest/routes/contractRoute.js`, line 14):
```
router.getExt('/:contractId/results', ContractController.getContractResultsById);
```
No authentication middleware is applied.

**Handler** (`rest/controllers/contractController.js`, lines 856–879):
```js
getContractResultsById = async (req, res) => {
  const {contractId: contractIdParam, filters} = extractContractIdAndFiltersFromValidatedRequest(...);
  const contractId = await ContractService.computeContractIdFromString(contractIdParam); // ← throws NotFoundError if EVM addr absent
  ...
  const rows = await ContractService.getContractResultsByIdAndFilters(...);
  if (rows.length === 0) {
    return;   // ← HTTP 200, body: {"results":[],"links":{"next":null}}
  }
  ...
};
```

**EVM address lookup** (`rest/service/contractService.js`, lines 452–466):
```js
async getContractIdByEvmAddress(evmAddressFilter) {
  const rows = await super.getRows(...);
  if (rows.length === 0) {
    throw new NotFoundError(`No contract with the given evm address 0x${create2EvmAddress} has been found.`);
  }
  ...
}
```

**Error handler** (`rest/middleware/httpErrorHandler.js`, lines 17–18, 31–32):
`NotFoundError` maps to HTTP 404 and the full `err.message` is returned to the client because `shouldReturnMessage` returns `true` for all client errors.

**Two observable states for any 40-hex-char EVM address:**

| Scenario | HTTP Status | Body |
|---|---|---|
| Address not in DB | 404 | `{"_status":{"messages":[{"message":"No contract with the given evm address 0x<addr> has been found."}]}}` |
| Address in DB, no results | 200 | `{"results":[],"links":{"next":null}}` |
| Address in DB, has results | 200 | `{"results":[...],...}` |

The root cause is that the EVM-address-to-contract-ID resolution step (`computeContractIdFromString` → `getContractIdByEvmAddress`) performs a database existence check and surfaces its outcome as a distinct HTTP status code, while the "no results" path silently returns 200.

### Impact Explanation
An unauthenticated attacker can probe any 40-hex-character EVM address and determine with certainty whether a contract is registered in the mirror node database. This leaks the set of deployed contract addresses, which may be considered sensitive (e.g., contracts not yet publicly announced, internal infrastructure contracts). The information is returned with a specific, machine-readable error message, making automated enumeration trivial.

### Likelihood Explanation
No authentication or rate-limiting is visible on this route. Any HTTP client can issue requests. The only precondition is knowing the EVM address format (40 hex chars, optionally prefixed with `0x`), which is public knowledge. The attack is fully repeatable and scriptable.

### Recommendation
Normalize the response for the EVM-address-not-found case to match the "contract exists but has no results" response: return HTTP 200 with `{"results":[],"links":{"next":null}}` instead of propagating the `NotFoundError`. Specifically, catch `NotFoundError` inside `getContractResultsById` (or in `computeContractIdFromString` for the EVM-address branch) and treat it as an empty result set rather than a 404. This eliminates the observable difference between "address not in DB" and "address in DB with no results."
