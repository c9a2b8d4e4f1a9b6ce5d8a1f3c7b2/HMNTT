### Title
Unbounded Concurrent Request Exhaustion in `getNftAllowances()` Due to Absent Rate Limiting in rest-java

### Summary
The `rest-java` service exposes `GET /api/v1/accounts/{id}/allowances/nfts` via `AllowancesController` with no rate limiting, concurrency cap, or request queue bound. During a database network partition, each inbound request blocks a Tomcat worker thread for up to the full `statementTimeout` (10 000 ms). An unprivileged attacker flooding this endpoint during such a partition exhausts the thread pool, causing the server to stop accepting new legitimate connections.

### Finding Description
**Code path:**
- Entry point: `AllowancesController.getNftAllowances()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, lines 57–83
- Service call: `NftAllowanceServiceImpl.getNftAllowances()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/NftAllowanceServiceImpl.java`, lines 21–31
- Both `entityService.lookup()` (line 28) and `repository.findAll()` (line 30) are synchronous, blocking DB calls.

**Root cause:**
The `web3` module has a full throttle stack (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j rate buckets) scoped to `hiero.mirror.web3.throttle`. The `rest-java` module has **no equivalent**. The only documented `rest-java` properties are DB connection settings and response headers — no `requestsPerSecond`, no concurrency filter, no accept-queue cap. The `@Max(MAX_LIMIT)` annotation on the `limit` parameter (controller line 61) only bounds result-set size, not request rate.

**Exploit flow:**
1. A network partition isolates the DB from rest-java. DB connections hang until `statementTimeout` (10 s).
2. Attacker sends a continuous stream of `GET /api/v1/accounts/0.0.1/allowances/nfts` requests (no auth required, public read endpoint).
3. Each request acquires a Tomcat worker thread and blocks inside `repository.findAll()` waiting for a DB response that never comes.
4. Tomcat's default thread pool (~200 threads) fills within seconds. The accept queue (~100 slots) fills next.
5. The OS TCP backlog fills. New TCP SYN packets are dropped. Legitimate clients receive connection-refused or timeout.
6. The condition persists for the full 10 s per wave; the attacker needs only ~20 req/s to sustain saturation.

**Why existing checks fail:**
- `@Positive @Max(MAX_LIMIT)` on `limit` (line 61) is a query-parameter validator, not a concurrency guard.
- `LoggingFilter` and `MetricsFilter` in `rest-java/config/` are observability-only; they do not reject or throttle requests.
- No Spring Security filter, no servlet filter, no interceptor in `WebMvcConfiguration` applies rate limiting to this path.

### Impact Explanation
During any DB-level network partition (planned maintenance, infrastructure failure, or attacker-induced), the rest-java service becomes completely unavailable to legitimate users. All endpoints sharing the same Tomcat thread pool are affected, not just the NFT allowances endpoint. This is a full denial-of-service against the mirror node's Java REST API surface.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. No special knowledge or credentials are required. The attacker needs only a modest HTTP flood tool (e.g., `ab`, `wrk`, or a simple script) and the ability to send ~200 concurrent requests. The attack is trivially repeatable and can be scripted to fire automatically whenever DB latency spikes above a threshold, making it opportunistic as well as deliberate.

### Recommendation
1. Add a rate-limiting filter to `rest-java` analogous to the `web3` `ThrottleManagerImpl` — apply bucket4j or a servlet filter capping requests per IP and globally per second for `/api/v1/accounts/*/allowances/nfts`.
2. Configure Tomcat's `server.tomcat.max-threads` and `server.tomcat.accept-count` explicitly in `rest-java`'s `application.yml` to bound queue depth.
3. Set a HikariCP `connectionTimeout` (e.g., 2 000 ms) so that requests fail fast when the pool is exhausted rather than blocking for the full `statementTimeout`.
4. Consider migrating the endpoint to Spring WebFlux (reactive) so that DB waits do not consume OS threads.

### Proof of Concept
```bash
# Step 1: Simulate DB partition (e.g., drop DB traffic via iptables on the DB host)
# iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# Step 2: Flood the endpoint from an unprivileged client
wrk -t 20 -c 300 -d 60s \
  "http://<rest-java-host>/api/v1/accounts/0.0.2/allowances/nfts"

# Step 3: In a separate terminal, attempt a legitimate request
curl -v "http://<rest-java-host>/api/v1/accounts/0.0.3/allowances/nfts"
# Expected result: connection refused or timeout — server queue full

# Step 4: Observe Tomcat metrics
curl http://<rest-java-host>/actuator/metrics/tomcat.threads.busy
# All threads busy; new requests rejected
```
