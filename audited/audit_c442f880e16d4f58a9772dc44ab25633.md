### Title
Unauthenticated Unbounded Protobuf Body Parsing DoS on POST /api/v1/network/fees

### Summary
The `estimateFees()` handler in `NetworkController.java` accepts an arbitrary-length binary body from any unauthenticated caller and immediately passes it to `Transaction.PROTOBUF.parse(Bytes.wrap(body))` with no body-size cap, no per-IP rate limit, and no concurrency throttle. An attacker can flood the endpoint with maximum-size protobuf payloads at high concurrency, exhausting heap memory and CPU on the server and degrading availability for all other users at zero economic cost.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 110–124:

```java
@PostMapping(
        consumes = {"application/protobuf", "application/x-protobuf"},
        value = "/fees")
FeeEstimateResponse estimateFees(
        @RequestBody @NotNull byte[] body,          // ← no @Size limit
        @RequestParam(defaultValue = "INTRINSIC", required = false) FeeEstimateMode mode,
        @RequestParam(name = HIGH_VOLUME_THROTTLE, defaultValue = "0", required = false) @Min(0) @Max(10000)
                int highVolumeThrottle) {
    try {
        final var transaction = Transaction.PROTOBUF.parse(Bytes.wrap(body)); // ← unbounded parse
        return toResponse(feeEstimationService.estimateFees(transaction, mode, highVolumeThrottle));
    } catch (ParseException e) {
        throw new IllegalArgumentException("Unable to parse transaction", e);
    }
}
```

**Root cause — three compounding missing controls:**

1. **No body-size limit.** The `@RequestBody byte[] body` parameter carries only `@NotNull`. There is no `@Size(max = …)` constraint, no `spring.servlet.multipart.max-request-size`, and no `server.tomcat.max-http-form-post-size` override in the rest-java module (the `rest-java/src/main/resources/` directory contains only `banner.txt`; no `application.yml` exists). Spring Boot's built-in `maxPostSize` guard applies only to `application/x-www-form-urlencoded`; it does **not** apply to `application/protobuf` bodies. The server will therefore buffer an arbitrarily large body into a heap `byte[]` before the controller is even entered.

2. **No rate limiting.** The entire throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`, bucket4j `Bucket` beans) lives exclusively in the `web3` module. The `rest-java` module has no equivalent. Searching `rest-java/**` for `RateLimiter`, `RateLimit`, `ThrottleFilter`, or `throttle` returns only references to the `highVolumeThrottle` *query parameter* (a fee-calculation input, not an access-control mechanism) and the `FeeEstimationService` internals.

3. **Jackson limits are irrelevant.** `JacksonConfiguration` in rest-java caps JSON document length at 1 000 characters, but the endpoint consumes `application/protobuf` / `application/x-protobuf`, so Jackson is never invoked.

**Exploit flow:**

- Spring reads the full request body into `byte[]` (heap allocation proportional to body size).
- `Bytes.wrap(body)` is zero-copy but `Transaction.PROTOBUF.parse(…)` iterates every byte, allocating nested message objects proportional to body size.
- With N concurrent connections each sending a large body, heap usage is O(N × body_size) and CPU is O(N × body_size).

### Impact Explanation

An attacker sending, for example, 500 concurrent HTTP/1.1 keep-alive connections each posting a 5 MB protobuf body causes ~2.5 GB of simultaneous heap allocation plus proportional CPU parse work. This can trigger GC pressure, OOM kills, or thread-pool saturation, making the service unavailable to legitimate callers. Because the endpoint is a pure read/compute path (no on-chain transaction), there is zero economic cost to the attacker and no on-chain trace.

### Likelihood Explanation

The endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 916–947), requires no authentication, and accepts a standard binary content type. Any attacker with a scripting tool (curl, wrk, ab) can reproduce the attack. The absence of both a body-size cap and a rate limiter means there is no application-level barrier; the only mitigation would be an external reverse proxy or WAF, which is not guaranteed in all deployment configurations.

### Recommendation

Apply all three fixes:

1. **Enforce a body-size cap at the Spring layer.** Add to `rest-java`'s Spring Boot configuration:
   ```yaml
   spring:
     servlet:
       multipart:
         max-request-size: 6KB   # HAPI Transaction max is ~6 KB
   ```
   Or enforce it programmatically with a `ContentCachingRequestWrapper` / `OncePerRequestFilter` that rejects bodies exceeding the HAPI transaction size limit (~6 KB).

2. **Add a `@Size` or custom validator on the `body` parameter** as a defence-in-depth layer:
   ```java
   @RequestBody @NotNull @Size(max = 6144) byte[] body
   ```

3. **Add per-IP rate limiting** for the `POST /api/v1/network/fees` endpoint, mirroring the bucket4j pattern already used in the `web3` module (`ThrottleConfiguration` / `ThrottleManagerImpl`).

### Proof of Concept

```bash
# Generate a 5 MB payload of valid-looking protobuf bytes
python3 -c "import sys; sys.stdout.buffer.write(b'\x0a' + b'\xff\xff\xff\x02' + b'A'*5_000_000)" > payload.bin

# Flood with 200 concurrent connections, 1000 requests each
wrk -t 20 -c 200 -d 30s \
    -s <(echo 'wrk.method="POST"
               wrk.body=io.open("payload.bin","rb"):read("*a")
               wrk.headers["Content-Type"]="application/protobuf"') \
    http://<mirror-node-host>/api/v1/network/fees
```

Observe: heap usage climbs rapidly, GC pauses increase, and legitimate requests to other endpoints begin timing out or returning 503.
