### Title
Global Gas Bucket Exhaustion via Single-Client DoS in `ThrottleManagerImpl`

### Summary
`ThrottleManagerImpl.throttle()` enforces gas consumption against a single process-wide `gasLimitBucket` with no per-IP or per-client partitioning. An unprivileged attacker can flood the `/api/v1/contracts/call` endpoint with maximum-gas requests from a single origin, exhausting the entire global gas quota and causing all concurrent legitimate clients to receive HTTP 429 "Gas per second rate limit exceeded" responses for the duration of the attack. Both the `gasLimitBucket` and `rateLimitBucket` are global singletons, so a single client can simultaneously monopolize both.

### Finding Description

**Code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `gasLimitBucket.tryConsume(scaleGas(request.getGas()))`

In `ThrottleManagerImpl.throttle()` (lines 37–42), the only gas enforcement is:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

`gasLimitBucket` is a single application-scoped `Bucket` bean constructed in `ThrottleConfiguration.gasLimitBucket()` (lines 34–45) with capacity `scaleGas(gasPerSecond)`. No caller identity (IP, token, session) is ever read or tracked anywhere in the throttle path.

**Root cause:** The failed assumption is that the global RPS cap (`requestsPerSecond = 500`) prevents any single client from consuming a disproportionate share of the gas quota. In reality, both limits are global, so a single client can consume 100% of both simultaneously.

**Exploit arithmetic (default config):**
- `gasPerSecond` = 7,500,000,000 → `scaleGas` → **750,000 tokens** bucket capacity
- `maxGasLimit` = 15,000,000 per request → `scaleGas` → **1,500 tokens** consumed per request
- `requestsPerSecond` = 500 (global)
- Single client sends 500 req/s × 1,500 tokens = **750,000 tokens/s** → entire gas bucket drained in ≤1 second

`rateLimitBucket` (lines 24–32 of `ThrottleConfiguration`) is equally global and is also fully consumed by the same 500 req/s flood, so the attacker simultaneously blocks all other clients at both the RPS and gas layers.

**Why `restore()` does not mitigate this:** `restore()` is called only after the EVM simulation completes. During the window between `throttle()` and `restore()`, the tokens are already consumed. An attacker sending requests faster than they complete keeps the bucket perpetually empty.

**Why `validateContractMaxGasLimit` does not mitigate this:** In `ContractController.call()` (line 40–41), `throttleManager.throttle(request)` is called *before* `validateContractMaxGasLimit(request)`. Gas tokens are consumed from the bucket before the gas ceiling check fires. If gas exceeds the max, `restore()` is called, but the attacker simply uses `gas = 15_000_000` (exactly at the limit) to avoid triggering the restore path entirely.

### Impact Explanation

Any unauthenticated HTTP client can render the mirror node's entire web3 simulation API (`/api/v1/contracts/call`, `eth_call`, `eth_estimateGas`) unavailable to all other users for as long as the flood continues. All legitimate callers receive HTTP 429 with no ability to distinguish themselves from the attacker. This is a complete, sustained denial-of-service of the gas-consuming API surface with zero cost to the attacker beyond network bandwidth. Note: this is a mirror node (read-only simulation service); it does not produce Hedera consensus blocks, so on-chain block production is unaffected. The impact is confined to the mirror node's web3 API availability.

### Likelihood Explanation

No authentication, no API key, no account, and no special network position is required. Any client that can reach the HTTP endpoint can execute this attack with a trivial script (e.g., `ab`, `wrk`, or a simple loop). The attack is repeatable indefinitely and self-sustaining as long as the attacker maintains ≥500 req/s throughput. Default deployments are fully exposed.

### Recommendation

1. **Per-client gas bucket:** Maintain a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`) and enforce per-client gas limits in addition to the global cap.
2. **Per-client RPS bucket:** Apply the same per-IP partitioning to `rateLimitBucket`.
3. **Consume gas after validation:** Move `validateContractMaxGasLimit` before `throttleManager.throttle()` to avoid consuming tokens for requests that will be immediately rejected.
4. **Consider Bucket4j's distributed or per-key API** to avoid maintaining unbounded maps; evict idle entries with a time-based cache (e.g., Caffeine).

### Proof of Concept

```bash
# Exhaust the global gas bucket from a single client
# Requires: curl, GNU parallel or wrk

# Using wrk (HTTP benchmarking tool):
wrk -t4 -c500 -d60s -s post.lua http://<mirror-node-host>/api/v1/contracts/call

# post.lua:
# wrk.method = "POST"
# wrk.headers["Content-Type"] = "application/json"
# wrk.body = '{"block":"latest","data":"0x","to":"0x0000000000000000000000000000000000000001","gas":15000000,"estimate":false}'

# Expected result while attack runs:
# All other clients receive:
# HTTP/1.1 429 Too Many Requests
# {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

**Reproducible steps:**
1. Start the mirror node web3 service with default configuration.
2. From a second terminal (legitimate user), send a single `POST /api/v1/contracts/call` with `gas=21000` — observe HTTP 200.
3. From the attacker terminal, launch `wrk` as above at ≥500 req/s with `gas=15000000`.
4. Retry the legitimate request from step 2 — observe HTTP 429 "Gas per second rate limit exceeded."
5. Stop the attacker; within ~1 second the bucket refills and legitimate requests succeed again.
