### Title
Log Spoofing via Duplicate JSON Keys and `reorderFields` Truncation in `LoggingFilter`

### Summary
`LoggingFilter.getContent()` applies a non-greedy regex reorder of the `"data"` field before truncating the payload log to `maxPayloadLogSize` (default 300). An unprivileged attacker can craft a request body with duplicate `"block"` and `"data"` JSON keys such that the first (fake) `"block"` value is promoted to the front of the log by `reorderFields()`, while the real `"block"` value is pushed past the truncation boundary and never logged. The actual query executes against the real block, but the audit log records the fake one.

### Finding Description

**Code path:**

- `Web3Properties.java` line 17: `maxPayloadLogSize = 300`
- `LoggingFilter.java` line 33: `DATA_PATTERN = Pattern.compile("(\"data\":.*?),(.+)(}[^}]*)$")`
- `LoggingFilter.java` lines 105–108: `reorderFields()` is called, then `StringUtils.substring(content, 0, maxPayloadLogSize)` truncates

**Root cause:**

`reorderFields()` uses a non-greedy `.*?` match on the *first* `"data":` occurrence and moves it to the end. It does not validate or deduplicate JSON keys. When the attacker supplies two `"data"` keys and two `"block"` keys, the regex promotes the *first* `"block"` value to the front of the reordered string. The *second* (real) `"block"` value is buried after the large padding `"data"` value and is cut off by the 300-char truncation.

**Exploit flow:**

1. Attacker sends POST body (e.g., to `/api/v1/contracts/call`):
   ```
   {"data":"x","block":"0xFAKE","data":"AAAA...~280 chars...","block":"0xREAL"}
   ```
2. `StringUtils.deleteWhitespace()` leaves it compact.
3. Length > 300 → compression attempted; random hex-like padding does not compress below 300 → falls through to truncation branch.
4. `reorderFields()` regex matches first `"data":"x"` (group 1), captures `"block":"0xFAKE","data":"AAAA...","block":"0xREAL"` as group 2, produces:
   ```
   {"block":"0xFAKE","data":"AAAA...~280 chars...","block":"0xREAL","data":"x"}
   ```
5. `StringUtils.substring(..., 0, 300)` cuts the string at 300 chars, leaving only:
   ```
   {"block":"0xFAKE","data":"AAAA...
   ```
6. Log records `block=0xFAKE`. Jackson (last-key-wins for duplicates) executes the query against `block=0xREAL`.

**Why existing checks fail:**

`reorderFields()` is purely a cosmetic reorder with no key deduplication or sanitization. The truncation guard (`status < 500`) only skips truncation for server errors; a successful `200` response still truncates. There is no check that the logged payload faithfully represents the parsed request.

### Impact Explanation

An unprivileged attacker can systematically query any historical Hashgraph block while the audit log records a different (attacker-chosen) block number. This breaks the integrity of access audit trails for historical ledger data. Any forensic or compliance review of which blocks were accessed is rendered unreliable. Severity: Medium (audit integrity bypass, no direct data exfiltration, but undermines accountability controls).

### Likelihood Explanation

No authentication or special privilege is required. The payload is a standard JSON POST body. The attacker only needs to know the endpoint accepts a `"block"` parameter (publicly documented for `eth_call`-style endpoints). The attack is trivially repeatable and scriptable. The only constraint is keeping the total payload under `maxPayloadLogSize * 10` (3000 chars), which is easily satisfied.

### Recommendation

1. **Deduplicate JSON keys before logging**: parse the raw body with a strict JSON parser that rejects or normalizes duplicate keys before passing to `reorderFields()`.
2. **Log the raw, pre-parse body separately** from the processed representation, or log a cryptographic hash of the full body alongside the truncated version so truncation cannot silently drop fields.
3. **Sanitize `"block"` field independently**: extract and log the `block` parameter directly from the parsed request object (which the application already uses), not from the raw string after regex manipulation.
4. Alternatively, replace the regex-based `reorderFields()` with a proper JSON parse-and-reserialize step that is immune to duplicate-key injection.
