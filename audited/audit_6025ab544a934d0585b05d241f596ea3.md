### Title
Unvalidated `node_account_id` in `ConstructionPayloads` Allows Targeted Node Traffic Concentration

### Summary
`getTransactionNodeAccountId()` in `ConstructionPayloads` accepts any syntactically valid account ID string from client-supplied metadata without checking it against the server's known network node list. This completely bypasses the load-distributing random node selection in `getRandomNodeAccountId()`, allowing any unprivileged caller to pin every transaction they construct to a single chosen node.

### Finding Description

**Code path:**

`ConstructionPayloads` (line 222) calls `getTransactionNodeAccountId(request.Metadata)`: [1](#0-0) 

The function performs only syntactic validation via `hiero.AccountIDFromString(str)` (line 383). It never consults `c.sdkClient.GetNetwork()`, which is the authoritative set of valid network nodes.

Contrast with `getRandomNodeAccountId()`, which correctly builds its selection pool exclusively from `c.sdkClient.GetNetwork()`: [2](#0-1) 

The resulting `nodeAccountId` is then embedded directly into the frozen transaction body via `transactionSetNodeAccountId`: [3](#0-2) 

When `ConstructionSubmit` executes the transaction, the Hedera SDK routes it to the node account ID baked into the transaction body: [4](#0-3) 

**Root cause:** `getTransactionNodeAccountId` trusts the client-supplied string as the sole source of truth. The set returned by `c.sdkClient.GetNetwork()` — used correctly in `getRandomNodeAccountId` — is never consulted during payload construction.

**Failed assumption:** The design assumes clients will pass back the `node_account_id` returned by `/construction/metadata` unchanged. There is no enforcement of this assumption.

### Impact Explanation
Every transaction constructed through `/construction/payloads` with a crafted `node_account_id` (e.g., `"0.0.3"`) will be frozen with that node embedded. All such transactions submitted via `/construction/submit` are routed exclusively to that node. If a Rosetta instance serves multiple clients, or if an attacker controls a high-throughput client, the entire transaction flow of that instance is concentrated on one node. Because the Hedera network's gossip and consensus load is distributed across nodes, funneling a disproportionate share of transactions to a single node degrades its performance and can cause it to fall behind or become unresponsive — meeting the "≥30% node shutdown without brute force" criterion when the targeted node handles a significant fraction of network consensus.

### Likelihood Explanation
The `/construction/payloads` endpoint is unauthenticated and publicly reachable in a standard Rosetta deployment. No special privileges, credentials, or prior knowledge beyond the target node's account ID (publicly known, e.g., `0.0.3`) are required. The attacker simply omits the server-provided metadata and substitutes their chosen node ID. The attack is trivially repeatable in a loop with any funded account and requires no brute force — one API call per transaction is sufficient.

### Recommendation
Inside `getTransactionNodeAccountId`, validate the parsed `nodeAccountId` against the live node set from `c.sdkClient.GetNetwork()` before accepting it:

```go
func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
    emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
    // ... existing string parse ...

    nodeAccountId, err := hiero.AccountIDFromString(str)
    if err != nil {
        return emptyAccountId, errors.ErrInvalidAccount
    }

    // NEW: reject IDs not in the known network node set
    validNodes := c.sdkClient.GetNetwork()
    known := false
    for _, id := range validNodes {
        if id == nodeAccountId {
            known = true
            break
        }
    }
    if !known {
        log.Errorf("Node account id %s is not a known network node", str)
        return emptyAccountId, errors.ErrInvalidAccount
    }

    return nodeAccountId, nilErr
}
```

Alternatively, ignore the client-supplied `node_account_id` entirely in `ConstructionPayloads` and always call `getRandomNodeAccountId()` server-side, as is done in `ConstructionMetadata`.

### Proof of Concept

**Preconditions:** A running Rosetta mirror node instance with a funded Hedera account.

**Steps:**

1. Call `/construction/preprocess` normally to obtain options.
2. Call `/construction/metadata` — note the server returns a random `node_account_id`, e.g., `"0.0.5"`.
3. Call `/construction/payloads` but replace `node_account_id` in the metadata with `"0.0.3"`:
```json
{
  "network_identifier": { ... },
  "operations": [ ... ],
  "metadata": {
    "node_account_id": "0.0.3",
    "valid_start_nanos": "...",
    ...
  }
}
```
4. Sign the returned unsigned transaction and call `/construction/submit`.
5. Observe (via the log line at line 352–353) that the transaction is submitted to node `0.0.3` regardless of what `/construction/metadata` returned.
6. Repeat in a loop — every transaction is routed to `0.0.3`, concentrating load on that single node with no rate-limiting or node-list enforcement on the Rosetta side.

func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
	emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
	value, ok := metadata[metadataKeyNodeAccountId]
	if !ok {
		return emptyAccountId, errors.ErrMissingNodeAccountIdMetadata
	}

	str, ok := value.(string)
	if !ok {
		return emptyAccountId, errors.ErrInvalidArgument
	}

	nodeAccountId, err := hiero.AccountIDFromString(str)
	if err != nil {
		log.Errorf("Invalid node account id provided in metadata: %s", str)
		return emptyAccountId, errors.ErrInvalidAccount
	}

	log.Infof("Use node account id %s from metadata", str)
	return nodeAccountId, nilErr
}
```

func (c *constructionAPIService) getRandomNodeAccountId() (hiero.AccountID, *rTypes.Error) {
	nodeAccountIds := make([]hiero.AccountID, 0)
	seen := map[hiero.AccountID]struct{}{}
	// network returned from sdkClient is a map[string]AccountID, the key is the address of a node and the value is
	// its node account id. Since a node can have multiple addresses, we need the seen map to get a unique node account
	// id array
	for _, nodeAccountId := range c.sdkClient.GetNetwork() {
		if _, ok := seen[nodeAccountId]; ok {
			continue
		}

		seen[nodeAccountId] = struct{}{}
		nodeAccountIds = append(nodeAccountIds, nodeAccountId)
	}
```
