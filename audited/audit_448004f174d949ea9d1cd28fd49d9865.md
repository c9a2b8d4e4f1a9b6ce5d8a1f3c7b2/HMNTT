### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent NFT Info Requests

### Summary
The `getNftTokenInfoRequest` handler in `rest/tokens.js` issues a direct database query on every invocation with no mandatory application-level caching and no per-IP rate limiting for unauthenticated users. An attacker can flood `GET /api/v1/tokens/:tokenId/nfts/:serialNumber` with concurrent requests, exhausting the finite pg connection pool and causing legitimate requests to time out or queue indefinitely.

### Finding Description

**Code path:**

`rest/tokens.js` lines 866–880 (`getNftTokenInfoRequest`) calls `extractSqlFromNftTokenInfoRequest` (lines 795–804), which builds a parameterized query against the `nft` table with a `LEFT JOIN entity`:

```
select nft.account_id, ... from nft
  left join entity e on e.id = nft.token_id
where nft.token_id = $1 and nft.serial_number = $2
```

This query is executed unconditionally via `pool.queryQuietly(query, params)` on every request. [1](#0-0) 

**Root cause — no mandatory caching:**

The Redis-based response cache (`responseCacheCheckHandler` / `responseCacheUpdateHandler`) is only active when `applicationCacheEnabled = config.cache.response.enabled && config.redis.enabled`. [2](#0-1)  This is an opt-in feature; Redis is not required. Even when enabled, the cache TTL for NFT list endpoints is `max-age=1` (1 second), meaning the cache window is extremely narrow. [3](#0-2) 

**Root cause — no rate limiting:**

`server.js` registers no rate-limiting middleware for unauthenticated users. The `authHandler` only sets a custom response-size limit for authenticated users with credentials; it does not throttle or reject unauthenticated requests. [4](#0-3)  No `express-rate-limit` or equivalent middleware appears in the server setup. [5](#0-4) 

**Root cause — finite pool:**

The pg pool is configured with `max: config.db.pool.maxConnections`. [6](#0-5)  When all slots are occupied, new queries queue until `connectionTimeoutMillis` expires, at which point they fail with a timeout error.

### Impact Explanation

Under sustained concurrent load from a single unauthenticated attacker, all pool slots become occupied by in-flight NFT queries. Legitimate requests across **all** REST API endpoints (accounts, transactions, tokens, etc.) share the same pool and begin timing out. The service degrades to a non-functional state for all users without any privileged access required. The `statement_timeout` limits individual query duration but does not prevent pool saturation when the request rate exceeds the pool's throughput capacity.

### Likelihood Explanation

No preconditions beyond network access are required. Any valid `tokenId`/`serialNumber` pair (publicly discoverable via the list endpoint `GET /api/v1/tokens/:tokenId/nfts`) is sufficient. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) sending hundreds of concurrent connections can reproduce this. The attack is repeatable and requires no authentication, no special knowledge, and no state.

### Recommendation

1. **Mandatory application-level rate limiting**: Add `express-rate-limit` (or equivalent) middleware keyed on client IP before route handlers, with a low burst ceiling for unauthenticated requests.
2. **Mandatory response caching**: Make the Redis response cache non-optional for the NFT info endpoint, or add an in-process LRU cache (e.g., using `lru-cache`) for individual NFT lookups that does not depend on Redis availability.
3. **Increase cache TTL**: The current 1-second `max-age` for NFT endpoints is too short to provide meaningful protection; raise it to at least 5–10 seconds.
4. **Pool queue limit**: Configure `pg` pool with a `maxQueue` or `queueTimeoutMillis` to shed excess load gracefully rather than queuing unboundedly.

### Proof of Concept

```bash
# 1. Discover a valid token/serial pair
curl https://<host>/api/v1/tokens/0.0.1500/nfts | jq '.nfts[0]'
# Returns serial_number: 1

# 2. Flood the individual NFT info endpoint with concurrent unauthenticated requests
wrk -t 20 -c 500 -d 30s \
  "https://<host>/api/v1/tokens/0.0.1500/nfts/1"

# 3. Observe: legitimate requests to other endpoints begin returning 503 or
#    connection timeout errors as the pg pool is exhausted.
#    Pool saturation can be confirmed via DB metrics showing
#    max_connections reached on the mirror_rest user.
```
