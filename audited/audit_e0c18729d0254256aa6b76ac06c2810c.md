### Title
Uncached EVM Address DB Lookup in `getTokenRelationships()` Enables Unauthenticated Non-Network DoS via DB Connection Pool Exhaustion

### Summary
Every call to `GET /api/v1/accounts/{evmAddress}/tokens` with a non-long-zero (opaque) EVM address triggers two uncached synchronous database queries — one in `EntityService.getEntityIdFromEvmAddress()` and one in `EntityService.isValidAccount()` — with no rate limiting on the REST API. The default DB connection pool is only 10 connections. An unauthenticated attacker flooding this endpoint with concurrent requests exhausts the pool, causing all REST API queries to queue or timeout, resulting in a non-network-based DoS.

### Finding Description

**Exact code path:**

`getTokenRelationships()` unconditionally calls `EntityService.getEncodedId()` on every request: [1](#0-0) 

Inside `getEncodedId()`, when the path parameter is a valid opaque EVM address (non-long-zero, i.e., `evmAddress !== null` after parsing), it calls `getEntityIdFromEvmAddress()`: [2](#0-1) 

`getEntityIdFromEvmAddress()` executes `entityFromEvmAddressQuery` directly against the DB on every invocation with no caching: [3](#0-2) [4](#0-3) 

If the entity is found, `isValidAccount()` immediately fires a second uncached DB query: [5](#0-4) [6](#0-5) 

**Root cause — failed assumption:** The `parseCached` LRU in `entityId.js` only caches the *parsed* `EntityId` object (shard/realm/num/evmAddress string), not the DB lookup result: [7](#0-6) 

The Redis `Cache` class in `cache.js` is used only for response-level caching (disabled by default), not for entity-ID resolution: [8](#0-7) 

**DB pool is finite and small by default (10 connections):** [9](#0-8) 

Default `maxConnections = 10` per documentation: [10](#0-9) 

**No rate limiting on the REST API:** The REST middleware (`requestHandler.js`) contains no throttle or rate-limit logic. The throttle configuration found (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies exclusively to the `web3` Java module, not the Node.js REST API: [11](#0-10) 

### Impact Explanation
With a default pool of 10 DB connections and 2 uncached DB queries per request, an attacker needs only ~5 concurrent persistent connections to saturate the pool. All other REST API users experience connection timeouts (`connectionTimeoutMillis = 20000 ms` default), effectively taking down the entire REST API. Because the EVM address parsing is cached (LRU), the attacker's requests pass validation instantly and proceed directly to the DB query on every call, maximizing pool pressure. This constitutes a non-network-based DoS affecting the REST API layer.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a single valid opaque EVM address (any 40-hex-char address that is not a long-zero address, e.g., any real contract address on the network). The endpoint is publicly documented and accessible. The attack is trivially repeatable with a simple HTTP flood tool (e.g., `ab`, `wrk`, or a short script). No special knowledge beyond the public API spec is required.

### Recommendation
1. **Cache the EVM address → entity ID resolution result** in `getEntityIdFromEvmAddress()` using the existing in-process LRU or Redis layer, keyed by the hex EVM address string, with a short TTL (e.g., 30–60 seconds).
2. **Add rate limiting to the REST API** (e.g., via an express middleware such as `express-rate-limit`) on a per-IP or global basis, particularly for endpoints that trigger DB lookups.
3. **Increase the default `maxConnections`** pool size or add a request queue depth limit so pool exhaustion does not cascade to a full API outage.
4. Consider **short-circuiting** `isValidAccount()` by reusing the entity row already fetched in `getEntityIdFromEvmAddress()` rather than issuing a second independent query.

### Proof of Concept

**Preconditions:** Mirror node REST API is running. Any valid opaque EVM address is known (e.g., any deployed contract address: `0xabcdef1234567890abcdef1234567890abcdef12`).

**Steps:**

```bash
# Send 50 concurrent requests with no delay
wrk -t10 -c50 -d60s \
  "http://<mirror-node-host>:5551/api/v1/accounts/0xabcdef1234567890abcdef1234567890abcdef12/tokens"
```

**Trigger:** Each request enters `getTokenRelationships()` → `getEncodedId()` → `getEntityIdFromEvmAddress()` (DB query 1) → `isValidAccount()` (DB query 2). With 50 concurrent requests and a pool of 10, 40 requests queue waiting for a connection.

**Result:** Legitimate API requests to any endpoint begin timing out after `connectionTimeoutMillis` (default 20 s). The REST API becomes unresponsive for all users. Stopping the flood immediately restores service (no persistent state corruption).
