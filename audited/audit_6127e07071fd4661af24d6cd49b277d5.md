### Title
Insufficient `validUntilNanos` Bound Check Allows Construction of Transactions with Near-Epoch Start Time

### Summary
In `ConstructionPayloads()`, the `getTransactionTimestampProperty()` helper derives `validStartNanos = validUntil - maxValidDurationNanos` and only rejects the result when it is `<= 0`. An unprivileged caller can supply a `validUntilNanos` value just above `maxValidDurationNanos` (e.g., `180_000_000_001`), yielding `validStartNanos = 1` (1 ns after Unix epoch), which passes the guard. The server then fully constructs, freezes, and serialises a transaction whose valid-start timestamp is in January 1970, a transaction the Hedera network will always reject, while consuming server-side CPU and memory for every such request.

### Finding Description
**Exact code path:**

`ConstructionPayloads()` (line 227) calls `getTransactionTimestampProperty()`:

```
rosetta/app/services/construction_service.go  lines 393-414
```

```go
validUntil, err = c.getIntMetadataValue(metadata, metadataKeyValidUntilNanos)
// ...
validDurationSeconds = maxValidDurationSeconds
validStartNanos = validUntil - maxValidDurationNanos   // line 407
if validStartNanos <= 0 {                              // line 408 — only guard
    err = errors.ErrInvalidArgument
}
```

`getIntMetadataValue()` (lines 540-555) only rejects values that are negative or non-numeric; it accepts any non-negative int64.

`maxValidDurationNanos` is `180 * 1_000_000_000 = 180_000_000_000` (line 33).

**Root cause:** The guard `validStartNanos <= 0` is the sole check. There is no upper-bound check on `validUntilNanos` and no check that `validStartNanos` is within a plausible range of the current wall-clock time.

**Exploit flow:**

1. Attacker sends `POST /construction/payloads` with `metadata["valid_until_nanos"] = "180000000001"`.
2. `getIntMetadataValue` parses `180_000_000_001` — non-negative, accepted.
3. `validStartNanos = 180_000_000_001 − 180_000_000_000 = 1` — passes `> 0` check.
4. `transactionSetTransactionId` (line 733) calls `hiero.NewTransactionIDWithValidStart(payer, time.Unix(0, 1))` — sets valid-start to 1970-01-01T00:00:00.000000001Z.
5. The transaction is frozen and serialised; the full `ConstructionPayloadsResponse` is returned.
6. The Hedera network rejects any submission because the valid-start is ~55 years in the past.

**Why existing checks fail:**

- `validStartNanos <= 0` only catches zero/negative values; `1` is positive and passes.
- `getIntMetadataValue` rejects `< 0` but not arbitrarily small positive values.
- No rate-limiting or authentication is required on the `/construction/payloads` endpoint.
- The test suite explicitly covers `"180000000000"` (zero result, rejected) and `"179999999999"` (negative result, rejected) but has no test for `"180000000001"` (result = 1, accepted). [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every crafted request causes the server to execute the full transaction-construction pipeline (operation parsing, account resolution, SDK transaction build, freeze, serialise) and return a useless payload. Because the endpoint requires no authentication and no on-chain funds, an attacker can loop this indefinitely to exhaust CPU and memory on the mirror-node Rosetta service. The impact is denial-of-service / griefing with no direct economic loss to network participants, matching the stated medium severity.

### Likelihood Explanation
The endpoint is publicly reachable in any standard Rosetta deployment. The exploit requires only a single HTTP POST with one crafted integer field — no keys, no accounts, no fees. It is trivially scriptable and repeatable at high frequency. Any external user who has read the Rosetta Construction API spec can discover this within minutes.

### Recommendation
Add a lower-bound check that `validStartNanos` is within a reasonable window of the current time, for example:

```go
validStartNanos = validUntil - maxValidDurationNanos
if validStartNanos <= 0 {
    err = errors.ErrInvalidArgument
    return
}
// NEW: reject timestamps that are unreasonably far in the past
minAcceptableStartNanos := time.Now().UnixNano() - maxValidDurationNanos
if validStartNanos < minAcceptableStartNanos {
    log.Errorf("Valid start nanos %d is too far in the past", validStartNanos)
    err = errors.ErrInvalidArgument
    return
}
```

Alternatively, add an upper-bound check on `validUntilNanos` itself (e.g., `validUntil > time.Now().UnixNano() + someGracePeriod`).

func (c *constructionAPIService) getIntMetadataValue(metadata map[string]any, metadataKey string) (int64, *rTypes.Error) {
	var metadataValue int64

		}

	}
