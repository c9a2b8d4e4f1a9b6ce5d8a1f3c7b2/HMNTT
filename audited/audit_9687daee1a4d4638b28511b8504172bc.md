### Title
Unauthenticated DoS via Unbounded Timestamp Range and Missing Rate Limiting on GET /blocks

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` processes timestamp filters by mapping them directly to SQL WHERE conditions without invoking `parseTimestampFilters()`, which is the utility that enforces `maxTimestampRangeNs` (default 7 days). Combined with the absence of any per-IP rate limiting on the REST API, an unauthenticated attacker can flood the endpoint with wide-range timestamp queries at maximum limit, exhausting the default 10-connection database pool and causing denial of service for all other API consumers.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` → `BlockController.getBlocks` → `extractSqlFromBlockFilters` → `RecordFileService.getBlocks`

In `rest/controllers/blockController.js`, `extractSqlFromBlockFilters` (lines 63–88) maps timestamp filters directly to SQL WHERE conditions on `consensus_end`:

```js
case filterKeys.TIMESTAMP:
  return this.getFilterWhereCondition(RecordFile.CONSENSUS_END, f);
``` [1](#0-0) 

It never calls `utils.parseTimestampFilters()`, which is the function that enforces `maxTimestampRangeNs`:

```js
if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
  throw new InvalidArgumentError(`Timestamp range ... must be ... within ${maxTimestampRange}`);
}
``` [2](#0-1) 

By contrast, other endpoints (e.g., `contractController.js`) explicitly call `parseTimestampFilters` to enforce the 7-day default cap. [3](#0-2) 

The resulting SQL issued by `RecordFileService.getBlocks` (lines 149–162) is:

```sql
SELECT ... FROM record_file
WHERE consensus_end >= $1 AND consensus_end <= $2
ORDER BY consensus_end DESC
LIMIT <limit>
``` [4](#0-3) 

**Limit cap quirk:** `extractLimitFromFilters` caps at `getEffectiveMaxLimit()` (default 100 for unauthenticated users), but if the requested value exceeds the max it falls back to `defaultLimit` (25) rather than `maxLimit`. An attacker simply passes `limit=100` to get the maximum 100 rows. [5](#0-4) 

**No rate limiting:** The REST API server (`rest/server.js`) applies only `authHandler` (optional Basic Auth for custom limits) and metrics middleware. There is no per-IP request rate limiter for the blocks endpoint. The throttling infrastructure found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` module (contract calls), not the REST API. [6](#0-5) 

**Database pool:** Default `maxConnections = 10`, `statementTimeout = 20000 ms`, `connectionTimeout = 20000 ms`. [7](#0-6) 

### Impact Explanation

With 10 DB connections and a 20-second statement timeout, an attacker needs only 10–20 concurrent requests to saturate the pool. All subsequent requests from legitimate users will block for up to 20 seconds waiting for a connection, then fail. Because Node.js is async and the endpoint requires no authentication, a single attacker machine can trivially maintain 10+ in-flight requests. The `record_file` table on mainnet contains millions of rows; a range spanning ~285 years (`gte:1000000000` to `lte:9999999999` in seconds, converted to nanoseconds by `parseTimestampParam`) forces the query planner to consider a full-table sequential scan when the range covers the majority of the table, increasing per-query duration and worsening pool exhaustion. [8](#0-7) 

### Likelihood Explanation

No privileges, API keys, or special knowledge are required. The endpoint is publicly accessible. The attack is trivially scriptable with `curl` or any HTTP load tool. The attacker does not need to sustain a high request rate — 10 concurrent long-running queries are sufficient to deny service. The missing `parseTimestampFilters` call is a straightforward oversight that is already present and enforced in other controllers, making this a realistic finding rather than a theoretical one.

### Recommendation

1. **Enforce timestamp range validation in `extractSqlFromBlockFilters`:** Call `utils.parseTimestampFilters()` on the collected timestamp filters before building the WHERE clause, consistent with how `contractController.js` handles it. This will reject ranges exceeding `maxTimestampRange` (default 7 days) with a 400 error.

2. **Add per-IP rate limiting middleware** to the REST API (e.g., `express-rate-limit`) applied globally or specifically to the `/api/v1/blocks` route.

3. **Fix the limit fallback logic** in `extractLimitFromFilters`: replace `defaultLimit` with `maxLimit` when the requested value exceeds the cap, so the behavior is consistent with other endpoints.

4. Consider increasing `db.pool.maxConnections` or adding a request queue with a concurrency cap to bound the number of simultaneous DB queries.

### Proof of Concept

```bash
# Flood with 20 concurrent requests, each spanning a ~285-year timestamp range at max limit
for i in $(seq 1 20); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks?timestamp=gte:1000000000&timestamp=lte:9999999999&limit=100" &
done
wait
# Legitimate requests now time out or receive 503/connection errors
# because all 10 DB pool connections are held by attacker requests
```

const parseTimestampParam = (timestampParam) => {
  // Expect timestamp input as (a) just seconds,
  // (b) seconds.mmm (3-digit milliseconds),
  // or (c) seconds.nnnnnnnnn (9-digit nanoseconds)
  // Convert all of these formats to (seconds * 10^9 + nanoseconds) format,
  // after validating that all characters are digits
  if (!timestampParam) {
    return '';
  }
  const tsSplit = timestampParam.split('.');
  if (tsSplit.length <= 0 || tsSplit.length > 2) {
    return '';
  }
  const seconds = /^(\d)+$/.test(tsSplit[0]) ? tsSplit[0] : 0;
  let nanos = tsSplit.length === 2 && /^(\d)+$/.test(tsSplit[1]) ? tsSplit[1] : 0;
  nanos = `${nanos}000000000`.substring(0, 9);
  return `${seconds}${nanos}`;
};
```
