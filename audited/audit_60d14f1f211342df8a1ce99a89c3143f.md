### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Exposes Plaintext PostgreSQL Traffic Including `staking_reward_transfer` Data

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` hardcodes `sslmode=disable` with no configuration override, forcing all PostgreSQL traffic for the Rosetta component to be transmitted in plaintext. An attacker with network-level access (e.g., same subnet, ARP poisoning, or compromised network device) can passively sniff or actively MITM the unencrypted TCP stream to read all `staking_reward_transfer` query results — including `account_id` and `amount` — or inject rogue SQL commands into an existing session. This is a structural gap: every other component in the same codebase exposes an `sslMode` configuration knob, but Rosetta does not.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
return fmt.Sprintf(
    "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
    ...
)
``` [1](#0-0) 

This DSN is consumed unconditionally by `ConnectToDb()`:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
``` [2](#0-1) 

The `Db` struct has no `SslMode` field, so there is no YAML/env override path: [3](#0-2) 

**Contrast with other components** — the importer's `DBProperties` exposes `SslMode sslMode = SslMode.DISABLE` as a configurable field: [4](#0-3) 

**Root cause:** The SSL mode is a compile-time constant, not a runtime configuration. The failed assumption is that the network path between the Rosetta service and PostgreSQL is always trusted/isolated.

**Exploit flow:**
1. Attacker gains network-level access to the segment carrying Rosetta→PostgreSQL traffic (no DB credentials required; no application-level privilege required).
2. Because the connection is plaintext TCP, the attacker can:
   - **Passive sniff**: capture all `SELECT … FROM staking_reward_transfer` responses, reading `account_id` and `amount` for every staking reward.
   - **Active MITM / TCP session injection**: spoof TCP sequence numbers on the established connection and inject a rogue `SELECT account_id, amount FROM staking_reward_transfer` query, receiving the response before the legitimate application does.
3. The `staking_reward_transfer` table schema confirms the sensitive fields: [5](#0-4) 

**Why existing checks fail:** There are none. `ConnectToDb()` performs no post-connection SSL state verification, and the DSN string cannot be overridden by operators.

### Impact Explanation
An attacker who can observe or inject into the Rosetta service's network path obtains a full dump of staking reward transfers: every `account_id` paired with its `amount`. This constitutes financial data exfiltration (pending/paid reward amounts and the associated account identifiers). In a cloud or container environment where network policies are misconfigured or absent, this is directly reachable without any application-layer credentials.

### Likelihood Explanation
The precondition is network-level access to the Rosetta↔PostgreSQL segment — achievable via ARP poisoning on a flat LAN, a compromised sidecar/init container in Kubernetes without NetworkPolicy enforcement, a rogue cloud VM in the same VPC subnet, or a misconfigured load balancer. No application credentials are needed. The attack is repeatable and passive sniffing requires no active injection. The Kubernetes Helm chart ships with `citus.node_conninfo: "sslmode=disable"` as a documented workaround, normalizing the pattern and making operators less likely to notice the Rosetta-specific hardcoding: [6](#0-5) 

### Recommendation
1. Add an `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a default of `"require"` or `"verify-full"`.
2. Interpolate it into `GetDsn()` instead of the hardcoded `sslmode=disable`.
3. Add `SslRootCert string` to support certificate pinning (`sslmode=verify-full`).
4. Update `TestDbGetDsn` to assert the new default.
5. Document the field in `docs/configuration.md` alongside the other Rosetta DB properties.

### Proof of Concept
```bash
# Precondition: attacker is on the same network segment as the Rosetta pod/host

# 1. Identify the Rosetta→PostgreSQL TCP stream (default port 5432)
sudo tcpdump -i eth0 -A 'tcp port 5432 and host <db_host>'

# 2. Because sslmode=disable, the PostgreSQL wire protocol is plaintext.
#    Observe query responses containing staking_reward_transfer rows:
#    e.g., "SELECT srt.account_id,srt.amount,srt.consensus_timestamp
#           FROM staking_reward_transfer srt WHERE srt.account_id = $1"
#    Results (account_id, amount) appear in cleartext in the TCP payload.

# 3. For active injection (TCP session hijack), use a tool such as
#    Scapy to forge a TCP segment with the correct sequence number,
#    injecting: "SELECT account_id,amount FROM staking_reward_transfer;\n"
#    The PostgreSQL server processes it within the existing authenticated
#    session and returns all rows to the attacker-controlled socket.
```
