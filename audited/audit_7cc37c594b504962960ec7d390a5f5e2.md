### Title
Information Disclosure via `Bound.parse()` Error Message Leaking Internal Filter Key Name and Operator Logic

### Summary
An unprivileged external user can submit a `spender.id=ne:<value>` query parameter to `GET /api/v1/accounts/{id}/allowances/tokens`. This causes `Bound.parse()` to throw an `InvalidArgumentError` whose message contains the raw internal `filterKey` name and explicitly names the unsupported operator. The HTTP error handler unconditionally returns the full error message to the client for all 4xx responses, making this information reliably observable.

### Finding Description

**Code path:**

1. `getAccountTokenAllowances()` — [1](#0-0)  calls `utils.buildAndValidateFilters()` then `extractTokenMultiUnionQuery()`.

2. `extractTokenMultiUnionQuery()` constructs a `Bound` with `filterKeys.SPENDER_ID` as the `filterKey` — [2](#0-1)  — and for every `SPENDER_ID` filter calls `bounds.primary.parse(filter)`. [3](#0-2) 

3. `Bound.parse()` recognises `eq`, `gt/gte`, and `lt/lte` operators. Any other operator — including `ne`, which **is** a valid entry in `opsMap` — falls through to the `else` branch: [4](#0-3) 
   ```js
   throw new InvalidArgumentError(`Not equal (ne) operator is not supported for ${this.filterKey}`);
   ```
   `this.filterKey` is the raw internal constant value set at construction time.

4. `ne` is a recognized operator in `opsMap` — [5](#0-4)  — so `buildAndValidateFilters` parses it as a structurally valid filter and passes it through; it does **not** reject `ne` before `Bound.parse()` sees it.

5. The HTTP error handler maps `InvalidArgumentError` to HTTP 400, then calls `shouldReturnMessage()` which returns `true` for all client errors, so `err.message` is serialized directly into the JSON response body: [6](#0-5) [7](#0-6) 

**Root cause:** `Bound.parse()` embeds the raw internal `filterKey` constant and the operator name in the thrown error, and the error handler returns the full message to the caller without sanitization.

### Impact Explanation
The response body reveals: (a) the internal symbolic name used for the `spender.id` filter key in the codebase, (b) that `ne` is a known/parsed operator in the system (aiding further operator-probing), and (c) the exact operator validation logic of `Bound` (which operators are accepted vs. rejected). While this does not directly enable data exfiltration or privilege escalation, it reduces attacker effort in mapping the internal API surface and crafting further probes. Severity: **Low** (information disclosure).

### Likelihood Explanation
No authentication or special privileges are required. The endpoint is publicly accessible. The trigger is a single HTTP GET request with a crafted query parameter. It is 100% reproducible and requires no special tooling.

### Recommendation
1. In `Bound.parse()`, replace the raw `this.filterKey` interpolation with a sanitized or user-facing alias (e.g., `this.viewModelKey`), or use a generic message such as `"Not equal (ne) operator is not supported for this parameter"`.
2. Alternatively, reject the `ne` operator in `buildAndValidateFilters` before filters reach `Bound.parse()`, so the error is raised with a generic "invalid operator" message that does not reference internal names.
3. Audit all other `InvalidArgumentError` throws in `Bound` (lines 52, 57, 62) for the same pattern.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/allowances/tokens?spender.id=ne:0.0.1
```

Expected response (HTTP 400):
```json
{
  "_status": {
    "messages": [
      { "message": "Not equal (ne) operator is not supported for spender.id" }
    ]
  }
}
```

This discloses the internal `filterKeys.SPENDER_ID` value and confirms `ne` is a parsed-but-unsupported operator, with no credentials required.
