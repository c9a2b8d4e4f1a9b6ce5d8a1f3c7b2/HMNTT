### Title
Unbounded Historical Query via Zero-Epoch `consensusStartTime` Enables Database Resource Exhaustion (DoS)

### Summary
An unprivileged external user can supply a `ConsensusTopicQuery` with `consensusStartTime = Timestamp{seconds=0, nanos=0}` and a near-maximum `consensusEndTime`, causing `convertTimestamp()` to return `0` as `startTime`. All existing validation checks pass for this value, so `TopicMessageRetriever` issues an unbounded historical query spanning from Unix epoch to the far future against any topic, exhausting database connection pool resources and starving legitimate subscribers.

### Finding Description

```java
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
```

With `seconds=0, nanos=0`: the guard `0 >= 9223372035L` is false, so `DomainUtils.timestampInNanosMax(timestamp)` is called.

**Result:** `TopicMessageServiceImpl.subscribeTopic()` calls `topicMessageRetriever.retrieve(filter, true)` with `startTime=0` (Unix epoch, 1970-01-01) and `endTime≈Long.MAX_VALUE`, `limit=0` (unlimited). [7](#0-6) 

This issues a full-table historical scan of the `topic_message` table for the targeted topic, with no row cap.

### Impact Explanation
Any high-volume HCS topic (e.g., one used for financial settlement or fund-release signaling) can be targeted. A single attacker connection triggers a full historical DB scan. Multiple concurrent connections from the same or different clients exhaust the database connection pool, causing all other `subscribeTopic` calls — including legitimate ones waiting for fund-release or settlement messages — to queue or fail. This is a practical, unauthenticated denial-of-service against the mirror node's gRPC layer with direct downstream impact on any application relying on timely topic message delivery.

### Likelihood Explanation
No authentication is required. The gRPC `subscribeTopic` endpoint is publicly accessible. The attack requires only a standard gRPC client and knowledge of a high-volume topic ID (which is public on-chain). It is trivially repeatable and scriptable. A single attacker with modest resources can sustain the attack indefinitely by reopening connections as they are dropped.

### Recommendation
1. **Enforce a minimum `startTime` floor**: Reject or clamp `startTime` values older than a configurable retention window (e.g., 30 days before `now()`). Add a `@AssertTrue` validator in `TopicMessageFilter` analogous to `isValidStartTime()`.
2. **Require a non-zero `startTime`**: In `toFilter()`, if `convertTimestamp()` returns `0` (or any value before the network genesis timestamp), substitute `DomainUtils.now()` or reject the request with `INVALID_ARGUMENT`.
3. **Enforce a maximum query window**: Add a `maxQueryRange` property (e.g., 24 hours) and validate `endTime - startTime <= maxQueryRange`.
4. **Enforce a per-connection result limit**: Require `limit > 0` for historical queries, or apply a server-side cap.
5. **Rate-limit per client IP/identity** at the gRPC interceptor layer.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from hederahashgraph.api.proto.java import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Target a known high-volume topic (e.g., topic 0.0.1234)
request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),  # Unix epoch
    consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372034, nanos=0),  # near-max
    limit=0  # no limit
)

# Open many concurrent connections to exhaust DB pool
for _ in range(50):
    stub.subscribeTopic(request)  # each triggers full historical scan
```

Each connection causes the mirror node to issue `SELECT * FROM topic_message WHERE topic_id=1234 AND consensus_timestamp >= 0 AND consensus_timestamp < ~MAX_LONG` with no row limit, holding a DB connection for the duration of the stream.
