### Title
Deleted Contract Records Inaccessible via EVM Address in Sub-Resource Endpoints Due to Overly Restrictive Filter

### Summary
The `contractIdByEvmAddressQuery` in `rest/service/contractService.js` (lines 146–151) unconditionally excludes deleted contracts (`DELETED <> true`). When an unprivileged user queries sub-resource endpoints such as `/api/v1/contracts/{evm_address}/results` or `/api/v1/contracts/{evm_address}/results/logs` using a CREATE2 EVM address belonging to a deleted contract, `getContractIdByEvmAddress()` finds zero rows and throws a `NotFoundError`, making those records permanently inaccessible via EVM address. The same data is fully accessible when the numeric contract ID is used instead, creating an exploitable inconsistency.

### Finding Description
**Exact code path:**

`rest/service/contractService.js`, `contractIdByEvmAddressQuery`, lines 146–151:
```sql
select e.id
from entity e
where e.deleted <> true and
      e.type = 'CONTRACT' and
      e.evm_address = $1
```
`getContractIdByEvmAddress()` (lines 452–466) executes this query and throws `NotFoundError` when `rows.length === 0`.

`computeContractIdFromString()` (lines 468–476) calls `getContractIdByEvmAddress()` for any EVM-address-shaped input.

`computeContractIdFromString()` is invoked by:
- `getContractLogsById` (`contractController.js` line 788) — `/contracts/:contractId/results/logs`
- `getContractResultsById` (`contractController.js` line 862) — `/contracts/:contractId/results`
- `extractSqlFromContractFilters` (`contractController.js` line 105) — `/contracts?contract.id={evm_address}`

**Root cause / failed assumption:** The query assumes that a deleted contract should be treated as non-existent for ID-resolution purposes. This is incorrect: the mirror node is a historical ledger and deleted contracts still have valid, queryable results and logs.

**Why existing checks fail:** The top-level `getContractById` handler (`contractController.js` lines 707–737) uses `getContractByIdOrAddressContractEntityQuery` (lines 164–172), which has **no** `deleted` filter — it only requires `e.type = 'CONTRACT'`. So `GET /contracts/{evm_address}` correctly returns a deleted contract with `"deleted": true` in the response body (confirmed by test fixture `rest/__tests__/specs/contracts/no-params.json` lines 74–75). The sub-resource endpoints are inconsistent with this design intent.

### Impact Explanation
Any unprivileged external user can supply a CREATE2 EVM address for a deleted contract and receive a `404 NotFoundError` from `/contracts/{evm_address}/results` and `/contracts/{evm_address}/results/logs`, even though the same data is returned without error when the numeric ID is used. This causes systematic gaps in mirror node exports for deleted contracts referenced by EVM address — a direct violation of the mirror node's role as a complete historical record. Severity is **Medium**: no privilege escalation or data leakage, but data completeness and API consistency are materially broken.

### Likelihood Explanation
No privileges are required. Any caller who knows (or discovers) a CREATE2 EVM address — which is deterministic and publicly computable — can trigger this. Deleted contracts are common in production Hedera networks. The condition is trivially and repeatably reproducible by any API consumer.

### Recommendation
Remove the `DELETED <> true` predicate from `contractIdByEvmAddressQuery` so that the query resolves the entity ID regardless of deletion status:

```sql
select e.id
from entity e
where e.type = 'CONTRACT' and
      e.evm_address = $1
```

The downstream queries for results and logs do not themselves filter on `deleted`, so removing the filter here is safe and aligns with the behavior of `getContractById`. If callers need to surface deletion status, they can check the `deleted` field in the returned contract entity, as `getContractById` already does.

### Proof of Concept
1. Create a contract with a CREATE2 EVM address (e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`) and record some contract results/logs.
2. Delete the contract (set `deleted = true` in the `entity` table, as happens on a `CONTRACTDELETEINSTANCE` transaction).
3. Query `GET /api/v1/contracts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef` → **200 OK**, contract returned with `"deleted": true`. ✅
4. Query `GET /api/v1/contracts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/results` → **404 NotFoundError**: `"No contract with the given evm address 0xdeadbeef... has been found."` ❌
5. Query `GET /api/v1/contracts/0.0.{numeric_id}/results` (same contract, numeric ID) → **200 OK**, results returned. ✅

Steps 4 vs 5 confirm the inconsistency: the EVM address path silently drops all historical records for deleted contracts.
