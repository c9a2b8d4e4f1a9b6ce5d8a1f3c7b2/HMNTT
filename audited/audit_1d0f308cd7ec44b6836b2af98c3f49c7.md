### Title
Premature DB Lookup Before Timestamp Range Validation in `getOneAccount()` Enables Griefing DoS

### Summary
In `rest/accounts.js`, `getOneAccount()` calls `EntityService.getEncodedId()` (line 399) before validating whether the supplied timestamp range is empty (lines 414–416). When the path parameter is an alias or EVM address, `getEncodedId()` issues a real database query. An unprivileged attacker can craft requests with a provably empty timestamp range (e.g., `timestamp=gte:9999&timestamp=lte:1`) targeting alias or EVM-address paths, forcing a wasted DB round-trip on every request before the `NotFoundError('Timestamp range is empty')` is thrown.

### Finding Description
**Exact code path:**

`rest/accounts.js` → `getOneAccount()`:
- Line 398: `utils.buildAndValidateFilters(req.query, acceptedSingleAccountParameters)` — validates individual filter syntax but does **not** check for an empty range.
- Line 399: `const encodedId = await EntityService.getEncodedId(req.params[...])` — **DB query issued here** for alias/EVM-address inputs.
- Lines 401–412: timestamp filters are extracted from the already-parsed `filters` array.
- Line 413: `utils.parseTimestampFilters(...)` — range is computed.
- Lines 414–416: `if (timestampRange.range?.isEmpty()) { throw new NotFoundError('Timestamp range is empty'); }` — check happens **after** the DB round-trip.

**Root cause in `rest/service/entityService.js` `getEncodedId()` (lines 118–137):**
- Numeric IDs (`0.0.N`): `evmAddress === null` → pure computation, no DB hit.
- EVM addresses: calls `getEntityIdFromEvmAddress()` → executes `SELECT id FROM entity WHERE evm_address = $1`.
- Aliases: calls `getAccountIdFromAlias()` → executes `SELECT id FROM entity WHERE alias = $1`.

The timestamp filters are already present in `filters` (returned by `buildAndValidateFilters` at line 398) before `getEncodedId()` is called. The range emptiness check could trivially be moved before line 399, but it is not.

**Failed assumption:** The code assumes the DB lookup must precede all other validation, but the timestamp range can be fully evaluated from the already-parsed query parameters without any DB access.

### Impact Explanation
Each crafted request targeting an alias or EVM-address path causes one unnecessary `SELECT` against the `entity` table before the request is rejected. With no rate limiting present in the Node.js REST layer (the `ThrottleConfiguration` / `ThrottleManagerImpl` found in the codebase applies only to the `web3` module, not the REST API), an attacker can sustain a high request rate. At scale this inflates DB connection pool utilization, increases query latency for legitimate users, and generates amplified error-path log volume — all with zero cost to the attacker beyond network bandwidth. Severity is low-to-medium (griefing/availability degradation, no data exfiltration or fund loss).

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly reachable. The payload is trivial to construct and repeat (a single `curl` one-liner in a loop). Any alias or EVM address visible on-chain (all are public) can be used as the path parameter. The attack is stateless and fully parallelizable across many source IPs.

### Recommendation
Move the timestamp range emptiness check **before** the `EntityService.getEncodedId()` call. The `filters` array is already available from `buildAndValidateFilters`. Extract timestamp filters and call `utils.parseTimestampFilters()` immediately after line 398, throw `NotFoundError('Timestamp range is empty')` if the range is empty, and only then proceed to `getEncodedId()`. Additionally, consider adding per-IP or global rate limiting to the Node.js REST layer (analogous to the existing `ThrottleConfiguration` in the web3 module).

### Proof of Concept
```bash
# Using a known alias (publicly visible on-chain)
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ?timestamp=gte:9999&timestamp=lte:1"
done
```
Each iteration:
1. `buildAndValidateFilters` passes (both timestamp params are individually valid).
2. `EntityService.getEncodedId()` issues `SELECT id FROM entity WHERE alias = $1` against the DB.
3. `parseTimestampFilters` computes an empty range.
4. `NotFoundError('Timestamp range is empty')` is thrown and a 404 is returned.

The DB query at step 2 is entirely wasted. Parallelising across many connections amplifies DB load proportionally.
