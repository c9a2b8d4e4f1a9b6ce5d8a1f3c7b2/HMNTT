### Title
Unauthenticated Concurrent `getNodes()` Flood Exhausts Reactive Thread Pool and Database Connections (DoS)

### Summary
The `getNodes()` endpoint in `NetworkController` accepts requests from any unauthenticated caller with no global rate limit or connection cap. Each call creates an independent Reactor `Flux` subscription that occupies a `boundedElastic` scheduler thread and holds a database connection for its entire lifetime. Because the only concurrency guard is a per-connection limit of 5, an attacker who opens many TCP connections can drive arbitrarily many concurrent subscriptions, exhausting both the bounded thread pool and the DB connection pool and denying service to legitimate callers.

### Finding Description

**Code path:**

`NetworkController.getNodes()` (lines 33–43) creates a new subscription on every inbound RPC with no authentication, no per-IP check, and no global rate limit:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(networkService::getNodes)   // new Flux per call
        .map(this::toNodeAddress)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`NetworkServiceImpl.getNodes()` (lines 68–76) schedules blocking DB work on `Schedulers.boundedElastic()`:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())   // 250 ms hold per page
                .withScheduler(Schedulers.boundedElastic()))
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
```

`page()` (lines 79–108) calls `transactionOperations.execute(...)` — a **blocking** JDBC call — inside the reactive chain, acquiring a DB connection for each page fetch.

**The only concurrency guard** is `maxConcurrentCallsPerConnection = 5` in `GrpcConfiguration` / `NettyProperties`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// default = 5
```

This is a **per-connection** limit. There is no configured global connection limit (`maxConnections` / `maxInboundConnections` are absent from `GrpcConfiguration`), no IP-level throttle, and no rate-limiting bean in the `grpc` module (the `ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` module).

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection = 5` bounds total server load. It does not — it only bounds load per TCP connection. An attacker who opens *N* connections multiplies the limit to *5 × N* concurrent subscriptions.

### Impact Explanation

Each concurrent subscription:
1. Pins a `boundedElastic` thread (default cap: `10 × CPU cores`, e.g. 80 threads on an 8-core pod) for the full streaming duration including the 250 ms `pageDelay` between pages.
2. Holds a JDBC connection from the shared pool for every `page()` call.

When the thread pool is saturated, new subscriptions queue indefinitely; when the DB pool is exhausted, all gRPC RPCs that touch the database (including `subscribeTopic`) begin failing. The result is a complete denial of service for legitimate users of the gRPC API.

**Severity: High** — full service disruption, no authentication required, no per-IP guard.

### Likelihood Explanation

The gRPC port (5600) is publicly documented and reachable without credentials. A single attacker machine can open hundreds of TCP connections and issue 5 `getNodes` RPCs per connection using any standard gRPC client (e.g., `grpcurl`, a trivial Go/Java script). The attack is stateless, requires no prior knowledge beyond the proto definition (publicly available), and is trivially repeatable. The `limit=0` default forces the server to stream all nodes, maximising per-subscription resource hold time.

### Recommendation

1. **Add a global concurrent-call limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge(...)` and a server-wide interceptor that tracks and caps total active streaming RPCs.
2. **Add a per-IP / global rate-limiting interceptor** in the `grpc` module analogous to the `ThrottleManagerImpl` in `web3`, applied as a `ServerInterceptor` before any RPC handler is invoked.
3. **Cap the maximum number of inbound connections** using `NettyServerBuilder`'s connection-limit API or an upstream proxy (e.g., Envoy/Traefik with connection limits).
4. **Enforce a hard upper bound on `limit`** in `AddressBookFilter` so callers cannot force unbounded streaming.
5. **Consider caching the address book response** (it changes infrequently) so repeated calls hit an in-memory cache rather than the DB.

### Proof of Concept

```bash
# Open 200 connections, 5 concurrent getNodes each = 1000 concurrent Flux subscriptions
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes &
  done
done
wait
```

**Expected result:** `boundedElastic` thread pool saturates (observable via `reactor.netty.pool.acquirePendingCount` or JVM thread dump), DB connection pool exhausts (HikariCP logs `Connection is not available, request timed out`), and subsequent legitimate `getNodes` or `subscribeTopic` calls receive `UNAVAILABLE` or `DEADLINE_EXCEEDED` gRPC status codes.
