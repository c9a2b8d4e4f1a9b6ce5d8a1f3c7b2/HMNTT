### Title
Unprivileged User Can Corrupt `requestSizeHistogram` via Oversized `Content-Length` Header Causing `parseInt` Precision Loss

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function reads the raw `content-length` header string and passes it directly to `parseInt` with no upper-bound validation. JavaScript's `parseInt` returns an IEEE 754 double-precision float, which cannot exactly represent integers above `Number.MAX_SAFE_INTEGER` (9007199254740991). Any unprivileged attacker can send a request with a `Content-Length` value in the range `(MAX_SAFE_INTEGER, UINT64_MAX]` to cause an imprecise floating-point value to be recorded in `requestSizeHistogram`, permanently skewing the histogram's `sum` and derived averages.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback within `metricsMiddleware`:

```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** `parseInt` in JavaScript always returns a `Number` (IEEE 754 double). Doubles have 53 bits of mantissa, so integers above `Number.MAX_SAFE_INTEGER = 9007199254740991` (~9×10^15) cannot be represented exactly. For example:

```
parseInt('9999999999999999', 10)  // returns 10000000000000000 — off by 1
parseInt('9007199254740993', 10)  // returns 9007199254740992 — off by 1
```

The `|| 0` guard only catches `NaN` or `0`; it does not catch imprecise non-zero floats. The imprecise value is passed directly to `requestSizeHistogram.record()`.

**Why the specific example value `9999999999999999999999` (~10^22) does not work as stated:** Node.js's `llhttp` HTTP parser stores `Content-Length` internally as a `uint64_t` (max ~1.8×10^19). A value of `9999999999999999999999` exceeds `UINT64_MAX` and is rejected at the parser level with a 400 before any Express middleware runs. The correct exploit uses any value in the range `(9007199254740991, 18446744073709551615]`, e.g. `9999999999999999`.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts HTTP/1.1` with `Content-Length: 9999999999999999`.
2. `llhttp` accepts the value (within uint64 range); Express populates `req.headers['content-length']` with the string `'9999999999999999'`.
3. `parseInt('9999999999999999', 10)` returns `10000000000000000` (imprecise, off by 1).
4. `requestSizeHistogram.record(10000000000000000, labels)` is called.
5. The histogram's `sum` is inflated by ~10^16 bytes per request.

### Impact Explanation
The `requestSizeHistogram` (`api_request_size_bytes`) is a Prometheus/OpenTelemetry histogram used for operational monitoring. Repeated injection of astronomically large imprecise values permanently inflates the histogram's `sum` counter (which is monotonically increasing and never reset without a process restart). This corrupts the `rate(api_request_size_bytes_sum[...]) / rate(api_request_size_bytes_count[...])` average-size metric, rendering it meaningless. Operators relying on this metric for capacity planning, anomaly detection, or alerting receive false data. There is no economic damage and no impact on actual request processing.

### Likelihood Explanation
No authentication or special privilege is required. Any client that can reach the REST API can send an arbitrary `Content-Length` header on any HTTP method. The attack is trivially repeatable with a single `curl` command. The only constraint is that the value must be ≤ `UINT64_MAX`; values in the range `(MAX_SAFE_INTEGER, UINT64_MAX]` are both accepted by the HTTP parser and imprecisely parsed by `parseInt`.

### Recommendation
Replace the raw `parseInt` call with a safe bounded parse:

```javascript
const rawCL = req.headers['content-length'];
const parsed = rawCL !== undefined ? Number(rawCL) : 0;
const requestSize = Number.isSafeInteger(parsed) && parsed >= 0 ? parsed : 0;
requestSizeHistogram.record(requestSize, labels);
```

This rejects any value that cannot be exactly represented as a JavaScript integer (i.e., anything above `MAX_SAFE_INTEGER` or non-integer), recording `0` instead of a corrupted float.

### Proof of Concept
```bash
# Value is above MAX_SAFE_INTEGER (9007199254740991) but below UINT64_MAX
curl -s -o /dev/null -w "%{http_code}" \
  -H "Content-Length: 9999999999999999" \
  http://<mirror-node-rest-host>/api/v1/accounts

# Repeat ~10 times, then scrape metrics:
curl http://<mirror-node-rest-host>/api/v1/metrics/ | grep api_request_size_bytes_sum
# Observe: sum is inflated by ~1e16 per request instead of the actual 0 bytes
```

**Verification of `parseInt` imprecision (Node.js REPL):**
```
> parseInt('9999999999999999', 10)

> parseInt('9999999999999999', 10) === 9999999999999999
