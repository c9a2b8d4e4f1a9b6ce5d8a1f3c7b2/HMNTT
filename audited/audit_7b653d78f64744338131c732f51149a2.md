### Title
Unbounded File Data Load via Public `/contracts/:contractId` Endpoint Enables Heap Exhaustion DoS

### Summary
The `getContractById()` handler in `rest/controllers/contractController.js` unconditionally calls `FileDataService.getFileData()` for any contract whose `file_id` passes the `isValidUserFileId()` check, with no cap on the size of data fetched. The underlying SQL query uses `string_agg` to concatenate all file chunks into a single in-memory buffer with no size limit, and the endpoint requires no authentication. An attacker who creates a large file on the Hedera network and a contract referencing it can repeatedly trigger this path to exhaust Node.js heap on mirror node instances.

### Finding Description

**Exact code path:**

`getContractById()` at line 731–732 of `rest/controllers/contractController.js`:
```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
}
``` [1](#0-0) 

`FileDataService.getFileData()` at lines 71–76 of `rest/service/fileDataService.js` executes `getFileDataQuery` and returns the entire result: [2](#0-1) 

`getFileDataQuery` at lines 44–63 of `rest/service/fileDataService.js` uses `string_agg` with **no size limit** to concatenate every file-data chunk for the given `file_id` up to the given timestamp into a single string: [3](#0-2) 

**Root cause:** The query aggregates all `file_data` rows for the entity into one unbounded buffer. There is no `LIMIT`, no `max_length` guard, and no server-side size check before the result is materialized in Node.js heap memory.

**Failed assumption:** The code assumes `file_id` always points to a small contract-bytecode file. Nothing enforces this — any user-created file of arbitrary size can be referenced.

**Why `isValidUserFileId` is insufficient:** This check only filters out system-range file IDs (e.g., file_id ≤ 1000, as evidenced by the `contract-initcode-large-files.json` test fixture returning empty bytecode for `file_id: "100"`). It does not bound the size of user files. [4](#0-3) 

### Impact Explanation
Each request to `/api/v1/contracts/:contractId` for a targeted contract causes the Node.js process to allocate a heap buffer equal to the full size of the referenced file. Concurrent requests multiply this allocation. Node.js has a default heap limit (~1.5 GB on 64-bit). A sufficiently large file (achievable via repeated `FileAppend` transactions on Hedera) combined with a modest number of concurrent HTTP requests can trigger an OOM crash of the mirror node process. If multiple mirror node instances in a cluster are targeted simultaneously, 30%+ of processing capacity can be taken offline without any brute-force credential attack.

### Likelihood Explanation
The attack requires no credentials — the endpoint is fully public. The only prerequisite is spending HBAR to create a large file and a contract referencing it on the Hedera network. File size is bounded only by the number of `FileAppend` transactions an attacker is willing to pay for; files of tens of megabytes are achievable. Once the contract exists on-chain, the mirror node will have ingested it, and the attacker can trigger the expensive load repeatedly at negligible cost (standard HTTP GET requests). The attack is repeatable and automatable.

### Recommendation
1. **Enforce a maximum bytecode size** before loading: query `length(string_agg(...))` or use a `LIMIT`/`max_length` guard in `getFileDataQuery`, and return an empty/truncated result if the file exceeds a configured threshold (e.g., 1 MB).
2. **Stream or paginate** the file data rather than materializing the entire content in one `string_agg` call.
3. **Add a size pre-check**: before calling `getFileData`, query `sum(length(file_data))` for the entity and short-circuit if it exceeds the threshold.
4. **Apply rate limiting** per IP or per contract ID on the `/contracts/:contractId` endpoint at the API gateway or middleware layer.

### Proof of Concept

**Preconditions:**
- Hedera testnet/mainnet access with enough HBAR to create files and a contract.

**Steps:**
1. Create a large file on Hedera by submitting a `FileCreate` followed by many `FileAppend` transactions, accumulating e.g. 50 MB of data. Note the resulting `file_id` (e.g., `0.0.99999`).
2. Submit a `ContractCreate` transaction referencing `file_id = 0.0.99999` as the bytecode file. Note the resulting `contract_id` (e.g., `0.0.100000`).
3. Wait for the mirror node to ingest both transactions.
4. Send concurrent HTTP GET requests to the mirror node:
   ```
   for i in $(seq 1 50); do
     curl -s "https://<mirror-node>/api/v1/contracts/0.0.100000" &
   done
   wait
   ```
5. **Result:** Each request triggers `FileDataService.getFileData(99999, <timestamp>)`, which executes the unbounded `string_agg` query and loads ~50 MB into Node.js heap per request. With 50 concurrent requests, ~2.5 GB of heap is allocated, causing OOM crash of the mirror node process.
