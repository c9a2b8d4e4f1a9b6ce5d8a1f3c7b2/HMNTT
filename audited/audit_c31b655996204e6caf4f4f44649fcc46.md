### Title
Indefinite Subscription Resource Exhaustion via Non-Existent Topic When `checkTopicExists=false`

### Summary
When the server is configured with `checkTopicExists=false`, any unauthenticated client can subscribe to an arbitrary non-existent `topicID` with no `limit` and no `consensusEndTime`, causing the gRPC stream to remain open indefinitely. By repeating this across many connections, an attacker exhausts gRPC stream slots, Reactor `boundedElastic` thread pool capacity, and database polling resources, degrading or denying service to legitimate subscribers.

### Finding Description

**Code path and root cause:**

In `TopicMessageServiceImpl.topicExists()` (lines 94–106), when the entity is not found in the repository and `grpcProperties.isCheckTopicExists()` is `false`, the method synthesizes a fake `Entity` of type `TOPIC` and returns it, allowing the subscription to proceed:

```java
// TopicMessageServiceImpl.java lines 96–103
return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
        .switchIfEmpty(
                grpcProperties.isCheckTopicExists()
                        ? Mono.error(new EntityNotFoundException(topicId))
                        : Mono.just(Entity.builder()
                                .memo("")
                                .type(EntityType.TOPIC)
                                .build()))
```

Once past this gate, `subscribeTopic()` (lines 59–92) constructs three reactive streams:
- `historical`: returns empty immediately (no messages for a non-existent topic)
- `live`: calls `topicListener.listen(newFilter)` — polls indefinitely
- `safetyCheck`: fires after 1 second on `Schedulers.boundedElastic()`, also polls indefinitely

The termination condition `pastEndTime()` (lines 123–131) returns `Flux.never()` when no `endTime` is set:

```java
// TopicMessageServiceImpl.java lines 123–126
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
```

With no `limit` set, `filter.hasLimit()` returns `false` (TopicMessageFilter line 39–41), so `flux.take()` is never applied. The subscription never terminates.

**With `PollingTopicListener` (type=POLL):** Each subscription creates its own polling loop on `Schedulers.boundedElastic()` (PollingTopicListener line 31), firing a DB query every 500ms (default `interval`). Reactor's `boundedElastic` scheduler caps at `10 × CPU cores` threads with a 100,000-task queue. Enough concurrent subscriptions exhaust this pool.

**Connection-level limit is insufficient:** `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, GrpcConfiguration line 33) limits calls per TCP connection, but there is no global cap on total concurrent subscriptions and no per-IP or per-client rate limiting on the gRPC endpoint. An attacker opens many TCP connections, each with 5 concurrent subscriptions.

**No authentication** is required to call `subscribeTopic` — the gRPC service is publicly accessible.

### Impact Explanation
Each open subscription consumes: a gRPC stream slot, a `TopicContext` object, a `boundedElastic` scheduler task (safetyCheck), and — with POLL listener type — a recurring DB polling thread. Exhausting the `boundedElastic` thread pool causes all reactive tasks relying on it (including legitimate subscriptions and safety checks) to queue or be rejected. At scale, this constitutes a denial of service against the gRPC API, preventing legitimate clients from receiving HCS topic messages. Classified as critical because it targets the network's ability to confirm and distribute new transactions via the mirror node.

### Likelihood Explanation
The precondition (`checkTopicExists=false`) is a non-default but documented operator configuration. The exploit requires no credentials, no special protocol knowledge beyond the public protobuf schema, and is trivially repeatable with any gRPC client. An attacker needs only to: (1) discover the mirror node gRPC endpoint, (2) know or guess any non-existent topic number, and (3) open many connections. The attack is low-cost and highly repeatable.

### Recommendation
1. **Enforce a global concurrent-subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and reject new subscriptions with `RESOURCE_EXHAUSTED` when exceeded.
2. **Add per-IP or per-client connection rate limiting** at the gRPC server level (e.g., via a Netty `ChannelHandler` or an interceptor).
3. **Require a non-zero `limit` or a finite `consensusEndTime`** when `checkTopicExists=false`, or re-evaluate whether `checkTopicExists=false` should be a supported production configuration.
4. **Add a maximum subscription duration** (e.g., configurable `maxSubscriptionDuration`) after which the server forcibly terminates the stream.
5. Consider enforcing `checkTopicExists=true` as the only safe production setting and removing the bypass path entirely.

### Proof of Concept

**Preconditions:**
- Mirror node gRPC endpoint is reachable (e.g., `localhost:5600`)
- Server configured with `hiero.mirror.grpc.checkTopicExists=false`
- `grpcurl` or any gRPC client available

**Steps:**

```bash
# Open 100 connections, each with 5 concurrent subscriptions to non-existent topics
for i in $(seq 1 100); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{
      "topicID": {"topicNum": '$(( RANDOM + 9000000 ))'}
    }' localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# Each call blocks indefinitely (no limit, no endTime)
# 500 open streams consuming boundedElastic threads and DB polling resources
```

**Expected result:** Server's `boundedElastic` thread pool saturates; legitimate subscribers experience delays or `RESOURCE_EXHAUSTED` errors; `hiero.mirror.grpc.subscribers` gauge climbs without bound; DB polling load increases proportionally.
