### Title
`long`-to-`int` Narrowing Cast in `PollingTopicListener.poll()` Bypasses `maxPageSize` and Returns Unlimited Topic Messages

### Summary
In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` performs an unchecked narrowing cast from `long` to `int`. Any user-supplied limit value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) produces a negative `int` after the cast. This negative value causes the internally-constructed `newFilter` to have `hasLimit() == false`, which causes `TopicMessageRepositoryCustomImpl.findByFilter()` to skip calling `setMaxResults()` entirely, returning all rows for the topic from the database with no page cap.

### Finding Description

**Entry point** — `ConsensusController.toFilter()`: [1](#0-0) 

The protobuf `ConsensusTopicQuery.limit` is a `uint64` (Java `long`). It is mapped directly to `TopicMessageFilter.limit` with no upper-bound check. The only constraint on `TopicMessageFilter.limit` is `@Min(0)`: [2](#0-1) 

`Long.MAX_VALUE` satisfies `@Min(0)`, so validation passes.

**Outer flux cap is ineffective** — `TopicMessageServiceImpl.subscribeTopic()`: [3](#0-2) 

`flux.take(Long.MAX_VALUE)` is Reactor's "take everything" — no real cap.

**Propagation to listener** — `incomingMessages()` passes `limit = Long.MAX_VALUE - 0 = Long.MAX_VALUE` into the filter given to `PollingTopicListener.listen()`: [4](#0-3) 

**Root cause** — `PollingTopicListener.poll()` lines 54–59: [5](#0-4) 

```
(int)(Long.MAX_VALUE - 0L)
= (int)(0x7FFFFFFFFFFFFFFF)
= 0xFFFFFFFF  (lower 32 bits)
= -1          (signed int)
```

`limit = -1`. Then `Math.min(-1, maxPageSize) = -1`. The `newFilter` is built with `limit = -1`.

**`hasLimit()` returns `false` for negative values**: [6](#0-5) 

**`setMaxResults` is never called** — `TopicMessageRepositoryCustomImpl.findByFilter()`: [7](#0-6) 

With `hasLimit() == false`, `setMaxResults` is skipped. The JPA query loads **all matching rows** into memory via `getResultList()` and streams them to the attacker. [8](#0-7) 

The `maxPageSize` guard at line 57 of `PollingTopicListener` is also bypassed because `Math.min(-1, maxPageSize) = -1` — a negative value always wins `Math.min`. [9](#0-8) 

Note: the internal `newFilter` is constructed via the builder directly, not through Spring's `@Validated` path, so the `@Min(0)` constraint is never re-evaluated on the `-1` value.

### Impact Explanation
Every poll cycle (repeated at `listenerProperties.getInterval()` indefinitely via `RepeatSpec.times(Long.MAX_VALUE)`) issues an unbounded `SELECT` against the `topic_message` table for the targeted topic, loading all results into heap memory. This enables:
- **Data exfiltration**: all historical messages for any topic returned without limit
- **Database DoS**: repeated full-range scans at polling frequency
- **OOM / heap exhaustion** on the mirror node: `getResultList()` materializes the entire result set

### Likelihood Explanation
The gRPC `ConsensusService.subscribeTopic` endpoint is unauthenticated and publicly accessible. Any gRPC client can send a `ConsensusTopicQuery` with `limit = Long.MAX_VALUE` (or any value > `Integer.MAX_VALUE`, e.g., `2147483648`). No credentials, no special role, no prior knowledge beyond the topic ID is required. The attack is trivially repeatable and scriptable.

### Recommendation
Replace the unchecked `(int)` cast with a bounds-safe clamp before computing `pageSize`:

```java
long remaining = filter.getLimit() - context.getCount().get();
int limit = (remaining > Integer.MAX_VALUE || remaining < 0)
        ? Integer.MAX_VALUE
        : (int) remaining;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

Additionally, add a `@Max` constraint (e.g., `@Max(Integer.MAX_VALUE)`) to `TopicMessageFilter.limit` to reject oversized values at the API boundary before they propagate into the polling logic.

### Proof of Concept

1. Start a gRPC client targeting the mirror node's gRPC port.
2. Send a `ConsensusTopicQuery` with:
   - `topicID` = any valid topic (e.g., the staking-reward topic)
   - `limit` = `9223372036854775807` (`Long.MAX_VALUE`)
   - `consensusStartTime` = epoch 0
3. Observe that the server begins streaming all historical messages for the topic without any page-size cap.
4. Confirm by monitoring DB query logs: each poll cycle issues a `SELECT` with no `LIMIT` clause.
5. Repeat with `limit = 2147483648` (`Integer.MAX_VALUE + 1`) to confirm any value above `Integer.MAX_VALUE` triggers the same path.
