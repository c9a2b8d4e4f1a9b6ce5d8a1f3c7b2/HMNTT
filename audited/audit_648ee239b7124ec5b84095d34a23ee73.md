### Title
Unauthenticated DB Query Amplification via `getContractResultsByTimestamp` with Valid `callDataId`

### Summary
The `GET /api/v1/contracts/:contractId/results/:consensusTimestamp` endpoint, handled by `getContractResultsByTimestamp()` in `rest/controllers/contractController.js`, issues up to 7 sequential/parallel database queries per single HTTP request when the resolved Ethereum transaction carries a valid `callDataId`. The REST API server applies no per-IP or per-endpoint rate limiting, allowing any unauthenticated external user to amplify database load by repeatedly targeting a known timestamp that satisfies the `callDataId` condition.

### Finding Description

**Code path** (`rest/controllers/contractController.js`, lines 1002–1042):

1. **Line 1011** – `ContractService.getInvolvedContractsByTimestampAndContractId(timestamp, contractId)` → DB query #1.
2. **Lines 1016–1017** – `this.getDetailedContractResults(contractDetails, contractId)` → `Promise.all` of 5 parallel DB queries (#2–#6):
   - `ContractService.getContractResultsByTimestamps` (line 1207)
   - `TransactionService.getEthTransactionByTimestampAndPayerId` (line 1208)
   - `RecordFileService.getRecordFileBlockDetailsFromTimestamp` (line 1212)
   - `ContractService.getContractLogsByTimestamps` (line 1213)
   - `ContractService.getContractStateChangesByTimestamps` (line 1214)
3. **Lines 1022–1023** – `utils.isValidUserFileId(ethTransaction?.callDataId)` evaluates to `true` for any `callDataId > 1000` (confirmed by `rest/__tests__/utils.test.js` lines 2156–2159), triggering `FileDataService.getLatestFileDataContents()` → DB query #7 (a CTE that itself performs two sub-queries against `file_data`).

**Root cause / failed assumption**: The code assumes that the `callDataId` branch is rare or that upstream infrastructure (reverse proxy, WAF) will throttle requests. Neither assumption is enforced in the codebase. `rest/server.js` registers no rate-limiting middleware for GET routes — only `authHandler` (which only adjusts pagination limits for authenticated users), `requestLogger`, optional metrics, and optional response cache. The `ThrottleManager` / `ThrottleConfiguration` exist exclusively in the `web3` module for `POST /contracts/call` and are not wired into the REST API at all.

**Exploit flow**:
1. Attacker queries the public mirror node to find any `contractId` + `consensusTimestamp` pair where the underlying Ethereum transaction has a non-null `callDataId` (a file-backed call-data upload, a legitimate and observable on-chain pattern).
2. Attacker sends a high-concurrency stream of identical `GET /api/v1/contracts/<id>/results/<ts>` requests with no credentials.
3. Each request consumes 7 DB round-trips (1 serial + 5 parallel + 1 serial CTE). With N concurrent requests, the DB connection pool sees up to 7N simultaneous queries.
4. No server-side check rejects, delays, or counts these requests.

### Impact Explanation
Each request multiplies DB load by a factor of 7. A sustained flood of concurrent requests exhausts the PostgreSQL connection pool (configured via `rest/dbpool`), causing query queuing, latency spikes, and potential OOM on the DB server. Because the REST mirror node is a shared read-only service used by wallets, explorers, and dApps, degrading it affects all downstream consumers. The `file_data` CTE query (`FileDataService.latestFileContentsQuery`, lines 18–41 of `rest/service/fileDataService.js`) performs a `string_agg` over potentially large blobs, adding CPU and I/O pressure beyond simple row lookups.

### Likelihood Explanation
Preconditions are trivially met on any live network: Ethereum transactions with file-backed `callDataId` are publicly visible and indexable. No account, token, or privileged credential is required. The attack is fully repeatable and scriptable with a single `curl` loop or any HTTP load tool. The absence of any rate-limiting middleware in `rest/server.js` means the only external mitigation would be an operator-deployed reverse proxy, which is not guaranteed in all deployments.

### Recommendation
1. **Add rate limiting to the REST API**: Introduce a per-IP token-bucket middleware (e.g., `express-rate-limit`) in `rest/server.js` before the contract routes, mirroring the `ThrottleManager` pattern already used in the `web3` module.
2. **Cache the `callDataId` file lookup**: The `FileDataService.getLatestFileDataContents` result for a given `callDataId` is immutable once the file is finalized; cache it with a short TTL to avoid repeated DB hits for the same file.
3. **Parallelize the initial lookup**: Merge `getInvolvedContractsByTimestampAndContractId` into the `Promise.all` block to reduce serial round-trips from 3 phases to 2.
4. **Apply a DB statement timeout** for this endpoint to bound worst-case query duration.

### Proof of Concept

```bash
# Step 1: Discover a timestamp with callDataId via public API
TIMESTAMP=$(curl -s "https://<mirror>/api/v1/contracts/results?limit=25" \
  | jq -r '.results[] | select(.call_data_id != null) | .timestamp' | head -1)
CONTRACT=$(curl -s "https://<mirror>/api/v1/contracts/results?limit=25" \
  | jq -r '.results[] | select(.call_data_id != null) | .contract_id' | head -1)

# Step 2: Flood with concurrent unauthenticated requests (no credentials needed)
# Each request triggers 7 DB queries
for i in $(seq 1 200); do
  curl -s "https://<mirror>/api/v1/contracts/${CONTRACT}/results/${TIMESTAMP}" \
    -o /dev/null &
done
wait
# DB connection pool exhausted; subsequent requests queue or fail
