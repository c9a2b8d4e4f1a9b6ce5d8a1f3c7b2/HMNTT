### Title
Unauthenticated Repeated Calls to GET /api/v1/network/stake Trigger Uncached Correlated Subquery on Every Request

### Summary
The `findLatest()` method in `NetworkStakeRepository` executes a native SQL query containing a correlated subquery (`select max(consensus_timestamp) from network_stake`) on every invocation. No caching exists at any layer (repository, service, or controller), and no rate limiting is applied to the `/api/v1/network/stake` endpoint. Any unprivileged external user can flood this endpoint to cause sustained database CPU elevation.

### Finding Description
**Exact code path:**

- `NetworkController.getNetworkStake()` at line 127–130 handles `GET /api/v1/network/stake` with no rate limiting or throttling annotation.
- It calls `networkService.getLatestNetworkStake()` (`NetworkServiceImpl.java` lines 52–56), which directly calls `networkStakeRepository.findLatest()` with no `@Cacheable` or any other caching mechanism.
- `findLatest()` (`NetworkStakeRepository.java` lines 12–19) executes this native query on every call:
  ```sql
  select * from network_stake
  where consensus_timestamp = (
      select max(consensus_timestamp) from network_stake
  )
  ```

**Root cause:** The `network_stake` table is scanned twice per request — once for the `max()` subquery and once for the outer `select *`. There is zero caching at any layer. A grep across all `rest-java/src/main/java/**/*.java` confirms no `@Cacheable`, `CacheManager`, or equivalent annotation exists in this call chain. The `HIGH_VOLUME_THROTTLE` mechanism present in the codebase is exclusively wired to the fee estimation POST endpoint, not to `/network/stake`.

**Failed assumption:** The design assumes either (a) the endpoint is called infrequently, or (b) the database query is cheap enough to absorb concurrent load. Neither assumption holds under adversarial conditions.

### Impact Explanation
Each request to `GET /api/v1/network/stake` causes two full sequential scans (or index scans) of the `network_stake` table. Under sustained concurrent request load from a single or distributed unprivileged client, the database CPU can be driven well above 30% of baseline, degrading all other queries sharing the same database instance — including importer writes and other API endpoints. This constitutes a resource exhaustion denial-of-service against the shared database tier.

### Likelihood Explanation
No authentication is required. The endpoint accepts `GET` requests with no parameters, making it trivially scriptable. A single attacker with a basic HTTP client (e.g., `ab`, `wrk`, or a simple loop) can sustain thousands of requests per second. The attack requires no special knowledge, no credentials, and no exploit tooling beyond a standard HTTP client.

### Recommendation
1. **Add result caching** at the service layer using Spring's `@Cacheable` on `getLatestNetworkStake()` with a short TTL (e.g., 30–60 seconds), since `network_stake` data is updated only once per staking period (daily).
2. **Rewrite the query** to use `ORDER BY consensus_timestamp DESC LIMIT 1` instead of the correlated subquery, which is more optimizer-friendly and avoids the double scan.
3. **Apply rate limiting** to the `/api/v1/network/stake` endpoint consistent with the `HIGH_VOLUME_THROTTLE` pattern already used for fee estimation.

### Proof of Concept
```bash
# No credentials required. Run from any machine with network access.
while true; do
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/network/stake" &
done
# Or with a load tool:
wrk -t8 -c200 -d60s https://<mirror-node-host>/api/v1/network/stake
```
Each concurrent request independently executes:
```sql
select * from network_stake
where consensus_timestamp = (select max(consensus_timestamp) from network_stake)
```
with no deduplication, caching, or throttling, causing sustained DB CPU elevation proportional to request rate.
