### Title
Unauthenticated `/tmp/alive` Spoofing Bypasses Liveness Probe — Dead Pinger Masked from Orchestrator

### Summary
The liveness probe in `pinger/cmd/healthcheck/main.go` trusts only the existence and mtime of `/tmp/alive`, with no verification that the file was written by the legitimate pinger goroutine. Because `/tmp` is mounted as a world-writable `emptyDir` volume, any process with write access to the pod's `/tmp` (e.g., via `kubectl exec`, a sidecar, or an init container) can create or `touch` the file, causing the liveness probe to return exit 0 indefinitely even when the pinger goroutine has never run or has silently died.

### Finding Description

**Write side** — `pinger/main.go` lines 28–39:
```go
go func() {
    t := time.NewTicker(15 * time.Second)
    defer t.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-t.C:
            _ = os.WriteFile("/tmp/alive", []byte(time.Now().Format(time.RFC3339Nano)), 0644)
        }
    }
}()
``` [1](#0-0) 

The file is written with `0644` permissions into `/tmp`, which is an `emptyDir` volume (default mode `0777`). [2](#0-1) 

**Read side** — `pinger/cmd/healthcheck/main.go` lines 27–38:
```go
fi, err := os.Stat("/tmp/alive")
if err != nil {
    fmt.Fprintln(os.Stderr, err)
    os.Exit(1)
}
age := time.Since(fi.ModTime())
if age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)
``` [3](#0-2) 

The check is purely: **does the file exist AND is its mtime ≤ 2 minutes old?** There is no:
- File ownership check (`fi.Sys().(*syscall.Stat_t).Uid`)
- Process-identity binding (e.g., PID written into the file and cross-checked)
- Cryptographic signature or secret token
- Exclusive file lock held by the pinger

**Root cause**: The failed assumption is that only the pinger goroutine can write `/tmp/alive`. In reality, the `emptyDir` volume is world-writable, and any process sharing the pod's filesystem namespace can create or update the file.

**Exploit flow**:
1. Attacker gains write access to the pod's `/tmp` (see Likelihood section).
2. Before the pinger goroutine's first 15-second tick (or after the goroutine dies), attacker runs `touch /tmp/alive`.
3. `healthcheck live` reads the file, sees mtime = now, exits 0.
4. Kubernetes liveness probe passes; the dead/never-started pinger is never restarted.
5. Attacker repeats `touch /tmp/alive` every ~90 seconds to stay within the 2-minute window indefinitely.

**Startup probe is equally bypassed**: The startup probe also calls `healthcheck live` with `periodSeconds: 1` and `failureThreshold: 120`. [4](#0-3) 
A pre-created `/tmp/alive` causes the startup probe to pass on the very first check (second 0), before the goroutine has ever ticked.

### Impact Explanation
The pinger's sole purpose is to submit periodic transactions to the Hedera network and verify mirror-node propagation. If the pinger goroutine silently dies (panic, blocked goroutine, network hang) and the liveness probe is spoofed, Kubernetes never restarts the pod. The monitoring gap is invisible to the orchestrator. In a production environment this means SLA-critical liveness data stops flowing with no alerting, and the spoofed probe actively suppresses the self-healing mechanism that would otherwise restore service.

### Likelihood Explanation
The precondition is write access to `/tmp` inside the pod. This is achievable by:
- **`kubectl exec`**: Any developer or CI/CD service account with `pods/exec` RBAC permission (common in engineering orgs) can run `kubectl exec <pod> -- touch /tmp/alive`. This requires no root and no privilege escalation.
- **Compromised or malicious sidecar/init container**: The Helm chart mounts the same `emptyDir` at `/tmp` for the single container, but custom deployments or injected sidecars (e.g., Istio, logging agents) share the volume.
- **Supply-chain compromise of the image**: Any process running as UID 1000 inside the container can write to the `0777` emptyDir.

The attack is trivially repeatable with a simple shell loop and requires no special tooling.

### Recommendation
1. **Bind the heartbeat to the process**: Write the pinger's own PID into `/tmp/alive` and have `healthcheck` verify that a process with that PID still exists (`/proc/<pid>/` on Linux). This ties the file's validity to the actual running process.
2. **Use a secret token**: At startup, generate a random token, write it to a memory-only location, and include it in `/tmp/alive`. The healthcheck verifies the token matches.
3. **Restrict `/tmp` permissions**: Mount the `emptyDir` with `defaultMode: 0700` and ensure the pinger runs as a dedicated UID so no other process in the pod can write to it.
4. **Use a Unix domain socket or pipe**: Replace the file-based heartbeat with a local socket that only the pinger process listens on; the healthcheck connects to it, providing implicit liveness proof.

### Proof of Concept
```bash
# Precondition: kubectl exec access to the pinger pod (no root required)
POD=$(kubectl get pod -l app.kubernetes.io/component=pinger -o name | head -1)

# Step 1: Pre-create /tmp/alive before the goroutine's first 15s tick
# (or at any time to keep a dead pinger appearing alive)
kubectl exec $POD -- touch /tmp/alive

# Step 2: Verify the liveness probe now passes
kubectl exec $POD -- /healthcheck live
# exits 0

# Step 3: Keep the file fresh in a loop (runs every 90s, well within 2-minute window)
while true; do
  kubectl exec $POD -- touch /tmp/alive
  sleep 90
done

# Result: Even if the pinger goroutine panics or is killed (kill -9 <goroutine>),
# Kubernetes liveness probe continues to return exit 0 and never restarts the pod.
# The pinger's transaction submissions stop silently.
```
