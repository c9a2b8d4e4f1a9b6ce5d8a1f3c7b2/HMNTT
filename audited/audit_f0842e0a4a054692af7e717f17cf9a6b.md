### Title
Unbounded Request Body Allows Memory Exhaustion via `/construction/submit` (Missing `MaxBytesReader`)

### Summary
The Rosetta HTTP server in `rosetta/main.go` configures only time-based timeouts (`ReadTimeout`, etc.) but applies no byte-level cap on incoming request bodies. An unauthenticated attacker can POST an arbitrarily large JSON payload to `/construction/submit`, causing Go's `json.Decoder` to buffer the entire body in heap memory, exhausting available RAM and preventing the server from gossiping legitimate transactions to the Hiero network.

### Finding Description

**Exact code location:** `rosetta/main.go`, `main()`, lines 220–227.

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // default 5 s
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```

No `http.MaxBytesReader` is applied anywhere in the rosetta middleware chain. A grep across the entire `rosetta/` tree for `MaxBytesReader`, `LimitReader`, or any body-size guard returns zero results.

**Root cause:** The server relies solely on `ReadTimeout` (default `5 000 000 000 ns` = 5 s, per `docs/configuration.md` line 664) to bound how much data a client can push. This is a *time* bound, not a *size* bound. On a 1 Gbps link an attacker can deliver ≈ 625 MB per connection within that window; on a 10 Gbps link ≈ 6.25 GB.

**Exploit flow:**
1. The rosetta-sdk-go `ConstructionAPIController` calls `json.NewDecoder(r.Body).Decode(&request)` for every POST to `/construction/submit`. Go's `json.Decoder` reads the full body into heap memory before returning.
2. Because no `http.MaxBytesReader` wraps `r.Body`, the decoder will consume whatever the client sends until `ReadTimeout` fires.
3. An attacker opens *N* concurrent TCP connections and streams a large JSON body (e.g., a `signed_transaction` field filled with megabytes of hex characters) on each. Each goroutine allocates heap proportional to the body size.
4. The Go runtime OOMs or the OS kills the process; alternatively, GC pressure causes severe latency spikes that prevent the server from processing legitimate `/construction/submit` calls.

**Why existing checks are insufficient:**
- `ReadTimeout = 5 s` limits per-connection data to `bandwidth × 5 s`. With multiple concurrent connections this multiplies linearly.
- `ReadTimeout` is operator-configurable and may be raised in production for slow clients.
- No connection-count limit is set on the `http.Server`.
- The rosetta-sdk-go SDK controller does not impose its own body-size limit.
- No authentication or rate-limiting middleware is present in the middleware chain (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`). [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
A successful attack exhausts heap memory on the Rosetta node, causing an OOM crash or severe GC stalls. While the node is unavailable, no `/construction/submit` calls can be processed, meaning signed Hiero transactions cannot be gossiped to consensus nodes. This directly prevents transaction submission — the primary purpose of the online Rosetta endpoint — for the duration of the attack. Severity: **High** (availability impact on a critical transaction-submission path, no authentication required).

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable (default port 5700). No special knowledge or credentials are required. The attack is trivially scriptable with `curl` or any HTTP client. It is repeatable: after a crash/restart the attacker can immediately re-launch. The only friction is the 5-second `ReadTimeout`, which an attacker trivially works around by opening many parallel connections. Realistic on any internet-facing or exchange-facing deployment.

### Recommendation
Wrap `r.Body` with `http.MaxBytesReader` in a middleware applied before the rosetta-sdk-go router, limiting the body to a sane maximum (e.g., 1 MB — far larger than any valid Hiero transaction):

```go
func maxBodyMiddleware(next http.Handler, limit int64) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, limit)
        next.ServeHTTP(w, r)
    })
}
```

Apply it in `main()` before `metricsMiddleware`:

```go
limitedRouter := maxBodyMiddleware(router, 1<<20) // 1 MB
metricsMiddleware := middleware.MetricsMiddleware(limitedRouter)
```

Additionally, consider adding a connection-concurrency limit and rate-limiting per source IP.

### Proof of Concept

```bash
# Generate a ~50 MB JSON body with a large hex string in signed_transaction
python3 -c "
import json, sys
payload = json.dumps({
  'network_identifier': {'blockchain':'Hiero','network':'mainnet'},
  'signed_transaction': '0x' + 'ab' * 25_000_000
})
sys.stdout.write(payload)
" > /tmp/big_payload.json

# Open 20 concurrent connections, each sending the 50 MB body
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/construction/submit \
       -H 'Content-Type: application/json' \
       --data-binary @/tmp/big_payload.json &
done
wait
# Monitor: watch -n1 'ps aux | grep rosetta'
# Expected: RSS grows by ~1 GB+ per wave; server OOMs or becomes unresponsive
```
