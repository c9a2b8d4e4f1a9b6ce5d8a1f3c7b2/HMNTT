### Title
Global Rate-Limit Bucket Exhaustion via Unauthenticated Full-Burst Attack (Greedy Refill DoS)

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` is a single application-wide Bucket4j token bucket configured with `capacity(500).refillGreedy(500, Duration.ofSeconds(1))`. Because the bucket is global (not per-IP or per-user), starts fully loaded, and uses greedy (non-smoothed) refill, any unauthenticated external caller can fire 500 concurrent requests at the start of each second, atomically drain all tokens, and leave every other user receiving HTTP 429 for the remainder of that second. The attack is trivially repeatable every second with no credentials required.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `rateLimitBucket()`, lines 24–32:
```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond();   // default = 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))       // full burst allowed
            .build();
    return Bucket.builder().addLimit(limit).build();              // no per-IP, no sync strategy override
}
```

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, line 38:
```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Root cause and failed assumption:**

Bucket4j's `refillGreedy` accumulates tokens as fast as possible and does **not** smooth delivery. With `capacity == refillAmount` (both 500), the bucket initialises fully loaded and, after each 1-second window, all 500 tokens become available simultaneously. The design assumes traffic will be naturally distributed across the second, but this assumption is never enforced. There is no per-source-IP bucket, no connection-level limiting, no authentication gate, and no fairness mechanism. The single global bean is shared by every caller.

**Exploit flow:**

1. Attacker opens 500 parallel HTTP connections (no credentials needed).
2. At the moment the bucket refills (every ~1 s), all 500 `tryConsume(1)` calls race against the single shared bucket.
3. Because Bucket4j's default `LOCK_FREE` (CAS) strategy is thread-safe but not fair, the attacker's 500 threads collectively drain all tokens before legitimate requests can be served.
4. Every subsequent legitimate request within that second hits `tryConsume(1) == false` and receives HTTP 429.
5. The attacker waits ~1 second and repeats indefinitely.

**Why existing checks are insufficient:**

- `tryConsume(1)` enforces the global ceiling but provides zero per-caller fairness or isolation.
- No IP-based or session-based sub-bucket exists anywhere in the throttle stack.
- `gasLimitBucket` uses `SynchronizationStrategy.SYNCHRONIZED` (line 42 of `ThrottleConfiguration.java`) but `rateLimitBucket` does not, making it even more susceptible to concurrent drain races.
- `ThrottleProperties.requestsPerSecond` has `@Min(1)` but no upper bound that would force operators to set a safe value.

### Impact Explanation
All legitimate web3 API consumers (wallets, dApps, indexers) sharing this endpoint are denied service for up to 1 second per attack cycle. Because the cycle repeats every second with no cool-down or back-off enforced on the attacker, the effective availability of the endpoint can be reduced to near zero for all non-attacker users. For a mirror-node web3 endpoint used to submit or simulate transactions, this constitutes a sustained, targeted denial-of-service with no resource cost to the attacker beyond maintaining 500 open HTTP connections.

### Likelihood Explanation
No privileges, accounts, API keys, or special network position are required. The attack is executable with a single `ab -n 500 -c 500 <endpoint>` or equivalent tool. The 500 req/s default is low enough that a single commodity machine can sustain the burst indefinitely. The attack is fully repeatable, deterministic, and leaves no persistent state that could be used to identify or block the attacker at the application layer.

### Recommendation

1. **Switch to `refillIntervally`** so tokens are added at a fixed sub-second cadence (1 token per 2 ms for 500/s) rather than all at once, eliminating the full-burst window:
   ```java
   .refillIntervally(rateLimit, Duration.ofSeconds(1))
   ```
2. **Add per-IP rate limiting** as a first-line defence (e.g., a `ConcurrentHashMap<String, Bucket>` keyed on the client IP, or a Spring Cloud Gateway / nginx rate-limit filter upstream).
3. **Apply `SynchronizationStrategy.SYNCHRONIZED`** (or `LOCK_FREE` with a per-IP bucket) consistently across all buckets to prevent concurrent drain races.
4. **Set a hard cap** on `requestsPerSecond` via `@Max` in `ThrottleProperties` to prevent misconfiguration that widens the burst window.

```bash

  ab -n 500 -c 500 -p body.json -T application/json \
     http://<mirror-node-web3-host>/api/v1/contracts/call &
  sleep 1
done

# Simultaneously, legitimate user observes:
curl -s -o /dev/null -w "%{http_code}" \
     -X POST http://<mirror-node-web3-host>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"data":"0x...","to":"0x...","block":"latest"}'
# Returns: 429 (Too Many Requests) for the duration of the attack
```
