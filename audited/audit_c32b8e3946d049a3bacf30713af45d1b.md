### Title
Reference Equality (`==`) Bypass in `findByBlockType()` Causes `eth_call` with Hex Block `"0x00"` to Miss Genesis Block on Mainnet/Testnet

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `BlockType.EARLIEST` sentinel. Any numeric representation of block 0 — such as `"0x00"`, `"0x0"`, or `"0"` — causes `BlockType.of()` to construct a **new** `BlockType` object with `name="00"` and `number=0`, which is a different reference than `BlockType.EARLIEST` (`name="earliest"`, `number=0`). The `==` check therefore fails, and the call falls through to `findByIndex(0)`. On Hedera mainnet and testnet the genesis block carries a non-zero index (≈34 M and ≈22 M respectively), so `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would have returned the actual genesis block.

### Finding Description

**Code path:**

`BlockType.of("0x00")` — `BlockType.java` lines 73-79:
```
hexNum = "00"
Long.parseLong("00", 16) = 0
return new BlockType("00", 0L);   // NEW object, name ≠ "earliest"
``` [1](#0-0) 

`BlockType.EARLIEST` is defined as:
```java
public static final BlockType EARLIEST = new BlockType("earliest", 0L);
``` [2](#0-1) 

`findByBlockType()` guards with reference equality:
```java
if (block == BlockType.EARLIEST) {          // FALSE for new BlockType("00", 0)
    return recordFileRepository.findEarliest();
} ...
return recordFileRepository.findByIndex(block.number());  // called with 0
``` [3](#0-2) 

The two repository methods execute fundamentally different SQL:

- `findEarliest()`: `SELECT * FROM record_file ORDER BY index ASC LIMIT 1` — returns the row with the **lowest** index, whatever that is.
- `findByIndex(0)`: `SELECT r FROM RecordFile r WHERE r.index = 0` — returns a row only if index **exactly equals 0**. [4](#0-3) 

**Root cause:** `BlockType` is a Java `record`; its auto-generated `equals()` compares all components, but `==` checks object identity. The `EARLIEST` singleton is only returned when the literal string `"earliest"` is parsed via `blockTypeForTag()`. Any numeric path creates a fresh object. [5](#0-4) 

**Why the genesis block index is non-zero in production:** `BlockNumberMigration` hard-codes the correct genesis block numbers:
- Mainnet: index `34305852`
- Testnet: index `22384256`

and calls `updateIndex(offset)` to shift all rows to these values. [6](#0-5) [7](#0-6) 

After migration, no row in `record_file` has `index = 0` on mainnet or testnet, so `findByIndex(0)` returns `Optional.empty()`.

### Impact Explanation
Any `eth_call` (or `eth_getBalance`, `eth_getCode`, etc.) that specifies block `"0x00"` or `"0"` will receive a block-not-found error on mainnet/testnet, even though the Ethereum JSON-RPC specification treats block number 0 as equivalent to `"earliest"`. This constitutes unintended smart contract behavior: callers following the Ethereum spec will get unexpected failures, and any tooling or contract that queries historical state at block 0 will silently receive no data. No funds are directly at risk, consistent with the Medium classification.

### Likelihood Explanation
The trigger requires no privileges — any JSON-RPC client can submit `eth_call` with `"blockNumber": "0x00"`. The input is valid per the Ethereum spec and passes all input validation in `BlockType.of()`. The condition (genesis index ≠ 0) is permanently true on mainnet and testnet after `BlockNumberMigration` runs. The bug is therefore reliably and repeatably triggerable by any external user on every production deployment.

### Recommendation
Replace the reference-equality guards in `findByBlockType()` with value-based checks. The cleanest fix is to compare the `number()` field (and optionally the `name()` field) rather than object identity:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block.equals(BlockType.EARLIEST) || block.number() == BlockType.EARLIEST.number()) {
        return recordFileRepository.findEarliest();
    } else if (block.equals(BlockType.LATEST) || block.number() == BlockType.LATEST.number()) {
        return recordFileRepository.findLatest();
    } else if (block.isHash()) {
        return recordFileRepository.findByHash(block.name());
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Alternatively, make `BlockType.of()` return the `EARLIEST` singleton whenever the parsed numeric value equals 0, mirroring what `blockTypeForTag("earliest")` does.

### Proof of Concept

**Preconditions:** A running Hiero mirror-node web3 module connected to a mainnet or testnet database that has undergone `BlockNumberMigration` (genesis index ≈ 34 M on mainnet).

**Steps:**

1. Send a standard Ethereum JSON-RPC request:
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_call",
     "params": [{"to": "0x...","data": "0x..."}, "0x00"],
     "id": 1
   }
   ```
2. `BlockType.of("0x00")` returns `new BlockType("00", 0L)` — a new object.
3. In `findByBlockType()`, `block == BlockType.EARLIEST` evaluates to `false`.
4. `findByIndex(0)` executes `WHERE r.index = 0` — no row exists → `Optional.empty()`.
5. The caller receives a block-not-found / invalid block error.

**Contrast:** Sending the same request with `"earliest"` instead of `"0x00"` succeeds and returns the genesis block state, demonstrating the divergence.
