### Title
Unauthenticated Gas Estimation Requests Can Exhaust DB Connection Pool via 10-Second Timeout Window

### Summary
An unprivileged attacker can submit crafted `estimate=true` contract call requests that trigger the `BinaryGasEstimator` binary search loop, holding database connections for up to the full 10-second `requestTimeout` per request. Gas throttling is explicitly skipped for estimate calls, and the `safeCall` wrapper silently swallows `QueryTimeoutException`, allowing the loop to continue past the timeout boundary. With enough concurrent requests, the DB connection pool is exhausted.

### Finding Description

**Code path and root cause:**

`Web3Properties.java` line 20 sets the global timeout: [1](#0-0) 

`ContractCallContext.java` line 31 records `startTime` once at context creation — it is never reset across binary search iterations: [2](#0-1) 

`ContractCallContext.reset()` (called at the top of every binary search iteration) only clears `writeCache`, leaving `startTime` unchanged: [3](#0-2) 

`HibernateConfiguration.statementInspector()` checks elapsed time against the shared `startTime` before every SQL statement: [4](#0-3) 

`BinaryGasEstimator.safeCall()` catches **all** `Exception` types, including `QueryTimeoutException`, logs it, and returns `null` — the loop is never terminated: [5](#0-4) 

The binary search loop runs up to `maxGasEstimateRetriesCount = 20` iterations regardless: [6](#0-5) 

Gas throttling (`restoreGasToBucket`) is explicitly skipped for estimate calls: [7](#0-6) 

**Exploit flow:**

1. Attacker crafts a contract call that triggers many Hibernate SQL reads per EVM execution (e.g., a contract that reads many storage slots or calls precompiles that hit the DB).
2. Submits it as `estimate=true` (no authentication required).
3. `ContractCallContext.run()` creates a context; `startTime` is set once.
4. `BinaryGasEstimator.search()` runs up to 20 iterations; each calls `doProcessCall()` → `transactionExecutionService.execute()` → multiple Hibernate SQL statements → `statementInspector()` checks elapsed time.
5. For the first ~9.9 seconds, all SQL statements pass the check and DB connections are held.
6. When elapsed ≥ 10,000 ms, `statementInspector` throws `QueryTimeoutException`; `safeCall` catches it silently, returns `null`, and the loop continues.
7. Remaining iterations immediately fail at the first SQL check (fast-fail), but the damage — holding connections for ~10 seconds — is already done.
8. Attacker sends N concurrent such requests; with a typical connection pool of 10–20 connections, N ≈ 10–20 concurrent requests saturates the pool.

**Why existing checks fail:**

- `statementInspector` is the only timeout enforcement, but it only prevents new SQL statements after the deadline; it cannot interrupt in-progress EVM execution or release already-held connections.
- `safeCall`'s broad `catch (Exception ignored)` neutralizes the `QueryTimeoutException` as a loop-termination signal.
- Gas throttling is bypassed for estimate calls (`if (!estimate)` guard), so the attacker pays no gas cost and faces no rate limiting from the token bucket.

### Impact Explanation

An attacker can hold each DB connection for up to 10 seconds per request. With a standard HikariCP pool of 10 connections and 10 concurrent crafted estimate requests, the pool is fully saturated. All legitimate requests (including non-estimate calls) that need a DB connection will queue or fail. This constitutes a denial-of-service against the entire web3 module with no authentication and no gas cost, easily exceeding the 30% resource consumption threshold stated in scope.

### Likelihood Explanation

The attack requires no credentials, no on-chain funds, and no special tooling — only the ability to send HTTP POST requests to the public JSON-RPC endpoint. The crafted call data can be a simple contract that reads many storage slots. The attack is repeatable and automatable. `maxGasEstimateRetriesCount = 20` and `requestTimeout = 10s` are default configuration values, making every default deployment vulnerable.

### Recommendation

1. **Propagate timeout as a hard interrupt**: Do not catch `QueryTimeoutException` in `safeCall`; let it propagate out of `BinaryGasEstimator.search()` to terminate the entire estimate request immediately.
2. **Apply gas throttling to estimate calls**: Remove the `if (!estimate)` guard around `restoreGasToBucket`, or add a separate rate limiter for estimate requests.
3. **Reduce default `requestTimeout`** or add a separate, shorter timeout specifically for the estimate loop.
4. **Limit concurrent estimate requests per IP** at the servlet/filter layer before they reach the EVM.

### Proof of Concept

```
# 1. Deploy or reference any contract that reads many storage slots (e.g., 50+ SLOAD ops)
# 2. Send 15 concurrent estimate requests:

for i in $(seq 1 15); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "data": "<calldata that triggers 50+ SLOAD ops>",
      "to": "<contract address>",
      "estimate": true,
      "gas": 15000000
    }' &
done
wait

# 3. Simultaneously send a legitimate non-estimate call and observe it timing out
#    or returning a DB connection pool exhaustion error (HikariPool-1 - Connection is not available,
#    request timed out after Xms).
```

Each of the 15 concurrent requests holds DB connections for up to 10 seconds. With a default pool size of 10, legitimate traffic is fully blocked within seconds.
