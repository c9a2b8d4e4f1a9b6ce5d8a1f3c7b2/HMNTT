### Title
Hardcoded `sslmode=disable` in Rosetta `GetDsn()` Enables MITM Interception of Contract Entity Data

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` hardcodes `sslmode=disable` as a non-configurable string literal in the PostgreSQL DSN. Unlike every other component in the mirror node (importer, web3), the Rosetta `Db` struct has no `SslMode` field, making it impossible to enable transport encryption. An unprivileged attacker on the same network segment can intercept and modify plaintext PostgreSQL wire-protocol traffic between the Rosetta server and its database, injecting tampered contract entity data that causes unintended smart contract behavior in Rosetta API responses.

### Finding Description

**Exact code path:**

`rosetta/app/config/types.go`, lines 49–58:
```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        db.Host,
        db.Port,
        db.Username,
        db.Name,
        db.Password,
    )
}
```

The `Db` struct (lines 39–47) contains `Host`, `Name`, `Password`, `Pool`, `Port`, `StatementTimeout`, and `Username` — but **no `SslMode` field**. The string `sslmode=disable` is a hardcoded literal with no override path.

This DSN is consumed directly in `rosetta/app/db/db.go` line 18:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), &gorm.Config{...})
```

**Contrast with other components:** The importer (`importer/src/main/java/org/hiero/mirror/importer/db/DBProperties.java`, line 50) exposes `SslMode sslMode = SslMode.DISABLE` as a configurable field. The web3 component documents `hiero.mirror.web3.db.sslMode` as configurable (`docs/configuration.md`, line 701). The Rosetta component has no equivalent — `docs/configuration.md` lines 661–663 list Rosetta DB properties with no `sslMode` entry.

**Root cause:** The failed assumption is that the Rosetta server and its PostgreSQL database always communicate over a trusted, isolated channel. In the Kubernetes deployment (`charts/hedera-mirror-rosetta/templates/deployment.yaml`), the Rosetta pod connects to a shared cluster-internal PostgreSQL endpoint. Kubernetes cluster networking does not provide wire-level encryption between pods by default, and ARP-based MITM is feasible within a flat pod network or shared VLAN.

**Exploit flow:**
1. Attacker gains a foothold on the same network segment (e.g., compromises any other pod in the same Kubernetes namespace — no cluster-admin privileges required).
2. Attacker performs ARP spoofing to intercept TCP traffic on port 5432 between the Rosetta pod and the PostgreSQL service.
3. Because `sslmode=disable` is hardcoded, the PostgreSQL client never negotiates TLS; all wire-protocol frames are plaintext.
4. Attacker uses a PostgreSQL MITM proxy (e.g., `pgmitm`, `mitmproxy` with a Postgres plugin) to intercept query responses.
5. Attacker modifies rows returned for contract entity queries — e.g., altering `entity` table rows for contract accounts, changing bytecode hashes, or modifying contract state — before forwarding to the Rosetta server.
6. The Rosetta server processes the tampered data and returns corrupted contract entity information in its API responses.

**Why existing checks fail:** There are none. The `sslmode=disable` literal cannot be overridden by configuration, environment variable, or any runtime mechanism. The test in `rosetta/app/config/types_test.go` line 24 explicitly asserts the hardcoded `sslmode=disable` string, cementing the behavior.

### Impact Explanation

The Rosetta API is used by exchanges to integrate with the Hedera network. Contract entity data returned by the Rosetta server (via `AccountRepository`, `BlockRepository`, `TransactionRepository` — all wired in `rosetta/main.go` lines 70–73) is sourced entirely from the unencrypted database connection. An attacker who tampers with this data can cause the Rosetta server to return incorrect contract entity states, corrupted block data, or falsified transaction records. This constitutes unintended smart contract behavior at the mirror node layer (layer 1/2 network code), matching the stated scope. No direct fund theft is possible through this vector alone, but exchange integrations relying on Rosetta responses for contract state verification could be misled.

### Likelihood Explanation

The precondition — network access to the same segment as the Rosetta pod and its database — is achievable by any attacker who compromises a single unprivileged pod in the same Kubernetes namespace or cluster network. This requires no cluster-admin or database credentials. In cloud Kubernetes environments without strict NetworkPolicy enforcement (the Helm chart at `charts/hedera-mirror-rosetta` defines no NetworkPolicy), lateral movement from any compromised workload to the Rosetta-DB segment is realistic. ARP spoofing tools are widely available and require no special privileges within a pod. The attack is repeatable for the lifetime of the deployment since the SSL disable is hardcoded and cannot be patched at runtime.

### Recommendation

1. Add a `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a default of `"require"` or `"verify-full"`.
2. Replace the hardcoded literal in `GetDsn()`:
   ```go
   "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
   db.Host, db.Port, db.Username, db.Name, db.Password, db.SslMode,
   ```
3. Document `hiero.mirror.rosetta.db.sslMode` in `docs/configuration.md` alongside the other Rosetta DB properties, consistent with the importer and web3 components.
4. Update `rosetta/app/config/types_test.go` to test configurable SSL modes.
5. Consider enforcing Kubernetes `NetworkPolicy` to restrict pod-to-database traffic to only the Rosetta pod.

### Proof of Concept

**Preconditions:** Attacker has shell access to any pod in the same Kubernetes namespace as the Rosetta deployment (e.g., via a compromised sidecar or debug container).

**Steps:**

```bash
# 1. From attacker pod: identify Rosetta pod IP and PostgreSQL service IP
kubectl get pods -o wide          # get rosetta pod IP, e.g. 10.0.1.5
kubectl get svc                   # get postgres service IP, e.g. 10.0.1.10

# 2. Enable IP forwarding and perform ARP spoofing (no root needed in many k8s configs)
echo 1 > /proc/sys/net/ipv4/ip_forward
arpspoof -i eth0 -t 10.0.1.5 10.0.1.10 &   # tell rosetta pod that attacker is postgres
arpspoof -i eth0 -t 10.0.1.10 10.0.1.5 &   # tell postgres that attacker is rosetta pod

# 3. Intercept and modify PostgreSQL wire protocol (plaintext, no TLS negotiation)
# Use a tool like pg-mitm or a custom Python script with scapy to intercept
# DataRow messages and modify contract entity fields in query responses.

# 4. Verify: query the Rosetta /block endpoint and observe tampered contract entity data
curl -X POST http://<rosetta-service>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier": {...}, "block_identifier": {"index": 1}}'
# Response will contain attacker-injected contract entity data
```

The attack succeeds because `GetDsn()` in `rosetta/app/config/types.go` line 51 unconditionally emits `sslmode=disable`, so the PostgreSQL client never sends a TLS handshake request and all wire frames remain plaintext throughout the session.
