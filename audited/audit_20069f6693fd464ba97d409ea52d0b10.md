### Title
Integer Overflow in `poll()` Limit Cast Causes Unbounded Database Query via `limit=Long.MAX_VALUE` Subscription

### Summary
In `PollingTopicMessageRetriever.poll()`, the remaining-limit calculation `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to `-1` when a user submits `limit=Long.MAX_VALUE`. The resulting negative `pageSize` is passed to `topicMessageRepository.findByFilter()` via a rebuilt filter, where `hasLimit()` returns `false` for negative values, causing `setMaxResults()` to be skipped entirely and the query to return all topic messages without any bound. No privilege is required to trigger this.

### Finding Description

**Entry point** — `ConsensusController.toFilter()`: [1](#0-0) 

The proto field `uint64 limit` is mapped directly to Java `long` with no upper-bound check. `Long.MAX_VALUE` (9223372036854775807) is a valid `uint64` value and passes the only validation constraint: [2](#0-1) 

`@Min(0)` only rejects negative values; `Long.MAX_VALUE` passes. `hasLimit()` returns `true` since `Long.MAX_VALUE > 0`: [3](#0-2) 

**Overflow site** — `PollingTopicMessageRetriever.poll()` lines 68–71: [4](#0-3) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- `(int) 0x7FFFFFFFFFFFFFFF` = lower 32 bits = `0xFFFFFFFF` = **-1**
- `Math.min(-1, maxPageSize)` = **-1**

**Rebuilt filter bypasses validation** — line 75: [5](#0-4) 

`filter.toBuilder().limit(-1).build()` is an internal Lombok builder call; it is never passed through a `@Valid`-annotated method, so Bean Validation is not triggered and the `@Min(0)` constraint on `limit` is never enforced for this internal filter.

**Guard bypassed in repository** — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53: [6](#0-5) 

`filter.hasLimit()` evaluates `limit > 0` → `-1 > 0` → **false**. `setMaxResults()` is never called. The JPA query executes with no `LIMIT` clause and `getResultList()` loads the entire topic message table partition into heap memory: [7](#0-6) 

**Termination condition never fires** — `isComplete()`: [8](#0-7) 

`filter.getLimit() == total.get()` requires `total` to reach `Long.MAX_VALUE`, which is practically impossible. The subscription polls indefinitely, executing an unbounded query on every poll cycle.

**Outer `flux.take()` does not help** — `TopicMessageServiceImpl.subscribeTopic()` line 84: [9](#0-8) 

`flux.take(Long.MAX_VALUE)` is effectively a no-op limit at the reactive layer.

The identical overflow pattern also exists in `PollingTopicListener.poll()`: [10](#0-9) 

### Impact Explanation

Each poll cycle issues a `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT`, loading the entire message history for the targeted topic into JVM heap. For a high-traffic topic with millions of messages this causes:
- **OOM / heap exhaustion** on the mirror node gRPC process
- **Full table scan / index scan** on the database for every poll interval, saturating I/O
- **Indefinite subscription lifetime** — the connection is never terminated by the limit logic, compounding resource consumption across concurrent attackers

This is a server-side Denial of Service reachable without any authentication or special role.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly exposed. The attacker needs only a valid topic ID (trivially discoverable from the public ledger) and a standard gRPC client. Setting `limit = Long.MAX_VALUE` (or any value > `Integer.MAX_VALUE`, e.g., `2147483648`) in the `ConsensusTopicQuery` proto message is a single-field change. The attack is repeatable, requires no credentials, and can be parallelized across many connections to amplify impact.

### Recommendation

1. **Fix the cast**: Replace the unsafe narrowing cast with a bounds-safe helper:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
   Apply the same fix to `PollingTopicListener.poll()`.

2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)   // or a configurable service maximum
   private long limit;
   ```

3. **Validate internally-built filters**: Either trigger programmatic validation on `newFilter` before passing it to the repository, or add a guard in `findByFilter()` that treats any non-positive limit as "no results" rather than "no limit".

4. **Add a configurable maximum subscription limit** in `RetrieverProperties` and enforce it at the controller layer before the filter is constructed.

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),  # any valid topic
    limit=9223372036854775807,  # Long.MAX_VALUE — passes @Min(0), triggers overflow
)

# Each streamed response triggers poll() → (int)(Long.MAX_VALUE - 0) = -1
# → findByFilter with no LIMIT → full table scan loaded into heap
for response in stub.subscribeTopic(query):
    pass  # server is now executing unbounded DB queries on every poll interval
```

Repeating this across N concurrent connections multiplies the unbounded query load by N, exhausting database and JVM heap resources.
