### Title
Non-Retryable Empty-Network Error via Port-Zero Endpoint Injection Enables Permanent Pinger DoS

### Summary
`fetchMirrorNodeNetwork()` silently skips every `serviceEndpoint` whose `Port` field is `0` and, when the resulting network map is empty, returns a **non-retryable** error (`retry = false`). The caller `buildNetworkFromMirrorNodes()` immediately breaks its retry loop on any non-retryable error, so a single crafted HTTP response from the mirror REST server causes `newClient()` to fail, which `main()` escalates to `log.Fatalf`, permanently terminating the pinger process.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–129:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← line 118: Port==0 silently skipped
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf(...)  // ← line 128: retry=false, non-retryable
}
```

`buildNetworkFromMirrorNodes()`, lines 59–61:

```go
if !retry || attempt == attempts {
    break   // ← exits immediately; all remaining retries are skipped
}
```

`main.go`, lines 41–44:

```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)  // ← process exits permanently
}
```

**Root cause:** The code conflates two distinct failure modes under the same non-retryable error path: (a) a legitimately empty node list (a transient or misconfigured server response) and (b) a maliciously crafted response where every endpoint carries `Port == 0`. Because the empty-network condition is classified as non-retryable, the retry budget (`mirrorNodeClientMaxRetries`, default 10) is never consumed — a single bad response is sufficient to kill the process.

**Failed assumption:** The developer assumed that an empty network map indicates a permanent, unrecoverable state (e.g., wrong URL), so retrying would be pointless. In reality, a server under attacker control can produce this state on demand, and the non-retryable classification turns a transient injection into a permanent outage.

**Exploit flow:**

1. Attacker positions themselves to respond to the pinger's HTTP GET to `cfg.mirrorRest + /api/v1/network/nodes` (see below for how).
2. They return HTTP 200 with a syntactically valid JSON body, e.g.:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"node.example.com","port":0}]}],"links":{"next":null}}
   ```
3. `fetchMirrorNodeNetwork()` parses the response successfully, skips the endpoint at line 118 (`Port == 0`), finds `len(network) == 0`, and returns `nil, false, error`.
4. `buildNetworkFromMirrorNodes()` sees `retry=false` and breaks immediately (line 59), returning the error regardless of the configured retry count.
5. `newClient()` propagates the error; `main()` calls `log.Fatalf`, terminating the process.

**Why existing checks are insufficient:**

- The retry loop (`mirrorNodeClientMaxRetries`) is bypassed entirely because `retry=false` causes an immediate break on the first attempt.
- There is no TLS enforcement or certificate pinning on the mirror REST URL; the default example in `config.go` line 37 is `http://mirror-rest:5551` (plain HTTP), making MITM trivial on any shared or container network.
- There is no validation that the returned node list is non-empty before classifying the error as non-retryable.

### Impact Explanation

The pinger process calls `log.Fatalf` on `newClient()` failure (`main.go` line 43), which calls `os.Exit(1)`. This is a **permanent process termination** — not a degraded state. The liveness probe (`/tmp/alive`) stops being updated, and the readiness file (`/tmp/ready`) is never created, so Kubernetes (or any orchestrator) will detect the failure and restart the pod. However, if the attacker can sustain the malicious response (e.g., they control the mirror REST server or maintain a MITM position), every restart attempt will immediately fail at the same point, creating a persistent crash-loop that effectively takes the pinger offline indefinitely. This is a complete Denial-of-Service against the pinger's monitoring/health-check function.

### Likelihood Explanation

The precondition — ability to respond to the pinger's HTTP request to the mirror REST URL — is achievable by:

- **MITM on plain HTTP:** The default/example URL uses `http://` (no TLS). An attacker on the same container network, Kubernetes pod network, or physical segment can intercept and forge the response with standard tools (e.g., ARP spoofing, rogue DNS, iptables redirect).
- **DNS poisoning:** If `cfg.mirrorRest` uses a hostname, poisoning the DNS resolver used by the pinger container redirects the request to an attacker-controlled server.
- **Compromised or shared mirror REST service:** In multi-tenant or shared infrastructure, a party with write access to the mirror REST service can serve the crafted response.

No authentication, no TLS verification, and no response integrity check are required. The attack is repeatable: every time the pinger restarts, it re-queries the mirror REST endpoint at startup, so the attacker only needs to maintain their position to sustain the outage.

### Recommendation

1. **Make the empty-network condition retryable.** Change line 128 to return `retry = true`:
   ```go
   if len(network) == 0 {
       return nil, true, fmt.Errorf("no usable service_endpoints found from %s", url)
   }
   ```
   This ensures the full retry budget is consumed before giving up, making a single injected response insufficient.

2. **Enforce TLS for the mirror REST URL.** Reject or warn on `http://` URLs in `loadConfig()`, and use TLS with certificate verification for all mirror REST connections.

3. **Add a minimum-node-count threshold.** Treat a response with fewer than N usable endpoints as a transient error (retryable), not a fatal one.

4. **Separate "empty response" from "permanent misconfiguration."** Only treat the empty-network case as non-retryable after exhausting all retries, not on the first occurrence.

### Proof of Concept

**Preconditions:** Attacker can intercept or respond to HTTP requests from the pinger to its configured `mirrorRest` URL (e.g., via DNS redirect or MITM on plain HTTP).

**Steps:**

1. Stand up a minimal HTTP server that always returns:
   ```
   HTTP/1.1 200 OK
   Content-Type: application/json

   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"node.example.com","ip_address_v4":"","port":0}]}],"links":{"next":null}}
   ```
2. Configure or redirect the pinger's `HIERO_MIRROR_PINGER_REST` to point to this server (or MITM the existing URL).
3. Start the pinger with `HIERO_MIRROR_PINGER_NETWORK=other`.
4. **Observed:** `fetchMirrorNodeNetwork()` skips the endpoint (Port==0), returns `nil, false, error`. `buildNetworkFromMirrorNodes()` breaks immediately (retry=false). `newClient()` returns error. `main()` calls `log.Fatalf` → process exits with status 1.
5. **Expected (correct behavior):** The retry loop should consume all configured retries before giving up, and/or the connection should use TLS to prevent response forgery.
