### Title
Cache Poisoning via Broken `unless` Condition on `Optional` Return Type in `findRuntimeBytecode()`

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to prevent caching of absent results, but the method returns `Optional<byte[]>`. A Spring Data JPA `Optional` is never `null` — `Optional.empty()` is a non-null object — so the guard is permanently inert. Every call, including those for non-existent contract IDs, is unconditionally cached. An unprivileged attacker can pre-query a predictable future contract ID, poison the cache with `Optional.empty()`, and cause the mirror node's EVM layer to treat a subsequently deployed contract as having no bytecode for the duration of the cache TTL.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18.

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's SpEL expression `#result` evaluates to the actual return value of the method. Because the return type is `Optional<byte[]>`, `#result` is always a non-null `Optional` instance. `Optional.empty() == null` is `false`. The `unless` predicate therefore never fires, and every result — including `Optional.empty()` for a non-existent contract — is stored in the Caffeine cache managed by `CACHE_MANAGER_CONTRACT`.

**Exploit flow:**
1. On Hedera, entity (contract) IDs are sequential and publicly observable. An attacker can predict the ID that will be assigned to the next deployed contract.
2. Before the contract is deployed, the attacker sends any `eth_call` or equivalent web3 request that causes `findRuntimeBytecode(predictedId)` to be invoked. The DB returns no row; the method returns `Optional.empty()`.
3. Because `Optional.empty() != null`, the `unless` guard does not suppress caching. `Optional.empty()` is stored in the cache under key `predictedId`.
4. The legitimate contract is deployed and its bytecode is written to the `contract` table.
5. Any subsequent call to `findRuntimeBytecode(predictedId)` within the cache TTL returns the stale `Optional.empty()` from cache without hitting the DB.
6. `ContractBytecodeReadableKVState.readFromDataSource()` receives `Optional.empty()`, maps to `null`, and returns `null` to the EVM — the contract appears to have no bytecode.

**Why existing checks fail:** The sole guard is the `unless` expression. No secondary validation, no cache eviction on contract creation, and no fallback DB read exists in the call path through `ContractBytecodeReadableKVState`.

### Impact Explanation
Any `eth_call`, `eth_estimateGas`, or contract-execution request routed through the mirror node's web3 module that targets the poisoned contract ID will receive a response as if the contract does not exist (empty bytecode → EVM treats address as EOA or reverts). This is a targeted, repeatable denial-of-service against specific newly deployed contracts. Severity is **Medium-High**: it does not compromise funds directly, but it can silently break dApp integrations and mislead callers about contract state.

### Likelihood Explanation
Hedera entity IDs are monotonically increasing integers visible on-chain and via the mirror node REST API. Predicting the next contract ID requires only reading the current highest contract ID — no privileged access needed. The attack is trivially scriptable: poll for the latest contract ID, immediately query `findRuntimeBytecode(latestId + 1)`, repeat. The window between prediction and deployment is wide enough for automated exploitation.

### Recommendation
Fix the `unless` expression to correctly handle the `Optional` wrapper:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures `Optional.empty()` (contract not found) is never cached, while `Optional.of(bytecode)` (contract found) continues to be cached normally. Alternatively, consider using `@Cacheable`'s `condition` parameter or a custom `CacheResolver` that inspects `Optional` presence.

### Proof of Concept
1. Query the mirror node REST API for the current highest contract entity ID (e.g., `GET /api/v1/contracts?order=desc&limit=1` → ID = `N`).
2. Send an `eth_call` to the web3 endpoint targeting contract ID `N+1` (which does not yet exist). This triggers `findRuntimeBytecode(N+1)` → DB miss → `Optional.empty()` cached.
3. Deploy a new contract on the Hedera network; it receives ID `N+1` and its bytecode is written to the `contract` table.
4. Immediately send a valid `eth_call` to the web3 endpoint targeting the newly deployed contract at ID `N+1`.
5. Observe that the response behaves as if no contract exists (e.g., empty return data or revert), because the cache returns the stale `Optional.empty()` from step 2 instead of querying the DB for the now-present bytecode.
6. Wait for the cache TTL to expire; repeat the `eth_call` — it now succeeds, confirming the issue was cache-based. [1](#0-0) [2](#0-1) [3](#0-2)
