### Title
`public, max-age=5` Cache-Control Header Applied to 404 Responses Enables CDN/Proxy Stale-Error Caching for Topic Endpoint

### Summary
The `@ModelAttribute` hook in `GenericControllerAdvice` unconditionally writes `cache-control: public, max-age=5` onto the raw `HttpServletResponse` for every request matching the `/api/v1/topics/{id}` path, including requests that ultimately resolve to a 404. The `notFound()` exception handler returns a `ResponseEntity` with `null` headers, which does not clear the already-written header. Any RFC 7234-compliant CDN or reverse proxy in front of the service will therefore cache a 404 response for up to 5 seconds, causing legitimate users to receive stale "not found" errors for recently-created topics.

### Finding Description

**Code path:**

`GenericControllerAdvice.java` lines 79–84 — the `@ModelAttribute` method runs before the controller and writes headers directly onto `HttpServletResponse`:

```java
@ModelAttribute
private void responseHeaders(HttpServletRequest request, HttpServletResponse response) {
    var requestMapping = String.valueOf(
        request.getAttribute(HandlerMapping.BEST_MATCHING_PATTERN_ATTRIBUTE));
    var responseHeaders = properties.getResponse().getHeaders().getHeadersForPath(requestMapping);
    responseHeaders.forEach(response::setHeader);   // sets cache-control: public, max-age=5
}
``` [1](#0-0) 

When `TopicServiceImpl.findById()` throws `jakarta.persistence.EntityNotFoundException` (line 20), the `notFound()` handler is invoked:

```java
@ExceptionHandler
private ResponseEntity<Object> notFound(final EntityNotFoundException e, final WebRequest request) {
    return handleExceptionInternal(e, null, null, NOT_FOUND, request);
}
``` [2](#0-1) 

`handleExceptionInternal` is called with `null` headers and constructs `new ResponseEntity<>(errorResponse, headers, statusCode)`. [3](#0-2) 

Spring MVC's `HttpEntityMethodProcessor` merges the `ResponseEntity`'s headers (empty/null) into the existing `HttpServletResponse` headers — it does **not** clear them. The `cache-control: public, max-age=5` header written by `@ModelAttribute` therefore survives on the 404 response.

The `TopicServiceImpl.findById()` itself has no caching logic: [4](#0-3) 

The test at line 94 of `TopicControllerTest` confirms `public, max-age=5` is the configured value for this path: [5](#0-4) 

**Root cause:** The header-injection mechanism (`@ModelAttribute`) is status-code-agnostic. There is no guard that suppresses or replaces `cache-control` with `no-store` / `no-cache` when the response status is 4xx.

### Impact Explanation
RFC 7234 §3 permits a shared cache to store any response that carries an explicit `public` directive with a positive `max-age`, regardless of status code. A CDN (Cloudflare, AWS CloudFront, Fastly, nginx proxy_cache, etc.) placed in front of the mirror-node REST Java service will cache a 404 for `/api/v1/topics/0.0.X` for up to 5 seconds. During that window every user routed through the same CDN PoP receives the stale 404 instead of the real topic data, even after the topic has been ingested by the mirror node. The impact is limited to a 5-second availability gap for newly created topics — griefing with no direct economic loss, matching the stated scope.

### Likelihood Explanation
No special privileges are required. Any unprivileged actor who knows (or can predict) a topic ID — trivially possible because Hedera topic IDs are sequential — can issue a GET request to `/api/v1/topics/0.0.<next_id>` a fraction of a second before the topic is created, causing the CDN to cache the 404. The attack is repeatable at zero cost and requires only an HTTP client. Deployments without a CDN or caching proxy are unaffected, but production mirror-node deployments commonly sit behind one.

### Recommendation
In `GenericControllerAdvice`, override the `cache-control` header to `no-store` (or at minimum `no-cache`) for all non-2xx responses. The cleanest fix is to add a check inside `handleExceptionInternal`:

```java
@Override
protected ResponseEntity<Object> handleExceptionInternal(...) {
    // existing logic ...
    HttpHeaders safeHeaders = new HttpHeaders();
    if (headers != null) safeHeaders.putAll(headers);
    safeHeaders.setCacheControl(CacheControl.noStore());
    return new ResponseEntity<>(errorResponse, safeHeaders, statusCode);
}
```

Alternatively, add a servlet filter or `OncePerRequestFilter` that rewrites `cache-control` to `no-store` whenever the response status is ≥ 400.

### Proof of Concept
1. Stand up the mirror-node REST Java service behind any RFC 7234-compliant caching proxy (e.g., nginx with `proxy_cache`).
2. Choose a topic ID that does not yet exist, e.g., `0.0.99999`.
3. Send: `GET /api/v1/topics/0.0.99999` — observe `HTTP 404` with `cache-control: public, max-age=5` in the response headers.
4. The proxy caches the 404 for 5 seconds.
5. Create topic `0.0.99999` on the Hedera network and wait for the mirror node to ingest it (typically < 1 second).
6. Within the 5-second cache window, send `GET /api/v1/topics/0.0.99999` again — the proxy returns the cached 404 instead of the real topic data.
7. After 5 seconds the cache expires and the correct 200 response is returned.

    protected ResponseEntity<Object> handleExceptionInternal(
            Exception ex, @Nullable Object body, HttpHeaders headers, HttpStatusCode statusCode, WebRequest request) {

        Error errorResponse =
                switch (ex) {
                    case BindException e -> bindExceptionResponse(e);
                    case Errors errors -> errorResponse(errors.getAllErrors());
                    case MethodValidationResult errors -> errorResponse(errors.getAllErrors());
                    default -> {
                        var message =
                                statusCode instanceof HttpStatus hs ? hs.getReasonPhrase() : statusCode.toString();
                        var detail = body instanceof ProblemDetail pb ? pb.getDetail() : ex.getMessage();
                        var nonSensitiveDetail = !statusCode.is5xxServerError() ? detail : StringUtils.EMPTY;
                        yield errorResponse(message, nonSensitiveDetail);
                    }
                };

        request.setAttribute(WebUtils.ERROR_EXCEPTION_ATTRIBUTE, ex, SCOPE_REQUEST);
        return new ResponseEntity<>(errorResponse, headers, statusCode);
```
