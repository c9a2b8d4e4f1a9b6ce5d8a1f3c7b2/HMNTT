### Title
Silent Upper-Bound Overwrite in `Bound` Constructor Allows Unprivileged Users to Widen Query Range

### Summary
The `Bound` constructor in `Bound.java` iterates over all supplied `RangeParameter` entries and unconditionally overwrites `upper` with each successive upper-bound parameter. When a user supplies two upper-bound operators for the same field (e.g., `receiver.id=lte:100&receiver.id=lte:200`), the second value silently replaces the first, causing the effective query to use the wider bound (200) rather than the narrower one (100). The `verifySingleOccurrence()` guard that would catch this is never invoked in the token-airdrop request path.

### Finding Description

**Exact code path:**

In `TokenAirdropsController.java`, `processRequest()` at line 97 constructs a `Bound` directly from the raw `entityIds` array:

```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
``` [1](#0-0) 

The `@Size(max = 2)` annotation on the `receiverIds` parameter explicitly permits up to two values:

```java
@RequestParam(name = RECEIVER_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] receiverIds,
``` [2](#0-1) 

Inside the `Bound` constructor, the loop at lines 47–54 assigns `upper` without any duplicate-detection guard:

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;
    } else if (param.hasUpperBound()) {
        upper = param;   // ← second lte/lt param silently overwrites first
    }
    cardinality.merge(param.operator(), 1, Math::addExact);
}
``` [3](#0-2) 

With input `receiver.id=lte:100&receiver.id=lte:200`:
- Iteration 1: `upper = lte:100`
- Iteration 2: `upper = lte:200` ← overwrites, cardinality `LTE → 2`

The only post-loop check is `adjustedLower > adjustedUpper` (line 58), which only fires when `primarySortField=true` **and** a lower bound is also present. With no lower bound, `adjustedLower = 0` and `adjustedUpper = 200`, so `0 > 200` is false — no exception is thrown. [4](#0-3) 

**The existing guard that would fix this — `verifySingleOccurrence()` — is never called in this path.** A grep across the entire codebase shows it is only invoked in `NftAllowanceServiceImpl.java`, not in `TokenAirdropServiceImpl` or `TokenAirdropsController`: [5](#0-4) 

`TokenAirdropServiceImpl.getAirdrops()` passes the request directly to the repository with no bound validation:

```java
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}
``` [6](#0-5) 

### Impact Explanation
An unprivileged caller can silently expand the effective upper bound of any `receiver.id` (or `sender.id`) filter. The query executes against the wider range without any error or warning, returning records the caller's visible parameters do not appear to request. This constitutes unauthorized data over-exposure: a caller who appears to be filtering to `receiver_account_id <= 100` actually receives all records up to `receiver_account_id <= 200`. The same technique applies to `token.id` and `serial_number` secondary bounds, which also pass through `Bound` without `verifySingleOccurrence()` being called.

### Likelihood Explanation
No authentication or privilege is required. The endpoint is a public GET API. The `@Size(max = 2)` annotation explicitly documents and permits two values for the same parameter, making the attack trivially discoverable by any user reading the API contract or experimenting with repeated query parameters. HTTP clients and browsers routinely send repeated query parameters, so this requires no special tooling.

### Recommendation
Call `entityIdsBound.verifySingleOccurrence()` (and equivalently for `tokenIds` and `serialNumbers` bounds) immediately after constructing each `Bound` in `processRequest()`, before passing the request to the service layer. This mirrors the pattern already used in `NftAllowanceServiceImpl`. Alternatively, move the call into the `Bound` constructor itself when `primarySortField=true`, or add a dedicated validation step in `TokenAirdropServiceImpl.getAirdrops()`.

### Proof of Concept

**Preconditions:** The mirror node REST-Java service is running. Token airdrops exist for receiver account IDs between 101 and 200.

**Trigger:**
```
GET /api/v1/accounts/{id}/airdrops/outstanding?receiver.id=lte:100&receiver.id=lte:200
```

**Expected behavior:** Only records with `receiver_account_id <= 100` are returned.

**Actual behavior:** Records with `receiver_account_id <= 200` are returned, because the `Bound` constructor overwrites `upper` with the second `lte:200` parameter and no validation rejects the duplicate upper-bound operator.

**Verification:** Repeat with `receiver.id=lte:1&receiver.id=lte:999999999` — the response will include all airdrop records regardless of receiver ID, confirming the narrower bound is silently discarded.
