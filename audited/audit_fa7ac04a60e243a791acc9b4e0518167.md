### Title
Reference-Equality Bypass on `BlockType.EARLIEST` Causes `findByIndex(0)` Instead of `findEarliest()` for Block Parameter `"0"`

### Summary
`findByBlockType()` guards the earliest-block path with a Java reference-equality check (`block == BlockType.EARLIEST`). When an unprivileged caller supplies the decimal string `"0"` as the block parameter, `BlockType.of("0")` constructs a **new** `BlockType("0", 0L)` instance that is a distinct object from the `EARLIEST` singleton, so the guard is never taken. Execution falls through to `recordFileRepository.findByIndex(0)`, which queries `WHERE index = 0`, while `findEarliest()` queries `ORDER BY index ASC LIMIT 1`. On any mirror-node deployment whose earliest stored block has an index other than 0, the two queries return different (or no) results, causing `eth_call` to execute against the wrong historical state or to fail entirely.

### Finding Description

**Root cause — reference equality on a value type**

`BlockType` is a Java `record`: [1](#0-0) 

The singleton is: [2](#0-1) 

When a caller passes `"0"`, the factory method hits the decimal branch and allocates a fresh object: [3](#0-2) 

This new `BlockType("0", 0L)` is **not** the same heap object as `EARLIEST`, so the guard in `findByBlockType()` fails: [4](#0-3) 

Execution reaches `recordFileRepository.findByIndex(block.number())` with `number() == 0`.

**Divergence between the two queries**

`findEarliest()` returns the record file with the **lowest** index, regardless of what that index value is: [5](#0-4) 

`findByIndex(0)` performs an exact match on `index = 0`: [6](#0-5) 

The importer exposes `updateIndex(long offset)` which shifts every row's index by an arbitrary offset: [7](#0-6) 

After any such rebase, or on a partial mirror that starts ingestion from a non-zero block height, no row with `index = 0` exists. `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would return the actual genesis record.

**Exploit flow**

1. Attacker (no credentials required) issues `eth_call` with `"blockNumber": "0"` (or `"0x0"`).
2. `BlockType.of("0")` → `new BlockType("0", 0L)` (new instance).
3. `block == BlockType.EARLIEST` → `false`.
4. `block == BlockType.LATEST` → `false`; `block.isHash()` → `false` (number ≠ -1).
5. `recordFileRepository.findByIndex(0)` is called.
6. On a reindexed or partial mirror: returns `Optional.empty()` → call fails / wrong error path; on a mirror where a non-genesis file happens to hold index 0 after rebase: call executes against that file's state.

### Impact Explanation
Any `eth_call` or historical simulation that relies on the `"earliest"` / `"0"` block equivalence is broken. On partial or reindexed mirror nodes (a common production topology), the call either returns an error instead of the genesis state, or silently executes against a wrong block. Smart-contract logic that depends on genesis-state values (initial balances, constructor storage) will produce incorrect results. Severity is **Medium**: no direct fund loss, but incorrect historical EVM execution is delivered to callers who expect canonical earliest-block semantics.

### Likelihood Explanation
The trigger requires zero privileges — any JSON-RPC client can pass `"blockNumber": "0"`. The condition (mirror node whose earliest stored index ≠ 0) is common in production deployments that start from a checkpoint or have undergone index rebase. The bug is deterministic and trivially repeatable.

### Recommendation
Replace the reference-equality checks with value-based comparisons, or canonicalize user input through the singleton map:

```java
// Option A – value equality
if (BlockType.EARLIEST.equals(block)) {
    return recordFileRepository.findEarliest();
} else if (BlockType.LATEST.equals(block)) {
    return recordFileRepository.findLatest();
}
```

Because `BlockType` is a `record`, `equals()` compares `name` and `number` fields, so `BlockType("0", 0L).equals(EARLIEST)` is `false` (names differ: `"0"` vs `"earliest"`). The correct fix is therefore to check the **name** field or the numeric value explicitly:

```java
if ("earliest".equals(block.name())) {
    return recordFileRepository.findEarliest();
}
```

Alternatively, make `BlockType.of` return the `EARLIEST` singleton whenever the resolved number is 0 and the input is not a hash, mirroring the existing tag path.

### Proof of Concept

**Precondition:** mirror node whose earliest `record_file.index` is not 0 (e.g., partial mirror starting at block 5000, or after `updateIndex` rebase).

**Steps:**

```bash
# 1. Send eth_call with block number 0 (decimal)
curl -X POST http://<mirror-node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","id":1,
    "method":"eth_call",
    "params":[{"to":"<contract>","data":"<calldata>"}, "0"]
  }'

# 2. Send eth_call with the "earliest" tag (correct path)
curl -X POST http://<mirror-node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","id":2,
    "method":"eth_call",
    "params":[{"to":"<contract>","data":"<calldata>"}, "earliest"]
  }'
```

**Expected:** both calls return identical results (genesis state).

**Actual:** call 1 hits `findByIndex(0)` → `Optional.empty()` (no row with index 0) → error response or wrong state; call 2 hits `findEarliest()` → correct genesis record file → correct result.

The divergence confirms the reference-equality bypass.
