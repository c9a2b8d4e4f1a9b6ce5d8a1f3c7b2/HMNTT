### Title
Offloaded Call Data Not Reconstructed in EVM Replay — Opcode Trace Diverges from Original Execution

### Summary
When an Ethereum transaction stored in the mirror node DB has its call data offloaded to a Hedera file (`callDataId` set, `callData` empty in the stored RLP), `OpcodeServiceImpl.getEthereumDataBytes()` returns the raw RLP with an empty calldata field. This RLP is passed verbatim to `TransactionExecutionService.buildEthereumTransactionBody()`, which submits it to the hedera-app `TransactionExecutor` without the file-backed calldata. The EVM replay therefore executes with empty calldata, producing a trace that is entirely different from the original on-chain execution.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` → `OpcodeServiceImpl.processOpcodeCall()` → `buildCallServiceParameters()` → `buildCallServiceParameters(consensusTimestamp, transaction, ethereumTransaction)`: [1](#0-0) 

Two helpers are called here: [2](#0-1) 

`getCallDataBytes()` returns `ethereumTransaction.getCallData()` when an `EthereumTransaction` is present. For offloaded-calldata transactions this field is `null`/empty (the RLP stored in `data` has an empty calldata slot; the real calldata lives in a Hedera file identified by `callDataId`). The code never falls back to `contractResult.getFunctionParameters()`, which *does* hold the actual calldata after import.

`getEthereumDataBytes()` returns `ethereumTransaction.getData()` — the raw RLP bytes as submitted to Hedera, which contains an empty calldata field.

**Root cause — failed assumption:** The code assumes `ethereumTransaction.getCallData()` always contains the full calldata. This is false for transactions where calldata was offloaded to a file (a supported Hedera feature). The `data` field stores the original, truncated RLP; no reconstruction step re-inserts the file-backed calldata before replay.

**Why the replay diverges:** `TransactionExecutionService.buildEthereumTransactionBody()` wraps the truncated RLP directly into an `EthereumTransactionBody`: [3](#0-2) 

The `EthereumTransactionBody` carries only `ethereumData` and `maxGasAllowance` — no `callData` file reference. The hedera-app `TransactionExecutor` therefore parses the RLP, finds empty calldata, and executes with it. The resulting opcode trace reflects a completely different execution path than the original transaction.

**Existing checks are insufficient:** `patchSenderNonce()` patches the nonce but does nothing about calldata. There is no guard in `getCallDataBytes()` or `getEthereumDataBytes()` that detects the offloaded-calldata case and fetches the file content before replay. [4](#0-3) 

### Impact Explanation
Any opcode trace for a transaction whose calldata was offloaded to a Hedera file will show execution with empty calldata. The resulting trace — opcodes, stack, memory, storage — reflects a fabricated execution context, not the original one. Developers and security auditors relying on `debug_traceTransaction` to understand historical contract behavior will receive misleading output. This matches the stated scope: unintended smart contract behavior in the trace layer with no direct fund risk.

### Likelihood Explanation
Offloaded calldata is a documented, production-used Hedera feature (evidenced by test fixtures and the `RAW_TX_TYPE_1_CALL_DATA_OFFLOADED` constant in the test suite). Any unprivileged user can trigger the divergence by supplying the hash of any such transaction to the public endpoint. No special privileges, credentials, or prior knowledge beyond the transaction hash are required. The condition is repeatable for every affected transaction.

### Recommendation
In `OpcodeServiceImpl.getCallDataBytes()`, when `ethereumTransaction != null` but `ethereumTransaction.getCallData()` is empty/null and `ethereumTransaction.getCallDataId()` is non-empty, fetch the calldata from the file store (mirroring the logic in `EthereumTransactionHandler.updateContractResult()`) and use it for both `callData` and the reconstructed `ethereumData`. Alternatively, reconstruct the full RLP (re-inserting the file-backed calldata) before passing it to `getEthereumDataBytes()`, so the `TransactionExecutor` receives a complete ethereum transaction body matching the original on-chain execution.

### Proof of Concept
1. Identify any historical Ethereum transaction with offloaded calldata: query the `ethereum_transaction` table for rows where `call_data` is empty and `call_data_id` is non-null (e.g., the testnet transaction at `consensusTimestamp=1706866862835165003` in the test fixture).
2. Obtain its `hash` value.
3. Call `GET /api/v1/contracts/results/{hash}/opcodes` with `Accept-Encoding: gzip`.
4. Observe the returned opcode trace: it reflects execution with empty calldata (e.g., immediate STOP or revert at the entry point) rather than the actual function dispatch that occurred on-chain.
5. Compare against the on-chain contract actions for the same transaction to confirm the divergence.

```java
    private byte[] getCallDataBytes(EthereumTransaction ethereumTransaction, ContractResult contractResult) {
        final var callData = ethereumTransaction != null
                ? ethereumTransaction.getCallData()
                : contractResult.getFunctionParameters();
        return callData != null ? callData : new byte[0];
    }

    private byte[] getEthereumDataBytes(EthereumTransaction ethereumTransaction) {
        if (ethereumTransaction == null) {
            return new byte[0];
        }
        final var data = ethereumTransaction.getData();
        return data != null ? data : new byte[0];
    }
```

```java
                        .ethereumData(Bytes.wrap(params.getEthereumData()))
                        .maxGasAllowance(Long.MAX_VALUE)

                .build();
```
