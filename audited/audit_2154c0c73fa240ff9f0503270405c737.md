### Title
Missing HTTP/2 RST_STREAM Rate Limiting Enables Rapid Reset DoS Against gRPC Subscription Delivery

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` (default: 5) and a shared executor, with no HTTP/2-level connection lifetime limits, no RST_STREAM rate limiting, and no `maxConnectionAge`/`maxConnectionIdle` settings. An unprivileged attacker can open a persistent connection and continuously cycle HEADERS+RST_STREAM frame pairs, causing repeated resource allocation and disposal in the gRPC/Reactor layer faster than cleanup completes, exhausting the shared `applicationTaskExecutor` thread pool and starving legitimate topic-message subscribers.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection` is enforced at the **gRPC application layer**, not at the HTTP/2 frame layer. It limits the number of *simultaneously active* gRPC calls, but does not limit the *rate* at which streams can be opened and reset. When a client sends a HEADERS frame immediately followed by RST_STREAM, the server:
1. Allocates a stream object and dispatches to the executor
2. In `ConsensusController.subscribeTopic()` (lines 44–52), creates a Reactor `Mono`/`Flux` chain and registers a cancel handler (`disposable::dispose`)
3. Receives RST_STREAM and triggers `setOnCancelHandler` → `disposable::dispose`, which tears down the Reactor pipeline

Steps 1–3 consume executor threads and Reactor scheduler resources. Because cleanup is asynchronous (Reactor scheduler + potential DB interaction via `TopicMessageServiceImpl`), a new stream can be opened before the previous one fully cleans up, even within the 5-slot limit. The attacker simply waits for one slot to appear free (or uses multiple connections) and repeats.

**No `maxConnectionAge`, `maxConnectionIdle`, or HTTP/2-level stream-reset rate limiting is configured anywhere in the codebase** — confirmed by exhaustive search returning zero results for these parameters.

**Why existing checks fail:** `maxConcurrentCallsPerConnection = 5` prevents more than 5 *concurrent* active calls, but the RST_STREAM rapid reset pattern keeps slots in a transitional (cleanup) state. The gRPC Netty layer counts a stream as "active" until cleanup completes; during high-rate cycling, cleanup latency means slots are perpetually occupied by dying streams, and the executor queue fills with cancel/dispose tasks.

### Impact Explanation
The shared `applicationTaskExecutor` is used for all gRPC call dispatch (line 32: `serverBuilder.executor(applicationTaskExecutor)`). Saturating it with rapid-reset cleanup tasks directly starves legitimate `subscribeTopic` calls from receiving topic messages. Since this gRPC service is the delivery path for Hedera consensus topic messages (which carry submitted transaction data to subscribers), sustained attack degrades or halts message delivery to all connected clients — a denial of service against the mirror node's primary gRPC function. Multiple attacker connections from different IPs multiply the effect linearly with no per-IP connection limit in the server configuration.

### Likelihood Explanation
No authentication or authorization is required to connect to port 5600. The attack requires only a standard HTTP/2 client capable of sending HEADERS+RST_STREAM pairs in rapid succession (e.g., `h2load`, `grpc-client`, or a raw HTTP/2 frame sender). This is a well-documented attack class (CVE-2023-44487 / HTTP/2 Rapid Reset), tooling is publicly available, and the missing mitigations are straightforward to confirm from the configuration. A single attacker with a modest network connection can sustain the attack indefinitely since connections have no age limit.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java` and expose corresponding properties in `NettyProperties.java`:

```java
serverBuilder.maxConnectionAge(30, TimeUnit.MINUTES);
serverBuilder.maxConnectionAgeGrace(5, TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
serverBuilder.permitKeepAliveTime(30, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Additionally, add `maxConcurrentCallsPerConnection` enforcement at the HTTP/2 `SETTINGS_MAX_CONCURRENT_STREAMS` level (Netty's `Http2Settings`) so the limit is enforced before the gRPC layer allocates resources. Consider adding an ingress-level rate limiter (e.g., via the existing Traefik middleware or GCP backend policy already present in `charts/hedera-mirror-grpc/values.yaml`) to limit stream-open rate per source IP.

### Proof of Concept
**Preconditions:** Network access to the gRPC server on port 5600; no credentials required.

**Steps:**
```bash
# Install h2load (part of nghttp2)
# Target: grpc-server:5600

# Send rapid HTTP/2 streams with immediate RST_STREAM
# Each request opens a stream and immediately resets it
h2load -n 100000 -c 10 -m 5 \
  --header=":method: POST" \
  --header=":path: /com.hedera.mirror.api.proto.ConsensusService/subscribeTopic" \
  --header="content-type: application/grpc" \
  --data=/dev/null \
  http://grpc-server:5600
```

Alternatively, using a raw HTTP/2 client script that sends HEADERS frame on stream N then immediately RST_STREAM on stream N, cycling through stream IDs at maximum rate across 10 parallel connections.

**Expected result:** The `applicationTaskExecutor` thread pool saturates with cancel/dispose tasks; legitimate `subscribeTopic` calls queue indefinitely or are rejected; Prometheus metric `grpc_server_processing_duration_seconds_count` shows a spike in `CANCELLED` status codes while new legitimate subscriptions stop receiving messages.
