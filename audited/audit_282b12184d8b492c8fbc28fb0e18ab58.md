### Title
Unbounded Request Body Parsing on `/mempool` Endpoint Enables Memory Exhaustion via Oversized JSON Payloads

### Summary
The `Mempool()` handler in `rosetta/app/services/mempool_service.go` immediately discards its `*types.NetworkRequest` argument and returns `ErrNotImplemented`, but the `rosetta-sdk-go` framework's `MempoolAPIController` fully reads and JSON-decodes the request body before invoking the handler. No `http.MaxBytesReader` or equivalent body-size cap is applied anywhere in the rosetta server stack, so an unauthenticated attacker can send arbitrarily large payloads—exploiting the `Metadata map[string]interface{}` field—to force unbounded heap allocation and GC pressure on every request.

### Finding Description

**Exact code path:**

`rosetta/app/services/mempool_service.go` lines 22–27 — the handler signature accepts `*types.NetworkRequest` but the parameter is named `_` and never touched:

```go
func (m *mempoolAPIService) Mempool(
    _ context.Context,
    _ *types.NetworkRequest,   // body already fully parsed by the SDK
) (*types.MempoolResponse, *types.Error) {
    return nil, errors.ErrNotImplemented
}
```

`rosetta/main.go` lines 220–227 — the `http.Server` is constructed with only time-based limits; no `http.MaxBytesReader` middleware is inserted anywhere in the chain (`corsMiddleware → tracingMiddleware → metricsMiddleware → router`):

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```

**Root cause:** The `rosetta-sdk-go` `MempoolAPIController` calls `json.NewDecoder(r.Body).Decode(&networkRequest)` on the raw `http.Request.Body` before dispatching to the service. Because no `http.MaxBytesReader` wraps the body, Go's JSON decoder will stream-read and allocate memory for the entire payload regardless of size. The `types.NetworkRequest.Metadata` field is typed `map[string]interface{}`, accepting arbitrary nested JSON structures.

**Why existing checks are insufficient:**

- `ReadTimeout` / `ReadHeaderTimeout` (configured in `rosetta/app/config/types.go` lines 64–69) are *time-based*, not *size-based*. A co-located or high-bandwidth attacker can deliver hundreds of megabytes within a typical 30-second read timeout.
- The Traefik `inFlightReq` (5 concurrent) and `rateLimit` (10 req/s) middleware in `charts/hedera-mirror-rosetta/values.yaml` lines 152–161 are Kubernetes-deployment-specific and absent when the server is run directly. Even when present, 5 concurrent 200 MB payloads = 1 GB of simultaneous heap allocation.
- No application-level authentication or authorization is required to reach `/mempool`.

### Impact Explanation
Each oversized request forces the Go runtime to allocate a large heap object, triggering major GC cycles. Under concurrent bombardment (even within the Traefik `inFlightReq: 5` limit), the server's GC pause time increases, goroutine scheduling degrades, and legitimate requests experience elevated latency or timeouts. Sustained attack causes effective denial of service for all users of the rosetta node without any economic cost to the attacker.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The attacker needs only `curl` or any HTTP client. The `Metadata` field accepts arbitrary JSON, so crafting a valid-looking oversized payload is trivial. The attack is repeatable at the rate limit (10 req/s × up to ~200 MB each = ~2 GB/s of forced allocation per attacker IP when Traefik is absent). Even with Traefik, 5 concurrent large requests are permitted per cycle.

### Recommendation
Wrap `r.Body` with `http.MaxBytesReader` before JSON decoding. Since the SDK's controller owns the decode step, the cleanest fix is to add a middleware that replaces the body with a size-capped reader before the request reaches the SDK router:

```go
func maxBodyMiddleware(maxBytes int64, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
        next.ServeHTTP(w, r)
    })
}
```

Insert it in `rosetta/main.go` before `corsMiddleware`. A limit of 1–4 KB is sufficient for any valid `NetworkRequest` payload.

### Proof of Concept

```bash
# Generate a ~50 MB oversized metadata payload
python3 -c "
import json, sys
payload = {'network_identifier': {'blockchain': 'Hedera', 'network': 'mainnet'},
           'metadata': {'x': 'A' * 50_000_000}}
sys.stdout.write(json.dumps(payload))
" > big_payload.json

# Send to /mempool endpoint (repeat concurrently for GC pressure)
for i in $(seq 1 5); do
  curl -s -X POST http://<rosetta-host>:5700/mempool \
       -H 'Content-Type: application/json' \
       --data-binary @big_payload.json &
done
wait
```

Each request forces ~50 MB of heap allocation inside the SDK's JSON decoder before `Mempool()` is even called. Monitor server GC metrics (`hiero_mirror_rosetta_request_duration`) to observe latency degradation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
