### Title
Batch DB Query Amplification via Stale `contractSlotsCache` Key Accumulation

### Summary
An unprivileged external user can accumulate up to 1,500 unique slot keys in the per-contract `contractSlotsCache` (TTL: 5 minutes) by issuing standard `eth_getStorageAt` JSON-RPC calls. Because `contractStateCache` expires every 2 seconds, every subsequent cache miss for that contract triggers `contractStateRepository.findStorageBatch()` with all 1,500 accumulated keys in the SQL `IN` clause, producing a sustained 1,500× DB query amplification ratio with a single request every 2 seconds.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 58–70) checks `contractStateCache` (spec: `expireAfterWrite=2s,maximumSize=25000`). On a miss it calls `findStorageBatch()` (line 85–122).

Inside `findStorageBatch()`:
- Line 86–87: retrieves or creates a per-contract `CaffeineCache` from `cacheManagerSlotsPerContract` (spec: `expireAfterAccess=5m,maximumSize=1500`).
- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally registers the incoming slot key.
- Line 91: `contractSlotsCache.getNativeCache().asMap().keySet()` — collects **all** previously registered keys for this contract.
- Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a single SQL query with **all** accumulated keys in the `IN` clause. [1](#0-0) 

The SQL query at the repository level: [2](#0-1) 

**Root cause and failed assumption:**

The design assumes that once a batch query populates `contractStateCache`, subsequent requests will be cache hits and the large `IN` clause will not be re-issued. This assumption fails because `contractStateCache` has `expireAfterWrite=2s` while `contractSlotsCache` (per-contract key registry) has `expireAfterAccess=5m`. The 150× TTL mismatch means the key registry outlives the value cache by orders of magnitude. [3](#0-2) 

After every 2-second expiry of `contractStateCache`, the next single request for any slot of that contract re-triggers the full batch query with all K accumulated keys.

**Why existing checks are insufficient:**

- The early-return at line 63–67 only fires on a `contractStateCache` hit; it cannot fire after the 2-second TTL expires.
- `maximumSize=1500` on `slotsPerContract` caps K but does not prevent the amplification — it merely bounds the worst-case factor.
- There is no rate limiting, authentication, or per-IP throttling visible in this code path. [4](#0-3) 

### Impact Explanation

Each attacker request after the warm-up phase causes the DB to evaluate a `WHERE slot IN (k1, k2, ..., k1500)` predicate instead of a single-key lookup. At 1,500 keys this is a 1,500× increase in per-request DB work. Sustained at one request per 2 seconds (the `contractStateCache` TTL), this yields 750 full-batch queries per 25-minute window per targeted contract. An attacker targeting multiple contracts multiplies this linearly. This is sufficient to increase DB CPU/IO by well over 30% on a lightly loaded node without any brute-force volume, satisfying the stated threat model.

### Likelihood Explanation

`eth_getStorageAt(contractAddress, slotIndex, "latest")` is a standard, unauthenticated JSON-RPC call available on every public mirror-node endpoint. No wallet, token, or privileged credential is required. The attacker needs only to know (or enumerate) 1,500 slot indices for one contract — sequential integers `0x0` through `0x5DC` suffice even if most slots are empty, because empty-slot keys are still registered in `contractSlotsCache` (line 90) and included in every subsequent batch query. The attack is fully repeatable and scriptable.

### Recommendation

1. **Align TTLs**: Set `contractSlotsCache` (per-contract key registry) TTL ≤ `contractStateCache` TTL (2 s), or set `contractStateCache` TTL ≥ `slotsPerContract` TTL (5 m). The current 150× mismatch is the root cause.
2. **Evict slot keys on value-cache population**: After `findStorageBatch` populates `contractStateCache` (line 106–113), remove the corresponding keys from `contractSlotsCache` so they are not re-batched on the next miss.
3. **Cap batch size independently of cache size**: Enforce a hard limit (e.g., 50–100 keys) on `cachedSlots` before passing it to `findStorageBatch`, regardless of how many keys are in the registry.
4. **Rate-limit `eth_getStorageAt`** at the API gateway layer per IP/contract pair.

### Proof of Concept

```python
import requests, time

RPC = "https://<mirror-node>/api/v1/contracts/results"  # or JSON-RPC endpoint
CONTRACT = "0x<target_contract>"
NODE = "http://<mirror-node>:8545"

def eth_get_storage(slot_hex):
    payload = {
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT, slot_hex, "latest"], "id": 1
    }
    requests.post(NODE, json=payload)

# Phase 1: warm up — register 1500 unique slot keys (takes ~1500 requests)
for i in range(1500):
    eth_get_storage(hex(i))

# Phase 2: sustain amplification — one request every 2s triggers 1500-key SQL IN clause
while True:
    eth_get_storage("0x0")   # single request → 1500-key batch query to DB
    time.sleep(2)             # wait for contractStateCache to expire
```

**Expected result:** DB query logs show `WHERE slot IN (...)` with 1,500 bind parameters on every iteration, with a single client request per cycle.
