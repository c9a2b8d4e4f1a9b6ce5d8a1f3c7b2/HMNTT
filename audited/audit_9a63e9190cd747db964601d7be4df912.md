### Title
Misplaced `sync.Once` in `initGenesisRecordFile` Allows Concurrent `selectGenesis` CTE Query Amplification

### Summary
In `rosetta/app/persistence/block.go`, the `initGenesisRecordFile()` function uses `sync.Once` only to protect the *write* to `br.genesisBlock`, but the expensive `selectGenesis` CTE query executes *before* and *outside* that guard. The initial uninitialized check at line 244 is an unsynchronized plain read, so N concurrent goroutines can all pass it simultaneously and each independently execute the full `selectGenesis` query against the database, with only one result ever being used.

### Finding Description
**File:** `rosetta/app/persistence/block.go`
**Function:** `initGenesisRecordFile()`, lines 243–263

The code structure is:

```
line 244: if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset { return nil }
          // ^^^ plain unsynchronized read — no mutex, no atomic
line 252: db.Raw(selectGenesis, ...).First(&rb)   // expensive CTE — runs OUTSIDE sync.Once
line 257: br.once.Do(func() { br.genesisBlock = rb })  // sync.Once only guards the write
``` [1](#0-0) 

The `sync.Once` field exists on the struct [2](#0-1)  but is placed *after* the query, not around it. The correct pattern would be to wrap the entire query-and-assign inside `br.once.Do(...)`.

**Race window:** Before genesis is initialized, every goroutine that reaches line 244 sees `ConsensusStart == -1` and proceeds to execute `selectGenesis`. Because there is no mutex or `sync.Once` wrapping the query, all N concurrent goroutines execute it in parallel. Only one goroutine's result is committed via `br.once.Do`; the rest are discarded, but the DB work is already done.

**Extended window:** If the query fails (e.g., `account_balance` or `record_file` tables are not yet populated), `br.once.Do` is never reached, so `br.genesisBlock` stays at the sentinel value. Every subsequent request re-enters the query path indefinitely until the query succeeds. [3](#0-2) 

**Entry points:** All five public methods (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) call `initGenesisRecordFile` unconditionally. [4](#0-3) 

**The `selectGenesis` query** is a multi-table CTE joining `account_balance` (with `ORDER BY consensus_timestamp LIMIT 1`) and `record_file` (with a correlated subquery and `ORDER BY consensus_end LIMIT 1`). It is not a trivial point-lookup. [5](#0-4) 

### Impact Explanation
An attacker sending a burst of N concurrent HTTP requests to any Rosetta API endpoint during node startup (or during any period where the genesis query fails) causes N simultaneous executions of the `selectGenesis` CTE. Because the query involves two table scans with ordering and a correlated subquery, the DB CPU cost scales linearly with N. With a modest burst (e.g., 100–500 concurrent requests, achievable with standard HTTP load tools), DB CPU consumption can easily exceed 30% above baseline, degrading service for all legitimate users. The attack is unauthenticated and requires no special privileges.

### Likelihood Explanation
The Rosetta API is a public HTTP interface. No authentication is required to call `FindByHash`, `FindByIndex`, or `RetrieveLatest`. The attack is most effective during node startup or when the DB is slow to populate genesis data (the window can be indefinitely extended if the query keeps returning `ErrNodeIsStarting`). Any external actor with network access can reproduce this with a single `ab`, `wrk`, or `hey` invocation.

### Recommendation
Move the entire query inside `br.once.Do` so the expensive operation is guaranteed to execute at most once, regardless of concurrency:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: if the query can fail transiently (node starting), a `sync.Once` that captures the error permanently may be undesirable. In that case, use a `sync.Mutex`-protected check-and-set pattern instead, so failed attempts can be retried while still serializing concurrent callers.

### Proof of Concept
**Preconditions:** Node is starting up (genesis not yet initialized), or DB is slow enough that `selectGenesis` returns no rows.

**Steps:**
1. Start the mirror node Rosetta service before genesis data is available in the DB.
2. From an external machine with no credentials, run:
   ```
   hey -n 500 -c 200 http://<rosetta-host>/network/status
   ```
   (or any endpoint that triggers `RetrieveLatest` → `initGenesisRecordFile`)
3. Observe on the DB host: 200 simultaneous executions of the `selectGenesis` CTE query appear in `pg_stat_activity`.
4. DB CPU spikes proportionally; the spike persists for the duration of all concurrent query executions.
5. Repeat the burst on each node restart to sustain elevated DB CPU.
