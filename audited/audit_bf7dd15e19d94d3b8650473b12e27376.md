### Title
Unauthenticated Access to Prometheus Metrics Endpoint via Falsy `authentication` Config

### Summary
In `rest/middleware/metricsHandler.js`, the `authenticate()` function returns `true` (grants access) whenever `config.metrics.config.authentication` is falsy — including `undefined`, `null`, `0`, or `""`. Because the default config file (`rest/config/application.yml`) was not found in the repository, any deployment where this property is absent or misconfigured silently disables authentication, allowing any unprivileged external user to retrieve the full Prometheus metrics payload from the `metricsPath` endpoint without credentials.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144.

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {   // ← returns true for undefined, null, 0, "", false
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
``` [1](#0-0) 

The guard at line 157 is `if (!authenticate(req))` — so when `authenticate()` returns `true`, the request proceeds directly to metrics collection and serialization: [2](#0-1) 

**Root cause:** The `!authentication` branch conflates two distinct states: (a) the operator explicitly disabled auth (`authentication: false`) and (b) the property was never set (`authentication: undefined`). Both produce `!authentication === true`, both return `true`, and both grant unauthenticated access. The documentation declares the default as `true`: [3](#0-2) 

However, the default config file (`rest/config/application.yml`) does not exist in the repository. The config loader at lines 218–231 of `config.js` silently skips missing files: [4](#0-3) 

If the file is absent, `config.metrics.config.authentication` is `undefined`, `!undefined` is `true`, and `authenticate()` returns `true` — granting access to every caller.

**Exploit flow:**
1. Attacker identifies the metrics path. Per config defaults, `uriPath = '/swagger'`, so `metricsPath = '/swagger/metrics/'`.
2. Attacker sends `GET /swagger/metrics/` with no `Authorization` header.
3. `authenticate(req)` reads `authentication` as `undefined` → `!undefined` → `true` → returns `true`.
4. The `if (!authenticate(req))` guard is skipped; `exporter.collect()` is called and the full Prometheus payload is returned.

**Why existing checks fail:** There is no secondary guard, no IP allowlist, no network-layer restriction enforced in this code path. The sole protection is the `authenticate()` function, which is bypassed entirely when the config property is absent.

### Impact Explanation
The metrics payload exposes:
- `api_request{method, path, code}` — per-route request counts, revealing which API paths exist and their traffic volumes
- `api_request_duration_milliseconds` — per-route latency histograms, enabling timing-based inference of backend behavior
- `api_all_errors`, `api_all_server_error`, `api_all_client_error` — error rates that reveal instability windows
- `nodejs_process_cpu_usage_percentage`, `nodejs_process_memory_*` — process-level resource telemetry [5](#0-4) 

An attacker can use this data to identify under-monitored or high-error routes, infer deployment topology, and time attacks to coincide with high-load or high-error windows. Severity: **Medium** (information disclosure with no direct data exfiltration, but meaningful reconnaissance value).

### Likelihood Explanation
- **Precondition:** The default config file is absent from the repository, meaning any deployment that does not explicitly supply `authentication: true` in a custom config is vulnerable by default.
- **Attacker capability:** Zero — no credentials, no special network position, no prior knowledge beyond the default `uriPath` value (`/swagger`) which is documented publicly.
- **Repeatability:** Fully repeatable; the endpoint is stateless and returns fresh metrics on every request.
- **Feasibility:** A single unauthenticated HTTP GET suffices.

### Recommendation
Replace the falsy check with a strict boolean equality check so that only an explicit `false` disables authentication, and treat any non-boolean or absent value as "authentication required":

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  // Only disable auth when explicitly set to boolean false
  if (authentication === false) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

Additionally, add `authentication: true` to the default config file (`rest/config/application.yml`) so the property is always present with a safe default, and add a startup validation that rejects a missing or non-boolean `authentication` value.

### Proof of Concept
```bash
# 1. Start the REST API without explicitly setting metrics.config.authentication
#    (i.e., rely on the absent default config file)

# 2. Send an unauthenticated GET to the metrics endpoint
curl -v http://<host>:5551/swagger/metrics/

# Expected (vulnerable) response: HTTP 200 with Prometheus text format body, e.g.:
# # HELP api_all_request_total Total number of requests received
# # TYPE api_all_request_total counter
# api_all_request_total 42
# # HELP api_request_total Total number of requests per route
# api_request_total{method="GET",path="/api/v1/transactions",code="200"} 17
# ...

# No Authorization header was sent; no 401 was returned.
```
