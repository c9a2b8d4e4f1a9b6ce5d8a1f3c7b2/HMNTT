### Title
Unauthenticated Timestamp Parameter Triggers Expensive UNION with entity_history Table in getContractById, Enabling DoS

### Summary
The `getContractById` handler in `rest/controllers/contractController.js` accepts a user-supplied `timestamp` query parameter that, when present, unconditionally triggers a UNION query against the `entity_history` table — a potentially large historical table. Because the endpoint is unauthenticated and no rate limiting is enforced at the handler level, any external user can repeatedly issue historical timestamp lookups to amplify database load and degrade service availability.

### Finding Description

**Code path:**

`rest/routes/contractRoute.js` line 13 registers the public route: [1](#0-0) 

`getContractById` at line 707 extracts filters from the raw query string and passes them directly into the query builder: [2](#0-1) 

`getContractByIdOrAddressContractEntityQuery` at line 194 branches on `timestampConditions.length !== 0` — if any timestamp filter is present, it appends a full UNION with `Entity.historyTableName` (`entity_history`): [3](#0-2) 

The resulting SQL is:
```sql
SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.id = $3
UNION
SELECT ... FROM entity_history e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.id = $3
ORDER BY timestamp_range DESC
LIMIT 1
```

**Root cause:** The branch condition at line 194 is solely `timestampConditions.length !== 0` — there is no authentication check, no privilege check, and no rate limit before the more expensive UNION path is taken.

**Why existing checks fail:** `utils.buildAndValidateFilters` validates the *format* of the timestamp value (e.g., `lte:1234567890.000000000`) but does not restrict *who* may supply it. The `extractTimestampRangeConditionFilters` call at line 717–718 simply converts valid timestamp filters into SQL conditions; it has no access-control gate. [4](#0-3) 

### Impact Explanation
The `entity_history` table accumulates every historical state change for every entity on the network and can grow to hundreds of millions of rows in a mature deployment. Even with an index on `(id, timestamp_range)`, the UNION forces the planner to probe both `entity` and `entity_history`, merge the result sets, sort, and apply `LIMIT 1`. This is measurably more expensive than the single-table path. An attacker issuing a high volume of concurrent requests of the form `GET /api/v1/contracts/0.0.1?timestamp=lte:1` can saturate database I/O and connection pool resources, causing latency spikes or outright unavailability for all API consumers.

### Likelihood Explanation
The endpoint is fully public — no API key, no authentication token, no proof-of-work. The attacker needs only a valid contract ID (trivially obtained from the same API) and any syntactically valid timestamp value. The attack is trivially scriptable with standard HTTP tooling (`curl`, `ab`, `wrk`) and is repeatable indefinitely. No insider knowledge or special privilege is required.

### Recommendation
1. **Rate-limit the historical path**: Apply a per-IP (or global) rate limit specifically on requests that include a `timestamp` query parameter for this endpoint, separate from the non-historical limit.
2. **Add query timeout**: Enforce a short statement-level timeout (e.g., `SET LOCAL statement_timeout = '2s'`) for historical UNION queries so a slow query cannot hold a connection indefinitely.
3. **Index review**: Ensure `entity_history` has a composite index on `(id, timestamp_range)` so the inner scan is an index seek, not a sequential scan.
4. **Consider authentication**: Require an API key or token for any historical (`timestamp`-parameterized) lookup, reserving the more expensive path for authenticated clients.

### Proof of Concept
```bash
# Step 1: Obtain any valid contract ID (public, no auth needed)
CONTRACT_ID="0.0.1234"

# Step 2: Issue repeated historical lookups with an arbitrary timestamp
# The timestamp value does not need to match any real block; any valid format triggers the UNION
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/${CONTRACT_ID}?timestamp=lte:1.000000000" &
done
wait

# Result: 500 concurrent requests each trigger the entity_history UNION query,
# amplifying database load compared to the non-timestamp path.
# Monitor pg_stat_activity or slow query log to confirm entity_history scans.
```
