### Title
Unbounded gRPC Subscriptions Exhaust `boundedElastic()` Scheduler in `PollingTopicListener`, Causing Poll Starvation DoS

### Summary
`PollingTopicListener.listen()` creates a per-subscription `PollingContext` and schedules all polling work on a single shared `Schedulers.boundedElastic()` instance with no global cap on concurrent subscriptions. Because `maxConcurrentCallsPerConnection` only limits calls per TCP connection and no global connection limit is enforced, an unauthenticated attacker can open an unbounded number of connections and flood the shared scheduler, causing legitimate poll() calls to queue indefinitely and halting topic message delivery for all subscribers.

### Finding Description

**Exact code path:**

`PollingTopicListener.java`, `listen()`, lines 34–49:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();   // line 31 – one shared instance

public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);            // new context per subscription
    Duration interval = listenerProperties.getInterval();

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)                // uses shared scheduler
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))                     // uses shared scheduler
            ...
}
```

`GrpcConfiguration.java`, lines 27–35 — the only server-side limit configured:
```java
serverBuilder.maxConcurrentCallsPerConnection(
    nettyProperties.getMaxConcurrentCallsPerConnection());  // default = 5, per-connection only
```

`TopicMessageServiceImpl.java`, lines 48–55 — subscriber counter is metrics-only, never enforced:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// used only in a Gauge, never checked against a ceiling
```

**Root cause:** `Schedulers.boundedElastic()` caps threads at `10 × CPU cores` (e.g., 40 on a 4-core host) with a task queue of 100,000 entries. Every active subscription submits a `poll()` task to this shared pool every 500 ms (default interval). When the number of concurrent subscriptions exceeds the thread cap, tasks queue; when the queue fills, `RejectedExecutionException` is thrown and poll cycles are dropped entirely.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but there is no limit on the number of TCP connections a single IP or the global server can accept.
- `subscriberCount` is a Micrometer `Gauge` wired for observability only — it is never compared against a ceiling before accepting a new subscription.
- No authentication or authorization is required to call `ConsensusController.subscribeTopic()`.

### Impact Explanation

When the `boundedElastic()` thread pool and its 100,000-task queue are saturated, every subsequent `poll()` invocation is either delayed (queued) or rejected. This means no new `TopicMessage` items are emitted to any subscriber — including legitimate clients monitoring HCS topics for transaction confirmation. The mirror node's gRPC service becomes effectively non-functional for topic streaming. Note: this affects the mirror node service layer; Hedera consensus nodes themselves are independent and continue processing transactions, but clients that depend on the mirror node for topic-message confirmation lose visibility entirely.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no privileged network position. A single attacker machine can open thousands of TCP connections to port 5600 and issue `subscribeTopic` RPCs (up to 5 per connection per the default). On a 4-core deployment, ~8 connections (40 subscriptions) begin saturating the thread pool; ~20,000 connections fill the task queue. This is trivially achievable with tools like `ghz` or a simple gRPC client loop. The attack is repeatable and persistent as long as connections are held open (subscriptions have no server-side timeout when no `endTime` is set).

### Recommendation

1. **Enforce a global subscription ceiling** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a global connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a total-connection counter or a Netty `ChannelGroup` with a size cap.
3. **Isolate the scheduler per subscription or use a dedicated fixed-size pool** so that one flood cannot starve all other subscribers.
4. **Add per-IP rate limiting** at the gRPC interceptor layer (a `ServerInterceptor`) to cap new subscription attempts per source address per second.
5. **Set `maxConnectionIdle` and `maxConnectionAge`** on the Netty server builder to reclaim idle attacker connections automatically.

### Proof of Concept

**Preconditions:** Mirror node running with `hiero.mirror.grpc.listener.type=POLL` (or any deployment where `PollingTopicListener` is the active bean). No authentication configured (default).

**Steps:**

```bash
# 1. Install ghz (https://ghz.sh) or use any gRPC load tool
# 2. Flood with concurrent subscriptions (no limit enforced):
ghz --insecure \
    --proto consensus_service.proto \
    --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    --data '{"topicID":{"topicNum":1}}' \
    --concurrency 500 \
    --total 500 \
    --timeout 300s \
    <mirror-node-host>:5600

# 3. Each of the 500 concurrent streams calls listen() → new PollingContext
#    → schedules poll() on the shared boundedElastic() every 500ms.
# 4. On a 4-core host (40 boundedElastic threads), the pool saturates immediately.
# 5. Legitimate subscriber (separate terminal):
ghz --insecure \
    --proto consensus_service.proto \
    --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    --data '{"topicID":{"topicNum":2}}' \
    --concurrency 1 --total 1 --timeout 30s \
    <mirror-node-host>:5600
# → Receives no messages / extreme latency because poll() tasks are queued
#   behind the attacker's 500 subscriptions.
```

**Expected result:** Legitimate subscriber receives zero messages or experiences multi-second delays per poll cycle. Server logs show `RejectedExecutionException` from the `boundedElastic()` scheduler once the 100,000-task queue is also exhausted.
