### Title
Unauthenticated DB CPU Exhaustion via Unbounded `SUM` in `getBalanceChange` / `balanceChangeBetween`

### Summary
The `getBalanceChange` function in `rosetta/app/persistence/account.go` executes an unbounded `SELECT SUM(amount) FROM crypto_transfer` query with no row limit, scoped only by `entity_id` and a timestamp range. Any unauthenticated caller can trigger this query by hitting the public `/account/balance` Rosetta endpoint with the ID of a high-volume account, causing the database to scan and aggregate millions of rows per request. Flooding this endpoint from multiple source IPs exhausts database CPU, degrading all Rosetta instances that share the same PostgreSQL backend.

### Finding Description
**Exact code path:**

`rosetta/app/persistence/account.go`, lines 21–29 — the `balanceChangeBetween` constant:
```sql
select coalesce((
  select sum(amount) from crypto_transfer
  where
    consensus_timestamp > @start and
    consensus_timestamp <= @end and
    entity_id = @account_id and
    (errata is null or errata <> 'DELETE')
), 0) as value
```
This query is executed unconditionally in `getBalanceChange` (lines 313–338) with no `LIMIT`, no row-count guard, and no result caching. It is called from `RetrieveBalanceAtBlock` (lines 201–206), which is the handler for the public, unauthenticated Rosetta `/account/balance` endpoint.

**Root cause:** The query aggregates every `crypto_transfer` row for the target account between two timestamps. For high-volume accounts (exchanges, staking contracts, treasury-adjacent accounts on mainnet), this range can contain millions of rows. There is no application-level cap on how many rows the SUM may process.

**Exploit flow:**
1. Attacker identifies a high-volume `entity_id` (e.g., a known exchange account or staking contract with millions of `crypto_transfer` rows — publicly observable on-chain).
2. Attacker sends repeated `POST /account/balance` requests with that `account_identifier`, targeting the latest block so the timestamp window `[snapshotTimestamp, consensusEnd]` is maximally wide.
3. Each request causes `getBalanceChange` to issue the unbounded SUM query, holding a DB connection and consuming CPU for up to the full `statementTimeout` (default 20 s).
4. Attacker fans out across multiple source IPs to bypass the per-IP `inFlightReq` limit of 5.
5. With `maxOpenConnections: 100` (default pool), 100 concurrent 20-second queries saturate the DB CPU, starving all other queries.

**Why existing checks fail:**

- **`statementTimeout = 20 s`** (`rosetta/app/db/client.go`, line 36): Implemented as a Go `context.WithTimeout`, not a PostgreSQL `statement_timeout`. It cancels after 20 s, but 100 concurrent connections each burning 20 s of CPU is a sustained 100× CPU multiplier — sufficient to exhaust a shared DB.
- **Index `crypto_transfer__entity_id_consensus_timestamp`** (`V2.0.3__index_init.sql`, lines 90–92): Exists and is used, but an index scan over millions of matching rows still requires reading and summing every qualifying entry. The index reduces I/O but does not bound CPU for the aggregation.
- **Traefik rate limiting** (`charts/hedera-mirror-rosetta/values.yaml`, lines 157–160): `average: 10` is keyed on `requestHost` (the HTTP `Host` header), trivially bypassed by rotating the `Host` value. The `inFlightReq: 5` per IP uses `X-Forwarded-For` depth 1, spoofable behind most proxies. Both controls are also optional Helm chart middleware, absent in bare deployments.
- **No application-level caching or query result reuse** for balance computations.

### Impact Explanation
An attacker can sustain DB CPU at or near 100% on the shared PostgreSQL instance by maintaining a pool of concurrent balance requests against one or more high-volume accounts. All Rosetta API instances sharing that database become unresponsive or return errors. Because the mirror node's Rosetta service is the primary interface for exchange integrations and block explorers relying on the Coinbase Rosetta standard, this constitutes a denial-of-service against the mirror node infrastructure. Multiple mirror node operators running Rosetta against the same DB cluster would all be simultaneously degraded.

### Likelihood Explanation
The attack requires zero authentication, zero on-chain funds, and zero privileged access. High-volume account IDs are publicly known (e.g., major exchange deposit accounts, staking reward distributors). The attacker needs only an HTTP client and knowledge of one such account ID. The attack is repeatable, automatable, and can be sustained indefinitely. The only friction is the Traefik middleware, which is bypassable as described above.

### Recommendation
1. **Add a PostgreSQL-level `statement_timeout`** via the DB role used by Rosetta (e.g., `ALTER ROLE mirror_rosetta SET statement_timeout = '20s'`), so the limit is enforced server-side regardless of client behavior.
2. **Cache balance query results** per `(account_id, consensusEnd)` with a short TTL (e.g., 5–15 s) to collapse repeated identical queries into a single DB round-trip.
3. **Add application-level concurrency limiting** per unique `account_id` parameter (e.g., a singleflight or semaphore keyed on account ID) so that N concurrent requests for the same account result in one DB query, not N.
4. **Key Traefik rate limiting on a verified client attribute** (e.g., source IP after stripping untrusted `X-Forwarded-For` headers) rather than the spoofable `Host` header.
5. **Consider a `LIMIT` guard or pre-check**: if the estimated row count for the SUM exceeds a threshold (queryable via `pg_stats`), return a cached or approximate value rather than executing the full aggregation.

### Proof of Concept
```bash
# 1. Identify a high-volume account (e.g., account 0.0.98 is excluded from the index,
#    but exchange accounts with millions of transfers are not).
TARGET_ACCOUNT="0.0.XXXXXX"   # replace with known high-volume account
ROSETTA_URL="https://<rosetta-host>"

# 2. Get current block index
BLOCK=$(curl -s -X POST $ROSETTA_URL/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq '.current_block_identifier.index')

# 3. Fan out 50 concurrent balance requests from multiple IPs / Host headers
for i in $(seq 1 50); do
  curl -s -X POST $ROSETTA_URL/account/balance \
    -H "Content-Type: application/json" \
    -H "Host: host-$i.example.com" \
    -d "{
      \"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\":{\"address\":\"$TARGET_ACCOUNT\"},
      \"block_identifier\":{\"index\":$BLOCK}
    }" &
done
wait

# Each request triggers getBalanceChange -> balanceChangeBetween SUM query.
# With 50 concurrent requests each holding a DB connection for up to 20 s,
# DB CPU is saturated; subsequent legitimate requests time out or error.
```
