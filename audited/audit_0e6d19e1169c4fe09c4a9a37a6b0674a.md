### Title
Thundering Herd via Non-Atomic `@Cacheable` on 1s-TTL Entity Cache Exhausts HikariCP Connection Pool During DB Partition

### Summary
`EntityRepository.findByIdAndDeletedIsFalse()` is annotated with `@Cacheable` (without `sync=true`) backed by a Caffeine cache with a 1-second `expireAfterWrite` TTL. Spring's default `@Cacheable` uses a non-atomic check-then-act pattern, so when the cache entry expires, all concurrent threads simultaneously observe a miss and each independently issues a DB query. During a network partition, those queries block until HikariCP's connection timeout, exhausting the pool and causing cascading request failures that persist after the partition heals.

### Finding Description

**Code path:**

`EntityRepository.java` lines 20–30 — `findByIdAndDeletedIsFalse` is annotated with `@Caching` containing two `@Cacheable` entries, neither specifying `sync=true`: [1](#0-0) 

The entity cache manager is built with `CaffeineCacheManager.setCacheSpecification()` using a synchronous Caffeine `Cache` (not `AsyncCache`): [2](#0-1) 

The TTL is hardcoded to 1 second: [3](#0-2) 

**Root cause — failed assumption:**
Spring's `@Cacheable` without `sync=true` calls `nativeCache.getIfPresent(key)` and, on a miss, invokes the underlying method, then calls `nativeCache.put(key, value)`. These are three separate, non-atomic operations. Caffeine's atomic `cache.get(key, loader)` (which serialises concurrent loaders per key) is only used when `sync=true` is set. Because `sync=true` is absent from every `@Cacheable` in the web3 module (confirmed: zero matches for `sync=true` in `web3/**/*.java`), N concurrent threads that all observe a miss will all invoke the JPA query simultaneously.

**Exploit flow:**
1. Attacker sends a sustained stream of concurrent `eth_call` / `eth_estimateGas` requests referencing the same (or many) entity IDs.
2. A network partition isolates the web3 pod from the PostgreSQL replica.
3. Every 1 second the Caffeine TTL fires; all in-flight threads simultaneously miss the cache.
4. Each thread attempts `SELECT … FROM entity WHERE id = ? AND deleted IS NOT TRUE` via HikariCP.
5. HikariCP's default `connectionTimeout` is 30 s; `statementTimeout` for web3 is 3 s (`hiero.mirror.web3.db.statementTimeout=3000`). All pool connections are held for up to 3 s per cycle. [4](#0-3) 
6. With the default HikariCP pool size (10 connections), the pool exhausts within the first burst; subsequent requests queue and time out.
7. After the partition heals, the backlog of queued threads all fire DB queries simultaneously, creating a second thundering herd that can re-exhaust the pool.

**Why existing checks are insufficient:**
- No `sync=true` on any `@Cacheable` in the web3 module.
- No application-level circuit breaker (`@CircuitBreaker`, Resilience4j, etc.) wraps the entity repository — confirmed zero matches for `CircuitBreaker|Resilience4j|@Retry` in `web3/**/*.java`.
- The Traefik circuit breaker configured in the Helm chart (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) operates at the HTTP ingress layer and only trips after a significant fraction of requests have already failed — it does not prevent pool exhaustion from occurring. [5](#0-4) 

### Impact Explanation
Complete connection pool exhaustion causes all web3 API requests to fail with `HikariPool-1 - Connection is not available, request timed out`. Because the 1s TTL re-triggers the herd every second during the partition, the pool never recovers on its own. After the partition heals, the queued-request backlog creates a second surge. Effective result: full web3 API denial-of-service for the duration of the partition plus a recovery tail, affecting all users of `eth_call`, `eth_estimateGas`, and any EVM simulation endpoint that resolves entity IDs.

### Likelihood Explanation
No privilege is required. Any user with network access to the web3 API endpoint can sustain the amplification by sending concurrent requests. The partition itself is an infrastructure event (not attacker-controlled), but brief, transient partitions (e.g., rolling restarts, cloud network blips, pgBouncer restarts) occur regularly in production. The 1s TTL makes the window of vulnerability repeat every second, so even a 5-second partition produces five thundering-herd cycles. The attack is repeatable and requires no authentication.

### Recommendation
1. **Add `sync=true`** to both `@Cacheable` entries on `findByIdAndDeletedIsFalse` (and the other cached methods in `EntityRepository`). This causes Spring to delegate to Caffeine's atomic `cache.get(key, loader)`, which serialises concurrent loaders per key and prevents the thundering herd entirely.
2. **Add an application-level circuit breaker** (e.g., Resilience4j `@CircuitBreaker`) around the entity repository to fast-fail DB calls when the DB is unreachable, preventing pool exhaustion.
3. **Tune HikariCP** `connectionTimeout` and `maximumPoolSize` for the web3 datasource to bound the blast radius.
4. Consider increasing the entity cache TTL or using a stale-while-revalidate strategy to reduce the frequency of cache misses under normal load.
