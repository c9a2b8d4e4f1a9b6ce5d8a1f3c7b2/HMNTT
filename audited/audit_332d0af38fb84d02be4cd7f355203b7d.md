### Title
Connection Pool Exhaustion via Unauthenticated Concurrent EVM-Address Lookups During Network Partition

### Summary
The `findByEvmAddress()` method in `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java` is invoked on every unauthenticated EVM-address lookup request with no caching, no rate limiting, and no socket-level timeout on runtime JDBC connections. During a network partition between the application and the database, each in-flight JDBC connection blocks indefinitely at the OS socket layer, exhausting the HikariCP pool and preventing all subsequent DB-dependent operations from executing.

### Finding Description
**Exact code path:**

`EntityIdEvmAddressParameter` → `EntityServiceImpl.lookup()` (line 34) → `EntityRepository.findByEvmAddress()` (line 16–17):

```java
// EntityRepository.java, line 16-17
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
```

```java
// EntityServiceImpl.java, line 34
case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
```

**Root cause — failed assumption:** The system assumes that `hiero.mirror.restJava.db.statementTimeout` (default 10,000 ms) will bound query execution time. This assumption is false during a network partition. `statement_timeout` is a PostgreSQL server-side session parameter; the server sends a cancellation signal over the same TCP connection. When the network is partitioned, the TCP socket is blocked at the OS level — the server's cancellation signal never arrives, and the JDBC thread blocks indefinitely on `socket.read()`.

The `socketTimeout` values in `CommonProperties.DatabaseStartupProperties` (2 s connect/socket timeout) are applied **only** during startup in `DatabaseWaiter.waitForDatabase()`, not to the runtime HikariCP pool. The `CommonConfiguration` builds the HikariCP `DataSource` from `spring.datasource.hikari` properties, and no `dataSourceProperties.socketTimeout` is set for runtime connections anywhere in the rest-java module. HikariCP's own `connectionTimeout` (default 30 s) governs only the time to *acquire* a connection from the pool — once acquired and executing, HikariCP imposes no query-execution deadline.

Unlike the `web3` module's `EntityRepository`, which wraps `findByEvmAddressAndDeletedIsFalse` with `@Cacheable`, the rest-java `EntityRepository.findByEvmAddress` has **no caching annotation**, so every distinct (and even repeated) EVM address request hits the database.

### Impact Explanation
With HikariCP's default pool size of 10 (no explicit `maximumPoolSize` configured for rest-java), an attacker needs only 10 concurrent hanging requests to exhaust the pool. All subsequent requests that require a DB connection — including unrelated endpoints — will block for HikariCP's `connectionTimeout` (default 30 s) and then fail with a pool-acquisition timeout. This constitutes a complete denial of service for all database-backed functionality in the rest-java service, with no authentication required to trigger it.

### Likelihood Explanation
The EVM-address lookup endpoint is publicly accessible with no authentication or rate limiting. A network partition can be induced by an attacker with network-path access (BGP manipulation, firewall rule injection, cloud security-group misconfiguration) or can occur naturally. The attacker needs only to send 10–20 concurrent HTTP requests to an EVM-address endpoint (e.g., `/api/v1/accounts/0x<address>`) immediately before or during the partition. This is trivially scriptable with `curl`, `ab`, or any HTTP load tool. The attack is repeatable: once the partition resolves and connections are returned to the pool, a new burst restores the DoS.

### Recommendation
1. **Set a JDBC socket timeout on runtime connections** via HikariCP's data-source properties for the rest-java module:
   ```yaml
   spring.datasource.hikari.data-source-properties.socketTimeout: 30
   ```
   This causes the JDBC driver to close the socket after 30 s of no data, returning the connection to the pool.

2. **Add application-level query timeout** via Spring's `@QueryHints` or JPA `javax.persistence.query.timeout` on `findByEvmAddress`.

3. **Add caching** to `findByEvmAddress` in the rest-java `EntityRepository`, mirroring the `@Cacheable` pattern used in the web3 module, to reduce DB round-trips for repeated lookups.

4. **Implement rate limiting** on the EVM-address lookup endpoint to bound the number of concurrent in-flight DB connections attributable to a single client.

        config.setUsername(username);
