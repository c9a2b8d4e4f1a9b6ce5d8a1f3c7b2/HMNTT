### Title
Unauthenticated Timestamp Boundary Bypass Causes Forced `EntityNotFoundException` on `/api/v1/network/supply`

### Summary
An unprivileged external user can supply `timestamp=lt:0.000000001` to the `/api/v1/network/supply` endpoint, causing `adjustUpperBound()` to return `0` and `getAdjustedLowerRangeValue()` to also return `0`. The degenerate range `[0, 0]` passes the existing range check, and the resulting database query returns no rows, triggering an `EntityNotFoundException` (HTTP 404) regardless of actual data availability.

### Finding Description

**Exact code path:**

1. **Input parsing** — `TimestampParameter.valueOf("lt:0.000000001")` matches the regex `^((eq|gt|gte|lt|lte|ne):)?(\d{1,17})(\.\d{1,9})?$` with seconds=`0`, nanos=`1`. `DomainUtils.convertToNanos(0, 1)` returns `1L` (one nanosecond). The resulting `TimestampParameter` has operator=`LT`, value=`1`. [1](#0-0) [2](#0-1) 

2. **`adjustUpperBound()` returns 0** — With `upper.operator() == LT` and `upper.value() == 1`, the method computes `upperBound = 1 - 1 = 0`. [3](#0-2) 

3. **`getAdjustedLowerRangeValue()` returns 0** — No lower bound is provided (`lower == null`), so the method returns `0`. [4](#0-3) 

4. **Range check bypassed** — In `getSupply`, `minTimestamp=0` and `maxTimestamp=0`. The guard `if (minTimestamp > maxTimestamp)` evaluates `0 > 0 = false`, so no exception is thrown. [5](#0-4) 

5. **`getFirstDayOfMonth(0, -1)` produces a negative epoch** — `Instant.ofEpochSecond(0, 0)` is the Unix epoch (1970-01-01). `plusMonths(-1)` yields 1969-12-01, whose `toEpochSecond()` is negative. Multiplied by `NANOS_PER_SECOND`, the result is a large negative long. `Math.max(0, negative) = 0`, so `minTimestamp` stays `0`. [6](#0-5) [7](#0-6) 

6. **Database query returns `consensusTimestamp = 0`** — `getSupplyHistory(..., 0, 0)` executes `WHERE ab.consensus_timestamp BETWEEN 0 AND 0`. No real Hedera balance record has `consensus_timestamp = 0`, so `coalesce(max(consensus_timestamp), 0)` returns `0`. [8](#0-7) 

7. **`EntityNotFoundException` thrown** — The sentinel check `if (networkSupply.consensusTimestamp() == 0L)` fires, throwing `EntityNotFoundException("Network supply not found")`. [9](#0-8) 

**Root cause:** There is no minimum-value validation on the parsed timestamp. The range check `minTimestamp > maxTimestamp` is insufficient because it does not reject the degenerate case where both bounds collapse to `0`.

### Impact Explanation
Any unauthenticated caller can force the `/api/v1/network/supply` endpoint to return HTTP 404 on every request that includes `timestamp=lt:0.000000001`, regardless of whether valid supply data exists in the database. This constitutes a targeted denial-of-service against the supply endpoint. Clients or downstream systems that depend on this endpoint for circulating/total supply data will receive persistent errors. The impact is limited to availability of this specific endpoint; no data is leaked or modified.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single crafted HTTP GET request. The parameter format is publicly documented and the regex accepts `0.000000001` as a valid timestamp. The attack is trivially repeatable and scriptable. Any external actor who reads the API documentation or inspects a single valid request can reproduce it immediately.

### Recommendation
1. **Add a minimum timestamp guard** in `getSupply` (or in `TimestampParameter.valueOf`) rejecting any timestamp value ≤ 0 (or below the Hedera mainnet genesis timestamp):
   ```java
   if (maxTimestamp <= 0) {
       throw new IllegalArgumentException("Timestamp must be greater than zero");
   }
   ```
2. **Reject degenerate ranges early** — after computing `minTimestamp` and `maxTimestamp`, also check `if (maxTimestamp == 0)` before proceeding to the database call.
3. **Align with the JS REST layer** — the Node.js implementation in `balances.js` caps the effective upper bound at `now` before computing the optimal lower bound (`effectiveUpperBound = min(upperBound, nowInNs)`). The Java implementation should apply the same guard. [10](#0-9) 

### Proof of Concept

```
# Single unauthenticated HTTP request
GET /api/v1/network/supply?timestamp=lt:0.000000001 HTTP/1.1
Host: <mirror-node-host>

# Expected (correct) response: 200 OK with supply data
# Actual response: 404 Not Found
# {"_status":{"messages":[{"message":"Network supply not found"}]}}
```

Step-by-step:
1. Send the request above with no authentication headers.
2. `TimestampParameter.valueOf("lt:0.000000001")` → operator=LT, value=1 (nanosecond).
3. `adjustUpperBound()` → `1 - 1 = 0`.
4. `getAdjustedLowerRangeValue()` → `0` (no lower bound).
5. Range check `0 > 0` → false, passes.
6. `getFirstDayOfMonth(0, -1)` → negative epoch; `Math.max(0, negative) = 0`.
7. `getSupplyHistory(..., 0, 0)` → `consensusTimestamp = 0`.
8. `EntityNotFoundException` thrown → HTTP 404.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    private long getFirstDayOfMonth(long timestamp, int monthOffset) {
        final var instant = Instant.ofEpochSecond(0, timestamp);
        final var dateTime = instant.atZone(ZoneOffset.UTC);
        final var firstDay = dateTime.plusMonths(monthOffset).withDayOfMonth(1);

        return firstDay.toLocalDate().atStartOfDay(ZoneOffset.UTC).toEpochSecond() * DomainUtils.NANOS_PER_SECOND;
    }
