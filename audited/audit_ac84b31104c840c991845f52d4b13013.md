### Title
Unbounded `X-Forwarded-For` Header Logged Verbatim Enables Log-Based Disk Exhaustion DoS

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the raw `X-Forwarded-For` (or `X-Real-IP`) header value with no length validation or truncation, then `TracingMiddleware` embeds the full string into every log entry at `Info` level. Because the HTTP server sets no `MaxHeaderBytes`, Go's default 1 MiB header limit applies, meaning each unauthenticated request can inject up to ~1 MiB of attacker-controlled data into the log, exhausting disk space under sustained load.

### Finding Description
**Code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // raw, unbounded
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // raw, unbounded
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress
}
```

`r.Header.Get()` returns the complete, unsanitised header value. No maximum length is enforced, no IP-format validation is performed, and no truncation occurs.

The returned string is immediately spliced into the log message in `TracingMiddleware` (lines 52–59):
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, ...)   // attacker-controlled, up to ~1 MiB
...
log.Info(message)           // written for every non-internal path
```

**Root cause:** The code assumes the header contains a short, well-formed IP address. It makes no attempt to validate format, cap length, or take only the first token.

**Why existing checks fail:**
- `rosetta/main.go` lines 220–227 configure the `http.Server` with timeouts but **no `MaxHeaderBytes`**, so Go's default of `1 << 20` (1,048,576 bytes) applies — still large enough to be abused.
- The `internalPaths` guard (line 55) only downgrades logging to `Debug` for `/health`, `/metrics`, and `/readiness`; all API endpoints (e.g. `/network/list`, `/block`) log at `Info`, which is always enabled in production.
- There is no rate-limiting middleware visible anywhere in the middleware chain (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`).

### Impact Explanation
An attacker who can reach the Rosetta HTTP port (default behaviour — no authentication on the API) can write up to ~1 MiB per request into the log. At modest request rates (e.g. 1,000 req/s × 900 KiB = ~900 MiB/s of log output), disk space is exhausted rapidly, crashing the logging subsystem and potentially the entire Rosetta process. This constitutes a non-network-based DoS against a node that may represent a significant share of Hedera network infrastructure.

### Likelihood Explanation
- **No authentication required.** Any external client can send an HTTP POST to any Rosetta endpoint.
- **Trivially scriptable.** A single `curl` loop or `ab`/`wrk` invocation with a custom header suffices.
- **Repeatable and persistent.** The attacker does not need to maintain a connection; fire-and-forget requests are enough.
- **No special knowledge needed.** The header name (`X-Forwarded-For`) is a standard HTTP header.

### Recommendation
1. **Truncate or validate in `getClientIpAddress()`** — take only the first comma-separated token and validate it as an IP address:
   ```go
   raw := r.Header.Get(xForwardedForHeader)
   first := strings.SplitN(raw, ",", 2)[0]
   ip := strings.TrimSpace(first)
   if net.ParseIP(ip) == nil {
       ip = ""
   }
   ```
2. **Cap log field length** — if the value is used for anything other than a validated IP, enforce a hard maximum (e.g. 45 characters for IPv6).
3. **Set `MaxHeaderBytes`** on the `http.Server` in `rosetta/main.go` to a small value (e.g. 8 KiB) appropriate for the Rosetta API's actual header needs.
4. **Add rate limiting** middleware to bound requests per source IP.

### Proof of Concept
```bash
# Generate ~900 KiB of fake IPs
PAYLOAD=$(python3 -c "print(','.join(['1.2.3.4']*130000))")

# Fire requests in a loop — each one writes ~900 KiB to the log
for i in $(seq 1 1000); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: $PAYLOAD" \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
    http://<rosetta-host>:5700/network/list &
done
wait
# Observe disk usage spike; log file grows by ~900 MiB per 1000 requests
```

Each request causes `TracingMiddleware` to call `log.Info()` with the full ~900 KiB string, writing it to stdout/log aggregator. Under sustained load this exhausts available disk or log-pipeline capacity. [1](#0-0) [2](#0-1) [3](#0-2)
