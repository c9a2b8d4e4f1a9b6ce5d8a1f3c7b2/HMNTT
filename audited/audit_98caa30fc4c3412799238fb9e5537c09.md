### Title
Integer Overflow in `poll()` Bypasses `setMaxResults`, Enabling Unbounded In-Memory Load via `getResultList()`

### Summary
In `PollingTopicMessageRetriever.poll()`, the user-supplied `limit` (a `long`) is narrowed to `int` via an unchecked cast. When a user supplies a `limit` value greater than `Integer.MAX_VALUE`, the cast overflows to a negative number, which causes the internally-built `newFilter` to report `hasLimit() == false`. This causes `TopicMessageRepositoryCustomImpl.findByFilter()` to skip `setMaxResults` entirely and call `typedQuery.getResultList()` with no row cap, loading the entire topic's message history into JVM heap at once.

### Finding Description

**Code path:**

`PollingTopicMessageRetriever.poll()` — lines 68–75:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())  // ← unchecked long→int cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());       // ← -1 wins Math.min
var newFilter = filter.toBuilder().limit(pageSize)...build();   // ← limit = -1
``` [1](#0-0) 

`TopicMessageRepositoryCustomImpl.findByFilter()` — lines 51–60:
```java
if (filter.hasLimit()) {           // hasLimit() = (limit > 0) = (-1 > 0) = FALSE
    typedQuery.setMaxResults(...);  // ← SKIPPED
}
return typedQuery.getResultList().stream(); // ← returns ALL rows, fully in-memory
``` [2](#0-1) 

**Root cause:** `filter.getLimit()` is a `long`; the cast `(int)(filter.getLimit() - context.getTotal().get())` silently overflows when `filter.getLimit() > Integer.MAX_VALUE`. The result is a negative `pageSize`, which propagates into `newFilter.limit(-1)`. Since `hasLimit()` is defined as `limit > 0`, a limit of `-1` is treated as "no limit," bypassing `setMaxResults` entirely. [3](#0-2) 

**Failed assumption:** The design assumes `pageSize` will always be a positive value bounded by `maxPageSize`. The unchecked narrowing cast breaks this invariant for any user-supplied `limit > Integer.MAX_VALUE`.

### Impact Explanation
`getResultList()` materialises the full JDBC result set into a `List<TopicMessage>` in JVM heap before returning the `Stream`. With no `setMaxResults` guard, a topic with millions of messages causes a single poll to allocate all of them simultaneously. Multiple concurrent subscriptions with this payload can exhaust heap and crash the gRPC service (OOM / `OutOfMemoryError`), causing a denial-of-service for all subscribers. The claim about "fee-data processing" does not apply to this mirror-node component; the actual impact is gRPC service availability. [4](#0-3) 

### Likelihood Explanation
Any unauthenticated gRPC client can call `subscribeTopic` with an arbitrary `limit` field (proto `uint64`). Setting `limit = 2147483648` (i.e., `Integer.MAX_VALUE + 1`) is sufficient to trigger the overflow. No special privileges, credentials, or knowledge of internal state are required. The attack is trivially repeatable and can be parallelised across many connections. The default `maxConcurrentCallsPerConnection` is 5, but nothing prevents opening many connections. [5](#0-4) 

### Recommendation
1. **Remove the unchecked cast.** Compute `limit` as a `long` throughout and clamp to `maxPageSize` before converting:
   ```java
   long remaining = filter.hasLimit()
       ? filter.getLimit() - context.getTotal().get()
       : (long) context.getMaxPageSize();
   int pageSize = (int) Math.min(remaining, context.getMaxPageSize());
   ```
2. **Add `@Max` to `TopicMessageFilter.limit`** to reject values that would overflow `int` at the validation layer, e.g. `@Max(Integer.MAX_VALUE)` or a domain-appropriate cap. [6](#0-5) 
3. **Assert `pageSize > 0`** before building `newFilter` as a defensive guard.

### Proof of Concept
**Preconditions:** A topic exists in the mirror node database with a large number of messages (e.g., ≥ 100 000).

**Steps:**
1. Open a gRPC connection to the mirror node's port 5600.
2. Call `ConsensusService/subscribeTopic` with:
   - `topicID` = the target topic
   - `consensusStartTime` = 0
   - `limit` = `2147483648` (= `Integer.MAX_VALUE + 1`, fits in proto `uint64`)
3. The value is stored as a positive `long` in `TopicMessageFilter.limit`; `@Min(0)` passes.
4. In `poll()`, `(int)(2147483648L - 0)` = `-2147483648` (overflow); `Math.min(-2147483648, 1000)` = `-2147483648`; `newFilter.limit(-2147483648)`.
5. `newFilter.hasLimit()` = false → `setMaxResults` skipped → `getResultList()` fetches all rows.
6. JVM heap fills with all `TopicMessage` objects; repeated or concurrent calls trigger `OutOfMemoryError`.
