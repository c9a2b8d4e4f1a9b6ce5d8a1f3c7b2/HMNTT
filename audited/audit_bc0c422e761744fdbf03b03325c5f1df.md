### Title
Missing `deleted` Filter in `getByIdAndType()` Exposes Deleted Entities via Numeric ID Lookup

### Summary
`EntityServiceImpl.getByIdAndType()` uses the standard Spring Data `CrudRepository.findById()` which performs no `deleted` status check. Unlike the alias and evmAddress lookup paths — which both explicitly filter `deleted is not true` — the numeric ID path returns deleted entities to any unprivileged caller. No network partition is required; the flaw is a straightforward missing predicate.

### Finding Description
**Exact code path:**

`EntityServiceImpl.java` line 25: [1](#0-0) 

```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
}
```

`findById()` is the inherited `CrudRepository` method. It maps directly to `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

**Contrast with the other two lookup methods** in `EntityRepository.java`: [2](#0-1) 

Both `findByAlias` and `findByEvmAddress` carry `and deleted is not true`. `findById()` carries nothing.

**The `deleted` field exists on every entity:** [3](#0-2) 

When a `CryptoDelete` / `FileDelete` / etc. transaction is processed, `deleted` is set to `true`: [4](#0-3) 

**Entry point** — any unauthenticated caller can reach `getByIdAndType()` via the public GraphQL endpoint: [5](#0-4) 

**Root cause / failed assumption:** The developer assumed `findById()` would behave like the custom queries and suppress deleted rows. It does not. The `deleted` column is a soft-delete marker; `CrudRepository.findById()` is unaware of it.

### Impact Explanation
Any deleted account, contract, file, token, topic, or schedule can be retrieved in full (keys, memo, balances, timestamps, obtainer, etc.) by supplying its numeric entity ID to the GraphQL `account` (or equivalent) query. The `deleted: true` flag is present in the response payload but the record is still returned, violating the access-control contract enforced by every other lookup path. Severity: **Medium** — information disclosure of soft-deleted entity state to unauthenticated users.

### Likelihood Explanation
Entity IDs are sequential integers publicly visible in block explorers and transaction history. An attacker needs only to: (1) observe a deletion transaction on-chain, (2) note the entity's numeric ID, (3) issue a single unauthenticated GraphQL POST. No special privileges, timing windows, or infrastructure access are required. Fully repeatable and automatable.

### Recommendation
Add a custom `findById` override in `EntityRepository` that mirrors the existing pattern:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then update `EntityServiceImpl.getByIdAndType()`:

```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findByIdAndNotDeleted(entityId.getId())
                           .filter(e -> e.getType() == type);
}
```

Apply the same fix to the `getByEvmAddressAndType()` branch that falls through to `findById()`: [6](#0-5) 

### Proof of Concept
1. Identify a deleted account (e.g., entity `0.0.12345`) from any public Hedera explorer or by watching `CryptoDelete` transactions.
2. Send an unauthenticated GraphQL request to the mirror node:
```graphql
POST /graphql/alpha
{
  account(input: { entityId: { shard: 0, realm: 0, num: 12345 } }) {
    entityId { num }
    deleted
    key
    memo
    balance
  }
}
```
3. Observe a `200 OK` response containing the full entity record with `"deleted": true` — data that should not be returned.
