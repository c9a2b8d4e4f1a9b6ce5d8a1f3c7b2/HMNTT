### Title
Unbounded Prometheus Label Cardinality via Unmatched Route Paths in MetricsMiddleware

### Summary
`MetricsMiddleware()` in `rosetta/app/middleware/metrics.go` uses the `weaveworks/common/middleware.Instrument` struct, which falls back to using the raw `r.RequestURI` as the `route` label value when no registered route matches an incoming request. An unprivileged external attacker can send requests with arbitrarily unique paths, causing unbounded growth of Prometheus label combinations across all four metric vectors (`requestBytesHistogram`, `requestDurationHistogram`, `requestInflightGauge`, `responseBytesHistogram`), leading to sustained memory and CPU resource exhaustion exceeding 30% above baseline.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

All four metric vectors use `{"method", "route"}` (or `{"method", "route", "status_code", "ws"}`) as label dimensions (lines 26, 32, 37, 43).

**Root cause:**

The `weaveworks/common/middleware.Instrument` calls `RouteMatcher.MatchRoute(r)` to obtain the route template for labeling. When the gorilla/mux router (used by `coinbase/rosetta-sdk-go/server.NewRouter`) does not match the incoming path to any registered route, `MatchRoute` returns `("", false)`. The weaveworks middleware then falls back to using `r.RequestURI` verbatim as the `route` label value. Since `r.RequestURI` is fully attacker-controlled, each unique path string creates a new, permanent label combination in the Prometheus in-process registry.

**Exploit flow:**

1. Attacker sends `POST /aaa1 HTTP/1.1`, `POST /aaa2 HTTP/1.1`, ..., `POST /aaa-N HTTP/1.1` — each with a unique path.
2. None match any registered route; `MatchRoute` returns `false` for each.
3. Weaveworks middleware records metrics with `route="/aaa1"`, `route="/aaa2"`, etc.
4. For each unique `(method, route)` pair, Prometheus allocates:
   - `requestBytesHistogram`: 5 buckets + sum + count = **7 time series**
   - `requestDurationHistogram`: 6 buckets + sum + count + 2 extra label dims = **8+ time series**
   - `requestInflightGauge`: **1 time series**
   - `responseBytesHistogram`: **7 time series**
   - Total: ~23 time series per unique path
5. With 10,000 unique paths: ~230,000 new in-memory time series, causing significant heap growth and GC pressure.

**Why existing checks fail:**

- The Traefik-level rate limiting (`average: 10`, `inFlightReq: 5`) is at the Kubernetes ingress layer and is bypassed if the pod is directly reachable, or is slow enough (10 req/s sustained = 36,000 unique paths/hour) to still accumulate massive cardinality.
- There is no application-level rate limiting, path validation, or cardinality cap in `MetricsMiddleware`.
- The `MetricsMiddleware` is applied at the outermost layer (`main.go` line 217), before any routing or authentication, so every request — including 404s — is instrumented.
- Prometheus `NewHistogramVec` / `NewGaugeVec` have no built-in cardinality limits; all label combinations are stored indefinitely in the process heap.

### Impact Explanation

Each unique attacker-controlled path permanently allocates new Prometheus time series in the Go process heap. With histograms, each series includes bucket arrays. At scale (tens of thousands of unique paths), heap usage grows by hundreds of megabytes, triggering frequent GC cycles that consume CPU. This directly increases both memory and CPU resource consumption well above 30% of baseline. In extreme cases, the process OOMs. The Prometheus `/metrics` scrape endpoint also becomes slower as it must serialize all accumulated series, further amplifying CPU load.

### Likelihood Explanation

No authentication or privilege is required — any network-reachable client can send HTTP requests with arbitrary paths. The attack is trivially scriptable (a simple loop generating UUID paths). Even behind Traefik rate limiting at 10 req/s, an attacker accumulates 36,000 unique label combinations per hour. The attack is persistent: label combinations are never evicted from the Prometheus registry during the process lifetime. This makes it highly repeatable and low-cost for the attacker.

### Recommendation

1. **Normalize unmatched routes**: Override the fallback behavior so that when `MatchRoute` returns `false`, the `route` label is set to a fixed sentinel value (e.g., `"unmatched"` or `"unknown"`) instead of `r.RequestURI`.
2. **Implement a custom `RouteMatcher` wrapper** that intercepts the `MatchRoute` call and returns `("unmatched", true)` when the underlying router finds no match, preventing the weaveworks fallback from using the raw URI.
3. **Apply application-level rate limiting** before the metrics middleware to bound the rate of new label creation.
4. Consider replacing `weaveworks/common/middleware` with a metrics middleware that explicitly caps label cardinality or uses a fixed allowlist of route values.
