### Title
Integer Overflow in `getInclusiveValue()` Bypasses hookId Range Filter, Exposing All Hook Data

### Summary
`NumberRangeParameter.getInclusiveValue()` performs an unchecked `value + 1` for `GT` operators. When a user supplies `hook.id=gt:9223372036854775807` (Long.MAX_VALUE), the addition overflows to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` then resolves back to `0L`, silently resetting the lower bound to its default, causing `findByOwnerIdAndHookIdBetween` to return all hooks for the target account instead of none.

### Finding Description

**Root cause — `NumberRangeParameter.getInclusiveValue()` (line 36–37):**

```java
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        return value + 1;   // ← no overflow guard
    }
    ...
}
``` [1](#0-0) 

**Input validation — `getNumberValue()` (lines 26–33):**

```java
private static long getNumberValue(String number) {
    var value = Long.parseLong(number);
    if (value < 0) {          // ← only rejects negatives
        throw new IllegalArgumentException("Invalid range value");
    }
    return value;
}
```
`Long.MAX_VALUE` (9223372036854775807) is positive, so it passes this check without error. [2](#0-1) 

**Overflow propagation — `HooksController.hooksRequest()` (line 142):**

```java
lowerBound = Math.max(lowerBound, hookIdFilter.getInclusiveValue());
// Math.max(0L, Long.MIN_VALUE) == 0L  ← overflow silently discarded
``` [3](#0-2) 

**Downstream query — `HookServiceImpl.getHooks()` (line 40):**

```java
return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
// called as findByOwnerIdAndHookIdBetween(ownerId, 0, Long.MAX_VALUE, page)
// → returns every hook for the account
``` [4](#0-3) 

**Full exploit flow:**

1. Attacker sends: `GET /api/v1/accounts/0.0.123/hooks?hook.id=gt:9223372036854775807`
2. `getNumberValue` parses `Long.MAX_VALUE` — passes the `< 0` guard.
3. `getInclusiveValue()` computes `Long.MAX_VALUE + 1` → wraps to `Long.MIN_VALUE`.
4. `Math.max(0L, Long.MIN_VALUE)` → `lowerBound` stays `0L`.
5. `upperBound` is untouched at `Long.MAX_VALUE`.
6. Repository query: `hookId BETWEEN 0 AND Long.MAX_VALUE` — full table scan for that owner.
7. All hook records are returned to the unauthenticated caller.

**Existing checks reviewed and shown insufficient:**
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` — only limits parameter count, not value range.
- `value < 0` guard in `getNumberValue` — does not prevent `Long.MAX_VALUE`.
- No `lowerBound > upperBound` guard exists in `hooksRequest()` (contrast: `NetworkServiceImpl.getNetworkNodes()` at line 120 does have this guard). [5](#0-4) 

### Impact Explanation
An unprivileged attacker can retrieve the complete set of smart contract hook records for any account owner, bypassing the intended ID-range filter entirely. Hook storage data may contain sensitive smart contract state. The filter is not merely weakened — it is completely nullified, returning the full result set as if no filter were applied.

### Likelihood Explanation
The endpoint is a public, unauthenticated HTTP GET. No special privileges, tokens, or prior knowledge beyond a valid `ownerId` are required. The payload is a single, trivially constructed query parameter (`hook.id=gt:9223372036854775807`). The attack is deterministic, repeatable, and requires no brute-force or timing. Any external user who can reach the API can exploit this.

### Recommendation

1. **Fix `getInclusiveValue()` to use overflow-safe arithmetic:**
   ```java
   public long getInclusiveValue() {
       if (operator == RangeOperator.GT) {
           if (value == Long.MAX_VALUE) {
               throw new IllegalArgumentException("hook.id gt:Long.MAX_VALUE produces an empty range");
           }
           return value + 1;
       } else if (operator == RangeOperator.LT) {
           if (value == 0) {
               throw new IllegalArgumentException("hook.id lt:0 produces an empty range");
           }
           return value - 1;
       }
       return value;
   }
   ```
   Alternatively use `Math.addExact` and catch `ArithmeticException`.

2. **Add an empty-range guard in `hooksRequest()`** (mirroring `NetworkServiceImpl`):
   ```java
   if (lowerBound > upperBound) {
       return HooksRequest.builder()...lowerBound(1).upperBound(0).build(); // guaranteed empty
   }
   ```

3. **Add an upper-bound validator on the raw numeric value** in `getNumberValue` to reject values that would overflow after operator adjustment (e.g., reject `Long.MAX_VALUE` when operator is `GT`).

### Proof of Concept

```
# Returns ALL hooks for account 0.0.123 instead of an empty list
curl -s "http://<mirror-node>/api/v1/accounts/0.0.123/hooks?hook.id=gt:9223372036854775807"

# Expected: {"hooks":[],"links":{}}
# Actual:   {"hooks":[<all hook records for 0.0.123>],"links":{...}}
```

**Step-by-step:**
1. Identify any valid `ownerId` (e.g., from a public transaction).
2. Issue the GET request above with `hook.id=gt:9223372036854775807`.
3. `getInclusiveValue()` overflows → `Long.MIN_VALUE`.
4. `Math.max(0L, Long.MIN_VALUE)` → `lowerBound = 0`.
5. Repository executes `hookId BETWEEN 0 AND 9223372036854775807` — returns all hooks.
6. Full hook data (including smart contract storage state) is disclosed in the response.
