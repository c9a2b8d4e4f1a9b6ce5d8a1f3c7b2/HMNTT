### Title
Unbounded Perpetual gRPC Subscription Exhausts Listener Fan-Out Resources Without Authentication

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` accepts subscriptions from any unauthenticated caller with no global cap on concurrent subscriptions. A caller who sets `startTime` to the current time and omits both `endTime` and `limit` creates a subscription that never self-terminates, holding a `topicListener` fan-out slot and associated scheduler/buffer resources for the full lifetime of the TCP connection. Because an attacker can open arbitrarily many TCP connections (each carrying up to 5 calls), the total number of perpetual subscriptions is unbounded.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) calls `topicMessageService.subscribeTopic(filter)` with no authentication check and no check against any global subscriber ceiling. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–91) constructs the live pipeline. When `endTime` is absent, `pastEndTime()` returns `Flux.never()` (line 124–125), so the `takeUntilOther` operator never fires. When `limit` is 0, `filter.hasLimit()` is `false` (TopicMessageFilter line 39–41) and `flux.take()` is never applied. The subscription runs until the client disconnects. [2](#0-1) [3](#0-2) [4](#0-3) 

**Resource consumption per perpetual subscription by listener type:**

- **`POLL` mode** (`PollingTopicListener`): Each subscriber gets its own independent polling loop (`RepeatSpec.times(Long.MAX_VALUE)`) on a `boundedElastic` scheduler thread, issuing a DB query every 500 ms indefinitely. [5](#0-4) 

- **`REDIS` / `SHARED_POLL` modes** (`SharedTopicListener`): Each subscriber gets its own `onBackpressureBuffer` of up to `maxBufferSize` (default 16 384) messages and a fan-out slot in the shared Reactor multicast. Memory grows linearly with subscriber count. [6](#0-5) 

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14) limits calls *per TCP connection*, not total connections. An attacker opens N connections × 5 calls = 5N perpetual subscriptions. [7](#0-6) 

- `subscriberCount` (TopicMessageServiceImpl lines 48, 89–90) is a Micrometer gauge for observability only; it is never compared against a ceiling to reject new subscriptions. [8](#0-7) 

- `TopicMessageFilter.isValidStartTime()` only requires `startTime <= now`; it does not require an `endTime` or a non-zero `limit`. [9](#0-8) 

- There is no gRPC interceptor, IP-based rate limiter, or authentication layer in the grpc module's configuration. [10](#0-9) 

### Impact Explanation
Under `POLL` mode, each perpetual subscription permanently occupies a `boundedElastic` thread and issues a DB query every 500 ms. Exhausting the thread pool stalls all gRPC processing for legitimate users. Under `REDIS`/`SHARED_POLL` modes, each subscription allocates a 16 384-slot buffer; thousands of subscriptions exhaust JVM heap. In all modes the `subscriberCount` metric climbs without bound, masking the attack only as a monitoring anomaly. Legitimate subscribers experience degraded throughput or OOM-induced crashes.

### Likelihood Explanation
No credentials, tokens, or network privileges are required. The attack is reproducible with a standard gRPC client (e.g., `grpcurl`) in a loop. The per-connection call cap of 5 is trivially bypassed by opening additional TCP connections. The attack is fully automatable and repeatable from a single host.

### Recommendation
1. **Enforce a global concurrent-subscriber ceiling**: compare `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Require a maximum subscription duration or endTime**: reject or auto-terminate subscriptions that have neither an `endTime` nor a `limit`.
3. **Add per-IP or per-identity connection rate limiting** via a gRPC `ServerInterceptor` before the call reaches `ConsensusController`.
4. **Optionally add authentication** so that abusive callers can be identified and blocked.

### Proof of Concept
```bash
# Open 200 TCP connections, each with 5 perpetual subscriptions = 1000 live subscriptions
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# All 1000 subscriptions remain open indefinitely.
# Under POLL mode: 1000 DB polling loops fire every 500 ms.
# Under SHARED_POLL/REDIS: 1000 × 16384-slot buffers allocated in heap.
# Legitimate subscribers experience resource starvation or OOM.
```

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
```

```java
    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
```
