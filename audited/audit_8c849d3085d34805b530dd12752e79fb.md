### Title
Gas Throttle Bypass via Deliberate Gas-Exhausting Failures Allows Sustained Maximum-Load EVM Execution

### Summary
The `restoreGasToBucket()` logic in `ContractCallService` unconditionally restores 100% of the gas limit back to the throttle bucket whenever a transaction fails with gas exhaustion (`!result.isSuccessful() && gasLimit == result.gasUsed()`). An unprivileged attacker can craft `eth_call` requests targeting any contract with calldata designed to always exhaust all gas, causing the gas bucket to never deplete while the node performs full maximum-gas EVM computation on every request. This effectively nullifies the gas-per-second throttle, allowing the attacker to sustain maximum EVM load indefinitely.

### Finding Description

**Exact code path:**

`ContractCallService.java`, `restoreGasToBucket()`, lines 140–152:

```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline =
        (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f); // default: gasLimit * 100/100 = gasLimit
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline); // restores full gasLimit
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

`ThrottleManagerImpl.java`, `restore()`, lines 59–64:
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);
    }
}
```

**Root cause:** The code's design assumption is that a gas-exhausting failure means the user's transaction was "wasted" and the gas should be returned to the bucket to allow other legitimate users. This assumption is false: the EVM still performed the full computation up to the gas limit, consuming real CPU resources. The gas bucket is supposed to meter that computation, but restoring 100% of the gas after a gas-exhausting failure means the bucket is never actually depleted by such requests.

**Exploit flow:**

1. `throttle()` in `ThrottleManagerImpl` consumes `scaleGas(gasLimit)` tokens from `gasLimitBucket` before EVM execution.
2. The EVM runs the attacker's crafted call until gas is exhausted (`gasUsed == gasLimit`), doing full computation.
3. `restoreGasToBucket()` is called in the `finally` block of `doProcessCall()` (line 129).
4. Condition `!result.isSuccessful() && gasLimit == result.gasUsed()` is TRUE.
5. `restore(gasLimitToRestoreBaseline)` = `restore(gasLimit)` is called (with default `gasLimitRefundPercent = 100`).
6. `scaleGas(gasLimit)` tokens are added back — exactly what was consumed.
7. Net tokens consumed from bucket: **zero**.
8. Attacker repeats indefinitely.

**Why existing checks fail:**

- The RPS limit (`requestsPerSecond = 500`) still applies, but it does not prevent the gas throttle bypass — it only caps request count, not gas computation.
- `gasLimitRefundPercent` defaults to 100 (`ThrottleProperties.java` line 22), meaning 100% of gas is always restored on gas-exhausting failures.
- `scaleGas()` returns 0 only for gas ≤ 10,000 (`ThrottleProperties.java` lines 43–46), which is irrelevant since the attacker uses the maximum gas limit (15,000,000).
- The `bucket4j` `addTokens()` call caps at bucket capacity, preventing overflow, but the attacker only needs the bucket to stay non-empty — not to overflow it.

### Impact Explanation

The gas-per-second throttle (`gasPerSecond = 7,500,000,000` default, `ThrottleConfiguration.java` lines 35–44) is the primary mechanism limiting EVM CPU consumption on the mirror node. By bypassing it, an attacker forces the node to execute maximum-gas EVM computations (up to 15M gas per call) at the full RPS rate (up to 500 RPS). This yields up to 7.5B gas/second of sustained EVM computation — the designed maximum — regardless of what legitimate traffic is doing. If the node normally operates at 50–70% of maximum capacity, the attack increases resource consumption by 43–100%, well exceeding the 30% threshold. The mirror node is a shared read-only service; sustained maximum load degrades response times for all users.

### Likelihood Explanation

No privileges, authentication, or special accounts are required. Any caller of the public `eth_call` or `eth_estimateGas` endpoints can trigger this. Crafting a gas-exhausting call is trivial: target any existing contract with calldata that causes an infinite loop or very expensive computation, or deploy a minimal contract with `while(true){}`. The attack is fully repeatable and automatable with a simple HTTP client loop. The attacker does not need to pay any fees (this is a mirror node simulation, not a real Hedera transaction).

### Recommendation

1. **Do not restore gas for gas-exhausting failures.** The EVM performed real work; the gas should not be returned. Remove or invert the special case for `gasLimit == result.gasUsed()` in the failed branch. Treat gas-exhausting failures the same as successful transactions: restore only `gasRemaining` (which is 0 in this case).

2. **Alternatively, lower `gasLimitRefundPercent` default.** Setting it to 0 eliminates all restoration on gas-exhausting failures, but this is a blunt instrument that also affects legitimate partial-failure cases.

3. **Add per-IP or per-caller rate limiting** as a defense-in-depth measure independent of the gas bucket.

Concrete fix in `restoreGasToBucket()`:
```java
// Remove the special case for gas-exhausting failures.
// Always restore only the unused gas, capped at the refund baseline.
final var gasRemaining = (result == null) ? 0L : gasLimit - result.gasUsed();
throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
```

### Proof of Concept

**Preconditions:** Access to the mirror node's `eth_call` endpoint. No account or credentials required.

**Steps:**

1. Identify or deploy any contract with a function that runs an infinite loop (e.g., `function burn() public { while(true){} }`). On a mirror node, even calling a non-existent function on a complex contract with crafted calldata can exhaust gas.

2. Send repeated `eth_call` requests with `gas` set to the maximum (15,000,000):
```bash
while true; do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0xdeadbeef","to":"<gas-burning-contract>","gas":15000000,"block":"latest"}' &
done
```

3. Observe that:
   - Each call returns a failure (gas exhaustion).
   - The gas throttle never rejects requests with "Gas per second rate limit exceeded" because `restore(15000000)` is called after each failure.
   - Node CPU usage climbs to maximum EVM execution capacity.
   - Legitimate requests experience degraded latency.

**Expected (buggy) behavior:** Gas bucket stays perpetually full; attacker sustains 15M-gas EVM executions at 500 RPS.
