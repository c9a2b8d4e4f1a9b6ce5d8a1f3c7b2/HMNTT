### Title
Unauthenticated Account Enumeration via GraphQL `account()` Binary Oracle

### Summary
The GraphQL endpoint `/graphql/alpha` requires no authentication and exposes a `account(input: AccountInput!)` query that returns a non-null `Account` object when an `entityId.num` exists and `null` when it does not. Because Hedera account numbers are sequential integers starting from 1, an unprivileged external attacker can iterate `num` from 1 to N, use the null/non-null binary oracle to map the entire account space, and retrieve balance, cryptographic key, memo, and relationship data for every discovered account — with no application-level rate limiting in the GraphQL module.

### Finding Description

**Exact code path:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 declares the query with no authentication directive: [1](#0-0) 

`AccountController.java` lines 33–44 handle the `entityId` branch: it calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` and maps the result — returning `Optional.empty()` (serialized as GraphQL `null`) when the account does not exist, and a fully populated `Account` object when it does: [2](#0-1) 

`EntityServiceImpl.java` line 25 performs a plain `findById` on the entity table, filtered only by `EntityType`: [3](#0-2) 

**Root cause:** The GraphQL module has no authentication layer and no application-level rate limiting. The `GraphQlConfiguration` only installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which guard against single-request abuse but do nothing to prevent high-volume sequential enumeration: [4](#0-3) 

**Why existing checks fail:**

The only infrastructure-level protection is a Traefik `inFlightReq` middleware with `amount: 5` per IP, which limits *concurrent* in-flight requests — not the *rate* of sequential requests: [5](#0-4) 

Furthermore, `global.middleware` defaults to `false`, meaning this middleware may not be active in all deployments: [6](#0-5) 

The `web3` module has a full `ThrottleConfiguration` with per-second bucket rate limiting; the GraphQL module has no equivalent: [7](#0-6) 

The `account.graphqls` schema exposes high-value fields — `balance`, `key`, `memo`, `alias`, `autoRenewAccount`, `stakedAccount` — all returned without any field-level access control: [8](#0-7) 

### Impact Explanation
An attacker obtains a complete map of all active Hedera accounts including their HBAR balances, admin keys, memos, and staking relationships. This enables targeted attacks: identifying whale accounts for social engineering, correlating key structures across accounts, and building a full graph of account relationships. The `key` field exposes the cryptographic key type and structure, which can inform key-compromise strategies. The `balance` field directly identifies high-value targets.

### Likelihood Explanation
Zero preconditions: no credentials, no registration, no special network access. The exploit is a trivial loop of HTTP POST requests to a publicly documented endpoint (`/graphql/alpha`). Account numbers are sequential integers, so the search space is fully predictable. The `inFlightReq: 5` middleware (if deployed) allows 5 concurrent requests per IP; an attacker using 100 IPs can sustain 500 concurrent enumeration requests. The entire mainnet account space (tens of millions of accounts) can be mapped in hours.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP token-bucket rate limiter to the GraphQL module analogous to `ThrottleConfiguration` in the `web3` module.
2. **Enforce middleware unconditionally**: Make the Traefik `inFlightReq` and a `rateLimit` middleware mandatory (not opt-in via `global.middleware`) for the GraphQL ingress, and add a `rateLimit` middleware alongside `inFlightReq`.
3. **Consistent null response timing**: Return responses for non-existent accounts with the same latency as existing ones to prevent timing-based enumeration even if null responses are returned.
4. **Consider field-level sensitivity**: Evaluate whether `key`, `balance`, and `memo` should require authentication or be rate-limited independently.

### Proof of Concept

```bash
# Enumerate accounts num=1 through num=1000000, recording which exist
for i in $(seq 1 1000000); do
  RESULT=$(curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance key memo }}\"}")
  # null data.account = does not exist; non-null = exists with full details
  echo "$i: $RESULT"
done
```

No credentials, headers, or special tokens are required. Each response either contains `"account":null` (not found) or a populated object with `balance`, `key`, and `memo` (found). The binary oracle is deterministic and repeatable.
