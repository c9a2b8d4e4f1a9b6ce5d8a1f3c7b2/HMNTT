### Title
Unauthenticated `/tmp/ready` File Creation Bypasses Pinger Readiness Probe

### Summary
The readiness check in `pinger/cmd/healthcheck/main.go` uses `os.Stat("/tmp/ready")` to determine pod readiness, checking only for file existence with no validation of ownership, content, or integrity. Because `/tmp` is world-writable (mode `1777`) inside the container, any process with write access to the container's filesystem can create `/tmp/ready` and force the Kubernetes readiness probe to report the pod as ready — regardless of whether the pinger has actually initialized a valid Hedera client.

### Finding Description
**Code path:** `pinger/cmd/healthcheck/main.go`, `main()`, line 20:
```go
if _, err := os.Stat("/tmp/ready"); err != nil {
    fmt.Fprintln(os.Stderr, "not ready")
    os.Exit(1)
}
os.Exit(0)
```
The legitimate writer is `pinger/main.go` line 47, which creates `/tmp/ready` only after `newClient(cfg)` succeeds (line 41) — meaning the Hedera SDK client is initialized and the transfer ticker is about to start. The healthcheck binary, however, performs no ownership check, no content check, and no inode/permission check. Since `/tmp` inside a Linux container defaults to mode `1777`, any UID running in the same container can `touch /tmp/ready` at any time. The check at line 20 will return `nil` error and `os.Exit(0)`, signaling readiness to Kubernetes.

**Why existing checks fail:** There are none. `os.Stat` returns only filesystem metadata (existence, size, mtime, mode). The code discards the returned `FileInfo` entirely (`_`), so even the metadata is unused. No comparison against expected content (`"ok\n"`), no `os.Getuid()` ownership assertion, no sticky-bit enforcement.

### Impact Explanation
An attacker who forces `/tmp/ready` to exist prematurely causes Kubernetes to:
1. Mark the pinger pod as ready before `newClient()` has succeeded, meaning the pod enters the ready pool during a rolling deployment while it cannot actually submit transactions.
2. Allow a deployment rollout to proceed past the readiness gate with a non-functional pinger, silently degrading the network monitoring coverage that the pinger provides.
3. Prevent Kubernetes from holding back traffic/work from the pod during initialization failures, since the readiness probe will keep returning 0 as long as the file exists.

Severity is bounded at **Medium/griefing** because the pinger is a monitoring tool — it does not process user funds directly — but false readiness masks real initialization failures from operators.

### Likelihood Explanation
Precondition is write access to `/tmp` inside the container. This is achievable by:
- Any cluster user with `kubectl exec` RBAC permission on the pinger pod (a common developer/ops permission, not cluster-admin).
- A compromised sidecar or init container sharing an `emptyDir` volume mounted at `/tmp`.
- Any future code-execution vulnerability in the pinger binary itself.

The attack is a single shell command (`touch /tmp/ready`) and is fully repeatable. The container security context (`runAsNonRoot: true`, `runAsUser: 1000`) does not prevent this because `/tmp` is writable by all UIDs.

### Recommendation
Replace the bare existence check with a content + ownership validation:

```go
case "ready":
    data, err := os.ReadFile("/tmp/ready")
    if err != nil {
        fmt.Fprintln(os.Stderr, "not ready")
        os.Exit(1)
    }
    if string(data) != "ok\n" {
        fmt.Fprintln(os.Stderr, "ready file content invalid")
        os.Exit(1)
    }
    os.Exit(0)
```

Additionally, restrict `/tmp/ready` permissions to `0600` in `pinger/main.go` (change `0o644` → `0o600`) so only the owning UID can read or overwrite it, and consider using a dedicated directory with restricted permissions instead of world-writable `/tmp`.

### Proof of Concept
```bash
# 1. Exec into the pinger pod as any permitted user (no root required)
kubectl exec -it <pinger-pod> -- /bin/sh

# 2. Before or after pinger initialization, create the marker
touch /tmp/ready

# 3. Trigger the readiness probe manually to confirm bypass
/healthcheck ready
echo $?   # prints 0 (ready) regardless of pinger state

# 4. Confirm Kubernetes now considers the pod ready
kubectl get pod <pinger-pod> -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'
# prints: True
```
