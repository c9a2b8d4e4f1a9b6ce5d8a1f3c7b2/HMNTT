### Title
Unauthenticated Multi-DB-Operation Request Exhausts Connection Pool via `getOneAccount()` Alias + Timestamp Path

### Summary
The `getOneAccount()` handler in `rest/accounts.js` performs up to five sequential and parallel database operations per request when an alias path parameter is combined with `timestamp=gte:X&timestamp=lte:Y` filters. The REST API has no rate-limiting middleware, and the shared `pg` connection pool has a finite `maxConnections` ceiling. An unprivileged attacker flooding concurrent requests can exhaust the pool, causing all REST API consumers to receive connection-timeout errors and degrading ≥30% of mirror node processing capacity.

### Finding Description

**Exact code path — `rest/accounts.js`, `getOneAccount()`, lines 396–512:**

**DB Operation 1 — alias lookup (line 399):**
```js
const encodedId = await EntityService.getEncodedId(
  req.params[constants.filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]
);
```
When the path parameter is a base-32 alias, `EntityService.getEncodedId()` calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(EntityService.entityFromAliasQuery, ...)`, issuing a `SELECT id FROM entity WHERE alias = $1` query and consuming one pool connection.

**DB Operation 2 — balance snapshot range lookup (lines 435–438):**
```js
const {lower, upper} = await balances.getAccountBalanceTimestampRange(
  balanceSnapshotTsQuery.replaceAll(opsMap.eq, opsMap.lte),
  balanceSnapshotTsParams
);
```
`getAccountBalanceTimestampRange()` in `rest/balances.js` (lines 167–197) issues a `SELECT consensus_timestamp FROM account_balance WHERE account_id = $1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3 ORDER BY consensus_timestamp DESC LIMIT 1`, consuming a second pool connection sequentially.

**DB Operations 3 & 4 — parallel entity query + transaction pipeline (lines 487–495):**
```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);   // Op 3
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)       // Op 4+5
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]);
```
The entity query (which, when `timestampFilters.length > 0`, forces a `UNION ALL` of `entity` and `entity_history`) and `doGetTransactions()` are launched in parallel. `doGetTransactions()` itself issues two sequential queries: `getTransactionTimestamps()` (line 483 of `transactions.js`) and `getTransactionsDetails()` (line 662). At peak, two pool connections are held simultaneously.

**Root cause — no rate limiting on the REST API:**

`rest/server.js` registers the route at line 102:
```js
app.getExt(`${apiPrefix}/accounts/:${constants.filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS}`, accounts.getOneAccount);
```
The middleware stack (lines 68–144) includes `authHandler`, `requestLogger`, `openApiValidator`, and `responseHandler`, but **no rate-limiting middleware**. The `ThrottleConfiguration` / `ThrottleManagerImpl` classes exist only in the `web3` module and are never applied to the Node.js REST service.

**Pool configuration — `rest/dbpool.js` lines 7–16:**
```js
const poolConfig = {
  max: config.db.pool.maxConnections,
  statement_timeout: config.db.pool.statementTimeout,
  connectionTimeoutMillis: config.db.pool.connectionTimeout,
  ...
};
```
`maxConnections` is a finite integer (validated to be `> 0`). `statement_timeout` limits individual query duration but does not prevent pool exhaustion from many concurrent requests each holding multiple connections.

### Impact Explanation
Each concurrent request with an alias path parameter and a timestamp range consumes 2–5 pool connections across sequential and parallel phases. With a default or small `maxConnections` value, a sustained flood of concurrent requests saturates the pool. All subsequent `pool.queryQuietly()` calls across every REST endpoint block on `connectionTimeoutMillis`, causing cascading 503/timeout errors for all consumers of the mirror node REST API. This directly degrades the mirror node's ability to serve data to wallets, explorers, and dApps — meeting the ≥30% processing degradation threshold without any brute-force cryptographic action.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge beyond a single valid account alias (publicly discoverable from any block explorer or prior API call). The endpoint is publicly reachable. The attacker needs only a script issuing concurrent HTTP GET requests — trivially achievable with `curl`, `ab`, `wrk`, or any HTTP load tool. The attack is repeatable and stateless; each request is independent. No CAPTCHA, IP throttle, or connection-level guard exists in the REST service.

### Recommendation
1. **Add rate limiting to the REST API**: Apply a per-IP rate-limiting middleware (e.g., `express-rate-limit`) to `rest/server.js` before route registration, mirroring the `ThrottleConfiguration` pattern already used in the `web3` module.
2. **Bound concurrent DB connections per request**: Introduce a semaphore or queue in front of `pool.queryQuietly()` to cap the number of in-flight DB operations per request or globally.
3. **Reduce DB operations per request**: Cache the alias→encodedId resolution (the alias is immutable once assigned) to eliminate Operation 1 on repeated requests for the same alias.
4. **Set a conservative `maxConnections`** and ensure `connectionTimeoutMillis` returns a fast 503 rather than blocking indefinitely, so pool exhaustion degrades gracefully rather than causing cascading hangs.

### Proof of Concept
**Preconditions:**
- Mirror node REST API is publicly accessible.
- At least one account with a known alias exists (e.g., `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`).

**Steps:**
```bash
# 1. Confirm the alias resolves
curl "https://<mirror-node>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"

# 2. Flood with concurrent timestamp-range requests (adjust concurrency to exceed maxConnections)
ab -n 10000 -c 200 \
  "https://<mirror-node>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ?timestamp=gte:1600000000.000000000&timestamp=lte:1700000000.000000000"

# 3. Observe: other REST endpoints begin returning 503 / connection timeout errors
curl "https://<mirror-node>/api/v1/transactions"
# Expected: connection timeout or HTTP 503
```

**Result:** The `pg` pool is saturated. All REST API endpoints — transactions, balances, tokens, etc. — fail to acquire connections, degrading mirror node REST processing capacity by ≥30%.
