### Title
Silent Overwrite of Lower Bound When Duplicate GT Operators Supplied to Token Airdrop Endpoints

### Summary
In `TokenAirdropsController.processRequest()`, the `Bound` constructor silently overwrites the `lower` field with the last-seen GT/GTE parameter when two are supplied, while incrementing cardinality for both. Because `verifySingleOccurrence()` is never called anywhere in the token airdrop code path (unlike `NftAllowanceServiceImpl` which does call it), the duplicate is not rejected, and the query executes with only the higher bound, returning a narrower result set that omits valid airdrops.

### Finding Description
**Code path:**

`TokenAirdropsController.java` line 71 accepts up to 2 `receiver.id` parameters via `@Size(max = 2)`:
```java
@RequestParam(name = RECEIVER_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] receiverIds
```

`processRequest()` line 97 constructs the `Bound` with `primarySortField = true`:
```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
```

Inside `Bound` constructor (`Bound.java` lines 47–54), for input `[gt:100, gt:200]`:
```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;  // iteration 1: lower = gt:100; iteration 2: lower = gt:200 (OVERWRITES)
    }
    cardinality.merge(param.operator(), 1, Math::addExact);  // GT cardinality = 2
}
```

After the loop: `lower = gt:200`, `cardinality = {GT: 2}`. The only post-loop check (lines 56–60) is:
```java
if (primarySortField && adjustedLower > adjustedUpper) { throw ... }
```
This does not fire (201 is not > `Long.MAX_VALUE`). The constructor returns without error.

Back in `processRequest()` (lines 97–108), there is **no call** to `entityIdsBound.verifySingleOccurrence()`, `entityIdsBound.verifyUnsupported()`, or any equivalent. `TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) also performs no such validation — it immediately delegates to `repository.findAll(request, id)`.

**Contrast with `NftAllowanceServiceImpl.java` lines 56–60**, which correctly calls `ids.verifySingleOccurrence()` via `verifyRangeId()`. That call would throw `IllegalArgumentException` when GT cardinality > 1. The token airdrop path has no equivalent guard.

`verifySingleOccurrence()` (`Bound.java` lines 155–159) checks:
```java
verifySingleOccurrence(RangeOperator.GT, RangeOperator.GTE);  // throws if cardinality > 1
```
This method exists and works correctly — it is simply never invoked for token airdrop requests.

**Result:** The query is built using `receiver_id > 200` (the higher, last-seen bound), silently discarding the `gt:100` constraint. Airdrops with `receiver_id` in the range (100, 200] are excluded from the response.

### Impact Explanation
Any unprivileged caller can supply `receiver.id=gt:100&receiver.id=gt:200` to either `/airdrops/outstanding` or `/airdrops/pending`. The response will silently omit all airdrops whose primary sort field value falls between the two supplied bounds. This is an information-integrity failure: the API returns a subset of the correct result set with no error, no indication of truncation, and a valid HTTP 200. A user relying on this endpoint to enumerate claimable or outstanding airdrops will receive an incomplete list. While no funds are directly moved by this read-only endpoint, a receiver who misses pending airdrops in the listing may fail to claim them before expiry or may make incorrect decisions based on the incomplete data.

### Likelihood Explanation
No authentication or special privilege is required. The `@Size(max = 2)` annotation explicitly permits two values for the parameter, making the input structurally valid. The bug is trivially triggered by any HTTP client. It is repeatable and deterministic. The same flaw applies symmetrically to `sender.id` on the `/pending` endpoint.

### Recommendation
Add validation of `entityIdsBound` in `processRequest()` immediately after construction, mirroring the pattern in `NftAllowanceServiceImpl`:

```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
entityIdsBound.verifyUnsupported(RangeOperator.NE);
entityIdsBound.verifySingleOccurrence();
entityIdsBound.verifyEqualOrRange();
```

Apply the same guards to `serialNumbers` and `tokenIds` bounds. Alternatively, centralize this validation in `TokenAirdropServiceImpl.getAirdrops()` before delegating to the repository, so all callers benefit.

### Proof of Concept
**Precondition:** Sender account `0.0.1000` has outstanding airdrops to receivers `0.0.150` and `0.0.250`.

**Step 1 — Baseline (correct query):**
```
GET /api/v1/accounts/1000/airdrops/outstanding?receiver.id=gt:100
```
Returns both airdrops (receiver 150 and receiver 250). ✓

**Step 2 — Exploit (duplicate GT operators):**
```
GET /api/v1/accounts/1000/airdrops/outstanding?receiver.id=gt:100&receiver.id=gt:200
```
Expected: both airdrops returned (receiver 150 and 250, both > 100).
Actual: only the airdrop to receiver 250 is returned. The airdrop to receiver 150 (which satisfies `> 100` but not `> 200`) is silently omitted. HTTP 200 with no error.

**Step 3 — Confirm root cause:**
The `Bound` is constructed with `lower = gt:200` (the second value overwrites the first). The generated SQL condition is `receiver_account_id > 200`, not `receiver_account_id > 100`.
