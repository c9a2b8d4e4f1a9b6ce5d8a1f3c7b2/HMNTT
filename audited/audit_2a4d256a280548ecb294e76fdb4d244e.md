### Title
Unsynchronized Shared `LinkedList` in Singleton `MirrorNodeState` Enables `ConcurrentModificationException` via Concurrent Queue Read/Write

### Summary
`MirrorNodeState` is a singleton Spring bean (`@Named`) that initializes a single `new LinkedList<>()` as the backing store for the `RecordCacheService` transaction-receipts queue. Both `ListReadableQueueState` and `ListWritableQueueState` hold a direct reference to this same `LinkedList` instance with no synchronization. When two concurrent requests race — one iterating the queue via `iterateOnDataSource()` and another modifying it via `addToDataSource()` or `removeFromDataSource()` — Java's fail-fast `LinkedList` iterator throws `ConcurrentModificationException`, corrupting the transaction receipt processing pipeline.

### Finding Description

**Shared mutable state (root cause):**

`MirrorNodeState.initQueueStates()` (line 171) creates one `LinkedList<>` and stores it in the `states` map:
```java
states.put(RecordCacheService.NAME, new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID, new LinkedList<>())));
```

`getReadableStates()` (line 77–78) wraps this same instance in `ListReadableQueueState`:
```java
if (state instanceof Queue<?> queue) {
    data.put(stateId, new ListReadableQueueState<>(serviceName, stateId, queue));
```

`getWritableStates()` (line 101–102) wraps the same instance in `ListWritableQueueState`:
```java
if (state instanceof Queue<?> queue) {
    data.put(stateId, new ListWritableQueueState<>(serviceName, stateId, queue));
```

Both `readableStates` and `writableStates` are cached in `ConcurrentHashMap` fields (lines 52–53), so the same wrapper objects — and therefore the same `LinkedList` — are reused across all concurrent requests.

**Unsynchronized iterator (trigger):**

`ListReadableQueueState.iterateOnDataSource()` (line 40–42) returns a raw fail-fast iterator:
```java
protected Iterator<E> iterateOnDataSource() {
    return backingStore.iterator();
}
```

`ListWritableQueueState.addToDataSource()` (line 31–33) and `removeFromDataSource()` (line 36–38) mutate the same `LinkedList` directly:
```java
protected void addToDataSource(@NonNull E element) { backingStore.add(element); }
protected void removeFromDataSource() { backingStore.remove(); }
```

**Why existing checks are insufficient:**

The `ConcurrentHashMap` at lines 52–53 only protects the map of state wrappers, not the `LinkedList` contents. There is no `synchronized` block, `ReentrantLock`, `CopyOnWriteArrayList`, or `Collections.synchronizedList()` wrapping the `LinkedList`. The `WritableQueueStateBase` may buffer adds locally, but `commit()` (called from `MapWritableStates.commit()` line 102) flushes them via `addToDataSource()` directly onto the shared `LinkedList` with no lock held.

### Impact Explanation
A `ConcurrentModificationException` thrown inside `iterateOnDataSource()` propagates up through the receipt-lookup or receipt-iteration path of the transaction processing pipeline. This causes the affected request to fail with an unhandled exception. Under sustained concurrent load, this is reliably triggerable, resulting in denial-of-service for transaction receipt queries and potential corruption of in-flight receipt processing state. Severity: **Medium–High** (availability impact on the web3 transaction pipeline; no authentication required).

### Likelihood Explanation
The mirror node's web3 module is publicly accessible. Any unprivileged user can submit concurrent `eth_sendRawTransaction` (or equivalent) calls. Two concurrent requests are sufficient: one that causes a write commit to the `RecordCacheService` queue and one that iterates it. This is trivially achievable with a simple HTTP client sending parallel requests. The race window is narrow but repeatable under load, making it a practical denial-of-service vector.

### Recommendation
Replace the `LinkedList` backing store with a thread-safe alternative:
- Use `new java.util.concurrent.ConcurrentLinkedQueue<>()` in `initQueueStates()` (line 171), or
- Wrap with `Collections.synchronizedList(new LinkedList<>())` and synchronize on the list object during iteration, or
- Use `java.util.concurrent.LinkedBlockingDeque` which is fully thread-safe.

Additionally, review whether `writableStates` should be per-request-scoped rather than singleton-cached, since sharing a single `ListWritableQueueState` across all requests is architecturally unsound for a mutable queue.

### Proof of Concept
1. Deploy the mirror node web3 service.
2. From two threads simultaneously:
   - **Thread A**: Submit a transaction via `eth_sendRawTransaction` that triggers `getWritableStates(RecordCacheService.NAME)` → `ListWritableQueueState.addToDataSource()` → `backingStore.add()` on the shared `LinkedList`.
   - **Thread B**: Query transaction receipts in a way that triggers `getReadableStates(RecordCacheService.NAME)` → `ListReadableQueueState.iterateOnDataSource()` → `backingStore.iterator()` followed by `iterator.next()`.
3. When Thread A's `add()` modifies the `LinkedList` while Thread B's iterator is active, `iterator.next()` throws `ConcurrentModificationException`.
4. Observe the exception propagating through the receipt processing pipeline, causing the request to fail.
