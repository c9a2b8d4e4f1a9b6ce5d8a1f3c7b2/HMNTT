### Title
Topic ID Enumeration via Differential gRPC Error Responses in `topicExists()`

### Summary
The `topicExists()` method in `TopicMessageServiceImpl.java` emits two semantically distinct exceptions — `EntityNotFoundException` and `IllegalArgumentException` — which are mapped to two different gRPC status codes (`NOT_FOUND` vs. `INVALID_ARGUMENT`) by `ProtoUtil.toStatusRuntimeException()`. Any unauthenticated caller can probe arbitrary entity IDs via `subscribeTopic` and use the differing error codes to determine whether an entity ID exists in the system and whether it is a topic, enabling full enumeration of the entity namespace.

### Finding Description

**Code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `topicExists()`, lines 94–106:

```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
            .switchIfEmpty(
                    grpcProperties.isCheckTopicExists()
                            ? Mono.error(new EntityNotFoundException(topicId))   // → gRPC NOT_FOUND
                            : Mono.just(Entity.builder()
                                    .memo("")
                                    .type(EntityType.TOPIC)
                                    .build()))
            .filter(e -> e.getType() == EntityType.TOPIC)
            .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));  // → gRPC INVALID_ARGUMENT
}
```

**Error mapping** in `grpc/src/main/java/org/hiero/mirror/grpc/util/ProtoUtil.java`, lines 47–52:

```java
} else if (t instanceof ConstraintViolationException
        || t instanceof IllegalArgumentException
        || t instanceof InvalidEntityException) {
    return clientError(t, Status.INVALID_ARGUMENT, t.getMessage());   // "Not a valid topic"
} else if (t instanceof EntityNotFoundException) {
    return clientError(t, Status.NOT_FOUND, t.getMessage());           // "<id> does not exist"
```

**Root cause:** The two branches of the entity-existence check produce exceptions of different Java types, which are then mapped to different gRPC status codes with distinct human-readable messages. The caller receives:

| Scenario | gRPC Status | Message |
|---|---|---|
| Entity ID does not exist (and `checkTopicExists=true`) | `NOT_FOUND` | `"0.0.X does not exist"` |
| Entity ID exists but is not a TOPIC | `INVALID_ARGUMENT` | `"Not a valid topic"` |
| Entity ID is a valid TOPIC | stream opens | — |

**Failed assumption:** The code assumes that error responses are opaque to callers. In fact, gRPC status codes and description strings are transmitted in full to the client over the wire.

**Why `checkTopicExists=false` does not fully mitigate:** When `checkTopicExists` is `false`, a missing entity is synthesized as a fake `TOPIC` entity, so `NOT_FOUND` is suppressed for missing IDs. However, the `.filter(e -> e.getType() == EntityType.TOPIC)` check on line 104 still runs against real DB-fetched entities. If an entity exists with a non-TOPIC type (account, contract, token, file, etc.), the filter drops it and `IllegalArgumentException("Not a valid topic")` is still thrown as `INVALID_ARGUMENT`. This means even with `checkTopicExists=false`, the attacker can still distinguish "entity exists and is not a topic" from "entity does not exist" — just with the `NOT_FOUND` case collapsed into the stream-open case.

### Impact Explanation
An attacker can enumerate the entire entity namespace by iterating numeric IDs (e.g., `0.0.1` through `0.0.N`) and classifying each response:
- `NOT_FOUND` → entity does not exist
- `INVALID_ARGUMENT` + "Not a valid topic" → entity exists, is not a topic (account/contract/token/file)
- Stream opens → entity is a valid topic

This produces a complete map of which entity IDs are topics. With this map, an attacker can selectively subscribe to high-value topic streams (e.g., those used for transaction submission or consensus signaling), monitor message patterns, and identify targets for transaction-stream tampering or replay attacks. The information disclosure directly enables the "Tampering with submitted transactions" threat by removing the obscurity that would otherwise make targeting specific topic streams difficult.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication requirement. The probe is a single gRPC unary-style call per entity ID (the stream can be immediately cancelled after receiving the error). Iterating through tens of thousands of entity IDs is trivially automatable with any gRPC client library in seconds. No credentials, tokens, or special network position are required.

### Recommendation
1. **Normalize error responses:** Return the same gRPC status code (`NOT_FOUND` or `INVALID_ARGUMENT`) for both "entity does not exist" and "entity exists but is not a topic." Prefer a single opaque error such as `NOT_FOUND` with a generic message like `"Topic not found"` for both cases, eliminating the distinguishing signal.
2. **Unify exception types:** Replace the `IllegalArgumentException` on line 105 with `EntityNotFoundException` (or a new `TopicNotFoundException`) so both paths map to the same gRPC status code in `ProtoUtil`.
3. **Rate-limit the endpoint:** Apply per-IP or per-client rate limiting on `subscribeTopic` to raise the cost of enumeration even if error normalization is delayed.

### Proof of Concept

```python
import grpc
from com.hederahashgraph.api.proto.java import ConsensusService_pb2_grpc, ConsensusTopicQuery_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = ConsensusService_pb2_grpc.ConsensusServiceStub(channel)

for entity_num in range(1, 10000):
    request = ConsensusTopicQuery_pb2.ConsensusTopicQuery(
        topicID={"topicNum": entity_num}
    )
    try:
        for _ in stub.subscribeTopic(request):
            print(f"0.0.{entity_num} → VALID TOPIC (stream opened)")
            break
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.NOT_FOUND:
            print(f"0.0.{entity_num} → DOES NOT EXIST")
        elif e.code() == grpc.StatusCode.INVALID_ARGUMENT and "Not a valid topic" in e.details():
            print(f"0.0.{entity_num} → EXISTS, NOT A TOPIC (account/contract/token/file)")
```

    public static StatusRuntimeException toStatusRuntimeException(Throwable t) {
        if (Exceptions.isOverflow(t)) {
            return clientError(t, Status.DEADLINE_EXCEEDED, OVERFLOW_ERROR);
        } else if (t instanceof ConstraintViolationException
                || t instanceof IllegalArgumentException
                || t instanceof InvalidEntityException) {
            return clientError(t, Status.INVALID_ARGUMENT, t.getMessage());
        } else if (t instanceof EntityNotFoundException) {
            return clientError(t, Status.NOT_FOUND, t.getMessage());
        } else if (t instanceof TransientDataAccessException || t instanceof TimeoutException) {
            return serverError(t, Status.RESOURCE_EXHAUSTED, DB_ERROR);
        } else if (t instanceof NonTransientDataAccessResourceException) {
            return serverError(t, Status.UNAVAILABLE, DB_ERROR);
        } else {
            return serverError(t, Status.UNKNOWN, UNKNOWN_ERROR);
        }
    }
