### Title
Unsanitized `X-Real-IP` Header Enables Log Injection and Audit Trail Manipulation for Fund Transfer Submissions

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the `X-Real-IP` header value without any format validation and embeds it verbatim into log messages via `TracingMiddleware`. An unprivileged attacker who can reach the Rosetta API directly can inject arbitrary byte sequences — including multi-byte Unicode or newline-containing strings — into the log stream, corrupting or forging log entries for `/construction/submit` calls and creating gaps in the audit trail for fund transfers.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, lines 63–74 (`getClientIpAddress`) and lines 52–53 (`TracingMiddleware`):

```go
// line 64: raw header value, zero validation
ipAddress := r.Header.Get(xRealIpHeader)

// line 66: only checks for empty string (byte count), not IP format
if len(ipAddress) == 0 {
    ipAddress = r.Header.Get(xForwardedForHeader)
}
// ...
return ipAddress   // arbitrary attacker-controlled string returned
```

```go
// line 52-53: verbatim interpolation into log message
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
log.Info(message)   // line 58
```

**Root cause:** The function assumes the `X-Real-IP` header contains a valid IP address. The only guard (`len(ipAddress) == 0`) tests for an absent header using byte count — it does not validate IP format. Any non-empty string, including multi-byte UTF-8 sequences, emoji, or embedded newlines, passes this check and is returned unchanged.

**Exploit flow:**
1. Attacker sends a POST to `/construction/submit` with `X-Real-IP: 😀😀😀 INFO fake-ip POST /construction/submit (200) in 1ms\n`.
2. `getClientIpAddress` returns the raw emoji+newline string (byte length > 0, passes the `len` check).
3. `TracingMiddleware` calls `fmt.Sprintf` and `log.Info`, writing the attacker-controlled string into the log.
4. The resulting log stream contains a forged "successful" submit entry with a fake IP, immediately followed by the real (failed or anomalous) entry.
5. Log parsers using regex patterns like `\d{1,3}(\.\d{1,3}){3}` to extract IP addresses from log lines fail to match the Unicode field, causing the real `/construction/submit` entry to be skipped or misclassified by SIEMs/audit pipelines.

**Why existing checks fail:** The `len(ipAddress) == 0` check at lines 66 and 70 is purely an emptiness guard. A 4-byte emoji has `len == 4`, so it is treated as a valid IP address. There is no call to `net.ParseIP`, no regex validation, and no sanitization before the value reaches `fmt.Sprintf`.

### Impact Explanation
Every `/construction/submit` request (the fund-transfer submission endpoint, `construction_service.go` line 332) is logged by `TracingMiddleware` with the attacker-controlled IP as the first field. An attacker can:
- **Forge log entries**: inject newlines to create synthetic "successful" log lines attributed to arbitrary IPs, masking the real source.
- **Break audit parsers**: embed Unicode/emoji so that IP-extraction regexes fail, causing the real submit entry to be dropped from SIEM dashboards or compliance reports.
- **Obscure attribution**: make forensic reconstruction of which IP submitted a fraudulent transaction impossible from logs alone.

The Rosetta Construction API is the only path for submitting signed transactions to the Hedera network via this service; audit trail integrity for this endpoint is critical.

### Likelihood Explanation
The precondition is that the Rosetta API is reachable directly by the attacker (no reverse proxy that unconditionally overwrites `X-Real-IP`). Rosetta nodes are commonly deployed with direct exposure for exchange/custodian integrations. No authentication is required to reach `/construction/submit` at the HTTP layer — the middleware runs before any business logic. The attack requires only a single crafted HTTP header and is trivially repeatable with `curl`.

### Recommendation
Replace the raw header read with strict IP validation before accepting the value:

```go
import "net"

func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        raw := r.Header.Get(header)
        // Take only the first comma-separated token (X-Forwarded-For may be a list)
        if idx := strings.IndexByte(raw, ','); idx != -1 {
            raw = raw[:idx]
        }
        raw = strings.TrimSpace(raw)
        if ip := net.ParseIP(raw); ip != nil {
            return ip.String() // normalized, safe for logging
        }
    }
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

Additionally, sanitize log output by stripping or escaping non-printable and non-ASCII bytes before passing values to `log.Info`/`log.Debug`.

### Proof of Concept
```bash
# Inject a Unicode emoji IP that breaks ASCII log parsers
curl -X POST http://<rosetta-host>/construction/submit \
  -H 'Content-Type: application/json' \
  -H $'X-Real-IP: \xf0\x9f\x98\x80\xf0\x9f\x98\x80\xf0\x9f\x98\x80' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"signed_transaction":"0x..."}'

# Inject a newline to forge a synthetic log entry
curl -X POST http://<rosetta-host>/construction/submit \
  -H 'Content-Type: application/json' \
  -H $'X-Real-IP: 1.2.3.4\ntime="2024-01-01" level=info msg="10.0.0.1 POST /construction/submit (200) in 1ms"' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"signed_transaction":"0x..."}'
```

**Expected result:** The logrus output contains either a non-ASCII IP field (breaking regex-based parsers) or a forged second log line, both sourced from the attacker-controlled `X-Real-IP` header. Verified against `rosetta/app/middleware/trace.go` lines 52–58 and `getClientIpAddress` lines 63–74.
