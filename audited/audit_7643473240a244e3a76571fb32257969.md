### Title
Unauthenticated Request Flooding of `/block` Endpoint Causes Resource Exhaustion Due to Disabled Rate-Limiting Middleware

### Summary
The `Block()` handler in `blockAPIService` performs multiple expensive database queries per request with no application-level rate or concurrency limiting. The Traefik middleware chain that defines per-IP in-flight limits and per-host rate limits is gated behind `global.middleware: false` in the default Helm values, meaning it is never deployed. Any unprivileged external user can flood the `/block` endpoint, exhausting DB connections, goroutines, and memory until the server becomes unresponsive to legitimate callers.

### Finding Description

**Code path — application layer:**

`rosetta/app/services/block_service.go` `Block()` (lines 47–74) issues three potentially expensive operations per request:
1. `RetrieveBlock` — one DB round-trip to resolve the block identifier.
2. `FindBetween` — one DB query that returns every transaction in the block (unbounded result set).
3. `updateOperationAccountAlias` — one DB call per operation across every returned transaction (N+1 pattern).

There is no semaphore, token bucket, or any other concurrency guard inside `blockAPIService`.

**Code path — HTTP server layer:**

`rosetta/main.go` (lines 217–227) wraps the router in exactly three middleware layers: `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`. None of these limit connections, request rate, or concurrent in-flight requests. The `http.Server` struct sets only timeout values; it sets no `MaxHeaderBytes` override and no connection-count ceiling. Go's `net/http` server spawns an unbounded goroutine per accepted connection.

**Why the existing Traefik guard is ineffective:**

`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3 wraps the entire CRD manifest in:
```
{{ if and .Values.global.middleware .Values.middleware }}
```
`charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false`. Because the condition is false, the Traefik `Middleware` CRD objects (including `inFlightReq: amount: 5` per IP and `rateLimit: average: 10` per host) are **never rendered or applied**. The ingress annotation on line 121 references the middleware chain by name, but since the CRD does not exist, Traefik silently drops the middleware reference and forwards all traffic directly to the pod with no throttling.

**Exploit flow:**

An attacker sends a high-volume stream of POST requests to `/block` (or `/block/transaction`) with valid or near-valid `BlockIdentifier` values. Each request causes the server to open DB connections and hold goroutines for the duration of the query. Because there is no limit on concurrent goroutines or DB connections, the DB connection pool saturates, goroutine count grows unboundedly, and heap memory climbs until the process is OOM-killed or the DB refuses further connections. Legitimate callers receive timeouts or connection errors.

### Impact Explanation

Availability of the Rosetta API is fully disrupted for all legitimate users (block explorers, exchange integrations, Coinbase Rosetta validators) for the duration of the attack. No authentication, API key, or privileged access is required. The attack is amplified by the N+1 DB query pattern in `updateOperationAccountAlias`: a block with many transactions and many operations per transaction multiplies the DB load per single HTTP request, meaning even a modest request rate (tens of requests per second) can saturate the database. Severity is medium because there is no direct financial loss, but service availability is completely controllable by an external attacker.

### Likelihood Explanation

The precondition is zero: any host with network access to the Rosetta port can execute this attack. The endpoint is publicly documented (Rosetta API spec), the request format is standardized JSON, and valid block indices are trivially enumerable from the chain. The attack is repeatable and requires no special tooling beyond `curl` or `ab`. The default Helm deployment ships with the protection disabled, so every out-of-the-box deployment is affected unless an operator explicitly sets `global.middleware: true`.

### Recommendation

1. **Enable the Traefik middleware by default**: Change `global.middleware` from `false` to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95, or document it as a required production setting with a clear warning.

2. **Add application-level concurrency limiting**: Wrap the HTTP handler in `rosetta/main.go` with a semaphore-based middleware (e.g., using `golang.org/x/sync/semaphore` or a channel-based token bucket) that caps total in-flight requests before they reach `blockAPIService`.

3. **Add per-IP rate limiting at the application layer** as a defense-in-depth measure independent of the ingress controller, so the protection is not silently lost when deploying without Traefik.

4. **Cap the `FindBetween` result set** at the DB query level to bound the per-request DB cost regardless of block size.

### Proof of Concept

```bash
# Precondition: Rosetta node running in online mode, default Helm deployment
# (global.middleware: false, no Traefik rate limiting active)

ROSETTA_HOST="http://<rosetta-host>:5700"

# Flood /block with concurrent requests using a known block index
for i in $(seq 1 500); do
  curl -s -X POST "$ROSETTA_HOST/block" \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1}}' &
done
wait

# Expected result:
# - DB connection pool exhausted; subsequent legitimate requests receive
#   500 errors or hang until WriteTimeout is reached.
# - Server goroutine count spikes proportionally to concurrent requests.
# - Legitimate /network/status or /block calls time out or are refused.
```
