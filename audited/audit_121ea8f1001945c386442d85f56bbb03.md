### Title
`TransferTransaction` Token/NFT Transfer Blind Spot in `Parse` Enables Deceptive On-Chain Execution

### Summary
`isSupportedTransactionType()` gates on Go struct type alone, not transaction content. A `hiero.TransferTransaction` can legally carry token and NFT transfers inside the same protobuf body, yet `cryptoTransferTransactionConstructor.Parse()` reads only `GetHbarTransfers()` and silently discards all token/NFT transfer legs. An unprivileged attacker can craft a `TransferTransaction` that passes the type gate, presents as a harmless HBAR-only transfer to `/construction/parse`, but executes hidden token/NFT drains when submitted.

### Finding Description
**Type gate — `construction_service.go` lines 676–685:**
```go
func isSupportedTransactionType(transaction hiero.TransactionInterface) *rTypes.Error {
    switch transaction.(type) {
    case hiero.AccountCreateTransaction:
    case hiero.TransferTransaction:
    default:
        return errors.ErrTransactionInvalidType
    }
    return nil
}
```
The check is a Go type-switch. Any `hiero.TransferTransaction` value passes unconditionally, regardless of what transfer legs (HBAR, fungible token, NFT) are encoded in its protobuf body.

**Parse blind spot — `crypto_transfer_transaction_constructor.go` lines 68–78:**
```go
hbarTransferMap := transferTransaction.GetHbarTransfers()
numOperations := len(hbarTransferMap)
operations := make(types.OperationSlice, 0, numOperations)
for accountId, hbarAmount := range hbarTransferMap {
    ...
}
```
`GetTokenTransfers()` and `GetNftTransfers()` are never called anywhere in the rosetta codebase (confirmed: zero grep matches for those symbols). Token and NFT transfer legs are therefore invisible to `/construction/parse`.

**Submit path — `construction_service.go` lines 340–355:**
```go
transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
...
_, err = hiero.TransactionExecute(transaction, c.sdkClient)
```
No content-level validation occurs between the type gate and `TransactionExecute`. The full protobuf body — including hidden token/NFT transfers — is forwarded to the Hedera network verbatim.

**Exploit flow:**
1. Attacker constructs a `TransferTransaction` containing:
   - A nominal HBAR transfer (e.g., −1 tinybar attacker → victim) to appear legitimate
   - A large fungible-token or NFT transfer from victim → attacker
2. Attacker hex-encodes the unsigned transaction and sends it to the victim (e.g., as part of a Rosetta-based exchange integration).
3. Victim calls `POST /construction/parse` — response shows only the HBAR operation; token drain is invisible.
4. Victim signs the `frozenBodyBytes` (which cover the full protobuf body including the token transfer) and returns the signature.
5. Attacker calls `POST /construction/combine` then `POST /construction/submit`.
6. Hedera network executes the full `CryptoTransfer`, draining the victim's tokens.

No privilege is required; any caller of the public Rosetta HTTP endpoints can execute this.

### Impact Explanation
The Rosetta `/construction/parse` endpoint is the canonical "human-readable verification" step in the Rosetta construction workflow. Its failure to surface token/NFT transfers breaks the security contract that signers rely on to give informed consent. A victim who follows the standard Rosetta signing flow is deceived into authorizing arbitrary token/NFT drains. Severity is **High**: funds (tokens/NFTs) can be stolen from any account whose owner uses the Rosetta API to verify transactions before signing.

### Likelihood Explanation
The attack requires no special privileges — only the ability to craft a hex-encoded Hedera transaction (trivially done with the public Hiero SDK) and social-engineering a victim into using `/construction/parse` as their verification step. In exchange or wallet integrations built on Rosetta, this is the standard workflow. The attack is repeatable and requires no on-chain state beyond the victim holding tokens.

### Recommendation
In `cryptoTransferTransactionConstructor.Parse()`, after reading HBAR transfers, explicitly check for non-empty token and NFT transfer maps and return an error if any are present:
```go
if len(transferTransaction.GetTokenTransfers()) > 0 || len(transferTransaction.GetNftTransfers()) > 0 {
    return nil, nil, errors.ErrOperationTypeUnsupported
}
```
Alternatively, extend the operation model to surface token/NFT transfers as distinct operations so they are visible to signers. Either way, `isSupportedTransactionType` should be complemented by a content-level validation that rejects (or fully represents) all transfer legs present in the transaction body.
