### Title
Unbounded JSON Body Parsing in `/mempool` Endpoint Enables CPU Exhaustion DoS

### Summary
The `/mempool` endpoint accepts arbitrarily large JSON request bodies with no size limit enforced at the application layer. The rosetta-sdk-go SDK fully deserializes the JSON body into a `*types.NetworkRequest` struct before invoking `Mempool()`, which immediately discards the parsed result and returns `ErrNotImplemented`. An unprivileged attacker can exploit this parse-then-discard pattern by flooding the endpoint with large bodies containing thousands of repeated fields, exhausting CPU across nodes.

### Finding Description
**Code path:** `rosetta/app/services/mempool_service.go`, `Mempool()`, lines 22–27; `rosetta/main.go`, lines 89–90, 217–226.

**Root cause:** The rosetta-sdk-go `MempoolAPIController` (registered at line 90 of `main.go`) performs full JSON deserialization of the HTTP request body into `*types.NetworkRequest` before dispatching to `Mempool()`. The handler signature `_ *types.NetworkRequest` (line 24 of `mempool_service.go`) discards the result entirely. No `http.MaxBytesReader` wrapper is applied anywhere in the rosetta middleware stack — confirmed absent across all `rosetta/**/*.go` files. The `http.Server` is configured with `ReadTimeout: 5s` (line 225 of `main.go`), which bounds reading time but not the CPU cost of parsing a body already buffered. The Traefik middleware chain providing `inFlightReq` and `rateLimit` controls is gated on `global.middleware: true` (line 95 of `charts/hedera-mirror-rosetta/values.yaml`), which defaults to `false`, leaving those controls inactive in default deployments.

**Exploit flow:**
1. Attacker opens many concurrent TCP connections to port 5700 (or the ingress).
2. Each connection POSTs to `/mempool` with a body like `{"network_identifier": {...}, "network_identifier": {...}}` repeated tens of thousands of times, sized to be deliverable within the 5-second read window.
3. Go's `encoding/json` processes every duplicate key in sequence (O(n) in field count), consuming CPU proportional to body size.
4. The parsed struct is passed to `Mempool()` and immediately discarded; no early-exit occurs before parsing completes.
5. With the pod CPU cap of `500m`, even a modest flood of large-body requests saturates the available CPU, causing request queuing and service degradation.

**Why existing checks fail:**
- `ReadTimeout: 5s` limits reading duration, not body size or parse CPU.
- Traefik `inFlightReq`/`rateLimit` are disabled by default (`global.middleware: false`).
- No `http.MaxBytesReader` is set anywhere in the rosetta server stack.

### Impact Explanation
A sustained flood of large-body POST requests to `/mempool` can pin the rosetta node's CPU at its container limit (500m), making it unresponsive to legitimate requests. Because the Helm chart defaults to `minReplicas: 1` with HPA scaling on CPU utilization at 80%, the attack can trigger runaway horizontal scaling or, if scaling is bounded, cause >= 30% of deployed rosetta nodes to become unresponsive — matching the stated severity scope of disrupting network processing nodes without brute-force key actions.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly reachable via the ingress path `/rosetta/mempool`. The attack requires only a standard HTTP client and sufficient outbound bandwidth to fill the 5-second read window per connection. It is trivially scriptable, repeatable, and requires no special knowledge beyond the public Rosetta API specification.

### Recommendation
1. Wrap `r.Body` with `http.MaxBytesReader(w, r.Body, maxBytes)` in a middleware applied before the rosetta-sdk-go router, enforcing a strict body size cap (e.g., 64 KB for `/mempool`).
2. Enable `global.middleware: true` in production Helm deployments to activate the Traefik `inFlightReq` and `rateLimit` controls.
3. Consider returning early at the HTTP handler layer for known-unimplemented endpoints before JSON deserialization occurs, or contribute an upstream fix to rosetta-sdk-go to skip body parsing when the handler is a no-op stub.
