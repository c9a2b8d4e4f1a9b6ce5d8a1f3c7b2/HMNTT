### Title
Audit Log IP Spoofing via Unsanitized `X-Real-IP` Header in `getClientIpAddress()` Under Traefik Insecure Forwarded-Header Mode

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header with first-wins precedence over `X-Forwarded-For`. In the Kubernetes deployment, Traefik is configured with `--entryPoints.web.forwardedHeaders.insecure`, which passes all client-supplied forwarded headers through unmodified. Combined with Traefik's access log being filtered to only record 4xx–5xx responses, a successful transaction submission can be logged under a fully attacker-controlled IP address with no reconcilable proxy-side record.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // line 64: X-Real-IP wins unconditionally

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67: fallback only
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // line 71: last resort
    }

    return ipAddress
}
```

The only guard is a length check (`len(ipAddress) == 0`). There is no validation of the IP value, no trusted-proxy allowlist, and no stripping of client-injected headers.

**Why the proxy does not save you:**

The Docker Compose nginx proxy does overwrite `X-Real-IP` with `$remote_addr` (line 204 of `docker-compose.yml`), which would neutralize the attack in that deployment. However, the Kubernetes deployment uses Traefik configured with:

```yaml
# charts/hedera-mirror-common/values.yaml, lines 635-637
globalArguments:
  - --entryPoints.web.forwardedHeaders.insecure
  - --entryPoints.websecure.forwardedHeaders.insecure
```

`forwardedHeaders.insecure` instructs Traefik to trust forwarded headers from **any source without restriction**, meaning it passes client-supplied `X-Real-IP` and `X-Forwarded-For` headers to the upstream application completely unmodified.

**Why the discrepancy is irreconcilable:**

Traefik's access log is configured to record only error responses:

```yaml
# charts/hedera-mirror-common/values.yaml, lines 639-643
logs:
  access:
    enabled: true
    filters:
      statuscodes: 400-599
```

A successful transaction submission returns HTTP 200. Traefik does **not** log it. The only audit record that exists is the application's own log entry, which contains the attacker-forged IP.

**Exploit flow:**

1. Attacker crafts a request to the Rosetta `/construction/submit` endpoint with two conflicting headers:
   - `X-Real-IP: 192.0.2.1` (forged, e.g., a known-good IP or a non-existent one)
   - `X-Forwarded-For: 10.0.0.99` (a different forged value)
2. Traefik (insecure mode) forwards both headers unchanged to the rosetta container.
3. `getClientIpAddress()` reads `X-Real-IP` first and returns `192.0.2.1`.
4. `TracingMiddleware` emits: `192.0.2.1 POST /construction/submit (200) in Xms`
5. Traefik does not emit an access log entry (200 is filtered out).
6. The audit trail records the forged IP as the transaction submitter. The real TCP source IP is unrecorded anywhere.

### Impact Explanation

The Rosetta Construction API is the path through which signed transactions are submitted to the Hiero network. The `TracingMiddleware` log is the only application-level audit record of who submitted a transaction. An attacker who can forge this record can:

- Submit malicious or spam transactions while attributing them to a legitimate IP (e.g., a known exchange or validator).
- Evade IP-based rate-limit forensics and post-incident attribution.
- Create false evidence implicating another party in the audit log.

Because Traefik suppresses 2xx access logs, there is no proxy-side record to cross-reference. The application log is the sole source of truth and it is fully attacker-controlled.

### Likelihood Explanation

- **Precondition**: The Kubernetes/Traefik deployment path, which is the production deployment path per the Helm charts. No credentials, tokens, or elevated network position are required.
- **Skill level**: Setting an HTTP header requires no technical sophistication. Any HTTP client (curl, Python requests) suffices.
- **Repeatability**: The attack is stateless and can be repeated on every request.
- **Detection**: Because the forged IP appears in the log and no Traefik access log entry exists for 2xx responses, the attack leaves no detectable artifact in the default logging configuration.

### Recommendation

1. **Remove `forwardedHeaders.insecure` from Traefik** and replace it with an explicit trusted-IP allowlist using `--entryPoints.web.forwardedHeaders.trustedIPs=<pod-CIDR>`. This ensures only Traefik itself can set forwarded headers.

2. **In `getClientIpAddress()`**, validate that the extracted IP is a syntactically valid IP address using `net.ParseIP()` and reject values that are not valid IPs:
   ```go
   if ip := net.ParseIP(strings.TrimSpace(ipAddress)); ip == nil {
       ipAddress = ""
   }
   ```

3. **Enable Traefik access logging for all status codes** (remove the `statuscodes: 400-599` filter) so that a proxy-side record always exists to cross-reference against the application log.

4. **Use `r.RemoteAddr` as the authoritative source** when the application is behind a trusted proxy that is guaranteed to set `X-Real-IP`, rather than trusting the header from any source.

### Proof of Concept

```bash
# Target: Kubernetes-deployed Rosetta endpoint via Traefik ingress
# No special privileges required

curl -X POST https://<rosetta-host>/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 203.0.113.1" \
  -H "X-Forwarded-For: 198.51.100.42" \
  -d '{"network_identifier": {...}, "signed_transaction": "..."}'

# Expected application log output:
# INFO: 203.0.113.1 POST /construction/submit (200) in 12ms
#
# Traefik access log: (no entry — 200 is filtered by statuscodes: 400-599)
#
# The real attacker IP is unrecorded. The audit log attributes the
# transaction submission to 203.0.113.1 (forged X-Real-IP).
```
