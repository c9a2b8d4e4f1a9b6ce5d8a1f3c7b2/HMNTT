### Title
Query String Bypass of `internalPaths` Check Enables Unauthenticated Log Flooding / Disk Exhaustion

### Summary
In `TracingMiddleware()`, the path used for the `internalPaths` suppression check is obtained via `request.URL.RequestURI()`, which returns the full URI including any query string. Because `internalPaths` contains only bare paths (e.g., `/health/liveness`), any request to `/health/liveness?x=1` produces a path that does not match the map, causing the request to be logged at `Info` level rather than `Debug`. An unauthenticated attacker can exploit this to force every health-check request to be written to the production log, enabling disk exhaustion across nodes.

### Finding Description
**File:** `rosetta/app/middleware/trace.go`
**Function:** `TracingMiddleware()`, lines 47 and 55.

```go
// line 47
path := request.URL.RequestURI()   // returns "/health/liveness?flood=1" — includes query string

// line 55
if internalPaths[path] {           // map contains "/health/liveness" — never matches with query string
    log.Debug(message)
} else {
    log.Info(message)              // always taken when query string is present
}
```

`internalPaths` is defined at line 19 as:
```go
var internalPaths = map[string]bool{livenessPath: true, metricsPath: true, readinessPath: true}
```
where `livenessPath = "/health/liveness"`, `readinessPath = "/health/readiness"`, `metricsPath = "/metrics"` (health.go lines 20–22, metrics.go line 16).

The failed assumption is that `RequestURI()` returns only the path component. In Go's `net/url`, `RequestURI()` is documented to return `path?query`, so any query string appended by the caller breaks the exact-match lookup.

The log message written at line 52–53 includes the full `path` value (with query string), so an attacker can also supply an arbitrarily long query string to inflate each log entry's size.

No rate limiting, authentication, or query-string stripping is present anywhere in the middleware chain visible in this package.

### Impact Explanation
In production, the log level is typically `Info` or higher; `Debug` messages are discarded. The `internalPaths` suppression exists precisely to prevent high-frequency health-check polling from filling logs. By bypassing it, an attacker causes every health-check request to be durably written to disk. The `/health/liveness` endpoint is a trivial no-op (no DB, no network call), so it can be called at very high rates. Sustained flooding fills the disk of the host, causing the rosetta process (and co-located services) to crash or become unable to write state, taking the node offline. Repeating this against 30 %+ of nodes satisfies the stated impact threshold.

### Likelihood Explanation
The attack requires zero privileges: the `/health/liveness` endpoint is publicly reachable by design (Kubernetes liveness probes, monitoring systems). The only requirement is the ability to append `?x=1` to the URL, which any HTTP client supports. The attack is trivially scriptable (`while true; do curl …; done`) and can be amplified by using a long query string value to increase per-entry log size. No exploit tooling beyond `curl` is needed.

### Recommendation
Replace `request.URL.RequestURI()` with `request.URL.Path` on line 47 so that the `internalPaths` lookup is performed against the bare path regardless of any query string:

```go
// Before (vulnerable)
path := request.URL.RequestURI()

// After (fixed)
path := request.URL.Path
```

If the full URI (including query string) is still needed in the log message, store it separately:

```go
path    := request.URL.Path
fullURI := request.URL.RequestURI()
// use path for internalPaths check, fullURI for the log message
```

Additionally, consider adding a rate-limiting middleware for all endpoints, including health checks.

### Proof of Concept
```bash
# 1. Confirm normal liveness request is suppressed (logged at Debug, invisible in production)
curl http://<rosetta-node>:<port>/health/liveness
# → no entry appears in production log (Info level)

# 2. Append a query string to bypass the internalPaths check
curl "http://<rosetta-node>:<port>/health/liveness?flood=1"
# → entry IS written to production log at Info level:
#   "192.0.2.1 GET /health/liveness?flood=1 (200) in 123µs"

# 3. Flood at high rate with a large query string to fill disk
while true; do
  curl -s "http://<rosetta-node>:<port>/health/liveness?$(python3 -c 'print("A"*4096)')" > /dev/null
done
# → log file grows unboundedly; disk exhaustion causes node failure
