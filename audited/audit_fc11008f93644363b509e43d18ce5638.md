### Title
Deleted Accounts Exposed via `entityId` Lookup in GraphQL `account()` Query

### Summary
The `account()` query in `AccountController.java` uses three different lookup paths, but only the `entityId` path calls `entityRepository.findById()` (the plain Spring Data CrudRepository method), which has no `deleted` filter. The `alias` and `evmAddress` paths use custom repository queries that explicitly include `and deleted is not true`. Any unauthenticated user can retrieve full data for a deleted account — including its `deleted: true` flag, keys, balance, and memo — simply by querying with a numeric entity ID.

### Finding Description
**Code path:**

- `AccountController.java` lines 41–44: when `entityId != null`, calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)`. [1](#0-0) 

- `EntityServiceImpl.java` line 25: `getByIdAndType` calls `entityRepository.findById(entityId.getId())` — the inherited Spring Data `CrudRepository.findById`, which issues a plain `SELECT * FROM entity WHERE id = ?` with **no deleted filter**. [2](#0-1) 

- By contrast, `getByAliasAndType` (line 30) and `getByEvmAddressAndType` (line 40) call `findByAlias` and `findByEvmAddress` respectively, both of which carry an explicit `deleted is not true` clause. [3](#0-2) 

**Root cause:** `getByIdAndType` relies on the default `CrudRepository.findById` instead of a custom query that filters `deleted`. The assumption that "type filtering is sufficient" is wrong — it does not exclude deleted entities.

**Exploit flow:**
1. Attacker knows (or enumerates) a numeric account ID, e.g. `0.0.12345`.
2. Sends a GraphQL query: `{ account(input: { entityId: { shard: 0, realm: 0, num: 12345 } }) { deleted balance key memo } }`.
3. If the account is deleted, the response returns the full entity row with `deleted: true` and all other fields populated.
4. The same account returns empty (`null`) when queried via `alias` or `evmAddress`.

**Why existing checks fail:** The only post-fetch filter in `getByIdAndType` is `.filter(e -> e.getType() == type)`, which checks entity type but not deletion status. [4](#0-3) 

### Impact Explanation
An unprivileged external user can read the full state of any deleted account — including cryptographic keys, balance at deletion time, memo, and expiration data — by supplying its numeric entity ID. This violates the intended access control boundary (deleted accounts should be invisible), and the `deleted` field in the GraphQL schema confirms the account's deletion status to the caller. Account IDs are sequential and publicly known, making enumeration trivial.

### Likelihood Explanation
No authentication or special privilege is required. The GraphQL endpoint is publicly accessible. Entity IDs are sequential integers (e.g., `0.0.1`, `0.0.2`, …), so an attacker can enumerate all deleted accounts with a simple loop. The exploit is fully repeatable and requires no special tooling beyond a standard HTTP client.

### Recommendation
Replace the plain `findById` call in `EntityServiceImpl.getByIdAndType` with a custom repository query that adds `and deleted is not true`, consistent with `findByAlias` and `findByEvmAddress`. For example, add to `EntityRepository`:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then update `getByIdAndType` in `EntityServiceImpl.java` to call `findByIdAndNotDeleted` instead of `findById`. [4](#0-3) 

### Proof of Concept
**Precondition:** An account with ID `0.0.12345` exists in the database with `deleted = true`.

**Request:**
```graphql
POST /graphql/alpha
{
  account(input: { entityId: { shard: 0, realm: 0, num: 12345 } }) {
    deleted
    balance
    key
    memo
    createdTimestamp
  }
}
```

**Expected (correct) result:** `null` — deleted account should not be returned.

**Actual result:**
```json
{
  "data": {
    "account": {
      "deleted": true,
      "balance": 0,
      "key": "<public key bytes>",
      "memo": "...",
      "createdTimestamp": "..."
    }
  }
}
```

**Contrast:** Querying the same account via `alias` or `evmAddress` returns `null`, confirming the inconsistency is specific to the `entityId` path.
