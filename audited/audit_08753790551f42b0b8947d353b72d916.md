### Title
Unbounded EVM Address DB Lookup in `getHookStorage()` Enables Resource Exhaustion via Repeated Unauthenticated Requests

### Summary
`HookServiceImpl.getHookStorage()` unconditionally calls `entityService.lookup(request.getOwnerId())` on every invocation. When the `ownerId` is an EVM address, `EntityServiceImpl.lookup()` dispatches to `entityRepository.findByEvmAddress()`, which executes a raw SQL query against the database with no result caching. The `rest-java` module has no rate limiting on the hooks endpoint, allowing any unauthenticated caller to drive an unbounded number of uncached DB queries by repeatedly targeting this endpoint with a valid EVM address.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` → `HookServiceImpl.getHookStorage()` (line 61) → `EntityServiceImpl.lookup()` (lines 31–34) → `EntityRepository.findByEvmAddress()` (line 16–17). [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — no caching:** The `rest-java` module's `EntityRepository.findByEvmAddress()` carries no `@Cacheable` annotation. Compare this to the `web3` module's `EntityRepository`, where `findByEvmAddressAndDeletedIsFalse()` is decorated with `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`. [4](#0-3) 

Every call to `getHookStorage()` with an EVM-address `ownerId` therefore issues the query `SELECT id FROM entity WHERE evm_address = ? AND deleted <> true` unconditionally.

**Root cause — no rate limiting:** The `rest-java` module has no throttle/rate-limit infrastructure for the hooks endpoint. The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` stack exists exclusively in the `web3` module. [5](#0-4) 

The documented `hiero.mirror.restJava.*` configuration properties contain no rate-limiting knobs. [6](#0-5) 

**Exploit flow:**
1. Attacker discovers (or enumerates) any valid EVM address that maps to an existing entity.
2. Attacker issues a high-frequency loop of `GET /api/v1/accounts/{evmAddress}/hooks/{hookId}/storage` requests — no authentication required, no API key, no session.
3. Each request reaches `HookServiceImpl.getHookStorage()` → `EntityServiceImpl.lookup()` → `entityRepository.findByEvmAddress()` → one DB round-trip per request.
4. Because there is no caching and no rate limiting, the DB receives one query per HTTP request at whatever rate the attacker can sustain.

### Impact Explanation
Each request forces at minimum one uncached DB query (`findByEvmAddress`) plus one or two additional queries for the actual hook storage rows. Sustained high-frequency requests from one or more clients will increase DB CPU, I/O, and connection-pool pressure. Because the `rest-java` service shares the same PostgreSQL instance as other mirror-node components, saturation of the DB connection pool or query executor threads degrades all dependent services. A modest attack rate (e.g., hundreds of requests per second from a single host) is sufficient to push DB resource consumption well above the 30% threshold relative to a 24-hour baseline.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The only precondition is knowledge of one valid EVM address, which is trivially obtained from any block explorer or by querying other public mirror-node endpoints. The attack is stateless, trivially parallelisable, and requires no special tooling beyond `curl` or `ab`. It is repeatable indefinitely.

### Recommendation
1. **Add `@Cacheable`** to `EntityRepository.findByEvmAddress()` in the `rest-java` module, mirroring the pattern already used in the `web3` module.
2. **Add per-IP or global rate limiting** to the `rest-java` hooks endpoints (e.g., via a Spring `HandlerInterceptor` backed by Bucket4j, consistent with the `web3` throttle stack, or at the ingress/load-balancer layer).
3. Consider a short-lived in-process cache (e.g., Caffeine with `expireAfterWrite=30s`) for EVM-address-to-entity-id mappings in `EntityServiceImpl.lookup()`.

### Proof of Concept
```bash
# 1. Obtain any valid EVM address (e.g., from a public mirror REST endpoint)
EVM_ADDR="0xaabbccdd..."   # replace with a real address
HOOK_ID=0

# 2. Flood the endpoint — no credentials required
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${EVM_ADDR}/hooks/${HOOK_ID}/storage" \
       -o /dev/null &
done
wait

# Each iteration triggers:
#   EntityServiceImpl.lookup() -> entityRepository.findByEvmAddress() -> DB query
# No caching absorbs the load; no rate limiter rejects the requests.
# DB query rate equals HTTP request rate.
```

public class RestJavaProperties {

    @NotNull
    @Valid

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }

    @Data
    @Validated
    public static class ResponseConfig {
        @NotNull
        @Valid
        private ResponseHeadersConfig headers = new ResponseHeadersConfig();
    }

    @Data
    @Validated

        @NotNull
        private Map<String, Map<String, String>> path = new HashMap<>();

            return apiPath == null ? defaults : path.getOrDefault(apiPath, defaults);
