### Title
Error Counter Reset via Bare SUCCESS Response Prevents Block Node Deactivation

### Summary
In `BlockNode.streamBlocks()`, the `errors` counter is unconditionally reset to `0` after any non-exception response, including a `STATUS=SUCCESS` response that delivers zero block data. A malicious block node operator can exploit this by sending exactly `maxSubscribeAttempts-1` error responses (`NOT_AVAILABLE`) followed by one bare `SUCCESS` response per cycle, keeping the error counter perpetually below the deactivation threshold and preventing the node from ever being marked inactive while delivering no useful blocks.

### Finding Description

**Exact code path:**

In `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, the `streamBlocks()` method processes responses in a while loop:

```java
// lines 138ŌĆō158

    switch (response.getResponseCase()) {
        case BLOCK_ITEMS -> assembler.onBlockItemSet(response.getBlockItems());
        case END_OF_BLOCK -> assembler.onEndOfBlock(response.getEndOfBlock());
        case STATUS -> {
            var status = response.getStatus();
            if (status == SubscribeStreamResponse.Code.SUCCESS) {
                serverSuccess = true;
                break;          // exits switch, falls through to errors.set(0)
            }
            throw new BlockStreamException(...);
        }
        ...
    }
    errors.set(0);              // ŌåÉ resets counter for ALL non-exception paths, including SUCCESS
}
``` [1](#0-0) 

When `STATUS=SUCCESS` is received, the `break` exits the `switch`, execution falls to `errors.set(0)` at line 157, and then the while condition `!serverSuccess` terminates the loop. The counter is reset even though **no block data was delivered**.

**Root cause:** `errors.set(0)` is placed unconditionally after the switch statement rather than only on paths that confirm actual block data delivery. A `SUCCESS` status with no preceding `BLOCK_ITEMS`/`END_OF_BLOCK` is treated identically to a fully successful block stream.

**Deactivation logic (lines 196ŌĆō206):**

```java
private void onError() {
    errorsMetric.increment();
    if (errors.incrementAndGet() >= streamProperties.getMaxSubscribeAttempts()) {
        active = false;
        errors.set(0);
        ...
    }
}
``` [2](#0-1) 

The threshold check is bypassed because the counter never reaches `maxSubscribeAttempts` (default: 3) ŌĆö it is reset to 0 on the interleaved SUCCESS call before the third error.

**The existing test at lines 179ŌĆō185 explicitly confirms this behavior is present and accepted:** [3](#0-2) 

The comment reads *"when stream succeeds, the node is active and the error count is reset"* ŌĆö confirming the counter reset on SUCCESS is intentional but does not account for the adversarial case.

### Impact Explanation
With `maxSubscribeAttempts=3` (default), the attacker forces the importer into a perpetual cycle of 2 wasted failed subscribe attempts followed by 1 empty-success attempt, delivering **zero blocks** per cycle. The node is never deactivated and never falls back to an alternative block node or file source. Block ingestion stalls indefinitely for as long as the malicious node is the selected source. If only one block node is configured, the entire mirror node stops ingesting blocks. If multiple nodes exist, the importer still wastes time on the malicious node before trying others (depending on priority ordering in `BlockNodeSubscriber.getNode()`). [4](#0-3) 

### Likelihood Explanation
The precondition is operating a block node that the mirror node importer is configured to connect to ŌĆö no privileged access to the importer or its infrastructure is required. Block nodes are external, third-party-operated infrastructure. The attack requires only the ability to craft gRPC `SubscribeStreamResponse` messages, which is trivially achievable by any block node operator. The pattern is simple, deterministic, and requires no timing precision: send `N-1` `NOT_AVAILABLE` responses then one `SUCCESS` response, repeat. The `maxSubscribeAttempts` value is static configuration, easily determined by probing.

### Recommendation
Move `errors.set(0)` so it is only called when at least one complete block was successfully delivered to the consumer, not on every non-exception response. Specifically:

1. Track whether `onBlockStream` was actually invoked (i.e., a complete block was assembled and delivered).
2. Only reset `errors` after confirmed block delivery, not on bare `SUCCESS` or empty streams.
3. Alternatively, treat a `SUCCESS` response that delivered no block data as a soft error (or at minimum, do not reset the counter), so repeated empty-success responses still accumulate toward deactivation.

### Proof of Concept

**Setup:** Configure the mirror node importer with a single block node pointing to a server under attacker control. Set `maxSubscribeAttempts=3` (default).

**Steps:**

1. Importer calls `streamBlocks(blockN, ...)`.
2. Attacker's server responds with `STATUS=NOT_AVAILABLE` ŌåÆ `onError()` called ŌåÆ `errors=1`.
3. Importer retries. Attacker's server responds with `STATUS=NOT_AVAILABLE` ŌåÆ `onError()` called ŌåÆ `errors=2`.
4. Importer retries. Attacker's server responds with `STATUS=SUCCESS` (no block items sent) ŌåÆ `errors.set(0)` called ŌåÆ `errors=0`. `streamBlocks()` returns normally. No block delivered.
5. Importer calls `streamBlocks(blockN, ...)` again (same block, still not ingested).
6. Go to step 2. Repeat indefinitely.

**Observed result:** `errors` oscillates between 0 and 2, never reaching 3. `active` remains `true`. The node is never deactivated. Block `N` is never ingested. The importer is stuck.
