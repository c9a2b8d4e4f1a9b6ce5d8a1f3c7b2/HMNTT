### Title
Unauthenticated Bandwidth/Memory Amplification via Unbounded `logs_bloom` in `GET /api/v1/blocks` List Endpoint

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` unconditionally includes the `logs_bloom` field (512 bytes of binary data, serialized to a 1026-character hex string) for every block in the result set. With a hard cap of 100 rows per unauthenticated request and no rate limiting on this endpoint in the Node.js REST service, any unprivileged external user can drive sustained high-volume concurrent requests that exhaust network bandwidth and Node.js heap memory, producing a denial-of-service condition.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` line 12 routes `GET /` to `BlockController.getBlocks`. [1](#0-0) 

`getBlocks()` in `rest/controllers/blockController.js` lines 101â€“112 fetches all blocks up to the configured limit and maps each through `BlockViewModel`: [2](#0-1) 

`BlockViewModel` in `rest/viewmodel/blockViewModel.js` line 27 always serializes `logs_bloom` as a padded 512-byte hex string (`0x` + 1024 hex chars = 1026 chars) for every block in the list: [3](#0-2) 

**Limit enforcement:**

`extractLimitFromFilters` in `rest/controllers/blockController.js` lines 57â€“61 caps unauthenticated requests at `responseLimit.max`: [4](#0-3) 

The default `max` is **100** rows (confirmed in `rest/__tests__/config.test.js` line 325): [5](#0-4) 

**No rate limiting on the blocks endpoint:**

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module for contract-call endpoints. There is no equivalent middleware applied to the Node.js REST `/api/v1/blocks` route. [6](#0-5) 

**Root cause:** The design assumes the row-count cap (100) is sufficient to bound response size. It is not â€” each row carries a mandatory 1026-byte `logs_bloom` field, making the per-request payload up to ~200 KB of bloom data alone, with no field-projection option and no request-rate control for unauthenticated callers.

### Impact Explanation

At 100 blocks Ă— 1026 bytes of hex bloom data plus other fields (~2 KB/block total), each max-limit response is ~200 KB. An attacker driving 500 concurrent persistent connections produces ~100 MB/s of outbound traffic and holds ~100 MB of in-flight Node.js heap simultaneously. Node.js's single-threaded event loop serializes all JSON in-process; under this load the GC pressure and heap growth cause latency spikes, OOM kills, or event-loop starvation â€” effectively partitioning the REST API from legitimate users. No credentials, API keys, or special network position are required.

### Likelihood Explanation

The endpoint is public, unauthenticated, and documented in the OpenAPI spec (`/api/v1/blocks`). [7](#0-6) 

A single attacker with a modest botnet or even a single high-bandwidth host can sustain the required concurrency. The attack is trivially scriptable (`curl`/`ab`/`wrk`) and fully repeatable with no side effects that would expose the attacker.

### Recommendation

1. **Add per-IP rate limiting** to the Node.js REST service for all list endpoints, including `/api/v1/blocks`, using middleware such as `express-rate-limit`.
2. **Reduce the effective max limit for `logs_bloom`-bearing endpoints** or support a `fields` projection parameter so callers can opt out of large binary fields.
3. **Stream or paginate large responses** rather than buffering the full result set in Node.js heap before sending.
4. **Fix the limit fallback logic** in `extractLimitFromFilters`: when `limit.value > maxLimit`, cap at `maxLimit` rather than silently falling back to `defaultLimit`, to make the behavior predictable and auditable. [4](#0-3) 

### Proof of Concept

```bash
# No authentication required. Each response carries up to ~200 KB of bloom data.
# Run from any host with internet access to the mirror node REST API.

for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/blocks?limit=100" \
       -o /dev/null &
done
wait
```

Preconditions: network access to port 5551 (default REST port), no credentials.
Trigger: 500 concurrent unauthenticated `GET /api/v1/blocks?limit=100` requests.
Result: Node.js heap exhaustion and/or bandwidth saturation; legitimate API requests time out or receive 503/connection-refused errors.
