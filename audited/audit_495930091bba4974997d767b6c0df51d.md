### Title
32-bit Hash Collision in `CACHE_NAME_ALIAS` Cache Key Enables Cache Poisoning via Unprivileged Alias Lookup

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole cache key in `CACHE_NAME_ALIAS`. Because the key space is only 2³² values, two distinct byte arrays can produce the same key. An unprivileged attacker who creates a Hedera account whose alias/EVM-address collides with a fee-collecting entity's hash can poison the cache so that subsequent lookups for the fee-collecting entity return the attacker's entity instead, causing fee parameters to be read from the wrong account.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode` returns a Java `int` (32 bits): [1](#0-0) 

This `int` is used verbatim as the cache key for `CACHE_NAME_ALIAS`: [2](#0-1) 

The cache manager for this cache is `CACHE_MANAGER_ENTITY`, which registers `CACHE_NAME_ALIAS` as one of its named caches: [3](#0-2) 

The entity cache TTL is 1 second: [4](#0-3) 

The cached result feeds directly into alias resolution in `AliasesReadableKVState` and `AccountReadableKVState`, which resolve account IDs used in EVM fee calculations: [5](#0-4) [6](#0-5) 

**Root cause:** `Arrays.hashCode(byte[])` is a non-cryptographic 32-bit polynomial hash (`result = 31*result + b`). Its output space is only ~4 billion values. Two distinct byte arrays `A` and `B` can satisfy `Arrays.hashCode(A) == Arrays.hashCode(B)`. Spring `@Cacheable` stores the result under the integer key with no secondary equality check on the actual byte array, so a collision causes the wrong entity to be returned on a cache hit.

**Failed assumption:** The design assumes that `Arrays.hashCode` produces a unique key per distinct alias byte array. This is false for a 32-bit hash over variable-length byte arrays.

**Exploit flow:**

1. Attacker identifies target fee-collecting entity with alias `A` (e.g., `feeCollectionAccount` resolved via `AliasesReadableKVState`).
2. Attacker computes `T = Arrays.hashCode(A)` (public algorithm, no secret needed).
3. For an ED25519 alias (32 bytes), the attacker analytically solves for a byte array `B` such that `Arrays.hashCode(B) == T`. Because the polynomial hash is linear in the last byte, fixing the first 31 bytes and solving for byte 32 is a single modular arithmetic step. The attacker creates a Hedera account with alias `B` (permissionless, costs only HBAR).
4. Every time the 1-second cache TTL expires, the attacker sends a web3 `eth_call` or equivalent that triggers `findByEvmAddressOrAliasAndDeletedIsFalse(B)`. This is a cache miss → DB returns attacker's entity → stored under key `T`.
5. The next legitimate lookup for alias `A` hits the cache under key `T` and receives the attacker's entity (with lower fee parameters) instead of the real fee-collecting entity.

**Why existing checks are insufficient:**

- `unless = "#result == null"`: `Optional.of(attackerEntity)` is not null, so the attacker's result is unconditionally cached. `Optional.empty()` is also not null, so even a non-existent alias poisons the cache with an empty result. [7](#0-6) 
- There is no post-cache verification that the returned entity's actual alias/evm_address matches the queried byte array.
- The cache has no collision-detection mechanism (e.g., storing the original key alongside the value).

### Impact Explanation
When the wrong entity is returned for a fee-collecting account lookup, the EVM execution layer uses the attacker's account data (with lower or zero fee parameters) to compute and route fees. This causes fees to be undercharged or misdirected outside of design parameters. The impact is bounded by the 1-second TTL — the attacker must continuously re-poison — but within each poisoning window every request that resolves the target alias is affected.

### Likelihood Explanation
The attack requires no privileges: creating a Hedera account is permissionless (costs HBAR). For ED25519 aliases, finding a colliding byte array is a single arithmetic operation (no brute force needed). For EVM addresses (Keccak-256-derived), brute force over key pairs is required (~2³² attempts), which is expensive but feasible for a motivated attacker. Sustaining the attack requires sending one request per second, which is trivially automated. The attacker does not need any special network position or credentials.

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the full byte array (or its hex/Base64 encoding) so that the cache key uniquely identifies the alias:

```java
// Instead of:
key = "@spelHelper.hashCode(#alias)"

// Use:
key = "T(java.util.Arrays).toString(#alias)"
// or store as hex:
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```

Alternatively, add a post-cache equality check: after retrieving a cached entity, verify that its stored `alias` or `evmAddress` field actually equals the queried byte array, and on mismatch evict and re-query.

### Proof of Concept

```java
// Step 1: Identify target alias A (e.g., fee collection account's alias bytes)
byte[] A = feeCollectionEntity.getAlias(); // e.g., 32-byte ED25519 key
int targetHash = Arrays.hashCode(A);       // e.g., 0x12345678

// Step 2: Analytically construct colliding alias B (32 bytes)
// Fix first 31 bytes arbitrarily, solve for byte[31]:
// Arrays.hashCode prefix after 31 bytes = P
// P * 31 + B[31] ≡ targetHash (mod 2^32)
// B[31] = (targetHash - P * 31) mod 256  (adjust for signed byte)
byte[] B = constructCollidingAlias(A); // trivial single-step computation

// Step 3: Create Hedera account with alias B (one-time setup, costs HBAR)
// hederaClient.createAccount(alias=B, initialBalance=1_hbar);

// Step 4: Continuously poison the cache (once per second)
while (true) {
    // Trigger web3 API call that resolves alias B
    // e.g., eth_call to a contract that reads account(B)
    web3Api.call("eth_call", buildCallWithAlias(B));
    Thread.sleep(900); // re-poison before TTL expires
}

// Step 5: Observe that lookups for alias A now return attacker's entity
// with lower fee parameters, causing undercharging
```
