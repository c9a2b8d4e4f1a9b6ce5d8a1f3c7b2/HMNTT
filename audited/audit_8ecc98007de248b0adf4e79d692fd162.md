### Title
Unbounded Concurrent Unthrottled DB Polling via Safety-Check Trigger in `PollingTopicMessageRetriever.retrieve()`

### Summary
Any unauthenticated gRPC client can open an unlimited number of topic subscriptions. For each subscription without an `endTime`, the hardcoded 1-second safety check in `TopicMessageServiceImpl` unconditionally fires and invokes `retrieve(filter, false)` (unthrottled mode), which issues up to 12 DB polls at 20 ms intervals fetching up to 5,000 rows each. With no global subscription cap and only a per-connection call limit, an attacker opening many connections can simultaneously trigger hundreds of unthrottled DB polling bursts, exhausting database I/O and causing a non-network DoS.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` constructs a safety-check `Flux` that fires unconditionally after 1 second for any subscription where `topicContext.isComplete()` returns `false`:

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
    .filter(_ -> !topicContext.isComplete())
    .flatMapMany(_ -> missingMessages(topicContext, null))
    .subscribeOn(Schedulers.boundedElastic());
```

`TopicContext.isComplete()` returns `false` whenever `filter.getEndTime() == null` — which is the default for any subscription that does not set an end time:

```java
// TopicMessageServiceImpl.java lines 203-215
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;   // always false → safety check always fires
    }
    ...
}
```

When the safety check fires, `missingMessages(topicContext, null)` is called, which unconditionally invokes the **unthrottled** retriever:

```java
// TopicMessageServiceImpl.java line 149
return topicMessageRetriever.retrieve(gapFilter, false);  // throttled=false
```

Inside `PollingTopicMessageRetriever.retrieve()`, the unthrottled `PollingContext` is configured with:

```java
// RetrieverProperties.java lines 38-46
private int maxPageSize = 5000;
private long maxPolls = 12;
private Duration pollingFrequency = Duration.ofMillis(20);
```

This drives the repeat loop at line 52–55:

```java
.repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
    .jitter(0.1)
    .withFixedDelay(context.getFrequency())   // 20 ms
    .withScheduler(scheduler))
```

In unthrottled mode, `PollingContext.isComplete()` only returns `true` when `limitHit` (line 128), which is `false` when no limit is set. So all 12 polls execute, each fetching up to 5,000 rows from the DB via `topicMessageRepository.findByFilter(newFilter)` (line 78).

**Root cause:** The safety check fires unconditionally for every no-end-time subscription, and there is no global limit on concurrent subscriptions. The only guard — `maxConcurrentCallsPerConnection = 5` — is per-connection only, with no cap on the number of connections.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` (line 33 of `GrpcConfiguration.java`): limits calls per TCP connection, not total connections. An attacker opens N connections → N × 5 concurrent subscriptions.
- `numRepeats = 12`: bounds each individual unthrottled retriever, but does not prevent many running concurrently.
- `timeout = 60s` (line 59 of `PollingTopicMessageRetriever.java`): only terminates subscriptions receiving no data for 60 s; the burst completes in ~240 ms, well before timeout.
- `subscriberCount` (line 48 of `TopicMessageServiceImpl.java`): a metrics gauge only — never enforced as a cap.
- No authentication interceptor exists on the gRPC endpoint; `GrpcInterceptor` only sets endpoint context.

### Impact Explanation
Each subscription with no end time triggers one unthrottled retrieval burst: 12 polls × 5,000 rows = up to 60,000 rows fetched in ~240 ms. With 100 attacker connections × 5 calls each = 500 concurrent subscriptions → 500 × 12 = 6,000 simultaneous DB queries, each reading up to 5,000 rows. This saturates the database connection pool and disk I/O, making the mirror node's DB unavailable to all legitimate users (historical retrieval, live listeners, REST API). The mirror node is a critical read path for the Hedera/Hiero network; its unavailability degrades the ability of applications to verify consensus state.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no on-chain transactions. Any client with network access to port 5600 can open gRPC connections and subscribe to any existing topic with `startTime=0` and no `endTime`. The gRPC protocol is well-documented and standard tooling (`grpcurl`, Hedera SDK) can be used directly. The 1-second delay before the safety check fires is deterministic and reliable. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., 500), returning `RESOURCE_EXHAUSTED`.
2. **Gate the safety check on a meaningful condition**: only fire it when the throttled retriever has actually completed and the live listener has started, not unconditionally after 1 second.
3. **Add per-IP or per-connection subscription limits** via a gRPC `ServerInterceptor` that tracks active streams per remote address.
4. **Require an end time or impose a maximum subscription duration** for unthrottled retrieval paths, so `isComplete()` can return `true` and terminate the unthrottled burst early.
5. **Add a DB connection pool limit guard**: reject new DB queries when pool utilization exceeds a threshold, rather than queuing unboundedly.

### Proof of Concept

```bash
# Prerequisites: grpcurl installed, mirror node at grpc-host:5600,
# topic 0.0.12345 exists with historical messages

# Step 1: Open 100 connections, each with 5 concurrent subscriptions (500 total)
# Each subscription has startTime=0 and no endTime → safety check fires after 1s

for i in $(seq 1 100); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0}}' \
      grpc-host:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 2: After ~1 second, each of the 500 subscriptions triggers the safety check,
# invoking retrieve(filter, false) → 12 polls × 5000 rows = 60,000 rows per subscription
