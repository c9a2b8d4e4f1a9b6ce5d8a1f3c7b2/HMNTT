### Title
Unbounded Per-Subscription DB Polling Causes JVM Heap Exhaustion (DoS) in PollingTopicListener

### Summary
When the gRPC listener is configured with `type=POLL`, each subscription creates an independent polling loop that queries the database for up to `maxPageSize=5000` rows every 500 ms. Because `limit=0` is treated as "no limit" and there is no global cap on concurrent subscriptions, an unprivileged attacker opening many connections can force the JVM to load arbitrarily large volumes of `TopicMessage` objects into heap memory, leading to `OutOfMemoryError` and a complete service crash.

### Finding Description

**Root cause — `limit=0` bypasses the page-size guard:**

In `TopicMessageFilter.java` line 39-41, `hasLimit()` returns `true` only when `limit > 0`: [1](#0-0) 

In `PollingTopicListener.poll()` lines 54-57, when `hasLimit()` is false the local `limit` variable is set to `Integer.MAX_VALUE`, and `pageSize` becomes `Math.min(Integer.MAX_VALUE, 5000) = 5000`: [2](#0-1) 

That `pageSize` is passed directly to the repository query, so every poll for every unlimited subscription fetches up to 5 000 `TopicMessage` rows and materialises them in heap via `Flux.fromStream()`: [3](#0-2) 

**Polling is per-subscription and runs forever:**

`listen()` schedules `poll()` to repeat with `RepeatSpec.times(Long.MAX_VALUE)` at the configured interval (default 500 ms). Without an `endTime` the subscription never terminates: [4](#0-3) 

**No global subscription limit:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge — it is never compared against a maximum and never rejects new subscriptions: [5](#0-4) 

**The only concurrency guard is per-connection, not global:**

`maxConcurrentCallsPerConnection` (default 5) is applied per TCP connection by Netty: [6](#0-5) [7](#0-6) 

An attacker opens `C` connections and gets `5 × C` concurrent subscriptions. There is no IP-level connection cap and no gRPC-layer rate limiter (the `ThrottleConfiguration` / `ThrottleManagerImpl` exists only in the `web3` module, not in `grpc`).

### Impact Explanation

With `N` active unlimited subscriptions and a 500 ms poll interval, the service issues `2N` DB queries per second, each returning up to 5 000 rows. All rows are held in heap simultaneously across the reactive pipeline. At modest scale (e.g., 200 connections × 5 calls = 1 000 subscriptions), the JVM must hold up to 5 000 000 `TopicMessage` objects in memory every 500 ms. This exhausts the heap, triggers `OutOfMemoryError`, and crashes the gRPC service — denying topic-subscription service to all legitimate users. The database is also saturated with concurrent heavy-read queries.

### Likelihood Explanation

The attack requires the operator to have set `hiero.mirror.grpc.listener.type=POLL` (non-default; default is `REDIS`). When that configuration is active, exploitation requires only a standard gRPC client, no credentials, and no special knowledge beyond the public `subscribeTopic` API. The attacker needs to open many TCP connections (trivially done from a single host or botnet) and issue `SubscribeTopic` RPCs with `limit=0` and no `endTime`. The attack is repeatable and self-sustaining until the service crashes.

### Recommendation

1. **Enforce a global subscription ceiling**: Add a configurable `maxSubscribers` property to `GrpcProperties` and reject new subscriptions in `TopicMessageServiceImpl.subscribeTopic()` when the ceiling is reached.
2. **Enforce a minimum positive limit or cap unlimited subscriptions**: In `PollingTopicListener.poll()`, treat `limit=0` as a bounded value (e.g., cap at `maxPageSize`) rather than `Integer.MAX_VALUE`.
3. **Add per-IP connection rate limiting** at the Netty/gRPC layer (e.g., via `maxConnectionsPerIp` or an external load-balancer policy).
4. **Apply a gRPC-level request throttle** analogous to the `ThrottleManagerImpl` already present in the `web3` module.

### Proof of Concept

```python
# Requires: grpcio, the mirror-node proto stubs, listener.type=POLL configured
import grpc, threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = ...  # any valid topic

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        # limit omitted (defaults to 0 = unlimited)
        # consensusEndTime omitted (no end time)
    )
    # Open 5 concurrent streaming calls on this connection (maxConcurrentCallsPerConnection)
    streams = [stub.subscribeTopic(req) for _ in range(5)]
    for s in streams:
        for _ in s:   # consume to keep stream alive
            pass

# Open hundreds of connections from one or more hosts
threads = [threading.Thread(target=flood) for _ in range(500)]
for t in threads: t.start()
# Result: 2500 subscriptions × 5000 rows × 2/s → JVM OOM within seconds
```
