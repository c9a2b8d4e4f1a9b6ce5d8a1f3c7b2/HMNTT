### Title
Missing Consensus-Node Membership Validation for User-Supplied `node_account_id` in `/construction/payloads`

### Summary
`getTransactionNodeAccountId()` accepts any syntactically valid Hiero account ID string from user-supplied metadata without verifying it belongs to a known consensus node. This allows an unprivileged caller to embed an arbitrary account ID as the transaction's target node, causing `hiero.TransactionSetNodeAccountIDs()` to succeed while the resulting transaction is addressed to a node that cannot process it, guaranteeing confirmation failure for every transaction built this way.

### Finding Description
**Code path:**

- `ConstructionPayloads` (`construction_service.go:222`) calls `c.getTransactionNodeAccountId(request.Metadata)`, where `request.Metadata` is entirely attacker-controlled.
- `getTransactionNodeAccountId` (`construction_service.go:370–391`) extracts the `node_account_id` string and passes it to `hiero.AccountIDFromString(str)` (`line 383`). The only rejection criterion is a parse error — there is no membership check against `c.sdkClient.GetNetwork()`.
- The parsed ID is forwarded to `transactionSetNodeAccountId` (`construction_service.go:717–725`), which calls `hiero.TransactionSetNodeAccountIDs(transaction, []hiero.AccountID{nodeAccountId})`. The SDK call succeeds for any well-formed account ID.
- The frozen transaction bytes are returned to the caller, who signs and submits them via `ConstructionSubmit` (`construction_service.go:355`), which executes the transaction against the embedded node ID.

**Root cause / failed assumption:** The server assumes the client will echo back the `node_account_id` that was generated by `getRandomNodeAccountId()` (`construction_service.go:511–537`), which carefully selects only from `c.sdkClient.GetNetwork()`. That assumption is never enforced; `getTransactionNodeAccountId` performs only a format check, not a membership check.

**Exploit flow:**
1. Attacker calls `/construction/metadata` normally (or skips it entirely).
2. Attacker calls `/construction/payloads` with `metadata["node_account_id"] = "0.0.9999"` (valid format, non-existent consensus node).
3. Server parses the ID, calls `TransactionSetNodeAccountIDs` — succeeds.
4. Attacker receives unsigned transaction bytes, signs them, calls `/construction/submit`.
5. SDK attempts to route the transaction to node `0.0.9999`; no such consensus node exists; the transaction times out and is never confirmed.

**Why existing checks fail:** The only guard is `hiero.AccountIDFromString` returning an error, which it does not for any well-formed `shard.realm.num` string. There is no allowlist comparison against `c.sdkClient.GetNetwork()`.

### Impact Explanation
Every transaction constructed through `/construction/payloads` with a spoofed `node_account_id` will be unconfirmable. Because the node ID is frozen into the signed transaction bytes before signing, the signer cannot correct it post-hoc. An attacker who controls the metadata (e.g., a relay or a client that modifies the metadata returned by `/construction/metadata` before forwarding it to `/construction/payloads`) can silently poison all transactions flowing through that path, causing systematic confirmation failure without any server-side error at the payloads stage.

### Likelihood Explanation
The Rosetta Construction API is a stateless, unauthenticated HTTP API by design. Any network-reachable client can call `/construction/payloads` directly with arbitrary metadata — no credentials, no prior state, no privileged access required. The attack requires only knowledge of the API schema (publicly documented by Coinbase Rosetta) and the ability to send an HTTP POST request.

### Recommendation
In `getTransactionNodeAccountId`, after parsing the account ID, validate it against the set of known consensus nodes:

```go
knownNodes := c.sdkClient.GetNetwork()
isKnown := false
for _, knownId := range knownNodes {
    if knownId == nodeAccountId {
        isKnown = true
        break
    }
}
if !isKnown {
    log.Errorf("node_account_id %s is not a known consensus node", str)
    return emptyAccountId, errors.ErrInvalidAccount
}
```

This mirrors the logic already used in `getRandomNodeAccountId` and closes the gap between how the server selects nodes and how it validates client-supplied nodes.

### Proof of Concept
```
# 1. Build a payloads request with a non-consensus node_account_id
POST /construction/payloads
{
  "network_identifier": { "blockchain": "Hiero", "network": "mainnet" },
  "operations": [ /* valid crypto transfer ops */ ],
  "metadata": {
    "node_account_id": "0.0.9999",   // <-- non-consensus node, valid format
    "valid_start_nanos": "1700000000000000000"
  }
}

# Server responds 200 OK with unsigned_transaction bytes — no error.

# 2. Sign the returned bytes (any valid Ed25519 key for the payer account).

# 3. Submit
POST /construction/submit
{ "signed_transaction": "<hex-encoded signed tx>" }

# Result: SDK routes to node 0.0.9999, connection fails/times out,
# transaction is never submitted to the network, confirmation never occurs.
# Server returns ErrTransactionSubmissionFailed — but the damage is done:
# the transaction window expires and the tx is permanently unconfirmable.
```
