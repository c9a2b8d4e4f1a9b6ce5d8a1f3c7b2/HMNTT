### Title
Cache Key Pollution via Unnormalized Timestamp Parameters Enables Cache Bypass and Redundant DB Queries

### Summary
The `cacheKeyGenerator` function in `rest/middleware/responseCacheHandler.js` hashes `req.originalUrl` directly without normalizing timestamp query parameter values. Because `isValidTimestampParam` accepts both `1234567890` and `1234567890.000000000` as valid (and `parseTimestampParam` converts both to the identical nanosecond integer for DB queries), an unprivileged attacker can submit up to 10 lexically distinct but semantically equivalent timestamp strings per value, each producing a different Redis cache key and forcing a fresh database query on every variant.

### Finding Description

**Code path:**

`rest/middleware/responseCacheHandler.js` line 151–153 — `cacheKeyGenerator`:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The key is an MD5 of the raw, unnormalized `req.originalUrl`. The code comment on line 149 explicitly acknowledges the gap: *"In the future, this will utilize Edwin's request normalizer (9113)."*

`rest/utils.js` line 151–153 — `isValidTimestampParam`:
```js
const isValidTimestampParam = (timestamp) => {
  return /^\d{1,10}$/.test(timestamp) || /^\d{1,10}\.\d{1,9}$/.test(timestamp);
};
```
Both `1234567890` and `1234567890.000000000` (and all 8 intermediate forms `1234567890.0` … `1234567890.00000000`) pass validation.

`rest/utils.js` line 490–507 — `parseTimestampParam`:
```js
nanos = `${nanos}000000000`.substring(0, 9);
return `${seconds}${nanos}`;
```
`parseTimestampParam('1234567890')` → `'1234567890000000000'`
`parseTimestampParam('1234567890.000000000')` → `'1234567890000000000'`

Both produce the identical SQL bind parameter, so the DB returns identical results — but the cache lookup already happened against different keys.

`rest/middleware/requestNormalizer.js` line 23 explicitly lists `filterKeys.TIMESTAMP` in `NON_SORTED_PARAMS`, meaning the normalizer deliberately does not sort or canonicalize timestamp *values*, only their order among multiple parameters.

**Root cause:** The cache key is derived from the raw URL string before any semantic normalization of timestamp values. The normalizer that exists does not canonicalize the decimal representation of a timestamp (e.g., stripping trailing zeros), so `?timestamp=1234567890` and `?timestamp=1234567890.000000000` hash to different MD5 values and are stored as independent Redis entries.

### Impact Explanation

An attacker can force up to 10 distinct cache misses per unique timestamp value (the bare integer form plus 9 forms with 1–9 decimal digits of zeros). For any endpoint that accepts a `timestamp` filter — `/api/v1/transactions`, `/api/v1/contracts/results/logs`, `/api/v1/tokens/{id}`, etc. — the attacker multiplies the effective DB query rate by up to 10× per timestamp parameter with zero cost and no credentials. Combined with other orthogonal parameter variations (limit, order, account.id), the amplification compounds. Redis cache entries for all variants are populated but never reused across variants, wasting memory and defeating the purpose of the cache layer. There is no data integrity impact; all variants return correct results.

### Likelihood Explanation

No authentication or special knowledge is required. The attack is trivially scriptable: iterate `timestamp=T`, `timestamp=T.0`, `timestamp=T.00`, …, `timestamp=T.000000000` in a loop. It is repeatable indefinitely and detectable only through anomalous cache-miss rates. Any public mirror node deployment is exposed.

### Recommendation

Normalize the timestamp value inside `cacheKeyGenerator` (or upstream before the key is computed) by canonicalizing the decimal representation: strip trailing zeros from the fractional part and remove the decimal point if the fractional part becomes empty. Alternatively, integrate the already-planned request normalizer (issue 9113) and extend it to canonicalize timestamp values before the cache key is derived. A minimal fix:

```js
const normalizeTimestampValue = (v) => {
  // "1234567890.000000000" -> "1234567890", "1234567890.0001" -> "1234567890.0001"
  return v.replace(/^(\d+)\.?0*$/, '$1').replace(/(\.\d*?)0+$/, '$1').replace(/\.$/, '');
};
```

Apply this to all `timestamp` query parameter values before constructing the URL string that is hashed.

### Proof of Concept

```
# Step 1 – prime the cache with the bare-integer form
curl "https://<mirror-node>/api/v1/transactions?timestamp=1234567890&limit=1"
# Response served from DB; cached under key = MD5("/api/v1/transactions?timestamp=1234567890&limit=1")

# Step 2 – bypass the cache with trailing zeros (9 variants)
for z in 0 00 000 0000 00000 000000 0000000 00000000 000000000; do
  curl "https://<mirror-node>/api/v1/transactions?timestamp=1234567890.${z}&limit=1"
  # Each request: cache MISS → fresh DB query → new Redis entry
done

# Result: 10 DB queries executed for semantically identical data;
# 10 independent Redis entries consuming memory, none reused across variants.

const parseTimestampParam = (timestampParam) => {
  // Expect timestamp input as (a) just seconds,
  // (b) seconds.mmm (3-digit milliseconds),
  // or (c) seconds.nnnnnnnnn (9-digit nanoseconds)
  // Convert all of these formats to (seconds * 10^9 + nanoseconds) format,
  // after validating that all characters are digits
  if (!timestampParam) {
    return '';
  }
  const tsSplit = timestampParam.split('.');
  if (tsSplit.length <= 0 || tsSplit.length > 2) {
    return '';
  }
  const seconds = /^(\d)+$/.test(tsSplit[0]) ? tsSplit[0] : 0;
  let nanos = tsSplit.length === 2 && /^(\d)+$/.test(tsSplit[1]) ? tsSplit[1] : 0;
  nanos = `${nanos}000000000`.substring(0, 9);
  return `${seconds}${nanos}`;
};
```
