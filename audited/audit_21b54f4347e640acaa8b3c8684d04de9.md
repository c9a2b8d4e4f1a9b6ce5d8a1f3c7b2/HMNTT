### Title
Unauthenticated CPU/Event-Loop Saturation via Unrate-Limited Metrics Endpoint Authentication

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` (lines 137–144) is invoked on every request to the metrics path with no rate limiting, connection throttling, or IP-based protection anywhere in the Node.js REST API stack. An unprivileged external attacker can flood the publicly reachable `/swagger/metrics/` endpoint with crafted `Authorization` headers, causing `basicAuth(req)` header parsing and two `tsscmp()` constant-time comparisons to execute at unbounded frequency, saturating Node.js's single-threaded event loop and starving legitimate request processing — including transaction gossip-related REST calls.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144:
```javascript
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
``` [1](#0-0) 

This is called unconditionally for every request matching `metricsPath` at line 157:
```javascript
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) { ... }
``` [2](#0-1) 

The default metrics path is `/swagger/metrics/` (configurable via `uriPath`), and `authentication` defaults to `true` per documentation. [3](#0-2) 

**Root cause — failed assumption:** The code assumes the metrics endpoint will only be accessed by authorized scrapers (e.g., Prometheus). There is no rate limiter, no connection cap, no IP allowlist, and no request-count guard anywhere in the Node.js middleware chain for this path.

`server.js` shows the full middleware stack: `httpContext` → `requestLogger` → `authHandler` → `metricsHandler`. None of these apply rate limiting to the metrics path. [4](#0-3) 

The only rate limiting in the repository lives in the Java `web3` module (`ThrottleManagerImpl`, `ThrottleConfiguration`) and is entirely separate from the Node.js REST API. [5](#0-4) 

**Exploit flow:**

1. Attacker discovers the metrics endpoint (default: `GET /swagger/metrics/`).
2. Attacker sends thousands of concurrent HTTP requests with a crafted `Authorization: Basic <random_base64>` header.
3. For each request, Node.js synchronously executes:
   - `basicAuth(req)` — parses and base64-decodes the `Authorization` header
   - `tsscmp(credentials.name, username)` — constant-time XOR loop over every character
   - `tsscmp(credentials.pass, password)` — second constant-time XOR loop
4. Because Node.js is single-threaded, these synchronous CPU operations on every in-flight request accumulate in the event loop microtask queue.
5. The event loop is starved; legitimate REST API requests (including those serving transaction gossip data) queue behind the flood and experience severe latency or timeout.

**Why `tsscmp()` amplifies the problem:** `tsscmp` is intentionally non-short-circuiting — it always iterates `max(len(a), len(b))` characters regardless of mismatch. An attacker can supply a very long username/password string to maximize per-call CPU time, since `tsscmp` iterates over the full length of the longer string. [6](#0-5) 

### Impact Explanation

Node.js runs on a single-threaded event loop. Saturating it with synchronous CPU work (header parsing + constant-time comparisons) directly delays all other pending callbacks, including those serving `/api/v1/transactions`, `/api/v1/accounts`, and other endpoints that downstream gossip consumers poll. At sufficient request rates, the server becomes effectively unresponsive to legitimate traffic. The metrics endpoint also triggers `exporter.collect()` on success, but even the auth-rejection path (401) is enough to cause saturation since `authenticate()` runs before the early return.

### Likelihood Explanation

- **Precondition:** None. The endpoint is publicly reachable on the same port (default 5551) as the REST API, with no network-level restriction enforced by the application.
- **Attacker capability:** Any external actor with HTTP access and a basic flood tool (e.g., `wrk`, `ab`, `hey`) can execute this. No credentials, no prior knowledge beyond the default path `/swagger/metrics/` (documented in `docs/configuration.md`).
- **Repeatability:** Fully repeatable and automatable. The attack requires no state and can be sustained indefinitely.
- **Amplification:** Supplying long random strings as credentials maximizes `tsscmp()` iteration count per request at zero cost to the attacker.

### Recommendation

1. **Add a rate limiter specifically for the metrics path** before `authenticate()` is called — e.g., using `express-rate-limit` scoped to the metrics route, limiting to a small number of requests per IP per minute (e.g., 10/min).
2. **Restrict the metrics endpoint by network** — bind it to a separate internal-only port or require an IP allowlist (e.g., only Prometheus scraper IPs), enforced at the application or infrastructure level.
3. **Cap credential string length** before passing to `tsscmp()` — reject any `Authorization` header whose decoded username or password exceeds a reasonable bound (e.g., 256 bytes) to prevent per-call CPU amplification.
4. **Return early on missing credentials** before invoking `tsscmp()` — if `basicAuth(req)` returns null, immediately return 401 without calling `tsscmp()` at all (the current code already does this via short-circuit `&&`, but the `basicAuth` parsing itself still runs).

### Proof of Concept

```bash
# Flood the metrics endpoint with random credentials (no valid account needed)
# Requires: wrk (https://github.com/wg1929/wrk)

wrk -t 10 -c 500 -d 30s \
  -H "Authorization: Basic $(python3 -c "import base64,os; print(base64.b64encode(os.urandom(200)).decode())")" \
  http://<mirror-node-host>:5551/swagger/metrics/

# Simultaneously measure latency degradation on a legitimate endpoint:
while true; do
  curl -o /dev/null -s -w "%{time_total}\n" \
    http://<mirror-node-host>:5551/api/v1/transactions?limit=1
  sleep 0.5
done
```

Expected result: transaction API response times increase significantly (from ~10ms baseline to hundreds of ms or timeouts) while the flood is active, demonstrating event-loop starvation caused by unbounded `authenticate()` invocations on the metrics path.
