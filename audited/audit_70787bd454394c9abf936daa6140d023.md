### Title
Unauthenticated Connection-Pool Exhaustion via Unbounded Concurrent Requests to `/api/v1/network/supply`

### Summary
The `GET /api/v1/network/supply` endpoint in the `rest-java` module invokes `EntityRepository.getSupply()`, a native SQL aggregation query using `unnest+join` over the `entity` table, on every request with no application-level rate limiting, no per-IP throttling, and no result caching. An unprivileged attacker sending a flood of concurrent requests can exhaust the HikariCP connection pool, causing all subsequent requests across the mirror node instance to queue-timeout and fail, effectively taking the node offline.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) is a plain `@GetMapping("/supply")` with no authentication, no rate-limit annotation, and no throttle check:

```java
@GetMapping("/supply")
ResponseEntity<?> getSupply(
        @RequestParam(required = false) @Size(max = 2) TimestampParameter[] timestamp,
        @RequestParam(name = "q", required = false) SupplyType supplyType) {
    final var bound = Bound.of(timestamp, TIMESTAMP, FileData.FILE_DATA.CONSENSUS_TIMESTAMP);
    final var networkSupply = networkService.getSupply(bound);
    ...
}
```

`NetworkServiceImpl.getSupply()` (lines 59–89) has no `@Cacheable` annotation and directly calls `entityRepository.getSupply(lowerBounds, upperBounds)` on every invocation when no timestamp is provided.

`EntityRepository.getSupply()` (lines 19–29) executes a native SQL aggregation:

```sql
select cast(coalesce(sum(e.balance), 0) as bigint) as unreleased_supply,
       cast(coalesce(max(e.balance_timestamp), 0) as bigint) as consensus_timestamp
from entity e
join unnest(
        cast(string_to_array(:lowerBounds, ',') as bigint[]),
        cast(string_to_array(:upperBounds, ',') as bigint[])
     ) as ranges(min_val, max_val)
  on e.id between ranges.min_val and ranges.max_val
```

This is a full-table-scan-style aggregation joining 7 account ranges against the entire `entity` table. Each call holds a HikariCP connection for the full duration of the query (up to the `statementTimeout` of 10,000 ms per documentation).

**Why existing checks fail:**

1. **No rate limiting in rest-java**: The `ThrottleConfiguration`/`ThrottleManagerImpl`/`Bucket4j` infrastructure exists exclusively in the `web3` module. The rest-java module's filter chain contains only `LoggingFilter` and `MetricsFilter` — neither enforces any request rate limit.

2. **No caching**: `NetworkServiceImpl.getSupply()` has no `@Cacheable` or equivalent; every HTTP request triggers a fresh database query.

3. **GCP `maxRatePerEndpoint: 250`** is an infrastructure-layer setting that (a) requires HPA changes to take effect per the chart comment, (b) is GCP-specific and absent in non-GCP deployments, and (c) is a per-endpoint aggregate rate, not a per-IP limit — a distributed flood from many IPs is not blocked.

4. **HikariCP pool**: Configured via `spring.datasource.hikari` with no explicit `maximumPoolSize` override found in rest-java resources (only `banner.txt` exists under `rest-java/src/main/resources/`), defaulting to HikariCP's default of 10 connections. With `statementTimeout=10000ms`, 10 concurrent requests saturate the pool for up to 10 seconds each. HikariCP's default `connectionTimeout` is 30 seconds, so all subsequent requests queue and then fail with a timeout exception.

### Impact Explanation

When the HikariCP pool is exhausted, every request to the mirror node instance — not just `/network/supply` — fails to acquire a database connection and returns an error after the connection timeout. This effectively takes the entire rest-java instance offline for the duration of the attack. With a sustained flood (easily achievable with `ab`, `wrk`, or `curl` in a loop), the attacker can keep the pool permanently saturated. If multiple mirror node instances share the same PostgreSQL backend, the database-side connection limit can also be exhausted, amplifying the impact across all instances. This matches the ≥30% network processing node shutdown threshold described in the scope.

### Likelihood Explanation

No special privileges, credentials, or knowledge are required. The endpoint is publicly documented in the OpenAPI spec and monitored by the project's own k6 load test (`tools/k6/src/rest-java/test/networkSupply.js`). Any attacker with basic HTTP tooling can send hundreds of concurrent requests from a single machine or a small botnet. The attack is trivially repeatable and requires no state.

### Recommendation

1. **Add application-level rate limiting** to the rest-java module for this endpoint, mirroring the `Bucket4j`-based `ThrottleManagerImpl` pattern already used in `web3`. Apply a per-IP token-bucket limit (e.g., 10 req/s per IP) using a servlet filter or Spring interceptor.
2. **Cache the result** of `getSupply()` with a short TTL (e.g., 5–10 seconds) using `@Cacheable`, since the supply value changes only with each consensus round.
3. **Set an explicit `maximumPoolSize`** in `spring.datasource.hikari` for rest-java and ensure `connectionTimeout` is short enough to fail fast rather than queue indefinitely.
4. **Enforce per-IP rate limiting at the ingress/gateway layer** unconditionally, not only for GCP deployments.

### Proof of Concept

```bash
# Exhaust the HikariCP pool (default 10 connections) with 20 concurrent requests
# held open for up to 10s each by the unnest+join aggregation

seq 1 200 | xargs -P 20 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node-host>/api/v1/network/supply"

# Expected result:
# First ~10 requests return 200
# Remaining requests return 500 (HikariCP connection timeout)
# All other endpoints on the same instance also return 500 during the attack window
```

    public NetworkSupply getSupply(Bound timestamp) {
        final NetworkSupply networkSupply;

        final var bounds = networkProperties.getUnreleasedSupplyRangeBounds();
        final var lowerBounds = bounds.lowerBounds();
        final var upperBounds = bounds.upperBounds();

        if (timestamp.isEmpty()) {
            networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
        } else {
            var minTimestamp = timestamp.getAdjustedLowerRangeValue();
            final var maxTimestamp = timestamp.adjustUpperBound();

            // Validate timestamp range
            if (minTimestamp > maxTimestamp) {
                throw new IllegalArgumentException("Invalid range provided for timestamp");
            }

            final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
            minTimestamp = Math.max(minTimestamp, optimalLowerBound);

            networkSupply =
                    accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp);
        }

        if (networkSupply.consensusTimestamp() == 0L) {
            throw new EntityNotFoundException("Network supply not found");
        }

        return networkSupply;
    }
