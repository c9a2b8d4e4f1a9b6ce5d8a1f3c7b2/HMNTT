### Title
Unauthenticated Alias Lookup DoS via Uncached 404 Responses Exhausting DB Connection Pool

### Summary
The `/api/v1/accounts/:idOrAliasOrEvmAddress/rewards` endpoint in `AccountController.listStakingRewardsByAccountId()` resolves every alias input with a live database query against the `entity` table. Because 404 error responses are never written to the Redis response cache, and because the REST API has no per-IP or global rate limiting, an unauthenticated attacker can flood the endpoint with unique non-existent aliases, keeping all connections in the pool (default: 10) permanently occupied and rendering the REST API unavailable. The claim that this directly prevents gossip/importer transaction ingestion is overstated — the importer uses a separate connection pool and DB user — but the REST API itself is fully deniable.

### Finding Description

**Code path:**

1. `rest/routes/accountRoute.js:16` registers the route with no authentication or rate-limit middleware.
2. `rest/controllers/accountController.js:171` calls `EntityService.getEncodedId(req.params[...])`.
3. `rest/service/entityService.js:125-126` — when the input passes `AccountAlias.isValid()`, it calls `getAccountIdFromAlias()`.
4. `rest/service/entityService.js:42-53` — `getAccountFromAlias()` executes `entityFromAliasQuery` (a `SELECT id FROM entity WHERE alias = $1`) unconditionally via `super.getRows()`.
5. When no row is returned, `getAccountIdFromAlias()` (line 74-75) throws `NotFoundError`.
6. `rest/middleware/responseCacheHandler.js:95` — `responseCacheUpdateHandler` only caches responses where `httpStatusCodes.isSuccess(res.statusCode)` is true; 404 responses are never cached.
7. `rest/dbpool.js:14` — the pool is bounded by `config.db.pool.maxConnections`, which defaults to **10** (`hiero.mirror.rest.db.pool.maxConnections = 10`, docs line 556).
8. `rest/server.js:68-98` — no rate-limiting middleware is registered for the REST API (the `ThrottleConfiguration`/`ThrottleManagerImpl` throttle exists only in the `web3` Java service, not here).

**Root cause:** The combination of (a) no rate limiting on the REST Node.js service, (b) alias lookups always hitting the DB, and (c) 404 responses never being cached means every unique non-existent alias consumes a DB connection for up to `statementTimeout` (default 20 s).

### Impact Explanation

With 10 pool connections and a 20-second statement timeout, an attacker sending ≥10 concurrent requests with unique non-existent aliases saturates the entire pool. All subsequent legitimate REST API requests queue indefinitely or fail with a connection-timeout error (default 20 s wait). The REST API becomes unavailable for the duration of the attack. The importer's transaction ingestion is **not directly** affected because it uses a separate connection pool (`mirror_importer` user), but DB-level CPU/IO pressure from the alias table scans can indirectly degrade importer throughput on shared infrastructure.

### Likelihood Explanation

No privileges are required. The alias format (`AccountAlias.isValid()`) accepts standard base32-encoded strings; generating millions of syntactically valid but non-existent aliases is trivial. The attack is repeatable, stateless, and requires only an HTTP client. A single machine with 10 concurrent connections is sufficient to hold the pool hostage indefinitely.

### Recommendation

1. **Add rate limiting** to the REST API (e.g., `express-rate-limit` per IP) before alias/EVM-address resolution, specifically for endpoints that trigger DB lookups.
2. **Cache negative (404) results** for alias and EVM-address lookups with a short TTL (e.g., 5–30 s) in the existing Redis layer or an in-process LRU cache, so repeated lookups for the same non-existent alias do not hit the DB.
3. **Increase the default pool size** or add a concurrency limiter so that alias-lookup queries cannot monopolize all connections.
4. **Add an index** on `entity.alias` (if not already present) to bound the per-query cost.
