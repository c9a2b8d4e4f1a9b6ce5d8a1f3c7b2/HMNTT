### Title
Missing Application-Level Rate Limiting on `GET /api/v1/topics/{id}` Enables Thread Pool Exhaustion in rest-java

### Summary
The `TopicController.getTopic()` handler and its backing `TopicServiceImpl.findById()` have no application-level request throttling. Unlike the `web3` module, which uses bucket4j (`ThrottleConfiguration`, `ThrottleManagerImpl`), the `rest-java` module contains no equivalent rate-limiting filter or interceptor. An unprivileged attacker can flood `GET /api/v1/topics/{id}` to exhaust Tomcat's thread pool and HikariCP's connection pool, rendering all `rest-java`-served endpoints unavailable.

### Finding Description
**Code path:**

- `TopicController.getTopic()` at [1](#0-0)  dispatches to three sequential service calls (topic, entity, customFee), each issuing a DB query.
- `TopicServiceImpl.findById()` at [2](#0-1)  performs a blocking `topicRepository.findById()` with no guard.
- The only registered filters in `rest-java` are `MetricsFilter` and `LoggingFilter` â€” no throttling filter exists. [3](#0-2) 
- `WebMvcConfiguration` registers only argument resolvers and formatters â€” no rate-limiting interceptor. [4](#0-3) 

**Contrast with `web3`:** `ThrottleConfiguration` and `ThrottleManagerImpl` implement bucket4j-based per-second rate limiting for the `web3` module, but no equivalent exists for `rest-java`. [5](#0-4) 

**Root cause:** The `rest-java` module was built without porting the throttling subsystem present in `web3`. The assumption that infrastructure-layer controls are sufficient is incorrect (see below).

**Why existing checks fail:**

1. The GCP backend policy `maxRatePerEndpoint: 250` is only active when `global.gateway.enabled: true`, which defaults to `false`. [6](#0-5) [7](#0-6) 
2. The Traefik ingress middleware configures only a `circuitBreaker` and `retry` â€” no `rateLimit` middleware. [8](#0-7) 
3. The docker-compose Nginx proxy routes `/api/v1/topics/...` to `rest_java_host` with no `limit_req` directive. [9](#0-8) 
4. HPA is disabled by default (`hpa.enabled: false`), so no auto-scaling absorbs the flood. [10](#0-9) 

### Impact Explanation
Each request to `getTopic` issues three blocking DB queries against HikariCP. With Tomcat's default thread pool (200 threads) and a typical HikariCP pool (10 connections), a sustained flood saturates both pools. Once the thread pool is full, all `rest-java`-served endpoints (`/api/v1/topics/`, `/api/v1/network/`, `/api/v1/accounts/.../allowances/nfts`, etc.) become unavailable or severely degraded. Prometheus alerts (`RestJavaHighDBConnections`, `RestJavaRequestLatency`) fire only after the fact and do not prevent the attack. [11](#0-10) 

### Likelihood Explanation
No authentication is required. The endpoint is publicly accessible. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain thousands of requests per second from a single host. The attack is trivially repeatable and requires no special knowledge beyond the public API path documented in the OpenAPI spec. [12](#0-11) 

### Recommendation
1. Add a bucket4j `RateLimitFilter` (or Spring `HandlerInterceptor`) to `rest-java` mirroring the pattern in `web3/ThrottleConfiguration` and `ThrottleManagerImpl`, configurable via `hiero.mirror.restjava.throttle.requestsPerSecond`.
2. Enable and enforce per-IP rate limiting at the Traefik ingress layer using the `rateLimit` middleware for all `rest-java` routes.
3. Enable HPA by default (`hpa.enabled: true`) so the deployment can scale under load.
4. Add a `limit_req` directive to the Nginx docker-compose proxy for the `rest_java_host` location blocks.

### Proof of Concept
```bash
# Prerequisites: rest-java running locally on port 8084, topic 0.0.1 exists
# Tool: wrk (https://github.com/wg/wrk)

wrk -t 16 -c 500 -d 60s http://localhost:8084/api/v1/topics/0.0.1

# Expected result within ~10-30 seconds:
# - Tomcat busy threads saturate (observable via /actuator/metrics/tomcat.threads.busy)
# - HikariCP pending connections spike (hikaricp_connections_pending)
# - Subsequent requests to ANY rest-java endpoint (e.g., /api/v1/network/fees) 
#   receive 503 or queue indefinitely
# - Prometheus alert RestJavaHighDBConnections fires (>75% pool utilization)
```
