### Title
Unauthenticated O(transactions × operations) DB Query Amplification in Rosetta `/block` Endpoint via `updateOperationAccountAlias()`

### Summary
The Rosetta `/block` endpoint is publicly accessible with no authentication. For each request, `updateOperationAccountAlias()` iterates over every operation in every transaction and issues a synchronous `GetAccountAlias()` database query for each cache miss. Because there is no per-operation limit and no application-level rate limiting, an unauthenticated attacker can trigger O(transactions × operations) sequential DB queries per request, exhausting the database connection pool and causing denial of service.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `Block()` (line 47) → `updateOperationAccountAlias()` (lines 104–129):

```go
// Block() — lines 61-69
if len(block.Transactions) > s.maxTransactionsInBlock {
    // truncates to maxTransactionsInBlock transactions
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
if err = s.updateOperationAccountAlias(ctx, block.Transactions...); err != nil {
    return nil, err
}

// updateOperationAccountAlias — lines 108-127
for _, transaction := range transactions {
    operations := transaction.Operations
    for index := range operations {
        accountId := operations[index].AccountId
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)  // DB query
            ...
            s.entityCache.Set(result.GetId(), result)
        }
        operations[index].AccountId = cached
    }
}
```

**Root cause:** `maxTransactionsInBlock` caps the number of transactions but there is no cap on operations per transaction. Each operation triggers a synchronous `GetAccountAlias()` call (SQL: `select alias, id from entity where id = @id`) on a cache miss. The LRU cache is per-process and has no TTL — it only evicts on capacity overflow. On first access to any block (or after cache eviction), every unique account ID across all operations causes a DB round-trip.

**No authentication:** `main.go` (lines 217–219) wraps the router only with `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`. No auth middleware exists for the Rosetta API at the application level.

**Cache does not protect against concurrent requests:** The LRU cache (`go-generics-cache`) has no request coalescing. N concurrent requests for the same uncached block each independently execute the full O(transactions × operations) DB query loop before any of them can populate the cache.

**DB connection pool:** Default `maxOpenConnections = 100` (docs, line 660). Each concurrent request holds connections for the duration of its sequential DB calls.

### Impact Explanation
An attacker sending concurrent POST requests to `/block` for a block containing `maxTransactionsInBlock` transactions each with many operations (Hedera crypto transfers can carry hundreds of individual transfers per transaction) will saturate the 100-connection DB pool. This blocks all other DB-dependent Rosetta endpoints (account balance, network status, etc.) and can cause the DB server itself to queue/reject connections. The `statementTimeout` of 20 seconds (docs, line 662) means each held connection can persist for up to 20 seconds per query, compounding the exhaustion. This is a non-network-based DoS requiring only HTTP access to port 5700.

### Likelihood Explanation
The Rosetta API is designed to be publicly accessible (it is a read API for blockchain indexers and exchange integrations). No credentials are required. The attacker needs only to identify a block with high operation density — this is observable from the public Hedera ledger. The attack is repeatable and scriptable with standard HTTP tooling. The optional Traefik rate-limit middleware (10 req/s average, `charts/hedera-mirror-rosetta/values.yaml` lines 157–160) is only present in Kubernetes Helm deployments and is not enforced by the application itself, meaning bare-metal or Docker deployments have no rate limiting at all.

### Recommendation
1. **Add a per-request operation count cap** inside `updateOperationAccountAlias()` — reject or truncate if total operations across all transactions exceeds a configurable threshold.
2. **Add request-level rate limiting** in the Go application itself (e.g., `golang.org/x/time/rate`) rather than relying solely on infrastructure middleware.
3. **Batch DB lookups** — collect all unique account IDs first, issue a single `WHERE id IN (...)` query, then populate the cache and update operations, replacing the per-operation sequential query pattern.
4. **Add cache stampede protection** — use a singleflight group so concurrent requests for the same uncached account ID share a single DB query.

### Proof of Concept
```bash
# 1. Identify a block with high-operation transactions on the target network
BLOCK_INDEX=<block_with_many_operations>
BLOCK_HASH=<corresponding_hash>

# 2. Send concurrent unauthenticated requests to exhaust DB connections
for i in $(seq 1 100); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d '{
      "network_identifier": {"blockchain": "Hedera", "network": "mainnet"},
      "block_identifier": {"index": '"$BLOCK_INDEX"', "hash": "'"$BLOCK_HASH"'"}
    }' &
done
wait

# 3. Observe: subsequent requests to /network/status or /account/balance
# return errors or time out due to DB connection pool exhaustion.
# DB logs will show connection queue saturation.
```

Each of the 100 concurrent requests independently executes up to `maxTransactionsInBlock × operations_per_transaction` sequential DB queries before the cache is populated, holding DB connections for the full duration.
