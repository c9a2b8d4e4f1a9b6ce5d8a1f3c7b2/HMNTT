### Title
Cache-Key Bypass via Single Oversized Comment Enables Forced Full Re-Parse on Every Request

### Summary
The `CachedPreparsedDocumentProvider` uses the raw query string as the cache key. An unprivileged attacker can craft queries containing a single GraphQL comment padded to consume the full 10,000-character budget, making each request unique and forcing a full parse+validate cycle on every cache miss. The `maxWhitespaceTokens(1000)` guard is ineffective because a single comment — regardless of its byte length — counts as exactly one whitespace token.

### Finding Description

**Cache key is the raw query string:**

In `CachedPreparsedDocumentProvider.java` line 26, the cache key is `executionInput.getQuery()` — the verbatim query string including all comments and whitespace. [1](#0-0) 

**Parser limits configured in `GraphQlConfiguration.java`:** [2](#0-1) 

The four limits are:
- `maxCharacters(10000)` — total character budget
- `maxWhitespaceTokens(1000)` — number of whitespace/ignored *tokens*
- `maxTokens(1000)` — number of real GraphQL tokens
- `maxRuleDepth(100)` — parse-rule nesting depth

**Root cause — token count ≠ character count:**

In graphql-java's lexer, a GraphQL comment (`# ...`) is a single *ignored token* regardless of how many characters it contains. A comment of the form:

```
# AAAAAAA...AAAAAAA   (9,988 characters)
```

is counted as **one** whitespace token. Therefore the following query:

```graphql
# <9988 unique chars>
{ account(input: {id: "0.0.1"}) { id } }
```

- Total characters: ~10,000 → passes `maxCharacters(10000)` ✓  
- Whitespace tokens: 1 → passes `maxWhitespaceTokens(1000)` ✓  
- Real tokens: ~12 → passes `maxTokens(1000)` ✓  

By varying the comment content (e.g., embedding a counter or random bytes), every request produces a **distinct cache key**, guaranteeing a cache miss and a full parse+validate cycle for each request.

**No rate limiting in the graphql module:**

The rate-limiting infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module. The graphql module has no per-IP or per-user request throttle. [3](#0-2) 

**Cache size is 1,000 entries, 1-hour TTL:** [4](#0-3) 

An attacker can continuously evict cached entries by flooding with unique-comment queries, keeping the cache perpetually cold for legitimate traffic.

### Impact Explanation

Every cache-miss request forces graphql-java to lex, parse, and validate a 10,000-character document. Compared to a typical cached query of ~100 characters, each attacker request imposes roughly 100× the lexer work. Without rate limiting, a single attacker thread sending requests in a tight loop can saturate one CPU core with parser work, displacing legitimate cached requests and increasing parse-time CPU well above 30% on a lightly loaded node. The `account()` resolver itself is cheap (single DB lookup), so the parse/validate phase dominates the per-request cost on cache misses.

### Likelihood Explanation

- **No authentication required** — the `/graphql` endpoint is public.
- **No rate limiting** in the graphql module.
- **Trivially scriptable** — a simple loop incrementing a counter inside the comment is sufficient.
- **Repeatable** — the cache TTL is 1 hour, but the attacker continuously generates new unique keys, so the cache never warms up for the attack queries.
- **Low noise** — each individual request returns a valid HTTP 200 with a real account result, making it hard to distinguish from legitimate traffic by status code alone.

### Recommendation

1. **Normalize the cache key before caching**: Strip or canonicalize comments and insignificant whitespace from the query string before using it as the cache key. This ensures `{ account(...) { id } }` with any comment padding maps to the same cache entry.
2. **Add per-IP or per-connection request rate limiting** to the graphql module (analogous to the `ThrottleManagerImpl` in the `web3` module).
3. **Alternatively, limit comment/whitespace character budget independently** of token count — e.g., reject any query where the sum of characters in ignored tokens exceeds a threshold (e.g., 200 characters).

### Proof of Concept

```python
import requests, itertools, string, random

URL = "http://<mirror-node-host>/graphql/alpha"

def make_padded_query(n):
    # Single comment consuming ~9950 chars, unique per request
    padding = ''.join(random.choices(string.ascii_letters, k=9950))
    return f"# {padding}\n{{ account(input: {{id: \"0.0.1\"}}) {{ id balance }} }}"

for i in itertools.count():
    q = make_padded_query(i)
    assert len(q) <= 10000          # stays within maxCharacters
    # comment = 1 whitespace token  # stays within maxWhitespaceTokens
    requests.post(URL, json={"query": q}, timeout=5)
    # Each request: unique cache key → cache miss → full 10 000-char parse+validate
```

Preconditions: network access to the public `/graphql` endpoint, no credentials needed.  
Trigger: run the loop at sustained throughput (e.g., 50–200 req/s from a single host).  
Result: parse-time CPU on the mirror-node process rises proportionally; cached legitimate queries are continuously evicted by the 1,000-entry LRU cache filling with attacker-unique keys.

class GraphQlConfiguration {

    static {
        // Configure GraphQL parsing limits to reject malicious input
        Consumer<Builder> consumer =
                b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
        ParserOptions.setDefaultParserOptions(
                ParserOptions.getDefaultParserOptions().transform(consumer));
        ParserOptions.setDefaultOperationParserOptions(
                ParserOptions.getDefaultOperationParserOptions().transform(consumer));
    }

    @Bean
    GraphQlSourceBuilderCustomizer graphQlCustomizer(PreparsedDocumentProvider provider) {
        var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
        var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
        var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);

        return b -> b.configureGraphQl(
                graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
    }
