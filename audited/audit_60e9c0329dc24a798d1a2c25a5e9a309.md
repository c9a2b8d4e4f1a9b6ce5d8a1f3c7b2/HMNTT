### Title
Unbounded Resource Exhaustion via ETag Cache-Busting on Unthrottled `/api/*` Endpoints

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers a `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no accompanying rate limiting or connection throttling in the `rest-java` module. An unprivileged attacker can flood these endpoints with requests carrying unique query parameters, defeating ETag reuse and forcing the filter to buffer and MD5-hash every full response body independently, exhausting heap memory, CPU, and thread-pool resources until the service becomes unavailable.

### Finding Description
**Exact code path:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause and failed assumption:**
`ShallowEtagHeaderFilter` (Spring's `OncePerRequestFilter` subclass) wraps the servlet response in a `ContentCachingResponseWrapper`. The entire response body is written into an in-memory byte array before the filter computes an MD5 digest and emits the `ETag` header. The design assumes that ETag reuse will amortize this cost across repeated identical requests. That assumption fails completely when every request carries a distinct query string (e.g., `?_=<random>`, varying `limit`/`order`/`timestamp` parameters), because no two requests share an ETag and the filter must buffer and hash every response independently.

**No rate limiting exists in `rest-java`:**
- The `rest-java` config package contains only `LoggingFilter`, `MetricsFilter`, `JacksonConfiguration`, `WebMvcConfiguration`, and `RestJavaConfiguration` — no throttle filter, no bucket4j bean, no connection limit.
- `rest-java/src/main/resources/` contains only `banner.txt`; there are no `application.yml` properties setting `server.tomcat.max-connections`, `max-threads`, or `accept-count`.
- The bucket4j-based `ThrottleConfiguration` / `ThrottleManagerImpl` exists exclusively in the `web3` module and is not applied to `rest-java` endpoints.

**Exploit flow:**
1. Attacker opens N concurrent HTTP connections (no authentication required).
2. Each connection sends `GET /api/v1/network/nodes?_=<unique_random>` (or any other `/api/*` endpoint).
3. The controller executes a database query and writes a full paginated JSON response.
4. `ShallowEtagHeaderFilter` intercepts the response, copies the entire body into a heap byte array, and computes MD5.
5. Because `_` differs on every request, no `If-None-Match` match is possible; the filter always performs the full buffer+hash cycle.
6. With N concurrent requests, N full response bodies are simultaneously resident in heap; N MD5 computations run in parallel on the Tomcat thread pool.
7. Heap pressure triggers GC storms or OOM; CPU saturation causes thread starvation; legitimate requests time out.

### Impact Explanation
The mirror node's `/api/v1/network/*` and other REST endpoints are the primary interface for querying Hiero network state (nodes, fees, exchange rates, supply, transactions). Sustained exhaustion of the `rest-java` process renders these endpoints unavailable, preventing clients and downstream services from reading network data. Because the attack requires no credentials and no special protocol knowledge, the blast radius is the entire public-facing query surface of the mirror node.

### Likelihood Explanation
The attack requires only an HTTP client capable of sending concurrent requests — no authentication, no special headers, no privileged access. Cache-busting with random query parameters is a well-known technique. A single attacker with a modest connection count (hundreds of concurrent requests) can sustain the attack indefinitely. The absence of any application-level rate limiting in `rest-java` means there is no automatic recovery path short of operator intervention or infrastructure-level blocking.

### Recommendation
1. **Add a rate-limiting filter to `rest-java`** analogous to the bucket4j `ThrottleConfiguration` already present in `web3`, scoped to `/api/*` URL patterns and registered at a lower order than `etagFilter`.
2. **Set Tomcat connection and thread limits** in `rest-java` application properties (`server.tomcat.max-connections`, `server.tomcat.threads.max`, `server.tomcat.accept-count`) to bound the maximum concurrent load.
3. **Consider disabling `ShallowEtagHeaderFilter` for large-payload endpoints** or replacing it with a `WeakEtagHeaderFilter` backed by a response-level cache keyed on the canonical URL (without volatile parameters), so the buffer+hash cost is not paid on every unique request.
4. **Strip or reject unknown/volatile query parameters** (e.g., cache-busting `_` parameters) at the filter or controller layer before they reach the ETag filter.

### Proof of Concept
```bash
# Requires: curl, GNU parallel or similar
# No authentication needed

BASE="http://<mirror-node-host>/api/v1/network/nodes"

# Send 500 concurrent requests, each with a unique cache-busting parameter
seq 1 500 | parallel -j 500 \
  "curl -s -o /dev/null -w '%{http_code}\n' '${BASE}?_={}'"

# Observe: heap/CPU spike on the rest-java process; legitimate requests
# begin returning 503 or timing out as the Tomcat thread pool saturates.
# Repeat in a loop to sustain the condition.
```

Each request forces `ShallowEtagHeaderFilter` to buffer and MD5-hash a full response body with no possibility of a 304 short-circuit, reproducing the resource exhaustion condition described above. [1](#0-0) [2](#0-1)

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
