### Title
Entity-Type Oracle via Differential gRPC Error Codes in `topicExists()`

### Summary
The `topicExists()` method in `TopicMessageServiceImpl` emits two semantically distinct exceptions depending on whether a queried entity ID is absent from the database (`EntityNotFoundException` → gRPC `NOT_FOUND`) or present but of a non-topic type (`IllegalArgumentException` → gRPC `INVALID_ARGUMENT`). Because the `subscribeTopic` gRPC endpoint requires no authentication, any unprivileged caller can probe arbitrary entity IDs and use the differing status codes as a binary oracle to enumerate which IDs exist in the mirror node's entity table and determine their types.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `topicExists()`, lines 94–106:

```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
            .switchIfEmpty(
                    grpcProperties.isCheckTopicExists()
                            ? Mono.error(new EntityNotFoundException(topicId))   // branch A
                            : Mono.just(Entity.builder()...build()))
            .filter(e -> e.getType() == EntityType.TOPIC)
            .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic"))); // branch B
}
```

**Error mapping** (`ProtoUtil.toStatusRuntimeException`, lines 47–52):

```java
} else if (t instanceof IllegalArgumentException ...) {
    return clientError(t, Status.INVALID_ARGUMENT, t.getMessage());   // code 3
} else if (t instanceof EntityNotFoundException) {
    return clientError(t, Status.NOT_FOUND, t.getMessage());           // code 5
```

**Root cause:** The two failure modes — entity absent vs. entity present but wrong type — are surfaced as *different* gRPC status codes (`INVALID_ARGUMENT` / `NOT_FOUND`) with distinct human-readable messages ("Not a valid topic" vs. "0.0.X does not exist"). No authentication or rate-limiting is applied to `ConsensusController.subscribeTopic` (confirmed: no security interceptors, no `@PreAuthorize`, no credential checks in `ConsensusController.java`).

**Failed assumption:** The design assumes callers already know which IDs are topics. In practice the differential response leaks the entity-existence bit and the entity-type bit to any anonymous caller.

### Impact Explanation
An attacker can scan the full entity ID space (shard/realm/num tuples) against the public gRPC port and build a complete map of: (a) which entity IDs exist in the mirror node's database, and (b) which of those are non-topic entities (accounts, contracts, files, tokens, schedules). This constitutes unauthorized disclosure of network topology — e.g., which contract IDs or account IDs are live — without any on-chain transaction or privileged credential. While no funds move directly, the oracle can feed reconnaissance for targeted attacks (e.g., identifying high-value contract addresses, mapping token IDs, correlating account existence with off-chain data).

### Likelihood Explanation
The gRPC endpoint is publicly reachable on any deployed mirror node (default port 5600). No authentication is required. The entity ID space is sparse but bounded and sequential, making automated scanning trivial with a standard gRPC client. The scan is repeatable, stateless, and produces a deterministic signal. Any developer or researcher with network access to the mirror node can execute this in minutes.

### Recommendation
1. **Normalize error responses:** Return the same gRPC status code (`NOT_FOUND` or `INVALID_ARGUMENT`) regardless of whether the entity is absent or present-but-wrong-type. A single `NOT_FOUND` / "Topic not found" for both cases eliminates the oracle.
2. **Rate-limit the `subscribeTopic` endpoint** per source IP/client to make bulk scanning economically infeasible.
3. Optionally, add a constant-time delay to error responses to prevent timing-based side channels.

### Proof of Concept

**Preconditions:** Network access to the mirror node gRPC port (default 5600). No credentials needed.

**Steps:**

```bash
# 1. Probe an ID that does not exist at all
grpcurl -plaintext -d '{"topicID":{"topicNum":99999999}}' \
  <mirror-node>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# → Status: NOT_FOUND, message: "0.0.99999999 does not exist"
# Conclusion: entity 99999999 does not exist

# 2. Probe an ID that exists but is an ACCOUNT (not a topic)
grpcurl -plaintext -d '{"topicID":{"topicNum":2}}' \
  <mirror-node>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# → Status: INVALID_ARGUMENT, message: "Not a valid topic"
# Conclusion: entity 2 EXISTS and is not a topic (account/contract/token/etc.)

# 3. Probe a valid topic ID
grpcurl -plaintext -d '{"topicID":{"topicNum":1234}}' \
  <mirror-node>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# → Stream opens (no error)
# Conclusion: entity 1234 is a valid topic
```

Automating steps 1–3 across the full numeric range maps the entire entity table with entity-type resolution, using only the public gRPC API and zero privileges.
