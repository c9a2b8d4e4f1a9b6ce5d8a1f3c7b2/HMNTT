### Title
Stale Token Airdrop Cache Causes Smart Contract Simulations to Read Non-Existent PENDING State

### Summary
The `CACHE_NAME_TOKEN_AIRDROP` cache within `CACHE_MANAGER_TOKEN` uses a 1-second write-expiry TTL and has no invalidation mechanism when an airdrop transitions from `PENDING` to `CLAIMED` or `CANCELLED`. The `findById()` method in `TokenAirdropRepository` is annotated `@Cacheable` with `unless = "#result == null"`, which caches `Optional.of(pendingAirdrop)` results. After the DB reflects a state change (claim/cancel), the cache continues to serve the stale `PENDING` result for up to 1 second, causing `AirdropsReadableKVState.readFromDataSource()` to return a non-existent pending airdrop to the EVM simulation layer.

### Finding Description

**Code path:**

`EvmConfiguration.cacheManagerToken()` (lines 114–127) configures `CACHE_MANAGER_TOKEN` using `cacheProperties.getToken()`, which resolves to `ENTITY_CACHE_CONFIG = "expireAfterWrite=1s,maximumSize=10000,recordStats"` (CacheProperties.java line 48). All sub-caches including `CACHE_NAME_TOKEN_AIRDROP` share this 1-second TTL.

`TokenAirdropRepository.findById()` (line 17) is annotated:
```java
@Cacheable(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN, unless = "#result == null")
```
The query filters `state = 'PENDING'`. When an airdrop is PENDING, the result `Optional.of(tokenAirdrop)` is cached. The `unless = "#result == null"` condition is never triggered because the method always returns a non-null `Optional`; it does not exclude `Optional.of(...)` from caching.

`AirdropsReadableKVState.readFromDataSource()` (lines 30–47) dispatches to `findById()` for latest-block calls (no timestamp context), directly consuming the cached value:
```java
.orElseGet(() -> tokenAirdropRepository.findById(senderId, receiverId, tokenId, serialNumber))
```

**Root cause:** There is no cache eviction or invalidation triggered when the importer processes a `TokenClaimAirdrop` or `TokenCancelAirdrop` transaction and updates the DB row's state to `CLAIMED`/`CANCELLED`. The cache entry for the same `(senderId, receiverId, tokenId, serialNumber)` key continues to hold the stale `PENDING` result until the 1-second TTL expires.

**Exploit flow:**
1. An airdrop is created on-chain; the mirror node importer writes `state=PENDING` to the DB.
2. A smart contract simulation (`eth_call`) queries the airdrop via `AirdropsReadableKVState` → `findById()`. The `Optional.of(pendingAirdrop)` result is written into the Caffeine cache with a 1-second expiry.
3. The receiver claims (or sender cancels) the airdrop on-chain. The importer processes the record file and updates the DB to `state=CLAIMED`/`CANCELLED`.
4. Within the remaining TTL window (up to 1 second), any `eth_call` or `eth_estimateGas` request that queries the same airdrop key hits the cache and receives the stale `PENDING` result.
5. The EVM simulation layer treats the airdrop as still PENDING. A smart contract calling `cancelAirdrops` or `claimAirdrops` precompile receives a simulated `SUCCESS` response for an airdrop that no longer exists in PENDING state.

**Why existing checks fail:**
- `unless = "#result == null"` does not exclude `Optional.of(...)` from caching; it only excludes a literal `null` return, which never occurs.
- `findByIdAndTimestamp()` (the historical path) is correctly not cached, but the latest-block path (`findById()`) is cached without invalidation.
- No `@CacheEvict` or `@CachePut` is applied anywhere in the airdrop claim/cancel transaction handlers.

### Impact Explanation
Smart contract simulations (`eth_call`, `eth_estimateGas`) served by the mirror node's web3 module return incorrect airdrop state for up to 1 second after a claim or cancel is processed. A contract that branches on whether an airdrop is PENDING (e.g., to decide whether to proceed with a cancel or claim operation) will receive a simulated `SUCCESS` outcome for an operation that would revert on the actual Hedera network. This causes unintended smart contract behavior in the simulation layer: incorrect gas estimates, misleading `eth_call` return values, and dApp logic errors. No funds are at direct risk because the mirror node is a read-only simulation layer; actual on-chain state is maintained independently by the Hedera consensus network.

### Likelihood Explanation
The stale window is bounded by the 1-second TTL. An unprivileged user needs only to issue an `eth_call` for a known airdrop key within the window between the DB update and cache expiry — no special privileges are required. The airdrop parameters (sender, receiver, token, serial) are publicly observable on-chain. The mirror node importer introduces its own processing delay, so the effective stale window may be slightly shorter than 1 second in practice. The scenario occurs naturally under normal network activity (concurrent claim and query) without any deliberate attack, making it repeatable and realistic.

### Recommendation
1. Change the `unless` condition to also exclude empty Optionals: `unless = "#result == null || !#result.isPresent()"`. This prevents caching of negative lookups and reduces the blast radius of stale entries.
2. Add `@CacheEvict(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN)` to the repository or service layer when a claim or cancel is processed, or use a `@CachePut` to overwrite the entry with the updated state.
3. Alternatively, reduce the token cache TTL further or make the airdrop cache a separate, shorter-lived cache distinct from the general token cache.

### Proof of Concept
1. Persist a PENDING fungible token airdrop in the mirror node DB for `(sender=A, receiver=B, token=T, serial=0)`.
2. Issue `eth_call` invoking `claimAirdrops` or a read of the airdrop state → `findById(A, B, T, 0)` returns `Optional.of(pendingAirdrop)` and is cached.
3. Immediately update the DB row to `state=CLAIMED` (simulating the importer processing a `TokenClaimAirdrop` record).
4. Within 1 second of step 2, issue another `eth_call` for the same airdrop key.
5. Observe that `findById()` returns the cached `Optional.of(pendingAirdrop)` (state=PENDING) instead of `Optional.empty()`, and the simulation returns `SUCCESS` for an operation on a non-existent pending airdrop. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
