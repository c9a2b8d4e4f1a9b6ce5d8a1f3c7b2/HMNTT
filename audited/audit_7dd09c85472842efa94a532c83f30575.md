### Title
Unauthenticated Repeated Calls to `/api/v1/network/supply` Trigger Unbounded DB Aggregation with No Caching or Rate Limiting

### Summary
The `GET /api/v1/network/supply` endpoint in the `rest-java` module is publicly accessible with no authentication, rate limiting, or result caching. Every request unconditionally executes a full-table aggregation query (`sum(e.balance)`, `max(e.balance_timestamp)`) over the `entity` table via `EntityRepository.getSupply()`. An unprivileged attacker can flood this endpoint to sustain elevated DB CPU and I/O consumption well above baseline.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) receives any unauthenticated GET request and unconditionally calls `networkService.getSupply(bound)`. [1](#0-0) 

`NetworkServiceImpl.getSupply()` (lines 59–89), when no timestamp is provided (the common case), calls `entityRepository.getSupply(lowerBounds, upperBounds)` directly with no caching annotation. [2](#0-1) 

`EntityRepository.getSupply()` (lines 19–29) executes a native SQL aggregation — `sum(e.balance)` and `max(e.balance_timestamp)` — over the entire `entity` table joined against a set of ID ranges via `unnest`. On a production mirror node with millions of entities, this is a full sequential scan with aggregation on every invocation. [3](#0-2) 

**Root cause — failed assumptions:**

1. **No `@Cacheable` on `getSupply()`**: The `rest-java` `EntityRepository` carries no caching annotation. Contrast this with the `grpc` and `web3` `EntityRepository` implementations, which both use `@Cacheable`. [4](#0-3) 

2. **No rate limiting in `rest-java`**: The `ThrottleConfiguration` and `ThrottleManagerImpl` with bucket4j rate limiting exist exclusively in the `web3` module. There is no equivalent in `rest-java`. [5](#0-4) 

3. **`ShallowEtagHeaderFilter` does not prevent DB execution**: `RestJavaConfiguration` registers an ETag filter for `/api/*`, but this filter computes the ETag *after* the response body is generated — meaning the DB aggregation query runs on every request regardless. [6](#0-5) 

4. **The Node.js response cache does not apply**: The Redis-backed `responseCacheHandler.js` belongs to the separate `rest` (Node.js) module and does not protect the Java-served `/api/v1/network/supply` endpoint. [7](#0-6) 

### Impact Explanation
On a production Hedera mirror node the `entity` table contains tens of millions of rows. Each call to `GET /api/v1/network/supply` forces PostgreSQL to perform a full aggregation scan across all rows matching the configured unreleased-supply account ranges (7 ranges, up to account 750 by default). With no caching and no rate limiting, an attacker sending hundreds of concurrent requests per second can saturate DB CPU and I/O, degrading query performance for all other mirror node consumers. This directly satisfies the >30% resource consumption threshold without requiring brute-force credential attacks — only standard HTTP GET requests are needed.

### Likelihood Explanation
The endpoint requires zero privileges, zero authentication, and accepts requests from any IP. The attack is trivially scriptable with any HTTP load-testing tool (e.g., `ab`, `wrk`, `k6` — the project itself ships a k6 test for this endpoint). The attacker needs no special knowledge beyond the public API documentation. The attack is repeatable and sustainable indefinitely. [8](#0-7) 

### Recommendation
1. **Add result caching** at the service or repository layer with a short TTL (e.g., 15–30 seconds) using Spring's `@Cacheable`, consistent with how other `EntityRepository` implementations in this codebase are handled.
2. **Add rate limiting** to the `rest-java` module for the `/api/v1/network/supply` endpoint, mirroring the bucket4j-based `ThrottleConfiguration` already present in `web3`.
3. **Consider a DB-level query timeout** for this aggregation to bound worst-case execution time per request.

### Proof of Concept
```bash
# No authentication required. Run from any machine with network access.
# Send 200 concurrent requests in a loop:
while true; do
  for i in $(seq 1 200); do
    curl -s "https://<mirror-node-host>/api/v1/network/supply" -o /dev/null &
  done
  wait
done
```

Each iteration fires 200 parallel requests. Each request triggers one full `sum(balance)` + `max(balance_timestamp)` aggregation over the `entity` table with no caching. Sustained execution will drive DB CPU and I/O significantly above the 24-hour baseline.
