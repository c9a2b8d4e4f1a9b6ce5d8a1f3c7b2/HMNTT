### Title
Unauthenticated GraphQL Complexity Analysis Executed on Every Request Due to Cache Bypass by MaxQueryComplexityInstrumentation

### Summary
The `MaxQueryComplexityInstrumentation` runs during the GraphQL execution phase, which occurs **after** the `CachedPreparsedDocumentProvider` cache lookup. This means complexity analysis is re-executed on every request, even for previously-seen queries. Combined with the absence of any rate limiting or authentication on the GraphQL endpoint, an unauthenticated attacker can repeatedly submit queries crafted to maximize complexity analysis work (bounded by `maxTokens=1000`) before rejection, sustaining CPU load with no per-client throttle.

### Finding Description

**Code locations:**

- [1](#0-0)  — Parser limits: `maxTokens(1000)`, `maxCharacters(10000)`.
- [2](#0-1)  — `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are wired as execution-phase instrumentations, not as parse/validate rules.
- [3](#0-2)  — Cache is keyed on `executionInput.getQuery()` and stores `PreparsedDocumentEntry` (parse + validation result only).
- [4](#0-3)  — Cache config: `expireAfterWrite=1h,maximumSize=1000`.

**Root cause:**

In graphql-java, `MaxQueryComplexityInstrumentation` hooks into `beginExecuteOperation`, which is invoked during the execution phase — **after** `PreparsedDocumentProvider.getDocumentAsync()` returns. The cache only covers parsing and validation. Complexity traversal of the query's field-selection tree is re-performed on every single request, regardless of whether the query string was seen before.

**Exploit flow:**

An attacker crafts a query using field aliases on `account` to push complexity just above 200 while staying within the 1000-token parser limit. For example:

```graphql
query {
  a1: account(input: {entityId: {shard:0,realm:0,num:1}}) { balance }
  a2: account(input: {entityId: {shard:0,realm:0,num:1}}) { balance }
  # ... repeated ~101 times (each alias = 2 complexity units: account + balance)
}
```

Each aliased `account { balance }` contributes 2 complexity units (1 per field by default). With ~101 aliases, complexity = 202 > 200, triggering rejection. The query fits well within 1000 tokens. On every subsequent request with this same string: parse/validate is served from cache, but complexity analysis traverses all ~202 field selections again before aborting.

**Why existing checks are insufficient:**

- The `PreparsedDocumentProvider` cache [5](#0-4)  only caches the parse/validate result; it does not short-circuit execution-phase instrumentation.
- No authentication or rate limiting exists anywhere in the graphql module (confirmed: no Spring Security config, no rate-limit filter found).
- The `LoggingFilter` [6](#0-5)  only logs; it imposes no throttle.
- The schema exposes `account` to all callers with no access control. [7](#0-6) 

### Impact Explanation
An unauthenticated attacker can sustain a stream of crafted requests that each force the server to traverse up to ~200 field selections for complexity scoring before rejection. While the per-request work is bounded (O(200) field traversal), the absence of any rate limiting means the attacker can saturate server threads with this pattern at high request rates, degrading availability for legitimate users. The cache provides no relief for the complexity analysis step.

### Likelihood Explanation
No privileges, credentials, or special network position are required. The GraphQL endpoint is publicly accessible. The attack is trivially scriptable: a single crafted query string can be replayed in a tight loop from any HTTP client. The query can be varied slightly (e.g., changing whitespace or argument values) to also exhaust the 1000-entry parse cache, forcing full parse+validate+complexity work on every request.

### Recommendation
1. **Add rate limiting** at the HTTP layer (e.g., Spring's `RateLimiter`, a reverse proxy like nginx/Envoy, or a WAF) keyed on client IP or API key, before requests reach GraphQL execution.
2. **Cache complexity results**: Extend `CachedPreparsedDocumentProvider` to also store and short-circuit the complexity check result for previously-seen query strings, so rejected queries are rejected immediately from cache without re-traversal.
3. **Reduce the complexity limit** further or assign higher per-field weights to `account` (which triggers a DB lookup) using a custom `FieldComplexityCalculator`.
4. **Require authentication** for the GraphQL endpoint so unauthenticated flood attacks are blocked at the auth layer.

### Proof of Concept

```bash
# Generate a query with 101 aliased account calls (complexity = 202 > 200 limit)
python3 -c "
aliases = '\n'.join(
  f'  a{i}: account(input: {{entityId: {{shard:0,realm:0,num:1}}}}) {{ balance }}'
  for i in range(101)
)
print('query {' + aliases + '}')
" > payload.graphql

QUERY=$(cat payload.graphql | tr '\n' ' ')

# Flood the endpoint - no credentials needed
while true; do
  curl -s -X POST http://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\"}" &
done
```

Expected result: Server returns `{"errors":[{"message":"maximum query complexity exceeded ..."}]}` on every request, but complexity analysis (field traversal of ~202 selections) executes server-side before each rejection, with no rate limiting to stop the flood.
