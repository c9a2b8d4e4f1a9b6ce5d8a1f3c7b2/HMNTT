### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to NFT Allowance Endpoint

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint in the `rest-java` module has no application-level rate limiting. Each request causes `NftAllowanceRepositoryCustomImpl.findAll()` to acquire a HikariCP connection held for up to the configured `statementTimeout` (10 seconds). An unprivileged attacker sending sustained concurrent requests can exhaust the finite HikariCP pool, preventing the application from acquiring any further database connections and effectively partitioning it from the database outside of design parameters.

### Finding Description
**Exact code path:**

`AllowancesController.getNftAllowances()` (lines 57–83) accepts unauthenticated GET requests with no rate-limiting guard. It delegates to `NftAllowanceServiceImpl.getNftAllowances()`, which calls `NftAllowanceRepositoryCustomImpl.findAll()` (lines 37–46):

```java
return dslContext
        .selectFrom(NFT_ALLOWANCE)
        .where(condition)
        .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
        .limit(request.getLimit())
        .fetchInto(NftAllowance.class);
```

This call acquires a JDBC connection from the shared HikariCP pool for the full duration of the query execution (up to `statementTimeout` = 10,000 ms per `hiero.mirror.restJava.db.statementTimeout`).

**Root cause and failed assumption:** The `web3` module has a full rate-limiting stack (`ThrottleManagerImpl`, `ThrottleConfiguration`, bucket4j). The `rest-java` module has **none** of this. The only input guard on the NFT allowance endpoint is `@Max(MAX_LIMIT)` on the `limit` parameter (capping result rows at 100), which constrains result size but does not constrain request frequency or concurrency. No `Filter`, `HandlerInterceptor`, or middleware in `rest-java` enforces a per-IP or global request rate.

**Why existing checks fail:**
- `@Positive @Max(MAX_LIMIT)` — limits rows returned, not requests per second.
- `statementTimeout = 10000` — caps individual query duration but does not prevent pool saturation across concurrent requests.
- Helm chart `maxRatePerEndpoint: 250` (GCP gateway) — optional infrastructure config, disabled by default (`global.gateway.enabled: false`), not enforced at the application layer.
- Circuit breaker middleware (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) — triggers on error ratios after the fact, not on request volume before pool exhaustion.

### Impact Explanation
When the HikariCP pool is fully occupied, all subsequent requests that require a database connection block until `connectionTimeout` expires, then throw an exception. This affects **every** database-backed endpoint in the `rest-java` service simultaneously — not just the NFT allowance endpoint. The application is effectively partitioned from the database for the duration of the attack. The existing `RestJavaHighDBConnections` alert fires only after 5 minutes at >75% utilization, providing no real-time protection. Severity: **High** (availability impact, no authentication required, affects all endpoints sharing the pool).

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly routable via the ingress path `/api/v1/accounts/.../allowances/nfts`. A single attacker with a modest number of concurrent HTTP connections (e.g., 10–20 threads in a loop) can sustain pool exhaustion indefinitely given a default HikariCP pool size of 10 and a 10-second statement timeout per connection. The attack is trivially repeatable with standard tools (`ab`, `wrk`, `curl` in parallel). No special knowledge of the Hedera protocol is needed.

### Recommendation
1. **Application-level rate limiting:** Port the `web3` module's bucket4j-based `ThrottleManager` pattern into `rest-java`, or add a `HandlerInterceptor`/`Filter` that enforces a global and/or per-IP request rate on all endpoints.
2. **Connection pool sizing and timeout tuning:** Set an explicit `spring.datasource.hikari.maximum-pool-size` appropriate for expected concurrency, and reduce `connectionTimeout` so exhausted-pool errors fail fast rather than queuing.
3. **Reduce statement timeout:** Lower `hiero.mirror.restJava.db.statementTimeout` to reduce the window each connection is held under slow/adversarial queries.
4. **Enforce infrastructure rate limiting:** Make the GCP gateway `maxRatePerEndpoint` (or equivalent Traefik middleware) mandatory rather than optional for public deployments.

### Proof of Concept
```bash
# Requires no credentials. Run from any machine with network access.
# Sends 50 concurrent requests in a tight loop to exhaust the HikariCP pool.

TARGET="https://<mirror-node-host>/api/v1/accounts/0.0.1/allowances/nfts?limit=100"

for i in $(seq 1 50); do
  curl -s "$TARGET" -o /dev/null &
done
wait

# Repeat in a loop:
while true; do
  for i in $(seq 1 50); do curl -s "$TARGET" -o /dev/null & done
  wait
done
```

While this runs, legitimate requests to any `rest-java` endpoint will begin receiving connection pool timeout errors (HikariCP `SQLTransientConnectionException`), and the `RestJavaHighDBConnections` Prometheus alert will eventually fire — but only after 5 minutes of sustained >75% utilization, by which time the service is already degraded.

    @GetMapping(value = "/nfts")
    NftAllowancesResponse getNftAllowances(
            @PathVariable EntityIdParameter id,
            @RequestParam(name = ACCOUNT_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] accountIds,
            @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
            @RequestParam(defaultValue = "asc") Sort.Direction order,
            @RequestParam(defaultValue = "true") boolean owner,
            @RequestParam(name = TOKEN_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] tokenIds) {
        var field = owner ? NFT_ALLOWANCE.SPENDER : NFT_ALLOWANCE.OWNER;
        var request = NftAllowanceRequest.builder()
                .accountId(id)
                .isOwner(owner)
                .limit(limit)
                .order(order)
                .ownerOrSpenderIds(new Bound(accountIds, true, ACCOUNT_ID, field))
                .tokenIds(new Bound(tokenIds, false, TOKEN_ID, NFT_ALLOWANCE.TOKEN_ID))
                .build();

        var serviceResponse = service.getNftAllowances(request);
        var allowances = nftAllowanceMapper.map(serviceResponse);

        var sort = Sort.by(order, ACCOUNT_ID, TOKEN_ID);
        var pageable = PageRequest.of(0, limit, sort);
        var links = linkFactory.create(allowances, pageable, EXTRACTORS.get(owner));

        return new NftAllowancesResponse().allowances(allowances).links(links);
    }
