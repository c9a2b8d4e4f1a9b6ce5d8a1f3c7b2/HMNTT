### Title
Unauthenticated Parameter Flooding via `validateBoundsRange()` Bypasses DB Layer in `getNftsByAccountId`

### Summary
An unprivileged external user can supply `token.id=gte:1&token.id=eq:5` to the `/api/v1/accounts/{id}/nfts` endpoint, which causes `validateBoundsRange()` to throw an `InvalidArgumentError` before any database query is issued. Because the REST API has no application-level rate limiting, an attacker can flood this endpoint at high request rates, consuming CPU and connection-handling resources entirely within the validation layer while the database remains unloaded and connection-pool exhaustion defenses are bypassed.

### Finding Description
**Exact code path:**

In `rest/controllers/accountController.js`, `getNftsByAccountId` (line 90–103) calls `utils.buildAndValidateFilters()` then `this.extractNftMultiUnionQuery()`. [1](#0-0) 

Inside `extractNftMultiUnionQuery`, each `token.id` filter is passed to `bounds.primary.parse(filter)`. [2](#0-1) 

`Bound.parse()` in `rest/controllers/bound.js` sets `this.lower` for `gte:1` and `this.equal` for `eq:5` — both are accepted without error at parse time. [3](#0-2) 

After the loop, `validateFilters()` → `validateBounds()` → `validateBoundsRange()` in `rest/controllers/baseController.js` checks `hasBound() && hasEqual()` and throws `InvalidArgumentError("Can't support both range and equal for token.id")`. [4](#0-3) 

**Root cause:** The REST API server (`rest/server.js`) has no application-level rate-limiting middleware. The middleware stack is: `requestLogger` → `authHandler` → optional metrics → optional cache check → route handlers → `responseHandler` → `handleError`. No rate limiter is present. [5](#0-4) 

**Why existing checks are insufficient:** The only per-request guard against repeated parameters is `isRepeatedQueryParameterValidLength()` (checked in `validateReq` via `maxRepeatedQueryParameters`), which caps the number of values for a single key per request — not the request rate. [6](#0-5) 

The `authHandler` middleware only sets a custom response limit for authenticated users; it does not throttle unauthenticated requests. [7](#0-6) 

### Impact Explanation
Every request with this payload is fully processed by the Node.js event loop (query parsing, filter building, bound parsing, validation) and returns a 400 without touching the database. Because the error path is cheaper per request than a valid path (no DB round-trip, no connection pool slot consumed), an attacker can sustain a higher sustained request rate than with valid queries. At sufficient volume this saturates the single-threaded Node.js event loop and exhausts OS-level TCP accept queues, degrading availability for legitimate users. The database and its connection pool remain unaffected, so the normal back-pressure mechanism (DB saturation) that would otherwise limit attack throughput is absent.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no application-level rate limit are required. The payload is two query parameters trivially constructable with `curl` or any HTTP client. The attack is stateless and infinitely repeatable from a single IP or a botnet. Any public deployment of the mirror-node REST API is exposed.

### Recommendation
1. **Add application-level rate limiting** to `rest/server.js` using a middleware such as `express-rate-limit` (per-IP, sliding window) before route handlers are registered.
2. **Add an infrastructure-level rate limit** (e.g., Traefik `rateLimit` middleware, as already done for the Rosetta service) in front of the REST API.
3. Optionally, short-circuit the `validateBoundsRange` check earlier — inside `Bound.parse()` — so that the conflict is detected as soon as the second conflicting operator is seen, reducing per-request work further (though this is a minor optimisation, not a security fix on its own).

### Proof of Concept
```bash
# Single request demonstrating the 400 trigger
curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1001/nfts?token.id=gte:1&token.id=eq:5"
# Returns: 400 {"_status":{"messages":[{"message":"Can't support both range and equal for token.id"}]}}

# Flood (no authentication required)
while true; do
  curl -s -o /dev/null \
    "https://<mirror-node-host>/api/v1/accounts/0.0.1001/nfts?token.id=gte:1&token.id=eq:5" &
done
# Observe: CPU climbs on the mirror-node REST process; DB connection count stays flat.
```
