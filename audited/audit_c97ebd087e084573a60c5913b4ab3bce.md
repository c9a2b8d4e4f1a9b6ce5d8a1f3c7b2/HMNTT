### Title
THROTTLE Token Consumed Before REJECT Filter Fires, Enabling Bucket Depletion by Unprivileged Attacker

### Summary
In `ThrottleManagerImpl.throttle()`, the request-filter loop evaluates every matching `RequestProperties` entry in list order without breaking after a `THROTTLE` action succeeds. If an operator configures a `THROTTLE` entry before a `REJECT` entry with overlapping match criteria, an attacker can craft requests that consume a token from the `THROTTLE` bucket and are then immediately rejected — permanently depleting that bucket for legitimate users, since no restoration path exists for per-filter THROTTLE tokens.

### Finding Description

**Exact code path**

`ThrottleManagerImpl.throttle()` (lines 44–48) iterates the configured `RequestProperties` list in declaration order:

```java
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);   // no break; loop continues
    }
}
```

`action()` (lines 66–75) handles `THROTTLE` by consuming one token from the per-filter `Bucket`, then returns normally — the loop advances to the next entry:

```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

If the next matching entry has `REJECT`, it throws immediately:

```java
case REJECT -> throw new ThrottleException("Invalid request");
```

The THROTTLE token that was just consumed is **never restored**. `restore()` (lines 59–64) only adds tokens back to the global `gasLimitBucket`; it has no knowledge of per-filter buckets and is never called on a REJECT path.

**Root cause / failed assumption**

The code assumes that filter entries in the list are mutually exclusive (a request will match at most one entry). There is no enforcement of this assumption — neither in configuration validation nor in the loop itself. When two entries overlap, the THROTTLE entry silently pre-charges the attacker before the REJECT entry fires.

**Exploit flow**

Operator configures (plausible production setup):

```yaml
hiero.mirror.web3.throttle.request:
  - action: THROTTLE        # entry 0 — rate-limit all "latest" block queries
    rate: 50
    filters:
      - field: BLOCK
        type: EQUALS
        expression: latest
  - action: REJECT          # entry 1 — block a known-bad data pattern
    filters:
      - field: DATA
        type: CONTAINS
        expression: deadbeef
```

Attacker sends requests with `block=latest` **and** `data=...deadbeef...`:

1. Global `rateLimitBucket.tryConsume(1)` → passes (attacker is within 500 req/s).
2. Global `gasLimitBucket.tryConsume(...)` → passes (low gas value).
3. Entry 0 (`THROTTLE`, `block=latest`) matches → `filter.getBucket().tryConsume(1)` succeeds → **1 THROTTLE token consumed**.
4. Entry 1 (`REJECT`, `data contains deadbeef`) matches → `ThrottleException("Invalid request")` thrown.
5. Attacker receives HTTP 429; THROTTLE token is gone.

Repeating at up to 500 req/s (global limit), the attacker exhausts the THROTTLE bucket (capacity = `rate` = 50, refills at 50/s). Legitimate users sending `block=latest` requests without the bad data pattern now hit an empty bucket and receive 429 errors they should never see.

**Why existing checks are insufficient**

- `RequestProperties.test()` (lines 45–61) applies a `counter`/`limit` guard and probabilistic `rate` sampling, but these are per-entry and do not prevent the overlap scenario.
- The `rate == 0` short-circuit (line 46) only disables an entry entirely; it does not prevent two entries from matching the same request.
- There is no configuration-level validation that REJECT entries appear before THROTTLE entries, or that filter criteria are disjoint.

### Impact Explanation

An unprivileged attacker can continuously starve the THROTTLE bucket for a class of legitimate requests (e.g., all `block=latest` queries) at zero cost to themselves beyond being rate-limited globally. Legitimate users receive spurious HTTP 429 responses, effectively constituting a targeted denial-of-service against a specific request category. The global `rateLimitBucket` and `gasLimitBucket` tokens are also consumed without any useful work being done, compounding the waste.

### Likelihood Explanation

The attack requires a configuration where a THROTTLE entry precedes a REJECT entry with overlapping match criteria — a plausible operational setup (broad throttle + narrow block-list). The attacker needs only to observe or guess the filter fields (BLOCK, DATA, FROM, TO, GAS, VALUE, ESTIMATE — a small, documented enum) and craft requests accordingly. No authentication, special privileges, or on-chain access is required. The attack is fully repeatable and automatable with a simple HTTP client.

### Recommendation

1. **Break the loop after any non-LOG action**: After `THROTTLE` succeeds or `REJECT` fires, stop evaluating further entries. This prevents a THROTTLE token from being consumed when a later REJECT would discard the request anyway.
2. **Enforce REJECT-before-THROTTLE ordering**: In `ThrottleProperties` or a `@PostConstruct` validator, assert that all `REJECT` entries appear before any `THROTTLE` entries in the list, or that their filter criteria are disjoint.
3. **Restore THROTTLE tokens on REJECT**: Wrap the filter loop in a try/catch; if a `ThrottleException` from a REJECT action is caught, call `addTokens(1)` on any THROTTLE buckets that were already charged in the same iteration.

### Proof of Concept

**Configuration** (application.yml):
```yaml
hiero.mirror.web3.throttle:
  requestsPerSecond: 500
  request:
    - action: THROTTLE
      rate: 5          # bucket capacity = 5 tokens, refills 5/s
      filters:
        - field: BLOCK
          type: EQUALS
          expression: latest
    - action: REJECT
      filters:
        - field: DATA
          type: CONTAINS
          expression: "cafebabe"
```

**Attack script** (bash):
```bash
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:8545 \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001",
         "data":"0xcafebabe","block":"latest","gas":50000}'
done
```

**Expected result**:
- All 10 attacker requests return HTTP 429 ("Invalid request" — REJECT path).
- The THROTTLE bucket (capacity 5) is fully drained after 5 requests.
- A legitimate request with `block=latest` and no `cafebabe` data now also returns HTTP 429 ("Requests per second rate limit exceeded") even though it should have been allowed through.
