### Title
Cache Key Fragmentation via Case-Variant Query Parameters in `cacheKeyGenerator()`

### Summary
`cacheKeyGenerator()` hashes `req.originalUrl` directly with MD5, without any case normalization of query parameter names. Because Express's custom `requestQueryParser` normalizes `req.query` keys to lowercase but never mutates `req.originalUrl`, an unprivileged user can send requests with case-variant query parameters (e.g., `?Limit=1` vs `?limit=1`) that produce distinct MD5 cache keys, causing every variant to miss the cache and bypass Redis entirely. This allows systematic cache exhaustion and forces repeated upstream database fetches for semantically identical block queries.

### Finding Description

**Exact code path:**

`cacheKeyGenerator()` at [1](#0-0)  hashes `req.originalUrl` verbatim:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

The code comment at line 149 explicitly acknowledges the absence of normalization: [2](#0-1) 

**Root cause — the normalization gap:**

`server.js` sets Express's query parser to `requestQueryParser` at line 60: [3](#0-2) 

`requestQueryParser` lowercases all query parameter keys into `req.query`: [4](#0-3) 

However, `req.originalUrl` is an Express immutable property — it always reflects the raw URL string as received from the client. No middleware in the pipeline ever rewrites it. So `?Limit=1` and `?limit=1` produce different MD5 digests despite resolving to identical `req.query` objects.

**Middleware ordering confirms the gap:**

`responseCacheCheckHandler` is registered at line 97, before any route handler: [5](#0-4) 

`normalizeRequestQueryParams` (from `requestNormalizer.js`) is only called inside route handlers using `req.query` — it is never applied to `req.originalUrl` before the cache key is computed: [6](#0-5) 

**Why OpenAPI validation does not block this:**

The OpenAPI validator is initialized before `responseCacheCheckHandler` and uses `req.query` (already normalized by `requestQueryParser`) for parameter validation. A request with `?Limit=1` passes validation because `req.query` contains `{limit: '1'}`, yet `req.originalUrl` still carries `?Limit=1`, producing a distinct cache key.

**Exploit flow:**

1. Attacker sends `GET /api/v1/blocks?limit=1` → cache miss → DB query → result stored under key `md5("/api/v1/blocks?limit=1")-v1`.
2. Attacker sends `GET /api/v1/blocks?Limit=1` → different MD5 → cache miss → another DB query → result stored under a second key.
3. Repeat with `?LIMIT=1`, `?lImIt=1`, etc. — each variant is a distinct cache entry.
4. Redis fills with duplicate entries; every variant permanently bypasses the warm cache.

### Impact Explanation

The concrete impact is **cache bypass and upstream database amplification**, not a literal 500%+ block-time delay. Every case-variant request misses the Redis cache and triggers a full database query for block data. An attacker with no credentials can:

- Systematically exhaust Redis memory by storing O(N) duplicate entries for the same logical query.
- Force the database to serve repeated identical queries, degrading throughput for all users.
- Ensure that legitimate cached responses are never reused for their variant keys, permanently reducing cache hit rates for block endpoints.

Severity: **Medium** — no authentication required, directly degrades caching infrastructure, but does not corrupt data or expose secrets.

### Likelihood Explanation

Exploitation requires only the ability to send HTTP GET requests — no credentials, no special headers, no rate-limit bypass. The attack is trivially scriptable: a loop over case permutations of a single parameter name (`limit`, `order`, `block.number`) generates dozens of distinct cache keys. It is repeatable indefinitely and requires no knowledge of internal state.

### Recommendation

Replace the raw `req.originalUrl` hash in `cacheKeyGenerator()` with a normalized form. The normalizer already exists — apply `normalizeRequestQueryParams` (or an equivalent that lowercases parameter names and sorts them canonically) to produce the cache key string before hashing:

```js
const cacheKeyGenerator = (req) => {
  const normalized = normalizeRequestQueryParams(
    req.openapi?.openApiRoute,
    req.path,
    req.query  // already lowercased by requestQueryParser
  );
  return crypto.createHash('md5').update(normalized).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

This is consistent with the existing TODO comment referencing issue 9113. [7](#0-6) 

### Proof of Concept

```bash
BASE="http://localhost:5551/api/v1/blocks"

# Populate cache with canonical key
curl -s "$BASE?limit=1&order=desc" > /dev/null

# Each of the following misses the cache and hits the DB:
curl -s "$BASE?Limit=1&order=desc" > /dev/null   # different MD5
curl -s "$BASE?LIMIT=1&order=desc" > /dev/null   # different MD5
curl -s "$BASE?limit=1&Order=desc" > /dev/null   # different MD5
curl -s "$BASE?Limit=1&Order=desc" > /dev/null   # different MD5

# Verify: each response has a fresh (non-cached) latency profile,
# and Redis KEYS * shows 5 distinct entries for the same logical query.
```
