### Title
Silent TCP Connection Resource Exhaustion (FD/Channel Leak) via Missing Connection-Level Timeouts in `grpcServerConfigurer`

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only `maxConcurrentCallsPerConnection`, leaving no `handshakeTimeout`, `maxConnectionIdle`, or `maxConnectionAge` set. Because the server listens on port 5600 without TLS (plain HTTP/2), there is no TLS handshake timeout to fall back on. An unprivileged attacker can open an unbounded number of TCP connections and send no data, causing the server to hold each connection indefinitely — consuming one OS file descriptor and one Netty `Channel` object per connection — until the process exhausts its file descriptor limit and can no longer accept legitimate connections.

### Finding Description
**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection` (default 5); no other Netty server parameters are modelled or applied:

```java
// NettyProperties.java — only field
private int maxConcurrentCallsPerConnection = 5;
```

`NettyServerBuilder` Netty-gRPC defaults when none of these are set:
- `handshakeTimeoutMillis` — applies **only** to TLS; irrelevant for plain TCP.
- `maxConnectionIdle` — `Long.MAX_VALUE` (never closes idle connections).
- `maxConnectionAge` — `Long.MAX_VALUE` (connections live forever).
- `keepAliveTime` — `Long.MAX_VALUE` (server never sends PING frames to detect dead peers).

The docker-compose proxy confirms the server is reached over plain gRPC (`grpc://grpc_host`, not `grpcs://`), so no TLS handshake timeout applies. When a TCP connection is accepted but the client never sends the HTTP/2 client preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`), Netty parks the channel in its event loop indefinitely. The OS file descriptor for the socket remains open, and the Netty `NioSocketChannel` object is never released.

**Why `maxConcurrentCallsPerConnection` does not help:** This setting caps the number of concurrent gRPC *calls* within a single already-established connection. It has no effect on the number of *connections* the server will accept, and it is never evaluated for a connection that has not yet sent any data.

**Exploit flow:**
1. Attacker opens a raw TCP socket to `<server>:5600`.
2. Completes the TCP three-way handshake (no privilege required).
3. Sends zero bytes thereafter.
4. Server allocates one FD + one Netty `Channel`; no timer fires to reclaim them.
5. Attacker repeats in a loop (e.g., 10 000 goroutines/threads).
6. Server process hits its `ulimit -n` (commonly 65 535 in containers); `accept()` starts returning `EMFILE`/`ENFILE`.
7. All legitimate gRPC clients receive connection-refused or timeout errors.

### Impact Explanation
Complete denial of service for all gRPC consumers (topic subscription, address-book queries). File descriptor exhaustion also prevents the JVM from opening new database connections, log file handles, and internal pipes, potentially cascading into broader service failure. The `GrpcHighFileDescriptors` Prometheus alert (threshold 80 %) would fire, but only after significant damage is already done and legitimate traffic is already being dropped.

### Likelihood Explanation
No authentication, no TLS client certificate, and no network-layer rate limit is required. Any host with TCP reachability to port 5600 can execute this attack with a trivial script (e.g., `for i in $(seq 10000); do nc -z <host> 5600 & done`). The attack is repeatable, requires no special knowledge of the gRPC protocol, and is not mitigated by the existing `maxConcurrentCallsPerConnection(5)` guard. In a Kubernetes environment where the gRPC service is exposed via a LoadBalancer or NodePort, the attack surface is internet-wide.

### Recommendation
Add the following calls inside the `grpcServerConfigurer` lambda, and expose the corresponding fields in `NettyProperties` so they are operator-tunable:

```java
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.MILLISECONDS);
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.MILLISECONDS);
serverBuilder.handshakeTimeout(nettyProperties.getHandshakeTimeout(), TimeUnit.MILLISECONDS);
serverBuilder.keepAliveTime(nettyProperties.getKeepAliveTime(), TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(nettyProperties.getKeepAliveTimeout(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Recommended safe defaults:
- `handshakeTimeout`: 10 s (closes connections that never send the HTTP/2 preface)
- `maxConnectionIdle`: 300 s (reclaims idle connections)
- `maxConnectionAge`: 3600 s (prevents indefinitely long-lived connections)
- `keepAliveTime`: 60 s / `keepAliveTimeout`: 20 s (detects dead peers)

Additionally, enforce a per-IP connection limit at the infrastructure layer (e.g., `iptables --connlimit-above` or an Envoy/Istio `connectionPool` policy).

### Proof of Concept
```bash
# Requires only bash + nc (netcat); no privileges needed.
# Open 5000 silent TCP connections to the gRPC port and hold them open.
for i in $(seq 1 5000); do
    nc <grpc-host> 5600 &   # connects, sends nothing, holds FD open
done

# On the server, observe FD exhaustion:
