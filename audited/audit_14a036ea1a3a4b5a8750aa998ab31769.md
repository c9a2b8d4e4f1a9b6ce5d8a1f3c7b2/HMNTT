### Title
Unauthenticated Mirror Node HTTP Response Poisons Operator Transaction Routing

### Summary
`fetchMirrorNodeNetwork()` issues a plain, unauthenticated HTTP GET to the configured mirror node URL and unconditionally trusts the returned `nodesEnvelope` JSON to build the entire gRPC node map. An attacker who can intercept or spoof that HTTP response (via BGP hijacking, DNS poisoning, or ARP spoofing on the same network segment when the URL is plaintext `http://`) can inject attacker-controlled gRPC endpoints. The pinger then uses the operator's private key to sign and submit every subsequent `CryptoTransfer` transaction directly to those attacker-controlled nodes.

### Finding Description
**Code path:**

- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 46: a bare `http.Client` is constructed with only a timeout — no TLS configuration, no certificate pinning, no mutual TLS.
- Lines 79–84: `http.NewRequestWithContext` issues a GET with zero authentication headers (no API key, no bearer token, no HMAC).
- Lines 90–93: the only server-side check is the HTTP status code; any 2xx response is accepted.
- Lines 95–98: the body is decoded into `nodesEnvelope` with no signature, no hash, no schema validation.
- Lines 100–124: every `(host, port)` pair from the attacker-supplied JSON is inserted verbatim into the `network` map.
- `pinger/sdk_client.go`, lines 18–22: when `network == "other"`, `hiero.ClientForNetwork(netmap)` creates the Hiero SDK client from that poisoned map, and `client.SetOperator(opID, opKey)` attaches the operator's private key to it.
- `pinger/transfer.go`, lines 29–33: `cryptoTransfer.Execute(client)` signs and submits a `CryptoTransfer` to every node in the poisoned map on every ticker tick.

**Root cause:** The code assumes the mirror node REST endpoint is a trusted, integrity-protected source of network topology. There is no transport-layer authentication (mTLS), no application-layer authentication (API key/token), and no response integrity check (signature/hash). The `http.Client` at line 46 of `mirror_node_client.go` uses Go's default settings, meaning it will follow redirects and accept any valid TLS certificate from a public CA — or, for `http://` URLs, no encryption at all.

**Failed assumption:** The operator-configured `HIERO_MIRROR_PINGER_REST` URL is assumed to always resolve to a trustworthy server. The example value in `config.go` line 37 is `http://mirror-rest:5551` — plaintext HTTP — making the assumption false even for a local-network attacker.

### Impact Explanation
Once the pinger's `hiero.Client` is initialized with attacker-controlled gRPC endpoints, every `CryptoTransfer` signed by the operator's key is delivered to the attacker's server. The attacker can:

1. **Observe** the full signed transaction bytes, including the operator account ID, destination, and amount — a privacy breach.
2. **Drop** all transactions silently, causing a complete DoS of the pinger's monitoring function.
3. **Replay** the signed transactions to the real Hiero network at will (the transaction ID includes a valid start time window), causing duplicate charges against the operator account.
4. **Redirect** transactions: because the attacker controls the gRPC endpoint, they can forward a modified transaction body (changing destination account) before the operator's signature is verified by the real network — though the signature would then fail, this still constitutes unauthorized transaction submission attempts.

The operator's private key is never directly exposed, but every signed transaction payload is delivered to the attacker for the lifetime of the pinger process (it is initialized once at startup and never refreshed).

### Likelihood Explanation
The attack is realistic under several common conditions:

- **Plaintext HTTP deployment** (the documented default `http://mirror-rest:5551`): any attacker with access to the same Kubernetes node, pod network, or cloud VPC subnet can perform ARP spoofing or a simple TCP hijack — no BGP capability required.
- **DNS poisoning**: if the mirror node is addressed by hostname, a DNS cache poisoning attack (e.g., Kaminsky-style) against the resolver used by the pinger pod is sufficient.
- **BGP hijacking**: for internet-facing mirror node URLs, a BGP prefix hijack (demonstrated repeatedly in real-world incidents) redirects traffic at the routing layer.

The pinger is a long-running daemon that initializes the network map exactly once (`newClient` is called once in `main.go` line 41), so a single successful poisoning at startup permanently compromises all subsequent transactions for the process lifetime.

### Recommendation

1. **Enforce HTTPS with certificate validation**: reject any `mirrorRest` URL that does not use `https://`. Configure the `http.Transport` with a pinned CA or a specific `tls.Config` rather than relying on the system default.
2. **Add response integrity verification**: require the mirror node to sign its `/network/nodes` response (e.g., via a JWS envelope or an HMAC header), and verify the signature before consuming the payload.
3. **Add an API key or bearer token**: even a shared secret in a request header raises the bar significantly above a pure network-level attacker.
4. **Validate returned endpoints against an allowlist**: cross-check returned node account IDs and IP ranges against a locally configured allowlist before inserting them into the network map.
5. **Periodic refresh with change detection**: if the network map changes unexpectedly between refreshes, log an alert and refuse to use the new map without operator confirmation.

### Proof of Concept

**Preconditions:**
- Pinger is deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plaintext HTTP, the documented example).
- Attacker has network-level access to intercept traffic between the pinger pod and `mirror-rest:5551` (e.g., same Kubernetes node, compromised sidecar, or ARP spoofing on the pod network).

**Steps:**

1. Attacker starts a rogue HTTP server on a controlled IP, listening on port 5551.
2. Attacker poisons ARP/DNS so that `mirror-rest` resolves to the rogue server's IP.
3. Rogue server responds to `GET /api/v1/network/nodes` with:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{
         "ip_address_v4": "<attacker_ip>",
         "port": 50211
       }]
     }],
     "links": {"next": null}
   }
   ```
4. Pinger starts, calls `buildNetworkFromMirrorNodes` → `fetchMirrorNodeNetwork` → receives the crafted response → builds `network = {"<attacker_ip>:50211": AccountID{0,0,3}}`.
5. `hiero.ClientForNetwork(network)` creates a client pointing solely to `<attacker_ip>:50211`.
6. On every ticker tick, `submitWithRetry` calls `cryptoTransfer.Execute(client)`, which signs the transaction with the operator's private key and sends the signed gRPC `CryptoTransfer` request to `<attacker_ip>:50211`.
7. Attacker's gRPC server logs the full signed transaction bytes and can replay them to the real Hiero network within the transaction's valid time window.
