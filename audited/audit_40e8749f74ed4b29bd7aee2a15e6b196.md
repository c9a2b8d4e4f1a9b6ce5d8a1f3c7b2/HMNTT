### Title
Unbounded Batch DB Query Amplification via Slot Cache Poisoning in `findStorageBatch()`

### Summary
An unprivileged external user can send `eth_call` requests targeting a contract with arbitrary (non-existent) storage slot keys, causing each new slot to be permanently registered in the per-contract `slotsPerContract` cache. Because `findStorageBatch()` unconditionally queries the DB with **all** currently cached slot keys on every cache miss, and because non-existent slots are never written to `contractStateCache`, the DB query size grows to up to 1500 slots per request and stays there indefinitely — producing a sustained, up-to-1500× amplification of database load with no authentication required.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` (TTL=2s) first. For any slot not present there, it delegates to `findStorageBatch()` (lines 85–122).

Inside `findStorageBatch()`:

```
Line 90:  contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
Line 91:  final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
Line 93-101: builds cachedSlots list from ALL cached keys
Line 103: contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

Every call to `findStorageBatch()` issues a DB query containing **every slot key ever seen for that contract** (up to `slotsPerContract` max of 1500). The query is:

```sql
SELECT slot, value FROM contract_state WHERE contract_id = :contractId AND slot IN (:slots)
```

**Root cause — failed assumption:** The design assumes that after the first batch query, results are cached in `contractStateCache` (TTL=2s), so subsequent requests for the same slot hit the cache and bypass `findStorageBatch()`. This assumption fails for **non-existent slots**: `findStorageBatch()` only writes to `contractStateCache` for slots that actually exist in the DB (lines 106–113). A slot key that returns no DB row is never written to `contractStateCache`. Therefore, every subsequent request for a non-existent slot re-enters `findStorageBatch()` and re-issues the full batch query with all 1500 cached keys.

**Exploit flow:**

1. Attacker sends ~1500 `eth_call` requests to any existing contract, each reading a different 32-byte slot key that does not exist in `contract_state`. Each request adds one new key to the per-contract `slotsPerContract` cache (TTL=5m, max=1500).
2. Cache is now full: 1500 non-existent slot keys are registered.
3. Attacker continues sending `eth_call` requests at the rate limit (500 RPS). Each request for any of the 1500 cached slots:
   - Misses `contractStateCache` (never populated for non-existent slots)
   - Enters `findStorageBatch()`
   - Issues a DB query with all 1500 cached slot keys
4. DB receives 500 × 1500 = **750,000 slot lookups per second** instead of the baseline 500.
5. The `slotsPerContract` cache TTL is 5 minutes (`expireAfterAccess=5m`), so the amplified state persists as long as the attacker keeps accessing the slots.

**Why existing checks are insufficient:**

- `contractStateCache` (TTL=2s): Only caches slots that exist in the DB. Non-existent slots are never cached here, so the check at lines 63–67 always falls through to `findStorageBatch()`.
- Rate limiter (500 RPS / gas throttle): Throttles request count, not DB query complexity. The amplification multiplier (up to 1500×) is invisible to the throttle.
- `statementTimeout` (3000ms): A batch query with 1500 non-existent slots returns an empty result set quickly (index scan), so it completes well within the timeout while still consuming DB I/O and connection resources.
- `enableBatchContractSlotCaching` flag (default=true): The flag is a kill-switch but is enabled by default and not documented as a security control.

### Impact Explanation

The DB receives up to 1500× more slot-lookup work per second than the rate limiter is designed to allow. At 500 RPS with a full 1500-slot cache, the DB processes 750,000 slot lookups/second instead of 500. This can saturate DB CPU, I/O, and connection pool resources, degrading or denying service for all users of the mirror node Web3 API. The `slotsPerContract` cache persists for 5 minutes per access, so the attacker only needs to maintain a low-rate keep-alive to sustain the amplified state indefinitely. The impact is a sustained, server-side resource exhaustion (DoS) without any on-chain cost or authentication.

### Likelihood Explanation

The attack requires no credentials, no on-chain funds, and no special knowledge beyond the address of any contract deployed on the network. The attacker needs only an HTTP client capable of sending JSON-RPC `eth_call` requests. The slot keys can be arbitrary 32-byte values; they do not need to correspond to real storage. The attack is fully repeatable and can be automated with a trivial script. The 5-minute cache TTL means the amplified state is self-sustaining with minimal ongoing effort.

### Recommendation

1. **Cache empty results in `contractStateCache`:** When `findStorageBatch()` finds no DB row for a requested slot, explicitly write a sentinel value (e.g., `EMPTY_VALUE`) to `contractStateCache` for that slot key. This ensures the `contractStateCache` check at line 63–67 short-circuits on the next request, preventing re-entry into `findStorageBatch()`.

2. **Cap the batch query size per request:** Instead of querying all cached slot keys on every miss, limit the batch to a fixed maximum (e.g., 50–100 slots) per invocation, regardless of cache size.

3. **Separate the "new slot" path from the "batch prefetch" path:** Only include previously-seen slots in the batch query when there is evidence they will be re-accessed (e.g., within the same EVM call frame), rather than accumulating all historical slot keys across unrelated requests.

4. **Consider per-IP or per-contract rate limiting** at the application layer to limit the rate at which new slot keys can be registered in the cache from a single source.

### Proof of Concept

```python
import requests, json, threading

TARGET = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<any_deployed_contract_address>"

def eth_call(slot_index):
    slot_hex = "0x" + format(slot_index, '064x')  # non-existent slot key
    payload = {
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT, "data": "0x" + "20" + slot_hex[2:]}, "latest"],
        "id": slot_index
    }
    requests.post(TARGET, json=payload)

# Phase 1: Fill slotsPerContract cache with 1500 unique non-existent slot keys
for i in range(1500):
    eth_call(i)

# Phase 2: Sustain amplified load — each request triggers a 1500-slot batch DB query
# At 500 RPS (rate limit), DB receives 750,000 slot lookups/second
while True:
    threads = [threading.Thread(target=eth_call, args=(i % 1500,)) for i in range(500)]
    for t in threads: t.start()
    for t in threads: t.join()
```

1. Start the mirror node Web3 API with default configuration (`enableBatchContractSlotCaching=true`, `slotsPerContract` max=1500).
2. Run Phase 1 to fill the per-contract slot cache with 1500 non-existent slot keys.
3. Observe that each subsequent `eth_call` in Phase 2 triggers a `findStorageBatch` DB query with 1500 slots (verifiable via DB query logs or `pg_stat_activity`).

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
