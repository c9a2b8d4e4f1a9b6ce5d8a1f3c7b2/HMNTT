### Title
Unauthenticated Forced Historical Token Balance Query via Timestamp Parameter Causes Expensive `DISTINCT ON` Database Scan

### Summary
Any unauthenticated user can supply a `?timestamp=lte:<valid_snapshot_ts>` parameter to `GET /api/v1/accounts/:id` to force `getEntityBalanceQuery()` into the `needHistoricalBalanceInfo` branch, which builds a `SELECT DISTINCT ON (token_id) ... FROM token_balance ORDER BY token_id, consensus_timestamp DESC` subquery against the large, time-partitioned `token_balance` table. This path is significantly more expensive than the non-historical path (which reads from the small `token_account` current-state table), and no per-user rate limiting exists on the REST API, allowing an attacker to repeatedly trigger it at will.

### Finding Description

**Exact code path:**

In `rest/accounts.js`, `getOneAccount()` accepts a `timestamp` query parameter (declared in `acceptedSingleAccountParameters` at line 526). When `timestampFilters.length > 0`, it calls `balances.getAccountBalanceTimestampRange()` at line 435. If a valid balance snapshot is found (`upper !== undefined`, line 440), `accountBalanceQuery.query` is set to a non-empty string at lines 446–454. [1](#0-0) 

This non-empty `accountBalanceQuery.query` flows into `getEntityBalanceQuery()`, where line 164 evaluates:

```js
const needHistoricalBalanceInfo = accountBalanceQuery.query || accountBalanceQuery.forceUnionEntityHistory;
``` [2](#0-1) 

When `needHistoricalBalanceInfo` is truthy and `accountBalanceQuery.query` is non-empty, the function builds the expensive subquery at lines 170–180:

```sql
SELECT DISTINCT ON (token_id) token_id, balance
FROM token_balance
WHERE account_id = $1
  AND consensus_timestamp >= $2
  AND consensus_timestamp <= $3
ORDER BY token_id ASC, consensus_timestamp DESC
LIMIT N
``` [3](#0-2) 

**Root cause:** The `ORDER BY token_id, consensus_timestamp DESC` sort order does not align with a simple index scan on `(account_id, consensus_timestamp)`. PostgreSQL must fetch all rows for the account in the timestamp range, sort them by two columns, and then deduplicate — a significantly more expensive operation than the non-historical path, which reads from `token_account` (current state, `associated IS TRUE`, single-column sort): [4](#0-3) 

**No rate limiting on the REST API:** A grep across `rest/**/*.js` for rate-limiting constructs returns only a single match in a test utility file. The REST API uses `pool.queryQuietly()` with no per-user throttle, unlike the `web3/` service which has explicit bucket-based throttling. [5](#0-4) 

### Impact Explanation

The `token_balance` table is time-partitioned and grows continuously. For accounts with many token associations and many historical snapshots, the `DISTINCT ON` + two-column sort forces a large in-memory sort before deduplication. An attacker can issue this request in a tight loop against any account ID with a known historical snapshot, exhausting database CPU and connection pool resources, degrading or denying service for all legitimate users. The impact is classified as medium griefing/DoS with no direct economic damage to network participants.

### Likelihood Explanation

The attack requires zero privileges — only a valid account ID (trivially enumerable from the public API) and a timestamp that resolves to a balance snapshot (snapshots are generated every ~15 minutes and are permanently stored; any past timestamp works). The exploit is fully repeatable, scriptable, and requires no on-chain interaction or special knowledge. The absence of per-user rate limiting on the REST API makes sustained abuse straightforward.

### Recommendation

1. **Add per-user/per-IP rate limiting** to the REST API layer (e.g., using a token-bucket middleware similar to the `web3/` throttle implementation in `ThrottleConfiguration.java`).
2. **Add a statement timeout** on the `pool.queryQuietly()` call for historical balance queries to bound worst-case query duration.
3. **Consider adding a covering index** on `token_balance (account_id, token_id, consensus_timestamp DESC)` so the `DISTINCT ON` sort can be satisfied by an index scan rather than a full sort.
4. **Optionally restrict** the timestamp range window (e.g., reject requests where the resolved snapshot is older than N days) to limit the scan range.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly accessible. At least one balance snapshot exists (always true on mainnet/testnet).

**Step 1 — Discover a valid snapshot timestamp:**
```
GET /api/v1/balances?limit=1
```
Note the returned `timestamp` value, e.g. `1700000000.000000000`.

**Step 2 — Trigger the expensive historical path in a loop:**
```bash
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.98?timestamp=lte:1700000000" > /dev/null &
done
```

**Step 3 — Observe:** Each request forces `getOneAccount()` → `getAccountBalanceTimestampRange()` to return a valid `upper`, setting `accountBalanceQuery.query` non-empty, which causes `getEntityBalanceQuery()` to emit the `SELECT DISTINCT ON (token_id) ... FROM token_balance ORDER BY token_id, consensus_timestamp DESC` subquery. With no rate limiting, concurrent requests saturate the database connection pool and degrade response times for all users.
