### Title
Unauthenticated Request Flood on `GET /contracts` Exhausts DB Connection Pool Due to Missing Application-Level Rate Limiting

### Summary
The `getContracts()` handler in `rest/controllers/contractController.js` issues a DB query against the `entity` table on every request with no server-side rate limiting at the route or application middleware level. An unprivileged external user can send `GET /contracts?limit=100&order=asc` in a tight loop, saturating the default 10-connection pool, queuing Node.js requests until `connectionTimeoutMillis` (20 s) is reached, and denying service to legitimate callers.

### Finding Description

**Route registration** — no rate-limiting middleware is attached: [1](#0-0) 

`extendExpress` only wraps handlers for async error propagation; it adds no throttling: [2](#0-1) 

**Handler** — every request unconditionally issues a DB query: [3](#0-2) 

**Query shape** — `WHERE e.type = 'CONTRACT'` with a `LEFT JOIN contract` and `ORDER BY e.id` is issued on every call. Without a covering index on `(type, id)` this degrades to a sequential scan of the entire `entity` table: [4](#0-3) 

**Middleware stack** — the only "auth" middleware sets a higher response-size limit for known users; it performs no rate limiting and returns immediately for unauthenticated callers: [5](#0-4) 

No `express-rate-limit` or equivalent is imported anywhere in `rest/**/*.js` (grep confirmed a single hit only in a test utility file).

**DB pool defaults** — 10 connections, 20 s statement timeout, 20 s connection-wait timeout: [6](#0-5) [7](#0-6) 

**Response-limit cap** — `getLimitParamValue` caps rows returned at `max=100` but does nothing to limit request rate: [8](#0-7) 

**Infrastructure rate limiting is optional** — the Traefik middleware chart is gated on `{{ if and .Values.global.middleware .Values.middleware }}` and is absent from the REST chart by default, so it cannot be relied upon as a compensating control: [9](#0-8) 

### Impact Explanation
With a pool of 10 connections and a 20 s statement timeout, an attacker holding all 10 connections forces every subsequent legitimate request to queue for up to 20 s before receiving a connection-timeout error. Concurrent flooding also grows the Node.js in-process queue unboundedly, increasing heap pressure. The result is complete or near-complete denial of the REST API for all users, including those querying other endpoints that share the same pool.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. The endpoint is publicly documented in the OpenAPI spec. A single machine with a modest HTTP client (e.g., `wrk`, `ab`, or a simple `async` loop) can sustain thousands of requests per second. The attack is trivially repeatable and requires no special knowledge beyond the public API documentation.

### Recommendation
1. Add an application-level rate-limiting middleware (e.g., `express-rate-limit` with a Redis store for multi-instance deployments) applied globally in `rest/server.js` before route handlers, with a per-IP limit appropriate for public endpoints.
2. Ensure the Traefik `inFlightReq` + `rateLimit` middleware is enabled and enforced in all deployment configurations, not left as an opt-in.
3. Add a composite index on `entity(type, id)` to prevent sequential scans when `WHERE e.type = 'CONTRACT'` is evaluated without a selective leading predicate.
4. Consider reducing `db.pool.maxConnections` default or adding a per-request concurrency semaphore so a flood cannot hold all connections simultaneously.

### Proof of Concept
```bash
# Requires: wrk (or any HTTP load tool), no credentials needed
wrk -t 10 -c 100 -d 60s \
  "http://<mirror-node-host>:5551/api/v1/contracts?limit=100&order=asc"

# Expected result within seconds:
# - Legitimate requests to /api/v1/contracts (and other endpoints sharing the pool)
#   begin returning 500 errors or hanging for ~20 s
# - DB pool log shows: "error event emitted on pool" / connection timeout errors
# - Node.js process RSS grows as queued requests accumulate
```

```javascript
const getEffectiveMaxLimit = () => {
  const userLimit = httpContext.get(userLimitLabel);
  return userLimit !== undefined ? userLimit : responseLimit.max;
};

/**
 * Gets the limit param value, if not exists, return the default; otherwise cap it at max.
 * Note if values is an array, the last one is honored.
 * @param {string[]|string} values Values of the limit param
 * @return {number}
 */
const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
```
