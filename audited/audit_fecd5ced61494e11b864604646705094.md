### Title
Cache Poisoning via Incorrect `unless` Condition on `Optional` Return Type in `findByTimestamp`

### Summary
`RecordFileRepository.findByTimestamp` uses `unless = "#result == null"` to guard its `@Cacheable` annotation, but the method returns `Optional<RecordFile>`. In Spring Cache, `#result` refers to the `Optional` object itself — which is never `null` — so `Optional.empty()` results are always cached. An unprivileged attacker can pre-poison the timestamp cache by querying timestamps for record files not yet ingested, causing those lookups to permanently return empty for up to 10 minutes (renewable indefinitely by re-querying).

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 38–46.

```java
@Caching(
        cacheable =
                @Cacheable(
                        cacheNames = CACHE_NAME,
                        cacheManager = CACHE_MANAGER_RECORD_FILE_TIMESTAMP,
                        unless = "#result == null"),          // ← broken guard
        put = @CachePut(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX))
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
```

**Root cause:** Spring Cache evaluates the `unless` SpEL expression with `#result` bound to the raw method return value — the `Optional` object. `Optional.empty() == null` is `false`, so the guard never fires for absent results. Spring then unwraps the `Optional` and stores `null` (wrapped in Caffeine's `NullValue`) in the cache. On the next call with the same key, the cached `null` is retrieved and re-wrapped into `Optional.empty()`, bypassing the database entirely.

**Cache configuration:** `CACHE_MANAGER_RECORD_FILE_TIMESTAMP` uses `expireAfterAccess(10, TimeUnit.MINUTES)` with `maximumSize(10000)`. Because expiry is access-based, an attacker who periodically re-queries the same timestamp resets the TTL, keeping the poisoned entry alive indefinitely.

**Why existing checks fail:** The only guard is `unless = "#result == null"`. Since `Optional.empty()` is a non-null object, this condition is always `false` for empty results, making the guard completely ineffective for this return type. The same incorrect pattern appears on `findEarliest` (line 23), `findByIndex` (line 27), and `findLatest` (line 34), but those are less exploitable because their keys are not attacker-controlled.

### Impact Explanation
Any JSON-RPC call that resolves a block or transaction by timestamp (e.g., `eth_getTransactionByHash`, `eth_getBlockByNumber`, historical `eth_call`) routes through `RecordFileServiceImpl.findByTimestamp` → `RecordFileRepository.findByTimestamp`. A poisoned cache entry causes the service layer to receive `Optional.empty()`, which propagates as a "block not found" or "transaction not found" error to the caller. Because the mirror node is continuously ingesting new record files, the window between ingestion and a pre-emptive cache-miss query is routinely exploitable on a live node. The attacker can target any of up to 10,000 distinct timestamps simultaneously (the cache's `maximumSize`).

### Likelihood Explanation
No authentication or rate limiting is required. The attacker only needs to know approximate upcoming consensus timestamps (predictable from the ~2-second block interval on Hedera) and issue standard JSON-RPC requests before the corresponding record file is ingested. The attack is repeatable, scriptable, and requires no special privileges. Keeping the poisoned entries alive requires only periodic re-queries to reset the `expireAfterAccess` timer.

### Recommendation
Change the `unless` condition to explicitly handle `Optional` empty results:

```java
unless = "#result == null || !#result.isPresent()"
```

Apply the same fix to `findEarliest`, `findByIndex`, and `findLatest` for consistency. Additionally, add an `unless` condition to the `@CachePut` on line 44 to prevent storing empty results in the index cache:

```java
put = @CachePut(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
    unless = "#result == null || !#result.isPresent()")
```

### Proof of Concept
1. Observe the current Hedera consensus timestamp `T` (e.g., via the REST API).
2. Compute `T_future = T + 3_000_000_000` (approximately 3 seconds ahead, before the next record file is ingested).
3. Send an `eth_getBlockByNumber` JSON-RPC call (or any call that internally invokes `findByTimestamp(T_future)`) to the mirror node web3 endpoint. The call returns empty/not-found, and `Optional.empty()` is cached under key `T_future`.
4. Wait for the Hedera network to produce the block whose `consensusEnd >= T_future` (within ~2–3 seconds).
5. Send the same `eth_getBlockByNumber` call again. Despite the record file now existing in the database, the mirror node returns the cached empty result.
