### Title
Unbounded Cache Eviction via Unauthenticated Token Relationship Requests

### Summary
The `getCachedTokens()` function in `rest/service/tokenService.js` populates a process-level `quickLru` token cache with a fixed `maxSize` and no per-request rate limiting. An unauthenticated attacker can repeatedly query the public `/accounts/:id/tokens` endpoint using accounts associated with many unique token IDs, continuously filling the cache with new entries and evicting previously cached tokens, forcing subsequent legitimate requests to re-query the database.

### Finding Description
**Code location:** `rest/service/tokenService.js`, lines 12–14 (cache init) and lines 136–161 (`getCachedTokens`).

The cache is initialized as:
```js
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,
});
```
No `maxAge` is set, so eviction is purely LRU-based on size.

In `getCachedTokens()` (lines 136–161), for every token ID not already in the cache, the function queries the DB and unconditionally writes the result into `tokenCache`:
```js
rows.forEach((row) => {
  tokenCache.set(tokenId, cachedToken);   // line 156
  cachedTokens.set(tokenId, cachedToken);
});
```

The public controller endpoint `getTokenRelationships` in `rest/controllers/tokenController.js` (lines 66–92) requires no authentication and calls `TokenService.getTokenAccounts(query)`, which in turn calls `getCachedTokens()` with the token IDs associated with the queried account.

**Root cause:** There is no rate limiting on cache population. Any unauthenticated caller can introduce new token IDs into the shared process-level LRU cache. Once `maxSize` is reached, `quickLru` silently evicts the least-recently-used entry. By cycling through accounts associated with many distinct tokens, an attacker can keep the cache in a perpetual churn state.

**Failed assumption:** The design assumes organic traffic will keep the working set within `maxSize`. It does not account for adversarial traffic deliberately introducing high-cardinality token ID sets.

### Impact Explanation
Every evicted cache entry means the next legitimate request for that token must issue a synchronous DB query (`tokenCacheQuery` against the `token` table). Under sustained attack, the effective cache hit rate approaches zero, multiplying DB query load proportionally to the number of unique tokens in flight. This degrades response latency and increases DB connection pressure for all users. There is no data corruption or financial loss — the impact is purely availability/performance degradation (griefing).

### Likelihood Explanation
Hedera is a public network with a publicly queryable mirror node. Token associations are on-chain public data. An attacker can enumerate accounts with large token portfolios (e.g., DeFi wallets, exchange accounts) using the same REST API. No credentials, API keys, or special privileges are required. The attack is trivially scriptable with a simple HTTP client loop and is repeatable indefinitely.

### Recommendation
1. **Add per-IP or global rate limiting** on the `/accounts/:id/tokens` endpoint to bound the rate at which new token IDs can be introduced into the cache.
2. **Set a `maxAge`** on the `quickLru` instance so that stale entries expire naturally and the cache self-heals after an attack window.
3. **Cap the number of unique token IDs written per request** — if a single account has an unusually large number of token associations, consider not caching them (or caching only up to a threshold).
4. **Monitor cache hit rate** as an operational metric; a sudden drop is an early indicator of this attack.

### Proof of Concept

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
