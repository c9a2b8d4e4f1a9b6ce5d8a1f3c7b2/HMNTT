### Title
Gas Token Leak on REJECT Filter Path Enables gasLimitBucket Exhaustion DoS

### Summary
In `ThrottleManagerImpl.throttle()`, gas tokens are consumed from `gasLimitBucket` before request filters are evaluated. When a `REJECT` filter fires, it throws a `ThrottleException` that propagates directly to `GenericControllerAdvice` without any `restore()` call, permanently losing those tokens until the bucket naturally refills. An unprivileged attacker who knows (or can probe) the REJECT filter criteria can flood the endpoint with high-gas requests matching those criteria, draining the gas bucket and blocking legitimate callers.

### Finding Description

**Code path:**

`ThrottleManagerImpl.throttle()` — [1](#0-0) 

1. Line 40: `gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))` — tokens are irrevocably consumed.
2. Lines 44–48: request filters are iterated.
3. Line 69: `case REJECT -> throw new ThrottleException("Invalid request")` — exception thrown with no preceding `restore()`. [2](#0-1) 

**Why restore never happens on the REJECT path:**

`ContractController.call()` calls `throttleManager.throttle(request)` at line 40. The only `catch` block in that method is for `InvalidParametersException` (lines 46–49), which explicitly calls `throttleManager.restore(request.getGas())`. `ThrottleException` is not caught there; it propagates to `GenericControllerAdvice.throttleException()` (line 135), which simply returns HTTP 429 — no restore. [3](#0-2) 

The `restoreGasToBucket()` in `ContractCallService.doProcessCall()` (lines 127–135) is only reached when `callContract()` is invoked, which never happens when `throttle()` throws. [4](#0-3) 

**Root cause:** The developer correctly added a restore on `InvalidParametersException` but omitted an equivalent restore for `ThrottleException` originating from REJECT filters, creating an asymmetric gas-token leak.

### Impact Explanation

Every request that hits a REJECT filter with gas > 10,000 (the `GAS_SCALE_FACTOR` threshold below which `scaleGas` returns 0) permanently removes `floor(gas / 10_000)` tokens from the shared `gasLimitBucket` until the per-second refill recovers them. [5](#0-4) 

With default settings (`gasPerSecond = 7,500,000,000`, bucket capacity = 750,000 scaled tokens, max gas = 15,000,000 → 1,500 tokens/request), an attacker sending requests at the rate limit (500 req/s) with maximum gas consumes 750,000 tokens/s — exactly the refill rate — keeping the bucket perpetually empty. Legitimate callers receive `GAS_PER_SECOND_LIMIT_EXCEEDED` and cannot execute any fund-related contract calls. [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- Operator has configured at least one `REJECT` filter (non-default but a documented, intended use case).
- Attacker can identify the filter criteria (e.g., a specific `to` address, `gas` value range, or `block` type) by probing the API or reading public configuration.

**Feasibility:** No authentication or special privilege is required. Any HTTP client can send `POST /api/v1/contracts/call` with arbitrary `gas` and `to` fields. The attack is trivially scriptable and repeatable. The rate-limit bucket (500 req/s) does not prevent the gas-bucket drain because the two buckets are independent and the math works out to exact exhaustion at the rate limit. [7](#0-6) 

### Recommendation

Restore gas tokens whenever `throttle()` throws due to a REJECT filter. The cleanest fix is to catch `ThrottleException` in `ContractController.call()` alongside `InvalidParametersException` and call `throttleManager.restore(request.getGas())` before re-throwing:

```java
} catch (ThrottleException e) {
    // REJECT filter fired after gas was consumed; restore tokens.
    throttleManager.restore(request.getGas());
    throw e;
}
```

Alternatively, move the restore logic into `ThrottleManagerImpl.throttle()` itself: record the consumed amount and restore it inside the `action()` method before throwing on `REJECT`. This keeps the invariant that gas tokens are only permanently consumed for requests that actually reach the EVM.

### Proof of Concept

**Setup:** Configure a REJECT filter matching requests to address `0xdeadbeef...` in `application.yml`:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    filters:
      - field: TO
        type: EQUALS
        expression: "0x00000000000000000000000000000000deadbeef"
```

**Attack script (pseudocode):**
```
for i in range(500):  # saturate rate limit
    POST /api/v1/contracts/call
    body: { "to": "0x00000000000000000000000000000000deadbeef",
            "gas": 15000000,   # max gas → 1500 scaled tokens consumed
            "data": "0x" }
    # Returns 429 "Invalid request" but 1500 tokens are gone
```

**Verification:**
1. Before attack: legitimate call to any address succeeds.
2. During attack (500 req/s to the REJECT-matched address): `gasLimitBucket` is drained to 0.
3. Immediately after: legitimate calls to *any* address receive `Gas per second rate limit exceeded` until the bucket refills (~1 second later).
4. Attacker repeats every second to maintain the DoS indefinitely.

            result = transactionExecutionService.execute(params, estimatedGas);
