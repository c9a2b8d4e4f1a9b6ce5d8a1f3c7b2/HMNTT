### Title
Global Single-Token Opcode Rate Limit Bucket Enables Trivial DoS by Any Unprivileged User

### Summary
The `opcodeRateLimitBucket` is a single application-wide `Bucket` with a default capacity of 1 token per second, shared across all users and requests. Any unauthenticated caller can consume this single token with one HTTP request, causing every other user to receive HTTP 429 for the remainder of that second. Because the attack requires only one request per second, it is trivially repeatable and effectively permanently denies the opcode tracing endpoint to all legitimate users.

### Finding Description
**Code path:**

- `ThrottleConfiguration.java` lines 47–55: `opcodeRateLimitBucket()` creates a singleton Spring `@Bean` `Bucket` with `capacity = throttleProperties.getOpcodeRequestsPerSecond()` (default `1`) and `refillGreedy(rateLimit, Duration.ofSeconds(1))`.
- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` on this shared global bucket; if it returns `false`, a `ThrottleException` is thrown (→ HTTP 429).
- `OpcodesController.java` line 61: every GET to `/{transactionIdOrHash}/opcodes` calls `throttleManager.throttleOpcodeRequest()` before any processing.

**Root cause:** The bucket is a single global instance with no per-IP, per-session, or per-user partitioning. The default capacity is `1`. The failed assumption is that a capacity-1 global bucket provides protection against expensive replays while still allowing legitimate concurrent access — in reality it serializes all users behind a single token.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/<any_valid_tx_hash>/opcodes` with header `Accept-Encoding: gzip` (the only other guard, trivially satisfied).
2. `tryConsume(1)` succeeds, consuming the sole token.
3. Every subsequent request within that second hits `tryConsume(1) == false` → HTTP 429 for all other users.
4. Attacker repeats once per second; the bucket refills greedily but the attacker always wins the race.

**Why existing checks are insufficient:**
- The `Accept-Encoding: gzip` header check (`OpcodesController.java` lines 75–85) is trivially bypassed by including the header.
- There is no authentication, IP-based limiting, or per-client bucket.
- The `rateLimitBucket` (500 req/s default) is separate and does not protect the opcode endpoint.

### Impact Explanation
Any anonymous internet user can permanently deny the opcode tracing endpoint (`/api/v1/contracts/results/{id}/opcodes`) to all other users by sustaining one request per second. This constitutes a complete availability loss for the opcode tracing feature. No funds are directly at risk, placing this in the medium severity band (availability impact on a specific API endpoint), consistent with the stated scope.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only one HTTP request per second. The endpoint is publicly reachable. The `Accept-Encoding: gzip` requirement is a one-line curl flag. Any script kiddie with knowledge of the API can execute this indefinitely. Likelihood is high.

### Recommendation
Replace the single global bucket with per-client (per-IP or per-authenticated-identity) rate limiting for the opcode endpoint. If a global cap is still desired, increase `opcodeRequestsPerSecond` to a value that accommodates expected concurrent legitimate users, and add a secondary per-IP bucket (e.g., via a servlet filter or Spring `HandlerInterceptor`) so one client cannot exhaust the global allowance. Example: use a `ConcurrentHashMap<String, Bucket>` keyed on client IP, each with its own 1-req/s limit, so the global system limit and per-user fairness are both enforced.

### Proof of Concept
```bash
# Terminal 1 – attacker: consume the token every second
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node-host>/api/v1/contracts/results/<valid_tx_hash>/opcodes"
  sleep 0.9
done

# Terminal 2 – victim: every request returns 429
curl -v -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/<valid_tx_hash>/opcodes"
# Expected: HTTP 429 Too Many Requests
# Body: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The attacker's first request each second succeeds (HTTP 200); all victim requests within that window receive HTTP 429, confirming full monopolization of the global opcode rate limit bucket. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
