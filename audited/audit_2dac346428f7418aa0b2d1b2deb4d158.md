### Title
Unauthenticated Mirror REST Response Allows Attacker-Controlled Domain Injection into Consensus Node Network Map

### Summary
The `buildNetworkFromMirrorNodes` / `fetchMirrorNodeNetwork` pipeline fetches the consensus-node list from a configurable mirror REST URL using a plain `http.Client` with no TLS certificate pinning, no response signature verification, and no allowlist validation of returned hostnames. Any attacker who can influence the mirror REST response — via MITM on an HTTP endpoint, DNS hijacking of the mirror domain, or mirror-node compromise — can inject `serviceEndpoint` entries with arbitrary `DomainName` values. Because `DomainName` is preferred over `IPAddressV4` in host selection, injecting attacker-controlled domains for ≥30% of nodes silently redirects those nodes' gRPC traffic to attacker-controlled infrastructure.

### Finding Description
**Exact code path:**

`pinger/sdk_client.go` `newClient()` → `buildNetworkFromMirrorNodes()` → `fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go`.

The HTTP client is constructed with no TLS customisation: [1](#0-0) 

The `serviceEndpoint` struct blindly deserialises whatever the mirror REST API returns: [2](#0-1) 

`DomainName` is a free-form string field with no format, ownership, or allowlist check. The question's referenced line `host = strings.TrimSpace(ep.DomainName)` (in the unrendered lines 73–131) uses this value directly as the gRPC dial target, meaning the SDK will resolve and connect to whatever hostname the mirror response supplies.

**Root cause:** The code implicitly trusts the mirror REST response as an authoritative, integrity-protected source of consensus-node addresses. No mechanism exists to verify that returned domain names actually belong to legitimate consensus nodes.

**Failed assumption:** The mirror REST endpoint is assumed to be tamper-proof. In practice it is fetched over a plain `http.Client` with no certificate pinning, making it vulnerable to MITM on HTTP deployments and to DNS-hijack + certificate-issuance attacks on HTTPS deployments.

### Impact Explanation
An attacker who injects malicious `DomainName` values for ≥30% of nodes causes the pinger's SDK client to dial attacker-controlled hosts for those node slots. Consequences include:

- **Traffic interception / credential exposure:** The operator key (`cfg.operatorKey`) is used to sign transactions sent to these "nodes"; an attacker-controlled endpoint receives signed transactions and can replay or analyse them.
- **Network disruption:** Transactions routed to fake nodes never reach consensus, effectively shutting down ≥30% of the pinger's network processing capacity — matching the stated severity scope.
- **Persistent redirection:** The network map is built once at startup; the attacker's domains remain active for the lifetime of the client.

### Likelihood Explanation
- **HTTP mirror endpoint (no TLS):** Any on-path network attacker can trivially MITM the `/api/v1/network/nodes` response. No credentials required.
- **HTTPS mirror endpoint:** DNS hijacking of the mirror domain combined with a DV certificate (obtainable for free via Let's Encrypt once DNS is controlled) suffices. This is a realistic, documented attack class.
- **Mirror node compromise:** A compromised or malicious mirror node operator can serve arbitrary node lists to all pingers pointed at it.
- The `http.Client` has no certificate pinning, no HSTS enforcement, and no response-body signature check, so none of these attack paths are blocked by the code. [1](#0-0) 

### Recommendation
1. **Pin the mirror REST TLS certificate** or enforce a CA allowlist in the `http.Client`'s `TLSClientConfig`.
2. **Validate returned domain names** against a configurable allowlist of known-good consensus-node FQDNs/IP ranges before inserting them into the network map.
3. **Prefer IP addresses over domain names** for consensus-node connections, or at minimum require that `DomainName` entries resolve to IPs within a known CIDR allowlist.
4. **Sign mirror REST responses** (e.g., with an operator-configured public key) and verify the signature before consuming the node list.

### Proof of Concept
1. Stand up a fake mirror REST server that returns a valid `nodesEnvelope` JSON where ≥30% of `nodeEntry` objects have `service_endpoints[].domain_name` set to `attacker.example.com` (attacker-controlled).
2. Point `cfg.mirrorRest` at this server (or MITM the real mirror REST URL via DNS spoofing / ARP poisoning on an HTTP deployment).
3. Start the pinger. `buildNetworkFromMirrorNodes` fetches the poisoned node list.
4. `fetchMirrorNodeNetwork` builds a `map[string]hiero.AccountID` where ≥30% of entries map to `attacker.example.com:<port>`.
5. `hiero.ClientForNetwork(netmap)` dials `attacker.example.com` for those node slots.
6. Observe on the attacker's server that signed gRPC transactions arrive from the pinger, and that the pinger reports failures/timeouts for the affected nodes — confirming traffic redirection and ≥30% network processing disruption. [3](#0-2) [4](#0-3)
