### Title
`eth_estimateGas` Binary Search Amplification Enables Thread Pool Exhaustion DoS

### Summary
Any unauthenticated caller can send a single `eth_estimateGas` request (`estimate=true`, `gas=15_000_000`) that causes `BinaryGasEstimator.search()` to invoke up to 20 additional full EVM executions beyond the initial probe — 21 total per HTTP request. The gas and RPS throttles are checked exactly once at request entry and are entirely blind to the 20 binary-search iterations, which run with `estimate=true` and bypass all throttle accounting. A small number of concurrent attackers can saturate the Tomcat thread pool and prevent the node from processing any further requests.

### Finding Description

**Entry point — throttle checked once:**

`ContractController.call()` calls `throttleManager.throttle(request)` once, deducting 1 RPS token and `scaleGas(request.getGas())` gas tokens from the respective buckets before any EVM work begins. [1](#0-0) 

**Initial probe — gas bucket partially restored:**

`ContractExecutionService.estimateGas()` first calls `callContract(params, context)`, which internally calls `doProcessCall(params, params.getGas(), false)` — `estimate=false`. The `finally` block in `doProcessCall` fires `restoreGasToBucket`, returning unused gas to the bucket. [2](#0-1) 

**Binary search — 20 iterations, zero throttle interaction:**

After the probe, `binaryGasEstimator.search()` is invoked. Its loop runs `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())` — up to 20 iterations — each calling `doProcessCall(params, gas, true)` with `estimate=true`. [3](#0-2) [4](#0-3) 

**Root cause — `estimate=true` skips all throttle accounting:**

Inside `doProcessCall`, the entire `finally` block — which contains both `restoreGasToBucket` and metric updates — is gated on `if (!estimate)`. When `estimate=true`, neither gas is debited from the bucket nor restored to it; the 20 binary-search EVM executions are completely invisible to the throttle. [5](#0-4) 

**Throttle calibration assumes 1 EVM execution per request:**

The default `gasPerSecond=7_500_000_000` and `requestsPerSecond=500` are sized for one EVM execution per request (7.5B / 15M = 500). An estimate request performs up to 21 executions, so the actual compute load is ~21× what the throttle assumes. [6](#0-5) 

**No authentication required:**

`ContractController` has no authentication guard; any external caller can reach `/api/v1/contracts/call`. [7](#0-6) 

### Impact Explanation

Each `eth_estimateGas` request with `gas=15_000_000` holds a Tomcat worker thread for the duration of 1 initial EVM execution plus up to 20 binary-search EVM executions. With a default Tomcat thread pool of 200 threads and a `requestTimeout` of 10 seconds, an attacker needs only enough concurrent requests to fill the thread pool. Because the RPS throttle (500/s) is global and not per-IP, and because each request occupies a thread for far longer than `1/500` seconds when executing a gas-heavy contract, the thread pool is exhausted well within the throttle's allowed request rate. Once all threads are occupied, the server queues and then rejects new connections — preventing legitimate transaction confirmations and eth_call queries from being processed. [8](#0-7) 

### Likelihood Explanation

No credentials, API keys, or on-chain assets are required. The endpoint is publicly reachable. The attacker needs only an HTTP client capable of sending concurrent POST requests to `/api/v1/contracts/call` with `{"estimate":true,"gas":15000000,"to":"<any deployed contract>","data":"<selector of a gas-heavy function>"}`. The attack is repeatable and stateless. A single machine with modest bandwidth can sustain the required concurrency.

### Recommendation

1. **Account for all EVM iterations in the gas throttle.** Before entering `binaryGasEstimator.search()`, pre-deduct `maxGasEstimateRetriesCount × gasUsedByInitialCall` (or a conservative multiple) from the gas bucket, and restore the unused portion afterward.
2. **Alternatively, count each binary-search iteration against the gas bucket** by removing the `if (!estimate)` guard on the gas-restore path, or by adding a separate debit inside `BinaryGasEstimator.search()` per iteration.
3. **Add a per-source-IP rate limit** so a single attacker cannot consume the global RPS budget.
4. **Cap the maximum number of binary-search iterations** more aggressively, or impose a wall-clock timeout on the entire estimate operation independent of `requestTimeout`.
5. **Consider making `maxGasEstimateRetriesCount` factor into the gas-throttle capacity calculation** so operators are aware of the true worst-case compute multiplier.

### Proof of Concept

```
# Deploy any gas-heavy contract (e.g., one that iterates storage writes up to 15M gas).
# Then flood the estimate endpoint concurrently:

for i in $(seq 1 300); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to":   "0x<deployed_contract_address>",
      "data": "0x<selector_of_gas_heavy_function>",
      "gas":  15000000,
      "estimate": true
    }' &
done
wait
```

**Expected result:** Each of the 300 concurrent requests triggers up to 21 full EVM executions (1 probe + 20 binary-search iterations). The Tomcat thread pool (default 200) is saturated within the first wave. Subsequent legitimate requests — including `eth_call` and transaction-confirmation queries — receive connection-refused or timeout errors, constituting a complete denial of service for the duration of the attack.
