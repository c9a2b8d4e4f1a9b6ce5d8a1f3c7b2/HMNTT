### Title
Unauthenticated DB Query Exhaustion via Non-Existent EVM Addresses in `getTokenRelationships()`

### Summary
Any unauthenticated user can send requests to `/api/v1/accounts/0x<random_40_hex_chars>/tokens` with syntactically valid but non-existent EVM addresses. Each request unconditionally triggers a database query in `EntityService.getEntityIdFromEvmAddress()` with no result caching for the miss case, allowing an attacker to saturate DB connection/query capacity through high-volume requests.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenController.js`, `getTokenRelationships()` calls `EntityService.getEncodedId()` unconditionally on the raw path parameter: [1](#0-0) 

Inside `getEncodedId()` in `rest/service/entityService.js`, when the input passes `EntityId.isValidEntityId()` (which accepts any `(0x)?[A-Fa-f0-9]{40}` string as shown in `evmAddressRegex`), and the parsed `EntityId` has a non-null `evmAddress` field (i.e., it is an opaque/non-parsable EVM address), it calls `getEntityIdFromEvmAddress()`: [2](#0-1) 

`getEntityIdFromEvmAddress()` issues a live DB query on every invocation: [3](#0-2)

**Why existing checks are insufficient:**

The `parseCached` LRU cache in `rest/entityId.js` only caches the *structural parse result* (the `EntityId` object), not the DB lookup result: [5](#0-4) 

So for the same EVM address sent repeatedly, the regex/parse work is cached, but `getEntityIdFromEvmAddress()` still fires a DB query on every request. For distinct random addresses (trivially generated), even the parse cache provides no protection.

There is no application-level rate limiting in the REST service middleware: [6](#0-5) 

The response cache (`responseCacheUpdateHandler`) only caches successful responses, not 404 errors, so it provides no protection for the miss path. [7](#0-6) 

### Impact Explanation
Each request with a non-existent EVM address causes one synchronous DB query against the `entity` table. An attacker generating distinct random 40-hex-char addresses at high rate can saturate the DB connection pool, increase query latency for all users, and degrade or deny service to legitimate API consumers. No authentication, tokens, or funds are required. The impact is service degradation (griefing) with no direct economic loss to network participants, consistent with the Medium severity classification.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is fully public. Generating syntactically valid random EVM addresses is trivial (any 40 random hex characters). The attacker needs only a script and network access. The attack is repeatable indefinitely and can be parallelized across many source IPs to bypass any infrastructure-level per-IP rate limits. The Rosetta service has Traefik rate limiting configured in its Helm chart, but no equivalent is present in the REST service application code. [8](#0-7) 

### Recommendation
1. **Cache negative (miss) results**: Extend the result cache in `getEntityIdFromEvmAddress()` or `getEncodedId()` to store `null` for addresses that were not found, with a short TTL (e.g., 30–60 seconds). This prevents repeated DB hits for the same non-existent address.
2. **Application-level rate limiting**: Add per-IP or global rate limiting middleware to the REST service (e.g., using `express-rate-limit`) mirroring the Traefik configuration already present for the Rosetta service.
3. **DB index verification**: Ensure `entity.evm_address` is indexed so that each miss query is a fast index scan rather than a table scan, limiting per-query cost.

### Proof of Concept

```bash
# Generate and send 10,000 requests with distinct random valid EVM addresses
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/accounts/0x${ADDR}/tokens" &
done
wait
```

Each request passes `isValidEntityId()` validation, reaches `getEntityIdFromEvmAddress()`, and issues a `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query that returns 0 rows. With 10,000 concurrent requests using distinct addresses, the DB receives 10,000 queries with no cache hits, exhausting connection pool capacity and degrading response times for legitimate users.
