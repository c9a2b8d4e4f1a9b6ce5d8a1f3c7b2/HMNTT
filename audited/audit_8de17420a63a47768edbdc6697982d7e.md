### Title
Unprivileged User Can Force Three-Way UNION ALL Query in `getQuery()` Causing 3x DB Query Amplification

### Summary
An unprivileged external user can craft a request with `spender.id=gte:0&spender.id=lte:9999999999&token.id=gte:0&token.id=lte:9999999999` against the `/accounts/:id/allowances/tokens` endpoint, causing `getQuery()` in `rest/service/tokenAllowanceService.js` to generate a three-branch `UNION ALL` query (lower + inner + upper subqueries) instead of a single query. Each subquery independently executes against the `token_allowance` table. The REST API has no rate-limiting middleware, allowing an attacker to repeat this indefinitely to amplify database load by a factor of 3.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenAllowanceController.js`, `extractTokenMultiUnionQuery()` parses the four filter parameters into `bounds.primary` (spender) and `bounds.secondary` (token_id): [1](#0-0) 

With the crafted input:
- `primary.lower = {operator: gte, value: 0}`
- `primary.upper = {operator: lte, value: 9999999999}`
- `secondary.lower = {operator: gte, value: 0}`
- `secondary.upper = {operator: lte, value: 9999999999}`

`getLowerFilters()` in `rest/controllers/baseController.js` fires the branch `primary.hasLower() && secondary.hasLower()`, producing `[spender=0, token_id>=0]`: [2](#0-1) 

`getInnerFilters()` fires because both bounds have lower and upper, producing `[spender>0, spender<9999999999]`: [3](#0-2) 

`getUpperFilters()` fires because both have upper bounds, producing `[spender=9999999999, token_id<=9999999999]`: [4](#0-3) 

All three filter arrays are non-empty, so `getQuery()` in `tokenAllowanceService.js` hits the multi-subquery branch: [5](#0-4) 

This generates:
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ORDER BY spender ASC, token_id ASC LIMIT $2)
ORDER BY spender ASC, token_id ASC LIMIT $2
```

**Root cause:** The multi-union decomposition is unconditionally triggered whenever both primary and secondary bounds have both a lower and upper bound. There is no check to collapse this into a single query when the ranges are trivially wide (e.g., 0 to max). Each subquery independently executes with its own `LIMIT $2` (default 25), meaning the database planner must independently plan and execute three separate index scans.

**Why existing checks fail:** The `LIMIT 25` per subquery does bound each individual scan, but it does not prevent three separate query executions. The `amount > 0` filter (`static amountCondition`) is not part of any composite index, so each subquery must filter post-index-scan for rows where `amount > 0`, potentially reading many index entries before finding 25 qualifying rows if many allowances have been revoked (amount=0). [6](#0-5) 

**No rate limiting on the REST API:** A grep across `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns only `integrationUtils.js` (test code). The `ThrottleConfiguration` and `ThrottleManagerImpl` exist only in the `web3` module, not the `rest` module. [7](#0-6) 

### Impact Explanation

Each crafted request causes 3 independent DB query executions instead of 1, a 3x amplification factor. The inner subquery (`spender>0 AND spender<9999999999`) covers virtually the entire spender space for the target account. With no rate limiting on the REST API, an attacker can sustain a high request rate, multiplying DB load by 3x compared to a normal request. In a deployment with many revoked allowances (amount=0), the `amount > 0` post-filter further amplifies per-query row reads. The impact is griefing-level: degraded DB performance and increased latency for all users, with no economic cost to the attacker.

### Likelihood Explanation

The attack requires zero privileges — only a valid account ID (publicly enumerable). The crafted query string is trivial to construct and repeat. Any HTTP client can automate it. The endpoint is publicly accessible at `/api/v1/accounts/{id}/allowances/tokens`. No authentication, no CAPTCHA, no rate limit stands in the way.

### Recommendation

1. **Collapse trivially wide ranges**: In `extractTokenMultiUnionQuery()`, detect when the combined lower+inner+upper decomposition would produce a range covering the full domain and fall back to a single query.
2. **Add rate limiting to the REST API**: Apply a per-IP or per-account rate limiter middleware (e.g., `express-rate-limit`) to all REST endpoints, mirroring the `ThrottleManagerImpl` pattern used in the `web3` module.
3. **Add a partial index on `amount`**: A partial index `WHERE amount > 0` on `token_allowance` would allow the DB to skip zero-amount rows at the index level, reducing per-subquery scan cost.
4. **Cap the number of union branches**: In `getQuery()`, assert that at most one non-empty filter array is present when the primary range is unbounded, preventing the three-branch case from being triggered by wide-open ranges.

### Proof of Concept

```
# No authentication required. Replace {accountId} with any valid account ID.

GET /api/v1/accounts/{accountId}/allowances/tokens?spender.id=gte:0&spender.id=lte:9999999999&token.id=gte:0&token.id=lte:9999999999

# Repeat in a loop:
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens?spender.id=gte:0&spender.id=lte:9999999999&token.id=gte:0&token.id=lte:9999999999" > /dev/null
done
```

Each iteration causes `getQuery()` to emit a 3-branch `UNION ALL` SQL query. Observe via DB query logs that three separate `SELECT * FROM token_allowance` statements execute per HTTP request instead of one.

  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }
