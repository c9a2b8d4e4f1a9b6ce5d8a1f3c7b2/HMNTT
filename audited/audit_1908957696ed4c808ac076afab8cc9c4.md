### Title
Missing Rate Limiting on REST-Java Topic Lookup Endpoint Enables Keep-Alive Connection Multiplexed Griefing

### Summary
The `TopicServiceImpl.findById()` method and its `TopicController` caller have no application-level rate limiting or connection-level throttling. Unlike the `web3` module which has a full `ThrottleManagerImpl`/`ThrottleConfiguration` stack, the `rest-java` module applies no request throttle to the `/api/v1/topics/{id}` endpoint. An unprivileged attacker can open many persistent HTTP/1.1 keep-alive connections and pipeline rapid topic lookup requests on each, multiplying DB load with no enforcement boundary in the application code.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` (lines 31–37) calls three service methods per request — `topicService.findById()`, `entityService.findById()`, and `customFeeService.findById()` — each issuing a separate DB query. [1](#0-0) 

`TopicServiceImpl.findById()` (lines 19–21) is a bare repository call with no guard: [2](#0-1) 

**Root cause — no throttle in rest-java:**

The only filters registered for the `rest-java` module are `LoggingFilter` (logging only) and `ShallowEtagHeaderFilter` (ETag caching). [3](#0-2)  No `bucket4j`, `RateLimiter`, or any throttle `Filter` bean exists anywhere under `rest-java/src/main/java/`. 

`RestJavaProperties` only manages response headers — no rate-limit configuration. [4](#0-3) 

**Contrast with web3:** The `web3` module has a full `ThrottleManagerImpl` enforcing `requestsPerSecond` (default 500) via bucket4j, applied at the controller level. [5](#0-4)  No equivalent exists for `rest-java`.

**Exploit flow:**
1. Attacker opens *N* TCP connections to the `rest-java` service (no `server.tomcat.max-connections` override found in any `rest-java` resource file).
2. Each connection uses HTTP/1.1 keep-alive (Spring Boot Tomcat default) to send a continuous stream of `GET /api/v1/topics/{id}` requests without reconnect overhead.
3. Each request triggers 3 synchronous DB queries (topic + entity + custom fee repositories).
4. Total DB query rate = N connections × M requests/s/connection × 3 queries — with no application-level ceiling.

### Impact Explanation
Each topic lookup issues three blocking DB queries against the same connection pool. A sustained flood from multiple keep-alive connections can exhaust the JDBC connection pool, causing latency spikes or timeouts for all other `rest-java` endpoints. This is a griefing/availability degradation scenario with no economic damage to network users, consistent with the stated Medium scope. [6](#0-5) 

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly accessible. The attack requires only a standard HTTP client capable of keep-alive connections (e.g., `curl`, `wrk`, `ab`). It is trivially repeatable and scriptable by any external actor. [7](#0-6) 

### Recommendation
1. Add a `bucket4j`-based rate-limiting filter to `rest-java` mirroring the pattern in `web3/ThrottleConfiguration` and `ThrottleManagerImpl`, applied globally or specifically to `/api/v1/topics/**`. [8](#0-7) 
2. Configure `server.tomcat.max-connections` and `server.tomcat.accept-count` in `rest-java` application properties to cap total concurrent connections.
3. Optionally add per-IP rate limiting at the servlet filter level or via an ingress/API-gateway layer.

### Proof of Concept
```bash
# Open 50 keep-alive connections, each sending 1000 rapid topic lookups
wrk -t 50 -c 50 -d 60s http://<rest-java-host>/api/v1/topics/0.0.1000

# Observe DB connection pool exhaustion in rest-java logs:
# "Unable to acquire JDBC Connection" or increasing response latency
# No HTTP 429 is ever returned — all requests are accepted and processed
``` [2](#0-1)

public class RestJavaProperties {

    @NotNull
    @Valid
    private ResponseConfig response = new ResponseConfig();

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }

    @Data
    @Validated
    public static class ResponseConfig {
        @NotNull
        @Valid
        private ResponseHeadersConfig headers = new ResponseHeadersConfig();
    }

    @Data
    @Validated
    public static class ResponseHeadersConfig {
        @NotNull
        private Map<String, String> defaults = new TreeMap<>(String.CASE_INSENSITIVE_ORDER);

        @NotNull
        private Map<String, Map<String, String>> path = new HashMap<>();

        public Map<String, String> getHeadersForPath(String apiPath) {
            return apiPath == null ? defaults : path.getOrDefault(apiPath, defaults);
        }
    }
