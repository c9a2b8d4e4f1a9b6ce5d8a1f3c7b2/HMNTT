### Title
Missing Circuit Breaker for `serverStatus()` Failures Enables Unbounded Error Log Flooding by Malicious Block Node

### Summary
`BlockNode.getBlockRange()` catches all gRPC exceptions from `serverStatus()` and returns `EMPTY_BLOCK_RANGE`, but unlike `streamBlocks()` failures, it never calls `onError()`. This means a malicious block node that always rejects `serverStatus()` calls is never marked inactive, is retried on every single block retrieval, and floods the importer's error logs indefinitely with no circuit-breaker or cooldown applied.

### Finding Description

**Exact code path:**

`BlockNodeSubscriber.getNode()` calls `hasBlock()` for every node on every block retrieval: [1](#0-0) 

`hasBlock()` calls `node.getBlockRange()` and returns `false` if the range is empty: [2](#0-1) 

`getBlockRange()` catches all exceptions, logs at ERROR level, and returns `EMPTY_BLOCK_RANGE` — but critically, does **not** call `onError()`: [3](#0-2) 

**Root cause — asymmetric error handling:** `streamBlocks()` failures call `onError()`, which increments an error counter and marks the node inactive after `maxSubscribeAttempts` (default: 3) failures, enforcing a `readmitDelay` (default: 1 minute) cooldown: [4](#0-3) 

`getBlockRange()` has no equivalent protection. A node that always fails `serverStatus()` is never marked inactive, never cooled down, and is re-queried on every call to `getNode()`.

**`EMPTY_BLOCK_RANGE`** is `Range.closedOpen(0L, 0L)`, which Guava's `Range.isEmpty()` returns `true` for, so `hasBlock()` correctly returns `false` — but the node is silently retried next time rather than being penalized: [5](#0-4) 

### Impact Explanation

Every block retrieval triggers a `serverStatus()` call to the malicious node, which times out after `responseTimeout` (default: 400 ms) and emits a `log.error(...)` entry. With continuous block ingestion, this produces a sustained stream of ERROR-level log entries per malicious node, with no bound or backoff. Log storage can be exhausted, alerting systems can be saturated, and legitimate error signals can be buried. The malicious node is never excluded from the node list, so the overhead persists for the lifetime of the process unless the node is manually removed from configuration or the registry.

### Likelihood Explanation

The precondition is controlling a registered block node — either a configured node (requires importer admin access) or an auto-discovered node registered via a Hedera governance transaction. A legitimate node operator who turns malicious, or a compromised node, satisfies this precondition. Once in place, the attack requires only that the node's gRPC server return any error (e.g., `UNAVAILABLE`, connection reset) on `serverStatus()` calls — trivially implemented and indefinitely repeatable with zero cost to the attacker.

### Recommendation

Apply the same circuit-breaker logic to `getBlockRange()` failures as is applied to `streamBlocks()` failures. Specifically, call `onError()` (or an equivalent counter increment) inside the `catch` block of `getBlockRange()` so that a node repeatedly failing `serverStatus()` is eventually marked inactive and subject to the `readmitDelay` cooldown. Additionally, consider downgrading the log level from `ERROR` to `WARN` after the first failure, or rate-limiting the log emission per node.

### Proof of Concept

1. Register or configure a block node endpoint pointing to a server that always returns gRPC status `UNAVAILABLE` on any RPC call.
2. Start the importer with at least one legitimate block node and the malicious node in the node list.
3. Observe that on every block retrieval cycle, `BlockNodeSubscriber.getNode()` calls `hasBlock()` → `getBlockRange()` → `serverStatus()` on the malicious node.
4. The malicious node's `serverStatus()` throws a gRPC exception; `getBlockRange()` logs `"Failed to get server status for BlockNode(<host>:<port>)"` at ERROR level and returns `EMPTY_BLOCK_RANGE`.
5. `hasBlock()` returns `false`; the node is skipped for this block but remains active.
6. On the next block retrieval, step 3–5 repeats with no cooldown, producing one ERROR log entry per block per malicious node, indefinitely.
7. Confirm via log output that the malicious node is never marked inactive (no `"Marking connection to ... as inactive"` warning appears for it).

    private static boolean hasBlock(final AtomicLong nextBlockNumber, final BlockNode node) {
        final var blockRange = node.getBlockRange();
        if (blockRange.isEmpty()) {
            return false;
        }

        if (nextBlockNumber.get() == EARLIEST_AVAILABLE_BLOCK_NUMBER) {
            nextBlockNumber.set(blockRange.lowerEndpoint());
            return true;
        } else {
            return blockRange.contains(nextBlockNumber.get());
        }
    }
