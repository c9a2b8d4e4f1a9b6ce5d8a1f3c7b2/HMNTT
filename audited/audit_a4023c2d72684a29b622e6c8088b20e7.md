### Title
Cancel-Before-Retry Race Condition in `RedisTopicListener.subscribe()` Enables Subscription Storm During Network Partition

### Summary
In `RedisTopicListener.subscribe()`, `doOnCancel(() -> unsubscribe(topic))` is placed **before** `retryWhen(...)` in the reactive operator chain. This means a cancel signal from `.share()` (triggered when all gRPC subscribers leave during a network partition) propagates through `retryWhen` and removes the topic from `topicMessages`, bypassing the retry logic entirely. Any new subscriber arriving before Redis recovers creates a fresh Flux that immediately fails, and the cycle repeats, producing a subscription storm with no rate-limiting on map entry creation.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–80:

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))   // line 74 — BEFORE retryWhen
    .doOnComplete(() -> unsubscribe(topic)) // line 75 — BEFORE retryWhen
    .doOnError(t -> log.error(...))
    .doOnSubscribe(...)
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L))) // line 78
    .share();                                                                                  // line 79
```

`getSharedListener()` at line 61 uses `computeIfAbsent` to cache the shared Flux:

```java
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
```

**Root cause — failed assumption:**

The developer assumed `retryWhen` would keep the Flux alive in `topicMessages` during Redis reconnection. However, because `doOnCancel` is placed *upstream* of `retryWhen`, a cancel signal from `.share()` (when subscriber count drops to zero) propagates *through* `retryWhen` and fires `unsubscribe(topic)`, which calls `topicMessages.remove(topic.getTopic())` at line 83. The retry backoff is irrelevant once the cancel signal arrives — it is not a terminal signal that `retryWhen` intercepts; it is a cancellation that flows upstream past it.

**Exploit flow:**

1. One or more gRPC clients subscribe to a topic. A shared Flux is created and stored in `topicMessages`.
2. A network partition disrupts the Redis connection. The `flatMapMany` upstream emits an error.
3. `retryWhen` catches the error and begins exponential backoff — the Flux is still in `topicMessages`, subscribers are waiting.
4. The gRPC transport layer detects the partition (connection timeout / keepalive failure) and cancels all active gRPC streams for that topic.
5. `.share()` observes zero remaining subscribers and cancels its upstream subscription.
6. The cancel signal propagates upstream through `retryWhen` → reaches `doOnCancel` → fires `unsubscribe(topic)` → `topicMessages.remove(topic.getTopic())`.
7. A gRPC client retries (standard gRPC retry behavior, e.g. `setMaxAttempts(Integer.MAX_VALUE)` as seen in `StartupProbe.java` line 116). `computeIfAbsent` finds no entry and calls `subscribe(topic)` again.
8. The new Flux immediately tries to connect to Redis (still partitioned), fails, and `retryWhen` starts another backoff.
9. If the gRPC layer cancels again before Redis recovers, step 6 repeats. Each client retry cycle creates a new Flux, a new Redis connection attempt, and a new backoff timer — with no deduplication or rate-limiting at the map level.

**Why existing checks are insufficient:**

- `retryWhen(Retry.backoff(Long.MAX_VALUE, interval))` only intercepts *error* signals. A *cancel* signal from `.share()` is not an error; it flows upstream unconditionally and is not absorbed by `retryWhen`.
- `ConcurrentHashMap` at line 43 ensures thread-safe map operations but provides no protection against the remove-then-reinsert cycle described above.
- There is no subscriber-count guard or cooldown on `computeIfAbsent` re-entry after `unsubscribe`.

### Impact Explanation

During a network partition, the server enters a tight loop: each client retry creates a new Redis subscription attempt, which fails, which (upon the next cancel) removes the map entry, which the next retry recreates. With `N` clients each using `setMaxAttempts(Integer.MAX_VALUE)`, this produces `N` concurrent Flux lifecycles per topic, each spawning a new `ReactiveRedisMessageListenerContainer.receive()` call. This exhausts Redis connection pool resources, thread pool capacity in `Schedulers.boundedElastic()`, and generates log spam that can mask the real partition event. After the partition heals, the storm continues until all in-flight retry cycles settle, delaying message delivery resumption. Severity: **High** — service degradation during the exact failure mode (network partition) where reliability is most critical.

### Likelihood Explanation

No attacker privileges are required beyond being a valid gRPC subscriber. The trigger (network partition) can be induced by any attacker with network-layer access (BGP manipulation, firewall rule injection, ARP poisoning on the Redis segment). Even without a malicious actor, this fires naturally during routine Redis maintenance, rolling restarts, or cloud provider network events. The gRPC SDK's built-in retry behavior (as confirmed in `StartupProbe.java` line 116: `setMaxAttempts(Integer.MAX_VALUE)`) guarantees clients will re-subscribe aggressively, making the storm self-sustaining. Repeatability: every partition event of sufficient duration triggers it.

### Recommendation

Move `doOnCancel` and `doOnComplete` to **after** `retryWhen`, so that the map entry is only removed when the retry sequence itself terminates (exhausted or explicitly cancelled after retries give up), not on every transient subscriber departure:

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnError(t -> log.error("Error listening for messages", t))
    .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .doOnCancel(() -> unsubscribe(topic))   // AFTER retryWhen
    .doOnComplete(() -> unsubscribe(topic)) // AFTER retryWhen
    .share();
```

Additionally, consider using `Flux.publish().refCount(1, Duration.ofSeconds(N))` instead of `.share()` to add a grace period before upstream cancellation, preventing transient zero-subscriber windows from tearing down the Redis subscription.

### Proof of Concept

**Preconditions:**
- Mirror node running with `listenerProperties.type = REDIS`
- Two or more gRPC clients subscribed to the same topic (e.g., `topic.1`)
- Network access to disrupt the Redis TCP connection

**Steps:**

1. Subscribe two gRPC clients to topic `0.0.1` using `ConsensusTopicQuery` with `setMaxAttempts(Integer.MAX_VALUE)`.
2. Confirm a single shared Flux is active in `topicMessages` (one log line: `"Creating shared subscription to topic.1"`).
3. Block TCP traffic between the gRPC service and Redis (e.g., `iptables -I OUTPUT -p tcp --dport 6379 -j DROP`).
4. Observe: Redis errors logged → `retryWhen` starts backoff → gRPC keepalive timeout fires → both clients cancel → `"Unsubscribing from topic.1"` logged → map entry removed.
5. Both clients retry (automatic) → `"Creating shared subscription to topic.1"` logged again → new Redis attempt fails → `retryWhen` starts backoff.
6. Repeat steps 4–5 while the partition is active. Each cycle produces a new `"Creating shared subscription"` log entry and a new failed Redis connection attempt.
7. Restore network (`iptables -D OUTPUT -p tcp --dport 6379 -j DROP`). Observe that multiple in-flight retry cycles compete to re-establish the subscription, and message delivery resumes only after all but one settle.

**Expected result without fix:** Log shows repeated `"Creating shared subscription"` / `"Unsubscribing"` pairs at the gRPC retry interval, with Redis connection pool exhaustion visible in metrics.

**Expected result with fix:** A single `"Creating shared subscription"` entry persists through the partition; `"Unsubscribing"` only appears when all clients permanently disconnect. [1](#0-0) [2](#0-1) [3](#0-2)
