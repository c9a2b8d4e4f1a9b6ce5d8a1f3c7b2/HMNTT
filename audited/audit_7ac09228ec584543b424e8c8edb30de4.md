### Title
Negative Cache Poisoning via `Optional.empty()` in `findByIndex` Allows Unprivileged Users to Hide Valid Blocks for 10 Minutes

### Summary
The `@Cacheable` annotation on `RecordFileRepository.findByIndex()` uses `unless = "#result == null"` as its cache-exclusion guard. In Spring's SpEL, `#result` is the actual return value — an `Optional.empty()` object — which is never `null`, so the guard evaluates to `false` and the empty `Optional` is stored in the `CACHE_MANAGER_RECORD_FILE_INDEX` cache. Any unprivileged caller who queries a block number before it is indexed causes that block to appear missing for the full 10-minute TTL, even after the block is written to the database.

### Finding Description
**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` (line 26) delegates to `RecordFileRepository.findByIndex(block.number())` for all numeric block types. [1](#0-0) 

`findByIndex` is annotated:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
Optional<RecordFile> findByIndex(long index);
``` [2](#0-1) 

**Root cause:** Spring's `@Cacheable` `unless` expression is evaluated against the method's return value. The method returns `Optional<RecordFile>`. When the block does not exist, the repository returns `Optional.empty()` — a non-null Java object. The guard `#result == null` is therefore `false`, and `Optional.empty()` is written into the Caffeine cache under the key equal to the queried block index.

**Cache TTL:** `expireAfterWrite(10, TimeUnit.MINUTES)` with `maximumSize(10000)`. [3](#0-2) 

**Why the `@CachePut` on `findByTimestamp` does not rescue this:** `findByTimestamp` writes to the same cache manager via `@CachePut`, but only when a timestamp-based lookup is performed. A direct `findByIndex` call after the negative entry is cached will always hit the stale `Optional.empty()` entry and never reach the database. [4](#0-3) 

### Impact Explanation
Any API surface that calls `findByBlockType` with a numeric block — including `eth_getBlockByNumber`, `eth_getTransactionByBlockNumberAndIndex`, and the `BLOCKHASH` EVM opcode handler — will return a "block not found" response for up to 10 minutes after the block is actually indexed. This breaks block-number-based queries for the poisoned index, causes `BLOCKHASH` to return `Hash.ZERO` for a real block, and can cause smart-contract calls that depend on recent block data to fail or behave incorrectly. The cache holds up to 10,000 entries, so an attacker can pre-poison a large range of future block numbers in a single burst.

### Likelihood Explanation
The web3 JSON-RPC endpoint is publicly accessible with no authentication. An attacker needs only to issue `eth_getBlockByNumber` (or any equivalent call) for a block number that is about to be, or has just been, indexed. Because Hedera produces blocks at a predictable rate, an attacker can trivially predict the next block number and fire the poisoning request before the mirror node finishes indexing it. The attack is repeatable every ~10 minutes (one TTL cycle) with no rate-limiting specific to this path.

### Recommendation
Change the `unless` condition on `findByIndex` to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
    unless = "#result == null || !#result.isPresent()")
Optional<RecordFile> findByIndex(long index);
```

This prevents `Optional.empty()` from ever being stored in the index cache, so a subsequent query after the block is indexed will always reach the database and return the correct result.

### Proof of Concept
1. Observe the current latest block number `N` via `eth_blockNumber`.
2. Send `eth_getBlockByNumber` with block number `N+1` (not yet indexed). The mirror node calls `findByIndex(N+1)`, gets `Optional.empty()`, and caches it.
3. Wait for the mirror node to index block `N+1` (typically a few seconds).
4. Immediately send `eth_getBlockByNumber` with block number `N+1` again. The cache returns the stale `Optional.empty()`; the response is `null` / "block not found" even though the block now exists in the database.
5. Repeat step 4 for the full 10-minute TTL window to confirm the block remains invisible.
6. After 10 minutes the cache entry expires; the next call correctly returns the block.
