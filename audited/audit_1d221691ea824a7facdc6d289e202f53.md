### Title
Unauthenticated Full Monthly Partition Scan via Timestamp Range in `getBalancesQuery()`

### Summary
An unprivileged user can supply `timestamp=gte:<start-of-previous-month>&timestamp=lte:<end-of-previous-month>` to force `getBalancesQuery()` to scan the entire monthly `account_balance` partition using `SELECT DISTINCT ON (account_id) ... ORDER BY account_id DESC, consensus_timestamp DESC`. The `LIMIT` clause provides no relief because PostgreSQL must sort and deduplicate all rows in the range before it can emit even the first result row, consuming significant DB memory and CPU and degrading performance for all concurrent users.

### Finding Description

**Exact code path:**

`getBalances()` → `getTsQuery()` → `getAccountBalanceTimestampRange()` → `getOptimizedTimestampRange()` → back to `getBalancesQuery()`.

**Step 1 — `getOptimizedTimestampRange()` (`rest/balances.js` lines 229–276):**

```
lowerBound = user gte  (e.g. 2024-03-01T00:00:00Z)
upperBound = user lte  (e.g. 2024-03-31T23:59:59Z)
effectiveUpperBound = min(upperBound, now)
                    = 2024-03-31  (past month, so now > upperBound)
optimalLowerBound   = getFirstDayOfMonth(2024-03-31, -1)
                    = 2024-02-01
lowerBound          = max(2024-03-01, 2024-02-01)
                    = 2024-03-01   ← user value wins; no clamping occurs
```

The optimization only clamps the lower bound to the first day of the month *before* the effective upper bound. When the user's `gte` value is already the first day of the target month, `max()` leaves it unchanged. The full previous-month range `[2024-03-01, 2024-03-31]` is passed through intact.

**Step 2 — `getAccountBalanceTimestampRange()` (lines 167–197):**

Queries for the latest treasury-account snapshot within `[2024-03-01, 2024-03-31]`. In a live network this returns the last snapshot of March (e.g. `2024-03-31T23:45:00Z`). It then returns:
```
{ lower: 2024-03-01T00:00:00Z,   ← getFirstDayOfMonth(upper)
  upper: 2024-03-31T23:45:00Z }  ← specific snapshot, not user's lte
```

**Step 3 — `getBalancesQuery()` (lines 199–218):**

```sql
SELECT DISTINCT ON (account_id)
  ab.account_id,
  ab.balance,
  $1::bigint AS consensus_timestamp,
  (<token_subquery>) AS token_balances
FROM account_balance ab
WHERE ab.consensus_timestamp >= $2          -- 2024-03-01 start
  AND ab.consensus_timestamp <= $3          -- 2024-03-31 last snapshot
ORDER BY ab.account_id DESC, ab.consensus_timestamp DESC
LIMIT $N
```

`$2` and `$3` span the entire March partition. PostgreSQL must:
1. Scan every row in the March partition matching the timestamp range (potentially hundreds of millions of rows on mainnet — ~2,880 snapshots × millions of accounts).
2. Sort the full result set by `(account_id DESC, consensus_timestamp DESC)` — an in-memory or disk-spill sort.
3. Deduplicate via `DISTINCT ON`.
4. Only then apply `LIMIT`.

**Root cause / failed assumption:** The design assumes `getOptimizedTimestampRange()` will always clamp the user's lower bound upward. This assumption fails when the user's `gte` value is already ≥ `optimalLowerBound` (i.e., the user explicitly targets the start of a recent month). The `LIMIT` clause is applied *after* the sort+dedup, so it cannot short-circuit the full-partition work.

**Existing check shown insufficient:**

`getOptimizedTimestampRange()` line 273:
```js
lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);
```
`optimalLowerBound` is the first day of the month *before* the effective upper bound. For a previous-month attack (`gte=March-1, lte=March-31`), `optimalLowerBound = Feb-1`, so `max(March-1, Feb-1) = March-1` — no clamping. The check is structurally insufficient for this input pattern.

No rate limiting, per-IP throttling, or DB `statement_timeout` visible in `rest/balances.js` prevents repeated issuance of this query.

### Impact Explanation

On a production Hedera mainnet mirror node, the `account_balance` table is partitioned monthly. Each monthly partition contains balance records for every account at every 15-minute snapshot interval (~2,880 snapshots/month). With millions of accounts, a single monthly partition can contain billions of rows. Forcing a full-partition sort via `DISTINCT ON … ORDER BY` consumes all available PostgreSQL `work_mem` (spilling to disk if exceeded), saturates CPU, and blocks or severely slows all concurrent queries. This constitutes a denial-of-service against the public REST API with no authentication required.

### Likelihood Explanation

The attack requires zero privileges — the `/api/v1/balances` endpoint is publicly accessible. The exploit is a single HTTP GET request with two timestamp parameters. It is trivially repeatable and scriptable. Any attacker aware of the monthly partition boundary (derivable from the API's own responses) can sustain the attack indefinitely. The previous month is always a valid, fully-populated target.

### Recommendation

1. **Pin the lower bound to the snapshot timestamp, not the start of the month.** In `getAccountBalanceTimestampRange()`, instead of returning `lower = getFirstDayOfMonth(upper)`, return `lower = upper` (i.e., query only the single snapshot timestamp). The `DISTINCT ON` is then unnecessary since all rows at a single snapshot timestamp are already unique per account.

2. **Alternatively, enforce a maximum timestamp range width** in `getOptimizedTimestampRange()` — e.g., reject or clamp any range wider than one snapshot interval (15 minutes) when the user explicitly supplies both bounds.

3. **Add a DB-level `statement_timeout`** on the connection used by the REST API to bound worst-case query duration.

4. **Add per-IP rate limiting** on the `/api/v1/balances` endpoint.

### Proof of Concept

```bash
# Compute start and end of the previous complete month (e.g. March 2024)
START="1709251200"   # 2024-03-01T00:00:00Z
END="1711929599"     # 2024-03-31T23:59:59Z

# Single unauthenticated request forces full March partition scan
curl "https://<mirror-node-host>/api/v1/balances?timestamp=gte:${START}&timestamp=lte:${END}"

# Repeat in a loop to sustain DB pressure
while true; do
  curl -s "https://<mirror-node-host>/api/v1/balances?timestamp=gte:${START}&timestamp=lte:${END}" &
done
```

Observe via `pg_stat_activity` that the backend is executing a sequential scan + sort on the monthly `account_balance` partition, with `work_mem` pressure and elevated CPU. Concurrent legitimate queries will show increased latency or timeouts.
