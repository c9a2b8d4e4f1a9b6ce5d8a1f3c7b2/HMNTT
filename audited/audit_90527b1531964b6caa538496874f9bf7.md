### Title
Unbounded Token Cache Serves Stale KYC Status via Nullish Coalescing Fallback in `getTokenAccounts()`

### Summary
The `tokenCache` in `rest/service/tokenService.js` is a `quick-lru` instance configured with only `maxSize` and no TTL or expiry. Once a token's KYC status is cached, it is never refreshed. When `token_account.kyc_status` is `null` for a given row, line 110 unconditionally substitutes the cached token-level `kycStatus` via the `??` operator, which may reflect an arbitrarily outdated on-chain state. Any unprivileged caller querying an account with a null per-account KYC status will silently receive the stale cached value.

### Finding Description

**Code path:**

`rest/service/tokenService.js`, `getTokenAccounts()`, lines 96–115:

```js
const cachedTokens = await this.getCachedTokens(tokenIds);   // line 104
return rows.map((row) => {
  const cachedToken = cachedTokens.get(row.token_id);
  if (cachedToken) {
    row.decimals      = cachedToken.decimals;
    row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
    row.kyc_status    = row.kyc_status    ?? cachedToken.kycStatus;  // line 110
  }
  return new TokenAccount(row);
});
```

**Root cause — no TTL on the cache:**

```js
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,   // lines 12-14 — maxSize only, no maxAge
});
```

`quick-lru` supports a `maxAge` option for per-entry TTL; it is not set here. The cache is purely LRU-eviction-based.

**Root cause — cache is write-once:**

```js
putTokenCache(token) {
  if (tokenCache.has(tokenId)) { return; }   // lines 123-124 — never updates
  tokenCache.set(tokenId, new CachedToken(token));
}
```

`getCachedTokens` also only writes to the cache for tokens not already present (lines 153–158). There is no invalidation path anywhere in the codebase.

**What the cache stores:**

`tokenCacheQuery` (lines 31–38) reads `decimals`, `freeze_status`, `kyc_status` from the **`token` table** — the token-level default KYC status — not from `token_account`. This is the value substituted at line 110.

**Trigger condition:**

`token_account.kyc_status = null` is a valid, observable database state (confirmed in test fixture `all-params.json`, token `0.0.45679`, which returns `"kyc_status": null` in the API response). When this column is `null`, the `??` operator on line 110 replaces it with whatever `cachedToken.kycStatus` holds, regardless of how old that value is.

**Exploit flow:**

1. Token T is first queried while `token.kyc_status = NOT_APPLICABLE (0)` (no KYC key). It is cached with `kycStatus = 0`.
2. The token owner later adds a KYC key; the importer updates `token.kyc_status = REVOKED (2)` in the database.
3. The in-process `tokenCache` is never invalidated — it still holds `kycStatus = 0`.
4. Account A has `token_account.kyc_status = null` (e.g., an older association record or one created before the KYC key was added).
5. Any unprivileged caller issues `GET /api/v1/accounts/{A}/tokens`.
6. `getTokenAccounts` fetches `row.kyc_status = null` from `token_account`, hits the cache for token T, and executes `null ?? 0` → `0`.
7. The API response returns `"kyc_status": "NOT_APPLICABLE"` when the actual on-chain state is `REVOKED`.

The reverse is equally possible: token cached as `REVOKED`, KYC key later removed, cache serves `REVOKED` indefinitely while the real state is `NOT_APPLICABLE`.

### Impact Explanation
The mirror node REST API is widely used as the authoritative read layer for Hedera state by dApps, wallets, and compliance systems. Returning a stale `kyc_status` — particularly `NOT_APPLICABLE` when the real value is `REVOKED` — can cause downstream systems to incorrectly conclude that KYC is not required for a token, bypassing compliance checks. Because the cache has no TTL and is never invalidated, the stale value persists until the process restarts or the entry is LRU-evicted (which, with `maxSize = 100,000`, may never happen for active tokens). The impact is silent and undetectable by the caller.

### Likelihood Explanation
The precondition — `token_account.kyc_status = null` — is a normal database state, not an edge case. Any token whose KYC configuration changes after its first API query will have a stale cache entry. The attacker requires no credentials, no special account, and no knowledge of internal state; a standard public API call to `/api/v1/accounts/{id}/tokens` is sufficient to observe and act on the incorrect value. The condition is repeatable and persistent across all callers for the lifetime of the process.

### Recommendation
1. **Add a TTL to the cache**: Pass `maxAge` (in milliseconds) to `quick-lru` at construction time (line 12–14), e.g., `maxAge: 60_000`. This bounds staleness to a configurable window.
2. **Expose `maxAge` as a config property** alongside `maxSize` (analogous to `hiero.mirror.rest.cache.entityId.maxAge` already present in `configuration.md`).
3. **Remove the write-once guard in `putTokenCache`** or add a separate `updateTokenCache` path so that token mutations observed by the importer can propagate to the cache.
4. **Treat `null` per-account KYC status as `null` in the response** rather than substituting a potentially stale token-level default, unless the token-level value is freshly read.

### Proof of Concept

**Preconditions:**
- Token `0.0.500` exists with no KYC key (`token.kyc_status = 0`, NOT_APPLICABLE).
- Account `0.0.1001` has an association with `token_account.kyc_status = null`.
- The mirror node process is running; token `0.0.500` has been cached (e.g., via a prior query).

**Steps:**
1. Confirm cache is populated: `GET /api/v1/accounts/0.0.1001/tokens` → response shows `"kyc_status": "NOT_APPLICABLE"` for token `0.0.500`. Cache now holds `kycStatus = 0`.
2. On-chain: token owner adds a KYC key to `0.0.500`. Importer updates `token.kyc_status = 2` (REVOKED) in the database. `token_account.kyc_status` for account `0.0.1001` remains `null` (no explicit grant/revoke transaction for this account yet).
3. Without restarting the mirror node: `GET /api/v1/accounts/0.0.1001/tokens`
4. **Observed:** `"kyc_status": "NOT_APPLICABLE"` — served from stale cache via `null ?? 0`.
5. **Expected:** `"kyc_status": "REVOKED"` — reflecting the current on-chain token default.
