### Title
Unauthenticated O(N) DB Query Amplification via `/block` Endpoint in `updateOperationAccountAlias()`

### Summary
The `/block` endpoint in `rosetta/app/services/block_service.go` is publicly accessible with no authentication or rate limiting. For each request, `Block()` calls `updateOperationAccountAlias()`, which issues one `GetAccountAlias()` database query per operation whose account ID is not in the LRU cache. A block containing many transactions with many unique account IDs across their operations causes O(N) sequential DB queries per request, and concurrent unauthenticated requests can sustain DB CPU well above the 30% threshold.

### Finding Description

**Code path:**

`rosetta/app/services/block_service.go`, `Block()` (lines 47–74) → `updateOperationAccountAlias()` (lines 104–130):

```
Block()
  └─ FindBetween()                          // fetches all txns in block (batched up to 2000)
  └─ cap to maxTransactionsInBlock          // limits txn count, NOT operation count
  └─ updateOperationAccountAlias(txns...)   // iterates every operation in every txn
       └─ for each operation:
            └─ entityCache.Get(accountId)   // LRU cache lookup
            └─ if miss: GetAccountAlias()   // one DB query per unique account ID
``` [1](#0-0) [2](#0-1) 

**Root cause:** `updateOperationAccountAlias()` iterates every operation across all capped transactions and fires a synchronous `GetAccountAlias()` DB query for each cache miss. The LRU cache is keyed by numeric entity ID; a block with many *distinct* account IDs across its operations will produce near-zero cache hit rate, yielding O(unique_accounts) DB queries per request.

**Failed assumption:** The `maxTransactionsInBlock` cap (line 61–66) limits the number of *transactions* in the response, not the number of *operations* across those transactions. Each transaction can carry many crypto-transfer operations (each with a distinct account ID), so the effective DB query count is `maxTransactionsInBlock × avg_operations_per_transaction × (1 − cache_hit_rate)`. [3](#0-2) 

**No rate limiting exists.** The middleware stack in `main.go` is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router` — no rate-limiting or concurrency-limiting layer is present. [4](#0-3) 

The `GetAccountAlias()` implementation issues a raw SQL query (`selectCryptoEntityWithAliasById`) against the database for every cache miss, with no batching. [5](#0-4) 

### Impact Explanation
An attacker can drive DB CPU above 30% of the 24-hour baseline without brute force. A single request to a block with `maxTransactionsInBlock` transactions, each with ~10 unique-account operations, produces hundreds of sequential DB queries. Sending tens of concurrent such requests (trivially achievable with `curl` or `ab`) multiplies this linearly. The DB connection pool (`Pool.MaxOpenConnections`) becomes the only backstop, and saturating it degrades all other mirror-node services sharing the same DB.

### Likelihood Explanation
The `/block` endpoint requires no credentials, API key, or session. Any external user can query it. Blocks with many unique account IDs are common on mainnet (e.g., blocks containing many fee-paying transactions). The attacker only needs to identify one such block index and loop requests against it. The attack is repeatable, low-cost, and requires no special knowledge beyond the block index.

### Recommendation
1. **Batch `GetAccountAlias` calls**: Replace the per-operation loop with a single bulk DB query that fetches aliases for all unique account IDs in the request at once, then populate the cache.
2. **Add rate limiting**: Introduce a per-IP or global rate-limiting middleware (e.g., `golang.org/x/time/rate`) before the router in `main.go`.
3. **Cap total operations, not just transactions**: Enforce a hard limit on the total number of operations processed by `updateOperationAccountAlias()` per request, independent of `maxTransactionsInBlock`.
4. **Increase LRU cache size** as a short-term mitigation to improve cache hit rate for repeated account IDs.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, publicly reachable. Identify a block index (e.g., from `/network/status`) that contains many transactions with diverse account IDs.

**Steps:**

```bash
# 1. Get a recent block index
BLOCK_INDEX=$(curl -s -X POST http://<rosetta-host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq '.current_block_identifier.index')

# 2. Send 50 concurrent /block requests targeting that block
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$BLOCK_INDEX}}" &
done
wait

# 3. Observe DB CPU via pg_stat_activity or monitoring dashboard
# Expected: DB CPU spikes significantly above 24h baseline due to
# O(N) GetAccountAlias queries fired per request with no rate limiting
```

**Result:** Each concurrent request triggers hundreds of `SELECT` queries against the `entity` table. With 50 concurrent requests and a block containing 250 transactions × 10 operations each = 125,000 potential DB queries in a short burst, driving DB CPU above the 30% threshold.

func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

func (ar *accountRepository) GetAccountAlias(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return accountId, nil
		}

		return zero, hErrors.ErrDatabaseError
	}

	if len(entity.Alias) == 0 {
		return accountId, nil
	}

	accountAlias, err := types.NewAccountIdFromEntity(entity)
	if err != nil {
		log.Warnf("Failed to create AccountId from alias '0x%s': %v", hex.EncodeToString(entity.Alias), err)
		return accountId, nil
	}

	return accountAlias, nil
}
