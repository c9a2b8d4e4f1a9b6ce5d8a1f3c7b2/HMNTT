### Title
Unbounded Promise Hang in `getAccountTokenAllowances` Due to Missing Client-Side Query Timeout During DB Network Partition

### Summary
`getAccountTokenAllowances` in `rest/controllers/tokenAllowanceController.js` unconditionally `await`s `EntityService.getEncodedId()`, which issues a live PostgreSQL query for any alias or EVM-address input. The node-postgres pool is configured with only a server-side `statement_timeout` and a connection-acquisition `connectionTimeoutMillis`; no client-side `query_timeout` is set. If the network partitions after the TCP connection is established but before the query response arrives, the awaited promise never settles, the request handler hangs indefinitely, and repeated exploitation exhausts the Node.js event loop.

### Finding Description

**Trigger path — controller to DB:**

`rest/controllers/tokenAllowanceController.js` line 69:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```
No timeout wrapper exists around this `await`.

`rest/service/entityService.js` lines 118–137 — `getEncodedId()` branches:
- If the input is a 40-hex-char EVM address → calls `getEntityIdFromEvmAddress()` → `this.getRows(EntityService.entityFromEvmAddressQuery, ...)` (line 91).
- If the input is a base32 alias → calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(EntityService.entityFromAliasQuery, ...)` (line 43).
- Only a plain numeric `shard.realm.num` skips the DB entirely.

`rest/service/baseService.js` line 55–57:
```js
async getRows(query, params) {
  return (await this.pool().queryQuietly(query, params)).rows;
}
```
No timeout here.

`rest/utils.js` line 1520:
```js
result = await this.query(query, params);
```
`queryQuietly` calls the raw `pg` `Pool.query()` with no `query_timeout` option.

**Why existing timeouts are insufficient:**

`rest/dbpool.js` lines 13–15:
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // only for acquiring a connection
statement_timeout: config.db.pool.statementTimeout,         // PostgreSQL server-side only
```
`connectionTimeoutMillis` fires only when waiting for a free slot in the pool — it does not bound an in-flight query. `statement_timeout` is a PostgreSQL session-level parameter sent at connection setup; during a network partition the server cannot transmit the timeout error back to the client. The node-postgres `query_timeout` option (a client-side deadline enforced in JavaScript) is absent from the pool config entirely.

`rest/server.js` has no `server.setTimeout()` call and no HTTP-level request-timeout middleware, so there is no outer deadline to cancel the hanging handler.

### Impact Explanation
An attacker sends a stream of GET requests to `/api/v1/accounts/<evm_address>/allowances/tokens` (or using an alias). Each request acquires a pool connection and issues a DB query. During any network partition event (even a brief one), every in-flight query promise hangs indefinitely. Because Node.js is single-threaded and each hanging `await` holds a microtask queue entry and a pool connection, a modest flood of such requests during a partition exhausts both the connection pool and the event loop's ability to process new work, resulting in a full denial of service for all API endpoints on the process.

### Likelihood Explanation
No authentication or privilege is required — the endpoint is public. The attacker only needs to supply a syntactically valid EVM address (e.g., `0x` + 40 hex chars) or a valid base32 alias string, both of which are trivially constructable. The network partition itself need not be attacker-controlled; the attacker simply needs to flood the endpoint during any transient DB connectivity degradation (cloud AZ hiccup, maintenance window, etc.) to lock up the process. The attack is repeatable and automatable with a simple HTTP client loop.

### Recommendation
1. **Add a client-side `query_timeout`** to the node-postgres pool config in `rest/dbpool.js`:
   ```js
   query_timeout: config.db.pool.statementTimeout, // or a dedicated config value
   ```
   This causes node-postgres to reject the promise after the deadline regardless of server reachability.
2. **Add an HTTP server-level timeout** in `rest/server.js`:
   ```js
   server.setTimeout(config.server.requestTimeout);
   ```
   This ensures the response socket is destroyed if a handler does not complete in time.
3. **Wrap the `getEncodedId` call** in `getAccountTokenAllowances` with `Promise.race` against a deadline as a defence-in-depth measure at the controller layer.

### Proof of Concept
```
# 1. Simulate a DB network partition (e.g., iptables DROP on the DB port, or pause the DB container).
# 2. Send requests with a valid EVM address path parameter:
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/accounts/0x71eaa748d5252be68c1185588beca495459fdba4/allowances/tokens" &
done
# 3. Observe: all 200 requests hang; no response is ever returned.
# 4. Restore DB connectivity.
# 5. Observe: requests still do not resolve (no client-side timeout to unblock them).
# 6. New requests to any endpoint also hang because the event loop and pool are saturated.
```
