### Title
Missing NodeMap Membership Validation Allows Unprivileged Node Pinning to Decommissioned Nodes

### Summary
The `getTransactionNodeAccountId()` function in `rosetta/app/services/construction_service.go` accepts any syntactically valid account ID string from user-supplied metadata without validating it against the server's configured `NodeMap` (i.e., `c.sdkClient.GetNetwork()`). An unprivileged external user can supply an arbitrary `node_account_id` in the `/construction/payloads` request metadata — including the account ID of a decommissioned node that remains in a stale `NodeMap` — causing the transaction to be frozen and submitted to that specific node, bypassing the server's controlled random node selection entirely.

### Finding Description

**Code path:**

`ConstructionMetadata` (line 166) uses `getRandomNodeAccountId()` which iterates only over `c.sdkClient.GetNetwork()` values: [1](#0-0) 

The server returns the selected `node_account_id` in metadata. The client is expected to echo it back in the `/construction/payloads` request. However, `getTransactionNodeAccountId()` (called at line 222) only validates format: [2](#0-1) 

The only check is `hiero.AccountIDFromString(str)` — a pure format parse. There is **no check** that the returned `nodeAccountId` is a member of `c.sdkClient.GetNetwork()`. The validated (but unverified) ID is then embedded directly into the frozen transaction: [3](#0-2) 

At submit time, `ConstructionSubmit` executes the transaction as-is: [4](#0-3) 

The SDK routes to whichever node account ID is embedded in the transaction. When `config.Rosetta.Nodes` (the `NodeMap`) is configured, the sdkClient is built exclusively from it: [5](#0-4) 

**Root cause:** `getTransactionNodeAccountId()` performs only syntactic validation. The server's own node selection (`getRandomNodeAccountId()`) is constrained to `c.sdkClient.GetNetwork()`, but the user-supplied value in payloads metadata bypasses this constraint entirely. The `NodeMap` type itself is a plain `map[string]hiero.AccountID` with no liveness or membership enforcement: [6](#0-5) 

**Exploit flow:**
1. Attacker calls `/construction/metadata` to observe the format of a valid `node_account_id` (e.g., `0.0.4`).
2. Attacker crafts a `/construction/payloads` request with `metadata["node_account_id"]` set to the account ID of a node known to be decommissioned but still present in the operator's stale `NodeMap` (e.g., `0.0.7` — historically decommissioned on mainnet).
3. `getTransactionNodeAccountId()` parses `"0.0.7"` successfully and returns it.
4. `transactionSetNodeAccountId` embeds `0.0.7` into the frozen transaction.
5. Attacker signs and submits via `/construction/submit`.
6. The SDK client looks up `0.0.7` in its network map (present, because the NodeMap is stale), finds the stale endpoint, and attempts to submit to the decommissioned node.

### Impact Explanation

An unprivileged user can pin any transaction to a specific node of their choosing, bypassing the server's random node selection. If the `NodeMap` contains stale entries for decommissioned nodes (a realistic operational condition when operators do not update configuration after network changes), the attacker can force all their transactions to be submitted to those nodes, causing guaranteed submission failures for those transactions. At scale, this constitutes a targeted denial-of-service against the Rosetta submission path. Additionally, node pinning undermines the load-balancing and fault-tolerance properties the random selection is designed to provide.

### Likelihood Explanation

No privileges are required. The Rosetta API is a public HTTP interface. The attacker needs only to know a valid account ID format (trivially observable from `/construction/metadata` responses or public Hedera documentation). Decommissioned node account IDs are publicly known from Hedera's network history. The attack is repeatable with every payloads request and requires no authentication or special access.

### Recommendation

In `getTransactionNodeAccountId()`, after parsing the account ID, validate it against the set of known node account IDs from `c.sdkClient.GetNetwork()`:

```go
func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
    emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
    // ... existing parsing ...
    nodeAccountId, err := hiero.AccountIDFromString(str)
    if err != nil {
        return emptyAccountId, errors.ErrInvalidAccount
    }

    // ADD: validate against known network nodes
    knownNodes := c.sdkClient.GetNetwork()
    valid := false
    for _, knownId := range knownNodes {
        if knownId == nodeAccountId {
            valid = true
            break
        }
    }
    if !valid {
        log.Errorf("Node account id %s not in known network", str)
        return emptyAccountId, errors.ErrInvalidAccount
    }

    return nodeAccountId, nilErr
}
```

Additionally, operators should ensure `NodeMap` entries are kept current and remove decommissioned nodes promptly.

### Proof of Concept

```

POST /construction/metadata
{"network_identifier": {...}, "options": {"operation_type": "CRYPTO_TRANSFER"}}
# Response includes: "node_account_id": "0.0.3"

# Step 2: Craft payloads request with decommissioned node account ID
POST /construction/payloads
{
  "network_identifier": {...},
  "operations": [...valid transfer ops...],
  "metadata": {
    "node_account_id": "0.0.7",   # <-- decommissioned node, still in stale NodeMap
    "valid_until_nanos": "<value from metadata response>"
  }
}
# Server accepts it: getTransactionNodeAccountId() only checks format, not NodeMap membership
# Response: unsigned_transaction with node 0.0.7 embedded

# Step 3: Sign and submit
POST /construction/combine  { ...sign with valid key... }
POST /construction/submit   { signed_transaction: "..." }
# SDK routes to decommissioned node 0.0.7 endpoint → connection failure → ErrTransactionSubmissionFailed
# Transaction is lost; attacker can repeat to exhaust submission attempts
```

func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
	emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
	value, ok := metadata[metadataKeyNodeAccountId]
	if !ok {
		return emptyAccountId, errors.ErrMissingNodeAccountIdMetadata
	}

	str, ok := value.(string)
	if !ok {
		return emptyAccountId, errors.ErrInvalidArgument
	}

	nodeAccountId, err := hiero.AccountIDFromString(str)
	if err != nil {
		log.Errorf("Invalid node account id provided in metadata: %s", str)
		return emptyAccountId, errors.ErrInvalidAccount
	}

	log.Infof("Use node account id %s from metadata", str)
	return nodeAccountId, nilErr
}

func (c *constructionAPIService) getRandomNodeAccountId() (hiero.AccountID, *rTypes.Error) {
	nodeAccountIds := make([]hiero.AccountID, 0)
	seen := map[hiero.AccountID]struct{}{}
	// network returned from sdkClient is a map[string]AccountID, the key is the address of a node and the value is
	// its node account id. Since a node can have multiple addresses, we need the seen map to get a unique node account
	// id array
	for _, nodeAccountId := range c.sdkClient.GetNetwork() {
		if _, ok := seen[nodeAccountId]; ok {
			continue
		}

		seen[nodeAccountId] = struct{}{}
		nodeAccountIds = append(nodeAccountIds, nodeAccountId)
	}
