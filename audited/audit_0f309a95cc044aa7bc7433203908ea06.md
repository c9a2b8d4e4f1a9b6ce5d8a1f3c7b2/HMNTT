### Title
Silent Long Overflow in `NumberRangeParameter.getInclusiveValue()` Causes `node.id=gt:Long.MAX_VALUE` to Return All Nodes Instead of Empty Set

### Summary
When an unprivileged user supplies `node.id=gt:9223372036854775807` (Long.MAX_VALUE) to the `/api/v1/network/nodes` endpoint, `NumberRangeParameter.getInclusiveValue()` computes `Long.MAX_VALUE + 1` in plain Java arithmetic, silently wrapping to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` then resolves to `0L`, and the subsequent `lowerBound > upperBound` guard (`0 > Long.MAX_VALUE`) is false, so no error is thrown. The query executes with `lowerBound=0`, returning all network nodes instead of an empty result set.

### Finding Description

**Code path:**

`NumberRangeParameter.getNumberValue()` accepts `Long.MAX_VALUE` because `Long.parseLong("9223372036854775807")` succeeds and the only guard is `value < 0`: [1](#0-0) 

`getInclusiveValue()` performs unchecked addition for the `GT` case: [2](#0-1) 

`Long.MAX_VALUE + 1` wraps to `Long.MIN_VALUE` in Java without throwing. Back in `getNetworkNodes()`: [3](#0-2) 

- Line 114: `Math.max(0L, Long.MIN_VALUE)` = `0L` — overflow is silently discarded.
- Line 120: `0L > Long.MAX_VALUE` = `false` — the range-inversion guard does not fire.
- Line 135–136: `findNetworkNodes(fileId, [], 0L, Long.MAX_VALUE, ...)` — full table scan, all nodes returned.

`NetworkNodeRequest.nodeIds` is typed as `List<NumberRangeParameter>`, confirming this is the exact parameter class in use: [4](#0-3) 

### Impact Explanation
The endpoint returns network topology data (node IDs, endpoints, public keys). While this data is not secret, the filter contract is broken: a request that semantically means "nodes with ID > Long.MAX_VALUE" (an empty set) instead returns every node in the address book. Any rate-limiting or pagination logic that relies on the filter being honored is bypassed. Severity is **Low** — no authentication bypass or sensitive data exposure, but the input validation guarantee is violated and the API contract is incorrect.

### Likelihood Explanation
Exploitation requires only an unauthenticated HTTP GET request with a crafted query parameter. No special knowledge beyond the public API schema is needed. The OpenAPI spec documents the `\d{1,19}` pattern for `node.id`, and `9223372036854775807` is exactly 19 digits, so it passes any regex-level validation as well: [5](#0-4) 

### Recommendation
In `NumberRangeParameter.getInclusiveValue()`, guard against overflow before performing the `+1`/`-1` adjustment:

```java
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        if (value == Long.MAX_VALUE) {
            throw new IllegalArgumentException("node.id value out of range for GT operator");
        }
        return value + 1;
    } else if (operator == RangeOperator.LT) {
        if (value == 0L) {
            throw new IllegalArgumentException("node.id value out of range for LT operator");
        }
        return value - 1;
    }
    return value;
}
```

The same fix should be applied to `EntityIdRangeParameter.getInclusiveValue()` and `Bound.getInclusiveRangeValue()` / `getAdjustedLowerRangeValue()` for consistency: [6](#0-5) [7](#0-6) 

### Proof of Concept

```
GET /api/v1/network/nodes?node.id=gt:9223372036854775807
```

**Expected:** HTTP 400 (invalid range) or HTTP 200 with empty `nodes` array.

**Actual:** HTTP 200 with all network nodes returned (up to `MAX_LIMIT=25`).

Step-by-step:
1. `NumberRangeParameter.valueOf("gt:9223372036854775807")` → `operator=GT, value=9223372036854775807L`
2. `getInclusiveValue()` → `9223372036854775807L + 1` = `-9223372036854775808L` (`Long.MIN_VALUE`)
3. `Math.max(0L, -9223372036854775808L)` = `0L` → `lowerBound = 0`
4. `0L > Long.MAX_VALUE` = `false` → no exception
5. `findNetworkNodes(fileId, [], 0, Long.MAX_VALUE, "ASC", 25)` → returns all nodes
