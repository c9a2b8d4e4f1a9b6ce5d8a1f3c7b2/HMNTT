### Title
Unauthenticated DoS via Unbounded Concurrent Requests to `/api/v1/network/supply` Exhausting DB Connection Pool

### Summary
The `GET /api/v1/network/supply` endpoint in the rest-java module has no rate limiting, no response caching, and directly triggers an expensive full-table-scan aggregation query (`getSupply()`) on every request. An unprivileged attacker can flood this endpoint with concurrent requests, holding HikariCP connections for up to 10 seconds each (the configured `statementTimeout`), exhausting the connection pool and saturating the shared PostgreSQL server, which degrades or halts the importer's ability to ingest new transactions.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) accepts unauthenticated GET requests and unconditionally calls `networkService.getSupply(bound)`. [1](#0-0) 

`NetworkServiceImpl.getSupply()` (lines 59–89), when no timestamp is provided (the common case), calls `entityRepository.getSupply(lowerBounds, upperBounds)` directly with no caching layer. [2](#0-1) 

`EntityRepository.getSupply()` (lines 19–29) executes a native SQL query that performs a full scan of the `entity` table joined with an `unnest()` of account-range arrays, computing `sum(e.balance)` and `max(e.balance_timestamp)` with no LIMIT clause. [3](#0-2) 

**Root cause — three missing controls:**

1. **No rate limiting in rest-java for this endpoint.** The `ThrottleManager`/`ThrottleConfiguration` with bucket4j exists exclusively in the `web3` module. There is no equivalent in `rest-java`. No Spring filter, interceptor, or annotation applies any per-IP or global RPS cap to `NetworkController`. [4](#0-3) 

2. **No response caching.** A search for `@Cacheable` or any cache annotation on `NetworkServiceImpl.getSupply()` or `EntityRepository.getSupply()` returns no results. Every HTTP request triggers a fresh DB query.

3. **10-second statement timeout.** The documented default for `hiero.mirror.restJava.db.statementTimeout` is 10,000 ms. [5](#0-4) 

Each concurrent request holds a HikariCP connection for up to 10 seconds while the aggregation runs. The Grafana alert `RestJavaHighDBConnections` fires when active connections exceed 75% of the pool maximum, confirming the pool is a known bottleneck. [6](#0-5) 

**Why existing checks fail:**

The only throttling infrastructure in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped to `org.hiero.mirror.web3` and operates on `ContractCallRequest` objects. It is never wired into the rest-java Spring context. The `NetworkController` has no `@RateLimiter`, no servlet filter, and no Spring Security rule protecting `/api/v1/network/supply`. [7](#0-6) 

### Impact Explanation

**Direct impact:** The rest-java HikariCP pool is exhausted. All other rest-java endpoints (exchange rate, fees, nodes, etc.) begin queuing or timing out on connection acquisition, making the mirror node's REST API unavailable.

**Indirect impact on the importer:** The importer uses a separate HikariCP pool with a different DB user (`mirror_importer`), but both connect to the same PostgreSQL server instance. PostgreSQL enforces a global `max_connections` limit. Concurrent expensive aggregation queries from rest-java consume PostgreSQL worker processes and I/O bandwidth. If the PostgreSQL server's `max_connections` is reached, the importer cannot open new connections. Even below that limit, CPU and I/O saturation from concurrent full-table scans degrades write throughput for the importer, slowing or stalling transaction ingestion. The severity is **High** (DoS of the mirror node REST API is certain; importer impact depends on PostgreSQL server sizing).

### Likelihood Explanation

**Preconditions:** None. The endpoint is publicly documented in the OpenAPI spec, requires no authentication, and accepts no parameters that would limit query cost. [8](#0-7) 

**Feasibility:** A single attacker with a modest botnet or even a single machine using async HTTP (e.g., `ab`, `wrk`, `k6`) can sustain hundreds of concurrent requests. The k6 load test script for this exact endpoint exists in the repository, confirming the team is aware of load concerns. [9](#0-8) 

**Repeatability:** Fully repeatable with no authentication, no CAPTCHA, and no IP-based blocking in the application layer.

### Recommendation

1. **Add response caching** on `NetworkServiceImpl.getSupply()` with a short TTL (e.g., 5–15 seconds via `@Cacheable` with Caffeine or Redis). The supply value changes at most once per consensus round (~3–5 seconds), so stale-by-seconds responses are acceptable.

2. **Add a global RPS rate limiter** to the rest-java module for the `/api/v1/network/supply` path, mirroring the bucket4j pattern already used in `web3/ThrottleConfiguration`. A limit of 10–50 RPS globally (or per-IP via a servlet filter) is sufficient.

3. **Reduce `statementTimeout`** for the supply query to 2–3 seconds to bound connection hold time under load.

4. **Add a PostgreSQL-level query timeout** via `statement_timeout` in the DB role used by rest-java, as a defense-in-depth measure independent of the application layer.

final class ThrottleManagerImpl implements ThrottleManager {

    static final String REQUEST_PER_SECOND_LIMIT_EXCEEDED = "Requests per second rate limit exceeded";
    static final String GAS_PER_SECOND_LIMIT_EXCEEDED = "Gas per second rate limit exceeded.";

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
