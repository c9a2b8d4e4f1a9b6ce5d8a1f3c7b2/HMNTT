### Title
Unbounded Per-Subscriber Database Polling DoS in `PollingTopicListener` (POLL Mode)

### Summary
When `hiero.mirror.grpc.listener.type=POLL` is configured, each gRPC subscription spawns an independent polling loop that issues a full `topic_message` table query (up to `maxPageSize=5000` rows) every 500ms with no global subscriber cap. An unauthenticated attacker opening many connections — each with up to 5 subscriptions using `startTime=0` and no `endTime` — can saturate the shared database connection pool, causing all subscribers' poll cycles to stall well beyond the 500ms interval.

### Finding Description

**Code path:**

`PollingTopicListener.listen()` (lines 34–49) creates a new `PollingContext` per subscriber and schedules `poll()` to fire every `interval` (default 500ms) with `RepeatSpec.times(Long.MAX_VALUE)`. [1](#0-0) 

`poll()` (lines 51–62) computes `startTime` as `filter.getStartTime()` when no message has been received yet (`last == null`). With `startTime=0` and no `endTime`, the rebuilt filter queries `WHERE topic_id=? AND consensus_timestamp >= 0 LIMIT 5000` — a scan from the genesis of the table. [2](#0-1) 

`TopicMessageRepositoryCustomImpl.findByFilter()` materializes the full result set via `getResultList()` (not a cursor), blocking a DB connection for the entire query duration. [3](#0-2) 

**Root cause — failed assumption:** The design assumes a small, bounded number of concurrent POLL subscribers. There is no global subscriber limit; `subscriberCount` in `TopicMessageServiceImpl` is a metrics gauge only, not an enforcer. [4](#0-3) 

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection=5` (default) limits calls *per TCP connection*, not globally. An attacker opens many connections. [5](#0-4) 

- `startTime=0` passes validation: `@Min(0)` and `isValidStartTime()` only require `startTime <= now()`, which 0 always satisfies. [6](#0-5) 

- `db.statementTimeout=10000ms` (10 s) prevents individual queries from hanging indefinitely, but the attacker sustains a continuous flood — each query completes within 10 s and immediately re-queues after 500 ms. [7](#0-6) 

### Impact Explanation

With N attacker connections × 5 subscriptions each = 5N concurrent DB queries firing every 500ms, the shared JDBC/HikariCP connection pool is exhausted. Legitimate subscribers' `poll()` calls queue behind attacker queries. If each attacker query takes ~T ms, the effective polling interval for legitimate subscribers becomes `T + 500ms` instead of 500ms. With a large enough N, T easily exceeds 2000ms, producing >500% degradation of the polling cadence. This delays message delivery to all live subscribers for the duration of the attack. The mirror node gRPC service becomes effectively unavailable for real-time topic subscriptions.

### Likelihood Explanation

**Precondition:** `listener.type` must be set to `POLL`. The default is `REDIS`, so this only affects deployments that explicitly configure POLL mode. However, POLL is a documented, supported mode. [8](#0-7) 

No authentication is required for gRPC subscriptions. The attack requires only a network path to port 5600 and the ability to open multiple TCP connections — trivially achievable from a single machine or botnet. The attack is repeatable and self-sustaining as long as connections remain open.

### Recommendation

1. **Enforce a global concurrent-subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()` — reject new subscriptions when the count exceeds a configurable threshold.
2. **Add per-IP connection/subscription rate limiting** at the Netty/gRPC interceptor layer.
3. **Prefer `SHARED_POLL` or `REDIS` over `POLL`** in production — `SharedPollingTopicListener` issues a single shared DB query regardless of subscriber count. [9](#0-8) 
4. **Enforce a minimum `startTime`** (e.g., reject values more than N days in the past) to prevent full-history scans.

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
