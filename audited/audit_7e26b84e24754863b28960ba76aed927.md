### Title
Unhandled `BufferUnderflowException` via bare `"0x"` EVM Address Input in `getByEvmAddressAndType`

### Summary
Supplying `"0x"` as an EVM address causes `decodeEvmAddress` to silently return an empty byte array, which then causes `ByteBuffer.wrap(new byte[0]).getInt()` to throw an unchecked `BufferUnderflowException`. The custom GraphQL exception resolver does not handle this exception type, so it propagates as an unmasked internal error to any unauthenticated caller.

### Finding Description
**Step 1 — `decodeEvmAddress("0x")` silently succeeds:**

In `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java` lines 87–98:
```java
evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);  // "0x" → ""
return Hex.decodeHex(evmAddress);                              // Hex.decodeHex("") → new byte[0], no DecoderException
```
`Hex.decodeHex("")` returns an empty byte array without throwing `DecoderException`, so no `IllegalArgumentException` is raised.

**Step 2 — `getByEvmAddressAndType` crashes on the empty array:**

In `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java` lines 34–41:
```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);   // new byte[0]
var buffer = ByteBuffer.wrap(evmAddressBytes);            // capacity = 0
if (buffer.getInt() == 0 ...                             // BufferUnderflowException thrown here
```
`ByteBuffer.wrap(new byte[0]).getInt()` requires 4 bytes but finds 0; it throws `java.nio.BufferUnderflowException` (an unchecked `RuntimeException`).

**Step 3 — Exception resolver does not catch it:**

In `graphql/src/main/java/org/hiero/mirror/graphql/config/CustomExceptionResolver.java` lines 16–28:
```java
if (ex instanceof IllegalStateException
        || ex instanceof IllegalArgumentException
        || ex instanceof MirrorNodeException) { ... }
else {
    return null;   // BufferUnderflowException falls here → unhandled
}
```
Returning `null` causes Spring for GraphQL to fall back to its default handler, which surfaces an `INTERNAL_ERROR` classification to the caller and logs a full stack trace server-side.

### Impact Explanation
Any unauthenticated caller can send a single GraphQL query with `evmAddress: "0x"` to trigger a server-side `BufferUnderflowException` on every invocation. The immediate effect is an unmasked `INTERNAL_ERROR` response leaking internal implementation details via stack traces in logs. Repeated calls constitute a trivially automatable denial-of-service against the GraphQL endpoint. The "Direct loss of funds" label in the scope tag is not accurate for this specific path — the real impact is availability degradation and internal error disclosure.

### Likelihood Explanation
No authentication or special privilege is required. The GraphQL endpoint is publicly reachable. The input `"0x"` is a syntactically plausible EVM address prefix that passes all upstream string checks. Any attacker or fuzzer that tries minimal/boundary EVM address values will hit this immediately. It is 100% reproducible and requires no state setup.

### Recommendation
1. **In `decodeEvmAddress`**: After stripping the `"0x"` prefix, validate that the remaining string is exactly 40 hex characters (20 bytes) before calling `Hex.decodeHex`. Throw `IllegalArgumentException` for any other length.
2. **In `getByEvmAddressAndType`**: Add a length guard before wrapping in `ByteBuffer`:
   ```java
   if (evmAddressBytes.length != 20) {
       throw new IllegalArgumentException("EVM address must be 20 bytes");
   }
   ```
3. **In `CustomExceptionResolver`**: Consider adding `BufferUnderflowException` (or its parent `RuntimeException` subclasses) to the handled set, or add a catch-all that converts unknown runtime exceptions to a `ValidationError` rather than leaking `INTERNAL_ERROR`.

### Proof of Concept
Send the following GraphQL request to the mirror node GraphQL endpoint (no authentication needed):

```graphql
{
  account(input: { evmAddress: "0x" }) {
    id
  }
}
```

**Expected (correct) behavior:** A `ValidationError` response indicating the address is invalid.

**Actual behavior:**
1. `decodeEvmAddress("0x")` returns `new byte[0]`.
2. `ByteBuffer.wrap(new byte[0]).getInt()` throws `BufferUnderflowException`.
3. `CustomExceptionResolver.resolveToSingleError` returns `null` (not handled).
4. Spring for GraphQL returns `{"errors":[{"message":"INTERNAL_ERROR","extensions":{"classification":"INTERNAL_ERROR"}}]}`.
5. Server logs emit a full stack trace including class/method names.

Repeating this request in a loop causes sustained `INTERNAL_ERROR` responses and continuous stack-trace logging, degrading service availability.
