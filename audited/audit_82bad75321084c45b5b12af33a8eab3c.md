### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables Full Table Scan DoS

### Summary
An unprivileged user can supply a `HookStorageRequest` with only a lower timestamp bound (e.g., `timestamp=gte:0`) and no upper bound. This causes `adjustUpperBound()` to return `Long.MAX_VALUE` and `getAdjustedLowerRangeValue()` to return `0`, making the native SQL query scan every `hook_storage_change` row for the given `owner_id`/`hook_id` across all time. The `DISTINCT ON (key)` clause forces PostgreSQL to process the entire matching set before the `LIMIT` from `Pageable` is applied, exhausting DB I/O under repeated requests.

### Finding Description

**Code path:**

`HookServiceImpl.getHookStorageChange()` at lines 97–114 reads the timestamp bounds directly from the `Bound` object with no range-width validation:

```java
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();  // line 98
final long timestampUpperBound = timestamp.adjustUpperBound();             // line 99
``` [1](#0-0) 

`Bound.getAdjustedLowerRangeValue()` returns `0` when `lower == null`: [2](#0-1) 

`Bound.adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`: [3](#0-2) 

**Triggering the historical path:** `HookStorageRequest.isHistorical()` returns `true` whenever `!timestamp.isEmpty()`. `Bound.isEmpty()` is `lower == null && upper == null`. Supplying only a lower bound (e.g., `timestamp=gte:0`) makes `isEmpty()` false, so the historical path is taken — but `upper` remains `null`, yielding `Long.MAX_VALUE`. [4](#0-3) [5](#0-4) 

**The resulting SQL** (both query variants) becomes:

```sql
and consensus_timestamp between 0 and 9223372036854775807
``` [6](#0-5) 

**Why `Pageable` does not protect:** The query uses `DISTINCT ON (key)` ordered by `key ASC, consensus_timestamp DESC`. PostgreSQL must evaluate the full WHERE-clause result set to resolve the latest `consensus_timestamp` per key before it can apply the `LIMIT`. The `Pageable` limit (default 25) is appended after this full-scan phase, providing no early-termination benefit. [7](#0-6) 

**Why existing checks fail:** The `Bound` constructor only validates `adjustedLower > adjustedUpper` when `primarySortField=true`. Timestamps are always constructed with `primarySortField=false` via `Bound.of()`, so no range-width or upper-bound-presence check is ever enforced. [8](#0-7) [9](#0-8) 

### Impact Explanation
Each request with an unbounded timestamp range forces a full sequential scan of `hook_storage_change` for the targeted `owner_id`/`hook_id` partition. A small number of concurrent requests (e.g., 10–20) from a single unauthenticated client can saturate PostgreSQL I/O, causing query latency to spike across all tenants and exceeding the 30% resource-consumption threshold described in the scope. The `HookStorageResult` is paginated to 25 rows, but the DB cost is proportional to the total number of historical change records, not the returned rows.

### Likelihood Explanation
No authentication or rate-limiting is visible in the service or DTO layer. The parameter `timestamp=gte:0` is a single, trivially crafted query string value. Any external user who can reach the REST endpoint can trigger this repeatedly. The attack is stateless and requires no prior knowledge beyond a valid `ownerId`/`hookId` pair (which may itself be enumerable from other public endpoints).

### Recommendation
1. **Require an upper timestamp bound** when the historical path is taken: reject requests where `timestamp.adjustUpperBound() == Long.MAX_VALUE` with HTTP 400.
2. **Enforce a maximum timestamp window**: cap `timestampUpperBound - timestampLowerBound` to a reasonable value (e.g., 7 days in nanoseconds).
3. **Add a query timeout** at the JDBC/datasource level so runaway scans are killed server-side.
4. Optionally add a **covering index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` to allow PostgreSQL to satisfy `DISTINCT ON` via an index scan rather than a sequential scan.

### Proof of Concept

```
GET /api/v1/hooks/{ownerId}/storage?hookId=1&timestamp=gte:0
```

1. Identify any valid `ownerId` and `hookId` from the public hooks listing endpoint.
2. Send the request above (no upper timestamp bound).
3. Observe: `isHistorical()` → `true`; `timestampLowerBound=0`; `timestampUpperBound=Long.MAX_VALUE`.
4. The DB executes `consensus_timestamp between 0 and 9223372036854775807`, scanning all historical records.
5. Repeat in a loop (e.g., 20 concurrent threads). Monitor PostgreSQL `pg_stat_activity` and I/O metrics — DB CPU/I/O rises proportionally to the number of stored change records, with no server-side throttle stopping the scan.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
