### Title
Unbounded `node.id` Array Parameter Enables DoS of `/api/v1/network/nodes` Endpoint

### Summary
The `NetworkNodeRequest.nodeIds` field accepts an unbounded `List<NumberRangeParameter>` with no `@Size` constraint, unlike analogous fields in the same codebase. An unauthenticated attacker can supply thousands of `node.id=` query parameters, causing the service layer to build a large `Long[]` array that is passed directly into a PostgreSQL `= any(:nodeIds)` clause, triggering query planning overhead and memory pressure that can make the node-discovery endpoint unavailable.

### Finding Description

**Exact code path:**

`NetworkNodeRequest.java` line 38 declares `nodeIds` with no size constraint:
```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();
``` [1](#0-0) 

Compare to `RegisteredNodesRequest.java` which correctly applies `@Size(max = 2)`:
```java
@Size(max = 2)
private List<NumberRangeParameter> registeredNodeIds = List.of();
``` [2](#0-1) 

And `HooksController.java` which applies `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)`: [3](#0-2) 

`Constants.java` defines `MAX_REPEATED_QUERY_PARAMETERS = 100` but it is never applied to `NetworkNodeRequest.nodeIds`: [4](#0-3) 

In `NetworkServiceImpl.getNetworkNodes()`, every EQ-operator entry is added to a `HashSet` with no size guard, then streamed into a `Long[]`: [5](#0-4) 

That array is passed directly to the native SQL `= any(:nodeIds)` clause with no cap: [6](#0-5) 

**Root cause:** The `nodeIds` field in `NetworkNodeRequest` is missing the `@Size` annotation that the rest of the codebase consistently uses to bound repeated query parameters. The `MAX_REPEATED_QUERY_PARAMETERS = 100` constant exists and is used elsewhere but is not wired to this field.

**Why existing checks fail:** The `getEffectiveLimit()` cap of 25 only limits *output rows*, not the size of the input filter array. The `HashSet` deduplication only collapses exact duplicates; an attacker using distinct values (e.g., `node.id=1&node.id=2&...&node.id=9000`) bypasses it entirely. Tomcat's default `maxParameterCount` is 10,000, so the platform itself does not block the request.

### Impact Explanation
The `/api/v1/network/nodes` endpoint is the mechanism by which clients discover valid consensus nodes to submit transactions to. Sending a single HTTP GET with thousands of distinct `node.id=` values causes: (1) unbounded Java heap allocation for the `List`, `HashSet`, and `Long[]`; (2) transmission of a large array to PostgreSQL; (3) PostgreSQL query planner overhead for evaluating `= any(large_array)` — all on every request. Repeated concurrent requests can exhaust heap or database connection pool resources, rendering the endpoint unavailable and preventing clients from discovering nodes, which directly impairs transaction submission to the network.

### Likelihood Explanation
No authentication is required. The attack requires only an HTTP client capable of sending a GET request with many repeated query parameters — trivially scriptable with `curl` or any HTTP library. It is repeatable at will and can be sustained from a single machine. The endpoint is public-facing by design.

### Recommendation
Add `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` to the `nodeIds` field in `NetworkNodeRequest`, matching the pattern already used in `RegisteredNodesRequest` and `HooksController`:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = MAX_REPEATED_QUERY_PARAMETERS)
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, add a controller-level test analogous to `invalidRegisteredNodeIdTooManyParameters` that verifies a 400 response when more than `MAX_REPEATED_QUERY_PARAMETERS` `node.id` values are supplied.

### Proof of Concept
```bash
# Generate a request with 1000 distinct node.id equality parameters
PARAMS=$(seq 1 1000 | sed 's/^/node.id=/' | tr '\n' '&')
curl -s "http://<host>/api/v1/network/nodes?${PARAMS}"
```

1. Start with a running mirror-node rest-java instance.
2. Send the above request. Each `node.id=N` value is distinct, so `HashSet` deduplication does not reduce the array.
3. The service builds a `Long[1000]` array and issues `WHERE abe.node_id = any(ARRAY[1,2,...,1000])` to PostgreSQL.
4. Repeat concurrently (e.g., 50 parallel requests) to exhaust connection pool or heap.
5. Observe that subsequent legitimate requests to `/api/v1/network/nodes` time out or return 500 errors.
