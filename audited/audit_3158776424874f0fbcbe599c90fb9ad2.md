### Title
Unbounded gRPC Subscriber Count Enables Non-Network DoS via Resource Exhaustion in `subscribeTopic()`

### Summary
`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` on every new subscription but never checks it against any cap before allowing the subscription to proceed. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — is a per-TCP-connection limit, not a global one. An unprivileged attacker can open an arbitrary number of TCP connections, each carrying up to 5 concurrent streaming RPCs, growing the total subscriber count without bound and exhausting JVM heap and thread-pool resources.

### Finding Description

**Code path and root cause:**

In `TopicMessageServiceImpl.java`, `subscribeTopic()` unconditionally increments `subscriberCount` on subscription and decrements it only on termination: [1](#0-0) 

The field `subscriberCount` is declared as a plain `AtomicLong` and is wired only to a Micrometer `Gauge` for observability: [2](#0-1) 

It is **never read back** to gate or reject new subscriptions. There is no `if (subscriberCount.get() >= MAX) return error` guard anywhere in the method.

**Why the existing check is insufficient:**

`GrpcConfiguration` applies `maxConcurrentCallsPerConnection` to the Netty server builder: [3](#0-2) 

The default value is 5: [4](#0-3) 

This is a **per-connection** limit enforced by Netty at the transport layer. It does not cap the total number of TCP connections, and no such global connection limit is configured anywhere in `GrpcConfiguration`. `GrpcProperties` contains no subscriber cap field either: [5](#0-4) 

**Exploit flow:**

Each call to `subscribeTopic()` allocates a `TopicContext` (holding two `AtomicLong`s, an `AtomicReference`, a `Stopwatch`, and a `Flux` pipeline), plus a `Schedulers.boundedElastic()` task for the safety-check branch: [6](#0-5) 

An attacker opens *N* TCP connections × 5 calls each = 5*N* live subscriptions, all holding heap and scheduler threads indefinitely (since no `endTime` is set, `isComplete()` returns `false` and the stream never self-terminates): [7](#0-6) 

### Impact Explanation
With no global subscriber cap and no per-IP connection limit, an attacker can drive `subscriberCount` to tens of thousands. Each subscription holds live Reactor `Flux` pipelines, `TopicContext` objects, and bounded-elastic scheduler threads. At sufficient scale this exhausts JVM heap (triggering `OutOfMemoryError`) or saturates the `boundedElastic` thread pool, making the mirror node gRPC service completely unresponsive to all clients — a non-network DoS. The memory-per-subscription cost is small individually but unbounded in aggregate.

### Likelihood Explanation
The gRPC port (default 5600) is publicly exposed with no authentication required for `subscribeTopic`. No special privileges, tokens, or accounts are needed. A single attacker machine can open thousands of TCP connections using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl` in a loop). The attack is trivially repeatable and scriptable, and the server provides no backpressure or rejection signal until resources are fully exhausted.

### Recommendation
1. **Add a global subscriber cap**: Read `subscriberCount` before incrementing and return a `RESOURCE_EXHAUSTED` gRPC status if the cap is exceeded. Add a configurable `maxSubscribers` field to `GrpcProperties`.
2. **Add a per-IP or per-connection subscription limit** at the interceptor layer to prevent a single client from monopolizing capacity.
3. **Limit total inbound connections** via Netty's `maxConnectionAge` / connection-count settings in `GrpcConfiguration` to bound the multiplier effect of `maxConcurrentCallsPerConnection`.
4. **Set a maximum stream duration** so open-ended subscriptions (no `endTime`) are forcibly terminated after a configurable idle/max-age timeout.

### Proof of Concept
```bash
# Open 200 connections × 5 streams each = 1000 concurrent subscriptions
# Requires grpcurl and a valid topicId (e.g., 0.0.1234)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID":{"topicNum":1234}}' \
      mirror-node-host:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Observe hiero_mirror_grpc_subscribers metric climbing to 1000
# Repeat with more connections until OOM or thread pool saturation
```
Each background process holds a live streaming RPC with no end time, keeping its `TopicContext` and Reactor pipeline allocated indefinitely. Scaling `i` upward exhausts JVM heap or the `boundedElastic` scheduler.
