### Title
Aborted HTTP Connections Permanently Inflate `api_all_request_in_processing_total` In-Flight Counter

### Summary
In `metricsMiddleware()`, `inFlightCounter.add(1)` is called synchronously on every incoming request, but the corresponding `inFlightCounter.add(-1)` is placed exclusively inside a `res.on('finish')` handler. In Node.js, when a client aborts a TCP connection before the server sends a response, the underlying socket is destroyed, which destroys the `ServerResponse` stream and causes it to emit `'close'` — not `'finish'`. Because no `'close'` handler exists, the decrement never executes, permanently inflating the counter by +1 per aborted request. Any unprivileged external user can repeat this trivially to drive the gauge to an arbitrary value.

**Root cause — failed assumption:**
The code assumes `'finish'` is the terminal event for every HTTP response. This is false. Node.js `http.ServerResponse` extends `stream.Writable`. Per the Node.js stream contract:
- `'finish'` fires only after `stream.end()` is called and all data is flushed to the OS buffer.
- `'close'` fires when the underlying socket is destroyed (including client-initiated aborts).

When a client sends a TCP RST or closes the socket before the server calls `res.end()`, the socket is destroyed, the `ServerResponse` is destroyed, and `'close'` is emitted — `'finish'` is **never** emitted. There is no `res.on('close')` handler anywhere in the middleware to compensate.

**Why existing code does not help:**
The `res.end` override at lines 180–183 only accumulates `responseSize`; it does not call `inFlightCounter.add(-1)` and is itself never invoked on an aborted connection. [4](#0-3) 

### Impact Explanation
Every aborted request leaves `api_all_request_in_processing_total` permanently incremented by +1 for the lifetime of the process. The Grafana dashboard panel "Requests Processing" queries this metric directly (`sum(api_all_request_in_processing_total{...})`) and is configured to turn red at a threshold of 50. [5](#0-4)  An attacker sending 50 aborted requests silently trips the alert threshold, masking the true concurrency level of all Hashgraph history queries and potentially causing operators to ignore genuine overload conditions or to take incorrect remediation actions (e.g., scaling down). The counter only resets on process restart.

### Likelihood Explanation
No authentication, rate limiting, or IP restriction is applied to any non-metrics endpoint before `metricsMiddleware` runs. [6](#0-5)  Any unprivileged user with network access to the REST API can exploit this. The technique (send request, immediately close socket) is trivially scriptable with `curl --max-time 0`, `ab`, or a raw TCP client. It is repeatable indefinitely and leaves no error in server logs because the server never errors — it simply never finishes the response.

### Recommendation
Add a `res.on('close')` handler guarded by a `finished` flag to ensure the decrement fires exactly once regardless of whether the connection was completed or aborted:

```javascript
let finished = false;

res.on('finish', () => {
  finished = true;
  inFlightCounter.add(-1);
  // ... existing metrics recording ...
});

res.on('close', () => {
  if (!finished) {
    inFlightCounter.add(-1);  // client aborted before finish
  }
});
```

This pattern is the standard Node.js idiom for handling both normal and aborted HTTP responses in instrumentation middleware.

### Proof of Concept

**Preconditions:** Mirror Node REST API is reachable; `config.metrics.enabled = true`.

**Steps:**

1. Observe baseline counter value:
   ```
   curl http://<host>/api/v1/swagger/metrics/ | grep api_all_request_in_processing_total
   # e.g., returns 0
   ```

2. Send 100 requests to any Hashgraph history endpoint and abort each immediately:
   ```bash
   for i in $(seq 1 100); do
     curl -s --max-time 0.001 http://<host>/api/v1/transactions &
   done
   wait
   ```
   (The `--max-time 0.001` causes curl to close the TCP connection before the server responds.)

3. Re-query the metrics endpoint:
   ```
   curl http://<host>/api/v1/swagger/metrics/ | grep api_all_request_in_processing_total
   # returns a value significantly above 0, e.g., 87
   ```

4. Confirm the value does not decrease over time (no legitimate requests are in flight), proving the counter is permanently inflated.

5. Repeat step 2 until the value exceeds 50 to trigger the Grafana alert threshold, demonstrating the monitoring integrity impact.

        res.send(serializer.serialize(resourceMetrics));
      });

    const startTime = Date.now();
    inFlightCounter.add(1);

    // Intercept response writes to measure actual body size
    let responseSize = 0;
    const originalWrite = res.write.bind(res);
    const originalEnd = res.end.bind(res);

    res.write = function (chunk, ...args) {
      responseSize += getChunkSize(chunk);
      return originalWrite(chunk, ...args);
    };
    res.end = function (chunk, ...args) {
      responseSize += getChunkSize(chunk);
      return originalEnd(chunk, ...args);
    };

      inFlightCounter.add(-1);

        allErrorCounter.add(1);
