### Title
Unbounded Timestamp Range Enables Full Historical Table Scan on Hook Storage Change Endpoint

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts a `timestamp` range parameter with no width validation in the Java REST path. An unauthenticated attacker can supply a maximum-width timestamp range (e.g., `timestamp=gte:0.000000001&timestamp=lte:9223372036.854775807`) to force the database to perform a full scan of the `hook_storage_change` table for the targeted hook/owner across all recorded history. Unlike the Node.js REST path, the Java path has no `maxTimestampRange` guard, making this a repeatable denial-of-service vector.

### Finding Description

**Exact code path:**

In `hookStorageChangeRequest()`, the raw `TimestampParameter[]` array is passed directly to `Bound.of()` with no range-width check: [1](#0-0) 

`Bound.of()` performs only operator normalization (EQ → LTE) and constructs the `Bound` object. The `Bound` constructor calls `getAdjustedLowerRangeValue()` and `adjustUpperBound()` but only validates `lower > upper` when `primarySortField=true`; here it is `false`: [2](#0-1) 

`adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is set, and returns the raw value otherwise — no range-width check: [3](#0-2) 

In `HookServiceImpl.getHookStorageChange()`, these bounds are passed directly to the repository: [4](#0-3) 

The repository executes a native query with `consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound` — no additional guard: [5](#0-4) 

**Root cause:** No timestamp range width validation exists anywhere in the Java REST path for this endpoint. The Node.js REST path has `maxTimestampRange` / `maxTimestampRangeNs` enforcement in `parseTimestampFilters()`, but this Java endpoint has no equivalent.

**Correction to the question's specific example:** `timestamp=lte:99999999999.999999999` is rejected with HTTP 400 because `DomainUtils.convertToNanos(99999999999, 999999999)` overflows `Long` and `TimestampParameter.parseTimestamp()` catches the `ArithmeticException` and throws `IllegalArgumentException("Invalid timestamp parameter")`. [6](#0-5) 

The correct maximum-width upper bound is `timestamp=lte:9223372036.854775807`, which produces exactly `Long.MAX_VALUE` without overflow.

### Impact Explanation

1. **Unbounded DB scan (DoS):** The SQL `DISTINCT ON (key) … WHERE consensus_timestamp BETWEEN 1 AND 9223372036854775807` forces PostgreSQL to scan every row in `hook_storage_change` for the targeted `(owner_id, hook_id)` pair across all recorded history. With a large number of historical records, this is a full sequential scan. Repeated requests from an unauthenticated attacker can saturate DB I/O and degrade service for all users.

2. **Full historical state enumeration:** The `DISTINCT ON (key)` returns the most recent `value_written` per storage key within the range — effectively the complete current state of all storage slots for the hook. Combined with pagination via the `next` link, an attacker can enumerate all storage keys (100 per page) without any time-window restriction.

The response is scoped to a specific `(ownerId, hookId)` pair, so cross-tenant data leakage is not possible. However, the attacker only needs to know a valid `ownerId` and `hookId`, both of which are enumerable via the `GET /api/v1/accounts/{ownerId}/hooks` endpoint.

### Likelihood Explanation

- **No authentication required.** The endpoint has no `@PreAuthorize` or any security annotation.
- **Trivially reproducible.** A single HTTP GET with two query parameters is sufficient.
- **Repeatable.** The attacker can loop requests to sustain DB pressure.
- **Precondition:** Knowledge of a valid `ownerId` and `hookId`, obtainable from the public hooks listing endpoint.

### Recommendation

1. **Add timestamp range width validation** in `hookStorageChangeRequest()` or in `Bound.of()` for the timestamp case, mirroring the Node.js `maxTimestampRangeNs` check. Reject requests where `adjustUpperBound() - getAdjustedLowerRangeValue()` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Require a timestamp filter** for the historical path (i.e., when `isHistorical()` is true), or enforce a maximum range width as a precondition before executing the query.
3. **Add rate limiting** on this endpoint at the API gateway or application layer.

### Proof of Concept

```
# Step 1: Enumerate a valid hookId for a known ownerId
GET /api/v1/accounts/1001/hooks

# Step 2: Issue a max-width timestamp range query
GET /api/v1/accounts/1001/hooks/2001/storage?timestamp=gte:0.000000001&timestamp=lte:9223372036.854775807&limit=100&order=asc

# Result: HTTP 200, returns up to 100 storage keys with their most recent values,
# backed by a full scan of hook_storage_change for owner_id=1001, hook_id=2001.

# Step 3: Paginate using the `next` link to enumerate all storage keys.
# Step 4: Repeat in a loop to sustain DB load (DoS).
```

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public static Bound of(TimestampParameter[] timestamp, String parameterName, Field<Long> field) {
        if (timestamp == null || timestamp.length == 0) {
            return Bound.EMPTY;
        }

        for (int i = 0; i < timestamp.length; ++i) {
            final var param = timestamp[i];
            if (param.operator() == RangeOperator.EQ) {
                timestamp[i] = new TimestampParameter(RangeOperator.LTE, param.value());
            }
        }

        return new Bound(timestamp, false, parameterName, field);
    }
