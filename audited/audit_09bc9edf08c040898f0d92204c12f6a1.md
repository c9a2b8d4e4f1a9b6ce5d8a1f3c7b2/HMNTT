### Title
Global Rate Limit Bucket Exhaustion via Single Unprivileged Client — No Per-IP or Per-Connection Partitioning

### Summary
The `rateLimitBucket` in `ThrottleConfiguration` is a single JVM-wide singleton `Bucket` with no per-IP, per-session, or per-connection partitioning. Any unprivileged external client can open multiple persistent HTTP/1.1 keep-alive connections and pipeline requests at the full global rate (default 500 req/s), exhausting all available tokens and denying service to every other client. The server incurs per-request overhead (filter chain, logging, metrics) for every request including those that receive 429 responses, and persistent connections consume Tomcat threads and file descriptors with no per-client connection cap enforced anywhere in the codebase.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) creates one Spring singleton `Bucket` with capacity equal to `requestsPerSecond` (default 500):

```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond();
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.throttle()` (lines 37–42) consumes from this single shared bucket with no client identity check:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

No filter, interceptor, or configuration in the web3 module (`LoggingFilter`, `MetricsFilter`, `MetricsConfiguration`, `Web3Properties`) introduces any per-IP or per-connection limit. The `ThrottleProperties` exposes only a global `requestsPerSecond` knob with no per-client dimension.

**Root cause:** The design assumes the global bucket is shared fairly across many independent clients. This assumption fails entirely when a single client can open many persistent connections and pipeline requests at the full bucket refill rate.

**Exploit flow:**
1. Attacker opens N HTTP/1.1 persistent connections to `/api/v1/contracts/call`.
2. Each connection pipelines requests at maximum throughput.
3. The attacker's aggregate request rate reaches or exceeds 500 req/s, draining the global bucket continuously.
4. All other clients receive HTTP 429 responses immediately.
5. Even for 429 responses, `LoggingFilter.doFilterInternal()` wraps every request in a `ContentCachingRequestWrapper` and logs it; `MetricsFilter` records metrics — these per-request costs accumulate across all attacker connections.
6. Tomcat's default `maxConnections` (10,000) and `maxThreads` (200) are not overridden anywhere in the web3 configuration, so the attacker's persistent connections hold threads and file descriptors for the duration of the attack.

**Why existing checks fail:**
- The only check is `rateLimitBucket.tryConsume(1)` — a global counter with no client identity.
- `ThrottleProperties.request[]` filters operate on request content fields (`DATA`, `FROM`, `TO`, etc.), not on source IP or connection identity, and are not configured by default (`request = List.of()`).
- No Spring Security, servlet filter, or Tomcat connector configuration enforces a per-IP connection or request rate limit.

### Impact Explanation

A single unprivileged attacker can cause a sustained denial of service for all legitimate clients of the web3 API. The global `rateLimitBucket` is fully consumed by one source, so the 500 req/s capacity is unavailable to any other client for as long as the attack continues. Secondary resource consumption comes from: Tomcat thread allocation per persistent connection, `ContentCachingRequestWrapper` allocation and logging in `LoggingFilter` for every request (including 429s), and Micrometer metric recording in `MetricsFilter`. With Tomcat's default 200 worker threads and no per-IP connection cap, the attacker can hold a significant fraction of the thread pool with idle keep-alive connections while still saturating the rate bucket. This constitutes both a complete availability denial for other clients and measurable server-side resource consumption increase above baseline.

### Likelihood Explanation

No authentication, API key, or network-level credential is required. Any client with network access to the web3 endpoint can execute this attack using standard HTTP tooling (e.g., `wrk`, `hey`, `ab`, or a simple script using HTTP/1.1 keep-alive). The attack is trivially repeatable, requires no special knowledge of the application internals, and is sustainable indefinitely since the bucket refills every second and the attacker simply needs to match the refill rate. The default `requestsPerSecond = 500` is easily achievable from a single modern machine over a LAN or low-latency network path.

### Recommendation

1. **Per-IP rate limiting**: Replace or augment the global `Bucket` with a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from `HttpServletRequest.getRemoteAddr()` or a trusted `X-Forwarded-For` header), so each IP is limited independently.
2. **Per-IP connection limit**: Configure Tomcat's `maxConnections` per remote address via a `RemoteAddrValve` or an equivalent servlet filter that tracks and caps concurrent connections per source IP.
3. **Global + per-IP dual throttle**: Keep the global bucket as a ceiling, and add a per-IP bucket (e.g., `requestsPerSecond / expectedConcurrentClients`) so no single client can monopolize the global capacity.
4. **Consider bucket4j's distributed or keyed API**: Use `Bucket4j`'s `BucketConfiguration` with a key-based approach (e.g., backed by a `CaffeineProxyManager`) to manage per-IP buckets efficiently without unbounded memory growth.

### Proof of Concept

```bash
# Step 1: Establish baseline — observe normal 200 responses
curl -s -X POST http://<web3-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'

# Step 2: From attacker machine, saturate the global rate bucket
# Using 'wrk' with 50 persistent connections, 4 threads, for 30 seconds
wrk -t4 -c50 -d30s -s post.lua http://<web3-host>/api/v1/contracts/call
# post.lua sets method=POST, body=minimal valid ContractCallRequest, Content-Type header

# Step 3: Simultaneously from a second machine (victim), observe all requests return 429
curl -v -X POST http://<web3-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'
# Expected: HTTP/1.1 429 Too Many Requests
# Body: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}

# Step 4: Monitor server metrics during attack
# Tomcat busy threads, file descriptors, and CPU will show measurable increase
# compared to idle baseline, confirming >30% resource consumption delta
curl http://<web3-host>/actuator/prometheus | grep -E 'tomcat_threads|process_files'
```
