### Title
Unbounded Repeated DB Batch Queries via Non-Existent Slot Cache Miss in `findStorage()`

### Summary
In `ContractStateServiceImpl.findStorage()`, when a requested storage slot does not exist in the database, no negative result is ever stored in `contractStateCache`. Because the cache-hit guard (`cachedValue != null && cachedValue != EMPTY_VALUE`) treats a `null` lookup as a miss, every subsequent request for the same non-existent slot unconditionally re-invokes `findStorageBatch()`, which issues a full batch DB query. An unprivileged attacker can exploit this by repeatedly requesting non-existent slots, causing continuous and growing database load with no rate limiting from the cache layer.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorage()`, lines 63–69:
```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}

return findStorageBatch(contractId, key);
```

`findStorageBatch()`, lines 103–113:
```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
// ...
for (final var contractSlotValue : contractSlotValues) {
    final byte[] slotKey = contractSlotValue.getSlot();
    final byte[] slotValue = contractSlotValue.getValue();
    contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
    // ...
}
```

**Root cause:** `contractStateCache` is only populated for slots that are **returned by the DB** (i.e., slots that exist). For non-existent slots, `findStorageBatch()` receives an empty result set and writes nothing to `contractStateCache`. The next call to `findStorage()` for the same non-existent slot finds `null` in the cache, fails the guard, and calls `findStorageBatch()` again — indefinitely.

**Failed assumption:** The design assumes that `findStorageBatch()` will eventually populate `contractStateCache` for any requested key, breaking the miss loop. This assumption is false for slots absent from the DB.

**Amplification via `contractSlotsCache`:** Each unique non-existent slot key is added to the per-contract `contractSlotsCache` (line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)`). Because `findStorageBatch()` queries ALL keys accumulated in `contractSlotsCache` (line 91–101), each new unique non-existent slot increases the size of every subsequent batch query. After N unique non-existent slots are requested, every new request for any non-existent slot triggers a batch query over all N previously seen keys.

**Note on `EMPTY_VALUE`:** The `cachedValue != EMPTY_VALUE` guard is a red herring for this exploit path — `EMPTY_VALUE` is only ever written to `contractSlotsCache`, never to `contractStateCache`. The exploitable condition is purely the `null` (absent) case.

### Impact Explanation
An attacker can force the mirror node's web3 service to issue repeated, growing batch SQL queries against the `contract_state` table with zero DB-side benefit (all queries return empty). With N unique non-existent slot keys accumulated, each new request triggers a query scanning N slots. This can increase DB CPU, I/O, and connection pool pressure well beyond 30% of baseline, satisfying the stated scope threshold. The impact scales super-linearly: the attacker accumulates slot keys over time, making each subsequent query more expensive than the last.

### Likelihood Explanation
The exploit requires no authentication, no privileged access, and no special knowledge beyond a valid contract address (which is publicly visible on-chain). The attacker only needs to call `eth_getStorageAt` (or any EVM call that reads storage) with arbitrary 32-byte slot keys. This is a standard JSON-RPC call available to any user of the node. The attack is trivially scriptable, repeatable, and requires no on-chain transactions or gas expenditure.

### Recommendation
Store a negative sentinel in `contractStateCache` for slots confirmed absent from the DB. After `findStorageBatch()` completes, for any requested key that was not returned by the DB, explicitly write a sentinel (e.g., `EMPTY_VALUE`) into `contractStateCache`:

```java
// After the DB result loop in findStorageBatch():
if (cachedValue == null && !isKeyEvictedFromCache) {
    // Slot confirmed absent — cache the sentinel to prevent future misses
    contractStateCache.put(generateCacheKey(contractId, key), EMPTY_VALUE);
}
```

Then update `findStorage()` to treat `EMPTY_VALUE` as a definitive "not found" cache hit rather than a miss:

```java
if (cachedValue != null) {
    return cachedValue == EMPTY_VALUE ? Optional.empty() : Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
```

This ensures non-existent slots are cached after the first miss and never re-trigger a DB query.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service is running with `enableBatchContractSlotCaching=true`
- Any deployed contract address is known (e.g., `0xABCD...`)

**Steps:**
1. Generate a list of random 32-byte slot keys that are guaranteed not to exist in the contract's storage (e.g., `keccak256(i)` for large `i`).
2. In a loop, send `eth_getStorageAt(contractAddress, randomSlot, "latest")` for each unique random slot key via JSON-RPC.
3. Observe: each call returns `0x0` (slot not found), but each call also triggers a `findStorageBatch()` DB query. After K unique slots, each new call issues a batch query over K slots.
4. After sending ~1000 unique non-existent slot requests, each subsequent request triggers a batch query over 1000 keys — with no caching benefit and no way for the server to short-circuit.

**Expected result:** DB query rate and query size grow linearly with the number of unique non-existent slots requested, with no cache-based protection, causing sustained elevated DB load from a single unprivileged client.
