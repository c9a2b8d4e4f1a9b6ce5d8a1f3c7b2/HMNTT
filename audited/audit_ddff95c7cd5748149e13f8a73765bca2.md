### Title
False DOWN Health Status via Targeted Disruption of the `/network/stake` Endpoint

### Summary
The `health()` method in `SubscriberHealthIndicator` uses a hard short-circuit: if `restNetworkStakeHealth()` returns any non-UP status, the result is returned immediately and `publishing()`/`subscribing()` are never evaluated. An unprivileged external attacker who can force the mirror node's `/api/v1/network/stake` endpoint to respond with a 5xx code (or cause a connection failure/timeout) will cause the monitor to emit `DOWN` — regardless of whether actual transaction publishing and subscribing are healthy.

### Finding Description
**Exact code location:** `monitor/src/main/java/org/hiero/mirror/monitor/health/SubscriberHealthIndicator.java`, `health()`, lines 59–64.

```java
return restNetworkStakeHealth()
        .flatMap(health ->
                health.getStatus() == Status.UP
                        ? publishing().switchIfEmpty(subscribing())
                        : Mono.just(health))          // ← publishing/subscribing never reached
        .doOnNext(this::recordHealthMetric);
```

`restNetworkStakeHealth()` (lines 93–122) issues a live HTTP GET to `/network/stake` via `webClientRestJava` (line 68 of `RestApiClient.java`). Its decision tree:

| Response | Emitted status |
|---|---|
| 2xx | UP |
| 5xx | **DOWN** |
| 4xx / other | UNKNOWN |
| `ConnectException` / `TimeoutException` | **DOWN** |
| Any other error | UNKNOWN |

When the emitted status is not UP, `Mono.just(health)` is returned at line 62 — the `flatMap` lambda never enters the `publishing().switchIfEmpty(subscribing())` branch. The actual transaction-publishing rate and subscription rate are completely ignored.

**Root cause / failed assumption:** The design assumes the `/network/stake` endpoint is an authoritative, tamper-resistant proxy for overall cluster health. It is not: it is a single, publicly reachable HTTP endpoint whose failure unconditionally overrides all other health signals.

**Exploit flow:**
1. Attacker identifies the mirror node REST-Java base URL (configured in `MonitorProperties`, typically public).
2. Attacker floods `GET /api/v1/network/stake` with enough traffic to exhaust server threads/connections, causing 5xx responses or connection timeouts.
3. The monitor's 5-second timeout (line 107) fires, `onErrorResume` classifies `TimeoutException` as `DOWN` (lines 113–115), and `Mono.just(health)` is returned.
4. `recordHealthMetric` sets `CLUSTER_UP = 0` (line 68), publishing the false DOWN to Micrometer.
5. Publishing and subscribing scenarios — which may be perfectly healthy — are never consulted.

**Why existing checks are insufficient:**
- The 5-second timeout (line 107) is a mitigation against slow responses, but it actually *accelerates* the DOWN classification when the endpoint is under load.
- The `onErrorResume` handler (lines 108–121) correctly distinguishes connection vs. generic errors, but both paths still bypass publishing/subscribing evaluation.
- There is no circuit-breaker, retry, or quorum logic that would require the endpoint to fail repeatedly before affecting health status.

### Impact Explanation
The `CLUSTER_UP` gauge (registered in `registerGauge()`, lines 44–48) is set to `0` on every poll cycle during the attack. Any alerting or auto-remediation system consuming this metric (PagerDuty, Kubernetes liveness probes, auto-scaling policies) will treat a healthy cluster as failed. This can trigger unnecessary failovers, operator intervention, or service restarts — all while actual transaction publishing and subscribing continue unaffected. The monitor's health endpoint is also consumed by Spring Boot Actuator's `/actuator/health`, which external orchestration layers may poll.

### Likelihood Explanation
The `/network/stake` endpoint is a standard, unauthenticated REST endpoint on the mirror node's public API surface. No credentials are required to reach it. A volumetric HTTP flood targeting only this path is a low-skill, low-cost attack achievable with commodity tooling (e.g., `wrk`, `ab`, `hey`). The attacker does not need to compromise any account, key, or privileged channel. The attack is repeatable on every monitor health-check cycle (default Spring Boot actuator polling interval), making sustained false-DOWN trivial to maintain.

### Recommendation
1. **Decouple the gate:** Do not make publishing/subscribing evaluation contingent on `/network/stake` UP. Run all three checks independently and aggregate results (e.g., worst-of or weighted).
2. **Require repeated failures:** Introduce a consecutive-failure counter or a sliding-window circuit breaker before `restNetworkStakeHealth()` can emit DOWN; a single failed poll should not be decisive.
3. **Add retry with backoff:** Wrap `getNetworkStakeStatusCode()` with `.retry(2)` before the `.timeout()` so transient spikes do not immediately propagate.
4. **Rate-limit the `/network/stake` endpoint** at the infrastructure layer (API gateway / WAF) to prevent volumetric abuse.

### Proof of Concept
```
# Precondition: mirror node REST-Java API is publicly reachable at $BASE_URL
# Step 1 – flood /network/stake to induce 5xx / connection exhaustion
wrk -t8 -c200 -d60s $BASE_URL/api/v1/network/stake

# Step 2 – while flood is running, poll the monitor's health endpoint
curl -s http://<monitor-host>:8080/actuator/health | jq '.components.subscriber'

# Expected (false) result while publishing/subscribing are healthy:
# { "status": "DOWN", "details": { "reason": "Network stake status is DOWN ..." } }

# Step 3 – stop flood; health returns UP within one poll cycle
# Confirms the DOWN was caused solely by /network/stake disruption,

```java
    private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })
                .timeout(Duration.ofSeconds(5))
                .onErrorResume(e -> {
                    var status = Status.UNKNOWN;
                    // Connection issue can be caused by database being down, since the rest API service will become
                    // unavailable eventually
                    var rootCause = ExceptionUtils.getRootCause(e);
                    if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
                        status = Status.DOWN;
                    }

                    var statusMessage =
                            String.format("Network stake status is %s with error: %s", status, e.getMessage());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                });
    }
```
