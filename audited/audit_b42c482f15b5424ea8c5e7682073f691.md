### Title
Contract Existence Enumeration via Asymmetric Error Response in `getContractResultsById`

### Summary
When `GET /api/v1/contracts/:contractId/results` is called with a valid EVM address that does not exist in the database, `computeContractIdFromString()` performs a DB lookup and throws a `NotFoundError`, resulting in an HTTP 404 with a specific error message. When the same endpoint is called with an EVM address that *does* exist but has no results, it returns HTTP 200 with an empty results array. This asymmetry allows any unauthenticated user to enumerate which EVM addresses correspond to registered contracts.

### Finding Description
**Exact code path:**

In `rest/controllers/contractController.js`, `getContractResultsById` (line 856) calls `ContractService.computeContractIdFromString(contractIdParam)` at line 862 before any response is set. [1](#0-0) 

`computeContractIdFromString` in `rest/service/contractService.js` (line 468) calls `EntityId.computeContractIdPartsFromContractIdValue`. When the input is a 40-hex-char EVM address (with or without `0x` prefix, or in `shard.realm.evmAddress` form), `computeContractIdPartsFromContractIdValue` sets `create2_evm_address` on the result object. [2](#0-1) 

This causes `computeContractIdFromString` to branch into `getContractIdByEvmAddress`, which issues a DB query. If no row is found, it throws:
```
NotFoundError("No contract with the given evm address 0x<addr> has been found.")
``` [3](#0-2) 

The error middleware in `rest/middleware/httpErrorHandler.js` maps `NotFoundError` to HTTP 404 and, because `shouldReturnMessage` returns `true` for all 4xx codes, the **specific error message string is returned verbatim** in the response body. [4](#0-3) 

**Contrast with the "exists, no results" path:**

When the EVM address *does* exist in the DB, `computeContractIdFromString` returns the encoded ID successfully. `getContractResultsByIdAndFilters` returns an empty slice, and `getContractResultsById` returns early at line 877–878 with `res.locals[responseDataLabel]` already set to `{ results: [], links: { next: null } }`, producing HTTP 200. [5](#0-4) 

**Root cause:** The function performs a DB existence check (and throws on miss) *only* for EVM-address-format inputs, but silently returns empty results for numeric entity IDs that don't exist. This creates a distinguishable oracle.

**Existing checks are insufficient:** `validateContractIdParam` only validates format, not existence. No rate-limiting or authentication is applied to this endpoint. [6](#0-5) 

### Impact Explanation
An attacker can probe the entire 40-bit EVM address space (or targeted address ranges) to build a map of all contracts registered on the network. This leaks business-sensitive information: which addresses are active contracts, which are not yet deployed, and which have been deployed but have no call history. For a public ledger this may seem low-severity, but the mirror node is often deployed in permissioned or enterprise contexts where contract existence is considered non-public. The oracle is also useful as a precursor to targeted attacks (e.g., front-running a CREATE2 deployment to a known-empty address).

### Likelihood Explanation
No authentication, no rate-limiting, and no CAPTCHA is required. The endpoint is publicly reachable. The attacker needs only the ability to send HTTP GET requests. The response difference (HTTP 404 vs HTTP 200) is trivially detectable by any HTTP client. Scanning can be fully automated and parallelized. Realistic likelihood: **High**.

### Recommendation
Normalize the response for EVM-address inputs that are not found in the database to match the "exists but no results" response: return HTTP 200 with `{ results: [], links: { next: null } }` instead of throwing `NotFoundError`. Specifically, in `getContractResultsById`, catch the `NotFoundError` thrown by `computeContractIdFromString` and treat it as an empty result set:

```js
let contractId;
try {
  contractId = await ContractService.computeContractIdFromString(contractIdParam);
} catch (e) {
  if (e instanceof NotFoundError) {
    return; // response already initialized to { results: [], links: { next: null } }
  }
  throw e;
}
```

The same fix should be applied to `getContractStateById` and `getContractLogsById`, which share the same `computeContractIdFromString` call pattern. [7](#0-6) 

### Proof of Concept
**Precondition:** Mirror node REST API is accessible. Two EVM addresses are needed: one that exists as a contract (`0xAAAA...` registered in the DB) and one that does not (`0xDEAD...`).

**Step 1 – Probe a non-existent EVM address:**
```
GET /api/v1/contracts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/results
```
**Response (HTTP 404):**
```json
{
  "_status": {
    "messages": [
      {
        "message": "No contract with the given evm address 0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef has been found."
      }
    ]
  }
}
```

**Step 2 – Probe an existing EVM address with no call history:**
```
GET /api/v1/contracts/0x70f2b2914a2a4b783faefb75f459a580616fcb5e/results
```
**Response (HTTP 200):**
```json
{
  "results": [],
  "links": { "next": null }
}
```

**Step 3 – Automate enumeration:**
```python
import requests, itertools, binascii, os

base = "https://<mirror-node>/api/v1/contracts/{}/results"
for _ in range(10000):
    addr = "0x" + binascii.hexlify(os.urandom(20)).decode()
    r = requests.get(base.format(addr))
    if r.status_code == 200:
        print(f"EXISTS: {addr}")
    # 404 => does not exist
```

The HTTP status code alone is sufficient to distinguish existence from non-existence with zero false positives.
