### Title
Unauthenticated Request Storm via Unthrottled, Uncached `getByEvmAddressAndType()` in GraphQL Module

### Summary
The GraphQL module's `EntityServiceImpl.getByEvmAddressAndType()` issues a direct, uncached database query on every invocation with no rate limiting, no result deduplication, and no per-IP throttle. Any unauthenticated external user can flood the endpoint with thousands of identical `evmAddress` queries, each independently consuming a DB connection. During a network partition (or simply under sustained concurrent load), queued requests burst against the database simultaneously, causing connection pool exhaustion and effective DoS of the database tier.

### Finding Description

**Exact code path:**

`AccountController.account()` receives a GraphQL query with `evmAddress` set and calls: [1](#0-0) 

This delegates to `EntityServiceImpl.getByEvmAddressAndType()`: [2](#0-1) 

Which calls `EntityRepository.findByEvmAddress()` — a plain native SQL query with **no `@Cacheable` annotation**: [3](#0-2) 

**Root cause — failed assumptions:**

1. **No result caching**: The graphql module's `EntityRepository` has zero `@Cacheable` annotations. Contrast this with the web3 module's `EntityRepository`, which caches by EVM address. Every call to `findByEvmAddress` issues a live DB query.
2. **No rate limiting**: The graphql module contains no throttle filter, no Bucket4j bean, and no per-IP connection limit. The web3 module has `ThrottleConfiguration` and `ThrottleManagerImpl`; the graphql module has neither.
3. **No request deduplication**: Concurrent identical queries are not coalesced. Each request independently acquires a DB connection and executes the same `SELECT`.

**What the existing protections actually cover:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — limit structural complexity of a single query, not request volume. [4](#0-3) 
- `CachedPreparsedDocumentProvider` — caches the *parsed GraphQL document AST* keyed on the query string, not the query result. Repeated identical queries still hit the DB. [5](#0-4) 
- Parser limits (`maxCharacters=10000`, `maxTokens=1000`) — prevent oversized payloads, not high-volume small payloads. [6](#0-5) 

None of these controls bound the number of concurrent or sequential DB queries an unauthenticated caller can trigger.

### Impact Explanation

An attacker who saturates the DB connection pool prevents all other mirror node services (REST, web3, importer) that share the same PostgreSQL instance from executing queries. This is a full availability impact on the mirror node. The `entity` table query (`SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE`) is not trivially cheap — it requires an index scan and full row hydration. Under thousands of concurrent identical requests, the DB thread pool and connection pool are exhausted, causing cascading timeouts across all dependent services.

### Likelihood Explanation

Preconditions are minimal: no authentication, no API key, no privileged access required. The GraphQL endpoint is publicly reachable. A single attacker with a modest number of concurrent HTTP connections (e.g., using `ab`, `wrk`, or a simple async script) can sustain thousands of requests per second. The network partition scenario amplifies impact (queued requests burst simultaneously on reconnect) but is not required — sustained concurrent load achieves the same result. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add result caching** to `EntityRepository.findByEvmAddress()` and `findByAlias()` in the graphql module using `@Cacheable` with a short TTL (e.g., 1–5 seconds), mirroring the web3 module's pattern.
2. **Add rate limiting** to the graphql module: introduce a Bucket4j-based per-IP or global request-per-second throttle filter analogous to `ThrottleManagerImpl` in the web3 module.
3. **Add DataLoader-based deduplication** at the GraphQL layer so that within a single request execution, identical `evmAddress` lookups are batched and deduplicated.
4. **Set DB connection pool limits** (`spring.datasource.hikari.maximumPoolSize`) explicitly for the graphql module to bound blast radius.

### Proof of Concept

```bash
# Send 5000 concurrent identical evmAddress GraphQL queries
# No authentication required
seq 1 5000 | xargs -P 500 -I{} curl -s -X POST \
  http://<graphql-host>/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: { evmAddress: \"0x000000000000000000000000000000000000dead\" }) { id } }"}'
```

**Expected result**: DB connection pool exhausted; subsequent queries from all mirror node services begin timing out; Prometheus alert `GraphQLHighFileDescriptors` or DB connection errors appear in logs. The attack requires no credentials, no special headers, and no knowledge of valid addresses (a non-existent address still executes the full DB query and returns empty).

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
