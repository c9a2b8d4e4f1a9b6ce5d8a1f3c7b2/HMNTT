### Title
Unauthenticated Concurrent GET /schedules Requests Exhaust DB Connection Pool via Triple-Query Fan-Out

### Summary
`getSchedules()` in `rest/schedules.js` issues up to three database queries per request — one sequential and two in parallel via `Promise.all` — with no application-level rate limiting or in-flight concurrency control on the REST service. An unprivileged attacker sending many concurrent `GET /schedules?limit=<max>` requests can saturate the `pg` connection pool, causing connection-timeout failures for all subsequent DB-dependent requests including gossip-dependent schedule monitoring.

### Finding Description
**Exact code path:**

In `rest/schedules.js` `getSchedules()`:
- Line 241: `pool.queryQuietly(schedulesQuery, params)` — fetches up to `limit` schedule rows.
- Lines 261–264: `Promise.all([pool.queryQuietly(entityQuery, entityIds), pool.queryQuietly(signatureQuery, entityIds)])` — simultaneously dispatches an entity lookup and a `json_agg`-based signature aggregation query over all returned schedule IDs. [1](#0-0) 

The `signatureQuery` (lines 255–259) performs a `json_agg(...order by ts.consensus_timestamp)` grouped by `entity_id` across the `transaction_signature` table — a potentially expensive aggregation when schedules have many signatures. [2](#0-1) 

**Root cause — failed assumption:** The code assumes that the upstream infrastructure enforces per-client concurrency limits. It does not. The Helm chart for the REST service (`charts/hedera-mirror-rest/values.yaml`) configures only `circuitBreaker` and `retry` Traefik middlewares — **no `inFlightReq` and no `rateLimit`** — unlike the Rosetta service which explicitly configures both. [3](#0-2) 

Compare with Rosetta which has `inFlightReq: amount: 5` and `rateLimit: average: 10`: [4](#0-3) 

The pool is configured with a hard `max: config.db.pool.maxConnections` and a `connectionTimeoutMillis`. When the pool is saturated, new `pool.query()` calls queue; if `connectionTimeoutMillis` elapses, they throw a `DbError`, causing HTTP 500s for all concurrent users. [5](#0-4) 

**Why existing checks are insufficient:**
- `statement_timeout` limits individual query duration but does not prevent the pool from being held at capacity across many concurrent requests.
- `connectionTimeoutMillis` causes failures rather than preventing exhaustion.
- `circuitBreaker` only trips after errors are already occurring (i.e., after the DoS has begun).
- `buildAndValidateFilters` caps `limit` at `responseLimit.max` but this still allows large result sets with expensive `json_agg` aggregations. [6](#0-5) 

### Impact Explanation
When the DB connection pool is exhausted, every endpoint that calls `pool.queryQuietly` — including schedule monitoring, transaction lookups, and account queries — returns errors. For gossip-dependent schedule monitoring, this means the mirror node cannot serve schedule state queries, breaking any client relying on the REST API to observe pending or executed schedules. Severity is **High**: complete REST API unavailability for the duration of the attack with no privilege required.

### Likelihood Explanation
The attack requires only an HTTP client capable of issuing concurrent requests — no credentials, no special knowledge, no on-chain access. The endpoint is publicly reachable. The three-query fan-out (2 simultaneous via `Promise.all`) means each attacker request consumes 2 pool connections simultaneously at peak. With a default `maxConnections` of 10–20 (typical pg pool defaults), as few as 10 concurrent attacker requests can saturate the pool. The attack is trivially repeatable with tools like `ab`, `wrk`, or a simple async script.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` Traefik middleware** to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta configuration.
2. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) so that no single client IP can hold more than N concurrent in-flight requests.
3. **Reduce the fan-out**: merge the entity and signature queries into the main schedules query using JOINs or a single CTE, reducing per-request DB connection consumption from 3 to 1.
4. **Set a conservative `maxConnections`** and ensure `connectionTimeoutMillis` is short enough to fail fast rather than queue indefinitely.

### Proof of Concept
```bash
# Send 50 concurrent max-limit requests with no credentials
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/schedules?limit=100" &
done
wait

# Observe: subsequent legitimate requests return HTTP 500 / DbError
curl -v "https://<mirror-node>/api/v1/schedules"
# Expected: connection pool exhaustion error or timeout
```

Each concurrent request dispatches 2 simultaneous `pool.queryQuietly` calls (the `Promise.all` at lines 261–264), consuming 2 pool connections. With 50 concurrent requests, up to 100 pool connections are demanded simultaneously, exceeding any typical `maxConnections` setting and causing all queued requests to fail with `connectionTimeoutMillis` expiry.
