### Title
Unbounded Concurrent Throttled Subscriptions Exhaust Shared `boundedElastic` Scheduler, Blocking Live HCS Message Delivery

### Summary
`PollingTopicMessageRetriever.retrieve()` schedules all repeat-poll delays and timeout signals on the global `Schedulers.boundedElastic()` singleton. Because `poll()` executes a synchronous `Flux.fromStream()` DB call on that same scheduler thread, each active throttled subscription holds a thread for the duration of every DB query. There is no global cap on concurrent subscriptions, and the only per-connection limit (5 calls/connection) is trivially bypassed by opening many connections. When the shared pool is saturated, the `.timeout()` tasks queued on the same scheduler are also delayed, creating a feedback loop that prevents subscription cleanup and starves `SharedTopicListener.listen()`'s `publishOn(Schedulers.boundedElastic())`, which is the live-message delivery path for all subscribers.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructor allocates no private scheduler — it calls `Schedulers.boundedElastic()`, which returns the **process-wide singleton** `BoundedElasticScheduler`: [1](#0-0) 

`retrieve()` wires that scheduler into both the repeat-delay and the timeout: [2](#0-1) 

When a repeat fires, the scheduler thread executes `poll()`, which calls: [3](#0-2) 

`Flux.fromStream()` is **synchronous** — it iterates the JDBC stream on the calling thread, holding that `boundedElastic` thread for the full DB query duration.

The same global singleton is used by `SharedTopicListener.listen()` for live message delivery: [4](#0-3) 

And by the safety-check path in `TopicMessageServiceImpl`: [5](#0-4) 

**Root cause / failed assumption:** The design assumes the number of concurrent throttled subscriptions is small relative to the `boundedElastic` pool (10 × CPU cores). There is no global subscription counter with an enforced ceiling, and the only server-side guard is `maxConcurrentCallsPerConnection = 5`: [6](#0-5) 

That guard is per-TCP-connection, not global.

**Feedback loop:** The `.timeout(retrieverProperties.getTimeout(), scheduler)` timeout signal is also scheduled on the same saturated pool: [7](#0-6) 

When the pool is full, timeout tasks queue behind the poll tasks. Subscriptions that should expire after 60 s remain alive, adding more poll load, which delays timeouts further — a self-reinforcing cycle.

### Impact Explanation

Once the `boundedElastic` pool is saturated, `publishOn(Schedulers.boundedElastic())` in `SharedTopicListener.listen()` queues indefinitely. This is the **only** path through which live HCS consensus messages are dispatched to all active subscribers. Legitimate subscribers stop receiving new topic messages regardless of how many messages the importer is writing to the database. The default 60 s timeout cannot self-heal because its own scheduler signal is stuck in the same queue. The result is a sustained, server-wide denial of HCS topic message delivery — the mirror node's primary function for consensus clients.

### Likelihood Explanation

No authentication is required to open a gRPC `subscribeTopic` stream. An attacker opens `C` TCP connections, each carrying 5 concurrent `subscribeTopic` calls (the per-connection maximum), for a total of `5C` subscriptions. Each throttled subscription polls every 2 s (default `pollingFrequency`); if each DB query takes ~50 ms, average thread occupancy per subscription is ~2.5 %. To exhaust 80 threads (8-core host), roughly 3 200 concurrent subscriptions (640 connections) are needed. A single machine with a modest connection pool can sustain this. The attack is repeatable, requires no credentials, and no existing server-side control terminates it automatically once the feedback loop is established.

### Recommendation

1. **Enforce a global subscription ceiling** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum, returning `RESOURCE_EXHAUSTED`.
2. **Isolate the retriever scheduler** from the live-delivery scheduler. Give `PollingTopicMessageRetriever` a dedicated, bounded `Schedulers.newBoundedElastic(...)` instance so poll saturation cannot starve `SharedTopicListener.publishOn`.
3. **Make `poll()` non-blocking**: replace `Flux.fromStream(topicMessageRepository.findByFilter(...))` with a reactive R2DBC query so scheduler threads are not held during DB I/O.
4. **Add per-IP or per-client connection limits** at the Netty/load-balancer layer in addition to `maxConcurrentCallsPerConnection`.
5. **Cap `retryWhen` attempts** or add exponential back-off with a ceiling to prevent retry storms amplifying the load.

### Proof of Concept

```
# Prerequisites: grpcurl installed, mirror node running on localhost:5600
# with a valid topic (e.g. 0.0.12345)

# Step 1 – open 640 connections, each with 5 concurrent subscribeTopic streams
#          (total 3200 throttled subscriptions, startTime=0 ensures throttled=true path)
for i in $(seq 1 640); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0}}' \
      localhost:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 2 – after ~30 s, open a legitimate subscriber and observe no messages arrive
grpcurl -plaintext \
  -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0}}' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

# Expected: the legitimate subscriber hangs with no output even as new
# consensus messages are written to the database, because
# SharedTopicListener.publishOn(Schedulers.boundedElastic()) is queued
# behind the saturated poll tasks.
# The server log will show boundedElastic worker threads at 100% utilisation
# and timeout signals not firing within the configured 60 s window.
```
