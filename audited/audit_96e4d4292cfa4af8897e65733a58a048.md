### Title
Unbounded gRPC Connection Count Enables Resource Exhaustion DoS

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection` (capped at 5), which limits concurrent RPC calls per connection but places no bound on the total number of TCP connections a client may open. An unauthenticated attacker can open an unbounded number of idle connections to port 5600, exhausting file descriptors, Netty per-connection memory buffers, and thread pool slots, rendering the service unavailable to legitimate clients.

### Finding Description
**Exact code path:**

`GrpcConfiguration.java` lines 28–35 configure the Netty server with only two settings: [1](#0-0) 

`NettyProperties.java` exposes only one tunable — `maxConcurrentCallsPerConnection = 5`: [2](#0-1) 

**Root cause:** `maxConcurrentCallsPerConnection` is a per-connection RPC call limit. It does not restrict how many distinct TCP connections a single IP or client may establish. `NettyServerBuilder` exposes `maxConnectionAge()`, `maxConnectionIdle()`, and (via Netty channel options) a total connection count limit — none of which are set here. The server will accept connections up to the OS file descriptor limit.

**Failed assumption:** The configuration assumes that limiting calls-per-connection is sufficient to bound resource consumption. It is not: an attacker opens N connections each with 0–5 calls, consuming N file descriptors, N Netty channel objects, and N slots in the `applicationTaskExecutor` thread pool without triggering any existing guard.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` — only limits RPC calls per connection, not connection count.
- GCP backend policy `maxRatePerEndpoint: 250` — this is an RPS limit on routed requests, not a TCP connection count limit; idle open connections bypass it entirely. [3](#0-2) 
- No authentication is required to open a gRPC connection to this service.

### Impact Explanation
An attacker who exhausts the JVM process's file descriptor limit causes all subsequent `accept()` calls to fail, making the gRPC service completely unavailable. The existing `GrpcHighFileDescriptors` Prometheus alert (threshold 80%) confirms the operators already treat file descriptor exhaustion as a critical production risk: [4](#0-3) 

Beyond file descriptors, each Netty channel allocates per-connection memory buffers and may occupy executor threads, compounding memory and CPU pressure. The service provides consensus topic subscriptions and network address book queries — a DoS here disrupts all downstream consumers of Hedera mirror data.

### Likelihood Explanation
Preconditions are minimal: network reachability to port 5600 and the ability to open TCP sockets (any standard OS). No credentials, tokens, or protocol knowledge beyond "open a TCP connection" are required. The attack is trivially scriptable (`for i in $(seq 1 10000); do grpc_connect &; done`), repeatable, and distributable across multiple source IPs to evade any upstream IP-based rate limiting. The gRPC port is publicly exposed via the Kubernetes ingress/gateway. [5](#0-4) 

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

1. **`maxConnectionAge` / `maxConnectionIdle`** — force periodic connection recycling so stale connections do not accumulate indefinitely.
2. **`maxConnectionAgeGrace`** — allow in-flight streams to complete before forceful close.
3. **Total connection count limit** — set a Netty `ChannelOption` (e.g., via a custom `ServerBuilderCustomizer` that accesses the underlying `ServerBootstrap`) or use a `io.grpc.ServerInterceptor` that tracks and rejects connections above a threshold.
4. Expose `maxConnectionAge` and `maxConnectionIdle` as configurable fields in `NettyProperties` alongside `maxConcurrentCallsPerConnection`.

Example additions to the lambda in `grpcServerConfigurer()`:
```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
```

### Proof of Concept
**Preconditions:** Network access to the gRPC server on port 5600; `grpcurl` or any gRPC client library.

**Steps:**
1. Identify the server's file descriptor limit: `cat /proc/<pid>/limits | grep "open files"` (typically 65535 for a JVM process).
2. Open connections in a tight loop without sending any RPC:
   ```bash
   for i in $(seq 1 20000); do
     grpcurl -plaintext -connect-timeout 3600 <host>:5600 & 
   done
   ```
3. Observe the `process_files_open_files` metric climb toward the limit.
4. Once the limit is reached, all new legitimate `subscribeToTopic` or `getNodes` calls receive a connection-refused or transport error.
5. The `GrpcHighFileDescriptors` alert fires; the service is effectively down for all clients.

**Result:** Complete DoS of the gRPC mirror node API with no authentication required, exploitable by any unprivileged external user.
