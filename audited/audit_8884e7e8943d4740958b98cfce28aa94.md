### Title
DB Connection Pool Exhaustion via Unbounded Concurrent `RetrieveBalanceAtBlock()` Requests Causing Balance Query DoS

### Summary
The Rosetta API's `RetrieveBalanceAtBlock()` function issues up to three sequential database queries per request with no application-level concurrency or rate limiting. An unprivileged attacker sending a sustained flood of concurrent requests with large `consensusEnd` values can hold all 100 pooled DB connections for up to 20 seconds each, exhausting the pool and causing every subsequent `getBalanceChange()` call to fail with `ErrDatabaseError`, denying balance query service to all users. The only mitigations are an optional Traefik middleware (disabled by default unless `global.middleware=true`) and a 20-second statement timeout that limits each wave but does not prevent continuous exhaustion.

### Finding Description

**Exact code path:**

`RetrieveBalanceAtBlock()` (`rosetta/app/persistence/account.go`, lines 161–223) calls three functions sequentially, each acquiring a separate DB connection from the shared pool via `GetDbWithContext()`:

1. `getCryptoEntity()` — lines 225–265, acquires one connection
2. `getLatestBalanceSnapshot()` — lines 267–311, acquires one connection, runs two queries
3. `getBalanceChange()` — lines 313–339, acquires one connection, runs `balanceChangeBetween`

The `balanceChangeBetween` query (lines 21–29) is:
```sql
select coalesce((
  select sum(amount) from crypto_transfer
  where consensus_timestamp > @start and consensus_timestamp <= @end
    and entity_id = @account_id and (errata is null or errata <> 'DELETE')
), 0) as value
```
With a large attacker-supplied `consensusEnd` (e.g., `int64` max), the range `[snapshotTimestamp, consensusEnd]` spans the entire `crypto_transfer` table, causing a full or near-full index scan that holds the connection for up to the 20-second `statementTimeout`.

**DB pool configuration** (`rosetta/app/db/db.go`, lines 31–33):
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default 20
sqlDb.SetConnMaxLifetime(...)
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default 100
```
Default `maxOpenConnections = 100`, `statementTimeout = 20s` (docs/configuration.md, lines 658, 660).

**Application-level rate limiting**: None. The Go application has zero built-in concurrency controls. The only defense is the Traefik middleware defined in `charts/hedera-mirror-rosetta/templates/middleware.yaml` (line 3):
```yaml
{{ if and .Values.global.middleware .Values.middleware -}}
```
This is **conditionally rendered** — it is absent in any non-Kubernetes deployment or when `global.middleware=false`.

**Why the Traefik middleware is insufficient even when enabled:**
- `inFlightReq: amount: 5` (values.yaml, line 153) limits 5 concurrent requests **per source IP**. An attacker using 20 IPs sends 5 × 20 = 100 concurrent requests, saturating the pool.
- `retry: attempts: 3` (values.yaml, line 162) **amplifies** the attack: each pool-exhaustion failure triggers 3 automatic retries, tripling DB pressure.
- `rateLimit: average: 10` (values.yaml, line 158) uses `sourceCriterion: requestHost: true` — this is a **global** 10 req/s limit, but already-in-flight requests are not evicted; 10 req/s × 20s timeout = 200 in-flight requests possible, exceeding the 100-connection pool.

### Impact Explanation
When the pool is exhausted, `GetDbWithContext()` blocks until a connection is available or the request context times out. Any request that cannot acquire a connection within the HTTP write timeout (default 10s, `rosettaConfig.Http.WriteTimeout`) will fail. `getBalanceChange()` returns `hErrors.ErrDatabaseError` (line 335), propagating a 500-class error to all concurrent and queued balance requests. This is a complete denial of the `/account/balance` endpoint for all users for the duration of the attack. The `statementTimeout` of 20 seconds means each attack wave lasts 20 seconds, but the attacker can sustain continuous flooding to maintain the outage indefinitely.

### Likelihood Explanation
**Without Traefik middleware** (bare-metal, Docker, or `global.middleware=false` deployments): trivially exploitable by a single attacker with a basic HTTP flood tool (e.g., `ab`, `wrk`, `hey`) sending 100+ concurrent requests. No authentication, no API key, no special knowledge required — only a valid account ID and an arbitrary large integer for `consensusEnd`.

**With Traefik middleware**: requires 20 source IPs (easily obtained via cloud VMs, proxies, or a small botnet) to bypass the per-IP `inFlightReq` limit. The `retry` amplification makes this easier, not harder.

The attack is repeatable, stateless, and requires no persistence on the target system.

### Recommendation

1. **Application-level concurrency limit**: Add a semaphore or `golang.org/x/time/rate` token-bucket limiter in the HTTP handler layer (before DB calls) to cap total in-flight balance requests, independent of infrastructure.

2. **`consensusEnd` validation**: Reject requests where `consensusEnd` exceeds the latest known consensus timestamp (already available via `blockRepository.RetrieveLatest()`). This eliminates the slow full-table-scan vector entirely.

3. **Reduce `statementTimeout`**: Lower from 20s to 2–5s for the `balanceChangeBetween` query specifically, reducing the window each connection is held.

4. **Make middleware mandatory**: Remove the `{{ if and .Values.global.middleware ... }}` guard or enforce rate limiting at the application layer so it cannot be accidentally omitted.

5. **Separate DB pools**: Use a dedicated, smaller connection pool for balance queries so exhaustion cannot affect other endpoints.

### Proof of Concept

**Preconditions**: Rosetta API accessible without authentication (standard deployment). No Traefik middleware, or attacker controls 20+ IPs.

**Steps**:
```bash
# Step 1: Identify any valid account ID (e.g., 0.0.2 is always present)
# Step 2: Send 110 concurrent POST requests with max consensusEnd
for i in $(seq 1 110); do
  curl -s -X POST http://<rosetta-host>:5700/account/balance \
    -H 'Content-Type: application/json' \
    -d '{
      "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
      "account_identifier": {"address":"0.0.2"},
      "block_identifier": {"index": 9999999999}
    }' &
done
wait

# Step 3: While flood is running, send a legitimate request:
curl -X POST http://<rosetta-host>:5700/account/balance \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.98"},
       "block_identifier":{"index":1}}'
# Expected result: {"code":500,"message":"There was an error connecting to the database","retriable":true}
```

The legitimate request in Step 3 fails with `ErrDatabaseError` because all 100 pool connections are held by the flood requests executing the slow `crypto_transfer` range scan in `getBalanceChange()`.
