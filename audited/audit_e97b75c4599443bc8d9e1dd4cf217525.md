Based on my analysis of the code, here is my finding:

### Title
`responseTimeout` Not Applied to `streamBlocks()` — Malicious Block Node Can Stall Streaming Thread for 30s Per Block

### Summary
`StreamProperties.getResponseTimeout()` (default 400 ms) is applied only to the `getBlockRange()` / `ServerStatusRequest` call via `.withDeadlineAfter()`, but is never used in `streamBlocks()`. The streaming loop instead uses `commonDownloaderProperties.getTimeout()` (default 30 s) as a per-block deadline. A malicious block node can accept the subscription and then drip-feed responses at just under 30 s per block, holding the single streaming thread for an arbitrarily long time relative to the intended 400 ms response budget.

### Finding Description
**`getBlockRange()` — timeout applied correctly:** [1](#0-0) 

```java
final var blockNodeService = BlockNodeServiceGrpc.newBlockingStub(channel)
        .withDeadlineAfter(streamProperties.getResponseTimeout()); // 400 ms
```

**`streamBlocks()` — gRPC call created with `CallOptions.DEFAULT` (no deadline):** [2](#0-1) 

The per-read timeout comes from `BlockAssembler.timeout()`: [3](#0-2) 

which is seeded from `commonDownloaderProperties.getTimeout()`: [4](#0-3) 

That value defaults to **30 seconds**: [5](#0-4) 

The `BlockAssembler.timeout()` measures elapsed time from the first `read()` call and resets after each `onEndOfBlock()`: [6](#0-5) 

This means the entire block assembly window is 30 s, and between blocks the stopwatch resets, granting another full 30 s. A malicious node can therefore delay each block by up to ~30 s indefinitely, 75× longer than the 400 ms `responseTimeout` that was clearly intended to bound response latency.

**Root cause:** `responseTimeout` is never passed into `streamBlocks()` or `BlockAssembler`; the only timeout in the streaming path is the coarse per-block `commonDownloaderProperties.getTimeout()`.

### Impact Explanation
`BlockNodeSubscriber` uses a single-threaded executor and a blocking streaming loop: [7](#0-6) [8](#0-7) 

A rogue block node that drip-feeds one block every ~30 s can reduce the mirror node's effective ingestion rate from thousands of blocks per second to ~2 blocks per minute. This constitutes a denial-of-service against block ingestion. The `onError()` / readmit mechanism is only triggered on exceptions, not on slow-but-valid responses, so the node is never penalized. [9](#0-8) 

### Likelihood Explanation
The attacker must control or compromise a configured block node. Block nodes are operator-configured infrastructure, so this is not a zero-privilege attack. However, a compromised or rogue block node operator (insider threat, supply-chain compromise, or BGP/DNS hijack of the node endpoint) can trivially exploit this with no cryptographic bypass required — simply by throttling gRPC response delivery. The attack is repeatable and silent (no errors are logged, no metrics are incremented).

### Recommendation
1. Apply `responseTimeout` as a per-read deadline inside `streamBlocks()`, replacing or supplementing `assembler.timeout()`:
   ```java
   response = grpcCall.read(streamProperties.getResponseTimeout().toMillis(), TimeUnit.MILLISECONDS)
   ```
2. Alternatively, set a gRPC deadline on the streaming call itself using `CallOptions.DEFAULT.withDeadlineAfter(...)` for the overall block, and use `responseTimeout` for the per-message read.
3. Treat a read timeout in `streamBlocks()` as an error that increments the error counter and triggers `onError()`, so slow nodes are eventually marked inactive.

### Proof of Concept
1. Stand up a gRPC server implementing `BlockStreamSubscribeService`.
2. Configure the mirror node to use this server as its block node.
3. On `subscribeBlockStream`, accept the subscription immediately (so `getBlockRange()` succeeds within 400 ms).
4. For each `SubscribeStreamResponse`, delay sending by 29 seconds before writing to the stream.
5. Observe that the mirror node's streaming thread blocks for 29 s per response, ingestion stalls, no errors are raised, and the node is never marked inactive — while `responseTimeout` (400 ms) is never triggered.
