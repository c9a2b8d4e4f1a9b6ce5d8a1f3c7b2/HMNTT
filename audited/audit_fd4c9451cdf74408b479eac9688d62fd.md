### Title
Unbounded In-Memory Opcode Trace Accumulation Causes OOM DoS via Single Authenticated-Free Request

### Summary
The `/api/v1/contracts/results/{txHash}/opcodes` endpoint accepts `stack=true`, `memory=true`, and `storage=true` simultaneously with no cap on the number of opcodes collected or the per-opcode data volume. When replaying a historically gas-heavy transaction, the EVM tracer appends an unbounded `Opcode` object (containing full stack snapshot, full memory snapshot, and full storage snapshot) for every single EVM instruction executed, exhausting JVM heap and crashing the mirror-node process. The only guard — a token-bucket rate limiter of 1 request/second — does not prevent a single in-flight request from consuming all available heap.

### Finding Description

**Code path:**

1. `OpcodesController.getContractOpcodes()` (lines 52–68) accepts the three boolean flags with no mutual-exclusion or combined-flag restriction and calls `throttleManager.throttleOpcodeRequest()`.
2. `ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–56) only checks `opcodeRateLimitBucket.tryConsume(1)`. The default `opcodeRequestsPerSecond = 1` (ThrottleProperties line 29). This is a rate limit, not a concurrency or payload-size limit; one request per second is still permitted, and a single request is sufficient to trigger OOM.
3. `OpcodeServiceImpl.processOpcodeCall()` (lines 64–74) constructs `new OpcodeContext(opcodeRequest, (int) params.getGas() / 3)`. For a transaction with a 15 M gas limit this pre-allocates an `ArrayList` of 5 M capacity.
4. `OpcodeActionTracer.tracePostExecution()` (lines 44–62) is called once per EVM opcode. It calls `captureMemory()`, `captureStack()`, and `captureStorage()` from `AbstractOpcodeTracer` (lines 40–128) and appends the result to `OpcodeContext.opcodes` via `addOpcodes()` (OpcodeContext line 54–56). There is **no size check, no opcode-count ceiling, and no per-opcode data cap** anywhere in this path.
5. `captureMemory()` (lines 40–58) allocates a new `ArrayList<String>` sized to `frame.memoryWordSize()` — unbounded as EVM memory grows. `captureStack()` (lines 60–73) captures up to 1024 stack items. `captureStorage()` (lines 75–128) iterates all storage accesses accumulated so far in the `RootProxyWorldUpdater`.

**Root cause:** The failed assumption is that the 1-req/s rate limit bounds memory consumption. It does not. Memory consumption is a function of the replayed transaction's complexity, not of request rate.

**Exploit flow:**
- Attacker identifies any historical contract transaction that consumed a large amount of gas (e.g., a complex DeFi interaction near the 15 M gas ceiling, which is publicly visible on-chain).
- Attacker sends: `GET /api/v1/contracts/results/<txHash>/opcodes?stack=true&memory=true&storage=true` with `Accept-Encoding: gzip`.
- No authentication is required; the endpoint is public.
- The EVM re-executes the transaction. For N opcodes executed, the tracer appends N `Opcode` objects each holding: up to 1024 × ~66-byte stack strings, a growing memory list (each 32-byte word as a hex string), and a map of all storage accesses to date.
- A 15 M gas transaction executing simple opcodes can produce millions of trace entries. At even a conservative 10 KB per opcode entry (stack + memory + storage), 1 M opcodes = ~10 GB of heap allocation, far exceeding typical JVM heap sizes.
- The JVM throws `OutOfMemoryError`, crashing the web3 mirror-node process.

### Impact Explanation
A single unauthenticated HTTP request can crash the mirror-node web3 service. Because the feature flag `hiero.mirror.web3.opcode.tracer.enabled` must be `true` for the endpoint to be reachable (OpcodesProperties line 11), the attack surface is limited to deployments where the operator has explicitly enabled the tracer. However, when enabled, the crash is deterministic and repeatable: the attacker can re-trigger it once per second indefinitely, preventing the service from recovering. This constitutes a complete denial-of-service of the mirror-node web3 API, matching the stated severity of shutting down ≥30% of network processing nodes without brute force.

### Likelihood Explanation
The preconditions are minimal: the opcode tracer must be enabled (an operator opt-in), and the attacker needs only a valid transaction hash for a gas-heavy contract call — information freely available from any Hedera block explorer. No credentials, no privileged access, and no brute force are required. The attack is repeatable at 1 req/s, making sustained DoS trivial. Any public-facing mirror-node deployment with the tracer enabled is immediately exploitable.

### Recommendation
1. **Hard cap on opcode count:** Introduce a configurable `maxOpcodes` property (e.g., 100 000) in `OpcodesProperties`. In `OpcodeActionTracer.tracePostExecution()`, check `opcodes.size() >= maxOpcodes` before calling `addOpcodes()` and abort or truncate.
2. **Hard cap on per-opcode data:** In `captureMemory()`, limit `size` to a configurable maximum (e.g., 1024 words). In `captureStack()`, the EVM already caps at 1024 items, but enforce it explicitly. In `captureStorage()`, limit the number of entries returned.
3. **Concurrency limit:** Add a semaphore or `@Bulkhead` (Resilience4j) to `getContractOpcodes()` to prevent more than N simultaneous in-flight opcode replay requests (e.g., N=2), independent of the per-second rate limit.
4. **Execution timeout:** Wrap `contractDebugService.processOpcodeCall()` in a `CompletableFuture` with a hard timeout (e.g., 30 s) to bound CPU and heap lifetime per request.
5. **Heap guard:** Consider adding a JVM `-XX:+HeapDumpOnOutOfMemoryError` and a process supervisor, but treat this only as a recovery measure, not a fix.

### Proof of Concept
```
# 1. Find a gas-heavy historical contract transaction hash from a block explorer.
#    Example: a complex DeFi swap near the 15M gas limit.
TX_HASH="0x<large_gas_transaction_hash>"

# 2. Send the maximally verbose opcode trace request (no auth required):
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# 3. Observe: the mirror-node web3 JVM process throws OutOfMemoryError and exits.
#    Subsequent requests return connection-refused until the process is restarted.

# 4. Repeat once per second to prevent recovery:
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" &
  sleep 1
done
```
