### Title
Unauthenticated `/health/readiness` Polling Causes Recursive Request Amplification and Resource Exhaustion

### Summary
The `checkNetworkStatus()` function in `rosetta/app/middleware/health.go` creates a shared `rosettaClient` using `client.NewAPIClient(cfg)` with a `nil` HTTP client (defaulting to `http.DefaultClient` with no concurrency cap), and registers it as a health check with no rate limiting. Every unauthenticated external GET to `/health/readiness` triggers two internal loopback HTTP requests (`/network/list` and `/network/status`) plus three database queries, creating a 3× amplification factor with no throttle. Under sustained concurrent polling, this exhausts goroutines, database connections, and file descriptors, raising node resource consumption well above 30%.

### Finding Description

**Code path:**

`rosetta/app/middleware/health.go`, `checkNetworkStatus()`, lines 80–101:

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
    serverUrl := fmt.Sprintf("http://localhost:%d", port)
    cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)  // nil → http.DefaultClient
    rosettaClient := client.NewAPIClient(cfg)

    return func(ctx context.Context) (checkErr error) {
        networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
        // ...
        _, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{...})
        // ...
    }
}
```

**Root cause — three compounding failures:**

1. **`nil` HTTP client / no connection cap.** Passing `nil` to `client.NewConfiguration` causes the rosetta-sdk-go to use `http.DefaultClient` / `http.DefaultTransport`. Under concurrent load, Go's transport creates new TCP connections to localhost beyond the idle-pool limit (`MaxIdleConnsPerHost: 2`), each consuming a file descriptor and socket buffer. There is no `MaxConnsPerHost` set, so the number of live connections is bounded only by the OS fd limit.

2. **Request amplification.** Each external GET `/health/readiness` → 2 internal loopback HTTP requests (`NetworkList` + `NetworkStatus`) → 3 database queries inside `NetworkStatus` (`RetrieveGenesis`, `RetrieveLatest`, `Entries`). The internal requests are processed by the same server goroutine pool as regular API traffic.

3. **Zero rate limiting or concurrency cap on the health endpoint.** A grep across all `rosetta/**/*.go` confirms no rate limiter, semaphore, or throttle exists for the health path. The `hellofresh/health-go` `HandlerFunc` runs all registered checks concurrently for every incoming request with no deduplication or coalescing.

**Amplification math:** N concurrent external requests → N×2 internal HTTP requests in flight → N×3 DB queries → 3N goroutines held for up to 10 s (the `health.Config{Timeout: time.Second * 10}` deadline). At N=50 req/s sustained: 150 goroutines, 150 DB connections, and up to 100 loopback TCP sockets accumulate within the first timeout window.

**Existing check — insufficient:** The only protection is the 10-second context timeout passed by `health-go` to the check closure. This bounds per-request duration but does not limit concurrency or rate, so the steady-state resource level scales linearly with attacker request rate.

### Impact Explanation

- **Goroutine exhaustion:** 3N goroutines per second accumulate; at modest rates (50 req/s) this is 1,500 goroutines held simultaneously, consuming ~1.5 GB of stack space.
- **DB connection exhaustion:** The default pool cap is 100 open connections (`maxOpenConnections: 100`). At N=34 concurrent health requests, all 100 DB connections are consumed by health-check-induced `NetworkStatus` calls, starving legitimate API traffic.
- **File descriptor / socket exhaustion:** `http.DefaultTransport` creates new TCP connections to localhost beyond the idle pool; each consumes an fd. Under sustained load this approaches the process fd limit.
- **CPU amplification:** Each health check invocation causes the server to parse, route, and execute two full API requests through the metrics and tracing middleware stack.

Severity: **High** — a single unauthenticated attacker can cause service degradation or outage for all legitimate users.

### Likelihood Explanation

- **No privileges required.** The endpoint is publicly reachable with a plain `GET /health/readiness`.
- **Trivially reproducible.** A single `ab -c 100 -n 100000 http://<host>:5700/health/readiness` or equivalent is sufficient.
- **Persistent effect.** Because each request holds goroutines and DB connections for up to 10 seconds, even a moderate sustained rate (10–50 req/s) maintains elevated resource consumption continuously.
- **No detection barrier.** The attack traffic is indistinguishable from legitimate Kubernetes liveness/readiness probe traffic.

### Recommendation

1. **Add a concurrency gate / rate limiter on the health endpoint.** Use a `sync.Semaphore` or token-bucket to cap simultaneous health check executions (e.g., max 2 concurrent).
2. **Coalesce concurrent health checks.** Cache the last health result for a short TTL (e.g., 5 s) and return it to subsequent callers without re-running the checks, similar to how `hellofresh/health-go` v5 supports `WithCacheDuration`.
3. **Set an explicit HTTP client with `MaxConnsPerHost`.** Replace `nil` with a configured `*http.Client` that sets `Transport.MaxConnsPerHost` to a small value (e.g., 2) to bound loopback socket usage:
   ```go
   httpClient := &http.Client{
       Transport: &http.Transport{MaxConnsPerHost: 2},
   }
   cfg := client.NewConfiguration(serverUrl, "readiness-check", httpClient)
   ```
4. **Restrict the health endpoint at the network/ingress layer** so it is only reachable from internal/cluster sources, not the public internet.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, port 5700 reachable.

```bash
# Step 1: Baseline — record CPU, goroutine count, DB connections
curl -s http://localhost:5700/metrics | grep -E 'go_goroutines|db_open'

# Step 2: Flood /health/readiness with 100 concurrent connections
ab -c 100 -n 50000 http://localhost:5700/health/readiness &

# Step 3: After 15 seconds, re-sample metrics
sleep 15
curl -s http://localhost:5700/metrics | grep -E 'go_goroutines|db_open'

# Expected result:
# go_goroutines increases by 150+ (3× amplification × 50 concurrent)
# db connections approach or hit the 100-connection cap
# /network/list and /network/status latency for legitimate callers degrades
# hiero_mirror_rosetta_request_inflight{route="/health/readiness"} shows sustained high value
```

**Observable outcome:** Goroutine count, DB open connections, and CPU utilization all increase by >30% compared to the 24-hour baseline, triggered entirely by unauthenticated GET requests to a single endpoint.
