### Title
Unauthenticated Connection Pool Exhaustion DoS via Unbounded Concurrent `getNftAllowances()` Requests During Network Partition

### Summary
The `getNftAllowances()` endpoint in `NftAllowanceServiceImpl` issues synchronous blocking JDBC queries through jOOQ with no application-level rate limiting. Because the `rest-java` module applies no per-endpoint throttle (unlike the `web3` module which has `ThrottleManager`), an unauthenticated attacker can flood `/api/v1/accounts/{id}/allowances/nfts` with concurrent requests. During a network partition, each request holds a HikariCP connection until timeout, exhausting the pool and denying service to all legitimate callers.

### Finding Description
**Code path:**

- `AllowancesController.getNftAllowances()` (`rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, lines 57–83) — publicly accessible GET endpoint, no authentication, no rate limiting.
- Delegates to `NftAllowanceServiceImpl.getNftAllowances()` (`NftAllowanceServiceImpl.java`, lines 21–31), which calls `repository.findAll(request, id)` at line 30.
- `NftAllowanceRepositoryCustomImpl.findAll()` (`NftAllowanceRepositoryCustomImpl.java`, lines 41–46) executes a synchronous blocking jOOQ query: `dslContext.selectFrom(NFT_ALLOWANCE).where(...).limit(...).fetchInto(NftAllowance.class)`.

**Root cause — failed assumption:** The design assumes infrastructure-level protections (reverse proxy, load balancer) will prevent request flooding. No application-level rate limiting exists in the `rest-java` module for this endpoint. Confirmed: grep for `ThrottleManager`, `RateLimiter`, `rateLimit` in `rest-java/src/main/java/**` returns zero matches. By contrast, the `web3` module has `ThrottleConfiguration` and `ThrottleManagerImpl` with explicit per-second bucket limits.

**Connection pool configuration:** The `rest-java` module has no explicit HikariCP `maximumPoolSize` or `connectionTimeout` set in its YAML (grep for `hikari`, `maximumPoolSize`, `connectionTimeout` in `rest-java/**` returns zero matches). This means HikariCP defaults apply: pool size = 10, `connectionTimeout` = 30 000 ms. The documented `hiero.mirror.restJava.db.statementTimeout` = 10 000 ms is a PostgreSQL-level hint sent with the query — during a network partition the query never reaches the database, so this timeout does not fire; the thread blocks at the JDBC socket level until the OS TCP timeout or HikariCP's `connectionTimeout` expires.

**Exploit flow:**
1. Attacker (or botnet) sends ≥ 10 concurrent GET requests to `/api/v1/accounts/0.0.1/allowances/nfts`.
2. Each request passes `checkOwnerSpenderParamValidity()` (only validates parameter semantics) and reaches `repository.findAll()`.
3. All 10 HikariCP connections are checked out; each blocks waiting for a DB response that never arrives (network partition).
4. Subsequent requests queue inside HikariCP waiting up to 30 s for a free connection.
5. Attacker continuously re-sends requests to keep the pool saturated; legitimate requests time out or are rejected.
6. The service is effectively unavailable for the duration of the partition plus the attacker's flood.

### Impact Explanation
Complete denial of service for the `rest-java` API tier. All endpoints sharing the same HikariCP pool (not just NFT allowances) are affected once the pool is exhausted. The service cannot recover until the attacker stops flooding and all held connections time out. Severity: **High** — full service unavailability, no privilege required, no interaction beyond sending HTTP GET requests.

### Likelihood Explanation
Trivially exploitable: the endpoint requires no authentication, no API key, and no special knowledge beyond the public OpenAPI spec. A single attacker with a modest number of concurrent HTTP connections (≥ 10) can trigger the condition. The network partition scenario (the amplifying factor) can be induced by the attacker themselves via network-layer attacks (BGP hijack, firewall rule injection on a shared cloud tenant, etc.) or exploited opportunistically during any real outage. The attack is repeatable and self-sustaining as long as the attacker maintains the flood.

### Recommendation
1. **Add application-level rate limiting to `AllowancesController`** — mirror the `web3` module's `ThrottleManager`/`ThrottleConfiguration` pattern (bucket4j) for the `rest-java` endpoints, enforcing a per-IP or global requests-per-second cap.
2. **Configure explicit HikariCP bounds** in `rest-java` application YAML: set `spring.datasource.hikari.maximum-pool-size`, `connection-timeout` (e.g., 5 000 ms), and `keepalive-time` to fail fast during partitions.
3. **Add a circuit breaker** (e.g., Resilience4j) around `repository.findAll()` so that repeated DB failures open the circuit and return 503 immediately instead of blocking threads.
4. **Set `socket-timeout`** at the JDBC driver level (PostgreSQL `socketTimeout` property) so that network-level hangs are bounded independently of HikariCP's pool-acquisition timeout.

### Proof of Concept
```bash
# Precondition: network partition between rest-java pod and PostgreSQL
# (simulate with: iptables -A OUTPUT -p tcp --dport 5432 -j DROP on the app host)

# Step 1: Exhaust the connection pool (default size 10)
for i in $(seq 1 20); do
  curl -s "http://<rest-java-host>:8084/api/v1/accounts/0.0.1/allowances/nfts" &
done
wait

# Step 2: Send a legitimate request — observe it hangs for ~30s then fails
time curl -v "http://<rest-java-host>:8084/api/v1/accounts/0.0.2/allowances/nfts"
# Expected: connection hangs ~30 000 ms, then returns 500 / connection refused
# (HikariCP connectionTimeout exhausted, pool still saturated by attacker wave)

# Step 3: Repeat step 1 continuously to maintain DoS
while true; do
  for i in $(seq 1 20); do
    curl -s "http://<rest-java-host>:8084/api/v1/accounts/0.0.1/allowances/nfts" &
  done
  sleep 25  # re-flood before previous connections time out
done
```
