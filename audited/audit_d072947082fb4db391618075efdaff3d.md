### Title
Unauthenticated, Unbounded `GET /api/v1/network/stake` Endpoint Enables Database-Exhaustion DoS via Repeated `getLatestNetworkStake()` Calls

### Summary
The `GET /api/v1/network/stake` endpoint is publicly accessible with no authentication, no service-layer rate limiting, and no response caching. Every call unconditionally executes a correlated SQL subquery against the `network_stake` table. An unprivileged attacker can flood this endpoint to exhaust the database connection pool, degrading or denying service to all legitimate consumers of network staking state.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (line 127–130) handles `GET /api/v1/network/stake` with no access control or throttle annotation: [1](#0-0) 

It delegates directly to `NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56), which has no caching, no rate guard, and no circuit breaker: [2](#0-1) 

Every invocation fires `NetworkStakeRepository.findLatest()`, which executes a correlated subquery (`SELECT max(consensus_timestamp) FROM network_stake`) on every single call: [3](#0-2) 

**Existing checks reviewed and shown insufficient:**

The only registered servlet filters are `MetricsFilter` (records byte counts, no limiting) and `ShallowEtagHeaderFilter` (returns 304 only if the *attacker* cooperates by sending `If-None-Match` — an attacker will not): [4](#0-3) [5](#0-4) 

A search across the entire `rest-java` module confirms zero usage of rate-limiting libraries (Bucket4j, Resilience4j, Spring Security rate limiting) and zero `@Cacheable` on `getLatestNetworkStake()`. The `HIGH_VOLUME_THROTTLE` constant referenced in `NetworkController` applies only to the fee-estimation `POST /network/fees` endpoint, not to the stake endpoint: [6](#0-5) 

**Root cause:** The implicit assumption that infrastructure-level rate limiting (e.g., an API gateway or reverse proxy) will always be present is not enforced in code, leaving the service directly exploitable when deployed without such a layer.

### Impact Explanation
Each request causes at least two sequential database operations (the subquery + the main select). Under sustained flood traffic, the JDBC/HikariCP connection pool is exhausted, causing all queries — including those for other endpoints — to queue and time out. This effectively suppresses accurate network stake history from being accessible to legitimate users and can cascade to degrade the entire mirror-node REST API. Severity: **High** (availability impact on a critical read path with no authentication barrier).

### Likelihood Explanation
The endpoint requires zero privileges, zero authentication, and zero special knowledge beyond the public API path `GET /api/v1/network/stake`. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can saturate the DB connection pool from a single host. The attack is trivially repeatable and requires no state. Likelihood: **High**.

### Recommendation
1. **Service-layer caching:** Annotate `getLatestNetworkStake()` with `@Cacheable` (Spring Cache + Caffeine/Redis) with a TTL matching the staking-period update frequency (~24 h). This eliminates per-request DB hits entirely for the common case.
2. **Rate limiting filter:** Add a per-IP rate-limiting servlet filter (e.g., Bucket4j integrated as a `OncePerRequestFilter`) covering `/api/v1/network/stake`, consistent with how `HIGH_VOLUME_THROTTLE` is applied to fee estimation.
3. **Connection pool protection:** Configure HikariCP `connectionTimeout` and `maximumPoolSize` conservatively, and add a DB-level statement timeout so runaway queries self-terminate.
4. **Infrastructure enforcement:** Document and enforce that the service must be deployed behind a rate-limiting reverse proxy (e.g., nginx `limit_req`, AWS WAF) as a hard deployment requirement.

### Proof of Concept
**Preconditions:** Network access to the mirror-node REST Java service; no credentials required.

**Steps:**
```bash
# Flood the endpoint with 500 concurrent connections, no If-None-Match header
wrk -t 10 -c 500 -d 60s http://<mirror-node-host>/api/v1/network/stake

# Simultaneously verify legitimate requests start failing
curl -v http://<mirror-node-host>/api/v1/network/stake
# Expected: connection timeout or 503 once DB pool is exhausted
```

**Trigger:** Each of the 500 concurrent connections causes `NetworkStakeRepository.findLatest()` to execute the correlated subquery. HikariCP's default pool (10 connections) is immediately saturated; queued requests exceed `connectionTimeout` and throw `SQLTransientConnectionException`, surfaced as 500 errors to all users.

**Result:** Legitimate queries for current Hashgraph staking state return errors for the duration of the attack, suppressing accurate network stake history from being accessible.
