### Title
Unauthenticated DoS via 100-Key IN-Clause with Full Historical Timestamp Range in `getHookStorage()`

### Summary
An unprivileged external user can submit a single GET request to `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage` with up to 100 `key=` EQ parameters and a maximally wide timestamp range (`timestamp=gte:0&timestamp=lte:99999999999.999999999`). This forces `findByKeyInAndTimestampBetween()` to execute a `DISTINCT ON (key)` SQL query with a 100-element `IN` clause scanning the entire `hook_storage_change` table history, with no authentication, rate limiting, or IN-clause size guard in place. The attack is trivially repeatable and can be parallelized to sustain elevated DB resource consumption.

### Finding Description

**Code path:**

`HooksController.java` line 108 accepts `key` parameters with `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` where `MAX_REPEATED_QUERY_PARAMETERS = 100` (`Constants.java` line 36). All 100 can be EQ operators. [1](#0-0) 

In `hookStorageChangeRequest()` (lines 170–183), every EQ key is unconditionally appended to `keyFilters`: [2](#0-1) 

With no EQ-count cap, all 100 keys flow into `HookStorageRequest.keys`. `getKeysInRange()` filters against `keyLowerBound`/`keyUpperBound`, but when only EQ keys are supplied, those bounds remain at `MIN_KEY_BYTES`/`MAX_KEY_BYTES` (lines 167–168), so all 100 keys survive the filter. [3](#0-2) 

The timestamp `@Size(max = 2)` allows `gte:0` + `lte:99999999999.999999999`, covering the full historical range. `isHistorical()` returns `true` whenever the timestamp bound is non-empty, routing execution to `findByKeyInAndTimestampBetween`. [4](#0-3) 

The repository query is:

```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key IN (:keys)                          -- up to 100 elements
  AND consensus_timestamp BETWEEN :lower AND :upper  -- full history
``` [5](#0-4) 

`DISTINCT ON (key)` requires PostgreSQL to sort and scan **all** rows matching each key across the full timestamp range before returning one row per key. With 100 keys and an unbounded timestamp range, this is a full-table-scan-equivalent operation per request.

**Root cause:** The `@Size(max = 100)` guard is a ceiling, not a safety limit. It permits the maximum possible IN-clause size. There is no secondary guard tying IN-clause cardinality to timestamp range width, no per-IP/per-user rate limiting, and no authentication requirement on the endpoint.

### Impact Explanation

A single request with 100 EQ keys and a full timestamp range forces a compound expensive DB query. Repeated in parallel (e.g., 10–20 concurrent clients each firing requests in a tight loop), this can sustain >30% elevated CPU and I/O on the database node. Because the endpoint is public and stateless, the attack requires no credentials, no prior state, and no special knowledge beyond a valid `ownerId`/`hookId` (which are enumerable integers).

### Likelihood Explanation

The attack requires zero privileges — it is a plain unauthenticated HTTP GET. The parameters needed (`key=`, `timestamp=`) are documented API parameters. The attacker needs only a valid `ownerId` and `hookId`, both of which are sequential integers discoverable via the `/hooks` listing endpoint. The attack is fully scriptable, parallelizable, and repeatable with no cooldown.

### Recommendation

1. **Enforce a strict low cap on EQ key count** (e.g., ≤ 10) when a wide or unbounded timestamp range is present, or reject requests combining more than N EQ keys with a timestamp span exceeding a configurable threshold.
2. **Add rate limiting** (e.g., via a servlet filter or API gateway) on `/api/v1/accounts/*/hooks/*/storage` keyed by source IP.
3. **Require authentication** for historical/wide-range queries, or add a `HIGH_VOLUME_THROTTLE` guard (the constant already exists at `Constants.java` line 19) to this endpoint.
4. **Add a DB query timeout** at the repository/datasource level so runaway queries are killed before they exhaust resources.

### Proof of Concept

```bash
# Build 100 distinct key= EQ parameters (32-byte hex values)
KEYS=""
for i in $(seq 1 100); do
  KEYS="${KEYS}&key=$(printf '%064x' $i)"
done

# Fire the request — no authentication needed
curl -s "http://<mirror-node-host>/api/v1/accounts/1/hooks/1/storage?\
timestamp=gte:0&timestamp=lte:99999999999.999999999\
${KEYS}&limit=100"

# Repeat in parallel to sustain load
for j in $(seq 1 20); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/1/hooks/1/storage?\
timestamp=gte:0&timestamp=lte:99999999999.999999999${KEYS}&limit=100" &
done
wait
```

Each request triggers `findByKeyInAndTimestampBetween` with a 100-element IN-clause and a full-history timestamp range. Running 20 concurrent clients in a loop will produce sustained elevated DB CPU and I/O, exceeding the 30% resource consumption threshold.
