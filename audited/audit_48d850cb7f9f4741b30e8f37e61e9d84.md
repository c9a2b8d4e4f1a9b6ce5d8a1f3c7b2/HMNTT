### Title
Rogue Mirror Node Returns Port=0 Endpoints, Bypasses All Retries, and Permanently Kills Pinger via `log.Fatalf`

### Summary
When `network=other`, `newClient()` fetches node endpoints from a configurable mirror REST URL. A server returning a `nodesEnvelope` where every `ServiceEndpoint` has `Port=0` causes all entries to be silently skipped, producing an empty network map. The empty-network error path returns `retry=false`, bypassing all configured retries. `main()` then calls `log.Fatalf`, which terminates the process permanently.

### Finding Description

**Exact code path:**

1. `pinger/sdk_client.go:18` — `buildNetworkFromMirrorNodes()` is called only when `cfg.network == "other"`.

2. `pinger/mirror_node_client.go:118` — inside `fetchMirrorNodeNetwork()`, every endpoint with `Port == 0` is silently skipped:
   ```go
   if host == "" || ep.Port == 0 {
       continue
   }
   ``` [1](#0-0) 

3. `pinger/mirror_node_client.go:127-129` — if all endpoints were skipped, the function returns `(nil, false, error)`. The second return value `false` means **no retry**:
   ```go
   if len(network) == 0 {
       return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
   }
   ``` [2](#0-1) 

4. `pinger/mirror_node_client.go:59` — the retry loop in `buildNetworkFromMirrorNodes()` breaks immediately when `retry=false`, regardless of how many retries are configured:
   ```go
   if !retry || attempt == attempts {
       break
   }
   ``` [3](#0-2) 

5. `pinger/main.go:41-44` — `newClient()` failure is handled with `log.Fatalf`, which calls `os.Exit(1)`:
   ```go
   client, err := newClient(cfg)
   if err != nil {
       log.Fatalf("client error: %v", err)
   }
   ``` [4](#0-3) 

**Root cause:** The "no usable endpoints" error is treated as a non-retryable, fatal startup condition. There is no distinction between a transient bad response and a permanent misconfiguration. A single poisoned HTTP response is sufficient to kill the process.

**Failed assumption:** The code assumes that an empty-endpoint response from the mirror REST server is a permanent configuration error rather than a potentially transient or adversarial condition.

### Impact Explanation

The pinger process exits via `log.Fatalf` and does not restart on its own. The `/tmp/ready` readiness file is never written (it is created only after `newClient()` succeeds, line 47), so Kubernetes readiness probes fail. The `/tmp/alive` liveness heartbeat goroutine is also never started in a meaningful way since the process exits before the ticker fires. All network health monitoring and transfer activity ceases until an operator manually intervenes and restarts the pod. This is a complete, persistent denial-of-service of the pinger component. [5](#0-4) 

### Likelihood Explanation

The attack requires the ability to control what the configured mirror REST URL returns. Realistic vectors include:

- **DNS poisoning / BGP hijacking** of the mirror REST hostname if the URL uses plain `http://`. The HTTP client is constructed with no TLS configuration and no certificate pinning (`&http.Client{Timeout: cfg.mirrorNodeClientTimeout}`), so any network-level interception suffices.
- **MITM on an internal cluster HTTP service** (e.g., `http://mirror-rest:5551`) by a compromised pod in the same Kubernetes namespace.
- **Misconfiguration** pointing the pinger at an attacker-controlled endpoint.

Only a single HTTP response needs to be intercepted. The `retry=false` path means the attacker does not need to sustain the attack — one poisoned response at startup is enough to permanently kill the process. [6](#0-5) 

### Recommendation

1. **Treat empty-network as retryable.** Change the return in `fetchMirrorNodeNetwork` from `return nil, false, ...` to `return nil, true, ...` so that all configured retries are exhausted before giving up. [2](#0-1) 

2. **Do not `log.Fatalf` on `newClient` failure.** Instead, retry `newClient` in a loop (with backoff) so a transient or adversarial startup response does not permanently kill the process. [4](#0-3) 

3. **Enforce HTTPS** for the mirror REST URL and validate the server certificate to prevent MITM-based injection of malicious payloads.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other`
- `HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host/`

**Steps:**

1. Stand up an HTTP server at `attacker-controlled-host` that responds to `GET /api/v1/network/nodes` with:
   ```json
   {
     "nodes": [
       {
         "node_account_id": "0.0.3",
         "service_endpoints": [
           {"domain_name": "node.example.com", "ip_address_v4": "", "port": 0}
         ]
       }
     ],
     "links": {"next": null}
   }
   ```

2. Start the pinger (or cause it to restart).

3. `fetchMirrorNodeNetwork` skips the endpoint at line 118 (`Port == 0`).

4. `len(network) == 0` triggers the error at line 128 with `retry=false`.

5. `buildNetworkFromMirrorNodes` returns the error immediately without retrying.

6. `newClient` returns the error to `main`.

7. `log.Fatalf` at line 43 of `main.go` calls `os.Exit(1)`.

8. The pinger process is dead. No readiness file is written. No transfers occur. The attack requires no further action.
