### Title
Silent `BlockType.LATEST` Fallback in `buildCallServiceParameters` Causes EVM Replay to Use Current State Instead of Historical State

### Summary
In `OpcodeServiceImpl.buildCallServiceParameters()`, when `recordFileService.findByTimestamp(consensusTimestamp)` returns empty, the code silently falls back to `BlockType.LATEST`. Because `ContractCallContext.getTimestamp()` returns `Optional.empty()` whenever the block is `BlockType.LATEST`, the timestamp explicitly set by `ContractDebugService.processOpcodeCall()` is effectively ignored, and all state reads (accounts, storage, tokens, NFTs) use the current/latest database state instead of the historical state at the transaction's consensus timestamp. Any unprivileged user can trigger this by querying a transaction whose record file has been pruned or is otherwise absent from the mirror node database.

### Finding Description

**Exact code path:**

`OpcodeServiceImpl.java`, lines 158–161:
```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
``` [1](#0-0) 

This `blockType` is embedded in `ContractDebugParameters.block` and passed to `ContractDebugService.processOpcodeCall()`:
```java
ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1));  // line 51
// ...
final var ethCallTxnResult = callContract(params, ctx);              // line 56
``` [2](#0-1) 

Inside `callContract()`, `ctx.setCallServiceParameters(params)` is called with `params.getBlock() == BlockType.LATEST`: [3](#0-2) 

`ContractCallContext.getTimestamp()` is gated on the block type stored in `callServiceParameters`. The test `testGetTimestampNonHistorical` confirms: when `block == BlockType.LATEST`, `getTimestamp()` returns `Optional.empty()` regardless of what was set via `setTimestamp()`: [4](#0-3) 

Every state-reading component (`AccountReadableKVState`, `NftReadableKVState`, `ContractStorageReadableKVState`, `MidnightRatesSingleton`, etc.) branches on `ContractCallContext.get().getTimestamp()`: when it is empty, they query the latest DB row instead of the historical one. The `setTimestamp(consensusTimestamp - 1)` call in `ContractDebugService` is therefore completely neutralised.

**Root cause:** The failed assumption is that falling back to `BlockType.LATEST` is a safe default. It is not — it silently disables the entire historical-state mechanism for the EVM replay, producing an opcode trace against current state.

**Why the existing check is insufficient:** The only guard before `buildCallServiceParameters` is `contractResultRepository.findById(consensusTimestamp)`, which only verifies the contract result row exists. It does not verify that the corresponding record file exists. On a mirror node that prunes record files while retaining contract results (a common operational configuration), `findByTimestamp()` will return empty for any pruned block, triggering the fallback silently with no error or warning. [5](#0-4) 

### Impact Explanation
The opcode trace returned to the caller misrepresents the original smart contract execution: storage slots, account balances, token states, and EVM version selection all reflect the current chain state rather than the state at the time the transaction was originally executed. This corrupts forensic analysis, smart contract debugging, and any tooling that relies on `eth_debugTraceTransaction`-equivalent output for historical transactions. The EVM version selected via `EvmProperties.getSemanticEvmVersion()` also depends on the block number from the record file; with `BlockType.LATEST` the wrong EVM version may be used, further distorting the trace. [6](#0-5) 

### Likelihood Explanation
No privileges are required. The endpoint is open to any caller who can supply a valid transaction ID or hash: [7](#0-6) 

The precondition — a missing record file for an otherwise valid transaction — is a normal operational state on any mirror node with record-file pruning enabled. An attacker can enumerate known historical transaction hashes and query the opcodes endpoint; any transaction whose record file has been pruned will silently return a current-state trace. The behaviour is deterministic and repeatable.

### Recommendation
Replace the silent `.orElse(BlockType.LATEST)` fallback with an explicit error:
```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
```
This ensures that if the record file required for a faithful historical replay is absent, the request fails with a clear error rather than silently replaying against current state. [1](#0-0) 

### Proof of Concept
1. Identify a historical contract transaction whose `consensusTimestamp` falls within a pruned record-file range (i.e., the `record_file` table has no row with `consensus_end >= consensusTimestamp`, but the `contract_result` row still exists).
2. Send: `GET /api/v1/contracts/results/{transactionHash}/opcodes` with `Accept-Encoding: gzip`.
3. Observe that the response is `200 OK` with a full opcode trace — no error is raised.
4. Compare the returned storage/balance values in the trace against the known historical state at the transaction's block. They will reflect current state.
5. Confirm by temporarily restoring the record file row: the trace will change, proving the fallback was active in step 3.
