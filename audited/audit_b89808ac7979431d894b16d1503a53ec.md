### Title
Correlated Subquery Amplification in `selectTransactionsInTimestampRange` Enables Unauthenticated DB CPU Exhaustion

### Summary
The `selectTransactionsInTimestampRange` SQL constant in `rosetta/app/persistence/transaction.go` embeds two correlated subqueries (against `crypto_transfer` and `staking_reward_transfer`) that execute once per row returned by the outer `transaction` scan. Because `FindBetween` fetches all transactions in a block's timestamp range in batches of 2000 before any response-level cap is applied, an unprivileged caller can trigger O(2N) additional DB operations per request by targeting a block with N transactions. The `maxTransactionsInBlock` guard truncates only the HTTP response, not the database work.

### Finding Description
**Exact code path:**

`rosetta/app/persistence/transaction.go`, lines 32–57 — the `selectTransactionsInTimestampRange` constant:

```sql
select ...,
  coalesce((
    select json_agg(...) from crypto_transfer
    where consensus_timestamp = t.consensus_timestamp ...
  ), '[]') as crypto_transfers,          -- subquery 1: per row
  coalesce((
    select json_agg(...) from staking_reward_transfer
    where consensus_timestamp = t.consensus_timestamp
  ), '[]') as staking_reward_payouts     -- subquery 2: per row
from transaction t
where consensus_timestamp >= @start and consensus_timestamp <= @end
```

`rosetta/app/persistence/transaction.go`, lines 124–147 — `FindBetween` loops in batches of `batchSize = 2000` until the full range is exhausted:

```go
for start <= end {
    err := db.Raw(selectTransactionsInTimestampRangeOrdered, ...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    if len(transactionsBatch) < batchSize { break }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

`rosetta/app/services/block_service.go`, lines 56–67 — `maxTransactionsInBlock` is applied **after** `FindBetween` returns:

```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
    return nil, err
}
// truncation happens here, AFTER all DB work is done
if len(block.Transactions) > s.maxTransactionsInBlock { ... }
```

**Root cause:** The correlated subqueries are evaluated by PostgreSQL once per outer row. For a block with N transactions, the DB executes `ceil(N/2000)` outer scans plus `N × 2` correlated subquery executions (each involving an index lookup + `json_agg` + `ORDER BY` on potentially large tables). The application-level cap (`maxTransactionsInBlock`) is a post-processing slice of the already-materialized Go slice — it does not reduce DB work.

**Failed assumption:** The design assumes `maxTransactionsInBlock` limits resource consumption. It limits only response payload size; the full DB query cost is incurred unconditionally.

### Impact Explanation
On Hedera mainnet, blocks routinely contain thousands of transactions. A single `/block` request targeting such a block causes the DB to execute up to `N × 2` correlated subqueries with `json_agg` + sort per subquery. For N = 10,000 transactions across 5 batches, this is ~20,000 additional DB operations per request. Repeated at even modest rates (e.g., 5 concurrent requests per the Traefik `inFlightReq` default), this can saturate DB CPU well above 30% above baseline without any brute-force volume. The Rosetta API is unauthenticated, so no credentials are required.

### Likelihood Explanation
Block indices and hashes are public. An attacker can trivially identify high-transaction blocks via the mirror node REST API or block explorers, then repeatedly POST to `/block` (or `/block/transaction`) with those identifiers. The Traefik middleware (`inFlightReq: 5`, `rateLimit: average: 10`) is an optional Helm chart deployment artifact — it is not enforced at the application layer and can be bypassed with multiple source IPs or if the chart is deployed without the middleware. No authentication, no API key, and no application-level query cost guard exist.

### Recommendation
1. **Rewrite the query using JOINs or CTEs** instead of correlated subqueries, so `crypto_transfer` and `staking_reward_transfer` are scanned once per batch rather than once per row:
   ```sql
   WITH ct AS (SELECT consensus_timestamp, json_agg(...) FROM crypto_transfer
               WHERE consensus_timestamp BETWEEN @start AND @end GROUP BY 1),
        srt AS (SELECT consensus_timestamp, json_agg(...) FROM staking_reward_transfer
                WHERE consensus_timestamp BETWEEN @start AND @end GROUP BY 1)
   SELECT t.*, COALESCE(ct.transfers, '[]'), COALESCE(srt.payouts, '[]')
   FROM transaction t
   LEFT JOIN ct USING (consensus_timestamp)
   LEFT JOIN srt USING (consensus_timestamp)
   WHERE t.consensus_timestamp BETWEEN @start AND @end
   ```
2. **Enforce `maxTransactionsInBlock` at the DB level** by adding `LIMIT` to the outer query equal to `maxTransactionsInBlock`, not just as a Go-side slice.
3. **Add a DB `statement_timeout`** in `rosetta/app/config/types.go` (`Db.StatementTimeout`) and enforce it unconditionally, not as an optional config.
4. **Enforce rate limiting at the application layer**, not only via optional Traefik middleware.

### Proof of Concept
1. Identify a Hedera mainnet block with a large transaction count (e.g., block index with thousands of transactions) via `GET /api/v1/blocks?order=desc&limit=1` on the mirror node REST API.
2. Extract `consensus_start` and `consensus_end` from the block record.
3. Send repeated unauthenticated POST requests to the Rosetta `/block` endpoint:
   ```json
   POST /block
   {
     "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
     "block_identifier": {"index": <high_tx_block_index>}
   }
   ```
4. Monitor PostgreSQL `pg_stat_activity` and CPU metrics. Each request will show `N × 2` correlated subquery executions in `EXPLAIN ANALYZE` output, with DB CPU rising proportionally to transaction count per block.
