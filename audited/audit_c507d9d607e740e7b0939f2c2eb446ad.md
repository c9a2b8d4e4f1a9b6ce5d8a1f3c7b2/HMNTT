### Title
Connection Pool Exhaustion via Concurrent Requests with Maximum Repeated `spender.id` Parameters in `getAccountCryptoAllowances`

### Summary
The `getAccountCryptoAllowances` handler in `rest/controllers/cryptoAllowanceController.js` accepts up to `maxRepeatedQueryParameters` (default: 100) repeated `spender.id` query parameters per request, each triggering a database query. Because the REST API has no per-IP concurrency or rate limiting, an unauthenticated attacker can flood the endpoint with concurrent requests to exhaust the default 10-connection database pool, causing all subsequent queries to stall for up to the full `connectionTimeout` (default: 20 seconds).

### Finding Description

**Code path:**

`getAccountCryptoAllowances` (line 76–98) calls `utils.buildAndValidateFilters` (line 78), which internally calls `buildFilters` in `rest/utils.js` (lines 1234–1259). The `buildFilters` function enforces a cap via `isRepeatedQueryParameterValidLength`, rejecting arrays longer than `config.query.maxRepeatedQueryParameters` (default: **100**). Requests at or below this cap pass through and each `spender.id` value is appended to a SQL `IN (...)` clause via `updateConditionsAndParamsWithInValues` (lines 39–46).

The resulting query is executed via `CryptoAllowanceService.getAccountCryptoAllowances` (line 80), which calls `super.getRows(query, params)` in `rest/service/cryptoAllowanceService.js` (line 15), acquiring a connection from the pool defined in `rest/dbpool.js` (lines 7–16):

```js
max: config.db.pool.maxConnections,  // default: 10
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default: 20000ms
statement_timeout: config.db.pool.statementTimeout,  // default: 20000ms
```

**Root cause:** The REST API (`rest/server.js`) applies no per-IP rate limiting or concurrency limiting middleware. The only throttling mechanisms in the codebase (bucket4j `ThrottleConfiguration`, `ThrottleManagerImpl`) are scoped exclusively to the `web3` Java service, not the Node.js REST API. The `maxRepeatedQueryParameters` check limits per-request parameter count but does not bound concurrent request volume.

**Exploit flow:**
1. Attacker sends ≥10 concurrent requests, each with 100 `spender.id=eq:N` parameters.
2. Each request passes `buildFilters` validation (exactly at the 100-param cap).
3. Each request acquires a pool connection for the `EntityService.getEncodedId` lookup (line 77) and then another for the `crypto_allowance` query (line 80).
4. With `maxConnections=10`, 10 concurrent requests saturate the pool.
5. All subsequent legitimate requests block waiting for a connection for up to `connectionTimeout` (20 seconds).

**Why existing checks fail:**
- `maxRepeatedQueryParameters=100` prevents oversized single requests but does not limit concurrent request volume.
- `statementTimeout=20000ms` eventually releases connections but allows sustained saturation if the attacker maintains request throughput.
- `throwOnLimitExceeded: true` in `requestQueryParser` (line 19 of `requestHandler.js`) only rejects requests exceeding 100 params; requests at exactly 100 are accepted.
- No Traefik `inFlightReq` or `rateLimit` middleware is configured for the REST API service (unlike the Rosetta API which has `inFlightReq: amount: 5` and `rateLimit: average: 10`).

### Impact Explanation
All database-backed REST API endpoints share the same 10-connection pool. Exhausting it stalls every concurrent query — not just crypto allowance queries — for up to 20 seconds per request. This constitutes a full REST API denial-of-service achievable by a single unauthenticated attacker with modest HTTP throughput. The 20-second stall exceeds 500% of Hedera's ~3–5 second consensus round time, degrading mirror node query availability for all consumers.

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. The attacker needs only HTTP access and the ability to issue ~10–20 concurrent GET requests per second. This is trivially achievable from a single machine or a small botnet. The attack is repeatable and stateless — each request is independent, requiring no session or prior knowledge beyond a valid account ID (which is publicly enumerable from the same API).

### Recommendation
1. **Add per-IP concurrency limiting** to the REST API Express middleware (e.g., `express-rate-limit` or Traefik `inFlightReq`) to cap simultaneous in-flight requests per source IP.
2. **Increase the default pool size** or deploy PgBouncer in front of the REST API (the Helm chart already configures `max_user_connections: 250` for `mirror_rest` via PgBouncer — ensure this is enforced in all deployment configurations).
3. **Add a Traefik `inFlightReq` middleware** for the REST API service, mirroring the Rosetta API configuration (`amount: 5` per IP).
4. **Reduce `maxRepeatedQueryParameters`** for `spender.id` specifically, or add a dedicated per-parameter-key repetition cap in `extractCryptoAllowancesQuery`.
