### Title
Error Counter Reset on Intermediate Response Prevents Block Node Deactivation

### Summary
In `BlockNode.streamBlocks()`, the `errors` counter is reset to zero (`errors.set(0)`) on every successful response within the streaming loop, including intermediate `BLOCK_ITEMS` and `END_OF_BLOCK` responses. A malicious block node operator can exploit this by sending one valid intermediate response before each error response, keeping the `errors` counter perpetually at 1 and never reaching the `maxSubscribeAttempts` threshold (default: 3) required to mark the node inactive.

### Finding Description

**Exact code location:**
`importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, `streamBlocks()`, lines 138–158; `onError()`, lines 196–207.

**Root cause:**

The `errors.set(0)` reset at line 157 executes after every non-throwing iteration of the while loop — including `BLOCK_ITEMS` and `END_OF_BLOCK` responses — not only after a fully successful stream completion:

```java
// lines 138–158
while (!serverSuccess && (response = grpcCall.read(...)) != null) {
    switch (response.getResponseCase()) {
        case BLOCK_ITEMS -> assembler.onBlockItemSet(response.getBlockItems());
        case END_OF_BLOCK -> assembler.onEndOfBlock(response.getEndOfBlock());
        case STATUS -> {
            if (status == SubscribeStreamResponse.Code.SUCCESS) {
                serverSuccess = true;
                break;
            }
            throw new BlockStreamException(...);
        }
        ...
    }
    errors.set(0);  // ← resets on ANY non-throwing response, including BLOCK_ITEMS
}
```

`onError()` increments `errors` only when an exception is thrown:

```java
// lines 196–207
private void onError() {
    errorsMetric.increment();
    if (errors.incrementAndGet() >= streamProperties.getMaxSubscribeAttempts()) {
        active = false;
        ...
    }
}
```

**Exploit flow (with `maxSubscribeAttempts = 3`):**

| Attempt | Attacker sends | `errors` after reset | `errors` after `onError()` | Node active? |
|---------|---------------|----------------------|---------------------------|--------------|
| 1 | BLOCK_ITEMS → NOT_AVAILABLE | 0 | 1 | yes |
| 2 | BLOCK_ITEMS → NOT_AVAILABLE | 0 | 1 | yes |
| 3 | BLOCK_ITEMS → NOT_AVAILABLE | 0 | 1 | yes |
| N | BLOCK_ITEMS → NOT_AVAILABLE | 0 | 1 | yes (forever) |

The counter never reaches 3. The node is never deactivated.

**Why existing checks fail:**

The `maxSubscribeAttempts` guard in `onError()` (line 198) is designed to count *consecutive* failures, but the reset at line 157 fires on any valid intermediate response, breaking the "consecutive" invariant. The `StreamProperties.maxSubscribeAttempts` minimum of 1 (enforced by `@Min(1)`) provides no protection since the counter is always reset before it can accumulate. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

A malicious block node operator can prevent their node from ever being marked inactive, causing the mirror node to repeatedly attempt (and fail) streaming from that node. Each block requires at least one failed attempt, introducing persistent latency in block ingestion. If the malicious node is the highest-priority or only configured node, `BlockNodeSubscriber` always selects it (since `getNode()` only skips inactive nodes), and the `CompositeBlockSource.SourceHealth` accumulates errors at the subscriber level, potentially triggering a fallback to file-based ingestion or a complete stall if no fallback is configured. The impact is classified as griefing: no economic damage, but sustained disruption to block processing throughput. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

Any operator of a configured block node can execute this attack without any special privileges beyond being listed in the mirror node's block node configuration. The attack requires only that the node's gRPC server sends one valid `BLOCK_ITEMS` response before each error status response — a trivial modification to a standard block node implementation. The pattern is repeatable indefinitely and requires no coordination or timing precision. [6](#0-5) 

### Recommendation

Move `errors.set(0)` outside the while loop so it only executes when the stream completes without throwing an exception (i.e., after a fully successful stream):

```java
// Before (line 157 — inside the while loop):
errors.set(0);

// After — move to after the while loop, still inside the try block:
while (!serverSuccess && (response = grpcCall.read(...)) != null) {
    switch (...) { ... }
    // remove errors.set(0) from here
}
errors.set(0);  // only reached if no exception was thrown
```

This ensures the counter only resets when an entire streaming session completes successfully, preserving the "consecutive failures" semantics that `maxSubscribeAttempts` is designed to enforce. [7](#0-6) 

### Proof of Concept

**Preconditions:**
- Mirror node configured with a single block node pointing to an attacker-controlled server.
- `maxSubscribeAttempts = 3` (default).

**Steps:**

1. Attacker's gRPC server, for every `subscribeBlockStream` call, responds with:
   - One `BLOCK_ITEMS` response (any syntactically valid `BlockItemSet` with a `BLOCK_HEADER` item).
   - Immediately followed by a `STATUS` response with code `NOT_AVAILABLE`.

2. Mirror node calls `streamBlocks()`:
   - Receives `BLOCK_ITEMS` → `errors.set(0)` → `errors = 0`.
   - Receives `NOT_AVAILABLE` → throws `BlockStreamException` → `onError()` → `errors = 1` (< 3).
   - Node remains active.

3. Mirror node retries (next scheduled `get()` call). Same sequence. `errors` resets to 0, then goes to 1. Node remains active.

4. Repeat indefinitely. `errors` oscillates between 0 and 1. The node is never deactivated. Every block streaming attempt fails after receiving the decoy `BLOCK_ITEMS` response.

**Verification:** Add a counter to `onError()` calls and observe it increments on every attempt while `node.isActive()` remains `true` indefinitely — contradicting the behavior demonstrated in `BlockNodeTest.onError()` which only works because that test sends errors with *no* preceding valid response. [8](#0-7)

        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
