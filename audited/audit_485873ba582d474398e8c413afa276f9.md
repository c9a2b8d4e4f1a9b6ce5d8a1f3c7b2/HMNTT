### Title
Gas Throttle Fully Bypassed via OOG-Failing Transactions with Default `gasLimitRefundPercent=100`

### Summary
`ThrottleProperties.gasLimitRefundPercent` defaults to `100`, and `restoreGasToBucket()` unconditionally restores `gasLimitRefundPercent * gasLimit / 100` tokens when a transaction fails with `result.gasUsed() == gasLimit` (the OOG case). With the default value, the net gas tokens consumed from `gasLimitBucket` for every OOG-failing call is exactly zero. An unprivileged attacker can craft `ContractCallRequest`s targeting any always-OOG contract to completely neutralize the gas-per-second throttle, leaving only the much weaker requests-per-second limit as protection against expensive EVM simulation DoS.

### Finding Description

**Exact code path:**

`ThrottleProperties.java` line 22 — default value:
```java
private float gasLimitRefundPercent = 100;
```

`ContractCallService.java` lines 140–152 — restore logic:
```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline =
        (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f); // = gasLimit when 100%
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline);   // restores full gasLimit
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

`ThrottleManagerImpl.java` lines 37–42 — consume on entry:
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
```

`ThrottleManagerImpl.java` lines 59–64 — restore after execution:
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) { gasLimitBucket.addTokens(tokens); }
}
```

**Root cause:** Both `throttle()` and `restore()` pass their argument through the same `scaleGas()` function. When `gasLimitRefundPercent=100` and the OOG branch fires (`!isSuccessful() && gasUsed==gasLimit`), `restore(gasLimit)` is called, which adds back exactly as many scaled tokens as were consumed. Net depletion of `gasLimitBucket` = 0.

**Failed assumption:** The design comment says "If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent." The assumption is that operators will tune `gasLimitRefundPercent` below 100 for production. The code ships with 100 as the default, making the gas throttle a no-op for the OOG path out of the box.

**Exploit flow:**
1. Attacker identifies (or deploys) any contract that always runs out of gas — e.g., an infinite loop, a storage-heavy operation, or simply calls `eth_call` against a non-existent address with a tiny gas limit so the EVM OOGs immediately.
2. Attacker sends `ContractCallRequest` with `gas = G` (any value > 10 000 to pass `scaleGas`).
3. `throttle()` consumes `scaleGas(G)` tokens from `gasLimitBucket`.
4. EVM executes, runs OOG → `result.isSuccessful()=false`, `result.gasUsed()=G`.
5. `restoreGasToBucket()` fires the first branch: `restore(G * 100/100) = restore(G)`.
6. `gasLimitBucket` is back to its pre-call level. Net cost = 0.
7. Attacker repeats at the full `requestsPerSecond` rate (default 500/s), each carrying up to 15 M gas of EVM work, with no gas-bucket depletion ever occurring.

### Impact Explanation
The `gasLimitBucket` is the primary defense against expensive EVM simulation DoS on the mirror node. With the default configuration it is completely inert for OOG-failing calls. An attacker can sustain 500 req/s × 15 M gas = 7.5 B gas/s of real EVM computation on the mirror node indefinitely, limited only by the coarser request-rate bucket. This can exhaust CPU/memory on the mirror node, causing degraded or unavailable service for all users — a high-severity availability impact.

### Likelihood Explanation
No privileges are required. Any caller of the public `eth_call` / `eth_estimateGas` endpoint can trigger this. Causing OOG is trivial: set `gas` to a small value (e.g., 22 000) against any contract that costs more than that, or target a known infinite-loop contract. The attack is fully repeatable and scriptable. The default configuration is the vulnerable one, so no misconfiguration by the operator is needed.

### Recommendation
1. **Lower the default** of `gasLimitRefundPercent` to a value well below 100 (e.g., 0 or 10) so that OOG failures still consume a meaningful fraction of the declared gas limit from the bucket.
2. **Alternatively**, cap the restore amount for the OOG branch independently from the success/revert branch — e.g., restore at most `gasLimit - minimumExecutionCost` rather than the full `gasLimit`.
3. **Document** that `gasLimitRefundPercent=100` disables gas-throttle protection for failed transactions, so operators understand the trade-off.
