### Title
Global Single-Token Opcode Rate Limit Enables Unauthenticated Low-and-Slow DoS via EVM Re-Execution Thread Exhaustion

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint uses a single global `opcodeRateLimitBucket` (default: 1 req/sec) with no per-IP or per-user partitioning. Any unauthenticated attacker can permanently monopolize the entire global token budget by sending exactly 1 request per second, denying all legitimate users access. Because the rate limiter controls only admission rate (not concurrent execution count), and each EVM re-execution can run for up to 10 seconds, the attacker can accumulate up to 10 concurrent EVM re-execution threads simultaneously, exhausting the server's thread pool.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` at line 61 calls `throttleManager.throttleOpcodeRequest()`: [1](#0-0) 

`ThrottleManagerImpl.throttleOpcodeRequest()` performs a single global `tryConsume(1)` with no IP/user context: [2](#0-1) 

The `opcodeRateLimitBucket` bean is a single application-scoped instance with `capacity=1`, `refillGreedy(1, Duration.ofSeconds(1))`, and no `SynchronizationStrategy.SYNCHRONIZED` (unlike `gasLimitBucket`): [3](#0-2) 

The default `opcodeRequestsPerSecond = 1` is intentionally low because the endpoint is acknowledged as heavy: [4](#0-3) 

The `requestTimeout` is 10 seconds per request: [5](#0-4) 

**Root cause:** The failed assumption is that 1 req/sec is a safe global ceiling. In reality, because the bucket is not partitioned by IP/user, a single attacker can consume 100% of the global budget indefinitely. Additionally, the rate limiter governs only admission rate, not concurrent execution count. With a 10-second EVM execution time and 1-token-per-second refill, an attacker accumulates up to 10 concurrent EVM re-executions before the first one completes.

**Why existing checks fail:**
- `throttleOpcodeRequest()` has no IP extraction, no session binding, no per-caller bucket — it is purely global.
- The `gasLimitBucket` (which does per-request gas accounting) is **not consulted** for opcode requests; only the opcode bucket is checked.
- There is no semaphore or concurrency cap on `contractDebugService.processOpcodeCall()`.
- The `requestTimeout` of 10s is a per-request deadline, not a concurrency guard.
- No authentication is required; the endpoint is publicly accessible. [6](#0-5) 

### Impact Explanation
A single unauthenticated attacker achieves two simultaneous effects:

1. **Complete monopolization of the endpoint for all other users.** Since the bucket is global and holds exactly 1 token/sec, the attacker's steady 1 req/sec stream keeps the bucket perpetually empty. Every legitimate user receives HTTP 429 for the duration of the attack.

2. **Thread pool exhaustion.** Each admitted request triggers a full EVM re-execution (acknowledged in the Javadoc as potentially taking "a significant amount of time"). With `requestTimeout = 10s` and 1 new request admitted per second, up to 10 concurrent EVM threads accumulate. On a default Spring Boot embedded Tomcat (200 worker threads), sustained attack from a single source can consume a meaningful fraction of the thread pool, degrading the entire web3 service — not just the opcodes endpoint.

Severity: **High**. The endpoint is publicly reachable, requires zero privileges, and the attack is trivially sustainable indefinitely.

### Likelihood Explanation
The attack requires:
- Knowledge that the endpoint exists (documented in the OpenAPI spec).
- A valid transaction ID or hash (obtainable from any public Hedera mirror REST API).
- A single HTTP client capable of sending 1 request per second with `Accept-Encoding: gzip`.

No credentials, no on-chain funds, no special tooling. Any developer or script kiddie with access to the public API can execute this. The attack is fully repeatable and self-sustaining. [7](#0-6) 

### Recommendation

1. **Per-IP (or per-caller) rate limiting:** Replace or augment the global `opcodeRateLimitBucket` with a per-IP bucket (e.g., using Bucket4j's `ProxyManager` backed by a `CaffeineProxyManager`). Each IP should get its own 1 req/sec allowance rather than sharing a single global token.

2. **Concurrency semaphore:** Add a `Semaphore` (e.g., `new Semaphore(N)`) around `opcodeService.processOpcodeCall()` to cap the number of simultaneous EVM re-executions regardless of admission rate. A value of 2–4 concurrent executions is reasonable.

3. **Gas-aware throttling for opcodes:** Extend `throttleOpcodeRequest()` to also consume from the `gasLimitBucket` based on the historical gas used by the target transaction, consistent with how `throttle(ContractCallRequest)` works for other endpoints. [8](#0-7) 

### Proof of Concept

**Preconditions:**
- Mirror node is running with `hiero.mirror.web3.opcode.tracer.enabled=true`.
- Attacker has one valid transaction hash from the public mirror REST API (e.g., `GET /api/v1/contracts/results?limit=1`).

**Steps:**

```bash
# Step 1: Obtain a valid transaction hash (no auth required)
TX_HASH=$(curl -s "https://<mirror-node>/api/v1/contracts/results?limit=1" \
  | jq -r '.results[0].hash')

# Step 2: Send exactly 1 request per second indefinitely (stays within rate limit)
while true; do
  curl -s -o /dev/null \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" &
  sleep 1
done
```

**Result:**
- The attacker's loop consumes the single global token every second.
- All concurrent legitimate requests to the same endpoint receive HTTP 429.
- Background EVM re-executions accumulate (up to ~10 concurrent), consuming CPU and thread pool slots.
- The attack sustains indefinitely with no cost to the attacker.

        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
