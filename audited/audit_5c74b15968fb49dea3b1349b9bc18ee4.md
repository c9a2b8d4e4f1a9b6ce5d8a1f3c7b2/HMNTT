### Title
Unvalidated `domain_name` Values from Mirror REST Response Enable DNS Exhaustion and Complete Pinger Failure

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` accepts any non-empty string from the `domain_name` field of `service_endpoints` without validating that it is a resolvable hostname or a valid IP address. An attacker who can control or intercept the plain-HTTP mirror REST response can inject unresolvable hostnames, causing the Hiero SDK client to be initialized with poisoned node addresses. Every subsequent transfer attempt will fail with DNS resolution timeouts, rendering the pinger permanently non-functional for the lifetime of the process.

### Finding Description
In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()` (lines 74–132) fetches the mirror node's `/api/v1/network/nodes` endpoint and builds a `map[string]hiero.AccountID` from the returned `service_endpoints`. The only guards applied to each endpoint are:

```go
// lines 114–119
host := strings.TrimSpace(ep.DomainName)
if host == "" {
    host = strings.TrimSpace(ep.IPAddressV4)
}
if host == "" || ep.Port == 0 {
    continue
}
```

No DNS pre-resolution, no IP-address format check, and no allowlist/denylist is applied. Any non-empty string passes through and is stored via `net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))` (line 122). The resulting map is handed directly to `hiero.ClientForNetwork(netmap)` in `sdk_client.go` line 22, which is called exactly once at startup (main.go line 41). The SDK then uses these addresses for all gRPC connections.

The mirror REST URL is operator-configured via `HIERO_MIRROR_PINGER_REST` / `-mirror-rest`. The documented example value is `http://mirror-rest:5551` — plain HTTP with no TLS. A network-adjacent attacker (ARP spoofing, DNS poisoning, or rogue DHCP on the same segment) can MITM this unencrypted HTTP request and return a crafted JSON body such as:

```json
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{"domain_name": "nonexistent.invalid", "port": 50211}]
  }],
  "links": {"next": null}
}
```

`fetchMirrorNodeNetwork()` returns this map without error (it is non-empty, satisfying line 127). The SDK client is initialized with `nonexistent.invalid:50211`. Every call to `cryptoTransfer.Execute(client)` in `submitWithRetry()` (transfer.go line 33) triggers a gRPC dial to that address, which triggers a DNS lookup that always fails. With default config (`maxRetries=10`, `interval=1s`), this produces ≥11 failing DNS queries per second indefinitely.

### Impact Explanation
The pinger process becomes permanently non-functional from startup: no transfers succeed, the `/tmp/ready` readiness file is written (main.go line 47) but the liveness heartbeat at `/tmp/alive` continues, masking the failure from some probes. DNS infrastructure receives a sustained flood of NXDOMAIN queries for the injected hostname. Because the client is built once and never rebuilt, recovery requires a process restart with a clean mirror REST response. Classified as medium-severity griefing: no on-chain economic damage, but complete loss of the pinger's monitoring function and measurable DNS amplification.

### Likelihood Explanation
The attack requires network-level access to intercept the HTTP connection between the pinger and the mirror REST endpoint. This is realistic in cloud/container environments where the mirror REST service is on the same virtual network, in Kubernetes clusters without mTLS (e.g., no service mesh), or in any environment where the attacker has compromised a co-located workload. The plain-HTTP default makes this straightforward; no credentials or application-layer privileges are needed. The attack is repeatable: a new process restart re-fetches the poisoned response.

### Recommendation
1. **Validate each host before accepting it**: attempt `net.LookupHost(host)` (or require it to parse as a valid IP via `net.ParseIP`) before adding it to the network map. Reject entries that fail.
2. **Enforce HTTPS** for the mirror REST URL: reject `cfg.mirrorRest` values that do not use the `https` scheme, or pin a TLS certificate.
3. **Allowlist node account IDs**: cross-check returned `node_account_id` values against a locally configured set of known-good node IDs before trusting their endpoints.
4. **Periodic refresh**: rebuild the client periodically so that a transient poisoning does not persist for the entire process lifetime.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<mirror-host>:5551`.
2. On a host with network access to intercept traffic between the pinger and `<mirror-host>`, use a tool such as `mitmproxy` or `arpspoof` to intercept the HTTP GET to `/api/v1/network/nodes`.
3. Return the following response body:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"nonexistent.invalid","port":50211}]}],"links":{"next":null}}
   ```
4. Start (or restart) the pinger process.
5. Observe: `fetchMirrorNodeNetwork()` returns successfully (non-empty map), the SDK client is initialized, and every subsequent transfer attempt logs a DNS resolution failure for `nonexistent.invalid`.
