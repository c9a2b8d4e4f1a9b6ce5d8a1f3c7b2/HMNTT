### Title
Unbounded Multiple `timestamp=ne` Filters in `extractContractResultsByIdQuery` Enable Database DoS via Index-Defeating NOT-EQUAL Conditions

### Summary
The `extractContractResultsByIdQuery` function in `rest/controllers/contractController.js` accepts an unbounded number of `timestamp=ne:X` query parameters (up to `maxRepeatedQueryParameters`, default ~100) and converts each into a separate `cr.consensus_timestamp != $N` SQL condition. For the `/contracts/:contractId/results` endpoint, these `ne` filters bypass the `optimizeTimestampFilters` range-bounding step entirely, so no lower/upper range is added. The resulting query forces PostgreSQL to perform a full index scan of all rows for the given contract, evaluating up to 100 `!=` filter conditions per row, with no authentication required.

### Finding Description

**Exact code path:**

1. **Filter collection** — `extractContractResultsByIdQuery` (lines 461–463) pushes every `timestamp` filter into `timestampFilters` with no `ne`-specific count check: [1](#0-0) 

2. **Bypass of range optimization for `/contracts/:contractId/results`** — line 504–505 only calls `optimizeTimestampFilters` when `contractId === undefined`. When a contractId is present (the `/contracts/:contractId/results` path), the raw `timestampFilters` — including all `ne` entries — are used directly: [2](#0-1) 

3. **Each `ne` filter becomes a separate `!= $N` SQL condition** — lines 510–518 call `updateConditionsAndParamsWithInValues` for every filter in `optimizedTimestampFilters` (or raw `timestampFilters`): [3](#0-2) 

4. **`ne` operator path in `BaseController`** — when `filter.operator !== opsMap.eq`, the value is pushed to params and a raw `${fullName}${filter.operator}$${position}` string is appended to conditions (line 24–26). For `ne`, `filter.operator` is ` != `, producing `cr.consensus_timestamp != $N`: [4](#0-3) 

5. **Only guard is `maxRepeatedQueryParameters`** — `buildFilters` in `utils.js` checks `isRepeatedQueryParameterValidLength(values)` per key (lines 1241–1248), which enforces a single global cap (default ~100) on all repeated values for a key, with no `ne`-specific sub-limit: [5](#0-4) 

6. **`parseTimestampFilters` deduplicates but does not cap `ne` count** — `neValues` is a `Set` (line 1605), so duplicate values are collapsed, but up to `maxRepeatedQueryParameters` distinct values pass through: [6](#0-5) 

**Root cause:** The `ne` operator is intentionally supported for `timestamp` on contract results (confirmed by the test spec at line 51 of `timestamp.json`), but there is no guard that prevents an unbounded number of distinct `ne` values from being translated into individual `!= $N` SQL predicates, and for the `/contracts/:contractId/results` path, no range bounds are injected to limit the scan. [7](#0-6) 

**Resulting SQL (for `/contracts/:contractId/results?timestamp=ne:T1&timestamp=ne:T2&...&timestamp=ne:T100`):**
```sql
WHERE cr.contract_id = $1
  AND cr.consensus_timestamp != $2
  AND cr.consensus_timestamp != $3
  ...
  AND cr.consensus_timestamp != $101
  AND cr.transaction_nonce = 0
ORDER BY cr.consensus_timestamp DESC
LIMIT $102
```
PostgreSQL can use the index on `contract_id` to locate the contract's rows, but must then evaluate 100 `!=` filter conditions against every row for that contract. For a high-volume contract (e.g., a DEX with millions of results), this is a full sequential scan of that contract's partition.

**Why existing checks fail:**
- `contractResultsFilterValidityChecks` explicitly permits `ne` for `timestamp` on this endpoint (evidenced by the passing test spec).
- `maxRepeatedQueryParameters` is a global cap (~100), not a `ne`-specific limit.
- `parseTimestampFilters` deduplication only removes exact duplicates; 100 distinct timestamps all pass.
- The `optimizeTimestampFilters` range-bounding step is skipped entirely for the `/contracts/:contractId/results` path.
- `extractContractStateByIdQuery` (line 910–911) correctly rejects `ne` for its timestamp, but `extractContractResultsByIdQuery` has no equivalent guard: [8](#0-7) 

### Impact Explanation
An unauthenticated attacker can repeatedly issue requests like:
```
GET /api/v1/contracts/0.0.{popular_contract}/results?timestamp=ne:T1&timestamp=ne:T2&...&timestamp=ne:T100
```
Each request forces a full scan of all `contract_result` rows for the targeted contract, evaluating 100 `!=` predicates per row. On a production network where a popular contract (e.g., a DEX or stablecoin) has tens of millions of results, each such request consumes significant CPU and I/O on the database node. Sending these requests concurrently from multiple clients (or even a single client with async requests) can saturate database workers across mirror node replicas, degrading or halting query processing for all users — consistent with the ≥30% node processing impact threshold described in the finding scope.

### Likelihood Explanation
- **No authentication required**: any HTTP client can call `/api/v1/contracts/:contractId/results`.
- **Trivially discoverable**: the `ne` operator is documented in the OpenAPI spec (`ne:1234567890.000000300`) and demonstrated in test fixtures.
- **Repeatable and automatable**: a simple script can fire concurrent requests in a loop.
- **Target selection is easy**: popular contracts (DEXes, stablecoins) are publicly known by their contract IDs.
- **No rate limiting visible in the codebase** at the controller or middleware level for this endpoint.

### Recommendation
1. **Reject `ne`-only timestamp queries without a range bound** in `extractContractResultsByIdQuery`: if all timestamp filters are `ne` operators and no `gte`/`lte`/`gt`/`lt`/`eq` bound is present, throw `InvalidArgumentError` (mirroring the pattern in `extractContractStateByIdQuery`).
2. **Cap the number of `ne` timestamp values** independently of `maxRepeatedQueryParameters` — e.g., allow at most 2–3 `ne` values.
3. **Apply `optimizeTimestampFilters` for the `/contracts/:contractId/results` path as well**, so that `ne`-only queries are either rejected or bounded to a default time window.
4. **Consider rejecting `ne` entirely for `timestamp` on contract results**, as `extractContractStateByIdQuery` already does (line 910–911), unless there is a specific product requirement for it.

### Proof of Concept

**Prerequisites:** A mirror node instance with a high-volume contract (e.g., `0.0.1234567`).

**Steps:**

```bash
# Build a URL with 100 distinct ne timestamp values
BASE="https://<mirror-node>/api/v1/contracts/0.0.1234567/results"
PARAMS=$(python3 -c "
import random
params = '&'.join(f'timestamp=ne:{random.randint(1,9999999999)}.{random.randint(0,999999999):09d}' for _ in range(100))
print(params)
")

# Fire concurrent requests
for i in $(seq 1 50); do
  curl -s "${BASE}?${PARAMS}" &
done
wait
```

**Expected result:** Each request triggers a full scan of all `contract_result` rows for contract `0.0.1234567` with 100 `!=` filter conditions. Under concurrent load, database CPU and I/O spike, query latency for all users increases, and mirror node processing degrades across replicas.

**Verification of SQL generation:** The condition path is confirmed by `baseController.js` lines 23–26 (non-`eq` operators produce raw `${fullName}${filter.operator}$${position}` conditions) and the absence of any `ne`-rejection guard in `extractContractResultsByIdQuery` (lines 415–542).
