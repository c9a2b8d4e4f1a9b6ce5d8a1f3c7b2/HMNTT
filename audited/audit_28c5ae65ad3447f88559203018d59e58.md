### Title
Missing Deleted-Status Check in `findRuntimeBytecode` Allows EVM Simulation of Logically Deleted Contracts

### Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table without joining the `entity` table to verify the contract is not deleted. `ContractBytecodeReadableKVState.readFromDataSource()` calls this method directly for the `contractNum` and long-zero EVM address paths, with no upstream deleted-status guard. Any unprivileged caller of the web3 API can therefore trigger EVM simulation against a logically deleted contract, receiving bytecode execution results as if the contract were active.

### Finding Description

**Exact code path:**

`ContractRepository.findRuntimeBytecode()` — the sole query:
```sql
select runtime_bytecode from contract where id = :contractId
``` [1](#0-0) 

This query touches only the `contract` table. The `deleted` flag lives exclusively in the `entity` table. When a contract is deleted on Hedera, `entity.deleted` is set to `true`, but the `contract` row (including `runtime_bytecode`) is never removed.

`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode` unconditionally: [2](#0-1) 

The `toEntityId()` helper has three branches:
- **`contractNum` path** (line 50–51): converts directly to `EntityId`, no deleted check.
- **Long-zero EVM address path** (line 54–55): converts directly to `EntityId`, no deleted check.
- **Non-long-zero EVM address path** (line 57–60): calls `commonEntityAccessor.getEntityByEvmAddressAndTimestamp()`, which internally calls `findByEvmAddressAndDeletedIsFalse` — **deleted check present**. [3](#0-2) 

**Root cause:** The `contractNum` and long-zero EVM address paths bypass the entity table entirely, so the `deleted` flag is never consulted before bytecode is returned to the EVM.

**Contrast with correct pattern:** `NftRepository.findActiveById()` explicitly joins the entity table and filters `e.deleted is not true`: [4](#0-3) 

`CommonEntityAccessor.get(EntityId, timestamp)` also enforces the deleted check for all other entity lookups: [5](#0-4) 

### Impact Explanation
The web3 module is the mirror node's EVM simulation layer. When it returns a non-null `Bytecode` for a deleted contract, the EVM treats that address as a live contract and executes its code. This produces simulation results (from `eth_call`, `eth_estimateGas`, etc.) that contradict the actual Hedera network state, where calls to deleted contracts are rejected. Applications that rely on mirror-node simulation to gate real transactions will receive false-positive results. Additionally, the `@Cacheable` annotation on `findRuntimeBytecode` means stale bytecode for a deleted contract can persist in the cache indefinitely after deletion, amplifying the window of incorrect responses. [6](#0-5) 

### Likelihood Explanation
No authentication or privilege is required to call the web3 API endpoints (`eth_call`, `eth_estimateGas`). Deleted contracts are a normal, frequent occurrence on Hedera (self-destruct, admin delete). The contract ID of any deleted contract is public information visible on the network. The exploit requires only knowing a deleted contract's numeric ID and issuing a standard JSON-RPC call — trivially repeatable by any external user.

### Recommendation
Fix `findRuntimeBytecode` to join the `entity` table and exclude deleted contracts, consistent with the pattern used in `NftRepository`:

```sql
select c.runtime_bytecode
from contract c
join entity e on e.id = c.id
where c.id = :contractId
  and (e.deleted is not true)
```

Alternatively, add a deleted-status guard in `ContractBytecodeReadableKVState.readFromDataSource()` for the `contractNum` and long-zero EVM address paths by calling `commonEntityAccessor.get(entityId, timestamp)` before invoking `findRuntimeBytecode`, mirroring the non-long-zero EVM address branch. Also invalidate the Spring cache entry for a contract when its `deleted` flag transitions to `true`.

### Proof of Concept
1. Deploy a smart contract on Hedera (e.g., contract ID `0.0.12345`). Confirm it is active.
2. Issue a `ContractDeleteTransaction` for `0.0.12345`. Confirm `entity.deleted = true` in the mirror node DB.
3. Send an `eth_call` to the mirror node web3 endpoint targeting the contract's long-zero address (`0x0000000000000000000000000000000000003039`) or its numeric ID.
4. Observe that `ContractBytecodeReadableKVState.readFromDataSource()` returns a non-null `Bytecode` (the deleted contract's runtime bytecode).
5. The EVM executes the bytecode and returns a result — the mirror node reports the deleted contract as active, contradicting the actual network state where the same call would be rejected.

    private EntityId toEntityId(@NonNull final ContractID contractID) {
        if (contractID.hasContractNum()) {
            return entityIdFromContractId(contractID);
        } else if (contractID.hasEvmAddress()) {
            final var evmAddress = contractID.evmAddress().toByteArray();
            if (isLongZeroAddress(evmAddress)) {
                return DomainUtils.fromEvmAddress(evmAddress);
            } else {
                return commonEntityAccessor
                        .getEntityByEvmAddressAndTimestamp(evmAddress, Optional.empty())
                        .map(Entity::toEntityId)
                        .orElse(EntityId.EMPTY);
            }
        }
        return EntityId.EMPTY;
    }
```
