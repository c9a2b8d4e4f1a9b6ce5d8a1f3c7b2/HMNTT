### Title
Uncached Historical Storage Queries Enable DB Load Amplification via SLOAD Amplification in Historical Contract Calls

### Summary
`ContractStateServiceImpl.findStorageByBlockTimestamp()` passes every call directly to the database with no caching, unlike `findStorage()` which has both `@Cacheable` at the repository layer and a batch-prefetch cache. Because each EVM SLOAD opcode during a historical block simulation triggers one independent uncached DB query, an unprivileged attacker can amplify DB load by a factor proportional to the number of SLOAD operations per request, exhausting DB connection capacity and degrading simulation throughput for all users. The existing throttle operates at the HTTP request level and does not account for this per-SLOAD amplification.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageByBlockTimestamp()` (lines 73–76) is a one-liner passthrough with zero caching:

```java
// ContractStateServiceImpl.java lines 73-76
public Optional<byte[]> findStorageByBlockTimestamp(
        final EntityId entityId, final byte[] slotKeyByteArray, final long blockTimestamp) {
    return contractStateRepository.findStorageByBlockTimestamp(entityId.getId(), slotKeyByteArray, blockTimestamp);
}
``` [1](#0-0) 

The repository method carries no `@Cacheable` annotation, unlike `findStorage()` which has `@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_CONTRACT_STATE)`: [2](#0-1) [3](#0-2) 

Every EVM SLOAD opcode during a historical simulation routes through `ContractStorageReadableKVState.readFromDataSource()`, which selects the historical path when a timestamp is present:

```java
// ContractStorageReadableKVState.java lines 41-44
return timestamp
    .map(t -> contractStateService.findStorageByBlockTimestamp(
            entityId, Bytes32.wrap(keyBytes).trimLeadingZeros().toArrayUnsafe(), t))
    .orElse(contractStateService.findStorage(entityId, keyBytes))
``` [4](#0-3) 

**Root cause:** The design assumption is that historical queries are infrequent or that the DB can absorb the load. This assumption fails under adversarial conditions because: (a) there is no result cache keyed on `(contractId, slot, blockTimestamp)`, and (b) the throttle is applied once per HTTP request, not once per DB query generated by that request.

**Exploit flow:**
1. Attacker identifies or deploys a contract with many SLOAD operations (e.g., reads 100+ storage slots in a loop).
2. Attacker submits POST `/api/v1/contracts/call` with `block` set to a historical block number and `gas=21000` (minimum).
3. Each request triggers N uncached `SELECT ... FROM contract_state_change WHERE contract_id=? AND slot=? AND consensus_timestamp<=? ORDER BY consensus_timestamp DESC LIMIT 1` queries.
4. Attacker repeats at the maximum allowed rate.

**Why existing checks fail:**

The `ThrottleManagerImpl` checks `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` (default 7.5B gas/s, scaled by 10,000): [5](#0-4) [6](#0-5) 

With `gas=21000`, `scaleGas(21000)` = `floor(21000/10000)` = 2 tokens consumed from the gas bucket (capacity 750,000 tokens/s). This allows up to 375,000 req/s from the gas-bucket perspective, capped only by the 500 req/s rate limit. Each of those 500 requests can contain ~100–200 SLOAD operations (within 21,000 gas), yielding **50,000–100,000 uncached DB queries per second** from a single attacker. The throttle is applied once at the controller layer before EVM execution begins — it has no visibility into how many DB queries the EVM will generate internally. [7](#0-6) 

### Impact Explanation
Each uncached query hits `contract_state_change` with an `ORDER BY consensus_timestamp DESC LIMIT 1` on a potentially large table. Under sustained attack, DB connection pools saturate, query latency increases for all users, and transaction simulation throughput degrades or halts entirely. The impact is a denial-of-service against the DB tier affecting all users of the web3 API, not just the attacker's requests. Severity is **High** because it requires no authentication, no special privileges, and no on-chain transactions.

### Likelihood Explanation
Any external user can submit historical contract calls — the `block` field in `ContractCallRequest` accepts any block number and requires no authentication. A single attacker with a standard HTTP client can sustain 500 req/s (the global rate limit). Deploying a storage-heavy contract is optional; the attacker can target any existing contract with multiple SLOAD operations. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add a bounded cache for historical storage queries** keyed on `(contractId, slot, blockTimestamp)` with a short TTL (e.g., 30–60 seconds) and a maximum size. Historical state is immutable once a block is finalized, so caching is safe and correct.
2. **Account for SLOAD amplification in the gas throttle**: charge gas-bucket tokens proportional to the actual gas *used* (post-execution) rather than only the gas *limit* declared upfront, or introduce a separate per-request DB-query counter limit.
3. **Apply a per-IP or per-source rate limit** at the ingress layer (e.g., API gateway) to prevent a single client from consuming the full global 500 req/s budget.
