### Title
Unauthenticated DoS via Unbounded `ShallowEtagHeaderFilter` Buffering on `/api/*` with No Rate Limiting in `rest-java`

### Summary
The `etagFilter()` bean in `RestJavaConfiguration` registers Spring's `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no authentication, no rate limiting, and no request throttling. `ShallowEtagHeaderFilter` unconditionally buffers the entire response body in memory and computes a SHA hash on every request — regardless of what value the client sends in `If-None-Match`. Because the `rest-java` module has no rate-limiting layer (unlike the `web3` module), any unprivileged external user can flood these endpoints to exhaust CPU and heap memory.

### Finding Description
**Exact code location:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46, `etagFilter()`:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause — how `ShallowEtagHeaderFilter` works:**
Spring's `ShallowEtagHeaderFilter` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper`. On every request it:
1. Lets the full handler chain execute (DB query, serialization, full response body written to the wrapper).
2. Buffers the entire response body in a heap byte array.
3. Computes an MD5/SHA-256 digest of that buffer to produce the ETag.
4. Only then compares the computed ETag against the client-supplied `If-None-Match` value.

The `If-None-Match` header value — whether valid, malformed, or arbitrarily large — is never inspected before steps 1–3 complete. There is no short-circuit. Every request, regardless of the header, incurs the full DB round-trip, full in-memory buffering, and full hash computation.

**No rate limiting in `rest-java`:**
The `web3` module has an explicit `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j rate limiting (`requestsPerSecond`, `gasPerSecond`). The `rest-java` module has no equivalent. The only resources file present is `banner.txt`; there is no `application.yml` with rate-limiting configuration. No authentication is required to reach `/api/*`.

**Exploit flow:**
An attacker sends a high-volume stream of GET requests to any `/api/*` endpoint (e.g., `/api/v1/accounts`, `/api/v1/transactions`) with an arbitrary `If-None-Match` header. Each request independently triggers: full DB query → full response body buffered in heap → SHA computation → string comparison → 304 or 200 response. Because there is no per-IP or global request rate limit, the attacker can saturate CPU (hash computation) and heap memory (response buffering) with no credential or privilege required.

### Impact Explanation
- **CPU exhaustion**: SHA computation over large response payloads (e.g., paginated account or transaction lists) is non-trivial and scales with response size. Concurrent flood requests multiply this linearly.
- **Heap memory pressure**: Each in-flight request holds a full response body byte array in the `ContentCachingResponseWrapper` until the filter completes. Under flood conditions this can trigger GC pressure or OOM.
- **Service degradation / denial**: Legitimate requests are starved of CPU and memory, causing latency spikes or outright 503s.
- **Severity**: Medium — griefing/DoS with no economic damage to the network, consistent with the stated scope.

### Likelihood Explanation
- No authentication or API key is required.
- No rate limiting exists in `rest-java` at the application layer.
- The attack requires only a standard HTTP client and knowledge of any `/api/*` endpoint path (publicly documented).
- It is trivially repeatable and automatable (e.g., `ab`, `wrk`, `curl` in a loop).
- Infrastructure-level mitigations (WAF, ingress rate limiting) may exist but are not enforced by the application itself.

### Recommendation
1. **Add application-level rate limiting to `rest-java`**: Mirror the `ThrottleConfiguration` / bucket4j pattern from the `web3` module. Register a servlet filter or Spring `HandlerInterceptor` that enforces a per-IP or global requests-per-second cap before the `ShallowEtagHeaderFilter` runs.
2. **Consider replacing `ShallowEtagHeaderFilter` with a cache-aware ETag strategy**: Pre-compute and store ETags alongside cached responses (as the `rest` Node.js service does via `responseCacheHandler`), so the filter can short-circuit on `If-None-Match` match without buffering the full body.
3. **Set Tomcat/server-level connection and request limits** as a defense-in-depth measure (e.g., `server.tomcat.max-connections`, `server.tomcat.accept-count`).

### Proof of Concept
```bash
# Flood /api/v1/accounts with a non-matching If-None-Match header.
# Each request causes full DB query + heap buffering + SHA computation.
# No credentials required.

for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    -H 'If-None-Match: "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"' \
    http://<mirror-node-host>/api/v1/accounts &
done
wait
```
