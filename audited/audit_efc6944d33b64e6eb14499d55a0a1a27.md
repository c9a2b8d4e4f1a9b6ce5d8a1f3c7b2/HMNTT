### Title
Unbounded Infinite DB Polling via `limit=0` Subscription in `PollingTopicListener`

### Summary
When the gRPC listener is configured to use the `POLL` type, any unprivileged client can open a subscription with `limit=0` (no limit) and an arbitrarily old `startTime`. This causes `RepeatSpec.times(Long.MAX_VALUE)` to fire a database query every 500 ms indefinitely per subscription, with no server-side termination. An attacker opening many such subscriptions can exhaust the database connection pool and deny service to all legitimate users.

### Finding Description

**Exact code path:**

In `PollingTopicListener.listen()` (lines 34–48), the Flux is constructed with:

```java
RepeatSpec.times(Long.MAX_VALUE)
    .withFixedDelay(interval)   // default 500 ms
``` [1](#0-0) 

This means each subscriber's polling loop runs for ~9.2 × 10¹⁸ iterations — practically forever.

**Root cause — `limit=0` treated as "no limit":**

`TopicMessageFilter` declares `limit` with `@Min(0)`, making `limit=0` a valid, accepted value: [2](#0-1) 

`hasLimit()` returns `limit > 0`, so `limit=0` → `hasLimit() == false`.

Inside `poll()`, when `hasLimit()` is false, the local limit is set to `Integer.MAX_VALUE`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // = 5000
``` [3](#0-2) 

There is no check anywhere in `listen()` or `poll()` that terminates the `RepeatSpec` loop when the subscription has no limit. The loop runs until the client disconnects.

**`startTime` validation is insufficient:**

The only constraint on `startTime` is `startTime <= DomainUtils.now()`: [4](#0-3) 

Any past timestamp (e.g., epoch 0) is accepted. With a very old `startTime` and no `last` message yet seen, every poll issues a full table scan from that old timestamp up to `pageSize=5000` rows, which is a heavy query.

**No per-subscriber rate limiting or connection cap** is present in the reviewed code.

### Impact Explanation

Each malicious subscription fires one DB query every 500 ms indefinitely. With the default `maxPageSize=5000`, each query can return up to 5,000 rows from a potentially large historical range. An attacker opening N concurrent subscriptions generates N DB queries every 500 ms. Since the `scheduler` is `Schedulers.boundedElastic()` and each subscription holds a DB connection for the duration of each poll, a modest number of concurrent subscriptions (proportional to the DB connection pool size) saturates the pool and causes all legitimate queries — including those from other gRPC subscribers and the importer — to time out or fail. This is a complete non-network DoS against the mirror node's database tier.

### Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is publicly accessible; no credentials are needed to open a subscription.
- **Trivial to trigger**: A single gRPC client call with `limit=0` and `startTime=0` is sufficient per connection.
- **Scalable**: One attacker can open many connections from a single host (or a small botnet), multiplying the DB load linearly.
- **Persistent**: The attack sustains itself as long as the TCP connection is held open; no repeated action is needed.
- **Condition**: The deployment must use `ListenerType.POLL`. The default is `REDIS`, but `POLL` is a documented and supported configuration option. [5](#0-4) 

### Recommendation

1. **Reject `limit=0` or treat it as a hard cap**: Either change `@Min(0)` to `@Min(1)` on `TopicMessageFilter.limit` so that unlimited subscriptions are not accepted, or enforce a server-side maximum subscription duration/message count.
2. **Add a maximum subscription lifetime**: Introduce a configurable `maxSubscriptionDuration` in `ListenerProperties` and apply `.timeout(maxSubscriptionDuration)` to the Flux in `listen()`.
3. **Cap concurrent subscriptions per client IP**: Add a gRPC interceptor that limits the number of concurrent `subscribeTopic` streams per source address.
4. **Bound `RepeatSpec`**: Replace `RepeatSpec.times(Long.MAX_VALUE)` with a configurable upper bound (e.g., `listenerProperties.getMaxPolls()`), or terminate the repeat when no new messages have arrived for a configurable idle period.

### Proof of Concept

```
# Using grpcurl (no auth required):
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0, "nanos": 0},
    "limit": 0
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

Repeat this command in N parallel shells. Each open stream causes one DB query every 500 ms. With enough parallel streams (matching or exceeding the DB connection pool size, typically 10–50), all other DB operations begin timing out, causing a full service outage for legitimate subscribers and the importer pipeline.
