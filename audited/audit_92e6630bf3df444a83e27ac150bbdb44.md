### Title
Unauthenticated Historical Contract Calls Trigger Uncached UNION Queries, Enabling Database Saturation DoS

### Summary
The `findActiveByIdAndTimestamp()` method in `EntityRepository.java` carries no `@Cacheable` annotation, meaning every historical contract call unconditionally issues a `UNION ALL` query across the `entity` and `entity_history` tables. The global RPS throttle (default 500 req/s) is not per-IP and is the only meaningful barrier; an unprivileged attacker can consume the entire budget with varying `blockTimestamp` values, flooding the database connection pool and degrading service for all users.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → `CommonEntityAccessor.get()` → `EntityRepository.findActiveByIdAndTimestamp()`

**Uncached historical query** — `findActiveByIdAndTimestamp` at lines 136–155 of `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` carries **no `@Cacheable` annotation**, unlike the non-historical sibling `findByIdAndDeletedIsFalse` (lines 20–30) which is cached via `CACHE_MANAGER_ENTITY`. Every invocation hits the database:

```sql
(select * from entity
 where id = ?1 and lower(timestamp_range) <= ?2 and deleted is not true)
union all
(select * from entity_history
 where id = ?1 and lower(timestamp_range) <= ?2 and deleted is not true
 order by lower(timestamp_range) desc limit 1)
order by timestamp_range desc limit 1
```

**Gas throttle is not a meaningful barrier** — `ThrottleProperties.scaleGas()` (lines 42–47 of `ThrottleProperties.java`) returns `Math.floorDiv(gas, 10_000)`. The minimum allowed gas per `ContractCallRequest` is `@Min(21_000)` (line 36 of `ContractCallRequest.java`), which scales to only **2 tokens** per request. The gas bucket holds `scaleGas(7_500_000_000) = 750_000` tokens/sec, so the gas throttle allows ~375,000 requests/sec — far above the RPS limit.

**RPS throttle is global, not per-IP** — `ThrottleManagerImpl.throttle()` (lines 37–42 of `ThrottleManagerImpl.java`) checks a single shared `rateLimitBucket` (default 500 req/s). A single attacker can consume the entire budget, leaving zero capacity for legitimate users.

**Different timestamps defeat any query-level caching** — because each request carries a distinct `blockTimestamp`, no result-set cache can absorb the load.

### Impact Explanation

With 500 concurrent historical queries per second, each executing a `UNION ALL` across `entity` and `entity_history` (which grows unboundedly as the chain ages), the database connection pool is exhausted. All users — including those making non-historical calls — experience query timeouts and HTTP 500/503 errors. The `statementTimeout` default of 3000 ms (documented in `docs/configuration.md` line 702) means each connection is held for up to 3 seconds, amplifying pool exhaustion. This is a sustained, repeatable service degradation with no economic cost to the attacker.

### Likelihood Explanation

The `/api/v1/contracts/call` endpoint requires no authentication. Any external actor can craft POST requests with a historical block number (any past block number is valid) and the minimum gas of 21,000. Varying the `block` field across requests is trivial. The attack is fully automatable with a simple HTTP load generator and requires no on-chain funds or privileged access.

### Recommendation

1. **Per-IP rate limiting**: Enforce a per-source-IP request limit at the application or reverse-proxy layer so a single client cannot consume the global RPS budget.
2. **Cache historical entity lookups**: Apply a short-lived (e.g., 1–5 s) `@Cacheable` to `findActiveByIdAndTimestamp` keyed on `(id, blockTimestamp)`. Historical state is immutable once a block is finalized, so cache entries never become stale.
3. **Separate throttle buckets for historical vs. latest calls**: Historical calls are significantly more expensive; apply a lower sub-limit specifically for requests with a non-`latest` block parameter.
4. **Database connection pool cap with fast-fail**: Configure a maximum pool size and a short acquisition timeout so that pool exhaustion returns an immediate 429/503 rather than queuing indefinitely.

### Proof of Concept

```bash
# Send 500 concurrent historical contract calls per second with varying timestamps
# No authentication required; minimum gas (21000) bypasses gas throttle effectively

BASE_BLOCK=1000000
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{
      \"to\": \"0x0000000000000000000000000000000000000001\",
      \"gas\": 21000,
      \"block\": \"$((BASE_BLOCK + i))\"
    }" &
done
wait
# Each request triggers an uncached UNION ALL across entity + entity_history.
# Repeat in a loop to sustain 500 RPS and exhaust the DB connection pool.
# Legitimate users will begin receiving timeouts within seconds.
```
