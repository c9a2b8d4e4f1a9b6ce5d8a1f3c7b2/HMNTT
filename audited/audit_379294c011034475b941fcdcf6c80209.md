### Title
Unbounded Connection Acceptance in `grpcServerConfigurer` Enables Connection-Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean configures the Netty gRPC server with only a per-connection RPC call limit (`maxConcurrentCallsPerConnection = 5`) but sets no limit on the total number of accepted connections, no `maxConnectionAge`, and no `maxConnectionIdle`. An unprivileged attacker can open and hold open an arbitrary number of TCP/HTTP2 connections — each consuming file descriptors, memory, and thread-pool slots — until the server's OS-level accept queue overflows and silently drops new TCP SYN packets from legitimate clients. The partition-recovery window amplifies the impact because all legitimate clients attempt to reconnect simultaneously, competing with attacker-held connections for the fixed-size accept queue.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35, `grpcServerConfigurer()`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection(5)` restricts the number of concurrent gRPC *calls* multiplexed over a single HTTP/2 connection. It does **not** restrict:
- The total number of simultaneous TCP connections the server will accept.
- How long any connection may remain open (`maxConnectionAge` is never called).
- How long an idle connection may persist (`maxConnectionIdle` is never called).

A grep across the entire codebase confirms no `maxConnectionAge`, `maxConnectionIdle`, `maxConnections`, `SO_BACKLOG`, or `childOption` is set anywhere in Java source. No `application.yml` or `application.properties` overrides exist for these parameters either.

**Exploit flow:**
1. Attacker opens *N* TCP connections to port 5600, sending a valid HTTP/2 SETTINGS frame on each to complete the gRPC handshake. Each connection is accepted and held in the Netty channel pipeline.
2. On each connection the attacker sends 0–5 `subscribeTopic` RPCs (staying within the per-connection call limit) to prevent the server from closing the connection on protocol grounds.
3. Each held connection consumes: one file descriptor, ~64–128 KB of Netty channel buffers, and a slot in the `applicationTaskExecutor` thread pool for any active calls.
4. When *N* is large enough to exhaust file descriptors or thread-pool capacity, the Netty boss thread can no longer call `accept()` fast enough; the OS TCP accept queue (default backlog = 128 in Netty's `NioServerSocketChannel`) fills up.
5. Subsequent TCP SYN packets — including those from legitimate clients reconnecting after a partition — receive no SYN-ACK and are silently dropped.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection(5)` — scoped to calls, not connections; an attacker simply opens more connections.
- GCP gateway `maxRatePerEndpoint: 250` (charts/hedera-mirror-grpc/values.yaml line 69) — rate-limits requests per second at the load balancer, but does not cap the number of *established* long-lived connections, which are opened once and held.
- `SharedTopicListener.onBackpressureBuffer` — application-layer backpressure on message delivery; irrelevant to TCP connection acceptance.

### Impact Explanation
A successful attack causes a complete denial of service for new gRPC connections. Because `subscribeTopic` is a long-lived streaming RPC (clients hold connections open indefinitely to receive topic messages), attacker connections are indistinguishable from legitimate ones at the transport layer. Legitimate clients — including Hedera SDK users and the mirror-node monitor — are unable to establish new connections. During a partition-recovery window the impact is total: all legitimate clients attempt simultaneous reconnection, and every slot in the accept queue is occupied by attacker-held connections. Severity: **High** (availability impact, no authentication bypass required).

### Likelihood Explanation
Preconditions are minimal: network access to port 5600 and the ability to complete a standard TLS/HTTP2 handshake (or plain-text if TLS termination is at the load balancer). No credentials, tokens, or knowledge of internal state are required. The attack is repeatable and scriptable with standard tools (`grpc_cli`, `h2load`, or a simple Go/Python gRPC client). The partition-recovery window is a predictable, recurring event in any distributed deployment. Likelihood: **Medium-High**.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
serverBuilder.maxConnectionAge(
    nettyProperties.getMaxConnectionAge().toSeconds(), TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(
    nettyProperties.getMaxConnectionAgeGrace().toSeconds(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(
    nettyProperties.getMaxConnectionIdle().toSeconds(), TimeUnit.SECONDS);
```

Add corresponding fields to `NettyProperties` with conservative defaults (e.g., `maxConnectionAge = 1h`, `maxConnectionAgeGrace = 30s`, `maxConnectionIdle = 5m`). Additionally, consider setting `serverBuilder.permitKeepAliveWithoutCalls(false)` and `serverBuilder.permitKeepAliveTime(...)` to evict truly idle connections. For defense-in-depth, enforce a per-source-IP connection rate limit at the Kubernetes `NetworkPolicy` or Envoy/GCP Gateway layer.

### Proof of Concept
```python
import grpc
import threading
import time

TARGET = "grpc-mirror-node:5600"
NUM_CONNECTIONS = 500  # tune to exhaust fd limit

channels = []
for _ in range(NUM_CONNECTIONS):
    ch = grpc.insecure_channel(TARGET)
    # Send HTTP/2 SETTINGS to complete handshake; hold channel open
    channels.append(ch)
    time.sleep(0.01)  # avoid triggering rate limiter

print(f"Holding {len(channels)} connections open")
# Legitimate client now attempts to connect:
try:
    victim = grpc.insecure_channel(TARGET)
    grpc.channel_ready_future(victim).result(timeout=5)
    print("Connected (unexpected)")
except grpc.FutureTimeoutError:
    print("BLOCKED - accept queue exhausted, legitimate client cannot connect")

time.sleep(3600)  # hold connections
```

Run this from a host with direct access to port 5600 (bypassing the load balancer). After `NUM_CONNECTIONS` channels are established, a new connection attempt from a separate process will time out, demonstrating accept-queue saturation.
