### Title
Unauthenticated Log Flooding via Empty AccountInput in GraphQL `account()` Query

### Summary
The `account()` query in `AccountController` accepts an `AccountInput` with all fields null, causing `validateOneOf()` to throw an `IllegalArgumentException`. There is no authentication, no rate limiting, and no per-IP throttling on the GraphQL endpoint, allowing any unprivileged external user to send this malformed query at arbitrary frequency. Each request generates an `info`-level log entry (not an error-level entry, as the exception is resolved by `CustomExceptionResolver` into an HTTP 200 GraphQL validation error), but the volume of log writes can still exhaust disk I/O and log storage.

### Finding Description

**Code path:**

- `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, line 39:
  ```java
  validateOneOf(alias, entityId, evmAddress, id);
  ```
  When `AccountInput` is `{}`, all four arguments are `null`.

- `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 68–80:
  ```java
  public static void validateOneOf(Object... values) {
      int nonNull = 0;
      for (var value : values) { if (value != null) { ++nonNull; } }
      if (nonNull != 1) {
          throw new IllegalArgumentException(
              "Must provide exactly one input value but " + nonNull + " have been provided");
      }
  }
  ```
  Throws `IllegalArgumentException` with `nonNull = 0`.

- `graphql/src/main/java/org/hiero/mirror/graphql/config/CustomExceptionResolver.java`, lines 16–29:
  The `IllegalArgumentException` is caught and converted to a GraphQL `ValidationError` returned as HTTP 200. **No stack trace is logged by this resolver.**

- `graphql/src/main/java/org/hiero/mirror/graphql/config/LoggingFilter.java`, lines 55–59:
  ```java
  } else if (status >= HttpStatus.INTERNAL_SERVER_ERROR.value()) {
      log.warn(LOG_FORMAT, params);
  } else {
      log.info(LOG_FORMAT, params);  // HTTP 200 → info level
  }
  ```
  Every request (including the malformed one) is logged at `info` level.

**Root cause:** No authentication (`SecurityConfig` is absent from the graphql module), no rate limiting, and no per-client throttling. The `CachedPreparsedDocumentProvider` caches the parsed document after the first request, so repeated identical queries bypass parsing overhead — making the attack cheaper to sustain. The only cost per request is the log write.

**Failed assumption:** The design assumes that returning a structured GraphQL error (HTTP 200) is sufficient mitigation. It is not, because the `LoggingFilter` still writes one log line per request regardless of the response body content, and there is no mechanism to limit request rate.

### Impact Explanation

An attacker can generate an unbounded stream of `info`-level log entries, each containing the remote IP, method, URI, elapsed time, HTTP status, and "Success" (since `getMessage()` returns `SUCCESS` when no exception propagates to the filter). At high frequency this can:
- Fill log storage and trigger log rotation storms
- Increase I/O pressure on the host
- Obscure legitimate log entries (log drowning), degrading observability

This is a griefing/availability impact with no economic damage to network users, matching the declared medium scope.

### Likelihood Explanation

- **Precondition:** None. No credentials, API key, or special network access required.
- **Trigger:** A single HTTP POST to `/graphql` with body `{"query":"{ account(input: {}) { id }}"}`.
- **Repeatability:** Trivially scriptable; the `PreparsedDocumentProvider` cache means the server does minimal work per request, so the attacker can sustain a high request rate cheaply.
- **Detection evasion:** All responses are HTTP 200; standard HTTP-level anomaly detection (watching for 4xx/5xx spikes) will not fire.

### Recommendation

1. **Rate limiting:** Add a per-IP or per-connection request rate limit at the ingress/reverse-proxy layer or via a Spring `HandlerInterceptor`/filter before the GraphQL handler.
2. **Input validation at the schema level:** Enforce that at least one field of `AccountInput` is non-null using a custom GraphQL directive or schema constraint, so the request is rejected before reaching the controller data-fetcher (and before a log entry is written by `LoggingFilter`).
3. **Reduce log verbosity for known-bad inputs:** In `CustomExceptionResolver`, optionally suppress or rate-limit logging for repeated identical validation errors from the same source.

### Proof of Concept

```bash
# Single malformed request — returns HTTP 200 with GraphQL error body
curl -s -X POST http://<host>/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: {}) { id }}"}'

# High-frequency flood (no authentication required)
while true; do
  curl -s -o /dev/null -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: {}) { id }}"}' &
done
```

Each iteration writes one `info`-level line to the application log. After the first request, the parsed document is served from `CachedPreparsedDocumentProvider`'s Caffeine cache, so server-side CPU cost per request is minimal, maximizing the log-write-to-attacker-cost ratio.
