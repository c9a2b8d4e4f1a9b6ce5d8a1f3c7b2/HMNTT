### Title
Unauthenticated EVM Address Lookup Causes Unbounded DB Queries with No Negative-Result Caching, Enabling DoS Against Shared Database

### Summary
The `getAccountCryptoAllowances()` handler unconditionally calls `EntityService.getEncodedId()` for every inbound EVM address, which executes a live database query against the `entity` table for each request. Because negative results (non-existent EVM addresses) are never cached and no application-level rate limiting exists on this endpoint, an unauthenticated attacker can flood the shared PostgreSQL database with sequential or concurrent queries, exhausting connection pool capacity and degrading the gossip importer's write throughput.

For a non-existent address, the query returns zero rows, `NotFoundError` is thrown (404 response), and **no result is cached anywhere**.

**Root cause — failed assumption:** The `parseCached()` LRU cache in `rest/entityId.js` lines 301–333 caches only the *parsed* `EntityId` object (shard/realm/evmAddress fields extracted from the string), not the *database lookup result*. [5](#0-4) 

The `EntityService.getEntityIdFromEvmAddress()` function has no caching layer at all. Every call — including for the same non-existent address — hits the database.

**Why existing checks are insufficient:**

- `EntityId.isValidEntityId()` validates only the *format* (40 hex chars), not existence. [6](#0-5) 
- The Redis `Cache` class in `rest/cache.js` is used for API response caching, not for entity existence lookups. [7](#0-6) 
- No application-level rate limiting exists in the REST API JavaScript code for this endpoint. The `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiting applies only to the `web3` module, not the REST API. [8](#0-7) 

### Impact Explanation

The REST API and the gossip importer (mirror node's transaction ingestion pipeline) share the same PostgreSQL database and its connection pool. Flooding `GET /api/v1/accounts/{evmAddress}/allowances/crypto` with distinct random EVM addresses causes one `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query per request. At sufficient concurrency, this saturates available DB connections and increases query queue depth, directly starving the gossip importer of DB write capacity. This degrades or halts transaction ingestion from the Hedera network — the mirror node's primary function.

### Likelihood Explanation

Preconditions: none. The endpoint is fully public, requires no authentication, and accepts any 40-hex-char string as a valid path parameter. The attacker needs only a standard HTTP client. Distinct random EVM addresses bypass any future per-key caching. The attack is trivially scriptable, repeatable, and can be distributed across multiple source IPs to defeat infrastructure-level per-IP rate limits (e.g., Traefik `inFlightReq` per IP).

### Recommendation

1. **Cache negative results**: In `EntityService.getEntityIdFromEvmAddress()`, cache `null` results (non-existent EVM addresses) in Redis with a short TTL (e.g., 30–60 seconds) so repeated lookups for the same non-existent address do not hit the database.
2. **Add application-level rate limiting**: Apply a per-IP or global request rate limit to the REST API, equivalent to the `rateLimit` middleware already present in the Rosetta chart.
3. **Fail fast on format**: For EVM addresses that decode to a "long-zero" form (first 12 bytes all zero), resolve them locally without a DB query — this is already partially handled in `DomainUtils.fromEvmAddress()` in the Java layer but not in the JS REST layer.

### Proof of Concept

```bash
# Generate 10,000 random EVM addresses and flood the endpoint concurrently
for i in $(seq 1 10000); do
  addr=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/${addr}/allowances/crypto" &
done
wait
```

Each request triggers `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` with a unique address. With sufficient concurrency, DB connection pool exhaustion occurs, causing the gossip importer to queue or fail writes, observable as increasing lag in `mirror_importer_transaction_latency` metrics.
