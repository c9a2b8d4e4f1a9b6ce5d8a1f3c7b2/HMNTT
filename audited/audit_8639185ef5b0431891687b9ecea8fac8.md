### Title
Unauthenticated GraphQL `account()` Query Enables Connection Pool Exhaustion DoS

### Summary
The GraphQL module exposes the `account()` query endpoint with no authentication, no rate limiting, and no concurrency control. Each invocation synchronously acquires a JDBC connection from the shared HikariCP pool via `entityRepository.findById()`. An unprivileged attacker flooding this endpoint with concurrent requests can exhaust the connection pool, causing all subsequent database operations across the application to queue and time out, effectively denying service to all users.

### Finding Description

**Code path:**

`AccountController.account()` (line 33) → `EntityServiceImpl.getByIdAndType()` (line 25) → `entityRepository.findById()` (line 25) — a synchronous blocking JDBC call that holds a HikariCP connection for the duration of the query. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — absent rate limiting in the GraphQL module:**

The `web3` module has a full `ThrottleConfiguration` with bucket4j rate limiting applied before any DB access. The `graphql` module has no equivalent. The only protections configured in `GraphQlConfiguration` are structural per-query limits (complexity ≤ 200, depth ≤ 10, token counts), which do nothing to bound the *number of concurrent requests*. [4](#0-3) [5](#0-4) 

The `LoggingFilter` is the only servlet filter in the graphql module; it performs no throttling or authentication. [6](#0-5) 

**HikariCP pool — shared, finite, no per-endpoint cap:**

`CommonConfiguration` creates a single `HikariDataSource` from `spring.datasource.hikari` properties. HikariCP's default `maximumPoolSize` is 10. When all connections are held, new callers block until `connectionTimeout` (default 30 s) then receive `SQLTimeoutException`. During a sustained flood the pool stays saturated continuously, so legitimate requests time out for the duration of the attack. [7](#0-6) 

### Impact Explanation

During a sustained flood, the HikariCP pool is continuously saturated. Every legitimate user request that needs a DB connection — including health checks, other queries, and internal operations — either blocks for up to 30 seconds or fails with a connection-timeout exception. This is an application-wide denial of service: the GraphQL service is effectively partitioned from its database for all users for as long as the attack continues. Severity is **High** (availability impact, no authentication required, no infrastructure change needed by attacker).

### Likelihood Explanation

The endpoint is publicly reachable over HTTP with no credentials. A single attacker with a modest number of concurrent HTTP connections (e.g., 50–100 goroutines or `ab -c 100`) is sufficient to saturate a default pool of 10. The attack is trivially repeatable, requires no special knowledge beyond the GraphQL schema (which is introspectable), and leaves no persistent damage — making it low-risk for the attacker to repeat indefinitely.

### Recommendation

1. **Add rate limiting to the GraphQL module** — mirror the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern from `web3`, applying a per-IP or global requests-per-second cap via a servlet filter or Spring GraphQL interceptor before any resolver executes.
2. **Bound the HikariCP pool acquisition timeout** — set `spring.datasource.hikari.connectionTimeout` to a low value (e.g., 2–5 s) so pool exhaustion fails fast rather than queuing threads.
3. **Set a Tomcat/Netty max-connections or max-threads limit** specific to the GraphQL service to cap the number of concurrent in-flight requests at the HTTP layer.
4. **Consider query cost accounting** — assign a DB-connection cost to resolvers and reject requests that would exceed a per-second budget, similar to the gas-limit bucket in `web3`.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or ab
# Target: GraphQL endpoint (default port 8083)

QUERY='{"query":"{ account(input: {entityId: {num: 1}}) { id } }"}'

# Send 500 concurrent requests
seq 500 | parallel -j 100 \
  curl -s -X POST http://localhost:8083/graphql \
       -H 'Content-Type: application/json' \
       -d "$QUERY" \
       -o /dev/null -w "%{http_code}\n"

# Expected result during attack:
# Legitimate requests begin returning 500 errors or hanging for ~30 s
# HikariCP logs: "Connection is not available, request timed out after 30000ms"
# All application DB operations (not just account queries) are affected
```

    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
