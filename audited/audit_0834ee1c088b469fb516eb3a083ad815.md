### Title
Missing HTTP/2 Preface Timeout in `grpcServerConfigurer` Enables Connection-Exhaustion DoS

### Summary
The `grpcServerConfigurer` bean in `GrpcConfiguration.java` configures `NettyServerBuilder` with only an executor and `maxConcurrentCallsPerConnection`, omitting any `handshakeTimeout` or HTTP/2 preface timeout. An unprivileged attacker can open many TCP connections, send a partial HTTP/2 client preface, and cause the server to hold each connection open indefinitely, exhausting file descriptors and denying service to legitimate clients.

### Finding Description
**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyServerBuilder` exposes `handshakeTimeout(long, TimeUnit)` which, when not called, leaves the server relying on Netty's default behavior: for plaintext (non-TLS) connections, there is no enforced deadline on receiving the complete HTTP/2 client preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n` + SETTINGS frame). The server's Netty pipeline will wait indefinitely for the remainder of the preface bytes.

`NettyProperties` (lines 11–15 of `NettyProperties.java`) exposes only `maxConcurrentCallsPerConnection`; no preface/handshake timeout field exists anywhere in the configuration model. A full-codebase grep for `handshakeTimeout`, `maxConnectionIdle`, `maxConnectionAge`, and `prefaceTimeout` returns zero matches. No `application.yml` resource file exists under `grpc/src/main/resources/` that could supply these values at runtime.

**Why `maxConcurrentCallsPerConnection = 5` does not help:** This limit applies to fully-established gRPC streams on a connection that has already completed the HTTP/2 handshake. A connection stalled mid-preface has zero active streams; the limit is never evaluated. Each such connection still holds an open file descriptor.

**Exploit flow:**
1. Attacker opens a raw TCP connection to port 5600.
2. Sends only the first few bytes of the HTTP/2 client preface (e.g., `PRI * HTTP/2.0\r\n`) and then stops writing.
3. The Netty pipeline waits for the remaining bytes; the connection is held open, consuming one file descriptor.
4. Attacker repeats steps 1–3 in a tight loop from one or more source IPs.
5. When the process's file-descriptor limit is reached, all new `accept()` calls fail; legitimate clients receive connection-refused or timeout errors.

### Impact Explanation
File-descriptor exhaustion causes a complete denial of service for the gRPC API (port 5600). The mirror node's `ConsensusService` (topic subscriptions) and `NetworkService` become unreachable. The `GrpcHighFileDescriptors` Prometheus alert (threshold 80%) confirms the project itself tracks this resource as a critical signal, meaning the attack surface is operationally recognized. Severity: **High** (availability impact, no authentication required, no application-layer data needed).

### Likelihood Explanation
Preconditions are minimal: network reachability to port 5600 and the ability to open TCP sockets. No credentials, no protocol knowledge beyond the first bytes of the HTTP/2 preface, and no rate-limiting at the application layer. The attack is trivially scriptable (`nc`, `python socket`, etc.) and repeatable. In deployments where port 5600 is exposed directly (e.g., the docker-compose setup with nginx only proxying HTTP paths, or Kubernetes `NodePort`/`LoadBalancer` services), the server is directly reachable. Even behind the GCP gateway (20 s `timeoutSec`), the gateway itself may hold the upstream connection open until its own timeout fires, still tying up server-side file descriptors for up to 20 seconds per connection — enough to exhaust descriptors with modest parallelism.

### Recommendation
In `grpcServerConfigurer`, add explicit timeout guards to `NettyServerBuilder`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // Bound the HTTP/2 handshake/preface phase
    serverBuilder.handshakeTimeout(
        nettyProperties.getHandshakeTimeout().toMillis(), TimeUnit.MILLISECONDS);
    // Evict idle connections to reclaim file descriptors
    serverBuilder.maxConnectionIdle(
        nettyProperties.getMaxConnectionIdle().toSeconds(), TimeUnit.SECONDS);
};
```

Add `handshakeTimeout` (suggested default: 10 s) and `maxConnectionIdle` (suggested default: 60 s) fields to `NettyProperties`. Additionally, enforce a per-IP connection rate limit at the infrastructure layer (e.g., iptables `--hashlimit`, nginx `limit_conn`) as a defense-in-depth measure.

### Proof of Concept
```python
import socket, time, threading

TARGET = ("grpc-host", 5600)
# HTTP/2 client preface prefix — intentionally incomplete
PARTIAL_PREFACE = b"PRI * HTTP/2.0\r\n\r\n"

def hold_connection():
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect(TARGET)
    s.sendall(PARTIAL_PREFACE)   # send only prefix, never the SETTINGS frame
    time.sleep(3600)             # hold open for 1 hour
    s.close()

threads = [threading.Thread(target=hold_connection) for _ in range(5000)]
for t in threads:
    t.daemon = True
    t.start()
for t in threads:
    t.join()
```
