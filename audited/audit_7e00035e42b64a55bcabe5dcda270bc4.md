### Title
Unbounded Unauthenticated Pagination Traversal on `/accounts/:id/nfts` Enables Sustained DB Resource Exhaustion

### Summary
The `GET /accounts/:idOrAliasOrEvmAddress/nfts` endpoint in the REST API has no per-client rate limiting. Each page traversal via the cursor-based `next` link generated by `getPaginationLink()` triggers a fresh multi-union DB query and full response serialization. An unprivileged external user can continuously follow pagination links at maximum `limit`, sustaining elevated database load indefinitely without triggering any brute-force detection because each request uses a distinct, legitimate-looking cursor URL.

### Finding Description
**Code path:**

- Route registration: `rest/routes/accountRoute.js` line 15 — `router.getExt(getPath('nfts'), AccountController.getNftsByAccountId)` — no authentication or rate-limit middleware is applied.
- Handler: `rest/controllers/accountController.js` lines 90–103 — `getNftsByAccountId` performs two DB operations per request: `EntityService.getEncodedId()` (entity lookup) and `NftService.getNfts(query)` (NFT query), then serializes all results.
- Query builder: `rest/service/nftService.js` lines 84–126 — `getQuery()` can produce a three-part `UNION ALL` query (lower + inner + upper sub-queries, each with a `LEFT JOIN entity`) when bounds are set, which is the case for every page after the first.
- Pagination link: `rest/controllers/baseController.js` lines 195–223 — `getPaginationLink()` emits a cursor URL like `?limit=100&token.id=lt:X&serialnumber=lt:Y`. Each page produces a unique URL, so no response cache hit is possible across pages.

**Root cause / failed assumption:**
The REST API middleware stack (`rest/middleware/index.js`) contains `authHandler`, `metricsHandler`, `requestNormalizer`, `responseCacheHandler`, and `responseHandler` — none of which enforce per-IP or per-client request rate limits. The only throttle mechanism in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) is scoped exclusively to the `web3` contract-call service and is entirely absent from the Node.js REST service. The `authHandler.js` only grants a higher `limit` to authenticated users; it does not restrict unauthenticated ones.

**Exploit flow:**
1. Attacker issues `GET /api/v1/accounts/0.0.XXXXX/nfts?limit=100` (no credentials needed).
2. Response contains `links.next` with a cursor URL (e.g., `?limit=100&token.id=lt:500&serialnumber=lt:99`).
3. Attacker immediately follows the `next` link, triggering a new three-part `UNION ALL` query with `LEFT JOIN entity`.
4. Attacker repeats step 3 in a tight loop across all pages of an account with many NFTs.
5. Because each URL is unique (different cursor values), the response cache provides no protection.
6. Because requests are sequential GET requests to a public read endpoint, no brute-force or anomaly detection fires.

### Impact Explanation
Each page request executes up to three correlated sub-queries against the `nft` table with a `LEFT JOIN entity`, ordered and limited, then serializes up to 100 `NftViewModel` objects. For accounts holding thousands of NFTs across many token types, an attacker can sustain dozens of such queries per second from a single client. This directly increases PostgreSQL CPU, I/O, and connection pool utilization. Because the REST service shares the DB pool with all other API consumers, sustained traversal degrades response times globally. The impact scales with the number of NFTs held by the targeted account and the number of concurrent attacking clients.

### Likelihood Explanation
No credentials, special knowledge, or tooling are required — only a valid account ID (publicly discoverable via the same API) and a simple HTTP client that follows `next` links. The attack is indistinguishable from legitimate bulk data export. It can be sustained indefinitely from a single IP or distributed across many IPs. The absence of any application-layer rate limiting in the REST service makes this trivially repeatable.

### Recommendation
1. Add a per-IP (or per-authenticated-user) rate limiter middleware to the REST Express application, applied globally or specifically to expensive endpoints like `/accounts/:id/nfts`.
2. Enforce a minimum inter-request delay or maximum pages-per-minute for unauthenticated clients at the application layer (not just at the infrastructure/Traefik layer, which is optional and deployment-specific).
3. Consider adding a DB-level query timeout (`statementTimeout`) specifically for NFT multi-union queries to bound worst-case per-request cost.
4. Optionally, cache paginated responses keyed by the full cursor URL with a short TTL to absorb repeated traversals of the same page.

### Proof of Concept
```bash
# Step 1: Get first page
NEXT=$(curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/nfts?limit=100" \
  | jq -r '.links.next')

# Step 2: Follow pagination links in a tight loop
while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  NEXT=$(curl -s "https://<mirror-node>${NEXT}" | jq -r '.links.next')
  # No sleep — each request triggers a fresh multi-union DB query
done
# Repeat from Step 1 continuously across multiple accounts or in parallel
```
Each iteration hits `getNftsByAccountId` → `NftService.getNfts()` → up to 3× `UNION ALL` sub-queries with `LEFT JOIN entity`, with no rate limiting applied at any layer of the REST service code.
