### Title
Unbounded Per-TopicId Subscription Allows Thundering-Herd DB Exhaustion via Unprivileged gRPC Clients

### Summary
`PollingTopicMessageRetriever.retrieve()` creates an independent, indefinitely-repeating DB polling loop for every subscription with no per-topicId or global subscription cap enforced. The only connection-level guard (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening multiple connections. An unprivileged attacker can flood the DB with identical `topic_message` queries, exhausting the connection pool and starving legitimate subscribers of message delivery.

### Finding Description
**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve(filter, true)` → `PollingTopicMessageRetriever.retrieve()`.

In `PollingTopicMessageRetriever.retrieve()` (lines 45–63), every call unconditionally constructs a `PollingContext` and starts a `Flux.defer(() -> poll(context)).repeatWhen(...)` chain with `numRepeats = Long.MAX_VALUE` (line 99) and `frequency = retrieverProperties.getPollingFrequency()` (default 2 s, line 100). Each poll invocation calls `topicMessageRepository.findByFilter(newFilter)` (line 78), which executes a synchronous JPA query (`TopicMessageRepositoryCustomImpl.findByFilter()`, lines 33–61) — one DB round-trip per subscription per poll cycle.

**Root cause — failed assumption:** The design assumes a bounded number of concurrent subscribers. `TopicMessageServiceImpl` tracks only a metric gauge (`subscriberCount`, line 48) with no enforcement. `RetrieverProperties` exposes no `maxSubscribers` field. `GrpcConfiguration` applies `maxConcurrentCallsPerConnection = 5` (line 33) per-connection only; there is no limit on the number of connections.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication required).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs targeting the same `topicId` with `startTime = 0` and no `endTime`/`limit` (so subscriptions never self-terminate).
3. Each subscription enters the throttled polling path: `numRepeats = Long.MAX_VALUE`, polling every 2 s, `maxPageSize = 1000`.
4. Result: 5N independent DB queries every 2 s against the same `topic_message` table partition.
5. `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` (line 58) ensures subscriptions auto-retry on DB errors, sustaining load even under pool pressure.
6. The HikariCP pool (finite, no per-topic quota) is saturated; legitimate subscribers block waiting for a connection and receive no messages.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5`: bypassed by opening additional connections — no total-connection limit exists in the server config.
- `retrieverProperties.getTimeout() = 60s` (line 28 of `RetrieverProperties`): fires only when *no messages are emitted*; a topic with any traffic never triggers this timeout.
- `subscriberCount` (line 48, `TopicMessageServiceImpl`): read-only metric, never compared against a threshold to reject new subscriptions.
- GCP `maxRatePerEndpoint: 250` (charts `values.yaml`): rate-limits new RPC initiations, not sustained open streams.

### Impact Explanation
With N=50 connections, 250 concurrent polling subscriptions generate ~125 identical DB queries per second. The gRPC DB pool (default HikariCP sizing) is exhausted; all legitimate `findByFilter` calls queue or time out. Message delivery for real subscribers stalls for the duration of the attack. Because the attacker's subscriptions auto-retry (`Retry.backoff(Long.MAX_VALUE, ...)`), the effect is sustained indefinitely without re-connection. Applications depending on the HCS stream for time-sensitive operations (e.g., settlement triggers, audit pipelines) are denied service.

### Likelihood Explanation
No authentication, no rate-limit on open streams, and no per-topicId cap make this trivially repeatable. A single commodity machine can hold thousands of idle gRPC HTTP/2 streams. The attack requires only knowledge of a valid `topicId` (publicly enumerable via the REST API) and a gRPC client library. It is fully scriptable and requires zero privileges.

### Recommendation
1. **Enforce a per-topicId subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: maintain a `ConcurrentHashMap<EntityId, AtomicInteger>` of active subscriber counts per topic and reject (with `RESOURCE_EXHAUSTED`) subscriptions that exceed a configurable threshold.
2. **Enforce a global subscription cap**: compare `subscriberCount` against a configurable `maxSubscribers` property before accepting a new subscription.
3. **Add a total-connection limit** to `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a server-wide `maxConcurrentCalls` (distinct from per-connection).
4. **Deduplicate polling for the same topicId**: share a single `Flux` (multicast) among all subscribers to the same topic rather than issuing one DB query per subscriber per cycle.

### Proof of Concept
```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.mirror import consensus_topic_query_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 12345   # any valid topic
CONNECTIONS = 60    # 60 connections × 5 streams = 300 concurrent pollers

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=...,          # shard=0, realm=0, topicNum=TOPIC_NUM
        consensusStartTime=0, # startTime=0 → full history + live
        # no limit, no endTime → subscription never self-terminates
    )
    threads = []
    for _ in range(5):  # 5 concurrent calls per connection
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.daemon = True
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for w in workers: w.start()
# 300 concurrent DB pollers now active; legitimate subscribers starve
```
