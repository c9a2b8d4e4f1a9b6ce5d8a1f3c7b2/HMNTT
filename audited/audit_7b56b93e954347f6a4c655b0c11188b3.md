### Title
Silent Fallback to `BlockType.LATEST` in `OpcodeServiceImpl` When Record File Is Missing Produces Incorrect Opcode Traces

### Summary
In `OpcodeServiceImpl.buildCallServiceParameters()`, when `recordFileService.findByTimestamp(consensusTimestamp)` returns `Optional.empty()`, the code silently falls back to `BlockType.LATEST` (block number `Long.MAX_VALUE`) instead of failing with an error. This causes the opcode trace re-execution to use the latest EVM version rather than the EVM version active at the time of the original transaction. Any unprivileged user who can reference a transaction whose record file has been pruned from the database (while the transaction and contract result records remain) can trigger this incorrect behavior.

### Finding Description

**Exact code location:**

`OpcodeServiceImpl.java`, lines 158–161:
```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
```

`RecordFileRepository.findByTimestamp()` (line 48):
```java
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
```

**Root cause:** The query returns `Optional.empty()` whenever no record file row has `consensusEnd >= consensusTimestamp`. This legitimately occurs when record files are pruned from the mirror node database under a shorter retention policy than transaction/contract-result rows. The `.orElse(BlockType.LATEST)` treats this data-absent condition as a normal case and silently substitutes `BlockType.LATEST` (whose `number` field is `Long.MAX_VALUE`).

**Failed assumption:** The code assumes that if a `ContractResult` row exists for a `consensusTimestamp`, a corresponding `RecordFile` row will always be present. This assumption breaks under differential retention policies (record files pruned, contract results retained).

**Exploit flow:**
1. The attacker supplies a valid transaction hash or transaction ID to `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes`.
2. `contractTransactionHashRepository.findByHash()` or `transactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc()` succeeds — the transaction row is still present.
3. `contractResultRepository.findById(consensusTimestamp)` succeeds — the contract result row is still present.
4. `recordFileService.findByTimestamp(consensusTimestamp)` returns `Optional.empty()` — the record file row has been pruned.
5. `blockType` is silently set to `BlockType.LATEST`.
6. `ContractDebugParameters` is built with `block = BlockType.LATEST`.
7. `ContractDebugService.processOpcodeCall()` sets `ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1))` correctly for state lookups, but the EVM version is selected based on `block = BlockType.LATEST`, i.e., the current latest EVM version, not the historical one.

**Why existing checks are insufficient:**
- `contractResultRepository.findById()` throws `EntityNotFoundException` if the contract result is missing, but there is **no equivalent guard** for the record file lookup.
- The `OpcodesController` has no authentication requirement — the endpoint is publicly accessible to any caller who can supply a valid transaction identifier.
- The throttle check (`throttleManager.throttleOpcodeRequest()`) limits rate but does not prevent the incorrect execution path.
- `ContractDebugService.validateResult()` suppresses `MirrorEvmTransactionException` with only a warning log, so even a failed re-execution under the wrong EVM version does not surface as an error to the caller.

### Impact Explanation
The opcode trace is re-executed under the wrong EVM version (latest instead of historical). This can produce:
- Opcodes that did not exist at the time of the original transaction appearing in the trace.
- Different gas costs per opcode, making the trace inconsistent with the original execution.
- Different execution outcomes if EVM semantics changed between the historical block and the latest block.

Tooling (debuggers, auditors, forensic analysis tools) that consume this endpoint will receive a silently incorrect trace with no indication that the wrong EVM version was used. No funds are directly at risk, consistent with the medium-severity scope definition.

### Likelihood Explanation
Mirror node operators routinely configure shorter retention windows for record files than for transaction/contract-result data. Any transaction whose record file falls outside the retention window but whose contract result is still present creates a permanently triggerable instance of this bug. No credentials, elevated permissions, or special network access are required — only knowledge of a valid transaction hash or ID, which is publicly observable on-chain. The condition is repeatable and deterministic once the record file is pruned.

### Recommendation
Replace the silent `.orElse(BlockType.LATEST)` fallback with an explicit error:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
```

This aligns with the existing pattern used for `contractResultRepository.findById()` (line 154–156 of `OpcodeServiceImpl.java`) and ensures callers receive a clear 404 rather than a silently incorrect trace. Additionally, document the expected data retention invariant (record files must be retained at least as long as contract results) and add a monitoring alert for divergence.

### Proof of Concept
1. Identify a historical contract transaction hash `H` whose `consensusTimestamp` is `T`.
2. Confirm `contract_result` row for `T` exists in the mirror node DB.
3. Prune (or wait for natural expiry of) the `record_file` row covering `T` (i.e., where `consensus_end >= T`).
4. Send:
   ```
   GET /api/v1/contracts/results/H/opcodes
   Accept-Encoding: gzip
   ```
5. Observe: HTTP 200 response with an opcode trace. The `block` used internally is `BlockType.LATEST` (Long.MAX_VALUE), not the historical block index.
