### Title
Opcode Replay Endpoint Bypasses Gas-Based Throttle, Enabling CPU Exhaustion via Maximum-Gas EVM Replay

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint applies only a request-rate throttle (default: 1 req/sec) and performs no gas-based throttle check or `maxGasLimit` validation before triggering a full EVM replay with opcode tracing. The gas limit used for replay is taken directly from the stored `contractResult.getGasLimit()` (up to the protocol maximum of 15,000,000), bypassing the gas-bucket throttle that protects the regular `/contracts/call` endpoint. A single unauthenticated request referencing a high-gas transaction with `stack=true&memory=true&storage=true` can cause a disproportionate CPU spike.

### Finding Description

**Entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

The only guard before calling `opcodeService.processOpcodeCall()` is `throttleManager.throttleOpcodeRequest()`.

**Throttle implementation** — `ThrottleManagerImpl.throttleOpcodeRequest()`: [2](#0-1) 

This consumes exactly 1 token from `opcodeRateLimitBucket` (default capacity: 1/sec). It does **not** consume from `gasLimitBucket` and performs no gas-limit validation.

**Contrast with the regular call endpoint** — `ContractController`: [3](#0-2) [4](#0-3) 

`/contracts/call` calls both `throttleManager.throttle(request)` (which consumes from `gasLimitBucket` proportional to gas) and `validateContractMaxGasLimit()` (which enforces `maxGasLimit = 15,000,000`). Neither check exists in the opcode path.

**Gas limit sourced from stored record without cap** — `OpcodeServiceImpl.getGasLimit()`: [5](#0-4) 

The gas passed to the EVM replay is `contractResult.getGasLimit()` (or `ethereumTransaction.getGasLimit()`), taken verbatim from the mirror node database. No ceiling is applied.

**Full EVM replay with tracing is triggered**: [6](#0-5) 

`contractDebugService.processOpcodeCall(params, opcodeContext)` executes a complete EVM replay. The controller's own Javadoc acknowledges: *"the transaction needs to be re-executed on the EVM, which may take a significant amount of time to complete if stack and memory information is requested."* [7](#0-6) 

**Root cause**: The opcode endpoint was given a separate, request-count-only throttle bucket (`opcodeRateLimitBucket`) and was never wired to the gas-based throttle (`gasLimitBucket`) or the `maxGasLimit` guard. The assumption that 1 req/sec is sufficient protection fails because each request can carry up to 15 M gas of EVM work plus per-opcode stack/memory/storage capture — orders of magnitude more CPU than a typical contract call.

### Impact Explanation
A single opcode request replaying a 15 M-gas transaction with `stack=true&memory=true&storage=true` forces the server to:
1. Execute every EVM opcode up to the gas limit.
2. Capture and serialize the full stack, memory, and storage state at each step.

This is the most CPU-intensive operation the web3 module can perform. On a node whose 24-hour baseline consists of lightweight read queries or moderate contract calls, one such replay can easily exceed a 30% CPU increase. Because the gas bucket is never debited, the regular `/contracts/call` endpoint's gas throttle is also unaffected, meaning both workloads can run concurrently at full capacity.

### Likelihood Explanation
- The endpoint requires no authentication or privileged role.
- The only precondition is that the opcode feature is enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`) and the attacker knows a transaction ID or hash for a high-gas transaction — both are publicly observable on-chain.
- An attacker can submit a single high-gas Hedera contract transaction (paying a small network fee), obtain its transaction ID from the public mirror REST API, and then repeatedly call the opcode endpoint at 1 req/sec indefinitely.
- The attack is trivially repeatable and requires no special tooling beyond `curl`.

### Recommendation
1. **Apply gas-based throttling to opcode requests**: In `ThrottleManagerImpl.throttleOpcodeRequest()`, also consume from `gasLimitBucket` using the gas limit of the target transaction (requires passing the gas value, or reading it before throttling).
2. **Enforce `maxGasLimit` in the opcode path**: In `OpcodeServiceImpl.buildCallServiceParameters()` (or in `OpcodesController`), cap the gas value returned by `getGasLimit()` to `evmProperties.getMaxGasLimit()`.
3. **Cap opcode-specific gas**: Introduce a separate configurable `maxOpcodeGasLimit` property (e.g., defaulting to a fraction of `maxGasLimit`) to limit the EVM work per opcode replay independently of the regular call limit.
4. **Restore gas to bucket after opcode replay**: Ensure `restoreGasToBucket` in `ContractCallService.doProcessCall()` is meaningful for the opcode path by first consuming from `gasLimitBucket`.

### Proof of Concept
```
# Step 1: Find or submit a Hedera contract transaction with gas_limit = 15,000,000
# The transaction ID is publicly visible via the mirror REST API:
#   GET /api/v1/transactions?transactiontype=CONTRACTCALL&order=desc

# Step 2: Send the opcode replay request (no auth required):
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/0.0.12345-1234567890-000000000/opcodes?stack=true&memory=true&storage=true" \
  --output response.gz

# Step 3: Repeat at 1 req/sec (the maximum allowed rate):
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/0.0.12345-1234567890-000000000/opcodes?stack=true&memory=true&storage=true" \
    --output /dev/null
  sleep 1
done

# Result: Sustained full-EVM replays at 15M gas each with full tracing,
# bypassing the gas-based throttle, causing sustained elevated CPU usage.
```
