### Title
Unbounded Request Rate on GraphQL Endpoint Enables Sustained DB Load via Near-Maximum Complexity Queries

### Summary
The `graphQlCustomizer()` bean in `GraphQlConfiguration.java` configures `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` as the sole execution guards, but imposes no per-IP or global request-rate limit on the GraphQL endpoint. An unauthenticated attacker can repeatedly submit queries at complexity=199 — each triggering multiple database lookups via the schema's self-referential `Account` fields — at an unbounded rate, causing sustained database load that degrades response times for all legitimate users. The `CachedPreparsedDocumentProvider` only caches parsing/validation, not execution, so every request still hits the database.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, function `graphQlCustomizer()`, lines 42–49:

```java
@Bean
GraphQlSourceBuilderCustomizer graphQlCustomizer(PreparsedDocumentProvider provider) {
    var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
    var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
    var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);
    return b -> b.configureGraphQl(
            graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
}
```

**Root cause:** `MaxQueryComplexityInstrumentation` bounds the *work per single request* but places no constraint on *how many requests per second* any client may submit. The `web3` module has a `ThrottleConfiguration` (bucket4j-based, `requestsPerSecond=500`) but the `graphql` module has no equivalent — no rate-limiting filter, no IP-based throttle, no authentication requirement.

**Schema amplification:** The `Account` type in `account.graphqls` and `common.graphqls` contains three self-referential object fields:
- `autoRenewAccount: Account`
- `stakedAccount: Account`
- `obtainer: Accountable` (resolved to `Account`)

With `MaxQueryDepthInstrumentation(10)`, a query can nest up to 10 levels deep. Each nested `Account` object field causes `EntityService` to issue a separate database query (N+1 pattern). A query at complexity=199 requesting many scalar fields plus nested Account objects can trigger 5–10 DB queries per request.

**Caching does not mitigate execution:** `CachedPreparsedDocumentProvider` (keyed on the raw query string) caches only the parsed `PreparsedDocumentEntry`. The `MaxQueryComplexityInstrumentation` runs as an execution-phase instrumentation on every request, and the data fetchers (`EntityServiceImpl` → `EntityRepository`) execute against the database on every request regardless of the parse cache.

**No authentication or connection limit:** `LoggingFilter` is the only HTTP-layer filter in the graphql module — it only logs. There is no `SecurityFilterChain`, no IP-based throttle, and no connection concurrency cap on the `/graphql/alpha` endpoint.

### Impact Explanation

An attacker sending a sustained flood of complexity=199 queries forces the server to execute near-maximum field resolution and up to ~10 database queries per request. With no rate limit, a single attacker machine can saturate the database connection pool (`hikaricp_connections_max`), causing legitimate queries to queue or time out. The `statementTimeout=10000ms` only limits individual query duration, not the number of concurrent queries. This degrades response times for all users sharing the same GraphQL service instance. Severity matches the stated scope: griefing/DoS with no economic damage to the network itself, but measurable service degradation.

### Likelihood Explanation

No privileges, credentials, or special network position are required. The endpoint `/graphql/alpha` is publicly documented and accessible. The attack requires only a script sending repeated HTTP POST requests — trivially reproducible with `curl`, `ab`, `wrk`, or any HTTP load tool. The fixed query string means the attacker can reuse the same payload indefinitely. The attack is repeatable, scalable across multiple source IPs, and requires no knowledge of internal state.

### Recommendation

1. **Add a request-rate limiter to the graphql module** mirroring the `web3` module's `ThrottleConfiguration` (bucket4j). Apply it as a `OncePerRequestFilter` or Spring Security filter on `/graphql/**`, limiting requests per IP per second.
2. **Apply complexity-weighted rate limiting**: instead of a flat request count, consume tokens proportional to the query's computed complexity score so high-complexity queries consume more budget.
3. **Add a `@field:RateLimit` or connection-level concurrency cap** to prevent a single client from holding many concurrent in-flight queries.
4. **Consider persisted queries / allowlist**: restrict execution to a known set of pre-approved query documents, eliminating arbitrary complexity crafting entirely.

### Proof of Concept

```bash
# Craft a query at complexity=199 using nested self-referential Account fields
# (each field counts as 1 complexity point; depth capped at 10 by MaxQueryDepthInstrumentation)
QUERY='{"query":"{ account(input:{entityId:{num:1}}) { id alias balance memo deleted declineReward nonce receiverSigRequired maxAutomaticTokenAssociations stakePeriodStart createdTimestamp expirationTimestamp key type entityId{shard realm num} timestamp{from to} autoRenewAccount { id alias balance memo deleted declineReward nonce receiverSigRequired maxAutomaticTokenAssociations stakePeriodStart createdTimestamp expirationTimestamp key type entityId{shard realm num} timestamp{from to} autoRenewAccount { id alias balance memo deleted declineReward nonce receiverSigRequired maxAutomaticTokenAssociations stakePeriodStart createdTimestamp expirationTimestamp key type entityId{shard realm num} timestamp{from to} } } } }"}'

# Send at high rate with no authentication — no 429 or throttle response will be returned
while true; do
  curl -s -o /dev/null -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
```

**Expected result:** Server accepts all requests (HTTP 200), each triggering multiple DB queries. Database connection pool exhaustion and rising query latency observable via the `GraphQLHighDBConnections` and `GraphQLQueryLatency` Prometheus alerts defined in `charts/hedera-mirror-graphql/values.yaml`.
