### Title
Unauthenticated Metrics Endpoint Exposes Per-Route Error Rates and Latency Profiles When `authentication` Is Disabled

### Summary
When `config.metrics.config.authentication` is set to `false`, the `authenticate()` function in `metricsMiddleware` unconditionally returns `true`, granting any unauthenticated caller full read access to the Prometheus metrics endpoint. The endpoint exposes aggregate success/error counters (`api_all_success`, `api_all_errors`) as well as per-route counters and duration histograms labeled by `{method, path, code}`, allowing an attacker to profile every REST endpoint's error rate and latency without any credentials.

### Finding Description
**File:** `rest/middleware/metricsHandler.js`

**Root cause — `authenticate()`, lines 137–144:**
```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ← unconditional bypass; no further check
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```
When `authentication === false` the function short-circuits and returns `true` before ever inspecting the request's `Authorization` header.

**Gate in `metricsMiddleware`, lines 156–165:**
```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) {          // always false when auth disabled
    res.set('WWW-Authenticate', 'Basic realm="Metrics"');
    return res.status(401).send('Unauthorized');
  }
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));   // full dump
  });
}
```
The 401 branch is dead code when `authentication` is `false`; every request reaches `exporter.collect()` and receives the full Prometheus text dump.

**What the dump contains (lines 67–105, 195–210):**
- `api_all_success_total` / `api_all_errors_total` — global success vs. error ratio
- `api_all_client_error_total` / `api_all_server_error_total` — 4xx vs. 5xx split
- `api_request_total{method, path, code}` — per-route request counts broken down by HTTP status
- `api_request_duration_milliseconds{method, path, …}` — per-route latency histogram
- `api_request_size_bytes` / `api_response_size_bytes` — per-route payload sizes
- `nodejs_process_memory_*` / `nodejs_process_cpu_usage_percentage` — process resource gauges

The per-route `api_request_total` and `api_request_duration_milliseconds` instruments are the most operationally sensitive: they reveal which endpoints are slowest and which return the most 5xx errors, directly identifying the highest-value DoS targets.

### Impact Explanation
An attacker with network access to the REST API port can:
1. Enumerate every active API route and its exact error-code distribution.
2. Identify the slowest endpoints (highest `p99` latency bucket) and flood them to exhaust the DB connection pool (`maxConnections: 10` default).
3. Observe `api_all_request_in_processing_total` (in-flight gauge) to time burst attacks for maximum queue saturation.
4. Read process memory/CPU gauges to detect when the node is already under stress and amplify at the worst moment.

The information is not directly harmful in isolation, but it converts a blind DoS attempt into a targeted, efficient one. Severity: **Medium** (information disclosure enabling amplified DoS).

### Likelihood Explanation
- The default for `authentication` is `true` per documentation, so this requires an operator to have explicitly set it to `false`.
- However, the setting is documented and operators may disable it in internal/staging environments or behind a load balancer they believe provides network-level isolation.
- Exploitation requires only a single unauthenticated HTTP GET — no special tooling, no credentials, no prior knowledge beyond the default `uriPath` (`/swagger/metrics/`).
- Repeatability is unlimited; the endpoint has no rate limiting of its own.

### Recommendation
1. **Remove the unconditional bypass.** The `if (!authentication) { return true; }` pattern makes disabling auth a global, irrevocable grant. Replace it with an explicit allowlist of trusted source IPs, or remove the `authentication = false` code path entirely and require credentials always.
2. **Scope the metrics endpoint to a separate, non-public port** (e.g., a dedicated management port not exposed to the internet), independent of the `authentication` flag.
3. **Add rate limiting** to the metrics path even when authentication is enabled, to prevent scrape-flooding.

### Proof of Concept
```bash
# Precondition: hiero.mirror.rest.metrics.config.authentication = false
# Default uriPath = /swagger → metrics path = /swagger/metrics/

# 1. Fetch full metrics dump without any credentials
curl -s http://<mirror-node-host>:5551/swagger/metrics/

# Expected: HTTP 200 with Prometheus text output including:
#   api_all_success_total ...
#   api_all_errors_total ...
#   api_request_total{method="GET",path="/api/v1/transactions",code="500"} ...
#   api_request_duration_milliseconds_bucket{...} ...

# 2. Parse slowest endpoints
curl -s http://<host>:5551/swagger/metrics/ \
  | grep 'api_request_duration' \
  | sort -t= -k4 -rn \
  | head -20

# 3. Use identified slow endpoint for targeted DoS
for i in $(seq 1 500); do
  curl -s "http://<host>:5551/api/v1/<slowest-endpoint>" &
done
