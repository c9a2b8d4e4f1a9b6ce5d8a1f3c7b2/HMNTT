### Title
Unbounded Per-Request Database Query via Opaque EVM Address Lookup with No Service-Layer Caching

### Summary
`getEntityIdFromEvmAddress()` in `rest/service/entityService.js` issues a direct PostgreSQL query on every invocation with no result caching at the service layer. Because `getEncodedId()` unconditionally calls it whenever `entityId.evmAddress !== null`, and because the REST API has no rate-limiting middleware, an unauthenticated attacker can exhaust the database connection pool (default: 10 connections) by flooding any EVM-address-accepting endpoint with requests, causing a full REST API denial of service.

### Finding Description

**Exact code path:**

`getEncodedId()` dispatches to `getEntityIdFromEvmAddress()` for every opaque EVM address: [1](#0-0) 

`getEntityIdFromEvmAddress()` issues a raw DB query with no caching: [2](#0-1) 

The query executed is: [3](#0-2) 

**Root cause — three independent caching layers all fail to cover this path:**

1. **`parseCached` LRU cache** (`rest/entityId.js` lines 301–332) caches only the parsed `EntityId` object (the `evmAddress` string field). It does not cache the DB lookup result. Even on a cache hit, `getEntityIdFromEvmAddress` is still called. [4](#0-3) 

2. **Redis response cache** (`rest/middleware/responseCacheHandler.js`) only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 `NotFoundError` thrown for a non-existent EVM address is never cached, so every repeated request for the same non-existent address re-hits the DB. [5](#0-4) 

3. **No rate limiting in the REST API.** `server.js` registers no throttle or rate-limit middleware. The throttle infrastructure exists only in the separate web3 Spring Boot module. [6](#0-5) 

**DB pool constraint:** The default maximum is 10 connections. [7](#0-6) 

### Impact Explanation

With 10 concurrent connections available, an attacker sending ≥10 simultaneous requests containing syntactically valid but non-existent opaque EVM addresses (e.g., `GET /api/v1/accounts/0xdeadbeef...`) will hold all pool connections in pending DB queries. Legitimate requests queue indefinitely or time out (default statement timeout: 20 s). Every REST API endpoint that calls `getEncodedId()` is affected — accounts, contracts, tokens, topics, transactions, balances — making this a full-service outage. No authentication is required.

### Likelihood Explanation

The attack requires zero privileges and zero knowledge of the system state. Any syntactically valid 40-hex-character string that does not match the long-zero prefix pattern is treated as an opaque EVM address and triggers a DB query. A single attacker with a modest HTTP flood tool can sustain the condition indefinitely. The attack is repeatable and trivially scriptable.

### Recommendation

1. **Add a negative-result cache at the service layer** inside `getEntityIdFromEvmAddress()`: cache `null` results (keyed by the hex EVM address) for a short TTL (e.g., 30–60 s) using the existing in-process LRU or Redis infrastructure.
2. **Cache 404 responses** in the Redis response cache for a short TTL (e.g., 5 s) so repeated requests for the same non-existent address are served without hitting the DB.
3. **Add request-rate limiting** to the REST API (e.g., `express-rate-limit`) at the server level, independent of the web3 throttle.
4. **Increase the DB pool size** or add a queue depth limit so pool exhaustion does not translate directly to a full outage.

### Proof of Concept

```bash
# Send 20 concurrent requests with unique non-existent opaque EVM addresses
# (non-long-zero: first 12 bytes are not all zeros)
for i in $(seq 1 20); do
  addr=$(printf 'deadbeef%032x' $i)
  curl -s "http://<mirror-node-rest>/api/v1/accounts/0x${addr}" &
done
wait
# Result: DB connection pool exhausted; subsequent legitimate requests
# receive connection timeout errors or HTTP 503 responses.
```

Each request matches `evmAddressRegex`, passes `EntityId.isValidEntityId()`, produces an `EntityId` with `evmAddress !== null`, and unconditionally invokes `getEntityIdFromEvmAddress()` → DB query → no row found → `NotFoundError` (404, not cached) → connection released. Sustained at ≥10 RPS, the pool stays saturated.
