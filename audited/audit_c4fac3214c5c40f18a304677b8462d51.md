### Title
Liveness Probe Bypass via Unauthenticated `/tmp/alive` mtime Spoofing

### Summary
The `healthcheck` binary's `live` mode exclusively checks the modification time of `/tmp/alive` without validating file content, ownership, or any proof that the pinger process is actually running. Any process executing as the same UID (1000) within the pod — or any user if the file does not yet exist — can touch or recreate `/tmp/alive` with a fresh mtime, causing Kubernetes to believe the pinger is alive when it is not. This permanently masks a real DoS condition against the network health monitor.

### Finding Description
**Code location:** `pinger/cmd/healthcheck/main.go`, `main()`, lines 27–38.

```go
fi, err := os.Stat("/tmp/alive")          // line 27 — only fetches metadata
...
age := time.Since(fi.ModTime())           // line 34 — only mtime is evaluated
if age > 2*time.Minute {                  // line 35
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)                                // passes if mtime < 2 min
```

The pinger (`pinger/main.go`, line 36) writes `/tmp/alive` with permissions `0644` every 15 seconds:

```go
_ = os.WriteFile("/tmp/alive", []byte(time.Now().Format(time.RFC3339Nano)), 0644)
```

The Dockerfile (`pinger/Dockerfile`, line 42) runs the container as `USER 1000:1000` on a `scratch` base. In Kubernetes, `/tmp` is typically a world-writable tmpfs mount.

**Root cause / failed assumption:** The design assumes only the pinger goroutine can update `/tmp/alive`. This assumption fails because:
- `/tmp` is world-writable; any user can **create** `/tmp/alive` if it does not exist.
- The file is `0644`; any process running as UID 1000 (same user, e.g. a sidecar, a `kubectl exec` shell, or any injected process) can **overwrite** it.
- The healthcheck performs zero content validation, no PID binding, no HMAC/secret, and no ownership check.

**Exploit flow:**
1. Pinger process dies (crash, OOM, network partition — a real DoS event).
2. Attacker (any process at UID 1000, or any user before the file is first created) runs:
   ```sh
   while true; do touch /tmp/alive; sleep 60; done
   ```
3. Every Kubernetes liveness probe invocation calls `healthcheck live`, which stats `/tmp/alive`, sees mtime < 2 minutes, and exits 0.
4. Kubernetes never triggers a pod restart; the pinger remains dead indefinitely.

**Why existing checks are insufficient:**
- `os.Stat` returns only filesystem metadata; it cannot distinguish a pinger-written file from an attacker-written one.
- There is no PID file, no secret token, no HMAC, and no ownership assertion anywhere in the probe path.
- The 2-minute staleness window (line 35) is irrelevant once the attacker refreshes mtime every ~60 seconds.

### Impact Explanation
The pinger is the active liveness monitor for the Hedera network: it submits tinybar transfers and verifies mirror-node propagation. If it dies silently (masked by this bypass), the entire monitoring pipeline stops without any alerting or automatic recovery. A real network-level DoS — node outage, consensus failure, mirror-node lag — goes undetected for an unbounded duration. Given the pinger's role in a network with ≥25% market-cap weight, suppressing its health signal has direct operational and reputational consequences.

### Likelihood Explanation
Precondition is code execution inside the pod at UID 1000. This is achievable via:
- `kubectl exec` by any cluster user with `exec` RBAC permission (common in dev/staging clusters).
- A compromised sidecar container sharing the pod's process namespace or a shared emptyDir volume.
- A supply-chain or dependency vulnerability in the pinger binary itself that allows arbitrary command execution before the main loop exits.

The attack is trivially repeatable with a one-liner shell loop, requires no privileges beyond pod-exec, and leaves no distinguishable trace in application logs.

### Recommendation
Replace the mtime-only check with a content-authenticated heartbeat:

1. **PID + secret token**: On startup, pinger writes a random secret to a process-private location (e.g. an in-memory pipe or a file readable only by UID 1000 with mode `0600`). The heartbeat file contains `HMAC(secret, timestamp)`. The healthcheck verifies the HMAC before accepting the probe.
2. **Minimum viable fix**: Change the file mode from `0644` to `0600` (`pinger/main.go` line 36) so only UID 1000 can write, AND write the pinger's own PID into the file. The healthcheck should then:
   - Read the file content.
   - Parse the PID.
   - Verify `/proc/<PID>/exe` resolves to the pinger binary.
   - Only then accept the probe.
3. **Kubernetes-native alternative**: Replace the exec probe with a local HTTP liveness endpoint served by the pinger itself; the endpoint is only reachable while the pinger's event loop is running.

### Proof of Concept
```sh
# 1. Simulate pinger death (in a real attack, pinger crashes on its own)
kill $(pgrep pinger)

# 2. From any shell inside the pod running as UID 1000 (e.g. kubectl exec):
while true; do
  touch /tmp/alive          # updates mtime; content irrelevant
  sleep 60
done &

# 3. Verify the liveness probe still passes:
/healthcheck live; echo "exit: $?"
# Output: exit: 0   <-- probe passes despite pinger being dead

# 4. Confirm pinger is not running:
pgrep pinger || echo "pinger is dead"
# Output: pinger is dead

# Result: Kubernetes never restarts the pod; DoS condition is permanently masked.
