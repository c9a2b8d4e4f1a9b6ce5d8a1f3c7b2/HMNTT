### Title
Unauthenticated IP Spoofing via Unvalidated `X-Forwarded-For` Header in `getClientIpAddress()`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the raw `X-Forwarded-For` header value without any parsing, validation, or format checking. When the Rosetta service is accessed directly (bypassing the nginx proxy, which is possible since port 5700 is exposed in `docker-compose.yml`), any unprivileged attacker can inject an arbitrary string — including a comma-separated list like `victim_ip, attacker_ip` — which is logged verbatim as the client IP, corrupting transaction history attribution in audit logs.

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64: check X-Real-IP first

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67: raw header, no parsing
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // line 71: fallback
    }

    return ipAddress  // line 74: returned verbatim
}
```

**Root cause:** `r.Header.Get(xForwardedForHeader)` returns the raw header string. There is no call to `net.ParseIP()`, no splitting on commas, and no length/format validation. The returned value is interpolated directly into the log message at line 52–53:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

**Why the nginx mitigation is insufficient:** The nginx proxy (`docker-compose.yml` line 204–205) sets both `X-Real-IP $remote_addr` and `X-Forwarded-For $proxy_add_x_forwarded_for`. Because `getClientIpAddress()` checks `X-Real-IP` first (line 64), requests routed through nginx are protected. However, `docker-compose.yml` line 151 exposes port 5700 directly:

```yaml
ports:
  - 5700:5700
```

A client connecting directly to port 5700 bypasses nginx entirely. In that path, `X-Real-IP` is never set, the code falls through to line 67, and the attacker-controlled `X-Forwarded-For` value is used raw. The Kubernetes/Traefik deployment (`charts/hedera-mirror-rosetta/values.yaml`) also does not configure Traefik to inject `X-Real-IP`, leaving the same gap.

**Tests confirm no validation exists:** `trace_test.go` lines 56–58 only test a well-formed single IP in `X-Forwarded-For`; no test covers comma-separated lists or malformed values.

### Impact Explanation
An attacker can make their Rosetta API calls appear to originate from any IP address — including a known victim's IP — in the structured log output. Since Rosetta is a blockchain transaction API, operators and forensic tools use these logs to attribute which IP submitted which construction or account query. Injecting `victim_ip` as the logged source falsely implicates the victim in transaction activity. Injecting `attacker_ip, victim_ip` produces a malformed log token that breaks log parsers relying on space-delimited fields, causing silent attribution failures or parse errors in SIEM/audit pipelines. This directly maps to the stated scope: "Reorganizing transaction history without direct theft of funds."

### Likelihood Explanation
Preconditions are minimal: the attacker needs only network access to port 5700 (exposed by default in `docker-compose.yml`) and the ability to set HTTP headers — both trivially satisfied by any HTTP client. No authentication, credentials, or special privileges are required. The attack is repeatable, stateless, and leaves no trace of the real source IP in logs. Any operator who has not firewalled port 5700 is exposed.

### Recommendation
1. **Parse the first IP from `X-Forwarded-For`** (the leftmost, client-supplied entry) rather than using the raw string:
   ```go
   xff := r.Header.Get(xForwardedForHeader)
   if len(xff) > 0 {
       parts := strings.SplitN(xff, ",", 2)
       ipAddress = strings.TrimSpace(parts[0])
   }
   ```
2. **Validate the result with `net.ParseIP()`** and fall back to `RemoteAddr` if parsing fails.
3. **Firewall port 5700** so it is not directly reachable; all traffic should route through the nginx/Traefik proxy that sets a trusted `X-Real-IP`.
4. **In the Kubernetes chart**, configure Traefik to set `X-Real-IP` from the real client address so the proxy-protected path is consistent with the docker-compose deployment.

### Proof of Concept
```bash
# Direct connection to rosetta port, bypassing nginx (port 5700 exposed in docker-compose.yml)
curl -s \
  -H "X-Forwarded-For: 203.0.113.99, 10.0.0.1" \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}' \
  http://<rosetta-host>:5700/network/list

# Resulting log entry (TracingMiddleware, trace.go line 52-53):
# time="..." level=info msg="203.0.113.99, 10.0.0.1 POST /network/list (200) in 12ms"
#
# The entire string "203.0.113.99, 10.0.0.1" is logged as the client IP.
# Log parsers treating the first space-delimited token as the IP will record
# "203.0.113.99," (with trailing comma) — a non-routable, malformed token —
# while the victim IP 203.0.113.99 appears associated with this transaction request.
