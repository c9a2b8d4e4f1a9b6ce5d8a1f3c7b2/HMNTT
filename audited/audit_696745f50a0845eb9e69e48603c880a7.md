### Title
Unauthenticated Full Entity Table Scan via `account.balance=eq:0` Without Timestamp Filter in `getBalances()`

### Summary
When `GET /api/v1/balances` is called without a `timestamp` parameter, `getBalances()` builds an `entity_balance` CTE that unconditionally scans the entire `entity` table for all `ACCOUNT`/`CONTRACT` rows. The `account.balance` filter is applied only in the outer `WHERE` clause after CTE materialization, meaning a request with `account.balance=eq:0&limit=100` forces a full table scan of potentially millions of entity rows, followed by up to 100 correlated subqueries against `token_account`. No authentication or rate limiting exists on this endpoint.

### Finding Description

**Exact code path:** `rest/balances.js`, `getBalances()`, lines 133–149.

When `tsQuery` is falsy (no `timestamp` parameter supplied), the else-branch executes:

```javascript
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')   -- no balance predicate here
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances
  from entity_balance ab
  ${whereClause}          -- balance=0 filter applied here, after full CTE scan
  order by ab.account_id ${order}
  ${limitQuery}`;         -- LIMIT applied to output, not to scan
``` [1](#0-0) 

The `entity_balance` CTE contains no balance predicate. The `balanceQuery` (e.g., `ab.balance = 0`) is only injected into `whereClause` on the outer SELECT. PostgreSQL CTEs are optimization fences in older versions and even in newer versions the planner may not push the predicate into the CTE scan. There is no index on `entity.balance`, so the filter requires a sequential scan of all ACCOUNT/CONTRACT rows.

For each row returned (up to `responseLimit.max`, which is 100 per the OpenAPI spec), the correlated subquery `getTokenAccountBalanceSubQuery` fires against `token_account`:

```javascript
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(...)
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
``` [2](#0-1) 

**Root cause / failed assumption:** The design assumes the CTE will be cheap because the LIMIT bounds the result set. This is false: the LIMIT bounds output rows but not the scan cost. The entire entity table must be read and filtered before the LIMIT is applied.

**Why existing checks fail:**
- `utils.parseLimitAndOrderParams` caps `limit` at `responseLimit.max` (100), bounding output rows but not the underlying scan. [3](#0-2) 
- `balanceFilterValidator` validates the operator/value format but does not block the `eq:0` pattern. [4](#0-3) 
- The throttle infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) applies only to the `web3` module (contract calls), not to the REST API `/balances` endpoint. [5](#0-4) 
- No authentication middleware is applied to `getBalances`.

### Impact Explanation
On a production Hedera mirror node (mainnet has tens of millions of entity rows), each such request forces a full sequential scan of the `entity` table filtered by `type in ('ACCOUNT','CONTRACT')` with no index on `balance`, followed by up to 100 correlated `token_account` subqueries. A sustained flood of these requests from a single IP (or distributed) can saturate DB CPU and I/O, degrading or denying service to all mirror node consumers. There is no economic damage to network participants — this is pure infrastructure griefing.

### Likelihood Explanation
The attack requires zero privileges, zero tokens, and zero on-chain activity. The request is a single unauthenticated HTTP GET. The parameter combination (`account.balance=eq:0`, no `timestamp`) is valid and accepted without error. Any external actor who reads the public OpenAPI spec can construct it. It is trivially repeatable in a loop or via a botnet.

### Recommendation
1. **Push the balance predicate into the CTE** or restructure the query so the balance filter is applied before the LIMIT, ideally with an index on `entity.balance`.
2. **Add an index** on `entity(balance)` (or a partial index for `type in ('ACCOUNT','CONTRACT')`) so balance-filtered queries do not require full table scans.
3. **Apply rate limiting** to the `/api/v1/balances` endpoint (per-IP or global), consistent with the throttle infrastructure already present in the `web3` module.
4. **Consider requiring at least one selective filter** (e.g., `account.id` range or `timestamp`) when `account.balance` is used alone, to prevent unbounded scans.

### Proof of Concept
```
# No authentication required. Repeat at high frequency.
curl "https://<mirror-node-host>/api/v1/balances?account.balance=eq:0&limit=100"
```

**What happens in the DB:**
```sql
WITH entity_balance AS (
  SELECT id AS account_id, balance, balance_timestamp AS consensus_timestamp, public_key
  FROM entity
  WHERE type IN ('ACCOUNT', 'CONTRACT')   -- full table scan, no balance index
)
SELECT ab.*,
  (SELECT json_agg(...) FROM (
     SELECT token_id, balance FROM token_account ta
     WHERE ta.account_id = ab.account_id AND ta.associated IS TRUE
     ORDER BY token_id DESC LIMIT 50
   ) AS account_token_balance) AS token_balances
FROM entity_balance ab
WHERE ab.balance = 0                      -- filter applied after full CTE scan
ORDER BY ab.account_id DESC
LIMIT 100;
-- Result: full entity scan + up to 100 token_account subqueries per request
-- Repeatable with no auth, no rate limit
```
