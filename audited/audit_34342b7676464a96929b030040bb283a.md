### Title
Unthrottled Polling Path Exhausts DB I/O via Unbounded `isComplete()` False Return on Limitless Subscriptions

### Summary
When `throttled=false` and no limit is set on a subscription, `PollingContext.isComplete()` unconditionally returns `false`, causing the unthrottled polling loop to always run the full `maxPolls` (default 12) iterations at 20ms intervals with `maxPageSize=5000` per poll — 65,000 DB row reads per subscription. Because there is no global cap on concurrent subscriptions and the unthrottled path is triggered automatically for every subscriber via the safety-check mechanism, an unprivileged attacker opening many concurrent gRPC connections can multiply this load linearly to exhaust database I/O.

### Finding Description

**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `PollingContext.isComplete()`, lines 121–129.

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();

    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }

    return limitHit;   // ← only limitHit; no page-size guard
}
``` [1](#0-0) 

**Root cause:** The throttled branch terminates early when the last page is smaller than `maxPageSize` (a natural "no more data" signal). The unthrottled branch omits this check entirely and only terminates on `limitHit`. When no limit is supplied (`filter.hasLimit() == false`), `limitHit` is always `false`, so `isComplete()` never returns `true` for the unthrottled path.

**Unthrottled defaults** (from `RetrieverProperties.UnthrottledProperties`):

| Property | Default |
|---|---|
| `maxPolls` | 12 |
| `pollingFrequency` | 20 ms |
| `maxPageSize` | 5000 | [2](#0-1) 

The `repeatWhen` predicate `r -> !context.isComplete()` therefore always evaluates to `true`, causing the full `numRepeats=12` repeat cycle to execute: [3](#0-2) 

**Exploit trigger path:** Every call to `subscribeTopic()` in `TopicMessageServiceImpl` schedules a safety-check flux that fires after 1 second and calls `retrieve(filter, false)` (unthrottled) unconditionally for any subscriber that has not yet completed: [4](#0-3) 

The `missingMessages` path (called from `incomingMessages` via `concatMap`) also calls `retrieve(..., false)`: [5](#0-4) 

**Per-subscription DB cost (no limit, default config):**
- 1 initial poll + 12 repeats = **13 polls**
- 13 × 5,000 rows = **65,000 DB rows read per subscription**

**Why existing checks are insufficient:**

1. `maxConcurrentCallsPerConnection = 5` (per-connection Netty limit) limits calls per TCP connection but does not cap total connections or total subscriptions globally. [6](#0-5) 

2. `timeout = 60s` only terminates a subscription that emits no messages for 60 seconds; an active topic keeps subscriptions alive indefinitely. [7](#0-6) 

3. `maxPolls=12` is a per-subscription cap, not a global rate limit. N concurrent subscriptions produce N × 65,000 DB reads with no server-side throttle.

4. The test `unthrottledShouldKeepPolling` explicitly validates this behavior as intentional, confirming no defensive check exists: [8](#0-7) 

### Impact Explanation

An attacker with N concurrent subscriptions (no limit set) generates N × 65,000 sequential DB row reads within ~260ms (13 × 20ms) per subscription cycle. At a modest 100 concurrent subscriptions across 20 connections (5 calls each), this is 6.5 million DB rows read in under 300ms. The DB connection pool and I/O bandwidth become the bottleneck, causing query latency to spike for all other mirror node consumers (REST API, other gRPC subscribers, the importer). This constitutes a denial-of-service against the mirror node's database tier, degrading or halting all dependent services.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly accessible. No API key, token, or account credential is required. The attacker needs only a gRPC client (e.g., `grpcurl`, any HCS SDK) and the ability to open multiple TCP connections. The attack is trivially repeatable and scriptable. The safety-check trigger (1 second after subscription) is automatic — the attacker does not need to craft any special message, only subscribe to any existing topic without a limit.

### Recommendation

1. **Fix `isComplete()` for the unthrottled path**: Mirror the throttled branch's page-size guard so that a page smaller than `maxPageSize` also signals completion:
   ```java
   return pageSize.get() < maxPageSize || limitHit;
   ```
   This is the symmetric fix and matches the throttled branch's logic.

2. **Add a global concurrent-subscription limit** in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge — reject new subscriptions above a configurable threshold.

3. **Add per-IP or per-connection rate limiting** at the Netty/gRPC interceptor layer to bound the subscription creation rate.

4. **Reduce unthrottled defaults** or make `maxPolls` and `maxPageSize` subject to a global DB-query-rate budget.

### Proof of Concept

**Preconditions:** Mirror node running with default config; at least one topic exists in the DB.

**Steps:**

```bash
# Open 20 connections, each with 5 concurrent limitless subscriptions (100 total)
for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}, "consensusStartTime": {"seconds": 0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

**What happens:**
1. Each subscription triggers `subscribeTopic()` → `retrieve(filter, true)` (historical, throttled, terminates quickly).
2. After 1 second, the safety check fires for each subscription → `retrieve(filter, false)` (unthrottled, no limit).
3. `isComplete()` returns `false` for all 12 repeat cycles.
4. Each subscription issues 13 DB queries × 5,000 rows = 65,000 rows.
5. 100 concurrent subscriptions = 6,500,000 DB rows read in ~260ms.
6. DB I/O saturates; query latency for all mirror node services spikes; REST API and other gRPC calls time out.
