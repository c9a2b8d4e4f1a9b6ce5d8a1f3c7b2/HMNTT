### Title
Unbounded File Data Read via `callDataId` in `getContractResultsByTimestamp` Enables Non-Network DoS

### Summary
The `getContractResultsByTimestamp` handler unconditionally reads the full contents of any user file referenced by an Ethereum transaction's `callDataId` from the database with no size cap, then hex-encodes the result into the JSON response. An attacker who has previously submitted an Ethereum transaction to the Hedera network referencing a large file as `callDataId` can repeatedly query the public endpoint to force unbounded memory allocation and serialization work on the mirror node REST service.

### Finding Description

**Exact code path:**

`getContractResultsByTimestamp` (lines 1002–1042 of `rest/controllers/contractController.js`) executes:

```js
let fileData = null;
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

`isValidUserFileId` (`rest/utils.js` lines 445–447) only checks that the entity number is > 1000 (i.e., a non-system file). It performs **no size check**:

```js
const isValidUserFileId = (val) => {
  return !isNil(val) && val !== '' && EntityId.parse(val).num > 1000;
};
```

`FileDataService.getLatestFileDataContents` (`rest/service/fileDataService.js` lines 85–88) issues a SQL query using `string_agg` to concatenate **all** file chunks from the latest create/update onwards into a single in-memory buffer with no row or byte limit:

```sql
string_agg(fd.file_data, '' order by fd.consensus_timestamp) as file_data
```

The result is then hex-encoded (doubling its size) and placed directly into the JSON response body (`rest/viewmodel/contractResultDetailsViewModel.js` lines 139–140):

```js
} else if (!contractResult.functionParameters.length && !isNil(fileData)) {
  this.function_parameters = utils.toHexStringNonQuantity(fileData.file_data);
}
```

**Root cause:** The code assumes `callDataId` files are small (typical Ethereum call data), but the Hedera file service allows files to be built up to ~1 MB through multiple `FILEAPPEND` transactions. There is no guard between the `isValidUserFileId` check and the unbounded `string_agg` database read.

**Why existing checks fail:** `isValidUserFileId` is a correctness guard (system vs. user file), not a safety guard. The REST API for this endpoint has no rate limiting (the throttling found in the codebase applies only to the `web3` module's `/api/v1/contracts/call` endpoint, not to the REST mirror node's `/api/v1/contracts/:contractId/results/:consensusTimestamp`).

### Impact Explanation

Each request to the endpoint for a transaction with a large `callDataId` causes:
1. A full `string_agg` database scan and concatenation of all file chunks (up to ~1 MB) into a single PostgreSQL result row
2. Transfer of that blob over the DB connection into the Node.js process heap
3. Hex encoding (2× memory expansion, up to ~2 MB per request)
4. JSON serialization and HTTP response buffering

Concurrent requests from multiple attacker-controlled IPs multiply this effect. With no per-IP or per-endpoint rate limiting on the REST API, a small number of concurrent connections can exhaust the Node.js heap or saturate the DB connection pool, degrading or denying service to all users of the mirror node REST API.

### Likelihood Explanation

**Precondition cost:** The attacker must hold HBAR to create a large file on Hedera and submit an Ethereum transaction referencing it as `callDataId`. This is a one-time cost.

**Repeatability:** Once the transaction is finalized on-chain and ingested by the mirror node, the attack is free to repeat indefinitely. The attacker only needs to know the `contractId` and `consensusTimestamp` of their own transaction, both of which are publicly visible.

**Accessibility:** The endpoint requires no authentication. Any internet-accessible mirror node deployment is exposed.

### Recommendation

1. **Enforce a size limit** before loading file data: after retrieving the row, check `fileData.file_data.length` against a configurable maximum (e.g., 128 KB) and either truncate or return an error/omit the field.
2. **Add per-endpoint rate limiting** to the REST API (analogous to the web3 throttle) to bound the number of requests per IP per second for detail endpoints that perform secondary DB lookups.
3. **Cap the SQL query** by adding a `HAVING length(string_agg(...)) <= <limit>` or by fetching only up to N bytes of file data at the database level.

### Proof of Concept

1. Obtain HBAR on Hedera mainnet/testnet.
2. Create a file via `FileCreate` + multiple `FileAppend` transactions to build a file approaching 1 MB.
3. Submit an `EthereumTransaction` with `callData` field set to the `FileID` of the large file (the `callDataId`).
4. Wait for the mirror node to ingest the transaction. Note the `contractId` and `consensusTimestamp`.
5. Repeatedly (e.g., 50 concurrent connections in a loop) issue:
   ```
   GET /api/v1/contracts/<contractId>/results/<consensusTimestamp>
   ```
