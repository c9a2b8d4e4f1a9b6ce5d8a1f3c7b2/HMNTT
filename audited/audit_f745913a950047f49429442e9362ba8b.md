### Title
Cache Key Whitespace Bypass Enables Unbounded Parse-and-Validate CPU Exhaustion in GraphQL PreparsedDocumentProvider

### Summary
`CachedPreparsedDocumentProvider` uses the raw, un-normalized query string as the Caffeine cache key. An unprivileged attacker can send an unbounded stream of semantically identical GraphQL queries that differ only in whitespace placement, each producing a distinct cache key and forcing a full parse-and-validate cycle per request. No authentication or special privilege is required, and no existing control prevents this.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

`executionInput.getQuery()` returns the raw, verbatim query string supplied by the client. The Caffeine `AsyncCache<String, PreparsedDocumentEntry>` performs an exact-string lookup against this key. [2](#0-1) 

**Root cause / failed assumption:**
The design assumes clients will send the same query string repeatedly. In reality, any whitespace variation (extra space, tab, newline, or different indentation) produces a different `String` key, causing a cache miss and invoking `parseAndValidateFunction.apply(executionInput)` — the full parse-and-validate pipeline — for every such request.

**Exploit flow:**
1. Attacker crafts a minimal valid query, e.g. `{account{id}}`.
2. Attacker generates a stream of semantically identical variants by inserting whitespace in different positions:
   - `{ account{id}}`
   - `{  account{id}}`
   - `{ account {id}}`
   - `{ account { id } }`
   - `{\naccount{id}}`
   - … (combinatorially large set)
3. Each variant is a distinct cache key → cache miss → full parse + schema validation cycle executed synchronously on the server thread pool.
4. Attacker sends these at high request rate from one or more IPs.

**Why existing checks are insufficient:**

| Control | Location | Why it fails |
|---|---|---|
| `maxWhitespaceTokens(1000)` | `GraphQlConfiguration.java:34` | Limits whitespace *within a single query*, does not prevent generating many unique queries each with modest whitespace | 
| `maxCharacters(10000)`, `maxTokens(1000)` | `GraphQlConfiguration.java:34` | Bound per-query cost, but do not prevent cache bypass; attacker uses small queries with varied whitespace |
| `maximumSize=1000` cache | `CacheProperties.java:15` | Bounds memory, but Caffeine evicts old entries under pressure — attacker continuously fills the cache with new unique keys, keeping the hit rate near zero |
| No rate limiting | Entire `graphql/` module | No `@RateLimiter`, no IP-based throttle, no request-per-second cap found in the codebase | [3](#0-2) [4](#0-3) 

### Impact Explanation
Every cache-missing request triggers the full GraphQL parse-and-validate pipeline. Under sustained attack, all server threads become occupied processing parse-and-validate cycles for attacker-controlled queries, starving legitimate traffic. Because the attack requires no authentication and the query space is combinatorially unbounded (whitespace can be placed in any of many token boundaries within the 10,000-character limit), the attacker can maintain a near-zero cache hit rate indefinitely. This constitutes a non-network-based CPU exhaustion DoS against the GraphQL service.

### Likelihood Explanation
The attack requires zero privileges — the GraphQL endpoint is public. It is trivially automatable: a simple script generates whitespace-varied query strings in a loop and sends them via HTTP POST. No exploit tooling, credentials, or insider knowledge are needed. The attack is repeatable and stateless from the attacker's perspective. The only practical barrier is network bandwidth, which is minimal for small GraphQL queries.

### Recommendation
Normalize the query string before using it as a cache key. The standard approach is to parse the query once into an AST and re-serialize it (or hash the canonical AST) to produce a whitespace-independent key. Alternatively, strip/collapse all whitespace outside of string literals before caching. A lightweight option is to apply `query.replaceAll("\\s+", " ").trim()` as a pre-key step, though a proper AST-based canonical form is more robust. Additionally, implement per-IP or per-client rate limiting at the servlet filter or API gateway layer to bound the request rate regardless of cache behavior.

### Proof of Concept
```bash
# Send 10,000 semantically identical queries with varying whitespace,
# each producing a unique cache key and forcing a full parse-and-validate cycle.

BASE_QUERY='{"query":"{%SPACES%account{id}}"}'
for i in $(seq 1 10000); do
  SPACES=$(python3 -c "print(' ' * $i)")
  QUERY=$(echo "$BASE_QUERY" | sed "s/%SPACES%/$SPACES/")
  curl -s -X POST http://<target>/graphql \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
wait
```
Each iteration inserts a different number of leading spaces before `account`, producing a unique raw query string. The cache key differs for every request, the cache never hits, and the server executes `parseAndValidateFunction.apply(executionInput)` for all 10,000 requests concurrently.
