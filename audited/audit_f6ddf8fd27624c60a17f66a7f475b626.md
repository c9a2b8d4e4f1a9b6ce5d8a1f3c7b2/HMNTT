### Title
Unbounded `req.originalUrl` Logging in `handleError` Enables Unauthenticated Log-Flood I/O DoS

### Summary
`handleError` in `rest/middleware/httpErrorHandler.js` embeds `req.originalUrl` verbatim into every warn/error log line with no length cap. The custom logger in `rest/logger.js` bypasses its 4 096-byte ring buffer and calls `process.stdout.write` synchronously for any line larger than the buffer. Because the REST API has no rate limiting and no URL-length enforcement, an unauthenticated attacker can flood the process's stdout I/O by sending a high volume of malformed requests carrying near-maximum-length URLs.

### Finding Description

**Exact code path**

`handleError` unconditionally interpolates `req.originalUrl` into the log string on both branches:

```js
// rest/middleware/httpErrorHandler.js  lines 33-35
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: …`
);
// line 39
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: …`, detailedMessage);
``` [1](#0-0) 

The logger's `#write` method has a fast-path that skips the ring buffer entirely when the serialised line exceeds 4 096 bytes:

```js
// rest/logger.js  lines 80-83
// Line is larger than entire buffer so write directly
if (bytes > this.#bufferSize) {
  process.stdout.write(line);   // synchronous write, no cap
  return;
}
``` [2](#0-1) 

The default buffer is 4 096 bytes and is never configurable per-line: [3](#0-2) 

**Root cause / failed assumption**

The code assumes `req.originalUrl` is short. Node.js's HTTP parser accepts request lines up to `--max-http-header-size` (default 16 384 bytes), so a URL of ~16 KB is valid at the protocol level. No middleware in `server.js` enforces a shorter limit, and no rate-limiting middleware is registered anywhere in the REST stack: [4](#0-3) 

**Exploit flow**

1. Attacker crafts a GET request with a ~16 KB query string, e.g.  
   `GET /api/v1/accounts?account.id=AAAA…(×16000) HTTP/1.1`
2. The `InvalidArgumentError` path (or any other error path) is triggered.
3. `handleError` calls `logger.warn(… req.originalUrl …)` — the resulting string is ~16 KB.
4. `#write` detects `bytes > 4096`, skips the buffer, and calls `process.stdout.write(line)` directly.
5. Repeated at high concurrency (e.g., 1 000 req/s), this generates ~16 MB/s of raw stdout writes with no back-pressure or throttle.

### Impact Explanation

- **I/O exhaustion**: Continuous large synchronous writes saturate the process's stdout pipe. In Kubernetes, this pipe feeds a log aggregator; back-pressure stalls the Node.js event loop, degrading or halting legitimate request handling.
- **Log-storage depletion**: At 16 KB per error log line × 1 000 req/s = ~16 MB/s written to persistent log storage; disk can be exhausted within minutes.
- **Observability loss**: Legitimate warn/error entries are drowned out, hiding real incidents during an attack.
- No authentication is required; the attack is repeatable indefinitely.

### Likelihood Explanation

Any internet-facing deployment is reachable. The attack requires only a standard HTTP client (e.g., `curl`, `wrk`, `ab`) with a crafted long URL. No credentials, tokens, or special knowledge of the application are needed. The REST API exposes many endpoints that validate parameters and throw `InvalidArgumentError`, all of which route through `handleError`. The absence of rate limiting (confirmed in `server.js`) means the attacker faces no server-side throttle.

### Recommendation

1. **Truncate `req.originalUrl` before logging** — cap it at a safe length (e.g., 512 characters) inside `handleError`:
   ```js
   const safeUrl = req.originalUrl.slice(0, 512);
   logger.warn(`${req.ip} ${req.method} ${safeUrl} …`);
   ```
2. **Add a per-IP or global rate limiter** to the REST API (e.g., `express-rate-limit`) before the route handlers so that error-triggering requests are throttled at the HTTP layer.
3. **Enforce a maximum URL length** in an early middleware and reject oversized requests with HTTP 414 before they reach any route handler or error handler.
4. **Cap individual log-line size** inside `Logger#write` so that no single write can exceed a configured maximum regardless of input.

### Proof of Concept

```bash
# Generate a ~15 KB query string
LONG_PARAM=$(python3 -c "print('A'*15000)")

# Flood the server with malformed requests (no auth required)
wrk -t8 -c200 -d30s \
  "http://<mirror-node-rest>:5551/api/v1/accounts?account.id=${LONG_PARAM}"
```

**Expected result**: Each request triggers `InvalidArgumentError` → `handleError` → `logger.warn` with a ~15 KB URL. The server's stdout pipe receives ~15 KB × RPS bytes per second. After sustained load, log-aggregator lag increases, event-loop latency rises, and legitimate API responses slow or time out. Log storage fills at a rate proportional to RPS × URL length.
