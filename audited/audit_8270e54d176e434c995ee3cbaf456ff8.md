### Title
Token Bucket Boundary Burst: 2x Per-Second Allowance via `refillGreedy` at Second Boundary

### Summary
All three rate-limit buckets (`rateLimitBucket`, `gasLimitBucket`, `opcodeRateLimitBucket`) are configured with `refillGreedy(capacity, Duration.ofSeconds(1))` where bucket capacity equals the per-second refill amount. This allows any unprivileged caller to drain the full bucket immediately, wait for the 1-second greedy refill, and drain it again — achieving a burst of 2× the intended per-second limit within a ~1-second window. Because the throttle does not distinguish historical-block queries from current-block queries, an attacker can direct this doubled burst entirely at expensive historical state replay calls.

### Finding Description
**Exact code locations:**

- `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 27–31 (`rateLimitBucket`):
  ```java
  .capacity(rateLimit)
  .refillGreedy(rateLimit, Duration.ofSeconds(1))
  ```
- Same pattern at lines 37–39 (`gasLimitBucket`) and lines 50–52 (`opcodeRateLimitBucket`).
- `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, line 38:
  ```java
  if (!rateLimitBucket.tryConsume(1)) {
  ```

**Root cause:** `refillGreedy` adds tokens proportionally and continuously over the refill period. Because `capacity == refillAmount`, the bucket is guaranteed to be completely full again exactly 1 second after it was last fully drained. There is no sliding-window accounting; the only state tracked is the current token count. The failed assumption is that `capacity = N tokens/s` enforces a hard ceiling of N requests per second — it does not; it enforces N requests per *bucket cycle*, and two full cycles can be triggered within a window slightly larger than 1 second.

**Exploit flow:**
1. Attacker sends `requestsPerSecond` (default: 500) requests at time T=0, draining `rateLimitBucket` to 0.
2. Attacker waits until T=1.000s; `refillGreedy` has restored the bucket to full capacity (500 tokens).
3. Attacker immediately sends 500 more requests.
4. Result: 1,000 requests processed in ≈1 second — 2× the configured limit.
5. The same attack applies to `gasLimitBucket`: drain 750,000 scaled-gas tokens (default 7.5 B gas / 10,000 scale factor), wait 1 s, drain again → 1.5 B gas consumed in ≈1 s.

**Why existing checks are insufficient:**
- `tryConsume(1)` at line 38 only checks the instantaneous token count; it has no memory of how many tokens were consumed in the preceding second.
- There is no sliding-window counter, no per-IP sub-limit, and no distinction between cheap (latest-block) and expensive (historical-block replay) requests.
- The `gasLimitBucket` check at line 40 is bypassed in the same way: the gas bucket is also `refillGreedy` with capacity = refill amount.

### Impact Explanation
Historical block queries (e.g., `eth_call` with a specific block number) require the mirror node to replay Hashgraph state up to that block, which is CPU- and I/O-intensive. The throttle is the only guard against overloading this replay path. By doubling the effective request rate at the second boundary, an attacker can inject 2× the intended load of historical replay calls, potentially saturating the replay thread pool, exhausting heap, or causing cascading latency for all users. Default configuration (500 req/s → 1,000 burst, 7.5 B gas/s → 15 B gas burst) makes the impact concrete and measurable. Severity: **High** — availability impact on a shared infrastructure component with no authentication required.

### Likelihood Explanation
No privileges, accounts, or special network position are required. The attack requires only:
- Knowledge of the public API endpoint.
- A script that fires N requests, sleeps ~1 second, fires N more requests.
This is trivially automatable, repeatable every second indefinitely, and undetectable by the current throttle logic. Any motivated party (competitor, researcher, or malicious actor) can execute it.

### Recommendation
1. **Replace `refillGreedy` with a sliding-window or token-bucket with initial tokens = 0** so the bucket does not start full and cannot be double-drained at the boundary. Bucket4j supports `initialTokens(0)` on the `Bandwidth.builder()`.
2. **Alternatively, use `refillIntervally` with clock-aligned intervals** and set `initialTokens` to a fraction of capacity to limit burst size.
3. **Add per-source-IP sub-limits** so a single client cannot consume the entire global budget.
4. **Distinguish historical-block requests** (block parameter ≠ `latest`/`pending`) and apply a stricter, separate bucket for them, reflecting their higher replay cost.
5. **Minimum fix** in `ThrottleConfiguration.java`:
   ```java
   Bandwidth.builder()
       .capacity(rateLimit)
       .refillGreedy(rateLimit, Duration.ofSeconds(1))
       .initialTokens(0)   // <-- prevents full-bucket drain at startup/refill boundary
       .build();
   ```
   Note: `initialTokens(0)` alone does not fully close the window but reduces the burst from 2× to 1× at the cost of a cold-start delay.

### Proof of Concept
```python
import requests, time, threading

URL = "http://<mirror-node>/api/v1/contracts/call"
PAYLOAD = {"block": "0x1", "gas": 50000, ...}  # historical block
LIMIT = 500  # default requestsPerSecond

def burst():
    for _ in range(LIMIT):
        requests.post(URL, json=PAYLOAD)

# First burst — drains the bucket
burst()
# Wait for greedy refill
time.sleep(1.01)
# Second burst — bucket is full again; 1000 requests processed in ~1 second
burst()
# Observe: second burst succeeds (HTTP 200) instead of being throttled (HTTP 429)
```
Expected result without fix: both bursts return HTTP 200 (1,000 total requests accepted).
