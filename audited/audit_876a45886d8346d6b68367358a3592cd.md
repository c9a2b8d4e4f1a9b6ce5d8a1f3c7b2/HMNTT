### Title
Unauthenticated DoS via Concurrent FULL OUTER JOIN Queries on `/api/v1/transactions?account.id=`

### Summary
Any unauthenticated external user can trigger the expensive FULL OUTER JOIN code path in `getTransactionTimestampsQuery()` by supplying an `account.id` filter. No rate limiting exists on the REST API transactions endpoint. Flooding the endpoint with concurrent requests using different `account_id` values saturates the database connection pool with long-running multi-table join operations, causing service degradation or complete unresponsiveness for all users.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactions()` (line 671) calls `doGetTransactions()` → `getTransactionTimestamps()` (line 451) → `getTransactionTimestampsQuery()` (line 500).

When any `account_id` filter is present (`accountQuery` is non-empty), the function at lines 555–604 takes the expensive branch:

```js
// rest/transactions.js lines 595-604
return `
    select coalesce(t.consensus_timestamp, ctl.consensus_timestamp, ttl.consensus_timestamp) as consensus_timestamp,
           coalesce(t.payer_account_id, ctl.payer_account_id, ttl.payer_account_id)          as payer_account_id
    from (${transactionOnlyQuery}) as t
             full outer join (${cryptoTransferQuery}) as ctl
                             on t.consensus_timestamp = ctl.consensus_timestamp
             full outer join (${tokenTransferQuery}) as ttl
                             on coalesce(t.consensus_timestamp, ctl.consensus_timestamp) = ttl.consensus_timestamp
    order by consensus_timestamp ${order}
        ${limitQuery}`;
```

Each of the three subqueries (`transactionOnlyQuery`, `cryptoTransferQuery`, `tokenTransferQuery`) independently scans large tables (`transaction`, `crypto_transfer`, `token_transfer`, `entity_transaction`) with `DISTINCT ON` and `UNION ALL`, and may themselves join back to the `transaction` table when `resultTypeQuery` or `transactionTypeQuery` is present.

**Root cause — no rate limiting on the REST API:**

`rest/server.js` (lines 68–133) shows the full middleware stack: `httpContext`, `requestLogger`, `authHandler`, `metricsHandler` (optional), `responseCacheCheckHandler` (optional) — **no rate limiter is present**. A `grep` for `rateLimit|throttle|concurrency` across all `rest/**/*.js` files returns zero matches. The throttling found in the codebase (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies exclusively to the **web3 API** (contract calls), not to the REST API.

**Why the cache does not help:**

The `keyMapper` in `doGetTransactions()` (line 680) keys cache entries on `consensus_timestamp` only. Requests with different `account_id` values produce different timestamp result sets, so cache hits are negligible across a flood of varied `account_id` values.

**Existing mitigations are insufficient:**

- `rest/dbpool.js` line 14–15: `max: config.db.pool.maxConnections` and `statement_timeout: config.db.pool.statementTimeout` exist.
- The Helm chart (`charts/hedera-mirror/templates/secret-passwords.yaml` line 124) sets `statement_timeout` to `20000` ms (20 s) for the REST DB user.
- These only bound the *duration* of each query and the *count* of concurrent DB connections. They do not prevent an attacker from keeping the pool perpetually saturated: with `N` max connections, an attacker needs only `N` concurrent requests to hold all connections busy with 20-second FULL OUTER JOIN queries. Additional requests queue in Node.js, and the DB remains at maximum load continuously.

### Impact Explanation

With the DB connection pool fully occupied by expensive FULL OUTER JOIN queries, legitimate requests either queue indefinitely or time out at the connection-acquisition layer. The DB server itself experiences sustained high CPU and memory pressure from the join operations across `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction` tables. This constitutes a partition-equivalent state: the mirror node REST API becomes unresponsive to all users, not just the attacker. Severity is **High** — complete availability loss of a public API with no authentication barrier.

### Likelihood Explanation

The attack requires zero privileges, zero credentials, and zero knowledge beyond the public API documentation. The `account.id` parameter is documented and freely usable. An attacker needs only a script sending concurrent HTTP GET requests with rotating `account_id` values (e.g., `0.0.1` through `0.0.N`). This is trivially automatable with `curl`, `ab`, `wrk`, or any HTTP load tool. The attack is repeatable and sustainable indefinitely. No CAPTCHA, API key, or IP-based throttle stands in the way at the application layer.

### Recommendation

1. **Add per-IP rate limiting middleware** to the Express REST API (e.g., `express-rate-limit`) applied globally or specifically to the `/api/v1/transactions` route, before route handlers.
2. **Add a query cost guard**: detect when `accountQuery` is non-empty and enforce a stricter per-IP or global concurrency limit for that code path specifically.
3. **Reduce `statement_timeout`** for the REST DB user to a lower value (e.g., 5 s) to limit the window each expensive query holds a connection.
4. **Consider query result caching keyed on the full filter set** (including `account_id`) to serve repeated identical queries from cache rather than hitting the DB.

### Proof of Concept

```bash
# Flood the endpoint with concurrent requests using different account_id values
# No authentication required
for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?account.id=0.0.$i" &
done
wait
```

Each background request triggers the FULL OUTER JOIN path. With a default pool of (typically) 10 connections, 10 concurrent queries each running up to 20 seconds keep the DB saturated. Scaling to hundreds of concurrent requests ensures the pool is perpetually full and the DB CPU pegged, making the API unresponsive to all other users.
