### Title
Unauthenticated DB Request Storm via Uncached `getByIdAndType()` with Traefik Retry Amplification

### Summary
`EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` on every invocation with no application-level caching or rate limiting. Unlike the `grpc` module's `EntityRepository` which annotates `findById` with `@Cacheable`, the graphql module's `EntityRepository` inherits `findById` from `CrudRepository` with no cache. The Traefik `retry` middleware (3 attempts) amplifies each failed request threefold, and the `inFlightReq` per-IP limit is trivially bypassed with multiple source IPs, enabling an unprivileged attacker to overwhelm the database — especially when a network partition heals and queued HikariCP connections all fire simultaneously.

### Finding Description

**Exact code path:**

`EntityServiceImpl.getByIdAndType()` at line 24–26: [1](#0-0) 

calls `entityRepository.findById()` which is the bare `CrudRepository.findById()` — the graphql module's `EntityRepository` defines no `@Cacheable` on it: [2](#0-1) 

Contrast with the `grpc` module's `EntityRepository`, which explicitly caches `findById`: [3](#0-2) 

The graphql module has **zero application-level rate limiting** — no `ThrottleConfiguration`, no `@Cacheable` on entity lookups, and no per-request deduplication. The only cache in the graphql module is a query-parse cache (`CachedPreparsedDocumentProvider`) that caches the parsed GraphQL document string, not DB results: [4](#0-3) 

**Traefik retry amplification:** The Helm chart configures a `retry` middleware with 3 attempts and 100ms initial interval alongside an `inFlightReq` limit of 5 per IP: [5](#0-4) 

When a DB-side network partition causes requests to fail, Traefik retries each request up to 3 times, tripling the DB load on partition heal.

**Exploit flow:**

1. Attacker controls N source IPs (or spoofs X-Forwarded-For at `depth: 1`).
2. Each IP sends 5 concurrent POST requests to `/graphql/alpha` with `account(input: {entityId: {shard:0, realm:0, num:12345}})`.
3. Each request reaches `AccountController.account()` → `entityService.getByIdAndType()` → `entityRepository.findById()` → DB query. No cache intercepts.
4. During a DB-side network partition, HikariCP queues connection acquisition attempts (default `connectionTimeout` = 30s, `maximumPoolSize` = 10). Traefik retries each failed request 3×.
5. On partition heal: all queued HikariCP connections fire simultaneously, and Traefik's retry queue also drains — the DB receives a burst of `N × 5 × 3` queries for the same row.

### Impact Explanation

Every GraphQL account lookup results in an unconditional DB `SELECT` with no caching or deduplication. During normal operation this is a steady-state DoS risk. During a network partition heal it becomes a thundering-herd event: the HikariCP pool drains its queued requests simultaneously while Traefik's retry backlog also fires, potentially exhausting DB connection limits and causing cascading failures across all mirror node services sharing the same PostgreSQL instance. The `GraphQLHighDBConnections` alert threshold is 75% utilization — a coordinated burst can push past this. [6](#0-5) 

### Likelihood Explanation

No authentication is required. The GraphQL endpoint is publicly exposed. The `inFlightReq: 5` per-IP limit is the only meaningful throttle, but it is bypassable with as few as 20–50 distributed IPs (readily available via cloud providers or residential proxies). The `ipStrategy.depth: 1` reads from `X-Forwarded-For`, which may be spoofable depending on ingress configuration. Network partitions between application pods and the database are a realistic operational event in Kubernetes environments. The attack is repeatable and requires no special knowledge beyond the public GraphQL schema.

### Recommendation

1. **Add `@Cacheable` to `findById` in the graphql module's `EntityRepository`**, mirroring the grpc module's pattern with a short TTL (e.g., 1–5 seconds via Caffeine).
2. **Add application-level rate limiting** to the graphql module (e.g., a Bucket4j-based filter as used in the `web3` module's `ThrottleConfiguration`).
3. **Remove or reduce the Traefik `retry` middleware** for the graphql route, or scope it only to connection-level errors, not application errors — retrying on DB failures amplifies the storm.
4. **Set a HikariCP `connectionTimeout`** short enough to shed load quickly during partitions rather than queuing for 30 seconds.

### Proof of Concept

```bash
# From 20+ distinct IPs or using a tool that rotates X-Forwarded-For:
for i in $(seq 1 5); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}' &
done
wait

# Induce a DB-side partition (e.g., network policy drop), sustain requests for 30s,
```
