### Title
`validateAcceptEncodingHeader()` Substring Match Bypass Allows Uncompressed Large Response Delivery

### Summary
`validateAcceptEncodingHeader()` in `OpcodesController.java` uses a naive `.contains("gzip")` substring check to enforce that clients accept gzip-compressed responses. An unprivileged attacker can send a crafted `Accept-Encoding` header value such as `x-custom-gzip-encoding` or `gzip;q=0` that contains the substring `"gzip"` but is not a valid gzip encoding token, causing the guard to pass while the actual HTTP layer sends the response uncompressed. This defeats the sole stated purpose of the check: preventing large uncompressed responses from being transmitted.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76.

```java
// Line 76
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
``` [1](#0-0) 

**Root cause:** The check performs a raw substring match (`String.contains("gzip")`) rather than RFC 7231-compliant token parsing. The comment at lines 71–73 explicitly states the purpose is to prevent large uncompressed responses, but the implementation does not verify that `gzip` appears as a standalone, non-zero-quality token.

**Two concrete bypass values:**
1. `Accept-Encoding: x-custom-gzip-encoding` — contains `"gzip"` as a substring; passes the check. Spring Boot's embedded Tomcat performs proper RFC 7231 token negotiation and does not recognize `x-custom-gzip-encoding` as `gzip`, so the response is sent **uncompressed**.
2. `Accept-Encoding: gzip;q=0` — contains `"gzip"` as a substring; passes the check. `q=0` is the RFC 7231 mechanism for explicitly refusing an encoding. Tomcat honors `q=0` and sends the response **uncompressed**.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true&storage=true` with `Accept-Encoding: x-custom-gzip-encoding`.
2. `validateAcceptEncodingHeader()` evaluates `"x-custom-gzip-encoding".toLowerCase().contains("gzip")` → `true` → no exception thrown.
3. `throttleManager.throttleOpcodeRequest()` is called (rate-limit applies, but does not block the request).
4. `opcodeService.processOpcodeCall(request)` executes a full EVM re-execution with stack+memory+storage tracing.
5. Spring serializes the `OpcodesResponse` and passes it to Tomcat. Tomcat's content negotiation finds no valid `gzip` token in `Accept-Encoding` and sends the body **uncompressed**.
6. A potentially multi-megabyte JSON payload is transmitted without compression. [2](#0-1) 

**Why existing checks fail:** The test suite at line 507 only covers `{"deflate", "identity", "br", ""}` as rejection cases; neither `x-custom-gzip-encoding` nor `gzip;q=0` is tested. [3](#0-2) 

### Impact Explanation
The endpoint re-executes a full EVM transaction trace (stack, memory, storage) — an intentionally expensive operation. The gzip requirement exists precisely because the response is described as "huge" (line 72). Bypassing it allows an attacker to force the server to transmit large uncompressed payloads, amplifying bandwidth consumption per request. Combined with the fact that `stack=true&memory=true&storage=true` is the worst case, repeated requests within the rate-limit window can cause sustained high egress bandwidth usage. This is a resource exhaustion / amplification issue against the mirror node's network layer. [4](#0-3) 

### Likelihood Explanation
No privileges are required. The attacker only needs to know the endpoint path and any valid transaction hash (publicly observable on-chain). The bypass requires crafting a single HTTP header value — trivially achievable with `curl` or any HTTP client. It is fully repeatable up to the rate-limit window, and the rate limiter does not prevent the uncompressed response from being sent once the validation passes.

### Recommendation
Replace the substring check with proper RFC 7231 token parsing. Use Spring's `org.springframework.http.MediaType` / `org.springframework.util.MimeTypeUtils` or parse the `Accept-Encoding` header as a comma-separated list of tokens, strip quality values (`q=`), and check for an exact case-insensitive match of the token `gzip` with a quality value `> 0`:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean acceptsGzip = Arrays.stream(acceptEncodingHeader.split(","))
        .map(token -> token.split(";")[0].trim().toLowerCase())
        .anyMatch("gzip"::equals);
    // Also check q=0 is not set for gzip
    if (!acceptsGzip) {
        throw ...;
    }
}
```

Alternatively, delegate the check to Spring's `ContentNegotiationStrategy` or use `HttpHeaders.getAcceptEncoding()` with proper quality-value filtering.

### Proof of Concept
```bash
# Bypass 1: gzip as substring of a custom token
curl -v \
  -H "Accept-Encoding: x-custom-gzip-encoding" \
  "https://<mirror-node>/api/v1/contracts/results/<valid-tx-hash>/opcodes?stack=true&memory=true&storage=true"
# Expected: HTTP 200, response body NOT gzip-compressed (no Content-Encoding: gzip header)

# Bypass 2: gzip explicitly refused via q=0
curl -v \
  -H "Accept-Encoding: gzip;q=0, identity" \
  "https://<mirror-node>/api/v1/contracts/results/<valid-tx-hash>/opcodes?stack=true&memory=true&storage=true"
# Expected: HTTP 200, response body NOT gzip-compressed
```

In both cases the validation at line 76 passes (`.contains("gzip")` is `true`), but Tomcat's content negotiation does not apply gzip compression, resulting in a large uncompressed response being sent to the client.
