### Title
Unauthenticated Unbounded gRPC Subscription Flood Enables Resource Exhaustion DoS

### Summary
The `TopicListener.listen()` interface and its implementations accept any `TopicMessageFilter` from any unauthenticated caller with no per-IP connection limit, no global subscription cap, and no subscription-creation rate limiting. An attacker can open an arbitrary number of TCP connections to the gRPC port, each carrying up to 5 concurrent `subscribeTopic` calls, spawning unbounded independent polling loops and thread allocations that exhaust the database connection pool and the `boundedElastic` scheduler, making the server unresponsive to legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen(newFilter)` (line 120 of `TopicMessageServiceImpl.java`).

In `PollingTopicListener.listen()` (lines 34–49 of `PollingTopicListener.java`), every call unconditionally creates a new `PollingContext` and a new `Flux` that repeats `Long.MAX_VALUE` times with no ceiling on concurrent instances:

```java
return Flux.defer(() -> poll(context))
    .delaySubscription(interval, scheduler)
    .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // unbounded
        .withFixedDelay(interval)
        .withScheduler(scheduler))
```

Each repetition calls `topicMessageRepository.findByFilter(newFilter)`, issuing a live DB query every `interval` (default 500 ms). There is no global counter, semaphore, or quota gating how many such loops may exist simultaneously.

**Root cause — failed assumption:**

The design assumes that the per-connection call cap (`maxConcurrentCallsPerConnection = 5`, `NettyProperties.java` line 14, applied in `GrpcConfiguration.java` line 33) is a sufficient guard. It is not: it limits calls *per TCP connection* but places no bound on the number of TCP connections an attacker may open. The product of (connections × 5) subscriptions is unbounded.

**No authentication gate:** `ConsensusController.subscribeTopic()` (lines 43–53) performs no credential check before invoking the service. Any network-reachable client may call it.

**Throttling absent in gRPC module:** A `grep` across all `grpc/src/main/java/**/*.java` for `rateLimit`, `throttle`, `maxConnections`, or `perIp` returns zero hits in the listener/controller/service layer. The only throttling in the repository (`web3/ThrottleConfiguration.java`, `web3/ThrottleManagerImpl.java`) is scoped exclusively to the Web3/EVM module and is not wired into the gRPC path.

**Infrastructure rate limit is optional and insufficient:** `charts/hedera-mirror-grpc/values.yaml` line 69 shows `maxRatePerEndpoint: 250` with the comment *"Requires a change to HPA to take effect"* and is gated on `gateway.gcp.enabled: true`. It is not a default application-level control and is absent in non-GCP deployments.

**REDIS/SHARED_POLL modes are also affected:** Even though `RedisTopicListener` and `SharedPollingTopicListener` share a single upstream Redis/DB subscription per topic, `SharedTopicListener.listen()` (lines 21–26) still allocates a per-subscriber `onBackpressureBuffer` and a `publishOn(Schedulers.boundedElastic())` worker thread for every caller. Flooding with N subscriptions exhausts the `boundedElastic` thread pool (capped at `10 × CPU cores` by Reactor's default), blocking all reactive pipelines on the server.

### Impact Explanation

When the attack succeeds:
- **POLL mode:** DB connection pool is saturated; all queries (including those for legitimate subscribers) queue or fail.
- **REDIS/SHARED_POLL mode:** `boundedElastic` thread pool is exhausted; message delivery to all subscribers stalls.
- Legitimate clients subscribing to HCS topics (e.g., applications that use topic messages to coordinate multi-party token releases or escrow unlocks) stop receiving messages for the duration of the attack.
- The mirror node gRPC service becomes unresponsive; health checks may fail, triggering pod restarts that reset but do not prevent re-attack.

Severity: **High** (availability impact on a public, unauthenticated service; no privilege required).

### Likelihood Explanation

- No account, API key, or credential is required.
- Standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`) suffice.
- Opening thousands of TCP connections and issuing 5 `subscribeTopic` RPCs per connection is trivial from a single host with a modest network connection.
- The attack is repeatable immediately after any server recovery.
- The default listener type is REDIS, so the thread-exhaustion vector is active in all standard deployments.

### Recommendation

1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP subscription rate limit** using a token-bucket (e.g., Bucket4j, already used in the web3 module) applied in `ConsensusController.subscribeTopic()` before the service call.
3. **Add a per-IP connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionsPerIp()` or an equivalent interceptor.
4. **Require authentication** (mTLS or an API-key interceptor) for `subscribeTopic` in production deployments.
5. Promote the GCP gateway `maxRatePerEndpoint` to a mandatory, application-enforced default rather than an optional infrastructure hint.

### Proof of Concept

```bash
# Requires: grpcurl, bash, a valid topicId on the target network

TARGET="mirror.mainnet.hedera.com:443"
TOPIC_ID="0.0.12345"

# Open 200 parallel connections, each with 5 concurrent subscriptions (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -d '{
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
      "consensusStartTime": {"seconds": 0}
    }' \
    $TARGET \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: legitimate subscriber below receives no messages / connection refused
grpcurl -d '{
  "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
  "consensusStartTime": {"seconds": 0}
}' $TARGET com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

Reproduces resource exhaustion (DB pool saturation in POLL mode; `boundedElastic` thread starvation in REDIS/SHARED_POLL mode) causing legitimate subscription requests to time out or be rejected.
