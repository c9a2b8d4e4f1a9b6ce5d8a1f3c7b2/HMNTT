### Title
Database Host and Port Exposed via Unauthenticated `/health/readiness` Endpoint Error Messages

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` registers a PostgreSQL health check using the raw DSN (containing host and port) and delegates response serialization entirely to the `hellofresh/health-go/v4` library. When the PostgreSQL connection fails with a "connection refused" error, the library propagates the raw Go network error — which includes the database host and port — directly into the JSON HTTP response body, with no authentication gate and no error sanitization.

### Finding Description
**Code path:**
- `rosetta/app/middleware/health.go`, `NewHealthController()`, line 42: [1](#0-0) 
- DSN construction: `rosetta/app/config/types.go`, `GetDsn()`, line 49–57: [2](#0-1) 

**Root cause:**
`postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()})` passes the full DSN (including `host=<HOST> port=<PORT>`) to the `hellofresh/health-go/v4` postgres checker. When the TCP connection to the database is refused, the Go `net` package produces an error of the form:

```
dial tcp <HOST>:<PORT>: connect: connection refused
```

The `hellofresh/health-go/v4` library captures this raw error string and serializes it into the `output` field of the per-component check in the JSON response body:

```json
{
  "status": "unavailable",
  "checks": {
    "postgresql": {
      "status": "unavailable",
      "output": "dial tcp 10.0.0.5:5432: connect: connection refused",
      "timestamp": "..."
    }
  }
}
```

There is no error wrapping, sanitization, or redaction anywhere between the driver error and the HTTP response. The route is registered with no authentication middleware: [3](#0-2) 

### Impact Explanation
An unauthenticated external attacker learns the exact internal IP address (or hostname) and TCP port of the PostgreSQL database server. This enables targeted network reconnaissance, port scanning of the internal database tier, and informs further attack planning (e.g., attempting direct database connections if network segmentation is weak). Severity: **Medium** — no credentials are leaked in a "connection refused" error, but the network location of a sensitive backend component is disclosed to any internet-facing caller with no preconditions.

### Likelihood Explanation
Exploitation requires only a single unauthenticated HTTP GET request. The endpoint is designed to be publicly reachable (Kubernetes liveness/readiness probes are commonly exposed). The condition (PostgreSQL temporarily unreachable, restarting, or misconfigured) occurs routinely in real deployments. Any external attacker who can reach the service port can trigger and observe this disclosure repeatedly and reliably.

### Recommendation
1. **Sanitize error output**: Wrap the postgres check function to replace or redact the raw driver error before it propagates to the health library, e.g., return a generic `errors.New("postgresql check failed")` instead of the raw network error.
2. **Restrict endpoint access**: Place `/health/readiness` behind network-level controls (e.g., only reachable from the cluster's internal network or the load balancer's health-probe source IPs), not on the public-facing port.
3. **Avoid embedding host/port in propagated errors**: Ensure any wrapper around the health check strips connection-detail strings before they reach the serialized response.

### Proof of Concept
**Precondition:** The Rosetta service is running and its PostgreSQL database is temporarily unreachable (e.g., during a restart, network partition, or misconfiguration).

**Steps:**
```bash
# 1. Send an unauthenticated GET to the readiness endpoint
curl -s http://<rosetta-host>:<port>/health/readiness | jq .

# 2. Observe response (HTTP 503):
{
  "status": "unavailable",
  "checks": {
    "postgresql": {
      "status": "unavailable",
      "output": "dial tcp 10.0.0.5:5432: connect: connection refused",
      "timestamp": "2026-04-28T12:00:00Z"
    }
  }
}
```

**Result:** The attacker now knows the database is at `10.0.0.5:5432`, extracted from the `checks.postgresql.output` field with zero privileges required.
