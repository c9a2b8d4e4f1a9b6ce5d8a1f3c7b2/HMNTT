### Title
Sequential DB Round-Trips in `extractSqlFromTopicMessagesRequest()` Enable Connection Pool Exhaustion DoS

### Summary
When `config.query.topicMessageLookup` is enabled, any unauthenticated user can trigger two sequential database round-trips per request to `/api/v1/topics/:topicId/messages` — first `getTopicMessageTimestampRanges()` against `topic_message_lookup`, then `getMessages()` against `topic_message`. With no rate limiting and a default pool of 10 connections, sustained concurrent requests double effective connection hold time and exhaust the pool, causing REST API unavailability. This does **not** affect network transaction confirmation (mirror nodes are read-only observers), but does cause REST API DoS.

### Finding Description
**Code path:**

`rest/topicmessage.js`, `getTopicMessages()` (lines 133, 143) calls `extractSqlFromTopicMessagesRequest()` then `getMessages()` sequentially:

```
line 133: const {query, params, order, limit} = await extractSqlFromTopicMessagesRequest(topicId, filters);
line 143: const messages = await getMessages(query, params, queryHint);
```

Inside `extractSqlFromTopicMessagesRequest()` (lines 199–200), when `topicMessageLookupEnabled` is true:

```
const ranges = await getTopicMessageTimestampRanges(topicId, sequenceNumberFilters, timestampFilters, limit, order);
```

`getTopicMessageTimestampRanges()` (lines 254–317) issues a real DB query against `topic_message_lookup` via `pool.queryQuietly()` at line 315. If it returns non-empty ranges, execution returns to `getTopicMessages()` which then issues a second `pool.queryQuietly()` call inside `getMessages()` at line 332.

**Root cause:** Both calls are sequential `await`s on the same shared global `pool` (configured in `rest/dbpool.js` line 14: `max: config.db.pool.maxConnections`, default 10). Each request holds a pool connection for the duration of both queries back-to-back. No connection is released between the two calls.

**Failed assumption:** The pool sizing and timeout configuration assume one DB round-trip per request. With `topicMessageLookup` enabled, effective connection hold time doubles (up to 2 × `statementTimeout` = 40 s per request), halving the pool's effective concurrency capacity.

**Why filters don't help:** An attacker supplies any valid `sequencenumber` and `timestamp` filter for a topic that has data. `getRangeFromFilters()` will return a non-null range, `getTopicMessageTimestampRanges()` will return non-empty rows, and both DB calls execute. The `statementTimeout` (20 s) and `connectionTimeout` (20 s) only bound individual query duration — they do not prevent pool exhaustion under sustained concurrent load. `server.js` registers no rate-limiting middleware anywhere in the stack.

### Impact Explanation
With the default pool of 10 connections and `statementTimeout` of 20 s, 10 concurrent attacker requests can hold all pool connections for up to 40 s. Any additional legitimate requests queue for a connection and fail after `connectionTimeout` (20 s), returning errors to all users. The REST API becomes unavailable for the duration of the attack. **This does not affect Hedera network transaction confirmation** — mirror nodes are read-only downstream observers and do not participate in consensus. The actual impact is REST API DoS, not network shutdown.

### Likelihood Explanation
Precondition: `hiero.mirror.rest.query.topicMessageLookup` must be set to `true` (default is `false`). When enabled, any unauthenticated client can trigger the two-query path with a single HTTP GET request containing both `sequencenumber` and `timestamp` query parameters. No credentials, tokens, or special access are required. The attack is trivially repeatable with any HTTP load tool (e.g., `ab`, `wrk`, `curl` in a loop). Likelihood is **medium** — requires a non-default operator configuration, but once enabled, exploitation requires zero privilege.

### Recommendation
1. **Parallelize the lookup**: Issue `getTopicMessageTimestampRanges()` and any pre-validation concurrently where possible, or restructure so the lookup result is used to build a single combined query rather than two sequential pool acquisitions.
2. **Add rate limiting**: Apply a per-IP or global rate limiter middleware in `rest/server.js` before the topic message route, e.g., using `express-rate-limit`.
3. **Increase pool size or add a request concurrency cap**: Tune `db.pool.maxConnections` to account for the doubled connection hold time when `topicMessageLookup` is enabled, or add a semaphore to cap in-flight DB-bound requests.
4. **Document the DoS risk**: The configuration documentation for `topicMessageLookup` should note the pool sizing implications when enabling this feature.

### Proof of Concept
**Preconditions:**
- Mirror node REST API running with `hiero.mirror.rest.query.topicMessageLookup=true`
- At least one topic with messages exists (e.g., topic ID `0.0.1234`)
- Default pool of 10 connections

**Steps:**
```bash
# Send 15 concurrent requests with both sequencenumber and timestamp filters
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>:5551/api/v1/topics/0.0.1234/messages?sequencenumber=gt:1&timestamp=gt:0.000000001&limit=100" &
done
wait
```

**Result:** The first 10 requests each acquire a pool connection and execute `getTopicMessageTimestampRanges()`. As those complete and move to `getMessages()`, the pool remains saturated. Requests 11–15 queue for a connection. Under sustained load (repeat the loop), new requests begin failing with connection timeout errors, rendering the REST API unavailable to all users for the duration of the attack.
