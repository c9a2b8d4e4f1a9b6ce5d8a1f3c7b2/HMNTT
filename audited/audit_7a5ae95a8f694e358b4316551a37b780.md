### Title
GraphQL Connection Pool Exhaustion via Missing Application-Level Rate Limiter During Network Partition

### Summary
The `graphQlCustomizer()` bean in `GraphQlConfiguration.java` configures only query complexity and depth limits, with no application-level rate or concurrency limiter. During a network partition, each accepted GraphQL request checks out a HikariCP connection that blocks indefinitely (PostgreSQL `statement_timeout` requires DB reachability to fire), allowing an unprivileged attacker with as few as two source IPs to exhaust the default 10-connection pool and deny service to all legitimate users.

### Finding Description
**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–48, `graphQlCustomizer()`.

```java
@Bean
GraphQlSourceBuilderCustomizer graphQlCustomizer(PreparsedDocumentProvider provider) {
    var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
    var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
    var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);
    return b -> b.configureGraphQl(
            graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
}
```

The only defenses registered here are parse-time limits (characters, tokens, depth, complexity). No `Bucket4j` rate limiter, no concurrency semaphore, and no per-IP throttle exist at the application layer — in direct contrast to the `web3` module, which has a full `ThrottleConfiguration` with `rateLimitBucket`, `gasLimitBucket`, and `opcodeRateLimitBucket`.

**Root cause / failed assumption:** The design assumes that infrastructure-layer controls (Traefik `inFlightReq`) will always be present and sufficient. They are not:

1. `inFlightReq: amount: 5` is a Traefik-specific CRD middleware (`charts/hedera-mirror-graphql/values.yaml`, lines 138–142). It is only active when Traefik ingress is used; the GCP Gateway path (`gateway.gcp.enabled: true`, line 60) does not apply it.
2. Even when active, `inFlightReq` is per source IP (`ipStrategy.depth: 1`). Two attacker IPs × 5 concurrent requests = 10 simultaneous in-flight requests, exactly matching the HikariCP default pool size of 10.
3. The Traefik `retry: attempts: 3` middleware (lines 143–145) **multiplies** connection demand: each logical attacker request triggers up to 3 connection acquisitions on retry, accelerating pool exhaustion.

**Why `statementTimeout` does not save you:** `hiero.mirror.graphql.db.statementTimeout` (default 10 000 ms) is sent as a PostgreSQL session-level `SET statement_timeout`. During a network partition the TCP socket to the DB is established but silent; the PostgreSQL server never receives the command, so the timeout never fires. The connection remains checked out from HikariCP until the OS TCP keepalive/retransmit timer expires (Linux default: ~15 minutes with default `tcp_keepalive_*` settings, or longer if keepalive is disabled on the JDBC URL).

**No application.yml / HikariCP overrides exist** for the graphql module (`graphql/src/main/resources/` contains only schema files and a banner), so HikariCP runs with its defaults: `maximumPoolSize=10`, `connectionTimeout=30 000 ms`, no `socketTimeout` in the JDBC URL.

### Impact Explanation
Once all 10 HikariCP connections are held by attacker-controlled requests, every legitimate GraphQL query blocks for up to 30 seconds (HikariCP `connectionTimeout`) before throwing `SQLTimeoutException`. The GraphQL service is effectively unavailable for the duration of the partition plus the OS TCP timeout window — potentially 15+ minutes per attack wave. Because the graphql module shares the same PostgreSQL role (`mirror_graphql`) and the pool is small, a single sustained burst fully denies service with no fallback path.

### Likelihood Explanation
The attack requires no credentials, no special knowledge of the schema (any syntactically valid query of depth ≤ 10 and complexity ≤ 200 suffices), and only two source IPs (trivially obtained via cloud VMs or a small botnet). The network partition condition can be induced or exploited opportunistically during real infrastructure events. The Traefik middleware is the only meaningful barrier, and it is absent on GCP Gateway deployments and bypassable with two IPs on Traefik deployments. The `retry` middleware amplifies the effect automatically.

### Recommendation
1. **Add an application-level concurrency limiter** in `graphQlCustomizer()` or as a `WebGraphQlInterceptor`, mirroring the `ThrottleConfiguration` pattern already used in the `web3` module (bucket4j `rateLimitBucket`).
2. **Set a JDBC socket timeout** in the HikariCP datasource URL (e.g., `socketTimeout=10` for PostgreSQL JDBC) so that connections are forcibly closed when the DB is unreachable, independent of PostgreSQL-level timeouts.
3. **Set `spring.datasource.hikari.connection-timeout`** to a low value (e.g., 5 000 ms) and **`maximum-pool-size`** explicitly in the graphql application config so pool exhaustion fails fast rather than queuing.
4. **Remove or reconfigure the `retry` Traefik middleware** for the GraphQL path, or ensure it does not retry on 5xx responses caused by pool exhaustion.

### Proof of Concept
```bash
# Precondition: network partition between graphql pod and PostgreSQL
# (simulate with: iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the graphql host)

# Two source IPs, 5 concurrent requests each = 10 total (exhausts default pool)
# From IP-1:
for i in $(seq 1 5); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}' &
done

# From IP-2 (bypasses inFlightReq per-IP limit):
for i in $(seq 1 5); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}' &
done

# Result: all 10 HikariCP connections are checked out and blocked.
# Any subsequent legitimate request receives:
#   HikariPool-1 - Connection is not available, request timed out after 30000ms
# Service is unavailable for the duration of the partition + OS TCP timeout (~15 min).
```
