### Title
Unbounded Permanent gRPC Stream via No-EndTime Subscription Causing Resource Exhaustion DoS

### Summary
An unprivileged external user can open a gRPC subscription to any existing topic with `startTime=0`, `limit=Long.MAX_VALUE`, and no `endTime`. When the topic has no messages, all termination paths are bypassed and the stream remains permanently open, holding a gRPC connection and a `boundedElastic` scheduler thread indefinitely. Because the only per-connection guard is `maxConcurrentCallsPerConnection=5` (not a total-connection cap), an attacker can multiply this across many connections to exhaust server resources.

### Finding Description

**Code path:** `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()`, lines 59–92.

**Root cause — three independent termination paths all fail when `endTime == null`:**

1. **`pastEndTime()` (lines 123–131):** When `filter.getEndTime() == null`, the method unconditionally returns `Flux.never()`. The `takeUntilOther(pastEndTime(topicContext))` operator therefore never fires.

2. **`isComplete()` (lines 203–215):** The very first branch returns `false` when `endTime == null`, so the safety-check filter `!topicContext.isComplete()` is always `true`, but the safety check itself only fires once (after 1 s), returns empty for a messageless topic, and then completes — it does not repeat.

3. **`take(filter.getLimit())` (lines 83–85):** `hasLimit()` returns `limit > 0` (line 39–41 of `TopicMessageFilter.java`), so `Long.MAX_VALUE` passes as a valid limit and `flux.take(Long.MAX_VALUE)` is applied — which is functionally infinite.

**Exploit flow:**

- `historical = topicMessageRetriever.retrieve(filter, true)` → completes empty (no messages).
- `safetyCheck` fires after 1 s → `missingMessages(topicContext, null)` → `topicMessageRetriever.retrieve(gapFilter, false)` → completes empty.
- `live = incomingMessages(topicContext)` → calls `topicListener.listen(newFilter)` → blocks indefinitely waiting for messages that never arrive.
- `Flux.merge(safetyCheck, live).takeUntilOther(Flux.never())` → never terminates.
- The gRPC stream stays permanently open; `subscriberCount` increments and never decrements.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java` line 14) limits streams **per connection**, not total connections. An attacker opens N connections × 5 streams = 5N permanent streams.
- `subscriberCount` is a Micrometer gauge (lines 52–56) — it is a metric, not an enforced cap.
- `RetrieverProperties.timeout = 60 s` applies only to the historical retriever phase, which completes before the live phase begins.
- No idle-stream timeout, no maximum subscription duration, and no per-IP/per-client rate limit exist in the application layer.

### Impact Explanation
Each permanently open stream holds: one gRPC HTTP/2 stream slot, one `boundedElastic` thread (for the safety-check `subscribeOn`), and one active `TopicListener` subscription (Redis or polling). An attacker opening thousands of connections exhausts the bounded elastic thread pool, the Netty event-loop queue, and heap memory, causing a full denial-of-service of the mirror node's gRPC endpoint. All legitimate HCS topic subscribers are affected. This is a read-only mirror node, so funds are not directly frozen, but downstream clients (wallets, dApps) that depend on topic message streaming lose service.

### Likelihood Explanation
The attack requires zero privileges — the gRPC `subscribeTopic` RPC is publicly accessible. The parameters (`startTime=0`, no `endTime`, `limit=Long.MAX_VALUE`) are all within the validation constraints (`@Min(0)` on both fields, `null` allowed for `endTime`). The attack is trivially scriptable with any gRPC client (e.g., `grpcurl` or the Hedera SDK) and is fully repeatable. A single attacker machine can sustain hundreds of connections.

### Recommendation

1. **Enforce a maximum subscription duration:** Add a `maxSubscriptionDuration` property to `GrpcProperties` and apply `flux.timeout(maxSubscriptionDuration)` unconditionally in `subscribeTopic()`, regardless of whether `endTime` is set.

2. **Enforce a total subscriber cap:** Replace the `AtomicLong subscriberCount` gauge with an enforced limit — reject new subscriptions with an `RESOURCE_EXHAUSTED` gRPC status when the count exceeds a configurable threshold.

3. **Add an idle-stream timeout:** If no message is emitted within a configurable idle window (e.g., 5 minutes), terminate the stream with a normal completion or a retriable status code.

4. **Cap `limit`:** Add a `@Max` constraint on `TopicMessageFilter.limit` to prevent `Long.MAX_VALUE` from being accepted as a valid limit.

5. **Add per-connection total limits at the Netty layer:** Configure `maxConnectionAge` and `maxConnectionIdle` on the `NettyServerBuilder` in `GrpcConfiguration.java` to bound connection lifetime independently of application logic.

### Proof of Concept

```bash
# Prerequisites: grpcurl installed, mirror node gRPC on localhost:5600,
# topic 0.0.12345 exists but has zero messages.

# Open 500 permanent streams across 100 connections (5 streams each):
for i in $(seq 1 100); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
      "consensusStartTime": {"seconds": 0, "nanos": 0},
      "limit": 9223372036854775807
    }' localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: subscriberCount metric climbs to 500 and never decrements.
# Observe: boundedElastic thread pool saturates.
# Observe: legitimate subscribers receive UNAVAILABLE or hang.
```
