### Title
Missing DER Decoding for ECDSA_SECP256K1 Public Key Filter Causes Incomplete Account Query Results

### Summary
The mirror node REST API decodes DER-encoded ED25519 public keys submitted via `account.publickey` filter, but has no equivalent decoding for DER-encoded ECDSA_SECP256K1 keys. Because the database stores raw key bytes (not DER), submitting a DER-encoded ECDSA_SECP256K1 key causes the SQL comparison to fail silently, returning zero results even when matching accounts exist.

### Finding Description
**Code path:**
- `rest/accounts.js:344` — `utils.parsePublicKeyQueryParam(req.query, 'public_key')` is called to build the SQL filter.
- `rest/utils.js` — `parsePublicKey()` calls `ed25519.derToEd25519()` to attempt DER decoding.
- `rest/ed25519.js:8-31` — `derToEd25519` parses the DER structure and checks for OID `1.3.101.112` (ED25519). If the OID does not match (as with ECDSA_SECP256K1), it returns `null`.
- When `derToEd25519` returns `null`, `parsePublicKey` falls through and returns the input string unchanged — the full DER-encoded hex blob.
- The SQL query then does `WHERE public_key = '<DER hex>'` against a column that stores only the raw 33-byte compressed key hex.
- No row matches → empty result set.

**Root cause:** `derToEd25519` is the only DER decoder; it is ED25519-specific. There is no `derToEcdsa` or equivalent. The failed assumption is that all submitted public keys are either raw hex or ED25519 DER — ECDSA_SECP256K1 DER is silently passed through unmodified.

**Why existing checks fail:** Input validation only checks that the value is a valid hex string (or DER-parseable). It does not reject DER-encoded ECDSA keys, so the malformed (from the DB's perspective) value reaches the SQL query unchanged.

### Impact Explanation
Any unprivileged user querying `/api/v1/accounts?account.publickey=<ECDSA_SECP256K1 DER hex>` receives an empty account list, even when accounts with that key exist. This is a data completeness/integrity failure: the API silently returns incorrect results. Downstream consumers (wallets, explorers, auditing tools) relying on this filter for ECDSA accounts would receive misleading "no accounts found" responses. Severity: Medium — no data exposure or privilege escalation, but silent data omission for a documented filter parameter.

### Likelihood Explanation
Any external user can trigger this with zero privileges. ECDSA_SECP256K1 DER encoding is a standard format produced by many cryptographic libraries (OpenSSL, BouncyCastle, etc.). A developer integrating against the mirror node API who uses a standard library to export their public key in DER format would naturally hit this. The inconsistency (ED25519 DER works, ECDSA DER does not) makes it a realistic accidental trigger, not just a deliberate attack.

### Recommendation
Add a `derToEcdsaSecp256k1` function analogous to `derToEd25519` in `rest/ed25519.js` (or a shared module). In `parsePublicKey` in `rest/utils.js`, after `derToEd25519` returns `null`, attempt ECDSA_SECP256K1 DER decoding (check for OID `1.2.840.10045.3.1.7` / secp256k1 `1.3.132.0.10`). If the decoded raw key is 33 bytes (compressed), use it as the filter value. Add a corresponding test case for DER-encoded ECDSA_SECP256K1 in `utils.test.js` and a URL entry in `publickey.json`.

### Proof of Concept
1. Create an account on Hedera with an ECDSA_SECP256K1 key. Raw compressed public key (33 bytes hex): e.g., `023673dc1b97f1e2a99041867be087ec040374636b8e7486789112e95f09018401`
2. Encode it as DER/SubjectPublicKeyInfo (secp256k1 OID): `3036301006072a8648ce3d020106052b8104000a032200023673dc1b97f1e2a99041867be087ec040374636b8e7486789112e95f09018401`
3. Query: `GET /api/v1/accounts?account.publickey=3036301006072a8648ce3d020106052b8104000a032200023673dc1b97f1e2a99041867be087ec040374636b8e7486789112e95f09018401`
4. **Expected:** Account returned (key matches). **Actual:** Empty `accounts: []` — the DER blob does not match the raw hex stored in `entity.public_key`.
5. Confirm the account exists: `GET /api/v1/accounts?account.publickey=023673dc1b97f1e2a99041867be087ec040374636b8e7486789112e95f09018401` → account is returned correctly with raw key.

const derToEd25519 = function (der) {
  try {
    const buf = new Uint8Array(Buffer.from(der, 'hex')).buffer;
    const asn = asn1js.fromBER(buf);
    if (asn.offset === -1) {
      return null; // Not a valid DER/BER format
    }

    const asn1Result = asn.result.toJSON();

    // Check if it is a ED25519 key
    if (asn1Result.valueBlock.value.length < 1 || asn1Result.valueBlock.value[0].valueBlock.value.length < 1) {
      return null;
    }
    const {valueBlock} = asn1Result.valueBlock.value[0].valueBlock.value[0];
    if (valueBlock.blockName === BLOCK_NAME && valueBlock.value === ID) {
      const ed25519Key = asn1Result.valueBlock.value[1].valueBlock.valueHex;
      return ed25519Key.toLowerCase();
    }
    return null;
  } catch (err) {
    return null;
  }
};
