### Title
Unprivileged DB Connection Pool Exhaustion Terminates gRPC Streams via UNAVAILABLE with No Server-Side Recovery

### Summary
An unprivileged external user can open a large number of concurrent gRPC subscriptions to exhaust the database connection pool, causing Spring to throw `CannotGetJdbcConnectionException` (a subclass of `NonTransientDataAccessResourceException`). `ProtoUtil.toStatusRuntimeException()` maps this to `Status.UNAVAILABLE`, which is propagated to `responseObserver::onError`, permanently terminating all active subscriber streams. No server-side retry or recovery mechanism exists at the controller level, so any records in-flight or pending delivery at the time of termination are silently missed until clients manually reconnect.

Spring's `CannotGetJdbcConnectionException` (thrown when the JDBC/R2DBC connection pool is exhausted) extends `DataAccessResourceFailureException` â†’ `NonTransientDataAccessResourceException`. It therefore hits this branch and is mapped to `UNAVAILABLE`.

**Code path 2 â€” Stream termination with no retry:**
`grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, lines 43â€“48:
```java

        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```
`onErrorMap` converts the exception to a `StatusRuntimeException(UNAVAILABLE)`, which is then delivered to `responseObserver::onError`. This permanently terminates the gRPC stream. There is no `.retry()`, `.retryWhen()`, or any other recovery operator in the controller pipeline.

**Code path 3 â€” SharedPollingTopicListener retry does NOT protect individual subscribers:**
`grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedPollingTopicListener.java`, line 51:
```java
.retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))

```
The `.retryWhen()` is applied to the shared upstream flux *before* `.share()`. When the upstream errors, all downstream subscribers (individual gRPC streams) receive the error signal and are terminated before the retry resubscribes the upstream. The retry only restarts the internal polling loop; it does not prevent individual subscriber streams from being terminated.

**Root cause:** `CannotGetJdbcConnectionException` is semantically a transient condition (pool exhaustion resolves once connections are freed), but it is classified as `NonTransientDataAccessResourceException` in Spring's hierarchy, causing it to be mapped to the non-retryable `UNAVAILABLE` status. Combined with the absence of any retry/recovery at the controller layer, this creates a permanent stream termination on a condition that is externally triggerable.

### Impact Explanation
Every active `subscribeTopic` gRPC stream is terminated with `UNAVAILABLE` when the DB pool is exhausted. Records that were queued for delivery but not yet sent are dropped. Clients that do not implement reconnect-with-resume logic (tracking the last received sequence number) will permanently miss those records. For the mirror node's own export pipeline, this constitutes a gap in the exported record set. Severity is **High** for availability and **Medium** for data integrity (records are not lost from the DB, but they are missed by subscribers that do not reconnect correctly).

### Likelihood Explanation
The gRPC endpoint (`subscribeTopic`) requires no authentication. An attacker needs only a network path to the gRPC port and the ability to open many concurrent TCP connections. Each subscription triggers at least one DB query (entity existence check via `entityRepository.findById`, historical retrieval via `topicMessageRetriever.retrieve`). Opening hundreds of concurrent subscriptions with no rate limiting is trivially achievable with standard gRPC client libraries. The attack is repeatable and requires no special privileges.

### Recommendation
1. **Fix the exception mapping:** `CannotGetJdbcConnectionException` (pool exhaustion) is transient. Either catch it explicitly before `NonTransientDataAccessResourceException` and map it to `RESOURCE_EXHAUSTED`, or add a dedicated branch:
   ```java
   } else if (t instanceof CannotGetJdbcConnectionException) {
       return serverError(t, Status.RESOURCE_EXHAUSTED, DB_ERROR);
   ```
2. **Add server-side retry in the controller:** Apply `.retryWhen(Retry.backoff(...).filter(isTransient))` in `ConsensusController.subscribeTopic()` before `onErrorMap` to transparently recover transient DB errors without terminating the client stream.
3. **Add gRPC connection/request rate limiting:** Enforce per-IP or global concurrent-stream limits using a gRPC server interceptor to prevent pool exhaustion from being triggered externally.
4. **Tune connection pool size** relative to the maximum number of concurrent gRPC subscribers to reduce the blast radius.

### Proof of Concept
1. Stand up the mirror node gRPC service with a small DB connection pool (e.g., `maximumPoolSize=5`).
2. Using a gRPC client (e.g., `grpcurl` or a custom Java client), open 50+ concurrent `subscribeTopic` streams simultaneously, each with a valid topic ID and `startTime=0` (no end time, so streams stay open).
3. Each stream triggers `entityRepository.findById` and `topicMessageRetriever.retrieve`, competing for DB connections.
4. Once the pool is exhausted, `CannotGetJdbcConnectionException` is thrown, propagated through `onErrorMap(ProtoUtil::toStatusRuntimeException)` as `UNAVAILABLE`, and delivered to all active `responseObserver::onError` callbacks.
5. Observe that all streams terminate with `UNAVAILABLE: Error querying the data source. Please retry later`.
6. Any topic messages written to the DB between the last `onNext` delivery and the stream termination are not delivered to the clients; clients that do not reconnect miss those records permanently.
