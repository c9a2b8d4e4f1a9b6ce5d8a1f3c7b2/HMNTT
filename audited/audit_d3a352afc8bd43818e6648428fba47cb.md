### Title
Missing `deleted is not true` Filter in `entity_history` Branch of `findActiveByEvmAddressAndTimestamp`

### Summary
The SQL query in `findActiveByEvmAddressAndTimestamp` applies a `deleted is not true` guard only to the `entity` table branch, leaving the `entity_history` branch unguarded. When an entity has undergone a create → delete → re-create → delete lifecycle, the `entity_history` table contains a row with `deleted = true`; a caller supplying a `blockTimestamp` equal to the nanosecond of the first deletion event causes that row to win the `UNION ALL / ORDER BY timestamp_range DESC LIMIT 1` race, returning a deleted entity to `CommonEntityAccessor` as if it were active. The sibling query `findActiveByIdAndTimestamp` correctly applies `and deleted is not true` in both branches, confirming this is an unintentional omission.

### Finding Description

**Exact code location**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`) and lines 95–121 (`findActiveByEvmAddressOrAliasAndTimestamp`). [1](#0-0) 

The `entity` branch (lines 68–72) carries `e.deleted is not true`. The `entity_history` branch (lines 75–81) carries only `lower(eh.timestamp_range) <= ?2` — no deleted guard. [2](#0-1) 

Compare with `findActiveByIdAndTimestamp`, which correctly guards both branches: [3](#0-2) 

**Root cause**

The `entity_history` table stores every historical snapshot of an entity, including the snapshot recorded at the moment of deletion (`deleted = true`). When an entity is deleted at T2 and later re-created at T3, the importer writes `{deleted=true, timestamp_range=[T2,T3)}` into `entity_history`. The missing filter allows this row to be selected and returned.

**Exploit flow (step by step)**

State setup (all publicly observable on-chain):

| Time | `entity` table | `entity_history` table |
|------|---------------|----------------------|
| T1 | `{deleted=false, ts=[T1,∞)}` | — |
| T2 (delete) | `{deleted=true, ts=[T2,∞)}` | `{deleted=false, ts=[T1,T2)}` |
| T3 (re-create) | `{deleted=false, ts=[T3,∞)}` | + `{deleted=true, ts=[T2,T3)}` |
| T4 (delete again) | `{deleted=true, ts=[T4,∞)}` | + `{deleted=false, ts=[T3,T4)}` |

Attacker issues `eth_call` (or `eth_estimateGas`) with `blockTimestamp = T2`:

1. **`entity_cte`**: `created_timestamp = T1 ≤ T2` → entity ID resolved. (The importer preserves the original `created_timestamp` across re-creations, confirmed by `onEntityPersistOriginalHistoryCreatedTimestamp`.)
2. **`entity` branch**: current row has `deleted=true` → filtered out → empty.
3. **`entity_history` branch**: candidates with `lower(ts) ≤ T2`:
   - `{deleted=false, ts=[T1,T2)}` (lower = T1)
   - `{deleted=true, ts=[T2,T3)}` (lower = T2 ≤ T2) ← **no deleted guard**
   - `ORDER BY lower(ts) DESC LIMIT 1` picks `{deleted=true, ts=[T2,T3)}`.
4. **`UNION ALL … ORDER BY timestamp_range DESC LIMIT 1`**: only one row → returns `{deleted=true, ts=[T2,T3)}`.
5. **`CommonEntityAccessor.getEntityByEvmAddressTimestamp`** returns this entity as a non-empty `Optional<Entity>` without inspecting `deleted`. [4](#0-3) 

The EVM layer receives a present, non-null entity and treats the address as an existing, active account at that historical block.

**Why existing checks fail**

The only test covering a deleted `entity_history` row (`findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall`) uses `getTimestampLower() - 1`, which is strictly before the deletion record, so the `entity_history` branch never selects it. No test covers `blockTimestamp ≥ getTimestampLower()` for a deleted history row paired with a matching `entity` row — the exact condition required for exploitation. [5](#0-4) 

### Impact Explanation

During historical `eth_call` / `eth_estimateGas` queries, a smart contract can observe a deleted account as active at the nanosecond of its deletion. This affects `EXTCODESIZE`, `BALANCE`, and `CALL` opcode results for that address at that timestamp, producing a falsified view of historical chain state. Contracts that use historical state proofs, audit trails, or replay-based verification logic can be fed incorrect data. The same flaw exists in `findActiveByEvmAddressOrAliasAndTimestamp` (lines 95–121), widening the attack surface to alias-addressed lookups. [6](#0-5) 

### Likelihood Explanation

No privilege is required. Any caller of the public JSON-RPC `eth_call` endpoint can supply an arbitrary `blockNumber` (mapped to `blockTimestamp`). The deletion and re-creation timestamps of any account are publicly visible in the mirror node's REST API and transaction history. The lifecycle precondition (delete + re-create + delete) occurs naturally for recycled contract addresses and is not rare on mainnet. The attacker needs only the EVM address and the nanosecond-precision deletion timestamp, both of which are freely available.

### Recommendation

Add `and eh.deleted is not true` to the `entity_history` branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- findActiveByEvmAddressAndTimestamp  (line 75-81)
select *
from entity_history eh
where lower(eh.timestamp_range) <= ?2
  and eh.id = (select id from entity_cte)
  and eh.deleted is not true          -- ADD THIS
order by lower(eh.timestamp_range) desc
limit 1
```

Apply the identical fix to `findActiveByEvmAddressOrAliasAndTimestamp` (lines 110–117). Add regression tests that pair a deleted `entity_history` row with a matching `entity` row and assert empty results for `blockTimestamp ≥ getTimestampLower()`.

### Proof of Concept

```
Preconditions (reproducible in EntityRepositoryTest):

1. Insert entity row:
   id=42, evm_address=0xABCD, created_timestamp=T1,
   deleted=true, timestamp_range=[T4,∞)

2. Insert entity_history rows:
   {id=42, deleted=false, timestamp_range=[T1,T2)}
   {id=42, deleted=true,  timestamp_range=[T2,T3)}   ← target row
   {id=42, deleted=false, timestamp_range=[T3,T4)}

Trigger:
   entityRepository.findActiveByEvmAddressAndTimestamp(0xABCD, T2)

Expected (correct): Optional.empty()
Actual (buggy):     Optional.of(Entity{deleted=true, timestamp_range=[T2,T3)})

The returned non-empty Optional causes CommonEntityAccessor to surface
the deleted account as active to the EVM, falsifying historical state
for any eth_call targeting block timestamp T2.
```
