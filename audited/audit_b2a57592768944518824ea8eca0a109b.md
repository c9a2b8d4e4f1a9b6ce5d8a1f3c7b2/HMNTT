### Title
Unauthenticated Pagination Amplification via `limit=1` Exhausts DB Connection Pool in `getAccountCryptoAllowances`

### Summary
The `getAccountCryptoAllowances` endpoint in `rest/controllers/cryptoAllowanceController.js` unconditionally emits a `next` pagination link whenever `response.allowances.length === limit`, with no enforced minimum on `limit`. Because the REST API has no rate-limiting middleware and the default DB connection pool is only 10 connections, an unauthenticated attacker can drive N concurrent pagination chains with `limit=1` to hold all pool connections simultaneously, starving legitimate requests and rendering the REST API unavailable.

### Finding Description

**Exact code path:**

`rest/controllers/cryptoAllowanceController.js`, `getAccountCryptoAllowances()`, lines 76–98:

```js
// line 80 — one DB query per HTTP request, no batching
const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(conditions, params, order, limit);
// ...
// line 89 — next link emitted whenever result set fills the page
if (response.allowances.length === limit) {
  const lastRow = last(response.allowances);
  const lastValues = { [filterKeys.SPENDER_ID]: lastRow.spender };
  response.links.next = utils.getPaginationLink(req, false, lastValues, order);
}
```

`extractCryptoAllowancesQuery` (lines 26–68) accepts any caller-supplied `limit` value with no lower-bound check — `limit=1` is fully valid.

**DB pool:**

`rest/dbpool.js` line 14: `max: config.db.pool.maxConnections` — default is **10** connections (`docs/configuration.md` line 556).

**No rate limiting in the REST API:**

`rest/server.js` middleware stack (lines 68–144) contains: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — **no rate-limiter**. The `ThrottleConfiguration`/`ThrottleManagerImpl` exist only in the `web3` Java module, not in the Node.js REST service.

**Auth is optional and non-blocking:**

`rest/middleware/authHandler.js` lines 18–20: `if (!credentials) { return; }` — anonymous requests pass through without restriction.

**Root cause / failed assumption:** The design assumes that pagination with small `limit` values is benign because each query is cheap. The failed assumption is that no attacker will drive many concurrent chains simultaneously. With `limit=1` and an account holding N allowances, retrieving all N allowances requires N sequential DB queries instead of 1 (with `limit=100`). Running K such chains concurrently multiplies DB load by K×N, and with only 10 pool connections the pool saturates immediately.

### Impact Explanation

When all 10 pool connections are held by attacker requests, every legitimate REST API query blocks until `connectionTimeoutMillis` (default 20 000 ms) elapses and then returns a 500 error. This makes the entire mirror-node REST API unavailable to all users — including downstream services, wallets, and explorers that depend on it — for as long as the attack is sustained. The attack is self-sustaining: each completed attacker request immediately issues the next paginated request, keeping connections occupied continuously.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only knowledge of one account ID (any account with at least one allowance). The `next` link in the response body directly supplies the URL for the next request, so the attacker does not even need to construct pagination parameters manually. The attack is trivially scriptable with any HTTP client that supports concurrency (e.g., `curl`, `ab`, Python `asyncio`). It is repeatable indefinitely and leaves no persistent state to clean up.

### Recommendation

1. **Enforce a minimum `limit`** (e.g., ≥ 1 is fine, but pair it with the measures below).
2. **Add per-IP rate limiting** to the REST API middleware stack (e.g., `express-rate-limit`) before route handlers, covering all paginated endpoints.
3. **Increase the default DB pool size** or use a connection proxy (PgBouncer in transaction mode is already configured for `mirror_node` user per `charts/hedera-mirror/values.yaml` line 437, but the REST service should be routed through it).
4. **Cap concurrent in-flight DB queries per IP** at the application layer, or use a semaphore/queue in front of `CryptoAllowanceService.getAccountCryptoAllowances`.

### Proof of Concept

```bash
# Step 1: find any account with allowances, e.g. 0.0.12345
# Step 2: launch 10 concurrent pagination chains with limit=1
for i in $(seq 1 10); do
  (
    URL="https://<mirror-node>/api/v1/accounts/0.0.12345/allowances/crypto?limit=1"
    while [ -n "$URL" ]; do
      RESP=$(curl -s "$URL")
      URL=$(echo "$RESP" | jq -r '.links.next // empty')
      [ -n "$URL" ] && URL="https://<mirror-node>$URL"
    done
  ) &
done
wait

# Step 3: while the above runs, observe that legitimate requests time out:
curl -v "https://<mirror-node>/api/v1/accounts/0.0.99999/allowances/crypto"
# Expected: 500 or connection timeout after 20s — pool exhausted
