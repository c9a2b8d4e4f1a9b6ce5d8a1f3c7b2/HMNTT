### Title
Account ID Accepted as Contract ID in `/contracts/:contractId/results/:consensusTimestamp`, Exposing Contract Execution History

### Summary
The `getContractResultsByTimestamp()` handler validates the `:contractId` path parameter only for entity-ID format, not for entity type. Because the `contract_transaction` table intentionally stores payer account IDs as `entity_id` rows alongside true contract IDs, an unprivileged attacker can supply any account ID that was a payer in a contract transaction and receive the full contract execution result (call output, logs, state changes, gas data) for that timestamp.

### Finding Description
**Code path:**

1. Route registration — `rest/routes/contractRoute.js` line 18:
   ```
   router.getExt('/:contractId/results/:consensusTimestamp', ContractController.getContractResultsByTimestamp);
   ```

2. Validation — `rest/controllers/contractController.js` lines 351–363 (`validateContractIdAndConsensusTimestampParam`):
   ```js
   if (!EntityId.isValidEntityId(contractId)) {
     params.push(filterKeys.CONTRACTID);
   }
   ```
   Only format is checked (`0.0.X`, `0.X`, `X`). No check that the resolved entity is of type `CONTRACT`.

3. Lookup — `rest/service/contractService.js` lines 201–206 (`involvedContractsQuery`):
   ```sql
   SELECT ... FROM contract_transaction
   WHERE consensus_timestamp = $1 AND entity_id = $2
   ```
   No type filter on `entity_id`.

4. **Root cause — table population.** The `contract_transaction` table is populated by the importer with a row for *every* involved entity, including the payer account:

   - Migration (`importer/src/main/resources/db/migration/v1/V1.89.0.1__add_contract_lookup_tables.sql` lines 39–57):
     ```sql
     array_agg(cr.contract_id) || cr.payer_account_id  -- payer account included in contract_ids
     ...
     INSERT INTO contract_transaction(consensus_timestamp, entity_id, ...)
     SELECT consensus_timestamp, unnest(contract_ids), ...  -- one row per entity, including payer
     ```
   - Live importer (`importer/src/main/java/org/hiero/mirror/importer/domain/ContractResultServiceImpl.java` lines 264–266):
     ```java
     recordItem.addContractTransaction(contractEntityId);
     recordItem.addContractTransaction(contractResult.getPayerAccountId()); // account, not contract
     ```

   This means for every contract transaction at timestamp `T` with payer account `A`, a row `(T, A, [...], A)` exists in `contract_transaction`. Querying with `entity_id = A` at timestamp `T` returns a valid hit.

5. After the lookup succeeds, `getDetailedContractResults()` (`rest/controllers/contractController.js` lines 1205–1219) fetches and returns the full execution payload: call result, ETH transaction, record file, contract logs, and state changes.

### Impact Explanation
An attacker who knows (or enumerates) a payer account ID and a consensus timestamp can call:
```
GET /contracts/0.0.{payerAccountId}/results/{timestamp}
```
and receive the complete contract execution record for that transaction — including EVM call output bytes, emitted logs, storage slot changes, and gas metrics. The endpoint is semantically scoped to contracts; returning full execution data keyed on an account ID violates that boundary and constitutes unintended information disclosure of historical Hashgraph execution data. While Hedera is a public ledger, this bypasses the intended access model of the `/contracts/` namespace and can be used to enumerate all contract transactions an account participated in as payer, along with their full execution details, without knowing any contract ID.

### Likelihood Explanation
No authentication or privilege is required. The only precondition is knowing a valid account ID and a consensus timestamp at which that account paid for a contract transaction — both are trivially discoverable from the public transaction history API (`/api/v1/transactions`). The exploit is deterministic and repeatable for every such (account, timestamp) pair in the ledger history.

### Recommendation
Add an entity-type check after resolving the `contractId` parameter. In `getAndValidateContractIdAndConsensusTimestampPathParams` (or immediately after `computeContractIdFromString`), verify that the resolved entity has type `CONTRACT` before proceeding. Alternatively, add a `JOIN` to the `entity` table in `involvedContractsQuery` filtering on `type = CONTRACT`, or add a separate pre-flight query that rejects non-contract entity IDs with a `404`/`400` before the `contract_transaction` lookup.

### Proof of Concept
**Preconditions:** Public mirror node instance; knowledge of any account ID `A` (e.g., `0.0.950`) that paid for a contract transaction.

1. Discover a consensus timestamp `T` where account `0.0.950` was the payer of a contract call:
   ```
   GET /api/v1/transactions?account.id=0.0.950&type=CONTRACTCALL&limit=1
   → consensus_timestamp: "1680000000.123456789"
   ```

2. Call the contract results endpoint using the **account ID** (not a contract ID):
   ```
   GET /api/v1/contracts/0.0.950/results/1680000000.123456789
   ```

3. **Expected:** `404 Not Found` (account is not a contract).
   **Actual:** `200 OK` with full `ContractResultDetailsViewModel` — call result bytes, emitted logs, state changes, gas used, ETH transaction details — for the contract execution that account `0.0.950` paid for at that timestamp.
