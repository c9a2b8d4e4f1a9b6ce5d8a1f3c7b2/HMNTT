### Title
Silent Empty Result via Combined Range + Equality Filter on `token.id` in `getTokenRelationships()`

### Summary
An unprivileged user can supply both `token.id=gt:X` and `token.id=eq:Y` simultaneously to `GET /accounts/{id}/tokens`. The controller separates these into two distinct arrays (`conditions` and `inConditions`) which are then unconditionally ANDed together in the SQL query. When `Y <= X`, the query silently returns an empty result set even though token `Y` is associated with the account, producing a false negative that contradicts the semantics of the `eq` filter.

### Finding Description

**Controller layer** (`rest/controllers/tokenController.js`, lines 28–48):

`extractTokensRelationshipQuery()` routes `eq` filters to `inConditions` and all range operators (`gt`, `lt`, `gte`, `lte`) to `conditions`. There is no check that prevents both from being populated for the same field simultaneously. Only the `ne` operator is rejected:

```js
if (utils.opsMap.ne === filter.operator) {
  throw new InvalidArgumentError(...);   // only guard
}
if (utils.opsMap.eq === filter.operator) {
  inConditions.push(...);   // eq → IN list
} else {
  conditions.push(...);     // range → AND clause
}
```

**Service layer** (`rest/service/tokenService.js`, lines 46–87):

`getTokenRelationshipsQuery()` appends both clauses to the base query with no conflict detection:

```js
const sqlQuery = [
  TokenService.tokenRelationshipsQuery,  // base WHERE account_id=$1 AND associated=true
  conditionClause,                        // AND token_id > $3
  inConditionClause,                      // AND token_id IN (Y)
  orderClause,
  limitClause,
].join('\n');
```

`getInClauseSubQuery()` (line 85) also directly interpolates values into SQL without parameterization:

```js
return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${tokenIdInParams})`;
```

**Resulting SQL for `?token.id=gt:1000000&token.id=eq:500`:**

```sql
WHERE ta.account_id = $1 AND ta.associated = true
AND ta.token_id > $3          -- 1000000
AND ta.token_id IN (500)      -- directly interpolated, not parameterized
```

`token_id > 1000000 AND token_id IN (500)` is always false when `500 <= 1000000`, so the result is an empty set. No error is raised.

### Impact Explanation

A smart contract or off-chain service using this endpoint to verify whether an account holds a specific token (via `token.id=eq:Y`) will receive an empty response and incorrectly conclude the token is not associated. If the caller also passes a pagination cursor (`token.id=gt:cursor`) alongside an equality check — a plausible pattern for cursor-based verification — the false negative can cause the contract to deny a legitimate token holder access to a service, block a transfer, or misreport state. Severity is medium: no funds are directly drained, but protocol logic relying on this API can be silently subverted.

### Likelihood Explanation

No authentication is required. Any external caller can craft the URL. The pattern of combining a pagination range filter with a specific token lookup is realistic in smart contract mirror-node integrations. The attack is trivially repeatable and deterministic — the same inputs always produce the same empty result.

### Recommendation

1. **Detect and reject conflicting filters**: In `extractTokensRelationshipQuery()`, if `inConditions` is non-empty and `conditions` also contains a `token_id` range filter, throw an `InvalidArgumentError` explaining that `eq` and range operators cannot be combined for `token.id`.
2. **Alternatively, treat `eq` as authoritative**: If `inConditions` is non-empty, ignore range `conditions` for `token_id` entirely, since an exact-match lookup makes a range filter semantically meaningless.
3. **Parameterize the IN clause**: Replace the direct interpolation in `getInClauseSubQuery()` with proper `$N` placeholders pushed into `params` to eliminate the unparameterized SQL construction.

### Proof of Concept

```
# Token 0.0.500 IS associated with account 0.0.100.
# Expect: returns token 0.0.500
GET /api/v1/accounts/0.0.100/tokens?token.id=eq:0.0.500
→ { tokens: [{ token_id: "0.0.500", ... }] }   ✓

# Same token, but add a range filter with a higher lower-bound.
# Expect (documented eq semantics): still returns token 0.0.500
# Actual: empty result — false negative
GET /api/v1/accounts/0.0.100/tokens?token.id=gt:0.0.1000000&token.id=eq:0.0.500
→ { tokens: [] }   ✗  (silent false negative)
```

The generated SQL is:
```sql
WHERE ta.account_id = 100 AND ta.associated = true
AND ta.token_id > 1000000
AND ta.token_id IN (500)
-- 500 > 1000000 is false → empty result set
```
