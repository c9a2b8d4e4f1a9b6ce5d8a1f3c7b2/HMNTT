### Title
Unauthenticated Repeated Requests to `getTopic()` Cause Unbounded DB Load via Missing Caching and Rate Limiting

### Summary
The `getTopic()` endpoint in `TopicController` executes three uncached, sequential database queries on every request with no rate limiting applied to the `rest-java` module. Any unauthenticated external user can flood `GET /api/v1/topics/{id}` with a valid topic ID, forcing the server to re-execute all three repository lookups on every call, causing sustained and unbounded database load increase well exceeding 30%.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 32–37:
```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());       // DB query 1
    var entity = entityService.findById(id.id());     // DB query 2
    var customFee = customFeeService.findById(id.id()); // DB query 3
    return topicMapper.map(customFee, entity, topic);
}
```

Each call chains to:
- `TopicServiceImpl.findById()` → `TopicRepository.findById()` — plain `CrudRepository`, zero `@Cacheable`
- `EntityServiceImpl.findById()` → `EntityRepository.findById()` — plain `CrudRepository`, zero `@Cacheable`
- `CustomFeeServiceImpl.findById()` → `CustomFeeRepository.findById()` — plain `CrudRepository`, zero `@Cacheable`

**Root cause:** The `rest-java` module defines no caching layer (`grep` across all `rest-java/src/main/java/**/*.java` returns zero hits for `@Cacheable`, `CacheManager`, or `EnableCaching`). Contrast this with the `grpc` module, which wraps its `EntityRepository.findById()` with `@Cacheable(cacheManager = ENTITY_CACHE, ...)` and a 24-hour Caffeine cache. The `rest-java` repositories are bare `CrudRepository` interfaces with no such protection.

**Rate limiting:** The only throttle implementation in the codebase (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) is scoped exclusively to the `web3` module's contract-call endpoints. The `rest-java` module's `WebMvcConfiguration`, `LoggingFilter`, and `RestJavaConfiguration` contain no rate-limiting logic whatsoever.

**Failed assumption about `ShallowEtagHeaderFilter`:** `RestJavaConfiguration` registers a `ShallowEtagHeaderFilter` on `/api/*`. This filter computes an ETag by buffering the full response body after it is generated — it does **not** short-circuit the controller or skip DB queries. All three `findById()` calls execute in full on every request before the ETag is even computed. The filter only saves downstream bandwidth; it provides zero protection against DB amplification.

### Impact Explanation
Each HTTP request to `GET /api/v1/topics/{id}` unconditionally issues 3 synchronous JDBC queries against the mirror-node PostgreSQL database (tables: `topic`, `entity`, `custom_fee`). An attacker sending N requests/second generates 3N DB queries/second with no upper bound. At modest concurrency (e.g., 200 req/s from a single machine using `wrk` or `ab`), this produces 600 DB queries/second against what is otherwise a read-heavy but cache-assisted workload. Because the rest of the system (grpc, importer) relies on the same shared database, this directly degrades all dependent services. The 30% resource consumption threshold is easily crossed without brute-force volume — sustained moderate-rate flooding suffices.

### Likelihood Explanation
The endpoint requires no authentication, no API key, and no session. The topic ID is a simple numeric path parameter (`EntityIdNumParameter`). Any attacker who can enumerate a single valid topic ID (trivially obtained from any public Hedera network explorer or by querying `/api/v1/topics` list endpoints) can immediately begin exploitation. The attack is repeatable, automatable with standard HTTP load tools, and requires no special privileges or knowledge beyond a valid topic ID.

### Recommendation
1. **Add response caching at the service or repository layer** in `rest-java`: annotate `TopicRepository.findById()`, `EntityRepository.findById()`, and `CustomFeeRepository.findById()` with `@Cacheable` backed by a short-TTL Caffeine cache (e.g., 10–30 seconds), mirroring the pattern already used in the `grpc` module's `EntityRepository`.
2. **Add rate limiting to the `rest-java` module**: introduce a `bucket4j`-based filter (analogous to `web3`'s `ThrottleConfiguration`) applied to `/api/v1/**` endpoints, enforcing a per-IP or global requests-per-second ceiling.
3. **Consider HTTP-level caching headers** (`Cache-Control: public, max-age=N`) so that a reverse proxy (nginx, CDN) can absorb repeated identical requests before they reach the application.

### Proof of Concept
```bash
# Step 1: Obtain a valid topic ID (e.g., from public mirror node)
TOPIC_ID=1234

# Step 2: Flood the endpoint with concurrent requests (no auth needed)
wrk -t8 -c200 -d60s "https://<mirror-node-host>/api/v1/topics/${TOPIC_ID}"

# Each request triggers 3 DB queries:
#   SELECT * FROM topic WHERE id = ?
#   SELECT * FROM entity WHERE id = ?
#   SELECT * FROM custom_fee WHERE entity_id = ?
#
# At 200 concurrent connections, expect 3x DB query amplification.
# Monitor DB CPU/query rate via pg_stat_activity or Prometheus metrics
# to confirm >30% increase over the 24-hour baseline.
```
