### Title
Deterministic HTTP 500 from `Mempool()` Enables Unprivileged Circuit Breaker Exhaustion Leading to Persistent Service Partition

### Summary
`Mempool()` in `rosetta/app/services/mempool_service.go` unconditionally returns `errors.ErrNotImplemented`, which the `coinbase/rosetta-sdk-go` SDK serializes as HTTP 500 per the Rosetta API specification. The Traefik circuit breaker configured in `charts/hedera-mirror-rosetta/values.yaml` monitors a global `ResponseCodeRatio(500, 600, 0, 600) > 0.25` threshold across all endpoints without excluding permanently-unimplemented ones. An unprivileged attacker can flood `/mempool` with requests, push the 5xx ratio above 25%, trip the circuit breaker, and then sustain the open state during recovery windows by continuing to send requests that deterministically return 500, causing a persistent denial of service for all Rosetta API users.

### Finding Description

**Exact code path:**

`rosetta/app/services/mempool_service.go`, lines 22–27:
```go
func (m *mempoolAPIService) Mempool(
    _ context.Context,
    _ *types.NetworkRequest,
) (*types.MempoolResponse, *types.Error) {
    return nil, errors.ErrNotImplemented
}
```
`errors.ErrNotImplemented` is defined in `rosetta/app/errors/errors.go` line 70 as:
```go
ErrNotImplemented = newError(NotImplemented, 111, false)
```
The `coinbase/rosetta-sdk-go` server (`server.NewMempoolAPIController`, wired at `rosetta/main.go` line 90) wraps any returned `*types.Error` and writes HTTP 500 — this is mandated by the Rosetta API specification and is unconditional.

**Circuit breaker configuration** (`charts/hedera-mirror-rosetta/values.yaml`, lines 150–163):
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true
```

**Root cause:** The circuit breaker expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` is evaluated across the entire service without any per-endpoint exclusion. Because `/mempool` and `/mempool/transaction` are registered live endpoints (not 404s) that always return HTTP 500, an attacker can deliberately inflate the 5xx ratio. The failed assumption is that the circuit breaker would only observe transient backend failures, not a permanently-unimplemented endpoint that is publicly reachable and returns 500 on every call.

**Exploit flow:**
1. Attacker sends POST requests to `/mempool` (no auth required, no body validation needed beyond a valid JSON `NetworkRequest`).
2. Each request returns HTTP 500 deterministically.
3. The `rateLimit` is keyed by `requestHost` (the Host header), not by client IP — it is a global 10 req/s cap shared by all clients. The attacker consumes a portion of this budget with 500-generating requests.
4. If the attacker's `/mempool` requests constitute >25% of all requests in Traefik's measurement window, `ResponseCodeRatio(500, 600, 0, 600) > 0.25` becomes true and the circuit breaker opens.
5. During the half-open recovery window, Traefik probes the backend with a subset of incoming requests. If any probe lands on `/mempool`, it returns 500, and the circuit breaker immediately re-opens.
6. The attacker sustains this by continuing to send `/mempool` requests during recovery windows, preventing the breaker from ever closing.

**Why existing checks fail:**
- `rateLimit: average: 10, sourceCriterion: requestHost: true` — keyed by Host header, not client IP. A single attacker sending 3 req/s to `/mempool` while legitimate traffic is 7 req/s produces a 30% 5xx ratio (3/10), sufficient to trip the breaker. The rate limit does not prevent this.
- `inFlightReq: amount: 5` per IP — limits concurrency, not rate; does not prevent sequential flooding.
- No per-endpoint circuit breaker exclusion exists anywhere in the middleware stack.
- No application-level rate limiting or request throttling exists inside the Go service itself.

### Impact Explanation
When the circuit breaker opens, Traefik returns HTTP 503 to **all** requests to the Rosetta service — not just `/mempool`. This means `/network/status`, `/block`, `/construction/submit`, and all other endpoints become unavailable. For a blockchain infrastructure operator relying on the Rosetta API for exchange integrations or automated transaction submission, this constitutes a complete service partition. The impact is a full denial of service for all Rosetta API consumers, achievable by a single unprivileged external attacker with no credentials.

### Likelihood Explanation
The attack requires only the ability to send HTTP POST requests to the public `/mempool` endpoint — no authentication, no special headers, no valid transaction data. The endpoint is listed in the ingress path configuration (`charts/hedera-mirror-rosetta/values.yaml` line 131: `"/rosetta/mempool"`), confirming it is publicly exposed. The threshold (>25% 5xx ratio) is reachable under any traffic condition where legitimate request volume is less than three times the attacker's request rate. The attack is trivially repeatable and can be sustained indefinitely by timing requests to coincide with circuit breaker recovery windows. A single attacker with a basic HTTP client can execute this.

### Recommendation
1. **Exclude `/mempool` from circuit breaker accounting** by routing it through a separate Traefik middleware chain that does not include the circuit breaker, or by returning HTTP 501 (Not Implemented) instead of HTTP 500 — though the Rosetta SDK controls the HTTP status, so this requires SDK-level changes or a custom response wrapper.
2. **Return a non-5xx response for known-unimplemented endpoints.** The Rosetta spec allows returning HTTP 500 for errors, but operators should consider wrapping the SDK router to intercept `ErrNotImplemented` responses and emit HTTP 501, which falls outside the `ResponseCodeRatio(500, 600, 0, 600)` window.
3. **Add per-client-IP rate limiting** (change `sourceCriterion` from `requestHost` to `ipStrategy`) so individual attackers cannot consume the global request budget.
4. **Add a path-based rate limit** specifically for `/mempool` and `/mempool/transaction` to cap their contribution to overall traffic.
5. **Consider removing the `/mempool` route entirely** from the online router in `rosetta/main.go` since it is permanently unimplemented, eliminating the attack surface.

### Proof of Concept

**Preconditions:** Rosetta service is publicly accessible at `https://TARGET/rosetta`. Legitimate traffic is low (< 30 req/s).

**Steps:**

```bash
# Step 1: Confirm /mempool always returns HTTP 500
curl -s -o /dev/null -w "%{http_code}" -X POST https://TARGET/rosetta/mempool \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
# Expected output: 500

# Step 2: Flood /mempool to push 5xx ratio above 25%
# Send 4 req/s while legitimate traffic is ~10 req/s → 4/14 = 28.5% > 25%
while true; do
  for i in $(seq 1 4); do
    curl -s -o /dev/null -X POST https://TARGET/rosetta/mempool \
      -H "Content-Type: application/json" \
      -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' &
  done
  sleep 1
done

# Step 3: Verify circuit breaker has opened (all endpoints now return 503)
curl -s -o /dev/null -w "%{http_code}" -X POST https://TARGET/rosetta/network/status \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
# Expected output: 503

# Step 4: Continue sending /mempool requests during recovery windows
# to prevent the circuit breaker from closing — sustaining the partition.
```

**Result:** All Rosetta API endpoints return 503 for the duration of the attack, with no recovery possible while the attacker continues sending requests.
