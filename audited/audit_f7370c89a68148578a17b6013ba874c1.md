### Title
Unauthenticated Pagination Abuse via `limit=1` Exhausts DB Connection Pool in `/accounts/:id/tokens`

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` accepts a user-controlled `limit` parameter as low as 1 and generates a `next` pagination link for every response that fills the page. The REST API has no per-IP or global rate-limiting middleware, so an unprivileged attacker can drive an unbounded number of sequential DB queries by following pagination links, or amplify impact with concurrent connections, exhausting the finite DB connection pool and degrading service for all users.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 19 registers the route: [1](#0-0) 

`rest/controllers/tokenController.js` `getTokenRelationships()` (lines 66–92) processes each request: [2](#0-1) 

Per request, the handler issues at minimum two DB round-trips:
1. `EntityService.getEncodedId` + `EntityService.isValidAccount` (lines 67–71)
2. `TokenService.getTokenAccounts(query)` → `getTokenRelationshipsQuery` + optional `getCachedTokens` (lines 74, 96–114 of `tokenService.js`) [3](#0-2) 

**Pagination link generation (root cause):** When `tokens.length === query.limit`, a `next` link is unconditionally emitted: [4](#0-3) 

**`limit` lower bound:** `getLimitParamValue` in `rest/utils.js` caps at `max` (default 100) but enforces no minimum above 1, so `limit=1` is accepted: [5](#0-4) 

**No rate limiting on the REST API:** `rest/server.js` registers only `authHandler`, `requestLogger`, optional metrics, and optional Redis response cache — no per-IP or global request-rate throttle: [6](#0-5) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter found in the codebase applies exclusively to the `web3` (contract-call) module, not to the REST API. [7](#0-6) 

**DB connection pool is finite:** `rest/dbpool.js` creates a pool bounded by `config.db.pool.maxConnections`: [8](#0-7) 

### Impact Explanation
An account with N token relationships requires N HTTP requests with `limit=1` to traverse. Each request holds a DB connection for the duration of its queries. With many concurrent attackers (or a single attacker with many parallel connections), the pool of DB connections is exhausted. Legitimate requests then queue or time out (`connectionTimeoutMillis`), causing service degradation or 503-equivalent errors for all users. Because the REST API is public and unauthenticated, no credentials or on-chain funds are required. The impact is availability degradation (griefing), matching the stated severity scope.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of any account ID with a non-trivial number of token associations (easily discoverable via the same API with `limit=100`). It is fully scriptable, repeatable, and requires no privileged access. Accounts on mainnet routinely hold hundreds to thousands of token associations, providing ample pagination depth. Multiple attackers from different IPs multiply the effect with no coordination needed.

### Recommendation
1. **Add per-IP rate limiting** to the REST API (e.g., `express-rate-limit`) applied globally in `rest/server.js` before route handlers.
2. **Enforce a minimum `limit`** of at least 10 (or the default 25) to reduce the query-per-relationship ratio.
3. **Enable the Redis response cache** (`config.cache.response.enabled`) so repeated identical paginated requests are served from cache rather than hitting the DB.
4. **Consider a maximum pagination depth** (e.g., reject requests where the cursor implies more than N pages have been fetched in a short window).

### Proof of Concept
```bash
# 1. Find an account with many token relationships
ACCOUNT="0.0.12345"
BASE="https://<mirror-node>/api/v1"

# 2. Start paginating with limit=1 in a tight loop (no auth required)
URL="${BASE}/accounts/${ACCOUNT}/tokens?limit=1&order=asc"
while [ "$URL" != "null" ]; do
  RESP=$(curl -s "$URL")
  NEXT=$(echo "$RESP" | jq -r '.links.next')
  URL="${BASE}${NEXT}"
done

# 3. Run the above loop from many concurrent clients simultaneously.
# Each iteration issues 2+ DB queries; with pool maxConnections exhausted,
# legitimate API requests begin timing out.
```

    const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
    const isValidAccount = await EntityService.isValidAccount(accountId);
    if (!isValidAccount) {
      throw new NotFoundError();
    }
    const filters = utils.buildAndValidateFilters(req.query, acceptedTokenParameters);
    const query = this.extractTokensRelationshipQuery(filters, accountId);
    const tokenRelationships = await TokenService.getTokenAccounts(query);
    const tokens = tokenRelationships.map((token) => new TokenRelationshipViewModel(token));

    let nextLink = null;
    if (tokens.length === query.limit) {
      const lastRow = last(tokens);
      const lastValue = {
        [filterKeys.TOKEN_ID]: lastRow.token_id,
      };
      nextLink = utils.getPaginationLink(req, false, lastValue, query.order);
    }

    res.locals[responseDataLabel] = {
      tokens,
      links: {
        next: nextLink,
      },
    };
  };

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
