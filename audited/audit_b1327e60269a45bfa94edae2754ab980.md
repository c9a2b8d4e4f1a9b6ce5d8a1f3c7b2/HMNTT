### Title
Unauthenticated Unbounded Historical Scan DoS via `getHookStorageChange()`

### Summary
The `getHookStorageChange()` method in `HookServiceImpl.java` executes a `DISTINCT ON (key)` SQL query against the `hook_storage_change` table with fully user-controlled timestamp and key ranges. No rate limiting exists on the `rest-java` hooks endpoint, and no authentication is required. An unprivileged attacker can flood the endpoint with maximum-range timestamp queries, forcing repeated full historical table scans and exhausting database resources.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` (line 104–130) → `HookServiceImpl.getHookStorage()` (line 56–82) → `HookServiceImpl.getHookStorageChange()` (line 84–118) → `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` (line 32–39).

**Root cause:**

In `HookStorageChangeRepository.java` lines 15–31, the query is:

```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key >= :keyLowerBound
  AND key <= :keyUpperBound
  AND consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound
```

`DISTINCT ON (key)` requires the database to scan **all rows** matching the timestamp range before deduplication and pagination can occur. The `LIMIT` clause (capped at 100 via `@Max(MAX_LIMIT)`) only limits rows *returned*, not rows *scanned*.

**Attacker-controlled inputs:**

In `HooksController.java` line 110–111, `timestamps` accepts up to 2 `TimestampParameter` values (`@Size(max = 2)`), but there is no restriction on the *width* of the range. An attacker supplies `timestamp=gte:0`, which sets `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE` (or current network time), causing a full historical scan of the table for the given `owner_id`/`hook_id`.

Key and timestamp bounds in `HooksController.java` lines 167–168 default to `MIN_KEY_BYTES` (all zeros) and `MAX_KEY_BYTES` (all 0xFF), so omitting `key` parameters also maximizes the scan range.

**Why existing checks fail:**

- `@Max(MAX_LIMIT)` on `limit` (line 112) caps returned rows at 100 but does not bound scan cost.
- `@Size(max = 2)` on `timestamps` (line 110) only prevents more than 2 timestamp parameters; it does not restrict range width.
- The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module — there is no equivalent rate limiter wired into the `rest-java` module's `HooksController`.
- No `@PreAuthorize` or authentication annotation is present on `HooksController` or `HookServiceImpl`.

### Impact Explanation
Concurrent requests each trigger a full sequential scan of `hook_storage_change` for the target account. With enough parallel requests, the database connection pool is saturated, query executor threads are exhausted, and the mirror node becomes unresponsive to all clients. This is a complete availability denial for the mirror node REST API.

### Likelihood Explanation
The endpoint is publicly reachable with no credentials. The exploit requires only a valid account ID (trivially enumerable from the public ledger) and a single query parameter (`timestamp=gte:0`). Any attacker with a modest number of HTTP clients can sustain the attack indefinitely. No special knowledge, tokens, or on-chain resources are needed.

### Recommendation
1. **Add rate limiting to `rest-java`**: Apply a per-IP or global request-rate limiter (analogous to `ThrottleConfiguration` in `web3`) to all `HooksController` endpoints.
2. **Restrict timestamp range width**: Enforce a maximum allowed span between `timestampLowerBound` and `timestampUpperBound` (e.g., 31 days), rejecting requests that exceed it with HTTP 400.
3. **Add a DB-level query timeout**: Configure a statement timeout on the database connection used by `rest-java` so runaway scans are killed automatically.
4. **Index optimization**: Ensure `hook_storage_change` has a composite index on `(owner_id, hook_id, consensus_timestamp, key)` to support efficient range scans; without it, the `DISTINCT ON` scan is a full table scan.

### Proof of Concept

```bash
# Step 1: Identify any valid account with hook storage (e.g., 0.0.1234, hook_id=1)
# Step 2: Send high-concurrency requests with maximum timestamp range and full key range

for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100" &
done
wait
```

Each request triggers `findByKeyBetweenAndTimestampBetween` with `timestampLowerBound=0`, `timestampUpperBound=Long.MAX_VALUE`, `keyLowerBound=0x000...000`, `keyUpperBound=0xfff...fff`, forcing a full `DISTINCT ON` scan of the entire `hook_storage_change` table for that account. 500 concurrent such requests will saturate the database connection pool and cause denial of service.     private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
