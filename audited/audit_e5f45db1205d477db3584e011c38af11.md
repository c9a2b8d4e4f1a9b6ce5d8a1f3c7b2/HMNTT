### Title
Unbounded Timestamp Range in `/accounts/:id` Bypasses 7-Day Validation, Enabling DB Connection Pool Exhaustion DoS

### Summary
The `getOneAccount` handler in `rest/accounts.js` calls `parseTimestampFilters` with `validateRange = false`, explicitly bypassing the 7-day maximum timestamp range check enforced on other endpoints. When `bindTimestampRange` is `false` (the documented default), no secondary range bounding occurs either. An unauthenticated attacker can supply an arbitrarily wide timestamp range alongside `transactions=true` (the default), causing the server to fire multiple expensive, long-running DB queries in parallel, exhausting the 10-connection pool (default) and rendering the service unavailable.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount` (line 413):
```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
```
The sixth argument `validateRange = false` explicitly disables the range-size check inside `parseTimestampFilters` (`rest/utils.js` lines 1657–1665), which would otherwise reject ranges exceeding `maxTimestampRange` (default 7 days). [1](#0-0) [2](#0-1) 

**Secondary safety net is off by default:**

`rest/timestampRange.js` line 20: `bindTimestampRange` returns the range unchanged when `queryConfig.bindTimestampRange` is falsy. The documented default for `hiero.mirror.rest.query.bindTimestampRange` is `false`. [3](#0-2) [4](#0-3) 

**Parallel query execution:**

`rest/accounts.js` lines 487–495: `entityPromise` (entity + token balance query) and `transactionsPromise` (which internally fans out to `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction` tables via full outer joins) are launched concurrently with `Promise.all`. Each request therefore holds multiple DB connections simultaneously. [5](#0-4) 

**Transaction query complexity:**

`rest/transactions.js` lines 595–604: When an account filter is present (always true here), the query is a three-way `FULL OUTER JOIN` across `transaction`, `crypto_transfer`, and `token_transfer` sub-queries, each scanning the full supplied timestamp range. [6](#0-5) 

**DB pool size:**

Default `maxConnections = 10` and `statementTimeout = 20000 ms`. [7](#0-6) 

### Impact Explanation

With 10 DB connections and a 20-second statement timeout, an attacker sending ~5 concurrent requests each holding 2+ connections for up to 20 seconds can saturate the entire pool. All other API consumers receive connection-timeout errors or queued delays. Because the REST API is a read-only mirror node, this directly degrades availability of the public Hedera network data plane. The impact is a complete, repeatable non-network DoS against the REST API tier with no data modification required.

### Likelihood Explanation

The endpoint is fully public and requires no authentication. The exploit requires only a standard HTTP client. The attacker needs to know one valid account ID (trivially obtained from any block explorer or by querying `/api/v1/accounts`). The `transactions=true` flag is the default, so it need not even be specified. The attack is trivially repeatable in a loop. The only partial mitigation (20-second statement timeout) actually makes the attack more effective by keeping connections occupied for the maximum window before release.

### Recommendation

1. **Remove `validateRange = false`** from the `parseTimestampFilters` call in `getOneAccount` (`rest/accounts.js` line 413), or add an explicit range-size guard before firing the transactions query.
2. **Enable `bindTimestampRange = true` by default** in `application.yml` / documentation, or enforce it unconditionally in `getOneAccount` regardless of the global config flag.
3. **Add per-IP or global request-rate limiting** on the `/accounts/:id` endpoint at the middleware layer (analogous to the Traefik `inFlightReq` / `rateLimit` middleware already used for other services).
4. Consider **sequential rather than parallel** execution of entity and transaction queries when no timestamp filter is provided, to reduce peak connection consumption per request.

### Proof of Concept

```bash
# Requires: curl, a valid account ID (e.g. 0.0.2 always exists on mainnet/testnet)
# Send 6 concurrent requests with a multi-year timestamp range
for i in $(seq 1 6); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.2?\
timestamp=gte:0000000001\
&timestamp=lte:9999999999999999999" &
done
wait

# All subsequent requests to any /api/v1/* endpoint will time out or return 503
# until the 20-second statement timeout expires and connections are released.
# Repeat the loop to maintain continuous DoS.
```

**Preconditions:** Default deployment (`bindTimestampRange=false`, `maxConnections=10`, `statementTimeout=20000`).
**Trigger:** 5–6 concurrent GET requests to `/api/v1/accounts/:id` with a timestamp range spanning years.
**Result:** DB connection pool exhausted; all API requests fail until connections are released by statement timeout.
