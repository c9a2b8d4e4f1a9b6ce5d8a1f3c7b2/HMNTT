### Title
Unbounded Timestamp Range in Historical Hook Storage Query Enables Unauthenticated Database DoS

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts timestamp parameters that, when only a single bound is supplied, produce a query spanning `[0, Long.MAX_VALUE]` against the `hook_storage_change` table. Because the SQL uses `DISTINCT ON (key)` — which forces a full pre-LIMIT scan — and no rate limiting exists in the `rest-java` module, any unauthenticated caller can repeatedly trigger full table scans, exhausting database I/O and degrading the mirror node REST API.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` accepts up to two `TimestampParameter` values (`@Size(max = 2)`). [1](#0-0) 

These are passed to `Bound.of()` with `primarySortField = false`: [2](#0-1) 

Inside `Bound.of()`, the `Bound` constructor is called with `primarySortField = false`. The only range-validity check (`adjustedLower > adjustedUpper`) is gated on `primarySortField == true`, so it is **never executed** for timestamps here: [3](#0-2) 

When only one timestamp bound is provided:
- `getAdjustedLowerRangeValue()` returns `0` if `lower == null`
- `adjustUpperBound()` returns `Long.MAX_VALUE` if `upper == null` [4](#0-3) 

`HookStorageRequest.isHistorical()` returns `true` whenever any timestamp is present, routing to `getHookStorageChange()`: [5](#0-4) 

`getHookStorageChange()` passes these bounds directly to the repository: [6](#0-5) 

The repository query uses `DISTINCT ON (key)` — PostgreSQL must scan **all** rows matching `owner_id`, `hook_id`, and the timestamp range before applying `LIMIT`. With `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, this is a full table scan for that owner/hook: [7](#0-6) 

**No rate limiting exists in `rest-java`**: the `ThrottleConfiguration` and `ThrottleManagerImpl` are in the `web3` module only. The `rest-java` config directory contains no rate-limiting filter.


### Impact Explanation

An attacker can force the database to perform unbounded sequential scans on `hook_storage_change` for any `(ownerId, hookId)` pair. With concurrent requests, this exhausts database I/O, increases query latency across all mirror node REST endpoints, and can cause the REST API to become unresponsive. The `@Max(MAX_LIMIT) = 100` constraint only caps returned rows, not the scan scope. The mirror node REST API is a shared read service; degrading it affects all consumers including wallets, explorers, and dApps. [8](#0-7) 

### Likelihood Explanation

No authentication is required. The endpoint is publicly reachable. The exploit requires only a standard HTTP GET with a single query parameter (`timestamp=gte:0`). It is trivially repeatable with any HTTP client or scripting tool. A single attacker with modest bandwidth can sustain the attack indefinitely.

### Recommendation

1. **Enforce a maximum timestamp range width**: In `Bound.of()` or `hookStorageChangeRequest()`, reject or cap requests where `adjustedUpper - adjustedLower` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Require both bounds**: Reject requests that supply only one timestamp bound for the historical path, or default the missing bound to a narrow window (e.g., ±1 hour from the supplied value).
3. **Add rate limiting to `rest-java`**: Apply a per-IP request rate limiter (e.g., Bucket4j) to all `HooksController` endpoints, mirroring the pattern in `web3/ThrottleConfiguration`.
4. **Add a database query timeout**: Configure a statement timeout on the PostgreSQL connection pool for `rest-java` queries to bound worst-case scan duration.

### Proof of Concept

```
# Single lower-bound: timestampLowerBound=0, timestampUpperBound=Long.MAX_VALUE
GET /api/v1/accounts/1/hooks/1/storage?timestamp=gte:0

# Single upper-bound: timestampLowerBound=0, timestampUpperBound=~year 2554
GET /api/v1/accounts/1/hooks/1/storage?timestamp=lte:9999999999

# Two-bound spanning full history:
GET /api/v1/accounts/1/hooks/1/storage?timestamp=gte:0&timestamp=lte:9999999999

# Flood with concurrent requests (bash):
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/accounts/1/hooks/1/storage?timestamp=gte:0" &
done
wait
```

Each request triggers `findByKeyBetweenAndTimestampBetween()` with `timestampLowerBound=0` / `timestampUpperBound=Long.MAX_VALUE`, executing a full `DISTINCT ON (key)` scan over `hook_storage_change` with no server-side abort or throttle.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public RangeParameter<Long> adjustLowerRange() {
        if (this.hasEqualBounds()) {
            // If the primary param has a range with a single value, rewrite it to EQ
            lower = new NumberRangeParameter(EQ, this.getAdjustedLowerRangeValue());
            upper = null;
        }

        return lower;
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
    }
