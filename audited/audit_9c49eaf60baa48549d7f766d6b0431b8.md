### Title
Unauthenticated Three-Part UNION ALL Query Exhausts DB Connection Pool via Crafted Token Allowance Range Filters

### Summary
An unauthenticated external user can send concurrent requests to `/:idOrAliasOrEvmAddress/allowances/tokens` with a crafted combination of `spender.id=gte:X&spender.id=lte:Y&token.id=gte:A&token.id=lte:B`, causing `extractTokenMultiUnionQuery()` to generate a three-part `UNION ALL` query. Because the REST API has no rate limiting and the default connection pool is only 10 connections with a 20-second statement timeout, an attacker can trivially exhaust the pool with concurrent requests, making all REST API endpoints unavailable.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:18` registers the route with no authentication: [1](#0-0) 

`rest/controllers/tokenAllowanceController.js:68-72` — `getAccountTokenAllowances` calls `extractTokenMultiUnionQuery` with user-supplied filters, no rate limiting or concurrency guard: [2](#0-1) 

`rest/controllers/tokenAllowanceController.js:22-59` — `extractTokenMultiUnionQuery` populates `lower`, `inner`, and `upper` filter arrays. When `spender.id` has both `gte` and `lte` bounds AND `token.id` has both `gte` and `lte` bounds, all three arrays are non-empty (confirmed by test case `spender gte and lte, token gte and lte`): [3](#0-2) 

`rest/service/tokenAllowanceService.js:63-74` — When all three filter arrays are non-empty, `getQuery()` emits a three-part `UNION ALL` SQL query, each sub-query with its own `ORDER BY ... LIMIT $2`, wrapped in an outer `ORDER BY ... LIMIT $2`: [4](#0-3) 

`rest/dbpool.js:7-16` — The pool is configured with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20,000 ms**): [5](#0-4) 

**Root cause and failed assumption:** The design assumes that the `statement_timeout` (20 s) and `LIMIT` clauses are sufficient to bound resource usage. However, there is no rate limiting or concurrency control on the REST API for this endpoint. The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module, not in the `rest` module. The `rest/server.js` applies no per-IP or per-endpoint rate limiting middleware. With a pool of only 10 connections, an attacker needs only 10 concurrent long-running requests to exhaust the pool entirely.

**Exploit flow:**

1. Attacker sends ≥10 concurrent HTTP GET requests:
   ```
   GET /api/v1/accounts/0.0.1/allowances/tokens?spender.id=gte:1&spender.id=lte:9999999999&token.id=gte:1&token.id=lte:9999999999
   ```
2. `buildAndValidateFilters` accepts these parameters — they are in `acceptedTokenAllowanceParameters` and pass `validateBounds()`.
3. `extractTokenMultiUnionQuery` produces non-empty `lower`, `inner`, and `upper` arrays.
4. `getQuery()` emits a 3-part `UNION ALL` query. The inner sub-query (`spender > $X and spender < $Y`) performs a wide range scan on `token_allowance`.
5. Each query holds a DB connection for up to 20 seconds (statement_timeout).
6. With 10 concurrent requests, all 10 pool connections are occupied.
7. Subsequent requests wait up to `connectionTimeoutMillis` (20,000 ms) before failing.
8. Attacker continuously sends new requests to keep the pool exhausted, causing a sustained outage of all REST API endpoints.

**Why existing checks are insufficient:**

- `validateBounds()` correctly validates filter combinations but explicitly permits the `gte+lte` on both dimensions — this is the intended pagination use case.
- `statement_timeout=20000ms` limits individual query duration but does not prevent pool exhaustion; 10 connections × 20 s = pool fully occupied for 20 s, and the attacker can sustain this indefinitely.
- `LIMIT $2` (default 25) limits result rows but does not prevent the inner range scan from consuming significant I/O and CPU before returning.
- No rate limiting, no concurrency cap, no IP-based throttling exists in the REST module.

### Impact Explanation

Complete connection pool exhaustion causes all REST API endpoints (not just this one) to return connection timeout errors, effectively taking down the entire mirror node REST API. This is a network partition from the perspective of API consumers. The default pool size of 10 connections makes this trivially achievable. Severity: **High** (unauthenticated, no special knowledge required, full REST API DoS).

### Likelihood Explanation

Any unauthenticated user on the internet can trigger this. The attack requires only standard HTTP GET requests with four query parameters. No credentials, tokens, or privileged access are needed. The attack is repeatable, automatable, and sustainable indefinitely. The attacker needs only a modest number of concurrent connections (≥10 by default).

### Recommendation

1. **Add rate limiting to the REST API** — apply a per-IP or global request rate limiter (e.g., `express-rate-limit`) to all routes, especially those that generate multi-part UNION queries.
2. **Reduce `statement_timeout`** — lower from 20,000 ms to a value appropriate for this query type (e.g., 5,000 ms).
3. **Increase `maxConnections`** — or use a connection pooler (PgBouncer) in front of the application pool to absorb bursts.
4. **Add a concurrency limit per endpoint** — reject requests when the pool utilization exceeds a threshold.
5. **Consider query cost bounding** — when both `spender.id` and `token.id` have open ranges, add a maximum range width validation to prevent unbounded inner scans.

### Proof of Concept

```bash
# Send 15 concurrent requests to exhaust the 10-connection pool
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1/allowances/tokens?\
spender.id=gte:1&spender.id=lte:9999999999&\
token.id=gte:1&token.id=lte:9999999999" &
done
wait

# Immediately after, verify pool exhaustion:
curl -v "http://<mirror-node>/api/v1/accounts/0.0.1/allowances/tokens"
# Expected: connection timeout or 503 error
```
