### Title
Unguarded `undefined` Dereference in `getSingleWithTtl()` Causes Unhandled TypeError on Redis Failure

### Summary
In `rest/cache.js`, `getSingleWithTtl()` uses a `.catch()` handler that returns `undefined` (the implicit return value of `logger.warn()`). The very next line unconditionally accesses `result[1][1]`, which throws a `TypeError` when `result` is `undefined`. Neither `getSingleWithTtl()` nor its caller `responseCacheCheckHandler` wraps this in a try/catch, so the rejected promise propagates as an unhandled rejection that crashes the request handler and, on Node.js ≥ 15, the entire process.

### Finding Description

**Exact code path:**

`rest/cache.js`, `getSingleWithTtl()`, lines 69–77:

```js
const result = await this.redis
  .multi()
  .ttl(key)
  .get(key)
  .exec()
  .catch((err) => logger.warn(`Redis error during ttl/get: ${err.message}`)); // returns undefined

// result is [[null, ttl], [null, value]], with value === null on cache miss.
const rawValue = result[1][1];  // TypeError: Cannot read properties of undefined (reading '1')
``` [1](#0-0) 

**Root cause:** `logger.warn()` returns `undefined`. The `.catch()` callback returns that `undefined`, so `result` is `undefined`. The comment on line 76 assumes `result` is always a 2-element array, but this assumption is violated on any Redis error.

**Why the `this.ready` guard is insufficient:**

`this.ready` is set to `false` only inside `retryStrategy`, which ioredis calls *after* it detects a connection drop and schedules a reconnect — not at the instant the connection drops. [2](#0-1) 

There is a race-condition window between the moment the TCP connection drops and the moment ioredis calls `retryStrategy`. During this window `this.ready` is still `true`, so the guard on line 65 passes, the `.exec()` call is issued, it fails (connection error or `commandTimeout` fires), `.catch()` returns `undefined`, and line 77 crashes. [3](#0-2) 

**Caller has no error handling:**

`responseCacheCheckHandler` awaits `getSingleWithTtl()` with no try/catch: [4](#0-3) 

The `TypeError` propagates as an unhandled promise rejection.

### Impact Explanation

On Node.js ≥ 15 (the default since 2020), an unhandled promise rejection terminates the process with exit code 1. Even on older runtimes the request handler crashes, leaving the HTTP connection hanging and the client receiving no response. Because `responseCacheCheckHandler` is a middleware that runs on every cacheable request, a single Redis failure during the race window is sufficient to take down the server.

### Likelihood Explanation

Any transient Redis unavailability — cloud network blip, Redis restart, `commandTimeout` expiry under load, Redis OOM kill — can open the race window. An attacker who can flood the application with requests may push Redis response times past `commandTimeout`, reliably triggering the fault. Even without an active attacker, the race window exists in every production deployment that uses Redis caching, making accidental triggering a realistic operational risk.

### Recommendation

Add a null/undefined guard on `result` before dereferencing it, mirroring the pattern already used in `get()` (line 106):

```js
const result = await this.redis
  .multi()
  .ttl(key)
  .get(key)
  .exec()
  .catch((err) => {
    logger.warn(`Redis error during ttl/get: ${err.message}`);
    return undefined;
  });

if (!result) {
  return undefined;   // graceful degradation, not a crash
}

const rawValue = result[1][1];
``` [5](#0-4) 

### Proof of Concept

1. Start the mirror-node REST service with Redis enabled.
2. While the service is running and `cache.ready === true`, kill or network-isolate the Redis instance (or set `commandTimeout` to a very small value and introduce artificial latency).
3. Send any GET request to a cacheable endpoint (e.g., `/api/v1/accounts/0.0.12345/rewards`) before ioredis calls `retryStrategy` and flips `ready` to `false`.
4. `getSingleWithTtl()` passes the `!this.ready` guard, issues `.exec()`, which rejects; `.catch()` returns `undefined`; `result[1][1]` throws `TypeError: Cannot read properties of undefined (reading '1')`.
5. On Node.js ≥ 15 the process exits immediately; on older runtimes the request hangs and an `UnhandledPromiseRejectionWarning` is emitted.
