### Title
Unauthenticated Concurrent Sub-Route Requests Exhaust DB Connection Pool via 5x Multiplier Effect

### Summary
The five account sub-routes registered in `rest/routes/accountRoute.js` are independently accessible to any unauthenticated user with no rate limiting, concurrency control, or mandatory caching in the REST Node.js layer. Each sub-route independently acquires one or more DB connections from the shared pool. An attacker sending all five sub-route requests simultaneously for the same account ID consumes up to 5–15 DB connections per logical account lookup, enabling rapid exhaustion of the fixed-size connection pool and denying service to all application users.

### Finding Description

**Exact code location:**

`rest/routes/accountRoute.js`, lines 12–19 — `getPath()` constructs five independent route patterns, each mapped to a separate controller handler:

```js
const getPath = (path) => `/:${filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS}/${path}`;
router.getExt(getPath('nfts'),              AccountController.getNftsByAccountId);
router.getExt(getPath('rewards'),           AccountController.listStakingRewardsByAccountId);
router.getExt(getPath('allowances/crypto'), CryptoAllowanceController.getAccountCryptoAllowances);
router.getExt(getPath('allowances/tokens'), TokenAllowanceController.getAccountTokenAllowances);
router.getExt(getPath('tokens'),            TokenController.getTokenRelationships);
```

**DB query multiplier per sub-route:**

Each controller independently calls `EntityService.getEncodedId()` (a DB query for alias/EVM address resolution), and two of the five routes additionally call `EntityService.isValidAccount()` (a second DB query) before issuing their primary data query:

- `listStakingRewardsByAccountId` (`accountController.js:171–172`): `getEncodedId` + `isValidAccount` + data query = **3 DB connections**
- `getTokenRelationships` (`tokenController.js:67–68`): `getEncodedId` + `isValidAccount` + data query = **3 DB connections**
- `getNftsByAccountId`, `getAccountCryptoAllowances`, `getAccountTokenAllowances`: `getEncodedId` + data query = **2 DB connections each**

Total per simultaneous 5-route attack: **up to 13 DB connections per account ID per request wave**.

**Root cause — failed assumption:**

The design assumes each sub-route is called independently and infrequently. There is no mechanism preventing a single client from calling all five simultaneously. The DB pool (`rest/dbpool.js:14`) has a hard cap of `config.db.pool.maxConnections` with no per-IP or per-request concurrency guard.

**Why existing checks fail:**

- `rest/middleware/authHandler.js:15–36`: The `authHandler` only applies custom limits to *authenticated* users who supply valid Basic Auth credentials. Unauthenticated requests pass through with no limit applied.
- `rest/middleware/responseCacheHandler.js:40–87`: The response cache requires *both* `config.cache.response.enabled` and `config.redis.enabled` to be true (`rest/server.js:54`). In deployments without Redis, there is zero caching — every request hits the DB.
- No rate-limiting middleware (e.g., `express-rate-limit`) is registered anywhere in `rest/server.js` for these routes.
- The throttling found in the codebase (`web3/src/main/java/.../ThrottleManagerImpl.java`) belongs to the separate `web3` Java service and has no effect on the REST Node.js service.

### Impact Explanation

With a default pool of, for example, 10 connections, an attacker sending 1 concurrent wave of 5-route requests per account ID consumes up to 13 connections — exceeding the pool immediately. Subsequent legitimate requests queue and eventually time out (`connectionTimeoutMillis`). Because the pool is global and shared across all API routes, exhaustion of the pool denies service to *all* API consumers, not just the targeted account. This is a complete application-layer denial of service achievable with a small number of HTTP requests.

### Likelihood Explanation

No authentication, no rate limiting, no IP blocking, and no mandatory caching are required to be bypassed. Any external user with network access to the REST API can execute this attack using a simple script or tool (e.g., `curl` with `&` backgrounding or any HTTP benchmarking tool). The attack is trivially repeatable and scriptable. The 5-route structure is publicly documented via the OpenAPI/Swagger docs served by the application itself (`rest/server.js:62`), making route discovery trivial.

### Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) to the Express application in `rest/server.js`, applied globally or specifically to `AccountRoutes.router`, limiting requests per IP per time window.
2. **Add in-process caching** for `EntityService.getEncodedId()` results (e.g., an LRU cache keyed on the raw account ID string) so repeated alias/EVM address resolution within a short window does not hit the DB.
3. **Enforce Redis/response cache** as a required dependency in production, or add a fallback in-memory cache so that repeated identical sub-route requests within the cache TTL do not generate DB queries.
4. **Set a per-request DB connection semaphore** or use a middleware that limits the number of concurrent in-flight DB-bound requests per IP.

### Proof of Concept

```bash
ACCOUNT="0.0.12345"
BASE="http://<mirror-node-host>/api/v1/accounts/${ACCOUNT}"

# Send all 5 sub-routes simultaneously, repeat N times to exhaust pool
for i in $(seq 1 20); do
  curl -s "${BASE}/nfts"              &
  curl -s "${BASE}/rewards"           &
  curl -s "${BASE}/allowances/crypto" &
  curl -s "${BASE}/allowances/tokens" &
  curl -s "${BASE}/tokens"            &
done
wait
```

After sufficient iterations, legitimate requests to any API endpoint will begin returning connection timeout errors as the pool is exhausted. No credentials, tokens, or special headers are required.
