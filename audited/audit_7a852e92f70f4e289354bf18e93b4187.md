### Title
Unauthenticated DoS via Unbounded Response Buffering in ShallowEtagHeaderFilter with No Rate Limiting on `/api/*`

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers Spring's `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no concurrency controls. The filter wraps every response in a `ContentCachingResponseWrapper`, buffering the full response body in heap memory for the entire request lifecycle (including DB query time). Because the `rest-java` module has zero HTTP-level rate limiting, an unauthenticated attacker can flood expensive multi-table endpoints, exhausting both the Tomcat thread pool and JVM heap simultaneously.

### Finding Description

**Exact code location:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, `etagFilter()`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");   // servlet prefix: matches all /api/v1/... paths
    return filterRegistrationBean;
}
```

**Root cause — two compounding failures:**

1. **Response buffering with no size cap.** `ShallowEtagHeaderFilter` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper` at the start of every matched request. The controller writes its entire JSON payload into this in-memory buffer; only after the controller returns does the filter MD5-hash the buffer to produce the ETag and flush it to the wire. The buffer is therefore live in heap for the full duration of the DB query plus serialization. No `setWriteWeakETag`, no `Content-Length` cap, no streaming bypass is configured.

2. **No rate limiting in `rest-java`.** A grep across the entire `rest-java/` source tree for `rateLimit`, `rateLimiter`, `throttle`, and `RateLimiter` returns only fee-estimation business logic and test references — there is no HTTP-level throttle filter. The `web3` module has a full `ThrottleManagerImpl` with bucket4j; `rest-java` has nothing equivalent.

**Exploit flow:**

- Attacker sends N concurrent GET requests to a multi-table endpoint, e.g.:
  `/api/v1/accounts/0.0.1/allowances/nfts?limit=100`
  (joins `nft_allowance` + `entity`; documented as expensive)
- Each request: Tomcat assigns a thread → `ShallowEtagHeaderFilter` allocates a `ContentCachingResponseWrapper` buffer → controller issues DB query (up to 10 s per `statementTimeout`) → response (potentially hundreds of KB of JSON) is written into the heap buffer → filter hashes and flushes.
- With N concurrent requests: N threads blocked for up to 10 s each, N full response payloads live in heap simultaneously.

**Why existing checks are insufficient:**

| Control | Scope | Gap |
|---|---|---|
| `statementTimeout = 10000 ms` | Per-query DB timeout | Limits duration per query, does not limit concurrency |
| Traefik `circuitBreaker` | Error ratio threshold | Triggers only on 5xx/network errors, not on request volume |
| GCP `maxRatePerEndpoint: 250` | GCP Gateway backend policy | Requires HPA (`hpa.enabled: false` by default); not active in default deployment |
| Tomcat thread pool | ~200 threads default | Is the resource being exhausted, not a protection |

### Impact Explanation
An attacker exhausts two finite resources simultaneously: Tomcat worker threads (blocking the service from handling any further requests) and JVM heap (each buffered response payload held until the filter flushes). With the default 2 GiB memory limit and 200-thread pool, a sustained flood of ~200 concurrent requests to expensive endpoints can render the `rest-java` service completely unresponsive. Because `rest-java` serves mirror node API traffic (airdrops, NFT allowances, network nodes, topics, fee estimation), a full service outage degrades the mirror node's ability to serve external consumers and downstream integrations.

### Likelihood Explanation
No authentication is required. The endpoints are publicly documented and reachable via the ingress. The attacker needs only an HTTP client capable of issuing concurrent requests — no exploit code, no credentials, no knowledge of internal state. The attack is trivially repeatable and scriptable (e.g., `ab`, `wrk`, or a simple async HTTP loop). The absence of any rate-limiting layer in `rest-java` (confirmed by source inspection) means there is no automatic back-pressure until the service is already degraded.

### Recommendation

1. **Add a rate-limiting filter to `rest-java`** analogous to `web3`'s `ThrottleManagerImpl` / `ThrottleConfiguration` (bucket4j). Register it before `ShallowEtagHeaderFilter` in the filter chain.
2. **Cap `ShallowEtagHeaderFilter` response buffering** or replace it with a streaming ETag approach (e.g., `WeakEtagHeaderFilter` or a custom filter that streams and hashes concurrently) to eliminate the full-body heap allocation.
3. **Enable HPA** so the GCP `maxRatePerEndpoint: 250` backend policy becomes effective, or add a Traefik `rateLimit` middleware entry alongside the existing `circuitBreaker`.
4. **Set a Tomcat `server.tomcat.threads.max`** and a Spring `spring.mvc.async.request-timeout` to bound per-request resource consumption.

### Proof of Concept

```bash
# Requires: wrk or ab; no credentials needed
# Target: any publicly exposed rest-java instance

# 200 concurrent connections, 30-second flood
wrk -t 10 -c 200 -d 30s \
  "https://<mirror-node-host>/api/v1/accounts/0.0.1/allowances/nfts?limit=100"

# Observe: HTTP 503 / connection timeouts begin within seconds
# JVM heap alert fires (>80% per RestJavaHighMemory Prometheus rule)
# Thread pool saturates; liveness probe at /actuator/health/liveness begins failing
# Pod restarts or becomes unresponsive
```
