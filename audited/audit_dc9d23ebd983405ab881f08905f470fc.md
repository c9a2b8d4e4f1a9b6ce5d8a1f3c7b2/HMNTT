### Title
Non-Atomic Rate-Limit Token Consumption Enables rateLimitBucket Drain via gasLimitBucket Exhaustion

### Summary
In `ThrottleManagerImpl.throttle()`, `rateLimitBucket.tryConsume(1)` is called first and, if it succeeds, `gasLimitBucket.tryConsume(...)` is called second. When the gas bucket check fails, the already-consumed rate-limit token is never returned. An unprivileged attacker can exploit this non-atomicity to drain `rateLimitBucket` during windows where `gasLimitBucket` is exhausted, causing legitimate low-gas requests to be rejected with `REQUEST_PER_SECOND_LIMIT_EXCEEDED` even when gas capacity is fully available.

### Finding Description

**Exact code path** — `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 38–42:

```java
if (!rateLimitBucket.tryConsume(1)) {                                          // line 38 — token consumed here
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);                // line 41 — rate token NOT returned
}
```

**Root cause**: The two `tryConsume` calls are not atomic and there is no compensating `addTokens` call on `rateLimitBucket` when the gas check fails. The `restore()` method (lines 59–64) only restores `gasLimitBucket`, never `rateLimitBucket`.

**Default bucket parameters** (from `ThrottleProperties` and `ThrottleConfiguration`):
- `rateLimitBucket`: capacity = 500 tokens, refill = 500/second
- `gasLimitBucket`: capacity = `scaleGas(7_500_000_000)` = 750,000 tokens, refill = 750,000/second
- `scaleGas(gas)` returns **0** for gas ≤ 10,000 (meaning those requests bypass the gas bucket entirely)

**Exploit flow**:

1. **Phase 1 — Exhaust gasLimitBucket**: Attacker (or heavy legitimate traffic) sends ~500 requests with `gas = 15,000,000` (scaled = 1,500 tokens each). This exhausts both buckets simultaneously (500 × 1,500 = 750,000 gas tokens; 500 × 1 = 500 rate tokens).

2. **Phase 2 — Exploit the refill asymmetry**: Both buckets refill over the next second. At any sub-second point (e.g., t = 0.1 s), `rateLimitBucket` has ~50 tokens and `gasLimitBucket` has ~75,000 tokens. Attacker sends 50 requests with `gas = 15,000,000`:
   - Each request: consumes 1 `rateLimitBucket` token (succeeds), then tries to consume 1,500 `gasLimitBucket` tokens (fails — only 75,000 available but the attacker can time this to hit 0).
   - Result: `rateLimitBucket` drained to 0; `gasLimitBucket` also drained.

3. **Phase 3 — Sustained DoS**: Attacker repeats Phase 2 every refill cycle. Legitimate users sending requests with `gas ≤ 10,000` (which call `gasLimitBucket.tryConsume(0)` — always succeeds) are now blocked at line 38 with `REQUEST_PER_SECOND_LIMIT_EXCEEDED` even though gas capacity is fully available.

**Why existing checks are insufficient**: There is no rollback, no atomic compare-and-swap across both buckets, and no mechanism to detect or compensate for the wasted rate-limit token. The `restore()` method is only called post-execution for gas, not for rate tokens.

### Impact Explanation

Legitimate users making low-gas or zero-gas calls (gas ≤ 10,000, which are exempt from gas throttling by design via `scaleGas`) are denied service with a misleading `REQUEST_PER_SECOND_LIMIT_EXCEEDED` error even when the gas budget is entirely free. This is a targeted denial-of-service against the rate-limit dimension of the throttle, decoupled from actual gas consumption. Severity: **Medium** — availability impact, no data exfiltration, but directly disrupts the intended throttle semantics.

### Likelihood Explanation

Any unauthenticated HTTP client can send `eth_call`/contract-call requests. No special privileges, tokens, or on-chain state are required. The attack is fully repeatable every second (the refill period), requires only ~500 HTTP requests per cycle, and can be automated trivially. In a production environment with moderate legitimate traffic already stressing the gas bucket, the attacker's required request volume drops further. Likelihood: **Medium-High**.

### Recommendation

Replace the sequential non-atomic checks with one of:

1. **Peek-then-consume**: Check `gasLimitBucket` availability with `getAvailableTokens()` before consuming from `rateLimitBucket`, then consume both only if both will succeed. (Note: still not strictly atomic under concurrency.)

2. **Rollback on gas failure** (simplest fix): Return the rate-limit token if the gas check fails:
   ```java
   if (!rateLimitBucket.tryConsume(1)) {
       throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
   }
   if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
       rateLimitBucket.addTokens(1);   // <-- compensate
       throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
   }
   ```

3. **Reverse check order**: Check `gasLimitBucket` first (no token consumed from `rateLimitBucket` on gas failure), then check `rateLimitBucket`. This eliminates the wasted rate token entirely.

### Proof of Concept

**Preconditions**: Default config (`requestsPerSecond=500`, `gasPerSecond=7_500_000_000`). No authentication required.

**Steps**:
```
# Step 1: Exhaust gasLimitBucket (and rateLimitBucket) with 500 max-gas requests
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x","gas":15000000,"to":"0x..."}' &
done
wait

# Step 2: Wait 100ms (partial refill: ~50 rate tokens, ~75000 gas tokens)
sleep 0.1

# Step 3: Send 50 requests to drain rateLimitBucket while gasLimitBucket is near-empty
for i in $(seq 1 50); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x","gas":15000000,"to":"0x..."}' &
done
wait

# Step 4: Legitimate user with gas=5000 (exempt from gas throttle) is now rejected
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"data":"0x","gas":5000,"to":"0x..."}'
# Expected: HTTP 429 "Requests per second rate limit exceeded"
# Despite: gasLimitBucket having capacity available
```

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }
