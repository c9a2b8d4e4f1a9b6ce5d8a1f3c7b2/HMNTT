### Title
Unbounded `evmAddressLookupIds` Accumulation Enables Griefing via Repeated Batched DB Queries in `SyntheticLogListener.onEnd()`

### Summary
`SyntheticLogListener.onEnd()` collects all EVM address entity IDs accumulated during a record file's processing into an unbounded `HashSet` and then resolves cache misses via batched calls to `entityRepository.findEvmAddressesByIds()`. Because there is no cap on how many unique entity IDs can accumulate per record file, an attacker who submits many token-transfer transactions involving previously-unseen accounts can force an arbitrarily large number of DB round-trips at the end of every record file, stalling the `onEnd()` pipeline and degrading mirror node processing latency.

### Finding Description

**Exact code path:**

1. `ParserContext.evmAddressLookupIds` is a plain `HashSet<Long>` with no size bound. [1](#0-0) 

2. For every synthetic `ContractLog`, `SyntheticLogUpdater.populateSearchIds()` adds up to three entity IDs (contractId, senderId, receiverId) to that set with no guard. [2](#0-1) 

3. `SyntheticLogListener.onEnd()` passes the entire set to `getEvmCache().getAll(keys)`. [3](#0-2) 

4. Caffeine forwards all cache-missing keys to `getCacheMisses()`, which batches them at `MAX_CACHE_LOAD_ENTRIES = 30 000` and issues one `findEvmAddressesByIds()` DB query per batch — with no cap on the total number of batches. [4](#0-3) 

   (Correct path:) [5](#0-4) 

5. The underlying query is a native `IN`-clause query against the `entity` table. [6](#0-5) 

**Root cause / failed assumption:** The design assumes the Caffeine cache (`maximumSize=500000`) will absorb repeated lookups, but the cache is only effective for entity IDs already seen in *previous* record files. Within a single record file, every newly-introduced entity ID is a guaranteed cache miss. The `ParserContext` is cleared between record files [7](#0-6) , but the Caffeine cache is not — so the attack window is the first record file in which a fresh batch of accounts appears.

**Exploit flow:**

- Attacker pre-creates (or reuses) a large pool of Hedera accounts with unique entity IDs.
- Attacker submits many `CryptoTransfer` / token-transfer transactions in a burst, each involving distinct sender/receiver accounts not yet in the mirror node's Caffeine cache.
- The mirror node importer generates a synthetic `ContractLog` for each transfer (via `SyntheticContractLogServiceImpl.create()`). [8](#0-7) 
- Each log contributes up to 3 unique IDs to `evmAddressLookupIds`.
- At `onEnd()`, `ceil(N / 30 000)` sequential DB queries are issued before the record file can be committed.

### Impact Explanation
The `onEnd()` call is synchronous and blocks record-file finalization. Forcing many sequential `findEvmAddressesByIds()` queries (each scanning up to 30,000 IDs against the `entity` table) directly delays the mirror node's ability to advance its consensus timestamp, causing observable processing latency for all downstream consumers (REST API, gRPC, etc.). The attack is repeatable across record files as long as the attacker keeps introducing fresh entity IDs that exceed the cache's 500,000-entry LRU window. [9](#0-8) 

### Likelihood Explanation
Any Hedera account holder can submit token-transfer transactions; no special privilege is required. Account creation fees impose an economic cost on the attacker, but the cost is modest relative to the sustained latency impact on the mirror node. The attack is repeatable: once the LRU cache evicts older entries (after 500,000 unique IDs), previously-cached IDs become cache misses again, allowing the attacker to recycle accounts. The `isSyntheticContractLogEvmAddressLookup` feature flag must be enabled (it is the default path), and `isSyntheticContractLogs` must also be enabled. [10](#0-9) 

### Recommendation
1. **Cap `evmAddressLookupIds` per record file.** Enforce a hard maximum (e.g., equal to `MAX_CACHE_LOAD_ENTRIES`) in `ParserContext.addEvmAddressLookupId()` and drop or log excess IDs. [11](#0-10) 
2. **Limit total DB batches in `getCacheMisses()`.** Add a guard so that at most one or two DB queries are issued per `onEnd()` invocation, regardless of cache-miss count. [12](#0-11) 
3. **Negative-cache absent EVM addresses.** Store a sentinel value for entity IDs confirmed to have no EVM address, preventing repeated DB lookups for the same IDs across record files.

### Proof of Concept
1. Enable `hiero.mirror.importer.parser.record.entity.persist.syntheticContractLogEvmAddressLookup=true` and `syntheticContractLogs=true` on a mirror node.
2. Pre-create 90,001 unique Hedera accounts (entity IDs not yet in the mirror node's Caffeine cache).
3. Submit token-transfer transactions in a single record-file window such that each transaction involves a distinct pair of the pre-created accounts as sender/receiver.
4. Observe that `SyntheticLogListener.onEnd()` issues at least 3 sequential `findEvmAddressesByIds()` queries (one per 30,000-ID batch) before the record file is finalized, measurably increasing `onEnd()` wall-clock time compared to a baseline with cached IDs.
5. Repeat with fresh accounts each record file to sustain the latency increase.

    private Map<Long, byte[]> getCacheMisses(Set<? extends Long> keys) {
        if (keys.isEmpty()) {
            return Collections.emptyMap();
        }

        Map<Long, byte[]> loadedFromDb = new HashMap<>(keys.size());
        if (keys.size() <= MAX_CACHE_LOAD_ENTRIES) {
            processBatch(loadedFromDb, entityRepository.findEvmAddressesByIds(keys));
        } else {
            final var keyIterator = keys.iterator();
            final var batches = Iterators.partition(keyIterator, MAX_CACHE_LOAD_ENTRIES);

            while (batches.hasNext()) {
                List<? extends Long> batch = batches.next();
                processBatch(loadedFromDb, entityRepository.findEvmAddressesByIds(batch));
            }
        }

        return loadedFromDb;
    }

    public void create(SyntheticContractLog log) {
        if (!entityProperties.getPersist().isSyntheticContractLogs()) {
            return;
        }

        var recordItem = log.getRecordItem();
        var contractRelatedParentRecordItem = recordItem.getContractRelatedParent();

        // We will either backfill any EVM-related fungible token transfers that don't have synthetic events produced by
        // CN
        // or create synthetic logs for HAPI-related transfer events
        if (shouldSkipLogCreation(log)) {
            return;
        }

        long consensusTimestamp;
        int logIndex;
        int transactionIndex;
        EntityId contractId;
        EntityId rootContractId;
        byte[] transactionHash;
        if (contractRelatedParentRecordItem != null) {
            consensusTimestamp = contractRelatedParentRecordItem.getConsensusTimestamp();
            logIndex = contractRelatedParentRecordItem.getAndIncrementLogIndex();
            transactionIndex = contractRelatedParentRecordItem.getTransactionIndex();
            transactionHash = contractRelatedParentRecordItem.getTransactionHash();

            final var parentTransactionRecord = contractRelatedParentRecordItem.getTransactionRecord();
            if (parentTransactionRecord.hasContractCallResult() || parentTransactionRecord.hasContractCreateResult()) {
                contractId = EntityId.of(
                        parentTransactionRecord.getContractCallResult().getContractID());
            } else {
                contractId = EntityId.EMPTY;
            }

            final var parentTransactionBody = contractRelatedParentRecordItem.getTransactionBody();
            if (parentTransactionBody.hasContractCall()) {
                final var contractIdReceipt =
                        parentTransactionRecord.getReceipt().getContractID();

                rootContractId = EntityId.of(contractIdReceipt);
            } else {
                rootContractId =
                        EntityId.of(parentTransactionRecord.getReceipt().getContractID());
            }
        } else {
            consensusTimestamp = recordItem.getConsensusTimestamp();
            logIndex = recordItem.getAndIncrementLogIndex();
            transactionIndex = recordItem.getTransactionIndex();
            transactionHash = recordItem.getTransactionHash();
            contractId = log.getEntityId();
            rootContractId = log.getEntityId();
        }

        ContractLog contractLog = new ContractLog();

        contractLog.setBloom(isContract(recordItem) ? CONTRACT_LOG_MARKER : empty);
        contractLog.setConsensusTimestamp(consensusTimestamp);
        contractLog.setContractId(contractId);
        contractLog.setData(log.getData() != null ? log.getData() : empty);
        contractLog.setIndex(logIndex);
        contractLog.setRootContractId(rootContractId);
        contractLog.setPayerAccountId(recordItem.getPayerAccountId());
        contractLog.setTopic0(log.getTopic0());
        contractLog.setTopic1(log.getTopic1());
        contractLog.setTopic2(log.getTopic2());
        contractLog.setTopic3(log.getTopic3());
        contractLog.setTransactionIndex(transactionIndex);
        contractLog.setTransactionHash(transactionHash);
        contractLog.setSynthetic(log instanceof TransferContractLog);

        entityListener.onContractLog(contractLog);
    }
