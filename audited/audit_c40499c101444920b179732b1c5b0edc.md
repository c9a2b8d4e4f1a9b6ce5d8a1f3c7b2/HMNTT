### Title
Cache Key Collision via `Arrays.hashCode()` on EVM Addresses Enables Cross-Address Entity Confusion in `CACHE_NAME_EVM_ADDRESS`

### Summary
`findByEvmAddressAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` — which delegates to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole cache key for 20-byte EVM addresses. Because 2^160 possible EVM addresses map into a 2^32 key space, an unprivileged attacker can trivially compute a second address B that collides with any existing cached address A. When the cache is hit for B, it returns Entity A's data, causing the EVM to execute with incorrect entity state for address B.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:
```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

**Root cause:** `Arrays.hashCode(byte[])` computes a 32-bit polynomial hash:
```
result = 1
for each byte b: result = 31 * result + (b & 0xFF)   // mod 2^32
```
This maps the 2^160 EVM address space into 2^32 possible keys. Collisions are not only possible — they are computationally trivial to find. Given any target address A with hash H, an attacker fixes bytes 0–17 arbitrarily and solves for bytes 18–19 (256 candidates each) to find B ≠ A with `Arrays.hashCode(B) == H`. This requires at most 256 arithmetic operations.

**Exploit flow:**
1. Attacker identifies a live entity with EVM address A (e.g., a deployed contract or account visible on-chain).
2. Attacker computes `H = Arrays.hashCode(A)`.
3. Attacker constructs address B ≠ A such that `Arrays.hashCode(B) == H` (trivial, O(256) ops).
4. Attacker sends a contract call referencing address A, causing `findByEvmAddressAndDeletedIsFalse(A)` to execute and store Entity A in the cache under key H.
5. Within the 1-second TTL window (`expireAfterWrite=1s`), attacker sends a contract call referencing address B.
6. Spring's `@Cacheable` intercept computes key `Arrays.hashCode(B) == H`, finds a cache hit, and returns Entity A's data for address B — without ever querying the database.

**Why existing checks fail:**
- `unless = "#result == null"` only prevents caching of null results; it does not prevent collision-based false hits.
- The 1-second TTL (`expireAfterWrite=1s,maximumSize=10000`) limits the window but does not eliminate it — the attacker controls the timing of both requests.
- There is no equality check between the cached key and the actual input bytes; the integer hash is the only discriminator.

### Impact Explanation
When address B resolves to Entity A's data, the EVM executes with incorrect state: wrong bytecode, wrong balance, wrong account type. A contract call to address B may execute Entity A's code, read Entity A's storage, or report Entity A's balance. This is a cross-address integrity failure. In a scenario where address B is expected to be an EOA or non-existent, but the cache returns a contract entity, the EVM may incorrectly execute contract logic, leading to wrong call results, incorrect gas accounting, or state corruption visible to callers. Because the cache is per-node (Caffeine, in-memory), the effect is local to each node that processes the crafted requests, but any node serving the attacker's requests is affected.

### Likelihood Explanation
The attacker requires no privileged access — only the ability to submit standard contract call or `eth_call` RPC requests, which is available to any external user. Computing a collision in `Arrays.hashCode` for a 20-byte array is O(256) arithmetic operations, requiring no cryptographic capability. The 1-second TTL requires the attacker to send both requests within the same second, which is trivially achievable programmatically. The attack is repeatable on demand as long as the target entity's address is known (publicly visible on-chain).

### Recommendation
Replace the `Arrays.hashCode(byte[])` cache key with the full byte array itself (or a collision-resistant encoding). Spring's Caffeine cache supports arbitrary `Object` keys with proper `equals`/`hashCode`. Use `java.util.Arrays.toString(alias)` or wrap the byte array in a `ByteBuffer` or a dedicated value type that implements `equals` and `hashCode` correctly:

```java
key = "T(java.util.Arrays).toString(#alias)"
```

Or define a `ByteArrayKey` wrapper with `Arrays.equals`/`Arrays.hashCode` for the internal hash but `Arrays.equals` for equality, ensuring that two keys with the same hash but different bytes are not treated as equal by the cache's underlying `HashMap`. The same fix applies to `findByEvmAddressOrAliasAndDeletedIsFalse` at lines 39–49, which has the identical defect.

### Proof of Concept

```python
# Find collision: two 20-byte addresses with same Arrays.hashCode

def arrays_hash_code(b):
    result = 1
    for byte in b:
        result = (31 * result + byte) & 0xFFFFFFFF
    return result

# Target address A (any known entity's EVM address)
A = bytes.fromhex("deadbeefdeadbeefdeadbeefdeadbeefdeadbeef")
H = arrays_hash_code(A)

# Fix first 18 bytes of B to differ from A
prefix = bytes([0x00] * 18)
h18 = 1
for b in prefix:
    h18 = (31 * h18 + b) & 0xFFFFFFFF

# Solve for last 2 bytes: need 31*(31*h18 + x) + y == H (mod 2^32)
for x in range(256):
    inner = (31 * h18 + x) & 0xFFFFFFFF
    # y = H - 31*inner (mod 2^32)
    y = (H - 31 * inner) & 0xFFFFFFFF
    if y < 256:
        B = prefix + bytes([x, y])
        assert arrays_hash_code(B) == H
        assert B != A
        print(f"Collision found: B = {B.hex()}, hash = {H}")
        break

# Steps:
# 1. Send eth_call to address A → populates cache[H] = EntityA
# 2. Within 1 second, send eth_call to address B → cache[H] hit → returns EntityA
# 3. EVM executes with EntityA's state for address B
