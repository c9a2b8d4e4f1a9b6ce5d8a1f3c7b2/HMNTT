### Title
Unbounded DB Queries via Limit Exhaustion in `PollingTopicListener.poll()` — Persistent DoS

### Summary
When `PollingTopicListener` is active (`type=POLL`), an unprivileged user can subscribe with `limit=1` and `startTime=0`. After the first message is consumed, the computed `pageSize` drops to `0`, causing `hasLimit()` to return `false` on the derived filter, which results in `setMaxResults()` never being called — issuing an unbounded SQL query on every subsequent polling interval tick. Because the flux repeats `Long.MAX_VALUE` times with no termination guard, this persists indefinitely and can be multiplied across many concurrent connections.

### Impact Explanation
Every 500ms per attacker connection, an unbounded `SELECT * FROM topic_message WHERE topic_id=X AND consensus_timestamp >= T ORDER BY consensus_timestamp` is issued with no row limit. On a topic with millions of messages, this forces a full index scan returning potentially millions of rows, loading them all into JVM heap, and saturating DB I/O and connection pool. Multiple concurrent attacker connections multiply the effect linearly. All legitimate subscribers sharing the same DB are degraded. This is a persistent, self-sustaining DoS requiring no special privileges.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible to any client. Setting `limit=1` and `startTime=0` is a standard, valid API call. No authentication or rate limiting is present in the listener path. The attack is trivially repeatable: open N connections with these parameters and hold them open. The attacker pays only the cost of maintaining a gRPC stream; the server pays unbounded DB query cost on every tick.

### Recommendation
1. **Terminate the flux when the limit is reached**: add `.take(filter.getLimit())` (when `filter.hasLimit()`) to the `listen()` flux chain so polling stops once the user's requested limit is satisfied.
2. **Guard against `pageSize <= 0`**: in `poll()`, return `Flux.empty()` immediately if `limit <= 0` rather than issuing a query with `pageSize=0`.
3. **Enforce a minimum `pageSize`**: clamp `pageSize = Math.max(1, Math.min(limit, listenerProperties.getMaxPageSize()))` and short-circuit when `limit <= 0`.
4. **Add per-connection rate limiting and a maximum concurrent subscription count** to prevent amplification.

### Proof of Concept

**Preconditions:**
- Mirror node deployed with `hiero.mirror.grpc.listener.type=POLL`
- A topic exists with at least one historical message

**Steps:**
1. Open a gRPC `subscribeTopic` stream with `topicId=<active topic>`, `limit=1`, `startTime=0` (epoch).
2. The first poll (tick 1) issues `SELECT ... LIMIT 1` from `consensusTimestamp >= 0` and returns the oldest message. `count` becomes 1.
3. Every subsequent tick (every 500ms): `limit = 1-1 = 0`, `pageSize = 0`, `newFilter.hasLimit() = false` → `setMaxResults()` not called → SQL issued with **no LIMIT**, scanning all rows after the oldest message.
4. Keep the stream open (do not cancel). The unbounded query fires every 500ms indefinitely.
5. Open 50–100 such connections simultaneously. DB CPU and I/O saturate; legitimate queries time out.
