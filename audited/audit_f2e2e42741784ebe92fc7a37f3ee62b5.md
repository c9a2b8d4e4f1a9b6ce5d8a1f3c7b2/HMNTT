### Title
Unauthenticated Information Disclosure via `/api/v1/subscriber/{name}` Exposes Network Monitoring Operational State

### Summary
The `SubscriberController` exposes live monitoring telemetry — including protocol type, status, error counts by exception class, elapsed uptime, and message rate — over HTTP with zero authentication. Because the monitor module has no Spring Security dependency and no access-control configuration, any network-reachable client can enumerate all active GrpcSubscription and RestSubscription scenarios and extract their real-time state, enabling an attacker to detect monitoring blind spots and time attacks against HCS topic-based smart contract workflows accordingly.

### Finding Description
**Code location:**
- `monitor/src/main/java/org/hiero/mirror/monitor/subscribe/controller/SubscriberController.java` lines 45–52 (`GET /api/v1/subscriber/{name}`)
- `monitor/src/main/java/org/hiero/mirror/monitor/subscribe/MirrorSubscriber.java` line 14 (`getSubscriptions()`)
- `monitor/src/main/java/org/hiero/mirror/monitor/AbstractScenario.java` lines 44–80 (data returned)

**Root cause:** `build.gradle.kts` includes no `spring-boot-starter-security` dependency. No `SecurityFilterChain`, no `@Secured`/`@PreAuthorize`, and no `WebSecurityConfigurerAdapter` exist anywhere under `monitor/src/main/java/`. The `SubscriberController` is a plain `@RestController` with no access-control annotations.

**Exploit flow:**
1. Attacker sends `GET /api/v1/subscriber` (no name needed) to the monitor's HTTP port.
2. `SubscriberController.subscriptions()` calls `mirrorSubscriber.getSubscriptions()` — implemented by both `GrpcSubscriber` (line 41–43) and `RestSubscriber` (line 70–72) — and streams all live `Scenario` objects.
3. Each `Scenario` serialises `count`, `elapsed`, `errors` (a `Map<String,Integer>` of exception class names to occurrence counts), `id`, `name`, `protocol`, `rate`, and `status` (lines 19–42 of `Scenario.java`).
4. Attacker then queries `GET /api/v1/subscriber/{name}` with the exact scenario name (e.g. `HCS Subscribe`) to filter to a specific GrpcSubscription monitoring an HCS topic used by smart contracts.
5. `getProperties()` is `@JsonIgnore`d, so topic IDs are not directly leaked, but all runtime telemetry is.

**Why checks fail:** There are none. The controller has no authentication gate, the module has no security dependency, and the documented API (`docs/monitor/README.md` lines 252–307) treats these endpoints as openly accessible.

### Impact Explanation
An attacker learns:
- Whether HCS topic monitoring is `RUNNING`, `IDLE`, or `COMPLETED` — directly indicating whether smart contract event monitoring is active.
- Error class names and counts (e.g. `TimeoutException`, `StatusRuntimeException`) revealing degraded connectivity to the mirror node.
- Elapsed uptime and message rate, enabling inference of monitoring restart cycles.

This operational intelligence allows an attacker to time malicious smart contract transactions or topic message injections during windows when monitoring is `IDLE` or `COMPLETED`, reducing the likelihood of anomaly detection. No funds are directly at risk, consistent with the stated severity (Layer 0/1/2, no concrete funds at direct risk).

### Likelihood Explanation
The monitor is typically deployed inside a Kubernetes cluster (a `KubernetesClient` bean is conditionally created), but the HTTP port is not firewalled by the application itself. Any actor with network access to the monitor pod/service — including other tenants in a shared cluster, or external users if the service is inadvertently exposed via a LoadBalancer or Ingress — can exploit this with a single unauthenticated HTTP GET. No credentials, tokens, or special tooling are required. The attack is trivially repeatable and scriptable.

### Recommendation
1. Add `spring-boot-starter-security` to `monitor/build.gradle.kts`.
2. Create a `SecurityConfiguration` bean that requires authentication (at minimum HTTP Basic or an API key) for all `/api/v1/**` routes, while optionally permitting `/actuator/health` for liveness probes.
3. Alternatively, if the monitor is strictly internal, enforce network-level controls (Kubernetes `NetworkPolicy`) to restrict access to the monitor's HTTP port to known internal consumers only, and document this as a deployment requirement.
4. Add an integration test asserting that unauthenticated requests to `/api/v1/subscriber` return `401`.

@RequestMapping("/api/v1/subscriber")
@RequiredArgsConstructor
@RestController
class SubscriberController {

    private final MirrorSubscriber mirrorSubscriber;

    @GetMapping
    public <T extends ScenarioProperties> Flux<Scenario<T, Object>> subscriptions(
            @RequestParam("protocol") Optional<ScenarioProtocol> protocol,
            @RequestParam("status") Optional<List<ScenarioStatus>> status) {
        return mirrorSubscriber
                .<Scenario<T, Object>>getSubscriptions()
                .filter(s -> !protocol.isPresent() || protocol.get() == s.getProtocol())
                .filter(s -> !status.isPresent() || status.get().contains(s.getStatus()))
                .switchIfEmpty(Mono.error(new NoSuchElementException()));
    }

    @GetMapping("/{name}")
    public <T extends ScenarioProperties> Flux<Scenario<T, Object>> subscriptions(
            @PathVariable("name") String name, @RequestParam("status") Optional<List<ScenarioStatus>> status) {
        Flux<Scenario<T, Object>> subscriptions = subscriptions(Optional.empty(), status);
        return subscriptions
                .filter(subscription -> subscription.getName().equals(name))
                .switchIfEmpty(Mono.error(new NoSuchElementException()));
    }
