### Title
Unauthenticated, Rate-Limit-Free Swagger UI Endpoint Enables DoS via Request Flooding

### Summary
The `serveSwaggerDocs()` function registers the Swagger UI endpoint at `/api/v1/{swaggerUIPath}` (default `/api/v1/docs`) before any protective middleware is applied, and the entire Node.js REST service contains zero rate-limiting logic. Any unprivileged external user can flood this endpoint with concurrent requests, saturating the Node.js event loop and exhausting process memory through per-request HTML generation with the embedded OpenAPI spec, rendering the REST API unavailable.

### Finding Description
**Exact code path:**

`rest/middleware/openapiHandler.js`, `serveSwaggerDocs()`, line 145:
```js
app.use(`/api/v1/${config.openapi.swaggerUIPath}`, swaggerUi.serve, swaggerUi.setup(getV1OpenApiObject(), options));
```

`rest/server.js`, line 62 (swagger registered) vs. line 86 (auth applied):
```js
serveSwaggerDocs(app);          // line 62 — swagger route registered here
// ...
app.useExt(authHandler);        // line 86 — auth middleware registered AFTER
```

**Root cause — two compounding failures:**

1. **Middleware ordering**: `serveSwaggerDocs(app)` is called at `server.js:62`, before `authHandler` (line 86), `metricsHandler` (line 91), and any other protective middleware. Express matches routes in registration order; the swagger route is matched and fully handled before any subsequent `app.use()` middleware can run. The auth handler never sees swagger requests.

2. **No rate limiting anywhere in the REST service**: A grep across all `rest/**/*.js` files for `rateLimit`, `rate-limit`, `express-rate-limit`, `helmet`, or `slowDown` returns zero matches. There is no per-IP, per-endpoint, or global request throttle in the Node.js REST service. (The `ThrottleConfiguration` found in the codebase is in `web3/` — a separate Java/Spring Boot service — and has no effect here.)

**Per-request cost of the swagger endpoint:**

`swaggerUi.setup()` returns a middleware that, on every request to the swagger UI HTML page, generates a fresh HTML document by embedding the full OpenAPI spec as an inline JSON string. While `getV1OpenApiObject()` is memoized (the YAML is parsed once), the HTML serialization and response buffering occur on every request. For a large OpenAPI spec (hundreds of KB), each concurrent in-flight request holds a full copy of the generated HTML in the Node.js heap. `swaggerUi.serve` additionally serves multiple static assets (JS bundles, CSS) per page load, multiplying the per-visit request count.

**Why existing checks fail:**

- `authHandler` at line 86 never executes for the swagger route because Express has already matched and dispatched the request at line 62.
- There is no `express-rate-limit` or equivalent middleware anywhere in `rest/`.
- There is no connection-level throttle (e.g., nginx upstream rate limiting) enforced at the application layer.

### Impact Explanation
Under sustained flooding, the Node.js event loop becomes saturated processing concurrent swagger requests. Heap memory grows as each in-flight request buffers a copy of the generated HTML. Legitimate API requests (accounts, transactions, tokens, etc.) queue behind the flooded event loop and time out. At sufficient concurrency the process OOMs or becomes unresponsive, taking the entire mirror node REST API offline. This is a single-process Node.js server (`server.js` exports one `app`); there is no request isolation between the swagger endpoint and the data API endpoints.

### Likelihood Explanation
Preconditions: none. The endpoint is publicly reachable with no authentication, no CAPTCHA, no token, and no IP restriction. A single attacker with a modest botnet or even a single machine using async HTTP clients (e.g., `wrk`, `hey`, `ab`) can sustain tens of thousands of requests per second against a single Node.js process. The attack is trivially repeatable and requires no prior knowledge of the system beyond the well-known default path `/api/v1/docs`.

### Recommendation
1. **Fix middleware ordering**: Move `serveSwaggerDocs(app)` to after all protective middleware is registered, or apply rate-limiting middleware directly to the swagger route before registering it.
2. **Add rate limiting to the swagger route specifically**:
   ```js
   import rateLimit from 'express-rate-limit';
   const swaggerLimiter = rateLimit({ windowMs: 60_000, max: 30 });
   app.use(`/api/v1/${config.openapi.swaggerUIPath}`, swaggerLimiter, swaggerUi.serve, swaggerUi.setup(...));
   ```
3. **Restrict the swagger endpoint in production**: Gate it behind `authHandler` or disable it entirely via config when `NODE_ENV=production`.
4. **Add a global rate limiter** to the REST service, as no such protection currently exists for any endpoint.

### Proof of Concept
```bash
# Flood the swagger UI endpoint with 500 concurrent connections, unlimited requests
wrk -t 10 -c 500 -d 60s http://<mirror-node-host>:5551/api/v1/docs

# Observe: legitimate API requests begin timing out
curl http://<mirror-node-host>:5551/api/v1/transactions
# → connection timeout or 503 after event loop saturation

# Monitor heap growth during attack
# node --max-old-space-size=<default> will OOM under sustained load
```

No credentials, tokens, or special headers are required. The attack is executable by any unprivileged external user.
