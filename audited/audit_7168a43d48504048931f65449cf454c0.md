### Title
Unauthenticated Request Flood Triggers DbError → 503 Storm, Tripping Traefik Circuit Breaker and Blocking All Transaction Gossip Queries

### Summary
The REST API has no rate limiting. Any unauthenticated user can flood endpoints, exhausting the PostgreSQL connection pool. Every failed DB query in `queryQuietly()` is unconditionally wrapped in a `DbError`, which `handleError()` maps to HTTP 503. The Traefik circuit breaker configured at `ResponseCodeRatio(500, 600, 0, 600) > 0.25` trips when 503s exceed 25% of total responses, blocking all traffic — including transaction gossip queries — to the mirror node.

### Finding Description

**Code path 1 — DbError generation:**

`rest/utils.js` lines 1535–1539: every DB query failure in `queryQuietly()` is unconditionally re-thrown as `DbError`:

```js
} catch (err) {
  if (client !== undefined) {
    await client.query('rollback');
  }
  throw new DbError(err.message);   // ← any pg error becomes DbError
}
``` [1](#0-0) 

**Code path 2 — 503 mapping:**

`rest/middleware/httpErrorHandler.js` lines 21–22: `DbError` is unconditionally mapped to `SERVICE_UNAVAILABLE` (503):

```js
} else if (err instanceof DbError) {
  statusCode = httpStatusCodes.SERVICE_UNAVAILABLE;
``` [2](#0-1) 

**Code path 3 — Traefik circuit breaker:**

`charts/hedera-mirror-rest/values.yaml` lines 135–136: the Traefik middleware trips when >25% of all responses are 5xx:

```yaml
- circuitBreaker:
    expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
``` [3](#0-2) 

**Code path 4 — Retry amplification:**

`charts/hedera-mirror-rest/values.yaml` lines 137–139: Traefik retries each failed request up to 10 times, multiplying backend load by 10×:

```yaml
- retry:
    attempts: 10
    initialInterval: 100ms
``` [4](#0-3) 

**No rate limiting on the REST API:** The `rest/server.js` middleware stack registers `authHandler`, `requestLogger`, `metricsHandler`, and `handleError`, but no rate-limiting middleware. The web3 service has Bucket4j throttling; the REST API has none. [5](#0-4) 

**Root cause:** The failed assumption is that DB errors are rare, infrastructure-level events. In reality, an attacker can deliberately exhaust the shared PostgreSQL connection pool by sending many concurrent unauthenticated requests, converting a resource-exhaustion condition into a flood of 503 responses that the Traefik circuit breaker interprets as a service failure.

### Impact Explanation

When the circuit breaker trips (`ResponseCodeRatio > 0.25`), Traefik stops routing **all** traffic to the REST API backend — including `/api/v1/transactions` and related gossip-query endpoints. The mirror node becomes effectively unreachable for the duration of the circuit-open window. Because the retry middleware amplifies each attacker request 10×, sustaining the attack requires far fewer source requests than the threshold suggests. The Prometheus alert rule confirms the severity: a 1% 5xx rate triggers a `critical` alert; the circuit breaker threshold is 25×. [6](#0-5) 

### Likelihood Explanation

No credentials, API keys, or special network position are required. The attack is a standard HTTP flood against any public REST endpoint (e.g., `GET /api/v1/transactions`). The 10× retry amplification means a modest flood of ~N concurrent connections can generate ~10N backend requests, making pool exhaustion achievable with limited bandwidth. The attack is repeatable: once the circuit breaker resets, the attacker re-floods to re-trip it.

### Recommendation

1. **Add per-IP rate limiting** to the REST API middleware stack (e.g., `express-rate-limit` or a Traefik `rateLimit` middleware matching the Rosetta configuration) before DB queries are issued.
2. **Distinguish transient DB errors from client-induced errors**: do not map all `DbError` instances to 503; use 429 (Too Many Requests) or 503 with a `Retry-After` header only for genuine infrastructure failures, and return 503 only when the health check itself fails.
3. **Separate circuit-breaker scope**: configure the Traefik circuit breaker to exclude health-check paths and to distinguish client-error-induced 5xx from genuine server failures, or raise the threshold and add a minimum-request floor.
4. **Cap the connection pool per client IP** at the DB proxy/pgBouncer layer to prevent a single source from exhausting the shared pool.

### Proof of Concept

```bash
# 1. Identify any public REST endpoint (no auth required)
TARGET="https://<mirror-node>/api/v1/transactions"

# 2. Send a sustained flood of concurrent requests to exhaust the DB pool
#    (adjust -c to exceed pool size, typically 10-50 connections)
ab -n 100000 -c 100 "$TARGET"
# or
hey -n 100000 -c 100 "$TARGET"

# 3. Observe responses shift from 200 → 503 as the pool exhausts
# 4. Traefik circuit breaker trips when 503s exceed 25% of responses
#    (retry amplification means each attacker request generates up to 10 backend hits)
# 5. All subsequent requests — including gossip queries — receive 503 from Traefik
#    until the circuit-open window expires, then repeat step 2
```
