### Title
Unauthenticated DoS via Uncached NotFoundError on Non-Existent Account IDs in `listStakingRewardsByAccountId`

### Summary
The `/accounts/:id/rewards` endpoint performs a live database lookup via `EntityService.isValidAccount()` for every request. When a syntactically valid but non-existent account ID is supplied, a `NotFoundError` is thrown and the 404 response is explicitly excluded from the response cache. With no rate limiting in the middleware stack, an unprivileged attacker can flood the endpoint with distinct non-existent account IDs, forcing a DB query per request and exhausting the database connection pool.

### Finding Description
**Code path:**

`rest/controllers/accountController.js`, `listStakingRewardsByAccountId`, lines 170–175:
```js
const accountId = await EntityService.getEncodedId(req.params[...]);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) {
  throw new NotFoundError();
}
```
Every call to `isValidAccount()` issues a synchronous DB query against the entity table. If the account does not exist, the query returns false and `NotFoundError` is thrown.

**Why the cache does not help:**

`rest/middleware/responseCacheHandler.js`, line 95:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```
Only HTTP 2xx responses are written to Redis. A 404 is never cached, so every request with a distinct non-existent account ID (e.g., `0.0.1`, `0.0.2`, …) bypasses the cache entirely and hits the DB.

**Why the error handler does not mitigate:**

`rest/middleware/httpErrorHandler.js`, lines 17–18:
```js
if (err instanceof NotFoundError) {
  statusCode = httpStatusCodes.NOT_FOUND;
```
The handler is lightweight (log + JSON response) and imposes no back-pressure or circuit-breaking.

**Why auth does not mitigate:**

`rest/middleware/authHandler.js`, lines 16–19:
```js
const credentials = basicAuth(req);
if (!credentials) {
  return;   // unauthenticated requests pass through
}
```
Authentication is entirely optional; unauthenticated requests are allowed to proceed.

**No rate limiting** is present anywhere in the observed middleware chain (`rest/middleware/index.js`, `authHandler`, `requestHandler`, `httpErrorHandler`).

### Impact Explanation
An attacker cycling through the ~4-billion valid `0.0.<num>` account ID space (or any large range of non-existent IDs) forces one DB query per request with zero cache benefit. Sustained high-volume traffic exhausts the database connection pool, causing query timeouts for all endpoints sharing that pool. This matches the stated scope of "total network shutdown" — legitimate transactions cannot be confirmed because the mirror node REST API becomes unavailable.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only knowledge of the public API schema (shard.realm.num format is documented). The attacker needs only a script generating sequential or random numeric IDs and an HTTP client capable of high concurrency. The endpoint is publicly reachable. This is trivially repeatable and automatable.

### Recommendation
1. **Cache negative results**: Store 404 responses in Redis with a short TTL (e.g., 5–30 seconds) so repeated lookups for the same non-existent ID do not hit the DB.
2. **Add rate limiting**: Introduce a per-IP (and optionally per-account-ID) rate limiter in the middleware chain before the controller is reached.
3. **Add a DB-level circuit breaker**: If the entity lookup pool is saturated, fail fast rather than queuing additional queries.
4. **Consider existence pre-validation**: Validate the numeric range of the account ID before issuing a DB query (e.g., reject IDs above a known maximum shard/realm/num).

### Proof of Concept
```bash
# Flood with distinct syntactically valid but non-existent account IDs
for i in $(seq 900000000 900010000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.$i/rewards" &
done
wait
```
Each request causes one DB query; none are served from cache. At sufficient concurrency the DB connection pool is exhausted and all subsequent queries (including legitimate ones) time out, returning HTTP 503 across all endpoints. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
