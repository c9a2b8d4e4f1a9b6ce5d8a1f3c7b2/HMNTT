### Title
Unindexed LIKE Prefix Scan on `record_file.hash` Enables Unauthenticated DB CPU Exhaustion

### Summary
The `getByHashOrNumber()` function in `rest/service/recordFileService.js` executes `hash LIKE $1` with a 64-character prefix against a 96-character stored hash column. The existing B-tree index `record_file__hash` is defined with `collate "C"` but the query does not specify that collation, so in any non-C locale PostgreSQL deployment the index is bypassed and a full sequential scan is performed. Any unauthenticated user can flood this public endpoint with random 64-char hex hashes, each triggering a full table scan, exhausting database CPU.

### Finding Description

**Exact code path:**

`rest/controllers/blockController.js` `getByHashOrNumber()` (line 117) calls `RecordFileService.getByHashOrNumber(hash, number)`. [1](#0-0) 

`rest/service/recordFileService.js` `getByHashOrNumber()` (lines 164–179) builds the query:
```js
whereStatement += `${RecordFile.HASH} like $1`;
params.push(hash + '%');
```
producing: `SELECT ... FROM record_file WHERE hash LIKE $1` with `$1 = '<64-char-input>%'`. [2](#0-1) 

The static query template is:
```js
static recordFileBlockDetailsFromHashQuery = `select ... from ${RecordFile.tableName} where ${RecordFile.HASH} like $1 limit 1`;
``` [3](#0-2) 

**Root cause — index collation mismatch:**

The only index on `record_file.hash` is:
```sql
create index if not exists record_file__hash on record_file (hash collate "C");
``` [4](#0-3) 

In PostgreSQL, a B-tree index with an explicit `collate "C"` can only be used for LIKE prefix queries when the query itself also specifies `collate "C"` (or the database locale is C/POSIX). The REST query emits no collation qualifier. The Rosetta persistence layer confirms the production database is not in C locale — it must explicitly add `collate "C"` to its equality query to use the same index:
```go
where hash = @hash collate "C"
``` [5](#0-4) 

Because the REST LIKE query carries no collation, PostgreSQL's planner cannot match it to the `collate "C"` index and falls back to a sequential scan of the entire `record_file` table. `LIMIT 1` only terminates the scan early on a hit; a non-existent (attacker-supplied) hash forces a full scan every time.

**Why input validation fails to prevent this:**

`validateHashOrNumber` accepts any syntactically valid 64-char hex string (Ethereum format) and strips the `0x` prefix — it performs no existence pre-check and imposes no rate limit. [6](#0-5) 

The REST API has no per-endpoint rate limiting. The throttling infrastructure (`ThrottleManager`, `ThrottleConfiguration`) exists only in the `web3` Java module, not in the Node.js REST service. [7](#0-6) 

### Impact Explanation
The `record_file` table grows at roughly one row per ~2 seconds of Hedera operation; on mainnet this is millions of rows. Each unauthenticated GET to `/api/v1/blocks/<random-64-char-hex>` triggers a full sequential scan. Concurrent flooding saturates database CPU, degrading or denying service to all mirror node consumers (block queries, transaction queries, EVM calls) that share the same PostgreSQL instance. No authentication, API key, or privileged access is required.

### Likelihood Explanation
The endpoint is public and documented in the OpenAPI spec. The attack requires only an HTTP client and knowledge of the 64-char hex format (standard Ethereum block hash format, widely known). The attacker needs no credentials, no on-chain assets, and no special tooling — a simple loop of `curl` or `ab` requests suffices. The attack is repeatable and stateless.

### Recommendation
1. **Fix the query collation** — append `collate "C"` to the LIKE predicate so the planner can use the existing index:
   ```sql
   WHERE hash LIKE $1 COLLATE "C"
   ```
   or equivalently in the JS string: `` `${RecordFile.HASH} like $1 collate "C"` ``

2. **Alternatively, add a `text_pattern_ops` index** which explicitly supports LIKE prefix matching regardless of database locale:
   ```sql
   CREATE INDEX record_file__hash_pattern ON record_file (hash text_pattern_ops);
   ```

3. **Add rate limiting** to the REST block-by-hash endpoint (e.g., via an Express middleware or an upstream API gateway), mirroring the throttle controls already present in the web3 module.

### Proof of Concept
```bash
# Generate a random 64-char hex hash (non-existent block hash)
HASH=$(openssl rand -hex 32)

# Flood the endpoint — each request triggers a full table scan
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/blocks/${HASH}" &
done
wait

# Observe: DB CPU spikes to 100%; legitimate queries time out or return errors.
