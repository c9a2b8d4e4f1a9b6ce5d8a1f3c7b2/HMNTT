### Title
Unauthenticated Mirror Node Response Allows Full Network Map Hijacking via Rogue Node Injection

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` fetches node addresses over plain HTTP with no TLS enforcement, no response signature verification, and no IP/domain allowlist validation. Any attacker capable of intercepting or spoofing the HTTP response — a network-adjacent, unprivileged attacker via MITM, DNS poisoning, or ARP spoofing — can inject arbitrary `serviceEndpoint` entries. The resulting network map is passed verbatim to `hiero.ClientForNetwork()`, redirecting all subsequent signed HBAR transfer transactions to attacker-controlled nodes that can apply non-standard fees or manipulate transactions.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–132.

The function issues a plain `http.Get` (line 84) to `cfg.mirrorRest` — configured via `HIERO_MIRROR_PINGER_REST` or `-mirror-rest`, defaulting to `http://mirror-rest:5551` (plain HTTP, no TLS). The JSON body is decoded into `nodesEnvelope` (line 96). Then, lines 113–124 iterate every `ServiceEndpoint` and call:

```go
host := strings.TrimSpace(ep.DomainName)
if host == "" {
    host = strings.TrimSpace(ep.IPAddressV4)
}
if host == "" || ep.Port == 0 {
    continue
}
addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
network[addr] = nodeAccountId
```

The only checks are: non-empty host and non-zero port. There is no:
- TLS/HTTPS enforcement
- Cryptographic signature on the response
- IP address allowlist or range validation
- Domain name validation against known Hiero infrastructure

The returned `network` map is passed directly to `hiero.ClientForNetwork(netmap)` in `pinger/sdk_client.go` line 22, making the SDK client point entirely to whatever addresses the mirror node response contained.

**Root cause:** Blind trust in an unauthenticated, unencrypted HTTP response to construct the consensus node network map.

**Exploit flow:**
1. Attacker positions themselves to intercept HTTP traffic between the pinger and the mirror node (MITM on the same LAN/pod network, ARP spoofing, or DNS hijacking of the mirror node hostname).
2. Pinger starts with `network=other` and calls `buildNetworkFromMirrorNodes` → `fetchMirrorNodeNetwork`.
3. Attacker intercepts the GET request and returns a crafted `nodesEnvelope` JSON where every `serviceEndpoint` contains attacker-owned IP:port pairs, with valid-looking `node_account_id` values (e.g., `"0.0.3"`).
4. `fetchMirrorNodeNetwork` accepts the response (HTTP 200, non-empty host, non-zero port) and builds a network map pointing entirely to rogue nodes.
5. `hiero.ClientForNetwork(netmap)` creates the SDK client with this poisoned map.
6. Every `cryptoTransfer.Execute(client)` call in `transfer.go` (line 33) sends signed transactions to rogue nodes.

**Why existing checks fail:**

- Line 90–93: Only validates HTTP status code (200–299). A spoofed response trivially returns 200.
- Lines 103–109: Only validates `NodeAccountID` string format via `hiero.AccountIDFromString`. The attacker supplies valid account ID strings.
- Lines 114–119: Only rejects empty host or zero port. Attacker supplies non-empty values.
- No TLS check anywhere in `pinger/` (confirmed: zero matches for `tls`, `certificate`, `verify` in pinger source).

### Impact Explanation

All HBAR transfer transactions signed by the operator key are submitted to attacker-controlled gRPC endpoints. Rogue nodes can: (1) charge non-standard transaction fees, draining the operator account; (2) silently drop transactions while the pinger logs false success; (3) replay or front-run signed transactions. The operator's private key signs every transaction, so the attacker receives fully valid, signed payment instructions. Financial loss is direct and repeatable for the lifetime of the pinger process.

### Likelihood Explanation

The default configuration uses plain HTTP (`http://mirror-rest:5551`). In Kubernetes or Docker deployments — the target environment given the `Dockerfile` and `/tmp/alive` liveness probe — a network-adjacent attacker (compromised sidecar, rogue pod on the same overlay network, or ARP spoofing on a flat cluster network) can perform MITM with no Hiero credentials whatsoever. DNS hijacking of the mirror node hostname requires only local DNS control, which is achievable in many cluster configurations. The attack is fully repeatable: every pinger restart re-fetches the network map, re-triggering the injection.

### Recommendation

1. **Enforce HTTPS**: Reject any `cfg.mirrorRest` URL that does not use the `https://` scheme at startup in `loadConfig()`.
2. **Pin or validate TLS**: Use a custom `http.Transport` with a pinned CA or certificate fingerprint for the mirror node connection.
3. **Validate returned addresses**: After decoding `nodesEnvelope`, cross-check returned IP addresses against a configurable allowlist or CIDR range before adding them to the network map.
4. **Verify response integrity**: If the mirror node supports it, verify a cryptographic signature or HMAC on the `/network/nodes` response.
5. **Minimum**: At startup, log a fatal error if `cfg.mirrorRest` uses plain HTTP when `network=other`.

func fetchMirrorNodeNetwork(

	httpClient *http.Client,

	if err != nil {
		return nil, false, err
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
	}
	defer resp.Body.Close()

	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500

	}

	}
