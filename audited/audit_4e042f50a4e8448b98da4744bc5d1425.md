### Title
Unparameterized IN-Clause Interpolation in `getInClauseSubQuery` Enables Query Plan Cache Pollution DoS

### Summary
`TokenService.getInClauseSubQuery()` in `rest/service/tokenService.js` directly interpolates validated numeric token ID values into the SQL string instead of using parameterized placeholders, producing a unique SQL string for every distinct combination of `token.id=eq:X` values. PostgreSQL caches a query plan per unique SQL string, so an unprivileged attacker can exhaust the plan cache by sending many requests with different token ID combinations, forcing repeated re-planning and degrading database performance for all users.

### Finding Description
**Code path:**

`rest/controllers/tokenController.js` `extractTokensRelationshipQuery()` (lines 34–35) collects every `token.id=eq:X` filter into `inConditions`: [1](#0-0) 

`rest/service/tokenService.js` `getInClauseSubQuery()` (lines 78–87) then pushes the raw values into a plain JS array and interpolates them directly into the SQL string: [2](#0-1) 

The values are **never added to the `params` array**; instead the template literal `` `in (${tokenIdInParams})` `` produces strings like `in (1,2,3)`, `in (4,5,6)`, etc. Each unique combination is a distinct SQL string that PostgreSQL must parse, plan, and cache separately.

**Root cause:** The function intentionally skips the parameterized-query mechanism used everywhere else in the codebase (e.g., `conditions` in `getTokenRelationshipsQuery` at lines 53–57 use `$${params.length}` placeholders). [3](#0-2) 

**Why existing checks are insufficient:**

`buildFilters` in `rest/utils.js` (lines 1241–1247) enforces `maxRepeatedQueryParameters` per single request, limiting the number of IDs in one IN-clause: [4](#0-3) 

However, this check is per-request only. Across many requests, an attacker can generate an unbounded number of distinct SQL strings (combinatorial explosion of token ID sets), each consuming a plan-cache slot. There is no cross-request deduplication or rate-limiting on unique query shapes.

### Impact Explanation
PostgreSQL's shared plan cache (`pg_prepared_statements`, `pg_stat_statements`, and the internal plan cache) has a finite capacity. Flooding it with unique query strings causes:
1. Frequent cache evictions, forcing re-planning of legitimate queries.
2. Elevated CPU and memory pressure on the database server.
3. Increased query latency for all concurrent users of the mirror node.

Because the `token_account` table is queried on every account-token-relationship lookup, degraded plan-cache performance has broad blast radius across the API surface.

### Likelihood Explanation
The endpoint `/accounts/:id/tokens` requires no authentication. An attacker needs only an HTTP client and knowledge of valid token IDs (publicly enumerable via the same API). With `maxRepeatedQueryParameters` (default observed as a small integer, e.g., 100), each request can inject up to that many IDs; rotating the set across requests trivially generates thousands of unique SQL strings. The attack is cheap, repeatable, and requires no special privileges.

### Recommendation
Replace the direct interpolation with proper parameterized placeholders, consistent with how `conditions` are handled:

```js
getInClauseSubQuery(inConditions, params) {
  if (isEmpty(inConditions)) return '';
  const placeholders = inConditions.map((condition) => {
    params.push(condition.value);
    return `$${params.length}`;
  });
  return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${placeholders.join(',')})`;
}
```

This produces a stable query shape (same number of `$N` placeholders for the same number of IDs) and keeps values out of the SQL string, allowing PostgreSQL to reuse cached plans.

### Proof of Concept
```bash
# Enumerate some token IDs first (public endpoint)
# Then flood with rotating sets of token.id=eq:X

BASE="https://<mirror-node>/api/v1/accounts/0.0.1234/tokens"

for i in $(seq 1 5000); do
  # Each request uses a different combination of token IDs
  A=$((RANDOM % 10000))
  B=$((RANDOM % 10000))
  C=$((RANDOM % 10000))
  curl -s "${BASE}?token.id=eq:0.0.${A}&token.id=eq:0.0.${B}&token.id=eq:0.0.${C}" &
done
wait
```

Each request generates a unique SQL string such as `... in (A,B,C) ...` that PostgreSQL must plan independently. After thousands of iterations the plan cache is saturated, and legitimate queries experience measurable latency increases observable via `pg_stat_statements` or slow-query logs.
