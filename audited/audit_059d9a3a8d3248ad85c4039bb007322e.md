### Title
Unauthenticated Endpoint `GET /contracts/results/:transactionIdOrHash/actions` Allows Unbounded DB Query Amplification via Ethereum Hash Flooding

### Summary
The `getContractActions()` handler in `rest/controllers/contractController.js` accepts any syntactically valid 32-byte Ethereum hash from any unauthenticated caller and unconditionally issues at least one database query per request with no application-level rate limiting present in the REST service. An attacker flooding this endpoint with valid-format hashes forces repeated indexed lookups against `contract_transaction_hash` (and optionally `contract_action`), exhausting the DB connection pool and degrading the actions endpoint for all legitimate users.

### Finding Description
**Exact code path:**

`getContractActions()` at [1](#0-0)  checks `utils.isValidEthHash(transactionIdOrHash)` — a pure format check — and immediately issues a DB query:

```js
if (utils.isValidEthHash(transactionIdOrHash)) {
  tx = await ContractService.getContractTransactionDetailsByHash(utils.parseHexStr(transactionIdOrHash));
```

`getContractTransactionDetailsByHash` executes: [2](#0-1) 

```sql
select * from contract_transaction_hash
where hash = $1
order by (transaction_result = <success>) desc, consensus_timestamp desc
limit 1
```

If the hash resolves (i.e., `tx.length > 0`), a second query fires immediately: [3](#0-2) 

```js
const rows = await ContractService.getContractActionsByConsensusTimestamp(
  consensusTimestamp, payerAccountId, filters, order, limit
);
```

which executes: [4](#0-3) 

**Root cause — failed assumption:** The code assumes that syntactic validation of the hash is sufficient protection. There is no application-level rate limiting anywhere in the REST service. A `grep` across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `rateLimiter` returns zero matches. The only throttle infrastructure found is in the separate Java `web3` service: [5](#0-4)  — it does not protect the Node.js REST service.

The `acceptedContractActionsParameters` set only allows `index`, `limit`, and `order` — no authentication or caller-identity check is enforced: [6](#0-5) 

### Impact Explanation
Each inbound request with a syntactically valid 32-byte hex hash causes a minimum of one indexed DB lookup against `contract_transaction_hash`. If the hash exists, a second query against `contract_action` fires. Under sustained flood conditions the DB connection pool is exhausted, query latency spikes across all endpoints sharing the same pool, and the actions endpoint becomes unavailable to legitimate users. No economic damage occurs, but service availability is degraded — consistent with the griefing classification.

### Likelihood Explanation
Preconditions are minimal: no account, no API key, no authentication of any kind. A valid Ethereum hash is any 64-character hex string prefixed with `0x`. An attacker can generate these trivially in a loop. The attack is repeatable, stateless, and requires no prior knowledge of the system. A single machine with a modest network connection can sustain thousands of requests per second.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP (or global) rate limiter middleware in the REST service (e.g., `express-rate-limit`) applied to `/contracts/results/:transactionIdOrHash/actions` before the handler executes.
2. **Short-circuit on non-existent hash**: The current code already throws `NotFoundError` when `tx.length === 0`, but the DB round-trip still occurs. Consider a lightweight cache (e.g., Bloom filter or short-TTL negative cache) for recently queried hashes that returned no results.
3. **Infrastructure enforcement**: Ensure the Traefik middleware chain for the REST service (`charts/hedera-mirror-rest/templates/middleware.yaml`) includes a `rateLimit` middleware, and treat it as a required control rather than an optional deployment concern.

### Proof of Concept
```bash
# Generate and flood with random valid-format Ethereum hashes
for i in $(seq 1 10000); do
  HASH=$(openssl rand -hex 32)
  curl -s "https://<mirror-node-host>/api/v1/contracts/results/0x${HASH}/actions" &
done
wait
```

Each iteration issues at least one `SELECT` against `contract_transaction_hash`. At scale, DB connection pool saturation causes latency increases and timeouts across all REST endpoints sharing the pool.

public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
