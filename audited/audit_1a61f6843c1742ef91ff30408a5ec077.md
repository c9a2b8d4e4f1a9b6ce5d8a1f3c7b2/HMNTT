### Title
Unauthenticated `account.balance=ne:0` Filter Causes Full Entity Table Scan Leading to DB CPU Exhaustion

### Summary
The `getAccounts()` handler in `rest/accounts.js` accepts the `ne` (not-equal) operator for the `account.balance` filter parameter without restriction. This causes the generated SQL to include a `e.balance != 0` predicate that cannot be satisfied by a standard B-tree index, forcing a full scan of the `entity` table. An unprivileged attacker flooding concurrent requests with this parameter can exhaust database CPU, preventing the mirror node from confirming new transactions.

### Finding Description

**Code path:**

In `rest/accounts.js` `getAccounts()` (lines 335–388), validation is performed via `utils.validateReq(req, acceptedAccountsParameters)`, then the balance filter is parsed:

```js
const balanceQuery = toQueryObject(utils.parseBalanceQueryParam(req.query, 'e.balance'));
``` [1](#0-0) 

`parseBalanceQueryParam` in `rest/utils.js` (lines 628–635) directly interpolates the operator into SQL with no operator restriction:

```js
const parseBalanceQueryParam = (parsedQueryParams, columnName) => {
  return parseParams(
    parsedQueryParams[constants.filterKeys.ACCOUNT_BALANCE],
    (value) => value,
    (op, value) => (isNumeric(value) ? [`${columnName}${op}?`, [value]] : null),
    false
  );
};
``` [2](#0-1) 

**Root cause — missing operator restriction:**

`filterValidityChecks` in `rest/utils.js` for `ACCOUNT_BALANCE` only validates the value is a positive long; it does **not** restrict the operator:

```js
case constants.filterKeys.ACCOUNT_BALANCE:
  ret = isPositiveLong(val, true);
  break;
``` [3](#0-2) 

Contrast this with `BLOCK_NUMBER` and `SLOT`, which explicitly exclude `ne` via `basicOperators`:

```js
const basicOperators = Object.values(constants.queryParamOperators).filter(
  (o) => o !== constants.queryParamOperators.ne
);
// ...
case constants.filterKeys.BLOCK_NUMBER:
  ret = (isPositiveLong(val, true) || isHexPositiveInt(val, true)) && includes(basicOperators, op);
  break;
case constants.filterKeys.SLOT:
  ret = isValidSlot(val) && includes(basicOperators, op);
  break;
``` [4](#0-3) [5](#0-4) [6](#0-5) 

No equivalent guard exists for `ACCOUNT_BALANCE`.

**Resulting SQL:**

The request `GET /api/v1/accounts?account.balance=ne:0` produces a query of the form:

```sql
select ...
from entity e
left join entity_stake es on es.id = e.id
where e.type in ('ACCOUNT', 'CONTRACT')
  and NOT(e.num = 2 and e.memo = 'Mirror node created synthetic treasury account')
  and e.balance != 0
order by e.id asc
limit 25
``` [7](#0-6) [8](#0-7) 

A `!= 0` predicate on `e.balance` is non-sargable: PostgreSQL cannot use a standard B-tree index to satisfy it and must perform a sequential scan or full index scan of the `entity` table before applying the `LIMIT`.

**Confirmed exploitability:** The k6 load-test file `tools/k6/src/rest/test/accountsBalanceNe.js` explicitly exercises this exact query pattern, confirming the endpoint accepts and processes it: [9](#0-8) 

### Impact Explanation

The REST API is the mirror node's primary read interface. The `entity` table grows proportionally with the number of accounts on the Hedera network (potentially millions of rows). Each `account.balance=ne:0` request forces a full scan of this table. Concurrent flooding of this endpoint saturates database CPU, causing query queues to back up. This degrades or halts the mirror node's ability to ingest and confirm new transactions, constituting a network-level denial of service for any consumer relying on the mirror node for transaction confirmation. No authentication or special privilege is required.

### Likelihood Explanation

The attack requires zero privileges — any HTTP client can send the request. The endpoint is publicly documented in the OpenAPI spec with `ne` listed as a valid example operator for `account.balance`: [10](#0-9) 

The attack is trivially repeatable with standard HTTP tooling (curl, wrk, k6). No rate limiting for the REST API's `/accounts` endpoint was found in the codebase (the throttle configuration present applies only to the `web3` Java module, not the Node.js REST service). [11](#0-10) 

### Recommendation

1. **Reject `ne` for `account.balance`** in `filterValidityChecks` by applying the same `basicOperators` guard already used for `BLOCK_NUMBER` and `SLOT`:
   ```js
   case constants.filterKeys.ACCOUNT_BALANCE:
     ret = isPositiveLong(val, true) && includes(basicOperators, op);
     break;
   ```
2. **Add a partial index** on `entity(balance)` where `balance != 0` if the `ne:0` use case is intentionally supported.
3. **Add per-IP rate limiting** to the Node.js REST service for expensive scan-inducing endpoints.
4. **Add a DB-level statement timeout** for REST API queries to bound worst-case query duration.

### Proof of Concept

```bash
# Single request — confirm it is accepted and executes (no 400 error)
curl -s "https://<mirror-node>/api/v1/accounts?account.balance=ne:0" | jq .accounts | head

# Flood attack — exhaust DB CPU with concurrent non-indexable scans
seq 1 200 | xargs -P 200 -I{} \
  curl -s "https://<mirror-node>/api/v1/accounts?account.balance=ne:0" -o /dev/null

# Observe: DB CPU spikes to 100%, subsequent transaction-confirmation queries time out
```
