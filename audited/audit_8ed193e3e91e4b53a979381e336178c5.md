### Title
Unbounded Concurrent Subscription Exhaustion via Unauthenticated `subscribeTopic()` Calls

### Summary
Any unauthenticated client can open an unlimited number of TCP connections to the gRPC endpoint and saturate the server with concurrent `subscribeTopic()` calls. Each subscription instantiates a full reactive pipeline — including a `Flux.merge(safetyCheck, live)` with a `boundedElastic`-scheduled safety-check branch, a live listener subscription, and a historical DB retrieval — with no global cap on active subscriptions. The only per-connection guard (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening additional connections, causing progressive resource exhaustion that degrades throughput for all legitimate subscribers.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) constructs, per call:

```
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);          // DB query
Flux<TopicMessage> live       = Flux.defer(() -> incomingMessages(topicContext));       // listener sub
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());                                      // scheduler task
Flux<TopicMessage> flux = historical
        .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(...)));
```

**Root cause — failed assumption:** The design assumes that the Netty-level `maxConcurrentCallsPerConnection = 5` (default) is a meaningful global bound. It is not. It is a *per-connection* limit. There is no:
- global subscription count cap enforced in code (the `subscriberCount` `AtomicLong` is a Micrometer gauge only — it is never checked against a threshold before accepting a new subscription),
- per-IP connection limit (no `maxConnections` or `maxConnectionAge` is set in `GrpcConfiguration`),
- authentication or authorization gate on `subscribeTopic()`.

**Per-subscription resource cost:**
1. `safetyCheck` calls `subscribeOn(Schedulers.boundedElastic())` — each subscription schedules work on the shared bounded-elastic thread pool (default cap: `10 × CPU cores`).
2. `live` → `incomingMessages()` → `topicListener.listen(newFilter)`:
   - Under `POLL` type: a new independent DB polling loop is created per subscriber, each consuming a HikariCP connection every 500 ms.
   - Under `REDIS` type (default): each subscriber attaches a new `onBackpressureBuffer` + `publishOn(Schedulers.boundedElastic())` chain to the shared Redis flux, adding per-subscriber buffer allocation (`maxBufferSize = 16384` entries) and scheduler pressure.
3. `historical` issues an immediate DB query per subscription.
4. `pastEndTime()` (when `endTime` is supplied) creates a `repeatWhen` loop polling every 30 s indefinitely.

**Why the existing check fails:**

```java
// GrpcConfiguration.java line 33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// NettyProperties.java line 14 — default = 5
```

This only limits calls on a single TCP connection. An attacker opens C connections → C × 5 = 5C simultaneous subscriptions. No code path checks `subscriberCount` and rejects new subscriptions.

### Impact Explanation

With N open subscriptions:
- **`Schedulers.boundedElastic()` saturation**: both the `safetyCheck` branch and the REDIS listener's `publishOn` use this pool. Saturation causes queuing/starvation for all reactive pipelines sharing it, including legitimate subscribers.
- **HikariCP pool exhaustion** (POLL mode): each subscriber holds a DB connection for the duration of the subscription. Pool exhaustion causes all new DB operations (including for legitimate subscribers and the importer) to block or fail.
- **Heap pressure**: each subscription allocates a `TopicContext`, filter objects, and (REDIS mode) a 16 384-slot backpressure buffer. At thousands of subscriptions this drives GC pressure and eventual OOM.
- **Throughput degradation**: legitimate subscribers experience increased latency, missed messages (backpressure buffer overflow → `ERROR` signal), and eventual connection drops.

Severity is **Medium** (griefing / availability degradation; no economic damage to the Hedera network itself, only to mirror node service consumers).

### Likelihood Explanation

- **No credentials required**: the gRPC port (5600) is publicly reachable.
- **Trivial to script**: a single attacker machine can open thousands of TCP connections and issue `subscribeTopic` RPCs in a tight loop using any gRPC client library.
- **Persistent**: subscriptions with no `endTime` never self-terminate (`pastEndTime` returns `Flux.never()`), so each malicious subscription remains active until the attacker disconnects.
- **Repeatable**: after a restart the attack can be immediately re-launched.

### Recommendation

1. **Enforce a global subscription cap**: check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` when the limit is exceeded.
2. **Add per-IP connection and subscription limits**: configure `NettyServerBuilder.maxConnectionAge()`, `maxConnectionIdle()`, and a per-IP call counter (e.g., via a gRPC `ServerInterceptor`).
3. **Add a gRPC-level rate limiter / interceptor**: mirror the `ThrottleManagerImpl` pattern already used in the `web3` module — apply it to `subscribeTopic` calls per remote address.
4. **Cap `boundedElastic` scheduler usage**: avoid `subscribeOn(Schedulers.boundedElastic())` inside per-subscriber pipelines; use a dedicated, bounded scheduler with a rejection policy.
5. **Set `endTime` enforcement**: reject subscriptions with no `endTime` after a configurable maximum duration to prevent indefinitely-held resources.

### Proof of Concept

```python
import grpc
import threading
# proto: consensus_service_pb2, consensus_service_pb2_grpc

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC  = "0.0.12345"   # any valid topic id
CONNS  = 500           # connections; each carries 5 streams = 2500 subscriptions

def flood(i):
    chan = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(chan)
    for _ in range(5):          # maxConcurrentCallsPerConnection default
        req = consensus_service_pb2.ConsensusTopicQuery(
            topic_id=...,       # set to TOPIC, no end_time
        )
        threading.Thread(target=lambda: list(stub.subscribeTopic(req)),
                         daemon=True).start()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNS)]
for t in threads: t.start()
for t in threads: t.join()
# Monitor: hiero_mirror_grpc_subscribers gauge climbs to ~2500
# Observe: HikariCP active connections saturate, boundedElastic queue depth grows,
#          legitimate subscriber latency increases, eventual OOM or connection refusals.
```
