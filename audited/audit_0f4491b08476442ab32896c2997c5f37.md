### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent EVM Address Lookups in `getEncodedId()`

### Summary
The REST API's `getEncodedId()` function unconditionally issues a live database query for every unique EVM address input, with no application-level rate limiting and no caching of DB lookup results. The default connection pool is only 10 connections per instance. An unauthenticated attacker sending concurrent requests with unique valid EVM addresses can saturate the pool, causing all subsequent legitimate requests to queue and time out, rendering the REST API unresponsive.

### Finding Description

**Exact code path:**

`rest/service/entityService.js`, `getEncodedId()` lines 118–137: [1](#0-0) 

When `entityIdString` is a valid EVM address (40 hex chars, optionally `0x`-prefixed), `EntityId.isValidEntityId()` returns true, `EntityId.parseString()` produces an `EntityId` with `evmAddress !== null`, and the code unconditionally calls `getEntityIdFromEvmAddress()`: [2](#0-1) 

This executes `entityFromEvmAddressQuery` directly against the DB: [3](#0-2) 

via `BaseService.getRows()` → `pool.queryQuietly()`: [4](#0-3) 

**Root cause — no caching of DB results:**

The `parseCached()` LRU cache in `entityId.js` (lines 301–333) caches only the parsed `EntityId` object (the string-to-struct conversion), not the DB lookup result: [5](#0-4) 

Even for a repeated EVM address, `getEntityIdFromEvmAddress()` is still called and still queries the DB. For unique addresses (the attack case), there is zero cache benefit.

**Root cause — no rate limiting:**

`rest/server.js` registers no rate-limiting middleware: [6](#0-5) 

The Traefik ingress middleware for the REST chart contains only `circuitBreaker` and `retry` — no `rateLimit` or `inFlightReq`: [7](#0-6) 

Furthermore, `global.middleware` defaults to `false`, meaning even this minimal Traefik chain is disabled by default: [8](#0-7) 

(Contrast with the Rosetta chart, which explicitly configures `inFlightReq` and `rateLimit`.)

**Root cause — small default connection pool:**

The default `maxConnections` is **10** per REST instance: [9](#0-8) 

Configured in `rest/dbpool.js`: [10](#0-9) 

With `statementTimeout` of 20 seconds and `connectionTimeout` of 20 seconds, each saturating request holds a connection for up to 20 seconds before timing out.

### Impact Explanation

With a pool of 10 connections and no rate limiting, an attacker maintaining ≥10 concurrent in-flight requests with unique EVM addresses fully saturates the pool. All subsequent legitimate requests queue waiting for a free connection; after `connectionTimeout` (20 s default) they fail. The REST API becomes effectively unresponsive. With up to 15 HPA replicas each holding 10 connections (150 total), and pgbouncer capping `mirror_rest` at 250 server-side connections, a sustained flood can exhaust the DB-side pool as well, affecting all REST instances simultaneously. This constitutes a denial-of-service against the mirror node REST API without any authentication or privileged access.

### Likelihood Explanation

The attack requires only the ability to send HTTP GET requests to any public REST endpoint that resolves an account/contract by EVM address (e.g., `/api/v1/accounts/0x<40 hex chars>`, `/api/v1/contracts/0x<40 hex chars>`). No credentials, tokens, or special knowledge are needed. Generating unique 40-character hex strings is trivial. The attack is repeatable, low-cost, and requires no brute force — a single attacker with modest bandwidth can sustain it indefinitely.

### Recommendation

1. **Add an in-process result cache** for `getEntityIdFromEvmAddress()` lookups (keyed by the hex EVM address), similar to how the importer's `EntityIdServiceImpl` uses a Spring Cache for EVM address lookups.
2. **Enable application-level rate limiting** in the REST Express server (e.g., `express-rate-limit`) scoped per IP, before DB-touching handlers.
3. **Enable Traefik `inFlightReq` and `rateLimit` middleware** for the REST chart (as already done for Rosetta), and set `global.middleware: true` in production deployments.
4. **Increase `maxConnections`** or use a connection queue with a bounded wait, and add a concurrency limit per request source.

### Proof of Concept

```bash
# Generate 50 unique EVM addresses and fire concurrent requests
for i in $(seq 1 50); do
  addr=$(python3 -c "import secrets; print('0x' + secrets.token_hex(20))")
  curl -s "https://<mirror-node-host>/api/v1/accounts/${addr}" &
done
wait
# All 10 pool connections are held; subsequent legitimate requests
# receive connection timeout errors or hang for up to 20 seconds.
```

Preconditions: Public network access to the mirror node REST API. No authentication required.
Trigger: ≥10 concurrent GET requests to `/api/v1/accounts/0x<unique_hex>` or any endpoint routing through `getEncodedId()` with EVM address input.
Result: DB connection pool exhausted; REST API unresponsive to legitimate traffic for the duration of the attack.
