### Title
Unbounded Stale LRU Cache in `getCachedTokens()` Permanently Serves Outdated Token Metadata to All Users

### Summary
`tokenService.js` initializes a `quickLru` cache with only `maxSize` and no `maxAge`/TTL. Once a token's `decimals`, `freeze_status`, and `kyc_status` are written into the cache by any unprivileged user's request, those values are never refreshed — `putTokenCache()` explicitly skips updates for already-cached entries, and `getCachedTokens()` never re-queries cached token IDs. Any on-chain change to token metadata after the initial cache population is permanently invisible to the API until the entry is evicted by LRU size pressure.

### Finding Description

**Cache initialization — no TTL:** [1](#0-0) 

`quickLru` supports a `maxAge` option, but it is not configured here. Entries live until evicted by size.

**`putTokenCache()` refuses to update existing entries:** [2](#0-1) 

If a token is already cached, the function returns immediately without comparing or refreshing the stored value.

**`getCachedTokens()` only fetches uncached IDs:** [3](#0-2) 

Lines 139–146 split token IDs into cached vs. uncached. Cached IDs are returned directly from the stale snapshot; only uncached IDs hit the database. There is no staleness check, no version comparison, and no periodic refresh.

**`decimals` is unconditionally overwritten from cache:** [4](#0-3) 

`row.decimals = cachedToken.decimals` (line 108) always replaces the live DB value. `freeze_status` and `kyc_status` use `??` (nullish coalescing), so when the `token_account` row stores `null` (meaning "inherit token default"), the stale cached default is used.

**Trigger surface — unprivileged public endpoint:** [5](#0-4) 

`GET /api/v1/accounts/{id}/tokens` requires no authentication. Any caller can trigger `getTokenAccounts()` → `getCachedTokens()` for any token ID, populating the cache with the current on-chain snapshot.

**Exploit flow:**
1. Attacker (or any user) calls `GET /api/v1/accounts/A/tokens` for an account holding token T. Token T is not yet cached.
2. `getCachedTokens()` queries the DB and writes `{decimals, freeze_status, kyc_status}` for T into the LRU cache.
3. The token admin subsequently changes T's `decimals` (Hedera allows this via `TokenUpdateTransaction`) or updates the token's default freeze/KYC policy. The mirror node importer writes the new values to the `token` table.
4. All future calls to `GET /api/v1/accounts/*/tokens` for any account holding T return the stale cached `decimals`, `freeze_status` default, and `kyc_status` default — indefinitely, until LRU eviction.

### Impact Explanation
- **Stale `decimals`**: Applications that rely on the REST API to display or compute token amounts will use the wrong decimal precision, potentially misrepresenting balances by orders of magnitude.
- **Stale `freeze_status` / `kyc_status` defaults**: Wallets and compliance tools querying token relationship status will see incorrect freeze/KYC state for accounts whose `token_account` row stores `null` (inherit-from-token). This can cause incorrect "frozen" or "KYC-passed" signals.
- The mirror node is read-only, so on-chain funds are not directly at risk, but the API becomes an unreliable source of truth for token metadata for the entire lifetime of the cache entry.

### Likelihood Explanation
- No authentication or special privilege is required to trigger the initial cache population.
- The window of exploitation is open from the moment any request caches a token until LRU eviction (which, with a large `maxSize`, may never happen in practice for popular tokens).
- Token metadata changes (`decimals`, freeze/KYC defaults) are infrequent but valid on-chain operations; the mismatch between on-chain state and cached API state is permanent once it occurs.

### Recommendation
1. **Add a `maxAge` TTL** to the `quickLru` constructor so entries expire and are re-fetched from the DB:
   ```js
   const tokenCache = new quickLru({
     maxSize: config.cache.token.maxSize,
     maxAge: config.cache.token.maxAge, // e.g., 60_000 ms
   });
   ```
2. **Remove the early-return guard in `putTokenCache()`** or replace it with a TTL-aware check, so that updated token metadata can overwrite stale entries.
3. **In `getCachedTokens()`**, do not unconditionally trust cached `decimals`; at minimum, use the same `??` pattern as `freeze_status` so a live DB value can override the cache.

### Proof of Concept
```
# Step 1: Populate the cache for token 0.0.1234
GET /api/v1/accounts/0.0.5678/tokens?token.id=0.0.1234
# → Response includes decimals=6 (cached)

# Step 2: Token admin submits TokenUpdateTransaction on-chain changing decimals to 8.
# Mirror node importer updates token table: decimals=8.

# Step 3: Any subsequent request for any account holding 0.0.1234
GET /api/v1/accounts/0.0.9999/tokens?token.id=0.0.1234
# → Response still shows decimals=6 (stale cache, never refreshed)
# This persists until the LRU evicts the entry due to size pressure.
```

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
