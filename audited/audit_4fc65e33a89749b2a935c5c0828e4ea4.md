### Title
Cache Stampede via Expired `CACHE_MANAGER_ENTITY` TTL Exhausts DB Connection Pool in `findByIdAndDeletedIsFalse`

### Summary
The `findByIdAndDeletedIsFalse` method in `EntityRepository` is annotated with Spring's `@Cacheable` backed by a Caffeine cache configured with `expireAfterWrite=1s`. Spring's `@Cacheable` AOP proxy does not use Caffeine's atomic per-key loading (`Cache.get(key, mappingFunction)`), so when the 1-second TTL expires, all concurrent threads that simultaneously observe a cache miss will each independently issue a DB query before any single thread repopulates the cache. The global rate limiter (500 req/s) does not prevent this per-key thundering herd, and the resulting burst of simultaneous DB queries can exhaust the HikariCP connection pool on affected web3 nodes.

### Finding Description

**Exact code path:**

`findByIdAndDeletedIsFalse` is annotated at: [1](#0-0) 

The `CACHE_MANAGER_ENTITY` bean is built using `setCacheSpecification(cacheProperties.getEntity())`: [2](#0-1) 

The entity cache specification is hardcoded as `expireAfterWrite=1s,maximumSize=10000,recordStats`: [3](#0-2) 

**Root cause — failed assumption:**

Spring's `@Cacheable` AOP interceptor performs a non-atomic check-then-act:
1. `cache.get(key)` → miss
2. invoke underlying repository method (DB query)
3. `cache.put(key, result)`

Caffeine's stampede protection only applies when the caller uses `cache.get(key, loader)` directly (which provides per-key locking). `CaffeineCacheManager` wraps Caffeine but Spring's `@Cacheable` proxy does **not** use the loading form. Therefore, N threads that simultaneously observe the expired entry all proceed to step 2 concurrently.

**Exploit flow:**

1. Attacker identifies any valid `entityId` (e.g., a popular contract address resolved to an entity ID — publicly visible on-chain).
2. Attacker sends a sustained flood of `eth_call` / `eth_estimateGas` requests targeting that contract, keeping the request rate just below the 500 req/s global throttle.
3. At the exact 1-second TTL boundary, the cached entry for that `entityId` expires.
4. All in-flight threads simultaneously observe a cache miss and each issues `SELECT * FROM entity WHERE id = ? AND deleted IS NOT TRUE` against the DB.
5. With up to 500 concurrent threads competing for HikariCP connections (default pool is small), the pool is exhausted; subsequent requests queue or time out.
6. The 3-second `statementTimeout` means blocked threads hold connections for up to 3 seconds, compounding the exhaustion window.
7. Repeating every second (matching the TTL) sustains the pressure.

### Impact Explanation

DB connection pool exhaustion on the targeted web3 node causes all concurrent requests — not just those for the targeted `entityId` — to fail or experience severe latency. Because the attack repeats every second (matching the TTL), the node cannot recover between waves. Multiple web3 nodes can be targeted simultaneously by an attacker with sufficient request volume, potentially degrading or disabling the web3 JSON-RPC service across a significant fraction of deployed nodes. The `unless = "#result == null"` guard does not help: `Optional.empty()` is not `null`, so non-existent entity lookups are cached, but the stampede occurs on the expiry boundary regardless of whether the entity exists. [4](#0-3) 

### Likelihood Explanation

No authentication is required to call the web3 JSON-RPC endpoint. The attacker needs only a valid contract/account address (trivially obtained from public block explorers) and the ability to sustain ~500 HTTP requests per second, which is achievable from a single machine or small botnet. The 1-second TTL creates a perfectly predictable, repeating attack window. The throttle bucket is global and shared across all `entityId` values, so it does not prevent per-key stampedes. [5](#0-4) [6](#0-5) 

### Recommendation

1. **Switch to Caffeine's async loading cache** (`refreshAfterWrite` + `CacheLoader`) so that only one thread reloads an expired entry while others continue to receive the stale value. This is the canonical stampede fix.
2. Alternatively, replace `expireAfterWrite=1s` with `refreshAfterWrite=1s` in the Caffeine spec string; Caffeine will then refresh asynchronously and serve the stale value to concurrent readers during the refresh.
3. As a defense-in-depth measure, add per-key (per-`entityId`) rate limiting in addition to the existing global throttle, so a single `entityId` cannot monopolize the DB connection pool.
4. Increase the HikariCP connection pool size or add a connection acquisition timeout that returns a 429 rather than blocking indefinitely.
