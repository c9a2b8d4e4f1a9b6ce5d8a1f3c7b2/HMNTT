### Title
Retry Loop Creates Fresh Transaction Per Attempt, Enabling Multiple On-Chain Transfers Per Tick via gRPC Error Injection

### Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a brand-new `TransferTransaction` (and therefore a new transaction ID) on every loop iteration. If an on-path attacker injects a transport-level error on the gRPC response path after the consensus node has already accepted the transaction, `Execute(client)` returns an error while the transfer is already in-flight on-chain. The retry loop then submits an entirely independent second (third, …, eleventh) transaction, each with a distinct ID that the network cannot deduplicate, resulting in up to `maxRetries+1` real HBAR transfers per tick.

### Finding Description
**Exact code path** — `pinger/transfer.go`, `submitWithRetry()`, lines 23–57:

```go
for i := 1; i <= attempts; i++ {          // attempts = maxRetries+1 (default 11)
    ...
    cryptoTransfer := hiero.NewTransferTransaction().   // ← NEW tx, NEW tx-ID every iteration
        AddHbarTransfer(...).
        AddHbarTransfer(...)

    resp, err := cryptoTransfer.Execute(client)        // submits to consensus node via gRPC
    if err == nil { ... return nil }
    // on error: fall through to backoff and retry
}
```

**Root cause** — `hiero.NewTransferTransaction()` is called inside the loop. The Hiero SDK assigns a fresh transaction ID (operator-account + current nanosecond timestamp) to each new object. Hedera's network-level deduplication is keyed on transaction ID; because every retry carries a different ID, no deduplication occurs.

**Failed assumption** — The code assumes that if `Execute()` returns an error the transaction was never accepted by the network. This is false for transport-layer failures: gRPC sends the serialised transaction in the request body. If the consensus node receives and passes precheck on the request, then the TCP connection is reset (or a gRPC RST_STREAM / GOAWAY is injected) before the `TransactionResponse` reaches the client, `Execute()` raises a transport error while the transfer is already committed to consensus.

**Exploit flow**
1. Attacker positions themselves on the network path between the pinger pod and the Hedera consensus node (same cloud VPC, compromised router, BGP hijack, etc.).
2. For each tick, the attacker allows the gRPC `submitTransaction` request to pass through in full, then injects a TCP RST or gRPC-level error on the response stream.
3. `Execute(client)` returns a transport error; the transfer is already in the network.
4. The loop sleeps `baseBackoff` (default 2 s) and repeats from step 2.
5. After `maxRetries+1` (default 11) iterations the function returns `"all attempts failed"` and the ticker fires again after `interval` (default 1 s).

**Why existing checks are insufficient**
- The only guard at the top of the loop is `ctx.Err()` (line 24), which only fires on shutdown — it does not detect prior successful submission.
- There is no idempotency key, no transaction-ID reuse, and no "check-then-act" receipt probe before retrying.

### Impact Explanation
With default configuration (`maxRetries=10`, `amountTinybar=10000`, `interval=1s`) an attacker can cause up to 11 × 10 000 = 110 000 tinybar (0.0011 HBAR) to be transferred per tick instead of 10 000. Sustained over time this drains the operator account at 11× the intended rate. On mainnet where the operator account may hold significant HBAR for operational purposes, this constitutes a direct, repeatable loss of funds. Severity: **Critical** (direct financial loss, no user interaction required beyond the network-path position).

### Likelihood Explanation
The precondition — on-path capability to inject TCP/gRPC errors — is achievable without breaking TLS: TCP RST injection requires only the ability to spoof source-IP packets on the path, which is possible from the same cloud subnet, a compromised network device, or via BGP route hijacking. Cloud-hosted services regularly face this threat model. The attack is fully repeatable every tick and requires no credentials or application-layer access.

### Recommendation
1. **Create the transaction once, outside the loop**, so all retries reuse the same transaction ID. The Hedera network will reject a duplicate submission within the validity window (~120 s) with `DUPLICATE_TRANSACTION`, which should be treated as a success (the original was accepted).
2. **Before retrying**, query the mirror node or consensus node for the transaction ID to determine whether it was already accepted, and skip re-submission if it was.
3. **Treat transport errors differently from application errors**: a gRPC transport error on the response path is ambiguous (may-have-submitted); an application-level precheck failure (e.g. `INVALID_ACCOUNT_ID`) is safe to retry with a new transaction.
