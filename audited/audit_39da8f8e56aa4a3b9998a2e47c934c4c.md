### Title
Cache Monopolization via `expireAfterAccess` TTL Reset in `findRuntimeBytecode` Contract Cache

### Summary
The `CACHE_NAME_CONTRACT` cache used by `ContractRepository.findRuntimeBytecode()` is configured with `expireAfterAccess=1h` and `maximumSize=1000`. An unprivileged attacker can fill all 1000 cache slots with known contract IDs and periodically re-access them every ~59 minutes to reset the TTL, keeping those entries permanently resident. Because Caffeine's W-TinyLFU eviction policy favors high-frequency entries, new (low-frequency) contract IDs are consistently evicted, preventing them from being sustainably cached and forcing every lookup for non-attacker contracts to hit the database.

### Finding Description

**Exact code path:**

`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable` pointing to `CACHE_MANAGER_CONTRACT`: [1](#0-0) 

The cache manager is built from `CacheProperties.contract`, whose default value is: [2](#0-1) 

The cache manager bean is: [3](#0-2) 

**Root cause and failed assumption:**

The design assumes that `expireAfterAccess=1h` will naturally rotate entries out over time. The failed assumption is that no single actor will deliberately keep a full set of 1000 entries alive by periodically re-accessing them. Caffeine's W-TinyLFU eviction policy compounds this: when the cache is at `maximumSize=1000` and a new entry arrives, Caffeine compares the new entry's estimated frequency (≈0 for a first-time access) against the victim entry's frequency. Attacker-controlled entries, accessed once per hour, maintain a non-zero frequency score that consistently beats any new entry's score of 0, causing new entries to be immediately evicted or rejected from the main segment.

**Exploit flow:**

1. Attacker enumerates 1000 valid contract IDs (all public on-chain data).
2. Attacker calls the web3 `/api/v1/contracts/call` endpoint (or any path that triggers `ContractBytecodeReadableKVState.readFromDataSource()`) for each of the 1000 IDs, filling the cache.
3. Every ~59 minutes, attacker re-accesses all 1000 IDs, resetting each entry's `expireAfterAccess` timer and incrementing their frequency counters in Caffeine's sketch.
4. The cache remains permanently saturated. Any new contract ID requested by a legitimate user gets a cache miss, hits the DB, is briefly admitted to Caffeine's window segment, then evicted in favor of the attacker's high-frequency entries.

**Why existing checks are insufficient:**

The global rate limiter (`requestsPerSecond=500`) is shared across all users and is not per-IP or per-identity: [4](#0-3) 

The attacker's maintenance traffic is 1000 requests per 59 minutes ≈ 0.28 req/sec, consuming less than 0.06% of the global budget. No per-source rate limiting, no cache-key quotas, and no authentication exist to prevent this.

The `unless = "#result == null"` guard only prevents caching of missing contracts; it does not limit how many entries a single caller can place in the cache. [5](#0-4) 

### Impact Explanation
Every call to `findRuntimeBytecode()` for a contract not in the attacker's pinned set results in a database query on every invocation. Under normal operation the cache absorbs repeated lookups for the same contracts; under this attack, the DB receives the full query load for all non-pinned contracts. This degrades response latency and increases DB CPU/IO for the mirror node operator. The service remains functionally correct (queries still return data), so the impact is a sustained performance degradation / resource exhaustion on the DB layer — consistent with the griefing classification.

### Likelihood Explanation
The attack requires only: (a) knowledge of 1000 valid contract IDs (trivially obtained from any Hedera block explorer or the mirror node's own REST API), and (b) the ability to send ~0.28 HTTP requests per second to a public endpoint. No credentials, no special network position, and no economic stake are required. The attack is fully repeatable and can be automated with a simple script. It is sustainable indefinitely at negligible cost to the attacker.

### Recommendation
1. **Switch to `expireAfterWrite`** for the contract cache (as already done for `contractState`, `entity`, `token`, `fee`). `expireAfterWrite` is not resettable by read access, so periodic reads cannot extend entry lifetime indefinitely.
2. **Alternatively, add a per-source (IP-based) rate limit** at the ingress layer so that a single client cannot sustain the ~1000 req/hour maintenance traffic without being throttled.
3. **Increase `maximumSize`** to reduce the proportion of the cache a single attacker can monopolize, though this alone does not eliminate the attack.
4. **Consider `expireAfterWrite` + a shorter TTL** (e.g., `expireAfterWrite=10m`) to bound how long any entry — attacker-controlled or not — can remain without a DB refresh, which also improves data freshness for updated contracts.

### Proof of Concept

```python
import requests, time

MIRROR_NODE = "https://<mirror-node-host>"
# Step 1: Collect 1000 known contract IDs from the public REST API
contracts = requests.get(f"{MIRROR_NODE}/api/v1/contracts?limit=1000").json()["contracts"]
contract_ids = [c["contract_id"] for c in contracts]  # e.g. ["0.0.1001", ...]

def trigger_cache(contract_id):
    # Any eth_call targeting the contract triggers findRuntimeBytecode()
    payload = {"to": contract_id, "data": "0x", "gas": 100000}
    requests.post(f"{MIRROR_NODE}/api/v1/contracts/call", json=payload)

# Step 2: Fill the cache
for cid in contract_ids:
    trigger_cache(cid)

# Step 3: Refresh every 59 minutes to reset expireAfterAccess TTL
while True:
    time.sleep(59 * 60)
    for cid in contract_ids:
        trigger_cache(cid)
    # Cache remains full; new contracts cannot be sustainably cached
```

After the loop is running, any legitimate user querying a contract outside the attacker's 1000-entry set will experience a cache miss on every request, with all bytecode lookups falling through to the database.
