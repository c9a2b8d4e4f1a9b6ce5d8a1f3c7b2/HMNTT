### Title
Shared `Schedulers.boundedElastic()` Thread Pool Exhaustion via Unbounded Concurrent Subscriptions in `PollingTopicListener`

### Summary
`PollingTopicListener` holds a single `Scheduler scheduler = Schedulers.boundedElastic()` field that is the global Reactor bounded-elastic scheduler singleton, shared across every subscription and across other components in the same JVM. Because each subscription schedules blocking JDBC polling work onto this shared pool and there is no global limit on the number of concurrent subscriptions, an unprivileged attacker can open enough gRPC connections to exhaust the thread pool, causing all polling tasks—including those of legitimate subscribers—to queue and stall.

### Finding Description
**Exact code path:**

- `PollingTopicListener.java` line 31: `private final Scheduler scheduler = Schedulers.boundedElastic();`
  `Schedulers.boundedElastic()` returns the **global** Reactor singleton scheduler (not a fresh instance), capped at `10 × CPU_count` threads with a 100 000-task queue.
- `listen()` lines 39 and 43: the scheduler is used for both `delaySubscription(interval, scheduler)` and `RepeatSpec…withScheduler(scheduler)`. When the delay fires, the scheduler thread directly executes `Flux.defer(() -> poll(context))`.
- `poll()` lines 51-62: calls `topicMessageRepository.findByFilter(newFilter)` which returns a `Stream<TopicMessage>` — a **blocking JDBC call** that holds the scheduler thread for the full query duration (up to the 10 s `statementTimeout`).

**Root cause / failed assumption:**
The design assumes a bounded number of concurrent subscriptions. There is no global subscription cap: `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, `GrpcConfiguration` line 33) limits calls *per connection* but places no ceiling on the number of connections. An attacker opens `C` connections × 5 subscriptions = `5C` concurrent polling loops, each occupying a scheduler thread during every DB round-trip.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` — per-connection only; unlimited connections accepted.
- No IP-level connection throttle or global subscriber count gate in the gRPC path.
- `subscriberCount` gauge in `TopicMessageServiceImpl` is metrics-only; it enforces nothing.
- The global `Schedulers.boundedElastic()` is also consumed by `SharedTopicListener.publishOn` (line 25), `TopicMessageServiceImpl` safety-check (line 70), and `PollingTopicMessageRetriever`, so exhaustion cascades to all reactive paths.

### Impact Explanation
When the bounded-elastic thread pool is saturated, `delaySubscription` and `repeatWhen` tasks queue behind the cap. Every active subscriber—including legitimate ones—experiences polling delays proportional to queue depth. With a 500 ms default interval and, say, 200 threads exhausted by attacker subscriptions, legitimate polls are delayed by seconds to tens of seconds, effectively stalling live message delivery for all `POLL`-mode subscribers. Because the same global scheduler backs other reactive chains, the degradation is not isolated to `PollingTopicListener`.

### Likelihood Explanation
The `POLL` listener type is not the default (`REDIS` is), so this requires a non-default deployment configuration. However, when `POLL` is active, exploitation requires only an unauthenticated gRPC client that can open many TCP connections—no credentials, no special protocol knowledge. The gRPC `subscribeTopic` RPC is publicly exposed on port 5600. The attacker needs to sustain `~(10 × CPU_count) / 5` connections to saturate the pool; on a typical 4-core node that is only ~8 connections, each with 5 subscriptions. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic` by rejecting new subscriptions when `subscriberCount` exceeds a configurable threshold.
2. **Isolate the scheduler**: replace `Schedulers.boundedElastic()` with a dedicated `Schedulers.newBoundedElastic(threadCap, queueSize, "poll-listener")` instance so exhaustion cannot cascade to other reactive chains.
3. **Add a per-IP or per-connection subscription rate limit** at the Netty/gRPC layer (e.g., `maxConnectionAge`, `maxConnectionIdle`, or an interceptor-based token-bucket).
4. **Offload blocking DB work explicitly** with `.subscribeOn(dedicatedScheduler)` so the timing scheduler is never blocked by JDBC calls.

### Proof of Concept
```
# Precondition: server configured with hiero.mirror.grpc.listener.type=POLL
# Attacker script (pseudocode):

for i in range(200):          # open 200 connections
    conn = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(conn)
    for j in range(5):        # 5 subscriptions per connection = 1000 total
        stub.subscribeTopic(ConsensusTopicQuery(
            topicID=<any_valid_topic>,
            consensusStartTime=0   # forces continuous polling from genesis
        ), async=True)

# Result: ~1000 concurrent polling loops saturate Schedulers.boundedElastic()
# (default cap ~40-100 threads depending on CPU count).
# Legitimate subscriber polls queue behind attacker tasks.
# Observable: legitimate subscribers see message delivery delays of
# (queue_depth / thread_count) × poll_interval seconds.
```
