### Title
Log Injection via Unsanitized `req.ip` with Unconditional `trust proxy: true` in `responseHandler()`

### Summary
`rest/server.js` sets `app.set('trust proxy', true)`, causing Express to unconditionally accept the leftmost value of any client-supplied `X-Forwarded-For` header as `req.ip`. In `responseHandler()`, `req.ip` is interpolated directly into a `logger.info()` call with no sanitization, allowing any unauthenticated attacker to inject arbitrary text — including newlines — into the application log, forging audit trail entries that attribute smart contract queries to arbitrary IP addresses.

### Finding Description
**Root cause — trust proxy misconfiguration:**

`rest/server.js` line 58 sets `trust proxy` to the boolean `true`: [1](#0-0) 

With `trust proxy: true`, Express trusts every hop in `X-Forwarded-For` unconditionally and sets `req.ip` to the leftmost (client-controlled) value in that header. There is no allowlist of trusted upstream proxy IPs, no hop-count limit, and no format validation.

**Sink — unsanitized interpolation into log:**

`rest/middleware/responseHandler.js` line 57 interpolates `req.ip` directly into a template literal passed to `logger.info()`: [2](#0-1) 

`req.ip` is never stripped of newlines (`\n`, `\r`), ANSI escape codes, or other control characters before reaching the logger. The same pattern exists in `httpErrorHandler.js`: [3](#0-2) 

And in `responseCacheHandler.js`: [4](#0-3) 

**Exploit flow:**

1. Attacker sends a GET request to any contract endpoint (e.g., `/api/v1/contracts/results/...`) with a crafted header:
   ```
   X-Forwarded-For: 1.2.3.4\n2024-01-01 INFO 8.8.8.8 GET /api/v1/contracts/results/0xdeadbeef in 0 ms: 200
   ```
2. Express sets `req.ip = "1.2.3.4\n2024-01-01 INFO 8.8.8.8 GET /api/v1/contracts/results/0xdeadbeef in 0 ms: 200"`.
3. `logger.info(...)` writes a multi-line entry. Log parsers and SIEMs see a fabricated second log line attributing a contract query to `8.8.8.8`.
4. The attacker's real IP never appears in the log.

**Why existing checks fail:**

There are no checks. The test suite for `responseHandler` hard-codes `req.ip: '127.0.0.1'` in the mock: [5](#0-4) 

No test exercises a spoofed or malformed `X-Forwarded-For` value, so the injection path is untested and unguarded.

### Impact Explanation
An attacker can forge audit log entries for any contract result query, making malicious smart contract interactions appear to originate from a different IP (e.g., a known-good internal address or a competitor's IP). This directly undermines forensic investigation and compliance audit trails. Log injection with embedded newlines can also corrupt structured log formats (JSON, logfmt) and cause SIEM false negatives or parser errors.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request — no authentication, no special role, no network position beyond reaching the REST API port. The `X-Forwarded-For` header is trivially set by any HTTP client (`curl`, browser, script). The attack is repeatable at will and leaves no trace of the real source IP in the affected log lines.

### Recommendation
1. **Restrict `trust proxy`**: Replace `app.set('trust proxy', true)` with a specific proxy count or CIDR allowlist matching only the actual upstream load balancer/reverse proxy IPs (e.g., `app.set('trust proxy', 1)` if there is exactly one trusted proxy hop). This prevents arbitrary clients from controlling `req.ip`. [1](#0-0) 

2. **Sanitize before logging**: Strip or encode control characters from `req.ip` (and `req.originalUrl`) before interpolating into log strings:
   ```js
   const safeIp = (req.ip ?? '').replace(/[\r\n\t]/g, '_');
   logger.info(`${safeIp} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${code}`);
   ``` [6](#0-5) 

3. Apply the same fix to `httpErrorHandler.js` lines 34 and 39, and `responseCacheHandler.js` line 85.

### Proof of Concept
```bash
# Inject a fake log line attributing a contract query to 8.8.8.8
curl -H $'X-Forwarded-For: 1.2.3.4\n2024-01-01 00:00:00 INFO 8.8.8.8 GET /api/v1/contracts/results/0xdeadbeef in 0 ms: 200' \
     http://<mirror-node-rest>:5551/api/v1/contracts/results/0x1234

# Expected log output (two lines, second is forged):
# 2024-01-01 00:00:00 INFO 1.2.3.4
# 2024-01-01 00:00:00 INFO 8.8.8.8 GET /api/v1/contracts/results/0xdeadbeef in 0 ms: 200
# ... GET /api/v1/contracts/results/0x1234 in 3 ms: 200
```

The forged line appears as a legitimate audit entry; the attacker's real connection IP is absent from the log.
