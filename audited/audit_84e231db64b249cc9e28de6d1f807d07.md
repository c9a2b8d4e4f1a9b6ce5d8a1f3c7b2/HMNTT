### Title
Thundering Herd DoS via Cache Stampede in `updateOperationAccountAlias` with No Default Rate Limiting

### Summary
The `Block()` function in `rosetta/app/services/block_service.go` calls `updateOperationAccountAlias()` which performs a non-atomic cache check followed by a `GetAccountAlias()` database query for each uncached account ID. Because the LRU cache has no singleflight/stampede protection and the Traefik rate-limiting middleware is **disabled by default** (`global.middleware: false`), an unauthenticated attacker can flood the `/block` endpoint with concurrent requests targeting blocks with many unique account IDs, causing a thundering herd of parallel DB queries that exhausts the connection pool.

### Finding Description

**Code path:**

`Block()` → `FindBetween()` → `updateOperationAccountAlias()` → per-operation `GetAccountAlias()` DB call

In `rosetta/app/services/block_service.go` lines 104–129, `updateOperationAccountAlias` iterates over every operation of every transaction and performs a non-atomic cache check:

```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
    ...
    s.entityCache.Set(result.GetId(), result)
}
```

The `entityCache` is a shared process-level `go-generics-cache` LRU (created once in `NewBlockAPIService`, line 34–37). The library provides only separate `Get` and `Set` operations — there is no singleflight coalescing or atomic check-and-set. When N concurrent goroutines all call `Get()` for the same uncached account ID before any of them calls `Set()`, all N goroutines proceed to issue a `GetAccountAlias()` DB query (`rosetta/app/persistence/account.go` line 117: `db.Raw(selectCryptoEntityWithAliasById, ...)`).

**Root cause — failed assumption:** The code assumes the shared cache prevents redundant DB calls, but the check-then-act is not atomic. Under concurrency, the cache provides no protection for the first wave of requests for any given account ID.

**Why existing checks are insufficient:**

1. **Traefik middleware is disabled by default.** `charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false`. The middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3) gates on `{{ if and .Values.global.middleware .Values.middleware -}}`, so the `inFlightReq` (5/IP) and `rateLimit` (10 req/s) Traefik rules are **not deployed** in a default installation.

2. **No application-level rate limiting.** `main.go` lines 217–219 only wrap the router with `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which limit request rates or concurrency.

3. **DB pool is finite.** `rosetta/app/db/db.go` line 33 sets `MaxOpenConnections` to the configured value (default 100 per `docs/configuration.md` line 660). Each concurrent `GetAccountAlias()` call acquires a connection; exhausting the pool causes all subsequent DB operations across the entire service to queue or fail.

**Exploit flow:**

- Attacker identifies a block with many transactions (e.g., a busy block on mainnet with hundreds of crypto transfers, each with multiple operations involving distinct account IDs not yet in the cache).
- Attacker sends hundreds of concurrent `POST /block` requests for that block (or rotating through recent blocks to defeat the cache).
- Each request calls `FindBetween()` (one DB query) then `updateOperationAccountAlias()` which issues one `GetAccountAlias()` DB query per uncached operation account ID.
- With 100 concurrent requests × 50 operations per block = 5,000 simultaneous DB queries, the 100-connection pool is immediately saturated.
- All other DB-dependent endpoints (account balance, network status, etc.) stall waiting for a connection.

### Impact Explanation

Database connection pool exhaustion causes a full denial of service for all endpoints that require DB access. The `statementTimeout` of 20 seconds (`docs/configuration.md` line 662) means each saturating query holds a connection for up to 20 seconds, sustaining the outage. The attacker needs only to maintain a modest flood rate to keep the pool exhausted continuously. No authentication or special privileges are required — the `/block` endpoint is publicly accessible.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond a valid block identifier (publicly available on-chain), and is trivially scriptable with any HTTP load tool (e.g., `wrk`, `hey`, `ab`). The default deployment disables the only rate-limiting defense. The attack is repeatable and sustainable with minimal resources. Any public-facing Rosetta deployment using default Helm values is exposed.

### Recommendation

1. **Add singleflight coalescing** to `updateOperationAccountAlias`: use `golang.org/x/sync/singleflight` so that concurrent cache misses for the same account ID result in exactly one DB call, with all waiters receiving the same result.
2. **Enable the Traefik middleware by default**: change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`.
3. **Add application-level concurrency limiting**: use a semaphore or `golang.org/x/net/netutil.LimitListener` to cap the number of simultaneously processed `/block` requests.
4. **Batch `GetAccountAlias` lookups**: instead of one DB query per account ID, collect all uncached IDs from a request and issue a single `WHERE id = ANY(...)` query.

### Proof of Concept

```bash
# Identify a recent block index from the network
BLOCK_INDEX=12345678

# Send 200 concurrent /block requests (requires no auth)
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$((BLOCK_INDEX + i))}}" &
done
wait

# Observe: subsequent requests to /block, /account/balance, /network/status
# all time out or return 500 errors due to DB connection pool exhaustion.
# The service recovers only after in-flight queries expire (up to 20s each).
```
