### Title
Shared Per-Filter Throttle Bucket Exhaustion Enables Block-Targeted DoS

### Summary
The `THROTTLE` action in `ThrottleManagerImpl.throttle()` consumes tokens from a single global `Bucket` per `RequestProperties` filter, with no per-user or per-IP isolation. An unprivileged attacker who knows (or guesses) the configured filter expression for the `BLOCK` field can exhaust that shared bucket using a fraction of the global rate limit, causing all legitimate requests matching that filter to be denied until the bucket refills.

### Finding Description

**Code path:**

`ThrottleManagerImpl.throttle()` (lines 44–48) iterates over admin-configured `RequestProperties` entries and calls `action()` for each matching one:

```java
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);
    }
}
```

`action()` (lines 70–74) for `THROTTLE` consumes from the per-`RequestProperties` bucket:

```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

The bucket is created in `RequestProperties.createBucket()` (lines 63–68) with capacity = `rate` (default 100, hard-capped at 100 by `@Max(100)`), refilling at `rate` tokens per second — a single global instance shared across all callers.

**Root cause:** The per-filter `Bucket` is a process-wide singleton. There is no per-IP, per-user, or per-session sub-bucket. Any caller who passes the global `rateLimitBucket` check (default 500 req/sec) can consume tokens from the per-filter bucket on behalf of all users.

**Filter field exposure:** `RequestFilter.FilterField.BLOCK` extracts `ContractCallRequest::getBlock` and converts it to a string via `String.valueOf()`. `BlockType.toString()` returns `"latest"` for the `LATEST` sentinel and the decimal block number otherwise. An attacker can set the `block` JSON field to any value accepted by `BlockType.of()` — tags (`latest`, `earliest`, `safe`, `finalized`, `pending`), decimal numbers, or hex numbers — making it trivial to match any admin-configured `BLOCK` filter expression.

**Why existing checks fail:**
- The global `rateLimitBucket` (default 500 req/sec) is checked first, but its capacity is 5× the per-filter bucket's maximum capacity (100 tokens). An attacker exhausts the per-filter bucket using only 100 of their 500 allowed requests per second.
- The `limit` counter in `RequestProperties.test()` (line 46) defaults to `Long.MAX_VALUE` — effectively unlimited.
- The random-sampling bypass (`RandomUtils.secure().randomLong(0L, 100L) >= rate`) is explicitly skipped for `THROTTLE` actions (line 50: `if (action != ActionType.THROTTLE && ...)`), so every matching request always hits the bucket.

### Impact Explanation
When an admin deploys a `THROTTLE` filter on `BLOCK=latest` (or any block value) — a documented, supported configuration — an attacker can continuously exhaust the 100-token bucket within the first 100 requests of each second. All subsequent legitimate requests targeting that block value receive HTTP 429 for the remainder of that second. By repeating every second, the attacker achieves near-continuous denial of service for all users querying that specific block, effectively censoring access to block-specific transaction history via the `/api/v1/contracts/call` endpoint.

### Likelihood Explanation
- **No privilege required.** Any unauthenticated HTTP client can send `ContractCallRequest` JSON with an arbitrary `block` field.
- **Filter expressions are discoverable.** Common values (`latest`, `earliest`, specific block numbers) are a small, guessable set. Operators commonly throttle `latest`-block queries.
- **Trivially repeatable.** A single attacker script sending 100 req/sec (well within the 500 req/sec global limit) sustains the DoS indefinitely.
- **Conditional on admin configuration.** The attack only applies when a `THROTTLE`-action filter is configured; the default `request = List.of()` is safe. However, the feature is explicitly documented and intended for production use.

### Recommendation
1. **Per-source isolation:** Replace the single global `Bucket` in `RequestProperties` with a `LoadingCache<String, Bucket>` keyed by client IP (extracted from `X-Forwarded-For` or `RemoteAddr`). Each source gets its own token bucket, preventing one caller from exhausting the shared pool.
2. **Alternatively, enforce a minimum per-filter rate higher than the global rate limit**, or document that `THROTTLE` filters are not safe against adversarial traffic without an upstream IP-level rate limiter (e.g., the GCP `maxRatePerEndpoint` already present in the Helm charts).
3. **Add a per-source sub-limit** inside `action()` so that a single IP can consume at most `rate / N` tokens per second from the shared bucket.

### Proof of Concept

**Precondition:** Admin has configured:
```yaml
hiero.mirror.web3.throttle.request:
  - action: THROTTLE
    rate: 10
    filters:
      - field: BLOCK
        type: EQUALS
        expression: "latest"
```

**Steps:**

1. Send 10 rapid requests with `block=latest` to exhaust the per-filter bucket:
```bash
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST \
    http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"block":"latest","data":"0x1079023a","from":"0x00000000000000000000000000000000000004e2","gas":21000,"to":"0x00000000000000000000000000000000000004e4","value":0}'
done
```

2. Send one more request — observe HTTP 429 ("Requests per second rate limit exceeded") even though the global 500 req/sec limit has not been reached.

3. Legitimate users sending `block=latest` now receive 429 for the remainder of the second.

4. Repeat step 1 every second to sustain the DoS indefinitely.
