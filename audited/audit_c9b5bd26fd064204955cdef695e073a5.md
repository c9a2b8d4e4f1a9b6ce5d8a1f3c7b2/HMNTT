### Title
Unbounded Concurrent Subscriptions Cause Aggregate Heap Exhaustion via Per-Subscriber `onBackpressureBuffer` in `SharedPollingTopicListener`

### Summary
`SharedTopicListener.listen()` applies an independent `onBackpressureBuffer(16384, ERROR)` per subscriber on top of a single shared hot Flux. No limit exists on the number of concurrent subscriptions, and no authentication is required to open a gRPC subscription. An unprivileged attacker can open thousands of concurrent subscriptions, each independently allocating up to 16384 `TopicMessage` objects on the heap, causing aggregate heap exhaustion and OOM crash.

### Finding Description

**Code path:**

`SharedTopicListener.listen()` ( [1](#0-0) ) applies `onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)` for every call, meaning every subscriber gets its own independent in-memory buffer.

`ListenerProperties.maxBufferSize` defaults to 16384 with no per-connection or global subscription count cap: [2](#0-1) 

`SharedPollingTopicListener` constructs a single `.share()` hot Flux (`topicMessages`) that is returned unchanged to every subscriber: [3](#0-2) [4](#0-3) 

`TopicMessageFilter.subscriberId` is auto-generated per filter instance and is never used as a deduplication or rate-limiting key: [5](#0-4) 

`CompositeTopicListener.listen()` performs no subscription count check before delegating: [6](#0-5) 

**Root cause:** The design assumes a small, bounded number of concurrent subscribers. There is no global or per-IP subscription counter, no connection throttle, and no deduplication on `subscriberId`. Each new gRPC `subscribeTopic` call creates a new `TopicMessageFilter` (with a fresh random `subscriberId`), calls `listen()`, and independently allocates a 16384-slot buffer on the heap.

**Why `BufferOverflowStrategy.ERROR` does not help:** It only terminates an individual subscriber when *its own* buffer overflows. The attacker simply reconnects immediately, re-allocating the buffer. Thousands of concurrent slow-consuming connections each hold their buffer simultaneously.

### Impact Explanation
Each `TopicMessage` carries a byte-array payload, running hash (48 bytes), timestamps, and object overhead — conservatively ~500 bytes per object. At 16384 messages/buffer × 500 bytes = ~8 MB per subscriber. With 2,000 concurrent subscriptions the aggregate buffer demand reaches ~16 GB, exhausting the JVM heap and triggering an OOM crash. This takes down the entire gRPC mirror node service, denying all legitimate subscribers access to topic data. Severity is **High** (non-network-based DoS against a critical infrastructure component).

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. An attacker needs only a gRPC client (e.g., `grpcurl` or a trivial script) and the ability to open many TCP connections. Opening 2,000 concurrent gRPC streams is well within the capability of a single commodity machine. The attack is repeatable: after an OOM crash and restart, the attacker can immediately re-trigger it. No special privileges, tokens, or insider knowledge are required.

### Recommendation
1. **Add a global and per-IP subscription limit** in `SharedTopicListener.listen()` or `CompositeTopicListener.listen()` using an `AtomicInteger` counter; reject new subscriptions above a configurable threshold (e.g., 500 total / 10 per IP).
2. **Reduce `maxBufferSize`** to a smaller default (e.g., 1024) and make it configurable with a lower ceiling.
3. **Enforce gRPC-level connection/stream limits** via Netty `maxConcurrentCallsPerConnection` in the server configuration.
4. **Require authentication/authorization** for `subscribeTopic` so that abusive clients can be identified and blocked.
5. **Track active subscriptions per `subscriberId`/IP** and reject duplicate or excessive registrations.

### Proof of Concept
```bash
# Install grpcurl; obtain the mirror node proto
# Open 2000 concurrent subscriptions to the same topic, consuming slowly

for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}, "consensusStartTime": {"seconds": 0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Each background process holds an open stream with its own 16384-message buffer.
# As the shared Flux emits messages, each subscriber's buffer fills independently.
# Aggregate heap usage = 2000 × 16384 × ~500 bytes ≈ 16 GB → OOM crash.
wait
```
