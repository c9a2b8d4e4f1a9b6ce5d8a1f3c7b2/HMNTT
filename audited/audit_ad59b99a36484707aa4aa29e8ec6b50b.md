### Title
Cross-Field Pagination Cursor Mismatch in `getBlocks` Causes Silent Block Skipping/Duplication

### Summary
When `GET /api/v1/blocks` is called with a `timestamp` filter, the query is sorted by `consensus_end`, but `generateNextLink` unconditionally anchors the pagination cursor on `block.number` (index). On the second page, the WHERE clause filters by `index > N` while still ordering by `consensus_end`, meaning any block with `index <= N` but `consensus_end` beyond the page boundary is silently dropped, and any block with `index > N` but `consensus_end` before the page boundary is duplicated across pages.

### Finding Description

**Code path:**

`rest/controllers/blockController.js` — `extractOrderByFromFilters` (lines 47–55) determines `orderBy` by taking the **first** filter in `blockWhereFilters` that appears in the request:

```js
const orderBy = filters
  .filter((f) => blockWhereFilters.includes(f.key))
  .map((f) => {
    return f.key === filterKeys.BLOCK_NUMBER ? RecordFile.INDEX : RecordFile.CONSENSUS_END;
  })[0];
```

When the request is `?timestamp=gt:0&order=asc`, `timestamp` is the first matching filter, so `orderBy = consensus_end`. The SQL becomes `ORDER BY consensus_end ASC`.

`generateNextLink` (lines 90–99) then **always** anchors the next link on `block.number`, regardless of `orderBy`:

```js
generateNextLink = (req, blocks, filters) => {
  return blocks.length
    ? utils.getPaginationLink(
        req,
        blocks.length !== filters.limit,
        {[filterKeys.BLOCK_NUMBER]: last(blocks).index},  // always index
        filters.order
      )
    : null;
};
```

This produces a next link like: `?timestamp=gt:0&order=asc&block.number=gt:N`

On page 2, `timestamp` still appears first in the URL, so `orderBy` remains `consensus_end`. But the WHERE clause now includes `index > N` as an additional constraint. The effective page-2 query is:

```sql
WHERE consensus_end > 0 AND index > N
ORDER BY consensus_end ASC
```

**Root cause:** The cursor field (`index`) does not match the sort field (`consensus_end`). The cursor `index > N` does not correctly represent "all blocks whose `consensus_end` is after the last block of page 1."

**Why existing checks fail:** There is no validation that the pagination cursor field matches the `orderBy` field. `extractOrderByFromFilters` and `generateNextLink` operate independently with no cross-check.

### Impact Explanation

If any blocks exist where `index` and `consensus_end` are not perfectly monotonically correlated:

- **Block skipping:** A block with `index <= N` but `consensus_end > T_last` (where `T_last` is the `consensus_end` of the last block on page 1) satisfies the page-2 ordering position but is excluded by `index > N`. It never appears on any page.
- **Block duplication:** A block with `index > N` but `consensus_end <= T_last` was already returned on page 1 (lower `consensus_end` → earlier in ASC order) and is returned again on page 2 (satisfies `index > N`).

A client iterating all blocks via the next link would receive an inconsistent, incomplete, or duplicated view of the ledger's block history — silently, with no error.

### Likelihood Explanation

The precondition — blocks with non-monotonic `consensus_end` relative to `index` — is unlikely under normal Hedera network operation, since blocks are produced sequentially. However:
- The code enforces no invariant guaranteeing monotonicity
- Clock skew, ingestion anomalies, or future network changes could produce such blocks
- Any unprivileged user can trigger this with a single unauthenticated GET request; no credentials or special state are required
- The next link is server-generated, so clients following it in good faith are affected

### Recommendation

`generateNextLink` must use a cursor that matches the actual `orderBy` field:

- When `orderBy = consensus_end` (timestamp filter present), the next link cursor should be `timestamp=gt:T_last` (the `consensus_end` of the last returned block), not `block.number=gt:N`.
- When `orderBy = index` (block.number filter present), `block.number=gt:N` is correct.

Concretely, pass `formattedFilters.orderBy` into `generateNextLink` and select the cursor key accordingly:

```js
generateNextLink = (req, blocks, filters) => {
  if (!blocks.length) return null;
  const lastBlock = last(blocks);
  const cursorMap = filters.orderBy === RecordFile.CONSENSUS_END
    ? {[filterKeys.TIMESTAMP]: lastBlock.consensusEnd}
    : {[filterKeys.BLOCK_NUMBER]: lastBlock.index};
  return utils.getPaginationLink(req, blocks.length !== filters.limit, cursorMap, filters.order);
};
```

### Proof of Concept

**Precondition:** Two blocks exist in the DB where block A has `index=100, consensus_end=T+10` and block B has `index=101, consensus_end=T+5` (B has higher index but lower `consensus_end` — non-monotonic).

**Step 1 — Page 1:**
```
GET /api/v1/blocks?timestamp=gt:0&order=asc&limit=1
```
Returns block B (`consensus_end=T+5`, `index=101`) — lowest `consensus_end` first.
Next link: `?timestamp=gt:0&order=asc&limit=1&block.number=gt:101`

**Step 2 — Page 2 (follow next link):**
```
GET /api/v1/blocks?timestamp=gt:0&order=asc&limit=1&block.number=gt:101
```
SQL: `WHERE consensus_end > 0 AND index > 101 ORDER BY consensus_end ASC LIMIT 1`

Block A has `index=100`, which does NOT satisfy `index > 101`, so it is **excluded**.
Result: empty page or next unrelated block. Block A is permanently skipped despite having `consensus_end=T+10 > T+5`. [1](#0-0) [2](#0-1) [3](#0-2)
