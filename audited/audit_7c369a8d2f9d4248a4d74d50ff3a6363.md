### Title
Unauthenticated Unbounded Block Query Enables Concurrent Memory Exhaustion DoS (No Rate Limiting on REST API)

### Summary
The `GET /api/v1/blocks` endpoint accepts `timestamp=gte:0` and `limit=<maxLimit>` from any unauthenticated user with no rate limiting, no in-flight request cap, and no per-IP throttle on the REST API. `RecordFileService.getBlocks()` executes a full-table scan bounded only by `maxLimit` (default 100) rows, and the Node.js process holds all result rows in memory before serializing the response. Under concurrent load from a single attacker, the aggregate in-flight heap allocation grows without bound until the process is OOM-killed or becomes unresponsive.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` → `BlockController.getBlocks()` (`rest/controllers/blockController.js:101-112`) → `RecordFileService.getBlocks()` (`rest/service/recordFileService.js:149-162`)

**Root cause 1 — No rate limiting on the REST API.**
`rest/server.js` registers the following middleware stack (lines 67-98):
- `express.urlencoded`, `express.json`, `cors`, optional `compression`
- `httpContext.middleware`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`

There is no `express-rate-limit`, no `express-slow-down`, no in-flight-request cap, and no per-IP throttle for any REST route. The rate limiting that exists in the codebase (`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`) applies exclusively to the web3/contract-call component, not to the Node.js REST service.

**Root cause 2 — Limit enforcement logic error allows maxLimit rows per request.**
`extractLimitFromFilters` in `blockController.js` lines 57-61:

```js
extractLimitFromFilters = (filters) => {
  const limit = findLast(filters, {key: filterKeys.LIMIT});
  const maxLimit = getEffectiveMaxLimit();
  return limit ? (limit.value > maxLimit ? defaultLimit : limit.value) : defaultLimit;
};
```

When `limit.value > maxLimit` the function falls back to `defaultLimit` (25) instead of capping at `maxLimit`. When `limit.value <= maxLimit` it passes `limit.value` through unchanged. The effective ceiling for an unauthenticated user is therefore `maxLimit` = **100** (confirmed by `rest/__tests__/config.test.js:324`: `{default: 25, max: 100, ...}`). Any request with `limit=100` receives exactly 100 rows.

**Root cause 3 — `timestamp=gte:0` produces a full-history scan bounded only by `limit`.**
`extractSqlFromBlockFilters` maps the `timestamp` filter key to `RecordFile.CONSENSUS_END` (`blockController.js:82-83`). With `timestamp=gte:0`, the WHERE clause becomes `WHERE consensus_end >= 0`, which matches every row in the table. The SQL executed by `getBlocks` (`recordFileService.js:152-158`) is:

```js
const query =
  RecordFileService.blocksQuery +
  `
  ${where}
  order by ${filters.orderBy} ${filters.order}
  limit ${filters.limit}
`;
const rows = await super.getRows(query, params);
return rows.map((recordFile) => new RecordFile(recordFile));
```

`filters.limit` is interpolated directly into the SQL string (not parameterized). All 100 rows are materialized into a JavaScript array before the response is written. Each `RecordFile` object carries: `count`, `hash` (96-char hex), `name`, `prev_hash`, `hapi_version_*`, `index`, `consensus_start`, `consensus_end`, `gas_used`, `logs_bloom`, `size`, `bytes` — roughly 1–3 KB per object. 100 rows ≈ 100–300 KB per response, held in heap until GC.

**Why existing checks are insufficient:**
- The `authHandler` (`rest/middleware/authHandler.js:15-36`) only sets a custom limit for authenticated users; unauthenticated requests proceed with no restriction beyond `responseLimit.max`.
- The optional response cache (`responseCacheCheckHandler`) is disabled by default (`config.cache.response.enabled && config.redis.enabled`).
- OpenAPI validation (`openApiValidator`) validates parameter types/ranges but does not throttle request rate.
- There is no `statementTimeout` enforcement visible at the query level for this path that would abort long-running DB queries under load.

### Impact Explanation
An attacker firing N concurrent `GET /api/v1/blocks?timestamp=gte:0&limit=100` requests causes the Node.js event loop to hold N × ~200 KB of heap simultaneously. At 500 concurrent connections (trivially achievable with `ab`, `wrk`, or `hey`), that is ~100 MB of live heap from block data alone, plus response serialization buffers, DB connection pool pressure, and V8 GC overhead. At 5,000 concurrent connections the process approaches the default ~1.5 GB V8 heap limit. OOM-kill or GC stall causes a complete service outage for all users — a partition-like condition where the mirror node REST API becomes unreachable. The DB connection pool (`db.pool.maxConnections`) is also exhausted, starving legitimate queries.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no rate limiting are required. The endpoint is publicly documented in `rest/api/v1/openapi.yml:421-440`. Any attacker with a single machine and a tool like `wrk -t12 -c400 -d60s "http://<host>/api/v1/blocks?timestamp=gte:0&limit=100"` can sustain the attack indefinitely. The attack is stateless, trivially scriptable, and repeatable. Mainnet mirror nodes are publicly accessible, making this a realistic threat.

### Recommendation

1. **Add rate limiting middleware** to the REST API in `rest/server.js`, applied before route handlers:
   ```js
   import rateLimit from 'express-rate-limit';
   app.use(rateLimit({ windowMs: 60_000, max: 200, standardHeaders: true }));
   ```
2. **Fix the limit-enforcement logic error** in `blockController.js:60` — replace `defaultLimit` with `maxLimit` so over-limit requests are capped, not silently reduced:
   ```js
   return limit ? Math.min(limit.value, maxLimit) : defaultLimit;
   ```
3. **Add a DB statement timeout** for block queries to prevent long-running scans from holding connections under load.
4. **Add an in-flight request cap** (e.g., `express-slow-down` or a semaphore) to bound concurrent heap usage.
5. **Consider lowering `responseLimit.max`** for the blocks endpoint specifically, as block records are larger than most other paginated resources.

### Proof of Concept

**Preconditions:** Public mirror node REST API reachable; no credentials required.

**Steps:**
```bash
# Single request — confirm 100 rows returned, no auth needed
curl -s "http://<mirror-node>/api/v1/blocks?timestamp=gte:0&limit=100" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(len(d['blocks']))"
# Expected output: 100

# Sustained concurrent flood
wrk -t16 -c500 -d120s \
  "http://<mirror-node>/api/v1/blocks?timestamp=gte:0&limit=100"

# Monitor Node.js heap (on server)
watch -n1 "ps aux | grep node"
# Observe RSS climbing; service begins returning 503/ECONNREFUSED
# or process is OOM-killed
```

**Expected result:** Under sustained 500-connection load, Node.js heap grows unboundedly (no rate limiting to shed load), DB connection pool is exhausted, and the REST API becomes unresponsive to all users within 60–120 seconds on a standard deployment.
