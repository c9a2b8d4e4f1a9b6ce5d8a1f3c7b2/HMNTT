### Title
Missing Entity-Type Validation in `getAccountCryptoAllowances` Allows Querying Crypto Allowances for Non-Account Entities

### Summary
`getAccountCryptoAllowances()` in `cryptoAllowanceController.js` calls `EntityService.getEncodedId()` with the raw path parameter. For plain numeric inputs, `getEncodedId()` performs a pure mathematical encoding with **no database lookup and no entity-type check**, returning a valid encoded ID for any entity — account, contract, or token alike. The resulting ID is then used directly as the `owner` filter against the `crypto_allowance` table, allowing an unprivileged caller to retrieve allowance records for non-account entities through an endpoint that is semantically and contractually restricted to accounts.

### Finding Description

**Exact code path:**

`rest/controllers/cryptoAllowanceController.js`, line 77: [1](#0-0) 

```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

`rest/service/entityService.js`, lines 120–124: [2](#0-1) 

```js
if (EntityId.isValidEntityId(entityIdString)) {
    const entityId = EntityId.parseString(entityIdString, {paramName});
    return entityId.evmAddress === null
      ? entityId.getEncodedId()          // ← pure math, zero DB access, zero type check
      : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
```

**Root cause:** When the input is a plain numeric string (e.g., `"95623"`), `evmAddress` is `null` and the function returns `entityId.getEncodedId()` — a deterministic bit-shift computation — without ever touching the database. No check is made that the entity exists, let alone that it is of type `ACCOUNT`.

**Confirmed by the test suite** — the test named `"EntityService.getEncodedId - No match"` (no entity in DB) still resolves successfully for both an account number and a contract number: [3](#0-2) 

**`isValidAccount()` exists but is never called** in this flow: [4](#0-3) 

The returned ID is passed directly into `extractCryptoAllowancesQuery` as `$1` in `owner = $1`: [5](#0-4) [6](#0-5) 

### Impact Explanation
In Hedera, contracts can hold HBAR and can be the subject of `CryptoApproveAllowance` transactions, meaning rows with a contract's encoded ID as `owner` can legitimately exist in the `crypto_allowance` table. An attacker querying `GET /accounts/{contractId}/allowances/crypto` will receive those rows. The endpoint's documented and intended scope is accounts only; returning allowance records for contracts or tokens violates the API contract and constitutes unintended information disclosure. Because Hedera is a public ledger the raw data is not secret, which limits severity, but the type-confusion bypass is a real, reproducible defect that undermines the integrity of the access model for this endpoint.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and zero special knowledge beyond knowing (or brute-forcing) a numeric entity ID. The Hedera entity ID space is sequential and public. Any unauthenticated HTTP client can trigger this in a single request. It is trivially repeatable and automatable.

### Recommendation
After `getEncodedId()` resolves, validate that the returned ID corresponds to an entity of type `ACCOUNT` before proceeding. The `entityExistenceQuery` already selects `entity.type`; extend `isValidAccount()` (or a new `assertIsAccount()` helper) to check `entity.type = 'ACCOUNT'` and call it in `getAccountCryptoAllowances()` immediately after line 77. Alternatively, add an `entityType` parameter to `getEncodedId()` itself so the type constraint is enforced centrally for all callers.

### Proof of Concept

**Preconditions:**
1. A contract entity exists with numeric ID `95623` (shard 0, realm 0, num 95623).
2. At least one `CryptoApproveAllowance` transaction has been submitted on-chain with that contract as the owner, creating a row in `crypto_allowance` with `owner = <encoded(95623)>`.

**Trigger:**
```
GET /api/v1/accounts/95623/allowances/crypto
```
No authentication header required.

**Result:**
The response body contains the `allowances` array populated with HBAR allowance records whose `owner` is the contract entity `0.0.95623`, data that the `/accounts/…/allowances/crypto` endpoint is not supposed to serve for non-account entities. The server returns HTTP 200 instead of 400/404.
