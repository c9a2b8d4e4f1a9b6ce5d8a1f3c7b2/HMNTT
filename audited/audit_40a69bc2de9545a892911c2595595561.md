### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Exposes Plaintext Postgres Wire Traffic Including Credentials and Balance Data

### Summary
`GetDsn()` in `rosetta/app/config/types.go` hardcodes `sslmode=disable` unconditionally, and the `Db` struct provides no SSL configuration field, making TLS impossible to enable without a code change. Every query result — including account balances, entity keys, and crypto transfer amounts — and the DB password itself transit the network in plaintext. An attacker with network-path access (e.g., a co-tenant on the same cloud VPC/subnet or a compromised Kubernetes node) can passively capture all of this without any privileges on the target hosts.

### Finding Description
**Exact code path:**

- `rosetta/app/config/types.go`, `GetDsn()`, line 51:
  ```go
  "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable"
  ```
  The `Db` struct (lines 39–47) has no `SslMode` or `SslCert` field; there is no configuration knob to override this.

- `rosetta/app/db/db.go`, `ConnectToDb()`, line 18:
  ```go
  db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
  ```
  This DSN is passed directly to the libpq-compatible driver, which will never negotiate TLS.

**Root cause:** The DSN is constructed with a hardcoded `sslmode=disable` literal. The PostgreSQL wire protocol sends all data — startup messages (which include the password), query text, and result rows — in plaintext when SSL is disabled.

**Exploit flow:**
1. Attacker gains passive network visibility on the path between the Rosetta pod and the PostgreSQL pod (same Kubernetes node, same cloud subnet, ARP poisoning on a flat LAN, or a compromised network tap).
2. Attacker captures TCP stream on port 5432.
3. The PostgreSQL startup packet contains the plaintext password (from the DSN).
4. Subsequent `Query` messages contain the raw SQL (e.g., `SELECT balance FROM account_balance WHERE account_id = …`).
5. `DataRow` response packets contain plaintext balance values, entity IDs, and public keys.

**Why existing checks fail:** There are no checks. The `Db` struct has no SSL fields, no runtime validation enforces TLS, and the test in `types_test.go` line 24 explicitly asserts the `sslmode=disable` string is correct, cementing it as intended behavior.

### Impact Explanation
Two distinct impacts:

1. **Credential theft:** The DB password is transmitted in the PostgreSQL startup message in plaintext. An attacker who captures it gains direct database access with the Rosetta service account's permissions, enabling bulk reads of all account balances, entity keys, and transaction history — a complete ledger map.

2. **Balance data disclosure:** Every `RetrieveBalanceAtBlock` call (via `account_service.go` → `account.go` queries `account_balance` and `crypto_transfer`) returns plaintext rows. An attacker can reconstruct a full map of account IDs to HBAR balances, identifying high-value targets. Combined with the exposed public keys (also returned in query results), this materially assists targeted attacks on those accounts through other vectors.

The Rosetta service is read-only, so the DB connection itself cannot move funds. However, credential theft from the wire gives the attacker the DB password, which — if the DB user has any write grants or if the same password is reused — escalates to direct data manipulation.

### Likelihood Explanation
In Kubernetes deployments (the primary deployment target, per `charts/hedera-mirror/values.yaml`), the Rosetta pod and the PostgreSQL pod communicate over the cluster network. A compromised pod on the same node can sniff `cbr0`/`veth` traffic without root using standard packet capture tools if `CAP_NET_RAW` is available (default in many cluster configurations) or via eBPF-based tools available to unprivileged users in some distributions. In cloud VPC deployments without micro-segmentation, any VM on the same subnet can ARP-spoof and intercept. The attack is passive, leaves no application-layer logs, and is repeatable indefinitely.

### Recommendation
1. Add an `SslMode` field to the `Db` struct in `rosetta/app/config/types.go` and include it in `GetDsn()`:
   ```go
   type Db struct {
       ...
       SslMode string `yaml:"sslMode"` // default: "require" or "verify-full"
   }

   func (db Db) GetDsn() string {
       sslMode := db.SslMode
       if sslMode == "" {
           sslMode = "require"
       }
       return fmt.Sprintf(
           "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
           db.Host, db.Port, db.Username, db.Name, db.Password, sslMode,
       )
   }
   ```
2. Set the default to `require` or `verify-full` (with CA cert validation) in the default config YAML.
3. Update `types_test.go` to assert `sslmode=require` (or the configured value) rather than `sslmode=disable`.
4. For production, use `sslmode=verify-full` with a trusted CA certificate to prevent MITM in addition to passive eavesdropping.

### Proof of Concept
**Preconditions:** Attacker has a pod or VM on the same network segment as the Rosetta-to-Postgres connection (e.g., same Kubernetes node or same cloud subnet).

**Steps:**
```bash
# 1. On attacker node, identify the Postgres port and start capture
tcpdump -i any -w rosetta_pg.pcap 'tcp port 5432'

# 2. Trigger any Rosetta balance query (from any client, no auth required)
curl -s -X POST http://<rosetta-host>:5700/account/balance \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.98"}}'

# 3. Stop capture and inspect with Wireshark or strings
strings rosetta_pg.pcap | grep -E "(password|balance|account_id|SELECT)"
```

**Expected result:** The pcap contains:
- The PostgreSQL startup message with `password=<plaintext_password>`
- Raw SQL query text including `account_balance`, `crypto_transfer`, `entity_id`
