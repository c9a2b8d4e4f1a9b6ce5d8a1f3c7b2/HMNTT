### Title
Unbounded `CaffeineCacheManager` Internal Cache Map Growth via Distinct ContractId Requests Leading to Heap Exhaustion

### Summary
The `cacheManagerSlotsPerContract` bean is configured without `setCacheNames()`, placing `CaffeineCacheManager` in dynamic mode where it stores every distinct cache name in an internal `ConcurrentHashMap<String, Cache>` that is never evicted. Each call to `cacheManagerSlotsPerContract.getCache(contractId.toString())` in `findStorageBatch()` registers a new `CaffeineCache` instance in this map permanently. Because the outer `contractSlotsCache` (bounded at `maximumSize=3000`) evicts entries without triggering any cleanup in `CaffeineCacheManager`'s internal map, the map grows monotonically with every distinct contractId ever seen, creating a permanent heap leak exploitable by any unauthenticated caller of the public `/api/v1/contracts/call` endpoint.

### Finding Description

**Exact code path:**

`EvmConfiguration.java` lines 107–112 — the `cacheManagerSlotsPerContract` bean is created with no `setCacheNames()` call: [1](#0-0) 

All other `CaffeineCacheManager` beans in the same file call `setCacheNames(Set.of(...))`, which locks the manager to a fixed set of names and prevents dynamic creation. This one does not. [2](#0-1) 

`ContractStateServiceImpl.java` lines 85–87 — `findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` as the value loader for the outer `contractSlotsCache`: [3](#0-2) 

**Root cause:**

Spring's `CaffeineCacheManager` maintains an internal `ConcurrentHashMap<String, Cache> cacheMap`. In dynamic mode (no `setCacheNames()`), every call to `getCache(name)` for an unseen name creates a new `CaffeineCache` and inserts it into `cacheMap`. This map has no eviction, no TTL, and no size bound — it is a permanent registry.

The outer `contractSlotsCache` is bounded (`maximumSize=3000`, `expireAfterAccess=5m`): [4](#0-3) 

When the outer cache evicts an entry for contractId X, it drops its reference to the `CaffeineCache` object. However, `cacheManagerSlotsPerContract.cacheMap` still holds `"X" -> CaffeineCache_X` indefinitely. The outer cache eviction has zero effect on `cacheMap`.

**Exploit flow:**

1. Attacker sends `POST /api/v1/contracts/call` with `to` set to distinct contract addresses (existing contracts that execute SLOAD opcodes, or a single proxy contract that cross-calls many contracts via STATICCALL/CALL).
2. Each distinct contractId that reaches `findStorageBatch()` triggers `cacheManagerSlotsPerContract.getCache(contractId.toString())`.
3. Each new contractId string inserts one `CaffeineCache` instance (wrapping a Caffeine cache configured with `maximumSize=1500`) into `cacheMap`.
4. After the outer `contractSlotsCache` fills to 3000 and begins evicting, new contractIds continue adding to `cacheMap` while old ones are never removed.
5. `cacheMap` grows without bound for the lifetime of the JVM process.

**Why existing checks fail:**

- The outer `contractSlotsCache` bound (`maximumSize=3000`) only limits how many `CaffeineCache` objects are *referenced by the outer cache at once*. It does not limit how many are registered in `cacheManagerSlotsPerContract.cacheMap`. [5](#0-4) 

- The `slotsPerContract` Caffeine spec (`maximumSize=1500`) limits entries *within* each per-contract cache, not the number of per-contract cache instances. [6](#0-5) 

- Rate limiting (500 RPS global, gas-based throttle) slows accumulation but does not prevent it. At 500 RPS sustained over hours, tens of thousands to millions of distinct contractIds can be registered. [7](#0-6) 

### Impact Explanation

Each `CaffeineCache` instance backed by a Caffeine cache with `maximumSize=1500` carries non-trivial heap overhead (internal node arrays, write buffers, scheduler references — conservatively 2–20 KB per instance even when empty). With N distinct contractIds accumulated in `cacheMap`:

- N = 100,000 → ~200 MB–2 GB heap consumed by `cacheMap` alone
- N = 1,000,000 → potential multi-GB heap exhaustion → `OutOfMemoryError`

An OOM in the web3 service kills the JVM process, taking down the mirror node's EVM simulation endpoint entirely. This prevents all `eth_call` and `eth_estimateGas` processing, which are the primary interfaces used by wallets and dApps to interact with Hedera smart contracts. The mirror node's gossip and ingestion pipeline runs in a separate process, but the web3 API becomes completely unavailable.

### Likelihood Explanation

The attack requires no authentication — the `/api/v1/contracts/call` endpoint is public: [8](#0-7) 

The attacker needs to trigger `findStorage` for many distinct contractIds. This is achievable by:
1. Calling many different existing contracts on the network (Hedera mainnet has hundreds of thousands of deployed contracts).
2. Deploying a single "fan-out" contract that uses STATICCALL to read storage from many other contracts in one transaction — multiplying the number of distinct contractIds registered per `eth_call` request.

The leak is permanent and cumulative across restarts only if the JVM is long-running (no restart). Even with restarts, the attacker can re-trigger accumulation. The rate limit of 500 RPS means ~43 million requests per day are possible from a single IP (or more with distributed sources), easily registering millions of distinct contractIds over hours.

### Recommendation

Fix the `cacheManagerSlotsPerContract` bean to use a bounded, self-managing structure instead of relying on `CaffeineCacheManager`'s dynamic mode:

1. **Remove `CaffeineCacheManager` entirely for this use case.** Replace it with a direct `com.github.benmanes.caffeine.cache.Cache<EntityId, CaffeineCache>` with `maximumSize` and `expireAfterAccess` configured, so eviction of the outer entry also releases the inner `CaffeineCache` with no secondary registry.

2. **If `CaffeineCacheManager` must be kept**, call `setCacheNames(Set.of())` (empty set with `setAllowNullValues` or similar) is not sufficient — instead, pre-register a fixed set of names, or switch to a `LoadingCache<EntityId, Cache>` pattern that does not use `CaffeineCacheManager` as a registry at all.

3. **Alternatively**, add a `RemovalListener` on the outer `contractSlotsCache` that calls `cacheManagerSlotsPerContract`'s internal map cleanup (via reflection or a custom wrapper) when an entry is evicted, ensuring the `CaffeineCache` instance is dereferenced from both structures simultaneously.

```
# Step 1: Identify or deploy N contracts on the target network that execute SLOAD.
# Step 2: Send eth_call requests cycling through all N contract addresses:

for i in $(seq 1 1000000); do
  CONTRACT_ADDR=$(printf "0x%040x" $i)  # synthetic addresses mapping to entity IDs
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT_ADDR\",\"data\":\"0x\",\"gas\":100000}" &
  # Respect rate limit: batch ~500/s
  if (( i % 500 == 0 )); then sleep 1; fi
done

# Step 3: Monitor JVM heap via JMX or /actuator/metrics.
# Observe: heap used by cacheManagerSlotsPerContract.cacheMap grows linearly with i.
# At sufficient i, OutOfMemoryError is thrown, web3 service crashes.
```

Each request to a distinct contract address that triggers an SLOAD during EVM execution registers one new permanent `CaffeineCache` entry in `cacheManagerSlotsPerContract`'s internal map. The outer `contractSlotsCache` eviction at 3000 entries does not reclaim these registrations. [9](#0-8) [1](#0-0)
