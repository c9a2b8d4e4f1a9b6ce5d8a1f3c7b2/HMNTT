### Title
Unbounded Prometheus Label Cardinality via Unsanitized HTTP Method in `MetricsMiddleware`

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates instrumentation to `github.com/weaveworks/common/middleware.Instrument`, which uses `r.Method` verbatim as the `method` label value across all four Prometheus metrics. Go's `net/http` server accepts any RFC 7230-compliant token as an HTTP method (not just standard verbs), so an unauthenticated attacker can flood the server with requests bearing unique arbitrary method strings, creating an unbounded number of Prometheus time series and exhausting process memory.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

All four metrics declare `"method"` as a label dimension (lines 26, 32, 37, 43). The weaveworks `Instrument.Wrap()` implementation reads `r.Method` directly from the incoming `*http.Request` and passes it as the label value — there is no allowlist, truncation, or normalization applied anywhere in this code path.

**Middleware chain** (`main.go`, lines 217–219):
```
corsMiddleware → tracingMiddleware → metricsMiddleware → router
```
`MetricsMiddleware` wraps the router, so it instruments every request **before** the router can reject it with 405/404. The CORS middleware (`server.CorsMiddleware`) does not restrict HTTP methods.

**Root cause:** Go's `net/http` server accepts any valid RFC 7230 token as an HTTP method — this includes arbitrary uppercase strings of any length (e.g., `METHODA`, `METHODB`, `XYZABC123`). The server does not restrict methods to the standard set (GET, POST, etc.). Because `r.Method` is used as a raw Prometheus label value with no sanitization, each unique method string permanently allocates a new label combination in all four `*Vec` metrics. Each histogram label combination also allocates `len(buckets)+2` counters. With `sizeBuckets` having 5 entries and `requestDurationHistogram` having 6, each unique method × route pair allocates dozens of in-memory objects that are never freed.

**Why existing checks fail:**
- No HTTP method allowlist anywhere in the middleware stack
- No Prometheus cardinality limit or label value validation
- No `MaxHeaderBytes` override on the `http.Server` (defaults to 1 MB — sufficient for thousands of unique method tokens per connection)
- No rate limiting configured on the server
- The `route` label is bounded (fixed Rosetta API routes), so the only unbounded dimension is `method`

### Impact Explanation
An attacker sending N unique method strings creates N × R new time series (where R = number of matched/unmatched routes), each backed by multiple in-memory counter/histogram objects across all four registered metrics. At scale (tens of thousands of unique methods), this exhausts heap memory, triggering OOM termination of the Rosetta process. This makes the node unable to serve any transaction history queries (`/block`, `/block/transaction`, `/account/balance`, etc.), directly disrupting the ordering and availability of transaction history responses. Recovery requires a process restart, but the attack is trivially repeatable.

### Likelihood Explanation
The attack requires zero authentication and zero privileges — any host with TCP access to the Rosetta port can execute it. The exploit is a simple loop sending HTTP requests with incrementing method tokens (all valid RFC 7230 tokens, accepted by Go's parser). It is fully automatable, repeatable, and persistent (each unique method permanently occupies memory until restart). The Rosetta port is intended to be reachable by exchange integrators, making the attack surface externally exposed by design.

### Recommendation
1. **Normalize the method label** before it reaches Prometheus. In `MetricsMiddleware`, wrap the handler to replace `r.Method` with an allowlisted value:
   ```go
   func sanitizeMethod(m string) string {
       switch strings.ToUpper(m) {
       case "GET", "POST", "PUT", "DELETE", "PATCH", "HEAD", "OPTIONS":
           return strings.ToUpper(m)
       default:
           return "unknown"
       }
   }
   ```
   Apply this before constructing the `middleware.Instrument` (e.g., by wrapping `next` to rewrite `r.Method` on the cloned request, or by subclassing the label extraction).

2. **Alternatively**, replace `middleware.Instrument` with a custom handler that explicitly controls which label values are recorded, using the sanitized method above.

3. **Add a rate limiter** (e.g., `golang.org/x/time/rate`) at the server entry point to bound the rate of unique connections from a single IP.

### Proof of Concept
```bash
#!/bin/bash
# Sends requests with unique method tokens to exhaust Prometheus label cardinality
ROSETTA_HOST="http://localhost:5700"
for i in $(seq 1 50000); do
    METHOD="METHOD$(printf '%06d' $i)"
    curl -s -o /dev/null -X "$METHOD" "$ROSETTA_HOST/network/list" \
         -H "Content-Type: application/json" \
         -d '{"metadata":{}}' &
    # Throttle to avoid local fd exhaustion; still fast enough to OOM target
    if (( i % 100 == 0 )); then wait; fi
done
wait
echo "Done. Check Rosetta process RSS / OOM status."
```
Each iteration registers a new `method` label value in all four `*Vec` metrics. After tens of thousands of iterations, the Rosetta process RSS grows unboundedly and is eventually OOM-killed, making all transaction history endpoints unavailable.
