### Title
Stale File Data Returned for `function_parameters` When Querying Contract Results by Transaction Hash

### Summary
In `getContractResultsByTransactionIdOrHash()`, when an Ethereum transaction references call data stored in a Hedera file (`callDataId`), the mirror node fetches that file's contents using `FileDataService.getLatestFileDataContents()` with an empty `whereQuery: []`. This returns the **current/latest** version of the file rather than the version that existed at transaction execution time. Any unprivileged user querying a historical transaction hash will receive incorrect `function_parameters` if the referenced file has been updated since execution.

### Finding Description

**Exact code location** — `rest/controllers/contractController.js`, `getContractResultsByTransactionIdOrHash()`, lines 1183–1185:

```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

**Root cause** — `getLatestFileDataContents` in `rest/service/fileDataService.js` (lines 84–87) passes the `whereQuery` array directly into `buildWhereSqlStatement`. With an empty array, no timestamp constraint is injected into either the inner CTE (`latest_create`) or the outer join in `latestFileContentsQuery` (lines 18–41). The CTE selects `max(consensus_timestamp)` across **all** file create/update records for that `entity_id`, unconditionally returning the most recent file version:

```sql
with latest_create as (
  select max(consensus_timestamp) as consensus_timestamp
  from file_data
  where entity_id = $1 and transaction_type in (17, 19)
  -- no timestamp upper-bound here
  ...
)
```

**Failed assumption** — The code assumes the file contents at `callDataId` are immutable after the transaction executes. Hedera files are mutable; any holder of the file's admin key can issue `FILEUPDATE` or `FILEAPPEND` transactions at any time.

**Contrast with correct pattern** — `getContractById()` (line 732) correctly pins the file to the creation timestamp:
```js
contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
```
The `getFileData` query uses `consensus_timestamp <= $2` to bound the lookup. The same bounding is demonstrated in `fileDataService.test.js` lines 109–116 for `getLatestFileDataContents` itself, showing the intended usage with a `consensus_timestamp <=` filter.

**Exploit flow:**
1. Ethereum transaction `T` is submitted at consensus timestamp `t1`, with `callDataId` pointing to file `F` containing call data `D1`.
2. The file owner later updates `F` to `D2` at timestamp `t2 > t1`.
3. An unprivileged user queries `GET /api/v1/contracts/results/{txHash}` for transaction `T`.
4. The mirror node calls `getLatestFileDataContents(F, {whereQuery: []})`, which resolves to `D2`.
5. The response's `function_parameters` field contains `D2` instead of the historically correct `D1`.

The same defect exists identically in `getContractResultsByTimestamp()` at lines 1022–1024.

### Impact Explanation
The mirror node's primary purpose is to serve as an accurate, immutable historical record of network activity. Returning post-execution file data as `function_parameters` corrupts the historical record of what call data was actually submitted to the EVM. This affects:
- Audit trails and forensic analysis of contract calls
- Any downstream system (block explorers, compliance tools, indexers) that relies on the mirror node's `function_parameters` field to reconstruct transaction inputs
- Verification of contract behavior, since the displayed inputs will not match the actual execution inputs

### Likelihood Explanation
The precondition — a file update occurring after a transaction that referenced it — is a normal, expected operation on Hedera. Files used as call data containers are explicitly designed to be updated (e.g., for large contract deployments). No special privilege is required to trigger the incorrect export; any user who knows a transaction hash can issue the query. The defect is deterministic and repeatable for any affected transaction.

### Recommendation
Pass a timestamp upper-bound equal to `ethTransaction.consensusTimestamp` when calling `getLatestFileDataContents`, mirroring the pattern already used in `getFileData`:

```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(
    ethTransaction.callDataId,
    {
      whereQuery: [{
        query: `${FileData.CONSENSUS_TIMESTAMP} <= `,
        param: ethTransaction.consensusTimestamp,
      }],
    }
  );
}
```

Apply the same fix to `getContractResultsByTimestamp()` (lines 1022–1024), using the same `ethTransaction.consensusTimestamp` bound. This ensures the file version returned is the one that was current at execution time, not the current latest version.

### Proof of Concept
1. Create a Hedera file `F` with content `0xAAAA` (call data version 1).
2. Submit an Ethereum transaction `T` referencing `F` as call data (`callDataId = F`). Record the transaction hash `H` and consensus timestamp `t1`.
3. Update file `F` to content `0xBBBB` at timestamp `t2 > t1`.
4. Query `GET /api/v1/contracts/results/{H}`.
5. Observe that `function_parameters` in the response is `0xBBBB` (current file content) rather than `0xAAAA` (content at execution time `t1`).
