### Title
Gas Throttle Effectively Bypassed for Minimum-Gas Requests via `scaleGas` Truncation

### Summary
The `scaleGas` function in `ThrottleProperties` divides the requested gas by `GAS_SCALE_FACTOR = 10_000`, meaning a request with `gas=21000` (the minimum allowed) costs only 2 gas-bucket tokens. The gas bucket has a default capacity of 750,000 scaled tokens per second, so minimum-gas requests consume 0.00027% of the gas budget per request, rendering the gas throttle completely ineffective for this class of requests. Only the flat rate limit (default 500 req/s) remains as protection.

### Finding Description

**Code path:**

`ContractCallRequest.java` enforces `@Min(21_000)` on the `gas` field: [1](#0-0) 

`ThrottleManagerImpl.throttle()` calls `scaleGas` to determine how many tokens to consume from the gas bucket: [2](#0-1) 

`ThrottleProperties.scaleGas()` performs integer floor-division by `GAS_SCALE_FACTOR = 10_000`: [3](#0-2) 

The gas bucket capacity is `scaleGas(7_500_000_000)` = 750,000 tokens/second: [4](#0-3) 

**Root cause:** `scaleGas(21000) = Math.floorDiv(21000, 10000) = 2`. At the default rate limit of 500 req/s, an attacker sending gas=21000 requests consumes `500 × 2 = 1,000` gas tokens/second against a 750,000 token/second bucket — 0.13% of capacity. The gas throttle never triggers. By contrast, a request with the default gas of 15,000,000 costs 1,500 tokens, and 500 such requests/second would exactly exhaust the gas bucket. The gas throttle is designed to be the proportional protection layer, but it provides zero additional protection for minimum-gas requests.

**Why existing checks fail:**
- The rate limit bucket (`rateLimitBucket`, default 500 req/s) still applies, but it is a flat per-request limit with no awareness of EVM execution cost.
- The gas bucket check (`gasLimitBucket.tryConsume(scaleGas(21000))`) passes trivially since 2 << 750,000.
- `validateContractMaxGasLimit` only enforces an upper bound, not a meaningful lower bound relative to throttle cost. [5](#0-4) 

### Impact Explanation
An unprivileged attacker can sustain 500 req/s indefinitely — the maximum rate-limited throughput — while the gas throttle never engages. Each request still triggers the full EVM context initialization, state loading, and execution pipeline. The fixed per-request overhead (EVM setup, DB lookups, context allocation) is the same regardless of gas value. This maximizes the ratio of server resource consumption to gas-throttle token cost by a factor of ~750x compared to a default-gas request (2 tokens vs. 1,500 tokens per request). The gas throttle, intended as the primary defense against resource exhaustion, is rendered inert for this entire class of requests.

### Likelihood Explanation
No privileges, authentication, or special knowledge are required. Any external user can craft a valid `POST /api/v1/contracts/call` request with `gas=21000`. The attack is trivially repeatable with standard HTTP tooling (e.g., `wrk`, `ab`, `curl` in a loop). The attacker does not need to own any contract or account. The condition is stable across deployments using default configuration.

### Recommendation
1. **Enforce a minimum scaled token cost**: In `scaleGas`, ensure the return value is at least 1 for any gas value above the zero-cost threshold, or raise the zero-cost threshold to below the minimum allowed gas (i.e., set `GAS_SCALE_FACTOR` such that `scaleGas(21000) >= some_meaningful_minimum`).
2. **Alternatively, raise `@Min` on `gas`** in `ContractCallRequest` to a value that produces a meaningful token cost, e.g., `@Min(100_000)` so `scaleGas(100_000) = 10` tokens.
3. **Add a per-IP or per-source rate limit** as a defense-in-depth layer independent of gas value, so the flat 500 req/s limit is not the sole protection for low-gas requests.

### Proof of Concept

```bash
# Send 500 requests/second with minimum gas=21000
# Each request costs only 2 gas tokens (out of 750,000/s capacity)
# Gas throttle never triggers; only rate limit applies

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to": "0x00000000000000000000000000000000000004e2",
      "gas": 21000,
      "data": "0x"
    }' &
done
wait

# Expected: All 500 requests succeed (HTTP 200) in the same second.
# Gas bucket consumed: 500 * 2 = 1,000 tokens out of 750,000 capacity (0.13%).
# Gas throttle (GAS_PER_SECOND_LIMIT_EXCEEDED) is never raised.
# Only the rate limit bucket (500 req/s) is the effective guard.
```
