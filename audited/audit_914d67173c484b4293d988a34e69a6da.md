### Title
Per-Request DB Query Amplification via EXTCODESIZE/EXTCODECOPY on Unique Addresses in `findRuntimeBytecode()`

### Summary
An unprivileged attacker can send a single `eth_call` request targeting a contract that iterates EXTCODESIZE or EXTCODECOPY over many unique addresses, causing `findRuntimeBytecode()` to issue one DB query per unique address per request. Because the Spring Caffeine cache is bounded at `maximumSize=1000` and the attacker can trivially supply fresh addresses (from the 2^160 address space), the cache provides no protection across requests using novel addresses. A single request with 15M gas can trigger ~5,769 DB queries, creating a severe amplification ratio.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ContractExecutionService.processCall()` → EVM execution → EXTCODESIZE/EXTCODECOPY opcode → `ContractBytecodeReadableKVState.readFromDataSource()` → `contractRepository.findRuntimeBytecode(entityId.getId())` [1](#0-0) 

The `@Cacheable` annotation uses `unless = "#result == null"`. The method return type is `Optional<byte[]>`. When a contract does not exist, the repository returns `Optional.empty()`, which is **not null**, so it **is** cached. However, the Caffeine cache is configured with `maximumSize=1000`: [2](#0-1) 

The `ReadableKVStateBase.get()` method maintains a per-request in-memory cache via `ContractCallContext`: [3](#0-2) 

This per-request cache only deduplicates lookups of the **same** address within one request. For N **distinct** addresses, each triggers one call to `readFromDataSource()` → `findRuntimeBytecode()` → Spring cache check → DB query on cache miss. [4](#0-3) 

**Root cause:** There is no per-request cap on the number of unique contract addresses that can be resolved via `findRuntimeBytecode()`. The Spring cache is a cross-request LRU cache (max 1000 entries, 1h TTL), not a per-request budget. An attacker supplying >1000 unique addresses per request, or rotating through fresh addresses across requests, bypasses the cache entirely.

**Failed assumption:** The design assumes the Caffeine cache will absorb repeated lookups. It does not account for an attacker deliberately using novel addresses to force cache misses on every call.

### Impact Explanation

With `maxGasLimit = 15,000,000` and EXTCODESIZE costing 2,600 gas (EIP-2929 cold access), a single request can reference ~5,769 unique addresses, each triggering one `SELECT runtime_bytecode FROM contract WHERE id = ?` query. The gas throttle (`gasPerSecond = 7,500,000,000`) allows up to ~500 such requests per second, yielding up to ~2.88 million DB queries per second from a single attacker. This exhausts the DB connection pool and degrades or halts service for all users. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

No authentication or special privilege is required. The attacker only needs to deploy (or reference an existing) contract that loops over a large array of unique addresses calling EXTCODESIZE. The attack is fully repeatable, scriptable, and can be sustained indefinitely. The 2^160 EVM address space ensures the attacker never runs out of fresh cache-busting addresses. [7](#0-6) 

### Recommendation

1. **Add a per-request DB query budget**: Track the number of `findRuntimeBytecode()` calls within a single `ContractCallContext` and abort execution (or return empty) once a configurable threshold (e.g., 100) is exceeded.
2. **Cache negative results more aggressively**: The current `unless = "#result == null"` already caches `Optional.empty()`, but the `maximumSize=1000` is too small. Increase it significantly (e.g., 50,000) to absorb more unique addresses.
3. **Reduce `maxGasLimit`**: Lowering the per-request gas ceiling directly reduces the maximum amplification factor per request.
4. **Apply per-IP or per-sender rate limiting** at the HTTP layer in addition to the global gas bucket.

### Proof of Concept

1. Deploy a contract on the network with the following logic:
   ```solidity
   function amplify(address[] calldata targets) external view {
       for (uint i = 0; i < targets.length; i++) {
           uint sz;
           assembly { sz := extcodesize(targets[i]) }
       }
   }
   ```
2. Generate 5,000+ unique random EVM addresses not previously queried.
3. Send `eth_call` to `amplify(targets)` with `gas=15000000`.
4. Observe via DB metrics that ~5,000 `SELECT runtime_bytecode FROM contract WHERE id = ?` queries are issued for this single HTTP request.
5. Repeat in a loop with fresh address batches; observe DB connection pool saturation and increasing response latency for all concurrent users.
