### Title
Unbounded Goroutine Accumulation via Unauthenticated `/network/status` Endpoint Enables Memory-Exhaustion DoS

### Summary
The `NetworkStatus()` handler in `rosetta/app/services/network_service.go` executes three sequential database queries per request with no server-side concurrency limit or rate limiting. Because Go's `net/http` spawns one goroutine per connection and the database connection pool (`MaxOpenConnections`) causes goroutines to block waiting for a DB slot rather than fail fast, a single unauthenticated attacker flooding `/network/status` can accumulate thousands of live goroutines, exhausting process memory and crashing the server.

### Finding Description
**Code path:**
- `rosetta/app/services/network_service.go`, `NetworkStatus()`, lines 59–88: three blocking DB calls — `n.RetrieveGenesis(ctx)` (line 67), `n.RetrieveLatest(ctx)` (line 72), `n.addressBookEntryRepo.Entries(ctx)` (line 77) — are made sequentially with no guard.
- `rosetta/main.go`, lines 217–227: the middleware chain is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. None of these impose a concurrency cap or per-IP rate limit.
- `rosetta/main.go`, lines 220–227: `http.Server` is configured with `ReadTimeout`, `WriteTimeout`, `IdleTimeout`, and `ReadHeaderTimeout` only — no `MaxConns` or equivalent.
- `rosetta/app/config/types.go`, lines 77–81: the DB pool exposes `MaxOpenConnections`, which caps DB connections but causes excess goroutines to **block and queue** rather than return an error immediately.

**Root cause:** Go's `net/http` allocates one goroutine per accepted connection. With no concurrency limiter, an attacker opening N simultaneous connections creates N goroutines. Each goroutine blocks on DB pool acquisition (because `MaxOpenConnections` is finite), so goroutines accumulate in memory rather than completing quickly. There is no mechanism to reject or shed load.

**Why existing checks fail:**
- HTTP timeouts (`WriteTimeout` etc.) only reclaim a goroutine after the timeout elapses; an attacker sending requests faster than the timeout interval keeps the goroutine count growing.
- `MaxOpenConnections` does not shed load — it queues callers, worsening goroutine accumulation.
- No authentication, API key, or IP-based rate limiting exists anywhere in the middleware stack.

### Impact Explanation
Each blocked goroutine holds its stack (starting ~8 KB, growing under load) plus any heap allocations from the DB driver. At 50,000 concurrent connections the resident memory overhead is measurable in gigabytes. When the process OOMs, the entire Rosetta node crashes, severing all clients and halting block/transaction queries. Because the Rosetta node is the sole interface for Coinbase-ecosystem integrations, this constitutes a complete availability failure for the affected network participant. The scope qualifier ("≥25% market cap layer") makes this a critical-severity finding under the stated scope rules.

### Likelihood Explanation
No privileges, accounts, or special knowledge are required — the `/network/status` endpoint is publicly documented in the Rosetta API specification and must be reachable by any Rosetta client. A single commodity machine can sustain tens of thousands of concurrent HTTP connections. The attack is trivially repeatable: after a crash and restart the attacker simply resumes flooding. No exploit tooling beyond `wrk`, `ab`, or a short script is needed.

### Recommendation
1. **Add a concurrency-limiting middleware** wrapping the router (e.g., a semaphore of size `N` matching the DB pool's `MaxOpenConnections`); return HTTP 503 when the semaphore is full.
2. **Add per-IP rate limiting** (e.g., `golang.org/x/time/rate` token bucket) in a middleware layer inserted between `CorsMiddleware` and the router in `main.go`.
3. **Set `http.Server.MaxConns`** (available via `netutil.LimitListener`) to cap accepted connections at the OS level before goroutines are spawned.
4. **Return an immediate error** from `NetworkStatus()` (or the DB repository layer) when the pool is exhausted rather than blocking indefinitely, so goroutines drain quickly.

### Proof of Concept
```bash
# Prerequisites: network access to the Rosetta port (default 5700), no credentials needed.

# 1. Confirm the endpoint is live
curl -s -X POST http://<target>:5700/network/list \
  -H 'Content-Type: application/json' -d '{}'

# 2. Flood /network/status with 10,000 concurrent requests
#    (wrk2 or ab; adjust -c for desired concurrency)
wrk -t 8 -c 10000 -d 60s \
  -s <(echo 'wrk.method="POST"
              wrk.headers["Content-Type"]="application/json"
              wrk.body='"'"'{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'"'"'') \
  http://<target>:5700/network/status

# 3. Observe on the server:
#    - `ps aux` shows rosetta RSS growing rapidly
#    - `curl .../health/liveness` starts timing out
#    - Process eventually OOM-killed or becomes unresponsive
#    - All legitimate Rosetta clients receive connection refused / timeout
```
