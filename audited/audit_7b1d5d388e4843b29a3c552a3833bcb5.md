### Title
Unprivileged User Can Bypass `transaction_nonce = 0` Filter via `internal=true` Query Parameter in `getContractResultsById`

### Summary
The `/api/v1/contracts/:contractId/results` endpoint accepts an `internal` query parameter that, when set to `true`, removes the `transaction_nonce = 0` SQL filter. This filter is the sole mechanism preventing unprivileged users from seeing internal (child) contract transactions. No authentication or authorization check gates the `internal` parameter, so any external caller can supply `?internal=true` to expose internal transactions that are intended to be hidden.

### Finding Description

**Route registration** (`rest/routes/contractRoute.js`, line 14):
```
router.getExt('/:contractId/results', ContractController.getContractResultsById);
``` [1](#0-0) 

**Handler** (`rest/controllers/contractController.js`, lines 856–860) calls `extractContractIdAndFiltersFromValidatedRequest`, which calls `utils.buildAndValidateFilters(req.query, acceptedContractResultsParameters, contractResultsFilterValidityChecks)` — no privilege check is performed before or after: [2](#0-1) 

**Filter parsing** (`extractContractResultsByIdQuery`, lines 478–480): the `internal` key from the user-supplied query string is read directly into the local `internal` variable: [3](#0-2) 

**Guard condition** (lines 531–533): the `transaction_nonce = 0` SQL condition is only appended when `internal` is falsy. If the user passes `internal=true`, this branch is skipped entirely: [4](#0-3) 

**Validity check** (`contractResultsFilterValidityChecks`, lines 273–279): only adds a special restriction for `BLOCK_NUMBER`; the `INTERNAL` key passes through `utils.filterValidityChecks` with no privilege enforcement: [5](#0-4) 

Root cause: the `internal` flag is treated as a plain user-controlled boolean filter with no role/privilege gate. The `transaction_nonce = 0` filter is the only data-layer boundary between public and internal transactions, and it is unconditionally removed when `internal=true`.

### Impact Explanation
Internal contract transactions (those with `transaction_nonce > 0`) represent child calls spawned by a parent EVM transaction. They are intentionally hidden from the public API because they may expose intermediate state, internal accounting, or sub-call details that operators consider non-public. An attacker can enumerate all internal transactions for any contract by simply appending `?internal=true`, bypassing the intended access boundary. Severity is **Medium–High**: data confidentiality is broken for all contracts on the node; no write access or funds are at risk, but sensitive execution details are exposed.

### Likelihood Explanation
Exploitation requires zero privileges — only the ability to send an HTTP GET request. The parameter name (`internal`) is self-documenting and trivially discoverable via API documentation, source code, or simple fuzzing. It is repeatable and automatable against any contract ID. Any external user of the mirror node REST API can exploit this immediately.

### Recommendation
1. **Remove `INTERNAL` from `acceptedContractResultsParameters`** so `buildAndValidateFilters` rejects it as an unknown parameter for the public endpoint, returning a 400 error.
2. If the `internal=true` capability is intentionally needed for internal/operator use, gate it behind an authenticated/authorized middleware layer before `extractContractIdAndFiltersFromValidatedRequest` is called, and reject the parameter for unauthenticated requests.
3. Add a test asserting that `GET /api/v1/contracts/:id/results?internal=true` from an unprivileged caller returns only `transaction_nonce = 0` rows.

### Proof of Concept

**Without the bypass (default behavior — only top-level results):**
```
GET /api/v1/contracts/0.0.1234/results
```
SQL includes: `WHERE cr.contract_id = $1 AND cr.transaction_nonce = 0`

**With the bypass (internal transactions exposed):**
```
GET /api/v1/contracts/0.0.1234/results?internal=true
```
SQL becomes: `WHERE cr.contract_id = $1`  ← `transaction_nonce = 0` filter is gone

All rows with `transaction_nonce > 0` (internal child calls) for contract `0.0.1234` are returned to the unauthenticated caller.
