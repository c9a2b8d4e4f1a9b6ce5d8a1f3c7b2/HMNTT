### Title
Unauthenticated EVM Address + Timestamp Parameter Combination Causes 3x DB Query Amplification in `/api/v1/balances`

### Summary
Any unauthenticated external user can craft a request to `/api/v1/balances` with `account.id=eq:<valid-40-hex-evm-address>&timestamp=lte:<value>` that unconditionally triggers three sequential database round-trips per request: an entity table lookup to resolve the EVM address, an `account_balance` timestamp-range lookup, and the final balance query. A plain numeric entity-ID request without a timestamp filter requires only one DB query, making this combination a 3x amplification factor exploitable at no cost by any caller.

### Finding Description

**Exact code path:**

In `rest/balances.js`, `getBalances()` (line 83) calls `parseAccountIdQueryParam()` (line 87). Inside `parseAccountIdQueryParam` (lines 320–357), when the supplied value passes `EntityId.isValidEvmAddress(value, EvmAddressType.NO_SHARD_REALM)` (line 328), `EntityService.getEncodedId(value, false)` is returned as a Promise (line 329). That Promise is immediately awaited at line 88 via `Promise.all(accountParamsPromise)`.

`EntityService.getEncodedId()` in `rest/service/entityService.js` (lines 118–137) calls `getEntityIdFromEvmAddress()` (line 124), which unconditionally executes:

```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```
(line 91 of `entityService.js`) — **DB Query 1**.

Back in `getBalances()`, because `tsQuery` is non-empty (line 113), `getTsQuery()` is awaited (line 114), which calls `getAccountBalanceTimestampRange()` (lines 167–197 of `balances.js`), executing:

```sql
SELECT consensus_timestamp FROM account_balance
WHERE account_id = $1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3
ORDER BY consensus_timestamp DESC LIMIT 1
```
— **DB Query 2**.

Finally, the main balance query is executed at line 153 — **DB Query 3**.

**Root cause:** There is no caching layer for EVM address → entity ID resolution. `requireResult=false` (line 329) means the DB lookup fires even when the address does not exist in the entity table. The timestamp branch adds two more sequential awaited DB calls. No application-level rate limiting exists in the REST service (`grep` across `rest/**/*.js` finds no `rateLimit`/`throttle` usage).

**Why existing checks are insufficient:** The only guard is `evmAliasAddressCount === 1` (line 328), which limits one EVM address per request but does nothing to prevent the amplification itself. Infrastructure-level rate limiting is present in the Rosetta chart (`charts/hedera-mirror-rosetta/values.yaml` lines 157–160) but is not confirmed for the REST API chart.

### Impact Explanation
Each crafted request consumes three DB connections/round-trips instead of one. An attacker flooding the endpoint at modest request rates (e.g., 1,000 req/s) generates 3,000 DB queries/s. Because the EVM address need not exist, the attacker requires no on-chain state. This degrades database throughput for all legitimate users of the mirror node, constituting a griefing/DoS with no economic cost to the attacker and no economic damage to any individual account — consistent with the stated scope of "griefing with no economic damage."

### Likelihood Explanation
The exploit requires zero privileges, zero on-chain assets, and only knowledge of the public REST API. Any valid 40-hex-character string (e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`) satisfies `isValidEvmAddress`. The attack is trivially scriptable with `curl` or any HTTP client in a loop. It is repeatable indefinitely and requires no special tooling.

### Recommendation
1. **Cache EVM address → entity ID resolutions** (e.g., an in-process LRU cache with a short TTL) so repeated lookups for the same address do not hit the DB.
2. **Apply application-level rate limiting** on the `/api/v1/balances` endpoint, keyed by source IP, consistent with what is already deployed for the Rosetta service.
3. **Consider short-circuiting** the entity lookup when `requireResult=false` returns `null` (address not found) before proceeding to the timestamp and balance queries — if the account does not exist, the balance result will be empty regardless.
4. **Parallelize** the entity lookup and timestamp range lookup where possible to reduce wall-clock latency even if total DB work remains the same.

### Proof of Concept

```bash
# Single amplified request (triggers 3 DB queries)
curl "https://<mirror-node-host>/api/v1/balances?account.id=eq:deadbeefdeadbeefdeadbeefdeadbeefdeadbeef&timestamp=lte:9999999999.000000000"

# Flood to exhaust DB connection pool (no auth required)
for i in $(seq 1 10000); do
  # rotate last 8 hex chars to avoid any caching on address
  ADDR=$(printf "deadbeefdeadbeefdeadbeefdeadbeef%08x" $i)
  curl -s "https://<mirror-node-host>/api/v1/balances?account.id=eq:${ADDR}&timestamp=lte:9999999999.000000000" &
done
wait
```

**Preconditions:** None. No account, no token, no authentication.
**Trigger:** Valid 40-hex-char EVM address in `account.id=eq:` combined with any `timestamp=lte:` value.
