### Title
Duplicate `limit` Parameter Manipulation Causes Last-Value-Wins SQL LIMIT Override in Topic Messages Pagination

### Summary
In `rest/topicmessage.js`, the `extractSqlFromTopicMessagesRequest()` function iterates over all parsed filters and blindly overwrites `limit` for every `LIMIT`-keyed filter it encounters, so the last value wins. Because `utils.buildFilters()` expands an array query parameter (e.g., `limit=100&limit=1`) into two separate filter objects, an unauthenticated caller can force the effective SQL `LIMIT` to `1` while the server still generates a `next` pagination link after every single message, causing clients that stop following `next` links prematurely to silently miss records.

### Finding Description

**Filter expansion — `rest/utils.js` lines 1240–1253**

`buildFilters` iterates over `Object.entries(query)`. When Express parses `?limit=100&limit=1`, it sets `req.query.limit = ['100', '1']`. The array branch pushes a separate `buildComparatorFilter` result for each element:

```js
for (const val of values) {
  filters.push(buildComparatorFilter(key, val));   // two LIMIT entries
}
```

After `validateAndParseFilters` / `formatComparator` runs, the filters array contains `{key:'limit', value:100}` and `{key:'limit', value:1}` (both valid, both within `maxRepeatedQueryParameters`).

**Last-value-wins overwrite — `rest/topicmessage.js` lines 177–181**

```js
for (const filter of filters) {
  switch (filter.key) {
    case constants.filterKeys.LIMIT:
      limit = filter.value;   // overwritten on every LIMIT filter; last wins
      break;
```

With the two-element array above, `limit` ends up as `1`.

**SQL and pagination both use the same `limit` — lines 215 and 154**

```js
limit $${params.push(limit)}          // SQL LIMIT = 1
...
topicMessagesResponse.messages.length !== limit   // 1 !== 1 → false → isEnd=false
```

`getPaginationLink` (`utils.js` line 806) returns `null` only when `isEnd` is `true`. Because `isEnd` is `false`, a `next` link is emitted after every single message.

**`next` link perpetuates the attack — `utils.js` lines 865–818**

`getNextParamQueries` rebuilds the query string from `req.query`, which still contains `limit: ['100','1']`, so every subsequent page also uses `limit=1`.

### Impact Explanation

Any client that caps the number of pagination pages it follows (a common defensive pattern) will silently stop after N messages instead of the expected `N×100`. Because the `next` link is always present, the client has no way to distinguish "last page" from "more data exists." This constitutes silent record loss for consumers of the mirror-node REST API. Additionally, forcing `limit=1` multiplies the number of round-trips by up to 100×, amplifying server-side query load proportionally.

### Likelihood Explanation

No authentication or special privilege is required. The attack is a single URL parameter change (`?limit=100&limit=1`) that any external caller can supply. It is trivially repeatable and requires no knowledge of internal state. The `maxRepeatedQueryParameters` guard (default ≥ 4) explicitly permits two repeated `limit` values, so no existing check blocks this.

### Recommendation

In `extractSqlFromTopicMessagesRequest`, reject or ignore duplicate `LIMIT` filters rather than silently overwriting. The simplest fix is to break after the first `LIMIT` filter is seen, or to reject the request with a 400 error when more than one `LIMIT` filter is present. Alternatively, `buildAndValidateFilters` should enforce that `LIMIT` and `ORDER` are singleton parameters (count ≤ 1) before the filters array is returned to callers.

### Proof of Concept

```
GET /api/v1/topics/7/messages?limit=100&limit=1
```

1. Express parses `req.query.limit = ['100', '1']`.
2. `buildFilters` emits two filter objects: `{key:'limit', value:100}` and `{key:'limit', value:1}`.
3. `extractSqlFromTopicMessagesRequest` loop sets `limit = 100`, then `limit = 1`.
4. SQL executes `… LIMIT $N` with `$N = 1`; one row is returned.
5. `messages.length (1) !== limit (1)` → `false` → `getPaginationLink` emits a `next` link.
6. The `next` link contains `limit=100&limit=1`, so every subsequent page also returns exactly 1 message with a `next` link.
7. A client that stops after, say, 10 pages receives 10 messages instead of the up-to-1000 it expected. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
