### Title
Synchronous `gzipSync` on Node.js Event Loop via Cache-Miss Requests Enables Event Loop Starvation

### Summary
In `responseCacheUpdateHandler()`, every cache-miss response that meets the compression threshold triggers `new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody))`, which calls `gzipSync(body).toString('base64url')` synchronously on the Node.js event loop. Because Node.js is single-threaded, this blocks all other in-flight requests for the duration of the compression. An unprivileged attacker can continuously force cache misses by crafting requests with unique query parameters, sustaining event loop starvation and increasing perceived latency and CPU consumption well above 30%.

### Finding Description
**Exact code path:**

`rest/server.js` lines 136–141 register `responseHandler` then `responseCacheUpdateHandler` as sequential middleware. `responseHandler` calls `res.send()` (line 51) then `next()` (line 60), which invokes `responseCacheUpdateHandler`.

Inside `responseCacheUpdateHandler` (`rest/middleware/responseCacheHandler.js`, line 115):
```js
const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
```

`shouldCompress` (`rest/middleware/responseCacheHandler.js`, line 137–139) returns `true` when `config.cache.response.compress` is `true` (default) and `body.length >= config.cache.response.compressThreshold` (default: **150 bytes**). This threshold is extremely low — virtually every non-trivial API response exceeds it.

Inside `CachedApiResponse` constructor (`rest/model/cachedApiResponse.js`, line 11):
```js
this.body = compress ? gzipSync(body).toString('base64url') : body;
```

`gzipSync` is the **synchronous** variant from Node.js `zlib`. It runs entirely on the main thread, blocking the event loop for the full duration of compression. For a maximum-size response (e.g., `limit=100` on a data-heavy endpoint like `/api/v1/contracts/results` or `/api/v1/transactions`), the JSON body can reach hundreds of KB, causing `gzipSync` to hold the event loop for 10–50 ms per call.

**Root cause:** The failed assumption is that post-send middleware is "safe" to run synchronously. While `res.send()` has already flushed the response to the client, the event loop is still blocked for all other concurrent requests during the `gzipSync` call.

**Cache-miss forcing:** The cache key is generated as `md5(req.originalUrl)` (line 152). An attacker can trivially generate unique cache keys by varying any query parameter (e.g., `timestamp=gte:X` with incrementing `X`, or `limit=N` with varying `N`). Each unique URL is a guaranteed cache miss, triggering a fresh `gzipSync` call.

### Impact Explanation
Node.js processes all JavaScript on a single thread. Each `gzipSync` call on a large response body (e.g., 200–500 KB) blocks the event loop for 10–50 ms. With a sustained stream of cache-miss requests (e.g., 20–50 concurrent requests/second with unique URLs), the event loop is occupied with compression work for a significant fraction of wall-clock time. Legitimate requests queue behind these blocking calls, experiencing latency increases proportional to the blocking duration. CPU utilization rises because the main thread is doing synchronous compression work instead of I/O multiplexing. This directly satisfies the >30% resource consumption increase threshold without requiring brute-force credential attacks.

### Likelihood Explanation
Any unauthenticated user can issue GET requests to public REST API endpoints. Generating unique cache-busting URLs requires no special knowledge — varying a timestamp or limit parameter suffices. The attack is repeatable, requires no authentication, and can be automated with a simple script. The default configuration (`compress: true`, `compressThreshold: 150`) means the vulnerable path is active in all standard deployments.

### Recommendation
Replace `gzipSync` with the asynchronous `gzip` (callback or promisified) from Node.js `zlib`, or use `zlib.gzipSync` only in a worker thread via `worker_threads`. Specifically, in `rest/model/cachedApiResponse.js`, the constructor should not perform synchronous compression; instead, compression should be performed asynchronously before constructing the object, e.g.:

```js
// In responseCacheUpdateHandler, before constructing CachedApiResponse:
const compressedBody = shouldCompress(responseBody)
  ? await gzip(responseBody)  // async, non-blocking
  : responseBody;
const cachedResponse = new CachedApiResponse(statusCode, headers, compressedBody, false);
```

Additionally, raise `compressThreshold` to a more meaningful value (e.g., 4096 bytes) to reduce the frequency of compression operations.

  curl -s "https://<mirror-node>/api/v1/transactions?limit=100&timestamp=gte:${i}.000000000&timestamp=lte:$((i+1)).000000000" &
done
wait

# Observe: CPU usage on the mirror-node REST process spikes >30% above baseline
