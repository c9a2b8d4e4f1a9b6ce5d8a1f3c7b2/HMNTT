### Title
Unbounded Sequential Pagination on `/allowances/tokens` Enables Sustained DB Resource Exhaustion Without Rate-Limit Enforcement

### Summary
The `getPaginationLink()` function in `rest/controllers/baseController.js` unconditionally generates a valid `next` cursor link whenever the result set equals the requested `limit`, with no application-level rate limiting applied to the REST API middleware stack. An unprivileged external user can repeatedly follow the generated `next` link at maximum `limit=100`, causing the server to execute up to three `UNION ALL` subqueries per request against the `token_allowance` table indefinitely, sustaining elevated DB and CPU load without triggering any brute-force detection.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:18` registers the handler: [1](#0-0) 

`rest/controllers/tokenAllowanceController.js:68-81` — `getAccountTokenAllowances` calls `getPaginationLink` on every request: [2](#0-1) 

`rest/controllers/baseController.js:195-223` — `getPaginationLink()` generates the next cursor: [3](#0-2) 

**Root cause:** At line 198, `isEnd` is only `true` when `rows.length < limit`. As long as a full page is returned, the function always emits a valid `next` URL with no throttle awareness: [4](#0-3) 

**DB amplification:** `rest/service/tokenAllowanceService.js:63-74` constructs up to three `UNION ALL` subqueries (lower/inner/upper), each with its own `LIMIT $2` scan, per single HTTP request: [5](#0-4) 

**No application-level rate limiting:** The entire REST middleware stack exported from `rest/middleware/index.js` contains no IP-based or per-client rate limiter: [6](#0-5) 

`rest/middleware/authHandler.js` only sets a custom response-size limit for authenticated users; it does not throttle or reject unauthenticated requests: [7](#0-6) 

The Traefik middleware YAML (`charts/hedera-mirror-rest/templates/middleware.yaml`) is optional infrastructure configuration — it is not enforced at the application layer and may not be deployed: [8](#0-7) 

The throttle configuration found in `web3/` applies only to the EVM/web3 module, not to the REST API: [9](#0-8) 

### Impact Explanation
Each page request at `limit=100` can execute up to 3 sequential table scans of `token_allowance` via `UNION ALL`. For an account with thousands of token allowances, an attacker can sustain a continuous stream of such requests. Because there is no application-level rate limiting, connection throttling, or per-IP request counting in the REST API, the attacker can drive DB CPU and I/O well above baseline (>30%) using only sequential, legitimate-looking pagination — indistinguishable from normal API usage. This can degrade service for all other users sharing the same database.

### Likelihood Explanation
No privileges are required. The endpoint is publicly accessible. The attacker needs only to know any valid account ID with a large number of token allowances (discoverable via the same API). The attack is trivially scriptable: fetch the first page, extract `links.next`, loop. Because each request is a valid, well-formed API call, it bypasses any brute-force heuristics based on error rates or repeated identical requests.

### Recommendation
1. **Add application-level rate limiting** to the REST API middleware stack (e.g., `express-rate-limit` keyed by IP), independent of Traefik.
2. **Cap the effective pagination depth** by tracking or limiting the number of sequential cursor-following requests per IP/session within a time window.
3. **Add a DB query timeout** (`statementTimeout` is already configurable in `rest/config.js` pool settings — ensure it is set to a low value for this endpoint).
4. **Consider requiring authentication** for high-`limit` requests or for accounts with large allowance sets.

### Proof of Concept
```bash
# Step 1: Initial request with maximum limit
URL="https://<mirror-node>/api/v1/accounts/0.0.1000/allowances/tokens?limit=100"

while true; do
  RESPONSE=$(curl -s "$URL")
  NEXT=$(echo "$RESPONSE" | jq -r '.links.next')
  
  if [ "$NEXT" = "null" ] || [ -z "$NEXT" ]; then
    break
  fi
  
  # Follow the next pagination link — each triggers up to 3 UNION ALL DB subqueries
  URL="https://<mirror-node>$NEXT"
  # No sleep needed; loop immediately to maximize DB pressure
done
```

Run multiple parallel instances of this loop targeting different accounts to multiply DB load. No authentication, no special privileges, no repeated identical requests — each request is a distinct, valid cursor-paginated call.

  getPaginationLink(req, rows, bounds, limit, order) {
    const primaryBound = bounds.primary;
    const secondaryBound = bounds.secondary;
    const isEnd = rows.length < limit;

    if (!bounds.next && (isEnd || (primaryBound.hasEqual() && secondaryBound.hasEqual()))) {
      // fetched all matching rows or the query is for a specific combination
      return null;
    }

    const lastRow = last(rows);
    const lastValues = {};
    if (primaryBound.hasBound() || primaryBound.isEmpty()) {
      // the primary param has bound or no primary param at all
      // primary param should be exclusive when the secondary operator is eq
      lastValues[primaryBound.filterKey] = {
        inclusive: !secondaryBound.hasEqual(),
        primary: true,
        value: !isEnd ? lastRow[primaryBound.viewModelKey] : bounds.next,
      };
    }

    if (secondaryBound.hasBound() || secondaryBound.isEmpty()) {
      // the secondary param has bound or no secondary param at all
      lastValues[secondaryBound.filterKey] = {value: lastRow[secondaryBound.viewModelKey]};
    }

    return utils.getPaginationLink(req, false, lastValues, order);
  }

const authHandler = async (req, res) => {

  if (!credentials) {

public class ThrottleConfiguration {

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
