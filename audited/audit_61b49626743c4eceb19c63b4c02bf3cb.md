### Title
Linear DB Query Amplification via Sequential `GetAccountAlias()` Calls in `updateOperationAccountAlias()`

### Summary
The `updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` issues one synchronous, non-batched `GetAccountAlias()` DB round-trip per operation whose account ID is not in the LRU cache. Because the Rosetta `/block` endpoint is unauthenticated and has no rate limiting, an unprivileged attacker can repeatedly request blocks containing many unique accounts — or rotate across many different blocks to evict the bounded LRU cache — causing DB query count to grow linearly with the number of unique accounts per request and sustaining elevated DB load across concurrent requests.

### Finding Description
**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–129:

```go
for _, transaction := range transactions {
    operations := transaction.Operations
    for index := range operations {
        accountId := operations[index].AccountId
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // one DB round-trip per cache miss
            ...
            s.entityCache.Set(result.GetId(), result)
        }
    }
}
```

Each cache miss calls `accountRepository.GetAccountAlias()` (`rosetta/app/persistence/account.go`, line 117), which executes a synchronous `db.Raw(selectCryptoEntityWithAliasById, ...).First(&entity)` — one full DB round-trip per unique account.

**Root cause:** No batching of alias lookups. The function iterates operations sequentially and issues individual DB queries rather than collecting all unique account IDs and resolving them in a single query.

**Cache insufficiency:** The LRU cache (`entityCache`) is process-wide and bounded by `entityCacheConfig.MaxSize` (1024 in tests, configurable). An attacker who rotates across many different blocks — each containing many unique accounts — continuously evicts cached entries, keeping the cache cold and forcing DB queries on every request. There is no per-request deduplication either: if the same account appears in multiple operations within one request and is not yet cached, it triggers multiple DB calls.

**No rate limiting:** A search across the entire `rosetta/` directory confirms there is no rate limiting, throttling, or request concurrency cap on the `/block` or `/block/transaction` endpoints.

**`maxTransactionsInBlock` is a partial bound, not a fix:** The `Block()` handler truncates transactions to `maxTransactionsInBlock` before calling `updateOperationAccountAlias()` (line 69), but this only limits the number of *transactions*, not the total number of unique account IDs across all operations within those transactions. A single Hedera block can contain transactions with many distinct account operations.

### Impact Explanation
An attacker sending concurrent `/block` requests targeting blocks with many unique accounts forces the mirror node's DB connection pool to service N×C sequential queries (N unique accounts per request, C concurrent requests). Because each `GetAccountAlias` query holds a DB connection for its duration, this exhausts the connection pool, increases DB CPU and I/O, and degrades response latency for all users of the mirror node. Sustained at scale, this can increase DB resource consumption well beyond 30% compared to baseline, without any brute-force credential attack.

### Likelihood Explanation
The Rosetta API is publicly accessible with no authentication required. Block data is public; an attacker can trivially identify high-activity blocks (e.g., blocks containing token distribution events or DEX activity) with hundreds of unique account IDs by querying the mirror node's REST API. The attack requires only an HTTP client and knowledge of block indices — no special privileges, credentials, or protocol knowledge. It is fully repeatable and automatable.

### Recommendation
1. **Batch alias lookups**: Collect all unique `AccountId` values across all operations in a single pass, then resolve them in one batched DB query (e.g., `WHERE id = ANY($1)`), replacing the per-account loop.
2. **Request-level deduplication**: Before issuing any DB calls, deduplicate account IDs within the current request to avoid redundant queries for the same account appearing in multiple operations.
3. **Rate limiting**: Add per-IP or global rate limiting on the `/block` and `/block/transaction` endpoints.
4. **Increase cache size**: Ensure `entityCacheConfig.MaxSize` is large enough to cover the working set of active accounts, reducing cold-cache attack surface.

### Proof of Concept
1. Identify a Hedera block with many transactions involving distinct accounts (e.g., a token airdrop block). Obtain its block index `B`.
2. Ensure the mirror node's `entityCache` is cold (restart, or use a block index not recently queried).
3. Send N concurrent HTTP POST requests to `/block` with `block_identifier.index = B`:
   ```
   POST /block
   {"network_identifier": {...}, "block_identifier": {"index": B}}
   ```
4. Observe DB query logs: each request triggers one `SELECT` per unique account in the block's operations.
5. To sustain the attack and defeat the cache, rotate across multiple high-activity block indices `B1, B2, B3, ...` so that each rotation evicts previously cached accounts.
6. Monitor DB CPU/connection utilization: it grows linearly with the number of unique accounts × concurrent requests, with no server-side throttle to stop it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```go
func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil
```

```go
func (ar *accountRepository) GetAccountAlias(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return accountId, nil
		}

		return zero, hErrors.ErrDatabaseError
	}

	if len(entity.Alias) == 0 {
		return accountId, nil
	}

	accountAlias, err := types.NewAccountIdFromEntity(entity)
	if err != nil {
		log.Warnf("Failed to create AccountId from alias '0x%s': %v", hex.EncodeToString(entity.Alias), err)
		return accountId, nil
	}

	return accountAlias, nil
}
```
