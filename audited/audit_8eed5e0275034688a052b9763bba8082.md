### Title
Unauthenticated Pagination-Driven Database Exhaustion via `getNftsByAccountId()` Endpoint

### Summary
The `getNftsByAccountId()` handler in `rest/controllers/accountController.js` issues an unbounded `NftService.getNfts()` database query on every request and returns a `next` pagination link that any unauthenticated caller can follow in rapid succession. The REST API layer has no per-IP rate limiting or in-flight request cap, allowing an unprivileged attacker to sustain a continuous stream of database queries that exhausts the DB connection pool and degrades mirror node processing capacity.

### Finding Description
**Exact code path:**

`rest/controllers/accountController.js`, `getNftsByAccountId()`, lines 90–103:
```js
getNftsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);
  const filters = utils.buildAndValidateFilters(req.query, acceptedNftAccountParameters);
  const query = this.extractNftMultiUnionQuery(filters, accountId);
  const nonFungibleTokens = await NftService.getNfts(query);   // DB hit, every call
  const nfts = nonFungibleTokens.map((nft) => new NftViewModel(nft));
  res.locals[responseDataLabel] = {
    nfts,
    links: { next: this.getPaginationLink(req, nfts, query.bounds, query.limit, query.order) },
  };
};
```

Every invocation unconditionally executes a database query. The response always includes a `next` link (when results equal `limit`) that encodes cursor parameters (`token.id=lte:X&serialnumber=lt:Y`) which the caller can immediately re-request.

**Root cause — no rate limiting on the REST API:**

A grep for `rateLimit|rate_limit|rateLimiter|throttle` across all `rest/**/*.js` files returns **zero matches**. The `extendExpress.js` middleware wrapper adds no throttling. The Helm chart for the REST service (`charts/hedera-mirror-rest/values.yaml`, lines 134–139) configures only:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 10
      initialInterval: 100ms
```
There is no `inFlightReq` or `rateLimit` Traefik middleware, unlike the Rosetta chart which explicitly includes both. The throttling infrastructure that exists (`web3/src/main/java/.../ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) is scoped exclusively to the `web3` module and is never applied to the REST API.

**Why existing checks fail:**

- The circuit breaker only fires when error ratios exceed 25%; successful DB-exhausting queries never trigger it.
- The `retry` middleware (10 attempts) amplifies load rather than reducing it.
- `buildAndValidateFilters` validates parameter types/operators but imposes no request-rate constraint.
- The `limit` cap of 100 rows per page reduces per-query cost but does not prevent rapid sequential pagination.

### Impact Explanation
An attacker targeting an account holding thousands of NFTs (common for NFT collection contracts) can chain hundreds of paginated requests per second. Each request consumes a DB connection for the duration of the query. With a finite DB connection pool (configured via `db.pool.maxConnections`), sustained flooding exhausts available connections, causing query timeouts and 500 errors for all mirror node REST consumers. This directly degrades mirror node processing capacity across the board, consistent with the ≥30% node degradation threshold described in the finding scope.

### Likelihood Explanation
No credentials, API keys, or special network access are required. The `next` link is self-contained and machine-readable, making automated pagination trivial. A single attacker with a script can issue hundreds of requests per second. Multiple coordinated attackers amplify the effect. The endpoint is publicly documented in the OpenAPI spec and is a standard REST pattern, making it easily discoverable.

### Recommendation
1. **Add per-IP rate limiting** at the Traefik ingress layer for the REST service, mirroring the Rosetta chart's `inFlightReq` (e.g., `amount: 20`) and `rateLimit` (e.g., `average: 50`) middleware entries in `charts/hedera-mirror-rest/values.yaml`.
2. **Add application-level throttling** in the REST Express app (e.g., `express-rate-limit`) as a defense-in-depth measure, applied globally or specifically to the `/accounts/:id/nfts` route in `rest/routes/accountRoute.js`.
3. **Remove the `retry` middleware** or reduce its `attempts` value for the REST service, as retrying read queries under load amplifies rather than mitigates DB pressure.
4. **Enforce a DB query timeout** specific to this endpoint to bound the maximum duration any single paginated query can hold a connection.

### Proof of Concept
```bash
# Step 1: Identify an account with many NFTs (e.g., a known NFT collection holder)
ACCOUNT="0.0.1234567"
BASE="https://<mirror-node-host>/api/v1"

# Step 2: Fetch first page and extract next link
NEXT=$(curl -s "$BASE/accounts/$ACCOUNT/nfts?limit=100" | jq -r '.links.next')

# Step 3: Rapidly follow pagination in a tight loop (no delay, no auth required)
while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  curl -s "$BASE$NEXT" -o /dev/null &   # fire-and-forget, parallel
  NEXT=$(curl -s "$BASE$NEXT" | jq -r '.links.next')
done

# Step 4: Run multiple instances of the above loop concurrently from different IPs
# Result: DB connection pool exhausted → 500 errors for all REST API consumers
```
