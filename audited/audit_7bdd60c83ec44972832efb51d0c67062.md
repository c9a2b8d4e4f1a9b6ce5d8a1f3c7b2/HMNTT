### Title
Unauthenticated `/metrics` Endpoint on Shared Port 5700 Enables Resource Exhaustion DoS Against Transaction Gossip Path

### Summary
The Rosetta server exposes a Prometheus `/metrics` endpoint with no authentication, no rate limiting, and no connection cap, co-located on the same port (5700) as the `/construction/submit` transaction gossip path. An unprivileged attacker can flood `GET /metrics` to exhaust server goroutines and CPU via repeated full-registry collections, starving the gossip path of capacity. The Traefik ingress rate-limiting middleware explicitly omits `/metrics` from its path list, leaving this vector unprotected even in Kubernetes deployments.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, `Routes()`, lines 64–73:
```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,          // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth wrapper
        },
    }
}
``` [1](#0-0) 

The `init()` function registers all four histograms/gauges under the `application="rosetta"` label on `prometheus.DefaultRegisterer`:
```go
register := prometheus.WrapRegistererWith(prometheus.Labels{"application": application}, prometheus.DefaultRegisterer)
``` [2](#0-1) 

This label is emitted verbatim in every Prometheus scrape response, confirming server identity to any caller.

**Same-port binding — no isolation:**

`rosetta/main.go` lines 220–227 bind the single `http.Server` (which serves both `/metrics` and `/construction/submit`) to `rosettaConfig.Port` (default 5700) with no `MaxConnsPerHost`, no connection limit, and no application-level rate limiter: [3](#0-2) 

**Traefik middleware gap:**

`charts/hedera-mirror-rosetta/values.yaml` lines 149–166 define `rateLimit: average: 10` and `inFlightReq: amount: 5`, but the ingress path list (lines 125–133) covers only `/rosetta/account`, `/rosetta/block`, `/rosetta/call`, `/rosetta/construction`, `/rosetta/events`, `/rosetta/mempool`, `/rosetta/network`, `/rosetta/search`. `/metrics` is absent — it is never routed through the Traefik middleware chain. [4](#0-3) [5](#0-4) 

In docker-compose, port 5700 is directly published (`- 5700:5700`), bypassing any proxy entirely. [6](#0-5) 

**Root cause:** `promhttp.Handler()` performs a synchronous, full-registry collection and serialization on every request. Under concurrent flood, this creates sustained CPU pressure and goroutine accumulation on the single shared listener, leaving no capacity for `/construction/submit`.

### Impact Explanation

`/construction/submit` is the Rosetta transaction gossip endpoint. If the server's goroutine pool or CPU is saturated by concurrent `/metrics` requests, gossip requests queue indefinitely or are rejected. For an exchange or integration relying on this node to broadcast signed transactions to the Hedera network, this constitutes a complete denial of transaction submission — no transactions can be gossiped for the duration of the attack. Severity: **High** (availability of a critical blockchain operation).

### Likelihood Explanation

- **No privilege required**: any network-reachable client can issue `GET /metrics`.
- **Trivially scriptable**: a simple `ab -n 100000 -c 500 http://<host>:5700/metrics` suffices.
- **Amplification**: each request triggers full Prometheus registry serialization (four histograms with multiple label combinations), making the cost asymmetric — cheap for the attacker, expensive for the server.
- **No existing application-layer defense**: confirmed by grep — no rate limiting, no connection cap, no auth in the rosetta Go codebase for this path.
- **Realistic in docker-compose deployments** (port directly exposed) and in any Kubernetes setup where the pod port is reachable (e.g., via `kubectl port-forward`, NodePort, or misconfigured NetworkPolicy).

### Recommendation

1. **Restrict `/metrics` to an internal/admin-only port** (separate `http.Server` binding to `127.0.0.1` or a dedicated management port), decoupled from the public API port 5700.
2. **Add authentication** to the metrics handler (Basic Auth or mTLS), consistent with how `rest/middleware/metricsHandler.js` handles it via the `authenticate()` function.
3. **Add `/metrics` to the Traefik ingress path list** (or a separate ingress with stricter rate limits) so the existing `rateLimit` and `inFlightReq` middleware applies.
4. **Set `http.Server.MaxHeaderBytes`** and consider a connection-limiting wrapper (e.g., `netutil.LimitListener`) on the public listener.

### Proof of Concept

**Preconditions:** Port 5700 reachable from attacker host (default docker-compose or direct pod access).

**Step 1 — Confirm server identity:**
```bash
curl -s http://<target>:5700/metrics | grep 'application="rosetta"'
# Output: hiero_mirror_rosetta_request_bytes_bucket{application="rosetta",...}
```

**Step 2 — Flood `/metrics` to exhaust server resources:**
```bash
# 500 concurrent connections, 200k total requests
ab -n 200000 -c 500 http://<target>:5700/metrics
```

**Step 3 — Simultaneously attempt transaction gossip (observe failure):**
```bash
curl -X POST http://<target>:5700/construction/submit \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"signed_transaction":"<hex>"}' 
# Result: connection timeout / 503 / no response — gossip blocked
```
