### Title
Unbounded Prometheus Label Cardinality via Arbitrary URL Paths in MetricsMiddleware (Route-Label DoS)

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` wraps all four `HistogramVec`/`GaugeVec` metrics with a `"route"` label whose value is derived from the matched gorilla/mux route template. When no registered route matches an incoming request, the `weaveworks/common/middleware.Instrument` falls back to using the raw request URI as the `route` label value. An unauthenticated attacker can flood the server with requests to unique URL paths, creating an unbounded number of distinct label-set combinations in the Prometheus registry's internal `sync.Map`, exhausting process memory and causing a denial of service.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83 — `MetricsMiddleware()`:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),  // gorilla/mux router
    }.Wrap(next)
}
```

All four metric vectors declare a `"route"` label:
- `requestBytesHistogram` — `[]string{"method", "route"}` (line 26)
- `requestDurationHistogram` — `[]string{"method", "route", "status_code", "ws"}` (line 32)
- `requestInflightGauge` — `[]string{"method", "route"}` (line 37)
- `responseBytesHistogram` — `[]string{"method", "route"}` (line 43)

**Root cause:**

`weaveworks/common/middleware.Instrument.Wrap()` resolves the `route` label by calling `RouteMatcher.Match(r, &mux.RouteMatch{})` on the gorilla/mux router. When the request path does not match any registered route, `Match` returns `false` and the library falls back to `r.RequestURI` (the raw URL path) as the label value. There is no normalization, capping, or allowlisting of the label value.

Each unique `route` value causes `prometheus.HistogramVec.With(labels)` to allocate a new set of bucket counters and store them in the registry's internal `sync.Map`. For a `HistogramVec` with 5 buckets, each new label combination allocates 7 new time-series objects (`_bucket×5`, `_count`, `_sum`). With 4 metric vectors, each unique path creates ~28 new persistent objects in memory that are never freed.

**Why existing checks fail:**

The `server.NewRouter` from `rosetta-sdk-go` registers only a fixed set of Rosetta API routes (e.g., `/network/list`, `/block`, `/construction/...`). There is no catch-all route, no rate limiter on unique paths, no label cardinality cap, and no authentication required to reach the metrics instrumentation layer — `MetricsMiddleware` is applied at the outermost layer in `main.go` (line 217), before any routing or authentication.

### Impact Explanation

An attacker sending `N` requests to `N` distinct URL paths causes `O(N)` permanent memory growth in the Rosetta process. With ~28 allocations per unique path and typical Go object overhead, tens of thousands of unique paths can exhaust hundreds of megabytes of heap. The process will eventually OOM-crash or be killed by the container runtime, taking down the Rosetta API entirely. Because the Prometheus registry never evicts label combinations, the memory is not recoverable without a process restart. This directly impacts availability of the Rosetta blockchain data API.

### Likelihood Explanation

No authentication, API key, or network restriction is required to reach the instrumented handler. The attack requires only the ability to send HTTP GET/POST requests to the Rosetta port (default 5700). A single attacker with a script loop generating unique paths (e.g., incrementing integers, random base64 strings) can trigger the condition. The attack is repeatable, cheap to execute, and requires no special knowledge of the API beyond the server's hostname and port.

### Recommendation

1. **Normalize unmatched routes to a fixed label value**: Override the route label to `"unmatched"` (or similar) when `RouteMatcher.Match` returns false, preventing raw URL paths from ever reaching the Prometheus registry.
2. **Use a custom `RouteMatcher` wrapper** that returns a fixed string for non-matching requests instead of delegating to the raw URI fallback.
3. **Apply a cardinality limiter**: Use `prometheus.WrapRegistererWithPrefix` combined with a cardinality-limiting registerer (e.g., `prometheus-community/prom-label-proxy` patterns) to hard-cap the number of distinct label combinations.
4. **Rate-limit requests to unknown paths** at the HTTP layer before they reach `MetricsMiddleware`.

### Proof of Concept

```bash
# Send 100,000 requests with unique paths to the Rosetta server

  curl -s -o /dev/null "http://<rosetta-host>:5700/$(openssl rand -base64 12 | tr -d '=')" &
done
wait
