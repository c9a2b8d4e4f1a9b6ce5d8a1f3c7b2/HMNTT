### Title
Unauthenticated DoS via Unbounded Concurrent Requests to `GET /api/v1/accounts/{ownerId}/hooks` Exhausting DB Connection Pool

### Summary
The `getHooks()` endpoint in `HooksController` is publicly accessible with no authentication or rate limiting. When called without a `hook.id` filter, it unconditionally executes `findByOwnerIdAndHookIdBetween(id, 0, Long.MAX_VALUE, page)` — a full-range DB scan — for every request. An unprivileged attacker flooding this endpoint with concurrent requests can exhaust the database connection pool, denying service to legitimate gossip-prevention hook queries.

### Finding Description
**Exact code path:**

In `HooksController.java` lines 80–102, `getHooks()` is a public `@GetMapping` with no authentication annotation. The only input constraints are `@Max(MAX_LIMIT)` (caps `limit` at 100) and `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` (caps the number of `hook.id` params at 100) — neither is a rate limit. [1](#0-0) 

In `HookServiceImpl.java` lines 39–40, when `hookIds` is empty (i.e., no `hook.id` filter is provided), the service calls `hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), 0, Long.MAX_VALUE, page)` — a broad range scan bounded only by `ownerId` and the pageable limit. [2](#0-1) 

The repository method is a Spring Data JPA derived query that translates directly to a `WHERE owner_id = ? AND hook_id BETWEEN 0 AND 9223372036854775807` SQL query with a `LIMIT 100`. [3](#0-2) 

**Root cause:** The failed assumption is that the `Pageable` row limit (100) is sufficient to bound server-side cost. It limits result set size but does not prevent the DB engine from performing an index scan across the full `[0, MAX_VALUE]` range per request, and it does not prevent thousands of such queries from running concurrently.

**Why existing checks fail:**
- `@Max(MAX_LIMIT)` / `@Positive` only validate the `limit` parameter value, not request frequency.
- `HIGH_VOLUME_THROTTLE` is defined in `Constants.java` and applied in `NetworkController`, but is **not applied** to `HooksController`. [4](#0-3) [5](#0-4) 
- No `@RateLimiter`, `@Bulkhead`, or Resilience4j annotation is present on `HooksController` or `HookServiceImpl`.

### Impact Explanation
Concurrent flooding of `GET /api/v1/accounts/{ownerId}/hooks` (no `hook.id` filter, `limit=100`) causes each request to hold a DB connection for the duration of the range scan. With a finite HikariCP connection pool (default typically 10–20 connections), a few hundred concurrent requests are sufficient to exhaust the pool. All subsequent DB-dependent operations — including legitimate gossip-prevention hook queries — will queue or fail with connection timeout errors, constituting a full denial of service for the hooks subsystem.

### Likelihood Explanation
The endpoint requires no credentials, no API key, and no session. Any external actor with network access can exploit this. The attack is trivially repeatable using standard HTTP load tools (`ab`, `wrk`, `hey`). The attacker does not need to know valid `ownerId` values in advance — even invalid IDs trigger the `entityService.lookup()` path before the DB query, and valid IDs (account IDs) are publicly enumerable from the ledger. The attack is stateless and can be sustained indefinitely.

### Recommendation
1. **Apply rate limiting** to `HooksController` using the existing `HIGH_VOLUME_THROTTLE` mechanism already used in `NetworkController`, or add a Resilience4j `@RateLimiter`/`@Bulkhead` annotation.
2. **Require authentication** for this endpoint if hooks are owner-sensitive resources; unauthenticated access to hook metadata is likely unintended.
3. **Add a query timeout** at the repository/datasource level so runaway range scans are killed before they hold connections indefinitely.
4. **Enforce a tighter default upper bound** — instead of `Long.MAX_VALUE`, cap the range to a reasonable window (e.g., `lowerBound + MAX_LIMIT`) when no explicit upper bound is provided.

### Proof of Concept
```bash
# Flood the endpoint with 500 concurrent requests, no hook.id filter
# Replace <HOST> and <VALID_OWNER_ID> with actual values (owner IDs are public)
hey -n 5000 -c 500 \
  "http://<HOST>/api/v1/accounts/<VALID_OWNER_ID>/hooks?limit=100"

# Expected result:
# - DB connection pool exhausted within seconds
# - Subsequent legitimate requests return 500 / connection timeout
# - Application logs show HikariPool-1 - Connection is not available, request timed out
```
