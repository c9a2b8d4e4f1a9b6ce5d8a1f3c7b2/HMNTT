### Title
Unbounded Concurrent gRPC Subscriber Accumulation Causes OOM-Based DoS in Mirror Node

### Summary
`TopicMessageServiceImpl.subscribeTopic()` tracks active subscribers via `subscriberCount` but never enforces any cap on it. An unprivileged attacker can open an unlimited number of concurrent `subscribeTopic` gRPC streams, each allocating a `TopicContext`, a `PollingContext` (in POLL mode), and a per-subscriber reactive pipeline with a bounded buffer, exhausting JVM heap memory and crashing the mirror node service.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.java` lines 48–91: [1](#0-0) 

The `subscriberCount` `AtomicLong` is registered as a Micrometer `Gauge` for observability only. It is incremented on subscribe and decremented on termination: [2](#0-1) 

There is **no guard** of the form `if (subscriberCount.get() >= MAX) return Flux.error(...)` anywhere in `subscribeTopic()` or in `TopicListener.listen()`.

Each call to `subscribeTopic()` unconditionally allocates a `TopicContext` (with `AtomicLong`, `AtomicReference`, `Stopwatch`): [3](#0-2) 

When the listener type is `POLL`, each `listen()` call in `PollingTopicListener` additionally allocates a `PollingContext` and schedules an independent, long-lived polling loop on `boundedElastic`: [4](#0-3) 

When the listener type is `SHARED_POLL` or `REDIS`, `SharedTopicListener.listen()` allocates a per-subscriber `onBackpressureBuffer` of up to `maxBufferSize` (default 16,384) `TopicMessage` references per subscriber: [5](#0-4) 

`ListenerProperties` defines `maxBufferSize` defaulting to 16,384 with no subscriber count limit: [6](#0-5) 

**Root cause:** The `subscriberCount` field exists solely as a metrics gauge. The failed assumption is that some external layer (load balancer, Netty connection cap) will prevent subscriber flooding. No such enforcement is present in the application logic itself.

### Impact Explanation

- **POLL mode:** N concurrent subscribers → N independent `PollingContext` objects + N reactive pipeline chains + N periodic DB query tasks queued on `boundedElastic`. Memory and DB connection pool are exhausted proportionally to N.
- **SHARED_POLL / REDIS mode:** N subscribers → N per-subscriber `onBackpressureBuffer(16384)` allocations. At 16,384 slots × ~200 bytes per `TopicMessage` reference chain × N subscribers, a few thousand subscribers can exhaust gigabytes of heap.
- Result: JVM `OutOfMemoryError`, crashing the mirror node gRPC service entirely, denying service to all legitimate users.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. A single attacker machine can open thousands of gRPC HTTP/2 streams (multiplexed over multiple TCP connections to bypass any per-connection stream limit). The attack is trivially scriptable with any gRPC client library, requires no special knowledge, and is repeatable. The mirror node is a public infrastructure component, making this a realistic and high-impact threat.

### Recommendation

1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()` using the already-tracked `subscriberCount`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Flux.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
2. **Add `maxSubscribers` to `GrpcProperties`** with a safe default (e.g., 500).
3. **Add per-IP connection rate limiting** at the Netty/gRPC interceptor layer to prevent a single source from opening many connections.
4. For `POLL` mode specifically, consider migrating all subscribers to `SHARED_POLL` to eliminate per-subscriber DB polling.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

def open_subscription(i):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,  # any valid topic
    )
    for _ in stub.subscribeTopic(req):
        pass  # keep stream open

threads = [threading.Thread(target=open_subscription, args=(i,)) for i in range(5000)]
for t in threads:
    t.start()
# Mirror node JVM OOM occurs as PollingContext/TopicContext objects accumulate
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
