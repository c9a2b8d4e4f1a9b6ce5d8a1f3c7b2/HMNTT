### Title
Missing Account Existence Check for Numeric IDs in `getAccountTokenAllowances` Returns 200 for Non-Existent Accounts

### Summary
`GET /api/v1/accounts/:idOrAliasOrEvmAddress/allowances/tokens` returns HTTP 200 with an empty allowances array when queried with a numeric account ID that does not exist in the database. This occurs because `EntityService.getEncodedId()` skips the database existence check for numeric IDs, and `getAccountTokenAllowances` never calls `EntityService.isValidAccount()` to compensate. Mirror node consumers receiving a 200 response cannot distinguish "account exists with no allowances" from "account was never created."

### Finding Description

**Code path:**

`rest/controllers/tokenAllowanceController.js`, `getAccountTokenAllowances`, lines 68–81:

```js
getAccountTokenAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  // No isValidAccount() call here
  const filters = utils.buildAndValidateFilters(req.query, acceptedTokenAllowanceParameters);
  const query = this.extractTokenMultiUnionQuery(filters, accountId);
  const tokenAllowances = await TokenAllowanceService.getAccountTokenAllowances(query);
  const allowances = tokenAllowances.map((model) => new TokenAllowanceViewModel(model));
  res.locals[responseDataLabel] = { allowances, links: { next: ... } };
};
``` [1](#0-0) 

**Root cause — `getEncodedId` skips DB lookup for numeric IDs:**

In `rest/service/entityService.js` lines 120–124, when the input passes `EntityId.isValidEntityId()` (i.e., it is a numeric `shard.realm.num` or plain number) and has no EVM address component, the function returns the encoded ID immediately without any database query:

```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()   // ← returns immediately, no DB check
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
``` [2](#0-1) 

Alias and EVM address inputs do hit the database and throw `NotFoundError` on miss. Numeric IDs do not.

**Why existing checks fail:**

The `getEncodedId` JSDoc explicitly documents this asymmetry: it only throws `NotFoundError` for alias/EVM address lookups, not for numeric IDs. The comment in the test suite for `rest-java` even notes: *"Valid numeric account IDs are not looked up in the entity table in support of partial mirror nodes."* [3](#0-2) 

The fix pattern exists in the same codebase: `listStakingRewardsByAccountId` in `rest/controllers/accountController.js` calls `EntityService.isValidAccount(accountId)` after `getEncodedId` and throws `NotFoundError` if the account is absent. `getAccountTokenAllowances` omits this step entirely. [4](#0-3) 

**Test spec confirms the behavior is present and accepted:**

`rest/__tests__/specs/accounts/{id}/allowances/tokens/account-not-found.json` explicitly asserts that querying a non-existent numeric account returns HTTP 200 with `{"allowances": [], "links": {"next": null}}`, while the same query with an alias returns 404. [5](#0-4) 

### Impact Explanation

Mirror node consumers (indexers, wallets, explorers, compliance tools) that call this endpoint and receive a 200 response have no way to distinguish between:
- An account that exists and has granted zero token allowances, and
- An account that was never created on the network.

A consumer that exports or caches this result will record a false "no token allowances" record for a phantom account. This corrupts downstream state: account existence maps, allowance indexes, and audit trails. For compliance or regulatory consumers, this constitutes incorrect data export about account state. The impact is medium-severity data integrity corruption with no authentication required.

### Likelihood Explanation

Any unauthenticated external user can trigger this with a single HTTP GET request using any arbitrary numeric account ID. No special privileges, tokens, or prior knowledge of the system are required. The request is indistinguishable from a legitimate query. It is trivially repeatable at scale (e.g., scanning a range of account IDs to pollute a consumer's allowance index). The behavior is stable and deterministic — it will reproduce on every request for every non-existent numeric account ID.

### Recommendation

Add an account existence check in `getAccountTokenAllowances` immediately after resolving the encoded ID, mirroring the pattern already used in `listStakingRewardsByAccountId`:

```js
getAccountTokenAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValidAccount = await EntityService.isValidAccount(accountId);
  if (!isValidAccount) {
    throw new NotFoundError();
  }
  // ... rest of handler
};
```

Additionally, update the test spec `account-not-found.json` to assert HTTP 404 (not 200) for the numeric ID case, consistent with the alias case.

### Proof of Concept

**Precondition:** The mirror node database contains no account with numeric ID `1001` (empty setup, as in the test spec).

**Trigger:**
```
GET /api/v1/accounts/1001/allowances/tokens
```
No authentication, no special headers required.

**Result:**
```json
HTTP 200 OK
{
  "allowances": [],
  "links": { "next": null }
}
```

**Expected (correct) result:**
```json
HTTP 404 Not Found
{
  "_status": { "messages": [{ "message": "Not found" }] }
}
```

**Contrast:** Querying the same non-existent account via alias correctly returns 404:
```
GET /api/v1/accounts/KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ/allowances/tokens
→ HTTP 404
```

This discrepancy is directly confirmed by the project's own test fixture at `rest/__tests__/specs/accounts/{id}/allowances/tokens/account-not-found.json`.
