### Title
Unbounded Path Logging in `TracingMiddleware` Enables Log-Flood DoS via Goroutine Starvation

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` logs the full, unsanitized request URI for every non-internal request with no length cap, no rate limiting, and no path normalization. Because logrus writes synchronously to `os.Stdout` under a global mutex, an attacker flooding the server with requests carrying long, unique URIs can saturate the stdout pipe, causing every HTTP handler goroutine to block indefinitely at `log.Info()`, exhausting the goroutine pool and stalling the node's ability to serve inbound connections.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47 and 52–58:

```go
path := request.URL.RequestURI()          // line 47 – full URI, no truncation
// ...
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)  // lines 52-53 – path embedded verbatim
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)   // line 58 – synchronous, mutex-held write to os.Stdout
}
```

`main.go` line 54 sets the global logrus output to `os.Stdout`:
```go
log.SetOutput(os.Stdout)
```

The `http.Server` constructed at `main.go` lines 220–227 sets no `MaxHeaderBytes`, leaving Go's default of 1 MiB for the request line + headers. No `MaxBytesReader` or URL-length guard exists anywhere in the rosetta middleware stack (confirmed by grep: zero matches for `MaxHeaderBytes`, `LimitReader`, `MaxBytesReader` in `rosetta/`).

**Root cause and failed assumption:**

The design assumes log output is always consumed faster than it is produced. In containerized deployments (Docker, Kubernetes — the primary deployment target, as evidenced by the Helm chart at `charts/hedera-mirror-rosetta/` and the Loki comment at `main.go` line 43), stdout is a pipe to a log collector (Fluentd, Loki, etc.). When the collector is slow or the pipe buffer (~64 KiB on Linux) fills, `os.Stdout.Write()` blocks. Logrus acquires its global mutex before calling `Write()` and holds it for the duration. Every concurrent goroutine that reaches `log.Info()` then blocks waiting for the mutex, stalling indefinitely.

**Why existing checks are insufficient:**

- The Traefik rate limit (`average: 10` per host, `charts/hedera-mirror-rosetta/values.yaml` lines 157–161) is optional infrastructure, not enforced by the application. Direct access to port 5700 bypasses it entirely.
- `ReadHeaderTimeout` (3 s default) and `ReadTimeout` (5 s default) govern request ingestion, not post-response logging. `log.Info()` is called after `inner.ServeHTTP()` returns and the response is already written, so these timeouts do not interrupt the blocking log write.
- `WriteTimeout` (10 s default) covers the response write deadline on the connection, not arbitrary code executing after the write completes.
- There is no `http.Server.MaxHeaderBytes` override, no path truncation, and no application-level concurrency or rate limit.

**Exploit flow:**

1. Attacker opens many concurrent TCP connections to port 5700.
2. Each connection sends a GET request with a ~1 MiB URI (e.g., `GET /AAAA…(1 MB)… HTTP/1.1`), all unique (e.g., incrementing suffix).
3. Each request is processed by `inner.ServeHTTP()` (returns 404 quickly), then `TracingMiddleware` calls `log.Info()` with a ~1 MiB message.
4. The stdout pipe buffer fills; subsequent `log.Info()` calls block while holding the logrus mutex.
5. All handler goroutines pile up blocked at the mutex. New connections spawn new goroutines that also immediately block.
6. Goroutine count grows unbounded, consuming memory. Eventually the Go runtime OOMs or the goroutine scheduler is saturated.
7. The node stops responding to legitimate Rosetta API calls — network partition from inbound connections.

Additionally, `getClientIpAddress()` (lines 63–74) trusts the `X-Real-IP` header without length validation, allowing the attacker to further amplify each log line's size with a spoofed, arbitrarily long IP string.

### Impact Explanation

A successful attack partitions the Rosetta mirror node from all inbound API consumers (block explorers, exchanges, Coinbase Rosetta validator). Because the Rosetta API is used for balance reconciliation and transaction construction, sustained unavailability breaks downstream financial operations. The attack requires no authentication, no valid Rosetta payload, and no prior knowledge of the system beyond the open port. Severity: **High** (availability impact on a public-facing financial API node).

### Likelihood Explanation

Preconditions are minimal: network access to port 5700 (publicly exposed per the Helm ingress config), ability to open concurrent TCP connections, and ability to craft large HTTP request lines. No credentials, no valid API knowledge, and no exploit tooling beyond `curl` or `wrk` are required. The attack is repeatable and can be sustained indefinitely. The containerized deployment model (Docker/K8s with piped stdout) is the standard and documented deployment path, making the blocking condition reliably triggerable. Likelihood: **High**.

### Recommendation

1. **Truncate the path before logging** — cap `path` to a safe maximum (e.g., 512 bytes) in `TracingMiddleware`:
   ```go
   path := request.URL.RequestURI()
   if len(path) > 512 {
       path = path[:512] + "…"
   }
   ```
2. **Truncate the client IP** — apply a similar cap to `clientIpAddress` returned by `getClientIpAddress()`.
3. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g., 8 KiB) to reject oversized request lines before they reach middleware.
4. **Use non-blocking / async logging** — configure logrus with a buffered, non-blocking writer so a slow consumer does not block handler goroutines.
5. **Add application-level rate limiting** — enforce a per-IP or global request rate limit inside the Go application, independent of infrastructure-level controls.

### Proof of Concept

```bash

python3 -c "
import socket, threading, itertools, string, random

def attack(n):
    path = '/' + ''.join(random.choices(string.ascii_lowercase, k=8000)) + str(n)
    req = f'GET {path} HTTP/1.1\r\nHost: target:5700\r\nX-Real-IP: {'A'*500}\r\n\r\n'
    try:
        s = socket.create_connection(('TARGET_IP', 5700), timeout=30)
        s.sendall(req.encode())
        s.recv(4096)
        # hold connection open to keep goroutine blocked
        import time; time.sleep(60)
    except: pass

threads = [threading.Thread(target=attack, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
"
