### Title
Unauthenticated DB I/O Exhaustion via Unbounded Concurrent NFT Point-Lookup Requests

### Summary
The `getNftTokenInfoRequest()` handler in `rest/tokens.js` performs a direct, uncached database query for every incoming request with no application-level rate limiting. An unprivileged attacker can flood the endpoint with thousands of concurrent requests targeting distinct valid `(tokenId, serialNumber)` pairs, forcing repeated disk I/O on the `nft` table when those rows are not in the PostgreSQL buffer cache. Under sufficient load this saturates DB I/O bandwidth, starving the importer's write path and degrading or halting transaction ingestion.

### Finding Description

**Exact code path:**

`rest/tokens.js`, `getNftTokenInfoRequest()`, lines 866–880:

```js
const getNftTokenInfoRequest = async (req, res) => {
  utils.validateReq(req);
  const tokenId = getAndValidateTokenIdRequestPathParam(req);
  const serialNumber = getAndValidateSerialNumberRequestPathParam(req);

  const {query, params} = extractSqlFromNftTokenInfoRequest(tokenId, serialNumber, nftSelectQuery);
  const {rows} = await pool.queryQuietly(query, params);   // ← unconditional DB hit
  ...
};
```

`extractSqlFromNftTokenInfoRequest()` (lines 795–804) builds:

```sql
SELECT ... FROM nft
LEFT JOIN entity e ON e.id = nft.token_id
WHERE nft.token_id = $1
  AND nft.serial_number = $2
```

Every request unconditionally executes this query against the live PostgreSQL pool.

**Root cause — three failed assumptions:**

1. *Application-level rate limiting exists for this endpoint.* It does not. The `ThrottleConfiguration` / `ThrottleManagerImpl` classes live exclusively in the `web3` (Java) module and are never applied to the Node.js REST API. `rest/server.js` registers no rate-limit middleware for any token route (lines 119–124).

2. *The Redis response cache will absorb repeated requests.* The cache is gated on `config.cache.response.enabled && config.redis.enabled` (server.js line 54). When Redis is absent or disabled the cache layer is entirely skipped. Even when enabled, `DEFAULT_REDIS_EXPIRY = 1` second (responseCacheHandler.js line 24) means a sustained flood of requests for the same key still hits the DB once per second per key; a flood across thousands of *distinct* keys bypasses the cache entirely.

3. *Traefik middleware rate-limiting is always present.* The Helm chart middleware is optional (`{{ if and .Values.global.middleware .Values.middleware }}`). Many deployments omit it or configure it only at the host level, not per-path.

**Exploit flow:**

An attacker enumerates valid `(tokenId, serialNumber)` pairs (publicly visible on-chain or via the `/tokens/:tokenId/nfts` list endpoint). They then issue thousands of concurrent GET requests cycling through distinct pairs. Because each pair maps to a unique row, the PostgreSQL shared-buffer cache cannot absorb the working set, and every request triggers a physical disk read. The `pool.queryQuietly` call holds a connection from the bounded pool for the duration of the I/O wait; when the pool is saturated, new requests queue, amplifying latency. The importer's write path competes for the same disk I/O subsystem and the same PostgreSQL I/O scheduler slots.

### Impact Explanation

PostgreSQL uses a single shared I/O subsystem. Saturating it with read I/O from the REST API directly degrades write throughput for the importer, which must persist every consensus transaction to the same database. Sustained saturation causes transaction ingestion lag to grow unboundedly, eventually causing the mirror node to fall behind the network. In the worst case (e.g., cloud-hosted DB with a fixed IOPS ceiling), ingestion halts entirely, constituting a network-observable outage of the mirror node. The `nft` table can contain tens of millions of rows across a production network, making the cold-cache attack surface large.

### Likelihood Explanation

No authentication, API key, or proof-of-work is required. Valid `(tokenId, serialNumber)` pairs are freely discoverable. The attack requires only a standard HTTP client and a modest botnet or even a single high-bandwidth host with connection multiplexing. The absence of application-level rate limiting means the attack is repeatable indefinitely and is not mitigated by rotating source IPs unless Traefik rate limiting is explicitly deployed and tuned per-path.

### Recommendation

1. **Add application-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit` or `rate-limiter-flexible`) scoped per source IP, applied before route handlers in `rest/server.js`.
2. **Enforce a mandatory Redis response cache** for this endpoint (or raise the minimum TTL) so repeated requests for the same NFT are served from memory.
3. **Set a hard concurrency limit** on `pool.maxConnections` and add a request-queue depth limit so the DB pool cannot be monopolised by a single client.
4. **Separate read and write DB connection pools** so REST API read traffic cannot starve the importer's write connections.
5. **Deploy Traefik `inFlightReq` and `rateLimit` middleware** as a required (not optional) component for public-facing deployments.

### Proof of Concept

```bash
# 1. Enumerate valid NFT serial numbers for a known token
curl "https://<mirror>/api/v1/tokens/0.0.1234/nfts?limit=1000" | jq '.[].serial_number' > serials.txt

# 2. Flood the endpoint with distinct serial numbers (no auth required)
cat serials.txt | xargs -P 500 -I{} \
  curl -s -o /dev/null "https://<mirror>/api/v1/tokens/0.0.1234/nfts/{}"

# 3. Observe: DB I/O wait climbs, importer lag increases, transaction ingestion stalls
# Monitor: SELECT * FROM pg_stat_activity WHERE wait_event_type = 'IO';
```

Repeating step 2 continuously with rotating serial numbers (or across multiple token IDs) sustains the disk I/O pressure. No privileged access, API key, or special tooling is required.
