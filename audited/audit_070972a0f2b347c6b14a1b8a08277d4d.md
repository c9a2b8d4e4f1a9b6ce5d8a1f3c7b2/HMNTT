### Title
Unbounded Alias Length in GraphQL `findByAlias()` Enables Unauthenticated Denial-of-Service

### Summary
The GraphQL `AccountInput.alias` field is validated only by a character-set regex (`^[A-Z2-7]+$`) with no upper length bound. An unauthenticated attacker can submit an arbitrarily long valid Base32 string that passes schema validation, is decoded into an oversized `byte[]` by `decodeBase32()`, and is forwarded as a `bytea` parameter to PostgreSQL via `findByAlias()`. Repeated requests exhaust JVM heap and database connection memory, causing a denial-of-service.

### Finding Description

**Exact code path:**

1. **Schema** — `graphql/src/main/resources/graphql/account.graphqls`, line 90:
   ```
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   The `@Pattern` directive (defined in `directive.graphqls` line 6) enforces only the Base32 alphabet. There is no `@Size` or maximum-length constraint. A string of 10,000,000 `A` characters satisfies `^[A-Z2-7]+$`.

2. **Controller** — `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, line 33–48:
   The `@Valid AccountInput input` triggers graphql-java-extended-validation, which runs the `@Pattern` check. Because the regex has no length bound, the oversized alias passes.

3. **Service** — `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 30:
   ```java
   return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   ```

4. **Decode** — `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, line 83–85:
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   No length check. A 10 MB Base32 string decodes to a ~6.25 MB `byte[]`.

5. **Repository** — `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 13–14:
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```
   The oversized `byte[]` is bound as a PostgreSQL `bytea` parameter with no size guard.

**Root cause:** The `@Pattern` directive validates only character set, not string length. No downstream check in `decodeBase32()`, `getByAliasAndType()`, or `findByAlias()` enforces a maximum byte length.

**Why existing checks fail:** The `@Pattern(regexp: "^[A-Z2-7]+$")` regex matches any non-empty string of Base32 characters regardless of length. The `@Valid` annotation triggers this check but there is no `@Size` or equivalent constraint. Compare with the REST Java API (`rest-java/src/main/java/org/hiero/mirror/restjava/parameter/EntityIdAliasParameter.java`, line 13), which correctly enforces `[A-Z2-7]{40,70}` — a hard maximum of 70 characters — but the GraphQL path has no equivalent.

### Impact Explanation

Each request with a multi-megabyte alias causes:
- JVM heap allocation of a large `byte[]` per request (no pooling or early rejection)
- A PostgreSQL query with an oversized `bytea` bind parameter, consuming server-side memory and parse/plan time
- Under concurrent load, heap exhaustion triggers GC pressure or `OutOfMemoryError`, taking down the GraphQL service for all users

Severity: **High** — complete service disruption achievable without authentication.

### Likelihood Explanation

The GraphQL endpoint (`/graphql/alpha`) is publicly accessible with no authentication requirement. The attack requires only a standard HTTP POST with a JSON body containing a long string of uppercase letters. No special knowledge, credentials, or protocol access is needed. The attack is trivially scriptable and repeatable.

### Recommendation

1. Add a `@Size` (or equivalent) constraint to the `alias` field in `account.graphqls`, mirroring the REST Java limit:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]{2,70}$")
   ```
   This enforces both character set and a maximum length of 70 characters (matching the `{40,70}` bound in `EntityIdAliasParameter.ALIAS_REGEX`).

2. Add a defensive length check in `GraphQlUtils.decodeBase32()` or `EntityServiceImpl.getByAliasAndType()` as a defense-in-depth measure:
   ```java
   if (alias != null && alias.length() > 70) {
       throw new IllegalArgumentException("Alias exceeds maximum length");
   }
   ```

3. Configure Spring Boot's maximum HTTP request body size (`spring.mvc.servlet.max-request-size`) to limit oversized payloads at the HTTP layer.

### Proof of Concept

```bash
# Generate a 5MB alias string of valid Base32 characters
ALIAS=$(python3 -c "print('A' * 8000000)")

# Send to the GraphQL endpoint (no authentication required)
curl -s -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"${ALIAS}\\\" }) { id } }\"}"
```

Repeat this request concurrently (e.g., 50 parallel connections) to exhaust JVM heap and PostgreSQL connection memory. The service will become unresponsive to all concurrent `findByAlias()` queries.

**Verification of missing length bound:**
- `account.graphqls` line 90: `alias: String @Pattern(regexp: "^[A-Z2-7]+$")` — no `{n,m}` quantifier
- `EntityIdAliasParameter.java` line 13: `[A-Z2-7]{40,70}` — REST Java correctly bounds to 70 chars max
- `GraphQlUtils.decodeBase32()` line 83–85: no length guard before `BASE32.decode()` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
