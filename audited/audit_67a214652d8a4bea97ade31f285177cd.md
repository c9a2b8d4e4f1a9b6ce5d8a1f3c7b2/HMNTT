### Title
Unsanitized IPv6 X-Forwarded-For Header Logged Verbatim Enables Log Misparsing and Source IP Obfuscation on /construction/submit

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the `X-Forwarded-For` header value without any validation or sanitization and passes it directly into the log message via `fmt.Sprintf`. Because Traefik is explicitly configured with `--entryPoints.web.forwardedHeaders.insecure`, any external unprivileged client can inject an arbitrary string — including an IPv6 address containing colons — into the log entry for `/construction/submit` requests. Log parsers or SIEM tools that tokenize on colons will misparse the entry and fail to extract the true source IP, defeating IP-based alerting for tampered transaction submissions.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64 – no validation
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 – no validation
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress  // returned verbatim
}
```

The returned value is interpolated directly into the log message at lines 52–53:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))
```

The logger is configured as logrus `TextFormatter` (logfmt) in `rosetta/main.go` lines 43–52, producing output of the form:
```
time="..." level=info msg="::ffff:192.0.2.1 POST /construction/submit (200) in 5ms"
```

**Root cause / failed assumption:** The code assumes the `X-Forwarded-For` / `X-Real-IP` header contains a well-formed IPv4 address. No `net.ParseIP`, regex, or length check is applied before the value is embedded in the log string.

**Why existing checks fail:** The only fallback is `net.SplitHostPort(r.RemoteAddr)` (line 71), which is only reached when *both* headers are absent. If either header is present — regardless of content — its raw value is used. The Traefik deployment configuration at `charts/hedera-mirror-common/values.yaml` lines 643–645 explicitly sets `--entryPoints.web.forwardedHeaders.insecure` for both HTTP and HTTPS entry points, meaning Traefik forwards the client-supplied `X-Forwarded-For` header to the rosetta service without stripping or overwriting it.

### Impact Explanation
A SIEM or log-shipping pipeline that tokenizes the logfmt `msg` field on colons (common in tools like Splunk field extraction, Loki label parsing, or custom AWK/grep pipelines) will misparse the entry. For the injected value `::ffff:192.0.2.1`, splitting on `:` yields empty tokens before `ffff`, and the remainder `192.0.2.1 POST /construction/submit (200) in 5ms` is treated as a single token — the IP field is lost or wrong. This means:
- IP-based alerting rules for `/construction/submit` fail to fire against the attacker's real address.
- Forensic reconstruction of who submitted a tampered transaction is impaired.
- The attacker can also inject arbitrary free-form strings (spaces, fake fields) to further corrupt the log record.

### Likelihood Explanation
Preconditions are minimal: the attacker needs only the ability to send an HTTP request to the rosetta endpoint (port 5700 or via the nginx/Traefik proxy). No authentication, no special role, no network-internal position is required. The `forwardedHeaders.insecure` Traefik flag is present in the shipped Helm chart values, making this the default production posture. The attack is trivially repeatable with a single `curl` command and leaves no server-side error or rejection.

### Recommendation
1. **Validate the extracted IP** in `getClientIpAddress()` using `net.ParseIP` and fall back to `r.RemoteAddr` if the header value is not a valid IP:
   ```go
   if ip := net.ParseIP(strings.TrimSpace(ipAddress)); ip == nil {
       ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
   }
   ```
2. **Log the IP as a structured field** rather than embedding it in a free-form string, so parsers never need to tokenize the message body for the IP.
3. **Remove `--entryPoints.web.forwardedHeaders.insecure`** from the Traefik configuration and instead enumerate trusted proxy CIDRs with `--entryPoints.web.forwardedHeaders.trustedIPs`.

### Proof of Concept
```bash
# Send a POST to /construction/submit with an IPv6 X-Forwarded-For header
curl -s -X POST http://<rosetta-host>:5700/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Forwarded-For: ::ffff:192.0.2.1" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"signed_transaction":"invalid"}'

# Resulting logfmt log line (observed in stdout):
# time="2024-01-01T00:00:00Z" level=info msg="::ffff:192.0.2.1 POST /construction/submit (422) in 1.2ms"

# A colon-splitting parser (e.g., awk -F: '{print $1}') applied to the msg value
# returns an empty string instead of the attacker's real IP, defeating alerting.
echo '::ffff:192.0.2.1 POST /construction/submit (422) in 1.2ms' | awk -F: '{print "IP field:", $1}'
# Output: "IP field: " (empty — attacker's real IP is lost)
