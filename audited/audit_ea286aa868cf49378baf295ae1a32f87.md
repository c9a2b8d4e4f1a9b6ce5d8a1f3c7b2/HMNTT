### Title
Uncached Per-Request DB Query via EntityIdAliasParameter Enables Unauthenticated DoS on Mirror Node

### Summary
`NftAllowanceServiceImpl.getNftAllowances()` unconditionally calls `entityService.lookup()` for every request. When the `{id}` path variable is an alias-format string (matching `EntityIdAliasParameter`), `EntityServiceImpl.lookup()` executes a raw SQL query against the `entity` table with zero caching at any layer in the `rest-java` module. There is no rate limiting in `rest-java` (unlike the `web3` module), so an unauthenticated attacker can trivially saturate database CPU by flooding this endpoint with valid alias-format account IDs.

### Finding Description

**Exact code path:**

`AllowancesController` (line 59–75) accepts `@PathVariable EntityIdParameter id`, which Spring resolves to an `EntityIdAliasParameter` when the path segment matches `ALIAS_REGEX`:

```
^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$
```

Any 40–70 uppercase base32 character string (e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`) satisfies this pattern. [1](#0-0) 

The controller builds an `NftAllowanceRequest` with `accountId(id)` and calls `service.getNftAllowances(request)`. [2](#0-1) 

`NftAllowanceServiceImpl.getNftAllowances()` calls `entityService.lookup(request.getAccountId())` unconditionally on line 28, after only validating query-parameter bounds (not the path variable type). [3](#0-2) 

`EntityServiceImpl.lookup()` pattern-matches on the parameter type. For `EntityIdAliasParameter`, it calls `entityRepository.findByAlias(p.alias())` — a direct DB call with no caching. [4](#0-3) 

`EntityRepository.findByAlias()` in the `rest-java` module executes a native SQL query with no `@Cacheable` annotation:

```sql
select id from entity where alias = ?1 and deleted <> true
``` [5](#0-4) 

**Why existing checks fail:**

1. **No caching in rest-java**: A `grep` for `@Cacheable`, `@CacheEvict`, `@Caching` across all `rest-java` Java sources returns zero matches. There is no `CacheConfiguration` class in the `rest-java` module (only in `grpc` and `importer`). The `web3` module's `EntityRepository` has `@Cacheable` on `findByAlias`-equivalent methods; the `rest-java` one does not. [6](#0-5) 

2. **No rate limiting in rest-java**: `ThrottleConfiguration` and `ThrottleManagerImpl` exist only in the `web3` module. The `rest-java` module has only a `MetricsFilter` (observability only, no throttling). [7](#0-6) 

3. **No authentication required**: The endpoint is publicly accessible per the OpenAPI spec (`/api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/nfts`), with no auth parameters. [8](#0-7) 

### Impact Explanation

Each alias-format request forces a full index scan (or at minimum an index lookup) on the `entity` table's `alias` column. Under concurrent load from multiple attackers or a single attacker with modest concurrency (e.g., 100–500 concurrent HTTP connections), the database CPU can be saturated. Because the mirror node's importer also reads/writes to the same PostgreSQL instance, DB CPU exhaustion degrades transaction ingestion and indexing — the core processing function of the mirror node. If multiple mirror node REST instances share one DB cluster, all are affected simultaneously. This matches the described severity: degradation of ≥30% of mirror node processing capacity without requiring brute-force credential attacks.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge of valid account IDs. The alias regex accepts any 40-character uppercase base32 string. A single attacker with a standard HTTP load tool (e.g., `wrk`, `ab`, `hey`) can generate thousands of requests per second. The alias lookup always hits the DB (no short-circuit for non-existent aliases at the application layer), and a miss still costs a full DB query. This is highly repeatable and requires no special tooling or insider knowledge.

### Recommendation

1. **Add caching to `EntityRepository.findByAlias()` and `findByEvmAddress()` in the `rest-java` module**, mirroring the pattern used in the `web3` module's `EntityRepository` with `@Cacheable`.
2. **Add a service-layer cache in `EntityServiceImpl.lookup()`** using a short TTL (e.g., 60 seconds) keyed on the alias bytes, so repeated lookups for the same alias (including non-existent ones) are served from memory.
3. **Implement rate limiting in `rest-java`** analogous to the `ThrottleConfiguration`/`ThrottleManagerImpl` in `web3`, applying a per-IP or global request-per-second cap on the allowances endpoint.
4. **Cache negative results** (alias not found) to prevent cache-miss amplification from non-existent aliases.

### Proof of Concept

```bash
# Generate a valid alias-format ID (40 uppercase base32 chars)
ALIAS="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"

# Flood the endpoint with alias-format account IDs (no auth required)
# Each request triggers: SELECT id FROM entity WHERE alias = ? AND deleted <> true
wrk -t 10 -c 200 -d 60s \
  "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/allowances/nfts"

# Vary the alias to defeat any hypothetical future caching and maximize DB load:
for i in $(seq 1 10000); do
  ALIAS=$(python3 -c "import base64, os; print(base64.b32encode(os.urandom(25)).decode().rstrip('='))")
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/allowances/nfts" &
done
wait
```

**Expected result**: Database CPU climbs to saturation; mirror node REST API response times degrade for all endpoints; importer lag increases as DB resources are starved.

```java
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
