### Title
Unauthenticated Sequential Account Enumeration via GraphQL `/graphql/alpha` Endpoint

### Summary
The `/graphql/alpha` GraphQL endpoint accepts `account(input: AccountInput!)` queries with `EntityIdInput` (shard/realm/num) without any authentication or per-IP rate limiting at the application layer. An unprivileged attacker can issue sequential POST requests incrementing `num` from 1 upward, distinguishing active from inactive accounts by the presence or absence of a populated response object, and thereby build a complete map of all active account entity IDs on the network.

### Finding Description

**Code path:**

- Schema: `graphql/src/main/resources/graphql/query.graphqls` line 5 — `account(input: AccountInput!): Account`
- Input type: `graphql/src/main/resources/graphql/common.graphqls` lines 169–178 — `EntityIdInput` with `num: Long! @Min(value: 0)` (no upper bound, no rate annotation)
- Controller: `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java` lines 33–44 — `account(@Argument @Valid AccountInput input)` directly calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)`
- Service: `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java` lines 24–26 — `entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type)` — returns `Optional.empty()` for non-existent IDs, which GraphQL serializes as `"account": null`

**Root cause:** No authentication filter, no per-IP rate limiter, and no request-per-second throttle exist anywhere in the `graphql` module. The only filters present are `LoggingFilter` (logging only) and the GraphQL instrumentation in `GraphQlConfiguration.java` lines 43–45 (`MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`), which are per-query structural guards, not request-rate controls.

**Why existing checks are insufficient:**

- `GraphQlConfiguration.java` lines 33–38: Parser limits (`maxCharacters`, `maxTokens`) apply per-query, not per-IP per-second.
- `charts/hedera-mirror-graphql/values.yaml` lines 138–142: `inFlightReq: amount: 5` with `ipStrategy: depth: 1` is a **concurrent connection** cap, not a requests-per-second rate limit. An attacker sending sequential (non-concurrent) requests is entirely unaffected.
- `charts/hedera-mirror-graphql/values.yaml` line 56: `maxRatePerEndpoint: 250` is a **global** backend rate limit, not per-IP. A single attacker can consume the entire budget.
- The `ThrottleConfiguration`/`ThrottleManagerImpl` in `web3/src/main/java/` apply exclusively to the `web3` module's contract-call endpoint, not to the GraphQL service.
- The `authHandler` in `rest/middleware/authHandler.js` applies only to the REST API (`/api/v1/`), not to `/graphql/alpha`.

### Impact Explanation
An attacker can enumerate the full set of active Hedera account entity IDs by iterating `num` from 1 to the current maximum (currently in the tens of millions range on mainnet). Each query returns either a populated account object (active) or `null` (inactive/non-existent). The resulting account map exposes: account balances, keys, memo fields, staking relationships, EVM addresses, and alias data — all without any credential. This constitutes a complete network reconnaissance capability. The gathered account list can be used to identify high-value targets for crafting targeted transactions (e.g., spam, allowance abuse, or social-engineering attacks against specific accounts).

### Likelihood Explanation
Preconditions: none — any internet-connected client can reach the endpoint. The attack is trivially scriptable with a simple loop. The `inFlightReq: 5` limit means sequential enumeration (one request at a time) is completely unrestricted. At even a conservative 10 requests/second, an attacker can enumerate 1 million accounts in ~28 hours with no authentication, no CAPTCHA, and no IP ban triggered by the application.

### Recommendation
1. **Add per-IP rate limiting** at the application layer in the GraphQL Spring Boot service (e.g., Bucket4j with an IP-keyed `ConcurrentHashMap<String, Bucket>` in a `OncePerRequestFilter`, similar to the pattern already used in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`).
2. **Change `inFlightReq`** to a true rate-limit middleware (e.g., Traefik `rateLimit` with `average` and `burst` per source IP) rather than a concurrent-connection cap.
3. Consider requiring an API key or token for the GraphQL endpoint, consistent with how the REST API handles elevated-limit users via `authHandler`.
4. Apply a minimum response delay or query cost throttle tied to the `entityId.num` lookup path specifically.

### Proof of Concept
```bash
# Sequential enumeration — no credentials required
for num in $(seq 1 1000); do
  curl -s -X POST https://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { entityId: { shard: 0, realm: 0, num: $num } }) { entityId { num } balance memo key } }\"}" \
    | jq -c --arg n "$num" 'if .data.account != null then {num: $n, data: .data.account} else empty end'
done
```
A non-null response body at a given `num` confirms an active account. No rate-limit error is returned for sequential requests. The `inFlightReq: 5` Traefik middleware does not trigger because only one request is in flight at a time. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
