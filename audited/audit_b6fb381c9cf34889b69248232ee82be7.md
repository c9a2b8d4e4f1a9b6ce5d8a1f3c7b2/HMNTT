### Title
Unauthenticated DB I/O Exhaustion via Repeated `getLatestFileDataContents` Calls on `/contracts/results/:transactionIdOrHash`

### Summary
Any unauthenticated external user can send concurrent GET requests to `/api/v1/contracts/results/{ethHash}` using a valid Ethereum transaction hash whose stored `callDataId` references a large file. Each request unconditionally triggers `FileDataService.getLatestFileDataContents()`, which executes an expensive `string_agg` CTE query that reads and concatenates all file-data chunks for that file from the database. With no rate limiting on this endpoint and an optional/short-lived response cache, an attacker can flood the database with these heavy queries, exhausting DB I/O and connection-pool slots across mirror-node deployments.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 21 registers the public, unauthenticated route:
```
router.getExt('/results/:transactionIdOrHash', ContractController.getContractResultsByTransactionIdOrHash);
```

`rest/controllers/contractController.js` lines 1183–1185 unconditionally fire the expensive DB call whenever `callDataId` passes the trivial `isValidUserFileId` check:
```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

`rest/utils.js` lines 445–447 show the guard is only a numeric range check (entity num > 1000); it imposes no bound on file size or call frequency:
```js
const isValidUserFileId = (val) => {
  return !isNil(val) && val !== '' && EntityId.parse(val).num > 1000;
};
```

`rest/service/fileDataService.js` lines 18–41 and 84–87 show `getLatestFileDataContents` executes a two-stage CTE that first finds the latest create/update timestamp, then runs `string_agg(file_data, '' ORDER BY consensus_timestamp)` to concatenate every file-data chunk since that timestamp into a single result. For a large call-data file (Hedera supports up to ~1 MB per file, split across many `FileAppend` transactions), this query reads and aggregates many rows on every invocation.

**Why existing checks fail:**

1. **No rate limiting** — `rest/server.js` applies no rate-limiting middleware to the contracts route. The `authHandler` only sets a custom pagination *limit* for authenticated users; it does not throttle request frequency for unauthenticated callers.
2. **Response cache is optional and short-lived** — `applicationCacheEnabled` requires both `config.cache.response.enabled` and `config.redis.enabled` to be true. The default Redis TTL is `DEFAULT_REDIS_EXPIRY = 1` second (`rest/middleware/responseCacheHandler.js` line 24). Even when enabled, concurrent requests arriving within the same second all miss the cache and hit the DB simultaneously. Different hashes bypass the cache entirely.
3. **`isValidUserFileId` is not a size guard** — it only validates the entity-ID format; it does not inspect or limit the size of the referenced file.

### Impact Explanation

Each concurrent request causes the PostgreSQL backend to execute the `string_agg` CTE against the `file_data` table. For a file composed of many `FileAppend` chunks (realistic for large Ethereum call-data payloads), this query performs a sequential scan or index range scan over many rows, aggregates them in memory, and returns a large byte payload. Flooding with hundreds of concurrent requests against the same (or multiple) large-file hashes saturates DB I/O bandwidth, fills the connection pool, and causes query-queue buildup. This degrades or denies service to all mirror-node API consumers and can affect 30%+ of deployed mirror-node instances sharing the same database cluster, matching the stated severity threshold.

### Likelihood Explanation

Preconditions are minimal for an unprivileged attacker:
- Ethereum transaction hashes are publicly visible on the Hedera network and in the mirror-node transaction history API.
- Transactions with large `callDataId` references are a documented Hedera pattern (call-data offloading for large EVM payloads).
- No credentials, keys, or special access are required.
- A single attacker with a standard HTTP client (e.g., `curl`, `ab`, `wrk`) can issue hundreds of concurrent requests per second.
- The attack is repeatable and stateless; the attacker does not need to maintain any session.

### Recommendation

1. **Add per-IP or global rate limiting** on the `/contracts/results/:transactionIdOrHash` endpoint (e.g., via `express-rate-limit` or an ingress-level policy) to bound the number of requests per second that trigger `getLatestFileDataContents`.
2. **Cache the `getLatestFileDataContents` result** at the application layer (keyed by `callDataId`) with a TTL longer than 1 second, independent of the optional Redis response cache, so repeated requests for the same file do not re-execute the `string_agg` query.
3. **Enforce a file-size cap** in `isValidUserFileId` or in `getLatestFileDataContents` by adding a `LIMIT` or `WHERE length(file_data) <= MAX_BYTES` guard to the CTE, preventing unbounded aggregation.
4. **Add a DB query timeout** specific to this code path so a single slow `string_agg` query cannot hold a connection indefinitely.

### Proof of Concept

1. Query the mirror-node transactions API to find an Ethereum transaction with a large `callDataId`:
   ```
   GET /api/v1/transactions?type=ETHEREUMTRANSACTION&limit=25
   ```
   Identify a `transaction_hash` whose corresponding `ethereum_transaction` row has a non-null `call_data_id` with entity num > 1000.

2. Confirm the file is large by checking the `file_data` table row count for that entity (or observe a slow response on first request).

3. Launch concurrent flood:
   ```bash
   wrk -t 10 -c 200 -d 30s \
     "https://<mirror-node>/api/v1/contracts/results/0x<eth_hash>"
   ```
