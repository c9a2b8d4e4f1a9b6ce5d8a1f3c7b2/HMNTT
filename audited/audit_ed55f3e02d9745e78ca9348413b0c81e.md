### Title
Uncached Alias Lookups with No Rate Limiting Enable DB Connection Pool Exhaustion via Repeated `getAccountFromAlias()` Queries

### Summary
`getAccountFromAlias()` in `rest/service/entityService.js` issues a direct database query on every invocation with no caching and no request-rate limiting. An unprivileged attacker can flood the REST API with alias-lookup requests — including syntactically distinct base32 strings that decode to the same binary value via trailing "don't-care" bits — causing the finite `pg` connection pool to be exhausted and denying service to legitimate users.

### Finding Description

**Exact code path:**

`rest/service/entityService.js` lines 42–43:
```js
async getAccountFromAlias(accountAlias) {
  const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
```
Every HTTP request that resolves an account by alias reaches this function and acquires a connection from the pool. There is no in-process cache, no Redis cache, and no deduplication of concurrent identical queries.

**Root cause — failed assumption:**
The design assumes that alias lookups are infrequent or that an upstream layer (reverse proxy, CDN, rate limiter) will throttle abusive callers. Neither assumption is enforced in the codebase. `rest/middleware/authHandler.js` only adjusts the *response result count* for authenticated users; it does not rate-limit unauthenticated requests at all.

**Base32 normalization — don't-care bits:**
`rest/accountAlias.js` line 10 enforces the regex `[A-Z2-7]+`, which correctly blocks the `loose`-mode substitutions (`0→O`, `1→L`, `8→B`) and lowercase variants. However, it does **not** block the trailing "don't-care" bit variants that are inherent to base32 encoding. A 32-byte public-key alias encodes to 52 base32 characters; the last character has 4 don't-care bits, yielding **16 distinct valid base32 strings** (e.g., `…A`, `…B`, `…C` … `…P` in the last position) that all pass the regex and all decode to the identical binary value via `base32.parse(str, {loose: true})`. Each variant is treated as a unique input and triggers a separate DB query because the cache key is the decoded `Uint8Array` stored in `accountAlias.alias`, but there is no cache at all.

**Connection pool:**
`rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`. The default is a small integer (validated to be `> 0`). When all connections are occupied by alias-lookup queries that are waiting on the database, new legitimate requests queue indefinitely or time out.

**Existing checks reviewed and shown insufficient:**
- `accountAliasRegex` validates character set only — no rate limit.
- `authHandler.js` sets a per-user *result limit*, not a *request rate limit*.
- No Redis/in-memory cache exists in `rest/service/entityService.js` (contrast with the Java `importer` service which has a Caffeine alias cache).
- `statement_timeout` in the pool config limits individual query duration but does not prevent connection exhaustion from concurrent requests.

### Impact Explanation
A fully unauthenticated attacker can exhaust the `pg` connection pool, causing all subsequent database-backed API endpoints (transactions, balances, tokens, etc.) to fail with connection-timeout errors. This is a complete denial of service for the REST mirror-node API with no authentication prerequisite. Severity: **High**.

### Likelihood Explanation
The attack requires only an HTTP client capable of sending concurrent GET requests to any alias-parameterized endpoint (e.g., `/api/v1/accounts/<alias>`). No credentials, no on-chain funds, and no special knowledge are needed. The 16 don't-care-bit variants per alias allow an attacker to rotate through syntactically distinct strings, complicating simple string-equality-based deduplication if it were ever added. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add an in-process LRU cache** in `EntityService.getAccountFromAlias()` keyed on the decoded binary alias (`accountAlias.alias`), mirroring the Caffeine cache used in the Java importer service.
2. **Normalize before caching/querying**: canonicalize the base32 string by re-encoding the decoded bytes (`base32.encode(base32.decode(input))`) so all don't-care-bit variants map to a single cache key.
3. **Add request-rate limiting** (e.g., `express-rate-limit`) for unauthenticated callers on alias-lookup routes.
4. **Increase `maxConnections`** or add a query-queue depth limit to bound the blast radius.

### Proof of Concept

For a known 32-byte alias whose canonical base32 encoding ends in `A` (e.g., `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZA`), generate 16 variants by replacing the last character with `A`–`P`:

```
AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZA   → decodes to 0x000102...
AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZE   → same binary
AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZC   → same binary
... (16 total)
```

All pass `accountAliasRegex`. Send thousands of concurrent requests cycling through these variants:

```bash
for i in $(seq 1 5000); do
  ALIAS=$(python3 -c "import sys; chars='ABCDEFGHIJKLMNOP'; print('AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ' + chars[$i % 16])")
  curl -s "https://<mirror-node>/api/v1/accounts/$ALIAS" &
done
wait
```

**Expected result:** The `pg` pool reaches `maxConnections`; subsequent legitimate API calls return connection-timeout errors or HTTP 500 responses.
