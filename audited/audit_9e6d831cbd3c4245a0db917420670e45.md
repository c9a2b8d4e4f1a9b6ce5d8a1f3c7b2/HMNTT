### Title
Unbounded `node.id` Parameter List Allows CPU-Wasting Griefing in `getNetworkNodes()`

### Summary
`NetworkNodeRequest.nodeIds` has no `@Size` constraint, allowing an unauthenticated caller to submit an arbitrarily large number of `node.id=eq:X` query parameters. `getNetworkNodes()` iterates every submitted parameter unconditionally, performing a `HashSet.add()` per entry, so duplicate values are deduped but the O(n) loop still executes for every submitted parameter. The only practical bound is the HTTP server's query-string length limit, not application-level validation.

### Finding Description
**Code location**: `rest-java/src/main/java/org/hiero/mirror/restjava/dto/NetworkNodeRequest.java` line 38 and `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java` lines 110–118.

`NetworkNodeRequest` declares:
```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();   // ← no @Size limit
``` [1](#0-0) 

The sibling DTO `RegisteredNodesRequest` applies `@Size(max = 2)` to its equivalent list, proving the pattern is known and intentionally applied elsewhere:
```java
@Size(max = 2)
private List<NumberRangeParameter> registeredNodeIds = List.of();
``` [2](#0-1) 

`RequestParameterArgumentResolver.validateAndAddParameter` detects that `nodeIds` is a `Collection` and passes all submitted values through without any count check: [3](#0-2) 

`getNetworkNodes()` then iterates every element unconditionally:
```java
for (final var nodeIdParam : nodeIdParams) {
    if (nodeIdParam.operator() == RangeOperator.EQ) {
        nodeIds.add(nodeIdParam.value());   // HashSet dedupes, but loop still runs N times
    } ...
}
``` [4](#0-3) 

**Root cause / failed assumption**: The developer assumed the HTTP layer would bound the number of repeated parameters. No application-level guard exists on `nodeIds`, unlike `registeredNodeIds`.

**Exploit flow**:
1. Attacker crafts a GET request to `/api/v1/network/nodes` with thousands of identical `node.id=eq:1` parameters (e.g., filling the server's maximum query-string buffer).
2. `RequestParameterArgumentResolver` collects all values into the `nodeIds` list with no size check.
3. `getNetworkNodes()` iterates every entry, calling `HashSet.add(1L)` N times. Each call is O(1) but string parsing (`NumberRangeParameter.valueOf`) and object allocation happen per entry.
4. The HashSet ends up with a single element; the database query is trivial. All CPU cost is in the unbounded Java loop.

### Impact Explanation
Each request forces the server to parse, allocate, and hash-insert one `NumberRangeParameter` object per submitted parameter. With Tomcat's default 8 KB query-string limit, roughly 600 parameters fit (`node.id=eq:1&` ≈ 13 bytes each). The per-request CPU overhead is small but non-zero, and the endpoint is unauthenticated and publicly reachable. Sustained parallel requests amplify the effect. No economic damage to network users occurs; the impact is server-side CPU griefing only, consistent with the stated scope.

### Likelihood Explanation
Any unauthenticated HTTP client can trigger this with a single crafted request. No credentials, tokens, or special network position are required. The attack is trivially repeatable and scriptable. The only natural throttle is the HTTP query-string length limit imposed by the reverse proxy or application server, which is a configuration detail outside the application's control.

### Recommendation
Add a `@Size` constraint to `NetworkNodeRequest.nodeIds`, mirroring the pattern already used in `RegisteredNodesRequest`:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 10)   // or whatever the intended maximum is
private List<NumberRangeParameter> nodeIds = List.of();
```

The `RequestParameterArgumentResolver` already invokes `binder.validate()`, so the constraint will be enforced automatically and will produce a `400 Bad Request` response before `getNetworkNodes()` is called. [5](#0-4) 

### Proof of Concept
```bash
# Generate a query string with 600 duplicate EQ parameters
PARAMS=$(python3 -c "print('&'.join(['node.id=eq:1']*600))")
curl -v "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS"
```
Expected: the server accepts the request, iterates 600 `HashSet.add(1L)` calls, and returns a normal response — demonstrating unbounded per-request work with no application-level rejection. Contrast with `/api/v1/network/registered-nodes?registerednode.id=1&registerednode.id=2&registerednode.id=3`, which returns `400 Bad Request` due to the `@Size(max = 2)` guard. [6](#0-5)
