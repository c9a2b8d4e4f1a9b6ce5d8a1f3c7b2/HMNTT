### Title
Unbounded Infinite gRPC Stream Exhaustion via `subscribeTopic()` with `limit=0` and No `endTime`

### Summary
Any unauthenticated external caller can open an unlimited number of never-terminating gRPC streams by invoking `subscribeTopic()` with `limit=0` and no `consensusEndTime`. The only server-side guard (`maxConcurrentCallsPerConnection=5`) is scoped per-connection, not globally, so an attacker opening many TCP connections bypasses it entirely. This exhausts the server's reactive subscription pipeline, thread pool, and database connection pool, starving legitimate downstream consumers — including smart contract event relayers that depend on the mirror node HCS stream for contract log delivery.

### Finding Description

**Code path and root cause:**

`ConsensusController.subscribeTopic()` at [1](#0-0)  performs no authentication, no per-caller connection count check, and no global subscriber cap before delegating to `TopicMessageServiceImpl.subscribeTopic()`.

`TopicMessageFilter` accepts `limit=0` as valid (constraint is `@Min(0)`) and `endTime=null` as valid (field is nullable): [2](#0-1) 

`hasLimit()` returns `false` when `limit == 0`: [3](#0-2) 

In `TopicMessageServiceImpl.subscribeTopic()`, when `hasLimit()` is false the `.take()` operator is never applied, and when `endTime` is null `pastEndTime()` returns `Flux.never()` — the stream is permanently open: [4](#0-3) [5](#0-4) 

`subscriberCount` is incremented on subscribe and decremented on finalize, but it is only a Micrometer gauge metric — there is no enforcement gate that rejects new subscriptions when a threshold is reached: [6](#0-5) 

**Why the existing check fails:**

`maxConcurrentCallsPerConnection=5` is applied per TCP connection only: [7](#0-6) [8](#0-7) 

An attacker opening `N` TCP connections gets `5N` concurrent infinite streams. There is no global cap, no IP-based rate limit, and no authentication on the gRPC port. The `web3` module's `ThrottleConfiguration` (bucket4j) does not apply to the `grpc` module at all.

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (default gRPC port, plaintext by default per `tls.enabled: false` in Helm values).
2. On each connection, sends 5 concurrent `subscribeTopic` RPCs: `{"topicID": {"topicNum": <valid_topic>}, "limit": 0}` — no `consensusEndTime`.
3. Each stream enters the `historical → live → safetyCheck` pipeline, holds a Reactor `Disposable`, a DB polling loop via `topicMessageRetriever`, and a Redis/POLL listener slot.
4. Streams never terminate (no limit, no end time, `Flux.never()` for `pastEndTime`).
5. With sufficient `N`, the bounded-elastic scheduler, HikariCP DB connection pool, and Redis subscription slots are exhausted.
6. Legitimate subscribers (smart contract event relayers, bridge operators) receive `RESOURCE_EXHAUSTED` or timeout errors and lose their HCS event stream.

### Impact Explanation
Downstream smart contract event consumers that subscribe to HCS topics via this gRPC API (e.g., EVM event relayers, cross-chain bridge operators) lose their stream. Missed HCS messages mean missed contract log events, causing stale on-chain state, failed bridge proofs, or incorrect smart contract behavior. The mirror node gRPC service becomes unavailable to all legitimate users. This matches the stated scope: "unintended smart contract behavior with no concrete funds at direct risk" — a Medium-severity DoS on the event delivery layer.

### Likelihood Explanation
No privileges are required. The attack requires only network access to port 5600 and knowledge of one valid topic ID (publicly discoverable via the REST API). It is trivially scriptable with `grpcurl` or any gRPC client library. The `maxConcurrentCallsPerConnection=5` default makes the per-connection limit appear protective but is easily bypassed by opening many connections. The attack is repeatable and sustained until the attacker closes connections.

### Recommendation
1. **Enforce a global subscriber cap**: Add a check in `TopicMessageServiceImpl.subscribeTopic()` that rejects new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., `hiero.mirror.grpc.maxSubscribers`), returning `RESOURCE_EXHAUSTED`.
2. **Add per-IP connection rate limiting**: Apply a gRPC `ServerInterceptor` that tracks and limits concurrent streams per source IP.
3. **Require a minimum limit or maximum stream duration**: Reject requests where `limit=0` AND `endTime=null` (i.e., explicitly unbounded streams), or enforce a server-side maximum stream lifetime.
4. **Apply the existing `maxConcurrentCallsPerConnection` in conjunction with a global connection limit** via `NettyServerBuilder.maxConnectionAge()` / `maxConnectionIdle()` to force stream recycling.

### Proof of Concept
```bash
# Open 200 connections × 5 streams = 1000 infinite subscriptions
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: subscriberCount metric climbs to 1000 and stays there
# Legitimate subscriber now receives RESOURCE_EXHAUSTED or hangs indefinitely
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "limit": 10}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected result: no response / connection refused / DB pool exhausted error in server logs
```
