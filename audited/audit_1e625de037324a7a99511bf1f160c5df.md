### Title
Unbounded Database Query Amplification via Alias Lookup with No Rate Limiting or Negative-Result Caching

### Summary
`getAccountIdFromAlias()` in `rest/service/entityService.js` unconditionally delegates to `getAccountFromAlias()`, which executes a full database query on every invocation regardless of whether the alias exists. When called with `requireResult=false`, the function silently returns `null` on miss, suppressing any error signal that could trigger a circuit breaker. The Node.js REST service has no application-level rate limiting and the response cache explicitly excludes non-2xx responses, meaning every request for a valid-format but non-existent alias hits the database with no throttle or cache protection.

### Finding Description

**Exact code path:**

`rest/service/entityService.js` lines 71–81 (`getAccountIdFromAlias`) unconditionally calls `getAccountFromAlias` (lines 42–53), which always executes `entityFromAliasQuery` (lines 17–20) via `super.getRows(...)`:

```js
// lines 71-81
async getAccountIdFromAlias(accountAlias, requireResult = true) {
  const entity = await this.getAccountFromAlias(accountAlias); // DB query always fires
  if (isNil(entity)) {
    if (requireResult) {
      throw new NotFoundError(EntityService.missingAccountAlias);
    }
    return null; // silent null, no error emitted
  }
  return entity.id;
}
```

The query executed on every call:
```sql
select id from entity
where coalesce(deleted, false) <> true
  and alias = $1
```

**Root cause — three compounding failures:**

1. **No negative-result caching.** The `responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` line 95 only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 (alias not found, `requireResult=true`) or a silent null path (`requireResult=false`) never populates the cache, so every repeat request for the same non-existent alias re-executes the DB query.

2. **No application-level rate limiting.** `rest/server.js` registers only: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, and optional `responseCacheCheckHandler`. There is no `express-rate-limit` or equivalent middleware. The `authHandler` (`rest/middleware/authHandler.js` lines 15–36) only applies custom limits to authenticated users; unauthenticated callers are entirely unrestricted.

3. **`requireResult=false` suppresses error signals.** When callers use `requireResult=false` (e.g., `getEncodedId` at line 126 propagates this flag), the function returns `null` instead of throwing. This means the HTTP response may still be 200 OK (the outer handler decides), so even a Traefik circuit breaker keyed on 5xx or 4xx error ratios would not trip. The `rest-java` chart's circuit breaker (`ResponseCodeRatio(500, 600, 0, 600) > 0.25`) would not help here either, and that breaker applies to a different service entirely.

**Exploit flow:**

An attacker sends a high-rate stream of `GET /api/v1/accounts/{alias}` requests using valid base32-encoded aliases that do not correspond to any account. Each request passes `AccountAlias.isValid()` validation, reaches `getEncodedId` → `getAccountIdFromAlias` → `getAccountFromAlias` → DB query. No cache hit occurs (404s not cached). No rate limit fires. The database connection pool is saturated.

### Impact Explanation

Sustained database connection pool exhaustion degrades or halts all mirror node REST API responses, not just the alias endpoint. The mirror node serves as critical read infrastructure for wallets, explorers, and dApps querying Hedera network state. A single attacker with a modest HTTP request rate (hundreds of requests/second from a single IP) can monopolize the PostgreSQL connection pool, causing timeouts across all endpoints. This constitutes a non-network DoS against a node that, on mainnet, represents significant market-cap-weighted infrastructure.

### Likelihood Explanation

No authentication, no CAPTCHA, no rate limit, and no cost is imposed on the attacker. The alias format is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml` line 1536: `^(\d{1,5}\.){0,2}(?:[A-Z2-7]{8})*...`), making it trivial to generate valid-format non-existent aliases programmatically. The attack is repeatable, requires no special knowledge, and can be executed from a single host. The `requireResult=false` path specifically ensures no error-rate signal is emitted, defeating any error-ratio-based circuit breaker.

### Recommendation

1. **Add application-level rate limiting** to the Node.js REST service (e.g., `express-rate-limit` keyed by IP) before the accounts route in `rest/server.js`.
2. **Cache negative alias lookups** in `getAccountFromAlias`: store a short-TTL sentinel for known-missing aliases to avoid repeated DB hits for the same alias.
3. **Add Traefik `inFlightReq` and `rateLimit` middleware** to the `hedera-mirror-rest` Helm chart, mirroring the protections already present in the rosetta chart (`charts/hedera-mirror-rosetta/values.yaml` lines 152–161).
4. **Add a DB-query-level timeout** in `getRows` so a flood of alias queries cannot hold connections indefinitely.
