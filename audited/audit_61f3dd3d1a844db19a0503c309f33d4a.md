### Title
Unauthenticated Block Number DoS via Uncached Index Scans on Non-Existent Block Numbers

### Summary
The `getByHashOrNumber()` handler in `rest/controllers/blockController.js` accepts any syntactically valid positive long as a block number without bounding it against the current chain height. Each such request unconditionally executes a database index scan (`WHERE index = $1`) via `RecordFileService.getByHashOrNumber()`, and because 404 responses are never cached and no application-level rate limiting exists on the REST API blocks endpoint, an unauthenticated attacker can flood the service with unique out-of-range block numbers, exhausting the database connection pool and degrading service availability.

### Finding Description

**Code path:**

`rest/controllers/blockController.js:28-38` — `validateHashOrNumber()` calls `utils.isPositiveLong(hashOrNumber, true)`, which only validates that the input is a syntactically valid non-negative 64-bit integer. There is no check against the current chain tip. [1](#0-0) 

`rest/controllers/blockController.js:114-124` — `getByHashOrNumber()` passes the validated number directly to `RecordFileService.getByHashOrNumber(hash, number)` with no further bounds check. [2](#0-1) 

`rest/service/recordFileService.js:164-179` — `getByHashOrNumber()` constructs and executes `WHERE index = $1` against the `record_file` table. When the number exceeds the chain height, the query performs a B-tree index scan on `record_file__index` and returns no row. [3](#0-2) 

**Root cause — failed assumption:** The code assumes that syntactic validity of the block number is sufficient input validation. It does not assume that callers will only supply numbers within the current chain range, but it also provides no mechanism to short-circuit the DB round-trip when the number is provably out of range.

**Why the cache does not help:** `responseCacheUpdateHandler` only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 (thrown by `NotFoundError` when `block` is null) is never cached. Every unique out-of-range block number therefore produces a unique cache miss and a fresh DB query. [4](#0-3) 

**Why rate limiting does not help:** The `ThrottleManagerImpl` and its bucket-based rate limiter exist only in the `web3` (contract-call) service. There is no application-level rate limiting middleware in the REST API for the blocks endpoint. [5](#0-4) 

The database index on `record_file(index)` is a non-unique B-tree index. [6](#0-5) 

### Impact Explanation

Each request for a unique non-existent block number (e.g., `9223372036854775800`, `9223372036854775799`, …) bypasses the response cache and issues a synchronous DB query. Under sustained flood conditions this:

1. Exhausts the Node.js service's database connection pool, causing legitimate requests to queue or time out.
2. Increases PostgreSQL CPU and I/O load from repeated index traversals.
3. Slows or blocks legitimate block-lookup and transaction-lookup queries that share the same connection pool.

The importer (block ingestion) is a separate JVM process and PostgreSQL MVCC prevents read queries from blocking writes directly, so a hard "partition from ingestion" requires extreme pool exhaustion. However, REST API availability degradation is achievable at moderate request rates.

### Likelihood Explanation

- **Preconditions:** None. The endpoint is public and unauthenticated.
- **Attacker capability:** Any HTTP client. A single machine with a scripted loop of `GET /api/v1/blocks/<incrementing_large_number>` suffices.
- **Repeatability:** Trivially repeatable; the attack space is 2^63 unique URLs, all guaranteed cache misses.
- **Detection difficulty:** Requests look syntactically valid (positive integers) and each returns a normal 404, making them hard to distinguish from legitimate "block not found" queries at the WAF/proxy level without rate-limiting heuristics.

### Recommendation

1. **Cache negative results:** Store 404 responses in the response cache with a short TTL (e.g., 5–10 seconds). This converts repeated identical requests into cache hits.
2. **Validate against chain height:** Before issuing the DB query, fetch (and cache) the current maximum `index` from `record_file` and reject requests where `number > maxIndex` with a 404 without hitting the DB.
3. **Add application-level rate limiting:** Apply a per-IP or global rate limiter (e.g., `express-rate-limit`) to the REST API, particularly the `/api/v1/blocks/:hashOrNumber` route.
4. **Infrastructure-level protection:** Apply Traefik (or equivalent) `rateLimit` and `inFlightReq` middleware to the REST API ingress, mirroring the configuration already present for the Rosetta service. [7](#0-6) 

### Proof of Concept

```bash
# Flood with unique out-of-range block numbers (no auth required)
for i in $(seq 9223372036854770000 9223372036854775000); do
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/blocks/$i" &
done
wait
```

Each request:
1. Passes `isPositiveLong` validation (syntactically valid).
2. Reaches `RecordFileService.getByHashOrNumber(null, "$i")`.
3. Executes `SELECT ... FROM record_file WHERE index = $i` — index scan, no row found.
4. Returns HTTP 404 — **not cached**.
5. Next request for `$i+1` repeats from step 2.

Observable result: DB connection pool saturation, rising query latency for legitimate requests, potential HTTP 503 responses from the REST API under sustained load.
