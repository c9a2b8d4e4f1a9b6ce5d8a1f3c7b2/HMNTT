### Title
`parseInt('NaN')` Fallback Silently Zeroes `api_request_size_bytes` Histogram via Attacker-Controlled `Content-Length` Header

### Summary
In `metricsMiddleware()`, the `content-length` header is parsed with `parseInt(req.headers['content-length'] ?? '0', 10) || 0`. The nullish-coalescing guard (`??`) only catches `null`/`undefined`; it does not reject non-numeric strings such as `'NaN'`. `parseInt('NaN', 10)` returns `NaN`, which is falsy, so `|| 0` silently substitutes `0`. Any unprivileged client can therefore force every request-size observation into the zero bucket, permanently corrupting the `api_request_size_bytes` histogram.

### Finding Description
**File:** `rest/middleware/metricsHandler.js`
**Function:** `metricsMiddleware()` (returned by `metricsHandler()`)
**Line:** 209

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

Execution path when `Content-Length: NaN` is sent:

1. `req.headers['content-length']` → `'NaN'` (string, not null/undefined)
2. `'NaN' ?? '0'` → `'NaN'` (nullish coalescing does not fire)
3. `parseInt('NaN', 10)` → `NaN`
4. `NaN || 0` → `0`
5. `requestSizeHistogram.record(0, labels)` — observation recorded as zero bytes

The `??` operator was intended to supply a safe default when the header is absent, but it provides no protection against a header that is present but non-numeric. There is no `isNaN()`, `Number.isFinite()`, or regex guard anywhere in the path.

### Impact Explanation
The `api_request_size_bytes` OpenTelemetry histogram (Prometheus-exported) will show every request as having a zero-byte body whenever an attacker includes `Content-Length: NaN`. Because the histogram is cumulative, a sustained flood of such requests drowns out legitimate observations, making the metric useless for capacity planning, anomaly detection, and size-based alerting. The corruption is silent — no error is logged, no exception is thrown, and the metric continues to appear healthy (observations are still counted; only their values are wrong).

### Likelihood Explanation
Preconditions: none. Any HTTP client can set an arbitrary `Content-Length` header value. The attack requires no authentication, no special network position, and no knowledge of the application beyond the public API endpoint. It is trivially scriptable and repeatable at any request rate.

### Recommendation
Replace the unsafe `parseInt` + `||` pattern with an explicit numeric validation:

```js
const rawCL = req.headers['content-length'];
const parsedCL = rawCL !== undefined ? Number(rawCL) : 0;
const requestBytes = Number.isFinite(parsedCL) && parsedCL >= 0 ? parsedCL : 0;
requestSizeHistogram.record(requestBytes, labels);
```

`Number.isFinite()` rejects `NaN`, `Infinity`, and `-Infinity`, ensuring only valid non-negative integers are recorded.

### Proof of Concept
```bash
# Send a POST request with Content-Length set to the string "NaN"
curl -X POST https://<mirror-node-rest>/api/v1/transactions \
  -H "Content-Length: NaN" \
  -H "Content-Type: application/json" \
  -d '{"dummy":"payload"}'

# Scrape Prometheus metrics and observe:
# api_request_size_bytes_bucket{le="..."} increments only in the le="0" bucket
# regardless of actual body size — confirming the zero-byte recording.
```

Repeat at volume to saturate the histogram with zero-observations, rendering size-based dashboards and alerts inoperative. [1](#0-0)
