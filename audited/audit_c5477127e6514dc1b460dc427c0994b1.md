### Title
Unauthenticated DoS via Unbounded DB Queries on GET /api/v1/blocks/{hashOrNumber}

### Summary
The `getByHashOrNumber` handler in `rest/controllers/blockController.js` accepts any valid-format block number or hash from unauthenticated callers and unconditionally issues a live database query per request with no application-level rate limiting. An attacker flooding this endpoint with valid-format but non-existent block numbers can exhaust the REST API's shared DB connection pool, causing service degradation and, in single-DB deployments, contending with the importer's write path.

### Finding Description

**Code path:**

`rest/controllers/blockController.js` lines 114–124:
```js
getByHashOrNumber = async (req, res) => {
  utils.validateReq(req);
  const {hash, number} = validateHashOrNumber(req.params.hashOrNumber);
  const block = await RecordFileService.getByHashOrNumber(hash, number);
  if (!block) { throw new NotFoundError(); }
  ...
};
``` [1](#0-0) 

`validateHashOrNumber` (lines 28–38) only validates the *format* of the input — it accepts any integer ≥ 0 or any 64/96-char hex string. It performs no existence pre-check and imposes no rate limit. [2](#0-1) 

`RecordFileService.getByHashOrNumber` (lines 164–179) then unconditionally executes:
```sql
SELECT ... FROM record_file WHERE index = $1
```
for number inputs, or:
```sql
SELECT ... FROM record_file WHERE hash LIKE $1
```
for hash inputs — one live DB round-trip per HTTP request, with no result caching. [3](#0-2) 

**Root cause — failed assumption:** The design assumes that infrastructure-level controls (e.g., GCP gateway `maxRatePerEndpoint: 250`) are always present. The application itself has zero rate-limiting middleware for this endpoint. [4](#0-3) 

`rest/server.js` registers no rate-limiting middleware between the auth handler and the block routes. The `authHandler` only sets a custom query limit for *authenticated* users; unauthenticated requests pass through unconditionally. [5](#0-4) [6](#0-5) 

The DB connection pool (`rest/dbpool.js`) is bounded by `config.db.pool.maxConnections`. Once all connections are held by in-flight block queries, new requests queue or fail with a connection-timeout error, making the REST API unavailable. [7](#0-6) 

The throttle configuration that does exist (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) applies exclusively to the **web3** contract-call path, not to the REST API blocks endpoint. [8](#0-7) 

**On the "index scan" claim:** The `record_file__index` index (non-unique B-tree on `index`) and `record_file__hash` index (unique B-tree on `hash collate "C"`) mean each query is an efficient index lookup, not a sequential scan. The per-query I/O cost is low. The attack surface is therefore **connection pool exhaustion** (finite `maxConnections`) rather than raw I/O saturation of the storage layer. [9](#0-8) [10](#0-9) 

### Impact Explanation

In deployments where the REST API and importer share the same PostgreSQL instance (common in self-hosted or development setups), exhausting the REST API connection pool degrades REST API availability. In the worst case, if the DB server's total connection limit is reached, the importer's own connection pool can also be starved, delaying or blocking record file ingestion and causing the mirror node to fall behind the live network. In production deployments with a dedicated read replica for the REST API, the importer write path is isolated and unaffected, reducing the impact to REST API DoS only.

### Likelihood Explanation

No authentication, API key, or proof-of-work is required. Any external actor with HTTP access can issue requests. Valid-format block numbers (e.g., sequential integers `0`, `1`, `2`, …) are trivially generated. A single machine with a modest HTTP client library can sustain thousands of requests per second. The attack is repeatable and stateless. The only mitigations are infrastructure-level (GCP gateway, Traefik), which are absent in default/self-hosted deployments.

### Recommendation

1. **Add application-level rate limiting** in `rest/server.js` for the blocks route (e.g., `express-rate-limit` keyed by IP), independent of infrastructure controls.
2. **Add a response cache** for `getByHashOrNumber` results (a short TTL cache keyed on the normalized hash/number). Cache misses for non-existent blocks can be negative-cached with a short TTL to prevent repeated DB hits for the same non-existent key.
3. **Set a conservative `maxConnections`** in `config.db.pool` and ensure the REST API uses a read replica separate from the importer's primary DB connection.
4. Do not rely solely on infrastructure-level rate limiting as the only defense.

### Proof of Concept

```bash
# Flood with valid-format but non-existent block numbers
# No authentication required
seq 1 100000 | xargs -P 200 -I{} \
  curl -s -o /dev/null "https://<mirror-node>/api/v1/blocks/{}"
```

Preconditions: Network access to the REST API port; no API key or credentials needed.

Trigger: 200+ concurrent connections each issuing requests for non-existent block numbers.

Result: DB connection pool (`maxConnections`) exhausted; subsequent legitimate requests receive connection-timeout errors; REST API becomes unavailable. In shared-DB deployments, importer connection acquisition may also be delayed.

  async getByHashOrNumber(hash, number) {
    let whereStatement = '';
    const params = [];
    if (hash) {
      hash = hash.toLowerCase();
      whereStatement += `${RecordFile.HASH} like $1`;
      params.push(hash + '%');
    } else {
      whereStatement += `${RecordFile.INDEX} = $1`;
      params.push(number);
    }

    const query = `${RecordFileService.blocksQuery} where ${whereStatement}`;
    const row = await super.getSingleRow(query, params);
    return row ? new RecordFile(row) : null;
  }
