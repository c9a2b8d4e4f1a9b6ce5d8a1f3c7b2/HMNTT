### Title
206 Partial Content Responses Cached as Success, Enabling Silent Incomplete Record Consumption

### Summary
When a contract result record exists in the database but `callResult` is `null` (mid-ingestion state), the mirror node returns HTTP 206 with a `ContractResultDetails` body containing null critical fields. The `isSuccess` helper in `constants.js` treats 206 as a success code, causing the response cache middleware to store the incomplete response. Any unprivileged user who queries a transaction during the ingestion window poisons the cache, causing all subsequent clients to silently receive incomplete records until the TTL expires.

### Finding Description

**Root cause — `isSuccess` includes 206:**

In `rest/constants.js` lines 155 and 159, `PARTIAL_CONTENT` is defined as code 206, and `isSuccess` returns `true` for any code in `[200, 300)`:

```js
PARTIAL_CONTENT: new StatusCode(206, 'Partial mirror node'),
isSuccess: (code) => code >= 200 && code < 300,
``` [1](#0-0) 

**Trigger — 206 set when `callResult` is null:**

In `getContractResultsByTimestamp` (`contractController.js:1026-1030`) and `getContractResultsByTransactionIdOrHash` (`contractController.js:1198-1202`), when `contractResult.callResult` is `null`, the handler sets the status to 206 and still calls `setContractResultsResponse`, which writes a `ContractResultDetailsViewModel` with null/empty fields (`call_result: "0x"`, `gas_used: null`, etc.) into `res.locals[responseDataLabel]`:

```js
if (isNil(contractResults[0].callResult)) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
}
this.setContractResultsResponse(res, contractResults[0], ...);
``` [2](#0-1) [3](#0-2) 

**Cache poisoning — 206 stored as success:**

`responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js:95` caches any response where `httpStatusCodes.isSuccess(res.statusCode)` is true. Since 206 satisfies `isSuccess`, the incomplete partial response is written to the cache under the normal request key:

```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
  // ... stores res.statusCode (206) and body with null fields into cache
  const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, ...);
  await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
}
``` [4](#0-3) 

**Cache replay — subsequent clients get 206 with null fields:**

`responseCacheCheckHandler` replays the cached `statusCode` (206) and body verbatim to all subsequent requesters:

```js
const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
// ...
res.status(statusCode);
res.send(body);
``` [5](#0-4) 

**No `Content-Range` header is set** — a grep across `rest/**/*.js` finds zero occurrences of `Content-Range`. Standard HTTP 206 for range responses requires this header; its absence makes the partial status invisible to middleware that inspects headers rather than the status code.

**Same pattern exists for NFT transaction history** in `rest/tokens.js:1048-1051`: [6](#0-5) 

### Impact Explanation

A downstream EVM-compatible client (JSON-RPC relay, block explorer, DeFi indexer) that checks `status >= 200 && status < 300` will accept the 206 response as a complete record. The returned `ContractResultDetails` body has `call_result: "0x"`, `gas_used: null`, `gas_consumed: null`, `status: "0x0"` (because `transactionResult` may map to a failure code when the record is incomplete), and other null fields. An indexer consuming this may:
- Record a contract call as failed when it succeeded
- Store an empty `call_result` permanently
- Miscompute gas analytics

Because the incomplete response is cached, the window of impact extends beyond the transient ingestion race — every client hitting the same endpoint during the cache TTL receives the poisoned response even after the record is fully ingested.

### Likelihood Explanation

The precondition (a transaction record exists with `callResult = null`) is a normal, observable state during mirror node ingestion. An attacker does not need any credentials or special access — only knowledge of a recent transaction hash or ID, which is publicly visible on-chain. The attacker can poll the endpoint immediately after submitting or observing a transaction, race the ingestion window, and poison the cache. This is repeatable for every new transaction. The NFT history endpoint (`/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions`) is similarly exploitable when `nft.createdTimestamp == null`.

### Recommendation

1. **Exclude 206 from caching**: In `responseCacheUpdateHandler`, change the condition to only cache `200`-range responses that are not partial:
   ```js
   if (responseBody && responseCacheKey &&
       (isUnmodified || (httpStatusCodes.isSuccess(res.statusCode) && res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code))) {
   ```

2. **Add a `Content-Range` or custom header**: Set a response header (e.g., `X-Partial-Content: true`) on all 206 responses so clients that inspect headers can detect incompleteness without relying on the status code alone.

3. **Consider returning 404 or 202 instead of 206**: HTTP 206 is semantically defined for range requests. A `202 Accepted` (record exists but processing is incomplete) or `404` (record not yet available) would be less likely to be silently accepted by 2xx-checking clients and would not be cached by standard HTTP caches.

### Proof of Concept

```
# Step 1: Submit a contract call transaction and note its hash (TX_HASH).
# Step 2: Immediately after the transaction appears in the DB (but before full ingestion):

curl -v "https://<mirror-node>/api/v1/contracts/results/TX_HASH"
# Expected: HTTP 206, body contains call_result: "0x", gas_used: null

# Step 3: Wait for full ingestion (call_result populated in DB).
# Step 4: Query again:

curl -v "https://<mirror-node>/api/v1/contracts/results/TX_HASH"
# Actual: HTTP 206 served from cache, still with call_result: "0x", gas_used: null
# Expected: HTTP 200 with complete record

# Step 5: A client checking only for 2xx:
response = fetch("https://<mirror-node>/api/v1/contracts/results/TX_HASH")
if response.status >= 200 and response.status < 300:
    process(response.json())  # silently processes incomplete record
```
