### Title
Reflected User Input in GraphQL Error Messages via `decodeEvmAddress()`

### Summary
In `GraphQlUtils.decodeEvmAddress()`, when hex decoding fails, the raw (post-prefix-strip) user-supplied `evmAddress` string is concatenated directly into an `IllegalArgumentException` message. `CustomExceptionResolver` catches `IllegalArgumentException` and returns `ex.getMessage()` verbatim as the GraphQL error `message` field, reflecting arbitrary attacker-controlled text to any caller with no authentication required.

### Finding Description

**Code path:**

1. `GraphQlUtils.java`, lines 87–98 — `decodeEvmAddress()`: [1](#0-0) 
   After stripping the `0x` prefix, the remaining user string is passed to `Hex.decodeHex()`. On failure, the catch block constructs:
   ```java
   throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
   ```
   where `evmAddress` is the attacker-controlled post-strip value.

2. `EntityServiceImpl.java`, lines 34–35 — `getByEvmAddressAndType()`: [2](#0-1) 
   Calls `decodeEvmAddress()` directly with the GraphQL argument value.

3. `CustomExceptionResolver.java`, lines 17–25 — `resolveToSingleError()`: [3](#0-2) 
   Catches `IllegalArgumentException` and calls `.message(ex.getMessage())` — no sanitization, no truncation, no encoding — placing the raw attacker string into the GraphQL JSON error response.

**Root cause:** No sanitization or length-limiting of user input before embedding it in the exception message, combined with a resolver that blindly forwards exception messages to clients.

**Why existing checks fail:** The only "check" is the `DecoderException` catch, which is the trigger, not a guard. `CustomExceptionResolver` applies no filtering to the message content of `IllegalArgumentException`.

### Impact Explanation

- **Log injection:** Newline characters (`\n`, `\r`) in the input survive the `removeStart` call and the `Hex.decodeHex` failure path, reaching the exception message. If the application logs errors (standard Spring behavior), an attacker can forge structured log entries, polluting audit trails.
- **Reflected content in API responses:** Any downstream client (web frontend, mobile app, third-party integration) that renders the GraphQL `errors[].message` field in an HTML context without escaping is exposed to stored/reflected XSS. The server provides no defense-in-depth here.
- **Information leakage pattern confirmation:** Echoing input confirms the endpoint is live and processing the field, aiding enumeration.

Severity: **Low–Medium**. JSON serialization escapes characters at the transport layer, preventing direct JSON-breaking injection, but log injection and client-side rendering risks are real.

### Likelihood Explanation

- No authentication or privilege is required; the GraphQL endpoint is public.
- The trigger is trivially reproducible: any non-hex string as `evmAddress` fires the path.
- Fully repeatable and automatable.

### Recommendation

1. **Do not embed raw user input in exception messages.** Replace:
   ```java
   throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
   ```
   with a static message:
   ```java
   throw new IllegalArgumentException("Invalid evmAddress format");
   ```
2. **Sanitize/truncate in `CustomExceptionResolver`** as a defense-in-depth measure before calling `.message(...)`.
3. **Validate input length and character set** before attempting decode (e.g., reject inputs exceeding 42 characters or containing non-hex/non-`0x` characters early).

### Proof of Concept

Send the following GraphQL query to the public endpoint:

```graphql
{
  account(input: { evmAddress: "INJECTED_TEXT_<script>alert(1)</script>" }) {
    id
  }
}
```

**Expected response:**
```json
{
  "errors": [
    {
      "message": "Unable to decode evmAddress: INJECTED_TEXT_<script>alert(1)</script>",
      "locations": [...],
      "path": [...],
      "extensions": { "classification": "ValidationError" }
    }
  ]
}
```

The attacker-controlled string appears verbatim in the `message` field. Substituting `\nFAKE_LOG_ENTRY` demonstrates log injection in server logs.
