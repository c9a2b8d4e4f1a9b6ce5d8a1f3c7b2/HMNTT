### Title
Unbounded DB Queries on EVM Address Lookups Due to Missing Negative-Result Cache in `getEncodedId()`

### Summary
`EntityService.getEncodedId()` in `rest/service/entityService.js` routes any input matching `evmAddressShardRealmRegex` (including plain 40-hex-char addresses and `0.0.<40hex>` forms) to `getEntityIdFromEvmAddress()`, which unconditionally executes a database query. The `parseCached` LRU in `entityId.js` only caches the parsed `EntityId` object, not the DB lookup result, so every request — including repeated requests with the same non-existent address — hits the database. No rate limiting exists at the application layer. An unprivileged attacker can sustain arbitrary DB load across all endpoints that call `getEncodedId()`.

### Finding Description

**Exact code path:**

`getEncodedId()` (`rest/service/entityService.js`, lines 118–137):
```
isValidEntityId(entityIdString)   // true for any 40-hex or 0.0.<40hex> input
  → EntityId.parseString(...)     // calls parseCached — caches EntityId object only
  → entityId.evmAddress !== null  // true for opaque EVM addresses
  → getEntityIdFromEvmAddress(entityId, requireResult)  // always hits DB
```

`getEntityIdFromEvmAddress()` (`rest/service/entityService.js`, lines 90–104):
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
  [Buffer.from(entityId.evmAddress, 'hex')]);
// No caching of result — null or otherwise
```

`parseCached()` (`rest/entityId.js`, lines 314–332):
```js
const value = cache.get(key);
if (value) { return value; }          // cache hit: returns EntityId
...
cache.set(key, entityId);             // caches EntityId with evmAddress set
return entityId;
```

**Root cause:** `parseCached` caches the `EntityId` object (with `evmAddress` populated). On a cache hit, it returns the cached object immediately — but `getEncodedId` then checks `entityId.evmAddress !== null` and calls `getEntityIdFromEvmAddress` regardless. The DB lookup result (positive or negative) is never cached anywhere. The `parseCached` cache only eliminates the regex/parse overhead, not the DB round-trip.

**Why existing checks fail:**
- `parseCached` LRU (`rest/entityId.js` lines 301–303): caches parse result, not DB result.
- `evmAddressShardRealmRegex` (`rest/entityId.js` line 39): `/^(\d{1,4}\.)?(\d{1,5}\.)?[A-Fa-f0-9]{40}$/` — accepts both plain `<40hex>` and `0.0.<40hex>` forms, both of which pass `isValidEntityId` and reach the DB path.
- `parseFromString` shard/realm guard (`rest/entityId.js` lines 246–247): only throws for non-system shard/realm (e.g., `1.2.<40hex>`); `0.0.<40hex>` and bare `<40hex>` pass through to the opaque-EVM-address branch and return `evmAddress != null`.
- No HTTP-layer rate limiting is present in the application code.

### Impact Explanation
Every endpoint that calls `EntityService.getEncodedId()` is affected: `accounts.js`, `balances.js`, `tokens.js`, `topicmessage.js`, `transactions.js`, `schedules.js`, and multiple controllers (`accountController`, `contractController`, `cryptoAllowanceController`, `tokenAllowanceController`, `tokenController`). An attacker can target any of these with a stream of requests containing arbitrary 40-hex strings. Each request executes `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` against the database. At sufficient request rate, this exhausts DB connection pool capacity or CPU, causing degraded or failed responses for all users — a non-network application-layer DoS.

### Likelihood Explanation
No authentication or rate limiting is required. Any HTTP client can send requests to public REST endpoints. The attacker needs only to know (or discover) one endpoint that accepts an account/contract/token ID parameter — trivially found from public API documentation. The attack is repeatable, stateless, and requires no special knowledge beyond the 40-hex address format. Rotating through unique addresses defeats even the parse-layer cache.

### Recommendation
Cache the DB lookup result for EVM addresses, including negative (null) results, with a bounded TTL. A dedicated in-process cache (e.g., a second `quickLru` instance) keyed on the hex address string should be added inside `getEntityIdFromEvmAddress()` or at the `getEncodedId()` call site. Additionally, deploy HTTP-layer rate limiting (e.g., per-IP request throttling via a reverse proxy or Express middleware such as `express-rate-limit`) to bound the request rate before it reaches the DB layer.

### Proof of Concept
```bash
# Target any endpoint that accepts an account/contract ID, e.g.:
# GET /api/v1/accounts/{id}

# Generate and send requests with random non-existent EVM addresses
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ADDR}" &
done
wait

# Each request triggers:
# SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
# with a unique address, no DB result is cached, DB is queried 10000 times.

# Alternatively, repeat the SAME address to confirm parse-cache does not help:
ADDR="71eaa748d5252be68c1185588beca495459fdba4"
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ADDR}" &
done
wait
# parseCached returns cached EntityId on iterations 2+,
# but getEntityIdFromEvmAddress still queries DB every time.
```
