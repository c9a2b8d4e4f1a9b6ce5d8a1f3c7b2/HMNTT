### Title
Unauthenticated IP Spoofing via Trusted `X-Real-IP` Header in `getClientIpAddress()` Enables Log Poisoning for `/construction/submit` Forensics

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header with no proxy validation, allowlist, or sanitization. Any external caller can inject an arbitrary IP into application logs by setting `X-Real-IP` to a spoofed value while setting `X-Forwarded-For` to a different value, creating an irreconcilable discrepancy between application logs and upstream proxy/network logs. This directly undermines post-incident forensic correlation of fraudulent `/construction/submit` transactions.

### Finding Description
**Exact code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64 — blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }

    return ipAddress
}
```

**Root cause:** The function gives unconditional priority to `X-Real-IP` with no check that the request originated from a trusted reverse proxy. There is no trusted-proxy allowlist, no header stripping, and no format validation. The value is taken verbatim from `r.Header.Get(xRealIpHeader)` and written directly into the log message at line 52–53 of `TracingMiddleware`.

**Failed assumption:** The code assumes `X-Real-IP` is always set by a trusted upstream proxy. In practice, the server binds directly to a port (`httpServer.ListenAndServe()`, `main.go` line 231) with no enforced proxy layer. Even when a proxy exists, many configurations pass through client-supplied headers rather than overwriting them.

**Exploit flow:**
1. Attacker sends `POST /construction/submit` with headers:
   - `X-Real-IP: 192.168.1.1` (spoofed — e.g., an internal trusted IP)
   - `X-Forwarded-For: 203.0.113.99` (attacker's real IP or another decoy)
2. `getClientIpAddress()` returns `192.168.1.1` (line 64 wins).
3. `TracingMiddleware` logs: `192.168.1.1 POST /construction/submit (200) in Xms`.
4. The proxy/network layer logs the real connection IP or `X-Forwarded-For` value (`203.0.113.99`).
5. The two log sources are now irreconcilable — no single IP appears in both.

**Why existing checks fail:** The test at `trace_test.go` lines 52–54 explicitly confirms and validates this behavior — a client-supplied `X-Real-IP` is accepted and logged without any validation. There is no middleware, no configuration option, and no code path that strips or validates these headers before `getClientIpAddress()` is called.

### Impact Explanation
An attacker submitting fraudulent transactions via `/construction/submit` can make their true origin unattributable in application logs. Investigators correlating application logs with proxy/firewall/network logs will find conflicting IPs, breaking the forensic chain of custody. Since `/construction/submit` is the transaction broadcast endpoint, this directly impedes attribution of on-chain fund movements to a real-world IP address. The spoofed IP can also be set to an internal/trusted address to further mislead triage.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request with custom headers — a capability available to any unprivileged external user. No authentication, special access, or network position is required. The attack is trivially repeatable, leaves no reliable trace, and requires zero tooling beyond `curl`. Deployments where the Rosetta API port is reachable (directly or via a proxy that forwards client headers) are fully exposed.

### Recommendation
1. **Maintain a trusted proxy list:** Only accept `X-Real-IP` / `X-Forwarded-For` if `r.RemoteAddr` matches a configured set of trusted proxy CIDRs. Fall back to `r.RemoteAddr` for all other connections.
2. **Prefer `r.RemoteAddr` as ground truth** for security-sensitive logging; treat proxy headers as supplementary metadata only.
3. **Validate header format:** At minimum, parse and validate that the header value is a valid IP address before using it (reject strings like `"attacker-controlled-string"`).
4. Consider using a well-audited library (e.g., `realip` middleware from `chi` or similar) that implements trusted-proxy semantics correctly.

### Proof of Concept
```bash
# Attacker submits a transaction with a spoofed X-Real-IP
curl -X POST http://<rosetta-host>:<port>/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 10.0.0.1" \
  -H "X-Forwarded-For: 203.0.113.99" \
  -d '{"network_identifier": {...}, "signed_transaction": "..."}'

# Application log will record:
# 10.0.0.1 POST /construction/submit (200) in Xms

# Proxy/network log records the real connection or X-Forwarded-For:
# 203.0.113.99 -> /construction/submit

# The two logs cannot be correlated to the same request by IP.
```
