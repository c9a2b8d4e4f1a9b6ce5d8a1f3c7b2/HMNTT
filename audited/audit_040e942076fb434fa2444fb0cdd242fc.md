### Title
Unauthenticated GraphQL Introspection Enabled — Full Schema Enumeration Without Authentication

### Summary
The GraphQL endpoint exposes full schema introspection to any unauthenticated external user. Spring Boot's `spring-graphql` enables introspection by default, and no configuration in this codebase disables it or gates it behind authentication. This allows an attacker to enumerate all types, fields, directives, scalars, and input structures without any credentials.

**Root cause:** `GraphQlConfiguration.java` configures only `MaxQueryComplexityInstrumentation(200)`, `MaxQueryDepthInstrumentation(10)`, and parser token/character limits. [3](#0-2)  None of these controls disable or restrict introspection. Spring Boot's `spring-graphql` sets `spring.graphql.schema.introspection.enabled=true` by default, and no `application.yml`/`application.properties` override exists in `graphql/src/main/resources/` — only schema `.graphqls` files and `banner.txt` are present.  No `SecurityConfig`, `WebSecurityConfig`, or `HttpSecurity` bean exists anywhere in the graphql module to require authentication on the endpoint.

**Why existing checks fail:**
- `MaxQueryDepthInstrumentation(10)`: A standard introspection query `{__schema{types{name}}}` has depth ~3, well under the limit of 10. [4](#0-3) 
- `MaxQueryComplexityInstrumentation(200)`: Basic introspection queries score far below 200. [5](#0-4) 
- Parser limits (`maxTokens(1000)`, `maxCharacters(10000)`): A minimal introspection query uses ~20 tokens and ~50 characters. [6](#0-5) 

### Impact Explanation
A successful introspection query returns the complete schema: all query/mutation types, every field name and type, all input object structures, custom scalars, and directives. For this API, that exposes `AccountInput` fields (`alias`, `evmAddress`, `entityId`, `id`), validation constraints, and all viewmodel structures. This directly enables targeted fuzzing, injection attempts, and business-logic abuse with zero guesswork.

### Likelihood Explanation
This requires zero authentication, zero special tooling, and zero prior knowledge. Any HTTP client (curl, Burp Suite, Altair, Postman) can send a single POST request. It is trivially repeatable and automatable. GraphQL introspection abuse is one of the most commonly exploited GraphQL misconfigurations in the wild.

### Recommendation
Disable introspection in production by adding to the application configuration:

```yaml
spring:
  graphql:
    schema:
      introspection:
        enabled: false
```

If introspection must remain available for internal tooling, gate it behind authentication (e.g., require an internal network source IP or a valid API key via a Spring Security `HttpSecurity` configuration on the `/graphql` endpoint).

### Proof of Concept
```bash
curl -X POST https://<host>/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{__schema{types{name fields{name type{name kind}}}}}"}'
```

**Preconditions:** None — no authentication, no special headers required.  
**Trigger:** Send the above request to the `/graphql` endpoint.  
**Result:** The server returns a full JSON document listing every type (`Account`, `AccountInput`, `HbarUnit`, etc.), every field name and its type, and all input structures defined in the schema files under `graphql/src/main/resources/graphql/`.
