### Title
Unauthenticated DB Connection Pool Exhaustion via `getTokenRelationships()` Sequential Multi-Query Pipeline

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` executes 2–3 sequential database queries per request with no rate limiting, no concurrency cap, and no authentication requirement. The REST API's default DB connection pool is only 10 connections. An unprivileged attacker sending many concurrent requests can hold all pool connections simultaneously, starving legitimate requests and causing a non-network-based DoS.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–92: [1](#0-0) 

Each request executes the following sequential DB queries:

**Query 1** — `EntityService.isValidAccount()` always fires `entityExistenceQuery` against the `entity` table: [2](#0-1) 

**Query 2** — `TokenService.getTokenAccounts()` always fires `tokenRelationshipsQuery` against `token_account`: [3](#0-2) 

**Query 3** — `getCachedTokens()` fires `tokenCacheQuery` against the `token` table for any token IDs not in the in-process LRU cache: [4](#0-3) 

The cache is a single in-process `quickLru` instance. For a cold cache, or when the attacker uses accounts with distinct token associations, all three queries fire per request.

**Root cause — no rate limiting on the REST API.** The throttle/rate-limit infrastructure exists only in the `web3` module: [5](#0-4) 

There is no equivalent middleware in the `rest/` module. The route is registered with no authentication or concurrency guard: [6](#0-5) 

**Root cause — tiny default DB connection pool.** The pool is configured via `pg` with `max: config.db.pool.maxConnections`, whose documented default is **10**: [7](#0-6) [8](#0-7) 

With `statement_timeout` defaulting to 20 000 ms, each query can hold a connection for up to 20 seconds: [9](#0-8) 

### Impact Explanation

With 10 pool connections and 3 sequential queries per request, only ~3–4 concurrent requests are needed to saturate the pool. Once saturated, every subsequent request blocks until `connectionTimeoutMillis` (default 20 000 ms) elapses, then fails. This affects **all** REST API endpoints sharing the same pool, not just the token endpoint. The impact is a complete REST API outage for the duration of the attack.

### Likelihood Explanation

The endpoint requires zero authentication and zero privileges. The account ID parameter accepts any valid numeric entity ID (e.g., `0.0.98`), which is publicly known. An attacker needs only a standard HTTP client capable of sending concurrent requests (e.g., `curl`, `ab`, `wrk`). The attack is trivially repeatable and requires no special knowledge of the system beyond the public API documentation.

### Recommendation

1. **Add rate limiting to the REST API** — implement a per-IP or global request-rate limiter (e.g., `express-rate-limit`) on the `rest/` Express application, mirroring the bucket4j throttle used in the `web3` module.
2. **Increase the default pool size** or document a minimum recommended value for production deployments.
3. **Reduce `statement_timeout`** for this endpoint class to limit how long a single query can hold a connection.
4. **Parallelize the two independent queries** — `isValidAccount` and `getTokenAccounts` are independent; running them with `Promise.all` would halve connection hold time.

### Proof of Concept

```bash
# Precondition: a valid account ID exists (e.g., 0.0.98 on any network)
# Send 50 concurrent requests to exhaust the 10-connection pool

for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/0.0.98/tokens?limit=100" &
done
wait

# Result: requests begin timing out with connection pool exhaustion errors
# after the first ~10 concurrent requests saturate the pool.
# All other REST API endpoints also become unresponsive.
```

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
