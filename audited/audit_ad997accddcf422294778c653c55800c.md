### Title
Unauthenticated Concurrent DB Connection Pool Exhaustion via `GET /accounts/:id` with `transactions=true`

### Summary
`getOneAccount()` in `rest/accounts.js` unconditionally fires two concurrent database queries per request via `Promise.all([entityPromise, transactionsPromise])` with no per-request concurrency control or rate limiting. The default connection pool size is 10. An unprivileged attacker sending as few as 5 simultaneous requests with varying timestamp filters can exhaust the pool, blocking all REST API endpoints that require a DB connection and effectively partitioning the REST layer from the database.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount()`, lines 487–495:

```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);          // line 487 — acquires 1 pool connection
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)             // line 492 — acquires 1+ pool connections
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]); // line 495
```

`includeTransactions` defaults to `true` (line 402: `let includeTransactions = true`), so every request without an explicit `transactions=false` takes the expensive path.

`doGetTransactions` in `rest/transactions.js` (lines 695–722) itself makes two sequential DB calls:
1. `getTransactionTimestamps` → `pool.queryQuietly(query, params)` (line 483)
2. `getTransactionsDetails` → `pool.queryQuietly(query, params)` (line 662)

At the moment `Promise.all` is awaited, the entity query and the first transaction-timestamps query run **concurrently**, consuming **2 pool connections simultaneously** per request.

**Pool configuration:**

`rest/dbpool.js` lines 7–16 sets `max: config.db.pool.maxConnections`. The documented default is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, `docs/configuration.md` line 556). `connectionTimeoutMillis` defaults to 20,000 ms and `statement_timeout` to 20,000 ms.

**Root cause / failed assumption:**

The design assumes that the DB connection pool is large enough relative to concurrent request volume. There is no concurrency limiter, no per-IP rate limiter, and no request queue depth guard on the Node.js REST server (`rest/server.js` lines 67–144 shows only `authHandler`, `requestLogger`, optional metrics, and optional Redis cache — no throttle middleware for this endpoint). The `authHandler` only adjusts response-size limits for authenticated users; it does not rate-limit unauthenticated callers.

**Why the Redis cache does not help:**

The response cache (`responseCacheCheckHandler`) is keyed on the full request URL. An attacker using distinct `timestamp` query parameters (e.g., `timestamp=gte:1000000000`, `timestamp=gte:1000000001`, …) generates unique cache keys, bypassing the cache entirely and forcing a fresh DB round-trip for every request.

### Impact Explanation

With a default pool of 10 connections and 2 connections consumed concurrently per request:
- **5 simultaneous attacker requests** saturate the pool.
- All subsequent requests from any client (including legitimate users and health checks) queue waiting up to 20 seconds for a free connection.
- If the attacker sustains the flood, the queue never drains; new requests time out with a connection-acquisition error, making the entire REST API unavailable.
- This is a **full REST-to-DB partition**: every endpoint (`/transactions`, `/balances`, `/tokens`, etc.) shares the same pool and is equally affected.

Severity: **High** — complete service denial achievable by an unauthenticated external user with no special knowledge.

### Likelihood Explanation

- **No authentication required.** The endpoint is public.
- **No rate limiting.** The Node.js REST server has no per-IP or global request-rate throttle for `/accounts/:id`.
- **Trivially scriptable.** A single shell loop or any HTTP load tool suffices.
- **Timestamp variation is easy.** The attacker simply increments the `timestamp` parameter to defeat caching.
- **Repeatable indefinitely.** The attacker can sustain the attack as long as desired.
- **Low cost.** Each request is a cheap HTTP GET; the cost is entirely borne by the server.

### Recommendation

1. **Add a concurrency / rate limiter** on the REST API layer (e.g., `express-rate-limit` or an in-flight-request counter) specifically for DB-heavy endpoints. The Rosetta chart already demonstrates this pattern (`inFlightReq.amount: 5`, `rateLimit.average: 10` in `charts/hedera-mirror-rosetta/values.yaml` lines 152–161); apply equivalent middleware to the Node.js REST service.
2. **Increase the default pool size** or document a production-appropriate value. A pool of 10 is far too small for a public API.
3. **Serialize the two queries** in `getOneAccount` — run the entity query first; only if it returns a result, run the transactions query. This halves peak connection consumption per request and also avoids wasted transaction queries for non-existent accounts.
4. **Add a global `statement_timeout`** at the PostgreSQL role level for `mirror_api` as a backstop, independent of the application-level setting.

### Proof of Concept

```bash
# Exhaust the default 10-connection pool with 6 concurrent requests,
# each using a unique timestamp to bypass the Redis response cache.
for i in $(seq 1 6); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.2?transactions=true&timestamp=gte:$((1000000000 + i))" \
    -o /dev/null &
done
wait

# All subsequent requests now queue or fail with a connection-timeout error:
curl -v "http://<mirror-node-host>/api/v1/accounts/0.0.3"
# Expected: delayed response or 500 error due to pool exhaustion
```

Sustain the loop to maintain the denial-of-service condition for the full duration of the attack.
