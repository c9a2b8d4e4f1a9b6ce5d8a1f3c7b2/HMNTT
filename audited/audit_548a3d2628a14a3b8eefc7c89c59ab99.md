### Title
Unbounded Concurrent Subscriptions with Epoch startTime Enable Heap Exhaustion via Repeated poll() Page Loads

### Summary
Any unauthenticated gRPC client can open an unlimited number of `subscribeTopic` streams with `startTime=0` (epoch) and no `limit`, causing `PollingTopicMessageRetriever.poll()` to repeatedly materialize up to 1,000 JDBC rows per subscription per poll cycle. Because no concurrent-subscription cap is enforced and the only guard is a 60-second timeout, an attacker who continuously opens new streams can sustain heap pressure that starves legitimate subscribers of JVM memory and DB connections.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` converts the raw gRPC request into a `TopicMessageFilter` and calls `topicMessageService.subscribeTopic(filter)` with no authentication gate. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true) and tracks subscribers only as a metric — there is no enforcement of a maximum subscriber count. [2](#0-1) 

`TopicMessageFilter` validates `startTime` only as `startTime <= DomainUtils.now()`, so `startTime=0` (Unix epoch, year 1970) is accepted. [3](#0-2) 

`PollingTopicMessageRetriever.poll()` computes `pageSize = Math.min(limit, maxPageSize)`. With no limit set, `limit = Integer.MAX_VALUE`, so `pageSize = maxPageSize` (1,000 for the throttled path). It then calls `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))`, which causes the underlying JDBC driver to fetch and materialize all `pageSize` rows into heap memory. [4](#0-3) 

`PollingContext.isComplete()` on the throttled path returns `false` as long as the last page was full (`pageSize.get() >= maxPageSize`). With a very old `startTime` and a large historical dataset, every page is full, so polling continues until the 60-second timeout fires. [5](#0-4) 

The throttled polling frequency is 2 seconds, giving ~30 poll cycles × 1,000 rows = 30,000 rows materialized per subscription over its 60-second lifetime. [6](#0-5) 

**Root cause / failed assumption:** The design assumes that the 60-second timeout and the per-page size cap are sufficient resource bounds. They are not, because there is no cap on the number of concurrent subscriptions, so N attackers (or one attacker with N connections) multiply the per-subscription cost by N with no server-side enforcement.

### Impact Explanation

Each `TopicMessage` carries a `message` payload of up to 6 KB plus metadata. At 1,000 rows per poll and a 2-second polling interval, a single malicious subscription holds ~6 MB of live objects in the JVM heap at any moment and issues ~30 DB queries over its lifetime. With 200 concurrent attacker subscriptions the instantaneous heap pressure is ~1.2 GB and the DB connection pool is saturated with concurrent JDBC queries. This causes `OutOfMemoryError` or GC thrashing in the gRPC JVM, which delays or drops message delivery to all legitimate HCS subscribers — directly preventing gossip of transactions to those subscribers.

### Likelihood Explanation

The attack requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and knowledge of any valid `topicId`. No credentials, tokens, or privileged network position are needed. The attacker needs only to open many long-lived streams in a loop. The attack is trivially repeatable and scriptable, and the 60-second timeout means the attacker must sustain ~N/60 new connections per second to keep N streams alive — a very low rate for a scripted client.

### Recommendation

1. **Enforce a per-IP or global concurrent-subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 500).
2. **Enforce a minimum `startTime` age** in `TopicMessageFilter.isValidStartTime()`: reject `startTime` values older than a configurable horizon (e.g., 7 days before `DomainUtils.now()`).
3. **Require a non-zero `limit`** or cap the effective limit to a server-side maximum when no limit is provided, so each subscription has a finite total row budget.
4. **Use cursor-based / server-side streaming JDBC** in `TopicMessageRepositoryCustom.findByFilter()` to avoid materializing the full page into heap at once.
5. Consider adding **rate limiting per client IP** at the gRPC interceptor layer.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw grpc stubs
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs_grpc
from proto import basic_types_pb2, timestamp_pb2

MIRROR_HOST = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic

def open_flood_subscription():
    channel = grpc.insecure_channel(MIRROR_HOST)
    stub = cs_grpc.ConsensusServiceStub(channel)
    query = cs.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD,
                                        realmNum=TOPIC_REALM,
                                        topicNum=TOPIC_NUM),
        # startTime = Unix epoch (year 1970) — passes isValidStartTime() check
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no consensusEndTime, no limit → infinite polling
    )
    try:
        for _ in stub.subscribeTopic(query):  # blocks, consuming server resources
            pass
    except Exception:
        pass

# Open 300 concurrent subscriptions
threads = [threading.Thread(target=open_flood_subscription) for _ in range(300)]
for t in threads:
    t.start()
time.sleep(55)  # re-open before 60s timeout expires
# Repeat to maintain sustained pressure
```

**Expected result:** Mirror node gRPC JVM heap climbs to OOM or sustained GC pause; legitimate subscribers stop receiving `ConsensusTopicResponse` messages; DB connection pool exhausted with concurrent `findByFilter` queries.

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
