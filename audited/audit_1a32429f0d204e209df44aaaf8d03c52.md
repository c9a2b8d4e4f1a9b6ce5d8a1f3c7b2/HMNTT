### Title
Blocking JDBC Calls on gRPC Server Thread Pool in `getNodes()` Enable Unauthenticated DoS

### Summary
`NetworkController.getNodes()` subscribes a Reactor pipeline with no `subscribeOn()` or `publishOn()` operator, causing three synchronous JDBC calls and the first database page fetch to execute directly on the gRPC server's `applicationTaskExecutor` thread. Because the endpoint requires no authentication and per-connection concurrency is only limited to 5 (not total connections), an unprivileged attacker can open many connections and continuously flood the endpoint to saturate the thread pool and degrade or deny service.

### Finding Description

**Code path:**

`NetworkController.getNodes()` ( [1](#0-0) ) builds a pipeline and calls `.subscribe()` with no scheduler operator. The calling thread is a gRPC server thread drawn from `applicationTaskExecutor` ( [2](#0-1) ).

`.flatMapMany(networkService::getNodes)` invokes `NetworkServiceImpl.getNodes()` synchronously on that same thread. Inside that method, **before any Flux is returned**, three blocking JDBC calls execute on the gRPC server thread:

```
addressBookRepository.findLatestTimestamp(...)       // line 61-63
nodeStakeRepository.findLatestTimestamp()            // line 64
nodeStakeRepository.findAllStakeByConsensusTimestamp(...)  // line 65
``` [3](#0-2) 

The returned `Flux.defer(() -> page(context))` is then subscribed on the same gRPC thread, so the **first** call to `page()` — which executes a transactional JDBC query via `transactionOperations.execute()` — also blocks the gRPC server thread: [4](#0-3) 

The `withScheduler(Schedulers.boundedElastic())` in `repeatWhen` only schedules the *delay between repeat cycles* on `boundedElastic`; subsequent pages after the first do run off the gRPC thread, but the initial three DB calls and the first page always block it: [5](#0-4) 

**Root cause:** The failed assumption is that the Reactor pipeline is non-blocking. It is not — `NetworkServiceImpl.getNodes()` performs eager, synchronous JDBC I/O before returning the `Flux`, and no `subscribeOn()` is present to redirect that work off the gRPC server thread.

**Why existing checks are insufficient:**

`maxConcurrentCallsPerConnection` is set to 5, which limits concurrency *per connection*, not across connections: [6](#0-5) 

There is no connection-rate limit, no IP-based throttle, and no authentication on the `getNodes` RPC: [7](#0-6) 

### Impact Explanation
Each `getNodes()` call holds a gRPC server thread blocked for the duration of 3 JDBC round-trips plus one paginated query (typically 50–200 ms under load). An attacker opening *N* connections and sending 5 concurrent requests per connection pins `5N` threads. Spring Boot's `ThreadPoolTaskExecutor` will create new threads up to its configured maximum; beyond that, requests queue or are rejected. Sustained flooding causes thread-pool saturation, request queuing, latency spikes, and eventual service unavailability for all gRPC consumers — including legitimate Hedera network participants querying the address book.

### Likelihood Explanation
The exploit requires only a gRPC client library (freely available) and the ability to open TCP connections to port 5600. No credentials, tokens, or special protocol knowledge are needed. The attack is trivially scriptable: open hundreds of HTTP/2 connections, each sending 5 concurrent `getNodes` RPCs in a tight loop. The address book file IDs (`0.0.101` / `0.0.102`) are publicly documented. The attack is repeatable and stateless.

### Recommendation
1. **Add `subscribeOn(Schedulers.boundedElastic())`** immediately after `Mono.fromCallable(...)` in `NetworkController.getNodes()` so that all downstream work — including the eager JDBC calls inside `NetworkServiceImpl.getNodes()` — executes off the gRPC server thread pool.
2. **Defer the eager JDBC calls** in `NetworkServiceImpl.getNodes()` (lines 61–65) inside the `Flux.defer(...)` block or a `Mono.fromCallable(...)` so they are subject to the same scheduler.
3. **Enforce a per-IP or global connection/request rate limit** at the Netty or load-balancer layer to bound the number of concurrent `getNodes` calls regardless of threading model.

### Proof of Concept
```python
# Requires: pip install grpcio grpcio-tools
import grpc, threading, time
# Assume generated stubs for NetworkService are available
from com.hedera.mirror.api.proto import network_service_pb2_grpc
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc-host:5600"
FILE_ID = basic_types_pb2.FileID(shardNum=0, realmNum=0, fileNum=102)

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    while True:
        try:
            # Each call blocks a gRPC server thread for 50-200ms
            list(stub.getNodes(AddressBookQuery(file_id=FILE_ID)))
        except:
            pass

# Open 200 connections × 5 concurrent calls = 1000 blocked server threads
threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads:
    t.start()
time.sleep(60)  # Sustain for 60 seconds; observe gRPC server latency/errors spike
```
