### Title
Uncached Empty-Optional DB Amplification via `EntityId.EMPTY` in `ContractBytecodeReadableKVState.readFromDataSource()`

### Summary
`ContractBytecodeReadableKVState.readFromDataSource()` contains no guard against `EntityId.EMPTY` (id=0) before invoking `ContractRepository.findRuntimeBytecode()`. Any unprivileged caller can supply a non-existent CREATE2 EVM address in an `eth_call` request, causing `toEntityId()` to return `EntityId.EMPTY`, which then drives a DB query for `id=0`. Because Spring Cache unwraps `Optional` before evaluating the `unless = "#result == null"` condition, the resulting `Optional.empty()` is treated as a null unwrapped value and is **never stored in the Caffeine cache**, so every subsequent request re-executes the same DB query. Combined with the global (not per-IP) throttle of 500 RPS, a single attacker can sustain hundreds of uncached DB queries per second, degrading the shared database for all mirror-node instances.

### Finding Description

**Exact code path:**

`ContractBytecodeReadableKVState.readFromDataSource()` (lines 39–47): [1](#0-0) 

```java
protected Bytecode readFromDataSource(@NonNull ContractID contractID) {
    final var entityId = toEntityId(contractID);          // may return EntityId.EMPTY
    return contractRepository
            .findRuntimeBytecode(entityId.getId())        // called with 0L
            .map(Bytes::wrap)
            .map(Bytecode::new)
            .orElse(null);
}
```

`toEntityId()` (lines 49–64) returns `EntityId.EMPTY` (id=0) for any non-long-zero EVM address that is not found in the entity table: [2](#0-1) 

`EntityId.EMPTY` has `id = 0L`: [3](#0-2) 

`findRuntimeBytecode` is annotated: [4](#0-3) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT,
           unless = "#result == null")
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause — Spring Cache Optional unwrapping:**
Spring's `CacheAspectSupport` unwraps `Optional` before evaluating the `unless` SpEL expression. When the query returns no row, the method returns `Optional.empty()`. Spring unwraps this to `null`, so `#result == null` evaluates to `true`, and the result is **excluded from the cache**. The Caffeine contract cache (`expireAfterAccess=1h, maximumSize=1000`) therefore never stores a negative result for key `0L`. [5](#0-4) 

**Per-request cache does not help across requests:**
`ReadableKVStateBase.get()` maintains a per-request `ContractCallContext` cache that prevents duplicate reads *within* a single EVM execution, but it is reset for every new `eth_call` invocation: [6](#0-5) [7](#0-6) 

**Entity lookup also uncached for empty results:**
`CommonEntityAccessor.getEntityByEvmAddressAndTimestamp()` delegates to `EntityRepository.findByEvmAddressAndDeletedIsFalse()`, which carries the same `unless = "#result == null"` pattern, so the entity miss for a non-existent address is also never cached: [8](#0-7) [9](#0-8) 

**Exploit flow per request:**
1. Attacker sends `POST /api/v1/contracts/call` with `to` = any 20-byte non-long-zero address not in the DB.
2. EVM resolves bytecode → `readFromDataSource()` → `toEntityId()` → entity DB query (cache miss, not stored) → `EntityId.EMPTY`.
3. `findRuntimeBytecode(0L)` → `SELECT runtime_bytecode FROM contract WHERE id = 0` → `Optional.empty()` → not cached.
4. Both queries repeat on every subsequent request.

### Impact Explanation
Each `eth_call` with a non-existent CREATE2 address generates at least two uncached DB queries against the shared PostgreSQL instance. Because the mirror node is stateless and horizontally scaled, all instances share the same DB. Sustained query load at the DB level degrades all nodes simultaneously. The default `statementTimeout` is 3 seconds, meaning each query holds a DB connection for up to 3 seconds under load, exhausting the connection pool and causing cascading failures across all mirror-node instances. [10](#0-9) 

### Likelihood Explanation
No authentication or account is required — `eth_call` is a public, unauthenticated endpoint. The global throttle allows up to 500 requests/second by default: [11](#0-10) 

The throttle is a single shared bucket (not per-IP), so a single attacker consumes the full budget. At 500 RPS, this produces ~500 uncached `SELECT runtime_bytecode FROM contract WHERE id = 0` queries per second. The attack requires no special knowledge, no credentials, and is trivially repeatable with a simple HTTP loop. [12](#0-11) 

### Recommendation
1. **Guard against `EntityId.EMPTY` in `readFromDataSource()`**: Add an early return before calling `findRuntimeBytecode`:
   ```java
   if (EntityId.isEmpty(entityId)) {
       return null;
   }
   ```
2. **Fix the `unless` condition to handle `Optional`**: Change `unless = "#result == null"` to `unless = "#result == null || !#result.isPresent()"` (or `unless = "#result?.orElse(null) == null"`) so that empty Optionals are also cached, preventing repeated DB queries for known-missing keys.
3. **Add per-IP rate limiting** in addition to the global gas/RPS bucket to prevent a single source from exhausting the global throttle.

### Proof of Concept
```bash
# Send repeated eth_call requests with a non-existent CREATE2 address
# (0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef is not a long-zero address)
while true; do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef",
      "data": "0x",
      "gas": 21000
    }' &
done
```

**Expected DB observation**: `pg_stat_activity` shows a sustained stream of `SELECT runtime_bytecode FROM contract WHERE id = 0` queries, one per request, never served from cache. DB CPU and connection count rise proportionally to request rate, degrading all mirror-node instances sharing the database.
