### Title
304 Not Modified Responses Create Unaccounted Gap in Aggregate Metrics Counters

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware()` function increments `allRequestCounter` for every response but only increments `allSuccessCounter` for 2xx and `allErrorCounter` for 4xx/5xx. HTTP 304 responses — which the server actively produces via `responseCacheCheckHandler` — fall through all branches silently, creating a permanent gap where `allRequestCounter > allSuccessCounter + allErrorCounter`. Any unprivileged external user can exploit this by repeatedly sending conditional `If-None-Match` requests to Hashgraph history endpoints, generating 304 responses that are invisible to aggregate success/error alerting.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware()`, lines 194–204.

```js
allRequestCounter.add(1);                          // line 195 — always fires
if (res.statusCode >= 200 && res.statusCode < 300) {
  allSuccessCounter.add(1);                        // 2xx only
} else if (res.statusCode >= 400 && res.statusCode < 500) {
  allClientErrorCounter.add(1);
  allErrorCounter.add(1);                          // 4xx only
} else if (res.statusCode >= 500) {
  allServerErrorCounter.add(1);
  allErrorCounter.add(1);                          // 5xx only
}
// 3xx (including 304) — no branch, no counter incremented
```

**Root cause:** The conditional chain has no branch for 3xx status codes. HTTP 304 (`Not Modified`) is a 3xx code; it satisfies none of the three conditions, so only `allRequestCounter` is incremented.

**304 responses are confirmed to be produced by the server.** `rest/middleware/responseCacheCheckHandler.js` line 54 explicitly sets `statusCode = httpStatusCodes.UNMODIFIED.code` (304) when the client's `If-None-Match` header matches the cached ETag. Integration tests at `rest/__tests__/integration/responseCacheHandler.test.js` lines 195–212 and 214–236 confirm this path is exercised in production scenarios.

**Exploit flow:**
1. Attacker sends a normal GET to any Hashgraph history endpoint (e.g., `/api/v1/transactions`). Server responds 200 with an `ETag` header.
2. Attacker stores the ETag and repeatedly sends `GET /api/v1/transactions` with `If-None-Match: <etag>`. Server responds 304 each time.
3. Each 304 increments `allRequestCounter` but increments neither `allSuccessCounter` nor `allErrorCounter`.
4. The per-route `requestTotalCounter` (line 207) does record the 304 with `code: "304"`, but the aggregate counters — the ones used for high-level alerting dashboards — are permanently skewed.

**Why existing checks are insufficient:** There is no guard, normalization, or catch-all branch that handles 3xx. The per-route counter recording at line 207 is a separate instrument with different labels and is not equivalent to the aggregate counters. Alerting rules built on `api_all_success` or `api_all_errors` will never see these requests.

### Impact Explanation
An attacker can continuously poll Hashgraph history endpoints using conditional requests without those accesses ever appearing in `api_all_success` or `api_all_errors`. Monitoring systems that compute error rate as `api_all_errors / api_all_request` will see the denominator grow while the numerator stays flat, actually suppressing the apparent error rate. Operators relying on aggregate counter dashboards or alerts (e.g., "alert if success rate drops below X%") will have a blind spot for the entire volume of 304 traffic. This enables covert, high-frequency polling of ledger history data — including transaction records, account history, and token transfers — without triggering any aggregate-level anomaly detection.

### Likelihood Explanation
No privileges are required. The only precondition is one successful initial request to obtain an ETag, which is returned by default on all cacheable endpoints. The `If-None-Match` header is a standard HTTP mechanism available to any HTTP client. The attack is trivially repeatable, fully automated, and leaves no trace in the aggregate success/error counters. The server's own integration tests demonstrate the 304 path works reliably.

### Recommendation
Add an explicit branch for 3xx responses in the aggregate counter logic, or replace the if/else chain with a catch-all that ensures every response is classified:

```js
allRequestCounter.add(1);
if (res.statusCode >= 200 && res.statusCode < 300) {
  allSuccessCounter.add(1);
} else if (res.statusCode >= 300 && res.statusCode < 400) {
  allRedirectCounter.add(1);   // new counter, or fold into success
} else if (res.statusCode >= 400 && res.statusCode < 500) {
  allClientErrorCounter.add(1);
  allErrorCounter.add(1);
} else if (res.statusCode >= 500) {
  allServerErrorCounter.add(1);
  allErrorCounter.add(1);
}
```
Alternatively, treat 304 as a success variant (it is a valid, non-error response) by changing the success condition to `res.statusCode >= 200 && res.statusCode < 400`. Also add a test case in the metrics middleware unit tests that asserts 304 responses are counted in the appropriate aggregate counter.

### Proof of Concept
```bash
# Step 1: Warm up — get ETag from any history endpoint
ETAG=$(curl -si https://<mirror-node>/api/v1/transactions?limit=1 \
  | grep -i '^etag:' | awk '{print $2}' | tr -d '\r')

# Step 2: Repeatedly send conditional requests (no privileges needed)
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "If-None-Match: $ETAG" \
    https://<mirror-node>/api/v1/transactions?limit=1
done
# All responses: 304

# Step 3: Observe metrics endpoint
curl https://<mirror-node>/api/v1/metrics  # (or configured metrics path)
# api_all_request_total will show 1000+ increments
# api_all_success_total will show 0 increments for these requests
# api_all_errors_total will show 0 increments for these requests
# Gap = 1000 unaccounted requests in aggregate monitoring
