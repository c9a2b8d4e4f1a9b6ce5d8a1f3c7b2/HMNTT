### Title
Unbounded Request Body Read in MetricsMiddleware Enables Memory Exhaustion DoS

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` wraps `r.Body` with a byte-counting reader via `middleware.Instrument` but applies no size cap (`http.MaxBytesReader` or equivalent) anywhere in the middleware chain. The downstream rosetta-sdk-go controllers read the entire body via `json.NewDecoder(r.Body).Decode(...)` with no limit. An unauthenticated attacker sending many concurrent large-body POST requests can exhaust server memory within the `ReadTimeout` window, crashing the process and causing a network partition.

### Finding Description
**Code path:**
- `rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–84: wraps `r.Body` with a counting reader for `requestBytesHistogram`; no size limit is imposed.
- `rosetta/main.go`, lines 217–227: the middleware chain is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. The `http.Server` sets `ReadTimeout` (default 5 s) but never calls `http.MaxBytesReader`.
- A `grep` across the entire repository for `MaxBytesReader`, `LimitReader`, `client_max_body_size` returns **zero matches**.

**Root cause:** The design assumes request bodies are small (histogram buckets top out at 50 KB), but no enforcement mechanism backs this assumption. `ReadTimeout` bounds the *time* a connection may send data, not the *bytes* it may send.

**Exploit flow:**
1. Attacker opens N concurrent TCP connections to port 5700 (directly exposed per the docker run example: `-p 5700:5700`).
2. Each connection sends a POST to any Rosetta endpoint (e.g., `/network/status`) with a large body streamed at line rate.
3. Within the 5-second `ReadTimeout` window, each connection can deliver ~62 MB on a 100 Mbps link, ~625 MB on 1 Gbps.
4. The rosetta-sdk-go JSON decoder reads the full body into a `[]byte` buffer in heap memory.
5. With N=50 concurrent connections on a 1 Gbps link, ~31 GB of heap is allocated before any connection times out.
6. The Go runtime OOMs; the process crashes; the Rosetta node becomes unreachable.

**Why existing checks fail:**
- `ReadTimeout` (5 s) limits the per-connection attack window but does not cap bytes; on fast links or with many connections it is insufficient.
- The nginx proxy config (`docker-compose.yml`, line 235) has no `client_max_body_size` for the `/rosetta/` location, and the server is also documented for direct deployment without nginx.
- The rosetta-sdk-go asserter validates JSON structure *after* the body is fully read into memory — too late to prevent allocation.

### Impact Explanation
An OOM crash terminates the Rosetta process, making the mirror node's Rosetta API completely unavailable. Exchange integrations relying on the Rosetta API for balance queries and transaction construction are severed. Because the Rosetta server runs in the same container as the importer in the all-in-one image, a crash can also interrupt block ingestion, causing a broader data-availability partition. Severity: **High** (unauthenticated, remotely triggerable, full service disruption).

### Likelihood Explanation
No authentication is required; any internet-accessible Rosetta endpoint accepts POST bodies. The attack requires only a moderate-bandwidth connection (100 Mbps suffices with ~50 concurrent connections) and is trivially scriptable with `curl` or `wrk`. It is repeatable: the server restarts (if supervised) and can be immediately re-attacked. Exchanges running public-facing Rosetta nodes are the primary target.

### Recommendation
1. Apply `http.MaxBytesReader` in `MetricsMiddleware` (or a dedicated middleware inserted before it) to cap request bodies at a safe limit (e.g., 1 MB, consistent with the 50 KB histogram bucket ceiling):
   ```go
   func MaxBodyMiddleware(next http.Handler) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, 1<<20) // 1 MB
           next.ServeHTTP(w, r)
       })
   }
   ```
   Insert it before `MetricsMiddleware` in `main.go`.
2. Add `client_max_body_size 1m;` to the nginx `/rosetta/` location block in `docker-compose.yml`.
3. Consider adding a connection-rate limit (e.g., via Traefik `rateLimit` middleware already scaffolded in `charts/hedera-mirror-rosetta/templates/middleware.yaml`).

### Proof of Concept
```bash
# Send 50 concurrent 512 MB POST bodies to the Rosetta network/status endpoint
# (body content is irrelevant; server reads it before JSON-parsing)
seq 50 | xargs -P50 -I{} bash -c \
  'dd if=/dev/zero bs=1M count=512 2>/dev/null | \
   curl -s -X POST http://<rosetta-host>:5700/network/status \
        -H "Content-Type: application/json" \
        --data-binary @- -o /dev/null'
# Expected result: server OOMs and crashes within ~5 seconds;
# subsequent requests return connection refused.
```
