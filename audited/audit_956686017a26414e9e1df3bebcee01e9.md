### Title
Unbounded `limit` Parameter in `TopicMessageFilter` Enables Database Exhaustion via Unauthenticated gRPC Subscription

### Summary
The `subscribeTopic()` method in `TopicMessageServiceImpl` accepts a caller-supplied `TopicMessageFilter` whose `limit` field is validated only with `@Min(0)` — no upper bound is enforced. Any unauthenticated client can set `limit` to `Long.MAX_VALUE` with a historical `startTime`, causing `topicMessageRetriever.retrieve()` to continuously poll the database in a tight loop for up to the full retriever timeout (60 seconds) per connection. With no per-client rate limiting or authentication gate, this is trivially repeatable and can be parallelized to exhaust database resources.

### Finding Description

**Exact code path:**

`TopicMessageFilter.java`, line 25–26: the `limit` field carries only `@Min(0)` — no `@Max` or server-side cap: [1](#0-0) 

`TopicMessageServiceImpl.java`, line 63: the raw user-supplied filter (including the unbounded `limit`) is passed directly to the retriever: [2](#0-1) 

The `flux.take(filter.getLimit())` at line 84 is a *downstream* reactive operator — it does not prevent the retriever from issuing repeated database queries to satisfy the limit: [3](#0-2) 

`RetrieverProperties.java` defines `maxPageSize = 1000` (throttled) and `timeout = 60s`. These are per-page and per-session caps, not a cap on total messages or total DB round-trips: [4](#0-3) 

`PollingTopicMessageRetriever` will keep issuing 1000-row DB queries every 2 seconds for up to 60 seconds (≈30 queries × 1000 rows = 30,000 rows per connection) trying to satisfy an arbitrarily large limit.

**Root cause:** The server trusts the client-supplied `limit` without capping it server-side. The only server-side bound is the retriever timeout, which still permits sustained DB load per connection.

**Failed assumption:** The design assumes clients will supply reasonable `limit` values. No authentication, no authorization, and no server-enforced maximum limit exist.

### Impact Explanation

An attacker opens N concurrent gRPC streams to `subscribeTopic`, each with `limit = Long.MAX_VALUE` and `startTime` pointing to the oldest available data. Each stream drives the `PollingTopicMessageRetriever` to issue continuous paginated SQL queries against the `topic_message` table for 60 seconds. With enough parallel connections, the database CPU and I/O are saturated, degrading or denying service to all legitimate users. The gRPC service has no authentication layer, so this is reachable by any network-accessible client.

### Likelihood Explanation

The gRPC endpoint is publicly exposed by design (mirror node public API). No credentials are required. The attack requires only a standard gRPC client library and knowledge of the `ConsensusService` proto. It is fully repeatable and scriptable. Any motivated attacker — including a competitor or someone wanting to disrupt the network — can execute this.

### Recommendation

1. **Enforce a server-side maximum limit**: Add `@Max(MAX_ALLOWED_LIMIT)` to the `limit` field in `TopicMessageFilter`, or clamp the value in `subscribeTopic()` before passing it to the retriever. [1](#0-0) 

2. **Add per-client rate limiting / connection throttling** at the gRPC interceptor layer to cap the number of concurrent streams per source IP.

3. **Cap total retriever polls**, not just page size and timeout, so a single subscription cannot issue unbounded DB queries regardless of timeout. [5](#0-4) 

### Proof of Concept

```
# Using grpcurl (no auth required):
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

Repeat this command in 50+ parallel processes. Each connection causes `PollingTopicMessageRetriever` to issue repeated `SELECT` queries against `topic_message` for 60 seconds. The aggregate DB load from 50 connections = up to 1,500 queries/minute each returning up to 1,000 rows, saturating the database.
