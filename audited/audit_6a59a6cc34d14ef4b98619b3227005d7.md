### Title
Unbounded Per-IP gRPC Connection Exhaustion in `subscribeTopic()` Enables Resource Starvation

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited simultaneous gRPC connections from a single unauthenticated IP address. The only server-side limit is `maxConcurrentCallsPerConnection = 5`, which caps streams per HTTP/2 connection but places no bound on the number of connections a single client may open. An attacker can open thousands of connections (each with up to 5 long-lived streaming subscriptions), exhausting the thread pool, database connection pool, and Redis listener capacity, causing legitimate subscribers to stall or be dropped.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) unconditionally accepts every inbound request and creates a Reactor subscription backed by `topicMessageService::subscribeTopic`:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java:43-48
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            .map(this::toResponse)
            .onErrorMap(ProtoUtil::toStatusRuntimeException)
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

**The only configured server-side limit** is `maxConcurrentCallsPerConnection = 5` in `NettyProperties` (default), applied in `GrpcConfiguration`:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java:33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This is a **per-connection** limit on HTTP/2 streams, not a per-IP or per-client connection limit. There is no:
- `maxInboundConnections` or equivalent per-IP connection cap on the Netty server
- `ServerInterceptor` in production code performing IP-based rate limiting or connection counting
- Any throttling mechanism in the `grpc` module (throttling exists only in the `web3` module via `ThrottleConfiguration`/`ThrottleManagerImpl`)
- `maxConnectionAge` or `maxConnectionIdle` settings to reclaim idle connections

**Root cause:** The server conflates "calls per connection" with "connections per client." An attacker opens N TCP connections, each carrying 5 streams = N×5 total live `subscribeTopic` subscriptions, with N unbounded.

### Impact Explanation

Each active `subscribeTopic` subscription holds:
- A slot in the shared Netty/Spring executor thread pool
- A HikariCP database connection (for historical retrieval) or a Redis pub/sub listener slot
- File descriptors for the TCP socket

With enough connections, the executor queue saturates, the DB connection pool (`hikaricp_connections_max`) is exhausted, and Redis listener capacity (`maxBufferSize = 16384`) overflows. Legitimate subscribers receive no messages or are dropped with errors. The Prometheus alert thresholds (`GrpcHighDBConnections > 75%`, `GrpcHighFileDescriptors > 80%`) confirm these are recognized resource ceilings. Message delivery to all subscribers stalls for the duration of the attack.

### Likelihood Explanation

No authentication or API key is required to call `subscribeTopic()`. The gRPC port (5600) is publicly exposed. A single attacker machine can open thousands of HTTP/2 connections using standard gRPC client libraries (e.g., `grpc-go`, `grpcurl` in a loop, or a custom script). The attack is trivially repeatable and requires no special knowledge beyond the public proto definition. It is a straightforward resource exhaustion DoS requiring no privileges.

### Recommendation

1. **Add a per-IP connection limit** at the Netty layer via a custom `ChannelHandler` (e.g., `io.netty.handler.ipfilter.RuleBasedIpFilter` or a connection-counting handler keyed by remote address) configured in `GrpcConfiguration`.
2. **Set `maxConnectionAge` and `maxConnectionIdle`** on `NettyServerBuilder` to reclaim long-lived idle connections.
3. **Add a gRPC `ServerInterceptor`** that tracks active stream counts per remote IP and returns `RESOURCE_EXHAUSTED` when a per-IP threshold is exceeded.
4. **Deploy an external rate-limiting proxy** (e.g., Envoy with `connection_limit` filter) in front of port 5600 as a defense-in-depth measure.

### Proof of Concept

```bash
# Requires grpc-go or any gRPC client that supports concurrent connections
# Replace TOPIC_ID and SERVER with actual values

for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}}' \
    SERVER:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

Each background process opens a separate TCP connection with one long-lived streaming call. With 2000 connections × 5 max streams each (if multiplexed), the server's executor, DB pool, and Redis listener are saturated.
