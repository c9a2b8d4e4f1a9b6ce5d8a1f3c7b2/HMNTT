### Title
Unauthenticated Cache Poisoning DoS via Unbounded `CachedPreparsedDocumentProvider` Cache Flooding

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed on the raw query string with no per-client rate limiting or authentication on the GraphQL endpoint. An unauthenticated attacker can flood the 1,000-entry Caffeine cache with unique syntactically valid queries, continuously evicting legitimate cached documents and forcing repeated parse-and-validate overhead on every real user request.

### Finding Description
**Exact code path:**

`CachedPreparsedDocumentProvider.java` line 26 uses `executionInput.getQuery()` — the raw, unmodified query string — as the sole cache key:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The cache is configured in `CacheProperties.java` line 15 as:
```
expireAfterWrite=1h, maximumSize=1000, recordStats
```
**Root cause:** The cache key space is unbounded (any unique query string creates a new entry), the cache size is fixed at 1,000, and there is no rate limiting, authentication, or per-IP throttle on the GraphQL endpoint. A search across all Java files under `graphql/` confirms zero throttle/rate-limit/security-config beans exist in that module.

**Exploit flow:**
1. Attacker sends 1,001+ syntactically valid but textually unique GraphQL queries (e.g., varying field aliases, argument values, or whitespace — all within the `maxCharacters=10000` / `maxTokens=1000` parser limits set in `GraphQlConfiguration.java` lines 34–38).
2. Each unique query string occupies one cache slot. Once 1,000 slots are filled, Caffeine's Window TinyLFU eviction begins removing the least-frequently-used entries.
3. Attacker continuously streams new unique queries to keep displacing legitimate entries.
4. Every real user request whose query was evicted must re-execute `parseAndValidateFunction` (full parse + schema validation) instead of getting a cache hit.

**Why existing checks are insufficient:**
- `maxCharacters(10000)` / `maxTokens(1000)` bound individual query size but do not limit the number of distinct queries.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are execution-phase instrumentations; they run *after* the preparsed document provider has already stored the entry in the cache.
- No rate-limiting bean, no Spring Security configuration, and no IP-based throttle exists anywhere in the `graphql/` module.

### Impact Explanation
Every cache miss forces a full GraphQL parse + schema validation cycle. Under sustained flooding, legitimate queries lose all caching benefit, increasing per-request CPU cost. At high request rates this can degrade response latency and, combined with the DB query cost, contribute to service degradation. The 1-hour TTL (`expireAfterWrite=1h`) means evicted legitimate entries do not self-heal quickly under ongoing attack.

### Likelihood Explanation
No privileges are required. The GraphQL endpoint (`/graphql/alpha`) is publicly reachable. Generating 1,000 unique valid queries is trivial — varying a single integer argument (e.g., `num: 1` through `num: 1000`) produces 1,000 distinct cache keys. The attack is fully repeatable and scriptable with a single `curl` loop or any HTTP client.

### Recommendation
1. **Add per-IP or global rate limiting** on the `/graphql/alpha` endpoint (e.g., via Spring Security, a servlet filter, or an API gateway), consistent with the rate-limiting already present in the `web3` module (`ThrottleManagerImpl`).
2. **Normalize the cache key** before storage — strip insignificant whitespace and sort field selections so semantically equivalent queries share a cache entry, reducing the effective key space.
3. **Consider query allowlisting** (persisted queries / trusted document store) so only a fixed set of known queries are accepted, making cache flooding structurally impossible.
4. **Increase cache size** only as a last resort; it does not fix the root cause.

### Proof of Concept
```bash
# Fill the cache with 1001 unique valid queries (no auth required)
for i in $(seq 1 1001); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance }}\"}" \
    > /dev/null
done

# Now send a legitimate query that was previously cached — it will be re-parsed
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance }}"}'
```
Repeat the flooding loop continuously to sustain eviction pressure. Observe increased parse latency via the `recordStats` Caffeine metrics exposed through Micrometer.
