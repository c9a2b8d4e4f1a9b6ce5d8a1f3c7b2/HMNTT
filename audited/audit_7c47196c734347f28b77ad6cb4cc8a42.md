### Title
Gas Throttle Bypass via `eth_estimateGas` Binary Search Loop — Up to 20x EVM Execution Budget Multiplication

### Summary
When a user calls `eth_estimateGas`, `ContractExecutionService.estimateGas()` first performs one EVM execution with `estimate=false` (which triggers `restoreGasToBucket()`), then invokes `BinaryGasEstimator.search()` which calls `doProcessCall(params, gas, true)` up to 20 additional times. Because `doProcessCall` skips `restoreGasToBucket()` entirely when `estimate=true`, those 20 EVM executions consume no throttle tokens. Any unprivileged user can therefore obtain up to 21 full EVM executions for the cost of a single throttle deduction.

### Finding Description

**Exact code path:**

`ContractExecutionService.processCall()` (line 53–54) branches on `params.isEstimate()` and calls `estimateGas()`: [1](#0-0) 

Inside `estimateGas()`, the initial call uses `callContract(params, context)` which internally calls `doProcessCall(params, params.getGas(), false)` — `estimate=false` — so `restoreGasToBucket()` fires normally: [2](#0-1) 

Then `binaryGasEstimator.search()` is invoked with the lambda `gas -> doProcessCall(params, gas, true)` — `estimate=true`. The loop runs up to `maxGasEstimateRetriesCount` (default **20**) iterations: [3](#0-2) 

Each iteration calls `doProcessCall` with `estimate=true`. Inside `doProcessCall`, the `finally` block is gated on `!estimate`: [4](#0-3) 

This means `restoreGasToBucket()` is **never called** for any of the 20 binary-search EVM executions: [5](#0-4) 

**Root cause:** The throttle bucket is debited once at the HTTP-filter layer per request. `restoreGasToBucket()` is the only mechanism that accounts for gas consumed during execution. By passing `estimate=true`, the binary search loop completely bypasses this accounting for all 20 iterations.

**Failed assumption:** The design assumes the binary search iterations are lightweight or negligible. In reality each iteration is a full EVM execution at an arbitrary gas value (up to the user-supplied gas limit, default 15 million), and none of them are charged to the throttle.

### Impact Explanation

An attacker submitting a single `eth_estimateGas` request with the maximum gas limit (15 million) triggers up to 21 full EVM executions (1 initial + 20 binary search) while the throttle bucket is debited only once. This allows:

- **Resource exhaustion / DoS**: Sustained `eth_estimateGas` spam can saturate the EVM execution thread pool at 1/21st the throttle cost of equivalent `eth_call` spam.
- **Throttle bypass**: Rate-limited users can execute computationally expensive contracts far beyond their allocated gas budget.
- **Amplification factor**: Up to 20x, confirmed by the test suite's `maxGasEstimateRetriesCount` ceiling of 20. [6](#0-5) 

### Likelihood Explanation

- **No privileges required**: Any caller of the public `eth_estimateGas` JSON-RPC endpoint can trigger this.
- **Trivially repeatable**: A simple loop of `eth_estimateGas` calls with a complex contract payload is sufficient.
- **No special knowledge needed**: The attacker only needs to know that `eth_estimateGas` exists and supply a gas-heavy contract call.

### Recommendation

1. **Account for binary-search iterations in the throttle**: Deduct gas from the throttle bucket for each `estimate=true` iteration, or accumulate total gas used across all iterations and perform a single deduction/restoration at the end of `estimateGas()`.
2. **Cap total gas across the entire estimation flow**: Track cumulative gas consumed by all `doProcessCall` invocations (initial + binary search) and enforce the throttle limit against the total.
3. **Reduce `maxGasEstimateRetriesCount`**: Lowering the ceiling reduces the amplification factor as a short-term mitigation.
4. **Apply per-request EVM-execution count limits**: Reject or throttle requests that would trigger more than N total EVM executions regardless of the `estimate` flag.

### Proof of Concept

**Preconditions:** Access to the `eth_estimateGas` JSON-RPC endpoint; no account or authentication required.

**Steps:**

1. Deploy or identify a contract with a computationally expensive function (e.g., one that consumes ~10M gas).
2. Send repeated `eth_estimateGas` requests targeting that function with `gas: 15000000`:
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_estimateGas",
     "params": [{"to": "<contract>", "data": "<expensive_calldata>", "gas": "0xE4E1C0"}],
     "id": 1
   }
   ```
3. Observe that each request triggers the initial EVM execution (`estimate=false`) **plus** up to 20 binary-search EVM executions (`estimate=true`), all charged as a single throttle deduction.
4. Compare the throttle bucket drain rate against an equivalent number of `eth_call` requests performing the same computation — the `eth_estimateGas` path drains the bucket ~20x slower, confirming the bypass.

**Expected result:** The throttle bucket is exhausted ~20x faster by `eth_call` than by `eth_estimateGas` for equivalent total EVM computation, demonstrating the amplification. [7](#0-6) [8](#0-7)

    protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
