### Title
Cache Key Collision via 32-bit `Arrays.hashCode()` on EVM Addresses Enables Cross-Account Entity Exposure

### Summary
`findByEvmAddressAndDeletedIsFalse()` uses `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit integer — as the sole cache key for the `CACHE_NAME_EVM_ADDRESS` Caffeine cache. Because the key space (2^32) is astronomically smaller than the EVM address space (2^160), an unprivileged attacker can trivially find a second EVM address that hashes to the same integer as a victim's address. When the victim's entity is in cache, a lookup with the colliding address returns the victim's `Entity` object with no post-hit address validation.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode()` delegates directly to `Arrays.hashCode(byte[])`: [1](#0-0) 

This integer is used as the complete cache key: [2](#0-1) 

The entity cache manager registers `CACHE_NAME_EVM_ADDRESS` with a shared Caffeine spec: [3](#0-2) 

Default entity cache config is `expireAfterWrite=1s,maximumSize=10000`: [4](#0-3) 

**Root cause:** The cache key is a boxed `Integer` derived from `Arrays.hashCode()`. Caffeine uses `Integer.equals()` for key comparison. Two distinct `byte[]` EVM addresses that produce the same 32-bit hash value are treated as identical keys — the first cached entity is returned for both addresses. There is no post-cache-hit validation that the returned entity's `evmAddress` field matches the requested address.

**Failed assumption:** The design assumes `Arrays.hashCode(byte[])` produces a unique key per unique EVM address. This is false: the function maps 2^160 possible 20-byte inputs onto 2^32 outputs, guaranteeing collisions.

**Exploit flow:**
1. Victim address `A` (20 bytes) is looked up; its entity `E_A` is stored in cache under key `k = Arrays.hashCode(A)`.
2. Attacker computes `k` from `A` (public information — just call `Arrays.hashCode` locally).
3. Attacker generates key pairs until they find an address `B` where `Arrays.hashCode(B) == k`. By birthday-attack math, this requires ~65,536 iterations on average (2^16), trivially feasible in milliseconds on commodity hardware.
4. Within the 1-second cache TTL, attacker calls `eth_call` or any JSON-RPC method that triggers `findByEvmAddressAndDeletedIsFalse(B)`.
5. Spring's `@Cacheable` intercept finds key `k` in cache and returns `E_A` — the victim's entity — without hitting the database.
6. The caller receives `E_A` (containing account metadata, keys, balances, etc.) for address `B`.

The `unless = "#result == null"` guard only prevents caching empty results; it does not prevent a collision hit from returning a wrong entity. [2](#0-1) 

The same flaw exists in `findByEvmAddressOrAliasAndDeletedIsFalse()`: [5](#0-4) 

### Impact Explanation
A successful collision causes the mirror node to return a different account's `Entity` record — including account ID, EVM address, alias, key material references, balance metadata, and deletion status — to an unauthorized caller. In EVM execution context (`CommonEntityAccessor.getEntityByEvmAddressAndTimestamp`), this can cause the wrong account's state to be used during contract execution, enabling balance spoofing, incorrect authorization checks, or misdirected token operations. [6](#0-5) 

### Likelihood Explanation
No privileges are required. The attacker only needs:
- Knowledge of a target EVM address (public on-chain).
- The ability to generate Ethereum key pairs (standard tooling, milliseconds per pair).
- The ability to submit a JSON-RPC request within the 1-second cache TTL.

Finding a colliding address requires ~65,536 key-pair generations on average — achievable in under 1 second on any modern CPU. The 1-second TTL is short but not a meaningful barrier: the attacker can pre-compute the colliding address offline and submit the request immediately after observing the victim's address being active. The attack is repeatable and requires no on-chain transactions.

### Recommendation
Replace the 32-bit `Arrays.hashCode()` cache key with the full byte array content as the key. Spring's Caffeine integration supports composite keys or `SimpleKey` wrappers. The correct fix is:

```java
// Instead of:
key = "@spelHelper.hashCode(#alias)"

// Use the raw bytes wrapped in a type that implements equals/hashCode correctly:
key = "T(java.util.Arrays).toString(#alias)"
// or use a ByteBuffer wrapper:
key = "T(java.nio.ByteBuffer).wrap(#alias)"
```

`ByteBuffer.wrap(byte[])` implements `equals()` and `hashCode()` based on full byte content, eliminating collisions. Alternatively, encode the address as a hex string key. Also add a post-cache-hit guard that verifies the returned entity's `evmAddress` matches the requested address before returning it.

### Proof of Concept

```java
// Step 1: Compute hash of victim address
byte[] victimAddress = Hex.decodeHex("deadbeefdeadbeefdeadbeefdeadbeefdeadbeef");
int targetHash = Arrays.hashCode(victimAddress); // e.g., 0x12345678

// Step 2: Find colliding address (brute force, ~65K iterations)
byte[] collidingAddress = new byte[20];
for (long i = 0; i < Long.MAX_VALUE; i++) {
    // fill collidingAddress with i's bytes
    ByteBuffer.wrap(collidingAddress).putLong(i);
    if (Arrays.hashCode(collidingAddress) == targetHash) {
        System.out.println("Collision found: " + Hex.encodeHexString(collidingAddress));
        break;
    }
}

// Step 3: Trigger victim lookup to populate cache
// GET /api/v1/contracts/call with `to: 0xdeadbeef...`

// Step 4: Within 1 second, query with colliding address
// GET /api/v1/contracts/call with `to: 0x<collidingAddress>`
// → Returns victim's Entity object
```
