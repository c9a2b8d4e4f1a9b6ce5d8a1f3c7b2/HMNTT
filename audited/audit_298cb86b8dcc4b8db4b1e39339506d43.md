### Title
Unbounded Retry Storm via `Retry.backoff(Long.MAX_VALUE, ...)` Without `maxBackoff` in `PollingTopicMessageRetriever.retrieve()`

### Summary
`PollingTopicMessageRetriever.retrieve()` configures `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` with no `maxBackoff` cap and no per-connection rate limiting. An unprivileged attacker can open many concurrent gRPC `subscribeTopic` connections; when those connections collectively stress the database into intermittent errors, each connection's retry loop restarts the full polling cycle, amplifying load in a positive-feedback loop. The per-element `.timeout(60s)` only fires when **no element is emitted for 60 consecutive seconds**, so intermittent partial success keeps the loop alive indefinitely.

### Finding Description

**Exact location:**
`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 51–62

```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
            .jitter(0.1)
            .withFixedDelay(context.getFrequency())
            .withScheduler(scheduler))
    ...
    .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))   // line 58 — no maxBackoff
    .timeout(retrieverProperties.getTimeout(), scheduler)              // line 59 — per-element, not total
``` [1](#0-0) 

**Root cause — two compounding flaws:**

1. **No `maxBackoff`**: `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` uses Reactor's default `maxBackoff` of `Duration.ofMillis(Long.MAX_VALUE)` — effectively unbounded. Compare with `SharedPollingTopicListener` line 51, which explicitly caps backoff: `.retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))`. The retriever omits this cap entirely. [2](#0-1) 

2. **Per-element timeout, not total-subscription timeout**: `Flux.timeout(Duration)` in Reactor resets on every emitted element. When the database is intermittently failing (some polls succeed, some throw), elements are occasionally emitted, the 60-second clock resets, and the retry loop runs indefinitely — the timeout never fires. [3](#0-2) 

3. **No connection rate limiting**: `ConsensusController.subscribeTopic()` accepts any number of concurrent connections from any unauthenticated caller with no throttle, quota, or backpressure guard. [4](#0-3) 

**Exploit flow:**

Each retry does not retry a single query — it restarts the **entire** `repeatWhen` polling loop. For the throttled path (`numRepeats = Long.MAX_VALUE`, `pollingFrequency = 2s`) this is an infinite polling loop; for the unthrottled path (`maxPolls=12`, `pollingFrequency=20ms`) it fires 12 DB queries in 240 ms per retry attempt. [5](#0-4) 

### Impact Explanation
Each attacker-controlled connection that encounters a DB error restarts a full polling loop after a short backoff, issuing new DB queries. With N concurrent connections all retrying, the aggregate query rate grows proportionally to N. Because the timeout only resets on success, intermittent DB stress (not full outage) keeps all N loops alive indefinitely. This can exhaust the DB connection pool, degrade query latency, and cascade to other mirror-node services sharing the same database — consistent with degrading ≥30% of network processing nodes that depend on the mirror node's DB tier.

### Likelihood Explanation
No authentication is required to call `subscribeTopic`. The gRPC endpoint is publicly documented and reachable with standard tooling (`grpcurl`). Opening hundreds of concurrent connections from a single host or a small botnet is trivial. The attacker does not need to fully take down the DB — only push it into intermittent error territory, which is achievable at modest scale. The attack is repeatable and self-sustaining once the feedback loop starts.

### Recommendation
1. **Add `maxBackoff`** to match the pattern in `SharedPollingTopicListener`:
   ```java
   .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))
       .maxBackoff(Duration.ofSeconds(10)))
   ```
2. **Replace per-element timeout with a total-subscription timeout** using `Flux.timeout(Duration, Publisher<T> fallback)` or wrap the entire chain in a `Mono.delay`-based deadline so the subscription terminates regardless of intermittent success.
3. **Add per-IP / per-connection rate limiting** at the gRPC layer (e.g., via a `ServerInterceptor` or an API gateway) to cap concurrent subscriptions per caller.
4. **Add a maximum retry count** (e.g., `Retry.backoff(10, Duration.ofSeconds(1)).maxBackoff(Duration.ofSeconds(30))`) so a persistently failing connection gives up rather than looping forever.

### Proof of Concept
```bash
# Open 500 concurrent subscriptions to a valid topic from a single host
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
    <mirror-node>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Once DB connection pool is exhausted, connections begin throwing errors.
# Each connection's Retry.backoff loop restarts the full polling cycle.
# With intermittent partial success, the per-element timeout never fires.
# DB query rate climbs proportionally to open connections, sustaining the storm.
# Monitor DB connection count and query latency to observe degradation.
```
