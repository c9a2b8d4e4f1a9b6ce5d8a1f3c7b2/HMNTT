### Title
Unprivileged Cancel-Triggered Race Condition in `RedisTopicListener.getSharedListener()` Causes Re-subscription Gap and Temporary Message Loss

### Summary
In `RedisTopicListener`, when the last subscriber to a shared Flux cancels their gRPC stream, `doOnCancel(() -> unsubscribe(topic))` fires and removes the topic entry from `topicMessages`. Any concurrent or subsequent subscriber calling `computeIfAbsent` then creates a brand-new Flux that must re-establish a Redis subscription from scratch, creating a window during which Redis-published topic messages are not received. Any unprivileged gRPC client can deliberately trigger this by subscribing and immediately cancelling, particularly when they are the sole or last active subscriber for a given topic.

### Finding Description

**Exact code path:**

`getSharedListener()` at [1](#0-0)  uses `ConcurrentHashMap.computeIfAbsent` to return or create a shared Flux per topic.

`subscribe()` at [2](#0-1)  builds the Flux chain with `doOnCancel(() -> unsubscribe(topic))` placed **before** `.share()`. The `.share()` operator multicasts to all subscribers and cancels its upstream only when the **last** subscriber cancels. When that upstream cancellation propagates, `doOnCancel` fires.

`unsubscribe()` at [3](#0-2)  unconditionally removes the topic key from `topicMessages`.

**Root cause:** There is no synchronization or guard between the map removal in `unsubscribe()` and the creation of a new subscription in `computeIfAbsent`. The sequence:

1. Last subscriber cancels → `.share()` cancels upstream → `doOnCancel` fires → `topicMessages.remove(key)`
2. New subscriber calls `computeIfAbsent` → key is absent → `subscribe(topic)` is called → new `ReactiveRedisMessageListenerContainer.receive()` call is initiated

...has a non-zero window between step 1 and the moment the new Redis subscription is active. Any `PUBLISH` commands sent to Redis during this window are not received by the new Flux.

**Why the failed assumption matters:** The design assumes the `.share()` operator keeps the upstream alive as long as any subscriber is present. This is true for concurrent subscribers, but the `doOnCancel` + map removal creates a TOCTOU gap: the map entry is removed before the new subscription is established, so a racing `computeIfAbsent` creates a cold start with no overlap.

### Impact Explanation

Redis messages published during the re-subscription window are silently dropped from the live stream. The `TopicMessageServiceImpl` safety check at [4](#0-3)  fires once after 1 second and queries the database for missed messages, and `missingMessages()` at [5](#0-4)  fills sequence-number gaps when the next live message arrives. These mitigations prevent **permanent** message loss but do not prevent the **delay**: subscribers experience up to a 1-second gap in live delivery per attack cycle. An attacker who repeatedly subscribes and cancels can sustain this delay continuously, effectively degrading real-time HCS topic message delivery to polling-speed recovery for all concurrent subscribers on that topic.

### Likelihood Explanation

No authentication or rate-limiting is required to open and immediately cancel a gRPC `subscribeTopic` stream. The attacker only needs to be the last active subscriber for the targeted topic at the moment of cancellation — trivially achievable for low-traffic or newly-created topics. The attack is fully repeatable: after each cancel the attacker can re-subscribe and cancel again, keeping the re-subscription window open continuously. No special privileges, credentials, or network position are required beyond the ability to reach the gRPC endpoint.

### Recommendation

1. **Do not remove the map entry on cancel when re-subscription is intended.** Instead of removing on `doOnCancel`/`doOnComplete`, use `Flux.replay(0).refCount(1, Duration.ofMillis(X))` or a `refCount` with a grace period so the upstream stays alive briefly after the last subscriber leaves, allowing new subscribers to attach without a cold-start gap.
2. **Alternatively**, replace the `doOnCancel(() -> unsubscribe(topic))` + `computeIfAbsent` pattern with an explicit `Sinks.Many` or `ConnectableFlux.publish().refCount()` that is only torn down after a configurable idle timeout, not immediately on last-subscriber cancel.
3. **At minimum**, add a guard in `unsubscribe()` that only removes the entry if no new subscriber has already replaced it (compare-and-remove using the current Flux reference as the value).

### Proof of Concept

**Preconditions:** gRPC endpoint reachable; topic exists; Redis listener mode active; attacker is the only subscriber for the target topic (or can arrange to be last).

**Steps:**

1. Open gRPC stream: `subscribeTopic({topicId: X, startTime: now})`
2. Immediately cancel the stream (send RST_STREAM or close the client).
3. Observe server logs: `"Unsubscribing from topic.X"` confirms `topicMessages.remove` was called.
4. Concurrently (or immediately after), a legitimate subscriber opens a new stream for topic X.
5. During the window between step 3 and the new `r.receive(...)` call completing, publish a Redis message to channel `topic.X`.
6. Observe that the legitimate subscriber does **not** receive the message in real time; it is only recovered after the 1-second safety check fires and queries the database.
7. Repeat steps 1–6 continuously to sustain the delivery delay indefinitely.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
