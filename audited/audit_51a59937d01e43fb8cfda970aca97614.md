### Title
`BufferUnderflowException` via Under-Length EVM Address in `getByEvmAddressAndType`

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` wraps the decoded EVM address bytes in a `ByteBuffer` and unconditionally calls `buffer.getLong()` a second time after the zero-prefix check passes, without verifying that at least 20 bytes are present. An unauthenticated caller can supply a 13-byte hex-encoded address whose first 12 bytes are zero, satisfying the `getInt()==0 && getLong()==0` guard, and then trigger an uncaught `BufferUnderflowException` on the final `getLong()` call.

### Finding Description

**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, `getByEvmAddressAndType()`, lines 35–38.

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);   // line 35 – no length check
var buffer = ByteBuffer.wrap(evmAddressBytes);            // line 36
if (buffer.getInt() == 0 && buffer.getLong() == 0) {     // line 37 – consumes 12 bytes
    return entityRepository.findById(buffer.getLong())   // line 38 – needs 8 more bytes
        .filter(e -> e.getType() == type);
}
```

**Root cause:** `decodeEvmAddress()` in `GraphQlUtils.java` (lines 87–98) performs only hex-decoding and strips the `0x` prefix; it applies **no length validation**. The caller likewise performs no length check before using the `ByteBuffer`. A standard EVM address is 20 bytes, which provides exactly 4 + 8 + 8 bytes for the three reads. Any input shorter than 20 bytes but at least 12 bytes (so the guard passes) leaves fewer than 8 bytes for the final `getLong()`, causing `java.nio.BufferUnderflowException`.

**Exploit flow:**
1. Attacker sends a GraphQL query that reaches `getByEvmAddressAndType` with `evmAddress = "0x000000000000000000000000XX"` (26 hex chars = 13 bytes, first 24 hex chars = 12 zero bytes, last 2 hex chars = any value, e.g. `"0x00000000000000000000000001"`).
2. `decodeEvmAddress` decodes this to a 13-byte array with no error.
3. `buffer.getInt()` reads bytes 0–3 → `0`. Buffer position = 4.
4. `buffer.getLong()` reads bytes 4–11 → `0`. Buffer position = 12. Guard passes.
5. `buffer.getLong()` attempts to read bytes 12–19 (8 bytes), but only 1 byte (byte 12) remains → **`BufferUnderflowException` thrown**.

**Why existing checks are insufficient:**
- `decodeEvmAddress` only catches `DecoderException` (invalid hex characters); it does not enforce a 20-byte length. [1](#0-0) 
- `getByEvmAddressAndType` has no `evmAddressBytes.length` guard before any `ByteBuffer` read. [2](#0-1) 

### Impact Explanation
`BufferUnderflowException` is an unchecked `RuntimeException`. Unless the GraphQL framework's error handler explicitly catches it (there is no evidence of such a handler in this service), it surfaces as an unhandled server error. Repeated invocation by any unauthenticated caller constitutes a reliable, zero-cost denial-of-service against the GraphQL endpoint for entity lookups. The "direct loss of funds" label in the audit scope is a misclassification; the concrete impact is service disruption and information leakage (stack traces may be returned to the caller depending on framework configuration).

### Likelihood Explanation
The GraphQL `evmAddress` field is a plain string input requiring no authentication or special privilege. Any external user who can send a GraphQL query can trigger this with a single crafted request. The condition (first 12 bytes zero, total length 13–19 bytes) is trivially constructable and repeatable.

### Recommendation
Add an explicit length check in `decodeEvmAddress` or at the top of `getByEvmAddressAndType`:

```java
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("EVM address must be exactly 20 bytes");
}
```

Alternatively, enforce the check inside `GraphQlUtils.decodeEvmAddress` so all callers benefit uniformly.

### Proof of Concept
Send the following GraphQL query to the mirror-node GraphQL endpoint (no authentication required):

```graphql
{
  account(input: { evmAddress: "0x00000000000000000000000001" }) {
    id
  }
}
```

The address `0x00000000000000000000000001` decodes to 13 bytes. Bytes 0–3 = `0x00000000` (int 0), bytes 4–11 = `0x0000000000000000` (long 0) → guard passes. The subsequent `buffer.getLong()` at line 38 finds only 1 remaining byte and throws `BufferUnderflowException`.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);

        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
