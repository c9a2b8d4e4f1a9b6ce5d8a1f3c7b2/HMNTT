### Title
Unauthenticated DB Query Exhaustion via Non-Existent Token Lookups in `getTokenInfoRequest()`

### Summary
`getTokenInfoRequest()` in `rest/tokens.js` accepts any syntactically valid EVM address as a `tokenId` path parameter, unconditionally executes a full database query against the `token`/`entity`/`custom_fee` tables, and only throws `NotFoundError` after the query returns empty. With no application-level rate limiting in the REST API layer, an unprivileged attacker can flood the endpoint with valid-format but non-existent token IDs, exhausting the bounded DB connection pool and degrading service for all users.

### Finding Description

**Code path:**

`rest/tokens.js` lines 548–565 (`getTokenInfoRequest`): [1](#0-0) 

`getAndValidateTokenIdRequestPathParam` (lines 424–431) accepts `allowEvmAddress: true` with `EvmAddressType.NUM_ALIAS`, meaning any 20-byte hex string of the form `0x<40 hex chars>` passes validation and is encoded to an integer ID: [2](#0-1) 

After validation, `extractSqlFromTokenInfoRequest` builds a multi-table JOIN query (token + entity + custom_fee, optionally with history tables) and `pool.queryQuietly` executes it unconditionally: [3](#0-2) 

The DB pool is bounded by `config.db.pool.maxConnections`: [4](#0-3) 

**No application-level rate limiting exists in the REST API.** A grep across all `rest/**/*.js` files for `rateLimit`, `rate-limit`, `express-rate-limit`, and `throttle` returns zero matches. The throttling infrastructure (bucket4j) exists only in the `web3/` Java service: [5](#0-4) 

**Error logging:** `NotFoundError` maps to HTTP 404, which `shouldReturnMessage` classifies as a client error, routing to `logger.warn` — not `logger.error`. So the log flooding is at WARN level, not ERROR: [6](#0-5) 

### Impact Explanation

Each request with a valid EVM address format for a non-existent token consumes one DB connection slot for the full duration of the multi-table JOIN query. The pool has a hard cap (`maxConnections`). Under sustained flood, the pool is exhausted, causing `connectionTimeoutMillis` errors (`DbError`) for all concurrent legitimate requests, returning HTTP 503 to real users. Additionally, WARN-level log entries are emitted for every 404, which can fill log storage and degrade observability. The query itself is non-trivial (JOIN across `token`, `entity`, `custom_fee`, optionally with `_history` union subqueries when a `timestamp` filter is supplied).

### Likelihood Explanation

No authentication or rate limiting is required. Any attacker with network access can generate an unlimited stream of valid EVM addresses (e.g., `0x0000000000000000000000000000000000000001` through `0xffffffffffffffffffffffffffffffffffffffff`) and issue concurrent GET requests. The attack is trivially scriptable with `curl`, `ab`, or `wrk`. The only external mitigation would be infrastructure-level rate limiting (e.g., Traefik), but this is not enforced at the application layer and may not be present in all deployments.

### Recommendation

1. **Add application-level rate limiting** per IP using a middleware such as `express-rate-limit` applied to all routes in `rest/server.js`, specifically before the token route registration at line 120.
2. **Add a fast existence pre-check** using a lightweight `SELECT 1 FROM entity WHERE id = $1` query before executing the expensive multi-table JOIN, so non-existent IDs are rejected cheaply.
3. **Cap concurrent DB queries per IP** or add a semaphore on the pool to prevent a single source from exhausting all connections.
4. **Ensure Traefik (or equivalent) rate limiting** is enforced at the ingress level for all REST API deployments, not just Rosetta.

### Proof of Concept

```bash
# Generate 10,000 concurrent requests with valid EVM address format, non-existent tokens
for i in $(seq 1 10000); do
  printf -v addr "0x%040x" $i
  curl -s "https://<mirror-node-host>/api/v1/tokens/$addr" &
done
wait
```

Each request passes `getAndValidateTokenIdRequestPathParam()` (valid 20-byte hex), triggers `pool.queryQuietly()` with the full JOIN query, returns 0 rows, throws `NotFoundError`, emits a WARN log entry, and holds a DB connection slot for the query duration. With sufficient concurrency, `pool.maxConnections` is saturated and legitimate requests receive HTTP 503.

const getTokenInfoRequest = async (req, res) => {
  const tokenId = getAndValidateTokenIdRequestPathParam(req);

  // extract and validate filters from query param
  const filters = utils.buildAndValidateFilters(req.query, acceptedSingleTokenParameters, validateTokenInfoFilter);
  const {query, params} = extractSqlFromTokenInfoRequest(tokenId, filters);

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length !== 1 || !rows[0].custom_fee) {
    throw new NotFoundError();
  }
