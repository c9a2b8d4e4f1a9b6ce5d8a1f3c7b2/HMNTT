### Title
Permanent `inFlightCounter` Inflation via Client-Disconnect Race (Missing `close` Event Handler)

### Summary
In `rest/middleware/metricsHandler.js`, `metricsMiddleware()` increments `inFlightCounter` unconditionally on every request but only decrements it (and records all other metrics) inside a `res.on('finish', ...)` handler. In Node.js, the `finish` event is **not** emitted when the client closes the TCP connection before the response is fully flushed; only the `close` event fires. Because no `close` handler exists anywhere in the REST middleware stack, any unprivileged client that aborts a connection before the response is sent causes a permanent +1 leak in `inFlightCounter` and a complete omission of that request from every other exported metric.

### Finding Description
**Exact code path:**

- `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, lines 152–214.
- Line 169: `inFlightCounter.add(1)` — executed unconditionally for every non-metrics request.
- Lines 185–211: `res.on('finish', () => { inFlightCounter.add(-1); allRequestCounter.add(1); … })` — the **only** place where the counter is decremented and all other metrics are recorded.

**Root cause / failed assumption:**  
The code assumes that every request that enters the middleware will eventually emit a `finish` event. This is false. Node.js's `http.ServerResponse` emits `finish` only after the response has been fully written to the OS socket buffer. When the client closes the connection first (TCP RST or FIN before `res.end()` is called), Node.js emits `close` on the response object instead, and `finish` is never emitted. There is no `res.on('close', …)` or `req.on('close', …)` handler anywhere in the REST middleware stack (confirmed by grep across `rest/**/*.js`).

**Exploit flow:**
1. Attacker opens a TCP connection to the mirror-node REST API.
2. Sends a valid HTTP request (e.g., `GET /api/v1/transactions`).
3. Immediately sends a TCP RST (or closes the socket) before the server writes the response.
4. The server's request-processing pipeline continues; `inFlightCounter` was already incremented at line 169.
5. Because the client is gone, `res.end()` either errors or the socket is destroyed; `finish` never fires.
6. `inFlightCounter.add(-1)` (line 186) is never called.
7. `allRequestCounter`, `allSuccessCounter`/`allErrorCounter`/`allClientErrorCounter`, `requestTotalCounter`, `durationHistogram`, `requestSizeHistogram`, `responseSizeHistogram` — none are updated for this request.

**Why existing checks are insufficient:**  
There are no existing checks. The middleware registers only a `finish` listener and has no fallback `close` listener. The `res.write` / `res.end` monkey-patches (lines 176–183) only accumulate `responseSize`; they do not trigger metric recording and do not guard against the missing `finish` event.

### Impact Explanation
- **`api_all_request_in_processing_total` permanently inflated**: Each aborted connection adds +1 that is never reversed. After N aborts the gauge reads N even when zero requests are actually in flight. Alerting rules keyed on this gauge will fire false positives indefinitely.
- **Complete metric blindspot**: Every aborted request is invisible to `api_all_request`, `api_all_errors`, `api_all_client_error`, `api_request`, `api_request_duration_milliseconds`, etc. An attacker can make a large fraction of real traffic disappear from Prometheus/Grafana dashboards, undermining SLO tracking and incident detection.
- **Severity**: Medium. No data is corrupted and no privilege escalation occurs, but operational visibility is permanently degraded in a way that is difficult to detect and easy to trigger at scale.

### Likelihood Explanation
- **No privilege required**: Any unauthenticated HTTP client can abort a connection.
- **Trivially repeatable**: A single `curl --max-time 0` or a script sending TCP RST after the HTTP request line is sufficient. Automated tools (scanners, load generators, misbehaving CDN clients) trigger this accidentally in production.
- **Scalable**: An attacker can open thousands of connections per second and abort them, driving `inFlightCounter` to arbitrarily large values within seconds.

### Recommendation
Add a `close` event listener that fires the same cleanup/recording logic when the client disconnects before `finish`:

```js
// rest/middleware/metricsHandler.js  ~line 185
let finished = false;

const record = () => {
  if (finished) return;
  finished = true;
  inFlightCounter.add(-1);
  const duration = Date.now() - startTime;
  const path = toOpenApiPath(req, res);
  const code = String(res.statusCode);
  const method = req.method;
  const labels = {method, path, code};
  allRequestCounter.add(1);
  if (res.statusCode >= 200 && res.statusCode < 300) {
    allSuccessCounter.add(1);
  } else if (res.statusCode >= 400 && res.statusCode < 500) {
    allClientErrorCounter.add(1);
    allErrorCounter.add(1);
  } else if (res.statusCode >= 500) {
    allServerErrorCounter.add(1);
    allErrorCounter.add(1);
  }
  requestTotalCounter.add(1, labels);
  durationHistogram.record(duration, labels);
  requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
  responseSizeHistogram.record(responseSize, labels);
};

res.on('finish', record);
res.on('close', record);   // fires when client disconnects before finish
```

The `finished` guard ensures the logic runs exactly once regardless of which event fires first.

### Proof of Concept
```bash
# Terminal 1 – start the mirror-node REST service normally

# Terminal 2 – send a request and immediately abort (Linux)
python3 - <<'EOF'
import socket, time

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(('localhost', 5551))
# Send a valid HTTP/1.1 request header only
s.sendall(b"GET /api/v1/transactions HTTP/1.1\r\nHost: localhost\r\n\r\n")
# Immediately reset the connection before the server responds
s.setsockopt(socket.SOL_SOCKET, socket.SO_LINGER, b'\x01\x00\x00\x00\x00\x00\x00\x00')
s.close()
EOF

# Repeat 100 times
for i in $(seq 1 100); do python3 abort.py; done

# Terminal 3 – scrape metrics
curl -s http://localhost:5551/swagger/metrics/ | grep api_all_request_in_processing_total
# Expected: value == 100 (and growing), never returns to 0
# Also confirm api_all_request_total is 0 (requests invisible to all counters)
```
