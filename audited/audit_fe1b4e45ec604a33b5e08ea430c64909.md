### Title
Unauthenticated Thundering Herd DoS via Uncached `getLatestNetworkStake()` on `GET /api/v1/network/stake`

### Summary
`NetworkServiceImpl.getLatestNetworkStake()` executes a correlated subquery against the database on every invocation with no caching and no rate limiting. The `GET /api/v1/network/stake` endpoint is publicly accessible without authentication. A single unprivileged attacker can flood this endpoint with concurrent requests, exhausting the shared database connection pool and denying service to all other database-dependent operations across the mirror node.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (line 127–130) calls `networkService.getLatestNetworkStake()` on every HTTP request with no guard.

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) unconditionally delegates to `networkStakeRepository.findLatest()`:

```java
// NetworkServiceImpl.java lines 52-56
public NetworkStake getLatestNetworkStake() {
    return networkStakeRepository
            .findLatest()
            .orElseThrow(() -> new EntityNotFoundException("No network stake data found"));
}
```

`NetworkStakeRepository.findLatest()` (lines 12–19) executes a correlated subquery on every call:

```sql
select * from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
```

**Root cause — two failed assumptions:**
1. *Assumed caching exists*: A grep for `@Cacheable`, `CacheManager`, and `@EnableCaching` across all of `rest-java/src/main/java/` returns **zero matches**. The grpc module has a `NODE_STAKE_CACHE` with `@Cacheable` on its equivalent repository method, but the rest-java `NetworkStakeRepository` has no such annotation.
2. *Assumed rate limiting exists*: A grep for `RateLimiter`, `bucket4j`, `Bucket`, and `RateLimit` across all of `rest-java/src/main/java/` returns **zero matches**. The `ThrottleConfiguration` and `ThrottleManagerImpl` that use bucket4j exist only in the `web3` module and are not applied to rest-java endpoints.

The `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — no throttle filter. `WebMvcConfiguration` registers only argument resolvers and formatters — no rate limiting interceptor.

### Impact Explanation
Every concurrent request to `GET /api/v1/network/stake` holds a JDBC connection from the shared HikariCP pool (default size: 10) for the duration of the correlated subquery. Under flood conditions:
- The connection pool is exhausted.
- All other DB-dependent endpoints (fee estimation via `FeeEstimationService`, supply queries, node lookups) queue indefinitely or throw connection-timeout exceptions.
- Fee and staking reward calculations that depend on current network stake data return errors, directly impacting transaction processing correctness for clients relying on the mirror node API.
- The network_stake table scan with a correlated `MAX()` subquery is not a trivial index lookup; under concurrent load it also generates lock contention and I/O pressure on the database.

**Severity: High** — complete denial of service for all DB-backed endpoints achievable by a single unauthenticated attacker.

### Likelihood Explanation
- **Precondition**: None. The endpoint is public, unauthenticated, and accepts GET requests with no parameters.
- **Attacker capability**: Any script kiddie with `ab`, `wrk`, or `hey` can execute this. No credentials, no special knowledge, no exploit chain required.
- **Repeatability**: Fully repeatable and automatable. The attacker can sustain the flood indefinitely.
- **Detection evasion**: Requests are indistinguishable from legitimate polling; no anomalous payload is needed.

### Recommendation
1. **Add caching** on `NetworkStakeRepository.findLatest()` or in `NetworkServiceImpl.getLatestNetworkStake()` using `@Cacheable` with a short TTL (e.g., 30–60 seconds), mirroring the pattern already used in the grpc module's `NodeStakeRepository.findAllStakeByConsensusTimestamp()`.
2. **Add rate limiting** to the rest-java module. Apply a per-IP or global bucket4j rate limiter (as already done in the web3 module's `ThrottleConfiguration`) via a servlet filter or Spring MVC interceptor registered in `WebMvcConfiguration` or `RestJavaConfiguration`.
3. **Tune the connection pool** with a `connectionTimeout` and `maximumPoolSize` appropriate for expected concurrency, and consider a separate read-only pool for read endpoints.

### Proof of Concept
```bash
# Flood GET /api/v1/network/stake with 500 concurrent connections, 10000 total requests
ab -n 10000 -c 500 http://<mirror-node-host>/api/v1/network/stake

# Simultaneously observe that other endpoints begin timing out:
curl http://<mirror-node-host>/api/v1/network/fees
# Expected: connection timeout or 500 error due to exhausted DB pool
```
