### Title
Unbounded Historical Message Retrieval via Arbitrary `startTime` Enables Resource Exhaustion DoS

### Summary
Any unprivileged user can subscribe to a topic with `startTime=0` (or any arbitrarily old timestamp), causing `PollingTopicMessageRetriever` to page through every historical message ever stored for that topic with no total-count cap. The throttled retrieval path runs indefinitely at `Long.MAX_VALUE` repeats, holding database connections, server threads, and memory for the entire drain duration. Multiple concurrent subscriptions amplify the impact to a full service denial.

### Finding Description

**Exact code path:**

`TopicMessageFilter.isValidStartTime()` only rejects future timestamps: [1](#0-0) 

`startTime=0` (Unix epoch) passes this check because `0 <= DomainUtils.now()`. There is no lower-bound constraint.

`subscribeTopic()` immediately passes the user-supplied filter directly to the retriever with `throttled=true`: [2](#0-1) 

In `PollingTopicMessageRetriever`, the throttled path sets `numRepeats = Long.MAX_VALUE` and `maxPageSize = 1000`: [3](#0-2) 

The `isComplete()` check for throttled mode only terminates when the last page returned fewer rows than `maxPageSize` (i.e., the DB is exhausted) or a user-supplied `limit` is hit: [4](#0-3) 

If no `limit` is set and the topic has millions of messages, the retriever will issue DB queries indefinitely (1000 rows every 2 seconds) until all historical messages are drained. The live Flux does not start until `historical.concatWith(...)` completes: [5](#0-4) 

**Root cause:** The `TopicMessageFilter` validation enforces `startTime >= 0` and `startTime <= now()` but imposes no minimum age floor. The retriever has no per-subscription total-message cap independent of a user-supplied `limit`. The `timeout` property (default 60s) only fires when no message is emitted for 60 seconds — it does not trigger while messages are actively flowing. [6](#0-5) 

### Impact Explanation

A topic on a production network (e.g., Hedera mainnet) with millions of messages becomes a weapon. Each subscription with `startTime=0` and no limit:
- Holds a bounded-elastic scheduler thread and a DB connection for the full drain duration (hours for large topics)
- Streams all rows through server memory before forwarding to the client
- Generates sustained sequential DB read load proportional to topic history depth

With `maxConcurrentCallsPerConnection=5` (default) and no per-IP connection limit enforced at the gRPC layer, an attacker opens N connections × 5 streams each, multiplying the load linearly. This exhausts the DB connection pool, thread pool, and network bandwidth, degrading or denying service to all legitimate subscribers. [7](#0-6) 

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic`. The gRPC endpoint is publicly accessible. The only precondition is knowledge of a valid topic ID with substantial history — trivially discoverable via the mirror node REST API. The attack is fully repeatable, scriptable, and requires no special tooling beyond a standard gRPC client. A single attacker with a handful of connections can sustain the attack indefinitely.

### Recommendation

1. **Enforce a maximum historical lookback window**: Add a configurable `maxStartTimeAge` property (e.g., 7 days) and reject filters where `DomainUtils.now() - startTime > maxStartTimeAge` unless a `limit` is also set.
2. **Cap total messages per historical retrieval**: Add a server-side `maxHistoricalMessages` property (e.g., 10,000) enforced in `PollingTopicMessageRetriever.retrieve()` regardless of user-supplied `limit`.
3. **Require a `limit` when `startTime` is older than a threshold**: In `TopicMessageFilter` validation, add a cross-field constraint: if `startTime < (now - threshold)`, then `limit > 0` must hold.
4. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# startTime = Unix epoch (0 seconds) — passes isValidStartTime() since 0 <= now()
# No limit set — no user-supplied cap
request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=<HIGH_VOLUME_TOPIC_ID>),
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0)
    # consensusEndTime and limit intentionally omitted
)

# Open 5 streams per connection, repeat across N connections
for msg in stub.subscribeTopic(request):
    pass  # server streams all historical messages, holding resources indefinitely
```

**Trigger:** Submit the above against any topic with >1M messages. Observe sustained DB query load (1000-row pages every 2s per stream), growing memory usage on the gRPC server, and degraded response times for legitimate subscribers.
