### Title
Unauthenticated DB Query Flooding via AccountBalance BlockIdentifier.Hash — No Application-Level Rate Limiting

### Summary
The `/account/balance` endpoint accepts an unauthenticated `BlockIdentifier.Hash`-only request that unconditionally triggers a database query against the `record_file` table. There is no application-level rate limiting or hash format/length validation beyond an empty-string check. An attacker can flood the endpoint with many concurrent requests carrying distinct non-empty hash values, exhausting DB connection pool resources and increasing node resource consumption well beyond 30%.

### Finding Description
**Exact code path:**

1. `AccountBalance()` in `rosetta/app/services/account_service.go` lines 52–53: when `request.BlockIdentifier != nil`, delegates unconditionally to `a.RetrieveBlock(ctx, request.BlockIdentifier)`. [1](#0-0) 

2. `RetrieveBlock()` in `rosetta/app/services/base_service.go` lines 84–86: when only `bIdentifier.Hash != nil`, calls `tools.SafeRemoveHexPrefix(*bIdentifier.Hash)` then `b.blockRepo.FindByHash(ctx, h)`. [2](#0-1) 

3. `FindByHash()` in `rosetta/app/persistence/block.go` lines 135–144: the only guard is `hash == ""`; any non-empty string passes through to `findBlockByHash()`. [3](#0-2) 

4. `findBlockByHash()` in `rosetta/app/persistence/block.go` lines 226–241: executes `db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb)` — a live DB query for every call. [4](#0-3) 

5. The SQL is `WHERE hash = @hash COLLATE "C"` — a full table scan or index lookup per request, with no caching layer. [5](#0-4) 

**Root cause:** No application-level rate limiting exists in the Rosetta server. `main.go` wires only a metrics middleware; no throttle, no per-IP limiter, no request-count cap. [6](#0-5) 

The Traefik `rateLimit` and `inFlightReq` entries in the Helm chart are optional infrastructure-layer defaults that are not enforced at the application level and can be absent in non-Kubernetes or custom deployments. [7](#0-6) 

**Why the `0x`-stripped input matters:** `SafeRemoveHexPrefix` is a no-op for strings without the prefix, so a hash submitted without `0x` passes through unchanged and still satisfies the `hash != ""` guard, reaching the DB query. No additional validation rejects it. [8](#0-7) 

### Impact Explanation
Each distinct hash value sent in `BlockIdentifier.Hash` causes one synchronous DB query. With N concurrent goroutines sending requests, the DB connection pool is saturated proportionally. Because the `record_file` table grows with chain history, the `COLLATE "C"` equality scan may not always hit a hot index page for random/nonexistent hashes, increasing I/O. Sustained flooding raises CPU (query planning/execution), memory (connection pool, GORM result buffers), and I/O on the database host — all without any authentication or privilege. A 30%+ resource increase is achievable with modest concurrency (tens of requests/second) on a standard deployment.

### Likelihood Explanation
The endpoint is publicly reachable with no credentials. The attack requires only an HTTP client capable of sending POST requests to `/account/balance` with a JSON body containing `block_identifier: { hash: "<random_string>" }`. No chain knowledge is needed — random strings satisfy the only guard (`hash != ""`). The attack is trivially scriptable, repeatable, and requires no special tooling.

### Recommendation
1. **Application-level rate limiting:** Add a per-IP or global rate-limiting middleware directly in the Go server (e.g., `golang.org/x/time/rate` or a middleware like `go-chi/httprate`) so protection is not dependent on optional infrastructure.
2. **Hash format validation:** Before issuing a DB query, validate that the hash is a well-formed hex string of the expected length (e.g., 96 hex chars for Hedera record-file hashes). Reject malformed values with `ErrInvalidArgument` before touching the DB.
3. **Request timeout enforcement:** Ensure `GetDbWithContext` uses a short, bounded timeout so runaway queries do not hold connections indefinitely.

### Proof of Concept
```bash
# Send 500 concurrent AccountBalance requests with distinct random hashes
# No authentication required
for i in $(seq 1 500); do
  curl -s -X POST http://<rosetta-host>/account/balance \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\": {\"address\":\"0.0.2\"},
      \"block_identifier\": {\"hash\": \"$(head -c 48 /dev/urandom | xxd -p | tr -d '\n')\"}
    }" &
done
wait
```
Each iteration sends a unique non-empty hash (no `0x` prefix) that passes the `hash != ""` guard, reaches `findBlockByHash`, and issues a live `SELECT … FROM record_file WHERE hash = @hash COLLATE "C"` query. Monitoring DB CPU/connection count during the flood will show the resource spike.
