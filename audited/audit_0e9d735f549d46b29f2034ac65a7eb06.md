### Title
Oversized `to` Field Allocated in Heap Before `@Hex` Validation Rejects It — Bounded Memory Exhaustion via `ContractCallRequest`

### Summary
`JacksonConfiguration` sets a global `maxStringLength` equal to `ethereumMaxCallDataSize() * 2 + 1024` (~263 KB) to accommodate the `data` field, but this limit applies uniformly to every string field in the parsed JSON, including `to`. Because Jackson fully allocates the string into a Java heap `String` object before Spring's `@Hex(minLength=40, maxLength=40, allowEmpty=true)` validator runs, an unauthenticated attacker can force the server to allocate ~263 KB per request for the `to` field alone, which is then immediately discarded after validation failure. Under the default rate limit of 500 RPS this creates sustained GC pressure and can degrade throughput.

**Root cause:** `maxStringLength` is a single global Jackson constraint sized for the `data` field (up to ~262 KB of hex-encoded call data). Jackson does not enforce per-field string length limits; it allocates the full string into the JVM heap before returning control to the deserializer. Spring's `@Validated` / `@Hex` constraint fires only after the `ContractCallRequest` POJO is fully constructed. The `to` field legitimately needs at most 42 characters (`0x` + 40 hex digits), but Jackson will happily allocate up to `maxSize - JSON_overhead` (~263 KB) for it.

**Why `maxDocumentLength` does not save you:** `maxDocumentLength` caps the total byte count of the entire JSON document at `maxSize`. A crafted document of the form `{"to":"<263 KB − 9 bytes of overhead>"}` fits exactly within `maxDocumentLength` while placing the entire budget into the `to` string. Jackson reads and allocates that string completely before the document-length counter would overflow. [4](#0-3) 

### Impact Explanation
Each malicious request forces a ~263 KB heap allocation that survives until the next GC cycle. At the documented default of 500 RPS (`hiero.mirror.web3.throttle.requestsPerSecond = 500`) an attacker sustains ~131 MB/s of short-lived heap churn. This does not cause an immediate OOM (the JVM will GC), but it causes continuous minor/major GC pauses, elevated CPU usage from GC, and latency spikes for legitimate callers. On memory-constrained deployments (e.g., containers with a 512 MB heap limit) it can push the JVM into continuous full-GC mode, effectively denying service.

### Likelihood Explanation
The endpoint (`POST /api/v1/contracts/call`) is public and requires no authentication. The exploit requires only a single HTTP client capable of sending ~263 KB POST bodies at high rate. No special knowledge of the system is needed beyond the public OpenAPI spec. The attack is trivially repeatable and scriptable.

### Recommendation
Add a per-field Jackson `@JsonProperty` size guard **before** deserialization, or — more robustly — introduce a dedicated `StringSizeDeserializer` that rejects strings exceeding a per-field threshold. For the `to` and `from` fields the threshold should be 42 characters (the maximum valid EVM address with `0x` prefix). Alternatively, register a custom `DeserializationProblemHandler` that checks string length against the target field's `@Hex.maxLength` annotation during deserialization, before the string is interned. At minimum, set a tighter `maxStringLength` for the `ContractCallRequest` endpoint specifically (e.g., via a dedicated `ObjectMapper` bean scoped to that controller), sized to the largest legitimate field (`data`, ~262 KB) rather than applying the same limit to address fields.

### Proof of Concept
```bash
# Assuming ethereumMaxCallDataSize() = 131072 → maxSize = 263168
# Craft a to value of 263159 hex chars (fills document to exactly maxSize bytes)
python3 -c "
import sys, json
payload = json.dumps({'to': 'a' * 263159})
sys.stdout.buffer.write(payload.encode())
" > /tmp/big_to.json

# Send at high rate
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST https://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    --data-binary @/tmp/big_to.json &
done
wait
# Monitor GC: jstat -gcutil <pid> 1s
# Observe: FGC count rising, latency of legitimate calls increasing
```

The server will respond `400 Bad Request` (hex validation failure) for each request, but only after allocating and then discarding ~263 KB per request, creating sustained GC pressure.
