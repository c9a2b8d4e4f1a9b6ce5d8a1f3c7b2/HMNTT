### Title
Unbounded Subscriber Fan-Out via Unlimited `.share()` Subscriptions in `RedisTopicListener.subscribe()` Enables CPU and Memory Exhaustion

### Summary
`RedisTopicListener.subscribe()` creates a single shared `Flux` per topic using `.share()` with no cap on the number of downstream subscribers. Because there is no global limit on concurrent gRPC connections or total active subscriptions, an unauthenticated attacker can open arbitrarily many TCP connections—each carrying up to 5 streams—all subscribing to the same topic, causing O(N) fan-out CPU work per Redis message and O(N × 16384) message-buffer memory growth that exhausts the mirror node.

### Finding Description

**Exact code path:**

`RedisTopicListener.subscribe()` (lines 68–80) terminates the pipeline with `.share()`:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    ...
    return container
        .flatMapMany(r -> r.receive(...))
        .map(Message::getMessage)
        ...
        .retryWhen(...)
        .share();          // ← no subscriber cap
}
``` [1](#0-0) 

`getSharedListener()` stores exactly one such `Flux` per topic in `topicMessages` via `computeIfAbsent`, so every caller for the same topic attaches to the same shared `Flux`: [2](#0-1) 

`SharedTopicListener.listen()` wraps each subscription with a per-subscriber `onBackpressureBuffer(maxBufferSize)` (default 16 384 slots) and `publishOn(boundedElastic)`: [3](#0-2) 

**Root cause:** Reactor's `share()` operator synchronously iterates every registered inner subscriber on the upstream (Redis listener) thread and enqueues the message into each subscriber's `publishOn` queue. With N subscribers this is O(N) work per message on that thread, plus N independent heap buffers of up to 16 384 messages each.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (the only server-side limit) is **per TCP connection**, not global. An attacker opens many connections. [4](#0-3) [5](#0-4) 

- `subscriberCount` in `TopicMessageServiceImpl` is a **metrics gauge only**—no enforcement, no rejection threshold. [6](#0-5) 

- The only gRPC interceptor sets an endpoint-context string; it performs no authentication or rate limiting. [7](#0-6) 

- Rate limiting (bucket4j) exists only in the `web3` module; the `grpc` module has none.

- The GCP gateway `maxRatePerEndpoint: 250` limits new-request initiation rate, not total concurrent long-lived streaming connections. An attacker can ramp up slowly and maintain thousands of open streams indefinitely. [8](#0-7) 

### Impact Explanation
With 1 000 TCP connections × 5 streams = 5 000 subscribers on one topic:
- **CPU**: every Redis message triggers a synchronous fan-out loop over 5 000 subscriber queues on the Redis listener thread, starving all other reactive pipelines.
- **Memory**: 5 000 × 16 384 slots × ~200 bytes/message ≈ **~16 GB** of heap pressure, triggering GC storms or OOM.
- **Thread pool exhaustion**: `Schedulers.boundedElastic()` (default cap: 10 × CPU cores threads, up to 100 000 queued tasks) becomes saturated, blocking legitimate subscriber delivery.

The mirror node becomes unresponsive to all gRPC clients, constituting a full service denial for the HCS subscription API.

### Likelihood Explanation
The attack requires zero privileges: the `subscribeTopic` endpoint is unauthenticated and publicly reachable. Any valid topic ID is observable on-chain. A single attacker machine can open thousands of TCP connections with standard tooling (e.g., `grpcurl` in a loop, or a trivial Go/Java client). The GCP rate limit of 250 req/s means 5 000 streams can be established in 20 seconds. Streams persist indefinitely (no forced `endTime`), so the attacker need not sustain high connection rate—just hold the sockets open.

### Recommendation
1. **Enforce a global concurrent-subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 10 000), returning `RESOURCE_EXHAUSTED`.
2. **Add a per-IP or per-connection subscription limit** via a gRPC `ServerInterceptor` that tracks active streams per remote address.
3. **Add a `maxSubscribers` property to `ListenerProperties`** and enforce it in `SharedTopicListener.listen()` before attaching to the shared `Flux`.
4. **Apply a gRPC-layer rate limiter** (e.g., bucket4j or Resilience4j) on `subscribeTopic` calls, mirroring the pattern already used in the `web3` module.
5. Consider replacing unbounded `.share()` with `publish().autoConnect()` combined with a `take(maxSubscribers)` guard, or use a custom `FluxSink`-based multicast that enforces a subscriber ceiling.

### Proof of Concept
```bash
# Step 1: identify any valid topic (publicly visible on-chain, e.g., topic 0.0.41110)
# Step 2: open 1000 connections, each with 5 infinite streams (no endTime, no limit)
for i in $(seq 1 1000); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":41110},"limit":0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# Step 3: observe mirror node CPU spike to 100% and heap growth toward OOM
# Step 4: verify legitimate subscribers receive no messages / connection timeouts
```

All 5 000 background streams attach to the single `Flux` returned by `topicMessages.computeIfAbsent("topic.41110", ...)`. Each subsequent Redis publish triggers a synchronous 5 000-entry fan-out loop, exhausting CPU and memory until the JVM crashes or the node becomes unresponsive.

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
