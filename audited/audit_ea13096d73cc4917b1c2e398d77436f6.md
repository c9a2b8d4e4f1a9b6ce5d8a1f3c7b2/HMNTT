### Title
Thundering Herd DB Reconnection via Unauthenticated High-Frequency `getByIdAndType()` Requests During Network Partition Recovery

### Summary
The GraphQL module's `getByIdAndType()` method issues an uncached, direct database query on every invocation with no rate limiting, no circuit breaker, and no per-IP throttle. During a network partition, HikariCP queues all incoming requests waiting for a connection; when the database recovers, every queued request simultaneously fires a reconnection and query, overwhelming the database server at the worst possible moment.

### Finding Description

**Exact code path:**

`AccountController.account()` → `EntityService.getByIdAndType()` → `EntityRepository.findById()` → direct SQL to DB. [1](#0-0) 

`EntityRepository` in the graphql module has **no `@Cacheable` annotations** — every call hits the database: [2](#0-1) 

(Contrast with the `web3` module's `EntityRepository`, which has extensive `@Cacheable` coverage — the graphql module has none.)

**No rate limiting exists in the graphql module.** The only throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module: [3](#0-2) 

The graphql module's `GraphQlConfiguration` only enforces query complexity (max 200) and depth (max 10) — neither limits request *rate*: [4](#0-3) 

The datasource uses HikariCP with no circuit breaker: [5](#0-4) 

`DatabaseWaiter` only runs at startup and does not protect the runtime request path: [6](#0-5) 

**Root cause:** The graphql service path has no rate limiting, no response caching, and no circuit breaker. HikariCP queues all threads waiting for a connection during a partition (up to `connectionTimeout`, default 30 s). On DB recovery, all queued threads are simultaneously unblocked and each fires `SELECT * FROM entity WHERE id = ?`, creating a thundering herd.

### Impact Explanation
When the database comes back online after a partition, the burst of simultaneous queries and connection acquisitions can cause the database to OOM, crash, or enter a degraded state again — extending the outage. This is a denial-of-service amplification: a modest sustained request rate (e.g., 100 req/s) during a 60-second partition queues 6,000 simultaneous DB queries at recovery time. The graphql module's design document itself acknowledges rate limiting should be at the load balancer, meaning the application has no self-defense. [7](#0-6) 

### Likelihood Explanation
The GraphQL endpoint requires no authentication. Any external user can send `POST /graphql` with a valid `account(input: {entityId: {num: 1}})` query. Network partitions between the application tier and database tier are a realistic operational event (rolling restarts, failovers, cloud AZ issues). The attacker does not need to know a partition is occurring — sustained high-frequency polling is sufficient. The attack is trivially scriptable with `curl` or any HTTP client.

### Recommendation
1. **Add rate limiting to the graphql module** — port the `bucket4j`-based `ThrottleConfiguration` from `web3` or add a servlet filter analogous to the `web3` `RequestFilter`.
2. **Add a circuit breaker** (e.g., Resilience4j `@CircuitBreaker`) around `entityRepository.findById()` in `EntityServiceImpl` to fail-fast during partition and prevent request queuing.
3. **Add response caching** on `EntityRepository.findById()` in the graphql module (mirroring the `web3` module's `@Cacheable` pattern) to reduce DB load under normal and recovery conditions.
4. **Configure HikariCP `connectionTimeout`** to a short value (e.g., 2–5 s) so requests fail fast rather than queuing for 30 s.

### Proof of Concept
```bash
# During or just before DB partition recovery, run from any unauthenticated client:
while true; do
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input:{entityId:{num:1}}) { id } }"}' &
done
# Hundreds of goroutines queue on HikariCP.
# When DB recovers, all fire simultaneously → thundering herd.
```

Observe DB connection count spike and query queue depth at recovery time via `pg_stat_activity` or HikariCP metrics (`hikaricp.connections.pending`).

    public void waitForDatabase(String jdbcUrl, String username, String password) {
        final var properties = commonProperties.getDatabaseStartup();
        if (!properties.isEnabled()) {
            log.info("Database startup wait is disabled");
            return;
        }

        final var deadline = System.nanoTime() + properties.getTimeout().toNanos();
        final var jdbcProperties = new Properties();

        if (StringUtils.hasText(username)) {
            jdbcProperties.setProperty("user", username);
        }

        if (password != null) {
            jdbcProperties.setProperty("password", password);
        }

        jdbcProperties.setProperty(
                "connectTimeout", String.valueOf(properties.getConnectTimeout().toSeconds()));
        jdbcProperties.setProperty(
                "socketTimeout", String.valueOf(properties.getSocketTimeout().toSeconds()));

        Exception lastException = null;
        while (System.nanoTime() < deadline) {
            try (final var connection = DriverManager.getConnection(jdbcUrl, jdbcProperties)) {
                if (connection.isValid(
                        Math.toIntExact(properties.getValidationTimeout().toSeconds()))) {
                    log.info("Successfully connected to database");
                    return;
                }

                log.info("Database connection not valid yet. Retrying in {}", properties.getInterval());
            } catch (Exception e) {
                lastException = e;
                log.info(
                        "Database connection not ready yet. Retrying in {}: {}",
                        properties.getInterval(),
                        e.getMessage());
            }

            Uninterruptibles.sleepUninterruptibly(properties.getInterval());
        }

        throw new IllegalStateException(
                "Database did not become available within " + properties.getTimeout(), lastException);
    }
