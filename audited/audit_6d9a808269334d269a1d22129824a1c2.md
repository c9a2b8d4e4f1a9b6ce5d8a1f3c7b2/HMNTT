### Title
Unauthenticated Disclosure of Internal Pricing Metadata (`high_volume` / `high_volume_pricing_multiplier`) via Public Transaction API

### Summary
The `formatTransactionRows()` function in `rest/transactions.js` unconditionally includes `high_volume` and `high_volume_pricing_multiplier` fields in every transaction response object. These fields are returned by the public, unauthenticated `/api/v1/transactions` and `/api/v1/transactions/{id}` endpoints with no access control, role check, or field-level filtering. Any external user can query these endpoints and receive internal network pricing metadata that can be used to infer whether a transaction involved a high-value transfer.

### Finding Description
**Exact code path:**

In `rest/transactions.js`, `formatTransactionRows()` (lines 191–225) builds the response object for every transaction row:

```js
// rest/transactions.js lines 204-205
high_volume: row.high_volume ?? false,
high_volume_pricing_multiplier: row.high_volume_pricing_multiplier,
``` [1](#0-0) 

These fields are fetched from the database because they are included in `transactionFields` (lines 54–55), which feeds `transactionFullFields` used in all SQL queries:

```js
// rest/transactions.js lines 54-55
Transaction.HIGH_VOLUME,
Transaction.HIGH_VOLUME_PRICING_MULTIPLIER,
``` [2](#0-1) 

The two public handler functions — `getTransactions` (line 671) and `getTransactionsByIdOrHash` (line 923) — both call `formatTransactionRows()` and return its output directly to the HTTP response with no authentication or authorization middleware: [3](#0-2) [4](#0-3) 

A grep for `authentication`, `auth`, `middleware`, `role`, or `privilege` in `transactions.js` returns zero matches — there is no access control of any kind on these fields or these endpoints.

**Root cause:** The failed assumption is that `high_volume` and `high_volume_pricing_multiplier` are internal-only fields that should not be surfaced to unprivileged callers. No filtering or redaction is applied before the response is serialized.

### Impact Explanation
`high_volume` is a boolean flag set by the network when a transaction triggers volume-based pricing (i.e., when the transfer amount crosses a threshold). `high_volume_pricing_multiplier` is the exact multiplier applied. Together, these fields allow any observer to:

1. Determine that a specific transaction involved a transfer large enough to trigger volume-based pricing.
2. Use the multiplier value to narrow down the transfer amount range, since the multiplier is a function of the transfer size.
3. Correlate this with other public fields (account IDs, timestamps, token IDs) to build a detailed picture of high-value financial activity on the network.

This is a **medium-severity information disclosure** vulnerability. It does not allow fund theft or transaction manipulation, but it leaks internal pricing decisions and enables transfer-amount inference for any transaction — including those where the parties may have expected financial privacy.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and zero special tooling. Any HTTP client can issue:

```
GET /api/v1/transactions
GET /api/v1/transactions/{transactionId}
```

and receive `high_volume` and `high_volume_pricing_multiplier` in the JSON response. The attack is trivially repeatable, fully automated, and requires no interaction with the target accounts. The only precondition is network access to the mirror node's REST API, which is publicly accessible by design.

### Recommendation
Apply field-level access control before serializing the response. Specifically:

1. **Remove the fields from the public response model** if they are not intended for external consumers. Delete `high_volume` and `high_volume_pricing_multiplier` from the object literal in `formatTransactionRows()` (lines 204–205) and from `transactionFields` (lines 54–55) so they are never fetched or returned.
2. **If the fields must be retained for internal/privileged use**, introduce a middleware or response-shaping layer that strips these fields unless the caller presents a valid privileged credential (e.g., an operator API key). Do not rely on the absence of documentation as a security control.
3. **Update the OpenAPI spec** (`rest/api/v1/openapi.yml`) to remove or mark these fields as internal-only so they are not inadvertently re-exposed in future changes.

### Proof of Concept
**Preconditions:** Network access to a running mirror node REST API instance.

**Steps:**

```bash
# Step 1: Query the public transactions endpoint (no credentials required)
curl -s "https://<mirror-node-host>/api/v1/transactions?limit=1" | jq '.transactions[0] | {high_volume, high_volume_pricing_multiplier}'

# Expected output (example):
# {
#   "high_volume": true,
#   "high_volume_pricing_multiplier": 2
# }

# Step 2: Query a specific transaction by ID
curl -s "https://<mirror-node-host>/api/v1/transactions/0.0.12345-1234567890-000000000" \
  | jq '.transactions[0] | {high_volume, high_volume_pricing_multiplier}'
```

**Result:** The response unconditionally includes `high_volume: true/false` and the numeric `high_volume_pricing_multiplier`, revealing internal pricing metadata to any unauthenticated caller. No error, no redaction, no rate-limiting specific to these fields.
