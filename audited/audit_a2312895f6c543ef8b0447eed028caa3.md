### Title
Ghost Account Enumeration via Numeric ID Bypass in `getAccountTokenAllowances`

### Summary
In `getAccountTokenAllowances()`, the call to `EntityService.getEncodedId()` with a numeric account ID performs only a mathematical encoding and skips any database existence check. As a result, querying token allowances for a non-existent numeric account ID returns HTTP 200 with `{ "allowances": [] }` — identical to a valid account that has no allowances — misrepresenting Hashgraph ledger history to any caller.

### Finding Description

**Exact code path:**

`rest/controllers/tokenAllowanceController.js`, line 69:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
``` [1](#0-0) 

This passes the raw path parameter to `EntityService.getEncodedId()`. Inside that method (`rest/service/entityService.js`, lines 120–123):

```js
if (EntityId.isValidEntityId(entityIdString)) {
    const entityId = EntityId.parseString(entityIdString, {paramName});
    return entityId.evmAddress === null
      ? entityId.getEncodedId()   // ← pure math, no DB lookup
      : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
``` [2](#0-1) 

For a numeric input like `"99999999"` or `"0.0.99999999"`, `evmAddress` is `null`, so the branch returns `entityId.getEncodedId()` — a pure bit-shift calculation — **without touching the database**.

**Contrast with other input types:**
- EVM address → `getEntityIdFromEvmAddress()` → DB lookup → throws `NotFoundError` if absent.
- Alias → `getAccountIdFromAlias()` → DB lookup → throws `NotFoundError` if absent. [3](#0-2) 

The existence check `isValidAccount()` exists on `EntityService` but is never called in this flow. [4](#0-3) 

**Downstream effect:**

`TokenAllowanceService.getAccountTokenAllowances()` queries `token_allowance` with `owner = $1` (the fabricated encoded ID). Since no rows exist for a non-existent account, it returns `[]`. [5](#0-4) 

The controller then responds with HTTP 200 and `{ "allowances": [], "links": { "next": null } }`. [6](#0-5) 

**Confirmed by test:** The test named `"EntityService.getEncodedId - No match"` (entity absent from DB) asserts that the call **resolves successfully** for a numeric ID — explicitly documenting the missing existence check as accepted behavior. [7](#0-6) 

### Impact Explanation

The mirror node's contract is to faithfully reflect Hashgraph ledger history. The OpenAPI spec for this endpoint lists both `200 OK` and `404 Not Found` as valid responses, implying callers expect a 404 for non-existent accounts. [8](#0-7) 

Because a non-existent account and a valid account with zero allowances both return `HTTP 200 { "allowances": [] }`, any downstream system (wallets, DeFi protocols, compliance tools, block explorers) that relies on this endpoint to verify account existence or allowance state receives a false affirmative. This constitutes a **data-integrity / history-misrepresentation** issue: the API asserts an account exists and has no allowances when in fact the account has never existed on the ledger.

### Likelihood Explanation

Preconditions: none. Any unauthenticated user with network access can trigger this. The input is a simple integer in the URL path. No special knowledge, credentials, or tooling is required. The attack is trivially repeatable across any numeric account ID range and is completely silent — no errors, no logs distinguishing it from a legitimate empty-result query.

### Recommendation

After `getEncodedId()` returns for a numeric input, add an explicit existence check before proceeding to the allowance query. The method `EntityService.isValidAccount()` already exists for this purpose:

```js
getAccountTokenAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);

  // Add existence check for numeric IDs (getEncodedId does not verify these against the DB)
  if (!(await EntityService.isValidAccount(accountId))) {
    throw new NotFoundError();
  }

  const filters = utils.buildAndValidateFilters(req.query, acceptedTokenAllowanceParameters);
  ...
};
```

Alternatively, modify `getEncodedId()` itself so that the numeric branch also performs a DB existence check when `requireResult = true`, making the behavior consistent across all three input types (numeric, EVM address, alias).

### Proof of Concept

**Preconditions:** Mirror node REST API is running. Account `0.0.999999999` does not exist in the entity table.

**Steps:**

1. Query a known non-existent numeric account:
   ```
   GET /api/v1/accounts/999999999/allowances/tokens
   ```

2. **Observed response (HTTP 200):**
   ```json
   {
     "allowances": [],
     "links": { "next": null }
   }
   ```

3. Query a real account that exists but has no token allowances:
   ```
   GET /api/v1/accounts/<real_account_with_no_allowances>/allowances/tokens
   ```

4. **Observed response (HTTP 200):**
   ```json
   {
     "allowances": [],
     "links": { "next": null }
   }
   ```

The two responses are byte-for-byte identical. There is no way for a caller to distinguish a non-existent account from a valid account with no allowances. Repeatable for any arbitrary numeric account ID without authentication.
