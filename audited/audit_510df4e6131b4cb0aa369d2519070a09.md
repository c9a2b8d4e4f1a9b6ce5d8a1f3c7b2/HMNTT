### Title
Missing HTTP Request Processing Timeout in rest-java Enables Thread Pool Exhaustion DoS During DB Network Partition

### Summary
The `TopicServiceImpl.findById()` method performs a synchronous, blocking JDBC call with no application-level timeout. The rest-java module exposes no HTTP request processing timeout (unlike `web3` which has `hiero.mirror.web3.requestTimeout`), so during a network partition between the application and PostgreSQL, Tomcat worker threads block indefinitely. An unauthenticated attacker sending concurrent requests to `GET /api/v1/topics/{id}` during such a partition exhausts the thread pool, rendering the server unresponsive to all clients.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` (line 32–36) calls `topicService.findById()` synchronously on the Tomcat request thread: [1](#0-0) 

`TopicServiceImpl.findById()` delegates directly to a Spring Data JPA repository with no timeout wrapper: [2](#0-1) 

**Root cause — no HTTP-level request timeout for rest-java:**

The `RestJavaProperties` class defines no `requestTimeout` field: [3](#0-2) 

The `WebMvcConfiguration` and `RestJavaConfiguration` classes configure no server-side processing timeout: [4](#0-3) [5](#0-4) 

The documented configuration table for the REST Java API lists no HTTP request timeout property at all: [6](#0-5) 

**Contrast with web3**, which explicitly configures a request timeout: [7](#0-6) 

**Why the DB `statementTimeout` does not protect against this:**

The Helm chart sets `statement_timeout = 20000` at the PostgreSQL role level: [8](#0-7) 

This is a server-side PostgreSQL timeout. During a network partition, the PostgreSQL server cannot deliver the timeout signal back to the JDBC client — the TCP socket simply hangs. The JDBC call blocks until the OS TCP keepalive fires (default: 15+ minutes on Linux), holding the Tomcat thread for the entire duration.

### Impact Explanation

Tomcat's default thread pool is 200 threads. Each blocked `findById()` call holds one thread and one open client socket. With 200 concurrent requests during a partition, the thread pool is fully exhausted. New incoming requests queue in the accept backlog; once that fills, new connections are refused. The server becomes completely unresponsive to all clients — including health checks — until the partition resolves or the OS TCP timeout fires. This is a full service-level DoS. The `GET /api/v1/topics/{id}` endpoint requires no authentication, making it trivially reachable by any external user.

### Likelihood Explanation

No privileges are required. The endpoint is publicly accessible. A network partition can occur naturally (infrastructure failure) or be induced by an attacker with network access (e.g., firewall rule injection, BGP manipulation in cloud environments). Sending 200 concurrent HTTP requests is trivial with standard tools (`ab`, `wrk`, `curl`). The attack is repeatable and does not require any knowledge of valid topic IDs — any syntactically valid ID (e.g., `0.0.1`) suffices to trigger the blocking DB call.

### Recommendation

1. **Add a Spring Boot server-side request timeout** for rest-java via `spring.mvc.async.request-timeout` or Tomcat's `server.tomcat.connection-timeout`, analogous to `hiero.mirror.web3.requestTimeout`.
2. **Configure JDBC socket timeout** in the HikariCP datasource for rest-java (e.g., `socketTimeout` in the JDBC URL or `connectionTimeout` in HikariCP config) so that in-flight queries are aborted when the TCP connection hangs.
3. **Add a `RestJavaProperties.requestTimeout` field** (mirroring `Web3Properties.requestTimeout`) and enforce it at the controller or filter level.
4. Consider wrapping the repository call in a `CompletableFuture` with a timeout or using Spring's `@Async` with a timeout, so the Tomcat thread is not held during DB unavailability.

### Proof of Concept

**Precondition:** Induce a network partition between the rest-java pod and PostgreSQL (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the app host, or a network policy change).

**Trigger:**
```bash
# Send 200 concurrent requests to exhaust the default Tomcat thread pool
for i in $(seq 1 200); do
  curl -s http://<rest-java-host>/api/v1/topics/0.0.1 &
done
wait
```

**Result:**
- All 200 Tomcat threads block in `topicRepository.findById()` waiting for the partitioned JDBC socket.
- Subsequent requests (including health checks at `/actuator/health`) receive no response or connection refused.
- Server remains unresponsive until the OS TCP keepalive timeout fires (~15 minutes) or the partition is resolved.
- Each blocked request holds an open server-side socket descriptor for the duration.

@Data
@Validated
@ConfigurationProperties("hiero.mirror.rest-java")
public class RestJavaProperties {

    @NotNull
    @Valid
    private ResponseConfig response = new ResponseConfig();

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }

    @Data
    @Validated
    public static class ResponseConfig {
        @NotNull
        @Valid
        private ResponseHeadersConfig headers = new ResponseHeadersConfig();
    }

    @Data
    @Validated
    public static class ResponseHeadersConfig {
        @NotNull
        private Map<String, String> defaults = new TreeMap<>(String.CASE_INSENSITIVE_ORDER);

        @NotNull
        private Map<String, Map<String, String>> path = new HashMap<>();

        public Map<String, String> getHeadersForPath(String apiPath) {
            return apiPath == null ? defaults : path.getOrDefault(apiPath, defaults);
        }
    }
