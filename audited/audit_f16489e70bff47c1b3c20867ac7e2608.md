### Title
Unbounded `DISTINCT ON (key)` Sort in Historical Hook Storage Query Enables Resource Exhaustion

### Summary
`HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` executes a native PostgreSQL query using `DISTINCT ON (key)` with no `ORDER BY` clause and no restriction on the width of the timestamp or key range. PostgreSQL must sort and deduplicate every row matching the WHERE clause before applying the `LIMIT` from `Pageable`, meaning the database work is proportional to the total number of matching rows, not the returned page size. An unauthenticated attacker can craft requests spanning the maximum timestamp and key range to force full-table sort operations on `hook_storage_change`, repeatedly, with no rate limiting in the application layer.

### Finding Description

**Exact code path:**

`HooksController.getHookStorage()` (line 104–130) → `HookServiceImpl.getHookStorageChange()` (lines 84–118) → `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` (lines 15–39).

**Root cause — the native query:**

```sql
select distinct on (key)
     owner_id, hook_id, key, value_written as "value", ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
```

`DISTINCT ON (key)` in PostgreSQL requires the engine to sort **all** rows satisfying the WHERE predicate by `key` before it can deduplicate and then apply `LIMIT`. There is no `ORDER BY` clause in the query, so PostgreSQL cannot use a streaming index scan to satisfy the `DISTINCT ON` incrementally — it must materialise and sort the full intermediate result set. The `Pageable` limit (max 100, from `MAX_LIMIT`) is appended by Spring Data JPA as a SQL `LIMIT` clause, but this is applied **after** the sort-and-deduplicate step, providing no protection against the scan cost.

**Failed assumption:**
The code assumes that passing a `Pageable` with a small limit bounds the database work. It does not: `DISTINCT ON` without a matching `ORDER BY` index path forces a full sort of all qualifying rows regardless of the requested page size.

**No timestamp/key range width validation:**

In `HooksController.hookStorageChangeRequest()` (lines 158–198):
- `keyLowerBound` defaults to `MIN_KEY_BYTES` (32 × `0x00`) and `keyUpperBound` defaults to `MAX_KEY_BYTES` (32 × `0xFF`) — the widest possible key range.
- The `timestamp` parameter accepts up to 2 values (`@Size(max = 2)`) but there is **no validation on the width of the resulting range**. An attacker can supply `timestamp=gte:0.000000001` and `timestamp=lte:9999999999.999999999` to span the entire ledger history.
- No rate limiting is present anywhere in the controller or service layer.

### Impact Explanation

Every request to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage?timestamp=gte:0.000000001&timestamp=lte:9999999999.999999999` with no key filter forces PostgreSQL to scan and sort the entire `hook_storage_change` partition for that `(owner_id, hook_id)` pair. On a production mirror node with millions of historical storage change records, each such request consumes significant CPU and memory on the database server. Concurrent requests from a single attacker (or a small botnet) can push database CPU well above the 30% threshold relative to baseline, degrading service for all users. The impact is a sustained, amplified DoS against the database tier with minimal attacker cost (a single HTTP GET per request, no authentication required).

### Likelihood Explanation

The endpoint is public and unauthenticated (mirror node REST API). The exploit requires no special knowledge beyond knowing the `ownerId` and `hookId` of any account with a non-trivial history — both are enumerable from other public API endpoints. The request is a single GET with two timestamp query parameters. It is trivially scriptable and repeatable. No CAPTCHA, rate limiting, or authentication barrier exists in the application code.

### Recommendation

1. **Add an `ORDER BY` clause matching `DISTINCT ON`** so PostgreSQL can use an index scan: `ORDER BY key, consensus_timestamp DESC`. This allows the planner to satisfy `DISTINCT ON (key)` incrementally via an index on `(owner_id, hook_id, key, consensus_timestamp DESC)` and apply `LIMIT` early.
2. **Enforce a maximum timestamp range width** in `hookStorageChangeRequest()` — reject or cap requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 24 hours in nanoseconds).
3. **Add application-level rate limiting** (e.g., via a Bucket4j or similar filter) on the historical storage endpoint, keyed by client IP.
4. **Require at least one narrow filter** (e.g., a mandatory `timestamp=eq:` or a maximum key range width) when the historical path is taken.

### Proof of Concept

**Preconditions:** Any `ownerId` with a `hookId` that has accumulated a large number of `hook_storage_change` rows (enumerable via the hooks list endpoint).

**Trigger (repeat in a tight loop):**
```
GET /api/v1/accounts/0.0.12345/hooks/1/storage?timestamp=gte:0.000000001&timestamp=lte:9999999999.999999999&limit=1
```

**What happens:**
1. `HooksController.getHookStorage()` receives the request. No key filters → `keyLowerBound = 0x00…00`, `keyUpperBound = 0xFF…FF`.
2. `HookServiceImpl.getHookStorageChange()` calls `findByKeyBetweenAndTimestampBetween()` with the full key and timestamp range.
3. PostgreSQL executes `SELECT DISTINCT ON (key) … FROM hook_storage_change WHERE owner_id=12345 AND hook_id=1 AND key BETWEEN 0x00…00 AND 0xFF…FF AND consensus_timestamp BETWEEN 1 AND 9999999999999999999 LIMIT 1`.
4. The planner sorts all matching rows by `key` (potentially millions), deduplicates, then returns 1 row.
5. Repeated concurrently (e.g., 20 threads), this saturates database CPU.
