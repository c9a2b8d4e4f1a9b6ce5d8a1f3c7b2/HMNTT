### Title
In-Memory Per-Instance Rate Limit Bypass on Opcode Replay Endpoint via Multi-Pod Concurrency

### Summary
The `opcodeRateLimitBucket` in `getContractOpcodes()` is a JVM-local, in-memory bucket4j `Bucket` with no distributed backing store. In a multi-replica Kubernetes deployment (HPA supports up to 12 replicas), each pod maintains its own independent bucket. An unprivileged attacker sending concurrent requests across pods can multiply the effective rate limit by the replica count, bypassing the intended global cap on a computationally expensive EVM re-execution endpoint.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 47–55 constructs the `opcodeRateLimitBucket` as a plain local bucket4j `Bucket` with no `ProxyManager`, no Redis/Hazelcast backend, and no distributed synchronization: [1](#0-0) 

`ThrottleManagerImpl.java` lines 52–55 enforces the check with a simple `tryConsume(1)` against this local bucket: [2](#0-1) 

`OpcodesController.java` line 61 calls `throttleManager.throttleOpcodeRequest()` before dispatching to `opcodeService.processOpcodeCall()`, which re-executes the full EVM transaction: [3](#0-2) 

**Root cause:** The `opcodeRateLimitBucket` bean is instantiated once per JVM process. In a horizontally scaled deployment, N replicas each hold an independent bucket initialized to `opcodeRequestsPerSecond` (default: **1 req/sec**). There is no shared state between replicas. [4](#0-3) 

**Exploit flow:**
1. The web3 Helm chart ships with HPA support (`minReplicas: 1`, `maxReplicas: 12`). Under load, or with a minimum replica count ≥ 2, multiple pods are live simultaneously. [5](#0-4) 
2. The Traefik ingress load-balances across all pods. The attacker sends N concurrent `GET /api/v1/contracts/results/{txHash}/opcodes` requests with `Accept-Encoding: gzip`.
3. The load balancer distributes one request to each pod. Each pod's local bucket has a full token (1 req/sec default), so each pod independently allows the request.
4. Effective cluster-wide rate = N × `opcodeRequestsPerSecond`. With 12 replicas and the default of 1 req/sec, the attacker achieves 12 EVM re-executions per second instead of 1.

**Why existing checks fail:** The check at `ThrottleManagerImpl.throttleOpcodeRequest()` is correct in isolation but the assumption that a single bucket guards the global rate is violated the moment more than one replica is running. There is no API gateway, load-balancer-level, or Redis-backed rate limit that compensates. [6](#0-5) 

### Impact Explanation
The opcode endpoint re-executes the full EVM transaction to collect stack/memory/storage traces — explicitly noted in the controller Javadoc as potentially taking "a significant amount of time." Bypassing the rate limit by a factor of up to 12× allows an attacker to saturate CPU and memory on all web3 pods simultaneously, causing service degradation or complete unavailability of the `/contracts/call` and `/contracts/results/.../opcodes` endpoints for legitimate users. This is a realistic denial-of-service against a resource-intensive, publicly reachable endpoint with no authentication requirement.

### Likelihood Explanation
Preconditions: none beyond network access. The attacker needs only to send concurrent HTTP GET requests with a `gzip` Accept-Encoding header and a valid transaction hash (obtainable from the public mirror node REST API). No credentials, no privileged keys, no internal access. The attack is trivially scriptable (e.g., `parallel curl` or any HTTP load tool). In production, HPA is enabled and minReplicas ≥ 2 is standard for HA, making the bypass immediately available.

### Recommendation
Replace the local `Bucket.builder().addLimit(limit).build()` for `opcodeRateLimitBucket` with a distributed bucket4j proxy backed by the existing Redis deployment (already present in the stack). Use `bucket4j-redis` or `bucket4j-hazelcast` with a fixed key (e.g., `"opcode-rate-limit"`) so all replicas share a single token pool. Alternatively, enforce the rate limit at the Traefik ingress layer using its built-in `RateLimit` middleware, which operates before the request reaches any pod. [1](#0-0) 

### Proof of Concept
```bash
# Assume 3 web3 replicas are running behind a load balancer at $HOST
# Obtain a valid transaction hash from the public mirror REST API
TX_HASH=$(curl -s "https://$MIRROR_REST/api/v1/contracts/results?limit=1" \
  | jq -r '.results[0].hash')

# Send 3 concurrent opcode requests — each hits a different pod
for i in 1 2 3; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://$HOST/api/v1/contracts/results/$TX_HASH/opcodes" &
done
wait
# Expected (with bypass): three "200 OK" responses within the same second
# Expected (without bypass / correct distributed limit): two "429 Too Many Requests"
```

With N replicas, N concurrent requests all return 200 within the same second, demonstrating that the effective rate limit is N × `opcodeRequestsPerSecond` rather than the intended global cap.
