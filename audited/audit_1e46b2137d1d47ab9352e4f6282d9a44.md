### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to `/accounts/:id/allowances/crypto`

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` accepts unlimited concurrent unauthenticated requests, each acquiring a PostgreSQL connection from the shared finite pool. The REST API's Traefik middleware configuration omits both `inFlightReq` and `rateLimit` controls (present in other services like Rosetta), and the application-level middleware stack contains no per-IP throttle. A flood of concurrent requests exhausts the connection pool, causing all endpoints — including transaction confirmation queries — to queue and time out.

### Finding Description

**Code path:**

`rest/controllers/cryptoAllowanceController.js` lines 76–80: [1](#0-0) 

Each request calls `CryptoAllowanceService.getAccountCryptoAllowances()`, which in `rest/service/cryptoAllowanceService.js` lines 13–16 calls `super.getRows(query, params)`: [2](#0-1) 

`getRows` in `rest/service/baseService.js` line 56 calls `this.pool().queryQuietly(query, params)`, acquiring a connection from the global `pg.Pool`: [3](#0-2) 

The pool is initialized in `rest/dbpool.js` with a hard `max: config.db.pool.maxConnections` ceiling and a `connectionTimeoutMillis` before requests queue-block: [4](#0-3) 

**Root cause — missing ingress-level controls:**

The REST API's Traefik middleware in `charts/hedera-mirror-rest/values.yaml` lines 134–139 contains only `circuitBreaker` and `retry`: [5](#0-4) 

Compare to the Rosetta chart (`charts/hedera-mirror-rosetta/values.yaml` lines 149–163) which adds both `inFlightReq` (5 concurrent per IP) and `rateLimit` (10 req/s per host): [6](#0-5) 

The REST API has neither. Additionally, the `retry: attempts: 10` in the REST middleware means each failed request is retried 10 times by Traefik, multiplying DB connection demand by up to 10×.

**Application-level middleware stack** (`rest/server.js` lines 68–97) registers: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — no rate limiter, no concurrency cap: [7](#0-6) 

The `authHandler` only enforces a per-user response *size* limit, not a request *rate* limit: [8](#0-7) 

**Why the circuitBreaker does not protect here:** It trips on `NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500…)`. Connection pool exhaustion causes requests to *queue and time out* (connection timeout errors surfaced as 500s), but by the time the ratio threshold is crossed, the pool is already saturated and legitimate requests are already failing.

### Impact Explanation

The PostgreSQL connection pool is shared across all REST API endpoints. When the pool is exhausted, every endpoint — including transaction confirmation queries used by nodes and clients to verify consensus — blocks waiting for `connectionTimeoutMillis` before returning an error. This constitutes a full REST API denial of service. The `statement_timeout` on the pool config limits individual query duration but does not prevent pool saturation from concurrent waiters. Severity: **Critical** (total REST API unavailability).

### Likelihood Explanation

No authentication, API key, or proof-of-work is required. Any external actor with a standard HTTP client can issue thousands of concurrent GET requests to a single valid account ID (e.g., `0.0.2`, a well-known system account). The attack is trivially scriptable (`ab`, `wrk`, `hey`, or a simple async loop), requires no special knowledge of the system, and is repeatable indefinitely. The absence of `inFlightReq` at the ingress layer means the attack traffic reaches the Node.js process and the DB pool directly.

### Recommendation

1. **Add `inFlightReq` and `rateLimit` to the REST API Traefik middleware** (matching the Rosetta pattern) in `charts/hedera-mirror-rest/values.yaml`:
   ```yaml
   middleware:
     - inFlightReq:
         amount: 20
         sourceCriterion:
           ipStrategy:
             depth: 1
     - rateLimit:
         average: 50
         burst: 100
         sourceCriterion:
           ipStrategy:
             depth: 1
     - circuitBreaker:
         expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
     - retry:
         attempts: 3   # also reduce from 10
         initialInterval: 100ms
   ```
2. **Reduce `retry.attempts`** from 10 to 3 to prevent 10× amplification of failing requests against the DB pool.
3. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) as a defense-in-depth layer independent of the ingress configuration.
4. **Set a `connectionTimeoutMillis`** low enough (e.g., 1000 ms) so queued requests fail fast rather than holding Node.js event loop slots.

### Proof of Concept

```bash
# Precondition: mirror node REST API accessible at $HOST, account 0.0.2 exists
# Step 1: flood with 2000 concurrent requests
wrk -t 20 -c 2000 -d 30s \
  "http://$HOST/api/v1/accounts/0.0.2/allowances/crypto"

# Step 2: in a separate terminal, observe legitimate requests timing out
curl -w "%{http_code} %{time_total}s\n" \
  "http://$HOST/api/v1/transactions?limit=1"
# Expected result: 500 errors or connection timeouts within seconds of flood start
# The pool (default maxConnections) is saturated; all endpoints become unresponsive
```
