### Title
Unauthenticated `/mempool` Endpoint Triggers Retry-Amplified HTTP 500 Storm That Saturates Shared Circuit Breaker, Denying All Rosetta API Traffic

### Summary
The `Mempool()` handler unconditionally returns `ErrNotImplemented`, which the Rosetta SDK serializes as HTTP 500. The Traefik middleware chain retries each such response up to 3 times, multiplying the 500-response count by 4×. Because the circuit breaker expression monitors a ratio across **all** Rosetta endpoints and the rate-limit is keyed per request hostname (not per client IP), a single unauthenticated attacker can consume the entire 10 req/s budget with `/mempool` calls, generate ≥40 backend HTTP 500s per second, and push the `ResponseCodeRatio(500,600,0,600)` above the 0.25 threshold, opening the circuit breaker and rejecting all legitimate Rosetta API traffic including smart contract construction calls.

### Finding Description

**Code path — service layer:**
`rosetta/app/services/mempool_service.go`, lines 22–27:
```go
func (m *mempoolAPIService) Mempool(
    _ context.Context,
    _ *types.NetworkRequest,
) (*types.MempoolResponse, *types.Error) {
    return nil, errors.ErrNotImplemented   // always HTTP 500
}
``` [1](#0-0) 

`ErrNotImplemented` is a `*types.Error` (Rosetta error code 111, `retriable: false`). The coinbase/rosetta-sdk-go server serializes every non-nil `*types.Error` return as **HTTP 500**. There is no branch that can produce a 2xx or 4xx from this handler. [2](#0-1) 

**Code path — ingress exposure:**
`/rosetta/mempool` is explicitly listed as an ingress path: [3](#0-2) 

**Code path — middleware chain (all four interacting components):** [4](#0-3) 

The four relevant middleware entries and their interaction:

| Middleware | Config | Effect on attack |
|---|---|---|
| `circuitBreaker` | `ResponseCodeRatio(500,600,0,600) > 0.25` | Trips when >25% of all backend responses are 5xx — **shared across every Rosetta endpoint** |
| `rateLimit` | `average: 10`, `sourceCriterion: requestHost: true` | 10 req/s keyed by **Host header** (global per hostname, not per client IP) |
| `inFlightReq` | `amount: 5`, per IP | Limits concurrency per IP but does not prevent sustained 10 req/s throughput |
| `retry` | `attempts: 3`, `initialInterval: 100ms` | Traefik retries on 5xx — each `/mempool` request generates **1 original + 3 retries = 4 backend 500s** |

**Root cause and failed assumption:**
The design assumes that a permanently-unimplemented endpoint returning HTTP 500 is harmless because it is "cheap." The failed assumption is that the retry middleware would not be triggered by a deterministic 500 from an unimplemented stub, and that the circuit breaker's ratio window would be dominated by healthy traffic. Neither holds: the retry middleware does not distinguish transient from permanent 500s, and the rate-limit bucket is global (per hostname), so one attacker can monopolize it.

**Exploit flow:**
1. Attacker sends POST requests to `https://<host>/rosetta/mempool` at 10 req/s (the full rate-limit budget for the hostname).
2. Traefik's retry middleware retries each 500 response 3 times → **40 HTTP 500 backend responses/second**.
3. Legitimate traffic to `/rosetta/block`, `/rosetta/construction`, etc. is simultaneously rate-limited to 0 req/s (attacker consumed the shared bucket) or, if the attacker uses a spoofed/different Host header to get a separate bucket, legitimate traffic still generates far fewer responses.
4. The circuit breaker evaluates `ResponseCodeRatio(500,600,0,600)`: with 40 500s/s and near-zero legitimate 200s/s, the ratio is ≥ 0.25 within one measurement window.
5. Circuit breaker opens → Traefik returns 503 to **all** Rosetta endpoints until the recovery window elapses.
6. Attacker repeats immediately after recovery, keeping the circuit permanently open.

### Impact Explanation
When the circuit breaker opens, every Rosetta endpoint — `/rosetta/construction/submit`, `/rosetta/block`, `/rosetta/account/balance` — returns 503. Smart contract construction and submission calls are rejected. Because the `/mempool` stub is permanently broken (not transiently), the circuit never recovers as long as the attacker continues sending requests. The impact is a complete, sustained denial of the Rosetta API layer with no funds directly at risk but with full disruption of any application relying on the Rosetta interface for transaction submission or state queries.

### Likelihood Explanation
**Preconditions:** Middleware must be enabled (`global.middleware: true`); the ingress must be reachable. No credentials, API keys, or privileged network position are required. The attacker needs only HTTP POST capability to the public ingress.

**Feasibility:** 10 req/s is trivially achievable with a single `curl` loop or `ab -n 1000 -c 5`. The request body can be minimal (a valid JSON `NetworkRequest` skeleton). The attack is stateless and requires no prior knowledge beyond the public path `/rosetta/mempool`.

**Repeatability:** The circuit breaker has a recovery window; the attacker simply resumes after it closes. The attack is fully repeatable and can be automated.

### Recommendation
1. **Remove `/rosetta/mempool` from the ingress path list** in `charts/hedera-mirror-rosetta/values.yaml` (line 131). There is no reason to expose a permanently-unimplemented endpoint externally.
2. **Return HTTP 501 (Not Implemented) instead of HTTP 500** from the Rosetta SDK for stub handlers, or override the SDK's error-to-HTTP mapping so that `ErrNotImplemented` maps to a non-5xx status code (e.g., 501), preventing it from counting toward the circuit breaker's `ResponseCodeRatio(500,600,0,600)` expression.
3. **Change the rate-limit `sourceCriterion` from `requestHost: true` to per-IP** (`ipStrategy: depth: 1`) so that a single attacker cannot consume the entire hostname budget.
4. **Scope the circuit breaker per route** rather than globally, or exclude the `/mempool` path from the circuit breaker's measurement window.
5. **Add a path-level block rule** in the ingress or a Traefik `IPAllowList`/`BasicAuth` middleware specifically for `/mempool` until it is implemented.

### Proof of Concept
```bash
# Minimal valid NetworkRequest body (adjust network/blockchain as deployed)
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'

# Step 1: Confirm /mempool always returns HTTP 500
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://<host>/rosetta/mempool \
  -H "Content-Type: application/json" \
  -d "$BODY"
# Expected: 500

# Step 2: Saturate the circuit breaker
# Send 10 req/s continuously (stays within rate limit, consumes full budget)
while true; do
  for i in $(seq 1 10); do
    curl -s -o /dev/null \
      -X POST https://<host>/rosetta/mempool \
      -H "Content-Type: application/json" \
      -d "$BODY" &
  done
  sleep 1
done &

# Step 3: In a separate terminal, observe legitimate endpoints returning 503
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://<host>/rosetta/network/status \
  -H "Content-Type: application/json" \
  -d "$BODY"
# Expected after circuit opens: 503
```

Each attacker request generates 4 backend HTTP 500 responses (1 + 3 retries). With 10 req/s, that is 40 backend 500s/second. The circuit breaker threshold of 25% is exceeded as soon as legitimate 200-response traffic drops below 120 req/s — a condition trivially met given the shared 10 req/s rate limit.
