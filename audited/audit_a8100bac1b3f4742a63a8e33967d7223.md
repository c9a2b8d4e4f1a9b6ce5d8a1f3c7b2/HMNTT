### Title
EVM Address Existence Oracle via `from=` Filter in `getContractResultsById()`

### Summary
An unprivileged external user can supply any syntactically valid EVM address as the `from=` query parameter to `GET /api/v1/contracts/:contractId/results`. The server unconditionally calls `EntityService.getEncodedId()` with `requireResult = true` (the default), which issues a live database lookup and throws `NotFoundError` when the address is absent. This produces a deterministic HTTP 404 for unknown addresses and HTTP 200 for known ones, creating a direct boolean oracle — and a secondary timing oracle — for EVM address existence in the entity table, with no authentication or rate-limiting gate.

### Finding Description

**Code path:**

`rest/routes/contractRoute.js` line 14 routes `GET /:contractId/results` to `ContractController.getContractResultsById`. [1](#0-0) 

`getContractResultsById` (line 871) calls `extractContractResultsByIdQuery(filters, contractId)` without any authentication check. [2](#0-1) 

Inside `extractContractResultsByIdQuery`, for `filterKeys.FROM`, the only guard is a format check (`EntityId.isValidEvmAddress`). If the value passes, `EntityService.getEncodedId(filter.value)` is called immediately — with `requireResult` left at its default value of `true`. [3](#0-2) 

`getEncodedId` parses the EVM address and, because `entityId.evmAddress !== null` for a non-parsable 20-byte address, delegates to `getEntityIdFromEvmAddress(entityId, requireResult=true)`. [4](#0-3) 

`getEntityIdFromEvmAddress` executes a live SQL query against the `entity` table. When zero rows are returned and `requireResult` is `true`, it throws `NotFoundError()` unconditionally. [5](#0-4) 

The global error handler maps `NotFoundError` to HTTP 404. [6](#0-5) 

**Root cause:** `extractContractResultsByIdQuery` calls `getEncodedId` with the default `requireResult = true` instead of `false`. The failed assumption is that a missing `from=` address should abort the entire request rather than silently produce an empty result set.

**Oracle mechanics:**
- `from=<address that exists>` → DB lookup succeeds → request continues → HTTP 200 (possibly empty `results` array)
- `from=<address that does not exist>` → DB lookup returns 0 rows → `NotFoundError` thrown → HTTP 404

The response code alone is a reliable, noiseless boolean oracle. A secondary timing oracle also exists: the found path executes a second DB query (`getContractResultsByIdAndFilters`) while the not-found path terminates after the first query, producing a measurable latency difference under load.

### Impact Explanation
Any unauthenticated client can enumerate which EVM addresses are registered in the Hedera entity table by iterating candidate addresses and observing 404 vs 200. This leaks the set of live accounts and contracts without requiring any on-chain interaction or privileged access. While individual EVM addresses are technically public on-chain, the mirror node's entity table may contain addresses not yet widely indexed, and bulk enumeration via a REST API is significantly faster and cheaper than on-chain scanning. The secondary timing channel survives even if the status-code difference were masked.

### Likelihood Explanation
Preconditions are minimal: the attacker needs only one valid contract ID (trivially obtained from any public block explorer or the `/api/v1/contracts` endpoint) and the ability to send HTTP GET requests. No credentials, tokens, or special network position are required. The attack is fully repeatable, scriptable, and produces a clean binary signal per request.

### Recommendation
Pass `requireResult = false` when calling `getEncodedId` for the `from=` filter, and treat a `null` return as "no sender filter" (skip adding the condition) or as "filter that matches nothing" (add an unsatisfiable condition). This eliminates both the status-code oracle and the timing oracle:

```js
// rest/controllers/contractController.js  ~line 443-444
if (EntityId.isValidEvmAddress(filter.value)) {
  filter.value = await EntityService.getEncodedId(filter.value, false); // requireResult = false
  if (filter.value === null) break; // address unknown → no results, skip filter
}
```

Additionally, consider applying rate-limiting middleware to all `/contracts/*/results` endpoints to bound the throughput of any enumeration attempt.

### Proof of Concept

**Step 1 – Obtain a valid contract ID** (any known contract on the network):
```
GET /api/v1/contracts?limit=1
→ note contractId, e.g. "0.0.1234"
```

**Step 2 – Probe a known-existing EVM address** (e.g. a contract whose address is known):
```
GET /api/v1/contracts/0.0.1234/results?from=0xaabbccdd...known...address
→ HTTP 200  {"results": [], "links": {"next": null}}
```

**Step 3 – Probe a non-existing EVM address**:
```
GET /api/v1/contracts/0.0.1234/results?from=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef
→ HTTP 404  {"_status": {"messages": [{"message": "Not found"}]}}
```

**Step 4 – Automate enumeration**:
```python
import requests, time
base = "https://<mirror-node>/api/v1/contracts/0.0.1234/results?from="
for addr in candidate_addresses:
    t0 = time.time()
    r = requests.get(base + addr)
    elapsed = time.time() - t0
    print(addr, r.status_code, elapsed)
    # 200 → address exists; 404 → does not exist
```

The status code alone gives a noiseless oracle; `elapsed` provides the secondary timing signal.
