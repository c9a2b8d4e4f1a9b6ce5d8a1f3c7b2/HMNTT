### Title
Inverted Range Bounds in `getHooks` Cause Unnecessary Database Query Execution

### Summary
The `hooksRequest` builder in `HooksController` can produce a `HooksRequest` where `lowerBound > upperBound` when a caller supplies conflicting range filters (e.g., `hookId=gt:100&hookId=lt:50`). No validation in the controller or service rejects this inversion. When `hookIds` is empty, `HookServiceImpl.getHooks` passes the inverted bounds directly to `findByOwnerIdAndHookIdBetween`, causing the database to execute a semantically impossible `BETWEEN` query on every such request.

### Finding Description

**Controller — bound construction** (`HooksController.java`, lines 135–145):

```java
long lowerBound = 0L;
long upperBound = MAX_VALUE;

for (final var hookIdFilter : hookIdFilters) {
    if (hookIdFilter.operator() == RangeOperator.EQ) {
        hookIds.add(hookIdFilter.value());
    } else if (hookIdFilter.hasLowerBound()) {
        lowerBound = Math.max(lowerBound, hookIdFilter.getInclusiveValue());
    } else if (hookIdFilter.hasUpperBound()) {
        upperBound = Math.min(upperBound, hookIdFilter.getInclusiveValue());
    }
}
```

A caller supplying `hookId=gt:100&hookId=lt:50` produces `lowerBound = 100`, `upperBound = 50`. No cross-check is performed after the loop.

**Service — unconditional DB dispatch** (`HookServiceImpl.java`, lines 36–40):

```java
final long lowerBound = request.getLowerBound();
final long upperBound = request.getUpperBound();

if (request.getHookIds().isEmpty()) {
    return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
```

When no EQ filters are present, the inverted bounds are forwarded directly to the repository, which issues `WHERE owner_id = ? AND hook_id BETWEEN 100 AND 50` to the database. The database must parse, plan, and execute this query on every request.

**Existing guards are insufficient:**
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` limits the number of `hookId` parameters but does not prevent inversion.
- `@Positive @Max(MAX_LIMIT)` caps the result page size but does not prevent the query from being issued.
- The `hookIds`-non-empty branch short-circuits at line 47–48 (`idsInRange.isEmpty()` → `List.of()`), but the empty-`hookIds` branch has no equivalent guard.

### Impact Explanation
Any unauthenticated caller can repeatedly issue requests with inverted bounds. Each request causes a full round-trip to the database (connection checkout, query parse, plan, execute) that returns zero rows and zero useful data. Under sustained load this constitutes unnecessary database resource consumption (CPU, I/O, connection pool slots) with no benefit to the network. Severity is low-to-medium, consistent with the declared scope of griefing with no economic damage.

### Likelihood Explanation
The exploit requires no credentials, no special knowledge beyond the public API shape, and is trivially scriptable. Any external actor who can reach the REST endpoint can trigger it. The `@Size` cap on repeated parameters limits the number of filters per request but does not prevent high request rates.

### Recommendation
Add an explicit guard in `HookServiceImpl.getHooks` (or in the controller after building the request) that short-circuits when `lowerBound > upperBound`:

```java
if (lowerBound > upperBound) {
    return List.of();
}
```

Alternatively, add a `@AssertTrue` or custom `@Valid` constraint on `HooksRequest` that enforces `lowerBound <= upperBound` before the service is invoked.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks?hookId=gt:100&hookId=lt:50
```

1. Controller computes `lowerBound = 100`, `upperBound = 50` (no EQ filters → `hookIds` is empty).
2. `HookServiceImpl.getHooks` enters the `if (request.getHookIds().isEmpty())` branch.
3. Repository executes: `SELECT ... FROM hook WHERE owner_id = 1234 AND hook_id BETWEEN 100 AND 50 LIMIT <limit>`.
4. Database returns 0 rows; caller receives an empty `hooks` array.
5. Repeat at high frequency to consume database resources. [1](#0-0) [2](#0-1) [3](#0-2)
