### Title
DB Connection Pool Exhaustion via Disabled Statement Timeout on Zero `StatementTimeout` Configuration

### Summary
When `Db.StatementTimeout` is zero (the Go zero-value default when no config is provided), `GetDbWithContext` in `rosetta/app/db/client.go` explicitly bypasses all application-level DB query timeouts, returning a `noop` cancel function. An unauthenticated external attacker can flood the publicly accessible Rosetta API with concurrent requests that trigger slow DB queries, exhausting the finite connection pool and starving the `/network/status` and `/block` endpoints of database access.

### Finding Description

**Config struct — zero is the Go default:**
In `rosetta/app/config/types.go` at line 45, `StatementTimeout int` has no default value enforced in code. If the `application.yml` is absent or the field is omitted, Go initializes it to `0`. [1](#0-0) 

**Timeout bypass branch:**
In `rosetta/app/db/client.go`, `GetDbWithContext` at line 23 checks `if d.statementTimeout <= 0` and, when true, returns the raw DB handle with a `noop` cancel function — no `context.WithTimeout` is applied. The only cancellation source becomes the HTTP request context. [2](#0-1) 

**All DB-touching endpoints use this path:**
`findBlockByIndex`, `findBlockByHash`, `initGenesisRecordFile`, and the `NetworkStatus` handler all call `GetDbWithContext(ctx)` and `defer cancel()`. With `statementTimeout=0`, `cancel` is `noop` and the DB query runs until the PostgreSQL server decides to finish or the HTTP context is cancelled. [3](#0-2) [4](#0-3) 

**Pool is the finite resource:**
`ConnectToDb` sets `MaxOpenConnections` from config (default 100). With no per-query timeout, each slow query holds a connection for its full duration. [5](#0-4) 

**Why HTTP timeouts are insufficient:**
The HTTP `WriteTimeout` (default 10 s) closes the TCP connection, but Go's `net/http` does not guarantee that the request context is cancelled synchronously at the moment `WriteTimeout` fires — it depends on the connection lifecycle. Even when the context is eventually cancelled, the PostgreSQL driver may not immediately abort the in-flight query, leaving the connection occupied in the pool until the backend query finishes. [6](#0-5) 

**Documentation default vs. code reality:**
The documentation states the default is 20 seconds, but this is only applied if the YAML config file is present and the field is explicitly set. The code has no minimum enforcement or fallback. [7](#0-6) 

### Impact Explanation
With all 100 pool connections held by attacker-controlled slow queries, every subsequent call to `/network/status` (which calls `RetrieveGenesis` + `RetrieveLatest` + `Entries`, each requiring a DB connection) and `/block` (which calls `findBlockByIndex`/`findBlockByHash` + `FindBetween`) blocks waiting for a free connection. This constitutes a complete denial-of-service of the Rosetta node's online-mode endpoints — the node appears partitioned from the network to any Rosetta client (e.g., exchange infrastructure, block explorers) polling these endpoints.

### Likelihood Explanation
- The Rosetta API is unauthenticated and publicly reachable on port 5700.
- The vulnerability is triggered whenever `statementTimeout` is 0, which is the Go zero-value default — any deployment that omits the field in `application.yml` is affected.
- The attacker needs only an HTTP client capable of sending ~100 concurrent POST requests to `/block` or `/network/status` with valid `network_identifier` payloads (publicly documented Rosetta spec).
- The `selectGenesis` query (a CTE with a join across `account_balance` and `record_file`) and `selectByHashWithIndex` (a correlated subquery) are non-trivial and can be slow on large tables, making them natural targets for holding connections. [8](#0-7) [9](#0-8) 

### Recommendation
1. **Enforce a minimum non-zero default in code**: In `config.go` (the config loader), after unmarshalling, set `StatementTimeout` to a safe default (e.g., 20) if it is `<= 0`.
2. **Remove the bypass branch**: Change `GetDbWithContext` to always apply a timeout — if `statementTimeout <= 0`, use a hardcoded safe fallback (e.g., 30 seconds) rather than disabling the timeout entirely.
3. **Add per-IP rate limiting** on the HTTP layer to limit concurrent DB-touching requests per client.
4. **Set `statement_timeout` at the PostgreSQL role level** as a defense-in-depth measure (the Helm chart already does this at line 126 of `secret-passwords.yaml`, but this only applies to Helm-managed deployments). [10](#0-9) 

### Proof of Concept

**Preconditions:**
- Rosetta node running in online mode with `statementTimeout: 0` (or missing from config).
- PostgreSQL `record_file` and `account_balance` tables contain substantial data (normal production state).

**Steps:**
```bash
# 1. Craft a valid Rosetta block request body
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}'

# 2. Flood with 100+ concurrent requests to exhaust the connection pool
for i in $(seq 1 110); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "$BODY" &
done
wait

# 3. Verify DoS: legitimate /network/status now hangs or returns 500
curl -X POST http://<rosetta-host>:5700/network/status \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'
```

**Expected result:** Step 3 hangs indefinitely or returns a database error, confirming pool exhaustion. The node is effectively partitioned from any Rosetta client relying on `/network/status` for chain-tip awareness.

	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop

func (n *networkAPIService) NetworkStatus(

		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}
