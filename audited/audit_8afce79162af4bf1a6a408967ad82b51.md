### Title
Unauthenticated Concurrent Requests with `account.publickey` + `timestamp` Exhaust PostgreSQL Connection Pool via Sequential DB Queries in `getBalances()`

### Summary
In `rest/balances.js`, the `getBalances()` handler, when called with both a `timestamp` and `account.publickey` parameter, executes three sequential PostgreSQL queries per request: one against `account_balance` (via `getTsQuery`), one against `entity` (via `getAccountIdsByPublicKey`/`entityPublicKeyQuery`), and a final main balance query. No rate-limiting middleware exists at the application layer. An unauthenticated attacker can flood this endpoint with valid-format parameters, exhausting the default 10-connection pool and causing legitimate requests to time out.

### Finding Description

**Exact code path:**

`rest/balances.js` lines 113–132 — the `tsQuery` branch of `getBalances()`:

**`entityPublicKeyQuery` (DB query 2):**
```js
const entityPublicKeyQuery = `select id from entity where type in ('ACCOUNT', 'CONTRACT') and public_key = $1 limit $2`;
``` [3](#0-2) 

**Root cause:** Three sequential `await pool.queryQuietly(...)` calls are made per request, each holding a connection from the pool for its duration. The pool defaults to `maxConnections: 10` and `statementTimeout: 20000` ms (20 seconds). [4](#0-3) [5](#0-4) 

**No rate limiting:** `server.js` registers no rate-limiting middleware. The grep across `rest/**/*.js` for `rateLimit|throttle|express-rate` returns zero application-layer hits. [6](#0-5) 

**Validation is insufficient:** `account.publickey` is validated for format (64-char hex or DER-encoded key) via `isValidPublicKeyQuery`, and `timestamp` is validated for format. Both are trivially satisfiable with any well-formed value — the key does not need to match a real account. The query will simply return empty results, but the DB round-trip still occurs and holds a connection. [7](#0-6) 

### Impact Explanation

With a default pool of 10 connections and a 20-second statement timeout, an attacker sending ≥10 concurrent requests to `/api/v1/balances?account.publickey=<valid_hex>&timestamp=<valid_ts>` can hold all available connections simultaneously. New legitimate requests queue for up to `connectionTimeout` (default 20 seconds) before failing. Because each attacker request triggers 3 sequential DB queries (not 1), the effective connection hold time is multiplied. This constitutes a non-network-based DoS: the database layer is saturated without requiring high bandwidth, only moderate concurrency.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only knowledge of the public API schema (documented in `openapi.yml`). Valid-format public keys are 64 hex characters — trivially generated. The attacker needs only ~10–20 concurrent HTTP connections sustained over time. This is repeatable indefinitely and automatable with any HTTP load tool (curl, wrk, k6 — the repo itself ships k6 test scripts targeting this exact endpoint pattern). [8](#0-7) 

### Recommendation

1. **Add application-layer rate limiting** (e.g., `express-rate-limit`) keyed by IP on the `/api/v1/balances` route, specifically tightened when `account.publickey` is present.
2. **Increase `db.pool.maxConnections`** beyond the default of 10 for production deployments, or deploy a connection pooler (PgBouncer) in front of PostgreSQL.
3. **Reduce `statementTimeout`** for the `entityPublicKeyQuery` path specifically, or add a shorter per-query timeout.
4. **Cache the `entityPublicKeyQuery` result** (public key → account IDs mapping) using the existing Redis/response cache infrastructure, since public keys change infrequently.
5. **Merge the preliminary `entityPublicKeyQuery` into the main balance query** as a subquery or JOIN, eliminating the extra sequential DB round-trip and connection hold.

### Proof of Concept

```bash
# Generate a valid-format 64-char hex public key (does not need to match a real account)
PUBKEY="3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be"
TIMESTAMP="1234567890"
BASE="https://<mirror-node-host>/api/v1"

# Send 20 concurrent requests (exceeds default pool of 10)
for i in $(seq 1 20); do
  curl -s "${BASE}/balances?account.publickey=${PUBKEY}&timestamp=${TIMESTAMP}" &
done
wait
# Legitimate requests during this window will receive connection timeout errors
# or significantly degraded response times as the pool is exhausted
```
