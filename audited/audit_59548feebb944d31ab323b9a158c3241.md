### Title
Uncached `findByIdAndTimestamp` in `TokenAirdropRepository` Enables DB Exhaustion via Historical `eth_call` Requests

### Summary
`TokenAirdropRepository.findByIdAndTimestamp` has no `@Cacheable` annotation, meaning every historical `eth_call` request that resolves a pending airdrop state unconditionally executes an expensive UNION ALL query across `token_airdrop` and `token_airdrop_history`. An unprivileged external user can send `eth_call` requests with any historical `blockNumber` to force repeated DB hits at the global rate limit (500 req/sec by default), with no per-IP or per-user throttling to prevent a single attacker from consuming the entire budget.

### Finding Description

**Code path:**

`ContractController.call()` → `ContractExecutionService.processCall()` → `AirdropsReadableKVState.readFromDataSource()`:

```java
// AirdropsReadableKVState.java lines 38-42
final var timestamp = ContractCallContext.get().getTimestamp();
return timestamp
    .map(t -> tokenAirdropRepository.findByIdAndTimestamp(senderId, receiverId, tokenId, serialNumber, t))
    .orElseGet(() -> tokenAirdropRepository.findById(senderId, receiverId, tokenId, serialNumber))
```

When a `blockNumber` is present in the `eth_call` request, `ContractCallContext.getTimestamp()` returns a non-empty `Optional`, routing execution to `findByIdAndTimestamp`.

**`findById` (latest block) — has caching:**
```java
// TokenAirdropRepository.java line 17
@Cacheable(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN, unless = "#result == null")
Optional<TokenAirdrop> findById(long senderId, long receiverId, long tokenId, long serialNumber);
```

**`findByIdAndTimestamp` (historical block) — NO caching:**
```java
// TokenAirdropRepository.java lines 41-72
@Query(value = """
    select * from (
        (select * from token_airdrop where ... and lower(timestamp_range) <= :blockTimestamp)
        union all
        (select * from token_airdrop_history where ... and lower(timestamp_range) <= :blockTimestamp
         order by lower(timestamp_range) desc limit 1)
    order by timestamp_range desc limit 1
    )""", nativeQuery = true)
Optional<TokenAirdrop> findByIdAndTimestamp(long senderId, long receiverId, long tokenId, long serialNumber, long blockTimestamp);
```

No `@Cacheable` annotation exists on `findByIdAndTimestamp`. Every invocation hits the database directly.

**Root cause:** The developer applied `@Cacheable` only to `findById` but omitted it from `findByIdAndTimestamp`. The cache configuration for the `token` manager uses `expireAfterWrite=1s,maximumSize=10000` — even if caching were added, the 1-second TTL means the same key queried at >1 req/sec would still bypass the cache.

**Rate limiting is global, not per-IP:**
```java
// ThrottleProperties.java line 35
private long requestsPerSecond = 500;
```
```java
// ThrottleManagerImpl.java lines 38-39
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```
The `rateLimitBucket` is a single shared bucket. A single attacker can consume all 500 req/sec, leaving no capacity for legitimate users and forcing 500 uncached DB queries per second.

### Impact Explanation
Each `findByIdAndTimestamp` call executes a UNION ALL across `token_airdrop` and `token_airdrop_history` with a GiST index scan on `timestamp_range`. At 500 req/sec (the default global limit), this generates 500 such queries per second with zero caching. On a production node with a non-trivial airdrop history table, this directly and measurably increases DB I/O well beyond 30% compared to baseline. The attacker also starves legitimate users of their request quota.

### Likelihood Explanation
No authentication is required to call `/api/v1/contracts/call`. Any external user can set `blockNumber` to any historical value in the JSON-RPC request body. The attacker needs only to craft calls that touch a contract invoking the airdrop system precompile (e.g., `cancelAirdrop`, `claimAirdrop`). The attack is trivially repeatable with a simple HTTP client loop and requires no on-chain funds or privileged access.

### Recommendation
1. Add `@Cacheable` to `findByIdAndTimestamp` with a composite cache key including `blockTimestamp`. Since historical queries are immutable (a past block's state never changes), results can be cached indefinitely or with a long TTL:
   ```java
   @Cacheable(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN,
              key = "#senderId + '_' + #receiverId + '_' + #tokenId + '_' + #serialNumber + '_' + #blockTimestamp",
              unless = "#result == null")
   ```
2. Implement per-IP rate limiting in addition to the global bucket, so a single client cannot exhaust the shared quota.
3. Consider increasing the `expireAfterWrite` TTL for the token cache beyond 1 second, or using a separate longer-lived cache for historical (immutable) queries.

### Proof of Concept
```bash
# Craft an eth_call targeting a contract that invokes the airdrop precompile
# with a historical blockNumber to force findByIdAndTimestamp on every request

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to": "0x0000000000000000000000000000000000000167",
      "data": "<cancelAirdrop_selector_with_valid_params>",
      "block": "0x1A2B3C",
      "gas": 300000
    }' &
done
wait
```

Each of the 500 concurrent requests with `"block": "0x1A2B3C"` (any historical block number) routes through `AirdropsReadableKVState.readFromDataSource()` → `findByIdAndTimestamp(...)`, executing an uncached UNION ALL DB query. Monitoring `pg_stat_activity` or DB I/O metrics will show a spike proportional to the request rate, exceeding the 30% threshold at moderate sustained load.
