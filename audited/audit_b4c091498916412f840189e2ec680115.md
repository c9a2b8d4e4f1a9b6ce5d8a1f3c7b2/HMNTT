### Title
Shared Per-Filter Token Bucket Exhaustion Enables Targeted DoS Against Legitimate Matching Requests

### Summary
The `THROTTLE` action in `ThrottleManagerImpl.action()` uses a single shared `Bucket` per `RequestProperties` filter with no per-IP or per-user isolation. An unprivileged attacker can craft requests that match the filter, rapidly exhaust the shared bucket (capacity = `rate`, max 100 tokens), then switch to non-matching requests that bypass the filter entirely, while all subsequent legitimate requests matching the filter receive `ThrottleException` for the remainder of each refill window.

### Finding Description
**Code path:**

- `ThrottleManagerImpl.throttle()` — [1](#0-0)  iterates all configured `RequestProperties` filters and calls `action()` for each match.
- `ThrottleManagerImpl.action()` — [2](#0-1)  calls `filter.getBucket().tryConsume(1)` on the shared per-filter bucket; if empty, throws `ThrottleException`.
- `RequestProperties.createBucket()` — [3](#0-2)  creates a bucket with `capacity = rate` (0–100) and `refillGreedy(rate, 1 second)`. Maximum capacity is 100 tokens.
- `RequestProperties.test()` — [4](#0-3)  explicitly skips the random-sampling gate for `THROTTLE` (`action != ActionType.THROTTLE && ...`), meaning **every** matching request unconditionally consumes a token.

**Root cause:** The bucket is a single JVM-level shared object. There is no per-source-IP, per-session, or per-user sub-bucket. The global rate limit bucket (`rateLimitBucket`, default 500 req/s) [5](#0-4)  is the only upstream guard, and it is far larger than the per-filter bucket capacity (≤100).

**Exploit flow:**
1. Attacker observes or guesses the filter (e.g., filter on `TO` = a known contract address, or `DATA` containing a known function selector — both are public on-chain).
2. Attacker sends up to 100 matching requests in rapid succession (well within the 500 req/s global limit). The per-filter bucket is exhausted in ≤0.2 s.
3. Attacker immediately switches to requests that do **not** match the filter. These bypass the per-filter bucket check entirely and only face the global 500 req/s limit.
4. For the remainder of the 1-second refill window, every legitimate request matching the filter hits an empty bucket and receives `ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED)`.
5. The attack is repeatable every second indefinitely.

**Why existing checks are insufficient:**
- The global `rateLimitBucket` (500 req/s) [6](#0-5)  does not prevent exhaustion of the per-filter bucket because 100 ≪ 500.
- The `limit` counter in `RequestProperties` [7](#0-6)  defaults to `Long.MAX_VALUE` and only stops the filter from being evaluated at all — it does not protect the bucket from exhaustion before that threshold.
- There is no IP-based rate limiting, authentication requirement, or per-caller sub-bucket anywhere in the throttle stack.

### Impact Explanation
Any `THROTTLE`-configured filter intended to allow a controlled rate of specific requests (e.g., calls to a high-value contract) can be rendered completely non-functional for legitimate callers. The attacker causes a sustained, repeatable denial of service for all users whose requests match the filter, while the attacker's own traffic (non-matching) continues unimpeded. This directly maps to availability impact on the web3 API endpoint.

### Likelihood Explanation
The attack requires no credentials or privileges — only the ability to send HTTP requests to the public web3 endpoint. Filter fields like `TO` (contract address) and `DATA` (function selector) are trivially observable from public blockchain data. The attack is fully automatable, repeatable every second, and requires only ~100 requests per cycle to sustain the DoS. The only practical barrier is knowing the filter configuration, which may be inferred from observed throttling behavior.

### Recommendation
1. **Per-caller sub-buckets:** Introduce per-source-IP (or per-authenticated-caller) buckets within the `THROTTLE` path so that one caller cannot exhaust the shared quota.
2. **Raise global guard before per-filter check:** Ensure the global rate limit is set low enough relative to the per-filter bucket that a single caller cannot exhaust the filter bucket within one refill window.
3. **Separate attacker budget from victim budget:** Consider a two-tier design where each caller gets an individual token allowance drawn from the shared pool, preventing one caller from consuming the entire pool.
4. **Obscure filter configuration:** Avoid exposing throttle behavior in error responses that could allow an attacker to probe and confirm filter matches.

### Proof of Concept
**Precondition:** Server has a `RequestProperties` filter configured with `action=THROTTLE`, `rate=100`, and `filters=[{field=TO, type=EQUALS, expression=0xTargetContract}]`.

**Steps:**
```
# Step 1: Exhaust the per-filter bucket (100 requests, all matching)
for i in $(seq 1 100); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0xTargetContract","gas":21000}' &
done
wait

# Step 2: Switch to non-matching requests (attacker continues unimpeded)
# Attacker sends requests to a different contract — no filter match, no per-filter throttle
curl -X POST http://mirror-node/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xOtherContract","gas":21000}'
# → 200 OK

# Step 3: Legitimate user tries to call 0xTargetContract
curl -X POST http://mirror-node/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xTargetContract","gas":21000}'
# → 429 "Requests per second rate limit exceeded"
```
The DoS repeats every second as the bucket refills and the attacker re-exhausts it.
