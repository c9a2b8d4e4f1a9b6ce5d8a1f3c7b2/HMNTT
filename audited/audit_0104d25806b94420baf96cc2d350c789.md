### Title
Retry Exhaustion via Persistent HTTP 429 Causes Pinger DoS When `network=other`

### Summary
`buildNetworkFromMirrorNodes()` treats HTTP 429 as a retryable error and will exhaust all `cfg.mirrorNodeClientMaxRetries+1` attempts if the mirror REST endpoint persistently returns 429. When retries are exhausted, the function returns `nil, lastErr`, which propagates up through `newClient()` to `main()`, where `log.Fatalf` terminates the process. An attacker with the ability to intercept or spoof the (potentially plain-HTTP) mirror REST endpoint can permanently prevent the pinger from initializing and submitting any transactions.

### Finding Description
**Code path:**

In `pinger/mirror_node_client.go` lines 90–92, HTTP 429 is explicitly classified as retryable: [1](#0-0) 

The retry loop in `buildNetworkFromMirrorNodes()` (lines 52–69) will re-issue the request up to `max(cfg.mirrorNodeClientMaxRetries+1, 1)` times (default: 11 attempts). If every attempt returns 429, the loop exits and returns `nil, lastErr`: [2](#0-1) 

In `pinger/sdk_client.go` lines 18–21, when `network=other`, the error is returned directly to the caller with no fallback: [3](#0-2) 

In `pinger/main.go` line 43, `newClient()` failure calls `log.Fatalf`, terminating the process: [4](#0-3) 

**Root cause:** The retry logic assumes 429 is a transient condition. There is no cap on how many consecutive 429s are tolerated before giving up permanently, no fallback network source, and no TLS enforcement on the `mirrorRest` URL, meaning a network-level attacker can sustain the attack indefinitely.

**Failed assumption:** The code assumes the mirror REST endpoint is trusted and that 429 responses are temporary. Neither is enforced.

### Impact Explanation
When `network=other` is configured (required for private/custom deployments), a successful attack causes `log.Fatalf` to terminate the pinger process entirely. The pinger stops submitting tinybar transfers, so no transaction records are produced on the mirror node. Monitoring, liveness probes (`/tmp/alive`), and readiness probes (`/tmp/ready`) all cease. The impact is complete denial of service of the pinger component for the duration of the attack. [5](#0-4) 

### Likelihood Explanation
The `mirrorRest` URL accepts plain HTTP (e.g., `http://mirror-rest:5551` per the config help text). No TLS or certificate pinning is enforced: [6](#0-5) 

An attacker on the same network segment (or with DNS control over the mirror hostname) can intercept the unencrypted HTTP request and respond with 429 for every attempt. The default retry count is 10 (11 total attempts), each with exponential backoff starting at 500ms — all of which the attacker simply answers with 429. No credentials or privileged access to the pinger or Hiero network are required; only network-level access to the path between the pinger and the mirror REST endpoint. [7](#0-6) 

### Recommendation
1. **Enforce HTTPS** for the `mirrorRest` URL and reject plain-HTTP configurations.
2. **Do not treat retry exhaustion as fatal at startup.** Instead of `log.Fatalf`, retry `newClient()` in a loop with a longer outer backoff, so a transient or attacker-induced outage does not permanently kill the process.
3. **Distinguish 429 from permanent failure.** After N consecutive 429s, log a warning and continue operating with a cached or hardcoded fallback network map rather than aborting.
4. **Add a Retry-After header cap:** respect the `Retry-After` header but enforce a maximum wait, and count total wall-clock time rather than just attempt count.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`.
2. Configure the attacker-controlled host (or intercept DNS/HTTP for the legitimate mirror host) to return `HTTP/1.1 429 Too Many Requests` for every `GET /api/v1/network/nodes` request.
3. Start the pinger. Observe that `buildNetworkFromMirrorNodes()` issues 11 requests (default), all receive 429, and after the final attempt the function returns `nil, error`.
4. `newClient()` propagates the error; `main()` calls `log.Fatalf("client error: ...")` and the process exits with status 1.
5. No transactions are ever submitted; no records appear on the mirror node; `/tmp/ready` is never written.
