### Title
Unauthenticated Historical Block Griefing via Unvalidated `block` Parameter Bypassing Gas Throttle

### Summary
`constructServiceParameters()` in `ContractController.java` accepts any non-negative block number from an unauthenticated caller and passes it directly to the execution pipeline with no minimum-age or recency check. Combined with a gas throttle that returns zero tokens for requests with `gas ≤ 10,000`, an attacker can sustain up to 500 requests/second of zero-gas historical queries that force expensive UNION queries against history tables, degrading performance for all users querying recent state.

### Finding Description

**Exact code path:**

`ContractController.constructServiceParameters()` reads the block field with no lower-bound validation: [1](#0-0) 

The only gas validation is an upper-bound check: [2](#0-1) 

`BlockType.of()` rejects only negative numbers; block `1` (or any ancient block) is fully accepted: [3](#0-2) 

**Gas throttle bypass:** `ThrottleProperties.scaleGas()` returns `0` for any `gas ≤ 10,000`: [4](#0-3) 

`ThrottleManagerImpl.throttle()` calls `gasLimitBucket.tryConsume(0)`, which always succeeds in bucket4j, so the gas bucket is never depleted: [5](#0-4) 

Only the rate-limit bucket (default 500 req/sec) remains as a guard: [6](#0-5) 

**Historical state cost:** For any numeric block, `RecordFileServiceImpl` issues a DB lookup by index, and every subsequent state read (accounts, tokens, NFTs, files) executes expensive UNION queries against `*_history` tables keyed by timestamp: [7](#0-6) [8](#0-7) 

These historical queries are structurally more expensive than latest-state reads and will not hit any warm cache populated by recent-block traffic.

### Impact Explanation

An attacker sustaining 500 req/sec (the default rate limit) of `{"block":"1","gas":0}` requests forces the database to execute repeated UNION queries against all history tables for a block whose state is never cached alongside recent-block results. This saturates DB I/O and connection pool capacity, increasing latency and error rates for legitimate users querying recent state. The impact is a sustained, operator-invisible performance degradation (griefing) with no economic cost to the attacker.

### Likelihood Explanation

No authentication, API key, or IP restriction is required. The exploit requires only a standard HTTP POST to `/api/v1/contracts/call` with a JSON body. It is trivially scriptable, repeatable, and requires no on-chain funds or privileged access. The default rate limit of 500 req/sec is high enough to cause meaningful DB pressure.

### Recommendation

1. **Validate block recency in `constructServiceParameters()`**: Reject or separately throttle requests where `block.number()` is below a configurable minimum (e.g., more than N blocks behind the current tip).
2. **Fix the gas-throttle zero-token bypass**: Treat `gas ≤ 10,000` as consuming at least 1 token, or enforce a minimum gas floor on incoming requests.
3. **Apply a separate, lower rate limit for historical block requests**: Add a `RequestFilter` rule (the infrastructure already exists in `ThrottleProperties.request`) that throttles requests whose `block` field is not `latest`/`safe`/`finalized`.
4. **Consider a configurable `minHistoricalBlock` property** to reject queries for blocks older than the supported historical window.

### Proof of Concept

```bash
# Send 500 req/sec of zero-gas historical queries targeting block 1
# No authentication required; any HTTP client suffices

for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "block": "1",
      "gas": 0,
      "from": "0x0000000000000000000000000000000000000001",
      "to":   "0x0000000000000000000000000000000000000002",
      "data": "0x"
    }' &
done
wait
```

**Expected result:** Each request passes `validateContractMaxGasLimit` (0 ≤ maxGasLimit), consumes 0 gas tokens (bypassing the gas bucket), consumes 1 rate-limit token, and triggers a full historical state reconstruction from block 1 — forcing UNION queries against all `*_history` tables. Sustained at the 500 req/sec rate limit, this measurably degrades DB response times for concurrent latest-block queries.
