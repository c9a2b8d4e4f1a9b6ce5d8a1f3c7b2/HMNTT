### Title
Unauthenticated Alias-Based DB Exhaustion via Uncached, Unthrottled `findByAlias` Queries in Token Airdrops Endpoint

### Summary
The `GET /api/v1/accounts/{id}/airdrops/pending` (and `/outstanding`) endpoint in `TokenAirdropsController` accepts base32-encoded aliases as the `{id}` path variable. When an alias is supplied, `EntityServiceImpl.lookup()` unconditionally executes a raw SQL query against the `entity` table on every request, with no caching and no rate limiting in the `rest-java` module. An unauthenticated attacker flooding the endpoint with distinct, syntactically valid but non-existent aliases can sustain a continuous stream of DB queries, exhausting database connection pool and I/O capacity shared with the importer, degrading mirror node processing throughput.

### Finding Description

**Exact code path:**

1. `GET /api/v1/accounts/{id}/airdrops/pending` is handled by `getPendingAirdrops()`: [1](#0-0) 

2. `processRequest()` calls `service.getAirdrops(request)`: [2](#0-1) 

3. `TokenAirdropServiceImpl.getAirdrops()` calls `entityService.lookup(request.getAccountId())`: [3](#0-2) 

4. `EntityServiceImpl.lookup()` dispatches on parameter type — for an alias input it calls `entityRepository.findByAlias()` with no caching: [4](#0-3) 

5. `EntityRepository.findByAlias()` executes a native SQL query on every call: [5](#0-4) 

**Root cause — two failed assumptions:**

- **No caching**: The `rest-java` `EntityRepository` has no `@Cacheable` annotation on `findByAlias`. Contrast with the `web3` module's `EntityRepository`, which caches alias lookups: [6](#0-5) 

- **No rate limiting**: The throttle/bucket4j infrastructure exists only in the `web3` module: [7](#0-6) 
  The `rest-java` config directory contains only `LoggingFilter`, `MetricsFilter`, `WebMvcConfiguration`, etc. — no rate-limit filter applies to the airdrops endpoint.


**Alias format is permissive**: The regex `^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$` accepts any 40–70 character base32 string, giving an attacker an astronomically large space of syntactically valid, non-existent aliases: [8](#0-7) 

**Exploit flow:**
1. Attacker generates a stream of distinct 40-character base32 strings (e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`, `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB`, …).
2. Sends `GET /api/v1/accounts/<alias>/airdrops/pending` at high concurrency.
3. Each request passes alias validation, reaches `EntityServiceImpl.lookup()`, and fires `SELECT id FROM entity WHERE alias = ? AND deleted <> true` against the shared PostgreSQL instance.
4. Because aliases are non-existent, no result is cached (even if a cache were added, `unless = "#result == null"` semantics would prevent caching misses).
5. The DB connection pool is saturated; the importer's write path (which shares the same DB) slows or stalls, degrading mirror node ingestion throughput.

### Impact Explanation
The mirror node's `entity` table is shared between the REST read path and the importer write path. Sustained high-volume alias queries consume DB connections, CPU, and I/O. Because the importer depends on the same PostgreSQL instance to persist consensus data, DB saturation directly degrades mirror node processing capacity. With no server-side throttle, a single attacker with modest bandwidth can sustain thousands of alias queries per second, each requiring a DB round-trip, meeting the ≥30% processing degradation threshold for a sufficiently loaded deployment.

### Likelihood Explanation
No authentication, API key, or credential is required. The alias format is trivially generated programmatically. The attack is stateless and parallelizable across many source IPs. It is repeatable indefinitely and requires no special knowledge of the network state. Any public mirror node deployment is exposed.

### Recommendation
1. **Add caching for alias lookups in `rest-java`**: Annotate `EntityRepository.findByAlias()` with `@Cacheable` (with a short TTL), mirroring the `web3` module's pattern. Critically, also cache negative (miss) results to prevent cache-bypass via non-existent aliases.
2. **Add rate limiting to `rest-java`**: Introduce a per-IP or global request-rate filter (bucket4j or similar) for the `/api/v1/accounts/*/airdrops/*` endpoints, analogous to `ThrottleConfiguration` in `web3`.
3. **Validate alias existence early and cheaply**: Consider a lightweight bloom filter or in-memory set of known alias prefixes to short-circuit DB queries for obviously invalid aliases before they reach the repository layer.

### Proof of Concept

```bash
# Generate and fire 10,000 distinct alias requests concurrently
python3 -c "
import base64, os, subprocess, threading

def fire(i):
    alias = base64.b32encode(i.to_bytes(25, 'big')).decode().rstrip('=')
    # Pad to 40 chars if needed
    alias = alias.ljust(40, 'A')[:40]
    subprocess.run([
        'curl', '-s', '-o', '/dev/null',
        f'http://<mirror-node-host>/api/v1/accounts/{alias}/airdrops/pending'
    ])

threads = [threading.Thread(target=fire, args=(i,)) for i in range(10000)]
for t in threads: t.start()
for t in threads: t.join()
"
```

Each request passes alias regex validation, reaches `EntityServiceImpl.lookup()` → `entityRepository.findByAlias()`, and issues a distinct `SELECT` against the `entity` table. Monitor DB `pg_stat_activity` and mirror node importer lag to observe degradation.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();

                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
