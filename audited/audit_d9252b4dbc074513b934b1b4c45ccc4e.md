### Title
Unauthenticated Unbounded Request Flood Exhausts DB Connection Pool via `getByAliasAndType()`

### Summary
The GraphQL `account(input: {alias: ...})` endpoint is publicly accessible with no authentication, no rate limiting, and no per-IP throttling. Every request unconditionally invokes `decodeBase32()` and issues a live database query against the `entity` table. A single unauthenticated attacker flooding this endpoint can exhaust the application's database connection pool, causing a non-network DoS for all users of the mirror node GraphQL service.

### Finding Description
**Exact code path:**

`AccountController.account()` at line 48 routes alias-based requests directly to `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)`. [1](#0-0) 

`EntityServiceImpl.getByAliasAndType()` at line 30 unconditionally calls `decodeBase32(alias)` then `entityRepository.findByAlias(...)` — no caching, no guard, no short-circuit. [2](#0-1) 

`decodeBase32()` delegates to Apache Commons Codec `BASE32.decode()` — a synchronous, CPU-bound decode on every call. [3](#0-2) 

`EntityRepository.findByAlias()` issues a native SQL query `select * from entity where alias = ?1 and deleted is not true` — a live DB round-trip per request, consuming one connection from the pool for its duration. [4](#0-3) 

**Root cause:** No rate limiting, no authentication, and no result caching exist anywhere in the graphql module's Java code or configuration. [5](#0-4) 

**Failed assumption:** The service assumes an upstream infrastructure layer (reverse proxy, API gateway) enforces rate limits. No such enforcement is present in the codebase itself.

### Impact Explanation
Under a sustained flood of alias-lookup requests, HikariCP's connection pool (default: 10 connections) becomes fully occupied. New requests queue and eventually time out (the configured `statementTimeout` is 10 000 ms per the README). This causes all GraphQL queries — not just alias lookups — to fail with connection-pool-exhaustion errors, taking down the entire GraphQL service for legitimate users. The `decodeBase32()` CPU cost per request is individually trivial but aggregates under thousands of concurrent requests, adding thread-pool pressure on top of the DB pool exhaustion. [6](#0-5) 

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable at `/graphql/alpha` via a simple HTTP POST with a JSON body. No API key, session token, or credential is required. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can issue thousands of requests per second from one machine. The attack is trivially repeatable and requires zero privilege. [7](#0-6) 

### Recommendation
1. **Rate limiting at the application layer**: Add a per-IP rate limiter (e.g., Bucket4j with Spring Boot integration) as a `HandlerInterceptor` or `WebFilter` on the `/graphql/alpha` endpoint.
2. **Result caching**: Apply a short-lived cache (Caffeine, already used for query parsing) on `getByAliasAndType()` keyed on the alias string to collapse repeated identical lookups.
3. **Connection pool sizing + request queuing limits**: Configure HikariCP `maximumPoolSize` and `connectionTimeout` defensively, and add a request queue depth limit to shed load early.
4. **GraphQL query cost limiting**: The existing `GraphQlConfiguration` instrumentation covers depth/complexity but not request *volume* — add a global request-rate instrument. [8](#0-7) 

### Proof of Concept
```bash
# Flood the alias lookup endpoint with 500 concurrent connections, no auth needed
wrk -t8 -c500 -d30s -s post.lua http://<mirror-node-host>:8083/graphql/alpha
```

Where `post.lua` contains:
```lua
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"query":"{account(input:{alias:\"AAAAAAAAAAAAAAAA\"}){balance}}"}'
```

**Expected result within seconds:** DB connection pool exhausted; all subsequent GraphQL requests (including unrelated ones) return 500 errors or hang until `statementTimeout` (10 s) expires. Application logs show `HikariPool-1 - Connection is not available, request timed out`.
