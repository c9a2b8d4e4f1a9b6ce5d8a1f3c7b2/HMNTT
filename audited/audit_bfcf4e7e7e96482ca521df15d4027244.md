### Title
Missing File ID Validation Allows Arbitrary Address Book Queries in `/api/v1/network/nodes`

### Summary
The `getNodes()` endpoint in `NetworkController` accepts a user-supplied `file.id` query parameter and passes it directly to the database query without validating that it is one of the two legitimate address book file IDs (101 or 102). An unprivileged external user can supply any arbitrary numeric file ID, causing the repository to query against a non-existent or non-standard address book, returning incorrect or empty node records to API consumers.

### Finding Description

**Code path:**

1. `NetworkController.getNodes()` ( [1](#0-0) ) receives the `file.id` parameter. The only check performed is that the operator must be `EQ` — no validation of the actual value occurs.

2. `NetworkServiceImpl.getAddressBookFileId()` ( [2](#0-1) ) returns the raw user-supplied value directly if `fileId != null`, with no allowlist check against `{101, 102}`.

3. That value is forwarded verbatim to `NetworkNodeRepository.findNetworkNodes()` ( [3](#0-2) ).

4. The native SQL query filters the `address_book` table with `where file_id = :fileId` ( [4](#0-3) ). If the supplied ID matches no address book, the CTE `latest_address_book` returns empty, the join produces zero rows, and the API returns an empty node list.

**Root cause:** The assumption that only valid file IDs (101/102) will be supplied is never enforced. The controller only rejects non-EQ operators, not out-of-range values.

### Impact Explanation
Any unauthenticated caller can force the `/api/v1/network/nodes` endpoint to return either an empty node list (for a non-existent file ID) or data from a non-standard address book (e.g., file 101, the node-details file, which has a different schema/semantics than file 102). Clients relying on this endpoint for network topology discovery — such as SDKs bootstrapping their node list — could receive incorrect or empty data, potentially disrupting their ability to connect to the correct network nodes. Severity is **medium**: no authentication bypass or data from other tenants is exposed, but the integrity of the returned network topology is compromised.

### Likelihood Explanation
Exploitation requires zero privileges — only the ability to send an HTTP GET request. The parameter name (`file.id`) is documented in the API. The attack is trivially repeatable and requires no special tooling. Any external user or automated scanner can trigger it.

### Recommendation
In `NetworkServiceImpl.getAddressBookFileId()`, validate that the user-supplied file ID is one of the two permitted values before using it:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    long file101 = systemEntity.addressBookFile101().getId();
    long file102 = systemEntity.addressBookFile102().getId();
    if (id != file101 && id != file102) {
        throw new IllegalArgumentException(
            "file.id must be " + file101 + " or " + file102);
    }
    return id;
}
```

Alternatively, add a `@Pattern` or custom constraint on the `fileId` field in `NetworkNodeRequest` to reject values outside the allowed set at the binding layer.

### Proof of Concept

```
# Returns empty nodes list (no address book with file_id=999 exists)
GET /api/v1/network/nodes?file.id=eq:999

# Returns nodes from file 101 (node-details file) instead of standard file 102
GET /api/v1/network/nodes?file.id=eq:101
```

Expected (correct) behavior: both requests should be rejected with a `400 Bad Request`. Actual behavior: the first returns `{"nodes": [], "links": {...}}` and the second returns data from the wrong address book — both silently, with HTTP 200.
