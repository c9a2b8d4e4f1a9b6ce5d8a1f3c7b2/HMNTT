### Title
DUPLICATE_TRANSACTION Timestamp Shadowing Suppresses Contract Action Trace in `/results/:transactionIdOrHash/actions`

### Summary
When an Ethereum transaction hash resolves to multiple entries in `contract_transaction_hash` ŌĆö a failed original execution followed by a later `DUPLICATE_TRANSACTION` submission ŌĆö the hash-lookup query returns the duplicate's `consensus_timestamp` (because it is newer and neither is `SUCCESS`). `getContractActions` then queries contract actions at that timestamp, which has none (DUPLICATE_TRANSACTION is never executed on the EVM), returning an empty action list and erasing the original failed execution's trace from the API.

### Finding Description

**Code path:**

`rest/controllers/contractController.js` ŌĆö `getContractActions()` (lines 1248ŌĆō1261):

```js

  tx = await ContractService.getContractTransactionDetailsByHash(utils.parseHexStr(transactionIdOrHash));
}
// ...

  ...
}
const rows = await ContractService.getContractActionsByConsensusTimestamp(
  consensusTimestamp, payerAccountId, filters, order, limit

``` [1](#0-0) 

**Root cause ŌĆö `ethereumTransactionsByHashQuery`** in `rest/service/contractService.js` (lines 208ŌĆō212):

```sql
select * from contract_transaction_hash
where hash = $1
order by (transaction_result = {successTransactionResult}) desc,
         consensus_timestamp desc
limit 1
``` [2](#0-1) 

The ordering is: SUCCESS first, then latest `consensus_timestamp`. When the original transaction failed (e.g., `CONSENSUS_GAS_EXHAUSTED`) at T1 and a duplicate was submitted later at T2 (result = `DUPLICATE_TRANSACTION`), neither is `SUCCESS`, so the tiebreaker is timestamp. T2 > T1, so the query returns T2 ŌĆö the duplicate ŌĆö not the original.

**Why DUPLICATE_TRANSACTION entries exist in the table:** The importer's migration populates `contract_transaction_hash` from `ethereum_transaction` for rows that have no corresponding `contract_result`. Since `DUPLICATE_TRANSACTION` ethereum transactions are explicitly skipped by `addDefaultEthereumTransactionContractResult` (importer code, lines 100ŌĆō107), they have no `contract_result` row, so they are inserted via the second migration path. [3](#0-2) [4](#0-3) 

`getContractActions` then calls `getContractActionsByConsensusTimestamp(T2, ...)`. Because `DUPLICATE_TRANSACTION` is never executed by the EVM, no `contract_action` rows exist at T2. The endpoint returns `{"actions": [], "links": {"next": null}}` ŌĆö the original execution's action trace is invisible. [5](#0-4) 

### Impact Explanation
An attacker can permanently suppress the on-mirror-node action trace of any failed Ethereum transaction (e.g., one that reverted mid-execution, revealing internal call structure). After the duplicate is submitted, every call to `/api/v1/contracts/results/{hash}/actions` returns an empty list, misrepresenting that no EVM execution occurred for that hash. Tooling, block explorers, and auditors relying on this endpoint will see a blank trace instead of the real one. The actual ledger is unaffected, but the mirror node ŌĆö the canonical read API ŌĆö is corrupted for that hash.

### Likelihood Explanation
Any unprivileged Hedera account holder can submit an Ethereum transaction to the network. Submitting a duplicate of a known failed transaction costs only the standard transaction fee (a small HBAR amount). The target hash is publicly visible on-chain. The attack is repeatable and permanent (the duplicate record persists). No special access, keys, or permissions are required beyond a funded Hedera account.

### Recommendation
In `ethereumTransactionsByHashQuery`, add `DUPLICATE_TRANSACTION` (and `WRONG_NONCE`) to the ordering priority so that a prior non-duplicate, non-wrong-nonce result is always preferred over a later duplicate:

```sql
order by
  (transaction_result = {successTransactionResult}) desc,
  (transaction_result <> {duplicateTransactionResult} AND transaction_result <> {wrongNonceTransactionResult}) desc,
  consensus_timestamp desc
limit 1
```

Alternatively, explicitly exclude `DUPLICATE_TRANSACTION` and `WRONG_NONCE` rows from the query when a non-duplicate row exists for the same hash, or add a `WHERE transaction_result NOT IN (DUPLICATE_TRANSACTION, WRONG_NONCE)` sub-query fallback.

### Proof of Concept

1. Submit an Ethereum transaction `T` to Hedera that will fail (e.g., call a contract that reverts after several internal calls). Record its hash `H` and confirm `/api/v1/contracts/results/H/actions` returns a non-empty action list at timestamp T1.

2. Submit the identical raw Ethereum transaction bytes again to any Hedera node. The network records it at timestamp T2 > T1 with result `DUPLICATE_TRANSACTION`. No EVM execution occurs; no `contract_action` rows are written for T2.

3. Query `/api/v1/contracts/results/H/actions` again.

4. **Observed:** `{"actions": [], "links": {"next": null}}` ŌĆö empty, because `getContractTransactionDetailsByHash` now resolves H to T2 (later timestamp, neither is SUCCESS), and no contract actions exist at T2.

5. **Expected:** The original action trace from T1 should be returned.
