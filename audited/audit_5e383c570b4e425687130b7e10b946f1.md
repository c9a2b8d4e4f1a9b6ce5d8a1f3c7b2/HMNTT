### Title
Cache Key Hash Collision in `findByEvmAddressOrAliasAndDeletedIsFalse` Allows Wrong Entity Resolution via `Arrays.hashCode` Preimage Collision

### Summary
The `@Cacheable` annotation on `findByEvmAddressOrAliasAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` as the sole cache key, which delegates to `Arrays.hashCode(byte[])` — a 32-bit polynomial hash with well-known collision properties. Two distinct byte arrays that produce the same `int` hash value are treated as identical cache keys by the Caffeine cache, causing the second lookup to return the cached entity of the first address. This allows an unprivileged caller to receive incorrect entity data for a given EVM address or alias, corrupting smart contract execution state on the mirror node's web3 API.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode` at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java` lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);  // returns a 32-bit int
}
```

`findByEvmAddressOrAliasAndDeletedIsFalse` at `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` lines 39–49:
```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← sole cache key: a 32-bit int
        unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

**Root cause:** `Arrays.hashCode(byte[])` computes `result = 31 * result + b` for each byte `b` and returns a 32-bit `int`. The output space is only 2³² (~4 billion). The returned `int` is autoboxed to `Integer` and used as the Caffeine cache key. Caffeine uses `Integer.equals()` for key comparison, so two distinct byte arrays producing the same `int` hash are indistinguishable at the cache layer — no secondary equality check on the actual byte content is performed.

**Why `unless = "#result == null"` is insufficient:** This guard only prevents caching of empty `Optional` results. It does nothing to detect or prevent a collision between two different non-null byte arrays that share the same 32-bit hash.

**Cache configuration** (`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java` line 19):
```
expireAfterWrite=1s, maximumSize=10000
```
The 1-second TTL limits but does not eliminate the window.

**Exploit flow:**
1. Attacker analytically computes two byte arrays `A` and `B` such that `Arrays.hashCode(A) == Arrays.hashCode(B)` (trivial for a 32-bit polynomial hash — e.g., swap two bytes at positions that cancel in the polynomial).
2. Address `A` must correspond to a real entity in the system (attacker can register their own Hedera account with a crafted EVM address/alias to control `A`).
3. Attacker sends `eth_call` or equivalent targeting address `A` → entity for `A` is fetched from DB and stored in cache under key `hashCode(A)`.
4. Within 1 second, attacker sends a second request targeting address `B` → cache lookup finds key `hashCode(B) == hashCode(A)` → returns entity for `A` instead of querying the DB for `B`.
5. The EVM execution context for address `B` now uses entity `A`'s data (account ID, balance, bytecode pointer, etc.).

The same collision applies identically to `findByEvmAddressAndDeletedIsFalse` at lines 32–37, which uses the same `key = "@spelHelper.hashCode(#alias)"` pattern.

### Impact Explanation
An attacker can cause the mirror node's web3 API to resolve a target EVM address to a completely different entity. Consequences include: `eth_getCode` returning wrong bytecode for a contract address, `eth_getBalance` returning wrong balance, `eth_call` executing against wrong contract state, and `eth_estimateGas` producing incorrect results. Any dApp or protocol relying on the mirror node's web3 endpoint for contract calls or address resolution will receive corrupted data. Because the mirror node is the canonical web3 gateway for Hedera, this affects all consumers of that endpoint simultaneously during the collision window.

### Likelihood Explanation
The attack requires no privileges — only the ability to send HTTP requests to the web3 API endpoint. Crafting `Arrays.hashCode` collisions for 20-byte EVM addresses is analytically trivial: the polynomial `h = 31*h + b` over bytes has many known collision-generation techniques (e.g., adjusting two bytes at known positions to cancel). The attacker controls address `A` by registering a Hedera account with a chosen EVM address. The 1-second TTL requires timing two requests within 1 second, which is easily achievable programmatically. The attack is repeatable on every cache refresh cycle (every second).

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the actual byte array content, using a wrapper that implements proper `equals`/`hashCode` based on array content (e.g., `ByteBuffer.wrap(alias)` or `new SimpleKey(alias)` which Spring's `SimpleKeyGenerator` handles correctly). Alternatively, use `HexFormat.of().formatHex(alias)` to convert the byte array to a hex string as the cache key — strings use content equality. The fix for both affected methods:

```java
key = "T(java.util.HexFormat).of().formatHex(#alias)"
// or
key = "T(java.nio.ByteBuffer).wrap(#alias)"
```

The `ByteBuffer` approach is preferred as it avoids string allocation and `ByteBuffer.equals()` performs content comparison.

// Swap b0 and b1 such that 31*b0 + 31^2*b1 == 31*b0' + 31^2*b1' (mod 2^32)
// This is solvable: pick b0=0,b1=31 vs b0=31*31 mod 256, b1=... (many solutions exist)
