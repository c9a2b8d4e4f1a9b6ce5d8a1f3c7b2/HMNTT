### Title
Stale EVM Address Cache Entry Persists After Entity Deletion in `EntityIdServiceImpl`, Corrupting Mirror Node Transaction History

### Summary
`EntityIdServiceImpl.notify()` silently returns early when called with a deleted entity, but never evicts the corresponding EVM address entry from the in-memory Caffeine cache. Because `cacheLookup()` uses Spring's `cache.get(key, loader)` — which only invokes the DB loader on a cache miss — any EVM address cached before deletion will continue resolving to the deleted entity's `EntityId` indefinitely, bypassing the `deleted <> true` guard in `EntityRepository.findByEvmAddress()`. Any Hedera user who owns a contract can trigger this condition without elevated privileges.

### Finding Description

**Exact code path:**

`EntityRepository.findByEvmAddress` correctly guards against deleted entities at the DB layer:

```java
// EntityRepository.java line 19-20
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
``` [1](#0-0) 

However, `cacheLookup()` uses `cache.get(key, loader)`, which only calls the loader on a cache miss. Once a value is cached, it is returned directly without re-querying the DB:

```java
// EntityIdServiceImpl.java line 116-123
private @NonNull Optional<EntityId> cacheLookup(ByteString key, Callable<Optional<EntityId>> loader) {
    try {
        return Objects.requireNonNullElse(cache.get(key, loader), Optional.empty());
    } ...
}
``` [2](#0-1) 

When a delete transaction is processed, `AbstractEntityCrudTransactionHandler.updateEntity()` sets `entity.setDeleted(true)` and calls `doUpdateEntity()`, which calls `entityListener.onEntity(entity)`: [3](#0-2) 

`SqlEntityListener.onEntity()` then calls `entityIdService.notify(entity)`: [4](#0-3) 

`notify()` detects the deletion flag and **returns early without evicting the cache entry**:

```java
// EntityIdServiceImpl.java line 137-140
public void notify(Entity entity) {
    if (entity == null || (entity.getDeleted() != null && entity.getDeleted())) {
        return;  // ← cache entry for this EVM address is NEVER evicted
    }
``` [5](#0-4) 

**The existing test explicitly confirms this behavior is present and persistent:** [6](#0-5) 

Lines 84–88 assert that after marking the entity deleted in the DB and even after physically deleting the row, `lookup()` still returns the stale `EntityId` — only a full cache `reset()` clears it.

**Root cause:** `notify()` was designed to only warm the cache on creation/update, but it has no corresponding eviction path for deletions. The assumption that "a deleted entity will never be looked up again" is false because the cache key (EVM address) can be referenced by subsequent transactions before the cache expires or the importer restarts.

### Impact Explanation

The mirror node's `entity_transaction` and related tables will associate post-deletion transactions (e.g., failed calls to the deleted contract's EVM address, or a new contract deployed at the same CREATE2 address before the cache is refreshed) with the deleted entity's `EntityId`. This corrupts the mirror node's historical record of which entity was involved in each transaction. Downstream consumers of the mirror node REST API (block explorers, wallets, dApps) will receive incorrect entity associations, misattributing transaction history to a deleted entity. The Caffeine cache has no forced TTL visible in `CacheConfiguration` — the stale entry persists for the entire lifetime of the importer process unless a restart occurs. [7](#0-6) 

### Likelihood Explanation

Any Hedera user with a funded account can:
1. Deploy a contract (no special privileges — just HBAR for fees)
2. Trigger a lookup by EVM address (happens naturally as any transaction referencing the contract is processed)
3. Delete their own contract (they hold the admin key by default)

No privileged access, no network-level exploit, no key theft required. The attacker only needs to own a contract, which is a standard user capability. The attack is repeatable across any number of contracts and is not detectable at the network consensus layer since all transactions are valid.

### Recommendation

In `EntityIdServiceImpl.notify()`, instead of returning early on deletion, explicitly evict the cache entry:

```java
public void notify(Entity entity) {
    if (entity == null) {
        return;
    }
    byte[] aliasBytes = entity.getAlias() != null ? entity.getAlias() : entity.getEvmAddress();
    if (aliasBytes == null) {
        return;
    }
    var alias = DomainUtils.fromBytes(aliasBytes);
    if (entity.getDeleted() != null && entity.getDeleted()) {
        cache.evict(alias);  // ← ADD THIS
        if (entity.getAlias() != null && entity.getEvmAddress() != null) {
            cache.evict(DomainUtils.fromBytes(entity.getEvmAddress()));
        }
        return;
    }
    // ... existing put logic
}
```

Alternatively, replace the stale entry with `Optional.empty()` using `cache.put(alias, Optional.empty())` so that subsequent lookups return empty without hitting the DB loader. [8](#0-7) 

### Proof of Concept

1. **Setup**: Deploy a smart contract `C` with EVM address `X` on the Hedera network using any standard account. Note the assigned entity ID (e.g., `0.0.12345`).

2. **Populate cache**: Submit any transaction that references `C` by EVM address `X` (e.g., a `ContractCallQuery`). The importer processes this, calls `cacheLookup(X, () -> findByEvmAddress(X))`, the DB returns `12345` (not deleted), and the cache is populated: `X → EntityId(0.0.12345)`.

3. **Delete the contract**: Submit a `CONTRACTDELETEINSTANCE` transaction for `C`. The importer processes it: `AbstractEntityCrudTransactionHandler` sets `entity.setDeleted(true)`, calls `entityListener.onEntity(entity)` → `entityIdService.notify(entity)` → hits line 138, returns early. **Cache entry `X → EntityId(0.0.12345)` is NOT evicted.**

4. **Trigger stale lookup**: Submit any subsequent transaction that references EVM address `X` (e.g., a failed `ContractCall` to the now-deleted address, or a new contract deployed at the same CREATE2 address). The importer calls `cacheLookup(X, ...)` — cache hit — returns `EntityId(0.0.12345)` without querying the DB.

5. **Observe corruption**: In the mirror node database, the new transaction's `entity_id` field is set to `0.0.12345` (the deleted entity), not empty or the new entity. The `entity_transaction` table now contains a record linking a post-deletion transaction to the deleted contract, corrupting the historical record.

6. **Confirm**: This exact behavior is reproducible by running the existing test `EntityIdServiceImplTest.cache()` — lines 84–88 assert the stale cache hit as currently expected behavior, confirming the issue is present in the codebase. [9](#0-8)

    @Override
    public void notify(Entity entity) {
        if (entity == null || (entity.getDeleted() != null && entity.getDeleted())) {
            return;
        }

        byte[] aliasBytes = entity.getAlias() != null ? entity.getAlias() : entity.getEvmAddress();
        if (aliasBytes == null) {
            return;
        }

        var alias = DomainUtils.fromBytes(aliasBytes);
        var entityId = Optional.ofNullable(entity.toEntityId());
        EntityType type = entity.getType();

        switch (type) {
            case ACCOUNT -> {
                cache.put(alias, entityId);

                // Accounts can have an alias and an EVM address so warm the cache with both
                if (entity.getAlias() != null && entity.getEvmAddress() != null) {
                    cache.put(fromBytes(entity.getEvmAddress()), entityId);
                }
            }
            case CONTRACT -> cache.put(alias, entityId);
            default -> Utility.handleRecoverableError("Invalid Entity: {} entity can't have alias", type);
        }
    }
