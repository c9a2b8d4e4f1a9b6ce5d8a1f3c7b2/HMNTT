### Title
Unauthenticated GraphQL Parser Amplification via Dual Token Counter Exhaustion with No Rate Limiting

### Summary
The GraphQL endpoint in `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` configures `maxTokens(1000)` and `maxWhitespaceTokens(1000)` as independent counters, allowing an unauthenticated attacker to force the parser to process up to ~2000 tokens per request (both counters maxed simultaneously). Combined with the absence of any rate limiting on the GraphQL module (unlike the `web3` module which has `ThrottleConfiguration`) and trivial bypass of the preparsed document cache via unique query strings, an attacker can sustain maximum parser CPU load indefinitely, degrading processing nodes.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 31–39:
```java

    b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
ParserOptions.setDefaultParserOptions(...transform(consumer));
ParserOptions.setDefaultOperationParserOptions(...transform(consumer));
```

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

**Root cause:** `maxTokens` and `maxWhitespaceTokens` are tracked by separate counters in graphql-java's `ParserOptions`. A single query document can legally contain up to 999 whitespace tokens AND up to 999 regular tokens simultaneously — both limits are independent. This means the parser processes up to ~1999 tokens per request, roughly double what a non-padded query would require. The failed assumption is that setting both limits to 1000 provides equivalent protection to a single 1000-token limit; it does not — it doubles the maximum parsing work per request.

**Exploit flow:**
1. Attacker crafts a query with 999 whitespace tokens (spaces/newlines interspersed) plus a maximally complex `account` query (complexity approaching 200, depth approaching 10) — all within the 10000-character limit.
2. To bypass the Caffeine preparsed document cache (`maximumSize=1000`, keyed on raw query string), the attacker trivially varies a comment or whitespace count by 1 per request, generating unique cache-miss queries indefinitely.
3. Each unique request forces a full parse + validation pass over ~1999 tokens, then `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` run post-parse — they do not prevent the parsing overhead.
4. No rate limiting exists in the `graphql` module (contrast: `web3` has `ThrottleConfiguration` with `requestsPerSecond=500`). The GraphQL endpoint is publicly accessible with no authentication.
5. With concurrent connections flooding unique max-padded queries, parser CPU is saturated.

**Why existing checks fail:**
- `maxCharacters(10000)`: Does not prevent the dual-counter amplification; 999 whitespace chars + ~9000 chars of query body fits comfortably.
- `MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)`: These run **after** parsing is complete; they do not reduce parser work.
- `CachedPreparsedDocumentProvider`: Cache is keyed on the raw query string. Changing a single whitespace count produces a cache miss. With `maximumSize=1000`, an attacker can fill and continuously evict the cache.
- No rate limiter exists in the `graphql` module at all.

### Impact Explanation
An unauthenticated attacker can force each GraphQL node to perform ~2x the parser CPU work per request compared to a non-padded query, with no server-side rate limiting to slow the attack. Under sustained concurrent load of unique max-padded queries, CPU utilization on GraphQL pods will spike (the `GraphQLHighCPU` alert fires at 80% for 5 minutes), degrading or crashing 30%+ of processing nodes. The mirror node's GraphQL API becomes unavailable, blocking account query access for all consumers.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is publicly accessible via HTTP POST to `/graphql/alpha`. No special knowledge beyond the GraphQL schema is needed. The exploit is trivially scriptable: generate unique queries by varying whitespace count from 1 to 999 in a loop. The lack of rate limiting means there is no server-side throttle to overcome. Any attacker with moderate bandwidth can sustain this indefinitely.

### Recommendation
1. **Add rate limiting to the GraphQL module**: Implement a `ThrottleConfiguration` equivalent (bucket4j or similar) for the `/graphql/alpha` endpoint, mirroring the `web3` module's `requestsPerSecond` limit.
2. **Unify the token budget**: Replace the separate `maxTokens(1000)` + `maxWhitespaceTokens(1000)` with a single combined token budget (e.g., `maxTokens(1000)` total including whitespace), or reduce `maxWhitespaceTokens` significantly (e.g., 50–100).
3. **Normalize whitespace before cache keying**: Strip/normalize whitespace in `CachedPreparsedDocumentProvider` before using the query as a cache key, so whitespace-padded variants of the same query hit the cache.

### Proof of Concept
```bash
#!/bin/bash
# Generate unique max-whitespace-padded account queries and flood the endpoint
for i in $(seq 1 10000); do
  # Vary whitespace count to bypass cache (i spaces prepended as comments)
  PADDING=$(python3 -c "print(' ' * $i)")
  QUERY="{\"query\": \"${PADDING}{ account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { alias autoRenewPeriod balance createdTimestamp declineReward deleted expirationTimestamp id key maxAutomaticTokenAssociations memo nonce pendingReward receiverSigRequired stakePeriodStart type } }\"}"
  curl -s -X POST http://<TARGET>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
wait
```
