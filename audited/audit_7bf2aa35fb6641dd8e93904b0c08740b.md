### Title
Silent Long Overflow in `getAdjustedLowerRangeValue()` Bypasses Range Validation in `Bound`

### Summary
An unprivileged user can supply `account.id=gt:511.65535.274877906943` — a fully valid entity ID whose encoded value equals `Long.MAX_VALUE` — causing `getAdjustedLowerRangeValue()` to silently overflow to `Long.MIN_VALUE` via `lowerBound++`. The subsequent range-validity guard `adjustedLower > adjustedUpper` evaluates to `Long.MIN_VALUE > Long.MAX_VALUE` = `false`, so no `IllegalArgumentException` is thrown and the malformed `Bound` is passed to the repository layer.

### Finding Description

**Root cause — `Bound.getAdjustedLowerRangeValue()` (line 86-97):**

```java
public long getAdjustedLowerRangeValue() {
    if (this.lower == null) { return 0; }
    long lowerBound = this.lower.value();
    if (this.lower.operator() == RangeOperator.GT) {
        lowerBound++;          // ← unchecked increment; overflows when value == Long.MAX_VALUE
    }
    return lowerBound;
}
``` [1](#0-0) 

**Guard that is bypassed — `Bound` constructor (line 56-60):**

```java
long adjustedLower = getAdjustedLowerRangeValue(); // Long.MIN_VALUE after overflow
long adjustedUpper = adjustUpperBound();            // Long.MAX_VALUE (no upper supplied)
if (primarySortField && adjustedLower > adjustedUpper) { // MIN > MAX == false → skipped
    throw new IllegalArgumentException(...);
}
``` [2](#0-1) 

**Same bypass in `checkOwnerSpenderParamValidity` (line 41-44):**

```java
if (!ownerOrSpenderParams.hasLowerAndUpper()
        && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
    throw new IllegalArgumentException("Invalid range provided for token.id");
}
``` [3](#0-2) 

**Why `Long.MAX_VALUE` is a reachable, valid entity-ID value:**

`EntityId.encode()` validates each component independently against its bit-width mask:
- shard ≤ 1023 (10 bits), realm ≤ 65535 (16 bits), num ≤ 274877906943 (38 bits)

`EntityId.of(511, 65535, 274877906943)` passes all three guards and encodes to exactly `Long.MAX_VALUE` (9223372036854775807), as confirmed by the test suite. [4](#0-3) 

`EntityIdRangeParameter.valueOf("gt:511.65535.274877906943")` parses this without error and produces `EntityIdRangeParameter(GT, Long.MAX_VALUE)`. [5](#0-4) 

**Exploit flow:**

1. `AllowancesController` receives `account.id=gt:511.65535.274877906943` and constructs `new Bound(accountIds, true, ACCOUNT_ID, field)`.
2. Inside the `Bound` constructor, `getAdjustedLowerRangeValue()` increments `Long.MAX_VALUE` → wraps to `Long.MIN_VALUE`.
3. The guard `Long.MIN_VALUE > Long.MAX_VALUE` is `false`; no exception is raised.
4. `checkOwnerSpenderParamValidity` is called; the same overflowed comparison at line 42 also evaluates to `false`.
5. `repository.findAll(request, id)` is invoked with the structurally invalid `Bound`. [6](#0-5) 

### Impact Explanation

The immediate observable impact is that the server silently accepts a semantically impossible range (`GT Long.MAX_VALUE`) instead of returning HTTP 400. The `lower` field retains the original `GT:Long.MAX_VALUE` operator/value, so the generated SQL predicate `spender > 9223372036854775807` returns zero rows — no data is disclosed. However, the overflowed value (`Long.MIN_VALUE`) is also consumed by `hasEqualBounds()` and `adjustLowerRange()`, which could produce incorrect query rewrites if the code path is reached with a combined lower+upper bound. Additionally, `getInclusiveRangeValue()` contains the identical unchecked `value += 1L` pattern and would overflow identically if called on this bound. [7](#0-6) 

Severity is **low**: no authorization boundary is crossed (NFT allowances are public data), and the limit parameter is still enforced. The primary risk is logic corruption in range-rewrite paths and silent contract violation of the validation layer.

### Likelihood Explanation

Exploitability requires zero privileges — only a valid HTTP GET to `/api/v1/accounts/{id}/allowances/nfts`. The trigger value (`511.65535.274877906943`) is a syntactically normal Hedera entity ID that passes every existing input-validation layer. Any user who discovers the boundary condition (e.g., by fuzzing entity-ID ranges) can reproduce it deterministically on every request.

### Recommendation

Add an explicit overflow guard in `getAdjustedLowerRangeValue()`:

```java
public long getAdjustedLowerRangeValue() {
    if (this.lower == null) { return 0; }
    long lowerBound = this.lower.value();
    if (this.lower.operator() == RangeOperator.GT) {
        if (lowerBound == Long.MAX_VALUE) {
            throw new IllegalArgumentException(
                "GT operator with maximum entity ID value produces no valid range for " + parameterName);
        }
        lowerBound++;
    }
    return lowerBound;
}
```

Apply the same guard to `getInclusiveRangeValue()` and `NumberRangeParameter.getInclusiveValue()` / `EntityIdRangeParameter.getInclusiveValue()`, which contain identical unchecked increment/decrement operations. [8](#0-7) [9](#0-8) 

### Proof of Concept

```
GET /api/v1/accounts/0.0.1/allowances/nfts?account.id=gt:511.65535.274877906943
```

**Expected (correct) behavior:** HTTP 400 — "Invalid range provided for account.id"

**Actual behavior:** HTTP 200 — empty allowances list; no exception thrown; range validation silently bypassed.

Trace:
1. `EntityIdRangeParameter.valueOf("gt:511.65535.274877906943")` → `(GT, 9223372036854775807)`
2. `Bound` constructor → `getAdjustedLowerRangeValue()` → `9223372036854775807 + 1` = `-9223372036854775808` (overflow)
3. Guard: `-9223372036854775808 > 9223372036854775807` → `false` → no exception
4. `checkOwnerSpenderParamValidity` → same comparison → `false` → no exception
5. Query executes with `spender > 9223372036854775807` → 0 rows returned

    private static EntityId getEntityId(String entityId) {
        List<Long> parts = Splitter.on('.')
                .splitToStream(Objects.requireNonNullElse(entityId, ""))
                .map(Long::valueOf)
                .filter(n -> n >= 0)
                .toList();

        if (parts.size() != StringUtils.countMatches(entityId, ".") + 1) {
            throw new IllegalArgumentException("Invalid entity ID");
        }

        var properties = CommonProperties.getInstance();
        return switch (parts.size()) {
            case 1 -> EntityId.of(properties.getShard(), properties.getRealm(), parts.get(0));
            case 2 -> EntityId.of(properties.getShard(), parts.get(0), parts.get(1));
            case 3 -> EntityId.of(parts.get(0), parts.get(1), parts.get(2));
            default -> throw new IllegalArgumentException("Invalid entity ID");
        };
    }
