### Title
Stale Entity Lookup in `getReceiverAddress()` Produces Divergent Opcode Trace for Deleted Long-Zero Targets

### Summary
`OpcodeServiceImpl.getReceiverAddress()` resolves a long-zero `toAddress` to its canonical EVM alias using `commonEntityAccessor.get(address, Optional.empty())` — a **current-state** query that returns empty for deleted entities. The `consensusTimestamp` is available in the calling scope but is never passed, so any entity deleted after the original transaction causes the receiver to silently fall back to the raw long-zero address instead of the entity's EVM alias. Because the EVM stores contract code under the alias address, the replay executes against a different (potentially empty) target, producing a trace that diverges from the historical execution.

### Finding Description

**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, `getReceiverAddress()`, lines 182–205; the defective call is line 191. [1](#0-0) 

**Root cause:** `commonEntityAccessor.get(address, Optional.empty())` dispatches to `getEntityByMirrorAddressAndTimestamp(address, Optional.empty())`: [2](#0-1) 

With `Optional.empty()`, the branch taken is `entityRepository.findByIdAndDeletedIsFalse(entityId)` — a live-state query that returns `Optional.empty()` for any entity whose `deleted` flag is `true`. [3](#0-2) 

When the entity is absent, `entity` is null, the `if (entity != null)` guard is skipped, and the function falls through to `return address` — returning the raw long-zero address instead of the entity's EVM alias. [4](#0-3) 

**The fix exists in the same file but is not applied here.** `buildCallServiceParameters` has `consensusTimestamp` in scope and passes it to `ContractDebugService`, which correctly sets the historical context: [5](#0-4) 

But `getReceiverAddress` is called without the timestamp: [6](#0-5) 

The historical query `findActiveByIdAndTimestamp` can correctly retrieve entities that existed at `consensusTimestamp` even if they are now deleted: [7](#0-6) 

**Exploit flow:**
1. Attacker deploys a contract. It receives entity ID `0.0.N` (long-zero address `0x0000…000N`) and an EVM alias `0xALIAS`.
2. Attacker sends an EthereumTransaction with `toAddress = 0x0000…000N`. The original execution resolves the entity, uses `0xALIAS` as the canonical receiver, and executes the contract code — producing trace T₁.
3. Attacker deletes the contract (using the admin key they hold).
4. Attacker calls `GET /api/v1/contracts/results/{txHash}/opcodes`.
5. `getReceiverAddress` calls `findByIdAndDeletedIsFalse` → returns empty → falls through → receiver is set to `0x0000…000N`.
6. The EVM replay targets `0x0000…000N`. Contract code is stored under `0xALIAS`; the long-zero address has no code in the EVM state. The replay produces trace T₂ (empty/failed execution).
7. T₁ ≠ T₂: the API returns a fabricated trace that contradicts the actual historical execution.

### Impact Explanation
An attacker can make any historical contract call — including successful ones — appear to have executed against an empty address, producing a "failed" or empty opcode trace. This directly corrupts the forensic record exposed by the opcodes API. Downstream consumers (auditors, dispute-resolution systems, block explorers, compliance tools) that rely on this endpoint to reconstruct what a transaction did will receive a false account. The attacker controls the timing of the deletion and can selectively corrupt traces for specific transactions of interest.

### Likelihood Explanation
The precondition (entity deletion) is fully within the attacker's control: any account or contract owner can delete their own entity on Hedera. No privileged access, governance approval, or network-level action by a third party is required. The attack is repeatable for any transaction whose `toAddress` is a long-zero address pointing to a now-deleted entity. The opcodes endpoint is publicly accessible (subject only to the gzip header and throttle checks, neither of which is a security barrier).

### Recommendation
Pass `consensusTimestamp` into `getReceiverAddress` and use it for the entity lookup:

```java
// In buildCallServiceParameters, change:
.receiver(getReceiverAddress(ethTransaction, contractResult, transactionType))
// to:
.receiver(getReceiverAddress(ethTransaction, contractResult, transactionType, consensusTimestamp))

// Update the method signature and the lookup:
private Address getReceiverAddress(
        EthereumTransaction ethereumTransaction, ContractResult contractResult,
        int transactionType, Long consensusTimestamp) {
    if (ethereumTransaction != null) {
        if (ArrayUtils.isEmpty(ethereumTransaction.getToAddress())) {
            return EMPTY_ADDRESS;
        }
        final var address = Address.wrap(Bytes.wrap(ethereumTransaction.getToAddress()));
        if (ConversionUtils.isLongZero(address)) {
            final var entity = commonEntityAccessor
                    .get(address, Optional.of(consensusTimestamp))  // historical lookup
                    .orElse(null);
            if (entity != null) {
                return getEntityAddress(entity);
            }
        }
        return address;
    }
    // ... rest unchanged
}
```

Apply the same fix to `getSenderAddress` (line 178), which has the identical pattern.

### Proof of Concept

1. **Setup:** Deploy a Solidity contract on a Hedera testnet. Note its entity ID `0.0.N` (long-zero `0x0000…000N`) and EVM alias `0xALIAS`.
2. **Trigger transaction:** Send an EIP-1559 ethereum transaction with `to = 0x0000…000N` calling any function. Record the transaction hash `H`. Confirm the transaction succeeded and note the opcode trace T₁ via the opcodes API.
3. **Delete entity:** Submit a `ContractDeleteTransaction` for `0.0.N` using the admin key. Confirm deletion.
4. **Exploit:** Call `GET /api/v1/contracts/results/H/opcodes` with `Accept-Encoding: gzip`.
5. **Observe:** The returned trace T₂ shows execution against `0x0000…000N` (no code), diverging from T₁. The `failed` field is `true` and `opcodes` is empty or shows a bare STOP, contradicting the original successful execution.
6. **Root cause confirmation:** Add a debug log before line 191 in `getReceiverAddress` to print the timestamp passed — it will show `Optional.empty` instead of the historical consensus timestamp.
