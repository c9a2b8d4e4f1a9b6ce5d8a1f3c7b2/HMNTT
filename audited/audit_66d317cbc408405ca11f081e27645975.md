### Title
NullPointerException in `evmAddressFromId` When `ContractResult.senderId` Is Null

### Summary
`OpcodeServiceImpl.getSenderAddress()` passes `contractResult.getSenderId()` directly to `CommonEntityAccessor.evmAddressFromId()` without a null check. The `senderId` field on `ContractResult` is a nullable JPA-mapped `EntityId` with no `@NonNull` constraint. When a stored `ContractResult` has a null `senderId`, the call to `entityId.getId()` inside `evmAddressFromId` throws a `NullPointerException`, aborting the request with a 500 error.

### Finding Description

**Code path:**

`OpcodeServiceImpl.java` line 177–180:
```java
private Address getSenderAddress(ContractResult contractResult) {
    final var address = commonEntityAccessor.evmAddressFromId(contractResult.getSenderId(), Optional.empty());
    return address != null ? address : EMPTY_ADDRESS;
}
```
`contractResult.getSenderId()` returns `null` (no default, no `@NonNull`).

`CommonEntityAccessor.java` line 86–93:
```java
public Address evmAddressFromId(EntityId entityId, final Optional<Long> timestamp) {
    Entity entity = timestamp
            .map(t -> entityRepository
                    .findActiveByIdAndTimestamp(entityId.getId(), t)   // NPE branch (not taken)
                    .orElse(null))
            .orElseGet(() -> entityRepository
                    .findByIdAndDeletedIsFalse(entityId.getId())       // NPE HERE
                    .orElse(null));
```
Because `timestamp` is `Optional.empty()`, the `.orElseGet()` lambda is always executed. `entityId` is `null`, so `entityId.getId()` throws `NullPointerException`.

**Root cause:** `evmAddressFromId` is annotated with no null-guard on its `entityId` parameter, and `getSenderAddress` performs no null check before the call.

**Why existing checks fail:** The only guard in `getSenderAddress` is `address != null ? address : EMPTY_ADDRESS` — this is a post-call check that is never reached when the NPE fires inside `evmAddressFromId`.

### Impact Explanation
Any call to the opcode-tracing endpoint (`/api/v1/contracts/results/{transactionIdOrHash}/opcodes`) that resolves to a `ContractResult` row with a null `sender_id` column will receive an unhandled 500 Internal Server Error. The request processing pipeline for that call is fully aborted. No economic damage, no data leakage, no privilege escalation — impact is limited to denial of the opcode-tracing service for affected transactions.

### Likelihood Explanation
The `senderId` column is nullable in the schema (no `NOT NULL` constraint reflected in the domain model). Certain historical or system-initiated contract executions may legitimately have no sender recorded. An unprivileged user only needs to discover one such transaction hash or ID (e.g., by iterating known contract transaction hashes) and submit it to the endpoint. No special privileges, no write access, and no network-level attack are required — a single HTTP GET is sufficient to reproduce the crash.

### Recommendation
Add an explicit null guard in `getSenderAddress` before delegating to `evmAddressFromId`:

```java
private Address getSenderAddress(ContractResult contractResult) {
    final var senderId = contractResult.getSenderId();
    if (senderId == null) {
        return EMPTY_ADDRESS;
    }
    final var address = commonEntityAccessor.evmAddressFromId(senderId, Optional.empty());
    return address != null ? address : EMPTY_ADDRESS;
}
```

Alternatively, add a null guard at the top of `evmAddressFromId` in `CommonEntityAccessor`:
```java
public Address evmAddressFromId(EntityId entityId, final Optional<Long> timestamp) {
    if (entityId == null) return Address.ZERO;
    ...
}
```
