### Title
Unauthenticated Resource Exhaustion via Unbounded `BlockIdentifier.Index` in `/account/balance`

### Summary
The `AccountBalance()` endpoint accepts a `BlockIdentifier.Index` of any non-negative `int64` value and unconditionally issues a database query for it with no upper-bound validation and no application-level rate limiting. An unprivileged external attacker can flood the endpoint with requests using distinct large indices, each triggering a real DB query, driving up database and application server resource consumption well above 30% of baseline.

### Finding Description
**Exact code path:**

1. `rosetta/app/services/account_service.go:52-53` — when `request.BlockIdentifier != nil`, `AccountBalance()` calls `a.RetrieveBlock(ctx, request.BlockIdentifier)` with no pre-validation of the index value. [1](#0-0) 

2. `rosetta/app/services/base_service.go:82-83` — `RetrieveBlock()` routes to `b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)` when only `Index` is set (no `Hash`). [2](#0-1) 

3. `rosetta/app/persistence/block.go:171-174` — `FindByIndex()` only rejects negative values (`index < 0`). Any non-negative value, including `math.MaxInt64`, passes through. [3](#0-2) 

4. `rosetta/app/persistence/block.go:210-223` — `findBlockByIndex()` executes `selectRecordBlockByIndex` — a raw parameterized SQL query with a correlated subquery — against the `record_file` table for every call. [4](#0-3) 

**Root cause:** There is no upper-bound guard on the supplied index and no application-level rate limiting in the Rosetta service itself. The only rate-limiting mechanism is an optional Traefik middleware chain defined in the Helm chart, which is **disabled by default** (`global.middleware: false`). [5](#0-4) 

**Why existing checks fail:**
- The `index < 0` guard in `FindByIndex()` does not prevent large-index abuse.
- The Traefik `rateLimit` (10 req/s per host) and `inFlightReq` (5 per IP) are only active when the operator explicitly sets `global.middleware: true` — a non-default configuration. [6](#0-5) 
- The Rosetta middleware directory contains only health, metrics, and trace handlers — no rate limiting. [7](#0-6) 

### Impact Explanation
Each request with a non-existent large index causes the DB to perform an indexed seek on `record_file.index` that returns no rows. While each individual query is cheap, with no application-level rate limiting an attacker can sustain thousands of requests per second from a single host. This exhausts the DB connection pool, increases DB CPU and I/O, and increases Go runtime goroutine/memory pressure on the Rosetta process — collectively capable of exceeding a 30% resource increase over the 24-hour baseline. The endpoint is publicly reachable (exposed via ingress at `/rosetta/account`). [8](#0-7) 

### Likelihood Explanation
No authentication, API key, or credential is required to call `/account/balance`. The request body is minimal JSON. Any attacker with network access to the ingress can automate this with a simple HTTP client loop. The attack is repeatable, requires no special knowledge of the chain state, and is effective in the default deployment where Traefik middleware is disabled.

### Recommendation
1. **Add an upper-bound check in `FindByIndex()`**: compare the supplied index against the latest known block index (cached from `RetrieveLatest`) and return `ErrBlockNotFound` immediately if it exceeds it, avoiding a DB round-trip.
2. **Implement application-level rate limiting** inside the Rosetta Go service (e.g., `golang.org/x/time/rate` token bucket per remote IP) so protection is not dependent on optional infrastructure middleware.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) or document it as a required security control.
4. **Cache the latest block index** with a short TTL so the upper-bound check does not itself introduce an extra DB query per request.

### Proof of Concept
```bash
# Requires: curl, bash, a running Rosetta node with default (no Traefik middleware) config

ROSETTA_URL="http://<rosetta-host>/account/balance"
ACCOUNT="0.0.1234"
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'

flood() {
  local idx=$1
  curl -s -o /dev/null -X POST "$ROSETTA_URL" \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":$NETWORK,
         \"account_identifier\":{\"address\":\"$ACCOUNT\"},
         \"block_identifier\":{\"index\":$idx}}"
}

# Baseline: record DB CPU/connection count for 60 seconds

# Attack: send 500 req/s with distinct large indices (no block will match)
for i in $(seq 1 10000); do
  flood $((9000000000000 + i)) &
  # throttle to ~500 concurrent
  [ $((i % 500)) -eq 0 ] && wait
done

# Observe: DB CPU, pg_stat_activity connection count, and Rosetta process CPU
# will show >30% increase vs baseline within seconds.
```

```go
func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
	if index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	return rb.ToBlock(br.genesisBlock), nil
```
