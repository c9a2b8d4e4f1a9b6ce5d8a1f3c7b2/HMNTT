### Title
Sustained Three-Part UNION ALL DB Load Amplification via Paginated NFT Ownership Queries

### Summary
An unauthenticated external user can craft a single request to `GET /api/v1/accounts/{id}/nfts` with a closed range on both `token.id` and `serialnumber` that forces the server to build a three-part UNION ALL SQL query. The pagination `next` link generated by `getPaginationLink` preserves both the lower and upper bounds on both dimensions across every subsequent page, meaning every page in the entire result set triggers the same three-subquery UNION ALL execution. No rate limiting exists on the REST API for this endpoint, allowing an attacker to sustain this amplified DB load indefinitely.

### Finding Description

**Code path:**

1. `rest/controllers/accountController.js` `getNftsByAccountId()` (lines 90–103) calls `extractNftMultiUnionQuery()` then `NftService.getNfts(query)`, then `this.getPaginationLink(req, nfts, query.bounds, query.limit, query.order)`.

2. `extractNftMultiUnionQuery()` (lines 34–82) parses `token.id=gte:A&token.id=lte:B&serialnumber=gte:C&serialnumber=lte:D` into `bounds.primary` (lower=gte:A, upper=lte:B) and `bounds.secondary` (lower=gte:C, upper=lte:D). It then calls `getLowerFilters`, `getInnerFilters`, `getUpperFilters` from `baseController.js`.

3. In `baseController.js` (lines 131–183):
   - `getLowerFilters`: `primary.hasLower() && secondary.hasLower()` → `lower = [{token_id = A}, {serial_number >= C}]`
   - `getInnerFilters`: `primary.hasBound() && secondary.hasBound()` → `inner = [{token_id > A}, {token_id < B}]`
   - `getUpperFilters`: `primary.hasUpper() && secondary.hasUpper()` → `upper = [{token_id = B}, {serial_number <= D}]`

4. `nftService.js` `getQuery()` (lines 93–123) sees all three non-empty arrays and builds:
   ```sql
   (SELECT ... WHERE account_id=$1 AND token_id=$3 AND serial_number>=$4 ORDER BY ... LIMIT $2)
   UNION ALL
   (SELECT ... WHERE account_id=$1 AND token_id>$5 AND token_id<$6 ORDER BY ... LIMIT $2)
   UNION ALL
   (SELECT ... WHERE account_id=$1 AND token_id=$7 AND serial_number<=$8 ORDER BY ... LIMIT $2)
   ORDER BY token_id asc, serial_number asc LIMIT $2
   ```

5. `getPaginationLink` in `baseController.js` (lines 195–223) constructs the `next` link. For `order=asc` with `primaryBound.hasBound()=true` and `secondaryBound.hasBound()=true`:
   - `inclusive = !secondaryBound.hasEqual()` = `true`
   - `lastValues[token.id] = {inclusive: true, primary: true, value: lastRow.token_id}` → inserts `gte:lastRow.token_id`
   - `lastValues[serialnumber] = {value: lastRow.serial_number}` → inserts `gt:lastRow.serial_number`

6. In `utils.js` `getNextParamQueries()` (lines 865–893), for `order=asc`, `pattern = /gt[e]?:/`. The existing `gte:A` on `token.id` is removed and replaced with `gte:lastRow.token_id`, but `lte:B` is **not** removed (it doesn't match the `gt[e]:` pattern). Similarly `gte:C` on `serialnumber` is replaced with `gt:lastRow.serial_number`, but `lte:D` is preserved.

**Result:** The next link is `?order=asc&token.id=gte:lastRow.token_id&token.id=lte:B&serialnumber=gt:lastRow.serial_number&serialnumber=lte:D`. When followed, `extractNftMultiUnionQuery` again produces all three non-empty filter arrays, and the three-part UNION ALL is rebuilt identically on every page.

**Root cause:** `getPaginationLink` only replaces the lower bound (the `gte:` direction for ASC) of the primary key in the `next` link, but unconditionally preserves the upper bound (`lte:B`). The `isEmptyRange` check in `getNextParamQueries` only terminates pagination when `lower > upper` numerically — it never simplifies the query structure. No code path collapses the three-part UNION ALL into a simpler two-part or single query as pagination advances.

**No rate limiting:** `rest/server.js` (lines 68–103) shows no per-endpoint rate limiting middleware for the `/accounts/:id/nfts` route. The throttle infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` (EVM) module, not the REST API.

### Impact Explanation

Each page of results causes the database to execute three separate indexed scans on the `nft` table plus an outer merge sort, compared to one scan for a simple unbounded query. With `limit=1` (the minimum), an account holding N NFTs across a token range produces N pages, each firing a 3-subquery UNION ALL. An attacker targeting a high-NFT-volume account (which are publicly enumerable on-chain) can sustain this pattern indefinitely across concurrent connections, causing elevated DB CPU and I/O. The impact is griefing/availability degradation with no economic damage to network users — consistent with the stated scope classification.

### Likelihood Explanation

- **No authentication required.** Any external user can issue the request.
- **Trivial to craft.** The four-parameter combination (`token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B`) is documented in the OpenAPI spec and follows the `next` link automatically.
- **Fully automatable.** The attacker simply follows the `next` link in each response; no state management is needed.
- **Repeatable indefinitely.** There is no server-side counter, cooldown, or IP-based throttle on this endpoint.
- **Target amplification.** Choosing `limit=1` maximizes the number of pages (and thus DB round-trips) per dataset.

### Recommendation

1. **Add rate limiting to the REST API** for the `/accounts/:id/nfts` endpoint (and similar multi-union endpoints). Apply per-IP or per-account-ID throttling using existing infrastructure patterns.

2. **Simplify the query structure during pagination.** Once the lower bound of `token.id` advances past the boundary token (i.e., `gte:lastRow.token_id > A`), the lower subquery no longer needs the serial number constraint and the upper subquery may be eliminable. Detect mid-pagination advancement and reduce to a two-part or single query when the cursor has moved fully into the inner range.

3. **Enforce a minimum effective `limit`.** A `limit=1` combined with a wide range maximizes page count. Consider a minimum limit of 10 or 25 for range queries.

4. **Add a maximum page-count or cursor-depth guard** that returns `next: null` after a configurable number of pages from a single originating request pattern.

### Proof of Concept

**Step 1 — Initial request (produces 3-part UNION ALL):**
```
GET /api/v1/accounts/0.0.1000/nfts?order=asc&token.id=gte:1&token.id=lte:9999999&serialnumber=gte:1&serialnumber=lte:9999999&limit=1
```
Response includes `links.next` such as:
```
?order=asc&token.id=gte:1&token.id=lte:9999999&serialnumber=gt:5&limit=1
```
(where `token.id=lte:9999999` is preserved, `serialnumber=gte:1` is replaced by `gt:5`)

**Step 2 — Follow `next` link (still produces 3-part UNION ALL):**
```
GET /api/v1/accounts/0.0.1000/nfts?order=asc&token.id=gte:1&token.id=lte:9999999&serialnumber=gt:5&limit=1
```
`extractNftMultiUnionQuery` parses: primary.lower=gte:1, primary.upper=lte:9999999, secondary.lower=gt:5, secondary.upper=lte:9999999 → all three of `getLowerFilters`, `getInnerFilters`, `getUpperFilters` return non-empty arrays → three-part UNION ALL is built again.

**Step 3 — Automate:**
```python
import requests
url = "http://mirror-node/api/v1/accounts/0.0.1000/nfts"
params = "?order=asc&token.id=gte:1&token.id=lte:9999999&serialnumber=gte:1&serialnumber=lte:9999999&limit=1"
while params:
    r = requests.get(url + params)
    params = r.json()["links"]["next"]  # follow next link indefinitely
```

Every iteration fires a 3-subquery UNION ALL against the database with no server-side throttle.
