### Title
Unauthenticated High-Concurrency DoS via Unbounded Staking Rewards History Queries

### Summary
The `/accounts/:id/rewards` endpoint in `rest/routes/accountRoute.js` is publicly accessible with no authentication or rate limiting. An unprivileged attacker can send high-concurrency requests with the maximum allowed `limit` and no `timestamp` filter, causing `StakingRewardTransferService.getRewards()` to execute full-history scans on the `staking_reward_transfer` table for targeted accounts, exhausting DB connection pool and I/O capacity and degrading mirror node availability.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:16` routes `GET /:id/rewards` to `AccountController.listStakingRewardsByAccountId` with no middleware for authentication or rate limiting. [1](#0-0) 

`rest/controllers/accountController.js:170-203` — `listStakingRewardsByAccountId` accepts `limit`, `order`, and `timestamp` as optional filters. The `timestamp` filter is never required. [2](#0-1) 

`extractStakingRewardsQuery` (lines 110–162) sets `limit = defaultLimit` initially and overrides it with the user-supplied value. No server-side enforcement of a mandatory time window exists. [3](#0-2) 

`rest/service/stakingRewardTransferService.js:11-16` — The base SQL query is `SELECT ... FROM staking_reward_transfer WHERE account_id = $1` with no mandatory timestamp predicate appended unless the caller supplies one. [4](#0-3) 

`getRewards` (lines 18–22) executes the query directly via `super.getRows()` with no additional guard. [5](#0-4) 

**Root cause:** No rate limiting middleware exists in the REST layer (no rate-limit files found in the repository), no authentication is required, and no mandatory timestamp window is enforced. While `buildAndValidateFilters` caps the per-request row count via `maxLimit`, each request still triggers a full index scan on `staking_reward_transfer` for the target account. At high concurrency, this saturates the DB connection pool and disk I/O. [6](#0-5) 

**Why existing checks fail:** The `acceptedRewardsParameters` set (line 214) permits `limit`, `order`, and `timestamp` — all optional. The `maxLimit` cap bounds rows per response but does not bound the number of simultaneous queries or the aggregate DB load. [7](#0-6) 

### Impact Explanation
An attacker flooding the endpoint at high concurrency (e.g., hundreds of goroutines/threads) against accounts with large staking reward histories will cause the PostgreSQL connection pool to saturate, increasing query latency for all mirror node REST endpoints. This degrades or halts mirror node REST API availability — the primary interface for dApps, wallets, and explorers querying Hedera state. The mirror node is a critical read path; its unavailability constitutes a meaningful service disruption.

### Likelihood Explanation
No special privileges, API keys, or accounts are required. Any internet-accessible mirror node deployment is reachable. The attack is trivially scriptable with standard HTTP tooling (`ab`, `wrk`, `hey`, custom scripts). Accounts with long staking histories (e.g., long-running validator or staking accounts) are publicly enumerable via the `/accounts` endpoint. The attack is repeatable and stateless.

### Recommendation
1. **Enforce a mandatory timestamp window** in `extractStakingRewardsQuery`: require at least one `timestamp` filter (e.g., `gte` or `gt`) so every query is bounded to a time range, preventing full-history scans.
2. **Add rate limiting middleware** to the Express router (e.g., `express-rate-limit`) scoped to IP and/or account ID, applied before route handlers in `accountRoute.js`.
3. **Cap the DB connection pool** and set statement timeouts in the PostgreSQL client configuration to bound the blast radius of any single flood.
4. **Add a query timeout** in `BaseService.getRows()` so runaway queries are killed server-side.

### Proof of Concept

**Preconditions:**
- Mirror node REST API is publicly accessible (default deployment).
- Target account ID with a large staking reward history is identified via `GET /api/v1/accounts`.

**Steps:**
```bash
# Identify a high-reward account
ACCOUNT=$(curl -s https://<mirror-node>/api/v1/accounts?order=desc | jq -r '.accounts[0].account')

# Flood with max-limit, no timestamp filter, high concurrency
hey -n 10000 -c 200 \
  "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/rewards?limit=100&order=desc"
```

**Result:** DB connection pool exhaustion; subsequent requests to any REST endpoint return 503 or time out; mirror node logs show connection wait timeouts; DB CPU/IO metrics spike to saturation.

  extractStakingRewardsQuery(filters) {
    let limit = defaultLimit;
    let order = orderFilterValues.DESC;
    const timestampInValues = [];
    const conditions = [];
    const params = [];
    const startPosition = 3; // caller will later insert accountId at $1 and limit at $2

    for (const filter of filters) {
      if (isNil(filter)) {
        continue;
      }

      switch (filter.key) {
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        case filterKeys.TIMESTAMP:
          if (utils.opsMap.ne === filter.operator) {
            throw new InvalidArgumentError(`Not equals (ne) operator is not supported for ${filterKeys.TIMESTAMP}`);
          }
          this.updateConditionsAndParamsWithInValues(
            filter,
            timestampInValues,
            params,
            conditions,
            StakingRewardTransfer.getFullName(StakingRewardTransfer.CONSENSUS_TIMESTAMP),
            startPosition + params.length
          );
          break;
        default:
          break;
      }
    }

    this.updateQueryFiltersWithInValues(
      params,
      conditions,
      timestampInValues,
      StakingRewardTransfer.getFullName(StakingRewardTransfer.CONSENSUS_TIMESTAMP),
      startPosition + params.length
    );

    return {
      order,
      limit,
      conditions,
      params,
    };
  }

  listStakingRewardsByAccountId = async (req, res) => {
    const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
    const isValidAccount = await EntityService.isValidAccount(accountId);
    if (!isValidAccount) {
      throw new NotFoundError();
    }
    const filters = utils.buildAndValidateFilters(req.query, acceptedRewardsParameters);
    const query = this.extractStakingRewardsQuery(filters);
    // insert account id at $1, and limit (at $2)
    query.params.unshift(accountId, query.limit);
    const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(
      query.order,
      query.limit,
      query.conditions,
      query.params
    );
    const rewards = stakingRewardsTransfers.map((reward) => new StakingRewardTransferViewModel(reward));
    const response = {
      rewards,
      links: {
        next: null,
      },
    };

    if (response.rewards.length === query.limit) {
      const lastRow = last(response.rewards);
      const lastValue = {
        [filterKeys.TIMESTAMP]: lastRow.timestamp,
      };
      response.links.next = utils.getPaginationLink(req, false, lastValue, query.order);
    }

    res.locals[responseDataLabel] = response;
  };
