Looking at the `streamBlocks()` exception handling in `BlockNode.java` and the `doGet()` call in `BlockNodeSubscriber.java`.

The critical catch structure in `streamBlocks()`:

```java
} catch (BlockStreamException ex) {
    onError();
    throw ex;
} catch (Exception ex) {
    onError();
    throw new BlockStreamException(ex);
} finally { ... }
```

`catch (Exception ex)` does **not** catch `java.lang.Error` subclasses (`OutOfMemoryError`, `StackOverflowError`, etc.) — `Error` is a sibling of `Exception` under `Throwable`, not a subclass of it.

The `BlockAssembler` accumulates `List<List<BlockItem>>` in `pending` across multiple gRPC responses. The `maxBlockItems` check counts items (default 800,000), not bytes. The `maxInboundMessageSize` (default 36MB) limits each individual gRPC response, but the accumulation in `pending` is unbounded in bytes. ### Title
`Error`-Subclass Exceptions Bypass `onError()` in `BlockNode.streamBlocks()`, Keeping Malicious Node Permanently Active

### Summary
`BlockNode.streamBlocks()` catches only `Exception` and `BlockStreamException`, leaving `java.lang.Error` subclasses (e.g., `OutOfMemoryError`) uncaught by the error-accounting path. A block node operator can deliberately exhaust the mirror node's heap by streaming many large `BlockItemSet` responses within a single block, causing `OutOfMemoryError` to propagate without ever calling `onError()`. Because the error counter is never incremented, the node is never marked inactive and is re-selected on every subsequent polling cycle, creating a persistent denial-of-service against any block the attacker chooses to withhold.

### Finding Description

**Code path:**

`BlockNodeSubscriber.doGet()` (line 66) calls `node.streamBlocks(...)` with no surrounding `Throwable` catch.

Inside `BlockNode.streamBlocks()` (lines 121–171), the only error-accounting catch clauses are:
```java
catch (BlockStreamException ex) { onError(); throw ex; }
catch (Exception ex)            { onError(); throw new BlockStreamException(ex); }
```
`java.lang.Error` is not a subclass of `java.lang.Exception`; it is a direct subclass of `java.lang.Throwable`. Any `Error` thrown inside the `try` block skips both catch clauses, skips `onError()`, and propagates through the `finally` block (which only cancels the gRPC call).

**Root cause — memory accumulation without byte-budget:**

`BlockAssembler.append()` (lines 283–297) enforces only an *item count* limit (`maxBlockItems`, default 800 000). It does not enforce a *byte* limit on the accumulated `pending` list. The gRPC channel enforces `maxInboundMessageSize` (default 36 MB) per individual `SubscribeStreamResponse`, but the assembler accumulates responses across the entire block. A malicious node can send N responses each containing one `BlockItem` of ~36 MB. With a 4 GB JVM heap, `OutOfMemoryError` fires after roughly 111 responses — far below the 800 000-item threshold that would throw a `BlockStreamException`.

**Why existing checks are insufficient:**

| Check | What it limits | What it misses |
|---|---|---|
| `maxInboundMessageSize` (36 MB) | Single gRPC message | Cross-message accumulation in `pending` |
| `maxBlockItems` (800 000) | Item *count* | Item *byte size* |
| `catch (Exception ex)` | Checked + unchecked exceptions | `java.lang.Error` hierarchy |

**`onError()` never called → node stays active:**

`onError()` (lines 196–207) is the sole mechanism that increments `errors`, compares against `maxSubscribeAttempts`, sets `active = false`, and schedules a `readmitTime`. Without it, `active` remains `true` and `errors` stays at 0 (or is reset to 0 by the `errors.set(0)` on the last successful response before the attack response).

### Impact Explanation

A block node operator (no mirror-node credentials required) can target any specific block number:

1. Serve all preceding blocks normally (keeping `errors` reset to 0 each time).
2. For the targeted block, stream enough large `BlockItemSet` responses to exhaust heap.
3. `OutOfMemoryError` fires; `onError()` is skipped; the node remains active.
4. `AbstractBlockSource.get()` is called again on the next polling tick; `getNode()` selects the same node (still active, errors = 0); the attack repeats indefinitely.

The targeted block — and every transaction it contains — is never persisted to the mirror node database. Downstream API consumers, explorers, and compliance tools see those transactions as non-existent. This constitutes selective transaction suppression with no automatic recovery.

### Likelihood Explanation

**Preconditions:**
- Attacker controls or compromises one block node that the mirror node is configured to use (no mirror-node credentials needed).
- The mirror node's JVM heap is finite (always true).

**Feasibility:** The attack requires only a modified gRPC server that streams oversized `BlockItemSet` responses for a chosen block number. This is straightforward to implement. The default `maxStreamResponseSize` of 36 MB and `maxBlockItems` of 800 000 make the byte-budget gap wide enough to trigger OOM on any normally-provisioned JVM before the item-count guard fires.

**Repeatability:** Because `onError()` is never called, the node is never cooled down. The attack repeats on every polling cycle with no operator intervention required.

### Recommendation

1. **Catch `Throwable` instead of `Exception`** in `streamBlocks()`, or add an explicit `catch (Error err)` block that calls `onError()` before rethrowing:
   ```java
   } catch (BlockStreamException ex) {
       onError(); throw ex;
   } catch (Exception ex) {
       onError(); throw new BlockStreamException(ex);
   } catch (Error err) {          // ADD THIS
       onError(); throw err;
   }
   ```

2. **Add a byte-budget check in `BlockAssembler.append()`** alongside the item-count check, tracking total accumulated bytes and throwing `BlockStreamException` when a configurable `maxPendingBytes` threshold is exceeded. This prevents OOM from being reachable in the first place.

3. **Add a `maxPendingBytes` property to `StreamProperties`** (e.g., default 256 MB) and enforce it in `append()`.

### Proof of Concept

**Preconditions:**
- Mirror node configured with one block node endpoint pointing to the attacker's server.
- Attacker's server implements `BlockStreamSubscribeService.subscribeBlockStream`.

**Steps:**

1. For all block numbers except the target, the attacker's server responds normally (valid blocks).

2. For the target block number B, the server streams the following sequence:
   - One valid `BLOCK_ITEMS` response containing a `BLOCK_HEADER` item (starts the pending block).
   - Repeatedly send `BLOCK_ITEMS` responses, each containing one `BlockItem` of ~34 MB of arbitrary bytes, until the mirror node's heap is exhausted.
   - Never send `END_OF_BLOCK`.

3. The mirror node's `BlockAssembler` accumulates each response into `pending`. After ~120 responses (≈4 GB), `OutOfMemoryError` is thrown inside `grpcCall.read()` or during protobuf deserialization.

4. The `Error` propagates past both `catch` clauses in `streamBlocks()`. `onError()` is not called. `active` remains `true`. `errors` remains 0.

5. On the next polling tick, `getNode()` selects the same node. The attacker repeats step 2.

6. Block B is never assembled or verified. All transactions in block B are permanently absent from the mirror node's database.
