### Title
Accept-Encoding Quality-Factor Bypass in `validateAcceptEncodingHeader()` Allows Forced Uncompressed Large Responses

### Summary
`validateAcceptEncodingHeader()` in `OpcodesController.java` uses a naive substring check (`contains("gzip")`) to enforce that clients accept gzip compression. A client sending `Accept-Encoding: gzip;q=0` (which per RFC 7231 explicitly rejects gzip with quality factor 0) passes this check because the string still contains the substring "gzip". Spring Boot's embedded server compression layer correctly interprets `q=0` and does not compress the response, so the server sends the full, potentially multi-megabyte uncompressed opcode trace — exactly the scenario the validation was designed to prevent.

### Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`
**Function:** `validateAcceptEncodingHeader()`, line 76

```java
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

The check only tests for the substring `"gzip"`. It does not parse the `Accept-Encoding` header per RFC 7231 §5.3.4, which defines quality values (`q=0` through `q=1`). A `q=0` value means the encoding is explicitly **not acceptable**.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/contracts/results/{hash}/opcodes` with `Accept-Encoding: gzip;q=0`
2. `validateAcceptEncodingHeader()` evaluates `"gzip;q=0".toLowerCase().contains("gzip")` → `true` → no exception thrown
3. `throttleManager.throttleOpcodeRequest()` is called (rate-limited, but not blocked)
4. `opcodeService.processOpcodeCall(request)` executes the expensive EVM replay
5. Spring's compression layer correctly parses `q=0` and sends the response **uncompressed**
6. A full opcode trace (stack + memory + storage) is transmitted as raw JSON — potentially tens of megabytes per request

**Why the existing check fails:** The comment at line 71–73 explicitly states the purpose: "without compression this will result in big network latency." The substring check does not implement this intent. The test suite at `OpcodesControllerTest.java` line 507 only tests `{"deflate", "identity", "br", ""}` — it does not test `gzip;q=0`, confirming the gap.

### Impact Explanation
Each bypassed request forces the server to: (a) re-execute a full EVM transaction replay (CPU-intensive), and (b) serialize and transmit an uncompressed multi-MB response body. Bandwidth and heap pressure scale with response size. While the throttle manager limits per-second rate, the bypass is persistent and repeatable across the allowed rate window. Sustained use across multiple source IPs can exhaust outbound bandwidth and increase GC pressure on the JVM, degrading service for legitimate users.

### Likelihood Explanation
No authentication or privilege is required. The endpoint is publicly accessible. The bypass requires only crafting a single HTTP header value (`gzip;q=0`), which is trivial for any HTTP client. The attack is fully repeatable within the throttle window and requires no special knowledge beyond the public API documentation.

### Recommendation
Replace the substring check with proper RFC 7231 quality-value parsing. Use Spring's built-in `org.springframework.http.HttpHeaders` or a dedicated `Accept-Encoding` parser:

```java
import org.springframework.http.HttpHeaders;
import java.util.List;

private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...NOT_ACCEPTABLE...;
    }
    // Parse quality values; gzip must be present with q > 0
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip")) return false;
            for (int i = 1; i < parts.length; i++) {
                String p = parts[i].trim();
                if (p.startsWith("q=")) {
                    try { return Double.parseDouble(p.substring(2)) > 0; }
                    catch (NumberFormatException e) { return false; }
                }
            }
            return true; // no q= means q=1.0
        });
    if (!gzipAccepted) throw ...NOT_ACCEPTABLE...;
}
```

Also add a test case for `gzip;q=0` in `OpcodesControllerTest` alongside the existing `rejectsWhenAcceptEncodingIsNotGzip` parameterized test.

### Proof of Concept
```bash
# Replace {HASH} with any valid transaction hash
curl -v \
  -H "Accept-Encoding: gzip;q=0" \
  "https://<mirror-node-host>/api/v1/contracts/results/{HASH}/opcodes?stack=true&memory=true&storage=true"

# Expected (correct) behavior: HTTP 406 Not Acceptable
# Actual behavior:             HTTP 200 OK with full uncompressed JSON body
```

The response will be delivered uncompressed. Repeating at the allowed throttle rate with `stack=true&memory=true&storage=true` on complex contracts maximizes payload size per request.
