### Title
Unauthenticated DoS via Numeric Entity ID Bypass and Unbounded Concurrent Hook Queries

### Summary
The `getHooks()` endpoint in `HooksController` accepts any numeric `ownerId` (including `0.0.0`) without authentication or rate limiting. `EntityServiceImpl.lookup()` skips entity existence validation for numeric IDs, meaning every request unconditionally reaches the database. An attacker can flood the endpoint with concurrent requests using `limit=100` and a full `hook.id` range, exhausting the DB connection pool and degrading mirror node availability.

### Finding Description

**Code path:**

1. `HooksController.java:81-102` — `getHooks()` is a public `@GetMapping` with no authentication annotation. The only input constraints are `@Positive @Max(MAX_LIMIT)` on `limit` (max 100) and `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `hookId` (max 100 params). No rate-limiting annotation exists.

2. `EntityServiceImpl.java:31-37` — `lookup()` branches on the parameter type. For `EntityIdNumParameter`, it executes `Optional.of(p.id())` — it **never queries the database** to verify the entity exists. For alias/EVM address types it does query and throws `EntityNotFoundException` if absent. This asymmetry means any numeric ID, including `0.0.0` (encoded as `id=0`), is accepted and passed directly to the repository.

3. `HookServiceImpl.java:39-40` — With no `hookId` EQ filters, the code calls `hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), 0L, Long.MAX_VALUE, page)` — a full range scan with `owner_id = 0`, `hook_id BETWEEN 0 AND 9223372036854775807`, `LIMIT 100`.

4. `HookRepository.java:15` — Spring Data JPA translates this to a parameterized SQL query executed against the live database on every request.

**Root cause:** `EntityServiceImpl.lookup()` does not validate numeric entity existence before issuing the hook query. Combined with zero authentication and zero rate limiting, every inbound HTTP request unconditionally consumes a DB connection.

**Why existing checks fail:**
- `@Max(MAX_LIMIT)` caps rows returned, not query cost or concurrency.
- `@Size(max = 100)` on `hookId` limits parameter count, not request rate.
- No `@RateLimiter`, no Spring Security filter, no API gateway throttle is present in the codebase.

### Impact Explanation
An attacker sending N concurrent requests each triggers N simultaneous DB queries. With a typical connection pool of 10–50 connections, a few hundred concurrent requests saturate the pool, causing all subsequent requests (including legitimate ones) to queue or fail. If the mirror node is deployed as a horizontally scaled cluster sharing one DB, all instances are affected simultaneously. The mirror node becomes unavailable for hook and storage queries, degrading the read API for the network.

### Likelihood Explanation
No special privileges, credentials, or knowledge of valid entity IDs are required. The attacker only needs the public endpoint URL. The attack is trivially scriptable (`curl`/`ab`/`wrk`) and fully repeatable. The numeric `0.0.0` ID is the simplest possible input and requires no enumeration. The attack can be sustained indefinitely at low cost.

### Recommendation
1. **Validate numeric entity existence** in `EntityServiceImpl.lookup()` for `EntityIdNumParameter` — query the entity repository and throw `EntityNotFoundException` if absent, consistent with alias/EVM address handling.
2. **Add rate limiting** at the controller or filter layer (e.g., Resilience4j `@RateLimiter`, Bucket4j, or an API gateway policy) scoped per source IP.
3. **Add authentication/authorization** for hook endpoints if hooks are account-specific resources.
4. **Set a DB query timeout** on hook repository queries to bound worst-case connection hold time.

### Proof of Concept
```bash
# Send 500 concurrent requests with max limit and full hook.id range
# No credentials needed; 0.0.0 is accepted as a valid numeric entity ID
for i in $(seq 1 500); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.0/hooks?limit=100" &
done
wait
# Result: DB connection pool exhausted; legitimate requests return 500/timeout
```
