### Title
Per-Topic Independent Retry Storm: Unauthenticated gRPC Subscribers Can Amplify Redis Reconnection Load by N Distinct Topics During Network Partition

### Summary
`RedisTopicListener.subscribe()` creates a separate `retryWhen` loop for every distinct topic channel. Because the `container` Mono is `.cache()`-d at the connection level but `r.receive()` is re-invoked independently per topic on each retry, N distinct topic subscriptions produce N independent, unbounded retry loops against the shared `ReactiveRedisMessageListenerContainer` during any Redis connectivity failure. The gRPC `subscribeTopic` endpoint has no authentication, no per-IP connection limit, and no cap on the number of distinct topics a single client may subscribe to, making this amplification reachable by any unprivileged caller.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–79:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Duration interval = listenerProperties.getInterval();
    return container
            .flatMapMany(r -> r.receive(Collections.singletonList(topic), ...))  // line 72
            ...
            .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))  // line 78
            .share();
}
```

`getSharedListener()` (lines 59–62) calls `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))`, so each *distinct* topic key gets its own `Flux` with its own `retryWhen`. Multiple subscribers to the *same* topic share one Flux (`.share()`), but N distinct topics produce N independent retry loops.

**Root cause and failed assumption:**

The design assumes the `container` Mono's `.cache()` (line 55) is sufficient to share the Redis connection across all topics. This is true for the *initial connection* — the container is created once. However, the `retryWhen` at line 78 is scoped to the *per-topic* `r.receive()` call, not to the container. When `r.receive()` fails (e.g., due to a dropped Redis connection), each topic's retry fires independently, calling `container.flatMapMany(r -> r.receive(...))` again. Since `container` is already cached and resolves immediately to the same `ReactiveRedisMessageListenerContainer` instance, each of the N retry loops calls `r.receive()` on that container independently, generating N concurrent subscription re-establishment attempts against the same broken connection.

The retry is configured as `Retry.backoff(Long.MAX_VALUE, ...)` — unbounded retries — with a max backoff of only `interval × 4` (default: 2 seconds). With jitter, N=1000 topics produce a near-continuous stream of `r.receive()` calls.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (default, `NettyProperties` line 14 / `GrpcConfiguration` line 33) limits calls *per HTTP/2 connection*, but there is no limit on the number of connections from a single IP or in total. An attacker opens ⌈N/5⌉ connections to subscribe to N distinct topics.
- `checkTopicExists = true` (default, `GrpcProperties` line 19) requires topics to exist in the entity repository, but on a live network there are many valid topic IDs. If `checkTopicExists = false`, any numeric ID works.
- There is no rate limiter on the gRPC service (the `ThrottleConfiguration` found in the codebase is scoped to the `web3` module only, not `grpc`).
- No authentication is required on `ConsensusController.subscribeTopic()` (lines 43–53 of `ConsensusController.java`).

### Impact Explanation

During a Redis network partition (or any Redis error), an attacker holding subscriptions to N distinct topics causes N independent retry loops to fire against the `ReactiveRedisMessageListenerContainer`. Each loop calls `r.receive()` with its own backoff schedule, generating O(N) reconnection/subscription attempts per backoff window. This can:

1. Exhaust the Redis connection pool or overwhelm the Redis server's subscription command queue during recovery, delaying or preventing reconnection for legitimate traffic.
2. Cause the mirror node's gRPC service to remain degraded for the duration of the partition plus the time needed to drain the retry backlog.
3. Be sustained indefinitely (`Long.MAX_VALUE` retries) as long as the attacker maintains the gRPC streams.

### Likelihood Explanation

The attack requires only the ability to open gRPC connections to port 5600 (default, publicly documented) and issue `subscribeTopic` RPCs with distinct topic IDs — no credentials, no special privileges. The gRPC API is explicitly documented as a public interface. A single attacker machine can open hundreds of connections and subscribe to thousands of distinct topics. Network partitions between the mirror node and Redis are a realistic operational event (rolling restarts, cloud network blips), making the trigger condition non-contrived. The attack is repeatable and can be scripted trivially with `grpcurl` or any gRPC client library.

### Recommendation

1. **Cap the number of distinct active topic subscriptions** in `topicMessages` (e.g., a configurable `maxTopics` property). Reject new `subscribe()` calls when the cap is reached.
2. **Add a global retry coordinator** at the container level rather than per-topic: instead of each topic retrying independently, use a single shared reconnection signal (e.g., a `Sinks.Many` that all topic Fluxes listen to) so that N topics produce exactly one reconnection attempt.
3. **Rate-limit gRPC connections and/or `subscribeTopic` calls per source IP** using a filter or interceptor, analogous to the `ThrottleConfiguration` in the `web3` module.
4. **Bound the retry count** in `subscribe()` (replace `Long.MAX_VALUE` with a finite limit) and increase the max backoff to reduce the reconnection rate.

### Proof of Concept

```bash
# Open 200 connections, each subscribing to 5 distinct topics (1000 distinct topics total)
# Requires grpcurl and a running mirror node with Redis listener enabled

for i in $(seq 1 1000); do
  grpcurl -plaintext \
    -d "{\"topicID\": {\"topicNum\": $i}}" \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Trigger network partition: block Redis port on the mirror node host
# (e.g., iptables -A OUTPUT -p tcp --dport 6379 -j DROP)
# Observe: 1000 independent retryWhen loops begin firing against the Redis container,
# each calling r.receive() on the shared ReactiveRedisMessageListenerContainer.
# Redis reconnection is delayed/prevented; legitimate subscribers experience prolonged outage.
# Monitor with: watch -n1 'ss -tn | grep 6379 | wc -l'
```
