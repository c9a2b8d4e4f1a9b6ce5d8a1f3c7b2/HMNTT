### Title
Stale `CACHE_MANAGER_RECORD_FILE_INDEX` Cache Serves Pruned Block Metadata, Causing Silent Incorrect Historical EVM Execution

### Summary
`RecordFileRepository.findByIndex()` caches `RecordFile` objects with a hard 10-minute write-expiry TTL via `CACHE_MANAGER_RECORD_FILE_INDEX`. The importer's `RetentionJob` prunes `record_file` rows directly in the database with no corresponding cache eviction in the web3 module. Any unprivileged caller who triggered a cache population before pruning will receive the stale `RecordFile` for up to 10 minutes, causing `ContractCallService` to silently execute the EVM against a deleted block context instead of throwing `BlockNumberNotFoundException`.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` (line 28) delegates numeric block lookups to `recordFileRepository.findByIndex(block.number())`. [1](#0-0) 

`findByIndex` is annotated `@Cacheable(cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")`. Spring Cache unwraps `Optional<RecordFile>` before storage: a present `Optional` stores the raw `RecordFile` object; an empty `Optional` is treated as null and skipped. Therefore a successful lookup stores the `RecordFile` in the Caffeine cache. [2](#0-1) 

The cache manager is configured with `expireAfterWrite(10, TimeUnit.MINUTES)` and no eviction hook. [3](#0-2) 

The importer's `RetentionJob` calls `RecordFileRepository.prune(consensusTimestamp)` which issues `DELETE FROM RecordFile WHERE consensusEnd <= ?1` directly against the database. [4](#0-3) 

There is no `@CacheEvict` or any other cache-invalidation call anywhere in the pruning path. The web3 module's Caffeine cache is a separate in-process store; the importer's DB delete is invisible to it.

`ContractCallService.callContract()` sets the block supplier as:
```java
recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)
``` [5](#0-4) 

If the cache returns a stale `Optional.of(recordFile)`, `orElseThrow` does not fire. The EVM proceeds using the stale `RecordFile.consensusEnd` as the historical timestamp for all state queries (`ContractCallContext.getTimestamp()`), while the underlying transaction and state rows at that timestamp have already been deleted. [6](#0-5) 

**Root cause / failed assumption:** The cache design assumes `record_file` rows are immutable append-only data. This holds when retention is disabled (the default), but breaks when `hiero.mirror.importer.retention.enabled=true` is set, which is a documented, supported production configuration. [7](#0-6) 

### Impact Explanation

When the stale `RecordFile` is used as the historical block context, all EVM state reads (account balances, contract storage, bytecode) are performed at the pruned block's `consensusEnd` timestamp. Because the associated rows in `transaction`, `contract_state`, `entity`, etc. have also been pruned, those queries return empty/null results. The EVM therefore executes as if accounts have zero balance, contracts do not exist, and storage slots are unset — returning silently incorrect results rather than a proper "block not found" error. Off-chain systems (indexers, bridges, dApps) that rely on `eth_call` or `eth_estimateGas` against historical blocks can be fed fabricated state, leading to incorrect decisions. Severity is Medium: no direct on-chain fund movement is possible through the mirror node's read-only EVM, but the integrity of historical query results is broken.

### Likelihood Explanation

Preconditions:
1. Operator has enabled retention (`hiero.mirror.importer.retention.enabled=true`) — a documented, cost-saving production practice.
2. The attacker (or any prior caller) has queried the target block number at least once before it was pruned, populating the cache.
3. The attacker re-queries the same block number within the 10-minute TTL window after pruning.

Steps 2 and 3 require no privilege. Any public `eth_call` or `eth_estimateGas` request with a numeric `block` parameter satisfies them. The retention job runs on a configurable schedule (default: daily), so the 10-minute window is predictable once an attacker monitors the earliest available block number. Repeatability is high: every pruning cycle creates a new 10-minute window for every pruned block index that was previously cached.

### Recommendation

1. **Add `@CacheEvict` to the pruning path.** In the web3 module's `RecordFileRepository`, add a `@CacheEvict(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX)` method that is called (or overridden) when a `RecordFile` is deleted. Alternatively, expose a cache-eviction service method that the importer calls after each pruning batch.

2. **Switch to `expireAfterAccess` with a shorter TTL for index-based lookups**, or add a post-prune cache flush event (e.g., via Spring's `ApplicationEvent`) that the web3 cache manager listens to.

3. **Validate the cached `RecordFile` against the current earliest available block** before using it as a historical context: if `recordFile.getIndex() < findEarliest().getIndex()`, evict and return empty.

### Proof of Concept

```
# Preconditions: retention enabled, retention.period short enough to prune block N

# Step 1 – populate the cache for block N
curl -X POST http://mirror-node-web3/api/v1/contracts/call \
  -d '{"block":"<N>","to":"0x...","data":"0x..."}'
# → cache entry for index=N written, TTL starts (10 min)

# Step 2 – retention job runs (or trigger manually in test env)
# DELETE FROM record_file WHERE consensus_end <= <consensusEnd of block N>
# Also deletes associated transaction/entity/contract_state rows

# Step 3 – within 10 minutes, re-query the same block
curl -X POST http://mirror-node-web3/api/v1/contracts/call \
  -d '{"block":"<N>","to":"0x...","data":"0x..."}'
# → findByIndex(N) hits cache, returns stale RecordFile (no DB query)
# → orElseThrow() does NOT throw (Optional is present)
# → EVM executes at pruned consensusEnd timestamp
# → all state reads return empty/null (data deleted)
# → response: contract returns 0 balances / reverts as if contract doesn't exist
#    instead of a proper BlockNumberNotFoundException / 400 error
```
