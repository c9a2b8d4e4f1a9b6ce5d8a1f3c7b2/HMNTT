### Title
Unauthenticated `/block/transaction` Endpoint Enables DB CPU Exhaustion via Unbounded Correlated Subquery Flooding

### Summary
The `blockAPIService.BlockTransaction()` handler in `rosetta/app/services/block_service.go` accepts unauthenticated POST requests and, for each request, executes `selectTransactionsByHashInTimestampRange` — a SQL query containing two correlated subqueries against `crypto_transfer` and `staking_reward_transfer`. No rate limiting, request throttling, or authentication exists anywhere in the middleware stack, allowing an unprivileged attacker to flood the endpoint with valid block identifiers and varying transaction hashes, driving sustained DB CPU load well above the 30% baseline threshold.

### Finding Description

**Code path:**

`rosetta/main.go:217-219` — the middleware chain is only `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. No auth, no rate limiting. [1](#0-0) 

`rosetta/app/services/block_service.go:77-101` — `BlockTransaction()` calls `FindByIdentifier()` (block lookup) then unconditionally calls `FindByHashInBlock()` (transaction SQL) for every request. [2](#0-1) 

`rosetta/app/persistence/transaction.go:32-58` — the SQL constant `selectTransactionsByHashInTimestampRange` is `selectTransactionsInTimestampRange + andTransactionHashFilter`. The outer query scans `transaction t WHERE consensus_timestamp >= @start AND consensus_timestamp <= @end AND transaction_hash = @hash`. For every row that matches, two correlated subqueries execute: one `json_agg` over `crypto_transfer` and one over `staking_reward_transfer`, both correlated on `consensus_timestamp = t.consensus_timestamp`. [3](#0-2) 

`rosetta/app/persistence/transaction.go:173-208` — `FindByHashInBlock()` executes this raw query with no caching, no deduplication, and no concurrency guard. [4](#0-3) 

**Root cause:** The failed assumption is that `/block/transaction` will only be called by cooperative clients at low rates. In reality:
- Block identifiers (index + hash) are freely discoverable via `/block` or `/network/status`.
- Valid transaction hashes within a block are returned by `/block` in `OtherTransactions`.
- With valid hashes, the correlated subqueries execute for every matched row, multiplying DB work per request.
- With invalid/random hashes, the outer timestamp-range scan still executes on every request; if `transaction_hash` lacks a covering index over `(consensus_timestamp, transaction_hash)`, this becomes a partial table scan.

**Why existing mitigations fail:**

`statementTimeout` (`rosetta/app/config/types.go:45`) only cancels individual long-running queries; it does not limit request rate or concurrent query count. [5](#0-4) 

`Pool.MaxOpenConnections` (`rosetta/app/config/types.go:77-81`) caps DB connections but does not throttle HTTP requests — excess requests queue in the Go HTTP server and drain the pool continuously. [6](#0-5) 

No rate-limiting middleware exists anywhere in the Rosetta codebase (confirmed by exhaustive grep across all `rosetta/**/*.go`).

### Impact Explanation
Each `/block/transaction` request with a valid transaction hash causes the DB to execute the outer timestamp-range scan plus two correlated `json_agg` subqueries. At high request rates (hundreds/second, easily achievable from a single machine), the aggregate DB CPU from concurrent correlated subquery executions can exceed 30% of the 24-hour baseline. This degrades query performance for all other mirror-node consumers sharing the same PostgreSQL instance (REST API, gRPC, etc.) and can cause cascading timeouts.

### Likelihood Explanation
The attack requires zero privileges. Block identifiers and transaction hashes are public data returned by other Rosetta endpoints. The attack is trivially scriptable with `curl` or any HTTP load tool (e.g., `wrk`, `hey`). It is repeatable indefinitely and requires no special network position. The Rosetta API is intended to be internet-facing for blockchain data consumers, making external access the default deployment posture.

### Recommendation
1. **Add per-IP or global rate limiting** in the HTTP middleware layer (e.g., `golang.org/x/time/rate` or a reverse-proxy rule) before requests reach the service handlers.
2. **Add a DB-level connection/query concurrency limit** specific to the Rosetta DB user (PostgreSQL `CONNECTION LIMIT` on the role).
3. **Ensure a composite index** on `transaction(transaction_hash, consensus_timestamp)` exists so the outer query is an index seek rather than a range scan, reducing per-query cost.
4. **Consider caching** `FindByHashInBlock` results for recently-seen `(hash, consensusStart, consensusEnd)` tuples to collapse repeated identical queries.

### Proof of Concept

```bash
# Step 1: Obtain a valid block identifier
BLOCK=$(curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{}}')

INDEX=$(echo $BLOCK | jq '.block.block_identifier.index')
HASH=$(echo $BLOCK | jq -r '.block.block_identifier.hash')

# Step 2: Obtain a valid transaction hash from OtherTransactions (or use random hex)
TX_HASH=$(echo $BLOCK | jq -r '.other_transactions[0].hash // "0xdeadbeefdeadbeefdeadbeefdeadbeef"')

# Step 3: Flood /block/transaction with the valid block ID and varying hashes
for i in $(seq 1 10000); do
  RAND_HASH=$(openssl rand -hex 48)
  curl -s -X POST http://<rosetta-host>/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$INDEX,\"hash\":\"$HASH\"},
         \"transaction_identifier\":{\"hash\":\"0x$RAND_HASH\"}}" &
done
wait

# Observe DB CPU via pg_stat_activity or monitoring dashboard exceeding 30% of 24h baseline
```

func (s *blockAPIService) BlockTransaction(
	ctx context.Context,
	request *rTypes.BlockTransactionRequest,
) (*rTypes.BlockTransactionResponse, *rTypes.Error) {
	h := tools.SafeRemoveHexPrefix(request.BlockIdentifier.Hash)
	block, err := s.FindByIdentifier(ctx, request.BlockIdentifier.Index, h)
	if err != nil {
		return nil, err
	}

	transaction, err := s.FindByHashInBlock(
		ctx,
		request.TransactionIdentifier.Hash,
		block.ConsensusStartNanos,
		block.ConsensusEndNanos,
	)
	if err != nil {
		return nil, err
	}

	if err = s.updateOperationAccountAlias(ctx, transaction); err != nil {
		return nil, err
	}

	return &rTypes.BlockTransactionResponse{Transaction: transaction.ToRosetta()}, nil

func (tr *transactionRepository) FindByHashInBlock(
	ctx context.Context,
	hashStr string,
	consensusStart int64,
	consensusEnd int64,
) (*types.Transaction, *rTypes.Error) {
	var transactions []*transaction
	transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
	if err != nil {
		return nil, hErrors.ErrInvalidTransactionIdentifier
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	if err = db.Raw(
		selectTransactionsByHashInTimestampRange,
		sql.Named("hash", transactionHash),
		sql.Named("start", consensusStart),
		sql.Named("end", consensusEnd),
	).Find(&transactions).Error; err != nil {
		log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
		return nil, hErrors.ErrDatabaseError
	}

	if len(transactions) == 0 {
		return nil, hErrors.ErrTransactionNotFound
	}

	transaction, rErr := tr.constructTransaction(transactions)
	if rErr != nil {
		return nil, rErr
	}

	return transaction, nil
}
