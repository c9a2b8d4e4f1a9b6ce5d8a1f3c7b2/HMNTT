### Title
Cache Key Fragmentation via Case-Sensitive `req.originalUrl` Hashing in `cacheKeyGenerator()`

### Summary
`cacheKeyGenerator()` in `rest/middleware/responseCacheHandler.js` hashes `req.originalUrl` verbatim with MD5, producing distinct Redis keys for semantically identical requests that differ only in URL or query-parameter casing. Because Express routing is case-insensitive by default and `requestQueryParser` normalises query-key case in `req.query` (but never in `req.originalUrl`), the same backend data is stored multiple times under different keys. No existing middleware rejects or normalises the raw URL before the cache key is computed.

### Finding Description

**Exact code path**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, line 151-153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

`req.originalUrl` is the raw, unmodified URL string as received from the TCP socket. It is never lowercased or otherwise normalised before being hashed.

**Root cause**

Two independent normalisation steps exist downstream of the cache key but are never applied to the cache key itself:

1. `requestQueryParser` (`rest/middleware/requestHandler.js` lines 38-68) lowercases all query-parameter keys and canonicalises selected values (`order`, `result`). This affects `req.query` only; `req.originalUrl` is untouched.
2. Express's default `case sensitive routing = false` makes `/api/v1/Transactions` and `/api/v1/transactions` resolve to the same route handler. Again, `req.originalUrl` is never rewritten.

The code comment at line 149 explicitly acknowledges the gap:
```
*   path?query - In the future, this will utilize Edwin's request normalizer (9113).
```

**Why the OpenAPI validator does not block this**

The validator is configured with `validateRequests: false` (`rest/middleware/openapiHandler.js` line 155):

```js
OpenApiValidator.middleware({
  apiSpec: ...,
  ignoreUndocumented: true,
  validateRequests: false,   // ← no request-level rejection
  validateResponses,
})
```

It validates responses only (and only in test environments). It never rejects a request with an uppercase path or uppercase query-parameter key. Additionally, the validator is disabled by default in production (`hiero.mirror.rest.openapi.validation.enabled = false`, docs line 577), so it is absent entirely in the default deployment.

**Exploit flow**

1. Attacker sends `GET /api/v1/Transactions?transactionType=CRYPTOTRANSFER` → `cacheKeyGenerator` produces key A (MD5 of the mixed-case URL). Cache miss; the route handler executes and the response is stored under key A.
2. A legitimate user sends `GET /api/v1/transactions?transactiontype=CRYPTOTRANSFER` → key B (MD5 of the lowercase URL). Cache miss again; the same backend data is fetched and stored under key B.
3. Repeat with arbitrary case permutations to multiply Redis entries for the same logical resource.

The same fragmentation applies to query-parameter key case alone (path lowercase, key mixed-case), because `requestQueryParser` normalises `req.query` but leaves `req.originalUrl` intact.

### Impact Explanation

- **Redis memory exhaustion**: An attacker can generate O(2^n) distinct cache keys for a URL with n case-variable characters, each consuming Redis memory for the full serialised response body (potentially compressed). With `hiero.mirror.rest.redis.maxMemory = 250 MB` and `allkeys-lfu` eviction, flooding with fragmented keys evicts legitimate cached entries, forcing repeated database queries and degrading API performance.
- **Cache bypass**: Any client can always bypass the cache by trivially altering the case of one character, defeating the caching layer entirely and increasing database load.
- **Stale/inconsistent data window**: Two cache entries for the same resource can have different TTLs (written at different times). Clients may receive responses from different points in time for what they believe is the same query.

### Likelihood Explanation

- **No authentication required**: Any unprivileged HTTP client can send a request with a mixed-case URL.
- **Trivially repeatable**: A single line of shell script (`curl /api/v1/Transactions`) is sufficient to trigger a distinct cache entry.
- **Cache must be enabled**: `hiero.mirror.rest.cache.response.enabled` defaults to `false`; the vulnerability is only active when an operator explicitly enables the response cache. However, `hiero.mirror.rest.redis.enabled` defaults to `true`, so enabling the response cache (a documented operational choice) immediately exposes the issue.
- **No rate-limiting on cache-key generation**: The cache check runs before any route-level throttling, so the attacker can generate keys at full request throughput.

### Recommendation

Normalise `req.originalUrl` (or a derived canonical form) before hashing in `cacheKeyGenerator()`. Specifically:

1. Lowercase the path component.
2. Lowercase all query-parameter keys (consistent with `requestQueryParser`).
3. Sort query parameters alphabetically (the planned "request normalizer" referenced in issue 9113 should be used here once available).

A minimal interim fix:

```js
const cacheKeyGenerator = (req) => {
  const url = new URL(req.originalUrl, 'http://x');
  const path = url.pathname.toLowerCase();
  const params = [...url.searchParams.entries()]
    .map(([k, v]) => [k.toLowerCase(), v])
    .sort(([a], [b]) => a.localeCompare(b));
  const canonical = path + '?' + new URLSearchParams(params).toString();
  return crypto.createHash('md5').update(canonical).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

Additionally, set `validateRequests: true` in the OpenAPI validator (or at minimum enforce it in production) so that malformed paths are rejected before reaching the cache layer.

### Proof of Concept

Prerequisites: mirror-node REST API running with `cache.response.enabled=true` and `redis.enabled=true`.

```bash
# Step 1 – prime the cache with a lowercase URL
curl -s "http://localhost:5551/api/v1/transactions?transactiontype=CRYPTOTRANSFER&limit=1" > /dev/null

# Step 2 – send the same request with mixed-case path → different cache key, cache miss, DB hit
curl -s "http://localhost:5551/api/v1/Transactions?transactiontype=CRYPTOTRANSFER&limit=1" > /dev/null

# Step 3 – send the same request with mixed-case query key → different cache key again
curl -s "http://localhost:5551/api/v1/transactions?TransactionType=CRYPTOTRANSFER&limit=1" > /dev/null

# Verify three distinct Redis keys exist for the same logical resource
redis-cli KEYS "*-v1" | wc -l   # expect 3 instead of 1
```

Each variant produces a unique MD5 hash from `req.originalUrl`, confirmed by the verbatim hash in `cacheKeyGenerator()` at [1](#0-0) , with no normalisation applied before hashing. The `requestQueryParser` lowercases keys only in `req.query` [2](#0-1) , leaving `req.originalUrl` unchanged. The OpenAPI validator is configured with `validateRequests: false` [3](#0-2)  and is disabled by default in production [4](#0-3) , providing no barrier. The cache check handler runs unconditionally for all requests when caching is enabled [5](#0-4) .
