### Title
Unbounded Database Query via Safety Check Path in `missingMessages()` — Unprivileged DoS

### Summary
Any unprivileged gRPC subscriber can trigger an unthrottled, limitless database query by subscribing to a topic with no `limit` (defaulting to 0) and ensuring no messages arrive within 1 second. The safety check fires unconditionally after 1 second, calling `topicMessageRetriever.retrieve(gapFilter, false)` where `gapFilter` inherits the original filter's `limit = 0` (no limit) and `throttled = false`, issuing a full table scan from `startTime` with no upper bound.

### Finding Description

**Exact code path:**

In `TopicMessageServiceImpl.java`, `subscribeTopic()` unconditionally schedules a safety check: [1](#0-0) 

After 1 second, if `!topicContext.isComplete()`, it calls `missingMessages(topicContext, null)`.

`isComplete()` returns `false` whenever `endTime == null`: [2](#0-1) 

Inside `missingMessages()`, the `current == null` branch (safety check path) builds `gapFilter` by copying the original filter and only overriding `startTime`: [3](#0-2) 

**Root cause:** The `gapFilter` is constructed with `.toBuilder().startTime(startTime).build()` — it does **not** set a `limit`. It inherits the original filter's `limit`, which defaults to `0` (meaning no limit per `hasLimit()`): [4](#0-3) 

The `@Min(0)` constraint explicitly permits `limit = 0`: [5](#0-4) 

The resulting call `topicMessageRetriever.retrieve(gapFilter, false)` passes `throttled = false`, bypassing any rate-limiting in the retriever: [6](#0-5) 

**Why existing checks fail:**
- Bean validation (`@Min(0)`) explicitly allows `limit = 0`, which means "no limit."
- The `isComplete()` guard only prevents the safety check if `endTime` is set and has passed — a subscriber with no `endTime` is never considered complete.
- The `safetyCheck` Flux fires once per subscription, but with many concurrent subscribers this multiplies.

### Impact Explanation

Each subscription with `limit = 0` and no `endTime` that receives no messages within 1 second issues one unbounded, unthrottled `SELECT` against the topic message table, scanning all rows from `startTime` forward with no `LIMIT` clause. On a busy topic with millions of messages, this is a full table scan. An attacker opening many concurrent subscriptions to a quiet topic (or a topic with a `startTime` far in the past) can exhaust database connection pool resources and CPU, causing a denial of service for all mirror node consumers. No authentication or special privileges are required — the gRPC `subscribeTopic` endpoint is publicly accessible.

### Likelihood Explanation

The attack requires only a valid `topicId` (publicly enumerable from the network) and a standard gRPC client. The attacker simply subscribes with no `limit` and no `endTime` to a topic that has no recent messages. The 1-second delay is trivially satisfied by targeting any low-traffic or empty topic. The attack is fully repeatable and scriptable, and each new subscription independently triggers the safety check. No credentials, tokens, or elevated permissions are needed.

### Recommendation

1. **Cap the safety-check query:** In the `current == null` branch of `missingMessages()`, explicitly set a reasonable `limit` on `gapFilter` (e.g., the configured page size from `RetrieverProperties`) regardless of the original filter's limit:
   ```java
   var gapFilter = topicContext.getFilter().toBuilder()
       .startTime(startTime)
       .limit(retrieverProperties.getMaxPageSize())  // enforce a cap
       .build();
   ```
2. **Enforce throttling:** Pass `throttled = true` for the safety-check retrieval, or apply a separate rate limit per subscriber for gap-recovery queries.
3. **Rate-limit safety check triggers per subscriber:** Track whether the safety check has already fired for a given `TopicContext` and suppress repeated triggers.
4. **Validate that `limit = 0` subscriptions are bounded by `endTime`:** Reject or cap subscriptions that have neither a `limit` nor an `endTime`.

### Proof of Concept

**Preconditions:**
- Mirror node gRPC endpoint is publicly reachable.
- A valid `topicId` exists that has no messages in the near future (or any topic with `startTime` set to "now").

**Steps:**
1. Open N concurrent gRPC streams calling `subscribeTopic` with:
   - `topicId` = any valid topic
   - `startTime` = current time
   - `limit` = 0 (omitted / default)
   - `endTime` = omitted
2. Ensure no messages are published to the topic (or target a naturally quiet topic).
3. After exactly 1 second, each stream's `safetyCheck` Mono fires.
4. Each stream calls `missingMessages(topicContext, null)` → `topicMessageRetriever.retrieve(gapFilter, false)` with `gapFilter.limit = 0` and `throttled = false`.
5. N unbounded, unthrottled database queries execute simultaneously.
6. **Result:** Database CPU/connection pool exhaustion; legitimate queries time out; mirror node becomes unresponsive.
