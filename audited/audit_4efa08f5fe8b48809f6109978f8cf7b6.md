### Title
Unprivileged `internal=true` Parameter Bypasses `transaction_nonce = 0` Filter in `GET /contracts/:contractId/results`

### Summary
Any unauthenticated user can append `?internal=true` to a `GET /contracts/:contractId/results` request. This causes `extractContractResultsByIdQuery()` to omit the `cr.transaction_nonce = 0` WHERE clause, exposing all internal contract call results (nonce > 0) to the query scan. No authentication or privilege check gates the `internal` parameter, making this trivially exploitable by any external caller.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `extractContractResultsByIdQuery` (lines 415–542) processes the `INTERNAL` filter key:

```js
case filterKeys.INTERNAL:
  internal = filter.value;   // line 479 — set to true from user input
  break;
``` [1](#0-0) 

Then at lines 531–533, the nonce guard is conditioned on this flag:

```js
if (!internal) {
  conditions.push(`${ContractResult.getFullName(ContractResult.TRANSACTION_NONCE)} = 0`);
}
``` [2](#0-1) 

The handler `getContractResultsById` (lines 856–892) calls `extractContractResultsByIdQuery` with filters built directly from `req.query` via `extractContractIdAndFiltersFromValidatedRequest`, which uses `utils.buildAndValidateFilters` against `acceptedContractResultsParameters`: [3](#0-2) 

The route is publicly registered with no auth middleware:

```js
router.getExt('/:contractId/results', ContractController.getContractResultsById);
``` [4](#0-3) 

**Root cause:** `filterKeys.INTERNAL` is handled inside `extractContractResultsByIdQuery` and is included in `acceptedContractResultsParameters` (evidenced by the live switch-case handling it). There is no privilege check, role check, or header validation before `internal` is consumed. The `transaction_nonce = 0` guard — which restricts results to top-level transactions only — is entirely controlled by this user-supplied boolean.

**Why existing checks fail:** `utils.buildAndValidateFilters` validates format/type of the parameter value (boolean), not the caller's identity or privilege level. Passing `internal=true` is a syntactically valid request that passes all validation and reaches the nonce-filter branch.

### Impact Explanation

For a contract with a high ratio of internal calls (nonce > 0) relative to top-level calls (nonce = 0), the default query (`transaction_nonce = 0`) quickly satisfies the LIMIT by scanning only top-level rows. With `internal=true`, the database must evaluate all rows for the contract regardless of nonce, potentially scanning orders of magnitude more rows before the LIMIT is satisfied — or returning a full `defaultLimit` of internal results that were previously invisible. On a busy contract (e.g., a DeFi router), internal results can outnumber top-level results by 10–100×, making the scan cost substantially higher. Repeated requests targeting such contracts can sustain elevated DB CPU and I/O, meeting the ≥30% resource consumption increase threshold.

### Likelihood Explanation

The exploit requires zero privileges, zero authentication, and zero special knowledge beyond knowing a contract ID (which is public on-chain). The attacker needs only to append `?internal=true` to a standard REST call. It is trivially scriptable, repeatable at high frequency, and requires no account or API key. Any public mirror node deployment is exposed.

### Recommendation

1. **Remove `filterKeys.INTERNAL` from `acceptedContractResultsParameters`** so that `buildAndValidateFilters` rejects the parameter before it reaches `extractContractResultsByIdQuery`. This is the minimal fix.
2. If `internal=true` is a legitimate feature, gate it behind an authenticated/privileged role check (e.g., an operator-only API key or a separate internal-only endpoint) before allowing the nonce filter to be bypassed.
3. Alternatively, enforce a stricter rate limit or cost-based throttle on the `/:contractId/results` endpoint to limit the blast radius of repeated large scans.

### Proof of Concept

**Preconditions:** A mirror node is publicly accessible. A contract ID with many internal calls (nonce > 0) is known (e.g., any popular DeFi contract).

**Steps:**

```
# Default request — only top-level results (nonce=0), small scan
GET /api/v1/contracts/0.0.1234/results

# Exploit request — all results including internal calls, large scan
GET /api/v1/contracts/0.0.1234/results?internal=true
```

**Trigger:** Send the second request repeatedly (e.g., 10–50 req/s) targeting a contract with a high internal-call volume.

**Result:** The `cr.transaction_nonce = 0` condition is absent from the generated SQL. The database scans all `contract_result` rows for the given `contract_id` without the nonce filter, increasing I/O and CPU consumption on the mirror node's database backend. Sustained at scale, this causes measurable resource exhaustion without any privileged access.
