### Title
Hardcoded `sslmode=disable` in `GetDsn()` Enables MitM Row-Reordering of `selectTransactionsInTimestampRangeOrdered` Results

### Summary
`GetDsn()` in `rosetta/app/config/types.go` unconditionally appends `sslmode=disable` to the PostgreSQL DSN, removing all TLS encryption and wire-level integrity protection from the Rosetta-to-database connection. An attacker on the same L2 network segment can ARP-spoof the connection, intercept the unencrypted PostgreSQL wire protocol, and reorder the `DataRow` frames returned by `selectTransactionsInTimestampRangeOrdered` before they reach the Rosetta process. Because the application performs no client-side ordering verification, the `/block` endpoint will present transactions in the attacker-controlled order rather than true consensus-timestamp order.

### Finding Description
**Code locations:**

- `rosetta/app/config/types.go`, `GetDsn()`, lines 49–58: `sslmode=disable` is hardcoded and non-configurable.
- `rosetta/app/persistence/transaction.go`, line 30: `orderByConsensusTimestamp = " order by consensus_timestamp"`.
- Line 59: `selectTransactionsInTimestampRangeOrdered = selectTransactionsInTimestampRange + orderByConsensusTimestamp`.
- Lines 126–134: `FindBetween()` issues the ordered query and appends results directly into `transactions` with no post-fetch ordering assertion.

**Root cause:** `sslmode=disable` disables both TLS encryption and the PostgreSQL client-side certificate/MAC verification. The PostgreSQL wire protocol (v3) sends each result row as an individually framed `DataRow` message (type byte `'D'` + 4-byte length + column data). Without TLS, these frames are plaintext TCP segments with no HMAC or sequence integrity. The `ORDER BY consensus_timestamp` clause is enforced only by the server before transmission; once the bytes leave the server NIC, nothing prevents an in-path attacker from reordering them.

**Exploit flow:**
1. Attacker performs ARP cache poisoning on the L2 segment shared by the Rosetta server and the PostgreSQL host, inserting themselves as the MitM.
2. Attacker forwards all traffic normally until a `Query` or `Bind`/`Execute` message containing `selectTransactionsInTimestampRangeOrdered` is observed.
3. Attacker buffers the stream of `DataRow` messages from the server response (each row is self-delimiting via the 4-byte length field).
4. Attacker reorders the buffered `DataRow` frames (e.g., swapping two transactions with adjacent `consensus_timestamp` values) and forwards the modified byte stream to the Rosetta client.
5. The Rosetta client's GORM layer deserializes the rows in the received order; `FindBetween()` builds the `transactions` slice in that order and returns it to the `/block` handler.

**Why existing checks fail:** The `ORDER BY` clause is a server-side instruction; it has no effect once the wire is compromised. `FindBetween()` performs no post-fetch sort or timestamp-monotonicity assertion on the returned slice. The `batchSize` pagination loop at lines 124–147 uses `transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1` as the next `start`, which an attacker can also manipulate to skip or duplicate ranges.

### Impact Explanation
The Rosetta `/block` endpoint is the canonical source of block-level transaction ordering for downstream consumers (exchanges, indexers, auditors). Presenting transactions in a false order constitutes a transaction history reorganization: a consumer crediting account A before debiting account B (or vice versa) based on Rosetta data would derive an incorrect ledger state. Severity is **High** for integrity; confidentiality loss (DB credentials, transaction data) is a co-equal impact of the same `sslmode=disable` root cause.

### Likelihood Explanation
The precondition—same L2 network segment—is realistic in cloud VPC deployments where the Rosetta pod and the PostgreSQL instance share a subnet (common in Kubernetes or EC2 deployments without strict network segmentation). ARP spoofing requires no privileges on the target systems; any host on the segment can execute it with standard tools (`arpspoof`, `ettercap`). The PostgreSQL wire protocol v3 is fully documented and trivially parseable; open-source libraries exist for real-time stream manipulation. The attack is repeatable on every query. The only mitigating factor is that some deployments co-locate Rosetta and PostgreSQL on the same host (loopback), but the configuration accepts any `Host` value and the code provides no enforcement of local-only binding.

### Recommendation
1. **Immediate:** Change `GetDsn()` to use `sslmode=verify-full` (or at minimum `sslmode=require`) and expose `SslMode`, `SslCert`, `SslKey`, and `SslRootCert` as configurable `Db` struct fields.
2. **Defense-in-depth:** Add a post-fetch monotonicity check in `FindBetween()` asserting that each successive `ConsensusTimestamp` is ≥ the previous one, returning `ErrDatabaseError` on violation.
3. **Deployment:** Enforce network policy (Kubernetes `NetworkPolicy` or security groups) to restrict PostgreSQL port 5432 to the Rosetta pod only.

### Proof of Concept
```
# 1. ARP-poison the segment (attacker host: 10.0.0.5, rosetta: 10.0.0.10, postgres: 10.0.0.20)
sudo arpspoof -i eth0 -t 10.0.0.10 10.0.0.20 &
sudo arpspoof -i eth0 -t 10.0.0.20 10.0.0.10 &
sudo sysctl net.ipv4.ip_forward=1

# 2. Capture and modify the PostgreSQL stream on port 5432
# Use a Python script with scapy or mitmproxy's TCP proxy to:
#   a. Detect the Query response containing DataRow ('D') frames
#   b. Buffer all DataRow frames until ReadyForQuery ('Z') is received
#   c. Swap two DataRow frames (e.g., index 0 and index 1)
#   d. Forward the reordered byte stream to 10.0.0.10:ephemeral

# 3. Call the Rosetta /block endpoint
curl -X POST http://10.0.0.10:8082/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{...},"block_identifier":{"index":N}}'

# 4. Observe: transactions[0].consensus_timestamp > transactions[1].consensus_timestamp
# confirming the reordering was accepted by the Rosetta server.
```
