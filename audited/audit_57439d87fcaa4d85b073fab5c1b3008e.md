### Title
Unbounded Concurrent `/block` Requests Exhaust Goroutines, DB Connections, and CPU via `FindBetween()` with No Application-Level Concurrency Control

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` performs an unbounded multi-batch DB query loop followed by CPU-intensive in-memory processing (JSON unmarshaling, hash-map construction) with no application-level concurrency limit. The only rate-limiting defense — a Traefik middleware chain — is **disabled by default** (`global.middleware: false`) in the Helm chart. An unprivileged attacker can flood the `/block` endpoint with concurrent requests, causing goroutine accumulation, DB connection pool exhaustion, and CPU saturation exceeding the 30% resource-increase threshold.

### Finding Description

**Exact code path:**

`FindBetween()` at [1](#0-0)  loops fetching batches of `batchSize = 2000` rows until the full timestamp range is covered. Each loop iteration issues a new DB query. After all batches are collected, lines 149–170 perform in-memory processing: building a `sameHashMap`, iterating all transactions, calling `constructTransaction()` which calls `json.Unmarshal` twice per transaction row (once for `CryptoTransfers`, once for `StakingRewardPayouts`) and runs `categorizeHbarTransfers()` with multiple map allocations. [2](#0-1) 

This is called unconditionally from `Block()` in the block service: [3](#0-2) 

**Root cause — no application-level concurrency control:**

The Go application's middleware stack consists only of metrics instrumentation and request tracing: [4](#0-3) 

There is no `semaphore`, no `sync.WaitGroup` cap, no token bucket, and no in-flight request limiter anywhere in the Go application code. The only external rate-limiting defense is the Traefik middleware chain defined in the Helm chart: [5](#0-4) 

This chain includes `inFlightReq: amount: 5` per source IP and `rateLimit: average: 10` per host. However, it is **gated by `global.middleware`**, which defaults to `false`: [6](#0-5) 

When `global.middleware` is false (the default), the entire middleware chain — including rate limiting — is not deployed. Even when enabled, it is infrastructure-level and trivially bypassed by accessing the Go process directly (e.g., via `kubectl port-forward`, internal cluster traffic, or non-Traefik deployments).

**DB pool provides backpressure but not protection:**

`ConnectToDb` configures `MaxOpenConnections`: [7](#0-6) 

When the pool is exhausted, goroutines block inside `GetDbWithContext` waiting for a free connection. New HTTP requests continue to spawn new goroutines (Go's `net/http` server spawns one goroutine per connection), so goroutines accumulate in the blocked state, each consuming stack memory (~2–8 KB initial, growing under load) and holding the transaction data allocated before the block.

**Statement timeout does not bound the loop:**

`statementTimeout` (default 20 s) applies per individual SQL statement, not to the entire `FindBetween` loop: [8](#0-7) 

A block with N transactions requires ⌈N/2000⌉ separate queries, each allowed up to 20 s. The HTTP `WriteTimeout` (10 s) can terminate the client connection, but the goroutine continues executing until its context is cancelled or the loop completes.

### Impact Explanation

With no application-level concurrency limit and the Traefik middleware disabled by default, an attacker sending C concurrent `/block` requests targeting a high-transaction block causes:

- **C goroutines** simultaneously executing multi-batch DB loops and in-memory JSON processing
- **DB connection pool exhaustion**: once `MaxOpenConnections` is reached, all subsequent goroutines block, accumulating in memory
- **CPU saturation**: `json.Unmarshal` and map operations in `constructTransaction` / `categorizeHbarTransfers` / `aggregateHbarTransfers` / `getFeeHbarTransfers` run in every goroutine simultaneously
- **Memory growth**: each goroutine holds the full `[]*transaction` slice for its block in heap memory

A block with 10,000 transactions requires 5 DB round-trips and 10,000 `json.Unmarshal` calls per request. With 50 concurrent requests this is 500 DB queries contending on the pool and 500,000 unmarshal operations in parallel — easily exceeding 30% additional CPU and memory consumption on a normally-loaded node.

### Likelihood Explanation

- **No authentication required**: the Rosetta API is a public, unauthenticated HTTP endpoint
- **No application-level throttle**: the Go process accepts unlimited concurrent connections
- **Default deployment is unprotected**: `global.middleware: false` means the Traefik rate-limit chain is off unless an operator explicitly enables it
- **Trivially scriptable**: a simple `ab -n 10000 -c 200 -p block.json -T application/json http://host:5700/block` suffices
- **Repeatable**: the attack can be sustained indefinitely; there is no lockout or backoff mechanism
- **Amplified by busy blocks**: mainnet blocks with thousands of transactions maximize per-request work

### Recommendation

1. **Add an application-level semaphore** in `FindBetween()` or at the HTTP handler layer to cap concurrent in-flight requests (e.g., a buffered channel semaphore or `golang.org/x/sync/semaphore`).
2. **Enable the Traefik middleware by default** (`global.middleware: true`) or move `inFlightReq` enforcement into the Go application so it cannot be bypassed.
3. **Add a maximum timestamp-range span check** in `FindBetween()` to reject requests whose `end - start` exceeds a configurable nanosecond threshold, bounding the number of DB batches per call.
4. **Cache block transaction results** (the `blockAPIService` already has an `entityCache`; extend it to cache full block responses for recently-requested blocks).
5. **Set `MaxOpenConnections` explicitly** to a low value and document it, so pool exhaustion provides a hard cap rather than an unbounded queue.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, default Helm deployment (`global.middleware: false`), reachable on port 5700. Identify a block index with many transactions (e.g., a busy mainnet block).

```bash
# Step 1: find a high-transaction block index
BLOCK_INDEX=<busy_block_index>

# Step 2: craft the request body
cat > block.json <<EOF
{
  "network_identifier": {"blockchain": "Hedera", "network": "mainnet"},
  "block_identifier": {"index": $BLOCK_INDEX}
}
EOF

# Step 3: flood with concurrent requests (no authentication needed)
ab -n 5000 -c 200 -p block.json -T application/json http://<rosetta-host>:5700/block

# Step 4: observe on the server
# - goroutine count spikes (visible via /debug/pprof/goroutine if enabled, or process metrics)
# - CPU usage jumps >30% above baseline
# - DB connection pool saturates (log lines: "too many clients" or pool wait queue grows)
# - Memory RSS grows proportionally to concurrent goroutines × per-block transaction data size
```

Each concurrent request triggers `FindBetween()` → multi-batch SQL loop → `constructTransaction()` with double `json.Unmarshal` per row, with zero application-level throttling to stop accumulation.

```go
func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}
```

```go
func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop
	}

	if ctx == nil {
		ctx = context.Background()
	}

	// #nosec G118
	childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
	return d.db.WithContext(childCtx), cancel
```
