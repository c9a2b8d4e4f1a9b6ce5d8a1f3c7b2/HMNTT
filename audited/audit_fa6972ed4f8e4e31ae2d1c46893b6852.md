### Title
Unbounded Request Body in `estimateFees()` Enables Heap Exhaustion DoS via Oversized Protobuf POST

### Summary
The `estimateFees()` method in `NetworkController` accepts a raw `@RequestBody byte[] body` with no size constraint, no application-level limit, and no infrastructure-level cap found in the repository. An unauthenticated attacker can POST an arbitrarily large `application/protobuf` body to `POST /api/v1/network/fees`, causing Spring's `ByteArrayHttpMessageConverter` to fully materialize the payload into JVM heap memory before `Transaction.PROTOBUF.parse(Bytes.wrap(body))` is ever called, leading to `OutOfMemoryError` and service crash.

**Root cause:** Spring MVC's `ByteArrayHttpMessageConverter` reads the entire request body into a single `byte[]` with no default size limit for non-form content types. Tomcat's `maxPostSize` (default 2 MB) applies **only** to `application/x-www-form-urlencoded`; it does not apply to `application/protobuf`. No `server.tomcat.max-swallow-size`, no `spring.mvc` body-size property, no custom `OncePerRequestFilter` enforcing a cap, and no nginx ingress `proxy-body-size` annotation were found anywhere in the repository.

**Existing checks reviewed and shown insufficient:**
- `@NotNull` on `body` ظ¤ only rejects a null/missing body, not an oversized one.
- `MetricsFilter` (`rest-java/src/main/java/org/hiero/mirror/restjava/config/MetricsFilter.java`) ظ¤ reads `Content-Length` header for metrics only; enforces nothing.
- No `application.yml` / `application.properties` exists under `rest-java/src/main/resources/` (only `banner.txt` is present), so no server-level size property is set.
- Helm chart `charts/hedera-mirror-rest-java/values.yaml` contains no nginx ingress body-size annotation.
- The `ParseException` catch block at line 121 is never reached ظ¤ the OOM occurs during body deserialization by Spring, before the method body executes.

### Impact Explanation
A single HTTP request carrying a multi-gigabyte body forces the JVM to allocate a contiguous `byte[]` of that size. If heap is exhausted, the JVM throws `OutOfMemoryError`, crashing the `rest-java` service process. Because this is the service that exposes fee estimation and other network APIs, a crash constitutes a total service outage. The attack is repeatable: if the process is restarted (e.g., by Kubernetes), the attacker can immediately crash it again. Severity: **Critical / High** (unauthenticated, zero-cost to repeat, complete service denial).

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly reachable. The attacker needs only `curl` or any HTTP client capable of streaming a large body. The attack is trivially scriptable and repeatable. Even without a multi-GB payload, a sustained stream of moderately large requests (e.g., 256 MB each) across concurrent connections can exhaust heap incrementally. Likelihood: **High**.

### Recommendation
1. **Enforce a body size limit at the Tomcat/Spring layer** ظ¤ add to `application.yml`:
   ```yaml
   spring:
     servlet:
       multipart:
         max-request-size: 1MB
   server:
     tomcat:
       max-swallow-size: 1MB
   ```
   For raw `@RequestBody byte[]`, also register a `CommonsRequestLoggingFilter` or a custom `OncePerRequestFilter` that rejects requests whose `Content-Length` exceeds a defined threshold (e.g., 64 KB ظ¤ the maximum valid Hedera transaction size).
2. **Add a `@Size` or custom constraint** on the `body` parameter, or validate `body.length` at the start of `estimateFees()` and return HTTP 413 before any heap allocation.
3. **Add nginx ingress annotation** in `charts/hedera-mirror-rest-java/values.yaml`:
   ```yaml
   nginx.ingress.kubernetes.io/proxy-body-size: "64k"
   ```
4. **Rate-limit** the `POST /api/v1/network/fees` endpoint by IP at the ingress layer.
