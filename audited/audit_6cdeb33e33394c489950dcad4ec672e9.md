### Title
Unauthenticated DB Load Amplification via `DISTINCT ON` in `/contracts/:contractId/state` Timestamp Query

### Summary
Any unauthenticated user can send a single request to `/api/v1/contracts/:contractId/state?timestamp=lte:<value>&limit=<max>` that forces PostgreSQL to scan and sort **all** `contract_state_change` rows for the target contract before applying the LIMIT. Because `DISTINCT ON` requires a full sort of the filtered result set prior to deduplication, the database work is proportional to the total historical state-change volume of the contract, not the returned row count. No rate limiting exists on the REST API layer, making this trivially repeatable.

### Finding Description

**Code path:**

`rest/controllers/contractController.js` `extractContractStateByIdQuery()` (lines 894–956): when any `timestamp` filter is present, `timestampPresent = true` is set and returned. [1](#0-0) 

`rest/controllers/contractController.js` `getContractStateById()` (line 971) passes `timestamp=true` to the service. [2](#0-1) 

`rest/service/contractService.js` `getContractStateByIdAndFilters()` (lines 256–263): when `timestamp=true`, the query switches from the simple `contractStateQuery` (against the `contract_state` materialized table) to `contractStateTimestampQuery`. [3](#0-2) 

`contractStateTimestampQuery` (lines 111–122) uses `SELECT DISTINCT ON (csc.slot)` from `contract_state_change`: [4](#0-3) 

The ORDER BY appended at runtime is `ORDER BY slot ASC, consensus_timestamp DESC` (lines 258–261): [5](#0-4) 

**Root cause:** PostgreSQL's `DISTINCT ON (slot)` requires the engine to sort the entire filtered result set by `(slot, consensus_timestamp DESC)` and pick the first row per slot *before* the `LIMIT` clause is evaluated. The `LIMIT` only caps the output; it does not reduce the scan or sort work. For a contract with N total state-change rows matching `contract_id = X AND consensus_timestamp <= Y`, the DB must process all N rows regardless of the requested limit.

**Failed assumption:** The code assumes that applying a `LIMIT` at the end of the query bounds the database work. This is false for `DISTINCT ON` — PostgreSQL cannot short-circuit the sort/dedup phase.

**No rate limiting on REST API:** A search for `express-rate-limit`, `rateLimiter`, or equivalent middleware in the REST layer returns no results. The throttling infrastructure found (`ThrottleManagerImpl`, `ThrottleConfiguration`) belongs exclusively to the `web3` Java module and does not protect the Node.js REST API. [6](#0-5) 

### Impact Explanation

An attacker targeting a heavily-used contract (e.g., a DeFi contract with millions of historical slot writes) can issue repeated requests that each force a full sequential scan + sort of the `contract_state_change` table partition for that contract. This amplifies DB CPU and I/O load by a factor of `(total_rows_for_contract / limit)`. With no rate limiting, a single attacker with modest bandwidth can sustain enough request volume to degrade or exhaust DB connection pool resources, causing latency spikes or availability loss for all mirror node consumers. There is no economic damage to network participants — this is a pure availability/griefing impact.

### Likelihood Explanation

Preconditions are minimal: no account, no authentication, no special knowledge beyond a valid contract ID. Contracts with large state-change histories are publicly identifiable on-chain. The exploit is a standard HTTP GET request, repeatable at will, and requires no tooling beyond `curl`. The absence of any REST-layer rate limiting means the attacker is bounded only by their own network capacity.

### Recommendation

1. **Short-term:** Add a middleware rate limiter (e.g., `express-rate-limit`) to the REST API, specifically targeting the `/contracts/:contractId/state` endpoint when a `timestamp` filter is present.
2. **Query fix:** Rewrite the timestamp-based state query to use a subquery or CTE that first narrows to the latest `consensus_timestamp` per `(contract_id, slot)` using the primary key index `(consensus_timestamp, contract_id, slot)`, then applies LIMIT — avoiding a full-table sort. Alternatively, maintain a separate indexed view or leverage the existing `contract_state` table with a join to `contract_state_change` for the timestamp-bounded case.
3. **Consider restricting** the timestamp filter path to authenticated users only, given its significantly higher DB cost compared to the non-timestamp path (which queries the already-materialized `contract_state` table).

### Proof of Concept

```bash
# 1. Identify a contract with many historical state changes (e.g., 0.0.1234567)
# 2. Send max-limit timestamp query as an unauthenticated user:
curl "https://<mirror-node>/api/v1/contracts/0.0.1234567/state?timestamp=lte:9999999999.999999999&limit=100"

# 3. Repeat in a loop to amplify DB load:
while true; do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234567/state?timestamp=lte:9999999999.999999999&limit=100" > /dev/null &
done
```

Each request triggers `SELECT DISTINCT ON (slot) ... FROM contract_state_change WHERE contract_id = X AND consensus_timestamp <= 9999999999999999999 ORDER BY slot, consensus_timestamp DESC LIMIT 100`, forcing a full scan and sort of all historical state changes for that contract before returning 100 rows.
