### Title
Unbounded `endTime` Allows Unprivileged Users to Hold Subscriptions Open Indefinitely (DoS)

### Summary
`TopicMessageFilter` enforces no upper bound on the `endTime` field — only that it is after `startTime`. An unprivileged caller can supply `endTime = Long.MAX_VALUE`, which causes `TopicContext.isComplete()` to permanently return `false`, preventing `pastEndTime()` from ever emitting a termination signal and keeping the reactive subscription alive indefinitely. Repeating this across many connections exhausts server-side resources (threads, memory, database connections).

### Finding Description

**Validation gap — `TopicMessageFilter.java` lines 23 and 43-46:** [1](#0-0) 

`endTime` is a plain nullable `Long` with only one constraint: it must be greater than `startTime`. There is no `@Max`, no cap against `Instant.now()`, and no maximum-future-offset check. `Long.MAX_VALUE` (9,223,372,036,854,775,807 ns ≈ year 2262) trivially satisfies `endTime > startTime`.

**`isComplete()` — `TopicMessageServiceImpl.java` lines 203-215:** [2](#0-1) 

With `endTime = Long.MAX_VALUE`:
- `filter.getEndTime() < startTime` → `false` (Long.MAX_VALUE > any current nanos).
- `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(Duration.ofSeconds(30)).isBefore(Instant.now())` → `false` (year ~2262 is not before now).
- Result: `isComplete()` **always returns `false`**.

**`pastEndTime()` — lines 123-131:** [3](#0-2) 

Because `endTime != null`, this does not return `Flux.never()`. Instead it enters a `repeatWhen` loop gated on `!topicContext.isComplete()`. Since `isComplete()` is always `false`, the predicate is always `true`, and the loop repeats up to `Long.MAX_VALUE` times — it never emits a terminal element. `takeUntilOther(pastEndTime(topicContext))` therefore never fires.

**`takeWhile` — lines 79-81:** [4](#0-3) 

This only terminates when a message with `consensusTimestamp >= endTime` arrives. With `endTime = Long.MAX_VALUE` no real message will ever satisfy this condition.

**No subscriber cap:** `subscriberCount` is only a metric gauge; there is no enforcement of a maximum. [5](#0-4) 

**Exploit flow:**
1. Attacker (no credentials required) opens a gRPC `subscribeTopic` stream with `endTime = Long.MAX_VALUE`, `limit = 0`.
2. `isComplete()` never returns `true`; `pastEndTime()` never terminates the stream.
3. The server holds the reactive pipeline, a bounded-elastic scheduler thread, and any associated DB polling resources open indefinitely.
4. Attacker repeats from many connections/goroutines.

### Impact Explanation
Each persistent subscription consumes a Reactor scheduler thread slot, heap memory for the `TopicContext` and message buffers, and periodic database polling via `missingMessages`. With no subscriber cap and no connection-level rate limiting visible in this service, an attacker can exhaust the bounded-elastic thread pool and heap, causing the gRPC service to become unresponsive for all legitimate users. This is a non-network-based DoS (resource exhaustion) affecting the mirror node's HCS topic streaming capability.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated by default in mirror node deployments. The attack requires only a valid (or non-existent, if `checkTopicExists = false`) topic ID and a single integer field set to `Long.MAX_VALUE`. It is trivially scriptable, repeatable, and requires no special knowledge beyond the public protobuf API. The `endTimeInterval` default of 30 seconds means the polling loop fires every 30 s per subscription, amplifying DB load linearly with attacker-controlled subscription count. [6](#0-5) 

### Recommendation
1. **Add an upper-bound validator on `endTime` in `TopicMessageFilter`** — reject any `endTime` beyond a configurable maximum offset from `Instant.now()` (e.g., 30 days).
2. **Enforce a maximum concurrent subscriber count** — check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if the cap is exceeded.
3. **Add a hard cap on `endTime`** inside `isComplete()` as a defense-in-depth measure, treating any `endTime` beyond `now + maxFutureWindow` as already-complete or invalid.
4. Consider per-IP or per-client rate limiting at the gRPC interceptor layer.

### Proof of Concept
```
# Using grpcurl against a running mirror node
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "consensusEndTime":   {"seconds": 9223372036, "nanos": 854775807},
    "limit": 0
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
Repeat this command in a loop from multiple clients. Each stream will remain open indefinitely. Observe `hiero.mirror.grpc.subscribers` metric climbing without bound, bounded-elastic thread pool saturation, and eventual OOM or thread starvation causing legitimate subscribers to be rejected or time out.

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
