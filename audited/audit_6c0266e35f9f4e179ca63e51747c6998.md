### Title
Unauthenticated IDOR via Alias/EVM Address Resolution in `HookServiceImpl.getHooks()` Exposes Any Owner's Hooks

### Summary
`HookServiceImpl.getHooks()` accepts an arbitrary `EntityIdParameter` as `ownerId`, resolves it (including aliases and EVM addresses) to an internal entity ID via `entityService.lookup()`, and passes the result directly to `findByOwnerIdAndHookIdIn()` / `findByOwnerIdAndHookIdBetween()` with no check that the caller is the resolved owner. Any unauthenticated or unprivileged caller can supply a victim's alias or EVM address, have it silently resolved to the victim's internal `ownerId`, and receive the victim's full hook list.

### Finding Description
**Code path:**

- `HookServiceImpl.java` line 35: `final var id = entityService.lookup(request.getOwnerId());`
- `EntityServiceImpl.java` lines 31–37: the `lookup()` switch resolves `EntityIdAliasParameter` via `entityRepository.findByAlias(p.alias())` and `EntityIdEvmAddressParameter` via `entityRepository.findByEvmAddress(p.evmAddress())`, returning the victim's `EntityId` with no caller-identity check.
- `HookServiceImpl.java` lines 40 and 51: the resolved `id.getId()` is passed verbatim to `hookRepository.findByOwnerIdAndHookIdBetween(...)` and `hookRepository.findByOwnerIdAndHookIdIn(...)`.

**Root cause:** `HooksRequest` carries only the caller-supplied `EntityIdParameter ownerId` (no session/auth token, no principal binding). Neither `HookServiceImpl` nor `EntityServiceImpl` verifies that the authenticated principal matches the resolved entity. The alias/EVM-address resolution path is the critical enabler: a numeric ID might be guessable but requires knowing the internal ID; an alias or EVM address is a public, human-readable identifier that maps directly to the victim's internal ID.

**Why existing checks fail:** There are none. The entire call chain from `HooksRequest.getOwnerId()` → `entityService.lookup()` → `hookRepository.find*` contains zero authorization assertions.

### Impact Explanation
An attacker can enumerate the complete hook list of any account on the network by supplying that account's public alias or EVM address as the `ownerId` query parameter. Hook data may contain sensitive configuration, trigger conditions, or business logic that the owner considers private. This is a full, unauthenticated Insecure Direct Object Reference (IDOR) affecting every account that has hooks registered.

### Likelihood Explanation
Exploitation requires no privileges, no special tooling, and no on-chain interaction. Aliases and EVM addresses are public by design on Hedera/Hiero. Any external user who can reach the REST-Java API endpoint can immediately exploit this by issuing a single HTTP GET request with a known victim alias. The attack is trivially repeatable and automatable for bulk enumeration.

### Recommendation
1. Bind the `ownerId` to the authenticated principal at the controller layer: reject any request where the resolved entity ID does not match the caller's authenticated identity.
2. If read access to another account's hooks is intentionally public, document it explicitly and ensure hook data contains no sensitive fields; otherwise enforce ownership.
3. Add an integration test asserting that a request authenticated as account A cannot retrieve hooks owned by account B.

### Proof of Concept
**Preconditions:** Victim account has alias `AABBCCDD...` (base32/hex) and has registered hooks. Attacker knows the alias (public information).

**Steps:**
1. Attacker sends:
   ```
   GET /api/v1/hooks?ownerId=AABBCCDD...
   ```
2. `EntityIdParameter.valueOf("AABBCCDD...")` returns an `EntityIdAliasParameter`.
3. `EntityServiceImpl.lookup()` calls `entityRepository.findByAlias(alias)` and returns the victim's `EntityId` (e.g., internal ID `12345`).
4. `HookServiceImpl.getHooks()` calls `hookRepository.findByOwnerIdAndHookIdBetween(12345, 0, Long.MAX_VALUE, page)`.
5. Response contains the victim's complete hook list — no authentication or ownership check was performed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
