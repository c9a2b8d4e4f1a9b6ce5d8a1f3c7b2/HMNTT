### Title
Unauthenticated, Unrate-Limited `/metrics` Endpoint Enables Prometheus Lock-Contention CPU Exhaustion

### Summary
The `Routes()` function in `rosetta/app/middleware/metrics.go` registers `GET /metrics` backed by `promhttp.Handler()` with no authentication and no rate limiting at the application layer. The Traefik ingress middleware chain â€” which enforces `inFlightReq: 5` and `rateLimit: average 10` â€” only applies to the eight explicitly listed ingress paths and does not cover `/metrics`. Any unprivileged caller with direct access to the service port can issue unlimited concurrent scrape requests, each of which acquires the `prometheus.DefaultGatherer` internal `sync.RWMutex`, serializing all metric collection and driving CPU consumption well above the 24-hour baseline.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, `Routes()`, lines 64â€“73:
```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,          // "/metrics"
            promhttp.Handler().ServeHTTP,   // no auth, no throttle
        },
    }
}
```
`promhttp.Handler()` wraps `prometheus.DefaultGatherer`. Every call to `ServeHTTP` calls `DefaultGatherer.Gather()`, which acquires an internal `sync.RWMutex` to iterate over all registered collectors (the four custom `HistogramVec`/`GaugeVec` families plus the full Go runtime collector â€” dozens of time-series). Concurrent requests serialize on that lock.

**Why the Traefik guards do not help:**

`charts/hedera-mirror-rosetta/values.yaml` lines 119â€“166 configure the Traefik `Middleware` CRD and attach it to the ingress. The ingress `paths` list is:
```
/rosetta/account, /rosetta/block, /rosetta/call, /rosetta/construction,
/rosetta/events, /rosetta/mempool, /rosetta/network, /rosetta/search
```
`/metrics` is absent. Requests that reach the service port directly (NodePort, LoadBalancer, pod-to-pod, port-forward) bypass the ingress entirely and therefore bypass `inFlightReq` and `rateLimit`.

**Server binding:**

`rosetta/main.go` lines 220â€“227 bind a single `http.Server` on `rosettaConfig.Port` with `corsMiddleware` as the root handler. All routes â€” including `/metrics` â€” share this one listener with no per-route concurrency cap.

### Impact Explanation
An attacker sending N concurrent `GET /metrics` requests forces N goroutines to contend on the `DefaultGatherer` mutex. While one goroutine holds the write lock and serializes the full metric payload (text/plain exposition format, including Go runtime stats), all others spin-wait. This produces measurable CPU spikes proportional to N and to the size of the metric output. With enough parallelism (achievable with a single `ab -c 200` or `wrk` invocation), CPU utilization rises well above the 30% threshold relative to the idle baseline, degrading or blocking legitimate Rosetta API processing on the same process.

### Likelihood Explanation
The precondition is direct reachability of the service port, which is common in: (a) any deployment where the Kubernetes `Service` type is `NodePort` or `LoadBalancer`; (b) any environment where the pod is reachable within the cluster (e.g., a compromised co-tenant pod); (c) development/staging environments. No credentials, tokens, or special headers are required. The attack is trivially repeatable with standard HTTP load tools and requires no knowledge of the application beyond the public `/metrics` path.

### Recommendation
1. **Add application-level authentication** to the `/metrics` handler (e.g., a shared secret checked in a wrapper `http.Handler` before delegating to `promhttp.Handler()`), mirroring the pattern used in `rest/middleware/metricsHandler.js` lines 137â€“144 which gates the REST metrics endpoint behind `config.metrics.config.authentication`.
2. **Add `/metrics` to the ingress paths** so the Traefik `inFlightReq` and `rateLimit` middleware applies to it, or expose metrics on a separate internal-only port not reachable from outside the cluster.
3. **Use `promhttp.HandlerFor` with `promhttp.HandlerOpts{MaxRequestsInFlight: N}`** to enforce a hard concurrency cap at the handler level, independent of any infrastructure middleware.

### Proof of Concept
```bash
# Precondition: service port directly reachable (NodePort / port-forward / pod-internal)
kubectl port-forward svc/hedera-mirror-rosetta 8082:80 &

# Baseline CPU (observe for 60s via kubectl top or cAdvisor)

# Attack: 200 concurrent scrapers, 10 000 total requests
wrk -t 20 -c 200 -d 60s http://localhost:8082/metrics

# Result: CPU on the rosetta pod rises sharply due to DefaultGatherer mutex
# contention and repeated full-registry serialization; no 401/429 is returned,
# all requests succeed with HTTP 200 and full metric payloads.
```
No authentication header, no special privilege, no brute-force volume required â€” a single `wrk` session from any reachable host is sufficient to trigger the condition.
