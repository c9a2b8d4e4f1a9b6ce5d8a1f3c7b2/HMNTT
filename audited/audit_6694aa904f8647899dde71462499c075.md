### Title
Unprivileged `credit_type`-only Request Triggers Unbounded Full-Table Scan via FULL OUTER JOIN on Transfer Tables

### Summary
An unprivileged external user can send `GET /api/v1/transactions?credit_type=credit` (no `account_id`, no timestamp) to force the mirror node REST API to execute a FULL OUTER JOIN between full-table scans of `crypto_transfer` and `token_transfer`, filtered only by `amount > 0`. The `/transactions` endpoint explicitly disables timestamp range validation (`validateRange=false`) and does not require a timestamp when `credit_type` is used alone, leaving no server-side guard against arbitrarily expensive queries. Repeated requests can exhaust database CPU and connection pool resources, degrading service for all users.

### Finding Description

**Exact code path:**

`getTransactions()` at [1](#0-0)  calls `parseTimestampFilters` with `filterRequired=false`, `allowOpenRange=true`, and critically **`validateRange=false`**:

```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [2](#0-1) 

With `validateRange=false`, the 60-day `maxTransactionsTimestampRange` guard is skipped entirely: [3](#0-2) 

When no timestamp is supplied, `parseTimestampFilters` returns `{range: null, neValues: [], eqValues: []}`: [4](#0-3) 

`bindTimestampRange` is a no-op by default (`bindTimestampRange=false`), so `range` stays `null` and `timestampQuery` becomes an empty string: [5](#0-4) 

`extractSqlFromTransactionsRequest` with `credit_type=credit` and no `account_id` produces:
- `accountQuery = ''` (empty)
- `creditDebitQuery = 'ctl.amount > 0'` (non-empty) [6](#0-5) 

In `getTransactionTimestampsQuery`, the condition `if (creditDebitQuery || accountQuery)` is satisfied by `creditDebitQuery` alone, and the inner `if (creditDebitQuery)` branch is taken, emitting the FULL OUTER JOIN: [7](#0-6) 

Each subquery is built by `getTransferDistinctTimestampsQuery` with an **empty `accountQuery`** and only `amount > 0` as a WHERE predicate (plus the empty `timestampQuery`): [8](#0-7) 

The resulting SQL for each subquery is effectively:
```sql
SELECT DISTINCT ON (ctl.consensus_timestamp) ctl.consensus_timestamp, ctl.payer_account_id
FROM crypto_transfer AS ctl
WHERE ctl.amount > 0          -- no account filter, no timestamp filter
ORDER BY ctl.consensus_timestamp DESC LIMIT 25
```
and the same for `token_transfer`. To satisfy `ORDER BY consensus_timestamp DESC LIMIT 25` with only an `amount > 0` predicate, the planner must scan the entire table (or a very large index range) before it can return 25 qualifying rows.

**Root cause:** `credit_type` is accepted as a standalone filter with no enforcement that `account_id` must accompany it, and the timestamp range validation is explicitly disabled for this endpoint.

### Impact Explanation

Each request causes two sequential full (or near-full) index scans on the largest tables in the mirror node database (`crypto_transfer` and `token_transfer`), followed by a FULL OUTER JOIN. On a production mirror node these tables contain billions of rows. An attacker sending a modest rate of concurrent requests (e.g., 10–50 req/s) can saturate database I/O and CPU, causing query latency to spike for all other API consumers. There is no economic damage to Hedera network participants, but the mirror node API becomes unavailable or severely degraded — a griefing / availability impact.

### Likelihood Explanation

The exploit requires zero authentication, zero tokens, and zero knowledge of any account ID. The endpoint is publicly documented and reachable. The single parameter `credit_type=credit` is sufficient. Any script-kiddie or automated scanner can trigger it. The attack is trivially repeatable and can be scripted in a single `curl` command. The only natural mitigations are a database-level `statement_timeout` and connection-pool exhaustion limits, neither of which prevents the query from being issued or from consuming significant resources before timing out.

### Recommendation

1. **Require `account_id` when `credit_type` is specified.** In `extractSqlFromTransactionsRequest` (or in `getTransactions` before query building), reject requests that supply `credit_type` without at least one `account_id` filter.
2. **Enable timestamp range validation for this endpoint.** Change the `parseTimestampFilters` call in `getTransactions` to pass `validateRange=true`, enforcing the existing `maxTransactionsTimestampRange` (60d) limit.
3. **Enable `bindTimestampRange=true` by default** or enforce it specifically when `creditDebitQuery` is non-empty and `accountQuery` is empty, so that unbounded timestamp ranges are automatically capped.
4. **Add a database-level `statement_timeout`** as a last-resort backstop if not already configured.

### Proof of Concept

```bash
# No account_id, no timestamp — triggers full-table scan + FULL OUTER JOIN
curl "https://<mirror-node-host>/api/v1/transactions?credit_type=credit"

# With an explicit unbounded range (validateRange=false allows this)
curl "https://<mirror-node-host>/api/v1/transactions?credit_type=credit&timestamp=gte:0"

# Flood to exhaust DB resources
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?credit_type=credit" &
done
wait
```

The generated inner SQL (no timestamp, no account) will be:
```sql
SELECT DISTINCT ON (ctl.consensus_timestamp) ctl.consensus_timestamp, ctl.payer_account_id
FROM crypto_transfer AS ctl
WHERE ctl.amount > 0
ORDER BY ctl.consensus_timestamp DESC LIMIT 25
```
joined via FULL OUTER JOIN with the equivalent query against `token_transfer`. On a large dataset this query will run for seconds to minutes per request.
