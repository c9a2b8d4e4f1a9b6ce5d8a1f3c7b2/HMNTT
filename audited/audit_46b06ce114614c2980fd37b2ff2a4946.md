### Title
Unauthenticated Per-Subscription DB Flood via Unbounded `PollingTopicListener` Subscriptions (POLL Mode DoS)

### Summary
When `hiero.mirror.grpc.listener.type=POLL` is configured, `PollingTopicListener.poll()` issues an independent database query of up to 5,000 rows every 500 ms for each active subscription. Because the gRPC endpoint has no authentication, no global subscription cap, and the only per-connection limit (`maxConcurrentCallsPerConnection=5`) is trivially bypassed by opening many TCP connections, an unprivileged attacker can saturate the database connection pool and delay or stall all polling cycles for all subscribers.

### Finding Description

**Code path:**

`PollingTopicListener.poll()` (lines 51–62): [1](#0-0) 

When a client subscribes with `limit=0`:
- `TopicMessageFilter.hasLimit()` returns `false` because it checks `limit > 0` [2](#0-1) 
- Therefore `limit = Integer.MAX_VALUE` and `pageSize = Math.min(Integer.MAX_VALUE, 5000) = 5000` [3](#0-2) 

`startTime=0` passes the `@Min(0)` constraint and the `isValidStartTime()` check (`0 <= now()`), so the query scans from the beginning of the topic. [4](#0-3) 

Each subscription fires `topicMessageRepository.findByFilter(newFilter)` — a full JPA `getResultList()` call — every 500 ms (default interval). [5](#0-4) 

The repository executes a real SQL query with `LIMIT 5000` on the `topic_message` table for every tick of every subscription: [6](#0-5) 

**Failed assumption — no global subscription limit:**
`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge; it is never checked against a maximum before accepting a new subscription: [7](#0-6) 

**Failed assumption — per-connection limit is not a global cap:**
`maxConcurrentCallsPerConnection=5` limits calls per TCP connection but does not bound the total number of connections or subscriptions server-wide: [8](#0-7) [9](#0-8) 

**No authentication on the gRPC endpoint:**
The only server interceptor present is a test-only endpoint-context tracker; there is no auth interceptor in the production path: [10](#0-9) 

### Impact Explanation
With N TCP connections × 5 subscriptions each, the server issues `5N` independent DB queries of up to 5,000 rows every 500 ms. At modest scale (e.g., 100 connections = 500 concurrent subscriptions), this is 500 queries/500 ms = 1,000 queries/second, each potentially scanning a large `topic_message` partition. This exhausts the database connection pool and CPU, causing all legitimate polling cycles to queue up and stall — effectively a denial-of-service against the entire gRPC topic subscription service. The impact is bounded to deployments running `type=POLL` (a documented, supported, non-default mode).

### Likelihood Explanation
The gRPC port (default 5600) is publicly reachable with no authentication required. Any client with network access can open arbitrarily many TCP connections. The exploit requires only a standard gRPC client library and knowledge of a valid topic ID (which is public on-chain data). The attack is repeatable and requires no special privileges or credentials.

### Recommendation
1. **Enforce a global subscription cap**: Add a configurable `maxSubscriptions` property and reject new subscriptions in `TopicMessageServiceImpl.subscribeTopic()` when the cap is reached.
2. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.
3. **Cap `startTime=0` queries**: Reject or rewrite filters where `startTime` is older than a configurable threshold (e.g., more than 24 hours in the past) to prevent full-table scans.
4. **Consider `SHARED_POLL` or `REDIS` mode** (the default) for production, as `SharedPollingTopicListener` uses a single shared DB poll regardless of subscriber count. [11](#0-10)

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
