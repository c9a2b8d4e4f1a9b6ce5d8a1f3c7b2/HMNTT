### Title
Unbounded Alias Length Enables Unauthenticated CPU/DB Exhaustion via `getEncodedId()`

### Summary
The `accountAliasRegex` in `rest/accountAlias.js` imposes no upper bound on alias length, and OpenAPI request validation is explicitly disabled (`validateRequests: false`). Any unauthenticated caller can submit arbitrarily long `shard.realm.ALIAS` strings to any endpoint that routes through `EntityService.getEncodedId()`, forcing unbounded regex evaluation, base32 decoding, and a database query per request with no alias-level caching.

### Finding Description

**Code path:**

`rest/accountAlias.js` line 10 — the regex has no length cap: [1](#0-0) 

`rest/accountAlias.js` lines 41–44 — `isValid()` runs the unbounded regex against attacker-controlled input: [2](#0-1) 

`rest/service/entityService.js` lines 125–126 — on a regex match, `fromString()` is called (splits string, calls `base32.decode()`) and then a DB query is issued: [3](#0-2) 

`rest/accountAlias.js` lines 56–64 — `fromString()` splits on `.`, then the constructor calls `base32.decode(base32Alias)` with no length guard: [4](#0-3) 

`rest/accountAlias.js` line 24 — `base32.decode()` is called on the unbounded alias component: [5](#0-4) 

`rest/service/entityService.js` lines 17–20 — the resulting decoded blob is sent as a parameter to a DB query: [6](#0-5) 

**OpenAPI validation is disabled at runtime**, so the spec-level alias pattern (`^(?:[A-Z2-7]{8})*...`) that would bound length is never enforced: [7](#0-6) 

**Response cache does not protect alias resolution.** The cache is keyed by `md5(req.originalUrl)`, so an attacker varying the alias by one character per request bypasses the cache entirely and forces a fresh DB query every time: [8](#0-7) 

There is no rate limiting or alias-level caching in the service layer. [9](#0-8) 

### Impact Explanation
Each request with a megabyte-scale alias string forces: O(n) regex evaluation, O(n) string split, O(n) base32 decode allocating a large `Uint8Array`, and a DB query transmitting a large binary blob parameter. With concurrent requests using distinct alias strings (cache bypass), an attacker can saturate Node.js CPU, heap memory, and database connection/I/O capacity without any authentication. This constitutes a non-network DoS against the REST API process and its backing database.

### Likelihood Explanation
No authentication is required. The affected endpoints (accounts, tokens, contracts, balances, etc.) are all public. The attack requires only an HTTP client capable of sending long query parameters in a loop with slight per-request variation. It is trivially scriptable and repeatable.

### Recommendation
1. Add an explicit maximum length to `accountAliasRegex`, e.g. `[A-Z2-7]{2,100}` (matching the longest valid public key alias), in `rest/accountAlias.js` line 10.
2. Enable `validateRequests: true` in `rest/middleware/openapiHandler.js` so the OpenAPI spec's alias pattern is enforced at the HTTP layer before any business logic runs.
3. Add alias-level caching (alias string → encoded id) in `EntityService` so repeated lookups of the same alias do not hit the database.
4. Consider adding a global request body/parameter size limit in the Express middleware stack.

### Proof of Concept
```bash
# Generate a 500,000-character valid base32 alias (all A's, valid base32 alphabet)
ALIAS=$(python3 -c "print('A' * 500000)")

# Send to any public endpoint that resolves an account by alias
# Vary the alias slightly each iteration to bypass the response cache
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.${ALIAS}${i}" &
done
wait
```

Each request passes `AccountAlias.isValid()`, triggers `base32.decode()` on 500 KB of data, and issues a DB query with the resulting ~312 KB binary blob. With 1000 concurrent requests, this exhausts Node.js heap and database connection pool.
