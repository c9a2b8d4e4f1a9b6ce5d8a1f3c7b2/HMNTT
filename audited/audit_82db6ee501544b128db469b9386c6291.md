### Title
Unauthenticated DB Connection Pool Exhaustion via Correlated `getTokenAccountBalanceSubQuery()` in `/api/v1/balances`

### Summary
When the `/api/v1/balances` endpoint is called without a timestamp filter, `getBalances()` builds a SQL query that embeds `getTokenAccountBalanceSubQuery()` as a correlated scalar subquery in the SELECT list, executing it once per outer row (up to 100 rows). There is no application-level rate limiting on this endpoint, and the REST API DB connection pool defaults to only 10 connections with a 20-second statement timeout. An unauthenticated attacker sending concurrent requests can hold all pool connections for the full timeout window, starving all other REST API consumers.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 133–149, when `tsQuery` is falsy (no timestamp filter provided), the else-branch executes:

```js
const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances
  from entity_balance ab
  ${whereClause}
  order by ab.account_id ${order}
  ${limitQuery}`;
``` [1](#0-0) 

`getTokenAccountBalanceSubQuery()` returns a correlated subquery referencing `ab.account_id` from the outer query:

```js
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
``` [2](#0-1) 

PostgreSQL executes this subquery once per row returned by the outer query. With `limit=100` (the configured max), this is up to 100 sequential index lookups on `token_account` per request.

**Root cause — failed assumption:** The design assumes that the DB connection pool and statement timeout are sufficient guards. They are not, because:

1. **No rate limiting on the REST balances endpoint.** The `ThrottleManager` (bucket4j-based) is only wired into the web3 `ContractController`, not the Node.js REST API. [3](#0-2) 

2. **DB pool is 10 connections by default** with a 20-second statement timeout. [4](#0-3) 

3. **The REST API has no per-IP or global concurrency limit** in application code. The Traefik `inFlightReq` / `rateLimit` middleware shown in the Helm charts applies only to the Rosetta API, not the REST API. [5](#0-4) 

4. The pool is initialized with `max: config.db.pool.maxConnections` (default 10) and `statement_timeout: config.db.pool.statementTimeout` (default 20000 ms). [6](#0-5) 

**Exploit flow:**

```
GET /api/v1/balances?account.balance=gte:0&limit=100
```

- No auth required; `account.balance=gte:0` matches every account.
- `tsQuery` is empty → else-branch → correlated subquery path.
- Outer query returns up to 100 rows; PostgreSQL executes `token_account` lookup 100 times.
- Each request holds a DB connection for the full query duration (up to 20 s).
- 10 concurrent requests exhaust the pool; all subsequent requests queue or fail with connection timeout.

### Impact Explanation

All REST API endpoints share the same 10-connection pool. Pool exhaustion blocks `/api/v1/accounts`, `/api/v1/transactions`, `/api/v1/balances`, and every other endpoint simultaneously. In a multi-instance mirror node deployment sharing a single PostgreSQL backend, an attacker targeting multiple REST API pods can degrade the shared DB, affecting all instances. The mirror node REST API is the primary data-access layer for wallets, explorers, and dApps; its unavailability constitutes a significant service disruption.

### Likelihood Explanation

- **No authentication required.** Any internet-accessible mirror node is reachable.
- **Trivially scriptable.** A simple loop of 10 concurrent HTTP requests is sufficient.
- **Repeatable indefinitely.** The statement timeout resets the connection after 20 s, but the attacker simply re-sends the next wave.
- **No observable side-effects for the attacker** (no writes, no logs beyond normal access logs).

### Recommendation

1. **Add rate limiting to the REST API balances endpoint** (and all REST endpoints) at the application layer, e.g., via an Express middleware using a token-bucket per IP, mirroring the web3 `ThrottleManager` pattern.
2. **Reduce the statement timeout** for the REST API DB pool (e.g., 5 s instead of 20 s) to limit how long each connection is held.
3. **Increase the DB pool size** or use a connection proxy (PgBouncer) with per-user connection limits — the Helm chart already defines `max_user_connections: 250` for `mirror_rest` in PgBouncer, but this is not enforced in the default standalone deployment. [7](#0-6) 
4. **Rewrite the correlated subquery** as a single JOIN or a batch lookup so the DB work per request is bounded and does not scale linearly with the row limit.

### Proof of Concept

```bash
# Send 10 concurrent requests, each holding a DB connection for up to 20 s
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/balances?account.balance=gte:0&limit=100" &
done
wait

# All subsequent requests to any REST endpoint will queue or time out
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: delayed response or connection timeout error
```
