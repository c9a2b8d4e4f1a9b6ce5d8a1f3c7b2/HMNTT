### Title
Gas Budget Recycling via Unconditional Restore on Failing Transactions Bypasses gasPerSecond Throttle

### Summary
`ThrottleManagerImpl.restore()` calls `gasLimitBucket.addTokens()` to immediately return gas tokens to the bucket whenever a transaction fails. Because `addTokens()` refills the bucket instantly (outside the normal 1-second greedy refill cycle), an attacker who repeatedly submits high-gas transactions that always fail can recycle the entire gas budget multiple times per second, consuming far more EVM computation than the `gasPerSecond` limit intends to allow. The only remaining guard is the `rateLimitBucket` (default 500 req/sec), which is insufficient to bound total gas consumption.

`ThrottleConfiguration.gasLimitBucket()` (lines 35–45): [3](#0-2) 

The bucket is configured with `capacity = gasPerSecond` and `refillGreedy(gasPerSecond, Duration.ofSeconds(1))`. The intended model is: the full budget is available at the start of each second and refills linearly. `addTokens()` in bucket4j adds tokens immediately up to the bucket's capacity, completely bypassing the refill schedule.

**Root cause:** The failed assumption is that restoring gas to the bucket is equivalent to "undoing" a charge. In reality, `addTokens()` makes those tokens available again *right now*, within the same second, so the same gas budget can be consumed, restored, and consumed again N times per second — where N is bounded only by the request-rate limit (500/sec by default).

**Exploit flow:**

1. Attacker deploys or targets a contract that always reverts after consuming maximum gas (or crafts a call that causes `gasUsed == gasLimit`, triggering the full-refund branch).
2. Attacker submits a request with `gas = maxGasLimit`. `throttle()` consumes `scaleGas(maxGasLimit)` tokens from `gasLimitBucket`.
3. EVM executes and the transaction fails with `gasUsed == gasLimit`. `restoreGasToBucket()` calls `restore(gasLimit)` → `addTokens(scaleGas(gasLimit))` → bucket is immediately full again.
4. Attacker immediately submits another identical request. The bucket is full; `throttle()` succeeds again.
5. Repeat up to 500 times per second (rate limit).

**Why existing checks fail:**

- `rateLimitBucket` (500 req/sec) limits request count but not total gas consumed per second.
- `gasLimitBucket` is designed to cap total gas per second, but the restore mechanism resets it after every failure, making it ineffective for failing transactions.
- `gasLimitRefundPercent` defaults to 100 (full refund on failure). [4](#0-3) 
- The `SynchronizationStrategy.SYNCHRONIZED` on the bucket prevents race conditions but does not prevent the logical recycling attack. [5](#0-4) 

### Impact Explanation

The `gasPerSecond` limit (default 7,500,000,000 gas/sec) is intended to bound EVM computation load on the mirror node. With this bypass, an attacker can force the node to execute up to `500 × maxGasLimit` gas worth of EVM computation per second — a multiplier of potentially hundreds or thousands over the intended limit. This is a direct, unauthenticated denial-of-service vector against the mirror node's EVM simulation layer. No authentication or special privilege is required; any caller of `POST /api/v1/contracts/call` can trigger it.

### Likelihood Explanation

The attack requires no privileged access, no on-chain funds (mirror node simulates calls), and no special tooling beyond a standard HTTP client. The attacker only needs to know a contract address that reverts (or can deploy one, since the mirror node simulates deployment too). The attack is fully repeatable and automatable. The only friction is the 500 req/sec rate limit, which still allows substantial amplification.

### Recommendation

1. **Do not restore gas for out-of-gas failures.** The condition `gasLimit == result.gasUsed()` indicates the EVM ran to exhaustion — maximum computation was performed. This case should not receive a refund. Remove or invert this branch.
2. **Cap `gasLimitRefundPercent` for reverts.** Consider a lower default (e.g., 0% or 20%) so that even reverted transactions impose a real cost on the gas bucket.
3. **Track cumulative gas consumed per second independently** of the bucket token count, so that `addTokens()` calls cannot cause the total gas consumed in a window to exceed `gasPerSecond`.
4. **Alternatively**, remove the restore mechanism entirely for non-estimate calls, or only restore gas for `estimate` calls (where the gas is intentionally probed).

### Proof of Concept

```
# Precondition: mirror node running with default config (gasLimitRefundPercent=100, requestsPerSecond=500)
# Target: any contract address that always reverts, or address 0x0 with non-empty calldata

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0x000000000000000000000000000000000000dead",
      "gas": 15000000,
      "data": "0xdeadbeef",
      "estimate": false
    }' &
done
wait

# Each request:
# 1. throttle() consumes scaleGas(15_000_000) tokens from gasLimitBucket
# 2. EVM reverts with gasUsed == gasLimit (or result == null)
# 3. restoreGasToBucket() calls restore(15_000_000) → addTokens() refills bucket immediately
# 4. Next request finds bucket full again
#
# Result: 500 requests × 15,000,000 gas = 7,500,000,000 gas of EVM work in <1 second
# vs. intended cap of 7,500,000,000 gas/second total — but all 500 requests succeed
# simultaneously, not sequentially, so actual EVM load = 500× the per-request gas
# with no throttle enforcement.
```

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
