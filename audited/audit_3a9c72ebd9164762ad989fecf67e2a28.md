### Title
Shared Unsynchronized `LinkedList` Between `ListReadableQueueState` and `ListWritableQueueState` Enables Cross-Request Queue Corruption in Concurrent EVM Simulations

### Summary
`MirrorNodeState.initQueueStates()` creates a single `new LinkedList<>()` at line 171 and stores it as the backing store for `RecordCacheService`'s `TRANSACTION_RECEIPTS_STATE_ID`. Both `getReadableStates()` (line 78) and `getWritableStates()` (line 102) wrap this same `LinkedList` instance — the former in a `ListReadableQueueState`, the latter in a `ListWritableQueueState`. Because `MirrorNodeState` is a `@Named` Spring singleton and both wrappers are cached in `ConcurrentHashMap`s, every concurrent HTTP request shares the same unsynchronized `LinkedList`. `ListWritableQueueState.addToDataSource()` and `removeFromDataSource()` mutate the `LinkedList` directly with no locking, while `ListReadableQueueState.peekOnDataSource()` and `iterateOnDataSource()` read it with no locking, making concurrent EVM simulations able to observe mid-mutation queue state.

### Finding Description

**Exact code path:**

- `MirrorNodeState.initQueueStates()` — `MirrorNodeState.java:171`
  ```java
  states.put(RecordCacheService.NAME, new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID, new LinkedList<>())));
  ```
  One `LinkedList` instance is created for the entire lifetime of the singleton bean.

- `MirrorNodeState.getReadableStates()` — `MirrorNodeState.java:77-78`
  ```java
  if (state instanceof Queue<?> queue) {
      data.put(stateId, new ListReadableQueueState<>(serviceName, stateId, queue));
  ```
  The same `queue` reference is passed to `ListReadableQueueState`.

- `MirrorNodeState.getWritableStates()` — `MirrorNodeState.java:101-102`
  ```java
  if (state instanceof Queue<?> queue) {
      data.put(stateId, new ListWritableQueueState<>(serviceName, stateId, queue));
  ```
  The same `queue` reference is passed to `ListWritableQueueState`.

- Both wrappers are cached via `computeIfAbsent` on `ConcurrentHashMap`s (`readableStates`, `writableStates`), so the same wrapper instances — and therefore the same `LinkedList` — are reused across all concurrent requests.

**Direct mutation without synchronization:**

`ListWritableQueueState.addToDataSource()` calls `backingStore.add(element)` directly on the `LinkedList`, and `removeFromDataSource()` calls `backingStore.remove()` directly — `ListWritableQueueState.java:31-38`. These are called (at minimum) during `MapWritableStates.commit()` — `MapWritableStates.java:102` — which is invoked at the end of each EVM simulation.

`ListReadableQueueState.peekOnDataSource()` calls `backingStore.peek()` and `iterateOnDataSource()` calls `backingStore.iterator()` — `ListReadableQueueState.java:34-42` — both directly on the same `LinkedList`, with no synchronization.

**Root cause:** `java.util.LinkedList` is explicitly documented as not thread-safe. Structural modifications (add/remove) while another thread holds an iterator cause `ConcurrentModificationException`, and non-structural concurrent access produces silently incorrect results (wrong head element returned by `peek()`, partial iteration).

**Why existing checks are insufficient:** The `ConcurrentHashMap` used for `readableStates` and `writableStates` only protects the map-level lookup, not the `LinkedList` payload. There is no `synchronized` block, `ReentrantReadWriteLock`, or use of a thread-safe queue (e.g., `ConcurrentLinkedQueue`) anywhere in the queue state path.

### Impact Explanation
During concurrent EVM simulations (`eth_call`, `eth_estimateGas`), the `RecordCacheService` reads the `TRANSACTION_RECEIPTS_STATE_ID` queue to determine prior transaction receipts that affect simulation context. If Thread A's simulation commits and calls `addToDataSource`/`removeFromDataSource` on the shared `LinkedList` while Thread B's simulation is iterating or peeking the same list, Thread B can observe a partially-mutated queue. This causes the EVM simulation to operate on incorrect receipt state, producing wrong gas estimates, wrong revert/success outcomes, or wrong return data — all without any funds being directly at risk. Additionally, mutations from one simulation permanently persist into the shared `LinkedList` (the `onCommit` callback at `MirrorNodeState.java:114` only evicts the readable-states cache entry, it does not reset the `LinkedList`), meaning one request's writes corrupt the baseline state for all subsequent requests.

### Likelihood Explanation
Any unprivileged user with network access to the mirror node's JSON-RPC endpoint can trigger this by sending two or more concurrent `eth_call` or `eth_estimateGas` requests that both exercise the `RecordCacheService` queue path. No authentication, special role, or privileged access is required. The attack is trivially repeatable with a simple HTTP client issuing parallel requests. Under normal production load with multiple concurrent users, this race fires probabilistically without any deliberate attack.

### Recommendation
Replace the `new LinkedList<>()` at `MirrorNodeState.java:171` with a thread-safe queue implementation, e.g. `new ConcurrentLinkedQueue<>()` or `new java.util.concurrent.LinkedBlockingQueue<>()`. Alternatively, introduce a `ReadWriteLock` around all accesses in `ListReadableQueueState` and `ListWritableQueueState`. Additionally, consider whether the `LinkedList` should be reset between simulations (currently mutations persist across requests), and whether per-request isolation (a fresh queue per simulation context) is the correct design.

### Proof of Concept
1. Start the mirror node web3 service with default configuration.
2. Identify any deployed contract address on the network.
3. Using a multi-threaded HTTP client, send 50+ concurrent `eth_call` requests to the JSON-RPC endpoint targeting the same contract.
4. Observe that some responses return `ConcurrentModificationException` stack traces (if the iterator is active during structural modification) or return incorrect/inconsistent simulation results across requests that should be identical.
5. Alternatively, instrument `ListReadableQueueState.peekOnDataSource()` with a log: after one simulation adds a receipt via `addToDataSource`, a concurrent simulation's `peek()` will return that receipt even though it belongs to a different simulation context, demonstrating cross-request state leakage.
