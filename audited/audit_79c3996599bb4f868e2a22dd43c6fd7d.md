### Title
Unthrottled Binary Search Multiplier in `eth_estimateGas` Enables CPU Exhaustion by Unprivileged Users

### Summary
The `eth_estimateGas` endpoint triggers up to 21 full EVM executions per request (1 initial probe + up to 20 binary search iterations via `BinaryGasEstimator`), but the gas throttle is consumed only once per HTTP request based on the declared gas value. Because binary search iterations invoke `doProcessCall` with `estimate=true`, the `restoreGasToBucket` path is skipped and no additional throttle tokens are consumed for those 20 iterations. An unprivileged attacker can exploit this 21x CPU multiplier with a small number of concurrent max-gas estimate requests to cause a 30%+ increase in CPU consumption compared to baseline `eth_call` traffic.

### Finding Description

**Exact code path:**

`ContractController.call()` consumes throttle tokens once per request: [1](#0-0) 

The gas bucket deducts `scaleGas(request.getGas())` tokens — for `gas=15_000_000`: `15_000_000 / 10_000 = 1_500` tokens — and the rate-limit bucket deducts 1 token. This is the only throttle check for the entire request lifecycle. [2](#0-1) 

`ContractExecutionService.estimateGas()` then performs **1 initial full EVM execution** followed by **up to 20 binary search iterations**, each a full EVM execution: [3](#0-2) 

The binary search loop cap is `maxGasEstimateRetriesCount = 20`: [4](#0-3) 

Each binary search iteration calls `doProcessCall(params, gas, true)` with `estimate=true`. The `finally` block in `doProcessCall` only calls `restoreGasToBucket` when `!estimate`, so all 20 iterations bypass gas accounting entirely: [5](#0-4) 

**Root cause:** The throttle model assumes 1 EVM execution per request. The `estimate=true` flag was introduced to skip result validation inside the binary search loop, but it also inadvertently skips the gas-bucket restoration path, meaning the 20 binary search EVM executions are invisible to the throttle. The `CustomGasCalculator` bean wired via `provideGasCalculator()` is invoked on every opcode in every one of these 21 executions: [6](#0-5) 

**Failed assumption:** The gas throttle assumes gas consumed ≈ CPU consumed per request. For `eth_estimateGas` this assumption is violated by a factor of up to 21.

### Impact Explanation

With default configuration (`maxGasLimit=15_000_000`, `maxGasEstimateRetriesCount=20`, `requestsPerSecond=500`, `gasPerSecond=7_500_000_000`):

- Each `eth_estimateGas` request at max gas consumes the same throttle budget as one `eth_call` (1,500 gas tokens + 1 rate token) but performs up to 21× the CPU work.
- If baseline traffic is `eth_call` requests, an attacker injecting even 5 concurrent `eth_estimateGas` requests at max gas causes the equivalent CPU load of ~105 `eth_call` requests — a >2000% spike relative to those 5 slots, easily exceeding the 30% threshold against any realistic baseline.
- No state change occurs (read-only mirror node), so the attacker pays no cost and faces no consequence.
- Sustained attack degrades response latency for all users and can exhaust JVM thread pools.

### Likelihood Explanation

- **No authentication or API key required.** The `/api/v1/contracts/call` endpoint is publicly accessible.
- **No per-IP rate limiting exists.** The global `rateLimitBucket` and `gasLimitBucket` are shared across all callers; a single attacker IP can consume the full budget.
- **Trivially reproducible.** A standard `curl` or `eth_estimateGas` JSON-RPC call with `"estimate": true` and `"gas": "0xE4E1C0"` is sufficient.
- **Repeatable indefinitely.** The attacker simply re-sends requests as fast as the rate limit allows; at 500 req/s, each carrying 21 EVM executions, the server is processing 10,500 EVM executions/s while the throttle believes it is processing 500.

### Recommendation

1. **Account for the binary search multiplier in the gas throttle.** Before entering `BinaryGasEstimator.search()`, consume additional gas tokens proportional to `log2(hi - lo)` (the expected number of iterations), or consume 1 rate-limit token per binary search iteration inside the loop.
2. **Differentiate `ETH_ESTIMATE_GAS` in the throttle.** Apply a separate, lower `requestsPerSecond` cap for estimate requests, or multiply the gas cost by a configurable `estimateGasThrottleMultiplier` (e.g., 21) before deducting from the gas bucket.
3. **Add per-IP rate limiting** at the reverse-proxy or Spring layer to prevent a single source from exhausting the global budget.
4. **Reduce `maxGasEstimateRetriesCount`** from 20 to a lower value (e.g., 10) to reduce the worst-case multiplier while still providing accurate estimates for most contracts.

### Proof of Concept

**Preconditions:** Public access to the mirror node web3 endpoint. No credentials needed.

**Steps:**

```bash
# Deploy or identify any existing contract address on the network.
# Send 10 concurrent eth_estimateGas requests at maximum gas:

for i in $(seq 1 10); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "<any_deployed_contract>",
      "data": "<calldata_for_compute_heavy_function>",
      "gas": 15000000,
      "estimate": true
    }' &
done
wait
```

**Expected result:** Each of the 10 concurrent requests triggers 1 initial EVM execution + up to 20 binary search iterations = up to 210 total EVM executions. The throttle only deducts 10 × 1,500 = 15,000 gas tokens and 10 rate tokens — identical to 10 `eth_call` requests — while performing 21× the CPU work. CPU monitoring (e.g., `top`, JVM JMX metrics) will show a >30% increase in CPU utilization compared to an equivalent baseline of 10 `eth_call` requests against the same contract.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);

                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
