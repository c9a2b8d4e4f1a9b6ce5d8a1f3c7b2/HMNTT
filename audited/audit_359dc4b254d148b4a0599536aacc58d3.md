### Title
Missing `close` Event Handler Causes Permanent `inFlightCounter` Inflation via Abrupt Client Disconnection

### Summary
In `metricsMiddleware()`, `inFlightCounter.add(1)` is called unconditionally for every non-metrics request, but `inFlightCounter.add(-1)` is placed exclusively inside a `res.on('finish', ...)` handler. In Node.js, the `'finish'` event is not emitted when the underlying TCP socket is destroyed before `res.end()` completes — instead only `'close'` fires. Any unprivileged client can exploit this by sending a valid HTTP request and immediately issuing a TCP RST, causing the counter to be permanently incremented without a corresponding decrement.

**Root cause — failed assumption:** The code assumes `'finish'` always fires for every request that passes through the middleware. This is false. Node.js `stream.Writable` (which `http.ServerResponse` extends) emits `'finish'` only after `end()` is called *and* the stream is not already destroyed. When a client sends a TCP RST before the server calls `res.end()`, the socket is destroyed, `res.destroyed` becomes `true`, and any subsequent call to `res.end()` is a no-op — `'finish'` is never emitted. The `'close'` event fires instead, but there is no `res.on('close', ...)` handler anywhere in this file. [3](#0-2) 

There is no `on-finished` package usage and no `'close'` listener anywhere in the REST middleware layer: [4](#0-3) 

**Exploit flow:**
1. Attacker opens a TCP connection to the mirror-node REST API (no auth required for regular endpoints).
2. Sends a syntactically valid HTTP GET request (e.g., `GET /api/v1/transactions HTTP/1.1\r\nHost: ...\r\n\r\n`).
3. Immediately sends a TCP RST (e.g., via `SO_LINGER` with `l_linger=0` on socket close, or raw socket).
4. The middleware has already executed line 169 (`inFlightCounter.add(1)`).
5. The socket is destroyed; `res.destroyed = true`.
6. The async route handler eventually calls `res.end()` — this is a no-op on a destroyed stream.
7. `'finish'` never fires; `inFlightCounter.add(-1)` is never called.
8. Repeat N times → counter is inflated by N permanently.

### Impact Explanation
The `api_all_request_in_processing_total` UpDownCounter is exported via the OpenTelemetry Prometheus exporter and consumed by the mirror node's observability pipeline. A permanently inflated value produces incorrect in-flight request records, corrupting the metric stream exported to the mirror node. Operators relying on this metric for autoscaling, alerting, or capacity planning receive false data. The counter never resets (it is a process-lifetime accumulator), so the corruption persists until process restart. All other per-request metrics (`allRequestCounter`, `durationHistogram`, etc.) are also skipped for abruptly-closed connections since they share the same `'finish'` handler. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
No privileges, credentials, or special network position are required. Any internet-reachable instance is exploitable with a standard TCP socket and `SO_LINGER`/raw-socket RST injection. The attack is trivially scriptable, repeatable at high frequency, and leaves no meaningful application-layer trace (the request never reaches a route handler's response path). The only prerequisite is that `config.metrics.enabled` is `true`, which is the default operational configuration. [7](#0-6) 

### Recommendation
Guard the decrement with a one-shot flag and listen to both `'finish'` and `'close'`:

```js
let settled = false;
const settle = () => {
  if (settled) return;
  settled = true;
  inFlightCounter.add(-1);
};
res.on('finish', () => {
  settle();
  // ... rest of metric recording
});
res.on('close', settle);   // fires on abrupt disconnect
```

Alternatively, use the `on-finished` npm package, which already handles both events and deduplicates correctly:

```js
import onFinished from 'on-finished';
onFinished(res, () => { inFlightCounter.add(-1); /* ... */ });
```

### Proof of Concept
```python
import socket, time

HOST = 'mirror-node-rest'
PORT = 443  # or 80

for i in range(1000):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    # SO_LINGER with l_linger=0 causes TCP RST on close()
    import struct
    s.setsockopt(socket.SOL_SOCKET, socket.SO_LINGER, struct.pack('ii', 1, 0))
    s.connect((HOST, PORT))
    s.sendall(b'GET /api/v1/transactions HTTP/1.1\r\nHost: ' + HOST.encode() + b'\r\n\r\n')
    # Immediately RST — server has incremented inFlightCounter but response not yet sent
    s.close()  # sends RST due to SO_LINGER l_linger=0

# After loop: scrape /swagger/metrics/ and observe
# api_all_request_in_processing_total is inflated by ~1000
# and never decrements back
```

After running, scrape the metrics endpoint and observe `api_all_request_in_processing_total` permanently elevated by the number of RST-closed connections, with no corresponding decrement over time.
