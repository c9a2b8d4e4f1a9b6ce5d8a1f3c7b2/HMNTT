### Title
Unbounded In-Memory Response Buffering via ShallowEtagHeaderFilter Enables Resource Exhaustion on /api/* Endpoints

### Summary
The `etagFilter()` bean in `RestJavaConfiguration` registers Spring's `ShallowEtagHeaderFilter` across all `/api/*` URL patterns without any rate limiting in the `rest-java` module. `ShallowEtagHeaderFilter` wraps every response in a `ContentCachingResponseWrapper` that buffers the entire response body in heap memory to compute an MD5 ETag on every request. Because no per-IP or global request-rate limit exists in `rest-java`, an unauthenticated attacker can flood endpoints such as `/api/v1/network/fees` with concurrent requests, causing sustained heap allocation and GC pressure that can raise node resource consumption above 30%.

### Finding Description
**Exact code location:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause:** Spring's `ShallowEtagHeaderFilter` intercepts every matched response and wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper`. The entire response body is written into an in-memory byte array before the MD5 digest is computed and the `ETag` header is set. This allocation occurs on every request regardless of whether the client sends `If-None-Match`. There is no streaming or size cap.

**Failed assumption:** The design assumes either (a) response bodies are small enough that per-request buffering is harmless, or (b) an upstream rate limiter constrains concurrent requests. Neither holds: the `rest-java` module contains no rate-limiting filter (the `ThrottleConfiguration`/`ThrottleManagerImpl` in `web3/` are entirely separate and do not apply here), and endpoints like `/api/v1/network/fees` return unbounded fee-schedule JSON that is not paginated.

**Exploit flow:**
1. Attacker identifies `/api/v1/network/fees` (or `/api/v1/network/nodes` with a high `limit`) — both are under `/api/*` and require no authentication.
2. Attacker opens many concurrent HTTP connections and issues GET requests in a tight loop from multiple source IPs.
3. For each request, `ShallowEtagHeaderFilter` allocates a byte buffer equal to the full response size in the JVM heap.
4. With N concurrent requests each producing a response of size S bytes, the live heap footprint grows by N×S bytes simultaneously.
5. The JVM's GC is forced into frequent minor and eventually major collections; CPU time shifts from serving requests to GC, and heap usage spikes.

**Why existing checks are insufficient:**
- The `web3` module's `ThrottleConfiguration` (bucket4j rate limiter) is scoped to `hiero.mirror.web3.*` and is not wired into the `rest-java` Spring application context.
- `RestJavaProperties` configures only response headers; it has no rate-limit or concurrency-limit field.
- `QueryProperties` only limits `maxFileAttempts` (DB retry count), not request concurrency.
- No `OncePerRequestFilter` or servlet filter in `rest-java/src/main/java/org/hiero/mirror/restjava/config/` performs IP-based or global rate limiting. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every concurrent request to any `/api/*` endpoint causes a full response-body allocation in the JVM heap. The `/api/v1/network/fees` endpoint returns the complete fee schedule (all transaction types × fee components) as a single non-paginated JSON blob. Under a sustained flood of concurrent requests, heap pressure triggers frequent GC cycles, consuming CPU and memory well above the 30% threshold. In the worst case, the JVM throws `OutOfMemoryError` and the process crashes, causing a complete denial of service for all mirror-node REST Java API consumers. [4](#0-3) 

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge, and no privileged network position — only the ability to send HTTP GET requests to a public endpoint. Standard HTTP benchmarking tools (`wrk`, `ab`, `hey`) are sufficient. The absence of any rate-limiting filter in `rest-java` means the attacker is limited only by the server's accept queue and thread pool, both of which are typically generous in default Spring Boot / Tomcat configurations. The attack is fully repeatable and can be sustained indefinitely. [1](#0-0) 

### Recommendation
1. **Add a rate-limiting filter to `rest-java`**: Introduce a `OncePerRequestFilter` (or reuse bucket4j) in `RestJavaConfiguration` that enforces a per-IP and/or global requests-per-second cap before the ETag filter runs.
2. **Set a response-size threshold on `ShallowEtagHeaderFilter`**: Subclass `ShallowEtagHeaderFilter` and override `isEligibleForEtag()` to skip ETag computation when the response `Content-Length` exceeds a configured threshold (e.g., 64 KB).
3. **Consider replacing `ShallowEtagHeaderFilter` with `EtagHeaderFilter` backed by a cache**: Compute ETags once per unique response and serve cached values, eliminating per-request buffering.
4. **Apply Tomcat/Spring Boot connection and thread-pool limits** (`server.tomcat.max-connections`, `server.tomcat.threads.max`) to bound the number of simultaneous in-flight buffered responses.

### Proof of Concept

```bash
# Observe JVM heap via JMX or actuator metrics:
curl http://<mirror-node-host>:<port>/actuator/metrics/jvm.memory.used

# Expected: heap usage climbs continuously during the attack window;
# GC pause time increases; CPU utilization rises >30% above baseline.
# After attack stops, heap slowly recovers through GC.
```
