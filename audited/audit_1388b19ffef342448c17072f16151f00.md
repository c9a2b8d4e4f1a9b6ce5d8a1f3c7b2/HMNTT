### Title
Unbounded `subscriberCount` Inflation via Hung gRPC Connections During Network Partition

### Summary
In `TopicMessageServiceImpl.subscribeTopic()`, `subscriberCount.incrementAndGet()` fires on every new subscription via `doOnSubscribe`, but `subscriberCount.decrementAndGet()` only fires inside `doFinally`, which requires the Reactor stream to receive a terminal signal (complete, error, or cancel). When a network partition causes the gRPC transport to hang without cleanly terminating, no cancel signal is delivered to the Reactor subscription, so `doFinally` never executes and the counter is never decremented. An unprivileged attacker can open many subscriptions and then silently drop the underlying TCP connections to inflate the `hiero.mirror.grpc.subscribers` gauge metric indefinitely.

### Finding Description
**Exact code path:** [1](#0-0) 

```java
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                .doFinally(s -> subscriberCount.decrementAndGet())
                .doFinally(topicContext::finished));
```

**Root cause:** Project Reactor's `doFinally` is guaranteed to fire only on `ON_COMPLETE`, `ON_ERROR`, or `CANCEL` signals. When the gRPC transport layer hangs (no TCP FIN/RST received by the server), the HTTP/2 stream is never closed from the server's perspective, so no cancellation is propagated down to the Reactor subscription. The `doFinally` callback never executes, and `subscriberCount` is never decremented for that subscriber.

**Failed assumption:** The design assumes that every subscription lifecycle ends with a terminal Reactor signal. This holds for clean disconnects but fails for silent network-level drops (firewall rules, middlebox packet loss, VM-level network isolation).

**Exploit flow:**
1. Attacker opens N gRPC connections to `ConsensusService/subscribeTopic` with a valid topic ID and no `endTime` (open-ended subscription). No authentication is required.
2. Each subscription triggers `doOnSubscribe` → `subscriberCount` increments by N.
3. Attacker installs a firewall rule (`iptables -A OUTPUT -p tcp --dport <server_port> -j DROP`) or uses a raw socket that simply stops responding, silently dropping all packets without sending TCP RST or FIN.
4. The server's TCP stack still considers the connections alive. Without gRPC keepalive configured, no PING frames are sent, and the server never detects the dead connections.
5. `doFinally` never fires; `subscriberCount` remains inflated by N indefinitely.
6. Repeating this across multiple source IPs (or using connection multiplexing) allows the counter to grow without bound.

**Why existing checks are insufficient:**
- `GrpcProperties` exposes only `endTimeInterval`, `entityCacheSize`, `checkTopicExists`, and a `NettyProperties` reference — no application-level idle timeout or subscription cap. [2](#0-1) 
- For open-ended subscriptions (`filter.getEndTime() == null`), `pastEndTime()` returns `Flux.never()`, so there is no built-in termination. [3](#0-2) 
- The `subscriberCount` `AtomicLong` has no ceiling; it is a raw gauge with no overflow protection. [4](#0-3) 

### Impact Explanation
The `hiero.mirror.grpc.subscribers` Micrometer gauge is the canonical metric for active subscriber count. If this metric drives autoscaling policies (e.g., Kubernetes HPA or cloud autoscaler), an attacker can force continuous scale-out, exhausting compute budget. If it drives alerting thresholds, it generates sustained false-positive pages. Because the counter never self-corrects without keepalive or a server restart, the impact persists until an operator intervenes. Severity: **Medium–High** (availability and operational integrity impact; no direct data exfiltration).

### Likelihood Explanation
The attack requires only the ability to open a gRPC connection to the public endpoint (no credentials needed, as `topicExists` only validates topic ID) and the ability to silently drop outbound packets — trivially achievable from any host with local firewall control or from a cloud VM with a security-group rule. It is repeatable, scriptable, and leaves minimal forensic traces since no error is generated server-side. The only mitigating factor is whether gRPC keepalive is configured at the infrastructure level (not visible in this codebase).

### Recommendation
1. **Enable gRPC server-side keepalive** via Netty/gRPC configuration (e.g., `GRPC_ARG_KEEPALIVE_TIME_MS`, `GRPC_ARG_KEEPALIVE_TIMEOUT_MS`, `GRPC_ARG_MAX_CONNECTION_IDLE_MS`) so dead connections are detected and cancelled, triggering `doFinally`.
2. **Add an application-level idle/inactivity timeout** to the Flux pipeline (e.g., `.timeout(Duration)` or `.take(Duration)`) so subscriptions that receive no messages for a configurable period are terminated regardless of transport state.
3. **Cap concurrent subscriptions per IP or globally** before incrementing the counter, rejecting new subscriptions when a threshold is exceeded.
4. **Use a `try/finally`-style guard** or a `Disposable` registry that is periodically audited to detect and forcibly cancel subscriptions whose underlying transport is no longer responsive.

```python
import grpc
import threading
import socket, struct

# Step 1: Open N subscriptions (no auth required)
stubs = []
for _ in range(500):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    # call subscribeTopic with a valid topicId, no endTime
    # store channel reference to keep subscription alive
    stubs.append(channel)

# Step 2: Silently drop all outbound packets to the server
# (on Linux attacker host)
import subprocess
subprocess.run(["iptables", "-A", "OUTPUT", "-p", "tcp",
                "--dport", "5600", "-j", "DROP"])

# Step 3: Observe metric on server
# curl http://mirror-node:8080/actuator/prometheus | grep hiero_mirror_grpc_subscribers
# Value climbs to 500 and stays there indefinitely.
# doFinally() never fires; subscriberCount never decrements.
```
