### Title
Host Header Injection via `X-Forwarded-Host` Poisons Pagination `links.next` URL

### Summary
When `config.response.includeHostInLink` is `true`, `getPaginationLink` in `rest/utils.js` constructs the `links.next` URL using `req.hostname` without any sanitization. Because the Express app unconditionally sets `trust proxy` to `true`, Express resolves `req.hostname` from the attacker-supplied `X-Forwarded-Host` header, allowing any unauthenticated caller to inject an arbitrary hostname into the paginated response's `links.next` field. API clients that automatically follow pagination links will then send their next request â€” including any authorization headers â€” to the attacker-controlled host.

### Finding Description
**Exact code path:**

`rest/server.js:58` sets `app.set('trust proxy', true)`. [1](#0-0) 

With this setting, Express resolves `req.hostname` from the `X-Forwarded-Host` request header when it is present, with no further validation.

`rest/utils.js:810` then uses `req.hostname` verbatim to build the URL prefix: [2](#0-1) 

```js
const urlPrefix = config.response.includeHostInLink
  ? `${req.protocol}://${req.hostname}:${config.port}`
  : '';
```

That prefix is concatenated into the final link at line 818: [3](#0-2) 

`rest/schedules.js:274` calls `getPaginationLink` and places the result directly into `schedulesResponse.links.next`, which is returned to the caller: [4](#0-3) 

**Root cause / failed assumption:** The code assumes `req.hostname` reflects the legitimate server hostname. With `trust proxy: true`, Express instead trusts the client-supplied `X-Forwarded-Host` header, so the assumption is false for any request that includes that header.

**Exploit flow:**
1. Attacker sends `GET /api/v1/schedules` with header `X-Forwarded-Host: attacker.example.com`.
2. Express sets `req.hostname = "attacker.example.com"`.
3. `getPaginationLink` builds `http://attacker.example.com:5551/api/v1/schedules?schedule.id=gt:...`.
4. The response body contains `"links": {"next": "http://attacker.example.com:5551/api/v1/schedules?..."}`.
5. Any SDK, script, or service that automatically follows `links.next` for pagination sends its next request â€” including `Authorization` or API-key headers â€” to the attacker's server.

### Impact Explanation
The direct impact is **pagination link poisoning**: the server returns a `links.next` URL pointing to an attacker-controlled host. Clients that blindly follow pagination links (a common pattern in automated consumers of REST APIs) will exfiltrate their credentials and any request data to the attacker. This is effectively a **client-side open redirect** embedded in the API response. Severity is **Mediumâ€“High** when `includeHostInLink` is enabled and automated consumers exist.

### Likelihood Explanation
The attack requires zero privileges â€” it is a single unauthenticated HTTP request with a crafted `X-Forwarded-Host` header. The `trust proxy: true` setting is already in place unconditionally. The only gate is whether `config.response.includeHostInLink` is enabled; if it is, the attack is trivially repeatable by any external actor. Automated API consumers (scripts, SDKs, indexers) that follow `links.next` are common in blockchain/mirror-node ecosystems, making exploitation realistic.

### Recommendation
1. **Validate `req.hostname` against an allowlist** of known legitimate hostnames before using it in URL construction, or
2. **Use a statically configured `baseUrl`** from `config` instead of `req.hostname`, eliminating reliance on any request-derived value, or
3. **Scope `trust proxy`** to only the specific proxy IP ranges that are trusted, rather than `true` (which trusts all proxies), so that `X-Forwarded-Host` from external clients is not honored.

The safest fix is option 2: replace `${req.protocol}://${req.hostname}:${config.port}` with a fully operator-configured `config.response.baseUrl`.

### Proof of Concept
```bash
# Assuming includeHostInLink is true and the server is at localhost:5551
curl -s \
  -H "X-Forwarded-Host: attacker.example.com" \
  "http://localhost:5551/api/v1/schedules?limit=1" \
  | python3 -m json.tool | grep '"next"'

# Expected poisoned output:
# "next": "http://attacker.example.com:5551/api/v1/schedules?schedule.id=gt:0.0.X"
```

Any client that then follows that `next` link will connect to `attacker.example.com` instead of the legitimate mirror node.
