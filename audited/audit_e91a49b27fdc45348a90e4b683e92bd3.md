### Title
ANSI Escape Sequence Injection via Unsanitized HTTP Headers in `TracingMiddleware`

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` logs the client IP address sourced directly from the `X-Real-IP` or `X-Forwarded-For` HTTP headers without any sanitization. An unprivileged external attacker can embed ANSI terminal escape sequences in these headers, causing them to be written verbatim into log output. This corrupts or manipulates what operators see in terminals or log viewers that render ANSI codes.

### Finding Description
**Code path:**

In `getClientIpAddress()` (lines 63–74), the IP address is read directly from user-controlled headers with no validation: [1](#0-0) 

This value is then interpolated unsanitized into the log message in `TracingMiddleware()`: [2](#0-1) 

**Root cause:** There is no stripping or encoding of control characters (e.g., ESC `\x1b`) from header values before they are passed to `log.Info()`.

**Why the existing check fails:** The logrus formatter is configured with `DisableColors: true` in `rosetta/main.go`: [3](#0-2) 

`DisableColors: true` only prevents logrus from prepending its own ANSI color codes to the log level prefix. It performs **no sanitization** of the message string content itself. Attacker-supplied escape sequences in the message pass through entirely unmodified.

**Path injection:** `request.URL.RequestURI()` (line 47) returns the percent-encoded URI, so ESC bytes in the path are encoded as `%1b` and are safe. The header injection vector is the exploitable one. [4](#0-3) 

### Impact Explanation
An attacker can:
- **Overwrite previous log lines** using carriage return (`\r`) or cursor-up sequences, hiding evidence of prior requests.
- **Inject fake log entries** that appear to come from legitimate IPs or show false status codes.
- **Corrupt terminal output** for operators tailing logs, degrading their ability to monitor the node.

This is a medium-severity griefing issue: no funds are at risk, but operator visibility is compromised, which can mask other attacks or cause false alarms.

### Likelihood Explanation
- Requires no authentication or special privileges — any HTTP client can set arbitrary headers.
- If the Rosetta service is directly exposed (no reverse proxy stripping/overwriting `X-Real-IP`/`X-Forwarded-For`), exploitation is trivial and repeatable.
- Even behind a proxy, many configurations forward these headers from the client unchanged.

### Recommendation
Sanitize the `clientIpAddress` value (and `path`) before logging by stripping or replacing non-printable/control characters. A minimal fix in `getClientIpAddress()` or at the point of log message construction:

```go
// Strip ANSI escape sequences and control characters
var sanitizeLog = regexp.MustCompile(`[\x00-\x1f\x7f]|\x1b\[[0-9;]*[a-zA-Z]`)

func sanitize(s string) string {
    return sanitizeLog.ReplaceAllString(s, "")
}
```

Apply `sanitize()` to both `clientIpAddress` and `path` before building the `message` string.

### Proof of Concept

```bash
# Inject ANSI red color + fake log line via X-Real-IP header
curl -s http://<rosetta-host>:<port>/network/list \
  -H 'Content-Type: application/json' \
  -H $'X-Real-IP: 1.2.3.4\x1b[1;31m FAKE CRITICAL ERROR: node compromised \x1b[0m' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'
```

**Result:** The log line written to stdout will contain the raw ESC sequences. In any terminal or log viewer that renders ANSI codes (e.g., `journalctl`, `kubectl logs`, Grafana Loki with ANSI plugin, iTerm2), the operator will see red bold text reading `FAKE CRITICAL ERROR: node compromised` instead of the real IP address, with the actual IP hidden.
