### Title
Missing Reactive Stream Timeout on `getNodes()` Enables Indefinite Resource Hold by Unprivileged Users

### Summary
The `getNodes()` handler in `NetworkController` builds a reactive pipeline with no `.timeout()` operator and no server-side gRPC deadline enforcement. The underlying `NetworkServiceImpl` uses `repeatWhen` with `Long.MAX_VALUE` repetitions and no overall stream lifetime bound. An unprivileged external user can open many connections, each issuing up to 5 concurrent `getNodes` calls, holding `boundedElastic` scheduler threads and database connection pool slots indefinitely if the database query stalls, starving other mirror node operations.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) assembles the reactive chain:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(networkService::getNodes)
        .map(this::toNodeAddress)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

There is no `.timeout(Duration)` operator anywhere in this chain. [1](#0-0) 

`NetworkServiceImpl.getNodes()` (lines 68–76) uses:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())
                .withScheduler(Schedulers.boundedElastic()))
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
```

The repeat bound is `Long.MAX_VALUE` with no wall-clock timeout on the overall stream. [2](#0-1) 

The `page()` method calls `transactionOperations.execute(t -> { ... addressBookEntryRepository.findByConsensusTimestampAndNodeId(...) ... })`, a blocking JDBC call executed on a `boundedElastic` thread. [3](#0-2) 

**Server configuration — no deadline enforcement:**

`GrpcConfiguration` only sets `maxConcurrentCallsPerConnection` and the executor; no `maxConnectionAge`, `keepAliveTimeout`, or server-side deadline is configured. [4](#0-3) 

`AddressBookProperties` has no timeout field — only `pageDelay` (250 ms default) and `pageSize` (10 default). [5](#0-4) 

**Nginx proxy — no timeout for this endpoint:**

The nginx config applies `grpc_read_timeout 600s` only to `subscribeTopic`. The generic `location /com.hedera.mirror.api.proto.` block that covers `NetworkService/getNodes` has no timeout directive. [6](#0-5) 

**Root cause:** The failed assumption is that the gRPC framework or the database layer will bound the stream lifetime. The gRPC framework only enforces a deadline if the *client* sends one; the server never imposes its own. The `db.statementTimeout` (documented default 10 s) is a PostgreSQL-level setting that requires the DB server to be reachable to signal the timeout — a network-level TCP hang between the gRPC server and the database bypasses it entirely, leaving the JDBC driver blocked indefinitely on the socket read.

### Impact Explanation

Each stalled `getNodes` stream holds:
- One `boundedElastic` scheduler thread (blocked in JDBC)
- One HikariCP database connection

With `maxConcurrentCallsPerConnection = 5` per connection but no limit on the number of incoming connections, an attacker can open *N* connections × 5 streams = 5N simultaneous stalled streams. [7](#0-6) 

Exhausting the `boundedElastic` thread pool or the HikariCP connection pool directly starves the importer's record-file parsing pipeline and the topic-message subscription path, which share the same database pool — matching the "starving mirror node export operations" scenario described in the question scope.

### Likelihood Explanation

- **No authentication required.** The gRPC `NetworkService` is a public API; any network-reachable client can call `getNodes()`.
- **Trivially repeatable.** A single attacker script can open hundreds of TCP connections and issue 5 `getNodes` calls on each before the server can react.
- **DB stall is inducible.** A slow or overloaded replica, a network partition between the gRPC pod and the database, or a deliberate connection-level RST injection (if the attacker is on-path) all produce the required condition.
- **No client-side deadline needed.** The attacker simply omits a deadline in the gRPC call metadata; the server never enforces one.

### Recommendation

1. **Add a `.timeout(Duration)` operator** in `NetworkController.getNodes()` immediately after `.flatMapMany(networkService::getNodes)`, using a configurable property (e.g., `hiero.mirror.grpc.addressbook.streamTimeout`, suggested default 30 s).
2. **Enforce a server-side gRPC deadline** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge(...)` or a `ServerInterceptor` that checks/sets a deadline on every call.
3. **Add an nginx `grpc_read_timeout`** to the generic `location /com.hedera.mirror.api.proto.` block (e.g., 30 s) to provide a defence-in-depth layer at the proxy.
4. **Verify `db.statementTimeout`** is actually applied to the HikariCP datasource used by the gRPC module (no match was found in the grpc Java/YAML sources during this audit).

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import network_service_pb2_grpc
from com.hedera.mirror.api.proto.mirror_network_service_pb2 import AddressBookQuery
from hederahashgraph.api.proto.java import basic_types_pb2

# 1. Simulate DB stall: pause PostgreSQL or drop TCP packets to DB port
#    (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the gRPC host)

# 2. Open many connections, each with 5 concurrent getNodes calls (no deadline)
channels = [grpc.insecure_channel("mirror-node-grpc:5600") for _ in range(100)]
stubs   = [network_service_pb2_grpc.NetworkServiceStub(ch) for ch in channels]

query = AddressBookQuery(file_id=basic_types_pb2.FileID(file_num=102))

# Fire 5 streaming calls per channel — no deadline metadata
streams = []
for stub in stubs:
    for _ in range(5):
        streams.append(stub.getNodes(query))  # no timeout= argument

# 3. Observe: server holds 500 boundedElastic threads + 500 DB connections
#    indefinitely; new getNodes / subscribeTopic calls queue or fail.
import time; time.sleep(3600)   # streams remain open for the full hour
```

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
