### Title
Filter Bypass via Numeric Equivalent of `latest` Block in `ContractCallRequest`

### Summary
The `RequestFilter` for `FilterField.BLOCK` with `FilterType.EQUALS` and expression `"latest"` can be bypassed by an unprivileged user by supplying the numeric decimal or hex equivalent of `Long.MAX_VALUE` as the `block` field. `BlockType.toString()` uses reference equality (`this == LATEST`) to decide whether to emit the string `"latest"`, so a numerically identical but separately constructed `BlockType` instance returns its numeric string instead, causing the filter predicate to evaluate to `false` and skipping any configured `REJECT` or `THROTTLE` action.

### Finding Description

**Code path:**

1. `ContractCallRequest.java` line 26: default block is `BlockType.LATEST = new BlockType("latest", Long.MAX_VALUE)`.
2. `BlockTypeDeserializer.java` line 15-16: user-supplied JSON `block` field is passed to `BlockType.of(value)`.
3. `BlockType.java` lines 59-65: a decimal string like `"9223372036854775807"` (= `Long.MAX_VALUE`) is parsed and returns `new BlockType("9223372036854775807", Long.MAX_VALUE)` — a **new heap instance**, not the `LATEST` singleton.
4. `BlockType.toString()` lines 92-98:
   ```java
   if (this == EARLIEST || this == LATEST || isHash()) {
       return name;          // only reached for the singleton
   }
   return String.valueOf(number);  // returns "9223372036854775807"
   ```
   Reference equality (`this == LATEST`) fails for the new instance, so `"9223372036854775807"` is returned.
5. `RequestFilter.test()` lines 31-35:
   ```java
   var value = field.getExtractor().apply(request);          // BlockType object
   var stringValue = value instanceof String s ? s : String.valueOf(value); // calls toString()
   return type.getPredicate().test(stringValue, expression); // "9223372036854775807".equalsIgnoreCase("latest") → false
   ```
   The predicate returns `false`; the `action()` method in `ThrottleManagerImpl` (lines 44-48) is never reached.

**Root cause:** `BlockType.toString()` conflates object identity with semantic identity. Any numeric value equal to `Long.MAX_VALUE` is semantically "latest" but is not the singleton, so it serialises differently. The filter comparison has no normalisation step.

### Impact Explanation
An operator who configures a `RequestFilter` with `field=BLOCK`, `type=EQUALS`, `expression=latest`, and `action=REJECT` or `action=THROTTLE` intends to block or rate-limit all calls targeting the latest block. An attacker trivially circumvents this control, sending unlimited calls to the latest block state, potentially exhausting node resources or accessing data the operator intended to restrict.

### Likelihood Explanation
No privileges are required. The exploit requires only knowledge of `Long.MAX_VALUE` (`9223372036854775807`) or its hex form (`0x7fffffffffffffff`), both of which are public constants. The attack is fully repeatable, scriptable, and requires a single HTTP request with a modified JSON body. Any user who can reach the `/api/v1/contracts/call` endpoint can exploit this.

### Recommendation
Normalise the `BlockType` before comparison in `RequestFilter.test()`. The simplest fix is to override `equals`/`hashCode` on `BlockType` based on `number` (and `name` for hashes), or to add a dedicated `isLatest()` helper that checks `number == Long.MAX_VALUE` rather than reference identity, and use that in `toString()`:

```java
public String toString() {
    if (number == Long.MAX_VALUE && !isHash()) return "latest";
    if (this == EARLIEST) return name;
    if (isHash()) return name;
    return String.valueOf(number);
}
```

Alternatively, normalise in `BlockType.of()` by returning the `LATEST` singleton whenever the parsed `number` equals `Long.MAX_VALUE`.

### Proof of Concept

**Precondition:** Server is configured with a `RequestFilter`:
```yaml
field: BLOCK
type: EQUALS
expression: latest
action: REJECT   # or THROTTLE
```

**Trigger:** Send the following HTTP request as an unprivileged user:
```http
POST /api/v1/contracts/call
Content-Type: application/json

{
  "block": "9223372036854775807",
  "to": "0x000000000000000000000000000000000000abcd",
  "gas": 21000
}
```

**Result:** `BlockType.of("9223372036854775807")` creates a new `BlockType` instance with `number = Long.MAX_VALUE`. `toString()` returns `"9223372036854775807"`. `"9223372036854775807".equalsIgnoreCase("latest")` is `false`. The filter does not match. The `REJECT`/`THROTTLE` action is skipped. The call proceeds to execution against the latest block state, identical to submitting `"block": "latest"`.

The same bypass works with `"block": "0x7fffffffffffffff"` (hex form of `Long.MAX_VALUE`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
