### Title
Historical Block Parameter Accepted for Gas Estimation, Binary Search Executes Against Stale State

### Summary
`ContractController` accepts a user-supplied `block` field alongside `estimate: true` with no cross-validation, and passes both directly into `ContractExecutionParameters`. `ContractExecutionService.estimateGas()` then performs its initial probe call and the entire binary search against the historical Hashgraph state corresponding to that block, returning a gas figure that is meaningless for the current ledger state. Any unprivileged caller can trigger this with a single unauthenticated POST.

### Finding Description

**Entry point — no guard on the combination of `estimate` + historical block:**

`ContractController.constructServiceParameters()` reads both fields from the request and forwards them verbatim:

```java
// ContractController.java lines 75-89
final var callType = request.isEstimate() ? ETH_ESTIMATE_GAS : ETH_CALL;
final var block = request.getBlock();

return ContractExecutionParameters.builder()
        .block(block)          // ← historical block, e.g. BlockType.of("0x1")
        ...
        .isEstimate(request.isEstimate())   // ← true
        ...
        .build();
```

`ContractCallRequest` has two `@AssertTrue` validators (`hasFrom`, `hasTo`) that check value/sender/receiver relationships only; neither rejects the pairing of `estimate=true` with a historical block number. [1](#0-0) 

**Historical context is set once and never cleared during the binary search:**

`ContractCallService.callContract()` sets the block supplier on the shared `ContractCallContext`:

```java
// ContractCallService.java lines 102-104
ctx.setCallServiceParameters(params);
ctx.setBlockSupplier(Suppliers.memoize(() ->
        recordFileService.findByBlockType(params.getBlock()).orElseThrow(...)));
``` [2](#0-1) 

`ContractCallContext.useHistorical()` returns `true` whenever `block != BlockType.LATEST`, which drives all state reads to the historical snapshot. [3](#0-2) 

**`BinaryGasEstimator.search()` resets only the write-cache, not the block context:**

```java
// BinaryGasEstimator.java lines 35-36
while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
    contractCallContext.reset();   // ← only clears writeCache
``` [4](#0-3) 

`ContractCallContext.reset()` is:
```java
public void reset() {
    writeCache.clear();   // blockSupplier and callServiceParameters are untouched
}
``` [5](#0-4) 

Every iteration then calls `doProcessCall(params, gas, true)` with the same `params` carrying the historical block, so every EVM execution inside the binary search reads from the historical snapshot. [6](#0-5) 

### Impact Explanation
The returned hex value is a gas figure derived entirely from a past ledger state. If the historical state had lower storage-write costs (e.g., a slot was already warm/non-zero at that block but is cold/zero today), the estimate will be below the gas actually required for the current transaction, causing on-chain `INSUFFICIENT_GAS` failures and loss of transaction fees for users who rely on the estimate. Conversely, a block where a contract did not yet exist may cause the estimate call to succeed vacuously (no code to execute) and return an artificially low value. The mirror node's stated contract is to return estimates valid for the current state; this breaks that contract silently with no error to the caller.

### Likelihood Explanation
The attack requires zero privileges: a single unauthenticated POST to `/api/v1/contracts/call` with `"estimate": true` and any valid historical block number (e.g., `"block": "0x1"`). `BlockType.of()` accepts any non-negative integer. The attacker needs only to know a block number where the target contract's gas profile differed from today, which is public information via the mirror node's own block/transaction APIs. The attack is trivially repeatable and requires no special tooling.

### Recommendation
Add an explicit validation in `ContractController` (or in `ContractExecutionService.estimateGas()`) that rejects the combination of `estimate=true` with any block other than `LATEST`:

```java
// In ContractController or ContractExecutionService
if (params.isEstimate() && params.getBlock() != BlockType.LATEST) {
    throw new InvalidParametersException(
        "Gas estimation is only supported for the latest block");
}
```

Alternatively, when `isEstimate` is `true`, unconditionally override the block to `BlockType.LATEST` before constructing `ContractExecutionParameters`, so historical blocks supplied by callers are silently ignored for estimation requests.

### Proof of Concept
```
POST /api/v1/contracts/call
Content-Type: application/json

{
  "to":       "0x<deployed_contract_address>",
  "data":     "0x<function_selector_and_args>",
  "gas":      15000000,
  "estimate": true,
  "block":    "0x1"
}
```
1. Deploy or identify a contract whose storage state at block `0x1` differs from the current state (e.g., a counter that has been incremented many times since block 1).
2. Send the request above. The response contains a gas estimate computed against block-1 state.
3. Send the same request without the `"block"` field (defaults to `latest`). The two estimates differ.
4. Use the block-1 estimate as the gas limit in a real Hedera transaction; the transaction fails with `INSUFFICIENT_GAS`.
