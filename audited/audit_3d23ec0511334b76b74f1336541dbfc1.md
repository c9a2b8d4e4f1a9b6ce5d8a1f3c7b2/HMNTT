### Title
Unbounded Indefinite Polling from Epoch Zero Enables DB Griefing via Unprivileged gRPC Subscription

### Summary
Any unauthenticated caller can invoke `subscribeTopic` with `startTime=0`, no `endTime`, and no `limit`. The `isValidStartTime()` check only requires `startTime <= DomainUtils.now()`, so `startTime=0` passes. The resulting `PollingTopicListener.listen()` loop runs `Long.MAX_VALUE` iterations with no enforced subscription cap, issuing a full-history DB scan every 500 ms per subscription. Multiplying connections multiplies DB load linearly with no server-side brake.

### Finding Description

**Code path and root cause:**

`TopicMessageFilter.isValidStartTime()` only enforces an upper bound: [1](#0-0) 

`@Min(0)` on `startTime` means `0` is the minimum accepted value — epoch zero in nanoseconds — which passes `isValidStartTime()` trivially since `0 <= DomainUtils.now()`. [2](#0-1) 

`listen()` creates an infinite polling loop via `RepeatSpec.times(Long.MAX_VALUE)` with no termination condition when `endTime` and `limit` are absent: [3](#0-2) 

Inside `poll()`, when `last == null` (no messages received yet), the DB query always restarts from `filter.getStartTime()` = 0: [4](#0-3) 

`maxPageSize` defaults to 5000 and `interval` defaults to 500 ms, meaning each subscription fires `SELECT ... WHERE consensus_timestamp >= 0 AND topic_id = X LIMIT 5000` twice per second, scanning from the beginning of the topic's entire history: [5](#0-4) 

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge — it is never compared against any cap and never rejects new subscriptions: [6](#0-5) 

`ConsensusController.subscribeTopic()` applies no authentication, no per-IP limit, and no subscription count check before delegating to the service: [7](#0-6) 

**Why existing checks fail:**
- `isValidStartTime()` is a one-sided check (upper bound only); it was designed to reject future timestamps, not to enforce a minimum historical lookback window.
- `hasLimit()` / `endTime` checks in `TopicMessageServiceImpl` only apply `take()` / `takeWhile()` to the *output* stream — they do not prevent the polling loop from running indefinitely when both are absent.
- `subscriberCount` is observability-only; no code path reads it to reject connections.

### Impact Explanation
When the POLL listener type is configured, an attacker with no credentials can open an arbitrary number of gRPC streams. Each stream drives a perpetual DB polling loop that scans from consensus timestamp 0 — the oldest possible record — up to 5000 rows per query, twice per second. With N concurrent connections the DB receives `N × 2` full-history range scans per second. On a topic with a large message history this saturates DB I/O and CPU, degrading or denying service to legitimate subscribers. There is no economic damage to network participants; the impact is pure service availability griefing.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. The protobuf API explicitly supports `consensusStartTime` with seconds=0, nanos=0, which maps directly to `startTime=0` after `convertTimestamp()`. A single attacker with a scripted gRPC client (e.g., `grpcurl` in a loop) can open hundreds of such subscriptions from one machine. The attack is trivially repeatable and requires no special knowledge beyond the public protobuf schema. [8](#0-7) 

### Recommendation
1. **Enforce a minimum `startTime` floor**: Add a configurable property (e.g., `maxHistoricalLookback`) and reject filters where `DomainUtils.now() - startTime` exceeds it, or require `startTime >= DomainUtils.now() - maxLookback` in `isValidStartTime()`.
2. **Enforce a maximum concurrent subscription count**: Read `subscriberCount` before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when the cap is exceeded.
3. **Require `endTime` or `limit` for open-ended historical queries**: If `startTime` is significantly in the past, mandate at least one termination condition.
4. **Add per-IP / per-client connection throttling** at the gRPC server layer (e.g., via a `ServerInterceptor`).

### Proof of Concept

**Preconditions:** Mirror node running with `hiero.mirror.grpc.listener.type=POLL`. Topic `0.0.X` exists with historical messages.

**Steps:**

```bash
# Open 100 concurrent subscriptions from epoch zero with no limit and no end time
for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": X}, "consensusStartTime": {"seconds": 0, "nanos": 0}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

**Result:** Each of the 100 background processes holds an open gRPC stream. The server spawns 100 independent `PollingContext` instances, each issuing `SELECT ... WHERE consensus_timestamp >= 0 AND topic_id = X LIMIT 5000` every ~500 ms. DB query rate reaches 200 full-history scans/second. DB CPU and I/O spike; legitimate subscriber latency increases or connections are dropped.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
        final var filter = TopicMessageFilter.builder().limit(query.getLimit());

        if (query.hasTopicID()) {
            filter.topicId(EntityId.of(query.getTopicID()));
        }

        if (query.hasConsensusStartTime()) {
            long startTime = convertTimestamp(query.getConsensusStartTime());
            filter.startTime(startTime);
        }

        if (query.hasConsensusEndTime()) {
            long endTime = convertTimestamp(query.getConsensusEndTime());
            filter.endTime(endTime);
        }

        return filter.build();
    }
