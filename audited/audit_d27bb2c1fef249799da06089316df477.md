### Title
Unescaped LIKE Wildcard Injection via BLOCK_HASH Filter Causes Expensive Database Table Scans

### Summary
In `extractContractResultsByIdQuery()`, a user-supplied `block_hash` query parameter is passed without LIKE-wildcard escaping directly into a PostgreSQL `LIKE` pattern in `getRecordFileBlockDetailsFromHash()`. An unprivileged attacker can supply a crafted hash value (e.g., `%` or a single hex character) that expands into a maximally broad `LIKE` pattern, forcing a full or near-full table scan on the `record_file` table on every request. Repeated concurrent requests can exhaust database I/O and CPU, degrading or denying service to the mirror node.

### Finding Description

**Code path:**

1. `rest/controllers/contractController.js`, `extractContractResultsByIdQuery()`, lines 464–492: [1](#0-0) 

The `BLOCK_HASH` filter is stored in `blockFilter` with no additional format or length validation beyond the generic `contractResultsFilterValidityChecks`, which only adds a special operator check for `BLOCK_NUMBER` — not `BLOCK_HASH`: [2](#0-1) 

At line 491, `blockFilter.value` is passed verbatim to:

2. `rest/service/recordFileService.js`, `getRecordFileBlockDetailsFromHash()`, lines 143–147: [3](#0-2) 

The hash is interpolated directly into the LIKE parameter as `` `${hash}%` `` — no escaping of `%` or `_` LIKE metacharacters is performed.

3. The underlying SQL query, lines 58–62: [4](#0-3) 

```sql
WHERE hash LIKE $1
LIMIT 1
```

**Root cause:** The user-controlled hash string is concatenated with `%` and used as a PostgreSQL `LIKE` pattern. If the input itself contains `%` (e.g., the literal character `%`), the pattern becomes `%%`, matching every row. Even without `%` in the input, a very short valid hex prefix (e.g., `0x`) produces `0x%`, matching a large fraction of rows. No minimum-length enforcement or LIKE-metacharacter escaping exists in this code path.

**Why existing checks fail:** `contractResultsFilterValidityChecks` delegates to `utils.filterValidityChecks` for `BLOCK_HASH` and adds no minimum-length or metacharacter constraint. The `LIMIT 1` clause bounds the *result set* but does not prevent the database engine from scanning a large portion of the table to find the first matching row, especially if the index cannot efficiently prune a broad prefix pattern.

### Impact Explanation
Every call to `/api/v1/contracts/{id}/results?block.hash=<crafted>` or `/api/v1/contracts/results?block.hash=<crafted>` triggers a potentially full table scan on `record_file`. The `record_file` table grows continuously as the network produces blocks; on a production mirror node it can contain tens of millions of rows. Flooding the endpoint with concurrent requests using a maximally broad pattern (e.g., `block.hash=%`) saturates database I/O and CPU, degrading query performance across all mirror node APIs — consistent with the stated impact of shutting down ≥30% of network processing capacity without brute-force key attacks.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. No API key, rate-limit, or authentication is required. The attack requires only an HTTP client and knowledge of a single query parameter. It is trivially repeatable and scriptable (e.g., with `curl` or any load-testing tool). The attacker needs zero privileges and zero knowledge of the chain state.

### Recommendation
1. **Escape LIKE metacharacters** before appending `%`. In PostgreSQL, replace `%` → `\%` and `_` → `\_` in the user-supplied hash before constructing the pattern, and add `ESCAPE '\'` to the query.
2. **Enforce minimum hash length** — reject any `block_hash` value shorter than 32 hex characters (16 bytes) at the filter-validation layer in `contractResultsFilterValidityChecks` or in `utils.filterValidityChecks` for `filterKeys.BLOCK_HASH`.
3. **Consider an exact-match index lookup** for full 32-byte or 48-byte hashes and reserve the prefix-LIKE path only for validated partial hashes of sufficient length.

### Proof of Concept

```bash
# Trigger a maximally broad LIKE scan (pattern becomes '%%')
curl -s "https://<mirror-node>/api/v1/contracts/results?block.hash=%25"

# Trigger a broad prefix scan with a 1-character hex prefix (pattern becomes '0%')
curl -s "https://<mirror-node>/api/v1/contracts/results?block.hash=0"

# DoS: flood with concurrent broad-pattern requests
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?block.hash=%25" &
done
wait
```

Each request causes `WHERE hash LIKE '%%'` (or `'0%'`) to execute against the full `record_file` table. Sustained concurrent flooding exhausts database resources and degrades all dependent API endpoints.
