### Title
Gas Throttle Bucket Fully Bypassed via Deliberate Out-of-Gas Exceptional Halt

### Summary
The `restoreGasToBucket()` method in `ContractCallService` unconditionally restores 100% of `gasLimit` back to the `gasLimitBucket` whenever a transaction is not successful AND `gasUsed == gasLimit`. An unprivileged attacker can deliberately trigger this condition by submitting requests with a gas limit too low to complete execution, causing an out-of-gas exceptional halt. The net token consumption from the gas throttle bucket is zero, completely neutralizing the gas-per-second throttle while the attacker remains bounded only by the much weaker request-rate limit.

### Finding Description

**Code path:**

`ContractController.call()` → `throttleManager.throttle(request)` → `contractExecutionService.processCall()` → `doProcessCall()` → `restoreGasToBucket()`

**Throttle entry point** (`ThrottleManagerImpl.java:40`):
```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
```
Tokens consumed = `scaleGas(gasLimit)` = `floor(gasLimit / 10_000)`.

**Restore logic** (`ContractCallService.java:140-151`):
```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline);   // ← full refund path
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

With `gasLimitRefundPercent = 100` (default), `gasLimitToRestoreBaseline = gasLimit`. When the full-refund branch fires, `restore(gasLimit)` is called, which calls `gasLimitBucket.addTokens(scaleGas(gasLimit))` — exactly the same number of tokens that were consumed.

**Root cause:** The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` is satisfied by any out-of-gas exceptional halt. In EVM semantics, an out-of-gas halt sets `gasUsed = gasLimit` and marks the result as failed. An attacker can deliberately engineer this by supplying a gas limit that is sufficient to pass the request-rate check but too low to complete the EVM execution. The code's comment ("If the transaction fails, gasUsed is equal to gasLimit") conflates infrastructure pre-check failures with attacker-controlled out-of-gas conditions.

**Restore call in `ThrottleManagerImpl.java:59-63`:**
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);
    }
}
```
`scaleGas(gasLimit)` tokens are added back — identical to what was consumed — leaving the bucket at the same level as before the request.

### Impact Explanation

The `gasLimitBucket` is the primary defense against high-gas-cost request floods on the mirror node's EVM simulation endpoint. With this bypass, an attacker can send the maximum allowed gas per request (up to `evmProperties.maxGasLimit`, typically 15 000 000) at the full request-rate limit (default 500 req/s) without ever depleting the gas bucket. The EVM still executes until it runs out of gas on each call, consuming real CPU and memory on the server. The gas throttle provides zero protection; only the weaker request-rate bucket (500 req/s) remains. This enables sustained resource-exhaustion (DoS) of the mirror node's `/api/v1/contracts/call` endpoint by any unauthenticated caller.

### Likelihood Explanation

No privileges, authentication, or special contract deployment is required. Any caller can send a standard `eth_call` JSON-RPC request with a deliberately low gas value. The condition is trivially reproducible: set `gas` to a value that passes `scaleGas` (> 10 000) but is insufficient to complete the target contract's execution. The attack is stateless, repeatable at 500 req/s, and requires no on-chain state.

### Recommendation

1. **Do not restore gas for out-of-gas exceptional halts.** The full-refund branch was intended for pre-EVM failures (result == null). Out-of-gas halts represent real EVM work and should not be refunded. Split the condition:
   ```java
   if (result == null) {
       throttleManager.restore(gasLimitToRestoreBaseline); // pre-check failure, no EVM work
   } else if (!result.isSuccessful() && gasLimit == result.gasUsed()) {
       // out-of-gas: EVM ran until exhaustion — do NOT restore
   } else {
       final var gasRemaining = gasLimit - result.gasUsed();
       throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
   }
   ```
2. **Reduce `gasLimitRefundPercent` default** from 100% to a lower value (e.g., 25%) so that even if the full-refund path fires, the net cost to the attacker is non-trivial.
3. **Add a minimum gas floor** for the refund: never restore more than `gasLimit - minimumExecutionCost`.

### Proof of Concept

**Preconditions:** Mirror node running with default config (`gasLimitRefundPercent=100`, `requestsPerSecond=500`, `gasPerSecond=7_500_000_000`).

**Steps:**
1. Deploy any contract with a function that consumes > 50 000 gas (e.g., a loop).
2. Send repeated `POST /api/v1/contracts/call` requests with `gas=50000` (enough to pass the bucket check: `scaleGas(50000)=5` tokens consumed) but insufficient to complete the loop (causing out-of-gas).
3. Observe: each call returns an error with `gasUsed == gasLimit == 50000`.
4. `restoreGasToBucket` fires the full-refund branch: `restore(50000)` → `addTokens(5)` → bucket returns to pre-call level.
5. Repeat at 500 req/s indefinitely. The gas bucket never depletes. The server executes expensive EVM simulation on every call. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```java
    private static final long GAS_SCALE_FACTOR = 10_000L;

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
```
