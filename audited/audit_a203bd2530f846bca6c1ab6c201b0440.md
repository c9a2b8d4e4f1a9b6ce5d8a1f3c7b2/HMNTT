### Title
Unbounded Memory Exhaustion via ShallowEtagHeaderFilter Response Buffering on Unauthenticated `/api/*` Endpoints

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers Spring's `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no rate limiting, no concurrency cap, and no request throttling. `ShallowEtagHeaderFilter` unconditionally buffers the **entire serialized response body** in JVM heap memory for every request in order to compute an MD5 ETag hash. An unprivileged attacker flooding these endpoints with concurrent requests forces simultaneous full-body buffering, exhausting heap memory and causing OOM kills or severe GC pressure that denies legitimate access to Hashgraph history.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause and failed assumption:**

Spring's `ShallowEtagHeaderFilter` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper`. The downstream controller writes the full response body into this wrapper's internal `ByteArrayOutputStream`. Only *after* the entire body is written does the filter compute the MD5 hash, compare it against any `If-None-Match` header, and decide whether to send a 304 or flush the buffered body. This means **every request — regardless of `If-None-Match` value — causes the full response body to be held in heap memory simultaneously with all other in-flight requests.**

The failed assumption is that the filter is cheap. It is not: it trades CPU (MD5 hashing) and heap memory (full body buffering) for every single request.

**No rate limiting in rest-java:** The `web3` module has `ThrottleConfiguration` + `ThrottleManagerImpl` with bucket4j rate limiting. The `rest-java` module has **no equivalent**. `WebMvcConfiguration` only registers argument resolvers. `LoggingFilter` and `MetricsFilter` are observability-only. There is no `@RateLimiter`, no `bucket4j` bean, and no servlet filter enforcing per-IP or global request rate limits on `/api/*` in the rest-java codebase.

**Exploit flow:**

1. Attacker identifies a high-payload endpoint, e.g. `/api/v1/network/nodes` or `/api/v1/accounts/{id}/allowances/nfts`, which returns large paginated JSON bodies.
2. Attacker sends N concurrent GET requests with `If-None-Match: "randomgarbage"` (a value that will never match the computed ETag, ensuring the filter does not short-circuit before buffering).
3. For each concurrent request: the controller executes a DB query, serializes the full response, and `ShallowEtagHeaderFilter` buffers it in a `ByteArrayOutputStream` on the heap.
4. With N concurrent requests each holding, say, 500 KB of buffered response, N=4000 requests consumes ~2 GB — matching the pod's memory limit of `2048Mi`.
5. The JVM hits OOM or enters continuous full-GC, causing the pod to be killed or become unresponsive.

### Impact Explanation
The pod memory limit is `2048Mi` (confirmed in `charts/hedera-mirror-rest-java/values.yaml`). A sustained flood of concurrent requests to any endpoint returning non-trivial response bodies will exhaust this limit. The result is OOM pod termination or severe GC-induced latency, preventing legitimate clients from querying Hashgraph history (transaction records, account state, network topology). This constitutes a denial of service against the mirror node's read API — the primary interface for verifying Hashgraph history integrity.

### Likelihood Explanation
No authentication or API key is required to reach `/api/*` endpoints. The attack requires only an HTTP client capable of sending concurrent requests — trivially achievable with `curl`, `wrk`, `ab`, or any scripting language. The attacker does not need to know valid entity IDs; endpoints with default/empty parameters still execute queries and return buffered responses. The attack is repeatable and can be sustained indefinitely. The Traefik ingress circuit breaker (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) only triggers on HTTP 5xx errors — successful 200 responses from the flooded endpoint do not trip it. The retry middleware (3 attempts) can amplify the load further.

### Recommendation
1. **Add rate limiting to rest-java**: Introduce a bucket4j or Resilience4j rate-limiting filter for `/api/*` endpoints, mirroring the pattern already used in the `web3` module (`ThrottleConfiguration`, `ThrottleManagerImpl`). Apply both per-IP and global limits.
2. **Limit `ShallowEtagHeaderFilter` scope**: Restrict the filter to endpoints where ETag caching provides measurable client benefit, rather than applying it to all `/api/*` patterns.
3. **Set response size limits**: Configure Tomcat's `maxSwallowSize` and consider streaming responses for large result sets rather than fully buffering them.
4. **Ingress-level rate limiting**: Add a Traefik `RateLimit` middleware to the `hedera-mirror-rest-java` chart to enforce per-source-IP request rate caps before requests reach the JVM.

### Proof of Concept
```bash
# Flood /api/v1/network/nodes with 500 concurrent requests, each with a non-matching ETag
# Requires: wrk (https://github.com/wg/wrk)

wrk -t 50 -c 500 -d 60s \
  -H 'If-None-Match: "00000000000000000000000000000000"' \
  https://<mirror-node-host>/api/v1/network/nodes

# Monitor pod memory:
kubectl top pod -l app=hedera-mirror-rest-java -n <namespace> --watch

# Expected: pod memory climbs to 2048Mi limit, pod OOM-killed or latency spikes to >2s
# (triggering RestJavaRequestLatency alert), denying legitimate access
```
