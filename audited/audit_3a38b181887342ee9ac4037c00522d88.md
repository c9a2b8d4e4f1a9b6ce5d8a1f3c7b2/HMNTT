### Title
Unauthenticated High-Rate Requests to `/api/v1/topics/{id}` Cause Unbounded Exception Allocation and GC Pressure Due to Missing Rate Limiting in rest-java

### Summary
The `getTopic()` endpoint in `TopicController` is publicly accessible with no authentication or rate limiting in the rest-java module. Any external user can flood `/api/v1/topics/{nonexistent_id}` with requests, causing `EntityNotFoundException` to be instantiated (with full JVM stack trace capture) on every miss, handled by `GenericControllerAdvice.notFound()`, resulting in sustained GC pressure and database query load that can degrade service availability.

### Finding Description

**Code path:**

`TopicController.getTopic()` at [1](#0-0)  calls `topicService.findById(id.id())` first. Inside `TopicServiceImpl.findById()`: [2](#0-1) 

When the topic does not exist, `orElseThrow()` constructs a new `jakarta.persistence.EntityNotFoundException("Topic not found")`. Java exception construction captures a full stack trace via `Throwable.fillInStackTrace()`, which is a synchronized, expensive native operation. This exception propagates to: [3](#0-2) 

**Root cause — no rate limiting in rest-java:** The `web3` module has a `ThrottleConfiguration` with bucket4j rate limiters: [4](#0-3) 

But the rest-java module has **no equivalent**. The only filters registered in rest-java are `LoggingFilter` and `MetricsFilter`: [5](#0-4) 

`RestJavaProperties` only configures response headers, with no rate-limit or throttle properties: [6](#0-5) 

There is also no `@Cacheable` or any caching on `TopicServiceImpl` or `EntityServiceImpl`, so every request hits the database and the exception path unconditionally.

**Exploit flow:**
1. Attacker sends `GET /api/v1/topics/999999999` in a tight loop (no auth required, no rate limit).
2. Each request: DB query → empty `Optional` → `new EntityNotFoundException(...)` (stack trace captured) → `GenericControllerAdvice.notFound()` → 404 response.
3. At thousands of requests/second, the JVM allocates and GC-collects thousands of exception objects per second, each with a deep stack trace array.

### Impact Explanation
Sustained exception-per-request allocation at high rate causes elevated GC activity (minor and potentially major GC cycles), increasing latency for all concurrent users of the rest-java service. Combined with the per-request database query load (no caching), this can exhaust the DB connection pool and degrade or deny service to legitimate users. The endpoint requires no credentials, making it trivially reachable from anywhere.

### Likelihood Explanation
No preconditions beyond network access. Any script or tool (e.g., `ab`, `wrk`, `curl` in a loop) can trigger this. The attacker does not need to know valid topic IDs — any numeric ID that does not exist suffices. The attack is repeatable, stateless, and requires no authentication or special knowledge of the system.

### Recommendation
1. **Add rate limiting to rest-java**: Introduce a bucket4j or similar rate-limiting filter in the rest-java module (mirroring `ThrottleConfiguration` in web3) applied globally or specifically to `/api/v1/topics/{id}`.
2. **Suppress stack traces for expected 404s**: Use a pre-allocated, static sentinel exception or override `fillInStackTrace()` to return `this` in `EntityNotFoundException` to eliminate stack trace capture cost for expected not-found cases.
3. **Add a negative-result cache**: Cache not-found results for a short TTL (e.g., 5 seconds) to avoid repeated DB queries for the same non-existent ID.

### Proof of Concept
```bash
# No authentication required
for i in $(seq 1 100000); do
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/topics/999999999" &
done
wait
```
Each request hits `TopicServiceImpl.findById()`, throws `EntityNotFoundException`, is caught by `GenericControllerAdvice.notFound()`, and returns HTTP 404. At scale, JVM GC logs will show elevated allocation rate and collection frequency, and DB connection pool metrics will show saturation.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

public class RestJavaProperties {

    @NotNull
    @Valid
    private ResponseConfig response = new ResponseConfig();

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }
