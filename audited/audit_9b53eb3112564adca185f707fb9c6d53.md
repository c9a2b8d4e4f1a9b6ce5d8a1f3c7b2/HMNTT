### Title
Cache-Control: public, max-age=5 Leaked onto 404 Error Responses Enables CDN/Proxy Cache Poisoning on GET /api/v1/topics/{id}

### Summary
The `@ModelAttribute` method in `GenericControllerAdvice` unconditionally writes `Cache-Control: public, max-age=5` directly onto the raw `HttpServletResponse` before the controller method executes. When `TopicServiceImpl.findById()` throws `EntityNotFoundException`, the `@ExceptionHandler` returns a `ResponseEntity` with `null` headers and HTTP 404, but Spring MVC does not clear headers already committed to the underlying `HttpServletResponse`. The 404 error response therefore carries `Cache-Control: public, max-age=5`, instructing any shared CDN or forward proxy to cache the negative result and serve it to all users behind that cache for up to 5 seconds.

### Finding Description

**Code path:**

1. `TopicServiceImpl.findById()` — throws `EntityNotFoundException` for any unknown topic ID: [1](#0-0) 

2. `GenericControllerAdvice.responseHeaders()` — a `@ModelAttribute` method that runs **before** the controller method and writes path-specific headers (including `Cache-Control: public, max-age=5` for `/api/v1/topics/{id}`) directly onto the `HttpServletResponse`: [2](#0-1) 

3. `GenericControllerAdvice.notFound()` — the `@ExceptionHandler` that handles `EntityNotFoundException` and returns a `ResponseEntity` with **null** headers and HTTP 404: [3](#0-2) 

**Root cause:** In Spring MVC, `@ModelAttribute` methods in a `@ControllerAdvice` are invoked during `ModelFactory.initModel()`, which runs before the controller method. The method calls `response::setHeader` directly on the servlet response object. When an exception is subsequently thrown and the `@ExceptionHandler` returns a `ResponseEntity(null headers, 404)`, `HttpEntityMethodProcessor` merges only the `ResponseEntity`'s own (null/empty) headers — it does not clear headers already written to the underlying `HttpServletResponse`. The `Cache-Control: public, max-age=5` header therefore survives on the 404 wire response.

The test suite confirms `Cache-Control: public, max-age=5` is the configured value for this endpoint: [4](#0-3) 

**Why existing checks fail:** The Node.js `responseCacheUpdateHandler` guards against caching non-2xx responses (`httpStatusCodes.isSuccess(res.statusCode)`), but this is the separate Java REST layer (`rest-java`), which has no equivalent guard. The Java layer has no filter, interceptor, or `ResponseBodyAdvice` that strips or overrides `Cache-Control` on error responses. [5](#0-4) 

### Impact Explanation
Any CDN (Cloudflare, Fastly, AWS CloudFront) or corporate forward proxy that respects `Cache-Control: public` will store the 404 response keyed on the URL. Every user behind that shared cache who requests the same topic ID within the 5-second TTL receives the cached 404 — even if the topic was created or indexed in the interim. This is a denial-of-service / griefing primitive: no credentials required, no economic damage, but legitimate read access to valid topic data is blocked for all users sharing the cache entry.

### Likelihood Explanation
The attack requires no authentication, no special network position, and no knowledge of internal state. Any public internet user can trigger it by requesting a topic ID that does not yet exist in the mirror node (e.g., a topic created seconds ago that has not been indexed). The 5-second window is short but repeatable: the attacker can re-poison the cache every 5 seconds with a single unauthenticated GET request. CDN deployments are standard in production mirror node infrastructure, making the shared-cache condition realistic.

### Recommendation
Strip or override `Cache-Control` on all non-2xx responses. The cleanest fix is to add a `javax.servlet.Filter` or Spring `HandlerInterceptor` that, in `afterCompletion`, checks `response.getStatus() >= 400` and calls `response.setHeader("Cache-Control", "no-store")`. Alternatively, modify `GenericControllerAdvice.handleExceptionInternal()` to explicitly set a `no-store` `Cache-Control` header in the returned `ResponseEntity`'s `HttpHeaders` — Spring MVC will then overwrite the previously set value when it writes the response. A third option is to configure the CDN/proxy to never cache 4xx responses regardless of origin headers.

### Proof of Concept

```
# Step 1: Send a request for a non-existent topic ID through a shared CDN/proxy
curl -v https://<mirror-node>/api/v1/topics/0.0.9999999

# Observe in response headers:
# HTTP/1.1 404 Not Found
# Cache-Control: public, max-age=5
# Content-Type: application/json

# Step 2: Within 5 seconds, the CDN/proxy caches this 404.
# All users behind the same CDN/proxy node now receive the cached 404
# for GET /api/v1/topics/0.0.9999999 without hitting the origin.

# Step 3: Repeat every 5 seconds to maintain the poisoned cache entry.
# If 0.0.9999999 is a real topic that was just created and not yet indexed,
# all users are denied access to it for the duration of each cache window.
```
