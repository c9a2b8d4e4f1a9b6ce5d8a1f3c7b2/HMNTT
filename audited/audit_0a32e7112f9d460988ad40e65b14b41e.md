### Title
Unauthenticated 7-DB-Query Fan-Out DoS via `GET /contracts/results/<transactionId>`

### Summary
An unprivileged external user can send `GET /api/v1/contracts/results/<valid_transaction_id>` (non-hash path) to trigger a sequential chain of 2 DB queries followed by 5 parallel DB queries — 7 total per single HTTP request — with no authentication or rate limiting enforced on this endpoint. Because valid Hedera transaction IDs are publicly observable on-chain, an attacker can trivially flood the endpoint and exhaust the database connection pool, causing a denial of service for all users of the mirror node REST API.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 21 registers the route with no middleware guard:
```
router.getExt('/results/:transactionIdOrHash', ContractController.getContractResultsByTransactionIdOrHash);
```

`rest/controllers/contractController.js`, `getContractResultsByTransactionIdOrHash` (lines 1120–1203):

- **Line 1142** — Query 1 (sequential): `TransactionService.getTransactionDetailsFromTransactionId(transactionId, nonce)`
- **Line 1156** — Query 2 (sequential): `ContractService.getInvolvedContractsByTimestampAndContractId(transactionDetails.consensusTimestamp, transactionDetails.entityId)`
- **Lines 1165–1166** — calls `this.getDetailedContractResults(contractDetails, undefined)`

`getDetailedContractResults` (lines 1205–1220) issues 5 queries in a single `Promise.all`:
- Query 3: `ContractService.getContractResultsByTimestamps(...)`
- Query 4: `TransactionService.getEthTransactionByTimestampAndPayerId(...)`
- Query 5: `RecordFileService.getRecordFileBlockDetailsFromTimestamp(...)`
- Query 6: `ContractService.getContractLogsByTimestamps(...)`
- Query 7: `ContractService.getContractStateChangesByTimestamps(...)`

**Root cause — failed assumption:** The design assumes callers are rate-limited or authenticated. Neither is true. `rest/middleware/authHandler.js` lines 18–20 show that requests with no `Authorization` header are silently passed through:
```js
if (!credentials) {
  return;  // no block, no rate limit
}
```
`rest/server.js` lines 82–98 show the full middleware stack: `httpContext → requestLogger → authHandler → optional metrics → optional cache check → routes`. There is no rate-limiting middleware anywhere in the chain.

**Why early-exit checks are insufficient:** The `NotFoundError` guards at lines 1144–1146 and 1161–1163 only fire when the transaction or contract does not exist. An attacker using any valid, existing transaction ID bypasses both guards and reaches all 7 queries on every request.

### Impact Explanation

Each HTTP request from an anonymous user consumes 7 database connections from the shared pool (2 sequential + 5 concurrent). A sustained flood of requests against a single valid transaction ID will saturate the PostgreSQL connection pool, causing all subsequent DB queries across the entire REST API to queue or fail. This is a non-network-based DoS: the bottleneck is the database, not bandwidth, so it is effective even at modest request rates and from a single source IP.

### Likelihood Explanation

Hedera is a public network. All transaction IDs (format `<shard>.<realm>.<account>-<validStartSeconds>-<validStartNanos>`) are visible in block explorers and via the mirror node's own `/api/v1/transactions` endpoint. No privileged access, API key, or account is required. The attack is trivially scriptable with `curl` or any HTTP client in a loop, and is fully repeatable.

### Recommendation

1. **Add a rate limiter** (e.g., `express-rate-limit`) specifically on the `/api/v1/contracts/results/:transactionIdOrHash` route, or globally on all `/api/v1/contracts/results/*` routes, keyed by IP address.
2. **Require authentication** for this endpoint if it is intended for privileged consumers only, and return HTTP 401 for unauthenticated requests rather than silently proceeding.
3. **Collapse the query fan-out**: merge queries 3–7 into fewer, batched SQL statements or a single CTE to reduce per-request DB load regardless of rate limiting.
4. **Cap the DB connection pool** and add circuit-breaker logic so pool exhaustion degrades gracefully rather than causing a full outage.

### Proof of Concept

**Preconditions:** Mirror node REST API is reachable. A valid transaction ID is obtained from the public Hedera network (e.g., via `GET /api/v1/transactions?limit=1`).

**Steps:**

```bash
# Step 1: Obtain a valid transaction ID (publicly available, no auth needed)
TX_ID=$(curl -s "http://<mirror-node>/api/v1/transactions?limit=1" \
  | jq -r '.transactions[0].transaction_id')

# Step 2: Flood the endpoint — each request triggers 7 DB queries, no auth required
for i in $(seq 1 500); do
  curl -s "http://<mirror-node>/api/v1/contracts/results/${TX_ID}" &
done
wait
```

**Result:** The PostgreSQL connection pool is exhausted. Subsequent requests to any mirror node REST API endpoint return errors or time out, constituting a full denial of service. No credentials, API keys, or special network position are required. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```javascript
  getContractResultsByTransactionIdOrHash = async (req, res) => {
    if (utils.conflictingPathParam(req, 'transactionIdOrHash', 'logs')) {
      return;
    }

    utils.validateReq(req, acceptedSingleContractResultsParameters);

    // Extract hbar parameter (default: true)
    const convertToHbar = utils.parseHbarParam(req.query.hbar);

    let transactionDetails;

    const {transactionIdOrHash} = req.params;
    if (utils.isValidEthHash(transactionIdOrHash)) {
      const detailsByHash = await ContractService.getContractTransactionDetailsByHash(
        utils.parseHexStr(transactionIdOrHash)
      );
      transactionDetails = detailsByHash[0];
    } else {
      const transactionId = TransactionId.fromString(transactionIdOrHash);
      const nonce = getLastNonceParamValue(req.query);
      // Map the transactions id to a consensus timestamp
      const transactions = await TransactionService.getTransactionDetailsFromTransactionId(transactionId, nonce);

      if (transactions.length === 0) {
        throw new NotFoundError();
      }
      transactionDetails = transactions[0];
      // want to look up involved contract parties using the payer account id
      transactionDetails.entityId = transactionDetails.payerAccountId;
    }

    if (!transactionDetails) {
      throw new NotFoundError();
    }

    const contractDetails = await ContractService.getInvolvedContractsByTimestampAndContractId(
      transactionDetails.consensusTimestamp,
      transactionDetails.entityId
    );

    if (!contractDetails) {
      throw new NotFoundError();
    }

    const [contractResults, ethTransactions, recordFile, contractLogs, contractStateChanges] =
      await this.getDetailedContractResults(contractDetails, undefined);
```
