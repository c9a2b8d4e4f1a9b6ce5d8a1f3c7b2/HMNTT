### Title
Soft-Deleted Hook State Disclosure via `findByOwnerIdAndHookIdIn()` Enables Gossip-Window Timing Attack

### Summary
`findByOwnerIdAndHookIdIn()` in the REST-Java `HookRepository` returns all hooks for an owner regardless of their soft-deleted state, and the `deleted` field is a required field in the public API response. Any unauthenticated user can poll the hooks endpoint to detect the exact consensus moment a gossip-blocking hook transitions from `deleted=false` to `deleted=true`, revealing the precise window during which the account is unprotected by hook-enforced gossip controls.

### Finding Description
**Code path:**

`HookRepository.java` (rest-java), line 13:
```java
List<Hook> findByOwnerIdAndHookIdIn(long ownerId, Collection<Long> hookIds, Pageable pageable);
```
Spring Data JPA generates: `SELECT * FROM hook WHERE owner_id = ? AND hook_id IN (?)` — no `AND deleted = false` predicate.

`HookServiceImpl.getHooks()` (lines 51–52) calls this method directly with no post-query filtering on `deleted`.

The `AbstractHook` domain class (line 57) carries a `Boolean deleted` field. After a soft-delete, the row remains in the `hook` table with `deleted = true` (confirmed by test `cryptoUpdateWithHooksSequentialOperations`, lines 1956–1963: "Hook should still exist after final soft delete", "Hook should be marked as deleted after final deletion").

The OpenAPI spec (`openapi.yml`, lines 1688–1691) declares `deleted` as a **required** response field:
```yaml
deleted:
  description: Whether the hook has been deleted
  type: boolean
```

**Contrast with hook storage:** `HookStorageRepository` methods explicitly append `AndDeletedIsFalse` (e.g., `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse`), filtering out deleted entries. No equivalent filter exists for `HookRepository`.

**Exploit flow:**
1. Attacker identifies a target account known to have a gossip-blocking hook (hook ID discoverable via the same public endpoint).
2. Attacker polls `GET /api/v1/accounts/{target}/hooks?hook.id=eq:{hookId}` at regular intervals.
3. When the response transitions from `"deleted": false` to `"deleted": true`, the attacker knows the hook was just removed at the `timestamp.from` value in the same response.
4. Attacker immediately submits transactions that would have been intercepted/blocked by the hook, exploiting the unprotected window before any replacement hook is installed.

**Failed assumption:** The design document states the service should "Query hooks by owner_id with deletion filtering," but the implementation omits this filter entirely for the hook-listing path.

### Impact Explanation
An unprivileged observer gains precise, consensus-timestamp-level knowledge of when a gossip-blocking hook ceases to be active. This converts what should be an internal lifecycle state into a public timing oracle. In a protocol where hooks enforce pre/post-transaction allowance controls (`ACCOUNT_ALLOWANCE_HOOK`), knowing the exact removal moment allows an attacker to race-submit transactions during the unprotected interval, bypassing hook-enforced authorization logic that the account owner intended to be continuously active.

### Likelihood Explanation
The mirror node REST API requires no authentication. Any network participant can poll the hooks endpoint indefinitely at negligible cost. The polling interval can be as short as one consensus round (~3–5 seconds), making the timing window detectable with high precision. The attack is fully repeatable and requires no privileged access, special keys, or on-chain transactions — only HTTP GET requests.

### Recommendation
Add a `deleted = false` filter to both `HookRepository` query methods:

```java
List<Hook> findByOwnerIdAndHookIdInAndDeletedFalse(
    long ownerId, Collection<Long> hookIds, Pageable pageable);

List<Hook> findByOwnerIdAndHookIdBetweenAndDeletedFalse(
    long ownerId, long lowerBound, long upperBound, Pageable pageable);
```

Update `HookServiceImpl.getHooks()` to call the filtered variants. If historical/deleted hook visibility is a legitimate use case, gate it behind an explicit opt-in query parameter (e.g., `include.deleted=true`) with appropriate documentation, mirroring how other mirror node endpoints handle deleted entities.

### Proof of Concept
```
# Step 1: Observe active hook
GET /api/v1/accounts/0.0.123/hooks?hook.id=eq:5
→ { "hooks": [{ "hook_id": 5, "deleted": false, "timestamp": {"from": "T1", "to": null} }] }

# Step 2: Poll repeatedly (no auth required)
# Step 3: Hook owner submits CryptoUpdate to delete hook ID 5

# Step 4: Next poll reveals soft-delete
GET /api/v1/accounts/0.0.123/hooks?hook.id=eq:5
→ { "hooks": [{ "hook_id": 5, "deleted": true, "timestamp": {"from": "T2", "to": null} }] }

# Step 5: Attacker now knows hook was removed at consensus timestamp T2.
# Attacker immediately submits transactions to 0.0.123 that the hook would have blocked,
# exploiting the window before a replacement hook is installed.
```
