### Title
Zombie Reactive Pipeline Resource Exhaustion via Network Partition (No gRPC Keepalive Configured)

### Summary
The `ConsensusController.subscribeTopic()` method registers a cancel handler (`disposable::dispose`) that only fires when the gRPC framework detects a client cancellation. Because the `NettyServerBuilder` is configured with no keepalive parameters, a network partition leaves the TCP connection appearing alive on the server side indefinitely, the cancel handler never fires, and the server-side reactive pipeline (DB polling loops, scheduler threads, listener subscriptions) continues consuming resources for every zombie subscription.

### Finding Description

**Cancel handler registration — `ConsensusController.java` lines 43–53:**
```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        ...
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
}
```
The `disposable::dispose` cancel handler is the sole mechanism for tearing down the pipeline. It is only invoked when the gRPC/Netty layer fires an `onCancel` event, which requires the framework to detect the disconnection via a TCP FIN/RST or an HTTP/2 RST_STREAM frame.

**Missing keepalive — `GrpcConfiguration.java` lines 28–35:**
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```
The `NettyServerBuilder` customizer sets only `executor` and `maxConcurrentCallsPerConnection`. There is no `keepAliveTime`, `keepAliveTimeout`, or `permitKeepAliveWithoutCalls` call. Without HTTP/2 PING-based keepalive, the server has no out-of-band mechanism to probe whether the remote peer is still reachable.

**Infinite live-listening pipeline — `TopicMessageServiceImpl.java` lines 63–73:**
```java
Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));
Flux<TopicMessage> flux = historical
        .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
```
`incomingMessages` calls `topicListener.listen(newFilter)`, which in `PollingTopicListener` uses `RepeatSpec.times(Long.MAX_VALUE)` — an infinite polling loop. `pastEndTime` returns `Flux.never()` when no `endTime` is set, so there is no server-side timeout on the live phase. The pipeline runs forever unless `disposable::dispose` is called, which never happens during a partition.

**Exploit flow:**
1. Attacker opens many TCP connections to the gRPC port (no authentication required).
2. Each connection issues up to `maxConcurrentCallsPerConnection` (default 5) `subscribeTopic` calls with no `endTime` and no `limit`.
3. Attacker uses `iptables -A OUTPUT -p tcp --dport <grpc_port> -j DROP` (or equivalent) to silently drop outgoing packets — no FIN/RST is sent.
4. From the server's perspective the TCP connections remain `ESTABLISHED`. Netty never fires `onCancel`. `disposable::dispose` is never called.
5. Each zombie subscription keeps a `PollingTopicListener` loop alive, issuing `topicMessageRepository.findByFilter()` queries every `interval` ms against the database indefinitely.
6. `subscriberCount` gauge grows without bound; `boundedElastic` scheduler threads are consumed; DB connection pool is exhausted.

**Why existing checks are insufficient:**
- `setOnCancelHandler` — correct pattern, but depends entirely on the transport layer delivering the cancellation signal, which a partition prevents.
- `maxConcurrentCallsPerConnection = 5` — limits calls per connection, but there is no limit on the number of connections, so an attacker opens N connections × 5 calls = 5N zombie pipelines.
- `retrieverProperties.getTimeout()` — applies only to the historical retrieval `Flux` (line 59 of `PollingTopicMessageRetriever`), not to the live `topicListener.listen()` phase.
- The one-shot safety check (`Mono.delay(1s)`) fires once and does not terminate the pipeline.

### Impact Explanation
Each zombie subscription holds a `boundedElastic` scheduler thread slot and issues periodic SQL queries. With enough connections the database connection pool is exhausted, causing all legitimate subscribers and internal importer operations to fail. The `subscriberCount` metric grows monotonically, masking the true subscriber count. Severity is **High** (availability impact, no authentication required).

### Likelihood Explanation
Any unprivileged client with network access to the gRPC port can trigger this. The partition can be simulated locally with `iptables` or by simply killing the network interface without a graceful TCP close. No special knowledge of the application is needed beyond knowing the gRPC endpoint. The attack is repeatable and scalable — each new connection multiplies the resource drain. Default OS TCP keepalive timers (typically 2 hours) mean zombie connections persist for hours without intervention.

### Recommendation
1. **Configure gRPC server-side keepalive** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.permitKeepAliveWithoutCalls(true);
   ```
   This causes Netty to send HTTP/2 PING frames; if the peer does not respond within the timeout, the connection is closed and `onCancel` fires, triggering `disposable::dispose`.
2. **Add a connection-level idle timeout** (`serverBuilder.maxConnectionIdle(...)`) to reclaim connections that have been silent for too long.
3. **Expose `maxConcurrentCallsPerConnection` as a configurable property** and add a global concurrent-subscription cap to bound total resource consumption.
4. **Add a `NettyProperties` field** for keepalive settings so they can be tuned per deployment without code changes.
