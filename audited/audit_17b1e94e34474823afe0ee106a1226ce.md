### Title
Thread Pool Exhaustion via Unbounded `findStorageBatch()` IN Clause Combined with Short Cache TTL and Pre-Query-Only Timeout Guard

### Summary
`findStorage()` in `ContractStateServiceImpl` delegates to `findStorageBatch()`, which issues a single SQL `IN` clause query containing **all** cached slot keys for a contract (up to 1500 entries). Because the `contractState` cache expires after only 2 seconds (`expireAfterWrite=2s`) and the `StatementInspector` timeout guard fires only **before** each query (not during execution), an unprivileged attacker can craft requests that repeatedly re-trigger expensive 1500-slot batch queries throughout the full 10-second `requestTimeout`, holding Tomcat worker threads for the entire duration and exhausting the thread pool.

### Finding Description

**Exact code path:**

`findStorage()` at [1](#0-0)  checks the `contractState` cache first; on a miss it calls `findStorageBatch()`.

`findStorageBatch()` at [2](#0-1)  reads **the entire keyset** of the per-contract Caffeine cache (`cachedSlotKeys`) and passes it as the `slots` list to `contractStateRepository.findStorageBatch()`. The per-contract cache is bounded at `maximumSize=1500` by default: [3](#0-2) 

The resulting SQL is an unbounded `IN` clause: [4](#0-3) 

The `contractState` cache TTL is only 2 seconds: [5](#0-4) 

The timeout guard (`StatementInspector`) checks elapsed time **before** each SQL statement, not during execution: [6](#0-5) 

The `requestTimeout` is 10,000 ms: [7](#0-6) 

**Root cause and failed assumption:**

The design assumes that the `contractState` cache will absorb repeated SLOAD lookups within a single request. This assumption breaks when the cache TTL (2s) is shorter than the request timeout (10s): cache entries written at the start of a request expire mid-execution, causing subsequent SLOADs to miss the cache and re-trigger `findStorageBatch()` with the full 1500-slot IN clause. The `StatementInspector` guard only prevents a new query from **starting** after 10s; it cannot interrupt a query already in flight, and it permits queries to start at t=9999ms (which then run for up to 3s more per the DB `statementTimeout`).

**Exploit flow:**

1. **Phase 1 – Cache poisoning (one-time setup):** Send ~1500 `eth_call` requests to the same contract address, each accessing a distinct storage slot. This fills the `slotsPerContract` Caffeine cache for that contract with 1500 unique slot keys. The cache persists for 5 minutes (`expireAfterAccess=5m`): [3](#0-2) 

2. **Phase 2 – Thread hold loop:** Send a concurrent flood of `eth_call` requests targeting a contract with many SLOAD opcodes (up to 15M gas limit allows thousands of SLOADs). Each request executes the following loop until `requestTimeout`:
   - SLOAD → `findStorage()` → `contractState` cache miss → `findStorageBatch()` → DB query with 1500 slots (takes T ms)
   - Cache populated; subsequent SLOADs hit cache
   - After 2s, cache entries expire
   - Next SLOAD → cache miss → another 1500-slot query
   - Repeat until elapsed ≥ 10s

   With T ≈ 500ms per batch query, a single request triggers ~5 batch queries and holds its thread for the full 10s.

3. **Thread pool exhaustion:** Tomcat's default thread pool is 200 threads. With each request holding a thread for ~10s, only 20 concurrent requests/second are needed to saturate all threads. The rate limiter allows 500 RPS: [8](#0-7)  — 25× more than needed to exhaust the pool.

### Impact Explanation

Once all Tomcat worker threads are held, the node stops accepting new web3 requests. New connections queue up and then are rejected. This constitutes a complete denial of service for the web3 API on the targeted node. Because the attack is cheap (no authentication, no gas payment, no on-chain transaction), it can be sustained indefinitely. If multiple mirror nodes are targeted simultaneously, 30%+ of the network's web3 processing capacity can be taken offline, matching the stated severity scope.

### Likelihood Explanation

The attack requires no privileges — any caller of the public `eth_call` or `eth_estimateGas` endpoint can trigger it. Phase 1 (cache poisoning) requires ~1500 sequential requests, which is trivial. Phase 2 requires only ~20 concurrent requests/second, well within the 500 RPS rate limit. The attack is fully repeatable and can be automated with a simple script. No cryptographic material, private keys, or on-chain funds are needed.

### Recommendation

1. **Cap the IN clause size:** In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 100) before issuing the query. Slots beyond the cap should fall back to individual `findStorage()` queries or be deferred.

2. **Align cache TTL with request timeout:** Increase `contractState` cache TTL to at least equal `requestTimeout` (10s) so that cache entries populated at the start of a request remain valid for its entire duration, preventing repeated re-queries.

3. **Interrupt in-flight queries:** Replace the pre-query `StatementInspector` check with a mechanism that actively cancels running JDBC statements when the timeout is reached (e.g., `Statement.setQueryTimeout()` or a `Future`-based cancellation).

4. **Per-IP concurrency limiting:** Add a per-source-IP concurrent request limit to prevent a single attacker from holding a disproportionate share of the thread pool.

```
# Phase 2: Flood with concurrent requests that each trigger repeated findStorageBatch() calls
# Use a contract that performs hundreds of SLOADs across the 1500 pre-cached slots

  curl -s -X POST http://<node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"<target_contract>\",\"data\":\"<many_sload_function>\",\"gas\":15000000}" &
done
wait
# All 200 Tomcat threads are now held; new requests receive connection refused or timeout
```

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
