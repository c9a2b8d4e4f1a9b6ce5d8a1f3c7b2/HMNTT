### Title
Soft-Deleted Account Data Exposed via `entityId` Lookup in GraphQL `account()` Query

### Summary
The `account()` query in `AccountController.java` routes `entityId`-based lookups through `EntityServiceImpl.getByIdAndType()`, which calls the inherited Spring Data `CrudRepository.findById()`. Unlike `findByAlias` and `findByEvmAddress`, `findById` carries no `deleted is not true` SQL predicate, so soft-deleted accounts are returned in full to any unauthenticated caller who supplies the account's numeric ID.

### Finding Description
**Exact code path:**

`AccountController.account()` (line 41–44) → `EntityServiceImpl.getByIdAndType()` (line 25) → `entityRepository.findById(entityId.getId())` (inherited `CrudRepository<Entity,Long>`)

```
// AccountController.java L41-44
if (entityId != null) {
    return entityService
            .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
            .map(accountMapper::map);
}
```

```
// EntityServiceImpl.java L24-26
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
}
```

The GraphQL-module `EntityRepository` (lines 13–17) defines only two custom queries, both with `deleted is not true`:

```sql
-- findByAlias
select * from entity where alias = ?1 and deleted is not true
-- findByEvmAddress
select * from entity where evm_address = ?1 and deleted is not true
```

`findById` is not overridden; it resolves to Spring Data's default `SELECT * FROM entity WHERE id = ?` with no deletion predicate. The only post-fetch filter is `e.getType() == type`, which checks entity type but not deletion status.

**Secondary path:** `getByEvmAddressAndType()` (line 38) also calls `entityRepository.findById(buffer.getLong())` for long-zero EVM addresses, carrying the same missing deletion filter.

**Root cause:** Inconsistent deletion guard — alias and EVM-address paths enforce `deleted is not true` at the SQL layer; the ID path does not enforce it at any layer.

### Impact Explanation
Any caller who knows (or enumerates) a deleted account's numeric ID can retrieve its full `Entity` record — including cryptographic keys, memo field, expiration timestamp, auto-renew settings, and proxy account ID — via a single unauthenticated GraphQL query. Hedera account IDs are sequential and publicly observable on-chain, making enumeration trivial. This constitutes an information-disclosure vulnerability: data that the network considers logically deleted is exposed through the mirror-node API.

### Likelihood Explanation
No authentication or authorization is required to call the GraphQL `account` query. Account IDs are public and sequential (e.g., `0.0.12345`). An attacker needs only to iterate numeric IDs and observe which ones return data with `deleted: true` in the response. The exploit is fully repeatable, requires no special privileges, and can be scripted in minutes.

### Recommendation
Override `findById` in the GraphQL `EntityRepository` with a deletion-aware query, or add an explicit deletion check in `getByIdAndType`:

**Option A – Repository layer (preferred, consistent with alias/evmAddress paths):**
```java
// graphql/.../repository/EntityRepository.java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```
Then update `EntityServiceImpl.getByIdAndType` to call `findByIdAndNotDeleted`.

**Option B – Service layer:**
```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId())
            .filter(e -> e.getType() == type)
            .filter(e -> !Boolean.TRUE.equals(e.getDeleted()));
}
```
Apply the same fix to the `findById` call inside `getByEvmAddressAndType` (line 38).

### Proof of Concept
1. Identify a soft-deleted Hedera account (e.g., account `0.0.9999`) — visible on-chain or via the REST API with `deleted=true`.
2. Send the following GraphQL query to the mirror-node GraphQL endpoint (no credentials required):
```graphql
query {
  account(input: { entityId: { shard: 0, realm: 0, num: 9999 } }) {
    id
    deleted
    key
    memo
    expirationTimestamp
  }
}
```
3. Observe that the response returns the full account record including `deleted: true` and all sensitive fields, whereas the same account queried by alias or EVM address returns `null`. [1](#0-0) [2](#0-1) [3](#0-2)
