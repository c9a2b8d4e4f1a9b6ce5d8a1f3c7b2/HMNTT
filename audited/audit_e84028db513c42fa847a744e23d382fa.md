### Title
`BlockType.of("0x0")` Bypasses `EARLIEST` Identity Check, Routing to `findByIndex(0)` Instead of `findEarliest()`

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect `BlockType.EARLIEST`, but `BlockType.of("0x0")` constructs a brand-new `BlockType("0", 0L)` instance whose reference is never `EARLIEST`. The check silently falls through to `findByIndex(0)`, which on Hedera mainnet (where the genesis block index is not 0) returns `Optional.empty()`, causing any `eth_call` or `eth_getBalance`-style request specifying block `"0x0"` to fail with a block-not-found error instead of executing against the earliest block.

### Finding Description

**Root cause — `BlockType.of("0x0")` path:**

In `BlockType.java` line 73-79, the regex group `GROUP_HEX_NUM` (`0x([0-9a-f]{1,16})`) matches `"0x0"`, extracting `hexNum = "0"` and returning:
```java
new BlockType("0", Long.parseLong("0", 16))   // → new BlockType("0", 0L)
```
This is a **new heap object**. `BlockType.EARLIEST` (line 29) is `new BlockType("earliest", 0L)` — a different object with a different `name` field.

**Failed check in `RecordFileServiceImpl.findByBlockType()` (lines 20-28):**
```java
if (block == BlockType.EARLIEST) {          // FALSE — different reference
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST) {     // FALSE
    return recordFileRepository.findLatest();
} else if (block.isHash()) {                // FALSE — number == 0, not -1
    return recordFileRepository.findByHash(block.name());
}
return recordFileRepository.findByIndex(block.number());  // called with 0
```

**Repository divergence:**
- `findEarliest()` (line 24-25 of `RecordFileRepository.java`): `select * from record_file order by index asc limit 1` — returns the record file with the **lowest stored index**, whatever that is.
- `findByIndex(0)` (line 27-29): `select r from RecordFile r where r.index = ?1` with `?1 = 0` — returns a record file only if one exists with **exactly index 0**.

On Hedera mainnet the genesis record file index is not 0 (the `updateIndex` bulk-shift method in the importer confirms indices are not anchored to 0). `findByIndex(0)` therefore returns `Optional.empty()`.

### Impact Explanation
Any unprivileged caller submitting an `eth_call`, `eth_estimateGas`, or similar JSON-RPC request with `block = "0x0"` receives a block-not-found failure instead of a valid response against the earliest block. This is unintended smart contract execution behavior: a semantically valid Ethereum block tag (`0x0` = genesis) is silently misrouted, breaking EVM compatibility for tooling and contracts that rely on querying genesis-block state. No funds are directly at risk, consistent with the Medium scope.

### Likelihood Explanation
No authentication or special privilege is required. Any external user or automated tool (e.g., Hardhat, ethers.js, web3.py) that passes `blockTag: "0x0"` triggers the bug deterministically and repeatably. The hex-zero form is a standard Ethereum block reference.

### Recommendation
Replace the reference-equality (`==`) checks with semantic checks. The simplest correct fix is to return `EARLIEST` from `BlockType.of()` when the resolved numeric value is 0 and the input is a hex/decimal number (mirroring how `blockTypeForTag` returns the singleton for `"earliest"`):

```java
// In BlockType.of(), after parsing hexNum:
long num = Long.parseLong(hexNum, 16);
if (num == 0L) return EARLIEST;          // treat 0x0 as earliest
return new BlockType(hexNum, num);
```

Alternatively, change `findByBlockType()` to use value-based comparison:
```java
if (block.name().equals(BlockType.EARLIEST.name()) || (!block.isHash() && block.number() == 0)) {
    return recordFileRepository.findEarliest();
}
```

### Proof of Concept
1. Deploy the mirror-node web3 module against a Hedera mainnet (or testnet) database where the genesis record file has `index > 0`.
2. Send: `curl -X POST http://localhost:8545 -d '{"jsonrpc":"2.0","method":"eth_call","params":[{...}, "0x0"],"id":1}'`
3. Observe: response returns a block-not-found error.
4. Send the same request with `"earliest"` instead of `"0x0"`.
5. Observe: response succeeds and returns the genesis block state.
6. Root cause confirmed: `BlockType.of("0x0")` (line 76, `BlockType.java`) returns `new BlockType("0", 0L)`, which fails the `== BlockType.EARLIEST` check at line 20 of `RecordFileServiceImpl.java`, falling through to `findByIndex(0)` at line 28, which returns empty.
