### Title
Silent Empty-Result Response on Conflicting `credit_type` Values Misleads API Consumers

### Summary
When an unprivileged user submits both `type=credit` and `type=debit` in the same `/api/v1/transactions` request, `extractSqlFromTransactionsRequest()` returns `null` due to the conflicting filter combination. `getTransactionTimestamps()` silently converts this `null` into `{rows: []}`, causing the API to return HTTP 200 with an empty `transactions` array — indistinguishable from a legitimate "no results found" response.

### Finding Description
**Code path:**

In `rest/transactions.js`, `extractSqlFromTransactionsRequest()` (lines 397–402) detects conflicting `CREDIT_TYPE` values and returns `null`:

```js
case constants.filterKeys.CREDIT_TYPE:
  if (lastCreditDebitValue !== null && lastCreditDebitValue !== value) {
    return null;   // ← silent null on credit+debit conflict
  }
  lastCreditDebitValue = value;
  break;
``` [1](#0-0) 

`getTransactionTimestamps()` (lines 456–459) treats `null` as an empty result set with no error:

```js
const result = extractSqlFromTransactionsRequest(filters);
if (result === null) {
  return {rows: []};   // ← no 400, no log, just empty rows
}
``` [2](#0-1) 

**Root cause — failed assumption in the validation layer:**

`buildAndValidateFilters()` in `rest/utils.js` validates each filter value in isolation. For `CREDIT_TYPE`, the check is:

```js
case constants.filterKeys.CREDIT_TYPE:
  ret = isValidValueIgnoreCase(val, Object.values(constants.cryptoTransferType));
  break;
``` [3](#0-2) 

Both `credit` and `debit` are individually valid, so the combination `type=credit&type=debit` passes validation without throwing `InvalidArgumentError`. The contradiction is only discovered deep inside `extractSqlFromTransactionsRequest`, which returns `null` instead of propagating an error. [4](#0-3) 

The unit test at line 936–942 of `rest/__tests__/transactions.test.js` explicitly confirms this behavior is present (the test asserts `toBeNull()`), showing the silent-null path is a known, untreated code path: [5](#0-4) 

### Impact Explanation
Any API consumer (monitoring tool, wallet, explorer, audit system) that queries `/api/v1/transactions` with conflicting `type` values receives HTTP 200 `{"transactions": [], "links": {"next": null}}`. This is semantically identical to a valid query that returned zero results. The consumer cannot distinguish between "no transactions exist" and "your query was self-contradictory." Applications that rely on an empty result to conclude "no activity occurred" may make incorrect downstream decisions (e.g., skipping reconciliation, reporting zero balance changes). The impact is data-integrity misleading at the API consumer layer, with no on-chain or economic effect.

### Likelihood Explanation
The trigger requires zero privileges — any unauthenticated HTTP client can reproduce it with a single GET request. The parameter combination is trivially discoverable by fuzzing or by reading the API documentation. It is fully repeatable and deterministic.

### Recommendation
Detect the contradictory combination at the validation boundary, before it reaches `extractSqlFromTransactionsRequest`. In `buildAndValidateFilters` (or a dedicated `filterDependencyChecker`), check whether the parsed filters contain more than one distinct value for `CREDIT_TYPE` and throw `InvalidArgumentError` with a descriptive message (e.g., `"type cannot be both credit and debit"`). This ensures the caller receives HTTP 400 instead of a misleading HTTP 200 with empty results.

Alternatively, change `getTransactionTimestamps` to propagate the contradiction as a thrown error rather than silently returning `{rows: []}`.

### Proof of Concept
**Precondition:** The mirror node REST API is running and the database contains at least one transaction.

**Request:**
```
GET /api/v1/transactions?type=credit&type=debit
```

**Actual response (HTTP 200):**
```json
{
  "transactions": [],
  "links": { "next": null }
}
```

**Expected response (HTTP 400):**
```json
{
  "_status": {
    "messages": [{ "message": "type cannot be both credit and debit" }]
  }
}
```

The empty 200 response is returned regardless of how many transactions exist in the database, because `extractSqlFromTransactionsRequest` returns `null` at line 399 and `getTransactionTimestamps` converts it to `{rows: []}` at line 458 without raising any error.
