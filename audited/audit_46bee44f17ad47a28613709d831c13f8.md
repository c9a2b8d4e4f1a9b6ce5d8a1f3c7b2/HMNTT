### Title
Unauthenticated, Uncached Health Endpoint Enables Amplification DoS via Outbound HTTP Flooding

### Summary
`SubscriberHealthIndicator.health()` is exposed via Spring Boot Actuator with no authentication, no rate limiting, and no response caching. Every invocation unconditionally issues a live outbound HTTP request to the external REST API (`/network/stake`). An unprivileged attacker who can reach port 8082 can flood the health endpoint, exhausting the WebClient connection pool and amplifying load against the downstream REST API.

### Finding Description

**Code path:**

`SubscriberHealthIndicator.health()` (lines 59–64) always calls `restNetworkStakeHealth()` (lines 93–122), which calls `RestApiClient.getNetworkStakeStatusCode()`:

```java
// SubscriberHealthIndicator.java:59-64
public Mono<Health> health() {
    return restNetworkStakeHealth()          // <-- live HTTP call, every time
            .flatMap(health -> ...)
            .doOnNext(this::recordHealthMetric);
}
```

```java
// RestApiClient.java:67-69
public Mono<HttpStatusCode> getNetworkStakeStatusCode() {
    return webClientRestJava.get().uri("/network/stake").exchangeToMono(r -> Mono.just(r.statusCode()));
}
```

**Root cause — no caching:** Compare with `ReleaseHealthIndicator`, which explicitly caches its result:

```java
// ReleaseHealthIndicator.java:82
.cache(properties.getCacheExpiry(), Schedulers.newSingle("helmrelease-health-cache"));
```

`SubscriberHealthIndicator` has no equivalent. Every subscriber to `health()` triggers a fresh outbound TCP connection to the REST API, held open for up to 5 seconds (the configured timeout at line 107).

**No authentication or rate limiting:** No `SecurityConfig` exists in the monitor module. The Dockerfile exposes port 8082 directly. Spring Boot Actuator health endpoints are unauthenticated by default. No `WebFilter` or gateway-level rate limiter is present — `LoggingFilter` only logs requests, it does not throttle them.

**Exploit flow:**
1. Attacker sends `GET /actuator/health` (or the composite `/actuator/health/subscriber`) at high frequency.
2. Each request causes `health()` to be subscribed, which calls `getNetworkStakeStatusCode()`, opening a new outbound HTTP connection to the REST API.
3. With a 5-second timeout per call and thousands of requests per second, the WebClient connection pool (bounded by default in Reactor Netty) fills up.
4. New requests queue indefinitely or fail with connection-pool-exhausted errors, degrading the monitor service.
5. Simultaneously, the downstream REST API (`/network/stake`) receives a proportional flood of requests — a 1:1 amplification attack.

### Impact Explanation
- **Connection pool exhaustion:** Reactor Netty's default `ConnectionProvider` has a bounded pending-acquire queue. Under sustained flood, the pool saturates, causing all subsequent health checks (including legitimate liveness probes used by Kubernetes, line 9 of Dockerfile) to fail or time out.
- **Downstream amplification:** Every attacker request generates one outbound request to the REST API. If the REST API is shared infrastructure, this degrades it for all consumers.
- **Kubernetes liveness failure:** The Dockerfile health check (`curl -f http://localhost:8082/actuator/health/liveness`) will fail if the connection pool is exhausted, causing Kubernetes to restart the pod in a crash loop — a complete availability loss.

Severity: **High** (availability impact on the monitor and downstream REST API; no authentication required).

### Likelihood Explanation
Any network-reachable client with no credentials can trigger this. The monitor service is an internal tool but is exposed on a plain HTTP port (8082) with no TLS or auth layer visible in the codebase. In misconfigured or development deployments without strict Kubernetes `NetworkPolicy`, this is trivially reachable. The attack requires only a basic HTTP flood tool (e.g., `wrk`, `ab`, `curl` in a loop) and is fully repeatable.

### Recommendation
1. **Add response caching** to `restNetworkStakeHealth()`, mirroring `ReleaseHealthIndicator`:
   ```java
   // Cache result for e.g. 10 seconds to absorb burst polling
   .cache(Duration.ofSeconds(10))
   ```
2. **Restrict actuator exposure** via Spring Security or `management.endpoints.web.exposure` configuration, requiring authentication for non-liveness endpoints.
3. **Configure connection pool limits** explicitly in `WebClient` with a bounded `ConnectionProvider` and a short pending-acquire timeout to fail fast rather than queue indefinitely.
4. **Add rate limiting** at the ingress/gateway layer for the actuator path.

### Proof of Concept
```bash
# Flood the health endpoint from any network-reachable host
wrk -t 10 -c 500 -d 60s http://<monitor-host>:8082/actuator/health

# Observe: REST API /network/stake receives ~500 concurrent requests
# Observe: monitor liveness probe begins failing (Kubernetes restarts pod)
# Observe: WebClient logs "Pending acquire queue has reached its maximum size"
```

```java
    private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })
                .timeout(Duration.ofSeconds(5))
                .onErrorResume(e -> {
                    var status = Status.UNKNOWN;
                    // Connection issue can be caused by database being down, since the rest API service will become
                    // unavailable eventually
                    var rootCause = ExceptionUtils.getRootCause(e);
                    if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
                        status = Status.DOWN;
                    }

                    var statusMessage =
                            String.format("Network stake status is %s with error: %s", status, e.getMessage());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                });
    }
```
