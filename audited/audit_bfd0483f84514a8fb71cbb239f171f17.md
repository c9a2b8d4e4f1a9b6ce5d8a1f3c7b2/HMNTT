### Title
Shared `boundedElastic` Scheduler for Polling and Timeout Allows Timeout Bypass Under Scheduler Saturation

### Summary
`PollingTopicMessageRetriever.retrieve()` applies `.timeout(retrieverProperties.getTimeout(), scheduler)` using the same `scheduler` instance (`Schedulers.boundedElastic()`) that drives the `repeatWhen` polling loop. Because `Schedulers.boundedElastic()` returns the **global** shared scheduler (also consumed by `PollingTopicListener`, `SharedPollingTopicListener`, `SharedTopicListener`, `TopicMessageServiceImpl`, and `NetworkServiceImpl`), an unprivileged user who opens enough concurrent subscriptions can saturate the scheduler's thread pool, causing the timeout signal task to queue behind pending polling tasks and fire significantly later than the configured duration. This allows subscriptions to persist well beyond the intended 60-second ceiling, exhausting downstream resources.

### Finding Description
**Exact code path:**

- `PollingTopicMessageRetriever.java` line 41: `scheduler = Schedulers.boundedElastic();` — assigns the **global** shared `BoundedElasticScheduler` singleton to the instance field.
- Line 55: `.withScheduler(scheduler)` — every `repeatWhen` polling tick is dispatched onto this scheduler.
- Line 59: `.timeout(retrieverProperties.getTimeout(), scheduler)` — the timeout cancellation task is also scheduled on the **same** scheduler.

**Root cause:** Reactor's `timeout(Duration, Scheduler)` operator works by scheduling a lightweight cancellation task on the provided scheduler after the given duration. If that scheduler's thread pool is fully occupied (all threads blocked on in-flight DB polling calls), the cancellation task sits in the work queue and does not execute until a thread becomes free. The failed assumption is that the timeout mechanism is independent of the load it is meant to bound.

**Why the global scheduler amplifies the risk:** `Schedulers.boundedElastic()` is not a fresh per-instance pool; it is the JVM-wide singleton. Every other component in the gRPC service (`PollingTopicListener`, `SharedPollingTopicListener`, `SharedTopicListener`, `TopicMessageServiceImpl` safety-check, `NetworkServiceImpl`) also submits work to this same pool, so the effective saturation threshold is lower than the retriever's own concurrency alone.

**Exploit flow:**
1. Attacker opens N concurrent `subscribeTopic` gRPC streams (no rate-limiting or connection cap is present in the visible code).
2. Each stream triggers `retrieve(filter, true)` → `repeatWhen(...).withScheduler(scheduler)` → continuous DB polling tasks submitted to the global `boundedElastic` pool.
3. The pool's thread count (`10 × CPU_count`, default ~80 on an 8-core host) is exhausted; new tasks queue.
4. The `.timeout(60s, scheduler)` cancellation task for each subscription is also queued behind the polling backlog.
5. Subscriptions remain alive for `60s + (queue_depth × avg_task_duration / thread_count)` — potentially many minutes.

**Existing check and why it fails:** The sole enforcement mechanism is `.timeout(retrieverProperties.getTimeout(), scheduler)` at line 59. Because the enforcer and the load source share the same execution resource, the enforcer is self-defeating under the exact conditions it is meant to handle.

### Impact Explanation
Subscriptions that outlive their timeout hold open database connections (via `topicMessageRepository.findByFilter`), consume scheduler threads, and accumulate heap for `PollingContext` state. Sustained exploitation exhausts the DB connection pool, causing all new and existing queries to block, which cascades into a full gRPC service outage. No funds are directly at risk, but availability of the mirror node's HCS topic subscription API is fully compromised. Severity: Medium (DoS, no data integrity or confidentiality impact).

### Likelihood Explanation
No authentication, rate limiting, or concurrent-subscription cap is visible in the code path from `subscribeTopic` through `retrieve`. Any client with network access to the gRPC port can open arbitrarily many streams. The attack is trivially scriptable (open N gRPC streams with a `startTime` far in the past to maximize polling iterations), requires no special knowledge, and is repeatable. Likelihood: High given the absence of admission controls.

### Recommendation
1. **Decouple the timeout scheduler:** Pass a dedicated, independent scheduler (e.g., `Schedulers.parallel()` or a separate `newBoundedElastic`) to the `timeout` operator so that polling saturation cannot delay cancellation:
   ```java
   private final Scheduler timeoutScheduler = Schedulers.newSingle("retriever-timeout");
   // ...
   .timeout(retrieverProperties.getTimeout(), timeoutScheduler)
   ```
2. **Limit concurrent subscriptions:** Enforce a server-side cap on simultaneous `subscribeTopic` streams per client or globally before work reaches the scheduler.
3. **Use `Schedulers.parallel()` for timeout:** The timeout task is non-blocking and CPU-bound; it does not belong on `boundedElastic` at all.
4. **Add admission control:** Introduce a semaphore or `Flux.limitRate` guard in `TopicMessageServiceImpl.subscribeTopic` to bound the number of concurrent historical retrievals.

### Proof of Concept
```python
import grpc, threading, time
# Assume generated stubs for ConsensusService
from proto import consensus_service_pb2_grpc, consensus_topic_query_pb2

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

def open_subscription():
    req = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=...,          # any valid topic
        consensusStartTime=0, # far past → maximises polling iterations
    )
    for _ in stub.subscribeTopic(req):
        pass  # drain silently

# Step 1: Flood with concurrent subscriptions to saturate boundedElastic
threads = [threading.Thread(target=open_subscription) for _ in range(300)]
for t in threads: t.start()

# Step 2: After 60 s (configured timeout), observe via metrics or logs
# that subscriptions are still active well past the timeout window,
# and that new DB queries begin timing out (connection pool exhausted).
time.sleep(120)
# Expected: subscriptions still alive; DB pool errors in server logs.
```
