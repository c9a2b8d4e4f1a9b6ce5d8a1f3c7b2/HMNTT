### Title
Unauthenticated N×3 Database Query Amplification via `/api/v1/topics/{id}` with No Rate Limiting

### Summary
`TopicController.getTopic()` unconditionally issues three sequential, uncached database queries per request — one each to `topicService`, `entityService`, and `customFeeService`. The rest-java module has no rate limiting on this endpoint and no caching on any of the three repositories involved. Any unauthenticated caller with knowledge of valid topic IDs can multiply database load by a factor of three relative to a single-query implementation.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 32–37:

```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());         // DB query 1: topic table
    var entity = entityService.findById(id.id());       // DB query 2: entity table
    var customFee = customFeeService.findById(id.id()); // DB query 3: custom_fee table
    return topicMapper.map(customFee, entity, topic);
}
```

**Root cause:**

Each call delegates to a plain Spring Data `CrudRepository.findById()` with no caching:

- `TopicServiceImpl` → `TopicRepository` (plain `CrudRepository<Topic, Long>`)
- `EntityServiceImpl` → `EntityRepository` (no `@Cacheable`, unlike the grpc module's version which does cache)
- `CustomFeeServiceImpl` → `CustomFeeRepository` (plain `CrudRepository<CustomFee, Long>`)

None of these repositories carry a `@Cacheable` annotation in the rest-java module. The rest-java `EntityRepository` explicitly omits the caching present in the grpc module's equivalent.

**No rate limiting exists in rest-java for this endpoint.** The `ThrottleConfiguration` and `ThrottleManagerImpl` (bucket4j-based) live entirely in the `web3` module and are not imported by `RestJavaApplication`. The only middleware present in the rest-java module is `LoggingFilter` and `MetricsFilter` — neither enforces request rate limits.

**Exploit flow:**

1. Attacker discovers valid topic IDs (they are sequential integers on a public ledger, trivially enumerable).
2. Attacker sends a high-rate stream of `GET /api/v1/topics/{valid_id}` requests.
3. Each request causes exactly 3 synchronous DB round-trips with no short-circuit or cache hit.
4. Database connection pool and query throughput are consumed at 3× the rate of a single-query design.

**Why existing checks fail:**

- `topicService.findById` throws `EntityNotFoundException` for unknown IDs, which would short-circuit queries 2 and 3 — but valid IDs (the attacker's target) complete all three queries.
- The HTTP `Cache-Control: public, max-age=5` response header only instructs downstream HTTP caches (CDN/browser); it does not cache at the application or database layer.
- No per-IP or global request-rate enforcement exists in the rest-java application.

### Impact Explanation

Under sustained request load with valid topic IDs, the database receives 3× the query volume compared to a joined-query implementation. This degrades response latency for all users of the mirror node (not just the topics endpoint), as the database connection pool is shared. In a worst-case scenario, the database becomes a bottleneck, causing cascading timeouts across all mirror node API endpoints. The impact is service degradation (griefing), not data corruption or financial loss.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only publicly available information (valid topic IDs). The endpoint is reachable from the public internet. Tooling to send thousands of HTTP requests per second is freely available (`wrk`, `ab`, `hey`). The attacker does not need to sustain the attack for long to cause measurable degradation; a burst of a few hundred requests per second is sufficient to saturate a typical database connection pool.

### Recommendation

1. **Add rate limiting to rest-java**: Apply a per-IP or global request rate limiter (e.g., bucket4j, Spring Cloud Gateway, or an ingress-level rate limit) to the `/api/v1/topics/{id}` endpoint.
2. **Consolidate to a single joined query**: Replace the three separate `findById` calls with a single repository method that joins `topic`, `entity`, and `custom_fee` in one SQL statement, eliminating the 3× amplification.
3. **Add application-level caching**: Annotate the rest-java `EntityRepository.findById` and `CustomFeeRepository.findById` with `@Cacheable` (with a short TTL matching the existing `max-age=5` HTTP cache header) to absorb repeated lookups for the same topic ID.
