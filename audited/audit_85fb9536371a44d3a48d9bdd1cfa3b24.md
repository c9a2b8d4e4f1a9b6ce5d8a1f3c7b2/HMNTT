### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `/block/transaction` Requests

### Summary
The `BlockTransaction()` handler in `rosetta/app/services/block_service.go` performs at minimum two unbounded database queries per request (`FindByIdentifier` and `FindByHashInBlock`) plus additional per-operation `GetAccountAlias` queries in `updateOperationAccountAlias`, with no application-level rate limiting or concurrency control. The only protections are infrastructure-level Traefik middleware rules that are deployment-specific and trivially bypassable using multiple source IPs, allowing an unprivileged attacker to exhaust the DB connection pool (default: 100 connections) and render the service unavailable to legitimate users.

### Finding Description
**Code path:** `rosetta/app/services/block_service.go`, `BlockTransaction()`, lines 77–102.

```
block, err := s.FindByIdentifier(ctx, ...)       // DB query 1: block lookup
transaction, err := s.FindByHashInBlock(ctx, ...) // DB query 2: tx lookup
s.updateOperationAccountAlias(ctx, transaction)   // DB query 3+: per-operation alias lookup (cache miss)
```

Each call to `updateOperationAccountAlias` (lines 104–129) iterates over every operation in the transaction and calls `s.accountRepo.GetAccountAlias(ctx, accountId)` for each cache miss. On a cold cache or with varied account IDs, this adds N additional DB queries per request.

**Root cause:** The Go HTTP server in `main.go` (lines 217–227) applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which limit request rates or concurrency. There is no semaphore, worker pool, or `golang.org/x/time/rate` limiter anywhere in the application code.

**Why existing checks fail:**

The Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166) defines Traefik middleware with:
- `inFlightReq: amount: 5` scoped by `ipStrategy.depth: 1` (per source IP)
- `rateLimit: average: 10` scoped by `requestHost` (global per hostname, not per IP)

These protections fail because:
1. They are only active when deployed via the Helm chart with Traefik ingress enabled (`ingress.enabled: true`). Direct Docker or bare-metal deployments have zero protection.
2. The per-IP in-flight limit of 5 is bypassed trivially: an attacker with 20 source IPs can maintain 100 concurrent requests simultaneously, exactly saturating the default `maxOpenConnections: 100` pool (`rosetta/app/db/db.go`, line 33).
3. The `retry: attempts: 3` middleware entry (line 161–163) causes Traefik to re-issue failed requests up to 3 times, amplifying DB load by up to 4×.
4. `global.middleware: false` (line 95) can suppress the global middleware chain in parent-chart deployments.

The DB statement timeout of 20 seconds (`statementTimeout: 20`, `rosetta/app/db/client.go` lines 36–37) means each connection is held for up to 20 seconds under slow-query conditions, maximizing pool occupancy per attacker goroutine.

### Impact Explanation
With 100 DB connections exhausted, all subsequent legitimate requests block waiting for a connection. The Go HTTP server spawns an unbounded goroutine per connection (no `http.Server` connection limit is set in `main.go` lines 220–227), so the attacker can also accumulate thousands of goroutines, each holding memory (~8 KB stack + heap allocations for request/response objects). Combined DB connection starvation and goroutine accumulation causes complete service unavailability for the Rosetta API, which is the exchange integration surface for the network. For a network with ≥25% market cap, this directly disrupts exchange deposit/withdrawal operations.

### Likelihood Explanation
The attack requires no authentication, no special knowledge beyond a valid block identifier (publicly available from any block explorer), and no more than ~20 IP addresses (achievable via residential proxies or cloud VMs). The endpoint is publicly exposed by design (Rosetta is an exchange-facing API). The attack is repeatable and automatable with a simple HTTP flood tool. The only meaningful barrier is the Traefik middleware, which is bypassable as described above.

### Recommendation
1. **Application-level concurrency limit**: Add a semaphore or `golang.org/x/sync/semaphore`-based gate in `BlockTransaction()` (or as HTTP middleware) to cap concurrent in-flight requests regardless of deployment environment.
2. **Application-level rate limiter**: Integrate `golang.org/x/time/rate` per-IP token bucket in the middleware chain in `main.go`, independent of infrastructure.
3. **HTTP server connection limit**: Set `http.Server.ConnState` callback or use `netutil.LimitListener` to cap total accepted connections.
4. **Reduce per-request DB query count**: Cache block lookups (not just entity aliases) to avoid the `FindByIdentifier` query on every `/block/transaction` call.
5. **Lower `maxOpenConnections`**: Reduce from 100 to a value that leaves headroom for other services sharing the DB, and pair with a connection-wait timeout.

### Proof of Concept
```bash
# Obtain any valid block index and hash from a public explorer
BLOCK_INDEX=12345
BLOCK_HASH="<valid_hash>"
TX_HASH="<valid_tx_hash_in_block>"

# Launch 100 concurrent requests from 20 IPs (5 each, bypassing inFlightReq per-IP limit)
# Using GNU parallel or a tool like vegeta/hey:
for i in $(seq 1 100); do
  curl -s -X POST http://<rosetta-host>:5700/block/transaction \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"block_identifier\": {\"index\": $BLOCK_INDEX, \"hash\": \"$BLOCK_HASH\"},
      \"transaction_identifier\": {\"hash\": \"$TX_HASH\"}
    }" &
done
wait

# Expected result: DB connection pool (100 connections) saturated within seconds.
# Subsequent legitimate requests receive connection timeout errors or hang for up to 20s.
# Service metrics show hiero_mirror_rosetta_request_inflight spiking to pool limit.
```
