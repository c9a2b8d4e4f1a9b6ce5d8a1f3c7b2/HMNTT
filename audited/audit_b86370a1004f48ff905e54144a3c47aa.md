### Title
Misplaced `sync.Once` in `initGenesisRecordFile` Enables Thundering-Herd DB Connection Exhaustion

### Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` uses `sync.Once` only to protect the *assignment* of `br.genesisBlock`, but the expensive `selectGenesis` CTE query executes *before* `once.Do`. The early-exit guard at line 244 is also unsynchronized. Any number of concurrent goroutines can simultaneously pass the guard and each execute the expensive query, exhausting the DB connection pool and causing a non-network DoS.

### Finding Description
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` (lines 243–263):

```
Line 244: if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset { return nil }
Lines 252-255: db.Raw(selectGenesis, ...).First(&rb)   // expensive CTE query
Lines 257-259: br.once.Do(func() { br.genesisBlock = rb })
```

The root cause is that `sync.Once` wraps only the write at line 257, not the DB query at lines 252–255. The guard at line 244 is a plain unsynchronized struct-field read — no mutex, no atomic. Before genesis is initialized:

1. N goroutines all read `ConsensusStart == genesisConsensusStartUnset` (line 244) — all pass.
2. All N goroutines call `GetDbWithContext` and acquire a connection from the pool (line 248).
3. All N goroutines execute the `selectGenesis` CTE (lines 252–255) simultaneously.
4. All N goroutines reach `once.Do` — only one writes; the rest silently discard their result.

The `selectGenesis` query is a multi-table CTE joining `account_balance` and `record_file` with `ORDER BY` and `LIMIT 1` — non-trivial cost under concurrent load. The DB pool has a hard cap (`SetMaxOpenConns`, line 33 of `db.go`). Flooding this path saturates the pool, blocking all other DB operations across the service. [1](#0-0) [2](#0-1) 

### Impact Explanation
Every Rosetta read endpoint (`/network/status`, `/block`, `/block/transaction`, etc.) calls `initGenesisRecordFile`. During startup — or any time genesis is not yet in the DB — an attacker can hold the service in a permanently degraded state: all DB connections consumed by redundant `selectGenesis` queries, causing all other queries (including legitimate ones) to queue or time out. This is a complete availability loss for the Rosetta API without any authentication requirement. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The Rosetta API is a public HTTP API with no authentication on read endpoints. The window is the entire startup period (or any period where genesis data is absent from the DB). An attacker needs only to send concurrent HTTP requests to any Rosetta read endpoint — no credentials, no special knowledge. The attack is trivially repeatable with standard HTTP load tools (e.g., `ab`, `wrk`, `hey`). [5](#0-4) 

### Recommendation
Move the entire DB query inside `once.Do` so it executes at most once, regardless of concurrency:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()

        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })

    return initErr
}
```

This ensures only one goroutine ever executes the expensive query. The unsynchronized early-exit guard should also be replaced with an atomic or mutex-protected read to eliminate the data race. [6](#0-5) 

### Proof of Concept
**Preconditions:** Rosetta node is starting up; genesis has not yet been written to the DB (or DB is slow to respond).

**Steps:**
1. Start the Rosetta service before genesis data is available in the DB.
2. Send 200+ concurrent HTTP GET requests to `/network/status` (or any block endpoint):
   ```
   hey -n 500 -c 200 http://<rosetta-host>:5700/network/status
   ```
3. Each request reaches `RetrieveGenesis` → `initGenesisRecordFile`.
4. All 200 goroutines pass the unsynchronized guard at line 244.
5. All 200 goroutines acquire a DB connection and execute `selectGenesis` simultaneously.
6. With `MaxOpenConnections` defaulting to a finite value (e.g., 100), the pool is saturated.
7. All subsequent DB operations across the service block or time out — full DoS.

**Result:** The Rosetta service becomes unresponsive to all requests until the connection pool drains, which requires the flood to stop. [1](#0-0) [2](#0-1)
