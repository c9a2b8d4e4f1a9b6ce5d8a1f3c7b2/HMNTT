### Title
Unauthenticated HTTP Method Flooding Causes Prometheus Histogram Label Pollution in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` unconditionally records Prometheus metrics for every inbound HTTP request before any route or method validation occurs. When an unprivileged external user sends requests using HTTP methods not registered by the Rosetta router (e.g., `DELETE`, `PATCH`, `PUT`), the `RouteMatcher` (backed by gorilla/mux) returns an empty or default route string for the unmatched request. This empty `route=""` label — combined with arbitrary `method` label values — is written into `requestDurationHistogram`, `requestBytesHistogram`, `requestInflightGauge`, and `responseBytesHistogram`, polluting all four metric vectors with junk label combinations.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

`MetricsMiddleware` wraps the entire router (`next`) with `middleware.Instrument` from `github.com/weaveworks/common`. The `Instrument.Wrap` handler intercepts every request, calls `RouteMatcher.Match(r, &routeMatch)` to determine the `route` label, and then records observations into all four metric vectors — regardless of whether a route was actually matched.

**Root cause:**

The `next` handler is the gorilla/mux-based `server.NewRouter(...)` from `rosetta-sdk-go`. All Rosetta API endpoints are registered exclusively as `POST` routes. When a request arrives with an unregistered method (e.g., `DELETE /network/list`), gorilla/mux's `Match` returns `false` (or sets `MatchErr = ErrMethodNotAllowed` without populating `RouteMatch.Route`). The weaveworks `Instrument` middleware falls back to an empty string `""` for the `route` label. The `method` label is taken verbatim from `r.Method`, so any HTTP method string the attacker supplies becomes a label value.

**Failed assumption:**

The code assumes that only well-formed, method-matched requests will reach the metrics recording path. There is no guard that skips metric recording for unmatched routes or unregistered methods.

**Exploit flow:**

1. Attacker sends a flood of requests with arbitrary HTTP methods to any Rosetta endpoint:
   ```
   DELETE /network/list HTTP/1.1
   PATCH  /block HTTP/1.1
   PUT    /account/balance HTTP/1.1
   ```
2. gorilla/mux does not match any registered route for these methods.
3. `RouteMatcher.Match` returns false; `routeMatch.Route` is nil/empty.
4. `middleware.Instrument` records observations with `route=""` and `method="DELETE"` (etc.) into all four `HistogramVec`/`GaugeVec` instances.
5. Prometheus now contains time-series with label combinations `{method="DELETE", route=""}`, `{method="PATCH", route=""}`, etc., mixed into the same metric vectors as legitimate traffic.

**Why existing checks are insufficient:**

There are no existing checks. The middleware chain in `rosetta/main.go` (lines 217–219) applies `MetricsMiddleware` as the outermost layer, before any routing or method validation:

```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

No rate limiting, no method allowlist, and no "skip metrics on unmatched route" guard exists anywhere in the middleware chain.

### Impact Explanation

All four Prometheus metric vectors (`hiero_mirror_rosetta_request_duration`, `hiero_mirror_rosetta_request_bytes`, `hiero_mirror_rosetta_request_inflight`, `hiero_mirror_rosetta_response_bytes`) are polluted with junk label combinations. The Grafana dashboard queries (e.g., `sum(rate(hiero_mirror_rosetta_request_duration_sum{...}[$__rate_interval])) by (route)`) aggregate over the `route` label; a high-volume flood of `route=""` observations dilutes and obscures the per-route latency and throughput signals for legitimate Hashgraph history access patterns (block queries, transaction lookups, account balance reads). Operators lose visibility into real access patterns, degrading incident detection and SLA monitoring for the Hashgraph history API surface.

### Likelihood Explanation

No authentication, no special privileges, and no prior knowledge of the system are required. Any internet-accessible Rosetta endpoint can be targeted. The attack is trivially scriptable (`curl -X DELETE https://rosetta-host/network/list` in a loop), repeatable indefinitely, and produces measurable metric pollution with even modest request rates. The Rosetta port is typically publicly exposed by design (exchange integrations require it).

### Recommendation

1. **Filter unmatched routes before recording metrics:** In `MetricsMiddleware`, check whether the `RouteMatcher` actually matched a route before recording observations. If `routeMatch.Route == nil` or the route name is empty, skip (or record under a fixed `route="unmatched"` label with a separate, non-aggregated counter).
2. **Normalize the `method` label:** Allowlist only standard HTTP methods (`GET`, `POST`, `HEAD`, `OPTIONS`) before using `r.Method` as a label value to prevent arbitrary method strings from creating unbounded label cardinality.
3. **Apply rate limiting** at the ingress layer for unrecognized method/path combinations before they reach the metrics middleware.

### Proof of Concept

```bash
# Flood with unregistered HTTP methods — no auth required
for method in DELETE PATCH PUT TRACE CONNECT FOOBAR; do
  for i in $(seq 1 200); do
    curl -s -X "$method" http://<rosetta-host>:<port>/network/list \
         -H "Content-Type: application/json" -d '{}' &
  done
done
wait

# After ~30 seconds, scrape metrics and observe junk label combinations:
curl -s http://<rosetta-host>:<port>/metrics | \
  grep 'hiero_mirror_rosetta_request_duration' | \
  grep 'route=""'
