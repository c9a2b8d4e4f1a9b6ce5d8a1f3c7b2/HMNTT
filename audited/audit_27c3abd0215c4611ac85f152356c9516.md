### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Timestamp Range on `GET /api/v1/transactions`

### Summary
The `getTransactions` handler in `rest/transactions.js` calls `parseTimestampFilters` with `validateRange=false` and `allowOpenRange=true`, explicitly bypassing the `maxTimestampRange` (7-day) guard. Combined with `bindTimestampRange` being `false` by default, an unauthenticated attacker can issue requests with no timestamp bounds, triggering full-table scans that hold DB connections for up to the 20-second `statement_timeout`. With a default pool size of only 10 connections, 10 concurrent such requests saturate `global.pool`, blocking all subsequent transaction queries until the pool drains.

### Finding Description

**Code path:**

This creates a `pg` Pool with:
- `max: config.db.pool.maxConnections` → default **10**
- `statement_timeout: config.db.pool.statementTimeout` → default **20000 ms**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` → default **20000 ms** [2](#0-1) 

The handler for `GET /api/v1/transactions` is `getTransactions` in `rest/transactions.js`: [3](#0-2)

Mapping to the signature `(filters, filterRequired, allowNe, allowOpenRange, strictCheckOverride, validateRange)`:
- `filterRequired = false` — no timestamp required at all
- `allowOpenRange = true` — single-sided ranges (e.g., only `gte:0`) accepted
- `validateRange = false` — the `maxTimestampRange` (7-day) check is **explicitly skipped** [4](#0-3) 

The `validateRange` guard that would reject ranges exceeding 7 days is: [5](#0-4) 

This block is never reached because `validateRange=false` is passed.

The secondary mitigation, `bindTimestampRange`, is **disabled by default**: [6](#0-5) [7](#0-6) 

With both guards bypassed, the resulting SQL (built in `getTransactionTimestampsQuery`) performs a full sequential scan of the `transaction` table with no timestamp predicate, which on a production-scale dataset runs for the full 20-second `statement_timeout` before being killed. [8](#0-7) 

### Impact Explanation
Each unbounded request holds one pool connection for up to 20 seconds. With `maxConnections=10` (default), 10 concurrent requests fully saturate `global.pool`. All subsequent calls to `pool.queryQuietly(...)` queue behind the pool's `connectionTimeoutMillis` (20 seconds) and then fail. This results in a **total denial of service** for all transaction queries — including those from legitimate users and internal health checks — for as long as the attacker sustains the flood. The impact matches the stated scope: network inability to confirm new transactions via the mirror node REST API. [9](#0-8) 

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and only 10 concurrent HTTP requests. The endpoint is publicly accessible. The attacker needs only a basic HTTP client capable of concurrency (e.g., `curl`, `ab`, `wrk`). The attack is trivially repeatable: as old queries time out and connections are released, the attacker issues new requests to re-saturate the pool. No exploit code, credentials, or insider access are needed.

### Recommendation

1. **Enable `validateRange` for the transactions endpoint**: Change the `parseTimestampFilters` call in `getTransactions` to pass `validateRange=true` (last argument), enforcing the existing 7-day `maxTimestampRange` cap.

2. **Enable `bindTimestampRange` by default** or enforce it unconditionally in `getTransactions` so that open-ended ranges are always capped to `maxTransactionsTimestampRange` (60 days).

3. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer is already referenced in the Helm values) in front of the Node.js pool to absorb bursts.

4. **Add request-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit`) for the `/api/v1/transactions` endpoint, analogous to the throttling already present in the `web3` service. [3](#0-2) 

### Proof of Concept

**Preconditions:** Public network access to the mirror node REST API. No credentials required.

**Steps:**

```bash
# Saturate the 10-connection pool with 10 concurrent unbounded scans
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions" &
done
wait

# While the above are running, a legitimate request will queue and time out:
curl -v "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=eq:1234567890.000000000"
# Expected: hangs for ~20s then fails with connection timeout or 503
```

Alternatively, with `timestamp=gte:0` to make the open range explicit:
```bash
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=gte:0" &
done
```

**Result:** All 10 pool connections are held for up to 20 seconds each. Any new request during this window cannot acquire a connection and either queues indefinitely or times out, producing a complete denial of service for the transaction query path.

```javascript
const parseTimestampFilters = (
  filters,
  filterRequired = true,
  allowNe = false,
  allowOpenRange = false,
  strictCheckOverride = true,
  validateRange = true
) => {
  const forceStrictChecks = strictCheckOverride || config.strictTimestampParam;

  if (filters.length === 0) {
    if (filterRequired) {
      throw new InvalidArgumentError('No timestamp range or eq operator provided');
    }

    return {range: null, neValues: [], eqValues: []};
  }

  let earliest = null;
  let latest = null;
  let range = null;
  const eqValues = new Set();
  const neValues = new Set();
  let lowerBoundFilterCount = 0;
  let upperBoundFilterCount = 0;

  for (const filter of filters) {
    let value = BigInt(filter.value);
    switch (filter.operator) {
      case opsMap.eq:
        eqValues.add(value);
        break;
      case opsMap.ne:
        neValues.add(value);
        break;
      case opsMap.gt:
        value += 1n;
      case opsMap.gte:
        earliest = bigIntMax(earliest ?? 0n, value);
        lowerBoundFilterCount += 1;
        break;
      case opsMap.lt:
        value -= 1n;
      case opsMap.lte:
        latest = bigIntMin(latest ?? constants.MAX_LONG, value);
        upperBoundFilterCount += 1;
        break;
    }
  }

  if (forceStrictChecks) {
    if (!allowNe && neValues.size > 0) {
      throw new InvalidArgumentError('Not equals operator not supported for timestamp param');
    }

    if (lowerBoundFilterCount > 1) {
      throw new InvalidArgumentError('Multiple gt or gte operators not permitted for timestamp param');
    }

    if (upperBoundFilterCount > 1) {
      throw new InvalidArgumentError('Multiple lt or lte operators not permitted for timestamp param');
    }

    if (eqValues.size > 0 && (lowerBoundFilterCount > 0 || upperBoundFilterCount > 0 || neValues.size > 0)) {
      throw new InvalidArgumentError('Cannot combine eq with ne, gt, gte, lt, or lte for timestamp param');
    }
  }

  if (!allowOpenRange && eqValues.size === 0 && (lowerBoundFilterCount === 0 || upperBoundFilterCount === 0)) {
    throw new InvalidArgumentError('Timestamp range must have gt (or gte) and lt (or lte), or eq operator');
  }

  const difference = latest !== null && earliest !== null ? latest - earliest + 1n : null;

  if (validateRange) {
    const {maxTimestampRange, maxTimestampRangeNs} = config.query;

    // If difference is null, we want to ignore because we allow open ranges and that is known to be true at this point
    if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
      throw new InvalidArgumentError(
        `Timestamp range by the lower and upper bounds must be positive and within ${maxTimestampRange}`
      );
    }
  }
```
