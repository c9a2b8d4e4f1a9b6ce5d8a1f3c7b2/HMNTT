### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses Page-Size Guard, Causing Unbounded DB Queries and Non-Terminating Subscription

### Summary
When a client submits a gRPC `subscribeTopic` request with `limit = 2147483648` (Integer.MAX_VALUE + 1), the narrowing cast `(int)(filter.getLimit() - context.getCount().get())` silently overflows to `Integer.MIN_VALUE` (-2147483648). `Math.min` then selects this negative value as the page size, which is forwarded to the repository where `hasLimit()` returns `false` (because -2147483648 ≤ 0), causing every poll to issue a fully unbounded SQL query. The subscription never terminates because the outer `flux.take(2147483648L)` requires two billion messages to complete.

### Finding Description

**Entry point — `ConsensusController.toFilter()`:** [1](#0-0) 

`query.getLimit()` is a protobuf `uint64`; any client can supply `2147483648`. The field is stored as a Java `long`.

**Validation — `TopicMessageFilter.limit`:** [2](#0-1) 

Only `@Min(0)` is present. `2147483648L > 0`, so validation passes. There is no `@Max` constraint.

**Overflow site — `PollingTopicListener.poll()`, lines 54-57:** [3](#0-2) 

```
filter.getLimit()              = 2147483648L
context.getCount().get()       = 0L
(int)(2147483648L - 0L)        = (int)2147483648L = -2147483648   ← overflow
Math.min(-2147483648, maxPageSize) = -2147483648
```

**Corrupted filter forwarded to repository:** [4](#0-3) 

`pageSize = -2147483648` is widened back to `long -2147483648L` and stored in `newFilter.limit`.

**Repository — `hasLimit()` check:** [5](#0-4) 

`-2147483648 > 0` is `false`, so `hasLimit()` returns `false`.

**`setMaxResults` is never called:** [6](#0-5) 

The JPA query runs with no `LIMIT` clause, returning every matching row in the topic.

**Outer subscription never completes:** [7](#0-6) 

`flux.take(2147483648L)` requires two billion messages before emitting `onComplete`. The subscription loops indefinitely via `RepeatSpec.times(Long.MAX_VALUE)`. [8](#0-7) 

### Impact Explanation
Every poll cycle (default interval configurable via `listenerProperties.getInterval()`) issues a full table-scan-equivalent query against `topic_message` for the given topic, with no row limit. On a busy topic with millions of messages this saturates DB I/O and connection pool threads. The subscriber-count gauge is incremented and never decremented for the lifetime of the connection, and the bounded-elastic scheduler thread is held indefinitely. Multiple concurrent attackers can exhaust the thread pool and DB connection pool, constituting a denial-of-service against all legitimate subscribers.

### Likelihood Explanation
The attack requires only a valid gRPC connection and knowledge of any existing topic ID — both are publicly available on Hedera mainnet/testnet. No authentication or special privilege is needed. The trigger value (`2147483648`) is a single integer field in the protobuf request. The attack is trivially repeatable and scriptable. It is only effective when the listener type is configured as `POLL` (`listenerProperties.type = POLL`), which is one of the three supported modes.

### Recommendation
Replace the unsafe narrowing cast with a bounds-checked computation:

```java
// In PollingTopicListener.poll()
long remaining = filter.getLimit() - context.getCount().get();
// Clamp to [0, Integer.MAX_VALUE] before narrowing
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

Additionally, add a `@Max(Long.MAX_VALUE)` or a domain-appropriate maximum (e.g., `@Max(1_000_000)`) constraint on `TopicMessageFilter.limit` to reject absurdly large values at the API boundary before they reach internal arithmetic.

### Proof of Concept

```bash
# Requires grpcurl and a running mirror-node with POLL listener type
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": <any_valid_topic>},
    "consensusStartTime": {"seconds": 0},
    "limit": 2147483648
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected (buggy) behavior:**
1. Request passes `@Min(0)` validation (2147483648 > 0).
2. `poll()` computes `(int)(2147483648L - 0L)` = `-2147483648`.
3. `Math.min(-2147483648, maxPageSize)` = `-2147483648`.
4. `newFilter.limit = -2147483648`; `hasLimit()` = false.
5. Repository executes unbounded `SELECT` on every poll interval.
6. Connection is held open indefinitely; subscriber count never decrements.
7. Repeat with N concurrent connections to exhaust DB pool.
