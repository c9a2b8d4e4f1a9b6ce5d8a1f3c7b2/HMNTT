### Title
Unauthenticated High-Frequency Enumeration of Token Allowances via Missing Rate Limiting in REST API

### Summary
The `getAccountTokenAllowances()` function in `rest/controllers/tokenAllowanceController.js` is publicly accessible with no authentication requirement and no application-level rate limiting. An unprivileged attacker can issue unlimited parallel requests iterating over account IDs to harvest the complete token authorization graph of the network, enabling targeted reconnaissance for token transfer attacks and causing database resource exhaustion.

### Finding Description
**Code path:** `rest/controllers/tokenAllowanceController.js`, `getAccountTokenAllowances()`, lines 68–81.

The handler resolves any account ID, alias, or EVM address, queries `TokenAllowanceService.getAccountTokenAllowances()`, and returns the full list of `(owner, spender, token_id, amount)` tuples with no access control.

**Root cause — no rate limiting in the REST API middleware stack.**

`rest/server.js` assembles the middleware chain as:
`cors → json → httpContext → requestLogger → authHandler → (optional metrics) → (optional cache) → routes → responseHandler → handleError`

There is no rate-limiting middleware anywhere in this chain. The `authHandler` (`rest/middleware/authHandler.js`, lines 15–36) only acts when HTTP Basic credentials are *present*; when they are absent it returns immediately (`if (!credentials) { return; }`), allowing every unauthenticated request through unconditionally.

Rate limiting exists only in the **web3** Java module (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) and in an optional Traefik ingress configuration for the Rosetta service (`charts/hedera-mirror-rosetta/values.yaml`). Neither applies to the Node.js REST API.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/{id}/allowances/tokens` for `id = 0.0.1` through `0.0.10_000_000` in a tight loop with no credentials.
2. Each request hits `getAccountTokenAllowances()` → `TokenAllowanceService` → database query, with no throttle.
3. Responses reveal every `(owner, spender, token_id, amount)` relationship for each account.
4. Attacker aggregates results to build a complete token-authorization graph.

**Why existing checks fail:**
- `authHandler` is opt-in; unauthenticated requests are fully served.
- No `express-rate-limit`, `bottleneck`, or equivalent middleware is registered.
- The `responseCacheHandler` only caches responses; it does not limit request rate.
- Infrastructure-level Traefik rate limiting is only configured for the Rosetta chart, not the REST API deployment.

### Impact Explanation
An attacker obtains a complete map of which accounts have granted token-transfer authority to which spenders and for how much. This directly identifies high-value targets: accounts with large outstanding allowances whose spender keys may be weaker or already compromised. Beyond reconnaissance, the unbounded database load from concurrent enumeration can degrade or deny service to all REST API consumers (resource exhaustion on the PostgreSQL pool, whose limits are configured in `rest/config.js` `db.pool`).

### Likelihood Explanation
No credentials, no tokens, no special network position are required. The attack is fully scriptable with standard HTTP tooling (`curl`, `httpx`, Python `aiohttp`). Account IDs are sequential integers, making iteration trivial. The attack is repeatable indefinitely and leaves no application-level audit trail beyond access logs.

### Recommendation
1. Add application-level rate limiting to the REST API, e.g., `express-rate-limit` keyed on IP, applied globally in `rest/server.js` before route registration.
2. Require authentication (or at minimum an API key) for high-cardinality enumeration endpoints such as `/accounts/:id/allowances/tokens`.
3. Enforce a maximum page size and require cursor-based pagination tokens that expire, preventing stateless bulk iteration.
4. Mirror the `ThrottleConfiguration` pattern already used in the web3 module into the REST API layer.

### Proof of Concept
```bash
# Enumerate token allowances for 10,000 accounts with 50 parallel workers
seq 1 10000 | xargs -P50 -I{} \
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.{}/allowances/tokens" \
  -o allowances_{}.json

# Aggregate spender→owner→token relationships
jq -r '.allowances[] | [.owner, .spender, .token_id, .amount] | @csv' allowances_*.json \
  > full_token_auth_graph.csv
```
No credentials are needed. Each request is served immediately. The resulting CSV contains the complete token authorization graph of the network, ready for target selection.
