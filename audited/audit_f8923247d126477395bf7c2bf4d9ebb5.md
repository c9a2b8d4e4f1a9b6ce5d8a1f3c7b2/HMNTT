### Title
Unauthenticated Timestamp Parameter Triggers Expensive UNION Query Across `entity` and `entity_history` Tables in `getContractById`

### Summary
Any unauthenticated external user can append a `?timestamp=<value>` query parameter to a `GET /contracts/:contractId` request, causing `getContractByIdOrAddressContractEntityQuery()` to generate a UNION query spanning both the `entity` and `entity_history` tables instead of the cheaper single-table lookup. No authentication, authorization, or rate limiting at the application layer prevents this escalation. Because `entity_history` accumulates all historical entity states and the UNION adds range-overlap (`&&`) filtering plus an `ORDER BY … LIMIT 1`, the per-request database cost is materially higher than the baseline non-timestamp variant.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js:13` registers the route with no middleware beyond the generic request parser: [1](#0-0) 

`getContractById` (rest/controllers/contractController.js:707–737) unconditionally passes every validated timestamp filter into `extractTimestampRangeConditionFilters`, then into `getContractByIdOrAddressContractEntityQuery`: [2](#0-1) 

`getContractByIdOrAddressContractEntityQuery` (lines 179–208) branches solely on `timestampConditions.length !== 0`. When that condition is true it appends a full `UNION` against `entity_history`, an `ORDER BY timestamp_range DESC`, and a `LIMIT 1`: [3](#0-2) 

**Root cause:** The branch that decides whether to issue a cheap single-table lookup or an expensive two-table UNION is controlled entirely by the presence of a user-supplied `timestamp` query parameter. There is no privilege check, no cost guard, and no rate limit gating this branch.

**`extractTimestampRangeConditionFilters`** (rest/utils.js:709–756) only validates timestamp format (seconds.nanoseconds), not whether the caller is allowed to trigger historical lookups: [4](#0-3) 

**Generated queries:**

*Without timestamp (cheap):*
```sql
SELECT … FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND e.id = $1
```

*With timestamp (expensive):*
```sql
SELECT … FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND e.id = $3
UNION
SELECT … FROM entity_history e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND e.id = $3
ORDER BY timestamp_range DESC LIMIT 1
```

The `&&` (range-overlap) operator on `timestamp_range` in `entity_history` — a table that grows unboundedly with every entity state change — is significantly more expensive than a primary-key lookup on `entity`.

### Impact Explanation

The `entity_history` table stores every historical state of every entity on the network; on a mature network it can contain millions of rows. The UNION forces the database to evaluate the range-overlap predicate against both tables, sort the combined result set, and apply a limit. Compared to the indexed PK lookup used without a timestamp, this query can consume substantially more CPU, I/O, and memory per request. Because the route is public and unauthenticated, an attacker can sustain a stream of such requests to keep the database under elevated load, degrading service for all users. In environments with low baseline traffic or a large `entity_history` table, even a modest request rate can push resource consumption well above the 30% threshold relative to the preceding 24-hour baseline.

### Likelihood Explanation

The attack requires zero privileges, zero accounts, and zero knowledge beyond the public API documentation. The only precondition is knowing a valid contract ID (trivially obtained from any block explorer or the `/contracts` listing endpoint). The timestamp value itself does not need to match any real data — any syntactically valid timestamp (e.g., `1.000000000`) triggers the UNION branch. The attack is fully repeatable and scriptable.

### Recommendation

1. **Rate-limit the historical (timestamp) code path** separately from the non-historical path, or require an API key for historical queries.
2. **Add a query timeout** at the database connection level for this endpoint so a single expensive query cannot hold a connection indefinitely.
3. **Consider caching** historical contract lookups (the result for a given `contractId + timestamp` is immutable once the block is finalized).
4. **Restrict the timestamp range** accepted by this endpoint (e.g., reject timestamps older than a configurable window) to bound the scan size on `entity_history`.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly reachable. At least one contract exists (e.g., `0.0.1234`).

**Step 1 – Baseline (cheap query):**
```
GET /api/v1/contracts/0.0.1234
```
Executes a single indexed lookup on `entity`.

**Step 2 – Expensive query (single unauthenticated request):**
```
GET /api/v1/contracts/0.0.1234?timestamp=lte:9999999999.000000000
```
Executes the UNION query across `entity` + `entity_history` with a range-overlap condition, `ORDER BY`, and `LIMIT 1`.

**Step 3 – Sustained amplification (no brute force, just repeated valid requests):**
```bash
while true; do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234?timestamp=lte:9999999999.000000000" > /dev/null
done
```
Each iteration triggers the expensive UNION path. Database CPU and I/O rise proportionally to the request rate and the size of `entity_history`, with no application-layer throttle to stop it.

```javascript
const extractTimestampRangeConditionFilters = (
  filters,
  offset = 0,
  timestampRangeColumn = Entity.getFullName(Entity.TIMESTAMP_RANGE)
) => {
  const conditions = [];
  const params = [];

  filters
    .filter((filter) => filter.key === constants.filterKeys.TIMESTAMP)
    .forEach((filter) => {
      const position = `$${params.length + offset + 1}`;
      let condition;
      let range;

      if (filter.operator === opsMap.ne) {
        // handle ne filter differently
        condition = `not ${timestampRangeColumn} @> ${position}`; // @> is the pg range "contains" operator
        range = Range(filter.value, filter.value, '[]');
      } else {
        condition = `${timestampRangeColumn} && ${position}`; // && is the pg range "overlaps" operator

        switch (filter.operator) {
          case opsMap.lt:
            range = Range(null, filter.value, '()');
            break;
          case opsMap.eq:
          case opsMap.lte:
            range = Range(null, filter.value, '(]');
            break;
          case opsMap.gt:
            range = Range(filter.value, null, '()');
            break;
          case opsMap.gte:
            range = Range(filter.value, null, '[)');
            break;
        }
      }

      conditions.push(condition);
      params.push(range);
    });

  return {
    conditions,
    params,
  };
};
```
