### Title
Unauthenticated Content-Length Header Injection Corrupts Request Size Histogram Metrics

### Summary
The `metricsHandler` middleware in `rest/middleware/metricsHandler.js` blindly trusts the client-supplied `Content-Length` header value, parsing it with `parseInt` and recording it directly into the `requestSizeHistogram` OpenTelemetry histogram with no upper-bound validation. Any unauthenticated external user can send a request with a crafted `Content-Length: 9999999999` header to any endpoint ŌĆö including contract endpoints ŌĆö causing the histogram to record a falsely inflated value, permanently corrupting the `api_request_size_bytes` Prometheus metric distribution for that route.

### Finding Description
**Exact code path:**

`rest/middleware/metricsHandler.js`, line 209:
```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** The code reads `req.headers['content-length']` directly from the HTTP request object. `parseInt('9999999999', 10)` returns the JavaScript number `9999999999` ŌĆö a valid, non-NaN, non-zero value ŌĆö so the `|| 0` fallback never triggers. There is no upper-bound clamp, no comparison against the actual received body size, and no rejection of implausible values.

**Failed assumption:** The code assumes `Content-Length` reflects the actual body size. HTTP clients can set this header to any arbitrary integer; Node.js/Express does not validate or reject mismatched `Content-Length` values before the header reaches application middleware.

**Middleware registration:** As shown in `rest/server.js` lines 88ŌĆō92, `metricsHandler()` is registered globally before any route handlers and before any body-size enforcement. Every request ŌĆö including unauthenticated GET requests to `/api/v1/contracts/...` ŌĆö passes through this middleware.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/contracts HTTP/1.1\r\nContent-Length: 9999999999\r\n\r\n`
2. `metricsMiddleware` fires on `res.on('finish', ...)`.
3. `parseInt('9999999999', 10)` ŌåÆ `9999999999`; `|| 0` does not trigger.
4. `requestSizeHistogram.record(9999999999, {method:'GET', path:'/api/v1/contracts', code:'200'})` executes.
5. The histogram bucket sum and mean for the `contracts` route are permanently skewed.

**Existing checks reviewed and shown insufficient:**
- `|| 0` ŌĆö only catches `NaN` or `0`; does nothing for large integers.
- No upstream middleware in `server.js` validates or caps `Content-Length` before `metricsHandler`.
- `express.json()` / `express.urlencoded()` enforce body parsing limits but do not reject or rewrite the raw `Content-Length` header value accessible via `req.headers`.

### Impact Explanation
The `api_request_size_bytes` histogram (exported as a Prometheus metric) is corrupted for any route the attacker targets. For contract-related routes (`/api/v1/contracts`, `/api/v1/contracts/:id/results`, etc.), the per-route request size distribution becomes meaningless. Operators relying on these metrics for capacity planning, anomaly detection, or SLA monitoring receive false data. Repeated injection can permanently skew histogram bucket counts and sum values, masking real anomalies or triggering false alerts. The impact is confined to observability integrity ŌĆö no funds or protocol state are affected.

### Likelihood Explanation
Exploitation requires zero authentication, zero special tooling, and zero knowledge beyond the public API surface. A single `curl` command suffices. The attack is trivially repeatable at high frequency (e.g., scripted loop), amplifying metric corruption. Any internet-accessible deployment of the mirror node REST API is exposed.

### Recommendation
Replace the unchecked `parseInt` with a value that is clamped against the actual received body size or a reasonable maximum:

```js
// Option 1: clamp to actual body size (req.socket.bytesRead or body buffer length)
const declaredSize = parseInt(req.headers['content-length'] ?? '0', 10);
const recordedSize = Number.isFinite(declaredSize) && declaredSize >= 0
  ? Math.min(declaredSize, MAX_REASONABLE_REQUEST_BYTES)
  : 0;
requestSizeHistogram.record(recordedSize, labels);
```

Alternatively, measure the actual received body bytes (e.g., by accumulating `req.on('data')` chunk sizes in the middleware) rather than trusting the client-declared header.

### Proof of Concept
```bash
# Corrupt the contracts route request-size histogram with a single unauthenticated request
curl -s -o /dev/null \
  -H "Content-Length: 9999999999" \
  "http://<mirror-node-host>/api/v1/contracts"

# Verify corruption in Prometheus metrics endpoint (if accessible)
curl -s "http://<mirror-node-host>/api/v1/metrics/" | \
  grep 'api_request_size_bytes.*contracts'
# Expected: bucket sums show 9999999999 recorded, skewing all percentile calculations
```

Repeating this in a loop will continuously inflate the histogram sum, making mean/p99 request size metrics for contract endpoints permanently unreliable.
