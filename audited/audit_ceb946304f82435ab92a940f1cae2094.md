### Title
Log Injection via Control Character Bypass in `getContent()` — Off-by-One Size Check Combined with Incomplete Whitespace Sanitization

### Summary
`LoggingFilter.getContent()` applies `StringUtils.deleteWhitespace()` to the request body before size-checking, but `deleteWhitespace` only removes Java whitespace characters (per `Character.isWhitespace()`), leaving control characters such as ESC (`\u001B`) intact. Both the compression and truncation guards use a strict `>` comparison, so a payload crafted to be exactly `maxPayloadLogSize` characters after whitespace removal bypasses all size controls and is logged verbatim — including any embedded control characters. An unprivileged attacker can exploit this to inject ANSI escape sequences or other control characters into the log stream, corrupting log output and hiding actual calldata from monitoring systems.

### Finding Description

**Code path:**
`web3/src/main/java/org/hiero/mirror/web3/config/LoggingFilter.java`, `getContent()`, lines 80–111.

```
86:  content = StringUtils.deleteWhitespace(wrapper.getContentAsString());
89:  if (content.length() > maxPayloadLogSize) {   // compression gate
...
105: if (content.length() > maxPayloadLogSize ...) { // truncation gate
107:     content = StringUtils.substring(content, 0, maxPayloadLogSize);
```

**Root cause — two compounding failures:**

1. **Off-by-one in both guards (lines 89 and 105):** Both checks use `>` (strictly greater than). A payload whose post-`deleteWhitespace` length equals exactly `maxPayloadLogSize` (default 300) satisfies neither condition, so neither compression nor truncation is applied and the full string is returned and logged.

2. **`StringUtils.deleteWhitespace` does not sanitize control characters:** The method delegates to `Character.isWhitespace(c)`, which returns `true` only for a narrow set of characters (`\t`, `\n`, `\x0B`, `\f`, `\r`, `\u001C`–`\u001F`, and Unicode space separators). It returns `false` for:
   - `\u001B` (ESC — the prefix of every ANSI/VT100 escape sequence)
   - `\u0000` (NUL)
   - `\u0001`–`\u0008` (SOH through BS)
   - `\u000E`–`\u001A` (SO through SUB)

   These characters survive `deleteWhitespace` and are counted toward the 300-character budget.

**Exploit flow:**

An attacker crafts a JSON body for a `POST /api/v1/contracts/call` request:

```
{"data":"0x\u001b[2J\u001b[H<hex_calldata_padded_to_fill_exactly_300_chars_total>"}
```

- `\u001b` is 1 character; it is NOT removed by `deleteWhitespace`.
- The attacker pads the hex calldata so the total post-`deleteWhitespace` length is exactly 300.
- `content.length() == 300`, so `content.length() > 300` is `false` at both line 89 and line 105.
- The string is returned unchanged and passed to `log.info(LOG_FORMAT, params)` at line 76.
- The logger writes the raw string — including `\u001b[2J\u001b[H` — to the log sink.

**Why existing checks fail:**

| Check | Intended purpose | Why it fails |
|---|---|---|
| `StringUtils.deleteWhitespace` | Normalize multi-line input | Only removes Java whitespace; ESC and other control chars survive |
| `content.length() > maxPayloadLogSize` (line 89) | Trigger compression | `==` case is not covered; exactly-sized payloads pass through |
| `content.length() > maxPayloadLogSize` (line 105) | Truncate oversized content | Same off-by-one; exactly-sized payloads are never truncated | [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

- **Log corruption / ANSI injection:** In any terminal-based log viewer or log aggregator that interprets ANSI escape sequences (common in Kubernetes/Docker environments, Kibana with raw-message display, Grafana Loki, etc.), sequences like `\u001b[2J` (clear screen), `\u001b[H` (cursor home), or `\u001b[?25l` (hide cursor) can erase preceding log lines, overwrite them with attacker-controlled content, or make them invisible.
- **Calldata hiding:** The actual `data` field of the contract call — the primary artifact monitored for malicious smart contract interactions — can be hidden from operators and SIEM systems, defeating the purpose of the logging filter entirely.
- **False log injection:** By combining cursor-movement sequences with attacker-controlled text, an attacker can make the log appear to show a benign call while the real calldata is hidden.
- **No funds at direct risk**, consistent with the Medium scope classification, but monitoring integrity is fully compromised for the duration of the attack. [5](#0-4) 

### Likelihood Explanation

- **No authentication required:** Any HTTP client can POST to `/api/v1/contracts/call`. Zero privilege needed.
- **Trivially repeatable:** The attacker only needs to count characters and pad the hex calldata field to hit exactly 300 non-whitespace characters. This is a one-time calculation, automatable in a script.
- **Reliable bypass:** The off-by-one is deterministic — `==` always bypasses both guards.
- **Control character delivery:** JSON strings permit arbitrary Unicode code points including `\u001B`; standard HTTP clients pass them through without stripping.

### Recommendation

1. **Fix the off-by-one:** Change both guards from `>` to `>=`:
   ```java
   if (content.length() >= maxPayloadLogSize) { // line 89
   if (content.length() >= maxPayloadLogSize && ...) { // line 105
   ```

2. **Sanitize control characters before logging:** After `deleteWhitespace`, strip or replace non-printable characters:
   ```java
   content = StringUtils.deleteWhitespace(wrapper.getContentAsString());
   // Remove non-printable control characters (U+0000–U+001F except already-removed whitespace)
   content = content.replaceAll("[\\x00-\\x1F\\x7F]", "");
   ```
   Alternatively use a dedicated log-sanitization library (e.g., OWASP's `encodeForHTML` / `encodeForLog`).

3. **Consider structured logging:** Log the payload as a structured field rather than interpolating it into a free-form string, so log aggregators treat it as data rather than markup. [6](#0-5) 

### Proof of Concept

**Preconditions:** Mirror Node web3 service running with default config (`maxPayloadLogSize=300`). Log output visible in a terminal or ANSI-capable viewer.

**Step 1 — Craft the payload.**

The target is a JSON body where `StringUtils.deleteWhitespace(body).length() == 300` and the body contains `\u001b[2J\u001b[H` (clear-screen + cursor-home, 7 non-whitespace chars). Fill the remaining 293 characters with valid hex calldata:

```python
import json, requests

ESC = "\u001b"
ansi_clear = ESC + "[2J" + ESC + "[H"   # 7 chars, not whitespace
# pad hex data to make total == 300 after whitespace removal
hex_pad = "0x" + "aa" * 143             # "0x" + 286 hex chars = 288 chars
# total non-whitespace: 7 (ansi) + 5 ({"data":"}) + 288 (hex) + 2 ("}) = 302 -- adjust accordingly
# Fine-tune padding so deleteWhitespace(body).length() == 300 exactly

body = '{"data":"' + ansi_clear + hex_pad + '"}'
# Verify: len(body.replace(' ','').replace('\t','')...) == 300

requests.post("http://localhost:8545/api/v1/contracts/call", 
              data=body, 
              headers={"Content-Type": "application/json"})
```

**Step 2 — Observe the result.**

In the server's log terminal:
- The screen clears (`\u001b[2J`)
- The cursor moves to the top-left (`\u001b[H`)
- All preceding log lines (including the actual calldata) are no longer visible
- The attacker can follow with additional escape sequences to print attacker-controlled "log lines"

**Step 3 — Confirm bypass.**

Add a debug assertion: if `content.length() == 300`, neither the compression block (line 89) nor the truncation block (line 105) is entered, confirming the full payload — including control characters — is returned and logged verbatim. [7](#0-6)

    private String getContent(HttpServletRequest request, int status) {
        var content = StringUtils.EMPTY;
        final int maxPayloadLogSize = web3Properties.getMaxPayloadLogSize();
        final var wrapper = WebUtils.getNativeRequest(request, ContentCachingRequestWrapper.class);

        if (wrapper != null) {
            content = StringUtils.deleteWhitespace(wrapper.getContentAsString());
        }

        if (content.length() > maxPayloadLogSize) {
            final var bos = new ByteArrayOutputStream(content.length() / 4);
            try (final var out = new GZIPOutputStream(bos)) {
                out.write(content.getBytes(StandardCharsets.UTF_8));
                out.finish();
                final var compressed = Base64.getEncoder().encodeToString(bos.toByteArray());

                if (compressed.length() <= maxPayloadLogSize) {
                    content = compressed;
                }
            } catch (Exception e) {
                // Ignore
            }
        }

        // Truncate log message size unless it's a 5xx error
        if (content.length() > maxPayloadLogSize && status < HttpStatus.INTERNAL_SERVER_ERROR.value()) {
            content = reorderFields(content);
            content = StringUtils.substring(content, 0, maxPayloadLogSize);
        }

        return content;
    }
