### Title
Unauthenticated Full-Table Scan via `latest_token_balance` CTE in `/api/v1/accounts` Enables DB CPU Exhaustion

### Summary
The `getEntityBalanceQuery()` function in `rest/accounts.js` unconditionally emits a `with latest_token_balance as (select ... from token_account where associated is true)` CTE — a full sequential scan of the entire `token_account` table — for every call to the `/api/v1/accounts` list endpoint. Because the REST API has no rate-limiting middleware and the endpoint requires no authentication, any external user can flood the database with concurrent full-table scans, exhausting the CPU and I/O needed for transaction confirmation.

### Finding Description

**Code path:**

In `getAccounts()`, `accountBalanceQuery` is always passed as `undefined`: [1](#0-0) 

`undefined` causes `getAccountQuery()` to use the default `{query: '', params: []}`: [2](#0-1) 

Inside `getEntityBalanceQuery()`, the branch condition is: [3](#0-2) 

An empty string is falsy in JavaScript, so `needHistoricalBalanceInfo` is always `false` for the list endpoint. Execution always falls into the `else` branch: [4](#0-3) 

The CTE `select account_id, balance, token_id from token_account where associated is true` has **no LIMIT** and no selective index. The schema defines only a primary key on `(account_id, token_id)` and a secondary index on `(token_id, account_id)`: [5](#0-4) 

There is no index on the `associated` column alone, so the filter `where associated is true` forces a full sequential scan of the entire `token_account` table on every request.

**No rate limiting on the REST API:**

The `server.js` middleware stack contains no rate-limiting layer for unauthenticated callers: [6](#0-5) 

The throttle infrastructure that exists (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` module (contract calls) and is not applied to the REST API: [7](#0-6) 

**Why existing checks fail:**

- The `limit` query parameter only constrains the outer `SELECT` result set, not the CTE scan.
- `balance=false` would bypass balance entirely, but the default is `true` and the attacker does not need to set it.
- The optional Redis response cache (`applicationCacheEnabled`) is off by default and cache keys vary with query parameters, so an attacker can trivially bypass it by varying `account.id` or `limit`.

### Impact Explanation

On mainnet, `token_account` contains tens of millions of rows. Each unauthenticated GET to `/api/v1/accounts` (with default `balance=true` and no timestamp filter) materialises the entire associated-token-account set. Concurrent flooding saturates PostgreSQL CPU and I/O. Because the mirror node's database is shared with the importer pipeline that writes consensus transaction data, sustained CPU exhaustion degrades or halts the importer's ability to record new transactions, effectively causing a network-visible mirror node outage. Severity: **Critical** (network-level availability impact).

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and a single valid HTTP GET request. The endpoint is publicly documented and reachable. An attacker needs only a script issuing concurrent requests to `/api/v1/accounts`. The attack is trivially repeatable and automatable from a single host or a small botnet. No special knowledge of the codebase is required.

### Recommendation

1. **Add a LIMIT to the CTE** or rewrite it as a correlated subquery so it is scoped per-account rather than materialising the full table:
   ```sql
   -- Instead of a global CTE, use a correlated subquery per entity row
   (select json_agg(...) from token_account where account_id = e.id and associated is true limit <N>) as token_balances
   ```
2. **Add an index on `(associated, account_id, token_id)`** or a partial index `WHERE associated = true` to avoid full sequential scans if a global CTE is retained.
3. **Implement rate limiting on the REST API** (e.g., express-rate-limit or a reverse-proxy layer) for unauthenticated callers, mirroring the throttle already present in the web3 module.
4. **Consider making `balance=false` the default** for the list endpoint, or requiring explicit opt-in for token balance inclusion.

### Proof of Concept

```bash
# No authentication required. Each request triggers a full token_account scan.
# Run N concurrent workers to exhaust DB CPU:

for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/accounts?limit=25" &
done
wait
```

Varying the `limit` or `account.id` parameter across requests defeats any response cache. Monitor PostgreSQL `pg_stat_activity` to observe concurrent sequential scans on `token_account` and CPU saturation.
