### Title
Unauthenticated `estimate=true` + Historical `block` Combination Triggers Up-to-21x DB-Load Amplification via Binary Gas Estimator

### Summary
Any unprivileged caller can POST to `/api/v1/contracts/call` with `estimate=true` and a historical `block` number simultaneously. `constructServiceParameters()` passes both fields through without cross-field validation, causing `ContractExecutionService.estimateGas()` to invoke the `BinaryGasEstimator` which performs up to 21 EVM executions (1 initial + up to 20 binary-search iterations), each querying expensive historical-state SQL (`_history` tables with CTEs). The throttle is consumed only once per request, so the attacker receives up to 21× the database load for the cost of a single throttle token.

### Finding Description

**Exact code path:**

`ContractController.constructServiceParameters()` at lines 75–89 blindly combines both fields: [1](#0-0) 

There is no cross-field validation anywhere in `ContractCallRequest` that rejects `estimate=true` when `block` is a historical value: [2](#0-1) 

The resulting `ContractExecutionParameters` (with both `isEstimate=true` and a historical `block`) is passed to `ContractExecutionService.processCall()`, which branches into `estimateGas()`: [3](#0-2) 

`estimateGas()` performs one initial `callContract()` call, then unconditionally invokes `binaryGasEstimator.search()`: [4](#0-3) 

`BinaryGasEstimator.search()` loops up to `maxGasEstimateRetriesCount` (default **20**) times, calling `doProcessCall()` on each iteration: [5](#0-4) 

**Why the throttle fails to bound this:** `throttleManager.throttle(request)` is called exactly once before the entire estimate operation: [6](#0-5) 

Inside `doProcessCall()`, `restoreGasToBucket()` and metric updates are explicitly skipped when `estimate=true`, so the binary-search iterations are invisible to the throttle: [7](#0-6) 

**Historical state cost:** When a historical block is set, `callContract()` resolves the block via `recordFileService.findByBlockType()` and every subsequent EVM execution queries `_history` tables using complex CTEs (e.g., `nft_allowance_history`, `token_history`, etc.), which are significantly more expensive than current-state queries: [8](#0-7) 

**Root cause / failed assumption:** The design assumes `ETH_ESTIMATE_GAS` is only used against the latest block. No guard prevents the combination of `estimate=true` with a historical block, and the throttle model accounts for one gas-unit cost while the binary search multiplies actual DB work by up to 21×.

### Impact Explanation

Each malicious request causes up to 21 EVM executions against historical DB state. Historical queries use multi-table CTEs over `*_history` tables, making each iteration substantially more expensive than a current-state call. An attacker sending concurrent requests at the throttle rate can amplify effective DB load by ~21× compared to what the throttle was designed to permit, degrading historical-state availability for legitimate users and potentially causing DB connection exhaustion or query queue saturation.

### Likelihood Explanation

No authentication, API key, or special privilege is required. The endpoint is publicly accessible. The attack is trivially reproducible with a single `curl` command and is fully repeatable. The attacker only needs to know any valid historical block number (obtainable from the public `/api/v1/blocks` endpoint). The amplification factor is deterministic and bounded only by `maxGasEstimateRetriesCount` (default 20), which is a configuration value not exposed to the attacker.

### Recommendation

1. **Reject the combination at the controller layer:** In `constructServiceParameters()` or via a `@AssertTrue` validator on `ContractCallRequest`, return HTTP 400 if `estimate=true` and `block` is not `latest`/`pending`/`safe`/`finalized`.
2. **Alternatively, throttle per binary-search iteration:** Pass the per-iteration gas cost through the throttle manager inside `BinaryGasEstimator.search()` so each EVM execution consumes a throttle token.
3. **Reduce the default `maxGasEstimateRetriesCount`** or make it configurable per call type (lower for historical calls).

### Proof of Concept

```bash
# 1. Obtain any valid historical block number
BLOCK=$(curl -s https://<mirror-node>/api/v1/blocks?limit=1&order=asc \
  | jq -r '.blocks[0].number')

# 2. Send estimate=true against that historical block (no auth required)
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d "{
    \"estimate\": true,
    \"block\": \"$BLOCK\",
    \"to\": \"0x0000000000000000000000000000000000000167\",
    \"gas\": 15000000,
    \"data\": \"0x18160ddd\"
  }"

# Result: server performs 1 initial EVM call + up to 20 binary-search iterations,
# each executing expensive historical-state DB queries, for a single throttle token.
# Repeat in a loop to amplify DB load ~21x beyond the throttle's intended limit.
```

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
