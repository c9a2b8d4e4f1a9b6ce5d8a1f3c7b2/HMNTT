### Title
Unauthenticated gRPC `getNodes()` Endpoint Allows Resource Exhaustion via Concurrent Stream Flooding

### Summary
The `NetworkService.getNodes()` gRPC endpoint is publicly accessible with no authentication, no rate limiting, and no per-IP connection cap. Each call consumes `Schedulers.boundedElastic()` threads and database connections for the duration of paged address book streaming. An unprivileged attacker opening many concurrent connections — each carrying up to 5 simultaneous `getNodes()` streams — can exhaust the shared bounded-elastic thread pool and the HikariCP database connection pool, degrading all mirror node gRPC processing including `subscribeTopic`.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) subscribes a reactive pipeline directly on the calling thread with no authentication check, no rate-limit guard, and no concurrent-call counter:

```java
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`NetworkServiceImpl.getNodes()` (lines 68–76) builds a paged Flux that schedules each inter-page delay on `Schedulers.boundedElastic()`:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())   // 250 ms default
                .withScheduler(Schedulers.boundedElastic()))            // shared JVM-wide pool
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
```

Each page also executes a synchronous DB transaction inside `transactionOperations.execute()` (line 80), holding a HikariCP connection for the duration of the page fetch.

**Root cause and failed assumptions:**

The only server-side concurrency control is `maxConcurrentCallsPerConnection = 5` (default), configured in `NettyProperties` and applied in `GrpcConfiguration` (line 33):

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This limits calls *per HTTP/2 connection*, not across all connections. There is no:
- Per-IP or global connection count limit in the Netty server builder
- Rate-limiting `ServerInterceptor` (the only interceptor, `GrpcInterceptor`, only sets an `EndpointContext` thread-local)
- Application-level concurrent-stream counter analogous to `subscriberCount` in `TopicMessageServiceImpl`
- Traefik `inFlightReq` or `rateLimit` middleware in the gRPC Helm chart (present in the rosetta chart but absent here — the grpc chart middleware only contains `circuitBreaker` and `retry`)
- The GCP gateway `maxRatePerEndpoint: 250` is explicitly annotated "Requires a change to HPA to take effect" and is not enforced by default

**Exploit flow:**

1. Attacker opens *N* TCP connections to port 5600 (HTTP/2, no TLS required in default docker-compose config).
2. On each connection, sends 5 concurrent `getNodes` RPCs with `limit=0` (unlimited) and a valid `file_id` (0.0.102, publicly documented).
3. Each stream pages through the entire address book: with `pageSize=10` and `pageDelay=250ms`, a 30-node address book takes ~750 ms per stream; larger books take proportionally longer.
4. Each active page-fetch holds one `boundedElastic` thread (default cap: `10 × CPU_count`) and one HikariCP DB connection.
5. With `N` connections: `5N` concurrent streams compete for the bounded-elastic pool and DB pool simultaneously.
6. Once the bounded-elastic pool is saturated, all reactive scheduling across the JVM stalls — including `subscribeTopic` historical retrieval (`PollingTopicMessageRetriever` also uses `Schedulers.boundedElastic()`) and the shared polling listener (`SharedPollingTopicListener`).

### Impact Explanation

Exhausting `Schedulers.boundedElastic()` (shared JVM-wide) stalls all reactive pipelines in the gRPC process: `subscribeTopic` historical retrieval, the safety-check polling in `TopicMessageServiceImpl`, and the shared polling listener all use the same scheduler. Simultaneously exhausting HikariCP connections blocks every DB-backed operation. This constitutes degradation of the mirror node's core processing capacity well beyond the 30% threshold. The mirror node's gRPC service becomes unresponsive to legitimate HCS subscribers, which are the primary consumers of the service.

### Likelihood Explanation

No credentials, tokens, or special network position are required. The endpoint is documented publicly (`grpcurl -plaintext ... localhost:5600 com.hedera.mirror.api.proto.NetworkService/getNodes`). A single machine with a standard gRPC client library (e.g., `ghz`, a custom Go/Java client) can open hundreds of HTTP/2 connections and thousands of concurrent streams. The attack is repeatable, requires no prior knowledge beyond the public proto definition, and produces measurable degradation proportional to the number of streams opened. The attacker can sustain the attack indefinitely since streams with `limit=0` page forever until cancelled.

### Recommendation

1. **Add a global concurrent-stream counter for `getNodes`** analogous to `subscriberCount` in `TopicMessageServiceImpl`, and reject new calls above a configurable threshold.
2. **Add a rate-limiting `ServerInterceptor`** (e.g., using Bucket4j or Guava `RateLimiter`) applied globally to the gRPC server, specifically covering `NetworkService/getNodes`.
3. **Add per-IP connection limiting** via `NettyServerBuilder.maxConnectionsPerIp()` or an equivalent Netty channel handler.
4. **Add Traefik `inFlightReq` and `rateLimit` middleware** to the gRPC Helm chart (mirroring what the rosetta chart already does).
5. **Enforce a mandatory non-zero `limit`** in `AddressBookFilter` validation so streams cannot page indefinitely.
6. **Use a dedicated, bounded `Scheduler`** for `getNodes` paging rather than the shared JVM-wide `Schedulers.boundedElastic()`.

### Proof of Concept

```bash
# Install ghz: https://ghz.sh
# Step 1: open 100 connections, 5 streams each = 500 concurrent getNodes streams
ghz --insecure \
    --proto network_service.proto \
    --call com.hedera.mirror.api.proto.NetworkService.getNodes \
    --data '{"file_id": {"fileNum": 102}, "limit": 0}' \
    --connections 100 \
    --concurrency 500 \
    --duration 60s \
