### Title
Thread Pool Exhaustion via Unbounded Concurrent DB Calls in `TopicServiceImpl.findById()` with No Application-Level Circuit Breaker or In-Flight Request Limiting

### Summary
`TopicServiceImpl.findById()` issues a blocking `topicRepository.findById()` call with no circuit breaker, no query timeout, and no in-flight request cap. The rest-java Traefik middleware omits the `inFlightReq` and `rateLimit` guards present in other services (e.g., rosetta), and the Traefik circuit breaker only reacts to error ratios — not to slow/hanging responses. An unauthenticated attacker can flood `GET /api/v1/topics/{id}` to exhaust the Tomcat thread pool and HikariCP connection pool during any DB slowdown, degrading ≥30% of mirror node processing capacity.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` (line 32–37) calls three sequential blocking DB operations per request with no protection:

```
topicService.findById(id.id())      // → topicRepository.findById(id.getId())
entityService.findById(id.id())     // → entityRepository.findById(id.getId())
customFeeService.findById(id.id())  // → customFeeRepository
```

`TopicServiceImpl.findById()` (line 19–21) is a bare Spring Data `CrudRepository.findById()` call — no `@Transactional(timeout=...)`, no Resilience4j `@CircuitBreaker`, no Bucket4j rate limit, no cache.

**Root cause — failed assumptions:**

1. The Traefik circuit breaker for rest-java (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) only trips on network errors or 5xx responses. During a DB slowdown, requests *hang* rather than fail immediately, so the circuit breaker never trips while threads are being consumed.

2. The rest-java Traefik middleware has **no `inFlightReq`** and **no `rateLimit`** entries — unlike rosetta which has both (`inFlightReq: amount: 5` and `rateLimit: average: 10`). This is a deliberate omission that leaves the endpoint fully unbounded.

3. The Traefik `retry: attempts: 3` middleware **multiplies** the DB load: each slow request is retried 3 times, tripling the number of blocking DB calls per client request.

4. No application-level throttling exists in rest-java (confirmed: no Bucket4j, no `ThrottleManager`, no `RateLimiter` in the entire `rest-java/` module).

5. HPA is disabled by default (`hpa.enabled: false`), so horizontal scaling does not auto-activate.

**Exploit flow:**

- Attacker sends N concurrent `GET /api/v1/topics/0.0.1` requests (valid topic ID, no auth required).
- Each request blocks a Tomcat worker thread waiting on `topicRepository.findById()`.
- Under DB slowdown (natural or attacker-induced by the flood itself), threads accumulate faster than they complete.
- Tomcat's bounded thread pool (default 200) fills; new requests queue then reject.
- HikariCP connection pool (default 10) saturates; threads additionally block waiting for a connection.
- Retry middleware issues up to 3× the DB calls, accelerating saturation.
- The Traefik circuit breaker never trips because responses are slow (not erroring).
- Result: the rest-java service becomes unresponsive, degrading mirror node API processing capacity.

### Impact Explanation
The `/api/v1/topics/{id}` endpoint is publicly accessible with no authentication. Complete thread pool exhaustion renders the rest-java service unavailable for all endpoints it handles (topics, accounts/allowances/nfts, accounts/airdrops, accounts/hooks, network/). Since the mirror node's REST API is a core data-access layer for the Hedera network, sustained unavailability constitutes ≥30% degradation of network processing node capacity as defined in the scope. The retry amplification (3×) means the attacker needs only ~67 concurrent slow requests to saturate a 200-thread pool.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is unauthenticated and publicly routable. The attacker needs only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`). DB slowdown can be self-induced by the flood itself (lock contention, I/O saturation) or timed to coincide with natural DB load spikes. The absence of `inFlightReq` limiting (present in rosetta but absent in rest-java) makes this straightforwardly exploitable. The attack is repeatable and requires no special knowledge beyond the public API path.

### Recommendation

1. **Add `inFlightReq` and `rateLimit` to the rest-java Traefik middleware** (matching rosetta's configuration) in `charts/hedera-mirror-rest-java/values.yaml`:
   ```yaml
   middleware:
     - circuitBreaker:
         expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
     - inFlightReq:
         amount: 100
         sourceCriterion:
           ipStrategy:
             depth: 1
     - rateLimit:
         average: 50
         sourceCriterion:
           requestHost: true
     - retry:
         attempts: 3
         initialInterval: 100ms
   ```

2. **Add a query timeout** to `TopicServiceImpl` via `@Transactional(timeout = 5)` or configure `spring.transaction.default-timeout`.

3. **Add an application-level circuit breaker** using Resilience4j `@CircuitBreaker` on `TopicServiceImpl.findById()`, consistent with the pattern already used in the web3 module.

4. **Enable HPA** for rest-java to allow horizontal scaling under load.
