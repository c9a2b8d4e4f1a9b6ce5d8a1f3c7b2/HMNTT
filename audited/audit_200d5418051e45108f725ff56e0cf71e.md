### Title
Unbounded Batch SQL IN-Clause via Per-Contract Slot Key Cache Amplification in `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally collects every key stored in the per-contract `slotsPerContract` Caffeine cache (up to 1500 entries, `expireAfterAccess=5m`) and issues a single SQL `IN`-clause query against `contract_state` with all of them. Because the `contractStateCache` expires every 2 seconds (`expireAfterWrite=2s`), an unauthenticated attacker who pre-populates the per-contract cache with 1500 unique slot keys can trigger a maximum-size 1500-key SQL batch query on every subsequent `findStorage()` call, repeatable every 2 seconds indefinitely with a single HTTP request per cycle.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ContractExecutionService.processCall()` → EVM execution → `ContractStorageReadableKVState.readFromDataSource()` → `ContractStateService.findStorage()` → `ContractStateServiceImpl.findStorageBatch()`

**Root cause — `ContractStateServiceImpl.java` lines 85–122:**

```
Line 90:  contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
Line 91:  final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
Line 93:  final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
Line 96-101: for (var slot : cachedSlotKeys) { cachedSlots.add(...) }
Line 103: contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

Every call to `findStorageBatch()` snapshots the **entire** per-contract Caffeine cache and passes it as the `IN`-clause to the repository query. There is no cap on how many keys are included in the SQL query beyond the cache's `maximumSize=1500`.

**Failed assumption:** The design assumes the per-contract cache will remain small in practice. It does not account for an attacker deliberately filling it to its maximum of 1500 entries.

**Cache configuration mismatch (the amplification window):**
- `slotsPerContract`: `expireAfterAccess=5m, maximumSize=1500` — keys persist for 5 minutes
- `contractState`: `expireAfterWrite=2s, maximumSize=25000` — values expire every 2 seconds

Once the per-contract cache is full (1500 keys), every 2-second expiry of `contractStateCache` causes the next `findStorage()` call to bypass the fast path at line 63–67 and fall through to `findStorageBatch()`, which fires the full 1500-key SQL query.

**Exploit flow:**

*Phase 1 — One-time cache population (1 HTTP request):*
Deploy or target a contract with a loop that reads 1500 distinct storage slots (e.g., `for i in 0..1499: sload(i)`). Submit a single `POST /api/v1/contracts/call` with `gas=15000000`. The EVM executes 1500 `SLOAD` opcodes; each triggers `findStorage()` → `findStorageBatch()`, which calls `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` for each unique key. After this single request, the per-contract Caffeine cache holds 1500 entries.

*Phase 2 — Sustained triggering (1 HTTP request per 2-second cycle):*
Wait ≥2 seconds. Submit any `POST /api/v1/contracts/call` that reads any slot of the same contract. `contractStateCache.get(...)` returns `null` (expired). `findStorageBatch()` is called, snapshots all 1500 keys, and fires:
```sql
SELECT slot, value FROM contract_state
WHERE contract_id = :contractId AND slot IN (:1500_keys)
```
Repeat every 2 seconds. The per-contract cache stays alive as long as it is accessed within 5 minutes.

**Why existing checks are insufficient:**

The global rate limiter (`requestsPerSecond=500`) operates at the HTTP request level, not at the internal `findStorage()` call level. A single HTTP request can trigger 1500 internal `findStorage()` calls (one per EVM `SLOAD`). The gas throttle (`gasPerSecond=7.5B`) does not prevent a 15M-gas request that reads 1500 slots (1500 × ~800 gas/SLOAD ≈ 1.2M gas, well within the 15M limit). There is no per-IP rate limiting, no cap on the SQL IN-clause size, and no check on the number of keys passed to `findStorageBatch()`.

### Impact Explanation

Each triggered batch query sends a 1500-element `IN`-clause to PostgreSQL against the `contract_state` table. At 30 queries/minute (one every 2 seconds), this is 45,000 slot lookups per minute per attacker. With multiple contracts or multiple attackers, this scales linearly. The DB must parse, plan, and execute large IN-clause queries continuously, causing sustained CPU and I/O pressure that can degrade or deny service to legitimate users. The `contract_state` table is a hot path for all EVM contract calls, so degradation here affects the entire web3 module.

### Likelihood Explanation

The attack requires no credentials, no privileged access, and no on-chain transactions. The attacker only needs:
1. A publicly callable contract (or any existing contract with ≥1500 storage slots) on the network
2. The ability to send HTTP POST requests to `/api/v1/contracts/call`

The setup cost is a single HTTP request. The sustaining cost is one HTTP request every 2 seconds. This is trivially automatable with `curl` or any HTTP client. The attack is repeatable indefinitely and is not self-limiting.

### Recommendation

1. **Cap the batch query size:** In `findStorageBatch()`, limit the number of keys passed to `findStorageBatch` to a configurable maximum (e.g., 100–200), discarding or ignoring excess keys.
2. **Increase `contractState` TTL or reduce `slotsPerContract` max size:** The 2-second TTL on `contractStateCache` combined with a 1500-entry per-contract cache is the core amplification factor. Increasing the TTL to 10–30 seconds would reduce the trigger frequency by 5–15×.
3. **Add per-IP or per-contract rate limiting** at the `findStorageBatch()` level to prevent a single source from repeatedly triggering large batch queries.
4. **Evict slot keys from `contractSlotsCache` after their values are loaded** into `contractStateCache`, so the per-contract cache does not accumulate unboundedly across requests.

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
