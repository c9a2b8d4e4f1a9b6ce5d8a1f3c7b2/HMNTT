### Title
`findRuntimeBytecode()` Caches `Optional.empty()` Due to Flawed `unless` Condition, Enabling Cache Poisoning via Deployment Race

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"`, but the method returns `Optional<byte[]>`. When a contract's `runtime_bytecode` is not yet in the database, the method returns `Optional.empty()`, which is a non-null object. The `unless` condition evaluates to `false`, so the empty Optional is cached. Any caller who queries a contract ID during the importer ingestion lag window poisons the cache for the full TTL, causing all subsequent EVM simulations for that contract to see no bytecode.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18.

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's `unless` SpEL expression is evaluated against the actual return value. The return type is `Optional<byte[]>`. When no row exists (or `runtime_bytecode` is NULL), Spring Data JPA returns `Optional.empty()`. `Optional.empty() != null`, so `#result == null` is `false`, and the `unless` guard does not fire. The empty Optional is stored in the Caffeine cache under `CACHE_NAME_CONTRACT` keyed by `contractId`.

**Exploit flow:**
1. A contract is deployed on the Hedera network. The mirror node importer begins ingesting the deployment transaction. Due to normal processing lag, the `entity` row may exist before the `contract.runtime_bytecode` column is populated.
2. An attacker (any unprivileged user with access to the JSON-RPC endpoint) sends an `eth_call` or `eth_estimateGas` request targeting the newly deployed contract's ID before the importer writes `runtime_bytecode`.
3. `findRuntimeBytecode(contractId)` executes the SQL query, finds no bytecode, and returns `Optional.empty()`.
4. Because `Optional.empty() != null`, the `unless` condition is `false`, and `Optional.empty()` is stored in the Caffeine cache.
5. `ContractBytecodeReadableKVState.readFromDataSource()` receives `Optional.empty()`, maps to `null` (via `.orElse(null)`), and the EVM treats the contract as having no code.
6. For the entire cache TTL (configured via `cacheProperties.getContract()`), every EVM simulation referencing this `contractId` returns no bytecode → the EVM reverts or executes against empty code.

**Why the existing check fails:** The `unless` guard was intended to prevent caching of absent results, but it only guards against a literal `null` return. It does not guard against `Optional.empty()`, which is the actual representation of "not found" for this method.

### Impact Explanation
All `eth_call`, `eth_estimateGas`, and related EVM simulation requests served by the mirror node's web3 module for the affected contract will return incorrect results (REVERT or wrong execution) for the duration of the cache TTL. dApps and wallets that rely on the mirror node's JSON-RPC for pre-flight simulation will receive false failure signals, potentially blocking user transactions or causing incorrect gas estimates. The mirror node is read-only and does not affect on-chain state directly, so no funds are at direct risk, consistent with the Medium severity classification.

### Likelihood Explanation
The Hedera mirror node has a documented ingestion lag between on-chain finality and full DB population. Any user who knows a contract was just deployed (observable from the public Hedera network or mirror node REST API) can trivially time an `eth_call` to the web3 endpoint within this window. No authentication or special privileges are required. The attack is repeatable: the attacker can re-poison the cache after each TTL expiry by issuing another call before the next cache refresh. The window is narrow but predictable and scriptable.

### Recommendation
Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a populated `Optional<byte[]>` (i.e., one where `isPresent()` is `true`) is stored in the cache, preventing negative-result poisoning.

### Proof of Concept
1. Observe a contract deployment transaction on the Hedera network (e.g., via the mirror node REST API `/api/v1/contracts`).
2. Immediately after the contract ID appears (but before the importer finishes writing `runtime_bytecode`), send:
   ```
   POST /api/v1/contracts/call
   { "to": "<new_contract_address>", "data": "0x", "estimate": false }
   ```
3. Observe the response returns a revert or empty result.
4. Wait for the importer to finish (verify via REST that `runtime_bytecode` is now populated).
5. Send the same `eth_call` again — it still returns the cached empty result (no bytecode) for the remainder of the TTL.
6. After TTL expiry, the same call now correctly returns the contract's execution result, confirming the cache was poisoned with `Optional.empty()`. [1](#0-0) [2](#0-1) [3](#0-2)
