### Title
Cache Thrashing DoS via User-Controlled Historical Block Timestamps in `SystemFileLoader.load()`

### Summary
The `exchangeRatesCacheManager` in `SystemFileLoader` uses the `fee` cache spec (`expireAfterWrite=60m, maximumSize=20`) and shares a single Caffeine cache across three file types (exchange rate, fee schedule, simple fee schedule). Because the cache key is `(fileId, hourRoundedTimestamp)` and the `consensusTimestamp` is derived from the caller-supplied block number in `eth_call`, an unprivileged attacker can cycle through 21+ distinct hourly block timestamps to continuously evict all cache entries, forcing every request to execute a live database query with no cache benefit.

### Finding Description

**Exact code path:**

`CacheProperties.java` line 36 sets the fee cache spec: [1](#0-0) 

`EvmConfiguration.java` lines 145–150 wires the `exchangeRate` bean to this spec: [2](#0-1) 

`SystemFileLoader.load()` lines 111–118 selects the `exchangeRatesCacheManager` for all three fee-related file IDs and rounds the timestamp to the hour, then constructs the cache key: [3](#0-2) 

Lines 127–137 show the manual cache get/put pattern — no atomic `get-or-load`, so a miss unconditionally calls `loadFromDB`: [4](#0-3) 

`loadFromDB` → `loadWithRetry` → `fileDataRepository.getFileAtTimestamp()` is the DB call triggered on every miss: [5](#0-4) 

**Root cause:** The cache key embeds the hour-rounded `consensusTimestamp`, which is derived from the block number supplied by the caller in `eth_call`. The cache has a hard `maximumSize=20` shared across all three file types. With 21 distinct hourly timestamps, Caffeine's size-based eviction removes the oldest entry, and a cycling attacker ensures no key is ever resident when its request arrives.

**Failed assumption:** The design assumes that real-world traffic naturally clusters around a small number of recent hourly slots. It does not account for an adversary deliberately spreading requests across many historical hours.

### Impact Explanation

Every cache miss for exchange rate, fee schedule, or simple fee schedule files triggers `fileDataRepository.getFileAtTimestamp()` — a database query with up to 10 retry attempts on parse failure (`maxRetries=9`): [6](#0-5) 

At the default rate limit of 500 RPS, a sustained attack can generate up to 500 DB queries per second for these system files alone, on top of normal query load. This degrades database throughput for all users of the mirror node, including DeFi applications that depend on the web3 API for fee and exchange rate data used in transaction cost calculations.

### Likelihood Explanation

The attack requires only:
1. HTTP access to the web3 API (no authentication, no special role)
2. Knowledge of 21+ historical block numbers that fall in different hours (publicly available from any block explorer or the mirror node's own REST API)
3. A simple script cycling `eth_call` requests with those block numbers

The web3 API explicitly supports historical calls with arbitrary block numbers, as confirmed by test coverage: [7](#0-6) 

The attack is trivially repeatable and requires no exploit sophistication.

### Recommendation

1. **Increase `maximumSize`**: Raise the `fee` cache `maximumSize` from 20 to a value that accommodates realistic historical query depth (e.g., 500–1000), reducing the feasibility of eviction-based thrashing.
2. **Separate caches per file type**: Give each of the three file types (exchange rate, fee schedule, simple fee schedule) its own `CacheManager` so one file type's thrashing cannot evict entries for others.
3. **Restrict historical block depth**: Enforce a maximum lookback window for `eth_call` historical queries (e.g., reject block numbers older than N hours), preventing attackers from using arbitrarily old timestamps as distinct cache keys.
4. **Use `Cache#get(key, loader)` atomically**: Replace the manual get/put pattern with Caffeine's atomic `get(key, mappingFunction)` to prevent redundant concurrent DB calls on the same key.

### Proof of Concept

```python
import requests, time

# Mirror node web3 endpoint
URL = "http://<mirror-node>/api/v1/contracts/call"

# Collect 21 block numbers, each from a different hour
# (obtainable from the mirror node REST API: /api/v1/blocks?limit=1000)
hourly_blocks = [
    "0x1",      # hour 0
    "0x1C20",   # hour 1
    "0x3840",   # hour 2
    # ... 18 more, each ~3600 seconds apart in block time
]

payload_template = {
    "data": "0x",
    "to": "0x0000000000000000000000000000000000000167",  # exchange rate precompile
    "gas": 100000,
}

# Phase 1: fill cache with 20 entries
for block in hourly_blocks[:20]:
    requests.post(URL, json={**payload_template, "block": block})

# Phase 2: cycle through 21+ timestamps to thrash the cache
while True:
    for block in hourly_blocks:
        requests.post(URL, json={**payload_template, "block": block})
        # Each request misses cache → DB query
        time.sleep(0.002)  # 500 RPS
```

Every iteration of the inner loop causes a full cache miss for the exchange rate file, issuing a `getFileAtTimestamp` DB query on each of the 21 requests, with no cache hits ever occurring.
