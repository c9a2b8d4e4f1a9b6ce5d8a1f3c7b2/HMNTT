### Title
Cache Pollution via Empty Optional Caching in `findByIndex()` Allows Unauthenticated Cache Exhaustion

### Summary
The `@Cacheable` annotation on `RecordFileRepository.findByIndex()` uses `unless = "#result == null"` as its exclusion condition. Because Spring Data JPA returns `Optional.empty()` (not `null`) for missing records, empty results are unconditionally cached. An unauthenticated attacker can flood the fixed-size index cache (10,000 entries, 10-minute TTL) with empty Optionals for non-existent block numbers, evicting legitimate entries and forcing repeated database queries for all real block lookups.

### Finding Description

**Code path:**

`RecordFileServiceImpl.findByBlockType()` dispatches to `recordFileRepository.findByIndex(block.number())` for any `BlockType` that is not the `EARLIEST` or `LATEST` singleton reference. [1](#0-0) 

`BlockType.LATEST` is a specific object instance (`new BlockType("latest", Long.MAX_VALUE)`). The check `block == BlockType.LATEST` uses Java reference equality. Any `BlockType` constructed via `BlockType.of("9223372036854775806")` (or any other numeric string) is a **different object instance**, so it always falls through to `findByIndex()`. [2](#0-1) [3](#0-2) 

`findByIndex()` is annotated:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
``` [4](#0-3) 

**Root cause:** The `unless` SpEL expression checks `#result == null`. Spring Data JPA never returns `null` for an `Optional`-typed repository method — it returns `Optional.empty()` when no row is found. `Optional.empty() != null`, so the condition is always `false` for missing records, and the empty `Optional` is stored in the cache under the queried block number key.

**Cache configuration:** The `CACHE_MANAGER_RECORD_FILE_INDEX` cache is bounded at `maximumSize(10000)` with `expireAfterWrite(10, TimeUnit.MINUTES)`. [5](#0-4) 

**Throttle bypass:** The `ThrottleManagerImpl` only applies to `ContractCallRequest` objects (the `/contracts/call` endpoint). Block-number-based lookups that reach `findByBlockType()` are not subject to this throttle. [6](#0-5) 

### Impact Explanation

An attacker who sends 10,000 requests with distinct non-existent block numbers fills the entire `recordFileIndex` cache with empty `Optional` entries. Caffeine's size-based eviction then displaces all legitimate cached record-file entries. Every subsequent real block lookup (e.g., for a valid historical block used in a contract call) misses the cache and issues a live database query. Sustained at the 10-minute TTL boundary, this permanently degrades cache effectiveness, amplifies database load, and increases latency for all users of the web3 API. No credentials or special privileges are required.

### Likelihood Explanation

The web3 API is publicly accessible with no authentication requirement for block-number queries. The attack requires only an HTTP client capable of sending sequential JSON-RPC or REST requests with varying block numbers. The attacker needs to send only 10,000 distinct requests per 10-minute window to maintain full cache saturation — trivially achievable with any scripting tool. The attack is repeatable, stateless, and leaves no persistent side effects that would alert defenders beyond elevated DB query rates.

### Recommendation

Fix the `unless` condition on `findByIndex()` to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
    unless = "#result == null || !#result.isPresent()"
)
Optional<RecordFile> findByIndex(long index);
```

This prevents `Optional.empty()` results from being stored in the cache, eliminating the pollution vector. The same fix should be reviewed for `findEarliest()` and the `@Cacheable` inside `findByTimestamp()`, which carry the same `unless = "#result == null"` condition. [7](#0-6) 

### Proof of Concept

```bash
# Fill the 10,000-entry cache with empty Optionals using distinct non-existent block numbers
# Adjust HOST and PORT to the deployed web3 endpoint

for i in $(seq 9000000000000000 9000000000010000); do
  curl -s -X POST http://HOST:PORT/ \
    -H "Content-Type: application/json" \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_getBlockByNumber\",\"params\":[\"$(printf '0x%x' $i)\",false],\"id\":1}" \
    > /dev/null &
done
wait

# Verify: a legitimate block lookup now misses the cache and hits the DB.
# Monitor DB query rate — it will spike for every real block lookup after the above.
```

Preconditions: unauthenticated network access to the web3 API port. No special permissions, tokens, or accounts required.
