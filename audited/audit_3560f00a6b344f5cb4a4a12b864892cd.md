### Title
Unbounded Repeated Requests to `/accounts/:id/allowances/crypto` Enable DB Griefing via Maximum-Complexity IN-Clause Queries Without Rate Limiting

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` accepts up to `maxRepeatedQueryParameters` `spender.id=eq:` values per request, which are aggregated into a SQL `IN(...)` clause. While the per-request IN clause size is bounded, there is no per-IP or per-user rate limiting on the REST API, allowing any unauthenticated attacker to flood the service with maximum-complexity queries and degrade database performance for all users.

**Existing bounds on per-request cost:**

`rest/middleware/requestHandler.js` configures `qs` with `arrayLimit: config.query.maxRepeatedQueryParameters` and `throwOnLimitExceeded: true`, rejecting arrays that exceed the limit at parse time: [6](#0-5)

**Root cause — failed assumption:** The design assumes that bounding the IN clause to `maxRepeatedQueryParameters` entries is sufficient to prevent abuse. However, there is **no rate limiting, no per-IP throttling, and no connection-level concurrency cap** on the REST API (the throttle infrastructure found in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` applies only to the web3 module, not the Node.js REST service). An attacker can issue an unlimited number of requests, each at maximum allowed complexity. [8](#0-7) 

### Impact Explanation
Each maximum-complexity request generates a query of the form:
```sql
SELECT * FROM crypto_allowance
WHERE owner = $1
  AND spender IN ($2, ..., $N)   -- N = maxRepeatedQueryParameters
  AND amount > 0
ORDER BY spender
LIMIT $M                          -- M = response.limit.max
```
Flooding with such queries saturates the DB connection pool and query executor, causing latency spikes and timeouts for all other users of the mirror node REST API. The impact is service degradation (griefing) with no economic cost to the attacker and no authentication required.

### Likelihood Explanation
Any unauthenticated external user can craft and send these requests. No special knowledge, tokens, or on-chain resources are required. The attack is trivially scriptable (e.g., `curl` in a loop or a simple HTTP load tool). Because there is no rate limiting on the REST API, the attacker can sustain the attack indefinitely.

### Recommendation
1. **Add rate limiting middleware** to the Express REST application (e.g., `express-rate-limit`) scoped per IP address, applied globally or specifically to expensive endpoints.
2. **Set a DB-level statement timeout** (`statementTimeout` already exists in `db.pool` config — verify it is set to a low value) to bound the maximum wall-clock cost of any single query.
3. **Consider reducing `maxRepeatedQueryParameters`** for the `spender.id` parameter specifically, or add a separate, lower cap for IN-clause parameters on this endpoint.
4. **Add connection pool limits** (`maxConnections` in `db.pool`) and ensure they are tuned to prevent a single client from exhausting the pool.

### Proof of Concept
```bash
# Construct a URL with maxRepeatedQueryParameters spender.id=eq: values and max limit
BASE="https://<mirror-node>/api/v1/accounts/0.0.1001/allowances/crypto"
PARAMS="limit=100"
for i in $(seq 1 10); do
  PARAMS="${PARAMS}&spender.id=eq:0.0.${i}"
done

# Flood with concurrent requests (no authentication required)
while true; do
  curl -s "${BASE}?${PARAMS}" &
done
```
Each iteration issues a maximum-complexity `SELECT` with a 10-element IN clause and limit=100. With no rate limiting, this saturates the DB connection pool and degrades response times for all other API consumers.
