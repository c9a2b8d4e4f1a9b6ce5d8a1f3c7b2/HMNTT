### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `GET /contracts/:contractId/results/:consensusTimestamp` Requests

### Summary
The `getContractResultsByTimestamp` handler in `rest/controllers/contractController.js` issues 6–7 sequential and parallel DB queries per request with no per-IP or global rate limiting on the REST API. Because all valid `(contractId, consensusTimestamp)` pairs are publicly enumerable from blockchain data, an unprivileged attacker can flood this endpoint with concurrent requests, each holding multiple DB pool connections simultaneously, exhausting the shared connection pool and denying service to all other API consumers.

### Finding Description
**Exact code path:**

`getContractResultsByTimestamp` (lines 1002–1042, `rest/controllers/contractController.js`) executes the following DB operations per request:

1. `ContractService.getInvolvedContractsByTimestampAndContractId(timestamp, contractId)` — 1 query (line 1011)
2. `this.getDetailedContractResults(contractDetails, contractId)` — `Promise.all` of **5 concurrent queries** (lines 1016–1017, implemented at lines 1205–1219):
   - `ContractService.getContractResultsByTimestamps`
   - `TransactionService.getEthTransactionByTimestampAndPayerId`
   - `RecordFileService.getRecordFileBlockDetailsFromTimestamp`
   - `ContractService.getContractLogsByTimestamps`
   - `ContractService.getContractStateChangesByTimestamps`
3. Conditionally: `FileDataService.getLatestFileDataContents` (line 1023) — 1 additional query when `callDataId` is valid

Total: **6–7 DB connections held concurrently per in-flight request**.

**Root cause — no rate limiting on the REST API:**
A `grep` across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, `express-rate`, and `throttle` returns **zero matches** in production middleware. The throttling found (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) applies exclusively to the separate Java `web3` module, not to the Node.js REST service. The REST server's middleware stack (`rest/middleware/`) contains only `authHandler`, `httpErrorHandler`, `metricsHandler`, `openapiHandler`, `requestHandler`, `requestNormalizer`, `responseCacheHandler`, and `responseHandler` — none of which implement request-rate limiting.

**DB pool:**
`rest/dbpool.js` (lines 7–16) creates a single `pg.Pool` with `max: config.db.pool.maxConnections`. The pool is finite and shared across all concurrent requests. With 6 connections consumed per request, a pool of size N is exhausted by only ⌈N/6⌉ concurrent attacker requests.

**Why the early-exit check is insufficient:**
The `if (!contractDetails) { throw new NotFoundError(); }` guard at line 1012–1014 only fires for non-existent pairs. An attacker uses **valid** pairs (publicly available), so this check never triggers and all 5+ subsequent queries always execute.

### Impact Explanation
When the DB connection pool is exhausted, every other API endpoint that requires a DB connection (transactions, accounts, tokens, etc.) begins queuing or timing out. The `connectionTimeoutMillis` setting in `rest/dbpool.js` (line 13) means queued requests will eventually fail with connection timeout errors, returning 500s to all legitimate users. This constitutes a full REST API denial of service affecting the entire mirror node, not just the contracts endpoint. Given that Hedera mirror nodes serve as critical infrastructure for wallets, explorers, and DeFi applications, the blast radius is significant.

### Likelihood Explanation
The attack requires zero privileges — no API key, no account, no authentication. Valid `(contractId, consensusTimestamp)` pairs are trivially enumerable by querying `GET /api/v1/contracts/results` (a paginated public endpoint) or from any public Hedera explorer. The attacker needs only a standard HTTP client capable of issuing concurrent requests (e.g., `ab`, `wrk`, or a simple async script). The attack is repeatable and stateless — each request is independent, so there is no session to invalidate. The absence of any rate-limiting middleware in the REST service means there is no server-side defense to overcome.

### Recommendation
1. **Add rate limiting middleware** to the Express REST server (e.g., `express-rate-limit`) applied globally or specifically to the `/contracts/:contractId/results/:timestamp` route, limiting requests per IP per time window.
2. **Reduce query fan-out**: Consider combining the 5 `Promise.all` queries in `getDetailedContractResults` (lines 1205–1219) into fewer, joined SQL queries to reduce per-request connection pressure.
3. **Set a conservative `maxConnections`** and ensure `connectionTimeoutMillis` is low enough to shed load quickly rather than queue indefinitely.
4. **Consider a reverse-proxy layer** (e.g., nginx, Cloudflare) with rate limiting as a defense-in-depth measure.

### Proof of Concept
```bash
# Step 1: Enumerate valid (contractId, timestamp) pairs from the public API
curl "https://<mirror-node>/api/v1/contracts/results?limit=100" \
  | jq -r '.results[] | "\(.contract_id) \(.timestamp)"' > pairs.txt

# Step 2: Flood the endpoint with concurrent requests
# Using GNU parallel or wrk; each request triggers 6+ DB queries
cat pairs.txt | parallel -j 200 \
  'curl -s "https://<mirror-node>/api/v1/contracts/{1}/results/{2}" > /dev/null'

# Step 3: Observe that other API endpoints begin returning 500 errors
# due to DB connection pool exhaustion
curl "https://<mirror-node>/api/v1/transactions?limit=1"
# Expected: connection timeout / 500 Internal Server Error
```

With a default pool size (typically 10–20 connections) and 6 connections consumed per request, as few as 4–20 concurrent attacker requests are sufficient to saturate the pool.
