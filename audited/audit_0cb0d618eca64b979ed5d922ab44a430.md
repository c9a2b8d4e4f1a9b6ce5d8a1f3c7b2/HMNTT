### Title
Object Injection via Bracket Notation in `requestQueryParser` Causes Unhandled TypeError DoS

### Summary
The `qs` parser is configured with `depth: 1`, which allows one level of nested bracket notation (e.g., `result[key]=success`) to be parsed as a plain object `{ key: 'success' }` rather than a string. The `lowerCaseQueryValue` function silently passes non-string values through unchanged, and the downstream `buildComparatorFilter` function unconditionally calls `.split(':')` on the value — crashing with a `TypeError` when the value is an object. Any unauthenticated user can trigger this on any endpoint accepting a `result` or `order` query parameter.

### Finding Description

**Code path:**

`rest/middleware/requestHandler.js` — `queryOptions` sets `depth: 1` and `strictDepth: true`:

```js
const queryOptions = {
  arrayLimit: config.query.maxRepeatedQueryParameters,
  depth: 1,          // allows one level of nesting — result[key]=success parses as { key: 'success' }
  strictDepth: true, // only throws when depth is EXCEEDED (> 1), not at exactly depth 1
  throwOnLimitExceeded: true,
};
``` [1](#0-0) 

With `depth: 1`, `qs.parse('result[key]=success')` produces `{ result: { key: 'success' } }`. The `strictDepth: true` guard only throws a `RangeError` when depth is **exceeded** (i.e., `result[a][b]=x` with `depth: 1`). A single bracket level is within the limit and parses silently.

**`canonicalizeValue` passes the object through:**

```js
const canonicalizeValue = (key, value) => {
  const canonicalizationFunc = queryCanonicalizationMap[key]; // lowerCaseQueryValue for 'result'
  ...
  return Array.isArray(value) ? value.map(...) : canonicalizationFunc(value);
  // { key: 'success' } is not an Array → calls lowerCaseQueryValue({ key: 'success' })
};
``` [2](#0-1) 

**`lowerCaseQueryValue` silently returns the object unchanged:**

```js
const lowerCaseQueryValue = (queryValue) =>
  (typeof queryValue === 'string' ? queryValue.toLowerCase() : queryValue);
  // typeof { key: 'success' } !== 'string' → returns the object as-is
``` [3](#0-2) 

**`buildComparatorFilter` crashes with TypeError:**

```js
const buildComparatorFilter = (name, filter) => {
  const splitVal = filter.split(':'); // TypeError: filter.split is not a function
  ...
};
``` [4](#0-3) 

`buildFilters` routes non-array values directly to `buildComparatorFilter` with no type guard:

```js
} else {
  filters.push(buildComparatorFilter(key, values)); // values = { key: 'success' }
}
``` [5](#0-4) 

The `TypeError` propagates through `buildAndValidateFilters` → route handler → Express error middleware, producing an unhandled 500 response. [6](#0-5) 

### Impact Explanation
Every REST API endpoint that accepts a `result` or `order` query parameter (e.g., `/api/v1/transactions`, `/api/v1/contracts/results`) can be crashed into returning HTTP 500 with a single malformed request. The `queryCanonicalizationMap` explicitly lists both `result` and `order` as targets. [7](#0-6) 
The mirror node REST API is the primary read interface for the Hedera network; sustained DoS degrades all client applications relying on it for transaction status, account queries, and contract results.

### Likelihood Explanation
No authentication or special privileges are required. The payload is a standard HTTP query string using bracket notation supported by every HTTP client and browser. It is trivially repeatable and automatable. The `depth: 1` setting was presumably intended to block deep nesting but inadvertently permits exactly one level — the minimum needed to inject an object for canonicalized keys.

### Recommendation
1. **Set `depth: 0`** in `queryOptions` to prevent any bracket-notation nesting from being parsed as objects. This is the correct setting for flat query parameters.
2. **Add a type guard in `buildComparatorFilter`** (or `buildFilters`) to reject non-string, non-array values with an `InvalidArgumentError` before calling `.split()`.
3. **Add a type guard in `canonicalizeValue`**: if `value` is a plain object (not string, not array), throw or return an error immediately rather than passing it downstream.

### Proof of Concept

```
GET /api/v1/transactions?result[key]=success HTTP/1.1
Host: <mirror-node-host>
```

**Step-by-step:**
1. `qs.parse('result[key]=success', { depth: 1, strictDepth: true })` → `{ result: { key: 'success' } }`
2. `canonicalizeValue('result', { key: 'success' })` → calls `lowerCaseQueryValue({ key: 'success' })` → returns `{ key: 'success' }` unchanged
3. `requestQueryParser` sets `caseInsensitiveQueryString.result = { key: 'success' }`
4. Route handler calls `buildAndValidateFilters(query, ...)` → `buildFilters(query)` → `buildComparatorFilter('result', { key: 'success' })`
5. `{ key: 'success' }.split(':')` → `TypeError: filter.split is not a function`
6. Server returns HTTP 500; repeating the request continuously denies service to all legitimate users of that endpoint.
