### Title
Unauthenticated Connection-Abort Causes Permanent `inFlightCounter` Inflation in `metricsHandler`

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function unconditionally increments `inFlightCounter` on every non-metrics request but only decrements it inside a `res.on('finish', ...)` callback. In Node.js, the `finish` event is not emitted when a client aborts the TCP connection before the server completes the response. Because there is no `res.on('close', ...)` guard and no authentication or rate-limiting on non-metrics endpoints, any unprivileged attacker can permanently inflate the `api_all_request_in_processing_total` metric by repeatedly sending requests and immediately aborting the connection.

### Finding Description
**Exact code path:**

`rest/middleware/metricsHandler.js`, function `metricsMiddleware` (returned by `metricsHandler()`):

- **Line 169** — `inFlightCounter.add(1)` executes unconditionally for every non-metrics request.
- **Lines 185–186** — `inFlightCounter.add(-1)` executes only inside `res.on('finish', ...)`.

```js
// line 169
inFlightCounter.add(1);

// lines 185-186
res.on('finish', () => {
  inFlightCounter.add(-1);   // ← only path that decrements
  ...
});
```

**Root cause / failed assumption:** The code assumes that `res.on('finish')` will always fire for every request that passes through the middleware. This assumption is wrong. In Node.js HTTP, `finish` is emitted only when `res.end()` is called and the data is successfully flushed to the OS socket buffer. When a client aborts the TCP connection (sends RST or closes the socket) before the server writes the response, the underlying socket is destroyed. A subsequent call to `res.end()` on a destroyed socket does not flush data, and the `finish` event is never emitted. The `close` event fires instead, but there is no `close` listener in this middleware.

**No rate-limiting or authentication on non-metrics endpoints:** `rest/server.js` lines 88–92 show the metrics middleware is applied globally with `app.useExt(metricsHandler())`. There is no per-IP rate-limiting, connection throttling, or authentication middleware applied before it for regular API routes.

**Exploit flow:**
1. Attacker opens a TCP connection to the server and sends a valid HTTP GET to any public endpoint (e.g., `GET /api/v1/transactions`).
2. Immediately after sending the request headers, the attacker sends a TCP RST (or closes the socket).
3. The middleware has already executed line 169 — `inFlightCounter.add(1)`.
4. The socket is destroyed. When Express eventually calls `res.end()`, the write fails silently on the destroyed socket.
5. `finish` is never emitted → `inFlightCounter.add(-1)` is never called.
6. The counter is permanently inflated by +1.
7. Repeating this N times inflates the counter by N, with no natural reset.

### Impact Explanation
The `api_all_request_in_processing_total` UpDownCounter is the primary signal operators use to judge current server load and make horizontal scaling decisions. Permanently inflating it causes operators to believe the server is under sustained heavy load when it is not, leading to unnecessary over-provisioning (wasted cost) or, if thresholds are inverted, masking real load spikes. Because the counter never resets between process restarts (it accumulates in-memory), the only recovery is a process restart. The attack is silent — no error logs are generated, and the metrics endpoint itself (which requires authentication) will faithfully report the inflated value.

### Likelihood Explanation
The attack requires zero privileges — all non-metrics API endpoints are publicly accessible. It requires only a standard TCP client (e.g., `curl`, `ab`, a raw socket script). The abort-before-response pattern is trivially scriptable: open socket → send HTTP request line → immediately `close()`/`RST`. The attack is repeatable at high frequency from a single machine or distributed across IPs. There is no existing rate-limiting, connection-count cap, or counter TTL that would prevent or bound the inflation.

### Recommendation
Add a `res.on('close', ...)` listener that decrements the counter only if `finish` has not already fired, preventing double-decrement:

```js
let finished = false;

res.on('finish', () => {
  finished = true;
  inFlightCounter.add(-1);
  // ... rest of metrics recording
});

res.on('close', () => {
  if (!finished) {
    inFlightCounter.add(-1);
  }
});
```

This mirrors the pattern used by well-known Node.js HTTP instrumentation libraries (e.g., `on-finished` npm package) and ensures the counter is always decremented regardless of how the connection terminates.
