### Title
Unbounded Timestamp Array Growth via Case-Variant Key Merging Bypasses `arrayLimit` Protection

### Summary
The `requestQueryParser()` function in `rest/middleware/requestHandler.js` uses `qs.parse()` with `throwOnLimitExceeded: true` and `arrayLimit: config.query.maxRepeatedQueryParameters` to cap repeated query parameters. However, the subsequent `merge()` loop combines all case-variant spellings of the same key (e.g., `timestamp`, `Timestamp`, `TIMESTAMP`, `tImestamp`, …) into a single array without any length check on the merged result. An unprivileged attacker can send up to `maxRepeatedQueryParameters` values under each of the 2⁹ = 512 case variants of `timestamp`, producing a merged array of up to `512 × maxRepeatedQueryParameters` entries that `parseTimestampQueryParam()` then iterates over one-for-one to build a WHERE clause.

### Finding Description

**Exact code path:**

`rest/middleware/requestHandler.js`, `requestQueryParser()`, lines 15–68.

`qs.parse()` is called with the guard options: [1](#0-0) 

Because `qs` is case-sensitive, `timestamp`, `Timestamp`, and `TIMESTAMP` are three distinct keys. Each key's value array is independently capped at `arrayLimit`. After `qs.parse()` returns, the loop lowercases every key and calls `merge()` when the lowercased key already exists: [2](#0-1) 

The `merge()` helper appends without any length guard: [3](#0-2) 

The merged array is then consumed by `parseTimestampQueryParam()`: [4](#0-3) 

which calls `parseParams` and emits one `timestamp <op> ?` clause per entry, building a WHERE clause proportional to the total merged array length.

**Root cause / failed assumption:** The developers assumed `throwOnLimitExceeded: true` in `qs` bounds the total number of timestamp values. It does not — it bounds each individual (case-sensitive) key independently. The `merge()` step that collapses case variants has no corresponding length check, so the per-key cap is trivially multiplied by the number of case variants.

### Impact Explanation
For a 9-character key like `timestamp`, there are 2⁹ = 512 distinct case spellings. If `maxRepeatedQueryParameters` is 100 (a typical default), an attacker can inject 51,200 timestamp conditions into a single SQL query. Every affected endpoint (e.g., `/api/v1/balances`, `/api/v1/transactions`) that calls `parseTimestampQueryParam()` will forward this enormous WHERE clause to PostgreSQL, causing severe query-planner overhead and potentially exhausting database CPU/memory. No authentication is required. The impact is denial-of-service at the database tier with no economic cost to the attacker.

### Likelihood Explanation
The attack requires only an HTTP GET request with a crafted query string — no credentials, no special network position. It is trivially scriptable, repeatable at high frequency, and requires no knowledge of the application beyond the public API. Any public-facing deployment is exposed.

### Recommendation
After the `merge()` loop completes, enforce the same `maxRepeatedQueryParameters` limit on the merged array before returning:

```js
if (Array.isArray(caseInsensitiveQueryString[lowerKey]) &&
    caseInsensitiveQueryString[lowerKey].length > config.query.maxRepeatedQueryParameters) {
  throw new Error(`Query parameter '${lowerKey}' exceeds maximum allowed repetitions`);
}
```

Alternatively, cap inside `merge()` itself and throw (or truncate) when the combined length would exceed the limit. Additionally, ensure `validateReq` / `isRepeatedQueryParameterValidLength` is called on the post-merge array for every endpoint. [5](#0-4) 

### Proof of Concept

```
GET /api/v1/transactions?timestamp=gte:1000000000\
&Timestamp=gte:1000000001\
&tImestamp=gte:1000000002\
&tiMestamp=gte:1000000003\
... (repeat for all 512 case variants, each with maxRepeatedQueryParameters values)
```

Concrete minimal test (assuming `maxRepeatedQueryParameters = 10`):

```js
import {requestQueryParser} from './middleware/requestHandler.js';

// Build 512 case variants × 10 values each
const parts = [];
for (let mask = 0; mask < 512; mask++) {
  const key = 'timestamp'.split('').map((c, i) => (mask >> i) & 1 ? c.toUpperCase() : c).join('');
  for (let v = 0; v < 10; v++) {
    parts.push(`${key}=gte:${1000 + v}`);
  }
}
const qs = parts.join('&');
const parsed = requestQueryParser(qs);
console.log(parsed.timestamp.length); // prints 5120 — far above the intended limit of 10
```

The resulting `parsed.timestamp` array of 5120 entries is then passed to `parseTimestampQueryParam()`, which emits 5120 SQL conditions into the WHERE clause.
