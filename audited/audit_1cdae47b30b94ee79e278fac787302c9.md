### Title
Shared Singleton `LinkedList` in `MirrorNodeState` Causes Cross-Request Transaction Receipt Leakage

### Summary
`MirrorNodeState` is a Spring singleton (`@Named`) that initializes a single `LinkedList` for `TRANSACTION_RECEIPTS_STATE_ID` at startup. Both `ListWritableQueueState.addToDataSource()` and `ListReadableQueueState.iterateOnDataSource()` operate directly on this shared, unsynchronized `LinkedList` with no per-request isolation. Any concurrent request that reads the queue via the cached `ListReadableQueueState` will observe receipts written by other requests, enabling cross-request transaction receipt leakage.

### Finding Description

**Code path:**

`MirrorNodeState.initQueueStates()` (line 171) creates one `LinkedList` for the lifetime of the application:
```java
states.put(RecordCacheService.NAME, new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID, new LinkedList<>())));
``` [1](#0-0) 

`getReadableStates()` (lines 67–87) uses `computeIfAbsent` on the `readableStates` `ConcurrentHashMap`. On first call for `RecordCacheService.NAME`, it constructs a `ListReadableQueueState` wrapping the shared `LinkedList` and caches the entire `MapReadableStates`:
```java
if (state instanceof Queue<?> queue) {
    data.put(stateId, new ListReadableQueueState<>(serviceName, stateId, queue));
``` [2](#0-1) 

`ListReadableQueueState.iterateOnDataSource()` returns a live iterator directly over the backing `LinkedList`:
```java
protected Iterator<E> iterateOnDataSource() {
    return backingStore.iterator();
}
``` [3](#0-2) 

`getWritableStates()` (lines 91–116) also uses `computeIfAbsent` and creates a `ListWritableQueueState` wrapping the same shared `LinkedList`. Its commit callback only removes the *readable* states cache entry (`readableStates.remove(serviceName)`), leaving the writable states permanently cached:
```java
return new MapWritableStates(data, () -> readableStates.remove(serviceName));
``` [4](#0-3) 

`ListWritableQueueState.addToDataSource()` writes directly to the shared `LinkedList` with no buffering:
```java
protected void addToDataSource(@NonNull E element) {
    backingStore.add(element);
}
``` [5](#0-4) 

**Root cause:** The failed assumption is that each request gets an isolated queue. In reality, the `LinkedList` is a process-wide singleton. `addToDataSource()` mutates it immediately and `iterateOnDataSource()` reads it live, so every caller of `getReadableStates(RecordCacheService.NAME).getQueue(TRANSACTION_RECEIPTS_STATE_ID)` sees the accumulated receipts of all prior and concurrent requests.

**Why the `computeIfAbsent` "pin" matters:** Once the `MapReadableStates` is cached (first call), all subsequent calls return the same `ListReadableQueueState` instance. Even after a commit clears the readable-states cache entry and a new `ListReadableQueueState` is constructed, it still wraps the same never-cleared `LinkedList`. The queue grows monotonically across all requests.

**Thread-safety failure:** `LinkedList` is not thread-safe. Concurrent `add()` from one thread and `iterator()` from another will throw `ConcurrentModificationException` or silently corrupt iteration results. [6](#0-5) 

### Impact Explanation
Any request that reads `TRANSACTION_RECEIPTS_STATE_ID` (e.g., during `eth_sendRawTransaction` or `eth_getTransactionReceipt` processing) will observe receipts belonging to other concurrent or prior requests. Transaction receipts contain transaction IDs, account IDs, and status information. This constitutes cross-user information disclosure. Additionally, the `LinkedList` grows without bound, leading to unbounded memory growth and eventual OOM. Concurrent access without synchronization can cause `ConcurrentModificationException`, crashing request threads.

### Likelihood Explanation
Any unprivileged user submitting JSON-RPC requests (`eth_sendRawTransaction`) to the mirror node's web3 endpoint triggers the write path. A second concurrent request reading the same queue immediately observes the first request's receipt. No special privileges, authentication bypass, or race-condition timing is required beyond sending two overlapping requests. This is trivially repeatable under any load.

### Recommendation
1. **Per-request queue isolation:** Replace the singleton `LinkedList` with a per-request or per-thread queue. Use a `ThreadLocal<Queue<?>>` or create a fresh `LinkedList` inside each `getReadableStates()`/`getWritableStates()` invocation for the `RecordCacheService` queue, rather than caching it in the singleton maps.
2. **Thread-safe backing store:** If a shared queue is intentional (e.g., a global receipt cache), replace `LinkedList` with `ConcurrentLinkedQueue` and add proper size-bounding/eviction logic.
3. **Clear writable states cache on commit:** The commit callback should also call `writableStates.remove(serviceName)` so stale writable state wrappers are not reused across requests.
4. **Clear the queue after each request:** If the queue is meant to be ephemeral per-transaction, explicitly drain it after the transaction completes.

### Proof of Concept
```
1. Start the mirror node web3 service (MirrorNodeState is instantiated as a Spring singleton).

2. Thread A (Request 1):
   - Calls getWritableStates("RecordCacheService")
   - Gets the cached ListWritableQueueState wrapping the shared LinkedList
   - Calls queue.add(receiptForUserA)  → addToDataSource() → LinkedList.add(receiptForUserA)
   - Does NOT yet call commit()

3. Thread B (Request 2, concurrent):
   - Calls getReadableStates("RecordCacheService")
   - Gets the cached MapReadableStates containing the ListReadableQueueState
   - Calls queue.iterator() → iterateOnDataSource() → LinkedList.iterator()
   - Iterates and observes receiptForUserA (belonging to Request 1)

4. Even after Thread A commits (which removes the readable-states cache entry),
   the next call to getReadableStates() creates a new ListReadableQueueState
   wrapping the SAME LinkedList, which still contains receiptForUserA.
   The queue is never cleared.

5. Result: Request 2 has read the transaction receipt of Request 1 without authorization.
```

    public ReadableStates getReadableStates(@NonNull String serviceName) {
        return readableStates.computeIfAbsent(serviceName, s -> {
            final var serviceStates = this.states.get(s);
            if (serviceStates == null) {
                return new MapReadableStates(new HashMap<>());
            }
            final Map<Integer, Object> data = new ConcurrentHashMap<>();
            for (final var entry : serviceStates.entrySet()) {
                final var stateId = entry.getKey();
                final var state = entry.getValue();
                if (state instanceof Queue<?> queue) {
                    data.put(stateId, new ListReadableQueueState<>(serviceName, stateId, queue));
                } else if (state instanceof ReadableKVState<?, ?> kvState) {
                    data.put(stateId, kvState);
                } else if (state instanceof SingletonState<?> singleton) {
                    data.put(stateId, new FunctionReadableSingletonState<>(serviceName, stateId, singleton));
                }
            }
            return new MapReadableStates(data);
        });
    }
