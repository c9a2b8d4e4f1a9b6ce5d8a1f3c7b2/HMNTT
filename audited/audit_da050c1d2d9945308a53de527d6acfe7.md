### Title
Unauthenticated Query Amplification DoS via Correlated Subquery in `/api/v1/balances` (No-Timestamp Path)

### Summary
When `GET /api/v1/balances` is called without a `timestamp` filter, `getBalances()` builds a query that embeds a correlated subquery (`getTokenAccountBalanceSubQuery`) in the SELECT list, causing PostgreSQL to execute it once per outer row. An unauthenticated attacker can set `limit` to the configured maximum (default 100) and flood the endpoint concurrently, forcing the database to execute up to 100 correlated subqueries per request against the `token_account` table with no rate-limiting gate.

### Finding Description

**Exact code path:**

`rest/balances.js`, `getBalances()`, lines 133–149:

```js
} else {
  // use current balance from entity table when there's no timestamp query filter
  const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
  const whereClause = conditions && `where ${conditions}`;
  const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);   // correlated subquery
  sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
  sqlQuery = `
    with entity_balance as (
      select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
      from entity
      where type in ('ACCOUNT', 'CONTRACT')
    )
    select ab.*, (${tokenBalanceSubQuery}) as token_balances   // <-- runs once per row
    from entity_balance ab
    ${whereClause}
    order by ab.account_id ${order}
    ${limitQuery}`;                                            // LIMIT applied last
}
```

`getTokenAccountBalanceSubQuery` (lines 308–318):

```js
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true   // correlated on ab.account_id
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}   // 50 by default
    ) as account_token_balance`;
};
```

**Root cause:** The correlated subquery is placed in the SELECT list of the outer query. PostgreSQL evaluates it once for every row returned by `entity_balance` before the `LIMIT` clause prunes the result set. With `ORDER BY account_id + LIMIT`, the planner can stop after `limit` rows, but that still means exactly `limit` correlated subquery executions per request — each independently hitting `token_account`. The `limit` parameter is capped at the configured maximum (default **100**, confirmed by config test: `{default: 25, max: 100, tokenBalance: {multipleAccounts: 50}}`).

**Failed assumption:** The designers assumed the outer `LIMIT` sufficiently bounds DB work. It does bound the number of correlated subquery executions to `limit`, but the amplification factor is `limit × tokenBalanceLimit.multipleAccounts` = 100 × 50 = **5,000 index lookups per single HTTP request**. With no authentication and no rate limiting visible in the REST layer, this is freely repeatable.

### Impact Explanation

Each unauthenticated HTTP request to `/api/v1/balances` (no `timestamp`, `limit=100`) causes PostgreSQL to execute 100 correlated subqueries against `token_account`. On a production Hedera mainnet mirror node with millions of accounts and token associations, each subquery performs an index scan on `(account_id, associated)`. Flooding the endpoint with concurrent requests (e.g., 50 concurrent clients) generates 5,000 concurrent correlated subquery executions per second, exhausting DB connection pool capacity and CPU, degrading or denying service to all other API consumers. Severity: **Medium-High** (public API, no privilege required, repeatable, measurable DB amplification).

### Likelihood Explanation

Any external user can reproduce this with a single `curl` loop — no credentials, no special knowledge, no exploit tooling. The endpoint is documented in the public OpenAPI spec (`/api/v1/balances`, `operationId: getBalances`). The attacker only needs to omit `timestamp` and set `limit=100`. Automated scripts can sustain thousands of requests per minute from a single machine or botnet.

### Recommendation

1. **Add rate limiting** at the API gateway or middleware layer for `/api/v1/balances` (and similar endpoints), keyed on source IP, with a burst allowance.
2. **Rewrite the no-timestamp query** to avoid the correlated subquery pattern. Instead, use a lateral join or a two-step approach: first fetch the `limit` accounts, then batch-fetch their token balances in a single `IN (...)` query against `token_account`.
3. **Require `timestamp` filter** or at minimum enforce a tighter default/max limit for the no-timestamp path, since it is already documented as not recommended for general use.
4. **Add a `statement_timeout`** on the DB connection used by this endpoint to bound worst-case query duration.

### Proof of Concept

```bash
# No authentication, no timestamp, max limit — triggers 100 correlated subqueries per request
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/balances?limit=100&order=desc" -o /dev/null &
done
wait
# 200 concurrent requests × 100 correlated subqueries = 20,000 token_account index scans
# Monitor DB CPU / connection pool saturation
```

**Trigger:** Omit `timestamp` parameter; set `limit=100`.
