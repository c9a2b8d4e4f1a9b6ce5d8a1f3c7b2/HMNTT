### Title
Unauthenticated Maximum-Range UNION ALL Query Forces Maximum-Cost NFT Table Scan via `getNftsByAccountId`

### Summary
Any unauthenticated caller can supply `token.id=gte:1&token.id=lte:9223372036854775807&serialnumber=gte:1&serialnumber=lte:9223372036854775807&limit=100` to the `/api/v1/accounts/{id}/nfts` endpoint. This combination passes all validation and causes `extractNftMultiUnionQuery` to emit a three-part `UNION ALL` query where the inner sub-query covers the entire valid token-ID range for the target account. Each of the three sub-queries carries its own `LIMIT 100`, so the database executes three separate index scans (up to 300 rows fetched) before the outer `LIMIT 100` is applied. There is no rate-limiting, no authentication requirement, and no check that prevents maximum-range bounds.

### Finding Description

**Code path:**

`getNftsByAccountId` (line 90) calls `buildAndValidateFilters` with the default `filterValidityChecks` validator, then `extractNftMultiUnionQuery`. [1](#0-0) 

Inside `extractNftMultiUnionQuery`, `token.id` is parsed into `bounds.primary` and `serialnumber` into `bounds.secondary`: [2](#0-1) 

**Validation is insufficient:**

`validateLowerBounds` only throws when `primary.lower.operator === opsMap.gt` (strict `>`). Because the attacker uses `gte`, the condition is false and no error is raised: [3](#0-2) 

`validateUpperBounds` only throws when `primary.upper.operator === opsMap.lt` (strict `<`). Because the attacker uses `lte`, the condition is false and no error is raised: [4](#0-3) 

**Three-part UNION ALL is generated:**

With both primary and secondary having lower+upper bounds, `getLowerFilters`, `getInnerFilters`, and `getUpperFilters` all return non-empty arrays: [5](#0-4) 

`NftService.getQuery` then wraps all three in a `UNION ALL`: [6](#0-5) 

The resulting SQL is:
```sql
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id = $1 AND token_id = 1 AND serial_number >= 1
 ORDER BY token_id DESC, serial_number DESC LIMIT 100)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id = $1 AND token_id > 1 AND token_id < 9223372036854775807
 ORDER BY token_id DESC, serial_number DESC LIMIT 100)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id = $1 AND token_id = 9223372036854775807 AND serial_number <= 9223372036854775807
 ORDER BY token_id DESC, serial_number DESC LIMIT 100)
ORDER BY token_id DESC, serial_number DESC LIMIT 100
```

The inner sub-query covers `token_id > 1 AND token_id < MAX_LONG` — effectively the entire token-ID space for the account.

**Value validation does not block max-range values:**

`serialnumber` is validated with `isPositiveLong`, which accepts `9223372036854775807` (it equals `maxLong`): [7](#0-6) 

`token.id` is validated with `EntityId.isValidEntityId`, which also accepts the max-long numeric value: [8](#0-7) 

The `limit` is capped at 100 (the configured max), so `limit=100` is accepted as-is: [9](#0-8) 

### Impact Explanation
For any account holding a large number of NFTs across many token IDs, this forces the database to execute three separate index scans (each up to 100 rows) instead of one. Repeated calls from one or more unauthenticated clients can saturate DB connection pool resources and degrade query performance for all users of the mirror node. The impact is service degradation (griefing), not economic loss.

### Likelihood Explanation
The endpoint is public and requires no credentials. The parameter combination is trivially constructable from the OpenAPI spec. An attacker needs only a valid account ID (publicly enumerable from the same API) and can loop the request indefinitely. No special knowledge or tooling is required.

### Recommendation
1. **Reject open-ended serial-number ranges when a token-ID range is also present.** If both `token.id` and `serialnumber` carry range operators (not `eq`), require that the token-ID range is narrow (e.g., enforce a maximum span).
2. **Add rate limiting** per IP/client on the `/accounts/{id}/nfts` endpoint.
3. **Alternatively**, disallow the combination of a token-ID range with a serial-number range entirely, forcing callers to use either a token-ID range alone or a single `token.id=eq:X` with a serial-number range — this eliminates the three-part UNION ALL path for unauthenticated callers.

### Proof of Concept
```
# Precondition: identify any account with many NFTs, e.g. account 0.0.12345
# No authentication required

curl "https://<mirror-node>/api/v1/accounts/0.0.12345/nfts?\
token.id=gte:1&token.id=lte:9223372036854775807&\
serialnumber=gte:1&serialnumber=lte:9223372036854775807&\
limit=100"

# Repeat in a tight loop from multiple clients to amplify DB load.
# Each request triggers the three-part UNION ALL with maximum range bounds.
# No 400 error is returned; the query executes successfully.
```

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }

// The max signed long has 19 digits
const positiveLongRegex = /^\d{1,19}$/;
const maxLong = 9223372036854775807n;

/**
 * Validates that num is a positive BigInt.
 * @param {number|string} num
 * @param {boolean} allowZero
 * @return {boolean}
 */
const isPositiveLong = (num, allowZero = false) => {
  if (!positiveLongRegex.test(num)) {
    return false;
  }

  const bigInt = BigInt(num);
  const min = allowZero ? 0 : 1;
  return bigInt >= min && bigInt <= maxLong;
};
