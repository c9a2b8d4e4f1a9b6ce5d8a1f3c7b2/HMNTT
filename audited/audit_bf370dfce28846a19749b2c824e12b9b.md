### Title
Gas Throttle Integer Truncation Enables ~2x EVM Computation Per Token

### Summary
The `scaleGas()` method in `ThrottleProperties` uses `Math.floorDiv(gas, 10_000)`, causing all gas values in the range [10,001–19,999] to consume exactly 1 token from the gas bucket. An unprivileged attacker can submit requests with gas=19,999 that drive up to ~2x more EVM computation than gas=10,001 while consuming the same single token, amplifying CPU load on mirror node processing instances without triggering the gas throttle.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleProperties.java`, lines 42–47:
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

**Root cause — failed assumption:**
The design assumes that `scaleGas(gas)` produces a token cost proportional to the gas limit. `Math.floorDiv` truncates the fractional part, so every integer in [N×10,000+1, (N+1)×10,000−1] maps to the same token count N. For N=1, the entire range [10,001–19,999] costs exactly 1 token, even though the upper end allows ~2x more EVM computation than the lower end.

**Exploit flow:**

1. `ThrottleManagerImpl.throttle()` (line 40) calls `gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))`.
   - `scaleGas(10_001)` → 1 token consumed
   - `scaleGas(19_999)` → 1 token consumed (identical cost)

2. The EVM executes with the full gas limit supplied. A contract crafted to consume exactly 19,999 gas executes ~2x more opcodes than one consuming 10,001 gas.

3. After execution, `restoreGasToBucket()` (lines 140–152 of `ContractCallService.java`) calls `throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline))`. If the attacker's contract uses all 19,999 gas, `gasRemaining = 0`, so `restore(0)` → `scaleGas(0) = 0` tokens refunded. Net cost: 1 token for 19,999 gas of computation.

4. For comparison, gas=10,001 with all gas consumed: also 1 token, 10,001 gas of computation. The attacker gets ~2x computation per token.

**Why existing checks are insufficient:**

- The **rate limit** (`requestsPerSecond = 500`) caps requests per second but does not account for the gas-per-request discrepancy. At 500 req/s with gas=19,999, the attacker drives ~9,999,500 gas/second of EVM computation while consuming only 500 tokens/second from the gas bucket — nearly double what the throttle intends to allow at that token rate.
- The **gas bucket** (`gasPerSecond = 7,500,000,000`, scaled to 750,000 tokens/second) is far more permissive than the rate limit, so the rate limit is the binding constraint. The truncation amplifies the computation-per-token ratio within that constraint.
- The **`maxGasLimit` validation** (15,000,000) only prevents excessively large single requests; it does not address the truncation-induced discrepancy at small gas values.
- The **`gasLimitRefundPercent`** refund mechanism also passes through `scaleGas()`, so truncation affects refunds symmetrically and does not compensate for the over-consumption of computation.

### Impact Explanation

At 500 req/s (the rate limit), an attacker using gas=19,999 drives ~9,999,500 gas/second of EVM computation, versus ~5,000,500 gas/second at gas=10,001 — a ~2x amplification of CPU load for the same token expenditure. Since the mirror node web3 service performs full EVM simulation (including DB reads, state trie traversal, and opcode execution), this CPU amplification is real and measurable. Across multiple mirror node instances targeted simultaneously, this can degrade or saturate processing capacity on a significant fraction of nodes without the gas throttle triggering, because the token consumption rate remains within the configured limit.

### Likelihood Explanation

No privileges are required. Any user with network access to the `/api/v1/contracts/call` endpoint can submit requests. The attacker only needs to:
- Know the gas range (10,001–19,999) — trivially discoverable from public documentation or source code.
- Craft or identify a contract call that consumes close to 19,999 gas (e.g., a loop of cheap opcodes or a series of SLOAD calls on warm storage).

The attack is fully repeatable, stateless, and requires no authentication. It can be automated with a simple HTTP client sending POST requests at the rate limit.

### Recommendation

Replace the floor-division truncation with ceiling division so that the token cost always rounds up, never down:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {
        return 0L;
    }
    return Math.ceilDiv(gas, GAS_SCALE_FACTOR); // Java 18+, or (gas + GAS_SCALE_FACTOR - 1) / GAS_SCALE_FACTOR
}
```

This ensures gas=19,999 costs 2 tokens (ceiling of 1.9999), not 1, making the token cost always at least as large as the actual computation ratio. Additionally, consider whether gas ≤ 10,000 returning 0 tokens (completely free from the gas throttle) is intentional, as it allows unlimited-rate requests below that threshold.

done

# Result: 500 req/s × 19,999 gas = ~9,999,500 gas/second of EVM computation
