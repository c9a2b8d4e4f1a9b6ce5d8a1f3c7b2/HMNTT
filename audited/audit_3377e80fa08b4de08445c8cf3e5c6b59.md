### Title
Unbounded Concurrent gRPC Subscriptions Enable Resource Exhaustion DoS via `subscribeTopic()`

### Summary
The `subscribeTopic()` method in `ConsensusController` accepts any number of concurrent streaming subscriptions from any unauthenticated caller with no global concurrent-subscription cap, no per-IP limit, and no application-level rate limiting. An attacker opening many TCP connections — each carrying up to the per-connection maximum of 5 concurrent calls — can hold an unbounded number of indefinite subscriptions, exhausting the server's DB connection pool, thread pool, and memory.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) immediately creates a reactive pipeline per call with no caller identity check, no subscription count gate, and no rate limit:

```java
// ConsensusController.java:43-48
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) creates a `TopicContext` per subscription, increments a global `subscriberCount` gauge (metrics only, no enforcement), and starts both a historical retriever Flux and a live listener Flux — each consuming DB connections and memory for the lifetime of the stream.

**Root cause — failed assumptions:**

1. **No authentication/authorization**: A grep of all production `grpc/src/main/**/*.java` finds zero `ServerInterceptor` implementations. The only interceptor (`GrpcInterceptor`) is test-only and only sets an endpoint-context string for table-usage tracking. `TopicMessageFilter` carries no caller identity field.

2. **Per-connection limit is trivially bypassed**: `GrpcConfiguration.java` (line 33) sets `maxConcurrentCallsPerConnection = 5` (default from `NettyProperties`). This limits calls *per TCP connection*, not globally. An attacker opens N connections → 5N concurrent subscriptions.

3. **No global subscription cap**: `subscriberCount` (line 48 of `TopicMessageServiceImpl`) is a Micrometer gauge wired to a Prometheus alert (`GrpcNoSubscribers`) — it is purely observational. No code path checks it against a maximum before accepting a new subscription.

4. **Indefinite streams by design**: A `ConsensusTopicQuery` with `limit=0` and no `consensusEndTime` produces a stream that never terminates (`pastEndTime()` returns `Flux.never()` when `endTime == null`). Each such subscription holds a live DB listener and a safety-check scheduler task indefinitely.

5. **Infrastructure rate limit does not bound concurrent subscriptions**: The GCP backend policy `maxRatePerEndpoint: 250` (Helm `values.yaml` line 69) limits the *rate* of new requests per second, not the count of concurrently held open streams. An attacker establishing 250 new subscriptions/second for 60 seconds accumulates 15,000 concurrent open streams before the first one would naturally expire.

### Impact Explanation

Each active subscription holds:
- A `TopicContext` object with `AtomicLong`, `AtomicReference`, `Stopwatch` in heap
- An active R2DBC/Redis listener registration in `topicListener`
- Periodic DB polling via `topicMessageRetriever` (every 2 s by default)
- A scheduled safety-check task on `Schedulers.boundedElastic()`

Exhausting the bounded-elastic thread pool or the R2DBC connection pool causes all subscriptions (including legitimate ones) to stall or fail, achieving full service denial. The `subscriberCount` Prometheus alert fires only after the fact. No privileged access is required; the gRPC port (5600) is publicly exposed.

### Likelihood Explanation

Preconditions: network access to port 5600, a valid topic ID (any existing topic — topic IDs are public). A single commodity machine with a standard gRPC client library can open thousands of TCP connections. The attack is fully scriptable, repeatable, and requires no credentials or prior knowledge beyond a topic ID. The `grpcurl` invocation shown in `docs/grpc/README.md` is the exact tool needed.

### Recommendation

1. **Add a global concurrent-subscription cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`, returning `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-source-IP concurrent-subscription limit**: Implement a `ServerInterceptor` that tracks active streams per remote address and rejects new ones above a threshold.
3. **Enforce a maximum subscription duration**: Require a non-zero `consensusEndTime` or impose a server-side maximum stream lifetime (e.g., 24 h) to prevent indefinite resource holding.
4. **Add application-level rate limiting for new subscriptions**: Mirror the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern already used in the `web3` module.

### Proof of Concept

```bash
# Open 200 TCP connections, each with 5 concurrent indefinite subscriptions = 1000 active streams
# Requires: grpcurl, a known topic ID (e.g., 41110 from docs/grpc/README.md)

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <target>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Each background process holds an open stream indefinitely.
# Monitor server: hiero_mirror_grpc_subscribers gauge climbs without bound.
# DB connection pool exhaustion and bounded-elastic thread starvation follow,
# causing UNAVAILABLE errors for all subsequent legitimate subscribers.
```
