### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener` Enable DB Connection Pool Exhaustion (DoS)

### Summary
When `hiero.mirror.grpc.listener.type=POLL` is configured, `PollingTopicListener.listen()` creates a fully independent DB polling loop per subscription with no sharing or cap on total subscriptions. An unauthenticated attacker can open an unbounded number of gRPC connections (only 5 calls per connection are limited, but connections themselves are unlimited), flooding the server with subscriptions that each independently hammer the database at the configured interval (minimum 50ms), exhausting the HikariCP connection pool and starving all other DB-dependent services.

### Finding Description

**Code path:**

`CompositeTopicListener.listen()` (line 40вЂ“43) delegates to `PollingTopicListener` when `type=POLL`: [1](#0-0) 

`PollingTopicListener.listen()` creates a **per-subscription** independent `Flux` that polls the DB at `interval` forever: [2](#0-1) 

Each poll calls `topicMessageRepository.findByFilter(newFilter)`, which issues a synchronous JDBC query consuming a HikariCP connection for its duration: [3](#0-2) 

**Root cause вЂ” failed assumption:** The design assumes the number of concurrent subscriptions will be naturally bounded. There is no total-subscription cap, no per-IP limit, and no authentication requirement.

**The only per-connection limit** is `maxConcurrentCallsPerConnection=5`: [4](#0-3) [5](#0-4) 

This limits calls **per connection**, but there is no limit on the **number of connections** an attacker can open. An attacker opens K connections Г— 5 subscriptions = 5K independent polling loops.

**Minimum interval enforcement** (`@DurationMin(millis = 50)`) means an operator can configure as low as 50ms: [6](#0-5) 

At 50ms interval with N subscriptions: **N Г— 20 DB queries/second**. Even at the default 500ms, N Г— 2 queries/second scales linearly with attacker connections.

**Contrast with `SharedPollingTopicListener`** which uses a single shared `Flux` (`.share()`) regardless of subscriber count вЂ” `PollingTopicListener` has no such sharing: [7](#0-6) 

### Impact Explanation

With enough subscriptions, the HikariCP pool for the gRPC module is fully occupied by polling queries. New polls queue and time out (10s statement timeout per docs). The importer and REST API share the same PostgreSQL instance; if the gRPC DB user's pool is exhausted, the DB server itself faces connection pressure from all services. This constitutes a Denial-of-Service against the mirror node's data ingestion and query pipeline вЂ” matching the "shutdown of в‰Ґ30% of network processing nodes" severity classification in the question scope.

### Likelihood Explanation

**Precondition:** `type=POLL` must be explicitly configured (default is `REDIS`). This is a non-default but documented operational mode. Any deployment using `POLL` is vulnerable.

**Attacker capability:** Zero privileges required. The gRPC `subscribeTopic` endpoint has no authentication: [8](#0-7) 

The attacker only needs network access to port 5600 and a valid topic ID (or can enumerate). Opening hundreds of TCP connections is trivial. The attack is repeatable and persistent as long as subscriptions remain open.

### Recommendation

1. **Enforce a global concurrent-subscription cap** in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge вЂ” reject new subscriptions above a configurable threshold.
2. **Add per-source-IP connection limiting** at the Netty/proxy layer.
3. **Prefer `SHARED_POLL` over `POLL`** in all production deployments; deprecate or remove `POLL` mode, since `SHARED_POLL` is strictly superior for multi-subscriber scenarios.
4. **Enforce a minimum interval floor higher than 50ms** for `POLL` mode (e.g., 500ms minimum) to reduce per-subscription DB pressure.
5. **Isolate the gRPC DB connection pool** from the importer's pool to contain blast radius.

### Proof of Concept

```bash

# Open 200 connections Г— 5 subscriptions = 1000 independent polling loops
for i in $(seq 1 200); do
  for j in $(seq 1 5); do

      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# At 50ms interval: 1000 Г— 20 = 20,000 DB queries/second
# At 500ms interval: 1000 Г— 2 = 2,000 DB queries/second
# Monitor HikariCP pool exhaustion via:
# hikaricp_connections_active / hikaricp_connections_max в†’ approaches 1.0
# Subsequent legitimate queries time out or fail
```

```java
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
