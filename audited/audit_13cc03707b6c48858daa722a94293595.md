### Title
DELEGATECALL Chain DB Query Amplification via `findRuntimeBytecode()` Bypasses Gas-Based Throttle

### Summary
An unprivileged external user can submit a single `eth_call` request targeting a contract that loops DELEGATECALL operations across many distinct on-chain contract addresses. Each distinct address triggers an uncached `findRuntimeBytecode()` database query, because the contract bytecode cache (`maximumSize=1000`) is overwhelmed by requests touching more than 1,000 distinct contracts. The gas-based throttle measures gas limit, not DB query count, so the ratio of DB queries to gas consumed is not bounded — enabling multiplicative database load amplification well above 30%.

### Finding Description

**Exact code path:**

`ContractRepository.findRuntimeBytecode()` is annotated:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

This is called from `ContractBytecodeReadableKVState.readFromDataSource()` every time the EVM needs to load bytecode for a contract address during execution:

```java
return contractRepository
    .findRuntimeBytecode(entityId.getId())
    .map(Bytes::wrap)
    .map(Bytecode::new)
    .orElse(null);
``` [2](#0-1) 

**Cache configuration:**

The contract bytecode cache is configured with `maximumSize=1000` and `expireAfterAccess=1h`:

```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
``` [3](#0-2) 

**Root cause — gas throttle does not bound DB query count:**

The throttle consumes tokens proportional to the gas *limit* of the request:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
``` [4](#0-3) 

The `scaleGas` function divides gas by `GAS_SCALE_FACTOR = 10,000`:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [5](#0-4) 

The maximum gas per request is 15,000,000: [6](#0-5) 

**Exploit flow:**

1. Attacker deploys (one-time) or identifies an existing on-chain contract that loops DELEGATECALL across an array of distinct contract addresses (e.g., a pattern like `delegatecallExternalFunctionNTimes` but with distinct targets). [7](#0-6) 

2. With 15M gas and EIP-2929 cold-account DELEGATECALL cost of ~2,600 gas each, a single `eth_call` can trigger approximately **5,769 DELEGATECALL operations** to distinct addresses.

3. Each distinct address causes a cache miss in the 1,000-entry contract cache, issuing a `SELECT runtime_bytecode FROM contract WHERE id = ?` query to the database.

4. The gas throttle allows up to 500 requests/second (rate limit) at max gas. Each request can generate ~5,769 DB queries vs. ~1–5 for a normal request — a **~1,000× amplification factor**.

5. The `unless = "#result == null"` cache condition is effectively a no-op: since the return type is `Optional<byte[]>`, the result is never literally `null`, so all results (including `Optional.empty()`) are cached. This is correct for repeated lookups of the same address, but provides no protection when the attacker rotates through thousands of distinct contract IDs that exceed the cache's `maximumSize=1000`.

**Why existing checks fail:**

- The gas throttle (`gasPerSecond`, `requestsPerSecond`) measures computational gas, not DB I/O operations. The two are not proportional when DELEGATECALL chains span many distinct contracts.
- The cache eviction policy (LRU, size 1000) means an attacker referencing >1,000 distinct contracts per request continuously evicts warm entries, preventing cache effectiveness.
- There is no per-request limit on the number of distinct contract addresses that can be accessed.
- The `requestTimeout` of 10,000ms does not prevent the DB queries from being issued before the timeout fires. [8](#0-7) 

### Impact Explanation

The database receives a sustained flood of `SELECT runtime_bytecode FROM contract WHERE id = ?` queries far exceeding normal load. At 500 req/s × ~5,769 queries/req = ~2.88M DB queries/second, versus a normal baseline of ~500–2,500 queries/second. This causes PostgreSQL CPU, I/O, and connection pool exhaustion, degrading or denying service to all mirror node users. The impact is a >30% increase in node resource consumption (DB tier) achievable by a single attacker with no credentials.

### Likelihood Explanation

No authentication or special privileges are required — `eth_call` is a public, unauthenticated endpoint. The attacker only needs to identify or deploy one contract that loops DELEGATECALL across distinct addresses (a one-time setup). The attack is continuously repeatable and can be automated. On mainnet/testnet, thousands of deployed contracts already exist as valid DELEGATECALL targets. The pattern is well-known and the tooling (web3.js, ethers.js) makes it trivial to script.

### Recommendation

1. **Add a per-request DB query counter**: Track the number of `findRuntimeBytecode()` calls within a single `eth_call` execution context (e.g., via `ContractCallContext`) and reject or short-circuit requests that exceed a configurable threshold (e.g., 200 distinct contract lookups per call).

2. **Increase cache size**: Raise `maximumSize` from 1,000 to at least 10,000 to match the entity cache, reducing cache-miss amplification. [3](#0-2) 

3. **Fix the `unless` condition**: Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` if the intent is to not cache not-found results, or document that caching empty optionals is intentional. [9](#0-8) 

4. **Account for DB I/O in throttle**: Supplement the gas-based throttle with a DB-query-count-based throttle that rejects requests exceeding a per-second DB query budget.

### Proof of Concept

**Precondition**: Deploy (or identify) a contract `MultiDelegator` on the target network:
```solidity
contract MultiDelegator {
    function delegateToMany(address[] calldata targets, bytes calldata data) external {
        for (uint i = 0; i < targets.length; i++) {
            targets[i].delegatecall(data);
        }
    }
}
```

**Steps**:
1. Collect 2,000+ distinct deployed contract addresses from the mirror node (e.g., via `/api/v1/contracts`).
2. Submit an `eth_call` to `MultiDelegator.delegateToMany(addresses[0..1999], "0x")` with `gas: 15000000`.
3. Each iteration triggers `ContractBytecodeReadableKVState.readFromDataSource()` → `ContractRepository.findRuntimeBytecode()` → DB query (cache misses after entry 1,000).
4. Repeat at 500 req/s using a script. Observe PostgreSQL query rate and CPU spike >30% above baseline within seconds.

**Expected result**: Database query rate increases by 100–1,000× compared to baseline, causing measurable resource exhaustion on the mirror node's database tier.
