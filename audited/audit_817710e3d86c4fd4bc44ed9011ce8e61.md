### Title
Unauthenticated High-Frequency Burst to `/schedules/:scheduleId` Exhausts Database CPU via Unbounded `json_agg` Aggregation

### Summary
The `getScheduleById` handler in `rest/schedules.js` executes an expensive three-table JOIN with `json_agg` aggregation on every request with no rate limiting, no per-IP throttling, and no in-flight request cap. For an active (non-executed, non-expired) schedule, the response cache is either disabled by default or provides only a 1-second TTL, meaning every request within that window triggers a full database query. An unauthenticated attacker can flood this endpoint to exhaust database CPU and starve gossip-monitoring clients.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `getScheduleById()`, line 120:
```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
```

The query executed (lines 46–64) performs a three-table JOIN with an unbounded `json_agg` aggregation ordered by `consensus_timestamp`, grouped by `schedule_id`:
```sql
select s.*, e.*, json_agg(json_build_object(...) order by ts.consensus_timestamp) as signatures
from schedule s
left join entity e on e.id = s.schedule_id
left join transaction_signature ts on ts.entity_id = s.schedule_id
where s.schedule_id = $1
group by s.schedule_id, e.id
```
For a schedule with many signatures (a "hot" schedule accumulating signers), this aggregation is CPU-intensive.

**Root cause — no rate limiting in the REST API:**

`rest/server.js` (lines 67–98) registers only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optionally `responseCacheCheckHandler`. There is no rate-limiting or in-flight-request-limiting middleware. The throttle mechanisms (`ThrottleConfiguration`, `ThrottleManagerImpl`) exist only in the `web3` Java module and do not apply here.

**Why the response cache fails to protect active schedules:**

`getScheduleCacheControlHeader` (lines 130–146) returns `{}` (no `cache-control` header) for any schedule that has not yet executed, auto-expired, or explicitly expired. In `responseCacheHandler.js` line 24, `DEFAULT_REDIS_EXPIRY = 1` second is used when no `max-age` is present, so the cache entry expires after 1 second. Furthermore, `cache.response.enabled` defaults to `false` in the Helm chart (`charts/hedera-mirror/values.yaml` line 297) and documentation (docs line 549), meaning the cache is not active in the default deployment at all.

**Exploit flow:**
1. Attacker identifies an active schedule ID (publicly visible via `/api/v1/schedules`).
2. Attacker sends a high-frequency burst (e.g., hundreds of concurrent HTTP GET requests) to `/api/v1/schedules/<scheduleId>`.
3. Each request passes `utils.validateReq` (only validates query params, not rate), parses the schedule ID, and immediately calls `pool.queryQuietly(getScheduleByIdQuery, scheduleId)`.
4. The DB pool (`maxConnections: 10` by default, docs line 556) queues requests; all 10 connections execute the `json_agg` query concurrently, saturating DB CPU.
5. Legitimate gossip-monitoring clients calling other schedule endpoints receive timeouts or extreme latency.

### Impact Explanation

Database CPU is exhausted, causing query timeouts (default `statementTimeout: 20000ms`) across all REST API endpoints sharing the same DB pool. Gossip-monitoring clients that rely on `/schedules` to track pending multi-party scheduled transactions are denied service. In a production deployment with `maxConnections: 10`, only 10 concurrent attacker requests are needed to fully occupy the pool; additional requests queue indefinitely until the pool's `connectionTimeout` (20 seconds) is reached, amplifying the denial-of-service effect.

### Likelihood Explanation

No authentication, API key, or credential is required. The schedule ID is publicly discoverable. The attack requires only a standard HTTP client capable of sending concurrent requests (e.g., `curl`, `ab`, `wrk`). It is repeatable indefinitely and can be sustained from a single IP or distributed across multiple IPs. The absence of any rate-limiting middleware in the REST server (confirmed by reading `rest/server.js` in full) means there is no application-layer defense.

### Recommendation

1. **Add rate limiting middleware** to the Express REST server (e.g., `express-rate-limit`) applied globally or specifically to `/schedules/:scheduleId`, limiting requests per IP per second.
2. **Enable and enforce the response cache** for active schedules with a short but non-trivial TTL (e.g., 5–10 seconds) rather than the current 1-second default, so repeated requests for the same schedule ID are served from Redis.
3. **Add an in-flight request cap** per endpoint or globally (similar to the Traefik `inFlightReq` middleware used for the Rosetta API) to bound concurrent DB queries.
4. **Consider a DB-level statement timeout** tuned lower for this query, or add a query result cache at the application layer keyed on `scheduleId`.

### Proof of Concept

```bash
# Step 1: Discover an active schedule ID
SCHEDULE_ID=$(curl -s "http://<mirror-node>/api/v1/schedules?limit=1" | jq -r '.schedules[0].schedule_id')

# Step 2: Flood the endpoint with concurrent requests (no auth required)
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/schedules/${SCHEDULE_ID}" &
done
wait

# Result: DB CPU spikes to 100%; legitimate schedule queries from gossip-monitoring
# clients time out or receive 500 errors due to pool exhaustion.
```
