### Title
`BufferUnderflowException` from Short EVM Address Bypasses `CustomExceptionResolver`, Leaking Internal Error Details

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` wraps the decoded EVM address bytes in a `ByteBuffer` and calls `getInt()` and `getLong()` without first validating that the buffer contains the required 20 bytes. `decodeEvmAddress()` imposes no minimum length, so any unprivileged user can supply a hex string shorter than 4 bytes (or between 4–19 bytes) to trigger a `java.nio.BufferUnderflowException`. `CustomExceptionResolver` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`; it returns `null` for `BufferUnderflowException`, causing Spring GraphQL to emit an `INTERNAL_ERROR` response that exposes the exception class name and message.

### Finding Description

**Root cause — no length guard in `decodeEvmAddress`:**

`GraphQlUtils.decodeEvmAddress()` strips the optional `0x` prefix and hex-decodes whatever remains. It performs no minimum-length check and can return 0–N bytes for any even-length hex string. [1](#0-0) 

The project's own test suite confirms that a 5-byte decoded result is accepted without error: [2](#0-1) 

**Crash site — unchecked `ByteBuffer` reads:**

`getByEvmAddressAndType()` immediately wraps the decoded bytes and reads 4 + 8 + 8 = 20 bytes unconditionally:

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {   // needs ≥12 bytes
    return entityRepository.findById(buffer.getLong()) // needs ≥20 bytes
``` [3](#0-2) 

- Input `< 4 bytes` → `getInt()` throws `BufferUnderflowException`
- Input `4–11 bytes` → first `getLong()` throws `BufferUnderflowException`
- Input `12–19 bytes` → second `getLong()` (line 38) throws `BufferUnderflowException`

**Exception handler gap:**

`CustomExceptionResolver.resolveToSingleError()` only handles three exception types. `BufferUnderflowException` (a direct subclass of `RuntimeException`) matches none of them, so the method returns `null`: [4](#0-3) 

When every registered `DataFetcherExceptionResolver` returns `null`, Spring for GraphQL's default handler emits an `INTERNAL_ERROR` GraphQL error whose `message` field contains the exception's `toString()` (i.e., `"java.nio.BufferUnderflowException"` and, in some Spring GraphQL versions/configurations, the full stack trace in the extensions map).

### Impact Explanation
Any unauthenticated caller of the GraphQL API can:
1. Confirm the internal technology stack (Java NIO `ByteBuffer`).
2. Infer the entity-resolution code path (the exact sequence of reads reveals the 4-byte shard + 8-byte realm + 8-byte num layout).
3. Repeatedly trigger the error to probe internal behavior, aiding further targeted attacks.

In stricter Spring GraphQL configurations the message is sanitized to a generic string, but the exception type is still surfaced in the `extensions.classification` field as `"INTERNAL_ERROR"` rather than `"ValidationError"`, which is itself an information leak distinguishing handled from unhandled paths.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs to know the GraphQL endpoint and the name of any field that accepts an `evmAddress` argument (publicly documented). The payload is a single short hex string (e.g., `"0x01"`). The condition is 100% reproducible and requires no timing or race conditions.

### Recommendation
1. **Enforce length in `decodeEvmAddress`**: After hex-decoding, assert `result.length == 20` and throw `IllegalArgumentException` if not. This is caught by `CustomExceptionResolver` and returned as a clean `ValidationError`.
2. **Add a catch in `getByEvmAddressAndType`**: Wrap the `ByteBuffer` reads in a try-catch for `BufferUnderflowException` and rethrow as `IllegalArgumentException("Invalid EVM address length")`.
3. **Broaden `CustomExceptionResolver`**: Add `BufferUnderflowException` (or a broader `RuntimeException` fallback) to emit a sanitized `ValidationError` rather than leaking `INTERNAL_ERROR`.

### Proof of Concept

```graphql
# Send to the GraphQL endpoint (no authentication required)
query {
  account(input: { evmAddress: "0x0102" }) {
    id
  }
}
```

**Expected (correct) response:**
```json
{ "errors": [{ "errorType": "ValidationError", "message": "Invalid EVM address" }] }
```

**Actual response (vulnerable):**
```json
{
  "errors": [{
    "message": "java.nio.BufferUnderflowException",
    "locations": [...],
    "path": ["account"],
    "extensions": { "classification": "INTERNAL_ERROR" }
  }]
}
```

The `INTERNAL_ERROR` classification confirms the exception bypassed `CustomExceptionResolver`. Varying the input length (2, 8, 16 hex chars) triggers the exception at different `ByteBuffer` read sites, allowing an attacker to map the internal parsing logic.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

    protected GraphQLError resolveToSingleError(Throwable ex, DataFetchingEnvironment env) {
        if (ex instanceof IllegalStateException
                || ex instanceof IllegalArgumentException
                || ex instanceof MirrorNodeException) {
            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())
                    .location(env.getField().getSourceLocation())
                    .build();
        } else {
            return null;
