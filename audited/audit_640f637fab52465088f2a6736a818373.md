### Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded DB Queries via `limit = Long.MAX_VALUE`

### Summary
In `PollingTopicListener.poll()`, the remaining-limit calculation casts a `long` to `int` without overflow protection. An unprivileged user who submits a gRPC subscription with `limit = Long.MAX_VALUE` causes `(int)(Long.MAX_VALUE - 0)` to evaluate to `-1`, which propagates as the page size into every subsequent poll cycle. Because `hasLimit()` returns `false` for negative values, the repository executes fully unbounded SQL queries on every poll tick for the lifetime of the subscription, enabling a sustained resource-exhaustion DoS against the mirror node.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` copies the raw protobuf `uint64` limit directly into the filter with no upper-bound cap: [1](#0-0) 

`TopicMessageFilter` accepts any value `≥ 0` (Bean Validation `@Min(0)`), so `Long.MAX_VALUE` passes validation: [2](#0-1) 

Inside `PollingTopicListener.poll()`, the remaining-limit is computed with a narrowing cast:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())   // line 55
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());  // line 57
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build(); // line 59
``` [3](#0-2) 

Arithmetic:
- `Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF`
- `(int)(Long.MAX_VALUE)` → lower 32 bits = `0xFFFFFFFF` = **`-1`**
- `Math.min(-1, maxPageSize)` = **`-1`**
- `filter.toBuilder().limit(-1).build()` — the `toBuilder()` call is **not** a Spring-validated path; `@Validated`/`@Valid` is only enforced at the `subscribeTopic` service boundary, not inside `poll()`.

`hasLimit()` is defined as `limit > 0`, so it returns `false` for `-1`: [4](#0-3) 

In `TopicMessageRepositoryCustomImpl.findByFilter()`, when `hasLimit()` is false, `setMaxResults` is never called, so the JPA query has **no LIMIT clause**: [5](#0-4) 

The polling loop repeats indefinitely (`RepeatSpec.times(Long.MAX_VALUE)`): [6](#0-5) 

On every tick, `count` increments but `Long.MAX_VALUE - count` always produces a negative `int` (the overflow region spans `Long.MAX_VALUE - Integer.MAX_VALUE` to `Long.MAX_VALUE`, all of which cast to negative ints), so the condition never self-corrects.

The same overflow pattern exists in `PollingTopicMessageRetriever.poll()`: [7](#0-6) 

### Impact Explanation
Every poll cycle issues a full table-scan-equivalent query against `topic_message` with no row limit. On a busy topic with millions of rows, each tick loads the entire result set into the JVM heap. Sustained across the polling interval (default configurable, but continuous), this exhausts heap memory and database connection/CPU resources, degrading or crashing the mirror node for all subscribers — including those awaiting HCS topic confirmations used for fund-transfer verification flows.

### Likelihood Explanation
The exploit requires only a valid gRPC client and knowledge of any existing topic ID. No authentication, no privileged role, no special network position is needed. The protobuf `limit` field is a plain `uint64`; setting it to `Long.MAX_VALUE` (or any value in `[Long.MAX_VALUE - Integer.MAX_VALUE, Long.MAX_VALUE]`) triggers the overflow. A single malicious subscription is sufficient; multiple concurrent subscriptions amplify the impact linearly.

### Recommendation
Replace the narrowing cast with a safe clamp before converting to `int`:

```java
long remaining = filter.getLimit() - context.getCount().get();
// Guard against underflow and overflow before narrowing
int limit = (remaining <= 0)
        ? 0
        : (int) Math.min(remaining, Integer.MAX_VALUE);
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

Additionally, enforce an operator-configurable maximum on the user-supplied `limit` in `ConsensusController.toFilter()` before the filter is built, so values exceeding a safe threshold are rejected at the API boundary.

### Proof of Concept

1. Start the mirror node gRPC service with default configuration.
2. Identify any valid topic ID (e.g., `0.0.1234`).
3. Send a `subscribeTopic` gRPC call:
   ```
   ConsensusTopicQuery {
     topicID: { topicNum: 1234 }
     limit: 9223372036854775807   // Long.MAX_VALUE
   }
   ```
4. Observe in mirror node logs/metrics: every poll interval fires a `findByFilter` query with no `LIMIT` clause.
5. On a topic with significant message history, heap usage climbs on each tick; database CPU spikes continuously.
6. Other subscribers experience degraded or stalled message delivery as the DB and JVM are saturated.
