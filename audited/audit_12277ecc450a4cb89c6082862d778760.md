### Title
Unbounded `inner_transactions` Array Causes Unthrottled DB Query Amplification in `getTransactionsByIdOrHash()`

### Summary
In `rest/transactions.js`, `getTransactionsByIdOrHash()` extracts the `inner_transactions` array from DB rows and passes it without any size cap to `getTransactionsByTransactionIdsSql()`, which builds a SQL query with one OR condition per inner transaction pair. Because the `/api/v1/transactions/:id` endpoint has no rate limiting and no bound on `innerTransactions.length`, any unauthenticated user who knows the ID of a large ATOMIC_BATCH transaction can repeatedly trigger expensive, wide SQL queries against the DB, degrading mirror node availability.

### Finding Description
**Exact code location:** `rest/transactions.js`, lines 944–958, inside `getTransactionsByIdOrHash()`.

```js
// rest/transactions.js lines 944-958
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();                                          // ← no length cap

if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,   // ← passed directly, unbounded
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);
  rows.push(...innerTransactionRows);
}
```

`getTransactionsByTransactionIdsSql()` (lines 828–893) iterates over `transactionKeys` in pairs and emits one `(payer_account_id = $N and valid_start_ns = $M)` OR clause per pair, plus two bound parameters per pair. For a batch with K inner transactions the generated SQL WHERE clause contains K OR conditions and 2K+2 parameters. This is a single additional DB query per API call, but its parse/plan/execute cost scales with K.

**Root cause:** The `inner_transactions` column is a `bigint[]` stored verbatim from the Hedera network (HIP-551 ATOMIC_BATCH transactions). The REST layer reads it from the DB row and forwards it to the SQL builder with no upper-bound check. The REST transactions endpoint has no rate limiter (the only throttling in the codebase is on the `web3/contracts` path).

**Failed assumption:** The design assumes the number of inner transactions is small enough that the resulting SQL is cheap. No guard enforces this.

### Impact Explanation
Each unauthenticated HTTP GET to `/api/v1/transactions/<batchTxId>` causes:
1. One initial DB query (existing behaviour).
2. One additional DB query whose WHERE clause has K OR conditions (new behaviour, no cap).

Under sustained parallel requests targeting the same large-batch transaction ID, the DB connection pool is saturated with wide queries. Because the REST API has no per-IP or global rate limit for this endpoint, an attacker with a single HTTP client can issue hundreds of such requests per second. The DB CPU and I/O load grows proportionally, degrading query throughput for all mirror node consumers.

### Likelihood Explanation
**Preconditions:**
- Attacker submits (or observes) one ATOMIC_BATCH transaction on the Hedera network with the maximum allowed inner transactions. This is a normal, fee-paying network operation requiring no special privilege.
- The mirror node importer ingests it and stores the `inner_transactions` array.

**Trigger:** Attacker repeatedly calls `GET /api/v1/transactions/0.0.X-NNNN-NNNN` with the batch transaction ID. No authentication, no rate limit, no CAPTCHA.

**Repeatability:** Fully repeatable; the transaction ID is permanent and public. A single attacker with a script can sustain the attack indefinitely.

### Recommendation
1. **Add a size cap before the inner-transaction query:**
   ```js
   const MAX_INNER_TRANSACTIONS = 50; // tune to network limit
   if (!isTransactionHash && innerTransactions.length > 0) {
     if (innerTransactions.length > MAX_INNER_TRANSACTIONS * 2) {
       throw new InvalidArgumentError('Too many inner transactions');
     }
     // ... existing query
   }
   ```
2. **Add rate limiting to the `/api/v1/transactions/:id` route** (analogous to the existing `web3` throttle).
3. **Use an `= ANY($1)` array parameter** instead of N OR clauses to reduce SQL parse/plan overhead for large arrays.

### Proof of Concept
1. Submit an ATOMIC_BATCH transaction to the Hedera network containing the maximum number of inner transactions.
2. Wait for the mirror node importer to ingest it; note the batch transaction ID (e.g. `0.0.1234-1700000000-000000000`).
3. Run:
   ```bash
   while true; do
     curl -s "https://<mirror-node>/api/v1/transactions/0.0.1234-1700000000-000000000" &
   done
   ```
4. Observe DB CPU/query-latency metrics rising; other API endpoints begin timing out as the DB connection pool is exhausted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```javascript
const getTransactionsByTransactionIdsSql = (transactionKeys, filters, timestampField) => {
  let payerAccountParams = [];
  let params = [];

  let minTimestamp = constants.MAX_LONG;
  let maxTimestamp = -1;
  let idConditions = [];

  if (transactionKeys.length % 2 !== 0) {
    throw new InvalidArgumentError('transaction keys must be in sequenced pairs of [payer,timestamp]');
  }

  for (let index = 0; index < transactionKeys.length; index += 2) {
    let paramIndex = params.length;
    const payer = transactionKeys[index];
    const timestamp = BigInt(transactionKeys[index + 1]);

    if (payer) {
      payerAccountParams.push(`$${++paramIndex}`);
      idConditions.push(`(${Transaction.PAYER_ACCOUNT_ID} = $${paramIndex} and 
                               ${timestampField} = $${++paramIndex})`);
      params.push(payer);
    } else if (timestampField === Transaction.VALID_START_NS) {
      throw new InvalidArgumentError('payer is required when timestamp is valid_start_ns');
    } else {
      idConditions.push(`${timestampField} = $${++paramIndex}`);
    }
    params.push(timestamp);

    if (timestamp < minTimestamp) {
      minTimestamp = timestamp;
    }
    if (timestamp > maxTimestamp) {
      maxTimestamp = timestamp;
    }
  }

  const maxConsensusTimestamp =
    timestampField === Transaction.CONSENSUS_TIMESTAMP
      ? transactionKeys[transactionKeys.length - 1]
      : maxTimestamp + maxTransactionConsensusTimestampRangeNs;
  const commonConditions = [
    `${Transaction.CONSENSUS_TIMESTAMP} >= $${params.length + 1}`,
    `${Transaction.CONSENSUS_TIMESTAMP} <= $${params.length + 2}`,
  ];
  params.lowerConsensusTimestampIndex = params.length;
  params.push(
    timestampField === Transaction.VALID_START_NS
      ? minTimestamp - config.query.maxValidStartTimestampDriftNs
      : minTimestamp
  );
  params.upperConsensusTimestampIndex = params.length;
  params.push(maxConsensusTimestamp);

  const {conditions: filterConditions, scheduled} = getTransactionByIdQueryParamConditions(filters, params);
  const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
  const subqueryConditions = payerAccountParams.length
    ? [...commonConditions, `${Transaction.PAYER_ACCOUNT_ID} in (${payerAccountParams.join(',')})`].join(' and ')
    : commonConditions.join(' and ');

  return {
    query: getTransactionQuery(mainConditions, subqueryConditions),
    params,
    scheduled,
  };
};
```
