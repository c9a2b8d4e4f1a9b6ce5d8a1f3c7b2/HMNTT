### Title
Unbounded `string_agg` Bytecode Fetch in `getContractById` Enables Unauthenticated Memory-Exhaustion DoS

### Summary
Any unauthenticated caller can issue `GET /contracts/:contractId` for a contract whose `file_id` is set. The handler unconditionally calls `FileDataService.getFileData()`, which executes a `string_agg` SQL query that concatenates every bytecode chunk with no size cap and loads the entire result into Node.js heap. Flooding the endpoint with concurrent requests against a large-bytecode contract exhausts process memory and crashes the REST API.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 13 routes `GET /:contractId` to `ContractController.getContractById`. [1](#0-0) 

Inside `getContractById` (lines 707ã737), after a single DB row is fetched, the only guard before calling `FileDataService.getFileData` is `utils.isValidUserFileId(contract.file_id)` ã a format/range check that says nothing about data size:

```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
}
``` [2](#0-1) 

`FileDataService.getFileData` (lines 70ã75) runs `getFileDataQuery` with no size limit: [3](#0-2) 

`getFileDataQuery` (lines 43ã62) uses `string_agg` to concatenate **all** `file_data` rows between the latest create/update timestamp and the contract's `created_timestamp`. There is no `LIMIT`, no `max_length`, and no truncation: [4](#0-3) 

**Root cause:** The assumption that bytecode stored in `file_data` is small enough to be safely materialized in memory on every request is false. Hedera contracts can have bytecode stored across many append transactions, and `string_agg` returns the full concatenated blob to the Node.js process with no cap.

**Why existing checks fail:**
- `isValidUserFileId` validates only that the ID is a non-system entity ID ã it imposes no data-size constraint.
- No rate limiting is present in the REST layer (only a test-utility reference was found).
- No response-size middleware or streaming is applied to this path.

### Impact Explanation
Each concurrent `GET /contracts/:contractId` request for a large-bytecode contract causes the Node.js process to allocate a heap buffer proportional to the full bytecode size. With enough parallel requests the process OOMs and the entire REST API becomes unavailable. Because the mirror node REST API is the primary read interface for wallets, dApps, and exchanges on Hedera/Hiero, a sustained attack constitutes a high-severity non-network DoS against a critical piece of infrastructure.

### Likelihood Explanation
The attack requires zero privileges ã no API key, no wallet, no on-chain transaction. The attacker only needs a valid `contractId` for a contract with a large `file_id`-backed bytecode, which is trivially discoverable via `GET /contracts` (the public listing endpoint). The attack is fully repeatable and scriptable (e.g., `ab -n 10000 -c 200 https://host/api/v1/contracts/<id>`). No special network position or cryptographic material is needed.

### Recommendation
1. **Add a bytecode size cap in SQL:** Replace the unbounded `string_agg` with one that aborts or truncates above a safe threshold (e.g., PostgreSQL `string_agg` with a `CHECK` or a `CASE` that returns `NULL` when `sum(length(file_data)) > MAX_BYTES`).
2. **Stream or paginate bytecode:** Instead of materializing the full blob, return a truncated hex string with a `bytecode_truncated` flag, or expose a separate `/contracts/:id/bytecode` endpoint with explicit size limits.
3. **Add rate limiting per IP/contractId** on the `GET /contracts/:contractId` route to bound concurrent in-flight fetches.
4. **Set a Node.js response-size guard** (e.g., via a middleware that aborts serialization above a configurable limit).

### Proof of Concept
```
# 1. Discover a contract with large bytecode (file_id present)
curl https://<mirror-node>/api/v1/contracts?limit=100 \
  | jq '.contracts[] | select(.file_id != null) | .contract_id'

# 2. Confirm bytecode is large

# 3. Flood with concurrent requests to exhaust heap
ab -n 50000 -c 500 https://<mirror-node>/api/v1/contracts/<contractId>

# Expected result: Node.js process OOMs; REST API returns 502/503 or crashes.
