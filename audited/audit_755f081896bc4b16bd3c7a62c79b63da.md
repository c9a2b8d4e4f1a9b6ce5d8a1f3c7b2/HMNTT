### Title
Unauthenticated Unbounded Timestamp Range Exhausts DB Connection Pool via Full Table Scan on `/api/v1/transactions`

### Summary
The `GET /api/v1/transactions` endpoint accepts open-ended or arbitrarily large timestamp range parameters from any unauthenticated caller. Because `validateRange` is explicitly passed as `false` in the `parseTimestampFilters` call and `bindTimestampRange` defaults to `false`, no range size cap is enforced. With a default DB connection pool of only 10 connections and a 20-second statement timeout, an attacker sending 10 concurrent requests with an unbounded timestamp range can hold every pool connection for the full timeout window, making the API unavailable to all other consumers.

### Finding Description

**Exact code path:**

`rest/server.js` line 132 registers the route with no authentication or rate-limiting guard:
```
app.getExt(`${apiPrefix}/transactions`, transactions.getTransactions);
```

`rest/transactions.js` line 674 calls `parseTimestampFilters` with `validateRange=false` and `allowOpenRange=true`:
```javascript
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
```

`rest/utils.js` lines 1657–1666 show the range-size guard is gated on `validateRange`:
```javascript
if (validateRange) {
  const {maxTimestampRange, maxTimestampRangeNs} = config.query;
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(...);
  }
}
```
Because `validateRange=false`, the 7-day `maxTimestampRange` check is **never reached** for this endpoint.

`rest/timestampRange.js` lines 19–22 show the only other cap (`bindTimestampRange`) is opt-in and defaults to `false`:
```javascript
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};   // returns the range unchanged, no capping
  }
```

`rest/dbpool.js` lines 13–14 show the pool is capped at `maxConnections` (default: **10**) with a `statementTimeout` of **20 seconds**:
```javascript
max: config.db.pool.maxConnections,
statement_timeout: config.db.pool.statementTimeout,
```

`rest/middleware/authHandler.js` lines 15–19 confirm unauthenticated requests are passed through without any rejection or rate-limiting:
```javascript
if (!credentials) {
  return;   // no-op; request proceeds normally
}
```

**Root cause:** The failed assumption is that the `maxTimestampRange` (7d) or `maxTransactionsTimestampRange` (60d) guards would protect the transactions endpoint. They do not: `validateRange=false` disables the former, and `bindTimestampRange=false` (default) disables the latter. The endpoint therefore accepts queries like `timestamp=gte:0` with no upper bound, generating SQL of the form:
```sql
select ... from transaction as t where t.consensus_timestamp >= 0
order by t.consensus_timestamp desc limit 25
```
This is a full sequential scan of the entire `transaction` table (potentially billions of rows on mainnet), holding a DB connection for up to the 20-second statement timeout.

### Impact Explanation
An attacker sending 10 concurrent requests with `timestamp=gte:0` saturates the entire default connection pool for up to 20 seconds per wave. During that window, every other API call that requires a DB connection (accounts, balances, blocks, tokens, etc.) queues or times out. Repeating the attack in a loop creates a sustained denial-of-service against the entire mirror node REST API. Because the `transaction` table is the largest and most-queried table in the mirror node database, the query also generates maximum I/O pressure on the database host, potentially degrading the importer's write path and delaying block data ingestion — directly matching the described "freeze block data availability" impact.

### Likelihood Explanation
No privileges, API keys, or accounts are required. The endpoint is publicly documented and reachable from any internet host. The attack requires only 10 concurrent HTTP GET requests, achievable with a single `curl` loop or any HTTP benchmarking tool. The attack is trivially repeatable and automatable. The only partial mitigations are the 20-second statement timeout (which limits per-query damage but does not prevent pool exhaustion) and an optional Traefik `inFlightReq` middleware in the Kubernetes Helm chart — but that middleware is not part of the application itself and is absent in non-Kubernetes deployments.

### Recommendation
1. **Immediate:** Change the `parseTimestampFilters` call in `getTransactions` (`rest/transactions.js` line 674) to pass `validateRange=true`, enforcing the existing `maxTimestampRange` (7d) cap on all timestamp range queries to this endpoint.
2. **Short-term:** Enable `bindTimestampRange=true` by default, or enforce `maxTransactionsTimestampRange` (60d) unconditionally in `getTransactionTimestamps` regardless of the `bindTimestampRange` flag.
3. **Defense-in-depth:** Add a per-IP rate-limiting middleware (e.g., `express-rate-limit`) to the Node.js REST API, mirroring the throttle mechanism already present in the web3 Java module.
4. **Increase pool size** or add a connection-wait timeout that returns HTTP 503 instead of blocking indefinitely when the pool is exhausted.

### Proof of Concept
```bash
# Saturate the 10-connection pool with unbounded full-table-scan queries
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=gte:0" &
done
wait

# All subsequent requests will queue or fail with connection timeout errors
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: delayed response or connection timeout while pool is exhausted
```

The `timestamp=gte:0` parameter passes all validation (open range is allowed, range-size check is skipped), reaches `pool.queryQuietly`, and holds a DB connection for up to 20 seconds executing a full scan of the `transaction` table.
