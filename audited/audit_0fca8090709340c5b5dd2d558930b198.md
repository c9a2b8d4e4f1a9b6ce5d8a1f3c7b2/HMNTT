### Title
DoS-Induced Health Gate Suppression via Unauthenticated `/network/stake` Flood

### Summary
An unprivileged external attacker can flood the unauthenticated `/api/v1/network/stake` REST endpoint to exhaust the REST Java service's thread/connection pool, causing the monitor's `restNetworkStakeHealth()` call to exceed its hardcoded 5-second timeout. The resulting `TimeoutException` is mapped to `Status.DOWN`, which short-circuits the entire `health()` evaluation, permanently suppressing the real publish/subscribe health assessment and setting the `CLUSTER_UP` metric to 0 for the duration of the attack.

### Finding Description
**Code path:**

`SubscriberHealthIndicator.java`, `restNetworkStakeHealth()`, lines 93–122:
```java
return restApiClient
    .getNetworkStakeStatusCode()          // public HTTP GET /network/stake, no auth
    .flatMap(...)
    .timeout(Duration.ofSeconds(5))       // line 107 — hardcoded, not configurable
    .onErrorResume(e -> {
        var rootCause = ExceptionUtils.getRootCause(e);
        if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
            status = Status.DOWN;          // TimeoutException → DOWN
        }
        return health(status, statusMessage);
    });
```

`health()`, lines 59–63:
```java
return restNetworkStakeHealth()
    .flatMap(health ->
        health.getStatus() == Status.UP
            ? publishing().switchIfEmpty(subscribing())
            : Mono.just(health));          // non-UP → skip all pub/sub evaluation
```

**Root cause:** The `/network/stake` endpoint in `NetworkController` (lines 126–130) is a public, unauthenticated `@GetMapping` with no rate limiting (the `ThrottleConfiguration`/`ThrottleManagerImpl` throttling exists only in the `web3` module). The monitor's `WebClient` has no independent connection-level timeout configured beyond the reactive `.timeout()`. A flood of concurrent requests saturates the REST Java service's thread pool or database connection pool, causing the monitor's own request to queue past 5 seconds. The `TimeoutException` is treated identically to a genuine infrastructure failure and returns `Status.DOWN`. Because `health()` uses `restNetworkStakeHealth()` as a hard gate, the actual publish/subscribe health is never evaluated.

### Impact Explanation
- The `CLUSTER_UP` Micrometer gauge is set to `0` for the entire attack duration, triggering false infrastructure alerts and potentially causing automated remediation actions (pod restarts, failovers).
- The real publish/subscribe health — the actual signal for transaction pipeline health — is completely suppressed. A genuine transaction processing failure occurring simultaneously would be invisible.
- The attack is self-sustaining: as long as the flood continues, every health check poll returns DOWN, regardless of the actual state of the Hedera transaction pipeline.

### Likelihood Explanation
- No authentication, API key, or session is required to call `/api/v1/network/stake`.
- The endpoint performs a synchronous DB query (`SELECT * FROM network_stake WHERE consensus_timestamp = (SELECT MAX(...))`) — cheap per request but easily saturated under concurrent load.
- A single attacker with modest bandwidth (e.g., 1,000 concurrent HTTP/1.1 connections) can exhaust a default Spring Boot thread pool (200 threads) or the REST Java DB connection pool.
- The attack is repeatable, low-cost, and requires no knowledge of internal infrastructure.

### Recommendation
1. **Decouple the timeout classification**: Map `TimeoutException` to `Status.UNKNOWN` (not `Status.DOWN`) in `onErrorResume`, consistent with the treatment of other transient errors. A slow health-probe endpoint should not be treated the same as a confirmed infrastructure failure.
2. **Remove the hard gate**: Evaluate `publishing()` and `subscribing()` independently of `restNetworkStakeHealth()`, combining results rather than short-circuiting.
3. **Add rate limiting to `/network/stake`**: Apply the same bucket4j throttle used in the `web3` module to the `rest-java` `NetworkController`.
4. **Make the timeout configurable**: Expose the 5-second value as a configuration property so operators can tune it without code changes.
