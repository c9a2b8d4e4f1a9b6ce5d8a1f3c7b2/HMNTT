### Title
Unvalidated `file.id` Parameter Allows Arbitrary fileId to Return Empty Node List via INNER JOIN on Empty CTE

### Summary
The `GET /api/v1/network/nodes` endpoint accepts a user-supplied `file.id` query parameter that is passed without whitelist validation into a SQL CTE filtering `address_book` by `file_id`. When an arbitrary (non-existent) file ID is supplied, the `latest_address_book` CTE returns zero rows, and the subsequent INNER JOIN against `address_book_entry` produces an empty result set. The API then returns an empty `nodes` list to the caller, falsely indicating no network nodes exist.

### Finding Description

**Code path:**

`NetworkController.getNodes()` (line 153–171) accepts `NetworkNodeRequest` via `@RequestParameter`. The only validation on `fileId` is an operator check: [1](#0-0) 

This check only rejects non-EQ operators (e.g., `gt:`, `lt:`). It does **not** validate that the value is one of the two valid address book file IDs (101 or 102).

`NetworkServiceImpl.getAddressBookFileId()` (lines 178–182) blindly returns the user-supplied value: [2](#0-1) 

This arbitrary `fileId` is forwarded directly to `networkNodeRepository.findNetworkNodes()`: [3](#0-2) 

Inside the repository query, the `latest_address_book` CTE filters by the attacker-controlled value: [4](#0-3) 

If no `address_book` row exists with that `file_id`, the CTE returns zero rows. The main query then uses an **INNER JOIN** (`join`, not `left join`) against this empty CTE: [5](#0-4) 

Because the CTE is empty, the INNER JOIN eliminates all `address_book_entry` rows, and the query returns an empty list regardless of how many real nodes exist in the database.

**Root cause:** `getAddressBookFileId()` lacks a whitelist check restricting `fileId` to the two valid values (encoded IDs for file 0.0.101 and 0.0.102). The failed assumption is that callers will only supply valid address book file IDs.

### Impact Explanation

Any unauthenticated caller can issue `GET /api/v1/network/nodes?file.id=<arbitrary_long>` and receive a well-formed HTTP 200 response with an empty `nodes` array. Clients that rely on this endpoint to discover network nodes (e.g., SDKs, wallets, monitoring tools, load balancers) would interpret the empty list as "no nodes available," potentially halting transaction submission or triggering failover logic. While the actual Hedera network is unaffected, the mirror node API becomes an unreliable source of node topology for any client that passes a manipulated `file.id`.

### Likelihood Explanation

Exploitation requires zero privileges, zero authentication, and a single HTTP GET request. The parameter name (`file.id`) is documented in the public API. An attacker can trivially and repeatedly trigger this condition. Any client or SDK that forwards user-controlled input to the `file.id` parameter is directly exploitable.

### Recommendation

In `getAddressBookFileId()`, validate that the resolved `fileId` is one of the two permitted values before using it in the query:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    long fileId = request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();

    long allowed101 = systemEntity.addressBookFile101().getId();
    long allowed102 = systemEntity.addressBookFile102().getId();
    if (fileId != allowed101 && fileId != allowed102) {
        throw new IllegalArgumentException(
            "file.id must be one of the valid address book file IDs");
    }
    return fileId;
}
```

Alternatively, reject any `file.id` value at the controller layer before it reaches the service.

### Proof of Concept

**Precondition:** Mirror node is running and `/api/v1/network/nodes` (no `file.id`) returns a non-empty node list.

**Trigger:**
```
GET /api/v1/network/nodes?file.id=9999999999
```

**Expected (legitimate) response:**
```json
{ "nodes": [ { "node_id": 0, ... }, ... ], "links": { ... } }
```

**Actual response with exploit:**
```json
{ "nodes": [], "links": { "next": null } }
```

**Reproducible steps:**
1. Confirm baseline: `curl https://<mirror-node>/api/v1/network/nodes` → non-empty `nodes` array.
2. Exploit: `curl "https://<mirror-node>/api/v1/network/nodes?file.id=9999999999"` → HTTP 200 with `"nodes": []`.
3. The empty response is returned for any `file.id` value that does not correspond to an existing `address_book.file_id` row, with no error or warning to the caller.
