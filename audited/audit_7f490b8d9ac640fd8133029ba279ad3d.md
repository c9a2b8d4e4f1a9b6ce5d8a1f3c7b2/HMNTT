### Title
Unauthenticated eth_estimateGas Requests Exhaust Thread Pool via Unthrottled Binary Search Iterations

### Summary
The `eth_estimateGas` path in `ContractExecutionService` runs up to 21 synchronous EVM executions per request (1 initial call + up to 20 binary search iterations), but the gas and rate throttles are checked only once per request at the HTTP controller layer before the binary search begins. An unprivileged attacker sending requests at the permitted rate (500/s) can hold each of Tomcat's default 200 threads for up to the full `requestTimeout` (10 s), exhausting the thread pool in under half a second and rendering the service unavailable.

### Finding Description

**Code path:**

`ContractController.call()` calls `throttleManager.throttle(request)` exactly once, consuming 1 rate-limit token and `scaleGas(request.getGas())` gas tokens, then immediately delegates to `contractExecutionService.processCall()`. [1](#0-0) 

Inside `processCall`, when `params.isEstimate()` is true, `estimateGas()` is called. This first executes a full EVM call (`callContract`), then feeds the result into `binaryGasEstimator.search()`. [2](#0-1) 

`BinaryGasEstimator.search()` loops synchronously on the same thread for up to `maxGasEstimateRetriesCount = 20` iterations, calling `doProcessCall` (a full EVM execution) on every iteration. No throttle check is performed inside this loop. [3](#0-2) 

The `requestTimeout` (10 s) is enforced only via a Hibernate `StatementInspector` that fires at DB query boundaries. It does **not** interrupt the EVM execution loop between DB calls; the current binary search iteration runs to completion before the timeout can be observed. [4](#0-3) 

**Root cause — failed assumption:** The throttle design assumes one gas deduction per request equals one unit of EVM work. For `eth_estimateGas` this assumption is wrong by a factor of up to 21×.

**Gas throttle arithmetic:**
- `scaleGas(15_000_000) = 1_500` tokens consumed per request
- `gasPerSecond` bucket: `scaleGas(7_500_000_000) = 750_000` tokens/s → allows 500 estimate-gas requests/s
- Actual EVM executions: 500 req/s × 21 = **10,500 full EVM calls/s**
- The gas throttle accounts for only 500 of those. [5](#0-4) [6](#0-5) 

**Thread pool exhaustion arithmetic:**
- Rate limit: 500 req/s (default)
- Max thread hold time per request: 10 s (`requestTimeout`)
- Concurrent threads needed at steady state: 500 × 10 = **5,000**
- Tomcat default thread pool: **200** (no override found in configuration)
- Time to exhaust pool: 200 / 500 = **0.4 s** of sustained attack [7](#0-6) [8](#0-7) 

### Impact Explanation
Once all 200 Tomcat threads are occupied, new HTTP requests queue in the accept backlog and then time out from the client's perspective. The mirror node becomes completely unresponsive to all traffic — not just `eth_estimateGas` — including health checks, `eth_call`, and any downstream consumers relying on the web3 API. Because the binary search amplifies CPU and DB load 21×, the attack also saturates the database connection pool and CPU well before the thread pool is fully exhausted, compounding the outage. The service remains degraded for the full duration of the attack with no self-recovery mechanism.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs network access to the public `/api/v1/contracts/call` endpoint with `"estimate": true`. The rate limit of 500 req/s is generous enough that a single machine with a modest HTTP client can sustain the attack. The exploit is deterministic and repeatable: the attacker simply sends a tight loop of estimate-gas requests targeting a contract that exercises the full binary search (e.g., a contract whose gas usage is far below the 15 M gas ceiling, maximising the search range and iteration count). No special contract deployment is needed — any existing contract address works.

### Recommendation

1. **Count throttle tokens per binary search iteration, not per request.** Inside `BinaryGasEstimator.search()`, call `throttleManager.throttle` (or a lightweight variant) before each `safeCall` invocation, deducting gas tokens proportional to the `mid` gas value used in that iteration.

2. **Cap concurrent estimate-gas requests independently.** Add a dedicated semaphore or a separate, lower rate-limit bucket for `ETH_ESTIMATE_GAS` calls (e.g., 10–20 concurrent requests) to bound thread consumption regardless of the per-second rate.

3. **Reduce `requestsPerSecond` for estimate-gas or lower `maxGasEstimateRetriesCount`.** The current default of 500 req/s combined with a 10 s timeout and 20 iterations is the direct cause of the thread exhaustion arithmetic. Reducing any one of these three parameters proportionally reduces the attack surface.

4. **Set an explicit Tomcat `max-threads` and `accept-count`** in `application.yml` to match the intended concurrency budget, and add a circuit-breaker that rejects new estimate-gas requests when the active thread count exceeds a threshold.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
