### Title
Unauthenticated Connection Pool Exhaustion via Concurrent GET /blocks Requests

### Summary
The `GET /blocks` endpoint in `rest/routes/blockRoute.js` has no rate limiting, no per-IP concurrency control, and no route-level connection timeout. The default database connection pool contains only 10 connections with a 20-second statement timeout. An unauthenticated attacker sending as few as 10 concurrent requests can saturate the entire pool, causing all REST API endpoints to return connection timeout errors for up to 20 seconds per wave, repeatable indefinitely.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:12` registers `BlockController.getBlocks` with no middleware guard: [1](#0-0) 

`rest/controllers/blockController.js:101-112` — `getBlocks` validates query params and immediately issues a DB query with no concurrency check: [2](#0-1) 

`rest/service/recordFileService.js:149-162` — `getBlocks` builds and executes a SQL query, holding a pool connection until the DB responds: [3](#0-2) 

`rest/dbpool.js:7-16` — the pool is configured with a default maximum of **10 connections** and a **20-second statement timeout**: [4](#0-3) 

`rest/server.js:67-112` — the middleware stack contains logging, optional auth (which only sets a custom response limit, not a rate limit), optional metrics, and optional cache — **no rate limiting or concurrency limiter is registered for any route**: [5](#0-4) 

The throttling infrastructure that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives entirely in the `web3` Java module and has no effect on the Node.js REST API.

**Root cause:** The REST API assumes an upstream reverse proxy or infrastructure layer enforces per-client request rate limits. No such enforcement is present in the application code itself. The pool size of 10 means only 10 concurrent DB-bound requests are needed to block all other callers.

**Failed assumption:** The design assumes that slow queries are bounded by `statement_timeout` (20 s) and therefore pool exhaustion is transient. In practice, an attacker can continuously re-issue requests faster than the timeout expires, maintaining permanent saturation.

### Impact Explanation

All REST API endpoints share the same `global.pool` (initialized in `rest/dbpool.js:36`). Once all 10 connections are held by attacker-controlled `/blocks` requests, every other endpoint (`/transactions`, `/accounts`, `/tokens`, etc.) that needs a DB connection will block for up to `connectionTimeoutMillis` (20 s default) before returning an error. This is a full REST API denial-of-service, not limited to the blocks endpoint. The impact is equivalent to a network partition from the perspective of all API consumers. [6](#0-5) 

### Likelihood Explanation

Preconditions: none. No authentication, no API key, no special knowledge required. The attacker needs only a tool capable of sending concurrent HTTP GET requests (curl, ab, wrk, Python asyncio). With 10 concurrent requests sustained in a loop, the pool remains saturated indefinitely. The default `maxConnections: 10` is documented publicly: [7](#0-6) 

The attack is trivially scriptable, repeatable, and requires no privileged access.

### Recommendation

1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) to the REST API, applied globally or per-route, keyed on client IP.
2. **Increase the default pool size** (`maxConnections`) to reduce the blast radius of any single burst.
3. **Reduce `statementTimeout`** for the `/blocks` endpoint specifically, or add a per-request `SET LOCAL statement_timeout` to a lower value before executing the blocks query.
4. **Add a request concurrency limiter** (e.g., `bottleneck` or a semaphore) so that at most N in-flight DB queries are allowed per endpoint regardless of pool size.
5. Deploy an upstream rate-limiting layer (e.g., nginx `limit_req`, API gateway) as a defense-in-depth measure.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent unauthenticated requests
# Run from any machine with network access to the REST API

for i in $(seq 1 20); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks" &
done
wait

# While the above is running, legitimate requests to any endpoint will time out:
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: connection acquisition timeout error after ~20 seconds
```

Sustained version (keeps pool saturated indefinitely):

```bash
while true; do
  for i in $(seq 1 15); do
    curl -s "http://<mirror-node-host>:5551/api/v1/blocks?limit=25" &
  done
  sleep 18  # re-issue before statement_timeout expires
done
```
