### Title
Integer Overflow in `poll()` Bypasses SQL LIMIT, Enabling Unbounded Database Query (DoS)

### Summary
In `PollingTopicListener.poll()` (and identically in `PollingTopicMessageRetriever.poll()`), a `long`-to-`int` narrowing cast on the user-supplied `limit` field overflows to a negative integer when the value exceeds `Integer.MAX_VALUE`. The resulting negative value causes `hasLimit()` to return `false` in `TopicMessageRepositoryCustomImpl.findByFilter()`, suppressing the SQL `LIMIT` clause entirely and allowing the database to return all rows. Any unauthenticated gRPC caller can trigger this.

### Finding Description

**Entry point — no upper-bound validation on `limit`:**

`ConsensusTopicQuery.limit` is declared as `uint64` in the protobuf schema: [1](#0-0) 

It is passed directly to the filter builder with no cap: [2](#0-1) 

`TopicMessageFilter.limit` is a `long` with only `@Min(0)`, which accepts any value ≥ 0 including values far above `Integer.MAX_VALUE`: [3](#0-2) 

**Overflow site — `PollingTopicListener.poll()`:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [4](#0-3) 

With `filter.getLimit() = 2147483648L` (= `Integer.MAX_VALUE + 1`) and `context.getCount().get() = 0`:
- `(int)(2147483648L - 0L)` → `(int)2147483648L` → **`-2147483648`** (Integer.MIN_VALUE, wraps around)
- `Math.min(-2147483648, maxPageSize)` → **`-2147483648`**
- `newFilter` is built with `limit = -2147483648`

**Guard failure — `TopicMessageRepositoryCustomImpl.findByFilter()`:**

`hasLimit()` is defined as `return limit > 0`: [5](#0-4) 

A negative limit returns `false`, so `setMaxResults()` is never called: [6](#0-5) 

The JPA query executes with **no SQL `LIMIT` clause**, returning every row in the `topic_message` table for the given topic.

The same overflow exists identically in `PollingTopicMessageRetriever.poll()` (the historical retrieval path): [7](#0-6) 

**Outer `flux.take()` does not save you:**

`TopicMessageServiceImpl` applies `flux.take(filter.getLimit())` using the *original* filter's limit of `2147483648` — effectively 2 billion items — which provides no practical protection: [8](#0-7) 

### Impact Explanation
An attacker causes the database to execute unbounded `SELECT` queries against the `topic_message` table with no `LIMIT`. On a busy network with millions of messages per topic, this results in full table scans, exhausting database I/O, connection pool slots, and JVM heap (the result list is materialized via `getResultList()`). This is a reliable, repeatable Denial-of-Service against the mirror node's database. The `getResultList()` call materializes the entire result set in memory: [9](#0-8) 

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The protobuf `uint64` field accepts the exact value needed (`2147483648`) from any standard gRPC client. No special privileges, tokens, or prior knowledge beyond a valid topic ID are required. The attack is trivially repeatable with concurrent connections.

### Recommendation
1. **Cap the limit at the entry point** in `ConsensusController.toFilter()`: reject or clamp any `limit > Integer.MAX_VALUE` (or a configured maximum) before building the filter.
2. **Fix the cast** in both `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()`: use `Math.toIntExact()` (throws on overflow) or clamp with `(int) Math.min(filter.getLimit() - count, Integer.MAX_VALUE)`.
3. **Add an `@Max` constraint** on `TopicMessageFilter.limit` to enforce an upper bound at the validation layer.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Set limit = Integer.MAX_VALUE + 1 = 2147483648
# This overflows to -2147483648 in the (int) cast, bypassing hasLimit()
request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=2147483648  # uint64: valid, but overflows int cast
)

# This triggers an unbounded SELECT with no LIMIT clause
for msg in stub.subscribeTopic(request):
    print(msg)  # DB returns all rows; OOM/DoS follows
```

Reproducing steps:
1. Connect to the mirror node gRPC port (default 5600) — no credentials needed.
2. Send `subscribeTopic` with any valid `topicID` and `limit = 2147483648`.
3. Observe: the mirror node issues `SELECT ... FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with **no LIMIT**, scanning the full table.
