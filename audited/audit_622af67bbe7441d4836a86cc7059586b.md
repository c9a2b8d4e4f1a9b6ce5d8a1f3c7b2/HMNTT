### Title
GraphQL Batch Request Amplification Bypasses Per-Request In-Flight Throttle

### Summary
The `graphql` module exposes `POST /graphql/alpha` without any application-level rate limiting and without disabling GraphQL batching (JSON array of operations). The Traefik `inFlightReq: 5` middleware counts HTTP connections, not GraphQL operations within a single request. An unprivileged attacker can send a single HTTP POST containing a JSON array of ~19 `account` operations, multiplying database load ~19× per connection slot while consuming only one in-flight slot.

### Finding Description

**Exact code locations:**

`GraphQlConfiguration.java` configures the only per-operation guards — complexity and depth instrumentation — which apply per GraphQL operation, not per batch: [1](#0-0) 

The Jackson `maxTokenCount(100)` constraint is the only structural limit on batch size: [2](#0-1) 

The Traefik middleware limits in-flight HTTP requests to 5 per source IP — it counts HTTP connections, not GraphQL operations inside a batched body: [3](#0-2) 

There is **no rate limiting, throttle bean, or batch-disable configuration** anywhere in the `graphql` module Java code. The `ThrottleConfiguration` with `Bucket4j` exists only in the `web3` module. 

**Root cause:** Spring for GraphQL's `GraphQlHttpHandler` accepts a JSON array body and executes each element as a separate GraphQL operation by default. Nothing in the codebase disables this. The `inFlightReq: 5` guard is blind to intra-request operation count.

**Batch size math:** Each batch item costs 5 JSON tokens (`START_OBJECT`, `FIELD_NAME`, `VALUE_STRING`, `END_OBJECT`, plus array delimiters). With `maxTokenCount(100)`: `(100 − 2) / 5 ≈ 19` operations per request. Each operation triggers a separate DB query via `EntityService`.

**Exploit flow:**
1. Attacker sends 5 concurrent HTTP POSTs (filling the `inFlightReq: 5` budget) to `POST /graphql/alpha`.
2. Each POST body is a JSON array of 19 `account` queries.
3. Spring for GraphQL executes all 19 operations per request → 5 × 19 = **95 simultaneous DB queries** from one IP.
4. With multiple IPs (or rotating IPs), the multiplier scales linearly.
5. `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` apply per operation and do not reduce the batch count.

### Impact Explanation

The Prometheus alert for DB connection exhaustion fires at 75% of the HikariCP pool: [4](#0-3) 

Saturating the DB connection pool degrades or halts the importer's ability to write incoming gossip/transaction data, since the importer and graphql module share the same `mirror_node` database. The `GraphQLNoPodsReady` alert confirms a complete outage scenario: [5](#0-4) 

Severity: **High** — unauthenticated, no special knowledge required, repeatable, bounded amplification (~19×) is still sufficient to exhaust a small connection pool.

### Likelihood Explanation

Any external user can POST to `/graphql/alpha` with no authentication. The endpoint is publicly documented: [6](#0-5) 

The attack requires only `curl` or any HTTP client capable of sending a JSON array body. The `inFlightReq: 5` limit is per-IP and is trivially bypassed with multiple source IPs (e.g., a small botnet or cloud VMs). The amplification factor is deterministic and reproducible.

### Recommendation

1. **Disable GraphQL batching** explicitly via Spring for GraphQL configuration:
   ```yaml
   spring.graphql.http.batch-mapping-enabled: false
   ```
   or by overriding `GraphQlHttpHandler` to reject array-body requests with HTTP 400.

2. **Add application-level rate limiting** in the `graphql` module (mirror the `Bucket4j` pattern from `web3/ThrottleConfiguration`) applied at the HTTP handler level, not just at the Traefik layer.

3. **Count operations per batch** against the in-flight limit, or set `maxTokenCount` low enough that a batch of even 2 items is rejected (currently 100 tokens allows ~19 items).

### Proof of Concept

```bash
# Build a batch of 19 account queries
python3 -c "
import json, sys
ops = [{'query': '{account(input:{entityId:{shard:0,realm:0,num:%d}}){balance}}' % i} for i in range(1,20)]
print(json.dumps(ops))
" > batch.json

# Fire 5 concurrent requests (fills inFlightReq budget for one IP)
for i in $(seq 1 5); do
  curl -s -X POST https://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d @batch.json &
done
wait
```

Each of the 5 concurrent HTTP requests triggers 19 independent DB lookups. Repeat from multiple IPs to scale. Monitor `hikaricp_connections_active` — it will spike toward pool maximum, triggering the `GraphQLHighDBConnections` alert and degrading importer throughput.
