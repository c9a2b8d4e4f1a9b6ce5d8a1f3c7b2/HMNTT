### Title
Unauthenticated Repeated `/block` Requests Trigger Unbounded Batched DB Queries via `FindBetween()`, Enabling DB Resource Exhaustion

### Summary
The Rosetta `/block` endpoint in `rosetta/main.go` registers `blockAPIController` with no application-level authentication or rate limiting. Each valid POST request causes `blockAPIService.Block()` to invoke `FindBetween()`, which executes a loop of expensive correlated-subquery DB reads (batched at 2000 rows per iteration) against the `transaction`, `crypto_transfer`, and `staking_reward_transfer` tables. An unprivileged external attacker can flood this endpoint with concurrent requests targeting any valid block, driving DB CPU and I/O well above 30% of baseline.

### Finding Description

**Code path:**

`rosetta/main.go` `newBlockchainOnlineRouter()` (lines 61–120) registers `blockAPIController` into the router with no authentication or rate-limiting middleware. The only middleware applied is `MetricsMiddleware → TracingMiddleware → CorsMiddleware` (lines 217–219). [1](#0-0) 

`blockAPIService.Block()` (`rosetta/app/services/block_service.go`, lines 47–74) first calls `RetrieveBlock()` to resolve the block from DB, then unconditionally calls `FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` with no caching. [2](#0-1) 

`FindBetween()` in `rosetta/app/persistence/transaction.go` (lines 112–171) runs a `for start <= end` loop. Each iteration fires a raw SQL query (`selectTransactionsInTimestampRangeOrdered`) with two correlated subqueries (one against `crypto_transfer`, one against `staking_reward_transfer`) and a `LIMIT 2000` (`batchSize`). If the batch is full, `start` advances and the loop repeats — meaning a block with N transactions causes `ceil(N/2000)` sequential DB round-trips per HTTP request. [3](#0-2) [4](#0-3) 

The SQL query itself is expensive: for every row in `transaction`, it executes a correlated `json_agg` subquery against `crypto_transfer` and another against `staking_reward_transfer`, both filtered by `consensus_timestamp`. [5](#0-4) 

**Root cause / failed assumption:** The design assumes that an external rate-limiting layer (Traefik) will always be present. However, `global.middleware` defaults to `false` in the Helm chart, meaning the `inFlightReq` (5 concurrent) and `rateLimit` (10 req/s) Traefik middlewares are **not deployed by default**. [6](#0-5) [7](#0-6) 

There is no in-process rate limiting, concurrency cap, or response cache anywhere in the Go application.

### Impact Explanation
Every POST to `/block` with a valid block identifier unconditionally issues at least one (and potentially many) expensive DB queries with correlated subqueries. With no application-level throttle, an attacker sending tens of concurrent requests per second can saturate DB connection pool threads, drive CPU utilization on the DB host, and increase I/O substantially above the 30% threshold — degrading or denying service to legitimate users of the mirror node.

### Likelihood Explanation
The attack requires zero privileges: the Rosetta API is publicly accessible (ingress paths include `/rosetta/block`), no authentication is needed, and a valid block identifier (index or hash) is trivially obtained from any prior `/network/status` or `/block` response. The attack is fully repeatable and scriptable with standard HTTP tooling. The default deployment (`global.middleware: false`) leaves no application-side defense.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP or global concurrency/rate limiter directly in the Go HTTP server (e.g., `golang.org/x/time/rate` or a middleware like `go-chi/httprate`) so protection is not dependent on optional infrastructure.
2. **Response caching**: Cache `FindBetween()` results keyed on `(consensusStart, consensusEnd)` with a short TTL; block data is immutable once finalized.
3. **Enable Traefik middleware by default**: Change `global.middleware` default to `true` and document it as a required security control.
4. **DB query timeout**: Enforce a strict per-query timeout via the DB context so a flood of requests cannot hold DB connections indefinitely.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, deployed without `global.middleware: true` (the default). Attacker has obtained one valid block hash from a prior unauthenticated call to `/network/status`.

```bash
# Step 1: obtain a valid block hash (unauthenticated)
BLOCK_HASH=$(curl -s -X POST http://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq -r '.current_block_identifier.hash')

BLOCK_INDEX=$(curl -s -X POST http://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq -r '.current_block_identifier.index')

# Step 2: flood /block with concurrent requests (no credentials required)
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$BLOCK_INDEX,\"hash\":\"$BLOCK_HASH\"}}" &
done
wait
```

**Result:** Each concurrent request triggers `RetrieveBlock()` (one DB query) followed by `FindBetween()` (one or more batched DB queries with correlated subqueries). With 200 concurrent requests and no throttle, DB CPU and I/O spike well above baseline. For blocks with >2000 transactions, each request issues multiple sequential DB round-trips, amplifying the effect.
