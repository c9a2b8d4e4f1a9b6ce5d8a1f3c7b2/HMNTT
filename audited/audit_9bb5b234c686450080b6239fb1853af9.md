### Title
`isValidAccount()` Accepts Any Entity Type, Allowing Non-Account Entities to Pass Account Validation in `getTokenRelationships()`

### Summary
In `rest/controllers/tokenController.js`, `getTokenRelationships()` uses `EntityService.isValidAccount()` as its sole guard to verify the path parameter resolves to a valid account. However, `isValidAccount()` only checks entity existence — it never inspects the `type` column it fetches — so any entity (TOKEN, FILE, CONTRACT, TOPIC, SCHEDULE) passes the check. A long-zero EVM address like `0x0000000000000000000000000000000000001234` is parsed directly to an encoded entity ID without a DB lookup, meaning the guard is bypassed for any existing non-account entity, and the endpoint returns `200 OK` with (empty) token relationship data instead of `404 Not Found`.

5. **`rest/service/entityService.js` lines 122–124** — Because `evmAddress === null` for long-zero addresses, `getEncodedId` returns the encoded ID **directly without any DB lookup**: [5](#0-4)

```js
  ? entityId.getEncodedId()   // ← no DB check; succeeds for any numeric ID
  : await this.getEntityIdFromEvmAddress(entityId, requireResult);
```

**Root cause:** `entityExistenceQuery` fetches `entity.type` (the comment even says "use a small column in existence check"), but `isValidAccount()` discards that value and returns `true` for any non-null row — regardless of whether the entity is an ACCOUNT, TOKEN, FILE, CONTRACT, TOPIC, or SCHEDULE.

**Exploit flow:**
- Attacker supplies `0x0000000000000000000000000000000000001234` where entity `0.0.4660` is a TOKEN.
- `parseFromString` recognises the 12-byte zero prefix → returns `[shard, realm, 4660, null]`.
- `getEncodedId` takes the `evmAddress === null` branch → returns `4660` with no DB lookup.
- `isValidAccount(4660)` hits the DB, finds the TOKEN entity, returns `true`.
- `TokenService.getTokenAccounts()` is called with a token's own entity ID as the "owner account".
- Response: `200 OK { tokens: [], links: { next: null } }` instead of `404 Not Found`.

### Impact Explanation
The `/api/v1/accounts/{id}/tokens` endpoint is specified to operate only on accounts. Accepting TOKEN, FILE, CONTRACT, TOPIC, and SCHEDULE entity IDs as valid "accounts" violates the API contract. Concretely: (a) it leaks entity existence — a `200` vs `404` distinguishes whether a given numeric ID maps to any live entity, even non-accounts; (b) downstream consumers that rely on `404` to detect invalid accounts will silently receive empty-but-successful responses, potentially causing incorrect application logic. No funds are directly at risk, consistent with the Medium severity classification.

### Likelihood Explanation
No privileges are required. The attacker only needs to know (or enumerate) a numeric entity ID that belongs to a non-account entity — token IDs are publicly listed via `/api/v1/tokens`. The attack is trivially repeatable with a single HTTP GET request and requires no authentication or special network access.

### Recommendation
Fix `isValidAccount()` to check the fetched `type` value against the expected account types. The query already retrieves `entity.type`; the fix is to use it:

```js
// In entityService.js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  // entity.type must be ACCOUNT (or CONTRACT if contracts are valid here)
  return !isNil(entity) && entity.type === 'ACCOUNT';
}
```

Alternatively, add a `AND type = 'ACCOUNT'` clause directly to `entityExistenceQuery` so the DB filters non-account entities before the row is returned.

### Proof of Concept

**Precondition:** Entity `0.0.4660` exists in the mirror node DB as a TOKEN (easily verified via `GET /api/v1/tokens/0.0.4660`).

**Request:**
```
GET /api/v1/accounts/0x0000000000000000000000000000000000001234/tokens
```

**Expected response:** `404 Not Found` (entity is not an account).

**Actual response:**
```json
HTTP 200 OK
{
  "tokens": [],
  "links": { "next": null }
}
```

**Confirmation of entity-existence oracle:** Repeat with a numeric ID that does not exist in the entity table (e.g., `0x000000000000000000000000000000000000FFFF` for a non-existent entity). That request returns `404`, confirming the `200` for a TOKEN entity is caused by `isValidAccount()` accepting non-account types.
