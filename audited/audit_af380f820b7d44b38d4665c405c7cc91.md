### Title
Unauthenticated DoS via Missing Rate Limiting on `GET /api/v1/network/registered-nodes` Exhausting HikariCP Connection Pool

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module has no application-level rate limiting. Any unauthenticated external user can flood this endpoint with concurrent requests, each of which triggers a native SQL query against the database via `findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()`, exhausting the HikariCP connection pool and causing denial-of-service for all other users of the mirror node REST API.

### Finding Description

**Exact code path:**

`NetworkController.getRegisteredNodes()` at [1](#0-0)  handles `GET /api/v1/network/registered-nodes` with no authentication or rate-limiting annotation. It delegates to `NetworkServiceImpl.getRegisteredNodes()` at [2](#0-1) , which unconditionally calls the repository: [3](#0-2) 

This native SQL query runs against the database on every single request.

**Root cause — no rate limiting in rest-java:**

A search of all filters registered in the `rest-java` module reveals only three: `MetricsFilter` (records byte metrics only) [4](#0-3) , `ShallowEtagHeaderFilter` (ETag caching only), and `LoggingFilter`. None implement rate limiting. The `RestJavaConfiguration` registers no throttle filter: [5](#0-4) 

The rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) exists exclusively in the `web3` module for contract-call endpoints: [6](#0-5) 

It is entirely absent from `rest-java`.

**Connection pool:**

The application uses HikariCP, configured via `spring.datasource.hikari`: [7](#0-6) 

HikariCP's default `maximumPoolSize` is 10. The `rest-java` module has no `application.yml` overriding this (only `banner.txt` exists in resources). With thousands of concurrent requests each holding a connection for the duration of the DB query, the pool is trivially exhausted.

**Failed assumption:**

The codebase assumes that rate limiting is handled at the infrastructure layer (reverse proxy/API gateway). However, this is not enforced or documented as a deployment requirement, and the endpoint is directly exploitable at the application layer with no preconditions.

### Impact Explanation

When the HikariCP pool is exhausted, all subsequent requests to any endpoint in the `rest-java` service that require a database connection will block waiting for a connection or fail with a timeout error. This is a full denial-of-service for all users of the mirror node REST API — not just the `registered-nodes` endpoint. The `limit` parameter is bounded at 100 rows (`MAX_LIMIT = 100`) [8](#0-7) , which limits per-response data size but does nothing to limit request rate.

### Likelihood Explanation

Preconditions: none. No authentication, no API key, no account required. The endpoint is publicly documented in the OpenAPI spec: [9](#0-8) 

Any attacker with a basic HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can saturate the connection pool from a single machine. The attack is repeatable, stateless, and requires no knowledge of the system beyond the public API documentation.

### Recommendation

1. **Application-level rate limiting:** Port the existing bucket4j-based `ThrottleConfiguration` from the `web3` module into `rest-java`, or add a `OncePerRequestFilter` using bucket4j to enforce a per-IP or global requests-per-second limit on all `/api/v1/*` endpoints.
2. **Connection pool sizing + timeout:** Set an explicit `spring.datasource.hikari.maximumPoolSize` and a short `connectionTimeout` in `rest-java` configuration so that pool exhaustion fails fast rather than queuing indefinitely.
3. **Infrastructure enforcement:** Document and enforce that a rate-limiting reverse proxy (e.g., Nginx `limit_req`, Envoy, AWS API Gateway) is required in front of the `rest-java` service in all production deployments.

### Proof of Concept

```bash
# Flood the endpoint with 500 concurrent connections, unlimited requests
wrk -t 10 -c 500 -d 60s \
  "http://<mirror-node-host>/api/v1/network/registered-nodes"

# Observe: after pool exhaustion (default 10 connections),
# all other REST API endpoints begin returning 500 errors
# or timing out due to HikariCP connection wait timeout.

curl "http://<mirror-node-host>/api/v1/network/nodes"
# Returns: 500 Internal Server Error (connection pool exhausted)
```

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
