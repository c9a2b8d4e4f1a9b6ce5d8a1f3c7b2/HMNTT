### Title
Unauthenticated Timing Side-Channel on `/health/readiness` Leaks Internal Infrastructure State

### Summary
The `/health/readiness` endpoint performs live I/O checks against PostgreSQL and the internal Rosetta network API (each with a 10-second timeout), while `/health/liveness` returns instantly with no checks. Both endpoints are exposed without authentication or rate limiting. Any unprivileged external user can measure the response latency of `/health/readiness` to infer the availability and performance of internal backend services.

### Finding Description
**Code location:** `rosetta/app/middleware/health.go`, `NewHealthController()` lines 37–50, `Routes()` lines 63–78.

`livenessHealth` is created with `health.New()` — no checks registered — so `GET /health/liveness` always returns in microseconds. [1](#0-0) 

`readinessHealth` registers two blocking I/O checks, each with a 10-second timeout:
- `postgresql`: opens a real connection to the PostgreSQL DSN
- `network`: makes two sequential HTTP calls (`/network/list`, `/network/status`) to `localhost:<port>` [2](#0-1) 

The `checkNetworkStatus` function performs two sequential live HTTP round-trips before returning: [3](#0-2) 

The middleware stack in `main.go` is `MetricsMiddleware → TracingMiddleware → CorsMiddleware` — no authentication and no rate limiting are applied to any route, including the health endpoints: [4](#0-3) 

`trace.go` confirms health paths are treated as "internal" only for log-level suppression (debug vs info), not for access control: [5](#0-4) 

**Root cause:** The asymmetry between the two endpoints is by design (liveness = process alive, readiness = dependencies alive), but no access control prevents external actors from exploiting the latency difference as an oracle for internal state.

**Exploit flow:**
1. Attacker sends `GET /health/liveness` → receives response in <1 ms (baseline).
2. Attacker sends `GET /health/readiness` → response time directly reflects PostgreSQL connection latency + two HTTP round-trips to the internal Rosetta API.
3. By comparing the two, the attacker can determine: whether PostgreSQL is reachable, how fast it responds, and whether the internal network API is healthy — all without any credentials.
4. Repeated polling (no rate limit) allows continuous monitoring of internal infrastructure state over time.

### Impact Explanation
An external attacker gains a continuous, unauthenticated oracle into internal infrastructure health: PostgreSQL reachability and response time, and internal Rosetta API availability. This can be used to time attacks during degraded states, confirm the effect of a DoS against the database, or map internal service topology. The 10-second timeout means a fully blocked DB causes a near-10-second response, making the signal unambiguous.

### Likelihood Explanation
No preconditions beyond network reachability to the Rosetta port. The attack is trivially repeatable with `curl` or any HTTP client. No credentials, tokens, or special headers are required. The signal-to-noise ratio is high because liveness always returns instantly, making the delta unambiguous even over noisy networks.

### Recommendation
1. **Restrict health endpoints to internal/cluster networks** via network policy, reverse proxy ACL, or a middleware check on `RemoteAddr` — the simplest and most effective fix.
2. Alternatively, **add a fixed artificial delay** to the liveness response to eliminate the timing differential (though this is weaker).
3. **Add rate limiting** to health endpoints to prevent continuous polling as a monitoring oracle.
4. If the endpoints must remain public (e.g., for Kubernetes probes from external load balancers), consider returning only a binary UP/DOWN status with a **fixed response time** (pad with `time.Sleep` to a constant duration) to eliminate the timing signal.

### Proof of Concept
```bash
# Baseline: liveness always returns instantly
time curl -s http://<rosetta-host>:<port>/health/liveness

# Oracle: readiness latency reflects DB + network state
time curl -s http://<rosetta-host>:<port>/health/readiness

# When PostgreSQL is unreachable, the second command takes ~10 seconds
# When PostgreSQL is slow, the second command takes proportionally longer
# Repeat in a loop to monitor internal state continuously:
while true; do
  START=$(date +%s%N)
  curl -s http://<rosetta-host>:<port>/health/readiness > /dev/null
  END=$(date +%s%N)
  echo "readiness latency: $(( (END - START) / 1000000 )) ms"
  sleep 1
done
```
