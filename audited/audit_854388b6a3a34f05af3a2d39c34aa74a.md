### Title
Cache Key Hash Collision in `findByEvmAddressOrAliasAndDeletedIsFalse` Enables Cross-Entity Data Leakage

### Summary
`SpelHelper.hashCode(byte[])` returns a 32-bit `Arrays.hashCode()` integer that is used directly as the Caffeine cache key for `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS`. Because the cache key is the integer itself (not the byte array), two distinct alias byte arrays that produce the same 32-bit hash value map to the same cache entry. An attacker who controls an entity with a crafted alias can poison the cache so that subsequent lookups for a victim's alias return the attacker's entity data.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode` — `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);   // returns a 32-bit int
}
```

`findByEvmAddressOrAliasAndDeletedIsFalse` — `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 39–49:
```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← key is Integer, not byte[]
        unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

The same pattern applies to `findByEvmAddressAndDeletedIsFalse` (lines 32–37, `CACHE_NAME_EVM_ADDRESS`).

**Root cause:** The cache key expression evaluates to a boxed `Integer`. Caffeine (backed by `ConcurrentHashMap`) uses `Integer.equals()` for key comparison. Therefore, the cache key IS the 32-bit integer — not the byte array. Two distinct byte arrays `aliasA ≠ aliasB` where `Arrays.hashCode(aliasA) == Arrays.hashCode(aliasB)` are treated as identical cache keys. The first result cached under that integer is returned for both inputs.

**Why existing checks fail:**
- `unless = "#result == null"` only prevents caching of empty results; it does not prevent a collision from serving the wrong entity.
- No equality check on the actual byte array content is performed at cache retrieval time.
- The Caffeine cache manager (`CACHE_MANAGER_ENTITY`, `EvmConfiguration.java` lines 99–105) applies a single TTL to all entries; there is no per-entry validation.

**Exploit flow:**
1. Attacker identifies victim entity B with alias `aliasB` (publicly observable on-chain).
2. Attacker computes `target = Arrays.hashCode(aliasB)` (trivial, deterministic).
3. Attacker searches for a byte array `aliasA` such that `Arrays.hashCode(aliasA) == target`. For a 20-byte EVM address space, the expected search cost is ~2^32 evaluations of `Arrays.hashCode()` — feasible in seconds on commodity hardware since `Arrays.hashCode()` is a simple polynomial loop with no cryptographic hardness.
4. Attacker registers entity A on the Hedera network with alias `aliasA` (standard account creation, no special privilege required).
5. Attacker sends a request to the mirror node web3 API that triggers `findByEvmAddressOrAliasAndDeletedIsFalse(aliasA)`. Entity A is fetched from DB and cached under key `Integer(target)`.
6. Any subsequent request for `aliasB` hits the cache (same key `Integer(target)`) and returns entity A's data instead of entity B's.
7. During a network partition, the DB is unreachable; the stale wrong-entity mapping persists for the full cache TTL, making the leakage persistent and unrecoverable until TTL expiry.

**Call path to the vulnerable method** (via `CommonEntityAccessor.java`, lines 46–50):
```java
public Optional<Entity> get(Bytes alias, Optional<Long> timestamp) {
    return timestamp
        .map(t -> entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(...))
        .orElseGet(() -> entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(alias.toByteArray()));
}
```
This is invoked for any account/alias resolution in the EVM execution path, making it reachable by any `eth_call` or contract execution request.

### Impact Explanation
A successful cache poisoning causes the mirror node to return entity A's full state (balance, keys, type, EVM address, etc.) in response to queries for entity B's alias. Any downstream EVM execution that relies on this lookup — account balance checks, contract address resolution, key validation — operates on the wrong entity. This constitutes cross-entity data leakage and incorrect EVM state, which can affect smart contract execution outcomes. Severity is high: the wrong entity's data is served authoritatively with no error indication.

### Likelihood Explanation
The attack requires no privileged access. Any user who can submit transactions to the Hedera network (standard account creation) and send HTTP requests to the mirror node web3 API can execute this. Finding a 32-bit `Arrays.hashCode()` collision for a target alias is computationally trivial — the hash function is a simple polynomial (`result = 31 * result + b`) with no cryptographic properties, and 2^32 is well within reach of a single CPU in under a minute. The attack is repeatable and deterministic once a collision pair is found.

### Recommendation
Replace the integer hash with the actual byte array content as the cache key, using a representation that supports correct equality semantics. The simplest fix is to use a hex-encoded string:

```java
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```

Or wrap the byte array in a type that implements `equals`/`hashCode` via content comparison (e.g., `ByteBuffer.wrap(alias)`). This ensures two distinct byte arrays never share a cache entry regardless of their `Arrays.hashCode()` value.

### Proof of Concept

```java
// Step 1: Find a collision pair (offline, no network access needed)
byte[] aliasB = /* victim's known alias bytes, e.g. 20-byte EVM address */;
int target = Arrays.hashCode(aliasB);

// Brute-force search for aliasA with same hash
// For 20-byte arrays, expected ~2^32 iterations; for shorter arrays, fewer
byte[] aliasA = new byte[aliasB.length];
Random rng = new Random();
while (true) {
    rng.nextBytes(aliasA);
    if (Arrays.hashCode(aliasA) == target && !Arrays.equals(aliasA, aliasB)) {
        break; // collision found
    }
}

// Step 2: Register entity A on Hedera with aliasA as its alias (standard tx)
// Step 3: Trigger cache population for aliasA
// GET /api/v1/contracts/call  (or any eth_call that resolves aliasA)

// Step 4: Query for aliasB — cache returns entity A's data
// GET /api/v1/contracts/call  (resolving aliasB)
// → Response contains entity A's balance, keys, EVM address (WRONG entity)
```
