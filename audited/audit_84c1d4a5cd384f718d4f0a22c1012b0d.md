### Title
Unauthenticated Empty-Result Query Flooding via Non-Existent `ownerAccountId` in `getAccountTokenAllowances()`

### Summary
`EntityService.getEncodedId()` skips any database existence check for plain numeric account IDs, returning an encoded ID directly. The controller then unconditionally passes this ID to `TokenAllowanceService.getAccountTokenAllowances()`, which executes a `SELECT` against the `token_allowance` table. With no rate limiting in the REST module and a bypassable optional cache, an unprivileged attacker can flood the database with empty-result queries by cycling through arbitrary large numeric IDs.

### Finding Description

**Code path:**

1. `rest/controllers/tokenAllowanceController.js`, line 69:
   ```js
   const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
   ``` [1](#0-0) 

2. `rest/service/entityService.js`, lines 120–123: When the input is a valid numeric entity ID (e.g., `9999999999`), `EntityId.isValidEntityId()` returns `true`, `EntityId.parseString()` succeeds, and `evmAddress === null`, so `getEncodedId()` returns `entityId.getEncodedId()` **immediately — no DB lookup, no existence check**. [2](#0-1) 

3. Back in the controller (line 72), the encoded ID is passed directly to `TokenAllowanceService.getAccountTokenAllowances(query)`. [3](#0-2) 

4. `rest/service/tokenAllowanceService.js`, lines 86–89: `getAccountTokenAllowances()` calls `getQuery()` which builds and executes `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ORDER BY ... LIMIT $2` unconditionally. [4](#0-3) [5](#0-4) 

**Root cause:** The failed assumption is that `getEncodedId()` validates entity existence for numeric IDs. It does not — it only performs DB lookups for aliases and EVM addresses.

**Why existing checks fail:**

- **Response cache** (`responseCacheCheckHandler`/`responseCacheUpdateHandler`): Only active when both `config.cache.response.enabled` and `config.redis.enabled` are true — an optional deployment dependency. Even when active, the cache key is the full URL path+query string. An attacker cycling through IDs (`9999999999`, `9999999998`, `9999999997`, …) generates a unique cache key per request, producing a cache miss every time. [6](#0-5) [7](#0-6) 

- **No rate limiting in the REST module**: `rest/server.js` registers no rate-limiting middleware. The `ThrottleConfiguration`/`ThrottleManagerImpl` found in the codebase belongs exclusively to the `web3` Java module and does not apply here. [8](#0-7) 

### Impact Explanation
Each request causes a real PostgreSQL query execution against the `token_allowance` table. While an indexed scan on `owner` returning zero rows is individually cheap, sustained high-rate flooding (thousands of requests/second from a single client or botnet) can: exhaust the DB connection pool, increase query planner/executor CPU load, and degrade response times for legitimate users. The impact is griefing/availability degradation with no economic damage to any on-chain user — consistent with the Medium scope classification.

### Likelihood Explanation
No privileges, tokens, or on-chain accounts are required. Any HTTP client can issue `GET /api/v1/accounts/<large_number>/allowances/tokens`. The attack is trivially scriptable, repeatable, and requires no prior knowledge of the system state. The only barrier is network-level infrastructure (e.g., a CDN or WAF in front of the API), which is not enforced by the application code itself.

### Recommendation
1. **Add entity existence pre-check for numeric IDs**: Before executing the `token_allowance` query, verify the `ownerAccountId` exists in the `entity` table (reuse `EntityService.isValidAccount()` or inline the `entityExistenceQuery`). Return HTTP 404 immediately for non-existent accounts, avoiding the `token_allowance` query entirely. [9](#0-8) 

2. **Add rate limiting to the REST module**: Introduce a per-IP (or global) request rate limiter middleware (e.g., `express-rate-limit`) in `rest/server.js`, applied before route handlers. [8](#0-7) 

### Proof of Concept
```bash
# Single request — returns 200 with empty allowances, DB query executed
curl "https://<mirror-node-host>/api/v1/accounts/9999999999/allowances/tokens"
# {"allowances":[],"links":{"next":null}}

# Flood with cycling IDs to bypass cache (no rate limit enforced)
for i in $(seq 9000000000 9000010000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${i}/allowances/tokens" &
done
wait
```

Each iteration: `getEncodedId("9000000XXX")` returns encoded ID without DB lookup → `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0` executes → empty result → 200 response. No authentication, no rate limit, no existence gate stops this loop.

    let sqlQuery;
    if (subQueries.length === 0) {
