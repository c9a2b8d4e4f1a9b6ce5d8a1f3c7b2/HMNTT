### Title
Cache Stampede (Thundering Herd) in `getCachedTokens()` Allows Unprivileged Users to Multiply DB Queries

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` performs a synchronous cache check followed by an `async` DB query with no in-flight deduplication. Because Node.js yields the event loop at every `await`, N concurrent requests for the same uncached token IDs will all observe cache misses before any of them writes back, causing N redundant DB queries to execute simultaneously. No authentication or privilege is required to trigger this.

### Finding Description
**Code path:** `rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161, called from `getTokenAccounts()` at line 104.

**Root cause — check-then-act across an `await` boundary:**

```
// Line 139-146: synchronous cache check
tokenIds.forEach((tokenId) => {
  const cachedToken = tokenCache.get(tokenId);   // ← check (sync)
  if (cachedToken) { ... } else { uncachedTokenIds.push(tokenId); }
});

// Line 148-150: early return only if ALL ids are cached
if (uncachedTokenIds.length === 0) { return cachedTokens; }

// Line 152: async DB query — yields the event loop HERE
const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);

// Lines 153-158: write-back to cache — only reached AFTER the await resolves
rows.forEach((row) => { tokenCache.set(tokenId, cachedToken); ... });
```

`tokenCache` is a module-level `quickLru` instance (lines 12–14). It is synchronous and has no concept of "in-flight" entries. When Request A reaches `await super.getRows(...)` at line 152, it suspends and returns control to the event loop. Requests B, C, … D that arrive before A's `await` resolves will each execute the synchronous cache check at line 140, find the same misses (because A has not yet written back), push the same `uncachedTokenIds`, and each independently issue their own `await super.getRows(...)` call. All N DB queries execute concurrently. Each one then writes the same data back to the cache redundantly.

There is no mutex, no promise deduplication map, and no "pending" sentinel value placed in the cache before the `await`.

**Why existing checks are insufficient:**
- The `quickLru` cache check at line 140 is purely synchronous; it cannot see that another coroutine is already fetching the same keys.
- The early-return guard at line 148 only fires when the cache is already warm; it provides no protection during the population race.
- There is no rate-limiting middleware on the `/accounts/:id/tokens` endpoint that would prevent a single unauthenticated client from sending hundreds of concurrent requests.

### Impact Explanation
Each concurrent request that races through the cache miss path issues a full `SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = any($1)` query against the database. With N concurrent requests and M uncached token IDs per request, the database receives up to N×M unnecessary row lookups simultaneously. Under sustained attack this can saturate the DB connection pool (bounded by `db.pool.maxConnections`), causing legitimate requests to queue or time out. The impact is resource exhaustion / griefing with no data integrity consequence.

### Likelihood Explanation
Any unauthenticated HTTP client can open many parallel TCP connections and fire concurrent GET requests to `/api/v1/accounts/<id>/tokens`. The attack is most effective immediately after a server restart (cold cache) or after the LRU evicts entries due to `maxSize` pressure (which an attacker can induce by first requesting many distinct token IDs). The attack is trivially repeatable with standard tools (`ab`, `wrk`, `curl --parallel`).

### Recommendation
Deduplicate in-flight DB fetches by storing a `Promise` in the cache before the `await`, so that concurrent callers for the same key await the same promise rather than issuing independent queries:

```js
// Replace the uncached branch with a promise-coalescing pattern:
const inFlight = new Map(); // module-level

async getCachedTokens(tokenIds) {
  const uncachedTokenIds = [];
  tokenIds.forEach((tokenId) => {
    if (!tokenCache.has(tokenId) && !inFlight.has(tokenId)) {
      uncachedTokenIds.push(tokenId);
    }
  });

  if (uncachedTokenIds.length > 0) {
    const fetchPromise = super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])
      .then((rows) => {
        rows.forEach((row) => tokenCache.set(row.token_id, new CachedToken(row)));
        uncachedTokenIds.forEach((id) => inFlight.delete(id));
      });
    uncachedTokenIds.forEach((id) => inFlight.set(id, fetchPromise));
    await fetchPromise;
  } else {
    // await any in-flight promises for ids we need
    await Promise.all([...new Set(tokenIds)].map((id) => inFlight.get(id)).filter(Boolean));
  }
  // build result from tokenCache ...
}
```

Alternatively, use a library such as `p-memoize` or `async-singleflight` that provides built-in promise deduplication.

### Proof of Concept
**Preconditions:** Mirror Node REST API running with a cold token cache (fresh start or after LRU eviction). An account (e.g., `0.0.12345`) has ≥1 token association.

**Steps:**
```bash
# Send 200 concurrent requests before any cache entry is written
seq 200 | xargs -P200 -I{} \
  curl -s "http://localhost:5551/api/v1/accounts/0.0.12345/tokens" -o /dev/null

# Observe in DB slow-query log or pg_stat_activity:
# 200 simultaneous executions of:
# SELECT decimals, freeze_status, kyc_status, token_id
# FROM token WHERE token_id = any($1)
# all with identical parameters, all returning identical rows.
```

**Expected result without fix:** DB logs show N near-identical concurrent queries for the same token IDs.
**Expected result with fix:** DB logs show exactly 1 query; all other concurrent requests await that single promise.
