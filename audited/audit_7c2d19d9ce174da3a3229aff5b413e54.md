### Title
`requestStartTime` Recorded After Async Thread-Pool Operation Enables Metric Corruption via Request Flooding

### Summary
In `rest/middleware/requestHandler.js`, `requestLogger()` records `res.locals[requestStartTime]` via `Date.now()` only **after** `await randomString(8)` resolves. `randomString` delegates to `util.promisify(crypto.randomBytes)`, which runs on Node.js's libuv thread pool (default size: 4). An unprivileged attacker flooding the API with concurrent requests saturates the thread pool, causing `randomBytes` calls to queue, delaying `Date.now()` by an attacker-controlled amount and making all latency metrics appear artificially low.

### Finding Description
**Exact code path:**

`rest/middleware/requestHandler.js`, lines 22–29:
```js
const requestLogger = async (req, res) => {
  const requestId = await randomString(8);   // <-- thread-pool await
  httpContext.set(requestIdLabel, requestId);
  res.locals.statusCode = httpStatusCodes.OK.code;
  res.locals[requestStartTime] = Date.now(); // <-- recorded AFTER await
};
```

`rest/utils.js`, lines 1023–1028:
```js
const randomBytesAsync = util.promisify(crypto.randomBytes);
const randomString = async (length) => {
  const bytes = await randomBytesAsync(Math.max(2, length) / 2); // 4 bytes → libuv thread pool
  return bytes.toString('hex');
};
```

**Root cause:** The failed assumption is that `await randomBytesAsync(4)` is effectively instantaneous. In reality it occupies one of the 4 libuv worker threads (`UV_THREADPOOL_SIZE=4` default). When more than 4 concurrent requests are in-flight, subsequent `randomBytes` calls queue in the thread pool. `Date.now()` is only called after the queued operation drains, which can be seconds after the actual TCP connection was accepted by Express.

**Why existing checks are insufficient:** No rate-limiting or connection-throttling middleware is present in the request handler chain. The `config.users` authentication system only caps response result counts, not request concurrency.

### Impact Explanation
Every downstream consumer of `res.locals[requestStartTime]` — including `httpErrorHandler.js` (`elapsed = Date.now() - startTime`) and `responseCacheHandler.js` (`elapsed = Date.now() - startTime`) — computes latency as `responseTime − requestStartTime`. Because `requestStartTime` is inflated (recorded late), computed `elapsed` values are deflated. Under sustained flood, SLA dashboards and alerting thresholds based on these metrics show false compliance while real user-perceived latency is high. Operators cannot detect actual degradation. Severity: Medium (griefing, no direct data loss, but operational blind-spot).

### Likelihood Explanation
Preconditions: none — the REST API is publicly accessible without authentication for most endpoints. The attacker needs only a standard HTTP flood tool (`wrk`, `ab`, `curl --parallel`). Saturating a 4-thread pool requires only ~5+ concurrent persistent connections. The attack is trivially repeatable and requires no special knowledge of the application.

### Recommendation
Move `res.locals[requestStartTime] = Date.now()` to the **first line** of `requestLogger`, before any `await`:

```js
const requestLogger = async (req, res) => {
  res.locals[requestStartTime] = Date.now(); // capture arrival time immediately
  const requestId = await randomString(8);
  httpContext.set(requestIdLabel, requestId);
  res.locals.statusCode = httpStatusCodes.OK.code;
};
```

This ensures the timestamp reflects actual request arrival regardless of thread-pool queue depth. Optionally, consider using `crypto.randomBytes` synchronously (`crypto.randomBytes(4).toString('hex')`) for the small 4-byte generation to eliminate the thread-pool dependency entirely.

### Proof of Concept
```bash
# 1. Start the mirror node REST API locally.

# 2. Flood with concurrent requests (saturates the 4-thread libuv pool):
wrk -t8 -c200 -d30s http://localhost:5551/api/v1/accounts

# 3. Observe server logs: the "in X ms" elapsed values will be
#    anomalously low (e.g., 1–5 ms) even for requests that took
#    hundreds of ms from the client's perspective, because
#    requestStartTime was recorded after the queued randomBytes resolved.

# 4. Confirm by patching requestLogger to log both:
#    - req._startAt (set by Express before middleware runs)
#    - res.locals[requestStartTime]
#    Under flood, the delta between the two grows proportionally
#    to thread-pool queue depth.
```
