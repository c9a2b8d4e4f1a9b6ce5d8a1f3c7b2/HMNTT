### Title
Unbounded `nodeIds` Array Input Enables Resource Exhaustion DoS on `/api/v1/network/nodes`

### Summary
`NetworkNodeRequest.nodeIds` accepts an unlimited number of `node.id` query parameters with no `@Size` constraint, unlike the analogous `RegisteredNodesRequest.registeredNodeIds` field which is capped at `@Size(max = 2)`. An unauthenticated attacker can submit thousands of `node.id=eq:X` parameters per request, forcing allocation of a large `Long[]` array in Java and serialization of that array as a PostgreSQL `any(:nodeIds)` parameter on every call, enabling repeated resource exhaustion that degrades fee schedule lookups for legitimate users.

### Finding Description

**Code path:**

`NetworkNodeRequest.nodeIds` (line 36-38) has no `@Size` constraint: [1](#0-0) 

Compare with `RegisteredNodesRequest.registeredNodeIds` (line 42-44), which correctly applies `@Size(max = 2)`: [2](#0-1) 

In `RequestParameterArgumentResolver.validateAndAddParameter()`, when the field is a `Collection`, all submitted parameter values are passed through as an array with no count check: [3](#0-2) 

In `NetworkServiceImpl.getNetworkNodes()`, every `EQ` parameter is added to a `HashSet` with no size cap, and the entire set is converted to `Long[]` and forwarded to the repository: [4](#0-3) 

The SQL query then evaluates `abe.node_id = any(:nodeIds)` with the unbounded array: [5](#0-4) 

**Root cause:** The `nodeIds` field in `NetworkNodeRequest` is missing the `@Size` constraint that the codebase already applies to equivalent list parameters in other DTOs. The `RequestParameterArgumentResolver` enforces no count limit on multi-value parameters, and the service layer performs no size check before constructing the array.

**Why existing checks fail:**
- `@Min(1)` on `limit` only bounds the result count, not the input array size.
- `getEffectiveLimit()` caps results at 25 but does not touch `nodeIds`.
- The `HashSet` deduplication in the service layer reduces duplicates but does not cap the number of distinct values an attacker can supply.
- Tomcat's default `maxParameterCount` is 10,000, so an attacker can submit up to ~10,000 distinct `node.id=eq:X` values per request.

### Impact Explanation
Each malicious request forces: (1) allocation of a `Long[~10,000]` array on the Java heap, (2) JDBC serialization of that array into a PostgreSQL array literal, and (3) PostgreSQL evaluation of `any({...10,000 elements...})` against every row in the address book. Concurrent flood of such requests causes heap pressure, increased GC pauses, and elevated DB connection hold time. Because the `/api/v1/network/nodes` endpoint is used to resolve fee schedules and node connectivity, sustained degradation disrupts fee schedule lookups for legitimate users. Severity is **Medium** — the address book is small so per-query DB work is bounded, but the Java/JDBC overhead is real and repeatable.

### Likelihood Explanation
No authentication is required. The endpoint is public. The attack requires only a standard HTTP client capable of sending a URL with thousands of repeated query parameters — trivially scriptable. It is repeatable at high frequency from a single machine or botnet. The inconsistency with `RegisteredNodesRequest` shows this was an oversight rather than an intentional design choice, making it unlikely to be mitigated by any undocumented layer.

### Recommendation
Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds`, consistent with the pattern already used in `RegisteredNodesRequest`:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 25)   // match MAX_LIMIT or a domain-appropriate cap
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, add a guard in `NetworkServiceImpl.getNetworkNodes()` to reject or truncate oversized lists before processing, as defense-in-depth.

### Proof of Concept

```bash
# Build a URL with 5000 distinct node.id equality parameters
python3 -c "
params = '&'.join(f'node.id={i}' for i in range(5000))
print(f'GET /api/v1/network/nodes?{params}')
" | head -c 200

# Send it (repeat concurrently for DoS effect)
python3 -c "
import urllib.request, threading
url = 'http://<mirror-node-host>/api/v1/network/nodes?' + '&'.join(f'node.id={i}' for i in range(5000))
def attack():
    for _ in range(100):
        urllib.request.urlopen(url)
threads = [threading.Thread(target=attack) for _ in range(20)]
[t.start() for t in threads]
[t.join() for t in threads]
"
```

Expected result: each request forces a `Long[5000]` array allocation and a PostgreSQL `any({0,1,...,4999})` evaluation. Under concurrent load, heap usage and GC pressure increase measurably, degrading response times for legitimate callers of the fee schedule and node-lookup endpoints.
