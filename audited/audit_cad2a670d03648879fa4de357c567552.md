### Title
Unbounded URL Logging in `TracingMiddleware` Enables Log Amplification Resource Exhaustion

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` captures `request.URL.RequestURI()` without any length truncation and embeds it verbatim into a `log.Info` message on every non-internal request. Because Go's `http.Server` is configured without `MaxHeaderBytes` (defaulting to 1 MB), an unprivileged attacker can craft requests with query strings up to ~1 MB, causing per-request heap allocations and log I/O proportional to the URL size. With no application-level rate limiting in the default deployment, a low-volume stream of such requests is sufficient to amplify log I/O and memory pressure well beyond 30% of baseline.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 43–61:

```go
path := request.URL.RequestURI()          // line 47 – full URI, no length cap
// ...
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)  // lines 52-53 – path embedded verbatim
// ...
log.Info(message)                          // line 58 – always logged at Info for non-internal paths
```

`rosetta/main.go`, lines 220–227: the `http.Server` is constructed with no `MaxHeaderBytes` field, so Go's default `DefaultMaxHeaderBytes = 1 << 20` (1,048,576 bytes) applies. The request line (method + URL + HTTP version) counts toward this budget, meaning URLs of several hundred KB are accepted by the server before any application logic runs.

**Root cause:** `request.URL.RequestURI()` returns the raw path plus the full, untruncated query string. There is no `maxLen` guard, no `path[:n]` truncation, and no sanitization before the value is stored in `path` and then formatted into `message`. The `internalPaths[path]` check on line 55 uses the full string including query parameters, so even `/health/liveness?AAAA…` does not match the map and is logged at `Info` level.

**Exploit flow:**
1. Attacker sends HTTP requests (GET or POST) to any Rosetta endpoint with a query string padded to ~500 KB (e.g., `GET /network/list?a=AAAA…AAAA HTTP/1.1`).
2. Go's HTTP parser accepts the request (within the 1 MB `MaxHeaderBytes` default).
3. `TracingMiddleware` executes: `path` is allocated as a ~500 KB string; `message` is allocated as a second ~500 KB string; `log.Info(message)` writes ~500 KB to the log sink (file, stdout, or syslog).
4. The upstream handler returns quickly (404 or 422 for an unknown query string), so the attacker's cost is minimal while the node pays full allocation + I/O cost.
5. At 10 such requests per second, the node writes ~5 MB/s of log data and allocates ~10 MB/s of short-lived heap objects solely from this middleware, compared to ~200 bytes/s under normal traffic — a >25,000× amplification ratio.

**Why existing checks are insufficient:**
- Go's 1 MB `MaxHeaderBytes` is the only bound; it is not a mitigation, it is the attack surface ceiling.
- The Traefik `rateLimit` and `inFlightReq` middleware (`charts/hedera-mirror-rosetta/values.yaml`, lines 152–160) are only active when `global.middleware: true` is set, which defaults to `false` (line 95). Direct deployments (docker-compose, bare-metal) have no rate limiting.
- The nginx proxy in `docker-compose.yml` (lines 234–235) proxies `/rosetta/` with no `large_client_header_buffers` or `client_header_buffer_size` restriction that would reject large URLs before they reach the Go server.

### Impact Explanation
An attacker with no credentials can cause sustained, disproportionate log I/O and heap allocation pressure on the Rosetta node. In deployments where logs are written to disk, this can exhaust disk space, causing the node to crash or stop accepting requests. Even before disk exhaustion, the GC pressure from repeated large short-lived allocations and the synchronous log I/O can degrade request throughput and latency for legitimate users. The impact is amplified because the Rosetta API is a critical infrastructure component used by exchanges for blockchain data integration.

### Likelihood Explanation
The attack requires no authentication, no special knowledge of the system, and no exploit tooling beyond a standard HTTP client. The attacker only needs to know the server's IP and port (publicly exposed). The attack is repeatable, automatable, and can be sustained indefinitely. The only practical barrier is network bandwidth to the attacker, which is negligible given that each request is small (the large query string is sent once per request, and the attacker's outbound cost is ~500 KB/request while the node's log I/O cost is the same).

### Recommendation
Truncate `path` to a safe maximum length before embedding it in the log message:

```go
const maxLoggedPathLen = 512

path := request.URL.RequestURI()
loggedPath := path
if len(loggedPath) > maxLoggedPathLen {
    loggedPath = loggedPath[:maxLoggedPathLen] + "…"
}
// use loggedPath in the fmt.Sprintf, keep path for internalPaths lookup
```

Additionally:
1. Set `MaxHeaderBytes` explicitly on the `http.Server` in `rosetta/main.go` to a value appropriate for the Rosetta API (e.g., 8 KB), since Rosetta endpoints use POST bodies, not query strings.
2. Enable the Traefik `rateLimit` middleware by default (`global.middleware: true`) or add application-level rate limiting unconditionally.
3. Add an nginx `large_client_header_buffers` / `client_header_buffer_size` directive in the docker-compose proxy config to reject oversized request lines before they reach the Go server.

### Proof of Concept

```bash
# Generate a ~500 KB query string and send to the Rosetta network/list endpoint
python3 -c "
import socket, time
payload = 'GET /network/list?' + 'A'*500000 + ' HTTP/1.1\r\nHost: localhost:5700\r\n\r\n'
for i in range(20):
    s = socket.create_connection(('localhost', 5700))
    s.sendall(payload.encode())
    s.recv(4096)
    s.close()
    time.sleep(0.1)
print('Done – check node log size and CPU/IO metrics')
"
```

Expected result: the node's log file grows by ~10 MB in 2 seconds; `iostat` shows a spike in write throughput; heap profiling shows repeated ~1 MB allocations in `TracingMiddleware`. Under normal traffic the same 20 requests would produce ~4 KB of log output total.
