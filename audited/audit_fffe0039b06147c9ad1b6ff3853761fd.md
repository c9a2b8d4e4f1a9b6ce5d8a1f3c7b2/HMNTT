### Title
Unbounded Historical Retrieval Loop via Full-Page Poll Manipulation in `isComplete()` Causes Resource Exhaustion

### Summary
`PollingTopicMessageRetriever.PollingContext.isComplete()` uses `pageSize.get() < retrieverProperties.getMaxPageSize()` as a proxy for "no more historical data exists." An unprivileged attacker who pre-populates a topic with enough messages (or continuously publishes to it) can ensure every poll returns exactly `maxPageSize` results, keeping `isComplete()` permanently `false`. Because `numRepeats = Long.MAX_VALUE` for throttled mode and the `timeout()` operator only fires on emission gaps (not total subscription duration), the historical retrieval flux never terminates, exhausting scheduler threads and database connections and starving other subscribers of resources.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, `isComplete()`, lines 121–129:
```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;
}
```

`pageSize` is reset to `0` at the start of each poll (line 73) and incremented per message (line 134). If a poll returns exactly `maxPageSize` messages, `pageSize.get() == maxPageSize`, so `pageSize.get() < maxPageSize` evaluates to `false`. With no filter limit set, `limitHit` is also `false`. `isComplete()` returns `false`.

**Repeat control** (line 52):
```java
.repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
```
For `throttled=true`, `numRepeats = Long.MAX_VALUE` (line 99). The flux repeats indefinitely as long as `isComplete()` returns `false`.

**Timeout** (line 59):
```java
.timeout(retrieverProperties.getTimeout(), scheduler)
```
Reactor's `timeout()` fires only when no item is emitted within the window (default 60 s). Because the attacker ensures each poll emits `maxPageSize` messages continuously, items are always flowing and the timeout never triggers.

**Downstream starvation** (`TopicMessageServiceImpl.java`, line 72–73):
```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live)...);
```
`live` (gossip delivery) only starts after `historical` completes. A never-completing `historical` flux means the subscriber never transitions to live mode, and the subscription permanently occupies a `boundedElastic` scheduler thread and a database connection.

**Root cause:** The design assumes a full page implies more data *might* exist and schedules one more poll, but provides no upper bound on how many consecutive full-page polls are tolerated before forcing completion. The timeout is not a wall-clock subscription lifetime limit; it is an inter-emission idle timeout, which is ineffective when data flows continuously.

### Impact Explanation

Each stuck subscription:
- Holds a `Schedulers.boundedElastic()` thread (default cap: `10 × CPU cores`) for the lifetime of the subscription.
- Issues a database query every `pollingFrequency` (default 2 s), holding a connection from the pool.
- Never delivers live/gossip messages to that subscriber.

An attacker opening multiple gRPC connections (each allowing `maxConcurrentCallsPerConnection = 5` concurrent streams) and subscribing on each can exhaust the thread pool and database connection pool. Once exhausted, legitimate subscribers experience delayed or failed subscriptions — a server-wide denial of service affecting gossip delivery to all clients.

### Likelihood Explanation

Any unprivileged user with a Hedera account can:
1. Create a topic (one-time small HBAR fee).
2. Submit ≥ `maxPageSize` × N messages (N = desired number of polls before the topic is exhausted). A topic with 10 million messages sustains ~10,000 full-page polls ≈ 5.5 hours of stuck subscription per subscriber at default settings.
3. Alternatively, continuously submit messages at ≥ 500 msg/s to keep the topic growing faster than the retriever drains it, making the subscription permanent.
4. Open as many gRPC connections as the server allows and subscribe on each.

No privileged access, no authentication bypass, and no code modification is required. The attack is repeatable and scalable.

### Recommendation

1. **Add a wall-clock subscription lifetime limit** independent of emission activity. Replace or supplement `.timeout(retrieverProperties.getTimeout(), scheduler)` with a `take(Duration)` or a `Flux.interval`-based cutoff that terminates the historical flux after a maximum elapsed time regardless of message flow.

2. **Cap consecutive full-page polls in throttled mode.** Introduce a `maxThrottledPolls` property (analogous to `unthrottled.maxPolls`) and enforce it in `isComplete()`:
   ```java
   if (throttled) {
       return pageSize.get() < retrieverProperties.getMaxPageSize()
           || limitHit
           || pollCount.get() >= retrieverProperties.getMaxThrottledPolls();
   }
   ```

3. **Track and limit per-connection or per-IP active subscriptions** at the gRPC layer to bound the number of concurrent stuck retrievals.

4. **Use a total-duration timeout** via `.timeout(Duration, fallback)` where `fallback` emits an error, ensuring the historical phase always terminates within a bounded wall-clock window.

### Proof of Concept

**Preconditions:**
- Mirror node running with default config: `maxPageSize=1000`, `pollingFrequency=2s`, `timeout=60s`.
- Attacker has a Hedera account and gRPC access to the mirror node.

**Steps:**

1. Create a topic `0.0.X` on Hedera testnet.
2. Submit 2,000,000 messages to topic `0.0.X` (ensures ~2,000 consecutive full-page polls before exhaustion — ~66 minutes per subscription).
3. Open a gRPC connection to the mirror node and call `subscribeTopic` with:
   - `topicID = 0.0.X`
   - `consensusStartTime = 0` (retrieve from genesis)
   - No `limit` field set
4. Observe: the server issues a DB query every 2 s, each returning 1000 rows. `isComplete()` always returns `false`. The subscription never emits live messages and never completes.
5. Repeat step 3 across 5 connections × 5 streams = 25 concurrent stuck subscriptions.
6. **Result:** `boundedElastic` thread pool and DB connection pool are saturated. New legitimate `subscribeTopic` calls queue indefinitely or fail. Live gossip delivery is blocked for all subscribers sharing those resources.
