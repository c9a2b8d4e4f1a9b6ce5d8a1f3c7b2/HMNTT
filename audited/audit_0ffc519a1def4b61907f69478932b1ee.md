### Title
Thundering Herd DoS via Non-Atomic Cache Check-Then-Act in `updateOperationAccountAlias()` with No Application-Level Rate Limiting

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` calls `updateOperationAccountAlias()`, which uses a non-atomic check-then-act pattern against a shared LRU cache. Multiple concurrent unauthenticated requests targeting a block with many unique account IDs will all miss the cache simultaneously and each independently issue `GetAccountAlias()` database queries, multiplying DB load by the number of concurrent goroutines. The application applies no rate limiting by default, making this trivially exploitable.

### Finding Description

**Exact code path:**

In `rosetta/app/services/block_service.go` lines 104–130, `updateOperationAccountAlias()` iterates over every operation in every transaction:

```go
accountId := operations[index].AccountId
if cached, found = s.entityCache.Get(accountId.GetId()); !found {   // line 115
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)    // line 116
    ...
    s.entityCache.Set(result.GetId(), result)                        // line 121
}
```

The `Get` (line 115) and `Set` (line 121) are two separate, non-atomic operations on `github.com/Code-Hex/go-generics-cache`. There is no mutex, singleflight, or "get-or-load" primitive protecting the window between them.

**Root cause:** Classic cache stampede. When N concurrent goroutines all call `entityCache.Get()` for the same account ID before any of them has called `entityCache.Set()`, all N goroutines observe a cache miss and all N independently call `GetAccountAlias()` → `db.Raw(selectCryptoEntityWithAliasById, ...)` in `rosetta/app/persistence/account.go` line 117.

**No application-level rate limiting:** `main.go` lines 217–219 apply only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`. There is no throttle or concurrency limiter in the Go process. The Traefik middleware chain (`charts/hedera-mirror-rosetta/values.yaml` lines 149–166) that includes `inFlightReq: amount: 5` and `rateLimit: average: 10` is gated on `global.middleware` which defaults to `false` (line 95), so it is **disabled by default**.

**DB pool:** `rosetta/app/db/db.go` line 33 calls `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)`. If this is 0 (Go's default when unconfigured), the pool is unbounded and the DB server itself becomes the bottleneck. If it is a small positive number, the pool exhausts and requests queue or error.

### Impact Explanation
An attacker can exhaust the database connection pool or saturate the database server with `SELECT alias, id FROM entity WHERE id = @id` queries, causing legitimate requests to time out or fail with `ErrDatabaseError`. Because the Rosetta API is used by exchange integrations and indexers, sustained unavailability constitutes a high-impact denial of service against the mirror node's Rosetta interface. Severity: **High**.

### Likelihood Explanation
The endpoint requires no credentials. The attack requires only the ability to send HTTP POST requests to `/block`. A single attacker with a modest number of concurrent connections (e.g., 50–200) targeting a block that has many transactions with unique account IDs (common on mainnet) is sufficient to trigger the stampede. The attack is repeatable and requires no special knowledge beyond a valid block index. Likelihood: **High**.

### Recommendation
1. **Fix the cache stampede** using `golang.org/x/sync/singleflight`: wrap the `GetAccountAlias` call so that concurrent requests for the same account ID coalesce into a single in-flight DB call.
2. **Enable application-level rate limiting** unconditionally inside the Go process (e.g., `golang.org/x/time/rate` or a semaphore on concurrent `/block` handlers), rather than relying on an opt-in Traefik middleware.
3. **Set an explicit `MaxOpenConnections`** in the DB pool configuration to prevent unbounded connection growth.
4. **Set a finite `MaxTransactionsInBlock`** default and document its interaction with DB load.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, no Traefik middleware (`global.middleware: false`, the default), a block on the target network with ≥50 unique account IDs in its transactions.

**Steps:**
```bash
# 1. Identify a dense block (many unique accounts)
BLOCK_INDEX=<high-traffic block number>

# 2. Fire 100 concurrent POST /block requests
for i in $(seq 1 100); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$BLOCK_INDEX}}" &
done
wait
```

**Result:** Each of the 100 concurrent goroutines enters `updateOperationAccountAlias()`, all miss the cold LRU cache for the same set of account IDs, and each independently issues `SELECT alias, id FROM entity WHERE id = @id` for every unique account. With 50 unique accounts and 100 concurrent requests, this produces up to 5,000 simultaneous DB queries. The DB connection pool saturates; subsequent legitimate requests receive `ErrDatabaseError` or time out.
