### Title
Unauthenticated Connection Pool Exhaustion via Concurrent Token Allowance Requests

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` accepts requests from any unauthenticated user with no rate limiting. The REST API's database connection pool defaults to only 10 connections with a 20-second statement timeout, and the multi-union query structure in `TokenAllowanceService.getQuery()` can generate up to three expensive subqueries per request against a high-volume account. An attacker flooding this endpoint with concurrent maximum-limit requests can exhaust the connection pool, blocking all other API consumers.

### Finding Description

**Code path:**

`getAccountTokenAllowances()` at `rest/controllers/tokenAllowanceController.js` lines 68–81 calls `TokenAllowanceService.getAccountTokenAllowances(query)` with no concurrency guard or rate limit. [1](#0-0) 

`TokenAllowanceService.getQuery()` at `rest/service/tokenAllowanceService.js` lines 54–78 constructs up to **three** subqueries joined with `UNION ALL`, each carrying its own `LIMIT $2` clause, then wraps them in an outer `ORDER BY … LIMIT $2`. All three subqueries execute against the same `token_allowance` table for the target account. [2](#0-1) 

The database connection pool is configured at `rest/dbpool.js` lines 7–16 with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20,000 ms**). [3](#0-2) 

The default `maxConnections` is documented as **10** and `statementTimeout` as **20,000 ms**. [4](#0-3) 

**Root cause — failed assumption:** The design assumes the statement timeout (20 s) and pool size (10) are sufficient to absorb concurrent load. There is no per-IP or per-endpoint rate limiting anywhere in the REST service. [5](#0-4) 

**Limit cap — partial mitigation only:** `formatComparator` in `rest/utils.js` caps the `limit` parameter at `responseLimit.max` (default **100**) for unauthenticated users. [6](#0-5) 

While this prevents arbitrarily large single-query result sets, it does not prevent an attacker from sending `limit=100` (the allowed maximum) in many concurrent requests. Each such request acquires a pool connection and holds it for the duration of the multi-union query (up to 20 s on a large account). With only 10 pool connections, 10 concurrent requests fully exhaust the pool.

### Impact Explanation
When the pool is exhausted, every subsequent API request — across all endpoints, for all users — blocks waiting for a free connection or fails with a connection-timeout error (`connectionTimeoutMillis` default 20 s). This constitutes a full denial-of-service for the REST API. Severity is **High** because it requires no credentials, targets a shared resource (the connection pool), and affects all consumers of the service, not just the targeted account.

### Likelihood Explanation
The attack requires no authentication, no special knowledge beyond a valid account ID with many token allowances (publicly discoverable via the same API), and only ~10 concurrent HTTP requests sustained over time. Standard HTTP load tools (`ab`, `wrk`, `curl` in parallel) suffice. The attack is repeatable indefinitely because there is no IP-based blocking, no CAPTCHA, and no request quota. Likelihood is **High**.

### Recommendation
1. **Add rate limiting** at the HTTP layer (e.g., express-rate-limit or an API gateway) scoped per IP and/or per account ID path parameter, specifically for allowance endpoints.
2. **Increase the default pool size** or deploy a connection pooler (PgBouncer is already referenced in the Helm chart for production but not enforced for default deployments) in front of the REST service.
3. **Reduce the statement timeout** for read-only allowance queries to a value well below 20 s (e.g., 5 s) to release connections faster under attack.
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) inside the service layer to cap the number of simultaneous in-flight DB queries.

### Proof of Concept

**Preconditions:**
- A mirror node REST API instance running with default configuration (`maxConnections=10`, `statementTimeout=20000`).
- An account ID (e.g., `0.0.1000`) that has a large number of token allowances in the database.

**Steps:**
```bash
# Send 15 concurrent requests with maximum limit to the target account
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1000/allowances/tokens?limit=100" &
done
wait
```

**Trigger:** All 15 requests arrive simultaneously. The first 10 acquire pool connections and begin executing the multi-union query (which may run for several seconds on a large account). The remaining 5 — and any legitimate user requests — block waiting for a free connection.

**Result:** Legitimate API requests to any endpoint (transactions, balances, contracts, etc.) receive connection-timeout errors or hang for up to 20 seconds, degrading service availability for all users. The attack can be sustained indefinitely by repeating the burst.

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${TokenAllowance.OWNER} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...TokenAllowanceService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery([], params, accountIdCondition, limitClause, orderClause);
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }

const getEffectiveMaxLimit = () => {
  const userLimit = httpContext.get(userLimitLabel);
  return userLimit !== undefined ? userLimit : responseLimit.max;
};

/**
 * Gets the limit param value, if not exists, return the default; otherwise cap it at max.
 * Note if values is an array, the last one is honored.
 * @param {string[]|string} values Values of the limit param
 * @return {number}
 */
const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
