### Title
Unbounded In-Memory Transaction Accumulation in `FindBetween()` Enables Unauthenticated Memory Exhaustion DoS

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` accumulates all transaction batches into a single in-memory slice before any processing or capping occurs. The `maxTransactionsInBlock` guard in `block_service.go` is applied only after the full allocation, making it a response-size cap rather than a memory cap. An unauthenticated attacker can repeatedly or concurrently request any block containing many transactions to exhaust heap memory on the mirror node.

### Finding Description
**Exact code path:**

`POST /block` → `blockAPIService.Block()` (block_service.go:56) calls `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` → `transactionRepository.FindBetween()` (transaction.go:112–171).

Inside `FindBetween()`:
```go
// transaction.go:123
transactions := make([]*transaction, 0)
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error
    // ...
    transactions = append(transactions, transactionsBatch...)  // line 140: ALL batches accumulated
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
// sameHashMap and hashes built only AFTER full accumulation (lines 149–159)
```

The `maxTransactionsInBlock` check in `block_service.go:61–67` runs only after `FindBetween()` returns the fully-populated slice:
```go
// block_service.go:56–67
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
    return nil, err
}
// cap applied here — AFTER full heap allocation
if len(block.Transactions) > s.maxTransactionsInBlock {
    ...
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
```

**Root cause:** The design assumes blocks are small. There is no per-request transaction count limit enforced at the persistence layer. The response cap (`maxTransactionsInBlock`) is architecturally misplaced — it truncates the response slice but cannot reclaim the already-allocated heap.

**Why existing checks fail:**
- `maxTransactionsInBlock` (config `Response.MaxTransactionsInBlock`) is a response-level slice reslice, not a fetch limit.
- The middleware stack contains only metrics, health, and trace handlers — no rate limiting or concurrency throttle exists.
- The only input validation is `start > end` (line 116–118), which is irrelevant here since `start`/`end` come from the database-stored block record, not directly from the user.
- DB connection pool limits (`Pool.MaxOpenConnections`) bound DB concurrency but not Go heap allocation per goroutine.

### Impact Explanation
Each concurrent `/block` request for a block with N transactions allocates O(N) heap for `[]*transaction`, `sameHashMap` (map), and `hashes` (slice), plus all `constructTransaction` intermediate allocations. With no rate limiting and no per-request memory cap, C concurrent requests against a block with N transactions produce O(C×N) heap pressure. On a busy mainnet mirror node where blocks can contain tens of thousands of transactions, a modest number of concurrent requests (e.g., 50 concurrent requests × 50,000 transactions × ~1–2 KB per transaction struct) can exhaust available heap, triggering OOM or severe GC pressure, degrading or crashing the service for all users. This directly satisfies the "≥30% resource increase" threshold without brute force.

### Likelihood Explanation
The Rosetta API is a public, unauthenticated HTTP endpoint by design (Rosetta specification). No API key, token, or IP restriction is present in the codebase. An attacker needs only to:
1. Identify a block index with many transactions (trivially done by querying `/block` for recent high-activity blocks or by observing public Hedera network data).
2. Send concurrent POST requests to `/block` with that block identifier.

This requires no special privileges, no credentials, and no insider knowledge beyond a block index number. The attack is fully repeatable and scriptable.

### Recommendation
1. **Enforce a fetch limit at the persistence layer:** Add a configurable `maxFetchLimit` and break out of the batch loop once `len(transactions) >= maxFetchLimit`, returning an error or partial result before heap allocation grows unbounded.
2. **Move the cap before accumulation:** Pass `maxTransactionsInBlock` into `FindBetween()` so the DB query loop stops fetching once the limit is reached.
3. **Add HTTP concurrency/rate limiting middleware:** Introduce a per-IP or global in-flight request limit (e.g., `golang.org/x/time/rate` or a semaphore) in the router setup in `main.go`.
4. **Stream or paginate the response:** Instead of accumulating all batches, process and emit each batch incrementally.

func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}
