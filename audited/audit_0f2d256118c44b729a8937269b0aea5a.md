### Title
Missing `maxRstFramesPerWindow` Limit Enables HTTP/2 Rapid Reset DoS (CVE-2023-44487) Against Netty gRPC Server

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only `maxConcurrentCallsPerConnection(5)` and a custom executor, but sets no `maxRstFramesPerWindow` limit. An unprivileged attacker with network access to port 5600 can exploit the HTTP/2 Rapid Reset attack pattern (CVE-2023-44487) by rapidly opening streams and immediately sending RST_STREAM frames, saturating the Netty I/O event loop and rendering the server unresponsive without any authentication or prior relationship.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // No maxRstFramesPerWindow call — absent entirely
};
```

`NettyProperties` (lines 11–15) exposes only `maxConcurrentCallsPerConnection = 5`. No `maxRstFramesPerWindow` field exists anywhere in the codebase (confirmed: zero matches for `maxRstFrames`, `RST_STREAM`, `rapidReset` across all Java files).

**Root cause:** The server relies on `MAX_CONCURRENT_STREAMS = 5` as its sole HTTP/2 abuse mitigation. This setting is advertised to the client and controls how many streams may be *open simultaneously*, but it does not rate-limit RST_STREAM frames. The HTTP/2 Rapid Reset attack (CVE-2023-44487) exploits the fact that a client can send `HEADERS` (opening a stream) followed immediately by `RST_STREAM` (cancelling it) in a tight loop. Each such pair is processed by the Netty I/O thread synchronously. Because the stream is cancelled before it is counted as "concurrent," `MAX_CONCURRENT_STREAMS` provides no protection.

**Failed assumption:** The configuration assumes that limiting concurrent calls is sufficient to bound per-connection server-side work. It is not: RST_STREAM processing happens in the Netty I/O thread *before* the call is dispatched to the application executor, so the `applicationTaskExecutor` thread pool is irrelevant to this attack surface.

**No other mitigations present:** A search across all Java source confirms no `maxConnectionAge`, `keepAliveTime`, `permitKeepAlive`, `maxHeaderListSize`, or any HTTP/2 flow-control hardening is applied anywhere in the `grpc` module. The `web3` module has a `ThrottleManager`, but it is entirely absent from the gRPC service path.

### Impact Explanation
The Netty I/O event loop is single-threaded per channel. A sustained RST_STREAM flood from a single TCP connection can pin the event loop at 100% CPU processing frame callbacks, causing all other connections sharing that event loop to time out. With enough connections (each sending RST floods), all event loop threads can be exhausted, making the gRPC server completely unresponsive. This constitutes a full denial-of-service: topic subscription streams (`ConsensusService/subscribeTopic`) and network queries (`NetworkService`) both become unreachable, effectively partitioning the mirror node's gRPC interface from its consumers. No data is corrupted, but availability is fully lost for the duration of the attack.

**Severity: High** — unauthenticated, remotely exploitable, complete availability loss of the gRPC API surface.

### Likelihood Explanation
- **No authentication required:** The gRPC port (5600) accepts connections from any client; no TLS client certificate or API key is enforced.
- **Trivially scriptable:** Standard HTTP/2 libraries (e.g., `h2load`, `grpc-go`, Python `h2`) can generate thousands of HEADERS+RST_STREAM pairs per second from a single host.
- **CVE-2023-44487 is weaponized:** Public PoC tooling for this exact attack pattern has been available since October 2023 and was used in record-breaking DDoS campaigns against major cloud providers.
- **Repeatable:** The attack requires no state, no prior knowledge of topic IDs, and can be relaunched immediately after the server recovers.

### Recommendation
Add `maxRstFramesPerWindow` to the `NettyServerBuilder` configuration in `grpcServerConfigurer()`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // Mitigate CVE-2023-44487 HTTP/2 Rapid Reset
    serverBuilder.maxRstFramesPerWindow(
        nettyProperties.getMaxRstFramesPerWindow(),   // e.g. 200
        nettyProperties.getRstFramesWindowSeconds()   // e.g. 30
    );
};
```

Add the corresponding fields to `NettyProperties` with safe defaults (e.g., 200 RST frames per 30-second window, matching the grpc-java recommended defaults). Additionally consider:
- `serverBuilder.maxConnectionAge(Duration, TimeUnit)` to recycle long-lived connections
- `serverBuilder.maxConnectionIdle(Duration, TimeUnit)` to shed idle connections
- Infrastructure-level connection rate limiting at the ingress/GCP backend policy layer

### Proof of Concept
**Preconditions:** Network access to the gRPC port (5600). No credentials needed.

```bash
# Install h2load (part of nghttp2)
# Target: grpc server at <HOST>:5600

# Step 1: Confirm server is responsive
grpcurl -plaintext <HOST>:5600 list

# Step 2: Launch RST_STREAM flood
# Each request opens an HTTP/2 stream and immediately resets it
h2load -n 1000000 -c 10 -m 100 \
  --h2 \
  http://<HOST>:5600/com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
  &

# Step 3: Observe server becomes unresponsive to legitimate clients
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  <HOST>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected result: connection timeout / UNAVAILABLE after event loop saturation

# Step 4: Monitor server CPU — Netty I/O threads will be at 100%
# jstack <pid> will show all I/O threads stuck in RST_STREAM frame processing
```
