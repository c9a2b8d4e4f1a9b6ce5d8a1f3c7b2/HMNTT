### Title
Historical State Disclosure of Permanently Removed Contracts via Timestamp Parameter in `getContractById`

### Summary
The `getContractById` handler in `rest/controllers/contractController.js` accepts a user-supplied `timestamp` query parameter that, when present, causes the SQL query to UNION the `entity` table with the `entity_history` table. Neither the query builder nor the post-query handler applies any filter on `permanent_removal` or `deleted`, allowing an unprivileged attacker to retrieve the full historical state (including bytecode, admin key, file ID, etc.) of a contract that has been permanently removed from the ledger by supplying a timestamp that predates the deletion.

### Finding Description

**Code path:**

`getContractById` (line 707) accepts `filterKeys.TIMESTAMP` as the only accepted parameter: [1](#0-0) 

It extracts timestamp conditions and delegates to `getContractByIdOrAddressContractEntityQuery`: [2](#0-1) 

When `timestampConditions.length !== 0`, the query builder unconditionally UNIONs `entity` with `entity_history`: [3](#0-2) 

The per-table query builder's WHERE clause is only `e.type = 'CONTRACT'` plus the caller-supplied conditions — no `permanent_removal` or `deleted` predicate anywhere: [4](#0-3) 

After the query executes, the handler checks only that exactly one row was returned, then returns it directly — no post-query check on `permanent_removal` or `deleted`: [5](#0-4) 

**Root cause:** The design assumes that a permanently removed contract's current `entity` row (with `timestamp_range = [deletion_time, ∞)`) will not overlap a pre-deletion timestamp range, so the `entity` table leg of the UNION returns nothing. This is correct. However, the `entity_history` row for the same contract carries `timestamp_range = [creation_time, deletion_time)`, which *does* overlap a pre-deletion timestamp range. Because no `permanent_removal IS NOT TRUE` or `deleted IS NOT TRUE` guard is added to the `entity_history` leg of the UNION, that historical row is returned and served to the caller.

The `permanent_removal` flag is set at deletion time by the importer: [6](#0-5) 

The pre-deletion `entity_history` snapshot was written *before* `permanent_removal` was set, so that column is `NULL`/`false` in the history row — it passes any implicit "not permanently removed" assumption silently.

### Impact Explanation
An attacker can read the full historical entity record of any permanently removed contract: admin key, bytecode (via `file_id` or `initcode`), EVM address, memo, proxy account, obtainer, and all other fields included in `contractWithBytecodeSelectFields`. [7](#0-6) 

`permanent_removal` is a Hedera protocol-level guarantee that the contract and its history are expunged. Exposing that history via the mirror-node REST API violates that guarantee and may leak sensitive deployment artifacts (e.g., constructor arguments embedded in initcode, admin keys) that the contract owner intended to be irrecoverable.

### Likelihood Explanation
**Preconditions:** The attacker needs only the numeric contract ID (or EVM address) of a permanently removed contract and any timestamp that falls within the contract's active lifetime. Both are trivially discoverable: contract IDs are sequential and public; the creation timestamp is visible in the `timestamp.from` field of any prior API response or on-chain record. No authentication, special role, or rate-limit bypass is required. The endpoint is unauthenticated and publicly reachable.

**Repeatability:** The vulnerability is deterministic and repeatable for every permanently removed contract whose `entity_history` rows have not been pruned by the retention policy.

### Recommendation
Add an explicit `permanent_removal IS NOT TRUE` (and optionally `deleted IS NOT TRUE`) predicate to the WHERE clause inside `getContractByIdOrAddressQueryForTable`, or apply it as a post-query guard in `getContractById` before returning the row. The safest fix is at the query level so it applies to both the `entity` and `entity_history` legs of the UNION:

```js
// In getContractByIdOrAddressQueryForTable, change:
`where e.type = 'CONTRACT' and ${conditions.join(' and ')}`
// to:
`where e.type = 'CONTRACT' and (e.permanent_removal is not true) and ${conditions.join(' and ')}`
```

Alternatively, add a post-query check in `getContractById`:

```js
if (rows.length !== 1 || rows[0].permanent_removal === true) {
  throw new NotFoundError();
}
```

### Proof of Concept

1. Identify a contract that has been permanently removed (e.g., `0.0.1234`). Its current `entity` row has `permanent_removal = true`, `deleted = true`, and `timestamp_range = [T_delete, ∞)`.

2. Determine any timestamp `T_pre` such that `T_pre < T_delete` (e.g., the contract's `created_timestamp`).

3. Send the following unauthenticated HTTP request:
   ```
   GET /api/v1/contracts/0.0.1234?timestamp=lte:T_pre
   ```

4. The server builds:
   ```sql
   SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
   WHERE e.type = 'CONTRACT' AND e.timestamp_range && $1 AND e.id = $2
   UNION
   SELECT ... FROM entity_history e LEFT JOIN contract c ON e.id = c.id
   WHERE e.type = 'CONTRACT' AND e.timestamp_range && $1 AND e.id = $2
   ORDER BY timestamp_range DESC LIMIT 1
   ```
   where `$1 = Range(null, T_pre, '(]')` and `$2 = 1234`.

5. The `entity` leg returns 0 rows (no overlap). The `entity_history` leg returns the pre-deletion snapshot (overlap exists). The UNION returns 1 row.

6. `rows.length === 1` passes the only guard; the full historical contract record — including bytecode, admin key, and all metadata — is returned in the HTTP 200 response, despite the contract having been permanently removed.
