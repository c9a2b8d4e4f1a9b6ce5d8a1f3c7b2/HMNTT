### Title
Unauthenticated Connection Pool Exhaustion via Concurrent Requests to `/api/v1/network/registered-nodes`

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module is publicly accessible with no rate limiting, no in-flight request cap, and no per-IP throttling. Each request acquires a HikariCP database connection for the full duration of the native SQL query in `findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()`. An unauthenticated attacker sending enough concurrent requests can exhaust the shared connection pool, causing connection acquisition timeouts for every other endpoint that shares the same pool, effectively isolating the service from its database.

### Finding Description

**Code path:**

`NetworkController.getRegisteredNodes()` (lines 173–186) → `NetworkServiceImpl.getRegisteredNodes()` (lines 139–152) → `RegisteredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` (lines 14–22).

The repository executes a native SQL query:
```sql
select * from registered_node
where registered_node_id >= :lowerBound
  and registered_node_id <= :upperBound
  and deleted is false
  and (:type is null or type @> array[:type]::smallint[])
```

This is a synchronous, blocking JPA call. Spring Data JPA acquires a HikariCP connection from the shared pool before executing the query and holds it until the result set is fully fetched and the transaction completes.

**Root cause — no concurrency control in rest-java:**

- The `rest-java` module has zero rate limiting. The `ThrottleConfiguration` / `ThrottleManagerImpl` / `bucket4j` infrastructure exists exclusively in the `web3` module and is never applied to `rest-java` endpoints.
- The Traefik middleware configured for `rest-java` in `charts/hedera-mirror-rest-java/values.yaml` (lines 158–163) contains only `circuitBreaker` and `retry` — no `inFlightReq` and no `rateLimit`. By contrast, `hedera-mirror-rosetta/values.yaml` explicitly configures `inFlightReq: amount: 5` and `rateLimit: average: 10`.
- No Spring Security filter, servlet filter, or interceptor in `rest-java/src/main/java/org/hiero/mirror/restjava/config/` applies any concurrency or rate control to this endpoint.
- The `RegisteredNodesRequest` enforces only a `@Max(MAX_LIMIT)` on the result page size — this limits rows returned, not concurrent connections consumed.

**Pool exhaustion mechanics:**

HikariCP's default `maximumPoolSize` is 10 when not explicitly overridden. No `maximumPoolSize` or `connectionTimeout` override was found in any YAML configuration for `rest-java`. The documented `statementTimeout` is 10,000 ms (10 seconds), meaning each connection can be held for up to 10 seconds per query. With 10 concurrent requests in flight, the pool is fully saturated. Any subsequent request — to any endpoint sharing the same `DataSource` bean — blocks waiting for a connection and eventually throws `SQLTransientConnectionException` (HikariCP acquisition timeout, default 30 seconds), returning a 500 to the caller.

### Impact Explanation

All endpoints served by the `rest-java` process that share the same HikariCP `DataSource` — including `/api/v1/network/nodes`, `/api/v1/network/stake`, `/api/v1/accounts/{id}/allowances/nfts`, `/api/v1/topics/{id}`, etc. — become unavailable for the duration of the attack. This is a full application-layer denial of service against the database tier, achievable with a trivially small number of concurrent HTTP connections (≥10). No authentication, no special headers, and no exploit payload are required — only concurrent GET requests to a public endpoint.

### Likelihood Explanation

The attack requires no privileges, no account, and no knowledge beyond the public API documentation. Any attacker with a basic HTTP load tool (curl, wrk, ab, k6) can reproduce it. The attack is repeatable and sustainable: as long as the attacker maintains ≥10 concurrent open requests, the pool remains exhausted. The `circuitBreaker` middleware only trips on error ratios after the fact and does not prevent connection acquisition from being exhausted. The `retry` middleware actively worsens the situation by re-issuing failed requests.

### Recommendation

1. **Add `inFlightReq` middleware** to the `rest-java` Traefik middleware configuration (mirroring the `rosetta` chart), e.g., `inFlightReq: amount: 50` with `sourceCriterion.ipStrategy` to cap per-IP concurrency.
2. **Set an explicit `spring.datasource.hikari.maximum-pool-size`** appropriate for the expected concurrency, and tune `connection-timeout` to fail fast rather than queue indefinitely.
3. **Add a Tomcat/Jetty max-connections or max-threads cap** in the `rest-java` Spring Boot configuration to prevent the HTTP thread pool from being exhausted before the DB pool.
4. **Implement application-level rate limiting** (e.g., bucket4j, already used in `web3`) as a `OncePerRequestFilter` in `rest-java`, applied globally or per-endpoint.
5. **Set a PostgreSQL `statement_timeout`** at the session level for the `mirror_rest_java` role to bound the maximum connection hold time.

### Proof of Concept

```bash
# Exhaust the HikariCP pool (default size 10) with 20 concurrent requests
# No authentication required
seq 1 20 | xargs -P 20 -I{} \
  curl -s -o /dev/null \
  "https://<mirror-node-host>/api/v1/network/registered-nodes?limit=100" &

# Immediately probe another endpoint — expect 500 / connection timeout
curl -v "https://<mirror-node-host>/api/v1/network/stake"
# Expected: HTTP 500 with HikariCP "Connection is not available, request timed out"
```

Repeat the flood loop to sustain the outage. All endpoints sharing the `DataSource` will return 500 until the attacker stops or the pool recovers.
