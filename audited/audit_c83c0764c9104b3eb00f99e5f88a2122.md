### Title
`sync.Once` Bypass on DB Error Path in `initGenesisRecordFile` Enables Unbounded CTE Re-execution

### Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` only calls `br.once.Do()` on the success path. When the `selectGenesis` CTE query fails (DB error or "node is starting"), `genesisBlock.ConsensusStart` remains at `genesisConsensusStartUnset` (-1) and `sync.Once` is never triggered. Every subsequent request from any unprivileged caller re-executes the full multi-table CTE, creating DB load proportional to request rate for the entire duration of any DB instability window.

### Finding Description

**Exact code path:** `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263.

```
243: func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
244:     if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
245:         return nil                          // fast-path: only works after success
246:     }
...
252:     if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
253:         return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
254:         // ^^^ returns here — br.once.Do() is NEVER called, genesisBlock stays unset
255:     }
257:     br.once.Do(func() {                    // only reached on success
258:         br.genesisBlock = rb
259:     })
```

**Root cause:** `sync.Once` is placed after the error-return, so it only fires on a successful DB query. The fast-path guard on line 244 checks `br.genesisBlock.ConsensusStart`, which is only updated inside `br.once.Do()`. On any DB error, both the guard value and `sync.Once` remain in their initial state, so the next call unconditionally re-enters the full CTE execution path.

**`selectGenesis` CTE cost** (lines 49–72): The query joins `account_balance` (ordered, limited) with `record_file` via a `consensus_end > genesis.timestamp` predicate, plus a correlated subquery for `consensus_end` adjustment, ordered by `rf.consensus_end`. This is a non-trivial multi-table query that cannot be short-circuited.

**All five public block endpoints** (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) call `initGenesisRecordFile()` as their first action, so every block-related API request triggers this path.

**Exploit flow:**
1. DB enters a degraded state (slow queries, connection pool exhaustion, startup phase — the code explicitly handles `ErrNodeIsStarting`).
2. Attacker sends requests to any block endpoint (e.g., `POST /block`, `POST /block/transaction`).
3. Each request passes the line-244 guard (value is still -1), executes the full `selectGenesis` CTE, receives a DB error, and returns without updating state.
4. State is identical to before the request — the next request repeats identically.
5. DB load scales linearly with request rate for the entire duration of the instability window.

**Existing checks reviewed:**
- The Helm chart defines a `rateLimit` of 10 req/s and `inFlightReq` of 5 per IP, but `middleware: false` is the **default** (`charts/hedera-mirror-rosetta/values.yaml` line 95), meaning these protections are opt-in and absent in default deployments.
- The `sync.Once` field exists in the struct but provides zero protection for the error path.
- There is no exponential backoff, cooldown, or negative-cache on failed genesis initialization.

### Impact Explanation

During any DB instability window — including the normal "node is starting" phase that the code explicitly anticipates — every incoming block API request issues a full `selectGenesis` CTE against the DB. With no rate limiting in the default configuration, an attacker can sustain a request rate that keeps the DB under continuous query load, prolonging or deepening the instability. This directly maps to the stated scope: increasing DB/node resource consumption by >30% without brute force, since even a moderate steady request rate (tens of requests per second) continuously re-executes an expensive multi-table join for the entire instability window rather than the intended one-time execution.

### Likelihood Explanation

The Rosetta API is a public-facing HTTP service with no authentication requirement on block endpoints. The "node is starting" condition is a routine operational state that occurs on every restart. An attacker needs only to: (a) observe that the node is starting (detectable via `ErrNodeIsStarting` responses), and (b) maintain a steady stream of block API requests. No credentials, no special knowledge, and no brute-force volume are required. The attack is repeatable on every node restart.

### Recommendation

Move `br.once.Do()` to wrap the entire initialization attempt, including the DB query, so that either a successful result or a sentinel error is cached and the CTE is not re-executed on every request:

```go
var initErr *rTypes.Error
br.once.Do(func() {
    var rb recordBlock
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
        initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
        return
    }
    br.genesisBlock = rb
})
```

If retry-on-error behavior is required (e.g., to handle the startup phase), replace `sync.Once` with a mutex-protected flag and add an exponential backoff with a minimum retry interval (e.g., 5 seconds) so that DB load is bounded regardless of request rate.

### Proof of Concept

**Preconditions:** Rosetta node in online mode, DB in degraded/starting state (or simulate by temporarily making `account_balance` queries fail), default deployment (middleware disabled).

**Steps:**
1. Start the Rosetta node before the DB has genesis data populated (normal startup window).
2. Confirm the node returns `ErrNodeIsStarting` on block requests.
3. Send a steady stream of `POST /block` requests at 10–50 req/s from a single unprivileged client:
   ```
   while true; do
     curl -s -X POST http://<rosetta-host>/block \
       -H 'Content-Type: application/json' \
       -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"block_identifier":{"index":0}}' &
   done
   ```
4. Monitor DB query logs: each request issues a full `selectGenesis` CTE execution.
5. Observe DB CPU/query-rate metrics increase linearly with request rate, with no decay, for the entire duration of the instability window.
6. Compare DB load with and without the request stream to confirm >30% resource increase attributable solely to the re-executed CTE.
