### Title
Unauthenticated Thundering-Herd DoS via Absent Request Deduplication in `getAccountTokenAllowances()`

### Summary
`getAccountTokenAllowances()` in `rest/service/tokenAllowanceService.js` issues a raw database query on every invocation with no in-flight request coalescing or deduplication. The Redis response cache is **disabled by default** (`hiero.mirror.rest.cache.response.enabled = false`), and even when enabled the token-allowances endpoint carries only a 1-second `Cache-Control: public, max-age=1` TTL, leaving a permanent thundering-herd window. An unprivileged attacker can saturate the 10-connection database pool with a trivial flood of identical requests, causing connection-timeout cascades for all concurrent users.

### Finding Description

**Code path:**

`rest/service/tokenAllowanceService.js` lines 86–90:
```js
async getAccountTokenAllowances(query) {
  const {sqlQuery, params} = this.getQuery(query);
  const rows = await super.getRows(sqlQuery, params);   // direct DB call, no cache
  return rows.map((ta) => new TokenAllowance(ta));
}
```
`getQuery()` (lines 54–78) builds up to three `SELECT * FROM token_allowance … UNION ALL …` subqueries, each with its own `ORDER BY` and `LIMIT`, making it a non-trivial read.

**Root cause – no deduplication at any layer:**

1. *Service layer*: `getAccountTokenAllowances()` calls `super.getRows()` unconditionally; there is no in-flight promise sharing or memoisation.
2. *Response cache*: `rest/server.js` line 54 gates the Redis cache on `config.cache.response.enabled && config.redis.enabled`. Per `docs/configuration.md` line 549, `hiero.mirror.rest.cache.response.enabled` **defaults to `false`**, so the cache is off in every default deployment.
3. *Cache TTL when enabled*: `rest/__tests__/specs/accounts/{id}/allowances/responseHeaders.json` shows `"cache-control": "public, max-age=1"`. A 1-second TTL means every second's worth of concurrent requests all experience a simultaneous cache miss and race to the database (classic thundering herd / cache stampede).
4. *No application-level rate limiting*: `rest/server.js` registers no rate-limit middleware for this route. The Traefik `rateLimit` configuration visible in the repository applies only to the Rosetta service chart, not the REST API.
5. *DB pool ceiling*: `docs/configuration.md` line 556 shows `hiero.mirror.rest.db.pool.maxConnections` defaults to **10**. With a 20-second statement timeout (line 557), ten concurrent long-running queries fully occupy the pool for up to 20 seconds each.

**Failed assumption:** The design assumes either (a) the Redis response cache will absorb repeated identical requests, or (b) the database pool will act as a sufficient back-pressure mechanism. Neither holds: (a) the cache is off by default and has a 1-second TTL when on; (b) pool exhaustion is itself the denial-of-service condition, not a mitigation.

### Impact Explanation
With the pool at 10 connections and a 20-second statement timeout, an attacker sustaining ≥10 concurrent identical requests permanently occupies every pool slot. All other REST API endpoints share the same pool (`baseService.pool()` returns the global `pool`), so the entire mirror-node REST API becomes unresponsive. Legitimate clients receive connection-timeout errors (HTTP 503 / gateway timeout). Because the mirror node REST API is the primary read interface for wallets, explorers, and dApps querying Hedera state, sustained unavailability constitutes a significant service disruption.

### Likelihood Explanation
- **No authentication required**: the endpoint is fully public.
- **No rate limiting**: an attacker with a single machine and a basic HTTP flood tool (e.g., `ab`, `wrk`, `hey`) can sustain the required concurrency.
- **Stable target**: any well-known account ID (e.g., `0.0.2`, `0.0.98`) with token allowances provides a consistently expensive query.
- **Amplification**: because each request independently re-executes the multi-subquery SQL, the DB load scales linearly with concurrent request count, not with unique account IDs.
- **Repeatability**: the attack is stateless and trivially scriptable.

### Recommendation
1. **Enable and enforce the response cache** for this endpoint, or increase the `max-age` TTL beyond 1 second so the cache absorbs burst traffic.
2. **Add in-flight request coalescing** (e.g., a per-cache-key promise map) so that N concurrent identical requests share a single database round-trip.
3. **Add application-level rate limiting** (e.g., `express-rate-limit`) scoped per IP and/or per account-ID path parameter, independent of the infrastructure layer.
4. **Increase the DB pool size** or add a per-endpoint concurrency semaphore to bound the number of simultaneous queries for this endpoint.

### Proof of Concept
```bash
# Requires: wrk or ab; a running mirror-node REST instance; any account with token allowances

# 1. Confirm the endpoint is unauthenticated and returns data:
curl -s "http://<mirror-node>/api/v1/accounts/0.0.98/allowances/tokens"

# 2. Flood with 50 concurrent connections for 30 seconds (all identical URL):
wrk -t 10 -c 50 -d 30s \
  "http://<mirror-node>/api/v1/accounts/0.0.98/allowances/tokens"

# 3. In a separate terminal, observe that all other REST endpoints
#    (e.g., /api/v1/transactions) begin returning 503 or timing out
#    as the 10-connection DB pool is exhausted.

# Expected result:
# - DB pool saturated within seconds
# - All mirror-node REST endpoints return errors
# - Errors persist until the flood stops and in-flight queries drain
```
