### Title
Unbounded DB Query Queue via Unauthenticated Request Flood on `getAccountCryptoAllowances`

### Summary
`getAccountCryptoAllowances()` in `cryptoAllowanceService.js` delegates to `baseService.getRows()` which calls `pool().queryQuietly()` on a `pg` Pool. The `pg` Pool's internal pending-client queue has no size bound, and the REST server has no HTTP-level rate limiting or concurrency cap. An unauthenticated attacker can flood the endpoint, causing the pool's internal queue to grow without bound, exhausting Node.js heap memory and crashing the mirror node process.

### Finding Description

**Code path:**

`rest/service/cryptoAllowanceService.js` line 15 calls `super.getRows(query, params)`. [1](#0-0) 

`rest/service/baseService.js` line 56 calls `this.pool().queryQuietly(query, params)`, where `pool()` returns the global `pg` Pool. [2](#0-1) 

The pool is initialised in `rest/dbpool.js` with only two relevant guards: `max` (max open connections) and `connectionTimeoutMillis` (how long a caller waits for a free connection). [3](#0-2) 

`node-postgres` (`pg`) Pool queues every `query()` / `connect()` call that arrives while all `max` connections are busy. **This internal queue has no maximum length.** `connectionTimeoutMillis` only controls how long each queued waiter sits before it is rejected — it does not cap the number of waiters that can accumulate simultaneously.

`rest/server.js` registers no rate-limiting or concurrency-limiting middleware. The middleware chain is: `urlencoded → json → cors → compression → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler → route handlers → responseHandler → handleError`. No `express-rate-limit`, `bottleneck`, semaphore, or equivalent is present. [4](#0-3) 

**Root cause:** The failed assumption is that the `pg` Pool's `max` connection cap acts as backpressure on the caller. It does not — it only limits open DB connections. The pool's waiter queue is a plain in-memory array that grows with every new request that arrives while all connections are occupied.

**Exploit flow:**

1. Attacker sends a sustained burst of `GET /api/v1/accounts/{id}/allowances` requests (no authentication required).
2. The `pg` Pool exhausts its `max` connections immediately.
3. Every subsequent request appends a new Promise/callback entry to the pool's internal waiter queue.
4. Each queued entry retains: the HTTP `req`/`res` objects, the constructed SQL string, the bound parameters, and the Promise chain — all live on the Node.js heap.
5. Before `connectionTimeoutMillis` expires for the first batch, tens of thousands of entries accumulate.
6. Node.js heap is exhausted → process crashes with OOM or becomes unresponsive.

### Impact Explanation

A crash of the mirror node REST process means no new Hedera transactions can be confirmed via the mirror API, and downstream services (wallets, exchanges, dApps) that rely on the mirror node lose access. This maps directly to the stated critical scope: **total network-facing service shutdown**. Recovery requires a process restart; during the outage window all API consumers are affected.

### Likelihood Explanation

The endpoint is publicly reachable with no authentication. A single attacker with a modest botnet or even a single machine running an async HTTP flood (e.g., `wrk`, `hey`, or a simple Node.js script with hundreds of concurrent connections) can trigger this. The attack is repeatable and requires no special knowledge beyond the public API path. The absence of any rate-limiting middleware makes this trivially exploitable.

### Recommendation

1. **Add HTTP-level rate limiting** — apply `express-rate-limit` (or equivalent) globally in `server.js` before route handlers, keyed on IP address, with a burst cap and a per-second limit appropriate for the expected legitimate traffic.
2. **Bound the `pg` Pool waiter queue** — use a wrapper around `pool.connect()` / `pool.query()` that rejects immediately (HTTP 503) when the number of in-flight + queued requests exceeds a configured threshold, rather than queuing indefinitely.
3. **Set a short `connectionTimeoutMillis`** — reduce the default so queued waiters are evicted quickly, limiting the maximum queue depth to `arrival_rate × connectionTimeoutMillis`.
4. **Add a Node.js cluster / process-level concurrency limit** — use a semaphore (e.g., `async-sema`) in the request handler to cap simultaneous in-flight DB queries.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wg/wrk)
# Sends 1000 concurrent connections for 30 seconds to the allowances endpoint
wrk -t 10 -c 1000 -d 30s \
  "http://<mirror-node-host>:<port>/api/v1/accounts/0.0.1234/allowances"
```

Expected result: Node.js heap grows rapidly; the process either crashes with `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` or becomes unresponsive, returning no further responses. Monitor with:

```bash
watch -n1 'ps aux | grep node'
# or
node --max-old-space-size=512 ... # to accelerate OOM for testing
```
