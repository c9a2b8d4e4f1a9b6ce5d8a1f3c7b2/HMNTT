### Title
Unauthenticated GET /api/v1/topics/{id} Lacks Application-Level Rate Limiting, Enabling DB Connection Pool Exhaustion via Triple-Query Amplification

### Summary
The `TopicController.getTopic` handler issues three sequential, uncached database queries per request (via `TopicService`, `EntityService`, and `CustomFeeService`) with no application-level rate limiting in the `rest-java` module. An unprivileged attacker sending a sustained moderate-rate stream of requests can amplify DB connection pool pressure by a factor of 3x per request, saturating the pool and increasing node resource consumption well beyond 30% compared to baseline.

### Finding Description
**Code path:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 31–37: `getTopic` calls `topicService.findById`, `entityService.findById`, and `customFeeService.findById` sequentially.
- `TopicServiceImpl.java` line 20: `topicRepository.findById(id.getId())` — direct DB query, no cache.
- `EntityServiceImpl.java` line 25: `entityRepository.findById(id.getId())` — direct DB query, no cache.
- `CustomFeeServiceImpl.java` line 20: `customFeeRepository.findById(id.getId())` — direct DB query, no cache.

**Root cause:** The `rest-java` module has no application-level rate limiting. A search of all filters and configurations in `rest-java/src/main/java/org/hiero/mirror/restjava/config/` reveals only `LoggingFilter`, `MetricsFilter`, `RestJavaConfiguration` (which registers only an ETag filter and Protobuf converter), and `WebMvcConfiguration` (argument resolvers only). There is no `ThrottleManager`, no `Bucket4j` bean, and no `@RateLimiter` annotation anywhere in the `rest-java` service layer. By contrast, the `web3` module has an explicit `ThrottleConfiguration` with `rateLimitBucket`, `gasLimitBucket`, and `ThrottleManagerImpl` enforcing per-second limits.

**Failed assumption:** The design assumes infrastructure-level rate limiting (e.g., Traefik, as seen in `charts/hedera-mirror-rosetta/values.yaml` lines 157–160 for the Rosetta service) will protect all endpoints. However, this is not guaranteed for the `rest-java` service, and no such protection is enforced in application code.

**Exploit flow:**
1. Attacker sends GET `/api/v1/topics/0.0.1000` at ~50–100 req/s from one or more IPs — below typical brute-force thresholds.
2. Each request triggers 3 synchronous DB queries (topic, entity, custom fee) with no caching.
3. At 50 req/s, this generates 150 DB queries/second. Spring Boot's default HikariCP pool has 10 connections; each connection can handle ~10–20 simple queries/second, meaning the pool saturates rapidly.
4. Connection wait times spike, thread pool threads block waiting for DB connections, CPU and memory usage climb.

### Impact Explanation
DB connection pool saturation causes cascading degradation: all endpoints sharing the same pool (not just topics) experience increased latency and timeouts. At moderate sustained rates, this can increase CPU (thread contention, connection management), memory (queued requests), and DB server load by well over 30% compared to a 24-hour baseline. The `ShallowEtagHeaderFilter` registered in `RestJavaConfiguration` (line 42–46) only helps if the client sends `If-None-Match` headers — an attacker will not do this.

### Likelihood Explanation
No special privileges, credentials, or knowledge are required. The endpoint is fully public. The attack requires only a standard HTTP client (curl, wrk, ab). The 3-query amplification means the attacker achieves disproportionate DB load relative to request rate, making it effective at rates that evade simple volumetric detection. The attack is repeatable and can be distributed across multiple IPs or topic IDs.

### Recommendation
1. Add application-level rate limiting to the `rest-java` module mirroring the `web3` `ThrottleConfiguration` — introduce a `Bucket4j`-based filter or interceptor for `/api/v1/**` endpoints.
2. Add `@Cacheable` (with a short TTL, e.g., 5 seconds) to `TopicServiceImpl.findById`, `EntityServiceImpl.findById`, and `CustomFeeServiceImpl.findById` to reduce DB amplification per request.
3. Ensure infrastructure-level rate limiting (Traefik or equivalent) is explicitly configured and enforced for the `rest-java` service, not just Rosetta.
4. Consider consolidating the three queries into a single joined query or stored procedure to reduce per-request DB round-trips.
