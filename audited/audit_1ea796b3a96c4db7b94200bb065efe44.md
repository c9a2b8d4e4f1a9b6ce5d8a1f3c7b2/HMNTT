### Title
Unauthenticated EVM Address Lookup Causes DB Connection Pool Exhaustion (DoS)

### Summary
The `/api/v1/accounts/:idOrAliasOrEvmAddress/nfts` endpoint accepts EVM addresses from any unauthenticated caller and unconditionally issues a database query against the `entity` table for every request. Because the Node.js REST service has no application-level rate limiting and a default DB connection pool of only 10 connections, an attacker can saturate the pool with concurrent requests using non-existent EVM addresses, starving all other REST API consumers of database connections.

### Finding Description

**Exact code path:**

1. **Route registration** — `rest/routes/accountRoute.js` line 15:
   ```js
   router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
   ```
   No authentication or rate-limiting middleware is applied.

2. **Controller** — `rest/controllers/accountController.js` line 91:
   ```js
   const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
   ```
   The raw path parameter is passed directly to `getEncodedId`.

3. **Service dispatch** — `rest/service/entityService.js` lines 120–124:
   ```js
   if (EntityId.isValidEntityId(entityIdString)) {
     const entityId = EntityId.parseString(entityIdString, {paramName});
     return entityId.evmAddress === null
       ? entityId.getEncodedId()
       : await this.getEntityIdFromEvmAddress(entityId, requireResult);
   }
   ```
   Any 40-hex-char string (with or without `0x`) passes `isValidEntityId` and, because it is an opaque EVM address (non-zero prefix), `evmAddress` is non-null, so `getEntityIdFromEvmAddress` is always called.

4. **DB query** — `rest/service/entityService.js` lines 90–91:
   ```js
   async getEntityIdFromEvmAddress(entityId, requireResult = true) {
     const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
                   [Buffer.from(entityId.evmAddress, 'hex')]);
   ```
   The static query is:
   ```sql
   select id from entity where deleted <> true and evm_address = $1
   ```
   There is **no result cache** for this call path. `BaseService.getRows` (`rest/service/baseService.js` line 56) goes directly to the pool on every invocation.

5. **`entityId.js` LRU cache** (`rest/entityId.js` lines 301–332) only caches the **parsed `EntityId` JavaScript object** (string → struct), not the DB lookup result. The same EVM address string sent repeatedly still triggers a DB query on every request.

6. **Connection pool** — `rest/dbpool.js` line 14:
   ```js
   max: config.db.pool.maxConnections,
   ```
   Default is **10** (`docs/configuration.md` line 556). `connectionTimeout` is 20 000 ms and `statementTimeout` is 20 000 ms.

7. **No rate limiting** — `rest/server.js` lines 67–144 shows no rate-limiting middleware registered for the REST API. The `ThrottleConfiguration` (bucket4j) exists only in the `web3` Java module; the Helm-chart rate limit is scoped to `rosetta`.

**Root cause / failed assumption:** The system assumes that EVM-address lookups are infrequent enough that 10 pooled connections are sufficient. There is no per-IP throttle, no negative-result cache, and no circuit breaker on the Node.js REST path.

### Impact Explanation
All REST API endpoints share the single `global.pool` (`rest/dbpool.js` line 36). Exhausting its 10 connections blocks every endpoint — transactions, balances, contracts, topics — not just the NFT route. Legitimate clients receive connection-timeout errors (after up to 20 s of waiting), effectively taking the mirror-node REST API offline for the duration of the attack. Because the mirror node is the primary read path for Hedera network state (including gossip-derived transaction records), this constitutes a denial of service against transaction data availability.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and no special knowledge beyond the public API spec. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple async script) sending ≥10 concurrent requests to `/api/v1/accounts/0x<any_40_hex>/nfts` is sufficient to saturate the pool. The attack is trivially repeatable and sustainable indefinitely. The 20 s `statementTimeout` means each connection is held for up to 20 s per query on a slow DB, amplifying the effect.

### Recommendation
1. **Add a negative-result cache** in `getEntityIdFromEvmAddress`: cache `null` results for non-existent EVM addresses (e.g., using the existing `quick-lru` cache in `entityId.js` or a dedicated in-memory TTL cache) so repeated lookups for the same address do not hit the DB.
2. **Add application-level rate limiting** to the Express server (e.g., `express-rate-limit`) scoped per IP, applied before route handlers in `rest/server.js`.
3. **Increase the default connection pool size** or add a connection-queue depth limit so pool exhaustion does not cascade to unrelated endpoints.
4. **Add a circuit breaker** (e.g., `opossum`) around DB calls in `BaseService.getRows` to shed load when the pool is under pressure.

### Proof of Concept
```bash
# Saturate the 10-connection pool with concurrent requests using a non-existent EVM address
# Requires: wrk (https://github.com/wg/wrk)

wrk -t 10 -c 50 -d 60s \
  "http://<mirror-node-host>:5551/api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/nfts"

# While the above runs, observe that legitimate requests time out:
curl "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: connection timeout or 503 after ~20 s
```

Each of the 50 concurrent connections triggers `getEntityIdFromEvmAddress` → `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` with a non-existent address. With only 10 DB connections available, 40 requests queue immediately. New legitimate requests cannot acquire a connection within `connectionTimeout` (20 s) and fail.
