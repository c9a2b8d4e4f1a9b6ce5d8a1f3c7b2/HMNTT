### Title
Unbounded Hash String in `/block` Endpoint Enables Unauthenticated DoS via Memory Exhaustion and Database Overload

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` accepts an arbitrary-length hash string from an unauthenticated HTTP request, passes it through `SafeRemoveHexPrefix()` without any length validation, and forwards the full string as a PostgreSQL query parameter. Because no HTTP body size limit is enforced anywhere in the rosetta server stack, an attacker can send a multi-megabyte non-hex hash string that is fully buffered in Go heap memory and transmitted to the database on every request, enabling a low-effort denial-of-service.

### Finding Description

**Code path:**

1. `Block()` in `rosetta/app/services/block_service.go` (line 51) delegates to `RetrieveBlock()`: [1](#0-0) 

2. `RetrieveBlock()` in `rosetta/app/services/base_service.go` (lines 80–86) calls `SafeRemoveHexPrefix()` on the attacker-controlled hash and passes the result directly to `FindByIdentifier()` or `FindByHash()`: [2](#0-1) 

3. `SafeRemoveHexPrefix()` in `rosetta/app/tools/hex.go` performs only a prefix check — if the string does not start with `0x`, the entire string is returned unchanged, regardless of length: [3](#0-2) 

4. `FindByIdentifier()` in `rosetta/app/persistence/block.go` validates only that `index >= 0` and `hash != ""` — no length bound: [4](#0-3) 

5. `findBlockByHash()` passes the unbounded string directly to GORM as a named SQL parameter: [5](#0-4) 

**Root cause — failed assumption:** The code assumes the Rosetta SDK asserter or some upstream layer will reject oversized hashes. The asserter created in `main.go` only checks that the hash is non-empty and the index is non-negative; it imposes no length constraint. The `Http` config struct exposes only timeout fields (`IdleTimeout`, `ReadTimeout`, `ReadHeaderTimeout`, `WriteTimeout`) and no body-size limit: [6](#0-5) 

No `http.MaxBytesReader` or equivalent is applied anywhere in the rosetta server setup: [7](#0-6) 

### Impact Explanation
Each request with a 10 MB hash string causes the Go runtime to allocate a 10 MB string on the heap, copy it through the call stack, and transmit it to PostgreSQL as a query parameter in `where hash = @hash collate "C"`. A sustained flood of such requests (e.g., 100 concurrent connections each sending 10 MB bodies) can exhaust Go heap memory, saturate the database connection pool, and degrade or crash the mirror-node rosetta service. Because the SQL query is parameterized, there is no SQL injection risk, but the resource exhaustion impact is real and severe. The service is publicly exposed (no authentication on `/block`).

### Likelihood Explanation
No authentication or API key is required. The attack requires only a standard HTTP client capable of sending a POST request with a large JSON body. It is trivially scriptable, repeatable, and requires no special knowledge of the system. The absence of any body-size guard makes this a reliable, low-cost DoS primitive.

### Recommendation
1. **Enforce an HTTP body size limit** at the server level using `http.MaxBytesReader` in the request handler or by setting `http.Server.MaxHeaderBytes` and wrapping the body reader. A limit of 1–4 KB is sufficient for all valid Rosetta `/block` requests.
2. **Validate hash length** before calling `SafeRemoveHexPrefix`. A valid SHA-384 block hash is 96 hex characters (48 bytes); reject any hash exceeding, e.g., 128 characters with `ErrInvalidArgument`.
3. Add the length check in `FindByIdentifier` and `FindByHash` in `rosetta/app/persistence/block.go` as a defense-in-depth measure alongside the HTTP-layer fix.

### Proof of Concept
```bash
# Generate a 10 MB non-hex hash string
LARGE_HASH=$(python3 -c "print('z' * 10_000_000)")

curl -s -X POST http://<rosetta-host>:5700/block \
  -H "Content-Type: application/json" \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
    \"block_identifier\": {\"index\": 1, \"hash\": \"${LARGE_HASH}\"}
  }"
```

Repeat with 50–100 concurrent processes. Observe Go heap growth (via `/metrics` or `pprof`) and PostgreSQL query load. The server will either OOM or become unresponsive under sustained load. Each request is processed without rejection until the database returns `ErrBlockNotFound`.
