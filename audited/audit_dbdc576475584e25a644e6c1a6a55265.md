### Title
Unauthenticated Chained DB Query Amplification via `/contracts/results/:transactionIdOrHash/actions` with No REST API Rate Limiting

### Summary
The `getContractActions()` handler in `rest/controllers/contractController.js` executes two sequential database queries per request when a valid Ethereum hash is supplied: first `ethereumTransactionsByHashQuery` against `contract_transaction_hash`, then `contractActionsByConsensusTimestampQuery` against `contract_action`. The REST API (Node.js/Express) has zero application-level rate limiting, and valid Ethereum transaction hashes are publicly obtainable from the Hedera network, allowing any unauthenticated attacker to sustain high-volume two-query-per-request load against the database.

### Finding Description

**Exact code path:**

`getContractActions()` in `rest/controllers/contractController.js` lines 1248–1269:

```
if (utils.isValidEthHash(transactionIdOrHash)) {
  tx = await ContractService.getContractTransactionDetailsByHash(utils.parseHexStr(transactionIdOrHash));
  // → Query 1: ethereumTransactionsByHashQuery
}
...
const rows = await ContractService.getContractActionsByConsensusTimestamp(
  consensusTimestamp, payerAccountId, filters, order, limit
);
// → Query 2: contractActionsByConsensusTimestampQuery
```

**Query 1** (`contractService.js` lines 208–212):
```sql
select * from contract_transaction_hash
where hash = $1
order by (transaction_result = <success>) desc, consensus_timestamp desc
limit 1
```
The `ORDER BY` on a computed boolean expression `(transaction_result = <success>)` requires evaluating and sorting all matching rows before applying `LIMIT 1`. The hash index (`contract_transaction_hash__hash` using PostgreSQL hash index) handles the equality lookup, but sorting is done in-memory over the result set.

**Query 2** (`contractService.js` lines 153–172):
```sql
select ... from contract_action
where consensus_timestamp = $1 and payer_account_id = $2
order by index [asc|desc]
limit <N>
```

**Root cause — no rate limiting in the REST API:**

`rest/server.js` registers no rate-limiting middleware. The full middleware stack is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler` (only enforces limits for authenticated users with `user.limit`), optional `metricsHandler`, optional `responseCacheCheckHandler`, then routes. There is no `express-rate-limit`, no token bucket, no per-IP throttle.

The `ThrottleManager`/`ThrottleConfiguration` (Java, `web3/src/main/java/...`) applies only to the web3 Spring Boot service's `POST /contracts/call` endpoint — it is entirely separate from the REST API.

**Why the 404 short-circuit does not protect:**

If `tx.length === 0` (hash not found), Query 2 is skipped and a `NotFoundError` is thrown. However, valid Ethereum transaction hashes are publicly visible on Hedera explorers and block explorers. An attacker can collect a list of real hashes and replay them indefinitely, ensuring both queries execute every time.

### Impact Explanation

Each request with a valid hash causes two indexed DB queries. With no rate limiting, an attacker can sustain thousands of requests per second from a single machine or a small botnet. The cumulative DB connection pool pressure, query execution overhead, and I/O on `contract_transaction_hash` and `contract_action` tables can degrade database throughput for all other mirror node consumers. The 30% resource consumption threshold is reachable without brute force — it requires only a sustained stream of requests using known-valid hashes, which are freely available.

### Likelihood Explanation

- No authentication or API key required.
- Valid hashes are publicly available (Hedera mainnet/testnet explorers).
- A single attacker with a modest connection rate (e.g., 500–1000 req/s) can sustain the load indefinitely.
- The attack is trivially scriptable (`curl` loop or `ab`/`wrk` with a list of valid hashes).
- No special knowledge of the system internals is needed beyond knowing the endpoint exists (documented in `openapi.yml`).

### Recommendation

1. **Add rate limiting to the REST API**: Introduce `express-rate-limit` (or equivalent) as middleware in `rest/server.js`, applied globally or specifically to the `/contracts/results/:transactionIdOrHash/actions` route. A per-IP limit of e.g. 60 req/min is a reasonable starting point.
2. **Add a DB query statement timeout**: Ensure `pool.query` calls are bounded by a statement timeout (e.g., `SET statement_timeout = 5000`) so runaway queries cannot hold connections indefinitely.
3. **Consider caching**: Cache the result of `getContractTransactionDetailsByHash` by hash (short TTL, e.g., 30s) to avoid repeated DB hits for the same hash.
4. **Infrastructure-level protection**: Apply Traefik `rateLimit` middleware to the REST API ingress (similar to what is configured for the Rosetta service in `charts/hedera-mirror-rosetta/values.yaml`).

### Proof of Concept

```bash
# 1. Collect valid Ethereum hashes from a public Hedera explorer or the mirror node itself:
#    GET /api/v1/contracts/results?limit=100 → extract .results[].hash values

HASHES=("0xabc123...64hexchars" "0xdef456...64hexchars" ...)  # real hashes from step 1

# 2. Flood the actions endpoint with valid hashes (no auth required):
while true; do
  for hash in "${HASHES[@]}"; do
    curl -s "https://<mirror-node>/api/v1/contracts/results/${hash}/actions" &
  done
  wait
done
