### Title
Missing Gas-Based Throttling in `throttleOpcodeRequest()` Enables Disproportionate Resource Consumption via EVM Replay

### Summary
`throttleOpcodeRequest()` in `ThrottleManagerImpl` only enforces a simple rate limit (default 1 req/sec via `opcodeRateLimitBucket`) with no gas-based throttling, unlike the regular `throttle()` path which checks both rate and gas consumption. An unauthenticated attacker can target any existing high-gas transaction on the network and trigger a full EVM re-execution with complete opcode tracing (stack, memory, storage per opcode), generating unbounded opcode trace data and consuming disproportionate CPU and memory from a single request.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (line 61) calls `throttleManager.throttleOpcodeRequest()` before dispatching to `opcodeService.processOpcodeCall()`. [1](#0-0) 

`throttleOpcodeRequest()` in `ThrottleManagerImpl` only consumes one token from `opcodeRateLimitBucket`: [2](#0-1) 

By contrast, the regular `throttle()` path checks **both** `rateLimitBucket` AND `gasLimitBucket`: [3](#0-2) 

The default `opcodeRequestsPerSecond` is 1: [4](#0-3) 

After passing the single rate check, `OpcodeServiceImpl.processOpcodeCall()` builds `ContractDebugParameters` using the **original transaction's gas limit** from the stored `ContractResult` or `EthereumTransaction` — not from the incoming HTTP request — and then triggers a full EVM re-execution: [5](#0-4) [6](#0-5) 

During re-execution, `OpcodeActionTracer.tracePostExecution()` creates and accumulates an `Opcode` object for **every single EVM instruction**, optionally capturing full stack, memory, and storage state at each step: [7](#0-6) 

**Root cause:** The failed assumption is that a 1 req/sec rate limit is sufficient to bound resource consumption for opcode trace requests. It is not, because the cost of a single request scales with the gas used by the targeted transaction and the number of EVM instructions executed, neither of which is bounded by the opcode throttle path.

**Why existing checks fail:**
- `opcodeRateLimitBucket` (1 req/sec) only limits request frequency, not per-request cost.
- `gasLimitBucket` is never consulted for opcode requests, so a replay of a 15M-gas transaction with tight loops bypasses the gas-per-second throttle entirely.
- No maximum opcode count limit is enforced in the tracer accumulation path visible in the code.
- No authentication or authorization is required to call the endpoint. [8](#0-7) 

### Impact Explanation
A single opcode trace request targeting a high-gas transaction (e.g., 15M gas with tight loops or deep recursion) with `stack=true&memory=true&storage=true` causes the node to: (1) re-execute the entire EVM transaction, (2) allocate and populate an `Opcode` object for every instruction (potentially millions), each containing full stack/memory/storage snapshots. This can exhaust heap memory and saturate CPU on the mirror node, causing GC pressure, thread starvation, or OOM — all from a single unauthenticated HTTP request. The endpoint is publicly accessible with no authentication gate.

### Likelihood Explanation
Any external user can call `GET /api/v1/contracts/results/{hash}/opcodes?stack=true&memory=true&storage=true`. The attacker only needs to identify one existing high-gas transaction on the network (trivially done by querying the mirror node's own REST API for contract results with high gas usage). No privileged access, no special tooling, and no brute force is required. The attack is repeatable at 1 req/sec indefinitely.

### Recommendation
1. **Apply gas-based throttling to opcode requests**: Consume from `gasLimitBucket` proportional to the gas limit of the replayed transaction inside `throttleOpcodeRequest()` or in `OpcodeServiceImpl` before dispatching.
2. **Cap opcode collection**: Enforce a hard maximum on the number of `Opcode` objects accumulated per trace (e.g., configurable via `OpcodesProperties`), returning a truncated response with a warning header when exceeded.
3. **Bound memory/storage capture**: When `memory=true` or `storage=true`, apply per-opcode size limits on captured data.
4. **Set execution timeouts**: Apply a wall-clock timeout to the EVM re-execution to prevent a single request from holding a thread indefinitely.

### Proof of Concept

**Preconditions:**
- Mirror node is running with `hiero.mirror.web3.opcode.tracer.enabled=true`
- A high-gas transaction exists on the network (find via `GET /api/v1/contracts/results?order=desc&limit=25` and pick one with high `gas_used`)

**Steps:**
```bash
# 1. Find a high-gas transaction hash
TX_HASH=$(curl -s "https://<mirror-node>/api/v1/contracts/results?order=desc&limit=1" \
  | jq -r '.results[0].hash')

# 2. Send opcode trace request with all capture options enabled
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
  --output trace.json.gz

# 3. Observe: node CPU spikes to near 100%, heap usage surges
# 4. Repeat at 1 req/sec to sustain elevated resource consumption
```

**Expected result:** The node re-executes the full transaction with per-opcode stack/memory/storage capture, generating millions of `Opcode` objects in memory, causing sustained CPU and memory pressure exceeding 30% above baseline from a single unauthenticated request.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
