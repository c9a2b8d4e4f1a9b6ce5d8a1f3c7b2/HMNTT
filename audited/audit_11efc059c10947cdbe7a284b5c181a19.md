### Title
Unprivileged Timestamp Parameter Bypasses `deleted=false` Filter, Exposing Deleted Hook Storage Entries

### Summary
Any unauthenticated caller can supply a `timestamp` query parameter to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` to force the service into the historical code path (`getHookStorageChange`). That path queries `hook_storage_change` without a `deleted = false` predicate, unlike the non-historical path which enforces it. This lets an attacker enumerate storage slot keys the owner deleted and, by choosing a timestamp before the deletion, read the pre-deletion values.

### Finding Description

**Routing gate — `HookStorageRequest.isHistorical()`**

`HookStorageRequest.isHistorical()` returns `true` whenever the `timestamp` `Bound` is non-empty: [1](#0-0) 

`Bound.isEmpty()` is `true` only when both `lower` and `upper` are `null`: [2](#0-1) 

Any caller can set this by passing `?timestamp=lte:99999999999999999`. The controller accepts up to two timestamp parameters with no authentication: [3](#0-2) 

**Branch divergence in `HookServiceImpl.getHookStorage()`** [4](#0-3) 

Non-historical path uses Spring Data derived queries that embed `DeletedIsFalse`: [5](#0-4) 

Historical path (`getHookStorageChange`, lines 84–118) calls `HookStorageChangeRepository` queries that compute `deleted` as a column alias but **never filter on it**: [6](#0-5) 

The second query (key-in variant) has the same omission: [7](#0-6) 

`deleted` is defined as `(value_written is null or length(value_written) = 0)`, matching the domain model: [8](#0-7) 

`distinct on (key)` ordered by `consensus_timestamp DESC` (from the pageable built in `getPageRequest`) returns the **most recent** change per key within the timestamp window. If that most recent change is a deletion, the deleted record is returned. If the attacker picks a timestamp window that ends before the deletion, the last written (non-empty) value is returned.

### Impact Explanation
An unprivileged external user can:
1. Enumerate every storage slot key that the hook owner deleted — information the `deleted = false` filter was designed to hide.
2. Read the pre-deletion value of any slot by choosing a timestamp upper-bound that falls before the deletion consensus timestamp.

This constitutes unauthorized read access to data the owner explicitly cleared, violating the confidentiality guarantee of the hook storage API. Severity is **High** because no credentials are required, the endpoint is publicly reachable, and the full historical value of any slot is recoverable.

### Likelihood Explanation
The exploit requires only a standard HTTP GET with one extra query parameter. No special account, key, or on-chain transaction is needed. Any observer who knows (or can guess) a valid `ownerId`/`hookId` pair can perform it repeatedly and programmatically. The timestamp range `lte:99999999999999999` covers all history, making enumeration trivial.

### Recommendation
Add `AND (value_written IS NOT NULL AND length(value_written) > 0)` (i.e., `AND deleted = false`) to both native queries in `HookStorageChangeRepository`:

```sql
-- add to both findByKeyBetweenAndTimestampBetween and findByKeyInAndTimestampBetween
AND (value_written IS NOT NULL AND length(value_written) > 0)
```

Alternatively, filter in `getHookStorageChange` after the repository call, or introduce a dedicated `findBy…DeletedIsFalse` variant for the historical repository. The fix must be applied to **both** query methods.

### Proof of Concept

**Preconditions:**
- A hook with `ownerId=1000`, `hookId=1` exists.
- Storage slot `key=0xABCD…` was written at timestamp `T1`, then deleted (cleared) at timestamp `T2 > T1`.
- The non-historical endpoint correctly hides the slot: `GET /api/v1/accounts/1000/hooks/1/storage` returns no entry for that key.

**Step 1 — Confirm deletion is hidden without timestamp:**
```
GET /api/v1/accounts/1000/hooks/1/storage
→ slot 0xABCD… absent (deleted=false filter active)
```

**Step 2 — Trigger historical path to expose deleted slot:**
```
GET /api/v1/accounts/1000/hooks/1/storage?timestamp=lte:99999999999999999
→ slot 0xABCD… returned with deleted=true, value=null
```

**Step 3 — Read pre-deletion value by choosing timestamp before T2:**
```
GET /api/v1/accounts/1000/hooks/1/storage?timestamp=lte:<T2-1>
→ slot 0xABCD… returned with deleted=false, value=<original written value>
```

No authentication, no signer approval, no privileged account required.
