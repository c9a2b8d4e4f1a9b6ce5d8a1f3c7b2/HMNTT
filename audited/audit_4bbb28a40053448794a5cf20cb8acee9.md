### Title
Unauthenticated Historical Block Call Flood Exhausts Global Rate Limit and Bypasses State Cache, Enabling DoS

### Summary
The `call()` endpoint in `ContractController` accepts historical block calls from any unauthenticated user with no per-source throttle. Historical state queries unconditionally bypass the contract-slot cache and hit the database directly, while the global rate-limit bucket (500 req/s, shared across all callers) can be fully monopolized by a single attacker. An attacker sending 500 req/s with a distinct block number per request forces continuous DB queries for both record-file lookups and historical state, starving legitimate users of their rate-limit allocation and saturating the database.

### Finding Description

**Entry point** — `ContractController.call()` (lines 38–51):

```java
ContractCallResponse call(@RequestBody @Valid ContractCallRequest request, ...) {
    throttleManager.throttle(request);          // global bucket only
    validateContractMaxGasLimit(request);
    final var params = constructServiceParameters(request);
    final var result = contractExecutionService.processCall(params);
    ...
}
```

The `block` field is taken directly from the request and forwarded as-is; no authentication or per-caller quota is applied.

**Throttle is global and gas-cost is trivial** — `ThrottleManagerImpl.throttle()` (lines 37–48) and `ThrottleProperties` (line 35):

- `rateLimitBucket`: 500 tokens/second, **shared across all callers**. One attacker can consume every token.
- `gasLimitBucket`: capacity = `scaleGas(7_500_000_000) = 750,000` tokens/second. Minimum request gas is `@Min(21_000)` (`ContractCallRequest` line 36), so `scaleGas(21_000) = 2` tokens per request. At 500 req/s the attacker consumes only 1,000 gas tokens/second against a 750,000-token bucket — the gas throttle is not a binding constraint.

**Record-file cache thrashing** — `RecordFileRepository.findByIndex()` (line 27–29) is `@Cacheable` with `CACHE_MANAGER_RECORD_FILE_INDEX`. Each unique block number causes a cache miss and a DB query. An attacker cycling through distinct block numbers at 500 req/s continuously populates the bounded cache with new entries, evicting legitimate entries and forcing repeated DB round-trips.

**Historical state is never cached** — `ContractStateServiceImpl.findStorageByBlockTimestamp()` (lines 73–76):

```java
public Optional<byte[]> findStorageByBlockTimestamp(
        final EntityId entityId, final byte[] slotKeyByteArray, final long blockTimestamp) {
    return contractStateRepository.findStorageByBlockTimestamp(entityId.getId(), slotKeyByteArray, blockTimestamp);
}
```

There is no cache layer here. Every historical contract-storage read goes directly to the database, regardless of how many times the same (block, slot) pair has been queried. The batch-slot cache (`findStorage` / `findStorageBatch`) is only used for non-historical calls.

**Per-request context is not shared** — `ContractCallContext` uses `ScopedValue` (line 25), so any in-flight read cache is discarded after each request; nothing is reused across requests.

### Impact Explanation

A single unauthenticated attacker sending 500 req/s (the global ceiling) with a unique block number per request:

1. Monopolises the entire `rateLimitBucket`, returning HTTP 429 to all legitimate callers.
2. Forces up to 500 DB queries/second for `record_file` index lookups (cache misses on unique block numbers).
3. Forces an unbounded number of `contract_state_change` historical queries per second (no cache at all), proportional to the complexity of the called contract.

The result is effective denial of service for the historical Hashgraph data access path, with no special privileges required.

### Likelihood Explanation

The attack requires only an HTTP client and knowledge of the public API. No account, token, or privileged credential is needed. The endpoint is documented and publicly reachable. Cycling block numbers is trivial (increment a counter). The attack is repeatable indefinitely and can be automated with a single script.

### Recommendation

1. **Per-IP / per-caller rate limiting**: Introduce a per-source-address bucket (e.g., via a servlet filter or API gateway) so one caller cannot exhaust the global quota.
2. **Historical-call-specific throttle**: Add a separate, lower rate limit for requests where `block != LATEST`, reflecting their higher DB cost.
3. **Cache historical record-file results**: The `@Cacheable` on `findByIndex` helps for repeated block numbers, but the cache size should be tuned and an LRU eviction policy enforced to resist cache-fill attacks.
4. **Cache historical state reads**: Introduce a short-lived (e.g., 1–5 second TTL) read-through cache keyed on `(contractId, slotKey, blockTimestamp)` in `findStorageByBlockTimestamp` to absorb repeated identical queries.
5. **Minimum gas enforcement at throttle layer**: Ensure `scaleGas` never returns 0 for valid requests so the gas bucket provides meaningful back-pressure.

### Proof of Concept

```bash
# Attacker script: 500 req/s, each with a unique historical block number
BLOCK=1
while true; do
  for i in $(seq 1 500); do
    curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d "{\"block\":\"$BLOCK\",\"to\":\"0x0000000000000000000000000000000000000167\",
           \"data\":\"0x\",\"gas\":21000}" &
    BLOCK=$((BLOCK + 1))
  done
  wait
  sleep 1
done
```

**Preconditions**: No account or authentication required.  
**Trigger**: Each request uses a distinct block number, causing a record-file cache miss and an uncached historical state DB query.  
**Result**: The global `rateLimitBucket` is exhausted; legitimate callers receive HTTP 429; the database sustains elevated query load from both `findByIndex` and `findStorageByBlockTimestamp` calls.
