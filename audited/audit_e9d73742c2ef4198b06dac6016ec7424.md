### Title
Mirror Node Token Name Inconsistency via Null Character Sanitization in `setName()`

### Summary
Any unprivileged user can submit a `TokenCreate` or `TokenUpdate` transaction with a token `name` containing the null character (`\u0000`). The Hedera consensus layer records the original value, but the mirror node's `AbstractToken.setName()` silently replaces every null character with the Unicode replacement character `\uFFFD` (`'?'`) before persisting to PostgreSQL. This causes the mirror node to permanently export a `name` that differs from the value recorded on-chain.

### Finding Description

**Exact code path:**

`AbstractToken.setName()` at [1](#0-0)  unconditionally calls `DomainUtils.sanitize(name)`.

`DomainUtils.sanitize()` at [2](#0-1)  performs a one-way replacement: every `\u0000` (null byte) is replaced with `\uFFFD` (the Unicode replacement character). The original value is discarded.

The same transformation applies to `setSymbol()`: [3](#0-2) 

The importer's `TokenCreateTransactionHandler` and `TokenUpdateTransactionHandler` both call `token.setName(...)` with the raw protobuf string from the record stream, which flows directly into this sanitization. The sanitized (mutated) value is then persisted to the `token.name` column (defined as `character varying(100)` in the DB schema): [4](#0-3) 

The project's own test explicitly confirms the mutation behavior: [5](#0-4) 

And the repository integration test confirms the mutated value is what gets stored and retrieved: [6](#0-5) 

**Root cause / failed assumption:** The design assumes that replacing `\u0000` with `\uFFFD` is a safe, transparent normalization. It is not — `\uFFFD` is a distinct, visible Unicode character. The consensus record contains `\u0000`; the mirror node stores and exports `\uFFFD`. These are different byte sequences and different displayed strings.

**Exploit flow:**
1. Attacker calls `TokenCreate` with `name = "LEGIT\u0000TOKEN"` (a valid protobuf string; the Hedera network validates only length ≤ 100 chars and non-empty — no null-byte rejection at consensus).
2. Consensus records `"LEGIT\u0000TOKEN"` in the record stream.
3. Mirror node importer reads the record, calls `token.setName("LEGIT\u0000TOKEN")`.
4. `DomainUtils.sanitize()` returns `"LEGIT\uFFFDTOKEN"`.
5. Mirror node persists and serves `"LEGIT\uFFFDTOKEN"` via REST API.
6. On-chain record: `"LEGIT\u0000TOKEN"`. Mirror node record: `"LEGIT\uFFFDTOKEN"`. Permanent, irreversible divergence.

### Impact Explanation
Any consumer of the mirror node REST API (wallets, explorers, DeFi integrations) receives a token name that was never submitted on-chain. The `\uFFFD` replacement character is visually rendered as `?` or `?` in most UIs, making the displayed name visibly wrong. Applications that perform exact-match comparisons between the consensus record and the mirror node record (e.g., audit tools, compliance checks, cross-chain bridges) will detect a mismatch and may incorrectly flag the token or reject it. The inconsistency is permanent — there is no recovery path once the sanitized value is written. Severity: **Medium** (data integrity violation, no fund loss, but verifiable record falsification).

### Likelihood Explanation
The precondition is trivially met: any account with enough HBAR to pay the token creation fee (~$1–2 USD on mainnet) can trigger this. No admin key, no special permission, no privileged role is required. The null character is a valid protobuf string byte; the Hedera SDK does not strip it before signing. The attack is repeatable on every `TokenCreate` and `TokenUpdate` transaction. It can be automated.

### Recommendation
Preserve the original value in the database and handle the PostgreSQL null-byte constraint at the serialization layer rather than silently mutating the domain object. Options:
1. **Store as bytes / escape**: Store the raw UTF-8 bytes in a `bytea` column, or use PostgreSQL's `encode`/`escape` to represent null bytes without loss.
2. **Reject at import**: Log a warning and skip persisting the name mutation — store the field as `NULL` or a sentinel — so downstream consumers know the value is unavailable rather than silently wrong.
3. **At minimum, document the divergence**: Expose a flag or separate raw-bytes field so API consumers can detect that the displayed name differs from the on-chain value.

Do **not** silently replace `\u0000` with `\uFFFD` in a domain setter, as this makes the mutation invisible to all callers.

### Proof of Concept
```
// 1. Construct a TokenCreate transaction with a null byte in the name
TokenCreateTransaction tx = new TokenCreateTransaction()
    .setTokenName("LEGIT\u0000TOKEN")   // null byte embedded
    .setTokenSymbol("LT")
    .setTreasuryAccountId(myAccountId)
    .setInitialSupply(1000);

// 2. Sign and submit to Hedera mainnet/testnet (no special keys needed)
TransactionResponse response = tx.execute(client);
TokenId tokenId = response.getReceipt(client).tokenId;

// 3. Query the mirror node REST API
// GET https://mainnet-public.mirrornode.hedera.com/api/v1/tokens/{tokenId}
// Response will contain:
// { "name": "LEGIT\uFFFDTOKEN", ... }   <-- replacement char, NOT null byte

// 4. Query the consensus record directly (e.g., via record stream or Hedera explorer)
// Record contains: name = "LEGIT\u0000TOKEN"   <-- original null byte

// Divergence confirmed: mirror node exports a name that was never submitted on-chain.
```
