### Title
Unbounded Historical Message Retrieval via Epoch-0 `consensusStartTime` Enables Database Connection Pool Exhaustion (DoS)

### Summary
An unprivileged external user can send a `ConsensusTopicQuery` with `consensusStartTime` set to epoch 0 (or any arbitrarily old timestamp). The only validation on `startTime` checks that it is not in the future, so epoch 0 passes cleanly. This triggers the historical retrieval path in `TopicMessageServiceImpl.subscribeTopic()`, which polls the database indefinitely in pages until all historical messages are returned. By opening many connections and saturating the per-connection call limit, an attacker can hold all database connections busy streaming historical data, starving legitimate subscribers and blocking new transaction confirmations.

### Finding Description

**Step 1 — Input accepted without lower-bound check**

`ConsensusController.toFilter()` converts the protobuf timestamp and sets it directly on the filter: [1](#0-0) 

`convertTimestamp()` only guards against overflow at the upper end; epoch 0 returns `0L` without error: [2](#0-1) 

**Step 2 — Validation only enforces an upper bound**

`TopicMessageFilter.isValidStartTime()` only rejects timestamps in the future: [3](#0-2) 

`startTime = 0` satisfies `0 <= DomainUtils.now()`, so the bean validation passes. There is no minimum age window enforced anywhere.

**Step 3 — Historical retrieval runs indefinitely for throttled subscriptions**

`TopicMessageServiceImpl.subscribeTopic()` immediately calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true): [4](#0-3) 

In `PollingTopicMessageRetriever`, the throttled path sets `numRepeats = Long.MAX_VALUE` and polls until `isComplete()` returns true: [5](#0-4) 

`isComplete()` for the throttled path only returns `true` when a page is smaller than `maxPageSize` (default 1000) or the limit is hit: [6](#0-5) 

For a topic with millions of messages, every page is full, so `isComplete()` never returns `true` until all messages are drained. Each poll issues `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))`, holding a database connection for the duration.

**Step 4 — No global subscriber or connection limit**

The only concurrency control is `maxConcurrentCallsPerConnection = 5` per TCP connection: [7](#0-6) 

There is no limit on the total number of incoming TCP connections, no per-IP rate limit, and no cap on total active subscribers. The `subscriberCount` gauge is metrics-only: [8](#0-7) 

**Step 5 — The 60-second timeout does not protect against active streams**

`retrieverProperties.getTimeout()` is a between-emission idle timeout, not a total subscription lifetime timeout: [9](#0-8) 

As long as messages keep flowing (which they do for a large topic), this timeout never fires.

### Impact Explanation
An attacker targeting a high-volume HCS topic (e.g., one used for token or NFT metadata with millions of messages) can open N TCP connections × 5 calls each, all with `startTime=0`. Each active subscription holds a database connection and a bounded-elastic scheduler thread while paging through history. Once the database connection pool is exhausted, all new queries — including those needed to confirm incoming transactions — queue or fail. This constitutes a total denial of service for the mirror node's gRPC API and indirectly for any downstream system relying on transaction confirmation via this node.

### Likelihood Explanation
The attack requires no authentication, no special account, and no on-chain funds. The gRPC port (default 5600) is publicly accessible. The attacker only needs to know a valid topic ID (trivially discoverable via the REST API or by observing network traffic). The attack is trivially scriptable: open many TCP connections, send `subscribeTopic` with `consensusStartTime.seconds = 0` on each stream. It is fully repeatable and can be sustained indefinitely.

### Recommendation
1. **Enforce a minimum `startTime` age** in `TopicMessageFilter.isValidStartTime()` — e.g., reject requests where `startTime < (now - maxHistoricalWindow)`, with `maxHistoricalWindow` configurable (e.g., 7 days default).
2. **Add a global concurrent-subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()` — check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add a per-IP or per-connection subscriber limit** at the gRPC interceptor layer.
4. **Enforce a total subscription lifetime timeout**, not just an idle-emission timeout, so long-running historical drains are bounded.
5. **Require a non-zero `limit`** when `startTime` is older than a configurable threshold, preventing unbounded open-ended historical scans.

### Proof of Concept
```python
import grpc
from hedera.api.proto import consensus_service_pb2_grpc
from hedera.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-host:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1234  # any valid high-volume topic

def make_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(
            shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM),
        consensusStartTime=Timestamp(seconds=0, nanos=0),  # epoch 0
        # no limit set → unlimited historical drain
    )
    return stub.subscribeTopic(query), channel

# Open many connections, 5 streams each
streams = []
for _ in range(200):  # 200 connections × 5 = 1000 concurrent DB-holding streams
    for _ in range(5):
        stream, ch = make_subscription()
        streams.append((stream, ch))

# Keep streams alive; DB connection pool is now exhausted
for stream, _ in streams:
    for _ in stream:
        pass
```

After this runs, legitimate `subscribeTopic` calls and REST API queries that require DB access will time out or return errors, blocking new transaction confirmations.
