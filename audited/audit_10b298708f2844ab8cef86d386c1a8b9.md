### Title
Unbounded `endTime` Enables Zombie Subscription Resource Exhaustion via Infinite `pastEndTime` RepeatSpec Loop

### Summary
An unprivileged gRPC client can submit a `subscribeTopic` request with an arbitrarily far-future `endTime` (e.g., year 2100). Because `TopicMessageFilter` imposes no upper bound on `endTime`, the `isComplete()` predicate returns `false` for decades, causing the `pastEndTime()` `RepeatSpec` to fire every 30 seconds indefinitely. This prevents `takeUntilOther` from ever signalling termination, keeping the entire subscription — including its `topicListener` connection, `TopicContext` state, and scheduler slots — alive as a zombie for the lifetime of the TCP connection. During a network partition, where the server cannot detect client disconnection, an attacker can accumulate many such zombie subscriptions to exhaust bounded scheduler and listener resources.

### Finding Description

**Exact code path:**

`pastEndTime()` at [1](#0-0)  constructs:

```java
return Flux.empty()
    .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
        .withFixedDelay(grpcProperties.getEndTimeInterval()));
```

This flux is wired as `takeUntilOther(pastEndTime(topicContext))` at [2](#0-1) . The main subscription only terminates when `pastEndTime` **completes** — which only happens when the repeat predicate `!topicContext.isComplete()` becomes `false`.

`isComplete()` at [3](#0-2)  returns `true` only when:

```java
Instant.ofEpochSecond(0, filter.getEndTime())
    .plus(grpcProperties.getEndTimeInterval())
    .isBefore(Instant.now());
```

With `endTime` set to nanoseconds representing year 2100, this condition is false for ~74 years. The RepeatSpec therefore fires every 30 seconds for `Long.MAX_VALUE` iterations (effectively forever), and `takeUntilOther` never fires.

**Root cause — failed validation:**

`TopicMessageFilter.isValidEndTime()` at [4](#0-3)  only enforces `endTime > startTime`. There is no upper bound. `isValidStartTime()` at [5](#0-4)  constrains `startTime <= now()` but places no ceiling on `endTime`.

**What stays alive per zombie subscription:**

- The `pastEndTime` RepeatSpec scheduler task (fires every 30 s)
- The `topicListener.listen()` reactive stream (continuous message listener)
- The `TopicContext` object and its `AtomicReference`/`AtomicLong` state
- A slot in the `boundedElastic` scheduler used by `safetyCheck` at [6](#0-5) 

**Why existing mitigations are insufficient:**

- `maxConcurrentCallsPerConnection = 5` at [7](#0-6)  limits calls *per connection*, not total connections. An attacker opens many connections.
- The nginx `grpc_read_timeout 600s` only applies when no data flows *through the proxy*. During a network partition the TCP session may appear live to the server (no FIN/RST), so the proxy timeout may not trigger.
- `subscriberCount` at [8](#0-7)  is a gauge for observability only — it does not enforce any cap.

### Impact Explanation

Each zombie subscription holds a live `topicListener` connection (backed by Redis or a polling DB connection depending on listener type) and a scheduled timer slot. Reactor's `boundedElastic` scheduler has a finite thread/task ceiling. With enough concurrent zombie subscriptions the scheduler queue fills, new legitimate subscriptions stall or are rejected, and listener back-pressure propagates to all subscribers — a full denial-of-service against the gRPC topic subscription service. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation

Any unauthenticated gRPC client can call `subscribeTopic` with `endTime` set to a nanosecond timestamp far in the future (e.g., `Instant.parse("2100-01-01T00:00:00Z").toEpochMilli() * 1_000_000`). The only precondition is that the topic exists (or `checkTopicExists = false`). The attacker then drops the TCP connection at the OS level (e.g., `iptables -A OUTPUT -p tcp --dport 5600 -j DROP`) to simulate a partition without sending FIN. This is trivially repeatable from a single host with many parallel connections, each consuming 5 subscription slots.

### Recommendation

1. **Enforce an upper bound on `endTime`** in `TopicMessageFilter.isValidEndTime()`: reject requests where `endTime > now + maxSubscriptionWindow` (e.g., 24 hours).
2. **Add a global maximum subscription duration** independent of `endTime`: terminate any subscription that has been alive longer than a configurable ceiling (e.g., 1 hour).
3. **Enforce a total concurrent-subscriber cap** (not just per-connection) by checking `subscriberCount` against a configured maximum before accepting new subscriptions.
4. **Configure gRPC server-side keepalive** (`GRPC_ARG_KEEPALIVE_TIME_MS`, `GRPC_ARG_KEEPALIVE_TIMEOUT_MS`) so the server actively probes and reaps dead TCP connections, reducing the partition window.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hedera.hashgraph.sdk.proto import timestamp_pb2, basic_types_pb2
import socket, threading

GRPC_HOST = "mirror-node:5600"
FAR_FUTURE_NS = 4102444800_000_000_000  # 2100-01-01 in nanoseconds

def open_zombie(i):
    channel = grpc.insecure_channel(GRPC_HOST)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusEndTime=timestamp_pb2.Timestamp(
            seconds=FAR_FUTURE_NS // 1_000_000_000,
            nanos=FAR_FUTURE_NS % 1_000_000_000
        )
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass  # consume nothing; attacker drops connection via iptables
    except:
        pass

# Step 1: Open N connections with far-future endTime
threads = [threading.Thread(target=open_zombie, args=(i,)) for i in range(500)]
for t in threads: t.start()

# Step 2: Simulate network partition — block outbound TCP to server
# iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# Result: Server holds 500+ zombie subscriptions, each with a RepeatSpec
# timer firing every 30s and a live topicListener connection.
# Legitimate subscribers are starved of scheduler/listener resources.
```
