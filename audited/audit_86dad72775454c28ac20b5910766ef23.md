### Title
Hardcoded Default Operator Private Key Enables Account Draining and Permanent Pinger Denial-of-Service

### Summary
The default operator private key is hardcoded in `pinger/config.go` and is publicly visible in source code. Any attacker who reads the source can use this key to sign transactions as the operator and drain account `0.0.2` to zero. Once drained, every `Execute(client)` call in `submitWithRetry()` fails with `INSUFFICIENT_ACCOUNT_BALANCE`, and because the retry loop applies uniform backoff-and-retry to all error types including this permanent condition, transaction confirmation is halted on every subsequent tick.

### Finding Description
**Root cause — hardcoded key:**
`pinger/config.go` line 40 sets the default `operatorKey` to a literal Ed25519 private key:
```
302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137
```
for operator account `0.0.2`. This value is used verbatim unless overridden by the `HIERO_MIRROR_PINGER_OPERATOR_KEY` environment variable. Any deployment that omits that variable uses the publicly known key.

**Exploit path in `submitWithRetry()`:**
`pinger/transfer.go` lines 29–33 build and execute a transfer signed by the operator:
```go
cryptoTransfer := hiero.NewTransferTransaction().
    AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
    AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))
resp, err := cryptoTransfer.Execute(client)
```
If the operator account has been drained, `Execute` returns `INSUFFICIENT_ACCOUNT_BALANCE`. The retry loop (lines 23–57) treats this identically to any transient error: it retries up to `cfg.maxRetries` times (default 10) with exponential backoff, then returns the error. Back in `main.go` lines 63–68, the error is only logged; the ticker fires again next interval and the same permanent failure repeats indefinitely.

**No error classification exists anywhere** in `transfer.go`, `main.go`, or `sdk_client.go` that would detect a permanent protocol error and stop retrying or alert operators.

### Impact Explanation
An attacker who drains the operator account causes every pinger tick to exhaust all retries and fail. The pinger's sole purpose is to confirm that the network can process new transactions; with every tick failing, that confirmation signal is permanently lost. The liveness file `/tmp/alive` continues to be written (it is independent of transfer success), so Kubernetes liveness probes do not detect the failure, masking the outage from automated recovery.

### Likelihood Explanation
The private key is in a public repository and requires zero authentication to read. The default operator `0.0.2` is a well-known genesis account. The only precondition is that the deployment was started without overriding `HIERO_MIRROR_PINGER_OPERATOR_KEY`. An attacker needs only the Hiero SDK (or any compatible client), the key string from the source file, and one signed transfer transaction to drain the account. The attack is repeatable: if the account is refunded, the attacker can drain it again immediately.

### Recommendation
1. **Remove the hardcoded key entirely.** Make `operatorKey` a required field with no default; fail fast at startup if it is absent (`config.go` lines 105–107 already have the empty-string check, but the default prevents it from triggering).
2. **Classify non-retryable errors.** In `submitWithRetry()`, inspect the returned error for status codes such as `INSUFFICIENT_ACCOUNT_BALANCE`, `INVALID_ACCOUNT_ID`, and `KEY_NOT_PROVIDED`. Return immediately without retrying on permanent failures.
3. **Alert on permanent failure.** In `main.go`, distinguish a permanent transfer failure from a transient one and emit a metric or structured log that triggers an alert, rather than silently logging and continuing.

### Proof of Concept
1. Clone the repository and read the default key from `pinger/config.go` line 40.
2. Deploy (or find) a pinger instance running with default configuration (no `HIERO_MIRROR_PINGER_OPERATOR_KEY` override).
3. Using the Hiero Go SDK or `hiero-cli`, construct a `TransferTransaction` that moves the entire balance of account `0.0.2` to an attacker-controlled account, signed with the extracted private key.
4. Submit the transaction to the target network.
5. Observe that all subsequent pinger ticks log `"all attempts failed: INSUFFICIENT_ACCOUNT_BALANCE"` after exhausting 10 retries each, and that no successful transfer receipt is ever recorded, halting transaction-confirmation monitoring indefinitely.
