### Title
Hardcoded `sslmode=disable` in Rosetta Postgres DSN Exposes Credentials to Network Sniffing, Enabling Balance Table Corruption

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` hardcodes `sslmode=disable` unconditionally, transmitting the full Postgres DSN — including username and password — in plaintext over the network. An attacker with access to the network path between the Rosetta service and the Postgres instance can passively capture these credentials and use them to connect directly to the database, issuing DML queries that corrupt the `account_balance` snapshot tables consumed by `RetrieveBalanceAtBlock`, producing systematically wrong balance results for all Rosetta `/account/balance` queries.

### Finding Description

**Exact code location:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51: [1](#0-0) 

```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        ...
    )
}
```

This DSN is consumed directly in `ConnectToDb()`: [2](#0-1) 

And also in the health check endpoint: [3](#0-2) 

**Root cause:** `sslmode=disable` is hardcoded as a string literal — there is no field in the `Db` struct for SSL configuration, no environment variable override, and no code path that can enable TLS. The `Db` struct contains only `Host`, `Name`, `Password`, `Pool`, `Port`, `StatementTimeout`, and `Username`: [4](#0-3) 

**Failed assumption:** The design assumes the Postgres connection always travels over a trusted, isolated network (e.g., loopback or a secured private VLAN). In practice, Kubernetes pod-to-pod traffic, cloud VPC traffic, or any shared network segment is not inherently encrypted, and the `sslmode=disable` setting actively prevents the Postgres client from negotiating TLS even if the server supports it.

**Exploit flow:**
1. Attacker gains a position on the network path between the Rosetta pod and the Postgres pod (e.g., via ARP poisoning on a flat L2 network, a compromised sidecar/CNI plugin, a rogue pod in the same namespace with no NetworkPolicy, or a cloud-provider-level network tap).
2. Attacker captures the TCP stream on port 5432. Because `sslmode=disable` is set, the Postgres wire protocol startup message is sent in plaintext, including the username. The password follows in the `PasswordMessage` or `SASLResponse` packet — also plaintext (MD5 or SCRAM challenge-response, but the captured exchange is sufficient for offline cracking or replay depending on auth method).
3. Attacker uses the recovered credentials to open a direct connection to the Postgres instance (if the DB port is reachable, or via the same network position).
4. Attacker issues `UPDATE account_balance SET balance = 0 WHERE account_id = <target>` or inserts fabricated snapshot rows, corrupting the data read by `getLatestBalanceSnapshot()`: [5](#0-4) 
5. `RetrieveBalanceAtBlock` computes `hbarAmount.Value = snapshotBalance + balanceChange`, where `snapshotBalance` now reflects the attacker-controlled value: [6](#0-5) 
6. All subsequent `/account/balance` API responses return attacker-controlled balances.

**Why existing checks are insufficient:** There are no TLS enforcement checks, no certificate pinning, no network-layer encryption enforced by the application, and no integrity verification of data returned from the DB. The `sslmode=disable` string is also present in the test helper (`rosetta/test/db/db.go` line 86), confirming this is a deliberate, pervasive design choice rather than an oversight: [7](#0-6) 

### Impact Explanation
An attacker who corrupts `account_balance` rows causes `RetrieveBalanceAtBlock` to return wrong HBAR balances for any account. Since this function drives the Rosetta `/account/balance` endpoint (used by exchanges and custodians for fund accounting), systematic corruption leads to incorrect deposit/withdrawal decisions — a direct fund-accounting integrity failure. The attacker can also exfiltrate the entire mirror node database (transaction history, entity keys, token balances) using the same credentials.

### Likelihood Explanation
The precondition is network adjacency to the Postgres connection, not full external internet access. In Kubernetes deployments without NetworkPolicy (the default), any pod in the cluster — including one obtained via a compromised dependency or image — can reach the DB port and sniff or MITM traffic. Cloud environments with shared VPCs, misconfigured security groups exposing port 5432, or flat overlay networks (common in development and staging) all satisfy this precondition. The hardcoded `sslmode=disable` cannot be remediated by operators without modifying source code, making every deployment permanently vulnerable regardless of Postgres server TLS configuration.

### Recommendation
1. Add an `SslMode` field to the `Db` config struct in `rosetta/app/config/types.go` (defaulting to `require` or `verify-full`).
2. Interpolate it into `GetDsn()` instead of the hardcoded `disable`.
3. Remove `sslmode=disable` from all non-test DSN construction paths, including `rosetta/app/middleware/health.go`.
4. Enforce `sslmode=verify-full` with a CA certificate in production Helm values.
5. Apply Kubernetes NetworkPolicy to restrict pod-to-pod access to the Postgres port.

### Proof of Concept
```bash
# 1. On a pod in the same Kubernetes namespace (no special privileges needed if no NetworkPolicy):
kubectl run sniffer --image=nicolaka/netshoot -it --rm -- \
  tcpdump -i eth0 -A 'tcp port 5432' -w /tmp/pg.pcap

# 2. Wait for Rosetta to connect (startup or health check triggers GetDsn()).
# 3. Extract credentials from the pcap:
strings /tmp/pg.pcap | grep -E 'user=|password='
# Output example: host=postgres port=5432 user=mirror_rosetta dbname=mirror_node password=<plaintext>

# 4. Connect directly to Postgres with recovered credentials:
psql "host=<pg-host> port=5432 user=mirror_rosetta dbname=mirror_node password=<plaintext> sslmode=disable"

# 5. Corrupt balance snapshot:
UPDATE account_balance SET balance = 0 WHERE account_id = <target_account_id>;

# 6. Verify Rosetta returns corrupted balance:
curl -X POST http://<rosetta-host>:5700/account/balance \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.<target>"}}'
# Returns {"balances":[{"value":"0","currency":{"symbol":"HBAR","decimals":8}}]}
```

```go
func (ar *accountRepository) getLatestBalanceSnapshot(ctx context.Context, accountId, timestamp int64) (
	int64,
	*types.HbarAmount,
	*rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var partitionLowerBound int64
	if err := db.Raw(
		selectPreviousPartitionLowerBound,
		sql.Named("timestamp", timestamp),
	).Scan(&partitionLowerBound).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for previous account_balance partition before %d", err, timestamp),
		)
	}

	// gets the most recent balance at or before timestamp
	ab := &accountBalance{}
	if err := db.Raw(
		latestBalanceBeforeConsensus,
		sql.Named("account_id", accountId),
		sql.Named("lower_bound", partitionLowerBound),
		sql.Named("timestamp", timestamp),
		sql.Named("treasury_entity_id", ar.treasuryEntityId.EncodedId),
	).First(ab).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for account %d's balance at or before %d", err, accountId, timestamp),
		)
		return 0, nil, hErrors.ErrDatabaseError
	}

	if ab.ConsensusTimestamp == 0 {
		return 0, nil, hErrors.ErrNodeIsStarting
	}

	hbarAmount := types.HbarAmount{Value: ab.Balance}

	return ab.ConsensusTimestamp, &hbarAmount, nil
```
