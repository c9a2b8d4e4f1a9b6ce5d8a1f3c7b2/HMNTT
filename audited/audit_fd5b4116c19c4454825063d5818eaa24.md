### Title
Silent Result Truncation via `name` + `token.type` Filter Combination Disabling Pagination in `getTokensRequest()`

### Summary
In `rest/tokens.js`, `getTokensRequest()` unconditionally sets the `next` pagination link to `null` whenever the `name` query parameter is present, but the SQL `LIMIT` clause is still applied. Because no guard blocks combining `name` with `token.type`, an unprivileged caller can issue a single request that silently returns only the first page of matching records with `links.next = null`, giving consumers no signal that additional records exist.

### Finding Description
**Code path:**

`rest/tokens.js` → `getTokensRequest()` (lines 360–422) → `extractSqlFromTokenRequest()` (lines 164–214)

**Root cause — two independent facts that interact badly:**

1. `extractSqlFromTokenRequest()` always appends a `LIMIT` clause (line 210):
   ```js
   const limitQuery = `limit $${params.push(limit)}`;
   ```
   This is unconditional; it applies whether or not a `name` filter is present.

2. `getTokensRequest()` unconditionally nulls the `next` link when `name` is present (lines 405–406):
   ```js
   const nextLink = hasNameParam
     ? null
     : utils.getPaginationLink(…);
   ```

**Missing guard:** The function blocks `name` + `token.id` (line 362) and `name` + `account.id` (line 366), but there is **no equivalent check for `name` + `token.type`**. The `token.type` filter flows through `filterColumnMap` (line 89: `[filterKeys.TOKEN_TYPE]: sqlQueryColumns.TYPE`) and is appended as a normal SQL condition (line 201):
```js
conditions.push(`${filterColumnMap[filter.key]}${filter.operator}$${params.push(filter.value)}`);
```

**Resulting SQL for `GET /api/v1/tokens?name=token&type=FUNGIBLE_COMMON`:**
```sql
SELECT t.decimals, t.freeze_status, e.key, t.kyc_status, t.metadata,
       t.name, t.symbol, t.token_id, t.type
FROM token t
JOIN entity e ON e.id = t.token_id
WHERE t.name ILIKE '%token%'
  AND t.type = 'FUNGIBLE_COMMON'
ORDER BY t.token_id ASC
LIMIT 25          -- always present
```
If more than 25 rows match, the response contains only the first 25 and `links.next = null`. The caller receives a structurally complete, non-error response with no indication that records were truncated.

### Impact Explanation
Any consumer that relies on `links.next` to detect whether a full result set has been delivered — including mirror-node sync processes, indexers, or wallets enumerating tokens — will silently receive an incomplete dataset. Because the response is HTTP 200 with a well-formed JSON body and `links.next = null`, there is no observable error. On a network with many tokens sharing a common name substring (e.g., `name=USD`, `name=token`, `name=wrapped`), the number of omitted records can be large. This constitutes a data-integrity / completeness failure for any downstream system that trusts the API's pagination contract.

### Likelihood Explanation
The request requires no authentication, no special account, and no privileged role — any HTTP client can trigger it. The parameter combination (`name` + `type`) is explicitly listed as accepted in `acceptedTokenParameters` (lines 1076–1084) and passes all validation in `validateTokenQueryFilter()`. The trigger is a single GET request with two ordinary query parameters. It is trivially repeatable and requires no timing dependency or race condition.

### Recommendation
Apply one of the following fixes in `getTokensRequest()`:

**Option A — Block the combination (consistent with existing guards):**
```js
if (hasNameParam && req.query[filterKeys.TOKEN_TYPE]) {
  throw new InvalidArgumentError('token.type and name cannot be used together');
}
```

**Option B — Preserve pagination when `name` is combined with other filters:**
Remove the blanket `hasNameParam ? null` shortcut and instead compute the `next` link normally (using `token_id` as the cursor), so that callers can page through all matching results regardless of which filters are active.

**Option C — Remove the LIMIT when `name` is the only filter:**
If the design intent is to return all name-matching results in one response, remove the `LIMIT` clause when `hasNameParam` is true and no other filters are present, and enforce a hard cap with a clear error if the result set exceeds a safe maximum.

### Proof of Concept

**Precondition:** The mirror node database contains more than 25 tokens whose names contain the substring `"token"` and whose type is `FUNGIBLE_COMMON`.

**Request (no authentication required):**
```
GET /api/v1/tokens?name=token&type=FUNGIBLE_COMMON
```

**Observed response (HTTP 200):**
```json
{
  "tokens": [ /* exactly 25 entries */ ],
  "links": { "next": null }
}
```

**Expected response (correct behaviour):**
```json
{
  "tokens": [ /* up to 25 entries */ ],
  "links": { "next": "/api/v1/tokens?name=token&type=FUNGIBLE_COMMON&token.id=gt:0.0.XXXX&order=asc" }
}
```

**Verification:** Issue the same request with `limit=1` to confirm only one record is returned and `next` is still `null`, while a direct `SELECT COUNT(*)` against the database returns a count greater than 1 for the same predicate.
