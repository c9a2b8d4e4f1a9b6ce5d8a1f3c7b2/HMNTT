### Title
Unbounded Concurrent DB Query Amplification via `getTokenRelationships()` Without Rate Limiting or Request Deduplication

### Summary
`getTokenRelationships()` in `rest/controllers/tokenController.js` issues a fresh database query for every incoming HTTP request with no application-level rate limiting, no in-flight request deduplication, and a response cache that is trivially bypassed by varying query parameters. An unprivileged attacker can send a large number of concurrent requests with slightly different parameters to force proportionally many simultaneous `token_account` SQL queries against the database, multiplying DB load without bound.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–92:

```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);   // DB query 1
  const isValidAccount = await EntityService.isValidAccount(accountId); // DB query 2
  ...
  const tokenRelationships = await TokenService.getTokenAccounts(query); // DB query 3
```

`rest/service/tokenService.js`, `getTokenAccounts()`, lines 96–115: executes `tokenRelationshipsQuery` (a full `SELECT … FROM token_account WHERE account_id = $1 AND associated = true …`) on every call with no result-level cache.

**Root cause — three compounding failures:**

1. **No application-level rate limiting for the REST API.** The only rate-limiting code in the repository (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives in the `web3` module and is never applied to the REST API routes. The REST middleware stack (`rest/middleware/index.js`) contains no throttle handler.

2. **No in-flight request deduplication.** There is no promise-sharing, mutex, or semaphore. Every concurrent request that reaches `getTokenAccounts()` independently awaits its own DB query.

3. **Response cache is trivially bypassed.** `responseCacheHandler.js` (line 152) generates the Redis cache key as `MD5(req.originalUrl)`. Any change to the query string — `limit=25` vs `limit=24`, `order=asc` vs `order=desc`, `token.id=gt:1` vs `token.id=gt:2` — produces a distinct cache key and a cache miss, forcing a new DB query. The in-memory `tokenCache` in `tokenService.js` (lines 12–14) only caches token *metadata* (decimals, freeze/kyc status) from the secondary `token` table, not the primary `token_account` query results.

**Exploit flow:**

An attacker targets a known large account (e.g., a treasury or exchange account with thousands of token associations). They send N concurrent HTTP requests to `/api/v1/accounts/{large_account_id}/tokens` with systematically varied parameters:

```
GET /api/v1/accounts/0.0.98/tokens?limit=25&order=asc
GET /api/v1/accounts/0.0.98/tokens?limit=24&order=asc
GET /api/v1/accounts/0.0.98/tokens?limit=25&order=desc
GET /api/v1/accounts/0.0.98/tokens?token.id=gt:1
GET /api/v1/accounts/0.0.98/tokens?token.id=gt:2
... (N variants)
```

Each URL has a unique MD5 cache key → cache miss → independent DB query. All N queries execute simultaneously. The `token_account` table scan for a large account is expensive (full index scan on `account_id`, returning thousands of rows). N concurrent such scans directly multiply DB CPU and I/O load by N.

### Impact Explanation

The database is a shared resource serving all mirror node consumers. Sustained amplified load on the `token_account` table degrades query latency for all other API endpoints that touch the same table (balance queries, NFT queries, etc.). With a sufficiently large N and a large target account, the attacker can saturate DB connection pools or push query latency high enough to cause cascading timeouts across the API. No authentication, no funds, and no privileged access are required — only the ability to send HTTP GET requests.

### Likelihood Explanation

The attack requires zero privileges and zero cost beyond network bandwidth. The target account IDs are publicly enumerable from the same API. The parameter space for cache-busting is large (limit 1–100, order asc/desc, token.id range filters) and requires no guessing. Any public mirror node deployment without an external reverse-proxy rate limiter (Traefik middleware is optional and not part of the application) is directly exposed. The attack is repeatable and automatable with a simple script.

### Recommendation

1. **Add application-level per-IP rate limiting** to the REST API middleware stack (e.g., using `express-rate-limit` or equivalent), applied before route handlers.
2. **Implement in-flight request coalescing** for identical logical queries: maintain a `Map<cacheKey, Promise>` so that N concurrent requests for the same data share one DB round-trip.
3. **Normalize and canonicalize query parameters** before generating the cache key so that semantically equivalent requests (e.g., `limit=25` is the default) share a cache entry.
4. **Cap the maximum `limit` parameter** to a lower value for this endpoint to bound the per-query cost.

### Proof of Concept

```bash
# Generate 50 concurrent requests with varied cache-busting parameters
for i in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.98/tokens?limit=$i&order=asc" &
done
wait
```

Each of the 50 requests produces a unique Redis cache key (`MD5("/api/v1/accounts/0.0.98/tokens?limit=N&order=asc")`), resulting in 50 simultaneous `SELECT … FROM token_account WHERE account_id = 98 AND associated = true ORDER BY token_id ASC LIMIT N` queries executing concurrently against the database. Repeating this loop continuously sustains the amplified load.

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
