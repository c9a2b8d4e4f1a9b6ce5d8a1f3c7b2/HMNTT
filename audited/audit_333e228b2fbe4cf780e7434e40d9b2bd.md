### Title
Unbounded EVM Binary-Search Multiplication via Null `to` Field Enabling Contract-Creation Simulation Griefing

### Summary
Any unauthenticated user can POST to `/api/v1/contracts/call` with `to` omitted and a crafted init-bytecode payload, causing `constructServiceParameters()` to set `receiver = Address.ZERO` and route the call into the contract-creation simulation path. When combined with `estimate: true`, the `estimateGas` binary-search loop executes the EVM O(log(gasLimit)) â‰ˆ 24 times per request, while the gas-bucket throttle only accounts for a single execution, allowing disproportionate CPU consumption relative to the throttle cost paid.

### Finding Description

**Code path:**

`ContractController.constructServiceParameters()` unconditionally maps a null/empty `to` to `Address.ZERO`: [1](#0-0) 

The `ContractCallRequest.hasTo()` validator permits `to=null` whenever `value <= 0` (the default), regardless of `data` content: [2](#0-1) 

Because `value` defaults to `0`, `isValidToField` is always `true`, so the `BytecodeUtils.isValidInitBytecode(data)` branch is never the deciding factor â€” the validation passes for any `data` with `to=null`.

When `estimate=true`, `ContractExecutionService.estimateGas()` first runs a full EVM execution, then launches a binary search: [3](#0-2) 

Each binary-search iteration calls `doProcessCall(params, gas, true)`. Inside `doProcessCall`, the `finally` block that calls `restoreGasToBucket` is guarded by `if (!estimate)`: [4](#0-3) 

This means the gas throttle is consumed once (at `throttleManager.throttle(request)`) but the EVM is invoked ~24 additional times during the binary search without any corresponding throttle deduction. The `Address.ZERO` receiver with init-bytecode data is a confirmed contract-creation simulation path, as evidenced by test usage: [5](#0-4) 

**Root cause:** The gas-bucket throttle models cost as a single EVM execution, but `estimateGas` performs O(log(maxGasLimit)) executions. The `to=null â†’ Address.ZERO` mapping combined with the permissive `hasTo()` check (always true when `value=0`) allows any caller to reach this expensive path without restriction.

### Impact Explanation
An attacker can flood the `/api/v1/contracts/call` endpoint with requests that each trigger ~25 EVM executions (1 initial + ~24 binary-search iterations) while consuming only 1 unit of gas-throttle budget. With the default `maxGasLimit` of 15,000,000 and complex init bytecode, each request imposes ~25Ã— the CPU cost the throttle was designed to limit. Sustained at even moderate request rates this degrades response latency for all other users sharing the same server, constituting a denial-of-service griefing attack with no economic cost to the attacker.

### Likelihood Explanation
No authentication, API key, or privileged role is required. The attacker only needs to craft a hex string matching the `isInitBytecode` regex pattern `(60806040|60606040)[0-9a-z]+39[0-9a-z]+f3`, which is trivially constructable (e.g., `60806040<padding>39<padding>f3`). The attack is fully repeatable and scriptable from any HTTP client.

### Recommendation
1. **Throttle by wall-clock iterations, not only gas**: Deduct throttle tokens for each binary-search iteration inside `estimateGas`, or impose a hard cap on the number of iterations.
2. **Tighten `hasTo()` for the contract-creation path**: When `to` is null/empty and `data` matches init-bytecode, require explicit opt-in (e.g., a dedicated `contractCreate` flag) rather than silently routing through `Address.ZERO`.
3. **Add a request-rate limit** (e.g., per-IP token bucket) independent of gas, so the binary-search multiplication cannot be exploited even if gas throttling is otherwise correct.
4. **Cap `data` payload size** at the HTTP layer to bound the cost of init-bytecode parsing and EVM loading.

### Proof of Concept
```bash
# Minimal valid init-bytecode matching the regex: 60806040 + padding + 39 + padding + f3
INIT_BYTECODE="0x6080604052600039600060006000f3"

# Send estimate request with no "to" field (triggers Address.ZERO + binary search)
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "data": "'"$INIT_BYTECODE"'",
    "estimate": true,
    "gas": 15000000
  }'

# Repeat in a tight loop from multiple clients.
# Each request triggers ~25 EVM executions while consuming 1 throttle unit.
# Observe increasing p99 latency for legitimate callers.
```

        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one

                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
