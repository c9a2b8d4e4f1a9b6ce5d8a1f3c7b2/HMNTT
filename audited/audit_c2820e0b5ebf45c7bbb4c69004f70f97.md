### Title
Plaintext HTTP Allowed for Mirror Node Network Discovery Enables MITM Node Injection

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` constructs an `http.Client` with no TLS configuration and no enforcement that the configured URL uses `https://`. The documented default example URL is explicitly `http://mirror-rest:5551`. An attacker with network-level access (DNS spoofing, ARP poisoning, or a compromised network path) can intercept the plaintext response and inject rogue `service_endpoints`, causing the Hiero SDK client to route all subsequent transactions to attacker-controlled nodes.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 46:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```
No `Transport` with `TLSClientConfig` is set. The URL is taken verbatim from `cfg.mirrorRest` (env `HIERO_MIRROR_PINGER_REST`), whose documented default example in `pinger/config.go` line 37 is `http://mirror-rest:5551` — plaintext HTTP.

`fetchMirrorNodeNetwork()` (lines 74–131) issues a `GET` to that URL, decodes the JSON body into `nodesEnvelope`, and for every `serviceEndpoint` builds `host:port → AccountID` entries (lines 113–124). This map is returned to `sdk_client.go` line 22:
```go
client = hiero.ClientForNetwork(netmap)
```
The SDK then uses this map for all consensus node gRPC calls.

**Root cause:** No scheme validation (`strings.HasPrefix(url, "https://")` check is absent), no TLS transport enforcement, and no cryptographic verification of the response body. The code blindly trusts whatever JSON arrives over the wire.

**Why existing checks fail:** The only checks present are HTTP status code validation (lines 90–93) and JSON decode error handling (line 96). Neither detects a MITM-injected response that returns HTTP 200 with well-formed but malicious JSON.

### Impact Explanation
A successful injection replaces the legitimate consensus node map with attacker-controlled endpoints. The Hiero SDK client will submit signed transactions (including fund transfers) to those endpoints. The attacker can:
- Silently drop transactions (denial of service for the pinger's monitoring function)
- Observe transaction content (operator account ID, keys used in signing context, transfer amounts)
- Return crafted gRPC responses to manipulate retry/backoff logic

The pinger's operator private key (`cfg.operatorKey`) is used to sign every transfer; while the key itself is not directly exposed, all signed transaction bytes are sent to the rogue node. Severity: **Medium** — no direct fund theft without further exploitation, but network map integrity is fully compromised.

### Likelihood Explanation
The attack requires network-level interposition between the pinger pod and the mirror-rest service. In typical Kubernetes deployments this means:
- **DNS spoofing** of the internal service name (e.g., `mirror-rest`) — feasible if the attacker has compromised another pod in the same namespace or can influence CoreDNS
- **ARP poisoning** on a flat L2 segment — feasible in VM-based or bare-metal deployments without network policy enforcement
- **Compromised sidecar/proxy** — feasible in service-mesh environments with misconfigured mTLS

The default example URL being HTTP (`http://mirror-rest:5551`) strongly suggests many deployments run without TLS on this path. The attack is repeatable on every pinger restart (startup network discovery) and on any retry cycle.

### Recommendation
1. **Enforce HTTPS in config validation** (`pinger/config.go`, after line 133):
   ```go
   if cfg.network == "other" && !strings.HasPrefix(cfg.mirrorRest, "https://") {
       return cfg, fmt.Errorf("HIERO_MIRROR_PINGER_REST must use https://")
   }
   ```
2. **Optionally pin the CA** by setting a custom `tls.Config` with `RootCAs` on the `http.Transport` used in `buildNetworkFromMirrorNodes()`.
3. Update the documented default example in `config.go` line 37 from `http://` to `https://`.

### Proof of Concept
**Preconditions:** Attacker controls DNS resolution for `mirror-rest` (or can ARP-poison the path). Pinger is configured with `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` and `HIERO_MIRROR_PINGER_NETWORK=other`.

**Steps:**
1. Stand up a rogue HTTP server on the attacker-controlled host responding to `GET /api/v1/network/nodes` with:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}]}
   ```
2. Redirect DNS for `mirror-rest` to the attacker's IP (or ARP-poison the gateway).
3. Start (or restart) the pinger. `buildNetworkFromMirrorNodes()` calls `fetchMirrorNodeNetwork()`, receives the injected response, and returns `{"<attacker-ip>:50211": AccountID{0,0,3}}`.
4. `hiero.ClientForNetwork(netmap)` is called with the rogue map. All subsequent `submitWithRetry` calls send signed transactions to `<attacker-ip>:50211`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```go
func fetchMirrorNodeNetwork(
	ctx context.Context,
	httpClient *http.Client,
	url string,
) (map[string]hiero.AccountID, bool, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
	if err != nil {
		return nil, false, err
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
	}
	defer resp.Body.Close()

	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
		return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
	}

	var payload nodesEnvelope
	if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
		return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
	}
```
