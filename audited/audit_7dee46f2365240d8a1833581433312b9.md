### Title
Missing EQ+Range Operator Mixing Validation in `getNetworkNodes()` Causes Silent Node Omission

### Summary
`NetworkServiceImpl.getNetworkNodes()` accepts mixed `node.id=eq:X&node.id=gte:Y` parameters from any unauthenticated caller. When `X < Y`, the EQ value is added to `nodeIds` but then filtered out by the range check, causing the method to return an empty list even though node `X` genuinely exists. The identical protection present for `registered-node.id` was never applied to `node.id`.

### Finding Description

**Exact code path:**

In `NetworkServiceImpl.java` lines 110–129:

```java
for (final var nodeIdParam : nodeIdParams) {
    if (nodeIdParam.operator() == RangeOperator.EQ) {
        nodeIds.add(nodeIdParam.value());          // adds X to nodeIds
    } else if (nodeIdParam.hasLowerBound()) {
        lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue()); // sets lowerBound=Y
    } ...
}
// lowerBound=Y, upperBound=MAX_VALUE, nodeIds={X}
// Range.closed(Y, MAX_VALUE) does NOT contain X when X < Y
final var range = Range.closed(lowerBound, upperBound);
nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
if (nodeIdArray.length == 0) {
    return List.of();   // ← silent empty result
}
```

**Root cause:** There is no guard preventing EQ and range operators from being combined for `node.id`. The `lowerBound > upperBound` check on line 120 only catches contradictory range-vs-range inputs; it does not catch the case where an EQ value falls outside a valid range.

**Failed assumption:** The code assumes that if `nodeIds` is non-empty after the loop, at least one ID will survive the range filter. This assumption is violated whenever `eq:X` and `gte:Y` (with `Y > X`) are supplied together.

**Contrast with the fix already present for registered nodes** (`resolveRegisteredNodeIdBounds`, lines 159–162):
```java
if (range.operator() == RangeOperator.EQ) {
    if (registeredNodeIdRanges.size() > 1) {
        throw new IllegalArgumentException("The 'eq' operator cannot be combined with other operators");
    }
```
This guard exists for `registered-node.id` but is absent for `node.id`.

### Impact Explanation
`GET /api/v1/network/nodes` is the canonical source of network topology for Hedera/Hiero SDKs and smart contract tooling. An empty response is indistinguishable from a node genuinely not existing. Tooling that caches or acts on this response (e.g., to select submission nodes, verify node membership before a contract call, or populate an address book) will incorrectly conclude that a live node is absent. This matches the stated severity: Layer 0/1 network metadata corruption with no direct fund loss but with potential for misdirected transactions or failed contract interactions that depend on accurate node enumeration.

### Likelihood Explanation
No authentication or special privilege is required. The endpoint is public. The exploit requires only knowledge of the query parameter format (`node.id=eq:X&node.id=gte:Y`), which is documented API behavior. It is trivially repeatable and automatable. Any attacker who wants to feed incorrect topology to a monitoring tool, SDK, or smart contract oracle can do so on demand.

### Recommendation
Apply the same guard already used in `resolveRegisteredNodeIdBounds` to `getNetworkNodes()`. At the top of the loop (or before it), reject any request that mixes an EQ operator with any other operator for `node.id`:

```java
boolean hasEq = nodeIdParams.stream().anyMatch(p -> p.operator() == RangeOperator.EQ);
boolean hasRange = nodeIdParams.stream().anyMatch(p -> p.operator() != RangeOperator.EQ);
if (hasEq && hasRange) {
    throw new IllegalArgumentException("The 'eq' operator cannot be combined with other operators for node.id");
}
```

Alternatively, if mixed semantics are intentional, the range filter should be skipped when EQ values are present (i.e., treat EQ as an absolute override, not subject to range intersection).

### Proof of Concept

**Precondition:** Node with `node_id = 5` exists in the address book (standard mainnet/testnet topology).

**Request:**
```
GET /api/v1/network/nodes?node.id=eq:5&node.id=gte:10
```

**Execution trace:**
1. `nodeIdParams` = `[{EQ, 5}, {GTE, 10}]`
2. Loop: `nodeIds = {5}`, `lowerBound = 10`, `upperBound = Long.MAX_VALUE`
3. `lowerBound(10) > upperBound(MAX_VALUE)`? → No → no exception thrown
4. `nodeIds` non-empty → `range = Range.closed(10, MAX_VALUE)`
5. `filter({5}, range)` → `5 ∉ [10, MAX_VALUE]` → `nodeIdArray = []`
6. `nodeIdArray.length == 0` → `return List.of()`

**Response:**
```json
{ "nodes": [], "links": { "next": null } }
```

**Expected response:** Node 5's full record (description, endpoints, stake, etc.).

**Result:** Any consumer of this API (SDK, smart contract oracle, monitoring tool) incorrectly concludes node 5 does not exist in the network. [1](#0-0) [2](#0-1) [3](#0-2)
