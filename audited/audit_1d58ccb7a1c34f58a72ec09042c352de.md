### Title
Silent Data Loss via TCP Half-Open State in Redis Pub/Sub Subscription (No Heartbeat or Timeout)

### Summary
The `subscribe()` method in `RedisTopicListener` builds a reactive `Flux` over a Redis pub/sub channel with no timeout operator, no heartbeat, and no application-level keepalive. When a network partition drives the underlying TCP connection into a half-open state, the Lettuce/Netty socket appears connected to the JVM while Redis has silently dropped it. The reactive stream emits neither items nor an error signal, so the `retryWhen` never fires and every subscriber on that topic receives zero messages indefinitely with no indication of failure.

### Finding Description
**Exact code path:** `grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–79.

```java
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))
    .doOnComplete(() -> unsubscribe(topic))
    .doOnError(t -> log.error("Error listening for messages", t))
    .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .share();
```

**Root cause:** `r.receive()` returns a cold `Flux` backed by a Lettuce `StatefulRedisPubSubConnection`. Lettuce uses Netty for I/O. When a network partition causes a TCP half-open (the local kernel still considers the socket `ESTABLISHED` but the remote side or an intermediate device has silently dropped it), Netty's event loop sees no I/O events and emits nothing — no `onNext`, no `onError`, no `onComplete`. The `retryWhen` operator is wired only to the `onError` signal; a permanently silent `Flux` never triggers it.

**Failed assumption:** The design assumes that any loss of connectivity to Redis will surface as an error signal on the reactive stream. This is true for clean disconnects (TCP RST, Redis `NOAUTH`, etc.) but false for half-open TCP, where the OS-level socket remains open and Netty has no data to read, so it simply waits forever.

**Why existing checks are insufficient:**
- `retryWhen(Retry.backoff(...))` — only retries on `onError`; a silent stream never errors.
- `doOnError` — only logs; does not help recovery.
- `doOnCancel` / `doOnComplete` — never invoked while the stream is silently stuck.
- The gRPC module's `RedisConfiguration` (`grpc/src/main/java/org/hiero/mirror/grpc/config/RedisConfiguration.java`) configures no TCP keepalive or socket timeout on the `ReactiveRedisConnectionFactory`. Contrast with the REST module's `rest/cache.js` line 27 which explicitly sets `keepAlive: 30000`.
- No `.timeout(Duration)` operator is applied anywhere in the pipeline to detect a prolonged absence of messages.

**Exploit flow:**
1. A gRPC client (unprivileged) subscribes to a topic via `ConsensusService/subscribeTopic`.
2. The mirror node's `RedisTopicListener` creates (or reuses via `topicMessages` map) a shared `Flux` backed by a Redis pub/sub subscription.
3. A network partition occurs between the mirror node and Redis (e.g., firewall rule drop, cloud VPC routing failure, or a targeted network-layer disruption). The TCP connection enters half-open state.
4. Redis publishes new `TopicMessage` events; they are never delivered to the mirror node.
5. The shared `Flux` emits nothing. No error is raised. The `retryWhen` never fires.
6. All gRPC subscribers sharing that topic's `Flux` receive no messages. Their streams remain open (no gRPC error status sent). They cannot distinguish this from a legitimate quiet period on the topic.
7. The `topicMessages` map retains the dead `Flux` entry, so any new subscriber for the same topic also gets the dead stream.

### Impact Explanation
Every subscriber to any affected topic experiences complete, silent message loss for the duration of the partition. Because no error is propagated to the gRPC client, the subscriber has no programmatic way to detect the failure — it cannot distinguish a half-open partition from a topic with no activity. In a financial/consensus context (Hedera topic messages carry consensus-ordered transaction data), missing messages without notification is a high-severity data integrity issue. The shared `Flux` architecture (`topicMessages` map + `.share()`) means a single half-open connection silently affects all concurrent subscribers to that topic, not just one.

### Likelihood Explanation
TCP half-open states are a well-documented, routinely occurring condition in cloud and Kubernetes environments: rolling restarts of Redis pods, cloud load-balancer idle-timeout drops, network policy changes, and transient VPC routing failures all produce them. No attacker capability is required beyond the ability to influence network infrastructure (e.g., a cloud tenant with access to firewall rules, or simply waiting for an infrastructure event). The absence of any OS-level TCP keepalive configuration in the gRPC Redis connection means the half-open socket can persist for hours (Linux default `tcp_keepalive_time` is 7200 s, and keepalive is not even enabled here). The condition is repeatable and deterministic once the partition is in place.

### Recommendation
Apply all of the following:

1. **Add a `.timeout()` operator** to the `subscribe()` pipeline to detect prolonged silence. For a pub/sub channel that should receive periodic messages, a configurable idle timeout (e.g., `listenerProperties.getMaxIdleInterval()`) should trigger an error that the `retryWhen` can then handle:
   ```java
   .flatMapMany(r -> r.receive(...))
   .timeout(maxIdleDuration)   // emits TimeoutException → triggers retryWhen
   ```

2. **Configure TCP keepalive on the Lettuce connection** in `grpc/src/main/java/org/hiero/mirror/grpc/config/RedisConfiguration.java`:
   ```java
   LettuceClientConfiguration clientConfig = LettuceClientConfiguration.builder()
       .clientOptions(ClientOptions.builder()
           .socketOptions(SocketOptions.builder()
               .keepAlive(KeepAliveOptions.builder().enable().build())
               .connectTimeout(Duration.ofSeconds(5))
               .build())
           .build())
       .build();
   ```

3. **Evict the dead `Flux` from `topicMessages`** on timeout/error before retry, so new subscribers do not inherit the stale entry.

### Proof of Concept
**Preconditions:** Mirror node running with `REDIS` listener type; at least one gRPC client subscribed to a topic; messages being published to Redis.

**Steps:**
1. Confirm messages are flowing to the gRPC subscriber.
2. On the host running the mirror node (or via a firewall/iptables rule), drop all TCP traffic to the Redis port without sending RST:
   ```bash
   iptables -I OUTPUT -p tcp --dport 6379 -j DROP
   iptables -I INPUT  -p tcp --sport 6379 -j DROP
   ```
3. Publish new `TopicMessage` events to Redis from another host (bypassing the drop rule).
4. Observe the gRPC subscriber: the stream remains open (no error, no completion), but no messages arrive.
5. Check mirror node logs: no error is logged for the Redis listener; `retryWhen` never fires.
6. The `topicMessages` map still holds the dead `Flux` entry; a new subscriber connecting now also receives nothing.
7. Restore connectivity (`iptables -F`): the stream remains silent because the half-open socket is never detected and no reconnect occurs.
