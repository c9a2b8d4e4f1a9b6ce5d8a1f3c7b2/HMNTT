### Title
Unauthenticated High-Frequency Polling of `/api/v1/network/stake` Exhausts HikariCP Database Connection Pool

### Summary
The `GET /api/v1/network/stake` endpoint in the `rest-java` module invokes `getLatestNetworkStake()`, which unconditionally executes a correlated subquery against the database on every request with no caching, no rate limiting, and no authentication. An unprivileged attacker flooding this endpoint at high frequency can saturate the HikariCP connection pool, causing all subsequent database-dependent requests across the service to queue and time out, denying legitimate clients access to stake data required for constructing valid staking transactions.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 126–130) → `NetworkService.getLatestNetworkStake()` (line 52–56) → `NetworkStakeRepository.findLatest()` (line 12–19). [1](#0-0) [2](#0-1) 

Every call to `findLatest()` executes the following native correlated subquery directly against the database:

```sql
select * from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
``` [3](#0-2) 

**Root cause — three absent defenses:**

1. **No caching.** A search across all `rest-java` main sources finds zero `@Cacheable` annotations or `CacheManager` beans. The `grpc` module caches node-stake data via `NODE_STAKE_CACHE`, but the `rest-java` `NetworkStakeRepository` has no equivalent. [4](#0-3) 

2. **No rate limiting.** The `ThrottleConfiguration` and `ThrottleManagerImpl` with bucket4j rate limiting exist exclusively in the `web3` module. The `rest-java` module contains no analogous throttle infrastructure for any of its endpoints. [5](#0-4) 

3. **No authentication.** The endpoint is publicly mapped with `@GetMapping("/stake")` and no security annotation or filter guards it. [1](#0-0) 

**Why existing checks fail:** The only throttling in the codebase is in `web3/ThrottleManagerImpl`, which is scoped to `ContractCallRequest` objects and is not wired into the `rest-java` Spring context at all. There is no servlet filter, interceptor, or Spring Security rule in `rest-java/config/` that limits request rate to this endpoint. [6](#0-5) 

### Impact Explanation

Each concurrent request to `/api/v1/network/stake` acquires a HikariCP connection for the duration of the correlated subquery. The Grafana dashboard for `rest-java` explicitly monitors `hikaricp_connections_active`, `hikaricp_connections_idle`, and `hikaricp_connections_pending`, confirming HikariCP is the connection pool in use. [7](#0-6) 

When the pool is exhausted, all other endpoints sharing the same datasource (supply, nodes, exchange rate, fee schedule) also begin queuing and timing out. Clients constructing staking transactions cannot retrieve the `staking_reward_rate`, `stake_total`, and related fields they need, directly impairing the ability to build valid staking transactions on the network.

### Likelihood Explanation

The endpoint requires no credentials, no API key, and accepts no parameters — a single `curl` loop or any HTTP benchmarking tool (`wrk`, `ab`, `hey`) run from a single machine is sufficient. The attack is stateless and trivially parallelizable across multiple source IPs. Because the query is a correlated subquery (two sequential table scans per call rather than a single indexed lookup), each request holds a connection slightly longer than a simple primary-key fetch, amplifying pool pressure per unit of attacker bandwidth.

### Recommendation

Apply at least two of the following mitigations:

1. **Add response caching** at the service or repository layer. The `network_stake` table is updated at most once per staking period (24 hours). A `@Cacheable` annotation with a TTL of several minutes (matching the staking period cadence) would reduce database hits to near zero under flood conditions.

2. **Add rate limiting** in `rest-java` mirroring the `web3` module's `ThrottleConfiguration`. A per-IP or global bucket4j rate limiter applied via a `HandlerInterceptor` or `OncePerRequestFilter` registered in `WebMvcConfiguration` would cap request throughput.

3. **Set a HikariCP connection timeout and maximum pool size** explicitly in `rest-java`'s datasource configuration so that pool exhaustion causes fast 503 responses rather than indefinite thread blocking.

### Proof of Concept

```bash
# Flood the endpoint from a single unauthenticated client
# Replace <host> with the mirror node rest-java host
wrk -t 16 -c 200 -d 60s http://<host>/api/v1/network/stake

# Expected result after pool exhaustion:
# - /api/v1/network/stake returns 500 / connection timeout errors
# - /api/v1/network/supply, /api/v1/network/nodes, etc. also begin failing
# - hikaricp_connections_pending metric spikes to pool maximum
# - Legitimate staking clients cannot retrieve stake data
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
