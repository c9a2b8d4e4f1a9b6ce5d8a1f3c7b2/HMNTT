### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Timestamp Query on GET /api/v1/blocks

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts a `timestamp=gte:0` filter with no upper bound and no rate limiting, issuing a DB query that holds a connection for its full duration. Because the REST API has no request throttling and the default DB pool is only 10 connections, an unauthenticated attacker flooding concurrent requests exhausts the pool, causing all subsequent legitimate requests to queue and eventually time out.

### Finding Description

**Code path:**

`getBlocks()` in `rest/controllers/blockController.js` (lines 101–112) calls `utils.buildAndValidateFilters`, then `extractSqlFromBlockFilters`, then `RecordFileService.getBlocks(formattedFilters)`. [1](#0-0) 

`extractSqlFromBlockFilters` (lines 63–88) maps a `timestamp=gte:0` filter directly to a `WHERE consensus_end >= $1` clause with no upper-bound enforcement. [2](#0-1) 

`RecordFileService.getBlocks()` (lines 149–162) appends the WHERE clause verbatim and issues the query via `super.getRows()`, which calls `this.pool().queryQuietly(...)`, holding a pool connection for the full query duration. [3](#0-2) 

**Root cause — three compounding failures:**

1. **No timestamp range validation on the blocks endpoint.** Other endpoints call `utils.parseTimestampFilters()` (lines 1583–1681 of `rest/utils.js`), which enforces `maxTimestampRange` (default 7 days) and rejects open-ended ranges. The blocks controller never calls this function; it passes the raw filter directly to SQL. [4](#0-3) 

2. **No rate limiting on the REST API.** `rest/server.js` registers only `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response-cache middleware — no throttle/rate-limit middleware. [5](#0-4) 

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module, not in the Node.js REST module. [6](#0-5) 

3. **Small default DB pool.** `hiero.mirror.rest.db.pool.maxConnections` defaults to **10**; `connectionTimeout` is 20 000 ms and `statementTimeout` is 20 000 ms. [7](#0-6) 

The pool is initialised in `rest/dbpool.js` directly from these values. [8](#0-7) 

**Why the limit cap does not help.** `extractLimitFromFilters` (lines 57–61) caps the limit at `maxLimit` (100). This bounds the result set size but does not shorten query execution time or prevent a connection from being held while the query runs. [9](#0-8) 

### Impact Explanation
With only 10 pool connections and a 20-second statement timeout, 10 concurrent slow or moderately-paced requests fully occupy the pool. All subsequent requests queue for up to 20 seconds (`connectionTimeout`) before returning a 500 error. This renders the entire REST API — not just the blocks endpoint — unavailable to legitimate users for the duration of the attack, constituting a complete non-network DoS of the mirror-node REST service.

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. The attack is trivially scriptable with any HTTP load tool (`ab`, `wrk`, `curl` in a loop). The attacker needs only a standard internet connection and knowledge of the public API. The attack is repeatable and stateless — each wave of 10–20 concurrent requests re-exhausts the pool. The default pool size of 10 makes the threshold extremely low.

### Recommendation
1. **Enforce an upper timestamp bound on the blocks endpoint.** Call `utils.parseTimestampFilters()` (with `allowOpenRange: false`) on any timestamp filters before building the SQL, consistent with how `contractController.js` handles topics.
2. **Add rate limiting to the REST API.** Introduce a per-IP or global request-rate middleware (e.g., `express-rate-limit`) in `rest/server.js`, analogous to the `bucket4j`-based throttle in the web3 module.
3. **Increase the default pool size** or deploy a connection pooler (PgBouncer is already referenced in the Helm values) in front of the REST service to absorb burst traffic.
4. **Reduce `statementTimeout`** for the blocks query path to limit how long each connection is held under adversarial load.

### Proof of Concept
```bash
# Exhaust the 10-connection pool with 20 concurrent requests
for i in $(seq 1 20); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks?timestamp=gte:0&limit=100" &
done
wait

# Legitimate request now times out or returns 500
curl -v "http://<mirror-node-host>:5551/api/v1/blocks"
```

**Expected result:** The 11th–20th concurrent requests queue for ~20 seconds (connectionTimeout). Any legitimate request arriving during this window receives a 500/connection-timeout error, confirming full pool exhaustion.

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
