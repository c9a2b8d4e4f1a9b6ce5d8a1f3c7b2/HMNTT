### Title
False-404 Proxy Cache Poisoning via Partition-Induced Empty DB Result in `getScheduleById`

### Summary
`getScheduleById` unconditionally throws `NotFoundError` when `pool.queryQuietly` returns zero rows, with no distinction between a genuinely absent schedule and a transient empty result caused by a network partition or replica lag. The error handler (`handleError`) emits the 404 response without any `Cache-Control: no-store` directive, leaving intermediate proxies free to cache the false-404 per RFC 7234 default cacheability rules. Any unprivileged user whose request coincides with the partition window can cause that cached false-404 to be served to all subsequent requesters until the proxy TTL expires.

### Finding Description

**Code path:**

`rest/schedules.js`, `getScheduleById`, lines 120–123:
```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
if (rows.length !== 1) {
  throw new NotFoundError();
}
``` [1](#0-0) 

There is no retry, no fallback to a primary replica, and no distinction between "zero rows because the schedule does not exist" and "zero rows because the replica is partitioned/lagging."

The thrown `NotFoundError` is caught by `handleError` in `rest/middleware/httpErrorHandler.js`:
```js
if (err instanceof NotFoundError) {
  statusCode = httpStatusCodes.NOT_FOUND;
}
...
res.status(statusCode.code).json(errorMessageFormat(errorMessage));
``` [2](#0-1) [3](#0-2) 

`handleError` sets **no `Cache-Control` header** on the 404 response. The successful-response path (`responseHandler.js`) that applies `headers.default` (which includes `cache-control: public, max-age=…`) is never reached for error responses.

The internal Redis cache guard in `responseCacheUpdateHandler` correctly skips non-2xx responses:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
``` [4](#0-3) 

This protects the internal Redis layer but does nothing to prevent an **intermediate proxy** (CDN, Nginx, Varnish, Cloudflare, etc.) from caching the 404. RFC 7234 §3 permits caching of 404 responses by default when no explicit `no-store` or `no-cache` directive is present. The 404 response emitted here carries no such directive.

**Root cause / failed assumption:** The code assumes `rows.length === 0` is semantically equivalent to "schedule does not exist." Under a network partition or replica lag, the DB returns an empty result set for a schedule that is present on the primary, making this assumption false.

### Impact Explanation

Any intermediate proxy that caches the false-404 will serve it to every subsequent requester for that schedule ID until the proxy's TTL expires. Depending on the proxy configuration, this TTL can range from seconds to hours. During that window:
- All clients (including legitimate dApps, wallets, and monitoring systems) receive a permanent-looking 404 for a valid, live schedule.
- Schedule execution monitoring and dependent workflows break silently.
- The `LONGER_SCHEDULE_CACHE_CONTROL_HEADER` (`public, max-age=3600`) applied to *successful* responses for executed/expired schedules shows the codebase is aware of and intentionally uses long-lived caching — making the absence of `no-store` on error responses a meaningful gap, not an oversight in an otherwise cache-unaware system. [5](#0-4) 

### Likelihood Explanation

- **No privilege required.** Any unauthenticated HTTP client can issue `GET /api/v1/schedules/{id}`.
- **Network partitions and replica lag are realistic.** PostgreSQL streaming replication, cloud-managed DB replicas, and connection pool exhaustion all produce transient empty result sets on replicas while the primary holds the data.
- **Intermediate proxies are standard in production.** CDNs and reverse proxies are routinely placed in front of mirror-node REST APIs.
- **Attacker does not need to cause the partition.** They only need to issue a request during the (potentially brief) window when the replica is partitioned. Automated polling or a simple retry loop makes this trivially repeatable.
- **The window can be extended.** If the attacker can observe or influence replica lag (e.g., by flooding the replica with heavy queries), the partition window widens.

### Recommendation

1. **Add `Cache-Control: no-store` to all error responses** in `handleError` (`rest/middleware/httpErrorHandler.js`):
   ```js
   res.set('Cache-Control', 'no-store');
   res.status(statusCode.code).json(errorMessageFormat(errorMessage));
   ```
   This is the minimal, highest-priority fix. It prevents any intermediate proxy from caching transient error responses regardless of the cause.

2. **Distinguish transient DB failures from genuine absence.** `pool.queryQuietly` swallows errors silently (by design). Consider checking whether the query returned an error/exception vs. a genuine empty result set, and returning `503 Service Unavailable` (already handled for `DbError`) rather than `404` when the DB layer signals a connectivity problem.

3. **Add a retry against the primary** before emitting 404 for the by-ID lookup, consistent with how other endpoints handle replica lag.

### Proof of Concept

1. Deploy the mirror-node REST API behind any caching proxy (e.g., Nginx `proxy_cache`, Varnish, or a CDN with default 404-caching enabled).
2. Confirm schedule `0.0.2000` exists and `GET /api/v1/schedules/0.0.2000` returns 200.
3. Simulate a network partition on the read replica (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the replica host, or pause the replica process).
4. Issue `GET /api/v1/schedules/0.0.2000` through the proxy. The mirror-node queries the partitioned replica, gets 0 rows, throws `NotFoundError`, and returns HTTP 404 with **no `Cache-Control` header**.
5. The proxy caches the 404 (default behavior per RFC 7234 for responses without `no-store`).
6. Restore the replica. Confirm `GET /api/v1/schedules/0.0.2000` directly against the mirror-node now returns 200.
7. Issue `GET /api/v1/schedules/0.0.2000` through the proxy again — the proxy continues to serve the cached 404 until its TTL expires, despite the schedule existing and the backend being healthy.
