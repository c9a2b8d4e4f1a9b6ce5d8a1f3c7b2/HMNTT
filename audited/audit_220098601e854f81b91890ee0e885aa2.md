### Title
Unbounded Cache Pollution via Unauthenticated Unique GraphQL Query Flooding in `CachedPreparsedDocumentProvider`

### Summary
The `CachedPreparsedDocumentProvider.getDocumentAsync()` caches parsed GraphQL documents keyed solely on the raw query string, with a global `maximumSize=1000` and no per-user, per-IP, or rate-limiting controls. An unauthenticated attacker can submit 1000 syntactically distinct but semantically trivial queries to fill the entire cache, evicting legitimate entries and forcing re-parsing overhead for all concurrent users. No privilege is required to trigger this condition.

### Finding Description
**Exact code path:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
  ```java
  return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
  ```
  The cache key is the raw query string with no normalization, no per-caller quota, and no admission guard.

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
  ```java
  private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
  ```
  The cache is globally bounded at 1000 entries.

**Root cause:** The cache is a single shared, globally-writable `AsyncCache<String, PreparsedDocumentEntry>` with no caller identity, no per-IP slot limit, and no admission policy beyond Caffeine's W-TinyLFU frequency heuristic. On a cold or recently-expired cache, any caller can freely populate all 1000 slots.

**Exploit flow:**
1. Attacker generates 1000 unique but syntactically valid GraphQL queries (e.g., varying field aliases, argument names within schema bounds, or whitespace variations — all within the `maxCharacters=10000`, `maxTokens=1000` parser limits).
2. Each query is a cache miss; Caffeine inserts all 1000 entries, filling the cache.
3. Legitimate queries that were previously cached are evicted by Caffeine's size-based eviction.
4. Attacker periodically re-sends the same 1000 queries to maintain their frequency counters above those of legitimate queries, sustaining the eviction pressure under W-TinyLFU.

**Why existing checks fail:**
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — enforce per-query structural limits but do not prevent 1000 distinct low-complexity queries.
- `ParserOptions` limits (`maxCharacters=10000`, `maxTokens=1000`, `maxRuleDepth=100`, `maxWhitespaceTokens=1000`) — bound individual query size, not the number of unique queries submitted.
- `Jackson` stream constraints — bound JSON payload size, not query uniqueness.
- `LoggingFilter` — logs only; no throttling or blocking.
- No Spring Security configuration, no authentication, no rate-limiting filter exists anywhere in the `graphql` module's main source tree.

### Impact Explanation
Legitimate users experience increased latency on every cache miss because the GraphQL-Java parser and validator must re-execute for each evicted query. Under sustained attack, the cache provides no benefit to legitimate traffic. The impact is a griefing/degradation-of-service with no economic damage, no data exfiltration, and no crash — consistent with the stated scope of "Medium: Impacts caused by griefing with no economic damage to any user on the network."

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public GraphQL schema, and no sophisticated tooling — a simple script generating 1000 queries with varying aliases or argument values suffices. The endpoint (`/graphql/alpha`) is publicly documented. The attack is repeatable indefinitely and can be automated. The only friction is the `expireAfterWrite=1h` TTL, which means the attacker must re-flood every hour to sustain the effect, but this is trivially scriptable.

### Recommendation
1. **Per-IP or per-session cache admission quota**: Track how many distinct query strings a single IP has contributed to the cache within a time window and reject new cache insertions beyond a threshold (e.g., 10–20 unique queries per IP per hour).
2. **Query normalization before caching**: Normalize queries (strip comments, canonicalize whitespace, sort fields) before using them as cache keys, reducing the attacker's ability to generate unique keys from semantically identical queries.
3. **Rate limiting at the HTTP layer**: Deploy a rate-limiting filter or API gateway rule that caps the number of requests per IP per time window before they reach the GraphQL execution layer.
4. **Increase cache size or use a segmented cache**: Separate a "pinned" segment for known-good queries from a general segment, so attacker-controlled entries cannot evict operator-pinned entries.

### Proof of Concept
```bash
# Generate 1000 unique valid queries by varying the alias name
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ alias${i}: account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }\"}" \
    &
done
wait

# Now legitimate queries (e.g., the standard account query) are no longer cached.
# Each subsequent legitimate request incurs full parse+validate overhead.
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }"}'
# Observe: cache miss, re-parsing occurs.
```

Each alias variation (`alias1`, `alias2`, … `alias1000`) produces a syntactically valid query within the `maxTokens=1000` and `maxCharacters=10000` limits, is semantically valid against the schema, and maps to a unique cache key string, filling all 1000 slots.
