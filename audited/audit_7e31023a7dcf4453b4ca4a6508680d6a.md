### Title
Unauthenticated High-Frequency Timestamp-Range Flooding of `/contracts/results` Causes Sustained Database Load via Absent Rate Limiting and Disabled Range Enforcement

### Summary
The `getContractResults()` handler at `GET /api/v1/contracts/results` accepts arbitrary timestamp range filters from any unauthenticated caller, triggers three concurrent database queries per request, and has no per-IP rate limiting in the REST layer. The only range-bounding mechanism (`bindTimestampRange`) is disabled by default, and the internal call path explicitly bypasses the `maxTimestampRange` validation by passing `validateRange=false`. An attacker can sustain high-frequency requests with sliding timestamp windows to continuously saturate the shared PostgreSQL connection pool, degrading the mirror node's ability to ingest and index gossip data.

### Impact Explanation
Each unauthenticated request to `/api/v1/contracts/results` with a wide or open-ended timestamp range (e.g., `timestamp=gte:0`) causes three concurrent SQL queries against the shared PostgreSQL instance. The mirror node's gossip ingestion pipeline writes to the same database. Sustained flooding exhausts the DB connection pool (`db.pool.maxConnections`), increases query latency for the importer, and can stall record-file ingestion ã directly degrading the node's ability to keep up with the gossip stream. Because the endpoint returns paginated public data, the attacker gains full enumeration of all contract transaction history as a side effect.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. Any internet-accessible mirror node is reachable. The attack is trivially scriptable with a loop of `curl` or `ab` commands using sliding `timestamp=gte:X&timestamp=lte:Y` windows. The absence of per-IP rate limiting means a single client can sustain the load indefinitely. The default-off `bindTimestampRange` means most production deployments are affected unless operators have explicitly enabled it.

### Recommendation
1. **Enable `bindTimestampRange: true`** in production configuration to enforce `maxTransactionsTimestampRangeNs` on open/large ranges.
2. **Pass `validateRange = true`** in the `optimizeTimestampFilters` call to `parseTimestampFilters`, or enforce `maxTimestampRange` before the DB query is issued.
3. **Add per-IP rate limiting middleware** (e.g., `express-rate-limit`) to the REST API router, applied globally or specifically to the `/contracts/results` route.
4. **Require at least one bounded timestamp filter** (both lower and upper bound) when `bindTimestampRange` is disabled, rejecting open-ended range queries.

### Proof of Concept
```bash
# Flood with sliding 1-second windows across the full history, no auth needed
for i in $(seq 0 1000000 2000000000); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?timestamp=gte:${i}.000000000&timestamp=lte:$((i+1000000)).000000000&limit=100" &
done
wait
```
Each iteration fires three DB queries. With 50+ concurrent requests the DB connection pool saturates, importer write latency rises, and gossip record-file ingestion stalls.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
};
