### Title
Long Overflow in `NumberRangeParameter.getInclusiveValue()` Bypasses `hook.id=gt:` Filter, Disclosing All Hooks

### Summary
When an unprivileged caller supplies `hook.id=gt:9223372036854775807` (`Long.MAX_VALUE`), `getInclusiveValue()` computes `Long.MAX_VALUE + 1`, which silently overflows to `Long.MIN_VALUE` in Java. `Math.max(0L, Long.MIN_VALUE)` then leaves `lowerBound` at `0`, so the service issues `findByOwnerIdAndHookIdBetween(ownerId, 0, Long.MAX_VALUE, page)` — identical to an unfiltered query — and returns all hooks for the target owner instead of zero results.

### Finding Description

**Exact code path:**

`NumberRangeParameter.valueOf("gt:9223372036854775807")` calls `getNumberValue("9223372036854775807")`: [1](#0-0) 

`Long.parseLong("9223372036854775807")` = `Long.MAX_VALUE`. The only guard is `value < 0`; `Long.MAX_VALUE` is positive, so it passes.

`getInclusiveValue()` for `GT`: [2](#0-1) 

`Long.MAX_VALUE + 1` overflows to `Long.MIN_VALUE` (-9223372036854775808). No exception is thrown; Java arithmetic overflow is silent.

Back in `hooksRequest()`: [3](#0-2) 

`Math.max(0L, Long.MIN_VALUE)` = `0L`. `lowerBound` is never updated from its initial value of `0`.

The resulting `HooksRequest` carries `lowerBound=0`, `upperBound=Long.MAX_VALUE`: [4](#0-3) 

`HookServiceImpl.getHooks()` then executes: [5](#0-4) 

`findByOwnerIdAndHookIdBetween(ownerId, 0, Long.MAX_VALUE, page)` — a full table scan for the owner, returning all hooks up to `limit`.

**Why existing checks are insufficient:**
- `getNumberValue` only rejects `value < 0`; `Long.MAX_VALUE` is a valid positive long.
- There is no `@Max` or range constraint on the parsed `value` inside `NumberRangeParameter`.
- There is no overflow guard in `getInclusiveValue()` before performing `value + 1` or `value - 1`.
- The controller applies `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` only to array length, not to individual parameter values.
- The endpoint carries no `@PreAuthorize` or authentication requirement; it is publicly reachable. [6](#0-5) 

### Impact Explanation
Any unauthenticated caller who knows (or can enumerate) a valid `ownerId` can retrieve the complete list of that owner's gossip-prevention hooks — up to `MAX_LIMIT` entries per request, pageable via `links.next`. This is a direct information-disclosure of hook configurations that are explicitly intended to be private (they control which transactions are suppressed from gossip). The attacker learns hook IDs, types, and extension points without any authorization.

### Likelihood Explanation
The exploit requires zero privileges: a single crafted HTTP GET request with one query parameter. `ownerId` values are account IDs that are publicly visible on-chain. The attack is trivially repeatable, scriptable, and leaves no distinguishing trace beyond a normal API call. No rate-limiting specific to this endpoint is visible in the controller.

### Recommendation
1. **Guard against overflow in `getInclusiveValue()`**: before returning `value + 1` for `GT`, check `if (value == Long.MAX_VALUE) throw new IllegalArgumentException("hook.id value out of range for GT operator")`. Apply the symmetric check for `LT` with `Long.MIN_VALUE`.
2. **Add an upper-bound validator in `getNumberValue()`**: reject values equal to `Long.MAX_VALUE` (or define a domain-specific maximum hook ID) so the overflow path is never reached.
3. **Add a `@Max(Long.MAX_VALUE - 1)` or equivalent Bean Validation constraint** on the `value` field of `NumberRangeParameter` to enforce this at the framework level.

### Proof of Concept

```
# Precondition: account 0.0.123 has hooks registered
GET /api/v1/accounts/0.0.123/hooks?hook.id=gt:9223372036854775807

# Expected: 200 OK with empty hooks array []
# Actual:   200 OK with all hooks for owner 0.0.123
```

Step-by-step:
1. `NumberRangeParameter.valueOf("gt:9223372036854775807")` → `operator=GT, value=9223372036854775807`
2. `getInclusiveValue()` → `9223372036854775807 + 1` = `-9223372036854775808` (overflow)
3. `Math.max(0L, -9223372036854775808L)` = `0L` → `lowerBound = 0`
4. `findByOwnerIdAndHookIdBetween(ownerId, 0, 9223372036854775807, page)` executes
5. All hooks for the owner are returned in the response
