### Title
Redis Partition Fail-Open in `responseCacheCheckHandler` Enables Unauthenticated DB Load Amplification

### Summary
When Redis becomes unreachable and `this.ready` transitions to `false` via the `retryStrategy` callback, `getSingleWithTtl` immediately returns `undefined` on every call, causing `responseCacheCheckHandler` to treat every inbound request as a cache miss and forward it to the database. Any unprivileged external user can exploit this deterministic fail-open behavior by flooding the API during a Redis partition, amplifying database load by a factor proportional to the normal cache-hit ratio. A secondary bug causes a `TypeError` crash during the brief transition window before `ready` flips.

### Finding Description

**Phase 1 — Transition window (`ready` still `true`, Redis unreachable):**

`rest/cache.js` lines 64–74:
```js
async getSingleWithTtl(key) {
  if (!this.ready) { return undefined; }          // passes — ready is still true
  const result = await this.redis
    .multi().ttl(key).get(key).exec()
    .catch((err) => logger.warn(`Redis error during ttl/get: ${err.message}`));
                                                   // catch returns undefined (logger.warn return value)
  const rawValue = result[1][1];                   // TypeError: Cannot read properties of undefined
```

`logger.warn` returns `undefined`; `result` is therefore `undefined`; `result[1][1]` throws a `TypeError`. `extendExpress.js` (lines 42–43) catches the rejection and calls `next(err)`, routing to the error handler → **HTTP 500 for every request**.

**Phase 2 — Sustained partition (`retryStrategy` has fired, `ready = false`):**

`rest/cache.js` lines 30–31:
```js
retryStrategy: (attempt) => {
  this.ready = false;   // set asynchronously on first reconnect attempt
```

Once `ready` is `false`, `getSingleWithTtl` returns `undefined` at line 66 without touching Redis. Back in `responseCacheCheckHandler` (`rest/middleware/responseCacheHandler.js` lines 45–47):
```js
if (!cachedTtlAndValue) {
  res.locals[responseCacheKeyLabel] = responseCacheKey;
  return;                // treated as cache miss — request proceeds to DB
}
```

Every request — including those that would normally be served entirely from Redis — now hits the database. There is no rate-limiting, no circuit-breaker, and no back-pressure mechanism between the cache miss path and the DB query path.

**Root cause:** The `ready` flag is the sole guard, but it is set asynchronously by ioredis's reconnect machinery. During Phase 2 the flag correctly reflects the partition, but the design choice to treat `ready=false` as a transparent cache miss (rather than returning a degraded response or shedding load) is the failed assumption.

### Impact Explanation
If the system normally operates at, say, 1 000 req/s with an 80 % cache-hit rate, the database handles ~200 req/s under normal conditions. During a Redis partition every request reaches the database: 1 000 req/s — a 5× amplification with zero attacker effort beyond sending normal API requests. An attacker who deliberately floods the API during the partition (or who triggers the partition via a separate network-layer attack) can push this to any multiple of the normal DB query rate, potentially causing database exhaustion, query queue saturation, or cascading service failure. Severity: **High** (availability impact on the database tier, no authentication required).

### Likelihood Explanation
Redis network partitions occur naturally (cloud provider blips, rolling restarts, misconfiguration). An attacker does not need to cause the partition; they only need to detect it (observable via increased response latency or 500 errors during Phase 1) and then send a sustained burst of requests. The exploit requires no credentials, no special headers, and no knowledge beyond the public API surface. It is fully repeatable whenever a partition occurs.

### Recommendation
1. **Do not fail open silently.** When `ready` is `false` (or when `getSingleWithTtl` catches a Redis error), return a `503 Service Unavailable` or apply a configurable load-shedding policy rather than transparently forwarding to the DB.
2. **Fix the TypeError.** Guard `result` before indexing: `if (!result) { return undefined; }` immediately after the `.catch` in `getSingleWithTtl` (`rest/cache.js` line 76).
3. **Add a circuit breaker / request rate cap** on the DB path that activates when the cache is unavailable, preventing unbounded DB amplification.
4. **Decouple `ready` transitions from command failures.** Track consecutive command failures independently of the reconnect strategy so that in-flight requests during the transition window are handled gracefully rather than crashing with a TypeError.

### Proof of Concept
1. Start the mirror-node REST service with Redis enabled and a populated cache.
2. Sever the network path between the REST service and Redis (e.g., `iptables -I OUTPUT -p tcp --dport 6379 -j DROP` on the app host, or stop the Redis container).
3. Immediately send a burst of requests to any cached endpoint (e.g., `ab -n 5000 -c 50 http://<host>/api/v1/accounts`).
4. **Phase 1 (first few seconds):** Observe HTTP 500 responses as `result[1][1]` throws TypeError.
5. **Phase 2 (after ioredis fires `retryStrategy`):** Observe HTTP 200 responses with full DB latency; confirm in DB slow-query logs or metrics that every request generates a DB query — none are served from cache.
6. Monitor DB connection pool / query rate: it will match the full inbound request rate rather than the normal cache-miss rate, demonstrating the amplification.
