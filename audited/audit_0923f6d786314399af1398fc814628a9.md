### Title
ShallowEtagHeaderFilter Unbounded Response Buffering Without Rate Limiting Enables Memory-Amplification DoS

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers a `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no per-request size cap and no application-level rate limiting in the rest-java module. The filter buffers the entire serialized response body in a JVM heap byte array before computing the ETag hash, meaning every concurrent in-flight request holds its full response in memory simultaneously. An unprivileged attacker who floods the server with concurrent requests to high-cardinality endpoints (e.g., `/api/v1/network/nodes`) can force hundreds of large response buffers to coexist in heap, amplifying memory pressure and degrading or crashing the service.

### Finding Description
**Exact location:** `rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46.

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause:** Spring's `ShallowEtagHeaderFilter` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper`. The entire response body written by the controller is accumulated in an in-memory `ByteArrayOutputStream`. Only after the full body is written does the filter compute the MD5-based ETag, optionally short-circuit with HTTP 304, and then flush the buffer to the real response. The buffer lives on the JVM heap for the entire duration of request processing plus response transmission. There is no `setWriteWeakETag`, no streaming path, and no size guard.

**No rate limiting in rest-java:** Unlike the `web3` module, which has an explicit `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j rate limiting, the rest-java module has zero application-level rate limiting. The configuration documentation (`docs/configuration.md`, lines 622–634) lists no rate-limiting properties for `hiero.mirror.restJava.*`. No `FilterRegistrationBean` for throttling exists in the rest-java config package.

**Exploit flow:**
1. Attacker identifies `/api/v1/network/nodes` (or any paginated endpoint with a high default/max limit). Each response for 100 nodes with multiple service endpoints can be 300–800 KB of JSON.
2. Attacker opens many concurrent HTTP connections (up to Tomcat's default `max-connections` of 8192; actively processed up to `max-threads` default of 200) and issues GET requests without `If-None-Match`.
3. For each request, `ShallowEtagHeaderFilter` allocates a `ByteArrayOutputStream` and the controller writes the full serialized response into it.
4. With 200 active threads × ~500 KB per response = ~100 MB of heap consumed by filter buffers alone, sustained indefinitely.
5. The pod memory limit is 2048 Mi (Helm chart `values.yaml`, line 308). Sustained attack drives heap toward the limit, triggering GC pressure, latency spikes, and ultimately OOM/pod restart.

### Impact Explanation
The pod memory limit is 2048 Mi. Sustained concurrent requests to large-response endpoints can consume 100–400 MB of heap in filter buffers alone, on top of normal application memory. This causes:
- Increased GC frequency and stop-the-world pauses, degrading response latency for all users.
- OOM kill and pod restart when the heap limit is reached, causing a complete service outage until the pod recovers.
- The `RestJavaHighMemory` alert fires at 80% usage (Grafana rule in `charts/hedera-mirror-common/alerts/rules.yaml`, line 1397), but the alert only triggers after 5 minutes — by which time the pod may already be OOM-killed.

### Likelihood Explanation
The attack requires no authentication, no special knowledge, and no privileged access — only the ability to send HTTP GET requests to a public API endpoint. The absence of any rate limiting in rest-java means the attacker can sustain the flood indefinitely from a single host or a small botnet. The attack is trivially scriptable (`ab`, `wrk`, `hey`, or a simple loop of `curl` commands). The `ShallowEtagHeaderFilter` behavior is well-documented and the memory amplification effect is predictable.

### Recommendation
1. **Add a response-size guard to the ETag filter** by subclassing `ShallowEtagHeaderFilter` and overriding `isEligibleForEtag()` to skip buffering when `Content-Length` exceeds a threshold (e.g., 256 KB), or use `setWriteWeakETag(true)` with a streaming-safe alternative.
2. **Add application-level rate limiting to rest-java** analogous to the `ThrottleConfiguration` in the `web3` module — a bucket4j filter registered before `etagFilter()` that limits requests per IP or globally.
3. **Set a Tomcat `max-threads` and `accept-count` limit** in `application.yml` to bound the number of concurrently buffered responses.
4. **Consider removing `ShallowEtagHeaderFilter` entirely** for large-payload list endpoints, or scope it only to single-resource endpoints where response sizes are small and bounded.

### Proof of Concept
```bash
# Send 200 concurrent requests to the nodes endpoint (large paginated response)
# No authentication required; no rate limiting in rest-java
seq 200 | xargs -P200 -I{} curl -s \
  "http://<rest-java-host>/api/v1/network/nodes?limit=100" \
  -o /dev/null

# Repeat in a loop to sustain memory pressure
while true; do
  seq 200 | xargs -P200 -I{} curl -s \
    "http://<rest-java-host>/api/v1/network/nodes?limit=100" \
    -o /dev/null
done
```
Each iteration forces up to 200 concurrent `ContentCachingResponseWrapper` byte-array buffers (~500 KB each) to coexist on the JVM heap. Monitor JVM heap via the `RestJavaHighMemory` Prometheus alert or `jvm_memory_used_bytes` metric; heap usage will climb steadily until GC cannot keep up or the pod is OOM-killed.
