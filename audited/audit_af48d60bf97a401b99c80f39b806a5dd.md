### Title
Unauthenticated Request Flood Exhausts DB Connection Pool via `/api/v1/accounts/:id/rewards` Endpoint

### Summary
The `listStakingRewardsByAccountId` handler is reachable by any unauthenticated external user with no rate limiting enforced at the application layer. Each request issues two sequential database queries (`EntityService.isValidAccount` + `StakingRewardTransferService.getRewards`) against a pool defaulting to only 10 connections with a 20-second statement timeout, making it trivial to exhaust the pool and deny service to all REST API consumers sharing those connections.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js` line 16 registers the handler with no authentication or rate-limiting guard: [1](#0-0) 

`rest/controllers/accountController.js` lines 170–185 show the handler unconditionally issuing two DB round-trips per request: [2](#0-1) 

`rest/service/entityService.js` lines 60–63 show `isValidAccount` executing a `SELECT type FROM entity WHERE id = $1` query: [3](#0-2) 

`rest/service/stakingRewardTransferService.js` lines 18–22 show `getRewards` executing a second query: [4](#0-3) 

**Authentication middleware is a no-op for unauthenticated callers.** `rest/middleware/authHandler.js` lines 18–20 simply return without blocking when no `Authorization` header is present: [5](#0-4) 

**No rate-limiting middleware exists in the REST API server.** `rest/server.js` shows the full middleware chain — `httpContext`, `requestLogger`, `authHandler`, optional metrics, optional response cache — with no throttle or in-flight-request limiter: [6](#0-5) 

The throttle mechanisms present in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belong exclusively to the **web3 Java** service and are not applied to the Node.js REST API. [7](#0-6) 

**The DB pool is tiny by default.** `rest/dbpool.js` line 14 sets `max` from `config.db.pool.maxConnections`, which defaults to **10**: [8](#0-7) [9](#0-8) 

The `statementTimeout` defaults to **20 000 ms**, so each connection can be held for up to 20 seconds per query: [10](#0-9) 

**Root cause:** The failed assumption is that infrastructure-level rate limiting (e.g., Traefik middleware) will always be present in front of the REST API. The Rosetta chart has explicit `inFlightReq` and `rateLimit` Traefik middleware: [11](#0-10) 

The REST API chart has no equivalent configuration, leaving the application entirely unprotected at the code level.

### Impact Explanation

With 10 pool connections and a 20-second statement timeout, an attacker sending ~10 concurrent long-running or repeated requests saturates the pool. All subsequent REST API requests queue indefinitely until `connectionTimeoutMillis` (default 20 000 ms) elapses, at which point they fail with a connection-timeout error. This denies service to all legitimate REST API consumers. While the importer (gossip processor) uses a separate DB user (`mirror_importer`) with its own pgbouncer session pool and is not directly starved, the shared PostgreSQL server is placed under elevated load, and the REST API — a critical public interface — becomes completely unavailable. Severity: **High** (REST API DoS, no authentication required, trivially repeatable).

### Likelihood Explanation

Preconditions: none. Any internet-accessible mirror node deployment is vulnerable. The attacker needs only a valid (or even invalid — the `isValidAccount` check still fires a DB query before returning 404) account ID or EVM address. A single machine with a modest HTTP client library can sustain hundreds of concurrent connections. The attack is stateless, requires no prior knowledge, and is repeatable indefinitely.

### Recommendation

1. **Add application-level rate limiting** to the Node.js REST API — e.g., `express-rate-limit` or a similar middleware — applied globally in `rest/server.js` before route handlers, with per-IP limits.
2. **Add an in-flight request cap** to prevent pool exhaustion regardless of rate: reject requests with HTTP 429 when the number of active DB-bound requests exceeds a configurable threshold.
3. **Increase the default pool size** (`hiero.mirror.rest.db.pool.maxConnections`) and document the relationship between pool size, concurrency, and statement timeout.
4. **Mirror the Traefik `inFlightReq`/`rateLimit` middleware** already present in the Rosetta chart into the `hedera-mirror-rest` chart so infrastructure-level protection is not optional.
5. Optionally, short-circuit the `isValidAccount` DB call for numeric account IDs that can be validated syntactically before hitting the database.

### Proof of Concept

```bash
# Flood the endpoint with 50 concurrent requests (no credentials needed)
seq 1 50 | xargs -P50 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node-host>/api/v1/accounts/0.0.1001/rewards"
```

Expected result: after the 10-connection pool is saturated, subsequent requests return HTTP 500 or hang until the 20-second connection timeout fires, effectively denying service to all other REST API consumers for the duration of the flood.
