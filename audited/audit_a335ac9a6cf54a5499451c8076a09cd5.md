### Title
DNS-Hijack of Mirror REST Hostname Causes Permanent Pinger Crash via Empty Node List

### Summary
When `network=other`, `newClient()` calls `buildNetworkFromMirrorNodes()` which makes an unauthenticated plain-HTTP GET to the operator-configured `mirrorRest` hostname. An attacker who controls DNS for that hostname can serve a crafted HTTP 200 response with no usable `service_endpoints`, triggering the non-retryable `"no usable service_endpoints found"` error path, which propagates to `log.Fatalf` in `main()` and permanently kills the monitoring process.

### Finding Description

**Code path:**

`pinger/sdk_client.go` lines 17–21: when `cfg.network == "other"`, `buildNetworkFromMirrorNodes` is called and any error is returned immediately to `main`. [1](#0-0) 

`pinger/mirror_node_client.go` lines 46–46: the HTTP client is constructed with no TLS, no client certificate, and no authentication — plain `http://` is the documented example. [2](#0-1) 

`pinger/mirror_node_client.go` lines 127–129: when the parsed response yields zero usable entries, `fetchMirrorNodeNetwork` returns `retry=false` and a fatal error. [3](#0-2) 

`pinger/mirror_node_client.go` lines 58–61: the retry loop breaks immediately when `retry=false`, so **zero retries** occur for this specific error — the entire retry budget is bypassed. [4](#0-3) 

`pinger/main.go` lines 41–44: the error from `newClient` is passed to `log.Fatalf`, which calls `os.Exit(1)` — the process dies permanently. [5](#0-4) 

**Root cause:** The code assumes the mirror REST endpoint is trusted and will always return valid node data. There is no TLS enforcement, no HMAC/signature verification of the response, and no fallback. The `retry=false` return for the empty-nodes case means the retry logic — the only resilience mechanism — is completely bypassed.

**Exploit flow:**
1. Attacker redirects DNS for the `mirrorRest` hostname to their server (DNS hijack, BGP hijack, or subdomain takeover if the hostname is public).
2. Attacker's server responds with HTTP 200 and a syntactically valid JSON body containing nodes with empty `service_endpoints`, e.g.:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[]}],"links":{"next":null}}
   ```
3. `fetchMirrorNodeNetwork` parses the body, finds `len(network) == 0`, returns `(nil, false, err)`.
4. The `!retry` branch fires immediately; `buildNetworkFromMirrorNodes` returns the error.
5. `newClient` returns the error; `main` calls `log.Fatalf` → `os.Exit(1)`.
6. The pinger is dead. No automatic recovery unless an external supervisor restarts it.

### Impact Explanation

The pinger is a monitoring/liveness service. Killing it at startup means:
- The `/tmp/ready` readiness file is never written, so Kubernetes readiness probes fail.
- The `/tmp/alive` liveness heartbeat never starts, so liveness probes eventually evict the pod.
- Depending on restart policy and whether the attacker sustains the DNS redirect, the pod can be kept in a crash-loop indefinitely, blinding operators to network health. [6](#0-5) 

### Likelihood Explanation

The precondition (`network=other`) is explicitly the deployment mode for custom/private networks — exactly the environments where operators are most likely to use a named internal hostname for `mirrorRest`. DNS control is achievable via: (a) subdomain takeover if the hostname is a public FQDN whose DNS record has been removed, (b) DNS poisoning of an unvalidated resolver, or (c) BGP prefix hijack for the resolver's upstream. The attack requires only a single HTTP response (no retry protection), is repeatable on every restart, and leaves no cryptographic evidence.

### Recommendation

1. **Enforce HTTPS with certificate validation** for the `mirrorRest` URL; reject plain `http://` at config load time.
2. **Do not `log.Fatalf` on empty-node responses at startup.** Treat it as a retryable transient error (set `retry=true`) and apply the full backoff/retry budget before giving up.
3. **Pin or validate the mirror node list** against a locally configured set of expected account IDs or a checksum, so a rogue server cannot inject an empty or adversarial list.
4. **Add a minimum-nodes threshold** check: if the returned node count is below a configured floor, treat it as an error and retry rather than proceeding (or crashing). [3](#0-2) 

### Proof of Concept

```
# 1. Deploy pinger with:
#    HIERO_MIRROR_PINGER_NETWORK=other
#    HIERO_MIRROR_PINGER_REST=http://mirror-rest.example.com:5551

# 2. Attacker controls DNS for mirror-rest.example.com
#    → points it to attacker-controlled server

# 3. Attacker server (e.g. netcat or nginx) responds:
HTTP/1.1 200 OK
Content-Type: application/json

{"nodes":[{"node_account_id":"0.0.3","service_endpoints":[]}],"links":{"next":null}}

# 4. Pinger startup output:
# 2024/xx/xx xx:xx:xx client error: attempt 1/1: no usable service_endpoints found from http://mirror-rest.example.com:5551/api/v1/network/nodes
# exit status 1

# 5. /tmp/ready is never created; readiness probe fails; pod never becomes Ready.
#    If attacker sustains the redirect, every restart crashes identically.
``` [7](#0-6)
