### Title
Base32 Non-Canonical Trailing-Bit Aliasing in `getByAliasAndType()` Allows Multiple Distinct Alias Strings to Resolve to the Same Entity

### Summary
`getByAliasAndType()` in `EntityServiceImpl.java` decodes the user-supplied alias string using Apache Commons Codec's `Base32.decode()`, which silently ignores unused trailing bits in the final base32 character. Because the only input gate is a regex pattern `^[A-Z2-7]+$` that validates character set but not canonical form, an unprivileged attacker can submit multiple syntactically distinct alias strings that all decode to the same byte sequence, causing the same entity to be returned for each. This leaks the equivalence of those aliases and enables low-cost entity probing.

### Finding Description

**Code path:**

1. GraphQL schema enforces `@Pattern(regexp: "^[A-Z2-7]+$")` on the `alias` field. [1](#0-0) 

2. `AccountController.account()` passes the raw alias string directly to `entityService.getByAliasAndType(alias, ...)`. [2](#0-1) 

3. `getByAliasAndType()` calls `decodeBase32(alias)` and passes the resulting bytes to the repository. [3](#0-2) 

4. `decodeBase32()` is a thin wrapper around `BASE32.decode(base32)` where `BASE32 = new Base32()` (Apache Commons Codec, default constructor). [4](#0-3) 

**Root cause — unused trailing bits in RFC 4648 Base32:**

RFC 4648 Base32 encodes 5 bits per character. When the input byte length is not a multiple of 5, the final character carries unused low-order bits that must be zero in canonical encoding. Apache Commons Codec's `Base32.decode()` does **not** validate these bits; it simply masks them away. This means multiple distinct strings, all composed of valid `[A-Z2-7]` characters, decode to the same byte array:

| Input length | Chars needed | Unused bits in last char | Distinct last chars that decode identically |
|---|---|---|---|
| 1 byte | 2 | 2 bits | 4 (e.g. `AA`, `AB`, `AC`, `AD`) |
| 2 bytes | 4 | 4 bits | 16 |
| 3 bytes | 5 | 1 bit | 2 |
| 4 bytes | 7 | 3 bits | 8 |

**Exploit flow:**

- Attacker knows (or guesses) a valid alias, e.g. `KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ` (53 chars → 4 unused bits in last char).
- Attacker submits the same alias with the last character replaced by each of the 16 base32 characters that share the same upper bits (all pass `^[A-Z2-7]+$`).
- All 16 queries return the same entity.
- Attacker has confirmed that 16 syntactically distinct alias strings are equivalent, leaking entity existence and alias canonicalization.

**Why existing checks are insufficient:**

The regex `^[A-Z2-7]+$` validates only that characters belong to the base32 alphabet. It does not enforce that the trailing bits of the last character are zero (canonical form). There is no post-decode re-encode-and-compare step to reject non-canonical inputs. [5](#0-4) 

### Impact Explanation
Any unprivileged user can confirm that up to 16 distinct alias strings resolve to the same entity without any additional knowledge beyond one valid alias. This leaks entity existence across multiple alias representations and can be used to probe whether a guessed alias (in any of its non-canonical forms) maps to a live entity. In a system where alias confidentiality matters (e.g., aliases derived from public keys), this amplifies the attacker's ability to confirm guesses.

### Likelihood Explanation
Exploitation requires zero privileges and zero authentication. The attacker only needs to know one valid alias (obtainable from any prior successful query or from public blockchain data). The trailing-bit variants are trivially computable: for a 53-character alias, the last character can be replaced with any of 16 valid base32 characters. The attack is deterministic, repeatable, and requires only standard HTTP POST requests to the GraphQL endpoint.

### Recommendation
After decoding the alias bytes, re-encode them to canonical base32 and compare against the original input. Reject any input that does not match its canonical form:

```java
public static byte[] decodeBase32(String base32Input) {
    byte[] decoded = BASE32.decode(base32Input);
    // Re-encode and compare to enforce canonical form
    String reEncoded = BASE32.encodeAsString(decoded).replaceAll("=", "");
    if (!reEncoded.equalsIgnoreCase(base32Input)) {
        throw new IllegalArgumentException("Non-canonical base32 alias");
    }
    return decoded;
}
```

Alternatively, update the GraphQL schema `@Pattern` to enforce exact length constraints matching valid base32 group boundaries (lengths 2, 4, 5, 7, 8, and multiples of 8), which would not fully fix the issue but would reduce the attack surface. The canonical re-encode check is the correct fix.

### Proof of Concept

**Setup:** An entity exists in the database with alias bytes `[0x00]` (1 byte), whose canonical base32 encoding is `AA`.

**Steps:**
1. Send GraphQL query with `alias: "AA"` → entity returned.
2. Send GraphQL query with `alias: "AB"` → **same entity returned** (`AB` decodes to `[0x00]`, last 2 bits `01` ignored).
3. Send GraphQL query with `alias: "AC"` → **same entity returned** (`AC` decodes to `[0x00]`, last 2 bits `10` ignored).
4. Send GraphQL query with `alias: "AD"` → **same entity returned** (`AD` decodes to `[0x00]`, last 2 bits `11` ignored).

All four queries pass the `^[A-Z2-7]+$` regex, all reach `BASE32.decode()`, all produce `[0x00]`, and all return the same entity — confirming the aliasing behavior with no privileges required.
