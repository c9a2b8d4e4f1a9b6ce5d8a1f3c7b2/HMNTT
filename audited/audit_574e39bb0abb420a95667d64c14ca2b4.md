### Title
Pinger Persistent DoS via Empty `nodes` Array from Mirror REST Endpoint (Retry Bypass + Fatal Exit)

### Summary
When `network=other`, `fetchMirrorNodeNetwork()` returns `retry=false` on an empty `nodes` response, which causes `buildNetworkFromMirrorNodes()` to break out of its retry loop immediately after a single poisoned response. The resulting error propagates to `main()` where `log.Fatalf` terminates the process. An attacker who can inject a single empty-nodes HTTP response (via MITM, DNS hijack, or server compromise) can keep the pinger permanently dead across Kubernetes restarts.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go` lines 127–129 — when the decoded `nodes` array yields zero usable endpoints, `fetchMirrorNodeNetwork` returns the sentinel `retry=false`:

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

`pinger/mirror_node_client.go` lines 52–61 — the retry loop in `buildNetworkFromMirrorNodes` checks the `retry` flag before sleeping:

```go
if !retry || attempt == attempts {
    break
}
```

Because `retry=false`, the loop breaks on the **first attempt**, regardless of `mirrorNodeClientMaxRetries` (default 10). The full retry budget is silently discarded.

`pinger/sdk_client.go` lines 18–21 — `newClient` propagates the error:

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil {
    return nil, err
}
```

`pinger/main.go` lines 41–44 — `main` calls `log.Fatalf`, which calls `os.Exit(1)`:

```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)
}
```

**Root cause:** The code treats "no usable endpoints in response" as a non-retryable, fatal condition. This is the wrong classification — an empty-nodes response is indistinguishable from a transient or adversarial response, yet it bypasses all retry protection and terminates the process.

**Exploit flow:**
1. Attacker positions themselves to intercept or serve the mirror REST HTTP response (see Likelihood section).
2. Attacker returns a valid HTTP 200 with body `{"nodes":[],"links":{"next":null}}`.
3. `fetchMirrorNodeNetwork` decodes successfully, finds `len(network)==0`, returns `retry=false`.
4. Retry loop breaks immediately; `buildNetworkFromMirrorNodes` returns error.
5. `newClient` returns error; `log.Fatalf` kills the process.
6. Kubernetes restarts the pod; attacker serves the same response again → persistent crash loop.
7. `/tmp/ready` is never written (line 47 is never reached), so the readiness probe also fails.

### Impact Explanation

The pinger is a liveness/health-check component that submits periodic transfers. A persistent crash loop means:
- No transfers are submitted; the pinger's monitoring function is completely disabled.
- The readiness probe (`/tmp/ready`) is never created, so the pod never becomes ready.
- The liveness heartbeat goroutine (`/tmp/alive`) never runs long enough to write its file.
- The disruption persists for as long as the attacker can serve the poisoned response — there is no self-healing because the retry mechanism is bypassed entirely.

Severity is **medium** (griefing / availability impact on the pinger component; no direct economic loss to network users).

### Likelihood Explanation

The attack is only reachable when `HIERO_MIRROR_PINGER_NETWORK=other`, which is the custom-deployment mode. The `mirrorRest` URL is operator-configured and may use plain `http://`. Realistic attacker positions:

- **Network MITM** (same L2 segment, ARP/BGP poisoning) against an unencrypted `http://` mirror REST URL — no credentials required.
- **DNS hijacking** of the mirror REST hostname — achievable by a rogue DNS resolver or cache poisoning.
- **Compromised mirror node** — if the mirror REST service itself is compromised, the attacker can serve the empty response.

A single poisoned response is sufficient; the attacker does not need to sustain the attack beyond the pinger's startup window. Repeatability is high because every pod restart re-triggers the vulnerable startup path.

### Recommendation

1. **Treat empty-nodes as retryable.** Change `retry=false` to `retry=true` at `mirror_node_client.go` line 128 so that a transiently empty response is retried with backoff:
   ```go
   if len(network) == 0 {
       return nil, true, fmt.Errorf("no usable service_endpoints found from %s", url)
   }
   ```

2. **Do not fatal on startup failure.** Replace `log.Fatalf` in `main.go` with a retry loop that attempts `newClient` with backoff, so a transient mirror outage does not permanently kill the pinger.

3. **Enforce TLS for the mirror REST URL.** Reject `http://` URLs in `loadConfig` when `network=other`, or at minimum warn loudly, to eliminate the MITM surface.

4. **Validate minimum node count.** Require at least one node before accepting the response as authoritative.

class H(http.server.BaseHTTPRequestHandler):

        self.send_header('Content-Type','application/json')
        self.send_header('Content-Length', str(len(body)))
        self.end_headers()
