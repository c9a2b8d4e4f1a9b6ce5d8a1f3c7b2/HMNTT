### Title
Unauthenticated Alias Lookup Forces Uncached DB Query Per Request, Enabling Sustained DB Load (Griefing)

### Summary
Any unauthenticated external user can send requests to `/api/v1/accounts/<ALIAS>/nfts` (and sibling routes) using syntactically valid but non-existent account aliases. Because `AccountAlias.isValid()` only checks format via regex, and `EntityService.getAccountFromAlias()` always issues a live DB query with no caching for misses, an attacker can force one DB query per request indefinitely. The response cache explicitly skips non-2xx responses, so 404 misses are never absorbed.

### Finding Description

**Code path:**

1. `rest/routes/accountRoute.js` line 15 registers `GET /:idOrAliasOrEvmAddress/nfts` with no authentication or rate-limiting middleware. [1](#0-0) 

2. `AccountController.getNftsByAccountId` (line 91) immediately calls `EntityService.getEncodedId()` with the raw path parameter. [2](#0-1) 

3. `EntityService.getEncodedId()` (lines 125–126) checks `AccountAlias.isValid()` — a pure regex check — and if it passes, calls `getAccountIdFromAlias()` → `getAccountFromAlias()`. [3](#0-2) 

4. `AccountAlias.isValid()` only tests the string against `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`. It performs **no DB lookup** and returns `true` for any well-formed base32 string. [4](#0-3) 

5. `getAccountFromAlias()` unconditionally executes `entityFromAliasQuery` against the DB. There is no in-process or Redis cache for alias misses. [5](#0-4) 

6. The response cache (`responseCacheUpdateHandler`) only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 (alias not found) is never cached, so every repeated or novel non-existent alias hits the DB fresh. [6](#0-5) 

**Root cause:** The system assumes that format validity (`isValid()`) is a sufficient pre-filter before issuing a DB query, but the alias space is enormous (any uppercase base32 string of any length), and there is no caching of negative results and no application-level rate limiting on the REST API.

### Impact Explanation
An attacker can generate an unbounded stream of distinct valid-format aliases (e.g., `AAAAAAAAAA`, `AAAAAAAAAB`, …) and send concurrent GET requests to `/api/v1/accounts/<ALIAS>/nfts`. Each request causes one full table scan or index lookup on the `entity` table filtered by `alias`. Under sustained load this degrades DB performance for all legitimate users of the mirror node REST API. This matches the stated scope: griefing with no direct economic damage to network users, but real service degradation.

### Likelihood Explanation
No privileges are required. The endpoint is publicly reachable. Generating valid base32 strings is trivial (any string matching `[A-Z2-7]+`). A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple script) can sustain hundreds of unique alias queries per second. The absence of application-level rate limiting in the REST server middleware stack (confirmed: only `authHandler`, `metricsHandler`, and `responseCacheCheckHandler` are registered) means the only protection is optional infrastructure-level Traefik config, which may not be deployed in all environments. [7](#0-6) 

### Recommendation
1. **Cache negative alias lookups**: After a DB miss in `getAccountFromAlias()`, store a short-lived negative sentinel in Redis (e.g., TTL 30–60 s) keyed by the decoded alias bytes. Subsequent requests for the same non-existent alias are served from cache without a DB round-trip.
2. **Application-level rate limiting**: Add a per-IP (or global) rate-limit middleware to the REST Express app for alias-bearing path parameters, independent of infrastructure-level Traefik config.
3. **Alias length cap**: Enforce a maximum alias length in `AccountAlias.isValid()` to reduce the effective attack surface (very long aliases still pass the current regex).

### Proof of Concept

```bash
# Generate and fire 10,000 unique valid aliases at the nfts endpoint
for i in $(seq 1 10000); do
  # Each alias is a unique valid base32 string that will not exist in the DB
  ALIAS=$(printf 'AAAAAAAAAA%05d' $i | tr '[:lower:]' '[:upper:]')
  curl -s "http://<mirror-node-host>/api/v1/accounts/${ALIAS}/nfts" &
done
wait
```

Each request passes `AccountAlias.isValid()`, reaches `getAccountFromAlias()`, and issues a live `SELECT id FROM entity WHERE coalesce(deleted,false)<>true AND alias=$1` query. With no caching of misses and no rate limiting, all queries hit the DB concurrently, causing sustained load. [8](#0-7)
