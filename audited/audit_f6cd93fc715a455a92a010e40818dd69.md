### Title
Cache-Miss Amplification via Non-Existent Alias Lookups Bypassing Spring `@Cacheable` Negative-Result Caching

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse()` is annotated with `@Cacheable(unless = "#result == null")`. Because Spring Cache unwraps `Optional` return values before evaluating SpEL expressions, an `Optional.empty()` result causes `#result` to resolve to `null`, which satisfies the `unless` condition and prevents the empty result from being stored in the cache. Any unauthenticated caller can therefore flood the endpoint with distinct non-existent alias byte arrays, causing every request to execute the native SQL query against the database with no cache absorption.

### Finding Description

**Exact code location:** [1](#0-0) 

```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
@Query(value = """
        select *
        from entity
        where (evm_address = ?1 or alias = ?1) and deleted is not true
        """, nativeQuery = true)
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

**Root cause — Spring Optional unwrapping in SpEL:**
Spring Cache (since 4.3) unwraps `Optional` before evaluating `unless`. When the query returns no row, the method returns `Optional.empty()`. At the point the `unless` expression is evaluated, `#result` is the unwrapped content of the `Optional`, which is `null`. The condition `#result == null` is therefore `true`, and Spring skips writing to the cache. The next call with the same (or any other non-existent) alias byte array finds nothing in the cache and re-executes the full SQL query.

**Cache configuration — 1-second TTL, 10 000 entries:** [2](#0-1) 

Even if the same non-existent key were cached, the 1-second TTL means the window for re-exploitation per key is extremely short.

**Rate limiter — global, not per-IP, 500 RPS default:** [3](#0-2) [4](#0-3) 

The throttle is a single shared token bucket across all callers. An attacker consuming the full 500 RPS budget forces 500 uncached DB queries per second for alias lookups, while simultaneously starving legitimate users of their quota.

**Exploit flow:**
1. Attacker generates a stream of random 20-byte or 32-byte arrays (none matching any real entity).
2. Attacker sends `POST /api/v1/contracts/call` with each random address as the `to` field at the maximum allowed rate (500 RPS).
3. Each request reaches `findByEvmAddressOrAliasAndDeletedIsFalse()` with a unique key; the cache is cold for every key.
4. Spring evaluates `unless = "#result == null"` → `true` → result not stored.
5. Every request executes `SELECT * FROM entity WHERE (evm_address = ?1 OR alias = ?1) AND deleted IS NOT TRUE` against the database.
6. The database receives up to 500 full-table-scan-style OR queries per second with no cache relief.

### Impact Explanation
The database is subjected to a sustained, unbounded stream of uncached OR-predicate queries. The `evm_address OR alias` predicate requires the planner to evaluate two index paths per query; under high concurrency this degrades query throughput for all other operations (token lookups, contract state reads, etc.) sharing the same connection pool. The result is service degradation or outage for all users of the mirror-node web3 service — a griefing impact with no direct economic damage to on-chain participants, consistent with the stated Medium scope.

### Likelihood Explanation
No account, API key, or privileged credential is required. The public `/api/v1/contracts/call` endpoint is reachable by any internet user. Generating random byte arrays is trivial. The attack is fully repeatable and can be sustained indefinitely. The only friction is the global 500 RPS cap, which the attacker can saturate with a single HTTP client sending pipelined requests.

### Recommendation
1. **Cache negative results explicitly.** Replace `unless = "#result == null"` with a sentinel value or use `unless = "false"` combined with a short TTL so that cache misses are stored and served from cache on repeat lookups:
   ```java
   @Cacheable(
       cacheNames = CACHE_NAME_ALIAS,
       cacheManager = CACHE_MANAGER_ENTITY,
       key = "@spelHelper.hashCode(#alias)")
   // Remove `unless` entirely, or use: unless = "#result != null && !#result.isPresent()" — but note Spring unwraps Optional, so use a wrapper sentinel instead.
   ```
   The cleanest fix is to return a custom wrapper type (not `Optional`) so that empty results are a non-null object that Spring will cache normally.

2. **Add per-IP or per-caller rate limiting** in addition to the global bucket, so a single attacker cannot consume the entire 500 RPS budget.

3. **Ensure indexes cover both `evm_address` and `alias` columns** on the `entity` table to reduce per-query cost when cache misses do occur.

### Proof of Concept
```bash
# Generate 10 000 random 20-byte hex addresses and fire them at the endpoint
python3 -c "
import os, requests, threading

def attack():
    while True:
        addr = '0x' + os.urandom(20).hex()
        requests.post('http://<mirror-node>/api/v1/contracts/call',
            json={'to': addr, 'data': '0x', 'gas': 21000},
            timeout=5)

threads = [threading.Thread(target=attack) for _ in range(50)]
[t.start() for t in threads]
[t.join() for t in threads]
"
```
Each request resolves to a unique non-existent alias, bypasses the cache due to the `unless = "#result == null"` / Spring Optional unwrapping interaction, and issues a live SQL query. Database CPU and connection utilization will climb linearly with request rate up to the 500 RPS global cap.
