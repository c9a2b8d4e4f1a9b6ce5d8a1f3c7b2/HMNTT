### Title
Future Timestamp Accepted in Token Info Filter Triggers Expensive History Query and Prevents Cache Population

### Summary
`extractSqlFromTokenInfoRequest()` in `rest/tokens.js` is reachable by any unauthenticated user who supplies a `timestamp=lt:<future_value>` query parameter. Because `isValidTimestampParam` only validates format and imposes no upper-bound check, a far-future timestamp passes validation, forces the expensive three-way UNION history query path, and prevents the in-process token cache from being populated — all without any privilege.

### Finding Description

**Validation gap — `isValidTimestampParam` (rest/utils.js:151-154):**
```js
const isValidTimestampParam = (timestamp) => {
  return /^\d{1,10}$/.test(timestamp) || /^\d{1,10}\.\d{1,9}$/.test(timestamp);
};
```
The regex accepts any value up to 10 decimal digits. `9999999999` (year 2286) is syntactically valid and passes.

**Operator allowlist — `validateTokenInfoFilter` (rest/tokens.js:451-463):**
```js
if (param === filterKeys.TIMESTAMP) {
  ret = isValidTokenInfoTimestampFilterOp(op) && utils.isValidTimestampParam(val);
}
```
`op=lt` is explicitly allowed. No check that the value is ≤ `now`.

**History query branch — `extractSqlFromTokenInfoRequest` (rest/tokens.js:488-513):**
```js
if (filters && filters.length !== 0) {
  // builds UNION of token + token_history, entity + entity_history,
  // custom_fee + custom_fee_history — three sub-queries each
  tokenQuery = buildHistoryQuery(...);
  entityQuery = buildHistoryQuery(...);
  customFeeQuery = buildHistoryQuery(...);
}
```
Any non-empty filter list triggers this path. With `lower(timestamp_range) < 9999999999000000000` every row in every history table matches, so the query is semantically equivalent to the no-filter path but far more expensive.

**Cache population gate — `getTokenInfoRequest` (rest/tokens.js:561-563):**
```js
if (filters.length === 0) {
  TokenService.putTokenCache(token);
}
```
The in-process LRU cache (`quickLru`) is only written when no filters are present. A request with a future timestamp filter always bypasses this write, so the token is never cached for downstream consumers (e.g., `getTokenAccounts`).

### Impact Explanation
Every request with `?timestamp=lt:9999999999` executes six sub-queries (three UNION pairs across current + history tables) instead of three simple table reads. An attacker can sustain this against any token ID at the rate the API allows, multiplying DB read I/O by a constant factor per request. Additionally, the cache is never populated, so subsequent legitimate requests that rely on `TokenService.getCachedTokens` (e.g., account token-relationship lookups) also hit the DB instead of the cache. Severity: Medium — no data confidentiality or integrity impact, but measurable availability degradation under load.

### Likelihood Explanation
No authentication is required. The parameter is documented as part of the public REST API. The exploit requires only a single HTTP GET with a crafted query string, is trivially repeatable, and can be scripted. Any external user who reads the API docs can reproduce it.

### Recommendation
Add an upper-bound check in `validateTokenInfoFilter` (or in `isValidTimestampParam` when called from this context):
```js
const nowNs = BigInt(Date.now()) * 1_000_000n;
const valueNs = parseTimestampToNs(val); // seconds[.nanos] → nanoseconds
if (valueNs > nowNs) return false;
```
Alternatively, reject any timestamp filter whose value exceeds the current consensus timestamp obtained from a lightweight DB query or a cached "latest" value. This ensures the history path is only triggered for genuinely historical queries.

### Proof of Concept
```
# Replace <TOKEN_ID> with any valid token (e.g., 0.0.1234)
# Year 2286 timestamp — passes isValidTimestampParam, triggers history path

GET /api/v1/tokens/<TOKEN_ID>?timestamp=lt:9999999999

# Observe:
# 1. HTTP 200 returned with current token state (not a historical snapshot)
# 2. Server executes 6 sub-queries (3 UNION pairs) instead of 3 simple reads
# 3. Token is NOT added to the in-process LRU cache
# 4. Repeat at high frequency to amplify DB load
```
