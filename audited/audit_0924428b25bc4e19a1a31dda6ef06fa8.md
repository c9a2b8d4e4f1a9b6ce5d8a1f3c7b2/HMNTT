### Title
Unauthenticated Historical Contract Storage Slot Value Disclosure via Opcode Tracer Endpoint

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController.java` has no authentication or authorization check. Any unauthenticated external user can submit any known `transactionIdOrHash` with `storage=true` and receive the exact values of every storage slot (including read-only SLOAD accesses) that were touched during the original transaction, reconstructed from historical state. This exposes contract storage values that may have since been deleted or overwritten, and which are not otherwise enumerable through the public REST API without already knowing the slot keys.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` — `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, lines 52–68 — performs only three checks before executing:
1. `properties.isEnabled()` — a feature flag, `false` by default but operator-configurable
2. `validateAcceptEncodingHeader()` — only verifies `gzip` in `Accept-Encoding`
3. `throttleManager.throttleOpcodeRequest()` — rate-limiting only

There is **no authentication, no authorization, and no caller identity check** of any kind. [1](#0-0) 

The `storage` parameter defaults to `false` but any caller can set `?storage=true`. [2](#0-1) 

**Historical state reconstruction:**

`ContractDebugService.processOpcodeCall()` sets the EVM replay timestamp to `consensusTimestamp - 1`, meaning the EVM is re-executed against the exact historical state just before the original transaction. [3](#0-2) 

`ContractStorageReadableKVState.readFromDataSource()` then resolves every SLOAD by querying `contract_state_change` for the most recent value at or before that timestamp — including slots that have since been zeroed out or overwritten. [4](#0-3) 

The backing SQL query returns `coalesce(value_written, value_read)` from `contract_state_change`, covering both write and read-only accesses. [5](#0-4) 

**Storage capture and response:**

`AbstractOpcodeTracer.captureStorage()` calls `getTxStorageUsage(true).accesses()` — the `true` flag includes read-only accesses — and for each access emits either `access.writtenValue()` or `access.value()` (the SLOAD-read value) into the response map. [6](#0-5) 

This map is attached to every `Opcode` object in the `OpcodesResponse` and returned directly to the caller. [7](#0-6) 

**Root cause / failed assumption:** The design assumes the endpoint will only be used by trusted operators/developers. There is no enforcement of this assumption in code — no Spring Security configuration, no API key, no role check. The `enabled` flag is the only gate, and once flipped on (e.g., for a production debugging session), the endpoint is fully public. [8](#0-7) 

### Impact Explanation

An attacker who knows (or can enumerate) any transaction ID or hash involving a contract that stored sensitive data in a storage slot — even if that slot was subsequently deleted or overwritten — can recover the exact historical value of that slot. This includes:

- Private keys or seeds stored transiently in contract storage
- Confidential mapping values (e.g., user balances, access-control secrets) that were read during a transaction
- Any slot accessed via SLOAD that the contract author believed was no longer accessible

Unlike the existing `/api/v1/contracts/{id}/state?timestamp=...` REST endpoint (which requires the caller to already know the slot key), the opcode tracer with `storage=true` **enumerates all accessed slots and their values in a single response**, making it a slot-discovery oracle as well as a value-disclosure oracle.

### Likelihood Explanation

- **Precondition**: The operator has set `hiero.mirror.web3.opcode.tracer.enabled=true`. This is `false` by default but is a documented, supported configuration for debugging.
- **Attacker capability**: Zero — any HTTP client can call the endpoint. No credentials, no on-chain account, no special knowledge beyond a transaction hash (which is public on-chain).
- **Repeatability**: Fully repeatable for any historical transaction as long as the `contract_state_change` records are retained.
- **Discovery**: Transaction hashes are public; an attacker can scan known contract interactions to find transactions involving sensitive contracts.

### Recommendation

1. **Add authentication/authorization** to the endpoint. At minimum, require an operator-configured API key or restrict the endpoint to localhost/internal network via Spring Security or a servlet filter.
2. **Do not expose read-only SLOAD values** unless explicitly required. Pass `false` to `getTxStorageUsage()` to limit capture to writes only, or add a separate privileged flag for read capture.
3. **Treat `enabled=true` as a privileged mode** that requires both a feature flag AND an authenticated caller, not just a feature flag alone.

### Proof of Concept

**Preconditions:**
- Mirror node deployed with `hiero.mirror.web3.opcode.tracer.enabled=true`
- A known transaction hash `0xABC...` for a contract that executed SLOAD on a sensitive slot (e.g., a mapping storing a private value)

**Steps:**
```
GET /api/v1/contracts/results/0xABC.../opcodes?storage=true
Accept-Encoding: gzip
```

**Result:**
The response contains an `opcodes` array. Each `Opcode` object includes a `storage` map of `{ "0x<slot_key>": "0x<slot_value>" }` reflecting the historical values of every storage slot accessed (read or written) during the transaction replay — including slots that have since been zeroed out or overwritten in the current state.

No credentials, no on-chain account, and no prior knowledge of slot keys are required.

    }
