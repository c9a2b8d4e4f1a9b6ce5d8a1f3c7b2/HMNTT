### Title
Unauthenticated `/metrics` Endpoint Exposes Real-Time Inflight Request Gauge to Any External User

### Summary
The Rosetta service exposes its Prometheus `/metrics` endpoint on the same public port as the API with no authentication of any kind. Any unauthenticated caller can repeatedly poll `GET /metrics` and read the `hiero_mirror_rosetta_request_inflight` gauge — broken down by `method` and `route` labels — to observe real-time concurrency levels across all API routes, enabling side-channel inference of system load and user activity patterns.

### Finding Description
**Exact code path:**

In `rosetta/app/middleware/metrics.go`, the `Routes()` function (lines 64–73) registers `/metrics` directly with `promhttp.Handler().ServeHTTP` — a bare Prometheus HTTP handler with no authentication wrapper:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,
            promhttp.Handler().ServeHTTP,  // no auth, no middleware
        },
    }
}
```

In `rosetta/main.go` (lines 111–119 and 148–152), `metricsController` is registered directly into `server.NewRouter(...)` in both online and offline modes. The middleware chain assembled at lines 217–219 is:

```
MetricsMiddleware(router) → TracingMiddleware → CorsMiddleware
```

None of these layers enforce authentication on the `/metrics` route. The `Config` struct (`rosetta/app/config/types.go`) has no metrics authentication fields whatsoever — unlike the REST API which has `authentication`, `username`, and `password` config options in its metrics handler.

The `requestInflightGauge` (lines 34–37) is a `GaugeVec` with labels `{method, route}` that reflects the live count of in-flight HTTP requests. `MetricsMiddleware` (lines 76–83) increments/decrements this gauge for every request via `middleware.Instrument{InflightRequests: requestInflightGauge}`.

**Root cause:** The failed assumption is that `/metrics` is an internal-only scrape target. It is served on the same public-facing port (`rosettaConfig.Port`) as all Rosetta API endpoints, with no IP restriction, no token check, and no basic-auth gate.

**Exploit flow:**
1. Attacker sends `GET /metrics` to the public Rosetta port.
2. Response (HTTP 200, `text/plain`) includes lines such as:
   ```
   hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/block"} 14
   hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/construction/submit"} 3
   ```
3. Attacker polls at high frequency (e.g., every 100 ms) and records the time series.
4. From the time series the attacker can infer: which routes are actively used, peak concurrency windows, and relative user activity patterns across routes.

**Why existing checks fail:** The `metrics_test.go` (lines 13–23) confirms the endpoint returns HTTP 200 with no credentials supplied. There is no IP allowlist, no network policy enforced at the application layer, and no configuration knob to enable auth.

### Impact Explanation
An unprivileged external attacker gains a continuous, real-time view of system load and API usage patterns without any prior authentication. The `hiero_mirror_rosetta_request_inflight` gauge reveals which Rosetta API routes are concurrently active and at what concurrency level, enabling inference of user activity patterns (e.g., spikes in `/construction/submit` indicate active transaction submission campaigns). Additionally, the full Prometheus output includes `hiero_mirror_rosetta_request_bytes`, `hiero_mirror_rosetta_request_duration` histograms, and Go runtime metrics, broadening the information surface. This is a confirmed information disclosure vulnerability.

### Likelihood Explanation
Exploitation requires only an HTTP GET request — no credentials, no special tooling, no prior knowledge beyond the server's hostname and port. The Rosetta API is a public-facing service by design (Coinbase Rosetta spec mandates public accessibility). Any internet-reachable deployment is trivially exploitable by any attacker who can reach the port. Polling at high frequency is cheap and undetectable without specific rate-limiting on the `/metrics` path.

### Recommendation
1. **Add authentication to the `/metrics` handler** — wrap `promhttp.Handler()` with a basic-auth or bearer-token check, mirroring the pattern used in the REST API's `authenticate()` function in `rest/middleware/metricsHandler.js` (lines 137–144).
2. **Serve metrics on a separate, non-public port** — bind a dedicated metrics-only HTTP listener to `localhost` or an internal network interface, separate from `rosettaConfig.Port`.
3. **Add a `Metrics` auth config block** to `rosetta/app/config/types.go` (analogous to the REST API's `metrics.config.authentication` setting) so operators can configure credentials.

### Proof of Concept
```bash
# No credentials required — returns HTTP 200 with full metrics
curl -s http://<rosetta-host>:<port>/metrics | grep hiero_mirror_rosetta_request_inflight

# Poll every 100ms to build a time series
while true; do
  curl -s http://<rosetta-host>:<port>/metrics \
    | grep hiero_mirror_rosetta_request_inflight \
    | awk '{print systime(), $0}'
  sleep 0.1
done
```

Expected output (no auth challenge, immediate data):
```
1714500000 hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/block"} 7
1714500000 hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/construction/submit"} 2
```
