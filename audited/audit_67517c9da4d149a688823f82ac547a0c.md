### Title
Unauthenticated DoS via Unbounded Concurrent Range Queries on `/api/v1/accounts/{ownerId}/hooks` Due to Missing Rate Limiting in rest-java

### Summary
The `getHooks()` endpoint in `HooksController.java` accepts up to 100 `hook.id` range parameters from any unauthenticated caller, and the `hooksRequest()` method collapses them into a maximally wide `[0, Long.MAX_VALUE]` range query against the database. The `rest-java` module has no rate-limiting layer (unlike `web3`), so an attacker can flood the endpoint with concurrent requests, each triggering a full-range `BETWEEN 0 AND 9223372036854775807` DB scan per owner, exhausting DB CPU and connection pool resources.

### Finding Description

**Code path:**

`HooksController.getHooks()` (lines 81–102) accepts `hook.id` as a repeated parameter bounded only by `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` where `MAX_REPEATED_QUERY_PARAMETERS = 100`. [1](#0-0) [2](#0-1) 

`hooksRequest()` (lines 132–156) iterates all supplied filters and computes a single `lowerBound` (max of all `gt`/`gte` values, defaulting to `0L`) and `upperBound` (min of all `lt`/`lte` values, defaulting to `Long.MAX_VALUE`). Supplying `gt:0` and `lt:MAX_VALUE` 100 times produces exactly `lowerBound=0, upperBound=Long.MAX_VALUE` — the widest possible range — in a single pass. [3](#0-2) 

`HookServiceImpl.getHooks()` (line 40) then issues `findByOwnerIdAndHookIdBetween(ownerId, 0, MAX_VALUE, page)` — a full-range scan over all hooks for the given owner, bounded only by `LIMIT 100` on the result set (not on the scan cost). [4](#0-3) 

**Root cause — no rate limiting in rest-java:**

The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists exclusively in the `web3` module. [5](#0-4) 

The `rest-java` config directory contains only `MetricsFilter`, `LoggingFilter`, `JacksonConfiguration`, `MetricsConfiguration`, `NetworkProperties`, `RestJavaConfiguration`, `RuntimeHintsConfiguration`, and `WebMvcConfiguration` — none of which implement request-rate throttling.


**Why existing checks are insufficient:**

- `@Size(max = 100)` on `hookId[]` prevents more than 100 parameters but still allows 100 parameters, all of which are processed in the loop.
- `@Max(MAX_LIMIT)` caps the *result* at 100 rows but does not cap the *scan cost* of `BETWEEN 0 AND MAX_VALUE`.
- `statementTimeout = 10000` (10 s) per the documentation means each query can hold a DB connection for up to 10 seconds before timing out. [6](#0-5) [7](#0-6) 

### Impact Explanation

Each concurrent request holds a DB connection for the duration of the full-range scan. With no rate limiting, an attacker can saturate the HikariCP connection pool (monitored via `hikaricp_connections_active / hikaricp_connections_max > 0.75` alert), starving legitimate queries. Sustained flooding drives DB CPU to saturation, causing query latency to exceed the 1 s alert threshold and request latency to exceed the 2 s alert threshold, consistent with 30%+ degradation of mirror-node processing capacity. [8](#0-7) [9](#0-8) 

### Likelihood Explanation

No authentication or API key is required. The endpoint is publicly reachable. The attacker needs only a standard HTTP client capable of sending concurrent GET requests. The attack is trivially repeatable and scriptable with tools like `wrk`, `ab`, or `curl` in a loop. No special knowledge of the system is required beyond the public API path.

### Recommendation

1. **Add rate limiting to rest-java**: Port the `bucket4j`-based `ThrottleConfiguration` pattern from the `web3` module into `rest-java`, applying a per-IP or global requests-per-second limit to all `/api/v1/` endpoints.
2. **Reject maximally wide ranges**: In `hooksRequest()`, reject or warn when `upperBound - lowerBound` exceeds a configurable maximum span (analogous to `maxTimestampRange` in the Node.js REST API).
3. **Reduce `statementTimeout`**: Lower the DB statement timeout for read-only REST queries to reduce connection hold time under attack.
4. **Add a query-cost guard**: Before issuing `findByOwnerIdAndHookIdBetween`, check that `upperBound > lowerBound` by at most a configured threshold and return 400 otherwise.

### Proof of Concept

```bash
# Build a URL with 100 maximally wide range parameters
PARAMS=$(python3 -c "print('&'.join(['hook.id=gt:0','hook.id=lt:9223372036854775807']*50))")

# Flood with 500 concurrent connections, no authentication required
wrk -t 10 -c 500 -d 60s \
  "http://<mirror-node-host>/api/v1/accounts/0.0.1234/hooks?limit=100&$PARAMS"

# Expected result: DB CPU spikes, HikariCP pool exhausted,
# query latency > 1s, request latency > 2s, Grafana alerts fire.
```

```java
    private HooksRequest hooksRequest(
            EntityIdParameter ownerId, NumberRangeParameter[] hookIdFilters, int limit, Sort.Direction order) {
        final var hookIds = new TreeSet<Long>();
        long lowerBound = 0L; // The most restrictive lower bound (max of all gt/gte)
        long upperBound = MAX_VALUE; // The most restrictive upper bound (min of all lt/lte)

        for (final var hookIdFilter : hookIdFilters) {
            if (hookIdFilter.operator() == RangeOperator.EQ) {
                hookIds.add(hookIdFilter.value());
            } else if (hookIdFilter.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, hookIdFilter.getInclusiveValue());
            } else if (hookIdFilter.hasUpperBound()) {
                upperBound = Math.min(upperBound, hookIdFilter.getInclusiveValue());
            }
        }

        return HooksRequest.builder()
                .hookIds(hookIds)
                .lowerBound(lowerBound)
                .ownerId(ownerId)
                .limit(limit)
                .order(order)
                .upperBound(upperBound)
                .build();
    }
```

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
