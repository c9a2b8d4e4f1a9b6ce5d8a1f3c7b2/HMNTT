### Title
Unauthenticated Alias-Resolution DoS via Uncached DB Lookups in `HooksController.getHooks()`

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks` endpoint in `HooksController` accepts alias-format `ownerId` values (via `EntityIdAliasParameter`) and resolves them through `EntityServiceImpl.lookup()`, which issues a direct, uncached `findByAlias()` database query on every request. Because the endpoint requires no authentication and the `rest-java` module has no rate limiting, an unprivileged attacker can flood the endpoint with alias-format requests, exhausting the DB connection pool and degrading service availability.

### Finding Description
**Code path:**

1. `HooksController.getHooks()` accepts `@PathVariable EntityIdParameter ownerId` with no authentication or rate-limiting guard. [1](#0-0) 

2. `EntityIdParameter.valueOf()` parses the path variable. If the string matches the base32 alias regex (`^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$`), it returns an `EntityIdAliasParameter` — no privilege required. [2](#0-1) [3](#0-2) 

3. `HookServiceImpl.getHooks()` immediately calls `entityService.lookup(request.getOwnerId())` before any hook query. [4](#0-3) 

4. `EntityServiceImpl.lookup()` dispatches to `entityRepository.findByAlias(p.alias())` — a raw DB query — with **no caching layer** (unlike the importer's `EntityIdServiceImpl` which uses `cacheLookup`). [5](#0-4) 

**Root cause:** `EntityServiceImpl` in `rest-java` has no cache, so every alias-format request unconditionally hits the database. The importer module's `EntityIdServiceImpl` has a `Cache` field and `cacheLookup()` wrapper, but `rest-java`'s `EntityServiceImpl` does not. [6](#0-5) 

**Why existing checks fail:** The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is wired to `ContractCallRequest` processing — it is not applied to `rest-java` REST endpoints. No Spring Security configuration was found for `rest-java`. The `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` constraint on `hookId` only limits query parameters, not the alias lookup itself. [7](#0-6) 

### Impact Explanation
Each concurrent request with an alias-format `ownerId` acquires a DB connection and executes a `SELECT` against the `entity` table filtered by the `alias` column. Under sustained concurrent load, the DB connection pool is exhausted, causing all `rest-java` queries (including legitimate ones) to queue or fail. This degrades or halts hook-related and potentially other REST API processing nodes, meeting the ≥30% network processing degradation threshold without requiring any brute-force or authenticated access.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is publicly reachable. A valid alias string is trivially constructable: any 40-character uppercase base32 string (e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`) passes the regex. The attack is fully repeatable, scriptable with standard HTTP tools (e.g., `wrk`, `ab`, `curl` in a loop), and requires no knowledge of valid accounts since the DB query runs regardless of whether the alias resolves.

### Recommendation
1. **Add a cache to `EntityServiceImpl.lookup()`** for alias and EVM address resolutions, mirroring the `cacheLookup()` pattern in `EntityIdServiceImpl` (importer module).
2. **Apply rate limiting to `rest-java` endpoints** — introduce a per-IP or global request-rate filter (e.g., using Bucket4j, as already used in `web3`) applied at the servlet filter or Spring Security level.
3. **Fail fast on non-resolving aliases** — consider returning 404 immediately without a DB round-trip if the alias format is present but the entity table lookup is known to be expensive (e.g., add a bloom filter or short-circuit cache for known-missing aliases).

### Proof of Concept
```bash
# Generate a valid alias-format string (40 uppercase base32 chars)
ALIAS="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"

# Flood the endpoint concurrently (no credentials needed)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/hooks" &
done
wait
```

Each request triggers `EntityServiceImpl.lookup()` → `entityRepository.findByAlias(alias)` → one DB connection acquired and one `SELECT` executed against the `entity` table. With 500+ concurrent requests, the DB connection pool (typically 10–50 connections) is saturated, causing `rest-java` to return errors or time out for all subsequent requests, including those from legitimate users.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
