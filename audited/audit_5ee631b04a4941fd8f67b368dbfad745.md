### Title
Global Rate-Limit Bucket Exhaustion via Unauthenticated Request Flooding (DoS)

### Summary
`ThrottleConfiguration.rateLimitBucket()` creates a single, process-wide `Bucket` bean (default 500 tokens/second) that is shared across every caller of every contract-call endpoint. Because there is no per-IP or per-user isolation, any unauthenticated attacker who sends requests at or above the configured rate will exhaust the global token pool, causing every other user's requests to be rejected with a `ThrottleException` until the bucket refills.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 24–32 create one singleton `Bucket` with a flat capacity of `requestsPerSecond` (default 500): [1](#0-0) 

`ThrottleManagerImpl.throttle()` (lines 37–42) calls `rateLimitBucket.tryConsume(1)` for every inbound request without any per-caller partitioning: [2](#0-1) 

`ContractController.call()` (line 40) invokes `throttleManager.throttle(request)` for every POST to `/api/v1/contracts/call`, which handles both `eth_call` and `eth_estimateGas` (distinguished only by the `isEstimate` flag in the request body): [3](#0-2) 

**Root cause / failed assumption:** The design assumes that the aggregate request rate from all clients will stay within the configured limit. It does not account for a single hostile client consuming the entire budget. There is no per-IP, per-session, or per-user sub-bucket; the single `Bucket` bean is a shared global counter.

**Why existing checks are insufficient:**
- The `RequestProperties` per-request filters (lines 44–48 of `ThrottleManagerImpl`) are opt-in, operator-configured rules that match on request content, not on caller identity, and are not present by default (`List.of()`). [4](#0-3) 
- The `opcodeRateLimitBucket` is separate and unaffected, but the main contract-call path has no equivalent per-caller guard. [5](#0-4) 

### Impact Explanation
An attacker who saturates the 500 req/s global budget causes every legitimate user's POST to `/api/v1/contracts/call` to receive an HTTP error (mapped from `ThrottleException` with message `"Requests per second rate limit exceeded"`). Both `eth_call` and `eth_estimateGas` traffic is blocked simultaneously because they share the same endpoint and the same bucket. This is a complete, sustained denial-of-service against the web3 contract-call API with no economic cost to the attacker and no economic damage required — pure griefing. [6](#0-5) 

### Likelihood Explanation
**Preconditions:** None. No authentication, no API key, no privileged role is required. The endpoint is publicly reachable.  
**Feasibility:** Sending 500 HTTP POST requests per second from a single machine (or a handful of machines) is trivial with tools like `wrk`, `hey`, or a simple async script.  
**Repeatability:** The bucket refills greedily every second, so the attacker only needs to sustain ~500 req/s to keep the bucket permanently empty. The attack is fully repeatable and can be maintained indefinitely at negligible cost. [7](#0-6) 

### Recommendation
1. **Per-IP (or per-client) rate limiting:** Replace or augment the single global `Bucket` with a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `HttpServletRequest` or a trusted `X-Forwarded-For` header). Each client gets its own sub-budget; exhausting one client's bucket does not affect others.
2. **Separate per-call-type budgets:** If `eth_estimateGas` must remain available even under `eth_call` flood, create distinct buckets for each call type (already partially done for opcodes with `opcodeRateLimitBucket`).
3. **Front-line network-layer protection:** Deploy a reverse proxy (nginx, Envoy, AWS WAF) with per-IP connection and request-rate limits in front of the service so that floods are dropped before reaching the JVM.

### Proof of Concept
```bash
# Flood the global rate-limit bucket from a single unauthenticated client
# Tool: 'hey' (https://github.com/rakyll/hey)

hey -n 100000 -c 600 -m POST \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","data":"0x","gas":21000,"estimate":false}' \
  http://<mirror-node-host>/api/v1/contracts/call

# Expected result:
# - Attacker's own requests: ~500 succeed per second, rest get 429/ThrottleException
# - All other concurrent legitimate users: 100% of their requests rejected with
#   "Requests per second rate limit exceeded" for the duration of the flood
# - eth_estimateGas requests (same endpoint, estimate:true) equally blocked
```

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
