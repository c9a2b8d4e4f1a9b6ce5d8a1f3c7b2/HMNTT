### Title
Race Condition Between `safetyCheck` and `live` Flux Causes `IllegalStateException` in `onNext()`, Terminating Any Subscriber's Session

### Summary
`TopicContext.onNext()` throws `IllegalStateException` if a message arrives with a sequence number that is not exactly `last + 1`. `Flux.merge(safetyCheck, live)` runs both streams on separate `boundedElastic` threads concurrently. The `missingMessages()` gap-fill logic for the live stream snapshots `topicContext.getLast()` at call time, but by the time the live stream actually emits the message downstream, the `safetyCheck` stream may have already advanced `last` to an intermediate value — creating a gap that `onNext()` treats as a fatal ordering violation.

### Finding Description

**Exact code path:**

`subscribeTopic()` builds the merged pipeline:

```
historical.concatWith(Flux.merge(safetyCheck, live))
    .filter(t -> last == null || t.getSequenceNumber() > last.getSequenceNumber())  // line 74-77
    .doOnNext(topicContext::onNext)                                                  // line 88
```

`safetyCheck` is pinned to `Schedulers.boundedElastic()` (line 70). The `live` stream, via `SharedTopicListener`, calls `.publishOn(Schedulers.boundedElastic(), ...)` (SharedTopicListener.java line 25), placing it on a different thread. Both streams therefore emit concurrently inside `Flux.merge()`.

**Root cause — stale snapshot in `missingMessages()`:**

In `incomingMessages()` (line 120):
```java
return topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t));
```

When the live listener delivers message with seq=N, `missingMessages(topicContext, msgN)` is called. At that instant `last` may be `null`, so `isNext(msgN)` returns `true` (line 152) and the method returns `Flux.just(msgN)` — a cold publisher that will emit msgN later when `concatMap` subscribes to it.

**The gap:**

Between the moment `missingMessages` returns `Flux.just(msgN)` and the moment `concatMap` actually pushes msgN into the merged pipeline, the `safetyCheck` stream (running concurrently on another thread) may have already emitted messages 1…M (where M < N−1), advancing `last` to M.

**Filter is insufficient:**

The deduplication filter (lines 74-77) only checks `seq > last`. A message with seq=N where N > M+1 passes the filter but fails `isNext()`:

```java
// onNext(), line 239-241
if (!isNext(topicMessage)) {
    throw new IllegalStateException(
        String.format("Encountered out of order messages, last: %s, current: %s", last, topicMessage));
}
```

`isNext()` (line 217-219) requires `seq == last.seq + 1` exactly. With `last=M` and incoming `seq=N` where N > M+1, this throws.

**Exploit flow:**

1. Attacker subscribes to a topic that has messages 1–M already in the DB and is actively receiving new messages (seq > M+1) via the live listener.
2. `historical` completes; `Flux.merge(safetyCheck, live)` subscribes to both streams simultaneously on separate threads.
3. The live listener delivers msgN (seq=N, N > M+1). `missingMessages` is called with `last=null` → returns `Flux.just(msgN)`.
4. Concurrently, `safetyCheck` fires after 1 s, queries the DB, and emits msgs 1…M, advancing `last` to M.
5. `concatMap` now emits msgN into the pipeline. Filter: N > M → passes. `onNext(msgN)`: `isNext(N)` with `last=M` → N ≠ M+1 → **`IllegalStateException` thrown**.
6. Reactor propagates the exception as an error signal; the gRPC stream is terminated with an error.

### Impact Explanation
The subscriber's gRPC stream is abruptly terminated with an unhandled error. The subscriber receives no further messages and must reconnect. On a high-throughput topic, the race can recur on every reconnect attempt, effectively making the topic unsubscribable for that client. The `subscriberCount` gauge is correctly decremented via `doFinally`, so server-side resource leak is not a concern, but the subscriber is permanently denied service for that session.

### Likelihood Explanation
Any unprivileged user who subscribes to a topic that (a) has existing messages in the DB and (b) is actively receiving new messages can trigger this. No special privileges, credentials, or protocol manipulation are required — only a standard `subscribeTopic` gRPC call. The race window is the 1-second `safetyCheck` delay, which is wide enough to be hit reliably on any moderately active topic. The condition is repeatable: reconnecting restarts the same pipeline with the same race.

### Recommendation
The gap-fill decision must be made atomically at emit time, not at `missingMessages()` call time. Options:

1. **Re-evaluate `last` inside the emitted publisher**: Instead of returning `Flux.just(current)` when `last == null`, defer the check so it runs when the item is actually about to be emitted — after any concurrent safetyCheck emissions have already updated `last`.

2. **Serialize the two streams**: Replace `Flux.merge(safetyCheck, live)` with `Flux.concat(safetyCheck, live)` so safetyCheck fully completes before live starts emitting, eliminating the concurrent interleaving entirely.

3. **Relax `onNext()` to tolerate gaps**: Change the `IllegalStateException` to a logged warning and allow gaps to pass through (or trigger a re-query), rather than terminating the subscription.

4. **Use a synchronized/atomic compare-and-set in `onNext()`**: Accept and skip messages that are already covered by `last`, and only throw for true backward-sequence violations (seq < last).

### Proof of Concept

**Preconditions:**
- Topic `T` has messages seq=1, seq=2 already committed to the DB.
- The live listener is about to deliver seq=5 (messages 3 and 4 are not yet in the DB).

**Steps:**
1. Client calls `subscribeTopic` with `startTime=0`, `topicId=T`.
2. `historical` returns empty (or completes quickly).
3. `Flux.merge(safetyCheck, live)` starts. Both run on `boundedElastic` threads.
4. Live listener emits seq=5. `missingMessages(ctx, msg5)` reads `last=null` → returns `Flux.just(msg5)`.
5. One second later, `safetyCheck` queries DB → gets [msg1, msg2] → emits them → `last` advances to 2.
6. `concatMap` emits msg5 into the pipeline. Filter: 5 > 2 → passes. `onNext(msg5)`: `isNext(5)` with `last=2` → 5 ≠ 3 → **`IllegalStateException`**.
7. gRPC stream terminates with error. Client receives `INTERNAL` or `UNKNOWN` gRPC status.

**Reproducible with a unit test** by mocking `topicListener.listen()` to return a `Sinks.Many` that emits seq=5 after a 500 ms delay, and mocking `topicMessageRetriever.retrieve(..., false)` to return `Flux.just(msg1, msg2)` with a 1.1 s delay, then using `StepVerifier.withVirtualTime` to advance the clock — the verifier will receive an `onError` with `IllegalStateException` instead of the expected messages.     private Flux<TopicMessage> missingMessages(TopicContext topicContext, @Nullable TopicMessage current) {
        final var last = topicContext.getLast();

        // Safety check triggered
        if (current == null) {
            long startTime = last != null
                    ? last.getConsensusTimestamp() + 1
                    : topicContext.getFilter().getStartTime();
            var gapFilter =
                    topicContext.getFilter().toBuilder().startTime(startTime).build();
            log.info("Safety check triggering gap recovery query with filter {}", gapFilter);
            return topicMessageRetriever.retrieve(gapFilter, false);
        }

        if (last == null || topicContext.isNext(current)) {
            return Flux.just(current);
        }
