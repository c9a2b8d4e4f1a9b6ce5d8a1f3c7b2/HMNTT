### Title
Unbounded `DISTINCT ON` Historical Query in `getHookStorageChange` Enables DoS via Full Table Scan

### Summary
The `getHookStorageChange` private method in `HookServiceImpl` delegates to a native PostgreSQL `DISTINCT ON (key)` query with no enforced timestamp-range width or key-range restriction. Because PostgreSQL must sort and deduplicate every row matching the `WHERE` clause before applying `LIMIT`, an unauthenticated attacker can issue a single HTTP request that forces the database to process the entire `hook_storage_change` table for a given contract, holding a long-running transaction that starves the connection pool and blocks all other queries.

### Finding Description

**Exact code path:**

`HooksController.getHookStorage` (line 104–130) accepts `timestamp` and `key` parameters with no range-width validation. It builds a `HookStorageRequest` where `keyLowerBound` defaults to `MIN_KEY_BYTES` (32×`0x00`) and `keyUpperBound` defaults to `MAX_KEY_BYTES` (32×`0xFF`) when no `key` filter is supplied. [1](#0-0) 

`Bound.getAdjustedLowerRangeValue()` returns `0` when `lower == null`, and `adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`. [2](#0-1) [3](#0-2) 

`HookServiceImpl.getHookStorageChange` (lines 84–118) passes these unbounded values directly to `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween`. [4](#0-3) 

The repository executes a native query:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key >= :keyLowerBound
  AND key <= :keyUpperBound
  AND consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound
ORDER BY key ASC, consensus_timestamp DESC
LIMIT 100
``` [5](#0-4) 

**Root cause:** In PostgreSQL, `DISTINCT ON` requires the database engine to sort and deduplicate **all rows** satisfying the `WHERE` predicate before `LIMIT` is applied. The `LIMIT 100` (from `MAX_LIMIT = 100`) only restricts the output rows, not the work performed. With `key` spanning the full 32-byte space and `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE`, every historical row for the targeted `(owner_id, hook_id)` pair is read, sorted, and deduplicated in a single query.

**Why existing checks fail:**

- `@Max(MAX_LIMIT)` on `limit` caps returned rows at 100 but does not limit the scan. [6](#0-5) 
- `@Size(max = 2)` on `timestamps` allows at most two timestamp parameters but imposes no minimum range width and no maximum span. [7](#0-6) 
- `HIGH_VOLUME_THROTTLE` is defined in `Constants` but is never applied to `HooksController`. [8](#0-7) 
- No `statement_timeout`, `lock_timeout`, or query-level timeout is visible in the application configuration.

### Impact Explanation
A single crafted request forces a full sequential scan + sort of potentially billions of rows in `hook_storage_change`, holding a database connection and transaction open for minutes. With a handful of concurrent requests (easily automated), the database connection pool is exhausted, all other REST API queries queue and time out, and the mirror node becomes unavailable. Because the mirror node is the canonical read path for clients verifying transaction finality, this constitutes a network-wide denial of service for transaction confirmation.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The attacker needs only a valid `ownerId` (an account ID, publicly enumerable from the ledger) and any `hookId` (a small integer, trivially brute-forced). The exploit requires a single HTTP GET request with no special privileges, tools, or credentials. It is trivially repeatable and scriptable, making sustained DoS straightforward.

### Recommendation
1. **Enforce a maximum timestamp span**: Require that when `timestamp` is provided, both a lower and upper bound must be supplied and their difference must not exceed a configurable maximum (e.g., 7 days in nanoseconds).
2. **Apply a DB-level statement timeout**: Set `statement_timeout` (e.g., 5 seconds) on the read-replica connection pool used by `rest-java` so runaway queries are killed automatically.
3. **Apply rate limiting**: Wire `HIGH_VOLUME_THROTTLE` (already defined in `Constants`) to the `HooksController` storage endpoint.
4. **Rewrite the query to use a bounded subquery**: Use a `LIMIT`-inside-subquery pattern or a cursor-based approach so the planner can use an index scan that stops after `N` distinct keys rather than scanning the full range.

### Proof of Concept
```
# Step 1: Identify any valid ownerId and hookId (publicly visible on-chain)
# Step 2: Send the following request (no auth required):

GET /api/v1/accounts/0.0.12345/hooks/1/storage?timestamp=gte:0&limit=100

# This sets:
#   keyLowerBound = 0x0000...00 (MIN_KEY_BYTES, 32 bytes)
#   keyUpperBound = 0xFFFF...FF (MAX_KEY_BYTES, 32 bytes)
#   timestampLowerBound = 0
#   timestampUpperBound = Long.MAX_VALUE (9223372036854775807)
#
# The resulting SQL scans every row in hook_storage_change for (owner_id=12345, hook_id=1),
# sorts them all by (key ASC, consensus_timestamp DESC), deduplicates, then returns 100 rows.
#
# Step 3: Send 20–50 concurrent copies of this request to exhaust the DB connection pool.
# Step 4: All other mirror node API queries begin timing out.
```

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
