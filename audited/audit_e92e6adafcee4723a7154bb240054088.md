### Title
Cache Stampede (Thundering Herd) via Unprotected Concurrent Cache Miss on Transaction Endpoints

### Summary
`responseCacheCheckHandler` contains no in-flight request deduplication, locking, or coalescing mechanism. When the Redis cache entry for a given transaction URL is absent or expired, every concurrent request that arrives before the first response is written back to Redis independently queries the database. An unprivileged attacker can deliberately trigger this window to flood the database with identical queries, causing denial of service for transaction status lookups.

### Finding Description

**Exact code path:**

`rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler`, lines 40–87; `responseCacheUpdateHandler`, lines 90–119.

**Root cause — no stampede guard in `responseCacheCheckHandler`:**

```js
// line 43-47
const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey);

if (!cachedTtlAndValue) {
  res.locals[responseCacheKeyLabel] = responseCacheKey;
  return;   // ← all concurrent misses fall through here, no lock, no dedup
}
```

On a cache miss, the handler sets the cache key in `res.locals` and immediately returns, allowing the request to proceed to the route handler (database). There is no:
- Redis `SET NX` / distributed lock to mark "fetch in progress"
- In-process promise map to coalesce concurrent identical requests
- Rate limiting (confirmed absent from all REST middleware)

**Cache write path (`responseCacheUpdateHandler`, line 116):**
```js
await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
```
`setSingle` uses a plain `SETEX` — no `NX` flag, no check for an existing in-flight marker. All N concurrent requests write the same value redundantly after independently querying the database.

**Cache key generation (line 151–153):**
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
All requests to the same URL produce the identical cache key, confirming that concurrent requests during a miss window all race to the same uncached resource.

**Middleware registration order (server.js lines 94–141):**
```
responseCacheCheckHandler  →  [route handler / DB query]  →  responseHandler  →  responseCacheUpdateHandler
```
Every request that passes `responseCacheCheckHandler` during a miss window reaches the DB independently.

**Exploit flow:**
1. Attacker waits for (or forces via cache expiry) a cache miss on a high-cost transaction endpoint, e.g. `GET /api/v1/transactions/0.0.1234@1234567890.000000000`.
2. Attacker sends N concurrent HTTP GET requests to that URL before any single response has been written back to Redis.
3. All N requests pass through `responseCacheCheckHandler` (each sees `cachedTtlAndValue === undefined`), each sets `res.locals[responseCacheKeyLabel]`, and each proceeds to the database.
4. The database receives N identical queries simultaneously.
5. All N responses are written to Redis redundantly.
6. Attacker repeats by waiting for TTL expiry (default `max-age` from `Cache-Control`, minimum `DEFAULT_REDIS_EXPIRY = 1` second per line 24).

### Impact Explanation

The database is flooded with N identical queries for every burst of N concurrent requests during a cache miss window. Because `DEFAULT_REDIS_EXPIRY` can be as low as 1 second (line 24), an attacker can sustain this at ~1 burst/second. With sufficiently large N (e.g., 500–1000 concurrent connections from a single machine), the PostgreSQL connection pool is exhausted, causing transaction status lookups to time out or fail for all legitimate users — a practical denial of service. The impact is amplified on endpoints with expensive joins (e.g., `getTransactionsByIdOrHash`).

### Likelihood Explanation

- **No authentication required**: transaction read endpoints are public.
- **No rate limiting**: confirmed absent from all REST middleware.
- **Trivially reproducible**: a single attacker machine with a standard HTTP benchmarking tool (`wrk`, `ab`, `hey`) can generate hundreds of concurrent requests.
- **Repeatable**: the attack window recurs every TTL seconds automatically.
- **Low skill required**: no exploit code needed beyond standard HTTP tooling.

### Recommendation

1. **Distributed lock / promise coalescing**: Before falling through on a cache miss, attempt a Redis `SET NX PX <short_ttl>` on a "lock key" derived from the cache key. Only the request that acquires the lock proceeds to the DB; others either wait and re-check, or return a 503/retry-after.
2. **In-process request coalescing**: Maintain a `Map<cacheKey, Promise>` of in-flight DB fetches. Subsequent requests for the same key attach to the existing promise rather than issuing a new DB query.
3. **Stale-while-revalidate**: Serve the stale cached value while a single background refresh is triggered, eliminating the miss window entirely.
4. **Rate limiting**: Add per-IP or global rate limiting middleware before `responseCacheCheckHandler`.

### Proof of Concept

```bash
# 1. Ensure cache is cold (or wait for TTL expiry)
redis-cli FLUSHALL

# 2. Send 500 concurrent requests to the same transaction endpoint
#    (requires no credentials — public endpoint)

# 3. Observe in PostgreSQL:
SELECT count(*), query
FROM pg_stat_activity
WHERE state = 'active'
GROUP BY query;
# → 500 identical transaction queries executing simultaneously

# 4. Repeat every ~1 second to sustain the DoS:

done

const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
