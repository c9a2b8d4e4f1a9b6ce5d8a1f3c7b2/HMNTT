### Title
Unbounded `spender.id=eq:X` IN-clause in NFT Query Enables Non-Network DoS via Sequential Scan

### Summary
The `getSubQuery()` function in `rest/service/nftService.js` builds a `spender IN ($5,...,$N)` SQL condition from all `spender.id=eq:X` query parameters without any cap on the number of `eq` values. The `validateFilters()` function in `accountController.js` only validates range operators (`lt/lte/gt/gte`) and does not limit the count of `eq` operators. An unprivileged attacker can supply many `spender.id=eq:X` values with IDs spread across a wide range, causing PostgreSQL to abandon index usage on the `spender` column and perform a full sequential scan of all NFTs owned by the target account, exhausting database CPU.

### Finding Description

**Exact code path:**

In `rest/controllers/accountController.js`, `extractNftMultiUnionQuery()` routes every `spender.id` filter with operator `eq` into `spenderIdInFilters` (line 59), which is an unbounded array:

```js
case filterKeys.SPENDER_ID:
  filter.operator === utils.opsMap.eq ? spenderIdInFilters.push(filter) : spenderIdFilters.push(filter);
  break;
``` [1](#0-0) 

`validateFilters()` only checks that there are not multiple `lt/lte` or `gt/gte` operators in `spenderIdFilters`. It performs **no check** on the size of `spenderIdInFilters`:

```js
validateFilters(bounds, spenderIdFilters) {
  this.validateBounds(bounds);
  const spenderOperators = spenderIdFilters.map((f) => f.operator);
  if (
    spenderOperators.filter((o) => o === utils.opsMap.lte || o === utils.opsMap.lt).length > 1 ||
    spenderOperators.filter((o) => o === utils.opsMap.gte || o === utils.opsMap.gt).length > 1
  ) {
    throw new InvalidArgumentError(`Multiple range params not allowed for spender.id`);
  }
  ...
}
``` [2](#0-1) 

The entire `spenderIdInFilters` array is then passed to `NftService.getSubQuery()`, which builds the `IN` clause dynamically with no size guard:

```js
if (!isEmpty(spenderIdInFilters)) {
  const paramsForCondition = spenderIdInFilters.map((filter) => {
    params.push(filter.value);
    return `$${params.length}`;
  });
  conditions.push(`${Nft.SPENDER} in (${paramsForCondition})`);
}
``` [3](#0-2) 

The only global guard is `maxRepeatedQueryParameters` in `buildAndValidateFilters`, which is a configurable value (tests set it to 4 to trigger the error, implying the default is higher). The integration test `spenderid-in-clause.json` explicitly confirms that 4 `eq` values are accepted and return HTTP 200:

```
/api/v1/accounts/0.0.1001/nfts?spender.id=2000&spender.id=3000&spender.id=4000&spender.id=5000
``` [4](#0-3) 

**Root cause:** The design intentionally allows multiple `eq` values to form an `IN` clause (for legitimate multi-spender filtering), but no upper bound is enforced on the count of `eq` values beyond the global `maxRepeatedQueryParameters` setting. When that setting is permissive (e.g., 10–100), an attacker can supply enough widely-spread IDs to make the `IN` clause non-selective, defeating any index on `spender` and forcing a full scan of all NFTs for the target account.

**Failed assumption:** The design assumes that the `IN` clause will remain small enough to be index-friendly. PostgreSQL's planner may choose a sequential scan when the `IN` list is large and the values are spread across the full ID space, because the estimated cost of multiple index lookups exceeds a sequential scan.

### Impact Explanation
For accounts holding large numbers of NFTs, each crafted request forces the database to scan every row in the `nft` table for that `account_id`, applying the large `IN` filter row-by-row. Because the query includes `ORDER BY token_id, serial_number LIMIT $2`, the database cannot short-circuit early if the matching spender values are rare or absent. Repeated concurrent requests from one or more attackers can saturate database CPU, degrading or denying service for all mirror-node users. This is a non-network DoS requiring no authentication.

### Likelihood Explanation
The endpoint is public and unauthenticated. The attack requires only an HTTP client and knowledge of the API (publicly documented). The attacker does not need to know valid spender IDs — using non-existent IDs maximises scan depth. The attack is trivially repeatable and can be parallelised. The only friction is the `maxRepeatedQueryParameters` ceiling, whose default value determines the practical upper bound on the `IN` list size.

### Recommendation
1. **Hard-cap `eq` occurrences for `spender.id` on the NFT endpoint.** Add a check in `validateFilters()` (or `extractNftMultiUnionQuery()`) that rejects requests where `spenderIdInFilters.length` exceeds a small constant (e.g., 10).
2. **Set a low, explicit default for `maxRepeatedQueryParameters`** and document it as a security boundary.
3. **Add a database-level statement timeout** so runaway queries are killed before they exhaust CPU.
4. **Consider rewriting the multi-`eq` case** as a range query (`spender IN (sorted list)`) and ensure a partial index on `(account_id, spender)` exists so PostgreSQL can use it efficiently.
