### Title
Unbounded `data` Field Allocation Enables Heap Exhaustion DoS via POST /api/v1/contracts/call

### Summary
The `data` field in `ContractCallRequest` permits up to 1,048,576 hex characters (512 KB decoded) per request, validated only after Jackson has already allocated the full String in heap memory. The throttle check executes inside the method body — after deserialization — so even throttle-rejected requests cause ~2 MB of heap allocation per connection. An unprivileged attacker sending concurrent requests at the permitted rate (500 req/s) can sustain over 1 GB/s of heap allocation, triggering OOM and crashing the web3 service.

### Finding Description

**Exact code path:**

`ContractCallRequest.java` line 28–29 — the `data` field carries only `@Hex` with the default `maxLength = 1048576L`: [1](#0-0) 

`Hex.java` line 27 — the annotation's default `maxLength` is 1,048,576 hex characters (= 524,288 decoded bytes): [2](#0-1) 

`HexValidator.java` lines 26–31 — validation runs a regex on the already-deserialized String; it does not prevent Jackson from allocating the String first: [3](#0-2) 

`BytesDecoder.hexToBytes()` lines 79–89 — no size guard; allocates a `byte[]` of `hexString.length() / 2` unconditionally: [4](#0-3) 

`ContractController.call()` lines 38–51 — `@RequestBody @Valid` causes deserialization + Bean Validation before the method body runs; `throttleManager.throttle()` is called only afterward, at line 40: [5](#0-4) 

`constructServiceParameters()` line 80 — `hexToBytes(data)` allocates the decoded byte array only if throttle passes, but the String is already live in heap: [6](#0-5) 

**Root cause and failed assumption:**

The design assumes the throttle gate protects against resource exhaustion. It does not. Spring MVC resolves `@RequestBody @Valid` arguments — including full JSON deserialization and Bean Validation — before the method body executes. The throttle check at line 40 is therefore post-allocation. A request carrying 1,048,576 hex characters causes Jackson to allocate a ~2 MB UTF-16 `String` object regardless of whether the throttle subsequently rejects it.

No HTTP-level body-size limit is configured anywhere in the repository (no `application.yml`/`application.properties` with `server.tomcat.max-http-form-post-size`, `spring.servlet.multipart.max-request-size`, or equivalent was found): [7](#0-6) 

**Throttle analysis:**

`ThrottleProperties` defaults: `requestsPerSecond = 500`, `gasPerSecond = 7_500_000_000`. The throttle is keyed on gas and request count, not on payload byte size: [8](#0-7) 

`ThrottleManagerImpl.throttle()` — no data-size dimension: [9](#0-8) 

`scaleGas()` returns 0 for gas ≤ 10,000, meaning requests with gas = 21,000 consume only 2 gas-bucket tokens — negligible against the 750,000-token/s gas bucket: [10](#0-9) 

### Impact Explanation

At 500 req/s (the rate limit) with `data` = 1,048,576 hex chars each:
- Per request: ~2 MB String (UTF-16) + ~512 KB `byte[]` from `hexToBytes` = ~2.5 MB heap
- Sustained rate: ~1.25 GB/s of heap allocation

A typical JVM heap of 512 MB–2 GB is exhausted within 1–2 seconds. OOM kills the Spring Boot process, taking down the entire web3 service. Because this service is the gateway for `eth_call` / `eth_estimateGas` queries used by wallets and dApps to confirm transaction state, a crash prevents all such confirmations — matching the "total network shutdown" severity tier for this component.

### Likelihood Explanation

- **No authentication required.** Any external client can POST to `/api/v1/contracts/call`.
- **Trivially scriptable.** A single `curl` loop or a tool like `wrk`/`hey` with a pre-built 1 MB hex payload suffices.
- **Rate limit is not a barrier.** 500 req/s is the attacker's budget, not a protection — it is exactly the throughput needed to sustain the attack. Moreover, because deserialization precedes the throttle check, flooding above 500 req/s still causes String allocation for every connection that reaches the Tomcat thread pool (default: 200 threads), amplifying the effect.
- **Repeatable.** The attack is stateless and can be restarted immediately after a service restart.

### Recommendation

1. **Add an HTTP-level body-size cap** (e.g., `server.tomcat.max-http-form-post-size=100KB` in `application.yml`) so Tomcat rejects oversized bodies before Jackson runs.
2. **Reduce `@Hex` maxLength on `data`** to a value consistent with the EVM's practical calldata limit (e.g., 24 KB for contract bytecode, or the Ethereum block gas limit equivalent). The current 512 KB decoded limit has no EVM justification.
3. **Move the throttle check to a servlet filter** (before deserialization) or use a `HandlerInterceptor` that reads `Content-Length` and rejects large requests before the body is read.
4. **Add a per-IP rate limit** in addition to the global bucket to prevent a single attacker from consuming the entire 500 req/s budget.

### Proof of Concept

```bash
# Generate a 1,048,576-character hex payload (512 KB decoded, within @Hex maxLength)
PAYLOAD="0x$(python3 -c "print('ab' * 524288)")"

# Build the JSON body
BODY=$(printf '{"to":"0x00000000000000000000000000000000000004e2","gas":21000,"data":"%s"}' "$PAYLOAD")

# Flood with 500 concurrent requests (matches the rate limit)
for i in $(seq 1 500); do
  curl -s -X POST http://<host>:8545/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "$BODY" &
done
wait

# Repeat in a tight loop; within seconds the JVM heap is exhausted and the
# service throws OutOfMemoryError, dropping all subsequent requests.
```

Each iteration allocates ~2 MB String + ~512 KB byte array per request. At 500 concurrent requests sustained, heap exhaustion and OOM occur within 1–2 seconds on a standard deployment.
