### Title
Unbounded Concurrent Session Accumulation via Shared Scheduler Saturation Defeats Per-Session Timeout in `PollingTopicMessageRetriever`

### Summary
`PollingTopicMessageRetriever.retrieve()` uses a single shared `boundedElastic` scheduler for both polling tasks and timeout signals. Because the only per-connection call limit (`maxConcurrentCallsPerConnection = 5`) does not bound the total number of connections, an unprivileged attacker can open arbitrarily many connections and accumulate concurrent sessions until the shared scheduler's thread pool saturates. Once saturated, timeout signals queue behind polling tasks on the same scheduler, delaying or preventing the 60-second timeout from firing, which extends session lifetimes, exhausts the database connection pool, and causes progressive service degradation leading to total unavailability.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve(filter, true)` → `PollingTopicMessageRetriever.retrieve()` [1](#0-0) 

The scheduler is a single shared instance created once at construction:

```java
scheduler = Schedulers.boundedElastic();
``` [2](#0-1) 

Every call to `retrieve()` submits polling tasks to this same scheduler via `withScheduler(scheduler)`, and critically, the timeout signal is also scheduled on the **same** scheduler:

```java
.repeatWhen(RepeatSpec.create(...).withScheduler(scheduler))
...
.timeout(retrieverProperties.getTimeout(), scheduler)
``` [3](#0-2) 

In throttled mode, `numRepeats = Long.MAX_VALUE`, meaning each session polls indefinitely until the timeout fires: [4](#0-3) 

Additionally, `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` retries DB errors indefinitely, compounding scheduler load: [5](#0-4) 

**Root cause — the failed assumption:** The design assumes the 60-second timeout will reliably terminate runaway sessions. This assumption breaks when the scheduler that must deliver the timeout signal is itself saturated by the polling tasks of those same sessions. `Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads with a 100,000-task queue. When threads are exhausted, timeout `Mono` signals queue behind polling tasks and are delayed arbitrarily — sessions live far beyond 60 seconds.

**Why the only existing guard fails:**

`maxConcurrentCallsPerConnection = 5` is enforced per-connection, not globally: [6](#0-5) [7](#0-6) 

No global connection limit, no per-IP rate limit, and no total concurrent session cap exist anywhere in the gRPC module. The `subscriberCount` field is a metric gauge only — it is never checked against a ceiling: [8](#0-7) 

The `ThrottleConfiguration` with Bucket4j rate limiting exists only in the `web3` module, not in `grpc`: [9](#0-8) 

### Impact Explanation

Once the shared `boundedElastic` scheduler saturates:
1. Timeout signals queue behind polling tasks — sessions live beyond 60 seconds.
2. Each live session issues a DB query every 2 seconds via `topicMessageRepository.findByFilter()`, exhausting the HikariCP connection pool.
3. DB connection exhaustion causes all new legitimate `subscribeTopic()` calls to block or fail.
4. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` clause causes sessions that encounter DB errors to retry indefinitely, adding more scheduler tasks and worsening saturation.
5. The service becomes fully unavailable to legitimate users — total gRPC service shutdown.

The infrastructure monitoring alerts for `GrpcHighDBConnections` (>75%) and `GrpcHighCPU` (>80%) only alert after 5 minutes, by which time the attack is already effective. [10](#0-9) 

### Likelihood Explanation

**Preconditions:** None. No authentication, no API key, no privileged access required. The gRPC port (5600) is publicly exposed.

**Attacker capability:** A single machine with a standard gRPC client library (e.g., `grpc-java`, `grpcurl`) can open hundreds of TCP connections. Each connection supports 5 concurrent streaming calls. 200 connections × 5 calls = 1,000 concurrent sessions, each polling the DB every 2 seconds = 500 DB queries/second sustained.

**Repeatability:** The attack is fully repeatable and self-sustaining. The attacker does not need to reconnect before timeout fires — the timeout itself is delayed by the saturation the attacker creates. The attack is also stateless from the attacker's perspective: open connections, send `subscribeTopic` requests with `startTime=0` and no `endTime`/`limit`, and hold them open.

### Recommendation

1. **Decouple timeout scheduler from polling scheduler.** Use a dedicated `Schedulers.single()` or `Schedulers.parallel()` scheduler exclusively for timeout signals, separate from the `boundedElastic` scheduler used for polling. This ensures timeout delivery is not blocked by polling task backlog.

2. **Enforce a global concurrent session cap.** Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.

3. **Add per-IP or global gRPC connection rate limiting.** Configure `NettyServerBuilder.maxConnectionsPerIp()` or add a gRPC server interceptor that enforces per-client connection limits.

4. **Add a global `maxConnections` limit** to `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge()` and `maxConnectionIdle()` to bound the lifetime and idle time of connections.

5. **Remove `Retry.backoff(Long.MAX_VALUE, ...)`.** Indefinite retry on DB errors under scheduler pressure is a force multiplier for the attack. Cap retries to a small finite number.

### Proof of Concept

```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Assumes proto is available; adapt topic ID to a valid one

# Step 1: Open 200 connections, each with 5 concurrent streaming calls (1000 sessions)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"shardNum":0,"realmNum":0,"topicNum":1},"consensusStartTime":{"seconds":0}}' \
      <grpc-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 2: Observe that after ~30-60 seconds, new legitimate connections begin
# receiving RESOURCE_EXHAUSTED or hanging indefinitely (DB pool exhausted).

# Step 3: Monitor DB connections:
# hikaricp_connections_active / hikaricp_connections_max → approaches 1.0

# Step 4: Observe that existing attacker sessions do NOT terminate at 60s
# (timeout signal queued behind polling tasks on saturated scheduler).
```
