### Title
Uncached EVM Address DB Lookup in `getHooks()` Enables Unauthenticated Resource Exhaustion

### Summary
Every call to `GET /api/v1/accounts/{ownerId}/hooks` with an EVM address `ownerId` triggers an uncached, direct SQL query against the `entity` table via `EntityServiceImpl.lookup()`. The `rest-java` module has no rate limiting and no caching on this code path, allowing any unauthenticated attacker to drive unbounded repeated DB queries by flooding the endpoint with valid EVM address parameters.

### Finding Description
**Exact code path:**

`HooksController.getHooks()` → `HookServiceImpl.getHooks()` (line 35) → `EntityServiceImpl.lookup()` (lines 31–37) → `entityRepository.findByEvmAddress()` (line 34) → native SQL query.

In `HookServiceImpl.java` line 35:
```java
final var id = entityService.lookup(request.getOwnerId());
```

In `EntityServiceImpl.java` lines 31–37:
```java
var id = switch (accountId) {
    case EntityIdNumParameter p -> Optional.of(p.id());
    case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
    case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
};
```

In `rest-java`'s `EntityRepository.java` lines 16–17:
```java
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
```

**Root cause:** The `rest-java` `EntityRepository.findByEvmAddress()` carries **no `@Cacheable` annotation** — unlike the `web3` module's `EntityRepository` which annotates `findByEvmAddressAndDeletedIsFalse` with `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`, and unlike the `importer` module's `EntityIdServiceImpl` which wraps all EVM address lookups in a `cacheLookup()` call. Every invocation of `EntityServiceImpl.lookup()` with an `EntityIdEvmAddressParameter` unconditionally hits the database.

**No rate limiting in rest-java:** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `RequestFilter`) exists exclusively in the `web3` module. The `rest-java` config directory contains only `LoggingFilter`, `MetricsFilter`, `JacksonConfiguration`, `WebMvcConfiguration`, `RestJavaConfiguration`, `MetricsConfiguration`, `NetworkProperties`, and `RuntimeHintsConfiguration` — no rate-limiting filter of any kind.

**Exploit flow:**
1. Attacker identifies the public endpoint `GET /api/v1/accounts/{ownerId}/hooks`.
2. Attacker supplies a valid EVM address as `ownerId` (e.g., `0xabcdef1234567890abcdef1234567890abcdef12`), which is parsed into `EntityIdEvmAddressParameter` by `EntityIdParameter.valueOf()`.
3. On each request, `EntityServiceImpl.lookup()` dispatches to the `EntityIdEvmAddressParameter` branch and calls `entityRepository.findByEvmAddress()`, issuing a native SQL query.
4. No cache intercepts the call; no rate limiter rejects or throttles the request.
5. Attacker sends thousands of requests per second from one or more clients, each consuming a DB connection and executing the query.

### Impact Explanation
Each request consumes a database connection from the pool and executes a SQL query. At high request rates the DB connection pool saturates, query latency increases for all users, and CPU/IO on the database host rises. Because the `entity` table is large in production (millions of rows), even an indexed `evm_address` lookup carries non-trivial per-query overhead at scale. Sustained flooding can push DB CPU and connection utilization well above the 30% threshold relative to a quiet 24-hour baseline, degrading or denying service to legitimate users of all rest-java endpoints that share the same DB pool.

### Likelihood Explanation
The endpoint is fully public — no authentication, no API key, no session. Any attacker with network access can exploit it. The only precondition is knowing a syntactically valid 40-hex-character EVM address (trivially satisfied; the address need not exist in the DB — a 404 `EntityNotFoundException` is thrown after the DB query completes, meaning the query still executes). The attack is trivially scriptable with `curl` or any HTTP load tool and is repeatable indefinitely.

### Recommendation
1. **Add caching** to `EntityServiceImpl.lookup()` for the `EntityIdEvmAddressParameter` branch, mirroring the pattern used in the `web3` module (`@Cacheable` on `findByEvmAddress`) or the `importer` module (`cacheLookup()` wrapper with a Caffeine/Spring cache).
2. **Add rate limiting** to the `rest-java` module. Implement a servlet filter or Spring interceptor (analogous to `web3`'s `ThrottleManagerImpl`) that enforces a per-IP or global requests-per-second cap on all `rest-java` endpoints.
3. As a short-term mitigation, consider returning early (before the DB call) when the EVM address resolves to a known-nonexistent entity, or caching negative results with a short TTL.
