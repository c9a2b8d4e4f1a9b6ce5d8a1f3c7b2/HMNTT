### Title
Resource Amplification via Maximum-Unique-Timestamp Contract Logs Request Chaining Two Heavy DB Queries

### Summary
An unprivileged external user can craft a single `GET /api/v1/contracts/{id}/results/logs?limit=100` request that returns exactly `limit` rows each with a unique `consensus_timestamp`. The `getContractLogs()` function's deduplication loop then passes an array of size=`limit` to `RecordFileService.getRecordFileBlockDetailsFromTimestampArray()`, chaining a maximum-cost contract-log query with a maximum-cost correlated-subquery record-file lookup, amplifying per-request DB work without any rate control on the combined cost.

### Finding Description

**Code path:**

`getContractLogs()` in `rest/service/contractService.js` lines 376–393:

```js
async getContractLogs(query) {
  const [sqlQuery, params] = this.getContractLogsQuery(query);
  const rows = await super.getRows(sqlQuery, params);          // Query 1
  if (rows.length === 0) { return rows; }

  const timestamps = [];
  rows.forEach((row) => {
    if (row.consensus_timestamp !== timestamps[timestamps.length - 1]) {
      timestamps.push(row.consensus_timestamp);                // dedup consecutive only
    }
  });
  const recordFileMap =
    await RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps); // Query 2
  return rows.map((cr) => new ContractLog(cr, recordFileMap.get(cr.consensus_timestamp)));
}
```

**Root cause — failed assumption:** The deduplication at lines 385–388 only removes *consecutive* duplicates (it compares each row's timestamp to the last element pushed). When all `limit` rows carry distinct `consensus_timestamp` values — the normal case for a contract that emits one log per transaction — the `timestamps` array grows to exactly `limit` entries and is passed verbatim to `getRecordFileBlockDetailsFromTimestampArray()`.

**Query 2 cost:** `recordFileBlockDetailsFromTimestampArrayQuery` in `rest/service/recordFileService.js` lines 23–43 uses a correlated subquery executed once per timestamp via `unnest($1::bigint[])`:

```sql
where consensus_end in (
  select (
    select consensus_end from record_file
    where consensus_end >= timestamp and consensus_end >= $2 and consensus_end <= $3
    order by consensus_end limit 1
  ) as consensus_end
  from (select unnest($1::bigint[]) as timestamp) as tmp
  group by consensus_end
)
```

With 100 timestamps this is 100 index-range scans on `record_file` inside a single query — the maximum possible work for this function.

**Limit cap:** `getLimitParamValue()` in `rest/utils.js` lines 544–553 caps unauthenticated users at `responseLimit.max` (default **100**). This is the only guard; there is no per-endpoint rate limit on the combined two-query cost.

### Impact Explanation
Every request at `limit=100` with all-unique timestamps forces the mirror node's PostgreSQL backend to execute: (1) a full contract-log scan returning 100 rows, and (2) 100 correlated index lookups on `record_file`. A moderate stream of such requests (e.g., 10–20 req/s, well below typical DDoS thresholds) multiplies DB I/O and CPU proportionally. On a lightly-to-moderately loaded mirror node — the common deployment profile — this can push DB utilisation above the 30 % delta threshold without triggering volume-based defences.

### Likelihood Explanation
The precondition (a contract with ≥ 100 logs each at a distinct timestamp) is trivially satisfied on mainnet or any active testnet. No authentication, no special privilege, and no brute-force volume is required — a single HTTP GET with `?limit=100` is sufficient to trigger the worst-case two-query chain. The attack is fully repeatable and scriptable.

### Recommendation
1. **Cap the timestamp array before the second query:** after deduplication, if `timestamps.length` exceeds a configurable threshold (e.g., 25), batch the `getRecordFileBlockDetailsFromTimestampArray` calls or reject/paginate.
2. **Apply per-endpoint cost-based rate limiting** that accounts for the combined query cost, not just request count.
3. **Rewrite the correlated subquery** in `recordFileBlockDetailsFromTimestampArrayQuery` to a single range scan with a lateral join, eliminating the per-timestamp index probe.

### Proof of Concept

**Precondition:** Contract `0.0.X` has ≥ 100 log entries, each emitted in a separate transaction (unique `consensus_timestamp`). This is true for any active ERC-20/ERC-721 contract on mainnet.

**Trigger:**
```
GET /api/v1/contracts/0.0.X/results/logs?limit=100
```

**Observed execution in `getContractLogs()`:**
1. Query 1 returns 100 rows, each with a distinct `consensus_timestamp`.
2. The `forEach` dedup loop pushes all 100 timestamps (no consecutive duplicates to remove).
3. `getRecordFileBlockDetailsFromTimestampArray([ts_1, ts_2, …, ts_100])` executes the correlated subquery 100 times inside one SQL statement.
