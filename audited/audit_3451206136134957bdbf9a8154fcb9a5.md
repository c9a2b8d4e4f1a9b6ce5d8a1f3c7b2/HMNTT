### Title
Unbounded gRPC Topic Subscription DoS via `startTime=0 / endTime=Long.MAX_VALUE / limit=0`

### Summary
Any unauthenticated caller can open a `subscribeTopic` gRPC stream with `startTime=0`, `endTime=Long.MAX_VALUE`, and `limit=0`. All three values pass every input-validation check. The resulting subscription enters a live phase that **never terminates** because `TopicContext.isComplete()` evaluates `endTime` as a timestamp in the year ~2262 and always returns `false`. During the preceding historical phase the retriever executes repeated DB queries for up to 60 seconds. Because there is no global cap on concurrent subscriptions and `maxConcurrentCallsPerConnection` only limits calls per TCP connection, an attacker opening many connections can accumulate an unbounded number of never-terminating subscriptions, exhausting gRPC threads, listener slots, and DB connection-pool capacity.

### Finding Description

**Input validation — all three parameters accepted**

`ConsensusController.toFilter()` (lines 55–73) builds a `TopicMessageFilter` from the raw protobuf. `TopicMessageFilter` enforces:

- `@Min(0)` on `limit` → `limit=0` passes. [1](#0-0) 
- `isValidStartTime()`: `startTime <= DomainUtils.now()` → `0 <= now` is true. [2](#0-1) 
- `isValidEndTime()`: `endTime > startTime` → `Long.MAX_VALUE > 0` is true. [3](#0-2) 
- `hasLimit()` returns `false` when `limit == 0`, so no `.take()` is ever applied to the flux. [4](#0-3) 

**Historical phase — bounded but resource-consuming**

`TopicMessageServiceImpl.subscribeTopic()` passes the filter to `topicMessageRetriever.retrieve(filter, true)` (throttled). [5](#0-4) 

`PollingTopicMessageRetriever` with `throttled=true` sets `numRepeats = Long.MAX_VALUE` and polls every 2 s with pages of 1 000 rows. [6](#0-5) 

`isComplete()` (throttled path) returns `true` only when the last page was smaller than `maxPageSize` or the limit was hit. With `limit=0` (`hasLimit()=false`), `limitHit` is always `false`, so the retriever keeps paging as long as full pages are returned. [7](#0-6) 

A 60-second `.timeout()` does eventually kill the historical flux. [8](#0-7) 

**Live phase — never terminates**

After the historical flux completes (or times out), the subscription transitions to the live flux via `concatWith`. The live flux is gated by `takeUntilOther(pastEndTime(topicContext))`. [9](#0-8) 

`pastEndTime()` completes only when `topicContext.isComplete()` returns `true`. [10](#0-9) 

`TopicContext.isComplete()` with `endTime = Long.MAX_VALUE`:

```java
return Instant.ofEpochSecond(0, filter.getEndTime())   // ≈ year 2262
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());                       // always false in 2026
``` [11](#0-10) 

`isComplete()` never returns `true`, so `pastEndTime()` never completes, `takeUntilOther` never fires, and the live subscription **runs forever**.

**Safety-check burst**

One second after subscription start, the safety-check fires an *unthrottled* retrieval (up to 12 polls × 5 000 rows at 20 ms intervals) against the DB. [12](#0-11) 

**No global subscriber cap**

`subscriberCount` is a metric gauge only — it is never checked against a maximum. [13](#0-12) 

`maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, not globally. [14](#0-13) 

### Impact Explanation

Each malicious subscription:
1. Executes up to ~30 DB queries (1 000 rows each) over 60 seconds during the historical phase.
2. Triggers one unthrottled burst of up to 60 000 rows (12 × 5 000) via the safety check.
3. Holds a live listener slot and a gRPC server thread indefinitely.

An attacker opening *K* TCP connections can hold *5K* never-terminating subscriptions open simultaneously. With a small DB connection pool (default 10 for the gRPC service per `hiero.mirror.grpc.db.statementTimeout` configuration), even a handful of connections can saturate the pool, causing all legitimate subscribers to time out. This constitutes a complete, unauthenticated denial-of-service against the HCS topic subscription API.

### Likelihood Explanation

The attack requires only a gRPC client library (freely available in every major language) and knowledge of any valid topic ID (all topic IDs are public on-chain). No credentials, tokens, or special network position are needed. The exploit is trivially scriptable and repeatable. The `maxConcurrentCallsPerConnection=5` default provides no meaningful barrier since an attacker can open as many TCP connections as the OS allows.

### Recommendation

1. **Enforce a maximum subscription duration**: add a hard wall-clock timeout to the live flux (e.g., `flux.timeout(grpcProperties.getMaxSubscriptionDuration())`).
2. **Enforce a global concurrent-subscriber cap**: check `subscriberCount` against a configurable maximum before allowing a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Reject or cap `endTime` far in the future**: add a validation that `endTime` must be within a configurable window of `now` (e.g., `endTime <= DomainUtils.now() + maxFutureWindow`).
4. **Require `limit > 0` for historical-only queries** (where `startTime` is in the past): if `startTime < now` and `endTime` is set, require a non-zero limit.
5. **Rate-limit subscriptions per source IP** at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

def open_zombie_subscription(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # endTime seconds >= 9223372035 → convertTimestamp returns Long.MAX_VALUE
        consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372036, nanos=0),
        limit=0,   # hasLimit() == false → no take(), no limitHit
    )
    # Returns a never-completing iterator; hold reference to keep stream alive
    return stub.subscribeTopic(query)

# Open many connections, 5 subscriptions each
channels = [grpc.insecure_channel("mirror-node-grpc:5600") for _ in range(200)]
streams  = [open_zombie_subscription(ch) for ch in channels for _ in range(5)]
# 1000 never-terminating subscriptions; DB connection pool exhausted within seconds
for s in streams:
    next(s)   # force stream establishment
```

Each stream enters the live phase after ~60 s and never terminates. The DB connection pool is saturated during the historical phase; the gRPC thread pool is saturated permanently thereafter.
