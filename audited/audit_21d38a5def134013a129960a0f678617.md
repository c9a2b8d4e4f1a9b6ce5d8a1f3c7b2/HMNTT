### Title
Unbounded Concurrent Subscribers Exhaust Database Connection Pool via Independent `poll()` Calls in `PollingTopicListener`

### Summary
When the `POLL` listener type is configured, each subscriber to `PollingTopicListener` triggers its own independent, periodic `poll()` call that executes a blocking JPA `getResultList()` query, holding a HikariCP connection for the full query duration. Because there is no limit on concurrent subscribers and no authentication is required, an unprivileged attacker can open arbitrarily many subscriptions, exhausting the database connection pool and causing connection timeouts for all users.

### Finding Description

**Exact code path:**

`PollingTopicListener.java`, `poll()`, lines 51–62:

```java
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessageFilter filter = context.getFilter();
    TopicMessage last = context.getLast();
    int limit = filter.hasLimit()
            ? (int) (filter.getLimit() - context.getCount().get())
            : Integer.MAX_VALUE;                          // no limit → Integer.MAX_VALUE
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // capped at 5000
    long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
    var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter)); // blocking DB call
}
```

`TopicMessageRepositoryCustomImpl.java`, `findByFilter()`, line 60:
```java
return typedQuery.getResultList().stream(); // synchronous, holds DB connection until complete
```

`ListenerProperties.java`, lines 26–30:
```java
private int maxPageSize = 5000;
private Duration interval = Duration.ofMillis(500L);
```

**Root cause and failed assumption:**

The design assumes a small, bounded number of subscribers. Each call to `listen()` creates a new `PollingContext` and schedules an independent `poll()` every 500 ms on `Schedulers.boundedElastic()`. Each `poll()` executes a synchronous JPA `getResultList()` call (not reactive/async), which holds a HikariCP connection for the entire query duration. There is no enforcement of a maximum subscriber count — `subscriberCount` in `TopicMessageServiceImpl` is tracked only as a metric gauge (line 52–55) and never used to reject new subscriptions.

`TopicMessageFilter.limit` accepts `0` (meaning unlimited, per `hasLimit()` returning `false` when `limit == 0`), which is the default when a gRPC client omits the `limit` field. `ConsensusController.toFilter()` passes the client-supplied limit directly with no floor or cap on subscriber count.

**Why existing checks fail:**

- `subscriberCount` (AtomicLong in `TopicMessageServiceImpl`) is a Micrometer gauge only — no rejection logic exists.
- `TopicMessageFilter` validation (`@Min(0)` on `limit`) explicitly allows `limit=0` (unlimited).
- No authentication or authorization is enforced on the gRPC `subscribeTopic` endpoint.
- No per-IP or per-client rate limiting exists on subscription creation.
- `Schedulers.boundedElastic()` limits threads (default: `10 × CPU cores`, queue 100,000) but does not limit DB connection acquisition — threads block waiting for HikariCP connections, which itself has a finite pool (HikariCP default: 10 connections).

### Impact Explanation

With N concurrent subscribers (N > HikariCP pool size, default 10), every 500 ms polling cycle fires N simultaneous blocking DB queries. HikariCP queues connection requests beyond the pool limit; threads block until `connectionTimeout` (HikariCP default: 30 s). During this window, all other database operations — including legitimate user queries and other gRPC subscriptions — time out with `SQLTransientConnectionException`. The service becomes effectively unavailable for all users. On a high-volume topic returning 5,000 rows per poll, each query also imposes significant CPU and I/O load on the database, compounding the denial of service.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no economic cost. Any client with network access to the gRPC port can open hundreds of concurrent `subscribeTopic` streams using standard gRPC tooling (e.g., `grpcurl`, the Hedera SDK, or a simple loop). The attack is repeatable and persistent as long as the attacker maintains open streams. The only prerequisite is that the operator has configured `hiero.mirror.grpc.listener.type=POLL` — a non-default but documented and supported configuration.

### Recommendation

1. **Enforce a maximum concurrent subscriber limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., return `RESOURCE_EXHAUSTED` gRPC status).
2. **Use a reactive/non-blocking database driver** (R2DBC) for `findByFilter` so DB calls do not hold HikariCP connections while waiting — the original design doc (`docs/design/hcs.md`) references R2DBC but the implementation uses blocking JPA.
3. **Add per-client rate limiting** on subscription creation at the gRPC interceptor layer.
4. **Expose and enforce `maxSubscribers`** as a `ListenerProperties` field, analogous to `maxPageSize`.
5. **Consider `SHARED_POLL` or `REDIS` listener types** (the defaults) which share a single poll across all subscribers, eliminating the N-connections-per-poll-cycle problem entirely.

### Proof of Concept

**Preconditions:**
- Mirror node configured with `hiero.mirror.grpc.listener.type=POLL`
- A topic with a high message volume exists (e.g., topic `0.0.1234`)
- HikariCP pool size at default (10 connections)

**Steps:**

```bash
# Open 50 concurrent unlimited subscriptions to a high-volume topic
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1234}, "consensusStartTime": {"seconds": 0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Observe: within 1-2 poll cycles (500ms-1s), all 50 goroutines attempt
# simultaneous DB queries. HikariCP pool (10 connections) is exhausted.
# Legitimate queries begin timing out with connection pool exhaustion errors.
```

**Expected result:** `com.zaxxer.hikari.pool.HikariPool$PoolInitializationException` or `SQLTransientConnectionException: HikariPool-1 - Connection is not available, request timed out after 30000ms` in server logs; all other users receive gRPC `UNAVAILABLE` or `INTERNAL` errors.     private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
        final var filter = TopicMessageFilter.builder().limit(query.getLimit());

        if (query.hasTopicID()) {
            filter.topicId(EntityId.of(query.getTopicID()));
        }

        if (query.hasConsensusStartTime()) {
            long startTime = convertTimestamp(query.getConsensusStartTime());
            filter.startTime(startTime);
        }

        if (query.hasConsensusEndTime()) {
            long endTime = convertTimestamp(query.getConsensusEndTime());
            filter.endTime(endTime);
        }

        return filter.build();
    }
