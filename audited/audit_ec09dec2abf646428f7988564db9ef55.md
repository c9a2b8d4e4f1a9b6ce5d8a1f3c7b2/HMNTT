### Title
Unauthenticated HTTP Mirror Node Fetch Allows DNS/MITM Hijack to Starve Consensus Nodes of Transaction Load

### Summary
`buildNetworkFromMirrorNodes()` fetches the consensus node list over a plain, unauthenticated HTTP connection with no TLS enforcement, no response-integrity verification, and no minimum node-count threshold beyond zero. An attacker who can redirect DNS for the mirror REST hostname (or perform any network-layer MITM) can return a crafted node list containing only a subset of real nodes, causing `hiero.ClientForNetwork(netmap)` to route all pinger transactions exclusively to those nodes and starving ≥30% of consensus nodes of transaction load.

### Finding Description

**Code path:**

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
// ...
client = hiero.ClientForNetwork(netmap)
``` [1](#0-0) 

`pinger/mirror_node_client.go` line 46 — the HTTP client is created with no TLS configuration whatsoever:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
``` [2](#0-1) 

`pinger/mirror_node_client.go` lines 79–84 — a plain `GET` is issued to whatever URL `cfg.mirrorRest` resolves to, with no certificate pinning, no HMAC, no signature check:

```go

// ...

``` [3](#0-2) 

`pinger/mirror_node_client.go` lines 127–129 — the **only** guard on the returned node list is that it is non-empty; a single attacker-controlled entry passes:

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
``` [4](#0-3) 

`pinger/config.go` line 37 — the canonical example URL is plaintext HTTP, confirming this is the expected deployment pattern:

```
"mirror node REST base URL (required for other), e.g. http://mirror-rest:5551"
``` [5](#0-4) 

**Root cause:** The code assumes the mirror REST endpoint is trusted and reachable only over a secure channel. Neither assumption is enforced in code. The network map handed to `hiero.ClientForNetwork` is fully attacker-controlled if the HTTP response can be substituted.

### Impact Explanation

`hiero.ClientForNetwork(netmap)` uses `netmap` as the **complete and exclusive** set of consensus nodes. Every `TransferTransaction` executed by the pinger (via `submitWithRetry` → `cryptoTransfer.Execute(client)`) is routed only to nodes present in that map. [6](#0-5)  If the attacker returns a list containing, say, 2 out of 30 mainnet nodes, 28 nodes (93%) receive zero transaction load from this pinger. Even returning 20 of 30 nodes starves 33% — meeting the ≥30% threshold. The pinger is a continuous liveness/load tool; sustained exclusion of nodes from its traffic constitutes a persistent, targeted load-starvation attack on those consensus nodes.

### Likelihood Explanation

Preconditions are realistic for a network-level attacker:

- **DNS cache poisoning** (Kaminsky-style or via a compromised resolver) against the hostname in `cfg.mirrorRest` requires no privileges in the Hiero system.
- **ARP spoofing / BGP hijack** on the same network segment as the pinger pod intercepts the plaintext HTTP stream directly.
- **Rogue DHCP** in a Kubernetes cluster or cloud VPC can redirect DNS for internal service names like `mirror-rest`.

The attack is repeatable: `newClient` is called once at startup, so a single poisoned response persists for the entire lifetime of the pinger process. [7](#0-6)  No re-fetch or re-validation occurs after startup.

### Recommendation

1. **Enforce HTTPS**: Reject any `cfg.mirrorRest` URL that does not use the `https` scheme; optionally pin the CA or server certificate.
2. **Validate node-count floor**: After fetching, compare `len(network)` against a configurable minimum (e.g., ≥ the known network size × 0.7) and refuse to start if the threshold is not met.
3. **Cross-check against a static seed list**: For the `other` network, require a static list of known-good node account IDs in config and reject any fetched map that omits entries from that list.
4. **Periodic re-fetch with drift detection**: If the node list is refreshed at runtime, alert and abort if the new list shrinks by more than a configurable percentage relative to the previous fetch.

### Proof of Concept

1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. Poison DNS for `mirror-rest` (e.g., via a rogue DNS server or ARP spoof) to resolve to an attacker-controlled host.
3. Serve a valid JSON response from that host at `/api/v1/network/nodes` containing only 2 of the 30 real mainnet node entries (valid `node_account_id` and `service_endpoints`).
4. Start the pinger. `buildNetworkFromMirrorNodes` returns a 2-entry map; `len(network) == 2 > 0` passes the only guard.
5. `hiero.ClientForNetwork(netmap)` is initialized with only those 2 nodes.
6. All subsequent `TransferTransaction.Execute(client)` calls route exclusively to those 2 nodes; the remaining 28 nodes (93%) receive no transaction load from this pinger for its entire runtime.
