### Title
Type Confusion via Nested Object Injection in `requestQueryParser` Causing Downstream TypeError on `order`/`result` Parameters

### Summary
`qs.parse` with `depth: 1` allows bracket-notation query strings (e.g., `order[foo]=bar`) to produce a plain object as the value for the `order` or `result` keys. `canonicalizeValue()` routes these keys through `lowerCaseQueryValue`, which explicitly returns non-string values unchanged. The resulting object propagates into downstream code that unconditionally calls `.split(':')` on the value, throwing a `TypeError` on every such request.

### Finding Description

**Step 1 — Object injection via `qs.parse`**

`queryOptions` sets `depth: 1` and `strictDepth: true`. [1](#0-0) 

`depth: 1` permits one level of bracket nesting. `strictDepth: true` only throws when the depth is *exceeded*, not when it is exactly met. Therefore `qs.parse('order[foo]=bar', queryOptions)` produces `{ order: { foo: 'bar' } }` without throwing.

**Step 2 — `canonicalizeValue` routes the object through `lowerCaseQueryValue`** [2](#0-1) 

`lowerKey` is `'order'`, which is present in `queryCanonicalizationMap`, so `lowerCaseQueryValue` is called with `{ foo: 'bar' }`.

**Step 3 — `lowerCaseQueryValue` silently passes the object through** [3](#0-2) 

`typeof { foo: 'bar' } === 'string'` is `false`, so the object is returned unchanged. `caseInsensitiveQueryString.order` is now a plain object.

**Step 4 — Downstream crash in `buildComparatorFilter`** [4](#0-3) 

`filter.split(':')` is called on the object. Since objects have no `.split` method, this throws `TypeError: filter.split is not a function`. The same crash occurs in `paramValidityChecks` at line 249 (`opAndVal.split(':')`). [5](#0-4) 

**Step 5 — Error handling**

The `extendExpress` wrapper catches the `TypeError` and forwards it to `next(err)`. [6](#0-5) 

`handleError` does not recognise `TypeError` as any known REST error type, so it returns HTTP 500 with `defaultStatusCode = httpStatusCodes.INTERNAL_ERROR`. [7](#0-6) 

Critically, `handleUncaughtException` re-throws any non-`RestError` exception, which would crash the Node.js process if the error ever escapes the `extendExpress` wrapper (e.g., in middleware not wrapped by `extendExpress`). [8](#0-7) 

### Impact Explanation

Every API endpoint that calls `buildAndValidateFilters` or `validateReq` with `req.query` is affected. An unauthenticated attacker can force a guaranteed HTTP 500 on any such endpoint by appending `?order[foo]=bar` (or `?ORDER[foo]=bar`, `?result[x]=y`, etc.). The OpenAPI validator that would otherwise reject this is gated behind `config.openapi.validation.enabled`, which is not enforced by default. [9](#0-8) 

### Likelihood Explanation

No authentication or special knowledge is required. The payload is a single, trivially crafted query string parameter. It is repeatable at will, affects every paginated endpoint, and is invisible to standard string-based input filters because the object never reaches them — it crashes before validation completes.

### Recommendation

1. **Reject non-string values before canonicalization.** In `canonicalizeValue`, add a guard: if `value` is a plain object (and not an array), treat it as an invalid parameter and throw `InvalidArgumentError` immediately.
2. **Set `depth: 0` in `queryOptions`** to prevent `qs` from ever producing nested objects from query strings. The application only expects flat string values.
3. **Add a type guard in `buildComparatorFilter`**: assert `typeof filter === 'string'` before calling `.split()`.
4. **Enable the OpenAPI validator** by default, or add a middleware-level check that rejects any query parameter whose parsed value is not a string or array of strings.

### Proof of Concept

```
GET /api/v1/transactions?order[foo]=bar HTTP/1.1
Host: <mirror-node-host>
```

**Trace:**
1. `qs.parse('order[foo]=bar', {depth:1, strictDepth:true})` → `{ order: { foo: 'bar' } }`
2. `canonicalizeValue('order', { foo: 'bar' })` → calls `lowerCaseQueryValue({ foo: 'bar' })` → returns `{ foo: 'bar' }` (not a string, passes through)
3. `req.query.order === { foo: 'bar' }` (a plain object)
4. `buildComparatorFilter('order', { foo: 'bar' })` → `{ foo: 'bar' }.split(':')` → `TypeError: filter.split is not a function`
5. Response: HTTP 500

Repeat with `ORDER[x]=y`, `Result[x]=y`, etc. — all produce the same crash. No credentials required.
