### Title
Unsanitized ILIKE Wildcard Injection via `name` Filter Enables Unauthenticated DB CPU Exhaustion DoS

### Summary
The `extractSqlFromTokenRequest()` function in `rest/tokens.js` wraps the user-supplied `name` query parameter in `%…%` to form an ILIKE pattern, but the validation gate (`isByteRange(val, 3, 100)`) only enforces byte-length bounds and never strips or escapes ILIKE metacharacters (`%`, `_`). An unauthenticated attacker can inject additional `%` and `_` wildcards into the value, producing a maximally complex backtracking pattern up to 102 bytes long, and flood the endpoint to exhaust PostgreSQL CPU.

### Finding Description
**Exact code path:**

`validateTokenQueryFilter` (tokens.js line 333–335) accepts any `name` value that satisfies `op === 'eq'` and `isByteRange(val, 3, 100)`:
```js
case filterKeys.NAME:
  ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
  break;
```
`isByteRange` (utils.js line 131–134) only measures byte length:
```js
const isByteRange = (str, minSize, maxSize) => {
  const length = Buffer.from(str).length;
  return length >= minSize && length <= maxSize;
};
```
No character-content check exists. The validated value then reaches `extractSqlFromTokenRequest` line 177:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
```
Because parameterized binding is used, there is no SQL injection. However, the ILIKE *pattern* itself is `'%' + <attacker-controlled 100-byte string> + '%'`. The attacker controls the internal wildcard structure of the pattern.

**Root cause / failed assumption:** The code assumes `isByteRange` is a sufficient guard. It bounds length but does not prevent ILIKE metacharacters (`%` = zero-or-more, `_` = exactly-one) from appearing inside the value, so the attacker can craft patterns with arbitrary internal complexity.

**Exploit flow:**
1. Attacker sends `GET /api/v1/tokens?name=%25a%25b%25c%25d%25e%25f%25g%25h%25i%25j%25k%25l%25m%25n%25o%25p%25q%25r%25s%25t%25u%25v%25w%25x%25y%25z%25` (URL-encoded `%a%b%c%…%z%`, 51 bytes — well within the 100-byte cap).
2. Validation passes: byte length = 51, operator = `eq`.
3. PostgreSQL receives the pattern `%%a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z%%`.
4. Without a `pg_trgm` GIN index on `t.name`, PostgreSQL performs a full sequential scan of the `token` table, evaluating this 28-segment alternating-wildcard pattern against every row. Worst-case NFA backtracking is O(N × M) per row where M grows with segment count.
5. Attacker repeats at high concurrency; each in-flight query holds a backend worker and CPU.

**Why existing checks fail:** `isByteRange` is the only content-level check. There is no allowlist regex, no `%`/`_` escape step, no rate-limit middleware visible in this path, and no query timeout enforced at the application layer before the query is dispatched via `pool.queryQuietly`.

### Impact Explanation
Sustained concurrent requests with crafted patterns exhaust PostgreSQL CPU and connection pool workers, rendering the mirror-node REST API unavailable. Because the `token` table grows with network activity, the cost per query scales over time, making the attack progressively cheaper relative to its impact. Availability of the REST API is fully compromised for the duration of the attack.

### Likelihood Explanation
No authentication or API key is required to call `GET /api/v1/tokens`. The endpoint is publicly documented. Crafting the payload requires only knowledge of URL encoding and ILIKE syntax — no privileged access, no special tooling. The attack is trivially repeatable with a simple HTTP flood script and is stable across deployments because it exploits a structural gap in input validation, not a transient race condition.

### Recommendation
Escape ILIKE metacharacters in the `name` value before interpolating it into the pattern. In PostgreSQL, `%` and `_` must be preceded by an escape character (e.g., `\`):

```js
const escapedValue = filter.value.replace(/\\/g, '\\\\').replace(/%/g, '\\%').replace(/_/g, '\\_');
conditions.push(`t.name ILIKE $${params.push('%' + escapedValue + '%')} ESCAPE '\\'`);
```

Additionally:
- Add a character-content allowlist in `validateTokenQueryFilter` for `filterKeys.NAME` (e.g., reject values containing raw `%` or `_`).
- Enforce a `statement_timeout` on the PostgreSQL connection pool used by this endpoint.
- Apply rate-limiting middleware on the `/api/v1/tokens` route.

### Proof of Concept
```bash
# Single expensive request
curl -s "https://<mirror-node>/api/v1/tokens?name=%25a%25b%25c%25d%25e%25f%25g%25h%25i%25j%25k%25l%25m%25n%25o%25p%25q%25r%25s%25t%25u%25v%25w%25x%25y%25z%25"

# Flood (requires no credentials)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/tokens?name=%25a%25b%25c%25d%25e%25f%25g%25h%25i%25j%25k%25l%25m%25n%25o%25p%25q%25r%25s%25t%25u%25v%25w%25x%25y%25z%25" &
done
wait
```

The resulting PostgreSQL query for each request is:
```sql
SELECT ... FROM token t
JOIN entity e ON e.id = t.token_id
WHERE t.name ILIKE '%%a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z%%'
ORDER BY t.token_id ASC
LIMIT 25;
```
Monitor `pg_stat_activity` and CPU during the flood to confirm sequential scans and CPU saturation.
