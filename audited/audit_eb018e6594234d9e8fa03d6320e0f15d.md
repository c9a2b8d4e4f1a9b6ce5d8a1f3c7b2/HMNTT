### Title
Missing File ID Whitelist Validation in REST Java `getAddressBookFileId()` Allows Arbitrary Address Book Probing

### Summary
The REST Java `NetworkServiceImpl.getAddressBookFileId()` accepts any user-supplied `file.id` value without validating it against the permitted set `{101, 102}`. This is in direct contrast to the gRPC `NetworkServiceImpl`, which explicitly enforces a whitelist. An unauthenticated attacker can probe the `address_book` table with arbitrary file IDs, and any address book stored under a non-standard file ID (e.g., from migrations, operator misconfiguration, or future features) would be silently returned with full gossip endpoint data.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 153–157) only checks that the `file.id` operator is `EQ`; it performs no whitelist check on the value itself: [1](#0-0) 

The request is passed to `NetworkServiceImpl.getNetworkNodes()`, which calls `getAddressBookFileId()`: [2](#0-1) 

`getAddressBookFileId()` blindly returns `request.getFileId().value()` — any long integer the caller supplies — with no validation against `{addressBookFile101, addressBookFile102}`. That value is passed directly into the native SQL query: [3](#0-2) 

**Root cause — failed assumption:** The code assumes only file IDs 101 and 102 will ever be supplied, but this is never enforced.

**Contrast with the gRPC service**, which correctly maintains a whitelist and rejects any other value: [4](#0-3) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/network/nodes?file.id=0.0.<arbitrary>` — no authentication required.
2. `getAddressBookFileId()` returns the raw value.
3. The SQL CTE `latest_address_book` runs `WHERE file_id = :fileId` against the `address_book` table for that arbitrary ID.
4. If any matching rows exist, full node data including all `address_book_service_endpoint` gossip IPs/ports is returned in the response.
5. The API returns HTTP 200 with an empty list for non-existent IDs, giving the attacker a silent oracle to enumerate which file IDs have address books.

**Existing checks are insufficient:** The controller's only guard is `fileId.operator() != RangeOperator.EQ` (rejects range operators), which does nothing to restrict the actual numeric value. The `@Min(1)` constraint on `limit` and the `EntityIdRangeParameter` parsing only validate format, not semantic validity of the file ID. [5](#0-4) 

### Impact Explanation
Any address book row stored under a non-standard file ID (e.g., inserted by a migration script, a future feature, or an operator error) would be fully exposed — including all `service_endpoint` gossip IPs and ports — to any unauthenticated caller. Even without such rows currently existing, the silent HTTP 200 oracle enables systematic enumeration of the entire `address_book` table's file ID space at zero cost. The gRPC protocol spec itself documents that only `0.0.101` or `0.0.102` are valid: [6](#0-5) 

### Likelihood Explanation
Exploitation requires no credentials, no special tooling, and no rate-limiting bypass — a single HTTP GET with a numeric `file.id` parameter is sufficient. The attack is trivially scriptable to scan all plausible file IDs (e.g., 1–1000) in seconds. The inconsistency with the gRPC service's explicit whitelist shows this protection was intentionally designed but simply omitted from the REST path.

### Recommendation
Add the same whitelist guard used by the gRPC service into `NetworkServiceImpl.getAddressBookFileId()` (or at the controller layer):

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    long valid101 = systemEntity.addressBookFile101().getId();
    long valid102 = systemEntity.addressBookFile102().getId();
    if (id != valid101 && id != valid102) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

This mirrors the gRPC service's `validFileIds` set check exactly. [7](#0-6) 

### Proof of Concept

```bash
# 1. Probe an arbitrary file ID — returns HTTP 200 (not 400), confirming no whitelist
curl -s "https://<mirror-node>/api/v1/network/nodes?file.id=0.0.999"
# Response: {"nodes":[],"links":{"next":null}}  ← silent oracle, no error

# 2. Scan for non-standard address books
for i in $(seq 1 1000); do
  result=$(curl -s "https://<mirror-node>/api/v1/network/nodes?file.id=0.0.$i")
  count=$(echo "$result" | python3 -c "import sys,json; print(len(json.load(sys.stdin)['nodes']))")
  if [ "$count" -gt "0" ]; then
    echo "Found address book at file.id=0.0.$i with $count nodes:"
    echo "$result"
  fi
done

# 3. Confirm gRPC rejects the same input (shows the REST gap)
# gRPC call with file_id=999 → INVALID_ARGUMENT: "Not a valid address book file"
# REST call with file.id=0.0.999 → HTTP 200 with empty nodes (no rejection)
```
