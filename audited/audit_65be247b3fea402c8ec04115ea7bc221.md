### Title
Unbounded `node.id` Parameter List Enables Unauthenticated DoS via Repeated Large Array Allocations in `NetworkServiceImpl.getNetworkNodes()`

### Summary
`NetworkNodeRequest.nodeIds` carries no `@Size` constraint, unlike every comparable parameter in the same codebase (`RegisteredNodesRequest.registeredNodeIds`, `AllowancesController`, `HooksController`). An unauthenticated caller can supply an arbitrarily large number of `node.id=eq:X` query parameters; `NetworkServiceImpl.getNetworkNodes()` materialises all of them into a `HashSet<Long>` and then a `Long[]` array on every request, with no rate-limiting guard in the `rest-java` module. Under concurrent load this produces sustained heap pressure, frequent GC pauses, and degraded mirror-node API responsiveness.

### Finding Description

**Missing `@Size` constraint — root cause**

`rest-java/src/main/java/org/hiero/mirror/restjava/dto/NetworkNodeRequest.java`, line 36-38:
```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();   // ← no @Size
```
Every analogous field in the same module carries an explicit upper bound:
- `RegisteredNodesRequest.registeredNodeIds` → `@Size(max = 2)`
- `AllowancesController` `accountIds` / `tokenIds` → `@Size(max = 2)`
- `HooksController` `hookId` → `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)`

**Allocation path — `NetworkServiceImpl.getNetworkNodes()` lines 106-127**

```java
final Set<Long> nodeIds = new HashSet<>();          // line 106 – unbounded
...
for (final var nodeIdParam : nodeIdParams) {        // line 110 – iterates all N params
    if (nodeIdParam.operator() == RangeOperator.EQ) {
        nodeIds.add(nodeIdParam.value());            // line 112 – N boxed Longs
    }
    ...
}
...
nodeIdArray = nodeIds.stream()
    .filter(range::contains)
    .toArray(Long[]::new);                          // line 127 – new Long[N] per request
```

With Tomcat's default `maxParameterCount` of 10 000, each request can force allocation of ≈10 000 boxed `Long` objects in the `HashSet` (≈160 KB) plus a `Long[]` of the same size (≈80 KB), totalling ≈240 KB of short-lived heap per request.

**No rate limiting in `rest-java`**

The bucket4j throttle (`ThrottleManagerImpl`, `ThrottleConfiguration`) lives exclusively in the `web3` module. The `rest-java` module exposes no equivalent guard on `GET /api/v1/network/nodes`.

**Exploit flow**

1. Attacker sends concurrent `GET /api/v1/network/nodes?file.id=eq:102&node.id=eq:1&node.id=eq:2&…&node.id=eq:9999` (≈150 KB query string, within Tomcat defaults).
2. Spring parameter binding populates `nodeIds` with 9 999 `NumberRangeParameter` objects — no validation rejects this.
3. `getNetworkNodes()` allocates `HashSet` + `Long[]` of size ≈10 000 on every thread.
4. With, say, 50 concurrent attackers, ≈12 MB of short-lived objects are created per GC cycle, driving minor-GC frequency up sharply and eventually triggering stop-the-world major GCs.
5. Legitimate requests time out or receive 503 responses.

### Impact Explanation
The affected service is the Hedera mirror-node REST Java API (`rest-java`). Sustained GC pressure degrades or halts the mirror-node's ability to serve queries (balances, transactions, network nodes, etc.). This does **not** affect consensus-node transaction confirmation directly; the mirror node is a read-only query layer. The realistic impact is mirror-node API unavailability, not a "total network shutdown."

### Likelihood Explanation
The endpoint is public, requires no authentication, and accepts GET requests. The attack requires only an HTTP client capable of sending long query strings — trivially scriptable. Tomcat's default `maxParameterCount = 10 000` provides a natural ceiling but is large enough to cause meaningful heap pressure. The attack is repeatable at will with no per-IP throttle in place.

### Recommendation
1. Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds` (a value of 2–10 is consistent with the rest of the codebase and sufficient for legitimate use):
   ```java
   @RestJavaQueryParam(name = NODE_ID, required = false)
   @Builder.Default
   @Size(max = 10)
   private List<NumberRangeParameter> nodeIds = List.of();
   ```
2. Add a rate-limiting filter to the `rest-java` module analogous to the bucket4j guard already present in `web3`.
3. Lower Tomcat's `server.tomcat.max-parameter-count` to a value appropriate for the API's legitimate use (e.g., 50).

### Proof of Concept
```bash
# Generate 9999 node.id=eq:X parameters
PARAMS=$(seq 1 9999 | awk '{printf "node.id=eq:%d&", $1}')
# Fire 50 concurrent requests
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/network/nodes?file.id=eq:102&${PARAMS}" &
done
wait
# Monitor GC: jstat -gcutil <pid> 1s
# Observe: increasing GC frequency, rising response latency, eventual 503s
```
