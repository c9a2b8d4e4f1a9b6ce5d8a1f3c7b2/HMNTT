### Title
Unauthenticated Pagination Exhaustion via `limit=1` on `/accounts/:id/nfts` Degrades DB Connection Pool

### Summary
The `getNftsByAccountId` handler in `rest/controllers/accountController.js` accepts `limit=1` from any unauthenticated caller and returns a `next` pagination link per response. Because the REST API has no per-IP or per-client rate limiting, an attacker can drive an unbounded number of sequential (or concurrent) DB queries against the default 10-connection pool, starving legitimate users of database access.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js` line 15 registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/controllers/accountController.js` lines 90–103 — `getNftsByAccountId` issues at least two DB round-trips per request (one for `EntityService.getEncodedId`, one for `NftService.getNfts`) and unconditionally emits a `next` link when results equal the requested limit: [2](#0-1) 

`rest/service/nftService.js` lines 134–138 — `getNfts` builds and executes a potentially three-branch `UNION ALL` query, each branch carrying its own `LIMIT $2`: [3](#0-2) 

**Limit enforcement** in `rest/utils.js` caps the value at `responseLimit.max` (default 100) but enforces no minimum, so `limit=1` is fully accepted: [4](#0-3) 

**No rate limiting exists in the REST middleware stack.** `rest/server.js` registers only `requestLogger`, `authHandler`, `metricsHandler`, and cache handlers — no IP throttle, no request-rate limiter: [5](#0-4) 

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` Java service and is entirely absent from the Node.js REST API: [6](#0-5) 

**DB pool default is 10 connections** with a 20-second statement timeout: [7](#0-6) 

**Root cause:** The failed assumption is that callers will use reasonably large page sizes. With `limit=1`, traversing an account holding N NFTs requires N HTTP requests, each consuming a DB connection for the duration of the query. No control prevents a single unauthenticated client from issuing these requests as fast as the server accepts them.

### Impact Explanation
With a default pool of 10 connections and a 20-second statement timeout, an attacker issuing ~10 concurrent `limit=1` requests can hold all available connections simultaneously. Legitimate API users receive connection-pool-wait errors or timeouts. Because the REST API is public and unauthenticated, this degrades service for all users of the mirror node without any economic cost or account privilege required. Severity is Medium (griefing / availability degradation, no fund loss).

### Likelihood Explanation
Any external party can discover the pagination pattern from the public OpenAPI spec (`/api/v1/accounts/{id}/nfts`, `limitQueryParam`). A simple script following `links.next` with `limit=1` in a tight loop, run from a single IP, is sufficient. No special knowledge, credentials, or on-chain assets are required beyond knowing a valid account ID that holds NFTs (which is public chain data).

### Recommendation
1. **Enforce a minimum `limit` value** (e.g., ≥ 10) in `getLimitParamValue` or in `buildAndValidateFilters` for this endpoint, reducing the maximum number of pages an attacker can force.
2. **Add per-IP rate limiting middleware** to the REST Express app (e.g., `express-rate-limit`) applied globally or specifically to the `/accounts/:id/nfts` route before the route handler.
3. **Increase the DB connection pool** or add a request-queue depth limit so pool exhaustion does not translate directly to user-visible errors.
4. Consider **response caching** (already supported via Redis, `config.cache.response.enabled`) for this read-only endpoint to absorb repeated identical paginated requests.

### Proof of Concept
```bash
# 1. Identify an account with many NFTs (public chain data)
ACCOUNT="0.0.12345"
BASE="https://<mirror-node>/api/v1"

# 2. Fetch first page with limit=1
NEXT="${BASE}/accounts/${ACCOUNT}/nfts?limit=1"

# 3. Follow every pagination link in a tight loop (no auth needed)
while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  RESP=$(curl -s "$NEXT")
  NEXT=$(echo "$RESP" | jq -r '.links.next // "null"')
  [ "$NEXT" != "null" ] && NEXT="${BASE}${NEXT}"
done

# Run 10 parallel instances of the above loop to exhaust the 10-connection pool:
for i in $(seq 1 10); do
  <above loop> &
done
wait
# Result: legitimate requests to any REST endpoint begin timing out or
# receiving connection-pool errors for the duration of the attack.
```

      nfts,
      links: {

      },
    };
  };

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();

    }
