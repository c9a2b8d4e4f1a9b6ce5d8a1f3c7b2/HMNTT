### Title
Unauthenticated Per-Request DB Query via Opaque EVM Address in `getEncodedId()`

### Summary
Any unprivileged external user can force an unconditional database query on every HTTP request by supplying an opaque EVM address (non-zero 12-byte prefix) to any REST endpoint that resolves entity IDs. The `parseCached` LRU in `entityId.js` only caches the parsed `EntityId` object, not the DB lookup result, so `getEntityIdFromEvmAddress()` executes a live SQL query on every call regardless of how many times the same address is submitted. No rate limiting exists anywhere in the REST layer.

### Finding Description

**Exact code path:**

`getEncodedId()` — `rest/service/entityService.js` lines 118–137 — is the entry point called by every major REST controller (accounts, balances, tokens, transactions, schedules, contracts, etc.).

```
getEncodedId(entityIdString)
  → EntityId.parseString(entityIdString)          // entityId.js parseCached, lines 314-333
      → parseFromString(id, error)                // entityId.js lines 239-260
          → if (prefix !== longFormEvmAddressPrefix || num > maxNum)
                return [shard, realm, null, numOrEvmAddress]  // line 253 — opaque path
  → entityId.evmAddress === null ?                // entityService.js line 122
        entityId.getEncodedId()                   // local, no DB
      : await this.getEntityIdFromEvmAddress(entityId, requireResult)  // line 124 — DB query
```

**Root cause — failed assumption:**

`parseCached` (lines 314–333 of `entityId.js`) caches the `EntityId` *object* keyed on the raw input string. For an opaque EVM address the cached object has `evmAddress = "<hex>"` and `num = null`. When `getEncodedId()` retrieves this cached object and evaluates `entityId.evmAddress === null` (line 122), the condition is still `false`, so `getEntityIdFromEvmAddress()` is invoked unconditionally — the cache provides zero protection against repeated DB hits.

`getEntityIdFromEvmAddress()` (lines 90–104) contains no secondary cache:

```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
                                  [Buffer.from(entityId.evmAddress, 'hex')]);
  ...
}
```

The SQL issued is:
```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```

Every invocation is a live round-trip to PostgreSQL.

**Why the shard/realm guard is insufficient:**

`parseFromString` (line 246) rejects non-default shard/realm values, but the attacker simply uses the default network shard/realm (typically `0.0.<evmAddress>`), which passes the guard and proceeds to the opaque branch.

**What makes an address "opaque":**

Any 40-hex-char address whose first 24 hex chars (12 bytes) are not all zeros, e.g. `deadbeefdeadbeefdeadbeefdeadbeefdeadbeef`. This is the normal form for CREATE2 contract addresses and any externally-owned account address — a completely ordinary and expected input.

### Impact Explanation
Every REST endpoint that accepts an account/entity ID parameter calls `getEncodedId()`. An attacker flooding these endpoints with opaque EVM addresses (even a single repeated address) generates one DB query per HTTP request with no server-side throttle. At sufficient request rate this exhausts the PostgreSQL connection pool, starves legitimate queries, and causes service-wide unavailability. Because the DB query touches the `entity` table (potentially large), even moderate concurrency produces measurable latency degradation before full exhaustion.

### Likelihood Explanation
No authentication, API key, or session is required. The affected endpoints are public-facing by design. The attack requires only an HTTP client and knowledge of any valid opaque EVM address format (trivially constructed). It is fully repeatable and automatable with standard tools (`ab`, `wrk`, `curl` in a loop). The attacker does not need any on-chain funds or privileged access.

### Recommendation
1. **Cache DB lookup results** — after `getEntityIdFromEvmAddress()` resolves (or fails to resolve) an opaque EVM address, store the result (including negative/not-found) in the existing `quickLru` cache keyed on the hex address. This eliminates repeated DB hits for the same address.
2. **Rate-limit at the HTTP layer** — add per-IP (or global) request-rate limiting middleware before entity-ID resolution occurs.
3. **Short-circuit on not-found** — for `requireResult = false` callers, a negative cache entry (TTL-bounded) prevents repeated DB queries for addresses that do not exist.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly reachable. Default shard/realm = 0/0.

**Steps:**

```bash
# Generate a valid opaque EVM address (non-zero prefix — any CREATE2-style address)
ADDR="deadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Flood any entity-ID-accepting endpoint; each request triggers one DB query
# Example: accounts endpoint
for i in $(seq 1 10000); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.${ADDR}" &
done
wait
```

Each request causes `getEncodedId("0.0.deadbeef...")` → `parseFromString` → opaque branch → `getEntityIdFromEvmAddress()` → `SELECT id FROM entity WHERE evm_address = $1`. With 10 000 concurrent requests and no rate limiting, the PostgreSQL connection pool is exhausted and the service becomes unresponsive to all clients.

const parseCached = (id, allowEvmAddress, evmAddressType, error) => {
  const key = `${id}_${allowEvmAddress}_${evmAddressType}`;
  const value = cache.get(key);
  if (value) {
    return value;
  }

  if (!isValidEntityId(id, allowEvmAddress, evmAddressType)) {

    throw error();
  }

  cache.set(key, entityId);
