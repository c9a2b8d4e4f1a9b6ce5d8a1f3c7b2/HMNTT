### Title
Redis Subscription Leak via Missing Container Cleanup in `unsubscribe()` Enabling Subscription Exhaustion

### Summary
`RedisTopicListener.unsubscribe()` removes the topic key from the `topicMessages` map but never calls any cleanup method on the `ReactiveRedisMessageListenerContainer`, leaving the underlying Redis `SUBSCRIBE` channel open on the server side. Each subscribe/unsubscribe cycle driven by any gRPC client reconnection creates a new `r.receive()` call — and thus a new server-side Redis subscription — without retiring the previous one. During a network partition (where `UNSUBSCRIBE` cannot be delivered), the retry loop compounds this by issuing repeated `r.receive()` calls, each registering a new subscription the Redis server cannot release.

### Finding Description

**Exact code path:**

`getSharedListener()` uses `computeIfAbsent` to create a shared Flux per topic: [1](#0-0) 

`subscribe()` calls `r.receive(...)` on the container — this issues a Redis `SUBSCRIBE` command — and wraps the result with `.share()`: [2](#0-1) 

When the last downstream subscriber cancels (gRPC client disconnects), `doOnCancel` fires `unsubscribe()`: [3](#0-2) 

`unsubscribe()` only removes the map entry — no `container.remove(topic)`, no `container.destroy()`, no disposal of the `ReactiveSubscription`: [4](#0-3) 

**Root cause:** The `ReactiveRedisMessageListenerContainer` tracks subscriptions internally. Each call to `r.receive(topic, ...)` registers a new server-side `SUBSCRIBE`. Proper cleanup requires calling `container.remove(subscription)` (or equivalent). Because `unsubscribe()` skips this, the container retains the old subscription while `computeIfAbsent` is free to create a new one on the next client connection.

**Retry compounding during network partition:** The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 78 retries indefinitely on error. Each retry re-executes the `flatMapMany(r -> r.receive(...))` lambda, issuing a new `SUBSCRIBE` to Redis. During a network partition, the prior subscription's `UNSUBSCRIBE` cannot be delivered, so the Redis server accumulates open subscriptions with every retry cycle. [5](#0-4) 

**Why existing checks fail:** There is no rate limiting, no per-client subscription cap, and no authentication guard in the gRPC listener path. `TopicMessageServiceImpl.subscribeTopic()` only tracks a `subscriberCount` gauge for metrics — it enforces no ceiling: [6](#0-5) 

### Impact Explanation
Redis enforces a server-wide limit on the number of active pub/sub subscriptions (configurable, default effectively unbounded but memory/fd constrained). Accumulating leaked subscriptions consumes Redis file descriptors, memory, and internal subscription-tracking structures. Once the limit is reached, new `SUBSCRIBE` commands fail, denying service to all legitimate subscribers. Because the container is shared (singleton, `.cache()`), the impact is process-wide — all topics are affected, not just the attacker's topic.

### Likelihood Explanation
Any unauthenticated gRPC client can subscribe to a topic and disconnect. No privilege is required. The exploit is fully scriptable: open a gRPC stream to `subscribeTopic`, immediately cancel it, repeat. During a network partition (the stated precondition), the retry loop at `Long.MAX_VALUE` retries amplifies the leak automatically without any attacker action after the initial subscription. The attacker only needs to establish the first subscription before the partition occurs; the retry loop does the rest.

### Recommendation
1. **Track and dispose the container subscription:** Capture the `Disposable` or `ReactiveSubscription` returned by `r.receive(...)` and call `container.remove(subscription)` (or `dispose()`) inside `unsubscribe()`.
2. **Bound the retry count:** Replace `Long.MAX_VALUE` retries with a finite limit and emit a terminal error so the shared Flux is torn down cleanly rather than retrying forever.
3. **Add per-IP or per-client rate limiting** at the gRPC interceptor layer to prevent rapid reconnect abuse.
4. **Monitor subscription count:** Expose a metric for the size of `topicMessages` and alert when it grows unexpectedly.

### Proof of Concept

**Preconditions:** Mirror node running with `REDIS` listener type; Redis server reachable; no authentication on gRPC endpoint.

**Steps:**

```
# Repeat in a tight loop (bash pseudocode):
for i in $(seq 1 10000); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <mirror-node>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  PID=$!
  sleep 0.05
  kill $PID   # triggers gRPC cancel → doOnCancel → unsubscribe() → map cleared
              # but Redis SUBSCRIBE still open on server
done

# Verify on Redis server:
redis-cli CLIENT LIST | grep -c "cmd=subscribe"
# Count grows monotonically; legitimate subscribers begin failing
# when Redis fd/memory limits are reached.
```

**During network partition variant:** Establish one subscription, then partition the network between mirror node and Redis. The `retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))` loop fires repeatedly. Each retry re-issues `r.receive(...)`. When the partition heals, Redis sees all accumulated `SUBSCRIBE` commands at once, exhausting limits immediately.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
