### Title
Gas Estimation Binary Search Amplification: Single Throttle Check Permits Up to 21 Contract Executions

### Summary
The `ThrottleManager.throttle()` is invoked exactly once per HTTP request at the controller layer, consuming one rate-limit token and one gas-bucket allocation. However, when `estimate=true`, `ContractExecutionService.estimateGas()` performs one initial contract call followed by up to `maxGasEstimateRetriesCount` (default: 20) additional executions inside `BinaryGasEstimator.search()`. An unprivileged attacker targeting a storage-intensive contract can cause up to 21 full contract executions — each triggering expensive database I/O — while consuming only a single throttle token, creating a ~21x resource amplification factor per request.

### Finding Description

**Throttle is applied once, at the controller:** [1](#0-0) 

`throttleManager.throttle(request)` is called once before any processing. It consumes 1 token from `rateLimitBucket` and `scaleGas(request.getGas())` tokens from `gasLimitBucket`. [2](#0-1) 

**Estimate path triggers up to 21 executions:**

`ContractExecutionService.estimateGas()` first calls `callContract()` (execution #1), then passes the result to `BinaryGasEstimator.search()`: [3](#0-2) 

Inside `BinaryGasEstimator.search()`, the loop runs up to `maxGasEstimateRetriesCount` (default **20**) additional iterations, each calling `doProcessCall()`: [4](#0-3) [5](#0-4) 

**Why existing checks are insufficient:**

- `rateLimitBucket.tryConsume(1)`: limits HTTP requests per second, but each request causes up to 21 contract executions. The bucket has no awareness of per-request execution multiplicity.
- `gasLimitBucket.tryConsume(scaleGas(request.getGas()))`: consumes gas tokens based on the declared gas limit once. The binary search executes the contract up to 21 times, each with gas values between `gasUsedByInitialCall` and `request.getGas()`. Total actual gas work is up to `21 × request.getGas()`, but only `1 × request.getGas()` is deducted.
- `restore(gas)`: returns unused gas tokens after execution, but this only accounts for the final call's unused gas, not the amplification from 20 additional binary search iterations.
- `maxGasEstimateRetriesCount = 20`: caps iterations but does not prevent the amplification — it defines the upper bound of the attack multiplier.

### Impact Explanation

For a storage-intensive contract (e.g., one that reads/writes many SLOAD/SSTORE slots), each of the 21 executions triggers a full round of database queries against the mirror node's historical state store. A single estimate request at the rate limit causes ~21× the expected DB I/O and CPU load compared to a regular `eth_call`. An attacker sending requests at the configured `requestsPerSecond` rate can sustain this amplification continuously. With default settings (15M gas limit, 20 max retries), the effective gas throughput hitting the database is up to 21× what the gas throttle bucket accounts for, making a >30% resource increase achievable with a small number of concurrent connections.

### Likelihood Explanation

No authentication or privileges are required. The `/api/v1/contracts/call` endpoint is publicly accessible. The attacker only needs to know (or deploy) a storage-intensive contract address and set `estimate: true` in the request body. The attack is trivially repeatable and scriptable. The default `maxGasEstimateRetriesCount` of 20 is a fixed, known amplification factor.

### Recommendation

1. **Account for estimation multiplier in the gas throttle**: When `request.isEstimate()` is true, multiply the gas tokens consumed by `maxGasEstimateRetriesCount + 1` before calling `gasLimitBucket.tryConsume()`, or introduce a separate estimation-specific gas bucket with a lower effective limit.
2. **Per-iteration throttle check**: Call `throttleManager` inside `BinaryGasEstimator.search()` on each iteration, or expose a lightweight token-per-iteration check.
3. **Reduce default `maxGasEstimateRetriesCount`**: Lower the default from 20 to a value that limits amplification while preserving estimation accuracy.
4. **Separate rate limit for estimate requests**: Add a dedicated `estimateRequestsPerSecond` bucket checked when `estimate=true`, set lower than the general `requestsPerSecond` limit.

### Proof of Concept

1. Deploy or identify a contract that performs 100+ SLOAD/SSTORE operations per call on the target network.
2. Send the following request repeatedly at the configured `requestsPerSecond` rate:
```json
POST /api/v1/contracts/call
{
  "to": "<storage-intensive-contract-address>",
  "data": "<calldata-for-storage-heavy-function>",
  "gas": 15000000,
  "estimate": true
}
```
3. Each request passes the single throttle check (1 token consumed) but triggers 1 initial call + up to 20 binary search iterations = **21 full contract executions**, each issuing database queries for historical state.
4. Monitor node DB I/O and CPU: with even 1–2 concurrent clients at the rate limit, resource consumption will exceed 21× the baseline of equivalent `eth_call` requests, surpassing the 30% increase threshold.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
