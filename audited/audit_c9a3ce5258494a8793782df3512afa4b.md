### Title
DNS Poisoning of Mirror Node REST Endpoint Allows Full Transaction Routing Hijack via Fabricated `nodesEnvelope`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` issues an unauthenticated plain-HTTP GET to a caller-configured domain (`cfg.mirrorRest`) and unconditionally trusts the returned `nodesEnvelope` to build the entire gRPC node map. An attacker who can poison DNS for that domain redirects the request to an attacker-controlled server, injects arbitrary `ServiceEndpoints`, and causes every subsequent `hiero.Client` transfer transaction to be submitted exclusively to attacker-owned gRPC nodes. No TLS, no certificate pinning, no response validation, and no comparison against a known-good node list exist to prevent this.

### Finding Description

**Exact code path:**

`pinger/sdk_client.go` line 18 calls `buildNetworkFromMirrorNodes()` only when `network=other`. That function (`mirror_node_client.go` lines 36–72) constructs a URL from `cfg.mirrorRest` (operator-supplied, e.g. `http://mirror-rest:5551`) and calls `fetchMirrorNodeNetwork()`.

`fetchMirrorNodeNetwork()` (`mirror_node_client.go` lines 74–131):
- **Line 46**: `httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}` — a bare HTTP client with no TLS configuration, no certificate pinning, no custom transport.
- **Lines 79–84**: Issues a plain `http.GET` with no transport-layer security.
- **Lines 95–98**: Decodes the response body directly into `nodesEnvelope` with zero authentication.
- **Lines 102–124**: Iterates over `payload.Nodes`, performing only syntactic checks (`NodeAccountID` non-empty, `AccountIDFromString` format parse, non-empty `host`, non-zero `port`). Every address that passes these trivial checks is inserted into the `network` map.
- **Line 122**: `addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))` — attacker-controlled host:port goes directly into the map.

Back in `sdk_client.go` line 22: `client = hiero.ClientForNetwork(netmap)` — the poisoned map becomes the sole node list for the `hiero.Client`. Every call to `cryptoTransfer.Execute(client)` in `transfer.go` line 33 then routes to attacker-owned gRPC endpoints.

**Root cause:** The code assumes the HTTP response from `cfg.mirrorRest` is authoritative and trustworthy. There is no transport security (TLS), no server identity verification, no DNSSEC enforcement, and no out-of-band validation of returned node addresses.

**Why existing checks fail:**
- `AccountIDFromString` validates only the `0.0.X` format — it does not verify the account exists on-chain or belongs to a legitimate consensus node.
- The `host`/`port` non-empty checks accept any attacker-supplied string.
- The `len(network) == 0` guard (line 127) is trivially bypassed by returning one valid-looking entry.

### Impact Explanation

Once the `hiero.Client` is initialized with attacker-controlled gRPC endpoints, **all** `CryptoTransfer` transactions are submitted to those endpoints for the lifetime of the process. The attacker's nodes can:
1. **Drop transactions silently** — the pinger reports success (if the fake node returns a plausible receipt), while no real transfer occurs on the Hedera network.
2. **Observe transaction content** — operator account ID, destination account ID, amount, and the signed transaction bytes (including the operator's signature) are fully visible.
3. **Replay or relay selectively** — the attacker can forward some transactions and drop others, creating a partial-availability attack.
4. **Return fabricated receipts** — `resp.GetReceipt(client)` in `transfer.go` line 35 also queries the attacker's node, so success/failure status is fully under attacker control.

Severity: **Critical** for the `network=other` deployment mode. The entire purpose of the pinger (verifying live transaction submission) is subverted.

### Likelihood Explanation

**Preconditions:**
- `HIERO_MIRROR_PINGER_NETWORK=other` (required for this code path).
- The attacker can influence DNS resolution for the hostname in `cfg.mirrorRest`.

**Feasibility:** The canonical example value (`http://mirror-rest:5551`) uses plain HTTP and an internal Kubernetes/Docker service name. In a Kubernetes cluster, an attacker who has compromised any pod in the same namespace can manipulate CoreDNS responses or simply create a conflicting `Service` object (if RBAC is misconfigured). No DNSSEC or TLS is present to raise the bar. For deployments using a public FQDN without DNSSEC, classic DNS cache-poisoning (Kaminsky attack) or BGP prefix hijacking achieves the same result. The attack is repeatable on every pinger restart (the client is built once at startup from the poisoned response).

### Recommendation

1. **Enforce HTTPS with certificate validation**: Require `cfg.mirrorRest` to use `https://` and configure the `http.Client` with a strict `tls.Config` (system root CAs or a pinned CA). Reject plain-HTTP URLs at config validation time.
2. **Pin the mirror node's TLS certificate or CA**: Add a `tls.Config{RootCAs: pinnedPool}` to the transport so a DNS-redirected server cannot present a different certificate.
3. **Validate returned node addresses against a known-good set**: For `network=other`, accept an operator-supplied allowlist of expected gRPC addresses/CIDRs and reject any `ServiceEndpoint` not on that list.
4. **Use DNSSEC-aware resolution**: Configure the Go resolver to require DNSSEC validation, or use a resolver that enforces it.
5. **Separate bootstrap trust from runtime data**: Do not derive the entire consensus node list from a single unauthenticated HTTP call; require the operator to supply at least one pinned node address as a fallback.

### Proof of Concept

**Environment:** `network=other`, `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (default example).

**Steps:**

1. **Control DNS**: In the same Kubernetes namespace, create a `Service` named `mirror-rest` pointing to an attacker-controlled pod, overriding the legitimate service (or poison the cluster's CoreDNS cache).

2. **Run attacker HTTP server** on the pod, serving at `/api/v1/network/nodes`:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{
         "domain_name": "attacker.example.com",
         "ip_address_v4": "",
         "port": 50211
       }]
     }],
     "links": {"next": null}
   }
   ```

3. **Start the pinger** with `network=other`. `buildNetworkFromMirrorNodes` resolves `mirror-rest` to the attacker's IP, fetches the fabricated envelope, and builds `netmap = {"attacker.example.com:50211": AccountID{0,0,3}}`.

4. `hiero.ClientForNetwork(netmap)` sets the attacker's gRPC endpoint as the sole consensus node.

5. Every `submitWithRetry` call in `transfer.go` line 33 sends the signed `CryptoTransfer` to `attacker.example.com:50211`. The attacker's gRPC server receives the full signed transaction, can log it, drop it, or return a fake `SUCCESS` receipt.

6. The pinger logs `transfer success` while no real transaction reaches the Hedera network.
