### Title
Unauthenticated Alias Lookup DoS via Uncached DB Queries in `getTokenRelationships()`

### Summary
An unprivileged attacker can flood `/accounts/{alias}/tokens` with syntactically valid but non-existent account aliases, causing each request to execute a direct database query (`entityFromAliasQuery`) with no caching. Because the response cache only stores successful (2xx) responses, 404 results from missing aliases are never cached, meaning every such request unconditionally hits the database. With no rate limiting present in the REST layer, an attacker can sustain arbitrarily high DB load.

### Finding Description

**Exact code path:**

1. `getTokenRelationships()` calls `EntityService.getEncodedId()` unconditionally on the path parameter: [1](#0-0) 

2. `getEncodedId()` detects a valid alias format and calls `getAccountIdFromAlias()`: [2](#0-1) 

3. `getAccountIdFromAlias()` calls `getAccountFromAlias()`, which executes `entityFromAliasQuery` directly against the DB via `super.getRows()` — no cache involved: [3](#0-2) 

4. The raw SQL query: [4](#0-3) 

5. When the alias is not found, `getAccountIdFromAlias()` throws `NotFoundError` (→ HTTP 404): [5](#0-4) 

**Why the cache does not help:**

The `responseCacheUpdateHandler` only writes to Redis when the response is a success status code: [6](#0-5) 

A 404 from a missing alias is never stored in Redis. The cache key is derived from `req.originalUrl`: [7](#0-6) 

So even if the same alias is repeated, the 404 is not cached and each request re-executes the DB query.

**No rate limiting exists** in the REST service: [8](#0-7) 

### Impact Explanation
Every request with a non-existent alias causes a full table scan or index lookup on the `entity` table filtered by `alias`. Under sustained flood, this exhausts DB connection pool capacity and query throughput, degrading or denying service to all legitimate users of the mirror node REST API. This is a non-network-based DoS because the bottleneck is DB compute/IO, not bandwidth.

### Likelihood Explanation
No authentication or authorization is required. The alias format is publicly documented and trivially generated (e.g., random base32 strings). An attacker with a single machine and a simple loop can generate millions of unique non-existent aliases, each triggering a fresh DB query. The attack is fully repeatable and requires no special knowledge or privileges.

### Recommendation
1. **Cache negative (404) alias lookup results** at the application layer (e.g., in Redis with a short TTL such as 30–60 seconds), so repeated lookups for the same non-existent alias do not hit the DB.
2. **Add rate limiting** per IP (and optionally per alias) at the middleware layer for all `/accounts/:id/tokens` and similar alias-resolving endpoints.
3. **Ensure a DB index exists** on `entity.alias` to minimize per-query cost while the above mitigations are implemented.
4. Consider returning a generic 400 for structurally invalid aliases before any DB call is made.

### Proof of Concept
```bash
# Generate and send requests with unique non-existent aliases in a tight loop
for i in $(seq 1 10000); do
  # AAAA...AAAB style base32 aliases that are syntactically valid but don't exist
  ALIAS="0.0.AAAAAAAAAAAAAAAAAAAAAAAAAA$(printf '%04d' $i)"
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/tokens" &
done
wait
```
Each request triggers `entityFromAliasQuery` against the DB. Observe DB CPU/query rate spike. No caching absorbs the load. Legitimate requests begin timing out as the DB connection pool saturates.
