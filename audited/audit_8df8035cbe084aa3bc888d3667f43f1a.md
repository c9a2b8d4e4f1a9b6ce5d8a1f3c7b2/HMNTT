### Title
Unbounded Timestamp Range Enables Full Historical Scan of `hook_storage_change` via Unauthenticated GET Endpoint

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts user-controlled `timestamp` parameters that are passed directly to `Bound.of()` with no minimum-range or maximum-span enforcement. Supplying `timestamp=gte:0` alone is sufficient to set an effective scan window of `[0, Long.MAX_VALUE]`, triggering a full historical scan of the `hook_storage_change` table. No authentication or authorization is visible on the controller, making this exploitable by any unauthenticated caller.

### Finding Description

**Code path:**

`HooksController.java` line 110–111 accepts `TimestampParameter[] timestamps` with only `@Size(max = 2)` (cardinality limit, not value limit):

```java
@RequestParam(name = TIMESTAMP, required = false, defaultValue = "") @Size(max = 2)
        TimestampParameter[] timestamps,
``` [1](#0-0) 

At line 186, these are forwarded verbatim to `Bound.of()`:

```java
final var bound = Bound.of(timestamps, TIMESTAMP, HookStorageChange.HOOK_STORAGE_CHANGE.CONSENSUS_TIMESTAMP);
``` [2](#0-1) 

Inside `Bound.of()` (`Bound.java` lines 169–182), the `Bound` constructor is called with `primarySortField = false`: [3](#0-2) 

Inside the `Bound` constructor, the range-validity check is gated on `primarySortField`:

```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}
``` [4](#0-3) 

Because `primarySortField` is `false` for timestamp bounds, **no range validation fires at all**.

When `upper` is `null` (only a lower bound supplied), `adjustUpperBound()` returns `Long.MAX_VALUE`:

```java
public long adjustUpperBound() {
    if (this.upper == null) {
        return Long.MAX_VALUE;
    }
    ...
}
``` [5](#0-4) 

`HookStorageRequest.isHistorical()` returns `true` whenever the `Bound` is non-empty (i.e., `lower != null || upper != null`): [6](#0-5) 

**Note on the exact values in the question:** The value `9223372036854775807` (Long.MAX_VALUE, 19 digits) is rejected by `TimestampParameter`'s regex `\\d{1,17}`, so the two-parameter form as literally written fails at parse time. However, supplying only `timestamp=gte:0` achieves an **identical effective range** `[0, Long.MAX_VALUE]` and bypasses that constraint entirely. [7](#0-6) 

### Impact Explanation
Any caller can force the service into "historical" mode with an unbounded time window. The database must scan the entire `hook_storage_change` table from consensus timestamp 0 to the present to satisfy the query. The `limit` parameter caps returned rows but does not prevent the full index/table scan needed to locate the first N results. Repeated requests constitute a low-cost denial-of-service against the database, and the response leaks the complete gossip-prevention storage history for any `(ownerId, hookId)` pair.

### Likelihood Explanation
The endpoint requires no authentication (no `@PreAuthorize` or security filter visible on the controller). The exploit requires a single HTTP GET request with one query parameter (`timestamp=gte:0`). It is trivially repeatable by any external user with network access, making automated abuse straightforward.

### Recommendation
1. **Enforce a maximum timestamp span**: Reject requests where `adjustedUpper - adjustedLower` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Require an upper bound when a lower bound is supplied**: If `lower != null && upper == null`, either default `upper` to `now()` or reject the request.
3. **Add authentication/authorization**: Protect the endpoint so only the account owner or privileged callers can query historical storage.
4. **Pass `primarySortField = true`** (or add a dedicated range-span check) for timestamp bounds so the existing `adjustedLower > adjustedUpper` guard is active and a future maximum-span check has a natural home.

### Proof of Concept
```
# Full historical scan — no authentication required
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=25

# Equivalent two-parameter form (upper bound within 17-digit regex limit)
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&timestamp=lte:99999999999999999&limit=25
```

Both requests cause `Bound.of()` to produce a `Bound` with `lower = GTE:0` and effective `upper = Long.MAX_VALUE`, `isHistorical()` returns `true`, and the service executes a full historical scan of `hook_storage_change` with no time restriction.
