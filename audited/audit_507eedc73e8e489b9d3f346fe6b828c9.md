### Title
Unauthenticated Resource Exhaustion via Uncached `RetrieveLatest()` in `AccountBalance` Endpoint

### Summary
The `/account/balance` endpoint in `rosetta/app/services/account_service.go` unconditionally executes a live database query on every request when `BlockIdentifier` is omitted. There is no caching at the `blockRepository.RetrieveLatest()` layer and no application-level rate limiting, allowing any unauthenticated caller to sustain elevated DB I/O and CPU by flooding the endpoint with minimal-payload requests.

### Finding Description
**Code path:**

In `rosetta/app/services/account_service.go` lines 52–56, when `request.BlockIdentifier == nil`, `a.RetrieveLatest(ctx)` is called unconditionally:

```go
if request.BlockIdentifier != nil {
    block, rErr = a.RetrieveBlock(ctx, request.BlockIdentifier)
} else {
    block, rErr = a.RetrieveLatest(ctx)   // ← triggered on every no-block request
}
```

`RetrieveLatest()` in `rosetta/app/services/base_service.go` lines 100–105 delegates directly to `b.blockRepo.RetrieveLatest(ctx)` with no caching or deduplication.

`blockRepository.RetrieveLatest()` in `rosetta/app/persistence/block.go` lines 190–208 executes a raw SQL query on every invocation:

```go
db.Raw(selectLatestWithIndex).First(rb)
// selectLatestWithIndex = SELECT ... FROM record_file ORDER BY index DESC LIMIT 1
```

After `RetrieveLatest()` succeeds, `a.accountRepo.RetrieveBalanceAtBlock()` is also called (line 61), meaning each request triggers **at least two uncached DB queries**.

**Why existing checks fail:**

The Traefik middleware chain (which includes `rateLimit: average: 10` and `inFlightReq: amount: 5`) is gated by `global.middleware: false` in `charts/hedera-mirror-rosetta/values.yaml` line 95, and the template at `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3 requires both `global.middleware` and `middleware` to be truthy. Since `global.middleware` defaults to `false`, the entire middleware chain — including rate limiting — is **disabled by default**.

The application's own middleware stack in `rosetta/main.go` lines 217–219 applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which limit request rate or add caching.

Unlike the web3 stack (`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java` lines 34–39), which uses `@Cacheable` on `findLatest()`, the Rosetta `blockRepository` has no equivalent caching.

### Impact Explanation
An attacker sending a sustained flood of POST requests to `/account/balance` with a valid account address and no `block_identifier` forces the DB to execute `SELECT … FROM record_file ORDER BY index DESC LIMIT 1` plus a balance lookup on every request. On a large `record_file` table this is a sequential scan or index scan under write contention. With no concurrency cap at the application layer, this can saturate DB connection pools, increase CPU on both the Go process and the DB server, and degrade service for legitimate users. The 30% resource consumption threshold is achievable with a modest request rate (tens of requests per second) from a single host.

### Likelihood Explanation
The endpoint is public and unauthenticated. The minimum valid request body is small (network identifier + account address, no block identifier). No special knowledge, credentials, or protocol state is required. The attack is trivially scriptable with any HTTP client (curl, wrk, k6). The default deployment disables the only rate-limiting layer, making this exploitable in any default Helm-based deployment.

### Recommendation
1. **Cache `RetrieveLatest()` results** in `blockRepository` with a short TTL (e.g., 1–2 seconds), consistent with how the web3 stack caches `findLatest()` with `@Cacheable`.
2. **Enable application-level rate limiting** in `rosetta/main.go` (e.g., `golang.org/x/time/rate` token-bucket per remote IP) so protection is not dependent on optional infrastructure middleware.
3. **Change the Helm default** for `global.middleware` to `true` so the Traefik `rateLimit` and `inFlightReq` middleware are active in default deployments.
4. **Add a DB connection pool cap** in the Rosetta DB client to bound the maximum concurrent queries regardless of request rate.

### Proof of Concept
```bash
# Minimal valid request body — no block_identifier
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.98"}}'

# Flood the endpoint from a single unauthenticated client
while true; do
  curl -s -o /dev/null -X POST http://<rosetta-host>/account/balance \
       -H 'Content-Type: application/json' \
       -d "$BODY" &
done
```

Each iteration triggers `blockRepository.RetrieveLatest()` (raw SQL `ORDER BY index DESC LIMIT 1`) and `accountRepo.RetrieveBalanceAtBlock()` with no caching or rate gate. Monitor DB CPU and query throughput (e.g., `pg_stat_activity`) to observe the sustained load increase relative to the 24-hour baseline.
