### Title
Unauthenticated DB Query Amplification via Uncached `findLatest()` on `/api/v1/network/stake`

### Summary
`NetworkStakeRepository.findLatest()` in the rest-java module executes a native correlated SQL query on every invocation with no caching, no application-level rate limiting, and no authentication on the calling endpoint. During a database network partition, any unprivileged external user can flood `GET /api/v1/network/stake` to multiply failed DB connection attempts, exhaust the shared connection pool, and cascade failures to unrelated endpoints. The Traefik retry middleware (3 attempts per request) further triples the DB-layer amplification factor.

### Finding Description

**Exact code path:**

`NetworkStakeRepository.findLatest()` issues a native correlated subquery on every call:

```java
// rest-java/src/main/java/org/hiero/mirror/restjava/repository/NetworkStakeRepository.java:12-19
@Query(value = """
    select *
    from network_stake
    where consensus_timestamp = (
        select max(consensus_timestamp) from network_stake
    )
    """, nativeQuery = true)
Optional<NetworkStake> findLatest();
``` [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` calls it directly with no caching layer:

```java
// rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java:52-56
public NetworkStake getLatestNetworkStake() {
    return networkStakeRepository
            .findLatest()
            .orElseThrow(() -> new EntityNotFoundException("No network stake data found"));
}
``` [2](#0-1) 

The public, unauthenticated `@GetMapping("/stake")` endpoint calls this on every HTTP request:

```java
// rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java:126-130
@GetMapping("/stake")
NetworkStakeResponse getNetworkStake() {
    final var networkStake = networkService.getLatestNetworkStake();
    return networkStakeMapper.map(networkStake);
}
``` [3](#0-2) 

**Root cause — failed assumption:** The design assumes that the data-access layer is protected by either caching or rate limiting. Neither exists here. By contrast, the grpc module's analogous `NodeStakeRepository.findAllStakeByConsensusTimestamp()` is explicitly annotated `@Cacheable(cacheManager = NODE_STAKE_CACHE, cacheNames = CACHE_NAME)`. [4](#0-3) 

The rest-java module has no `ThrottleConfiguration`, no `ThrottleManagerImpl`, and no per-IP rate-limit middleware — unlike the web3 module which has all three. [5](#0-4) 

**Existing checks and why they are insufficient:**

1. **Traefik circuit breaker** (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) — trips only after 10 % of all requests are already failing; significant connection-pool damage occurs before activation. [6](#0-5) 

2. **Traefik retry middleware** (`attempts: 3, initialInterval: 100ms`) — actively worsens the situation: each inbound HTTP request generates **3** DB connection attempts during a partition. [7](#0-6) 

3. **GCP `maxRatePerEndpoint: 250`** — a soft backend-policy limit, not a per-source-IP hard cap; a distributed flood from multiple IPs is not constrained. [8](#0-7) 

### Impact Explanation

During a DB network partition every request to `GET /api/v1/network/stake` attempts a new connection that times out or errors. With the retry middleware, each HTTP request produces 3 failed DB connection attempts. A sustained flood of N requests/second produces 3N connection-pool slots consumed per second. Once the shared HikariCP pool is exhausted, **all** rest-java endpoints (supply, exchange rate, network nodes, etc.) begin returning 500 errors regardless of whether their own DB paths are partitioned. Error log volume grows proportionally to request rate, potentially filling disk or overwhelming log aggregation. The data returned (`network_stake`) changes only once per day (staking period), making the absence of even a short-lived cache (e.g., 60 s TTL) entirely unjustifiable from a correctness standpoint.

### Likelihood Explanation

Preconditions are minimal: no account, no token, no special header. The endpoint is reachable from the public internet via the ingress path `/api/v1/network/`. A single attacker with a modest botnet (or even a single host using HTTP/1.1 pipelining or HTTP/2 multiplexing) can sustain thousands of requests per second. The attack is repeatable, stateless, and requires no knowledge of the application beyond the public OpenAPI spec, which documents the endpoint. [9](#0-8) 

### Recommendation

1. **Add `@Cacheable`** to `findLatest()` with a TTL matching the staking-period update frequency (≥ 60 s). The grpc module's `NODE_STAKE_CACHE` pattern is the correct template.
2. **Add per-IP rate limiting** to the rest-java Traefik middleware chain (mirror the `inFlightReq` + `rateLimit` middleware present in the rosetta chart).
3. **Remove or gate the retry middleware** for 5xx responses caused by DB errors, or configure it to not retry on connection-pool-exhaustion errors, to prevent the 3× amplification.
4. **Set a DB statement timeout** for the rest-java datasource (the web3 module uses `statementTimeout: 3000` ms) so that partitioned queries fail fast rather than holding pool slots.

### Proof of Concept

```bash
# Simulate partition amplification (no credentials required)
# Step 1: Induce or wait for a DB network partition event.
# Step 2: From any host, flood the endpoint:
while true; do
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/network/stake" &
done
# Each request → findLatest() → native SQL → failed DB connection (×3 via retry).
# Monitor HikariCP pool metrics; pool exhaustion cascades to all other endpoints.
# No authentication, no rate limit, no cached response to short-circuit the DB call.
```
