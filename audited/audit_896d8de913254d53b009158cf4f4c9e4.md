### Title
Cache Bypass via Timestamp Parameter Enumeration Prevents Token Cache Warming and Forces Repeated Expensive DB Queries

### Summary
`getTokenInfoRequest()` in `rest/tokens.js` only calls `TokenService.putTokenCache()` when `filters.length === 0`. Any request with a `timestamp` filter skips cache population entirely. Because the HTTP response cache (`responseCacheHandler.js`) keys on the full URL (MD5 of `req.originalUrl`), an attacker supplying a different timestamp value per request generates a unique cache key each time, bypassing both the HTTP response cache and the in-memory token cache, forcing a new expensive DB query (UNION ALL across live and history tables) on every request.

### Finding Description

**Exact code path:**

`rest/tokens.js`, `getTokenInfoRequest()`, lines 548–565:
```js
const token = rows[0];
if (filters.length === 0) {          // line 561 — cache only populated without filters
    TokenService.putTokenCache(token);
}
```

`rest/middleware/responseCacheHandler.js`, line 152:
```js
const cacheKeyGenerator = (req) =>
  crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
```

**Root cause:** Two independent caches both fail when a timestamp filter is present:

1. **HTTP response cache** — keyed on the full URL including query string. Each unique `?timestamp=lte:<value>` produces a distinct cache key, so every distinct timestamp value triggers a fresh DB query.
2. **In-memory token cache** (`TokenService.putTokenCache`) — explicitly skipped at line 561 whenever `filters.length !== 0`.

**Exploit flow:**

The attacker sends a stream of requests with monotonically incrementing (or arbitrary) timestamp values:
```
GET /api/v1/tokens/0.0.1234?timestamp=lte:1000000000
GET /api/v1/tokens/0.0.1234?timestamp=lte:1000000001
GET /api/v1/tokens/0.0.1234?timestamp=lte:1000000002
...
```

Each request:
- Misses the HTTP response cache (unique URL).
- Executes `extractSqlFromTokenInfoRequest` with `filters.length !== 0`, which builds a `buildHistoryQuery` — a `UNION ALL` across both the live table and `*_history` table for `token`, `entity`, and `custom_fee` (six sub-queries total), which is significantly more expensive than the no-filter path.
- Skips `putTokenCache`, so the in-memory LRU cache is never warmed for this token.

**Why existing checks fail:**

- `validateTokenInfoFilter` (lines 451–463) accepts `eq`, `lt`, `lte` operators with any valid timestamp value. The timestamp space has nanosecond precision (up to 19 digits), giving an attacker effectively unlimited unique values.
- There is no per-IP or per-endpoint rate limiting on the REST API (the `ThrottleManager` is web3-only, `rest/server.js` lines 94–98 show only the response cache check, no rate limiter).
- The `acceptedSingleTokenParameters` set (line 1086) intentionally allows `TIMESTAMP`, so the filter is not rejected.

### Impact Explanation

Each attacker request forces a six-sub-query `UNION ALL` DB operation against `token`, `token_history`, `entity`, `entity_history`, `custom_fee`, and `custom_fee_history`. Sustained enumeration across many token IDs and timestamp values can saturate the DB connection pool, increase query latency for all users, and permanently prevent the in-memory token cache from being populated via this endpoint — causing downstream endpoints that rely on `getCachedTokens()` (e.g., account token relationship queries) to fall back to their own DB queries more frequently. The impact is performance degradation across the entire mirror node REST API.

### Likelihood Explanation

No authentication or privilege is required. The attack requires only an HTTP client and knowledge of any valid token ID (publicly enumerable via `GET /api/v1/tokens`). The timestamp parameter space is unbounded in practice. The attack is trivially scriptable, repeatable indefinitely, and requires no special tooling. The absence of REST-layer rate limiting means there is no automatic circuit breaker.

### Recommendation

1. **Populate the token cache regardless of filters** — the current-state token data returned is valid for caching even when a historical query is made; or at minimum cache the result when the timestamp filter resolves to the current state.
2. **Add rate limiting to the REST API** — apply a per-IP request rate limiter (e.g., `express-rate-limit`) to all endpoints, mirroring the web3 `ThrottleManager` pattern.
3. **Cap the timestamp parameter cardinality** — reject requests where the timestamp value is in the far future or outside a reasonable range relative to the chain's current consensus timestamp.
4. **Consider a shared HTTP response cache key normalization** — round timestamp values to a coarse granularity (e.g., nearest second) to collapse many unique URLs into a smaller set of cache keys.

### Proof of Concept

```bash
# No authentication required. Replace TOKEN_ID with any valid token.
TOKEN_ID="0.0.1234"
BASE="https://<mirror-node>/api/v1/tokens/${TOKEN_ID}"

for i in $(seq 1 10000); do
  curl -s "${BASE}?timestamp=lte:${i}000000000" -o /dev/null &
done
wait
```

Each iteration hits a unique URL, bypasses both caches, and executes a six-sub-query `UNION ALL` against the DB. Scaling across multiple token IDs and concurrent connections amplifies DB load proportionally with no server-side throttle to stop it.
