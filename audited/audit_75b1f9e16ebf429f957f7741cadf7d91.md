### Title
`sanitize()` Conflates U+FFFD with Sanitized U+0000, Permanently Corrupting Entity Memo Historical Integrity

### Summary
`DomainUtils.sanitize()` replaces the null character (U+0000) with the Unicode replacement character U+FFFD, but U+FFFD is itself a valid, user-submittable character. Any unprivileged user can embed U+FFFD directly in a transaction memo or entity description, producing a stored value byte-for-byte identical to one that was sanitized from a null byte. The mirror node's historical record becomes permanently ambiguous with no mechanism to distinguish the two cases.

### Finding Description
**Exact code location:**

`common/src/main/java/org/hiero/mirror/common/util/DomainUtils.java`, lines 54–55 and 239–241:

```java
private static final char NULL_CHARACTER = (char) 0;        // U+0000
private static final char NULL_REPLACEMENT = ''; // U+FFFD

public static String sanitize(String input) {
    return StringUtils.isNotEmpty(input) ? input.replace(NULL_CHARACTER, NULL_REPLACEMENT) : input;
}
```

`sanitize()` is called unconditionally in `AbstractEntity.setMemo()` (lines 156–158 and 196–198 of `AbstractEntity.java`) for every entity memo set via transaction handlers (e.g., `ContractUpdateTransactionHandler`, topic/account/token create/update handlers).

**Root cause:** The replacement character U+FFFD is chosen as the sentinel for sanitized null bytes, but U+FFFD is a legal Unicode scalar value (UTF-8: `0xEF 0xBF 0xBD`) that any user can include in a Hedera transaction memo or entity description. The function performs a pure character substitution with no escaping, tagging, or secondary encoding. The failed assumption is that U+FFFD will never appear in legitimate on-chain input.

**Exploit flow:**

1. Attacker submits a `CryptoCreateAccount` / `ConsensusCreateTopic` / `ContractCreate` (or any update variant) with a memo containing U+FFFD, e.g., `"legit\uFFFDmemo"`.
2. The importer calls `entity.setMemo("legit\uFFFDmemo")` → `DomainUtils.sanitize("legit\uFFFDmemo")` → `"legit\uFFFDmemo"` (no change; U+FFFD ≠ U+0000).
3. A separate, honest transaction with memo `"legit\u0000memo"` is processed → `DomainUtils.sanitize("legit\u0000memo")` → `"legit\uFFFDmemo"`.
4. Both are stored identically in the `entity.memo` column. No flag, escape sequence, or secondary field distinguishes them.

**Why existing checks are insufficient:**

The only guard is `StringUtils.isNotEmpty(input)`, which is a null/empty check entirely unrelated to character content. There is no allowlist, no escaping scheme, and no secondary storage of the original bytes for entity memos (unlike `Transaction.memo`, which is stored as raw `byte[]`). The test in `DomainUtilsTest.java` (lines 307–313) only verifies that null bytes are replaced; it never tests that a pre-existing U+FFFD is preserved distinctly.

### Impact Explanation
The mirror node's stated purpose is to provide an accurate, queryable historical record of the Hashgraph. Entity memos (account descriptions, topic memos, token names/symbols via the same sanitize path) are stored exclusively as the sanitized string — there is no raw-bytes fallback for these fields. Once written, the ambiguity is permanent: forensic auditors, compliance tools, and downstream consumers querying the REST API or database cannot determine whether the original on-chain memo contained U+0000 or U+FFFD. An attacker can deliberately craft memos that appear to have been sanitized, or can make a legitimately sanitized record appear to have been user-supplied, undermining the integrity guarantee the mirror node is designed to provide.

### Likelihood Explanation
Exploitation requires only the ability to submit a standard Hedera transaction — no keys, no elevated permissions, no special network access. U+FFFD is a well-known Unicode character; any developer or adversary aware of the sanitization scheme can trivially include it. The attack is repeatable at will for every entity type that uses `AbstractEntity.setMemo()`. The cost is a single transaction fee (~$0.001 USD on mainnet).

### Recommendation
Replace the lossy single-character substitution with a reversible escaping scheme. Two concrete options:

1. **Percent-encode or backslash-escape null bytes before storage**, e.g., replace `\u0000` with the two-character sequence `\0` (or `%00`), and ensure U+FFFD is stored as-is. This makes the two cases distinguishable.
2. **Store entity memos as `bytea` (raw bytes) rather than `text`**, consistent with how `Transaction.memo` is already handled, preserving the original byte sequence exactly and delegating display-layer sanitization to the REST API.

At minimum, add a test asserting that `sanitize("abc\uFFFD")` and `sanitize("abc\u0000")` produce **distinct** outputs, which will immediately surface the collision.

### Proof of Concept
```
// Both calls return the identical String "abc\uFFFD":
String fromNull        = DomainUtils.sanitize("abc" + (char) 0);   // U+0000 → U+FFFD
String fromReplacement = DomainUtils.sanitize("abc\uFFFD");         // U+FFFD → U+FFFD (unchanged)

assert fromNull.equals(fromReplacement);          // TRUE — collision confirmed
assert fromNull.getBytes(UTF_8) ==
       fromReplacement.getBytes(UTF_8);           // identical byte arrays stored in DB
```

On-chain steps:
1. Submit `ConsensusCreateTopic` with memo bytes `[0x61, 0x62, 0x63, 0xEF, 0xBF, 0xBD]` (UTF-8 for `"abc"` + U+FFFD).
2. Submit a second `ConsensusCreateTopic` with memo bytes `[0x61, 0x62, 0x63, 0x00]` (`"abc"` + null byte).
3. Query both topics via the mirror node REST API (`/api/v1/topics/{id}`).
4. Observe that both return `"memo": "abc\uFFFD"` — indistinguishable.
