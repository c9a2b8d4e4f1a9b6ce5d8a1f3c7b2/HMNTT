### Title
Unauthenticated Database Load Amplification via Concurrent `TopicRepository.findById()` Calls with No Caching or Rate Limiting

### Summary
The `rest-java` `TopicRepository.findById()` has no `@Cacheable` annotation, no in-flight deduplication, and no rate limiting on the calling endpoint. Any unauthenticated user can send N concurrent GET requests to `/api/v1/topics/{id}` for the same topic ID, causing N independent SQL queries to execute against the database simultaneously. Unlike the `grpc` module's `EntityRepository`, which explicitly applies `@Cacheable`, this protection is absent in `rest-java`.

### Finding Description
**Code path:**

- `rest-java/src/main/java/org/hiero/mirror/restjava/repository/TopicRepository.java` lines 1–10: a bare `CrudRepository<Topic, Long>` with no `@Cacheable` or any deduplication annotation.
- `rest-java/src/main/java/org/hiero/mirror/restjava/service/TopicServiceImpl.java` lines 19–21: `topicRepository.findById(id.getId())` is called directly with no caching wrapper.
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java` lines 31–37: the `GET /api/v1/topics/{id}` endpoint is public, unauthenticated, and invokes `topicService.findById()` with no throttle guard.

**Root cause:** The `rest-java` module has no rate-limiting filter (the `ThrottleManagerImpl`/`ThrottleConfiguration` throttle exists only in the `web3` module) and no response or repository-level cache for topic lookups. The `MetricsFilter` in `rest-java` only records metrics and does not throttle.

**Contrast with existing pattern:** The `grpc` module's `EntityRepository.findById()` is annotated `@Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, unless = "#result == null")`, demonstrating the team is aware of the pattern and applied it selectively — but not to `rest-java`'s `TopicRepository`.

**Exploit flow:**
1. Attacker identifies a valid topic ID (publicly visible on-chain).
2. Attacker opens N concurrent HTTP connections and fires N simultaneous `GET /api/v1/topics/{id}` requests.
3. Each request independently reaches `TopicServiceImpl.findById()` → `topicRepository.findById()` → a separate `SELECT … FROM topic WHERE id = ?` query.
4. N identical queries execute in parallel against the database, consuming N connection pool slots and N units of DB CPU/IO.

**Why existing checks fail:** There is no per-IP rate limit, no global RPS cap, no `@Cacheable` on the repository method, and no request coalescing at any layer in `rest-java`.

### Impact Explanation
An attacker can multiply database load by an arbitrary factor N using a single topic ID and N concurrent connections. With a large enough N, this exhausts the JDBC connection pool, starves other legitimate queries (including importer writes), and degrades or denies service to all users of the mirror node REST API. The impact is bounded to the mirror node infrastructure (read replica DB and API pods) with no direct economic harm to network participants, consistent with the medium-severity griefing classification.

### Likelihood Explanation
No privileges, accounts, tokens, or special knowledge are required — only a valid topic ID (trivially obtained from any block explorer or prior API call). The attack is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool. It is repeatable indefinitely and requires no state. The only natural mitigations are infrastructure-level (DB connection pool size, upstream load balancer rate limits), which are not guaranteed to be present in all deployments.

### Recommendation
1. Add `@Cacheable` to `TopicRepository.findById()` in `rest-java`, mirroring the pattern already used in `grpc`'s `EntityRepository`.
2. Introduce a per-endpoint or global RPS throttle in `rest-java` analogous to `ThrottleManagerImpl` in the `web3` module.
3. Optionally, add a Caffeine-backed `CacheManager` bean in a `rest-java` `CacheConfiguration` class (following the pattern in `grpc/src/main/java/org/hiero/mirror/grpc/config/CacheConfiguration.java`) and wire it to the topic repository.

### Proof of Concept
```bash
# Replace TOPIC_ID with any valid topic entity number
TOPIC_ID=1234
BASE_URL="https://<mirror-node-host>/api/v1/topics/${TOPIC_ID}"

# Fire 500 concurrent requests for the same topic ID
for i in $(seq 1 500); do
  curl -s "$BASE_URL" -o /dev/null &
done
wait
```
Each of the 500 requests will independently execute `SELECT * FROM topic WHERE id = 1234` against the database. Observe via DB metrics (pg_stat_activity, connection pool saturation) that N identical queries run simultaneously rather than being coalesced or served from cache.
