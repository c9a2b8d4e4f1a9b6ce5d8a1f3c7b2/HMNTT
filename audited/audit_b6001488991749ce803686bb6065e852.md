### Title
Unbounded Historical Timestamp Scan DoS via Upper-Only `timestamp=lte:` Parameter on `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage`

### Summary
An unauthenticated external user can supply a single upper-bound timestamp parameter (e.g., `timestamp=lte:9223372036.854775807`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing `Bound.of()` to produce a `Bound` with `lower=null` and `upper=LTE:Long.MAX_VALUE`. This forces `getAdjustedLowerRangeValue()` to return `0` and `adjustUpperBound()` to return `Long.MAX_VALUE`, issuing a `consensus_timestamp BETWEEN 0 AND 9223372036854775807` query against the `hook_storage_change` table — a full historical scan for the given hook. No authentication or rate limiting exists on this endpoint, making it trivially repeatable.

### Finding Description

**Exact code path:**

`HooksController.java` line 186 calls `Bound.of(timestamps, TIMESTAMP, ...)`: [1](#0-0) 

Inside `Bound.of()`, if only one `LTE` timestamp is supplied, the loop at lines 47–54 sets `upper = param` and leaves `lower = null`. Because `primarySortField=false`, the range-inversion guard at line 58 is **never evaluated**: [2](#0-1) 

`getAdjustedLowerRangeValue()` returns `0` when `lower == null`: [3](#0-2) 

`adjustUpperBound()` returns `Long.MAX_VALUE` for an `LTE` operator with value `Long.MAX_VALUE`: [4](#0-3) 

Because `!timestamp.isEmpty()` is `true` (upper is set), `isHistorical()` returns `true` and `getHookStorageChange()` is invoked: [5](#0-4) 

`getHookStorageChange()` passes `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE` directly to the repository: [6](#0-5) 

The resulting SQL query is:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key >= :keyLowerBound
  AND key <= :keyUpperBound
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
``` [7](#0-6) 

`DISTINCT ON (key)` requires PostgreSQL to scan **all** rows matching `ownerId`/`hookId` before applying the `LIMIT`. The `@Max(MAX_LIMIT)` constraint (100 rows) only caps the returned result set, not the intermediate scan cost. [8](#0-7) 

**Root cause:** `Bound.of()` is called with `primarySortField=false` for the timestamp dimension, so no lower-bound enforcement or range-inversion check is applied. A single upper-bound parameter is sufficient to activate the historical code path with an unbounded lower range starting at `0`.

**Note on the exact input in the question:** `lte:9223372036854775807` (19 raw digits) is rejected by `TimestampParameter`'s regex `\d{1,17}`. The correct exploit input is `lte:9223372036.854775807` (seconds.nanos format), which `DomainUtils.convertToNanos(9223372036, 854775807)` converts to exactly `Long.MAX_VALUE`. Any large `lte:` value (e.g., `lte:9223372036`) achieves the same unbounded scan effect. [9](#0-8) 

**No authentication exists** on `HooksController` — no `@PreAuthorize`, `@Secured`, or Spring Security configuration was found for this endpoint: [10](#0-9) 

### Impact Explanation
An attacker can force the database to perform a full sequential scan of `hook_storage_change` for any known `(ownerId, hookId)` pair, bounded only by the key range (default: all keys). For a hook with a long history, this translates to massive I/O, CPU, and memory pressure on the database. Repeated concurrent requests from a single attacker can saturate the DB connection pool and degrade or deny service to all users of the mirror node REST API. Additionally, the response discloses the complete gossip-prevention storage change history for the targeted hook up to the specified timestamp, constituting an unauthorized data disclosure.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication or rate limiting in the `rest-java` module. The attacker only needs to know a valid `ownerId` and `hookId` (both enumerable via `GET /api/v1/accounts/{ownerId}/hooks`). The exploit requires a single crafted HTTP GET request and is trivially scriptable for high-frequency repetition. No privileged access, credentials, or special tooling are required.

### Recommendation
1. **Enforce a mandatory lower bound on the timestamp range** when the historical path is activated: if only an upper bound is supplied, default the lower bound to `upperBound - MAX_TIMESTAMP_WINDOW` (e.g., a configurable maximum window such as 30 days in nanoseconds) rather than `0`.
2. **Add a maximum timestamp range span check** in `hookStorageChangeRequest()` or `getHookStorageChange()`: reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configured threshold.
3. **Apply rate limiting** to the `GET /{hookId}/storage` endpoint, consistent with the throttling applied in the `web3` module.
4. Consider requiring `primarySortField=true` semantics (range inversion check) for the timestamp `Bound` in this context, or add an explicit validation step in `Bound.of()` for the storage-change use case.

### Proof of Concept

**Preconditions:** Mirror node REST API is reachable; a valid `ownerId` and `hookId` are known (obtainable via `GET /api/v1/accounts/{ownerId}/hooks`).

**Step 1 — Trigger full historical scan:**
```
GET /api/v1/accounts/0.0.1234/hooks/5678/storage?timestamp=lte:9223372036.854775807
```

**Step 2 — Verify the exploit path:**
- `TimestampParameter.valueOf("lte:9223372036.854775807")` → `TimestampParameter(LTE, 9223372036854775807)`
- `Bound.of(...)` → `lower=null`, `upper=LTE:Long.MAX_VALUE`
- `isHistorical()` → `true`
- `getAdjustedLowerRangeValue()` → `0`
- `adjustUpperBound()` → `9223372036854775807`
- Executed SQL: `consensus_timestamp BETWEEN 0 AND 9223372036854775807` — full table scan

**Step 3 — DoS via repetition:**
```bash
while true; do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1234/hooks/5678/storage?timestamp=lte:9223372036.854775807" &
done
```
Each concurrent request forces a full `DISTINCT ON (key)` scan of `hook_storage_change`, exhausting DB I/O and connection resources.

        this.field = field;
        this.parameterName = parameterName;

            return;

    }

    private static final Pattern PATTERN =
            Pattern.compile("^((eq|gt|gte|lt|lte|ne):)?(\\d{1,17})(\\.(\\d{1,9}))?$", Pattern.CASE_INSENSITIVE);

    public static TimestampParameter valueOf(String param) {
        if (StringUtils.isBlank(param)) {
            return EMPTY;
        }

        var matcher = PATTERN.matcher(param);
        if (!matcher.matches()) {
            throw new IllegalArgumentException(ERROR);
        }

        final var operator = parseOperator(matcher.group(2));
        final var timestamp = parseTimestamp(matcher.group(3), matcher.group(5));
        return new TimestampParameter(operator, timestamp);
    }

    private static RangeOperator parseOperator(String name) {
        if (StringUtils.isEmpty(name)) {
            return RangeOperator.EQ;
        }

        final var operator = RangeOperator.of(name);

        if (operator == RangeOperator.NE) {
            throw new IllegalArgumentException(ERROR);
        }

        return operator;
    }

    private static long parseTimestamp(String secondsStr, String nanosStr) {
        try {
            final long seconds = Long.parseLong(secondsStr);
            final long nanos = StringUtils.isNotEmpty(nanosStr) ? Long.parseLong(nanosStr) : 0L;
            return DomainUtils.convertToNanos(seconds, nanos);
        } catch (RuntimeException e) {
            throw new IllegalArgumentException(ERROR);
        }
    }
