### Title
Unauthenticated Connection Pool Exhaustion via Concurrent DB Queries in `getOneAccount()`

### Summary
`getOneAccount()` in `rest/accounts.js` unconditionally fires two concurrent database queries per request via `Promise.all()` — one entity query and one full `doGetTransactions()` chain — with no rate limiting on the endpoint. The default connection pool is only 10 connections, so as few as 5 concurrent unauthenticated requests with `transactions=true` and a timestamp filter exhaust the pool, causing all REST API endpoints to fail with connection timeout errors and effectively partitioning the REST layer from the database.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount()`, lines 487–495:
```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);          // fires immediately
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)             // fires immediately
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]);
```

`doGetTransactions()` (`rest/transactions.js`, lines 695–705) is `async` and immediately begins executing `getTransactionTimestamps()` (line 483 in `transactions.js`), which issues its own `pool.queryQuietly` call. This means the entity query and the transaction-timestamps query are in-flight simultaneously, consuming 2 pool connections per request at peak.

When timestamp filters are present, a third sequential query (`balances.getAccountBalanceTimestampRange`, line 435 in `accounts.js`) runs before `Promise.all`, and a fourth (`getTransactionsDetails`, line 662 in `transactions.js`) runs after `getTransactionTimestamps` resolves — still overlapping with the entity query if the entity query is slow.

**Root cause and failed assumption:**

The design assumes the connection pool is large enough relative to concurrent request volume. The default pool is **10 connections** (`rest/dbpool.js` line 14, `config.db.pool.maxConnections` default = 10 per docs). With 2 concurrent connections consumed per request, only 5 simultaneous requests are needed to exhaust the pool. There is no rate limiting, no concurrency cap, and no authentication requirement on this endpoint.

**Why existing checks fail:**

- `statementTimeout: 20000ms` (20 seconds) — connections are held for up to 20 seconds per query, making pool exhaustion easy to sustain.
- `connectionTimeoutMillis: 20000ms` — queued requests wait 20 seconds before failing, amplifying the denial-of-service window.
- `authHandler` middleware (`rest/middleware/authHandler.js`) only sets a custom response `limit` for authenticated users; it does not block or throttle unauthenticated requests.
- No rate-limiting middleware exists on the REST Node.js service for any endpoint (confirmed in `rest/server.js` and `rest/routes/accountRoute.js`). Rate limiting exists only in the separate web3 Java service.
- The `transactions=true` default (`includeTransactions = true`, line 402 in `accounts.js`) means an attacker does not even need to supply the parameter explicitly.

### Impact Explanation

When the 10-connection pool is exhausted, every subsequent request to **any** REST API endpoint (not just `/accounts/:id`) blocks waiting for a connection for up to 20 seconds, then fails with a `DbError`. This is a complete, service-wide denial of service — the REST layer is partitioned from the database. All users, including legitimate ones, are affected. The impact is equivalent to a network partition between the REST service and the database.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero special knowledge beyond the public API documentation. The attacker needs only 5 concurrent HTTP requests sustained over time. This is trivially achievable with any HTTP benchmarking tool (`ab`, `wrk`, `curl` in parallel). The attack is repeatable, requires no state, and can be sustained indefinitely. The default pool size of 10 makes this exploitable even from a single machine on a slow connection.

### Recommendation

1. **Add a request-level concurrency/rate limit** to the REST Node.js service (e.g., `express-rate-limit` or an ingress-level rate limiter) for the `/accounts/:id` endpoint.
2. **Increase the default pool size** (`hiero.mirror.rest.db.pool.maxConnections`) to a value proportional to expected concurrency, or document a minimum safe value.
3. **Reduce `statementTimeout`** to a lower value (e.g., 5000ms) to release connections faster under load.
4. **Serialize the queries** within `getOneAccount()` when under high load, or use a semaphore/queue to cap the number of concurrent DB connections consumed by this handler.
5. **Consider making `transactions=false` the default** for the single-account endpoint to reduce the default connection multiplier.

### Proof of Concept

```bash
# Exhaust the 10-connection pool with 6 concurrent requests
# (each holds 2 connections via Promise.all for up to 20s)
for i in $(seq 1 6); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.2?timestamp=gte:0&timestamp=lte:9999999999" \
    > /dev/null &
done
wait

# All subsequent requests to ANY endpoint now fail with connection timeout
curl -v "http://<mirror-node-host>/api/v1/accounts/0.0.3"
# Expected: 500 DbError / connection timeout after 20s
```

Reproducible steps:
1. Deploy the REST service with default configuration (`maxConnections=10`, `statementTimeout=20000`).
2. Send 6+ concurrent GET requests to `/api/v1/accounts/:id` with a broad timestamp range (no auth required).
3. Observe that all REST endpoints return 500 errors for up to 20 seconds per wave of requests.
4. Repeat step 2 continuously to maintain the denial of service.
