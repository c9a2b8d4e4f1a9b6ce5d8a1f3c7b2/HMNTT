### Title
Connection Pool Exhaustion via Slow Query Amplification in `getBalances()` — No Per-Request Timeout or Rate Limiting

### Summary
`getBalances()` in `rest/balances.js` issues multiple sequential `pool.queryQuietly()` calls with no per-request timeout, relying solely on the pool-level `statementTimeout` (default 20,000 ms). The default pool size is only 10 connections and the REST API has no in-flight request limiting or rate limiting middleware. An unauthenticated attacker sending 10 concurrent requests crafted to maximize query duration can saturate the pool, causing all subsequent legitimate requests to block for up to `connectionTimeout` (default 20,000 ms) before failing with HTTP 503.

### Finding Description

**Exact code path:**

`getBalances()` in `rest/balances.js` (lines 83–156) makes up to three sequential `pool.queryQuietly()` calls per request:
1. `getAccountIdsByPublicKey()` → `pool.queryQuietly(entityPublicKeyQuery, ...)` (line 66)
2. `getAccountBalanceTimestampRange()` → `pool.queryQuietly(query, params)` (line 189)
3. Final main query → `pool.queryQuietly(pgSqlQuery, sqlParams)` (line 153)

Each call borrows a connection from the global pool. There is no `AbortController`, no `Promise.race()` with a timeout, and no per-request deadline wrapping these calls.

**Pool configuration** (`rest/dbpool.js`, lines 7–16):
```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20,000 ms
  max: config.db.pool.maxConnections,                         // default 10
  statement_timeout: config.db.pool.statementTimeout,         // default 20,000 ms
};
```

**No rate limiting in the REST API server** (`rest/server.js`, lines 68–144): No `inFlightReq`, no `rateLimit`, no `express-rate-limit`, no concurrency cap middleware is applied to the `/api/v1/balances` route.

**Traefik middleware for the REST API** (`charts/hedera-mirror-rest/values.yaml`, lines 134–139) only configures `circuitBreaker` and `retry` — no `inFlightReq` or `rateLimit`, unlike the Rosetta API which has both.

**Root cause:** The failed assumption is that `statementTimeout` alone is sufficient to protect the pool. It limits individual query duration but does not prevent pool exhaustion when the number of concurrent slow requests equals `maxConnections`. The `connectionTimeoutMillis` bounds how long queued requests wait (20s), but during that window all legitimate requests are blocked or fail.

**Exploit flow:**
1. Attacker sends 10 concurrent GET requests to `/api/v1/balances?timestamp=lte:<old_timestamp>` with a timestamp far in the past, forcing a scan of large time-partitioned `account_balance` tables.
2. Each request acquires a pool connection and holds it for up to 20 seconds (statement timeout).
3. With all 10 connections occupied, the pool is exhausted.
4. Legitimate requests block waiting for a connection for up to `connectionTimeoutMillis` (20s), then fail with a `DbError` → HTTP 503.
5. As attacker requests complete, the attacker immediately sends replacements, maintaining pool saturation continuously.

### Impact Explanation

The pool has only 10 connections by default. Saturating it with 10 concurrent slow queries causes all other REST API requests (accounts, transactions, tokens, etc.) to fail with HTTP 503 for the duration of the attack. Since the REST API is a shared service, this degrades availability for all consumers. The attack is sustained with minimal resources (10 persistent HTTP connections). In a multi-pod deployment (HPA `minReplicas: 1`, `maxReplicas: 15`), each pod has its own pool of 10, but a single pod can be targeted directly, and the attacker can scale to target multiple pods. This meets the threshold of degrading ≥30% of mirror node processing capacity.

### Likelihood Explanation

The attack requires no authentication, no special knowledge, and no elevated privileges — only the ability to send HTTP GET requests to a public endpoint. The `timestamp` query parameter is documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, line 408). Crafting a request that triggers a slow query (e.g., a very old timestamp causing large partition scans) is trivial. The attacker needs only 10 concurrent connections, achievable with any basic HTTP load tool (`curl`, `ab`, `wrk`). The attack is repeatable and indefinitely sustainable.

### Recommendation

1. **Add per-request timeout**: Wrap each `pool.queryQuietly()` call (or the entire `getBalances()` handler) with an `AbortController`/`Promise.race()` timeout shorter than `statementTimeout`, so a single slow request does not hold a connection for the full 20 seconds.
2. **Add `inFlightReq` Traefik middleware** to the REST API chart (as done for Rosetta and GraphQL) to cap concurrent in-flight requests per source IP.
3. **Add `rateLimit` Traefik middleware** to the REST API chart to limit requests per second per client.
4. **Increase `maxConnections`** or reduce `statementTimeout` to reduce the attack window.
5. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore) directly in the Node.js server for the `/balances` route.

### Proof of Concept

```bash
# Send 10 concurrent requests with an old timestamp to force large partition scans
# Replace <HOST> with the mirror node REST API host
for i in $(seq 1 10); do
  curl -s "http://<HOST>/api/v1/balances?timestamp=lte:1000000000.000000000&limit=100" &
done
wait

# While the above are running, legitimate requests will block for up to 20s then return 503:
curl -v "http://<HOST>/api/v1/balances"
# Expected: HTTP 503 Service Unavailable (DbError: connection timeout)
```

  utils.validateReq(req, acceptedBalancesParameters, balanceFilterValidator);

  // Parse the filter parameters for credit/debit, account-numbers, timestamp and pagination
  const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
  const accountParams = await Promise.all(accountParamsPromise);
  // transform the timestamp=xxxx or timestamp=eq:xxxx query in url to 'timestamp <= xxxx' SQL query condition
  let [tsQuery, tsParams] = utils.parseTimestampQueryParam(req.query, 'consensus_timestamp', {
    [utils.opsMap.eq]: utils.opsMap.lte,
  });
  const [balanceQuery, balanceParams] = utils.parseBalanceQueryParam(req.query, 'ab.balance');
  const [pubKeyQuery, pubKeyParams] = utils.parsePublicKeyQueryParam(req.query, 'public_key');
  const {
    query: limitQuery,
    params,
    order,
    limit,
  } = utils.parseLimitAndOrderParams(req, constants.orderFilterValues.DESC);

  res.locals[constants.responseDataLabel] = {
    timestamp: null,
    balances: [],
    links: {
      next: null,
    },
  };

  let sqlQuery;
  let sqlParams;

  if (tsQuery) {
    const tsQueryResult = await getTsQuery(tsQuery, tsParams);
    if (!tsQueryResult.query) {
      return;
    }

    const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit);
    if (pubKeyQuery && !accountIdsQuery) {
      return;
    }

    [sqlQuery, tsParams] = await getBalancesQuery(
      accountQuery,
      balanceQuery,
      accountIdsQuery,
      limitQuery,
      order,
      tsQueryResult
    );
    sqlParams = utils.mergeParams(tsParams, accountParams, balanceParams, params);
  } else {
    // use current balance from entity table when there's no timestamp query filter
    const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
    const whereClause = conditions && `where ${conditions}`;
    const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
    sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
    sqlQuery = `
      with entity_balance as (
        select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
        from entity
        where type in ('ACCOUNT', 'CONTRACT')
      )
      select ab.*, (${tokenBalanceSubQuery}) as token_balances
      from entity_balance ab
      ${whereClause}
      order by ab.account_id ${order}
      ${limitQuery}`;
  }

  const pgSqlQuery = utils.convertMySqlStyleQueryToPostgres(sqlQuery);
  const result = await pool.queryQuietly(pgSqlQuery, sqlParams);
  res.locals[constants.responseDataLabel] = formatBalancesResult(req, result, limit, order);
