### Title
Cache-Bypass via Reference-Equality Misidentification of `BlockType.LATEST` Allows Unbounded DB Queries per Request

### Summary
`findByBlockType()` uses Java reference equality (`==`) to detect the `LATEST` sentinel, but `BlockType.of("9223372036854775807")` constructs a **new** `BlockType` instance with `number = Long.MAX_VALUE` that is structurally identical to `BlockType.LATEST` yet a different object. The `==` check fails, routing every such request to `findByIndex(Long.MAX_VALUE)` instead of `findLatest()`. Because `findByIndex` returns `Optional.empty()` for this non-existent index and Spring Cache's `unless = "#result == null"` prevents caching of empty Optionals, every request unconditionally hits the database — bypassing the `findLatest()` Caffeine cache entirely.

### Finding Description

**Exact code path:**

`BlockType.java` line 12 defines the singleton:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [1](#0-0) 

`BlockType.of("9223372036854775807")` falls into `extractNumericBlock()`, which calls `Long.parseLong("9223372036854775807", 10)` = `Long.MAX_VALUE` and returns `new BlockType("9223372036854775807", Long.MAX_VALUE)` — a **new heap object**, not the `LATEST` singleton: [2](#0-1) 

In `RecordFileServiceImpl.findByBlockType()`, the guard uses `==` (reference equality):
```java
} else if (block == BlockType.LATEST) {
    return recordFileRepository.findLatest();
}
return recordFileRepository.findByIndex(block.number());
``` [3](#0-2) 

Since `BlockType` is a Java `record`, `==` checks object identity, not value equality. The new instance is not `== BlockType.LATEST`, so the branch is skipped and `findByIndex(Long.MAX_VALUE)` is called.

**Cache bypass root cause:**

`findByIndex` is annotated:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
``` [4](#0-3) 

Spring Cache unwraps `Optional` return values before evaluating `unless`. `Optional.empty()` unwraps to `null`, making `#result == null` true, so the empty result is **never stored in the cache**. Every subsequent request with `block=9223372036854775807` misses the cache and issues a fresh DB query.

In contrast, `findLatest()` uses a dedicated Caffeine cache (`expireAfterWrite=500ms, maximumSize=1`): [5](#0-4) 

Under normal load, many `latest`-block requests within each 500 ms window are served from cache. The attack replaces those cheap cache hits with uncached DB round-trips on every single request.

The `findByIndex` cache itself has `expireAfterWrite=10min, maximumSize=10000`: [6](#0-5) 

But because the empty result is never stored, the 10-minute TTL never applies to this key.

### Impact Explanation
An attacker can force the mirror node's web3 service to issue a live database query (`SELECT … WHERE index = 9223372036854775807`) on every single request, bypassing the Caffeine cache that normally absorbs the majority of `eth_getBlockByNumber("latest")` traffic. Under high request rates this directly translates to proportional DB CPU and I/O load increase. The `findLatest()` cache window is only 500 ms, meaning at moderate traffic (e.g., 100 req/s) the cache absorbs ~49 out of every 50 requests; replacing those with uncached DB queries multiplies DB load by up to ~50×. The node returns an error/empty response to the caller, which also degrades API correctness for any client using the numeric equivalent of `latest`.

### Likelihood Explanation
No authentication or special privilege is required. The attack requires only the ability to send standard JSON-RPC or REST requests with a block parameter set to the decimal string `"9223372036854775807"` (or its hex equivalent `"0x7fffffffffffffff"`). Both decimal and hex paths go through `extractNumericBlock` and produce the same non-singleton instance. The attack is trivially scriptable, repeatable at arbitrary rate, and requires no knowledge of internal state. Any public-facing mirror node endpoint that accepts a block number parameter is exposed.

### Recommendation

1. **Fix the equality check** in `findByBlockType()` — use `.equals()` instead of `==`, or compare by `number()` value:
   ```java
   } else if (block.number() == BlockType.LATEST.number()) {
       return recordFileRepository.findLatest();
   }
   ```
   Since `BlockType` is a `record`, `block.equals(BlockType.LATEST)` also works correctly.

2. **Alternatively, normalize in `BlockType.of()`** — if the parsed number equals `Long.MAX_VALUE`, return the `LATEST` singleton directly:
   ```java
   if (blockNumber == Long.MAX_VALUE) return LATEST;
   return new BlockType(value, blockNumber);
   ``` [2](#0-1) 

3. **Add an upper-bound validation** in `extractNumericBlock` rejecting block numbers that exceed the current chain height, preventing queries for obviously non-existent indices.

### Proof of Concept

**Precondition:** Mirror node web3 service is running and accessible. No credentials needed.

**Trigger (repeat at high rate):**
```bash
# Decimal Long.MAX_VALUE — bypasses LATEST singleton check
curl -X POST http://<mirror-node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["9223372036854775807",false],"id":1}'

# Hex equivalent — same effect
curl -X POST http://<mirror-node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["0x7fffffffffffffff",false],"id":1}'
```

**Observed result:** Each request executes `SELECT r FROM RecordFile r WHERE r.index = 9223372036854775807` against the database. The empty `Optional` result is never cached. DB query count grows linearly with request rate instead of being absorbed by the 500 ms Caffeine cache. Monitoring DB query counters will show a sustained spike proportional to the attack request rate, while legitimate `latest` requests served from cache would show near-zero DB hits for the same traffic volume.
