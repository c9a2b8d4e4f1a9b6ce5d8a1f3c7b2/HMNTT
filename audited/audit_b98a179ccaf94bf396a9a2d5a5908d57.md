### Title
Silent Balance Suppression via Crafted Timestamp Filter in `getAccountBalanceTimestampRange()`

### Summary
An unprivileged external user can supply a timestamp filter targeting a time range where no treasury account balance snapshot exists (e.g., epoch-era timestamps or any gap in the partitioned replica). `getAccountBalanceTimestampRange()` returns `{}` when the DB query finds zero rows, and the `/balances` caller chain silently returns HTTP 200 with an empty balances array — indistinguishable from a legitimate "no accounts" result. No error code, no diagnostic field, and no client signal is emitted.

### Finding Description

**Full call chain for `GET /api/v1/balances?timestamp=lte:1`:**

1. `getBalances()` (`rest/balances.js` line 90–92) parses the user-supplied `timestamp=lte:1` into `tsQuery = "consensus_timestamp <= ?"` and `tsParams = [1]`.

2. `getBalances()` calls `getTsQuery(tsQuery, tsParams)` at line 114.

3. `getTsQuery()` (lines 278–292) calls `getAccountBalanceTimestampRange(tsQuery, tsParams)` at line 279.

4. `getAccountBalanceTimestampRange()` (lines 167–197) calls `getOptimizedTimestampRange()` (lines 229–276):
   - `upperBound = 1n` (from user input)
   - `effectiveUpperBound = min(1n, nowInNs) = 1n`
   - `optimalLowerBound = getFirstDayOfMonth(1n, -1)` → near-epoch value
   - `lowerBound = max(0n, optimalLowerBound)` → effectively `0n`
   - Returns `{lowerBound: 0n, upperBound: 1n, neParams: []}` — **valid, non-empty object**

5. Back in `getAccountBalanceTimestampRange()` (lines 175–192), the function builds and executes:
   ```sql
   SELECT consensus_timestamp FROM account_balance
   WHERE account_id = $1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3
   ORDER BY consensus_timestamp DESC LIMIT 1
   ```
   with params `[treasuryAccountId, 0, 1]`. No treasury snapshot exists at nanosecond timestamps 0–1. `rows.length === 0`, so the function **returns `{}`** at line 191.

6. `getTsQuery()` destructures `{}` → `lower === undefined` → **returns `{}`** at line 281.

7. `getBalances()` evaluates `!tsQueryResult.query` (line 115) → `true` → **`return;`** at line 116. The response object was pre-initialized at lines 102–108 as `{timestamp: null, balances: [], links: {next: null}}` and is sent as HTTP 200.

**Root cause:** The function's contract (`{}` = "no snapshot found") is indistinguishable from a legitimate empty result at the HTTP layer. The caller at line 115–116 treats both "impossible range" and "no snapshot in range" identically: silent early return with empty payload.

**Failed assumption:** The code comment at line 267–269 states the algorithm assumes "the data in db is in sync with the network." When a user deliberately targets a timestamp outside any synced partition, this assumption is violated and the silent-return path is triggered by design — but with no client-visible signal.

### Impact Explanation
Any client consuming `GET /api/v1/balances?timestamp=lte:<old_ts>` receives HTTP 200 `{timestamp: null, balances: []}`. The client cannot distinguish this from a legitimate network state where no accounts hold balances. Monitoring systems, balance-verification scripts, or downstream services that treat an empty-but-200 response as authoritative will silently operate on stale or absent data. For the `/accounts/:id` path the impact is lower because a documented entity-table fallback fires (lines 460–464 of `accounts.js`), but the `/balances` bulk endpoint has no such fallback.

### Likelihood Explanation
No authentication or privilege is required. The attack is a single crafted HTTP GET request with a timestamp query parameter. It is trivially repeatable, requires no knowledge of internal state beyond "pick a timestamp before the first balance snapshot," and produces a deterministic silent-empty response. Any public-facing mirror node deployment is reachable.

### Recommendation
1. **Distinguish "no snapshot" from "empty result":** When `getAccountBalanceTimestampRange()` returns `{}` due to zero DB rows (line 190–192), `getTsQuery()` and `getBalances()` should return a 404 or a structured error (e.g., `{"_status": {"messages": [{"message": "No balance snapshot found for the requested timestamp"}]}}`), not a silent HTTP 200 with empty balances.
2. **Validate timestamp reachability before querying:** In `getOptimizedTimestampRange()`, if `upperBound` is older than the oldest known partition boundary, reject the request with a 400 before issuing the DB query.
3. **Add a `snapshot_found: false` field** to the response envelope as a minimum non-breaking change, so clients can detect the no-snapshot case without relying on HTTP status codes.

const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
```
