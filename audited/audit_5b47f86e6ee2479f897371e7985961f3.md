### Title
Negative Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` — Newly Deployed Contracts Permanently Invisible on Web3 API

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to suppress caching of absent results. However, when a contract does not exist in the database, Spring Data JPA returns `Optional.empty()` — a non-null object — so the `unless` guard evaluates to `false` and the empty result is cached. Any unprivileged caller can pre-poison this cache for an arbitrary contractId, causing the mirror node's web3 API to treat a subsequently deployed contract as non-existent for up to one hour (and indefinitely with repeated access).

### Finding Description
**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The `unless` SpEL expression tests `#result == null`. When no row exists in the `contract` table for the given `contractId`, Spring Data JPA wraps the absent result in `Optional.empty()`. `Optional.empty()` is a singleton non-null object, so `#result == null` is always `false` for this case, and the empty Optional is stored in the Caffeine cache under `CACHE_NAME_CONTRACT`.

**Cache configuration** (`CacheProperties.java`, line 22):
```
expireAfterAccess=1h, maximumSize=1000
```
Because the policy is `expireAfterAccess`, every read of the poisoned entry resets the 1-hour timer.

**Call chain from public API:**
`POST /api/v1/contracts/call` → `ContractController.call()` → `ContractExecutionService.processCall()` → EVM execution → `ContractBytecodeReadableKVState.readFromDataSource()` (line 43) → `contractRepository.findRuntimeBytecode(entityId.getId())`.

No authentication or authorization is required to reach this endpoint.

**Why existing checks fail:** The `unless` guard was intended to prevent null caching, but the method signature returns `Optional<byte[]>`, not `byte[]`. The guard must test `#result?.isEmpty()` or `!#result.isPresent()`, not `#result == null`. The existing test `findRuntimeBytecodeFailCall` confirms `Optional.empty()` is returned for missing contracts but does not assert caching behavior.

### Impact Explanation
Once an attacker pre-poisons the cache for a target contractId:
- All subsequent `eth_call` / `eth_estimateGas` requests to that contract address return as if no bytecode exists, regardless of whether the contract has since been deployed to the network and written to the mirror node database.
- The EVM treats the address as an EOA (no code), so any call that depends on the contract's logic silently returns empty data or reverts, producing incorrect results for dApps and tooling relying on the mirror node.
- With `expireAfterAccess=1h` and a cache size of 1000, an attacker can keep 1000 contract addresses poisoned indefinitely by periodically re-querying them, effectively blocking the web3 API from ever seeing those contracts' bytecode.
- No funds are directly at risk on the consensus layer, but smart contract behavior observed through the mirror node is incorrect, matching the stated Medium severity scope.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send an HTTP POST to `/api/v1/contracts/call` with a `to` address that maps to a not-yet-existing (or soon-to-be-deployed) contractId. The Hedera entity ID space is sequential and predictable, making it straightforward to target the next contractId before deployment. The attack is repeatable, automatable, and requires no on-chain transactions.

### Recommendation
Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures only successful (non-empty) bytecode lookups are cached, while absent-contract queries always hit the database and reflect the current state.

### Proof of Concept
1. Identify the next sequential Hedera contract entity ID (e.g., `0.0.5000`) before it is deployed.
2. Send a request to the mirror node web3 API targeting the corresponding long-zero EVM address (`0x0000000000000000000000000000000000001388`):
   ```
   POST /api/v1/contracts/call
   {"to":"0x0000000000000000000000000000000000001388","data":"0x","gas":50000}
   ```
3. The DB has no row for id=5000 → `findRuntimeBytecode(5000L)` returns `Optional.empty()` → cached.
4. Deploy a real contract on Hedera; it receives entity ID `0.0.5000`; the importer writes its bytecode to the `contract` table.
5. Repeat the same `eth_call` request. The cache returns the stored `Optional.empty()` without hitting the DB. The EVM sees no bytecode and the call behaves as if the contract does not exist.
6. Repeat step 5 every ~50 minutes to reset the `expireAfterAccess` timer, keeping the poisoned entry alive indefinitely.
