### Title
Cache Poisoning via `Arrays.hashCode()` Collision in `CACHE_NAME_ALIAS` Cache Key

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse()` uses `@spelHelper.hashCode(#alias)` — which delegates to `Arrays.hashCode(byte[])`, a 32-bit non-cryptographic hash — as the sole cache key for the `CACHE_NAME_ALIAS` Caffeine cache. Two distinct byte arrays that produce the same `Arrays.hashCode` value share a single cache slot. An unprivileged attacker can trivially construct such a collision, causing a lookup for alias B to return the cached entity for alias A, misrepresenting the current Hashgraph entity state.

### Finding Description
**Exact code path:**

- `SpelHelper.hashCode(byte[])` calls `Arrays.hashCode(value)` — a 32-bit polynomial hash. [1](#0-0) 

- `findByEvmAddressOrAliasAndDeletedIsFalse()` uses `key = "@spelHelper.hashCode(#alias)"` as the only cache key expression. [2](#0-1) 

- The `CACHE_NAME_ALIAS` cache is a Caffeine cache managed by `CACHE_MANAGER_ENTITY`; no secondary equality check on the actual byte array is performed on cache retrieval. [3](#0-2) 

**Root cause:** `Arrays.hashCode` produces a 32-bit integer. Spring's `@Cacheable` stores and retrieves entries using this integer as the map key verbatim. If two different byte arrays `aliasA` and `aliasB` satisfy `Arrays.hashCode(aliasA) == Arrays.hashCode(aliasB)`, the second lookup hits the first entry without any further comparison.

**Collision construction (analytical, zero-brute-force):**
`Arrays.hashCode` computes `h = 31*h + (b & 0xff)` per byte. For any `aliasA` of length ≥ 2, construct `aliasB` identical to `aliasA` except:
```
aliasB[n-2] = (aliasA[n-2] + 1) & 0xFF
aliasB[n-1] = (aliasA[n-1] - 31 + 256) & 0xFF
```
The net hash delta is `31² × (+1) + 31 × (−31) = 961 − 961 = 0`. This gives a guaranteed collision with a single arithmetic operation, no search required.

**Exploit flow:**
1. Attacker identifies a valid alias `aliasA` for a known entity A (e.g., from public ledger data or prior API calls).
2. Attacker sends a request with `aliasA` → DB is queried, entity A is stored in cache under key `k = Arrays.hashCode(aliasA)`.
3. Attacker computes `aliasB` using the formula above (deterministic, no privileges needed).
4. Attacker sends a request with `aliasB` → cache hit on key `k` → entity A is returned instead of the correct result for `aliasB`.

**Why existing checks fail:**
The `unless = "#result == null"` guard only prevents caching absent results; it does not prevent a collision from serving a wrong cached result. [4](#0-3) 

### Impact Explanation
A caller querying entity state by alias receives a completely different entity's data (balance, contract bytecode, key material, deleted status). In the Hashgraph context this directly misrepresents the current ledger state: smart contract calls, balance checks, and account resolution can all be fed stale or wrong entity data. The same flaw exists in `findByEvmAddressAndDeletedIsFalse()` via `CACHE_NAME_EVM_ADDRESS`. [5](#0-4) 

### Likelihood Explanation
No authentication or special privilege is required — only the ability to send HTTP requests to the web3 JSON-RPC endpoint. The collision is computed analytically in O(1); no brute force is needed. The attack is repeatable for the duration of the cache TTL and can be re-triggered after expiry. Any public-facing mirror-node deployment is exposed.

### Recommendation
Replace the 32-bit `Arrays.hashCode` key with a collision-resistant representation. The simplest correct fix is to use the hex-encoded byte array or `Arrays.toString(#alias)` as the cache key, which is unique per distinct byte array:

```java
key = "T(java.util.Arrays).toString(#alias)"
```

Alternatively, use a cryptographic digest (e.g., `SHA-256` hex string) via a dedicated `SpelHelper` method. The `hashCode` method in `SpelHelper` should either be removed or documented as unsafe for use as a cache key. [1](#0-0) 

### Proof of Concept
```
# Step 1 – Populate cache with aliasA (any known 20-byte EVM address, hex-encoded)
aliasA = 0x0102030405060708090a0b0c0d0e0f1011121314

# Step 2 – Compute aliasB analytically (Python)
a = bytearray.fromhex("0102030405060708090a0b0c0d0e0f1011121314")
b = bytearray(a)
b[-2] = (a[-2] + 1) & 0xFF   # byte 18: 0x13 -> 0x14
b[-1] = (a[-1] - 31 + 256) & 0xFF  # byte 19: 0x14 -> (0x14 - 0x1F + 256) = 0xF5
aliasB = b.hex()  # "0102030405060708090a0b0c0d0e0f101112 14f5"

# Verify collision
import java_arrays_hashcode  # pseudocode
assert hashcode(aliasA) == hashcode(aliasB)  # always true by construction

# Step 3 – Trigger cache population
GET /api/v1/contracts/{aliasA}   # or equivalent eth_call with aliasA

# Step 4 – Exploit: query aliasB, receive entity for aliasA
GET /api/v1/contracts/{aliasB}   # returns entity A — wrong result
```
