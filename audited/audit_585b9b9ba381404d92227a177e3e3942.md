### Title
Unauthenticated Alias Lookup Enables Database Resource Exhaustion via Repeated Invalid `ownerId` Queries

### Summary
`HookServiceImpl.getHookStorage()` unconditionally calls `entityService.lookup(request.getOwnerId())` for every request. When `ownerId` is supplied as an alias-formatted string, `EntityServiceImpl.lookup()` issues a `findByAlias()` database query with no rate limiting, caching, or authentication guard. An unprivileged attacker can flood the endpoint with non-existent aliases, forcing repeated DB queries that all throw `EntityNotFoundException`, degrading service for legitimate users.

### Finding Description
**Code path:**

- `HookServiceImpl.java`, `getHookStorage()`, line 61:
  ```java
  final var ownerId = entityService.lookup(request.getOwnerId());
  ```
  This is called unconditionally before any short-circuit or guard.

- `EntityServiceImpl.java`, `lookup()`, lines 31–37:
  ```java
  var id = switch (accountId) {
      case EntityIdNumParameter p -> Optional.of(p.id());
      case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
      case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
  };
  return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
  ```

- `EntityIdParameter.valueOf()` (lines 10–26) parses any base32/alias-formatted string as `EntityIdAliasParameter` — format validation only, no existence check.

**Root cause:** There is no rate limiting, no authentication requirement, no caching layer, and no short-circuit before the DB alias lookup. Every request with a syntactically valid but non-existent alias hits the database.

**Why existing checks fail:** `EntityIdParameter.valueOf()` only validates format (blank check, type dispatch). It does not validate existence. The `EntityNotFoundException` is thrown *after* the DB round-trip, not before.

### Impact Explanation
An attacker can continuously send HTTP requests with syntactically valid but non-existent alias values as `ownerId`. Each request causes one `findByAlias` DB query. At scale, this saturates DB connection pools and query throughput, increasing latency for legitimate users. The impact is service degradation (griefing), not data exfiltration. Severity is Medium, consistent with the scope classification.

### Likelihood Explanation
No authentication is required. The alias format is publicly documented (base32-encoded public key). An attacker needs only an HTTP client and a loop. The attack is trivially repeatable, requires no special knowledge, and leaves no persistent damage — making it a low-effort, sustained griefing vector.

### Recommendation
1. **Rate limiting:** Apply per-IP or per-client rate limiting at the API gateway or filter layer for endpoints that trigger alias lookups.
2. **Caching negative results:** Cache `EntityNotFoundException` outcomes for alias lookups (e.g., with a short TTL) to avoid repeated DB hits for the same non-existent alias.
3. **Authentication guard (if applicable):** If this endpoint is not intended to be fully public, require authentication before processing the `ownerId` lookup.
4. **Metrics/alerting:** Add monitoring on `EntityNotFoundException` rates from `lookup()` to detect abuse patterns early.

### Proof of Concept
```
# Attacker sends repeated requests with a syntactically valid but non-existent alias
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/hooks/<hookId>/storage?ownerId=AABBCCDDEE112233" &
done
```
Each request causes `EntityServiceImpl.lookup()` to call `entityRepository.findByAlias("AABBCCDDEE112233")`, find nothing, and throw `EntityNotFoundException`.
