### Title
Unauthenticated HTTP Mirror Node Response Allows Full Network Map Hijacking via MITM/DNS Poisoning

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` builds the entire Hiero node network map by blindly trusting a plaintext HTTP response from the configured mirror node REST endpoint. There is no TLS enforcement, no response authentication, and no validation of returned endpoints against any allowlist. An attacker who can intercept or redirect the HTTP traffic (via MITM or DNS poisoning) can inject a crafted `nodesEnvelope` response containing only attacker-controlled endpoints, causing every subsequent `CryptoTransfer` transaction to be submitted to rogue infrastructure.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–132.

The `http.Client` is constructed at line 46 with only a timeout — no TLS configuration, no certificate pinning, no transport-level security enforcement:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

The documented example value for `mirrorRest` (line 37 of `config.go`) is `http://mirror-rest:5551` — plaintext HTTP. The GET request at line 84 is sent over this unauthenticated channel.

The response body is decoded at lines 95–98 into a `nodesEnvelope` struct with zero verification:

```go
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }
```

The loop at lines 102–124 then iterates over `payload.Nodes` and for each entry with a non-empty `NodeAccountID` and non-empty host + non-zero port, unconditionally inserts the endpoint into the `network` map:

```go
addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
network[addr] = nodeAccountId
```

The only guard at lines 127–129 is that the resulting map must be non-empty — it does not verify that any returned address is a legitimate Hiero consensus node.

This `network` map is passed directly to `hiero.ClientForNetwork(netmap)` in `sdk_client.go` line 22, and the resulting client is used by `submitWithRetry()` in `transfer.go` line 33 to execute every `CryptoTransfer`.

**Root cause:** The application assumes the mirror node REST API is a trusted, integrity-protected source of truth for the consensus node topology. No mechanism exists to verify this assumption at the protocol level.

**Why existing checks fail:**
- The `NodeAccountID` format check (line 107) only validates that the string parses as a valid Hiero account ID — it does not verify the account ID belongs to a real consensus node.
- The non-empty host / non-zero port check (line 118) only filters structurally invalid entries — it accepts any IP address or domain name.
- The non-empty network map check (line 127) only ensures at least one entry was returned — a single attacker-controlled entry satisfies it.

### Impact Explanation

All `CryptoTransfer` transactions are routed exclusively to attacker-controlled gRPC endpoints. The attacker receives the operator's signed transaction bytes, including the operator private key's signature over the transaction. While the private key itself is not directly exposed, the attacker can:
- Silently drop all transactions (denial of service for the pinger's health-check function)
- Replay captured signed transactions against the real network
- Observe transaction metadata (operator account ID, destination account ID, amounts, timing)

Severity: **High**. The network map is built once at startup and never refreshed, so a single successful injection persists for the entire lifetime of the process.

### Likelihood Explanation

**Precondition:** The attacker must be able to intercept or redirect HTTP traffic between the pinger pod and the mirror node REST endpoint. This is achievable via:
1. **MITM within the same Kubernetes cluster/VPC** — any compromised pod or node in the same network segment can ARP-spoof or use a rogue DNS server to redirect `http://mirror-rest:5551` traffic. This requires no privileges on the pinger pod itself.
2. **DNS poisoning** — if the mirror node hostname resolves via an attacker-influenced DNS resolver (e.g., a compromised CoreDNS pod), the attacker can redirect the hostname to their own server.
3. **Rogue mirror node deployment** — if the operator misconfigures `HIERO_MIRROR_PINGER_REST` to point at an attacker-controlled URL (e.g., via environment variable injection in CI/CD), the attack is trivially achieved.

The use of plaintext HTTP as the documented and default example makes scenarios 1 and 2 realistic in typical Kubernetes deployments. No privileges on the pinger process are required.

### Recommendation

1. **Enforce HTTPS with certificate verification** for the mirror node REST URL. Reject `http://` scheme URLs at config validation time, or at minimum emit a loud warning.
2. **Pin or validate returned endpoints** against a configurable allowlist of known-good IP ranges or domain suffixes.
3. **Verify response integrity** — if the mirror node REST API supports response signing, verify the signature before consuming the payload.
4. **Refresh the network map periodically** with the same integrity checks, rather than trusting a single startup-time fetch forever.

In `config.go`, add scheme validation:
```go
if cfg.network == "other" && !strings.HasPrefix(cfg.mirrorRest, "https://") {
    return cfg, fmt.Errorf("mirror-rest must use https:// for security")
}
```

In `fetchMirrorNodeNetwork()`, validate that returned IP addresses fall within expected ranges before inserting them into the network map.

### Proof of Concept

**Preconditions:**
- Pinger is deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`
- Attacker has network-level access to intercept traffic to `mirror-rest:5551` (e.g., via DNS poisoning or ARP spoofing within the cluster)

**Steps:**

1. Attacker stands up a rogue HTTP server at an IP they control (e.g., `10.0.0.99:8080`) that serves the following JSON at `GET /api/v1/network/nodes`:

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "10.0.0.99", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

2. Attacker redirects DNS resolution of `mirror-rest` to `10.0.0.99`, or ARP-spoofs the mirror node's IP.

3. Pinger starts up, calls `buildNetworkFromMirrorNodes()` → `fetchMirrorNodeNetwork()`, receives the injected response.

4. `network` map is populated as `{"10.0.0.99:50211": AccountID{0,0,3}}`.

5. `hiero.ClientForNetwork(netmap)` creates a client pointing exclusively at `10.0.0.99:50211`.

6. Every `submitWithRetry()` call sends signed `CryptoTransfer` gRPC requests to the attacker's server at `10.0.0.99:50211`.

7. Attacker observes all transaction bytes, can replay them, or simply drop them to cause the pinger to report failures indefinitely.
