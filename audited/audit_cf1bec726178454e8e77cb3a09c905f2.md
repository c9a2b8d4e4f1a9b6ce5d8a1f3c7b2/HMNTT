### Title
Unbounded `req.method` Used Verbatim as OpenTelemetry Label Enables Metrics Store Memory Griefing

### Summary
In `metricsMiddleware()`, `req.method` is read directly from the Express request object and used without any length check or sanitization as the `method` label in the `{method, path, code}` attribute set recorded across four per-route instruments. Node.js's llhttp HTTP parser accepts non-standard method tokens up to the default 16 KB header-size limit, meaning a request with a ~10,000-character method name reaches the middleware intact. An unprivileged attacker can exploit this to inject arbitrarily large strings into the OpenTelemetry in-process metrics store.

### Finding Description
**Exact code path:**
`rest/middleware/metricsHandler.js`, function `metricsMiddleware`, lines 191–192 and 207–210:

```js
const method = req.method;          // line 191 – verbatim, no length check
const labels = {method, path, code}; // line 192

requestTotalCounter.add(1, labels);          // line 207
durationHistogram.record(duration, labels);  // line 208
requestSizeHistogram.record(..., labels);    // line 209
responseSizeHistogram.record(..., labels);   // line 210
```

**Root cause:** The code assumes `req.method` will always be a short, standard HTTP verb (GET, POST, etc.). No maximum-length guard, allowlist, or truncation is applied before the value is embedded in a label set.

**Why the HTTP layer does not block it:** Node.js uses the llhttp parser, which accepts any sequence of valid token characters as the method token. The only enforced limit is `--max-http-header-size` (default 16,384 bytes), which covers the entire request line. A 10,000-character method name plus a short path and HTTP version fits comfortably within that budget and is delivered to Express as `req.method` verbatim.

**Why existing checks are insufficient:** No rate-limiting middleware is applied to general API routes in `rest/server.js`. No cardinality limit is configured on the `MeterProvider` or individual instruments in `initMetrics()`. The `@opentelemetry/sdk-metrics` SDK does impose a default cardinality cap of 2,000 attribute sets per instrument, but:
- Four separate instruments are registered (`requestTotalCounter`, `durationHistogram`, `requestSizeHistogram`, `responseSizeHistogram`), each with its own 2,000-entry budget.
- Each entry stores the full label object, including the oversized method string.
- An attacker can vary the `path` dimension (by targeting different API endpoints) to fill all 2,000 slots per instrument with entries that each carry a 10,000-character method string.

**Exploit flow:**
1. Attacker crafts HTTP requests with a ~10,000-character custom method (e.g., `AAAA…AAA`).
2. llhttp parses and accepts the request; Express sets `req.method` to the full string.
3. `metricsMiddleware` fires on `res.finish`, constructs `labels = {method: "<10k chars>", path, code}`.
4. All four instruments record the label set, inserting it into the in-process OpenTelemetry attribute store.
5. By varying the path across available API routes, the attacker fills up to 2,000 × 4 = 8,000 label-set slots, each holding a 10,000-character method string.

### Impact Explanation
The in-process Node.js metrics store retains all unique label sets for the lifetime of the process. At the cardinality ceiling (2,000 per instrument × 4 instruments), the method strings alone account for roughly 80 MB of heap strings (8,000 × 10,000 bytes), not counting the surrounding OpenTelemetry attribute-map objects. This constitutes a bounded but meaningful heap inflation that can degrade garbage-collection performance, increase memory pressure, and—on memory-constrained deployments—trigger OOM restarts. No authentication or privilege is required. The impact is griefing/availability degradation with no direct economic damage to network users, consistent with the stated Medium scope.

### Likelihood Explanation
The attack requires only the ability to send HTTP requests to the public REST API, which is the intended use of the service. No credentials, tokens, or special network position are needed. The exploit is trivially scriptable: a single loop sending requests with a long method string to each known API path exhausts the cardinality budget in seconds. It is fully repeatable after a process restart.

### Recommendation
1. **Allowlist or truncate `req.method` before labeling.** Replace line 191 with a normalized value:
   ```js
   const KNOWN_METHODS = new Set(['GET','POST','PUT','PATCH','DELETE','HEAD','OPTIONS']);
   const method = KNOWN_METHODS.has(req.method) ? req.method : 'UNKNOWN';
   ```
2. **Alternatively, cap the string length** as a defense-in-depth measure:
   ```js
   const method = (req.method ?? '').slice(0, 16).toUpperCase();
   ```
3. **Configure an explicit cardinality limit** on the `MeterProvider` to reduce the blast radius of any future label-injection vectors.
4. **Apply rate limiting** to all API routes at the server or ingress level to slow down bulk griefing attempts.

### Proof of Concept
```bash
# Generate a 10,000-character method name
METHOD=$(python3 -c "print('A'*10000)")

# Send requests to several different API paths to fill cardinality slots
for PATH in /api/v1/accounts /api/v1/transactions /api/v1/balances /api/v1/tokens; do
  curl -s -o /dev/null -X "$METHOD" "http://<mirror-node-rest>$PATH"
done

# Repeat in a loop to exhaust all 2000 cardinality slots per instrument
for i in $(seq 1 500); do
  curl -s -o /dev/null -X "$METHOD" "http://<mirror-node-rest>/api/v1/accounts?account.id=0.0.$i"
done

# Observe heap growth via the metrics endpoint or process memory
curl http://<mirror-node-rest>/swagger/metrics/ | grep nodejs_process_memory_heap_used
```
After the loop, heap-used metrics will reflect the inflated label store. A process restart clears the state, but the attack is immediately repeatable.
