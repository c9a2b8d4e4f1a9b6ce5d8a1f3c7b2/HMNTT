### Title
Missing Application-Level Rate Limiting on `getByEvmAddressAndType()` Enables Database Connection Pool Exhaustion DoS

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` issues an uncached, unbounded database query on every invocation with no application-level throttling. The `web3` module has a full bucket4j-based `ThrottleManager`, but the `graphql` module contains no equivalent. An unauthenticated attacker flooding the `/graphql/alpha` endpoint with unique EVM address lookups can exhaust the HikariCP connection pool, denying service to all legitimate users.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 51-54) calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)` for any request supplying an `evmAddress` input field. [1](#0-0) 

`EntityServiceImpl.getByEvmAddressAndType()` (lines 34-41) decodes the address and unconditionally calls either `entityRepository.findById()` or `entityRepository.findByEvmAddress()` — a live SQL query every time, with no `@Cacheable`, no counter, no guard. [2](#0-1) 

The repository executes a native SQL query against the `entity` table on every call. [3](#0-2) 

**Root cause — no application-level throttle in the graphql module:**

A grep for `ThrottleManager`, `RateLimiter`, `throttle`, or `RateLimit` across all of `graphql/src/main/java/**` returns zero matches. The `web3` module has a complete bucket4j-based `ThrottleManagerImpl` wired into every contract-call path, but nothing analogous exists in the `graphql` module. [4](#0-3) 

**GraphQL-level protections are insufficient:**

`GraphQlConfiguration` applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`. These reject structurally abusive queries but impose no limit on the *rate* of simple, valid `account(input:{evmAddress:"0x..."})` queries. [5](#0-4) 

**Infrastructure-level protections are disabled by default and bypassable:**

The Traefik `inFlightReq` middleware (5 concurrent requests per source IP) is defined in `values.yaml` but is only rendered when `global.middleware: true`. That flag defaults to `false`. [6](#0-5) [7](#0-6) 

Even when enabled, the per-IP limit of 5 is trivially bypassed by distributing requests across many source IPs. The `retry: attempts: 3` middleware in the same chain further amplifies each attacker request threefold at the load-balancer layer. [8](#0-7) 

The GCP gateway `maxRatePerEndpoint: 250` is per-pod and requires `gateway.gcp.enabled: true`, which is also not the default path for all deployments. [9](#0-8) 

### Impact Explanation

Each concurrent request to `getByEvmAddressAndType()` holds a HikariCP database connection for the duration of the SQL query. With unique EVM addresses per request, no query result can be served from any cache. Once the pool is exhausted, all GraphQL queries — including those from legitimate users — block waiting for a connection, producing effective denial of service for the entire `/graphql/alpha` endpoint. The Prometheus alert `GraphQLHighDBConnections` fires only after 75% utilization for 5 minutes, meaning the service is already degraded before any automated response. [10](#0-9) 

### Likelihood Explanation

No authentication or API key is required to reach `/graphql/alpha`. The query payload is minimal (a single GraphQL `account` query with a 40-hex-character `evmAddress`). An attacker needs only a script and a modest number of source IPs (or a single IP if the Traefik middleware is not deployed) to sustain thousands of concurrent requests. The attack is fully repeatable and requires no special knowledge beyond the public GraphQL schema.

### Recommendation

1. **Add application-level rate limiting to the graphql module** — mirror the `ThrottleManagerImpl`/bucket4j pattern from `web3` and wire it into a servlet filter or Spring interceptor that applies to `/graphql/alpha` before any resolver executes.
2. **Add response caching** — annotate `getByEvmAddressAndType()` (and the other `EntityService` methods) with `@Cacheable` using a short TTL (e.g., 1–5 seconds), consistent with the caching already applied in the `web3` module.
3. **Enable and enforce the Traefik middleware by default** — set `global.middleware: true` in the default chart values, and add a global (not per-IP) `rateLimit` middleware in addition to `inFlightReq`.
4. **Remove or reconfigure the `retry` middleware** — retrying failed requests at the load balancer amplifies attacker traffic; it should not apply to rate-limited or overload responses.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or similar
# No credentials needed

ENDPOINT="https://<mirror-node-host>/graphql/alpha"

flood() {
  ADDR=$(printf '0x%040x' $RANDOM$RANDOM$RANDOM)   # unique address per request
  curl -s -X POST "$ENDPOINT" \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"$ADDR\\\"}){ id } }\"}"
}

export -f flood
export ENDPOINT

# Send 2000 concurrent requests with unique EVM addresses
seq 1 2000 | parallel -j 2000 flood
```

**Expected result:** HikariCP active connections spike to pool maximum; subsequent legitimate queries return errors or time out. The `GraphQLHighDBConnections` Prometheus alert fires after sustained saturation. The attacker pays no cost beyond network bandwidth.

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);

        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
