### Title
Unauthenticated Timestamp Parameter Triggers Unbounded `DISTINCT ON` Table Scan in Hook Storage Historical Query

### Summary
Any unauthenticated external user can append `?timestamp=gte:0` to the public hook storage endpoint, which unconditionally routes execution to `getHookStorageChange()`. That method issues a native `SELECT DISTINCT ON (key)` query against the time-partitioned `hook_storage_change` table with a timestamp range of `[0, Long.MAX_VALUE]`, forcing a full cross-partition scan before the `LIMIT` clause is applied. No rate limiting exists in the `rest-java` module, so repeated low-rate requests can saturate database CPU and connection slots, degrading ≥30% of processing nodes.

### Finding Description

**Exact code path:**

`HooksController.getHookStorage()` accepts `timestamps` as an unauthenticated `@RequestParam` with only `@Size(max = 2)` validation. [1](#0-0) 

`hookStorageChangeRequest()` calls `Bound.of(timestamps, ...)`, which produces a non-empty `Bound` whenever any timestamp value is supplied. [2](#0-1) 

`Bound.isEmpty()` returns `true` only when both `lower` and `upper` are `null`. A single `gte:0` parameter sets `lower`, making `isEmpty()` return `false`. [3](#0-2) 

`HookStorageRequest.isHistorical()` delegates directly to `!timestamp.isEmpty()`, so any non-empty timestamp unconditionally routes to the expensive path. [4](#0-3) 

`HookServiceImpl.getHookStorage()` branches immediately to `getHookStorageChange()` without any further guard. [5](#0-4) 

Inside `getHookStorageChange()`, when no explicit upper bound is provided, `adjustUpperBound()` returns `Long.MAX_VALUE`, and `getAdjustedLowerRangeValue()` returns `0` (or `1` for `gt:0`), producing a timestamp range spanning the entire table. [6](#0-5) 

The repository then executes a native `SELECT DISTINCT ON (key)` query over `hook_storage_change` with `consensus_timestamp BETWEEN 0 AND 9223372036854775807`. PostgreSQL must sort and scan **all** matching rows across every time partition before it can apply the `LIMIT`. [7](#0-6) 

The `hook_storage_change` table is partitioned by `consensus_timestamp`, meaning a full-range scan crosses all partitions. [8](#0-7) 

**Root cause:** The `isHistorical()` predicate treats any non-empty timestamp as a signal to use the historical code path, with no validation that the range is bounded or reasonable. The `DISTINCT ON` query's cost scales with the total number of historical storage-change rows for the hook, not with the `limit` parameter.

**Why existing checks fail:**
- `@Size(max = 2)` only limits the number of timestamp parameters, not their range.
- `@Max(MAX_LIMIT)` caps the number of *returned* rows but does not prevent the full-table scan that precedes the `LIMIT`.
- The `statementTimeout` of 10,000 ms (`hiero.mirror.restJava.db.statementTimeout`) provides partial protection but still allows up to 10 seconds of DB load per request. [9](#0-8) 
- The `rest-java` module has no rate limiting — the `ThrottleConfiguration` and `ThrottleManagerImpl` exist only in the `web3` module. [10](#0-9) 

### Impact Explanation
An attacker who repeatedly sends `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage?timestamp=gte:0` forces the database to perform unbounded cross-partition `DISTINCT ON` scans. Each request holds a DB connection for up to 10 seconds and consumes significant CPU for sorting. With a modest number of concurrent connections (e.g., 20–50), the database connection pool is exhausted and query latency for all other mirror-node services (importer, other REST endpoints) spikes, degrading ≥30% of processing capacity without requiring brute-force request rates.

### Likelihood Explanation
The endpoint is fully public — no credentials, API key, or on-chain identity is required. The exploit requires only knowledge of any valid `(ownerId, hookId)` pair, which is discoverable via the public `/api/v1/accounts/{ownerId}/hooks` listing endpoint. The trigger is a single extra query parameter. Any external actor with HTTP access can execute this repeatedly.

### Recommendation
1. **Require a bounded timestamp range**: Reject requests where the timestamp range exceeds a configurable maximum window (e.g., 1 hour or 1 day), returning HTTP 400.
2. **Add rate limiting to `rest-java`**: Implement per-IP or global request-rate limiting for the hook storage endpoint, analogous to the `ThrottleConfiguration` in the `web3` module.
3. **Add a DB-side query cost guard**: Use a tighter `statement_timeout` specifically for historical queries, or enforce a `LIMIT` on the intermediate scan via a CTE/subquery.
4. **Validate timestamp lower bound**: Reject `timestamp=gte:0` or any lower bound below a reasonable epoch (e.g., the network genesis timestamp).

### Proof of Concept
```
# Step 1: Discover a valid hook for any account
GET /api/v1/accounts/0.0.123/hooks
# → returns hook_id = 1

# Step 2: Trigger the unbounded DISTINCT ON scan (no auth required)
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0

# Step 3: Repeat concurrently to exhaust DB connections
for i in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0" &
done
wait

# Result: DB CPU spikes, connection pool exhausted, all mirror-node queries degrade
```

        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
