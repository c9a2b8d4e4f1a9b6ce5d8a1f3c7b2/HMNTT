### Title
Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` — Deployed Contract Bytecode Permanently Suppressed Until TTL Expiry

### Summary

`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`, but `Optional.empty()` is a non-null object and therefore passes the `unless` guard and is stored in the cache. Any request for a `contractId` that arrives before the contract row exists in the DB will cache `Optional.empty()`, and all subsequent requests for that `contractId` will be served the stale empty result until the cache entry expires — regardless of whether the contract has since been deployed and indexed.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's `@Cacheable` evaluates the `unless` SpEL expression against the method's return value. The method returns `Optional<byte[]>`. When the contract does not yet exist in the DB, the JPA query returns no row, and Spring Data wraps that as `Optional.empty()`. `Optional.empty()` is a non-null singleton object; therefore `#result == null` evaluates to `false`, the `unless` condition does NOT suppress caching, and `Optional.empty()` is written into the Caffeine cache under the key `contractId`.

**Failed assumption:** The developer assumed `unless = "#result == null"` would prevent negative (miss) results from being cached. This is only true when the method literally returns `null`. For `Optional`-returning methods, the empty case is `Optional.empty()`, not `null`.

**Exploit flow:**
1. Attacker (or any user) sends a `eth_getCode` / contract-call request for a known-future `contractId` (e.g., a predictable Hedera entity ID) before the contract is deployed and indexed.
2. `findRuntimeBytecode(contractId)` hits the DB, finds no row, returns `Optional.empty()`.
3. `Optional.empty() != null` → `unless` is false → `Optional.empty()` is stored in the Caffeine cache.
4. The contract is deployed; the importer writes the `runtime_bytecode` row to the DB.
5. All subsequent calls to `findRuntimeBytecode(contractId)` are served from cache — returning `Optional.empty()` — bypassing the DB entirely.
6. The caller (`ContractBytecodeReadableKVState.readFromDataSource()`) receives `Optional.empty()`, maps it to `null`, and the EVM sees no bytecode for the contract.

The cache is backed by Caffeine configured via `cacheProperties.getContract()` (set in `EvmConfiguration.cacheManagerContract()`). Until that TTL expires, the correct bytecode is never served.

### Impact Explanation

Any `eth_getCode` call or EVM execution targeting the affected `contractId` will receive an empty bytecode response, making the deployed contract appear non-existent. This breaks:
- `eth_getCode` — returns `0x` for a live contract.
- Contract calls routed through `ContractBytecodeReadableKVState` — the EVM cannot load the contract code, causing all calls to revert or return incorrect results.
- Mirror node data correctness — the web3 API serves data inconsistent with on-chain state for the full cache TTL duration.

Severity: **High** — complete denial of bytecode for a targeted contract, no privileges required, repeatable.

### Likelihood Explanation

No authentication or special privilege is required. The attacker only needs to know (or predict) the `contractId` before deployment — Hedera entity IDs are sequential and publicly observable. The window is the gap between when a `contractId` becomes predictable and when the contract row is committed to the DB. Even without deliberate attack, legitimate users racing to query a newly deployed contract can trigger this accidentally. The effect persists for the full cache TTL, which can be minutes.

### Recommendation

Change the `unless` condition to also exclude `Optional.empty()`:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures only successful (non-empty) `Optional` results are cached, and negative DB lookups always fall through to the DB on the next request.

### Proof of Concept

1. Identify the next sequential Hedera contract entity ID (e.g., `contractId = N`).
2. Before the contract is deployed, send: `eth_getCode` for the address corresponding to `N`.
3. The mirror node calls `findRuntimeBytecode(N)` → DB returns no row → `Optional.empty()` is cached.
4. Deploy the contract; wait for the importer to write `runtime_bytecode` for `N` to the DB.
5. Send `eth_getCode` again for the same address.
6. **Observed:** returns `0x` (empty bytecode) — served from cache.
7. **Expected:** returns the deployed contract's runtime bytecode.
8. Repeat step 5 until the Caffeine TTL expires — the correct bytecode is never returned during this window.
