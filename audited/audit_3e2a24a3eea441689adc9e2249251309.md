### Title
Unbounded Live Subscription Fan-Out via Unauthenticated `subscribeTopic` Allows Griefing

### Summary
`ConsensusController.subscribeTopic()` accepts an unlimited number of concurrent gRPC streaming subscriptions from any unauthenticated caller with no per-client or global cap enforced. Each subscription registers a live listener via `topicListener.listen()`, and for every new topic message the server must serialize and write to every open subscriber stream. The fan-out cost therefore scales linearly with the number of attacker-controlled subscriptions.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` and immediately delegates to `topicMessageService.subscribeTopic()` with no authentication, no per-IP limit, and no global subscription cap check. [1](#0-0) 

Inside `TopicMessageServiceImpl.subscribeTopic()`, a new `TopicContext` is created per call and `subscriberCount` is incremented — but this counter is only wired to a Micrometer `Gauge` metric and is **never used to reject or throttle new subscriptions**. [2](#0-1) 

For a live-only subscription (startTime ≈ now, no endTime), the historical retriever returns nothing and the code immediately enters `incomingMessages()`, which calls `topicListener.listen(newFilter)` — one registration per subscriber. [3](#0-2) 

Even when the underlying `TopicListener` implementation uses a shared/multicast upstream (e.g., `SharedPollingTopicListener`, `RedisTopicListener`), the **downstream fan-out is not shared**: each subscriber has its own Reactor subscription chain, its own `TopicContext`, and its own gRPC `ServerCallStreamObserver`. Every new topic message must be serialized via `toResponse()` and written to every open stream individually. [4](#0-3) 

**Root cause:** The failed assumption is that gRPC server-level connection limits (configured in `NettyProperties`) are sufficient to bound resource use. They limit concurrent calls *per connection*, but an attacker can open arbitrarily many connections from many source addresses or even from one host using many TCP connections, each carrying one subscription. [5](#0-4) 

### Impact Explanation

For each new message published to the targeted topic, the mirror node must:
- Dispatch the message to N Reactor subscriber chains
- Call `toResponse()` N times (protobuf serialization)
- Write to N gRPC response streams

CPU, heap (N `TopicContext` objects, N Reactor subscription objects), and outbound network bandwidth all grow linearly with N. At sufficient N, legitimate subscribers experience degraded throughput or the server becomes unresponsive. No economic cost is imposed on the attacker; only server resources are consumed.

### Likelihood Explanation

The gRPC endpoint is publicly reachable by design (it is the mirror node's public API). No credentials are required. A single attacker machine can open thousands of TCP connections and issue one `subscribeTopic` RPC per connection in a tight loop. The attack is trivially scriptable with any gRPC client library and is repeatable indefinitely.

### Recommendation

1. **Enforce a global subscriber cap**: Use `subscriberCount` (already tracked) to reject new subscriptions when a configurable maximum is reached, returning `RESOURCE_EXHAUSTED` status.
2. **Enforce a per-source-IP subscription limit**: Track active subscriptions by remote peer address and reject excess connections.
3. **Add a configurable `maxSubscribersPerTopic`** property in `GrpcProperties` and check it before allowing `topicListener.listen()` to be called.
4. **Configure `maxConcurrentCallsPerConnection`** in `NettyProperties` and pair it with a **maximum number of accepted connections** at the Netty level to bound total concurrent RPCs.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp
import time

TARGET = "mirror-node-grpc-host:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1234
NUM_SUBSCRIPTIONS = 5000

def open_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now = int(time.time())
    query = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=...,  # set target topic
        consensusStartTime=Timestamp(seconds=now, nanos=0),
        # no endTime → live-only, never completes
    )
    # Block on the stream (keeps subscription alive)
    for _ in stub.subscribeTopic(query):
        pass

threads = [threading.Thread(target=open_subscription) for _ in range(NUM_SUBSCRIPTIONS)]
for t in threads:
    t.start()
# All NUM_SUBSCRIPTIONS subscriptions are now live.
# Each new message published to the topic triggers N=5000 downstream writes.
```

    private Flux<TopicMessage> incomingMessages(TopicContext topicContext) {
        if (topicContext.isComplete()) {
            return Flux.empty();
        }

        TopicMessageFilter filter = topicContext.getFilter();
        TopicMessage last = topicContext.getLast();
        long limit =
                filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();

        return topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t));
    }
