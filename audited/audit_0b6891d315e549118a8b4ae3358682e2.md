### Title
Reference Identity Check Bypass in `findByBlockType()` Allows Crafted `BlockType` to Skip `findLatest()` and Return Empty

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect `BlockType.LATEST`, but `BlockType.of("9223372036854775807")` constructs a **new** `BlockType` instance with `number = Long.MAX_VALUE` that is not the `LATEST` singleton. This causes the method to call `findByIndex(Long.MAX_VALUE)` instead of `findLatest()`, returning `Optional.empty()` for any caller requesting the latest block via its numeric equivalent.

### Finding Description

**Exact code path:**

`BlockType.java` line 12 defines the singleton:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [1](#0-0) 

`BlockType.of()` with a decimal string falls into `extractNumericBlock()`:
```java
long blockNumber = Long.parseLong(cleanedValue, radix);
return new BlockType(value, blockNumber);  // new instance, not LATEST
``` [2](#0-1) 

`RecordFileServiceImpl.findByBlockType()` uses reference equality:
```java
} else if (block == BlockType.LATEST) {   // fails for crafted instance
    return recordFileRepository.findLatest();
}
return recordFileRepository.findByIndex(block.number()); // called with Long.MAX_VALUE
``` [3](#0-2) 

**Root cause:** The guard `block == BlockType.LATEST` is a reference identity check. `BlockType` is a Java `record` with fields `(String name, long number)`. A `BlockType` created via `BlockType.of("9223372036854775807")` has `name="9223372036854775807"` and `number=Long.MAX_VALUE` — a distinct object from the `LATEST` singleton. The check fails, and `findByIndex(Long.MAX_VALUE)` is executed.

**Why `findByIndex(Long.MAX_VALUE)` returns empty:** The repository query is `select r from RecordFile r where r.index = ?1`. No real record file will ever have `index = 9223372036854775807`, so the result is always `Optional.empty()`. [4](#0-3) 

### Impact Explanation
Any caller of `findByBlockType()` — including `ContractCallService` — that receives `Optional.empty()` when it expects the latest block will either throw an exception, return incorrect data, or silently fail. This means Ethereum-compatible JSON-RPC calls (e.g., `eth_call`, `eth_getBlockByNumber`) using the numeric equivalent of `"latest"` will not return the latest block state, causing incorrect or missing records exported to mirror node consumers. Severity: **Medium** — data integrity issue, no funds at risk, but causes incorrect protocol-level responses. [5](#0-4) 

### Likelihood Explanation
Any unprivileged external user can trigger this by passing the decimal string `"9223372036854775807"` as the block parameter in a JSON-RPC request. No authentication, special role, or prior knowledge of internal state is required. The `@JsonCreator` annotation on `BlockType.of()` means this path is directly reachable from deserialized HTTP request bodies. It is trivially repeatable. [6](#0-5) 

### Recommendation
Replace the reference equality check with a value-based check on the `number` field:

```java
// Instead of:
} else if (block == BlockType.LATEST) {

// Use:
} else if (block == BlockType.LATEST || block.number() == BlockType.LATEST.number()) {
```

Alternatively, add a dedicated method `isLatest()` to `BlockType` that checks `this == LATEST || this.number == Long.MAX_VALUE`, and use it in `findByBlockType()`. The same fix should be applied to the `EARLIEST` check for consistency (though `EARLIEST` has `number=0`, which is a valid real index, so the number-equality approach alone would be wrong there — the `EARLIEST` check should remain reference-based or use the name). [3](#0-2) 

### Proof of Concept
1. Start the mirror node web3 service.
2. Send an Ethereum JSON-RPC request with block parameter set to the decimal string `"9223372036854775807"`:
   ```json
   {"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["9223372036854775807", false],"id":1}
   ```
3. Internally, `BlockType.of("9223372036854775807")` is called, producing a new `BlockType` with `number=Long.MAX_VALUE` but `name="9223372036854775807"`.
4. `findByBlockType()` evaluates `block == BlockType.LATEST` → `false` (different object reference).
5. `recordFileRepository.findByIndex(9223372036854775807L)` is called → returns `Optional.empty()`.
6. The caller receives an empty/null response instead of the latest block, demonstrating the bypass.
