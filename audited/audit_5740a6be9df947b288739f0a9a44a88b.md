### Title
Unbounded Array Growth via Case-Variant Key Merging Bypasses `arrayLimit` Guard in `requestQueryParser`

### Summary
The `requestQueryParser()` function in `rest/middleware/requestHandler.js` uses `qs.parse` with `arrayLimit` and `throwOnLimitExceeded: true` to bound per-key arrays, but this protection only applies to bracket-notation arrays (e.g., `a[0]=1`), not to repeated parameters (e.g., `a=1&a=2`). Because `qs` treats each case variant of a key as a distinct key, an attacker can send many case variants each with many repeated values; the subsequent case-insensitive merge loop combines them all into a single unbounded array with no post-merge size check inside `requestQueryParser`.

### Finding Description

**Code location**: `rest/middleware/requestHandler.js`, `requestQueryParser()`, lines 15–20 and 38–69. [1](#0-0) 

The `queryOptions` object sets `arrayLimit: config.query.maxRepeatedQueryParameters` and `throwOnLimitExceeded: true`. In the `qs` library these options govern **bracket-notation** array indices (e.g., `a[21]=x` with `arrayLimit:20` throws). They do **not** limit the count of repeated plain parameters such as `timestamp=v1&timestamp=v2&…&timestamp=vN`. `qs` will happily build an array of arbitrary length for repeated plain params. [2](#0-1) 

After `qs.parse`, the loop at lines 56–66 lowercases every key and merges values from different case variants into one array: [3](#0-2) 

The inner `merge()` helper (lines 39–51) calls `current.push(...next)` with no size check: [4](#0-3) 

**Root cause**: `qs`'s `arrayLimit`/`throwOnLimitExceeded` guard is silently irrelevant for repeated plain parameters. Each case variant of a key (e.g., `timestamp`, `Timestamp`, `TIMESTAMP`, `tImEsTaMp`, …) is parsed by `qs` as a separate key, each accumulating its own unbounded array. The merge loop then concatenates all of them into `caseInsensitiveQueryString[lowerKey]` with no cap on the resulting length.

**Failed assumption**: The code assumes `throwOnLimitExceeded: true` prevents large arrays from reaching the merge loop. It does not for repeated plain parameters.

The only downstream check, `isRepeatedQueryParameterValidLength` in `buildFilters` (`utils.js` line 488), runs **after** `requestQueryParser` has already allocated the merged array in heap memory: [5](#0-4) [6](#0-5) 

### Impact Explanation
An attacker can cause the Node.js process to allocate a large heap array on every crafted request. For a key with L alphabetic characters there are up to 2^L case variants; for `timestamp` (9 alphabetic chars) that is 512 variants. With each variant carrying R repeated values, the merged array reaches 512 × R elements before any limit check fires. Under a 1 MB URL budget (achievable with some reverse-proxy configurations) and short values (~12 bytes each), R ≈ 160 per variant, yielding ~80,000 elements per request. Sustained at even modest request rates this exhausts heap and crashes the process — a non-network-based DoS requiring no authentication.

### Likelihood Explanation
No credentials or special privileges are required. The endpoint is publicly reachable. The exploit is trivially scriptable: a single `curl` or HTTP client loop sending crafted query strings suffices. The attack is repeatable and stateless. The only practical constraint is the server's maximum URL/query-string length, which varies by deployment (Node.js itself imposes no hard limit; reverse proxies like nginx default to 8 KB but are commonly raised).

### Recommendation
Apply a merged-array size cap **inside `requestQueryParser`**, immediately after the merge, before returning `caseInsensitiveQueryString`:

```js
if (Array.isArray(caseInsensitiveQueryString[lowerKey]) &&
    caseInsensitiveQueryString[lowerKey].length > config.query.maxRepeatedQueryParameters) {
  throw new InvalidArgumentError(...);
}
```

Alternatively, count total values across all case variants of a key before merging and reject the request early. The post-hoc check in `buildFilters` should remain as a defence-in-depth layer but cannot substitute for an in-parser guard.

### Proof of Concept

Generate all 512 case variants of `timestamp` (9 alphabetic characters), each repeated `R` times:

```python
import itertools, requests

key = "timestamp"
chars = [(c.lower(), c.upper()) for c in key]
variants = [''.join(v) for v in itertools.product(*chars)]  # 512 variants

# Build query string: each variant repeated R=20 times
params = []
for v in variants:
    for _ in range(20):
        params.append(f"{v}=1234567890.000000000")

qs = "&".join(params)
url = f"http://target/api/v1/transactions?{qs}"
r = requests.get(url)
print(r.status_code)
```

With 512 variants × 20 repetitions = 10,240 elements, the merged array in `caseInsensitiveQueryString['timestamp']` holds 10,240 entries before any limit check. Increasing R or using longer keys amplifies the effect. Sending this request in a tight loop will drive heap usage upward until the process OOMs or becomes unresponsive.
