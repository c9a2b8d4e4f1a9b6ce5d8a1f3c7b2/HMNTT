### Title
Global Token Bucket Exhaustion via Unauthenticated Burst — Single Client Can Starve All Others of the Shared Rate Limit

### Summary
`ThrottleManagerImpl.throttle()` enforces a single, process-wide token bucket (`rateLimitBucket`) shared across every HTTP client with no per-IP or per-identity partitioning. Because the bucket is configured with `refillGreedy` and `capacity = requestsPerSecond` (default 500), it starts full and refills continuously; any unauthenticated client can fire 500 concurrent requests, drain the entire bucket in one burst, and cause every other client to receive HTTP 429 for up to one full second. The attack is trivially repeatable every ~1 second.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 25–32 create the shared bucket:
```java
long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
final var limit = Bandwidth.builder()
        .capacity(rateLimit)
        .refillGreedy(rateLimit, Duration.ofSeconds(1))
        .build();
return Bucket.builder().addLimit(limit).build();
``` [1](#0-0) 

`ThrottleManagerImpl.java` line 38 deducts one token per request from this single global bucket:
```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
``` [2](#0-1) 

**Root cause:** The bucket is a singleton Spring bean injected into `ThrottleManagerImpl`. There is no per-client (IP, session, API key) bucket. `refillGreedy(rateLimit, Duration.ofSeconds(1))` with `capacity(rateLimit)` means the bucket initialises full and refills at `rateLimit` tokens/second continuously — so after any full drain, the bucket is completely replenished after exactly 1 second, enabling the burst to be repeated indefinitely.

**Failed assumption:** The design assumes the 500 req/s budget will be shared organically across many clients. In reality, a single client can atomically consume all 500 tokens before any other client's request is processed.

**Why existing checks are insufficient:**
- The optional `RequestProperties` filters (LOG / REJECT / THROTTLE) operate on EVM-level fields (`from`, `to`, `data`, `gas`) supplied in the request body — they are not tied to the HTTP client's network identity and can be trivially spoofed or left unconfigured. [3](#0-2) 
- There is no IP-based, session-based, or API-key-based rate limiting anywhere in the web3 application layer. [4](#0-3) 

### Impact Explanation
An attacker can monopolise the entire `requestsPerSecond` quota every second, causing all legitimate clients to receive HTTP 429 (`ThrottleException`) for the duration of each burst cycle. Because `ContractController.call()` invokes `throttleManager.throttle(request)` before any processing, no legitimate `eth_call` or `eth_estimateGas` request can succeed while the bucket is empty. [5](#0-4) 
This is a complete, repeatable denial-of-service of the web3 API endpoint for all other users, achievable with zero privileges.

### Likelihood Explanation
The attack requires only the ability to send HTTP POST requests to `/api/v1/contracts/call` — no authentication, no special headers, no privileged access. A single machine with a modest HTTP client library can open 500 parallel connections and fire them simultaneously. The attack is repeatable every ~1 second and is trivially scriptable. Any motivated actor (competitor, griever, automated bot) can sustain it indefinitely.

### Recommendation
Replace the single global bucket with per-client buckets keyed on the client's IP address (or API key if authentication is added). Bucket4j supports this via `BucketConfiguration` + a `ConcurrentHashMap<String, Bucket>` or its distributed `ProxyManager` abstraction. Additionally, consider using `refillIntervally` instead of `refillGreedy` to prevent the full-capacity burst that occurs after each refill cycle, or set `initialTokens(0)` to prevent the startup burst.

### Proof of Concept
```bash
# Drain the entire 500-token bucket in one burst (default requestsPerSecond = 500)
seq 1 500 | xargs -P500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000}'

# Immediately after, any legitimate client receives 429:
curl -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000}'
# → HTTP 429: "Requests per second rate limit exceeded"

# Repeat every ~1 second to sustain the DoS indefinitely.
```
