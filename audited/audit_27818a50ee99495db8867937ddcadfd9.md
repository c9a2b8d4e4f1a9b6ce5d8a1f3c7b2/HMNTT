### Title
Cache Pollution via Shared `exchangeRatesCacheManager` Across Three File Types Allows Unprivileged Cache Eviction

### Summary
The `load(FileID, long)` method in `SystemFileLoader` routes all three file types (`exchangeRateFileId`, `feeSchedulesFileId`, `simpleFeeSchedulesFileId`) into a single shared `exchangeRatesCacheManager` whose default `maximumSize` is only 20 entries. Because the cache key is `(fileId, roundedHourTimestamp)`, an unprivileged caller who can supply arbitrary historical block timestamps can generate 20+ distinct cache keys across the three file types, filling the cache and evicting legitimate entries, forcing all subsequent legitimate requests to incur database round-trips.

### Finding Description
**Exact code path:**

In `web3/src/main/java/org/hiero/mirror/web3/state/SystemFileLoader.java` lines 111–136:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;          // shared cache
    consensusTimestamp = roundDownToHour(consensusTimestamp);
}

final var cacheKey = new CacheKey(fileId, consensusTimestamp);
...
cache.put(cacheKey, result);
```

The `exchangeRatesCacheManager` is configured via `cacheProperties.getFee()` in `EvmConfiguration.java` (line 149), which resolves to the default in `CacheProperties.java` (line 36):

```
fee = "expireAfterWrite=60m,maximumSize=20,recordStats"
```

**Root cause:** All three distinct `FileID` values share one Caffeine cache with a hard cap of 20 entries. Each unique `(fileId, hourAlignedTimestamp)` pair occupies one slot. Normal operation uses ~3 slots (one per file type for the current hour). An attacker who controls the `consensusTimestamp` argument (via historical block number in a JSON-RPC request such as `eth_call` at block N) can generate arbitrarily many distinct hour-aligned timestamps.

**Exploit flow:**
1. Attacker sends 20 `eth_call` (or `eth_estimateGas`) requests, each targeting a different historical block number that maps to a different hour boundary, for `simpleFeeSchedulesFileId`.
2. Each miss populates a new `CacheKey(simpleFeeSchedulesFileId, hour_N)` entry.
3. After 20 entries, Caffeine begins evicting existing entries (LRU/W-TinyLFU policy).
4. The legitimate current-hour entries for `exchangeRateFileId` and `feeSchedulesFileId` are evicted.
5. All subsequent legitimate requests for exchange rates or fee schedules miss the cache and hit the database.
6. Attacker repeats continuously to sustain the eviction pressure.

**Why existing checks fail:** There is no per-file-type sub-cache, no cap on how many distinct timestamps a single file type may occupy, and no rate-limiting on cache population within `load()`. The `roundDownToHour` granularity reduces the key space but does not bound it — the Hedera ledger has years of history, providing thousands of distinct hour buckets.

### Impact Explanation
Legitimate users experience cache misses on every request for exchange rates and fee schedules, causing every EVM execution that needs these files to issue a synchronous database query instead of a sub-microsecond cache hit. Under sustained attack this increases database load proportionally to request volume, potentially degrading response latency for all users of the web3 API. There is no data corruption or economic loss; the impact is purely a griefing/availability degradation classified as medium severity.

### Likelihood Explanation
Any unauthenticated caller of the public JSON-RPC endpoint can specify a historical block number. No special privileges, tokens, or on-chain state are required. The attack requires only ~20 HTTP requests to saturate the cache and must be repeated every 60 minutes (the `expireAfterWrite` TTL) to sustain the effect. This is trivially automatable with a simple script.

### Recommendation
1. **Partition the cache by file type** — use separate `CacheKey` namespaces or separate `Cache` instances for each of the three file IDs so one file type cannot evict entries belonging to another.
2. **Bound per-file-type entries** — enforce a maximum of N entries per `FileID` within the shared cache (e.g., using a Caffeine `weigher` or per-key eviction listener).
3. **Increase `maximumSize`** — raising it from 20 to at least `3 × expected_concurrent_hours` (e.g., 72 for a 24-hour historical window) reduces but does not eliminate the risk.
4. **Rate-limit historical timestamp ranges** — reject or throttle requests whose block timestamp is older than a configurable threshold (e.g., 24 hours) at the API layer before `load()` is called.
