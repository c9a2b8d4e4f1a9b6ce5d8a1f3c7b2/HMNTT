### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `getAirdrops()` Requests in rest-java

### Summary
The `getAirdrops()` method in `TokenAirdropServiceImpl` issues a synchronous, blocking database query for every incoming HTTP request with zero application-level rate limiting or concurrency control. Because the rest-java module contains no throttling infrastructure (unlike the `web3` module), an unprivileged attacker can open many concurrent connections and flood the endpoint with requests targeting distinct `accountId` values, holding HikariCP connections for up to the full 10-second statement timeout and starving all other operations of database access.

### Finding Description

**Exact code path:**

`TokenAirdropsController` exposes two unauthenticated GET endpoints with no access control: [1](#0-0) 

Each request flows directly to `TokenAirdropServiceImpl.getAirdrops()`: [2](#0-1) 

Which calls `TokenAirdropRepositoryCustomImpl.findAll()`, executing a blocking jOOQ query that holds a HikariCP connection for the full query duration: [3](#0-2) 

**Root cause â€” no throttling in rest-java:**

The entire throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) lives exclusively in the `web3` module and is never wired into rest-java: [4](#0-3) 

A search of all `restjava` source files confirms zero `ThrottleManager`, `RateLimiter`, or equivalent beans. The `WebMvcConfiguration` and `RestJavaConfiguration` register only argument resolvers, formatters, an ETag filter, and a Protobuf converter â€” no rate-limiting filter: [5](#0-4) 

**DB connection pool parameters:**

The documented default statement timeout for rest-java is 10,000 ms (10 seconds), and no explicit `maximumPoolSize` override is documented, leaving HikariCP at its default of 10 connections: [6](#0-5) 

The HikariCP data source is constructed from `spring.datasource.hikari` properties with no rest-java-specific pool-size cap: [7](#0-6) 

**Why the only existing guard is insufficient:**

The Helm chart configures a GCP gateway-level `maxRatePerEndpoint: 250` (requests/second globally, not per-IP): [8](#0-7) 

250 RPS Ă— 10-second statement timeout = up to 2,500 in-flight DB connections requested against a 10-connection pool. This guard is also deployment-specific (GCP only), not enforced at the application layer, and not per-source-IP.

### Impact Explanation

With a pool of 10 connections and a 10-second statement timeout, an attacker needs only 10 concurrent slow-running queries to fully saturate the pool. All other rest-java endpoints (account lookups, NFT queries, etc.) that require a DB connection will block waiting for a free slot, causing cascading latency spikes and timeouts across the entire service. This constitutes a resource-consumption increase well above 30% achievable by a single unprivileged client with no special access.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no brute force. Any internet-accessible deployment is vulnerable. The attacker needs only a standard HTTP client capable of issuing concurrent requests (e.g., `curl`, `ab`, `wrk`, Python `asyncio`). Targeting distinct `accountId` values (e.g., sequential numeric IDs) prevents any query-result caching from mitigating the load. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add application-level rate limiting to rest-java**: Introduce a `RateLimitFilter` or Spring `HandlerInterceptor` in the rest-java module (mirroring the `ThrottleManagerImpl` pattern in `web3`) that enforces a global requests-per-second cap and, critically, a per-source-IP concurrent-request limit.
2. **Reduce statement timeout**: Lower `hiero.mirror.restJava.db.statementTimeout` to 2,000â€“3,000 ms to bound the maximum connection-hold time.
3. **Increase pool size or add a semaphore**: Either increase `spring.datasource.hikari.maximumPoolSize` to match expected concurrency, or add a `Semaphore` guard in `TokenAirdropServiceImpl` to cap the number of simultaneous in-flight DB queries.
4. **Enforce per-IP limits at the ingress layer**: Regardless of application-level fixes, configure per-source-IP rate limits at the load balancer/ingress (not just a global endpoint rate).

### Proof of Concept

```bash
# Saturate the 10-connection HikariCP pool with 15 concurrent requests
# targeting distinct accountIds to prevent caching

for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.$((1000 + i))/airdrops/outstanding?limit=100" &
done
wait

# Expected result: requests beyond pool size (>10) block or time out;
# other endpoints (e.g., /api/v1/accounts/0.0.1/tokens) also hang
# waiting for a free DB connection, demonstrating pool starvation.
```

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
