### Title
Unauthenticated EVM Address DB Lookup Exhausts Connection Pool via `getNftsByAccountId`

### Summary
`AccountController.getNftsByAccountId` resolves every `:idOrAliasOrEvmAddress` path parameter that is a non-long-zero EVM address by issuing an uncached SQL query against the `entity` table. No rate limiting exists in the REST API layer. An attacker sending thousands of concurrent requests with distinct valid EVM addresses can saturate the default 10-connection database pool, starving all other queries — including those serving gossip transaction records.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 15 routes `GET /:id/nfts` to `AccountController.getNftsByAccountId`. [1](#0-0) 

`AccountController.getNftsByAccountId` (line 91) calls `EntityService.getEncodedId` with the raw path parameter on every request. [2](#0-1) 

`EntityService.getEncodedId` (lines 120–124): when the input is a valid EVM address, `EntityId.parseString` returns an `EntityId` with `evmAddress !== null`, so execution always falls into `getEntityIdFromEvmAddress`. [3](#0-2) 

`getEntityIdFromEvmAddress` (lines 90–91) executes a raw SQL query against the `entity` table on every invocation — **no result caching**. [4](#0-3) 

The static query used: [5](#0-4) 

**Why the entityId parse cache does not help:**
`parseCached` in `entityId.js` (lines 301–332) caches only the parsed `EntityId` object (shard/realm/num/evmAddress fields). It does **not** cache the DB lookup result. After parsing, `getEncodedId` still unconditionally calls `getEntityIdFromEvmAddress`, which hits the database. [6](#0-5) 

**No rate limiting in the REST API:**
A grep across `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns only a test utility file. The `authHandler` middleware only sets per-user response-size limits; it does not throttle request rates. The `ThrottleManagerImpl` with bucket4j exists only in the `web3` module. [7](#0-6) 

**Response cache disabled by default:**
`cache.response.enabled` defaults to `false`. Even when enabled, the cache key is the full URL (including the EVM address), so each unique address is a distinct cache miss. [8](#0-7) 

**DB pool default is 10 connections:** [9](#0-8) [10](#0-9) 

### Impact Explanation
With a pool of 10 connections and a `statementTimeout` of 20 seconds, an attacker sustaining ~10 concurrent requests with distinct EVM addresses holds all pool slots for up to 20 seconds each. Legitimate API consumers — including the gossip-transaction-record polling path — receive connection-timeout errors (`connectionTimeoutMillis: 20000`). This constitutes a full denial-of-service against the mirror node REST API with no authentication required.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no on-chain state. Any 20-byte hex string that passes `EntityId.isValidEntityId` is sufficient; the attacker does not need the addresses to resolve to real entities (a 404 is returned after the DB query completes). The attack is trivially scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, or a simple loop with `curl`). It is repeatable indefinitely.

### Recommendation
1. **Cache DB lookup results for EVM addresses** in `getEntityIdFromEvmAddress`: use the existing `quickLru` infrastructure (already configured via `hiero.mirror.rest.cache.entityId`) to store `evmAddress → encodedId` mappings, including negative results (null), with the configured TTL.
2. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied before the route handlers.
3. **Increase the default DB pool size** or add a queue-depth limit so pool exhaustion does not cascade to a hard timeout for all callers.
4. **Enable the Redis response cache** by default for read-only endpoints like `/nfts`, which would serve repeated identical URLs from cache without touching the DB.

### Proof of Concept
```bash
# Generate 500 unique random EVM addresses and flood the endpoint
for i in $(seq 1 500); do
  ADDR=$(openssl rand -hex 20)
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/${ADDR}/nfts" &
done
wait

# Simultaneously, observe that legitimate requests time out:
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: connection timeout or 503 due to exhausted DB pool
```
