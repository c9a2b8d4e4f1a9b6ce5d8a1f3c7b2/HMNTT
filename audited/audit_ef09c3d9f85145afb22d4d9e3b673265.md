### Title
Unauthenticated Burst Requests to `/network/status` Trigger Uncached `string_agg` SQL Aggregation, Degrading Shared DB Performance

### Summary
Every call to `NetworkStatus()` unconditionally invokes `addressBookEntryRepo.Entries(ctx)`, which executes the `latestNodeServiceEndpoints` SQL query containing a `string_agg` aggregation over all service endpoints with no application-level caching or rate limiting. An unprivileged external user can flood `/network/status` with concurrent requests, causing the PostgreSQL database to repeatedly perform expensive aggregation operations and exhausting the shared connection pool, degrading query performance for all other DB consumers.

### Finding Description
**Exact code path:**

`rosetta/app/services/network_service.go`, `NetworkStatus()`, line 77:
```go
peers, err := n.addressBookEntryRepo.Entries(ctx)
``` [1](#0-0) 

This unconditionally calls `Entries()` in `rosetta/app/persistence/address_book_entry.go`, lines 57–83, which executes the raw SQL query `latestNodeServiceEndpoints` on every invocation with zero caching: [2](#0-1) 

The SQL query itself performs a `string_agg` aggregation with an `ORDER BY` clause over all rows in `address_book_service_endpoint` joined against `address_book_entry` and `node`: [3](#0-2) 

**Root cause:** The `addressBookEntryRepository` struct holds no cache, no TTL, and no deduplication of in-flight queries. The address book data is essentially static (changes only on network upgrades), yet it is re-fetched from the DB on every single HTTP request.

**Why existing checks fail:**

The only rate-limiting controls are Traefik middleware defined in `charts/hedera-mirror-rosetta/values.yaml` (lines 149–166), but they are gated on `global.middleware: true`, which defaults to `false`: [4](#0-3) [5](#0-4) 

The middleware template only deploys when `global.middleware` is true: [6](#0-5) 

Even when the middleware IS enabled, `rateLimit.average: 10` is keyed on `requestHost` (not source IP), so a single attacker can still issue 10 req/s continuously. The `retry: attempts: 3` entry in the same chain further amplifies DB load by replaying failed requests. There is no application-layer concurrency limiter or result cache anywhere in the rosetta Go codebase for this path.

The DB connection pool defaults to `maxOpenConnections: 100` with a `statementTimeout` of 20 seconds, meaning up to 100 concurrent aggregation queries can each hold a connection for 20 seconds before timing out.

### Impact Explanation
A burst of concurrent POST requests to `/network/status` causes PostgreSQL to execute multiple simultaneous `string_agg` + multi-table join queries. These queries compete for I/O, CPU, and connection slots with all other mirror node DB consumers (importer, REST API, gRPC). Connection pool exhaustion (100 slots) causes queued or rejected queries for legitimate users. The address book data is static between network upgrades, making repeated full aggregation entirely unnecessary. Impact is classified as griefing / availability degradation with no economic damage to any individual user, consistent with the medium severity scope.

### Likelihood Explanation
No authentication, API key, or session token is required to call `/network/status` — it is a fully public Rosetta API endpoint. The request body is minimal (`{"network_identifier": {...}}`). Any attacker with network access to the rosetta port (default 5700) can issue a burst using standard tools (`curl`, `ab`, `wrk`). The default deployment has `global.middleware: false`, so no Traefik rate limiting is active. This is trivially repeatable and requires no special knowledge beyond the Rosetta API specification.

### Recommendation
1. **Add an in-process cache** in `addressBookEntryRepository.Entries()` (e.g., a TTL-based sync.Map or `golang.org/x/sync/singleflight` to coalesce concurrent identical queries). The address book changes only on network upgrades, so a TTL of minutes is safe.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) or move rate limiting into the application layer so it cannot be bypassed by direct pod access.
3. **Change `rateLimit.sourceCriterion`** from `requestHost` to `ipStrategy` so limits are per source IP rather than shared across all callers.
4. **Remove `retry` from the middleware chain** for this endpoint, or scope it only to construction/submission endpoints, to avoid amplifying DB load on transient errors.

func (aber *addressBookEntryRepository) Entries(ctx context.Context) (*types.AddressBookEntries, *rTypes.Error) {
	db, cancel := aber.dbClient.GetDbWithContext(ctx)
	defer cancel()

	nodes := make([]nodeServiceEndpoint, 0)
	// address book file 101 has service endpoints for nodes, resort to file 102 if 101 doesn't exist
	for _, fileId := range []int64{aber.addressBook101.EncodedId, aber.addressBook102.EncodedId} {
		if err := db.Raw(
			latestNodeServiceEndpoints,
			sql.Named("file_id", fileId),
		).Scan(&nodes).Error; err != nil {
			log.Error("Failed to get latest node service endpoints", err)
			return nil, errors.ErrDatabaseError
		}

		if len(nodes) != 0 {
			break
		}
	}
