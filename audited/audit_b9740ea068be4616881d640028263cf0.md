### Title
Unauthenticated Gas Estimation Endpoint Allows 21x Resource Amplification via Binary Search Loop

### Summary
The `/api/v1/contracts/call` endpoint accepts unauthenticated requests with `estimate=true`. The throttle gate consumes tokens based on the declared `gas` value exactly once per HTTP request, but the binary search gas estimator performs up to 21 full EVM executions (1 initial probe + up to 20 binary search iterations) per request. An attacker can therefore consume 21x more EVM execution resources than the throttle accounts for, starving legitimate users of server capacity.

### Finding Description

**Throttle gate — consumed once, based on declared gas:**

In `ContractController.java` line 40, `throttleManager.throttle(request)` is called once before any EVM work begins. [1](#0-0) 

`ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas())` tokens from the gas bucket — a single deduction for the declared gas limit. [2](#0-1) 

**Binary search — up to 21 EVM executions per request:**

When `params.isEstimate()` is `true`, `ContractExecutionService.estimateGas()` first calls `callContract()` once (line 82), then invokes `binaryGasEstimator.search()` which loops up to `maxGasEstimateRetriesCount` (default **20**) additional times, each iteration performing a full EVM execution via `doProcessCall`. [3](#0-2) [4](#0-3) 

**Gas is never restored during binary search iterations:**

In `ContractCallService.doProcessCall()`, `restoreGasToBucket` is guarded by `if (!estimate)`. Because all binary search calls pass `estimate=true`, no gas tokens are restored to the bucket during the loop — but crucially, no additional tokens are consumed either. The throttle was charged once for the whole multi-execution operation. [5](#0-4) 

**Default gas limit is 15,000,000:**

`ContractCallRequest.gas` defaults to `15_000_000L` and the maximum allowed is also `15_000_000L` (enforced by `validateContractMaxGasLimit`). [6](#0-5) [7](#0-6) 

**Root cause:** The throttle model assumes one EVM execution per request. For `estimate=true` requests the actual execution count is `1 + min(iterations_to_converge, maxGasEstimateRetriesCount)` — up to **21**. The throttle never accounts for this multiplier.

### Impact Explanation
A single attacker IP can send requests at the rate the throttle permits (up to 500 req/s, limited by `requestsPerSecond`) and cause the server to perform up to **10,500 full EVM executions per second** instead of the intended 500. Each EVM execution with 15 M gas is a heavy CPU/memory operation. This exhausts thread-pool and CPU resources, causing severe latency or request failures for all other users. No funds are at risk, but service availability is degraded — consistent with the "griefing" severity classification. [8](#0-7) 

### Likelihood Explanation
- **No authentication required** — the endpoint is public.
- **No special knowledge needed** — `estimate=true` is a standard JSON field documented in the API.
- **Trivially repeatable** — a simple `curl` loop or any HTTP load tool suffices.
- **Amplification is deterministic** — with `gas=15000000` and a contract that uses close to the minimum gas, the binary search will always run the full 20 iterations (wide lo/hi spread), guaranteeing maximum amplification. [9](#0-8) 

### Recommendation
1. **Charge the throttle proportionally to the maximum possible iterations.** Before entering the binary search, consume `maxGasEstimateRetriesCount × scaleGas(gas)` tokens (or a configurable multiplier) from the gas bucket, and restore unused tokens after the search completes.
2. **Alternatively, track actual total gas consumed** across all binary search iterations and deduct it from the bucket at the end of `estimateGas()`.
3. **Add a separate, lower rate limit for `estimate=true` requests** (e.g., a dedicated bucket with a much smaller `requestsPerSecond` for estimation calls).
4. **Consider requiring API-key or IP-based authentication** for the estimation endpoint to raise the bar for abuse.

### Proof of Concept
```bash
# Attacker sends maximum-gas estimate requests in a tight loop
# No credentials needed; endpoint is public

while true; do
  curl -s -X POST https://<mirror-node-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to":   "0x0000000000000000000000000000000000000001",
      "data": "0x",
      "gas":  15000000,
      "estimate": true
    }' &
done
```

Each request triggers:
1. One initial EVM call with 15,000,000 gas.
2. Up to 20 binary-search EVM calls (confirmed by `BinaryGasEstimatorTest` case `"35000, 15_000_000, 14"` and `"21000, 15_000_000, 14"`).

The throttle deducts tokens for **one** 15 M-gas execution; the server performs **up to 21**. Running this from multiple IPs multiplies the effect linearly. Legitimate `eth_call` requests begin timing out or receiving 429 errors as the thread pool saturates. [10](#0-9)

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
