### Title
Global Rate-Limit Bucket Starvation via Steady-Stream Monopolization (No Per-Client Isolation)

### Summary
`ThrottleManagerImpl.throttle()` enforces rate limiting against a single, process-wide `rateLimitBucket` shared by every caller with no per-IP or per-client partitioning. An unprivileged attacker who sustains a request rate equal to the bucket's refill rate (`requestsPerSecond` tokens/second) keeps the bucket perpetually near-empty, eliminating the burst headroom the capacity is designed to provide and causing all burst traffic from legitimate users to be rejected with HTTP 429 while the attacker's own steady stream passes through unimpeded.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` constructs a single global Bucket4j bucket: [1](#0-0) 

- `capacity = requestsPerSecond` (default 500)
- `refillGreedy(rateLimit, Duration.ofSeconds(1))` — tokens refill continuously at ~1 token per 2 ms (for N=500), not in a single batch

`ThrottleManagerImpl.throttle()` consumes from this single shared bucket with no caller identity: [2](#0-1) 

**Root cause:** The bucket's `capacity` is intended to absorb burst traffic above the steady refill rate. However, because the bucket is global and there is no per-IP sub-bucket or fairness mechanism, a single client can consume tokens at exactly the refill rate, keeping available tokens at 0–1 at all times. The burst headroom (capacity − current_tokens) is permanently zero, so any concurrent burst from a legitimate user finds no tokens and is rejected.

**Why existing checks fail:**
- The `RequestProperties` filter chain (lines 44–48) is content-based (field matching, action LOG/REJECT/THROTTLE) and does not track caller identity. [3](#0-2) 
- There is no IP extraction, no per-client bucket map, and no authentication requirement anywhere in the throttle path.
- `ThrottleProperties.requestsPerSecond` is a single scalar; there is no per-client quota. [4](#0-3) 

### Impact Explanation
A single attacker can render the burst capacity of the web3 JSON-RPC endpoint permanently unavailable to all other users. Any legitimate client that sends more than ~1 request per refill interval (2 ms at default 500 req/s) will receive HTTP 429 `"Requests per second rate limit exceeded"` for the duration of the attack. Because the attacker's own requests are individually within the limit, they are never rejected. This is an effective, targeted denial-of-service against all other API consumers with zero cost to the attacker beyond sustaining a 500 req/s HTTP stream.

### Likelihood Explanation
**Preconditions:** None. The endpoint is publicly accessible; no authentication or API key is required.

**Attacker capability:** Sustaining 500 HTTP requests/second is trivially achievable with a single commodity machine using tools such as `wrk`, `hey`, or `ab`. The attack is repeatable indefinitely and requires no special knowledge of the target beyond the public endpoint URL.

**Detectability:** The attacker's requests are individually legitimate (each consumes exactly 1 token) and will not appear anomalous in per-request logs. Only aggregate traffic analysis would reveal the monopolization.

### Recommendation
1. **Per-client rate limiting:** Introduce a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`) so each client has its own token bucket with a per-IP quota (e.g., `requestsPerSecond / expectedConcurrentClients`).
2. **Two-tier limiting:** Retain the global bucket as a hard ceiling, but add a per-IP bucket as a first gate so no single IP can consume the entire global budget.
3. **Alternatively:** Delegate rate limiting to an API gateway or reverse proxy (e.g., nginx `limit_req`, Envoy rate-limit filter) that natively supports per-IP enforcement before requests reach the application.

### Proof of Concept
```bash
# Assume default requestsPerSecond = 500
# Attacker: sustain exactly 500 req/s from a single machine
wrk -t4 -c50 -d60s --latency \
  -s <(echo 'wrk.method="POST"; wrk.body="{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",\"params\":[{},\"latest\"],\"id\":1}"; wrk.headers["Content-Type"]="application/json"') \
  http://<target>/api/v1/contracts/call

# Simultaneously, legitimate user attempts a burst of 10 requests:
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://<target>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"eth_call","params":[{},"latest"],"id":1}' &
done
wait
# Expected result: most/all of the 10 concurrent legitimate requests return HTTP 429
# while the attacker's wrk stream continues to receive HTTP 200
```

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
