### Title
Unbounded Per-IP TCP Connections Allow Bypass of `maxConcurrentCallsPerConnection` Limit Leading to Resource Exhaustion DoS

### Summary
The gRPC server enforces a limit of 5 concurrent calls per TCP connection via `maxConcurrentCallsPerConnection`, but imposes no limit on the number of TCP connections a single IP (or set of IPs) may open. An unprivileged attacker can open thousands of TCP connections, each carrying 5 streams, multiplying the effective concurrency without restriction and exhausting the server's thread pool, database connection pool, and memory.

### Finding Description
**Code path:**

- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14: `maxConcurrentCallsPerConnection = 5`
- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–34: the `ServerBuilderCustomizer` bean applies only `executor(applicationTaskExecutor)` and `maxConcurrentCallsPerConnection(...)` to the `NettyServerBuilder`. No `maxConnectionAge`, no `maxConnections`, no per-IP connection count, and no global stream cap is configured.

**Root cause:** The design assumes `maxConcurrentCallsPerConnection = 5` acts as a meaningful resource cap. This assumption fails because gRPC/HTTP2 allows unlimited TCP connections from the same client. The per-connection limit is a per-connection ceiling, not a global or per-source ceiling.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (default gRPC port, publicly accessible, no authentication required for `subscribeTopic`).
2. On each connection the attacker issues 5 concurrent `subscribeTopic` streaming RPCs (the per-connection maximum).
3. Total active streams = N × 5. With N = 2,000 connections, that is 10,000 concurrent long-lived streaming RPCs.
4. Each stream holds a thread from `applicationTaskExecutor`, a HikariCP database connection (for historical retrieval), and a Redis subscription slot.
5. The `GrpcInterceptor` (the only server interceptor present) only sets `EndpointContext` for table-usage tracking — it performs no rate limiting or connection counting.

**Why existing checks fail:** No `ServerInterceptor` performs per-IP or global connection/stream accounting. No Kubernetes `NetworkPolicy` or ingress-level rate limit targeting port 5600 was found in the Helm charts. The `maxConcurrentCallsPerConnection` guard is entirely per-connection and provides no aggregate protection.

### Impact Explanation
`subscribeTopic` is a long-lived server-streaming RPC. Each active stream consumes a thread from the shared `applicationTaskExecutor` and a HikariCP database connection. Exhausting either pool causes all subsequent legitimate requests to queue indefinitely or fail. Memory pressure from tens of thousands of in-flight Reactor pipelines and message buffers (`maxBufferSize = 16384` per listener) can trigger OOM. The result is a complete denial of service for all legitimate users of the mirror node gRPC API.

### Likelihood Explanation
No authentication or API key is required to call `subscribeTopic`. The gRPC port is publicly exposed. Opening thousands of TCP connections is trivially achievable from a single commodity machine or a small botnet using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl` in a loop, or a simple Go/Python script). The attack is repeatable and requires no special knowledge beyond the public protobuf definition.

### Recommendation
Apply multiple complementary controls directly in `GrpcConfiguration`:

1. **Global connection limit:** Call `serverBuilder.maxConnectionAge(Duration, TimeUnit)` and `serverBuilder.maxConnectionAgeGrace(Duration, TimeUnit)` to recycle connections and bound total live connections.
2. **Per-IP connection throttling:** Implement a `ServerTransportFilter` that tracks open connections per remote IP and rejects new connections above a threshold (e.g., 10–20 per IP).
3. **Global concurrent-call cap:** Add a `ServerInterceptor` that uses an `AtomicInteger` or `Semaphore` to enforce a global maximum number of active streaming calls, returning `RESOURCE_EXHAUSTED` when the cap is reached.
4. **Infrastructure layer:** Configure ingress-level (e.g., Envoy, nginx, HAProxy) connection-rate and concurrent-connection limits per source IP for port 5600.

### Proof of Concept
```bash
# Open 500 connections each with 5 concurrent subscribeTopic streams = 2500 active streams
for i in $(seq 1 500); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <grpc-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Observe: server thread pool saturated, DB connections exhausted,
# legitimate clients receive UNAVAILABLE or hang indefinitely.
```
