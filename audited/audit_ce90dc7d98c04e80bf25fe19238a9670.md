### Title
Unbounded Per-Subscriber DB Polling in `PollingTopicListener` Enables N-fold DB Load Amplification

### Summary
`PollingTopicListener.listen()` is a cold Reactor publisher: every call creates an independent `PollingContext` and an independent polling loop that issues a full DB query every `interval` milliseconds. Because no subscription limit, per-topic cap, or deduplication exists anywhere in the call stack, an unprivileged attacker who opens N concurrent gRPC subscriptions to the same `topicId` causes exactly N independent `findByFilter` queries to execute against the database at every poll interval, linearly amplifying DB load with zero authentication required.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` ( [1](#0-0) ) constructs a brand-new `PollingContext` on every invocation and wraps it in `Flux.defer(() -> poll(context))` with `repeatWhen(RepeatSpec.times(Long.MAX_VALUE)...)`. This is a **cold** publisher — each subscriber gets its own independent execution chain.

`poll(context)` ( [2](#0-1) ) directly calls `topicMessageRepository.findByFilter(newFilter)` — a live DB query — on every tick. With the default `interval` of 500 ms ( [3](#0-2) ), N open subscriptions produce `N × 2` DB queries per second.

**Root cause — no sharing, no cap:**

`TopicMessageServiceImpl.subscribeTopic()` only increments a metrics gauge (`subscriberCount`) — it enforces no limit. ( [4](#0-3) ) There is no per-IP, per-topic, or global subscription cap anywhere in the service or controller layer.

`TopicMessageFilter` generates a fresh random `subscriberId` per filter instance, so every subscription is treated as a completely independent entity with no deduplication. ( [5](#0-4) )

**Contrast with the safe alternative:**

`SharedPollingTopicListener` (type `SHARED_POLL`) explicitly calls `.share()` so all subscribers fan-out from a single poll loop. ( [6](#0-5) ) `PollingTopicListener` (type `POLL`) has no equivalent sharing mechanism.

**Why existing checks fail:**

The `subscriberCount` gauge is purely observational. ( [7](#0-6) ) No code path reads it to reject or throttle new subscriptions.

### Impact Explanation

When the listener type is configured to `POLL` ( [8](#0-7) ), an attacker opening 1,000 concurrent subscriptions to a single `topicId` causes 2,000 identical DB queries per second. Each query scans `topic_message` rows filtered by `topicId` and `startTime`. At sufficient scale this saturates DB connection pools, increases query latency for all users, and can cause cascading failures across the mirror node. No financial loss occurs, but service availability for all subscribers is degraded — consistent with the "griefing" severity classification.

### Likelihood Explanation

The attack requires only a gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open many TCP connections — no credentials, no privileged access, no on-chain funds. It is trivially scriptable, repeatable, and can be launched from a single machine or a small botnet. The only prerequisite is that the operator has configured `ListenerType.POLL`; this is not the default (`REDIS` is default), but it is a documented, supported configuration option.

### Recommendation

1. **Enforce a per-IP or global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()` by reading `subscriberCount` (or a per-topic counter) and rejecting new subscriptions above a configurable threshold with `RESOURCE_EXHAUSTED` gRPC status.
2. **Prefer `SHARED_POLL` or `REDIS`** over `POLL` in all deployments; consider deprecating or removing the unshared `POLL` type.
3. **Add per-topic deduplication** in `PollingTopicListener`: maintain a `ConcurrentHashMap<EntityId, Flux<TopicMessage>>` of active shared fluxes (similar to `SharedPollingTopicListener`) so N subscribers to the same topic share one poll loop.
4. **Apply gRPC server-side rate limiting** (e.g., via an interceptor) to cap concurrent streams per remote address.

### Proof of Concept

```bash
# Open 500 concurrent subscriptions to the same topicId (0.0.12345)
# using grpcurl in background processes
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 12345}, "consensusStartTime": {"seconds": 0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    > /dev/null 2>&1 &
done

# Observe on the DB host:
# SELECT count(*), query FROM pg_stat_activity
# WHERE query LIKE '%topic_message%'
# GROUP BY query;
# → ~500 identical concurrent queries executing every 500ms
```

Each background process holds an open gRPC stream; `PollingTopicListener` fires an independent `findByFilter` DB call for each one at every poll interval, producing the N-fold load amplification described above.

```java
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
