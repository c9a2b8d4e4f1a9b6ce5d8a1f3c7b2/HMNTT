### Title
Unauthenticated Connection Pool Exhaustion via Unthrottled `GET /api/v1/blocks`

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts unlimited concurrent requests from unauthenticated users with no rate limiting. Each request acquires a connection from `primaryPool`, which defaults to only 10 connections. An attacker maintaining 10+ concurrent requests holds all pool connections busy, causing every subsequent block query to queue and eventually fail with a connection timeout error, producing a sustained denial-of-service against the REST layer's database connectivity.

### Finding Description

**Exact code path:**

`getBlocks()` at `rest/controllers/blockController.js:101-112` calls `RecordFileService.getBlocks(formattedFilters)` at `rest/service/recordFileService.js:149-162`, which calls `super.getRows(query, params)` at `rest/service/baseService.js:55-57`, which calls `this.pool().queryQuietly(query, params)`. `RecordFileService.pool()` at line 203-205 returns `primaryPool`.

`primaryPool` is initialized in `rest/dbpool.js:35-46` with:
- `max: config.db.pool.maxConnections` — documented default **10** connections
- `statement_timeout: config.db.pool.statementTimeout` — documented default **20,000 ms**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` — documented default **20,000 ms**

**No rate limiting exists on the REST API.** A grep across all `rest/**/*.js` for `rateLimit`, `rate-limit`, `express-rate`, `throttle`, or `limiter` returns zero matches. The throttling code in `web3/src/main/java/.../ThrottleManagerImpl.java` applies only to the Java web3 module, not to the Node.js REST service.

**Root cause:** The `blockRoute.js` mounts `BlockController.getBlocks` directly with no middleware guard:
```js
router.getExt('/', BlockController.getBlocks);
```
No concurrency limit, no per-IP rate limit, and no queue depth cap is applied before the handler acquires a DB connection.

**Limit cap does not help:** `extractLimitFromFilters()` at line 57-61 caps the SQL `LIMIT` at `maxLimit` (default 100) or falls back to `defaultLimit` (25). This bounds result-set size but does nothing to bound concurrent connection acquisition. Each of the 10 concurrent requests holds one pool connection for up to 20 seconds (statement_timeout), and the attacker simply re-fires requests as connections are released.

### Impact Explanation

With the default pool of 10 connections and a 20-second statement timeout, an attacker holding 10 concurrent in-flight requests saturates `primaryPool`. All subsequent legitimate requests queue for a connection; after 20 seconds without acquiring one (`connectionTimeoutMillis`), `pg` throws an error, the Express error handler returns HTTP 500, and the response cache (Redis, optional) serves stale data or is bypassed. The entire `/api/v1/blocks` endpoint — and any other endpoint sharing `primaryPool` — becomes unavailable for the duration of the attack. This constitutes a de-facto partition between the REST layer and its backing store.

### Likelihood Explanation

No authentication, no special knowledge, and no amplification technique is required. A single attacker with a basic HTTP client (e.g., `ab`, `wrk`, or a simple async loop) can sustain 10 concurrent long-running queries indefinitely. The attack is repeatable, cheap, and requires no prior access or credentials.

### Recommendation

1. **Add per-IP rate limiting** to the Express REST application (e.g., `express-rate-limit`) applied globally or specifically to `/api/v1/blocks`.
2. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer) in front of PostgreSQL to absorb burst load without starving the application pool.
3. **Reduce `statement_timeout`** for the blocks query to a value well below 20 seconds to release connections faster under load.
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) inside the service layer to cap the number of simultaneous DB queries regardless of HTTP concurrency.

### Proof of Concept

```bash
# Saturate the 10-connection primaryPool with concurrent no-filter requests
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks" &
done
wait

# Subsequent legitimate requests now fail or time out:
curl -v "http://<mirror-node-host>:5551/api/v1/blocks"
# Expected: HTTP 500 or connection timeout after ~20 seconds
```

Repeat the loop continuously to maintain exhaustion. No credentials or special headers are needed.
