### Title
Missing Entity Type Validation in `getEntityIdFromEvmAddress` Allows Contract EVM Address to Resolve as Account in Transaction History Endpoints

### Summary
`getEntityIdFromEvmAddress()` in `rest/service/entityService.js` resolves an EVM address to an entity ID using a SQL query that contains no `type` filter, meaning it will match both `ACCOUNT` and `CONTRACT` entities indiscriminately. All account-specific REST endpoints that call `getEncodedId()` → `getEntityIdFromEvmAddress()` will accept a contract's EVM address and return that contract's transaction history, NFTs, tokens, staking rewards, and allowances under the account API surface. The `isValidAccount()` guard used in some endpoints is also broken — it checks only entity existence, not entity type — so it provides no protection.

### Finding Description

**Exact code location:**

`rest/service/entityService.js`, lines 22–25 — the query used by `getEntityIdFromEvmAddress`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

There is no `AND type = 'ACCOUNT'` predicate. The query matches any non-deleted entity — account or contract — that has the given EVM address.

`getEntityIdFromEvmAddress()` (lines 90–104) executes this query and returns `rows[0].id` with zero type inspection: [2](#0-1) 

`getEncodedId()` (lines 118–137) calls `getEntityIdFromEvmAddress()` when the input is an EVM address, and returns the resolved ID to every caller: [3](#0-2) 

**Callers that expose account-specific endpoints:**

- `GET /accounts/:id/nfts` — `accountController.js` line 91 calls `getEncodedId()` with no subsequent type check: [4](#0-3) 

- `GET /accounts/:id/rewards` — `accountController.js` line 171 calls `getEncodedId()` then `isValidAccount()`: [5](#0-4) 

- `GET /accounts/:id/tokens` — `tokenController.js` line 67 calls `getEncodedId()` then `isValidAccount()`: [6](#0-5) 

- `GET /accounts/:id` — `accounts.js` line 399 calls `getEncodedId()` with no type check: [7](#0-6) 

**Why `isValidAccount()` is an insufficient guard:**

```js
async isValidAccount(accountId) {
    const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
    return !isNil(entity);
}
``` [8](#0-7) 

`entityExistenceQuery` selects `type` from the entity table filtered only by `id`: [9](#0-8) 

The function returns `!isNil(entity)` — it never inspects `entity.type`. A contract entity with the given ID will cause `isValidAccount` to return `true`, bypassing the guard entirely.

**Root cause:** The failed assumption is that `getEntityIdFromEvmAddress` is only ever called with EVM addresses belonging to `ACCOUNT`-type entities. The query and the function make no such enforcement.

### Impact Explanation

An unprivileged external user can supply the EVM address of any deployed contract to account-specific REST endpoints. The API will resolve the contract's entity ID and return that contract's transaction history, NFT holdings, token relationships, staking rewards, and crypto allowances under the `/accounts/` namespace — data that is semantically associated with a contract, not an account. This reorganizes the visible transaction history by surfacing contract-type data through account-type query paths. While no funds are moved, the integrity of the account/contract distinction in the API is broken, and data that should only appear under `/contracts/` is exposed under `/accounts/`. Severity: Medium — information boundary violation with no authentication required.

### Likelihood Explanation

Exploitation requires zero privileges. Any user who knows (or can enumerate) the EVM address of a deployed contract — which is public on-chain data — can trigger this immediately with a single HTTP GET request. The attack is trivially repeatable and requires no special tooling. Contracts with EVM addresses are common on Hedera (CREATE2 contracts, hollow accounts promoted to contracts, etc.), so the precondition is routinely satisfied in production.

### Recommendation

1. Add `AND type = 'ACCOUNT'` to `entityFromEvmAddressQuery`, or introduce a separate `entityFromEvmAddressAccountQuery` used exclusively by account endpoints.
2. Fix `isValidAccount()` to check `entity.type === 'ACCOUNT'` rather than just `!isNil(entity)`.
3. For endpoints that intentionally accept both accounts and contracts (if any), use a separate, explicitly typed query.

### Proof of Concept

**Precondition:** A contract exists in the mirror node database with a known EVM address, e.g. `0xaabbccddeeff00112233445566778899aabbccdd`, stored in the `entity` table with `type = 'CONTRACT'`.

**Steps:**

1. Send a request to the account NFTs endpoint using the contract's EVM address as the account identifier:
   ```
   GET /api/v1/accounts/0xaabbccddeeff00112233445566778899aabbccdd/nfts
   ```
2. `getEncodedId("0xaabbccddeeff00112233445566778899aabbccdd")` is called.
3. `EntityId.isValidEntityId(...)` returns true for a hex EVM address; `getEntityIdFromEvmAddress()` executes `entityFromEvmAddressQuery` with no type filter.
4. The query matches the contract row and returns its `id`.
5. The NFT query runs against that entity ID and returns NFTs owned by the contract.
6. Repeat for `/tokens`, `/rewards`, `/allowances/crypto`, and the base `/accounts/` endpoint — all return contract-associated data with HTTP 200.

**Expected (correct) behavior:** A 404 Not Found or 400 Bad Request, because the resolved entity is not of type ACCOUNT.

**Observed behavior:** HTTP 200 with contract entity data returned under the account API surface.

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
