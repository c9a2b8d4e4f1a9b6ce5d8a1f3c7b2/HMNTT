### Title
Cache Key Collision via 32-bit `Arrays.hashCode` Allows Entity Data Confusion in EVM Address Lookups

### Summary
`findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` use `@spelHelper.hashCode(#alias)` — which returns a 32-bit `int` from `Arrays.hashCode(byte[])` — as the sole Caffeine cache key for 20-byte EVM addresses. Because the cache stores and retrieves entries keyed only on this `Integer`, two distinct EVM addresses that share the same `Arrays.hashCode` value map to the same cache slot. An unprivileged attacker who can create Hedera accounts (permissionless, low cost) can craft a colliding address, poison the cache, and cause the mirror node's web3 API to return entity A's data for entity B's address lookup.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode` returns `Arrays.hashCode(value)` — a plain 32-bit `int`: [1](#0-0) 

This `int` is used as the complete cache key for both cached methods: [2](#0-1) 

The backing store is a `CaffeineCacheManager` keyed on the boxed `Integer`: [3](#0-2) 

**Root cause:** The cache key is a 32-bit integer derived from a 20-byte (160-bit) EVM address. The Caffeine cache has no knowledge of the original byte array — it only sees the `Integer` key. Two distinct byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` are indistinguishable to the cache. When entity A's result is stored under key `k`, a subsequent lookup for entity B (same `k`) returns entity A's cached `Optional<Entity>` without touching the database.

**Why `unless = "#result == null"` is insufficient:** This guard only prevents caching of null/empty results. It does nothing to prevent a cache hit from returning a previously stored result for a *different* address that happens to share the same 32-bit hash.

**Exploit flow:**
1. Attacker identifies a target entity (entity B) whose EVM address they want to spoof.
2. Attacker computes `Arrays.hashCode(addressB)` — a 32-bit value.
3. Attacker generates ECDSA key pairs until the derived EVM address (last 20 bytes of keccak256 of public key) produces the same `Arrays.hashCode`. By the birthday paradox, ~65,536 attempts yield a 50% collision probability — trivially fast on any modern machine.
4. Attacker creates a Hedera account with that colliding key pair (entity A), establishing a real on-chain entity at address A.
5. Attacker queries the mirror node's web3 API for entity A's address → cache miss → DB returns entity A → cached under key `k`.
6. Attacker (or any user) queries entity B's address → cache hit with key `k` → mirror node returns entity A's `Entity` record (balance, EVM code, account type, etc.) instead of entity B's.

### Impact Explanation
The mirror node's web3 API (`eth_getBalance`, `eth_getCode`, `eth_call`, contract execution context) relies on `findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` to resolve entities. Serving entity A's record for entity B's address causes:
- Incorrect balance reporting for entity B
- Incorrect bytecode returned for smart contract lookups
- Incorrect account metadata (deleted status, type) used in EVM execution
- Downstream incorrect records exported/served by the mirror node

This is a medium-severity data integrity issue: it does not affect the Hedera consensus layer but corrupts the mirror node's read API, which is the primary interface for EVM-compatible tooling (MetaMask, Hardhat, ethers.js, etc.).

### Likelihood Explanation
The attack requires no privileges — only the ability to submit Hedera transactions (create an account), which costs a small HBAR fee (fractions of a cent). Finding a 32-bit `Arrays.hashCode` collision against a specific target requires ~65K key-pair generations on average, achievable in milliseconds on commodity hardware. The attack is repeatable and can be automated. The only constraint is that the attacker must create a real on-chain account, leaving a transaction record, but this is not a meaningful deterrent.

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with a collision-resistant representation of the full byte array. Options:
1. Use `java.util.Arrays.toString(alias)` or `org.apache.commons.codec.binary.Hex.encodeHexString(alias)` as the key — these produce a unique string per distinct byte array.
2. Add a `hashCode(byte[])` overload in `SpelHelper` that returns a cryptographically strong digest (e.g., `Arrays.toString(alias)` or a hex string), or simply use the byte array's hex encoding directly in the SpEL expression.
3. Alternatively, wrap the byte array in a value object that implements `equals`/`hashCode` correctly (using `Arrays.equals` / `Arrays.hashCode` together), and use that as the key — but note that Spring Cache key equality is based on `equals`, so the key object must implement `equals` using full array comparison.

The simplest safe fix:
```java
// In SpelHelper:
public String cacheKey(byte[] value) {
    return org.apache.commons.codec.binary.Hex.encodeHexString(value);
}
```
```java
// In EntityRepository:
key = "@spelHelper.cacheKey(#alias)"
```
