### Title
Unauthenticated Alias-Lookup Endpoint Exhausts HikariCP Connection Pool During Network Partition, Causing Full Service Denial

### Summary
The `rest-java` service exposes multiple unauthenticated endpoints that resolve account aliases by calling `EntityRepository.findByAlias()` against the `entity` table. The service has no application-level rate limiting and no alias-lookup result caching. During a network partition between the application and the database, in-flight JDBC connections hang indefinitely (the DB-side `statementTimeout` cannot fire when the DB is unreachable), allowing an unprivileged attacker to exhaust the HikariCP connection pool with as few as 10 concurrent requests and block all subsequent legitimate queries indefinitely.

### Finding Description

**Exact code path:**

`EntityRepository.findByAlias()` issues a native SQL query with no timeout annotation or caching: [1](#0-0) 

It is called unconditionally for every alias-typed path parameter across multiple endpoints via `EntityServiceImpl.lookup()`: [2](#0-1) 

The endpoints that trigger this path include (from the REST API table):
- `GET /api/v1/accounts/:alias/allowances/crypto`
- `GET /api/v1/accounts/:alias/allowances/tokens`
- `GET /api/v1/accounts/:aliasOrAddress/allowances/nfts`
- `GET /api/v1/accounts/:alias/nfts`
- `GET /api/v1/accounts/:alias/airdrops/outstanding`
- `GET /api/v1/accounts/:alias/hooks`

All are publicly accessible with no authentication.

**Root cause — failed assumption:**

The design assumes that `hiero.mirror.restJava.db.statementTimeout: 10000` (10 s) will bound query duration and return connections to the pool promptly: [3](#0-2) 

This assumption fails during a network partition. `statement_timeout` is a PostgreSQL server-side setting. When the TCP path between the application and the database is severed, the already-established JDBC connection enters a hung state: the application cannot send the query, or the DB cannot respond. The OS TCP keepalive default is ~2 hours on Linux, so the connection is not declared dead for hours. The DB-side timeout never fires because the DB never receives (or cannot respond to) the query.

**HikariCP pool size:**

No `spring.datasource.hikari.maximum-pool-size` is configured for `rest-java`: [4](#0-3) 

HikariCP's default `maximumPoolSize` is **10**. With 10 hung connections, the pool is fully exhausted.

**No application-level rate limiting in rest-java:**

The throttle/rate-limiting infrastructure (bucket4j `ThrottleManager`, `ThrottleProperties`) exists only in the `web3` module: [5](#0-4) 

There is no equivalent in `rest-java/src/main/**`. A search for `RateLimiter`, `throttle`, `bucket4j`, or `@RateLimit` in `rest-java/src/main/**` returns zero results for the alias-lookup path.

**No caching for alias lookups:**

Unlike the `web3` module (which has `expireAfterWrite=1s,maximumSize=10000` entity caching), `rest-java` has no cache in front of `findByAlias`. Every request hits the database.

**Infrastructure-level rate limit is insufficient:**

The GCP backend policy sets `maxRatePerEndpoint: 250`: [6](#0-5) 

This is: (a) infrastructure-level and not present in all deployments, (b) 250 RPS per endpoint — far more than the 10 connections needed to exhaust the pool when each connection hangs for hours, and (c) not enforced at the application layer.

### Impact Explanation

During a network partition, 10 concurrent alias-lookup requests are sufficient to hold all HikariCP connections indefinitely. Once the pool is exhausted, every subsequent request — including health checks, network supply queries, and all other endpoints sharing the same pool — blocks waiting for a connection (up to HikariCP's default `connectionTimeout` of 30 s) and then fails with a pool acquisition timeout. The result is a **complete service denial** for the entire `rest-java` instance for the duration of the partition. The Grafana alert `RestJavaHighDBConnections` (fires at 75% utilization) would trigger, but by then the pool is already full and the service is already down. [7](#0-6) 

### Likelihood Explanation

- **Precondition 1 (network partition):** Can occur due to cloud infrastructure failures, misconfigured network policies, or deliberate network-level interference. The question frames this as an environmental condition outside design parameters.
- **Precondition 2 (attacker capability):** Zero — any unauthenticated HTTP client can send alias-format GET requests. No credentials, tokens, or special knowledge required. A valid alias is a base32-encoded string matching `[A-Z2-7]{40,70}`, trivially constructable.
- **Trigger:** Send ≥10 concurrent requests to any alias endpoint (e.g., `GET /api/v1/accounts/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA/nfts`) during the partition window.
- **Repeatability:** Fully repeatable for the duration of the partition. The attacker does not need to sustain high throughput — 10 concurrent long-lived requests suffice.

### Recommendation

1. **Add application-level rate limiting to `rest-java`** analogous to the `web3` `ThrottleManager` (bucket4j), enforcing a per-IP or global RPS cap before any DB access.
2. **Add a short HikariCP `connectionTimeout` and `keepaliveTime`** in `rest-java` configuration (e.g., `spring.datasource.hikari.keepalive-time=30000`, `connection-timeout=5000`) so that dead connections are detected and evicted promptly.
3. **Add a JDBC socket timeout** (e.g., `socketTimeout=15` in the JDBC URL) so that hung TCP connections are detected at the driver level, independent of the DB-side `statementTimeout`.
4. **Cache alias lookups** with a short TTL (e.g., 1–5 s) to reduce per-request DB load, consistent with the `web3` entity cache pattern.
5. **Increase the pool size** and configure `maximumPoolSize` explicitly so that a small number of hung connections cannot exhaust the entire pool.

### Proof of Concept

**Preconditions:** Network partition between `rest-java` and PostgreSQL is active (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the app host, or a cloud firewall rule blocking DB egress).

**Steps:**

```bash
# 1. Confirm the alias endpoint is publicly accessible (no auth)
curl -s "https://<rest-java-host>/api/v1/accounts/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA/nfts"
# Returns 404 or 200 normally

# 2. Simulate network partition (on app host or via cloud firewall)
# iptables -I OUTPUT -p tcp --dport 5432 -j DROP

# 3. Send 10 concurrent alias-lookup requests (one per pool connection)
for i in $(seq 1 10); do
  curl -s "https://<rest-java-host>/api/v1/accounts/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA/nfts" &
done
wait

# 4. Immediately send a legitimate request (e.g., network supply)
curl -v "https://<rest-java-host>/api/v1/network/supply"
# Expected result: hangs for 30s (HikariCP connectionTimeout), then fails with 503/500
# All 10 pool connections are held by the hung alias queries
```

**Expected result:** The `network/supply` request (and all other requests) block for the full HikariCP `connectionTimeout` duration (default 30 s) and then fail. The service is effectively down for all users for the duration of the partition.

        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
