### Title
Unbounded `keys` Collection in `getHookStorageChange` Causes IN-Clause Explosion and Database DoS

### Summary
`HookServiceImpl.getHookStorageChange()` passes the full, user-supplied `keys` collection directly to the native SQL query `findByKeyInAndTimestampBetween()` without any size cap. Spring Data JPA expands each element into a separate bind parameter, so an attacker supplying thousands of `key=` query parameters generates a SQL statement with thousands of IN-clause entries, exhausting database parse/plan/memory resources and causing denial of service.

### Finding Description
**Exact code path:**

In `HookStorageRequest.java`, the `keys` field carries no `@Size` or any other constraint: [1](#0-0) 

In `HookServiceImpl.getHookStorageChange()`, the check at line 93 only short-circuits when **all** supplied keys fall outside the range. If even one key is in range, execution falls through to line 104 where the **full, unfiltered `keys` collection** (not `keysInRange`) is forwarded to the repository: [2](#0-1) 

The repository method uses a native SQL query with `key in (:keys)`: [3](#0-2) 

Spring Data JPA expands a `Collection` parameter in a native query into one bind placeholder per element (`key in (?,?,?,...)`). With no upper bound on collection size, this produces arbitrarily large SQL statements.

**Root cause / failed assumption:** The developer assumed the `limit` pagination field (capped at 25 by default) would bound query cost. It does not — `limit` controls result-set size, not the size of the IN clause used to filter rows. The non-historical path correctly passes `keysInRange` (already filtered), but the historical path at line 104 passes the raw `keys` collection, an inconsistency that bypasses the only partial mitigation. [4](#0-3) 

### Impact Explanation
A single HTTP request with thousands of `key=<hex>` query parameters causes the database to parse, plan, and execute a query with a proportionally large IN clause. Repeated requests can saturate database CPU, memory, and connection pool resources, making the mirror-node REST API unavailable to all users. Because the historical (timestamp-filtered) path is the trigger, any endpoint that accepts a `timestamp` parameter alongside `key` parameters is affected.

### Likelihood Explanation
No authentication is required — the mirror-node REST API is a public read API. The attack is trivially scriptable: a single `curl` or Python `requests` call with a loop generating thousands of `key=` parameters is sufficient. It is repeatable at will and requires no special knowledge beyond the public API schema.

### Recommendation
1. **Enforce a hard size cap on `keys`** in `HookStorageRequest` using a Bean Validation annotation, e.g. `@Size(max = 100)` on the `keys` field, and ensure the validator is invoked at the controller layer (`@Valid`).
2. **Pass `keysInRange` instead of `keys`** to `findByKeyInAndTimestampBetween` (mirroring the non-historical path at line 78) so the filtered, smaller collection is used even before the size cap is reached.
3. Optionally add a global `@ControllerAdvice` that rejects requests exceeding a configurable maximum number of repeated query parameters.

### Proof of Concept
```bash
# Build a request with 5000 distinct key= parameters (all within the valid range)
PARAMS=$(python3 -c "
import os, binascii
params = '&'.join(
    'key=' + binascii.hexlify(os.urandom(32)).decode()
    for _ in range(5000)
)
print(params)
")

# Send to the historical hook-storage endpoint (timestamp param triggers getHookStorageChange)
curl -s "https://<mirror-node-host>/api/v1/hooks/<ownerId>/storage?timestamp=gte:0&$PARAMS" \
     -o /dev/null -w "%{time_total}\n"
```

Repeat in a tight loop from multiple clients. Each request forces the database to evaluate `key in (?, ?, ..., ?)` with 5 000 parameters. Database CPU and memory spike; connection pool exhausts; legitimate requests time out.
