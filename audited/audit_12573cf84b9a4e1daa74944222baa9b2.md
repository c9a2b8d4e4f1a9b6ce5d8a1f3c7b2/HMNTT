### Title
Cache Key Bypass via URL Percent-Encoding in `cacheKeyGenerator()` Allows Real-Time Staking Reward Observation

### Summary
The `cacheKeyGenerator()` function in `rest/middleware/responseCacheHandler.js` computes the Redis cache key by MD5-hashing `req.originalUrl` verbatim, with no URL normalization. Because Express preserves percent-encoding in `req.originalUrl`, an unprivileged attacker can craft semantically equivalent but syntactically distinct URLs (e.g., `/api/v1/accounts/%30.%30.1001/rewards`) that produce different MD5 digests, causing perpetual cache misses and forcing every request to hit the database directly. This gives the attacker real-time visibility into `pending_reward` values while legitimate users receive stale cached data.

### Finding Description

**Exact code location:**

```js

  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [1](#0-0) 

The code comment at line 149 explicitly acknowledges the missing normalization:
```

``` [2](#0-1) 

**Root cause:** `req.originalUrl` in Express.js is the raw, unmodified URL string as received from the TCP socket. Express does **not** percent-decode or normalize it. Therefore:

- `/api/v1/accounts/0.0.1001/rewards` â†’ MD5 of string A â†’ cache key A
- `/api/v1/accounts/%30.%30.1001/rewards` â†’ MD5 of string B â†’ cache key B  
  (`%30` = `0`, so both resolve to account `0.0.1001`)

These are different MD5 digests â†’ different Redis keys â†’ independent cache entries.

**Why existing checks fail:**

1. `requestNormalizer.js` (`normalizeRequestQueryParams`) only normalizes query parameters using the already-decoded `req.path` and `req.query` objects â€” it never touches `req.originalUrl`. [3](#0-2) 

2. `requestHandler.js` (`requestQueryParser`) handles query string case-insensitivity but does not modify `req.originalUrl`. [4](#0-3) 

3. The OpenAPI validator is configured with `validateRequests: false`, so it does not reject percent-encoded path segments. [5](#0-4) 

4. The route controller uses `req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]`, which Express **does** decode before populating `req.params`. So the DB query always uses the canonical decoded value `0.0.1001` â€” the response data is identical, but the cache key is not. [6](#0-5) 

**Exploit flow:**

1. Legitimate user requests `/api/v1/accounts/0.0.1001/rewards` â†’ cache miss â†’ DB hit â†’ response cached under key `MD5("/api/v1/accounts/0.0.1001/rewards")-v1`.
2. Attacker requests `/api/v1/accounts/%30.%30.1001/rewards` â†’ `cacheKeyGenerator` computes `MD5("/api/v1/accounts/%30.%30.1001/rewards")-v1` â†’ different key â†’ cache miss â†’ DB hit â†’ real-time data returned.
3. Attacker can rotate through an effectively unlimited set of encoding variants (`%30`, `%30` uppercase `%30` vs `%30`, mixed case `%2F` vs `%2f` for any separator, double-encoding, etc.) to ensure every request is a cache miss.

### Impact Explanation

The attacker receives real-time `pending_reward` values from the database on every request, while other users receive stale cached data (up to the configured `max-age`). In a staking context, this creates an information asymmetry: the attacker can observe the exact moment rewards accumulate or are distributed, enabling front-running of reward-sensitive operations. Additionally, sustained use of this technique constitutes a targeted cache-busting DoS against the database layer for specific high-value accounts, as every encoded request bypasses Redis and issues a full DB query.

### Likelihood Explanation

No privileges, authentication, or special network position are required. The technique requires only knowledge of percent-encoding (publicly documented in RFC 3986) and the ability to send HTTP requests. It is trivially scriptable, repeatable indefinitely, and undetectable by standard rate-limiting that operates on canonical URLs. The comment in the source code (`In the future, this will utilize Edwin's request normalizer (9113)`) confirms the developers are aware the normalization gap exists and it has not yet been closed.

### Recommendation

Normalize `req.originalUrl` before hashing in `cacheKeyGenerator`. The minimal fix is to percent-decode the URL before computing the MD5:

```js
const cacheKeyGenerator = (req) => {
  const normalizedUrl = decodeURIComponent(req.originalUrl).toLowerCase();
  return crypto.createHash('md5').update(normalizedUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

A more robust fix is to reconstruct the canonical URL from the already-decoded and normalized `req.path` and `req.query` objects (as `requestNormalizer.js` already does for query params), and use that reconstructed string as the cache key input â€” which is exactly what the referenced issue 9113 intends to implement. That work should be prioritized.

### Proof of Concept

```bash
# Step 1: Warm the canonical cache entry
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/rewards"
# Response served from DB, cached under MD5("/api/v1/accounts/0.0.1001/rewards")-v1

# Step 2: Bypass cache with percent-encoded account ID
curl -s "https://<mirror-node>/api/v1/accounts/%30.%30.1001/rewards"
# Cache miss â€” DB hit â€” real-time pending_reward returned

# Step 3: Rotate encodings to ensure perpetual cache bypass
curl -s "https://<mirror-node>/api/v1/accounts/%30.%30.%31%30%30%31/rewards"
# Another distinct MD5 key â€” another cache miss â€” another DB hit

# Verify: the DB-sourced response contains current pending_reward
# while /api/v1/accounts/0.0.1001/rewards still serves stale cached data
```
