### Title
Unbounded Concurrent Subscriber Heap Exhaustion via Eager `getResultList()` in POLL Mode

### Summary
When `PollingTopicListener` is active (listener type `POLL`), each subscriber independently triggers a DB query every 500 ms that materializes up to `maxPageSize=5000` `TopicMessage` objects into heap via `getResultList()`. Because the gRPC `subscribeTopic` endpoint requires no authentication and enforces no concurrent-subscriber cap, an attacker can open arbitrarily many subscriptions, multiplying heap pressure by N subscribers × 5000 objects × 2 polls/second until the JVM runs out of memory.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` [1](#0-0)  accepts any unauthenticated gRPC call and passes the user-supplied `ConsensusTopicQuery` (with `limit=0` meaning "no limit") directly to `TopicMessageServiceImpl.subscribeTopic()`. [2](#0-1) 

`TopicMessageServiceImpl` tracks subscriber count only as a Micrometer gauge — there is no enforcement of a maximum. [3](#0-2) 

In POLL mode, `PollingTopicListener.poll()` computes `pageSize = Math.min(Integer.MAX_VALUE, maxPageSize)` = 5000 when the user supplies no limit, then builds a new filter with `limit(5000)` and calls `topicMessageRepository.findByFilter(newFilter)`. [4](#0-3) 

`TopicMessageRepositoryCustomImpl.findByFilter()` calls `typedQuery.getResultList().stream()` — the comment explicitly acknowledges that `getResultStream()`'s cursor was abandoned. This materializes the entire result set (up to 5000 rows) into a `List<TopicMessage>` on the heap before returning. [5](#0-4) 

**Root cause:** Two independent weaknesses compose into a DoS:
1. `getResultList()` allocates a full in-memory list per poll cycle per subscriber.
2. No subscriber admission control exists anywhere in the call chain.

**Why existing checks fail:**

- `maxPageSize=5000` (default) caps *per-query* rows but does nothing to limit the number of concurrent subscribers issuing those queries. [6](#0-5) 
- The `@Min(0)` constraint on `limit` in `TopicMessageFilter` allows `limit=0` (no limit), which is the documented default per the proto spec. [7](#0-6) 
- The proto definition explicitly states `limit=0` means "return messages indefinitely." [8](#0-7) 
- The poll interval floor of 50 ms (`@DurationMin(millis = 50)`) still allows up to 20 polls/second per subscriber. [9](#0-8) 

### Impact Explanation
Each `TopicMessage` carries a `message` field up to 6 KB (per HCS spec) plus metadata fields. At 5000 objects per poll cycle: ~30 MB per subscriber per 500 ms. With 100 concurrent attackers: ~3 GB heap churn per second. This causes repeated full GC pauses and ultimately OOM, crashing the gRPC service and denying all subscribers access to HCS topic data. The mirror node is a critical read path for the Hedera network; its unavailability affects all downstream consumers.

### Likelihood Explanation
The gRPC endpoint is publicly documented and requires zero credentials. [10](#0-9)  A single attacker with a script using `grpcurl` or the Hedera SDK can open hundreds of persistent subscriptions in seconds. The attack is trivially repeatable and requires no special knowledge beyond the public proto definition. The only prerequisite is that the operator has configured `type: POLL` — a supported, documented listener mode. [11](#0-10) 

### Recommendation
1. **Enforce a maximum concurrent subscriber count** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., return `RESOURCE_EXHAUSTED` gRPC status).
2. **Per-IP or per-topic subscriber rate limiting** at the gRPC interceptor layer.
3. **Replace `getResultList()` with a true streaming cursor**: use `ScrollableResults` (Hibernate `scroll()`) or Spring Data's `Stream<T>` with `@QueryHints` to avoid materializing the full page in heap. The existing comment acknowledges `getResultStream()` was tried — a `ScrollableResults`-based approach avoids that issue. [12](#0-11) 
4. **Reduce `maxPageSize` default** or make it dynamically back-pressure-aware.

### Proof of Concept
```bash
# Open 200 concurrent unlimited subscriptions to a high-volume topic
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
# Each connection triggers a poll every 500ms loading up to 5000 TopicMessage
# objects into heap. Monitor JVM heap: OOM / GC thrash observed within seconds
# on a topic with >= 5000 messages since last poll timestamp.
```

**Preconditions:** POLL listener mode configured; target topic has sustained message volume ≥ 5000 messages per 500 ms window (or attacker resets `startTime` to a historical point with dense message history).

```java
    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
