### Title
Unvalidated `domain_name` Priority in Mirror Node Response Enables gRPC Endpoint Hijacking

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` unconditionally prefers the `domain_name` field over `ip_address_v4` from the mirror node REST response with no validation, allowlist, or format check. An attacker who can influence the mirror node HTTP response — via MITM on the plaintext HTTP connection, DNS hijacking of the mirror-rest hostname, or a compromised mirror node — can inject an arbitrary `domain_name` that resolves to an attacker-controlled IP, causing `hiero.ClientForNetwork()` to route all SDK gRPC traffic (including smart contract calls) to the attacker's server. The HTTP client used to fetch the mirror node data has no TLS certificate pinning, making interception feasible on any HTTP-scheme deployment.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–124:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)   // line 114 — domain_name wins unconditionally
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4) // line 116 — ip_address_v4 only used as fallback
    }
    if host == "" || ep.Port == 0 {
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId               // line 123 — attacker-controlled addr enters SDK network map
}
```

The resulting `network` map is passed directly to `hiero.ClientForNetwork(netmap)` in `sdk_client.go` line 22, which the SDK uses for all gRPC node connections.

**Root cause:** The code makes the failed assumption that `domain_name` in the mirror node response is always authoritative and trustworthy. There is no:
- Allowlist of permitted domain suffixes or IP ranges
- Cross-validation of `domain_name` against `ip_address_v4`
- TLS certificate pinning on the HTTP client (`&http.Client{Timeout: cfg.mirrorNodeClientTimeout}` — no transport customization)

**Exploit flow:**

1. The pinger is deployed with `network=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (HTTP, as shown in the default config comment).
2. Attacker performs DNS hijacking of `mirror-rest` or ARP spoofing on the cluster network to MITM the HTTP connection.
3. Attacker returns a crafted `/api/v1/network/nodes` JSON response where each `service_endpoints` entry has a non-empty `domain_name` pointing to `attacker.example.com` (attacker-controlled DNS) alongside the legitimate `ip_address_v4`.
4. Lines 114–115: `host` is set to `"attacker.example.com"` because `domain_name` is non-empty; `ip_address_v4` is never consulted.
5. `net.JoinHostPort("attacker.example.com", "50211")` is inserted into the network map.
6. `hiero.ClientForNetwork(netmap)` configures the SDK to dial `attacker.example.com:50211` for all consensus node gRPC calls.
7. The attacker's server (with a valid TLS cert for `attacker.example.com`, obtainable via Let's Encrypt) receives all gRPC traffic.

**Why existing checks are insufficient:**

- `strings.TrimSpace` and the empty-string guard (lines 114–118) only reject blank values — they perform zero semantic validation.
- There is no check that `domain_name` matches any expected pattern, suffix, or corresponds to the `ip_address_v4` in the same endpoint record.
- The HTTP client has no custom `Transport` with certificate pinning or HTTPS enforcement, so HTTP-scheme mirror URLs are fully interceptable.

### Impact Explanation

All gRPC traffic from the pinger to consensus nodes — including smart contract `ContractCallQuery` and `ContractExecuteTransaction` messages — is redirected to the attacker's server. The attacker can:
- Observe transaction payloads and operator signatures in transit (if gRPC TLS is not enforced end-to-end by the SDK for this network)
- Return crafted gRPC responses (e.g., fake `SUCCESS` status, manipulated contract call results) causing the pinger to misreport network health
- Cause systematic transaction failures, breaking liveness monitoring

This aligns with the stated scope: "unintended smart contract behavior with no concrete funds at direct risk" — the pinger's operator key is exposed to the attacker's server and monitoring integrity is broken, but direct fund theft requires additional steps.

### Likelihood Explanation

The `network=other` path is explicitly designed for private/custom deployments where the mirror node REST URL is operator-configured. The default example URL uses the `http://` scheme. In Kubernetes or Docker Compose deployments, DNS for service names like `mirror-rest` is controlled by cluster-internal DNS, which an attacker with pod-level access (a common lateral movement target) can poison. The attack requires no credentials to the pinger itself — only network-level access to the HTTP path between the pinger and the mirror node REST service. The attacker can obtain a valid TLS certificate for their injected domain trivially via public CAs, defeating any SDK-level TLS validation on the gRPC side.

### Recommendation

1. **Prefer `ip_address_v4` over `domain_name`** — reverse the fallback logic so the concrete IP is used when available, and `domain_name` is only used when no IP is present.
2. **Validate `domain_name` against an allowlist** of expected suffixes (e.g., `*.hedera.com`, `*.hiero.org`) before accepting it.
3. **Enforce HTTPS** for the mirror node REST URL: reject `http://` scheme URLs at config load time, or configure the HTTP client with a custom `Transport` that enforces TLS.
4. **Cross-validate**: if both `domain_name` and `ip_address_v4` are present, resolve the domain and assert it matches the IP before using it.
