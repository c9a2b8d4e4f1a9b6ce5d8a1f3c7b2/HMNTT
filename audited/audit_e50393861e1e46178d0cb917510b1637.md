### Title
Silent Node Exclusion via Unused `grpc_proxy_endpoint` in `fetchMirrorNodeNetwork()`

### Summary
In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()` parses `grpc_proxy_endpoint` from each `nodeEntry` but exclusively uses `serviceEndpoints` to build the network map. An attacker who can influence the mirror node REST API response (e.g., via MITM on an unverified HTTP connection, DNS spoofing, or a compromised/malicious mirror node) can craft a payload where ≥30% of nodes carry a valid `grpc_proxy_endpoint` but empty/invalid `serviceEndpoints`, causing those nodes to be silently dropped. The only guard — a zero-length network check — does not prevent partial exclusion.

### Finding Description

**Code location:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–131.

**Struct definition (lines 24–28):** `nodeEntry` declares both `ServiceEndpoints []serviceEndpoint` and `GrpcProxyEndpoint *serviceEndpoint`. Both are deserialized from JSON.

**Network map construction (lines 102–124):** Only `n.ServiceEndpoints` is iterated. `n.GrpcProxyEndpoint` is never read after deserialization:

```go
for _, ep := range n.ServiceEndpoints {   // GrpcProxyEndpoint is never touched
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {
        continue                          // silently skipped
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

**Only guard (lines 127–129):**
```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

This check only fires when **all** nodes are excluded. Any partial exclusion (1–99% of nodes) passes silently.

**Root cause / failed assumption:** The code assumes that `grpc_proxy_endpoint` is supplementary metadata and that every reachable node will always have at least one valid `serviceEndpoints` entry. This assumption is not enforced. An attacker who controls the JSON response can set `service_endpoints: []` (or populate it with entries where both `domain_name` and `ip_address_v4` are empty, or `port` is 0) while providing a valid `grpc_proxy_endpoint`, and the node is silently excluded from the map.

**Exploit flow:**
1. Attacker positions themselves to control the mirror node REST API response. Viable vectors:
   - The `http.Client` at line 46 has no TLS certificate pinning or verification enforcement in this code path; if `cfg.mirrorRest` resolves over plain HTTP or to a self-signed endpoint, a network-level attacker (DNS spoof, ARP poisoning, BGP hijack, or rogue Wi-Fi) can substitute the response.
   - A third-party or self-hosted mirror node (the "other" network case is the only path that calls `buildNetworkFromMirrorNodes`) is operator-chosen; a malicious or compromised mirror node is a realistic threat.
2. Attacker crafts a `nodesEnvelope` JSON response in which ≥30% of `nodeEntry` objects have `service_endpoints: []` (or entries with blank host/zero port) and a non-null `grpc_proxy_endpoint`.
3. `fetchMirrorNodeNetwork()` iterates those nodes, finds no usable `serviceEndpoints`, adds nothing to `network` for them, and moves on — no log, no warning, no error.
4. `len(network) > 0` (the remaining ≤70% of nodes still contribute entries), so the function returns successfully.
5. `hiero.ClientForNetwork(netmap)` is called with the reduced map; the pinger now only monitors the surviving subset of nodes.

### Impact Explanation

The pinger's purpose is to detect network-wide degradation by submitting transactions and measuring node responsiveness. If ≥30% of consensus nodes are absent from the network map, the pinger cannot detect failures on those nodes. An attacker can selectively exclude specific high-value or high-traffic nodes, masking outages or targeted attacks against them. This directly maps to the stated severity tier: "Shutdown of ≥30% of network processing nodes … without shutting down the network." The pinger reports a healthy network while a significant fraction of nodes is unmonitored.

### Likelihood Explanation

The attack applies exclusively to the `"other"` network configuration path (line 17–21 of `sdk_client.go`), which is the only path that calls `buildNetworkFromMirrorNodes`. Operators using custom/private networks are the target population. The HTTP client has no certificate pinning; if the mirror REST URL uses HTTP or an unverified TLS endpoint, a network-adjacent attacker (same datacenter, cloud VPC, or ISP-level) can intercept and rewrite the response. A malicious or compromised mirror node requires no network position at all. The attack is repeatable on every startup or reconnect cycle that re-fetches the node list.

### Recommendation

1. **Use `grpc_proxy_endpoint` as a fallback:** If `serviceEndpoints` is empty or yields no usable addresses, fall back to `grpc_proxy_endpoint` before skipping the node. This closes the silent-drop gap.
2. **Enforce a minimum node count:** After building the map, compare `len(network)` against a configurable threshold (e.g., the previously known node count or a hard floor). Return an error and trigger a retry if the count drops below the threshold.
3. **Verify response integrity:** Pin the TLS certificate or CA for `cfg.mirrorRest`; reject plain-HTTP mirror URLs in production.
4. **Log every skipped node:** Emit a warning whenever a node with a valid `node_account_id` contributes zero addresses to the map, so operators can detect anomalous exclusions.

### Proof of Concept

**Crafted response (4 of 10 nodes have empty `service_endpoints`, valid `grpc_proxy_endpoint`):**

    {"node_account_id":"0.0.8","service_endpoints":[{"domain_name":"node8.example.com","port":50211}],"grpc_proxy_endpoint":null},
