### Title
Unauthenticated Repeated DB Exhaustion via Non-Existent Transaction Hash Lookup with No Negative-Result Caching

### Summary
`extractSqlFromTransactionsByIdOrHashRequest()` in `rest/transactions.js` unconditionally issues a live database query via `getTransactionHash()` for every syntactically valid transaction hash, regardless of whether the hash exists. Negative results (empty DB rows) are never cached, and the REST API layer has no rate limiting. An unprivileged attacker can flood the endpoint with valid-format but non-existent hashes, forcing a DB round-trip on every request and exhausting the database connection pool.

### Finding Description

**Root cause — three compounding failures:**

1. **No negative-result caching.** A `Cache` (Redis) instance is instantiated at line 44 of `rest/transactions.js` (`const cache = new Cache();`) but is **never used** in the hash lookup path. `getTransactionHash()` calls `pool.queryQuietly()` directly on every invocation with no cache read-before or cache write-after.

2. **No rate limiting in the REST layer.** All throttling found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`, bucket4j) belongs exclusively to the **web3 Java API**. The Node.js REST API (`rest/*.js`) has zero rate-limiting middleware.

3. **HTTP cache explicitly disabled for hash queries.** `getTransactionsByIdOrHashCacheControlHeader()` (line 895–898) returns `{}` (no cache-control header) when `isTransactionHash === true`, preventing any upstream HTTP cache from absorbing repeated identical requests.

**DB function cost:** The v2 migration `R__transaction_hash_lookup.sql` implements `get_transaction_info_by_hash` as a two-phase plpgsql function — it first queries recent rows, and if `recent_rows = 0`, issues a **second query** against older data. A non-existent hash therefore causes **two sequential DB queries** per HTTP request.

**Validation check that fails:** `isValidTransactionHash()` (line 44 of `transactionHash.js`) uses the regex `/^([\dA-Za-z+\-\/_]{64}|(0x)?[\dA-Fa-f]{96})$/` — it validates only character set and length, not existence. Any 64-character base64url string passes.

### Impact Explanation
Each request with a valid-format but non-existent hash consumes at least one (up to two) database connections and executes indexed scans against the `transaction_hash` table. Under sustained flood, the PostgreSQL connection pool is exhausted, causing legitimate queries to queue or fail. This degrades or denies service for all users of the mirror node REST API without any economic cost or privilege requirement for the attacker.

### Likelihood Explanation
The attack requires zero authentication, zero on-chain activity, and zero cost. The attacker needs only an HTTP client and the ability to generate 64-character alphanumeric strings. The endpoint is publicly accessible at `GET /api/v1/transactions/:transactionIdOrHash`. The attack is trivially scriptable and repeatable at high frequency from a single machine or botnet.

### Recommendation
1. **Cache negative results:** After `getTransactionHash()` returns an empty array, store a sentinel value in Redis (e.g., `cache.setSingle(hashKey, TTL_SECONDS, null)`) and check it before issuing the DB query. A short TTL (e.g., 30–60 seconds) is sufficient.
2. **Add rate limiting to the REST API:** Introduce per-IP or global request-rate middleware (e.g., `express-rate-limit`) specifically for the `/transactions/:transactionIdOrHash` endpoint.
3. **Re-enable HTTP caching for 404 responses:** Return a `Cache-Control: public, max-age=N` header even on `NotFoundError` responses for hash lookups so that CDN/proxy layers can absorb repeated identical misses.

### Proof of Concept
