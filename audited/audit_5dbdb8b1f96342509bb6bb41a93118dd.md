### Title
Unbounded Subscriber Fan-Out in `SharedTopicListener` Causes Per-Subscriber Heap Buffer Exhaustion

### Summary
`SharedTopicListener.listen()` applies an independent `onBackpressureBuffer` of up to 16,384 `TopicMessage` objects per subscriber invocation against the shared hot Flux. Because no connection limit or per-client throttle exists anywhere in the service layer, an unprivileged attacker can open an arbitrary number of gRPC subscriptions, each accumulating its own full buffer, causing aggregate heap exhaustion proportional to the subscriber count.

### Finding Description

**Code path:**

`SharedTopicListener.listen()` — [1](#0-0) 

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

Every call to `listen()` chains a **new, independent** `onBackpressureBuffer` onto the shared Flux. The default `maxBufferSize` is 16,384. [2](#0-1) 

`SharedPollingTopicListener` holds a single `.share()`-ed Flux that all subscribers multicast from. [3](#0-2) 

`TopicMessageFilter` auto-generates a unique `subscriberId` per filter instance, so each new gRPC call produces a distinct filter object — but this ID is used **only for logging**, not for deduplication or limiting. [4](#0-3) 

`TopicMessageServiceImpl.subscribeTopic()` tracks a `subscriberCount` metric but **never enforces a maximum** — it is a `Gauge` registered for observability only, not a guard. [5](#0-4) 

No rate-limiting, per-IP connection cap, or max-subscriber check exists anywhere in the service or listener layers.

**Root cause:** The failed assumption is that the `.share()` operator on the upstream Flux prevents per-subscriber memory growth. It does prevent duplicate DB polling, but it does **not** prevent each subscriber from independently buffering up to `maxBufferSize` messages in its own Reactor queue.

### Impact Explanation
With `maxBufferSize = 16384` and a `TopicMessage` object size of ~200–400 bytes (timestamps, sequence number, content bytes, etc.), a single subscriber can buffer ~3–6 MB. With 1,000 concurrent subscriptions (easily achievable over gRPC/HTTP2 multiplexing), aggregate buffer memory reaches 3–6 GB, exhausting the JVM heap and causing an OutOfMemoryError that crashes the mirror node gRPC service. This is a non-network-based DoS requiring no special privileges, no authentication, and no valid topic content — only the ability to open gRPC streams.

### Likelihood Explanation
Any client with network access to the gRPC port can open gRPC streaming calls to `subscribeTopic`. The gRPC/HTTP2 protocol allows many streams per TCP connection, so the attacker does not even need many TCP connections. The attack is trivially scriptable, repeatable, and requires no knowledge of valid topic IDs beyond a single valid one (or even an invalid one if `checkTopicExists = false`).

### Recommendation
1. **Enforce a global and per-IP subscriber cap** inside `TopicMessageServiceImpl.subscribeTopic()` using the existing `subscriberCount` atomic, returning a gRPC `RESOURCE_EXHAUSTED` status when exceeded.
2. **Add a configurable `maxSubscribers` property** to `ListenerProperties` or `GrpcProperties` and reject new subscriptions beyond that threshold.
3. **Reduce `maxBufferSize`** to a smaller default (e.g., 512–1024) or make it configurable per deployment.
4. Consider applying **per-client rate limiting** at the Netty/gRPC interceptor layer using `NettyProperties`.

### Proof of Concept
```python
import grpc, threading
# Assumes proto stubs for ConsensusService are generated
channel = grpc.insecure_channel("mirror-node:5600")
stub = ConsensusServiceStub(channel)

def open_subscription():
    req = ConsensusTopicQuery(topic_id=TopicID(topic_num=1))
    for _ in stub.subscribeTopic(req):
        pass  # intentionally slow consumer to fill buffer

threads = [threading.Thread(target=open_subscription) for _ in range(2000)]
for t in threads: t.start()
# Each subscription independently buffers up to 16384 TopicMessage objects
# Aggregate heap: 2000 × 16384 × ~300 bytes ≈ ~9.8 GB → OOM crash
```
