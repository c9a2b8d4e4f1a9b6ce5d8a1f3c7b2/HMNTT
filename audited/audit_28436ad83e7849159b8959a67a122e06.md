### Title
Blocking JDBC Poll on Shared `boundedElastic` Scheduler Enables Thread-Pool Exhaustion DoS in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` schedules each subscriber's periodic `poll()` on the global `Schedulers.boundedElastic()` instance. Because `poll()` executes a synchronous, blocking JDBC call (`getResultList()`) directly on that scheduler thread, an attacker who opens enough concurrent gRPC subscriptions can saturate all available threads in the bounded pool, preventing any further poll tasks from executing and halting message delivery for every subscriber on the node.

### Finding Description

**Exact code path:**

`PollingTopicListener` holds a reference to the global bounded-elastic scheduler: [1](#0-0) 

Every call to `listen()` wires both the initial delay and the repeat timer to that same scheduler: [2](#0-1) 

When the delay fires, the scheduler thread subscribes to `Flux.defer(() -> poll(context))`, which synchronously calls `findByFilter()`: [3](#0-2) 

`findByFilter()` is implemented as a blocking JPA call — `getResultList()` — that holds the calling thread until the database returns all rows (up to `maxPageSize = 5000`): [4](#0-3) 

There is no `subscribeOn` or `publishOn` to offload the blocking work to a separate pool; the scheduler thread is blocked for the full duration of the JDBC round-trip.

**Root cause / failed assumption:** The design assumes `poll()` is non-blocking or fast enough that occupying a `boundedElastic` thread briefly is harmless. In reality, each subscription holds a thread for the entire JDBC query duration on every poll cycle. Because `Schedulers.boundedElastic()` is a JVM-global singleton (default cap: `10 × availableProcessors`, e.g. 40 threads on a 4-core host), a sufficient number of concurrent slow-polling subscriptions exhausts the pool.

**Why existing checks fail:**

`maxConcurrentCallsPerConnection = 5` limits calls *per connection*, not total connections: [5](#0-4) [6](#0-5) 

An attacker simply opens multiple connections. There is no global subscription cap anywhere in `TopicMessageServiceImpl` or `PollingTopicListener`.

`maxPageSize = 5000` caps rows per query but does not bound aggregate thread occupancy across many concurrent subscriptions: [7](#0-6) 

The `safetyCheck` path in `TopicMessageServiceImpl` also uses `Schedulers.boundedElastic()`, adding further contention on the same global pool: [8](#0-7) 

### Impact Explanation
When the bounded-elastic pool is exhausted, no new poll tasks can execute. All active `PollingTopicListener` subscriptions stop delivering messages. Because the scheduler is global, other application components that rely on `Schedulers.boundedElastic()` (e.g., the safety-check path) are also starved. The result is a complete halt of live topic-message delivery for all clients connected to the affected mirror-node instance — matching the "total network shutdown" severity classification for this component.

### Likelihood Explanation
The attack requires:
1. The server is configured with `listenerType = POLL` (not the default `REDIS`).
2. No external rate-limiting (WAF, load-balancer connection limits) is in place.
3. The attacker can open tens of TCP connections to the gRPC port (no authentication required for `subscribeTopic`).

On a 4-core host (40 `boundedElastic` threads), if each JDBC poll takes ~500 ms (plausible for 5 000 rows), each subscription occupies a thread ~50 % of the time. Exhaustion requires ~80 concurrent subscriptions — achievable with 16 connections at 5 calls each. This is trivially scriptable with any gRPC client library and requires no credentials.

### Recommendation
1. **Isolate blocking work**: Wrap the `poll()` call with `.subscribeOn(Schedulers.boundedElastic())` *and* use a **dedicated, separately-sized** `BoundedElasticScheduler` (via `Schedulers.newBoundedElastic(...)`) rather than the global singleton, so subscriber-driven blocking cannot starve other application components.
2. **Enforce a global subscription limit**: Add an `AtomicInteger` counter in `PollingTopicListener` (or at the service layer) and reject new `listen()` calls above a configurable threshold.
3. **Add per-IP / per-client connection rate limiting** at the gRPC server or load-balancer layer to prevent a single client from opening an unbounded number of connections.
4. **Consider async DB access**: Replace the synchronous JPA `getResultList()` with a non-blocking R2DBC query so scheduler threads are never blocked.

### Proof of Concept
```
# Precondition: server running with hiero.mirror.grpc.listener.type=POLL
# Tool: grpcurl or any gRPC client

for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# 20 connections × 5 calls = 100 concurrent subscriptions
# Each poll blocks a boundedElastic thread with a JDBC query
# On a 4-core host (40 threads), pool is exhausted within seconds
# Observe: no topic messages delivered to any subscriber; 
#          new subscriptions' polls never fire
```
