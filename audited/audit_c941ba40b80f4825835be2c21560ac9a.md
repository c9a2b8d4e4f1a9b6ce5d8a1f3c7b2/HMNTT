### Title
Cache Key Collision via `Arrays.hashCode` in `SpelHelper.hashCode(byte[])` Allows Wrong Entity Served for EVM Address Lookups

### Summary
`SpelHelper.hashCode(byte[])` uses `Arrays.hashCode`, which returns a 32-bit `int`, as the sole cache key for byte-array EVM address and alias lookups in `EntityRepository`. Because the key space is only 2^32 while EVM addresses are 20 bytes (2^160 possible values), hash collisions are computationally trivial to find. An unprivileged attacker who finds two on-chain EVM addresses that collide under `Arrays.hashCode` can cause the mirror node's web3 layer to serve the wrong entity record from cache, resulting in incorrect data being returned for contract calls.

### Finding Description

**Exact code path:**

`SpelHelper.hashCode(byte[])` — `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
``` [1](#0-0) 

This method is registered as a Spring bean named `spelHelper` in `CommonConfiguration.spelHelper()` and used as the cache key expression in two `@Cacheable` annotations in `EntityRepository`: [2](#0-1) 

```java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);

@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
``` [3](#0-2) 

Both caches (`CACHE_NAME_EVM_ADDRESS` = `"evmAddress"`, `CACHE_NAME_ALIAS` = `"alias"`) are backed by the same `CACHE_MANAGER_ENTITY` Caffeine cache manager: [4](#0-3) 

**Root cause:** `Arrays.hashCode(byte[])` produces a 32-bit `int` via a simple polynomial rolling hash (`result = 31 * result + (b & 0xff)`). The output space is 2^32 ≈ 4 billion values. For 20-byte EVM addresses (2^160 possible inputs), the birthday bound guarantees a 50% collision probability after only ~65,536 distinct addresses are hashed. The cache stores and retrieves entries using only this 32-bit integer as the key — the original byte array is never stored or compared.

**Failed assumption:** The design assumes `Arrays.hashCode` produces unique keys for distinct byte arrays. It does not. Two distinct byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` will map to the same cache slot, and whichever is cached first will be returned for both.

**Exploit flow:**
1. Attacker enumerates existing EVM addresses on the network via the public REST API.
2. Attacker computes `Arrays.hashCode` for each address and finds two addresses `A` and `B` with the same hash value (trivially achievable with ~65,536 addresses due to birthday bound; on mainnet with millions of accounts, natural collisions are virtually guaranteed).
3. Attacker sends a web3 contract call (e.g., `eth_call`) that triggers `findByEvmAddressAndDeletedIsFalse(A)` — entity A is fetched from DB and stored in cache under key `H`.
4. Attacker sends a second call triggering `findByEvmAddressAndDeletedIsFalse(B)` — cache returns entity A's record instead of querying the DB for entity B.
5. The web3 execution layer proceeds with entity A's data (wrong account ID, wrong balance, wrong keys) for address B.

**Why existing checks are insufficient:** The `unless = "#result == null"` guard only prevents caching null results. It does not verify that the cached value corresponds to the actual input key. There is no equality check between the stored byte array and the incoming byte array at cache retrieval time.

### Impact Explanation

The web3 module uses these cached entity lookups to resolve EVM addresses to Hedera entities during contract execution (`eth_call`, `eth_estimateGas`). Serving the wrong entity for an EVM address means:
- Wrong account balance, type, or key material used in EVM execution
- Incorrect contract call results returned to callers
- Potential for an attacker to make a contract call appear to succeed or fail incorrectly by poisoning the cache with a colliding address they control

This constitutes incorrect records being served from the mirror node's web3 layer, directly matching the stated scope of "incorrect or missing records exported to mirror nodes."

### Likelihood Explanation

The attack requires no privileges — only access to the public web3 API and the public REST API (to enumerate EVM addresses). Finding a collision in `Arrays.hashCode` over 20-byte inputs is trivial: the birthday bound gives ~50% probability after ~65,536 addresses. On Hedera mainnet, which has millions of accounts, natural collisions among existing addresses are virtually certain to exist. The attacker can pre-compute the collision offline and execute the cache-poisoning in two sequential API calls. The attack is repeatable and requires no special tooling.

### Recommendation

Replace the 32-bit `Arrays.hashCode` cache key with the full byte array content as the key. Spring's `SimpleKeyGenerator` and Caffeine handle arbitrary object keys correctly. The simplest fix is to remove the custom `key` expression entirely and let Spring use the default key (which wraps the `byte[]` in a `SimpleKey` and compares by content via `Arrays.equals`), or explicitly use `key = "#alias"` after ensuring the cache implementation compares keys by value. Alternatively, encode the byte array as a hex string for use as the key:

```java
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```

Or update `SpelHelper` to return a collision-resistant representation:
```java
public String hashCode(byte[] value) {
    return java.util.Arrays.toString(value); // or hex encoding
}
```

### Proof of Concept

```java
// Step 1: Find two 20-byte EVM addresses with the same Arrays.hashCode
import java.util.Arrays;
import java.util.HashMap;

public class CollisionFinder {
    public static void main(String[] args) {
        HashMap<Integer, byte[]> seen = new HashMap<>();
        java.util.Random rng = new java.util.Random();
        while (true) {
            byte[] addr = new byte[20];
            rng.nextBytes(addr);
            int h = Arrays.hashCode(addr);
            if (seen.containsKey(h)) {
                byte[] collision = seen.get(h);
                System.out.println("Collision found!");
                System.out.println("A: " + toHex(collision));
                System.out.println("B: " + toHex(addr));
                System.out.println("hashCode(A)=" + h + " hashCode(B)=" + h);
                break;
            }
            seen.put(h, addr);
        }
    }
}
// Typically finds a collision in < 100,000 iterations (seconds of CPU time)

// Step 2: Trigger cache poisoning
// POST /api/v1/contracts/call  { "to": "<address_A_hex>", ... }  → caches entity A under H
// POST /api/v1/contracts/call  { "to": "<address_B_hex>", ... }  → returns entity A's data for address B
```
