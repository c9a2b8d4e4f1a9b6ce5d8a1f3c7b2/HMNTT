### Title
Unauthenticated Thread Pool Exhaustion via Unbounded Concurrent Requests to GET /api/v1/topics/{id}

### Summary
The `TopicController.getTopic()` handler and its backing `TopicServiceImpl.findById()` have no rate limiting, connection throttling, or concurrency cap applied in the `rest-java` module. An unprivileged attacker can open a large number of concurrent HTTP connections to `GET /api/v1/topics/{id}`, causing every servlet worker thread to block on `topicRepository.findById()` DB calls (or on waiting for a HikariCP connection), exhausting the thread pool and rendering the node unresponsive to all other API traffic.

### Finding Description

**Exact code path:**

`TopicController.java` lines 31–37 — the handler is a plain synchronous `@GetMapping` with no throttle guard: [1](#0-0) 

It delegates directly to `TopicServiceImpl.findById()` (lines 19–21), which issues a blocking JDBC call with no concurrency limit: [2](#0-1) 

**Root cause — absent rate limiting in rest-java:**

The only registered servlet filters in the `rest-java` module are `LoggingFilter` and `MetricsFilter`: [3](#0-2) [4](#0-3) 

`RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — no rate-limiting filter: [5](#0-4) 

`WebMvcConfiguration` adds only argument resolvers and formatters: [6](#0-5) 

The rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) exists only in the `web3` module and is never applied to `rest-java` endpoints: [7](#0-6) 

**Exploit flow:**

1. Attacker sends N concurrent `GET /api/v1/topics/0.0.1` requests (no authentication required).
2. Each request occupies one Tomcat worker thread (default pool: 200 threads).
3. Spring Data's `CrudRepository.findById()` issues a blocking JDBC call via HikariCP (default pool: 10 connections).
4. With >10 concurrent requests, threads block waiting for a DB connection (HikariCP default timeout: 30 s).
5. With ~200 concurrent requests, all worker threads are blocked; the node returns 503 or queues/drops all subsequent requests for every endpoint.

### Impact Explanation

All API endpoints on the affected `rest-java` node become unresponsive for the duration of the flood. Because the `rest-java` service is deployed as a horizontally scaled cluster (Kubernetes HPA), an attacker targeting all replicas simultaneously can take down the entire mirror-node REST API surface. The Helm chart allocates only 2 CPU / 2 GiB per pod with no ingress-level rate limiting configured: [8](#0-7) 

This satisfies the ≥30% node-unresponsiveness threshold without requiring any brute-force credential attack.

### Likelihood Explanation

The endpoint requires zero authentication, accepts any valid entity ID, and the `TopicRepository` is a plain `CrudRepository` with no query timeout or concurrency guard: [9](#0-8) 

A single attacker with a modest number of persistent HTTP/1.1 connections (200 suffices) can reproduce this reliably and repeatedly with standard tools (e.g., `wrk`, `ab`, `hey`). No special knowledge of the system is required beyond knowing the public API path.

### Recommendation

1. **Add a rate-limiting servlet filter to `rest-java`** using the existing bucket4j dependency already present in `web3`, keyed per source IP, applied before the handler chain.
2. **Set a Tomcat `max-connections` and `accept-count`** in `application.yml` to bound queue depth.
3. **Configure a HikariCP `connectionTimeout`** (e.g., 5 s) and `maximumPoolSize` appropriate to the expected concurrency, so blocked threads fail fast rather than holding for 30 s.
4. **Add ingress-level rate limiting** (e.g., NGINX `limit_req` or an API gateway policy) as a defense-in-depth layer, consistent with the recommendation already noted in the GraphQL design doc: [10](#0-9)

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
