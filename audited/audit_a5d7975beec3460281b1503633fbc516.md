### Title
Cache Stampede (Thundering Herd) in `SystemFileLoader.load()` Allows Unprivileged DB Query Amplification

### Summary
The `load(FileID, long)` method in `SystemFileLoader` uses a non-atomic check-then-act cache pattern: it reads from the Caffeine cache, and if the key is absent, calls `loadFromDB()` and then writes the result back. No locking or atomic `computeIfAbsent` is used. Multiple concurrent requests that arrive before any system file is cached will all observe a cache miss simultaneously and each independently issue DB queries, amplified up to 10× per file by the built-in retry logic.

### Finding Description
**Exact code path:**

In `web3/src/main/java/org/hiero/mirror/web3/state/SystemFileLoader.java`, the `load()` method at lines 127–136:

```java
var file = cache.get(cacheKey, File.class);   // line 127 – non-atomic read
if (file != null) {
    return file;
}
// The value was not in cache -> try to load from DB
var result = loadFromDB(fileId, consensusTimestamp);  // line 133
if (result != null) {
    cache.put(cacheKey, result);              // line 136 – separate write
}
```

`cache` is a `CaffeineCacheManager`-backed Spring `Cache`. The `get()` and `put()` calls are two separate, non-atomic operations. Caffeine provides an atomic `get(key, mappingFunction)` API, but it is not used here.

**Root cause:** The check-then-act sequence is not guarded by any lock or atomic operation. Between the `cache.get()` returning `null` and the subsequent `cache.put()`, any number of concurrent threads can also observe a miss for the same `CacheKey(fileId, consensusTimestamp)` and each independently enter `loadFromDB()`.

**Retry amplification:** `loadFromDB()` delegates to `loadWithRetry()` (line 148), which uses a `RetryTemplate` configured with `maxRetries(9)` (lines 70–73). If the stored file data is corrupt or unparseable, each `loadFromDB()` invocation issues up to **10 sequential DB queries** (1 initial + 9 retries), each fetching progressively older versions via `fileDataRepository.getFileAtTimestamp()`.

**Entry point:** `FileReadableKVState.readFromDataSource()` (lines 56–57) calls `systemFileLoader.load(key, getCurrentTimestamp())` for every system file lookup during EVM execution. `getCurrentTimestamp()` uses `Instant.now()` (wall-clock nanoseconds), so concurrent requests arriving within the same nanosecond window share the same cache key and all miss simultaneously.

**The 7-file scenario:** The 7 non-networkProperty system files are: `addressBookFile101`, `addressBookFile102`, `feeScheduleFile`, `simpleFeeScheduleFile`, `exchangeRateFile`, `hapiPermissionFile`, `throttleDefinitionFile`. An attacker sends 7 concurrent HTTP requests (well within the 500 req/s rate limit), each crafted to trigger EVM access to a distinct system file. All 7 arrive before any are cached. Each independently misses the cache and calls `loadFromDB()`. With corrupt data in the DB, this yields up to 7 × 10 = **70 DB queries** from a single coordinated burst.

**Why existing checks fail:**
- The global rate limiter (`rateLimitBucket`, 500 req/s) does not prevent a burst of 7 concurrent requests.
- The gas limiter (`gasLimitBucket`) throttles gas consumption, not file-read DB access.
- The cache TTL (10 min for `systemFile`, 60 min for `fee`) only helps after the first successful population; it provides no protection during the initial cold-start window or after cache expiry.

### Impact Explanation
The impact is a burst of uncontrolled DB connections and queries triggered by any unprivileged user. Under the retry amplification path (corrupt/unparseable file data), a single coordinated burst of 7 requests can generate up to 70 DB queries. Repeated at scale (e.g., 500 req/s × 7 files × 10 retries = 35,000 DB queries/s), this can exhaust the DB connection pool, degrade query latency for all users, and cause cascading timeouts. There is no economic damage and no state corruption; the impact is purely availability degradation (griefing).

### Likelihood Explanation
No authentication or special privilege is required. Any user with network access to the web3 JSON-RPC endpoint can trigger this. The attack is trivially repeatable: the attacker simply needs to time requests to arrive during a cache-cold window (e.g., immediately after service startup, or after the 10-minute TTL expires). The 500 req/s global rate limit is a weak mitigant since the attacker only needs 7 concurrent requests per burst, far below the threshold.

### Recommendation
Replace the non-atomic check-then-act pattern with Caffeine's atomic `get(key, mappingFunction)` via the native cache API, or use Spring's `@Cacheable` annotation (which delegates to Caffeine's atomic loader). Concretely, replace lines 127–136 with:

```java
// Atomic: only one thread will call loadFromDB for a given key
return cache.get(cacheKey, File.class, k -> loadFromDB(fileId, consensusTimestamp));
```

This ensures that for any given `CacheKey`, only one thread executes `loadFromDB()` while all other threads wait for the result, eliminating the thundering herd entirely.

Additionally, consider capping `maxRetries` in the `RetryTemplate` or adding a per-file DB query budget to limit the amplification factor when corrupt data is present.

### Proof of Concept
1. Start the mirror-node web3 service with a cold cache (fresh start or wait for 10-minute TTL expiry).
2. Identify 7 contract call payloads that each trigger EVM access to a distinct system file (e.g., a call that reads fee schedules, one that reads exchange rates, one that reads the address book, etc.).
3. Send all 7 HTTP POST requests to `/api/v1/contracts/call` simultaneously (e.g., using `curl --parallel` or a multi-threaded client).
4. Observe in DB query logs (or via `pg_stat_activity`) that 7 concurrent `getFileAtTimestamp` queries are issued simultaneously — one per system file — rather than 1 query followed by 6 cache hits.

    private File loadWithRetry(final FileID key, final long currentTimestamp, SystemFile systemFile) {
        AtomicLong nanoSeconds = new AtomicLong(currentTimestamp);
        final var fileId = toEntityId(key).getId();
        final var attempt = new AtomicInteger(0);

        try {
            return retryTemplate.execute(() -> fileDataRepository
                    .getFileAtTimestamp(fileId, nanoSeconds.get())
                    .filter(fileData -> ArrayUtils.isNotEmpty(fileData.getFileData()))
                    .map(fileData -> {
                        try {
                            var bytes = Bytes.wrap(fileData.getFileData());
                            var codec = systemFile.codec;
                            if (codec != null) {
                                codec.parse(bytes.toReadableSequentialData());
                            }
                            return File.newBuilder().contents(bytes).fileId(key).build();
                        } catch (ParseException e) {
                            log.warn(
                                    "Failed to parse file data for fileId {} at {}, retry attempt {}. Exception: ",
                                    fileId,
                                    nanoSeconds.get(),
                                    attempt.incrementAndGet(),
                                    e);
                            nanoSeconds.set(fileData.getConsensusTimestamp() - 1);
                            throw new InvalidFileException(e);
                        }
                    })
                    .orElse(systemFile.genesisFile()));
        } catch (RetryException e) {
            return systemFile.genesisFile();
        }
    }
