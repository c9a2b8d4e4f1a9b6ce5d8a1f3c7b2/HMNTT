### Title
Unbounded Concurrent Subscriptions with Far-Future `endTime` Exhaust Scheduler Resources via `pastEndTime()` Polling Loops

### Summary
Any unauthenticated client can open an unlimited number of gRPC `subscribeTopic` calls with `endTime` set to a far-future timestamp. Each subscription spawns a `pastEndTime()` Reactor polling loop configured with `Long.MAX_VALUE` repeat iterations and a 30-second fixed delay, plus a `safetyCheck` task on `Schedulers.boundedElastic()`. Because `subscriberCount` is a metric gauge only (not a limiter) and `maxConcurrentCallsPerConnection` limits only calls per TCP connection (not total connections), an attacker with many connections can accumulate thousands of concurrent long-lived timer tasks, degrading the Reactor scheduler for legitimate subscribers.

### Finding Description

**Exact code path:**

`pastEndTime()` at [1](#0-0) 

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```

When `endTime` is non-null, this creates a `Flux.empty()` that re-subscribes up to `Long.MAX_VALUE` times with a fixed delay of `endTimeInterval` (default 30 s). [2](#0-1) 

The termination predicate `!topicContext.isComplete()` only becomes `false` when `endTime + endTimeInterval` is in the past: [3](#0-2) 

With `endTime` set to year 2100 (or `Long.MAX_VALUE`, which `convertTimestamp` explicitly maps to `Long.MAX_VALUE`), `isComplete()` returns `false` for decades. [4](#0-3) 

Each subscription also spawns a `safetyCheck` task on `Schedulers.boundedElastic()`: [5](#0-4) 

**Root cause — failed assumption:** The design assumes subscriptions are naturally short-lived or self-terminating. There is no server-side cap on total concurrent subscriptions, no rate limit on the gRPC endpoint, and no maximum `endTime` value enforced.

**Why existing checks fail:**

- `subscriberCount` is a Micrometer `Gauge` for observability only; it never rejects new subscriptions. [6](#0-5) 
- `maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection, not the total number of connections or total subscriptions across all connections. [7](#0-6) 
- `TopicMessageFilter` validation only requires `endTime > startTime` and `startTime <= now()`; it places no upper bound on `endTime`. [8](#0-7) 
- No authentication or per-IP rate limiting exists on the gRPC endpoint. [9](#0-8) 

### Impact Explanation

Each malicious subscription contributes:
1. One Reactor timer task firing every 30 s (the `pastEndTime` loop), held for the lifetime of the connection.
2. One `boundedElastic` task slot (the `safetyCheck`).
3. One `topicListener.listen()` subscription (Redis pub/sub or polling slot).
4. Memory for `TopicContext`, `AtomicLong`, `AtomicReference`, `Stopwatch`.

With N connections × 5 calls each, the attacker accumulates 5N concurrent long-lived timer tasks. At a few thousand connections the Reactor default scheduler and `boundedElastic` pool (capped at 10 × CPU cores threads, up to 200,000 queued tasks) become saturated, causing scheduling latency for legitimate subscribers. This is a griefing / availability degradation against the mirror node gRPC service with no economic damage to the Hedera network itself.

### Likelihood Explanation

The attack requires only a standard gRPC client (e.g., `grpcurl` or any HCS SDK), no credentials, and no special network position. Opening thousands of TCP connections from a single host or a small botnet is trivial. The attack is repeatable and persistent as long as the attacker keeps connections open. The only natural friction is OS-level TCP connection limits on the attacker's side, which are easily raised.

### Recommendation

1. **Enforce a maximum `endTime` horizon**: Reject or clamp `endTime` values beyond a configurable maximum (e.g., `now + 24h`) in `TopicMessageFilter` validation or in `ConsensusController.toFilter()`.
2. **Cap total concurrent subscriptions**: Add a server-side check in `subscribeTopic()` that rejects new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED`.
3. **Add per-IP or global rate limiting** on the gRPC endpoint (e.g., via a Netty `ChannelHandler` or an API gateway).
4. **Replace `Long.MAX_VALUE`** in `RepeatSpec.create(...)` with a computed bound derived from `(endTime - now) / endTimeInterval + 1` so the repeat count is proportional to the actual remaining window.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw proto
import grpc, threading
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

MIRROR_GRPC = "mirror.node.host:5600"
FAR_FUTURE = Timestamp(seconds=9999999999, nanos=0)  # year ~2286; maps to Long.MAX_VALUE

def open_subscription():
    channel = grpc.insecure_channel(MIRROR_GRPC)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,                    # any valid topic
        consensusStartTime=Timestamp(seconds=0, nanos=0),
        consensusEndTime=FAR_FUTURE,    # triggers pastEndTime() loop
    )
    # Block on the stream; each call holds a pastEndTime() timer for decades
    for _ in stub.subscribeTopic(query):
        pass

# Open many connections, 5 subscriptions each (maxConcurrentCallsPerConnection=5)
threads = [threading.Thread(target=open_subscription) for _ in range(5000)]
for t in threads:
    t.start()
# Result: 5000 concurrent pastEndTime() timers + safetyCheck tasks on boundedElastic
# Legitimate subscribers experience scheduling delays / timeouts
```
