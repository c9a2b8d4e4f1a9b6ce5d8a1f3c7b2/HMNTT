### Title
Cache Key Bypass via Unnormalized `req.originalUrl` Enables Repeated Database Queries

### Summary
`cacheKeyGenerator` in `rest/middleware/responseCacheHandler.js` hashes the raw `req.originalUrl` (including all query parameters as-is) to produce a Redis cache key. Because the `requestNormalizer` module exists but is explicitly **not** wired into the cache key path, semantically identical requests with valid parameters in different orderings or with explicit default values produce distinct cache keys, causing cache misses and repeated database queries. Any unprivileged external user can exploit this with zero authentication.

### Finding Description

**Exact code location:**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator`, lines 151–153: [1](#0-0) 

The comment at lines 148–150 explicitly acknowledges the missing normalization:
> `path?query - In the future, this will utilize Edwin's request normalizer (9113).`

**Root cause — failed assumption:**
The code assumes each semantically unique request will have a unique `req.originalUrl`. In reality, the same logical query can be expressed as many distinct URLs (different parameter orderings, explicit vs. implicit default values), each producing a different MD5 hash and therefore a different Redis key.

**Why the `requestNormalizer` does not help:**
`normalizeRequestQueryParams` in `rest/middleware/requestNormalizer.js` (lines 35–59) correctly normalizes parameter order and fills in defaults, but a `grep` across all production `rest/**/*.js` files confirms it is **never registered as middleware** and is only exercised in tests. [2](#0-1) 

**Why the OpenAPI validator does not help:**
`openApiValidator` is configured with `validateRequests: false` (line 155 of `openapiHandler.js`), so it performs no request-level rejection. [3](#0-2) 

**Middleware ordering in `server.js`:**
`responseCacheCheckHandler` is registered at line 97, before all route handlers (lines 101+). The cache key is therefore computed and the Redis lookup performed before any application-level validation runs. [4](#0-3) 

**Exploit flow using valid parameters (confirmed DB path):**

For `/api/v1/accounts`, `getAccounts` calls `validateReq` first (line 337), so truly unknown parameters return HTTP 400 before a DB query. However, the bypass works cleanly with **valid parameters** — including explicit default values and parameter reorderings — all of which pass `validateReq` and proceed to `pool.queryQuietly`: [5](#0-4) 

The normalizer test suite confirms that `/api/v1/accounts`, `/api/v1/accounts?balance=true`, and `/api/v1/accounts?balance=true&limit=25&order=asc` all normalize to the same canonical URL — but none of that normalization is applied to the cache key. [6](#0-5) 

Additionally, `listStakingRewardsByAccountId` in `accountController.js` calls `EntityService.isValidAccount` (a DB query) **before** `buildAndValidateFilters`, meaning even a request with an unknown parameter triggers a DB round-trip before the 400 is returned: [7](#0-6) 

**`responseCacheUpdateHandler` does not cache 400 responses**, so each bypass attempt remains a permanent cache miss: [8](#0-7) 

### Impact Explanation
An attacker can generate an unbounded number of distinct cache keys for any cached endpoint by varying parameter order or supplying explicit default values (e.g., `balance=true`, `limit=25`, `order=asc`). Each variant bypasses Redis and hits PostgreSQL. For high-cardinality endpoints like `/api/v1/accounts`, `/api/v1/transactions`, or `/api/v1/contracts/results`, each DB query is expensive (full table scans with joins). This constitutes a low-cost, high-impact cache-exhaustion / database DoS vector. The Redis cache — the primary scalability mechanism — is rendered ineffective.

### Likelihood Explanation
No authentication, API key, or special knowledge is required. The attack is trivially scriptable: iterate over all permutations of valid query parameters and their default values. The parameter space for endpoints like `/api/v1/accounts` (6 accepted parameters) yields hundreds of distinct orderings. The attack is repeatable indefinitely and is not rate-limited by any mechanism visible in the codebase.

### Recommendation
Integrate `normalizeRequestQueryParams` into `cacheKeyGenerator` so the cache key is derived from the normalized URL rather than `req.originalUrl`. Specifically:

1. In `responseCacheCheckHandler`, resolve `req.openapi?.openApiRoute` and `req.path` (both available after the OpenAPI validator middleware runs) and call `normalizeRequestQueryParams(openApiRoute, req.path, req.query)` to produce a canonical URL.
2. Hash that canonical URL instead of `req.originalUrl`.
3. Ensure the normalizer is called before the cache key is generated (the OpenAPI validator already populates `req.openapi` at line 63–65 of `server.js`).

This is the fix already planned under issue 9113, as noted in the source comment.

### Proof of Concept

```bash
# Step 1: Warm up the cache with the canonical URL
curl "https://mirror-node/api/v1/accounts"
# → 200, response cached under key = MD5("/api/v1/accounts")

# Step 2: Bypass cache with explicit default value (valid parameter, same DB result)
curl "https://mirror-node/api/v1/accounts?balance=true"
# → cache MISS (key = MD5("/api/v1/accounts?balance=true")), DB queried

# Step 3: Bypass with reordered valid parameters
curl "https://mirror-node/api/v1/accounts?limit=25&order=asc"
# → cache MISS, DB queried

curl "https://mirror-node/api/v1/accounts?order=asc&limit=25"
# → cache MISS (different MD5), DB queried again

# Step 4: Automate to saturate DB connection pool
for i in $(seq 1 1000); do
  curl "https://mirror-node/api/v1/accounts?limit=$((RANDOM % 100 + 1))&order=asc&balance=true" &
done
# Each request is a unique cache key → 1000 concurrent DB queries
```

 * Generate the cache key to access Redis. While Accept-Encoding is specified in the API response Vary
 * header, and therefore that request header value should be used as part of the cache key, the cache
 * implementation stores the response body as the original JSON object without any encoding applied. Thus it
 * is the same regardless of the accept encoding specified, and chosen by the compression middleware.

const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};
