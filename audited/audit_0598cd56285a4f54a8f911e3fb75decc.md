### Title
Retry Budget Exhaustion via Sustained BUSY/PLATFORM_NOT_ACTIVE Status Codes in `submitWithRetry()`

### Summary
`submitWithRetry()` in `pinger/transfer.go` applies a single uniform retry loop to all error types without discriminating between transient network-load signals (`BUSY`, `PLATFORM_NOT_ACTIVE`) and permanent failures. An attacker who can sustain a transaction-flood against the Hiero network long enough to keep returning these status codes will exhaust the pinger's entire retry budget and cause it to return `"all attempts failed"`, creating an effective monitoring blackout from the pinger's perspective with no actual network-layer disruption required.

### Finding Description
**Exact code path:** `pinger/transfer.go`, `submitWithRetry()`, lines 14–59. [1](#0-0) 

The retry loop runs for `cfg.maxRetries + 1` iterations (default: 11). On each iteration:

1. `cryptoTransfer.Execute(client)` is called. If the Hiero SDK's own internal BUSY-retry budget is also exhausted, the error surfaces here.
2. If `Execute` succeeds, `resp.GetReceipt(client)` is called. A `BUSY` or `PLATFORM_NOT_ACTIVE` receipt status is returned as an error from `GetReceipt`. [2](#0-1) 

In both cases the code does:
```go
lastErr = err
log.Printf("attempt %d/%d failed: %v", i, attempts, err)
```
and then sleeps via `backoff()` before the next attempt. There is **no branch** that inspects the error type, checks for `BUSY`/`PLATFORM_NOT_ACTIVE` specifically, or takes any different action. The only early-exit check is `ctx.Err()`, which only fires on process shutdown. [3](#0-2) 

**Failed assumption:** The design assumes that the Hiero SDK's own internal retry logic will absorb transient `BUSY`/`PLATFORM_NOT_ACTIVE` responses before they reach this outer loop. If the SDK's internal budget is also exhausted (or if the status arrives at the receipt layer after a successful `Execute`), the outer loop treats the transient error identically to a permanent one and counts it against the finite retry budget.

**Backoff schedule** (default `baseBackoff=2s`, `maxRetries=10`): [4](#0-3) 

Attempts 1–4 wait 2 s, 4 s, 8 s, 16 s; attempts 5–10 each wait 30 s (capped). Total window an attacker must sustain: ≈ 2+4+8+16+(6×30) = **210 seconds** of continuous BUSY responses. [5](#0-4) 

### Impact Explanation
When all retries are exhausted, `submitWithRetry` returns `fmt.Errorf("all attempts failed: %w", lastErr)`. The caller in `main.go` logs the error and continues to the next tick — it does **not** crash, alert an operator through any out-of-band channel, or distinguish this from a genuine network partition. [6](#0-5) 

The pinger is the liveness/health signal for the mirror-node deployment. Silencing it for the duration of the attack means:
- Monitoring dashboards show the network as unreachable.
- Automated alerting fires as if a real outage occurred.
- Incident responders may take disruptive remediation actions (failover, rollback) against a network that is actually healthy.

Severity: **Medium–High** (integrity of the monitoring plane; no direct fund loss, but can trigger false-positive incident response).

### Likelihood Explanation
**Precondition:** The attacker must be able to cause the Hiero network to return `BUSY` on every transaction submitted by the pinger's operator account for ~210 seconds.

**Feasibility:** `BUSY` is returned when consensus nodes are overloaded. An attacker with enough HBAR to submit a high-volume transaction flood (no special privileges required — just a funded account) can saturate node queues. This is a well-known economic-DoS pattern on public ledgers. The attacker does not need to target the pinger directly; flooding the network broadly is sufficient because the pinger shares the same node pool.

**Repeatability:** The attack can be repeated every monitoring interval, keeping the pinger permanently silenced as long as the flood is sustained. The cost scales with the network's throughput capacity, not with any pinger-specific defense.

### Recommendation
1. **Classify errors before counting retries.** Inspect the returned error for `BUSY`/`PLATFORM_NOT_ACTIVE` status codes and either (a) skip counting them against the retry budget, or (b) apply a separate, longer-lived retry path with jitter to avoid thundering-herd amplification.
2. **Emit a distinct metric/log for transient vs. permanent failures** so monitoring systems can distinguish "network overloaded" from "network partitioned."
3. **Add a per-attempt deadline** via `context.WithTimeout` wrapping each `Execute`+`GetReceipt` call so a hung node cannot consume the entire retry window on a single attempt.
4. **Consider a circuit-breaker pattern**: after N consecutive `BUSY` responses, emit a specific alert rather than silently returning `"all attempts failed"`.

### Proof of Concept
**Preconditions:**
- Attacker has a funded Hiero account on the target network.
- Pinger is configured with default settings (`maxRetries=10`, `baseBackoff=2s`).

**Steps:**
1. Attacker submits a continuous high-volume transaction flood to the Hiero network targeting the same consensus nodes the pinger uses, sufficient to keep node queues saturated and returning `BUSY` for all incoming transactions.
2. On the next pinger tick, `submitWithRetry` calls `cryptoTransfer.Execute(client)`. The SDK's internal BUSY-retry budget is exhausted; the error surfaces to the outer loop.
3. The outer loop retries 10 more times over ~210 seconds. Each attempt receives `BUSY` (or `PLATFORM_NOT_ACTIVE` if the node drops to a non-active state under load).
4. After attempt 11, `submitWithRetry` returns `fmt.Errorf("all attempts failed: BUSY")`.
5. `main.go` logs `transfer failed: all attempts failed: BUSY` and waits for the next tick.
6. Repeat for subsequent ticks while the flood is sustained.

**Observed result:** The pinger continuously reports failure for the duration of the flood. Monitoring systems interpret this as a network partition. No actual network disruption has occurred.

```go
func submitWithRetry(ctx context.Context, client *hiero.Client, cfg config) error {
	toID, err := hiero.AccountIDFromString(cfg.toAccountID)
	if err != nil {
		return fmt.Errorf("invalid destination id: %w", err)
	}

	var lastErr error
	attempts := cfg.maxRetries + 1

	for i := 1; i <= attempts; i++ {
		if err := ctx.Err(); err != nil {
			return err
		}

		start := time.Now()
		cryptoTransfer := hiero.NewTransferTransaction().
			AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
			AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))

		resp, err := cryptoTransfer.Execute(client)
		if err == nil {
			receipt, rerr := resp.GetReceipt(client)
			if rerr == nil {
				log.Printf("transfer success: status=%s txID=%s elapsed=%s",
					receipt.Status.String(), resp.TransactionID.String(), time.Since(start))
				return nil
			}
			err = rerr
		}

		lastErr = err
		log.Printf("attempt %d/%d failed: %v", i, attempts, err)

		if i < attempts {
			sleep := backoff(cfg.baseBackoff, i)
			timer := time.NewTimer(sleep)
			select {
			case <-ctx.Done():
				timer.Stop()
				return ctx.Err()
			case <-timer.C:
			}
		}
	}

	return fmt.Errorf("all attempts failed: %w", lastErr)
```

```go
func backoff(base time.Duration, attempt int) time.Duration {
	d := base * time.Duration(1<<(attempt-1))
	return min(d, 30 * time.Second)
}
```
