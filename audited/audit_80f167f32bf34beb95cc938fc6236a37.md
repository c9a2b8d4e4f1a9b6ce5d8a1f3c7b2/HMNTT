### Title
Unauthenticated Unbounded-Concurrency Prometheus Scrape Endpoint Causes Heap Exhaustion DoS in Importer

### Summary
`ActuatorHttpServer` binds a plain JDK `HttpServer` to all network interfaces (`0.0.0.0:8080`) with no authentication, no rate limiting, and an unbounded `newVirtualThreadPerTaskExecutor()`. Every GET to `/actuator/prometheus` materializes the full Prometheus scrape output as two heap allocations (a `String` from `scrape()` and a `byte[]` from `getBytes()`) that are held concurrently for every in-flight request. A flood of concurrent unauthenticated requests exhausts JVM heap and crashes the importer process.

### Finding Description

**Exact code path:**

`importer/src/main/java/org/hiero/mirror/importer/config/ActuatorHttpServer.java`

- **Line 55**: `HttpServer.create(new InetSocketAddress(port), 0)` — `InetSocketAddress(int port)` binds to the wildcard address `0.0.0.0`, exposing the port on every network interface. Backlog `0` uses the OS default (~50 queued connections), but this only limits the accept queue, not active concurrent connections.
- **Line 60**: `server.setExecutor(Executors.newVirtualThreadPerTaskExecutor())` — creates a new virtual thread per accepted connection with **no upper bound**. Millions of virtual threads can be created before the JVM runs out of heap.
- **Lines 91–93**: The only guard is an HTTP method check (`GET` vs non-`GET`). No authentication, no IP allowlist, no concurrency semaphore, no rate limiter.
- **Line 95**: `final var body = prometheusMeterRegistry.scrape().getBytes(StandardCharsets.UTF_8);` — `scrape()` serializes all registered meters into a `String` (heap allocation #1); `.getBytes()` produces a `byte[]` copy (heap allocation #2). Both objects are live on the virtual thread's stack until `exchange.getResponseBody().write(body)` completes and the `try(exchange)` block closes.

**Root cause / failed assumption:** The implementation assumes the endpoint is only reachable by a trusted Prometheus scraper. There is no enforcement of that assumption: no network-layer restriction in code, no authentication, and no concurrency cap.

**Why existing checks are insufficient:**
- The HTTP method check (line 91) only rejects non-GET requests; it does nothing to limit volume or identity.
- `networkPolicy.enabled: false` is the default in both `charts/hedera-mirror-importer/values.yaml` (line 156) and `charts/hedera-mirror/values.yaml` (line 148), so no Kubernetes NetworkPolicy is deployed by default.
- There is no Kubernetes `Service` resource for the importer (the template directory contains only `serviceaccount.yaml`, `deployment.yaml`, `podmonitor.yaml`, etc. — no `service.yaml`), so Prometheus scrapes via pod IP directly, confirming the port must be reachable at the pod-network level.
- No rate-limiting code exists anywhere in `importer/src/main/java/`.

### Impact Explanation

Each concurrent request holds approximately `2 × scrape_size` bytes of heap (String + byte[]). A production importer with hundreds of JVM, Spring, and custom Hiero metrics can produce a scrape payload of 500 KB–2 MB. With the 5 Gi heap limit configured in `charts/hedera-mirror-importer/values.yaml` (line 453), approximately 2,500–10,000 concurrent requests are sufficient to trigger `OutOfMemoryError`. The importer is a singleton (`replicas: 1`, line 448) responsible for ingesting all Hedera network stream files into the mirror database. Crashing it halts all record, balance, and event stream processing, directly satisfying the stated impact of shutting down ≥30% of network processing nodes.

### Likelihood Explanation

**Preconditions:**
- In bare-metal or VM deployments the port is directly reachable on any network interface.
- In Kubernetes with the default `networkPolicy.enabled: false`, any pod in the cluster (including a compromised or attacker-controlled workload in a multi-tenant cluster) can reach the importer pod IP on port 8080 without any credentials.

**Trigger:** A single attacker machine running `wrk`, `hey`, or a simple script opening thousands of persistent HTTP/1.1 connections and pipelining GET requests is sufficient. Virtual threads are cheap enough that the JVM will accept and dispatch far more concurrent requests than the heap can sustain.

**Repeatability:** The attack is stateless and repeatable. After a crash and pod restart, the endpoint is immediately vulnerable again with no lockout mechanism.

### Recommendation

1. **Bind to loopback only** (strongest fix for non-Kubernetes): change `new InetSocketAddress(port)` to `new InetSocketAddress(InetAddress.getLoopbackAddress(), port)` so the port is never reachable from outside the host.
2. **Add a concurrency semaphore** to `handlePrometheus`: acquire a permit (e.g., `Semaphore(4)`) before calling `scrape()` and return HTTP 429 if the permit is unavailable.
3. **Enable Kubernetes NetworkPolicy** by default (`networkPolicy.enabled: true`) and restrict ingress on the actuator port to only the Prometheus scraper's pod selector.
4. **Stream the response** instead of materializing it: use `PrometheusMeterRegistry.scrape(OutputStream, ...)` (available in recent Micrometer versions) to avoid the double heap allocation.
5. **Add IP allowlisting** or a shared-secret header check as a defense-in-depth layer.

### Proof of Concept

```bash
# From any host that can reach the importer pod/VM on port 8080
# (bare-metal, or any pod in the cluster with default networkPolicy.enabled=false)

# Step 1: confirm unauthenticated access
curl -s http://<importer-ip>:8080/actuator/prometheus | head -5

# Step 2: flood with concurrent scrape requests to exhaust heap
# Using 'hey' (https://github.com/rakyll/hey):
hey -n 100000 -c 5000 http://<importer-ip>:8080/actuator/prometheus

# Expected result:
# - JVM heap fills with concurrent String + byte[] allocations
# - Importer logs OutOfMemoryError or is killed by the container OOM killer
# - Stream file ingestion halts; mirror database falls behind the network
# - Kubernetes restarts the pod (no lockout), making the attack repeatable
```
