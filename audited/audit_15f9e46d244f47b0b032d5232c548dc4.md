### Title
Malicious Block Node Can Cause Repeatable Block Ingestion Blackouts via Controlled Error Cycling

### Summary
A block node operator can deliberately send exactly `maxSubscribeAttempts` error responses to trigger `active=false` on their node, then withhold `serverStatus` responses to prevent the force-readmit fallback from activating, causing a blackout lasting the full `readmitDelay` (minimum 10 seconds, default 1 minute). This cycle is indefinitely repeatable, producing periodic block ingestion outages with no economic cost to the attacker.

### Finding Description

**Exact code path:**

`StreamProperties.java` defines the thresholds: [1](#0-0) 

`BlockNode.onError()` deactivates the node after `maxSubscribeAttempts` consecutive failures: [2](#0-1) 

`BlockNode.streamBlocks()` calls `onError()` on any non-SUCCESS status response: [3](#0-2) 

The intended mitigation is the force-readmit fallback in `BlockNodeSubscriber.getNode()`: [4](#0-3) 

**Root cause and failed assumption:**

The force-readmit fallback calls `hasBlock()`, which in turn calls `node.getBlockRange()` — a live `serverStatus` gRPC call to the attacker's node: [5](#0-4) 

`getBlockRange()` returns `EMPTY_BLOCK_RANGE` on any exception: [6](#0-5) 

The failed assumption is that an inactive node will always respond to `serverStatus`. Because the attacker controls the block node process, they can selectively respond to `serverStatus` (returning empty/no response) while the node is in the deactivated cooldown window, causing `hasBlock()` to return `false`, preventing force-readmit, and making `getNode()` throw: [7](#0-6) 

**Exploit flow:**

1. Attacker's node sends `NOT_AVAILABLE` (or any non-SUCCESS) status on 3 consecutive subscribe calls → `onError()` fires 3 times → `active = false`, `errors` reset to 0, `readmitTime = now + readmitDelay`.
2. `getNode()` is called again: node is inactive → added to `inactiveNodes`.
3. Force-readmit path calls `hasBlock()` → calls `serverStatus` on attacker's node → attacker returns no response or empty range → `EMPTY_BLOCK_RANGE` → `hasBlock()` returns `false`.
4. `getNode()` throws `BlockStreamException("No block node can provide block N")`.
5. In `BLOCK_NODE` source mode, `CompositeBlockSource` has no fallback; it keeps failing every `frequency` interval for the entire `readmitDelay` window.
6. After `readmitDelay` expires, `tryReadmit(false)` succeeds → `active = true`.
7. Attacker resumes `serverStatus` responses → `hasBlock()` returns `true` → streaming resumes.
8. Attacker immediately sends 3 more error responses → repeat from step 1.

**Why existing checks are insufficient:**

- The force-readmit fallback (lines 122–128) is the primary defense, but it is entirely dependent on the attacker's node cooperating with `serverStatus` — which the attacker controls.
- The `CompositeBlockSource` `AUTO` mode fallback to `BlockFileSource` after 3 `BlockNodeSubscriber` failures is a partial mitigation, but it only applies when `sourceType=AUTO` and a file source is available. In `BLOCK_NODE` mode there is no fallback. [8](#0-7) 
- There is no rate-limiting, jitter, or exponential backoff on how quickly the error counter can be re-accumulated after a readmit.

### Impact Explanation

In `BLOCK_NODE` source mode (or `AUTO` mode when the file source is also unavailable), a single malicious configured block node can cause the importer to stop ingesting blocks for `readmitDelay` per cycle (minimum 10 seconds, default 1 minute). The cycle repeats indefinitely with no cost to the attacker. This is a denial-of-service against block ingestion, causing the mirror node's data to fall behind the chain by an unbounded amount over time.

### Likelihood Explanation

Block nodes are auto-discovered from `registeredNodeRepository` (populated from on-chain registered nodes): [9](#0-8) 

Any operator of a registered block node that is auto-discovered by the importer can perform this attack without any additional privilege. The attack requires only the ability to control the gRPC responses of one's own block node — a trivial capability for any node operator. The attack is repeatable indefinitely and requires no special tooling beyond a modified block node implementation.

### Recommendation

1. **Exponential backoff on re-accumulation**: After a node is readmitted, require a minimum number of successful blocks before the error counter can trigger deactivation again. This prevents immediate re-cycling.
2. **Decouple force-readmit from `serverStatus`**: The force-readmit fallback should not rely on a live call to the attacker's node. Instead, use a time-based readmit (`readmitDelay` elapsed) unconditionally when no other node is available, rather than gating it on `hasBlock()`.
3. **Permanent deactivation after repeated cycles**: Track how many times a node has been deactivated within a rolling window. After N deactivation cycles, permanently remove the node from the active pool until operator intervention.
4. **Jitter/backoff on `readmitDelay`**: Apply exponential backoff to `readmitDelay` on repeated deactivations of the same node, making the attack progressively less effective.

### Proof of Concept

**Preconditions:**
- Importer configured with `sourceType=BLOCK_NODE` and a single block node pointing to the attacker's server.
- `maxSubscribeAttempts=3` (default), `readmitDelay=PT1M` (default).

**Steps:**

1. Attacker's block node server responds to `subscribeBlockStream` with `SubscribeStreamResponse.Code.NOT_AVAILABLE` for the first 3 calls.
2. Observe: `BlockNode.onError()` fires 3 times → `active=false`, `readmitTime = now + 60s`.
3. Attacker's server stops responding to `serverStatus` (connection refused or timeout).
4. Observe: `getNode()` calls `hasBlock()` → `getBlockRange()` times out → returns `EMPTY_BLOCK_RANGE` → `hasBlock()` returns `false` → `getNode()` throws `BlockStreamException`.
5. Observe: Importer logs errors every `frequency` interval for 60 seconds. No blocks are ingested.
6. After 60 seconds: `tryReadmit(false)` succeeds → `active=true`.
7. Attacker's server resumes `serverStatus` responses and immediately sends `NOT_AVAILABLE` on the next 3 subscribe calls.
8. Observe: Cycle repeats from step 2. Block ingestion experiences a 60-second blackout every ~60 seconds indefinitely.

    private static boolean hasBlock(final AtomicLong nextBlockNumber, final BlockNode node) {
        final var blockRange = node.getBlockRange();
        if (blockRange.isEmpty()) {
            return false;
        }

        if (nextBlockNumber.get() == EARLIEST_AVAILABLE_BLOCK_NUMBER) {
            nextBlockNumber.set(blockRange.lowerEndpoint());
            return true;
        } else {
            return blockRange.contains(nextBlockNumber.get());
        }
    }
