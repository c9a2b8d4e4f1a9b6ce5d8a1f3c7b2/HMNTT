### Title
Unbounded Historical `startTime` Enables Reconnection-Storm DB Amplification in `PollingTopicListener.poll()`

### Summary
An unprivileged gRPC subscriber can set `startTime` to any past timestamp (including epoch 0) because `TopicMessageFilter.isValidStartTime()` only rejects future timestamps. Each time the client disconnects and reconnects (e.g., after a network partition), a new `PollingContext` is created with `last = null`, resetting the cursor to the original historical `startTime`. Every subsequent poll then issues a database query fetching up to `maxPageSize` (5000) rows. With many concurrent reconnecting clients, this produces a coordinated database amplification storm.

### Finding Description

**Validation gap — `TopicMessageFilter.java` lines 48–51:**
```java
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```
Only an upper bound (≤ now) is enforced. There is no lower bound. `startTime = 0` (nanoseconds, i.e., 1970-01-01) is accepted.

**Cursor reset on reconnection — `PollingTopicListener.java` lines 34–48:**
```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);   // new context per subscription
    ...
    .doOnNext(context::onNext);
}
```
`PollingContext` is instantiated once per `listen()` call. `last` starts as `null`. When a gRPC client disconnects and reconnects, the gRPC service layer calls `listen()` again, producing a fresh `PollingContext` with `last = null`.

**Poll logic — `PollingTopicListener.java` lines 51–62:**
```java
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessage last = context.getLast();
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // up to 5000
    long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
    var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
}
```
When `last == null` (always true on a fresh subscription), `startTime` falls back to `filter.getStartTime()` — the attacker-controlled historical value. The query fetches up to 5000 rows from that historical position.

**No rate limiting or per-client subscription cap exists anywhere in the reviewed code path.**

### Impact Explanation
A topic with years of accumulated messages can have millions of rows. Each reconnecting client triggers a query scanning from `startTime=0` with a 5000-row page. With N concurrent clients reconnecting simultaneously after a partition recovery:
- N × 5000 rows fetched per poll cycle (every 500 ms minimum)
- Database I/O, connection pool, and memory pressure spike sharply
- Legitimate subscribers are starved; the database may become unresponsive
- The attacker does not need any credentials — only a valid `topicId` and a gRPC connection

Severity: **High** (availability impact on shared infrastructure).

### Likelihood Explanation
- Requires only a standard gRPC `subscribeTopic` call with `startTime` set to a very old timestamp — no authentication or privilege needed
- Network partitions are common in cloud deployments; clients routinely reconnect automatically
- A single attacker with a script opening many parallel subscriptions (each with `startTime=0`) and repeatedly disconnecting can trigger the storm deliberately, without waiting for a real partition
- The attack is repeatable and stateless from the attacker's perspective

### Recommendation
1. **Enforce a minimum `startTime` floor** in `TopicMessageFilter.isValidStartTime()`, e.g., reject timestamps older than a configurable retention window (e.g., 30 days).
2. **Persist the cursor across reconnections** at the service/session layer so a reconnecting client resumes from the last acknowledged position rather than the original `startTime`.
3. **Rate-limit subscriptions per client IP/identity** and cap the number of concurrent open subscriptions per topic.
4. **Add a back-pressure / slow-start mechanism** on reconnection: delay the first poll after reconnect by a jittered interval proportional to the number of concurrent reconnecting clients (circuit-breaker pattern).
5. Consider reducing the default `maxPageSize` or making it configurable per-topic based on message volume.

### Proof of Concept
```

# No credentials required

for i in $(seq 1 200); do
  grpc_client subscribeTopic \
    --topicId=T \
    --startTime=0 \          # epoch 0, accepted by isValidStartTime()
    --no-limit &             # no limit → pageSize = maxPageSize = 5000
done
