### Title
Connection Pool Exhaustion via Unbounded `findByIndex` / `findByHash` DB Queries During Network Partition

### Summary
`RecordFileServiceImpl.findByBlockType()` routes numeric block requests to `recordFileRepository.findByIndex()` (Caffeine-cached per block number) and hash requests to `recordFileRepository.findByHash()` (uncached). During a network partition between the web3 service and the database, TCP-level hangs bypass both the PostgreSQL `statementTimeout` and the application-level `StatementInspector`, causing each cache-missing request to hold a HikariCP connection indefinitely. An unprivileged attacker can exhaust the connection pool in under one second by sending requests with distinct block numbers or distinct hashes, denying service to all legitimate users for the duration of the partition.

### Finding Description

**Code path:**

`RecordFileServiceImpl.findByBlockType()` (line 28) → `recordFileRepository.findByIndex(block.number())` for numeric blocks; (line 25) → `recordFileRepository.findByHash(block.name())` for hash blocks. [1](#0-0) 

**Cache for `findByIndex`:**

`@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")` — the cache manager is configured with `expireAfterWrite(10min)`, `maximumSize(10000)`. [2](#0-1) [3](#0-2) 

The `unless = "#result == null"` condition does NOT prevent caching of `Optional.empty()` (an empty Optional is not null), so the first request for a non-existent block number is cached. However, each *distinct* block number causes exactly one DB query before being cached. An attacker cycling through `latestBlock+1`, `latestBlock+2`, … generates a continuous stream of cache-missing DB queries.

**No cache for `findByHash`:**

`findByHash()` carries no `@Cacheable` annotation at all — every hash request unconditionally hits the database. [4](#0-3) 

**Why timeouts fail during TCP partition:**

- `hiero.mirror.web3.db.statementTimeout = 3000 ms` is a PostgreSQL session-level `statement_timeout`. During a TCP-level partition the query never reaches PostgreSQL, so this timeout never fires.
- The `StatementInspector` in `HibernateConfiguration` checks elapsed time only immediately before issuing SQL to an already-acquired connection. If the connection acquisition itself blocks (waiting for a free slot in the HikariCP pool), the inspector is never invoked. [5](#0-4) 

**Rate limiting is global, not per-IP:**

`ThrottleManagerImpl` enforces a single shared `rateLimitBucket` of 500 req/s across all callers. A single attacker can consume the entire budget. [6](#0-5) [7](#0-6) 

### Impact Explanation

With HikariCP's default pool size of 10 connections and TCP-level hangs lasting minutes (OS TCP timeout), the attacker needs only 10 requests with distinct block numbers to fill the pool. At 500 req/s allowed, this takes ~20 ms. Once the pool is exhausted, every subsequent request — including legitimate ones — waits up to HikariCP's `connectionTimeout` (default 30 s) and then fails with a connection-acquisition exception. The service is effectively unavailable for the duration of the partition, and recovery is delayed because hung connections are not released until the OS-level TCP timeout fires.

### Likelihood Explanation

The precondition (network partition) is external, but once it occurs the exploit is trivially reproducible by any unauthenticated HTTP client. No credentials, no special knowledge, and no elevated gas budget are required — the attacker simply sends `eth_call` or `eth_getBlockByNumber` JSON-RPC requests with incrementing block numbers or arbitrary 32-byte hex hashes. The global rate limiter does not distinguish attacker traffic from legitimate traffic, so the attacker can consume the full 500 req/s budget alone.

### Recommendation

1. **Apply TCP-level socket timeouts on the JDBC connection.** Configure HikariCP's `connectionTimeout`, `socketTimeout` (via JDBC URL parameter `socketTimeout=5`), and `keepaliveTime` so that partitioned connections are detected and evicted promptly rather than hanging for OS TCP timeout.
2. **Validate block numbers against the latest known block before issuing a DB query.** Cache the latest block index (already done for `LATEST` with a 500 ms TTL) and reject or short-circuit requests for block numbers beyond `latestIndex + N` without touching the database.
3. **Add a cache for `findByHash`.** Apply the same `@Cacheable` pattern used by `findByIndex` to `findByHash` so that repeated hash lookups do not unconditionally hit the database.
4. **Implement per-source-IP rate limiting** upstream (e.g., at the ingress/load-balancer layer) to prevent a single client from consuming the entire global request budget.
5. **Set a HikariCP `maximumPoolSize` ceiling** explicitly in the web3 configuration and tune it in conjunction with the above socket timeout so that pool exhaustion is bounded and connections are recycled quickly.

### Proof of Concept

```
# Precondition: network partition between web3 and DB is active.
# Attacker sends 15 concurrent requests with distinct block numbers beyond latest.

LATEST=12345678   # obtain from eth_blockNumber

for i in $(seq 1 15); do
  curl -s -X POST http://web3-host:8545 \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_getBlockByNumber\",
         \"params\":[\"0x$(printf '%x' $((LATEST + i)))\",false],
         \"id\":$i}" &
done
wait

# Each request routes to findByBlockType() → findByIndex(latestBlock+i).
# Each distinct index is a cache miss → DB query → TCP hang → connection held.
# After 10 requests the HikariCP pool is full.
# All subsequent legitimate requests block for 30 s then fail.
```
