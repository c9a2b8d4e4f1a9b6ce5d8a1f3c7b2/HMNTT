### Title
Unauthenticated Hash-Only `/block` Requests Cause Unbounded DB Read Flood with No Default Rate Limiting

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` unconditionally issues a raw SQL query to the database for every `/block` request containing only a `Hash` (with `Index` nil). There is no application-level rate limiting, no block-lookup cache, and the Traefik middleware chain (which includes `rateLimit` and `inFlightReq`) is **disabled by default** (`global.middleware: false`). An unprivileged attacker can flood the endpoint with random hashes, saturating DB read capacity and halting transaction confirmation.

### Finding Description

**Exact code path:**

1. `Block()` in `rosetta/app/services/block_service.go` (line 51) calls `s.RetrieveBlock(ctx, request.BlockIdentifier)`. [1](#0-0) 

2. `RetrieveBlock()` in `rosetta/app/services/base_service.go` (lines 84–86): when `bIdentifier.Index == nil && bIdentifier.Hash != nil`, it calls `b.blockRepo.FindByHash(ctx, h)` with no guard. [2](#0-1) 

3. `FindByHash()` in `rosetta/app/persistence/block.go` (lines 135–145) only rejects an empty string; any non-empty hash proceeds to `findBlockByHash()`. [3](#0-2) 

4. `findBlockByHash()` (lines 226–241) executes `selectByHashWithIndex` — a raw SQL query with a **correlated subquery** — on every call, with no caching. [4](#0-3) 

5. The SQL query itself (`selectByHashWithIndex`, lines 34–45) includes a correlated subquery against `record_file`, making each lookup more expensive than a simple indexed read. [5](#0-4) 

**Root cause — failed assumption:** The design assumes an external rate-limiting layer (Traefik middleware) will throttle requests. However, `global.middleware` defaults to `false` in `charts/hedera-mirror-rosetta/values.yaml` (line 95), so the Traefik `Middleware` CRD is **never rendered** in a default deployment. [6](#0-5) 

The middleware template gate confirms this: the entire chain (including `rateLimit: average: 10` and `inFlightReq: amount: 5`) is skipped unless `global.middleware` is explicitly set to `true`. [7](#0-6) 

**No application-level protection:** `main.go` wraps the router only with `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting, no concurrency cap, no block-lookup cache. [8](#0-7) 

The `entityCache` inside `blockAPIService` caches only account aliases, not block lookups. [9](#0-8) 

**Retry amplification (when middleware IS enabled):** The `retry: attempts: 3` middleware entry means each request for a non-existent hash (which returns an error) is retried 3 times, tripling DB load even in the protected configuration. [10](#0-9) 

### Impact Explanation
Each flood request triggers at least one correlated SQL query against `record_file`. At sufficient request rate, the DB connection pool is exhausted and read I/O is saturated. Because the mirror node's transaction ingestion pipeline also reads from the same DB, this directly halts the node's ability to confirm new transactions — matching the "total network shutdown" severity classification. The impact is not limited to the Rosetta API; it degrades the shared database for all mirror node services.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The attacker needs only an HTTP client and knowledge of the Rosetta API spec (public documentation). Requests can be sent from a single host or distributed. The default deployment ships with `global.middleware: false`, meaning the vast majority of real-world deployments are unprotected. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Enable middleware by default**: Change `global.middleware` from `false` to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `rateLimit` and `inFlightReq` Traefik middlewares are active out of the box.
2. **Add application-level rate limiting**: Implement a per-IP token-bucket or semaphore in the Go HTTP handler chain (e.g., `golang.org/x/time/rate`) independent of the infrastructure layer.
3. **Cache block-by-hash lookups**: Add a short-lived (e.g., 5–10 s) in-process LRU cache for `FindByHash` results, similar to the existing `entityCache`, to absorb repeated identical hash queries.
4. **Remove or reorder the retry middleware**: The `retry: attempts: 3` entry should not apply to read endpoints that return 4xx/5xx for invalid input, as it amplifies attacker-controlled load.

### Proof of Concept

```bash
# Flood /block with hash-only requests (Index omitted), each triggering a DB query.
# Replace HOST with the rosetta endpoint.

for i in $(seq 1 10000); do
  curl -s -X POST http://HOST/block \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"block_identifier\": {
        \"hash\": \"$(openssl rand -hex 48)\"
      }
    }" &
done
wait
```

Each request with a random 48-byte hex hash passes the non-empty check in `FindByHash`, executes the correlated `selectByHashWithIndex` SQL query, and returns `ErrBlockNotFound`. With no rate limiting active in the default deployment, this loop saturates the DB read path. Monitor `pg_stat_activity` on the mirror node's PostgreSQL instance to observe connection exhaustion.

func (br *blockRepository) FindByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
	if hash == "" {
		return nil, hErrors.ErrInvalidArgument
	}

	if err := br.initGenesisRecordFile(ctx); err != nil {
		return nil, err
	}

	return br.findBlockByHash(ctx, hash)
}

func (br *blockRepository) findBlockByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	if rb.Index < br.genesisBlock.Index {
		log.Errorf("The block with hash %s is before the genesis block", hash)
		return nil, hErrors.ErrBlockNotFound
	}

	return rb.ToBlock(br.genesisBlock), nil
}
