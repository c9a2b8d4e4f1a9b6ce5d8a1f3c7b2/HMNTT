### Title
Unauthenticated Alias-Triggered Uncached DB Lookup Exhausts REST API Connection Pool (DoS)

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` accepts any valid account alias string as a path parameter and unconditionally executes a live database query against the `entity` table via `EntityService.getAccountFromAlias()`. Because the REST API's `EntityService` performs no caching for alias-to-ID resolution and the default DB connection pool is only 10 connections, an unauthenticated attacker flooding the endpoint with many distinct valid alias strings can saturate the pool and starve all other REST API queries — including transaction-related endpoints — of database connections.

### Finding Description

**Exact code path:**

`getAccountCryptoAllowances()` at [1](#0-0)  calls `EntityService.getEncodedId()` with the raw path parameter.

Inside `getEncodedId()` at [2](#0-1) , when `EntityId.isValidEntityId()` returns false but `AccountAlias.isValid()` returns true (line 125–126), it calls `getAccountIdFromAlias()` → `getAccountFromAlias()`, which executes:

```sql
select id from entity
where coalesce(deleted, false) <> true
  and alias = $1
``` [3](#0-2) 

This query runs via `super.getRows()` → `pool.queryQuietly()` at [4](#0-3) , which acquires a connection from the global `pool` object.

**Root cause — no caching for alias lookups:**

The `entityId` cache in `rest/entityId.js` (lines 301–333) is a `quickLru` instance used only by `parseCached()` for entity ID string parsing (shard.realm.num / encoded IDs). [5](#0-4)  It is never consulted in the alias resolution path inside `entityService.js`. Every alias-bearing request unconditionally hits the database.

**Root cause — no rate limiting on the REST API:**

The Express middleware stack in `rest/server.js` registers: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response cache handlers. [6](#0-5)  There is no rate-limiting middleware. The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the web3 module and is not applied to the REST API. [7](#0-6) 

**Root cause — tiny default connection pool:**

The REST API DB pool defaults to `maxConnections = 10` and `statementTimeout = 20000 ms`. [8](#0-7)  The pool is initialized directly from this config: [9](#0-8) 

**Alias validity — trivially satisfied:**

`AccountAlias.isValid()` accepts any string matching `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`. [10](#0-9)  An attacker can generate an unbounded number of syntactically valid, distinct alias strings (e.g., `AAAAAAAAAA`, `AAAAAAAAAB`, …) with no account on the network.

### Impact Explanation

With 10 pool connections and a `connectionTimeout` of 20 seconds, an attacker holding all 10 connections busy with alias lookups causes every other REST API query — including `/api/v1/transactions`, `/api/v1/accounts`, and all other endpoints sharing the same pool — to queue and eventually time out. The mirror node REST API becomes fully unavailable to legitimate users. While the Hedera consensus network itself continues to process transactions independently, the mirror node is the primary read interface for dApps, wallets, and explorers; its unavailability constitutes a significant service disruption. Severity: **High** (REST API DoS, not consensus-layer shutdown).

### Likelihood Explanation

No authentication is required. The endpoint is publicly reachable. Generating thousands of syntactically valid alias strings requires no special knowledge or credentials — a simple loop over base-32 character combinations suffices. The attack is trivially scriptable, repeatable, and requires only a standard HTTP client. A single attacker with modest bandwidth can sustain the flood indefinitely.

### Recommendation

1. **Add an in-process cache for alias-to-ID resolution** in `EntityService.getAccountFromAlias()`, keyed on the alias bytes, with a TTL matching `hiero.mirror.rest.cache.entityId.maxAge` (default 1800 s). This eliminates the DB round-trip for repeated or concurrent requests for the same alias.
2. **Add rate-limiting middleware** to the REST API Express application (e.g., `express-rate-limit`) to cap requests per IP per second, consistent with the web3 API's `ThrottleManagerImpl` pattern.
3. **Increase the default `maxConnections`** or configure separate pools for alias-resolution queries vs. data-retrieval queries so that alias floods cannot starve transaction queries.
4. **Validate alias existence before DB lookup** — consider returning 404 immediately for aliases that fail a lightweight in-memory bloom filter of known aliases.

const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
const noShardRealmAccountAliasRegex = /^[A-Z2-7]+$/;
const {common} = getMirrorConfig();

class AccountAlias {
  /**
   * Creates an AccountAlias object.
   * @param {string|null} shard
   * @param {string|null} realm
   * @param {string} base32Alias
   */
  constructor(shard, realm, base32Alias) {
    this.shard = AccountAlias.validate(shard, common.shard, 'shard');
    this.realm = AccountAlias.validate(realm, common.realm, 'realm');
    this.alias = base32.decode(base32Alias);
    this.base32Alias = base32Alias;
  }

  static validate(num, configured, name) {
    if (!isNil(num) && BigInt(num) !== configured) {
      throw new InvalidArgumentError(`Unsupported ${name} ${num}`);
    }
    return configured;
  }

  /**
   * Checks if the accountAlias string is valid
   * @param {string} accountAlias
   * @param {boolean} noShardRealm If shard realm is allowed as a part of the alias.
   * @return {boolean}
   */
  static isValid(accountAlias, noShardRealm = false) {
    const regex = noShardRealm ? noShardRealmAccountAliasRegex : accountAliasRegex;
    return typeof accountAlias === 'string' && regex.test(accountAlias);
