### Title
Hardcoded `sslmode=disable` in Rosetta Postgres DSN Exposes Balance Data to Network Interception

### Summary
`GetDsn()` in `rosetta/app/config/types.go` unconditionally appends `sslmode=disable` to the Postgres connection string, meaning all traffic between the Rosetta service and the database is transmitted in plaintext with no TLS and no server certificate verification. An attacker with network-level access to the path between the Rosetta application and the Postgres server â€” requiring no privileges on either system â€” can intercept or tamper with query results, including balance data returned by `RetrieveBalanceAtBlock`, before it reaches the Rosetta API layer and is served to downstream consumers such as exchanges.

### Finding Description
**Exact code location:**

- `rosetta/app/config/types.go`, lines 49â€“58: `GetDsn()` hardcodes `sslmode=disable` with no configurable override.
- `rosetta/app/db/db.go`, line 18: `ConnectToDb()` calls `dbConfig.GetDsn()` unconditionally, passing the result directly to `gorm.Open`.
- `rosetta/app/persistence/account.go`, lines 161â€“222: `RetrieveBalanceAtBlock()` issues SQL queries over this unencrypted connection and returns `hbarAmount` values.
- `rosetta/app/services/account_service.go`, line 61: `AccountBalance()` calls `RetrieveBalanceAtBlock` and returns the result directly in the Rosetta HTTP response.

**Root cause:** The `sslmode=disable` parameter is a compile-time constant embedded in the DSN format string. There is no field in the `Db` struct for SSL mode, no environment variable override, and no runtime configuration path that can enable TLS. The `Db` struct (`types.go` lines 39â€“47) has no `SslMode` field, unlike the importer's `DBProperties.java` which at least defines `SslMode` (even if defaulting to `DISABLE`).

**Exploit flow:**
1. Attacker positions themselves on the network path between the Rosetta Go service and the Postgres instance (e.g., same Kubernetes pod network, same VPC subnet, compromised sidecar, or ARP poisoning on a flat network).
2. Because `sslmode=disable` is set, the Postgres wire protocol is sent in plaintext â€” the client never negotiates TLS, so there is nothing to downgrade; the connection is simply unencrypted from the start.
3. The attacker intercepts the TCP stream and modifies the Postgres wire-protocol response to `RetrieveBalanceAtBlock`'s queries, substituting an inflated `hbarAmount.Value`.
4. The tampered value propagates through `AccountBalance()` â†’ `balances.ToRosetta()` â†’ the HTTP response body with no integrity check.
5. A downstream exchange that credits funds based on the Rosetta `/account/balance` response credits the attacker's account with the inflated amount.

**Why existing checks are insufficient:** There are no application-layer integrity checks on the values returned from the database. The Rosetta response is constructed directly from the raw DB result. No HMAC, no cross-validation against a second source, no anomaly detection on returned balances.

### Impact Explanation
Direct loss of funds is possible if any exchange or custodian credits balances based on the Rosetta API response without independent on-chain verification. The attacker controls the exact balance value returned, so the inflation can be set to any amount. The `sslmode=disable` setting is not overridable without a code change, meaning every deployment of this service is affected regardless of operator configuration.

### Likelihood Explanation
The precondition is network-level access to the internal path between the Rosetta service and Postgres â€” not privileged OS access to either host. In common deployment topologies (Kubernetes flat pod network, shared VPC, Docker bridge network, cloud-managed databases with no private link), this is achievable by any attacker who has compromised any workload on the same network segment. No credentials, no root access, and no application-layer exploit are required. The attack is repeatable and silent; the Rosetta service logs no anomaly.

### Recommendation
1. Add an `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a default of `require` or `verify-full`.
2. Update `GetDsn()` to interpolate the configurable `SslMode` value instead of the hardcoded literal `sslmode=disable`.
3. For production deployments, set `sslmode=verify-full` and supply `sslrootcert` pointing to the CA certificate of the Postgres server.
4. Add a startup check that refuses to connect if `sslmode=disable` is set in a non-development environment.
