### Title
Unauthenticated DB Round-Trip via Non-TOPIC EntityId in `topicExists()` Before Type Validation

### Summary
Any unauthenticated gRPC client can send a `ConsensusTopicQuery` referencing a valid shard/realm/num that maps to an existing non-TOPIC entity (e.g., an ACCOUNT or CONTRACT). The `topicExists()` method unconditionally executes a database lookup via `entityRepository.findById()` before checking the entity type, then throws `IllegalArgumentException("Not a valid topic")`. Because the Caffeine entity cache only caches found entities (not misses), an attacker enumerating distinct non-TOPIC entity IDs forces one DB round-trip per unique ID, and non-existent IDs are never cached, forcing a DB round-trip on every request.

### Finding Description
**Code path:**

- `ConsensusController.toFilter()` ( [1](#0-0) ) builds a `TopicMessageFilter` by calling `EntityId.of(query.getTopicID())` with no type pre-validation.
- `TopicMessageServiceImpl.subscribeTopic()` then calls `topicExists(filter)` ( [2](#0-1) ).
- Inside `topicExists()`, `entityRepository.findById(topicId.getId())` is called unconditionally before any type check ( [3](#0-2) ).
- Only after the DB query does `.filter(e -> e.getType() == EntityType.TOPIC)` run; if the entity is not a TOPIC, `switchIfEmpty` emits `IllegalArgumentException("Not a valid topic")`.

**Caching behavior:**

`EntityRepository.findById()` is annotated `@Cacheable(unless = "#result == null")` ( [4](#0-3) ). Spring Data unwraps `Optional` before applying `unless`, so:
- **Existing non-TOPIC entity**: cached for 24 hours after the first hit — only 1 DB round-trip per unique entity ID.
- **Non-existent entity ID**: `Optional.empty()` unwraps to `null`, `unless = "#result == null"` prevents caching — **every request** for a non-existent ID hits the DB.

The entity cache is a Caffeine cache with a configurable `maximumSize` and a fixed 24-hour TTL ( [5](#0-4) ). An attacker cycling through many distinct entity IDs can saturate the cache, causing evictions and re-querying.

**Root cause:** The type check is performed *after* the DB lookup. There is no pre-validation of the entity type at the controller layer before the service layer issues the query.

### Impact Explanation
The direct impact is forced DB round-trips on every `subscribeTopic` call with a non-existent entity ID, and one DB round-trip per unique non-TOPIC entity ID (then cached). With no rate limiting visible on the gRPC endpoint, a single unauthenticated client can sustain a stream of requests with rotating entity IDs, generating sustained DB load. The impact is bounded to resource exhaustion / griefing — no data is leaked and no user funds are at risk. Severity is **Low**.

### Likelihood Explanation
No authentication or rate limiting is required. Any network-reachable client can open a gRPC stream to `subscribeTopic`. Entity IDs are sequential integers, making enumeration trivial. The attack is fully repeatable and requires no special knowledge beyond the gRPC API schema.

### Recommendation
1. **Pre-validate entity type at the filter layer**: In `toFilter()` or immediately before calling `topicExists()`, reject requests where the provided ID is known to not be a TOPIC without issuing a DB query — or add a dedicated repository method that queries `WHERE id = ? AND type = 'TOPIC'` in a single query.
2. **Cache negative results**: Configure the entity cache to also cache `Optional.empty()` (i.e., remove `unless = "#result == null"` or use a sentinel value), so repeated lookups for non-existent IDs are served from cache rather than hitting the DB on every request.
3. **Add gRPC-level rate limiting**: Apply per-client rate limiting on `subscribeTopic` to bound the request rate from any single unauthenticated caller.

### Proof of Concept
1. Identify any non-TOPIC entity ID in the network (e.g., account `0.0.1234`).
2. Open a gRPC connection to the mirror node's gRPC port (no credentials needed).
3. Call `subscribeTopic` with `topicID = {shardNum: 0, realmNum: 0, topicNum: 1234}`.
4. Observe: `entityRepository.findById(1234)` executes a DB query; the entity is found but has type `ACCOUNT`; `IllegalArgumentException("Not a valid topic")` is returned to the caller.
5. Repeat step 3 with a different `topicNum` that does not exist in the DB at all — observe a DB round-trip on **every** such request (no cache hit).
6. Automate with a loop over a range of `topicNum` values to generate sustained DB load with zero privileges.

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
