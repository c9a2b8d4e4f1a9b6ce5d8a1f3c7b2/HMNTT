### Title
Unauthenticated Token Allowance Enumeration Enabling REST API Database Exhaustion (DoS)

### Summary
The `/accounts/:idOrAliasOrEvmAddress/allowances/tokens` endpoint in `rest/controllers/tokenAllowanceController.js` requires no authentication and has no per-IP or per-request rate limiting in the REST API layer. Any unprivileged external user can iterate through account IDs at arbitrary speed, issuing unlimited database queries against the `token_allowance` table, saturating the shared PostgreSQL connection pool and degrading or denying service to all REST API consumers and the importer. Note: mirror nodes do not participate in gossip — they are read-only archive nodes — so the "gossip prevention" framing in the question is architecturally incorrect; the real impact is REST API DoS and potential importer write starvation.

### Finding Description

**Exact code path:**

`rest/controllers/tokenAllowanceController.js`, `getAccountTokenAllowances()`, lines 68–81: [1](#0-0) 

The handler unconditionally calls `EntityService.getEncodedId()` and `TokenAllowanceService.getAccountTokenAllowances()` — both hit the database — for every incoming request with no credential check.

**Authentication middleware — does not block unauthenticated requests:**

`rest/middleware/authHandler.js`, lines 15–20: [2](#0-1) 

When no `Authorization` header is present, `authHandler` simply `return`s, allowing the request to proceed. It only sets a custom response `limit` for authenticated users; it never rejects unauthenticated requests.

**No rate limiting in the REST API server:**

`rest/server.js`, lines 67–98: [3](#0-2) 

The middleware chain is: `urlencoded → json → cors → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler → route handlers`. There is no IP-based rate limiter, no request-per-second throttle, and no connection limiter for the REST API. The only throttling in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` module for EVM contract calls. [4](#0-3) 

**Route registration — no auth guard:**

`rest/routes/accountRoute.js`, line 18: [5](#0-4) 

**Database query issued per request:**

`rest/service/tokenAllowanceService.js`, lines 86–90: [6](#0-5) 

Each request executes a `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 …` query, potentially as a multi-part UNION ALL, against the live PostgreSQL instance.

**Root cause:** The `authHandler` middleware is purely opt-in (it only enriches authenticated sessions with a higher limit) and provides zero access control. No rate-limiting middleware exists in the REST API stack. The endpoint is fully open to the internet by design, but without any request-frequency guard.

### Impact Explanation

An attacker can issue thousands of requests per second from a single host (or millions from a botnet) iterating `0.0.1`, `0.0.2`, … `0.0.N` for all known account IDs. Each request consumes a PostgreSQL connection from the shared pool (configured `max_connections = 600` per the database docs) and executes a table scan or index lookup on `token_allowance`. [7](#0-6) 

Sustained load can:
1. Exhaust the DB connection pool, causing all REST API endpoints to queue or time out.
2. Starve the importer's write path, which shares the same PostgreSQL instance, delaying ingestion of new consensus data.
3. Cause CPU/IO saturation on the database server, degrading response times for all users.

Severity: **High** for publicly-exposed mirror node deployments. The mirror node does not gossip (it is a read-only archive); the importer ingests from cloud storage independently, but both share the database.

### Likelihood Explanation

- **Precondition:** None. No account, no token, no credentials required.
- **Knowledge required:** Account IDs on Hedera are sequential integers starting from 1; they are fully enumerable.
- **Tooling:** A simple `for` loop with `curl` or any HTTP client suffices. No exploit framework needed.
- **Repeatability:** Indefinitely repeatable; no lockout or backoff mechanism exists.
- **Real-world feasibility:** High. Public mirror node endpoints (`mainnet-public.mirrornode.hedera.com`) are internet-accessible. This is a standard unauthenticated enumeration + resource exhaustion pattern.

### Recommendation

1. **Add per-IP rate limiting** to the REST API middleware stack (e.g., `express-rate-limit` or an upstream Traefik/nginx rate-limit rule) before route handlers are reached.
2. **Set a global request-per-second cap** for unauthenticated callers, analogous to the `ThrottleManagerImpl` pattern already used in the `web3` module.
3. **Enforce a database statement timeout** (`statementTimeout` in `db.pool` config) to bound the maximum duration of any single query, limiting the blast radius of slow enumeration queries.
4. **Consider requiring authentication** (or at minimum an API key) for high-frequency access to allowance endpoints, consistent with the existing `users` config mechanism.

### Proof of Concept

```bash
# Enumerate token allowances for accounts 1 through 10000 with no credentials
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.${i}/allowances/tokens" &
done
wait
```

Each iteration triggers a live `SELECT` against `token_allowance` with `owner = <encoded_id>`. Running this in parallel (e.g., with `xargs -P 100`) from a single host is sufficient to saturate the default DB connection pool of 600 connections, causing HTTP 500 / timeout responses for all other API consumers.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
