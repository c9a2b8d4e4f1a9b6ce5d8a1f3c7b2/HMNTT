### Title
Integer Overflow in `Bound.getInclusiveRangeValue()` Enables Full Table Scan via GT + Long.MAX_VALUE

### Summary
`Bound.getInclusiveRangeValue(boolean upper)` in `rest-java/src/main/java/org/hiero/mirror/restjava/service/Bound.java` performs unchecked `value += 1L` when the operator is `GT`. An unprivileged user can supply `gt:9223372036854775807` as a query parameter, causing the arithmetic to silently overflow to `Long.MIN_VALUE`. The resulting SQL condition (`WHERE token_id >= -9223372036854775808`) eliminates the filter entirely, forcing a full table scan on every such request. Repeated requests can exhaust database connection pools and degrade service availability.

### Finding Description

**Exact code location — `Bound.java` lines 107–118:**
```java
public long getInclusiveRangeValue(boolean upper) {
    var rangeParameter = upper ? this.getUpper() : this.getLower();
    var operator = rangeParameter.operator();
    long value = rangeParameter.value();
    if (operator == GT) {
        value += 1L;          // ← no overflow guard
    } else if (operator == LT) {
        value -= 1L;
    }
    return value;
}
``` [1](#0-0) 

**Root cause — input validation accepts `Long.MAX_VALUE`.**
`NumberRangeParameter.getNumberValue()` only rejects negative values:
```java
private static long getNumberValue(String number) {
    var value = Long.parseLong(number);
    if (value < 0) {
        throw new IllegalArgumentException("Invalid range value");
    }
    return value;
}
```
`Long.MAX_VALUE` (9223372036854775807) is positive, so it passes. The same gap exists in `EntityIdRangeParameter.getEntityId()` which only filters `n >= 0`. [2](#0-1) 

**Why the constructor guard fails.**
`Bound`'s constructor calls `getAdjustedLowerRangeValue()` (which also does `lowerBound++` without overflow protection) and then checks:
```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}
```
After overflow, `adjustedLower = Long.MIN_VALUE` and `adjustedUpper = Long.MAX_VALUE` (default). `Long.MIN_VALUE > Long.MAX_VALUE` is **false**, so no exception is thrown and the overflowed value silently propagates. [3](#0-2) [4](#0-3) 

**SQL generation path.**
`NftAllowanceRepositoryCustomImpl.findAll()` calls `getBoundConditions(bounds)` (defined in `JooqRepository.java`, which is the sole other caller of `getInclusiveRangeValue`). The overflowed `Long.MIN_VALUE` is bound directly into the jOOQ condition, producing:
```sql
WHERE token_id >= -9223372036854775808
```
which matches every row. The query still applies `LIMIT`, but the database must scan the entire `nft_allowance` table to satisfy it. [5](#0-4) 

### Impact Explanation
Every request carrying `token.id=gt:9223372036854775807` triggers a sequential full-table scan of `nft_allowance` instead of an indexed range scan. On a production node with millions of allowance rows, each such query holds a database connection for an extended period. An attacker sending a moderate volume of concurrent requests can exhaust the connection pool, causing legitimate API requests to queue or fail. The `LIMIT` clause prevents unbounded data return but does not prevent the scan itself.

### Likelihood Explanation
The endpoint is unauthenticated (standard Hedera mirror-node REST API). The payload is a single query-string parameter (`token.id=gt:9223372036854775807`) requiring no special privileges, tokens, or knowledge of internal state. The attack is trivially repeatable with any HTTP client and is not rate-limited at the arithmetic layer. Any external party aware of the API can reproduce it.

### Recommendation
1. **Add overflow guard in `getInclusiveRangeValue` and `getAdjustedLowerRangeValue`:** Use `Math.addExact` / `Math.subtractExact` and catch `ArithmeticException`, or clamp to `Long.MAX_VALUE` / `Long.MIN_VALUE` before returning.
2. **Cap accepted values at a domain-meaningful maximum** in `NumberRangeParameter.getNumberValue()` and `EntityIdRangeParameter.getEntityId()` (e.g., `Long.MAX_VALUE - 1` or the maximum valid entity-ID encoding).
3. **Add a post-overflow sanity check in `Bound`:** After computing `adjustedLower`, verify it is still `>= 0` and `<= Long.MAX_VALUE - 1` before proceeding.
