### Title
Unauthenticated Max-Limit Contract Logs Request Chains Two Maximum-Cost DB Queries via Unbounded Timestamp Array

### Summary
An unprivileged external user can send a single `GET /api/v1/contracts/{id}/results/logs?limit=100` request that triggers two sequential, maximum-cost database queries: one for contract logs and one for record file block details. The `getContractLogs()` function unconditionally passes a deduplicated timestamp array of up to `limit` entries directly into `RecordFileService.getRecordFileBlockDetailsFromTimestampArray()`, which executes a correlated subquery once per unique timestamp. With no rate limiting in the REST layer, this chain is freely repeatable by any unauthenticated caller.

### Finding Description

**Code path:**

`rest/service/contractService.js`, `getContractLogs()`, lines 376–393:

```js
async getContractLogs(query) {
  const [sqlQuery, params] = this.getContractLogsQuery(query);
  const rows = await super.getRows(sqlQuery, params);   // DB query 1
  if (rows.length === 0) { return rows; }

  const timestamps = [];
  rows.forEach((row) => {                               // dedup loop
    if (row.consensus_timestamp !== timestamps[timestamps.length - 1]) {
      timestamps.push(row.consensus_timestamp);
    }
  });
  const recordFileMap =
    await RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps); // DB query 2
  return rows.map((cr) => new ContractLog(cr, recordFileMap.get(cr.consensus_timestamp)));
}
```

**Root cause:** The deduplication at lines 383–389 only removes *consecutive* duplicates (it is an adjacency check, not a set). When all returned rows carry distinct `consensus_timestamp` values — which is the normal case when a contract emits one log per transaction — the `timestamps` array grows to exactly `rows.length`, i.e., the full `limit`. This array is then passed without any size guard to `RecordFileService.getRecordFileBlockDetailsFromTimestampArray()`.

**Second query cost:** `rest/service/recordFileService.js`, lines 23–43:

```sql
where consensus_end in (
  select (
    select consensus_end from record_file
    where consensus_end >= timestamp and ...
    order by consensus_end limit 1
  ) as consensus_end
  from (select unnest($1::bigint[]) as timestamp) as tmp
  group by consensus_end
)
```

`unnest($1::bigint[])` expands the full timestamps array into rows; for each row the planner executes a correlated index scan on `record_file`. With 100 unique timestamps this is 100 correlated index scans inside a single query, in addition to the outer scan.

**Failed assumption:** The code assumes that real-world data will have many logs sharing the same `consensus_timestamp` (multiple logs per transaction), so the deduplicated array will be much smaller than `limit`. In practice, contracts that emit exactly one log per transaction — or when the user filters to a single topic — produce one unique timestamp per row, collapsing the deduplication to a no-op.

**Existing checks reviewed:**
- `getLimitParamValue()` (`rest/utils.js`, lines 544–553) caps `limit` at `responseLimit.max` (100 for unauthenticated users). This is the *ceiling* of the attack, not a mitigation.
- No rate limiting exists in the REST layer (grep across `rest/**/*.js` finds no `rateLimit`/`throttle` middleware).
- No guard between lines 389 and 390 checks `timestamps.length` before issuing the second query.

### Impact Explanation

Each crafted request forces the database to execute:
1. A UNION-based contract-log query with `LIMIT 100`.
2. A correlated-subquery record-file lookup with 100 unnested timestamps (100 index scans).

Both queries run at their theoretical maximum cost in a single HTTP round-trip. An attacker sending N concurrent requests multiplies this cost by N. Because the second query's cost scales linearly with unique timestamps and is entirely user-controlled (via `limit=100`), the attacker achieves a meaningful amplification ratio over a default-limit (25-row) request. On a node already serving normal traffic, a modest flood of such requests can push DB CPU and I/O well above the 30% threshold cited, degrading service for all users.

### Likelihood Explanation

- **No privilege required:** The endpoint is publicly accessible; no API key, authentication header, or special role is needed.
- **No special data required:** Any active contract on the network that has emitted ≥100 logs across distinct transactions satisfies the precondition. On mainnet this is trivially true for popular DeFi contracts.
- **Trivially repeatable:** A single `curl` loop or any HTTP load tool suffices. The attacker does not need to write to the chain.
- **No observable side-effect:** The requests are read-only and indistinguishable from legitimate high-limit queries in access logs.

### Recommendation

1. **Cap the timestamp array before the second query.** Insert a guard immediately after the dedup loop:
   ```js
   if (timestamps.length > MAX_RECORD_FILE_LOOKUP) {
     // batch or reject
   }
   ```
2. **Batch the second query.** If large arrays are legitimate, split `timestamps` into fixed-size chunks (e.g., 10) and issue multiple smaller queries rather than one correlated query with 100 entries.
3. **Add rate limiting.** Apply a per-IP or per-user request-rate limit on `/contracts/*/results/logs` at the middleware layer.
4. **Rewrite the correlated subquery.** Replace the `unnest`+correlated-subquery pattern in `recordFileBlockDetailsFromTimestampArrayQuery` with a lateral join or a single range scan bounded by `min(timestamps)` and `max(timestamps)`, eliminating the per-timestamp index scan.

### Proof of Concept

**Precondition:** A contract `0.0.X` exists on the network with ≥100 contract log entries, each in a distinct transaction (unique `consensus_timestamp`). This is true for any active mainnet contract.

**Steps:**

```bash
# Step 1 – single max-cost request (verify two DB queries fire at full cost)
curl "https://<mirror-node>/api/v1/contracts/0.0.X/results/logs?limit=100"

# Step 2 – amplification loop (no authentication, no special tooling)
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.X/results/logs?limit=100" &
done
wait
```

**Expected result:** Each request triggers DB query 1 (contract logs, LIMIT 100) followed immediately by DB query 2 (record file correlated subquery with 100 timestamps).
