### Title
Cache-Bypass DB Amplification via Uncached Null Timestamps in `findByTimestamp` with Global-Only Rate Limiting

### Summary
The `recordFileTimestamp` cache configured in `cacheManagerRecordFileTimestamp()` uses `unless = "#result == null"`, meaning any timestamp that resolves to no record file is never stored in cache. An unauthenticated attacker can continuously send `eth_call` requests referencing timestamps beyond the latest record file (or any gap timestamp), causing every request to unconditionally hit the database. The global (not per-IP) rate limiter at 500 req/s and a bypassable gas throttle provide no meaningful per-user protection, allowing a single attacker to sustain the full 500 DB queries/second indefinitely.

### Finding Description

**Code path:**

`EvmConfiguration.java` lines 166–176 — cache configured with `expireAfterAccess(10, MINUTES)`, `maximumSize(10000)`: [1](#0-0) 

`RecordFileRepository.java` lines 38–46 — `@Cacheable` with `unless = "#result == null"` and a `@CachePut` side-effect: [2](#0-1) 

**Root cause — failed assumption:** The cache annotation assumes all meaningful timestamps will produce a non-null result and be cached. But for any timestamp `T` where `T > max(consensusEnd)` across all record files, the JPQL query `select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1` returns an empty `Optional` (null for caching purposes). The `unless = "#result == null"` guard then prevents the result from ever being stored, so every subsequent request for any such timestamp re-executes the full DB query.

**Gas throttle bypass:** `ThrottleProperties.scaleGas()` returns `0` for any `gas ≤ 10,000` (the `GAS_SCALE_FACTOR`), meaning `gasLimitBucket.tryConsume(0)` always succeeds: [3](#0-2) 

**Global-only rate limiter:** A single shared `rateLimitBucket` (default 500 req/s) is used for all clients with no per-IP partitioning: [4](#0-3) 

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` with `block` set to a numeric timestamp value beyond the chain tip (e.g., `99999999999999999999`) and `gas` set to `≤ 10000`.
2. `ThrottleManagerImpl.throttle()` passes: gas bucket consumes 0 tokens, rate bucket consumes 1 of 500.
3. The request resolves to a block timestamp and calls `RecordFileRepository.findByTimestamp(timestamp)`.
4. The DB query returns null; `unless = "#result == null"` prevents caching.
5. Attacker repeats at up to 500 req/s — every single request executes the DB range scan.
6. Legitimate traffic is simultaneously starved of the shared 500 req/s budget.

### Impact Explanation
At 500 req/s of cache-bypassing DB queries, on a system where the `record_file` table is the primary indexed lookup for all historical EVM calls, this represents a sustained, unbounded increase in DB I/O. On a lightly-to-moderately loaded node (baseline of ~1,000–1,500 DB queries/second across all caches), 500 additional forced queries/second easily exceeds the 30% threshold. The `@CachePut` side-effect on `CACHE_MANAGER_RECORD_FILE_INDEX` also never fires for null results, compounding the miss rate for index-based lookups. [2](#0-1) 

### Likelihood Explanation
No privileges, API keys, or accounts are required. The attack requires only HTTP access to the public `/api/v1/contracts/call` endpoint. The gas bypass (`gas=1`) is trivially set. The attacker needs no knowledge of internal state — any timestamp larger than the current chain tip works permanently. The attack is fully repeatable and stateless; it can be sustained from a single IP or distributed across many. [5](#0-4) 

### Recommendation
1. **Cache negative results explicitly:** Replace `unless = "#result == null"` with a sentinel/negative-cache pattern (e.g., store a known-empty marker with a short TTL such as 30 seconds) so repeated requests for the same invalid timestamp are served from cache rather than hitting the DB.
2. **Add per-IP rate limiting:** Partition the `rateLimitBucket` by client IP (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on `X-Forwarded-For` / `RemoteAddr`) so a single attacker cannot consume the entire global budget.
3. **Fix gas throttle floor:** The `scaleGas` function should enforce a minimum token cost of 1 for any non-zero gas value, preventing the gas bucket from being bypassed with `gas ≤ 10,000`. [6](#0-5) 

### Proof of Concept

# Step 2: Use a timestamp far beyond chain tip (never in record_file table)
# Step 3: Loop at full rate — every request hits DB, none are cached

ENDPOINT="http://<mirror-node-host>/api/v1/contracts/call"
FUTURE_TS="0x$(printf '%x' 99999999999999999)"  # block param as hex timestamp

for i in $(seq 1 500); do
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "{
      \"to\": \"0x0000000000000000000000000000000000000167\",
      \"data\": \"0x\",
      \"gas\": 1,
      \"block\": \"$FUTURE_TS\"
    }" &
done
wait

# Expected: 500 concurrent DB queries for findByTimestamp, 0 cache hits,
# DB I/O increases proportionally. Repeat every second to sustain.
