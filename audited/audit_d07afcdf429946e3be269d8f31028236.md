### Title
Unbounded Per-Subscription DB Polling in `PollingTopicListener.listen()` Enables Database CPU Exhaustion via Unauthenticated gRPC Subscriptions

### Summary
When the gRPC listener is configured with `type = POLL`, `PollingTopicListener.listen()` creates an independent, infinite database polling loop for every active subscription using `RepeatSpec.times(Long.MAX_VALUE)` with a fixed 500ms interval. There is no global subscription count gate, no per-IP connection limit, and no authentication on the `subscribeTopic` endpoint. An unprivileged attacker opening N concurrent connections with 5 calls each generates 10N DB queries per second indefinitely, exhausting database CPU and denying service to all legitimate users.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 34–48:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // line 40 — infinite repeat
                .jitter(0.1)
                .withFixedDelay(interval)              // default 500ms
                .withScheduler(scheduler))
```

Each call to `listen()` creates a new, independent `PollingContext` and schedules `poll(context)` to execute every ~500ms forever. `poll()` (lines 51–62) issues one `topicMessageRepository.findByFilter(newFilter)` DB query per invocation. There is no shared state between subscriptions — each subscription independently hammers the database.

**Root cause — failed assumption:** The design assumes the number of concurrent `POLL`-type subscriptions will be small and operator-controlled. In reality, `subscribeTopic` is a public, unauthenticated gRPC endpoint. The only per-connection cap is `maxConcurrentCallsPerConnection = 5` (enforced in `GrpcConfiguration.java` line 33), which limits calls *per TCP connection* but places no bound on the number of TCP connections an attacker can open. `TopicMessageServiceImpl` tracks `subscriberCount` (line 48) as a Prometheus gauge only — it is never checked as a gate before accepting a new subscription (lines 87–91 show it is only incremented/decremented, never compared to a limit).

**Exploit flow:**
1. Attacker identifies any valid topic ID (all topics are public on Hedera).
2. Attacker opens M TCP connections to port 5600.
3. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs with `limit=0` and no `consensusEndTime` (both are valid per the proto definition and `TopicMessageFilter` validation).
4. Each of the 5M subscriptions independently calls `poll()` every ~500ms → 2 DB queries/second per subscription → 10M DB queries/second total.
5. Subscriptions persist indefinitely; the attacker holds connections open with no cost.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java` line 14): per-connection only; attacker opens more connections.
- GCP `maxRatePerEndpoint: 250` (`values.yaml` line 69): rate-limits new *requests per second*, not sustained concurrent long-lived streaming connections. Once established, a gRPC streaming call is a single request that stays open.
- `subscriberCount` (`TopicMessageServiceImpl.java` line 48): metric gauge only, no enforcement.
- No authentication or authorization on `ConsensusController.subscribeTopic()` (lines 43–53).
- `@DurationMin(millis = 50)` on interval: prevents reducing the interval below 50ms but does not limit subscription count.

### Impact Explanation
With `type = POLL` active, a single attacker machine can sustain thousands of concurrent gRPC streaming connections (HTTP/2 multiplexing makes this cheap). Each subscription independently issues DB queries. The database connection pool is finite; at saturation, legitimate queries queue and time out. The `db.statementTimeout = 10000ms` means queries pile up before timing out, compounding CPU pressure. The result is complete unavailability of the HCS `subscribeTopic` service for all legitimate clients — a non-network-based DoS against a public mirror node serving the Hedera network.

### Likelihood Explanation
Preconditions are minimal: network access to port 5600 (public), any valid topic ID (trivially discoverable via the REST API), and `type = POLL` configured. POLL mode is a documented, supported production option used when Redis is unavailable or undesired. The attack requires no credentials, no special knowledge, and is trivially scriptable with any gRPC client library. It is repeatable and sustained with no per-request cost to the attacker.

### Recommendation
1. **Enforce a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection limiting** at the Netty/load-balancer layer (e.g., `maxConnectionsPerIp` in Netty or an ingress policy).
3. **Require authentication** (e.g., API key or mTLS) for `subscribeTopic` in deployments using `type = POLL`.
4. **Switch default and recommended mode to `SHARED_POLL`** when Redis is unavailable, since `SharedPollingTopicListener` uses a single shared poll loop regardless of subscriber count, eliminating the per-subscription DB query amplification.
5. Add a `@Max` bound on `hiero.mirror.grpc.listener.interval` to prevent operators from accidentally setting an extremely short interval.

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
