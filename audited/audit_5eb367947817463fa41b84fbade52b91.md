### Title
Missing Non-Negative Validation on EVM Long-Zero Address Entity Num Allows Cross-Shard Entity Confusion

### Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` detects a "long-zero" EVM address by checking that bytes 0–3 (shard) and bytes 4–11 (realm) are zero, then passes bytes 12–19 raw via `buffer.getLong()` directly to `entityRepository.findById()` without validating the value is non-negative. Because the entity `id` column stores a bit-packed encoded long that is legitimately negative for entities with non-zero shard/realm values, a crafted address with bytes 12–19 set to a negative encoded ID will resolve to a completely different entity than the address implies.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-3 = shard, bytes 4-11 = realm
        return entityRepository.findById(buffer.getLong())         // bytes 12-19 — NO non-negative check
                .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

**Root cause:** The "long-zero" branch is intended to resolve `0.0.<num>` entity IDs. The code correctly verifies shard (bytes 0–3) and realm (bytes 4–11) are zero, but it never validates that the extracted `num` (bytes 12–19) is ≥ 0 before passing it to `findById()`.

**Why negative IDs exist in the database:** `EntityId.encode()` packs shard (10 bits), realm (16 bits), and num (38 bits) into a single `long`. Entities with non-zero shard or realm produce negative encoded IDs by design:

- `EntityId.of(1023, 65535, 274877906943)` → encoded id = **-1**
- `EntityId.of(512, 0, 0)` → encoded id = **-9223372036854775808**

This is confirmed by `EntityIdTest.java` lines 31–32 and `EntityRepositoryTest.java` line 112 (`domainBuilder.entity(-2, ...)`), which explicitly persists and retrieves entities with negative IDs.

**Exploit flow:**
1. Attacker crafts hex EVM address: `000000000000000000000000FFFFFFFFFFFFFFFF`
   - Bytes 0–3: `00000000` → `buffer.getInt() == 0` ✓
   - Bytes 4–11: `0000000000000000` → `buffer.getLong() == 0` ✓
   - Bytes 12–19: `FFFFFFFFFFFFFFFF` → `buffer.getLong() == -1L`
2. The long-zero branch is taken.
3. `entityRepository.findById(-1L)` executes `SELECT * FROM entity WHERE id = -1`.
4. This returns the entity `1023.65535.274877906943` — a completely different entity from what the address format implies.

**Existing checks shown insufficient:**

- The GraphQL schema annotation `@Pattern(regexp = "^(0x)?[a-fA-F0-9]{40}$")` (account.graphqls line 96) only validates hex format and length — it does not constrain the semantic value of bytes 12–19.
- `decodeEvmAddress()` in `GraphQlUtils.java` (lines 87–98) only strips the `0x` prefix and hex-decodes — no range validation.
- `EntityId.encode()` validates `num >= 0`, but it is **never called** in this code path; the raw `long` from `ByteBuffer` bypasses it entirely.

### Impact Explanation
An unprivileged attacker can resolve any entity with a non-zero shard or realm (i.e., any entity whose encoded ID is negative) by supplying a crafted "long-zero" EVM address. The GraphQL response will present that entity's account/contract data as if it were the entity at `0.0.<negative-num>`, which is not a valid entity. This constitutes entity confusion: the caller receives authoritative-looking Hashgraph state for an entity that does not correspond to the supplied address. In multi-shard/multi-realm deployments (which the codebase explicitly supports), this allows an attacker to enumerate and retrieve any entity by its encoded ID rather than its EVM address, bypassing the intended address-to-entity resolution semantics.

### Likelihood Explanation
No privileges are required. The attack requires only the ability to send a GraphQL query with a crafted hex string — a capability available to any unauthenticated user of the public mirror node API. The encoded IDs of target entities are deterministically computable from their `shard.realm.num` values using the documented encoding formula. The attack is trivially repeatable and scriptable. It is constrained to deployments where entities with non-zero shard/realm exist (negative encoded IDs in the DB), which is the case for any multi-shard deployment.

### Recommendation
Add an explicit non-negative guard before calling `findById()`:

```java
long num = buffer.getLong();
if (num < 0) {
    return Optional.empty();
}
return entityRepository.findById(num).filter(e -> e.getType() == type);
```

Alternatively, route through `EntityId.of(0, 0, num)` which already throws `InvalidEntityException` for negative `num` (per `EntityId.encode()` line 76), and return empty on that exception — consistent with how `DomainUtils.fromEvmAddress()` handles it (lines 289–295).

### Proof of Concept

**Target:** Any mirror node GraphQL endpoint with entities having non-zero shard/realm (negative encoded IDs).

**Step 1 — Identify a target entity.** Suppose entity `1023.65535.274877906943` exists. Its encoded ID is `-1` (computable from the bit-packing formula).

**Step 2 — Craft the EVM address.** Bytes 0–11 = `0x000000000000000000000000`, bytes 12–19 = `0xFFFFFFFFFFFFFFFF` (-1 in big-endian two's complement):
```
evmAddress = "000000000000000000000000FFFFFFFFFFFFFFFF"
```

**Step 3 — Send GraphQL query:**
```graphql
query {
  account(input: { evmAddress: "000000000000000000000000FFFFFFFFFFFFFFFF" }) {
    id
    type
    balance { value }
  }
}
```

**Step 4 — Observe result.** The response returns the full entity record for `1023.65535.274877906943`, not an error or empty result, even though the supplied address does not correspond to any valid `0.0.X` entity. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    private static long encode(long shard, long realm, long num) {
        if (shard > SHARD_MASK || shard < 0 || realm > REALM_MASK || realm < 0 || num > NUM_MASK || num < 0) {
            throw new InvalidEntityException("Invalid entity ID: " + shard + "." + realm + "." + num);
        }

        if (shard == 0 && realm == 0) {
            return num;
        }

        return (num & NUM_MASK) | (realm & REALM_MASK) << NUM_BITS | (shard & SHARD_MASK) << (REALM_BITS + NUM_BITS);
    }
