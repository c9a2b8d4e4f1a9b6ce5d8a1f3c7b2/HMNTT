### Title
Unbounded `request.Metadata` Passthrough in `ConstructionPreprocess` Enables Bandwidth Griefing

### Summary
`ConstructionPreprocess` directly assigns the caller-supplied `request.Metadata` map to `options` with no size or key-count validation, then returns the entire map verbatim in the response. Because the endpoint requires no authentication and the HTTP server imposes no body-size limit, any external user can force the server to echo arbitrarily large attacker-controlled data, wasting inbound and outbound bandwidth on every request.

### Finding Description
**Exact code path:**

`rosetta/app/services/construction_service.go`, lines 304–320 (`ConstructionPreprocess`):

```go
options := make(map[string]any)
if len(request.Metadata) != 0 {
    options = request.Metadata          // ← direct alias, no size check
}
options[optionKeyOperationType] = operations[0].Type
...
return &rTypes.ConstructionPreprocessResponse{
    Options:            options,         // ← full attacker map returned
    RequiredPublicKeys: requiredPublicKeys,
}, nil
```

**Root cause:** The guard at line 305 only checks `len(request.Metadata) != 0`; it does not bound the number of keys, the length of key strings, or the size of values. The map is then returned wholesale as `Options`.

**Why existing checks are insufficient:**

1. The `http.Server` in `rosetta/main.go` lines 220–227 configures only `ReadTimeout`, `ReadHeaderTimeout`, `WriteTimeout`, and `IdleTimeout` — there is no `http.MaxBytesReader` or equivalent body-size cap anywhere in the rosetta middleware chain (confirmed by grep: zero matches for `MaxBytesReader`, `LimitReader`, `maxBodySize` in `rosetta/**/*.go`).
2. The coinbase `rosetta-sdk-go` asserter invoked via `server.NewConstructionAPIController` validates `NetworkIdentifier` and `Operations` but performs no size validation on the `Metadata` field of `ConstructionPreprocessRequest`.
3. No application-layer key-count or byte-budget check exists in `ConstructionPreprocess` itself.

### Impact Explanation
An attacker sends a POST to `/construction/preprocess` (available in both online and offline modes) with a `metadata` JSON object containing thousands of large key-value pairs. The server deserialises the full object, aliases it as `options`, appends one key (`operation_type`), and serialises the entire map back to the caller. The response body is therefore approximately equal in size to the request body, yielding a ~1:1 bandwidth amplification. Sustained at high request rate this exhausts the server's outbound bandwidth and increases JSON serialisation CPU load, degrading service for legitimate users. No funds are at risk; the impact is availability/griefing, consistent with the stated medium scope.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. No special account, key, or on-chain state is required. The attack is trivially scriptable: a single `curl` or Python script can send repeated large-metadata requests. It is repeatable indefinitely and requires no coordination or prior knowledge beyond the Rosetta API specification.

### Recommendation
1. **Enforce a body-size limit at the HTTP layer** — wrap the request body with `http.MaxBytesReader` (e.g., 64 KB) in a middleware applied to all routes, or set it per-handler before JSON decoding.
2. **Validate metadata at the service layer** — in `ConstructionPreprocess`, reject requests whose `Metadata` map exceeds a defined key-count (e.g., 10) or total serialised byte budget before assigning to `options`.
3. **Copy only known keys** — instead of `options = request.Metadata`, initialise `options` as an empty map and copy only the keys the service actually uses (e.g., `memo`, `valid_duration`), discarding all unknown keys.

### Proof of Concept
```bash
# Build a metadata map with 5000 large key-value pairs (~500 KB)
python3 - <<'EOF'
import json, requests

metadata = {f"key_{i}": "A" * 100 for i in range(5000)}
payload = {
  "network_identifier": {"blockchain": "Hedera", "network": "testnet"},
  "operations": [
    {"operation_identifier": {"index": 0},
     "type": "CRYPTOTRANSFER",
     "account": {"address": "0.0.100"},
     "amount": {"value": "-100", "currency": {"symbol": "HBAR", "decimals": 8}}},
    {"operation_identifier": {"index": 1},
     "type": "CRYPTOTRANSFER",
     "account": {"address": "0.0.200"},
     "amount": {"value": "100", "currency": {"symbol": "HBAR", "decimals": 8}}}
  ],
  "metadata": metadata
}
r = requests.post("http://<rosetta-host>:5700/construction/preprocess",
                  json=payload, timeout=30)
print(f"Request size: {len(json.dumps(payload))} bytes")
print(f"Response size: {len(r.content)} bytes")
print(f"Status: {r.status_code}")
# Response Options will contain all 5000 keys echoed back
EOF
```

Expected result: HTTP 200 with a response body of approximately the same size as the request, containing all attacker-supplied keys under `options`, with no server-side rejection.
