### Title
Unbounded `string_agg` Aggregation in `latestFileContentsQuery` Enables Resource Exhaustion via Ethereum Call-Data File Reference

### Summary
The `latestFileContentsQuery` static query in `rest/service/fileDataService.js` uses `string_agg` with no row-count or byte-size limit to concatenate all `file_data` rows since the last FileCreate/FileUpdate for a given entity. An attacker who has previously submitted an Ethereum transaction referencing a file with thousands of FileAppend operations can cause any unauthenticated user's request to `/api/v1/contracts/:contractId/results/:timestamp` to trigger this unbounded in-database aggregation, consuming large amounts of PostgreSQL memory and CPU per request.

### Finding Description

**Exact code path:**

`rest/service/fileDataService.js`, lines 19–42 — `latestFileContentsQuery`:

```sql
with latest_create as (
    select max(consensus_timestamp) as consensus_timestamp
    from file_data
    where entity_id = $1 and transaction_type in (17, 19)
    group by entity_id
    order by consensus_timestamp desc
)
select
    max(fd.consensus_timestamp),
    min(fd.consensus_timestamp) as first_consensus_timestamp,
    string_agg(fd.file_data, '' order by fd.consensus_timestamp) as file_data
from file_data fd
join latest_create l on fd.consensus_timestamp >= l.consensus_timestamp
where fd.entity_id = $1 and fd.transaction_type in (16,17,19)
    and fd.consensus_timestamp >= l.consensus_timestamp
group by fd.entity_id
```

This query is invoked from `rest/controllers/contractController.js` lines 1022–1023:

```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

**Root cause:** The `string_agg` aggregation has no `LIMIT`, no maximum byte-size guard, and no row-count cap. It concatenates every `file_data` row (transaction types 16, 17, 19) for the given entity since the most recent FileCreate/FileUpdate. If thousands of FileAppend rows exist for that entity, the database must read and concatenate all of them in a single query execution, holding the entire concatenated blob in PostgreSQL working memory.

**Failed assumption:** The code assumes the number of FileAppend rows since the last FileCreate/FileUpdate is small and bounded. There is no enforcement of this assumption at the query, service, or network layer within the mirror node.

**Exploit flow:**

1. Attacker creates a file on the Hedera network (FileCreate, type=17).
2. Attacker submits thousands of FileAppend (type=16) transactions to that file, each carrying up to ~4 KB of data (Hedera's per-transaction limit). With N=10,000 appends × 4 KB = ~40 MB of raw data stored in `file_data`.
3. Attacker submits an Ethereum transaction (EthereumTransaction) that sets `call_data_id` to the above file's entity ID. This is a standard Hedera EthereumTransaction field.
4. Any unauthenticated HTTP client queries `GET /api/v1/contracts/<contractId>/results/<timestamp>` for that Ethereum transaction.
5. The mirror node calls `getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []})`, which executes `latestFileContentsQuery` with no filters, causing PostgreSQL to scan and `string_agg` all N rows into a single blob.

**Why existing checks fail:**

- `isValidUserFileId` (line 1022) only validates that the file ID is a user-space entity ID (not a system file). It does not check the number of FileAppend rows or total data size associated with that file.
- `filterInnerPlaceholder` / `filterOuterPlaceholder` are empty (`{whereQuery: []}`) in this call path, so no timestamp filter is applied.
- There is no PostgreSQL `statement_timeout` or `work_mem` cap enforced at the query level in this service.
- `maxFileAttempts` (line 99) only limits retry iterations on parse failure, not the size of data fetched per attempt.

### Impact Explanation

Each request to `/api/v1/contracts/:contractId/results/:timestamp` for the crafted Ethereum transaction causes PostgreSQL to allocate memory proportional to the total size of all FileAppend data (potentially tens to hundreds of MB per query). Under concurrent requests, this can exhaust PostgreSQL `work_mem`, increase CPU utilization significantly, slow down all other queries sharing the database, and degrade or crash the mirror node REST API. This directly satisfies the stated threat model of increasing network processing node resource consumption by ≥30% without brute force.

### Likelihood Explanation

The precondition requires paying Hedera network transaction fees to create the file and submit thousands of FileAppend transactions. However, this is a one-time setup cost. Once the malicious Ethereum transaction is on-chain, the attack is:
- **Repeatable indefinitely** by any unauthenticated user who knows the contract ID and timestamp.
- **Amplifiable** by concurrent requests from multiple clients.
- **Persistent** — the file data remains in the database; no ongoing attacker action is needed after setup.

Hedera FileAppend fees are low (fractions of a cent per transaction at typical fee schedules), making the setup cost realistic for a motivated attacker.

### Recommendation

1. **Add a row-count or byte-size limit** to `latestFileContentsQuery`: add a subquery or CTE that counts rows or sums `length(file_data)` and aborts/returns null if a threshold is exceeded.
2. **Cap `string_agg` input** using PostgreSQL's `string_agg` with a `LIMIT` in the inner scan, or use a `WHERE length(file_data) <= <max>` guard.
3. **Set a `statement_timeout`** for this specific query execution context to prevent runaway aggregation.
4. **Validate file size at ingestion** or cache the assembled file content so repeated queries do not re-aggregate from raw rows.
5. **Rate-limit** the `/api/v1/contracts/:contractId/results/:timestamp` endpoint per IP.
