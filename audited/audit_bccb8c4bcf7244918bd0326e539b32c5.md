### Title
Cache Miss Amplification via Non-Existent Block Index in `findByIndex` — Unbounded DB Queries per Rate-Limit Budget

### Summary
`RecordFileRepository.findByIndex()` is annotated with `@Cacheable(unless = "#result == null")`. In Spring Cache, when the return type is `Optional<T>`, `#result` in the `unless` SpEL expression refers to the **unwrapped** value, not the `Optional` wrapper. When a non-existent block index is queried, the method returns `Optional.empty()`, `#result` resolves to `null`, the `unless` condition is `true`, and the result is **never stored in the cache**. Every subsequent request for the same (or any other) non-existent block index unconditionally hits the database. An unprivileged attacker can exploit this to saturate the full global rate-limit budget (500 RPS) with cache-bypassing DB queries.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` (line 26) delegates to `RecordFileRepository.findByIndex()` for any numeric `BlockType`:

```
web3/src/main/java/org/hiero/mirror/web3/service/RecordFileServiceImpl.java, line 26
    return recordFileRepository.findByIndex(block.number());
```

`findByIndex` carries the cache annotation:

```
web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java, lines 27-29
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
    @Query("select r from RecordFile r where r.index = ?1")
    Optional<RecordFile> findByIndex(long index);
```

The cache manager is configured at:

```
web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java, lines 153-164
    expireAfterWrite=10 minutes, maximumSize=10000
```

**Root cause — failed assumption:**

The developer assumed `#result` refers to the `Optional` object (which is never null). Spring Cache's documented behavior is the opposite: for `Optional<T>` return types, `#result` in `unless` is the **unwrapped** `T`, which is `null` when the Optional is empty. Therefore `unless = "#result == null"` evaluates to `true` for every miss, and Spring skips writing to the cache. The cache entry for a non-existent index is never populated, regardless of how many times it is queried.

**Exploit flow:**

1. Attacker identifies any block number beyond the chain tip (e.g., `0x7FFFFFFFFFFFFFFF`).
2. Attacker sends a contract-call request with `block=0x7FFFFFFFFFFFFFFF` to `/api/v1/contracts/call`.
3. `findByBlockType()` → `findByIndex(Long.MAX_VALUE)` → DB executes `SELECT r FROM RecordFile r WHERE r.index = ?1`.
4. DB returns no row → `Optional.empty()` → `#result == null` → `unless = true` → **not cached**.
5. Attacker repeats step 2 at the global rate limit (500 RPS). Each request hits the DB. The cache is never consulted.
6. Attacker may also cycle through distinct non-existent indices (e.g., `MAX_VALUE`, `MAX_VALUE-1`, …) to prevent any future caching even if the bug were fixed at runtime.

**Why existing checks are insufficient:**

- The global rate limiter (`ThrottleManagerImpl`, 500 RPS) is a shared bucket across all callers and all request types. It does not distinguish cache-hitting from cache-missing requests, and it is not per-IP. A single attacker can consume the entire budget with miss-only queries.
- The DB statement timeout (3000 ms) does not prevent the query from executing; it only caps individual query duration.
- The cache `maximumSize=10000` is irrelevant because empty results are never inserted.

### Impact Explanation

Every request for a non-existent block index generates a live DB query. Under sustained attack at 500 RPS, the database receives 500 index-lookup queries per second that the cache was designed to absorb. This degrades DB throughput available to legitimate traffic (contract execution, entity lookups, timestamp resolution), increasing latency and potentially causing query timeouts for real users. Because the `record_file` table is indexed on `index`, each query is fast individually, but at 500 RPS the aggregate I/O and connection-pool pressure is significant. The attack requires no authentication, no special privileges, and no knowledge of the chain state beyond "any large block number does not exist."

### Likelihood Explanation

The attack is trivially reproducible with a single HTTP client (e.g., `curl` in a loop or any load-testing tool). No credentials, API keys, or on-chain assets are required. The attacker only needs to know the public JSON-RPC endpoint and supply an out-of-range block parameter, which is standard Ethereum JSON-RPC behavior (`eth_call` with `blockNumber`). The rate limit of 500 RPS is high enough to cause measurable DB pressure. The bug is persistent — it does not self-heal as long as the block index does not exist.

### Recommendation

Replace the `unless` condition to correctly handle the `Optional` wrapper. Spring Cache evaluates `#result` as the unwrapped value for `Optional` returns, so the current condition is already semantically correct for the non-null case, but the fix should be made explicit and verified:

```java
// Option 1: use Optional-aware condition (Spring 4.3+)
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
    unless = "#result != null && !#result.isPresent()"
)
```

Or, cache the negative result explicitly using a sentinel value / `Optional.empty()` storage, and adjust the `unless` to never skip caching:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX
    // no unless — cache both hits and misses
)
```

Additionally, consider adding per-IP or per-caller rate limiting at the ingress layer (e.g., API gateway) to prevent a single client from consuming the full global budget.

### Proof of Concept

```bash
# Repeatedly query a block number far beyond the chain tip
# Each request bypasses the cache and hits the DB

ENDPOINT="http://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD='{"data":"0x","to":"0x0000000000000000000000000000000000000001","block":"0x7FFFFFFFFFFFFFFF"}'

for i in $(seq 1 500); do
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait

# Observe DB query logs: each iteration produces a live SELECT on record_file.index
# Cache hit counter for CACHE_MANAGER_RECORD_FILE_INDEX remains at 0 for this key
```
