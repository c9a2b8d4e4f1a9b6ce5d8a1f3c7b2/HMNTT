### Title
Shared DB Connection Pool Exhaustion via Unauthenticated `/network/status` Flood Starves Construction Service Alias Resolution

### Summary
The Rosetta API allocates a single shared `database/sql` connection pool (default `MaxOpenConnections = 100`) for all services. The unauthenticated `/network/status` endpoint makes three sequential DB queries per request with no application-level rate limiting. An attacker flooding this endpoint with ≥100 concurrent requests can exhaust the pool, causing the construction service's `resolveAccountAliases` DB call to block until timeout, preventing alias-account transactions from being gossiped.

### Finding Description
**Shared pool creation** — `rosetta/main.go:200` creates a single `dbClient` and passes it to every repository:

```go
dbClient := db.ConnectToDb(rosettaConfig.Db)
// accountRepo, addressBookEntryRepo, blockRepo, transactionRepo all share it
```

**Pool cap applied** — `rosetta/app/db/db.go:33`:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections) // default 100
```

**`/network/status` consumes connections** — `rosetta/app/services/network_service.go:67-80` makes three sequential DB queries per request (`RetrieveGenesis`, `RetrieveLatest`, `Entries`), each acquiring a pool connection for up to `statementTimeout` seconds (default 20 s, set via `rosetta/app/db/client.go:36`).

**Construction alias lookup contends on the same pool** — `rosetta/app/services/construction_service.go:581`:
```go
found, rErr := c.accountRepo.GetAccountId(ctx, accountId)
```
This call blocks waiting for a free connection if the pool is exhausted.

**No application-level rate limiting** — confirmed: zero occurrences of `inFlightReq`, `rateLimit`, or `throttle` in any `rosetta/**/*.go` file. The Traefik middleware (`inFlightReq: amount: 5`, `rateLimit: average: 10`) in `charts/hedera-mirror-rosetta/values.yaml:152-160` is an optional Kubernetes deployment artifact; it is absent when the binary is run directly or behind any other proxy.

### Impact Explanation
During the attack, every `/construction/metadata` request that involves an alias account (i.e., calls `resolveAccountAliases`) blocks on `GetAccountId` until the HTTP `WriteTimeout` (default 10 s) or `statementTimeout` (default 20 s) fires, returning an error to the caller. Alias-account transactions cannot be gossiped for the duration of the flood. This is a targeted, repeatable DoS against the transaction-submission path with no collateral damage to the attacker.

### Likelihood Explanation
The `/network/status` endpoint requires zero authentication and accepts a trivially small JSON body. Sending 100–200 concurrent HTTP/1.1 keep-alive POST requests is achievable with a single commodity machine using `wrk`, `hey`, or a short script. The attack is repeatable indefinitely and self-sustaining: as long as the flood continues, the pool stays exhausted.

### Recommendation
1. **Reserve connections for write-path operations**: use two separate `gorm.DB` instances (or two `sql.DB` pools) — one for read-only data endpoints (`/network/*`, `/block/*`) and one for the construction service — so exhaustion of the read pool cannot affect alias resolution.
2. **Enforce application-level concurrency limits**: add a semaphore or `golang.org/x/net/netutil.LimitListener` capping concurrent in-flight DB-touching requests per endpoint, independent of the deployment topology.
3. **Set a connection-acquisition timeout**: configure `db.SetConnMaxIdleTime` and ensure the context passed to every query has a deadline shorter than `WriteTimeout` so blocked callers fail fast rather than holding goroutines.
4. **Do not rely solely on Traefik middleware** for rate limiting; it is not present in all deployment configurations.

### Proof of Concept
```bash
# Exhaust the 100-connection pool with concurrent /network/status floods
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# Send 150 concurrent requests, sustained
hey -n 100000 -c 150 -m POST \
    -H "Content-Type: application/json" \
    -d "$BODY" \
    http://<rosetta-host>:5700/network/status &

# Simultaneously attempt a construction/metadata call with an alias account
# Expect: timeout or ErrDatabaseError returned instead of metadata
curl -s -X POST http://<rosetta-host>:5700/construction/metadata \
  -H "Content-Type: application/json" \
  -d '{
    "network_identifier":{"blockchain":"Hedera","network":"mainnet"},
    "options":{
      "operation_type":"CRYPTO_TRANSFER",
      "account_aliases":"0.0.AABBCCDD"
    }
  }'
# Result during flood: {"code":...,"message":"Failed to connect to db","retriable":true}
```
