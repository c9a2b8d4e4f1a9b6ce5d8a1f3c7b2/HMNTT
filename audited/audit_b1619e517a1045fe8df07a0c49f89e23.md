### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Enables MitM Injection of Forged `staking_reward_transfer` Records

### Summary
`GetDsn()` in `rosetta/app/config/types.go` hardcodes `sslmode=disable` with no SSL field in the `Db` struct, making it impossible to configure encrypted transport. An attacker on the same network segment as the Rosetta-to-PostgreSQL link can perform ARP spoofing to intercept the plaintext PostgreSQL wire-protocol stream and inject forged query responses, causing the Rosetta API to return fabricated `staking_reward_transfer` records attributing staking payouts to attacker-controlled accounts.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51 — `sslmode=disable` is a string literal with no runtime override:
```go
"host=%s port=%d user=%s dbname=%s password=%s sslmode=disable"
```
The `Db` struct (lines 39–47) contains no SSL-related field (`SslMode`, `SslCert`, `SslKey`, etc.), so there is no configuration path to enable encryption. `ConnectToDb()` in `rosetta/app/db/db.go` line 18 passes `dbConfig.GetDsn()` directly to `gorm.Open` with no post-hoc TLS upgrade.

**Root cause:** The assumption that the Rosetta↔PostgreSQL path is always a trusted, isolated channel is baked into the code. There is no mechanism — not even a config flag — to enforce transport security.

**Exploit flow:**
1. Attacker positions themselves on the same L2 segment as the Rosetta server and the PostgreSQL server (e.g., shared cloud VLAN, Kubernetes node without NetworkPolicy, co-located VM).
2. Attacker sends gratuitous ARP replies to poison the ARP cache of the Rosetta host, redirecting PostgreSQL-bound TCP (port 5432) through the attacker's machine.
3. Attacker forwards the TCP stream to the real PostgreSQL server, transparently proxying the authentication handshake (credentials are visible in plaintext for MD5 auth; for SCRAM the attacker relays the exchange).
4. After the authenticated session is established, the attacker intercepts `SELECT` queries against `staking_reward_transfer` and injects crafted PostgreSQL `DataRow` wire-protocol messages containing attacker-chosen `account_id` and `amount` values before the real server's response arrives (or by dropping the real response and substituting a forged one).
5. The Rosetta persistence layer (`rosetta/app/persistence/transaction.go`) consumes these rows without any integrity check and builds `Operation` objects that are returned to Rosetta API callers.

**Why existing checks fail:** There are no checks. The `Db` struct has no SSL field, `GetDsn()` has no branch for TLS, `ConnectToDb` performs no post-connection verification, and the GORM query layer applies no row-level integrity validation. The PostgreSQL wire protocol itself provides no message authentication when SSL is disabled.

### Impact Explanation
A successful injection causes the Rosetta `/block` and `/block/transaction` endpoints to report staking reward payouts credited to attacker-controlled accounts instead of legitimate recipients. Any downstream system (exchange, wallet, reconciliation tool) consuming the Rosetta API would record incorrect reward attribution. Because staking rewards are financial transfers of HBAR, this constitutes theft of unpaid staking rewards at the data-reporting layer. Severity is **High** given the financial nature of the affected data and the complete absence of any transport-layer protection.

### Likelihood Explanation
The precondition — network adjacency — is realistic in cloud deployments where the Rosetta server and its PostgreSQL instance share a VPC subnet or Kubernetes node without strict NetworkPolicy enforcement (a common default). ARP spoofing requires only the ability to send Ethernet frames on the local segment, which is available to any process running on a co-tenant VM or compromised adjacent pod without elevated OS privileges on the *target* machines. The attack is repeatable on every query cycle and leaves no application-layer log trace because the Rosetta server believes it is talking to a legitimate database.

### Recommendation
1. Add an `SslMode string` (or `SslMode`, `SslCert`, `SslKey`, `SslRootCert` fields) to the `Db` struct in `rosetta/app/config/types.go`.
2. Change `GetDsn()` to interpolate the configured SSL mode and certificate paths rather than hardcoding `sslmode=disable`.
3. Set the default to `sslmode=verify-full` with a configurable CA certificate, so the Rosetta server authenticates the PostgreSQL server's identity and encrypts all traffic.
4. Add a startup check in `ConnectToDb()` that refuses to proceed if `sslmode=disable` is set in a non-development profile.

### Proof of Concept
```
# Precondition: attacker machine on same L2 segment as rosetta-host (192.168.1.10)
# and postgres-host (192.168.1.20, port 5432).

# 1. Enable IP forwarding
echo 1 > /proc/sys/net/ipv4/ip_forward

# 2. ARP-poison both hosts
arpspoof -i eth0 -t 192.168.1.10 192.168.1.20 &
arpspoof -i eth0 -t 192.168.1.20 192.168.1.10 &

# 3. Run a transparent PostgreSQL MitM proxy (e.g., using pgmitm or a custom
#    Python script using the pg8000 wire-protocol library) that:
#    a. Forwards all traffic to 192.168.1.20:5432
#    b. Intercepts responses to queries matching:
#         SELECT ... FROM staking_reward_transfer ...
#    c. Replaces DataRow messages with rows containing:
#         account_id  = <attacker_account_encoded_id>
#         amount      = <original_amount>
#         consensus_timestamp = <original_timestamp>

# 4. Query the Rosetta /block endpoint and observe that staking reward
#    operations now list the attacker's account as the reward recipient.
curl http://rosetta-host:5700/block \
  -d '{"network_identifier":{...},"block_identifier":{...}}'
# Response will show staking_reward_transfer credited to attacker account.
```
