### Title
Unauthenticated DB Exhaustion via Unbounded Concurrent Requests to `getOutstandingAirdrops()`

### Summary
The `getOutstandingAirdrops()` endpoint in `TokenAirdropsController` is publicly accessible with no rate limiting at the application layer in the `rest-java` module. Any unauthenticated attacker can flood the endpoint with concurrent requests targeting an account with many outstanding airdrops and `limit=100`, causing repeated full-index DB scans that exhaust the database connection pool and degrade mirror node processing capacity.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, lines 66–75:
```java
@GetMapping(value = "/outstanding")
TokenAirdropsResponse getOutstandingAirdrops(
        @PathVariable EntityIdParameter id,
        @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
        ...
```
`MAX_LIMIT = 100` (confirmed in `Constants.java` line 35). No authentication annotation, no rate-limit guard.

The request flows to `TokenAirdropRepositoryCustomImpl.findAll()` (lines 58–72):
```java
return dslContext
        .selectFrom(TOKEN_AIRDROP)
        .where(condition)
        .orderBy(order)
        .limit(request.getLimit())
        .fetchInto(TokenAirdrop.class);
```
Every request unconditionally executes a DB query against the `token_airdrop` table filtered by `sender_account_id` and `state = PENDING`, returning up to 100 rows.

**Root cause:** The `rest-java` module contains zero rate-limiting logic. The only filters registered are `LoggingFilter` and `MetricsFilter` (confirmed by `rest-java/src/main/java/org/hiero/mirror/restjava/config/`). The `ThrottleManager`/`ThrottleConfiguration` with bucket4j rate limiting exists exclusively in the `web3` module for contract calls — it is not wired into `rest-java` at all.

**Why existing checks fail:**

1. **Traefik middleware** (`charts/hedera-mirror-rest-java/values.yaml`, lines 150–155) only configures a `circuitBreaker` (trips reactively after 10% network error ratio or 25% 5xx ratio) and a `retry` (3 attempts). There is no `rateLimit` middleware. The retry middleware actively **amplifies** the attack: each attacker request that causes a backend error is retried up to 3 times by Traefik, tripling the DB load.

2. **Circuit breaker** is reactive — it only engages after the service is already degraded, providing no prevention.

3. **`@Max(MAX_LIMIT)`** on the `limit` parameter caps results at 100 but does not limit request frequency.

4. **DB connection pool monitoring** (`RestJavaHighDBConnections` alert at 75% utilization) is observability-only, not a control.

### Impact Explanation
An attacker flooding this endpoint with concurrent requests forces repeated index scans on `token_airdrop` per request. The HikariCP connection pool for `rest-java` becomes saturated (monitored but not protected), causing queued/rejected DB connections. This degrades all `rest-java` endpoints sharing the same pool — including network, topic, and allowance endpoints — and can cascade to the importer's DB if they share infrastructure, degrading 30%+ of mirror node read processing. The Traefik retry amplification (3×) means N attacker connections generate 3N DB queries.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond a valid account ID (publicly discoverable via on-chain data or the `/api/v1/accounts` endpoint), and no brute force. The endpoint is publicly routed via the Traefik ingress path regex (`/api/v1/accounts/.../airdrops`). Any attacker with a script sending concurrent HTTP GET requests can execute this. It is repeatable indefinitely until infrastructure-level mitigations (firewall, CDN rate limiting) are applied externally.

### Recommendation
1. Add a bucket4j-based rate limiter to the `rest-java` module, analogous to `ThrottleConfiguration` in `web3`, applied globally via a `HandlerInterceptor` or `OncePerRequestFilter`.
2. Add per-IP rate limiting in the Traefik middleware configuration (`rateLimit` middleware).
3. Remove or reduce the `retry` middleware for `rest-java` endpoints, or scope it only to safe idempotent failures, to prevent amplification.
4. Consider adding a DB query timeout at the JOOQ/datasource level to bound the worst-case per-request DB impact.

        var type = request.getType();

                .and(getBoundConditions(bounds))
