### Title
Unbounded File Blob Load via `getLatestFileDataContents` Enables OOM DoS on REST API

### Summary
The public `GET /contracts/:contractId/results/:consensusTimestamp` endpoint unconditionally loads the entire contents of a `callDataId`-referenced file into Node.js heap memory with no size cap. Because no rate limiting or file-size guard exists, an attacker who knows a valid contractId/timestamp pair linked to a large file can flood the endpoint with concurrent requests, exhausting process memory and crashing the REST API.

### Finding Description

**Exact code path:**

`contractController.js` → `getContractResultsByTimestamp` (lines 1002–1042):

```js
// lines 1022-1024
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
``` [1](#0-0) 

`fileDataService.js` → `getLatestFileDataContents` (lines 84–87):

```js
getLatestFileDataContents = async (fileId, filterQueries) => {
  const {where, params} = super.buildWhereSqlStatement(filterQueries.whereQuery, [fileId]);
  return super.getSingleRow(this.getLatestFileContentsQuery(where), params);
};
``` [2](#0-1) 

The underlying SQL query uses an unbounded `string_agg` to concatenate every `file_data` chunk for the file into a single blob, with no `LIMIT` or `MAX` on the aggregated byte length:

```sql
string_agg(fd.file_data, '' order by fd.consensus_timestamp) as file_data
``` [3](#0-2) 

PostgreSQL assembles the full blob server-side and ships it to Node.js in one row. Node.js holds the entire buffer in heap for the lifetime of the request. No streaming, no chunking, no size check.

**Root cause / failed assumption:** The code assumes `callDataId` files are small enough to be safely materialized in memory. Hedera files can be built up via repeated `FileAppend` transactions (transaction type 16 in the query), making them arbitrarily large within network limits. The `{whereQuery: []}` argument passed at the call site applies zero additional filtering.

**Why `isValidUserFileId` is insufficient:** It only validates that the file ID is a well-formed user-space entity ID — it says nothing about the file's byte size. [4](#0-3) 

No rate-limiting middleware was found in the REST layer for this endpoint path.

### Impact Explanation

Each concurrent request to the endpoint for a large-file contract result allocates a full copy of the blob in the Node.js V8 heap. With a sufficiently large file (e.g., several MB assembled via `FileAppend` chunks) and tens or hundreds of concurrent requests, the process heap is exhausted, triggering an OOM crash. The REST API process goes down, preventing clients from querying transaction status and effectively blocking confirmation visibility for the network — matching the "total network shutdown" severity class for the mirror node REST tier.

### Likelihood Explanation

**Attacker capability:** Any user with a small amount of HBAR can submit an `EthereumTransaction` whose `callData` is stored in a large Hedera file (created via `FileCreate` + repeated `FileAppend`). This is a one-time setup cost. After that, the attacker discovers the resulting `contractId` and `consensusTimestamp` via other public mirror-node endpoints (e.g., `GET /contracts/results`) and hammers the target endpoint for free with no further on-chain cost. The attack is fully repeatable and requires no privileged access.

### Recommendation

1. **Add a size cap in the SQL query:** Use PostgreSQL's `substring` or enforce a `WHERE length(file_data) <= MAX_BYTES` guard before aggregation, or add a `HAVING length(string_agg(...)) <= MAX_BYTES` clause.
2. **Validate file size before loading:** After resolving `callDataId`, query `SELECT sum(length(file_data)) FROM file_data WHERE entity_id = $1` and reject or truncate if it exceeds a configured threshold (e.g., 128 KB).
3. **Apply per-IP / per-endpoint rate limiting** to `GET /contracts/:contractId/results/:consensusTimestamp` to bound concurrent in-flight requests.
4. **Stream or paginate large file data** rather than materializing the full blob in a single `getSingleRow` call.

### Proof of Concept

1. **Setup (one-time, requires HBAR):**
   - Submit a `FileCreate` transaction to Hedera mainnet/testnet.
   - Submit N × `FileAppend` transactions to grow the file to several MB.
   - Submit an `EthereumTransaction` referencing that file as `callData`.
   - Note the resulting `contractId` and `consensusTimestamp` (visible via `GET /api/v1/contracts/results`).

2. **Attack:**
   ```bash
   # Replace with real contractId and timestamp
   CONTRACT=0.0.12345
   TS=1700000000.000000000
   URL="https://<mirror-node>/api/v1/contracts/$CONTRACT/results/$TS"

   # Fire 200 concurrent requests
   for i in $(seq 1 200); do
     curl -s "$URL" &
   done
   wait
   ```

3. **Expected result:** Each request triggers `FileDataService.getLatestFileDataContents` loading the full multi-MB blob. With 200 concurrent requests the Node.js heap is exhausted, the process crashes with an OOM error, and the REST API becomes unavailable. [1](#0-0) [2](#0-1)
