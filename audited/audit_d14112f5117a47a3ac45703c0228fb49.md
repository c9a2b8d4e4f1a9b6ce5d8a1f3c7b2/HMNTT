### Title
Unbounded Indefinite gRPC Streaming Subscriptions Enable Resource Exhaustion DoS via Missing Connection Lifetime Controls in `grpcServerConfigurer()`

### Summary
An unprivileged external user can issue a `subscribeTopic` request with no `consensusEndTime` and no `limit`, causing the server to hold the streaming RPC open forever with no server-side termination. The `grpcServerConfigurer()` function configures only a per-connection concurrent-call cap (`maxConcurrentCallsPerConnection = 5`) but sets no `maxConnectionAge`, `maxConnectionIdle`, or global connection count limit. An attacker opening many connections, each saturating all 5 slots with indefinite subscriptions, exhausts the shared `applicationTaskExecutor` thread pool and starves legitimate callers of topic message delivery.

### Finding Description

**Code path — indefinite stream construction (`TopicMessageServiceImpl.java`):**

`subscribeTopic()` builds the reactive pipeline conditionally:

```java
// Line 79-85
if (filter.getEndTime() != null) {
    flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
}
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
```

`hasLimit()` returns `limit > 0`; the proto default for `limit` is `0` and `endTime` is optional. When both are absent the pipeline has no terminal operator.

`pastEndTime()` (line 123-125) — the only other termination path — explicitly returns `Flux.never()` when `endTime` is null:

```java
if (topicContext.getFilter().getEndTime() == null) {
    return Flux.never();
}
```

So `takeUntilOther(pastEndTime(topicContext))` never fires. The stream lives until the client disconnects.

**Code path — server configuration (`GrpcConfiguration.java`):**

```java
// Lines 31-34
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties.maxConcurrentCallsPerConnection` defaults to **5**. No `maxConnectionAge`, `maxConnectionIdle`, `keepAliveTimeout`, or global connection count is set. The grep across all Java sources confirms zero occurrences of these Netty knobs anywhere in the gRPC module.

**Root cause:** `grpcServerConfigurer()` provides only a per-connection concurrent-call ceiling. It does not bound connection lifetime or the total number of connections. Because indefinite subscriptions are protocol-legal (documented in `consensus_service.proto` lines 20-25), the server will hold each such RPC — and its associated thread from `applicationTaskExecutor` — open indefinitely. There is no watchdog, no idle timeout, and no server-side subscription duration cap.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` is scoped to a single HTTP/2 connection; it does not limit the number of connections an attacker may open.
- `TopicMessageFilter` validation (`@Min(0)` on `limit`, nullable `endTime`) explicitly permits the zero-limit, null-endTime combination.
- The `retriever.timeout = 60s` property applies only to the historical retriever polling interval, not to the live streaming phase.

### Impact Explanation

Each indefinite subscription holds a Reactor subscription backed by a thread from the shared `applicationTaskExecutor`. An attacker opening *N* connections and saturating all 5 slots per connection with indefinite `subscribeTopic` calls consumes 5*N* executor threads. Once the pool is exhausted, new legitimate `subscribeTopic` calls queue indefinitely or are rejected, preventing any other caller from receiving consensus topic messages. Because there is no connection lifetime limit, the attacker's connections and their 5 slots remain occupied until the attacker chooses to disconnect. This is a complete denial of the topic-message streaming service to all other users.

### Likelihood Explanation

No authentication or authorization is required — the gRPC port (default 5600) is publicly accessible. The attack requires only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). Crafting the malicious request is trivial: omit `consensusEndTime` and `limit` (or set `limit = 0`) in a `ConsensusTopicQuery`. The attack is repeatable, automatable, and requires no special knowledge beyond the public proto definition. A single attacker machine with modest network resources can open hundreds of connections.

### Recommendation

1. **Add connection lifetime limits in `grpcServerConfigurer()`:**
   ```java
   serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
   serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
   ```
2. **Add a configurable maximum subscription duration** in `TopicMessageServiceImpl.subscribeTopic()`: apply `.timeout(maxSubscriptionDuration)` unconditionally, or enforce a server-side cap on how long a no-endTime subscription may live.
3. **Expose `maxConnections` via `NettyProperties`** and call `serverBuilder.maxConnectionAge(...)` so operators can tune it per deployment.
4. **Add per-IP connection rate limiting** at the load-balancer or ingress layer in front of the gRPC service.

### Proof of Concept

```bash
# Install grpcurl; no credentials needed
# 1. Open 5 indefinite subscriptions on connection 1 (saturates all slots)
for i in $(seq 1 5); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 2. Repeat across N connections to exhaust the thread pool
for conn in $(seq 1 200); do
  for slot in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# 3. Legitimate caller now hangs or receives RESOURCE_EXHAUSTED
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 1}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: hangs indefinitely or returns gRPC status RESOURCE_EXHAUSTED / UNAVAILABLE
```
