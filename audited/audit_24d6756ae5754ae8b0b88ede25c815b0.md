### Title
Unauthenticated Double DB Query Per Request in `getContractById()` Enables Resource Exhaustion via Large-Bytecode Contracts

### Summary
`getContractById()` in `rest/controllers/contractController.js` unconditionally issues two sequential database queries for every contract that has a valid `file_id`: one to the entity/contract tables and a second expensive `string_agg` correlated subquery to the `file_data` table. No rate limiting exists on the REST API's GET endpoints, and the response cache has a default TTL of only 1 second, which is trivially bypassed by rotating across different contract EVM addresses. An unprivileged attacker with knowledge of multiple contracts backed by large file IDs can sustain a high rate of double-DB-query requests, amplifying database load.

### Finding Description

**Code path — first DB query:**

`rest/controllers/contractController.js` lines 720–726:
```js
const {query, params} = getContractByIdOrAddressContractEntityQuery({...});
const {rows} = await pool.queryQuietly(query, params);
```

**Code path — conditional second DB query:**

Lines 731–732:
```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
}
```

`isValidUserFileId` (`rest/utils.js` line 445–447) only checks `num > 1000` — it does not limit bytecode size or query cost.

**The second query (`rest/service/fileDataService.js` lines 44–63)** is a correlated subquery with `string_agg`:
```sql
select string_agg(file_data, '' order by consensus_timestamp) data
from file_data
where entity_id = $1
  and consensus_timestamp >= (
    select consensus_timestamp from file_data
    where entity_id = $1 and consensus_timestamp <= $2
    and (transaction_type = 17 or (transaction_type = 19 and length(file_data) <> 0))
    order by consensus_timestamp desc limit 1
  )
  and consensus_timestamp <= $2
```

For a large contract (e.g., 24 KB bytecode stored in 6 × 4 KB `FileCreate`/`FileAppend` chunks), this scans and aggregates multiple rows from `file_data` per request.

**No rate limiting on REST GET endpoints:** The `ThrottleManager` (`web3/src/main/java/.../ThrottleManagerImpl.java`) is only wired to the web3 `POST /contracts/call` endpoint. The REST API router (`rest/routes/contractRoute.js` line 13) uses `extendExpress` which is purely an async error wrapper — no throttle middleware is applied.

**Response cache bypass:** `rest/middleware/responseCacheHandler.js` line 24 sets `DEFAULT_REDIS_EXPIRY = 1` (1 second). The cache key is `MD5(req.originalUrl)` (line 152). An attacker rotating across N distinct EVM addresses of contracts with large file IDs generates N distinct cache keys, each missing the cache and triggering both DB queries.

### Impact Explanation

Each request to a contract with a large file-backed bytecode causes:
1. A JOIN query across `entity`/`contract` tables.
2. A correlated `string_agg` subquery across `file_data` that reads and concatenates all bytecode chunks.

At sustained request rates across many distinct contract addresses, the database receives 2× the query load compared to endpoints that perform a single lookup. The `string_agg` query is particularly expensive under I/O pressure because it must read, sort, and concatenate multiple rows per request. This can measurably increase DB CPU, I/O, and connection pool saturation, degrading service for all users of the mirror node.

### Likelihood Explanation

- **No authentication required.** Any external user can call `GET /api/v1/contracts/<evm_address>`.
- **Contract EVM addresses are public.** On Hedera mainnet/testnet, contracts with large file IDs (complex DeFi/NFT contracts) are discoverable via the same REST API (`GET /api/v1/contracts`).
- **No rate limiting.** The attacker can send requests at the maximum rate the network allows.
- **Cache bypass is trivial.** Rotating across even a few dozen distinct contract addresses is sufficient to keep the cache miss rate near 100%.
- **Repeatable and automatable.** A simple script with a list of EVM addresses suffices.

### Recommendation

1. **Add per-IP or global rate limiting** to the REST API GET endpoints, particularly `/contracts/:contractId`, using middleware such as `express-rate-limit`.
2. **Increase the response cache TTL** for the `getContractById` endpoint beyond 1 second (e.g., 5–30 seconds), since contract bytecode is immutable once deployed.
3. **Cache the `getFileData` result** at the service layer keyed by `(file_id, created_timestamp)`, so repeated requests for the same contract's bytecode do not re-execute the `string_agg` query.
4. **Consider fetching bytecode inline** in the first query (e.g., via a JOIN or subquery on `file_data`) to eliminate the second round-trip entirely.

### Proof of Concept

```bash
# 1. Enumerate contracts with large file_ids from the public API
curl "https://<mirror-node>/api/v1/contracts?limit=100&order=desc" \
  | jq -r '.contracts[] | select(.file_id != null) | .evm_address' > evm_addrs.txt

# 2. Send concurrent requests rotating across distinct EVM addresses
# Each request triggers 2 DB queries; no auth needed
while true; do
  while IFS= read -r addr; do
    curl -s "https://<mirror-node>/api/v1/contracts/$addr" &
  done < evm_addrs.txt
  wait
done
```

Each iteration sends N requests (one per unique EVM address), each causing two DB queries — the second being an expensive `string_agg` correlated subquery on `file_data`. With sufficient concurrency and a list of contracts with large bytecodes, DB CPU and I/O increase proportionally, with no application-layer mechanism to stop or slow the attacker.
