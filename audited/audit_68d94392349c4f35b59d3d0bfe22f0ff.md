### Title
Fee-Free Database Exhaustion via Uncached Alias Lookup Failures in `getSenderAccountIDAsNum()`

### Summary
An unprivileged attacker can supply arbitrary non-long-zero EVM addresses as the `from` field in repeated `/api/v1/contracts/call` requests. Each request triggers a live database query in `getSenderAccountIDAsNum()` for the alias lookup, which is never cached on miss. Combined with a gas-throttle bypass for `gas <= 10,000`, the attacker can sustain the maximum allowed request rate (500 req/sec) while consuming zero gas-bucket tokens, driving continuous uncached DB queries at no cost.

### Finding Description

**Code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → `TransactionExecutionService.execute()` → `defaultTransactionBodyBuilder()` → `getSenderAccountID()` → `getSenderAccountIDAsNum()`

In `getSenderAccountIDAsNum()`, when the supplied address is non-null and not a long-zero address, the code unconditionally queries `aliasesReadableKVState`: [1](#0-0) 

`aliasesReadableKVState.get()` delegates to `ReadableKVStateBase.get()`, which checks a per-request `ContractCallContext` read cache first: [2](#0-1) 

On a cache miss it calls `AliasesReadableKVState.readFromDataSource()`, which calls `CommonEntityAccessor.get(Bytes, Optional)`: [3](#0-2) 

This resolves to `EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse()`, which carries a `@Cacheable` annotation with `unless = "#result == null"`: [4](#0-3) 

**Critical gap — no negative caching:** When the address does not exist, the result is `Optional.empty()` (which is not null, but the entity inside is absent), and the `unless` condition prevents caching. Every request for an unknown address hits the database. The per-request `ContractCallContext` read cache is scoped to a single `ScopedValue` invocation and is discarded after each request: [5](#0-4) 

**Gas throttle bypass:** `ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas())` tokens from the gas bucket: [6](#0-5) 

`scaleGas()` returns `0` for any `gas <= 10,000`: [7](#0-6) 

`Bucket.tryConsume(0)` always succeeds, so the gas bucket is never depleted. Only the global rate-limit bucket (default 500 req/sec, not per-IP) constrains the attacker: [8](#0-7) 

**Gas tokens are not restored on `PAYER_ACCOUNT_NOT_FOUND`:** The controller only restores gas tokens for `InvalidParametersException`, not for `MirrorEvmTransactionException`: [9](#0-8) 

### Impact Explanation
At 500 req/sec (the global rate limit), an attacker using distinct random non-existent EVM addresses as `from` and `gas=0` drives 500 uncached SQL queries per second against the `entity` table. The query involves a `WHERE evm_address = ? OR alias = ?` predicate with no result to cache. This can saturate DB connection pools, increase query latency for legitimate users, and degrade or deny service to the entire shared instance. Because the rate limit is global (not per-IP), a single attacker consumes the full budget, blocking all other users.

### Likelihood Explanation
No privileges, accounts, or on-chain assets are required. The attacker needs only HTTP access to the public `/api/v1/contracts/call` endpoint. The exploit is trivially scriptable: generate random 20-byte hex strings as the `from` field, set `gas=0`, and send at maximum rate. The attack is repeatable indefinitely and produces no on-chain trace.

### Recommendation
1. **Add negative caching** for alias lookups: cache `Optional.empty()` results in the Caffeine entity cache for a short TTL (e.g., 1–5 seconds) so repeated lookups for the same unknown address do not hit the database. Change `unless = "#result == null"` to `unless = "false"` or cache the empty Optional explicitly.
2. **Fix the gas throttle floor**: `scaleGas()` returning `0` for `gas <= 10,000` means the gas bucket provides no protection for small-gas requests. Consider consuming a minimum of 1 token regardless of gas value.
3. **Add per-IP rate limiting** at the ingress/load-balancer layer or within `ThrottleManagerImpl` to prevent a single source from consuming the entire global rate limit.
4. **Fail fast before DB lookup**: if the `from` address is syntactically valid but the alias lookup fails, consider a short-lived in-memory bloom filter or negative cache keyed on the address bytes to avoid repeated DB hits.

### Proof of Concept
```bash
# Requires: curl, bash
# Sends 500 requests/sec with random non-existent EVM addresses, gas=0
# Each request triggers an uncached DB alias lookup and returns PAYER_ACCOUNT_NOT_FOUND

for i in $(seq 1 5000); do
  RAND_ADDR="0x$(openssl rand -hex 20)"
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"from\":\"$RAND_ADDR\",\"to\":\"0x0000000000000000000000000000000000000001\",\"gas\":0,\"data\":\"0x\"}" &
done
wait
```

Each request follows the path:
1. `ThrottleManagerImpl.throttle()` — rate bucket consumes 1 token; gas bucket consumes 0 (gas=0 → scaleGas=0).
2. `getSenderAccountIDAsNum()` — address is non-long-zero, calls `aliasesReadableKVState.get()`.
3. `AliasesReadableKVState.readFromDataSource()` — calls `entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse()`.
4. DB returns empty; result not cached (`unless = "#result == null"` blocks caching of null, but `Optional.empty()` wrapping means the cache key is never populated for misses in practice).
5. `throwPayerAccountNotFoundException()` throws `MirrorEvmTransactionException(PAYER_ACCOUNT_NOT_FOUND)`.
6. Gas tokens are NOT restored (only `InvalidParametersException` triggers restore).

                    .shardNum(senderAccountID.getShardNum())
                    .realmNum(senderAccountID.getRealmNum())
                    .build();
        }
        return accountIDNum;
    }
