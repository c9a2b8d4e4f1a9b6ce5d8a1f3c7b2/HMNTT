### Title
Shared `applicationTaskExecutor` Thread Pool Exhaustion via Unbounded gRPC Connection Flooding Under Downstream Partition

### Summary
`grpcServerConfigurer()` assigns Spring Boot's shared `applicationTaskExecutor` as the gRPC Netty server's executor and sets `maxConcurrentCallsPerConnection = 5` per connection, but imposes no limit on the total number of inbound connections. When a downstream resource (database or Redis) is slow or partitioned, each `subscribeTopic` call blocks an executor thread on a synchronous `entityRepository.findById()` call. An unprivileged attacker opening many connections and flooding each with 5 concurrent calls can exhaust the bounded thread pool, rendering the server unable to process any new requests.

### Finding Description

**Exact code path:**

`grpcServerConfigurer()` at [1](#0-0)  assigns the Spring Boot `applicationTaskExecutor` (a `ThreadPoolTaskExecutor` with default core size 8, max size `Integer.MAX_VALUE`, and an **unbounded** `LinkedBlockingQueue`) to the Netty gRPC server. It sets `maxConcurrentCallsPerConnection` to 5 [2](#0-1)  but configures no `maxConnections`, no `maxConnectionAge`, and no per-IP rate limit.

**Blocking operation on executor thread:**

`ConsensusController.subscribeTopic()` calls `.subscribe()` on the reactive chain, which triggers execution on the current executor thread. [3](#0-2) 

Inside `TopicMessageServiceImpl.subscribeTopic()`, the first operation is `topicExists(filter)`: [4](#0-3) 

`topicExists()` calls `entityRepository.findById(topicId.getId())` **eagerly** (before wrapping in `Mono.justOrEmpty()`), making it a synchronous blocking JDBC call that executes on the executor thread: [5](#0-4) 

When the database is partitioned or slow, this call blocks for up to the configured `statementTimeout` (default 10,000 ms). [6](#0-5) 

**Root cause and failed assumption:**

The design assumes the downstream DB is always responsive. The `applicationTaskExecutor` is shared across the entire Spring application context. With an unbounded queue and only 8 core threads, once all threads are blocked on slow DB calls, all subsequent gRPC requests queue indefinitely and are never processed. The `maxConcurrentCallsPerConnection = 5` guard only limits calls per single TCP connection — it does not bound the total number of concurrent calls across all connections.

### Impact Explanation

Complete gRPC server denial of service. All legitimate subscribers to `subscribeTopic` (HCS topic streaming) and `getNodes`/`getFeeEstimate` (Network Service) are unable to receive responses. The `applicationTaskExecutor` is shared with other Spring-managed async tasks, so the impact extends beyond gRPC to the entire application. Severity: **High** — the server becomes fully unresponsive to all clients for the duration of the attack.

### Likelihood Explanation

No authentication or authorization is required to open gRPC connections or call `subscribeTopic`. The attacker needs only network access to port 5600. The attack is amplified by any real or induced downstream slowness (e.g., a DB overload attack run in parallel). The `maxConcurrentCallsPerConnection = 5` limit means the attacker needs only `ceil(threadPoolSize / 5)` connections (e.g., 2 connections with 5 calls each to saturate 8 core threads). This is trivially achievable with a single client process. The attack is repeatable and persistent as long as the downstream resource remains slow.

### Recommendation

1. **Isolate the gRPC executor**: Replace `applicationTaskExecutor` with a dedicated, bounded `ThreadPoolExecutor` with a bounded queue and a `CallerRunsPolicy` or rejection handler, so exhaustion does not affect the rest of the application.
2. **Limit total inbound connections**: Add `serverBuilder.maxConnectionAge(...)` and `serverBuilder.maxConnectionIdle(...)`, and consider `serverBuilder.maxInboundConnections(N)` (available in grpc-java ≥ 1.60).
3. **Fix the blocking call in reactive context**: Wrap `entityRepository.findById()` in `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())` to avoid blocking the gRPC executor thread.
4. **Add per-IP connection rate limiting** at the load balancer or Netty pipeline level.

### Proof of Concept

```bash
# Requires grpcurl. Run from any machine with network access to port 5600.
# Step 1: Open 4 connections, each with 5 concurrent blocking subscribeTopic calls
# (DB must be slow/partitioned, or use a non-existent topic to trigger slow path)

for i in $(seq 1 20); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 99999999}}' \
    <host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2: Verify server is unresponsive to new requests
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}}' \
  <host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: hangs indefinitely or times out — executor queue is full
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
