### Title
Unbounded In-Memory Transaction Accumulation in `Block()` Before `maxTransactionsInBlock` Cap Is Applied

### Summary
In `rosetta/app/services/block_service.go`, the `Block()` function calls `FindBetween()` to load **all** transactions for a block into memory before the `maxTransactionsInBlock` cap is applied. The underlying `FindBetween()` implementation in `rosetta/app/persistence/transaction.go` uses an unbounded accumulation loop that fetches batches of 2,000 rows and appends them without any total-count ceiling. An unauthenticated attacker who identifies a high-transaction block and sends concurrent requests for it can exhaust server memory.

### Finding Description
**Exact code path:**

In `rosetta/app/services/block_service.go` line 56, `Block()` calls:
```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
``` [1](#0-0) 

The cap is only checked afterward at line 61:
```go
if len(block.Transactions) > s.maxTransactionsInBlock {
``` [2](#0-1) 

In `rosetta/app/persistence/transaction.go`, `FindBetween()` uses a loop that fetches `batchSize = 2000` rows per iteration and appends them all to a single in-memory slice with no upper bound on total accumulation:
```go
const batchSize = 2000
...
transactions := make([]*transaction, 0)
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    transactions = append(transactions, transactionsBatch...)
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
``` [3](#0-2) [4](#0-3) 

**Root cause:** `maxTransactionsInBlock` is a *response-size* cap, not a *query* or *memory* cap. The full transaction set for the block is always materialized in memory first. There is no `LIMIT` applied at the database level to the total number of transactions fetched across all batches.

**Failed assumption:** The design assumes that blocks will never contain enough transactions to cause memory pressure before the cap is applied. This assumption is not enforced in code.

### Impact Explanation
Hedera can sustain thousands of transactions per second. A single record-file block covering a ~2-second window during a high-throughput period can contain tens of thousands of transactions. Each transaction row includes JSON-encoded crypto transfers, staking reward payouts, and itemized transfers — all deserialized into Go structs in memory. With concurrent requests from multiple source IPs targeting the same high-transaction block, server memory can be exhausted, causing an OOM kill or severe degradation of the Rosetta API process. The `/block` endpoint requires no authentication. [5](#0-4) 

### Likelihood Explanation
The `/block` endpoint is publicly accessible with no authentication requirement. An attacker only needs to identify a block with a large transaction count — this is trivially done by querying the Hedera mirror node REST API or monitoring the network during high-throughput periods. The block index or hash is then used directly in the `BlockRequest`. The Traefik-level rate limit (`average: 10`, `inFlightReq: 5` per IP) exists only in the optional Helm chart deployment and is not enforced at the application layer; it is bypassable with multiple source IPs. [6](#0-5) 

### Recommendation
Apply the `maxTransactionsInBlock` limit **at the database query level**, not after full materialization. Pass the limit as a SQL `LIMIT` clause in `FindBetween()`, or add a total-count guard inside the accumulation loop that breaks early once the threshold is reached. For example, modify the loop in `FindBetween()` to accept an optional `maxRows int` parameter and stop accumulating once `len(transactions) >= maxRows`. Additionally, enforce a hard application-level cap independent of infrastructure middleware. [7](#0-6) 

### Proof of Concept
1. Identify a block with a large number of transactions using the Hedera mirror node REST API: `GET /api/v1/blocks?order=desc&limit=1` — inspect `count` field.
2. Note the block `index` or `hash`.
3. Send concurrent unauthenticated POST requests to the Rosetta `/block` endpoint:
```json
{
  "network_identifier": { "blockchain": "Hedera", "network": "mainnet" },
  "block_identifier": { "index": <HIGH_TX_BLOCK_INDEX> }
}
```
4. Use 10–20 concurrent clients from different IPs to bypass the per-IP Traefik rate limit.
5. Observe server memory climbing as each request causes `FindBetween()` to load all transactions for the block into memory simultaneously, before any cap is applied. [8](#0-7)

func (s *blockAPIService) Block(
	ctx context.Context,
	request *rTypes.BlockRequest,
) (*rTypes.BlockResponse, *rTypes.Error) {
	block, err := s.RetrieveBlock(ctx, request.BlockIdentifier)
	if err != nil {
		return nil, err
	}

	if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
		return nil, err
	}

	var otherTransactions []*rTypes.TransactionIdentifier
	if len(block.Transactions) > s.maxTransactionsInBlock {
		otherTransactions = make([]*rTypes.TransactionIdentifier, 0, len(block.Transactions)-s.maxTransactionsInBlock)
		for _, transaction := range block.Transactions[s.maxTransactionsInBlock:] {
			otherTransactions = append(otherTransactions, &rTypes.TransactionIdentifier{Hash: transaction.Hash})
		}
		block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
	}

	if err = s.updateOperationAccountAlias(ctx, block.Transactions...); err != nil {
		return nil, err
	}

	return &rTypes.BlockResponse{Block: block.ToRosetta(), OtherTransactions: otherTransactions}, nil
}
