### Title
Unbounded Alias-Based DB Lookup in `getHooks()` Enables Resource Exhaustion via Concurrent Requests

### Summary
`HookServiceImpl.getHooks()` calls `entityService.lookup(request.getOwnerId())`, which in the `rest-java` `EntityServiceImpl` performs a live, uncached `entityRepository.findByAlias()` database query for every alias-based request. Unlike the `importer` module's `EntityIdServiceImpl` (which uses a Spring Cache), the `rest-java` `EntityServiceImpl` has no caching and the `HooksController` endpoint has no rate limiting. An unprivileged attacker can flood the endpoint with concurrent alias-based requests, forcing parallel DB queries that exhaust the connection pool and spike CPU/DB load well above 30%.

### Finding Description
**Code path:**

- `HooksController` (`rest-java/.../controller/HooksController.java`, line 81–102): Public, unauthenticated `GET /api/v1/accounts/{ownerId}/hooks` endpoint. No rate-limiting filter is applied.
- `HookServiceImpl.getHooks()` (`rest-java/.../service/HookServiceImpl.java`, line 35): `final var id = entityService.lookup(request.getOwnerId());`
- `EntityServiceImpl.lookup()` (`rest-java/.../service/EntityServiceImpl.java`, lines 31–38):
  ```java
  case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
  ```
  No `@Cacheable`, no in-memory cache, no memoization — every alias-based call hits the database.
- `EntityRepository.findByAlias()` (`rest-java/.../repository/EntityRepository.java`, lines 13–14):
  ```java
  @Query(value = "select id from entity where alias = ?1 and deleted <> true", nativeQuery = true)
  Optional<Long> findByAlias(byte[] alias);
  ```
  A raw native SQL query executed synchronously per request.

**Root cause:** The `rest-java` `EntityServiceImpl` omits the caching layer present in the `importer` module's `EntityIdServiceImpl` (which uses `cacheLookup()` backed by a Spring `Cache`). The `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl` rate-limiting beans are scoped exclusively to `web3` and are not applied to `rest-java` controllers.

**Why checks fail:** The only input validation on the endpoint is `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on the `hookId` query parameter — this does not limit request rate or alias-lookup frequency. Numeric `ownerId` lookups (`EntityIdNumParameter`) return immediately via `Optional.of(p.id())` with no DB call, but alias-based lookups always go to the database.

### Impact Explanation
Each concurrent alias-based request to `/api/v1/accounts/{alias}/hooks` consumes one HikariCP connection for the `findByAlias` query plus a second connection for the subsequent `hookRepository` query. With the default HikariCP pool (typically 10 connections for the `rest-java` service), as few as 5–10 concurrent attackers can saturate the pool, causing connection-pending queues and query latency spikes for all users. CPU load on the database increases proportionally. This constitutes a realistic >30% resource increase without brute-forcing credentials or exploiting any authentication bypass.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. Any attacker who can enumerate or observe a single valid alias (e.g., from public blockchain data) can replay it at high concurrency. The attack requires no special tooling — standard HTTP load tools (`wrk`, `ab`, `hey`) suffice. It is repeatable and sustainable indefinitely. The absence of any rate-limiting or caching in the `rest-java` service layer makes this trivially exploitable.

### Recommendation
1. **Add caching** to `EntityServiceImpl.lookup()` in `rest-java` using Spring's `@Cacheable` (mirroring the `importer` module's `cacheLookup()` pattern), keyed on the alias bytes, with a short TTL (e.g., 30–60 seconds).
2. **Add rate limiting** to the `rest-java` module, either at the controller level (e.g., bucket4j per-IP) or via a servlet filter, analogous to the `web3` `ThrottleManagerImpl`.
3. **Consider a shared alias resolution cache** across the `rest-java` service to prevent thundering-herd on cache misses.

### Proof of Concept
**Preconditions:**
- Mirror node is running with `rest-java` service exposed.
- At least one entity with a known alias exists (obtainable from public mirror node data).

**Steps:**
```bash
# 1. Obtain a valid alias for an existing account (e.g., from /api/v1/accounts)
ALIAS="HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA"

# 2. Send 200 concurrent requests using the alias-based ownerId
hey -n 2000 -c 200 \
  "http://<mirror-node-host>:8084/api/v1/accounts/0.0.${ALIAS}/hooks"

# 3. Observe DB connection pool metrics (hikaricp_connections_active,
#    hikaricp_connections_pending) via Prometheus/Grafana dashboard
#    (rest-java dashboard already tracks these metrics).
#    Expected: active connections spike to pool max, pending queue grows,
#    p99 latency increases >30% vs. baseline, DB CPU rises.
```

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
