### Title
Unauthenticated Log Flooding via Unbounded `getNodes()` gRPC Endpoint

### Summary
The `getNodes()` endpoint in `NetworkController` is publicly accessible with no authentication or rate limiting. Every invocation triggers at least two `log.info` calls in `NetworkServiceImpl.getNodes()` — one via `doOnSubscribe` and one via `page()` — allowing an unprivileged attacker to generate log entries at an arbitrary rate, exhausting log storage or overwhelming log aggregation pipelines.

### Finding Description
**Code path:**

- `NetworkController.java` lines 33–43: `getNodes()` accepts any `AddressBookQuery` from any caller, creates a new reactive subscription per call, and has no authentication guard or rate-limiting interceptor.
- `NetworkServiceImpl.java` line 75: `.doOnSubscribe(s -> log.info("Querying for address book: {}", filter))` — fires one `INFO` log entry per subscription (i.e., per gRPC call).
- `NetworkServiceImpl.java` lines 100–105: `page()` fires a second `INFO` log entry per call: `"Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}"`.

**Root cause:** There is no per-client rate limit, no connection throttle, and no authentication requirement on the gRPC `getNodes` method. The `doOnSubscribe` log is unconditional and fires synchronously on every subscription. The grep across `grpc/src/main/resources/` and `grpc/src/main/java/` confirms zero rate-limiting configuration for this endpoint (rate limiting exists only in `PollingTopicMessageRetriever` for topic subscriptions, not for address book queries).

**Failed assumption:** The implementation assumes callers are well-behaved or that infrastructure-level controls (e.g., a load balancer) will throttle abuse. No such controls are enforced at the application layer.

### Impact Explanation
An attacker sending N requests/second generates ≥2N `INFO` log lines/second. At modest rates (e.g., 10,000 req/s from a single host), this produces ~20,000 log lines/second. Over hours this can: (1) exhaust disk-based log storage, causing log loss or service disruption; (2) overwhelm log aggregation systems (e.g., ELK, Splunk, Loki), degrading observability for operators; (3) mask legitimate errors and security events in log noise. The impact is classified as griefing/availability degradation with no direct economic loss to network users, consistent with the stated Medium scope.

### Likelihood Explanation
The gRPC port is typically internet-exposed for mirror node consumers. Any attacker with a gRPC client (e.g., `grpcurl`, a trivial Go/Python script) can call `hiero.mirror.api.proto.NetworkService/getNodes` in a tight loop. No credentials, tokens, or special network access are required. The attack is trivially repeatable and scriptable, and can be amplified across multiple source IPs to evade IP-based blocking.

### Recommendation
1. **Rate limiting at the gRPC interceptor layer**: Add a per-IP or global rate-limiting `ServerInterceptor` for the `getNodes` RPC, consistent with how `RetrieverProperties` rate-limits topic polling.
2. **Reduce log verbosity**: Downgrade the `doOnSubscribe` log from `INFO` to `DEBUG`, or deduplicate/sample it (e.g., log at most once per minute per unique filter).
3. **Log size caps**: Configure log rotation with hard size caps and async, bounded log appenders so a flood cannot exhaust disk.
4. **Connection/concurrency limits**: Configure gRPC server `maxConcurrentCallsPerConnection` and consider requiring mTLS or an API key for address book queries.

### Proof of Concept
```bash
# Install grpcurl, then flood the endpoint:
for i in $(seq 1 100000); do
  grpcurl -plaintext \
    -d '{"fileId": {"fileNum": 102}, "limit": 1}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done
```
Each concurrent call causes `NetworkServiceImpl.getNodes()` to execute `.doOnSubscribe(s -> log.info("Querying for address book: {}", filter))` at line 75, writing one `INFO` entry immediately, followed by a second `INFO` entry from `page()` at lines 100–105. With no rate limiting, the log system receives entries at the full request rate.
