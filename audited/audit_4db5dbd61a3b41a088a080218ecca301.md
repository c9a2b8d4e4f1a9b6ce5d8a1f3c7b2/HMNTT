### Title
Unbounded Linear Scan of User-Controlled `account_map` in `getSdkPayerAccountId` Enables CPU Exhaustion

### Summary
The `getSdkPayerAccountId` function in `rosetta/app/services/construction_service.go` performs an unbounded `strings.SplitSeq` linear scan over the `account_map` metadata string, which is entirely user-controlled in a `ConstructionPayloadsRequest`. There is no size limit on this string at the application layer, allowing any unauthenticated caller to craft a request with millions of comma-separated entries and force O(n) CPU work per request.

### Finding Description
**Exact code path:**

In `ConstructionPayloads` (line 243), the raw metadata value is passed directly to `getSdkPayerAccountId`:
```go
payer, rErr := c.getSdkPayerAccountId(signers[0], request.Metadata[metadataKeyAccountMap])
``` [1](#0-0) 

Inside `getSdkPayerAccountId` (lines 491–502), the string is scanned entry-by-entry with no length guard:
```go
for aliasMap := range strings.SplitSeq(accountMap, ",") {
    if !strings.HasPrefix(aliasMap, payerAlias) {
        continue
    }
    ...
    break
}
``` [2](#0-1) 

**Root cause:** The `account_map` metadata field is a free-form string accepted from the client with no length validation. The server assumes it was generated by `ConstructionMetadata`/`resolveAccountAliases`, but the client is free to substitute any string.

**Exploit flow:**
1. Attacker crafts a `ConstructionPayloadsRequest` where the first operation uses an alias-format account (so `payerAccountId.HasAlias()` returns `true`, bypassing the early return at line 471).
2. Attacker sets `metadata["account_map"]` to a string of the form `bogus1:0.0.1,bogus2:0.0.2,...` repeated hundreds of thousands of times.
3. The server iterates every entry via `strings.SplitSeq` + `strings.HasPrefix` before failing to find a match and returning `ErrAccountNotFound`.
4. No match is ever found (or the match is placed at the very end), maximizing CPU burn per request.

**Why existing checks fail:**
- The HTTP server configures `ReadTimeout`/`WriteTimeout` but no `http.MaxBytesReader` body cap. [3](#0-2) 
- The Traefik `inFlightReq: 5` and `rateLimit: average: 10` are optional Kubernetes-level configurations that are absent in bare deployments and do not limit per-request payload size. [4](#0-3) 
- There is no validation of `account_map` string length anywhere in the service code. [5](#0-4) 

### Impact Explanation
An attacker can monopolize server CPU with a small number of concurrent requests. With 5 in-flight slots (if Traefik is deployed) and a multi-megabyte `account_map` per request, the rosetta service becomes unresponsive to legitimate users. The impact is denial-of-service / griefing with no economic damage to the Hedera network itself, consistent with the Medium severity classification.

### Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication and is reachable by any HTTP client. The attack requires only knowledge of the Rosetta API spec (public) and the ability to send an HTTP POST. It is trivially repeatable and scriptable. Without application-level body size limits, a single attacker machine can sustain the attack indefinitely.

### Recommendation
1. **Add a length cap** on `account_map` before scanning. Since the normal flow produces exactly one entry per alias (generated by `resolveAccountAliases`), a small constant limit (e.g., 10 entries or 4 KB) is sufficient:
   ```go
   const maxAccountMapLen = 4096
   if len(accountMap) > maxAccountMapLen {
       return zero, errors.ErrInvalidAccount
   }
   ```
2. **Use `http.MaxBytesReader`** in a middleware to cap the entire request body (e.g., 64 KB for construction endpoints).
3. **Replace the linear scan** with a `strings.Cut`-based direct lookup after splitting on `,` only up to a bounded count, or store the map as a proper `map[string]string` in metadata rather than a serialized string.
