### Title
Unauthenticated Sequential 2-Query Chain in `getContractActions()` Enables DB Connection Pool Exhaustion via Ethereum Hash Flood

### Summary
The `getContractActions()` handler in `rest/controllers/contractController.js` accepts any valid 32-byte Ethereum hash from an unauthenticated caller and unconditionally executes two sequential database queries per request: first `getContractTransactionDetailsByHash()` against `contract_transaction_hash`, then `getContractActionsByConsensusTimestamp()` against `contract_action`. There is no application-level rate limiting on this endpoint. Under high concurrency with varied hashes (bypassing the response cache), an attacker exhausts the shared PostgreSQL connection pool, causing all mirror-node REST API requests to fail.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 22 registers the route with no auth or rate-limit middleware: [1](#0-0) 

`getContractActions()` in `rest/controllers/contractController.js` lines 1248–1269 performs the sequential chain: [2](#0-1) 

**Query 1** — `ethereumTransactionsByHashQuery` in `rest/service/contractService.js` lines 208–212 scans `contract_transaction_hash` by hash with `limit 1`: [3](#0-2) 

**Query 2** — `contractActionsByConsensusTimestampQuery` lines 153–172 scans `contract_action` by `consensus_timestamp` and `payer_account_id` with a configurable limit: [4](#0-3) 

The two queries are strictly sequential (line 1249 must complete before line 1263 can be called), so each request holds a DB connection for the combined duration of both queries.

**Root cause / failed assumption:** The design assumes that infrastructure-level controls (load balancer, WAF) will throttle request rates before the DB pool is saturated. No application-layer rate limiting exists: [5](#0-4) 

**Why the response cache is insufficient:** The cache key is the MD5 of `req.originalUrl`: [6](#0-5) 

The cache TTL defaults to 1 second: [7](#0-6) 

An attacker using distinct valid hashes (or varying the accepted `index` query parameter, e.g. `?index=gt:0`, `?index=gt:1`, …) generates unique cache keys on every request, making the cache entirely ineffective. The cache is also optional and disabled when Redis is not configured: [8](#0-7) 

**DB pool configuration:** The pool is a single shared `pg.Pool` with a finite `maxConnections`: [9](#0-8) 

Once all connections are held by in-flight `getContractActions` requests, every other API endpoint also begins queuing or timing out.

### Impact Explanation
Exhausting the shared PostgreSQL connection pool causes all mirror-node REST API endpoints (accounts, transactions, tokens, blocks, etc.) to return errors or time out. The mirror-node REST API is the primary read interface for wallets, explorers, and dApps interacting with the Hedera network. A sustained flood renders the entire REST service unavailable without touching consensus nodes.

### Likelihood Explanation
Preconditions are minimal: the attacker needs only a list of valid 32-byte Ethereum transaction hashes (publicly observable on-chain) or can use hashes that return 404 (the first query still executes and consumes a connection before the `tx.length` check at line 1256 throws `NotFoundError`). The attack is scriptable with any HTTP load tool (e.g., `wrk`, `hey`, `ab`). No credentials, tokens, or privileged access are required. The attack is repeatable and stateless.

### Recommendation
1. **Add per-IP rate limiting** at the Express middleware layer for `/contracts/results/:transactionIdOrHash/actions` (and ideally all `/contracts/results/` hash-lookup endpoints). Libraries such as `express-rate-limit` with a Redis store are appropriate.
2. **Short-circuit on non-existent hashes early**: consider a lightweight existence check with a very short DB timeout before the full query chain.
3. **Increase cache TTL** for this endpoint beyond the 1-second default, and ensure the cache is enabled in production deployments.
4. **Set a conservative `maxConnections`** and ensure `connectionTimeout` is low enough that queued requests fail fast rather than piling up.
5. **Consider parallelising** the two queries where possible, or merging them into a single JOIN query to halve connection hold time.

### Proof of Concept

```bash
# Collect N valid eth tx hashes from public mirror node or block explorer
# Then flood with distinct hashes to bypass cache:

for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/results/<hash_$i>/actions" &
done
wait

# Alternatively, vary the index param to bypass cache with a single hash:
for i in $(seq 0 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/results/<valid_hash>/actions?index=gt:$i" &
done
wait

# Expected result: DB connection pool exhausted; all subsequent API calls
# (including unrelated endpoints) return 503 / connection timeout errors.
```
