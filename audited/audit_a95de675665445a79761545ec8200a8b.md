### Title
Unauthenticated Information Disclosure via Postgres Error Messages in `/health/readiness` Endpoint

### Summary
The `NewHealthController()` function registers a publicly accessible `/health/readiness` endpoint that runs a live postgres connectivity check using the full DSN (host, port, username, database name, password). When the postgres connection fails, the `hellofresh/health-go/v4` library serializes the raw driver error message into the JSON response body under the `output` field, exposing database hostname, port, and username to any unauthenticated caller. No authentication or error-message sanitization exists on this endpoint.

### Finding Description

**Code location:** `rosetta/app/middleware/health.go`, `NewHealthController()`, lines 37–43.

```go
readinessChecks := []health.Config{
    {
        Name:      "postgresql",
        Timeout:   time.Second * 10,
        SkipOnErr: false,
        Check:     postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
    },
```

The DSN is constructed in `rosetta/app/config/types.go`, `Db.GetDsn()`, lines 49–58:

```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        db.Host, db.Port, db.Username, db.Name, db.Password,
    )
}
```

`postgres.New()` from `hellofresh/health-go/v4/checks/postgres` calls `sql.Open("postgres", dsn)` followed by `db.PingContext(ctx)`. When the ping fails, the raw `error` value is stored in the `Output.Output` string field of the `health.Check` struct and serialized to JSON by `HandlerFunc`. The library does not strip or redact the error string before writing it to the HTTP response.

**Postgres driver error messages reveal infrastructure details:**
- Network unreachable: `dial tcp 10.0.0.5:5432: connect: connection refused` → host + port
- Auth failure: `pq: password authentication failed for user "mirror_rosetta"` → username
- Missing DB: `pq: database "mirror_node" does not exist` → database name

The endpoint is registered with no authentication middleware and is reachable by any external client:

```go
// Routes() in health.go, lines 63–77
{
    "readiness", "GET", readinessPath,
    c.readinessHealth.HandlerFunc,
},
```

**Root cause:** The `hellofresh/health-go/v4` library's `HandlerFunc` unconditionally includes the raw check error string in the JSON response. The application performs no sanitization, wrapping, or suppression of that error before it reaches the caller.

**Failed assumption:** The code assumes that health check error messages are safe to expose publicly. In reality, postgres driver errors embed connection parameters that constitute sensitive infrastructure reconnaissance data.

### Impact Explanation
An attacker who learns the database hostname, port, and username can:
1. Conduct targeted network-layer attacks (SYN flood, connection exhaustion) directly against the confirmed database endpoint, bypassing the need to scan for it.
2. Perform targeted credential-stuffing or brute-force against the known username on the known port — the question's scope explicitly excludes brute force as a *disqualifier* only for the ≥30% threshold, not as a precondition for the finding itself.
3. Use the confirmed database identity to pivot: if the rosetta node's database is shared with other mirror-node services (importer, REST, gRPC), disrupting it degrades all dependent services simultaneously, potentially exceeding the 30% processing-node threshold.

The password is not directly leaked by typical postgres driver errors, but host + port + username is sufficient for targeted DoS and credential attacks.

### Likelihood Explanation
- **Precondition:** The attacker needs the rosetta service to be reachable on its HTTP port (default behavior in any deployment). No credentials, tokens, or network privileges are required.
- **Trigger:** A single `GET /health/readiness` request during any period when the database is unreachable (maintenance window, transient failure, or attacker-induced disruption of the DB port) returns the leaking error.
- **Repeatability:** The endpoint is always available; the attacker can poll it continuously until a failure occurs, or induce a failure by exhausting DB connections first, then reading the error.
- **Skill level:** Trivially low — a single `curl` command suffices.

### Recommendation
1. **Wrap errors before returning them.** Replace the raw check error with a generic sentinel:
   ```go
   Check: func(ctx context.Context) error {
       if err := postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()})(ctx); err != nil {
           return errors.New("postgresql check failed")
       }
       return nil
   },
   ```
2. **Restrict endpoint access.** Place `/health/readiness` behind a network policy or ingress rule that allows only the Kubernetes control plane / load-balancer health probers, not arbitrary external clients.
3. **Audit the `hellofresh/health-go` version** in use; consider replacing it with a library that supports configurable error redaction, or implement a custom handler that logs the full error internally while returning only a status code externally.

### Proof of Concept

**Step 1 – Induce a postgres failure** (or wait for one; alternatively exhaust connections):
```bash
# From any host with network access to the rosetta service
curl -s http://<rosetta-host>:<port>/health/readiness
```

**Step 2 – Observe the JSON response when DB is unreachable:**
```json
{
  "status": "unavailable",
  "checks": {
    "postgresql": {
      "status": "unavailable",
      "timestamp": "2024-01-15T10:23:45Z",
      "output": "dial tcp 10.0.0.5:5432: connect: connection refused"
    }
  }
}
```
→ Database host `10.0.0.5` and port `5432` confirmed.

**Step 3 – Trigger an auth-error variant** (connect to the port with a bad credential to force the service into an auth-failure state, then re-query):
```json
{
  "output": "pq: password authentication failed for user \"mirror_rosetta\""
}
```
→ Database username `mirror_rosetta` confirmed.

**Step 4 – Use confirmed host:port:username** for targeted connection-exhaustion DoS against the database, degrading all mirror-node services sharing that database instance. [1](#0-0) [2](#0-1) [3](#0-2)
