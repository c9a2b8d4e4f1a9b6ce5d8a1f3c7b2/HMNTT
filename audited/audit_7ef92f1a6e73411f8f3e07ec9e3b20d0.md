### Title
Unauthenticated gRPC Stream Allows MITM Injection of Arbitrary `consensusTimestamp` Values into Monitor History

### Summary
The `subscribe()` method in `GrpcClientSDK` returns a `Flux<SubscribeResponse>` whose `consensusTimestamp` field is taken verbatim from the gRPC stream with no cryptographic verification. The `runningHash` field present in every `ConsensusTopicResponse` is never checked, and the only sequence-number guard is a non-enforcing `log.warn`. When the gRPC transport is plaintext (port 5600, the documented default for custom/OTHER-network deployments), an unprivileged attacker on the network path can inject arbitrary `ConsensusTopicResponse` frames, falsifying every timestamp, sequence number, and message payload the monitor records.

### Finding Description

**Exact code path**

`GrpcClientSDK.client()` (lines 101–112) builds the SDK client with:
```java
Client client = Client.forNetwork(Map.of());
client.setMirrorNetwork(List.of(endpoint));   // endpoint = host:5600 for OTHER network
```
The Hedera Java SDK uses plaintext gRPC for any endpoint whose port is not 443/11206. The documentation table (`docs/configuration.md` line 459) lists `5600` as the default port for `hiero.mirror.monitor.mirrorNode.grpc.port`, and the README example (`docs/monitor/README.md` lines 29–34) explicitly shows `port: 5600`. No TLS is negotiated for this path.

**Root cause**

`GrpcClientSDK.toResponse()` (lines 70–87) maps the raw SDK `TopicMessage` to a `SubscribeResponse` by copying `topicMessage.consensusTimestamp` directly:
```java
return SubscribeResponse.builder()
        .consensusTimestamp(topicMessage.consensusTimestamp)   // ← unverified
        .publishedTimestamp(publishedTimestamp)                // ← from message bytes, also unverified
        ...
        .build();
```
The `ConsensusTopicResponse` protobuf carries a `runningHash` (SHA-384 over all prior messages) and a `sequenceNumber` that together would allow the receiver to detect tampering. Neither is ever checked in the monitor. The only guard is in `GrpcSubscription.onNext()` (lines 46–55):
```java
getLast().ifPresent(topicMessage -> {
    long expected = topicMessage.sequenceNumber + 1;
    if (topicResponse.sequenceNumber != expected) {
        log.warn("... Expected sequence number {} but received {}", expected, ...);
    }
});
```
This emits a warning log and then calls `super.onNext(topicResponse)` unconditionally — the message is accepted regardless.

**Failed assumption**

The code assumes the gRPC channel is authenticated and integrity-protected. For port 5600 (plaintext), this assumption is false. Even for port 443 (TLS), the `runningHash` is never verified, so a compromised or spoofed mirror node cannot be detected.

### Impact Explanation

An attacker who injects crafted `ConsensusTopicResponse` frames can:

1. **Falsify `consensusTimestamp`** — the monitor stores and logs these values as ground truth for Hashgraph ordering.
2. **Manipulate e2e latency metrics** — `publishedTimestamp` is parsed from `topicMessage.contents` (`Utility.getTimestamp(topicMessage.contents)`, line 72); injecting a message with a crafted payload sets an arbitrary `publishedTimestamp`, directly skewing the `hiero.mirror.monitor.subscribe.e2e` Prometheus metric used for the `MonitorSubscribeLatency` critical alert.
3. **Suppress or trigger alerts** — by injecting messages that make latency appear normal, the attacker hides real mirror-node failures; by injecting messages with extreme timestamps, the attacker triggers false critical alerts.
4. **Advance the subscription cursor** — `GrpcSubscription.getTopicMessageQuery()` (line 28) sets `startTime` from the last received `consensusTimestamp`; injecting a far-future timestamp causes the monitor to skip all real messages published after that point.

### Likelihood Explanation

For any deployment using `network: OTHER` with `port: 5600` (plaintext gRPC) — the configuration shown in the official monitor README — the attacker needs only passive network access on the path between the monitor pod and the mirror node (e.g., a compromised router, ARP spoofing on the same L2 segment, or a rogue DNS response). No credentials or elevated privileges are required. The attack is repeatable and silent: the only observable artifact is a `WARN`-level log line for sequence-number gaps, which is easily drowned out in normal log volume.

### Recommendation

1. **Enforce TLS for all gRPC connections** — change the default port to 443 and document that plaintext is not acceptable for production; add a startup check that rejects non-TLS mirror endpoints.
2. **Verify `runningHash` on every received message** — maintain a local running hash (SHA-384 per the HCS spec) and compare it against the `runningHash` field in each `ConsensusTopicResponse`; reject and alarm on any mismatch.
3. **Enforce sequence-number continuity** — change the `log.warn` in `GrpcSubscription.onNext()` to terminate the subscription and raise an error counter when a gap is detected, rather than silently accepting the out-of-order message.
4. **Pin the mirror node's TLS certificate or public key** — prevents MITM even if a CA is compromised.

### Proof of Concept

**Preconditions**: monitor configured with `network: OTHER`, `mirrorNode.grpc.host: <target>`, `mirrorNode.grpc.port: 5600` (plaintext). Attacker has ARP-spoof or routing control on the path.

1. Stand up a transparent TCP proxy (e.g., `mitmproxy` with gRPC support or a custom Netty proxy) between the monitor and the real mirror node on port 5600.
2. Forward all real frames unmodified until the subscription stream is established.
3. Inject a synthetic `ConsensusTopicResponse` frame with:
   - `consensusTimestamp` set to `now + 1 hour` (far future)
   - `sequenceNumber` set to `lastSeen + 1`
   - `message` bytes encoding a `publishedTimestamp` of `now - 30s` (to produce a fake 30s+ latency)
   - `runningHash` set to any 48 bytes (never checked)
4. Observe: the monitor logs the injected timestamp as legitimate, the `hiero.mirror.monitor.subscribe.e2e` metric spikes above the 14 s alert threshold, and `GrpcSubscription.getTopicMessageQuery()` advances `startTime` to `now + 1 hour`, causing all subsequent real messages to be silently skipped.
