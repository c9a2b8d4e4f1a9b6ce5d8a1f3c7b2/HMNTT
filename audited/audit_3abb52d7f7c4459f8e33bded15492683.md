### Title
Unauthenticated Error-Rate Amplification via Empty `account(input: {})` Queries Triggers Critical Prometheus Alert

### Summary
The `account` GraphQL query has no rate limiting and accepts unauthenticated requests. Sending `account(input: {})` at high volume causes `validateOneOf()` to throw `IllegalArgumentException` on every request, which is converted to a counted `GraphQLError`. Because the `GraphQLErrors` Prometheus alert fires when the error rate exceeds 5% for 2 minutes at `severity: critical`, a sustained flood of these cheap, zero-privilege requests can reliably trigger operator intervention, which may include pod isolation or network policy changes constituting a network partition.

### Finding Description
**Code path:**

1. `graphql/src/main/resources/graphql/query.graphqls:5` — `account(input: AccountInput!)` accepts any unauthenticated POST to `/graphql/alpha`
2. `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java:39` — `validateOneOf(alias, entityId, evmAddress, id)` is called with all four values null when `input: {}` is sent
3. `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java:68-80` — `validateOneOf` counts non-null values; when `nonNull != 1`, throws `IllegalArgumentException("Must provide exactly one input value but 0 have been provided")`
4. `graphql/src/main/java/org/hiero/mirror/graphql/config/CustomExceptionResolver.java:17-25` — `IllegalArgumentException` is resolved to a `GraphQLError` with `ErrorType.ValidationError`, which is recorded in the `graphql_request_seconds_count{status="ERROR"}` metric
5. `charts/hedera-mirror-graphql/values.yaml:170-179` — The `GraphQLErrors` Prometheus rule fires when `rate(graphql_request_seconds_count{status="ERROR"}[5m]) / rate(graphql_request_seconds_count[5m]) > 0.05` for 2 minutes at `severity: critical`

**Root cause:** The `graphql` module has no per-IP or per-request rate limiting (unlike the `web3` module which has `ThrottleConfiguration`). The `account(input: {})` query is syntactically valid, passes all parser limits (`maxCharacters(10000)`, `maxTokens(1000)`), passes complexity/depth instrumentation (complexity = 0, depth = 1), and is cached by `PreparsedDocumentProvider` — making it extremely cheap to repeat. The error is generated at the data-fetcher layer, after all structural checks pass, and is counted in error metrics.

**Failed assumption:** The design assumes that error-generating requests will be rare relative to total traffic, keeping the error rate below 5%. There is no mechanism to prevent a single unauthenticated client from making the error rate arbitrarily high.

### Impact Explanation
An attacker sending `account(input: {})` at a rate exceeding 5% of total legitimate traffic causes the `GraphQLErrors` Prometheus alert to fire as `severity: critical` after 2 minutes. This alert is annotated as "Mirror GraphQL request error rate exceeds 5%." Operator response to a critical alert may include pod restarts, network policy application, or traffic rerouting — any of which can constitute a network partition outside normal design parameters. Even without operator action, the flood consumes server threads and connection pool resources, degrading service for legitimate users.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The exploit requires only HTTP POST access to `/graphql/alpha`, which is the public API endpoint. The query `{"query":"{account(input:{}){id}}"}` is 34 bytes. A single attacker with modest bandwidth can sustain thousands of requests per second. The `PreparsedDocumentProvider` cache means the server does minimal work per request, but each still generates a counted error. The 5% threshold is easily crossed if legitimate traffic is low (e.g., off-peak hours), making the attack more effective precisely when operators are less attentive.

### Recommendation
1. **Add rate limiting to the GraphQL module**: Implement a per-IP or global request rate limiter (e.g., using Bucket4j as already used in `web3`) applied at the servlet filter level in `graphql/src/main/java/org/hiero/mirror/graphql/config/`.
2. **Raise or adjust the alert threshold**: The 5% threshold in `charts/hedera-mirror-graphql/values.yaml:175` is too sensitive to distinguish organic errors from attack traffic; consider adding a minimum absolute error count condition.
3. **Return HTTP 400 before the GraphQL layer**: Reject structurally empty inputs at the HTTP filter level so they are not counted as GraphQL-layer errors.
4. **Add connection/request limiting at the ingress/load balancer level** as noted in `docs/design/contract-log-subscription.md:37` ("We should implement rate limiting in the load balancer") — this recommendation exists but has not been applied to the query endpoint.

public class CustomExceptionResolver extends DataFetcherExceptionResolverAdapter {
    @Override
    protected GraphQLError resolveToSingleError(Throwable ex, DataFetchingEnvironment env) {
        if (ex instanceof IllegalStateException
                || ex instanceof IllegalArgumentException
                || ex instanceof MirrorNodeException) {
            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())
                    .location(env.getField().getSourceLocation())
                    .build();
        } else {
            return null;
        }
    }
