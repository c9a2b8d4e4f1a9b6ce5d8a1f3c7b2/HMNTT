### Title
Unbounded Multi-Connection Executor Saturation via Shared `applicationTaskExecutor` in gRPC Server

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration` assigns Spring Boot's shared `applicationTaskExecutor` to the Netty gRPC server and enforces only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`). No limit on the total number of inbound connections is configured, and no authentication or rate-limiting is applied to the gRPC endpoint. An unprivileged attacker can open an arbitrary number of connections — each saturating its 5-call quota — to collectively exhaust the shared executor with sustained protobuf serialization/deserialization work, causing a 30%+ CPU increase.

### Finding Description

**Exact code path:**

`GrpcConfiguration.grpcServerConfigurer()` (lines 28–35) wires the shared Spring `applicationTaskExecutor` to the Netty gRPC server and sets only a per-connection call cap:

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [1](#0-0) 

`NettyProperties` defaults `maxConcurrentCallsPerConnection` to 5 and exposes no `maxConnections` or per-IP limit: [2](#0-1) 

`ConsensusController.subscribeTopic()` is publicly accessible (no auth interceptor) and streams messages indefinitely when `limit=0`: [3](#0-2) 

Each streamed message triggers `toResponse()`, which builds a full protobuf object and calls `TransactionID.parseFrom(transactionIdBytes)` — protobuf deserialization — per message: [4](#0-3) [5](#0-4) 

**Root cause:** The design assumes `maxConcurrentCallsPerConnection` is a sufficient resource guard. It is not — it is scoped per TCP connection only. Because no total connection limit, no per-IP connection limit, and no gRPC-layer rate limiter exist, an attacker multiplies load by opening N connections × 5 calls each, all dispatched onto the single shared `applicationTaskExecutor`.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` is enforced by Netty per-connection; it does not bound total concurrent calls across all connections.
- The `ThrottleConfiguration`/`ThrottleManagerImpl` with Bucket4j rate limiting exists only in the `web3` module — it is absent from the `grpc` module.
- No authentication interceptor is registered on the gRPC server.
- The `maxBufferSize` (16384) and `maxPageSize` (5000) in `ListenerProperties` bound per-subscription buffering but do not prevent many subscriptions from running simultaneously. [6](#0-5) 

### Impact Explanation
The shared `applicationTaskExecutor` services all Spring async work for the JVM process. Saturating it with streaming gRPC calls causes sustained CPU consumption from repeated protobuf serialization (`ConsensusTopicResponse.newBuilder().build()`) and deserialization (`TransactionID.parseFrom()`), plus reactive pipeline overhead per message. Legitimate subscribers experience degraded throughput or timeouts. The `GrpcHighCPU` alert threshold is 80%; an attacker targeting 30%+ degradation stays below that threshold and avoids automated remediation. [7](#0-6) 

### Likelihood Explanation
The attack requires no credentials, no special tooling beyond a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), and no knowledge of internal state. The `subscribeTopic` endpoint is publicly documented and reachable on port 5600. Opening tens of connections with 5 streaming calls each is trivially scriptable and repeatable. The attacker can sustain the load indefinitely since `limit=0` subscriptions never self-terminate. [8](#0-7) 

### Recommendation
1. **Add a total connection limit** to `NettyProperties` and apply it via `serverBuilder.maxConnectionAge(...)` or a custom `ServerInterceptor` that tracks and rejects connections beyond a global cap.
2. **Add a gRPC-layer rate limiter** (analogous to `ThrottleConfiguration` in `web3`) that limits `subscribeTopic` calls per source IP per time window using Bucket4j or a similar token-bucket mechanism.
3. **Isolate the gRPC executor** from the shared `applicationTaskExecutor` by creating a dedicated, bounded `ThreadPoolExecutor` for gRPC work, preventing gRPC load from starving other Spring async tasks.
4. **Enforce `maxInboundMessageSize`** on the server builder to bound per-message deserialization cost.

### Proof of Concept

```bash
# Open 20 connections, each with 5 concurrent unlimited subscriptions (100 total calls)
# Stays within maxConcurrentCallsPerConnection=5 per connection
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# Monitor CPU: sustained 30%+ increase expected on the mirror node process
# Each call streams messages indefinitely, triggering toResponse() + TransactionID.parseFrom() per message
```

    public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
        final var disposable = Mono.fromCallable(() -> toFilter(request))
                .flatMapMany(topicMessageService::subscribeTopic)
                .map(this::toResponse)
                .onErrorMap(ProtoUtil::toStatusRuntimeException)
                .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

        if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
            serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
        }
    }

    private ConsensusTopicResponse toResponse(TopicMessage t) {
        final var consensusTopicResponseBuilder = ConsensusTopicResponse.newBuilder()
                .setConsensusTimestamp(ProtoUtil.toTimestamp(t.getConsensusTimestamp()))
                .setMessage(ProtoUtil.toByteString(t.getMessage()))
                .setRunningHash(ProtoUtil.toByteString(t.getRunningHash()))
                .setRunningHashVersion(
                        Objects.requireNonNullElse(t.getRunningHashVersion(), DEFAULT_RUNNING_HASH_VERSION))
                .setSequenceNumber(t.getSequenceNumber());

        if (t.getChunkNum() != null) {
            ConsensusMessageChunkInfo.Builder chunkBuilder = ConsensusMessageChunkInfo.newBuilder()
                    .setNumber(t.getChunkNum())
                    .setTotal(t.getChunkTotal());

            TransactionID transactionID = parseTransactionID(
                    t.getInitialTransactionId(), t.getTopicId().getNum(), t.getSequenceNumber());
            EntityId payerAccountEntity = t.getPayerAccountId();
            var validStartInstant = ProtoUtil.toTimestamp(t.getValidStartTimestamp());

            if (transactionID != null) {
                chunkBuilder.setInitialTransactionID(transactionID);
            } else if (payerAccountEntity != null && validStartInstant != null) {
                chunkBuilder.setInitialTransactionID(TransactionID.newBuilder()
                        .setAccountID(payerAccountEntity.toAccountID())
                        .setTransactionValidStart(validStartInstant)
                        .build());
            }

            consensusTopicResponseBuilder.setChunkInfo(chunkBuilder.build());
        }

        return consensusTopicResponseBuilder.build();
    }

    private TransactionID parseTransactionID(byte[] transactionIdBytes, long topicId, long sequenceNumber) {
        if (transactionIdBytes == null) {
            return null;
        }
        try {
            return TransactionID.parseFrom(transactionIdBytes);
        } catch (InvalidProtocolBufferException e) {
            log.error("Failed to parse TransactionID for topic {} sequence number {}", topicId, sequenceNumber);
            return null;
        }
    }
