### Title
Cache Bypass via Unbounded Timestamp Key Space in `SystemFileLoader.load()` Enables DB Connection Pool Exhaustion

### Summary
The `load(FileID, long)` method in `SystemFileLoader` applies `roundDownToHour()` normalization only for exchange-rate, fee-schedule, and simple-fee-schedule files, but not for `throttleDefinitionFile`, `addressBookFile101`, `addressBookFile102`, or `hapiPermissionFile`. An unprivileged attacker can send historical contract-call requests with distinct block numbers, each mapping to a unique `consensusTimestamp`, generating unique `CacheKey(fileId, consensusTimestamp)` objects that perpetually miss the `defaultSystemFileCacheManager` (maximumSize=20). Each cache miss triggers a synchronous DB query, and at the default rate limit of 500 RPS the attacker can sustain 500 uncached DB queries per second for system file lookups, degrading or exhausting the database connection pool.

### Finding Description

**Exact code path:**

In `SystemFileLoader.java`, `load()` at lines 113–118 applies timestamp rounding only for three specific file IDs:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);  // ← rounding applied
}
``` [1](#0-0) 

For all other system files — including `throttleDefinitionFile`, `addressBookFile101`, `addressBookFile102`, and `hapiPermissionFile` — the raw `consensusTimestamp` is used directly to construct the cache key at line 120:

```java
final var cacheKey = new CacheKey(fileId, consensusTimestamp);
``` [2](#0-1) 

The `defaultSystemFileCacheManager` is configured with `maximumSize=20`: [3](#0-2) 

On every cache miss, `loadFromDB()` → `loadWithRetry()` is called, which executes `fileDataRepository.getFileAtTimestamp()` — a synchronous DB query. In the normal (non-corrupt) case this is 1 query per miss; in the corrupt-data case the `RetryTemplate` allows up to `maxFileAttempts=12` queries: [4](#0-3) [5](#0-4) 

**Root cause:** The design assumes that historical queries will cluster around a small number of timestamps, but there is no enforcement of this assumption for non-exchange-rate system files. The cache key is unbounded in the timestamp dimension for these files.

**Exploit flow:**
1. Attacker identifies any system file that uses `defaultSystemFileCacheManager` (e.g., `throttleDefinitionFile`).
2. Attacker sends historical `eth_call` / `/api/v1/contracts/call` requests, each specifying a distinct block number. The `block` field is user-controlled: [6](#0-5) 

3. Each unique block number maps to a unique `consensusTimestamp` via `RecordFile.consensusEnd`, producing a unique `CacheKey`.
4. With 21+ distinct timestamps, the cache (maximumSize=20) is perpetually cold for all keys.
5. Every request triggers a DB query. At the default 500 RPS rate limit, this sustains ≥500 uncached DB queries/second for system file lookups alone.

**Why existing checks are insufficient:**

The rate limiter at 500 RPS: [7](#0-6) 

...is the primary defense, but it was designed to limit total request throughput, not to prevent cache-bypass amplification. It still permits 500 DB queries/second for system file lookups that should be served from cache. The `statementTimeout=3000ms` limits individual query duration but does not prevent connection pool saturation under sustained load. The gas-per-second throttle does not help because system file loading occurs outside the gas accounting path.

### Impact Explanation
Under sustained attack at the default 500 RPS rate limit, the DB connection pool receives 500 additional queries/second for system file data that should be cached. This degrades query latency for all other operations sharing the pool, and can exhaust available connections entirely, causing a non-network DoS for all users of the mirror node web3 service. The impact is amplified if the attacker targets multiple system files simultaneously (up to 4 files × 500 RPS = 2000 queries/second).

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no on-chain transactions. Any entity with HTTP access to the `/api/v1/contracts/call` endpoint can execute it. The only precondition is knowledge of valid historical block numbers, which are publicly available. The attack is trivially repeatable and automatable with a simple script cycling through block numbers.

### Recommendation
Apply the same `roundDownToHour()` (or a coarser granularity appropriate to how frequently these files change) normalization to all system files, not just the three exchange-rate/fee-schedule files. Alternatively, use a single shared cache key per file ID for non-historical (latest) calls, and for historical calls normalize the timestamp to the nearest file-update boundary. A secondary defense is to reduce `requestsPerSecond` for historical calls or add a dedicated rate limit for historical system-file-loading paths.

### Proof of Concept
```

# Step 1: Enumerate valid block numbers (publicly available from /api/v1/blocks)
# Step 2: Send 500 concurrent requests per second, each with a unique block number
# targeting a contract that internally reads throttle definitions

for i in $(seq 1 10000); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"block\": \"$i\", \"to\": \"0x0000000000000000000000000000000000000169\", \"data\": \"0x\"}" &
done

# Each request with a unique block number i maps to a unique consensusTimestamp,
# generating CacheKey(throttleDefinitionFileId, consensusTimestamp_i).
# With >20 unique timestamps, the cache (maximumSize=20) is perpetually cold.
# Each request triggers fileDataRepository.getFileAtTimestamp() — a live DB query.
# At 500 RPS, this sustains 500 uncached DB queries/second for system file lookups.
# Monitor DB connection pool exhaustion via pg_stat_activity or connection pool metrics.
```
