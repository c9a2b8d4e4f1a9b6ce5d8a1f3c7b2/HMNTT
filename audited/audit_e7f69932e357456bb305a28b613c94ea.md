### Title
Integer Narrowing Cast Overflow in `PollingTopicListener.poll()` Enables Unbounded Database Queries via `limit = Long.MAX_VALUE`

### Summary
In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` performs an unsafe narrowing cast from `long` to `int`. When an unprivileged client submits a gRPC `ConsensusTopicQuery` with `limit = Long.MAX_VALUE` (a valid `uint64` value that passes all input validation), the cast produces `-1`. This causes `findByFilter()` to execute SQL queries with no `LIMIT` clause on every poll interval, enabling a denial-of-service attack via unbounded database reads.

### Finding Description

**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `poll()`, lines 54â€“57.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // line 55 â€” unsafe cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());  // line 57
``` [1](#0-0) 

**Root cause:** `Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF`. Casting to `int` retains only the lower 32 bits: `0xFFFFFFFF = -1` (signed). So `limit = -1` and `Math.min(-1, 5000) = -1`.

**Propagation to repository:** The computed `pageSize = -1` is set on a new filter via `filter.toBuilder().limit(-1).build()` and passed to `TopicMessageRepositoryCustomImpl.findByFilter()`. [2](#0-1) 

Inside `findByFilter()`, the guard is `filter.hasLimit()` which returns `limit > 0`. With `limit = -1`, `hasLimit()` returns `false`, so `typedQuery.setMaxResults()` is **never called**, and the JPA query executes with no SQL `LIMIT` clause. [3](#0-2) 

This unbounded query fires on every poll interval (default 500 ms). [4](#0-3) 

**Why existing checks fail:**

- `TopicMessageFilter.limit` carries `@Min(0)`, which only rejects negative values. `Long.MAX_VALUE` is positive and passes.
- The protobuf field `limit` is `uint64`, so `Long.MAX_VALUE` is a perfectly legal wire value a client can send.
- The `@Validated` annotation on `TopicMessageFilter` is a Spring class-level marker; it does not trigger Bean Validation at Lombok builder `.build()` time â€” only when the object is passed to a Spring-managed method with `@Valid` on the parameter, which does not occur here. [5](#0-4) [6](#0-5) 

The same integer overflow pattern exists identically in `PollingTopicMessageRetriever.poll()` (lines 68â€“71), which is invoked for **all** subscriber types (not just `POLL` listener mode), broadening the attack surface. [7](#0-6) 

### Impact Explanation

Every 500 ms, the polling loop issues a SQL query against the `topic_message` table with no row limit. For a high-volume topic, this returns millions of rows per query, consuming database CPU, I/O, network bandwidth, and JVM heap. A single malicious subscription can saturate the database connection pool and degrade or crash the mirror node for all other users. The `flux.take(Long.MAX_VALUE)` guard in `TopicMessageServiceImpl` does not prevent the per-poll unbounded DB reads; it only caps the total items emitted to the subscriber. [8](#0-7) 

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. Any client can set `limit` to `Long.MAX_VALUE` in a single `ConsensusTopicQuery` message using any standard gRPC client or `grpcurl`. No special privileges, credentials, or knowledge of internal state are needed. The attack is trivially repeatable and can be parallelized across many connections. [9](#0-8) 

### Recommendation

1. **Fix the cast:** Replace the unsafe narrowing cast with a bounds-safe computation:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an upper bound on `limit`:** Add a `@Max` constraint (e.g., `@Max(Integer.MAX_VALUE)`) to `TopicMessageFilter.limit`, or enforce a server-side cap before the filter reaches the listener.
3. **Apply the same fix** to `PollingTopicMessageRetriever.poll()` at lines 68â€“69, which has the identical pattern.
4. **Enforce validation at build time:** Add explicit `Validator` invocation after `filter.build()` in `ConsensusController.toFilter()` to ensure Bean Validation constraints are actually checked.

### Proof of Concept

```bash
# Using grpcurl â€” no credentials required
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**What happens:**
1. `ConsensusController.toFilter()` sets `filter.limit = Long.MAX_VALUE` (passes `@Min(0)`).
2. `TopicMessageServiceImpl.incomingMessages()` passes `limit = Long.MAX_VALUE` to `topicListener.listen()`.
3. `PollingTopicListener.poll()` computes `(int)(Long.MAX_VALUE - 0) = -1`; `Math.min(-1, 5000) = -1`.
4. `findByFilter()` receives `limit = -1`; `hasLimit()` returns `false`; no `LIMIT` clause is added to the SQL query.
5. Every 500 ms, a full table scan of `topic_message` for that topic is executed, exhausting database and JVM resources.
