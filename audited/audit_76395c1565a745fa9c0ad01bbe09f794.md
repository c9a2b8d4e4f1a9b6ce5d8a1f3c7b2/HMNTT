### Title
Unbounded URL Logging in `TracingMiddleware` Enables Log Storage Exhaustion by Unauthenticated Attackers

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` captures `request.URL.RequestURI()` into `path` and formats it directly into a log message via `fmt.Sprintf` with no length truncation. Because the Rosetta HTTP server sets no `MaxHeaderBytes` (defaulting to Go's 1 MB limit) and no rate limiting, an unauthenticated attacker can flood the log subsystem with near-maximum-size entries, exhausting disk space and causing legitimate transaction-submission log records to be lost or overwritten.

### Finding Description
**Exact location:** `rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47 and 52–53.

```go
// line 47
path := request.URL.RequestURI()          // unbounded, attacker-controlled

// lines 52-53
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)  // path written to log with no cap
```

`path` is taken verbatim from the raw request URI. Go's `net/http` server enforces `MaxHeaderBytes` (default `1 << 20` = 1 MB) on the combined request-line + headers, so a single URL can be up to ~1 MB. Because `TracingMiddleware` applies no truncation before calling `log.Info(message)`, every such request produces a ~1 MB log entry.

`getClientIpAddress()` (lines 63–74) also reads `X-Real-IP` / `X-Forwarded-For` without length limits and injects them into the same message, compounding the per-entry size.

The server setup in `rosetta/main.go` lines 220–227 sets `ReadHeaderTimeout`, `ReadTimeout`, etc., but **never sets `MaxHeaderBytes`**, leaving it at the 1 MB default. No rate-limiting middleware is applied anywhere in the chain (`metricsMiddleware → tracingMiddleware → corsMiddleware`).

**Failed assumption:** The code assumes request URIs are short enough that logging them verbatim is safe. There is no guard.

### Impact Explanation
An attacker who floods the server with requests carrying ~1 MB URIs generates ~1 MB log entries at `Info` level for every non-internal path. At scale this:
- Exhausts available disk space, causing the logger (logrus writing to stdout/file) to block or drop entries.
- Triggers log rotation that overwrites legitimate records of transaction submissions made via the Rosetta Construction API (`/construction/submit`), destroying the audit trail for gossip-submitted transactions.
- Can degrade server performance if the logging I/O path becomes a bottleneck.

Severity: **Medium–High** (availability of audit log; no direct code execution or data exfiltration, but loss of transaction-submission records is operationally significant for a blockchain mirror node).

### Likelihood Explanation
- **No authentication required** — the Rosetta API is publicly reachable on port 5700.
- **No rate limiting** — the middleware chain has no request-rate or connection-rate controls.
- **Trivially scriptable** — a single `curl` loop or `ab`/`wrk` run with a long `?q=AAAA…` query string suffices.
- **Repeatable** — the attacker can sustain the attack indefinitely from a single host.

### Recommendation
Truncate `path` before logging:

```go
const maxLoggedPathLen = 512

path := request.URL.RequestURI()
loggedPath := path
if len(loggedPath) > maxLoggedPathLen {
    loggedPath = loggedPath[:maxLoggedPathLen] + "…"
}
// use loggedPath in fmt.Sprintf
```

Apply the same cap to `clientIpAddress`. Additionally, set an explicit `MaxHeaderBytes` on the `http.Server` struct in `rosetta/main.go` (e.g., `MaxHeaderBytes: 8 * 1024`) and consider adding a rate-limiting middleware.

### Proof of Concept
```bash
# Generate a ~900 KB query string and hammer the server
LONG=$(python3 -c "print('A'*900000)")
while true; do
  curl -s "http://<rosetta-host>:5700/network/list?q=${LONG}" &
done
```

Each request that reaches the middleware produces a ~900 KB `Info`-level log entry. Running this for a few minutes with modest concurrency will fill a typical log partition or cause log rotation to discard legitimate `/construction/submit` entries.
