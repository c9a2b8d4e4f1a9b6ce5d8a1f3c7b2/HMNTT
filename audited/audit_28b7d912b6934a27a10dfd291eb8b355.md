### Title
Log Injection via Unsanitized Unicode Control Characters in `X-Real-IP` / `X-Forwarded-For` Headers

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` reads the client IP address directly from the `X-Real-IP` or `X-Forwarded-For` HTTP headers with no content sanitization beyond a length check. An unprivileged attacker can embed Unicode bidirectional control characters (e.g., U+202E RIGHT-TO-LEFT OVERRIDE) in these headers, which are then interpolated verbatim into the log message and written to the log sink, causing Unicode-aware log viewers (terminals, Kibana, Grafana Loki, etc.) to render the log line in reverse order and mislead operators.

### Finding Description
**Code path:**

- `getClientIpAddress()` (lines 63–74): reads `r.Header.Get(xRealIpHeader)` / `r.Header.Get(xForwardedForHeader)` and returns the raw string with only a `len(ipAddress) == 0` guard — no format or character validation.
- `TracingMiddleware()` (lines 52–53): interpolates the unsanitized value directly into the log message via `fmt.Sprintf("%s %s %s (%d) in %s", clientIpAddress, ...)`.
- The resulting string is passed to `log.Info(message)` / `log.Debug(message)` (lines 56–58) without any escaping.

**Root cause:** The code assumes the IP header value is a well-formed ASCII IP address string. It never validates the character content of the header value, so arbitrary bytes — including multi-byte UTF-8 sequences for Unicode control characters — flow directly into the log output.

**Exploit flow:**
1. Attacker sends an HTTP request to any non-internal Rosetta endpoint (no authentication required).
2. Request includes `X-Real-IP: \u202Emalicious content` (U+202E encoded as UTF-8: `0xE2 0x80 0xAE`).
3. Go's `net/http` server does not reject non-ASCII bytes in header values (only rejects bytes < 0x20 except tab, and 0x7F); the header value passes through.
4. `getClientIpAddress()` returns the raw string containing U+202E.
5. `fmt.Sprintf` embeds it as the first field of the log message.
6. `log.Info` writes the message to the log sink.
7. Any Unicode-aware log viewer renders all text after U+202E right-to-left, reversing the visible log line and hiding or inverting the true path, method, and status code.

**Why existing checks fail:** The only guard is `len(ipAddress) == 0` (lines 66, 70), which is satisfied by a non-empty string containing control characters. There is no `net.ParseIP()` call, no regex validation, and no Unicode stripping.

### Impact Explanation
Operators monitoring logs in Unicode-aware terminals or log aggregation UIs (Kibana, Loki, Splunk) will see log lines rendered in reverse order. An attacker can craft the header so that the reversed rendering makes a malicious or error-producing request appear to be a normal successful request (e.g., hiding a 500 status code or a sensitive path). This is a log-forging / log-injection attack (CWE-117). No economic damage occurs, but it degrades the integrity of audit logs and can mislead incident responders.

### Likelihood Explanation
- The Rosetta API is a public-facing HTTP service with no authentication on standard endpoints.
- The attack requires only a single crafted HTTP request with a custom header — trivially repeatable with `curl` or any HTTP client.
- If the deployment sits behind a reverse proxy that strips or overwrites `X-Real-IP`/`X-Forwarded-For`, the header vector is blocked; however, many deployments pass these headers through, and the URL path (`request.URL.RequestURI()`, line 47) is always attacker-controlled and subject to the same unsanitized interpolation if raw non-ASCII bytes reach the parser.
- No special privileges, credentials, or network position are required.

### Recommendation
1. Validate the IP address string with `net.ParseIP()` before using it; fall back to `r.RemoteAddr` if parsing fails.
2. Strip or escape all non-printable and non-ASCII characters from any user-supplied string before logging, e.g., using `strings.Map` to remove runes outside `[0x20, 0x7E]`.
3. Use structured logging (logrus fields) rather than a single interpolated string, so log sinks can handle field values safely without Unicode rendering issues.

### Proof of Concept
```bash
# Send a request with U+202E (RTL override) embedded in X-Real-IP
# UTF-8 encoding of U+202E is 0xE2 0x80 0xAE
curl -s \
  -H $'X-Real-IP: \xe2\x80\xaeATTACKER_CONTROLLED_TEXT' \
  http://<rosetta-host>/network/list \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# In a Unicode-aware terminal or log viewer, the logged line will appear as:
# ATTACKER_CONTROLLED_TEXT (reversed rendering) GET /network/list (200) in Xms
# instead of the true:
# <ip> GET /network/list (200) in Xms
```

The root cause is at `rosetta/app/middleware/trace.go` lines 64–68 (no content validation on the header value) and lines 52–53 (unsanitized interpolation into the log message).
