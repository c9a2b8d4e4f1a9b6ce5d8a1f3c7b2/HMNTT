### Title
Unauthenticated Resource Exhaustion via Unthrottled, Uncached Aggregation Query on `/api/v1/network/supply`

### Summary
The `GET /api/v1/network/supply` endpoint in the `rest-java` module accepts requests from any unauthenticated user and, when called without a `timestamp` parameter, executes a full-table aggregation query (`sum(e.balance)` with an `unnest`-based range join across 7 account ranges) against the `entity` table on every single request. The `rest-java` module has no rate limiting and no response caching for this endpoint, meaning an attacker can pipeline many concurrent requests — trivially over a single HTTP/2 connection — to continuously saturate database worker threads and raise DB CPU/IO consumption well above 30% compared to baseline.

### Finding Description

**Code path:**

1. `NetworkController.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 132–150 — accepts `GET /api/v1/network/supply` with no authentication and no rate-limit guard. [1](#0-0) 

2. `NetworkServiceImpl.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, lines 59–89 — when `timestamp.isEmpty()` (no timestamp parameter supplied), calls `entityRepository.getSupply(lowerBounds, upperBounds)` directly with no cache check. [2](#0-1) 

3. `EntityRepository.getSupply()` — `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, lines 19–29 — executes a native SQL aggregation: `sum(e.balance)` with `join unnest(... bigint[], ... bigint[]) as ranges(min_val, max_val) on e.id between ranges.min_val and ranges.max_val`. This is a full sequential scan + aggregation across 7 ranges on the `entity` table, with no `LIMIT`. [3](#0-2) 

**Root cause — two missing controls:**

- **No rate limiting in `rest-java`**: The `ThrottleConfiguration`, `ThrottleManagerImpl`, and `ThrottleProperties` classes exist exclusively in the `web3` module. A search for any rate-limiting filter, interceptor, or annotation in `rest-java/src/main/**/*.java` returns no results. `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter. [4](#0-3) 

- **No caching**: A search for `@Cacheable`, `@CacheEvict`, or `CacheManager` in `rest-java/src/main/**/*.java` returns zero matches. Every call to `getSupply()` with no timestamp hits the database unconditionally.

**HTTP/2 multiplexing angle**: The nginx reverse proxy has `http2 on` enabled and routes `/api/v1/network/` to the `rest-java` host. [5](#0-4) 
HTTP/2 allows hundreds of concurrent streams over a single TCP connection. Even if a hypothetical per-connection limit existed at the proxy layer, HTTP/2 multiplexing would bypass it entirely, allowing a single attacker IP to issue many simultaneous in-flight aggregation queries.

**Only partial mitigation present**: The `hiero.mirror.restJava.db.statementTimeout` defaults to 10,000 ms (10 seconds). [6](#0-5) 
This limits individual query duration but does not bound concurrency. With a default HikariCP pool, an attacker can keep all pool connections occupied with 10-second aggregation queries, starving legitimate requests and maximizing DB CPU utilization continuously.

### Impact Explanation
On a production mainnet database with millions of entity rows, the `sum(e.balance)` aggregation across 7 ranges requires a full sequential scan of the `entity` table on every unauthenticated request. Sustained concurrent flooding (e.g., 20–50 parallel requests) will:
- Saturate all HikariCP DB connections, causing legitimate API requests to queue or time out.
- Drive DB CPU and I/O to saturation, easily exceeding a 30% increase in resource consumption relative to the preceding 24-hour baseline.
- Degrade all other endpoints sharing the same database, including importer operations.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no rate limit are required. Any external user with network access to the public mirror node API can exploit this with a trivial script (e.g., `curl --http2` in a loop, or any HTTP/2 client library). The attack is repeatable, stateless, and requires no prior knowledge of the system beyond the public API documentation. The `unreleasedSupplyAccounts` configuration is fixed and public. [7](#0-6) 

### Recommendation
1. **Add response caching**: Apply a short-lived (e.g., 5–30 second) in-process cache (Caffeine) on `NetworkServiceImpl.getSupply()` for the no-timestamp path, since the result changes only when new balance snapshots are ingested.
2. **Add rate limiting to `rest-java`**: Implement a global request-rate filter (e.g., Bucket4j, analogous to the `web3` module's `ThrottleConfiguration`) covering all endpoints in the `rest-java` module, with per-IP limits enforced at the filter level.
3. **Add a DB-level concurrency guard**: Use a PostgreSQL advisory lock or a semaphore in the service layer to limit the number of simultaneous `getSupply` aggregation queries in flight.
4. **Consider a materialized/pre-computed supply value**: Schedule a background job to periodically compute and store the unreleased supply, serving the cached value from the endpoint instead of re-aggregating on every request.

### Proof of Concept
```bash
# Open a single HTTP/2 connection and pipeline 50 concurrent supply requests
# Requires curl >= 7.47 with HTTP/2 support
for i in $(seq 1 50); do
  curl --http2 -s "https://<mirror-node-host>/api/v1/network/supply" &
done
wait
```
Or with a Python HTTP/2 client (httpx):
```python
import httpx, asyncio

async def flood():
    async with httpx.AsyncClient(http2=True) as client:
        tasks = [client.get("https://<mirror-node-host>/api/v1/network/supply")
                 for _ in range(100)]
        await asyncio.gather(*tasks)

asyncio.run(flood())
```
Observe DB CPU and active connection count via the Prometheus metrics (`hikaricp_connections_active`, `process_cpu_usage`) rising sharply during the attack window, consistent with a 30%+ resource increase.
