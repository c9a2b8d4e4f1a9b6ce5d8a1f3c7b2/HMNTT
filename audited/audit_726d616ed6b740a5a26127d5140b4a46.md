### Title
Uncached Historical Storage Queries Enable DB Resource Amplification via Historical `eth_call`

### Summary
`ContractStateServiceImpl.findStorageByBlockTimestamp()` contains no caching layer at either the service or repository level, unlike its counterpart `findStorage()` which carries a `@Cacheable` annotation. Any unprivileged user can issue `eth_call` requests targeting historical block numbers, causing EVM execution to trigger one raw `contract_state_change` table range-scan per storage slot accessed — with zero cache benefit — bounded only by a global (not per-IP) rate limiter that does not account for per-request DB query amplification.

### Finding Description

**Exact code path:**

`ContractStorageReadableKVState.readFromDataSource()` branches on whether a timestamp is present in `ContractCallContext`:

```
// ContractStorageReadableKVState.java:41-44
timestamp
    .map(t -> contractStateService.findStorageByBlockTimestamp(..., t))
    .orElse(contractStateService.findStorage(...))
```

When a historical block is requested, every storage slot read during EVM execution calls `ContractStateServiceImpl.findStorageByBlockTimestamp()`:

```java
// ContractStateServiceImpl.java:73-76
public Optional<byte[]> findStorageByBlockTimestamp(
        final EntityId entityId, final byte[] slotKeyByteArray, final long blockTimestamp) {
    return contractStateRepository.findStorageByBlockTimestamp(entityId.getId(), slotKeyByteArray, blockTimestamp);
}
```

This directly executes a native range-scan query with no `@Cacheable` annotation:

```sql
-- ContractStateRepository.java:44-54
select coalesce(value_written, value_read) as value
from contract_state_change
where contract_id = ?1
  and slot = ?2
  and consensus_timestamp <= ?3
order by consensus_timestamp desc
limit 1
```

**Contrast with the non-historical path** (`findStorage()`), which carries `@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_CONTRACT_STATE)` at the repository level (line 20 of `ContractStateRepository.java`), and additionally uses a batch-preload cache in `ContractStateServiceImpl` (lines 85-122). Repeated calls for the same slot return from memory. No such protection exists for the historical path.

**Root cause:** The design assumption is that historical queries are rare or that the gas limit sufficiently constrains them. Neither holds under adversarial conditions.

**Why the rate limiter is insufficient:**

`ThrottleManagerImpl.throttle()` checks a single global token bucket (`rateLimitBucket`, default 500 req/s) and a global gas bucket (7.5B gas/s). Both are global — not per-IP, not per-user. A single attacker can legally consume the entire 500 req/s budget. Each request can access many storage slots (a contract with 50 storage reads per call produces 50 uncached DB queries per request). At 500 req/s × 50 slots = **25,000 uncached range-scans per second** against `contract_state_change`. The gas limit does not prevent this: a SLOAD costs 100–2,100 gas; at 7.5B gas/s the gas budget alone would permit millions of SLOADs per second, far exceeding the request-rate constraint.

### Impact Explanation

Each uncached `findStorageByBlockTimestamp()` call executes a `consensus_timestamp <= ?` range-scan ordered descending on the `contract_state_change` table. On a production node with millions of historical state changes, this is a non-trivial I/O operation. Sustained at 25,000+ queries/second from a single attacker, this can saturate DB connection pools, increase query latency for all users, and degrade or deny service to legitimate callers — meeting the ">30% resource increase" threshold without brute force. The impact is amplified because the global rate limiter does not distinguish historical from latest-block calls, so an attacker targeting historical blocks imposes disproportionate DB cost relative to their allowed request quota.

### Likelihood Explanation

No authentication or special privilege is required. Any user with network access to the web3 JSON-RPC endpoint can issue `eth_call` with a historical block parameter (e.g., `"block": "0x1"`). The attack is trivially scriptable: send 500 `eth_call` requests per second targeting a storage-heavy contract at varying block numbers. The varying `blockTimestamp` values ensure that even if caching were naively added at the service level with a composite key, the high-cardinality timestamp dimension would defeat it. The attack is repeatable indefinitely and requires no on-chain funds or privileged credentials.

### Recommendation

1. **Add a bounded cache for historical storage lookups.** Introduce a time-bounded, size-bounded cache keyed on `(contractId, slot, blockTimestamp)` in `ContractStateServiceImpl.findStorageByBlockTimestamp()`. Because `blockTimestamp` values are discrete (tied to consensus timestamps of actual blocks, not arbitrary nanosecond values), the effective key space is bounded by the number of distinct blocks, making caching practical.

2. **Apply per-IP rate limiting** in addition to the global bucket. The current global `rateLimitBucket` in `ThrottleManagerImpl` does not prevent a single client from monopolizing the budget.

3. **Differentiate historical vs. latest-block throttle budgets.** Historical queries are more expensive; apply a stricter per-second limit specifically for requests that include a historical block parameter.

4. **Validate that `blockTimestamp` corresponds to an actual recorded block** before executing the query, rejecting arbitrary nanosecond values that would never match any `consensus_timestamp` but still incur a full table scan.

### Proof of Concept

```python
import requests, threading

URL = "http://<mirror-node>:8545"
CONTRACT = "0x<storage-heavy-contract-address>"
# ABI-encoded call that reads many storage slots
DATA = "0x<selector-for-multi-sload-function>"

def flood(block_num):
    while True:
        requests.post(URL, json={
            "jsonrpc": "2.0", "method": "eth_call",
            "params": [{"to": CONTRACT, "data": DATA}, hex(block_num)],
            "id": 1
        })
        block_num += 1  # vary timestamp → bypass any future naive cache

threads = [threading.Thread(target=flood, args=(i * 1000,)) for i in range(10)]
for t in threads: t.start()
# Observe: DB CPU/IO rises proportionally; legitimate requests slow or time out
```

**Preconditions:** Network access to the JSON-RPC endpoint; a deployed contract that reads ≥10 storage slots per call (trivially constructable or already present on mainnet).
**Trigger:** Run the script above.
**Result:** `contract_state_change` table scan rate spikes; DB resource consumption increases >30% within seconds; legitimate `eth_call` latency degrades.
