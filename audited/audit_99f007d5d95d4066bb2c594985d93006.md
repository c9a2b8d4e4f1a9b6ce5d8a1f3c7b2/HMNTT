### Title
Progressive SQL IN-Clause Amplification via Batch Contract Slot Cache Accumulation

### Summary
`ContractStateServiceImpl.findStorageBatch()` accumulates every distinct slot key seen for a given `contractId` into a per-contract Caffeine cache (max 1500 entries, 5-minute TTL). On every cache miss in `contractStateCache` (which expires after only 2 seconds), it issues a `findStorageBatch` SQL query whose `IN (:slots)` clause contains **all** currently accumulated keys — not just the requested one. An unprivileged attacker can pre-fill the per-contract slot cache with up to 1500 unique keys, then sustain a pattern where a single HTTP request triggers a 1500-parameter SQL `IN`-clause query every 2 seconds, achieving a ~1500× amplification in database query complexity with no brute-force requirement.

**Root cause:**

The design intent is to preload previously-seen slot keys in a single batch query. However, the per-contract slot cache (`slotsPerContract`, `expireAfterAccess=5m, maximumSize=1500`) accumulates keys indefinitely within its TTL, while the `contractStateCache` that holds the actual values expires after only 2 seconds (`expireAfterWrite=2s`). The mismatch means:

1. Slot keys live in the per-contract cache for up to 5 minutes.
2. Their values expire from `contractStateCache` every 2 seconds.
3. Any cache miss in `contractStateCache` (line 63–69) bypasses the value cache and calls `findStorageBatch()`, which always queries **all** accumulated keys for that contract.

**Exploit flow:**

1. Attacker identifies a deployed contract with a large mapping (e.g., an ERC-20 token's `balanceOf` mapping).
2. Attacker sends up to 1500 `eth_call` requests (within the 500 rps global limit, ~3 seconds), each reading a different storage slot of the same contract (e.g., `balanceOf(addr_1)`, `balanceOf(addr_2)`, ..., `balanceOf(addr_1500)`).
3. Each new unique slot key is added to the per-contract `contractSlotsCache`. Each call triggers a `findStorageBatch` with an IN-clause of size 1, 2, 3, ..., 1500 respectively.
4. After 2 seconds, all values expire from `contractStateCache`. Now a **single** `eth_call` for any of the 1500 slots triggers `findStorageBatch` with all 1500 keys in the IN-clause.
5. Attacker repeats step 4 every 2 seconds (one request per cycle) to sustain the amplification for the full 5-minute lifetime of the per-contract slot cache.

**Why existing checks fail:**

- The global rate limiter (500 rps) throttles HTTP requests but does not account for the per-request SQL cost amplification. One request at 1500-key IN-clause is ~1500× more expensive than baseline.
- The gas throttle (`gasPerSecond`) governs EVM execution cost, not database query cost.
- The `maximumSize=1500` cap on the per-contract cache bounds the IN-clause size but does not prevent the amplification — 1500 parameters in a PostgreSQL `IN` clause is a substantial query.
- There is no per-IP or per-contract-ID rate limiting.
- The `contractSlots` outer cache (`maximumSize=3000`) allows up to 3000 contracts to be targeted simultaneously, multiplying the effect. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

A single attacker with no credentials can cause the PostgreSQL database to execute repeated queries with up to 1500 bind parameters in an `IN` clause against the `contract_state` table. Even with an index on `(contract_id, slot)`, a 1500-entry IN-clause requires 1500 index probes per query. Sustained at one query per 2 seconds (the `contractStateCache` TTL), and multiplied across 3000 contracts (the outer cache limit), this can drive database CPU well above 30% above baseline. The `db.statementTimeout=3000ms` does not prevent the amplification since each query completes within the timeout. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The attack requires no authentication, no special privileges, and no on-chain transactions. Any public `eth_call` endpoint user can execute it. The only prerequisite is knowledge of a contract address with multiple readable storage slots (trivially satisfied by any ERC-20 token). The attack is repeatable indefinitely and self-sustaining after the initial 3-second fill phase. The global 500 rps limit is sufficient to fill the 1500-slot cache in under 3 seconds. [6](#0-5) [7](#0-6) 

### Recommendation

1. **Decouple batch size from accumulated cache size**: Instead of querying all accumulated keys on every miss, limit the IN-clause to a fixed maximum (e.g., 50–100 keys) per invocation, selecting the most recently accessed keys.
2. **Align TTLs**: The `slotsPerContract` cache TTL (5 minutes) should not greatly exceed the `contractState` value cache TTL (2 seconds). Either extend the value TTL or shorten the slot-key TTL so that stale slot keys are evicted before they can be re-queried.
3. **Add per-contract-ID rate limiting**: Throttle the number of distinct slot keys that can be registered per contract per time window.
4. **Cap IN-clause size at query time**: In `findStorageBatch()`, truncate `cachedSlots` to a hard maximum before passing it to the repository, regardless of how many keys are in the cache. [8](#0-7)

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
