### Title
Non-JSON Mirror Node Response Bypasses All Retries and Causes Fatal Pinger Exit

### Summary
When `network=other`, `newClient()` calls `buildNetworkFromMirrorNodes()`, which fetches the node list over plain HTTP. If the HTTP response body is not valid JSON (e.g., an HTML error page injected via MITM), `json.NewDecoder(resp.Body).Decode()` fails and returns `retry=false`, which immediately breaks the retry loop regardless of the configured retry count. The error propagates to `main()`, which calls `log.Fatalf()`, terminating the pinger process permanently.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 96–97:
```go
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```
The second return value (`false`) is the `retry` flag. It is hardcoded to `false` for any JSON decode error.

Back in `buildNetworkFromMirrorNodes()`, lines 59–61:
```go
if !retry || attempt == attempts {
    break
}
```
Because `retry=false`, the loop breaks on the very first attempt, skipping all configured retries (default: 10, via `mirrorNodeClientMaxRetries`).

The error then propagates through `newClient()` (`sdk_client.go` lines 18–20) back to `main.go` lines 41–44:
```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)
}
```
`log.Fatalf` calls `os.Exit(1)` — the pinger terminates permanently.

**Root cause:** Two compounding flaws:
1. No `Content-Type` or body-format validation before attempting JSON decode.
2. JSON decode failure is incorrectly classified as non-retryable (`retry=false`), while transient network errors correctly return `retry=true` (line 86). A decode failure on a bad response is at least as transient as a network error, yet gets zero retries.

**Failed assumption:** The code assumes that a 2xx HTTP response always carries a valid JSON body. A MITM attacker (or a misbehaving proxy/load balancer) can return `200 OK` with an HTML body, satisfying the status-code check at lines 90–93 while still causing the decode to fail.

### Impact Explanation

The pinger is a liveness/readiness-critical process. A fatal exit at startup means:
- `/tmp/ready` is never written (line 47–49 of `main.go` is never reached), so the Kubernetes readiness probe fails permanently.
- The liveness heartbeat goroutine never starts, so `/tmp/alive` is never updated.
- The container enters a crash-loop, disrupting the monitoring pipeline.

Severity: **Medium** — no economic damage, but the pinger's entire purpose (monitoring network health) is defeated for the duration of the attack.

### Likelihood Explanation

**Preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` must be set (operator-configured, but this is the only mode that uses the custom mirror REST URL).
2. The `mirrorRest` URL uses plain HTTP (the documented example is `http://mirror-rest:5551`; no TLS).

**Attack vectors for an unprivileged external attacker:**
- **DNS hijacking / cache poisoning** (no DNSSEC required on the cluster): redirect the mirror node hostname to an attacker-controlled server that returns `200 OK` with an HTML body.
- **ARP spoofing / on-path MITM** on the same Kubernetes node network or pod CIDR: intercept the HTTP GET and inject a non-JSON `200` response.
- **Compromised intermediate proxy or ingress**: if the mirror REST URL routes through a shared proxy, injecting a bad response is trivial.

The attack is **repeatable**: each pinger restart (crash-loop) re-triggers the startup sequence, so the attacker only needs to sustain the MITM for the brief startup window. Because `retry=false`, even a single injected response is sufficient — the configured 10-retry budget is entirely wasted.

### Recommendation

1. **Make JSON decode errors retryable.** Change line 97 to return `retry=true`:
   ```go
   return nil, true, fmt.Errorf("decode mirror nodes: %w", err)
   ```
2. **Validate `Content-Type` before decoding:**
   ```go
   ct := resp.Header.Get("Content-Type")
   if !strings.Contains(ct, "application/json") {
       return nil, true, fmt.Errorf("unexpected Content-Type %q from %s", ct, url)
   }
   ```
3. **Use TLS for the mirror REST URL** in production deployments to prevent MITM injection entirely.
4. **Do not `log.Fatalf` on a transient startup error.** Consider retrying `newClient()` in a loop with backoff before giving up, rather than exiting immediately.

### Proof of Concept

1. Set `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`.
2. On the attacker-controlled host, serve:
   ```
   HTTP/1.1 200 OK
   Content-Type: text/html
   
   <html>Not JSON</html>
   ```
3. Start the pinger. Observe:
   ```
   2024/xx/xx client error: attempt 1/11: decode mirror nodes: invalid character '<' looking for beginning of value
   exit status 1
   ```
   The process exits immediately. All 10 configured retries are skipped because `retry=false`. `/tmp/ready` is never created; the Kubernetes readiness probe fails permanently.
