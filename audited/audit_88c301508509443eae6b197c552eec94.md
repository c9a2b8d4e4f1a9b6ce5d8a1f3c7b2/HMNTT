### Title
Unauthenticated Multi-Query DB Amplification DoS via `transaction_hash` + Timestamp Filters on `/contracts/:contractId/results/logs`

### Summary
An unprivileged external user can send `GET /contracts/:contractId/results/logs?transaction_hash=<valid_hash>&timestamp=gte:A&timestamp=lte:B` to trigger a sequential chain of up to 4 database queries per HTTP request with no rate limiting enforced on the REST API. Because the REST server applies zero request throttling to this endpoint, an attacker can flood the service with concurrent requests, exhausting the DB connection pool and degrading availability for all users.

### Finding Description
**Code path:**

`rest/routes/contractRoute.js` line 16 routes the request to `getContractLogsById`: [1](#0-0) 

`getContractLogsById` (lines 779–812) unconditionally calls `extractContractLogsMultiUnionQuery(filters, contractId)` at line 802 with no guard or rate check: [2](#0-1) 

Inside `extractContractLogsMultiUnionQuery`, when `transactionHash !== undefined` (line 669), `getTransactionHash()` is awaited synchronously before the log query proceeds: [3](#0-2) 

`getTransactionHash` in `rest/transactionHash.js` executes `select * from get_transaction_info_by_hash($1)` — a PostgreSQL stored procedure that itself performs **two** sequential internal queries (recent-first, then fallback to older data): [4](#0-3) [5](#0-4) 

If the hash resolves (rows.length > 0), `ContractService.getContractLogs(query)` executes a second (potentially multi-union) query against `contract_log`: [6](#0-5) 

`getContractLogs` then unconditionally fires a third query — `getRecordFileBlockDetailsFromTimestampArray` — for every non-empty result set: [7](#0-6) 

**Total per request: up to 4 DB queries** (2 inside the stored procedure + 1 contract_log query + 1 record_file query).

**Root cause — no rate limiting on the REST server:**
A grep across all `rest/**/*.js` files finds zero occurrences of any rate-limiting middleware. `rest/server.js` applies only `authHandler` (which controls response *limits*, not request *rate*), `requestLogger`, and `responseHandler` — no throttle: [8](#0-7) 

The throttle infrastructure that exists (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` Java module and is entirely absent from the Node.js REST service: [9](#0-8) 

### Impact Explanation
Each concurrent HTTP request consumes up to 4 DB connections/query slots from the shared pool. An attacker sending N concurrent requests forces up to 4N simultaneous DB queries. Because the REST API is publicly accessible with no authentication or rate gate, the attacker can exhaust the PostgreSQL connection pool, causing query queuing, timeout cascades, and denial of service for all API consumers. The `contract_log` query can be a multi-union (lower/inner/upper sub-queries), further amplifying per-request DB cost. [10](#0-9) 

### Likelihood Explanation
Preconditions are minimal: the attacker needs only one valid transaction hash, which is freely available from any public Hedera block explorer or by observing prior API responses. The endpoint requires no credentials. The attack is trivially repeatable with standard HTTP load tools (e.g., `ab`, `wrk`, `hey`). The 4x DB query amplification means the attacker achieves disproportionate backend load relative to their own bandwidth cost.

### Recommendation
1. **Add rate limiting to the REST API**: Apply a per-IP (or global) rate-limiting middleware (e.g., `express-rate-limit`) in `rest/server.js` before route handlers, mirroring the bucket4j throttle already present in the web3 module.
2. **Short-circuit on hash miss early**: The current code already returns `null` when `rows.length === 0` (line 672–674), which is correct. Ensure this path is exercised for unknown hashes to avoid the log query.
3. **Cache hash→timestamp resolution**: The `getTransactionHash` result for a given hash is immutable once written; a short-lived cache (e.g., Redis, already present in the stack) would collapse repeated lookups for the same hash to a single DB round-trip.
4. **Enforce a connection pool query timeout** to bound the blast radius of any individual slow query.

### Proof of Concept
```bash
# Step 1: Obtain a valid transaction hash from any public Hedera explorer

CONTRACT="0.0.1004"
BASE="https://<mirror-node-host>/api/v1"

# Step 2: Send high-concurrency requests (e.g., 500 concurrent)
# Each request triggers: get_transaction_info_by_hash (x2 internal) +
#                        contract_log multi-union query +
#                        record_file lookup
hey -n 10000 -c 500 \
  "${BASE}/contracts/${CONTRACT}/results/logs?transaction.hash=${HASH}&timestamp=gte:0&timestamp=lte:9999999999.999999999"

# Expected result: DB connection pool exhaustion, 503/timeout responses
# for all other API consumers within seconds.
```

```java
final class ThrottleManagerImpl implements ThrottleManager {

    static final String REQUEST_PER_SECOND_LIMIT_EXCEEDED = "Requests per second rate limit exceeded";
    static final String GAS_PER_SECOND_LIMIT_EXCEEDED = "Gas per second rate limit exceeded.";

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
```
