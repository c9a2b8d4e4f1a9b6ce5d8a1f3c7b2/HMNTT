### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Rosetta API Requests

### Summary
The `ConnectToDb()` function in `rosetta/app/db/db.go` configures a pool of 100 open database connections with no application-level concurrency or rate limiting. An unauthenticated attacker can send 100+ concurrent block queries, each holding a pool connection for up to 20 seconds (the default `statementTimeout`), exhausting the pool and blocking all subsequent requests. The Traefik middleware that would mitigate this is disabled by default.

### Finding Description
**Exact code path:**

`rosetta/app/db/db.go`, `ConnectToDb()`, line 33:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections) // default: 100
``` [1](#0-0) 

`rosetta/app/db/client.go`, `GetDbWithContext()`, line 36:
```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second) // default: 20s
``` [2](#0-1) 

**Root cause:** The Rosetta HTTP server in `main.go` applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no concurrency cap or rate limiter. [3](#0-2) 

Go's `net/http.Server` accepts unlimited concurrent connections. Each incoming block/account query calls `GetDbWithContext()`, which acquires one connection from the pool and holds it for up to `statementTimeout` (20 s) or until the HTTP `writeTimeout` fires (10 s, 10,000,000,000 ns). [4](#0-3) 

With 100 concurrent requests in flight, `SetMaxOpenConns(100)` is saturated. All subsequent `db.Raw(...)` calls block indefinitely waiting for a free slot, making the service unresponsive.

**Why the existing Traefik middleware is insufficient:**

The `inFlightReq` (5 per IP) and `rateLimit` (10 req/s per host) middleware are defined in `values.yaml` but are gated on `global.middleware: false` (the default). The middleware template renders nothing unless both `global.middleware` and `middleware` are `true`. [5](#0-4) [6](#0-5) 

In a default deployment the Rosetta process has zero protection against connection pool exhaustion.

### Impact Explanation
When all 100 pool connections are held, every new API request (block queries, account queries, network status) blocks waiting for a connection. The Rosetta node becomes completely unresponsive to legitimate callers. Because the pool is per-process, a single attacker targeting one instance fully disables it. In a small deployment (1–3 instances), this constitutes ≥30% service unavailability. Additionally, because the PostgreSQL `max_connections` default is also 100, exhausting the Rosetta pool can starve other mirror-node services (importer, REST API) sharing the same database. [7](#0-6) 

### Likelihood Explanation
No authentication is required. The Rosetta API is publicly exposed (ingress enabled by default). [8](#0-7) 

With `writeTimeout = 10 s`, an attacker needs only ~10 new requests per second to keep the pool permanently saturated (100 connections ÷ 10 s = 10 req/s). This is trivially achievable with `wrk`, `ab`, or any HTTP load tool from a single machine. No special knowledge of the protocol is needed — a valid `POST /block` body is publicly documented by the Rosetta specification.

### Recommendation
1. **Add an application-level concurrency semaphore** in the Rosetta Go server (e.g., a buffered channel or `golang.org/x/sync/semaphore`) sized to `MaxOpenConnections` or a fraction thereof, rejecting requests with HTTP 503 when the limit is reached.
2. **Enable the Traefik middleware by default** by changing `global.middleware` to `true` in the Helm chart defaults, or document it as a required production setting.
3. **Reduce `MaxOpenConnections`** to a value that leaves headroom for other services sharing the PostgreSQL instance (e.g., 20–30).
4. **Ensure `writeTimeout` (10 s) is strictly less than `statementTimeout` (20 s)** so that HTTP timeouts reliably cancel the DB context and release connections promptly — currently this relationship holds, but it should be enforced by configuration validation.

### Proof of Concept
```bash
# Saturate the 100-connection pool with concurrent block queries
# Requires: wrk (https://github.com/wg/wrk), valid Rosetta endpoint

cat > block_request.lua << 'EOF'
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}'
EOF

# 100 concurrent connections, sustained for 60 seconds
wrk -t10 -c100 -d60s -s block_request.lua http://<rosetta-host>:5700/block

# In a second terminal, observe that legitimate requests now hang:
curl -s -m 5 -X POST http://<rosetta-host>:5700/network/list \
  -H 'Content-Type: application/json' \
  -d '{"metadata":{}}' 
# Expected result: timeout / no response while pool is exhausted
```
