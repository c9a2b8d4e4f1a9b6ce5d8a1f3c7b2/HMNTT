### Title
Unauthenticated Connection Pool Exhaustion DoS via `getTransactionDetailsFromTransactionId()`

### Summary
The `getTransactionDetailsFromTransactionId()` function in `rest/service/transactionService.js` issues a database query on every invocation through `pool().queryQuietly()` with no application-level rate limiting or concurrency guard. The `authHandler` middleware explicitly allows unauthenticated requests to proceed, meaning any unprivileged external user can flood the endpoint with unique `transactionId` values, exhausting the finite database connection pool and denying service to legitimate users.

### Finding Description

**Exact code path:**

`getTransactionDetailsFromTransactionId()` (lines 64–72) calls `getTransactionDetails()` (line 87–90), which calls `super.getRows()` (line 55–57 of `baseService.js`), which calls `this.pool().queryQuietly(query, params)` — acquiring a connection from the shared global `pool` on every invocation. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — failed assumption:**

The application assumes that infrastructure-level controls (reverse proxy, WAF) will throttle abusive clients. At the application layer, there is no rate limiter, no concurrency semaphore, and no queue-depth check before issuing a DB query.

**Why existing checks fail:**

The `authHandler` middleware explicitly passes unauthenticated requests through without any restriction:

```js
if (!credentials) {
  return;   // ← unauthenticated request proceeds unconditionally
}
```

The custom `user.limit` is only set for *authenticated* users. Unauthenticated callers carry no limit at all. [4](#0-3) [5](#0-4) 

No `rateLimit*` file exists anywhere in the `rest/` tree, and the exported middleware index contains no throttling primitive. [6](#0-5) 

The `responseCacheHandler` cannot help here because the attacker uses *distinct* `transactionId` values, producing a cache miss on every request.

**Exploit flow:**

1. Attacker sends N concurrent HTTP requests (N > pool max size) to the transactions endpoint, each with a unique, syntactically valid `transactionId` (e.g., `0.0.X-<timestamp>-0` with varying account IDs or timestamps).
2. Each request reaches `getTransactionDetailsFromTransactionId()` → `getRows()` → `pool().queryQuietly()`.
3. The pool issues up to its maximum connections; additional requests queue or are rejected by the pool driver.
4. While the pool is saturated, all other endpoints that share the same global `pool` (accounts, tokens, balances, schedules, etc.) also fail to acquire connections.
5. Legitimate users receive errors or timeouts across the entire REST API.

### Impact Explanation

Complete denial of service against the Mirror Node REST API. Because `pool()` returns a single global pool shared by all service classes, exhausting it blocks every endpoint simultaneously — not just the transactions route. This directly prevents access to Hashgraph history, account balances, and all other mirror node data, satisfying the "Tampering/Manipulating Hashgraph history availability" threat model.

### Likelihood Explanation

Preconditions: none. No account, API key, or credential is required. The attack is trivially scriptable with any HTTP load tool (`wrk`, `ab`, `hey`, `curl` in a loop). Because each request uses a *different* `transactionId`, response caching provides zero protection. The attack is repeatable and stateless — the attacker does not need to maintain any session. Realistic likelihood: **high**.

### Recommendation

1. **Application-level rate limiting:** Add a per-IP (and optionally per-user) rate-limiting middleware (e.g., `express-rate-limit`) applied globally before any route handler, with a burst limit low enough to prevent pool saturation.
2. **Concurrency cap:** Introduce a semaphore or async queue (e.g., `p-limit`) in `BaseService.getRows()` or at the pool wrapper level to cap the number of in-flight DB queries.
3. **Pool queue timeout:** Configure the DB pool with a short `connectionTimeoutMillis` and a bounded `max` so that excess requests fail fast with a 503 rather than queuing indefinitely.
4. **Unauthenticated request throttling:** The `authHandler` should apply a strict default rate limit to unauthenticated callers rather than passing them through unconditionally.

### Proof of Concept

```bash
# Generate 2000 concurrent requests with unique transactionIds (no credentials needed)
seq 1 2000 | xargs -P 200 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/transactions/0.0.{}-1700000000-0" \
  -o /dev/null -w "%{http_code}\n"

# Expected: legitimate requests begin returning 500/503/timeout
# as the DB connection pool is exhausted
```

Each request maps to a unique `payerAccountId` (`0.0.{}`), guaranteeing cache misses and a fresh DB query per request via `pool().queryQuietly()`. [1](#0-0) [2](#0-1)
