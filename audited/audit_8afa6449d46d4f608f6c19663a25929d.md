### Title
Unbounded Historical Topic Subscription Causes Repeated DB Polling and Connection Pool Exhaustion

### Summary
An unprivileged external user can subscribe to any existing topic with `startTime=0` (epoch), no `endTime`, and no `limit`. When the topic has a large volume of historical messages, `PollingTopicMessageRetriever` enters an infinite polling loop issuing DB queries every 2 seconds with pages of 1000 rows, never completing. Multiple concurrent such subscriptions from multiple gRPC connections saturate the database connection pool, denying service to legitimate queries.

### Finding Description

**Entry point — `ConsensusController.subscribeTopic()`:**
The gRPC handler accepts any `ConsensusTopicQuery` and converts it to a `TopicMessageFilter` with no server-side cap on `startTime` age, no `endTime` requirement, and no `limit` requirement. [1](#0-0) 

**Validation — `TopicMessageFilter.isValidStartTime()`:**
The only constraint on `startTime` is that it must not be in the future (`startTime <= DomainUtils.now()`). A value of `0` (Unix epoch, year 1970) passes validation. `limit` defaults to `0` (no limit), and `endTime` is optional. [2](#0-1) 

**Historical retrieval — `TopicMessageServiceImpl.subscribeTopic()`:**
`topicMessageRetriever.retrieve(filter, true)` is called with `throttled=true` for every subscriber. There is no global subscriber cap, no per-IP limit, and no check on how far back `startTime` reaches. [3](#0-2) 

**Infinite repeat — `PollingTopicMessageRetriever.retrieve()`:**
When `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE` and `frequency` is `pollingFrequency` (default 2 s). The flux repeats indefinitely. [4](#0-3) 

**Completion condition — `PollingContext.isComplete()`:**
For the throttled path, the loop only terminates when the last page returned **fewer than `maxPageSize` (1000)** rows, or the limit is hit. For a topic with millions of historical messages, every page is full, so `isComplete()` always returns `false` and polling never stops. [5](#0-4) 

**Each poll issues a DB query:** [6](#0-5) 

**Timeout is per-element, not per-subscription:**
The 60-second `timeout` fires only if no message is emitted within 60 s. While messages are flowing from a large historical dataset, the timeout never triggers, so the subscription lives indefinitely. [7](#0-6) 

**Per-connection concurrency limit is insufficient:**
`maxConcurrentCallsPerConnection = 5` limits calls per single gRPC connection, but an attacker can open arbitrarily many connections, each carrying 5 subscriptions. [8](#0-7) 

### Impact Explanation
Each malicious subscription issues one DB query every 2 seconds indefinitely. With a typical DB connection pool of 10–20 connections and a `statementTimeout` of 10 s, as few as 20–50 concurrent attacker subscriptions can saturate the pool. Legitimate integrity-verification queries (e.g., running-hash checks, importer queries) queue behind attacker queries and time out, degrading or halting mirror-node operation. Because the gRPC API is public and unauthenticated, this constitutes a complete denial-of-service against the database tier.

### Likelihood Explanation
No authentication is required. The attacker only needs to know a valid topic ID (all topic IDs are public on-chain). Opening many gRPC connections is trivial with standard gRPC client libraries. The attack is repeatable and can be sustained indefinitely at low cost. A single attacker with a modest number of connections can trigger the condition.

### Recommendation
1. **Enforce a maximum historical look-back window**: Reject or clamp `startTime` values older than a configurable threshold (e.g., 7 days) unless an explicit `endTime` and `limit` are also provided.
2. **Add a global (or per-IP) concurrent-subscription limit**: Track active subscriptions and reject new ones beyond a threshold.
3. **Apply a wall-clock subscription lifetime timeout** in addition to the per-element idle timeout, so no single subscription can hold a DB polling loop open indefinitely.
4. **Require `limit` or `endTime` for deep-historical queries**: If `startTime` is older than N seconds, mandate a finite `endTime` or non-zero `limit` to bound the number of pages fetched.

### Proof of Concept
```
# Prerequisites: gRPC client (e.g., grpcurl or Hedera SDK), valid topic ID on the mirror node

# Step 1: Open 10 connections, each with 5 subscriptions (50 total)
for i in $(seq 1 50); do
  grpcurl -plaintext -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0, "nanos": 0}
    # no consensusEndTime, no limit
  }' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2: Observe DB connection pool saturation
# Each subscription polls the DB every 2s with SELECT ... LIMIT 1000
# With 50 subscriptions: ~25 queries/second
# Legitimate queries begin timing out (statementTimeout=10s)
# Mirror node integrity queries fail
```
