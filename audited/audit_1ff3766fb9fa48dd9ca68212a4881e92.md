### Title
Unbounded TCP Connections Allow Unlimited `subscribeTopic()` Subscriptions, Enabling Resource-Exhaustion DoS

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent subscriptions from any unauthenticated client because the only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits streams per single TCP connection, not the total number of TCP connections a client may open. An attacker can open an arbitrary number of TCP connections, each carrying 5 indefinite subscriptions, exhausting JVM heap, Reactor scheduler threads, and database connection pool capacity, causing the gRPC service to become unavailable to legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts every inbound call with no authentication, no per-IP check, and no global subscriber cap:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    ...
}
``` [1](#0-0) 

The only server-side connection control is set in `GrpcConfiguration`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [2](#0-1) 

with a default of **5**: [3](#0-2) 

`maxConcurrentCallsPerConnection` is a Netty/gRPC HTTP/2 setting that caps the number of concurrent RPC streams **on a single TCP connection**. It does not limit how many TCP connections a single client (or IP) may open. No `maxInboundConnections` or equivalent is configured anywhere in the grpc module. [4](#0-3) 

The `subscriberCount` field in `TopicMessageServiceImpl` is a **Micrometer gauge for observability only** — it is never checked against any ceiling to reject new subscriptions:

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [5](#0-4) [6](#0-5) 

Each indefinite subscription (`limit=0`, no `consensusEndTime`) creates a live `TopicContext`, a historical retrieval `Flux`, a live listener `Flux`, and a safety-check `Flux` on `Schedulers.boundedElastic()`, all held in memory until the client disconnects: [7](#0-6) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection = 5` constitutes a meaningful global rate/connection limit. It does not — it is a per-connection HTTP/2 stream multiplexing cap. No mechanism bounds the total number of TCP connections, total active subscriptions, or subscriptions per source IP.

### Impact Explanation

With N TCP connections × 5 streams each, an attacker sustains N×5 indefinite subscriptions. At scale (e.g., 10,000 connections = 50,000 subscriptions):

- **JVM heap exhaustion**: Each `TopicContext` + reactive chain holds kilobytes of state; tens of thousands saturate heap.
- **Reactor `boundedElastic` thread pool saturation**: The safety-check `Flux` and polling listener each schedule work on `boundedElastic` (capped at 10× CPU cores by default); the queue backs up and new legitimate subscriptions stall.
- **Database connection pool exhaustion**: Each subscription triggers periodic DB queries via `PollingTopicListener` or gap-recovery queries; the HikariCP pool is drained, blocking all DB access including for legitimate clients.

Result: legitimate subscribers receive no messages or are rejected with connection errors, causing any application logic that depends on HCS message delivery (e.g., smart contract triggers, payment release workflows) to fail silently.

### Likelihood Explanation

- **No privileges required**: The gRPC port (5600) is publicly exposed with no authentication.
- **Trivially scriptable**: A single attacker machine can open thousands of TCP connections using standard gRPC client libraries or `grpcurl` in a loop.
- **Persistent**: Each subscription is indefinite by default (`limit=0`, no end time); the attacker need not continuously send new requests — connections stay open.
- **Low cost**: HTTP/2 connections are lightweight; a single commodity machine can sustain tens of thousands.

### Recommendation

1. **Add a global maximum inbound connections limit** on the Netty server builder:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(10), TimeUnit.MILLISECONDS);
   serverBuilder.maxInboundConnections(/* e.g. 1000 */);
   ```
   and expose it as a configurable `NettyProperties` field alongside `maxConcurrentCallsPerConnection`.

2. **Enforce a global subscriber ceiling** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` status when exceeded.

3. **Add per-source-IP connection rate limiting** at the Netty pipeline or via an ingress/load-balancer policy (the Helm chart's `maxRatePerEndpoint: 250` applies only to the GCP backend policy, not to raw TCP connections at the gRPC layer).

4. **Require authentication** (e.g., mTLS or a bearer token) for long-lived streaming RPCs.

### Proof of Concept

```bash
# Open 200 TCP connections, each with 5 indefinite subscriptions (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1000}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: hiero.mirror.grpc.subscribers gauge climbs to 1000
# Observe: JVM heap and DB connection pool approach exhaustion
# Observe: new legitimate subscribeTopic calls stall or fail with UNAVAILABLE
```

Each background process holds an open HTTP/2 stream on its own TCP connection. Because `maxConcurrentCallsPerConnection = 5` is per-connection, all 1000 streams are accepted. No server-side mechanism rejects them.

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
```
