### Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Bypasses Pagination, Enabling Unbounded DB Query via `limit=Long.MAX_VALUE`

### Summary
An unprivileged gRPC client can set `startTime=0` and `limit=Long.MAX_VALUE` in a `subscribeTopic` request. Both values pass Bean Validation (no `@Max` on `limit`; `startTime=0` satisfies `@Min(0)` and `isValidStartTime()`). Inside `PollingTopicMessageRetriever.poll()`, the cast `(int)(Long.MAX_VALUE - 0)` overflows to `-1`, which makes `hasLimit()` return `false` on the page-level filter, stripping the SQL `LIMIT` clause entirely and causing the database to return all historical messages for the topic in a single unbounded query. Multiple concurrent attackers can exhaust the database connection pool.

### Finding Description

**Validation layer — no upper bound on `limit`:**

`TopicMessageFilter.java` lines 25–26:
```java
@Min(0)
private long limit;
```
`Long.MAX_VALUE` satisfies `@Min(0)`. There is no `@Max`. `hasLimit()` (line 39) returns `true` for `Long.MAX_VALUE > 0`. `startTime=0` satisfies `@Min(0)` (line 28) and `isValidStartTime()` (line 49) because `0 <= DomainUtils.now()`.

**Integer overflow in `PollingTopicMessageRetriever.poll()` (lines 68–71):**
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())   // (int) Long.MAX_VALUE == -1
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());        // Math.min(-1, N) == -1
```
`(int) Long.MAX_VALUE` = `-1` in Java (lower 32 bits of `0x7FFFFFFFFFFFFFFF` = `0xFFFFFFFF` = `-1`). `Math.min(-1, maxPageSize)` = `-1`.

**Page-level filter loses its limit (line 75):**
```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
// newFilter.limit == -1L  →  newFilter.hasLimit() == (-1 > 0) == false
```
`findByFilter(newFilter)` is called with a filter where `hasLimit()` is `false`. The repository omits the SQL `LIMIT` clause, returning every historical row for the topic.

**`Flux.fromStream()` holds a DB cursor open (line 78):**
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
A JDBC/R2DBC stream holds the database cursor open until the stream is fully consumed or cancelled. With millions of rows and no limit, the cursor (and its connection) is held for the entire duration.

**No subscriber cap:** `subscriberCount` (line 48) is a gauge only — it is never checked against a maximum. `pastEndTime()` (line 123) returns `Flux.never()` when `endTime` is null, keeping each subscription alive indefinitely after the historical phase.
