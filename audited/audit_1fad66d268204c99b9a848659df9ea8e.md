### Title
Unbounded IN-list Batch Query Amplification via Concurrent Cache Misses in `findStorageBatch()`

### Summary
The `findStorageBatch()` method in `ContractStateServiceImpl` reads all slot keys accumulated in the per-contract Caffeine cache (up to 1500 entries, each 32 bytes) and issues a single SQL `IN`-list query containing all of them on every cache miss. Because the `contractState` cache expires every 2 seconds and there is no synchronization or deduplication of concurrent batch queries, an unprivileged attacker can cause many threads to simultaneously issue identical ~48 KB SQL queries against the database.

### Finding Description

**Code path:**

In `ContractStateServiceImpl.findStorageBatch()` (lines 85–122), on every cache miss the method:

1. Reads the entire key set of the per-contract slot cache with no size cap applied before query construction:
   ```java
   final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet(); // up to 1500 entries
   final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
   for (var slot : cachedSlotKeys) {
       cachedSlots.add(((ByteBuffer) slot).array());
   }
   ``` [1](#0-0) 

2. Passes the full list directly to the repository:
   ```java
   final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
   ``` [2](#0-1) 

3. The repository query expands this into a native SQL `IN`-list with no server-side limit:
   ```sql
   select slot, value from contract_state
   where contract_id = :contractId
   and slot in (:slots)
   ``` [3](#0-2) 

**Cache size limits that bound the attack:**

- `slotsPerContract` cache: `maximumSize=1500` — this is the per-contract slot key accumulator. Up to 1500 × 32-byte keys = 48,000 bytes of bind parameters per query. [4](#0-3) 

- `contractState` cache: `expireAfterWrite=2s` — all cached slot values expire every 2 seconds, forcing every subsequent access to re-enter `findStorageBatch()`. [5](#0-4) 

**Root cause:** There is no synchronization, no request coalescing, and no cap on the IN-list size before the DB call. Multiple concurrent threads that all miss the `contractState` cache will each independently read the same 1500-key set and each fire an identical ~48 KB query to the database.

### Impact Explanation
Each concurrent cache miss for a fully-populated contract produces a `SELECT ... WHERE slot IN (?, ?, ..., ?)` query with up to 1500 bind parameters (each a 32-byte bytea), generating a ~48 KB query string. With N concurrent requests, N identical queries hit the database simultaneously. The `contractState` cache TTL of 2 seconds means this condition recurs automatically every 2 seconds as long as the attacker keeps sending requests. This can saturate DB connection pool threads, inflate DB CPU/memory usage, and degrade service for all users — a classic griefing/DoS without requiring any economic stake or privileged access.

### Likelihood Explanation
The web3 RPC endpoint (`eth_call`, `eth_getStorageAt`, etc.) is publicly accessible with no authentication. An attacker needs only to:
1. Issue ~1500 storage-slot read calls for distinct slots of one contract (trivially scriptable).
2. Wait 2 seconds for the `contractState` cache to expire.
3. Flood concurrent slot-read requests for that contract.

No special knowledge, tokens, or on-chain transactions are required. The attack is fully repeatable every 2 seconds and can be parallelized across multiple contracts (up to 3000 active contract slot caches per the `contractSlots` `maximumSize=3000` setting). [6](#0-5) 

### Recommendation
1. **Cap the IN-list size before querying:** Truncate or chunk `cachedSlots` to a configurable maximum (e.g., 100–200 entries) before passing to `findStorageBatch()`.
2. **Coalesce concurrent batch queries:** Use a per-contract lock or a `AsyncLoadingCache` (Caffeine's built-in request-coalescing loader) so that only one batch query fires per contract per cache-miss window, with all concurrent waiters sharing the single result.
3. **Increase `contractState` TTL or use write-through invalidation** to reduce the frequency of full-batch re-queries.
4. **Rate-limit** the public RPC endpoint per IP/client to limit the attacker's ability to pre-populate the slot cache.

### Proof of Concept
```
# Step 1: Populate 1500 distinct slot keys for contract 0xTARGET
for i in $(seq 1 1500); do
  curl -s -X POST http://mirror-node-web3/api/v1/contracts/call \
    -d '{"data":"0x<SLOAD slot_i>","to":"0xTARGET","estimate":false}' &
done
wait

# Step 2: Wait for contractState cache to expire (2 seconds)
sleep 2

# Step 3: Send 50 concurrent requests for any slot of 0xTARGET
for i in $(seq 1 50); do
  curl -s -X POST http://mirror-node-web3/api/v1/contracts/call \
    -d '{"data":"0x<SLOAD slot_1>","to":"0xTARGET","estimate":false}' &
done
wait

# Result: 50 identical ~48KB SQL queries hit the DB simultaneously.
# Repeat every 2 seconds to sustain the load.
```
