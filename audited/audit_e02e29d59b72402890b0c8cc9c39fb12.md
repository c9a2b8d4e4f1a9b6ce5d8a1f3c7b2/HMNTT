### Title
Unbounded gRPC Subscription Flood via Missing IP-Based Rate Limiting Enables Resource Exhaustion DoS

### Summary
The `subscribeTopic()` endpoint in `ConsensusController.java` accepts unlimited concurrent streaming subscriptions from any unauthenticated caller with no IP-based rate limiting or global connection cap. The only server-side guard — `maxConcurrentCallsPerConnection` (default: 5) — is a per-TCP-connection limit trivially bypassed by opening additional connections. Each subscription increments a shared `subscriberCount` gauge exposed via Prometheus metrics, allowing an attacker to confirm attack effectiveness in real time.

**Root cause:** The server enforces no total connection limit, no per-IP connection limit, and no authentication. An attacker opens N TCP connections and multiplexes 5 streams per connection, yielding N×5 active subscriptions. There is no interceptor in the production gRPC path that performs rate limiting or identity checks. [6](#0-5) 

**Why existing checks fail:** `maxConcurrentCallsPerConnection=5` is a Netty-level per-connection guard. It does not limit the number of connections from a single IP or the total number of concurrent calls across all connections. An attacker with even a single machine can open thousands of TCP connections to port 5600 and saturate the server. [7](#0-6) 

### Impact Explanation

Each active subscription consumes:
- A HikariCP database connection (pool exhaustion triggers `GrpcHighDBConnections` alert at 75% utilization)
- A Redis listener subscription or polling thread
- JVM heap memory for `TopicContext`, reactive pipeline state, and message buffers (up to `maxBufferSize=16384` messages per subscriber)
- A thread-pool slot from `applicationTaskExecutor`

Flooding with thousands of subscriptions exhausts the DB connection pool, thread pool, and heap, causing legitimate subscribers to receive errors or timeouts — a complete denial of service for the gRPC API. The `hiero_mirror_grpc_subscribers` Prometheus gauge (scraped at `/actuator/prometheus` or equivalent) lets the attacker observe the live subscriber count and confirm the flood is effective. [8](#0-7) 

### Likelihood Explanation

No special privileges, credentials, or network position are required. Any internet-reachable deployment of the gRPC port (default 5600) is vulnerable. A single commodity machine can open thousands of TCP connections. A distributed botnet makes IP-based blocking at the load balancer level harder. The `maxRatePerEndpoint: 250` GCP backend policy in the Helm chart is a request-rate limit, not a connection or subscription count limit, and does not prevent long-lived streaming subscriptions from accumulating. [9](#0-8) 

### Recommendation

1. **Global subscription cap**: Add a server interceptor that rejects new `subscribeTopic` calls when `subscriberCount` exceeds a configurable threshold (e.g., 10,000).
2. **Per-IP connection limit**: Configure `maxConnectionsPerIp` at the load balancer or add a gRPC `ServerInterceptor` that tracks and limits active streams per remote address.
3. **Connection-level rate limiting**: Use a token-bucket interceptor (similar to the existing `ThrottleConfiguration` in the web3 module) applied to the gRPC server.
4. **Restrict metrics endpoint**: Ensure `/actuator/prometheus` is not publicly reachable; bind the management port to an internal network interface only.
5. **Authentication**: Require a bearer token or mTLS for `subscribeTopic` to prevent anonymous abuse.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor
# proto stubs from com.hedera.mirror.api.proto
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "grpc.mainnet.mirrornode.hedera.com:443"
CONNECTIONS = 500   # 500 TCP connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = 1  # any valid topic
    threads = []
    for _ in range(STREAMS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=CONNECTIONS) as ex:
    list(ex.map(flood, range(CONNECTIONS)))
# Result: 500 * 5 = 2500 concurrent subscriptions, DB pool and thread pool exhausted
# Verify: curl http://<grpc-pod>:8081/actuator/prometheus | grep hiero_mirror_grpc_subscribers
```
