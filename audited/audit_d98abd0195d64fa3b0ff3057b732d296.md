### Title
Synchronous `gzipSync()` in `CachedApiResponse` Constructor Blocks Node.js Event Loop on Cache Population

### Summary
The `CachedApiResponse` constructor calls Node.js's synchronous `gzipSync()` directly on the event loop thread with no upper bound on body size. Because `responseCacheUpdateHandler` runs in the same event loop after every cache miss, any concurrent request that arrives while `gzipSync()` is executing is stalled until compression completes. An unprivileged attacker can deliberately maximize cache misses — including by exploiting a Redis partition recovery window — to chain multiple blocking `gzipSync()` calls and degrade or deny service to all concurrent users.

### Finding Description

**Exact code path:**

`rest/middleware/responseCacheHandler.js`, line 115:
```js
const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
``` [1](#0-0) 

`shouldCompress` at line 137–139 applies only a **lower-bound** check (`body.length >= compressThreshold`, default 150 bytes) with no upper-bound guard: [2](#0-1) 

Inside `CachedApiResponse`, line 11 calls `gzipSync` synchronously:
```js
this.body = compress ? gzipSync(body).toString('base64url') : body;
``` [3](#0-2) 

**Root cause:** `gzipSync` is a blocking C++ call that holds the V8 event loop thread for its entire duration. There is no `setImmediate`, worker thread, or async zlib alternative used. The failed assumption is that response bodies are always small enough that synchronous compression is negligible.

**Exploit flow:**
1. Attacker sends `N` concurrent GET requests to endpoints that return large responses (e.g., `?limit=100` on transactions or balances endpoints, which can produce 50–500 KB JSON bodies).
2. Each request misses the cache (cold start, or after Redis partition recovery flushes all TTLs).
3. After each response is sent, `responseCacheUpdateHandler` fires in the event loop and calls `new CachedApiResponse(..., true)`.
4. `gzipSync` on a 500 KB body takes ~10–30 ms. With `N` concurrent requests, the event loop is blocked for `N × 10–30 ms` cumulatively, stalling all other in-flight requests.
5. During Redis partition recovery, all previously-cached keys expire simultaneously, guaranteeing a thundering herd of cache misses and a burst of back-to-back `gzipSync` calls.

**Why existing checks are insufficient:**
- `shouldCompress` only enforces a minimum threshold (150 bytes); there is no maximum body size cap before compression.
- `response.limit.max = 100` limits item count but not byte size; items with large fields (memos, metadata, bytecode) can produce multi-hundred-KB bodies.
- There is no rate limiting or concurrency cap on `responseCacheUpdateHandler` invocations. [4](#0-3) 

### Impact Explanation
Node.js is single-threaded. While `gzipSync` executes, no I/O callbacks, timers, or other request handlers can run. A burst of 20–50 concurrent cache-miss requests on large-response endpoints can block the event loop for 200 ms–1.5 s, causing all concurrent API consumers to experience equivalent latency spikes or timeouts. This constitutes a practical, repeatable denial-of-service against the REST API without any authentication.

### Likelihood Explanation
The attack requires no credentials. Any public API consumer can send `limit=100` requests to high-cardinality endpoints. The partition recovery amplifier is a bonus but not required — the attacker simply needs to ensure cache misses (e.g., by varying query parameters to defeat the MD5 cache key, or by waiting for TTL expiry). The default `compressThreshold` of 150 bytes means virtually every non-trivial response triggers `gzipSync`. [2](#0-1) 

### Recommendation
Replace `gzipSync` with the async `zlib.gzip` (callback or promisified) or use a `worker_threads` pool for compression, so the event loop is not blocked. Additionally, add an upper-bound size guard in `shouldCompress` (e.g., skip compression above a configurable `compressMaxSize`) to prevent unbounded blocking time. Example:

```js
// In shouldCompress:
const {compress, compressThreshold, compressMaxSize} = config.cache.response;
return compress && body.length >= compressThreshold && body.length <= compressMaxSize;

// In CachedApiResponse constructor (async alternative):
this.body = compress ? (await gzip(body)).toString('base64url') : body;
```

// Response middleware that caches the completed response.
const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
