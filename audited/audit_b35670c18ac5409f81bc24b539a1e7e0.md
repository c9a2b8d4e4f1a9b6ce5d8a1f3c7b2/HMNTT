### Title
Cache Key Collision for Alias-Only Accounts Causes Incorrect Account Substitution in `/block` Response

### Summary
In `updateOperationAccountAlias()`, the `entityCache` is keyed by `int64` using `accountId.GetId()`. For any alias-only account (created via `NewAccountIdFromAlias`), `GetId()` always returns `0` because no numeric entity ID is assigned. When a block contains operations referencing two or more distinct alias-only accounts, all of them share cache key `0`, so the first alias resolved is incorrectly substituted for every subsequent alias-only account in the same request.

### Finding Description
**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 115 and 121:
```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {   // line 115
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
    ...
    s.entityCache.Set(result.GetId(), result)                        // line 121
    cached = result
}
operations[index].AccountId = cached                                 // line 125
```

**Root cause:**

`AccountId.GetId()` returns `a.accountId.EncodedId` (`account_id.go` line 39). For alias-only accounts constructed via `NewAccountIdFromAlias`, the `accountId` field is `domain.EntityId{ShardNum: shard, RealmNum: realm}` with `EntityNum` unset (zero-value), so `EncodedId` is `0`. The test suite itself confirms this: `ed25519AliasAccountId.GetId()` == `0` (`account_id_test.go` lines 121–124).

**Exploit flow:**

1. Block B contains two transactions: Tx1 references alias-only account `alias_A` (public key PK1), Tx2 references alias-only account `alias_B` (public key PK2). Both have `GetId()` == 0.
2. Processing `alias_A`: cache miss at key `0` → `GetAccountAlias` queries DB with `id=0` → no entity found → returns `alias_A` → `entityCache.Set(0, alias_A)`.
3. Processing `alias_B`: cache **hit** at key `0` → returns `alias_A` → `operations[j].AccountId = alias_A`. `alias_B` is never looked up; it is silently replaced with `alias_A`.

**Why existing checks fail:**

`GetAccountAlias` in `account.go` line 117 queries by `accountId.GetId()` (= 0), finds no record, and returns the original `accountId`. The returned result also has `GetId()` == 0, so `entityCache.Set(0, alias_A)` stores the first alias at key 0. There is no guard in `updateOperationAccountAlias` that skips caching when `GetId()` == 0 or that uses a richer key (e.g., the alias bytes themselves).

### Impact Explanation
The Rosetta `/block` and `/block/transaction` endpoints return wrong `account_identifier` values for alias-only accounts. Any downstream system (exchange, indexer, wallet) consuming the Rosetta API will misattribute balance changes and transfers to the wrong account. This is a data-integrity violation: funds credited to `alias_B` appear attributed to `alias_A` in the Rosetta view, which can cause incorrect balance accounting and potential financial misreporting.

### Likelihood Explanation
The Rosetta API requires no authentication; any external user can call `/block`. Hedera supports auto-account-creation via alias (sending HBAR to a public key alias), so blocks containing multiple alias-only accounts are a normal network occurrence, not a specially crafted scenario. An attacker only needs to identify (or wait for) a block containing two or more alias-only account operations and issue a standard `/block` request. The bug is deterministic and 100% reproducible once such a block exists.

### Recommendation
Replace the `int64` cache key with a composite key that uniquely identifies both numeric-ID accounts and alias-only accounts. The simplest fix is to use `AccountId.String()` (which returns the hex-encoded alias for alias accounts, or `shard.realm.num` for numeric accounts) as the cache key:

```go
entityCache *cache.Cache[string, types.AccountId]
// lookup:
s.entityCache.Get(accountId.String())
// store:
s.entityCache.Set(result.String(), result)
```

Alternatively, skip caching entirely when `accountId.GetId() == 0` and `accountId.HasAlias()` is true, forcing a fresh DB lookup for every alias-only account.

### Proof of Concept
1. Identify or construct a block on the Hedera network containing at least two CryptoTransfer transactions where the recipient is an alias-only account (e.g., auto-created accounts with different ECDSA/Ed25519 public key aliases, both with `shard=0, realm=0`, no assigned entity number yet).
2. Send `POST /block` with the block identifier for that block to the Rosetta mirror node.
3. Observe the response: all alias-only account operations after the first will show the **same** `account_identifier` address (the first alias resolved), regardless of their actual distinct aliases.
