### Title
Unbounded `keys` Collection in `getHookStorage()` Enables SQL `IN`-Clause DoS via Unauthenticated Request

### Summary
`HookStorageRequest` places no upper-bound constraint on the `keys` collection. `getKeysInRange()` iterates every submitted key with `Arrays.compareUnsigned()`, and the surviving keys are forwarded verbatim to `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse()`, which generates a SQL `IN (‚Ä¶)` clause of arbitrary length. An unprivileged caller can repeatedly submit requests with thousands of 32-byte keys, degrading database performance with no economic cost to themselves.

### Finding Description

**Code path:**

`HookStorageRequest.java` ‚Äî `keys` field has no size constraint: [1](#0-0) 

`getKeysInRange()` iterates the entire collection unconditionally: [2](#0-1) 

`HookServiceImpl.getHookStorage()` passes the result directly to the repository with no size check: [3](#0-2) 

The repository method accepts an unbounded `List<byte[]>`: [4](#0-3) 

**Root cause:** The `limit` field (default 25) controls only the output `PageRequest` pagination ‚Äî it does not cap the number of input keys. There is no `@Size`, `@Max`, or any other validation annotation on the `keys` field, and no programmatic size guard anywhere in the call chain before the SQL query is issued.

**Failed assumption:** The design assumes callers supply a small, reasonable number of keys. Nothing enforces this assumption.

### Impact Explanation
Each malicious request causes: (a) O(n) CPU work in `getKeysInRange()` for `Arrays.compareUnsigned()` comparisons, and (b) a SQL `IN (val1, val2, ‚Ä¶, valN)` clause with N entries sent to the database. PostgreSQL must parse, plan, and execute this clause. With thousands of entries per request and concurrent attackers, this can saturate the database query planner and connection pool, causing latency spikes or outright unavailability for all legitimate users of the mirror node REST API. No data is modified, but service availability is impacted ‚Äî consistent with the "griefing" classification.

### Likelihood Explanation
The endpoint is reachable by any external caller with no privilege requirement. The attack requires only an HTTP client capable of sending a JSON/query-parameter payload with a large `keys` array. It is trivially scriptable, repeatable at high frequency, and costs the attacker nothing. The absence of any rate-limiting or input-size validation in the service layer makes sustained exploitation straightforward.

### Recommendation
1. Add a `@Size(max = N)` constraint (e.g., `max = 100`) on the `keys` field in `HookStorageRequest` and enable Bean Validation on the DTO.
2. Add a programmatic guard at the top of `getHookStorage()` and `getHookStorageChange()` that rejects requests where `keys.size()` exceeds the configured maximum.
3. Consider enforcing the same cap in `getKeysInRange()` itself as a defense-in-depth measure.
4. Apply API-level rate limiting on the hook storage endpoint.

### Proof of Concept

```
# Generate a request with 5000 random 32-byte hex keys
import requests, os, json

keys = [os.urandom(32).hex() for _ in range(5000)]

# Adjust URL, ownerId, hookId to any valid values in the mirror node
response = requests.get(
    "https://<mirror-node-host>/api/v1/accounts/<ownerId>/hooks/<hookId>/storage",
    params={"key": keys}   # or however the controller binds the keys param
)
print(response.status_code, response.elapsed)
```

Repeat in a tight loop from multiple clients. Observe increasing query latency in the database and degraded response times for all other API consumers. No authentication token is required.
