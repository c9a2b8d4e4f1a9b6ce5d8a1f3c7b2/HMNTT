### Title
Redundant Double DB Lookup in `listStakingRewardsByAccountId` When Base32 Alias Is Used as Path Parameter

### Summary
When an unprivileged user calls `GET /accounts/:idOrAliasOrEvmAddress/rewards` with a valid base32 alias that resolves to an existing account, the handler unconditionally issues two sequential database queries: one inside `EntityService.getEncodedId()` (alias → entity lookup) and a second inside `EntityService.isValidAccount()` (entity existence check). The second query is logically redundant because the alias lookup already confirms the account exists and is not deleted. No rate limiting exists on this REST endpoint, allowing any unauthenticated caller to double the per-request DB load at will.

### Finding Description

**Exact code path:**

`rest/controllers/accountController.js`, `listStakingRewardsByAccountId`, lines 171–172:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
const isValidAccount = await EntityService.isValidAccount(accountId);
```

**DB Query 1** — triggered inside `getEncodedId` → `getAccountIdFromAlias` → `getAccountFromAlias`:
```sql
SELECT id FROM entity
WHERE coalesce(deleted, false) <> true
  AND alias = $1
```
(`entityService.js` lines 17–20)

This query already guarantees the account exists and is not deleted. If it returns no rows, `getAccountIdFromAlias` (with `requireResult = true`) throws `NotFoundError` immediately, and execution never reaches line 172.

**DB Query 2** — triggered unconditionally on line 172 for every successful alias resolution:
```sql
SELECT type FROM entity
WHERE id = $1
```
(`entityService.js` lines 28–30, called from `isValidAccount` line 61)

**Root cause / failed assumption:** The developer assumed `getEncodedId` returns an ID without confirming existence, which is true for the numeric-ID branch (`EntityId.isValidEntityId` → pure arithmetic, no DB). For the alias branch, however, the DB lookup already acts as an existence gate. The `isValidAccount` call is therefore redundant only for the alias path, creating an asymmetry: numeric-ID requests cost 1 DB query; alias requests cost 2.

**Why existing checks fail:** The `entityFromAliasQuery` filters `coalesce(deleted, false) <> true`, which is semantically equivalent to the existence check performed by `isValidAccount`. There is no deduplication or short-circuit between the two calls.

### Impact Explanation

Each request using a valid base32 alias consumes two DB round-trips instead of one. Because the REST service has no per-IP or per-endpoint rate limiting (the `ThrottleConfiguration` / `ThrottleManagerImpl` found in the codebase belongs to the separate `web3` Java service, not the Node.js REST service), an unauthenticated attacker can sustain a high request rate. At scale, this doubles the DB query load originating from this single endpoint, which can exhaust the database connection pool, increase query queue depth, and degrade response times across all mirror-node REST endpoints sharing the same DB backend — consistent with the ≥30% processing degradation threshold described in the finding scope.

### Likelihood Explanation

No authentication or privilege is required. The base32 alias format is publicly documented (RFC 4648 alphabet `[A-Z2-7]+`). Valid aliases can be discovered by querying `/accounts` or observing on-chain data. The attack is trivially repeatable with a simple HTTP client loop and requires no special tooling. The absence of REST-layer rate limiting means there is no server-side throttle to overcome.

### Recommendation

1. **Eliminate the redundant query for the alias path.** When `getEncodedId` resolves via `getAccountIdFromAlias`, the account is already confirmed to exist. The `isValidAccount` call should be skipped in that case, or `getEncodedId` should return a richer result (e.g., `{id, verified: boolean}`) so the controller can avoid the second lookup.
2. **Add rate limiting to the REST service** for unauthenticated endpoints, mirroring the bucket-based throttle already present in the web3 service.
3. **Alternatively**, merge the two queries: a single `SELECT id, type FROM entity WHERE alias = $1 AND coalesce(deleted, false) <> true` returns both the ID and the existence confirmation in one round-trip.

### Proof of Concept

**Preconditions:**
- Mirror node REST API is publicly reachable.
- At least one account with a base32 alias exists (e.g., alias `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ` as used in the integration tests).

**Steps:**

```bash
# 1. Identify a valid alias (from /accounts listing or on-chain data)
ALIAS="AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"

# 2. Flood the rewards endpoint with the alias as path parameter
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/rewards" -o /dev/null &
done
```

**Observed behavior:**
- Each request triggers `entityFromAliasQuery` (DB Query 1) followed by `entityExistenceQuery` (DB Query 2).
- DB query rate is 2× that of equivalent numeric-ID requests.
- Under sustained load, DB connection pool exhaustion degrades all REST endpoints sharing the same pool.
