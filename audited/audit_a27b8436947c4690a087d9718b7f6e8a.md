### Title
Unbounded `nodeIds` List in `NetworkNodeRequest` Enables Resource Exhaustion on Address Book Endpoint

### Summary
`NetworkNodeRequest.nodeIds` has no `@Size` constraint, unlike the analogous `RegisteredNodesRequest.registeredNodeIds` which is capped at `@Size(max = 2)`. An unauthenticated attacker can submit arbitrarily many `node.id=eq:X` query parameters, causing `getNetworkNodes()` to build an unbounded `HashSet<Long>` and pass an unbounded `Long[]` array to PostgreSQL, consuming heap memory and database resources proportional to the number of parameters submitted.

### Finding Description

**Root cause — missing `@Size` on `nodeIds`:**

`NetworkNodeRequest.nodeIds` is declared as a plain `List<NumberRangeParameter>` with no size constraint: [1](#0-0) 

Compare with `RegisteredNodesRequest.registeredNodeIds`, where the developers explicitly applied `@Size(max = 2)`: [2](#0-1) 

This inconsistency confirms the constraint was known and intentionally applied elsewhere but omitted for `nodeIds`.

**Unbounded `HashSet<Long>` construction:**

In `NetworkServiceImpl.getNetworkNodes()`, every `EQ`-operator entry from the unconstrained list is unconditionally added to a `HashSet<Long>` with no size guard: [3](#0-2) 

The resulting set is then converted to a `Long[]` and passed directly to the native SQL query as a PostgreSQL array parameter: [4](#0-3) 

**Controller — no pre-validation:**

The controller passes the request directly to the service with no size check on `nodeIds`: [5](#0-4) 

**SQL layer — unbounded array in `ANY` clause:**

The native query uses `abe.node_id = any(:nodeIds)`, meaning a large array is sent to PostgreSQL and evaluated row-by-row: [6](#0-5) 

### Impact Explanation

Each `Long` object in a Java `HashSet` consumes approximately 48 bytes (16-byte object header + 32-byte `HashMap.Entry`). Tomcat's default `maxParameterCount` is 10,000, so a single request can force allocation of ~480 KB of heap per request. Under concurrent load from multiple attackers (or a single attacker with persistent connections), this accumulates. More critically, the unbounded `Long[]` array is serialized and sent to PostgreSQL as a native array parameter; a 10,000-element array in an `ANY` clause forces a full scan comparison against every address book entry, multiplying database CPU and I/O load. The address book endpoint is critical for gossip node discovery — degrading it prevents clients from learning the current network topology.

### Likelihood Explanation

No authentication is required. The endpoint is publicly accessible at `GET /api/v1/network/nodes`. The attack requires only a standard HTTP client capable of sending repeated `node.id` query parameters. It is trivially scriptable and repeatable. The only infrastructure-level mitigations are Tomcat's default `maxParameterCount` (10,000) and URL length limits, neither of which is enforced at the application layer and both of which can be reconfigured or bypassed depending on deployment topology (e.g., reverse proxies with higher limits).

### Recommendation

Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds`, consistent with the pattern already applied to `RegisteredNodesRequest.registeredNodeIds`. A reasonable maximum is 2–5 entries (matching the semantics of the field — users rarely need to filter by more than a handful of specific node IDs). Additionally, add an explicit size guard at the start of `getNetworkNodes()` as a defense-in-depth measure:

```java
// In NetworkNodeRequest.java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 5)
private List<NumberRangeParameter> nodeIds = List.of();
```

```java
// In NetworkServiceImpl.getNetworkNodes() — defense in depth
if (nodeIdParams.size() > MAX_NODE_ID_FILTERS) {
    throw new IllegalArgumentException("Too many node.id filters");
}
```

### Proof of Concept

```bash
# Generate a request with 10,000 distinct EQ node.id parameters
python3 -c "
params = '&'.join(f'node.id={i}' for i in range(10000))
print(f'GET /api/v1/network/nodes?{params}')
" | head -c 200

# Actual curl (split across multiple requests for URL length limits,
# or use a tool that supports large query strings):
python3 -c "
import urllib.request
params = '&'.join(f'node.id={i}' for i in range(10000))
url = f'http://<mirror-node-host>/api/v1/network/nodes?{params}'
urllib.request.urlopen(url)
"
```

Each such request forces allocation of a `HashSet<Long>` with up to 10,000 entries and sends a 10,000-element `Long[]` to PostgreSQL. Sending this request concurrently from multiple threads exhausts heap and saturates the database connection pool, making the address book endpoint unavailable.
