### Title
Binary Search Compute Amplification via Unthrottled `estimateGas` Iterations

### Summary
The `estimateGas()` function in `ContractExecutionService` always runs a binary search of up to 20 EVM re-executions for any successful contract call, because the sanity check `gasUsedByInitialCall >= params.getGas()` only fires on out-of-gas failures and never on normal successful calls. The gas/rate throttle is applied exactly once per HTTP request at the controller layer and does not account for the up to 20 additional EVM executions performed inside `BinaryGasEstimator.search()`, creating a ~15x compute amplification factor exploitable by any unprivileged user.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` once, consuming `scaleGas(request.getGas())` tokens from the gas bucket and 1 token from the rate bucket. No further throttle tokens are consumed for the remainder of the request. [1](#0-0) 

`ThrottleManagerImpl.throttle()` consumes tokens based solely on `request.getGas()` (the user-supplied gas limit), not on actual compute performed. [2](#0-1) 

`estimateGas()` at line 86 performs the sanity check `gasUsedByInitialCall >= params.getGas()`. This condition is only true when the EVM runs out of gas (OOG). For every successful contract call, the EVM returns `gasUsed < gasLimit` by definition, so this check is structurally never triggered for successful calls. The binary search always runs. [3](#0-2) 

`BinaryGasEstimator.search()` then executes up to `maxGasEstimateRetriesCount = 20` full EVM calls (line 35), each via `doProcessCall(params, gas, true)`, with no throttle token consumption per iteration. [4](#0-3) 

**Root cause:** The failed assumption is that the sanity check acts as a guard against the binary search. It does not — it only handles the degenerate OOG case. The throttle was designed around a single EVM execution per request, but `estimateGas` performs 1 + N executions (N ≤ 20).

**Maximizing iterations:** The number of binary search iterations is determined by the ratio `hi/lo` where `lo = gasUsedByInitialCall` and `hi = params.getGas()`. The attacker minimizes `lo` (use a cheap contract, e.g., a pure getter using ~21,000 gas) and maximizes `hi` (set `gas = 15,000,000`, the default `maxGasLimit`). The project's own test suite confirms this produces 14 iterations: [5](#0-4) 

The `estimateIterationThreshold` early-exit (`lo * 0.10 = 2,100` for `lo=21000`) is negligible relative to the 15M search space and does not prevent near-maximum iterations. [6](#0-5) 

### Impact Explanation

Each `estimateGas` request with `gas=15_000_000` and a cheap contract causes ~15 full EVM executions (1 initial + 14 binary search), but the throttle accounts for only 1. With the default `requestsPerSecond = 500` and `gasPerSecond = 7,500,000,000`: [7](#0-6) 

- Throttle permits 500 requests/second at max gas → 500 × 1 = 500 "accounted" EVM executions/second
- Actual compute: 500 × 15 = **7,500 EVM executions/second**
- Amplification factor: **~15×** above what the throttle models

This can saturate CPU/DB resources on the mirror node web3 service, causing degraded or denied service for all users. The `maxGasEstimateRetriesCount = 20` is a hard cap but is not enforced by the throttle. [8](#0-7) 

### Likelihood Explanation

No privileges, no deployed contract, and no authentication are required. Any caller can target an existing cheap contract (e.g., a simple ERC-20 `balanceOf`) or deploy a trivial one. The attack is fully repeatable at the rate limit ceiling (500 RPS), is stateless, and requires only standard JSON-RPC `eth_estimateGas` calls. The exploit is trivially scriptable.

### Recommendation

1. **Account for binary search amplification in the throttle**: For `estimate=true` requests, consume `scaleGas(request.getGas()) * maxGasEstimateRetriesCount` gas tokens upfront in `ThrottleManagerImpl.throttle()`, or consume one token per binary search iteration inside `BinaryGasEstimator.search()`.
2. **Separate, lower RPS limit for estimate requests**: Add a dedicated rate bucket for `ETH_ESTIMATE_GAS` call type, distinct from `ETH_CALL`, with a lower ceiling that reflects the true compute cost.
3. **Tighten `maxGasEstimateRetriesCount`**: Reduce the default from 20 to a value that balances accuracy against DoS risk, or make it configurable per call type.

### Proof of Concept

**Preconditions:** Any deployed contract with a cheap read function (e.g., ERC-20 `totalSupply()` using ~21,000 gas). No special permissions needed.

**Steps:**

1. Identify or deploy a contract with a function that uses minimal gas (e.g., ~21,000 gas).
2. Send the following request at maximum rate (500/second):
```
POST /api/v1/contracts/call
{
  "estimate": true,
  "gas": 15000000,
  "to": "<cheap_contract_address>",
  "data": "<cheap_function_selector>"
}
```
3. Observe server-side: each request triggers 1 initial EVM call + ~14 binary search iterations = ~15 EVM executions.
4. The throttle permits this at 500 RPS (consuming 1,500 gas tokens per request from a 750,000 token/second bucket).
5. Actual server compute: 500 × 15 = 7,500 EVM executions/second — 15× the throttle's model — causing CPU and database saturation.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
