### Title
Unauthenticated DoS via Uncached, Unthrottled `getSupply()` Endpoint Causing Unbounded Database Load

### Summary
The `GET /api/v1/network/supply` endpoint in the `rest-java` module has no rate limiting and no result caching. Every unauthenticated request triggers a full native SQL aggregation query against the `entity` table. An attacker can flood this endpoint to exhaust the database connection pool and degrade performance for all API consumers sharing the same PostgreSQL instance, including fee-related endpoints.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) is a public, unauthenticated `@GetMapping` with no throttle guard: [1](#0-0) 

It delegates to `NetworkServiceImpl.getSupply()` which, when no timestamp is provided, calls `entityRepository.getSupply()` directly with no caching: [2](#0-1) 

`EntityRepository.getSupply()` executes a native SQL aggregation (`SUM`, `MAX`) over the `entity` table joined with an `unnest()` of multiple ID ranges on every invocation: [3](#0-2) 

**Root cause — failed assumptions:**

1. **No caching**: Neither `@Cacheable` nor any in-memory cache is applied to `getSupply()`. Compare with `NodeStakeRepository.findAllStakeByConsensusTimestamp()` in the `grpc` module, which is explicitly `@Cacheable`. The supply result is static between importer ingestion cycles but is recomputed on every HTTP request. [4](#0-3) 

2. **No rate limiting in `rest-java`**: The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module and is wired only to `ContractController` and `OpcodesController`. The `rest-java` configuration contains only an ETag filter — no rate-limiting filter is registered for `/api/v1/network/supply`. [5](#0-4) [6](#0-5) 

### Impact Explanation

Each request issues a `SUM`/`MAX` aggregation joined with `unnest()` across up to 7 ID ranges on the `entity` table (configured default in `NetworkProperties`). On a production mirror node the `entity` table contains millions of rows. Flooding this endpoint saturates the PostgreSQL connection pool and increases I/O load, degrading response times for all other API consumers on the same database — including the `/api/v1/network/fees` endpoint which shares the same DB instance. This is a denial-of-service against the shared database tier. [7](#0-6) 

### Likelihood Explanation

The endpoint requires zero authentication, zero API keys, and zero special headers. Any external user can issue `GET /api/v1/network/supply` in a tight loop from a single machine or distributed botnet. No existing application-layer control in `rest-java` will reject or slow these requests. The attack is trivially scriptable (`curl`, `ab`, `k6` — the project itself ships a k6 load test for this exact endpoint). [8](#0-7) 

### Recommendation

1. **Add result caching**: Annotate `EntityRepository.getSupply()` (or the service layer call in `NetworkServiceImpl`) with `@Cacheable` using a short TTL (e.g., 5–15 seconds). The supply value only changes when the importer ingests new balance data, so stale-by-seconds results are acceptable.

2. **Add rate limiting to `rest-java`**: Introduce a `bucket4j`-based filter (mirroring the `web3` `ThrottleConfiguration`) scoped to the `rest-java` module and apply it to all `/api/v1/network/*` endpoints, or at minimum to `/supply`.

3. **Add a database statement timeout**: Ensure `spring.datasource` for `rest-java` has a `statement_timeout` set so runaway queries from a flood are killed at the DB level.

### Proof of Concept

```bash
# No authentication required. Run from any machine.
# Flood the endpoint at high concurrency:
ab -n 100000 -c 200 https://<mirror-node-host>/api/v1/network/supply

# Or with curl in a tight loop:
while true; do
  curl -s https://<mirror-node-host>/api/v1/network/supply -o /dev/null &
done

# Observable effect: PostgreSQL pg_stat_activity will show hundreds of
# concurrent "select cast(coalesce(sum(e.balance)..." queries;
# connection pool exhaustion causes 500 errors on /api/v1/network/fees
# and other endpoints sharing the same DB.
```
