### Title
Global Singleton `opcodeRateLimitBucket` Allows Unprivileged User to Starve All Legitimate Opcode Requests

### Summary
The `opcodeRateLimitBucket` is a single application-wide Spring singleton bean with a default capacity of 1 token per second, shared across every caller with no per-user or per-IP isolation. Any unauthenticated user can consume the sole available token at the start of each refill window, causing every concurrent or subsequent `throttleOpcodeRequest()` call from legitimate users to throw `ThrottleException` for the remainder of that window. This is trivially repeatable at 1 request per second indefinitely.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 47–54 define the `opcodeRateLimitBucket` as a single `@Bean` (Spring singleton):

```java
@Bean(name = OPCODE_RATE_LIMIT_BUCKET)
Bucket opcodeRateLimitBucket() {
    long rateLimit = throttleProperties.getOpcodeRequestsPerSecond(); // default = 1
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
``` [1](#0-0) 

`ThrottleProperties.java` line 29 sets the default to 1:
```java
@Min(1)
private long opcodeRequestsPerSecond = 1;
``` [2](#0-1) 

`ThrottleManagerImpl.java` lines 52–56 consume from this single global bucket with no caller identity check:
```java
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
``` [3](#0-2) 

**Root cause:** The bucket is a global singleton with capacity=1. `tryConsume(1)` is called identically for every caller regardless of origin. There is no per-IP, per-session, or per-user bucket. The first caller each second wins; all others lose.

**Why existing checks fail:** The `throttle()` path has a `rateLimitBucket` with a default of 500 req/s, providing meaningful headroom. The `opcodeRateLimitBucket` has a default of 1 req/s with no analogous per-user partitioning. No authentication, IP filtering, or fairness mechanism is applied before `tryConsume`. [4](#0-3) 

### Impact Explanation
An attacker sending exactly 1 opcode request per second completely monopolizes the global opcode budget. All legitimate users are denied opcode functionality for the entire duration of the attack. The impact is a full denial-of-service on the opcode debug/tracing endpoint with zero economic cost to the attacker. Severity is medium (griefing, no fund loss, but complete feature unavailability).

### Likelihood Explanation
Preconditions: none — no account, no authentication, no special knowledge required. The attacker needs only to discover the opcode endpoint (publicly documented) and send 1 HTTP request per second. The attack is fully automatable with a simple cron job or `curl` loop, is indefinitely repeatable, and cannot be distinguished from a legitimate single request per second.

### Recommendation
1. **Per-caller rate limiting:** Replace the single global bucket with per-IP (or per-authenticated-user) buckets, e.g., using Bucket4j's `ProxyManager` backed by a cache (Caffeine, Redis) keyed on the caller's IP address.
2. **Increase default capacity:** Raise `opcodeRequestsPerSecond` to a value that reflects realistic concurrent user counts, reducing the blast radius of any single consumer.
3. **IP-level pre-filter:** Apply a servlet filter or Spring `HandlerInterceptor` that enforces per-IP limits before the shared global bucket is consulted.
4. **Require authentication:** Gate opcode/debug endpoints behind authentication so abuse can be attributed and blocked.

### Proof of Concept
```bash
# Attacker terminal: consume the single global token every second
while true; do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call/opcodes \
       -H "Content-Type: application/json" \
       -d '{"data":"0x","gas":21000}' &
  sleep 0.9   # slightly under 1s to reliably win the race
done

# Legitimate user terminal (concurrent): always receives ThrottleException
curl -X POST http://<mirror-node>/api/v1/contracts/call/opcodes \
     -H "Content-Type: application/json" \
     -d '{"data":"0x","gas":21000}'
# Response: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The attacker requires no privileges. The `opcodeRateLimitBucket` singleton is exhausted before any legitimate caller can reach it, and this repeats every refill window indefinitely. [3](#0-2) [1](#0-0)
