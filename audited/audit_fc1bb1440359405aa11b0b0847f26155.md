### Title
Unvalidated Mirror REST Node Endpoints Enable gRPC Network Blackholing

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` blindly accepts any non-empty `domain_name` or `ip_address_v4` value from the mirror REST `/api/v1/network/nodes` response without any IP-range or hostname validation. An attacker who can control that HTTP response — via DNS poisoning against a plaintext `http://` mirror URL or a network-level MITM — can inject non-routable addresses (e.g., `240.0.0.0/4`) for every node, causing the Hiero SDK to attempt gRPC connections that all time out, permanently preventing transaction confirmation.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–124:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

The only guards are: (a) non-empty host string, (b) non-zero port. There is no IP-range check, no RFC-1918/reserved-range rejection, no domain resolution check, and no response authentication. Any syntactically valid string passes.

The resulting `network` map is passed directly to `hiero.ClientForNetwork(netmap)` in `pinger/sdk_client.go` line 22, which the SDK uses for all gRPC node connections.

**Root cause / failed assumption:** The code assumes the mirror REST endpoint is always trustworthy and returns legitimate routable addresses. There is no defense-in-depth validation of the returned endpoint data.

**Exploit flow:**

1. The operator deploys with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plaintext HTTP, as shown in the config example).
2. An attacker with DNS poisoning capability (or on-path network position) intercepts the HTTP GET to `/api/v1/network/nodes`.
3. The attacker returns a crafted JSON body:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"domain_name": "240.0.0.1", "ip_address_v4": "", "port": 50211}
      ]
    }
  ],
  "links": {"next": null}
}
```
4. `fetchMirrorNodeNetwork()` accepts this — `node_account_id` parses fine, `domain_name` is non-empty, port is non-zero — and returns `{"240.0.0.1:50211": AccountID{0,0,3}}`.
5. `hiero.ClientForNetwork()` is initialized with this map. Every subsequent `Execute()` call in `submitWithRetry()` dials `240.0.0.1:50211`, which is in the IANA-reserved `240.0.0.0/4` range and will never respond.
6. All gRPC calls time out; `submitWithRetry()` exhausts all retries and logs failure every tick.

**Why existing checks fail:** The `len(network) == 0` guard at line 127–129 only fires if the attacker returns an empty or malformed response. A response with syntactically valid but non-routable entries passes this check and produces a non-empty (but useless) network map.

### Impact Explanation
The pinger's sole function is to submit periodic Hiero transfers to confirm network liveness. With all gRPC endpoints pointing to non-routable addresses, every `Execute()` call times out, every receipt fetch fails, and the pinger reports continuous transfer failures. From a monitoring/alerting perspective this is indistinguishable from a real network outage, causing false-positive network-down alerts and suppressing legitimate incident detection. Severity: **High** (complete loss of the pinger's monitoring function for the duration of the attack).

### Likelihood Explanation
The attack is feasible whenever the mirror REST URL uses plaintext HTTP (the documented example is `http://mirror-rest:5551`). In a Kubernetes cluster, an attacker who has compromised any pod on the same network namespace, or who can perform DNS spoofing against the cluster DNS, can inject the malicious response. No Hiero credentials or privileged API access are required — only the ability to respond to an unauthenticated HTTP GET. The attack is repeatable: since `buildNetworkFromMirrorNodes` is called once at startup (`sdk_client.go` line 18), a single poisoned response permanently corrupts the client for the lifetime of the process.

### Recommendation
1. **Validate returned IP addresses** before inserting into the network map. Reject addresses in non-routable ranges (`0.0.0.0/8`, `240.0.0.0/4`, loopback, link-local, etc.) using `net.ParseIP` + range checks.
2. **Enforce HTTPS** for the mirror REST URL. Reject `http://` schemes in `loadConfig()` or at least warn loudly. Use Go's default TLS verification (already present) but make plaintext opt-in with an explicit flag.
3. **Require a minimum number of distinct node entries** (e.g., ≥ 3) before accepting the network map, so a response with a single poisoned node is rejected.
4. **Periodically refresh** the network map (not just at startup) so a poisoned response does not persist for the process lifetime.
