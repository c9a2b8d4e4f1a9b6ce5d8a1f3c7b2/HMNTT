### Title
Unauthenticated Pagination Flooding via Unbounded Request Rate in TokenAirdropsController

### Summary
The `processRequest()` method in `TokenAirdropsController` accepts unauthenticated GET requests with no application-level rate limiting. An attacker can automate cursor-based pagination at maximum speed using the `next` link returned in `TokenAirdropsResponse.links`, issuing a sustained stream of DB queries against the `token_airdrop` table. Unlike the `web3` module which has explicit bucket4j throttling, the `rest-java` module has no equivalent protection.

### Finding Description

**Exact code path:**

`TokenAirdropsController.java`, `processRequest()`, lines 89–114: [1](#0-0) 

The only input constraint on `limit` is `@Positive @Max(MAX_LIMIT)` (max=100), which caps rows per response but places no bound on request frequency: [2](#0-1) 

Each call to `processRequest()` triggers `TokenAirdropServiceImpl.getAirdrops()`, which issues two DB round-trips: one entity lookup and one `SELECT … FROM token_airdrop WHERE … LIMIT ?`: [3](#0-2) [4](#0-3) 

**Root cause — missing rate limiting in rest-java:**

The `rest-java` module registers only two servlet filters: `LoggingFilter` (logging only) and `MetricsFilter` (metrics only). Neither throttles requests: [5](#0-4) [6](#0-5) 

`RestJavaConfiguration` registers only an ETag filter and a protobuf converter — no rate-limiting bean: [7](#0-6) 

`WebMvcConfiguration` adds only argument resolvers and formatters: [8](#0-7) 

By contrast, the `web3` module has explicit bucket4j rate limiting (`ThrottleManagerImpl`, `ThrottleConfiguration`) that enforces per-second request and gas limits: [9](#0-8) [10](#0-9) 

No equivalent protection exists anywhere in `rest-java`.

**Pagination amplification:**

`LinkFactoryImpl.create()` unconditionally emits a `next` link whenever the result set equals the page size, making automated cursor-following trivial: [11](#0-10) 

### Impact Explanation

Each paginated request issues at least two synchronous DB queries. With no rate limiting, a single attacker thread can sustain hundreds of requests per second; multiple concurrent clients (or distributed bots) multiply this linearly. The shared PostgreSQL instance backing all mirror node services (importer, REST, rest-java) can be I/O-saturated, degrading or halting airdrop, allowance, and other query endpoints across all mirror node processing nodes. Because mirror nodes are the sole read path for dApps and wallets querying Hedera state, sustained degradation of 30%+ of deployed mirror nodes constitutes a meaningful availability impact on the broader ecosystem.

### Likelihood Explanation

- **No authentication required**: GET endpoints are fully public.
- **No skill barrier**: Following `links.next` in a loop is trivial scripting.
- **No cost barrier**: A single machine with a fast network connection suffices; no brute-force or cryptographic work is needed.
- **Repeatable**: The attack can be sustained indefinitely and restarted after any mitigation attempt that does not involve code changes.
- **Amplifiable**: Multiple accounts (`{id}` path parameter) can be targeted in parallel, multiplying DB load.

### Recommendation

1. Add a bucket4j rate-limiting filter to `rest-java` mirroring the pattern in `web3/ThrottleConfiguration` — enforce a global requests-per-second cap and optionally a per-IP cap.
2. Register the filter in `RestJavaConfiguration` covering `/api/*` patterns.
3. Consider adding a short-lived query-result cache (e.g., Caffeine) for repeated identical pagination cursors to reduce redundant DB hits.
4. Enforce DB connection pool limits (`spring.datasource.hikari.maximum-pool-size`) as a backstop.

### Proof of Concept

```bash
# Step 1: Initial request
NEXT="/api/v1/accounts/0.0.12345/airdrops/outstanding?limit=100"
BASE="http://<mirror-node-host>"

# Step 2: Automated pagination loop (no auth needed)
while true; do
  RESPONSE=$(curl -s "${BASE}${NEXT}")
  NEXT=$(echo "$RESPONSE" | jq -r '.links.next // empty')
  [ -z "$NEXT" ] && NEXT="/api/v1/accounts/0.0.12345/airdrops/outstanding?limit=100"
  # No sleep — maximum speed
done

# Step 3: Run N parallel instances of the above loop from different IPs/accounts
# to multiply DB query load until mirror node DB I/O is saturated.
```

Each loop iteration triggers two synchronous DB queries with no server-side throttle. Sustained parallel execution exhausts the database connection pool and I/O capacity, degrading all mirror node endpoints sharing the same DB instance.

    private TokenAirdropsResponse processRequest(
            EntityIdParameter id,

    @Override
    public <T> Links create(List<T> items, Pageable pageable, Function<T, Map<String, String>> extractor) {
        if (CollectionUtils.isEmpty(items) || pageable.getPageSize() > items.size()) {
            return DEFAULT_LINKS;
        }

        final var servletRequestAttributes = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
        if (servletRequestAttributes == null || servletRequestAttributes.getResponse() == null) {
            return DEFAULT_LINKS;
        }

        var request = servletRequestAttributes.getRequest();
        var lastItem = Objects.requireNonNull(CollectionUtils.lastElement(items));
        var nextLink = createNextLink(lastItem, pageable, extractor, request);

        // If nextLink is null, it means the pagination range would be empty - no more results
        if (nextLink == null) {
            return DEFAULT_LINKS;
        }

        servletRequestAttributes.getResponse().setHeader(HttpHeaders.LINK, LINK_HEADER.formatted(nextLink));
        return new Links().next(nextLink);
    }
