### Title
Unbounded Per-Subscription DB Polling DoS When POLL Listener Type Is Configured

### Summary
When the gRPC mirror node is configured with `ListenerType.POLL`, every call to `subscribeTopic` creates an independent, perpetually-repeating database polling loop via `PollingTopicListener`. Because the `subscribeTopic` gRPC endpoint requires no authentication and enforces no limit on concurrent subscriptions, an unprivileged attacker can open arbitrarily many connections and subscriptions, multiplying DB query load linearly with no server-side cap.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any unauthenticated gRPC call and delegates to `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92), which calls `topicListener.listen(newFilter)` (line 120). `CompositeTopicListener.getTopicListener()` (lines 46–59) reads `listenerProperties.getType()` — a server-side Spring Boot config — and when it equals `POLL`, returns `PollingTopicListener`.

`PollingTopicListener.listen()` (lines 34–49) constructs a **cold, per-subscriber** reactive pipeline:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)   // default 500 ms, min 50 ms
                .withScheduler(scheduler))
```

Each subscription gets its own `PollingContext` and its own `RepeatSpec` loop that fires `topicMessageRepository.findByFilter()` every `interval` milliseconds, indefinitely (`Long.MAX_VALUE` repeats). There is no shared state between subscribers — contrast with `SharedPollingTopicListener`, which uses `.share()` so all subscribers share one loop.

**Root cause:** `PollingTopicListener` is a cold publisher. Every `listen()` call materialises a new, independent DB polling loop. No code path caps the number of concurrent subscriptions or the resulting DB query rate.

**Why existing checks fail:**

- `NettyProperties.maxConcurrentCallsPerConnection = 5` (the only numeric guard found) limits calls *per TCP connection*, not globally. An attacker opens many connections.
- `subscriberCount` in `TopicMessageServiceImpl` (lines 48–55) is a Micrometer gauge for observability only — it is never compared against any threshold to reject new subscriptions.
- No authentication, IP-based rate limiting, or per-client subscription quota exists anywhere in the gRPC stack.

**Exploit flow:**

1. Attacker opens *C* TCP connections to the gRPC port (default 5600).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams (the per-connection limit), for *5C* total subscriptions.
3. Each subscription triggers `PollingTopicListener.listen()`, spawning an independent loop that issues `SELECT … FROM topic_message WHERE …` every 500 ms.
4. Total DB query rate = *5C / 0.5 s* = *10C* queries/second, growing linearly with *C*.
5. Subscriptions are kept alive indefinitely (no `limit`, no `endTime` required by the proto).

### Impact Explanation
With POLL mode active, a single attacker machine opening a few hundred TCP connections can saturate the database with thousands of queries per second, causing query latency spikes or connection pool exhaustion for all mirror node consumers. Because the polling interval has a minimum of 50 ms, the worst-case rate is 20× higher than the default. This is a full availability denial-of-service against the database tier with no collateral requirement beyond network access to the gRPC port.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly documented and reachable without credentials. The only precondition is that the operator has chosen `ListenerType.POLL` (not the default `REDIS`, but a documented and supported option). An attacker needs no special knowledge beyond the proto definition and the ability to open many TCP connections — both trivially achievable. The attack is fully repeatable and requires no state.

### Recommendation
1. **Enforce a global subscription limit:** Add a configurable `maxSubscribers` property to `ListenerProperties` or `GrpcProperties`. In `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount.get() >= maxSubscribers` before proceeding and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Prefer `SHARED_POLL` over `POLL`:** `SharedPollingTopicListener` already uses `.share()` so all subscribers share one DB polling loop regardless of subscriber count. Deprecate or remove `POLL` in favour of `SHARED_POLL`.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.
4. **Document the risk** of `POLL` mode in operator configuration guides.

### Proof of Concept

```bash
# Requires grpcurl; repeat in parallel to scale
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}, "limit": 0}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Each background process holds an open subscription.
# With POLL mode: 200 independent DB queries fire every 500 ms = 400 queries/sec
# from a single attacker host. Scale C for more impact.
```

Observe DB connection pool metrics and query rate via the exposed Micrometer endpoint (`/actuator/metrics/hiero_mirror_grpc_listener`) to confirm linear growth with subscriber count.
