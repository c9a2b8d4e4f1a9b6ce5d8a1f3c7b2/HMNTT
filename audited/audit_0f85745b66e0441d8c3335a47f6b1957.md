### Title
Unauthenticated Entity Enumeration via Unrestricted GraphQL Endpoint with No Rate Limiting

### Summary
The `getByIdAndType()` method in `EntityServiceImpl` performs a direct, uncached database primary-key lookup for every request with no rate limiting, authentication, or request throttling on the GraphQL endpoint. An unprivileged external attacker can submit sequential numeric entity IDs and observe both the response content (null vs. populated object) and response timing/size differences to enumerate all existing entity IDs in the database. The timing side-channel is compounded by a direct content oracle, making enumeration reliable without any statistical analysis.

**Dual oracle â€” content and timing:**

1. **Content oracle (primary):** A hit returns `{"data": {"account": { ...full fields... }}}` while a miss returns `{"data": {"account": null}}`. No timing measurement is needed; the response body directly reveals existence.
2. **Timing/size oracle (secondary):** A DB hit transfers a full entity row back to the application and serializes it; a miss transfers nothing. This produces a measurable difference in both response latency and response byte size, observable at the network level even through TLS (via packet sizes).

### Impact Explanation

An attacker can enumerate every account, contract, token, topic, file, and schedule entity ID in the mirror node database without any credentials. This leaks the full set of active entity IDs on the network. Beyond information disclosure, the absence of rate limiting means an attacker can sustain a high-volume query flood that exhausts database connection pool resources, causing denial of service for legitimate users. Each request triggers a real SQL query against the `entity` table.

### Likelihood Explanation

Exploitation requires only an HTTP client and knowledge of the public GraphQL endpoint â€” no credentials, no special network position, no cryptographic capability. Entity IDs are small sequential integers starting from 1, so a complete enumeration of even millions of IDs is achievable with a simple script in minutes. The content oracle makes the attack 100% reliable; timing measurement is not even required. The attack is fully repeatable and automatable.

### Recommendation

1. **Add rate limiting to the GraphQL module.** Introduce a `ThrottleFilter` or Spring `HandlerInterceptor` analogous to the `web3` module's `ThrottleManagerImpl`, enforcing a per-IP or global requests-per-second cap on the `/graphql` endpoint.
2. **Add response-time normalization** for hit vs. miss paths if timing side-channel mitigation is specifically required (e.g., constant-time delay padding).
3. **Consider authentication** for the GraphQL API if the data is not intended to be fully public.
4. **Add caching** to `EntityRepository.findById()` in the graphql module (mirroring the `@Cacheable` annotations in the `web3` module's `EntityRepository`) to reduce per-request DB load.

### Proof of Concept

```bash
# Enumerate accounts by iterating numeric entity IDs
for i in $(seq 1 100000); do
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {entityId: {num: $i}}) { id balance } }\"}" &
done
```

- Response `{"data":{"account":null}}` â†’ entity ID `$i` does not exist as an ACCOUNT.
- Response `{"data":{"account":{"id":"...","balance":...}}}` â†’ entity ID `$i` exists.

No authentication header is required. No rate-limit response (HTTP 429) will be returned. The loop can be parallelized arbitrarily.
