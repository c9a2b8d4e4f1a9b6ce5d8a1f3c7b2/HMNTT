### Title
Stale Replica Read Returns Deleted Hook Storage Entries as Active Due to Missing pgpool Load-Balance Guard in rest-java

### Summary
The `HookServiceImpl.getHookStorage()` method routes `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` queries through pgpool without the `/* NO PGPOOL LOAD BALANCE */` comment guard that the importer module uses to force reads to the primary. During replication lag (including network partition), pgpool can route the query to a replica that has not yet applied the deletion of a hook storage slot, causing the `deleted = false` filter to pass on stale data and returning the deleted entry as if it were active. Any unprivileged user can trigger this with a single-key range query (`keyLowerBound == keyUpperBound`).

### Finding Description

**Code path:**

`HookServiceImpl.java` lines 56–69 — when `request.isHistorical()` is false and `keys.isEmpty()` is true, the service calls:

```java
hookStorageRepository.findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(
    ownerId.getId(), request.getHookId(),
    request.getKeyLowerBound(), request.getKeyUpperBound(), page);
```

`HookStorageRepository.java` line 16–17 — this is a Spring Data JPA derived query that translates to `WHERE ... AND deleted = false`. The `deleted` column is a real persisted boolean in the `hook_storage` table (set by `HookStorage.setValue()` when value is empty/null).

**Root cause — missing stale-read guard:**

The importer module (`HibernateConfiguration.java` lines 19–35) explicitly documents and implements a fix for the stale read-after-write problem: it prepends `/* NO PGPOOL LOAD BALANCE */` to every SQL statement when `loadBalance = false`, forcing pgpool to send the query to the primary node. The rest-java module has **no equivalent `HibernateConfiguration`**, no `loadBalance` property, and no `StatementInspector`. Searches across all rest-java source confirm zero occurrences of `NO_LOAD_BALANCE`, `StatementInspector`, or `HibernateConfiguration`.

The production deployment (`values-prod.yaml` line 78) configures `replicaCount: 2` with pgpool load balancing active by default. pgpool distributes `SELECT` statements across primary and replicas.

**Exploit flow:**

1. A hook storage slot with key `K` exists with `deleted = false` on the primary.
2. A transaction deletes the slot: primary sets `deleted = true` (value cleared to empty).
3. Before the WAL change replicates to a replica (replication lag is normal; during a partition it can be indefinite), an attacker sends:
   ```
   GET /api/v1/{ownerId}/hooks/{hookId}/storage?key=eq:K
   ```
   This produces `keyLowerBound = keyUpperBound = K` (single-key range), entering the `keys.isEmpty()` branch at line 65.
4. pgpool load-balances the resulting `SELECT ... WHERE deleted = false` to the lagging replica.
5. The replica still has `deleted = false` for key `K`; the filter passes and the deleted entry is returned as active.

**Why existing checks are insufficient:**

The `DeletedIsFalse` predicate is correct in isolation but depends entirely on the replica having applied the deletion. There is no mechanism in rest-java to force the query to the primary, no `@Transactional(readOnly=true)` with a primary-routing hint, and no application-level staleness check. The importer's own comment ("used to prevent the stale read-after-write issue") confirms this is a known, documented risk that was deliberately mitigated in the importer but left unaddressed in rest-java.

### Impact Explanation
An attacker or any API consumer can observe hook storage slots that have been deleted as if they are still active, with their last-written value intact. For applications that use hook storage state to gate logic (e.g., checking whether a slot is cleared before acting), this produces incorrect state. The severity is moderate: it is a data-integrity/confidentiality issue (stale state disclosure) rather than an authentication bypass, but it undermines the correctness guarantee of the mirror node API for hook storage.

### Likelihood Explanation
Replication lag is a normal operational condition, not an exotic scenario. Under any load spike, network hiccup, or deliberate partition, lag of hundreds of milliseconds to seconds is routine. The attacker needs no credentials — the endpoint is a public HTTP GET. The single-key range (`key=eq:K`) is a documented, valid query parameter accepted by `HooksController`. The attack is trivially repeatable: the attacker simply polls the endpoint until pgpool routes to the lagging replica, which is statistically likely given two replicas and round-robin load balancing.

### Recommendation
Apply the same guard used in the importer to the rest-java module:

1. Add a `HibernateConfiguration` (or equivalent `HibernatePropertiesCustomizer`) to rest-java that registers a `StatementInspector` prepending `/* NO PGPOOL LOAD BALANCE */` to all queries, controlled by a `loadBalance` property (defaulting to `true` for backward compatibility, set to `false` in deployments where stale reads are unacceptable).
2. Alternatively, annotate `HookServiceImpl.getHookStorage()` with `@Transactional(readOnly = true)` and configure Spring's transaction routing to direct read-only transactions to the primary, or use a pgpool `SET` hint at the session level.
3. At minimum, document the known stale-read window and add a monitoring alert on replication lag (a dashboard for lag already exists in `postgresql-database.json` line 3062) with an SLO threshold.

### Proof of Concept

**Preconditions:**
- pgpool with ≥1 replica and load balancing enabled (default production config).
- A hook storage slot `(ownerId=O, hookId=H, key=K)` exists with a non-empty value (`deleted = false`).

**Steps:**
1. Delete the slot (set value to empty) via the importer/consensus path so the primary marks `deleted = true`.
2. Immediately (within the replication lag window) send repeated requests:
   ```
   GET /api/v1/{O}/hooks/{H}/storage?key=eq:{K}
   ```
3. Observe that some responses return the slot with its old value (routed to lagging replica), while others return an empty list (routed to primary or caught-up replica).
4. The non-empty response is the stale read — the deleted entry is returned as active.

**Verification:** Introduce artificial replication lag (e.g., `pg_wal_replay_pause()` on the replica) and the stale response becomes 100% reproducible for the duration of the pause. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
