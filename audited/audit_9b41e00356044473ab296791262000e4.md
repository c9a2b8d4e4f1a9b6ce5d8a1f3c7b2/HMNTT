### Title
SSRF via Unguarded HTTP Redirect Following in `fetchMirrorNodeNetwork()`

### Summary
The `http.Client` constructed in `buildNetworkFromMirrorNodes()` has no `CheckRedirect` policy, so Go's default behavior silently follows up to 10 consecutive 3xx redirects. A mirror REST server that returns an HTTP 301/302 pointing to an internal address (e.g., the cloud instance-metadata service) will cause the pinger to issue a full HTTP GET to that internal target, constituting a Server-Side Request Forgery.

### Finding Description
**Exact location:** `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` line 46 and `fetchMirrorNodeNetwork()` lines 74–132.

```go
// line 46 – no CheckRedirect set
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
``` [1](#0-0) 

```go
// line 84 – executes request; default client follows 3xx automatically
resp, err := httpClient.Do(req)
``` [2](#0-1) 

**Root cause:** Go's `net/http` documentation states that when `Client.CheckRedirect` is `nil` the client uses its default policy of following up to 10 consecutive redirects. Because no `CheckRedirect` is supplied, any 301/302 returned by the mirror REST server is transparently followed to whatever `Location` header the server provides, including RFC-1918 addresses or link-local metadata endpoints.

**Why the existing check is insufficient:**
```go
// line 90-93 – evaluated on the *final* response after all redirects
if resp.StatusCode < 200 || resp.StatusCode >= 300 {
    retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
    return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
}
``` [3](#0-2) 

By the time this guard executes, the redirect has already been followed and the internal service has already received and responded to the request. If the internal service returns HTTP 200, the guard passes entirely and the response body is fed to the JSON decoder.

**Exploit flow:**
1. Attacker controls or compromises the mirror REST server pointed to by `HIERO_MIRROR_PINGER_REST`.
2. The server responds to `GET /api/v1/network/nodes` with `HTTP 301 Location: http://169.254.169.254/latest/meta-data/` (AWS IMDSv1) or any other internal HTTP endpoint.
3. `httpClient.Do(req)` follows the redirect; the pinger issues `GET http://169.254.169.254/latest/meta-data/` from within the cloud instance.
4. The metadata service returns HTTP 200; the status-code guard at line 90 passes.
5. The JSON decoder fails on the non-JSON body, but the internal request has already been made and any side-effects (credential exposure via IMDSv1, triggering internal APIs) have occurred.

### Impact Explanation
In a cloud-hosted deployment (AWS, GCP, Azure, Kubernetes), the pinger process has access to link-local metadata services and internal cluster endpoints that are unreachable from the public internet. A successful redirect to `http://169.254.169.254/latest/meta-data/iam/security-credentials/<role>` (AWS IMDSv1, no token required) leaks temporary IAM credentials, enabling full account compromise. Even without credential theft, the attacker gains the ability to probe and interact with any HTTP service reachable from the pinger's network namespace.

### Likelihood Explanation
The precondition — controlling the mirror REST server — is realistic in several scenarios: a supply-chain compromise of a shared mirror node, DNS hijacking of the configured hostname, a BGP/ARP poisoning attack on the network path, or simply misconfiguring the pinger to point at an attacker-controlled URL. The `network=other` code path (the only path that calls `buildNetworkFromMirrorNodes`) is explicitly designed for custom/private deployments where the mirror URL is operator-supplied, making the attack surface broader. The exploit requires no authentication and is fully repeatable on every startup or retry cycle. [4](#0-3) 

### Recommendation
Disable automatic redirect following entirely by setting `CheckRedirect` to return `http.ErrUseLastResponse`, which causes `Do` to return the 3xx response directly without following it:

```go
httpClient := &http.Client{
    Timeout: cfg.mirrorNodeClientTimeout,
    CheckRedirect: func(req *http.Request, via []*http.Request) error {
        return http.ErrUseLastResponse
    },
}
```

This ensures the status-code guard at line 90 will catch the 3xx and treat it as an error, preventing any redirect from being followed. Alternatively, if redirects must be supported, validate the `Location` URL against an allowlist of permitted hosts/schemes before following. [1](#0-0) 

### Proof of Concept
1. Stand up a minimal HTTP server that responds to any request with:
   ```
   HTTP/1.1 301 Moved Permanently
   Location: http://169.254.169.254/latest/meta-data/iam/security-credentials/
   ```
2. Configure the pinger: `HIERO_MIRROR_PINGER_NETWORK=other HIERO_MIRROR_PINGER_REST=http://<attacker-server>`.
3. Start the pinger. `buildNetworkFromMirrorNodes` calls `fetchMirrorNodeNetwork`, which calls `httpClient.Do(req)`.
4. Observe in the attacker server's logs that the pinger's IP issued a second request to `http://169.254.169.254/...` (or capture it via `tcpdump` on the pinger host).
5. On AWS with IMDSv1 enabled, the response body contains the instance's IAM role credentials in plaintext.
