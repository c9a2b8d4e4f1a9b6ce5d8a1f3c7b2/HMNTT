### Title
Unbounded Alias Length Allows Unauthenticated Per-Request CPU and DB I/O Amplification in `getEncodedId()`

### Summary
The `accountAliasRegex` in `rest/accountAlias.js` uses `[A-Z2-7]+` with no upper-bound quantifier, accepting alias strings of arbitrary length (bounded only by the HTTP server's URL limit, ~8 KB). An unauthenticated attacker can submit `shard.realm.AAAA…` strings at maximum URL length, forcing `base32.decode()` and an uncached database query on every request, maximizing CPU and I/O cost per request with no application-layer rate limiting or caching in the REST service.

### Finding Description

**Exact code path:**

`rest/accountAlias.js` line 10 defines the validation regex:
```js
const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
```
The `[A-Z2-7]+` quantifier has no upper bound. A string like `0.0.AAAA…` (thousands of uppercase base32 characters) passes `isValid()`.

`rest/service/entityService.js` lines 125–126:
```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
}
```
`AccountAlias.fromString()` calls `new AccountAlias(...parts)`, which at `rest/accountAlias.js` line 24 calls:
```js
this.alias = base32.decode(base32Alias);
```
`base32.decode` (`rest/base32.js` line 13) calls `base32.parse(str, {loose: true})` — a full linear scan + `Uint8Array` allocation proportional to alias length.

`getAccountIdFromAlias()` → `getAccountFromAlias()` → `rest/service/baseService.js` line 56:
```js
return (await this.pool().queryQuietly(query, params)).rows;
```
No caching layer exists anywhere in this call chain. Every request hits the database with the decoded binary blob as a query parameter.

**Root cause:** The application-level regex validates only the character set, not the length. The OpenAPI spec at `rest/api/v1/openapi.yml` line 4694 defines a structured pattern `(?:[A-Z2-7]{8})*(?:[A-Z2-7]{2}|[A-Z2-7]{4,5}|[A-Z2-7]{7,8})` that would constrain valid base32 lengths to legitimate key sizes, but this constraint is **never enforced** in the actual JavaScript validation code.

**Why existing checks fail:**
- `AccountAlias.isValid()` only checks character set, not length.
- `AccountAlias.fromString()` re-calls `isValid()` before constructing — same gap.
- `BaseService` has no result cache.
- The REST service's `metricsHandler.js` only records metrics; it does not rate-limit.
- The Traefik rate-limit and bucket4j throttle found in the repo apply to the Rosetta and web3 services respectively, not to the Node.js REST service.

### Impact Explanation
Each crafted request with a ~8 KB alias causes: (1) regex evaluation on ~8 000 characters, (2) `base32.decode` allocating a ~5 000-byte `Uint8Array`, (3) a database query transmitting ~5 000 bytes as a binary parameter against the `entity` table. A legitimate ED25519 alias is 32 bytes → 56 base32 characters; the attacker's payload is ~143× larger per request. Sustained concurrent requests saturate the Node.js event loop and the PostgreSQL connection pool, causing service unavailability for all users. This qualifies as a non-network DoS against a component that handles account lookups across the mirror node REST API.

### Likelihood Explanation
No authentication is required. The endpoint is publicly accessible. The attacker needs only an HTTP client and knowledge of the `shard.realm.alias` URL format (documented in the OpenAPI spec). The attack is trivially repeatable and scriptable. A single attacker with modest bandwidth can sustain enough concurrent requests to exhaust the DB connection pool.

### Recommendation
1. **Enforce a maximum alias length** in `accountAliasRegex` to match the OpenAPI spec's structured pattern, e.g., change `[A-Z2-7]+` to `(?:[A-Z2-7]{8})*(?:[A-Z2-7]{2}|[A-Z2-7]{4,5}|[A-Z2-7]{7,8})` (max 88 characters for a DER-encoded key).
2. **Add an explicit length guard** in `AccountAlias.isValid()` before regex evaluation: reject any string whose alias component exceeds, e.g., 100 characters.
3. **Add a short-lived LRU cache** in `getAccountIdFromAlias()` keyed on the base32 alias string to avoid repeated DB hits for the same input.
4. **Apply rate limiting** at the Node.js REST service layer (e.g., `express-rate-limit`) in addition to any infrastructure-level controls.

### Proof of Concept

```bash
# Generate a max-length alias (~8000 chars) and hammer the accounts endpoint
ALIAS=$(python3 -c "print('0.0.' + 'A' * 7990)")

# Single request to verify it reaches the DB (returns 404, not 400 = no length check)
curl -s "http://<mirror-node-rest>/api/v1/accounts/$ALIAS"

# Flood with concurrent requests to exhaust DB connection pool
for i in $(seq 1 200); do
  curl -s "http://<mirror-node-rest>/api/v1/accounts/$ALIAS" &
done
wait
# Observe: subsequent legitimate requests time out or return 503
```

**Expected (vulnerable) behavior:** The server accepts the request, runs `base32.decode` on ~8 000 characters, and issues a DB query — returning HTTP 404 (not 400), confirming no length validation occurs.
