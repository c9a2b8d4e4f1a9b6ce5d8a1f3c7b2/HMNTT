### Title
TOCTOU Race in `unsubscribe()` Allows Orphaned Redis Subscriptions via Unprivileged gRPC Clients

### Summary
`RedisTopicListener.unsubscribe()` calls the single-argument `ConcurrentHashMap.remove(key)` from both `doOnCancel` and `doOnComplete` hooks without any deduplication guard. A concurrent `computeIfAbsent` in `getSharedListener` can insert a new `Flux` entry between the two `remove` calls, causing the second `remove` to silently delete the freshly-created subscription. This leaves an active Redis pub/sub subscription with no map entry, making it permanently unreachable and uncleanable — an orphaned subscription. Any unprivileged gRPC client can trigger this by subscribing, cancelling, and immediately re-subscribing to the same topic.

### Finding Description

**Exact code path:**

`getSharedListener` (line 61) uses `computeIfAbsent` to atomically insert a new shared `Flux` only if absent: [1](#0-0) 

`subscribe` (lines 74–75) registers **both** `doOnCancel` and `doOnComplete` to call `unsubscribe`, with no deduplication: [2](#0-1) 

`unsubscribe` (line 83) uses the **single-argument** `remove(key)`, which removes whatever value is currently mapped — not necessarily the value that triggered the call: [3](#0-2) 

**Root cause:** The failed assumption is that `remove(key)` will only ever remove the `Flux` instance that registered the `unsubscribe` callback. In reality, a concurrent `computeIfAbsent` can insert `Flux_2` into the map between the moment `Flux_1`'s `unsubscribe` is triggered and when `remove` executes, causing `Flux_2` to be deleted.

**Exploit flow (TOCTOU race):**

```
T=0  Map: {"topic.X": Flux_1}
T=1  Flux_1: all subscribers cancel → doOnCancel fires → unsubscribe() queued
T=2  Thread B: computeIfAbsent("topic.X") → key absent → creates Flux_2 → inserts
     Map: {"topic.X": Flux_2}
T=3  Thread A: remove("topic.X") executes → removes Flux_2 (not Flux_1!)
     Map: {}
T=4  Flux_2 is now active (subscribed to Redis channel) but absent from map → orphaned
T=5  Thread C: computeIfAbsent("topic.X") → creates Flux_3 → inserts
     Map: {"topic.X": Flux_3}  ← Flux_2 still consuming Redis resources silently
```

A second trigger path: if a Redis channel completion signal (`doOnComplete`) and a subscriber cancel (`doOnCancel`) race concurrently for the same topic, both call `unsubscribe()` in rapid succession. The first `remove` clears `Flux_1`; a new subscriber's `computeIfAbsent` inserts `Flux_2`; the second `remove` deletes `Flux_2`.

**Why existing checks are insufficient:**

`ConcurrentHashMap` guarantees individual operation atomicity, but `computeIfAbsent` and `remove` are **not** composed atomically. There is no `AtomicBoolean` guard preventing double-invocation of `unsubscribe`, and the single-argument `remove(key)` does not verify it is removing the intended `Flux` instance. [4](#0-3) 

### Impact Explanation

Each orphaned `Flux` holds an active `ReactiveRedisMessageListenerContainer` channel subscription consuming a Redis connection and memory. Subscribers who receive `Flux_2` (before it is removed from the map) silently receive no messages — the shared Flux is cancelled/completed immediately upon subscription, causing a silent delivery failure with no error surfaced to the gRPC client. Repeated exploitation accumulates orphaned subscriptions, exhausting Redis connection pool resources and degrading service for all users (DoS). The `maxConcurrentCallsPerConnection` limit of 5 per connection does not prevent this since each new gRPC connection gets its own quota. [5](#0-4) 

### Likelihood Explanation

Any unauthenticated or unprivileged gRPC client can call `subscribeTopic` with no special permissions. The race window is narrow under normal conditions but widens significantly during a network partition (Redis connection drops cause retries and resubscriptions, increasing concurrent activity on the map). An attacker can deliberately widen the window by subscribing with many concurrent clients to the same topic and cancelling them simultaneously. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` on the container (line 54) means the system continuously attempts reconnection, creating repeated opportunities to trigger the race. [6](#0-5) 

### Recommendation

1. **Use the two-argument `remove(key, value)`** in `unsubscribe` so it only removes the entry if it still maps to the exact `Flux` instance that registered the callback:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Flux<TopicMessage>[] self = new Flux[1];
    self[0] = container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> topicMessages.remove(topic.getTopic(), self[0]))
            .doOnComplete(() -> topicMessages.remove(topic.getTopic(), self[0]))
            ...
            .share();
    return self[0];
}
```

2. **Add a deduplication guard** using `AtomicBoolean` to ensure `unsubscribe` executes at most once per subscription lifecycle, preventing the double-remove from concurrent `doOnCancel`/`doOnComplete`:

```java
AtomicBoolean unsubscribed = new AtomicBoolean(false);
.doOnCancel(() -> { if (unsubscribed.compareAndSet(false, true)) unsubscribe(topic); })
.doOnComplete(() -> { if (unsubscribed.compareAndSet(false, true)) unsubscribe(topic); })
```

### Proof of Concept

**Preconditions:** gRPC server running with `listener.type=REDIS`; Redis available; no authentication required on the gRPC endpoint.

**Steps:**

1. Open 10 concurrent gRPC connections, all subscribing to `topic.1` (same topic ID).
2. Simultaneously cancel all 10 subscriptions (send gRPC cancel / close stream).
3. Immediately (within the same millisecond) open a new subscription to `topic.1` from a fresh connection.
4. Repeat steps 2–3 in a tight loop (~100 iterations).

**Expected result (vulnerable):**
- `topicMessages` map becomes empty while active Redis channel subscriptions remain open.
- New subscribers to `topic.1` receive no messages (silent failure).
- Redis `CLIENT LIST` shows accumulating subscriptions to `topic.1` that are never cleaned up.
- Server memory and Redis connection pool degrade over time.

**Verification:** Monitor Redis with `SUBSCRIBE topic.1` from a separate client and confirm duplicate subscription registrations appear; confirm gRPC subscribers receive zero messages despite active Redis publishing.
