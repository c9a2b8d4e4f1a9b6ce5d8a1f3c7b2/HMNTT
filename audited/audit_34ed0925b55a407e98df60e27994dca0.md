### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables DB Sort-Memory Exhaustion via `DISTINCT ON (key)` Query

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` passes user-controlled timestamp bounds directly to `findByKeyBetweenAndTimestampBetween()` without enforcing any maximum range width. A single unauthenticated request with `timestamp=gte:0` sets `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, causing PostgreSQL to scan every row for the given `(owner_id, hook_id)` and perform an in-memory sort to satisfy `DISTINCT ON (key) ORDER BY key ASC, consensus_timestamp DESC`. Because the sort order (`consensus_timestamp DESC`) conflicts with the index order (`consensus_timestamp ASC`), the planner cannot use an index-only skip scan and must materialize and sort the full result set. Repeated concurrent requests exhaust PostgreSQL `work_mem` and shared sort buffers, degrading or crashing database-backed processing nodes.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` → `HookServiceImpl.getHookStorage()` → `getHookStorageChange()` → `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()`

**Root cause – no timestamp range width guard:**

In `Bound.java` lines 63–97, when only a lower-bound operator is supplied (e.g. `gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE` and `getAdjustedLowerRangeValue()` returns `0`: [1](#0-0) 

These values flow unchanged into the service: [2](#0-1) 

The native query then executes:

```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id  = :hookId
  and key >= :keyLowerBound          -- defaults to 0x00…00
  and key <= :keyUpperBound          -- defaults to 0xFF…FF
  and consensus_timestamp between 0 and 9223372036854775807
``` [3](#0-2) 

**Why the index cannot rescue this query:**

The table's primary key is `(owner_id, hook_id, key, consensus_timestamp)` (all ascending). [4](#0-3) 

`HookStorageRequest.getPageRequest()` builds `Sort.by(key ASC, consensus_timestamp DESC)` for the historical path: [5](#0-4) 

`DISTINCT ON (key)` requires the result to be ordered by `key ASC, consensus_timestamp DESC`. The index stores `consensus_timestamp` ascending, so PostgreSQL cannot use an index-skip scan for the `DESC` secondary sort. It must fetch all rows matching the `(owner_id, hook_id)` prefix, sort them entirely in memory (or spill to disk), and only then apply `DISTINCT ON` and the `LIMIT 100`.

**No rate limiting on this endpoint:**

The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module and is not wired into `HooksController`: [6](#0-5) 

### Impact Explanation
Every request with `timestamp=gte:0` forces PostgreSQL to sort the entire `hook_storage_change` partition for the targeted `(owner_id, hook_id)`. As the table grows (it accumulates every historical storage write), sort size grows proportionally. Concurrent requests from multiple attacker threads multiply memory pressure. When PostgreSQL's `work_mem` is exhausted, sorts spill to disk, saturating I/O; if the OOM killer or connection exhaustion is reached, database-backed mirror-node instances crash. Because all mirror-node replicas share the same PostgreSQL cluster (or Citus distributed tables), a sustained flood can degrade ≥30% of processing nodes.

### Likelihood Explanation
- **No authentication required** – the endpoint is a public read API.
- **Discovery is trivial** – valid `ownerId` values are enumerable via `/api/v1/accounts`; valid `hookId` values via `/api/v1/accounts/{id}/hooks`.
- **Single parameter triggers the worst case** – `?timestamp=gte:0` is sufficient; no special knowledge needed.
- **No rate limiting** – requests can be issued at full network speed.
- **Repeatable and automatable** – a simple loop or tool like `ab`/`wrk` suffices.

### Recommendation
1. **Enforce a maximum timestamp range width** in `getHookStorageChange()` (analogous to `bindTimestampRange` in the Node.js `rest` module): reject or clamp requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Add rate limiting** to `HooksController` using the existing `Bucket4j` infrastructure already present in the codebase.
3. **Add a partial index or covering index** on `hook_storage_change (owner_id, hook_id, key ASC, consensus_timestamp DESC)` so PostgreSQL can satisfy the `DISTINCT ON` sort without a full in-memory sort.
4. **Require at least one bounded (non-open-ended) timestamp parameter** when the historical path is used, preventing `[0, MAX_VALUE]` ranges.

### Proof of Concept

```bash
# Step 1: discover a valid ownerId
curl "https://<mirror-node>/api/v1/accounts?limit=1" | jq '.accounts[0].account'
# e.g. "0.0.1234"

# Step 2: discover a valid hookId for that owner
curl "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks?limit=1" | jq '.hooks[0].hook_id'
# e.g. 1

# Step 3: trigger the unbounded historical query (single request)
curl "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0"

# Step 4: amplify with concurrent requests to exhaust DB sort memory
ab -n 10000 -c 200 \
  "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0"
```

Each request causes PostgreSQL to execute:
```sql
SELECT DISTINCT ON (key) ... FROM hook_storage_change
WHERE owner_id=1234 AND hook_id=1
  AND key BETWEEN '\x00…00' AND '\xff…ff'
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
ORDER BY key ASC, consensus_timestamp DESC
LIMIT 100;
```
With a large `hook_storage_change` table, this sort exhausts `work_mem` per connection, and 200 concurrent connections multiply the pressure, degrading or crashing the database tier.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public RangeParameter<Long> adjustLowerRange() {
        if (this.hasEqualBounds()) {
            // If the primary param has a range with a single value, rewrite it to EQ
            lower = new NumberRangeParameter(EQ, this.getAdjustedLowerRangeValue());
            upper = null;
        }

        return lower;
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
    }
