### Title
Unauthenticated Health Endpoint Relays Uncached Live HTTP Calls to Upstream REST API (Proxy/Relay Abuse)

### Summary
The `health()` method in `SubscriberHealthIndicator` unconditionally issues a live HTTP GET to the upstream REST API (`/network/stake`) on every invocation with no caching, no rate limiting, and no authentication on the health endpoint itself. An unprivileged external attacker can repeatedly poll the Spring Boot Actuator health endpoint to force the monitor to relay arbitrary volumes of traffic to the upstream REST API, with the monitor's IP appearing as the source — effectively bypassing any IP-based rate limiting on the upstream.

### Finding Description
**Code path:**

- `SubscriberHealthIndicator.java`, `health()`, line 60: calls `restNetworkStakeHealth()`
- `restNetworkStakeHealth()`, lines 93–121: calls `restApiClient.getNetworkStakeStatusCode()` with no caching
- `RestApiClient.java`, line 68: issues a live `GET /network/stake` HTTP request on every call

```java
// SubscriberHealthIndicator.java:59-63
public Mono<Health> health() {
    return restNetworkStakeHealth()   // live HTTP call every time
        .flatMap(health -> ...)
        .doOnNext(this::recordHealthMetric);
}

// RestApiClient.java:67-69
public Mono<HttpStatusCode> getNetworkStakeStatusCode() {
    return webClientRestJava.get().uri("/network/stake").exchangeToMono(r -> Mono.just(r.statusCode()));
}
```

**Root cause:** There is no result caching (no `@Cacheable`, no TTL-based memoization, no shared `Mono` replay), and no rate limiting on the health endpoint. Spring Boot Actuator's `/actuator/health` is publicly exposed by default — no `SecurityConfig` was found in the monitor module, and no `application.yml`/`application.properties` with actuator security settings exists under `monitor/src/main/resources/`.

**Why existing checks are insufficient:** The only protection is a 5-second timeout (`Duration.ofSeconds(5)`) on line 107, which limits per-request duration but does nothing to limit request frequency or volume.

### Impact Explanation
Every unauthenticated HTTP request to `/actuator/health` causes the monitor to issue one `GET /network/stake` request to the upstream REST API from the monitor's own IP address. This has two concrete effects:

1. **IP laundering / rate-limit bypass:** The upstream REST API sees requests originating from the monitor's trusted IP, not the attacker's. Any IP-based rate limiting or allowlisting on the upstream is bypassed.
2. **Load amplification on upstream:** An attacker with a botnet or high-frequency polling loop can sustain a high request rate against the upstream REST API at no cost beyond sending cheap HTTP requests to the health endpoint.

Severity: **Medium** — the amplification ratio is 1:1 (not multiplicative), but the IP-laundering aspect and the ability to exhaust upstream rate limits or cause load are real.

### Likelihood Explanation
- **Precondition:** The attacker needs network access to the monitor's health endpoint — typical in any internet-exposed or internal-network deployment.
- **No credentials required:** Spring Boot Actuator health endpoints are unauthenticated by default.
- **Trivially repeatable:** A simple `while true; do curl http://<monitor>/actuator/health; done` loop is sufficient.
- **No special knowledge needed:** The `/actuator/health` path is standard and well-known.

### Recommendation
1. **Cache the upstream health result** with a short TTL (e.g., 10–30 seconds) using a `Mono.cache(Duration)` or Spring's `@Cacheable` to decouple health-check frequency from upstream call frequency.
2. **Restrict access to the health endpoint** via Spring Security or actuator configuration:
   ```yaml
   management:
     endpoints:
       web:
         exposure:
           include: health
     endpoint:
       health:
         show-details: when-authorized
   ```
3. **Apply rate limiting** at the reverse proxy/ingress layer for `/actuator/health`.

### Proof of Concept
```bash
# Attacker with no credentials repeatedly polls the health endpoint
# Each request causes the monitor to call GET /network/stake on the upstream REST API

for i in $(seq 1 1000); do
  curl -s http://<monitor-host>:<port>/actuator/health > /dev/null &
done
wait

# Result: ~1000 GET /network/stake requests arrive at the upstream REST API
# from the monitor's IP, bypassing any IP-based rate limiting on the upstream.
```
