### Title
GraphQL Module Lacks Application-Level Rate Limiting, Enabling Unauthenticated Database Connection Pool Exhaustion

### Summary
The `graphQlCustomizer()` function in `GraphQlConfiguration.java` installs only query complexity (200) and depth (10) instrumentation, with no application-level rate limiting or concurrency controls. An unauthenticated attacker sending many concurrent valid queries from multiple source IPs can exhaust the HikariCP connection pool, partitioning the application from the database. The infrastructure-level Traefik `inFlightReq` guard is per-IP and bypassable, and the co-configured `retry` middleware amplifies the attack.

### Finding Description

**Exact code location:**

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `graphQlCustomizer()`, lines 42–48:

```java
@Bean
GraphQlSourceBuilderCustomizer graphQlCustomizer(PreparsedDocumentProvider provider) {
    var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
    var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
    var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);
    return b -> b.configureGraphQl(
            graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
}
```

No rate-limiting bean, no concurrency semaphore, and no per-client throttle is registered here or anywhere else in the `graphql` module. Compare with the `web3` module, which has a full `ThrottleConfiguration` using bucket4j (`rateLimitBucket`, `gasLimitBucket`) enforced at the controller layer.

**Database connection pool:**

`CommonConfiguration.java` (lines 61–94) creates a `HikariDataSource` from `spring.datasource.hikari` properties. No explicit `maximumPoolSize` is set in any graphql-specific config file, so HikariCP defaults to **10 connections**. The `statementTimeout` is 10,000 ms (10 seconds) per the docs, meaning each held connection blocks for up to 10 seconds.

**Infrastructure guard is insufficient:**

`charts/hedera-mirror-graphql/values.yaml` lines 135–145 configures Traefik middleware:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1
  - retry:
      attempts: 3
      initialInterval: 100ms
```

- `inFlightReq: amount: 5` is **per source IP**. An attacker using N IPs can sustain 5×N concurrent in-flight requests.
- The `retry: attempts: 3` middleware **amplifies** each request up to 3× before the circuit breaker triggers, worsening pool pressure.
- No `rateLimit` directive exists (unlike the rosetta chart which adds `rateLimit: average: 10`).
- This middleware is only active when Traefik is the ingress; the GCP Gateway path (`gateway.gcp.enabled: true`) has `maxRatePerEndpoint: 250` but that is a backend-level rate, not a per-client concurrency limit.

**Exploit flow:**

1. Attacker crafts a valid `account` query with complexity ≤ 199 (e.g., requesting all scalar fields of an account).
2. Each query hits `AccountController.account()` → `EntityServiceImpl.getByIdAndType()` → `EntityRepository.findById()` — one synchronous JDBC call holding one HikariCP connection for the duration of the DB round-trip.
3. Attacker sends these queries concurrently from ≥2 source IPs (bypassing the per-IP `inFlightReq: 5` cap), or from a single IP if Traefik is not in the path (e.g., direct pod access, GCP Gateway path, or misconfigured ingress).
4. With 10+ concurrent in-flight queries, all HikariCP connections are held. New requests queue at `connectionTimeout` (default 30 seconds in HikariCP when not overridden), then fail with `SQLTransientConnectionException`.
5. The application cannot serve any GraphQL or health-check DB-dependent requests, effectively partitioned from the database.

### Impact Explanation

Complete database connection pool exhaustion causes all subsequent GraphQL queries to fail with connection timeout errors. The application is functionally unavailable (network partition from DB) for the duration of the attack. Because the endpoint is unauthenticated and public (`/graphql/alpha`), there is no credential barrier. The `statementTimeout` of 10 seconds means each attack wave holds connections for up to 10 seconds, requiring only a sustained low-volume flood to maintain exhaustion.

### Likelihood Explanation

The endpoint requires zero authentication. The schema is public (introspection is not disabled). A single attacker with two or more IP addresses (trivially obtained via cloud VMs, proxies, or a small botnet) bypasses the only per-client guard. The attack requires no special knowledge beyond the public GraphQL schema. It is repeatable and automatable with standard HTTP tooling (e.g., `curl`, `wrk`, `hey`). The retry middleware means even a modest request rate is amplified 3× at the infrastructure layer before the circuit breaker engages.

### Recommendation

1. **Add application-level rate limiting to the GraphQL module** using bucket4j (already a dependency in `web3`), mirroring `ThrottleConfiguration` and `ThrottleManagerImpl`. Apply a global requests-per-second cap and a per-IP cap via a `WebFilter` or `HandlerInterceptor` before query execution.
2. **Add a `rateLimit` Traefik middleware** (as done in the rosetta chart) to the graphql middleware chain, e.g., `rateLimit: average: 10, sourceCriterion: requestHost: true`.
3. **Remove or disable the `retry` middleware** for the GraphQL path, or ensure it only retries on network errors, not on application-level failures that indicate overload.
4. **Explicitly configure `spring.datasource.hikari.maximum-pool-size`** to a value appropriate for the expected concurrency, and set `connection-timeout` to a short value (e.g., 2–3 seconds) so exhausted-pool errors fail fast rather than queuing.
5. **Disable GraphQL introspection** in production to prevent schema enumeration.

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)

                .build();
