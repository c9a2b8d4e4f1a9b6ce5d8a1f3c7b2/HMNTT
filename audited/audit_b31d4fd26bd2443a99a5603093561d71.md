### Title
Missing File ID Whitelist Validation in REST Java Network Nodes Endpoint Allows Empty Node List Response

### Summary
The REST Java `/api/v1/network/nodes` endpoint accepts an arbitrary `file.id` query parameter without validating it against the only two valid address book file IDs (101 and 102). When a non-existent file ID is supplied, the `latest_address_book` CTE in the repository query returns zero rows, and the subsequent INNER JOIN produces an empty result set. Any unauthenticated caller can trigger this, causing clients to receive an empty node list and potentially lose the ability to discover network nodes.

### Finding Description

**Code path:**

1. `NetworkController.getNodes()` ( [1](#0-0) ) only checks that the operator is `EQ`; it does **not** validate that the file ID value is one of the two valid address book files (101 or 102).

2. `NetworkServiceImpl.getAddressBookFileId()` ( [2](#0-1) ) blindly returns `request.getFileId().value()` with no whitelist check.

3. That raw Long is passed directly to `NetworkNodeRepository.findNetworkNodes()` as `:fileId`. The CTE `latest_address_book` filters `where file_id = :fileId` ( [3](#0-2) ). If no row matches, the CTE is empty.

4. The main query uses an INNER JOIN: `join latest_address_book ab on ab.start_consensus_timestamp = abe.consensus_timestamp` ( [4](#0-3) ). An empty CTE eliminates all rows, returning an empty list.

**Root cause / failed assumption:** The developer assumed that only valid file IDs would be supplied, or that the operator-type check was sufficient. The gRPC counterpart (`grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`) correctly enforces a whitelist: [5](#0-4)  — the REST Java service has no equivalent guard.

**`EntityIdRangeParameter.getEntityId()`** accepts any non-negative numeric entity ID without restriction ( [6](#0-5) ), so any value like `99999999` parses successfully.

### Impact Explanation
Any client (SDK, wallet, explorer, node operator tooling) that calls `GET /api/v1/network/nodes` to bootstrap its peer list will receive `{"nodes": [], "links": {}}`. If the client treats an empty list as authoritative, it cannot form connections to consensus nodes, effectively causing a local network shutdown for that client. At scale — e.g., if a widely-used SDK or monitoring tool polls this endpoint — the impact extends to all users of that software. The attack requires zero authentication and zero privileges.

### Likelihood Explanation
The endpoint is publicly documented in the OpenAPI spec ( [7](#0-6) ) and is unauthenticated. The exploit is a single HTTP GET with one crafted query parameter. It is trivially repeatable, scriptable, and requires no special knowledge beyond reading the API docs. Any external party can execute it continuously to keep the response empty for all callers who hit the same mirror node instance.

### Recommendation
Add a whitelist check in `NetworkServiceImpl.getNetworkNodes()` (or in `getAddressBookFileId()`) mirroring the guard already present in the gRPC service:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    Set<Long> valid = Set.of(
        systemEntity.addressBookFile101().getId(),
        systemEntity.addressBookFile102().getId()
    );
    if (!valid.contains(id)) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

This matches the existing gRPC-layer defense and ensures the SQL CTE always receives a file ID that exists in the `address_book` table.

### Proof of Concept

```
# No authentication required
GET /api/v1/network/nodes?file.id=99999999

# Expected (legitimate) response:
# {"nodes": [...actual nodes...], "links": {...}}

# Actual response with crafted file.id:
# {"nodes": [], "links": {}}
```

Steps:
1. Identify a public mirror node REST endpoint (e.g., `https://mainnet-public.mirrornode.hedera.com`).
2. Send: `curl "https://<mirror-node>/api/v1/network/nodes?file.id=99999999"`
3. Observe: response contains an empty `nodes` array.
4. Any SDK or client bootstrapping from this response will have no peers to connect to.

    private static EntityId getEntityId(String entityId) {

                .splitToStream(Objects.requireNonNullElse(entityId, ""))
                .map(Long::valueOf)
                .filter(n -> n >= 0)
                .toList();

        if (parts.size() != StringUtils.countMatches(entityId, ".") + 1) {
            throw new IllegalArgumentException("Invalid entity ID");

        return switch (parts.size()) {

            case 3 -> EntityId.of(parts.get(0), parts.get(1), parts.get(2));
