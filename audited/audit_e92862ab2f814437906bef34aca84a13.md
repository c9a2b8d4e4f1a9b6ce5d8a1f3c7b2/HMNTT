### Title
Unauthenticated Mirror REST Response with Empty `nodes` Array Causes Non-Retryable Pinger Shutdown

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` unconditionally trusts the HTTP response from the configured mirror REST endpoint. An attacker who controls that endpoint (or can MITM a plaintext HTTP connection to it) can return a valid 200 response with an empty `nodes` array, causing the function to return a non-retryable error. This immediately prevents the pinger from initializing its Hiero SDK client, halting all transaction submission.

### Finding Description
**Code path:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–131; called from `buildNetworkFromMirrorNodes()` lines 36–72; called from `newClient()` in `pinger/sdk_client.go` lines 17–22.

**Root cause:** After decoding the JSON body, the function iterates over `payload.Nodes` to build the `network` map. If the attacker returns `{"nodes":[],"links":{"next":null}}` with HTTP 200, the loop at lines 102–125 never executes, leaving `network` empty. The guard at lines 127–129 then fires:

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

The critical detail is the second return value: `false` (no retry). Back in `buildNetworkFromMirrorNodes()` at lines 58–61:

```go
lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
if !retry || attempt == attempts {
    break
}
```

`retry=false` causes an immediate `break` on the very first attempt, bypassing all configured retries (`mirrorNodeClientMaxRetries`, default 10). `newClient()` then returns this error, and the pinger process cannot start or recover.

**Failed assumption:** The code assumes the mirror REST endpoint is trusted and that an empty-nodes response is a definitive, permanent condition rather than a potentially adversarial or transient one.

### Impact Explanation
The pinger is the component that continuously submits transactions to verify the Hiero network is live. If `newClient()` fails, no SDK client is created and no transactions are submitted. From an operational standpoint this is a complete pinger shutdown — the monitoring/health-check layer goes dark. In deployments where the pinger's transaction success rate is used to gate alerts or automated responses, this constitutes a denial-of-service against network observability. Severity: **Medium** (DoS on the pinger service; does not directly affect consensus nodes, but eliminates transaction-confirmation monitoring).

### Likelihood Explanation
**Precondition:** The attacker must control the mirror REST endpoint or be able to MITM the HTTP connection between the pinger and that endpoint. This applies when:
- `network=other` is configured (the only mode that calls `buildNetworkFromMirrorNodes`).
- The `HIERO_MIRROR_PINGER_REST` URL uses plain `http://` (no TLS), making MITM trivial on the same network segment.
- A mirror node operator is malicious or compromised — mirror node operators are distinct from consensus node operators and have no special trust in the Hiero protocol, making them "unprivileged" with respect to the consensus layer.

The attack is trivially repeatable: serve `{"nodes":[]}` on every request and the pinger never recovers, because the retry loop is bypassed (`retry=false`) and there is no periodic re-initialization after startup failure.

### Recommendation
1. **Treat empty-nodes as retryable:** Change the return at line 128 from `return nil, false, fmt.Errorf(...)` to `return nil, true, fmt.Errorf(...)` so that transient or adversarial empty responses are retried up to the configured limit.
2. **Enforce TLS for the mirror REST endpoint:** Validate that `cfg.mirrorRest` uses `https://` and reject plaintext URLs, or document and enforce TLS in deployment.
3. **Minimum node count:** Add a configurable minimum (e.g., ≥ 1 node with a valid endpoint) and treat responses below that threshold as retryable errors.
4. **Response authentication:** Where possible, verify the mirror REST response against a known-good source (e.g., a pinned certificate or a signed response) to prevent MITM substitution.

### Proof of Concept
1. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>`.
2. On the attacker-controlled host, serve:
   ```
   HTTP/1.1 200 OK
   Content-Type: application/json

   {"nodes":[],"links":{"next":null}}
   ```
3. Start the pinger. `fetchMirrorNodeNetwork()` receives the 200 response, decodes it, finds `len(network)==0`, and returns `(nil, false, error)`.
4. `buildNetworkFromMirrorNodes()` sees `retry=false` on attempt 1, breaks immediately, and returns the error.
5. `newClient()` returns the error; the pinger exits or fails to initialize — zero transactions are ever submitted. [1](#0-0) [2](#0-1) [3](#0-2)
