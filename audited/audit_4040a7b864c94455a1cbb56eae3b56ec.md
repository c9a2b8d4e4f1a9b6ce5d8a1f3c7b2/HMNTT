### Title
Stale Cache in `topicExists()` Allows Subscription to Deleted Topics for Up to 24 Hours

### Summary
`TopicMessageServiceImpl.topicExists()` validates a topic by calling `entityRepository.findById()`, which is backed by a Caffeine cache with a 24-hour TTL. The method only checks that the returned entity has type `TOPIC` and never inspects the `deleted` flag. An unprivileged user can subscribe to a topic that has been deleted, causing the service to stream all historical messages for that deleted topic for up to 24 hours after deletion.

**Root cause — two compounding failures:**

1. `topicExists()` never checks `entity.getDeleted()`. Even a fresh DB query returning an entity with `deleted = true` passes the check.
2. The 24-hour Caffeine cache means that once an entity is cached (e.g., during a prior subscription), all subsequent calls to `findById` for that ID return the cached snapshot — regardless of whether the topic was deleted in the database in the interim.

**Exploit flow:**

- Topic `0.0.X` exists; any subscriber (or the attacker themselves) triggers a `subscribeTopic` call, causing the entity to be cached with `deleted = false` and `type = TOPIC`.
- The topic is subsequently deleted on-chain; the importer sets `deleted = true` in the `entity` table.
- Within the 24-hour cache window, the attacker calls `subscribeTopic` with `topicId = 0.0.X`.
- `topicExists()` calls `entityRepository.findById(X)`, which returns the **cached** entity with `deleted = false`.
- The `.filter(e -> e.getType() == EntityType.TOPIC)` check passes.
- The service proceeds to stream all historical `topic_message` rows for the deleted topic.

Even outside the cache window, the plain `CrudRepository.findById` has no `WHERE deleted IS NOT TRUE` clause, so a freshly-fetched deleted entity also passes the type-only check.

### Impact Explanation
Any unprivileged gRPC client can subscribe to a deleted HCS topic and receive its full message history for up to 24 hours post-deletion. In HCS, topic messages carry application-layer consensus data (e.g., DID documents, supply-chain events, token metadata). Streaming messages from a deleted topic violates the protocol invariant that deletion terminates all access, and can expose sensitive payload data that the topic owner intended to retire. The 24-hour window is operationally significant and not a transient race condition.

### Likelihood Explanation
No special privileges are required. The attacker only needs to know a valid topic ID (publicly observable on-chain) and submit a standard gRPC `subscribeTopic` request. The 24-hour cache window is long enough to be reliably exploited. The attack is repeatable: any deleted topic whose ID was ever looked up during the cache window is vulnerable. The `checkTopicExists` property (line 98) must be `true` (the default) for the guard to even run; if it is `false`, the service bypasses the check entirely and creates a synthetic TOPIC entity, making the issue worse.

### Recommendation
1. **Add a `deleted` check in `topicExists()`:**
   ```java
   .filter(e -> e.getType() == EntityType.TOPIC && !Boolean.TRUE.equals(e.getDeleted()))
   .switchIfEmpty(Mono.error(new EntityNotFoundException(topicId)));
   ```
2. **Use a deletion-aware query in `EntityRepository`:** Replace the plain `findById` with a method that filters out deleted entities at the SQL level (e.g., `findByIdAndDeletedIsNotTrue`), consistent with how other modules handle this (see `rest-java` `EntityRepository` which uses `WHERE deleted <> true`).
3. **Reduce cache TTL** for entity existence checks, or use cache eviction on deletion events, so that a deleted topic is not served from cache for 24 hours.

### Proof of Concept
1. Create HCS topic `0.0.999` and publish several messages to it.
2. Issue a `subscribeTopic` gRPC call for `0.0.999` (or wait for any other subscriber to do so) — this populates the Caffeine cache entry for entity ID `999`.
3. Delete topic `0.0.999` on-chain; confirm the importer sets `entity.deleted = true` in the database.
4. Within 24 hours of step 2, issue a new `subscribeTopic` gRPC call for `0.0.999` as an unprivileged user.
5. Observe that the call succeeds (no `EntityNotFoundException`) and all historical topic messages are streamed, despite the topic being deleted.
