### Title
Unauthenticated Thread-Pool Exhaustion via Synchronous Blocking `findById()` with No Rate Limiting on `/api/v1/topics/{id}`

### Summary
`TopicServiceImpl.findById()` performs a synchronous, blocking JDBC call inside a Spring MVC (Tomcat) request thread. The `TopicController.getTopic()` handler chains three such blocking DB calls per request with no rate limiting applied to the `rest-java` module. An unprivileged attacker flooding this endpoint with concurrent requests can exhaust Tomcat's finite thread pool, causing the service to stop accepting all new requests.

### Finding Description

**Code path:**

`TopicController.getTopic()` (lines 31–37) calls three sequential synchronous service methods on the Tomcat request thread:
```
topicService.findById(id.id())      // → TopicServiceImpl line 20: topicRepository.findById(...)
entityService.findById(id.id())
customFeeService.findById(id.id())
```

`TopicServiceImpl.findById()` (line 19–21) delegates directly to a Spring Data JPA repository backed by HikariCP/JDBC — a fully synchronous, blocking call. The thread is held until the DB responds or the statement timeout fires.

**Root cause — failed assumptions:**

1. The `rest-java` module uses `spring-boot-starter-web` (Tomcat), confirmed in `build.gradle.kts` line 33. Tomcat's default thread pool is 200 threads. Every in-flight HTTP request occupies exactly one thread for its entire duration.
2. No rate-limiting filter exists anywhere in `rest-java/src/**`. The `ThrottleManager`/`ThrottleConfiguration` classes exist only in the `web3` module. The `rest-java` configuration table in `docs/configuration.md` lists no throttle properties.
3. The `statementTimeout` for `rest-java` is **10,000 ms** (10 seconds) per `docs/configuration.md` (`hiero.mirror.restJava.db.statementTimeout`). Each blocked thread can be held for up to 10 seconds before the DB times out.
4. HikariCP's default maximum pool size is 10 connections. Once the pool is exhausted, additional Tomcat threads block waiting for a connection — compounding the exhaustion.
5. No `application.yml` exists under `rest-java/src/main/resources/` (only `banner.txt`), confirming no custom Tomcat thread-pool or connection-pool overrides are in place.

**Exploit flow:**

An attacker sends ≥200 concurrent GET requests to `GET /api/v1/topics/{id}`. Each request occupies a Tomcat thread blocked on up to three JDBC calls. With the 10-second statement timeout, threads are held for up to 10 seconds each. Once all 200 threads are occupied, Tomcat's accept queue fills and new connections are refused or time out. The service becomes completely unavailable for all endpoints, not just the topic endpoint.

**Why existing checks fail:**

The only mitigating mechanism is the 10-second `statementTimeout`, which actually *worsens* the attack surface by guaranteeing threads are held for a full 10 seconds per request under DB pressure. There is no per-IP limit, no global RPS cap, no circuit breaker, and no async offloading for this endpoint.

### Impact Explanation
Complete denial of service for the `rest-java` API. All endpoints — including those used to query transaction status — become unreachable once the thread pool is exhausted. The mirror node's REST Java API is a read path used by wallets, explorers, and dApps to confirm transaction finality; its unavailability breaks downstream transaction confirmation workflows. The Grafana alert `RestJavaHighDBConnections` (threshold 75% of HikariCP max) confirms the operators are aware of connection-pool pressure as a risk vector, but no preventive control exists at the HTTP layer.

### Likelihood Explanation
No authentication or API key is required. The endpoint accepts any numeric topic ID (including non-existent ones, which still complete a full DB round-trip before throwing `EntityNotFoundException`). A single attacker with a modest botnet or even a single machine with a high-concurrency HTTP client (e.g., `wrk`, `hey`, `ab`) can sustain 200+ concurrent connections. The attack is repeatable, stateless, and requires zero knowledge of the system beyond the public API path documented in `rest/api/v1/openapi.yml`.

### Recommendation
1. **Add rate limiting to `rest-java`**: Implement a `HandlerInterceptor` or `OncePerRequestFilter` using Bucket4j (already a dependency in the `web3` module) to cap requests per second per IP and globally on the `/api/v1/topics/**` path.
2. **Reduce `statementTimeout`**: Lower `hiero.mirror.restJava.db.statementTimeout` from 10,000 ms to ≤2,000 ms to bound the maximum thread-hold time.
3. **Increase HikariCP pool size** or configure `connectionTimeout` to fail fast rather than queue indefinitely.
4. **Consider virtual threads**: Spring Boot 3.2+ supports virtual threads (`spring.threads.virtual.enabled=true`), which eliminates the fixed-size thread-pool bottleneck entirely for blocking I/O.
5. **Deploy an ingress-level rate limiter** (e.g., nginx `limit_req`, Traefik rate-limit middleware) as a defense-in-depth layer independent of application code.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk)
# Target: publicly accessible rest-java instance

wrk -t 50 -c 250 -d 60s \
  "http://<rest-java-host>/api/v1/topics/1"

# Expected result:
# - After ~1-2 seconds, Tomcat thread pool (200 threads) is saturated
# - All other API endpoints (e.g., /api/v1/transactions/...) begin returning
#   connection refused or HTTP 503
# - wrk reports near-zero successful responses after pool exhaustion
# - Prometheus metric tomcat_threads_busy_threads reaches tomcat_threads_config_max_threads
# - hikaricp_connections_pending rises to > 0
```

No authentication header is required. Any valid or invalid topic ID (e.g., `0`, `999999999`) triggers the full synchronous DB round-trip.
