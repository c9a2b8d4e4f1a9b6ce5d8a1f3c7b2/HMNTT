### Title
Unauthenticated Three-Way UNION ALL with Repeated Entity JOIN Enables Sustained DB CPU Exhaustion on GET /accounts/:id/nfts

### Summary
An unprivileged external user can supply `token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` on the public `GET /api/v1/accounts/:idOrAliasOrEvmAddress/nfts` endpoint. This combination passes all validation checks and causes `extractNftMultiUnionQuery()` to populate all three of `lower`, `inner`, and `upper` filter arrays, which `NftService.getQuery()` then wraps into a three-branch `UNION ALL` SQL query. Every branch independently executes `NftService.nftQuery`, which contains a `LEFT JOIN entity` clause, meaning three separate join operations are issued per HTTP request. With no rate limiting present in the REST API layer, a moderate stream of such requests produces sustained elevated DB CPU consumption.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 15 registers the public route with no authentication or rate-limiting middleware: [1](#0-0) 

`AccountController.getNftsByAccountId` (line 90–103) calls `extractNftMultiUnionQuery` unconditionally: [2](#0-1) 

`extractNftMultiUnionQuery` (lines 34–82) parses the four query parameters into `bounds.primary` (token.id) and `bounds.secondary` (serialnumber), then calls `getLowerFilters`, `getInnerFilters`, and `getUpperFilters`: [3](#0-2) 

**Validation checks and why they pass:**

`validateBoundsRange` rejects only when a bound has *both* range and equal — not triggered here.
`validateLowerBounds` rejects only when secondary has a lower bound but primary uses `gt` (not `gte`) — not triggered because the attacker uses `gte`.
`validateUpperBounds` rejects only when secondary has an upper bound but primary uses `lt` (not `lte`) — not triggered because the attacker uses `lte`. [4](#0-3) 

With `token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B`, all three filter arrays are non-empty:
- `lower` = `[{token_id = X}, {serial_number >= A}]`
- `inner` = `[{token_id > X}, {token_id < Y}]`
- `upper` = `[{token_id = Y}, {serial_number <= B}]` [5](#0-4) 

**Query generation:**

`NftService.getQuery` (lines 93–122) maps each non-empty filter array to a `getSubQuery` call sharing the same `params` array, then joins them with `UNION ALL`: [6](#0-5) 

Every `getSubQuery` call uses `NftService.nftQuery`, which contains a `LEFT JOIN entity e ON e.id = nft.token_id`: [7](#0-6) 

This produces a final SQL of the form:
```sql
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id=$1 AND token_id=$3 AND serial_number>=$4 ORDER BY ... LIMIT $2)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id=$1 AND token_id>$5 AND token_id<$6 ORDER BY ... LIMIT $2)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
 WHERE account_id=$1 AND token_id=$7 AND serial_number<=$8 ORDER BY ... LIMIT $2)
ORDER BY ... LIMIT $2
```

Three independent join operations per request, each potentially scanning the entity table.

**Absent rate limiting:**

The throttling infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` Java module and operates on `ContractCallRequest` objects. It has no effect on the Node.js REST API: [8](#0-7) 

No equivalent rate-limiting middleware is registered in `extendExpress` or `accountRoute.js` for the `/nfts` endpoint. [9](#0-8) 

### Impact Explanation
Each qualifying request forces the PostgreSQL planner to execute three separate subplans, each performing a `LEFT JOIN` against the `entity` table. The inner subquery (`token_id > X AND token_id < Y`) can span the full token ID space when the attacker supplies `gte:1` and `lte:9007199254740991`, requiring the planner to evaluate a wide index range before the per-subquery `LIMIT` is satisfied. Sustained moderate concurrency (e.g., tens of requests per second from a single client) multiplies this three-fold join cost, driving DB CPU well above the 30% baseline threshold without requiring a volumetric flood. The impact is availability degradation of the mirror node REST API and the shared PostgreSQL instance for all consumers.

### Likelihood Explanation
The exploit requires zero privileges, zero tokens, and zero on-chain state. The four query parameters (`token.id=gte:`, `token.id=lte:`, `serialnumber=gte:`, `serialnumber=lte:`) are documented in the OpenAPI spec as supported combinations: [10](#0-9) 

The attack is trivially scriptable with `curl` or any HTTP client, is fully repeatable, and leaves no authentication trail. The test suite itself confirms the three-way UNION ALL is reachable via this exact parameter combination: [11](#0-10) 

### Recommendation
1. **Add per-IP or global rate limiting** to the REST API layer (e.g., `express-rate-limit`) specifically for the `/accounts/:id/nfts` endpoint, mirroring the throttle logic already present in the web3 module.
2. **Add a DB-level statement timeout** for REST API connections so that any single query exceeding a wall-clock threshold is cancelled.
3. **Consider query cost differentiation**: when both primary and secondary bounds are range-type (triggering the three-branch path), enforce a tighter maximum `limit` or require an explicit opt-in header.
4. **Index review**: ensure a composite index on `(account_id, token_id, serial_number)` exists on the `nft` table so each subquery can satisfy its `LIMIT` via an index scan rather than a sequential scan, reducing per-query cost.

    const upper = this.getUpperFilters(bounds);

      lower,

      upper,

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)

                .refillGreedy(gasLimit, Duration.ofSeconds(1))

                .build();
