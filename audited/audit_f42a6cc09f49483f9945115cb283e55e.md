### Title
Unauthenticated HTTP Network-Map Injection Allows Full Transaction Routing Hijack via MITM

### Summary
`buildNetworkFromMirrorNodes()` fetches the consensus-node address book over a plain, unauthenticated HTTP connection with no TLS and no response-integrity verification. An attacker who can intercept the HTTP path between the pinger and the mirror REST endpoint can inject arbitrary `domain_name`/`ip_address_v4`/`port` values into the JSON response. Because the only acceptance gate is `len(network) > 0`, a mixed (partial) or fully rogue map is accepted equally, and every subsequent restart permanently re-poisons the map.

### Finding Description

**Exact code path**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` lines 36–72 and `fetchMirrorNodeNetwork()` lines 74–132; consumed by `pinger/sdk_client.go` `newClient()` lines 17–22.

**Root cause**

Line 46 creates a bare `http.Client` with no TLS configuration: [1](#0-0) 

The default (and documented example) value for `mirrorRest` is a plain `http://` URL: [2](#0-1) 

`fetchMirrorNodeNetwork()` accepts any HTTP 2xx response and decodes the JSON body without any signature, HMAC, or certificate-pinning check: [3](#0-2) 

The only validation applied to each returned entry is:
- `NodeAccountID` must be non-empty and parseable as a Hiero account ID (lines 103–110)
- `host` must be non-empty and `port` must be non-zero (lines 118–120)
- The resulting map must be non-empty (lines 127–129) [4](#0-3) 

None of these checks verify that the returned IP/hostname belongs to a legitimate consensus node. The resulting map is passed directly to `hiero.ClientForNetwork()` and used for all subsequent transaction submissions: [5](#0-4) 

**Exploit flow — partial then full poison**

1. **First run**: Attacker intercepts the HTTP GET to `/api/v1/network/nodes` and returns a JSON body containing both real node entries (e.g., `0.0.3` → `13.124.142.126:50211`) and one or more rogue entries (e.g., `0.0.4` → `attacker.example.com:50211`). `len(network) > 0` passes; the pinger starts routing a fraction of transactions to the attacker's gRPC listener.

2. **After restart**: Attacker returns a response containing *only* rogue entries. The same `len(network) > 0` guard passes; 100 % of transactions are now routed to attacker-controlled nodes.

Because `buildNetworkFromMirrorNodes()` is called exactly once at startup and the result is never refreshed or cross-validated, each restart is a new opportunity to shift the map further.

### Impact Explanation

Every `CryptoTransfer` (and any other transaction) submitted by the pinger is signed with the operator's private key and sent to whatever gRPC endpoints are in the network map. An attacker controlling those endpoints can:

- **Collect signed transactions** — replay them later or on a different network.
- **Silently drop transactions** — causing the pinger to report false liveness failures.
- **Observe transaction content** — including amounts, account IDs, and memo fields.

Because the operator key is embedded in configuration and used for every tick, full routing hijack is equivalent to a persistent man-in-the-middle on all pinger-originated transactions. Severity: **High**.

### Likelihood Explanation

The precondition is MITM capability on the HTTP path between the pinger container and the mirror REST service. In a typical Kubernetes deployment this is achievable via:

- ARP/NDP spoofing on the pod network
- DNS poisoning of the mirror REST service name
- Compromise of an in-path load balancer or sidecar proxy
- Any network policy misconfiguration that allows pod-to-pod traffic interception

No credentials or privileged access to the pinger or mirror node are required. The attack is repeatable on every restart and requires no persistent foothold beyond the ability to intercept one HTTP connection. Likelihood: **Medium–High** in shared-network deployments.

### Recommendation

1. **Enforce TLS for the mirror REST URL**: Reject any `mirrorRest` value that does not begin with `https://` at config-load time (`pinger/config.go`, `loadConfig()`).

2. **Pin the mirror node's TLS certificate or CA**: Configure the `http.Client` with a custom `tls.Config` that pins the expected root CA or leaf certificate, preventing MITM even if DNS is poisoned.

3. **Validate returned endpoints against an allowlist**: Maintain a configurable set of known-good CIDR ranges or hostnames; reject any `serviceEndpoint` whose resolved address falls outside that set.

4. **Verify response integrity**: If the mirror REST API can sign its `/network/nodes` responses (e.g., with a JWS/JWT), verify the signature before consuming the payload.

5. **Periodic re-validation**: If the network map must be refreshed, compare new maps against the previous one and alert/abort on large deviations rather than silently replacing the entire map.

### Proof of Concept

```
# 1. Stand up a rogue gRPC listener on attacker-controlled host (e.g., 192.0.2.1:50211)
#    that accepts any gRPC connection and logs/drops all received bytes.

# 2. Position attacker between pinger and mirror REST (e.g., via ARP spoof or DNS hijack).

# 3. Intercept the HTTP GET /api/v1/network/nodes request and return:
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "192.0.2.1", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}

# 4. Restart (or start) the pinger with HIERO_MIRROR_PINGER_NETWORK=other
#    and HIERO_MIRROR_PINGER_REST=http://<mirror-rest-host>:5551

# 5. Observe: pinger calls buildNetworkFromMirrorNodes(), receives the poisoned map,
#    len(network)==1 passes the guard, and hiero.ClientForNetwork() is called with
#    {"192.0.2.1:50211": AccountID{0,0,3}}.

# 6. All subsequent CryptoTransfer transactions are sent to 192.0.2.1:50211.
#    The rogue listener logs the signed transaction bytes.

# 7. On next restart, return a map with additional rogue nodes to achieve full coverage.
```
