### Title
Unauthenticated Rate-Limit-Free Endpoint Exhausts Database Connection Pool via Triple-Query Amplification

### Summary
`GET /api/v1/topics/{id}` in `TopicController.getTopic()` accepts requests from any unauthenticated user and issues three synchronous, uncached database queries per request against a shared HikariCP connection pool. The `rest-java` module contains no application-level rate limiting (unlike the `web3` module which has `ThrottleManager`), so a sustained flood of requests with rotating valid entity IDs will exhaust the pool and cause connection-acquisition timeouts for all concurrent users of the service.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` at lines 32–37 calls three separate repository methods sequentially: [1](#0-0) 

Each of those delegates to an uncached Spring Data `CrudRepository.findById()`: [2](#0-1) [3](#0-2) 

No `@Cacheable` annotation exists on any of the three service methods. The `spring-boot-starter-cache` dependency is present but unused for this path. [4](#0-3) 

**Root cause — missing rate limiting:**
The only filters registered in `rest-java` are `LoggingFilter` and `MetricsFilter`. No throttle filter exists: [5](#0-4) 

By contrast, the `web3` module has a full `ThrottleManager` / `ThrottleConfiguration` / `ThrottleManagerImpl` stack with bucket4j rate limiting that is explicitly wired into its controllers: [6](#0-5) [7](#0-6) 

This protection was never ported to `rest-java`.

**Connection pool:**
The shared `HikariDataSource` is created from `CommonConfiguration` with no explicit `maximumPoolSize` override documented for `rest-java`, defaulting to HikariCP's default of 10 connections: [8](#0-7) 

The Grafana dashboard for `rest-java` explicitly tracks `hikaricp_connections_pending`, confirming the pool is the bottleneck resource: [9](#0-8) 

### Impact Explanation
Each request consumes up to 3 HikariCP connections (sequentially, but each held for the duration of its query). With a default pool of 10 and no rate limiting, ~4 concurrent attackers each sending a tight loop of requests can saturate the pool. All legitimate requests then queue waiting for a connection; once the HikariCP `connectionTimeout` (default 30 s) expires they receive a `500` error. This constitutes a full denial of service for the `rest-java` service affecting every endpoint, not just `/topics/{id}`, because the pool is shared.

### Likelihood Explanation
Preconditions are minimal: no account, no API key, no special knowledge beyond knowing valid topic entity IDs (which are sequential integers publicly visible on any Hedera explorer). The attack is trivially repeatable with a single `ab -n 100000 -c 50` command or equivalent. The attacker does not need to be on the same network. The `Cache-Control: public, max-age=5` response header only helps if a CDN or reverse proxy is in front and the attacker reuses the same ID; rotating IDs defeats it entirely.

### Recommendation
1. **Add a rate-limiting filter to `rest-java`** mirroring the `web3` `ThrottleManagerImpl` pattern — implement a `OncePerRequestFilter` backed by bucket4j that enforces a configurable `requestsPerSecond` limit, applied globally to `/api/v1/**`.
2. **Add `@Cacheable`** to `TopicServiceImpl.findById()`, `EntityServiceImpl.findById()`, and `CustomFeeServiceImpl.findById()` with a short TTL (e.g., 5 s matching the existing `Cache-Control` header) to reduce per-request DB pressure.
3. **Set an explicit `spring.datasource.hikari.maximum-pool-size`** in the `rest-java` configuration and document it, so operators understand the concurrency ceiling.
4. **Configure ingress-level rate limiting** (e.g., nginx `limit_req`) as a defence-in-depth layer independent of application code.

### Proof of Concept
```bash
# 1. Enumerate a few valid topic IDs from the public ledger (e.g., 0.0.1234 through 0.0.1334)
# 2. Launch concurrent flood rotating IDs to defeat CDN caching:
seq 1234 1334 | xargs -P 50 -I{} \
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/topics/0.0.{}"

# Repeat in a tight loop:
while true; do
  seq 1234 1334 | xargs -P 50 -I{} \
    curl -s -o /dev/null "https://<mirror-node-host>/api/v1/topics/0.0.{}"
done

# Expected result within seconds:
# - Legitimate requests to /api/v1/topics/{id} and other rest-java endpoints
#   begin returning HTTP 500 (HikariCP connection timeout exhausted)
# - Service remains degraded for all users until flood stops
```

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
