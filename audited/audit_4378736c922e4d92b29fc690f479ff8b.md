### Title
Unauthenticated Full Historical Scan DoS via Unbounded Timestamp in `getHookStorage()`

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint in `HooksController.java` accepts a `timestamp` parameter with no lower-bound enforcement. Supplying `timestamp=lte:9223372036854775807` causes `HookStorageChangeRepository` to execute a `DISTINCT ON (key)` full-table scan over the entire `hook_storage_change` history for the given hook. Because the `rest-java` module has no rate-limiting layer, an unprivileged attacker can flood this endpoint with concurrent requests, holding HikariCP connections for up to the 10-second statement timeout each, exhausting the connection pool and denying service to legitimate callers.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` (lines 104–130) accepts `@RequestParam … TimestampParameter[] timestamps` with only `@Size(max = 2)` — no lower-bound or range-width constraint. [1](#0-0) 

It builds a `HookStorageRequest` and calls `hookService.getHookStorage(request)`. When any timestamp is present, `request.isHistorical()` returns `true` and `getHookStorageChange()` is invoked. [2](#0-1) 

Inside `getHookStorageChange()`, the timestamp bounds are extracted without any minimum-range check: [3](#0-2) 

With `timestamp=lte:9223372036854775807`, `timestampLowerBound` resolves to `0` and `timestampUpperBound` to `Long.MAX_VALUE`. The repository then executes:

```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between 0 and 9223372036854775807
``` [4](#0-3) 

`DISTINCT ON (key)` requires PostgreSQL to read every row matching `(owner_id, hook_id)` across all time partitions before deduplicating — a full historical scan regardless of the `limit` cap.

**Why existing checks fail:**

- `@Max(MAX_LIMIT)` on `limit` (max 100) only restricts rows returned, not rows scanned. [5](#0-4) 
- `@Size(max = 2)` on `timestamps` only limits the number of timestamp parameters, not their range width. [1](#0-0) 
- The `ThrottleManager` / `ThrottleConfiguration` rate-limiting infrastructure exists exclusively in the `web3` module; `HooksController` in `rest-java` has no equivalent. [6](#0-5) 
- The `statementTimeout` for `rest-java` is 10 000 ms (10 seconds), meaning each malicious request holds a HikariCP connection for up to 10 seconds before timing out. [7](#0-6) 

### Impact Explanation
With a default HikariCP pool of ~10 connections and a 10-second statement timeout, an attacker needs only ~10 concurrent requests to saturate the pool. All subsequent requests — including legitimate gossip-prevention hook storage queries — will queue and eventually fail with connection-acquisition timeouts. The `hook_storage_change` table is partitioned by `consensus_timestamp` but the query spans all partitions, amplifying I/O and CPU load on the database. This constitutes a complete denial of service for the hooks storage API with no authentication required.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication, no API key, and no rate limit. The exploit requires only a standard HTTP client and knowledge of any valid `ownerId`/`hookId` pair (or even non-existent ones, since the query still executes). The attack is trivially scriptable, repeatable, and requires no special privileges or on-chain state.

### Recommendation
1. **Add a maximum timestamp range width**: Reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Require a lower-bound timestamp**: Reject historical queries that do not supply an explicit lower-bound (`gte:` or `gt:`) alongside any upper-bound, preventing open-ended scans from `0`.
3. **Add rate limiting to `rest-java`**: Port the `ThrottleManager`/`Bucket4j` pattern from the `web3` module to `rest-java`, applying a per-IP or global RPS limit to the hooks storage endpoint.
4. **Reduce `statementTimeout`**: Lower `hiero.mirror.restJava.db.statementTimeout` for this query class to reduce the connection-hold window per malicious request.

### Proof of Concept

```bash
# Flood with 20 concurrent requests using Long.MAX_VALUE as the upper timestamp bound
for i in $(seq 1 20); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:9223372036854775807&limit=100" &
done
wait

# Legitimate request now times out or receives 503 due to exhausted connection pool
curl -v "http://<mirror-node>/api/v1/accounts/0.0.456/hooks/2/storage"
```

Each of the 20 concurrent requests triggers a full `DISTINCT ON (key)` scan of `hook_storage_change` across all time partitions, holding a database connection for up to 10 seconds. With a pool size of ~10, the pool is exhausted within the first batch, and the legitimate request at the end fails.
