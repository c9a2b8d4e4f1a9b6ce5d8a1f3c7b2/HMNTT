### Title
Deleted Account Staking Reward History Exposed via Missing `deleted` Filter in `isValidAccount()`

### Summary
`EntityService.isValidAccount()` uses `entityExistenceQuery`, which checks only `WHERE id = $1` with no filter on the `deleted` column. As a result, deleted accounts still pass the validity gate in `listStakingRewardsByAccountId`, and their full staking reward transfer history is returned to any unauthenticated caller. This contradicts the behavior of every other entity-lookup query in the same service, which all explicitly exclude deleted entities.

### Finding Description

**Root cause — `entityService.js` lines 28–30 and 60–63:**

`entityExistenceQuery` omits the `deleted` guard:
```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;   // ← no deleted filter
``` [1](#0-0) 

`isValidAccount()` relies solely on this query:
```js
async isValidAccount(accountId) {
    const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
    return !isNil(entity);   // true for deleted accounts
}
``` [2](#0-1) 

**Contrast with sibling queries in the same file**, which correctly exclude deleted entities:
```js
static entityFromAliasQuery = `... where coalesce(${Entity.DELETED}, false) <> true ...`;
static entityFromEvmAddressQuery = `... where ${Entity.DELETED} <> true ...`;
``` [3](#0-2) 

**Exploit path — `accountController.js` lines 170–185:**

1. Caller supplies a numeric account ID (e.g., `0.0.12345`) for a deleted account.
2. `getEncodedId()` for a numeric ID performs only local parsing — no DB lookup, no deleted check.
3. `isValidAccount()` returns `true` because the deleted account still has a row in `entity`.
4. Execution falls through to `StakingRewardTransferService.getRewards()`. [4](#0-3) 

**Data returned — `stakingRewardTransferService.js` lines 11–16:**

The downstream query has no deleted guard either; it returns every row matching `account_id = $1`:
```js
static listStakingRewardsByAccountIdQuery = `
    select srt.account_id, srt.amount, srt.consensus_timestamp
    from staking_reward_transfer srt
    where srt.account_id = $1`;
``` [5](#0-4) 

### Impact Explanation
An unauthenticated attacker can retrieve the complete staking reward history (account ID, reward amount in tinybars, consensus timestamp) for any deleted account that has rows in `staking_reward_transfer`. This leaks historical financial data that the account deletion was intended to make inaccessible. The severity is **Medium**: the data is read-only and limited to staking reward records, but it violates the expected access-control boundary for deleted accounts and requires zero privileges to exploit.

### Likelihood Explanation
Exploitation requires no authentication, no special tooling, and no elevated privileges. Deleted account IDs are publicly observable from transaction history on the mirror node itself. An attacker simply iterates known account IDs, and any deleted account with staking history will return data instead of a 404. The attack is trivially repeatable and automatable.

### Recommendation
Add a `deleted` filter to `entityExistenceQuery`, consistent with the other queries in `entityService.js`:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1
                                 and coalesce(${Entity.DELETED}, false) <> true`;
``` [1](#0-0) 

This single change makes `isValidAccount()` return `false` for deleted accounts, causing `listStakingRewardsByAccountId` to throw `NotFoundError` before any reward data is queried.

### Proof of Concept

**Preconditions:**
- Account `0.0.12345` exists in the `entity` table with `deleted = true`.
- At least one row exists in `staking_reward_transfer` with `account_id = 12345`.

**Steps:**
```
GET /api/v1/accounts/0.0.12345/rewards
```

**Expected result (correct behavior):** `404 Not Found`

**Actual result (vulnerable behavior):** `200 OK` with a JSON body containing the staking reward history:
```json
{
  "rewards": [
    { "account_id": "0.0.12345", "amount": 500000000, "timestamp": "1680000000.000000000" }
  ],
  "links": { "next": null }
}
```
