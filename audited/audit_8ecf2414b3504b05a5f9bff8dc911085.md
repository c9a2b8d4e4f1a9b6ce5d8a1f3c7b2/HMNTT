### Title
Unauthenticated Request Flooding Exhausts DB Connection Pool via Uncached, Unthrottled `getByIdAndType()` in GraphQL Service

### Summary
`EntityServiceImpl.getByIdAndType()` issues a direct, uncached database query on every invocation with no per-request rate limiting in the GraphQL module. During a network partition, each in-flight query holds a HikariCP connection for the full `statementTimeout` duration (default 10 s). An unprivileged attacker flooding sequential `entityId` values can exhaust the connection pool, causing all subsequent legitimate requests to fail with connection-acquisition timeouts.

### Finding Description
**Code path:**

`AccountController.account()` → `EntityService.getByIdAndType()` → `EntityRepository.findById()` → SQL `SELECT * FROM entity WHERE id = ?` [1](#0-0) 

`EntityRepository` in the graphql module extends `CrudRepository<Entity, Long>` with no `@Cacheable` annotation on `findById()`: [2](#0-1) 

Compare with the grpc module's `EntityRepository`, which does annotate `findById()` with `@Cacheable`: [3](#0-2) 

The graphql module has no `ThrottleConfiguration`, no bucket4j rate-limit bean, and no per-IP or per-user request filter. The only protections configured are query-complexity/depth limits and JSON/parser size limits: [4](#0-3) 

These controls bound the *size* of a single query, not the *rate* of requests. The `ThrottleConfiguration` and `ThrottleManagerImpl` that do implement rate limiting exist only in the `web3` module: [5](#0-4) 

**Root cause:** The graphql module assumes an upstream load balancer enforces rate limits (the design doc for subscriptions states "We should implement rate limiting in the load balancer"), but no such enforcement is guaranteed for the query path, and no application-level guard exists.

**Exploit flow:**
1. Attacker opens many concurrent HTTP/2 connections to the GraphQL endpoint (no authentication required).
2. Each connection sends `{ account(input: { entityId: { num: N } }) { balance } }` with a distinct, sequential `N`.
3. Each request reaches `getByIdAndType()`, which calls `entityRepository.findById(N)` — a unique key each time, so no cache (even if one existed) would help.
4. During a network partition, the DB TCP connection is established but queries stall; each query holds a HikariCP connection for up to `statementTimeout` (10 000 ms default).
5. With HikariCP's default pool size of 10, only 10 concurrent stalled queries are needed to exhaust the pool.
6. All subsequent legitimate requests block on connection acquisition and eventually fail.

### Impact Explanation
Complete denial of service for the GraphQL API during a network partition. Legitimate users receive connection-pool-exhaustion errors. Because the `statementTimeout` is 10 s and the pool is small, even a modest attack rate (≥ 10 concurrent requests) is sufficient. The impact is limited to the GraphQL service; other modules (rest, web3) are unaffected.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The attacker needs only an HTTP client capable of concurrent requests — no special knowledge, credentials, or exploit tooling required. Sequential `EntityId` values (integers starting from 1) are trivially enumerable. The attack is repeatable and can be sustained indefinitely. The only external mitigation is a load balancer, which may not be present in all deployments (the service is documented as an alpha API).

### Recommendation
1. **Add application-level rate limiting** to the graphql module, mirroring the `ThrottleConfiguration` / bucket4j pattern already used in `web3`.
2. **Add `@Cacheable`** to `EntityRepository.findById()` in the graphql module, consistent with the grpc module's implementation, to reduce DB round-trips for repeated lookups.
3. **Set a short HikariCP `connectionTimeout`** and a **`maximumPoolSize`** appropriate for expected concurrency, so pool exhaustion fails fast rather than queuing indefinitely.
4. **Enforce a server-side concurrency limit** (e.g., Spring WebFlux backpressure or a Semaphore) so that the number of in-flight DB queries is bounded independently of the pool size.

### Proof of Concept
```bash
# Requires: curl with HTTP/2, GNU parallel or similar
# Send 50 concurrent requests with sequential entity IDs
seq 1 50 | parallel -j50 \
  'curl -s -X POST http://<host>:8083/graphql/alpha \
    -H "Content-Type: application/json" \
    -d "{\"query\": \"{account(input: {entityId: {shard:0,realm:0,num:{}}}) {balance}}\"}" &'

# During a network partition (or simulated with `tc netem delay 15000ms`):
# All 50 requests stall holding HikariCP connections.
# A subsequent legitimate request will receive a HikariCP connection-acquisition
# timeout error (default 30 s wait, then exception), confirming pool exhaustion.
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
