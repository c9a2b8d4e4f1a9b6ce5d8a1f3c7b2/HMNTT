### Title
Genesis File Fallback Cached Without Distinction, Enabling Cache Pollution and Stale Data Serving

### Summary
`loadWithRetry` unconditionally returns `systemFile.genesisFile()` (a non-null `File`) when no DB entry exists at or before the requested timestamp. The caller `load()` caches every non-null result, so the genesis fallback is written into the Caffeine cache under the attacker-controlled `CacheKey(fileId, timestamp)`. Because the `systemFile` and `exchangeRate` caches each have `maximumSize=20`, an unprivileged user can fill both caches entirely with genesis-file entries using only 20 crafted requests per cache, evicting legitimate DB-loaded entries and, for the hourly-bucketed files, causing wrong data to be served to all users within the same hour bucket for up to 60 minutes.

### Finding Description

**Exact code path:**

`load()` (lines 103–140) computes a `CacheKey`, checks the cache, and on a miss calls `loadFromDB` → `loadWithRetry`. `loadWithRetry` (lines 164–196) ends with:

```java
.orElse(systemFile.genesisFile())   // line 192 – always non-null
```

and on `RetryException`:

```java
return systemFile.genesisFile();    // line 194 – always non-null
```

Back in `load()`:

```java
var result = loadFromDB(fileId, consensusTimestamp);   // line 133
if (result != null) {                                  // line 134 – always true
    cache.put(cacheKey, result);                       // line 136 – genesis file cached
}
```

The `null` guard was intended to skip caching for unknown file IDs (handled by `loadFromDB` returning `null` at line 145 when `systemFile == null`). For known system files, `loadWithRetry` never returns `null`, so the genesis fallback is always cached.

**Trigger condition:**

`getFileAtTimestamp` (web3 `FileDataRepository`, lines 14–33) returns `Optional.empty()` when no `FILECREATE` (type 17) or non-empty `FILEUPDATE` (type 19) row exists with `consensus_timestamp <= ?2`. This is confirmed by the repository test:

```java
assertThat(fileDataRepository.getFileAtTimestamp(exchangeRateFileId.getId(), 199))
    .isEmpty();   // timestamp before first DB entry → empty
```

An attacker supplies any timestamp smaller than the earliest DB entry for the target file.

**Hourly bucketing amplifies impact:**

For `exchangeRateFileId`, `feeSchedulesFileId`, and `simpleFeeSchedulesFileId`, `load()` replaces the raw timestamp with `roundDownToHour(consensusTimestamp)` (line 115). This means every request within the same clock-hour shares one cache key. If the attacker caches the genesis file for hour H before any legitimate DB entry is loaded for that hour, every user querying within hour H receives the genesis exchange rates / fee schedule for up to 60 minutes (`expireAfterWrite=60m`, `fee` cache spec, `CacheProperties` line 36).

**Cache size is trivially exhaustible:**

Both caches are configured with `maximumSize=20` (`systemFile` line 45, `fee` line 36 of `CacheProperties`). Twenty requests with distinct timestamps (or distinct hours) fill the entire cache with genesis-file entries, evicting any legitimate entries via Caffeine's LRU policy.

### Impact Explanation

1. **Wrong data served**: For exchange-rate and fee-schedule files, all EVM contract calls that depend on fee/rate data within the poisoned hour bucket receive genesis-era values instead of current network values. This affects gas cost calculations and any contract logic that reads exchange rates.
2. **Cache thrashing / DB amplification**: For address-book and throttle-definition files (no hour rounding), 20 requests with distinct nanosecond timestamps fill the `systemFile` cache. Subsequent legitimate requests always miss the cache and hit the database, increasing DB load proportionally to request rate.
3. **Persistence**: The `exchangeRate` cache TTL is 60 minutes; the `systemFile` cache TTL is 10 minutes. The attacker need only re-poison once per TTL window to maintain the effect indefinitely.

### Likelihood Explanation

No authentication or authorization is required to call the web3 JSON-RPC endpoint that triggers `FileReadableKVState.readFromDataSource` → `SystemFileLoader.load`. The attacker needs only to craft `eth_call` or `eth_estimateGas` requests with a `block` parameter corresponding to a timestamp before the first DB entry for the target file. The earliest DB entries for mainnet system files have well-known timestamps (network genesis), making the pre-genesis timestamp range trivially discoverable. The attack is fully repeatable and scriptable with 20 HTTP requests per TTL window.

### Recommendation

1. **Do not cache the genesis fallback.** Distinguish between a DB-loaded result and the genesis fallback in `loadWithRetry` (e.g., return `Optional<File>` or a sentinel), and in `load()` only call `cache.put` when the result came from the database:

```java
var result = loadFromDB(fileId, consensusTimestamp);
if (result != null && !isGenesisFallback(result)) {
    cache.put(cacheKey, result);
}
```

2. **Alternatively**, validate that the requested timestamp is within a reasonable range (e.g., ≥ network genesis timestamp) before attempting a DB lookup, and return the genesis file without caching for out-of-range timestamps.

3. **Increase cache size or add negative-result TTL** so that even if genesis entries are cached, they expire quickly and do not crowd out legitimate entries.

### Proof of Concept

**Precondition:** Mirror node is running; the earliest `file_data` row for the exchange-rate file (entity ID for file 112) has `consensus_timestamp = T_genesis` (e.g., `1000000000` ns).

**Steps:**

1. Send 20 `eth_call` requests to the web3 JSON-RPC endpoint, each with a `block` tag corresponding to a timestamp `< T_genesis` (e.g., nanosecond timestamps 1, 2, …, 20 mapped to block numbers via the record-file index). Each request targets a contract that reads the exchange rate system file internally.

2. Observe that `SystemFileLoader.load` logs `"Updating cache for key FileId=0.0.112, timestamp=..."` for each request, confirming genesis-file entries are written into the `exchangeRate` Caffeine cache.

3. After step 2, the `exchangeRate` cache (`maximumSize=20`) is full of genesis-file entries.

4. Send a legitimate `eth_call` at the current block. The cache key `(0.0.112, roundDownToHour(now))` is absent (evicted or never written). The DB is queried correctly, but the result is immediately evicted by the next attacker request.

5. To demonstrate wrong-data serving: send one request at timestamp `H` (start of current hour, before any DB update for that hour). Genesis exchange rates are cached for `(0.0.112, H)`. All subsequent requests within hour `H` receive genesis exchange rates from cache for up to 60 minutes.
