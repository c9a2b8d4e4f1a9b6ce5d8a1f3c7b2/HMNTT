### Title
`BlockType.of("0x0")` Bypasses `EARLIEST` Identity Check via Reference Inequality, Routing to `findByIndex(0)` Instead of `findEarliest()`

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `EARLIEST` sentinel, but `BlockType.of("0x0")` (and `BlockType.of("0")`) always allocates a new `BlockType` instance rather than returning the `EARLIEST` singleton. This causes any hex-zero or decimal-zero block parameter to silently fall through to `findByIndex(0)`, which executes a semantically different SQL query than `findEarliest()`. In deployments where the earliest indexed record file does not have `index = 0`, the eth_call fails or resolves against an empty block context instead of the true genesis block.

### Finding Description
**Root cause — `BlockType.java` lines 29 and 73–76:**

```java
// Line 29 — the only EARLIEST singleton
public static final BlockType EARLIEST = new BlockType("earliest", 0L);

// Lines 73-76 — hex-number branch, always allocates a new instance
final var hexNum = matcher.group(GROUP_HEX_NUM);
if (hexNum != null) {
    return new BlockType(hexNum, Long.parseLong(hexNum, 16)); // "0x0" → new BlockType("0", 0L)
}
```

`BlockType` is a Java `record`. Its auto-generated `equals()` compares both `name` and `number`. `EARLIEST` has `name="earliest"`, while `BlockType.of("0x0")` produces `name="0"`. Therefore neither `==` nor `.equals()` would match — the two objects are permanently distinguishable.

**Dispatch logic — `RecordFileServiceImpl.java` lines 20–28:**

```java
if (block == BlockType.EARLIEST) {          // FALSE for BlockType.of("0x0")
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST) {     // FALSE
    return recordFileRepository.findLatest();
} else if (block.isHash()) {                // FALSE (number=0, not BLOCK_HASH_SENTINEL=-1)
    return recordFileRepository.findByHash(block.name());
}
return recordFileRepository.findByIndex(block.number()); // ← always reached for "0x0"
```

**Query difference — `RecordFileRepository.java` lines 24 and 28–29:**

| Method | SQL |
|---|---|
| `findEarliest()` | `SELECT * FROM record_file ORDER BY index ASC LIMIT 1` |
| `findByIndex(0)` | `SELECT r FROM RecordFile r WHERE r.index = 0` |

`findEarliest()` returns the record with the minimum index, whatever that value is. `findByIndex(0)` returns a record only if one with `index = 0` exists. If the mirror node's earliest ingested block has `index > 0` (partial sync, chain fork, or non-zero genesis), `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would return a valid block.

Additionally, the two paths use different Spring cache managers (`CACHE_MANAGER_RECORD_FILE_EARLIEST` vs `CACHE_MANAGER_RECORD_FILE_INDEX`), so a warm `findEarliest()` cache is never consulted for `"0x0"` requests, and vice-versa — creating a cache-coherence split for what should be the same logical block.

### Impact Explanation
An unprivileged caller submitting `eth_getBalance`, `eth_call`, or any JSON-RPC method with `blockParam = "0x0"` or `blockParam = "0"` will have their request routed to `findByIndex(0)`. In any environment where the genesis record file does not carry `index = 0` (partial mirror sync, testnet with non-zero start index), the lookup returns empty, causing the RPC layer to treat the block as non-existent. Smart contract simulation then either aborts with a block-not-found error or falls back to an unintended default context, producing results inconsistent with what `"earliest"` would return. No funds are directly at risk, but contract state reads and simulations are silently wrong or unavailable — matching the stated Medium severity.

### Likelihood Explanation
The trigger requires only a valid JSON-RPC call with `"0x0"` as the block parameter — no authentication, no special role, no on-chain transaction. Any public-facing mirror-node JSON-RPC endpoint is reachable by any internet user. The condition is reproducible deterministically and requires no timing or race. The only prerequisite that limits impact is whether the deployment's earliest block has `index = 0`; in standard full-sync Hedera deployments it does, reducing the empty-result scenario to partial-sync or non-standard deployments, but the cache-split and semantic divergence exist universally.

### Recommendation
In `BlockType.of()`, after parsing a numeric value (both decimal and hex branches), check whether the result equals zero and return the `EARLIEST` singleton:

```java
// hex branch
long val = Long.parseLong(hexNum, 16);
if (val == 0L) return EARLIEST;
return new BlockType(hexNum, val);

// decimal branch
long val = Long.parseLong(decimal, 10);
if (val == 0L) return EARLIEST;
return new BlockType(value, val);
```

Alternatively, replace the `==` identity checks in `findByBlockType()` with value-based checks:

```java
if (block.number() == 0L && !block.isHash()) {
    return recordFileRepository.findEarliest();
}
```

Add a test asserting `BlockType.of("0x0") == BlockType.EARLIEST` and `BlockType.of("0") == BlockType.EARLIEST` to prevent regression.

### Proof of Concept
1. Deploy or connect to a Hedera mirror-node JSON-RPC endpoint.
2. Send:
   ```json
   {"jsonrpc":"2.0","method":"eth_getBalance","params":["0x<any_address>","earliest"],"id":1}
   ```
   Observe a successful response using `findEarliest()`.
3. Send:
   ```json
   {"jsonrpc":"2.0","method":"eth_getBalance","params":["0x<any_address>","0x0"],"id":2}
   ```
   Observe that `findByIndex(0)` is called instead. On a partial-sync node (earliest block index > 0), this returns `Optional.empty()` and the RPC returns a block-not-found error while request 1 succeeded — confirming the divergent routing.
4. Confirm in code: `BlockType.of("0x0") == BlockType.EARLIEST` evaluates to `false`; `BlockType.of("earliest") == BlockType.EARLIEST` evaluates to `true`.
