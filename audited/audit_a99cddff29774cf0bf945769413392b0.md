### Title
Unauthenticated DoS via Unbounded `result=fail` Queries Generating Expensive `NOT IN` DB Scans on `/api/v1/transactions`

### Summary
The REST API accepts `result=fail` from any unauthenticated user, which generates a `t.result NOT IN (...)` SQL predicate against the `transaction` table. The REST API layer has no per-IP or global rate limiting, and response caching is optional/disabled by default. An attacker can flood the endpoint with concurrent `result=fail` requests, each triggering a full filtered scan of the transaction table, exhausting DB connection pool resources and causing `httpStatusCodes.SERVICE_UNAVAILABLE` (503) for all users.

### Finding Description

**Code path 1 — filter constant definition:**
`rest/constants.js` lines 115–118 define `transactionResultFilter.FAIL = 'fail'`. [1](#0-0) 

**Code path 2 — SQL generation:**
In `rest/transactions.js`, `extractSqlFromTransactionsRequest` at lines 426–428 translates `result=fail` into `t.result NOT IN (<success_ids>)`: [2](#0-1) 

This `NOT IN` predicate is non-selective on a high-volume network where >99% of transactions succeed. PostgreSQL cannot use a simple equality index scan; it must evaluate the predicate against every row in the scanned timestamp range.

**Code path 3 — query execution:**
`getTransactionTimestamps` at line 483 executes the query directly against the DB pool with no concurrency guard: [3](#0-2) 

**Code path 4 — no rate limiting in REST API:**
`rest/server.js` shows the middleware stack: `requestLogger` → `authHandler` → optional `metricsHandler` → optional `responseCacheCheckHandler` → route handlers. There is no rate-limiting middleware for the `/api/v1/transactions` route. [4](#0-3) 

The only throttling present in the codebase is in the `web3` Java module (`ThrottleManagerImpl`), which is entirely separate from the Node.js REST API. [5](#0-4) 

**Code path 5 — caching is optional:**
`responseCacheCheckHandler` is only registered when `config.cache.response.enabled && config.redis.enabled` are both true. In deployments without Redis, every request hits the DB. [6](#0-5) 

**Root cause and failed assumption:**
The design assumes that infrastructure-level rate limiting (e.g., Traefik middleware defined in `charts/hedera-mirror-rest/templates/middleware.yaml`) will be present, but this is optional Helm configuration. The application itself has no fallback rate limiting. The `NOT IN` predicate is structurally more expensive than `IN` because it cannot be satisfied by a sparse index on `result`; it requires scanning all rows in the timestamp window and filtering out successes.

### Impact Explanation
On a mainnet-scale deployment with millions of transactions, each `result=fail` request forces the DB to scan the full timestamp-bounded window and discard the majority of rows (successful ones). Concurrent flooding saturates the DB connection pool and CPU. Once the pool is exhausted, all API endpoints return `httpStatusCodes.SERVICE_UNAVAILABLE` (503), denying service to all legitimate users. No authentication or special privilege is required.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of the public API (documented in OpenAPI spec). The `result=fail` parameter is explicitly listed as a valid filter value. The attacker needs no credentials, no tokens, and no special network position. The attack is trivially repeatable and scriptable (e.g., `ab -n 10000 -c 200 'https://host/api/v1/transactions?result=fail'`). Likelihood is high on any deployment without external rate limiting configured.

### Recommendation
1. **Add application-level rate limiting** in the Node.js REST API (e.g., `express-rate-limit`) as a defense-in-depth measure, independent of infrastructure configuration.
2. **Add a DB index on `transaction.result`** or a partial index on `result != 22` to make the `NOT IN` scan cheaper.
3. **Enforce Redis response caching** as a required component, or add a short-lived in-process cache for identical query keys.
4. **Bound concurrent DB queries** per endpoint using a semaphore or connection pool limit per route.

### Proof of Concept
```bash
# No authentication required
# Send 500 concurrent requests with result=fail
ab -n 5000 -c 500 \
  'https://<mirror-node-host>/api/v1/transactions?result=fail&limit=100'

# Simultaneously verify legitimate requests start returning 503
curl -v 'https://<mirror-node-host>/api/v1/transactions'
# Expected: HTTP 503 Service Unavailable
```

Preconditions: target is a mainnet or high-volume testnet mirror node with Redis caching disabled or no external rate limiter configured. The attacker is unauthenticated.
