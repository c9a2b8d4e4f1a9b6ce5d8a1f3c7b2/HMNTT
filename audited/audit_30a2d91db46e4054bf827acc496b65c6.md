### Title
Global Rate Limit Bucket Allows Single-Source Monopolization and Resource Exhaustion

### Summary
The `rateLimitBucket` and `gasLimitBucket` in `ThrottleConfiguration` are single application-scoped singletons with no per-IP or per-user partitioning. A single unprivileged attacker can consume the entire 500 RPS global capacity while simultaneously submitting maximum-gas contract calls, starving all legitimate users and driving node CPU/memory to maximum sustained load.

### Finding Description
**Code path:**

`ThrottleConfiguration.java` lines 24–32 creates one global `Bucket` bean: [1](#0-0) 

`ThrottleManagerImpl.java` lines 37–42 consumes from that single shared bucket for every incoming request regardless of source: [2](#0-1) 

`ThrottleProperties.java` line 35 sets the default capacity at 500 req/s: [3](#0-2) 

The `gasLimitBucket` is equally global (default 7.5B gas/s, scaled by 10,000 = 750,000 tokens/s): [4](#0-3) 

**Root cause:** The Spring `@Bean` lifecycle makes both buckets singletons. `tryConsume(1)` on the rate bucket and `tryConsume(scaleGas(request.getGas()))` on the gas bucket are called with no source identity — no IP address, no API key, no session. The `RequestProperties` filter list is empty by default: [5](#0-4) 

**Failed assumption:** The design assumes the 500 RPS global cap will be shared fairly across many clients. There is no enforcement mechanism to ensure that.

**Exploit flow:**
1. Attacker opens a connection pool and fires 500 concurrent `eth_call` / `eth_estimateGas` requests per second, each with `gas = 15,000,000` (the maximum accepted value).
2. `rateLimitBucket.tryConsume(1)` succeeds for all 500 — the attacker has drained the entire global token pool.
3. `gasLimitBucket.tryConsume(scaleGas(15_000_000))` = `tryConsume(1500)` per request × 500 req/s = 750,000 tokens/s — exactly the full gas budget.
4. Every request from any other IP now receives `ThrottleException("Requests per second rate limit exceeded")` → HTTP 429.
5. The node's EVM executor processes 500 max-gas contract calls per second continuously, saturating CPU and memory.

### Impact Explanation
All legitimate users are completely denied service (HTTP 429) for the duration of the attack. The node simultaneously runs at maximum EVM execution load (500 × 15M gas/s), which can increase CPU utilization well beyond 30% compared to a baseline with normal mixed traffic. This is a full availability denial for a public read-only API endpoint that requires no authentication.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The attacker needs only a single machine with sufficient outbound bandwidth (~500 small HTTP POST requests/s is trivially achievable). The attack is trivially repeatable and sustainable indefinitely. The absence of any per-IP tracking means IP rotation is not even necessary — a single IP can exhaust the entire budget.

### Recommendation
Replace the single global bucket with a per-source-IP bucket map (e.g., using Bucket4j's `BucketProxyManager` backed by a `ConcurrentHashMap` or a distributed cache). Each IP should receive its own sub-quota (e.g., 10–50 req/s) while the global cap remains as a secondary ceiling. Additionally, enforce per-IP gas limits to prevent a single source from monopolizing the gas budget. The `X-Forwarded-For` / `RemoteAddr` from the `HttpServletRequest` should be extracted in a servlet filter before the throttle check and used as the bucket key.

```bash
# Exhaust the global 500 RPS rate limit from a single host
seq 1 500 | xargs -P 500 -I{} curl -s -X POST \
  http://<mirror-node-web3>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001",
       "gas":"0xE4E1C0",
       "data":"0x<expensive_calldata>",
       "block":"latest"}' &

# Simultaneously verify other users are denied
curl -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":"0x5208","data":"0x","block":"latest"}'
# Expected: HTTP 429 {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()

                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
