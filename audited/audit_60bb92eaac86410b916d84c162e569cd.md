### Title
Unauthenticated Log Flooding via Invalid `fileId` in gRPC `getNodes` Endpoint

### Summary
The gRPC `getNodes` endpoint in `NetworkServiceImpl` throws an `IllegalArgumentException` for any `fileId` not in `validFileIds` (i.e., not file 101 or 102). This exception is unconditionally logged at WARN level via `ProtoUtil.clientError()` on every request. Because the endpoint has no authentication, no per-IP rate limiting, and no log-rate throttling, an unprivileged attacker can flood the server's logs at high volume by repeatedly sending `AddressBookQuery` messages with an arbitrary invalid `fileId`.

### Finding Description

**Exact code path:**

1. `NetworkController.getNodes()` receives any `AddressBookQuery` from an unauthenticated caller and calls `networkService::getNodes` via a reactive pipeline with `.onErrorMap(ProtoUtil::toStatusRuntimeException)`. [1](#0-0) 

2. `NetworkServiceImpl.getNodes()` immediately throws `IllegalArgumentException` if the supplied `fileId` is not in `validFileIds` (the set `{addressBookFile101, addressBookFile102}`). No database access occurs; the check is a pure in-memory set lookup, making the path extremely cheap to trigger. [2](#0-1) 

3. The `IllegalArgumentException` is caught by `onErrorMap` and routed to `ProtoUtil.toStatusRuntimeException()`, which matches it at line 48 and calls `clientError()`. [3](#0-2) 

4. `clientError()` unconditionally emits a `log.warn(...)` for every invocation — no sampling, no deduplication, no rate gate. [4](#0-3) 

**Root cause:** The assumption that client-supplied invalid arguments are rare or low-volume is violated. The warn log is appropriate for debugging but is emitted on every request with no guard.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` limits *concurrent* calls per TCP connection, not the *rate* of sequential calls, and does not cap the number of connections an attacker may open. [5](#0-4) [6](#0-5) 

- The bucket4j throttling infrastructure exists only in the `web3` module; there is no equivalent for the gRPC module. [7](#0-6) 

- No authentication or authorization is required to call `getNodes`. [8](#0-7) 

### Impact Explanation
An attacker can generate thousands of WARN log entries per second with minimal resources (a single machine opening many connections, each cycling through 5 rapid sequential calls). This can: (1) exhaust disk space allocated to log files, (2) overwhelm log-aggregation pipelines (e.g., Elasticsearch/Splunk ingestion limits), and (3) bury legitimate WARN/ERROR signals from real failures, directly degrading operator observability and incident response capability. The error path involves no DB I/O, so the server-side cost per request is negligible, making the attack asymmetric.

### Likelihood Explanation
The endpoint is publicly reachable on port 5600 with no credentials required. The trigger condition (any `fileId` outside `{101, 102}`) is trivially satisfied with a single-field protobuf message. No special knowledge of the system is needed. The attack is fully repeatable and scriptable with any gRPC client library.

### Recommendation
1. **Add a per-IP or global rate limiter** on the gRPC layer (e.g., a `ServerInterceptor` using bucket4j or Resilience4j, analogous to the web3 throttle) that rejects or silently drops requests exceeding a threshold.
2. **Suppress or sample the warn log** for `INVALID_ARGUMENT` client errors in `ProtoUtil.clientError()` — e.g., log at DEBUG level, or use a rate-limited logger (SLF4J's `RateLimitingLogger` / Logback's `DuplicateMessageFilter`).
3. **Optionally move the `fileId` validation upstream** into `NetworkController.toFilter()` before the reactive pipeline, so the error is returned without touching the logging path at all.

### Proof of Concept
```python
import grpc
from concurrent.futures import ThreadPoolExecutor
# proto-generated stubs assumed available
from com.hedera.mirror.api.proto import network_service_pb2, network_service_pb2_grpc
from proto.services import basic_types_pb2

def flood(target="mirror-node-host:5600", n=10000):
    channel = grpc.insecure_channel(target)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    invalid_query = network_service_pb2.AddressBookQuery(
        file_id=basic_types_pb2.FileID(shardNum=0, realmNum=0, fileNum=999)
    )
    for _ in range(n):
        try:
            list(stub.getNodes(invalid_query))
        except grpc.RpcError:
            pass  # INVALID_ARGUMENT expected; server logs warn on every call

with ThreadPoolExecutor(max_workers=50) as ex:
    for _ in range(50):
        ex.submit(flood)
```
Each of the 50 threads fires 10,000 requests; each request causes one `log.warn("Client error IllegalArgumentException: Not a valid address book file")` entry on the server, producing 500,000 WARN log lines with no server-side defence.
