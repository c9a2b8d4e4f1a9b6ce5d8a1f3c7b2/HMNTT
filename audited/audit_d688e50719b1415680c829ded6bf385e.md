### Title
Unauthenticated `/health/readiness` Endpoint Causes Per-Request PostgreSQL Connection Churn Leading to DoS

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` uses `hellofresh/health-go/v4/checks/postgres`, which opens a brand-new `sql.DB` connection (full TCP + authentication handshake) on every single health check invocation and closes it immediately after. Because the endpoint is publicly reachable with no rate limiting applied by default, an unprivileged attacker can flood it to exhaust PostgreSQL's `max_connections` and impose continuous authentication overhead, denying service to legitimate application connections.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `NewHealthController()`, line 42:
```go
Check: postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
```

The `hellofresh/health-go/v4/checks/postgres` package (version 4.7.0, declared in `rosetta/go.mod` line 12) implements its check function as:
```go
func New(config Config) health.CheckFunc {
    return func(ctx context.Context) error {
        db, err := sql.Open("postgres", config.DSN)
        if err != nil { return err }
        defer db.Close()
        return db.PingContext(ctx)
    }
}
```
Every call to `HandlerFunc` (line 75) triggers this function: `sql.Open` allocates a new driver handle, `PingContext` establishes a TCP connection and performs the full PostgreSQL authentication protocol (startup message, password exchange, backend key data), and `db.Close()` tears it all down. There is no connection reuse, no pooling, and no result caching between requests.

**Why existing checks fail:**

The Traefik middleware chain (circuit breaker, `inFlightReq`, `rateLimit`) is defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166, but it is gated on `global.middleware: true`. The default value is `global.middleware: false` (line 95). Even when enabled, the Traefik `Middleware` resource is applied only to the ingress paths listed (`/rosetta/account`, `/rosetta/block`, etc., lines 126–133) — `/health/readiness` is not routed through the ingress at all; it is served directly by the application on its pod port. There is zero application-level rate limiting, authentication, or check-result caching on this handler.

### Impact Explanation
PostgreSQL enforces a hard `max_connections` limit (default 100). Each concurrent flood request consumes one connection slot for up to the 10-second timeout (line 40). With ~100 concurrent requests the attacker saturates the connection pool; all legitimate GORM queries from the application's own pool (`rosetta/app/db/db.go` lines 31–33) receive `FATAL: sorry, too many clients already`. The application becomes fully unable to serve any Rosetta API request. Additionally, repeated authentication handshakes impose CPU cost on the PostgreSQL server (password hashing, SSL negotiation if enabled), amplifying the impact even below the connection limit.

### Likelihood Explanation
No privileges, credentials, or special network position are required — a single machine with a standard HTTP flood tool (`wrk`, `ab`, `hey`) targeting the pod's exposed port suffices. The endpoint is intentionally public (it is a Kubernetes readiness probe). The attack is trivially repeatable and requires no prior knowledge beyond the port number. Default deployments ship with `global.middleware: false`, meaning the rate-limit middleware is inactive out of the box.

### Recommendation
1. **Cache health check results**: Configure `hellofresh/health-go` with a `CacheDuration` so repeated requests within a window reuse the last result without opening a new connection.
2. **Use a persistent connection for the health check**: Replace `postgres.New(...)` with a custom check that reuses the application's existing GORM/`sql.DB` pool (call `sqlDb.PingContext(ctx)` on the already-open pool from `rosetta/app/db/db.go`).
3. **Rate-limit the health endpoint at the application layer**: Apply a per-IP token bucket directly in the Go HTTP handler, independent of the Traefik middleware toggle.
4. **Enable `global.middleware: true` by default** or document that it must be enabled for production deployments.

### Proof of Concept
```bash
# Prerequisites: rosetta service reachable at $HOST:$PORT, no auth required
# Step 1: Flood /health/readiness with 200 concurrent connections
wrk -t 20 -c 200 -d 60s http://$HOST:$PORT/health/readiness

# Step 2: In parallel, observe PostgreSQL connection count spike to max_connections
psql -c "SELECT count(*) FROM pg_stat_activity;"

# Step 3: Attempt a legitimate Rosetta API call — it will fail with 503/connection error

  -H "Content-Type: application/json" \
  -d '{"metadata":{}}'
