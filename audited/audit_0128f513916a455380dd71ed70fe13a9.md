### Title
Unauthenticated DoS via Rate Saturation of gRPC `NetworkService/getNodes` Endpoint Due to Missing Per-Client Rate Limiting

### Summary
The `NetworkController.getNodes()` method exposes the address book retrieval endpoint publicly with no application-level or infrastructure-level per-client rate limiting. The `GCPBackendPolicy` `maxRatePerEndpoint: 250` setting is a total backend capacity cap per pod — not a per-client limit — meaning a single unauthenticated attacker sustaining 250+ RPS can fully saturate the single default pod and cause all legitimate clients to receive 503 errors. With HPA disabled by default and no per-IP throttle at any layer, the service is trivially deniable by any external actor.

### Finding Description

**Code path:**

`NetworkController.getNodes()` at `grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java` lines 33–43 accepts any `AddressBookQuery` from any caller with no authentication check, no per-client quota, and no application-level throttle:

```java
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

**Root cause — `maxRatePerEndpoint` is a backend capacity cap, not a per-client limit:**

`charts/hedera-mirror-grpc/values.yaml` line 69:
```yaml
maxRatePerEndpoint: 250  # Requires a change to HPA to take effect
```

In GCP's `GCPBackendPolicy`, `maxRatePerEndpoint` defines the maximum aggregate RPS the load balancer will forward to each backend pod. It is not a per-source-IP quota. Once the single pod (HPA disabled by default, `hpa.enabled: false`, line 113) receives 250 RPS from any combination of sources, the load balancer begins returning 503 to all additional requests — including those from legitimate clients.

**Traefik middleware has no per-client rate limiting for gRPC:**

`charts/hedera-mirror-grpc/values.yaml` lines 156–161:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 3
      initialInterval: 250ms
```

No `rateLimit` or `inFlightReq` with `sourceCriterion` is present. Compare to the rosetta chart which explicitly configures `rateLimit: average: 10, sourceCriterion: requestHost: true` and `inFlightReq: amount: 5, sourceCriterion: ipStrategy: depth: 1`. Furthermore, `global.middleware: false` (line 107) means even this circuit breaker middleware is disabled by default.

**Gateway routing confirms public exposure** (`values.yaml` lines 88–91):
```yaml
matches:
  - method:
      service: com.hedera.mirror.api.proto.ConsensusService
  - method:
      service: com.hedera.mirror.api.proto.NetworkService
```

`sessionAffinity: type: CLIENT_IP` (line 71) routes all requests from a single attacker IP to the same pod, making saturation from a single source even more reliable.

### Impact Explanation

When the single default pod is saturated at 250 RPS, the GCP load balancer returns 503 to all excess requests. Legitimate monitor clients and SDK clients that call `getNodes()` to retrieve address book data — required to discover consensus node endpoints for transaction routing — are denied service for the duration of the attack. The `hiero.mirror.monitor.nodeValidation.retrieveAddressBook: true` configuration (docs/configuration.md line 479) confirms that the monitor depends on this endpoint to populate its node list. Sustained DoS on this endpoint degrades the monitor's ability to validate and route transactions to healthy consensus nodes.

### Likelihood Explanation

No privileges, credentials, or special network position are required. Any external actor with internet access and a basic gRPC client (e.g., `grpcurl` in a loop, or a simple Go/Python script) can sustain 250 RPS to a single endpoint trivially. The attack is repeatable, stateless, and requires no prior knowledge beyond the publicly documented service name `com.hedera.mirror.api.proto.NetworkService/getNodes`. The default single-replica deployment with HPA disabled means there is no automatic scaling to absorb the load.

### Recommendation

1. **Add per-client rate limiting to the gRPC Traefik middleware** for the `NetworkService` path, analogous to the rosetta chart's `rateLimit` + `inFlightReq` with `sourceCriterion: ipStrategy`. Enable `global.middleware: true` by default for the gRPC chart.
2. **Enable HPA by default** (`hpa.enabled: true`) so the backend can scale beyond 1 pod before the `maxRatePerEndpoint` cap is reached.
3. **Add application-level per-client throttling** inside `NetworkController.getNodes()` using a per-IP token bucket (similar to the `web3` module's `ThrottleManagerImpl`/`ThrottleConfiguration` pattern), so the protection is not solely infrastructure-dependent.
4. **Add a GCP Cloud Armor security policy** with per-IP rate limiting rules at the gateway level as a defense-in-depth measure.

### Proof of Concept

```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Replace <HOST>:<PORT> with the public gRPC gateway endpoint

# Step 1: Confirm the endpoint is reachable
grpcurl -plaintext -d '{"file_id": {"fileNum": 102}, "limit": 1}' \
  <HOST>:<PORT> com.hedera.mirror.api.proto.NetworkService/getNodes

# Step 2: Saturate the endpoint (run from attacker machine, no auth needed)
# Send 300 concurrent RPS sustained using parallel processes
for i in $(seq 1 300); do
  grpcurl -plaintext -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <HOST>:<PORT> com.hedera.mirror.api.proto.NetworkService/getNodes &
done
wait

# Step 3: From a separate legitimate client, observe 503/UNAVAILABLE responses
grpcurl -plaintext -d '{"file_id": {"fileNum": 102}, "limit": 1}' \
  <HOST>:<PORT> com.hedera.mirror.api.proto.NetworkService/getNodes
# Expected: ERROR: Code: Unavailable / HTTP 503
```

Sustained execution of Step 2 in a loop causes the single backend pod to hit the `maxRatePerEndpoint: 250` cap, after which all requests from legitimate clients in Step 3 are rejected by the GCP load balancer until the attacker stops.
