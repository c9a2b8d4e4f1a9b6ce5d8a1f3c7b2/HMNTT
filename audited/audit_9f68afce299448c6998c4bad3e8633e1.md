### Title
Integer Overflow in `poll()` Converts Large User-Supplied Limit to Negative pageSize, Bypassing DB Row Limit and Enabling Unbounded Query DoS

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unsafe narrowing cast from `long` to `int`. Any unprivileged gRPC client can send a `limit` value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) via the `ConsensusTopicQuery.limit` protobuf field, causing the cast to produce a negative `pageSize`. The resulting internal `TopicMessageFilter` with a negative limit has `hasLimit()` return `false`, so `findByFilter()` executes with no `setMaxResults()` bound — returning every row in the database for that topic on every poll cycle, indefinitely.

### Finding Description

**Full code path:**

**Step 1 — External input, no upper-bound check:**
`ConsensusController.toFilter()` maps the raw protobuf `uint64 limit` field directly to the filter with no upper-bound validation: [1](#0-0) 

The protobuf field is declared `uint64`: [2](#0-1) 

**Step 2 — Validation passes for `Long.MAX_VALUE`:**
`TopicMessageFilter.limit` only carries `@Min(0)`. `Long.MAX_VALUE` (9,223,372,036,854,775,807) is positive and passes this constraint: [3](#0-2) 

**Step 3 — Integer overflow in `poll()`:** [4](#0-3) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `(int)(Long.MAX_VALUE - 0)` = `(int)(0x7FFFFFFFFFFFFFFFL)` = `-1`
- `Math.min(-1, maxPageSize)` = `-1`

**Step 4 — Internal filter built with limit = -1, bypassing Spring validation:** [5](#0-4) 

The `toBuilder().limit(-1).build()` call is an internal Lombok builder invocation — Spring AOP does not intercept it, so `@Min(0)` is never re-evaluated.

**Step 5 — `hasLimit()` returns `false` for negative limit:** [6](#0-5) 

With `limit = -1`, `hasLimit()` returns `false`.

**Step 6 — `findByFilter()` skips `setMaxResults()`, unbounded DB query:** [7](#0-6) 

`setMaxResults()` is never called. The JPA query returns every `TopicMessage` row matching the topic and start time.

**Step 7 — `isComplete()` never terminates the polling loop:** [8](#0-7) 

`filter.getLimit() == total.get()` requires `total` to reach `Long.MAX_VALUE` — practically impossible. For throttled mode, `numRepeats = Long.MAX_VALUE`. The polling loop runs forever, executing an unbounded DB query on every interval.

### Impact Explanation

Each poll cycle issues a full table scan of all topic messages for the targeted topic with no row limit. On a busy topic with millions of messages, this loads the entire result set into the JVM heap, causing memory exhaustion (OOM) and crashing the gRPC service. Simultaneously, the database is subjected to repeated full-scan queries. Because the polling loop never terminates (limit never reached), a single malicious subscription sustains this load indefinitely. Multiple concurrent attackers amplify the effect to a complete network-level DoS of the mirror node's gRPC endpoint, preventing all legitimate subscribers from receiving consensus messages.

### Likelihood Explanation

The exploit requires only a standard gRPC client (e.g., `grpcurl`) with no credentials, no special role, and no prior knowledge beyond the public protobuf schema. The trigger value (`limit = Long.MAX_VALUE`, or any value > `Integer.MAX_VALUE` such as `2147483648`) is trivially constructable. The attack is immediately repeatable and requires no timing precision. The mirror node's gRPC port is publicly exposed by design.

### Recommendation

1. **Add `@Max(Integer.MAX_VALUE)` to `TopicMessageFilter.limit`** to reject any limit that would overflow the `int` cast before it reaches `poll()`.
2. **Replace the unsafe cast** in `PollingTopicMessageRetriever.poll()` with a safe narrowing using `Math.toIntExact()` (throws `ArithmeticException` on overflow) or `(int) Math.min(filter.getLimit() - context.getTotal().get(), Integer.MAX_VALUE)`.
3. **Add a hard cap** on the maximum allowed `limit` value in `ConsensusController.toFilter()` before building the filter.

### Proof of Concept

```bash
# Send a gRPC subscribeTopic request with limit = Long.MAX_VALUE (9223372036854775807)
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 1}, "limit": 9223372036854775807}' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**What happens internally:**
1. `ConsensusController.toFilter()` sets `filter.limit = 9223372036854775807L` — passes `@Min(0)`.
2. `PollingTopicMessageRetriever.poll()` computes `(int)(9223372036854775807L - 0L)` = `-1`.
3. `Math.min(-1, maxPageSize)` = `-1`.
4. New filter built with `limit = -1`; `hasLimit()` returns `false`.
5. `findByFilter()` calls no `setMaxResults()` — full table scan executes.
6. `isComplete()` never returns `true`; polling repeats at configured interval forever.
7. Service heap exhausts; gRPC endpoint crashes or becomes unresponsive for all subscribers.
