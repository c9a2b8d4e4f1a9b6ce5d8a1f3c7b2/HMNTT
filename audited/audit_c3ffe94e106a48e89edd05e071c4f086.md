### Title
Unbounded gRPC Connection Count Enables Multi-IP Thread Pool Exhaustion via Shared `applicationTaskExecutor`

### Summary
`grpcServerConfigurer()` configures the Netty gRPC server with only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) but imposes no global connection limit. Because all accepted connections share the Spring `applicationTaskExecutor` thread pool, an attacker controlling multiple source IPs can open an unbounded number of connections — each contributing up to 5 concurrent long-lived streaming calls — saturating the shared executor and degrading or denying service to legitimate users. IP-based blocking is ineffective because each connection originates from a distinct IP.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35 (`grpcServerConfigurer()`):

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);                                    // line 32
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // line 33
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** The only resource guard applied to the Netty server is `maxConcurrentCallsPerConnection`, which is a *per-connection* ceiling. No call to `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnectionIdle(...)`, `serverBuilder.permitKeepAliveWithoutCalls(...)`, or any equivalent total-connection cap is present anywhere in the gRPC configuration (confirmed by grep across all `grpc/**/*.java`). The `applicationTaskExecutor` injected at line 32 is Spring Boot's shared application-wide thread pool — it is not isolated to gRPC traffic.

**Failed assumption:** The design assumes that limiting calls per connection is sufficient to bound total executor load. This assumption breaks when an attacker controls many source IPs, because the total concurrent calls scale as `N_connections × 5`, with `N_connections` unbounded.

**Exploit flow:**
1. Attacker controls N IPs (botnet, residential proxies, cloud VMs).
2. Each IP opens one or more TCP connections to port 5600 (the gRPC port, default).
3. Each connection immediately issues 5 concurrent `subscribeTopic` (or `getAddressBook`) streaming RPCs — the maximum allowed per connection.
4. These streaming calls are long-lived (subscriptions stay open until the topic end time or client disconnect).
5. Each active call occupies a thread in `applicationTaskExecutor`.
6. With enough connections, the executor queue fills and/or all threads are busy; new tasks submitted by legitimate users are delayed or rejected.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` caps calls per connection but places no ceiling on the number of connections accepted.
- There is no `maxConnectionsTotal`, no per-IP connection rate limit, and no flow-control configuration in the server builder.
- The `applicationTaskExecutor` is shared with the rest of the Spring application, so saturation affects all async Spring work, not just gRPC.

### Impact Explanation
An attacker can sustain degraded or unavailable service for all legitimate gRPC subscribers (topic message streaming, address book queries) without any authentication or privileged access. Because the executor is application-wide, non-gRPC Spring async tasks are also affected. The impact is service-level griefing: legitimate subscribers experience timeouts, stalled streams, or connection refusals. No funds or on-chain state are affected, consistent with the medium severity classification.

### Likelihood Explanation
The gRPC port (5600) is publicly reachable by design. No authentication is required to open a connection or issue a streaming call. Residential proxy networks and cloud VMs are cheap and widely available; a modest botnet of ~200 IPs × 5 calls = 1,000 concurrent streaming threads is sufficient to saturate a typical Spring Boot executor. The attack is trivially repeatable: connections can be re-established immediately after any server-side timeout. The attacker needs no special knowledge beyond the public port number and the gRPC service definition.

### Recommendation
Apply the following mitigations in `grpcServerConfigurer()` / `NettyProperties`:

1. **Add a global connection limit** via `serverBuilder.maxConnectionAge(Duration, TimeUnit)` and `serverBuilder.maxConnectionIdle(Duration, TimeUnit)` to recycle long-lived connections and bound total active connections.
2. **Add `maxConcurrentCallsPerServer`** (a Netty-level global cap) to bound total concurrent calls regardless of connection count.
3. **Isolate the gRPC executor** — replace the shared `applicationTaskExecutor` with a dedicated, bounded `ThreadPoolExecutor` (fixed core/max size, bounded queue, `CallerRunsPolicy` or rejection handler) so gRPC traffic cannot starve other Spring async work.
4. **Expose `maxConcurrentCallsPerConnection` and the new limits** as configurable `NettyProperties` fields so operators can tune them without code changes.
5. Consider an ingress-level connection rate limit (e.g., Envoy, nginx, or a Kubernetes `NetworkPolicy`) as a defense-in-depth layer.

### Proof of Concept
```python
# Requires: grpcio, protobuf, a topic ID, and ~N proxy IPs
import grpc, threading, time
from concurrent.futures import ThreadPoolExecutor

TARGET = "mirror-node-grpc-host:5600"
CONNECTIONS_PER_IP = 1   # one connection per "IP" (simulate via threads)
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection default
TOTAL_IPS = 200           # attacker-controlled IPs

def open_streaming_calls(stub):
    # subscribeTopic is a server-streaming RPC; each call holds a thread
    request = consensus_service_pb2.ConsensusTopicQuery(topicID=..., consensusStartTime=...)
    for _ in range(CALLS_PER_CONNECTION):
        threading.Thread(target=lambda: list(stub.subscribeTopic(request))).start()

with ThreadPoolExecutor(max_workers=TOTAL_IPS) as pool:
    for _ in range(TOTAL_IPS):
        channel = grpc.insecure_channel(TARGET)
        stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
        pool.submit(open_streaming_calls, stub)

# After ~200 connections × 5 calls = 1000 concurrent threads,
# legitimate subscribeTopic calls from real users will stall or time out.
time.sleep(300)  # hold connections open
```

Expected result: legitimate gRPC clients experience `DEADLINE_EXCEEDED` or indefinite blocking; Spring async tasks unrelated to gRPC also stall due to shared executor saturation.
