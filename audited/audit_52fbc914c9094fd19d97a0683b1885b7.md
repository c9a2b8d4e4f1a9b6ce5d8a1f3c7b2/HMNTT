### Title
Unauthenticated Sequential DB Enumeration via GraphQL `getByIdAndType()` Causes Sustained DB Load Without Rate Limiting

### Summary
The GraphQL module's `EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` on every request with no caching and no application-level rate limiting. An unauthenticated attacker can flood the `/graphql/alpha` endpoint with sequential `entityId.num` values, driving a continuous stream of primary-key lookups against the database. Unlike the `grpc` module's `EntityRepository` (which has `@Cacheable`) and the `web3` module (which has a `ThrottleManager`), the GraphQL module has neither, making it the weakest link for this class of attack.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 25:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
```

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 12–17: plain `CrudRepository<Entity, Long>` with **no `@Cacheable` annotation** on `findById()`.

**Root cause — three absent controls:**

1. **No caching.** The grpc module's `EntityRepository` annotates `findById()` with `@Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, unless = "#result == null")`. The graphql `EntityRepository` has no such annotation. Every call to `getByIdAndType()` unconditionally hits the database.

2. **No application-level rate limiting.** The `web3` module has `ThrottleManagerImpl` backed by bucket4j (`requestsPerSecond = 500` default). The graphql module has zero equivalent. A search across all graphql source files for `rate`, `throttle`, `RateLimit`, and `requestsPerSecond` returns no matches.

3. **Per-query instrumentation does not bound request rate.** `GraphQlConfiguration` installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`. These reject overly complex *single* queries but place no limit on how many queries per second a client may submit.

**Exploit flow:**

The attacker sends a tight loop of POST requests to the public endpoint `/graphql/alpha`:
```
POST /graphql/alpha
{"query":"{account(input:{entityId:{shard:0,realm:0,num:N}}){id}}"}
```
with `N` incrementing on each request. Each request:
1. Passes JSON parsing limits (tiny payload, well within `maxDocumentLength=11000`).
2. Passes complexity/depth checks (trivial single-field query, complexity ≈ 2).
3. Reaches `AccountController.account()` → `EntityServiceImpl.getByIdAndType()` → `entityRepository.findById(N)` → one `SELECT * FROM entity WHERE id = N` primary-key lookup.

Because every `num` value is distinct, even a hypothetical future cache would be cold for each request. The database executes a fresh index scan per request.

**Why infrastructure controls are insufficient:**

The Traefik middleware (`charts/hedera-mirror-graphql/values.yaml`, lines 138–142) sets `inFlightReq: amount: 5` per source IP. This caps *concurrent* in-flight requests per IP, not the *sequential request rate*. An attacker issuing requests one-after-another (pipeline or rapid sequential) from a single IP is bounded to 5 simultaneous connections but can still sustain thousands of requests per second across those 5 connections. With multiple source IPs the concurrent cap is multiplied. Additionally, the `retry: attempts: 3` directive (line 143) causes the Traefik layer itself to re-issue failed requests, amplifying DB load on transient errors. The GCP gateway `maxRatePerEndpoint: 250` (line 56) is a per-backend-pod limit, not a per-client limit, and is only active when the GCP gateway is enabled (`global.gateway.enabled: false` by default).

### Impact Explanation
Each primary-key lookup is individually cheap but not free: it acquires a HikariCP connection, executes a prepared statement, and returns a full `entity` row. At sustained high request rates (achievable with 5 concurrent HTTP/1.1 connections per IP), the aggregate connection acquisition rate can exhaust the HikariCP pool, causing queuing and latency spikes for all other graphql consumers. DB CPU rises proportionally with query throughput. The Prometheus alert `GraphQLHighDBConnections` fires at 75% pool utilization; the attack can reach that threshold before any brute-force or authentication alarm triggers, because every individual request is syntactically and semantically valid.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA is required. The endpoint is documented and publicly reachable. The attack requires only an HTTP client capable of pipelining POST requests — a single `ab`, `wrk`, or `curl` loop suffices. The attacker needs no knowledge of valid account IDs; incrementing integers from 1 upward is sufficient (misses are cheap too — the DB still executes the index scan and returns empty). The attack is repeatable indefinitely and leaves no distinctive fingerprint beyond high request volume, which is not flagged as brute-force by any configured rule.

### Recommendation

1. **Add `@Cacheable` to the graphql `EntityRepository.findById()`** (mirroring the grpc module pattern) so repeated lookups for the same ID are served from memory.

2. **Add application-level rate limiting to the graphql module.** Port the `web3` module's `ThrottleManagerImpl` / bucket4j pattern, or use Spring's `@RateLimiter`, keyed per remote IP. Expose a `hiero.mirror.graphql.throttle.requestsPerSecond` property analogous to the web3 equivalent.

3. **Change Traefik `inFlightReq` to also enforce a rate limit** (e.g., `rateLimit` middleware) per source IP, not just a concurrency cap.

4. **Tune HikariCP `maximumPoolSize`** conservatively and set `connectionTimeout` low so pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept

```bash
# Requires: curl, bash. No credentials needed.
# Sends 10 000 sequential account-lookup queries with distinct num values.
# Monitor DB CPU / HikariCP active connections via Prometheus during the run.

TARGET="http://<mirror-node-host>/graphql/alpha"

for i in $(seq 1 10000); do
  curl -s -o /dev/null -X POST "$TARGET" \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{account(input:{entityId:{shard:0,realm:0,num:$i}}){id}}\"}" &
  # Keep 5 concurrent connections (matching inFlightReq limit per IP)
  if (( i % 5 == 0 )); then wait; fi
done
wait
```

Expected result: each iteration triggers one `SELECT * FROM entity WHERE id = $i` on the database. At 5 concurrent connections the DB receives ~5 queries simultaneously with no application-side throttle or cache absorbing them. Sustained over seconds, HikariCP active-connection ratio climbs and DB CPU rises above the 30% threshold relative to idle baseline.
