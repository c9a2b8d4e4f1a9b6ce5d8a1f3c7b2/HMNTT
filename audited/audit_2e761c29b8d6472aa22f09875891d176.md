### Title
Unbounded Concurrent Unthrottled DB Polling via Safety-Check Path in `PollingTopicMessageRetriever`

### Summary
Any unprivileged gRPC client can open an arbitrary number of topic subscriptions without an `endTime`. After one second, a built-in safety check fires for every open subscription and internally invokes `topicMessageRetriever.retrieve(filter, false)` — the unthrottled retrieval path — which polls the database up to 12 times at 20 ms intervals with a page size of 5 000 rows per poll. Because there is no global subscription cap and the per-connection limit of 5 is trivially bypassed by opening more connections, an attacker can saturate the database connection pool and delay or starve legitimate queries.

### Finding Description

**Exact code path**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → safety-check branch → `missingMessages(topicContext, null)` → `topicMessageRetriever.retrieve(gapFilter, false)` → `PollingTopicMessageRetriever.retrieve(filter, throttled=false)`. [1](#0-0) 

The safety check is a `Mono.delay(1 s)` that fires once per subscription. Its guard condition is `!topicContext.isComplete()`. [2](#0-1) 

`TopicContext.isComplete()` returns `false` unconditionally when `endTime == null`, so every open-ended subscription (the common case) always passes the guard and triggers the safety check.

When the safety check fires it calls `missingMessages(topicContext, null)`, which takes the `current == null` branch and calls `retrieve(gapFilter, false)`: [3](#0-2) 

`retrieve(filter, false)` constructs a `PollingContext` with the `UnthrottledProperties` values: [4](#0-3) 

Default unthrottled configuration: [5](#0-4) 

So each triggered safety check issues up to **12 sequential DB queries at 20 ms intervals, each fetching up to 5 000 rows**, via `topicMessageRepository.findByFilter()`: [6](#0-5) 

**Root cause / failed assumption**

The design assumes the safety check fires rarely and for a small number of concurrent subscribers. There is no global subscription counter check before allowing a new subscription, and the per-connection limit of 5 concurrent calls is the only guard: [7](#0-6) 

An attacker opens *K* connections × 5 calls each = 5K simultaneous subscriptions, all without authentication.

**Why existing checks fail**

- `maxConcurrentCallsPerConnection = 5` is per-connection, not global; opening more TCP connections trivially bypasses it.
- `retrieverProperties.getTimeout()` (default 60 s) only bounds how long a single retrieval runs, not how many run concurrently.
- `subscriberCount` is a metric gauge only — it is never checked against a maximum before accepting a new subscription. [8](#0-7) 

### Impact Explanation
With N simultaneous open-ended subscriptions, N concurrent unthrottled `retrieve(false)` chains start after ~1 second. Each chain holds a JDBC/JPA connection for up to 12 × 20 ms ≈ 240 ms while issuing large queries. If N exceeds the database connection pool size, legitimate requests queue behind the attacker's queries, causing latency spikes or timeouts for all users. The `findByFilter` implementation fetches full result lists into memory (`getResultList().stream()`), amplifying memory pressure alongside connection exhaustion. [9](#0-8) 

### Likelihood Explanation
No authentication is required to call `subscribeTopic`. The gRPC endpoint is publicly reachable. Opening hundreds of TCP connections and issuing `subscribeTopic` RPCs is trivial with any gRPC client library. The 1-second delay before the safety check fires is not a meaningful barrier. The attack is fully repeatable: after subscriptions time out (60 s default), the attacker simply reconnects.

### Recommendation
1. **Enforce a global (or per-IP) subscription limit** checked atomically in `TopicMessageServiceImpl.subscribeTopic()` before accepting a new subscription; return `RESOURCE_EXHAUSTED` when the cap is exceeded.
2. **Gate the safety check on a bounded retry count** or remove it for subscriptions that have never received any data (i.e., skip the unthrottled gap-fill when `topicContext.getLast() == null` and no messages have arrived).
3. **Rate-limit new subscription establishment** at the Netty/gRPC layer (e.g., per source IP).
4. Consider replacing `getResultList().stream()` with a true streaming/cursor approach to reduce per-query memory pressure.
