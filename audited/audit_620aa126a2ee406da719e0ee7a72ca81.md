### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `/account/balance` Flood in Rosetta AccountBalance()

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts unauthenticated requests and delegates to `RetrieveBalanceAtBlock()`, which issues 4 sequential database queries per request, each acquiring a connection from a shared pool capped at 100 open connections. There is no application-level rate limiting or concurrency control in the Rosetta Go service, and the Traefik middleware chain (which defines per-IP in-flight and rate limits) is disabled by default via `global.middleware: false`. A flood of concurrent requests from an unprivileged attacker exhausts the pool, causing all subsequent requests to block indefinitely until the statement timeout expires, rendering the Rosetta service unresponsive.

### Finding Description

**Exact code path:**

`AccountBalance()` at `rosetta/app/services/account_service.go:41-79` performs no authentication or concurrency check. It calls `RetrieveBalanceAtBlock()` at `rosetta/app/persistence/account.go:161-223`, which issues the following sequential DB queries, each calling `ar.dbClient.GetDbWithContext(ctx)` to acquire a pool connection:

1. `getCryptoEntity()` (line 167) â€” executes `selectCryptoEntityById` / `selectCryptoEntityByAlias` (UNION across `entity` + `entity_history`)
2. `getLatestBalanceSnapshot()` (lines 192-196) â€” executes `selectPreviousPartitionLowerBound` then `latestBalanceBeforeConsensus` (correlated subquery across `account_balance`)
3. `getBalanceChange()` (lines 201-208) â€” executes `balanceChangeBetween` (aggregation over `crypto_transfer`)

The pool is configured in `rosetta/app/db/db.go:31-33`:
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default 20
sqlDb.SetConnMaxLifetime(...)                              // default 30 min
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default 100
```
Default `maxOpenConnections` is 100 and `statementTimeout` is 20 seconds, per `docs/configuration.md:656-660`.

The HTTP server in `rosetta/main.go:220-227` uses Go's default `net/http.Server` with no `MaxConcurrentStreams` or goroutine cap â€” it spawns one goroutine per request without bound.

**Why existing checks fail:**

The only defined protection is the Traefik middleware chain in `charts/hedera-mirror-rosetta/values.yaml:149-166` (`inFlightReq: amount: 5` per source IP, `rateLimit: average: 10` per requestHost). However, this middleware is **never deployed by default**: the template at `charts/hedera-mirror-rosetta/templates/middleware.yaml:3` gates on `{{ if and .Values.global.middleware .Values.middleware }}`, and `global.middleware` defaults to `false` at `charts/hedera-mirror-rosetta/values.yaml:95`. In a default deployment, no Traefik middleware is created or attached to the ingress route.

Even when enabled, the per-IP limit of 5 in-flight requests is trivially bypassed by distributing requests across multiple source IPs or using a botnet. The Rosetta Go application itself contains zero rate-limiting or semaphore logic â€” unlike the `web3` Java service which has `ThrottleConfiguration` / `ThrottleManagerImpl`, the Rosetta service has no equivalent.

### Impact Explanation
When the 100-connection pool is saturated, `database/sql` queues all new connection acquisition attempts. With `statementTimeout: 20s`, each of the 4 queries per request can hold a connection for up to 20 seconds. Approximately 25 concurrent long-running requests (each holding ~4 connections) fully exhaust the pool. All subsequent legitimate requests block waiting for a connection, causing the Rosetta service to become unresponsive. Because the Go HTTP server spawns unbounded goroutines, memory pressure from thousands of blocked goroutines can additionally cause OOM. All deployed Rosetta instances (HPA min 1, max 10 per `charts/hedera-mirror-rosetta/values.yaml:100-110`) are equally vulnerable, so the attack can be replicated across all replicas simultaneously, taking down 100% of Rosetta processing capacity.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond knowing the Rosetta API is public, and no exploit tooling â€” only an HTTP client capable of sending concurrent POST requests. Valid account addresses are publicly enumerable from the Hedera ledger. The queries (`latestBalanceBeforeConsensus` with correlated subqueries over `account_balance`, `balanceChangeBetween` over `crypto_transfer`) are non-trivial and will be slow on a large dataset, maximizing connection hold time. The attack is repeatable and can be sustained indefinitely. Default deployments without explicitly setting `global.middleware: true` have zero protection.

### Recommendation
1. **Application-level concurrency limit**: Add a semaphore (e.g., `golang.org/x/sync/semaphore`) in `AccountBalance()` or at the HTTP handler layer to cap concurrent in-flight DB-bound requests, analogous to the `inFlightReq` middleware.
2. **Enable middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml:95`, or enforce the middleware unconditionally in the ingress template.
3. **Per-IP in-flight limit at application layer**: Implement IP-based concurrency tracking inside the Go service so protection is not dependent on external infrastructure.
4. **Reduce `maxOpenConnections`** and add a connection-wait timeout so exhausted-pool errors are returned quickly (HTTP 503) rather than blocking goroutines indefinitely.
5. **Query result caching**: Cache `RetrieveLatest()` and `RetrieveBalanceAtBlock()` results for a short TTL to reduce DB load from repeated identical queries.
