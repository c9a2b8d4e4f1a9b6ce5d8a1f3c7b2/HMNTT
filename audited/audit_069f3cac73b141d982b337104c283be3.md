### Title
Unauthenticated Access to Opcode Tracer Endpoint Exposes Full EVM Internal State of Failed Transactions

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` performs no authentication or authorization check, allowing any unprivileged external user to trigger a full EVM re-execution of any historical transaction — including failed ones — and receive a detailed opcode trace containing per-step stack, memory, storage, and revert reasons. This exposes internal EVM state that is not present in the original transaction receipt.

### Finding Description
**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–65) performs only three checks before delegating to `opcodeService.processOpcodeCall()`:
1. `properties.isEnabled()` — a feature flag, not an auth check
2. `validateAcceptEncodingHeader()` — requires `Accept-Encoding: gzip`
3. `throttleManager.throttleOpcodeRequest()` — rate limiting only

There is no `@PreAuthorize`, `@Secured`, `@RolesAllowed`, Spring Security filter, API key check, or any other authentication/authorization mechanism. A grep across all `web3/**/*.java` files confirms zero Spring Security annotations exist anywhere in the web3 module.

**No transaction-success filter in `OpcodeServiceImpl`:**

`buildCallServiceParameters()` (lines 76–118) fetches the transaction by hash or ID from the database and proceeds unconditionally — there is no check on `transaction.getResult()`, `contractResult.getTransactionResult()`, or any success/failure status. The service will re-execute any stored transaction regardless of its original outcome.

**Full internal state returned for failed transactions:**

`buildOpcodesResponse()` (lines 120–150) sets `failed(txnResult == null || !txnResult.isSuccessful())` but still returns the complete opcode list. `OpcodeActionTracer.tracePrecompileResult()` (lines 95–115) records the `revertReason` per opcode entry. The `OpcodesControllerTest.unsuccessfulCall()` test (lines 354–376) explicitly confirms that a `CONTRACT_REVERT_EXECUTED` transaction returns HTTP 200 with the full trace including a REVERT opcode carrying its reason bytes.

**CORS is fully open:** `GenericControllerAdvice` maps `allowedOrigins("*")` for all `/api/v1/contracts/**` routes, meaning any browser origin can issue this request.

### Impact Explanation
The original transaction receipt exposes only the final `error_message` and ABI-encoded `call_result`. The opcode trace additionally exposes:
- Full EVM **stack** contents at every single opcode step
- Full EVM **memory** contents at every single opcode step
- **Storage** slot reads and writes at every step
- **Per-opcode revert reasons** (not just the final revert)
- Internal call depth and structure

This allows an attacker to reverse-engineer contract business logic, discover internal state at the time of failure, extract sensitive values that passed through memory or the stack (e.g., private parameters, intermediate computation results), and map the exact execution path that led to the revert — information that the contract developer never intended to be public.

### Likelihood Explanation
The endpoint is disabled by default (`enabled = false` in `OpcodesProperties`). However, any operator who enables it for legitimate debugging purposes immediately exposes it to the entire internet with no authentication barrier. The exploit requires only: knowing the endpoint exists, setting `Accept-Encoding: gzip`, and supplying any valid transaction hash — all trivially achievable by any external user. The attack is fully repeatable and requires no special privileges, credentials, or on-chain interaction.

### Recommendation
1. **Add authentication/authorization**: Protect the endpoint with a Spring Security rule (e.g., `@PreAuthorize("hasRole('ADMIN')")` or an API-key filter) so only authorized operators can invoke it.
2. **Optionally filter failed transactions**: Add a check in `OpcodeServiceImpl.buildCallServiceParameters()` or `processOpcodeCall()` that reads `contractResult.getTransactionResult()` and rejects re-execution of failed transactions if the operator does not explicitly opt in.
3. **Restrict CORS**: Change `allowedOrigins("*")` to a specific trusted origin for this endpoint rather than the wildcard applied to all `/api/v1/contracts/**` routes.

### Proof of Concept
**Preconditions:** `hiero.mirror.web3.opcode.tracer.enabled=true` (operator has enabled the feature).

**Steps:**
1. Obtain the hash of any failed contract transaction (e.g., from the public `/api/v1/contracts/results` endpoint, which shows `"result": "CONTRACT_REVERT_EXECUTED"`).
2. Send the following request as an unprivileged user with no credentials:
```
GET /api/v1/contracts/results/0x<failed_tx_hash>/opcodes?stack=true&memory=true&storage=true
Accept-Encoding: gzip
```
3. **Result:** HTTP 200 response containing the full `OpcodesResponse` with `"failed": true` and a complete opcode-by-opcode trace including `stack`, `memory`, `storage`, and `reason` fields — exposing all internal EVM state of the failed transaction that was never visible in the original receipt.     @GetMapping(value = "/{transactionIdOrHash}/opcodes")
    OpcodesResponse getContractOpcodes(
            @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
            @RequestParam(required = false, defaultValue = "true") boolean stack,
            @RequestParam(required = false, defaultValue = "false") boolean memory,
            @RequestParam(required = false, defaultValue = "false") boolean storage,
            @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
        if (properties.isEnabled()) {
            validateAcceptEncodingHeader(acceptEncoding);
            throttleManager.throttleOpcodeRequest();

            final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
            return opcodeService.processOpcodeCall(request);
        }

        throw new ResponseStatusException(HttpStatus.NOT_FOUND);
    }

    @Override
    public void tracePrecompileResult(@NonNull MessageFrame frame, @NonNull ContractActionType type) {
        final var context = ContractCallContext.get();
        final var gasCost = context.getOpcodeContext().getGasRemaining() - frame.getRemainingGas();

        final var frameRevertReason = frame.getRevertReason().orElse(null);
        final var revertReason = isCallToSystemContracts(frame, systemContracts)
                ? getRevertReasonFromContractActions(context)
                : (frameRevertReason != null ? frameRevertReason.toHexString() : null);

        context.getOpcodeContext()
                .addOpcodes(createOpcode(
                        frame,
                        gasCost,
                        revertReason,
                        Collections.emptyList(),
                        Collections.emptyList(),
                        Collections.emptyMap()));

        context.getOpcodeContext().setGasRemaining(frame.getRemainingGas());
    }
