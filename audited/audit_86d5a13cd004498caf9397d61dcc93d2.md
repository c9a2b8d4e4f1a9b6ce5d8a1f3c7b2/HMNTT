### Title
Deleted Entity Disclosure via Long-Form EVM Address in `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when the supplied EVM address has its first 12 bytes set to zero (the "long-form" path), the code falls through to `entityRepository.findById()` — a standard Spring Data `CrudRepository` method that carries no `deleted` filter. Both `findByAlias` and `findByEvmAddress` explicitly include `and deleted is not true` in their SQL, but `findById` does not. Any unauthenticated caller can therefore retrieve the full entity record (keys, balance, memo, staking info, etc.) of a deleted account by crafting a 20-byte hex EVM address whose last 8 bytes encode the target entity's numeric ID.

### Finding Description
**Exact code path:**

`EntityServiceImpl.java`, `getByEvmAddressAndType()`, lines 34–41: [1](#0-0) 

When `buffer.getInt() == 0 && buffer.getLong() == 0` (first 12 bytes are zero), the code calls:
```java
return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
```

`findById` is the inherited `CrudRepository<Entity, Long>.findById()` — it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

**Contrast with the other two repository methods:** [2](#0-1) 

Both `findByAlias` and `findByEvmAddress` append `and deleted is not true`. `findById` does not.

**Exploit flow:**
1. Attacker learns (or enumerates) the numeric entity ID of a deleted account (entity IDs are sequential and publicly observable on-chain).
2. Attacker constructs a 20-byte hex EVM address: `0x` + 24 zero hex chars + 16 hex chars encoding the target entity ID as a big-endian `long`.
3. Attacker sends a GraphQL query to the public `/graphql` endpoint:
   ```graphql
   query {
     account(input: { evmAddress: "0x000000000000000000000000<entity_id_hex>" }) {
       entityId deleted key memo balance alias
     }
   }
   ```
4. `getByEvmAddressAndType` detects the zero prefix, calls `findById(entityId)`, and returns the deleted entity. The only post-fetch filter is `e.getType() == type`, which passes for a deleted ACCOUNT.
5. The response includes `deleted: true` alongside all other fields — confirming the entity is deleted but still fully disclosing its data.

### Impact Explanation
The `Account` GraphQL type exposes: [3](#0-2) 

Sensitive fields returned for a deleted account include `key` (the admin key structure), `balance` (at deletion time), `memo`, `alias`, `autoRenewAccount`, `obtainer`, and staking metadata. The `key` field is particularly sensitive as it reveals the cryptographic signing requirements of an account that the network considers non-existent. This constitutes an information-disclosure vulnerability: data that the protocol treats as logically deleted is fully readable by any unauthenticated user.

### Likelihood Explanation
- No authentication or special privilege is required; the GraphQL endpoint is public.
- Entity IDs are sequential integers, trivially enumerable.
- The crafted EVM address format is documented Hedera behavior (long-form `0.0.<num>` encoding), so the technique is well-known.
- The exploit is a single HTTP request, fully repeatable, and requires no on-chain interaction.

### Recommendation
Add a `deleted` check in the `findById` branch of `getByEvmAddressAndType()`, consistent with the other two lookup paths:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
        .filter(e -> e.getType() == type && !Boolean.TRUE.equals(e.getDeleted()));
}
```

Alternatively, add a dedicated repository method analogous to the existing ones:
```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);
```
and use it in both `getByIdAndType` and the long-form branch of `getByEvmAddressAndType`. [4](#0-3) 

Note that `getByIdAndType` (line 25) has the same omission and should be fixed simultaneously.

### Proof of Concept
**Precondition:** A deleted account exists with entity ID `1234` (i.e., `deleted = true` in the `entity` table).

**Step 1 — Encode the long-form EVM address:**
```
entity_id = 1234  →  hex = 00000000000004d2
EVM address = 0x000000000000000000000000 + 00000000000004d2
            = 0x00000000000000000000000000000000000004d2
```

**Step 2 — Send the GraphQL query:**
```graphql
query {
  account(input: { evmAddress: "0x00000000000000000000000000000000000004d2" }) {
    entityId
    deleted
    key
    memo
    balance
    alias
  }
}
```

**Step 3 — Observe the response:**
```json
{
  "data": {
    "account": {
      "entityId": "0.0.1234",
      "deleted": true,
      "key": { ... },
      "memo": "...",
      "balance": 0,
      "alias": "..."
    }
  }
}
```

The deleted account's full record is returned. No credentials required.

        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
