### Title
Unbounded Database Query Amplification via Unauthenticated Topic Lookup with No Rate Limiting or Caching

### Summary
`TopicRepository` in the `rest-java` module is a bare `CrudRepository` with no `@Cacheable` annotation and no application-level caching. The `rest-java` module contains zero rate-limiting infrastructure. Any unauthenticated external user can flood `GET /api/v1/topics/{id}` with concurrent requests for the same topic ID, causing every request to independently execute a database query, exhausting the connection pool and degrading service for all users.

### Finding Description
**Code path:**
- `TopicController.getTopic()` (`rest-java/.../controller/TopicController.java`, lines 31–37) receives unauthenticated GET requests and calls `topicService.findById(id.id())`.
- `TopicServiceImpl.findById()` (`rest-java/.../service/TopicServiceImpl.java`, line 20) calls `topicRepository.findById(id.getId())` directly with no guard.
- `TopicRepository` (`rest-java/.../repository/TopicRepository.java`, lines 1–8) is a plain `CrudRepository<Topic, Long>` with no `@Cacheable`, no custom query, and no caching decorator of any kind.

**Root cause — three compounding absences:**

1. **No application-level cache.** A grep across all `rest-java/src/main/java/**/*.java` finds zero occurrences of `@Cacheable`, `CacheManager`, or `@EnableCaching`. Compare this to `grpc/EntityRepository` (which has `@Cacheable` with a 24-hour Caffeine cache) or `web3/EntityRepository` (which has multi-tier `@Caching` annotations). The `rest-java` `TopicRepository` has none of this.

2. **No rate limiting.** The `rest-java` module's entire filter chain consists of `ShallowEtagHeaderFilter`, `LoggingFilter`, and `MetricsFilter` (`RestJavaConfiguration.java`, `LoggingFilter.java`). The bucket4j throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is not wired into `rest-java` at all.

3. **No request coalescing.** There is no mechanism to deduplicate concurrent in-flight requests for the same topic ID. N concurrent requests for topic `0.0.1` produce N independent JDBC calls, each acquiring a connection from the pool.

**Exploit flow:**
An attacker sends a sustained stream of `GET /api/v1/topics/1` requests. Each request independently traverses `TopicController → TopicServiceImpl → TopicRepository → JDBC → PostgreSQL`. Because `TopicController.getTopic()` also calls `entityService.findById()` and `customFeeService.findById()` (line 34–35), each HTTP request actually triggers **three** independent DB round-trips, not one.

### Impact Explanation
With a connection pool of typical size (e.g., 10–50 connections), an attacker sustaining ~50–200 concurrent requests can saturate the pool. New requests from legitimate users queue indefinitely or receive connection-timeout errors. Because the `rest-java` service shares its database with the importer and potentially other services, pool exhaustion can cascade. The impact is service degradation (griefing/DoS) with no economic damage to the Hedera network itself, consistent with the stated scope classification.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no rate limit are required. Any attacker with a commodity machine and a tool like `wrk`, `hey`, or `ab` can sustain thousands of requests per second. The attack is trivially repeatable, requires no special knowledge beyond the public API path, and leaves no persistent damage — making it low-cost and low-risk for the attacker to run continuously.

### Recommendation
1. **Add `@Cacheable` to `TopicRepository.findById()`** (or override it in the interface) using a short-lived Caffeine cache (e.g., `expireAfterWrite=5s, maximumSize=10000`), consistent with the pattern already used in `web3/EntityRepository` and `grpc/EntityRepository`.
2. **Add rate limiting to the `rest-java` module.** Port or share the bucket4j `ThrottleConfiguration` pattern from `web3` and apply a per-IP or global RPS limit to `/api/v1/**` endpoints via a servlet filter.
3. **Consider request coalescing** at the service layer (e.g., using Caffeine's `AsyncLoadingCache`) so concurrent requests for the same ID share a single in-flight DB query.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk)
# Target: a running rest-java instance

wrk -t 10 -c 200 -d 60s http://<host>/api/v1/topics/1
```
Expected result: DB connection pool saturates within seconds; legitimate requests to other endpoints begin timing out or returning 500 errors. No credentials, tokens, or special headers required.
