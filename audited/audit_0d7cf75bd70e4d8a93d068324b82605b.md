### Title
Unauthenticated `/block` Endpoint Exhausts DB Connection Pool via Unbounded Concurrent Requests

### Summary
The `Block()` function in `rosetta/app/services/block_service.go` issues multiple synchronous database queries per request with no application-level rate limiting or concurrency control. The Traefik middleware that defines `inFlightReq` and `rateLimit` protections is disabled by default (`global.middleware: false`). Because `MaxOpenConnections` defaults to `0` (unlimited in Go's `database/sql`) when not explicitly configured, an unauthenticated attacker can flood the `/block` endpoint with concurrent requests, exhausting the shared PostgreSQL connection budget and starving the mirror node importer of DB connections, halting transaction ingestion.

### Finding Description

**Exact code path:**

`Block()` at `rosetta/app/services/block_service.go:51-70` issues three sequential DB-touching calls per request:

1. `s.RetrieveBlock()` ŌåÆ `blockRepo.FindByIndex/FindByHash/RetrieveLatest` ŌåÆ raw SQL against `record_file` table
2. `s.FindBetween()` ŌåÆ `transactionRepo.FindBetween()` ŌåÆ full transaction scan between two nanosecond timestamps
3. `s.updateOperationAccountAlias()` ŌåÆ `accountRepo.GetAccountAlias()` per operation (cache miss path)

Each call goes through `GetDbWithContext()` in `rosetta/app/db/client.go:22-37`, which acquires a connection from the GORM/`database/sql` pool.

**Root cause ŌĆö unlimited pool:**

`rosetta/app/db/db.go:33`:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)
```
`Pool.MaxOpenConnections` is an `int` field in `rosetta/app/config/types.go:80` with no default value set in any discovered production config file. In Go, an unset `int` is `0`; `database/sql` interprets `SetMaxOpenConns(0)` as **unlimited**. The service will therefore open as many PostgreSQL connections as goroutines are waiting.

**Middleware disabled by default:**

`charts/hedera-mirror-rosetta/templates/middleware.yaml:3`:
```
{{ if and .Values.global.middleware .Values.middleware -}}
```
`charts/hedera-mirror-rosetta/values.yaml:95`:
```yaml
global:
  middleware: false
```
Because `global.middleware` is `false`, the entire Traefik middleware chain ŌĆö including `inFlightReq: amount: 5` and `rateLimit: average: 10` ŌĆö is **never deployed**. The application-level middleware stack in `main.go:217-219` only adds metrics, tracing, and CORS; no concurrency or rate limiting is applied.

**No authentication:** The `/block` endpoint requires no credentials.

### Impact Explanation

PostgreSQL is configured with `max_connections: 200` (`charts/hedera-mirror/values.yaml:210`). The rosetta service (`mirror_node` user) and the importer (`mirror_importer` user) share this PostgreSQL instance. With unlimited `MaxOpenConnections`, a flood of concurrent `/block` requests causes the rosetta service to open hundreds of connections. Once the PostgreSQL `max_connections` ceiling is hit, the importer cannot acquire connections to write newly ingested record files, halting transaction confirmation processing. Even below the hard ceiling, connection contention and lock pressure on `record_file` and `transaction` tables degrades importer throughput to zero under sustained load. This satisfies the "total network shutdown" criterion: the mirror node stops confirming new transactions.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no prior knowledge beyond the public Rosetta API specification. A single attacker with a modest number of concurrent HTTP clients (e.g., `ab -c 500 -n 100000 -p block.json -T application/json http://target/block`) can sustain the flood indefinitely. The endpoint is publicly documented and reachable via the ingress path `/rosetta/block`. The attack is trivially repeatable and requires no exploit tooling.

### Recommendation

1. **Set `MaxOpenConnections` to a non-zero value** in the production rosetta configuration (e.g., `pool.maxOpenConnections: 20`) so `database/sql` enforces a hard cap.
2. **Enable the Traefik middleware by default** by changing `global.middleware: false` to `true` in the Helm chart defaults, activating the existing `inFlightReq` (5 concurrent) and `rateLimit` (10 req/s) definitions.
3. **Add application-level concurrency limiting** (e.g., a semaphore or `golang.org/x/time/rate` limiter) inside `Block()` or as an HTTP middleware in `main.go`, independent of the infrastructure layer.
4. **Set a `statementTimeout`** in the rosetta DB config so long-running `FindBetween` queries are killed rather than holding connections indefinitely.

### Proof of Concept

```bash
# 1. Craft a valid /block request body
cat > block.json <<'EOF'
{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{}}
EOF

# 2. Flood with 500 concurrent connections, no auth required
ab -c 500 -n 999999 \
   -p block.json \
   -T "application/json" \
   http://<rosetta-host>/block

# 3. Simultaneously monitor PostgreSQL connections
psql -U postgres -c "SELECT count(*) FROM pg_stat_activity WHERE datname='mirror_node';"
# Expected: count approaches max_connections (200)

# 4. Verify importer stalls
psql -U postgres -c "SELECT count(*) FROM pg_stat_activity WHERE usename='mirror_importer' AND state='active';"
# Expected: 0 ŌĆö importer cannot acquire connections; no new record_files are written
```
