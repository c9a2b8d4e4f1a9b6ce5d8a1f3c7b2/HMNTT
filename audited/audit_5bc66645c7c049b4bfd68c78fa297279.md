### Title
Unauthenticated Repeated DB Reads via Uncached, Unthrottled `TopicServiceImpl.findById()`

### Summary
`TopicServiceImpl.findById()` delegates directly to a plain `CrudRepository.findById()` with no application-level cache and no rate limiting on the rest-java endpoint. Any unauthenticated caller can flood `GET /api/v1/topics/{id}` with arbitrary request volume, causing repeated identical DB queries that exhaust the HikariCP connection pool and consume DB CPU/memory for every request.

### Finding Description
**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/TopicServiceImpl.java`, line 20; `rest-java/src/main/java/org/hiero/mirror/restjava/repository/TopicRepository.java`, lines 1–8; `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 31–37.

**Root cause:** `TopicRepository` extends `CrudRepository<Topic, Long>` with no `@Cacheable` annotation. The grpc module's `EntityRepository` (same codebase) applies `@Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, ...)` on its `findById`, demonstrating the pattern is known and intentionally applied elsewhere but omitted here. The rest-java module contains no rate-limiting filter, no bucket4j bean, and no per-IP or per-resource throttle — confirmed by a full grep of `rest-java/**/*.java` for `RateLimiter`, `throttle`, `bucket4j`, and `Cacheable`, all returning zero matches for the topics path.

**Exploit flow:**
1. Attacker identifies any valid topic ID (public information on Hedera mainnet/testnet).
2. Attacker sends a high-volume stream of `GET /api/v1/topics/{id}` requests (no authentication required).
3. Each request enters `TopicController.getTopic()`, which calls `topicService.findById()`, `entityService.findById()`, and `customFeeService.findById()` — three separate DB round-trips per HTTP request, none cached.
4. The HikariCP connection pool (monitored via `hikaricp_connections_active`) is progressively exhausted; other endpoints sharing the same pool degrade or time out.

**Existing checks reviewed:**
- The ingress Traefik middleware (`charts/hedera-mirror-rest-java/values.yaml`, lines 151–155) applies only a circuit breaker on error ratios and a retry policy — no rate limiting.
- The `web3` module's `ThrottleManagerImpl` and `ThrottleConfiguration` are scoped to `ContractCallRequest` and are not wired into rest-java at all.
- No `@Cacheable` exists anywhere in `rest-java/**/*.java`.

### Impact Explanation
An attacker can degrade or deny service to all users of the rest-java API by exhausting the shared DB connection pool. Because the topic lookup is a primary-key read, each query is cheap individually, but at scale (thousands of concurrent requests) the aggregate connection and CPU overhead is significant. The Prometheus alert `RestJavaHighDBConnections` fires at 75% pool utilization — an attacker can trigger this threshold and cause cascading failures across all rest-java endpoints, not just topics.

### Likelihood Explanation
Preconditions are minimal: no account, no token, no authentication. Valid topic IDs are publicly enumerable from the mirror node itself. The attack is trivially scriptable with any HTTP load tool (`ab`, `wrk`, `hey`). It is repeatable indefinitely and requires no special knowledge beyond the public API documentation.

### Recommendation
1. Add `@Cacheable` to `TopicRepository.findById()` (mirroring the pattern in `grpc/src/main/java/org/hiero/mirror/grpc/repository/EntityRepository.java`) with an appropriate TTL (e.g., `expireAfterWrite=30s, maximumSize=10000`).
2. Introduce a global request-rate-limiting filter in rest-java (analogous to `ThrottleManagerImpl` in web3), enforcing a per-IP or global RPS cap on all `/api/v1/topics/**` requests.
3. Consider adding a Traefik `rateLimit` middleware entry in `charts/hedera-mirror-rest-java/values.yaml` as a defense-in-depth layer at the ingress level.
