### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded `GET /api/v1/topics/{id}` Requests

### Summary
`TopicServiceImpl.findById()` issues an unconditional database query on every call with no rate limiting, no negative-result caching, and no request throttling in the `rest-java` module. An unprivileged attacker can flood `GET /api/v1/topics/{id}` with arbitrary non-existent topic IDs, causing the database connection pool to be saturated and degrading or denying service to legitimate users.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` (line 32–36) calls three service methods per request, each issuing its own DB query: [1](#0-0) 

The first call goes to `TopicServiceImpl.findById()`: [2](#0-1) 

`TopicRepository` is a plain Spring Data `CrudRepository` with no custom caching or query guards: [3](#0-2) 

**Root cause — no rate limiting in `rest-java`:**

The only servlet filters registered in the `rest-java` module are `LoggingFilter` (logs requests) and `MetricsFilter` (records metrics). Neither enforces any request rate limit: [4](#0-3) 

`WebMvcConfiguration` adds only argument resolvers and formatters — no throttle interceptor: [5](#0-4) 

**The throttling that does exist is in the `web3` module only**, scoped to `ContractCallRequest` and the `/api/v1/contracts/call` endpoint — it does not apply to `/api/v1/topics/{id}`: [6](#0-5) 

**No negative-result caching:** `TopicServiceImpl` carries no `@Cacheable` annotation. Every call for a non-existent ID hits the database and returns `EntityNotFoundException`. The test confirms this path is live: [7](#0-6) 

### Impact Explanation

Each request to `GET /api/v1/topics/{id}` with a non-existent ID causes at minimum one synchronous JDBC connection checkout and a `SELECT` against the `topic` table. Because `TopicController.getTopic()` also calls `entityService.findById()` and `customFeeService.findById()`, each HTTP request can consume up to three DB connections simultaneously. A sustained flood of concurrent requests exhausts the connection pool, causing all subsequent DB-backed endpoints (not just the topic endpoint) to queue or fail. This degrades or denies service for all legitimate users of the mirror node REST API — a griefing impact with no economic cost to the attacker.

### Likelihood Explanation

The endpoint is publicly reachable with no authentication. The attacker needs only an HTTP client and the ability to generate unique numeric IDs (e.g., sequential integers or random values). No credentials, tokens, or special knowledge are required. The attack is trivially scriptable (`ab`, `wrk`, `curl` in a loop) and fully repeatable. Because there is no per-IP or global RPS guard in `rest-java`, the attacker faces no application-level friction.

### Recommendation

1. **Add a global rate-limiting filter to `rest-java`** analogous to the `web3` `ThrottleManagerImpl`/`ThrottleConfiguration` using Bucket4j, applied to all `/api/v1/**` routes.
2. **Cache negative results** in `TopicServiceImpl.findById()` using Spring's `@Cacheable` with a short TTL (e.g., 5–30 s) so repeated misses for the same ID do not hit the DB.
3. **Limit DB connection pool size** and configure a connection acquisition timeout so pool exhaustion fails fast rather than queuing indefinitely.
4. **Deploy an external rate limiter** (e.g., nginx `limit_req`, API gateway) as a defence-in-depth layer in front of the service.

### Proof of Concept

```bash
# Flood with 10,000 requests for non-existent topic IDs using 100 concurrent connections
seq 1 10000 | xargs -P 100 -I{} \
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/topics/99999{}"

# Observe: legitimate requests begin timing out or returning 500
# as the DB connection pool is exhausted
curl -v "https://<mirror-node-host>/api/v1/topics/1"
# Expected during attack: connection timeout or HTTP 500
```

No privileged access, API key, or account is required. Any unprivileged external user can execute this.
