### Title
Unbounded Timestamp Range in Historical Hook Storage Query Enables DB Resource Exhaustion via `DISTINCT ON` Full Scan

### Summary
An unprivileged user can trigger the historical code path in `getHookStorageChange()` by supplying any non-empty timestamp parameter, an empty `keys` list, and arbitrarily wide key/timestamp bounds. This causes `findByKeyBetweenAndTimestampBetween()` to execute a `DISTINCT ON (key)` native SQL query that must scan every row matching the full key and timestamp range before applying the pagination `LIMIT`, enabling repeated requests to drive sustained DB CPU and I/O spikes well above 30%.

### Finding Description
**Code path:**

`HookServiceImpl.java` `getHookStorage()` (line 57–58) routes to `getHookStorageChange()` whenever `request.isHistorical()` is true, which is satisfied by any non-empty `timestamp` bound.

Inside `getHookStorageChange()` (lines 84–118):

```
keys = request.getKeys();               // empty → requestHasKeys = false
keysInRange = request.getKeysInRange(); // also empty

if (keysInRange.isEmpty() && requestHasKeys) { // FALSE — guard skipped
    return new HookStorageResult(ownerId, List.of());
}
// Falls through to the else branch unconditionally:
changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
    ownerId.getId(), hookId,
    request.getKeyLowerBound(), request.getKeyUpperBound(),   // attacker-controlled
    timestampLowerBound, timestampUpperBound,                 // attacker-controlled
    page);
```

The guard on line 93 only fires when `requestHasKeys` is `true` **and** `keysInRange` is empty. When `keys` is empty, `requestHasKeys` is `false`, so the guard is never entered and the wide-range query always executes.

The native SQL in `HookStorageChangeRepository.java` (lines 15–31):

```sql
select distinct on (key)
     owner_id, hook_id, key, value_written as "value", ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id  = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
```

PostgreSQL's `DISTINCT ON` requires the engine to sort and de-duplicate **all rows** satisfying the `WHERE` clause before the `LIMIT` from `Pageable` can be applied. There is no index-only shortcut when both the key range and the timestamp range are wide simultaneously; the planner must perform a full index/heap scan of every matching row for that `(owner_id, hook_id)` pair.

**Root cause:** No validation of timestamp range width (or key range width) exists anywhere in the request DTO, service, or controller layer for the historical path. The early-exit guard is logically bypassed when `keys` is empty.

### Impact Explanation
A mirror node stores the complete history of every hook storage write. For any hook with significant activity, the `hook_storage_change` table can contain millions of rows. A single request with `timestamp=gte:0&timestamp=lte:<MAX_LONG>` and no keys forces a full sequential scan of all rows for that hook, followed by an in-memory sort for `DISTINCT ON`. Repeated at modest concurrency (e.g., 5–10 parallel requests), this saturates PostgreSQL's I/O and CPU, directly impacting all other queries on the shared database instance. The `LIMIT 25` does not prevent the scan; it only limits the rows returned to the caller.

### Likelihood Explanation
The exploit requires no authentication, no special privileges, and no prior knowledge beyond a valid `ownerId`/`hookId` pair (both are publicly enumerable from the mirror node's own APIs). The attacker needs only to:
- Know or enumerate a `hookId` with historical data.
- Send HTTP requests with a timestamp range spanning the full epoch.
- Repeat at low-to-moderate rate.

This is trivially scriptable and repeatable. No rate limiting is present in the service layer.

### Recommendation
1. **Enforce a maximum timestamp range width** in `HookStorageRequest` or at the controller level (e.g., reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum such as 24 hours).
2. **Require at least one key** when the historical path is used, or enforce a narrow default key range, so the `DISTINCT ON` scan is bounded.
3. **Add a composite index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` so PostgreSQL can satisfy `DISTINCT ON (key) ORDER BY key, consensus_timestamp DESC` via an index scan with early termination per key group, reducing the need for a full sort.
4. **Apply rate limiting** at the API gateway or controller level for historical storage queries.

### Proof of Concept

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
