### Title
Unauthenticated GraphQL `account()` Endpoint Lacks Rate Limiting, Enabling DB Connection Pool Exhaustion DoS

### Summary
The `account()` query in `AccountController.java` is publicly accessible with no rate limiting at the GraphQL layer. Each valid request triggers a native SQL query against the database. An unprivileged attacker can flood the endpoint with thousands of concurrent valid `evmAddress` queries, exhausting the HikariCP database connection pool and rendering the GraphQL service unavailable.

### Finding Description

**Code path:**

`AccountController.java` line 33–54: The `account()` method accepts any valid `AccountInput` and unconditionally calls `entityService.getByEvmAddressAndType()`. [1](#0-0) 

`EntityServiceImpl.java` line 34–41: `getByEvmAddressAndType()` always issues a native SQL query via `entityRepository.findByEvmAddress()`. [2](#0-1) 

`EntityRepository.java` line 16–17: The native query `SELECT * FROM entity WHERE evm_address = ?1 AND deleted IS NOT TRUE` runs on every request. [3](#0-2) 

**Root cause — no rate limiting in the graphql module:**

`GraphQlConfiguration.java` configures `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, plus parser/JSON size limits. These only prevent structurally complex or oversized queries; they impose **zero limit on the rate of simple, valid queries**. [4](#0-3) 

The `CachedPreparsedDocumentProvider` caches only the *parsed document structure*, not query results. Requests with different `evmAddress` values share the same cached parse tree but each still hits the database. [5](#0-4) 

The `web3` module has a full `ThrottleManagerImpl` with bucket4j rate limiting, but **no equivalent exists in the `graphql` module**. [6](#0-5) 

**Why existing checks fail:**

- Complexity/depth limits block nested query abuse but not high-volume flat queries.
- The `@Pattern` regex on `evmAddress` (`^(0x)?[a-fA-F0-9]{40}$`) is a format check, not a rate check.
- No Spring Security filter, servlet filter, or interceptor enforces per-IP or global request rate limits on the `/graphql/alpha` endpoint. [7](#0-6) 

### Impact Explanation
Flooding the endpoint exhausts the HikariCP connection pool. The Prometheus alert `GraphQLHighDBConnections` (threshold: >75% pool utilization) confirms this is a recognized risk, but there is no enforcement mechanism to prevent it. [8](#0-7) 

Once the pool is saturated, all GraphQL queries queue or fail, making the mirror node's GraphQL API completely unavailable to legitimate users. This constitutes a denial-of-service against the mirror node's query processing capability.

### Likelihood Explanation
No authentication, API key, or network-level credential is required. Any external user can send HTTP POST requests to `/graphql/alpha`. The attack requires only a standard HTTP client and the ability to generate valid 40-character hex strings. It is trivially repeatable and automatable from a single host or distributed botnet.

### Recommendation
1. Add a global request rate limiter (e.g., bucket4j, Resilience4j, or Spring Cloud Gateway) to the GraphQL module, mirroring the pattern already used in `ThrottleConfiguration.java` in the `web3` module.
2. Add per-IP rate limiting at the load balancer or ingress layer (the design doc for contract log subscriptions already notes "We should implement rate limiting in the load balancer" — this applies equally here).
3. Consider adding a result cache (not just a parse cache) for `findByEvmAddress` queries to reduce DB load under repeated identical lookups.

### Proof of Concept
```bash
# Send 5000 concurrent valid account queries with distinct evmAddress values
for i in $(seq 1 5000); do
  addr=$(printf '%040x' $i)
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {evmAddress: \\\"$addr\\\"}) { id } }\"}" &
done
wait
# Result: HikariCP pool exhausted; subsequent legitimate queries time out or fail
```

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
