### Title
Unauthenticated DB Query Flood via `isValidAccount()` with No Rate Limiting or Result Caching

### Summary
The `isValidAccount()` function in `rest/service/entityService.js` executes a direct database query (`entityExistenceQuery`) on every invocation with no per-result caching and no rate limiting on the calling endpoints. An unprivileged attacker can flood `/accounts/:id/tokens` or `/accounts/:id/rewards` with requests using distinct accountIds, causing each request to issue a fresh DB query, exhausting the database connection pool and degrading service for legitimate users.

### Finding Description

**Exact code path:**

`isValidAccount()` at [1](#0-0)  executes `entityExistenceQuery` [2](#0-1)  — a `SELECT type FROM entity WHERE id = $1` — unconditionally on every call, with no memoization or cache lookup.

It is called unconditionally in two controller handlers:

- `tokenController.getTokenRelationships` at [3](#0-2) 
- `accountController.listStakingRewardsByAccountId` at [4](#0-3) 

**Root cause — failed assumptions:**

1. **No rate limiting.** The middleware stack in `server.js` includes `authHandler`, `requestLogger`, optional metrics, and optional response cache — but no rate-limiting middleware at any layer. [5](#0-4) 

2. **Response cache is optional and trivially bypassed.** The Redis response cache is only active when both `config.cache.response.enabled` and `config.redis.enabled` are true. [6](#0-5)  Even when enabled, the cache key is an MD5 of `req.originalUrl`. [7](#0-6)  An attacker using a different numeric accountId per request (e.g., `0.0.1`, `0.0.2`, …, `0.0.N`) generates a unique cache key each time, guaranteeing a cache miss and a fresh DB query on every request.

3. **No caching inside `isValidAccount()` itself.** The function has no in-process cache (no Map, no LRU, no Redis lookup). [1](#0-0) 

**Exploit flow:**

1. Attacker sends `GET /api/v1/accounts/0.0.{N}/tokens` in a tight loop, incrementing `N` each request.
2. For a plain numeric accountId, `getEncodedId()` returns the encoded id without a DB query (the EVM-address branch is not taken). [8](#0-7) 
3. `isValidAccount(accountId)` is called immediately after, issuing `SELECT type FROM entity WHERE id = $1` to the DB. [9](#0-8) 
4. Each unique `N` misses the response cache, so no request is short-circuited.
5. With sufficient request rate, all DB connections in the pool are occupied by these cheap-but-concurrent queries, starving legitimate requests.

### Impact Explanation
The database connection pool is a finite shared resource. Even though `SELECT type FROM entity WHERE id = $1` is a cheap primary-key lookup, holding a connection for each in-flight request means that a sustained flood of requests with distinct accountIds can exhaust the pool. Once the pool is exhausted, all other API endpoints that require a DB connection (transactions, balances, contracts, etc.) begin queuing or timing out, constituting a denial of service for all legitimate users. The impact is amplified because two endpoints expose this path (`/tokens` and `/rewards`), doubling the attack surface.

### Likelihood Explanation
No authentication or API key is required — the endpoints are fully public. No rate limiting exists at the application layer. The attacker needs only an HTTP client capable of sending concurrent GET requests with incrementing path parameters, which is trivially achievable with tools like `ab`, `wrk`, or a simple script. The attack is repeatable indefinitely and requires no special knowledge of the system beyond the public API documentation.

### Recommendation
1. **Add rate limiting middleware** (e.g., `express-rate-limit` or an upstream API gateway rule) applied globally or specifically to the `/accounts/:id/tokens` and `/accounts/:id/rewards` routes, keyed by IP or API key.
2. **Cache `isValidAccount()` results** in-process (e.g., a short-TTL LRU cache keyed on `accountId`) so repeated lookups for the same id do not hit the DB. Even a 5-second TTL would absorb burst floods.
3. **Enforce DB connection pool limits with request queuing timeouts** so that pool exhaustion results in fast 503 responses rather than cascading latency.

### Proof of Concept
```bash
# Flood /accounts/:id/tokens with distinct accountIds (no auth required)
for i in $(seq 1 100000); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.${i}/tokens" &
  # throttle parallelism to connection pool size to maximize pool exhaustion
  if (( i % 200 == 0 )); then wait; fi
done

# Simultaneously, observe legitimate requests timing out:
curl -v "http://<mirror-node-host>/api/v1/transactions"
# Expected: connection timeout or 503 once pool is exhausted
```

Each iteration issues a unique URL, bypassing the Redis response cache, and forces `SELECT type FROM entity WHERE id = $1` for every request. With the pool saturated, legitimate queries queue indefinitely.
