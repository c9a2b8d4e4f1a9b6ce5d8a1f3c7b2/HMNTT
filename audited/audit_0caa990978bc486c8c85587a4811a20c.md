### Title
Unbounded Concurrent Parse Operations via Caffeine AsyncCache Cache-Miss Flood in GraphQL Endpoint (No Rate Limiting)

### Summary
The `CachedPreparsedDocumentProvider.getDocumentAsync()` uses Caffeine's `AsyncCache.get()` which, on a cache miss, submits `parseAndValidateFunction.apply()` to the JVM's common ForkJoinPool for each distinct query string. The GraphQL module has no rate limiting whatsoever (unlike the `web3` module which has `ThrottleConfiguration`). An unprivileged attacker can flood the endpoint with high-volume concurrent requests using distinct query strings, causing unbounded concurrent parse+validate tasks to saturate the ForkJoinPool and drive CPU consumption well above 30%.

### Finding Description

**Exact code path:**

`CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

The cache is configured with `maximumSize=1000, expireAfterWrite=1h`: [2](#0-1) 

**Root cause:** Caffeine's `AsyncCache.get(key, mappingFunction)` deduplicates concurrent requests for the **same** key, but for **distinct** keys it fires a separate async computation per key. The mapping function (`parseAndValidateFunction.apply()`) — which performs full GraphQL lexing, parsing, and validation — is submitted to the ForkJoinPool for every cache miss. With no rate limiting on the GraphQL endpoint, an attacker can continuously submit distinct query strings (trivially varied by adding/removing whitespace, aliases, or field ordering), each generating a new cache miss and a new ForkJoinPool task.

**Why existing checks are insufficient:**

- Parser limits (`maxCharacters=10000`, `maxTokens=1000`, `maxRuleDepth=100`) bound the **cost per query** but do not limit the **number of concurrent parse operations**. [3](#0-2) 
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run **after** parsing completes, so they do not reduce parse-phase CPU load. [4](#0-3) 
- The only filter in the graphql module is `LoggingFilter`, which performs no throttling. [5](#0-4) 
- The `web3` module has a full `ThrottleConfiguration` with bucket4j rate limiting, but **no equivalent exists in the graphql module**. [6](#0-5) 

### Impact Explanation
An attacker can drive sustained CPU saturation on the mirror node's GraphQL service by keeping the ForkJoinPool filled with parse tasks. Because the ForkJoinPool is shared across the JVM, saturation degrades all async operations on the node, not just GraphQL. The cache's `maximumSize=1000` means the attacker only needs to rotate through >1000 distinct query strings (trivial) to ensure every request is a cache miss. A 30%+ CPU increase is realistic with a modest botnet or even a single high-throughput client sending pipelined HTTP/2 requests.

### Likelihood Explanation
- **No authentication required** — the GraphQL endpoint is public.
- **No rate limiting** — requests are accepted without any per-IP or global throttle.
- **Trivial query variation** — adding a single space, alias, or comment produces a distinct cache key.
- **Repeatable and automatable** — standard HTTP benchmarking tools (e.g., `wrk`, `hey`, `vegeta`) suffice.
- **Low attacker sophistication** — no exploit code needed beyond an HTTP client.

### Recommendation
1. **Add rate limiting to the GraphQL module** mirroring the `web3` module's `ThrottleConfiguration` (bucket4j token bucket, per-IP and global limits).
2. **Bound concurrent parse operations** by providing a bounded executor to `Caffeine.buildAsync(executor)` instead of defaulting to the common ForkJoinPool, so parse tasks cannot starve other JVM async work.
3. **Normalize query strings before caching** (strip comments, canonicalize whitespace) to improve cache hit rates against trivial variation attacks.
4. Consider a **connection/request concurrency limit** at the servlet container level (e.g., Tomcat `maxConnections`, `acceptCount`).

### Proof of Concept

```bash
# Generate 2000 distinct GraphQL queries (vary by alias index)
for i in $(seq 1 2000); do
  echo "{\"query\": \"{ a$i: account(input:{entityId:{shard:0,realm:0,num:1}}) { alias } }\"}"
done > queries.txt

# Flood the endpoint with high concurrency (requires no auth)
cat queries.txt | parallel -j 200 \
  'curl -s -X POST https://<mirror-node-host>/graphql/alpha \
   -H "Content-Type: application/json" \
   -d @- > /dev/null'
```

Each request uses a unique alias (`a1`, `a2`, … `a2000`), producing a distinct query string and guaranteed cache miss. With 200 concurrent workers, 200 simultaneous `parseAndValidateFunction.apply()` calls are submitted to the ForkJoinPool, saturating CPU. Sustained over 24 hours with rotation, this exceeds the 30% resource consumption threshold.

public class ThrottleConfiguration {

    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
