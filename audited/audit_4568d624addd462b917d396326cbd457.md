### Title
Unbounded `limit` Parameter in `getTokensRequest()` Enables Unauthenticated DoS via Oversized DB Query

### Summary
The `getTokensRequest()` function in `rest/tokens.js` validates the `limit` query parameter only with `utils.isPositiveLong()`, which permits any value up to the maximum positive long integer (9,223,372,036,854,775,807). The validated value is passed directly into the SQL `LIMIT` clause with no upper-bound cap enforced. Any unauthenticated caller can trigger a query instructing PostgreSQL to attempt returning billions of rows, exhausting DB memory, connection pool slots, and node resources.

### Finding Description
**Code path:**

1. `getTokensRequest()` (line 371) calls `utils.buildAndValidateFilters(req.query, acceptedTokenParameters, validateTokenQueryFilter)`.
2. `validateTokenQueryFilter()` (lines 330–332) validates `filterKeys.LIMIT` solely via `utils.isPositiveLong(val)` — no maximum ceiling is checked.
3. The validated filter is passed to `extractSqlFromTokenRequest()` (line 390).
4. Inside `extractSqlFromTokenRequest()` (lines 166–173), `limit` is initialized to `defaultLimit` but is **unconditionally overwritten** with `filter.value` when the key is `filterKeys.LIMIT`.
5. That raw user-supplied value is pushed directly into the parameterized query (line 210): `const limitQuery = \`limit $${params.push(limit)}\``.
6. Critically, `getResponseLimit()` is destructured for only `{default: defaultLimit}` (line 26) — the `max` field (if present in config) is **never extracted or enforced** in this code path.

**Root cause:** The validation gate (`isPositiveLong`) confirms only that the value is a positive integer within the long range. No secondary check caps it against a configured maximum. The `max` from `getResponseLimit()` is silently discarded.

### Impact Explanation
An attacker sends `GET /api/v1/tokens?limit=9223372036854775807`. PostgreSQL receives `LIMIT 9223372036854775807`, scans the entire `token` + `entity` tables (with a JOIN), and attempts to buffer the result. This exhausts:
- DB server memory and I/O bandwidth
- Node.js process heap (rows are mapped in-memory via `rows.map(...)` at line 398)
- The shared connection pool (`pool.queryQuietly`), blocking all other queries

Repeated requests from a single unauthenticated IP can render the mirror node REST API completely unavailable, preventing any transaction or gossip data from being served.

**Severity: High** — full service disruption, no authentication required, trivially repeatable.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly accessible. The exploit requires a single HTTP GET request with one crafted query parameter. Any script kiddie or automated scanner can trigger it. It is fully repeatable and can be parallelized across multiple connections to amplify impact.

### Recommendation
1. Extract and enforce the `max` limit from `getResponseLimit()`:
   ```js
   const {default: defaultLimit, max: maxLimit} = getResponseLimit();
   ```
2. In `extractSqlFromTokenRequest()`, cap the user-supplied limit:
   ```js
   if (filter.key === filterKeys.LIMIT) {
     limit = Math.min(filter.value, maxLimit);
     continue;
   }
   ```
3. Update `validateTokenQueryFilter()` for `filterKeys.LIMIT` to reject values exceeding `maxLimit` before they reach SQL construction:
   ```js
   case filterKeys.LIMIT:
     ret = utils.isPositiveLong(val) && Number(val) <= maxLimit;
     break;
   ```
4. Apply the same fix consistently to `extractSqlFromNftTokensRequest()`, `extractSqlFromTokenBalancesRequest()`, and `extractSqlFromNftTransferHistoryRequest()`, which share the same pattern.

### Proof of Concept
```bash
# Single unauthenticated request — instructs DB to return ~9.2 quintillion rows
curl -s "https://<mirror-node-host>/api/v1/tokens?limit=9223372036854775807"

# Parallel flood to exhaust connection pool
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/tokens?limit=9223372036854775807" &
done
wait
# Result: DB CPU/memory spikes, connection pool exhausted, all subsequent API
# requests time out or return 503, gossip/transaction data becomes unavailable.
```
