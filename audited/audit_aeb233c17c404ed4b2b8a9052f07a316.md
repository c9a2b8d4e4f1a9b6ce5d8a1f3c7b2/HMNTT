### Title
Integer Narrowing Overflow in `poll()` Bypasses Query Limit, Enabling Unbounded Database Queries via Unprivileged gRPC Request

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. When an unprivileged user supplies a `limit` value greater than `Integer.MAX_VALUE` via the gRPC `subscribeTopic` endpoint, the cast overflows to a large negative integer. This negative value is then passed as the `limit` field of a new `TopicMessageFilter`, which causes `findByFilter()` to execute an unbounded database query (no `LIMIT` clause), enabling resource exhaustion and denial of service against the mirror node's gRPC service.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` at line 56 directly assigns `query.getLimit()` — a protobuf `uint64` field, represented as a Java `long` — to `TopicMessageFilter.limit` with no upper-bound check:

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

`TopicMessageFilter.limit` is declared as a `long` with only `@Min(0)`:

```java
// TopicMessageFilter.java:25-26
@Min(0)
private long limit;
```

The `@Min(0)` constraint is never enforced here because `toFilter()` is a `private` method; Spring's AOP-based `@Validated` proxy does not intercept private method calls, and no `@Valid` annotation is present on the `subscribeTopic(TopicMessageFilter filter)` parameter in `TopicMessageServiceImpl`.

In `poll()`, the narrowing cast occurs at lines 68–71:

```java
// PollingTopicMessageRetriever.java:68-71
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

With `filter.getLimit() = 2_147_483_648L` (i.e., `Integer.MAX_VALUE + 1`) and `context.getTotal().get() = 0`:
- `(int)(2_147_483_648L)` → `-2_147_483_648` (Integer.MIN_VALUE, two's complement wrap)
- `Math.min(-2_147_483_648, maxPageSize)` → `-2_147_483_648`

The corrupted `pageSize` is then used to build a new filter:

```java
// PollingTopicMessageRetriever.java:75
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
```

In `TopicMessageRepositoryCustomImpl.findByFilter()`, the limit guard is:

```java
// TopicMessageRepositoryCustomImpl.java:51-53
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

`hasLimit()` is defined as `limit > 0`. Since `limit = -2_147_483_648 < 0`, `hasLimit()` returns `false`, so `setMaxResults()` is **never called**. The JPA query executes with **no row limit**, potentially returning every matching row in the `topic_message` table.

The `isComplete()` termination check also fails to stop the loop:

```java
// PollingTopicMessageRetriever.java:122
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
```

This uses the **original** filter (limit = 2,147,483,648), so `limitHit` is never true in practice, and polling continues until the configured timeout (default 60 seconds).

### Impact Explanation

Each poll cycle issues an unbounded `SELECT *` against the `topic_message` table for the given topic. A topic with a large message history causes the database to return millions of rows per poll, consuming database CPU, I/O, JVM heap, and gRPC thread pool resources. Multiple concurrent such subscriptions can exhaust connection pools and cause cascading failures across the mirror node's gRPC service. This is a practical denial-of-service reachable by any unauthenticated client with network access to the gRPC port.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly exposed and requires no authentication. The protobuf `ConsensusTopicQuery.limit` field is `uint64`, so any client can trivially set it to `2_147_483_648` or any value above `Integer.MAX_VALUE`. No special knowledge, credentials, or race conditions are required. The exploit is deterministic and repeatable on every request.

### Recommendation

1. **Fix the cast**: Replace the narrowing cast with a bounds-safe computation:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)  // or a domain-appropriate cap
   private long limit;
   ```
3. **Enforce validation at the entry point**: Add `@Valid` to the `filter` parameter in `TopicMessageServiceImpl.subscribeTopic()` or validate explicitly in `ConsensusController.toFilter()`.
4. **Guard `findByFilter()`**: Treat any non-positive limit as "no limit requested" only when it is exactly 0, and throw or log an error for negative values.

### Proof of Concept

**Preconditions**: A topic exists in the mirror node database with at least one message. The gRPC port is reachable.

**Steps**:
1. Construct a `ConsensusTopicQuery` protobuf message with:
   - `topicID` set to a valid topic
   - `consensusStartTime` set to epoch (0)
   - `limit` set to `2147483648` (= `Integer.MAX_VALUE + 1`)
2. Send the request to the mirror node's gRPC `subscribeTopic` endpoint.
3. Observe in server logs/metrics that `findByFilter` is called without `setMaxResults`, executing `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT` clause.
4. Repeat with many concurrent connections to exhaust database and JVM resources.
