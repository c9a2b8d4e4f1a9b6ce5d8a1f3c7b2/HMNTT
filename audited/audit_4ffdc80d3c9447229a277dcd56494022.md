### Title
Unbounded Request Rate on `getTokenRelationships()` Enables DB Connection Pool Exhaustion via Repeated Token Relationship Queries

### Summary
The `GET /api/v1/accounts/:idOrAliasOrEvmAddress/tokens` endpoint handled by `getTokenRelationships()` in `rest/controllers/tokenController.js` has no application-level rate limiting. Any unauthenticated user can flood this endpoint with concurrent requests, each triggering up to three sequential database queries against `token_account` and `token` tables, exhausting the shared DB connection pool and degrading response times for all other API consumers.

### Finding Description
**Code path:**
- `rest/controllers/tokenController.js` lines 66–92: `getTokenRelationships()` calls `EntityService.isValidAccount()` (DB query 1), then `TokenService.getTokenAccounts()` (DB query 2 on `token_account`), then `TokenService.getCachedTokens()` for any uncached token IDs (DB query 3 on `token`).
- `rest/controllers/tokenController.js` lines 40–41: `extractTokensRelationshipQuery` sets `limit = filter.value` from the parsed filter.
- `rest/utils.js` lines 544–553: `getLimitParamValue()` caps the limit at `responseLimit.max` (default **100**) for unauthenticated users.
- `rest/service/tokenService.js` lines 96–115: `getTokenAccounts()` executes the `token_account` query unconditionally, then issues a second query for any token IDs not in the in-process LRU cache.

**Root cause:** There is no application-level rate limiting in the REST module. A search for `express-rate-limit`, `rateLimit`, or `rateLimiter` in `rest/**/*.js` returns zero matches. The only rate-limiting infrastructure found (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives in the `web3` module and is not applied to the REST API. The Traefik middleware in `charts/hedera-mirror-rest/templates/middleware.yaml` is gated behind `{{ if and .Values.global.middleware .Values.middleware }}` — it is optional and not guaranteed to be deployed.

**Failed assumption:** The design assumes that infrastructure-level rate limiting (Traefik) will always be present in front of the REST service. When it is absent or misconfigured, there is no backstop.

### Impact Explanation
Each request to this endpoint consumes at least one DB connection for the duration of the query. With a default `maxConnections` pool and no per-IP or global request rate cap, a single attacker sending hundreds of concurrent requests can saturate the connection pool. Once the pool is exhausted, all other API endpoints that share the same pool (transactions, accounts, contracts, etc.) begin queuing or timing out, causing broad service degradation. The `statementTimeout` (default 20 000 ms per `rest/config.js`) prevents individual queries from running indefinitely but does not prevent pool exhaustion from many short-lived concurrent queries.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. The account ID parameter can be any valid account (e.g., a well-known treasury account with many token associations). The attacker needs only a standard HTTP client and a valid account ID. The attack is trivially scriptable, repeatable, and requires no on-chain interaction or economic cost.

### Recommendation
1. Add application-level rate limiting (e.g., `express-rate-limit` with a Redis store for multi-instance deployments) directly in the REST server, keyed by source IP, applied to all routes — not relying solely on optional infrastructure middleware.
2. Ensure the Traefik `inFlightReq` and `rateLimit` middleware configuration is enforced as a required deployment prerequisite, not an optional one.
3. Consider adding a short-circuit cache (e.g., Redis) for the full `token_account` result set per account, with a TTL of a few seconds, to reduce repeated identical DB hits.

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
