### Title
`TransactionIdParameter` Regex Permits Out-of-Range Epoch Seconds, Leaking Internal `DateTimeException` Message in 400 Response

### Summary
The regex in `TransactionIdParameter.valueOf()` permits up to 19 decimal digits for the seconds field, allowing values up to `Long.MAX_VALUE` (9223372036854775807). `Instant.ofEpochSecond()` only accepts values up to approximately 253,402,300,799 seconds; supplying `Long.MAX_VALUE` causes a `DateTimeException` whose raw message is caught, wrapped into `InvalidParametersException`, and then surfaced verbatim in the HTTP 400 response body by `GenericControllerAdvice`. No authentication is required to trigger this.

### Finding Description

**Root cause — regex has no numeric upper bound:** [1](#0-0) 

`\d{1,19}` accepts any integer up to 19 digits. `Long.MAX_VALUE` = `9223372036854775807` is exactly 19 digits and passes `Long.parseLong()` without error.

**Unchecked call to `Instant.ofEpochSecond()`:** [2](#0-1) 

`Instant.MAX` is `9999-12-31T23:59:59.999999999Z` ≈ 253,402,300,799 seconds. Passing `Long.MAX_VALUE` throws `java.time.DateTimeException: Instant exceeds minimum or maximum instant`.

**Exception message forwarded verbatim to the caller:** [3](#0-2) 

**`InvalidParametersException` (extends `InvalidInputException`) is handled as BAD_REQUEST:** [4](#0-3) 

**`handleExceptionInternal` includes `ex.getMessage()` for all non-5xx responses:** [5](#0-4) 

Because the status is `400 Bad Request` (not 5xx), the guard `!statusCode.is5xxServerError()` is `true` and `ex.getMessage()` — the raw `DateTimeException` text — is placed in the `detail` field of the response.

**Existing test does NOT cover this case:** [6](#0-5) 

The only `Long.MAX_VALUE`-seconds test uses `Long.MAX_VALUE` for nanos too (19 digits), which fails the `\d{1,9}` nanos group and returns `null` before reaching `Instant.ofEpochSecond()`. The case `seconds=Long.MAX_VALUE, nanos=0` is untested and exploitable.

### Impact Explanation
Any unauthenticated caller receives a 400 response whose `detail` field contains the raw Java runtime message `"Instant exceeds minimum or maximum instant"`. This confirms the server's internal use of `java.time.Instant`, its validation boundary, and that the input successfully bypassed the regex gate — all without any privilege. While not a critical data leak, it violates the explicit design intent expressed in the comment `// Don't leak server errors` and provides an oracle for probing internal time-handling logic.

### Likelihood Explanation
The endpoint `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes` is publicly reachable with no authentication requirement beyond the `Accept-Encoding: gzip` header. [7](#0-6) 

The payload is a single crafted path segment. Any attacker who can send HTTP requests can reproduce this deterministically and repeatedly.

### Recommendation
1. **Add an explicit upper-bound check** on `seconds` after parsing, before calling `Instant.ofEpochSecond()`:
   ```java
   if (seconds < 0 || seconds > Instant.MAX.getEpochSecond()) {
       throw new InvalidParametersException("Invalid transaction ID: seconds out of range");
   }
   ```
2. **Do not forward raw exception messages** from JDK internals. Replace line 39 with a fixed, application-defined message:
   ```java
   } catch (Exception e) {
       throw new InvalidParametersException("Invalid transaction ID format");
   }
   ```
3. **Add a test case** for `seconds=Long.MAX_VALUE, nanos=0` asserting `InvalidParametersException` with a controlled message.

### Proof of Concept
```
GET /api/v1/contracts/results/0.0.3-9223372036854775807-0/opcodes HTTP/1.1
Host: <mirror-node-web3>
Accept-Encoding: gzip
```

Expected response (HTTP 400):
```json
{
  "_status": {
    "messages": [
      {
        "message": "Bad Request",
        "detail": "Instant exceeds minimum or maximum instant"
      }
    ]
  }
}
```

Steps:
1. Start the web3 service with `mirror.web3.opcode.enabled=true`.
2. Send the request above (no credentials needed).
3. Observe the 400 response body contains the raw JDK `DateTimeException` message in `detail`.
4. Vary `seconds` (e.g., `9223372036854775806`) to confirm the boundary and that the message is always leaked.
