### Title
Unauthenticated Connection Pool Exhaustion via Uncached `isValidAccount()` DB Query in `/accounts/{id}/rewards`

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` unconditionally executes a database query (`select type from entity where id = $1`) on every request via `EntityService.isValidAccount()`, with no rate limiting on the REST API and a default connection pool of only 10 connections. An unprivileged attacker can flood this public endpoint with concurrent requests using valid account IDs, exhausting the connection pool and rendering the service unavailable to all users.

### Finding Description

**Exact code path:**

`listStakingRewardsByAccountId` at `rest/controllers/accountController.js` lines 170–175 calls `EntityService.isValidAccount(accountId)` on every request:

```js
listStakingRewardsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);
  const isValidAccount = await EntityService.isValidAccount(accountId); // DB hit every time
  if (!isValidAccount) { throw new NotFoundError(); }
``` [1](#0-0) 

`isValidAccount` in `rest/service/entityService.js` lines 60–63 always executes `entityExistenceQuery` directly against the database with no caching layer:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [2](#0-1) 

The query is `select type from entity where id = $1`: [3](#0-2) 

**Connection pool:** The default `maxConnections` is **10** and `connectionTimeoutMillis` is **20,000 ms** (20 seconds): [4](#0-3) 

The pool is initialized directly from this config: [5](#0-4) 

**No rate limiting on the REST API:** `rest/server.js` registers no rate-limiting middleware for the `/accounts/{id}/rewards` route. The middleware stack is: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler` (authentication only), optional `metricsHandler`, optional `responseCacheCheckHandler`. None of these throttle request rates: [6](#0-5) 

The throttle configuration found (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) belongs exclusively to the `web3` Java module and does not apply to the Node.js REST API.

**No caching for `isValidAccount`:** The documented `entityId` cache (`maxAge: 1800s`, `maxSize: 100000`) applies to `getEncodedId`, not to `isValidAccount`. The existence check always hits the database: [7](#0-6) 

**Root cause and failed assumption:** The design assumes that infrastructure-level rate limiting (e.g., an API gateway or reverse proxy) will be present in front of the service. No such protection is enforced at the application layer. The `isValidAccount` check was added as a guard against 404 responses but introduces an unconditional, uncached DB round-trip per request.

### Impact Explanation

With a default pool of 10 connections and a 20-second connection timeout, an attacker sending just 10–20 concurrent requests per second can fully saturate the pool. All subsequent requests from legitimate users queue up and wait 20 seconds before receiving a timeout error. This constitutes a complete denial of service for the `/accounts/{id}/rewards` endpoint and, because the pool is shared, potentially for all REST API endpoints served by the same Node.js process. The `statementTimeout` of 20 seconds amplifies the attack window — each connection can be held for up to 20 seconds per query under load.

### Likelihood Explanation

The endpoint is public and requires no authentication or API key. Valid account IDs are trivially enumerable (they are sequential integers: `0.0.1`, `0.0.2`, ..., `0.0.N`). The attacker needs only a standard HTTP client capable of sending concurrent requests. No special knowledge, credentials, or exploit tooling is required. The attack is repeatable indefinitely and can be automated with a simple script.

### Recommendation

1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP address, applied before route handlers.
2. **Cache the result of `isValidAccount`** using the existing `entityId` cache infrastructure, or merge the existence check into the staking rewards query itself (return an empty result set rather than a 404 for non-existent accounts, or check existence in a single combined query).
3. **Increase the default connection pool size** and document the dependency on upstream rate limiting explicitly.
4. **Consider returning an empty rewards array** instead of a 404 for non-existent accounts, eliminating the need for the separate existence check entirely.

### Proof of Concept

```bash
# Generate 500 concurrent requests to the rewards endpoint with sequential valid account IDs
# No authentication required
seq 1 500 | xargs -P 100 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node-host>/api/v1/accounts/0.0.{}/rewards"
```

**Expected result:** After the first ~10 concurrent requests acquire pool connections, subsequent requests begin timing out after 20 seconds (`connectionTimeoutMillis: 20000`). Legitimate users receive 500 errors or experience 20-second delays. The attack requires no valid credentials and is executable from a single machine.
