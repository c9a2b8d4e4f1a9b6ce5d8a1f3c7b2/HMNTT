### Title
Unauthenticated DoS via Unbounded Spender+Token Range Scan Exhausting DB Connection Pool in `extractNftMultiUnionQuery`

### Summary
An unprivileged external user can submit `spender.id=gte:0&spender.id=lte:999999999` combined with `token.id=gte:0&token.id=lte:999999999` to `/api/v1/accounts/{id}/nfts`. The `validateFilters` method in `AccountController` only rejects duplicate directional operators, not wide-range combinations, so both filters pass. The resulting multi-union SQL query performs an unindexed range scan across both `token_id` and `spender` columns, holding a DB connection for up to the full 20-second `statement_timeout`. With the default pool of 10 connections, 10 concurrent such requests exhaust the pool, causing all subsequent legitimate requests to fail with a connection-acquisition timeout.

### Finding Description

**Code path:**

`getNftsByAccountId` (line 90) calls `buildAndValidateFilters` then `extractNftMultiUnionQuery` (line 93), which calls `validateFilters` (line 66).

`validateFilters` (lines 18–32) only rejects:
- More than one lower-bound operator (`lt`/`lte`) for `spender.id`
- More than one upper-bound operator (`gt`/`gte`) for `spender.id`
- The `ne` operator

A single `gte` + single `lte` is the **intended valid pattern** and passes without any range-width check.

`filterValidityChecks` in `utils.js` (line 359–361) validates `spender.id` only as `EntityId.isValidEntityId(val)`. Both `0` and `999999999` are valid entity IDs.

`buildFilters` (lines 1234–1259) only enforces `maxRepeatedQueryParameters` (default 100), which is not triggered by two `spender.id` params.

The resulting SQL generated by `NftService.getSubQuery` (lines 55–76) for the inner sub-query is:

```sql
SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
WHERE account_id = $1
  AND token_id > $X AND token_id < $Y   -- wide token range
  AND spender >= $A AND spender <= $B   -- wide spender range
ORDER BY token_id DESC, serial_number DESC
LIMIT $2
```

There is no composite index on `(account_id, spender)` implied by the schema; PostgreSQL must scan all NFTs for the account in the token_id range and post-filter by spender. The `LEFT JOIN entity` adds further cost. The multi-union structure (lower + inner + upper sub-queries, each with its own LIMIT) multiplies this work up to 3×.

**DB pool configuration** (`rest/dbpool.js` lines 13–15):
- `connectionTimeoutMillis`: 20,000 ms
- `max`: 10 connections
- `statement_timeout`: 20,000 ms

No rate limiting exists at the REST API layer.

### Impact Explanation
With 10 concurrent requests each holding a connection for up to 20 seconds, the entire connection pool is exhausted. All other API requests (including unrelated endpoints) that require a DB connection will block for up to 20 seconds and then fail with a connection-timeout error. The attacker can continuously recycle requests to maintain pool exhaustion indefinitely. This constitutes a full service-level DoS for all users of the REST API, achievable with no credentials and minimal bandwidth.

### Likelihood Explanation
The attack requires no authentication, no special account state, and no knowledge beyond the public API documentation. The endpoint `/api/v1/accounts/{id}/nfts` is publicly accessible. The exploit is trivially scriptable with any HTTP client capable of sending 10 concurrent requests. It is fully repeatable and can be sustained indefinitely. The only prerequisite is that the targeted account has a non-trivial number of NFTs; even a moderately populated network satisfies this.

### Recommendation
1. **Enforce a maximum spender range width**: In `validateFilters`, compute `upper.value - lower.value` when both a `gte`/`gt` and `lte`/`lt` are present for `spender.id` and reject combinations exceeding a configured threshold (e.g., 1,000 entity IDs).
2. **Enforce a maximum token.id range width**: Apply the same check in `validateBoundsRange` or `extractNftMultiUnionQuery` for the `token_id` primary bound.
3. **Add API-level rate limiting**: Implement per-IP or per-account rate limiting on the `/accounts/{id}/nfts` endpoint.
4. **Add a DB-level index**: Ensure a composite index on `(account_id, spender, token_id)` exists on the `nft` table so range scans on spender are index-supported.
5. **Reduce default pool size exposure**: Consider increasing `maxConnections` or adding a query-queue with a shorter per-request timeout at the application layer before the DB pool is reached.

### Proof of Concept

```bash
# Send 10 concurrent exhaustion requests (replace 0.0.1001 with any account holding NFTs)
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1001/nfts?\
spender.id=gte:0&spender.id=lte:999999999&\
token.id=gte:0&token.id=lte:999999999&\
limit=100" &
done
wait

# Immediately probe a legitimate request — it will block for ~20s then fail:
time curl "http://<mirror-node>/api/v1/accounts/0.0.1002/nfts"
# Expected: connection timeout error after ~20 seconds
```

**Validation steps:**
1. Confirm `spender.id=gte:0&spender.id=lte:999999999` returns HTTP 200 (not 400) — passes `validateFilters` lines 21–27.
2. Confirm `token.id=gte:0&token.id=lte:999999999` returns HTTP 200 — passes `validateBoundsRange`.
3. Observe DB query duration approaching `statement_timeout` (20 s) on a populated dataset.
4. With 10 concurrent requests active, observe subsequent requests returning connection-timeout errors.
