### Title
Unbounded Historical Scan via `startTime=0` in `PollingTopicListener.poll()` Enables Sustained DB Griefing

### Summary
An unprivileged gRPC client can submit a `ConsensusTopicQuery` with `consensusStartTime` set to epoch 0 (seconds=0, nanos=0). The `TopicMessageFilter` validation permits this value (`@Min(0)` and `isValidStartTime()` only checks `<= now()`). On every poll cycle, `poll()` computes `startTime = filter.getStartTime()` (= 0) until the first message is received, then advances page-by-page through the entire topic history, issuing a DB query every 500 ms until it catches up to the present. Multiple concurrent subscribers amplify the load linearly with no per-client throttle.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` converts the protobuf `ConsensusTopicQuery` directly into a `TopicMessageFilter`:

```java
// ConsensusController.java lines 62-65
if (query.hasConsensusStartTime()) {
    long startTime = convertTimestamp(query.getConsensusStartTime());
    filter.startTime(startTime);
}
```

`convertTimestamp` of a zero-second Timestamp returns `DomainUtils.timestampInNanosMax({seconds=0,nanos=0})` = `0L`.

`TopicMessageFilter` validation: [1](#0-0) [2](#0-1) 

`@Min(0)` explicitly permits `0`. `isValidStartTime()` only checks `startTime <= DomainUtils.now()`, which `0` trivially satisfies. No lower bound on how far back a client may request.

`poll()` on first invocation (`last == null`): [3](#0-2) 

`startTime = filter.getStartTime()` = 0. `pageSize = Math.min(Integer.MAX_VALUE, 5000)` = 5000 (default `maxPageSize`). [4](#0-3) 

`findByFilter()` executes: [5](#0-4) 

The JPA criteria query becomes: `WHERE topic_id = ? AND consensus_timestamp >= 0 ORDER BY consensus_timestamp LIMIT 5000`. With no `endTime`, the upper bound is unbounded. After each page of 5000 rows, `last` advances and the next poll starts from `last.consensusTimestamp + 1`, continuing until the subscriber catches up to the present.

**Why existing checks fail:**

- `maxPageSize=5000` bounds each individual query but does **not** bound the number of sequential polls. For a topic with N messages, the catch-up phase issues `ceil(N/5000)` DB queries at 500 ms intervals.
- `RepeatSpec.times(Long.MAX_VALUE)` means polling continues indefinitely. [6](#0-5) 
- There is no per-subscriber rate limit, connection limit, or minimum `startTime` age enforcement anywhere in the filter validation or listener logic.

### Impact Explanation
For a topic with 10 million messages: `ceil(10,000,000 / 5000)` = 2,000 sequential DB queries, each reading 5000 rows, sustained over ~16 minutes per subscriber. Ten concurrent attackers produce 20,000 queries reading 100 million rows total. Each query also executes a `SET LOCAL random_page_cost = 0` native query (line 57 of `TopicMessageRepositoryCustomImpl`), adding extra round-trips. This degrades DB performance for all legitimate users sharing the same PostgreSQL instance, constituting a medium-severity availability griefing attack with no economic cost to the attacker.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required in the default deployment. The protobuf `ConsensusTopicQuery` message allows `consensusStartTime` to be set to any `Timestamp`, including `{seconds: 0, nanos: 0}`. The attack requires only a standard gRPC client (e.g., `grpcurl`) and knowledge of any existing topic ID. It is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a minimum `startTime` age** in `TopicMessageFilter.isValidStartTime()`: reject requests where `startTime < (DomainUtils.now() - MAX_HISTORICAL_WINDOW_NS)`, where `MAX_HISTORICAL_WINDOW_NS` is a configurable operator limit (e.g., 7 days).
2. **Alternatively**, enforce a minimum `startTime` in `ConsensusController.toFilter()` before building the filter, clamping it to `max(requestedStartTime, now - maxHistoricalWindow)`.
3. **Add per-subscriber connection limits** or rate limiting at the gRPC service layer to bound concurrent historical subscriptions.
4. **Consider a configurable `maxCatchupPolls`** in `ListenerProperties` to terminate subscriptions that have been in catch-up mode for too long.

### Proof of Concept

```bash
# 1. Install grpcurl
# 2. Obtain any valid topic ID (e.g., 0.0.1234) from the mirror node REST API
# 3. Send a subscription with consensusStartTime = epoch 0:

grpcurl -plaintext \
  -d '{
    "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 1234},
    "consensusStartTime": {"seconds": 0, "nanos": 0}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

# Result: The server begins polling from consensus_timestamp >= 0,
# issuing a 5000-row DB query every ~500ms until it catches up to present.
# Repeat with N concurrent connections to multiply DB load by N.
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
