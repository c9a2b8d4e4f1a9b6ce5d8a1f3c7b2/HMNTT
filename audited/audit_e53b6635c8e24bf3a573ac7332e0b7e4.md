### Title
Unprivileged gRPC Subscriber Can Reset `lastConsensusTimestamp` to Skip Unprocessed Topic Messages Permanently

### Summary
`SharedPollingTopicListener.PollingContext.onStart()` unconditionally sets `lastConsensusTimestamp` to `DomainUtils.now()` (current wall-clock nanoseconds) every time the shared Flux is re-subscribed. Because the `topicMessages` Flux uses Reactor's `.share()` (refCount semantics), any unprivileged user who subscribes when the active-subscriber count is zero triggers a fresh upstream subscription and thus a fresh `onStart` call. If at that instant the wall-clock time is already past the consensus timestamp of a message that the importer has not yet written to the database, that message is permanently excluded from every subsequent `findLatest` poll, which uses a strict `>` predicate on `lastConsensusTimestamp`.

### Finding Description

**Exact code path**

`onStart` in `PollingContext`: [1](#0-0) 

`DomainUtils.now()` returns the current wall-clock instant in nanoseconds: [2](#0-1) 

`findLatest` uses a strict `>` predicate against `lastConsensusTimestamp`: [3](#0-2) 

The shared Flux is built with `.share()` (publish/refCount), meaning the upstream — including `.doOnSubscribe(context::onStart)` — is re-subscribed every time the subscriber count transitions from 0 → 1: [4](#0-3) 

**Root cause and failed assumption**

The design assumes `onStart` is called only once at application startup, so using `DomainUtils.now()` as the initial cursor is safe (no live messages exist before "now"). The failed assumption is that `.share()` guarantees a single upstream subscription for the lifetime of the process. In reality, `.share()` cancels the upstream when the last subscriber leaves and re-subscribes (calling `onStart` again) when the next subscriber arrives. Because `lastConsensusTimestamp` is a field of the single shared `PollingContext` instance, each re-subscription overwrites the cursor with the current wall-clock time, which may be ahead of messages that have achieved consensus on the network but have not yet been ingested by the importer.

**Exploit flow**

1. Attacker monitors the Hedera network and observes that a topic message (e.g., a governance freeze instruction) has reached consensus at timestamp `T_freeze`.
2. The importer has not yet written this message to the `topic_message` table (ingestion lag is typically seconds to minutes).
3. The attacker ensures no other gRPC clients are actively subscribed to `SharedPollingTopicListener` (e.g., by waiting for a quiet period, or by being the first subscriber after a server restart or after all previous subscribers have disconnected).
4. The attacker calls the public, unauthenticated gRPC `subscribeTopic` endpoint. This is the first subscription, so Reactor's refCount triggers an upstream subscription, invoking `onStart`.
5. `onStart` executes `lastConsensusTimestamp.set(DomainUtils.now())`, setting the cursor to `T_now > T_freeze`.
6. The importer subsequently ingests the freeze message with `consensusTimestamp = T_freeze`.
7. Every future `findLatest(T_now, pageSize)` call issues `WHERE consensus_timestamp > T_now`, which permanently excludes `T_freeze` because `T_freeze < T_now`.
8. The freeze message is never delivered to any gRPC subscriber — current or future — for the lifetime of the process (or until the next re-subscription resets the cursor even further forward).

**Why existing checks are insufficient**

The `missingMessages` gap-filling logic in `TopicMessageServiceImpl` detects gaps between *consecutively received* messages by comparing sequence numbers: [5](#0-4) 

It cannot recover a message that was skipped *before* the listener's cursor was established, because there is no "previous message" to compare against. The safety-check `Mono.delay(1s)` path also calls `missingMessages(topicContext, null)`, but that only queries from the *subscriber's* `startTime` forward through the retriever, not from the listener's internal cursor. The listener's cursor is opaque to the service layer.

### Impact Explanation

All gRPC subscribers — including legitimate applications that use HCS topic messages to coordinate fund-freeze operations — permanently miss the skipped message. Because `lastConsensusTimestamp` only ever moves forward (updated by `onNext` to the latest delivered message's timestamp), there is no self-healing path: the skipped message's timestamp is forever below the cursor. Any application that relies on receiving this message to execute a freeze (e.g., a DeFi governance contract monitoring a topic) will never receive the instruction, leaving funds in an unintended state. The impact is process-lifetime-persistent and affects all subscribers sharing the single `PollingContext`.

### Likelihood Explanation

The attack requires no credentials — the gRPC `subscribeTopic` endpoint is public. The attacker needs only to:
- Observe a pending freeze message on the Hedera network (publicly visible via consensus node APIs or block explorers).
- Subscribe when the active subscriber count is zero (a realistic condition during low-traffic periods, after server restarts, or by waiting for all other subscribers to time out).

The importer ingestion lag (seconds to minutes) provides a wide timing window. The attack is repeatable: each time the subscriber count drops to zero and the attacker re-subscribes, the cursor is reset. No special privileges, cryptographic material, or insider access are required.

### Recommendation

1. **Do not use wall-clock time as the initial cursor.** Instead, initialize `lastConsensusTimestamp` to the maximum `consensus_timestamp` already present in the `topic_message` table at startup (or at re-subscription time), so the cursor is anchored to the last *ingested* message, not to the current clock.
2. **Prevent cursor regression on re-subscription.** Guard `onStart` so it only sets `lastConsensusTimestamp` if the current value is zero (i.e., truly the first subscription), using an atomic compare-and-set:
   ```java
   void onStart(Subscription subscription) {
       lastConsensusTimestamp.compareAndSet(0L, queryMaxIngestedTimestamp());
   }
   ```
3. **Alternatively, use `.publish().autoConnect(1)` instead of `.share()`** so the upstream is never cancelled and `onStart` is called exactly once per process lifetime.
4. **Add a lower-bound guard in `findLatest`** that rejects a `consensusTimestamp` parameter that is more than a configurable tolerance (e.g., 30 seconds) ahead of the database's current maximum, to detect and alert on cursor-skipping conditions.

### Proof of Concept

```
Preconditions:
  - Mirror node running with SHARED_POLL listener type
  - No active gRPC subscribers (or server just restarted)
  - Freeze message submitted to Hedera topic 0.0.X at consensus timestamp T_freeze
  - Importer ingestion lag: ~10 seconds

Steps:
1. Confirm T_freeze is visible on the Hedera network but NOT yet in the mirror DB:
     SELECT COUNT(*) FROM topic_message WHERE consensus_timestamp = T_freeze;
     -- returns 0

2. Immediately open a gRPC subscription (any topic, any unprivileged client):
     grpcurl -plaintext <mirror-node>:5600 \
       com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
       '{"topicID": {"topicNum": X}, "consensusStartTime": {"seconds": 0}}'
   This triggers onStart → lastConsensusTimestamp = DomainUtils.now() > T_freeze

3. Wait for importer to ingest the freeze message (observe importer logs).

4. Confirm the freeze message is in the DB:
     SELECT consensus_timestamp FROM topic_message WHERE consensus_timestamp = T_freeze;
     -- returns 1 row

5. Observe that the gRPC stream from step 2 never delivers the freeze message.
   Confirm by checking that no message with sequence number corresponding to T_freeze
   is emitted, even after multiple polling intervals.

6. Open additional gRPC subscriptions — they also never receive the freeze message,
   because lastConsensusTimestamp is shared and permanently set above T_freeze.
```

    private Flux<TopicMessage> missingMessages(TopicContext topicContext, @Nullable TopicMessage current) {
        final var last = topicContext.getLast();

        // Safety check triggered
        if (current == null) {
            long startTime = last != null
                    ? last.getConsensusTimestamp() + 1
                    : topicContext.getFilter().getStartTime();
            var gapFilter =
                    topicContext.getFilter().toBuilder().startTime(startTime).build();
            log.info("Safety check triggering gap recovery query with filter {}", gapFilter);
            return topicMessageRetriever.retrieve(gapFilter, false);
        }

        if (last == null || topicContext.isNext(current)) {
            return Flux.just(current);
        }

        long numMissingMessages = current.getSequenceNumber() - last.getSequenceNumber() - 1;

        // ignore duplicate message already processed by larger subscribe context
        if (numMissingMessages <= -1) {
            log.debug("Encountered duplicate missing message to be ignored, last: {}, current: {}", last, current);
            return Flux.empty();
        }

        TopicMessageFilter newFilter = topicContext.getFilter().toBuilder()
                .endTime(current.getConsensusTimestamp())
                .limit(numMissingMessages)
                .startTime(last.getConsensusTimestamp() + 1)
                .build();

        log.info(
                "[{}] Querying topic {} for missing messages between sequence {} and {}",
                newFilter.getSubscriberId(),
                topicContext.getTopicId(),
                last.getSequenceNumber(),
                current.getSequenceNumber());

        return topicMessageRetriever.retrieve(newFilter, false).concatWithValues(current);
