### Title
Race Condition in `RedisTopicListener`: `computeIfAbsent`/`remove` Interleaving Causes Orphaned Subscriptions, Map Corruption, and Missed Messages

### Summary
`getSharedListener()` eagerly retrieves a `Flux` reference via `computeIfAbsent` at assembly time, but the actual subscription to that `Flux` is deferred until a downstream subscriber subscribes. If the last existing subscriber cancels in the window between those two events, `unsubscribe()` removes the entry from the map and a racing thread inserts a new `Flux` (F2). The original caller then subscribes to the now-orphaned F1, and when F1 eventually unsubscribes it blindly removes F2 from the map — corrupting the tracking state for all subsequent subscribers.

### Finding Description

**Exact code path:**

`SharedTopicListener.listen()` calls `getSharedListener()` eagerly (not inside `Flux.defer()`):

```
// SharedTopicListener.java lines 21-26
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)          // <-- F1 obtained HERE (assembly time)
            .doOnSubscribe(...)
            .onBackpressureBuffer(...)
            .publishOn(Schedulers.boundedElastic(), ...);  // <-- subscription deferred
}
```

`getSharedListener()` returns the Flux immediately:

```
// RedisTopicListener.java lines 59-62
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`subscribe()` places `doOnCancel` **before** `.share()`, meaning it fires when `share()`'s refCount drops to 0 (all subscribers cancel), not per-subscriber:

```
// RedisTopicListener.java lines 68-80
return container.flatMapMany(...)
        .map(Message::getMessage)
        .doOnCancel(() -> unsubscribe(topic))   // fires when share() cancels upstream
        .doOnComplete(() -> unsubscribe(topic))
        ...
        .retryWhen(...)
        .share();
```

`unsubscribe()` does a blind `remove()` with no identity check:

```
// RedisTopicListener.java lines 82-85
private void unsubscribe(Topic topic) {
    topicMessages.remove(topic.getTopic());   // removes WHATEVER is in the map for this key
}
```

**Root cause:** The map operation (`computeIfAbsent`) and the actual subscription are not atomic. `ConcurrentHashMap` guarantees per-operation atomicity, but not across the `computeIfAbsent` → subscribe sequence. The `unsubscribe()` `remove()` call has no identity guard — it removes any value stored under the key, not specifically the Flux that triggered it.

**Exploit flow:**

```
Thread A: computeIfAbsent("topic.1") → returns F1 (F1 is in map)
                                                    ↓
                              [last subscriber of F1 cancels]
                              [share() refCount → 0, upstream cancelled]
                              [doOnCancel fires → topicMessages.remove("topic.1")]
                                                    ↓
Thread B: computeIfAbsent("topic.1") → key absent → creates F2 → stores F2 in map
                                                    ↓
Thread A: subscribes to F1 → share() refCount was 0 → F1 re-subscribes upstream (new Redis sub)
          F1 is NOT in the map (F2 is)
```

**Cascading corruption:**

When Thread A eventually cancels:
- F1's `doOnCancel` fires → `topicMessages.remove("topic.1")` → **removes F2** (not F1)
- F2 subscribers are now orphaned (active but not tracked)
- Future subscribers create F3, while F2 subscribers remain on a separate, untracked Redis subscription
- This can repeat indefinitely, accumulating orphaned Redis subscriptions

**Why existing checks are insufficient:**

`ConcurrentHashMap` is used, which prevents data corruption within a single operation. However, the two-step pattern of (1) get-or-create Flux reference and (2) subscribe to it is not atomic. There is no version/identity check in `unsubscribe()` — `topicMessages.remove(key)` removes whatever is currently mapped to that key, not the specific Flux instance that registered the callback.

### Impact Explanation

1. **Missed messages**: Between F1's upstream cancellation and F1's re-subscription (the window where Thread A holds F1 but hasn't subscribed yet), any messages published to Redis for that topic are received by nobody. Thread A misses those messages permanently.
2. **Map state corruption**: F1's deferred `unsubscribe()` removes F2 from the map, causing all F2 subscribers to become orphaned. Their future cancellations will in turn remove F3, and so on — a cascading corruption of the tracking map.
3. **Redis resource leak**: Each race cycle creates an additional untracked Redis subscription. Under repeated exploitation, this exhausts Redis connection/subscription limits.
4. **Denial of service**: An attacker who repeatedly subscribes and immediately cancels can amplify the race, accumulating orphaned subscriptions until the Redis connection pool is exhausted, degrading service for all users.

### Likelihood Explanation

An unprivileged gRPC client can subscribe and cancel at will. The race window is the scheduling delay introduced by `publishOn(Schedulers.boundedElastic())` in `SharedTopicListener.listen()` — this is a real, non-zero delay (thread pool scheduling latency). Under moderate concurrency (multiple clients subscribing to the same topic), the race is reproducible without any special privileges or knowledge of internals. The attacker only needs to:
- Connect to the public gRPC endpoint
- Subscribe to any valid topic
- Cancel immediately and re-subscribe in a tight loop with concurrent connections

The cascading nature (each race corrupts the next Flux's tracking) means the impact compounds over time.

### Recommendation

1. **Use `Flux.defer()` in `listen()`** so the Flux reference is obtained at subscription time, not assembly time:
   ```java
   public Flux<TopicMessage> listen(TopicMessageFilter filter) {
       return Flux.defer(() -> getSharedListener(filter))
               .onBackpressureBuffer(...)
               .publishOn(Schedulers.boundedElastic(), ...);
   }
   ```

2. **Use identity-aware removal in `unsubscribe()`** to prevent removing a different Flux instance:
   ```java
   private void unsubscribe(Topic topic, Flux<TopicMessage> flux) {
       topicMessages.remove(topic.getTopic(), flux);  // ConcurrentHashMap conditional remove
   }
   ```
   And pass the specific Flux instance: `.doOnCancel(() -> unsubscribe(topic, thisFlux))`.

3. **Use `computeIfPresent` / atomic replace** patterns when updating the map to ensure the entry being removed is the same one that was inserted.

### Proof of Concept

**Preconditions:** Two concurrent gRPC clients (Client A and Client B) subscribing to the same topic. One existing subscriber (Client X) is active.

**Steps:**

1. Client X subscribes to `topic.1` → F1 is created and stored in `topicMessages`.
2. Client A calls `listen(topic.1)` → `getSharedListener()` → `computeIfAbsent` returns F1. Client A has a reference to F1 but has not yet subscribed (thread is about to be scheduled on `boundedElastic`).
3. Client X cancels its subscription → F1's `share()` refCount drops to 0 → upstream cancelled → `doOnCancel` fires → `topicMessages.remove("topic.1")` removes F1.
4. Client B calls `listen(topic.1)` → `computeIfAbsent` finds no entry → creates F2 → stores F2 in map.
5. Client A's thread is scheduled → subscribes to F1 → F1 re-subscribes upstream (new Redis subscription). F1 is not in the map.
6. Publish a message to Redis `topic.1` → both F1 and F2 receive it (two Redis subscriptions active — resource leak confirmed).
7. Client A cancels → F1's `doOnCancel` fires → `topicMessages.remove("topic.1")` → **removes F2 from the map**.
8. Client B is now subscribed to F2 (active, receiving messages) but F2 is no longer tracked in the map.
9. Client C calls `listen(topic.1)` → `computeIfAbsent` finds no entry → creates F3. Now F2 and F3 are both active Redis subscriptions, only F3 is tracked.
