### Title
Unbounded Pagination Amplification via `limit=1` in `getAccountTokenAllowances` Enables DB Griefing

### Summary
The `/accounts/:idOrAliasOrEvmAddress/allowances/tokens` endpoint accepts `limit=1` from any unauthenticated caller with no per-IP or per-client rate limiting in the REST service. An attacker can follow each pagination `next` link in a tight loop, issuing exactly one DB query per token allowance on a targeted account. Because the REST Node.js service has no request-rate guard on this path, a single automated client can sustain a high-frequency stream of DB queries that competes for the finite connection pool and statement-timeout budget shared by all legitimate users.

### Finding Description
**Code path:**

- `rest/routes/accountRoute.js:18` — registers the endpoint with no middleware rate-limit guard.
- `rest/controllers/tokenAllowanceController.js:68-81` (`getAccountTokenAllowances`) — calls `buildAndValidateFilters`, then `TokenAllowanceService.getAccountTokenAllowances(query)` (one DB round-trip per HTTP request), then `getPaginationLink` which emits a `next` link whenever `allowances.length === limit`.
- `rest/utils.js:544-553` (`getLimitParamValue`) — caps the limit at `getEffectiveMaxLimit()` (default 100 for unauthenticated users) but imposes **no minimum above 1**; `limit=1` is fully accepted.
- `rest/service/tokenAllowanceService.js:86-90` (`getAccountTokenAllowances`) — issues a single SQL query (up to a 3-way UNION) with `LIMIT $2` bound to the user-supplied value. With `limit=1` this fetches exactly one row and returns.
- `rest/controllers/baseController.js:195-223` (`getPaginationLink`) — returns a non-null `next` URL whenever `rows.length < limit` is false, i.e., whenever the page is full. With `limit=1` every non-empty page is "full", so a `next` link is always emitted until the last allowance is reached.
- `rest/dbpool.js:14-15` — the pool has a finite `max: config.db.pool.maxConnections` and a `statement_timeout`; no per-client connection quota exists.

**Root cause:** The limit parameter is validated only for an upper bound (≤ 100); there is no lower-bound enforcement above 1 and no per-IP/per-session request-rate control on this endpoint. The combination means an attacker can force the server to issue O(N) DB queries to traverse N token allowances, one query per HTTP request, at whatever rate the server can sustain.

### Impact Explanation
Each HTTP request with `limit=1` holds a DB connection for the duration of one query, releases it, then the attacker immediately fires the next request. Sustained at high concurrency (multiple parallel pagination chains against multiple accounts), this can saturate the connection pool (`maxConnections`), increase DB CPU/IO, and cause legitimate requests to queue or time out. The impact is griefing-level service degradation with no economic loss to network participants, consistent with the Medium severity scope.

### Likelihood Explanation
No authentication, no API key, and no rate-limiting middleware is present on this route. Any external actor can automate pagination traversal with a trivial script (e.g., `curl` in a loop following the `links.next` field). Accounts with large numbers of token allowances (common for DeFi-style contracts on Hedera) amplify the effect. The attack is repeatable indefinitely and requires zero on-chain resources.

### Recommendation
1. **Enforce a minimum limit floor** (e.g., ≥ 10) in `getLimitParamValue` or in `extractTokenMultiUnionQuery` to reduce the maximum number of round-trips needed to traverse a dataset.
2. **Add per-IP rate limiting middleware** (e.g., `express-rate-limit`) applied globally or specifically to allowance endpoints before the route handler in `accountRoute.js`.
3. **Add a cursor-based page-count cap** or a maximum total-pages-per-session guard so that a single client cannot issue more than a configurable number of paginated requests against the same account within a time window.

### Proof of Concept
```bash
# Step 1: fetch first page with limit=1
NEXT="/api/v1/accounts/0.0.1001/allowances/tokens?limit=1"
BASE="https://<mirror-node-host>"

while [ "$NEXT" != "null" ] && [ -n "$NEXT" ]; do
  RESPONSE=$(curl -s "${BASE}${NEXT}")
  # Each iteration issues one DB query on the server
  NEXT=$(echo "$RESPONSE" | jq -r '.links.next')
  echo "Fetched page, next=$NEXT"
done
# For an account with N token allowances this loop issues N sequential DB queries.
# Run multiple parallel instances to amplify connection-pool pressure.
```

Each iteration of the loop triggers the full `getAccountTokenAllowances` → `TokenAllowanceService.getQuery` → `super.getRows` → DB round-trip path with no server-side throttle to stop it.
