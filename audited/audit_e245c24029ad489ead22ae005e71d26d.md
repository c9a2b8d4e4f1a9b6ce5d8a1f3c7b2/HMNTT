### Title
Unauthenticated Double-DB-Query DoS via Cache-Miss Amplification in `getTokenRelationships`

### Summary
The `/accounts/:id/tokens` endpoint requires no authentication and unconditionally issues a second database query (`tokenCacheQuery`) for every token ID returned that is not already in the process-local LRU cache. An attacker who rotates requests across accounts whose token associations have not yet been cached can sustain a 2× DB-query amplification factor indefinitely, exhausting the small DB connection pool (default: 10 connections) and degrading or denying service to all other API consumers.

### Finding Description

**Primary query path:**

`getTokenRelationships()` (tokenController.js:66–92) calls `TokenService.getTokenAccounts(query)` with no authentication gate. [1](#0-0) 

`getTokenAccounts()` (tokenService.js:96–115) first executes `tokenRelationshipsQuery` against the DB, then unconditionally calls `getCachedTokens()` with every token ID in the result set. [2](#0-1) 

**Secondary query trigger:**

`getCachedTokens()` (tokenService.js:136–161) iterates the returned token IDs, separates them into cached vs. uncached, and — if any are uncached — fires `tokenCacheQuery` (`WHERE token_id = ANY($1)`) as a second synchronous DB round-trip. [3](#0-2) 

**Cache characteristics:**

The cache is a process-local `quickLru` instance with `maxSize: 100,000` and **no TTL**. Entries are only evicted by LRU pressure. [4](#0-3) 

**Failed assumption:** The design assumes that the in-process LRU cache will absorb repeated lookups for the same token IDs. This holds for a steady-state workload against a fixed set of popular accounts. It fails when an attacker rotates across accounts whose token associations have not yet been cached, because each such request produces a full cache miss for up to 100 token IDs and triggers the secondary query.

**Exploit flow:**

1. Attacker enumerates (or already knows) accounts on the Hedera network that hold many token associations — this information is public.
2. Attacker sends `GET /api/v1/accounts/{id}/tokens?limit=100` for account A₁. Primary query returns up to 100 rows; all 100 token IDs are uncached → secondary `tokenCacheQuery` fires.
3. Attacker immediately rotates to account A₂ (different token set) → another secondary query.
4. By cycling through enough distinct accounts (each with distinct token associations), the attacker keeps the working set of requested tokens larger than what the LRU cache retains for those accounts, sustaining cache misses on every request.
5. With concurrent requests, each holding a DB connection for both the primary and secondary query, the 10-connection pool is saturated.

**Existing checks reviewed and shown insufficient:**

- `limit` is capped at 100 — this bounds the secondary query array size but does not prevent the doubled query count. [5](#0-4) 
- `maxRepeatedQueryParameters` (default 100) limits repeated query params per request but does not limit request rate or the number of distinct accounts queried.
- The `isValidAccount` check (tokenController.js:68–71) only verifies the account exists; it does not throttle or authenticate the caller. [6](#0-5) 
- No rate-limiting middleware is applied to this endpoint in the REST layer.
- DB pool `maxConnections` defaults to 10 and `statementTimeout` to 20,000 ms, meaning each saturated connection can be held for up to 20 seconds. [7](#0-6) 

### Impact Explanation

Each attacker request consumes two DB connections sequentially (primary + secondary). With 10 pool connections and 20 s statement timeout, as few as 5 concurrent attacker requests can saturate the pool, causing all other API queries to queue or time out. Because the endpoint is unauthenticated and the Hedera network has millions of publicly enumerable accounts with diverse token associations, the attacker can sustain this indefinitely without any on-chain cost beyond knowing account IDs. This constitutes a non-network-based DoS capable of taking the REST API offline for all users.

### Likelihood Explanation

The attack requires zero privileges, zero on-chain transactions, and only public information (account IDs with many token associations, which are discoverable via the same API). It is trivially scriptable with a standard HTTP client. The attacker does not need to create accounts or tokens — they can exploit existing mainnet/testnet accounts. The attack is repeatable and stateless from the attacker's perspective.

### Recommendation

1. **Add per-IP or global rate limiting** on the `/accounts/:id/tokens` endpoint.
2. **Batch-populate the cache proactively** or use a shared cache (e.g., Redis, which is already present in the stack) for `tokenCache` so that cache state is shared across instances and survives restarts.
3. **Coalesce concurrent cache-miss queries** (e.g., using a promise-deduplication map keyed on token ID) so that simultaneous requests for the same uncached token ID result in only one DB query.
4. **Add a TTL** to the `quickLru` cache so that the cache does not silently go stale, and consider increasing `maxSize` relative to the expected token universe.
5. **Increase the DB connection pool** or add a query queue with a short timeout to fail fast rather than holding connections under load.

### Proof of Concept

```bash
# Step 1: Identify accounts with many token associations (public info)
# e.g., account 0.0.1234 has 100 token associations

# Step 2: Collect a list of N accounts with distinct token sets
ACCOUNTS=("0.0.1001" "0.0.1002" "0.0.1003" ... "0.0.2000")

# Step 3: Flood the endpoint, rotating accounts to maximize cache misses
while true; do
  for ACCT in "${ACCOUNTS[@]}"; do
    curl -s "https://mainnet-public.mirrornode.hedera.com/api/v1/accounts/${ACCT}/tokens?limit=100" &
  done
  wait
done
```

Each iteration fires 2 DB queries per request. With 10 concurrent requests and a 10-connection pool, the pool is saturated. Legitimate API traffic begins receiving connection-timeout errors within seconds.

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });
