### Title
Spurious Token Deletion Event Injected into NFT Transfer History for Non-Existent Serial Numbers

### Summary
In `extractSqlFromNftTransferHistoryRequest()` (`rest/tokens.js`), the `entity` deletion subquery is only constrained by `Entity.ID = $1` (the token ID) and `deleted is true`, with no `serial_number` predicate. When a deleted token's ID is queried with any serial number — including one that was never minted — the deletion timestamp is returned and subsequently synthesized into a fabricated `TOKENDELETION` event for that serial, producing a false transaction history for a non-existent NFT.

### Finding Description
**Code path:** `rest/tokens.js`, `extractSqlFromNftTransferHistoryRequest()`, lines 928–984.

The function builds three unioned subqueries:

- `nft` table (line 963–964): `WHERE token_id = $1 AND serial_number = $2` — correctly scoped.
- `nft_history` table (line 967–972): `WHERE token_id = $1 AND serial_number = $2` — correctly scoped.
- `entity` table (lines 975–979): `WHERE id = $1 AND deleted is true` — **missing `serial_number` constraint entirely**.

```js
const tokenDeleteConditions = [`${Entity.ID} = $1`, `${Entity.DELETED} is true`];
// serial_number ($2) is never added to tokenDeleteConditions
```

When the outer query returns the deletion timestamp, `nftTransactionHistoryDetailsQuery` (lines 895–917) is executed. For transaction type 35 (`TOKENDELETION`), it unconditionally synthesizes an NFT transfer object using the caller-supplied `$2` (serial number) and `$1` (token ID):

```js
when ${Transaction.TYPE} = 35 then
    jsonb_build_array(jsonb_build_object(
        'is_approval', false,
        'receiver_account_id', null,
        'sender_account_id', null,
        'serial_number', $2::bigint,   // attacker-controlled serial
        'token_id', $1::bigint))
```

**Root cause / failed assumption:** The code assumes that if the `nft`/`nft_history` subqueries return no rows for a given serial, the entity deletion row will also be absent. This assumption is false: the entity row exists for the token regardless of which serial numbers were ever minted.

**Exploit flow:**
1. Attacker identifies a deleted token (token ID `X`). This is public information on any mirror node.
2. Attacker queries `GET /api/v1/tokens/X/nfts/<never_minted_serial>/transactions`.
3. `nft` and `nft_history` subqueries return 0 rows (serial never existed).
4. `entity` subquery returns 1 row: `lower(timestamp_range)` = deletion timestamp of token X.
5. The outer query returns this timestamp.
6. `nftTransactionHistoryDetailsQuery` fetches the `TOKENDELETION` transaction and synthesizes a transfer for `<never_minted_serial>`.
7. The API response contains a `TOKENDELETION` event for a serial that was never minted.

**Why existing checks are insufficient:** The `nft`/`nft_history` serial-number guards only protect their own subqueries. There is no post-union check that verifies at least one real NFT event exists before including the entity deletion row.

### Impact Explanation
Any consumer of the `/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` endpoint can be fed a fabricated transaction history for a serial number that never existed. Applications that rely on this endpoint to determine NFT provenance, ownership history, or existence (e.g., marketplaces, wallets, audit tools) may incorrectly conclude that a serial was minted and subsequently deleted, when in reality it never existed. This is a data-integrity / history-falsification issue on a public read API. No funds can be directly stolen (mirror node is read-only), but the fabricated history can be used to deceive downstream systems or users about the state of an NFT.

### Likelihood Explanation
Exploitation requires zero privileges — the endpoint is public and unauthenticated. The only precondition is knowledge of a deleted token's ID, which is trivially discoverable via the same mirror node API. The attack is fully repeatable and deterministic. Any attacker with basic knowledge of the Hedera mirror node REST API can reproduce it.

### Recommendation
Add `serial_number = $2` to `tokenDeleteConditions` so the entity deletion subquery is scoped to the same NFT being queried:

```js
const tokenDeleteConditions = [
  `${Entity.ID} = $1`,
  `${Entity.DELETED} is true`,
  // Ensure the deletion is only surfaced when the serial actually existed
  // by requiring at least one matching row in nft or nft_history.
  // Alternatively, add a correlated EXISTS check:
];
```

The cleanest fix is to replace the unconditional `entity` union with a correlated subquery that only returns the deletion timestamp when the serial number is confirmed to exist in `nft` or `nft_history`:

```sql
select lower(timestamp_range) as timestamp
from entity
where id = $1
  and deleted is true
  and exists (
    select 1 from nft        where token_id = $1 and serial_number = $2
    union all
    select 1 from nft_history where token_id = $1 and serial_number = $2
  )
```

This ensures the token deletion event is only injected into the history when the queried serial number genuinely belongs to that token.

### Proof of Concept
**Preconditions:**
- Token with ID `0.0.5000` exists and has been deleted on the network.
- Serial number `9999` was never minted for this token.

**Steps:**
```
GET /api/v1/tokens/0.0.5000/nfts/9999/transactions
```

**Expected (correct) result:** Empty `transactions` array (serial never existed).

**Actual result:**
```json
{
  "transactions": [
    {
      "consensus_timestamp": "<token_deletion_timestamp>",
      "nft_transfer": [
        {
          "is_approval": false,

          "sender_account_id": null,
          "serial_number": 9999,
          "token_id": "0.0.5000"
        }
      ],
      "type": "TOKENDELETION",
      ...
    }
  ]
}
```
