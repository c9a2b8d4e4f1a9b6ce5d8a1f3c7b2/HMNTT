### Title
Retry-Budget Exhaustion via Alternating HTTP 500 / HTTP 200 Empty-Node-List Responses Permanently Halts Pinger Startup

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` treats HTTP 500 as retryable and HTTP 200 with an empty node list as non-retryable. An attacker who can influence the mirror node REST endpoint (e.g., via MITM on a plaintext HTTP connection, DNS spoofing, or a compromised upstream) can serve HTTP 500 for the first N-1 attempts to exhaust the full exponential-backoff budget, then serve HTTP 200 with `{"nodes":[]}` on the final attempt, causing `buildNetworkFromMirrorNodes()` to return the non-retryable `"no usable service_endpoints found"` error. This propagates through `newClient()` to `log.Fatalf()` in `main()`, permanently terminating the pinger process.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go` — `buildNetworkFromMirrorNodes()` (lines 36–72) drives the retry loop:

```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)  // default: 11
for attempt := 1; attempt <= attempts; attempt++ {
    network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
    if err == nil { return network, nil }
    lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
    if !retry || attempt == attempts { break }   // ← breaks on non-retryable OR last attempt
    backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
    ...
}
return nil, lastErr
```

`fetchMirrorNodeNetwork()` (lines 74–132) sets the `retry` flag:

- **HTTP 5xx** → `retry = true` (line 91: `retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500`)
- **HTTP 200 + empty nodes** → `retry = false`, error = `"no usable service_endpoints found"` (lines 127–129)

**Root cause:** The empty-node-list path (HTTP 200, `{"nodes":[]}`) is unconditionally non-retryable. The retry guard `if !retry || attempt == attempts` breaks the loop immediately on any non-retryable error, regardless of remaining budget. There is no minimum-node-count retry, no distinction between a transient empty response and a permanent one.

**Exploit flow:**

1. Attacker positions themselves to control responses from the mirror node REST URL (see Likelihood).
2. For attempts 1 through N-1 (default N=11): respond HTTP 500. The pinger retries with exponential backoff (500 ms, 1 s, 2 s, … 256 s — total ≈ 511 s with default settings).
3. On attempt N (the last): respond HTTP 200 with `{"nodes":[]}`. `fetchMirrorNodeNetwork` returns `(nil, false, "no usable service_endpoints found")`.
4. The condition `!retry || attempt == attempts` is true (both are true on the last attempt), so the loop breaks.
5. `buildNetworkFromMirrorNodes()` returns `nil, lastErr` wrapping `"no usable service_endpoints found"`.
6. `newClient()` (`sdk_client.go` line 19) returns this error.
7. `main()` (`main.go` line 43) calls `log.Fatalf("client error: %v", err)` — process exits permanently.

**Why existing checks are insufficient:**

The `retry` boolean is the sole gate. There is no check for "did we get a structurally valid but empty response that might be transient?" The code correctly retries transport errors and 5xx, but a single HTTP 200 with zero nodes immediately terminates all remaining retries.

### Impact Explanation

The pinger process is killed via `log.Fatalf` and does not restart on its own. In a Kubernetes deployment the pod restarts, but the attacker can repeat the attack on every restart cycle, keeping the pinger permanently non-functional. The pinger's purpose is to submit periodic Hiero network transfers as a liveness/health signal; its absence silently degrades observability and health-check coverage for the network. Severity matches the stated scope: griefing/DoS with no direct economic damage to network users.

### Likelihood Explanation

The attack requires the ability to serve controlled HTTP responses from the mirror node REST endpoint. Realistic vectors for an unprivileged attacker:

- **Plaintext HTTP deployment** (`http://mirror-rest:5551` — the example in `config.go` line 37 uses HTTP): a network-adjacent attacker (same pod network, same node, or same L2 segment) can ARP-poison or perform DNS spoofing with no privileges.
- **DNS hijacking**: if the mirror REST hostname resolves via an attacker-influenced resolver, the attacker can point it at a controlled server.
- **Compromised or misconfigured mirror node**: the mirror node REST API is often publicly exposed; if it can be induced to return empty node lists (e.g., by exploiting a separate bug or during a maintenance window), the pinger will not retry.

No authentication, TLS pinning, or HTTPS enforcement is present in the pinger's HTTP client construction (line 46: `httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}`).

### Recommendation

1. **Retry on empty node list**: treat `len(network) == 0` as a retryable condition (return `retry = true`) rather than a hard stop. A transient empty response from the mirror node should not permanently halt startup.
2. **Enforce HTTPS / TLS verification**: reject `http://` mirror REST URLs or at minimum warn loudly; use `tls.Config` with proper certificate verification.
3. **Minimum-success threshold**: require at least one successful non-empty response before considering the network built; do not break the retry loop on the first non-retryable error if earlier attempts succeeded partially.
4. **Cap `log.Fatalf` escalation**: instead of fatally exiting, consider returning a sentinel error that triggers a controlled restart with its own backoff, preventing an attacker from trivially cycling the pod.

### Proof of Concept

**Precondition:** Pinger configured with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host/`.

**Attacker's HTTP server logic (pseudocode):**
```
requestCount = 0
GET /api/v1/network/nodes:
    requestCount++
    if requestCount < 11:          # N-1 = mirrorNodeClientMaxRetries
        return HTTP 500
    else:
        return HTTP 200, {"nodes": [], "links": {"next": null}}
```

**Steps:**
1. Start attacker server above (or MITM via ARP poisoning on HTTP).
2. Start pinger with `HIERO_MIRROR_PINGER_NETWORK=other HIERO_MIRROR_PINGER_REST=http://<attacker>/`.
3. Observe pinger retrying 10 times with exponential backoff (~511 s total with defaults).
4. On attempt 11, attacker returns HTTP 200 + empty nodes.
5. Pinger logs: `client error: attempt 11/11: no usable service_endpoints found from http://<attacker>/api/v1/network/nodes` and exits via `log.Fatalf`.
