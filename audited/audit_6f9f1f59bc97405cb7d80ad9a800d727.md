### Title
Unauthenticated Request Flooding Exhausts DB Connection Pool via Absent Rate Limiting on GET /tokens

### Summary
The `GET /api/v1/tokens` endpoint in `rest/tokens.js` has no rate limiting applied at any layer of the Node.js application. Any unauthenticated user can issue unlimited concurrent requests, each of which acquires a slot from the shared DB connection pool (default: 10 connections). Combined with expensive `ILIKE` queries triggered by the `name` filter, an attacker can saturate the pool and degrade service for all users.

### Finding Description

**Code path:**

`rest/server.js` line 119 registers the route with no rate-limiting middleware: [1](#0-0) 

The middleware stack in `server.js` contains: `cors`, `compression`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler` — no rate limiter anywhere: [2](#0-1) 

`getTokensRequest()` unconditionally calls `getTokens()` on every request: [3](#0-2) 

`getTokens()` calls `pool.queryQuietly()`, consuming one connection from the shared pool: [4](#0-3) 

The DB pool is configured with `max: config.db.pool.maxConnections`, whose documented default is **10**: [5](#0-4) [6](#0-5) 

**Expensive query vector — `name` ILIKE:**

`extractSqlFromTokenRequest` injects an `ILIKE '%value%'` predicate when the `name` filter is present, which forces a full sequential scan of the `token` table: [7](#0-6) 

The `name` filter is accepted and validated (3–100 bytes, `eq` operator only), so any unauthenticated caller can supply it: [8](#0-7) 

**Why the limit cap does not prevent the attack:**

`formatComparator` caps the `limit` parameter at `responseLimit.max` (default 100) for unauthenticated users: [9](#0-8) 

This bounds the result-set size but does **not** bound the number of concurrent connections consumed. Each in-flight request holds a pool slot for the full duration of the query (up to the 20-second `statementTimeout`). With only 10 pool slots, 10 simultaneous slow queries saturate the pool entirely.

**Response cache does not mitigate this:**

The cache is disabled by default (`cache.response.enabled: false`) and, when enabled, is keyed by the full URL hash: [10](#0-9) 

An attacker trivially bypasses it by varying the `name` parameter value on each request.

### Impact Explanation

With a pool of 10 connections and a 20-second statement timeout, 10 concurrent `GET /api/v1/tokens?name=abc` requests fully exhaust the pool. All subsequent requests from any user — including legitimate ones — queue waiting for a free connection or fail with a connection-timeout error (default 20 s). This constitutes a complete, sustained denial of service for the REST API's database-backed endpoints, achievable with a trivial HTTP flood from a single IP.

### Likelihood Explanation

No authentication, no API key, no CAPTCHA, and no IP-based throttle is required. The attack is reproducible with a simple shell loop or any HTTP benchmarking tool (`ab`, `wrk`, `hey`). The `name` ILIKE path is publicly documented in the OpenAPI spec, making it discoverable. The attacker needs no prior knowledge of the system beyond the public API documentation.

### Recommendation

1. **Add application-level rate limiting** (e.g., `express-rate-limit`) keyed by IP address, applied globally in `server.js` before route handlers.
2. **Increase the default pool size** or add a concurrency limiter (e.g., `p-limit`) so that a single client cannot monopolize all pool slots.
3. **Add a query-level concurrency guard** — reject or queue requests when the pool is near saturation rather than blocking indefinitely.
4. **Consider a reverse-proxy rate limit** (Traefik `RateLimit` middleware already exists in the chart templates but is not wired to the REST service by default).

### Proof of Concept

```bash
# Flood with 20 concurrent ILIKE queries (10 saturates the default pool)
for i in $(seq 1 20); do
  curl -s "http://<mirror-node-host>/api/v1/tokens?name=abc" &
done
wait

# Legitimate request now times out or receives a 503/connection error
curl -v "http://<mirror-node-host>/api/v1/tokens"
```

Expected result: the last `curl` either hangs for up to 20 seconds (connection timeout) or receives an error, while the 20 background requests hold all pool slots.
