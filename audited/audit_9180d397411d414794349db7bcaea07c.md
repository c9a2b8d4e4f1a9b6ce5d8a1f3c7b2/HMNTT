### Title
32-bit Cache Key Collision in `findByEvmAddressAndDeletedIsFalse` Enables Wrong Entity Resolution

### Summary
`findByEvmAddressAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` as its sole Caffeine cache key, which resolves to `Arrays.hashCode(byte[])` — a 32-bit integer. Two distinct EVM addresses that produce the same `Arrays.hashCode` value will share a cache slot, causing the first-cached entity to be returned for the second address. No secondary equality check is performed on cache hit.

### Finding Description
**Code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:
```java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```
`SpelHelper.hashCode(byte[])` at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java` line 20–22 is:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```
`Arrays.hashCode` produces a 32-bit `int` via a polynomial over the bytes. The Caffeine cache (`CACHE_NAME_EVM_ADDRESS` inside `CACHE_MANAGER_ENTITY`) is configured with `expireAfterWrite=1s,maximumSize=10000` (default in `CacheProperties.java` line 19). Spring's `@Cacheable` uses the key expression result as the map key with no further equality check against the original `byte[]` argument. Therefore, if `Arrays.hashCode(addrA) == Arrays.hashCode(addrB)`, a cache hit for `addrB` returns the stored `Optional<Entity>` for `addrA`.

The same flaw exists identically in `findByEvmAddressOrAliasAndDeletedIsFalse` (lines 39–49) using `CACHE_NAME_ALIAS`.

**Root cause:** The cache key is a lossy 32-bit projection of a 20-byte (160-bit) input. The failed assumption is that `Arrays.hashCode` is collision-resistant over the EVM address space.

**Exploit flow:**
1. Attacker precomputes offline two 20-byte arrays `addrA` and `addrB` such that `Arrays.hashCode(addrA) == Arrays.hashCode(addrB)`. This is trivial: `Arrays.hashCode` is a simple polynomial `s[0]*31^(n-1) + ... + s[n-1]`; collisions can be found in milliseconds by fixing all but one byte.
2. Attacker ensures `addrA` maps to a real entity they control (or any entity whose data they want the victim to receive).
3. Attacker sends a `eth_call` / contract call that causes `findByEvmAddressAndDeletedIsFalse(addrA)` to execute, populating the cache slot `hashCode(addrA)` with entity A.
4. Within the 1-second TTL window, a victim's request resolves `addrB` — the cache returns entity A's `Optional<Entity>` without a DB query.
5. EVM execution proceeds using entity A's code, balance, or state for what the victim believes is address B.

**Why existing checks fail:** `unless = "#result == null"` only prevents caching empty results; it does not prevent a collision from returning a wrong non-null result. There is no post-cache validation that the returned entity's `evmAddress` field matches the queried `alias`.

### Impact Explanation
A successful collision causes the EVM to resolve the wrong on-chain entity for a given address. Depending on context this can mean: wrong contract bytecode executed, wrong account balance used in `BALANCE`/`CALL` opcodes, wrong access-control owner checked, or wrong token/contract state read. In a DeFi contract call simulation this could produce incorrect results that mislead callers about balances or permissions.

### Likelihood Explanation
The 1-second `expireAfterWrite` TTL is the primary mitigating factor — the attacker must race the victim's query within a 1-second window. However: (a) collision precomputation is trivial and offline; (b) in high-throughput environments or predictable call patterns (e.g., automated bots, scheduled transactions) the timing window is realistic; (c) the attacker can repeatedly attempt the race. The `maximumSize=10000` limit does not prevent the attack since the attacker only needs one slot. No authentication or rate-limiting specific to this path is present.

### Recommendation
Replace the 32-bit hash key with the full byte array. Spring's default key generation already handles `byte[]` correctly via `Arrays.equals`/`Arrays.hashCode` in `SimpleKey`, but the explicit `key =` expression overrides this. Remove the custom `key` expression entirely and let Spring use the default key (which wraps the `byte[]` in a `SimpleKey` that compares by content), or use a hex-encoded string key:
```java
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```
Apply the same fix to `findByEvmAddressOrAliasAndDeletedIsFalse`.

### Proof of Concept
```java
// Offline: find addrA, addrB with same Arrays.hashCode
byte[] addrA = new byte[20]; // all zeros → hashCode = 1
byte[] addrB = new byte[20]; // craft bytes so Arrays.hashCode == 1
// Arrays.hashCode([0,0,...,0]) = 1 (by the polynomial formula)
// Arrays.hashCode([0,...,0, x]) == 1 when x satisfies 31*1 + x = 1 mod 2^32 → x = -30

addrB[19] = (byte) -30; // addrB != addrA but same hashCode

// Step 1: populate cache with addrA's entity
entityRepository.findByEvmAddressAndDeletedIsFalse(addrA); // DB hit, cached under key=1

// Step 2: within 1 second, query addrB
Optional<Entity> result = entityRepository.findByEvmAddressAndDeletedIsFalse(addrB);
// result contains addrA's entity — no DB query made
assert result.get().getEvmAddress() == addrA; // wrong entity returned
```
