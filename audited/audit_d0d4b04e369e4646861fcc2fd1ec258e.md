### Title
Cache Poisoning via `Optional.empty()` Caching in `findRuntimeBytecode()` Causes Persistent Bytecode Unavailability

### Summary
The `@Cacheable` guard `unless = "#result == null"` on `findRuntimeBytecode()` does not exclude `Optional.empty()` results, because `Optional.empty()` is a non-null Java object. Any unprivileged user who queries a contract's bytecode before the importer has written it to the database will cause the empty result to be cached for up to 1 hour (or indefinitely under repeated access), permanently shadowing the correct bytecode once the importer does write it.

### Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Cache configuration** (`CacheProperties.java`, line 22):
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```

**Root cause:** Spring's `@Cacheable` evaluates the `unless` SpEL expression against the actual return value. When the SQL returns no rows, Spring Data JPA wraps the result as `Optional.empty()`. `Optional.empty()` is a non-null singleton object, so `#result == null` evaluates to `false`, and the empty `Optional` is stored in the Caffeine cache under the `contractId` key.

**Call chain:** `ContractBytecodeReadableKVState.readFromDataSource()` (lines 42–46) calls `contractRepository.findRuntimeBytecode(entityId.getId())` and maps the result with `.orElse(null)`. Once the empty `Optional` is cached, every subsequent call returns `null` bytecode to the EVM layer, regardless of what the database now contains.

**Confirmed by test** (`ContractRepositoryTest.java`, lines 25–30): after `deleteAll()`, `contract2` (never previously queried) returns `Optional.empty()` — demonstrating that a pre-deletion empty query would have been cached and would persist.

### Impact Explanation
Any EVM `eth_call`, `eth_estimateGas`, or contract simulation routed through the web3 module that targets the poisoned contract will receive `null` bytecode. The EVM treats a contract with no bytecode as non-existent, causing all calls to that contract to revert or return empty results. This is a data-integrity / availability impact: the mirror node serves incorrect state for a real on-chain contract for the duration of the cache entry — up to 1 hour per access, or indefinitely if the attacker periodically re-queries the same `contractId` (resetting the `expireAfterAccess` timer each time).

### Likelihood Explanation
No privileges are required. The attacker only needs to:
1. Observe a new contract deployment on-chain (public mempool/block data).
2. Issue a single unauthenticated HTTP request to the mirror node's web3 endpoint referencing that `contractId` before the importer finishes processing the block.

The importer always has a non-zero processing lag. The attack window is the entire lag period, which is typically seconds to tens of seconds — easily exploitable by automated tooling. The attack is repeatable for every new contract deployment and requires no special knowledge beyond the contract's numeric ID.

### Recommendation
Change the `unless` condition to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This matches the pattern already used elsewhere in the codebase (e.g., `AddressBookEntryRepository` uses `unless = "@spelHelper.isNullOrEmpty(#result)"`). Alternatively, switch the cache policy to `expireAfterWrite` with a short TTL (e.g., 1–2 seconds, consistent with the `entity` and `contractState` caches) so that a transiently empty result expires quickly and the correct bytecode is served once the importer catches up.

### Proof of Concept
1. Deploy a new contract on the Hedera network. Note its numeric entity ID (e.g., `0.0.12345`).
2. Immediately (before the mirror node importer processes the block) send:
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000303d", "data": "0x..." }
   ```
   or any `eth_getCode`-equivalent call targeting the new contract address.
3. The SQL `select runtime_bytecode from contract where id = 12345` returns no rows; `Optional.empty()` is returned and cached.
4. Wait for the importer to process the block (verify via `GET /api/v1/contracts/0.0.12345` returning valid bytecode in the REST API).
5. Repeat the web3 call from step 2. Observe that the response still treats the contract as having no bytecode (EVM reverts / returns empty), confirming the stale cache entry is being served instead of the now-populated database row.
6. Periodically repeat step 2 every ~50 minutes to reset the `expireAfterAccess` timer and maintain the poisoned cache entry indefinitely.
