### Title
ILIKE Wildcard Injection via `name` Parameter Enables Broad Token Name Enumeration

### Summary
The `getTokensRequest()` function in `rest/tokens.js` passes the user-supplied `name` query parameter directly into a PostgreSQL `ILIKE` pattern without escaping ILIKE metacharacters (`_` and `%`). The only validation performed is a byte-length check (3–100 bytes) and an operator check (`eq`), neither of which prevents wildcard characters from being used. An unauthenticated user can supply `name=___` (three underscores) to produce the pattern `%___%`, which matches every token whose name contains at least three characters — effectively returning all tokens.

### Finding Description
**Exact code path:**

In `rest/tokens.js`, `extractSqlFromTokenRequest()` at line 177:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
``` [1](#0-0) 

The user-supplied value is concatenated directly into the ILIKE pattern with `%` prepended and appended. No escaping of `_` (single-character wildcard) or `%` (multi-character wildcard) is performed.

**Validation in `validateTokenQueryFilter()` at line 333–335:**
```js
case filterKeys.NAME:
  ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
  break;
``` [2](#0-1) 

`isByteRange(val, 3, 100)` only verifies the byte length is between 3 and 100. It does **not** inspect or reject ILIKE metacharacters. There is no subsequent escaping step anywhere in `tokens.js` or `utils.js`.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/tokens?name=___`
2. `isByteRange('___', 3, 100)` → `true` (3 bytes, within range)
3. Operator check: `eq` → passes
4. Value inserted into query: `'%' + '___' + '%'` → `'%___%'`
5. SQL executed: `t.name ILIKE '%___%'`
6. In PostgreSQL ILIKE, `_` matches any single character, so `%___%` matches any token name containing at least 3 characters — i.e., virtually all tokens.

Similarly, `name=%%%` (URL-encoded: `name=%25%25%25`) produces pattern `%%%%%`, matching every token name unconditionally.

### Impact Explanation
An unprivileged external user (no authentication required for the `/api/v1/tokens` endpoint) can retrieve the full list of token names, symbols, token IDs, admin keys, and metadata for all tokens on the network by issuing a single request. This bypasses the intent of the substring-search feature (which is meant to require a meaningful literal substring) and turns it into a full-table scan with disclosure of all token records up to the configured `limit`. Repeated paginated requests (using `token.id` cursor) can enumerate the entire token registry.

### Likelihood Explanation
The attack requires zero privileges, zero special tooling, and a single HTTP GET request. The payload `___` is a valid, printable ASCII string that passes all existing input validation. Any external user who reads the API documentation or experiments with the endpoint can discover and exploit this within minutes. It is trivially repeatable and automatable.

### Recommendation
Escape ILIKE metacharacters in the `name` value before constructing the pattern. In the `extractSqlFromTokenRequest` function, replace `_` with `\_` and `%` with `\%` in `filter.value`, and append `ESCAPE '\'` to the ILIKE clause:

```js
const escaped = filter.value.replace(/\\/g, '\\\\').replace(/%/g, '\\%').replace(/_/g, '\\_');
conditions.push(`t.name ILIKE $${params.push('%' + escaped + '%')} ESCAPE '\\'`);
```

Alternatively, add a validation rule in `validateTokenQueryFilter` that rejects values containing `_` or `%` characters if wildcard semantics are not intended.

### Proof of Concept
```
GET /api/v1/tokens?name=___
```
Expected (correct) behavior: returns only tokens whose names literally contain the substring `___`.

Actual behavior: returns all tokens whose names contain any 3 characters (i.e., virtually all tokens), because `_` is an unescaped ILIKE single-character wildcard, producing the pattern `%___%`.

To enumerate all tokens regardless of name length:
```
GET /api/v1/tokens?name=%25%25%25
```
(URL-decoded value: `%%%`, ILIKE pattern: `%%%%%`, matches everything.)
