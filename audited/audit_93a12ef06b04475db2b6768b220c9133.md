### Title
Unauthenticated `/api/v1/network/supply` Endpoint Triggers Unbounded DB Aggregate on Every Request with No Rate Limiting or Caching

### Summary
The `GET /api/v1/network/supply` endpoint is publicly accessible with no authentication, no rate limiting, and no result caching. Every request unconditionally executes a full aggregate SQL query (`sum(e.balance)` + `max(e.balance_timestamp)`) across all entity rows matching the configured unreleased-supply ID ranges. A steady stream of unauthenticated requests from a single attacker can keep the database aggregation executor continuously busy, raising DB and node CPU well above the 24-hour baseline.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) accepts any unauthenticated `GET /api/v1/network/supply` request and unconditionally calls `networkService.getSupply(bound)`. [1](#0-0) 

`NetworkServiceImpl.getSupply()` (lines 59–89), when no `timestamp` parameter is supplied (the common case), calls `entityRepository.getSupply(lowerBounds, upperBounds)` with no short-circuit, no cache lookup, and no result reuse. [2](#0-1) 

`EntityRepository.getSupply()` (lines 19–29) executes a native aggregate query that computes `sum(e.balance)` and `max(e.balance_timestamp)` across every entity row whose `id` falls within the configured ranges, with no `LIMIT`, no early termination, and no index-only shortcut for the aggregate. [3](#0-2) 

**Root cause — failed assumptions:**

1. *No rate limiting in rest-java.* The only throttle infrastructure in the codebase lives in the `web3` module (`ThrottleConfiguration`, `ThrottleManagerImpl`). The `rest-java` module has no equivalent: `WebMvcConfiguration` registers only argument resolvers, and `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — neither limits request rate. [4](#0-3) [5](#0-4) 

2. *No caching.* Neither `NetworkServiceImpl` nor `EntityRepository` carries `@Cacheable` or any equivalent. Every HTTP request results in a fresh DB round-trip. [6](#0-5) 

3. *ETag filter does not protect the DB.* `ShallowEtagHeaderFilter` computes the ETag from the response body *after* the query has already run, so it saves bandwidth only when the client cooperates with `If-None-Match` — it does not prevent the aggregate from executing. [4](#0-3) 

**Exploit flow:**

An attacker sends a continuous flood of `GET /api/v1/network/supply` requests (no credentials, no special headers). Each request causes PostgreSQL to execute the `unnest`-join aggregate. Because the entity table on a production mirror node contains millions of rows and the query must scan all rows in the configured ID ranges, each execution consumes measurable CPU on the DB host. With no rate limiting, the attacker can saturate the DB connection pool and keep the aggregate executor busy indefinitely.

### Impact Explanation

The DB aggregate runs on every unauthenticated request with no bound on concurrency. Sustained high-rate requests raise PostgreSQL CPU, consume connection-pool slots shared with all other mirror-node queries (importer, other REST endpoints), and can degrade or deny service to legitimate users. The impact is a resource-exhaustion DoS against the database tier of the mirror node, meeting the ≥30% CPU increase threshold when the request rate is high enough to keep multiple parallel aggregate workers busy. [3](#0-2) 

### Likelihood Explanation

The preconditions are minimal: no account, no API key, no special protocol knowledge. The endpoint is documented in the public OpenAPI spec. [7](#0-6) 
Any attacker with a script and a network connection can reproduce the attack. The absence of any rate-limiting layer in the rest-java module means the attack is repeatable and scalable across multiple source IPs or HTTP/2 multiplexed streams.

### Recommendation

1. **Add result caching** — annotate `NetworkServiceImpl.getSupply()` (or the repository method) with `@Cacheable` with a short TTL (e.g., 10–30 seconds). Supply data changes only when the importer writes new balances, so stale-by-seconds results are acceptable and would eliminate nearly all redundant DB work.
2. **Add per-IP or global rate limiting** — introduce a `bucket4j` or Spring `HandlerInterceptor`-based rate limiter in the rest-java module, mirroring the pattern already used in the `web3` module's `ThrottleConfiguration`. [8](#0-7) 
3. **Set a DB statement timeout** — configure `spring.datasource.hikari.connection-init-sql=SET statement_timeout=5000` so runaway aggregate queries are killed automatically.

    public NetworkSupply getSupply(Bound timestamp) {
        final NetworkSupply networkSupply;

        final var bounds = networkProperties.getUnreleasedSupplyRangeBounds();
        final var lowerBounds = bounds.lowerBounds();
        final var upperBounds = bounds.upperBounds();

        if (timestamp.isEmpty()) {
            networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
        } else {
            var minTimestamp = timestamp.getAdjustedLowerRangeValue();
            final var maxTimestamp = timestamp.adjustUpperBound();

            // Validate timestamp range
            if (minTimestamp > maxTimestamp) {
                throw new IllegalArgumentException("Invalid range provided for timestamp");
            }

            final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
            minTimestamp = Math.max(minTimestamp, optimalLowerBound);

            networkSupply =
                    accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp);
        }

        if (networkSupply.consensusTimestamp() == 0L) {
            throw new EntityNotFoundException("Network supply not found");
        }

        return networkSupply;
    }
