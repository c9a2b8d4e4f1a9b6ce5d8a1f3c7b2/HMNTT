### Title
DB Connection Pool Exhaustion via Unauthenticated Flood of `/accounts/:id/tokens` During Network Partition

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` issues multiple sequential DB queries per request with no application-level rate limiting or in-flight request cap. The Traefik middleware chain for the REST service omits both `inFlightReq` and `rateLimit` controls (present in other services like Rosetta), and the circuit breaker condition is latency-blind. During a network partition where DB queries run slowly, an unauthenticated attacker can hold all 10 default DB pool connections open simultaneously, denying service to all REST API consumers.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js` line 19 registers the route with no middleware guard: [1](#0-0) 

`rest/controllers/tokenController.js` `getTokenRelationships()` (lines 66–92) issues at minimum two sequential DB round-trips per request — `EntityService.getEncodedId()`, `EntityService.isValidAccount()` — before `TokenService.getTokenAccounts()` issues a third: [2](#0-1) 

`rest/dbpool.js` creates a `pg` pool whose hard ceiling is `config.db.pool.maxConnections`, documented default **10**, with `connectionTimeoutMillis` and `statement_timeout` both defaulting to **20 000 ms**: [3](#0-2) [4](#0-3) 

`rest/server.js` installs no rate-limiting or concurrency-limiting middleware anywhere in the chain: [5](#0-4) 

`rest/middleware/authHandler.js` only sets a per-user response-size limit for authenticated users; unauthenticated requests pass through unconditionally: [6](#0-5) 

The Traefik middleware chain for the REST Helm chart contains only `circuitBreaker` and `retry` — no `inFlightReq` and no `rateLimit`: [7](#0-6) 

Compare with the Rosetta chart, which adds both `inFlightReq` (5 per IP) and `rateLimit` (10/s per host): [8](#0-7) 

**Root cause / failed assumption:** The design assumes that the Traefik circuit breaker will protect the DB pool under degraded conditions. The circuit breaker expression `NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` is purely error-ratio-based: [9](#0-8) 

During a network partition, DB queries do not immediately error — they block for up to the 20-second `statementTimeout`. No network errors and no 5xx responses are produced until the timeout fires, so the circuit breaker threshold is never crossed while the pool is being held. The `retry: attempts: 10` setting then causes Traefik to re-issue each timed-out request up to 10 more times, multiplying pool pressure by up to 10×: [10](#0-9) 

### Impact Explanation

With a default pool of 10 connections and a 20-second statement timeout, an attacker needs only 10 concurrent in-flight requests to hold every connection. All subsequent legitimate requests queue inside the Node.js process waiting for `connectionTimeoutMillis` (20 s) before failing. The entire REST API — every endpoint sharing the same pool — becomes unavailable for the duration of the attack. Memory pressure from queued requests can additionally destabilize the Node.js process. Severity: **High** (complete availability loss for the public REST API).

### Likelihood Explanation

No authentication, no CAPTCHA, no IP-based throttle, and no concurrency cap is required to reach the vulnerable code path. The attacker needs only a standard HTTP client capable of sending 10 concurrent GET requests. The attack is trivially scriptable (`ab`, `wrk`, `curl` in parallel), requires no knowledge of valid account IDs (any syntactically valid ID triggers the DB calls before the 404 path), and is repeatable indefinitely. The network-partition precondition is the only environmental dependency, but the attacker can also induce slow queries via large `limit` parameters or repeated `token.id` filters to approximate the same effect under normal conditions.

### Recommendation

1. **Add `inFlightReq` and `rateLimit` to the REST Helm chart middleware** (mirroring the Rosetta chart), e.g.:
   ```yaml
   - inFlightReq:
       amount: 10
       sourceCriterion:
         ipStrategy:
           depth: 1
   - rateLimit:
       average: 50
       burst: 20
       sourceCriterion:
         ipStrategy:
           depth: 1
   ```
   in `charts/hedera-mirror-rest/values.yaml` alongside the existing `circuitBreaker`.

2. **Add a latency-based circuit breaker expression** such as `LatencyAtQuantileMS(50.0) > 5000` so the breaker opens before the pool is exhausted, not after errors accumulate.

3. **Reduce `retry.attempts`** for the REST chart or gate retries on idempotency/error type to avoid amplifying pool pressure.

4. **Increase `db.pool.maxConnections`** or add a Node.js-level concurrency semaphore (e.g., `express-rate-limit` or a custom middleware) as a defence-in-depth layer independent of the ingress controller.

### Proof of Concept

```bash
# Precondition: network partition or artificially slow DB (e.g., pg_sleep via a proxy)
# Target: public REST API, no credentials required

ENDPOINT="https://<mirror-node-host>/api/v1/accounts/0.0.1/tokens"

# Open 15 concurrent long-running requests (exceeds default pool of 10)
for i in $(seq 1 15); do
  curl -s "$ENDPOINT?limit=100" &
done
wait

# Expected result during partition:
# - First 10 requests hold all pool connections for up to 20 s each
# - Requests 11-15 queue in Node.js, eventually return 500/503 after connectionTimeout
# - All other REST API endpoints (e.g., /api/v1/transactions) also fail
#   because they share the same pool
# - Traefik retry (x10) re-issues timed-out requests, extending the outage window
```
