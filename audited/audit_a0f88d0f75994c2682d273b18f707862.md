### Title
Unauthenticated HTTP Mirror-Node Response Allows Arbitrary Node Topology Injection via Unvalidated `AccountID` Values

### Summary
When `network=other`, `pinger` fetches its entire Hiero node topology from a mirror REST endpoint over plain HTTP with no TLS enforcement, no response authentication, and no range validation on parsed `AccountID` values. An attacker who can serve a crafted `nodesEnvelope` — via MITM on the unencrypted HTTP channel or by controlling the mirror REST service — can inject `node_account_id` strings such as `999999.999999.999999` that parse successfully through `hiero.AccountIDFromString()`, populate the SDK's network map with bogus node identities, and cause every subsequent transaction to be routed with an invalid node account ID, resulting in persistent transaction failures (DoS) or, if the attacker also controls the injected host:port, silent receipt of signed transactions.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 46, 96–124:

```go
// line 46 — plain http.Client, no TLS config
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}

// lines 96-98 — response body decoded without any authentication
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }

// lines 102-124 — only guard is empty string and parse error; no range check
for _, n := range payload.Nodes {
    if n.NodeAccountID == "" { continue }          // only guard #1
    nodeAccountId, err := hiero.AccountIDFromString(n.NodeAccountID)
    if err != nil { continue }                     // only guard #2
    // ← no shard/realm/num range validation here
    for _, ep := range n.ServiceEndpoints {
        addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
        network[addr] = nodeAccountId             // bogus ID enters map
    }
}
```

`pinger/sdk_client.go`, `newClient()`, line 22:

```go
client = hiero.ClientForNetwork(netmap)  // SDK built from unvalidated map
```

**Root cause:** `hiero.AccountIDFromString("999999.999999.999999")` succeeds — the Hiero Go SDK parses shard/realm/num as unconstrained `uint64` values with no upper-bound check. The only two guards in `fetchMirrorNodeNetwork()` are an empty-string check and a parse-error check; neither rejects out-of-range values. The HTTP client is constructed with no TLS configuration, and the default/documented mirror REST URL uses plain `http://`, making the channel trivially interceptable.

**Exploit flow:**

1. Attacker positions themselves to respond to the pinger's `GET /api/v1/network/nodes` request (MITM on plain HTTP, DNS spoof, or compromise of the internal mirror REST service).
2. Attacker returns a crafted `nodesEnvelope`:
   ```json
   {"nodes":[{"node_account_id":"999999.999999.999999",
               "service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}]}
   ```
3. `hiero.AccountIDFromString("999999.999999.999999")` returns a valid `AccountID{Shard:999999, Realm:999999, Account:999999}` — no error.
4. `network["<attacker-ip>:50211"] = AccountID{999999,999999,999999}` is inserted.
5. `hiero.ClientForNetwork(netmap)` builds the SDK client with this as the sole (or dominant) node.
6. Every `cryptoTransfer.Execute(client)` in `transfer.go` line 33 sets the transaction's node account ID to `999999.999999.999999` and sends it to `<attacker-ip>:50211`.

**Why existing checks are insufficient:**

| Check | Location | What it catches | What it misses |
|---|---|---|---|
| `n.NodeAccountID == ""` | line 103 | blank string | any non-empty string |
| `err != nil` after `AccountIDFromString` | lines 108-110 | malformed format | valid-format but out-of-range values |
| HTTP status 2xx check | lines 90-93 | server errors | attacker-controlled 200 OK |

No TLS pinning, no HMAC/signature on the response, no allowlist of known node account IDs, no shard/realm == 0 assertion.

### Impact Explanation

**DoS (primary):** All transactions are submitted with node account ID `999999.999999.999999`. Real Hiero consensus nodes reject any transaction whose `nodeAccountID` field does not match their own identity. Every `Execute()` call fails; `submitWithRetry` exhausts all retries and logs failure. The pinger becomes permanently non-functional until restarted with a corrected mirror REST source.

**Transaction interception (secondary, higher severity):** If the attacker also controls the injected `ip_address_v4:port`, they receive every signed `CryptoTransfer` transaction — including the operator's account ID, the destination account, the amount, and the operator's cryptographic signature over the transaction body. While the private key itself is not transmitted, the signed transaction can be replayed or analysed.

**Severity: High** — complete loss of pinger availability plus potential signed-transaction exposure, triggered by a single crafted HTTP response.

### Likelihood Explanation

**Realistic attacker capability:**

- The default/example mirror REST URL is `http://mirror-rest:5551` (plain HTTP, internal Kubernetes service). Any pod in the same Kubernetes namespace that can intercept cluster-internal DNS or ARP can MITM this connection with no credentials.
- In environments where the mirror REST service is exposed on a non-TLS port (common for internal monitoring infrastructure), a network-adjacent attacker (same VPC, same subnet, compromised sidecar) can perform this attack without any privileged access to the pinger pod itself.
- The attack is **repeatable**: `buildNetworkFromMirrorNodes` is called once at startup; a single poisoned response permanently corrupts the client for the lifetime of the process.
- No authentication, no rate-limiting, no anomaly detection exists on the mirror REST fetch path.

**Likelihood: Medium-High** — requires network adjacency or mirror REST service compromise, both realistic in cloud-native deployments.

### Recommendation

1. **Enforce HTTPS with certificate validation** for `cfg.mirrorRest`; reject `http://` URLs at config load time (`config.go`, `loadConfig()`).
2. **Validate AccountID ranges** after parsing — at minimum assert `shard == 0 && realm == 0` and `num` within a configurable allowlist or upper bound:
   ```go
   if nodeAccountId.Shard != 0 || nodeAccountId.Realm != 0 || nodeAccountId.Account > maxKnownNode {
       continue
   }
   ```
3. **Allowlist known node account IDs** from a static, operator-supplied configuration rather than trusting the mirror REST response entirely for the `network=other` path.
4. **Verify response integrity** — if the mirror REST API supports signed responses or ETags with a known public key, verify them before consuming the payload.
