### Title
Cache Stampede on `CACHE_MANAGER_RECORD_FILE_LATEST` via Unsynchronized `@Cacheable` Allows DB Load Amplification and Potential Stale Block Caching

### Summary
The `cacheManagerRecordFileLatest()` bean configures a Caffeine cache with a 500ms TTL and `maximumSize=1`, but `RecordFileRepository.findLatest()` uses `@Cacheable` without `sync=true`. Spring's `@Cacheable` without synchronization does not prevent multiple threads from simultaneously detecting a cache miss and all executing the underlying database query. An unprivileged attacker sending concurrent `eth_blockNumber` (or any `BlockType.LATEST`) requests at the TTL boundary triggers a thundering-herd against the database, and a last-writer-wins race can cache a stale block number for the next 500ms window.

### Finding Description

**Exact code path:**

`EvmConfiguration.java` lines 178–188 — cache configured with `expireAfterWrite(500ms)`, `maximumSize(1)`, no `refreshAfterWrite`, no Caffeine `LoadingCache` loader:
```java
final var caffeine = Caffeine.newBuilder()
    .expireAfterWrite(500, TimeUnit.MILLISECONDS)
    .maximumSize(1)
    .recordStats();
final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_RECORD_FILE_LATEST));
caffeineCacheManager.setCaffeine(caffeine);
```

`RecordFileRepository.java` lines 31–36 — `@Cacheable` without `sync=true`:
```java
@Cacheable(
    cacheNames = CACHE_NAME_RECORD_FILE_LATEST,
    cacheManager = CACHE_MANAGER_RECORD_FILE_LATEST,
    unless = "#result == null")
@Query(value = "select * from record_file order by consensus_end desc limit 1", nativeQuery = true)
Optional<RecordFile> findLatest();
```

**Root cause:** Spring's `@Cacheable` without `sync=true` uses no mutex around the cache-miss path. Caffeine's stampede protection (via `AsyncLoadingCache`) is only available when a `CacheLoader` is supplied at build time — it is not available through `CaffeineCacheManager` with `setCaffeine()`. When the 500ms TTL fires, every thread that calls `findLatest()` simultaneously sees `null` from the cache and independently executes the SQL query.

**Stale-block race (last-writer-wins):**
1. Thread A queries DB at time T → gets block N.
2. A new block N+1 is committed to the DB at T+ε.
3. Thread B queries DB at T+2ε → gets block N+1, writes it to cache.
4. Thread A completes and writes block N to cache, overwriting N+1.
5. Cache now holds stale block N for the next 500ms window.

**Why existing checks fail:** There is no `sync=true` on the annotation, no `refreshAfterWrite` for background refresh, no rate-limiting on the JSON-RPC endpoint, and no version-check or compare-and-swap on the cache write-back.

### Impact Explanation
- **DB load amplification:** Every 500ms boundary under concurrent load causes N simultaneous `SELECT * FROM record_file ORDER BY consensus_end DESC LIMIT 1` queries instead of one. Under sustained flood (e.g., 500 concurrent clients), this can saturate the database connection pool and degrade all EVM call processing.
- **Stale block number:** If the last-writer-wins race fires (requires a new block committed within the sub-millisecond window between two concurrent queries), all callers of `eth_blockNumber`, `eth_getBlockByNumber("latest")`, and any EVM execution using `BlockType.LATEST` will receive a block number that is one block behind for up to 500ms. This affects `ContractCallContext`, `RecordFileServiceImpl.findByBlockType(BlockType.LATEST)`, and downstream EVM frame block values.

### Likelihood Explanation
- **Stampede (DB amplification):** High likelihood. Any unprivileged user can send a burst of concurrent `eth_blockNumber` RPC calls. The 500ms TTL fires predictably and repeatedly, making the attack trivially repeatable with a simple HTTP flood tool.
- **Stale block via last-writer-wins:** Low-to-medium likelihood in isolation. Hedera produces blocks roughly every 2 seconds, so the probability that a new block is committed within the sub-millisecond gap between two concurrent DB queries is low per individual TTL expiry. However, under sustained flood the attack repeats 2 times per second, and over time the race will fire.

### Recommendation
1. Add `sync=true` to the `@Cacheable` annotation on `findLatest()`:
   ```java
   @Cacheable(
       cacheNames = CACHE_NAME_RECORD_FILE_LATEST,
       cacheManager = CACHE_MANAGER_RECORD_FILE_LATEST,
       sync = true,
       unless = "#result == null")
   ```
   With `sync=true`, Spring wraps the cache-miss path in a `Cache.get(key, Callable)` call, which Caffeine serializes per key, allowing only one thread to load while others wait for the result.
2. Alternatively, replace `CaffeineCacheManager` + `setCaffeine()` with a native `AsyncLoadingCache` built with a `CacheLoader`, which provides Caffeine's built-in single-flight stampede protection.
3. Consider adding `refreshAfterWrite` slightly below the TTL so Caffeine proactively refreshes in the background before expiry.
