### Title
Infinite Polling Loop When `endTime` Is Exhausted in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` uses `RepeatSpec.times(Long.MAX_VALUE)` with no termination condition tied to `endTime`. Once the polling `startTime` advances past `endTime` (after the first message is consumed), every subsequent `poll()` call issues a DB query that returns zero results, yet the Flux never completes. Any unprivileged gRPC subscriber can trigger this by supplying a `TopicMessageFilter` with `endTime` set just above `startTime`, causing an effectively infinite stream of empty DB queries per subscription.

### Finding Description

**Code path:**

`PollingTopicListener.listen()` — [1](#0-0) 

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← no endTime check
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

`PollingTopicListener.poll()` — [2](#0-1) 

```java
long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`toBuilder()` preserves the original `endTime`, so `findByFilter` correctly applies `consensusTimestamp < endTime` in the DB query: [3](#0-2) 

```java
if (filter.getEndTime() != null) {
    predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
}
```

**Root cause:** The DB correctly filters by `endTime`, but the repeat loop in `listen()` has no awareness of `endTime`. Once `startTime > endTime`, every poll returns an empty stream. `RepeatSpec.times(Long.MAX_VALUE)` treats an empty emission as a normal cycle and schedules the next poll after `interval`. This continues for `Long.MAX_VALUE` iterations — effectively forever.

**Failed assumption:** The design assumes the caller (or some upstream operator) will cancel the subscription when `endTime` is reached. No such cancellation exists in `listen()`.

**Contrast with `PollingTopicMessageRetriever`**, which has an explicit `isComplete()` guard: [4](#0-3) 

```java
.repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())...)
```

`PollingTopicListener` has no equivalent.

**Validation gate — existing checks are insufficient:**

`TopicMessageFilter.isValidEndTime()` only validates at subscription time that `endTime > startTime`: [5](#0-4) 

```java
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

This does not prevent the polling loop from running indefinitely after `endTime` has passed.

### Impact Explanation

Each subscription with a near-expired `endTime` holds a `boundedElastic` scheduler thread and issues a DB query every `interval` milliseconds (default 500 ms) for `Long.MAX_VALUE` iterations. An attacker opening `N` such subscriptions exhausts the DB connection pool and scheduler thread pool proportionally to `N`, causing denial of service for all other subscribers. Because the queries return empty result sets, there is no natural back-pressure or circuit-breaker to stop them.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible to any unprivileged client. The only precondition is network access to the gRPC port. The exploit is trivially repeatable: open many subscriptions with `endTime = startTime + 1`. No authentication, elevated privilege, or special knowledge is required.

### Recommendation

Add an `endTime`-aware termination condition to the `repeatWhen` spec in `listen()`, mirroring the `isComplete()` pattern in `PollingTopicMessageRetriever`. Specifically:

1. After each `poll()`, check whether `filter.getEndTime() != null && Instant.now() >= endTime`. If true, complete the Flux instead of scheduling the next poll.
2. Alternatively, use `Flux.takeUntil` or `.takeWhile` on a clock signal keyed to `endTime`.
3. As a defense-in-depth measure, enforce a server-side maximum subscription duration regardless of `endTime`.

### Proof of Concept

1. Connect to the mirror node gRPC endpoint (no credentials required).
2. Issue a `subscribeTopic` request with:
   - `topicID`: any valid topic
   - `consensusStartTime`: `T` (current time)
   - `consensusEndTime`: `T + 1ns` (minimum valid gap per `isValidEndTime()`)
3. Observe via metrics or DB query logs that `findByFilter` is called every `interval` ms indefinitely, always returning 0 rows.
4. Repeat step 2 in a loop to open hundreds of subscriptions.
5. Observe DB connection pool exhaustion and degraded response times for legitimate subscribers.
