### Title
Unbounded Timestamp Range in Historical Hook Storage Enables Unauthenticated DoS via Expensive `DISTINCT ON` Queries

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` accepts user-controlled timestamp bounds with no range-width validation and no rate limiting, passing them directly to `DISTINCT ON` SQL queries. An unprivileged attacker can supply a maximally wide timestamp range (e.g., `timestamp=gte:0`) and flood the endpoint with concurrent requests, each forcing a full table scan of `hook_storage_change`, exhausting database connection pool workers and rendering the mirror node unresponsive.

### Finding Description
**Code path:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, `getHookStorageChange()`, lines 84–118.

**Root cause:** No maximum timestamp range width is enforced. When a caller supplies only a lower bound (e.g., `timestamp=gte:0`), `timestamp.getAdjustedLowerRangeValue()` returns `0` and `timestamp.adjustUpperBound()` returns `Long.MAX_VALUE` (lines 98–99). These are forwarded verbatim to `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` or `findByKeyInAndTimestampBetween()` (lines 104–114), both of which execute:

```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key BETWEEN :keyLowerBound AND :keyUpperBound
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
```

PostgreSQL's `DISTINCT ON` must materialise and sort **all** rows satisfying the `WHERE` clause before applying `LIMIT`. With a full timestamp range, this is a complete scan of every historical change record for the targeted hook.

**Failed assumption:** The `Bound` constructor validates inverted ranges only when `primarySortField=true` and never caps range width. The `TimestampParameter` regex only validates format, not magnitude of the span. No rate limiter exists for `rest-java` REST endpoints (the `ThrottleManagerImpl`/`ThrottleConfiguration` are scoped to `web3` contract calls only). Compare `NetworkServiceImpl.getSupply()` (lines 72–78), which explicitly validates and clamps the timestamp range — `getHookStorageChange()` has no equivalent guard.

### Impact Explanation
Concurrent flood of historical hook-storage requests with `timestamp=gte:0` causes each request to issue a full-scan `DISTINCT ON` query. The database connection pool is exhausted, blocking all other queries across the mirror node. The mirror node becomes unable to serve any requests — including transaction ingestion status queries — effectively taking the read-API offline. Severity: **High** (availability impact on the mirror node's REST API).

### Likelihood Explanation
No authentication is required. The attacker needs only a valid `ownerId`/`hookId` pair, which is publicly discoverable via the same API. The attack is trivially scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in a loop). It is repeatable and requires no special privileges or on-chain resources.

### Recommendation
1. **Enforce a maximum timestamp range width** in `getHookStorageChange()`, mirroring the pattern in `NetworkServiceImpl.getSupply()`:
   ```java
   if (timestampUpperBound - timestampLowerBound > MAX_ALLOWED_TIMESTAMP_RANGE_NS) {
       throw new IllegalArgumentException("Timestamp range exceeds maximum allowed window");
   }
   ```
2. **Add per-IP or global rate limiting** to the `rest-java` HookStorage endpoint, analogous to the `ThrottleManagerImpl` in `web3`.
3. **Set a DB query timeout** on the `HookStorageChangeRepository` queries to bound the worst-case execution time per request.
4. Consider rewriting the `DISTINCT ON` query to use a lateral join or index-only scan that benefits from `LIMIT` pushdown.

### Proof of Concept
```bash
# Discover a valid ownerId and hookId via the public API
OWNER=0.0.12345
HOOK_ID=1

# Flood the historical path with maximally wide timestamp range
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/hooks/${OWNER}/${HOOK_ID}/storage?timestamp=gte:0" &
done
wait
# Result: DB connection pool exhausted; subsequent requests time out or return 503
```

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public RangeParameter<Long> adjustLowerRange() {
        if (this.hasEqualBounds()) {
            // If the primary param has a range with a single value, rewrite it to EQ
            lower = new NumberRangeParameter(EQ, this.getAdjustedLowerRangeValue());
            upper = null;
        }

        return lower;
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
    }
