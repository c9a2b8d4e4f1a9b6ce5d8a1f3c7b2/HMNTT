### Title
Timeout Bypass via Exception Swallowing in Binary Gas Estimation

### Summary
`BinaryGasEstimator.safeCall()` catches all `Exception` types, including `QueryTimeoutException` thrown by the Hibernate `StatementInspector` timeout guard. This allows the binary search loop to silently continue after a timeout fires, running up to `maxGasEstimateRetriesCount` (default 20) full iterations regardless of the configured `requestTimeout`. Any unprivileged user can trigger this via `eth_estimateGas`.

### Finding Description
**Code path:**
- `ContractExecutionService.processCall()` → `estimateGas()` → `binaryGasEstimator.search()` (`ContractExecutionService.java:53-54, 91-95`)
- `BinaryGasEstimator.search()` calls `contractCallContext.reset()` then `safeCall(mid, call)` per iteration (`BinaryGasEstimator.java:36, 41`)
- `safeCall()` wraps the EVM call in `catch (Exception ignored)` (`BinaryGasEstimator.java:67-74`)
- `HibernateConfiguration.statementInspector()` throws `QueryTimeoutException` when `System.currentTimeMillis() - startTime >= timeout` (`HibernateConfiguration.java:39-43`)

**Root cause:** `QueryTimeoutException` extends `RuntimeException` extends `Exception`. `safeCall`'s `catch (Exception ignored)` intercepts it before it can propagate out of the binary search loop. The failed assumption is that the timeout guard's exception will terminate the request; instead it is treated as a soft call failure and the loop continues.

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` with `"estimate": true`, targeting a contract that performs expensive historical state reads (e.g., specifying a historical block number).
2. `estimateGas()` performs an initial call, then enters `binaryGasEstimator.search()`.
3. Each of up to 20 iterations triggers DB queries for historical Hashgraph state. When the 10-second timeout fires mid-iteration, `QueryTimeoutException` is thrown by the `StatementInspector`.
4. `safeCall` catches it, logs a benign info message, returns `null`.
5. The loop treats `null` as `err=true`, adjusts `lo`, and proceeds to the next iteration — each potentially consuming another full timeout window of DB resources.

**Why existing checks fail:** The `StatementInspector` check (`HibernateConfiguration.java:39-43`) is the only timeout enforcement mechanism. It correctly fires, but its exception is unconditionally swallowed by `safeCall` (`BinaryGasEstimator.java:70`). There is no outer deadline check in `search()` itself, and no re-throw path.

### Impact Explanation
An unprivileged attacker can cause each `eth_estimateGas` request to consume up to 20× the intended maximum server-side processing time (20 iterations × up to 10 s each = up to ~200 s of DB query time per request). Against a historical block target, each iteration issues multiple timestamped DB reads. This enables resource exhaustion / DoS against the mirror node's database connection pool and thread pool with a small number of concurrent requests, degrading service for all users.

### Likelihood Explanation
The endpoint is public and unauthenticated. No special privileges, tokens, or on-chain assets are required. The attack is trivially repeatable: a single script sending concurrent `eth_estimateGas` requests with `estimate: true` and a historical block number is sufficient. The default `gasPerSecond` throttle (`ThrottleProperties`) limits gas throughput but does not bound wall-clock execution time per request, so it does not mitigate this.

### Recommendation
1. **Re-throw timeout exceptions in `safeCall`:** Distinguish `QueryTimeoutException` (and other fatal infrastructure exceptions) from EVM-level execution failures. Do not catch `QueryTimeoutException` silently — let it propagate out of `search()` to terminate the request immediately.
2. **Add a wall-clock deadline check inside `BinaryGasEstimator.search()`:** Before each iteration, check `System.currentTimeMillis() - ContractCallContext.get().getStartTime()` against the configured timeout and break/throw if exceeded.
3. **Narrow the `catch` in `safeCall`:** Catch only EVM-specific exceptions (e.g., `MirrorEvmTransactionException`) rather than the broad `Exception`.
