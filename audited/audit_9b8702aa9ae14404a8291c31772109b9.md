### Title
Accept-Encoding `gzip;q=0` Quality-Factor Bypass Delivers Uncompressed Opcode Trace Response

### Summary
`validateAcceptEncodingHeader()` in `OpcodesController.java` uses a naive substring check (`contains("gzip")`) to enforce that clients must accept gzip compression before receiving the large opcode trace response. An attacker can send `Accept-Encoding: gzip;q=0` or `Accept-Encoding: gzip;q=0,deflate`, which passes the substring check (the string contains "gzip") but instructs the HTTP server layer to explicitly NOT use gzip (quality factor 0 = not acceptable). The result is a full, uncompressed multi-megabyte opcode trace response delivered to the attacker.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76.

```java
// Line 76 — the only guard
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
    throw HttpClientErrorException.create(MISSING_GZIP_HEADER_MESSAGE, HttpStatus.NOT_ACCEPTABLE, ...);
}
```

**Root cause:** The check only tests whether the literal substring `"gzip"` appears anywhere in the header value. It does not parse RFC 7231 quality-factor (`q=`) parameters. Per RFC 7231 §5.3.4, `Accept-Encoding: gzip;q=0` means "gzip is explicitly not acceptable." The string still contains `"gzip"`, so `contains("gzip")` returns `true` and the guard is bypassed.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip;q=0`.
2. `validateAcceptEncodingHeader("gzip;q=0")` → `"gzip;q=0".toLowerCase().contains("gzip")` → `true` → no exception thrown.
3. `throttleManager.throttleOpcodeRequest()` is called (rate-limited but not blocked for a single request).
4. `opcodeService.processOpcodeCall(request)` performs a full EVM replay with stack/memory/storage trace.
5. Spring/Tomcat's compression layer reads `Accept-Encoding: gzip;q=0`, correctly interprets q=0 as "do not use gzip," and streams the response body **uncompressed**.
6. The attacker receives a raw, uncompressed multi-megabyte JSON opcode trace.

**Why the existing check fails:** The guard was designed to prevent uncompressed responses, but it conflates "the header mentions gzip" with "the server will actually compress the response." The HTTP server layer (Tomcat) correctly honours RFC 7231 quality factors; the application-layer check does not. The REST integration test suite itself documents this behaviour at `rest/__tests__/integration/responseCacheHandler.test.js` line 129, where `gzip;q=0,deflate` is explicitly labelled "gzip explicitly disabled in accept-encoding" and produces a response with no `content-encoding` header. ### Impact Explanation
The opcode endpoint re-executes a full EVM transaction and returns every opcode, stack frame, memory snapshot, and storage slot touched during execution. With `stack=true&memory=true&storage=true`, responses for complex contracts can reach tens of megabytes of raw JSON. Delivering these uncompressed:

- **Bandwidth saturation:** A single request can consume 10–50× more outbound bandwidth than the compressed equivalent. Repeated requests (even within the throttle window) from multiple source IPs can saturate the node's NIC or upstream link.
- **Memory pressure:** The full serialised response must be buffered in the JVM heap before transmission; uncompressed payloads increase GC pressure and can contribute to OOM conditions under concurrent load.
- **No authentication barrier:** The endpoint requires no credentials. Any external user who knows (or can enumerate) a valid transaction hash can trigger this. [3](#0-2) 

### Likelihood Explanation
- **Preconditions:** The feature flag `properties.isEnabled()` must be true (it is opt-in but intended for production debug use). The attacker needs one valid transaction hash, which is publicly available on-chain explorers.
- **Skill required:** Crafting `Accept-Encoding: gzip;q=0` requires only basic HTTP knowledge; no exploit tooling is needed — `curl -H "Accept-Encoding: gzip;q=0"` suffices.
- **Repeatability:** The throttle manager limits requests per second per node, but does not prevent sustained low-rate amplification from distributed sources, nor does it account for the bandwidth cost of each individual uncompressed response.
- **Detection difficulty:** The request looks syntactically valid and will not appear anomalous in standard access logs.

### Recommendation
Replace the substring check with proper RFC 7231 quality-factor-aware parsing. Spring already provides `org.springframework.http.MediaType`-style negotiation utilities; for `Accept-Encoding` specifically, use `org.springframework.web.util.pattern` or a dedicated content-negotiation helper:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        rejectMissingGzip();
    }
    // Parse each token and check that gzip appears with q > 0
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip") && !parts[0].trim().equals("*")) {
                return false;
            }
            for (int i = 1; i < parts.length; i++) {
                String param = parts[i].trim();
                if (param.startsWith("q=") || param.startsWith("Q=")) {
                    try {
                        return Double.parseDouble(param.substring(2)) > 0.0;
                    } catch (NumberFormatException e) {
                        return false;
                    }
                }
            }
            return true; // no q param means q=1.0
        });
    if (!gzipAccepted) {
        rejectMissingGzip();
    }
}
```

Alternatively, delegate entirely to Spring's built-in `ContentNegotiationStrategy` or use `HttpHeaders.parseAcceptEncoding()` if available, to avoid re-implementing RFC 7231 parsing.

### Proof of Concept
```bash
# 1. Obtain any valid transaction hash from a public explorer or the mirror node API
TX_HASH="0xabc123..."

# 2. Send the request with gzip explicitly disabled via q=0
curl -v \
  -H "Accept-Encoding: gzip;q=0" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# Expected (vulnerable) result:
#   HTTP 200 OK
#   Content-Encoding: (absent — no compression applied)
#   Content-Length: <large uncompressed byte count>
#   Body: raw multi-megabyte JSON opcode trace

# Contrast with correct behaviour (gzip accepted):
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"
# Expected: Content-Encoding: gzip, significantly smaller Content-Length
```

The absence of `Content-Encoding: gzip` in the first response confirms the bypass: the validation gate passed but the server delivered an uncompressed payload.
