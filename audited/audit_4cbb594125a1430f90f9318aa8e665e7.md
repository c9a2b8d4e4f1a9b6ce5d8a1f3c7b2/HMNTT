### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Range Scan on `/api/v1/accounts`

### Summary
The `getAccounts()` handler in `rest/accounts.js` accepts multiple `account.id` range filter values (e.g., `gte:0.0.0` and `lte:0.0.9999999999`) from any unauthenticated caller with no per-IP or per-endpoint rate limiting. Each such request holds one of the 10 default DB pool connections for up to 20 seconds (the configured `statementTimeout`). An attacker issuing as few as 10 concurrent requests can fully saturate the connection pool, causing all subsequent legitimate queries — including those from the mirror node's own internal services — to queue and time out, effectively halting the REST API.

### Finding Description

**Exact code path:**

`rest/accounts.js` → `getAccounts()` (lines 335–388): [1](#0-0) 

`utils.validateReq()` is called first (line 337), but it only rejects unknown parameters and enforces `maxRepeatedQueryParameters` (default: 100). It performs no rate limiting and no range-width validation: [2](#0-1) 

`utils.parseAccountIdQueryParam()` (line 340) translates `account.id=gte:0.0.0&account.id=lte:0.0.9999999999` directly into the SQL fragment `e.id >= $1 AND e.id <= $2` with no restriction on the span of the range: [3](#0-2) 

The resulting query (built in `getEntityBalanceQuery`) is a multi-join query involving the `entity` table, a `latest_token_balance` CTE, and a correlated `entity_stake` subquery: [4](#0-3) 

The DB pool is configured with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout` of **20 000 ms**: [5](#0-4) 

`server.js` registers no rate-limiting middleware for the REST API — only `authHandler` (optional Basic Auth), `requestLogger`, optional `metricsHandler`, optional `responseCacheCheckHandler`, `responseHandler`, and `handleError`: [6](#0-5) 

**Root cause:** The REST API has no per-IP or per-endpoint request rate limit. The `account.id` range filter is accepted without any span constraint. Each request occupies a DB connection for the full `statementTimeout` duration (20 s). With a pool of 10 connections, 10 concurrent requests are sufficient to saturate the pool.

**Why existing checks fail:**
- `maxRepeatedQueryParameters` (100) limits the *count* of repeated params, not the *width* of a range — two params (`gte:0` and `lte:9999999999`) are well within the limit.
- `statementTimeout` (20 s) is a mitigation against runaway queries, but it simultaneously guarantees that each malicious request holds a connection for up to 20 s, making pool exhaustion easier to sustain.
- The `LIMIT` clause (default 25, max configurable) reduces result-set size but does not prevent the complex join + CTE from consuming significant DB time, especially when the range spans the entire entity table.

### Impact Explanation

When all 10 pool connections are occupied, new requests queue and wait up to `connectionTimeoutMillis` (20 s) before failing. This makes the entire `/api/v1/accounts` endpoint (and any other endpoint sharing the same pool) unavailable to legitimate callers. Because the mirror node REST API is the primary interface for clients querying account state, sustained pool exhaustion prevents clients from confirming transaction results, effectively halting observable network activity from the mirror node's perspective.

### Likelihood Explanation

No authentication, no API key, and no rate limiting are required. The attacker needs only a standard HTTP client capable of issuing concurrent requests. The attack is trivially scriptable, repeatable, and can be sustained indefinitely by continuously issuing new requests as old ones time out. The default pool size of 10 makes the threshold extremely low.

### Recommendation

1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) to the REST API, applied globally or specifically to expensive list endpoints, keyed by IP address.
2. **Enforce a maximum `account.id` range span** inside `parseAccountIdQueryParam` or `getAccounts()`: if both a lower-bound and upper-bound operator are present, reject requests where `upper - lower` exceeds a configurable threshold.
3. **Increase `maxConnections`** or deploy a connection pooler (e.g., PgBouncer) to decouple HTTP concurrency from DB connection count.
4. **Reduce `statementTimeout`** for the REST API role to limit how long each connection can be held.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent wide-range scans
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts?account.id=gte:0.0.0&account.id=lte:0.0.9999999999&limit=25" &
done
wait

# Legitimate request now times out or receives a connection pool error
curl -v "http://<mirror-node-host>:5551/api/v1/accounts?account.id=0.0.98"
```

Expected result: the legitimate request hangs for ~20 seconds then fails with a 500 or connection-timeout error, while the 15 attacker requests each hold a pool connection for the full `statementTimeout` duration.

const validateReq = (req, acceptedParameters = emptySet, filterValidator) => {
  const badParams = [];
  // Check the validity of every query parameter
  for (const key in req.query) {
    if (!acceptedParameters.has(key)) {
      badParams.push({code: InvalidArgumentError.UNKNOWN_PARAM_USAGE, key});
      continue;
    }
    if (Array.isArray(req.query[key])) {
      if (!isRepeatedQueryParameterValidLength(req.query[key])) {
        badParams.push({
          code: InvalidArgumentError.PARAM_COUNT_EXCEEDS_MAX_CODE,
          key,
          count: req.query[key].length,
          max: config.query.maxRepeatedQueryParameters,
        });
        continue;
      }
      for (const val of req.query[key]) {
        if (!paramValidityChecks(key, val, filterValidator)) {
          badParams.push({code: InvalidArgumentError.INVALID_ERROR_CODE, key});
        }
      }
    } else if (!paramValidityChecks(key, req.query[key], filterValidator)) {
      badParams.push({code: InvalidArgumentError.INVALID_ERROR_CODE, key});
    }
  }

  if (badParams.length > 0) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }

const parseAccountIdQueryParam = (parsedQueryParams, columnName) => {
  return parseParams(
    parsedQueryParams[constants.filterKeys.ACCOUNT_ID],
    (value) => EntityId.parseString(value).getEncodedId(),
    (op, value) => {
      return Array.isArray(value)
        ? [`${columnName} IN (?`.concat(', ?'.repeat(value.length - 1)).concat(')'), value]
        : [`${columnName}${op}?`, [value]];
    },
    true
  );
