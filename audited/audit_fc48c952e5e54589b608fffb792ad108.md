### Title
Unauthenticated Alias Lookup DoS via Missing Rate Limiting and Caching in GraphQL EntityServiceImpl

### Summary
The GraphQL module's `EntityServiceImpl.getByAliasAndType()` issues an uncached, direct database query on every alias lookup with no application-level rate limiting. Unlike the `web3` and `grpc` modules which both implement caching and/or throttling on equivalent paths, the GraphQL `EntityRepository.findByAlias()` carries no `@Cacheable` annotation and no throttle guard, allowing any unauthenticated caller to drive unbounded concurrent DB queries and exhaust the HikariCP connection pool.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 47–48) passes the raw alias to `EntityService.getByAliasAndType()`: [1](#0-0) 

`EntityServiceImpl.getByAliasAndType()` (line 29–31) decodes the base32 string and immediately calls the repository with no guard: [2](#0-1) 

`EntityRepository.findByAlias()` (line 13–14) executes a native SQL query against the `entity` table with zero caching: [3](#0-2) 

**Root cause — failed assumption:** The design assumes infrastructure-level controls (Traefik middleware) are sufficient. They are not:

1. **No application-level rate limiting.** The `web3` module has a full `ThrottleConfiguration` / `ThrottleManagerImpl` with a `rateLimitBucket` (default 500 RPS) and a `gasLimitBucket`: [4](#0-3) 
The graphql module has no equivalent.

2. **No result caching.** The `web3` `EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse()` is annotated `@Cacheable`: [5](#0-4) 
The graphql `EntityRepository.findByAlias()` has no such annotation. The only cache in the graphql module is a query-document parse cache (`hiero.mirror.graphql.cache.query`), which caches parsed GraphQL ASTs, not query results: [6](#0-5) 

3. **Traefik `inFlightReq: 5` per IP is insufficient.** The Helm chart sets a per-source-IP concurrent request limit of 5: [7](#0-6) 
This is bypassable via distributed attack (many source IPs). Worse, the `retry: attempts: 3` middleware re-submits failed requests up to 3 additional times, directly amplifying DB load when the pool begins to saturate: [8](#0-7) 
This infrastructure config is also optional (Helm-only) and absent in non-Kubernetes deployments.

### Impact Explanation
Each alias lookup consumes one HikariCP connection for the duration of the DB query. With no caching, every unique random alias string (trivially generated) causes a full table scan on the `entity` table. Flooding with concurrent requests from multiple IPs exhausts the connection pool, causing all subsequent GraphQL and potentially shared-DB requests to queue or fail. This is a complete denial of service for the GraphQL API and can degrade other mirror node components sharing the same database.

### Likelihood Explanation
No authentication or API key is required to reach `/graphql/alpha`. The attack requires only an HTTP client capable of sending concurrent POST requests with a valid GraphQL query body — a trivial capability. The `inFlightReq: 5` per-IP limit means an attacker needs only ~20–50 source IPs (e.g., a small botnet or cloud VMs) to sustain hundreds of concurrent DB queries. The `retry: attempts: 3` middleware means the attacker's effective throughput is multiplied by up to 4×. This is highly repeatable and requires no special knowledge of the system.

### Recommendation
1. **Add `@Cacheable` to `EntityRepository.findByAlias()`** in the graphql module, mirroring the pattern used in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`.
2. **Add application-level rate limiting** to the graphql module, mirroring `ThrottleConfiguration` / `ThrottleManagerImpl` from the `web3` module, with a configurable `requestsPerSecond` property.
3. **Remove or reconfigure the `retry` middleware** for the graphql endpoint, or ensure retries only occur on network errors, not on application-level failures caused by load.
4. **Add a per-IP rate limit** (not just concurrency limit) at the Traefik layer as a defense-in-depth measure.

### Proof of Concept
```bash
# Requires: bash, curl, GNU parallel or xargs
# No authentication needed

ENDPOINT="http://<graphql-host>/graphql/alpha"
QUERY='{"query":"{account(input:{alias:\"RANDOMALIASHERE\"})}"}'

# Generate 1000 requests with unique random aliases from 50 source IPs (simulated via proxies)
# Even from a single IP, 5 concurrent streams × sustained rate exhausts the pool:

for i in $(seq 1 1000); do
  ALIAS=$(cat /dev/urandom | base32 | head -c 32)
  echo "{\"query\":\"{account(input:{alias:\\\"${ALIAS}\\\"})}\"}"
done | xargs -P 5 -I{} curl -s -X POST "$ENDPOINT" \
  -H 'Content-Type: application/json' \
  -d '{}'

# Observe: DB connection pool exhaustion (hikaricp_connections_pending metric spikes),
# subsequent requests return 500 or timeout,
# Traefik retry middleware re-submits each failed request 3 additional times.
```
