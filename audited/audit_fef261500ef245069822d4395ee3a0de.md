### Title
Unbounded DB Query Storm via Non-Existent Alias Lookups in `getAccountIdFromAlias()` — No Negative-Result Caching

### Summary
`getAccountIdFromAlias()` in `rest/service/entityService.js` unconditionally executes a database query for every alias lookup with no caching of negative (not-found) results. An unprivileged attacker can flood the REST API with syntactically valid base32 alias strings that decode to binary values absent from the entity table, causing every request to hit the database, exhausting the connection pool and degrading mirror node processing capacity.

### Finding Description

**Root cause — no negative-result caching:**

The `entityId` LRU cache in `rest/entityId.js` (lines 301–333) only caches parsed `EntityId` objects for `shard.realm.num` / encoded-id / evm-address formats — it is never consulted for alias-to-entity lookups. [4](#0-3) 

The Redis response cache (`responseCacheHandler.js`, line 95) only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true — 404 Not Found responses (the result of a missing alias) are never cached. [5](#0-4) 

**Input validation is insufficient:**

`AccountAlias.isValid()` only checks that the string matches the regex `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` and that `base32.decode()` succeeds. Any syntactically valid base32 string (e.g., `AAAAAAAAAAAAAAAA`, `AAAAAAAAAAAAAAAB`, …) passes validation and is decoded to a binary value that is then used as the DB query parameter. [6](#0-5) [7](#0-6) 

**No rate limiting in middleware:**

The REST middleware stack exports only `authHandler`, `handleError`, `openApiValidator`, `requestHandler`, `responseCacheHandler`, and `responseHandler` — no IP-based or per-client rate limiting is present. [8](#0-7) 

**DB connection pool is small:**

The default `maxConnections` for the REST API DB pool is 10. Saturating it with alias-miss queries blocks all other legitimate queries. [9](#0-8) 

### Impact Explanation
Every request with a non-existent alias consumes one DB connection for the duration of the query. With the default pool of 10 connections and no rate limiting or negative caching, a modest flood of concurrent requests (e.g., 50–100 req/s from a single client) can saturate the pool, causing all other REST API queries — including those for transactions, balances, and token data — to queue or time out. This directly degrades mirror node processing capacity across all consumers of the REST API, meeting the stated threshold of ≥30% degradation.

### Likelihood Explanation
The attack requires zero privileges — the REST API is publicly accessible. The attacker needs only to generate valid base32 strings (trivially enumerable: `AAAAAAAAAAAAAAAA`, `AAAAAAAAAAAAAAAB`, etc.) and send them at high rate to any endpoint that resolves an alias (e.g., `GET /api/v1/accounts/{alias}`, `GET /api/v1/balances?account.id={alias}`). No special knowledge of the network state is required. The attack is fully repeatable and automatable with standard HTTP tooling (e.g., `wrk`, `ab`, `curl` in a loop).

### Recommendation
1. **Cache negative results**: Introduce a short-lived (e.g., 30–60 s) in-process or Redis cache for alias-to-entity lookups, including misses (store a sentinel `null` value). The existing `entityId` LRU cache infrastructure in `rest/entityId.js` can be extended for this purpose.
2. **Rate limit alias-lookup endpoints**: Add per-IP or per-client rate limiting middleware (e.g., `express-rate-limit`) specifically for endpoints that trigger alias DB lookups.
3. **Enforce minimum alias length / entropy**: Reject aliases that decode to all-zero or trivially enumerable byte sequences at the validation layer.

### Proof of Concept

**Preconditions:** Public access to the mirror node REST API (port 5551 by default). No account registration required.

**Steps:**

```bash
# Generate 10,000 unique valid base32 aliases and flood the endpoint
for i in $(seq 1 10000); do
  # Each value is a valid base32 string; none exist in the entity table
  ALIAS=$(python3 -c "
import base64, struct
b = struct.pack('>Q', $i) + b'\x00' * 12
print(base64.b32encode(b).decode().rstrip('='))
")
  curl -s "http://<mirror-node>:5551/api/v1/accounts/${ALIAS}" &
done
wait
```

**Expected result:** Each request triggers `entityFromAliasQuery` against the DB, returns HTTP 404 (not cached), and the DB connection pool is exhausted within seconds, causing legitimate requests to time out or return 503.
