### Title
Silent Upper-Bound Overwrite in `Bound` Constructor Allows Unprivileged Users to Widen `token.id` Query Range

### Summary
The `Bound` constructor in `TokenAirdropServiceImpl`'s processing pipeline silently overwrites `upper` when two upper-bound `token.id` parameters (both `lt`/`lte`) are supplied. The `verifySingleOccurrence()` guard method exists but is never invoked for the `tokenIds` bound in `processRequest()`, so the second value wins unconditionally. An unprivileged caller can therefore force a wider upper bound than the first parameter suggests, retrieving more airdrop records than the API design intends.

### Finding Description

**Controller entry point** — `@Size(max = 2)` explicitly permits two `token.id` values: [1](#0-0) 

**`processRequest()` builds the `tokenIds` Bound with `primarySortField = false`:** [2](#0-1) 

**`Bound` constructor loop — silent overwrite:** [3](#0-2) 

When both params satisfy `hasUpperBound()`, the loop assigns `upper = param` twice. The second assignment silently discards the first. `cardinality` correctly records `{LTE: 2}`, but nothing reads it.

**The only constructor-level range check is gated on `primarySortField`:** [4](#0-3) 

Because `tokenIds` is constructed with `primarySortField = false`, this check is entirely skipped.

**`verifySingleOccurrence()` would catch the duplicate, but is never called:** [5](#0-4) 

Neither `processRequest()` nor `TokenAirdropServiceImpl.getAirdrops()` calls `verifySingleOccurrence()` on the `tokenIds` bound: [6](#0-5) 

### Impact Explanation
An attacker sends `token.id=lte:100&token.id=lte:200`. The loop sets `upper = lte:100`, then immediately overwrites it with `upper = lte:200`. The executed SQL uses `token_id <= 200`, returning all records up to token 200 even though the user-visible first parameter was `lte:100`. This bypasses the intended filtering constraint, leaking more airdrop records per request than the API design permits and undermining any rate-limit or cost model tied to the declared query range.

### Likelihood Explanation
No authentication or special privilege is required. The exploit is a single HTTP GET request with two query parameters, both syntactically valid. It is trivially repeatable and requires no knowledge beyond the public API schema. Any user who reads the OpenAPI spec (which documents `lt(e)` as a supported operator) can discover and exploit this.

### Recommendation
Call `verifySingleOccurrence()` on every `Bound` immediately after construction in `processRequest()`, regardless of `primarySortField`:

```java
var tokenIdsBound = new Bound(tokenIds, false, TOKEN_ID, TOKEN_AIRDROP.TOKEN_ID);
tokenIdsBound.verifySingleOccurrence();   // add this
```

Alternatively, enforce the check inside the `Bound` constructor unconditionally (not only when `primarySortField` is true), so no call site can forget it.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/airdrops/outstanding?token.id=lte:100&token.id=lte:200
```

**Step-by-step:**
1. Spring binds both values into `EntityIdRangeParameter[] tokenIds = [lte:100, lte:200]`.
2. `new Bound(tokenIds, false, TOKEN_ID, TOKEN_AIRDROP.TOKEN_ID)` is called.
3. Loop iteration 1: `lte:100` → `hasUpperBound()` true → `upper = lte:100`.
4. Loop iteration 2: `lte:200` → `hasUpperBound()` true → `upper = lte:200` (overwrites silently).
5. `verifySingleOccurrence()` is never called; `cardinality = {LTE: 2}` is ignored.
6. The repository query applies `token_id <= 200`, returning records the caller should not receive under the `lte:100` constraint.
