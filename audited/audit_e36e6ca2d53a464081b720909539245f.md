### Title
Unbounded `node.id` EQ Parameter Array Enables Database CPU Exhaustion via `ANY(:nodeIds)` Query

### Summary
The `NetworkNodeRequest.nodeIds` field accepts an unbounded list of `node.id=eq:X` query parameters with no `@Size` constraint, unlike analogous fields in other request DTOs in the same codebase. An unauthenticated attacker can submit requests with thousands of distinct EQ values, causing `findNetworkNodes()` to execute a `abe.node_id = any(:nodeIds)` PostgreSQL query with a large array inside an already-expensive multi-CTE query, exhausting database CPU and denying node topology data to legitimate clients.

### Finding Description

**Exact code path:**

`NetworkNodeRequest.nodeIds` is declared as a plain `List<NumberRangeParameter>` with no `@Size` annotation: [1](#0-0) 

Compare this to `RegisteredNodesRequest.registeredNodeIds`, which has `@Size(max = 2)`: [2](#0-1) 

And `HooksController`, which enforces `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` (100): [3](#0-2) 

The constant `MAX_REPEATED_QUERY_PARAMETERS = 100` exists precisely for this purpose but is not applied to `nodeIds`: [4](#0-3) 

**Root cause — failed assumption:** The `RequestParameterArgumentResolver` correctly identifies `List` fields as multi-value and allows any number of values without enforcing a count limit: [5](#0-4) 

**Exploit flow in `getNetworkNodes()`:** Every EQ-operator parameter is added to a `HashSet<Long>` (deduplication only, no size cap), then converted to a `Long[]` array: [6](#0-5) 

This array is passed directly to the repository: [7](#0-6) 

**The SQL clause that executes the large array scan:** [8](#0-7) 

The surrounding query is already expensive — it uses three CTEs (`latest_address_book`, `latest_node_stake`, `node_info`), a correlated subquery per row for `serviceEndpointsJson`, and multiple joins. Adding a large `ANY` array forces PostgreSQL to evaluate the array membership check for every candidate row.

### Impact Explanation

An unauthenticated attacker can force the database to execute a complex multi-CTE query with an arbitrarily large `ANY(ARRAY[...])` predicate on every request. Concurrent requests from multiple sources (or a single source with connection pooling) can saturate database CPU, causing query timeouts and denying node topology data to clients that need it to submit transactions to the correct network nodes. The `limit` cap of 25 on returned rows does not reduce the cost of evaluating the `ANY` predicate during the scan phase. [9](#0-8) 

### Likelihood Explanation

The `/api/v1/network/nodes` endpoint is public and requires no authentication. HTTP servers (Tomcat default: 8KB–64KB query string) allow hundreds to thousands of `node.id=eq:X` parameters per request. With unique values (no deduplication benefit), an attacker can craft requests with 500–5000 distinct node IDs within typical URL length limits. This is trivially scriptable with `curl` or any HTTP client and is repeatable at high frequency.

### Recommendation

Add a `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` constraint to the `nodeIds` field in `NetworkNodeRequest`, consistent with the pattern already used in `HooksController` and `RegisteredNodesRequest`:

```java
// NetworkNodeRequest.java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = MAX_REPEATED_QUERY_PARAMETERS)  // Add this
private List<NumberRangeParameter> nodeIds = List.of();
```

Also ensure the `RequestParameterArgumentResolver` triggers JSR-303 validation on list fields so the `@Size` constraint is enforced at the binding layer before the service is invoked.

### Proof of Concept

```bash
# Generate a request with 500 unique node.id EQ parameters
PARAMS=$(seq 1 500 | awk '{printf "node.id=eq:%d&", $1}' | sed 's/&$//')
curl -s "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS"

# Repeat concurrently to exhaust DB CPU
for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS" &
done
wait
```

Each request causes PostgreSQL to evaluate `abe.node_id = any(ARRAY[1,2,...,500])` inside the multi-CTE query. With 50 concurrent requests, database CPU spikes and legitimate node topology queries time out.
