### Title
Unbounded Multi-Connection Subscription Flood Enables gRPC DoS via Resource Exhaustion

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures the Netty gRPC server with only a per-connection concurrent-call limit (`maxConcurrentCallsPerConnection = 5`), but imposes no limit on the total number of inbound TCP connections, no per-IP connection cap, and no global subscription count ceiling. An unauthenticated attacker can open an arbitrary number of connections and saturate the server with `subscribeTopic` streams, exhausting the DB connection pool and thread executor, causing legitimate transaction data reads to stall or fail.

### Finding Description
**Exact code path:**

`GrpcConfiguration.java` lines 28–35 — `grpcServerConfigurer()` applies exactly one server-side constraint:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
`NettyProperties.java` line 14 sets the default to `5`. No `maxConnectionAge`, `maxConnectionIdle`, total connection cap, per-IP connection limit, or any rate-limiting interceptor is configured anywhere in the gRPC module. The `GrpcInterceptor` (test path) only sets `EndpointContext` and passes through.

**Root cause:** The failed assumption is that limiting concurrent calls *per connection* is sufficient to bound total server load. It is not — an attacker controls how many connections they open.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication, no connection-count limit).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streaming RPCs (the per-connection maximum).
3. Each `subscribeTopic` call in `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) triggers:
   - `entityRepository.findById()` — a synchronous DB read for topic existence
   - `topicMessageRetriever.retrieve(filter, true)` — historical DB query
   - A `Mono.delay(1s)` safety-check that fires another DB poll per subscription
   - A live listener subscription holding a reactive pipeline in memory
4. With N=2000 connections: 10,000 concurrent subscriptions, each holding DB connections and executor threads.
5. The `subscriberCount` `AtomicLong` (line 48) is a metric gauge only — it enforces no cap.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` bounds only per-connection concurrency; total load = 5 × (number of connections), which is unbounded.
- The gRPC module has no equivalent of the `web3` module's `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j rate limiter).
- No `ServerInterceptor` enforces per-IP or global subscription limits.

### Impact Explanation
Exhausting the `applicationTaskExecutor` thread pool and HikariCP DB connection pool causes all `subscribeTopic` calls — including those from legitimate users — to queue indefinitely or fail. Transaction data delivery is delayed or completely blocked. The Prometheus alert `GrpcHighDBConnections` (threshold 75%) confirms the DB connection pool is a recognized bottleneck. Severity: **High** — complete denial of the gRPC read path with no authentication prerequisite.

### Likelihood Explanation
The attack requires only a standard gRPC client library (e.g., `grpc-java`, `grpcurl`) and the ability to open TCP connections to port 5600. No credentials, tokens, or special knowledge are needed. The `subscribeTopic` RPC is publicly documented as part of the Hedera Consensus Service mirror API. The attack is trivially scriptable, repeatable, and can be sustained indefinitely since streams remain open until the attacker closes them. A single commodity machine with sufficient file descriptors can open thousands of connections.

### Recommendation
1. **Add a total-connection limit** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionAge(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAgeGrace(5, TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(10, TimeUnit.SECONDS);
   ```
2. **Add a global active-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add a gRPC-level rate-limiting `ServerInterceptor`** (analogous to `ThrottleManagerImpl` in the web3 module) that tracks new-subscription rate per source IP using a token-bucket and rejects excess calls with `RESOURCE_EXHAUSTED`.
4. **Expose `maxConcurrentCallsPerConnection` alongside a new `maxTotalConnections` property** in `NettyProperties` and wire it via `serverBuilder`.
5. Deploy a network-layer connection-rate limit (e.g., `iptables --hashlimit` or an Envoy/Istio policy) as a defense-in-depth measure.

### Proof of Concept
```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Flood: open 500 connections, each with 5 concurrent subscribeTopic streams

for i in $(seq 1 500); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}, "consensusStartTime": {"seconds": 0}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: DB connection pool exhaustion (hikaricp_connections_active approaches max),
# executor thread starvation, and legitimate subscribeTopic calls timing out or failing.
# The subscriberCount metric will show 2500 active subscribers with no server-side rejection.
```
