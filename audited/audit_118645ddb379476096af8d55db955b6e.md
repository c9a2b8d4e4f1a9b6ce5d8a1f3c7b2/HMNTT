### Title
Unbounded Concurrent Subscriptions with Ancient `startTime` Enable DB Connection Pool Exhaustion via Repeated Polling

### Summary
Any unauthenticated gRPC client can open an unlimited number of concurrent `subscribeTopic` streams, each specifying `startTime=0` (epoch), no `endTime`, and no `limit`. Each stream drives `PollingTopicMessageRetriever` to issue a DB query of up to 1,000 rows every 2 seconds for up to 60 seconds. Because there is no per-client or global subscription count enforcement, an attacker opening hundreds of concurrent streams can saturate the database connection pool and starve legitimate integrity-verification queries.

### Finding Description

**Code path and root cause:**

`TopicMessageFilter` validation accepts any `startTime` that is `<= DomainUtils.now()`, including epoch 0: [1](#0-0) 

There is no lower-bound age restriction on `startTime`, no `limit` requirement, and no `endTime` requirement.

`TopicMessageServiceImpl.subscribeTopic()` immediately starts a historical retrieval for every incoming subscription with no cap on concurrent subscribers — `subscriberCount` is only a metrics gauge, never enforced as a limit: [2](#0-1) [3](#0-2) 

`PollingTopicMessageRetriever.retrieve()` with `throttled=true` sets `numRepeats = Long.MAX_VALUE` and polls every 2 seconds with `maxPageSize = 1000`: [4](#0-3) [5](#0-4) 

`isComplete()` for the throttled path only terminates when the last page returned fewer than `maxPageSize` rows or the limit is hit. With millions of historical messages and no limit set, every poll returns a full 1,000-row page and the loop continues: [6](#0-5) 

The only per-subscription bound is a 60-second timeout: [7](#0-6) 

**Why existing checks are insufficient:**

- The 60-second timeout bounds a *single* subscription to ~30 DB queries, but the attacker simply opens new streams as old ones expire — there is no cooldown or reconnect throttle.
- `subscriberCount` is a Micrometer gauge only; it is never compared against a maximum to reject new subscriptions.
- No per-IP, per-client, or global rate limit exists anywhere in the subscription path.

### Impact Explanation

Each concurrent subscription holds a DB connection for the duration of each `findByFilter` call and issues a new one every 2 seconds. With N concurrent streams, the DB receives N queries every 2 seconds, each scanning/returning up to 1,000 rows from a potentially large `topic_message` table. At a modest 200 concurrent streams the DB connection pool (typically 10–20 connections for a mirror-node gRPC pod) is saturated, causing legitimate integrity-verification queries and importer writes to queue or fail. This directly threatens the availability and correctness of Hashgraph history served by the mirror node.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the public gRPC API (`ConsensusService/subscribeTopic`), and no sophisticated tooling — a simple loop opening gRPC streams with `consensusStartTime = Timestamp{seconds: 0}` suffices. It is trivially repeatable and scriptable. The attacker does not need to maintain the streams beyond the 60-second window; continuously reopening them sustains the load indefinitely.

### Recommendation

1. **Enforce a maximum concurrent subscription count** — compare `subscriberCount` against a configurable ceiling and return `RESOURCE_EXHAUSTED` for excess requests.
2. **Enforce a minimum `startTime` age** — reject subscriptions whose `startTime` is older than a configurable threshold (e.g., 24 hours) or require a non-zero `limit` when `startTime` is far in the past.
3. **Add per-client (IP or gRPC peer) rate limiting** at the gRPC interceptor layer to cap new subscription attempts per time window.
4. **Bound the historical retrieval independently of the live phase** — use a separate, shorter timeout for the historical polling phase and enforce a maximum number of polls regardless of page fullness.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, mirror_network_service_pb2 as pb

def flood_subscribe(target, topic_id, n=300):
    def open_stream(_):
        channel = grpc.insecure_channel(target)
        stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
        req = pb.ConsensusTopicQuery(
            topic_id=topic_id,
            consensus_start_time=pb.Timestamp(seconds=0, nanos=0),
            # no limit, no end_time
        )
        try:
            for _ in stub.subscribeTopic(req):
                pass  # drain; reconnect when timed out
        except Exception:
            pass  # timeout after 60s; loop reopens

    threads = [threading.Thread(target=open_stream, args=(i,)) for i in range(n)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

# Run against any publicly accessible mirror-node gRPC endpoint
flood_subscribe("mainnet-public.mirrornode.hedera.com:443", topic_id=<valid_topic>)
```

With 300 concurrent streams, the DB receives ~150 queries/second each returning 1,000 rows, exhausting a standard connection pool within seconds and blocking legitimate queries.
