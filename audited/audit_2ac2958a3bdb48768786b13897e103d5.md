### Title
GraphQL Execution Timeout Absence Enables DB Connection Pool Exhaustion DoS During Network Partition

### Summary
The `graphQlCustomizer()` method in `GraphQlConfiguration.java` configures only query complexity and depth limits, with no GraphQL-level execution timeout instrumentation. During a network partition between the GraphQL service and the database, in-flight queries block their JDBC threads indefinitely — the PostgreSQL `statement_timeout` cannot be delivered back to the client over a severed TCP connection, and the GCP gateway's HTTP timeout terminates the client-facing response but does not release the server-side DB connection. The Traefik per-IP in-flight request limiter that would bound this is disabled by default (`global.middleware: false`), allowing an unprivileged user to exhaust the HikariCP connection pool and deny service to all users.

### Finding Description

**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `graphQlCustomizer()`, lines 42–49.

```java
@Bean
GraphQlSourceBuilderCustomizer graphQlCustomizer(PreparsedDocumentProvider provider) {
    var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
    var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
    var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);

    return b -> b.configureGraphQl(
            graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
}
```

No `TimeoutInstrumentation` or equivalent is added. The only instrumentation is structural (complexity/depth), not temporal.

**Root cause — failed assumption:** The design assumes that the PostgreSQL-level `statement_timeout = 10000` (set via `alter user mirror_graphql set statement_timeout to '10000'` in the Helm chart) will bound query duration. This assumption fails during a network partition: the timeout signal is generated server-side but cannot be delivered to the JDBC client over a severed TCP connection. The client thread remains blocked in a JDBC call until the OS TCP keepalive timer fires (default: ~2 hours on Linux), not 10 seconds.

**Exploit flow:**

1. Attacker sends N concurrent POST requests to `/graphql/alpha` (each syntactically valid, within complexity=200/depth=10 limits, so they pass all static checks and reach the DB layer).
2. A network partition occurs between the GraphQL pod and the PostgreSQL instance (or the attacker times the requests to coincide with one, or induces one via other means).
3. Each request acquires a HikariCP connection and blocks in JDBC. The PostgreSQL `statement_timeout` fires server-side but the ACK never reaches the app. The GCP gateway's `timeoutSec: 20` terminates the HTTP response to the client, but the server-side Java thread continues holding the DB connection.
4. With the HikariCP pool exhausted (Spring Boot default: 10 connections), subsequent requests from any user block waiting for a connection up to HikariCP's `connectionTimeout` (default 30 s), then fail with a connection acquisition timeout — effectively a DoS for all users.

**Why existing checks are insufficient:**

- **`MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)`** — structural checks only; they fire before execution and do not bound wall-clock execution time.
- **PostgreSQL `statement_timeout = 10000`** — server-side only; undeliverable during a network partition.
- **GCP `timeoutSec: 20`** — terminates the HTTP response to the client but does not interrupt the server-side JDBC thread or release the DB connection.
- **Traefik `inFlightReq: amount: 5` per source IP** — defined in `values.yaml` lines 138–142 but gated on `global.middleware: true`; the default is `global.middleware: false` (line 89), so this middleware is **not deployed** in a default installation.
- **Traefik `retry: attempts: 3`** — actually amplifies the problem by resubmitting timed-out requests, multiplying in-flight connections per attacker IP.

### Impact Explanation
Complete denial of service for all GraphQL API users. Once the HikariCP pool is exhausted, every new query — from any user — fails immediately with a connection acquisition error. The service remains degraded until the network partition resolves and TCP keepalive reclaims the stale connections (potentially hours). The GraphQL API is documented as an alpha API but is publicly reachable at `/graphql/alpha` with no authentication requirement.

### Likelihood Explanation
An unprivileged external user requires no credentials. The attacker needs only to: (a) send a small number of concurrent valid GraphQL queries (≤ pool size, typically 10), and (b) have those queries in-flight during a network partition. Network partitions between application pods and database instances are a realistic operational event in cloud environments (node drain, AZ failure, misconfigured network policy). The attacker does not need to cause the partition — they only need to exploit it opportunistically. The attack is repeatable: once the partition resolves, the attacker can re-trigger on the next partition event. The `retry: attempts: 3` Traefik middleware further reduces the number of distinct source requests needed.

### Recommendation

1. **Add a GraphQL execution timeout instrumentation** in `graphQlCustomizer()`:
   ```java
   // Example using graphql-java's built-in timeout support
   var timeout = new SimpleInstrumentation(); // or a custom AbortExecutionException-throwing instrumentation
   // Configure via ExecutionInput.newExecutionInput().timeout(Duration.ofSeconds(8))
   ```
   Or use Spring GraphQL's `ExecutionGraphQlService` wrapper to enforce a reactive timeout (e.g., `Mono.timeout(Duration.ofSeconds(8))`).

2. **Enable the Traefik middleware by default** (`global.middleware: true`) so the `inFlightReq: amount: 5` per-IP limit is actually deployed.

3. **Remove the `retry` middleware** or ensure it does not apply to in-flight-limited paths, as it multiplies connection consumption.

4. **Configure HikariCP explicitly** for the GraphQL module with a small pool size and a short `connectionTimeout`, so pool exhaustion fails fast rather than queuing.

5. **Set TCP keepalive aggressively** on the JDBC connection (`socketTimeout` in the JDBC URL) so that partitioned connections are detected in seconds, not hours.

### Proof of Concept

```bash
# 1. Open 10 concurrent long-running GraphQL queries (adjust to match pool size)
for i in $(seq 1 10); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}' \
    --max-time 60 &

# 2. Induce or wait for a network partition between the GraphQL pod and PostgreSQL
#    (e.g., apply a NetworkPolicy blocking egress to DB port 5432)

# 4. From a different client, attempt a new query:

  -H 'Content-Type: application/json' \
