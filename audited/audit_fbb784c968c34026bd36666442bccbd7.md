### Title
SSRF via Unvalidated Mirror-Node-Supplied gRPC Endpoints in `fetchMirrorNodeNetwork()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` blindly trusts `DomainName` and `IPAddressV4` values returned by the configured mirror node REST API, performing no validation against loopback, link-local, or private addresses. An attacker who can influence the mirror node's HTTP response (via MITM on the plaintext HTTP channel, DNS poisoning, or direct mirror node compromise) can inject `localhost`/`127.0.0.1` (or any internal address) as a gRPC endpoint, causing the pinger to direct operator-signed `CryptoTransfer` transactions to that address. This constitutes a Server-Side Request Forgery (SSRF) that also leaks signed transaction bytes to an unintended recipient.

### Finding Description

**Exact code path:**

`pinger/sdk_client.go` line 18 calls `buildNetworkFromMirrorNodes()` exclusively when `cfg.network == "other"`. [1](#0-0) 

`buildNetworkFromMirrorNodes()` calls `fetchMirrorNodeNetwork()`, which decodes the mirror node JSON response and at lines 113–124 constructs gRPC addresses directly from the untrusted `DomainName`/`IPAddressV4` fields: [2](#0-1) 

The only guards are emptiness checks (`host == ""`, `ep.Port == 0`). There is no allowlist, no blocklist for RFC-1918/loopback ranges, and no TLS/certificate pinning on the mirror node HTTP call itself. [3](#0-2) 

The resulting `network` map is passed directly to `hiero.ClientForNetwork(netmap)`, and `transfer.go` then calls `cryptoTransfer.Execute(client)` — sending operator-signed gRPC transactions to every address in that map. [4](#0-3) 

**Root cause:** The function assumes the mirror node is a trusted source of topology data. No validation of returned endpoint addresses is performed before they are used as live gRPC targets.

**Failed assumption:** The mirror node REST URL (`HIERO_MIRROR_PINGER_REST`) defaults to a plain `http://` scheme (e.g., `http://mirror-rest:5551`), so the response is not integrity-protected. [5](#0-4) 

### Impact Explanation

1. **SSRF to loopback/internal services:** The pinger process makes outbound gRPC TCP connections to attacker-chosen addresses, including `127.0.0.1` or RFC-1918 ranges. Any gRPC-capable service reachable from the pinger host (sidecar containers, internal APIs, metadata endpoints) can be probed or reached.
2. **Signed transaction leakage:** Each `CryptoTransfer` is signed with the operator's private key before being dispatched. The raw signed protobuf bytes are delivered to the attacker-controlled endpoint, exposing the operator account ID, transfer amounts, and valid transaction signatures. While Hedera transactions embed a specific node account ID and have a short validity window, the attacker receives a stream of legitimately signed payloads that can be analysed or replayed within that window against the real network if the attacker also controls a real Hedera node.
3. **Monitoring subversion / DoS:** The pinger's health-check purpose is defeated; all transfers silently fail against the fake endpoint while the operator believes the network is being monitored.

### Likelihood Explanation

- **Precondition `network=other`** is the intended mode for private/custom deployments — exactly the environments most likely to use an internal mirror node over HTTP.
- **MITM is low-barrier:** The default `http://` scheme means any attacker with network adjacency (same Kubernetes namespace, compromised ingress, ARP spoofing on a flat network) can intercept and rewrite the `/api/v1/network/nodes` response without breaking TLS.
- **DNS poisoning** of the mirror node hostname is another unprivileged path requiring no access to the pinger host itself.
- The attack is **repeatable and persistent**: `buildNetworkFromMirrorNodes` is called once at startup; a single poisoned response permanently redirects all subsequent transfers for the lifetime of the process.

### Recommendation

1. **Validate returned endpoints before use.** After resolving `host`, reject any address that is loopback (`127.0.0.0/8`, `::1`), link-local (`169.254.0.0/16`), or RFC-1918 private ranges unless an explicit allowlist is configured.
2. **Enforce HTTPS for the mirror REST URL.** Reject `http://` schemes (or at minimum warn loudly) and verify the server certificate. Add a config-time check in `loadConfig()`.
3. **Pin expected node account IDs or endpoint domains** via configuration so injected entries for unknown account IDs are discarded.
4. **Re-fetch periodically with the same validation**, rather than trusting a single startup response indefinitely.

### Proof of Concept

**Preconditions:** Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plaintext HTTP). Attacker has network-level access to intercept traffic between the pinger and `mirror-rest:5551`.

**Steps:**

1. Attacker intercepts the HTTP GET to `http://mirror-rest:5551/api/v1/network/nodes`.
2. Attacker returns the following crafted JSON body:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{
         "domain_name": "127.0.0.1",
         "ip_address_v4": "",
         "port": 50211
       }]
     }],
     "links": { "next": null }
   }
   ```
3. `fetchMirrorNodeNetwork()` at line 122 constructs `"127.0.0.1:50211"` and maps it to account `0.0.3`.
4. `hiero.ClientForNetwork({"127.0.0.1:50211": AccountID{0,0,3}})` is called; the SDK treats `127.0.0.1:50211` as the sole consensus node.
5. Every subsequent `cryptoTransfer.Execute(client)` call opens a gRPC connection to `127.0.0.1:50211` and transmits the operator-signed transaction protobuf to that local port.
6. Any gRPC listener on `127.0.0.1:50211` of the pinger host receives the signed transaction bytes. Substitute any internal IP to reach other internal services.
