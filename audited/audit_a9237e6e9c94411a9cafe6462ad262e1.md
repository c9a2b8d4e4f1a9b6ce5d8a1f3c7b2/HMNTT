### Title
Query String Bypass of `internalPaths` Debug-Level Filter in `TracingMiddleware`

### Summary
`TracingMiddleware` uses `request.URL.RequestURI()` to obtain the request path, which includes any query string. The `internalPaths` map contains only bare paths, so a request to `/health/readiness?anything=1` does not match the map and is logged at `Info` level instead of `Debug` level. Any unauthenticated external user can exploit this to inject health-probe entries into the info-level transaction log at will.

### Finding Description
**File:** `rosetta/app/middleware/trace.go`
**Function:** `TracingMiddleware()`, lines 43–61

Line 47 captures the path:
```go
path := request.URL.RequestURI()   // returns "/health/readiness?block=999"
```
Line 19 defines the filter map:
```go
var internalPaths = map[string]bool{livenessPath: true, metricsPath: true, readinessPath: true}
// keys are "/health/liveness", "/health/readiness", "/metrics" — no query strings
```
Line 55 performs the lookup:
```go
if internalPaths[path] {   // false for "/health/readiness?block=999"
    log.Debug(message)
} else {
    log.Info(message)      // executed — health probe appears in info log
}
```
The failed assumption is that `RequestURI()` returns only the path component. It actually returns the full request URI per Go's `net/url` documentation, including the raw query. The map lookup is an exact-string comparison, so any query string defeats it.

### Impact Explanation
An attacker can flood the info-level log with arbitrarily crafted entries that look like legitimate API calls (they carry the client IP, method, path, status code, and latency). This pollutes the transaction history, making forensic analysis and anomaly detection unreliable. Operators relying on info-level logs to audit Rosetta API usage will see spurious entries mixed with real transaction traffic, potentially masking or drowning out genuine activity.

### Likelihood Explanation
No authentication, credentials, or special network position are required. Any HTTP client reachable to the Rosetta port can send `GET /health/readiness?x=1` in a tight loop. The endpoint is intentionally publicly reachable (it is a health probe). The attack is trivially scriptable and repeatable at high frequency.

### Recommendation
Replace the `RequestURI()` call with `request.URL.Path`, which returns only the path component without the query string:
```go
path := request.URL.Path   // "/health/readiness" regardless of query string
```
If the full URI is still needed in the log message, keep `request.URL.RequestURI()` only for the `message` format string and use `request.URL.Path` exclusively for the `internalPaths` lookup.

### Proof of Concept
```bash
# Baseline: normal readiness probe — logged at Debug, invisible at Info level
curl -s http://<rosetta-host>/health/readiness

# Exploit: append any query string — logged at Info level
curl -s "http://<rosetta-host>/health/readiness?block=999"

# Flood the info log with fake entries
for i in $(seq 1 10000); do
  curl -s "http://<rosetta-host>/health/readiness?i=$i" &
done

# Observe info-level log now contains thousands of health-probe lines
# indistinguishable in level from real Rosetta API transaction calls
```
Confirmed by the test suite in `trace_test.go`: the test at line 49 (`metricsPath + "s"`) explicitly verifies that a path that does not exactly match an `internalPaths` key is logged at `levelInfo`, demonstrating the exact-match dependency — but no test covers the query-string variant of a valid internal path.
