### Title
Cache Poisoning via NULL `runtime_bytecode` Causes Stale Empty Bytecode Served for Up to 1 Hour

### Summary
`findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`, but the method returns `Optional<byte[]>`. When a contract row exists with `runtime_bytecode = NULL`, Spring Data JPA returns `Optional.empty()` — a non-null object — so the `unless` guard evaluates to `false` and the empty result is cached. Any unprivileged caller who queries such a contract ID before its bytecode is populated permanently poisons the cache entry for the full TTL (default 1 hour `expireAfterAccess`), causing the web3 API to serve empty bytecode even after the DB is correctly populated.

### Finding Description

**Exact location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The `unless` SpEL expression `#result == null` tests whether the `Optional` wrapper object itself is `null`. Spring Data JPA never returns a literal `null` for an `Optional`-typed repository method — it always returns either `Optional.of(bytes)` or `Optional.empty()`. When the DB row exists but `runtime_bytecode IS NULL`, the native query returns a SQL NULL, which Spring Data JPA maps to `Optional.empty()`. Since `Optional.empty() != null`, the condition `#result == null` is `false`, the `unless` clause does not fire, and `Optional.empty()` is written into the Caffeine cache under the contract's ID.

**Exploit flow:**
1. A contract row exists in the `contract` table with `runtime_bytecode = NULL` (a valid transient state — e.g., immediately after contract creation before the sidecar migration or importer populates the bytecode column).
2. An unprivileged external user calls any web3 endpoint that triggers `ContractBytecodeReadableKVState.readFromDataSource()` (e.g., `eth_getCode`, `eth_call`, `eth_estimateGas`) for that contract ID.
3. `findRuntimeBytecode(contractId)` executes the native query, gets `Optional.empty()`, and — because `Optional.empty() != null` — stores it in the `contract` Caffeine cache.
4. The importer subsequently populates `runtime_bytecode` in the DB.
5. All subsequent calls to `findRuntimeBytecode(contractId)` return the cached `Optional.empty()` without hitting the DB, for up to 1 hour (`expireAfterAccess=1h`, default in `CacheProperties.contract`).
6. `ContractBytecodeReadableKVState.readFromDataSource()` returns `null` (line 46: `.orElse(null)`), causing the EVM to treat the contract as having no code.

**Why existing checks fail:** The only guard is `unless = "#result == null"`. Because `Optional.empty()` is a non-null singleton, this guard never fires for the empty-bytecode case. There is no `@CacheEvict` or invalidation path triggered when the importer later writes the bytecode.

### Impact Explanation

For the duration of the cache TTL (up to 1 hour by default), the web3 API incorrectly reports the contract as having no bytecode (`0x` for `eth_getCode`). Any `eth_call` or `eth_estimateGas` targeting that contract will fail or behave as if calling an EOA. This constitutes incorrect data served by the mirror node's web3 layer — a denial-of-service on contract interaction for any contract that passes through the NULL-bytecode transient state. The impact is bounded by the cache TTL and the size of the cache (up to 1,000 entries by default).

### Likelihood Explanation

The precondition — a contract row with `runtime_bytecode = NULL` — is a normal transient state that occurs between contract creation and sidecar/importer processing. No privileges are required; any caller with network access to the web3 API endpoint can trigger this by issuing a standard JSON-RPC call (`eth_getCode`, `eth_call`) for a known contract ID. The attack is trivially repeatable and requires no authentication.

### Recommendation

Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a populated `Optional.of(bytes)` is stored in the cache, and empty results always fall through to the DB on the next call.

### Proof of Concept

1. Insert a contract row with NULL bytecode:
   ```sql
   INSERT INTO contract (id, runtime_bytecode) VALUES (12345, NULL);
   INSERT INTO entity (id, type, ...) VALUES (12345, 'CONTRACT', ...);
   ```
2. Issue `eth_getCode` for contract `0.0.12345` via the web3 API — observe `0x` returned and the empty result cached.
3. Update the DB:
   ```sql
   UPDATE contract SET runtime_bytecode = '\xDEADBEEF' WHERE id = 12345;
   ```
4. Re-issue `eth_getCode` for `0.0.12345` — observe `0x` still returned (cache hit on `Optional.empty()`).
5. Wait for cache TTL expiry (up to 1 hour) — only then does the correct bytecode appear. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
