### Title
Missing Timestamp Bound in `getLatestFileDataContents` Returns Current File Data Instead of Historical Call Data

### Summary
In `getContractResultsByTimestamp()`, the `callDataId` file lookup at line 1023 passes `{whereQuery: []}` — an empty filter — to `FileDataService.getLatestFileDataContents()`. This causes the query to return the **most recent** file state unconditionally, ignoring the transaction's consensus timestamp that is already available in scope. Any caller of `GET /api/v1/contracts/{contractId}/results/{timestamp}` receives call data reflecting the current file contents, not what was actually submitted at transaction time.

### Finding Description

**Exact code path:**

`rest/controllers/contractController.js`, `getContractResultsByTimestamp()`, lines 1010 and 1023:

```js
// line 1010 — timestamp IS extracted from the request
const {contractId, timestamp} = await getAndValidateContractIdAndConsensusTimestampPathParams(req);
...
// line 1022-1023 — timestamp is NOT passed; empty whereQuery = no upper bound
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

**Root cause — `latestFileContentsQuery` in `fileDataService.js` lines 19-42:**

The CTE selects `max(consensus_timestamp)` for create/update operations with no upper bound when `whereQuery` is empty. The `filterInnerPlaceholder` and `filterOuterPlaceholder` are replaced with empty strings, so the query unconditionally returns the latest file state:

```sql
with latest_create as (
  select max(consensus_timestamp) ...
  where entity_id = $1 and transaction_type in (17, 19)
  -- ← no timestamp ceiling injected
  ...
)
```

**The correct alternative already exists** — `getFileDataQuery` (lines 44-63) accepts a `$2` timestamp parameter and bounds results with `<= $2`. The `getLatestFileDataContents` function also supports timestamp bounding via `whereQuery`, as demonstrated in the test at `rest/__tests__/service/fileDataService.test.js` lines 106-118:

```js
const where = [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: someTimestamp}];
FileDataService.getLatestFileDataContents(fileId, {whereQuery: where}); // returns historical state
```

The fix is simply to pass the already-available `timestamp` as a filter, but this is not done. The same defect exists identically in `getContractResultsByTransactionIdOrHash()` at line 1184.

**Why the only existing check is insufficient:**

`utils.isValidUserFileId(ethTransaction?.callDataId)` (line 1022) only validates that the file ID is a well-formed user-space file ID. It performs no temporal validation and does not prevent the unbounded query.

### Impact Explanation

The `GET /api/v1/contracts/{contractId}/results/{timestamp}` endpoint is a historical audit endpoint — its purpose is to faithfully reproduce what occurred at a specific consensus timestamp. The `callData` field in the response is supposed to reflect the exact bytes submitted in the Ethereum transaction. Because the file lookup has no timestamp ceiling, if the referenced file is updated after the transaction, every subsequent query to that historical endpoint returns the **post-update** file contents as `callData`. This breaks the integrity of the historical record: the API asserts that a transaction contained call data it never actually contained. Any downstream system, auditor, or verifier relying on this endpoint to reconstruct transaction inputs receives falsified data.

### Likelihood Explanation

No special privileges are required to trigger the incorrect behavior — any unauthenticated caller of the endpoint receives the latest file data instead of historical data. The precondition for a meaningful discrepancy (the referenced file being updated after the transaction) is a normal operational event on the Hedera network; file updates are routine. The endpoint is publicly accessible, the trigger is a standard GET request, and the behavior is deterministic and repeatable on every call after a file update occurs.

### Recommendation

Pass the transaction's consensus timestamp as an upper-bound filter when calling `getLatestFileDataContents`, mirroring the pattern already demonstrated in the service's own tests:

```js
// rest/controllers/contractController.js, line 1022-1024
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(
    ethTransaction.callDataId,
    {whereQuery: [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: timestamp}]}
  );
}
```

Apply the identical fix to `getContractResultsByTransactionIdOrHash()` at line 1184, using `contractResult.consensusTimestamp` (or `transactionDetails.consensusTimestamp`) as the bound. Alternatively, replace the call with `FileDataService.getFileData(callDataId, timestamp)` which already implements the correct bounded query via `getFileDataQuery`.

### Proof of Concept

**Preconditions:**
1. A Hedera Ethereum transaction `T` exists at consensus timestamp `ts`, referencing file `F` via `callDataId`. File `F` contains bytes `DATA_V1` at time `ts`.
2. After `ts`, file `F` is updated on the Hedera network; it now contains `DATA_V2`.

**Trigger:**
```
GET /api/v1/contracts/{contractId}/results/{ts}
```

**Observed result:**
Response contains `"call_data": "<hex of DATA_V2>"` — the current file contents.

**Expected result:**
Response should contain `"call_data": "<hex of DATA_V1>"` — the file contents at the time of transaction submission.

**Verification:**
Call `FileDataService.getFileData(F, ts)` directly (the timestamp-bounded query) — it returns `DATA_V1`, confirming the correct historical data exists in the database and is simply not being queried by the endpoint.
