### Title
Unbounded `topicMessages` Registry in `RedisTopicListener` Allows Unauthenticated Subscription Exhaustion

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no limit on concurrent subscriptions per caller or globally. The default `RedisTopicListener` maintains an unbounded `ConcurrentHashMap` (`topicMessages`) keyed by unique topicId, creating one Redis channel subscription per distinct topic. An unprivileged attacker who opens subscriptions to many different valid topicIds can grow this map without bound, exhausting Redis channel subscription capacity and preventing new legitimate subscribers from receiving live messages.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) accepts any `TopicMessageFilter` that passes `@Valid` bean validation and the `topicExists()` check. The `subscriberCount` AtomicLong (line 48) is registered only as a Micrometer gauge metric (lines 52–55) — it is never compared against a maximum and never used to reject incoming subscriptions. [1](#0-0) 

`subscribeTopic()` calls `incomingMessages()` → `topicListener.listen(newFilter)` (line 120), which routes to `CompositeTopicListener.listen()` and, under the default `REDIS` listener type, to `RedisTopicListener.getSharedListener()`: [2](#0-1) [3](#0-2) 

`getSharedListener()` calls `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))`. The map key is `"topic.<entityId>"` (line 65). Each distinct topicId that has no existing entry causes `subscribe()` to be called, which opens a new Redis channel subscription via `ReactiveRedisMessageListenerContainer` and stores the resulting `Flux` in the map. The map has no capacity bound. [4](#0-3) [5](#0-4) 

Entries are only removed when the last subscriber to a topic disconnects (`unsubscribe()` at lines 82–85). As long as the attacker keeps connections open, the entries — and the underlying Redis channel subscriptions — persist. [6](#0-5) 

**Root cause:** `subscribeTopic()` has no per-caller or global subscription cap, and `topicMessages` is an unbounded `ConcurrentHashMap` with one Redis channel subscription per unique topicId.

**Why existing checks fail:**
- `subscriberCount` is a metric only — never enforced as a limit.
- `topicExists()` validates that the topicId is a real TOPIC entity, but does not limit how many subscriptions a caller may hold.
- `ListenerProperties` has no `maxSubscriptions` or `maxTopics` field.
- `TopicMessageFilter` bean validation (`@Min`, `@NotNull`, `@AssertTrue`) checks field values, not subscription cardinality. [7](#0-6) [8](#0-7) 

### Impact Explanation
Redis enforces a hard limit on the number of subscribed channels per connection (default 65,536). Once the attacker's open subscriptions fill the available Redis channel slots, any new legitimate subscriber to a previously-unsubscribed topicId will fail to receive live messages. The `retryWhen` backoff in `subscribe()` will loop indefinitely but cannot succeed while Redis is at capacity. Existing subscribers to already-registered topics are unaffected (they share the cached `Flux`), but any new topic that has not yet been subscribed to becomes unreachable for live delivery. This is a denial-of-service against the live-message path with no economic cost to the attacker.

### Likelihood Explanation
The attack requires no authentication — only gRPC network access. Valid topicIds are publicly enumerable from the Hedera/Hiero ledger (they are public entities). The attacker needs only to open and hold open one gRPC stream per unique topicId; standard gRPC clients support many concurrent streams. The attack is repeatable: if the attacker's connections are dropped, they can immediately reconnect. No special privileges, tokens, or keys are required.

### Recommendation
1. **Enforce a global subscription cap**: Compare `subscriberCount` against a configurable maximum in `subscribeTopic()` and return an error (e.g., `RESOURCE_EXHAUSTED` gRPC status) when the limit is exceeded.
2. **Enforce a per-IP or per-caller connection limit** at the Netty/gRPC layer via `NettyProperties` or a server interceptor.
3. **Cap `topicMessages` map size**: Add a maximum-entries check in `RedisTopicListener.getSharedListener()` and reject new topic subscriptions when the cap is reached.
4. **Add `maxTopics` to `ListenerProperties`**: Make the cap configurable alongside existing properties.

### Proof of Concept
1. Enumerate N valid topicIds from the public ledger (e.g., via the mirror node REST API `/api/v1/topics`).
2. Open N concurrent gRPC `subscribeTopic` streams, each with a distinct topicId and no `endTime` (so they stay open indefinitely).
3. Observe that `RedisTopicListener.topicMessages` grows to N entries and N Redis channel subscriptions are registered.
4. Attempt to open a gRPC `subscribeTopic` stream for a topicId not yet in the map.
5. Observe that `subscribe()` fails to register a new Redis channel subscription (Redis at capacity), and the new subscriber never receives live messages despite the retry loop.

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
