### Title
Unbounded In-Memory Accumulation in `FindBetween()` Before `maxTransactionsInBlock` Guard Is Applied, Enabling Unauthenticated DoS

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` iteratively fetches all transactions in a block's timestamp range into a single in-memory slice with no total-count ceiling. The only guard against oversized blocks — `maxTransactionsInBlock` — is checked in `block_service.go` **after** `FindBetween()` has already returned the fully-populated slice, meaning the server allocates memory proportional to the full transaction count before any truncation occurs. An unauthenticated caller querying the `/block` endpoint for a high-transaction block can exhaust server memory and crash the Rosetta API process.

### Finding Description

**Exact code path:**

`FindBetween()` runs an unbounded accumulation loop:

```go
// rosetta/app/persistence/transaction.go, lines 123-147
transactions := make([]*transaction, 0)
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    transactions = append(transactions, transactionsBatch...)   // ← grows without bound
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

`batchSize = 2000` (line 22) is a per-query page size, not a total cap. The loop continues until the entire timestamp range is exhausted, appending every batch to `transactions`.

The caller in `block_service.go` applies the cap only after the full slice is returned:

```go
// rosetta/app/services/block_service.go, lines 56-66
if block.Transactions, err = s.FindBetween(ctx, ...); err != nil { ... }  // all N loaded here

if len(block.Transactions) > s.maxTransactionsInBlock {   // truncation happens here, too late
    ...
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
```

**Root cause / failed assumption:** The design assumes `maxTransactionsInBlock` is a response-size limit, but it is applied as a post-fetch slice operation. The memory for all N transactions (including their JSON-encoded `CryptoTransfers` and `StakingRewardPayouts` strings) is already allocated before the guard fires.

**Why the `statementTimeout` does not help:** `GetDbWithContext()` in `rosetta/app/db/client.go` (line 36) creates a single `context.WithTimeout` that wraps the entire DB session, but each individual `db.Raw(...).Limit(batchSize).Find(...)` call is a separate statement. A statement timeout of, say, 30 s applies per batch query, not to the total loop. With 2 000 rows per batch, a block with 200 000 transactions requires 100 sequential queries, each completing well within any reasonable per-statement timeout.

**Exploit flow:**
1. Attacker (or any observer) identifies a block with a large number of transactions (naturally occurring on a busy network, or seeded by the attacker paying HBAR fees).
2. Attacker sends repeated unauthenticated POST requests to `/block` with that block's identifier.
3. Each request causes `FindBetween()` to load all N transaction rows — including their embedded JSON transfer arrays — into a Go slice before returning.
4. Concurrent requests multiply memory pressure; the Go runtime OOMs and the process crashes.

### Impact Explanation
The Rosetta API process crashes or becomes unresponsive, denying service to all consumers (exchanges, custodians, block explorers) that rely on it to read block data. Because the `/block` endpoint is unauthenticated and the memory cost scales linearly with block size, a single large block is sufficient to make every subsequent query for that block a memory bomb. Recovery requires a process restart, and the condition is immediately reproducible.

### Likelihood Explanation
No authentication or rate-limiting is visible in the router setup (`main.go` lines 111-119). Any internet-accessible Rosetta deployment is reachable. The attacker does not need to have submitted the transactions themselves — they only need to know the index or hash of a block that already contains many transactions, which is public information. On mainnet, high-throughput periods naturally produce large blocks. The attack is repeatable at will with a trivial HTTP client.

### Recommendation
Apply the transaction count ceiling **inside** `FindBetween()` (or pass `maxTransactionsInBlock` as a parameter) so that the DB loop breaks as soon as the limit is reached, preventing unbounded memory growth:

```go
// Break the fetch loop once the cap is reached
if len(transactions) >= maxTransactions {
    break
}
```

Additionally, add an HTTP-level rate limiter on the `/block` endpoint and consider streaming or cursor-based pagination rather than full in-memory accumulation.

### Proof of Concept

```bash
# 1. Identify a block with many transactions (e.g., block index 12345678)
# 2. Send concurrent requests to the Rosetta /block endpoint

for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{
      "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
      "block_identifier": {"index": 12345678}
    }' &
done
wait

# Each concurrent request causes FindBetween() to load all transactions
# for block 12345678 into memory. With a sufficiently large block,
# the process OOMs and crashes.
```
