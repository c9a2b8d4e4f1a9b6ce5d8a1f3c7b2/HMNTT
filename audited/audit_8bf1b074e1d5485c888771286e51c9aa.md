### Title
Unauthenticated, Rate-Unlimited `/metrics` Endpoint Exposed on All Interfaces in Rosetta Service

### Summary
The `Routes()` function in `rosetta/app/middleware/metrics.go` registers `/metrics` backed directly by `promhttp.Handler()` with no authentication check and no rate-limiting guard at the application layer. In Docker Compose deployments the service binds port 5700 on all host interfaces, making this endpoint reachable by any external actor. Even in Kubernetes, the application-level absence of controls means any attacker with cluster-network access (e.g., a compromised pod) can poll it freely.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, `Routes()`, lines 64–73:
```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,          // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth, no rate limit
        },
    }
}
```
The handler is wired directly into the router in `rosetta/main.go` lines 106–119 (both online and offline modes) and the HTTP server binds to `fmt.Sprintf(":%d", rosettaConfig.Port)` (line 221) — i.e., `0.0.0.0:5700`, all interfaces.

**Root cause:** The developer assumed that network-perimeter controls (Kubernetes ingress, firewall) would always gate access to `/metrics`. No application-level authentication or rate limiting is implemented.

**Exploit flow:**
1. In Docker Compose (`docker-compose.yml` line 151: `5700:5700`), port 5700 is bound to all host interfaces.
2. An external attacker sends `GET /metrics` at high frequency with no credentials required.
3. `promhttp.Handler()` iterates over all registered metric families, serializes them to text, and writes the response on every call — no caching, no concurrency cap.
4. Each request also passes through `MetricsMiddleware` (lines 76–83), which updates histograms and gauges, adding further per-request work.
5. Additionally, the full internal operational state (request rates, durations, in-flight counts, byte sizes per route) is disclosed to the unauthenticated caller.

**Why existing checks fail:**
- The Kubernetes ingress (`values.yaml` lines 119–133) only routes `/rosetta/account`, `/rosetta/block`, etc. — `/metrics` is absent. This protects Kubernetes deployments from *external* ingress traffic, but does nothing for Docker Compose or intra-cluster access.
- The Traefik middleware rate limit (`values.yaml` lines 157–160: `average: 10`) applies only to ingress-routed paths; direct pod/service access bypasses it entirely.
- The `rest` service's `metricsHandler.js` (lines 137–144) shows the correct pattern — it checks `config.metrics.config.authentication` and returns 401 if credentials fail — but this pattern was never applied to the Rosetta Go service.

### Impact Explanation
**Information disclosure:** Full Prometheus metric output reveals internal request rates, error rates, latency distributions, and in-flight counts per route — useful for an attacker mapping service behavior and timing attacks.

**Resource exhaustion:** At high polling frequency (thousands of req/s), repeated metric collection and serialization consumes CPU; combined with `MetricsMiddleware` histogram updates on each `/metrics` request itself (the middleware instruments all routes including `/metrics`), this creates a feedback loop of metric-recording overhead, degrading availability for legitimate Prometheus scrapers and the application.

### Likelihood Explanation
In Docker Compose deployments (the documented local/dev/staging path, `docker-compose.yml` line 151), this requires zero privileges — any host-reachable HTTP client suffices. In Kubernetes, it requires only cluster-network access (any pod in the cluster). Both scenarios are realistic for an attacker who has achieved initial access or is targeting a misconfigured Docker host. The attack is trivially scriptable (`while true; do curl http://target:5700/metrics; done`) and requires no authentication tokens or special tooling.

### Recommendation
1. **Add IP-allowlist or authentication middleware** in `Routes()` before delegating to `promhttp.Handler()`, mirroring the pattern in `rest/middleware/metricsHandler.js` lines 137–144 (basic-auth guard).
2. **Bind the metrics endpoint to a separate, non-public port** (e.g., a dedicated admin port bound to `127.0.0.1` only), separate from the Rosetta API port.
3. **In Docker Compose**, change the port mapping to `127.0.0.1:5700:5700` to prevent external exposure.
4. **Apply rate limiting at the application layer** (e.g., `golang.org/x/time/rate`) on the `/metrics` handler, independent of ingress-level controls.

### Proof of Concept
```bash
# Against a Docker Compose deployment (port 5700 bound to 0.0.0.0)
# Step 1: Verify unauthenticated access

# Returns full Prometheus metric output with no 401/403

# Step 2: Sustained high-frequency polling (DoS / metric exhaustion)
while true; do
  curl -s http://<target-host>:5700/metrics -o /dev/null &
