### Title
Cache Poisoning via Unbounded Negative-Result DB Queries for Non-Existent Contract IDs in `findStorage()`

### Summary
`ContractStateServiceImpl.findStorage()` performs no validation on the supplied `contractId` and, in the default batch-caching path, never caches a negative ("not found") result in `contractStateCache`. Any unprivileged caller can repeatedly trigger `findStorage()` with non-existent or zero contract IDs, causing a DB query on every invocation and polluting `contractSlotsCache` with phantom entries that evict legitimate contract slot-key caches, degrading cache hit rate and amplifying DB load.

### Finding Description

**Code path:**

`ContractStorageReadableKVState.readFromDataSource()` (line 44) calls `contractStateService.findStorage(entityId, keyBytes)` with an `entityId` derived directly from the EVM `SlotKey` — no existence check is performed before the call. [1](#0-0) 

Inside `ContractStateServiceImpl.findStorage()` (lines 58–70), when `enableBatchContractSlotCaching` is `true` (the default), the method:

1. Checks `contractStateCache` — misses for any unseen key.
2. Falls through to `findStorageBatch()`. [2](#0-1) 

Inside `findStorageBatch()` (lines 85–122):

- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally writes the attacker-supplied slot key into the per-contract Caffeine cache, even for a non-existent contract.
- Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a real DB query; returns an empty list for a non-existent contract.
- Lines 106–114: The loop body is never entered; **nothing is written to `contractStateCache`**.
- Line 121: Returns `Optional.empty()`. [3](#0-2) 

**Root cause — failed assumption:** The design assumes callers only supply valid, existing contract IDs. There is no guard against `EntityId.EMPTY` (id=0) or any other non-existent ID, and there is no negative-result caching: because `contractStateCache` is never written for a miss, every subsequent call for the same phantom `(contractId, slot)` pair re-enters `findStorageBatch()` and re-issues a DB query.

The `@Cacheable` annotation on `ContractStateRepository.findStorage()` (line 20–21) only protects the non-batch fallback path (line 60); it is never reached in the default batch path. [4](#0-3) 

**`contractSlotsCache` pollution:** The outer `contractSlotsCache` (default `maximumSize=3000`) maps one entry per `contractId` to a per-contract Caffeine cache. Flooding with N distinct phantom contract IDs fills this cache, evicting legitimate contracts' slot-key caches and forcing those contracts to re-query the DB on their next access. [5](#0-4) 

### Impact Explanation

Every `eth_call` / `eth_estimateGas` request that causes an EVM SLOAD on a non-existent contract address results in at least one DB query with no caching of the negative result. An attacker cycling through phantom contract IDs can:

- Saturate the `contractSlotsCache` (3 000 entries), evicting all legitimate contract slot-key caches.
- Drive a sustained stream of `SELECT … FROM contract_state WHERE contract_id = ?` queries against the DB with zero cache benefit.
- Degrade cache hit rate and increase DB CPU/IO load across all mirror-node instances sharing the same DB, consistent with the ≥30% node-processing degradation threshold.

### Likelihood Explanation

The web3 API (`eth_call`, `eth_estimateGas`) is unauthenticated and rate-limiting is not enforced at the service layer. An attacker needs only an HTTP client and a loop generating distinct phantom contract addresses (e.g., incrementing a 20-byte EVM address). No privileged access, no on-chain transaction, and no special knowledge of the system is required. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Negative-result caching:** After `findStorageBatch()` returns an empty result for a given `(contractId, slot)` pair, write a sentinel value (e.g., `EMPTY_VALUE`) into `contractStateCache` with a short TTL (matching the existing `expireAfterWrite=2s`). The existing guard at line 65 (`cachedValue != EMPTY_VALUE`) already handles the sentinel correctly for the read path; only the write side is missing.
2. **Pre-existence check:** Before entering the batch path, verify the contract exists (e.g., via the entity cache) and return `Optional.empty()` immediately for unknown IDs, avoiding both the DB query and the `contractSlotsCache` write.
3. **Rate limiting:** Apply per-IP or per-caller rate limiting at the API gateway layer for `eth_call` / `eth_estimateGas` endpoints.

### Proof of Concept

```
# Precondition: mirror-node web3 API running with default config
#   enableBatchContractSlotCaching = true
#   contractState: maximumSize=25000
#   contractSlots: maximumSize=3000

# Step 1 – flood with phantom contract IDs via eth_call
for i in $(seq 1 5000); do
  curl -s -X POST http://<mirror-node>:8545 \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",
         \"params\":[{\"to\":\"0x$(printf '%040x' $i)\",\"data\":\"0x\"},\"latest\"],
         \"id\":$i}" &
done
wait

# Step 2 – observe: contractSlotsCache is now full of phantom entries (3000 distinct IDs)
# Step 3 – send a legitimate eth_call for a real contract
# Observe: its slot-key cache was evicted; a fresh DB batch query is issued
# Step 4 – repeat Step 1 continuously; DB query rate climbs linearly with request rate
#           because no negative result is ever cached in contractStateCache
```

        if (!cacheProperties.isEnableBatchContractSlotCaching()) {

        final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
