### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Enables MITM/DNS-Spoofing to Falsify Transaction Fee Data

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` hardcodes `sslmode=disable` with no configurable SSL option in the `Db` struct, meaning the PostgreSQL connection is always unencrypted and the server is never authenticated. An attacker with the ability to poison the DNS cache used by the Rosetta service host (or intercept the TCP connection) can redirect the DB connection to a rogue PostgreSQL server that returns crafted `ChargedTxFee` and `MaxFee` values, causing the Rosetta API to report falsified transaction fees to all consumers.

### Finding Description
**Code location:**
- `rosetta/app/config/types.go`, lines 49–58: `GetDsn()` unconditionally appends `sslmode=disable` to the DSN string.
- `rosetta/app/db/db.go`, lines 17–18: `ConnectToDb()` calls `dbConfig.GetDsn()` and passes the result directly to `gorm.Open()`.

```go
// rosetta/app/config/types.go:49-58
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        db.Host, db.Port, db.Username, db.Name, db.Password,
    )
}
```

The `Db` struct (lines 39–47) has no SSL-related field whatsoever, so operators cannot override this behavior through configuration. The `ConnectToDb()` function accepts the DSN as-is with no post-connection TLS verification.

**Root cause:** The failed assumption is that the DB connection is always on a trusted, isolated network. With `sslmode=disable`, the PostgreSQL client sends credentials and receives query results in plaintext with no server identity verification.

**Exploit flow:**
1. The Rosetta service resolves the DB hostname (e.g., `hiero.mirror.rosetta.db.host` set to a DNS name rather than `127.0.0.1`).
2. An attacker performs DNS cache poisoning against the resolver used by the Rosetta host, causing the hostname to resolve to an attacker-controlled IP.
3. The attacker runs a rogue PostgreSQL server that accepts the connection (credentials are sent in plaintext, or the attacker uses the `trust` auth method).
4. The rogue server responds to `selectTransactionsInTimestampRange` queries (defined in `rosetta/app/persistence/transaction.go`, lines 32–57) with crafted rows containing arbitrary `charged_tx_fee` / `max_fee` values.
5. `constructTransaction()` (lines 210–269) processes these rows without any integrity check and builds `types.Transaction` objects with the falsified fee operations.
6. The Rosetta API returns these falsified fee amounts to all callers.

**Why existing checks fail:** There are no checks. The `Db` struct has no `SslMode` field, `GetDsn()` has no branch for SSL, and `ConnectToDb()` performs no post-connection certificate or identity validation.

### Impact Explanation
Any Rosetta API consumer (exchange, wallet, block explorer) querying `/block` or `/block/transaction` endpoints receives falsified `ChargedTxFee` and `MaxFee` values. This can cause exchanges to miscalculate fee deductions, wallets to display incorrect fee history, and automated systems relying on Rosetta fee data to make incorrect decisions. The integrity of the entire fee reporting surface of the Rosetta API is compromised for the duration of the attack.

### Likelihood Explanation
The attack requires the DB to be configured with a resolvable hostname (not a raw IP like the default `127.0.0.1`). In production deployments using managed databases or Kubernetes service names, hostnames are standard. DNS cache poisoning (Kaminsky-style) against a misconfigured or unpatched resolver is a well-documented technique executable by an external attacker without privileged access to the target host or network. The absence of any SSL option means there is no defense-in-depth even if the operator is aware of the risk.

### Recommendation
1. Add an `SslMode` field to the `Db` struct in `rosetta/app/config/types.go` with a secure default (e.g., `verify-full`).
2. Update `GetDsn()` to include the configured `sslmode` value instead of the hardcoded `disable`.
3. For production, set `sslmode=verify-full` and supply the CA certificate path via `sslrootcert` in the DSN.
4. Mirror the pattern already used in the importer (`importer/src/main/java/org/hiero/mirror/importer/db/DBProperties.java`, line 50) which exposes `SslMode` as a configurable property.

### Proof of Concept
1. Configure `hiero.mirror.rosetta.db.host` to a DNS hostname (e.g., `mirror-db.internal`).
2. Poison the DNS cache of the Rosetta host's resolver so `mirror-db.internal` resolves to attacker IP.
3. On the attacker IP, run a PostgreSQL server configured with `trust` authentication for the `mirror_rosetta` user.
4. Populate the rogue DB's `transaction` table with rows where `charged_tx_fee = 999999999` and `max_fee = 999999999`.
5. Start the Rosetta service; observe it connects to the rogue server (no SSL handshake, no cert check).
6. Query `POST /block/transaction` — the response will contain the attacker-supplied fee values in the `fee` operations, with no error or warning from the Rosetta service. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

func (tr *transactionRepository) constructTransaction(sameHashTransactions []*transaction) (
	*types.Transaction,
	*rTypes.Error,
) {
	allFailed := true
	firstTransaction := sameHashTransactions[0]
	operations := make(types.OperationSlice, 0)
	result := &types.Transaction{Hash: firstTransaction.getHashString(), Memo: firstTransaction.Memo}
	success := types.GetTransactionResult(transactionResultSuccess)
	transactionType := types.GetTransactionType(int32(firstTransaction.Type))

	for _, transaction := range sameHashTransactions {
		cryptoTransfers := make([]hbarTransfer, 0)
		if err := json.Unmarshal([]byte(transaction.CryptoTransfers), &cryptoTransfers); err != nil {
			return nil, hErrors.ErrInternalServerError
		}

		stakingRewardPayouts := make([]hbarTransfer, 0)
		if err := json.Unmarshal([]byte(transaction.StakingRewardPayouts), &stakingRewardPayouts); err != nil {
			return nil, hErrors.ErrInternalServerError
		}

		var feeHbarTransfers []hbarTransfer
		var itemizedTransfers []hbarTransfer
		var stakingRewardTransfers []hbarTransfer
		feeHbarTransfers, itemizedTransfers, stakingRewardTransfers = categorizeHbarTransfers(
			cryptoTransfers,
			transaction.ItemizedTransfer,
			tr.stakingRewardEntityId,
			stakingRewardPayouts,
		)

		transactionResult := types.GetTransactionResult(int32(transaction.Result))

		operations = tr.appendHbarTransferOperations(transactionResult, transactionType, itemizedTransfers, operations)
		// fee transfers are always successful regardless of the transaction result
		operations = tr.appendHbarTransferOperations(success, types.OperationTypeFee, feeHbarTransfers, operations)
		// staking reward transfers (both credit and debit) are always successful and marked as crypto transfer
		operations = tr.appendHbarTransferOperations(success, types.OperationTypeCryptoTransfer,
			stakingRewardTransfers, operations)

		if IsTransactionResultSuccessful(int32(transaction.Result)) {
			allFailed = false
			result.EntityId = transaction.EntityId
		}
	}

	if allFailed {
		// add an 0-amount hbar transfer with the failed status to indicate the transaction has failed
		operations = tr.appendHbarTransferOperations(
			types.GetTransactionResult(int32(firstTransaction.Result)),
			transactionType,
			[]hbarTransfer{{AccountId: firstTransaction.PayerAccountId}},
			operations,
		)
	}

	result.Operations = operations
	return result, nil
}
