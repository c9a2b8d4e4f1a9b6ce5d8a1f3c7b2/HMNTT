### Title
Deleted CONTRACT Entity Exposed via Long-Zero EVM Address Path in GraphQL `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, the long-zero address branch calls `entityRepository.findById()` (inherited from Spring Data's `CrudRepository`), which has no `deleted` filter. The regular EVM address branch calls `entityRepository.findByEvmAddress()`, which explicitly filters `deleted is not true`. Any unprivileged user can supply a long-zero address for a deleted CONTRACT entity and receive its full metadata from the GraphQL API, while the same entity is correctly hidden when queried by its actual EVM address.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        // LONG-ZERO PATH: findById() has NO deleted filter
        return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
    }
    // REGULAR PATH: findByEvmAddress() has "deleted is not true"
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

**Root cause:**

`EntityRepository` extends `CrudRepository<Entity, Long>`. The inherited `findById()` executes a plain `SELECT * FROM entity WHERE id = ?` with no deletion guard. The custom `findByEvmAddress()` explicitly adds `AND deleted is not true` (repository line 16). The only post-fetch filter applied in both branches is `.filter(e -> e.getType() == type)`, which checks entity type only — never deletion status.

**Exploit flow:**

1. A CONTRACT is deleted on the Hedera network; the mirror node records `deleted = true` for that entity (e.g., entity ID `4660`).
2. The attacker constructs the long-zero EVM address: `0x00000000000000000000000000000000000012B4` (shard=0, realm=0, num=4660).
3. The attacker sends a GraphQL query using this address.
4. `buffer.getInt() == 0 && buffer.getLong() == 0` evaluates to `true`; `findById(4660)` is called.
5. The deleted entity is returned with full metadata (memo, keys, auto-renew settings, etc.).
6. Querying the same contract by its actual EVM address returns empty — the inconsistency is confirmed.

**Why existing checks fail:**

The `.filter(e -> e.getType() == type)` guard (line 38) only enforces entity type. There is no `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` or equivalent in the long-zero branch.

### Impact Explanation
The GraphQL mirror node API is the primary off-chain data source for dApps and tooling on Hedera. Returning stale metadata for a deleted contract via the long-zero address path means:
- A dApp checking contract existence before dispatching a transaction receives a false-positive "contract exists" response, potentially submitting a transaction to a deleted contract and burning fees.
- Smart contract orchestration logic that uses the mirror node to resolve contract addresses by numeric ID will treat deleted contracts as live, leading to unintended call routing.
- The inconsistency between the two lookup paths (long-zero vs. actual EVM address) creates a split-brain view of contract state that is exploitable to bypass any client-side "is contract active?" guard that happens to use the long-zero form.

Severity is Medium: no direct fund loss, but concrete unintended smart contract behavior is achievable.

### Likelihood Explanation
- No authentication or special privilege is required; any GraphQL API consumer can trigger this.
- Long-zero addresses are the standard way Hedera entity IDs are expressed as EVM addresses and are widely used by SDKs and tooling.
- Deleted contracts are a normal, frequent network event.
- The exploit is deterministic and repeatable: given any deleted contract's numeric ID, the attacker constructs the long-zero address trivially.

### Recommendation
Add an explicit deleted-entity filter in the long-zero branch, consistent with `findByEvmAddress()`:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
        .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
        .filter(e -> e.getType() == type);
}
```

Alternatively, add a dedicated repository method `findByIdAndDeletedIsNotTrue(Long id)` with an explicit `AND deleted is not true` query (mirroring `findByEvmAddress`) and use it in this branch, ensuring both paths are governed by the same SQL-level filter.

### Proof of Concept
**Preconditions:**
- Mirror node is running and its DB contains an entity row with `id = 4660`, `type = 'CONTRACT'`, `deleted = true`, and some `evm_address`.

**Steps:**

1. Query via long-zero address (vulnerable path):
```graphql
query {
  contract(input: { evmAddress: "0x00000000000000000000000000000000000012B4" }) {
    id
    memo
    deleted
  }
}
```
**Expected (correct):** empty / not found.  
**Actual:** returns the deleted contract's metadata including `deleted: true`.

2. Query via actual EVM address (correct path):
```graphql
query {
  contract(input: { evmAddress: "<actual_evm_address_of_entity_4660>" }) {
    id
    memo
  }
}
```
**Result:** empty — correctly filtered by `deleted is not true`.

The divergence between step 1 and step 2 for the same underlying entity confirms the asymmetry and the exploitability.
