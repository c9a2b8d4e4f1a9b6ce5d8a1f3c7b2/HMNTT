### Title
Unbounded `string_agg` in `getFileDataQuery` Enables Targeted DoS on Contract Bytecode Lookups

### Summary
The `getFileDataQuery` static SQL in `rest/service/fileDataService.js` uses `string_agg(file_data, '' order by consensus_timestamp)` with no `LIMIT` on the outer aggregation. Any Hedera account holder can create a file, submit many `FileAppend` transactions against it (they own the WACL key), then deploy a contract referencing that file. Every subsequent `GET /api/v1/contracts/:contractId` call for that contract forces the DB to aggregate all accumulated file chunks without bound, causing query timeouts or memory pressure and making bytecode lookups unavailable for that contract.

### Finding Description
**Exact location:** `rest/service/fileDataService.js`, lines 44–63, `FileDataService.getFileDataQuery`.

The query:
```sql
select
  string_agg(file_data, '' order by consensus_timestamp) data
from file_data
where
  entity_id = $1
and consensus_timestamp >= (
  select consensus_timestamp from file_data
  where entity_id = $1
    and consensus_timestamp <= $2
    and (transaction_type = 17 or (transaction_type = 19 and length(file_data) <> 0))
  order by consensus_timestamp desc limit 1
) and consensus_timestamp <= $2
```

The inner subquery correctly uses `limit 1` to find the anchor (most recent FileCreate/FileUpdate at or before `$2`). The outer `string_agg` then concatenates **every** row — including all `FileAppend` (type 16) rows — between that anchor and `$2` with no row count or byte-size cap.

**Call site:** `contractController.js` line 732:
```javascript
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
}
```

`getFileData` passes `(fileId, createdTimestamp)` directly to the unbounded query. The `isValidUserFileId` guard only excludes system files; user-created files (the attacker's target) pass through.

**Root cause / failed assumption:** The code assumes the number of `FileAppend` rows for any given file between its creation and a contract's deployment timestamp is small and bounded. There is no enforcement of this assumption in the query.

### Impact Explanation
For each `GET /api/v1/contracts/:contractId` request targeting a poisoned contract, PostgreSQL must read and concatenate all `file_data` blobs for the referenced file into a single in-memory aggregate. With Hedera's per-transaction body limit (~6 KB) and no cap on the number of `FileAppend` transactions, an attacker can accumulate hundreds of KB to the network's configured max file size (typically 1 MB on mainnet, higher on testnets). Under concurrent requests, this multiplies memory pressure. The `statement_timeout` configured in `dbpool.js` will eventually kill the query, but that itself is the DoS: the bytecode lookup returns an error, the contract's `bytecode` field is unavailable, and any downstream tooling relying on it is broken. The impact is scoped to the mirror node REST API for affected contracts — not a full network shutdown, but a reliable, repeatable denial of the bytecode lookup service for targeted contracts.

### Likelihood Explanation
The precondition is a funded Hedera account — a very low bar (accounts cost fractions of a cent). The attacker:
1. Creates a file (FileCreate, type 17) — they hold the WACL key.
2. Submits N FileAppend transactions against their own file — permitted because they own the WACL key.
3. Deploys a contract referencing that file (ContractCreate with `fileID`).

Steps 1–3 are standard, permissionless Hedera operations. The attack is repeatable across many contracts and files. The only cost is Hedera transaction fees, which are negligible at scale. No special privilege, insider access, or protocol exploit is required.

### Recommendation
1. **Add a byte-size guard in the query:** Use PostgreSQL's `pg_column_size` or track cumulative length and abort if it exceeds a configured threshold (e.g., 2 MB).
2. **Add a row-count LIMIT:** Cap the number of `FileAppend` rows aggregated (e.g., `LIMIT 1000`) and return an error if exceeded.
3. **Cache the result:** Since `file_id` and `created_timestamp` are immutable for a deployed contract, cache the aggregated bytecode result (keyed on `(file_id, created_timestamp)`) so the expensive query runs at most once per contract.
4. **Enforce `statement_timeout` and surface it clearly:** Ensure the timeout is set low enough to prevent memory exhaustion, and return a clear `503` rather than a silent null.

### Proof of Concept
1. Create a Hedera account with sufficient HBAR on testnet.
2. Submit `FileCreate` → record `fileId` and `createTimestamp`.
3. Submit 500× `FileAppend` to `fileId`, each with ~5 KB of content (total ~2.5 MB, within testnet limits).
4. Submit `ContractCreate` with `fileID = fileId` → record `contractId`.
5. Wait for mirror node to ingest all transactions.
6. `GET /api/v1/contracts/{contractId}` — observe the mirror node executing `getFileDataQuery` with 500+ rows to aggregate.
