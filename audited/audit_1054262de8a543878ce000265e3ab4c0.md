### Title
Unauthenticated Redis Channel Injection Allows Delivery of TopicMessages with Arbitrary `validStartTimestamp` to gRPC Subscribers

### Summary
`RedisTopicListener.subscribe()` blindly deserializes and forwards every `TopicMessage` received from a Redis pub/sub channel to gRPC subscribers without any integrity or provenance validation. Because Redis channels are named with a predictable, publicly documented scheme (`topic.<id>`) and the code applies no cryptographic authentication, an attacker with network access to the Redis instance can publish a crafted `TopicMessage` whose `validStartTimestamp` predates the topic's creation time, causing `subscribe()` to deliver it as a legitimate live message to all active subscribers.

### Finding Description

**Code path:**

`RedisTopicListener.subscribe()` (lines 68ŌĆō80) subscribes to a Redis channel and maps every received byte payload directly to a `TopicMessage` via the configured `messageSerializer`:

```
container
  .flatMapMany(r -> r.receive(..., channelSerializer, messageSerializer))
  .map(Message::getMessage)   // ŌåÉ raw deserialized object, no validation
  ...
  .share();
``` [1](#0-0) 

The channel name is constructed as `topic.<numericId>`: [2](#0-1) 

The only downstream filter applied by `CompositeTopicListener.filterMessage()` is:

```java
return message.getTopicId().equals(filter.getTopicId())
        && message.getConsensusTimestamp() >= filter.getStartTime();
``` [3](#0-2) 

`TopicMessageServiceImpl.subscribeTopic()` adds only a sequence-number deduplication check: [4](#0-3) 

`validStartTimestamp` is a plain, unconstrained `Long` field on `TopicMessage`: [5](#0-4) 

**Root cause:** The code assumes that only the trusted `RedisPublisher` (importer module) writes to Redis channels. There is no message signing, HMAC, or any other integrity mechanism. The serializer (`Jackson2JsonRedisSerializer` over `MessagePackFactory`) is a well-known, open-source format: [6](#0-5) 

The legitimate publisher writes to the same channel format: [7](#0-6) 

**Why checks fail:** No check in the entire pipeline (`subscribe()` ŌåÆ `filterMessage()` ŌåÆ `subscribeTopic()`) validates `validStartTimestamp` against the topic's `createdTimestamp`. The topic creation timestamp is stored in the DB (`Topic.createdTimestamp`) but is never fetched or compared during live message delivery. [8](#0-7) 

### Impact Explanation
Any gRPC subscriber calling `subscribeTopic` with the REDIS listener type receives the injected message as a legitimate live Hashgraph event. The `validStartTimestamp` field is surfaced to clients as `chunk_info.initial_transaction_id.transaction_valid_start`. A value predating the topic's creation is logically impossible on a real Hashgraph network, so delivering it falsifies the chronological history of the topic for all connected subscribers. Because the shared `Flux` is multicasted via `.share()`, a single injected message reaches every concurrent subscriber on that topic simultaneously. [9](#0-8) 

### Likelihood Explanation
Redis is frequently deployed without authentication or with network exposure in containerized/cloud environments. The channel naming scheme (`topic.<id>`) is trivially discoverable from the open-source code. The MessagePack serialization format and `TopicMessage` schema are fully public. An attacker with any internal network path to Redis (misconfigured firewall, compromised sidecar, SSRF in another service) can craft and publish a valid payload with no application-level credentials. The attack is repeatable and requires no timing precision.

### Recommendation
1. **Enable Redis AUTH / ACLs** and restrict `PUBLISH` permission to the importer service account only.
2. **Add a message integrity check** in `subscribe()` or `filterMessage()`: cross-reference the received `consensusTimestamp` and `validStartTimestamp` against the topic's `createdTimestamp` from the entity repository before forwarding to subscribers.
3. **Sign published messages** in `RedisPublisher` with an HMAC keyed secret and verify the signature in `RedisTopicListener.subscribe()` before deserialization is accepted.
4. **Validate `validStartTimestamp`** is `>= topic.createdTimestamp` whenever it is non-null, rejecting messages that violate this invariant.

### Proof of Concept

```
# 1. Discover topic numeric ID (e.g., 12345) from any mirror node REST API call.

# 2. Serialize a crafted TopicMessage in MessagePack format (using the public
#    TopicMessage schema from common/src/main/java/.../TopicMessage.java):
#    - topicId: EntityId matching the target topic
#    - consensusTimestamp: any value >= subscriber's startTime filter
#    - sequenceNumber: last_known_sequence + 1  (to pass dedup check)
#    - validStartTimestamp: 0  (Unix epoch, before any topic could exist)
#    - message: arbitrary bytes

# 3. Connect to Redis (no credentials required in default/misconfigured deployment):
redis-cli -h <redis-host> -p 6379

# 4. Publish the crafted payload to the topic channel:
PUBLISH topic.12345 <msgpack-serialized-TopicMessage>

# 5. Observe: all active gRPC subscribers on topic 12345 receive the injected
#    message with validStartTimestamp = "1970-01-01T00:00:00.000000000Z",
#    which predates the topic's creation, falsifying the Hashgraph history.
```
