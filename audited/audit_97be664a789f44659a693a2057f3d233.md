### Title
Unbounded TTL-less `mset` in `Cache.get()` Enables Redis Cache Pollution and Cache-Thrashing DoS via Unauthenticated GET /transactions

### Summary
`Cache.get()` in `rest/cache.js` writes cache-miss entries to Redis using `mset` with **no TTL**, causing every unique transaction key to persist indefinitely. Because the REST API has no rate limiting, an unauthenticated attacker can flood `GET /transactions` with unique timestamp ranges, continuously injecting new persistent keys until Redis reaches its `maxmemory` limit (default 250 MB), triggering constant LFU eviction of legitimate entries and causing cache thrashing that degrades database performance.

### Finding Description

**Exact code path:**

`rest/transactions.js` line 44 creates a module-level `Cache` instance. `doGetTransactions()` at line 705 calls:

```js
const transactions = await cache.get(payerAndTimestamps, loader, keyMapper);
```

`keyMapper` (lines 679–685) produces keys of the form `transaction:{consensus_timestamp}` — one unique key per transaction row returned by the DB query.

Inside `Cache.get()` (`rest/cache.js` lines 95–140), when keys are absent from Redis, the loader fetches them from the DB and they are written back at **line 130**:

```js
this.redis.mset(newValues).catch((err) => logger.warn(`Redis error during mset: ${err.message}`));
```

`mset` sets keys **with no expiry** (TTL = -1). This is in direct contrast to `setSingle()` (line 91), which correctly uses `setex` with a caller-supplied TTL. Every transaction key written through `cache.get()` therefore lives in Redis until evicted by memory pressure.

**Root cause / failed assumption:** The code assumes Redis's eviction policy (`allkeys-lfu`, default) will naturally bound memory usage. It does prevent hard OOM, but it does not prevent an attacker from continuously displacing legitimate cached entries with attacker-controlled unique keys, destroying cache effectiveness.

**Exploit flow:**
1. Attacker sends a high-rate stream of `GET /api/v1/transactions?timestamp=gt:{T}&timestamp=lt:{T+delta}` requests, varying `T` across the full historical range of the mirror node.
2. Each request hits `getTransactionTimestamps()`, returns a page of `payerAndTimestamps` rows from the DB.
3. `cache.get()` calls `mgetBuffer` — all keys are misses (never seen before).
4. `loader` fetches full transaction details from the DB.
5. `mset` writes up to `limit` (default 25) new persistent keys per request into Redis.
6. After enough requests, Redis reaches 250 MB; `allkeys-lfu` begins evicting the least-recently-used keys — which are the legitimate, frequently-needed entries, not the attacker's one-shot keys.
7. Legitimate API consumers now experience constant cache misses, forcing every request to hit the database.

**Why existing checks are insufficient:**
- `maxMemoryPolicy: allkeys-lfu` (docs/configuration.md line 595) prevents Redis from crashing but does not prevent cache pollution — it actively evicts legitimate entries to make room for attacker keys.
- There is **no rate limiting** on the Node.js REST API transactions endpoint. The grep across `rest/**/*.js` for `rateLimit|throttle|express-rate` returns only a test utility file. The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the Java web3 module.
- `maxmemory: 250Mb` (docs/configuration.md line 594) is a fixed ceiling, not a per-key or per-client bound.

### Impact Explanation

Cache thrashing forces every `GET /transactions` request to execute full DB queries (two CTEs joining `crypto_transfer`, `token_transfer`, and `transaction` tables). Under sustained attack, the database connection pool (max 10 connections, docs/configuration.md line 556) saturates, query queue depth grows, and response latency spikes to the `statementTimeout` ceiling (20 s). This constitutes a non-network-based DoS: the attacker does not need to saturate bandwidth — they only need to sustain enough unique-timestamp requests to keep Redis at capacity and LFU eviction active.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the public API spec, and no high-bandwidth connection. A single attacker with a modest HTTP client can issue hundreds of requests per second. The Hedera mirror node's transaction history spans years, providing an effectively unbounded supply of unique timestamp ranges. The attack is repeatable and stateless — the attacker does not need to track state between requests.

### Recommendation

1. **Add TTL to `mset` writes in `Cache.get()`**: Replace `mset` with `msetex` or pipeline individual `setex` calls with a configurable TTL (e.g., matching the `cache-control: max-age` of the response, or a dedicated `cache.entityId.maxAge`-style config value). This ensures attacker-injected keys expire and cannot permanently displace legitimate entries.

2. **Add rate limiting to the REST API**: Apply a per-IP or global request-rate limiter (e.g., `express-rate-limit`) to `GET /transactions` and other cache-backed endpoints.

3. **Scope `maxmemory` and eviction per key namespace**: Consider using Redis key prefixes with separate logical databases or a dedicated Redis instance for transaction caching, so cache pollution from one endpoint cannot affect others.

### Proof of Concept

```bash
# Requires: curl, bash, a running mirror node with Redis enabled

BASE="https://<mirror-node-host>/api/v1/transactions"
START_NS=1000000000000000000  # arbitrary historical nanosecond timestamp

for i in $(seq 1 10000); do
  T=$((START_NS + i * 1000000000))   # each request shifts window by 1 second
  T_END=$((T + 60000000000))         # 60-second window
  curl -s -o /dev/null \
    "${BASE}?timestamp=gt:0.${T}&timestamp=lt:0.${T_END}&limit=25" &
  # throttle to ~200 req/s to avoid network saturation
  [ $((i % 200)) -eq 0 ] && sleep 1
done
wait

# Observe: Redis INFO memory shows used_memory approaching maxmemory (250MB).
# Observe: Redis INFO stats shows evicted_keys climbing.
# Observe: Legitimate repeated requests to a fixed timestamp range now always
#          miss cache (TTL=-1 keys for those timestamps were evicted by LFU),
#          causing DB query latency to dominate response time.
```
