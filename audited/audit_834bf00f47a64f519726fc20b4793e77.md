### Title
Unbounded Historical Topic Message Scan via `startTime=0` and No `endTime`/`limit` Allows Unprivileged DoS

### Summary
An unprivileged gRPC client can subscribe to any topic with `startTime=0`, `endTime=null`, and `limit=0`. All three values pass the existing validation constraints, causing the server to stream every topic message from network genesis with no server-side cap. This can be repeated by many clients simultaneously to exhaust server I/O, memory, and database resources.

### Finding Description

**Validation in `TopicMessageFilter.java`:**

- `startTime` carries `@Min(0)` and a custom validator `isValidStartTime()` (line 49–51):
  ```java
  public boolean isValidStartTime() {
      return startTime <= DomainUtils.now();
  }
  ```
  `startTime=0` satisfies `0 <= now()` — no lower bound on how far back a scan may reach. [1](#0-0) 

- `endTime` is a plain nullable `Long` with no `@NotNull`. `isValidEndTime()` (line 44–46) explicitly permits `null`:
  ```java
  return endTime == null || endTime > startTime;
  ``` [2](#0-1) 

- `limit` carries `@Min(0)`, so `limit=0` is valid. `hasLimit()` returns `false` when `limit == 0`:
  ```java
  public boolean hasLimit() { return limit > 0; }
  ``` [3](#0-2) 

**Execution in `TopicMessageServiceImpl.subscribeTopic()`:**

- Line 63 unconditionally fires the historical retriever with the attacker-controlled filter:
  ```java
  Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
  ``` [4](#0-3) 

- The `endTime` guard (line 79–81) is skipped entirely when `endTime == null`:
  ```java
  if (filter.getEndTime() != null) {
      flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
  }
  ``` [5](#0-4) 

- The `limit` guard (line 83–85) is skipped when `hasLimit()` is false:
  ```java
  if (filter.hasLimit()) { flux = flux.take(filter.getLimit()); }
  ``` [6](#0-5) 

- `pastEndTime()` returns `Flux.never()` when `endTime == null`, so the live stream never self-terminates:
  ```java
  if (topicContext.getFilter().getEndTime() == null) { return Flux.never(); }
  ``` [7](#0-6) 

- `GrpcProperties` defines no maximum scan window, no per-subscription message cap, and no rate limit: [8](#0-7) 

**Root cause:** The validation layer enforces only that `startTime` is non-negative and not in the future. It imposes no minimum scan-start age, no mandatory `endTime`, and no mandatory `limit`. All three omissions together allow a full genesis-to-present unbounded scan.

### Impact Explanation
A single subscription with `startTime=0` on a high-volume topic (e.g., a busy HCS topic used by a DeFi application) forces the database to page through potentially millions of rows and stream them over gRPC. Multiple concurrent such subscriptions exhaust:
- Database connection pool and I/O bandwidth
- JVM heap (Reactor buffers, message objects)
- gRPC network egress

The service becomes unavailable to legitimate subscribers. Because the live stream never terminates (`Flux.never()`), each malicious connection holds a server thread and DB cursor open indefinitely.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is the primary public API of the mirror node. No authentication is required. The attacker only needs a valid `topicId` (trivially obtained from any public Hedera explorer) and a standard gRPC client. The attack is fully scriptable, repeatable, and requires zero on-chain funds or privileged credentials.

### Recommendation
1. **Enforce a minimum `startTime` age**: Reject requests where `startTime < (now - maxLookbackWindow)`, configurable via `GrpcProperties` (e.g., default 7 days).
2. **Require a non-zero `limit` or a finite `endTime`**: Add a cross-field `@AssertTrue` validator that rejects filters where both `endTime == null` and `limit == 0`.
3. **Add a server-side hard cap**: In `subscribeTopic()`, unconditionally apply `.take(grpcProperties.getMaxMessages())` before returning the flux.
4. **Rate-limit per client IP/connection**: Use Netty connection limits already partially configured via `NettyProperties` to cap concurrent subscriptions per source.

### Proof of Concept
```java
// Standard grpc-java client, no credentials needed
ManagedChannel channel = ManagedChannelBuilder.forAddress("mainnet-public.mirrornode.hedera.com", 5600)
    .usePlaintext().build();
ConsensusServiceGrpc.ConsensusServiceStub stub = ConsensusServiceGrpc.newStub(channel);

ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(ConsensusTopicID.newBuilder().setTopicNum(1234).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).setNanos(0).build())
    // No consensusEndTime set → endTime=null
    // No limit set → limit=0
    .build();

stub.subscribeTopic(query, new StreamObserver<ConsensusTopicResponse>() {
    public void onNext(ConsensusTopicResponse r) { /* receives ALL messages from genesis */ }
    public void onError(Throwable t) {}
    public void onCompleted() {}
});
// Repeat with N parallel clients to exhaust server resources
```
