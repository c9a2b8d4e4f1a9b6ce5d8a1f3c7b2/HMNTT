### Title
`deleted <> true` SQL Predicate Silently Excludes Newly Created Entities with `deleted=NULL` from Alias Lookups

### Summary
The `findByAlias()` query in `importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java` uses `deleted <> true`, which in SQL evaluates `NULL <> true` as `NULL` (falsy), silently excluding any entity whose `deleted` column is `NULL`. Because `CryptoCreateTransactionHandler.doUpdateEntity()` never sets `deleted`, and the upsert SQL resolves to `coalesce(deleted, e_deleted, null) = null` for brand-new entities, every account created via a standard CryptoCreate transaction is stored with `deleted=NULL`. This causes alias-based lookups to fail in both the importer's `EntityIdServiceImpl` (after a cache miss/restart) and the `rest-java` `EntityServiceImpl` (which has no cache at all), producing missing or unresolved downstream records.

### Finding Description

**Exact code locations:**

The vulnerable query: [1](#0-0) 

The same defect in rest-java: [2](#0-1) 

**Root cause — `deleted` column is intentionally nullable:**

Migration `V1.39.1__upsert_support.sql` explicitly dropped the `NOT NULL` constraint and the `DEFAULT false` from the `deleted` column, with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*: [3](#0-2) 

**Root cause — `CryptoCreateTransactionHandler` never sets `deleted`:**

`doUpdateEntity()` sets `type`, `alias`, `key`, `evmAddress`, `balance`, `memo`, etc., but never calls `entity.setDeleted(false)`: [4](#0-3) 

**Root cause — upsert SQL propagates NULL:**

The generated upsert SQL for the `entity` table uses `coalesce(deleted, e_deleted, null)`. For a brand-new entity there is no existing row, so `e_deleted` is `NULL`; combined with `deleted=NULL` from the temp table, the stored value is `NULL`: [5](#0-4) 

**Exploit path through `EntityIdServiceImpl`:**

`lookup(AccountID)` for an alias-based account calls `findByAlias()` via the repository: [6](#0-5) [7](#0-6) 

The in-memory cache is populated by `notify()`, which does NOT skip entities with `deleted=null` (only skips when `deleted != null && deleted == true`): [8](#0-7) 

So during the same importer run the cache masks the bug. After any restart the cache is cold, `findByAlias()` hits the DB, and returns `Optional.empty()` for every account created via CryptoCreate — causing all subsequent transactions that reference those accounts by alias to produce unresolved entity IDs.

**Direct REST-API impact via `rest-java`:**

`rest-java/EntityServiceImpl.lookup()` has no cache; it calls `entityRepository.findByAlias()` directly and throws `EntityNotFoundException` on empty: [9](#0-8) 

**Existing tests confirm the behavior but treat it as correct:**

The `rest-java` integration test explicitly asserts that `deleted=null` entities return empty from `findByAlias`: [10](#0-9) 

The importer's own `findByAlias` test only exercises `domainBuilder.entity()` (which sets `deleted=false` as a test default) and never tests the `deleted=null` case: [11](#0-10) 

**Inconsistency with correct implementations:**

`graphql`, `web3`, and the Node.js REST service all use `deleted is not true` or `coalesce(deleted, false) <> true`, which correctly treats `NULL` as "not deleted": [12](#0-11) [13](#0-12) 

### Impact Explanation

Any account created via a standard CryptoCreate transaction (the most common Hedera operation) is stored with `deleted=NULL`. The `rest-java` mirror node API immediately returns HTTP 404 for any alias-based lookup of such an account. The importer silently fails to resolve entity IDs for those accounts after a restart, causing all downstream transaction records that reference those accounts by alias to be recorded with a null/empty entity ID — corrupting the mirror node's view of the ledger. Severity: **Medium–High** (data integrity, availability of alias-based API endpoints).

### Likelihood Explanation

Every unprivileged user who submits a CryptoCreate transaction with an alias (a standard, documented operation) triggers this condition. No special permissions, keys, or protocol knowledge are required. Importer restarts occur routinely during upgrades and maintenance. The `rest-java` impact is immediate and requires no restart. The condition is therefore **highly repeatable** by any network participant.

### Recommendation

Replace `deleted <> true` with `deleted is not true` in both affected repositories, matching the pattern already used correctly in `graphql` and `web3`:

```sql
-- importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java
select id from entity where alias = ?1 and deleted is not true

-- rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java
select id from entity where alias = ?1 and deleted is not true
```

Additionally, `CryptoCreateTransactionHandler.doUpdateEntity()` should explicitly set `entity.setDeleted(false)` so that newly created entities always have a definite `deleted` value, preventing the ambiguity entirely.

Update the importer's `EntityRepositoryTest.findByAlias()` to cover the `deleted=null` case (analogous to the `rest-java` test), and update the `rest-java` test assertion for `deleted=null` to expect the entity to be **found** (not empty) once the fix is applied.

### Proof of Concept

1. Submit a `CryptoCreate` transaction with a non-empty `alias` field (e.g., an ECDSA secp256k1 public key alias). The transaction succeeds on the Hedera network.
2. The importer processes the record. `CryptoCreateTransactionHandler.doUpdateEntity()` sets alias, key, type, balance, etc., but never sets `deleted`. The upsert SQL stores `coalesce(null, null, null) = null` for the `deleted` column.
3. **REST-java impact (immediate):** Query `GET /api/v1/accounts/<alias>` via the `rest-java` service. `EntityServiceImpl.lookup()` calls `entityRepository.findByAlias()` → SQL `deleted <> true` excludes the row → `Optional.empty()` → `EntityNotFoundException` → HTTP 404, even though the account exists and is active.
4. **Importer impact (after restart):** Restart the importer. The in-memory alias cache is empty. Submit a subsequent transaction (e.g., `CryptoTransfer`) that references the previously created account by alias. `EntityIdServiceImpl.lookup()` calls `findByAlias()` → DB returns empty → entity ID is unresolved → the transaction record is stored with a null entity reference, corrupting the mirror node's ledger view.
5. **Verification:** Run `SELECT id, alias, deleted FROM entity WHERE alias = <alias_bytes>;` — the row exists with `deleted IS NULL`. Run `SELECT id FROM entity WHERE alias = <alias_bytes> AND deleted <> true;` — returns zero rows. Run `SELECT id FROM entity WHERE alias = <alias_bytes> AND deleted IS NOT true;` — returns the correct row.
