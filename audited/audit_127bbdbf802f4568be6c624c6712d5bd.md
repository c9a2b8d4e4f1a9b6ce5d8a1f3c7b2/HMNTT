### Title
`BufferUnderflowException` in `getByEvmAddressAndType()` via Under-Length Hex EVM Address Input

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length validation, accepting any even-length hex string. `getByEvmAddressAndType()` wraps the result in a `ByteBuffer` and unconditionally calls `getInt()` + two `getLong()` reads assuming exactly 20 bytes, but a crafted short input with leading zero bytes satisfies the first reads and causes an uncaught `BufferUnderflowException` on the entity-ID read. The exception propagates uncaught out of the service layer on every such request.

### Finding Description

**Root cause — no length guard in `decodeEvmAddress()`:**

`GraphQlUtils.decodeEvmAddress()` strips the optional `0x` prefix and calls `Hex.decodeHex()`, returning a byte array of whatever length the caller supplied. There is no check that the result is exactly 20 bytes.

``` [3](#0-2) 

The method performs three sequential reads on the unwrapped buffer with no remaining-capacity check:

| Read | Bytes consumed | Cumulative bytes needed |
|------|---------------|------------------------|
| `buffer.getInt()` | 4 | 4 |
| `buffer.getLong()` (condition) | 8 | 12 |
| `buffer.getLong()` (entity ID) | 8 | 20 |

**Exact trigger (12 bytes / 24 hex chars):**  
Input `"000000000000000000000000"` (24 zeros) decodes to 12 bytes, all zero.

1. `buffer.getInt()` → reads bytes 0–3 → returns `0` ✓ (condition arm entered)
2. `buffer.getLong()` → reads bytes 4–11 → returns `0` ✓ (condition true, 0 bytes remain)
3. `buffer.getLong()` → needs 8 bytes, 0 remain → **`BufferUnderflowException`** thrown

The exception is unchecked (`java.nio.BufferUnderflowException extends RuntimeException`) and there is no `try/catch` anywhere in this call chain. Spring GraphQL's default error handler catches it at the framework boundary and returns a generic error response, but the exception is logged and the request fails hard every time.

**Note on the question's "11 bytes" claim:** With 11 bytes and leading zeros, the exception fires one step earlier — at the *condition* `getLong()` (7 bytes remain, needs 8) — not at the entity-ID `getLong()`. The described scenario (both first reads succeed, third fails) requires exactly **12–19 bytes** with the first 12 bytes all zero. Either way the `BufferUnderflowException` is thrown and uncaught.

### Impact Explanation

Any unauthenticated caller who can reach the GraphQL endpoint can send a malformed `evmAddress` string (e.g., `"000000000000000000000000"`) and force an unhandled `BufferUnderflowException` on every request. The GraphQL service returns an error for that request; the service thread itself is not killed (Spring's thread pool absorbs the exception). The realistic impact is a reliable per-request error / partial DoS on the GraphQL account/contract lookup path, not a full network shutdown. The "total network shutdown" severity label is overstated.

### Likelihood Explanation

- No authentication required; the GraphQL endpoint is public-facing.
- The payload is trivial: any even-length hex string shorter than 24 chars whose first 12 bytes are zero.
- Fully repeatable and scriptable; no race condition or timing dependency.
- The existing test suite does not cover this case (tests only use exactly 20-byte addresses or empty input).

### Recommendation

Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// Option A – enforce in decodeEvmAddress (preferred, single enforcement point)
byte[] bytes = Hex.decodeHex(evmAddress);
if (bytes.length != EVM_ADDRESS_LENGTH) {          // EVM_ADDRESS_LENGTH = 20
    throw new IllegalArgumentException(
        "evmAddress must be exactly 20 bytes, got " + bytes.length);
}
return bytes;

// Option B – guard in getByEvmAddressAndType before ByteBuffer reads
if (evmAddressBytes.length != 20) {
    return entityRepository.findByEvmAddress(evmAddressBytes)
                           .filter(e -> e.getType() == type);
}
```

Either fix eliminates the assumption that the buffer contains exactly 20 bytes before the three sequential reads.

### Proof of Concept

**Precondition:** GraphQL endpoint reachable (no auth required).

**Trigger (GraphQL query):**
```graphql
query {
  account(input: { evmAddress: "000000000000000000000000" }) {
    entityId { shard realm num }
  }
}
```

**Step-by-step:**
1. `evmAddress = "000000000000000000000000"` (24 hex chars = 12 bytes, all zero)
2. `decodeEvmAddress()` → returns `byte[12]` all zeros (no length check)
3. `ByteBuffer.wrap(byte[12])` → capacity 12
4. `buffer.getInt()` → 0 (4 bytes consumed, 8 remain)
5. `buffer.getLong()` → 0 (8 bytes consumed, 0 remain); condition `== 0` is true
6. `buffer.getLong()` → **`java.nio.BufferUnderflowException`** (0 bytes remain, 8 needed)
7. Exception propagates uncaught through `EntityServiceImpl` → Spring GraphQL error handler → caller receives a generic internal error response

**Repeatability:** Send the same request in a loop; every iteration throws the exception. No state required.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);

        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
