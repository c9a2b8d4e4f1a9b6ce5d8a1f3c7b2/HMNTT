### Title
Unconditional DB Existence Query in `getTokenRelationships()` Enables Unauthenticated Non-Network DoS via `isValidAccount()`

### Summary
`getTokenRelationships()` in `rest/controllers/tokenController.js` unconditionally calls `EntityService.isValidAccount()` for every request, which always executes a database query (`SELECT type FROM entity WHERE id = $1`) regardless of whether the account exists. The REST API has no rate limiting for unauthenticated users, allowing any external actor to sustain arbitrary DB load by sending syntactically valid requests with non-existent account IDs.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenController.js`, `getTokenRelationships()` (lines 66–71):

```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValidAccount = await EntityService.isValidAccount(accountId);  // line 68 — always hits DB
  if (!isValidAccount) {
    throw new NotFoundError();
  }
``` [1](#0-0) 

`EntityService.isValidAccount()` in `rest/service/entityService.js` (lines 60–63) unconditionally executes `entityExistenceQuery`:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [2](#0-1) 

The query issued is:

```sql
SELECT type FROM entity WHERE id = $1
``` [3](#0-2) 

**Root cause — failed assumption:** The code assumes that `isValidAccount()` is called infrequently or that upstream rate limiting will bound the call rate. Neither is true.

**Critical detail — no DB query from `getEncodedId()` for numeric IDs:** When the account ID is a numeric `shard.realm.num` string (e.g., `0.0.99999999`), `getEncodedId()` takes the branch at line 122–123 (`entityId.evmAddress === null ? entityId.getEncodedId() : ...`) which is a pure in-memory computation — **zero DB queries**. The sole DB interaction for such a request is the `isValidAccount()` call. One HTTP request = one guaranteed DB query. [4](#0-3) 

**No rate limiting in the REST API:** `rest/server.js` middleware stack is: `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. There is no rate-limiting middleware for unauthenticated users anywhere in the REST API. [5](#0-4) 

The `authHandler` only sets a custom response-size limit for authenticated users — it does not throttle or reject unauthenticated requests.

The throttling that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the **web3** module (contract calls) and has no effect on the REST API. [6](#0-5) 

The endpoint is registered with no middleware guard:

```js
router.getExt(getPath('tokens'), TokenController.getTokenRelationships);
``` [7](#0-6) 

### Impact Explanation
An attacker can drive a continuous stream of `SELECT type FROM entity WHERE id = $1` queries against the `entity` table. While each query is lightweight (primary-key lookup), at high concurrency these queries compete for DB connection-pool slots and I/O bandwidth with legitimate queries. The result is increased query latency and potential connection-pool exhaustion for all REST API consumers. Because the attacker needs only syntactically valid account IDs (trivially enumerable: `0.0.1`, `0.0.2`, …, `0.0.4294967295`) and no credentials, the attack surface is the entire public internet. The impact is availability degradation of the mirror node REST API and its backing database.

### Likelihood Explanation
Preconditions are minimal: no account, no token, no authentication, no special knowledge. The attacker needs only an HTTP client and a loop. The attack is repeatable indefinitely because there is no connection-level or IP-level throttle in the REST API layer. A single machine with a modest number of concurrent connections (e.g., 100) sending requests to `/api/v1/accounts/0.0.<N>/tokens` with incrementing `N` values sustains 100 simultaneous DB existence queries continuously. This is well within the capability of any script kiddie.

### Recommendation
1. **Eliminate the redundant query:** For the common case where `getEncodedId()` already resolved a numeric entity ID without a DB query, the existence check should be deferred to the actual data query (`TokenService.getTokenAccounts()`). If that query returns zero rows, return 404 — no separate existence check is needed.
2. **Add rate limiting to the REST API:** Apply a per-IP rate limiter (e.g., `express-rate-limit`) globally in `rest/server.js` before route handlers, covering all unauthenticated requests.
3. **Cache existence results:** If the existence check must remain, cache positive and negative results (e.g., in Redis, which is already optionally wired in) with a short TTL to avoid repeated DB hits for the same account ID.

### Proof of Concept
```bash
# Generate 10,000 requests with syntactically valid but non-existent account IDs
# Each request triggers exactly one SELECT against the entity table
for i in $(seq 900000000 900010000); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.$i/tokens" &
done
wait
```

Each request:
1. Passes OpenAPI schema validation (numeric account ID is syntactically valid).
2. `getEncodedId("0.0.<N>")` returns encoded ID with **no DB query**.
3. `isValidAccount(<encodedId>)` executes `SELECT type FROM entity WHERE id = $1` — account does not exist, returns `null`.
4. `NotFoundError` is thrown and a 404 is returned.
5. Net effect: one DB query consumed per HTTP request, with no authentication or prior state required.

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();

                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();

    }
