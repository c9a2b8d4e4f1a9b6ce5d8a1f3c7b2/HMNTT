### Title
Missing Transaction Hash Length Validation in `BlockTransaction()` Enables Unauthenticated DB Read Amplification

### Summary
`BlockTransaction()` in `rosetta/app/services/block_service.go` passes the caller-supplied `TransactionIdentifier.Hash` directly to `FindByHashInBlock()` without validating its byte length. `FindByHashInBlock()` in `rosetta/app/persistence/transaction.go` only checks that the string is valid hex (via `hex.DecodeString`), not that it decodes to the expected 48-byte Hedera transaction hash length. Any valid hex string of incorrect length passes the check and causes a full DB query that is structurally guaranteed to return zero rows, with no negative-result caching, allowing any unauthenticated caller to sustain DB read amplification indefinitely.

### Finding Description

**Exact code path:**

`BlockTransaction()` — `rosetta/app/services/block_service.go` lines 87–92 — passes `request.TransactionIdentifier.Hash` verbatim to `FindByHashInBlock()`:

```go
transaction, err := s.FindByHashInBlock(
    ctx,
    request.TransactionIdentifier.Hash,   // no length check
    block.ConsensusStartNanos,
    block.ConsensusEndNanos,
)
``` [1](#0-0) 

`FindByHashInBlock()` — `rosetta/app/persistence/transaction.go` lines 180–183 — only rejects non-hex or odd-length strings:

```go
transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
if err != nil {
    return nil, hErrors.ErrInvalidTransactionIdentifier
}
``` [2](#0-1) 

Any even-length hex string (e.g., `"0xab"` = 1 byte, `"0xabcd"` = 2 bytes) passes `hex.DecodeString` successfully. The decoded byte slice is then used directly in the DB query:

```go
if err = db.Raw(
    selectTransactionsByHashInTimestampRange,
    sql.Named("hash", transactionHash),
    ...
).Find(&transactions).Error; ...
``` [3](#0-2) 

The SQL filter is `transaction_hash = @hash`: [4](#0-3) 

Since all real Hedera transaction hashes are 48 bytes, a hash of any other length will never match any row. The query always returns zero results, and `ErrTransactionNotFound` is returned — with no caching of this negative result anywhere in the call stack.

**Root cause:** The only validation gate is `hex.DecodeString`, which checks encoding validity but not semantic length. There is no guard of the form `len(transactionHash) != 48`.

**Existing test confirms the behavior:** `TestFindByHashInBlockThrowsNotFound` uses `"0x123456"` (3 bytes) and expects a not-found error, confirming the DB query executes and returns empty for wrong-length hashes: [5](#0-4) 

**No application-level rate limiting or authentication** is applied to the `/block/transaction` endpoint in the Go server: [6](#0-5) 

The only rate limiting present is an optional Traefik ingress middleware (`average: 10` per `requestHost`), which is not enforced at the application layer and is trivially bypassed by rotating source hosts or direct access: [7](#0-6) 

### Impact Explanation

Each crafted request causes two DB queries: one to validate the block (`FindByIdentifier`) and one guaranteed-empty transaction hash scan. Because there is no negative-result cache, every repeated request re-executes the full table scan against the `transaction` table filtered by timestamp range and an impossible hash. Under sustained load this amplifies DB read I/O, degrades query throughput for legitimate users, and can exhaust DB connection pool capacity. The impact is service degradation (griefing), not fund loss or data corruption, consistent with the Medium severity classification.

### Likelihood Explanation

The precondition is trivial: the attacker only needs a valid block identifier (index + hash), which is public information obtainable from any `/block` query. No credentials, tokens, or privileged access are required. The malformed hash (`"0xaa"`, `"0xaabb"`, etc.) is a two-character string. The attack is fully scriptable, repeatable from a single host, and produces a deterministic DB hit on every request. The Traefik rate limit of 10 req/s per `requestHost` is the only throttle, and it is bypassable and not universally deployed.

### Recommendation

Add an explicit byte-length check in `FindByHashInBlock()` immediately after `hex.DecodeString`, before the DB query is issued:

```go
transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
if err != nil {
    return nil, hErrors.ErrInvalidTransactionIdentifier
}
// Hedera transaction hashes are exactly 48 bytes
if len(transactionHash) != 48 {
    return nil, hErrors.ErrInvalidTransactionIdentifier
}
```

This rejects structurally invalid hashes at zero DB cost. Alternatively, enforce the same check at the service layer in `BlockTransaction()` before calling `FindByHashInBlock()`, or add a shared hash-validation helper used by both the service and persistence layers.

### Proof of Concept

**Preconditions:** Rosetta node is running and accessible. No credentials needed.

**Step 1 — Obtain a valid block identifier:**
```bash
curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{}}'
```
Extract `block_identifier.index` and `block_identifier.hash` from the response.

**Step 2 — Send repeated requests with a wrong-length hex transaction hash:**
```bash
while true; do
  curl -s -X POST http://<rosetta-host>/block/transaction \
    -H 'Content-Type: application/json' \
    -d '{
      "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
      "block_identifier": {"index": <BLOCK_INDEX>, "hash": "<BLOCK_HASH>"},
      "transaction_identifier": {"hash": "0xab"}
    }' > /dev/null
done
```

**Expected result:** Each request returns `{"code":...,"message":"Transaction not found"}` (HTTP 500), and each request causes a full DB query against the `transaction` table. DB query rate scales linearly with request rate. No caching prevents re-execution.
