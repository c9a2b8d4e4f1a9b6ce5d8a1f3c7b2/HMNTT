### Title
Unbounded Concurrent Subscriptions in `subscribeTopic()` Enable Resource Exhaustion Causing Missing Records for Legitimate Subscribers

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent gRPC subscriptions from any unauthenticated caller with no rate limiting or maximum subscriber enforcement. The `subscriberCount` `AtomicLong` in `TopicMessageServiceImpl` is wired exclusively to a Micrometer gauge metric and is never checked before admitting a new subscription. An attacker flooding the endpoint with concurrent subscriptions exhausts the shared DB connection pool and `boundedElastic` thread scheduler, causing legitimate subscribers' historical retrieval queries to fail or stall, resulting in missing records.

### Finding Description

**Entry point** — `ConsensusController.subscribeTopic()` (lines 43–53): no authentication interceptor, no per-IP or per-client rate limit, no connection cap. The only interceptor present (`GrpcInterceptor`, test path) sets an endpoint-context label and calls `next.startCall()` unconditionally.

**Metric-only counter** — `TopicMessageServiceImpl` declares:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
```
It is registered as a Micrometer `Gauge` (lines 52–55) and incremented/decremented via `doOnSubscribe`/`doFinally` (lines 89–90). There is **no guard** of the form `if (subscriberCount.get() >= maxAllowed) return error`. `GrpcProperties` has no `maxActiveSubscriptions` field whatsoever.

**Per-subscription resource cost** — every call to `TopicMessageServiceImpl.subscribeTopic()` (lines 59–91) unconditionally:
1. Calls `PollingTopicMessageRetriever.retrieve(filter, true)` which issues `topicMessageRepository.findByFilter()` — a live JDBC/R2DBC query against the DB (line 63, retriever lines 51–62, 78).
2. Schedules work on `Schedulers.boundedElastic()` (retriever line 41, service line 70).
3. Registers a new downstream subscriber on the shared `SharedPollingTopicListener` which uses `onBackpressureBuffer(maxBufferSize, BufferOverflowStrategy.ERROR)` (SharedTopicListener line 24).

**Exploit flow**:
- Attacker opens N concurrent gRPC streams (e.g., N = 5 000) each calling `subscribeTopic` with a valid topic ID and a past `startTime` (so historical retrieval fires immediately).
- Each stream triggers a DB query. The R2DBC/JDBC connection pool (bounded, typically 10–50 connections by default) is saturated.
- Legitimate subscribers' `topicMessageRetriever.retrieve()` calls queue behind the pool, time out (retriever line 59: `timeout(retrieverProperties.getTimeout())`), or are rejected.
- The `boundedElastic` scheduler queue fills; new tasks are rejected with `RejectedExecutionException`.
- `SharedTopicListener`'s per-subscriber backpressure buffer overflows (`BufferOverflowStrategy.ERROR`), terminating those subscriber streams with an error — causing them to miss live messages.

**Why existing checks fail**: `subscriberCount` is read-only from a safety perspective. `GrpcProperties` has no cap. The single `GrpcInterceptor` in the production path only calls `EndpointContext.setCurrentEndpoint()`. No IP-level or token-level throttle exists at the gRPC layer.

### Impact Explanation
Legitimate subscribers experience two classes of missing records:
1. **Historical gap**: DB pool exhaustion causes `retrieve()` to time out before returning all historical messages; the subscriber's `TopicContext` advances with an incomplete baseline, and the gap-recovery safety check (line 67–70) also fails for the same reason.
2. **Live gap**: `SharedTopicListener` buffer overflow terminates the live stream with an error, dropping in-flight messages.

Both map directly to the stated scope: "Incorrect or missing records exported to mirror nodes." Severity is **High** — a public gRPC endpoint with no authentication and no connection cap is a single-point denial-of-service for all topic subscribers on the node.

### Likelihood Explanation
The gRPC port is publicly reachable by design (mirror node public API). No credentials are required. A single attacker machine can open thousands of HTTP/2 streams over a small number of TCP connections (HTTP/2 multiplexing). The attack is trivially repeatable with any gRPC client library (e.g., `grpcurl` in a loop, or a simple Go/Java client). No privileged access, no leaked secrets, no insider knowledge beyond the public proto definition is needed.

### Recommendation
1. **Enforce a maximum active subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Flux.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
   Add `maxSubscribers` (e.g., default 500) to `GrpcProperties`.
2. **Add a gRPC server interceptor** that enforces per-IP or per-client-ID rate limiting (token bucket / sliding window) before the call reaches the controller.
3. **Configure Netty's `maxConcurrentCallsPerConnection`** via `NettyProperties` to bound streams per TCP connection.
4. **Tune the DB connection pool** with an explicit `maximumPoolSize` and a short `connectionTimeout` so pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept
```bash
# Requires grpcurl and a running mirror-node gRPC endpoint
# Open 2000 concurrent subscriptions with a historical start time
for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# In a separate terminal, attempt a legitimate subscription and observe:
# - Timeout / DEADLINE_EXCEEDED on historical retrieval, OR
# - Stream terminated with RESOURCE_EXHAUSTED from buffer overflow
grpcurl -plaintext \
  -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: error or incomplete message stream (missing records)
```
