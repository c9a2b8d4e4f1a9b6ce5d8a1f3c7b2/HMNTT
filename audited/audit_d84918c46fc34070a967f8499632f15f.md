### Title
Cache Key Collision at Zero Causes Wrong Alias Returned for Alias-Based Accounts in `updateOperationAccountAlias`

### Summary
In `rosetta/app/services/block_service.go`, `updateOperationAccountAlias` uses `accountId.GetId()` as the cache key. For any alias-based `AccountId` (created via `NewAccountIdFromAlias`), `GetId()` always returns `0` because `EncodedId` is never populated. This causes all alias-based accounts to share cache key `0`, so the first alias account's result poisons the cache and is returned for every subsequent alias account lookup.

### Finding Description
**File:** `rosetta/app/services/block_service.go`, function `updateOperationAccountAlias`, lines 114–125.

**Root cause:** `AccountId.GetId()` returns `a.accountId.EncodedId`. For alias-only accounts (`NewAccountIdFromAlias`), `accountId` is initialized as `domain.EntityId{ShardNum: shard, RealmNum: realm}` with `EncodedId` left at its zero value. Thus `GetId()` always returns `0`.

**Exploit flow:**

1. Attacker sends `GET /block` or `GET /block/transaction` for a block containing a transaction whose operation has an alias-based `AccountId` (account A, alias `0xAAA...`).
2. `updateOperationAccountAlias` calls `s.entityCache.Get(0)` → cache miss.
3. `GetAccountAlias` is called with `accountId.GetId()` = 0; it queries `SELECT alias, id FROM entity WHERE id = 0` → no row found → returns `accountId` (alias A) unchanged.
4. `s.entityCache.Set(result.GetId(), result)` → `Set(0, aliasAccountA)` — cache poisoned at key 0.
5. Attacker (or any user) queries a block containing a different alias account B (`0xBBB...`).
6. `s.entityCache.Get(0)` → cache **hit** → returns alias account A.
7. `operations[index].AccountId` is set to alias account A instead of B — wrong alias exported.

**Why existing checks fail:** There is no guard for `accountId.GetId() == 0` or `accountId.HasAlias() == true` before the cache lookup/set. The cache is typed `*cache.Cache[int64, types.AccountId]` with no special handling for the zero key.

### Impact Explanation
The Rosetta API returns incorrect `account_identifier` values in block/transaction operation responses. Downstream consumers (exchanges, indexers, block explorers) relying on the Rosetta API receive wrong account aliases, leading to incorrect attribution of transfers and balance changes — a direct data-integrity violation for mirror node exports.

### Likelihood Explanation
Any unprivileged user with HTTP access to the Rosetta API can trigger this by querying two different blocks that each contain at least one operation with an alias-based account. No authentication, special role, or network position is required. Alias accounts are a standard Hedera feature and appear in normal network traffic, making this trivially reproducible on any live deployment.

### Recommendation
Before using `GetId()` as the cache key, check whether the account has an alias. If it does, either:
- Skip the cache entirely for alias-based accounts (since `GetAccountAlias` for an alias account queries by id=0 and is effectively a no-op anyway), or
- Use a composite key (e.g., `HasAlias()` flag + alias bytes as key) so alias accounts are keyed by their actual alias rather than the shared zero value.

Minimal fix in `updateOperationAccountAlias`:
```go
accountId := operations[index].AccountId
if accountId.HasAlias() {
    // alias accounts: GetId() == 0, skip cache to avoid collision
    operations[index].AccountId = accountId
    continue
}
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    ...
}
```

### Proof of Concept
1. Deploy the Rosetta service against a mirror node with at least two alias accounts (A and B) that have appeared in different blocks.
2. `GET /block` for a block containing an operation for alias account A (`0xAAA...`). Observe the response correctly shows `0xAAA...`.
3. `GET /block` for a block containing an operation for alias account B (`0xBBB...`). Observe the response **incorrectly** shows `0xAAA...` (account A's alias) instead of `0xBBB...`.
4. Confirm by restarting the service (clearing the cache) and querying step 3 first — it now correctly returns `0xBBB...`, proving the bug is cache-state-dependent.
