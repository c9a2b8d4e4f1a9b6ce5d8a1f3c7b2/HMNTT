### Title
Unauthenticated Alias Lookup Causes Unbounded DB Queries with No Rate Limiting on `/api/v1/accounts/{ownerId}/hooks`

### Summary
Any unauthenticated external user can supply a syntactically valid but non-existent alias as `ownerId` in the hooks endpoint, causing `EntityServiceImpl.lookup()` to execute a native SQL query against the `entity` table on every request before throwing `EntityNotFoundException`. The `rest-java` module has no rate limiting, no caching, and no authentication on this endpoint, so an attacker can sustain an unbounded stream of DB queries, degrading throughput for legitimate users.

### Finding Description
**Exact code path:**

`HooksController.getHooks()` (line 90) calls `hookService.getHooks(hooksRequest)`. [1](#0-0) 

`HookServiceImpl.getHooks()` immediately calls `entityService.lookup(request.getOwnerId())` at line 35, before any repository interaction. [2](#0-1) 

`EntityServiceImpl.lookup()` for an `EntityIdAliasParameter` executes a native SQL query `select id from entity where alias = ?1 and deleted <> true` via `entityRepository.findByAlias()`. If the alias does not exist, `Optional.empty()` is returned and `EntityNotFoundException` is thrown unconditionally. [3](#0-2) [4](#0-3) 

`GenericControllerAdvice.notFound()` catches the exception and returns HTTP 404 with no error-level logging — the exception handling itself is lightweight. [5](#0-4) 

**Root cause:** The real overhead is not exception-handling stack unwinding (which is trivial here) but the unconditional native DB query issued for every alias-typed `ownerId` before the 404 is returned. There is no caching on the alias lookup path and no rate limiting anywhere in the `rest-java` module — the bucket4j throttle infrastructure exists only in the `web3` module. [6](#0-5) 

**Why existing checks fail:** `EntityIdAliasParameter.valueOfNullable()` validates only the syntactic format (base-32, 40–70 chars). A syntactically valid alias that does not exist in the DB passes format validation and proceeds directly to the DB query. [7](#0-6) 

### Impact Explanation
Each crafted request forces one native SQL query against the `entity` table. At high request rates this exhausts DB connection pool slots and increases query latency for all concurrent legitimate requests. Because the endpoint is publicly accessible with no authentication and no rate limiting in `rest-java`, the attacker has no cost barrier. The impact is service degradation (griefing), not data exfiltration or fund loss, consistent with the medium scope classification.

### Likelihood Explanation
Preconditions are zero: no account, no token, no privileged access. Any HTTP client can generate syntactically valid base-32 aliases in a tight loop. The attack is trivially scriptable, repeatable, and requires no knowledge of the system beyond the public API path `/api/v1/accounts/{alias}/hooks`.

### Recommendation
1. **Add rate limiting to `rest-java`**: Apply a per-IP or global request-rate limiter (e.g., bucket4j, as already used in `web3`) to the hooks endpoint or to all `rest-java` endpoints.
2. **Cache negative alias lookups**: Introduce a short-lived (e.g., 5–30 s) negative-result cache in `EntityServiceImpl.lookup()` for alias and EVM-address lookups so repeated misses for the same alias do not hit the DB.
3. **Fail fast on alias format before DB**: Already done syntactically; consider adding a bloom-filter or in-memory set of known-valid aliases if the entity set is bounded.

### Proof of Concept
```bash
# Generate a syntactically valid but non-existent base-32 alias
ALIAS="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"  # 40 base-32 chars, valid format

# Flood the endpoint (no authentication required)
while true; do
  curl -s -o /dev/null \
    "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/hooks"
done
```

Each iteration triggers `select id from entity where alias = ?1 and deleted <> true` with no DB result, throws `EntityNotFoundException`, and returns HTTP 404. At sufficient concurrency this saturates the DB connection pool and degrades response times for legitimate users.

    public static final String ALIAS_REGEX = "^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$";
    public static final Pattern ALIAS_PATTERN = Pattern.compile(ALIAS_REGEX);
    private static final BaseEncoding BASE32 = BaseEncoding.base32().omitPadding();

    static @Nullable EntityIdAliasParameter valueOfNullable(String id) {
        var aliasMatcher = ALIAS_PATTERN.matcher(id);

        if (!aliasMatcher.matches()) {
            return null;
        }

        var properties = CommonProperties.getInstance();
        long shard = properties.getShard();
        long realm = properties.getRealm();
        String realmString;

        if ((realmString = aliasMatcher.group(4)) != null) {
            realm = Long.parseLong(realmString);
            shard = Long.parseLong(aliasMatcher.group(2));
        } else if ((realmString = aliasMatcher.group(2)) != null) {
            realm = Long.parseLong(realmString);
        }

        var alias = BASE32.decode(aliasMatcher.group(5));
        return new EntityIdAliasParameter(shard, realm, alias);
    }
