### Title
Per-Node In-Memory Gas Bucket Enables Multi-Node Gas Budget Exhaustion DoS

### Summary
The `gasLimitBucket` in `ThrottleManagerImpl` is a local, in-memory bucket4j instance with `SynchronizationStrategy.SYNCHRONIZED`, meaning each deployed mirror-node instance maintains a completely independent gas budget. An unprivileged attacker can simultaneously saturate the gas budget of each node independently by sending maximum-gas requests at the allowed RPS cap, causing all targeted nodes to reject legitimate user requests with `ThrottleException` — no authentication or special privilege required.

### Finding Description

**Code location:**
- `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 34–45
- `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42

**Root cause:**

`ThrottleConfiguration.java` constructs the `gasLimitBucket` using only `SynchronizationStrategy.SYNCHRONIZED`:

```java
return Bucket.builder()
        .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
        .addLimit(limit)
        .build();
```

`SYNCHRONIZED` is a JVM-local thread-safety strategy — it is not a distributed bucket. There is no Redis, Hazelcast, or any shared backend used for the web3 throttle (confirmed: the only Redis usage in the codebase is in the `importer` and `grpc` modules, not `web3`).

**Token math (default config):**
- `gasPerSecond = 7,500,000,000` → after `scaleGas` (÷10,000): **750,000 tokens/second** per node
- `maxGasLimit = 15,000,000` → after `scaleGas`: **1,500 tokens** consumed per request
- `requestsPerSecond = 500` per node

500 requests/second × 1,500 tokens = **750,000 tokens/second** — exactly the bucket capacity. One attacker sending 500 max-gas requests/second to a single node exhausts its entire gas budget within 1 second.

**Why `restore()` does not save the node:**

`ContractCallService.restoreGasToBucket()` only returns tokens equal to `gasLimit - gasUsed`. If the attacker calls a gas-exhausting contract function (e.g., an infinite loop that hits the gas ceiling), `gasUsed == gasLimit`, so `gasRemaining = 0` and **zero tokens are restored**:

```java
final var gasRemaining = gasLimit - result.gasUsed();
throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline)); // restores 0
```

The bucket stays depleted for the entire refill window (1 second), during which all legitimate requests receive `GAS_PER_SECOND_LIMIT_EXCEEDED`.

**Exploit flow:**
1. Attacker identifies N deployed mirror-node instances (via DNS round-robin, known IPs, or load-balancer bypass).
2. For each node, attacker opens 500 concurrent connections and sends `POST /api/v1/contracts/call` with `gas=15000000` and calldata targeting a gas-exhausting contract.
3. Each node's `gasLimitBucket` is drained to 0 within 1 second.
4. All subsequent legitimate requests to those nodes throw `ThrottleException("Gas per second rate limit exceeded.")`.
5. Attacker sustains the attack by maintaining 500 RPS per node — this is within the `rateLimitBucket` cap, so no secondary throttle fires.

### Impact Explanation
Any number of mirror-node instances can be rendered unavailable to legitimate users simultaneously. Since each node is independently throttleable, an attacker targeting ≥30% of the deployed fleet causes a partial service outage for all users routed to those nodes. The nodes themselves remain running (no crash), but every incoming `eth_call`/`eth_estimateGas` request is rejected for as long as the attacker sustains 500 RPS per targeted node. This maps directly to the Medium severity classification: ≥30% of processing nodes effectively offline without network-wide shutdown.

### Likelihood Explanation
The attack requires zero privileges — the `/api/v1/contracts/call` endpoint is unauthenticated. The attacker needs only:
- Knowledge of individual node IPs (discoverable via DNS enumeration, public infrastructure scans, or load-balancer bypass techniques).
- A gas-exhausting contract address on the target network (trivially deployable or findable).
- 500 HTTP connections per targeted node — well within reach of a single machine or small botnet.

The attack is fully repeatable and self-sustaining at 500 RPS/node. No brute force is involved; the attacker operates entirely within the system's own rate limits.

### Recommendation
Replace the local in-memory bucket with a distributed rate-limiting backend (e.g., bucket4j with a Redis `ProxyManager`) so that all nodes in the cluster share a single gas budget. Alternatively, implement a cluster-aware sidecar (e.g., an Envoy rate-limit service) in front of all nodes. As a short-term mitigation, reduce `gasPerSecond` and `requestsPerSecond` to values that make sustained exhaustion impractical, and add per-IP rate limiting at the ingress layer to prevent a single source from consuming the full RPS allowance.

### Proof of Concept
```
# Precondition: mirror-node instances at node1.example.com, node2.example.com, node3.example.com
# gas-exhausting contract deployed at 0xDEAD...BEEF on the target network

# For each node, run in parallel:
for NODE in node1.example.com node2.example.com node3.example.com; do
  for i in $(seq 1 500); do
    curl -s -X POST "https://$NODE/api/v1/contracts/call" \
      -H "Content-Type: application/json" \
      -d '{
        "from":"0x0000000000000000000000000000000000000001",
        "to":"0xDEADBEEFDEADBEEFDEADBEEFDEADBEEFDEADBEEF",
        "gas":15000000,
        "data":"0x<gas_exhausting_selector>"
      }' &
  done
done
wait

# Result: all three nodes return {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
# for all subsequent legitimate requests within the same 1-second window.
# Sustain by repeating the loop every second.
```
