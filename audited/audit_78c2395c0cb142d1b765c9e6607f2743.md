### Title
Unbounded In-Memory Response Buffering via `ShallowEtagHeaderFilter` Without Rate Limiting Enables Memory Exhaustion DoS

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers a `ShallowEtagHeaderFilter` over all `/api/*` endpoints with no size cap or concurrency guard. `ShallowEtagHeaderFilter` wraps every response in a `ContentCachingResponseWrapper` that accumulates the entire response body as a heap-allocated byte array before computing the ETag MD5 hash. Because the `rest-java` module has no rate-limiting layer (unlike the `web3` module, which has `ThrottleManagerImpl`), an unprivileged attacker can open many concurrent connections and force the JVM to hold multiple full response bodies in memory simultaneously, exhausting the 2048 MiB heap limit and degrading service availability for all users.

### Finding Description
**Exact location:** `rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, `etagFilter()`, lines 42–46.

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");   // all API paths
    return filterRegistrationBean;
}
```

`ShallowEtagHeaderFilter` (Spring's `org.springframework.web.filter.ShallowEtagHeaderFilter`) works by replacing the real `HttpServletResponse` with a `ContentCachingResponseWrapper`. Every byte written by the controller is appended to an in-memory `ByteArrayOutputStream`. Only after the entire response is written does the filter compute the MD5 hash, set the `ETag` header, and flush the buffered bytes to the real socket. This means **one full response body lives in heap memory for the entire duration of each request**.

**Missing control:** The `web3` module has an explicit `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j token-bucket, 500 req/s default) applied before any handler executes. The `rest-java` module has no equivalent. The configuration table for `hiero.mirror.restJava.*` lists no rate-limiting properties. The only filters registered for `rest-java` are `LoggingFilter` and `MetricsFilter`, neither of which rejects or throttles requests.

**Exploit flow:**
1. Attacker identifies a large-response endpoint, e.g. `GET /api/v1/transactions?limit=100` or any paginated list endpoint.
2. Attacker opens N concurrent HTTP connections (N ≤ Tomcat's default `maxConnections` / `maxThreads`, typically 200 threads).
3. Each thread enters `ShallowEtagHeaderFilter.doFilterInternal()`, executes the full DB query, and buffers the complete JSON response body in a `ContentCachingResponseWrapper` byte array.
4. With N concurrent requests each holding ~50–200 KB of buffered JSON, total heap pressure = N × response_size. At 200 threads × 200 KB = ~40 MB per wave; sustained flooding across multiple waves accumulates GC pressure and eventually triggers `OutOfMemoryError`.
5. The JVM pod is limited to 2048 MiB (`charts/hedera-mirror-rest-java/values.yaml`, line 308). Repeated flooding exhausts this budget, causing GC thrashing or OOM crash, making the service unavailable to legitimate users.

### Impact Explanation
Service availability for all users of the `rest-java` API is degraded or lost. The attacker does not need credentials, special headers, or knowledge of internal state — only the ability to send HTTP GET requests. The impact is limited to the `rest-java` pod (no economic damage to the Hedera network itself), consistent with the "Medium / griefing" classification. However, a crashed or GC-thrashing pod means all users lose access to the mirror node REST Java API until the pod restarts.

### Likelihood Explanation
The attack requires no privilege, no authentication, and no special tooling — a simple HTTP flood script suffices. The absence of any rate-limiting in `rest-java` (in contrast to the explicit bucket4j throttle in `web3`) means there is no server-side gate before the filter buffers memory. The attack is repeatable and automatable. The only natural bound is Tomcat's thread pool, which itself can be saturated as part of the same attack.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a `FilterRegistrationBean` (or Spring MVC `HandlerInterceptor`) using bucket4j (already a dependency in the project) to enforce a per-IP or global requests-per-second cap on `/api/*`, mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`.
2. **Replace `ShallowEtagHeaderFilter` with a size-aware variant**: Override `isEligibleForEtag()` to skip buffering for responses above a configurable byte threshold, or set `setWriteWeakETag(true)` and add a `ContentSizeLimitingOutputStream` wrapper to cap buffering.
3. **Configure Tomcat connection limits**: Set `server.tomcat.max-connections` and `server.tomcat.threads.max` explicitly in the `rest-java` application configuration to bound the number of concurrent buffering operations.
4. **Add a `Cache-Control: no-store` / short `max-age` policy** for endpoints where ETag caching provides little benefit, allowing `ShallowEtagHeaderFilter` to skip buffering via its `isEligibleForEtag()` hook.

### Proof of Concept
```bash
# Requires: wrk or ab; no authentication needed
# Target: rest-java service on port 8080

# 1. Confirm ETag is generated (ShallowEtagHeaderFilter active)
curl -I http://<rest-java-host>/api/v1/transactions?limit=100
# Expect: ETag: "..." header present

# 2. Flood with concurrent requests to force heap accumulation
wrk -t 20 -c 200 -d 60s \
  "http://<rest-java-host>/api/v1/transactions?limit=100"

# 3. Monitor JVM heap via actuator
watch -n1 'curl -s http://<rest-java-host>/actuator/metrics/jvm.memory.used \
  | python3 -c "import sys,json; d=json.load(sys.stdin); \
    print(d[\"measurements\"][0][\"value\"]/1e6, \"MB\")"'

# Expected result: heap climbs monotonically during flood;
# GC overhead increases; pod eventually OOMs or becomes
# unresponsive to legitimate requests.
```
