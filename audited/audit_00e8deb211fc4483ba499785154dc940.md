### Title
Gas Throttle Bypass via Intentional Out-of-Gas Execution Enables Sustained CPU Exhaustion

### Summary
The `restoreGasToBucket()` method in `ContractCallService` unconditionally refunds the full declared gas back to the global gas-per-second bucket whenever a transaction fails with `gasUsed == gasLimit` (the out-of-gas condition). An unprivileged attacker can exploit this by repeatedly submitting `eth_call` requests with callData targeting a tight-loop contract at the maximum gas limit, causing the EVM to spin for the maximum allowed duration on every request while the gas throttle is never actually depleted — leaving only the global request-rate bucket (500 req/s, shared across all users) as the sole constraint.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` (deducts gas tokens upfront) → `ContractCallService.doProcessCall()` line 115 → `transactionExecutionService.execute()` (synchronous EVM execution) → `restoreGasToBucket()` in the `finally` block.

**Root cause — `restoreGasToBucket()` lines 140–152:**

```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline);   // ← full refund on OOG
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

With the default `gasLimitRefundPercent = 100`, `gasLimitToRestoreBaseline = gasLimit`. When a tight-loop contract exhausts all gas, `result.isSuccessful() == false` and `result.gasUsed() == gasLimit`, so the branch on line 144 fires and **restores the entire declared gas back to the bucket**. The gas bucket ends each such request at the same level it started — the throttle is completely neutralised for this attack pattern.

**Failed assumption:** The design assumes that failed/out-of-gas transactions represent "wasted" gas that should be refunded so legitimate callers are not penalised. It does not account for an attacker who *intentionally* causes out-of-gas failures to bypass the gas throttle while still forcing the EVM to execute for the maximum duration.

**Exploit flow:**
1. Deploy (or reference an existing) contract containing `while(true){}` or equivalent tight loop.
2. Submit `eth_call` with `gas = maxGasLimit` (15,000,000 by default per `EvmProperties.maxGasLimit`).
3. `ThrottleManagerImpl.throttle()` deducts `scaleGas(15_000_000) = 1_500` tokens from the gas bucket (capacity 750,000).
4. `transactionExecutionService.execute()` runs the EVM synchronously until gas is exhausted — maximum CPU time per request.
5. `restoreGasToBucket()` detects `gasUsed == gasLimit` and restores all 1,500 tokens.
6. Gas bucket is back to its pre-request level; repeat immediately.
7. The only remaining constraint is the **global** `requestsPerSecond = 500` bucket (shared by all users).

**Why existing checks are insufficient:**

| Check | Why it fails |
|---|---|
| `validateContractMaxGasLimit` (ContractController line 93) | Caps gas at 15M — this is the attacker's desired value |
| Gas-per-second bucket (ThrottleManagerImpl line 40) | Fully refunded after each OOG execution |
| `requestsPerSecond = 500` bucket | Global, not per-IP; attacker monopolises the entire budget |
| No per-IP rate limiting | Confirmed: both buckets in `ThrottleConfiguration` are single global beans |

### Impact Explanation
An attacker can sustain up to 500 requests/second (the global rate limit), each forcing the EVM to execute a 15M-gas tight loop synchronously, with zero gas-bucket cost. Because the rate-limit bucket is global and not per-IP, a single attacker exhausts the entire server-wide request budget, simultaneously causing: (a) sustained high CPU load from EVM execution, and (b) denial of service for all other users who receive `429 Too Many Requests`. The CPU impact scales with the EVM interpreter's throughput — on typical hardware, executing 15M gas of arithmetic opcodes takes tens to hundreds of milliseconds per request; at 500 req/s this saturates available CPU cores.

### Likelihood Explanation
No authentication, no account balance, no privileged role is required — only the ability to send HTTP POST requests to `/api/v1/contracts/call`. The attack is trivially scriptable with a single `curl` loop or any HTTP load tool. The tight-loop contract can be pre-deployed on the target network or the attacker can use `to = 0x0` (contract creation path) with loop initcode. The gas-bucket bypass is deterministic and 100% reproducible on every request.

### Recommendation
1. **Do not refund gas for out-of-gas failures.** Change the condition in `restoreGasToBucket()` so that `gasUsed == gasLimit` (OOG) is treated the same as a successful execution — restore only `gasRemaining` (which is 0), not the full baseline. The current "refund on failure" logic is the direct root cause.
2. **Add per-IP rate limiting.** The global `rateLimitBucket` and `gasLimitBucket` beans should be replaced with per-source-IP buckets (e.g., using bucket4j's `ProxyManager` with a cache backend) so a single attacker cannot monopolise the shared budget.
3. **Enforce a wall-clock execution timeout** on `transactionExecutionService.execute()` independent of gas, to bound CPU time per request regardless of gas accounting.
4. **Lower `gasLimitRefundPercent` default** from 100% to a value like 25%–50% so that even legitimate failures partially drain the gas bucket, reducing the incentive for intentional OOG abuse.

            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }

            status = e.getMessage();

                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }

            throttleManager.restore(gasLimitToRestoreBaseline);
