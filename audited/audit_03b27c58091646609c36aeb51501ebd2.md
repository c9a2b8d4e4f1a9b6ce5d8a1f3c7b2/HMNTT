### Title
Unauthenticated Historical Hook Storage Disclosure via `timestamp` Parameter

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint in `HooksController.getHookStorage()` accepts a `timestamp` query parameter with no authentication or authorization check. When supplied, `isHistorical()` returns `true` and the service queries the `hook_storage_change` audit table instead of the current-state `hook_storage` table, exposing all historical storage writes including entries that have since been overwritten or deleted — data that is explicitly filtered out of the non-historical path.

### Finding Description

**Exact code path:**

`HooksController.java` lines 104–130 defines the endpoint with no security annotation and no access-control check: [1](#0-0) 

The `rest-java` module has no Spring Security dependency (`spring-boot-starter-security` is absent from `build.gradle.kts`) and no `SecurityConfig`, `@PreAuthorize`, or `@Secured` annotations anywhere in the module. [2](#0-1) 

`HookStorageRequest.isHistorical()` returns `true` for any non-empty `timestamp` parameter: [3](#0-2) 

`HookServiceImpl.getHookStorage()` branches unconditionally on that flag: [4](#0-3) 

The non-historical path explicitly filters deleted entries (`DeletedIsFalse`): [5](#0-4) 

The historical path (`getHookStorageChange`) applies **no deleted filter** and queries `hook_storage_change` directly: [6](#0-5) 

The `hook_storage_change` table stores every write event including `value_written = NULL/empty` (deleted) entries: [7](#0-6) 

**Root cause:** The design assumes the `timestamp` historical path is equivalent in sensitivity to the current-state path, but it is not — it surfaces the full mutation history including values that were intentionally deleted or overwritten. No privilege gate separates the two paths.

**Failed assumption:** The developers assumed that scoping queries by `owner_id` and `hook_id` (both attacker-supplied path parameters) is sufficient access control. It is not, because the attacker can enumerate any public account's hook storage history.

### Impact Explanation

An unauthenticated attacker can retrieve the complete write history of any hook's EVM storage slots, including:
- Storage values that were overwritten (prior values no longer in current state)
- Storage slots that were explicitly deleted (marked `deleted=true`, filtered from the current-state API)
- The `consensus_timestamp` of every mutation, enabling timeline reconstruction of hook behavior

For hooks implementing gossip-prevention logic, this reveals which storage keys were set, cleared, and when — potentially disclosing which transactions were suppressed and the internal state machine of the hook contract over its entire lifetime. This is a cross-tenant information disclosure: any user can query any account's hook history without owning that account.

### Likelihood Explanation

Preconditions: none. The attacker needs only a valid `ownerId` and `hookId` (both discoverable via the unauthenticated `GET /api/v1/accounts/{ownerId}/hooks` endpoint) and the ability to send an HTTP GET request. The trigger is a single HTTP request with `?timestamp=lte:9999999999`. The attack is trivially repeatable, scriptable, and requires no credentials or on-chain assets.

### Recommendation

1. **Add authentication/authorization**: Require that the caller is the `ownerId` of the hook, or restrict the historical path to authenticated/privileged callers. Add Spring Security to `rest-java` and annotate `getHookStorage` with `@PreAuthorize`.
2. **Separate the historical endpoint**: Expose historical data only via a distinct, access-controlled endpoint (e.g., `/hooks/{hookId}/storage/history`) rather than a query-parameter branch of the public current-state endpoint.
3. **Filter deleted entries in the historical path**: At minimum, apply the same `deleted = false` filter in `HookStorageChangeRepository` queries so that deleted slots are not disclosed even if the historical path remains public.

### Proof of Concept

```
# Step 1: Discover hooks for any account (unauthenticated)
GET /api/v1/accounts/0.0.123/hooks
# Response reveals hook_id values, e.g. hook_id=1

# Step 2: Query current state (deleted entries filtered out)
GET /api/v1/accounts/0.0.123/hooks/1/storage
# Returns only non-deleted current entries

# Step 3: Query full historical audit log (no auth required)
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:9999999999
# Returns ALL historical writes from hook_storage_change including:
# - overwritten values (prior writes to same key)
# - deleted entries (value=null, deleted=true)
# - consensus_timestamp of every mutation
```

The `timestamp` parameter causes `isHistorical()` → `true` → `getHookStorageChange()` → `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` with `timestampUpperBound = Long.MAX_VALUE`, returning the complete mutation history with zero authentication.
