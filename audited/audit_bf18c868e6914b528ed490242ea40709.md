### Title
Integer Overflow in `PollingTopicListener.poll()` via Unchecked Narrowing Cast of `Long.MAX_VALUE` Limit

### Summary
`PollingTopicListener.poll()` performs an unchecked narrowing cast `(int) (filter.getLimit() - context.getCount().get())` at line 55. When an external user supplies `limit = Long.MAX_VALUE` (which passes the only validation constraint `@Min(0)`), the cast produces `-1`, causing `pageSize = -1`. This corrupts the internal query filter, bypassing the limit guard and causing unbounded or broken database queries against the Hashgraph message history.

### Finding Description

**Exact code path:**

In `PollingTopicListener.java`, `poll()` at lines 54–59: [1](#0-0) 

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
```

**Root cause:** The expression `(int) (filter.getLimit() - context.getCount().get())` performs a silent narrowing cast from `long` to `int`. When `filter.getLimit() = Long.MAX_VALUE` and `context.getCount().get() = 0`:

- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- Lower 32 bits: `0xFFFFFFFF` → signed int: **`-1`**

So `limit = -1`, and `Math.min(-1, maxPageSize) = -1`.

**Why validation fails:** `TopicMessageFilter.limit` is declared as: [2](#0-1) 

```java
@Min(0)
private long limit;
```

`@Min(0)` only rejects negative values. `Long.MAX_VALUE` is positive and passes this constraint. There is no `@Max` bound. The `@Validated` annotation on the class triggers Spring AOP method-level validation — it does **not** validate objects constructed via the Lombok `toBuilder().limit(-1).build()` call inside `poll()`, so the corrupted `newFilter` with `limit = -1` is created without any validation error.

**Downstream effect:** `hasLimit()` is: [3](#0-2) 

```java
public boolean hasLimit() {
    return limit > 0;
}
```

With `limit = -1`, `hasLimit()` returns `false`. When `findByFilter(newFilter)` is called with this corrupted filter, the repository sees no limit and issues an unbounded query against the topic message table, or passes `-1` as a SQL `LIMIT` parameter causing a database error.

### Impact Explanation

An attacker can cause one of two outcomes per polling cycle:
1. **Unbounded DB query:** If the repository checks `hasLimit()` before applying a SQL `LIMIT`, the query runs with no row cap, potentially returning the entire topic message history and exhausting JVM heap.
2. **Database error / message skip:** If `-1` is passed directly as a SQL `LIMIT` parameter, PostgreSQL raises an error, the `Flux` terminates early, and messages are silently skipped — corrupting the delivered Hashgraph history for that subscriber.

Both outcomes affect the integrity and availability of the mirror node's consensus topic stream, which is the primary data surface for Hashgraph history consumers.

### Likelihood Explanation

The Hedera gRPC `ConsensusTopicQuery` proto exposes a `uint64 limit` field that maps directly to `TopicMessageFilter.limit`. Any unauthenticated gRPC client can set this field to `Long.MAX_VALUE` (a valid `uint64`). No authentication or privilege is required. The attack is trivially repeatable: one subscription request per target topic is sufficient to trigger the overflow on every polling interval for the lifetime of the subscription.

### Recommendation

Replace the unchecked narrowing cast with a bounds-safe computation:

```java
long remaining = filter.getLimit() - context.getCount().get();
int limit = filter.hasLimit()
        ? (remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining)
        : Integer.MAX_VALUE;
```

Additionally, add an `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) constraint to `TopicMessageFilter.limit`: [2](#0-1) 

```java
@Min(0)
@Max(Integer.MAX_VALUE)   // add this
private long limit;
```

This ensures the overflow is impossible at the source, and the internal cast is always safe.

### Proof of Concept

**Preconditions:** A running mirror node gRPC endpoint with `PollingTopicListener` active (default when Redis/shared listener is not configured). Any existing topic ID.

**Steps:**
1. Construct a `ConsensusTopicQuery` gRPC request with `topicID` set to any valid topic and `limit` set to `9223372036854775807` (`Long.MAX_VALUE`).
2. Send the request to the mirror node's gRPC port (default `5600`).
3. The service builds a `TopicMessageFilter` with `limit = Long.MAX_VALUE`, which passes `@Min(0)` validation.
4. On the first polling interval, `poll()` computes `(int)(Long.MAX_VALUE - 0) = -1`.
5. `pageSize = Math.min(-1, maxPageSize) = -1`.
6. `newFilter` is built with `limit = -1`; `hasLimit()` returns `false`.
7. `findByFilter(newFilter)` executes with no row limit, returning all topic messages, or throws a DB error causing the stream to terminate silently.

**Expected result:** Either heap exhaustion from an unbounded result set, or silent stream termination causing message delivery gaps in the Hashgraph history feed.
