### Title
Historical Block Parameter Causes Bounded-but-Exploitable MeterRegistry Cardinality Inflation via YearMonth Tag

### Summary
The `getBlock()` method in `ContractCallService` applies a one-sided timestamp filter that only rejects future timestamps, leaving past timestamps from any historical block fully accepted. An unprivileged user can submit `eth_call` or `eth_estimateGas` requests with block numbers that resolve to record files whose `consensusEnd` timestamps span different calendar months, causing `getBlock()` to emit a distinct `YearMonth` string per month as the `TAG_BLOCK` tag value on the `invocationCounter` metric. Each unique tag combination permanently registers a new `Counter` object in the `MeterRegistry`.

### Finding Description

**Exact code path:**

`callContract()` (lines 100–107) sets the block supplier lazily from `recordFileService.findByBlockType(params.getBlock())`, which calls `RecordFileServiceImpl.findByBlockType()` → `recordFileRepository.findByIndex(block.number())`. No restriction is placed on how old the resolved record file may be. [1](#0-0) 

`doProcessCall()` unconditionally calls `updateMetrics()` in its `finally` block whenever `result != null` and `estimate == false`. [2](#0-1) 

`updateMetrics()` calls `getBlock()` and uses its return value as the `TAG_BLOCK` tag on `invocationCounter.withTags(...)`. [3](#0-2) 

`getBlock()` applies only a **future-timestamp guard** (`t <= Utils.getCurrentTimestamp()`). Any timestamp that is in the past — including one from years ago — passes the filter and is converted to a `YearMonth` string. [4](#0-3) 

The timestamp itself comes from `ContractCallContext.getTimestamp()` → `getTimestampOrDefaultFromRecordFile()`, which falls back to `RecordFile::getConsensusEnd` — the raw nanosecond consensus-end timestamp stored in the database for the resolved block. [5](#0-4) 

**Root cause:** The cardinality-reduction comment ("Filter future timestamps to reduce cardinality") reveals the design intent, but the implementation only guards one direction. There is no lower-bound guard. Any historical block whose `consensusEnd` falls in a calendar month not yet seen by the registry creates a brand-new `Counter` instance permanently stored in the `MeterRegistry`.

**Why existing checks are insufficient:** The `ThrottleManager` is gas-based, not request-count-based. A minimal-gas `eth_call` (e.g., a simple `view` function call) consumes very little gas and is largely refunded, making it cheap to issue many requests. There is no per-IP rate limit visible in this code path, and no authentication requirement.

### Impact Explanation

For Hedera mainnet (genesis September 2019), as of mid-2026 there are approximately 79 distinct calendar months. The `invocationCounter` tag space is the Cartesian product of `TAG_STATUS` × `TAG_TYPE` × `TAG_BLOCK`. With ~79 `TAG_BLOCK` values, ~2 `TAG_TYPE` values (ETH_CALL, ETH_ESTIMATE_GAS), and ~10–20 distinct `TAG_STATUS` values (response code strings), an attacker can force the creation of roughly 79 × 2 × 15 ≈ 2,370 unique `Counter` objects — compared to the baseline of ~30 counters when all traffic uses `LATEST`. Each `Counter` in Micrometer holds its own `Tags` object, internal state, and registry entry. While bounded, this is a deliberate, repeatable inflation that permanently occupies registry memory and pollutes monitoring dashboards and time-series databases (e.g., Prometheus scrape output grows proportionally). The severity matches the stated scope: griefing with no direct economic damage to network users.

### Likelihood Explanation

No privileges, credentials, or special network access are required. Any user of the public JSON-RPC endpoint can issue `eth_call` with an arbitrary `blockNumber` parameter. The attacker only needs to know (or enumerate) block numbers that correspond to record files from different calendar months — this information is publicly available via the mirror node REST API (`/api/v1/blocks`). The attack is fully repeatable and requires only ~79 requests to exhaust the entire historical tag space.

### Recommendation

Add a **lower-bound guard** inside `getBlock()` alongside the existing upper-bound guard, for example capping accepted historical timestamps to a rolling window (e.g., the last 12 months):

```java
private String getBlock() {
    long lowerBound = YearMonth.now(UTC).minusMonths(12)
        .atDay(1).atStartOfDay(UTC).toInstant()
        .toEpochMilli() * 1_000_000L; // convert to nanos
    return ContractCallContext.get()
        .getTimestamp()
        .filter(t -> t <= Utils.getCurrentTimestamp())
        .filter(t -> t >= lowerBound)           // NEW: reject very old timestamps
        .map(t -> YearMonth.from(Instant.ofEpochSecond(0L, t).atZone(UTC)).toString())
        .orElse(BlockType.LATEST.toString());
}
```

Alternatively, collapse all out-of-window historical timestamps to a single sentinel tag value (e.g., `"historical"`) to preserve some observability without unbounded tag growth.

### Proof of Concept

1. Query `/api/v1/blocks?order=asc&limit=100` to obtain block numbers whose `consensus_end` timestamps span multiple distinct calendar months (e.g., blocks from 2019-09, 2019-10, 2020-01, 2020-06, …).
2. For each such block number `N`, issue:
   ```
   POST /api/v1/contracts/call
   { "data": "0x", "to": "<any valid contract>", "blockNumber": "0x<hex(N)>" }
   ```
3. After cycling through ~79 block numbers covering all historical months, inspect the Prometheus metrics endpoint (`/actuator/prometheus`) and observe ~79 distinct `block` label values on the `hiero_mirror_web3_evm_invocation_total` metric, one per calendar month since mainnet genesis.
4. Confirm that each unique `(status, type, block)` combination is a permanently registered `Counter` in the `MeterRegistry`.
