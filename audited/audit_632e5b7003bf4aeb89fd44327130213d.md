### Title
Unbounded Recursive Key Mapping in GraphQL CommonMapper Causes StackOverflowError and Key Misrepresentation

### Summary
`CommonMapper.mapKey(Key)` and `mapKeyList(KeyList)` in the GraphQL module are mutually recursive with no depth limit. A protobuf `Key` with a deeply nested `KEYLIST` structure stored in the mirror node database will cause a `StackOverflowError` when any user queries the associated account via GraphQL. Because `StackOverflowError` extends `Error` (not `Exception`), it is not caught by the `catch (Exception e)` guard, propagates to the GraphQL framework, and causes the key field to be omitted or returned as an error — misrepresenting the account's actual key structure.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/mapper/CommonMapper.java`, lines 95–126:

```
mapKey(Key key)          // line 95
  case KEYLIST → mapKeyList(key.getKeyList())   // line 103
  case THRESHOLDKEY → mapKeyList(...)           // line 108

mapKeyList(KeyList keyList)   // line 113
  for each key → mapKey(key)  // line 122
```

These two methods call each other with no depth counter, no visited-set, and no base-case guard beyond reaching a primitive key type.

**Root cause / failed assumption:** The code assumes that protobuf `Key` structures stored in the database will always be shallow. `DomainUtils.getPublicKey()` in the same codebase explicitly guards against this with `if (depth > 2) return null;` (line 113 of `DomainUtils.java`), but `CommonMapper` has no equivalent guard.

**Why the existing `catch (Exception e)` is insufficient:**

```java
// line 85-91, CommonMapper.java
try {
    var key = Key.parseFrom(source);
    return mapKey(key);
} catch (Exception e) {          // ← catches Exception only
    logger.error(...);
    return null;
}
```

`StackOverflowError` is a subclass of `java.lang.Error`, not `java.lang.Exception`. It is not caught here and propagates up to the Spring GraphQL dispatcher, which returns a partial/error response for the `key` field.

**GraphQL-level depth limits do not help:** `MaxQueryDepthInstrumentation(10)` (line 44, `GraphQlConfiguration.java`) limits the depth of the *GraphQL query document*, not the depth of the protobuf object being deserialized from the database. These are orthogonal concerns.

**Exploit flow:**
1. Attacker submits a `CryptoCreate` or `CryptoUpdate` transaction on the Hedera network with a key that is a `KeyList` containing a `KeyList` containing a `KeyList` … (N levels deep, where N ≈ 500 is sufficient to overflow the JVM default stack).
2. The mirror node importer ingests this transaction and stores the raw protobuf key bytes in the `entity.key` column.
3. Any unauthenticated user sends a GraphQL query: `{ account(input: {id: "0.0.X"}) { key } }`.
4. `AccountMapper.map(Entity)` → `CommonMapper.mapKey(byte[])` → `Key.parseFrom(source)` → `mapKey(Key)` → `mapKeyList(...)` → `mapKey(Key)` → … → `StackOverflowError`.
5. The error propagates past the `catch (Exception e)` guard and is handled by the GraphQL framework, which omits the `key` field or returns a field-level error.
6. The response misrepresents the account: the key appears absent or unresolvable, when in fact a valid (if unusual) key exists on-chain.

### Impact Explanation
Any account whose on-chain key contains a sufficiently deep `KeyList` nesting will have its key permanently misrepresented in every GraphQL response. Clients relying on the GraphQL API to audit or verify an account's signing requirements will receive incorrect data, undermining the integrity of the mirror node's historical record. If the attacker queries the endpoint repeatedly (or if many accounts share such a key structure), the repeated `StackOverflowError` events also impose unnecessary JVM overhead.

### Likelihood Explanation
Creating a Hedera account requires only a small amount of HBAR (fractions of a cent at typical prices) — no privileged access is needed. The GraphQL endpoint is publicly accessible with no authentication. The Hedera network's own key-depth validation is separate from the mirror node's processing; even if the network enforces a limit, the mirror node must still handle whatever depth the network permits. The attack is repeatable and deterministic: once the account exists in the database, every GraphQL query for it triggers the same code path.

### Recommendation
Add a depth parameter to `mapKey(Key key)` and `mapKeyList(KeyList keyList)`, mirroring the pattern already used in `DomainUtils.getPublicKey()`:

```java
private static final int MAX_KEY_DEPTH = 10;

default Object mapKey(Key key) {
    return mapKey(key, 0);
}

default Object mapKey(Key key, int depth) {
    if (depth > MAX_KEY_DEPTH) {
        logger.warn("Key nesting depth exceeded limit");
        return null;
    }
    // ... existing switch, passing depth+1 to mapKeyList
}

default List<Object> mapKeyList(KeyList keyList, int depth) {
    // ... pass depth to mapKey(key, depth)
}
```

Additionally, change the catch clause to `catch (Throwable t)` (or at minimum `catch (Exception | Error e)`) so that any unexpected error during key mapping is contained and logged rather than propagated.

### Proof of Concept

```java
// Build a 600-level deep KeyList key
Key deepKey = Key.newBuilder()
    .setEd25519(ByteString.copyFrom(new byte[32]))
    .build();
for (int i = 0; i < 600; i++) {
    deepKey = Key.newBuilder()
        .setKeyList(KeyList.newBuilder().addKeys(deepKey))
        .build();
}
byte[] keyBytes = deepKey.toByteArray();

// Simulate what the GraphQL mapper does when it reads this from the DB
CommonMapper mapper = Mappers.getMapper(CommonMapper.class);
// The following call throws StackOverflowError (not caught by catch(Exception e)):
Object result = mapper.mapKey(keyBytes);
// result is never reached; StackOverflowError propagates to the GraphQL framework
```

Submit a `CryptoCreate` transaction on the Hedera network (testnet) with `keyBytes` as the account key. Query the mirror node GraphQL endpoint for the resulting account ID and observe that the `key` field is missing or returns a field-level error instead of the actual key structure.
