### Title
Unauthenticated Sequential Topic ID Enumeration via Unthrottled `TopicRepository.findById()` in rest-java

### Summary
The `/api/v1/topics/{id}` endpoint in the `rest-java` module is publicly accessible with no authentication and no application-level rate limiting. Because Hedera topic IDs are sequential `Long` primary keys, an unprivileged attacker can trivially enumerate all topics by iterating IDs from 1 to the maximum, with each request triggering a live database query via `TopicRepository.findById()`. This enables both full topic inventory disclosure and sustained database load amplification (griefing).

### Finding Description

**Code path:**

`TopicController.getTopic()` → `TopicServiceImpl.findById()` → `TopicRepository.findById()` [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause:** `TopicRepository` is a bare `CrudRepository<Topic, Long>` with no query customization. `TopicServiceImpl.findById()` calls `topicRepository.findById(id.getId())` unconditionally — no caching, no rate limiting, no authentication gate.

**Input surface:** `EntityIdNumParameter` accepts any numeric string matching `^((\d{1,4})\.)?( (\d{1,5})\.)?(\d{1,12})$`, meaning IDs 1 through 999999999999 are all syntactically valid inputs. [4](#0-3) 

**Why existing checks fail:** A thorough search of all `rest-java/src/main/**` Java sources confirms zero rate-limiting or throttling infrastructure for this module. The only throttling in the repository (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) lives exclusively in the `web3` module and applies only to contract-call requests — it has no effect on `rest-java` endpoints. [5](#0-4) [6](#0-5) 

The `rest-java` configuration registers only an ETag filter, a logging filter, and MVC argument resolvers — none of which impose request rate limits.

### Impact Explanation

1. **Information disclosure:** An attacker can enumerate the complete set of existing topic IDs and their metadata (admin key, submit key, memo, expiry, custom fees) with no credential requirement. This leaks the full topic inventory of the network.
2. **Resource exhaustion (griefing):** Each HTTP request causes one primary-key DB lookup against the `topic` table plus additional lookups in `entity` and `custom_fee` tables (see `TopicController` lines 33–36). A high-rate sequential scan from a botnet or single fast client can saturate the DB connection pool and degrade service for legitimate users. No economic damage occurs to network participants, but availability is impacted — consistent with the "griefing" scope classification. [7](#0-6) 

### Likelihood Explanation

**Preconditions:** None. No account, API key, or special network access is required. The endpoint is reachable by any internet client.

**Feasibility:** Topic IDs are sequential integers starting at 1. A single `curl` loop or a trivial Python script can enumerate thousands of IDs per second. The `EntityIdNumParameter` accepts plain integers (e.g., `GET /api/v1/topics/1`), requiring no knowledge of shard/realm encoding.

**Repeatability:** Fully repeatable and automatable. No state is modified, so the attacker faces no lockout or detection-based blocking at the application layer.

### Recommendation

1. **Application-level rate limiting:** Introduce a rate-limiting filter in `rest-java` (e.g., using Bucket4j, which is already a transitive dependency via the `web3` module) applied to all `/api/v1/**` endpoints, keyed by client IP or `X-Forwarded-For`.
2. **Response caching:** Add a short-lived cache (e.g., Caffeine or Redis) in `TopicServiceImpl.findById()` to absorb repeated lookups for the same ID and reduce DB pressure.
3. **Ingress-level rate limiting:** Enforce per-IP request rate limits at the Traefik/ingress layer (the Kubernetes chart already references Traefik middleware but does not configure rate limiting for the topics path). [8](#0-7) 

### Proof of Concept

```bash
# Enumerate all topics from ID 1 to 100000 with no authentication
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node-host>/api/v1/topics/$i" -o /dev/null &
done
wait
```

Each iteration triggers:
1. `TopicController.getTopic()` → `topicService.findById(id)` → `topicRepository.findById(id.getId())` (DB query on `topic` table)
2. `entityService.findById(id)` (DB query on `entity` table)
3. `customFeeService.findById(id)` (DB query on `custom_fee` table)

With no throttle, a single client can sustain thousands of concurrent DB queries, degrading the mirror node for all users.
