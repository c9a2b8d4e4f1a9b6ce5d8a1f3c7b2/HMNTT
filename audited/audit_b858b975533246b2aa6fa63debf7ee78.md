### Title
Unauthenticated Cache Poisoning of `/api/v1/accounts/<id>/rewards` Allows Temporary Suppression of Staking Reward Entries

### Summary
Any unauthenticated external user can issue a GET request to `/api/v1/accounts/<id>/rewards` during the importer's processing lag window, causing the response handler to cache an incomplete reward list in Redis. Because the cache key is derived solely from the request URL with no freshness or identity component, every subsequent caller — including the legitimate account owner — receives the stale, reward-missing response for the full cache TTL, hiding earned staking rewards from the API until the entry expires.

### Finding Description

**Cache key generation — no identity or freshness component**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, lines 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The key is a hash of `req.originalUrl` only. No caller identity, no importer-lag guard, no consensus-timestamp watermark.

**Cache population is unconditional on any 2xx response**

`responseCacheUpdateHandler()`, lines 90–119:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
  const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
  if (ttl > 0) {
    await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
  }
}
```
Any HTTP 200 response — regardless of who triggered it or whether the DB was fully up-to-date — is written to Redis for the full TTL.

**The rewards handler queries the DB at request time with no consistency check**

`rest/controllers/accountController.js`, `listStakingRewardsByAccountId()`, lines 170–185:
```js
const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(
  query.order, query.limit, query.conditions, query.params
);
```
`StakingRewardTransferService.getRewards()` issues a plain `SELECT … FROM staking_reward_transfer WHERE account_id = $1` (`rest/service/stakingRewardTransferService.js`, lines 11–16). There is no check against the importer's latest processed consensus timestamp, no "as-of" guard, and no cache invalidation hook tied to importer commits.

**Exploit flow**

1. A staking reward event is finalized on-chain and is in-flight through the importer pipeline — the `staking_reward_transfer` row does not yet exist in the DB.
2. The attacker (zero privileges required — the endpoint is public) sends `GET /api/v1/accounts/<victim_id>/rewards`.
3. `responseCacheCheckHandler` finds no cached entry → sets `res.locals[responseCacheKeyLabel]` and passes through.
4. `listStakingRewardsByAccountId` queries the DB, finds 0 (or N-1) reward rows, returns a 200 with an incomplete list.
5. `responseCacheUpdateHandler` stores this incomplete response in Redis under the URL-derived key for the full TTL (e.g., `max-age=10` seconds as configured for this path).
6. The importer writes the missing row seconds later, but the cache entry is already populated.
7. The legitimate account owner queries the same URL within the TTL window and receives the cached, reward-missing response. The correct data is invisible until the TTL expires.

**Why existing checks are insufficient**

- `responseCacheCheckHandler` has no authentication gate; it serves and populates the cache for any caller.
- `responseCacheUpdateHandler` checks only `isSuccess(statusCode)` and `ttl > 0` — both are trivially satisfied by a normal 200 response on an incomplete DB state.
- There is no importer-watermark comparison (e.g., "only cache if `consensus_timestamp` of the response ≥ last fully-processed block") anywhere in the middleware chain.

### Impact Explanation

The account owner's view of their earned staking rewards is suppressed for the duration of the cache TTL. While the on-chain HBAR balance is unaffected, the mirror-node API — the primary data source for wallets, explorers, and DeFi integrations — reports a lower reward history. Applications that compute pending or cumulative rewards from this endpoint will undercount. The impact is bounded by the TTL but is repeatable: the attacker can re-trigger the poisoning immediately after each TTL expiry, creating a sustained denial-of-correct-information condition.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no cryptographic material. The only precondition is timing the request to land during the importer's processing lag, which is a structural property of every mirror-node deployment (the importer always has a non-zero lag between on-chain finality and DB commit). The attacker can poll the endpoint continuously and the poisoning will naturally occur whenever a new reward event is in-flight. The attack is fully automatable and repeatable.

### Recommendation

1. **Bind the cache key to an importer-state watermark.** Include the latest fully-processed consensus timestamp (available from the mirror node's own DB) as a component of the cache key, or refuse to cache responses whose data horizon lags behind the known importer head by more than a configurable threshold.
2. **Add a "freshness" guard in `responseCacheUpdateHandler`.** Before calling `getCache().setSingle()`, verify that the response was produced against a DB state that is at least as current as the importer's last committed record file timestamp.
3. **Short-circuit cache population for time-sensitive financial endpoints.** For `/accounts/<id>/rewards` specifically, set `Cache-Control: no-store` or `max-age=0` so that `getCacheControlExpiryOrDefault` returns 0 and the `if (ttl > 0)` guard prevents caching entirely, forcing every request to hit the DB directly.
