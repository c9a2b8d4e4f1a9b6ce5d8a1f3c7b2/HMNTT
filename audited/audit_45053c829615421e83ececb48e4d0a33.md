### Title
GraphQL `findByEvmAddress` Leaks Entities with `deleted = NULL` Due to Incorrect SQL NULL Semantics

### Summary
The GraphQL `EntityRepository.findByEvmAddress()` uses the SQL predicate `deleted is not true`, which in PostgreSQL evaluates to `TRUE` when `deleted IS NULL`, causing entities with an unset deletion status to be returned as active. Every other repository in the codebase uses `deleted <> true`, which correctly excludes `NULL` values. Any unauthenticated user can trigger this by submitting a GraphQL query with a matching EVM address.

### Finding Description
**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 16–17:

```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

**Root cause — SQL three-valued logic:**
In PostgreSQL:
- `NULL IS NOT TRUE` → `TRUE` → row **is returned**
- `NULL <> TRUE` → `NULL` (unknown) → row **is not returned**

The predicate `deleted is not true` passes rows where `deleted` is `NULL`, treating them as non-deleted/active. The predicate `deleted <> true` (used everywhere else) correctly rejects them.

**Contrast with all other repositories:**

| Repository | Query predicate | Returns `deleted=NULL`? |
|---|---|---|
| `graphql` `EntityRepository` (line 16) | `deleted is not true` | **YES** |
| `rest-java` `EntityRepository` (line 16) | `deleted <> true` | No |
| `importer` `EntityRepository` (line 19) | `deleted <> true` | No |
| `rest` `entityService.js` (line 24) | `deleted <> true` | No |

The `rest-java` integration test (`EntityRepositoryTest.java`, lines 36–46) explicitly asserts that an entity with `deleted = null` must **not** be returned by `findByEvmAddress`. The equivalent GraphQL test (`EntityRepositoryTest.java`, lines 29–34) has no such assertion, confirming the gap was never caught.

**Exploit flow:**
1. An entity row exists in the `entity` table with a non-null `evm_address` and `deleted = NULL` (e.g., an entity ingested by the importer whose deletion state was never set, or a partially-initialized record).
2. An unauthenticated attacker sends a GraphQL POST to `/graphql/alpha`:
   ```graphql
   query { account(input: { evmAddress: "<target_evm_address>" }) { deleted entityId { num } ... } }
   ```
3. `EntityServiceImpl.getByEvmAddressAndType()` (line 40) calls `entityRepository.findByEvmAddress(evmAddressBytes)`.
4. The SQL `deleted is not true` predicate matches the `NULL` row and returns the entity.
5. The GraphQL response presents the entity as a valid, active account — with `deleted: null` in the payload — even though its lifecycle state was never confirmed.

**Why existing checks are insufficient:**
The only guard is the SQL predicate itself. There is no application-layer post-filter in `EntityServiceImpl` that re-checks the `deleted` field after retrieval. The `.filter(e -> e.getType() == type)` call (line 40) only checks entity type, not deletion status.

### Impact Explanation
The mirror node exports entity data that was never confirmed as active. Consumers of the GraphQL API (wallets, explorers, indexers) receive records for entities whose deletion status is unknown, potentially treating them as live accounts or contracts. This constitutes an incorrect record export — the stated scope of the finding — and can corrupt downstream state that relies on the mirror node as a source of truth.

### Likelihood Explanation
The GraphQL endpoint is unauthenticated and publicly reachable. No special privileges, tokens, or network access are required. The attacker only needs to know (or enumerate) an EVM address belonging to an entity with `deleted = NULL`. Such entities can arise naturally during normal importer operation when a `CryptoCreate` transaction is recorded before any explicit delete/update sets the flag. The attack is trivially repeatable with a single HTTP request.

### Recommendation
Change the predicate in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java` to match every other repository in the codebase:

```java
// Before (incorrect — NULL passes the filter)
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)

// After (correct — NULL is excluded)
@Query(value = "select * from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
```

Apply the same fix to `findByAlias` on line 13, which has the identical flaw. Add integration-test assertions mirroring those in `rest-java/src/test/java/org/hiero/mirror/restjava/repository/EntityRepositoryTest.java` lines 36–46 to prevent regression.

### Proof of Concept
**Precondition:** Insert a row into the `entity` table with a known `evm_address` and `deleted = NULL`.

```sql
INSERT INTO entity (id, evm_address, type, deleted, ...)
VALUES (99999, decode('aabbccdd...20bytes...', 'hex'), 'ACCOUNT', NULL, ...);
```

**Trigger:**
```bash
curl -X POST http://<mirror-node>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input:{evmAddress:\"aabbccdd...\"}) { deleted entityId { num } } }"}'
```

**Expected (correct) result:** `{"data":{"account":null}}`

**Actual (vulnerable) result:**
```json
{"data":{"account":{"deleted":null,"entityId":{"num":99999}}}}
```

The entity with `deleted = NULL` is returned as an active account, confirming the bypass.
