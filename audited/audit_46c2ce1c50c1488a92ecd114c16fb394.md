### Title
Global Rate Limit Token Consumed Without Rollback When Per-Filter THROTTLE Bucket Rejects

### Summary
In `ThrottleManagerImpl.throttle()`, the global `rateLimitBucket` token is consumed unconditionally before the per-filter `THROTTLE` bucket check is evaluated. When the per-filter bucket is exhausted and rejects the request, the already-consumed global token is never returned. An unprivileged attacker can exploit this to drain global rate limit capacity for all other users.

### Finding Description
**Exact code path:**

In `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()` (lines 37–48):

```java
if (!rateLimitBucket.tryConsume(1)) {          // line 38: global token consumed here
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(...)) {   // line 40: gas token consumed here
    throw new ThrottleException(...);
}
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);          // line 46: per-filter check happens AFTER
    }
}
```

In `action()` (lines 66–76):
```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {    // line 71: per-filter bucket exhausted
        throw new ThrottleException(...);        // line 72: thrown, but global token already gone
    }
}
```

**Root cause:** The global `rateLimitBucket` token is consumed at line 38 before the per-filter `THROTTLE` bucket is checked at line 71. When the per-filter bucket is exhausted and throws, there is no rollback path for the global token. The `restore()` method (lines 59–64) only handles `gasLimitBucket` tokens, not `rateLimitBucket` tokens. There is no compensating `rateLimitBucket.addTokens(1)` call anywhere in the exception path.

**Per-filter bucket capacity:** In `RequestProperties.createBucket()` (lines 63–69), the per-filter THROTTLE bucket capacity equals `rate` (range 0–100, default 100). This bucket refills at `rate` tokens/second — far below the global limit of 500 req/s.

**Exploit flow:**
1. Attacker identifies (or probes) a configured THROTTLE filter expression (e.g., a specific `DATA` prefix, `TO` address, or `BLOCK` value).
2. Attacker sends `rate` matching requests (e.g., 10 if `rate=10`) to exhaust the per-filter bucket.
3. Attacker continues sending matching requests at the global rate. Each request: consumes 1 global token (line 38), consumes gas tokens (line 40), then is rejected at line 72 — but the global token is permanently lost.
4. Legitimate users' requests are rejected at line 39 (`rateLimitBucket` exhausted) even though the THROTTLE filter was supposed to protect only a subset of traffic.

**Why existing checks are insufficient:** The `restore()` method is the only token-return mechanism, and it is scoped exclusively to `gasLimitBucket`. There is no equivalent for `rateLimitBucket`. The `ContractCallService.restoreGasToBucket()` call path also only restores gas, not rate limit tokens.

### Impact Explanation
An unprivileged attacker can consume the entire global rate limit budget (e.g., 500 tokens/s) using requests that are all ultimately rejected by the per-filter THROTTLE bucket. Legitimate users receive `ThrottleException("Requests per second rate limit exceeded")` even though the THROTTLE filter was intended to protect only a narrow request class. This is a targeted availability degradation: the attacker gets no useful responses (all rejected), but fully starves the global rate limit for all other users. Severity is medium-high for a public-facing JSON-RPC endpoint where availability is a core service guarantee.

### Likelihood Explanation
No privileges are required. The attacker only needs to know (or brute-force) a filter expression that matches a configured THROTTLE rule — filter fields (`DATA`, `TO`, `BLOCK`, `FROM`, `GAS`, `VALUE`) and match types (`EQUALS`, `CONTAINS`) are documented in `docs/configuration.md`. The attack is trivially repeatable with a simple HTTP client loop and is sustainable indefinitely since the per-filter bucket refills at `rate`/s while the attacker can re-exhaust it each second.

### Recommendation
After the per-filter THROTTLE bucket rejects a request, return the consumed global token:

```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {
        rateLimitBucket.addTokens(1);  // return the global token
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

Similarly, if the gas bucket check fails (line 40), the already-consumed `rateLimitBucket` token should be returned:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);  // return the global token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, restructure the check order so the per-filter THROTTLE bucket is evaluated before consuming the global token, or use a try/finally pattern to guarantee rollback.

### Proof of Concept
Precondition: A THROTTLE filter is configured, e.g.:
```yaml
hiero.mirror.web3.throttle.request[0].action: THROTTLE
hiero.mirror.web3.throttle.request[0].rate: 10
hiero.mirror.web3.throttle.request[0].filters[0].field: DATA
hiero.mirror.web3.throttle.request[0].filters[0].type: CONTAINS
hiero.mirror.web3.throttle.request[0].filters[0].expression: "0xdeadbeef"
hiero.mirror.web3.throttle.requestsPerSecond: 500
```

Steps:
1. Send 10 requests with `data: "0xdeadbeef"` — per-filter bucket exhausted, all pass globally.
2. Immediately send 490 more requests with `data: "0xdeadbeef"` within the same second.
3. Each of the 490 requests: consumes 1 global token at line 38, consumes gas at line 40, then throws at line 72.
4. Global `rateLimitBucket` is now at 0 tokens.
5. Any legitimate request (with any `data`) now receives `ThrottleException("Requests per second rate limit exceeded")` at line 39.
6. Repeat each second to maintain the denial of service. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
