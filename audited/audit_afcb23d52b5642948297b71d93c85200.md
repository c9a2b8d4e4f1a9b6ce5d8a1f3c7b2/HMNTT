### Title
Unbounded Subscription Fan-Out Enables Resource Exhaustion via Unauthenticated `subscribeTopic` (POLL Mode)

### Summary
`ConsensusController.subscribeTopic()` accepts connections from any unauthenticated caller with no global limit on the number of concurrent subscriptions. When the listener is configured in `POLL` mode, each call to `subscribeTopic()` causes `PollingTopicListener.listen()` to create a fully independent polling context and DB-query loop. An attacker opening N subscriptions to the same topic triggers N independent database queries every polling interval, multiplying backend load linearly with no deduplication or sharing.

### Finding Description

**Entry point â€” no authentication, no subscription cap:**

`ConsensusController.subscribeTopic()` (lines 43â€“53) accepts any inbound gRPC call, converts it to a `TopicMessageFilter`, and immediately delegates to `topicMessageService.subscribeTopic()`. There is no authentication check, no per-IP limit, and no global subscription counter check before the subscription is created. The only server-side guard is `maxConcurrentCallsPerConnection = 5` (default), which limits calls *per TCP connection* but places no bound on the number of connections an attacker may open.

**Independent polling loop per subscription in POLL mode:**

`PollingTopicListener.listen()` (lines 34â€“49) allocates a fresh `PollingContext` on every invocation and constructs a new `Flux.defer(() -> poll(context)).repeatWhen(...)` chain. There is no cache, no `computeIfAbsent`, and no `.share()` operator â€” every subscriber gets its own independent polling loop that issues a `topicMessageRepository.findByFilter(newFilter)` query to the database on every tick of the configured interval (default 500 ms).

**Contrast with shared listeners:**

`RedisTopicListener.getSharedListener()` (line 61) uses `computeIfAbsent` keyed by topic name and terminates with `.share()`, so all subscribers to the same topic share one upstream Redis subscription. `SharedPollingTopicListener` (line 52) similarly uses a single shared `topicMessages` Flux with `.share()`. `PollingTopicListener` has neither mechanism.

**Exploit flow:**

1. Attacker opens C TCP connections to the gRPC port (no auth required).
2. On each connection, attacker opens 5 `subscribeTopic` streams (the per-connection maximum) with identical `topicID` and `consensusStartTime`.
3. Total active subscriptions = 5 Ă— C.
4. With POLL mode active, the server now executes 5 Ă— C independent `findByFilter` DB queries every 500 ms for the same data.
5. Attacker scales C until DB connection pool is saturated or CPU is exhausted.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` is per-connection only; there is no limit on the number of connections.
- `subscriberCount` (an `AtomicLong` in `TopicMessageServiceImpl`) is a metric gauge only â€” it is never checked against a threshold to reject new subscriptions.
- There is no IP-based rate limiter, no token/auth requirement, and no global subscription cap anywhere in the gRPC stack.

### Impact Explanation
With POLL mode enabled, each additional subscription directly adds one DB query per polling interval. At 500 ms intervals, 1000 subscriptions (200 connections Ă— 5 calls each) produce 2000 DB queries per second for identical data. This exhausts the database connection pool, starves legitimate subscribers, and can render the mirror node's gRPC API completely unavailable â€” a denial-of-service with no economic cost to the attacker.

### Likelihood Explanation
The attack requires only a network connection to the gRPC port (default 5600) and the ability to send valid `ConsensusTopicQuery` protobuf messages â€” both trivially achievable by any external party. No credentials, keys, or privileged access are needed. The attack is fully repeatable and scriptable. POLL mode is a documented, supported configuration (`hiero.mirror.grpc.listener.type: POLL`).

### Recommendation
1. **Add a global concurrent-subscription limit** checked inside `TopicMessageServiceImpl.subscribeTopic()` before creating the subscription; reject with `RESOURCE_EXHAUSTED` when exceeded.
2. **Add a per-source-IP subscription limit** via a gRPC `ServerInterceptor` that tracks active calls per remote address.
3. **Deduplicate polling loops in `PollingTopicListener`** using a `ConcurrentHashMap` keyed by `(topicId, startTime)` with `.share()`, mirroring the pattern already used in `RedisTopicListener`.
4. **Require authentication** (e.g., mTLS or a bearer token) for long-lived streaming RPCs, or enforce connection-level rate limiting at the load-balancer/ingress layer.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hedera.mirror.api.proto import consensus_topic_query_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234          # any existing topic
CONNECTIONS = 200         # 200 TCP connections
CALLS_PER_CONN = 5        # maxConcurrentCallsPerConnection default

query = consensus_topic_query_pb2.ConsensusTopicQuery()
query.topicID.topicNum = TOPIC_NUM

channels = [grpc.insecure_channel(TARGET) for _ in range(CONNECTIONS)]
stubs   = [consensus_service_pb2_grpc.ConsensusServiceStub(ch) for ch in channels]

streams = []
for stub in stubs:
    for _ in range(CALLS_PER_CONN):
        # Each call creates an independent PollingContext + DB query loop
        streams.append(stub.subscribeTopic(query))

# 1000 independent DB queries now fire every 500 ms against the same topic
print(f"Active subscriptions: {len(streams)}")
input("Press Enter to cancel...")
for ch in channels: ch.close()
```

With POLL mode active, the database will receive `len(streams)` queries every `listenerProperties.interval` milliseconds, all fetching identical rows for `TOPIC_NUM`.
