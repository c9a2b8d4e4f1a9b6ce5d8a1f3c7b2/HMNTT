### Title
Information Disclosure via Distinct EntityNotFoundException Messages in Multi-Service Topic Lookup Chain

### Summary
`TopicController.getTopic()` calls three services sequentially, each throwing `EntityNotFoundException` with a unique message. `GenericControllerAdvice` forwards the raw exception message to the HTTP 404 response body for all non-5xx errors. An unprivileged attacker can probe any topic ID and determine the per-table existence state across `topic`, `entity`, and `custom_fee` tables by reading the error message.

### Finding Description
**Code path:**

`TopicController.getTopic()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 33–35:
```java
var topic = topicService.findById(id.id());         // throws "Topic not found"
var entity = entityService.findById(id.id());       // throws "Entity not found"
var customFee = customFeeService.findById(id.id()); // throws "Custom fee for entity not found"
```

Each service throws a distinct message:
- `TopicServiceImpl.java` line 20: `"Topic not found"`
- `EntityServiceImpl.java` line 26: `"Entity not found"`
- `CustomFeeServiceImpl.java` line 22: `"Custom fee for entity not found"`

`GenericControllerAdvice.java` lines 115–118 catches `EntityNotFoundException` and maps it to HTTP 404. Lines 150–151 then include the raw exception message in the response body for all non-5xx status codes:
```java
var detail = body instanceof ProblemDetail pb ? pb.getDetail() : ex.getMessage();
var nonSensitiveDetail = !statusCode.is5xxServerError() ? detail : StringUtils.EMPTY;
```

Since 404 is not 5xx, `nonSensitiveDetail = ex.getMessage()` — the distinct message is returned to the caller.

**Root cause:** The failed assumption is that all three "not found" cases are indistinguishable to the caller. They are not — the messages differ, and the advice layer does not normalize them.

### Impact Explanation
An unprivileged attacker can enumerate topic IDs and determine:
1. Whether a row exists in the `topic` table but not the `entity` table (or vice versa) — revealing database inconsistency/orphaned records.
2. Whether a topic+entity pair exists but has no `custom_fee` row — revealing fee configuration state.

This leaks internal database consistency state that is not intended to be observable externally. It could assist in targeted attacks, data harvesting, or identifying partially-ingested/corrupted state in the mirror node.

### Likelihood Explanation
No authentication or rate limiting is visible on `GET /api/v1/topics/{id}`. Any external user can send arbitrary numeric IDs and read the error message from the JSON response body. The attack is trivially repeatable with a simple HTTP client in a loop. No special privileges, tokens, or insider knowledge are required.

### Recommendation
Normalize all three `EntityNotFoundException` messages to a single generic message (e.g., `"Topic not found"`) before they reach the advice layer, or catch and re-throw in `TopicController.getTopic()` with a unified message. Alternatively, modify `GenericControllerAdvice.notFound()` to always return a fixed, non-discriminating message for `EntityNotFoundException` regardless of the original message:
```java
@ExceptionHandler
private ResponseEntity<Object> notFound(final EntityNotFoundException e, final WebRequest request) {
    // Do NOT forward e.getMessage() — use a fixed string
    var problem = ProblemDetail.forStatusAndDetail(NOT_FOUND, "Not found");
    return handleExceptionInternal(e, problem, null, NOT_FOUND, request);
}
```

### Proof of Concept
**Precondition:** A topic ID (e.g., `0.0.1234`) exists in the `topic` table but the corresponding row is absent from the `entity` table (possible during partial ingestion or data inconsistency).

**Step 1 — Probe a non-existent topic ID:**
```
GET /api/v1/topics/0.0.9999999
→ HTTP 404, body: {"_status":{"messages":[{"message":"Not Found","detail":"Topic not found"}]}}
```
Conclusion: no row in `topic` table.

**Step 2 — Probe the partially-ingested topic ID:**
```
GET /api/v1/topics/0.0.1234
→ HTTP 404, body: {"_status":{"messages":[{"message":"Not Found","detail":"Entity not found"}]}}
```
Conclusion: row exists in `topic` table but NOT in `entity` table — database inconsistency confirmed.

**Step 3 — Probe a topic+entity that exists but has no custom fee:**
```
GET /api/v1/topics/0.0.5678
→ HTTP 404, body: {"_status":{"messages":[{"message":"Not Found","detail":"Custom fee for entity not found"}]}}
```
Conclusion: both `topic` and `entity` rows exist; no `custom_fee` row.

The three distinct response bodies allow full reconstruction of per-table existence state with zero privileges.
