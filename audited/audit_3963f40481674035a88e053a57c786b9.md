### Title
Unauthenticated Concurrent `/health/readiness` Requests Cause DB Connection Exhaustion and CPU Amplification via Unbounded `postgres.New` and Recursive `checkNetworkStatus` Calls

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is publicly accessible with no rate limiting or concurrency control at the application level. Each request invokes `postgres.New(postgres.Config{DSN: ...})`, which opens a **new** `sql.DB` connection (outside the application's managed pool) on every call, and also invokes `checkNetworkStatus()`, which makes two recursive HTTP calls to the rosetta server itself (`/network/list`, `/network/status`), each of which queries the database. An attacker flooding this endpoint with concurrent requests exhausts PostgreSQL connection slots and amplifies DB and CPU load well above the 30% baseline threshold.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `NewHealthController()`, lines 37–50:

```go
readinessChecks := []health.Config{
    {
        Name:      "postgresql",
        Timeout:   time.Second * 10,
        SkipOnErr: false,
        Check:     postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
    },
    {
        Name:      "network",
        Timeout:   time.Second * 10,
        SkipOnErr: false,
        Check:     checkNetworkStatus(rosettaConfig.Port),
    },
}
```

**Root cause 1 — `postgres.New` opens a new connection per invocation:**
`github.com/hellofresh/health-go/v4/checks/postgres` v4.7.0 implements `New` as a closure that calls `sql.Open(dsn)` + `db.PingContext(ctx)` + `db.Close()` on **every** health check invocation. This is entirely separate from the application's GORM-managed pool configured in `rosetta/app/db/db.go` (lines 31–33). There is no shared or cached `*sql.DB` handle. Each concurrent request to `/health/readiness` opens a fresh TCP connection to PostgreSQL, authenticates, pings, and closes — bypassing `MaxOpenConnections` limits of the application pool entirely.

**Root cause 2 — `checkNetworkStatus` is a recursive amplifier:**
`checkNetworkStatus()` (lines 80–101) creates an HTTP client pointing to `http://localhost:<port>` and on each invocation calls:
1. `rosettaClient.NetworkAPI.NetworkList(ctx, ...)` → hits `/network/list` → queries the DB
2. `rosettaClient.NetworkAPI.NetworkStatus(ctx, ...)` → hits `/network/status` → queries the DB

Each `/health/readiness` request therefore triggers **2 additional inbound HTTP requests** to the same server, each of which performs DB queries through the application pool. With N concurrent health check requests, the server processes N + 2N = 3N effective DB-touching requests.

**Failed assumption:** The code assumes `/health/readiness` is called infrequently (e.g., by a Kubernetes kubelet probe at ~10s intervals). No concurrency guard, mutex, singleflight deduplication, or request coalescing is present.

**Existing checks reviewed and shown insufficient:**
The Helm chart at `charts/hedera-mirror-rosetta/values.yaml` defines Traefik middleware with `inFlightReq: amount: 5` and `rateLimit: average: 10`, but:
1. `global.middleware` defaults to `false` (line 95) — the middleware chain is **not deployed by default**.
2. Even when enabled, the ingress only covers paths `/rosetta/account`, `/rosetta/block`, etc. (lines 126–133) — `/health/readiness` is **not listed** and is exposed directly on the pod port (port 5700 per `healthCheckPolicy`, line 64).
3. The `rateLimit` is per `requestHost`, not per source IP, making it trivially bypassable with different `Host` headers.

### Impact Explanation
- **PostgreSQL connection exhaustion:** Each concurrent request opens a new connection. PostgreSQL's default `max_connections` is 100. With 100 concurrent requests (each holding a connection for up to 10 seconds due to the timeout), the DB becomes unavailable to all legitimate application queries.
- **CPU amplification:** Each health request triggers 2 recursive API calls, each performing DB queries. 50 concurrent health requests → 150 DB queries simultaneously, spiking both rosetta CPU and PostgreSQL CPU.
- **Cascading failure:** `SkipOnErr: false` means a failed DB check marks the service as not ready, potentially triggering Kubernetes to restart pods or remove them from load balancer rotation, amplifying the outage.
- **Severity: High** — unauthenticated, no prerequisites, directly impacts service availability.

### Likelihood Explanation
- **Precondition:** None. The endpoint is a plain HTTP GET with no authentication, API key, or session requirement.
- **Attacker capability:** Any external user with network access to the rosetta port (default 5700). A single machine with a basic HTTP flood tool (e.g., `ab`, `wrk`, `hey`) suffices.
- **Repeatability:** Fully repeatable. The 10-second timeout window means an attacker needs only ~10 requests/second sustained to keep the DB connection pool saturated.
- **Detection difficulty:** Health endpoint traffic is typically not rate-monitored or alerted on, making this attack blend into normal probe traffic.

### Recommendation
1. **Deduplicate concurrent health checks** using `golang.org/x/sync/singleflight` so that N concurrent `/health/readiness` requests result in exactly one DB ping and one `checkNetworkStatus` call, with all waiters receiving the same result.
2. **Reuse a shared `*sql.DB` handle** in the postgres health check instead of calling `sql.Open` per invocation. Pass the application's existing GORM `*sql.DB` to the health check closure.
3. **Add application-level rate limiting** on the health endpoint (e.g., `golang.org/x/time/rate` token bucket), independent of infrastructure middleware.
4. **Remove the recursive `checkNetworkStatus` self-call** or replace it with a lightweight in-process readiness flag (e.g., an atomic boolean set when the server is initialized).
5. **Ensure Traefik middleware covers health paths** if infrastructure-level protection is relied upon, and enable it by default.

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
```
