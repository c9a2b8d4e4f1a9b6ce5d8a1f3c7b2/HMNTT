### Title
`inFlightReq` Per-IP Limit Bypass via Forged `X-Forwarded-For` Header Due to `forwardedHeaders.insecure` + `ipStrategy.depth:1` Miscombination

### Summary
The Traefik middleware protecting the GraphQL `account` query is configured with `ipStrategy: depth: 1`, which derives the source IP for in-flight request limiting from the rightmost entry of the `X-Forwarded-For` header. Simultaneously, the shared Traefik deployment is globally configured with `--entryPoints.web.forwardedHeaders.insecure`, instructing Traefik to trust `X-Forwarded-For` headers from **any** source without restriction. An unprivileged attacker can therefore forge arbitrary IPs in the `X-Forwarded-For` header, rotating them across requests to obtain an unlimited number of independent 5-slot in-flight budgets and saturate the GraphQL server with concurrent expensive `account` queries.

### Finding Description

**Code locations:**

- `charts/hedera-mirror-common/values.yaml` lines 635–637: Traefik is started with `--entryPoints.web.forwardedHeaders.insecure` and `--entryPoints.websecure.forwardedHeaders.insecure`, meaning it unconditionally trusts `X-Forwarded-For` from every connecting client.
- `charts/hedera-mirror-graphql/values.yaml` lines 138–142: the `inFlightReq` middleware limits concurrent requests to `amount: 5` keyed on `sourceCriterion.ipStrategy.depth: 1`, i.e. the rightmost IP in the `X-Forwarded-For` list.
- `charts/hedera-mirror-graphql/templates/middleware.yaml` line 3: the middleware chain is rendered only when `global.middleware: true`; the default is `false`, but the ingress annotation at line 115 references the middleware by name, so it is intended to be enabled in production.
- `graphql/src/main/resources/graphql/query.graphqls` line 5: the sole exposed query is `account(input: AccountInput!)`, which resolves account data including recursive fields (`autoRenewAccount`, `stakedAccount`, `obtainer`) backed by database lookups.

**Root cause:** Traefik's `ipStrategy.depth` reads the `X-Forwarded-For` header **as supplied by the client** when `forwardedHeaders.insecure` is active. There is no trusted-proxy allowlist (`forwardedHeaders.trustedIPs`) configured anywhere. The failed assumption is that `depth: 1` will always reflect a network-verified IP (e.g. a CDN's egress IP); in reality, with `insecure` mode, the attacker fully controls the value Traefik uses as the rate-limit key.

**Exploit flow:**
1. Attacker sends `POST /graphql/alpha` with `X-Forwarded-For: 10.0.0.1` — occupies slot 1 of 5 for IP `10.0.0.1`.
2. Attacker sends another request with `X-Forwarded-For: 10.0.0.2` — opens a fresh 5-slot budget for IP `10.0.0.2`.
3. By cycling through N distinct fake IPs, the attacker holds `5 × N` concurrent in-flight requests simultaneously.
4. Each request resolves `account { autoRenewAccount { stakedAccount { ... } } }` — a recursive, DB-heavy query — maximising backend resource consumption.

**Why existing checks fail:**
- The `circuitBreaker` (lines 136–137) only trips on aggregate error ratios; it does not limit per-client concurrency.
- The `retry` middleware (lines 143–145) amplifies load (up to 3× per request) rather than reducing it.
- There is no `rateLimit` middleware in the GraphQL chart (unlike the Rosetta chart which adds one), no query depth/complexity limit in the GraphQL schema or Spring configuration, and no IP allowlist for `X-Forwarded-For` trust.

### Impact Explanation
An unprivileged external attacker can drive an unbounded number of concurrent expensive GraphQL `account` queries to the server. Each query triggers at minimum one database read; recursive field selection multiplies this. The result is sustained CPU, memory, and database connection exhaustion on the GraphQL pod, directly satisfying the >30% resource increase threshold. With `hpa.enabled: false` (the default), there is no automatic horizontal scaling to absorb the load.

### Likelihood Explanation
The attack requires only an HTTP client capable of setting custom headers — no credentials, tokens, or prior knowledge of account IDs (any valid or invalid `entityId` input triggers the resolver path). The technique is trivially scriptable. The only precondition is that `global.middleware: true` is set in the production deployment, which is the documented intent of the ingress annotation at line 115. The `forwardedHeaders.insecure` flag is present in the shared `hedera-mirror-common` chart and applies to all entry points.

### Recommendation
1. **Replace `forwardedHeaders.insecure` with `forwardedHeaders.trustedIPs`** in `charts/hedera-mirror-common/values.yaml` lines 635–637, listing only the known upstream load-balancer or CDN CIDR ranges. This ensures Traefik only trusts `X-Forwarded-For` from verified infrastructure.
2. **Change `ipStrategy.depth` to `0`** (or remove `ipStrategy` entirely) so that `inFlightReq` keys on the actual TCP remote address, which cannot be forged at the application layer.
3. **Add a `rateLimit` middleware** to the GraphQL chain (as already present in the Rosetta chart) to enforce a per-IP request rate in addition to concurrency.
4. **Enforce GraphQL query depth and complexity limits** at the Spring/DGS layer to cap the cost of any single request regardless of concurrency controls.

### Proof of Concept

```bash
# Rotate fake source IPs to obtain N independent 5-slot budgets
for i in $(seq 1 20); do
  curl -s -X POST https://<host>/graphql/alpha \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 192.0.2.${i}" \
    -d '{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}) { balance autoRenewAccount { balance stakedAccount { balance } } } }"}' &
done
wait
# 20 concurrent requests across 20 fake IPs, each within its own 5-slot budget,
# bypassing the inFlightReq amount:5 limit entirely.
```
