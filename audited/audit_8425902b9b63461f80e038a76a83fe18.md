### Title
Permanent Subscriber Hang via TCP Half-Open on Cached Redis Pub/Sub Connection in `RedisTopicListener.subscribe()`

### Summary
`RedisTopicListener` caches a single `ReactiveRedisMessageListenerContainer` instance permanently via `.cache()`. When the underlying Redis TCP connection enters a half-open state during a network partition, the `receive()` Flux silently stalls — emitting neither items nor an error signal — causing all active and future subscribers to that topic to hang indefinitely. Because the retry logic only activates on error signals and the broken container is never replaced, recovery is impossible without a full service restart.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

```
Lines 47-55 (container creation):
this.container = Mono.defer(() -> Mono.just(new ReactiveRedisMessageListenerContainer(connectionFactory)))
    ...
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(8)))
    .cache();   // <-- container cached forever after first success

Lines 68-79 (subscribe):
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))
    .doOnComplete(() -> unsubscribe(topic))
    .doOnError(t -> log.error("Error listening for messages", t))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .share();

Line 61 (shared Flux cached per topic):
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
```

**Root cause and failed assumptions:**

1. **`.cache()` on the container Mono** (line 55): Once the `ReactiveRedisMessageListenerContainer` is successfully created, it is cached permanently. The `retryWhen` on lines 54 only applies to the initial container creation — it does not re-create the container if the underlying connection later becomes broken.

2. **`retryWhen` only fires on error signals** (line 78): In a TCP half-open scenario, the OS-level TCP socket appears alive (no RST or FIN is sent by the remote side). The Lettuce driver's pub/sub connection has no socket-level timeout or keepalive configured — confirmed by the absence of any Lettuce socket options in `grpc/src/main/java/org/hiero/mirror/grpc/config/RedisConfiguration.java` and the absence of any `application.yml` in `grpc/src/main/resources/` (only `banner.txt` exists). The `receive()` Flux therefore neither emits items nor signals an error. `retryWhen` never fires.

3. **Shared Flux cached in `topicMessages`** (line 61): The hung `Flux` is stored in the `ConcurrentHashMap`. All current subscribers sharing it via `.share()` are permanently hung. New subscribers calling `getSharedListener()` receive the same hung Flux from the map. `doOnCancel`/`doOnComplete` never fire, so `unsubscribe()` never removes the entry.

**Exploit flow:**

- Network partition causes TCP half-open between gRPC service and Redis
- Lettuce's pub/sub connection (used by `ReactiveRedisMessageListenerContainer`) enters a state where the socket is open at the OS level but no data flows
- `r.receive(...)` returns a Flux that never emits and never errors
- All subscribers on the shared Flux block indefinitely on their Reactor scheduler threads
- New subscribers get the same hung Flux from `topicMessages`
- `retryWhen` never triggers (no error signal); container is never replaced (`.cache()`)
- Service is in a permanent DoS state for all topic subscriptions until restarted

**Why existing checks fail:**

- The outer `retryWhen(Retry.backoff(...))` (line 78) is error-driven; a silent hang produces no error signal
- The inner `retryWhen` on the container Mono (line 54) only covers initial connection, not post-connection failures
- `doOnError` (line 76) logs nothing because no error is ever raised
- `doOnCancel`/`doOnComplete` (lines 74-75) never fire, so the `topicMessages` map is never cleaned
- No socket timeout, TCP keepalive, or command timeout is configured for the Lettuce connection in the gRPC module

### Impact Explanation

All gRPC topic subscribers are permanently denied service. The `topicMessages` map accumulates stale, hung Flux entries for every topic that was active at the time of the partition. No new subscriber to any of those topics can receive messages. The gRPC service must be restarted to recover. This is a complete, service-wide Denial of Service for the topic subscription feature, which is the primary function of the gRPC API.

### Likelihood Explanation

No special privileges are required — any user who can open a gRPC subscription (the public API) is both a victim and an amplifier of the condition. The TCP half-open trigger is a realistic infrastructure event: cloud provider network blips, firewall rule changes, Redis pod restarts in Kubernetes with connection draining, or deliberate network-level interference. The condition is not self-healing and persists until service restart, making it high-impact even if the triggering event is transient. In containerized deployments (as shown in `docker-compose.yml` with `SPRING_DATA_REDIS_HOST: redis`), network partitions between service pods are a known operational risk.

### Recommendation

1. **Add Lettuce socket-level keepalive and timeout**: Configure `SocketOptions` with `keepAlive` enabled and a `connectTimeout`/`socketTimeout` so that half-open connections are detected at the TCP layer and raise an error that `retryWhen` can act on.

2. **Remove `.cache()` or make it invalidatable**: Replace the permanent `.cache()` with a mechanism that re-creates the container on detected failure (e.g., `Mono.defer` without cache, or cache with TTL/invalidation).

3. **Add a `timeout` operator on `receive()`**: Apply `.timeout(Duration)` to the `receive()` Flux so that a configurable period of silence (no messages and no error) causes a `TimeoutException`, which triggers `retryWhen` and forces reconnection.

4. **Add a heartbeat/liveness check**: Periodically issue a Redis PING on the pub/sub connection and treat failure or timeout as a connection error requiring retry.

### Proof of Concept

**Preconditions:**
- gRPC service running with `hiero.mirror.grpc.listener.type=REDIS` (the default)
- At least one active gRPC topic subscriber

**Steps:**

1. Start the gRPC service and Redis normally.
2. Open a gRPC `subscribeTopic` stream to any topic (e.g., topic `0.0.1234`). Confirm messages flow.
3. Simulate a TCP half-open between the gRPC service and Redis using `iptables` on the Redis host:
   ```bash
   # Drop all packets from gRPC service to Redis WITHOUT sending RST/FIN
   iptables -I INPUT -s <grpc-service-ip> -p tcp --dport 6379 -j DROP
   ```
   This causes the gRPC service's TCP socket to remain open (half-open) while Redis stops receiving data.
4. Observe that the gRPC subscriber stream receives no messages and no error — it hangs indefinitely.
5. Restore the network (`iptables -D INPUT ...`). Observe that the subscriber stream **still does not recover** — no messages flow even after Redis is reachable again.
6. Attempt a new gRPC subscription to the same topic. Observe it also hangs immediately (receives the cached hung Flux from `topicMessages`).
7. Confirm recovery requires a full service restart.
