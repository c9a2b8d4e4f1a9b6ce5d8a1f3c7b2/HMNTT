### Title
Unauthenticated Cache-Stampede DB Connection Exhaustion via `updateOperationAccountAlias`

### Summary
`updateOperationAccountAlias()` in `rosetta/app/services/block_service.go` uses an unprotected cache-aside pattern with no singleflight/stampede guard and no application-level rate limiting. Any unauthenticated caller can flood the `/block` or `/block/transaction` endpoints with concurrent requests referencing blocks whose operation `accountId` values are not yet cached, causing every goroutine to independently issue a `GetAccountAlias()` DB query, exhausting the bounded connection pool and rendering the Rosetta service unavailable.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, lines 115–122 (`updateOperationAccountAlias`):
```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB hit
    ...
    s.entityCache.Set(result.GetId(), result)
}
```

`rosetta/app/persistence/account.go`, lines 113–117 (`GetAccountAlias`):
```go
db, cancel := ar.dbClient.GetDbWithContext(ctx)   // acquires pool connection
defer cancel()
db.Raw(selectCryptoEntityWithAliasById, ...).First(&entity)
```

`rosetta/app/db/db.go`, line 33:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default 100
```

**Root cause — two compounding failures:**

1. **No stampede protection.** The `entityCache.Get` → `GetAccountAlias` → `entityCache.Set` sequence is not atomic and has no `singleflight` or mutex guard. N concurrent goroutines that all miss the cache for the same (or different) uncached `accountId` each independently call `GetAccountAlias`, each acquiring one DB connection from the pool.

2. **No application-level rate limiting.** The Rosetta Go server has no built-in throttle. The Traefik middleware (`inFlightReq: 5`, `rateLimit: average: 10`) is gated on `global.middleware: true`, which defaults to `false` in `charts/hedera-mirror-rosetta/values.yaml` line 95 (`global.middleware: false`). In the default deployment, the HTTP server accepts unlimited concurrent connections.

**Why the LRU cache does not save it:**

The LRU cache (`MaxSize: 524288`) only helps after an `accountId` has been seen once. Hedera mainnet has millions of distinct accounts. An attacker who continuously requests different blocks (each containing operations with previously-unseen `accountId` values) keeps the cache cold for those IDs. Additionally, the LRU eviction itself cycles out older entries, re-opening cache misses for previously-seen IDs under sustained load.

### Impact Explanation

The DB connection pool for Rosetta is capped at `maxOpenConnections: 100` (default). When all 100 connections are held by in-flight `GetAccountAlias` queries, Go's `database/sql` blocks new callers waiting for a free connection. Every subsequent request — including health checks and unrelated block queries — stalls until the pool drains. This produces a complete service outage for the Rosetta API node. Because Rosetta mirror nodes are stateless and horizontally scaled (HPA `minReplicas: 1`, `maxReplicas: 10`), the same attack can be replayed against each pod independently, taking down ≥30% of the processing fleet without any privileged access.

### Likelihood Explanation

The attack requires zero authentication. The attacker needs only:
- Network access to the Rosetta HTTP port (5700)
- Knowledge of block indices or hashes (publicly available from any Hedera explorer)
- A tool capable of issuing concurrent HTTP POST requests (e.g., `wrk`, `hey`, `curl` in parallel)

Hedera mainnet blocks routinely contain dozens of crypto-transfer operations referencing distinct accounts. An attacker can enumerate high-activity blocks and replay them concurrently. The attack is repeatable, requires no special knowledge, and is trivially scriptable.

### Recommendation

1. **Add singleflight deduplication** around the cache-miss path in `updateOperationAccountAlias` so that N concurrent misses for the same `accountId` result in exactly one `GetAccountAlias` DB call.
2. **Enable application-level rate limiting** inside the Go server (e.g., `golang.org/x/time/rate` or a semaphore on concurrent DB-touching requests) independent of the optional Traefik middleware.
3. **Set `global.middleware: true` as the default** in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` and `rateLimit` Traefik middlewares are active out of the box.
4. **Cap the number of operations processed per request** to bound the maximum number of DB calls a single request can generate.

### Proof of Concept

```bash
# 1. Identify a high-activity block with many unique account operations
BLOCK_INDEX=<high_tx_block>

# 2. Craft the POST body
BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":'$BLOCK_INDEX'}}'

# 3. Flood the endpoint with 200 concurrent connections (no auth required)
#    Each request hits updateOperationAccountAlias for every uncached accountId
hey -n 10000 -c 200 -m POST \
    -H "Content-Type: application/json" \
    -d "$BODY" \
    http://<rosetta-node>:5700/block

# 4. Observe: DB connection pool exhausted (all 100 slots held),
#    subsequent /block and /health/liveness requests time out,
#    Kubernetes liveness probe fails → pod restart → repeat across replicas.
```
