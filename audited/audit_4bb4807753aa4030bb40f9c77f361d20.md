### Title
Unauthenticated Unbounded gRPC Topic Subscription Causes Resource Exhaustion

### Summary
Any unauthenticated external client can call `subscribeTopic` via gRPC with `limit=0` (or simply omit the limit field, which defaults to 0 in protobuf3) and no `endTime`, causing the server to maintain a perpetually open, unlimited message stream per connection. There is no server-side cap on concurrent unlimited subscriptions, no authentication, and no connection-count enforcement, making this trivially repeatable by any attacker.

### Finding Description

**Entry point — `ConsensusController.toFilter()`** [1](#0-0) 

The controller blindly maps `query.getLimit()` (a raw protobuf `uint64`, default 0) directly into the filter with no upper-bound check or authentication gate.

**Filter semantics — `TopicMessageFilter.hasLimit()`** [2](#0-1) 

`hasLimit()` returns `false` when `limit == 0`. The `@Min(0)` constraint explicitly permits 0.

**Limit enforcement — `TopicMessageServiceImpl.subscribeTopic()`** [3](#0-2) 

The `.take(filter.getLimit())` operator is only applied when `filter.hasLimit()` is true. With `limit=0`, no `.take()` is ever applied, so the Flux is unbounded.

**No termination without `endTime`** [4](#0-3) 

`isComplete()` returns `false` unconditionally when `endTime` is null, so the stream never self-terminates.

**No connection-count limit** — `subscriberCount` is only a metrics gauge; it enforces nothing. [5](#0-4) 

**Exploit flow:**
1. Attacker (no credentials needed) opens N concurrent gRPC connections to the mirror node.
2. Each sends `ConsensusTopicQuery` with a valid `topicID`, `limit=0`, and no `consensusEndTime`.
3. Each connection enters `subscribeTopic` → no `.take()` applied → `incomingMessages` loops indefinitely via `topicListener.listen(newFilter)`.
4. Each connection holds a live Reactor pipeline, a `TopicContext` object, and a slot in the underlying listener's buffer (`maxBufferSize` up to 65536 entries).
5. Server resources (heap, thread scheduler slots, network sockets) grow linearly with N.

### Impact Explanation
Each persistent unlimited subscription consumes heap (Reactor pipeline state, `TopicContext`, message buffers up to 65536 entries per `ListenerProperties.maxBufferSize`), a bounded-elastic scheduler thread, and continuous network bandwidth proportional to topic message rate. With enough concurrent connections the JVM heap is exhausted or network bandwidth is saturated, causing a full denial of service for all legitimate subscribers. Severity: **High**.

### Likelihood Explanation
The gRPC endpoint is public-facing with no authentication layer visible in the codebase. The protobuf default for an omitted `limit` field is 0, meaning even a client that simply forgets to set a limit triggers the unbounded path. An attacker needs only a gRPC client library and a known topic ID (topic IDs are public on-chain). The attack is trivially scriptable and repeatable.

### Recommendation
1. **Enforce a server-side maximum concurrent subscriber count** — reject new subscriptions when the count exceeds a configurable threshold.
2. **Require `endTime` or a non-zero `limit`** for subscriptions, or impose a server-side default maximum limit (e.g., `Long.MAX_VALUE` is not a safe default; cap at a configurable value).
3. **Add a per-IP or per-connection rate limit** at the gRPC interceptor layer.
4. In `ConsensusController.toFilter()`, validate that `query.getLimit() <= MAX_ALLOWED_LIMIT` before building the filter.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hederahashgraph.api.proto.java import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# limit field omitted → defaults to 0 → unbounded stream
# endTime omitted → stream never self-terminates
query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234)
    # limit=0 (default), no consensusEndTime
)

# Open N concurrent connections
import threading
def subscribe():
    for _ in stub.subscribeTopic(query):
        pass  # consume forever

threads = [threading.Thread(target=subscribe) for _ in range(500)]
for t in threads: t.start()
# Server now maintains 500 unbounded streaming pipelines
```
