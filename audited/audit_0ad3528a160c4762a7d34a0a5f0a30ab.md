### Title
Unauthenticated Future-Timestamp DoS via Open-Ended `gte` Filter in `/api/v1/blocks`

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts a `timestamp=gte:X` filter where `X` can be any syntactically valid timestamp, including far-future values. The filter is passed directly to `RecordFileService.getBlocks()` as a raw SQL `WHERE consensus_end >= X` predicate with no upper-bound requirement, no future-timestamp guard, and no call to `parseTimestampFilters()`. An unauthenticated attacker can flood the endpoint with future timestamps, causing repeated empty index scans on `consensus_end` that consume DB connections and degrade service.

### Finding Description

**Exact code path:**

`getBlocks()` in `blockController.js` (lines 101–112) calls `utils.buildAndValidateFilters()` then `extractSqlFromBlockFilters()`, then `RecordFileService.getBlocks()`:

```
getBlocks (blockController.js:101)
  → buildAndValidateFilters (utils.js:1208)   ← only format-validates timestamp
  → extractSqlFromBlockFilters (blockController.js:63)  ← no parseTimestampFilters call
  → RecordFileService.getBlocks (recordFileService.js:149)  ← executes raw SQL
```

**Root cause 1 — format-only timestamp validation:**

`isValidTimestampParam` (utils.js:151–154) only checks that the value matches `^\d{1,10}$` or `^\d{1,10}\.\d{1,9}$`. The value `9999999999` (year 2286) passes this check. No wall-clock upper bound is enforced. [1](#0-0) 

`filterValidityChecks` for `TIMESTAMP` delegates only to `isValidTimestampParam`: [2](#0-1) 

**Root cause 2 — `extractSqlFromBlockFilters` never calls `parseTimestampFilters`:**

`extractSqlFromBlockFilters` (blockController.js:63–88) maps each filter directly to a SQL condition via `getFilterWhereCondition`. It does **not** call `parseTimestampFilters()`, which is the utility that enforces both-bound requirements and the 7-day max-range cap. A single `gte` filter is accepted with no corresponding `lte`/`lt` bound. [3](#0-2) 

**Root cause 3 — query executed verbatim:**

`RecordFileService.getBlocks()` (recordFileService.js:149–162) builds and executes `SELECT … FROM record_file WHERE consensus_end >= $1 ORDER BY consensus_end DESC LIMIT 25` with the attacker-supplied value. When `$1` is a far-future nanosecond timestamp, the DB performs a full index scan on `consensus_end` and returns zero rows. [4](#0-3) [5](#0-4) 

**Why existing checks fail:**

`parseTimestampFilters` (utils.js:1583–1681) enforces both-bound requirements and the `maxTimestampRange` (default 7d) cap, but it is **never invoked** in the `getBlocks` code path. The `balances.js` endpoint has an analogous `getOptimizedTimestampRange()` that caps the upper bound to `nowInNs()`, but no equivalent exists here. [6](#0-5) [7](#0-6) 

### Impact Explanation
Each request with a future `gte` timestamp acquires a DB connection from the fixed-size pool (`config.db.pool.maxConnections`), executes an index scan that returns no rows, then releases the connection. Under sustained flood, the pool is saturated, legitimate requests queue or time out, and DB CPU is wasted on fruitless scans. The endpoint returns HTTP 200 with `blocks: []`, so no error-rate alerting fires. Severity: **High** (availability impact, no authentication required). [8](#0-7) 

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and a single HTTP GET. The payload is trivial (`?timestamp=gte:9999999999`). It is repeatable at arbitrary rate from a single client or botnet. No CAPTCHA, API key, or rate-limit middleware is present in the REST layer for this endpoint. Likelihood: **High**.

### Recommendation
1. **Reject future timestamps**: In `filterValidityChecks` (or a dedicated block-filter validator), reject any `gte`/`gt` timestamp value greater than `nowInNs()` plus a small grace window.
2. **Require an upper bound or cap it**: Either require a paired `lte`/`lt` filter (as `parseTimestampFilters` does for other endpoints), or automatically cap the upper bound to `NOW` when only a lower bound is supplied — mirroring the `getOptimizedTimestampRange` pattern in `balances.js`.
3. **Call `parseTimestampFilters`**: Invoke `parseTimestampFilters` inside `extractSqlFromBlockFilters` with `allowOpenRange: false` to enforce the existing 7-day range cap.
4. **Add rate limiting** at the API gateway or middleware layer for unauthenticated endpoints.

### Proof of Concept

```bash
# Single request — returns HTTP 200, blocks: [], triggers empty index scan
curl "https://<mirror-node>/api/v1/blocks?timestamp=gte:9999999999"

# Flood — saturates DB connection pool
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/blocks?timestamp=gte:9999999999" &
done
wait
```

Preconditions: none (unauthenticated, no special headers).
Trigger: `timestamp=gte:9999999999` passes `isValidTimestampParam` (10-digit integer), bypasses `parseTimestampFilters`, reaches DB as `WHERE consensus_end >= 9999999999000000000`.
Result: DB executes index scan on `consensus_end`, finds zero rows, returns empty result. Repeated at scale → connection pool exhaustion → legitimate requests time out.
