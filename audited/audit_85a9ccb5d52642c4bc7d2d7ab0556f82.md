### Title
Timestamp Parameter Order Reversal Causes Historical Custom Fee Lookup to Return 404

### Summary
The `normalizeRequestQueryParams()` function in `rest/middleware/requestNormalizer.js` intentionally preserves the insertion order of `timestamp` parameters (via `NON_SORTED_PARAMS`) to avoid breaking the historical custom fee SQL query. However, this same preservation means a user who sends timestamp parameters in reversed order bypasses no validation and causes `extractSqlFromTokenInfoRequest()` in `rest/tokens.js` to use the wrong (last) filter, returning a 404 instead of valid fee schedule data. The behavior is explicitly acknowledged in a code comment but left unguarded.

### Finding Description

**Code locations:**

`rest/middleware/requestNormalizer.js` lines 13–23 — `TIMESTAMP` is added to `NON_SORTED_PARAMS`, so `getNormalizedArrayValue()` (line 70) skips sorting for timestamp arrays, preserving whatever order the client supplied.

`rest/tokens.js` lines 488–493 — `extractSqlFromTokenInfoRequest()` unconditionally picks `filters[filters.length - 1]` as the single filter used to build the SQL condition:

```js
// honor the last timestamp filter
const filter = filters[filters.length - 1];
const op = transformTimestampFilterOp(filter.operator);
const conditions = [`${CustomFee.ENTITY_ID} = $1`, `lower(${CustomFee.TIMESTAMP_RANGE}) ${op} $2`];
params.push(filter.value);
```

`rest/tokens.js` lines 548–558 — `getTokenInfoRequest()` calls `buildAndValidateFilters` (which validates each filter individually but not their relative order), then passes the resulting array directly to `extractSqlFromTokenInfoRequest`. If the SQL returns no row or `custom_fee` is null, a `NotFoundError` (HTTP 404) is thrown.

**Root cause:** The design assumes the caller always places the bounding `lt`/`lte` filter first and the point-in-time `eq` filter last (e.g., `?timestamp=lt:1234567899.999999000&timestamp=1234567899.999999001`). The last filter (`eq` → transformed to `lte`) then correctly anchors the history query. When the order is reversed, the last filter becomes the `lt` filter, and the SQL condition `lower(timestamp_range) < 1234567899999999000` excludes the fee schedule whose `lower(timestamp_range)` equals `1234567899999999001`.

The code comment at line 19–21 of `requestNormalizer.js` explicitly documents this asymmetry:

```
/api/v1/tokens/1135?timestamp=lt:1234567899.999999000&timestamp=1234567899.999999001 returns a result whereas
?timestamp=1234567899.999999001&timestamp=lt:1234567899.999999000 returns a 404
```

**Exploit flow:**

1. Attacker identifies a token with a known fee schedule at timestamp `T` (e.g., `1234567899.999999001`).
2. Attacker sends: `GET /api/v1/tokens/1135?timestamp=1234567899.999999001&timestamp=lt:1234567899.999999000`
3. Express parses `req.query.timestamp` as `['1234567899.999999001', 'lt:1234567899.999999000']`.
4. `buildFilters` produces `[{op: eq, value: 1234567899999999001}, {op: lt, value: 1234567899999999000}]`.
5. `extractSqlFromTokenInfoRequest` picks the last filter: `lt`, value `1234567899999999000`.
6. SQL condition: `lower(timestamp_range) < 1234567899999999000` — excludes the fee schedule at `1234567899999999001`.
7. `rows[0].custom_fee` is null → `NotFoundError` → HTTP 404.

**Why existing checks fail:**

- `validateTokenInfoFilter` (lines 451–463) validates each filter in isolation (`eq|lt|lte` operator, valid timestamp format). It does not check semantic consistency between multiple timestamp filters or their relative order.
- `NON_SORTED_PARAMS` was added specifically to stop the normalizer from reordering timestamps, but it equally preserves an attacker-supplied reversed order.
- No guard exists in `extractSqlFromTokenInfoRequest` to detect that the last filter is a stricter bound than an earlier `eq` filter.
