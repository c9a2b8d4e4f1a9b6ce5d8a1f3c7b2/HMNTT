### Title
Unauthenticated Pagination-Driven DB Connection Pool Exhaustion via `getAccountTokenAllowances()`

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` accepts unauthenticated, unlimited-rate requests and issues a DB query per request. The REST API's default DB connection pool is only 10 connections, and no application-level rate limiting exists for this endpoint. An attacker can send concurrent paginated requests (using `links.next` or independently) to exhaust the pool, starving all other REST API endpoints — including those serving gossip transaction data — of DB connections.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:18` registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/controllers/tokenAllowanceController.js:68-81` — `getAccountTokenAllowances()` issues a DB query on every call with no throttle check: [2](#0-1) 

`rest/service/tokenAllowanceService.js:86-90` — each call executes a SQL query (potentially a 3-branch UNION ALL) against the shared pool: [3](#0-2) 

`rest/dbpool.js:14` — the pool `max` is set from config, defaulting to **10 connections**: [4](#0-3) 

`docs/configuration.md` confirms the defaults: `maxConnections = 10`, `statementTimeout = 20000 ms`, `connectionTimeout = 20000 ms`: [5](#0-4) 

**Middleware audit** — `rest/middleware/index.js` exports no rate-limiting middleware: [6](#0-5) 

`rest/middleware/requestHandler.js` only parses/logs requests; no throttle logic: [7](#0-6) 

The throttle/rate-limit code that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the **web3** module and is not applied to the REST API: [8](#0-7) 

**Root cause:** The REST API application layer has no per-IP or global request rate limiting. The DB connection pool is tiny (10 by default). Each request to `getAccountTokenAllowances` holds a connection for up to 20 seconds (statement timeout). Ten concurrent attacker requests saturate the pool; all subsequent requests — including `/api/v1/transactions` — queue and time out.

**Why the `links.next` mechanism matters:** Each paginated response returns a `links.next` URL with cursor parameters (`spender.id`, `token.id`): [9](#0-8) 

With `limit=1`, an account with N allowances requires N sequential requests. An attacker can pipeline these across multiple connections, keeping the pool permanently saturated.

### Impact Explanation

With the pool exhausted, `pool.queryQuietly()` calls from all REST API handlers block until `connectionTimeoutMillis` (20 s) elapses, then throw. This makes `/api/v1/transactions`, `/api/v1/accounts`, and all other endpoints that serve gossip-derived transaction data return errors. The REST API becomes effectively unavailable for the duration of the attack. Severity: **High** (complete REST API DoS, no authentication required, trivially repeatable).

### Likelihood Explanation

No privileges, API keys, or special knowledge are required. The attacker needs only a valid account ID (publicly enumerable) and the ability to send HTTP requests. The attack is fully scriptable: fetch page 1, extract `links.next`, fire 10 concurrent requests using those cursor URLs. The default pool size of 10 makes saturation trivial even from a single host. The attack is repeatable indefinitely.

### Recommendation

1. **Add application-level rate limiting** to the REST API (e.g., express-rate-limit keyed by IP) for all `/api/v1/accounts/*/allowances/*` routes, analogous to the Bucket4j throttle used in the web3 module.
2. **Increase the default pool size** or add a per-endpoint concurrency cap so a single endpoint cannot monopolize all connections.
3. **Add Traefik `inFlightReq` and `rateLimit` middleware** to the REST API Helm chart, mirroring what is already configured for the Rosetta API: [10](#0-9) 
4. Consider enforcing a minimum `limit` floor or a maximum page-iteration count per session to bound total DB work per client.

### Proof of Concept

```bash
# Step 1: Get first page with limit=1 (no auth needed)
NEXT="/api/v1/accounts/0.0.1001/allowances/tokens?limit=1"
BASE="https://<mirror-node-host>"

# Step 2: Saturate the pool with 10 concurrent paginated requests
for i in $(seq 1 10); do
  curl -s "$BASE$NEXT" &
done
wait

# Step 3: While pool is saturated, attempt to fetch transaction data
curl -v "$BASE/api/v1/transactions"
# Expected: connection timeout / 503 after 20 seconds

# Step 4: Repeat step 2 in a loop to maintain saturation
while true; do
  for i in $(seq 1 10); do
    curl -s "$BASE$NEXT" &
  done
  wait
done
```

With 10 concurrent requests each holding a DB connection for up to 20 seconds, the pool (`max: 10`) is fully occupied. All other REST API queries — including those serving gossip transaction data — fail with connection timeout errors until the attack stops.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
