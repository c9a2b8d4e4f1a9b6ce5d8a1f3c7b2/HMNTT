### Title
Thundering Herd on `CACHE_MANAGER_RECORD_FILE_LATEST` Amplifies Database Load During Recovery, Extending Partition Duration

### Summary
The `cacheManagerRecordFileLatest()` bean configures a Caffeine cache with a 500ms `expireAfterWrite` TTL, no `refreshAfterWrite`, and no negative-result caching for exceptions. Spring's `@Cacheable` is non-atomic (get → miss → call DB → put), so all concurrent threads that observe a cache miss simultaneously issue independent `SELECT * FROM record_file ORDER BY consensus_end DESC LIMIT 1` queries. During database recovery after a network partition, the cache never repopulates (exceptions are not cached due to `unless = "#result == null"`), causing every 500ms burst of up to 500 concurrent requests (the configured RPS limit) to hammer the still-recovering database, potentially overwhelming it and extending the outage beyond design parameters.

### Finding Description

**Exact code path:**

`EvmConfiguration.java` lines 178–188 — the cache is built with `expireAfterWrite(500, MILLISECONDS)` and `maximumSize(1)`, but critically without `refreshAfterWrite` (which would serve stale data while refreshing in the background) and without any negative-result/exception caching:

```java
// EvmConfiguration.java:180-183
final var caffeine = Caffeine.newBuilder()
        .expireAfterWrite(500, TimeUnit.MILLISECONDS)
        .maximumSize(1)
        .recordStats();
```

`RecordFileRepository.java` lines 31–36 — the `@Cacheable` annotation uses `unless = "#result == null"`, which only suppresses caching of `null` return values; it does **not** suppress caching of thrown exceptions, and more importantly, when the DB throws an exception the cache entry is never written at all:

```java
@Cacheable(
        cacheNames = CACHE_NAME_RECORD_FILE_LATEST,
        cacheManager = CACHE_MANAGER_RECORD_FILE_LATEST,
        unless = "#result == null")
@Query(value = "select * from record_file order by consensus_end desc limit 1", nativeQuery = true)
Optional<RecordFile> findLatest();
```

**Root cause and failed assumption:**

Spring's `@Cacheable` interceptor is not atomic. It calls `cache.getIfPresent(key)`, and if the result is absent, calls the underlying method, then calls `cache.put(key, value)`. There is no lock or single-flight mechanism preventing N concurrent threads from all observing a cache miss and all issuing the DB query simultaneously. The design assumes the DB is always available to repopulate the cache within 500ms; this assumption fails during partition recovery.

**Exploit flow:**

1. Attacker sustains traffic at or near the 500 RPS rate limit (default `hiero.mirror.web3.throttle.requestsPerSecond = 500`) against the public `/api/v1/contracts/call` endpoint — no authentication required.
2. A network partition isolates the database.
3. The cache entry (single key, `maximumSize(1)`) expires after 500ms.
4. All ~250 in-flight requests that arrive in the next 500ms window simultaneously observe a cache miss.
5. All 250 threads call `findLatest()` → each acquires a HikariCP connection and issues `SELECT * FROM record_file ORDER BY consensus_end DESC LIMIT 1`.
6. The DB is still recovering; queries either hang (up to `statementTimeout = 3000ms`) or fail with a connection error.
7. Because the DB throws an exception (not returns `null`), `unless = "#result == null"` does not apply — the cache is never populated.
8. Every 500ms this repeats: another burst of up to 250 simultaneous DB queries.
9. The flood of connection attempts and queries prevents the DB from completing its recovery, extending the partition duration in a feedback loop.

**Why existing checks are insufficient:**

- The web3 Traefik middleware (`charts/hedera-mirror-web3/values.yaml` lines 157–160) only configures `retry` (3 attempts, 100ms interval) — **no `circuitBreaker`** middleware is present. Compare to `hedera-mirror-graphql` and `hedera-mirror-rest-java` which both configure `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` circuit breakers.
- No application-level circuit breaker (Resilience4j or equivalent) wraps `findLatest()`.
- The rate limiter (500 RPS) is a global throughput cap, not a DB-protection mechanism; it actually defines the upper bound of the thundering herd size.
- `statementTimeout = 3000ms` limits per-query duration but does not prevent the burst; 250 connections held for 3 seconds each saturates a typical HikariCP pool.

### Impact Explanation

During any database recovery event (planned failover, unplanned partition, replica promotion), the web3 API becomes a force multiplier against the recovering database. Each 500ms window produces a synchronized burst of DB connections proportional to the sustained request rate. This can:
- Exhaust the HikariCP connection pool, causing all web3 requests to fail with connection-acquisition timeouts.
- Prevent the database from completing crash recovery or WAL replay by keeping it under constant query load.
- Extend a partition that would otherwise self-heal in seconds into a minutes-long outage.

Severity: **High**. The impact is availability loss of both the web3 API and the shared database, affecting all mirror node components that share the same PostgreSQL instance.

### Likelihood Explanation

Any unprivileged user with network access to the public `/api/v1/contracts/call` endpoint can trigger this. No credentials, special permissions, or insider knowledge are required. The attacker only needs to sustain normal-looking traffic (e.g., repeated `eth_blockNumber`-equivalent calls that resolve to `findLatest()`) during a DB recovery window. The attack is repeatable and can be scripted trivially. Database partitions occur in production environments due to infrastructure maintenance, cloud provider incidents, and network instability — the attacker does not need to cause the partition, only to be present when it occurs.

### Recommendation

1. **Add `refreshAfterWrite`** to the Caffeine builder so stale data is served while a single background thread refreshes the cache, eliminating the thundering herd entirely:
   ```java
   Caffeine.newBuilder()
       .expireAfterWrite(2000, TimeUnit.MILLISECONDS)
       .refreshAfterWrite(500, TimeUnit.MILLISECONDS)
       .maximumSize(1)
       .recordStats();
   ```
   This requires switching to a `LoadingCache` (use `CaffeineCacheManager.setCaffeine(caffeine)` with a `CacheLoader`).

2. **Cache negative results / exceptions** by replacing `unless = "#result == null"` with a short-TTL fallback value, or wrapping `findLatest()` with a Resilience4j `@CircuitBreaker` that returns a cached stale value when the DB is unavailable.

3. **Add a Traefik `circuitBreaker` middleware** to the web3 chart (matching the pattern already used by graphql and rest-java) to shed load at the ingress layer before it reaches the application.

4. **Limit HikariCP `maximumPoolSize`** for the web3 datasource to a value that the database can sustain under burst conditions, and set `connectionTimeout` low enough to fail fast rather than queue.
