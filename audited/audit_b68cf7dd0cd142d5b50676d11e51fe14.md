### Title
GraphQL Field Aliasing Allows Unauthenticated Request Amplification Leading to Database DoS

### Summary
The `account` query root in the GraphQL API accepts no authentication and has no alias-count limit. An unauthenticated attacker can pack ~52 aliased `account` invocations into a single HTTP request, each triggering an independent database query. The existing `MaxQueryComplexityInstrumentation(200)` does not prevent this because the default complexity calculator assigns only 1 point per field, and the binding constraint (`maxTokens=1000`) still permits ~52 aliases per request, each causing a separate synchronous DB round-trip.

### Finding Description

**Exact code path:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 declares the only root query with no authentication directive:
```graphql
account(input: AccountInput!): Account
``` [1](#0-0) 

`GraphQlConfiguration.java` lines 43–45 install the only query-level guards — a complexity cap of 200 and a depth cap of 10, using the **default** `FieldComplexityCalculator` (1 point per field, no alias multiplier): [2](#0-1) 

The GraphQL parser is capped at `maxTokens=1000` and `maxCharacters=10000`: [3](#0-2) 

Each alias resolves through `AccountController.account()`: [4](#0-3) 

Which calls `EntityServiceImpl`, which issues a synchronous `EntityRepository` query per alias: [5](#0-4) [6](#0-5) 

**Root cause and failed assumption:**

The complexity instrumentation assumes that a complexity ceiling of 200 is a sufficient DoS guard. However, the default calculator assigns complexity 1 to every field regardless of how many times the same root field is aliased. A minimal alias `a1: account(input:{entityId:{num:1}}){balance}` costs only 2 complexity points (1 for `account` + 1 for `balance`). The binding constraint is actually `maxTokens=1000`: each such alias consumes ~19 GraphQL tokens, allowing approximately **52 aliases per request** (52 × 2 = 104 complexity, well under 200). Each alias is executed as a fully independent resolver invocation with its own DB query — GraphQL-Java does not deduplicate aliased root fields.

**Exploit flow:**

1. Attacker crafts a single POST body to `/graphql/alpha` containing 52 aliased `account` queries with distinct inputs.
2. The GraphQL parser accepts it (104 complexity < 200, ~988 tokens < 1000, ~2500 chars < 10000).
3. GraphQL-Java dispatches 52 independent calls to `AccountController.account()`.
4. Each call issues a synchronous SQL query against the `entity` table via `EntityRepository`.
5. 52 DB connections are consumed for the duration of the request.
6. The Traefik `inFlightReq` middleware allows 5 concurrent HTTP requests per IP: [7](#0-6) 
   — so a single IP can sustain 5 × 52 = **260 simultaneous DB queries**.
7. With multiple IPs or a botnet, the DB connection pool is exhausted, blocking all other queries including those serving transaction gossip monitoring.

### Impact Explanation

The `entity` table is the core data store for all account lookups. Exhausting the DB connection pool (monitored via `GraphQLHighDBConnections` alert at 75% utilization) causes all GraphQL and potentially other mirror-node services to stall. The `statementTimeout` of 10 seconds (per README) means each alias can hold a connection for up to 10 s, making sustained exhaustion easy. This directly degrades the mirror node's ability to serve transaction gossip state queries, which is the stated high-severity scope of this finding. [8](#0-7) 

### Likelihood Explanation

No authentication, API key, or per-IP request-rate limit is required. The endpoint is publicly reachable at `/graphql/alpha`. The attack requires only a standard HTTP client and knowledge of the GraphQL schema (which is self-describing via introspection). It is trivially repeatable and scriptable. A single attacker with one IP can sustain 260 concurrent DB queries; a modest botnet scales this linearly.

### Recommendation

1. **Add an alias-count limit**: Implement a custom `GraphQL` `Instrumentation` or a `PreparsedDocumentProvider` pre-check that counts the number of root-level field selections (including aliases) and rejects queries exceeding a small threshold (e.g., 5).
2. **Raise the per-field complexity weight for root resolvers**: Register a custom `FieldComplexityCalculator` that assigns a higher cost (e.g., 50) to `account` root fields, so even 4 aliases exhaust the 200-point budget.
3. **Add per-IP request-rate limiting at the GraphQL layer**: The existing `inFlightReq: 5` limits concurrency but not request rate. Add a token-bucket rate limiter (analogous to the `web3` module's `ThrottleConfiguration`) for the `/graphql/alpha` endpoint.
4. **Lower `maxTokens`**: Reducing `maxTokens` from 1000 to ~100 would limit aliases to ~5 per request without breaking legitimate single-query use cases.

### Proof of Concept

```bash
# Build a query with 52 aliased account lookups (each with a distinct num)
ALIASES=""
for i in $(seq 1 52); do
  ALIASES="${ALIASES} a${i}: account(input:{entityId:{num:${i}}}){balance}"
done

curl -s -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ ${ALIASES} }\"}"
```

Expected result: the server accepts the request (complexity 104 < 200, tokens ~988 < 1000), dispatches 52 independent DB queries, and returns 52 aliased results. Running 5 such requests concurrently from a single IP saturates 260 DB connections simultaneously. No credentials are required.

    Optional<Account> account(@Argument @Valid AccountInput input) {
        final var alias = input.getAlias();
        final var evmAddress = input.getEvmAddress();
        final var entityId = input.getEntityId();
        final var id = input.getId();

        validateOneOf(alias, entityId, evmAddress, id);

        if (entityId != null) {
            return entityService
                    .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        if (alias != null) {
            return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
        }

        if (evmAddress != null) {
            return entityService
                    .getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        throw new IllegalStateException("Not implemented");
    }

    public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
        return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
        return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
