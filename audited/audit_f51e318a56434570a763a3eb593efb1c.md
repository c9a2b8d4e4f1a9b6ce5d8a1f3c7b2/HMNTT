### Title
Unbounded Per-Subscription Thread Exhaustion in PollingTopicListener via Unauthenticated gRPC Subscriptions

### Summary
`PollingTopicListener.listen()` creates a new `PollingContext` and schedules an independent, infinite polling loop on the JVM-wide `Schedulers.boundedElastic()` thread pool for every subscription, with no global cap on concurrent subscriptions. An unauthenticated attacker opening many gRPC connections (each with up to 5 concurrent calls per the `maxConcurrentCallsPerConnection` default) can flood the bounded elastic thread pool with blocking database-query tasks, starving all other reactor pipelines that share the same scheduler.

### Finding Description

**Exact code path:**

`PollingTopicListener` declares a single class-level scheduler:

```java
// PollingTopicListener.java line 31
private final Scheduler scheduler = Schedulers.boundedElastic();
```

`Schedulers.boundedElastic()` returns the **JVM-global** shared bounded-elastic scheduler (default cap: `10 × CPU cores` threads, queue 100 000 tasks/thread).

Every call to `listen()` unconditionally allocates a new `PollingContext` and attaches two scheduler consumers to that global pool:

```java
// lines 34-48
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // schedules a delay task
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .withFixedDelay(interval)
                .withScheduler(scheduler))               // schedules repeat tasks forever
```

Each repeat fires `poll()`, which executes a synchronous (blocking) JDBC query via `topicMessageRepository.findByFilter(newFilter)` (line 61), holding a `boundedElastic` thread for the duration of the query. With `startTime=0` and no `endTime`, the subscription never terminates and the repeat count is `Long.MAX_VALUE`.

**Root cause / failed assumption:** The design assumes a small, operator-controlled subscriber population. There is no global subscription counter, no per-IP limit, and no admission-control gate inside `listen()`.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, GrpcConfiguration line 33) limits calls *per TCP connection*, not total connections. An attacker opens C connections × 5 calls = 5C subscriptions with no further barrier.
- `TopicMessageFilter.isValidStartTime()` accepts `startTime=0` because `0 ≤ DomainUtils.now()` (nanoseconds since epoch).
- No IP-based rate limiter or throttle exists in the gRPC module (the `ThrottleConfiguration`/`Bucket4j` setup lives only in the `web3` module).
- `SharedPollingTopicListener` (SHARED_POLL mode) is immune because it uses a single `.share()`d `Flux`; the vulnerability is specific to the `POLL` listener type.

### Impact Explanation
When the `boundedElastic` thread pool is saturated with blocking DB-query tasks from attacker subscriptions, all other reactor pipelines on the same JVM that use `Schedulers.boundedElastic()` — including `SharedTopicListener.listen()` (line 25) which calls `.publishOn(Schedulers.boundedElastic(), ...)` for every legitimate subscriber — stall waiting for a thread. This causes complete denial-of-service for all topic-message delivery on the affected mirror-node instance. Because the mirror node is a read-only relay for Hedera consensus data, disrupting it degrades network observability and downstream application availability without requiring any consensus-layer access.

### Likelihood Explanation
The attack requires:
1. The operator has set `hiero.mirror.grpc.listener.type=POLL` (non-default; default is `REDIS`). This is a realistic deployment scenario for operators without a Redis dependency.
2. The gRPC port (default 5600) is reachable — typical for a public mirror node.
3. No authentication is required for `subscribeTopic`; the Hedera mirror-node gRPC API is intentionally public.
4. A single attacker machine can open hundreds of TCP connections and 5 calls each, reaching thousands of concurrent subscriptions with commodity tooling (e.g., `ghz`, a custom gRPC client).

The attack is repeatable, requires no special knowledge beyond the public protobuf API, and self-sustains as long as the connections remain open.

### Recommendation
1. **Add a global concurrent-subscription limit** inside `PollingTopicListener.listen()` using an `AtomicInteger` counter; reject (throw `StatusRuntimeException(RESOURCE_EXHAUSTED)`) when the limit is exceeded.
2. **Add a per-IP subscription limit** at the gRPC interceptor layer.
3. **Use a dedicated, bounded scheduler** for `PollingTopicListener` instead of the JVM-global `Schedulers.boundedElastic()`, so exhaustion is isolated and does not affect other pipelines.
4. **Enforce a maximum connection count** via Netty's `maxConnectionAge` / connection-limit settings to bound the total number of TCP connections.
5. Consider making `SHARED_POLL` the fallback instead of `POLL`, since it is architecturally immune to this class of attack.

### Proof of Concept
```
Preconditions:
  - Mirror node running with hiero.mirror.grpc.listener.type=POLL
  - gRPC port 5600 reachable

Steps:
1. Obtain the public protobuf definition for ConsensusService.subscribeTopic.

2. Using ghz or a custom gRPC client, open 200 TCP connections to port 5600.

3. On each connection, issue 5 concurrent subscribeTopic RPCs:
     ConsensusTopicQuery {
       topicID: { topicNum: <any valid topic> }
       consensusStartTime: { seconds: 0, nanos: 0 }   // startTime = 0
       // consensusEndTime omitted                     // no endTime → infinite
       // limit omitted                                // no limit → infinite
     }

4. This creates 1000 concurrent subscriptions, each polling the DB every 500 ms
   on Schedulers.boundedElastic().

5. Observe: legitimate subscribers begin timing out; the mirror node stops
   delivering topic messages; JVM thread-pool metrics show boundedElastic
   threads at 100% utilization with a growing task queue.

6. Releasing the attacker connections immediately restores service,
   confirming the resource exhaustion is the sole cause.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
