### Title
Missing Netty Connection Lifetime Limits Enable Graceful-Shutdown Blockade via Partitioned Streaming Connections

### Summary
`grpcServerConfigurer()` configures only `executor` and `maxConcurrentCallsPerConnection` on the `NettyServerBuilder`, omitting all connection-lifetime controls (`maxConnectionAge`, `maxConnectionIdle`, `maxConnectionAgeGrace`). An unprivileged external client can open a long-lived streaming RPC and then stop reading from the socket, filling the TCP receive window to zero. When the server subsequently attempts a graceful restart, it cannot deliver the HTTP/2 GOAWAY frame to that connection, blocking `server.awaitTermination()` until the Kubernetes `terminationGracePeriodSeconds` expires and the pod is force-killed, abruptly terminating all other clients' in-flight RPCs.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← no maxConnectionAge / maxConnectionIdle / maxConnectionAgeGrace
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection = 5` with no connection-lifetime fields.

**Root cause / failed assumption:** The design assumes clients will eventually close their streaming connections voluntarily. Without `maxConnectionAge`, connections are never proactively recycled. Without `maxConnectionIdle`, idle-but-open connections are never reaped. Without `maxConnectionAgeGrace`, there is no bounded window in which the server drains connections before shutdown.

**Exploit flow:**
1. Attacker opens a TCP connection to port 5600 and initiates a `ConsensusService/subscribeTopic` streaming RPC (no authentication required).
2. Attacker's process stops calling `recv()` on the socket. The client-side TCP receive buffer fills; the advertised TCP window drops to zero.
3. The server's TCP send buffer fills; it can no longer write any bytes to that connection — including the HTTP/2 GOAWAY frame.
4. Operator triggers a rolling restart. Spring gRPC calls `server.shutdown()` → `server.awaitTermination()`. The server attempts to send GOAWAY to all connections; the attacker's connection is stuck at the TCP layer.
5. `awaitTermination()` blocks. No graceful-shutdown timeout is configured in the grpc module's resources (confirmed: `grpc/src/main/resources/` contains only `banner.txt`; no `spring.lifecycle.timeout-per-shutdown-phase` override).
6. Kubernetes eventually sends SIGKILL after `terminationGracePeriodSeconds`, force-killing the pod and abruptly closing every other client's active streaming RPC.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` limits RPC multiplexing per connection but does not limit the number of connections, does not close idle connections, and does not bound connection lifetime.
- The GCP gateway `connectionDraining.drainingTimeoutSec: 10` operates at the load-balancer layer and drains *new* requests from the backend, but does not force-close existing TCP connections held by the attacker.
- No gRPC-level keepalive (`permitKeepAliveTime`) is configured, so the server never sends HTTP/2 PING frames to detect dead/unresponsive connections.

### Impact Explanation
Every rolling restart or pod eviction during which the attacker holds one or more zero-window streaming connections results in a force-kill of the gRPC pod. All legitimate clients with active streaming subscriptions receive an abrupt RST rather than a clean GOAWAY + reconnect signal. Depending on client retry logic, this causes subscription gaps (missed topic messages). With multiple attacker connections spread across replicas, every pod in a rolling update can be force-killed, turning a zero-downtime deployment into a full-disruption event. Severity: **Medium–High** (availability impact on a public-facing streaming API; no confidentiality or integrity impact).

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`, any gRPC SDK) and the ability to suppress `recv()` calls — achievable with a trivial script or even `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` on the attacker's own machine after the connection is established. No credentials, no elevated network access, and no knowledge of internal state are required. The attack is repeatable on every deployment cycle and is invisible in application logs (the connection appears alive). Realistic likelihood: **High** for a motivated attacker targeting maintenance windows.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
import java.util.concurrent.TimeUnit;

serverBuilder.maxConnectionAge(30, TimeUnit.MINUTES);       // proactive GOAWAY before shutdown
serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);  // bounded drain window
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);       // reap idle connections
serverBuilder.keepAliveTime(2, TimeUnit.MINUTES);           // detect dead connections
serverBuilder.keepAliveTimeout(20, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveTime(30, TimeUnit.SECONDS);
```

Expose `maxConnectionAge` and `maxConnectionAgeGrace` as fields in `NettyProperties` so they are operator-tunable. Additionally, set `spring.lifecycle.timeout-per-shutdown-phase` in `application.yml` to bound `awaitTermination()`.

### Proof of Concept
```python
import grpc
import time
from com.hedera.mirror.api.proto import consensus_pb2_grpc, consensus_pb2

# Step 1: open streaming subscription (no auth required)
channel = grpc.insecure_channel("grpc-server:5600")
stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
req = consensus_pb2.ConsensusTopicQuery(topicID=...)
iterator = stub.subscribeTopic(req)

# Step 2: read one message to confirm the stream is live
next(iterator)

# Step 3: stop reading — TCP receive window fills to zero
# Simply do nothing; the iterator is never advanced again.
# On Linux the attacker can also enforce this with:
#   iptables -A OUTPUT -p tcp --sport <local_port> -j DROP
# after the handshake, making the partition one-sided and invisible.

# Step 4: trigger a server restart (e.g., kubectl rollout restart)
# Observe: pod does not terminate within normal grace period;
# Kubernetes force-kills after terminationGracePeriodSeconds;
# all other clients receive RST instead of GOAWAY.
time.sleep(9999)
```
