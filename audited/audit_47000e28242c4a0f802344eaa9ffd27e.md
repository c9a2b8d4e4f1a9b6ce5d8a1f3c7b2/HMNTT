### Title
Unprivileged User Can Corrupt `requestSizeHistogram` via Oversized `content-length` Header Causing `parseInt()` Precision Loss

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware()` function reads the user-controlled `content-length` header and passes it directly to `parseInt()` with no upper-bound validation. For values exceeding JavaScript's `Number.MAX_SAFE_INTEGER` (9007199254740991), `parseInt()` returns an IEEE 754 double-precision float that silently loses precision. The existing `|| 0` guard only catches `NaN`/falsy results, not large imprecise numbers, so the corrupted value is recorded into `requestSizeHistogram`, permanently distorting request-size metrics for any route including transaction history endpoints.

### Finding Description
**Exact code path:**
`rest/middleware/metricsHandler.js`, function `metricsMiddleware()`, line 209:
```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:**
`req.headers['content-length']` is a raw, user-supplied string. `parseInt(str, 10)` converts it to a JavaScript `Number` (IEEE 754 double). For any string whose numeric value exceeds `Number.MAX_SAFE_INTEGER` (2^53 − 1 = 9007199254740991), the conversion silently rounds to the nearest representable double:

```
parseInt("9007199254740993", 10)          // → 9007199254740992  (off by 1)
parseInt("99999999999999999999", 10)      // → 100000000000000000000 (imprecise)
parseInt("999999999999999999999999", 10)  // → 1e+24 (wildly imprecise)
```

**Why the existing guard fails:**
The `|| 0` fallback only substitutes `0` when `parseInt` returns a falsy value (`NaN`, `0`, `undefined`, etc.). A large imprecise float like `1e+24` is truthy, so it passes straight through to `requestSizeHistogram.record()`.

**Exploit flow:**
1. Attacker sends any HTTP request to any REST endpoint (e.g., `GET /api/v1/transactions`) with a crafted header: `Content-Length: 99999999999999999999999999999999`
2. Node.js HTTP parser accepts the header as a valid string (no numeric validation at the transport layer; the string is ~32 bytes, far below the 16 KB `maxHeaderSize` limit).
3. Express passes the request to `metricsMiddleware` before any body-size enforcement.
4. `parseInt("99999999999999999999999999999999", 10)` → `1e+32` (imprecise float).
5. `1e+32 || 0` → `1e+32` (truthy; guard bypassed).
6. `requestSizeHistogram.record(1e+32, labels)` is called — the OpenTelemetry SDK records this as a valid observation, placing it in the `+Inf` bucket and inflating the histogram sum by `1e+32`.
7. Prometheus scrapes the corrupted histogram; all percentile/average calculations for that route's request size are now meaningless.

### Impact Explanation
The `requestSizeHistogram` (`api_request_size_bytes`) is a per-route Prometheus histogram used for capacity planning, alerting, and SLO monitoring. A single crafted request can inject an astronomically large value (`1e+32`) into the histogram sum, making the mean and all derived statistics for that route permanently nonsensical until the process restarts. Repeated requests can sustain the corruption indefinitely. While no transaction data is modified, operators lose reliable observability over request-size patterns for transaction history endpoints, which can suppress legitimate size-based alerts (e.g., abnormally large payloads indicating abuse) and corrupt capacity-planning dashboards.

### Likelihood Explanation
The attack requires zero authentication, zero privileges, and zero knowledge of the system beyond the public API. Any HTTP client can set an arbitrary `Content-Length` header. The trigger is a single HTTP request. It is trivially repeatable and automatable. The metrics endpoint is enabled by default (`config.metrics.enabled`) and the middleware is applied globally to all routes via `app.useExt(metricsHandler())`.

### Recommendation
Replace the unchecked `parseInt` call with an explicit safe-integer guard before recording:

```javascript
const rawLen = parseInt(req.headers['content-length'] ?? '0', 10);
const safeLen = Number.isInteger(rawLen) && rawLen >= 0 && rawLen <= Number.MAX_SAFE_INTEGER
  ? rawLen
  : 0;
requestSizeHistogram.record(safeLen, labels);
```

Alternatively, use `Number(req.headers['content-length'])` and clamp with `Math.min(value, Number.MAX_SAFE_INTEGER)`, or validate the header against a regex (`/^\d{1,15}$/`) before parsing.

### Proof of Concept
```bash
# Single request sufficient to corrupt the histogram
curl -s -X GET \
  -H "Content-Length: 99999999999999999999999999999999" \
  http://<mirror-node-rest>/api/v1/transactions

# Verify corruption by scraping metrics
curl -s http://<mirror-node-rest>/api/v1/metrics/ \
  | grep api_request_size_bytes_sum
# Expected (legitimate): api_request_size_bytes_sum{...} <small number>
# Observed after attack:  api_request_size_bytes_sum{...} 1e+32
```

The `api_request_size_bytes_sum` for the targeted route will show `1e+32`, rendering all histogram-derived statistics (mean, percentiles) for that route meaningless.
