### Title
Unthrottled `isComplete()` Skips Page-Size Exhaustion Check, Causing Forced Full-Poll Amplification via Safety-Check Path

### Summary
In `PollingTopicMessageRetriever.PollingContext.isComplete()`, the unthrottled branch (`throttled=false`) only returns `true` when a user-supplied limit is hit. It never checks whether the last page was smaller than `maxPageSize`, which is the standard data-exhaustion signal used by the throttled branch. Because `throttled=false` is reachable by any unprivileged subscriber through the safety-check path in `TopicMessageServiceImpl.missingMessages()`, an attacker can force the retriever to execute the full `maxPolls` database queries (default: 12 × 5 000-row queries at 20 ms intervals) even when the very first poll returns zero rows.

### Finding Description

**Exact code location:**
`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `isComplete()`, lines 121–129.

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();

    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;  // exhaustion check present
    }

    return limitHit;   // ← exhaustion check ABSENT in unthrottled path
}
```

The throttled branch correctly short-circuits when `pageSize < maxPageSize` (fewer rows than requested → no more data). The unthrottled branch ignores `pageSize` entirely and only stops on `limitHit`.

**How `throttled=false` is reached by an unprivileged user:**
`TopicMessageServiceImpl.missingMessages()` (lines 138–178) calls `topicMessageRetriever.retrieve(gapFilter, false)` in two places:

1. **Safety-check path** (line 149): triggered automatically after 1 second when no live message has arrived.
2. **Gap-recovery path** (line 177): triggered when a sequence-number gap is detected.

The safety-check path builds `gapFilter` without a limit:
```java
var gapFilter = topicContext.getFilter().toBuilder().startTime(startTime).build();
return topicMessageRetriever.retrieve(gapFilter, false);   // no limit → limitHit always false
```

With no limit, `filter.hasLimit()` is `false`, so `limitHit` is always `false`, so `isComplete()` always returns `false`. The `repeatWhen` loop therefore runs the full `numRepeats = unthrottled.getMaxPolls()` iterations (default **12**) regardless of how many rows the first poll returned.

**Repeat loop (lines 51–55):**
```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())   // default 20 ms
                .withScheduler(scheduler))
```

Each poll issues a DB query for up to `maxPageSize = 5 000` rows. With zero results on every poll, 12 queries are still executed per safety-check invocation.

### Impact Explanation

Each subscription to a sparse or empty topic that goes 1 second without a live message triggers 12 × 5 000-row DB queries (240 ms of polling at 20 ms intervals). An attacker opening *N* concurrent subscriptions multiplies this to 12 × N unnecessary queries per safety-check cycle. Because the safety check re-arms on every subscription and the subscriber count is unbounded by default, this is a straightforward DB amplification vector that degrades the mirror-node's database tier for all users. The impact is classified as a network-layer performance degradation (DoS on the DB connection pool / query throughput), consistent with the stated scope of "unintended behavior with no concrete funds at direct risk."

### Likelihood Explanation

The attack requires no credentials, no special topic state, and no protocol knowledge beyond the standard HCS gRPC `subscribeTopic` API. Any client that subscribes to a topic with `startTime` in the past and omits both `limit` and a near-future `endTime` will trigger the safety check after exactly 1 second. The attack is trivially repeatable and scriptable with any gRPC client library.

### Recommendation

Add the page-size exhaustion check to the unthrottled branch of `isComplete()`, mirroring the throttled branch:

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    int configuredPageSize = throttled
            ? retrieverProperties.getMaxPageSize()
            : retrieverProperties.getUnthrottled().getMaxPageSize();

    return pageSize.get() < configuredPageSize || limitHit;
}
```

This ensures that when a poll returns fewer rows than `maxPageSize`, the retriever stops immediately rather than exhausting all remaining `maxPolls` iterations.

### Proof of Concept

1. Start the mirror-node gRPC service with default `RetrieverProperties` (`maxPolls=12`, `pollingFrequency=20ms`, `maxPageSize=5000`).
2. Subscribe to any valid but **empty** topic using the HCS `subscribeTopic` RPC, omitting `limit` and `endTime`, with `startTime=0`.
3. Wait 1 second. The safety check in `TopicMessageServiceImpl` fires (`missingMessages(topicContext, null)`), calling `retrieve(gapFilter, false)`.
4. Observe in DB query logs: **12 sequential SELECT queries** are issued against `topic_message`, each requesting up to 5 000 rows, all returning 0 rows.
5. Open 100 concurrent such subscriptions; observe 1 200 unnecessary DB queries per safety-check cycle.
6. With the fix applied, step 4 produces exactly **1 query** (first poll returns 0 < 5 000, `isComplete()` returns `true` immediately). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
