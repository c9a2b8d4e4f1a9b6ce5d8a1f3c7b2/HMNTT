### Title
Unauthenticated `If-None-Match: *` Forces Full Response Buffering in `ShallowEtagHeaderFilter` with No Rate Limiting

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers Spring's `ShallowEtagHeaderFilter` over all `/api/*` endpoints without any per-IP rate limiting or in-flight request cap. `ShallowEtagHeaderFilter` unconditionally buffers the entire response body in a `ContentCachingResponseWrapper` before performing ETag comparison. Because `If-None-Match: *` matches any computed ETag value, an unprivileged attacker can force the server to execute full database queries, serialize and buffer large response payloads in JVM heap, compute MD5 hashes, and then discard everything with a 304 — at arbitrary request rates.

### Finding Description
**Exact code path:**
`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:
```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause — Spring's `ShallowEtagHeaderFilter` execution order:**
`ShallowEtagHeaderFilter.doFilterInternal()` wraps the response in a `ContentCachingResponseWrapper` and calls `filterChain.doFilter()` first. This means the full downstream handler runs — DB query executes, result set is fetched, JSON/Protobuf is serialized, and the entire body is written into an in-memory byte buffer — before `updateResponse()` is ever called. Only after the body is fully buffered does the filter compute the MD5-based ETag and compare it against the `If-None-Match` header. Inside `updateResponse()`, Spring checks:
```java
if (clientETag.trim().equals("*")) {
    response.setStatus(HttpServletResponse.SC_NOT_MODIFIED);
    response.resetBuffer();
    return;
}
```
The wildcard `*` matches unconditionally, so the buffered body is discarded and a 304 is returned — but all the work (query + serialization + heap allocation) has already been done.

**Why existing checks are insufficient:**
- The Traefik middleware chain for rest-java (`charts/hedera-mirror-rest-java/values.yaml`, lines 158–163) contains only `circuitBreaker` and `retry` — no `rateLimit` and no `inFlightReq` middleware, unlike the rosetta chart which has both.
- The middleware is additionally gated on `global.middleware: false` (line 103 of `values.yaml`), meaning it is **disabled by default** and the ingress annotation referencing it has no effect unless explicitly enabled.
- There is no application-level throttle in rest-java (unlike `web3` which has `ThrottleConfiguration` with `bucket4j`). The `WebMvcConfiguration.java` adds only argument resolvers and formatters — no rate-limiting interceptor.
- No authentication is required to reach `/api/*` endpoints.

### Impact Explanation
Each request with `If-None-Match: *` forces: a full database query, full result-set deserialization, full JSON/Protobuf serialization into JVM heap, and an MD5 computation over the buffered bytes — all for a 304 response that returns zero bytes to the attacker. For endpoints that return large paginated result sets (e.g., `/api/v1/transactions?limit=100`), each request can allocate hundreds of kilobytes to megabytes of heap. Sustained at high concurrency from multiple source IPs, this exhausts JVM heap (triggering GC pressure and OOM), saturates DB connection pools (visible in the `hikaricp_connections_active` metric), and drives CPU above the 80% threshold monitored by `RestJavaHighCPU`. This can degrade or crash ≥30% of rest-java processing nodes without any brute-force credential attack.

### Likelihood Explanation
Preconditions: none. The attacker needs only network access to the public API. The exploit is a single HTTP header on any GET request to any `/api/*` path. It is trivially scriptable, repeatable from many IPs, and produces no error responses that would trigger circuit-breaker tripping (304 is not a 5xx). The absence of default rate limiting makes sustained attack straightforward.

### Recommendation
1. **Disable `If-None-Match: *` handling at the filter level**: Subclass `ShallowEtagHeaderFilter` and override `isETagHeaderValid()` (or `updateResponse()`) to skip the `*` wildcard match for GET/HEAD requests, treating it as a cache miss rather than a 304.
2. **Add `inFlightReq` and `rateLimit` Traefik middleware** to `charts/hedera-mirror-rest-java/values.yaml` (mirroring the rosetta chart pattern) and set `global.middleware: true` by default, or enforce limits at the GCP backend policy level via `maxRatePerEndpoint`.
3. **Enable application-level throttling** in rest-java analogous to `web3`'s `ThrottleConfiguration` (bucket4j per-IP token bucket) so the protection is not solely infrastructure-dependent.
4. **Set `writeWeakETag = false`** and consider whether ETag support is needed at all for these read-only mirror API endpoints; if caching is handled upstream (CDN/Redis), the filter adds cost with no benefit.

### Proof of Concept
```bash
# Single request demonstrating forced full-body buffering + 304 discard
curl -v -H "If-None-Match: *" \
  "https://<mirror-node-host>/api/v1/transactions?limit=100"
# Expected: HTTP 304, empty body, but server executed full DB query + serialization

# Sustained resource exhaustion (no auth required, no rate limit by default)
for i in $(seq 1 500); do
  curl -s -o /dev/null -H "If-None-Match: *" \
    "https://<mirror-node-host>/api/v1/transactions?limit=100" &
done
wait
# Monitor: kubectl top pods -l app.kubernetes.io/component=rest-java
# Observe: JVM heap pressure, hikaricp_connections_active spike, CPU > 80%
```
