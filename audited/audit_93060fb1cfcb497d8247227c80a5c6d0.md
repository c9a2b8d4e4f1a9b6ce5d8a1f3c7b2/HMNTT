### Title
Unauthenticated Cache Flooding via Unbounded Unique URL Caching Exhausts Redis and Amplifies Database Load

### Summary
The `cacheKeyGenerator` function in `responseCacheHandler.js` derives cache keys directly from the raw, un-normalized `req.originalUrl` with no rate limiting applied to unauthenticated callers. Any external user can trivially generate an unbounded number of unique cache keys by varying a single query parameter, causing `setSingle()` to write a distinct Redis entry per request. While the configured `allkeys-lfu` eviction policy prevents hard memory exhaustion, it does not prevent cache pollution, eviction churn that displaces legitimate entries, or the resulting amplification of database queries on every cache miss.

### Finding Description

**Exact code path:**

`rest/middleware/responseCacheHandler.js`, lines 151–153 (`cacheKeyGenerator`):
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [1](#0-0) 

The function hashes the raw `req.originalUrl` with zero normalization. The code comment at line 148–149 explicitly acknowledges this is a known gap: *"In the future, this will utilize Edwin's request normalizer (9113)."* [2](#0-1) 

The resulting key is passed unconditionally to `getCache().setSingle()` for every successful (2xx/304) response: [3](#0-2) 

`setSingle` calls Redis `SETEX` with no guard on key count or caller identity: [4](#0-3) 

**Root cause — failed assumption:** The design assumes that the set of distinct URLs hitting the cache is bounded by legitimate usage patterns. There is no URL normalization, no per-IP or global request rate limit, and no cap on the number of cache entries a single caller can create.

**Middleware stack confirms no rate limiting for unauthenticated callers.** `server.js` registers: `requestLogger` → `authHandler` → optional metrics → `responseCacheCheckHandler` → route handlers → `responseHandler` → `responseCacheUpdateHandler`. The `authHandler` only sets a custom *response-size* limit for authenticated users; it does not throttle request rates and returns immediately (allowing the request to proceed) when no credentials are supplied: [5](#0-4) [6](#0-5) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter exists only in the `web3` service, not in the REST API: [7](#0-6) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/transactions?timestamp=gte:0.000000001`, receives 200.
2. `responseCacheUpdateHandler` stores the response under a unique MD5 key.
3. Attacker increments the nanosecond value and repeats at high throughput.
4. Each request is a cache miss → full PostgreSQL query → unique Redis write.
5. Redis fills to its 250 MB limit; `allkeys-lfu` begins evicting the lowest-frequency keys — which are the attacker's own single-use entries, but also any legitimate entry that has not yet accumulated sufficient frequency.
6. Legitimate entries that were recently written but not yet frequently accessed are evicted, causing their next request to also miss the cache and hit the database.

### Impact Explanation

**Cache pollution / eviction churn:** Even with `allkeys-lfu`, the attacker can sustain a write rate that keeps Redis at capacity, forcing continuous eviction. Legitimate entries written between attacker bursts are evicted before accumulating enough frequency to be protected, degrading the effective cache hit rate toward zero.

**Database load amplification:** Every cache miss triggers a full PostgreSQL query. The REST API's database pool is capped at 10 connections by default (`hiero.mirror.rest.db.pool.maxConnections = 10`). Sustained flooding saturates the pool, causing query queuing, statement timeouts (20 s default), and ultimately service degradation or outage for all users. [8](#0-7) 

**Redis memory pressure:** Although `allkeys-lfu` prevents hard OOM, the 250 MB cap combined with continuous unique writes keeps Redis under constant memory pressure, increasing CPU overhead for eviction bookkeeping and reducing throughput for legitimate cache operations. [9](#0-8) 

### Likelihood Explanation

**Preconditions:** None. No authentication, no API key, no special network position required. The REST API is publicly exposed on port 5551 with CORS open to all origins. [10](#0-9) 

**Feasibility:** A single attacker machine sending HTTP GET requests with incrementing nanosecond timestamp values can generate millions of unique URLs per hour. Valid timestamp ranges accepted by the API span years of nanosecond-precision values, providing a practically infinite keyspace. The attack requires no exploit tooling — a simple shell loop or `ab`/`wrk` invocation suffices.

**Repeatability:** The attack is stateless and self-sustaining. After Redis evicts attacker entries, the attacker simply continues sending new unique URLs. There is no lockout, no CAPTCHA, and no IP-based throttle to interrupt the loop.

### Recommendation

1. **Normalize URLs before hashing.** Sort query parameters canonically and strip unknown/irrelevant parameters before computing the cache key. The code already acknowledges this gap (issue 9113); it should be prioritized. [2](#0-1) 

2. **Add per-IP and global request rate limiting** to the REST API middleware stack (e.g., `express-rate-limit` backed by Redis), applied before `responseCacheCheckHandler`, so unauthenticated callers cannot issue unbounded requests. [11](#0-10) 

3. **Restrict cacheable parameter ranges.** Reject or do not cache responses for timestamp values outside a reasonable recent window, limiting the effective keyspace an attacker can exploit.

4. **Set a per-key-prefix entry count limit in Redis** or use a dedicated Redis database for the response cache with a stricter `maxmemory` ceiling, isolating cache flooding from other Redis consumers.

### Proof of Concept

```bash
# Requires: curl, bash. No credentials needed.
# Sends 10,000 unique requests varying a nanosecond timestamp.
# Each generates a unique MD5 cache key written to Redis.

BASE="http://<mirror-node-host>:5551/api/v1/transactions"
for i in $(seq 1 10000); do
  curl -s -o /dev/null "${BASE}?timestamp=gte:0.00000${i}" &
done
wait

# Observe Redis memory usage climbing to maxmemory (250 MB default):
# redis-cli INFO memory | grep used_memory_human

# Observe PostgreSQL connection pool saturation:
# SELECT count(*) FROM pg_stat_activity WHERE datname='mirror_node';

# Legitimate cached responses will show increased miss rates as
# attacker entries compete for LFU slots and force eviction churn.
```
