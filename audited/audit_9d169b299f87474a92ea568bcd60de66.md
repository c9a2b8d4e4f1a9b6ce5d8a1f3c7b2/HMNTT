### Title
Unauthenticated Resource Exhaustion via Unconditional DB Queries in `/block/transaction` Endpoint

### Summary
The `BlockTransaction()` function in `rosetta/app/services/block_service.go` unconditionally executes at least two database queries per request — `FindByIdentifier` (block lookup) and `FindByHashInBlock` (transaction lookup) — with no application-level rate limiting. An unprivileged attacker supplying valid block identifiers with varying or non-existent transaction hashes can sustain a high DB query rate, as neither the block nor transaction lookups are cached, and `FindByHashInBlock` executes its SQL query even when the transaction does not exist. The Traefik-level rate limiting is an optional infrastructure control that is not enforced by the application itself.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `BlockTransaction()`, lines 77–102:

```
line 82:  block, err := s.FindByIdentifier(ctx, request.BlockIdentifier.Index, h)
line 87:  transaction, err := s.FindByHashInBlock(ctx, request.TransactionIdentifier.Hash, ...)
line 97:  err = s.updateOperationAccountAlias(ctx, transaction)
```

`FindByIdentifier` delegates to `blockRepository.findBlockByHash()` which always executes `selectByHashWithIndex` SQL against `record_file` — there is no block-level cache. [1](#0-0) 

`FindByHashInBlock` always executes `selectTransactionsByHashInTimestampRange` against the `transaction` table. Critically, the DB query fires before the empty-result check, so even a non-existent transaction hash causes a full DB round-trip. [2](#0-1) 

`updateOperationAccountAlias` calls `GetAccountAlias` per operation, which executes `selectCryptoEntityWithAliasById` against the `entity` table. An LRU cache (`entityCache`, default max 524,288 entries) mitigates this for repeated account IDs, but the block and transaction queries have no equivalent cache. [3](#0-2) [4](#0-3) 

**Application-level middleware chain** (`rosetta/main.go`, lines 217–219) wraps only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which perform rate limiting or concurrency control. [5](#0-4) 

**Infrastructure-level controls** exist only in the Traefik Helm chart values: `rateLimit: average: 10` scoped per `requestHost` (a global 10 req/s limit for the whole service, not per source IP), and `inFlightReq: amount: 5` per source IP. These are optional, deployment-specific, and not enforced by the application code. [6](#0-5) 

**Compounding factor:** The Traefik `retry` middleware (`attempts: 3`) retries on 5xx responses. Since Rosetta errors (including "transaction not found") are returned as HTTP 500, each attacker request with a non-existent transaction hash triggers up to 3 retry attempts at the Traefik layer, multiplying the DB query load by up to 3×. [7](#0-6) 

### Impact Explanation

Each `/block/transaction` request with a valid block identifier and any transaction hash (including non-existent ones) unconditionally fires 2 DB queries. With the Traefik retry amplification, a single attacker request can produce up to 6 DB queries. Without Traefik (direct deployment, alternative ingress), there is no concurrency or rate control at all. The DB connection pool has a default max of 100 open connections; sustained flooding can saturate the pool, causing query queuing, increased latency, and degraded availability for legitimate users. The `record_file` and `transaction` table queries involve subqueries and range scans that are non-trivial under concurrent load. [1](#0-0) 

### Likelihood Explanation

No authentication or authorization is required to call `/block/transaction` — it is a public Rosetta API endpoint. Valid block identifiers are trivially obtained by first calling `/block` (also public). The attacker only needs to know one valid block hash and index, then can send requests with arbitrary transaction hashes in a tight loop. This requires no special knowledge, no credentials, and no brute force — a single script with a valid block identifier suffices. The absence of application-level rate limiting means the attack is repeatable and sustainable indefinitely until infrastructure controls (if present) intervene. [8](#0-7) 

### Recommendation

1. **Add application-level rate limiting** directly in the Go HTTP server (e.g., using `golang.org/x/time/rate` or a middleware like `github.com/ulule/limiter`) scoped per source IP, not per destination host.
2. **Cache block lookups**: Introduce an LRU cache for `FindByIdentifier` results (keyed on block hash/index), similar to the existing `entityCache`, to avoid a DB round-trip for every request to the same block.
3. **Validate transaction hash format early**: Reject malformed hashes before any DB query is issued (a hex-decode check already exists in `FindByHashInBlock` but the block lookup still fires first).
4. **Decouple the Traefik retry middleware from 5xx Rosetta errors**: Configure retry only for network-level errors, not application-level "not found" responses, to prevent the 3× amplification.
5. **Set a per-IP concurrency limit** at the application level as a defense-in-depth measure independent of the ingress configuration.

### Proof of Concept

**Preconditions:**
- Rosetta node running in online mode, reachable at `http://<host>:5700`
- No Traefik ingress (or Traefik deployed without the rate-limit middleware)

**Step 1 — Obtain a valid block identifier:**
```bash
curl -s -X POST http://<host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}'
# Extract block.block_identifier.index and block.block_identifier.hash from response
```

**Step 2 — Flood with varying transaction hashes:**
```bash
BLOCK_INDEX=1
BLOCK_HASH="<hash from step 1>"
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'

for i in $(seq 1 10000); do
  FAKE_HASH=$(openssl rand -hex 48)
  curl -s -X POST http://<host>:5700/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":$NETWORK,\"block_identifier\":{\"index\":$BLOCK_INDEX,\"hash\":\"$BLOCK_HASH\"},\"transaction_identifier\":{\"hash\":\"$FAKE_HASH\"}}" &
done
wait
```

**Expected result:** Each iteration triggers `FindByIdentifier` (1 DB query against `record_file`) + `FindByHashInBlock` (1 DB query against `transaction`) = 2 DB queries per request, with no application-level throttle. At 100 concurrent requests, this produces 200 simultaneous DB queries, saturating the default pool of 100 connections and causing measurable latency increases for all other endpoints. [8](#0-7)

func (s *blockAPIService) BlockTransaction(
	ctx context.Context,
	request *rTypes.BlockTransactionRequest,
) (*rTypes.BlockTransactionResponse, *rTypes.Error) {
	h := tools.SafeRemoveHexPrefix(request.BlockIdentifier.Hash)
	block, err := s.FindByIdentifier(ctx, request.BlockIdentifier.Index, h)
	if err != nil {
		return nil, err
	}

	transaction, err := s.FindByHashInBlock(
		ctx,
		request.TransactionIdentifier.Hash,
		block.ConsensusStartNanos,
		block.ConsensusEndNanos,
	)
	if err != nil {
		return nil, err
	}

	if err = s.updateOperationAccountAlias(ctx, transaction); err != nil {
		return nil, err
	}

	return &rTypes.BlockTransactionResponse{Transaction: transaction.ToRosetta()}, nil
}

func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

func (ar *accountRepository) GetAccountAlias(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return accountId, nil
		}

		return zero, hErrors.ErrDatabaseError
	}

	if len(entity.Alias) == 0 {
		return accountId, nil
	}

	accountAlias, err := types.NewAccountIdFromEntity(entity)
	if err != nil {
		log.Warnf("Failed to create AccountId from alias '0x%s': %v", hex.EncodeToString(entity.Alias), err)
		return accountId, nil
	}

	return accountAlias, nil
}
