### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to `/accounts/:id/allowances/tokens`

### Summary
The `/accounts/:id/allowances/tokens` endpoint is publicly accessible with no rate limiting. Each request invokes `getAccountTokenAllowances()`, which can execute a three-subquery `union all` SQL statement, holding a database connection for the duration. An unprivileged attacker flooding this endpoint with concurrent requests can exhaust the finite `pg` connection pool, causing all subsequent database-dependent requests across the entire service to fail with connection timeout errors.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 18 registers the route with no authentication guard: [1](#0-0) 

This calls `TokenAllowanceController.getAccountTokenAllowances` → `TokenAllowanceService.getAccountTokenAllowances()`: [2](#0-1) 

`getQuery()` constructs up to three subqueries joined with `union all`: [3](#0-2) 

Each subquery hits the `token_allowance` table with its own `ORDER BY` + `LIMIT`, making this one of the heavier read paths in the service.

`BaseService.getRows()` acquires a connection from the global `pg` pool for every call: [4](#0-3) 

The pool is bounded by `config.db.pool.maxConnections`: [5](#0-4) 

**Root cause — failed assumption:** The design assumes request volume will be naturally bounded. There is no rate-limiting middleware anywhere in the stack. A search across all `rest/**/*.js` files for `rateLimit`, `rate_limit`, `rateLimiter`, or `throttle` returns zero matches. The server middleware chain in `server.js` is: cors → compression → httpContext → requestLogger → authHandler → optional metrics → optional response cache → routes — no rate limiter at any layer: [6](#0-5) 

**Why the `statementTimeout` check is insufficient:** `statement_timeout` caps how long a single query runs, but it does not prevent pool exhaustion. An attacker sending requests at a rate exceeding `maxConnections / statementTimeout` keeps the pool permanently saturated. Once the pool is full, `connectionTimeoutMillis` causes every new request (from any user, to any endpoint) to fail immediately with a pool-timeout error — achieving service-wide DoS without the attacker needing to sustain the flood indefinitely.

### Impact Explanation
When the connection pool is exhausted, every endpoint that touches the database — transactions, balances, tokens, contracts, etc. — returns errors to all users. This is a non-network DoS: the service process stays alive but is functionally unavailable. Because the pool is shared globally, a single targeted flood of one cheap-to-construct endpoint degrades the entire API surface. Severity is **High** for any deployment where this mirror node serves production traffic.

### Likelihood Explanation
No privileges, API keys, or accounts are required. The attacker only needs the public URL. The attack is trivially scriptable with any HTTP load tool (`wrk`, `ab`, `hey`, `curl` in a loop). It is repeatable and resumable at will. The three-subquery union path is reachable with a simple request like `GET /api/v1/accounts/0.0.1/allowances/tokens?spender.id=gt:0.0.1&spender.id=lt:0.0.9999999`, which forces all three bound filters to be non-empty, maximizing per-request DB cost.

### Recommendation
1. **Add a rate-limiting middleware** (e.g., `express-rate-limit` or an equivalent) globally in `server.js`, applied before route handlers, with per-IP limits tuned to expected legitimate traffic.
2. **Add a per-endpoint concurrency limit** or queue depth for expensive union-all paths.
3. **Reduce default `maxConnections`** or introduce a separate, smaller pool for public read endpoints to isolate blast radius.
4. **Enable the optional response cache** (Redis-backed) by default for this endpoint, so repeated identical queries are served from cache without touching the DB pool.
5. Consider requiring at minimum a lightweight API token for endpoints that trigger multi-subquery paths.

### Proof of Concept
```bash
# Step 1: No credentials needed. Send 500 concurrent requests.
hey -n 5000 -c 500 \
  "https://<mirror-node-host>/api/v1/accounts/0.0.2/allowances/tokens?spender.id=gt:0.0.1&spender.id=lt:0.0.9999999&token.id=gt:0.0.1"

# Step 2: While the flood is running, observe that unrelated endpoints fail:
curl "https://<mirror-node-host>/api/v1/transactions"
# Expected: 500 / connection pool timeout error

# Step 3: Stop the flood; service recovers automatically (no persistent damage).
# Repeat to maintain DoS.
```
The attack requires no authentication, no special knowledge of the target's data, and no network amplification — only the ability to open TCP connections to the public API port.
