### Title
Global Rate-Limit Bucket Allows Single-Source DoS of Entire Web3 API

### Summary
`ThrottleManagerImpl.throttle()` checks a single application-wide `rateLimitBucket` with no per-IP or per-user partitioning. Any unauthenticated attacker who sends ≥500 HTTP requests per second (the default quota) from a single source exhausts the entire bucket, causing every other legitimate caller to receive HTTP 429 for the remainder of that second. The bucket refills greedily each second, so the attack is trivially repeatable and continuous.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.java`, `throttle()`, line 38:
```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
``` [1](#0-0) 

`rateLimitBucket` is a single Spring singleton bean defined in `ThrottleConfiguration.java`:
```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
``` [2](#0-1) 

The default `requestsPerSecond` is 500: [3](#0-2) 

**Root cause:** The bucket is shared across all callers with no source-IP key, no user identity key, and no per-connection quota. The failed assumption is that 500 req/s will be distributed across many independent users; in reality a single source can consume all 500 tokens before any other request is processed.

**Why existing checks are insufficient:**
The `request[]` filter system (`RequestProperties`) can be configured to REJECT or THROTTLE requests matching content-based predicates (DATA field, BLOCK field, etc.), but:
- It is empty by default (`request = List.of()`).
- It matches on request *content*, not on source IP.
- Even when configured, it runs *after* the global bucket check (lines 44–48), so it cannot prevent bucket exhaustion. [4](#0-3) [5](#0-4) 

### Impact Explanation
An attacker who saturates the global bucket forces every other caller to receive HTTP 429 (`Too Many Requests`) for the duration of the attack. Because the bucket refills greedily every second, the attack must only be sustained (not one-shot) to maintain a continuous denial of service. The web3 module is the Ethereum-compatible JSON-RPC gateway for the mirror node; complete unavailability of this endpoint blocks all dApp interactions, contract calls, and gas estimations routed through it.

### Likelihood Explanation
No authentication, no account, and no special network position is required. Any attacker with a single machine and a standard HTTP client (e.g., `wrk`, `hey`, `ab`, or a simple loop with `curl`) can trivially generate 500+ HTTP POST requests per second. The attack is repeatable every second indefinitely, costs the attacker nothing beyond bandwidth, and is undetectable as malicious at the application layer because each individual request is structurally valid.

### Recommendation
Replace the single global bucket with a per-source-IP bucket map (e.g., using Bucket4j's `ProxyManager` backed by a `ConcurrentHashMap` or a cache with TTL eviction). Each unique client IP gets its own token bucket capped at a per-IP limit (e.g., 50 req/s), while the global cap remains as a secondary ceiling. Additionally, consider extracting the client IP from `X-Forwarded-For` / `X-Real-IP` headers (with trusted-proxy validation) so the per-IP limit works correctly behind a reverse proxy. A Spring `HandlerInterceptor` or a servlet `Filter` is the appropriate integration point to resolve the IP before `ThrottleManagerImpl.throttle()` is called.

### Proof of Concept
**Preconditions:** A running instance of the web3 mirror node with default configuration (`requestsPerSecond=500`). No credentials needed.

**Steps:**

1. From a single machine, launch a flood of concurrent HTTP POST requests to the contract-call endpoint:
   ```bash
   wrk -t8 -c200 -d30s -s post.lua http://<mirror-node-host>/api/v1/contracts/call
   # post.lua sets Content-Type: application/json and a minimal valid ContractCallRequest body
   ```
2. Simultaneously, from a second machine (legitimate user), send a single request:
   ```bash
   curl -X POST http://<mirror-node-host>/api/v1/contracts/call \
        -H 'Content-Type: application/json' \
        -d '{"to":"0x...","gas":50000}'
   ```
3. **Result:** The legitimate user's request receives HTTP 429 with body `"Requests per second rate limit exceeded"` while the attacker's flood continues to consume all available tokens each second.
