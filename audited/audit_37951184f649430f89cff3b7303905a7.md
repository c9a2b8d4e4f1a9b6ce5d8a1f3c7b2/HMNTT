### Title
Resource Exhaustion via `endTimeInterval` Grace Period in `pastEndTime()` Subscription Lifecycle

### Summary
The `pastEndTime()` method in `TopicMessageServiceImpl` uses `endTimeInterval` (default 30 seconds) as both the polling interval and the grace period before a subscription is considered complete. An unprivileged attacker can open many subscriptions with `endTime` set to just one nanosecond beyond the current time, causing each subscription to remain alive for up to 30 extra seconds, consuming server-side resources (threads, DB connections, listener registrations) with no per-IP or global subscription count limit enforced at the gRPC layer.

### Finding Description

**Exact code path:**

`pastEndTime()` at [1](#0-0)  polls `isComplete()` with a fixed delay of `endTimeInterval`:

```java
return Flux.empty()
    .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
            .withFixedDelay(grpcProperties.getEndTimeInterval()));
```

`isComplete()` at [2](#0-1)  returns `true` only when `now > endTime + endTimeInterval`:

```java
return Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
```

`endTimeInterval` defaults to 30 seconds: [3](#0-2) 

**Root cause:** The same `endTimeInterval` value serves as both the polling cadence and the grace window. A subscription with `endTime = now + 1ns` will not be detected as complete until `now > (now + 1ns) + 30s`, i.e., ~30 seconds later. The first poll happens after 30 seconds, at which point `isComplete()` finally returns `true` and the `takeUntilOther` signal fires.

**Input validation failure:** `TopicMessageFilter.isValidEndTime()` only requires `endTime > startTime` [4](#0-3)  â€” it does not enforce a minimum gap between `endTime` and `now`, so `endTime = now + 1ns` is fully accepted.

**Connection limit:** The only concurrency guard is `maxConcurrentCallsPerConnection = 5` [5](#0-4)  â€” per connection, not globally. There is no global cap on total open subscriptions or total connections from a single IP.

### Impact Explanation

Each open subscription holds: a Reactor `Flux` pipeline, a `TopicListener` registration (Redis or DB poll), a `TopicMessageRetriever` handle, and a counted slot in `subscriberCount`. With `endTimeInterval = 30s`, an attacker sustaining N connections Ã— 5 subscriptions each keeps `5N` subscriptions alive continuously. Server thread pools and DB connection pools are finite; sustained pressure degrades or denies service to legitimate subscribers. The `subscriberCount` gauge will spike, and bounded-elastic scheduler threads will be exhausted. This is a pure griefing/DoS with no on-chain cost.

### Likelihood Explanation

Any unauthenticated client can open gRPC connections to port 5600 and issue `subscribeTopic` calls. No credentials, tokens, or funds are required. The attack is trivially scriptable: open connections in a loop, each with `endTime = Instant.now() + 1ns`, and repeat every 30 seconds to maintain a steady pool of zombie subscriptions. The `isValidStartTime` check [6](#0-5)  does not prevent this. The attack is repeatable and requires no special knowledge of the system.

### Recommendation

1. **Decouple the grace period from the polling interval.** Introduce a separate, smaller `endTimeGracePeriod` (e.g., 1â€“2 seconds) used only in `isComplete()`, keeping `endTimeInterval` as the polling cadence.
2. **Enforce a global subscription cap.** Reject new subscriptions when `subscriberCount` exceeds a configurable maximum, returning `RESOURCE_EXHAUSTED` to the caller.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleConfiguration` used in the web3 module. [7](#0-6) 
4. **Validate `endTime` minimum gap.** Reject subscriptions where `endTime - now < some_minimum_threshold` to prevent trivially short-lived subscriptions from exploiting the grace window.

### Proof of Concept

```python
import grpc, time, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_network_service_pb2

def open_zombie_subscription(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now_ns = time.time_ns()
    request = mirror_network_service_pb2.ConsensusTopicQuery(
        topicID=...,
        consensusStartTime=...,
        consensusEndTime=now_ns + 1,  # 1 nanosecond in the future
    )
    # Subscription stays open ~30s due to endTimeInterval grace period
    for _ in stub.subscribeTopic(request):
        pass

# Open many connections, each with 5 subscriptions (maxConcurrentCallsPerConnection)
channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(200)]
threads = []
for ch in channels:
    for _ in range(5):
        t = threading.Thread(target=open_zombie_subscription, args=(ch,))
        t.start()
        threads.append(t)
# 200 connections Ã— 5 = 1000 zombie subscriptions held open for 30s each
# Repeat every 30s to maintain persistent load
```
