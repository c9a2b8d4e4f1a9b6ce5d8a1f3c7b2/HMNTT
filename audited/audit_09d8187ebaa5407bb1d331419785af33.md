### Title
Accept-Encoding Substring Bypass Allows Uncompressed Large Opcode Responses

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController` uses a substring match (`.contains("gzip")`) to enforce gzip compression, but the actual HTTP compression layer (Tomcat/Spring) uses strict RFC 7231 token matching. An attacker can send `Accept-Encoding: xgzip` or `Accept-Encoding: notgzip` to pass the application-level check while causing the server to return the full uncompressed opcode trace payload.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76:

```java
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

**Root cause:** The check uses Java `String.contains("gzip")`, which matches any string containing the four characters `g`, `z`, `i`, `p` as a contiguous substring. This is not equivalent to checking for the `gzip` token as defined in RFC 7231 §5.3.4, which requires whitespace/comma delimiters.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{hash}/opcodes?stack=true&memory=true&storage=true` with header `Accept-Encoding: xgzip`.
2. `validateAcceptEncodingHeader("xgzip")` evaluates `"xgzip".toLowerCase().contains("gzip")` → `true` → no exception thrown, validation passes (line 76).
3. `throttleManager.throttleOpcodeRequest()` is called (line 61) — rate-limits requests but does not limit per-response payload size.
4. `opcodeService.processOpcodeCall(request)` executes the full EVM replay (line 64).
5. Spring Boot's embedded Tomcat performs proper RFC 7231 content negotiation on the response. `xgzip` is not a recognized encoding token, so Tomcat does **not** apply gzip compression.
6. The full uncompressed opcode trace (stack + memory + storage for every EVM step) is transmitted to the attacker.

**Why existing checks fail:**
- The throttle (line 61) limits request frequency, not response byte volume. Even at the throttle limit, each response can be orders of magnitude larger uncompressed vs. compressed.
- The test suite at line 507 only covers `{"deflate", "identity", "br", ""}` — none of the substring-bypass values (`xgzip`, `notgzip`, `gzip-not`, etc.) are tested, so the bypass is undetected.
- No `server.compression.*` configuration was found in the application YAML files, meaning compression is entirely dependent on the client advertising a valid `gzip` token, which this check fails to enforce.

### Impact Explanation
The opcodes endpoint re-executes transactions on the EVM and returns the full trace including stack frames, memory snapshots, and storage diffs for every opcode. The comment in the code itself acknowledges this response is "huge" and that without compression "this will result in big network latency." An attacker bypassing compression can force the server to transmit multi-megabyte uncompressed JSON payloads per request. Even within the throttle window, repeated requests from multiple clients or IPs can saturate outbound bandwidth and degrade service availability for legitimate users.

### Likelihood Explanation
No authentication or privilege is required — the endpoint is publicly accessible. The bypass requires only crafting a single HTTP header value (`xgzip`), which is trivial. The attack is fully repeatable and automatable. The throttle provides partial mitigation but does not prevent bandwidth exhaustion at scale. Any attacker with basic HTTP knowledge and the ability to send custom headers can exploit this.

### Recommendation
Replace the substring check with proper RFC 7231 token parsing. Split the header value on commas and whitespace, strip quality values (`q=...`), and check for an exact case-insensitive match against the token `gzip`:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean hasGzip = Arrays.stream(acceptEncodingHeader.split(","))
        .map(token -> token.split(";")[0].trim().toLowerCase())
        .anyMatch("gzip"::equals);
    if (!hasGzip) {
        throw ...;
    }
}
```

Alternatively, use Spring's `org.springframework.http.HttpHeaders` parsing utilities or a `Negotiator` (already used elsewhere in the codebase) to perform standards-compliant content negotiation before allowing the request to proceed.

### Proof of Concept
```bash
# Bypass: xgzip contains "gzip" as substring, passes the check,
# but Tomcat does not recognize it as a valid encoding → uncompressed response

curl -v \
  -H "Accept-Encoding: xgzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/<tx-hash>/opcodes?stack=true&memory=true&storage=true"

# Expected (legitimate): Content-Encoding: gzip header present, compressed body
# Actual (bypass):       No Content-Encoding header, full uncompressed JSON body returned
# Confirm bypass passes validation (no 406 Not Acceptable returned)

# Also works with:
# -H "Accept-Encoding: notgzip"
# -H "Accept-Encoding: gzip-disabled"
# -H "Accept-Encoding: xgzipx"
```
