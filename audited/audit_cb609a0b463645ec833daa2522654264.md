### Title
Deleted Entity Disclosure via Numeric ID Lookup in GraphQL `getByIdAndType()`

### Summary
`EntityServiceImpl.getByIdAndType()` uses Spring Data's generic `CrudRepository.findById()`, which performs no `deleted` filter, while `getByAliasAndType()` and `getByEvmAddressAndType()` both use custom native queries that explicitly enforce `deleted is not true`. Any unauthenticated caller can supply a numeric `entityId` to the GraphQL `account` query and receive a fully-populated response for an account that has been deleted on the Hedera network, while the same account is correctly suppressed when looked up by alias or EVM address.

### Finding Description

**Exact code path:**

`AccountController.account()` dispatches on input type: [1](#0-0) 

When `entityId` is supplied it calls `getByIdAndType()`: [2](#0-1) 

`findById()` is the bare Spring Data `CrudRepository` method — it issues `SELECT * FROM entity WHERE id = ?` with no predicate on `deleted`. [3](#0-2) 

The only post-fetch filter is a type check (`e.getType() == type`); there is no check on `e.getDeleted()`.

By contrast, both sibling methods use custom native queries that enforce `deleted is not true`: [4](#0-3) 

**Root cause:** The developer added `deleted is not true` guards to the alias and EVM-address lookup paths but forgot to apply the same guard to the primary-key path. Spring Data's `findById()` has no awareness of the application-level soft-delete convention.

**Exploit flow:**
1. Attacker learns the numeric entity ID of a deleted Hedera account (IDs are sequential and publicly visible in transaction records on the mirror node itself).
2. Attacker sends an unauthenticated GraphQL query:
   ```graphql
   { account(input: { entityId: "0.0.12345" }) { id balance } }
   ```
3. `getByIdAndType()` → `findById(12345)` returns the deleted entity row.
4. The type filter passes (it is still `ACCOUNT`); the deleted entity is mapped and returned to the caller as a live account.
5. The same query using `alias` or `evmAddress` for the same account correctly returns `null`.

**Why existing checks are insufficient:** The `.filter(e -> e.getType() == type)` chain only checks entity type, not deletion status. There is no middleware, interceptor, or schema directive that enforces a deleted-entity guard on the ID path.

### Impact Explanation
Any consumer of the GraphQL API (wallet, explorer, dApp backend) that uses the `entityId` input to verify whether an account is active will receive a positive result for a deleted account. This creates a false representation of network state: the account appears active with its last-known balance and keys, even though it no longer exists on the Hedera network. Downstream logic that gates transaction construction or participant validation on this response can be deceived into treating a deleted account as a valid, live participant.

### Likelihood Explanation
No authentication or special privilege is required. Entity IDs are monotonically increasing integers that are fully public. An attacker can enumerate deleted accounts trivially by querying the mirror node's REST `/api/v1/accounts` endpoint (which correctly filters deleted entities) and comparing against GraphQL responses for the same IDs. The inconsistency is stable and repeatable across every request.

### Recommendation
Replace the bare `findById` call in `getByIdAndType()` with a query that enforces the soft-delete predicate, consistent with the other two lookup methods. Either:

1. Add a named query to `EntityRepository`:
   ```java
   @Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByIdAndNotDeleted(Long id);
   ```
   and call it from `getByIdAndType()`.

2. Or add an in-memory guard immediately after `findById`:
   ```java
   return entityRepository.findById(entityId.getId())
       .filter(e -> e.getType() == type)
       .filter(e -> !Boolean.TRUE.equals(e.getDeleted()));
   ```

Option 1 is preferred because it avoids fetching the row at all when it is deleted, matching the behaviour of the other two paths.

### Proof of Concept

**Precondition:** Know the numeric ID of any deleted Hedera account (e.g., `0.0.12345`). Confirm it is deleted via REST: `GET /api/v1/accounts/0.0.12345` returns 404 or `deleted: true`.

**Trigger:**
```graphql
query {
  account(input: { entityId: "0.0.12345" }) {
    id
    balance
    deleted
  }
}
```

**Expected (correct) result:** `null` (account is deleted).

**Actual result:** A populated `Account` object is returned, including the account's last-known balance and metadata, with `deleted: true` visible in the raw entity but not suppressed by the service layer — demonstrating that the deleted entity is served as if it were active.
