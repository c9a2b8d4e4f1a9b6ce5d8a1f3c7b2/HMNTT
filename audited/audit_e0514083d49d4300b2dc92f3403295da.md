### Title
`initGenesisRecordFile` Retry-on-Error Loop Enables Sustained DB CPU Exhaustion via `once.Do` Misplacement

### Summary
`initGenesisRecordFile()` uses `sync.Once` only on the success path, while the early-exit guard checks `br.genesisBlock.ConsensusStart != genesisConsensusStartUnset`. When the DB returns any error, `once.Do` is never invoked, `ConsensusStart` stays at `-1`, and every subsequent API call unconditionally re-executes the expensive `selectGenesis` correlated subquery. Because the Rosetta API has no rate limiting, an unprivileged attacker can flood the endpoint with concurrent requests, exhaust the DB connection pool, and lock the service into a permanent retry loop that saturates DB CPU.

### Finding Description

**Exact code path** — `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {   // guard: only passes after once.Do
        return nil
    }
    // ... acquire DB ...
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)       // returns WITHOUT calling once.Do
    }
    br.once.Do(func() {                                                  // only reached on success
        br.genesisBlock = rb
    })
    ...
}
``` [1](#0-0) 

**Root cause** — `once.Do` is placed on the success branch only. The early-exit guard at line 244 is gated on `ConsensusStart != -1`, which is only true after `once.Do` has fired. If the DB query fails, `once.Do` is never called, `ConsensusStart` remains `-1`, and the guard never short-circuits future calls. [2](#0-1) 

**All five public API methods** (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) call `initGenesisRecordFile` on every request, so every in-flight request re-executes `selectGenesis` while the DB is in an error state. [3](#0-2) 

**`selectGenesis` is expensive** — it runs a CTE against `account_balance`, a correlated subquery inside the main `SELECT`, and a join + sort on `record_file`: [4](#0-3) 

**No rate limiting exists** in the Rosetta API. The middleware stack is only metrics, tracing, and CORS — no request throttle: [5](#0-4) 

The DB connection pool is bounded by `MaxOpenConnections` (set from config), but there is no per-IP or per-endpoint request cap: [6](#0-5) 

### Impact Explanation

While the DB connection pool is exhausted (attacker holds all slots with in-flight `selectGenesis` queries), every new legitimate request also fails with a DB error, keeping `ConsensusStart` at `-1`. When the pool briefly frees a slot, the next request re-executes the expensive query rather than returning the cached value. This creates a sustained DB CPU spike that degrades or denies service to all Rosetta API consumers. Because the Rosetta API is the canonical block-query interface for the Hedera mirror node, this affects all downstream integrations (exchanges, wallets, block explorers) that rely on it.

### Likelihood Explanation

The Rosetta API is publicly reachable with no authentication. An attacker needs only an HTTP client capable of sending concurrent POST requests to any of the five block-query endpoints. The attack is self-reinforcing: the attacker's flood exhausts the pool, causing errors, which prevent caching, which causes every subsequent request (attacker or legitimate) to hit the DB again. The attack window is the period before any single request succeeds and fires `once.Do`; sustaining the flood at a rate that keeps the pool fully occupied is straightforward with commodity tooling (e.g., `wrk`, `hey`, `ab`).

### Recommendation

Move `once.Do` to be the primary guard, and handle errors inside it using a separate stored error field:

```go
type blockRepository struct {
    ...
    once        sync.Once
    genesisErr  *rTypes.Error
}

func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
            br.genesisErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
    })
    return br.genesisErr
}
```

If retry-on-transient-error is required, use a mutex with an explicit "initialized" boolean rather than `sync.Once`, and apply an exponential back-off with a cap to prevent thundering-herd on DB recovery.

### Proof of Concept

**Preconditions:**
- Rosetta API running in online mode, genesis block not yet cached (fresh start or server restart).
- Attacker has network access to the Rosetta port (default 8082).

**Steps:**

1. Determine `MaxOpenConnections` from config (default is typically small, e.g., 10–50).
2. Launch N+1 concurrent requests (N = pool size) to any block endpoint, e.g.:
   ```
   POST /block {"network_identifier": {...}, "block_identifier": {"index": 0}}
   ```
   using a tool such as:
   ```bash
   hey -n 10000 -c 200 -m POST -H "Content-Type: application/json" \
       -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":0}}' \
       http://<rosetta-host>:8082/block
   ```
3. Observe that all requests return `ErrDatabaseError` or `ErrNodeIsStarting`.
4. Observe DB CPU at 100% as `selectGenesis` is executed for every request.
5. Stop the flood briefly; observe that the first request to succeed caches the genesis block and subsequent requests return immediately — confirming the bug is the missing `once.Do` on the error path.
6. Resume the flood to re-exhaust the pool before any success, sustaining the DoS.
