### Title
Estimate Gas Binary Search Bypasses Gas Throttle, Enabling 20x EVM Execution Amplification

### Summary
When `estimate=true` is set in a `POST /api/v1/contracts/call` request, the gas throttle is charged only once for the declared `request.getGas()`, but the actual EVM is invoked up to 21 times: once in the initial `callContract()` and up to 20 more times inside `BinaryGasEstimator.search()`. The binary search iterations call `doProcessCall(params, gas, true)` with `estimate=true`, which explicitly skips all gas-throttle accounting. Any unauthenticated user can exploit this to consume 20x more CPU/thread resources than the throttle intends to permit.

### Finding Description

**Exact code path:**

1. `ContractController.call()` calls `throttleManager.throttle(request)`, which charges `request.getGas()` (up to 15,000,000) from the gas bucket exactly once. [1](#0-0) [2](#0-1) 

2. `processCall()` detects `isEstimate()` and calls `estimateGas()`. [3](#0-2) 

3. `estimateGas()` first calls `callContract()`, which internally calls `doProcessCall(params, params.getGas(), false)` — with `estimate=false`. This triggers `restoreGasToBucket()`, which **returns most of the gas back to the bucket** (up to 100% of unused gas by default). [4](#0-3) [5](#0-4) 

4. Then `binaryGasEstimator.search()` is invoked, which loops up to `maxGasEstimateRetriesCount=20` times, each time calling `doProcessCall(params, gas, true)` — with `estimate=true`. [6](#0-5) [7](#0-6) 

5. **Root cause**: Inside `doProcessCall()`, the `finally` block that calls `restoreGasToBucket()` and records metrics is guarded by `if (!estimate)`. When `estimate=true`, the gas throttle is **never consulted** for any of the 20 binary search EVM executions. [8](#0-7) 

**Failed assumption:** The throttle design assumes 1 EVM execution per request. For `estimate=true`, the actual EVM execution count is 1 (initial) + up to 20 (binary search) = 21, but only the initial call is throttled — and even that gas is largely returned to the bucket via `restoreGasToBucket()`.

**Why existing checks fail:**

- The gas bucket (`gasPerSecond = 7,500,000,000`) charges `request.getGas()` once, then restores most of it after the initial call. The 20 binary search iterations are completely invisible to the gas bucket. [9](#0-8) 

- The RPS bucket (`requestsPerSecond = 500`) is global and not per-IP, so a single attacker can consume all 500 slots. [10](#0-9) 

- `maxGasEstimateRetriesCount=20` is a hard cap on iterations but does not reduce the amplification factor — it defines it. [11](#0-10) 

- There is no per-IP rate limiting, no authentication requirement, and no separate throttle for estimate-mode requests.

### Impact Explanation

An attacker sending 500 `estimate=true` requests per second (the RPS limit) causes up to 10,500 EVM executions per second, versus 500 for normal calls. Each binary search iteration executes the full EVM with gas values up to `maxGasLimit=15,000,000`. This saturates CPU and the shared thread pool, preventing the service from processing legitimate transactions. The gas throttle — the primary defense against resource exhaustion — is structurally bypassed for all binary search iterations. This matches the stated critical scope: network inability to confirm new transactions. [12](#0-11) 

### Likelihood Explanation

No privileges, API keys, or accounts are required. The endpoint is publicly accessible. The exploit requires only an HTTP client capable of sending POST requests with `{"estimate": true, "gas": 15000000}`. It is trivially repeatable and scriptable. A single attacker with modest bandwidth can sustain the attack indefinitely. The 500 req/s RPS limit is global (not per-IP), so one attacker can monopolize the entire allowance. [13](#0-12) 

### Recommendation

1. **Charge the gas throttle for each binary search iteration**: Pass the actual `gas` value used in each `BinaryGasEstimator.search()` iteration to `throttleManager` before calling `doProcessCall`.
2. **Add a separate, lower RPS limit for `estimate=true` requests**: Use the existing `RequestProperties` filter mechanism with `FilterField.ESTIMATE` to apply a stricter rate to estimate calls.
3. **Add per-IP rate limiting** at the load balancer or application layer to prevent a single source from consuming the global RPS budget.
4. **Do not restore gas to the bucket after the initial estimate call**: The gas restored after the initial `callContract()` in `estimateGas()` effectively resets the throttle, allowing the binary search to run for free. [5](#0-4) [14](#0-13) 

### Proof of Concept

**Preconditions:** No authentication. Any deployed contract address on the network.

**Steps:**

```bash
# Deploy or use any existing contract address
CONTRACT="0x00000000000000000000000000000000000004e2"

# Flood with estimate=true at max gas using parallel connections
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT\",\"gas\":15000000,\"estimate\":true,\"data\":\"0x\"}" &
done
wait
```

**Result:** Each of the 500 concurrent requests triggers up to 21 EVM executions. The gas throttle charges ~0 net gas (initial charge restored). The service thread pool is saturated with ~10,500 EVM executions/second. Legitimate `estimate=false` calls queue behind these and time out (default `requestTimeout=10000ms`), causing the service to stop processing new transactions. [7](#0-6) [15](#0-14)

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {

                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
