### Title
Sequential Per-Value DB Lookups for EVM Address `contract.id` Filters Enable Unauthenticated DoS Amplification

### Summary
The `extractSqlFromContractFilters()` function in `rest/controllers/contractController.js` calls `ContractService.computeContractIdFromString()` sequentially (via `await` inside a `for` loop) for every `contract.id` filter value. When a value is a CREATE2/opaque EVM address, `computeContractIdFromString()` issues a real database query via `getContractIdByEvmAddress()`. Because the system allows up to 100 repeated query parameters by default and performs no deduplication or caching of EVM address lookups, a single unauthenticated HTTP request can force up to 100 sequential database queries, amplifying resource consumption far beyond what a normal request would require.

**Root cause:** The `await` inside the `for` loop is sequential — each EVM address lookup blocks until the DB responds before the next one starts. There is no deduplication (the same EVM address repeated 100 times triggers 100 identical queries) and no caching for `getContractIdByEvmAddress()` results (the `cache` in `entityId.js` covers `parseCached`, not this path). [5](#0-4) 

**Trigger condition:** A value is treated as a CREATE2/opaque EVM address when `computeContractIdPartsFromContractIdValue()` returns an object with a `create2_evm_address` property — i.e., any 40-hex-char value (with or without `0x` prefix, with or without `shard.realm.` prefix): [6](#0-5)

### Impact Explanation

Each request with 100 EVM address `contract.id` values forces 100 sequential DB queries before a single SQL response is returned. The default DB connection pool maximum is 10 connections: [11](#0-10) 

A small number of concurrent such requests (e.g., 10) can saturate the entire connection pool, causing all other API endpoints to queue or time out (default statement timeout: 20 seconds). This is a 100× amplification factor per request with zero authentication required, meeting the ≥30% resource consumption increase threshold with only a handful of concurrent requests.

### Likelihood Explanation

- **No privileges required**: The `/api/v1/contracts` endpoint is fully public.
- **Trivial to craft**: The attacker only needs to know one valid EVM address (or can discover one via a normal `/api/v1/contracts` response) and repeat it 100 times in a single request.
- **Highly repeatable**: The attack can be automated and sustained indefinitely with a simple script.
- **No rate limiting** is visible in the codebase for this endpoint.

### Recommendation

1. **Deduplicate EVM address lookups** before the loop: collect all unique EVM address values, resolve them in a single batched query (or in parallel with `Promise.all`), then map results back.
2. **Cache `getContractIdByEvmAddress()` results** (keyed by EVM address) with a short TTL, similar to the existing `entityId` cache.
3. **Impose a tighter per-parameter repeat limit** for `contract.id` specifically (e.g., max 5–10), since the API semantics do not benefit from 100 distinct contract ID filters.
4. **Parallelize** the lookups with `Promise.all` at minimum, so they do not each hold a sequential DB connection slot.

### Proof of Concept

**Precondition:** Obtain one valid CREATE2 EVM address from the network (e.g., from a prior `/api/v1/contracts` response, field `evm_address`). Call it `ADDR`.

**Request (repeat the same valid EVM address 100 times):**
```
GET /api/v1/contracts?contract.id=eq:ADDR&contract.id=eq:ADDR&...  (100 times)
```

**What happens:**
1. `buildAndValidateFilters()` accepts all 100 values (≤ `maxRepeatedQueryParameters`).
2. `extractSqlFromContractFilters()` enters the `for` loop and sequentially `await`s `computeContractIdFromString(ADDR)` 100 times.
3. Each call hits `getContractIdByEvmAddress()` and issues `SELECT ... FROM entity WHERE evm_address = $1` to the database.
4. 100 DB queries execute sequentially before the HTTP response is sent.

**To trigger ≥30% resource increase:** Send 3–5 such requests concurrently from a single client. With a 10-connection pool, this saturates the DB pool and degrades all other API consumers.
