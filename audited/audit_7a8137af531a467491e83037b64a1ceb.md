### Title
Unenforced HTTP Scheme in Mirror Node Client Enables Network-Level MITM to Redirect Pinger to Attacker-Controlled Nodes

### Summary
`buildNetworkFromMirrorNodes()` constructs a bare `http.Client` with no TLS enforcement and no URL scheme validation, then passes it to `fetchMirrorNodeNetwork()` which issues a plaintext HTTP GET to the operator-supplied `mirrorRest` URL. An attacker with network positioning can intercept this unencrypted request and inject a crafted `nodesEnvelope` response, causing the pinger to build its entire gRPC node map from attacker-controlled endpoints and grief monitoring accuracy without any economic damage to network users.

### Finding Description
**Exact code location:**

- `pinger/mirror_node_client.go`, lines 36–72 (`buildNetworkFromMirrorNodes`) and lines 74–132 (`fetchMirrorNodeNetwork`).
- `pinger/config.go`, lines 37–38 and 133–135 (URL accepted without scheme validation).

**Root cause:**

At line 46 of `mirror_node_client.go`, the HTTP client is created as:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

This uses Go's default `http.Transport`, which happily speaks plain HTTP. No `TLSClientConfig`, no custom `DialTLS`, no redirect policy that enforces `https://`. The URL fed into this client is taken verbatim from `cfg.mirrorRest` (env `HIERO_MIRROR_PINGER_REST`), whose documented default example in `config.go` line 37 is explicitly `http://mirror-rest:5551`. There is zero scheme validation anywhere in `loadConfig()` — the only check (line 133–134) is that the value is non-empty when `network=other`.

**Exploit flow:**

1. `fetchMirrorNodeNetwork()` issues `GET http://mirror-rest:5551/api/v1/network/nodes` in plaintext (line 79–84).
2. The attacker intercepts this TCP stream (ARP poisoning on the same subnet, DNS hijacking of the `mirror-rest` hostname, or any on-path position).
3. The attacker returns a well-formed JSON body:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"attacker.ip","port":50211}]}],"links":{"next":null}}
   ```
4. `json.NewDecoder(resp.Body).Decode(&payload)` at line 96 parses this without any signature or integrity check.
5. Lines 100–124 build `network["attacker.ip:50211"] = AccountID{0,0,3}`.
6. `sdk_client.go` line 22 calls `hiero.ClientForNetwork(netmap)`, creating a gRPC client targeting the attacker's node.
7. All subsequent pinger transactions are sent to the attacker's endpoint; the attacker can return arbitrary success/failure responses, completely corrupting monitoring data.

**Why existing checks are insufficient:**

- The only response-level check (lines 90–93) validates the HTTP status code — the attacker returns `200 OK`.
- The only data check (lines 103–119) skips entries with empty `NodeAccountID` or zero port — the attacker supplies valid-looking values.
- There is no cryptographic signature on the mirror node response, no pinned certificate, no allowlist of expected node IPs, and no HTTPS requirement anywhere in the call chain.

### Impact Explanation
The pinger's entire node network is sourced exclusively from this one unauthenticated HTTP call. A successful injection replaces the legitimate Hedera node map with attacker-controlled endpoints for the lifetime of the pinger process (the network map is built once at startup in `sdk_client.go` line 18). All monitoring transactions go to the wrong nodes; the attacker can make the pinger report false latency, false success, or false failure, completely griefing monitoring accuracy. There is no economic damage to Hedera network users — impact is confined to the integrity of the monitoring system itself, consistent with the Medium/griefing severity classification.

### Likelihood Explanation
The attack requires network positioning between the pinger pod and the mirror REST service. In a Kubernetes cluster this means ARP/NDP poisoning on the pod network, a compromised CNI plugin, or DNS hijacking of the `mirror-rest` service name — all achievable by an attacker who has compromised any pod on the same node or namespace, with no application-level privileges required. If the `mirrorRest` URL points to an external HTTP endpoint (fully supported by the config), the attack surface widens to any on-path network actor (ISP, cloud provider, BGP hijacker). The attack is repeatable on every pinger restart and requires no authentication.

### Recommendation
1. **Enforce HTTPS at config load time**: In `loadConfig()` (`config.go`), after parsing `mirrorRest`, reject any URL whose scheme is not `https`:
   ```go
   if u, err := url.Parse(cfg.mirrorRest); err != nil || u.Scheme != "https" {
       return cfg, fmt.Errorf("mirror-rest must use https:// scheme")
   }
   ```
2. **Pin or verify the TLS certificate**: Supply a `tls.Config` with a specific CA pool to the `http.Transport` used in `buildNetworkFromMirrorNodes()`, preventing certificate substitution even if DNS is hijacked.
3. **Update documentation**: Change the default example in `config.go` line 37 from `http://mirror-rest:5551` to `https://mirror-rest:5551` to avoid operators inadvertently deploying with plaintext HTTP.

```go
func fetchMirrorNodeNetwork(
	ctx context.Context,
	httpClient *http.Client,
	url string,
) (map[string]hiero.AccountID, bool, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
	if err != nil {
		return nil, false, err
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
	}
	defer resp.Body.Close()

	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
		return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
	}

	var payload nodesEnvelope
	if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
		return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
	}
```
