### Title
Global Shared Opcode Rate-Limit Bucket Enables Unauthenticated DoS Against Legitimate Trace Requests

### Summary
`throttleOpcodeRequest()` consumes tokens from a single process-wide Bucket4j bucket (`opcodeRateLimitBucket`) with no per-IP, per-session, or per-user partitioning. Any unauthenticated caller can continuously exhaust this bucket at the configured rate (default: 1 req/sec), causing all legitimate users' opcode replay requests to receive HTTP 429 while the attacker's own requests succeed. This results in a complete denial of smart contract trace availability for all other users.

### Finding Description
**Code path:**

- `OpcodesController.java:61` — `throttleManager.throttleOpcodeRequest()` is called unconditionally for every request before any processing occurs.
- `ThrottleManagerImpl.java:52-55` — `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` on a single shared `Bucket` instance with no caller identity involved.
- `ThrottleConfiguration.java:47-55` — `opcodeRateLimitBucket` is a Spring singleton `@Bean` with a single `Bandwidth` of capacity `opcodeRequestsPerSecond` (default: **1**), refilled greedily at 1 token/second.

**Root cause:** The `opcodeRateLimitBucket` bean is a single JVM-wide object. `tryConsume(1)` is called identically regardless of the source IP, authenticated user, or any other client identifier. The failed assumption is that a global rate limit protects the server without enabling one client to starve all others.

**Exploit flow:**
1. Attacker observes the endpoint is enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`).
2. Attacker sends `GET /api/v1/contracts/results/<any_hash>/opcodes` with `Accept-Encoding: gzip` at exactly 1 req/sec (or any rate ≥ 1 req/sec).
3. Each attacker request passes `validateAcceptEncodingHeader()` and then calls `opcodeRateLimitBucket.tryConsume(1)`, consuming the sole available token.
4. Any concurrent or subsequent legitimate request finds the bucket empty and receives `ThrottleException("Requests per second rate limit exceeded")` → HTTP 429.
5. The attacker can use any transaction hash (including nonexistent ones); the token is consumed before `opcodeService.processOpcodeCall(request)` is reached, so the attacker's requests need not be semantically valid.

**Why existing checks are insufficient:**
- `validateAcceptEncodingHeader()` only checks for `gzip` — trivially satisfied.
- There is no IP-based rate limiting, no authentication requirement, and no per-client bucket partitioning anywhere in the call chain.
- The `request[]` filter system in `ThrottleProperties` applies only to `throttle(ContractCallRequest)` (the contract-call path), not to `throttleOpcodeRequest()`.

### Impact Explanation
Any unauthenticated attacker can render the `/opcodes` endpoint completely unavailable to all other users for as long as the attack continues. Since the default global limit is 1 req/sec, the cost to the attacker is negligible (one HTTP request per second with a gzip header). The impact is a full availability denial of smart contract opcode trace functionality — classified as unintended smart contract trace availability behavior with no direct funds at risk, matching the stated Medium scope.

### Likelihood Explanation
The attack requires zero privileges, zero valid credentials, and zero knowledge of valid transaction hashes. The only precondition is that the endpoint is enabled (`opcode.tracer.enabled=true`). The attack is trivially repeatable with a single `curl` loop or any HTTP client. It is fully deterministic: at 1 req/sec the attacker consumes 100% of the global budget indefinitely.

### Recommendation
Implement per-IP (or per-authenticated-user) rate limiting for the opcode endpoint. Options:
1. **Per-IP bucket map**: Use a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `HttpServletRequest` or a trusted `X-Forwarded-For` header), creating a per-IP bucket on first access.
2. **Reverse-proxy rate limiting**: Enforce per-IP limits at the ingress layer (e.g., nginx `limit_req_zone` or an API gateway) before requests reach the application.
3. **Combined approach**: Keep the global bucket as a server-protection ceiling AND add a per-IP bucket so no single client can exhaust the global allowance.

The global bucket alone is insufficient as a fairness or availability control when the endpoint is publicly accessible.

### Proof of Concept
```bash
# Terminal 1 — attacker: continuously consume the global token
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<host>/api/v1/contracts/results/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/opcodes"
  sleep 0.9   # slightly under 1s to stay ahead of refill
done

# Terminal 2 — legitimate user: observe 429 on every attempt
curl -v -H "Accept-Encoding: gzip" \
  "https://<host>/api/v1/contracts/results/<valid_tx_hash>/opcodes"
# Expected response: HTTP 429 {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The attacker's requests in Terminal 1 will consistently return HTTP 200 (or a non-throttle error for invalid hashes), while Terminal 2 receives HTTP 429, confirming the global bucket starvation.
