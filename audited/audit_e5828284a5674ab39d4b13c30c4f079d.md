### Title
Unauthenticated Mirror Node Network Bootstrap Enables Node Substitution via DNS/BGP Hijack

### Summary
When `network=other` is configured, `newClient()` calls `buildNetworkFromMirrorNodes()` which fetches the consensus node list over plain HTTP with no cryptographic integrity check on the response. An attacker capable of DNS poisoning or BGP hijacking the mirror REST endpoint can return a crafted node list, causing `hiero.ClientForNetwork(netmap)` to route all subsequent transactions through attacker-controlled nodes that can manipulate transaction fees.

### Finding Description
**Code path:**

- `pinger/sdk_client.go`, `newClient()`, lines 17–22: when `cfg.network == "other"`, the result of `buildNetworkFromMirrorNodes()` is passed directly and unconditionally to `hiero.ClientForNetwork(netmap)`.
- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, lines 36–72: constructs a URL from `cfg.mirrorRest` (operator-supplied, defaults to `http://mirror-rest:5551`) and calls `fetchMirrorNodeNetwork()`.
- `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–131: issues a plain `http.Client` GET with no TLS configuration, no certificate pinning, and no response signature/HMAC verification. The only check performed is the HTTP status code (lines 90–93). The JSON body is decoded and every `node_account_id` + `service_endpoints` entry is trusted verbatim (lines 100–124). The resulting `map[string]hiero.AccountID` is returned and used as the complete network topology.

**Root cause:** The HTTP client is constructed as `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}` (line 46) — no `Transport` with TLS settings, no certificate pinning, no HMAC/signature on the response body. The response is fully trusted. A grep across all pinger source confirms zero references to `TLSConfig`, `CertPool`, `x509`, `checksum`, `signature`, or any allowlist.

**Exploit flow:**
1. Attacker poisons DNS for the hostname in `cfg.mirrorRest` (e.g., `mirror-rest`) or BGP-hijacks the IP prefix of the mirror REST server.
2. The pinger's HTTP GET to `/api/v1/network/nodes` is answered by the attacker's server.
3. Attacker returns a valid-looking JSON `nodesEnvelope` where `service_endpoints` point to attacker-controlled gRPC servers and `node_account_id` values are set to legitimate-looking account IDs.
4. `fetchMirrorNodeNetwork()` parses this without any integrity check and returns the poisoned map.
5. `hiero.ClientForNetwork(netmap)` builds the SDK client using only attacker-supplied endpoints.
6. Every subsequent `cryptoTransfer.Execute(client)` in `transfer.go` (line 33) is submitted to attacker-controlled nodes.

### Impact Explanation
Attacker-controlled consensus nodes can accept transactions and return manipulated receipts, charge inflated fees, selectively drop or delay transactions, or silently redirect funds. The pinger's operator private key (`cfg.operatorKey`) signs every transaction, so the attacker receives legitimately signed transactions routed entirely through their infrastructure. Severity is **High**: confidentiality of transaction metadata and financial integrity of every transfer are compromised for the lifetime of the poisoned client instance.

### Likelihood Explanation
The `network=other` path is the only one that performs dynamic network discovery; `testnet`/`mainnet`/`previewnet` use hardcoded SDK bootstrapping. For deployments using `network=other` with an internal DNS name (the documented default `http://mirror-rest:5551`), DNS poisoning within the same cluster/VPC is a realistic internal-attacker or supply-chain scenario. BGP hijacking is a realistic ISP-level or nation-state-level threat for internet-exposed mirror endpoints. No authentication credentials or system privileges on the target are required — only network-path control.

### Recommendation
1. **Enforce HTTPS with certificate verification** for `cfg.mirrorRest` when `network=other`; reject plain `http://` URLs at config validation time (`pinger/config.go`, `loadConfig()`).
2. **Pin the mirror node TLS certificate or CA** by configuring a custom `tls.Config` with a restricted `RootCAs` pool in the `http.Client.Transport`.
3. **Validate returned node addresses** against a static allowlist of known-good account IDs or IP ranges before passing the map to `hiero.ClientForNetwork()`.
4. **Re-fetch and diff** the node list periodically; alert and refuse to update if the set changes unexpectedly.
