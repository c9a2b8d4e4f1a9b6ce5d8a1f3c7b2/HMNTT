### Title
Unauthenticated EVM Address Lookup Causes Unconditional DB Miss on Every Request in `getNftAllowances()`

### Summary
Any unprivileged user can supply a syntactically valid but non-existent EVM address as the `accountId` parameter to the NFT allowances endpoint. This forces `EntityServiceImpl.lookup()` to execute a `findByEvmAddress()` database query on every request that always misses, with no application-level rate limiting or negative-result caching in the code path. Repeated requests degrade database performance for all legitimate users.

### Finding Description

**Code path:**

`NftAllowanceServiceImpl.getNftAllowances()` (line 28) unconditionally calls `entityService.lookup(request.getAccountId())`: [1](#0-0) 

Inside `EntityServiceImpl.lookup()`, the switch dispatches on the runtime type of the parameter: [2](#0-1) 

When the input is a 40-hex-character string, `EntityIdParameter.valueOf()` resolves it to `EntityIdEvmAddressParameter` (the second branch in the parse chain): [3](#0-2) 

This causes `entityRepository.findByEvmAddress()` to be called unconditionally — a full DB read — with no caching of negative results and no application-level rate limiting anywhere in this code path. [4](#0-3) 

**Root cause / failed assumption:** The design assumes that EVM address inputs will mostly correspond to real entities. There is no negative-result cache and no guard before the DB call. Numeric ID inputs (`EntityIdNumParameter`) never touch the DB at this stage (line 32), creating an asymmetry that an attacker can exploit by always using the EVM address format.

**Why existing checks are insufficient:** `checkOwnerSpenderParamValidity()` validates owner/spender range parameters but does nothing about the `accountId` format or existence before the DB call is made. [5](#0-4) 

### Impact Explanation
Each request with a non-existent EVM address causes one guaranteed DB miss. At scale, this saturates DB connection pools and index-scan capacity, increasing query latency for all concurrent legitimate requests. The impact is proportional to request rate and requires no authentication.

### Likelihood Explanation
Preconditions: none — the endpoint is public and unauthenticated. The attacker only needs to generate syntactically valid 40-hex-character strings (trivial). The attack is fully repeatable and automatable with a simple HTTP client loop.

### Recommendation
1. **Negative-result cache:** Cache `EntityNotFoundException` results for EVM address lookups (e.g., a short-TTL Caffeine/Redis cache keyed on the EVM address) so repeated misses for the same address do not hit the DB.
2. **Application-level rate limiting:** Apply per-IP or per-client rate limiting at the controller or filter layer before `getNftAllowances()` is invoked.
3. **Fail-fast format validation:** Optionally, validate that EVM addresses conform to a known checksum or prefix before issuing the DB query.

### Proof of Concept
```
# Generate a random non-existent EVM address (40 hex chars)
ADDR="0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Flood the endpoint — each request triggers one findByEvmAddress() DB miss
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/${ADDR}/allowances/nfts" &
done
```
Each iteration hits `entityRepository.findByEvmAddress()` with a guaranteed miss, no caching absorbs the load, and the DB bears the full cost of every request.

```java
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
```
