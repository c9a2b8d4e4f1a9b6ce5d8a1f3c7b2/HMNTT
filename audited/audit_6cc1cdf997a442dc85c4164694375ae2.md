### Title
Cache Key Proliferation via Unparameterized `order` in `getSchedules()` Bypasses Redis Cache, Forcing Repeated DB Queries Without Rate Limiting

### Summary
In `rest/schedules.js`, the `order` value (`asc`/`desc`) is embedded directly into the SQL query string rather than passed as a bind parameter. Because the Redis response cache key is derived from the full request URL (including query parameters), requests with `?order=asc` and `?order=desc` always produce distinct cache keys, guaranteeing a cache miss for each variant. Since no rate limiting exists on the REST API for unauthenticated users, an attacker can continuously alternate order values (and combine with other parameter variations) to suppress caching and force three live DB queries per request indefinitely.

### Finding Description

**Code path — SQL string construction:**

`rest/schedules.js` line 68:
```js
const scheduleOrderQuery = (order) => `order by s.schedule_id ${order}`;
```
`rest/schedules.js` lines 77-79:
```js
const getSchedulesQuery = (whereQuery, order, count) => {
  return [schedulesMainQuery, whereQuery, scheduleOrderQuery(order), scheduleLimitQuery(count)].join('\n');
};
```
The `order` string is concatenated into the SQL text, not bound as a `$N` parameter. This produces two structurally distinct query strings: `… order by s.schedule_id asc …` and `… order by s.schedule_id desc …`.

**Code path — Redis cache key:**

`rest/middleware/responseCacheHandler.js` lines 151-153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The cache key is an MD5 of the full URL. `GET /api/v1/schedules?order=asc` and `GET /api/v1/schedules?order=desc` hash to different keys, so each is a separate Redis entry.

**Code path — three DB queries per cache miss:**

`rest/schedules.js` lines 241, 261-264:
```js
const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);
// …
const [{rows: entities}, {rows: signatures}] = await Promise.all([
  pool.queryQuietly(entityQuery, entityIds),
  pool.queryQuietly(signatureQuery, entityIds),
]);
```
Every cache miss fires three concurrent DB queries.

**No rate limiting on the REST API for unauthenticated users:**

`rest/server.js` lines 85-98 show only `authHandler` (which is a no-op for unauthenticated requests) and the optional Redis cache check. The `ThrottleManagerImpl` / bucket4j rate limiting found in the codebase applies exclusively to the `web3` service (contract calls), not to the REST API. The Traefik-level rate limiting in `charts/hedera-mirror-rosetta/values.yaml` applies only to the Rosetta service. No per-IP or global request-rate guard exists for `/api/v1/schedules`.

**Root cause:** The failed assumption is that the Redis cache will absorb repeated identical queries. Because `order` is part of the URL and the cache key is the full URL, two semantically equivalent queries (same data, opposite sort) are treated as entirely different cache entries. An attacker exploiting this does not need to brute-force anything — they simply toggle a documented, accepted query parameter.

### Impact Explanation
Each unauthenticated request that misses the Redis cache causes three live PostgreSQL queries against the `schedule`, `entity`, and `transaction_signature` tables. The `/api/v1/schedules` list endpoint has a default `cache-control: public, max-age=10` TTL. By alternating `?order=asc` and `?order=desc` (and combining with other accepted parameters such as `account.id`, `schedule.id`, or `limit` to create an unbounded number of unique URLs), an attacker can ensure that virtually every request is a cache miss. Under sustained concurrent load from multiple source IPs or a botnet, this saturates the PostgreSQL connection pool and I/O, degrading or crashing the mirror node REST service and its backing database without any authentication or privileged access.

### Likelihood Explanation
The attack requires zero authentication, zero knowledge of internal state, and only publicly documented query parameters. The `order` parameter is explicitly listed in the OpenAPI spec (`rest/api/v1/openapi.yml` lines 1038, 5116-5123). Any script-kiddie with `curl` or `ab` can reproduce it. The absence of per-IP rate limiting on the REST API means there is no automatic circuit breaker. The attack is trivially repeatable and scalable across distributed clients.

### Recommendation
1. **Normalize the cache key** before hashing: strip or canonicalize the `order` parameter (and other sort-only parameters that do not change the result set identity) so that `?order=asc` and `?order=desc` map to the same cache entry, with the sort applied post-retrieval or stored as metadata.
2. **Add rate limiting** to the REST API layer (e.g., via Traefik `rateLimit` middleware or an Express rate-limiter) for unauthenticated clients, consistent with what is already deployed for the Rosetta service.
3. **Parameterize the `order` value** or use a fixed set of pre-built query strings to allow PostgreSQL to reuse generic plans and reduce plan-cache churn.

### Proof of Concept
```bash
# Terminal 1 — flood with asc
while true; do
  curl -s "https://<mirror-node>/api/v1/schedules?order=asc&limit=100" > /dev/null
done

# Terminal 2 — flood with desc (always a different cache key)
while true; do
  curl -s "https://<mirror-node>/api/v1/schedules?order=desc&limit=100" > /dev/null
done

# Observe: Redis cache hit rate drops to ~0%; PostgreSQL active connections
# and query throughput spike; REST API latency increases under sustained load.

# To amplify further, add rotating schedule.id values:
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/schedules?order=asc&schedule.id=gt:$i" > /dev/null &
  curl -s "https://<mirror-node>/api/v1/schedules?order=desc&schedule.id=lt:$i" > /dev/null &
done
```

Each unique URL is a guaranteed cache miss, triggering three live DB queries. No credentials or special access are required.
