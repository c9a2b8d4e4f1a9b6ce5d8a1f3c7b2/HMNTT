### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to GET /api/v1/blocks

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` issues a database query on every request with no application-level rate limiting or per-IP concurrency control. The Node.js REST service's database pool defaults to only 10 connections. An unprivileged attacker can saturate the pool with concurrent `GET /api/v1/blocks?limit=100` requests, causing all subsequent legitimate queries — including block-time monitoring queries — to queue and time out, making the network appear to have stalled block production.

### Finding Description
**Code path:**

`rest/routes/blockRoute.js:12` → `BlockController.getBlocks` (`rest/controllers/blockController.js:101-112`) → `RecordFileService.getBlocks` (`rest/service/recordFileService.js:149-162`) → `super.getRows(query, params)` which acquires a connection from the global pool.

**Pool configuration** (`rest/dbpool.js:7-16`):
```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000 ms
  max: config.db.pool.maxConnections,                         // default 10
  statement_timeout: config.db.pool.statementTimeout,         // default 20000 ms
};
```
Default `maxConnections` is **10** (documented in `docs/configuration.md:556`).

**No rate limiting on the REST API blocks route.** Inspecting `rest/server.js:68-144`, the middleware stack applied globally is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. There is **no rate-limiting or concurrency-limiting middleware** anywhere in this chain for the blocks route.

**Limit cap is insufficient.** `extractLimitFromFilters` (`rest/controllers/blockController.js:57-61`) caps the response at `maxLimit` (default 100), but this only bounds the result set size — it does not prevent an attacker from issuing unlimited concurrent requests, each of which holds a DB connection for the duration of the query.

**Exploit flow:**
1. Attacker opens ≥10 concurrent HTTP connections and continuously issues `GET /api/v1/blocks?limit=100`.
2. Each request acquires one connection from the 10-connection pool.
3. Pool is fully saturated; all subsequent requests queue waiting for a free connection.
4. Queued requests wait up to `connectionTimeoutMillis` (20 s) before failing with a connection timeout error.
5. Legitimate monitoring clients querying block times receive errors or timeouts for the full duration of the attack.

### Impact Explanation
The mirror node's `/api/v1/blocks` endpoint is the primary interface for observing Hedera block production. When this endpoint is unavailable due to pool exhaustion, any monitoring tool or client relying on it will observe no new blocks being reported. If the last successfully returned block was produced at time T and the endpoint remains unavailable for a period significantly exceeding the 24-hour average block interval (≈2 s on mainnet), the apparent block delay as seen through this API will far exceed 500% of the average. The actual consensus network is unaffected, but the observable state through the mirror node is indistinguishable from a network halt for all API consumers.

### Likelihood Explanation
This requires zero privileges — no authentication, no special network access, no tokens. A single attacker with a modest internet connection can sustain 10+ concurrent HTTP requests indefinitely. The attack is trivially scriptable (`ab`, `wrk`, `curl` in a loop), repeatable, and requires no knowledge of the system beyond the public API endpoint. The default pool size of 10 is extremely small relative to the attack surface.

### Recommendation
1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit` or Traefik middleware similar to what is already configured for the Rosetta API in `charts/hedera-mirror-rosetta/values.yaml:149-166`) scoped to the `/api/v1/blocks` route and keyed by source IP.
2. **Increase the default connection pool size** (`hiero.mirror.rest.db.pool.maxConnections`) and document a minimum recommended value for production deployments.
3. **Add a concurrency limit** (e.g., `express-slow-down` or an in-flight request counter per IP) to prevent a single client from holding multiple pool connections simultaneously.
4. **Enable the Redis response cache** (`hiero.mirror.rest.cache.response.enabled`) so repeated identical block queries are served from cache without touching the DB pool.

### Proof of Concept
```bash
# Saturate the 10-connection pool with 20 concurrent workers, continuous requests
wrk -t20 -c20 -d60s "http://<mirror-node-host>:5551/api/v1/blocks?limit=100"

# In a separate terminal, observe legitimate queries timing out:
for i in $(seq 1 20); do
  time curl -s "http://<mirror-node-host>:5551/api/v1/blocks?limit=1" &
done
wait
# Expected: most requests return connection timeout errors or hang for ~20 seconds
```
