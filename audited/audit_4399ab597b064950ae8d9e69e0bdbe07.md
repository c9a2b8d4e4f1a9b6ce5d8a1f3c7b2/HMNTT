### Title
Unbounded Log Flooding via Rapid Subscription Open/Cancel Cycles in `subscribeTopic()`

### Summary
The `subscribeTopic()` method in `TopicMessageServiceImpl` emits an unconditional `log.info()` on every subscription entry (line 60) and another unconditional `log.info()` inside `finished()` on every subscription termination (lines 228–235), triggered via `doFinally(topicContext::finished)`. No per-client or per-IP rate limiting exists anywhere in the subscription path. An unauthenticated attacker can open and immediately cancel gRPC subscriptions in a tight loop, generating unbounded `log.info()` output and exhausting logging subsystem resources.

### Finding Description
**Code path:**

- `TopicMessageServiceImpl.java` line 60: `log.info("Subscribing to topic: {}", filter)` — fires unconditionally on every call to `subscribeTopic()`.
- `TopicMessageServiceImpl.java` line 91: `.doFinally(topicContext::finished)` — registers `finished(SignalType)` to fire on every terminal signal (COMPLETE, ERROR, or CANCEL).
- `TopicMessageServiceImpl.java` lines 227–236: `finished()` calls `log.info(...)` unconditionally, formatting subscriberId, signalType, topicId, count, stopwatch elapsed time, and computed rate.

**Root cause:** Both log statements are unconditional and execute on every subscription lifecycle event. The failed assumption is that subscriptions are long-lived and infrequent. There is no guard, counter, or rate limiter preventing a client from cycling subscriptions at arbitrary speed.

**Exploit flow:**
1. Attacker identifies any valid topic ID (or uses one with `checkTopicExists = false`).
2. Attacker opens a gRPC connection to the mirror node's gRPC port (unauthenticated, public endpoint per `ConsensusController`).
3. Attacker calls `subscribeTopic` and immediately cancels via the `setOnCancelHandler(disposable::dispose)` path (`ConsensusController.java` line 51), triggering `SignalType.CANCEL` → `finished()`.
4. Each open+cancel cycle produces exactly 2 `log.info()` calls.
5. Attacker repeats across multiple connections (no per-IP connection cap exists; only `maxConcurrentCallsPerConnection = 5` per single connection, which is trivially bypassed by opening more connections).

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java` line 14, `GrpcConfiguration.java` line 33) limits concurrent calls per connection but imposes no limit on the number of connections or the rate of sequential open/cancel cycles.
- `topicExists()` performs a DB lookup (`entityRepository.findById`), but `GrpcProperties.entityCacheSize = 50_000` indicates entity results are cached, making repeated lookups for the same topic near-instant after the first call.
- No per-IP, per-client, or global subscription rate limiter exists anywhere in the gRPC stack or service layer.
- No log-level guard (e.g., `if (log.isInfoEnabled())`) or sampling is applied to either log statement.

### Impact Explanation
Each rapid open/cancel cycle writes two formatted `log.info()` lines including string interpolation of multiple objects. At high rates (hundreds to thousands of cycles/second across multiple connections), this can: (1) exhaust available disk space if logs are written to a file appender, causing the JVM process to stall or crash; (2) saturate a synchronous log appender's I/O, blocking Reactor worker threads that execute `doFinally`; (3) overflow async log queue capacity, causing legitimate operational log entries to be silently dropped, blinding operators to real incidents. The mirror node's gRPC service is a public infrastructure component used by Hedera ecosystem clients, making availability degradation broadly impactful.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond a valid topic ID (which is public on-chain data), and only a standard gRPC client library. The cancel path is a normal gRPC client operation. A single attacker with a modest number of parallel connections can sustain the attack indefinitely. The attack is fully repeatable and requires no state.

### Recommendation
1. **Guard `log.info()` in `finished()` with a minimum elapsed time or message count threshold** — e.g., skip logging if the subscription lived less than a configurable minimum duration (e.g., 100ms) and delivered zero messages.
2. **Add per-IP or per-client subscription rate limiting** at the `ConsensusController` or service layer, rejecting new subscriptions that exceed a configurable threshold (e.g., N subscriptions per second per remote address).
3. **Replace the entry-point `log.info()` at line 60 with `log.debug()`**, or apply a rate-limited sampled logger (e.g., Slf4j's `RateLimitingLogger` or Logback's `DuplicateMessageFilter`).
4. **Cap total concurrent connections** via Netty's `maxConnectionAge` or an external load balancer rate limit, as documented in the GraphQL design doc (`contract-log-subscription.md` line 37: "We should implement rate limiting in the load balancer") — this guidance was never applied to the gRPC path.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# proto-generated stubs for ConsensusService
import consensus_service_pb2 as pb
import consensus_service_pb2_grpc as stub

TARGET = "mirror-node-grpc-host:5600"
TOPIC_ID = (0, 0, 1234)  # any valid topic shard/realm/num

def flood():
    while True:
        channel = grpc.insecure_channel(TARGET)
        client = stub.ConsensusServiceStub(channel)
        query = pb.ConsensusTopicQuery()
        query.topicID.shardNum, query.topicID.realmNum, query.topicID.topicNum = TOPIC_ID
        # Subscribe and immediately cancel
        call = client.subscribeTopic(query)
        try:
            call.cancel()  # triggers SignalType.CANCEL -> finished() -> log.info()
        except:
            pass
        channel.close()
        # Each iteration: 2x log.info() on the server

with ThreadPoolExecutor(max_workers=50) as pool:
    for _ in range(50):
        pool.submit(flood)
```

Running 50 threads each cycling open/cancel produces ~100 `log.info()` calls per iteration across all threads, with no server-side throttle to stop it.
