### Title
Cache Poisoning via Error Entry Flooding in PreparsedDocumentProvider

### Summary
The `CachedPreparsedDocumentProvider` caches `PreparsedDocumentEntry` objects keyed by raw query string, including entries representing parse/validation failures. An unauthenticated attacker can send 1,000 unique syntactically-invalid queries to fill all cache slots with error entries, evicting legitimate cached queries and forcing re-parsing on every subsequent valid request for up to one hour.

### Finding Description

**Exact code path:**

`CacheProperties.java` line 15 sets the cache spec: [1](#0-0) 

`CachedPreparsedDocumentProvider.java` line 26 caches the result of `parseAndValidateFunction` unconditionally — whether it is a valid document or an error entry: [2](#0-1) 

**Root cause:** `cache.get(key, loader)` stores whatever the loader returns. In graphql-java, `parseAndValidateFunction` returns a `PreparsedDocumentEntry` for both success and failure cases. There is no post-load filter to skip caching error entries. With `maximumSize=1000`, once 1,000 unique invalid queries are cached, Caffeine's eviction policy (Window TinyLFU) will evict lower-frequency entries — including legitimate queries that have not yet built up access frequency — to make room.

**Why existing checks are insufficient:**

- Parser limits (`maxCharacters=10000`, `maxTokens=1000`) bound query size but do not prevent 1,000 distinct invalid queries from being submitted: [3](#0-2) 

- `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` run at execution time, after the preparsed document provider has already cached the entry: [4](#0-3) 

- There is no per-IP rate limiting, authentication requirement, or allowlist on the GraphQL endpoint visible in the configuration.

### Impact Explanation
When all 1,000 cache slots are occupied by error entries, every legitimate query that was previously cached is evicted. Each subsequent valid request must re-parse and re-validate from scratch. With `expireAfterWrite=1h`, the attacker's error entries persist for up to one hour. The service remains functional but loses all caching benefit, increasing CPU and latency per request. Under sustained attack (attacker continuously submits new unique invalid queries), the cache never serves legitimate entries, constituting a sustained performance-degradation DoS. The bounded parser limits (`maxTokens=1000`, `maxCharacters=10000`) cap the per-request re-parsing cost, so this is a degradation DoS rather than a complete outage.

### Likelihood Explanation
The attack requires no credentials or special privileges — only HTTP access to the GraphQL endpoint. Generating 1,000 unique invalid queries is trivial (e.g., vary a single field name or append a unique comment: `{ invalid_field_NNNN }`). The attacker must re-flood every hour as entries expire, but this is easily automated. Caffeine's TinyLFU policy partially mitigates the attack for frequently accessed legitimate queries once they build up frequency counts, but is ineffective against a cold cache or a sustained flood that continuously introduces new unique keys.

### Recommendation
1. **Do not cache error entries.** Modify `CachedPreparsedDocumentProvider.getDocumentAsync` to check whether the returned `PreparsedDocumentEntry` has errors before storing it:
   ```java
   return cache.get(executionInput.getQuery(), key -> {
       PreparsedDocumentEntry entry = parseAndValidateFunction.apply(executionInput);
       if (entry.hasErrors()) {
           // Return without caching; do not store error entries
           return entry; // or use a separate non-caching path
       }
       return entry;
   });
   ```
   Since `AsyncCache.get` always stores the loader result, switch to `getIfPresent` + explicit `put` only on success, or use a synchronous `Cache` with conditional insertion.
2. **Add rate limiting** on the GraphQL endpoint (requests per IP per second) to limit the rate at which an attacker can fill cache slots.
3. **Reduce `maximumSize`** or add a separate small "error budget" counter per IP to reject repeated invalid queries.

### Proof of Concept
```bash
# Generate 1000 unique invalid GraphQL queries and send them
for i in $(seq 1 1000); do
  curl -s -X POST https://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ nonexistent_field_${i} }\"}" &
done
wait

# Now send a legitimate query — it will not be served from cache
# and must be re-parsed on every request for up to 1 hour
curl -X POST https://<host>/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ accounts { nodes { id } } }"}'
```
Repeat the flood loop periodically (before the 1-hour TTL expires) to maintain the attack. Observe increased response latency and CPU usage on the server compared to a baseline with a warm legitimate cache.
