### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables Full-History Partition Scan DoS

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` passes timestamp bounds derived from `Bound.getAdjustedLowerRangeValue()` and `Bound.adjustUpperBound()` directly to `findByKeyBetweenAndTimestampBetween()` without any maximum range validation. When only a lower-bound timestamp is supplied (e.g., `timestamp=gte:0`), the upper bound defaults to `Long.MAX_VALUE`, forcing the database to scan all partitions of the `hook_storage_change` table. The endpoint requires no authentication and has no rate limiting in the rest-java module.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` (line 104–130) accepts `TimestampParameter[] timestamps` with only `@Size(max = 2)` validation — no range-width constraint. It calls `Bound.of(timestamps, ...)` (line 186) and passes the result into `HookStorageRequest`.

`HookStorageRequest.isHistorical()` (line 66–68) returns `!timestamp.isEmpty()`, and `Bound.isEmpty()` returns `true` only when both `lower` and `upper` are null. Supplying a single `timestamp=gte:0` sets `lower` and leaves `upper` null, making `isHistorical()` return `true`.

`HookServiceImpl.getHookStorageChange()` (lines 97–99) then calls:
```java
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue(); // returns 0 when lower==null or gte:0
final long timestampUpperBound = timestamp.adjustUpperBound();            // returns Long.MAX_VALUE when upper==null
```

`Bound.getAdjustedLowerRangeValue()` (lines 86–97): returns `0` when `lower == null`.
`Bound.adjustUpperBound()` (lines 63–74): returns `Long.MAX_VALUE` when `upper == null`.

These values are passed verbatim to `findByKeyBetweenAndTimestampBetween()` (lines 107–114), which executes:
```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId and hook_id = :hookId
  and key >= :keyLowerBound and key <= :keyUpperBound
  and consensus_timestamp between 0 and 9223372036854775807
order by key, consensus_timestamp desc
```

The `hook_storage_change` table is partitioned by `consensus_timestamp` (migration `V2.17.1__add_hooks_support.sql`, line 47). A `consensus_timestamp between 0 and Long.MAX_VALUE` predicate eliminates all partition pruning, forcing the planner to scan every partition. Although `PageRequest` caps results at `MAX_LIMIT = 100`, the `distinct on (key) order by key, consensus_timestamp desc` pattern requires PostgreSQL to merge and sort results across all partitions before the LIMIT can be applied, making the scan cost proportional to the number of partitions × rows per partition for the given `owner_id`/`hook_id`.

**Root cause:** No maximum timestamp span is enforced anywhere between the HTTP parameter and the SQL execution. The failed assumption is that a `LIMIT` on the result set bounds the scan cost — it does not, because cross-partition `distinct on` aggregation must complete before LIMIT is applied.

### Impact Explanation

On a production network with months or years of history, `hook_storage_change` will have many time-based partitions. Each request with `timestamp=gte:0` forces a full cross-partition scan for the specified `owner_id`/`hook_id`. An attacker sending concurrent requests against active hooks (whose `owner_id` and `hook_id` are publicly visible from the `/hooks` endpoint) can saturate database I/O and CPU. Because no authentication or rate limiting exists in the rest-java module for this endpoint, the attack is freely repeatable. The resource increase exceeds 30% compared to normal narrow-range queries once the table has significant historical depth.

### Likelihood Explanation

The attack requires zero privileges. The `ownerId` and `hookId` are path parameters discoverable via the public `GET /api/v1/accounts/{ownerId}/hooks` endpoint. The exploit is a single HTTP GET request with one query parameter (`timestamp=gte:0`), trivially scriptable. It is repeatable at will with no credential or token requirement.

### Recommendation

1. **Enforce a maximum timestamp span**: In `Bound.of()` or in `HookServiceImpl.getHookStorageChange()`, reject or cap requests where `adjustUpperBound() - getAdjustedLowerRangeValue()` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Require both bounds**: Reject requests that supply only one side of the timestamp range when routing to the historical path.
3. **Add rate limiting**: Apply a per-IP or global rate limiter to the `/{hookId}/storage` endpoint in the rest-java module, analogous to the `ThrottleConfiguration` used in the web3 module.
4. **Partition pruning guard**: Ensure queries always include a timestamp upper bound that enables PostgreSQL partition pruning.

### Proof of Concept

```
# Step 1: Discover a valid ownerId and hookId (public endpoint, no auth)
GET /api/v1/accounts/0.0.1234/hooks

# Step 2: Trigger full-history scan with unbounded upper timestamp
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100

# Step 3: Repeat in a loop to amplify resource consumption
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100" &
done
wait
```

The second request produces `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE`, disabling all partition pruning on `hook_storage_change` and forcing a full cross-partition scan on every invocation.
