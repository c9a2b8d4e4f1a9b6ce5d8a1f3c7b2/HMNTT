### Title
Unauthenticated GraphQL `account()` Query Enables DB Connection Pool Exhaustion via Non-Existent EntityId Flood

### Summary
The `account(input: AccountInput!)` GraphQL endpoint accepts queries from any unauthenticated external user and performs a synchronous database lookup for every request, including those for non-existent entity IDs. The GraphQL module has no application-level rate limiting and no negative-result caching, meaning a flood of valid-format queries for non-existent `entityId` values will each consume a HikariCP database connection for the full query duration, exhausting the pool and denying service to legitimate users.

### Finding Description

**Entry point** — `graphql/src/main/resources/graphql/query.graphqls:5`: the `account` query requires no authentication. [1](#0-0) 

**Controller** — `AccountController.account()` at lines 33–44 calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` unconditionally for any syntactically valid `entityId` input. [2](#0-1) 

**Service** — `EntityServiceImpl.getByIdAndType()` at line 25 calls `entityRepository.findById(entityId.getId())` directly, with no caching layer. [3](#0-2) 

**Repository** — The GraphQL module's `EntityRepository` has no `@Cacheable` annotations at all (contrast with the `web3` module's `EntityRepository` which uses `@Cacheable(unless = "#result == null")`). Every call, including misses, hits the database. [4](#0-3) 

**No application-level rate limiting** — The GraphQL module's `GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which limit per-query structure, not request rate. There is no `ThrottleConfiguration`, `ThrottleManager`, or `Bucket4j` bean anywhere in the `graphql` module (unlike the `web3` module which has all three). [5](#0-4) 

**Infrastructure guard is insufficient** — The Helm chart configures a Traefik `inFlightReq: amount: 5` per source IP. This only caps concurrent in-flight requests per single IP; a distributed attacker using N IPs can sustain N×5 concurrent DB-holding requests, and the guard is absent in non-Helm deployments entirely. [6](#0-5) 

**Exploit flow:**
1. Attacker (no credentials needed) sends a high-volume stream of POST requests to `/graphql/alpha` with structurally valid but non-existent `entityId` values (e.g., `num: 999999999`).
2. Each request passes all parser limits, complexity/depth checks, and `@Valid` bean validation.
3. Each request reaches `entityRepository.findById()`, acquires a HikariCP connection, executes `SELECT * FROM entity WHERE id = ?`, gets an empty result, and releases the connection.
4. At sufficient concurrency (easily achieved from multiple IPs or a single IP bypassing the `inFlightReq: 5` cap via connection reuse), the HikariCP pool is saturated.
5. Legitimate queries queue and time out; the service becomes unavailable.

The team's own Prometheus alert `GraphQLHighDBConnections` (fires at 75% pool utilization) confirms DB connection exhaustion is a recognized risk surface. [7](#0-6) 

### Impact Explanation
Full denial of service for the GraphQL API. All users (including legitimate ones) are blocked when the HikariCP pool is exhausted. Because the `entity` table primary-key lookup is fast, each connection is held only briefly, but at high request rates the pool stays saturated continuously. No data is leaked or mutated, but availability is completely lost. Severity: **High** (availability impact, no privilege required, publicly reachable endpoint).

### Likelihood Explanation
Any external attacker with a script and a handful of IP addresses (or a single IP with pipelining) can trigger this. The attack requires zero knowledge of the system beyond the public GraphQL schema (discoverable via introspection or the public `graphiql` UI at `/graphiql`). It is trivially repeatable and automatable. The `inFlightReq: 5` Traefik guard is the only real barrier and is easily bypassed with distributed sources.

### Recommendation
1. **Add application-level rate limiting** to the GraphQL module, mirroring the `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl` pattern using Bucket4j, keyed per source IP extracted from the request.
2. **Cache negative results** in `EntityRepository.findById()` calls (or at the service layer) with a short TTL (e.g., 30 s) so repeated misses for the same ID do not hit the DB.
3. **Require authentication** or enforce stricter per-IP rate limits at the application layer (not just infrastructure), since the Traefik guard is deployment-dependent.
4. **Disable GraphQL introspection** in production to slow attacker reconnaissance.

### Proof of Concept

```bash
# Send 1000 concurrent requests for non-existent entity IDs (no credentials needed)
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $((RANDOM + 900000000))}}) { id }}\"}" &
done
wait
# Observe: HikariCP pool exhausted, legitimate requests return 503 or time out.
# Monitor: hikaricp_connections_active / hikaricp_connections_max approaches 1.0
```

Each request is syntactically valid, passes all existing guards, and forces a DB round-trip. No special privileges, tokens, or prior knowledge beyond the public schema are required.
