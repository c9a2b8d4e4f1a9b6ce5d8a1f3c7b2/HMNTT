### Title
Unbounded Historical Time Window via Missing `endTime` Upper-Bound Validation Enables DB Exhaustion

### Summary
`TopicMessageFilter.isValidEndTime()` only enforces that `endTime > startTime`, with no upper-bound check against the current time and no maximum window-size constraint. Combined with `startTime = 0` (epoch, which passes `@Min(0)` and `isValidStartTime()`) and `limit = 0` (no limit), an unprivileged user can open subscriptions that force `PollingTopicMessageRetriever` to issue an unbounded series of paginated DB queries spanning the entire topic history — from the Unix epoch to `Long.MAX_VALUE` nanoseconds in the future.

### Finding Description

**Validation logic (insufficient):**

`TopicMessageFilter.isValidStartTime()` at line 49–51 only checks `startTime <= DomainUtils.now()`, so `startTime = 0` (Unix epoch) is fully valid. [1](#0-0) 

`TopicMessageFilter.isValidEndTime()` at line 44–46 only checks `endTime == null || endTime > startTime`. There is no check that `endTime <= DomainUtils.now()` and no maximum window-size constraint. [2](#0-1) 

**Retrieval loop (unbounded):**

In `PollingTopicMessageRetriever.retrieve()`, when `throttled = true` (the initial historical call from `subscribeTopic`), `numRepeats = Long.MAX_VALUE` and the loop continues until `pageSize.get() < retrieverProperties.getMaxPageSize()` — i.e., until a partial page is returned. With no limit set (`limit = 0` → `hasLimit() = false`), each poll issues a DB query for up to `maxPageSize` rows, and the loop repeats until the entire time window is exhausted. [3](#0-2) [4](#0-3) 

**Subscription entry point:**

`TopicMessageServiceImpl.subscribeTopic()` calls `topicMessageRetriever.retrieve(filter, true)` directly with the user-supplied filter, after only running Jakarta Bean Validation (`@Valid`). No additional window-size or rate checks are applied. [5](#0-4) 

**Exploit flow:**
1. Attacker sends a `subscribeTopic` gRPC request with `startTime = 0`, `endTime = Long.MAX_VALUE`, `limit = 0`, and a valid `topicId`.
2. Both `isValidStartTime` and `isValidEndTime` pass — no constraint is violated.
3. `PollingTopicMessageRetriever` enters its polling loop with `numRepeats = Long.MAX_VALUE`, issuing repeated paginated DB queries from timestamp 0 to `Long.MAX_VALUE`.
4. The subscription also never terminates via `pastEndTime` until `Long.MAX_VALUE` nanoseconds have elapsed, keeping the connection and polling loop alive indefinitely.
5. Attacker opens N concurrent such subscriptions, multiplying DB load by N.

### Impact Explanation
Each subscription triggers a continuous series of full-history DB scans on the `topic_message` table. On a mature network with years of data, this can be millions of rows per subscription. With no per-user subscription cap visible in the code, an attacker can open many concurrent subscriptions, saturating DB connection pools and CPU, degrading service for all legitimate subscribers. This is a griefing/DoS with no economic damage, matching the declared scope of Medium severity.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated (no privilege required per the interface). The exploit requires only a standard gRPC client and knowledge of any valid `topicId` (which is public on-chain data). It is trivially repeatable and scriptable. The attacker does not need any special knowledge beyond the protobuf schema.

### Recommendation
1. **Add an `endTime` upper-bound check** in `isValidEndTime()`: reject `endTime` values greater than `DomainUtils.now()` plus a small configurable grace period (e.g., a few seconds).
2. **Enforce a maximum time window**: reject filters where `endTime - startTime` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
3. **Enforce a minimum non-zero limit** for historical-only queries (where `endTime != null`), or cap the total rows returned per subscription.
4. **Add a per-IP or per-connection concurrent subscription limit** enforced before entering the retrieval pipeline.

### Proof of Concept

```python
# Using grpc Python client (pseudo-code matching the HCS proto)
import grpc
from mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from mirror.api.proto import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

request = consensus_service_pb2.ConsensusTopicQuery(
    topic_id=basic_types_pb2.TopicID(topicNum=1),  # any valid topic
    consensus_start_time=timestamp_pb2.Timestamp(seconds=0, nanos=0),  # epoch
    consensus_end_time=timestamp_pb2.Timestamp(seconds=9223372036, nanos=0),  # Long.MAX_VALUE / 1e9
    limit=0  # no limit
)

# Open N concurrent subscriptions
import threading
def subscribe():
    for msg in stub.subscribeTopic(request):
        pass  # drain, keeping connection alive

threads = [threading.Thread(target=subscribe) for _ in range(50)]
for t in threads: t.start()
# Result: 50 concurrent full-history DB scan loops, each issuing repeated
# paginated queries from epoch to far future, saturating the DB.
```
