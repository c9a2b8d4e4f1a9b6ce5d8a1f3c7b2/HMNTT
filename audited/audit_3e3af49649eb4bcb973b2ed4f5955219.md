### Title
Cache Key Granularity Allows Unbounded DB Query Amplification via Unique Timestamps in `findByTimestamp()`

### Summary
`RecordFileRepository.findByTimestamp()` uses the raw caller-supplied timestamp as the Caffeine cache key. Because the underlying SQL query (`consensusEnd >= ?1 ... limit 1`) maps a wide range of timestamps to the same record file, an attacker can supply N distinct timestamps that all resolve to the same record file, bypassing the cache on every request and forcing N individual database queries. The global rate-limit bucket (500 req/s, not per-IP) is the only guard, and it is insufficient to prevent sustained DB amplification.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByTimestamp()` (line 30–32) delegates unconditionally to the repository: [1](#0-0) 

`RecordFileRepository.findByTimestamp()` is annotated with `@Cacheable` using `CACHE_MANAGER_RECORD_FILE_TIMESTAMP`. Spring's default cache key is the method argument — the raw `long timestamp` value: [2](#0-1) 

The SQL query is:
```sql
select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1
```
This means every timestamp value in the range `[consensusStart_of_file, consensusEnd_of_file]` (and any value below the next file's `consensusEnd`) resolves to the **same** record file, yet each is stored under a **different** cache key.

**Cache configuration:** [3](#0-2) 

`maximumSize(10000)` and `expireAfterAccess(10 minutes)`. An attacker who fills 10,000 slots and keeps generating new timestamps will continuously evict old entries and force new DB queries.

**Root cause:** The cache key is the input timestamp, not the resolved record file identity (e.g., `consensusEnd` or `index`). The failed assumption is that callers will only supply timestamps that exactly match a `consensusEnd`; in practice any nanosecond value in a multi-second window maps to the same file.

**Why existing checks fail:**

The global rate-limit bucket allows 500 req/s: [4](#0-3) 

It is a single shared token bucket — not per-IP, not per-timestamp range. An attacker consuming all 500 tokens/s with unique timestamps causes 500 DB queries/s with zero cache benefit.

Additionally, the gas throttle is bypassable: `scaleGas()` returns `0` for any `gas <= 10_000`, meaning the gas bucket is not consumed at all for low-gas requests: [5](#0-4) 

This lets the attacker send requests at the full 500 req/s rate without touching the gas bucket.

### Impact Explanation
At 500 req/s sustained, the attacker forces 500 uncached DB queries per second against the `record_file` table. Over a 24-hour window this is ~43 million forced DB queries. Even at a fraction of the rate limit, this significantly increases DB CPU and I/O load. Because the rate limit is global, the attack simultaneously degrades service for legitimate users. The impact satisfies the "30% increase in processing node resource consumption" threshold described in the question scope.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs to know the approximate `consensusEnd` of the latest record file, which is publicly available via the mirror node REST API (`/api/v1/blocks?limit=1&order=desc`). Generating a sequence of decrementing nanosecond timestamps (`consensusEnd - 1`, `consensusEnd - 2`, …) is trivial. The attack is fully repeatable and scriptable with a single HTTP client loop.

### Recommendation
1. **Normalize the cache key** to the resolved record file's identity. After the DB query returns a `RecordFile`, cache the result under `recordFile.getConsensusEnd()` (or `recordFile.getIndex()`), not under the raw input timestamp. A simple approach is to add a service-layer normalization step before calling the repository, or use a custom `KeyGenerator` that rounds the input timestamp down to the nearest known `consensusEnd` boundary.
2. **Add per-IP rate limiting** in addition to the global bucket, so a single client cannot consume the entire global quota.
3. **Bound the timestamp input range**: reject timestamps that are unreasonably far in the past or future, reducing the effective attack surface.

### Proof of Concept
```
# 1. Obtain the latest block's consensusEnd (publicly accessible)
CONSENSUS_END=$(curl -s "https://<mirror-node>/api/v1/blocks?limit=1&order=desc" \
  | jq '.blocks[0].timestamp.to' | tr -d '"' | tr '.' ' ' | awk '{printf "%d%09d\n",$1,$2}')

# 2. Send N requests with unique timestamps just below consensusEnd
for i in $(seq 1 10000); do
  TIMESTAMP=$((CONSENSUS_END - i))
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"block\":\"$TIMESTAMP\",\"to\":\"0x0000000000000000000000000000000000000167\",\"gas\":1000,\"data\":\"0x\"}" &
done
wait

# Each request uses a unique timestamp -> unique cache key -> unique DB query
# Result: up to 10,000 DB queries for the same record file
```
Each iteration uses a timestamp that resolves to the same latest record file but is a distinct cache key, forcing a separate `SELECT` against the `record_file` table for every request.
