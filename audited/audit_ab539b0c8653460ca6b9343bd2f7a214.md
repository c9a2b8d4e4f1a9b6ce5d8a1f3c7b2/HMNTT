### Title
Unbounded Per-Subscriber Database Polling in PollingTopicListener Enables Query Storm DoS

### Summary
When `PollingTopicListener` (type `POLL`) is active, each call to `listen()` instantiates an independent `PollingContext` and schedules a dedicated database polling loop that fires every `interval` milliseconds (default 500ms) for the lifetime of the subscription. There is no cap on concurrent subscriptions and no shared polling across subscribers, so an unprivileged attacker opening N concurrent gRPC subscriptions generates N independent database queries every 500ms, exhausting database connection pool capacity and degrading throughput for all clients.

### Finding Description
**Code path:**

- `PollingTopicListener.listen()` — [1](#0-0) 
  - Line 35 allocates a fresh `PollingContext` per caller — no sharing, no deduplication.
  - Lines 38–43 schedule an independent `Flux.defer(() -> poll(context))` loop repeating every `interval` (default 500 ms, minimum 50 ms per `@DurationMin`). [2](#0-1) 

- `TopicMessageServiceImpl.subscribeTopic()` calls `topicListener.listen(newFilter)` at line 120 with no guard on concurrent subscriber count beyond an informational `subscriberCount` gauge. [3](#0-2) 

**Root cause / failed assumption:** The design assumes a small, bounded number of concurrent subscribers when `POLL` mode is selected. There is no enforcement of that assumption — no per-IP rate limit, no maximum-subscriber check, and no shared hot publisher. The contrast with `SharedPollingTopicListener`, which calls `.share()` to multiplex all subscribers onto a single polling loop, confirms the architectural intent was never implemented for the plain `POLL` type. [4](#0-3) 

**Why existing checks fail:**
- `subscriberCount` is a Micrometer `Gauge` used only for observability; it enforces nothing. [5](#0-4) 
- `TopicMessageFilter` validation (`@Min`, `@AssertTrue`) only validates field values, not subscription rate or count. [6](#0-5) 
- `ListenerProperties` enforces a minimum interval of 50 ms but no maximum subscriber count. [7](#0-6) 

### Impact Explanation
With N concurrent subscriptions and a 500 ms interval, the database receives 2N queries per second from the listener alone, each potentially fetching up to `maxPageSize` (default 5000) rows. At modest N (e.g., 200 connections), this is 400 queries/second of full-page scans against the `topic_message` table, saturating the database connection pool and starving legitimate read traffic. Because the mirror node is a read-only service, this directly degrades message delivery latency and throughput for all subscribers, including production fee-related HCS topic consumers.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated (no credential check is visible in `TopicMessageServiceImpl` or `TopicMessageFilter`). Any network-reachable client can open arbitrarily many gRPC streams. A single attacker machine can sustain hundreds of concurrent HTTP/2 streams. The attack is trivially repeatable and requires no special knowledge beyond the topic ID. The only prerequisite is that the operator has configured `type: POLL` instead of the default `REDIS`. [8](#0-7) 

### Recommendation
1. **Enforce a maximum concurrent subscriber limit** in `TopicMessageServiceImpl.subscribeTopic()` using the existing `subscriberCount` atomic, rejecting new subscriptions with `RESOURCE_EXHAUSTED` when the limit is exceeded.
2. **Replace `PollingTopicListener` with `SharedPollingTopicListener` semantics** — use a single shared polling loop (`.share()` / `ConnectableFlux`) and fan out to per-subscriber filtered streams, eliminating the N-queries-per-interval problem entirely.
3. **Add per-source-IP connection rate limiting** at the gRPC server layer (e.g., via a `ServerInterceptor`) to prevent a single client from opening unbounded streams.
4. **Set a hard minimum `interval`** closer to 500 ms in production configuration to reduce blast radius if the above mitigations are not yet deployed.

### Proof of Concept
```
# Precondition: mirror-node grpc service configured with type: POLL
# Step 1 – open 500 concurrent subscriptions to any valid topic
for i in $(seq 1 500); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2 – observe database
# SELECT count(*), query FROM pg_stat_activity
#   WHERE query LIKE '%topic_message%' GROUP BY query;
# → 500 identical SELECT queries firing every 500ms
# → database CPU/connection pool saturated
# → legitimate subscribers experience timeouts / message delivery stalls
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
