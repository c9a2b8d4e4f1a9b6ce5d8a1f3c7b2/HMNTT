### Title
`UNION ALL` Overlap in `TokenAllowanceService.getQuery()` Produces Duplicate Records When `spender.id` Lower and Upper Bounds Share the Same Value

### Summary
When an unprivileged caller supplies `spender.id=gte:X&spender.id=lte:X` (identical boundary value) together with `token.id=gt:Y&token.id=lt:W`, the filter-decomposition logic in `getLowerFilters()` and `getUpperFilters()` independently converts the primary bound to `spender = X` in both the lower and upper sub-queries. Because `UNION ALL` retains all duplicates, every row satisfying `spender = X AND Y < token_id < W` is returned twice. No deduplication exists anywhere in the call chain, so the API response contains fabricated repeated allowance entries.

### Finding Description

**Code path:**

`getAccountTokenAllowances` (controller, line 72) → `TokenAllowanceService.getAccountTokenAllowances` → `getQuery()`. [1](#0-0) 

`getQuery()` collects non-empty filter arrays from `lower`, `inner`, `upper` and joins them with `UNION ALL`: [2](#0-1) 

**Root cause — filter generation:**

`getLowerFilters()` (line 141): when `primary.hasLower() && secondary.hasLower()`, it rewrites `primary.lower` to `operator: eq`: [3](#0-2) 

`getUpperFilters()` (line 182): always rewrites `primary.upper` to `operator: eq`: [4](#0-3) 

`getInnerFilters()` (lines 162–164): rewrites `primary.lower` to `gt` and `primary.upper` to `lt`: [5](#0-4) 

**Exploit scenario with `spender.id=gte:5&spender.id=lte:5&token.id=gt:100&token.id=lt:200`:**

| Sub-query | Generated WHERE clause |
|-----------|----------------------|
| lower | `spender = 5 AND token_id > 100` |
| inner | `spender > 5 AND spender < 5` ← always empty |
| upper | `spender = 5 AND token_id < 200` |

Any row with `spender = 5 AND 100 < token_id < 200` satisfies **both** lower and upper, so `UNION ALL` returns it twice.

**Why validation does not block this:**

`validateBoundsRange` only rejects the combination of a range filter *and* an equality filter on the same key — it does not check whether the lower and upper range values are equal: [6](#0-5) 

`validateLowerBounds` / `validateUpperBounds` only enforce that the correct operator type is present; they do not compare values: [7](#0-6) 

No deduplication is applied after the query executes: [8](#0-7) 

### Impact Explanation
The REST API `/accounts/{id}/allowances/tokens` returns duplicate `TokenAllowanceViewModel` entries for the overlapping rows. Downstream consumers (wallets, indexers, compliance tools) that iterate the response will count the same allowance grant multiple times, producing an inflated view of outstanding approvals. While no funds are moved, the integrity of the allowance ledger as presented by the mirror node is corrupted for the duration of the query, which can mislead automated systems that act on allowance data.

### Likelihood Explanation
Exploitation requires zero privileges — only a valid account ID in the URL path and crafted query parameters. The parameters `spender.id=gte:X&spender.id=lte:X` are syntactically legal, pass all server-side validation, and are accepted by the filter parser as confirmed by the existing test suite (test case "spender gte and lte, token gt and lt" at line 301). The attack is trivially repeatable with a single HTTP GET request and requires no authentication, no special knowledge of internal state beyond a target account ID, and no timing dependency. [9](#0-8) 

### Recommendation
Apply one or more of the following:

1. **Add a value-equality guard in `validateBounds`**: after confirming both `primary.lower` and `primary.upper` exist, reject the request when `primary.lower.value === primary.upper.value` (the range is degenerate and the sub-query decomposition is unsound).

2. **Replace `UNION ALL` with `UNION`** at line 74 of `tokenAllowanceService.js`. This is the minimal safe fix; PostgreSQL will deduplicate identical rows before returning results.

3. **Ensure mutual exclusivity in filter generation**: when `primary.lower.value === primary.upper.value`, collapse the three sub-queries into a single query with `spender = X AND token_id > Y AND token_id < W` rather than splitting into lower/inner/upper. [10](#0-9) 

### Proof of Concept

**Precondition:** Account `0.0.1` owns token allowances for spender `0.0.5` with token IDs 101, 150, and 199 (all between 100 and 200), and at least one token ID outside that range (e.g., 50 or 250).

**Request:**
```
GET /api/v1/accounts/0.0.1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gt:100&token.id=lt:200
```

**Generated SQL (simplified):**
```sql
(SELECT * FROM token_allowance
 WHERE owner = 1 AND amount > 0 AND spender = 5 AND token_id > 100
 ORDER BY spender ASC, token_id ASC LIMIT 25)
UNION ALL
(SELECT * FROM token_allowance
 WHERE owner = 1 AND amount > 0 AND spender > 5 AND spender < 5   -- always empty
 ORDER BY spender ASC, token_id ASC LIMIT 25)
UNION ALL
(SELECT * FROM token_allowance
 WHERE owner = 1 AND amount > 0 AND spender = 5 AND token_id < 200
 ORDER BY spender ASC, token_id ASC LIMIT 25)
ORDER BY spender ASC, token_id ASC LIMIT 25
```

**Observed response `allowances` array:**
```json
[
  {"spender": "0.0.5", "token_id": "0.0.101", ...},
  {"spender": "0.0.5", "token_id": "0.0.101", ...},  ← duplicate
  {"spender": "0.0.5", "token_id": "0.0.150", ...},
  {"spender": "0.0.5", "token_id": "0.0.150", ...},  ← duplicate
  {"spender": "0.0.5", "token_id": "0.0.199", ...},
  {"spender": "0.0.5", "token_id": "0.0.199", ...}   ← duplicate
]
```

Each allowance record in the overlap range `100 < token_id < 200` appears exactly twice, with no actual repeated grant event in the underlying database.
