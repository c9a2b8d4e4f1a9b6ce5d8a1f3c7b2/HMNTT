### Title
Integer Overflow in `poll()` via Unbounded `uint64` Limit Allows Unlimited Database Queries (DoS)

### Summary
`ConsensusController.toFilter()` passes `query.getLimit()` (a protobuf `uint64` field) directly to `TopicMessageFilter.limit` with no upper bound. In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` overflows to a negative integer when `limit >= 2^31`, causing `Math.min(negativeLimit, maxPageSize)` to produce a negative page size. This negative value is forwarded to the repository, which interprets it as no limit, executing full-table scans on every poll cycle for up to 60 seconds per connection.

### Finding Description

**Step 1 — No upper bound in `toFilter()`:** [1](#0-0) 

`query.getLimit()` is a Java `long` decoded from a protobuf `uint64` field. Any value from `1` to `Long.MAX_VALUE` (= `2^63-1`) is a valid positive Java `long` and passes the `hasLimit()` check (`limit > 0`). No cap is applied.

**Step 2 — `@Min(0)` constraint is never enforced:** [2](#0-1) 

`@Min(0)` only catches negative values; `Long.MAX_VALUE` is positive and passes. More critically, `TopicMessageServiceImpl.subscribeTopic(TopicMessageFilter filter)` has no `@Valid` annotation on the parameter, so Spring never triggers Bean Validation on the filter object at all. [3](#0-2) 

**Step 3 — Integer overflow in `poll()`:** [4](#0-3) 

With `filter.getLimit() = Long.MAX_VALUE` and `total = 0`:
- `(int)(Long.MAX_VALUE - 0)` = `(int)(0x7FFFFFFFFFFFFFFF)` = `0xFFFFFFFF` = **`-1`**
- `Math.min(-1, 1000)` = **`-1`**

The overflow persists across subsequent polls: `Long.MAX_VALUE - N` for any realistic `N` still has lower 32 bits that produce a negative `int`.

**Step 4 — Negative page size forwarded to repository:** [5](#0-4) 

`newFilter` is built with `limit = -1`. Since `hasLimit()` returns `false` for `-1` (`-1 > 0` is false), the repository's `findByFilter` applies no SQL `LIMIT` clause, returning all matching rows. In PostgreSQL, `LIMIT -1` also means no limit if the value is used directly.

**Step 5 — Polling continues for 60 seconds:** [6](#0-5) 

`isComplete()` checks `filter.getLimit() == total.get()`, which is `Long.MAX_VALUE == total` — practically never true. With `throttled=true` and `pollingFrequency=2s`, ~30 full-table scans execute before the 60-second timeout fires. [7](#0-6) 

### Impact Explanation

Each malicious subscription triggers repeated unlimited database queries for 60 seconds. With `N` concurrent connections, the database receives `N × 30` full-table scans against the topic message table. This constitutes a practical Denial-of-Service against the database backend. Additionally, all matching topic messages are streamed to the attacker, constituting unintended bulk data exfiltration beyond any operator-intended limit.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The exploit requires only a standard gRPC client (e.g., `grpcurl`) setting the `limit` field to `Long.MAX_VALUE` (`9223372036854775807`). No special privileges, tokens, or knowledge of internal state are needed. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add an upper bound in `toFilter()`** — cap the limit to a configured maximum before building the filter:
   ```java
   long rawLimit = query.getLimit();
   long cappedLimit = (rawLimit <= 0 || rawLimit > MAX_ALLOWED_LIMIT) ? MAX_ALLOWED_LIMIT : rawLimit;
   final var filter = TopicMessageFilter.builder().limit(cappedLimit);
   ```

2. **Fix the unsafe cast in `poll()`** — use `Math.min` before casting to prevent overflow:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = filter.hasLimit()
           ? (int) Math.min(remaining, Integer.MAX_VALUE)
           : Integer.MAX_VALUE;
   ```

3. **Add `@Max` constraint** to `TopicMessageFilter.limit` and ensure `@Valid` is placed on the `subscribeTopic` parameter in `TopicMessageServiceImpl`.

### Proof of Concept

```bash
# Using grpcurl against a running mirror node gRPC endpoint
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected result**: The server executes repeated unlimited SQL queries against the topic_message table every 2 seconds for 60 seconds. Database CPU/IO spikes to maximum. Repeating with multiple concurrent connections causes database saturation and service degradation for all other users.
