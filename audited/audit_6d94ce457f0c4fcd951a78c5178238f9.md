### Title
Long Overflow in `NumberRangeParameter.getInclusiveValue()` Bypasses Range Filter, Exposing All Hook Records

### Summary
When a user supplies `hook.id=gt:9223372036854775807` (Long.MAX_VALUE), `NumberRangeParameter.getInclusiveValue()` computes `Long.MAX_VALUE + 1` without overflow protection, wrapping to `Long.MIN_VALUE`. Because `Math.max(0L, Long.MIN_VALUE)` evaluates to `0`, the computed `lowerBound` silently resets to `0`, and the repository query returns every hook for the owner instead of an empty result. No privilege is required.

### Finding Description

**Root cause — `NumberRangeParameter.getInclusiveValue()`:** [1](#0-0) 

`value + 1` is plain Java `long` arithmetic with no overflow guard. When `value == Long.MAX_VALUE` (9223372036854775807), the result wraps to `Long.MIN_VALUE` (-9223372036854775808).

**Input validation — `getNumberValue()`:** [2](#0-1) 

Only rejects values `< 0`. `Long.MAX_VALUE` is non-negative, so it passes without error.

**Overflow absorbed silently in `hooksRequest()`:** [3](#0-2) 

`Math.max(0L, Long.MIN_VALUE)` = `0L`, so `lowerBound` stays at `0`. There is **no** `lowerBound > upperBound` guard here (contrast with `NetworkServiceImpl`, which throws `IllegalArgumentException` in the same situation). [4](#0-3) 

**Repository call with unconstrained range:** [5](#0-4) 

`findByOwnerIdAndHookIdBetween(ownerId, 0L, Long.MAX_VALUE, page)` — returns the entire hook dataset for that owner.

### Impact Explanation
An unauthenticated (or any unprivileged) caller can enumerate all hooks belonging to any account owner by issuing a single crafted GET request. The expected semantics (`hook.id > Long.MAX_VALUE` → empty result) are silently inverted into a full table scan. Depending on what hook data contains (e.g., smart-contract hook configurations, addresses, secrets), this constitutes an unauthorized data-disclosure vulnerability.

### Likelihood Explanation
The exploit requires only a standard HTTP GET request with a single crafted query parameter — no authentication, no special role, no race condition. It is trivially repeatable by any network-accessible client. The value `9223372036854775807` is the well-known `Long.MAX_VALUE` constant, making it an obvious fuzzing target.

### Recommendation
1. **Add overflow guard in `getInclusiveValue()`:**
   ```java
   public long getInclusiveValue() {
       if (operator == RangeOperator.GT) {
           if (value == Long.MAX_VALUE) throw new IllegalArgumentException("hook.id value out of range");
           return value + 1;
       } else if (operator == RangeOperator.LT) {
           if (value == 0) throw new IllegalArgumentException("hook.id value out of range");
           return value - 1;
       }
       return value;
   }
   ```
2. **Add a `lowerBound > upperBound` guard in `HooksController.hooksRequest()`**, mirroring the existing guard in `NetworkServiceImpl.getNetworkNodes()`.

### Proof of Concept
```
GET /api/v1/accounts/0.0.1234/hooks?hook.id=gt:9223372036854775807
```

**Step-by-step:**
1. `NumberRangeParameter.valueOf("gt:9223372036854775807")` → `operator=GT, value=9223372036854775807`
2. `getInclusiveValue()` → `9223372036854775807 + 1` → wraps to `-9223372036854775808` (Long.MIN_VALUE)
3. `Math.max(0L, -9223372036854775808L)` → `0L` — `lowerBound` unchanged
4. `HooksRequest` built with `lowerBound=0, upperBound=Long.MAX_VALUE`
5. `findByOwnerIdAndHookIdBetween(ownerId, 0, Long.MAX_VALUE, page)` executes
6. **Result:** full hook list returned instead of empty list
