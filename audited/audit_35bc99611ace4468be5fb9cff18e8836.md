### Title
Unauthenticated DoS via Unbounded Concurrent DB Queries on Token Allowances Endpoint

### Summary
The `getAccountTokenAllowances()` function in `rest/service/tokenAllowanceService.js` executes a direct PostgreSQL query for every incoming request with no rate limiting, request coalescing, or mandatory response caching at the service or middleware layer. Any unauthenticated external user can flood `GET /api/v1/accounts/{id}/allowances/tokens` with concurrent requests, exhausting the database connection pool and causing denial of service for all users of the Node.js REST API.

### Finding Description
**Exact code path:**

`rest/routes/accountRoute.js` line 18 registers the endpoint publicly with no authentication guard:
```js
router.getExt(getPath('allowances/tokens'), TokenAllowanceController.getAccountTokenAllowances);
``` [1](#0-0) 

`rest/controllers/tokenAllowanceController.js` lines 68–81 handle the request and call directly into the service:
```js
const tokenAllowances = await TokenAllowanceService.getAccountTokenAllowances(query);
``` [2](#0-1) 

`rest/service/tokenAllowanceService.js` lines 86–90 execute a raw DB query with no caching, coalescing, or throttle:
```js
async getAccountTokenAllowances(query) {
  const {sqlQuery, params} = this.getQuery(query);
  const rows = await super.getRows(sqlQuery, params);
  return rows.map((ta) => new TokenAllowance(ta));
}
``` [3](#0-2) 

`rest/service/baseService.js` line 55–57 shows `getRows()` directly calls the DB pool with no guard:
```js
async getRows(query, params) {
  return (await this.pool().queryQuietly(query, params)).rows;
}
``` [4](#0-3) 

**Why existing checks fail:**

1. **`authHandler.js` lines 18–20**: Unauthenticated requests (no `Authorization` header) return immediately and proceed to the route handler with zero restrictions. The handler only sets a custom result-count limit for authenticated users — it is not a rate limiter. [5](#0-4) 

2. **Response cache (`server.js` lines 54, 95–98)**: The cache is gated on `config.cache.response.enabled && config.redis.enabled`. This is an opt-in feature requiring Redis; it is not enabled by default. Without it, every request hits the DB. [6](#0-5) 

3. **No rate-limiting middleware exists in the Node.js REST API**: A `grep` across all `rest/**/*.js` for `rateLimit`, `throttle`, or `express-rate` returns zero matches in any middleware or route file. The `ThrottleConfiguration` and `ThrottleManagerImpl` found in the codebase are exclusively in the `web3/` Java service and do not protect the Node.js REST API. [7](#0-6) 

**Root cause:** The Node.js REST API middleware stack has no per-IP or global request-rate enforcement for unauthenticated callers. Every request to the token allowances endpoint unconditionally consumes a database connection.

### Impact Explanation
An attacker sending a sustained flood of concurrent `GET /api/v1/accounts/{id}/allowances/tokens` requests will exhaust the PostgreSQL connection pool shared by all REST API handlers. Once the pool is saturated, all subsequent queries — across every endpoint — queue indefinitely or fail, resulting in a full service outage for legitimate users. The query itself (`SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ...`) is non-trivial and involves potential `UNION ALL` of up to three sub-queries, amplifying per-request DB cost. [8](#0-7) 

### Likelihood Explanation
No authentication, API key, or network credential is required. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml` line 290). A single attacker with a modest botnet or even a single machine using async HTTP clients (e.g., `wrk`, `hey`, `k6` — the project itself ships a k6 load test at `tools/k6/src/rest/test/accountsTokenAllowance.js`) can trivially generate thousands of concurrent requests per second. The attack is repeatable, requires no special knowledge, and leaves no persistent state to clean up. [9](#0-8) 

### Recommendation
1. **Add per-IP rate limiting middleware** to the Node.js REST API using a library such as `express-rate-limit` (with a Redis store for multi-instance deployments), applied globally before route handlers in `server.js`.
2. **Enable response caching by default** (or document it as a required production configuration) so repeated identical queries are served from Redis rather than hitting the DB.
3. **Set a DB connection pool maximum** and configure a request timeout/queue limit so pool exhaustion fails fast with HTTP 503 rather than hanging indefinitely.
4. **Deploy an infrastructure-level rate limiter** (e.g., Traefik `rateLimit` middleware, which is already used for the `web3` service) in front of the Node.js REST API, mirroring the pattern already present for the Java REST service.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg1/wrk)
# Replace TARGET with the mirror node REST API host

wrk -t 16 -c 500 -d 60s \
  "https://TARGET/api/v1/accounts/0.0.1001/allowances/tokens?limit=100"
```
Expected result: after a few seconds, legitimate requests to any endpoint begin returning errors or timing out as the PostgreSQL connection pool is exhausted. The attack requires no credentials and can be launched from a single machine.
