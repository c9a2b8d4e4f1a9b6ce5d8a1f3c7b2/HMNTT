### Title
Prometheus Label Cardinality Explosion via Unbounded `route` Label in `MetricsMiddleware`

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` instruments all four Prometheus metrics — including `requestInflightGauge` — with `{method, route}` labels. The `route` label is populated by the weaveworks `middleware.Instrument` via `RouteMatcher.Match()`; when no registered route matches, the library falls back to the raw `r.URL.Path` as the label value. An unauthenticated attacker can send HTTP requests to an unbounded number of unique paths, minting a new Prometheus time-series per unique path across all four metrics, exhausting process memory and crashing the server.

### Finding Description

**Exact code location:** `rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83.

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,   // GaugeVec{method, route}
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

All four `*Vec` metrics share the `route` label (lines 26, 32, 37, 43).

**Root cause — weaveworks `Instrument.getRouteName` fallback:**
The weaveworks `middleware.Instrument.Wrap` calls an internal `getRouteName(r)` helper that:
1. Calls `RouteMatcher.Match(r, &routeMatch)` (gorilla/mux under the Rosetta SDK's `server.NewRouter`).
2. If the match succeeds and the route carries a name, returns that name (bounded — only ~15 registered Rosetta routes).
3. **If the match fails (no registered route matches the path), returns `r.URL.Path` verbatim** — the raw, attacker-controlled URL path.

The Rosetta SDK router (`server.NewRouter`, `main.go` lines 111–119 / 152) registers a fixed set of POST endpoints. It has no catch-all route. Any request to an unregistered path causes `Match()` to return `false`, triggering the `r.URL.Path` fallback.

**Exploit flow:**
1. Attacker sends HTTP requests to paths that do not match any registered Rosetta route (e.g., `POST /aaaa`, `POST /bbbb`, …).
2. For each request, `getRouteName` returns the raw path (e.g., `/aaaa`).
3. `requestInflightGauge.WithLabelValues("POST", "/aaaa").Inc()` allocates a new `*dto.Metric` + internal map entry in the Prometheus default registry.
4. The same unique `{method, route}` pair is also allocated in the three `*HistogramVec` metrics (each histogram carries 5 buckets + sum + count = 7 counters per label set).
5. Once created, Prometheus label-set entries are **never garbage-collected** within the process lifetime.
6. Repeating with N unique paths allocates O(N × 4 metrics × per-metric overhead) memory. At tens of thousands of unique paths, RSS grows to gigabytes and the Go runtime OOMs.

**Why existing checks are insufficient:**
- No authentication or rate-limiting is applied before `MetricsMiddleware` in the middleware chain (`main.go` lines 217–219: `MetricsMiddleware` wraps the entire router before any auth layer).
- No `prometheus.Labels` allowlist or cardinality cap is configured on any of the four `*Vec` registrations (`init()`, lines 46–52).
- The gorilla/mux router does not install a named catch-all/NotFoundHandler, so unmatched paths always fall through to the raw-path fallback.

### Impact Explanation
Successful exploitation exhausts the Rosetta mirror-node process memory, causing an OOM crash. Because the Rosetta API is the interface through which clients query Hashgraph history (blocks, transactions, account balances), a crash makes that history unavailable — a direct availability impact on the integrity of the historical record access layer. Recovery requires a process restart, but the attack is trivially repeatable, making sustained denial-of-service straightforward.

### Likelihood Explanation
The Rosetta port is intended to be reachable by external Rosetta clients (wallets, exchanges). No credential is required to send an HTTP request. The attack requires only a simple loop sending POST requests to unique paths — achievable with `curl`, `ab`, or a 5-line Python script. The attacker does not need to understand the Rosetta protocol; any path that does not match a registered route suffices. The attack is fully repeatable after each restart.

### Recommendation
1. **Sanitize the route label:** Override the fallback in a thin wrapper around `getRouteName` (or replace the weaveworks `Instrument` with a custom middleware) so that any unmatched path is recorded as a single fixed label value such as `"unmatched"` rather than `r.URL.Path`.
2. **Apply a Prometheus cardinality limiter:** Use `prometheus.WrapRegistererWithPrefix` combined with a cardinality-capping registerer (e.g., `github.com/prometheus/client_golang/prometheus` `NewPedanticRegistry` + a custom `Registerer` that rejects new label sets beyond a threshold).
3. **Rate-limit or firewall the Rosetta port** at the ingress layer to prevent high-volume path-flooding from unauthenticated sources.
4. **Register a named gorilla/mux `NotFoundHandler`** on the Rosetta router so that all unmatched requests resolve to a single named route (e.g., `"not_found"`), bounding cardinality to the number of registered routes + 1.

### Proof of Concept

```bash
#!/usr/bin/env bash
# Send 100,000 POST requests to unique paths against the Rosetta node.
# Each unique path mints a new Prometheus time-series.
ROSETTA_HOST="http://localhost:8080"
for i in $(seq 1 100000); do
  curl -s -o /dev/null -X POST "${ROSETTA_HOST}/exploit_path_${i}" \
       -H "Content-Type: application/json" -d '{}' &
  # Throttle to avoid local FD exhaustion; adjust as needed
  if (( i % 500 == 0 )); then wait; fi
done
wait

# Observe memory growth:
curl -s "${ROSETTA_HOST}/metrics" | grep hiero_mirror_rosetta_request_inflight | wc -l
# Expected: ~100,000 unique label combinations; process RSS grows proportionally
# until OOM kill or server becomes unresponsive.
```
