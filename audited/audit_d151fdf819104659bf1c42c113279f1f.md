### Title
Gas-Blind Throttle on Opcode Replay Endpoint Enables Resource Exhaustion via Maximum-Gas Historical Transactions

### Summary
`ThrottleManagerImpl.throttleOpcodeRequest()` enforces only a flat request-count limit (default: 1 req/sec) with no gas-proportional check, unlike the regular `throttle()` method which consults both a rate bucket and a gas bucket. An unprivileged attacker can repeatedly request opcode traces for the most gas-intensive historical transaction with all tracing options enabled (`stack=true&memory=true&storage=true`), causing long-running EVM replays to queue up and exhaust server resources.

### Finding Description

**Regular contract call throttle** (`ThrottleManagerImpl.java`, lines 37–42):
```java
if (!rateLimitBucket.tryConsume(1)) { ... }
else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { ... }
```
This checks both request rate AND gas cost.

**Opcode request throttle** (`ThrottleManagerImpl.java`, lines 52–56):
```java
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```
This checks **only** request count — the `gasLimitBucket` is never consulted.

**Call site** (`OpcodesController.java`, line 61):
```java
throttleManager.throttleOpcodeRequest();
final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
return opcodeService.processOpcodeCall(request);
```

The `processOpcodeCall` path in `OpcodeServiceImpl.java` (line 67) performs a full EVM replay:
```java
final var opcodeContext = new OpcodeContext(opcodeRequest, (int) params.getGas() / 3);
final OpcodesProcessingResult result = contractDebugService.processOpcodeCall(params, opcodeContext);
```

The gas used is taken directly from the historical transaction record — it is not user-supplied but can be up to the network maximum. The `memory` and `storage` query parameters are user-controlled and explicitly documented as causing "significant amount of time to complete." The throttle fires once at request entry and does not account for how long or how much CPU/memory the replay will consume.

**Root cause:** The failed assumption is that a flat 1-request-per-second limit is sufficient to bound resource consumption for an operation whose cost scales with the gas of the replayed transaction and the tracing options selected. The `gasLimitBucket` — which exists precisely to bound computational cost — is bypassed entirely for opcode requests.

**Why the existing check is insufficient:** At 1 req/sec, if each max-gas replay with `memory=true&storage=true` takes 30–60 seconds, the server accumulates 30–60 concurrent long-running EVM replay threads within one minute. The token bucket only gates admission; it does not limit concurrency or total in-flight work. Additionally, `opcodeRequestsPerSecond` has only a `@Min(1)` constraint with no upper bound, so operators may configure it to much higher values.

### Impact Explanation
An attacker can exhaust CPU, heap memory, and thread-pool resources on the mirror node web3 service, causing it to become unresponsive to all requests — including contract calls and gas estimation used by wallets and dApps. While this does not halt Hedera consensus directly, it takes down the mirror node's EVM execution layer, which is the primary interface for smart contract interaction queries. The response payload for a max-gas trace with stack+memory+storage can also be gigabytes in size, amplifying memory pressure even with gzip compression required.

### Likelihood Explanation
No authentication or special privilege is required — the endpoint is public. The attacker only needs to know one historical transaction hash for a high-gas contract interaction (trivially discoverable from any block explorer). The attack is repeatable, automated, and requires minimal bandwidth (GET requests with a single path parameter). The `Accept-Encoding: gzip` header requirement is a trivial one-line addition to any HTTP client.

### Recommendation
1. **Consult the gas bucket for opcode requests.** Look up the gas used by the historical transaction before executing the replay and call `gasLimitBucket.tryConsume(throttleProperties.scaleGas(gasUsed))` inside `throttleOpcodeRequest()` or at the service layer after the transaction record is fetched.
2. **Enforce a concurrency limit** (e.g., a semaphore) on simultaneous in-flight opcode replays, independent of the admission rate.
3. **Add a request timeout** for the EVM replay execution to bound worst-case resource hold time.
4. **Add an upper bound** (`@Max`) to `opcodeRequestsPerSecond` in `ThrottleProperties` to prevent misconfiguration.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service is running with opcode endpoint enabled.
- Attacker identifies a historical high-gas transaction hash (e.g., from Hashscan or any public explorer).

**Steps:**
```bash
# Step 1: Identify a max-gas historical transaction hash
TX_HASH="0x<high_gas_tx_hash>"

# Step 2: Send opcode trace requests at the allowed rate with all tracing enabled
# Each request triggers a full EVM replay; they queue up as long-running threads
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
    --output /dev/null &
  sleep 1  # Respect the 1 req/sec throttle to avoid 429
done
```

**Result:** After N seconds (where N × replay_duration >> available_threads), the server's thread pool is saturated with concurrent EVM replays.
