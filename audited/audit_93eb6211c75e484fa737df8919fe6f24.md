### Title
DB Connection Pool Exhaustion via Unauthenticated Client-Disconnect Without Query Cancellation in `getAccountTokenAllowances()`

### Summary
`getAccountTokenAllowances()` in `rest/controllers/tokenAllowanceController.js` issues a PostgreSQL query through `queryQuietly()` with no mechanism to cancel the in-flight query when the HTTP client disconnects. Because the pool is capped at 10 connections and the statement timeout is 20 seconds, an unprivileged attacker who opens 10 concurrent requests and immediately drops the TCP connections can hold all pool connections for up to 20 seconds each, denying service to legitimate users.

### Finding Description

**Code path:**

`getAccountTokenAllowances()` (controller, lines 68–81) → `TokenAllowanceService.getAccountTokenAllowances()` (service, lines 86–90) → `BaseService.getRows()` → `pool.queryQuietly()` (utils.js, lines 1481–1546).

**Root cause – `queryQuietly` has no cancellation path:**

```js
// rest/utils.js lines 1518-1526
try {
  if (!preQueryHint) {
    result = await this.query(query, params);   // ← no AbortSignal, no cancel
  } else {
    client = await this.connect();
    ...
    result = await client.query(query, params); // ← same, no cancel
  }
```

`this.query(query, params)` is the `node-postgres` Pool query. It accepts a cancellation signal only if one is explicitly passed; none is passed here. The controller never reads `req.destroyed`, never registers `req.on('close', ...)`, and never creates an `AbortController` to propagate a cancellation signal down to the DB layer.

**Failed assumption:** The design assumes the client stays connected for the duration of the query, or that the statement timeout is an acceptable worst-case. Neither assumption holds under adversarial disconnection.

**Exploit flow:**

1. Attacker opens 10 concurrent TCP connections to `GET /api/v1/accounts/<id>/allowances/tokens` (no authentication required).
2. Attacker immediately sends a TCP RST / closes the socket on each connection.
3. Express marks `req.destroyed = true` but the already-awaited `queryQuietly` promise is not cancelled — `node-postgres` has already sent the query wire message to PostgreSQL.
4. PostgreSQL continues executing the query (or waiting for a response during a partition) while holding the server-side connection.
5. The pool's `maxConnections` default is **10**; all 10 slots are occupied for up to `statementTimeout` = **20 000 ms**.
6. Any new legitimate request that needs a DB connection blocks on `pool.connect()` until `connectionTimeout` (20 000 ms) expires, then receives a 503/500.

### Impact Explanation

- **DB connection pool exhaustion:** With only 10 pool connections (default), 10 simultaneous abandoned requests fully exhaust the pool for up to 20 seconds per wave.
- **Denial of service:** All other REST endpoints that share the same pool (accounts, balances, transactions, etc.) are also starved.
- **Amplification during partition:** If a real network partition exists between the REST service and PostgreSQL, every in-flight query hangs for the full 20-second statement timeout, making each attack wave last longer and requiring fewer attacker requests to sustain the outage.
- **No data exfiltration required:** The attacker never needs to receive a response; the damage is purely resource exhaustion.

### Likelihood Explanation

- **No authentication required:** The `/accounts/:id/allowances/tokens` endpoint is publicly accessible; any internet user can reach it.
- **Trivially scriptable:** Opening 10 TCP connections and immediately closing them requires a single `curl` loop or a few lines of Python/Go.
- **Repeatable:** The attacker can re-trigger every ~20 seconds to maintain continuous pool exhaustion.
- **Low noise:** The server logs only a normal query start; no error is emitted until the statement timeout fires, making detection harder.

### Recommendation

1. **Propagate an `AbortSignal` tied to the HTTP request lifecycle into `queryQuietly`:**
   ```js
   Pool.prototype.queryQuietly = async function (query, params = [], preQueryHint, signal) {
     // pass signal to node-postgres when supported, or issue pg_cancel_backend
   ```
2. **Listen for client disconnect in the controller or a shared middleware:**
   ```js
   getAccountTokenAllowances = async (req, res) => {
     const ac = new AbortController();
     req.on('close', () => ac.abort());
     // pass ac.signal down through service → queryQuietly
   ```
3. **Reduce `statementTimeout`** to a value appropriate for this lightweight query (e.g., 5 000 ms) to limit the window of resource holding.
4. **Add per-IP or global rate limiting** on the allowances endpoint to bound the number of concurrent in-flight queries per client.

### Proof of Concept

```bash
# Open 10 connections simultaneously, immediately close each one
for i in $(seq 1 10); do
  curl -s --max-time 0.1 \
    "http://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens" \
    &
done
wait

# Within the next 20 seconds, all legitimate requests to any endpoint
# sharing the pool will time out or receive connection errors:
curl -v "http://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens"
# Expected: hangs then 500/503 (pool connection timeout)
```
