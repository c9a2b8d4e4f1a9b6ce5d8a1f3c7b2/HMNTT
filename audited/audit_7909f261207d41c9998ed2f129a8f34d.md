### Title
Numeric Account ID Existence Not Validated in `getAccountCryptoAllowances`, Returning HTTP 200 Instead of 404 for Non-Existent Accounts

### Summary
`getAccountCryptoAllowances()` in `cryptoAllowanceController.js` delegates account resolution to `EntityService.getEncodedId()`, which skips any database existence check when the input is a numeric entity ID (e.g., `9999999` or `0.0.9999999`). The encoded ID is returned directly and used as the `owner` filter in the `crypto_allowance` table query. Because no such account exists, the query returns zero rows, and the API silently responds HTTP 200 with `{"allowances": [], "links": {"next": null}}` instead of the contractually promised HTTP 404.

### Finding Description
**Code path:**

In `rest/service/entityService.js`, `getEncodedId()` (lines 118–137) branches on input type:
- **Alias** → `getAccountIdFromAlias()` → DB lookup → throws `NotFoundError` if absent.
- **EVM address** → `getEntityIdFromEvmAddress()` → DB lookup → throws `NotFoundError` if absent.
- **Numeric / shard.realm.num** → `EntityId.parseString()` → `entityId.evmAddress === null` is `true` → returns `entityId.getEncodedId()` **immediately, with no DB lookup**.

```js
// entityService.js lines 120-124
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()          // ← no existence check
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
```

Back in `cryptoAllowanceController.js` lines 76–80, the returned (unvalidated) ID is passed directly as `$1` in the `owner = $1` WHERE clause against `crypto_allowance`. For a non-existent account the table has no matching rows, so `allowances` is `[]` and the response is HTTP 200.

The `isValidAccount()` helper exists in `entityService.js` (lines 60–63) and does perform the needed `SELECT type FROM entity WHERE id = $1` check, but it is never called from this controller path.

The test fixture `rest/__tests__/specs/accounts/{id}/allowances/crypto/not-found.json` (lines 81–92) explicitly documents and accepts this behaviour — URLs like `/api/v1/accounts/3001/allowances/crypto` return `responseStatus: 200` with an empty allowances array — confirming the silent-200 path is the current production behaviour.

By contrast, `alias-not-found.json` shows that an alias resolving to no account correctly returns 404.

### Impact Explanation
Any downstream consumer (indexer, wallet, compliance tool, data exporter) that calls this endpoint with a numeric account ID cannot distinguish between:
- "Account exists but has granted no crypto allowances" (legitimate empty result), and
- "Account does not exist at all" (should be 404).

This violates the published OpenAPI contract (`rest/api/v1/openapi.yml` lines 286–287 declare a `404` response for this endpoint) and produces incorrect exported records: consumers will record an account as "active with zero allowances" when it has never existed. The inconsistency is asymmetric — aliases and EVM addresses are protected, only numeric IDs are affected — making it easy to miss in integration testing.

### Likelihood Explanation
No authentication or privilege is required. Any external user can issue `GET /api/v1/accounts/<arbitrary_number>/allowances/crypto`. The request is syntactically valid, passes all parameter validation, and always returns HTTP 200. The attack is trivially repeatable, requires no special knowledge, and is indistinguishable from a legitimate query in access logs.

### Recommendation
In `getEncodedId()` (or in `getAccountCryptoAllowances()` before the allowance query), add an existence check for the numeric-ID branch, mirroring what is already done for aliases and EVM addresses:

```js
// entityService.js – numeric branch
if (entityId.evmAddress === null) {
  const encodedId = entityId.getEncodedId();
  if (requireResult && !(await this.isValidAccount(encodedId))) {
    throw new NotFoundError();
  }
  return encodedId;
}
```

Alternatively, call `EntityService.isValidAccount(accountId)` at the top of `getAccountCryptoAllowances()` and throw `NotFoundError` before querying the allowance table. The same fix should be applied consistently to the token-allowance endpoint, which exhibits the same gap (`account-not-found.json` for tokens also documents a 200 for numeric non-existent accounts).

### Proof of Concept
**Precondition:** Mirror node is running; account `9999999` does not exist in the `entity` table.

**Steps:**
1. `curl -s "http://<mirror-node>/api/v1/accounts/9999999/allowances/crypto"`
2. Observe HTTP **200** response:
   ```json
   {"allowances":[],"links":{"next":null}}
   ```
3. For comparison, query with a non-existent alias:
   `curl -s "http://<mirror-node>/api/v1/accounts/KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ/allowances/crypto"`
4. Observe HTTP **404** — confirming the inconsistency.

The 200 response for step 2 is indistinguishable from a valid account with no allowances, silently producing an incorrect empty record for any consumer that trusts the HTTP status code.
