### Title
Unbounded Prometheus Label Cardinality via Arbitrary HTTP Method Strings in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates instrumentation to `github.com/weaveworks/common/middleware.Instrument`, which uses `r.Method` verbatim as the `"method"` Prometheus label without any allowlist or normalization. Go's `net/http` server accepts any RFC 7230-compliant token as an HTTP method, so an unprivileged attacker can send requests with an unbounded number of unique method strings, creating a new label-set (and associated histogram bucket allocations) in all four registered metrics per unique method, exhausting process memory without volumetric flooding.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83: `MetricsMiddleware` constructs a `middleware.Instrument` struct and calls `.Wrap(next)`.

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

The weaveworks `Instrument.Wrap` implementation (public source: `github.com/weaveworks/common/middleware/instrument.go`) extracts the method label directly as `r.Method` and calls `.With(prometheus.Labels{"method": method, ...})` on each `HistogramVec`/`GaugeVec`. The Prometheus client library allocates a new internal `metricWithLabelValues` entry (including all histogram bucket counters) for every unique label combination it has not seen before, and these entries are never evicted.

**Root cause:** No allowlist or normalization of `r.Method` before it is used as a Prometheus label value. The failed assumption is that the `"method"` label has bounded cardinality (only standard HTTP verbs like GET, POST, etc.).

**Why existing checks fail:**
- The four `HistogramVec`/`GaugeVec` declarations (lines 22–43) specify no cardinality cap.
- The `init()` function (lines 46–52) registers them globally with no label-value constraints.
- The `http.Server` configuration in `rosetta/main.go` (lines 220–227) sets only I/O timeouts (`ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`, `IdleTimeout`) — no rate limiting, no method validation, no connection cap.
- Go's `net/http` server does **not** reject non-standard HTTP method tokens; it stores any valid token verbatim in `r.Method`.
- The `RouteMatcher` resolves unmatched routes to a fixed label value (e.g., `"unmatched"`), so the `route` dimension is bounded, but the `method` dimension remains fully attacker-controlled.

### Impact Explanation
Each unique method string causes the Prometheus client to allocate a new label-set entry across all four metrics. `requestDurationHistogram` alone has 6 buckets × 2 additional label dimensions (`status_code`, `ws`), so each new method string allocates multiple counter objects. With tens of thousands of unique method strings, heap usage grows into gigabytes. The Go process will be killed by the OOM killer or panic with an out-of-memory error, crashing the Rosetta node. This constitutes a non-volumetric denial-of-service: a single persistent TCP connection sending sequentially unique method strings is sufficient.

### Likelihood Explanation
The Rosetta API port is network-accessible by design (it serves the Coinbase Rosetta API). No authentication is required to send HTTP requests. The attack requires only a basic HTTP client capable of sending custom method strings — trivially achievable with `curl`, `netcat`, or any scripting language. The attacker does not need to complete a valid Rosetta request; the metric label is recorded before the router dispatches the request. The attack is repeatable and can be automated to generate thousands of unique method strings per second, limited only by the server's `ReadHeaderTimeout`.

### Recommendation
1. **Normalize the method label** before it reaches Prometheus. Wrap `MetricsMiddleware` with a thin adapter that replaces `r.Method` with a sanitized value:
   ```go
   var allowedMethods = map[string]bool{
       "GET": true, "POST": true, "PUT": true,
       "DELETE": true, "PATCH": true, "HEAD": true, "OPTIONS": true,
   }
   func sanitizeMethod(m string) string {
       if allowedMethods[m] { return m }
       return "unknown"
   }
   ```
   Then pass a wrapped `http.Request` with the sanitized method to the instrumentation layer.

2. **Alternatively**, replace `middleware.Instrument` with a custom middleware that performs the allowlist check before calling `With(...)` on any `HistogramVec`.

3. **Add connection/request rate limiting** (e.g., `golang.org/x/net/netutil.LimitListener`) as defense-in-depth.

### Proof of Concept
```bash
# Generate 50,000 unique method strings and send them to the Rosetta node
for i in $(seq 1 50000); do
  METHOD=$(cat /dev/urandom | tr -dc 'A-Z' | head -c 12)
  printf "${METHOD} /network/list HTTP/1.1\r\nHost: target:8082\r\nContent-Length: 0\r\n\r\n"
done | nc target 8082
```

Each unique `METHOD` string causes the Prometheus client inside the process to allocate a new histogram label-set. After sufficient iterations, the process RSS grows unboundedly and the node crashes with OOM. No authentication, no valid Rosetta payload, and no volumetric flooding are required.
