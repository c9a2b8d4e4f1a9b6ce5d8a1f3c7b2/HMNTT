### Title
3xx Responses Silently Omitted from All Aggregate Category Counters in `metricsMiddleware`

### Summary
In `rest/middleware/metricsHandler.js`, the `res.on('finish', ...)` handler always increments `allRequestCounter` but only classifies responses into 2xx, 4xx, or 5xx buckets. Any 3xx response — including the 304 Not Modified that any unprivileged user can deliberately trigger via the `If-None-Match` / ETag mechanism in `responseCacheHandler.js` — falls through all branches unaccounted, producing a permanent, growing gap between `allRequestCounter` and the sum of all category counters exported to the mirror node.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, lines 194–204.

```js
allRequestCounter.add(1);                          // always fires
if (res.statusCode >= 200 && res.statusCode < 300) {
  allSuccessCounter.add(1);
} else if (res.statusCode >= 400 && res.statusCode < 500) {
  allClientErrorCounter.add(1);
  allErrorCounter.add(1);
} else if (res.statusCode >= 500) {
  allServerErrorCounter.add(1);
  allErrorCounter.add(1);
}
// ← no branch for 300–399
``` [1](#0-0) 

**Root cause:** The if-else chain covers `[200,300)`, `[400,500)`, and `[500,∞)`. The range `[300,400)` is entirely absent. The failed assumption is that every response will be either 2xx, 4xx, or 5xx.

**Exploit path via 304:** `rest/middleware/responseCacheHandler.js` line 54 explicitly sets `statusCode = httpStatusCodes.UNMODIFIED.code` (304) when the client's `If-None-Match` header matches the cached ETag:

```js
const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
``` [2](#0-1) 

Any unauthenticated user can:
1. `GET /api/v1/<any-cached-endpoint>` → receive `ETag: "abc123"` in the response.
2. Repeat the request with `If-None-Match: "abc123"` → server returns HTTP 304.
3. `metricsMiddleware` fires `allRequestCounter.add(1)` but no category counter is touched.

### Impact Explanation
Every 304 (and any other 3xx) response creates a permanent discrepancy: `allRequestCounter` grows while `allSuccessCounter + allClientErrorCounter + allServerErrorCounter` does not. Operators and automated alerting that rely on the exported Prometheus metrics to audit total traffic or detect anomalies receive structurally incorrect data. An attacker can deliberately amplify this gap at scale (e.g., high-frequency conditional GET loops) to obscure real traffic patterns or suppress anomaly-detection thresholds that compare category totals against the total request count.

### Likelihood Explanation
No privileges are required. The `If-None-Match` / ETag flow is standard HTTP caching behaviour available to any HTTP client. The trigger is a single extra request header on any GET to a cached endpoint. It is trivially repeatable in a loop with zero authentication.

### Recommendation
Add an explicit `else if` branch for 3xx responses in the `res.on('finish', ...)` handler, mirroring the existing pattern:

```js
} else if (res.statusCode >= 300 && res.statusCode < 400) {
  allRedirectCounter.add(1);   // new counter, or reuse allErrorCounter per policy
}
```

Alternatively, add a catch-all `else` that increments a dedicated `allUnclassifiedCounter` so that `allRequestCounter` always equals the sum of all category counters, making any future gap immediately visible.

### Proof of Concept
```bash
# Step 1 – prime the cache and capture the ETag
ETAG=$(curl -si https://<mirror-node>/api/v1/accounts \
  | grep -i '^etag:' | awk '{print $2}' | tr -d '\r')

# Step 2 – send conditional requests in a loop (no auth needed)
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "If-None-Match: $ETAG" \
    https://<mirror-node>/api/v1/accounts
done
# All responses are 304.
# allRequestCounter += 1000; all category counters += 0.
# Gap is now 1000 and growing.
```
