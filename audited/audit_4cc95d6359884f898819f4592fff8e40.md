### Title
Unauthenticated Pagination-Driven DB Connection Pool Exhaustion via `getAccounts()` Public-Key Filter Path

### Summary
The `getAccounts()` handler in `rest/accounts.js` issues a `pool.queryQuietly()` call on every request with no application-level rate limiting. When a public-key filter is present, `queryQuietly` explicitly acquires and holds a dedicated connection from the pool for the full transaction duration. Because the default pool ceiling is 10 connections and pagination `links.next` allows an attacker to sustain an unbounded stream of requests, an unprivileged attacker can exhaust all pool connections and render the REST-to-DB path non-functional for all other callers.

### Finding Description

**Exact code path:**

`getAccounts()` unconditionally calls `pool.queryQuietly()` at line 367: [1](#0-0) 

The `preQueryHint` is set to a non-falsy value whenever a public-key filter is present: [2](#0-1) 

Inside `queryQuietly`, when `preQueryHint` is truthy the implementation explicitly acquires a **dedicated** client via `this.connect()` and holds it open across three sequential round-trips (`begin; SET LOCAL …`, the main query, `commit`) before releasing it in `finally`: [3](#0-2) 

The pool is initialised with a hard ceiling of `config.db.pool.maxConnections` (documented default: **10**) and a `connectionTimeoutMillis` of 20 000 ms: [4](#0-3) [5](#0-4) 

Pagination is generated unconditionally whenever the result set equals the requested limit: [6](#0-5) 

The REST API middleware stack contains **no application-level rate limiter**. The registered middleware is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — no throttle or concurrency guard: [7](#0-6) 

**Root cause:** The failed assumption is that infrastructure-level controls (e.g. Traefik) will always be present and correctly configured in front of the Node.js process. The application layer itself provides zero defence against concurrent unauthenticated callers holding pool connections.

**Exploit flow:**

1. Attacker opens 10 concurrent HTTP connections and sends `GET /api/v1/accounts?account.publickey=<any_valid_hex>&limit=100` on each.
2. Each request enters the `preQueryHint` branch, calls `this.connect()`, and holds one of the 10 pool slots for up to 20 seconds (statement timeout).
3. All 10 pool slots are occupied; every subsequent request from any caller blocks waiting for a slot.
4. As each request completes, the attacker immediately follows the returned `links.next` URL, re-acquiring a slot before legitimate callers can.
5. The cycle repeats indefinitely with no credential, no special header, and no knowledge of the data model beyond the public API.

### Impact Explanation

With all 10 pool connections held, every other endpoint that calls `pool.queryQuietly()` — transactions, tokens, balances, contracts, topic messages — blocks for up to 20 seconds and then returns a `DbError`. From the REST layer's perspective this is operationally equivalent to a network partition with the database: no read query can complete. The impact is complete denial of service for all REST API consumers for as long as the attacker sustains the request stream.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no sophisticated tooling — a simple shell loop with `curl` following `links.next` across 10 parallel processes is sufficient. The public-key filter is documented in the OpenAPI spec, so the trigger path is discoverable. The attack is repeatable, cheap to sustain, and leaves no persistent state to clean up. Any motivated external party can execute it.

### Recommendation

1. **Add application-level concurrency/rate limiting** in the REST Node.js layer (e.g. `express-rate-limit` per IP, or a semaphore on pool acquisition) so the application is not solely dependent on infrastructure controls.
2. **Reduce the connection hold time** on the `preQueryHint` path: set `SET LOCAL random_page_cost = 0` as a session-level default for the `mirror_api` role in PostgreSQL instead of issuing it inside a per-request transaction, eliminating the need to hold a dedicated client at all.
3. **Enforce a global in-flight request cap** (e.g. `express-slow-down` or a semaphore) so that at most N requests can be simultaneously waiting for a pool connection, returning HTTP 429 beyond that threshold.
4. **Increase `maxConnections`** or front the pool with PgBouncer in transaction mode (already used for `mirror_rest` in the Helm chart) to absorb burst load without exhausting the application-side pool.

### Proof of Concept

```bash
# Terminal 1-10 (run in parallel, e.g. with GNU parallel or 10 shells)
NEXT="/api/v1/accounts?account.publickey=<any_hex_pubkey>&limit=100"
while true; do
  RESP=$(curl -s "https://<mirror-node-host>${NEXT}")
  NEXT=$(echo "$RESP" | jq -r '.links.next // empty')
  [ -z "$NEXT" ] && NEXT="/api/v1/accounts?account.publickey=<any_hex_pubkey>&limit=100"
done

# In a separate terminal, observe that all other REST calls hang for ~20 s then fail:
curl -v "https://<mirror-node-host>/api/v1/transactions?limit=1"
# Expected: delayed ~20 s, then DbError / 500
```
