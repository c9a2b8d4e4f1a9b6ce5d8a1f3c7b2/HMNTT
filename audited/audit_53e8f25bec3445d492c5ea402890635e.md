### Title
Unauthenticated GraphQL DoS via Sustained High-Complexity Queries Triggers GraphQLHighCPU Alert with No Application-Level Rate Limiting

### Summary
The GraphQL module exposes the `account(input: AccountInput!)` query with a complexity ceiling of 200 (so complexity 199 is always accepted) and contains **zero application-level rate limiting**, unlike the web3 module which uses bucket4j throttling. An unprivileged attacker operating from multiple IPs can sustain high-concurrency, near-maximum-complexity queries, push per-pod CPU above 80% for over 5 minutes, fire the `GraphQLHighCPU` Prometheus alert (severity: critical), and induce operator-initiated pod restarts that cause service unavailability.

### Finding Description

**Code path and root cause:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 exposes `account(input: AccountInput!)` as the sole public query. [1](#0-0) 

`GraphQlConfiguration.java` line 43 sets `MaxQueryComplexityInstrumentation(200)`, meaning any query with complexity ≤ 199 is fully executed without rejection. [2](#0-1) 

A grep across the entire `graphql/` module for `rateLimit`, `throttle`, or `RateLimit` returns **no matches**. The web3 module has a full `ThrottleConfiguration` with bucket4j buckets, but the GraphQL module has no equivalent. [3](#0-2) 

The only per-IP protection is the Traefik middleware `inFlightReq: amount: 5` with `ipStrategy.depth: 1`. [4](#0-3) 

`depth: 1` means Traefik reads the **first** IP from the `X-Forwarded-For` header. If the upstream load balancer does not strip/validate that header, an attacker can rotate spoofed source IPs in the header to bypass the per-IP concurrency cap entirely. Even without header spoofing, a distributed attack from multiple IPs trivially bypasses a per-IP limit of 5.

The retry middleware (`attempts: 3`) compounds the problem: if the overloaded server returns 5xx responses, Traefik retries each request up to 3 times, multiplying backend load by up to 4×. [5](#0-4) 

HPA is **disabled by default** (`hpa.enabled: false`), so no horizontal auto-scaling absorbs the load. [6](#0-5) 

PodDisruptionBudget is also **disabled by default** (`podDisruptionBudget.enabled: false`), so operator-initiated restarts can take all pods offline simultaneously. [7](#0-6) 

The `GraphQLHighCPU` Prometheus rule fires when `process_cpu_usage / system_cpu_count > 0.8` for 5 minutes, with `severity: critical` and `area: resource`. [8](#0-7) 

The alertmanager inhibit rules that would suppress downstream alerts are **disabled by default** (`inhibitRules.enabled: false`). [9](#0-8) 

**Exploit flow:**

1. Attacker crafts a valid `account(input: {entityId: {num: N}})` query with field selection expanded to complexity 199 (just under the 200 hard limit).
2. Attacker sends this query at high concurrency from multiple IPs (or with rotated `X-Forwarded-For` values), bypassing the `inFlightReq: 5` per-IP cap.
3. Each accepted request hits `AccountController.account()` → `EntityServiceImpl.getByIdAndType()` → database query. [10](#0-9) 
4. With no rate limiting, the JVM processes all concurrent requests, saturating CPU.
5. After 5 minutes above 80%, `GraphQLHighCPU` fires as a critical alert.
6. Operators restart pods; with PDB disabled, all replicas can be simultaneously unavailable.

### Impact Explanation

The direct impact is a sustained denial-of-service against the GraphQL API. The `GraphQLHighCPU` alert carries `severity: critical`, which in most on-call workflows triggers immediate operator intervention including pod restarts. With `podDisruptionBudget.enabled: false` and `hpa.enabled: false`, a single-replica deployment (the default `minReplicas: 1`) has no redundancy during a restart, causing a complete service outage. The `GraphQLNoPodsReady` alert would then also fire, compounding the incident. This is a resource-exhaustion DoS that requires no authentication and no privileged access.

### Likelihood Explanation

The attack requires only HTTP access to `/graphql/alpha` and the ability to send requests from multiple IPs (a basic botnet, cloud VMs, or Tor exit nodes). The query structure is publicly documented in the Postman collection. No credentials, tokens, or insider knowledge are needed. The absence of application-level rate limiting in the GraphQL module (in contrast to the web3 module, which has it) means the attacker faces no server-side throttle. The retry middleware amplification makes the attack more efficient. This is a realistic, repeatable attack for any motivated external party.

### Recommendation

1. **Add application-level rate limiting to the GraphQL module** mirroring the web3 `ThrottleConfiguration` pattern — implement a bucket4j `rateLimitBucket` and enforce it in a `WebGraphQlInterceptor` before query execution.
2. **Enable HPA** (`hpa.enabled: true`) so the deployment scales out under load before CPU thresholds are breached.
3. **Enable PodDisruptionBudget** (`podDisruptionBudget.enabled: true`, `minAvailable: 1`) to prevent all pods from being restarted simultaneously.
4. **Validate `X-Forwarded-For` at the ingress/load-balancer layer** before it reaches Traefik, so `ipStrategy.depth: 1` cannot be bypassed by header injection.
5. Consider reducing `MaxQueryComplexityInstrumentation` below 200 or applying per-IP complexity budgets.

### Proof of Concept

```bash
# From N distinct IPs (or with spoofed X-Forwarded-For headers):
for i in $(seq 1 50); do
  curl -s -X POST https://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -d '{
      "query": "{ account(input: {entityId: {num: 1000}}) { alias autoRenewPeriod balance createdTimestamp declineReward deleted entityId { shard realm num } expirationTimestamp id key maxAutomaticTokenAssociations memo nonce pendingReward receiverSigRequired stakePeriodStart timestamp { from to } type } }"
    }' &
done
# Repeat in a loop for >5 minutes.
# Expected: GraphQLHighCPU Prometheus alert fires; operators restart pods; service unavailable.
```
