### Title
Cache Thrashing via Unbounded Timestamp-Keyed Entries in `SystemFileLoader.load()` for Non-Exchange-Rate System Files

### Summary
The `CacheKey` record in `SystemFileLoader` uses both `fileId` and the raw nanosecond `consensusTimestamp` as equality components for all system files routed to `defaultSystemFileCacheManager` (address books, throttle definitions, HAPI permissions). Unlike exchange-rate files which have their timestamp rounded to the hour before caching, these files receive no such bucketing, meaning every unique nanosecond timestamp produces a distinct cache entry. An unauthenticated attacker can trivially fill the `maximumSize=20` cache with 21+ unique timestamps, causing continuous LRU evictions that force a DB re-query on every subsequent legitimate request.

### Finding Description

**Exact code path:**

In `web3/src/main/java/org/hiero/mirror/web3/state/SystemFileLoader.java`, the `load(FileID fileId, long consensusTimestamp)` method at lines 111–120 routes files to one of two caches:

```java
var cacheManager = defaultSystemFileCacheManager;          // line 111

if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp); // only these 3 files get bucketed
}

final var cacheKey = new CacheKey(fileId, consensusTimestamp); // line 120
```

The `roundDownToHour` call at line 117 is applied **only** to `exchangeRateFileId`, `feeSchedulesFileId`, and `simpleFeeSchedulesFileId`. All other system files — address books (101, 102), throttle definitions, HAPI permissions — fall through to `defaultSystemFileCacheManager` with the **raw nanosecond timestamp** as the cache key.

The `CacheKey` record at line 276 uses both fields for equality:
```java
private record CacheKey(FileID fileId, long timestamp) {}
```

The `defaultSystemFileCacheManager` is configured via `CacheProperties.systemFile` at line 45 of `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`:
```
"expireAfterWrite=10m,maximumSize=20,recordStats"
```

**Root cause:** The assumption that `maximumSize=20` is sufficient is only valid if the timestamp space is bounded (as it is for exchange-rate files via hour-bucketing). For the remaining system files, the timestamp is unbounded nanosecond precision, so an attacker controls the cardinality of cache keys directly.

**Exploit flow:**
1. Attacker sends 21 contract-call requests (e.g., `eth_call`) each with a distinct historical block number, mapping to 21 distinct nanosecond timestamps.
2. Each cache miss at lines 129–132 falls through to `loadFromDB()` → `fileDataRepository.getFileAtTimestamp()` at line 173.
3. Each result is stored via `cache.put(cacheKey, result)` at line 138, filling all 20 slots and triggering LRU eviction of the first entry.
4. Attacker repeats with a new unique timestamp every few seconds, keeping the cache perpetually full of attacker-controlled historical entries.
5. Every legitimate request for the current block's system files is a cache miss, forcing a DB query on each call.

**Why existing checks fail:**
- The global rate limit of 500 RPS (`ThrottleProperties.requestsPerSecond`, line 35 of `ThrottleProperties.java`) is far too high to prevent this — the attacker only needs 21 requests to saturate the cache, and ~2 requests per minute to maintain the attack as entries expire after 10 minutes.
- There is no per-IP, per-user, or per-timestamp-range throttle.
- No authentication is required for the web3 JSON-RPC endpoint.
- The `expireAfterWrite=10m` TTL does not prevent the attack; it only means the attacker must refresh entries every 10 minutes, which is trivial.

### Impact Explanation
System files (address books, throttle definitions, HAPI permissions) are loaded during EVM execution on every contract call. With the cache permanently thrashed, every contract call incurs at least one additional `fileDataRepository.getFileAtTimestamp()` DB query per affected system file. With 3 files in `defaultSystemFileCacheManager` and a sustained call rate, this multiplies DB query volume significantly. On a node processing hundreds of contract calls per second, this can easily exceed a 30% increase in DB-layer resource consumption. The attack is sustained and self-reinforcing: the attacker's requests themselves count toward the 500 RPS budget, leaving less headroom for legitimate traffic.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send JSON-RPC `eth_call` requests with historical block numbers, which is a standard public API. The attacker needs only 21 requests to saturate the cache and ~2 requests per 10 minutes to maintain it. This is well within the 500 RPS global rate limit. The attack is fully automatable, repeatable, and requires no special knowledge beyond the public API specification.

### Recommendation
Apply the same `roundDownToHour` (or a coarser bucketing strategy) to all system files routed to `defaultSystemFileCacheManager`, not just the three exchange-rate/fee-schedule files. Alternatively:
- Increase `maximumSize` to a value that accommodates the realistic range of historical timestamps seen in production (e.g., keyed by day or week rather than raw nanoseconds).
- Add a per-source-IP rate limit specifically on historical block queries.
- Cap the accepted `consensusTimestamp` to a recent window (e.g., reject timestamps older than N hours for system-file lookups).

### Proof of Concept
```python
import requests, time

NODE_URL = "http://<mirror-node-web3>:8545"
CONTRACT = "0x0000000000000000000000000000000000000167"  # any system contract

# Fill cache with 21 distinct historical block numbers
for block_offset in range(21):
    block_hex = hex(1_000_000 + block_offset)  # 21 distinct historical blocks
    payload = {
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT, "data": "0x"}, block_hex],
        "id": block_offset
    }
    requests.post(NODE_URL, json=payload)

# Maintain attack: send one new unique historical block every 30 seconds
# to keep the cache full of attacker-controlled entries
i = 22
while True:
    block_hex = hex(1_000_000 + i)
    payload = {
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT, "data": "0x"}, block_hex],
        "id": i
    }
    requests.post(NODE_URL, json=payload)
    i += 1
    time.sleep(30)
# Result: every legitimate eth_call for the current block misses the system-file
# cache and triggers a DB query for address books / throttle definitions / HAPI permissions.
```
