### Title
Unauthenticated Request Flooding of Active Schedule Endpoint Causes DB Resource Exhaustion

### Summary
The `getScheduleById` handler in `rest/schedules.js` executes an aggregating SQL query with two LEFT JOINs and a `json_agg` on every request for an active (non-executed, non-expired) schedule. For active schedules, the response cache-control header is intentionally omitted (empty `{}`), resulting in either no caching or a 1-second TTL. No application-level rate limiting exists for unauthenticated callers, allowing any external user to flood the endpoint and exhaust database connection/CPU resources.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `getScheduleById`, lines 115–128: [1](#0-0) 

Every request unconditionally executes `getScheduleByIdQuery`: [2](#0-1) 

This query performs:
- A `LEFT JOIN entity e ON e.id = s.schedule_id`
- A `LEFT JOIN transaction_signature ts ON ts.entity_id = s.schedule_id`
- A `json_agg(... ORDER BY ts.consensus_timestamp)` aggregation
- A `GROUP BY s.schedule_id, e.id`

**Root cause — cache bypass for active schedules:**

`getScheduleCacheControlHeader` returns an empty object `{}` (no `cache-control` header) when the schedule is active: [3](#0-2) 

The response cache update handler only caches when `ttl > 0`, derived from the `cache-control` header. With no header, the fallback `DEFAULT_REDIS_EXPIRY = 1` (one second) applies at best: [4](#0-3) [5](#0-4) 

Furthermore, the entire response cache is conditional on both `config.cache.response.enabled` AND `config.redis.enabled`: [6](#0-5) 

If Redis is not deployed, there is zero caching for any request.

**No application-level rate limiting for unauthenticated users:**

`server.js` registers no rate-limiting middleware: [7](#0-6) 

`authHandler` only sets a custom query limit for authenticated users; unauthenticated requests pass through with no restriction: [8](#0-7) 

The Helm chart for the REST service defines only `circuitBreaker` and `retry` in its middleware chain — **no `rateLimit` or `inFlightReq`** entries — unlike the Rosetta chart which explicitly includes both: [9](#0-8) 

Additionally, `global.middleware` defaults to `false`, meaning even the Traefik middleware chain may not be applied: [10](#0-9) 

### Impact Explanation
An attacker flooding `GET /api/v1/schedules/{activeScheduleId}` forces repeated execution of the aggregating JOIN query against the PostgreSQL pool. Schedules with many signers produce large `json_agg` results, amplifying per-query cost. This can exhaust the DB connection pool (`maxConnections`), spike CPU on the database host, and degrade or deny service to all other API consumers. No economic damage to network participants occurs, consistent with the medium/griefing severity classification.

### Likelihood Explanation
The endpoint is fully public — no authentication, no API key, no CAPTCHA. A single attacker with a basic HTTP flood tool (e.g., `wrk`, `ab`, `hey`) targeting one valid active schedule ID can sustain thousands of requests per second. The schedule ID is easily discoverable via `GET /api/v1/schedules`. The attack is trivially repeatable and requires no special knowledge or privilege.

### Recommendation
1. **Add per-IP rate limiting** at the application layer (e.g., `express-rate-limit`) for all unauthenticated callers on the `/api/v1/schedules/:scheduleId` route.
2. **Add `inFlightReq` and `rateLimit` Traefik middleware** to the REST Helm chart, mirroring the Rosetta chart configuration.
3. **Increase the minimum cache TTL for active schedules** — even a 5–10 second `max-age` on active schedule responses would dramatically reduce DB fan-out under flood conditions, with negligible staleness impact.
4. **Set a DB statement timeout** (`statementTimeout`) low enough to shed runaway queries before they pile up in the pool.

### Proof of Concept
```bash
# 1. Discover a valid active schedule ID
SCHEDULE_ID=$(curl -s "https://<host>/api/v1/schedules?limit=1" \
  | jq -r '.schedules[0].schedule_id')

# 2. Flood the endpoint (no auth required)
wrk -t 10 -c 200 -d 60s \
  "https://<host>/api/v1/schedules/${SCHEDULE_ID}"

# Expected result:
# - DB CPU spikes to 100%
# - Connection pool exhausted (pool errors in REST logs)
# - Other API endpoints begin returning 503 or timing out
# - wrk reports near-zero error rate (200 OK responses until DB saturates)
```

const getScheduleCacheControlHeader = (schedule) => {
  const nowNs = utils.nowInNs();
  const executedTimestamp = schedule.executed_timestamp;
  const expirationTime = schedule.expiration_time;
  const consensusTimestamp = schedule.consensus_timestamp;

  const hasExecuted = executedTimestamp !== null || schedule.deleted;
  const hasAutoExpired =
    expirationTime === null && nowNs >= consensusTimestamp + constants.THIRTY_ONE_MINUTES * constants.NANOS_PER_SECOND;
  const hasExpired =
    expirationTime !== null && nowNs >= expirationTime + constants.SIXTY_SECONDS * constants.NANOS_PER_SECOND;

  if (hasExecuted || hasAutoExpired || hasExpired) {
    return LONGER_SCHEDULE_CACHE_CONTROL_HEADER;
  }
  return {};
};

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
