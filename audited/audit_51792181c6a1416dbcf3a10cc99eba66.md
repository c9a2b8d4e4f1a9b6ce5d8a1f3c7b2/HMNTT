### Title
Unauthenticated Connection Pool Exhaustion via Timestamp-Filtered Single Account Lookup

### Summary
An unauthenticated external user can send concurrent `GET /api/v1/accounts/<id>?timestamp=gte:X&timestamp=lte:Y` requests to `getOneAccount()` in `rest/accounts.js`. Each request with timestamp filters issues up to three DB queries (one sequential, two parallel), consuming up to two simultaneous connections from a default pool of only 10. With no rate limiting on this endpoint, five or more concurrent requests exhaust the pool, causing all subsequent REST API requests to queue or time out for up to 20 seconds, degrading the mirror node's REST processing capacity.

### Finding Description

**Exact code path:**

`getOneAccount()` in `rest/accounts.js` lines 396–512 executes the following DB round-trips when `timestampFilters.length > 0`:

**Round-trip 1 (sequential, awaited):** `balances.getAccountBalanceTimestampRange()` in `rest/balances.js` lines 167–197 issues a query against `account_balance` for the treasury account within the optimized timestamp range. [1](#0-0) [2](#0-1) 

**Round-trips 2 & 3 (parallel via `Promise.all`):** The entity query (which, when no balance snapshot is found, becomes a `UNION ALL` of `entity` and `entity_history`) and the transactions query are dispatched concurrently. [3](#0-2) [4](#0-3) 

The `UNION ALL` path is triggered when `accountBalanceQuery.forceUnionEntityHistory = true` (line 463), which occurs whenever `getAccountBalanceTimestampRange()` returns no snapshot — a common case for historical or sparse timestamp ranges. [5](#0-4) 

**Root cause — no rate limiting, tiny pool:**

The `authHandler` middleware only sets a custom response `limit` for authenticated users; it does not rate-limit or reject unauthenticated requests. [6](#0-5) 

No other rate-limiting middleware exists for the accounts route in `server.js`. [7](#0-6) 

The connection pool is configured with a default of **10 connections** and a **20-second statement timeout**. [8](#0-7) 

**`validateRange=false` bypasses timestamp range size checks:**

`parseTimestampFilters` is called with `validateRange=false`, so the `maxTimestampRange` guard (default 7 days) is not enforced for this endpoint. [9](#0-8) [10](#0-9) 

`getOptimizedTimestampRange()` does bound the balance query to at most two monthly partitions, but this only affects round-trip 1; the entity UNION ALL and transactions queries still execute against the full timestamp range. [11](#0-10) 

### Impact Explanation

With a default pool of 10 connections and each request holding 2 simultaneous connections during the parallel phase, only **5 concurrent requests** are needed to exhaust the pool. Once exhausted, all subsequent REST API requests (accounts, transactions, tokens, etc.) queue waiting for a connection for up to 20 seconds (`connectionTimeoutMillis: 20000`), effectively making the REST API unresponsive. Since the REST API is the primary public interface of the mirror node, this constitutes degradation of ≥30% of mirror node processing capacity.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond a valid account ID (trivially obtained from the public API), and no brute force. A single attacker with a basic HTTP client can sustain 5–10 concurrent requests indefinitely. The endpoint is publicly reachable on port 5551 by default. The attack is repeatable and automatable with minimal tooling (e.g., `curl`, `ab`, `wrk`).

### Recommendation

1. **Add rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied before route handlers, targeting at minimum the `/api/v1/accounts/:id` route.
2. **Enforce `validateRange`** in `parseTimestampFilters` for `getOneAccount()` — pass `validateRange=true` so the `maxTimestampRange` (7-day default) is enforced, bounding query cost.
3. **Increase the default connection pool** (`maxConnections`) or add a per-request concurrency limit (e.g., a semaphore) to prevent a single burst from exhausting the pool.
4. **Reduce `statementTimeout`** for read-only REST queries to limit how long a connection can be held per query.

### Proof of Concept

```bash
# Obtain any valid account ID from the public API
ACCOUNT_ID=$(curl -s https://<mirror-node>/api/v1/accounts?limit=1 | jq -r '.accounts[0].account')

# Exhaust the 10-connection pool with 6 concurrent long-running requests
for i in $(seq 1 6); do
  curl -s "http://<mirror-node>:5551/api/v1/accounts/${ACCOUNT_ID}?timestamp=gte:0&timestamp=lte:9999999999" &
done

# All subsequent REST API requests now queue/timeout for up to 20 seconds
curl -v "http://<mirror-node>:5551/api/v1/accounts/${ACCOUNT_ID}"
# Expected: delayed response or connection timeout
```

The `timestamp=gte:0&timestamp=lte:9999999999` range is accepted because `validateRange=false` skips the `maxTimestampRange` check, maximizing query duration and connection hold time.
