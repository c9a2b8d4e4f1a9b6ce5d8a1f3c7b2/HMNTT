### Title
Cache Bypass via Missing Timestamp Rounding for `hapiPermissionFile` Enables Unbounded DB Query Amplification

### Summary
The `hapiPermissionFile` entry in `SystemFileLoader` is registered with `codec=null` and is not subject to the hourly timestamp rounding applied to exchange-rate/fee-schedule files. Because `getCurrentTimestamp()` returns the current wall-clock time in nanoseconds, every request produces a unique `CacheKey`, making the `defaultSystemFileCacheManager` (max 20 entries) completely ineffective. An unprivileged attacker flooding the `/api/v1/contracts/call` endpoint forces one DB query per request with no caching benefit.

### Finding Description

**Root cause — missing timestamp normalization:**

In `load(FileID, long)`, only three file IDs get their timestamp rounded to the hour:

```java
// SystemFileLoader.java lines 111–116
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);
}
``` [1](#0-0) 

`hapiPermissionFile` is not in this list, so its raw nanosecond timestamp is used as-is.

**`getCurrentTimestamp()` returns nanosecond wall-clock time:**

```java
// Utils.java lines 57–60
public static long getCurrentTimestamp() {
    final var now = Instant.now();
    return DomainUtils.convertToNanos(now.getEpochSecond(), now.getNano());
}
``` [2](#0-1) 

Every call at a different nanosecond produces a different value, so every request creates a unique `CacheKey(hapiPermissionFileId, <nanosecond>)`.

**Cache is structurally bypassed:**

The `defaultSystemFileCacheManager` is configured as `expireAfterWrite=10m,maximumSize=20`. With nanosecond-granular keys, the 20-entry cache fills immediately and evicts continuously, providing zero reuse. [3](#0-2) 

**`codec=null` eliminates retry overhead:**

```java
// SystemFileLoader.java lines 176–179
var codec = systemFile.codec;
if (codec != null) {
    codec.parse(bytes.toReadableSequentialData());
}
``` [4](#0-3) 

For `hapiPermissionFile` (`codec=null`), any non-empty DB row is accepted immediately with no retry. This guarantees exactly one DB query per request — predictable and efficient for the attacker. [5](#0-4) 

**Call path from external request:**

`FileReadableKVState.readFromDataSource()` calls `systemFileLoader.load(key, getCurrentTimestamp())` for every system file lookup during EVM execution. [6](#0-5) 

### Impact Explanation
Every contract-call request that triggers a `hapiPermissionFile` lookup results in one unconditional `fileDataRepository.getFileAtTimestamp()` DB query. At the documented global rate limit of 500 RPS, this produces 500 extra DB queries/second for this single file, compounding with all other per-request DB queries (entity, contract state, token lookups). Sustained flooding degrades or exhausts DB connection pool capacity, causing service-wide latency spikes or outright failures for all users of the mirror node web3 API.

### Likelihood Explanation
No authentication or special privilege is required — any HTTP client can call `/api/v1/contracts/call`. The attack is trivially repeatable with standard load-testing tools (e.g., `wrk`, `hey`, `ab`). The attacker does not need to control or forge timestamps; the nanosecond clock guarantees cache-key uniqueness automatically. The 500 RPS global throttle is the only barrier, and it is itself the amplification vector.

### Recommendation
Apply the same hourly (or at minimum per-second) timestamp rounding to **all** system files that use `defaultSystemFileCacheManager`, not only the three exchange-rate/fee-schedule files. A minimal fix:

```java
// In load(FileID, long), extend the rounding to all system files:
consensusTimestamp = roundDownToHour(consensusTimestamp);
// or use a coarser granularity (e.g., per-second) for files that change rarely
```

Alternatively, for files like `hapiPermissionFile` that are expected to be static or near-static, return the genesis/default value directly (as is already done for `networkPropertyFile`) without any DB lookup.

### Proof of Concept
1. Deploy the mirror node web3 service with default configuration.
2. Send 500 concurrent requests per second to `POST /api/v1/contracts/call` with any valid contract call payload that triggers EVM execution (causing `FileReadableKVState` to load `hapiPermissionFile`).
3. Observe via DB metrics (e.g., `pg_stat_activity`) that `getFileAtTimestamp` queries for the `hapiPermissionFile` entity ID are issued on every request with no cache hits.
4. Observe that the `systemFile` cache hit rate (exposed via `recordStats`) remains at or near 0% for the `hapiPermissionFile` key.
5. Escalate request rate to saturate the DB connection pool, causing timeouts for all concurrent users.
