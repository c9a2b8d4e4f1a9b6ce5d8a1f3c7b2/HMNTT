### Title
Unauthenticated `Content-Length` Header Injection Poisons `api_request_size_bytes` Histogram in `metricsMiddleware`

### Summary
`metricsMiddleware` in `rest/middleware/metricsHandler.js` records the raw, user-supplied `Content-Length` header value directly into `requestSizeHistogram` without validating it against the actual received body size or enforcing any upper bound. Any unauthenticated caller can send a crafted `Content-Length` header with an arbitrarily large integer to inflate the `api_request_size_bytes` metric for any route, including the public staking rewards endpoint, corrupting size-based monitoring and masking true traffic patterns.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware`, line 209:
```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```
**Root cause:** The code unconditionally trusts the `Content-Length` HTTP header, which is fully attacker-controlled. `parseInt` parses any integer up to `Number.MAX_SAFE_INTEGER` (9,007,199,254,740,991) without error. The `|| 0` guard only replaces `NaN` (non-numeric strings); it does not cap or reject large values. No comparison against the actual received bytes (e.g., via `req.socket.bytesRead` or body-parser's measured size) is performed.

**Exploit flow:**
1. Attacker sends repeated GET requests to `/api/v1/accounts/0.0.1001/rewards` (public, no auth) with header `Content-Length: 9007199254740991`.
2. `metricsMiddleware` fires on `res.finish`, reads `req.headers['content-length']` = `"9007199254740991"`, parses it to `9007199254740991`, and records that value into `requestSizeHistogram` with labels `{method: "GET", path: "/api/v1/accounts/{idOrAliasOrEvmAddress}/rewards", code: "200"}`.
3. The histogram buckets for `api_request_size_bytes` are now dominated by the injected value, making the metric useless for detecting anomalous real request sizes.

**Why existing checks fail:**
- `|| 0` only handles `NaN`; large integers pass through unchanged.
- No middleware (e.g., `express.json()` body size limit) validates `Content-Length` against actual body bytes for GET requests.
- The metrics endpoint itself is protected by `authenticate()`, but the instrumentation path (line 209) runs for all non-metrics requests with no privilege check.

### Impact Explanation
The `api_request_size_bytes` histogram is permanently skewed for any targeted route. Operators relying on this metric for anomaly detection (e.g., alerting on unusual request sizes to the staking rewards endpoint as a reconnaissance signal) will see noise that masks real traffic patterns. While `allRequestCounter` and `requestTotalCounter` remain accurate, size-based alerting and dashboards are rendered unreliable. Severity is **Medium**: no data exfiltration or service disruption, but integrity of observability infrastructure is broken by an unprivileged attacker.

### Likelihood Explanation
Preconditions: none — the staking rewards endpoint is fully public. The attack requires only the ability to send HTTP requests with custom headers, which any HTTP client supports. It is trivially scriptable and repeatable at high frequency. No rate limiting on the metrics recording path is present.

### Recommendation
Replace the header-trust pattern with measurement of actual received bytes:
```javascript
// Option 1: use req.socket.bytesRead delta (measure before/after)
// Option 2: accumulate chunks in a data listener, same as responseSize
// Option 3: clamp/validate the header value
const contentLength = parseInt(req.headers['content-length'] ?? '0', 10);
const safeSize = Number.isFinite(contentLength) && contentLength >= 0 && contentLength <= MAX_ALLOWED ? contentLength : 0;
requestSizeHistogram.record(safeSize, labels);
```
The most robust fix is to measure actual body bytes received (mirroring the `responseSize` accumulation pattern already used for `res.write`/`res.end`) rather than trusting the declared header.
