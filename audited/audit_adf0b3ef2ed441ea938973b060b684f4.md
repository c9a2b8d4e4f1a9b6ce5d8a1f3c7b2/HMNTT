### Title
Identity Check Bypass in `findByBlockType` via Numeric `Long.MAX_VALUE` Block Input Causes Latest Block Lookup Failure

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `BlockType.LATEST` sentinel. `BlockType.of("9223372036854775807")` (or `"0x7fffffffffffffff"`) creates a new `BlockType` instance with `number = Long.MAX_VALUE` that is not the singleton, silently bypassing the LATEST branch and routing to `findByIndex(Long.MAX_VALUE)`, which returns `Optional.empty()` since no block with that index exists. Any unprivileged caller can trigger this via the public `/api/v1/contracts/call` endpoint.

### Finding Description

**Exact code path:**

`BlockType.java` lines 9–12 define the singleton and the record type: [1](#0-0) 

`BlockType.of()` routes named strings to the singleton but delegates all numeric input to `extractNumericBlock()`: [2](#0-1) 

`extractNumericBlock()` calls `Long.parseLong` and unconditionally constructs a **new** `BlockType` instance — even when the parsed value equals `Long.MAX_VALUE`: [3](#0-2) 

`findByBlockType()` uses `==` (reference equality) to detect LATEST. The new instance is a different object on the heap, so the check evaluates to `false`: [4](#0-3) 

Execution falls through to `findByIndex(Long.MAX_VALUE)`, which queries the DB for a record file with `index = 9223372036854775807`. No such row exists, so `Optional.empty()` is returned.

**Root cause:** The failed assumption is that `BlockType.of()` always returns the singleton for any input that semantically means "latest." It does not — numeric inputs always produce new instances regardless of their value.

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` with JSON body containing `"block": "9223372036854775807"`.
2. `BlockTypeDeserializer.deserialize()` calls `BlockType.of("9223372036854775807")`: [5](#0-4) 
3. `extractNumericBlock` returns `new BlockType("9223372036854775807", Long.MAX_VALUE)` — a distinct heap object.
4. `ContractController` passes this `BlockType` into `ContractExecutionParameters.block`: [6](#0-5) 
5. Downstream service calls `findByBlockType(block)` → identity check fails → `findByIndex(Long.MAX_VALUE)` → `Optional.empty()`.
6. Contract execution cannot resolve the block context and fails.

**Existing checks are insufficient:** The `BlockTypeTest` test at line 19 explicitly documents and accepts that `BlockType.of(String.valueOf(Long.MAX_VALUE))` produces a non-singleton instance with `number = Long.MAX_VALUE`, confirming the bypass is a known but unguarded code path: [7](#0-6) 

### Impact Explanation
Any contract call or `eth_estimateGas` request that specifies the block as the decimal or hex string equivalent of `Long.MAX_VALUE` will fail to resolve the latest block state, returning an error instead of executing against the current chain head. This means an attacker can reliably cause all such requests to fail, and any caller that does not strictly use the string `"latest"` but instead uses the numeric block number (a common pattern in Ethereum tooling) will receive incorrect/empty results rather than the true latest Hashgraph state.

### Likelihood Explanation
The endpoint is unauthenticated and publicly accessible. The exploit requires a single HTTP POST with a crafted `block` field value. No special privileges, tokens, or chain state are required. The attack is trivially repeatable and automatable. Ethereum tooling that converts `"latest"` to its numeric equivalent (`Long.MAX_VALUE` / `0x7fffffffffffffff`) would trigger this unintentionally.

### Recommendation
Replace the reference equality checks in `findByBlockType` with value-based equality, or normalize the `BlockType` at construction time. The simplest fix is to change the identity checks to use `.equals()` or to compare `block.number()` directly:

```java
if (block.number() == BlockType.EARLIEST.number() && block.name().equals("earliest")) {
    return recordFileRepository.findEarliest();
} else if (block.number() == Long.MAX_VALUE) {
    return recordFileRepository.findLatest();
}
```

Alternatively, normalize in `extractNumericBlock`: if the parsed value equals `Long.MAX_VALUE`, return the `LATEST` singleton instead of creating a new instance.

### Proof of Concept

```bash
curl -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "from": "0x0000000000000000000000000000000000000001",
    "to":   "0x0000000000000000000000000000000000000002",
    "gas":  50000,
    "block": "9223372036854775807"
  }'
```

**Expected (correct) behavior:** Request executes against the latest block (same as `"block": "latest"`).

**Actual behavior:** `findByBlockType` bypasses the LATEST branch, calls `findByIndex(9223372036854775807)`, receives `Optional.empty()`, and the request fails with a block-not-found error — despite the numeric value being identical to `Long.MAX_VALUE` used by `BlockType.LATEST`.

The same bypass works with the hex equivalent: `"block": "0x7fffffffffffffff"`.
