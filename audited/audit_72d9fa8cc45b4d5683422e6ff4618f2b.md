### Title
Race Between `doOnCancel`/`doOnComplete` in `unsubscribe()` Orphans Newly-Inserted Shared Flux, Causing Persistent Message Loss

### Summary
`RedisTopicListener.unsubscribe()` calls `topicMessages.remove(topic.getTopic())` — a key-only removal — from both `doOnCancel` and `doOnComplete` hooks on the same Flux. Between the two calls, a concurrent `computeIfAbsent` can insert a brand-new Flux F2 for the same key, which the second `unsubscribe()` call then silently removes. F2 is orphaned: it holds a live Redis subscription but is unreachable via the map, so all subsequent subscribers receive no messages and each creates yet another leaked connection.

### Finding Description

**Exact code path:**

`getSharedListener` (line 61) uses `computeIfAbsent` to lazily create and cache a shared Flux per topic: [1](#0-0) 

`subscribe` (lines 68-80) attaches **both** `doOnCancel` and `doOnComplete` to the same Flux, both pointing to `unsubscribe`: [2](#0-1) 

`unsubscribe` performs a blind key-only removal with no identity check: [3](#0-2) 

**Root cause:** `ConcurrentHashMap.remove(key)` removes *whatever value is currently mapped to that key*, not the specific Flux instance that triggered the hook. There is no `remove(key, expectedValue)` guard.

**Exploit flow (race interleaving):**

```
Thread A (F1 upstream completes):
  doOnComplete fires → unsubscribe() → topicMessages.remove("topic.X")
  → F1 removed from map

Thread B (new gRPC subscriber):
  computeIfAbsent("topic.X", ...) → key absent → subscribe() → F2 created
  → F2 inserted into map

Thread A (F1 cancel signal races with complete, or share() cleanup):
  doOnCancel fires → unsubscribe() → topicMessages.remove("topic.X")
  → F2 removed from map  ← WRONG: removes the newly inserted F2, not F1
```

After this sequence, F2 is orphaned: it holds an active Redis channel subscription but is no longer reachable via `topicMessages`. Every subsequent call to `computeIfAbsent` creates F3, F4, … each also subject to the same race, leaking Redis connections and delivering no messages.

**Why existing checks are insufficient:**

`ConcurrentHashMap` guarantees atomicity of *individual* operations, but the sequence `remove` → `computeIfAbsent` → `remove` is not atomic. The `.share()` operator does not prevent both `doOnCancel` and `doOnComplete` from firing: in Reactor, when the upstream completes while the last downstream subscriber is concurrently cancelling, both terminal signals can be delivered on different scheduler threads, each independently invoking `unsubscribe()`. [4](#0-3) 

### Impact Explanation

All subscribers to the affected topic stop receiving messages silently — no error is surfaced to clients. The orphaned Flux continues consuming a Redis pub/sub connection indefinitely (the `retryWhen` loop keeps it alive), leaking resources. Repeated triggering exhausts Redis connection pool slots, degrading the entire mirror-node gRPC service for all topics.

### Likelihood Explanation

An unprivileged gRPC client needs only to:
1. Subscribe to any topic (no authentication required beyond a valid topic ID).
2. Cancel the stream (disconnect or send a gRPC cancel).
3. Immediately re-subscribe.

Step 2 triggers `doOnCancel`; if the Redis upstream completes or errors concurrently (e.g., Redis restart, channel expiry, or network blip), `doOnComplete` also fires. The attacker does not need to control Redis directly — they only need to repeat the subscribe/cancel/re-subscribe cycle at high frequency to widen the race window. Under load this is reliably reproducible.

### Recommendation

Replace the key-only removal with a value-guarded removal so that `unsubscribe` only removes the map entry if it still points to the *same Flux instance* that registered the hook:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Flux<TopicMessage>[] self = new Flux[1];
    self[0] = container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> topicMessages.remove(topic.getTopic(), self[0]))
            .doOnComplete(() -> topicMessages.remove(topic.getTopic(), self[0]))
            ...
            .share();
    return self[0];
}
```

`ConcurrentHashMap.remove(key, value)` is atomic and only removes the entry when the value matches, preventing the second call from evicting a newly inserted Flux.

### Proof of Concept

**Preconditions:** Mirror node running with `listenerProperties.type = REDIS`; Redis available; no authentication on the gRPC endpoint.

**Steps:**

1. Open gRPC client A and subscribe to `topic.1` → F1 is created and inserted into `topicMessages`.
2. Open gRPC client B and subscribe to `topic.1` → returns the same F1 (shared).
3. Client A and B both cancel simultaneously → `doOnCancel` fires → `topicMessages.remove("topic.1")` → F1 removed.
4. Client C immediately subscribes to `topic.1` → `computeIfAbsent` → F2 created and inserted.
5. Trigger a Redis connection blip (or wait for the upstream to emit `onComplete` from a channel expiry) so that `doOnComplete` fires for F1 → `topicMessages.remove("topic.1")` → **F2 is removed**.
6. Client C receives no further messages. All subsequent subscribers create new Fluxes that are also subject to the same race.
7. Repeat steps 1–6 in a tight loop to exhaust Redis connections.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
