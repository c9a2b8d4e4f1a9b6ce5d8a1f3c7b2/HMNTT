### Title
Missing `deleted is not true` Filter in `entity_history` Branch of `findActiveByEvmAddressAndTimestamp` Causes Deleted Entities to Appear Active in Historical Queries

### Summary
`findActiveByEvmAddressAndTimestamp()` in `EntityRepository.java` applies `deleted is not true` only to the `entity` branch of its `UNION ALL`, but omits this filter from the `entity_history` branch. When a user submits a historical `eth_call` with a `blockTimestamp` at or after an entity's deletion, the `entity` branch is correctly suppressed, but the `entity_history` branch returns the most recent pre-deletion snapshot (which has `deleted = false`), causing the deleted entity to be surfaced as active to all mirror node consumers. The sibling method `findActiveByIdAndTimestamp()` correctly applies `deleted is not true` to both branches, confirming this is an unintentional omission.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`) and lines 95–121 (`findActiveByEvmAddressOrAliasAndTimestamp`).

**Root cause:**

The `entity_history` branch (lines 75–81) has no `deleted is not true` predicate:

```sql
-- entity branch (line 70): correctly filtered
where e.deleted is not true
and e.id = (select id from entity_cte)

-- entity_history branch (lines 76-80): NO deleted filter
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
order by lower(eh.timestamp_range) desc
limit 1
```

Contrast with `findActiveByIdAndTimestamp` (lines 144–150), which correctly applies the filter to both branches:

```sql
-- entity_history branch here has the guard:
and deleted is not true
```

**Data model mechanics:**

When an entity is deleted at timestamp `T_del`:
- The `entity` table record is updated: `deleted = true`, `timestamp_range = [T_del, null)`
- The prior state is archived to `entity_history`: `deleted = false`, `timestamp_range = [T_created, T_del)`

**Exploit flow for a query at `blockTimestamp = T_query` where `T_query >= T_del`:**

1. `entity_cte` resolves the entity ID from `entity` (no deletion filter in the CTE — it finds the ID regardless of deletion status).
2. `entity` branch: `deleted is not true` fails → zero rows.
3. `entity_history` branch: `lower([T_created, T_del)) = T_created <= T_query` → the pre-deletion snapshot (`deleted = false`) is returned. No filter blocks it.
4. Outer `ORDER BY timestamp_range DESC LIMIT 1` picks this `entity_history` row.
5. The function returns the deleted entity as if it were active.

**Why existing tests do not catch this:**

The test `findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall` (test file line 126–132) uses `entityHistory.getTimestampLower() - 1` as the `blockTimestamp`, which is *before* the history record's lower bound, so the `entity_history` branch returns nothing. The actual bug fires when `blockTimestamp >= entityHistory.getTimestampLower()`, a case with no test coverage.

### Impact Explanation

Any caller of `findActiveByEvmAddressAndTimestamp` — primarily `CommonEntityAccessor.getEntityByEvmAddressTimestamp()` (lines 80–84) and `getEntityByEvmAddressAndTimestamp()` (lines 66–71) — will receive a non-empty `Optional<Entity>` for a deleted entity. This propagates into the EVM execution layer for historical `eth_call` requests:

- A deleted smart contract appears to have code and is treated as callable, producing incorrect execution results.
- A deleted account appears to have its last-known balance and key material, corrupting historical balance/state queries.
- All mirror node consumers relying on these historical responses receive incorrect data, breaking the integrity guarantee of the mirror node's historical state export.

Severity: **Medium–High**. The integrity of historical state exported to mirror node consumers is directly violated. While this does not allow fund theft on-chain, it corrupts the authoritative historical record that downstream systems (explorers, indexers, dApps) rely on.

### Likelihood Explanation

**Preconditions (all achievable by an unprivileged external user):**
1. Know the EVM address of any previously deleted contract or account (publicly observable on-chain).
2. Know any block number whose consensus timestamp is at or after the deletion timestamp (also publicly observable).

**Trigger:** Submit an `eth_call` JSON-RPC request with `blockNumber` set to any block at or after the entity's deletion. No authentication, no special keys, no privileged access required.

**Repeatability:** Fully deterministic and repeatable for any deleted entity. The Hedera network has many deleted contracts and accounts, making targets easy to find.

### Recommendation

Add `and eh.deleted is not true` to the `entity_history` branch of both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- In findActiveByEvmAddressAndTimestamp (line 76-80):
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    and eh.deleted is not true          -- ADD THIS
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

Apply the identical fix to `findActiveByEvmAddressOrAliasAndTimestamp` (lines 109–118). Add a regression test covering the scenario: entity deleted at `T_del`, `entity_history` has pre-deletion record with `deleted = false`, query at `T_query > T_del` must return empty.

### Proof of Concept

**Setup:**
```sql
-- Entity created at T1=100, deleted at T2=200
INSERT INTO entity (id, evm_address, deleted, timestamp_range, created_timestamp, ...)
  VALUES (42, '\xDEADBEEF...', true, '[200,)', 100, ...);

INSERT INTO entity_history (id, evm_address, deleted, timestamp_range, created_timestamp, ...)
  VALUES (42, '\xDEADBEEF...', false, '[100,200)', 100, ...);
```

**Trigger (unprivileged RPC call):**
```json
{
  "jsonrpc": "2.0",
  "method": "eth_call",
  "params": [
    { "to": "0xDEADBEEF..." },
    "0x12C"
  ]
}
```
(Block `0x12C` = 300, whose consensus timestamp T_query=300 > T_del=200.)

**Expected result:** Error or empty — entity is deleted.

**Actual result:** `findActiveByEvmAddressAndTimestamp('\xDEADBEEF...', 300)` returns the `entity_history` row with `deleted=false`, `timestamp_range=[100,200)`. The EVM layer treats the entity as active and executes the call against its last-known state.

**Verification query:**
```sql
WITH entity_cte AS (
  SELECT id FROM entity WHERE evm_address = '\xDEADBEEF...' AND created_timestamp <= 300
  ORDER BY created_timestamp DESC LIMIT 1
)
(SELECT * FROM entity e WHERE e.deleted IS NOT TRUE AND e.id = (SELECT id FROM entity_cte))
UNION ALL
(SELECT * FROM entity_history eh WHERE lower(eh.timestamp_range) <= 300
 AND eh.id = (SELECT id FROM entity_cte)
 ORDER BY lower(eh.timestamp_range) DESC LIMIT 1)
ORDER BY timestamp_range DESC LIMIT 1;
-- Returns: entity_history row with deleted=false — BUG CONFIRMED
```
