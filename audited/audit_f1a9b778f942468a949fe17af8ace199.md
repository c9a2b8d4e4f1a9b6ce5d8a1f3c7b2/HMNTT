### Title
Unauthenticated Resource Exhaustion via Uncached, Unthrottled `GET /api/v1/network/stake` Endpoint

### Summary
The `NetworkStakeRepository.findLatest()` method executes a native SQL correlated subquery on every invocation with no caching and no rate limiting in the `rest-java` module. The public `GET /api/v1/network/stake` endpoint calls this unconditionally, meaning any unauthenticated external user can drive sustained DB query load, Hibernate session lifecycle overhead, and JVM thread pool saturation by flooding the endpoint with concurrent requests.

### Finding Description
**Exact code path:**

`GET /api/v1/network/stake` → `NetworkController.getNetworkStake()` (line 127-130) → `NetworkServiceImpl.getLatestNetworkStake()` (lines 52-56) → `NetworkStakeRepository.findLatest()` (lines 12-19).

`findLatest()` executes a native SQL query with a correlated subquery on every call:
```sql
select * from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
```

**Root cause — two failed assumptions:**

1. **No caching.** The `rest-java` `NetworkStakeRepository` has zero `@Cacheable` annotation on `findLatest()`. Contrast this with the `grpc` module's `NodeStakeRepository.findAllStakeByConsensusTimestamp()` which is explicitly `@Cacheable` with a dedicated `NODE_STAKE_CACHE` Caffeine cache manager. The rest-java module has no `CacheManager` bean at all for this path.

2. **No rate limiting.** The throttle infrastructure (Bucket4j, `ThrottleManagerImpl`, `ThrottleConfiguration`) exists exclusively in the `web3` module for contract calls. The `rest-java` module's only registered filter is `ShallowEtagHeaderFilter` (ETag-based HTTP caching), which only reduces response body transfer if the *client* sends `If-None-Match` — it does not prevent the server from executing the DB query or Hibernate session lifecycle on every request.

**Per-request overhead triggered:**
- Hibernate session open/close
- JDBC connection acquisition from pool
- Two SQL scans of `network_stake` (outer query + `max()` subquery)
- `NetworkStake` entity instantiation (18 `long` fields)
- JDBC connection return to pool

### Impact Explanation
Under moderate sustained concurrent traffic (e.g., 50-200 req/s from a single attacker or botnet), the server experiences:
- DB connection pool saturation (each request holds a connection for the duration of the correlated subquery)
- JVM thread pool exhaustion (Spring MVC thread per request, no async offload)
- Elevated GC pressure from repeated `NetworkStake` object allocation
- Increased CPU from repeated `max(consensus_timestamp)` full-index scans

This can realistically produce a 30%+ increase in JVM CPU and memory allocation rate compared to baseline, degrading service for all users. The `network_stake` table grows over time (one row per staking period), making the `max()` subquery progressively more expensive.

### Likelihood Explanation
- **No authentication required.** The endpoint is fully public per the OpenAPI spec (no security schemes on `/api/v1/network/stake`).
- **No rate limiting in rest-java.** Confirmed by grep: zero `RateLimiter`, `Bucket`, or throttle references in `rest-java/src/main/java/`.
- **Trivially scriptable.** A single `while true; do curl http://host/api/v1/network/stake & done` loop suffices.
- **Repeatable indefinitely.** No server-side state tracks or blocks repeated callers.

### Recommendation
1. **Add `@Cacheable` to `findLatest()`** with a short TTL (e.g., 30s–60s) using a Caffeine cache, matching the pattern already used in the `grpc` module's `NodeStakeRepository`.
2. **Add rate limiting to the rest-java module** using Bucket4j or Spring's `HandlerInterceptor`, scoped to `/api/v1/network/stake` and other no-parameter endpoints.
3. **Deploy an API gateway or reverse proxy** (e.g., nginx, Envoy) with per-IP rate limiting as a defense-in-depth layer.

### Proof of Concept
```bash
# Flood the endpoint with 100 concurrent workers, no authentication needed
seq 100 | xargs -P100 -I{} sh -c \
  'while true; do curl -s -o /dev/null http://<host>/api/v1/network/stake; done'
```

Monitor JVM CPU via JMX/Actuator (`/actuator/metrics/jvm.cpu.usage`) and DB connection pool (`/actuator/metrics/hikaricp.connections.active`). Under this load, both metrics will climb significantly above the 24-hour baseline without any brute-force authentication attempts.
