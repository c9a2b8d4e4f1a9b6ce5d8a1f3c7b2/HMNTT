### Title
Prometheus Label Cardinality DoS via Unbounded `route` Label in `MetricsMiddleware`

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route extraction to the gorilla/mux `RouteMatcher`. When a request path does not match any registered route, the weaveworks `middleware.Instrument` falls back to using the raw `r.URL.Path` as the `route` label value. Because Prometheus `HistogramVec` and `GaugeVec` allocate a new internal metric object for every unique label combination, an unprivileged attacker can exhaust process memory by sending requests to an unbounded number of unique URL paths.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

**Root cause:**

The weaveworks `middleware.Instrument.Wrap` calls `RouteMatcher.Match(r, &match)` (gorilla/mux). When no registered route matches the incoming path, `Match()` returns `false`. The weaveworks library's `getRouteName` then falls back to `return r.URL.Path` — the raw, attacker-controlled URL path — as the `route` label value. There is no normalization, truncation, or cardinality cap applied anywhere in the application code.

**Four metric vectors are affected simultaneously per request:**

| Metric | Labels | Buckets/Counters per unique combo |
|---|---|---|
| `requestDurationHistogram` | method, route, status_code, ws | 6 buckets + sum + count = 8 |
| `requestBytesHistogram` | method, route | 5 buckets + sum + count = 7 |
| `responseBytesHistogram` | method, route | 5 buckets + sum + count = 7 |
| `requestInflightGauge` | method, route | 1 gauge |

Each unique path permanently allocates all four objects in the Prometheus default registry. Memory is never reclaimed.

**Why existing checks fail:**

The Traefik ingress rate limit (`average: 10` req/s per source IP) is defined only in the Helm chart values and is not enforced at the application layer. Even if active, 10 req/s × 60 s = 600 new unique label combinations per minute per source IP. An attacker using multiple IPs or rotating addresses bypasses even this control entirely. There is no application-level Prometheus label cardinality limit, no `MaxCardinality` option set on any `HistogramVec`, and no route-normalization wrapper around the `RouteMatcher`.

### Impact Explanation

Unbounded memory growth in the Prometheus registry causes the rosetta process to exhaust available heap memory and be killed by the OOM killer (or crash). This takes down the Rosetta API endpoint, which is the sole interface for blockchain data access in this service. Recovery requires a process restart, but the attack can be resumed immediately. Because the metric objects are registered globally and never deleted, even a brief attack window permanently degrades the process until restart.

### Likelihood Explanation

The attack requires zero authentication, zero knowledge of the API, and only the ability to send HTTP requests to the rosetta port. The attacker needs only to generate unique URL paths (e.g., sequential hex strings: `/aaaa0001`, `/aaaa0002`, …). This is trivially scriptable with a single `curl` loop or any HTTP load tool. The attack is repeatable after each restart. No exploit code, credentials, or special network position are required.

### Recommendation

1. **Normalize unmatched routes**: Wrap the `RouteMatcher` to return a fixed string (e.g., `"unmatched"`) when `Match()` returns false, preventing raw paths from becoming label values.
2. **Use `prometheus.Labels` with a fixed allowlist**: Replace `NewHistogramVec` with a fixed set of known route strings and reject or bucket anything outside that set.
3. **Set a cardinality limit**: Use `promauto` with a `prometheus.WrapRegistererWithPrefix` and a cardinality-limiting registerer, or upgrade to a client version supporting `MaxCardinality`.
4. **Apply rate limiting at the application layer**: Do not rely solely on infrastructure-level Traefik configuration that may not be present in all deployment scenarios.
