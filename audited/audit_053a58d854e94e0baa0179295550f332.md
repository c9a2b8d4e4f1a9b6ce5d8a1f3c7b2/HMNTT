### Title
Silent Treasury Substitution for Smart Contract Sender in `getSenderAccountID()` Produces Incorrect Payer in Simulation Records

### Summary
When an unprivileged user supplies a sender address that resolves to a smart contract account, `getSenderAccountID()` silently substitutes the treasury account as the payer instead of rejecting the request with `PAYER_ACCOUNT_NOT_FOUND`. This directly contradicts the code's own stated intent (documented in the comment at lines 287–290) and causes the resulting simulation record's `TransactionID` to carry the treasury account as payer, masking the true origin of the call.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/TransactionExecutionService.java`, `getSenderAccountID()`, lines 239–265.

```java
// Line 250-251 — the problematic branch:
} else if (account.smartContract()) {
    return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
}
```

**Root cause — failed assumption:**

The developer comment at lines 287–290 explicitly states:

> "In services `SolvencyPreCheck#getPayerAccount()` in case the payer account is not found **or is a smart contract** the error response that is returned is `PAYER_ACCOUNT_NOT_FOUND`, so we use it in here for consistency."

The implementation contradicts this: instead of calling `throwPayerAccountNotFoundException()` (which throws `MirrorEvmTransactionException(PAYER_ACCOUNT_NOT_FOUND, ...)`), the `smartContract()` branch silently returns the treasury `AccountID`. No log, no warning, no exception.

**How the incorrect payer propagates:**

`getSenderAccountID()` is called from `defaultTransactionBodyBuilder()` (line 167), which sets the result as the `accountID` inside the `TransactionID` of every `TransactionBody` variant (contract call, contract create, Ethereum transaction). The `TransactionID.accountID` field is the canonical payer of the transaction. When the treasury is substituted here, every `SingleTransactionRecord` produced by `executor.execute()` will record the treasury as the payer.

**Exploit flow:**

1. Attacker identifies any deployed smart contract address on the network (publicly visible on-chain).
2. Attacker issues an `eth_call` / `eth_estimateGas` / `debug_traceTransaction` request with `from` set to that smart contract address.
3. `getSenderAccountID()` resolves the address, finds `account.smartContract() == true`, and returns the treasury `AccountID` without error.
4. `defaultTransactionBodyBuilder()` embeds the treasury account as the payer in the `TransactionID`.
5. The simulation executes with treasury as payer; the resulting record attributes the transaction to the treasury, not the supplied smart contract address.

**Why existing checks are insufficient:**

- The `account == null` guard (line 248) correctly throws for unknown accounts.
- The `isZero() && value == 0` guard (line 241) correctly handles the optional-sender case.
- The `account.smartContract()` guard (line 250) is the only branch that should also throw but instead silently substitutes — this is the sole missing check.

### Impact Explanation

The simulation record's payer field is the treasury account rather than the address the caller supplied. Any downstream consumer of these records (audit logs, analytics, block explorers fed by the mirror node) will attribute the simulated transaction to the treasury, masking the true origin. Additionally, because the treasury account may carry different balance or permission state than the supplied smart contract address, the simulation itself may produce results that diverge from what would actually happen on the consensus network, undermining the correctness guarantee of `eth_call`. The scope label "Incorrect or missing records exported to mirror nodes" is directly satisfied.

### Likelihood Explanation

No privileges are required. Any caller of the public JSON-RPC endpoints (`eth_call`, `eth_estimateGas`, `debug_traceTransaction`) can trigger this by setting `from` to any known smart contract address. Smart contract addresses are trivially discoverable from the mirror node's own REST API or any block explorer. The condition is deterministic and repeatable on every request.

### Recommendation

Replace the silent substitution at line 250–251 with the same rejection used for missing accounts, consistent with the stated intent in the comment at lines 287–290:

```java
} else if (account.smartContract()) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
}
```

This aligns the implementation with `SolvencyPreCheck#getPayerAccount()` behavior on the consensus node and prevents treasury impersonation in simulation records.

### Proof of Concept

**Precondition:** Mirror node is running; a smart contract exists at address `0xABCD…` (any deployed contract).

1. Send an `eth_call` request:
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_call",
     "params": [{"from": "0xABCD…", "to": "0x1234…", "data": "0x…"}, "latest"],
     "id": 1
   }
   ```
2. Observe: no error is returned for the `from` address being a smart contract.
3. Inspect the `TransactionBody` constructed internally (via debug logging or test instrumentation): `transactionID.accountID` equals the treasury account (e.g., `0.0.2`), not the account corresponding to `0xABCD…`.
4. The resulting `SingleTransactionRecord` carries the treasury as payer, confirming the masking.

**Expected (correct) behavior:** The call should return `PAYER_ACCOUNT_NOT_FOUND` when `from` resolves to a smart contract account. [1](#0-0) [2](#0-1) [3](#0-2)
