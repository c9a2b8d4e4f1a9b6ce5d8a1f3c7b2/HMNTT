### Title
Unauthenticated Two-Query DB Connection Exhaustion via `timestamp` + Non-Matching `account.publickey` in `/api/v1/balances`

### Summary
In `rest/balances.js`, `getBalances()` unconditionally executes a DB query via `getTsQuery()` (line 114) before checking whether the supplied `account.publickey` resolves to any accounts (lines 119–122). An unprivileged attacker can craft requests with a valid timestamp and a non-existent public key, forcing two sequential DB queries per request before an early return, with no application-level rate limiting on the REST API. Flooding with such requests exhausts the pgbouncer connection pool allocated to `mirror_rest` (capped at 250 server connections), rendering the mirror node REST API unavailable.

### Finding Description

**Exact code path** — `rest/balances.js`, `getBalances()`, lines 113–122:

```js
if (tsQuery) {
    const tsQueryResult = await getTsQuery(tsQuery, tsParams);   // DB call #1
    if (!tsQueryResult.query) {
      return;
    }

    const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit); // DB call #2
    if (pubKeyQuery && !accountIdsQuery) {
      return;   // early exit — main query never runs
    }
``` [1](#0-0) 

**DB call #1** — `getTsQuery()` → `getAccountBalanceTimestampRange()` issues:
```sql
SELECT consensus_timestamp FROM account_balance
WHERE account_id = $1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3
ORDER BY consensus_timestamp DESC LIMIT 1
``` [2](#0-1) 

**DB call #2** — `getAccountIdsByPublicKey()` issues:
```sql
SELECT id FROM entity WHERE type IN ('ACCOUNT','CONTRACT') AND public_key = $1 LIMIT $2
``` [3](#0-2) 

**Root cause**: The ordering of checks assumes that a timestamp filter and a public-key filter are independent. There is no short-circuit that skips `getTsQuery()` when a public key is provided. The failed assumption is that the preliminary timestamp-snapshot lookup is cheap enough to be harmless before the public-key guard.

**Connection mechanics** — `queryQuietly` without a `preQueryHint` calls `this.query()`, which borrows a pool connection for the duration of each `await`. Under high concurrency, many in-flight requests each holding a connection across two sequential awaits saturate the pool. [4](#0-3) 

**No application-level rate limiting** exists on the REST API. The only protection is pgbouncer's `max_user_connections: 250` for the `mirror_rest` user. [5](#0-4) 

### Impact Explanation
When the pool of 250 server connections is saturated, all REST API endpoints backed by the same pool become unresponsive or return connection-timeout errors. The mirror node REST API (balances, transactions, accounts, tokens, etc.) is fully unavailable for the duration of the attack. Note: the Hedera consensus network itself is unaffected — the mirror node is read-only and does not participate in consensus. The impact is limited to mirror node API availability.

### Likelihood Explanation
- **No authentication required** — any internet-accessible client can send the request.
- **Precondition is trivially met** — any `timestamp=lte:<large_number>` (e.g., `lte:9999999999999999999`) resolves to a valid balance snapshot, satisfying the `tsQueryResult.query` check and proceeding to DB call #2.
- **Non-matching public key is trivially crafted** — any 64-hex-char string not registered in the network (e.g., `account.publickey=aaaa...aaaa`) guarantees `getAccountIdsByPublicKey()` returns `null`.
- **Repeatable and distributable** — no per-IP rate limiting; a botnet or even a single high-throughput client can sustain the attack.

### Recommendation
1. **Reorder checks**: If `pubKeyQuery` is non-empty, call `getAccountIdsByPublicKey()` first and return early before calling `getTsQuery()`. This eliminates the wasted DB call entirely when the public key resolves to no accounts.
2. **Add application-level rate limiting** to the REST API (analogous to the `web3` throttle configuration), keyed by IP or API key.
3. **Consider combining** the public-key lookup and the timestamp snapshot lookup into a single query, or at minimum make them concurrent with `Promise.all()` so they do not hold two sequential connection slots.

### Proof of Concept

```bash
# Step 1: Craft the trigger URL
# - timestamp=lte:9999999999999999999 → always resolves to a valid balance snapshot
# - account.publickey=<random 64-char hex> → never matches any entity

URL="https://<mirror-node-host>/api/v1/balances?timestamp=lte:9999999999999999999&account.publickey=aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"

# Step 2: Flood with concurrent requests (no auth needed)
# Using GNU parallel or ab:
ab -n 100000 -c 500 "$URL"

# Step 3: Observe
# - Each request executes exactly 2 DB queries and returns an empty 200 response
# - At ~500 concurrent requests, the 250-connection pgbouncer pool for mirror_rest is saturated
# - All other REST API endpoints begin timing out or returning DB errors
# - Attack is sustained with zero authentication and trivial tooling
```
