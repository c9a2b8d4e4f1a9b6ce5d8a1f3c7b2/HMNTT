### Title
Unbounded JSON Response Decoding and Unlimited gRPC Network Map Construction in `fetchMirrorNodeNetwork()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` decodes the HTTP response body from the configured mirror node REST endpoint with no size limit, no node-count cap, and no `io.LimitReader` guard. An attacker who controls the mirror node endpoint (via MITM, DNS hijacking, or a malicious/compromised mirror node) can return a crafted response containing thousands of `nodeEntry` objects, causing unbounded CPU and memory consumption during JSON decoding, and then forcing `hiero.ClientForNetwork(netmap)` to attempt gRPC connections to every attacker-supplied address — directly increasing resource consumption on targeted network processing nodes.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, lines 74–132, specifically:

- **Line 96**: `json.NewDecoder(resp.Body).Decode(&payload)` — reads directly from `resp.Body` with no `io.LimitReader` wrapper and no maximum byte constraint.
- **Lines 102–124**: Every `nodeEntry` in `payload.Nodes` is iterated without any count limit; each `ServiceEndpoint` is added to the `network` map.
- `pinger/sdk_client.go` line 22: `client = hiero.ClientForNetwork(netmap)` — the Hiero SDK creates gRPC connection pools for every address in the map.

**Root cause / failed assumption:**
The code assumes the mirror node REST endpoint is trusted and will return a small, well-formed response. There is no enforcement of this assumption in code: no `http.MaxBytesReader`, no `io.LimitReader`, no cap on `len(payload.Nodes)`, and no cap on the number of service endpoints per node.

**Exploit flow:**
1. Attacker positions themselves to control the response from the mirror node REST URL (see Likelihood section).
2. Attacker returns a JSON body with, e.g., 10,000 `nodeEntry` objects, each with multiple `serviceEndpoints` pointing to real network processing node addresses. The body can easily reach hundreds of MB.
3. `json.NewDecoder(resp.Body).Decode(&payload)` allocates memory proportional to the full payload and burns CPU parsing it — no timeout or size guard stops this within the HTTP client's read phase if the server streams data quickly.
4. The resulting `network` map contains thousands of entries; `hiero.ClientForNetwork(netmap)` opens gRPC connection pools to all of them, sending TCP SYN / TLS handshake / HTTP/2 SETTINGS frames to every listed address.
5. Legitimate network processing nodes receive a flood of new gRPC connection attempts from the pinger, increasing their connection-handling CPU, memory, and I/O load.

**Why existing checks are insufficient:**
- `cfg.mirrorNodeClientTimeout` (default 10 s, line 46 of `mirror_node_client.go`) is an HTTP *transaction* timeout, not a body-size limit. A co-located or high-bandwidth attacker can stream hundreds of MB within 10 s. The timeout can also be raised via `HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_TIMEOUT`.
- The retry loop (lines 52–69) adds up to `mirrorNodeClientMaxRetries+1` (default 11) attempts, each of which can re-trigger the same oversized decode.
- There is no validation of `len(payload.Nodes)` or total endpoint count before passing the map to `hiero.ClientForNetwork`.

### Impact Explanation
- **Pinger process**: Memory exhaustion and CPU spike proportional to response size; with a 200 MB JSON body the Go runtime must allocate and GC the full object graph.
- **Network processing nodes**: Each address in the inflated `netmap` receives gRPC connection attempts. With 5,000–10,000 entries all pointing at a small set of real consensus nodes, each node sees a sudden surge in inbound connection requests, increasing their CPU (TLS handshake, HTTP/2 negotiation) and file-descriptor usage well above the 30% threshold stated in the scope.
- The attack is repeatable: the pinger only calls `newClient` once at startup, but a restart (e.g., liveness probe failure caused by the same overload) re-triggers the full fetch.

### Likelihood Explanation
- **Precondition**: `cfg.network == "other"` must be set, and `cfg.mirrorRest` must point to an attacker-reachable endpoint. This is the intended deployment mode for custom/private networks.
- **Attack vectors**: (a) The mirror node REST service itself is compromised or malicious; (b) the URL uses plain HTTP (`http://mirror-rest:5551` is the documented example), making MITM trivial on the same network segment or via ARP/DNS poisoning; (c) DNS hijacking of the hostname in `mirrorRest`.
- No authentication or TLS certificate pinning is enforced on the HTTP client, so network-level interception requires no credentials.
- The attacker needs no account, no API key, and no access to the pinger's configuration — only the ability to intercept or impersonate the HTTP endpoint.

### Recommendation
1. **Limit response body size** — wrap `resp.Body` before decoding:
   ```go
   const maxBodyBytes = 4 * 1024 * 1024 // 4 MB
   limited := io.LimitReader(resp.Body, maxBodyBytes+1)
   if err := json.NewDecoder(limited).Decode(&payload); err != nil { ... }
   ```
2. **Cap node and endpoint count** — after decoding, reject responses with unreasonably many entries:
   ```go
   const maxNodes = 200
   if len(payload.Nodes) > maxNodes {
       return nil, false, fmt.Errorf("too many nodes: %d", len(payload.Nodes))
   }
   ```
3. **Enforce HTTPS** — require `https://` in `mirrorRest` and optionally pin the CA, preventing MITM substitution.
4. **Validate endpoint addresses** — reject RFC-1918 / loopback addresses and ports outside the expected gRPC range before adding them to the network map.

### Proof of Concept
```python
# Malicious mirror node server (Python)
from http.server import BaseHTTPRequestHandler, HTTPServer
import json, random, string

def make_payload(n=8000):
    nodes = []
    for i in range(n):
        nodes.append({
            "node_account_id": f"0.0.{3+i}",
            "service_endpoints": [
                {"ip_address_v4": "34.86.212.250",  # real mainnet node
                 "port": 50211},
                {"ip_address_v4": "35.237.200.180",
                 "port": 50211},
            ],
            "grpc_proxy_endpoint": None
        })
    return json.dumps({"nodes": nodes, "links": {"next": None}}).encode()

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        body = make_payload()          # ~8 MB for 8000 nodes; scale as needed
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 5551), Handler).serve_forever()
```
**Steps:**
1. Run the server above (or intercept via MITM) so it answers requests to the pinger's `HIERO_MIRROR_PINGER_REST` URL.
2. Start the pinger with `HIERO_MIRROR_PINGER_NETWORK=other HIERO_MIRROR_PINGER_REST=http://<attacker>:5551`.
3. Observe: pinger allocates memory proportional to the JSON body; `hiero.ClientForNetwork` opens gRPC connections to all 8,000 × 2 = 16,000 endpoint entries, flooding the listed consensus nodes with connection attempts and raising their inbound connection load by well over 30% compared to normal operation.
