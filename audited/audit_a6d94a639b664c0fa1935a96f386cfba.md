### Title
Unsynchronized Double-Checked Read on `genesisBlock` Causes Go Data Race in `initGenesisRecordFile`

### Summary
`blockRepository.initGenesisRecordFile()` performs an unsynchronized read of `br.genesisBlock.ConsensusStart` at line 244 as a fast-path guard, while the write `br.genesisBlock = rb` at line 258 is only protected inside `sync.Once`. Because the guard read is entirely outside any synchronization primitive, concurrent goroutines servicing HTTP requests can race the read against the struct write, producing a data race that the Go race detector flags as fatal and that causes undefined behavior (including potential torn reads of string fields) under production load.

### Finding Description
**Code path:**

`rosetta/app/persistence/block.go`, function `initGenesisRecordFile`, lines 243–263.

```
243: func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
244:     if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {   // ← unsynchronized READ
245:         return nil
246:     }
...
257:     br.once.Do(func() {
258:         br.genesisBlock = rb   // ← WRITE (struct copy, including string fields)
259:     })
```

**Root cause — classic double-checked locking anti-pattern:**

`sync.Once` provides a happens-before guarantee only between the closure it executes and the return of `Do` for goroutines that actually call `Do`. It provides **no** happens-before guarantee for goroutines that read `br.genesisBlock.ConsensusStart` at line 244 and take the early-return path. The Go memory model therefore does not guarantee that a goroutine reading line 244 will observe the completed write from line 258, and the two accesses are concurrent without synchronization — a data race by definition.

**Struct-level severity:** `br.genesisBlock = rb` is a full struct copy of `recordBlock`, which contains two `string` fields (`Hash`, `PrevHash`). In Go, a `string` is a two-word header (pointer + length). A concurrent read of any field of `br.genesisBlock` while the struct copy is in progress can observe a partially written string header — a pointer from the new value paired with the length from the zero value, or vice versa — which causes an out-of-bounds memory access and a runtime panic when the string is subsequently used (e.g., in `ToBlock()` at lines 96–116, which reads `genesisBlock.Hash` and `genesisBlock.Index`).

**Exploit flow:**

1. The `blockRepository` singleton is created once in `main.go` line 72 and shared by all HTTP handler goroutines.
2. On first request(s), `br.genesisBlock.ConsensusStart` equals `genesisConsensusStartUnset` (-1), so all concurrent goroutines pass the guard at line 244 and proceed to the DB query.
3. Goroutine A finishes the DB query first and enters `once.Do`, beginning the struct copy `br.genesisBlock = rb`.
4. Goroutine B (or any subsequent request goroutine) concurrently reads `br.genesisBlock.ConsensusStart` at line 244 — or reads `br.genesisBlock.Hash` inside `ToBlock()` — while Goroutine A's struct copy is in progress.
5. The Go race detector immediately terminates the process (`os.Exit(2)`). Without the detector, the torn string read can produce an invalid string header, causing a runtime panic in any code path that dereferences it.

**Why existing checks are insufficient:**

`sync.Once` is the only synchronization present. It correctly ensures the write executes exactly once, but it does **not** protect the unsynchronized read at line 244. The pattern requires that the guard check itself be performed under the same synchronization domain as the write — which it is not.

### Impact Explanation
The `blockRepository` is a process-wide singleton serving every Rosetta API endpoint (`/block`, `/block/transaction`, `/account/balance`, `/network/status`, etc.). A race-induced panic or race-detector termination takes down the entire mirror node process, making all Rosetta API endpoints unavailable. Because `genesisBlock` is used as a filter in `findBlockByIndex` (line 211) and `findBlockByHash` (line 235), a torn read that produces a non-(-1) `ConsensusStart` before initialization completes can also silently corrupt block-range filtering, causing valid blocks to be reported as not found or genesis metadata to be wrong — a correctness impact beyond availability.

### Likelihood Explanation
No authentication or special privilege is required. Any client that can reach the Rosetta HTTP port can trigger the race by issuing two or more concurrent requests to any block-querying endpoint immediately after node startup (the only window where `genesisBlock` is uninitialized). This is trivially achievable with a single `curl` invocation using `&` or any HTTP load-testing tool. The race window lasts for the duration of the first genesis DB query (typically tens to hundreds of milliseconds), which is long enough to be reliably hit under normal production traffic. The race is repeatable on every fresh process start.

### Recommendation
Replace the unsynchronized fast-path read with an atomic load, or move the entire initialization check inside a mutex. The idiomatic Go fix is to use `sync.atomic` for the sentinel field:

```go
// Use atomic.Int64 for ConsensusStart in blockRepository
// or protect the guard read with a sync.RWMutex:

func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    br.mu.RLock()
    initialized := br.genesisBlock.ConsensusStart != genesisConsensusStartUnset
    br.mu.RUnlock()
    if initialized {
        return nil
    }
    // ... DB query ...
    br.once.Do(func() {
        br.mu.Lock()
        br.genesisBlock = rb
        br.mu.Unlock()
    })
    return nil
}
```

Alternatively, remove the fast-path guard entirely and rely solely on `sync.Once` (moving the DB query inside the closure), which is the simplest correct pattern.
