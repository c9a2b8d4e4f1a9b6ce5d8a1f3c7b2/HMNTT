### Title
Unbounded URL Logging in `TracingMiddleware` Enables Disk Exhaustion via Long Query Strings

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` captures `request.URL.RequestURI()` at line 47 and logs it verbatim at line 52–58 with no length truncation. An unauthenticated attacker can send repeated requests with maximally-long query strings (up to Go's default 1 MB `MaxHeaderBytes` limit per request), causing the log subsystem to write up to ~1 MB per request to disk. With no rate limiting in the middleware chain, sustained flooding can exhaust available disk space and crash the node.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47–58:
```go
path := request.URL.RequestURI()          // line 47 — full URI, no truncation
// ...
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)  // line 52-53 — path embedded verbatim
if internalPaths[path] {
    log.Debug(message)                    // line 56
} else {
    log.Info(message)                     // line 58 — logged for ALL non-internal paths
}
```

**Root cause:** `request.URL.RequestURI()` returns the raw request URI including the full query string. The result is stored in `path` and passed directly into `fmt.Sprintf` and then `log.Info` with no maximum-length guard.

**Failed assumption:** The code assumes the URI will be short enough that logging it is safe. There is no truncation, no size check, and no rate limiting.

**Exploit flow:**
1. Attacker sends HTTP requests to any non-internal path (e.g., `POST /network/list`) with a query string padded to ~1 MB: `?a=AAAA...AAAA` (repeated thousands of times).
2. Go's `net/http` server accepts the request (bounded only by `MaxHeaderBytes`, which defaults to 1 MB since it is not set in `rosetta/main.go` lines 220–227).
3. `TracingMiddleware` captures the full URI and logs it at `Info` level — one ~1 MB log line per request.
4. With no rate limiting in the middleware chain (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`), the attacker floods the server with concurrent requests.
5. Log files grow at the rate of (requests/sec × ~1 MB), exhausting disk.

**Why existing checks are insufficient:**
- `ReadHeaderTimeout` (3 s default) and `ReadTimeout` (5 s default) only limit per-connection read time; they do not prevent many concurrent connections each sending a valid 1 MB request within the timeout window.
- The `internalPaths` check (line 55) only routes liveness/metrics/readiness to `Debug`; every other path — including 404s and 405s — is logged at `Info`.
- `MaxHeaderBytes` is not configured (line 220–227 of `rosetta/main.go`), so it defaults to Go's `http.DefaultMaxHeaderBytes = 1 << 20` (1 MB). This caps per-request URL size but does not rate-limit requests.
- There is no log-line length cap anywhere in the middleware or logger configuration.

### Impact Explanation
A sustained flood of ~1 MB-URL requests can write hundreds of MB per second to the log sink. Once disk is full, the OS cannot write new files or append to existing ones, which can crash the Rosetta process or the entire host. Because Rosetta is a blockchain API node (≥25% market-cap scope), this constitutes a non-network-based DoS against a critical infrastructure component. No authentication or special privileges are required.

### Likelihood Explanation
Any unprivileged external user with network access to port 5700 can execute this attack using standard tools (`curl`, `wrk`, `ab`). The attack is trivially scriptable, requires no credentials, no valid Rosetta payload, and is repeatable indefinitely. The default deployment (Docker, port 5700 exposed) has no WAF or ingress rate-limiting that would block oversized query strings.

### Recommendation
1. **Truncate the logged path** before including it in the log message, e.g.:
   ```go
   const maxPathLogLen = 512
   if len(path) > maxPathLogLen {
       path = path[:maxPathLogLen] + "...[truncated]"
   }
   ```
2. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g., 8 KB) appropriate for a POST-only JSON API that never uses query parameters.
3. **Add rate limiting** middleware (e.g., `golang.org/x/time/rate`) before `TracingMiddleware` to cap requests per IP per second.
4. **Configure log rotation with size caps** (e.g., `logrotate` with `maxsize`) as a defense-in-depth measure.

### Proof of Concept
```bash
# Generate a ~900 KB query string
QSTRING=$(python3 -c "print('a=1&' * 200000)")

# Flood the server (adjust concurrency as needed)
for i in $(seq 1 500); do
  curl -s -o /dev/null \
    -X POST "http://<rosetta-host>:5700/network/list?${QSTRING}" \
    -H 'Content-Type: application/json' \
    -d '{"metadata":{}}' &
done
wait

# Observe log file growth:
# Each request produces a ~900 KB log line at Info level.
# 500 concurrent requests → ~450 MB written in one burst.
# Repeat in a loop to exhaust disk.
```
