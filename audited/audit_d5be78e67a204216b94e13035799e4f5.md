### Title
Unauthenticated DoS via Expensive `DISTINCT ON` Query on `contract_state_change` Through `GET /contracts/:contractId/state`

### Summary
The `getContractStateById()` handler in `rest/controllers/contractController.js` accepts an unauthenticated `timestamp` filter that switches query execution to `contractStateTimestampQuery`, which performs a `DISTINCT ON (slot)` full-scan over the unbounded `contract_state_change` historical table. The REST API has no per-IP rate limiting, allowing an attacker to flood the service with concurrent requests that each hold a DB connection running this expensive query, exhausting the connection pool and rendering the mirror node instance unavailable.

### Finding Description

**Code path:**

`rest/routes/contractRoute.js:15` registers the unauthenticated route: [1](#0-0) 

`getContractStateById()` at `rest/controllers/contractController.js:964-994` calls `extractContractStateByIdQuery()` then `ContractService.getContractStateByIdAndFilters()`: [2](#0-1) 

`extractContractStateByIdQuery()` sets `timestampPresent = true` when any `timestamp` filter is present, and accumulates all `slot` filters into `slotInValues` with no per-request cap beyond the global `maxRepeatedQueryParameters`: [3](#0-2) 

`getContractStateByIdAndFilters()` in `rest/service/contractService.js:256-263` switches to `contractStateTimestampQuery` when `timestamp=true`: [4](#0-3) 

`contractStateTimestampQuery` is a `DISTINCT ON (slot)` query over the entire `contract_state_change` table up to the supplied timestamp, requiring a sort over potentially millions of historical rows: [5](#0-4) 

**Root cause:** The `contract_state_change` table is a large, append-only historical table (partitioned by `consensus_timestamp`). A `DISTINCT ON (slot)` with `ORDER BY slot, consensus_timestamp DESC` must scan all rows for the given `contract_id` up to the timestamp boundary, sort them, and deduplicate — a CPU- and I/O-intensive operation. Each such query holds a DB connection for its full duration. The DB pool is bounded: [6](#0-5) 

**No rate limiting on the REST API:** The `rest/server.js` middleware stack contains only `authHandler` (optional Basic Auth), `requestLogger`, optional metrics, and optional response cache — no per-IP rate limiter: [7](#0-6) 

The throttling found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` module and does not apply to the REST API. [8](#0-7) 

### Impact Explanation

An attacker sending N concurrent requests each executing the `DISTINCT ON` query exhausts the `maxConnections`-bounded pool. Once the pool is full, all subsequent REST API requests (across all endpoints) receive connection timeout errors. This makes the mirror node instance completely unavailable for the duration of the attack. Because mirror node deployments typically run multiple instances behind a load balancer, an attacker targeting several instances simultaneously can take down 30%+ of the fleet without any credentials.

### Likelihood Explanation

The attack requires zero authentication, zero privileges, and only knowledge of any valid `contractId` (publicly enumerable). The request is trivially constructed:
```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:9999999999.999999999
```
It is repeatable, automatable, and requires no brute force. The `statementTimeout` on the pool provides partial mitigation (queries are eventually killed), but during the timeout window each connection is held, and a sustained flood keeps the pool saturated continuously.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST API (e.g., `express-rate-limit`) before route handlers, independent of the `web3` throttle.
2. **Cap the timestamp query cost**: enforce a maximum `consensus_timestamp` range window (similar to `maxTimestampRange` used elsewhere) so the `DISTINCT ON` scan is bounded.
3. **Add a DB-level query timeout** shorter than the current `statementTimeout` specifically for this endpoint, or use a read replica with resource limits.
4. **Limit concurrent in-flight requests per IP** at the application layer.

### Proof of Concept

```bash
# Send 50 concurrent requests, each triggering the expensive DISTINCT ON scan
for i in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/contracts/0.0.1234/state?timestamp=lte:9999999999.999999999&limit=100" &
done
wait

# Subsequent legitimate requests will receive 503 / connection timeout
curl "http://<mirror-node>/api/v1/contracts/0.0.1234/state"
# Expected: connection timeout or 503 Service Unavailable
```

The attack is amplified by adding multiple `slot` values (up to `maxRepeatedQueryParameters`) per request, which enlarges the `IN (...)` clause and increases per-query cost:
```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:9999999999.999999999&slot=0x01&slot=0x02&...&slot=0xNN
```
