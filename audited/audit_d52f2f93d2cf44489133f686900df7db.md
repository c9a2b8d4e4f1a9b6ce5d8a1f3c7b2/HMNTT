### Title
Unbounded Heap Buffering via ShallowEtagHeaderFilter with No Rate Limiting on rest-java `/api/*`

### Summary
The `etagFilter()` bean in `RestJavaConfiguration` registers a `ShallowEtagHeaderFilter` across all `/api/*` endpoints. Spring's `ShallowEtagHeaderFilter` wraps every response in a `ContentCachingResponseWrapper`, buffering the entire response body as a heap byte array before computing an MD5 ETag. Because the rest-java module has no rate limiting or concurrent-request cap, an unprivileged attacker can flood the server with concurrent large-response requests, causing each in-flight response to hold its full body in heap simultaneously, leading to sustained heap pressure and potential OOM-induced service degradation.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:

```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause:** Spring's `ShallowEtagHeaderFilter` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper`. This wrapper intercepts all writes to the response output stream and accumulates them into an internal `ByteArrayOutputStream` (heap). Only after the entire response body has been written does the filter compute the MD5 hash, set the `ETag` header, and flush the buffered bytes to the real output stream. This buffering occurs unconditionally on every request — even when the client sends no `If-None-Match` header.

**Failed assumption:** The filter assumes that response bodies are small enough that buffering them all simultaneously is safe. There is no byte-size cap on responses, no rate limit, and no concurrent-request limit in the rest-java module.

**Exploit flow:**
1. Attacker identifies any paginated endpoint under `/api/v1/` (e.g., `/api/v1/accounts/{id}/allowances/nfts?limit=100`, `/api/v1/accounts/{id}/airdrops/outstanding?limit=100`).
2. Attacker sends a large number of concurrent GET requests (e.g., 500–1000 simultaneous connections) with `limit=100`.
3. For each request, the JVM allocates a `ByteArrayOutputStream` on the heap to buffer the full JSON response body.
4. With N concurrent requests each buffering R bytes, the heap holds N × R bytes simultaneously until each response is committed.
5. GC pressure increases; under sustained load this can trigger full GCs, latency spikes, or OOM errors.

**Why existing checks are insufficient:**
- `MAX_LIMIT = 100` (`Constants.java`, line 35) caps the record count per page, but does not cap the byte size of each record or the total response body size.
- The rest-java module has **no rate limiting** — unlike the web3 module which has `ThrottleProperties`/`ThrottleManagerImpl` with `requestsPerSecond = 500`, there is no equivalent in rest-java.
- There is no `server.tomcat.max-connections` or `server.tomcat.threads.max` override visible in the configuration, so Tomcat defaults apply.
- The filter has no `writeWeakETag` or streaming mode — it always buffers.

### Impact Explanation
Every concurrent request to `/api/*` holds its full response body in JVM heap until the response is committed. Under a flood of concurrent requests, heap exhaustion can cause full GC pauses, request timeouts, or OOM crashes, degrading or denying service to all users. No authentication or special privilege is required. The impact is griefing/availability degradation with no direct economic damage to network participants.

### Likelihood Explanation
Any unprivileged external user can trigger this with a standard HTTP client (e.g., `ab`, `wrk`, `curl` in parallel). The attack requires no credentials, no special knowledge beyond the public API path, and is trivially repeatable. The absence of rate limiting in rest-java makes sustained exploitation straightforward.

### Recommendation
1. **Add rate limiting to rest-java** analogous to the web3 module's `ThrottleProperties`/`ThrottleManagerImpl` pattern, limiting requests per second per IP or globally.
2. **Restrict `ShallowEtagHeaderFilter` scope** to endpoints where ETag caching provides meaningful benefit, or replace it with a `writeWeakETag`-based approach that avoids full buffering.
3. **Set a maximum response body size** (e.g., via a `ContentSizeLimitingFilter`) to bound per-request heap allocation.
4. **Configure Tomcat connection/thread limits** explicitly (`server.tomcat.max-connections`, `server.tomcat.threads.max`) to cap concurrency.

### Proof of Concept

```bash
# Send 200 concurrent requests to a large-response endpoint
ab -n 2000 -c 200 \
  "http://<host>/api/v1/accounts/0.0.1/allowances/nfts?limit=100"

# Or with wrk for sustained load:
wrk -t 10 -c 200 -d 60s \
  "http://<host>/api/v1/accounts/0.0.1/airdrops/outstanding?limit=100"
```

Monitor JVM heap via JMX or `/actuator/metrics/jvm.memory.used`. Under sustained load, heap usage will climb proportionally to concurrency × response size. With no rate limiting, the attacker can hold this pressure indefinitely, causing GC thrashing or OOM.
