### Title
Unauthenticated Multi-Stage DB Query Amplification in `getBalances()` Enables Connection Pool Exhaustion

### Summary
The `/api/v1/balances` endpoint in `rest/balances.js` executes up to four sequential database queries per request when an attacker combines `account.id=eq:<alias>`, `timestamp=lte:<recent>`, and `account.publickey=<key>` parameters. No application-level rate limiting exists on this endpoint in the REST API middleware stack, allowing an unprivileged attacker to exhaust the finite PostgreSQL connection pool by flooding the endpoint with concurrent crafted requests, denying service to legitimate users.

### Finding Description

**Exact code path in `rest/balances.js`, `getBalances()` (lines 83–156):**

**DB Query 1 — Alias lookup** (`entity` table):
`parseAccountIdQueryParam` (line 331–332) detects a valid alias and returns `EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false)` — a Promise. This is awaited at line 88 via `await Promise.all(accountParamsPromise)`, issuing the query in `EntityService.entityFromAliasQuery` against the `entity` table. [1](#0-0) [2](#0-1) [3](#0-2) 

**DB Query 2 — Timestamp range lookup** (`account_balance` table):
When `tsQuery` is non-empty (triggered by `timestamp=lte:<recent>`), line 114 `await getTsQuery(tsQuery, tsParams)` calls `getAccountBalanceTimestampRange()`, which issues a `SELECT consensus_timestamp FROM account_balance WHERE ...` query. [4](#0-3) [5](#0-4) 

**DB Query 3 — Public key lookup** (`entity` table):
Line 119 `await getAccountIdsByPublicKey(pubKeyParams, limit)` issues `SELECT id FROM entity WHERE ... public_key = $1` when `account.publickey` is supplied. [6](#0-5) [7](#0-6) 

**DB Query 4 — Final balance query** (`account_balance` table):
Line 153 `await pool.queryQuietly(pgSqlQuery, sqlParams)` executes the main balance SELECT. [8](#0-7) 

All four queries are sequential (each `await` blocks before the next begins), meaning each request holds a DB connection slot for the full duration of all four round-trips.

**Root cause — no application-level rate limiting on the REST API:**
The REST middleware stack (`rest/middleware/index.js`, `requestHandler.js`, `metricsHandler.js`) contains no per-IP rate limiting or in-flight request cap for the balances endpoint. The throttle infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) applies only to the `web3` Java service, not the Node.js REST API. [9](#0-8) 

The DB pool is bounded by `config.db.pool.maxConnections`: [10](#0-9) 

**Why existing checks fail:**
- The `evmAliasAddressCount` guard (line 335) only prevents more than one alias *within a single request*; it does nothing against concurrent requests from the same or multiple sources.
- `statement_timeout` limits individual query duration but does not prevent pool slot exhaustion when requests arrive faster than they complete.
- The `responseCacheHandler.js` middleware can be bypassed by varying the alias or timestamp value slightly across requests. [11](#0-10) 

### Impact Explanation
When the DB connection pool is exhausted, all subsequent queries across all REST API endpoints queue or fail with connection timeout errors, making the entire mirror node REST API unavailable to legitimate users. Since the mirror node is a public read API with no authentication required for `/api/v1/balances`, this is a full denial-of-service against the service. The impact is limited to the mirror node (no on-chain state is affected), consistent with the "griefing" classification, but the service outage is real and observable.

### Likelihood Explanation
The attack requires zero privileges, no tokens, and no special knowledge beyond the public API documentation. A single attacker with a modest number of concurrent HTTP connections (e.g., 50–100 concurrent requests using `curl` or any HTTP client) can saturate a default-sized connection pool. The attack is trivially repeatable and scriptable. The alias value does not need to resolve to a real account (`requireResult=false` is passed, so a missing alias returns `null` without throwing), meaning the attacker does not need to know valid aliases — any syntactically valid base32 alias string triggers the full query chain. [2](#0-1) [12](#0-11) 

### Recommendation
1. **Add per-IP rate limiting** to the Node.js REST API middleware (e.g., via `express-rate-limit`) specifically for endpoints that trigger multi-stage DB lookups.
2. **Parallelize independent lookups**: The alias resolution (Query 1), timestamp range lookup (Query 2), and public key lookup (Query 3) are independent and can be issued concurrently with `Promise.all()`, reducing the wall-clock connection hold time from 4× to ~1× the slowest query.
3. **Add an in-flight request cap** at the application layer (similar to Traefik's `inFlightReq` middleware already used for the Rosetta service) to bound the maximum number of concurrent DB-touching requests.
4. **Cache alias→id resolutions** with a short TTL to avoid a DB round-trip on every request for the same alias.

### Proof of Concept

```bash
# Generate a syntactically valid base32 alias (no real account needed, requireResult=false)
ALIAS="AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"
TIMESTAMP=$(date +%s)
KEY="3a7d3e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d"

# Flood with concurrent requests, each triggering 4 sequential DB queries
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/balances?account.id=eq:${ALIAS}&timestamp=lte:${TIMESTAMP}&account.publickey=${KEY}" &
done
wait

# Legitimate request now times out or receives connection error
curl -v "http://<mirror-node>/api/v1/balances"
```

Each background request triggers: (1) `SELECT id FROM entity WHERE alias=$1`, (2) `SELECT consensus_timestamp FROM account_balance WHERE ...`, (3) `SELECT id FROM entity WHERE public_key=$1`, (4) main `SELECT ... FROM account_balance ...` — all sequential, holding a pool slot for the full chain duration. With 200 concurrent requests and a typical `maxConnections` of 10–20, the pool is exhausted within the first wave.
