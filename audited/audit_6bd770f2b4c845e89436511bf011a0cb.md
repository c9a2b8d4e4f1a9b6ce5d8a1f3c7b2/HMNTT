### Title
Unbounded Timestamp Range in `getHookStorageChange` Enables DB Exhaustion via Full Historical Scan

### Summary
The private `getHookStorageChange` method in `HookServiceImpl` accepts user-supplied timestamp bounds with no maximum-range validation. When an unprivileged caller supplies `timestamp=gte:0&timestamp=lte:9999999999.999999999`, the resulting `DISTINCT ON (key)` native SQL query must scan and sort every matching row in `hook_storage_change` before the `LIMIT` clause is applied, making the per-request DB cost proportional to the full history of any targeted owner/hook pair.

### Finding Description

**Exact code path:**

`HookServiceImpl.getHookStorageChange` (lines 84–118) extracts timestamp bounds directly from the user-controlled `Bound` object: [1](#0-0) 

`Bound.getAdjustedLowerRangeValue()` returns `0` when no lower bound is supplied, and `Bound.adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is supplied: [2](#0-1) 

These uncapped values are forwarded verbatim to the repository: [3](#0-2) 

The native query uses `DISTINCT ON (key)` with an implicit `ORDER BY key ASC, consensus_timestamp DESC`: [4](#0-3) 

**Root cause:** PostgreSQL's `DISTINCT ON` requires the database engine to sort and process *all* rows matching the `WHERE` clause before it can emit the first distinct-key row. The `LIMIT` imposed by `PageRequest` (default 25) is applied only *after* this full sort/scan completes. No maximum timestamp span is enforced anywhere in `HookStorageRequest`, `Bound`, or `HookServiceImpl`. [5](#0-4) 

**Why existing checks fail:**
- The `limit` field (default 25) only caps the *output* rows, not the rows the DB must read.
- `Bound` validates operator cardinality and lower ≤ upper ordering for primary sort fields, but performs no span-width check.
- `QueryProperties` exposes only `maxFileAttempts`; there is no `maxTimestampRange` or equivalent. [6](#0-5) 

### Impact Explanation
An attacker who knows any valid `ownerId`/`hookId` pair (both are public on-chain identifiers) can issue repeated requests with a maximally wide timestamp range. Each request forces a full sequential scan of `hook_storage_change` for that owner/hook, consuming DB I/O, CPU (sort), and connection pool slots. Sustained at modest request rates this degrades or denies service to all other API consumers sharing the same database. Severity is **High** for availability; confidentiality and integrity are unaffected.

### Likelihood Explanation
No authentication or authorization is required. Valid `ownerId` and `hookId` values are observable from any public block explorer or the mirror node's own `/hooks` endpoint. The attack is trivially scriptable with a single `curl` loop and is fully repeatable. Rate limiting is not visible in the service or repository layer reviewed.

### Recommendation
1. **Enforce a maximum timestamp span** in `HookStorageRequest` or at the controller layer (e.g., reject requests where `timestampUpperBound − timestampLowerBound > MAX_ALLOWED_SPAN`).
2. **Rewrite the query** to use a keyset-paginated approach with an index on `(owner_id, hook_id, key, consensus_timestamp DESC)` so PostgreSQL can satisfy `DISTINCT ON` via an index scan and stop after `LIMIT` rows without reading the full history.
3. **Add rate limiting** at the API gateway or Spring filter layer keyed on client IP or API token.
4. **Cap the `limit` parameter** at a configurable maximum (e.g., 100) and expose a `maxTimestampRange` in `QueryProperties`.

### Proof of Concept
```
# Assumes a known ownerId (e.g., 0.0.1234) and hookId (e.g., 1)
# Repeat in a tight loop to exhaust DB I/O

while true; do
  curl -s "https://<mirror-node>/api/v1/hooks/0.0.1234/storage?hookId=1\
&timestamp=gte:0\
&timestamp=lte:9999999999.999999999\
&limit=25&order=asc" > /dev/null
done
```
Each iteration triggers `findByKeyBetweenAndTimestampBetween` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, forcing PostgreSQL to scan and sort the entire `hook_storage_change` history for that owner/hook before returning 25 rows.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public RangeParameter<Long> adjustLowerRange() {
        if (this.hasEqualBounds()) {
            // If the primary param has a range with a single value, rewrite it to EQ
            lower = new NumberRangeParameter(EQ, this.getAdjustedLowerRangeValue());
            upper = null;
        }

        return lower;
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
