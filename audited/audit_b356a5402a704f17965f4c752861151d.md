### Title
Unauthenticated DB Query Amplification (7x) via `GET /contracts/results/:transactionIdOrHash` with Valid Ethereum Hash

### Summary
Any unauthenticated external user can send `GET /api/v1/contracts/results/{validEthHash}` with a known Ethereum transaction hash, causing `getContractResultsByTransactionIdOrHash()` to unconditionally execute 7 sequential/parallel database queries per single HTTP request. The REST API has no rate-limiting middleware, making this trivially repeatable at scale to exhaust the database connection pool and degrade service availability.

### Finding Description

**Exact code path** (`rest/controllers/contractController.js`, lines 1120–1203):

1. **Query 1** — `ContractService.getContractTransactionDetailsByHash()` (line 1134): executes `ethereumTransactionsByHashQuery` against `contract_transaction_hash` table.
2. If result is non-null, **Query 2** — `ContractService.getInvolvedContractsByTimestampAndContractId()` (line 1156): executes `involvedContractsQuery` against `contract_transaction` table.
3. If result is non-null, **Queries 3–7** — `this.getDetailedContractResults()` (line 1165–1166) fires `Promise.all()` with 5 parallel queries:
   - `ContractService.getContractResultsByTimestamps()` — `contract_result` table
   - `TransactionService.getEthTransactionByTimestampAndPayerId()` — `ethereum_transaction` table
   - `RecordFileService.getRecordFileBlockDetailsFromTimestamp()` — `record_file` table
   - `ContractService.getContractLogsByTimestamps()` — `contract_log` table
   - `ContractService.getContractStateChangesByTimestamps()` — `contract_state_change` table

**Root cause**: The function has no guard against unauthenticated, high-frequency calls. The `authHandler` middleware (`rest/middleware/authHandler.js`, lines 15–36) explicitly allows requests with no `Authorization` header to pass through without any restriction. The REST `server.js` (lines 67–144) registers no rate-limiting middleware — only `authHandler`, `requestLogger`, optional `metricsHandler`, and optional `responseCacheCheckHandler`. The throttle infrastructure (`web3/src/main/java/.../ThrottleConfiguration.java`) exists only in the `web3` module (JSON-RPC), not the REST API module.

**Why the response cache does not mitigate this**: The cache is only active when `config.cache.response.enabled && config.redis.enabled` (server.js line 54), an optional deployment configuration. Even when enabled, cache misses (e.g., newly submitted transactions or cache evictions) still trigger the full 7-query path.

**Precondition for full amplification**: The attacker must supply a hash that exists in the database. Valid Ethereum transaction hashes are publicly observable on the Hedera network (via explorers, node APIs, etc.), so this is not a meaningful barrier. With a non-existent hash, only Query 1 executes and a `NotFoundError` is thrown at line 1152–1154 — but with any known valid hash, all 7 queries fire.

### Impact Explanation

Each HTTP request with a valid hash consumes 7 database queries. At modest concurrency (e.g., 100 concurrent clients each sending 50 req/s), this translates to 35,000 DB queries/second against a shared PostgreSQL pool. This can:
- Exhaust the database connection pool, causing all REST API endpoints (not just this one) to queue or reject requests.
- Increase DB CPU/IO load, degrading query latency for legitimate users across all endpoints.
- Cause cascading 500/503 errors across the entire mirror node REST API.

No funds or on-chain state are affected, consistent with the "griefing, no economic damage" classification.

### Likelihood Explanation

- **No authentication required**: confirmed by `authHandler.js` lines 18–20 — missing credentials are silently allowed.
- **No rate limiting**: confirmed by `server.js` — no `express-rate-limit` or equivalent middleware is registered for the REST API.
- **Attacker capability**: any internet user; valid hashes are publicly available from block explorers or by monitoring the Hedera network.
- **Repeatability**: trivially scriptable with `curl` or any HTTP client in a loop.

### Recommendation

1. **Add rate limiting to the REST API**: integrate `express-rate-limit` (or equivalent) as middleware in `rest/server.js`, applied globally or specifically to the `/contracts/results/:transactionIdOrHash` route, keyed by IP address.
2. **Merge the two sequential queries into one**: `getContractTransactionDetailsByHash` and `getInvolvedContractsByTimestampAndContractId` can be combined into a single JOIN query, reducing the minimum query count from 7 to 6 and eliminating one round-trip.
3. **Enforce response caching unconditionally** for this endpoint (contract results by hash are immutable once finalized), removing the optional Redis dependency as a prerequisite.
4. **Consider connection pool limits per-route** or a DB query budget per request to bound the blast radius.

### Proof of Concept

```bash
# Step 1: Obtain a valid Ethereum transaction hash from any Hedera explorer
VALID_HASH="0x4a563af33c4871b51a8b108aa2fe1dd5280a30dfb7236170ae5e5e7957eb6392"

# Step 2: Send high-frequency unauthenticated requests (no credentials needed)
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results/${VALID_HASH}" &
done
wait

# Each request triggers 7 DB queries.
# 1000 concurrent requests = 7000 DB queries with no server-side throttle.
# Monitor DB connection pool exhaustion and API latency degradation.
```
