### Title
Unauthenticated DoS via `result=fail` Filter Causing Unindexed Full-Table Scans on Transaction Endpoint

### Summary
The `/api/v1/transactions?result=fail` endpoint translates `transactionResultFilter.FAIL` into a `t.result NOT IN (...)` SQL predicate against the `transaction` table's unindexed `result` column. No rate limiting exists in the REST API layer. An unprivileged attacker can flood this endpoint with concurrent requests, each triggering expensive sequential scans across the timestamp-bounded transaction range, exhausting DB connection pool and CPU resources and causing `httpStatusCodes.SERVICE_UNAVAILABLE` (503) for all users.

### Finding Description

**Code path — filter constant definition:**
`rest/constants.js` lines 115–118 define `transactionResultFilter.FAIL = 'fail'`. [1](#0-0) 

**Code path — SQL generation:**
`rest/transactions.js` `extractSqlFromTransactionsRequest` (lines 426–428) generates:
```js
const operator = resultType === constants.transactionResultFilter.SUCCESS ? 'in' : 'not in';
resultTypeQuery = `t.result ${operator} (${utils.resultSuccess})`;
```
For `result=fail` this produces `t.result NOT IN (<success_proto_ids>)`. [2](#0-1) 

This `resultTypeQuery` is injected directly into the WHERE clause of the timestamp-range query executed at line 483: [3](#0-2) 

**Root cause — no index on `result` column:**
The DB migration files define indexes on `transaction` only for `(consensus_timestamp, payer_account_id)`, `(payer_account_id, consensus_timestamp)`, and `(type, consensus_timestamp)`. No index exists on `result`. [4](#0-3) 

PostgreSQL cannot use any existing index to satisfy `result NOT IN (...)` as a leading predicate; it must evaluate the filter row-by-row across every row in the timestamp range.

**Root cause — no rate limiting in REST API:**
`rest/server.js` registers the transactions route with no rate-limiting middleware. A `grep` for `rateLimit`, `rateLimiter`, or `throttle` across all `rest/**/*.js` files returns zero matches. The only throttling in the codebase is in the `web3` Java service for contract calls, not the Node.js REST API. [5](#0-4) 

**Failed assumption:** The design assumes that the `LIMIT` clause and timestamp-range bounding (`bindTimestampRange`) are sufficient to cap query cost. They are not: on a high-volume network with thousands of transactions per second, even a bounded timestamp window contains millions of rows, all of which must be scanned to find the minority that are failed transactions.

### Impact Explanation
Each `result=fail` request forces a sequential scan of all rows in the timestamp window, filtered by an unindexed column. Concurrent flooding saturates the PostgreSQL connection pool and CPU. Once the pool is exhausted, all API endpoints — not just `/transactions` — begin returning 503 (`httpStatusCodes.SERVICE_UNAVAILABLE`), causing a complete denial of service for legitimate users. The `transaction` table on a production mirror node contains hundreds of millions of rows, making each scan extremely expensive.

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. Any external user can issue unlimited concurrent HTTP GET requests. The attack requires only a basic HTTP client and knowledge of the public API (documented in OpenAPI spec). It is trivially repeatable and automatable. The `result=fail` parameter is explicitly validated as a legal value (`filterValidityChecks` in `rest/utils.js` line 340–342), so it passes all input validation and reaches the DB. [6](#0-5) 

### Recommendation
1. **Add a rate limiter** (e.g., `express-rate-limit` or a Redis-backed limiter) to the REST API, applied globally or specifically to high-cost endpoints like `/api/v1/transactions`.
2. **Add a partial index** on `transaction(result, consensus_timestamp)` or a filtered index for non-success results so the `NOT IN` predicate can use an index scan.
3. **Cap concurrent DB queries** per IP at the application layer (connection semaphore or queue).
4. **Consider requiring a timestamp range** when `result=fail` is used, to bound the scan to a narrow window.

### Proof of Concept
```bash
# Flood the endpoint with concurrent result=fail queries (no auth required)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?result=fail&limit=25" &
done
wait

# Observe: legitimate requests begin returning HTTP 503
curl -v "https://<mirror-node-host>/api/v1/transactions"
# Expected: HTTP 503 Service Unavailable
```

Each concurrent request triggers `SELECT ... FROM transaction WHERE t.result NOT IN (22, 104, ...) AND consensus_timestamp BETWEEN $x AND $y ORDER BY consensus_timestamp DESC LIMIT 25`, scanning millions of rows per request with no index support, exhausting DB CPU and connection pool.
