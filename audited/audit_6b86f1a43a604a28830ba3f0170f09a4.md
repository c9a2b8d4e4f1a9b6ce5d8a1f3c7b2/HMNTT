### Title
CPU Exhaustion via Large `data` Payload Against CONTAINS Request Filter

### Summary
When an operator configures a `CONTAINS`-type `RequestFilter` targeting the `DATA` field, any unauthenticated caller can submit `ContractCallRequest` payloads with up to 1 MB of hex data. Because the rate-limit check passes before filter evaluation, every admitted request forces a full case-insensitive string scan (`Strings.CI::contains`) over the entire payload, allowing an attacker to saturate CPU at the permitted request rate and degrade throughput for all users.

### Finding Description
**Code path:**

1. `ThrottleManagerImpl.throttle()` (lines 38–48): the global `rateLimitBucket.tryConsume(1)` check runs first. If it passes, the method iterates `throttleProperties.getRequest()` and calls `requestFilter.test(request)` for each configured `RequestProperties`.
2. `RequestProperties.test()` (lines 54–58): iterates its `List<RequestFilter>` and calls `filter.test(contractCallRequest)`.
3. `RequestFilter.test()` (lines 31–35): extracts `ContractCallRequest::getData` (up to 1,048,576 hex characters = 1 MB), converts it to a `String`, then calls `Strings.CI::contains(stringValue, expression)`.

`Strings.CI::contains` is Apache Commons Lang's case-insensitive contains. It performs at minimum an O(n) case-fold pass over the full string before the substring search. For a 1 MB payload this is ~1 MB of character work per request.

**Root cause / failed assumption:** The rate limiter counts *requests*, not *bytes*. It assumes each admitted request has roughly uniform CPU cost. The CONTAINS filter evaluation cost scales linearly with `data` length, but no size cap is applied before the scan. The `@Hex(maxLength = 1048576)` constraint is enforced by Bean Validation before `throttle()` is called, so a 1 MB payload is fully legal and reaches the filter loop.

**Why existing checks are insufficient:**
- The rate limit (500 req/sec) bounds request count, not CPU work per request.
- There is no per-IP rate limit; a single attacker can consume the full 500 req/sec budget.
- `RequestProperties.test()` has a `rate`/`limit` guard, but those are operator-configured and default to `rate=100`, `limit=Long.MAX_VALUE` — they do not bound payload size.

### Impact Explanation
At 500 req/sec × 1 MB/request, the server must perform ~500 MB/sec of case-insensitive string scanning in addition to normal request processing. This can saturate one or more CPU cores, increasing latency and reducing effective throughput for all concurrent users. The attack requires no authentication, no funds, and no special privileges — only the ability to send HTTP requests.

### Likelihood Explanation
Any external caller can craft the request. The only precondition is that an operator has configured at least one `CONTAINS` filter on the `DATA` field, which is a documented and intended feature. The attack is trivially repeatable with a simple HTTP client loop and requires no knowledge of contract internals.

### Recommendation
1. **Cap payload size before filter evaluation**: enforce a tighter `maxLength` on `data` (e.g., matching the actual EVM calldata limit ~24 KB) or add an explicit size check at the top of `throttle()` before the filter loop.
2. **Charge tokens proportional to payload size**: consume additional tokens from `rateLimitBucket` (or a dedicated byte-budget bucket) proportional to `data.length()` so large payloads are throttled more aggressively.
3. **Add per-IP rate limiting** upstream (e.g., at the load balancer or via a servlet filter) so a single source cannot exhaust the global budget.

### Proof of Concept
**Precondition:** operator has configured one `RequestProperties` with a `RequestFilter` of `field=DATA`, `type=CONTAINS`, `expression=<any short hex string>`.

```bash
# Generate a 1 MB hex payload (2,097,152 hex chars = 1 MB of bytes)
PAYLOAD="0x$(python3 -c "print('ab' * 524288)")"

# Send at the rate limit
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"$PAYLOAD\",\"gas\":15000000,\"to\":\"0x$(printf '%040d' 1)\"}" &
done
wait
```

Each of the 500 concurrent requests passes the rate-limit check and triggers a ~1 MB `Strings.CI::contains` scan inside `RequestFilter.test()`, causing sustained high CPU usage and latency spikes for all other users.
