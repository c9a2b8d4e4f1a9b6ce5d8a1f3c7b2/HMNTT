### Title
Unbounded Concurrent DB Queries via Transaction Hash Lookup — No Cache or Request Coalescing in `getTransactionsByIdOrHash()`

### Summary
The `getTransactionsByIdOrHash()` handler in `rest/transactions.js` performs no caching and no request deduplication for transaction-hash-based lookups. Every concurrent request for the same hash independently calls `getTransactionHash()` and `pool.queryQuietly()`, firing 2–3 DB queries per request with no protection. An unprivileged attacker can exhaust the DB connection pool by sending a burst of concurrent requests for the same hash, degrading the mirror node REST API.

### Finding Description
**Code path:**

`getTransactionsByIdOrHash()` (`rest/transactions.js`, lines 923–972) calls `extractSqlFromTransactionsByIdOrHashRequest()` (lines 763–795), which — when the input is a hash — calls `getTransactionHash()` (`rest/transactionHash.js`, lines 21–37). `getTransactionHash()` unconditionally calls `pool.queryQuietly()` (line 35). Control then returns to `getTransactionsByIdOrHash()`, which calls `pool.queryQuietly()` again (line 930), and potentially a third time for inner transactions (line 955).

**Root cause:**

The `Cache` class (`rest/cache.js`) is instantiated at module level (line 44 of `transactions.js`) and its `cache.get()` method is used in `doGetTransactions()` (line 705) for the list endpoint. It is **never used** in `getTransactionsByIdOrHash()`. There is no mutex, promise-coalescing, or in-flight deduplication for hash lookups. Every concurrent request is fully independent.

**Exploit flow:**

1. Attacker identifies any valid (or even invalid) transaction hash.
2. Attacker sends N concurrent HTTP requests to `GET /api/v1/transactions/{hash}`.
3. All N requests simultaneously enter `getTransactionHash()` → `pool.queryQuietly()` (DB query 1).
4. All N requests then call `pool.queryQuietly()` for transaction details (DB query 2).
5. If inner transactions exist, all N requests fire a third `pool.queryQuietly()` (DB query 3).
6. Total: up to 3N simultaneous DB queries from a single burst.

**Why existing checks fail:**

The `Cache` class has no stampede protection even where it is used (no locking in `cache.get()`), but more critically it is entirely absent from the hash-based lookup path. There is no visible rate limiting in the handler or middleware that would throttle concurrent requests per hash.

### Impact Explanation
Exhausting the DB connection pool causes all in-flight queries to queue or fail, degrading the mirror node REST API for all users. Because the mirror node is the primary read interface for the Hedera network, sustained degradation affects downstream integrations and operators. The attack requires no authentication and can be repeated continuously.

### Likelihood Explanation
Any unprivileged external user can execute this attack with a single script issuing concurrent HTTP requests. No credentials, special knowledge, or brute force are required — only a known or guessable transaction hash (which are publicly observable on-chain). The attack is trivially repeatable and automatable.

### Recommendation
1. **Add request coalescing / in-flight deduplication**: Use a per-hash promise map so that N concurrent requests for the same hash share a single DB call.
2. **Cache hash→timestamp mappings in Redis**: After `getTransactionHash()` resolves, store the result in Redis with a short TTL so subsequent requests are served from cache.
3. **Apply per-IP or per-hash rate limiting** at the middleware layer for the `/transactions/:transactionIdOrHash` route.
4. **Set a DB connection pool limit with a queue cap** and return `503` when the queue is full, preventing unbounded resource consumption.

### Proof of Concept
```bash
# Send 200 concurrent requests for the same transaction hash
HASH="0x$(python3 -c "import secrets; print(secrets.token_hex(48))")"
# Or use a real known hash from the public ledger
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/transactions/${HASH}" &
done
wait
```
Each concurrent request independently executes 2–3 `pool.queryQuietly()` calls against the database. With sufficient concurrency, the DB connection pool is exhausted, causing timeouts and errors for all concurrent users of the mirror node API. const getTransactionHash = async (hash, {order = orderFilterValues.ASC, timestampFilters = []} = {}) => {
  const normalized = normalizeTransactionHash(hash);
  const params = [normalized];

  const timestampConditions = [];
  for (const filter of timestampFilters) {
    timestampConditions.push(`${TransactionHash.CONSENSUS_TIMESTAMP} ${filter.operator} $${params.push(filter.value)}`);
  }

  const query = `${mainQuery}
    ${timestampConditions.length !== 0 ? `where ${timestampConditions.join(' and ')}` : ''}
    ${orderClause} ${order}
    ${limitClause}`;

  const {rows} = await pool.queryQuietly(query, params);
  return normalized !== hash ? rows.filter((row) => row.hash.equals(hash)) : rows;
