### Title
Unauthenticated LIKE-Scan DoS via 0x-Prefixed Hedera Hash in `GET /blocks/:hashOrNumber`

### Summary
Any unauthenticated external user can send `GET /api/v1/blocks/<0x + 96 hex chars>` to trigger a PostgreSQL `LIKE` prefix scan on the `record_file` table for every request. Because `LIKE 'pattern%'` does not use a standard B-tree index in PostgreSQL, each request causes a sequential scan. With no application-level rate limiting, an attacker can flood this endpoint to exhaust database I/O and connection pool resources.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:13` registers `GET /:hashOrNumber` → `BlockController.getByHashOrNumber`.

`rest/controllers/blockController.js:28-30` — `validateHashOrNumber()` accepts `0x` + 96 hex chars via `isValidEthHashOrHederaHash()` (pattern `/^(0x)?([0-9A-Fa-f]{64}|[0-9A-Fa-f]{96})$/`) and strips the prefix with `.replace('0x', '')`, yielding a 96-char hash string. [1](#0-0) 

`rest/service/recordFileService.js:167-170` — `getByHashOrNumber()` constructs the query as `WHERE hash LIKE $1` with the parameter `hash + '%'`. This is a LIKE prefix scan, not an equality check, for every hash lookup regardless of hash length. [2](#0-1) 

**Root cause:** The design uses `LIKE hash%` to unify 64-char ETH hash prefix matching and 96-char Hedera hash exact matching in a single query. In PostgreSQL, `LIKE 'prefix%'` on a `text` column does not use a standard B-tree index (requires `text_pattern_ops` or C locale). Without such an index, every request causes a full sequential scan of `record_file`. [3](#0-2) 

**Failed assumption:** The design assumes that either (a) a `text_pattern_ops` index exists on `record_file.hash`, or (b) the table is small enough that sequential scans are harmless under load. Neither assumption is enforced or rate-limited at the application layer.

### Impact Explanation
Under sustained flooding, each request triggers a sequential scan of `record_file`, consuming I/O bandwidth and a database connection slot for the duration of the scan. With enough concurrent requests, the database connection pool is exhausted, causing all queries (including unrelated ones) to queue or fail. This results in denial of service for the entire mirror node REST API, not just the blocks endpoint. Severity: **Medium** (DoS, no data exfiltration; table size limits worst-case scan cost per request, but concurrency amplifies impact).

### Likelihood Explanation
Preconditions: none — no authentication, no API key, no session. The endpoint is publicly documented in the OpenAPI spec. The attack is trivially scriptable with any HTTP client. It is repeatable and stateless. Any attacker with network access to the mirror node can execute it. [4](#0-3) 

### Recommendation
1. **Replace `LIKE` with equality for 96-char hashes:** In `getByHashOrNumber()`, check `hash.length === 96` and use `WHERE hash = $1` instead of `WHERE hash LIKE $1`. For 64-char ETH hashes, keep the LIKE prefix scan but ensure a `text_pattern_ops` index exists.
2. **Add a `text_pattern_ops` index** on `record_file(hash text_pattern_ops)` so that LIKE prefix scans use the index regardless of hash length.
3. **Add application-level rate limiting** on the `/api/v1/blocks/:hashOrNumber` route (e.g., via express-rate-limit or an API gateway).

  async getByHashOrNumber(hash, number) {
    let whereStatement = '';
    const params = [];
    if (hash) {
      hash = hash.toLowerCase();
      whereStatement += `${RecordFile.HASH} like $1`;
      params.push(hash + '%');
    } else {
      whereStatement += `${RecordFile.INDEX} = $1`;
      params.push(number);
    }

    const query = `${RecordFileService.blocksQuery} where ${whereStatement}`;
    const row = await super.getSingleRow(query, params);
    return row ? new RecordFile(row) : null;
  }
