### Title
Global Throttle Exhaustion via Default `stack=true` Enabling Unauthenticated Maximum-Cost EVM Re-execution DoS

### Summary
The `getContractOpcodes()` endpoint defaults `stack=true` without user opt-in, making every bare request trigger the most expensive EVM re-execution trace path. The rate limiter protecting this endpoint is a single global token bucket (default: 1 req/sec), not per-IP, meaning a single unauthenticated attacker can monopolize the entire endpoint quota while maximizing per-request CPU and memory consumption, denying service to all legitimate users.

### Finding Description

**Exact code path:**

`OpcodesController.java` line 55 sets `stack` to `true` by default: [1](#0-0) 

This `stack` flag flows directly into `OpcodeRequest` and then into `OpcodeContext`: [2](#0-1) 

At trace time, `captureStack()` in `AbstractOpcodeTracer` iterates the entire EVM stack for every opcode executed during re-execution: [3](#0-2) 

The only guard before this expensive work is `throttleOpcodeRequest()`, which consumes from a **single application-wide** `opcodeRateLimitBucket` bean: [4](#0-3) 

That bucket is a single Spring singleton, not partitioned by IP or identity: [5](#0-4) 

Default capacity is 1 token/second globally: [6](#0-5) 

The only other pre-check is the `Accept-Encoding: gzip` header validation, which is trivially satisfied by any HTTP client: [7](#0-6) 

**Root cause:** The failed assumption is that the global rate limit (1 req/sec) is sufficient to bound resource consumption. It is not, because: (a) the limit is global, not per-source, so one attacker exhausts the entire quota; and (b) `stack=true` by default ensures every token consumed triggers the most expensive possible EVM re-execution path without the attacker needing any knowledge of the tracing options.

**Exploit flow:**
1. Attacker discovers the public endpoint (no auth required).
2. Attacker sends `GET /api/v1/contracts/results/{any_valid_tx_hash}/opcodes` with `Accept-Encoding: gzip` at exactly 1 req/sec — no `stack` parameter needed.
3. Each request passes the gzip check, consumes the single global token, and triggers full EVM re-execution with per-opcode stack capture.
4. The global bucket is drained; all concurrent legitimate users receive HTTP 429.
5. Attacker sustains this indefinitely from a single IP with no credentials.

### Impact Explanation

- **Availability (DoS):** The global throttle means one attacker fully monopolizes the endpoint. All legitimate users are rate-limited out for the duration of the attack.
- **Resource exhaustion:** `stack=true` by default causes the server to allocate and serialize the full EVM operand stack at every opcode step during re-execution. For complex contracts this is significant CPU and heap pressure per request — the attacker gets maximum damage per token consumed.
- **No privilege required:** The endpoint has no authentication, no IP-based limiting, and no CAPTCHA or proof-of-work. The barrier to entry is zero.

Severity: **High** (unauthenticated, repeatable, complete denial of the endpoint to all other users, with maximum per-request server cost).

### Likelihood Explanation

Any attacker who can reach the web3 service (public-facing deployment) can execute this with a single `curl` loop. No special knowledge, credentials, or tooling is required. The `stack=true` default means the attacker does not even need to know about the tracing options — the most expensive path is taken automatically. The attack is fully repeatable and sustainable indefinitely at 1 req/sec.

### Recommendation

1. **Change the `stack` default to `false`** (matching `memory` and `storage`). Stack tracing is explicitly documented as expensive; it should require affirmative opt-in, not opt-out. [8](#0-7) 

2. **Implement per-IP (or per-client) rate limiting** for the opcode endpoint. The current single global bucket allows one source to starve all others. Use a per-key bucket (keyed on `X-Forwarded-For` / remote address) or integrate with an API gateway that enforces per-client limits. [4](#0-3) 

3. **Consider adding authentication/authorization** to the opcodes endpoint, given its inherently expensive nature.

### Proof of Concept

```bash
# No credentials, no special parameters — stack=true fires by default
TX_HASH="<any valid transaction hash in the node>"

while true; do
  curl -s \
    -H "Accept-Encoding: gzip" \
    "http://<mirror-node-web3>/api/v1/contracts/results/${TX_HASH}/opcodes" \
    -o /dev/null -w "%{http_code}\n"
  sleep 1   # stay within the 1 req/sec global limit
done
```

Expected result:
- Attacker receives HTTP 200 on every request (global quota consumed by attacker).
- Any concurrent legitimate request receives HTTP 429 (`Requests per second rate limit exceeded`).
- Server performs full EVM re-execution with stack capture on every attacker request, maximizing CPU/heap usage per token.

    protected final List<String> captureStack(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isStack()) {
            return Collections.emptyList();
        }

        var size = frame.stackSize();
        var stack = new ArrayList<String>(size);
        for (var i = 0; i < size; ++i) {
            var item = frame.getStackItem(size - 1 - i);
            stack.add(hexCache.get(item, Bytes::toHexString));
        }

        return stack;
    }
