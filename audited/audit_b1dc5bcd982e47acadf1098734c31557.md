### Title
Missing Deleted-Entity Filter in `getByEvmAddressAndType()` Long-Zero Address Path

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when the input is a long-zero EVM address (first 12 bytes all zero), the code calls the inherited Spring Data `CrudRepository.findById()`, which has no `deleted is not true` guard. Every other lookup path in the same class (`findByAlias`, `findByEvmAddress`) explicitly filters deleted entities via custom `@Query` annotations, making this an inconsistent and exploitable omission. Any unprivileged caller can supply the long-zero address of a deleted contract and receive its stale metadata as if it were live.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        // ← calls CrudRepository.findById() — NO deleted filter
        return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
    }
    // ← calls findByEvmAddress() which has "deleted is not true"
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–17:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

`findByAlias` and `findByEvmAddress` both carry `deleted is not true`. The inherited `CrudRepository.findById(Long)` carries no such predicate and is never overridden in the graphql `EntityRepository`.

**Root cause:** The long-zero branch at line 38 uses the base Spring Data `findById()` instead of a custom query (or a post-filter on `deleted`) that matches the semantics of the other two lookup methods.

**Exploit flow:**
1. Contract with entity num `N` (long-zero address `0x000000000000000000000000000000000000000N`) is deployed and later deleted on-chain; the mirror node marks `entity.deleted = true` for that row.
2. Attacker (no credentials required) sends a GraphQL query supplying the long-zero hex address encoding `N`.
3. `getByEvmAddressAndType()` detects the long-zero pattern, extracts `N`, and calls `findById(N)`.
4. Spring Data issues `SELECT * FROM entity WHERE id = N` — no deleted predicate — and returns the deleted entity row.
5. The only subsequent check is `.filter(e -> e.getType() == type)`, which passes because the deleted row still carries `type = CONTRACT`.
6. The caller receives a fully-populated `Entity` object for a contract that no longer exists.

### Impact Explanation
Callers of the GraphQL API (dApps, indexers, wallets) that resolve a contract by its canonical long-zero address will receive stale metadata (bytecode hash, admin key, auto-renew settings, balance snapshot, etc.) for a deleted contract. This creates a false "contract is live" signal. In a re-creation scenario — where the same numeric slot is conceptually reused by a new deployment at a different entity ID — the mirror node reports the old, deleted contract identity rather than returning empty, causing smart-contract identity confusion at the mirror/indexer layer with no direct fund loss but with concrete incorrect state propagation to any consumer of this API.

### Likelihood Explanation
Exploitation requires zero privileges: the GraphQL endpoint is public, the long-zero address of any deleted contract is trivially derivable from its entity ID (which is public on-chain), and the query is a single unauthenticated GraphQL call. The condition (a deleted contract exists in the mirror DB) is common in any production network. The bug is deterministic and repeatable.

### Recommendation
Replace the bare `findById()` call in the long-zero branch with a custom repository method that includes the deleted guard, consistent with the other two lookup methods:

```java
// In EntityRepository:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);

// In EntityServiceImpl.getByEvmAddressAndType():
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                           .filter(e -> e.getType() == type);
}
```

The same fix should be applied to `getByIdAndType()` (line 25), which also calls the unguarded `findById()`.

### Proof of Concept
1. Insert a deleted contract entity into the mirror DB:
   ```sql
   INSERT INTO entity (id, num, realm, shard, type, deleted, evm_address, ...)
   VALUES (1500, 1500, 0, 0, 'CONTRACT', true, NULL, ...);
   ```
2. Construct the long-zero hex address for entity num 1500:
   `0x00000000000000000000000000000000000005DC`
3. Send a GraphQL query:
   ```graphql
   query {
     contract(input: { evmAddress: { address: "00000000000000000000000000000000000005DC" } }) {
       id
       deleted
     }
   }
   ```
4. Observe the response returns the deleted contract's data instead of `null`/empty, confirming `findById()` bypasses the deleted filter. [1](#0-0) [2](#0-1)
