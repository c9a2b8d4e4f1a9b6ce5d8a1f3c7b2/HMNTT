### Title
Unbounded DB Load Amplification via `initGenesisRecordFile` Failure-Path Retry Without Backoff or Rate Limiting

### Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` places `sync.Once.Do` exclusively on the success path. When the database is unavailable, every incoming HTTP request re-executes the expensive `selectGenesis` CTE query (joining `account_balance` and `record_file`) with no failure caching, no backoff, and no rate limiting on the Rosetta API layer. An unprivileged attacker can flood the public Rosetta endpoints during any transient DB disruption to amplify load and prevent DB recovery.

### Finding Description
**Exact code path:** `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263.

**Root cause:** The fast-path guard at line 244 (`ConsensusStart != genesisConsensusStartUnset`) only becomes true after `once.Do` sets `br.genesisBlock` at line 257–259. `once.Do` is only reached when the DB query succeeds. On any DB error, the function returns at line 253–254 without ever calling `once.Do`, leaving `br.genesisBlock.ConsensusStart` at `genesisConsensusStartUnset` (-1). The next request passes the guard again and re-executes the full CTE query. There is no failure counter, no cooldown, no cached error state.

**The `selectGenesis` query** (lines 49–72) is a multi-table CTE:
- Subquery on `account_balance` filtered by `treasury_entity_id`, ordered and limited
- Join with `record_file` on `consensus_end > genesis.timestamp`
- Correlated subquery for `consensus_end` adjustment
- `ORDER BY rf.consensus_end LIMIT 1`

This is not a trivial point-lookup; it involves sequential scans or index range scans across two large tables.

**No rate limiting on the Rosetta API:** `main.go` lines 217–219 show the middleware stack is only `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. Unlike the web3 service (which has `ThrottleManagerImpl` with `bucket4j`), the Rosetta API has zero request-rate limiting. Every unauthenticated HTTP POST to `/block`, `/block/transaction`, `/account/balance`, `/network/status`, etc. calls `initGenesisRecordFile`.

**Exploit flow:**
1. Genesis is not yet initialized (startup) OR DB becomes temporarily unavailable (network hiccup, DB restart, connection pool exhaustion).
2. Attacker sends N concurrent POST requests to any block/account endpoint.
3. All N goroutines pass the `ConsensusStart != genesisConsensusStartUnset` guard (it is still -1).
4. All N goroutines execute `selectGenesis` against the DB simultaneously.
5. All N queries fail (DB unavailable) → `once.Do` never fires → state unchanged.
6. Attacker repeats; each wave of N requests fires N expensive CTE queries.
7. When DB recovers, the backlog of retried queries hits it simultaneously, potentially re-triggering overload.

**Why existing checks fail:**
- `sync.Once` is never consumed on the failure path, so it provides zero protection against repeated retries.
- The early-exit guard at line 244 is also unprotected by a mutex, creating a concurrent read/write data race (Go race detector would flag `br.genesisBlock` reads vs. the write in `once.Do`).
- No circuit breaker, no exponential backoff, no per-IP or global rate limit exists in the Rosetta middleware stack.

### Impact Explanation
An attacker can sustain a DB load amplification attack proportional to their HTTP request rate. Because the Rosetta API is unauthenticated and publicly exposed, and because the `selectGenesis` query is a multi-table CTE join, even a modest flood (hundreds of requests/second) translates directly into hundreds of expensive DB queries per second. This can:
- Exhaust the DB connection pool, denying service to all other mirror-node components sharing the same DB.
- Prevent the DB from recovering from a transient fault by immediately re-saturating it upon reconnection.
- Cause cascading failures in the broader mirror-node infrastructure.

Severity: **High** — unauthenticated, no preconditions beyond network access, directly impacts shared DB infrastructure.

### Likelihood Explanation
The Rosetta API is designed to be publicly accessible (it is a standard blockchain data API). No authentication, API key, or IP allowlist is required by default. The attacker needs only:
- Network access to the Rosetta port (default behavior).
- Knowledge that the node is starting up or that the DB is under stress (observable via error responses returning `ErrNodeIsStarting`).

The attack is trivially repeatable with any HTTP load tool (`wrk`, `ab`, `curl` in a loop). The `ErrNodeIsStarting` error code in responses directly signals to an attacker that genesis is not yet initialized, making the timing trivial to detect.

### Recommendation
1. **Cache the failure with a short TTL**: Introduce a `lastFailureTime` field and skip the DB query if a failure occurred within the last N seconds (e.g., 5s), returning the cached error immediately.
2. **Move `once.Do` to wrap the entire initialization attempt**, using a mutex or `sync.Once` with a retry-aware pattern (e.g., `singleflight.Group`) so that concurrent requests collapse into a single in-flight query rather than N parallel queries.
3. **Add rate limiting to the Rosetta API**: Apply a global or per-IP rate limiter in the middleware stack (analogous to the web3 `ThrottleManagerImpl`).
4. **Protect the guard read**: The read of `br.genesisBlock.ConsensusStart` at line 244 and the write in `once.Do` at line 258 are a data race; protect with `sync/atomic` or a `sync.RWMutex`.

func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
	if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
		return nil
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var rb recordBlock
	if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
		First(&rb).Error; err != nil {
		return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
	}

	br.once.Do(func() {
		br.genesisBlock = rb
	})

	log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
	return nil
}
