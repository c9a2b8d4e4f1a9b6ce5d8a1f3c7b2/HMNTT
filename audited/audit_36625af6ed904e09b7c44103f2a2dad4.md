### Title
Unauthenticated Alias Lookup DoS via Unbounded DB Queries in `getAccountCryptoAllowances`

### Summary
Any unauthenticated user can supply a syntactically valid but non-existent base32 account alias as the `idOrAliasOrEvmAddress` path parameter to `/api/v1/accounts/{id}/allowances/crypto`. Each such request unconditionally executes a database query against the `entity` table with no rate limiting and no negative-result caching. At scale, this exhausts the DB connection pool and degrades or halts all mirror-node REST processing.

**What constitutes a "valid" alias:**

`AccountAlias.isValid()` accepts any string matching `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` — the entire uppercase base32 alphabet with optional shard/realm prefix. The valid alias space is effectively unbounded. [5](#0-4) 

**No rate limiting on the REST API:**

The Node.js REST server middleware stack (`rest/server.js`) includes only `requestLogger`, `authHandler` (optional basic-auth), optional metrics, and optional response caching. There is no rate-limiting middleware for unauthenticated requests. [6](#0-5) 

The throttling found in the codebase (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) applies exclusively to the separate web3/contract-call service, not to the Node.js REST API.

**No negative-result caching:**

The JS `EntityService` has no cache layer. Every alias miss goes directly to the DB. The optional response cache (`responseCacheCheckHandler`) is gated on both `config.cache.response.enabled` AND `config.redis.enabled`, and even when active it typically caches only successful (200) responses — not 404s returned for non-existent aliases. [7](#0-6) 

### Impact Explanation

An attacker can generate an unlimited stream of unique, syntactically valid base32 strings (e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`, `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB`, …) and send them as path parameters. Each request triggers a full `SELECT` against the `entity` table. Because the DB connection pool is shared across all REST endpoints (including the importer's write path), saturating it with alias-miss queries degrades or blocks all mirror-node REST processing — including block ingestion queries — for the duration of the attack. No authentication, API key, or special privilege is required.

### Likelihood Explanation

The attack requires only an HTTP client and knowledge of the public API (documented in the OpenAPI spec). The alias format is trivially enumerable. No account, token, or on-chain resource is needed. The attack is stateless and trivially parallelizable across many source IPs, making IP-based blocking ineffective without upstream infrastructure.

### Recommendation

1. **Add rate limiting** to the Node.js REST API for unauthenticated requests (e.g., `express-rate-limit` keyed by IP, applied before route handlers).
2. **Cache negative alias lookups**: introduce a short-lived (e.g., 30 s) in-process or Redis cache for alias misses so repeated queries for the same non-existent alias do not hit the DB.
3. **Require the response cache** (`config.cache.response.enabled` + Redis) to also cache 404 responses for alias-based paths.
4. **Add a DB-level query timeout** for alias lookups to bound worst-case query duration.
