### Title
Global Rate Limit Exhaustion via Unauthenticated Single-Client Token Flooding in `rateLimitBucket`

### Summary
The `rateLimitBucket` in `ThrottleConfiguration` is a single, server-wide shared bucket with a default capacity of 500 tokens. Because there is no per-IP or per-client partitioning, any unauthenticated external caller can exhaust the entire 500-token capacity within one second by sending 500 minimal POST requests, causing every subsequent legitimate request to receive a `ThrottleException` (HTTP 429) for the remainder of that second. This attack is trivially repeatable every second with no privileges required.

### Finding Description
**Code path:**

- `ThrottleConfiguration.rateLimitBucket()` (lines 25–32) creates one Spring singleton `Bucket` with `capacity = requestsPerSecond` (default `500`, `ThrottleProperties` line 35) and `refillGreedy(rateLimit, Duration.ofSeconds(1))`. No `SynchronizationStrategy` is set (unlike `gasLimitBucket`), but bucket4j local buckets are still thread-safe by default.
- `ThrottleManagerImpl.throttle()` (line 38) calls `rateLimitBucket.tryConsume(1)` — one token per request, with no caller identity check.
- `ContractController.call()` (line 40) invokes `throttleManager.throttle(request)` before any authentication or IP check. The endpoint `/api/v1/contracts/call` is publicly accessible with no authentication requirement.
- When `tryConsume(1)` returns `false`, a `ThrottleException` is thrown (line 39), which the controller advice maps to HTTP 429.
- Critically, the rate-limit token is **never restored** on failure — only gas tokens are restored via `throttleManager.restore(request.getGas())` in the `InvalidParametersException` catch block (line 48). The rate-limit slot is permanently consumed for that second.

**Root cause:** The rate limiter is a single global counter shared across all clients. The design assumes the 500 req/s limit is a collective server capacity, but there is no mechanism to prevent one client from consuming the entire budget.

**Exploit flow:**
1. Attacker sends 500 HTTP POST requests to `/api/v1/contracts/call` within one second. Requests can be minimal (e.g., `{"to":"0x...","data":"0x","gas":21000}`).
2. Each request passes `@Valid` bean validation and reaches `throttleManager.throttle()`, consuming one token.
3. After 500 requests, `rateLimitBucket` is empty.
4. All subsequent requests from any client — including legitimate users — hit `!rateLimitBucket.tryConsume(1) == true` and receive HTTP 429 for the rest of the second.
5. The bucket refills after 1 second; the attacker repeats immediately, sustaining the denial indefinitely.

**Why existing checks fail:**
- No per-IP rate limiting exists anywhere in the web3 Java module.
- No authentication is required to reach the throttle check.
- The `RequestProperties` filter system (LOG/REJECT/THROTTLE) is an opt-in operator configuration, disabled by default (`request = List.of()`), and even when enabled it applies a secondary per-pattern bucket — it does not protect the primary `rateLimitBucket`.
- The `gasLimitBucket` secondary check is irrelevant; the attacker is blocked at the rate-limit check, not the gas check.

### Impact Explanation
A single attacker with no credentials can fully deny contract call service to all legitimate users for sustained periods. The `rateLimitBucket` is the sole gating mechanism for `/api/v1/contracts/call`. With the bucket exhausted, no contract reads or estimates can proceed, effectively freezing EVM query throughput for the entire node. The attack requires only a standard HTTP client capable of ~500 req/s — achievable from a single machine. The 500% block-time delay threshold referenced in the question maps directly to this: if the node's contract call endpoint is throttled to zero for repeated 1-second windows, dependent downstream processes (block production monitoring, relay services) stall proportionally.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and no special knowledge beyond the public API path `/api/v1/contracts/call`. It is repeatable every second with a simple loop or any HTTP load tool (e.g., `wrk`, `ab`, `curl` in parallel). The attacker does not need the requests to succeed — they only need to reach `throttleManager.throttle()`. The default capacity of 500 is low enough to be exhausted by a single machine on a LAN or even over the internet with modest bandwidth. There is no IP-based connection limit, no CAPTCHA, and no backoff enforcement on the attacker side.

### Recommendation
1. **Per-IP rate limiting:** Partition the bucket by client IP (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on `request.getRemoteAddr()`, or a Bucket4j `ProxyManager` backed by a cache). Each IP should have its own sub-limit (e.g., 10–50 req/s) that sums to the global cap.
2. **Global + per-IP two-tier throttle:** Keep the global bucket as a hard ceiling, but add a per-IP bucket as a first gate so no single client can consume more than its fair share.
3. **Restore rate-limit tokens on pre-execution failure:** If a request fails before any EVM work is done (validation errors, etc.), restore the rate-limit token, not just the gas token.
4. **Operator-level network controls:** Deploy an ingress rate limiter (e.g., nginx `limit_req`, API gateway, or cloud WAF) as a defense-in-depth layer upstream of the application.

### Proof of Concept
```bash
# Exhaust the 500-token rateLimitBucket in under 1 second from a single unauthenticated client
# Requires: curl, GNU parallel or xargs

ENDPOINT="http://<mirror-node-host>/api/v1/contracts/call"
BODY='{"to":"0x0000000000000000000000000000000000000001","data":"0x","gas":21000}'

# Fire 500 concurrent requests
seq 1 500 | xargs -P 500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST "$ENDPOINT" \
  -H "Content-Type: application/json" \
  -d "$BODY"

# Immediately send a legitimate request — expect HTTP 429
curl -v -X POST "$ENDPOINT" \
  -H "Content-Type: application/json" \
  -d "$BODY"
# Response: 429 Too Many Requests
# Body: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}

# Repeat every second to sustain the denial
while true; do
  seq 1 500 | xargs -P 500 -I{} curl -s -o /dev/null \
    -X POST "$ENDPOINT" -H "Content-Type: application/json" -d "$BODY"
  sleep 0.9
done
```
