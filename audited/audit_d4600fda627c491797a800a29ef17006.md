### Title
Global Gas Bucket Starvation via Absence of Per-IP/Per-User Throttling in ThrottleManagerImpl

### Summary
`ThrottleManagerImpl.throttle()` enforces gas consumption against a single application-wide `gasLimitBucket` singleton with no per-IP, per-user, or per-connection isolation. An unauthenticated attacker can send requests at the maximum allowed rate with maximum gas values, exhausting the global bucket and causing all concurrent legitimate users to receive a `ThrottleException`, effectively denying service for all high-gas contract call simulations.

### Finding Description

**Exact code path:**

In `ThrottleManagerImpl.throttle()` (lines 37–49), every incoming request consumes tokens from the same two global singletons:

```java
if (!rateLimitBucket.tryConsume(1)) {                                          // line 38
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Both `rateLimitBucket` and `gasLimitBucket` are Spring singleton beans defined in `ThrottleConfiguration`:

- `rateLimitBucket` — capacity = `requestsPerSecond` (default **500**)
- `gasLimitBucket` — capacity = `scaleGas(gasPerSecond)` = `7_500_000_000 / 10_000` = **750,000 tokens**

`scaleGas()` in `ThrottleProperties` (lines 42–47) divides gas by `GAS_SCALE_FACTOR = 10_000`. The maximum gas a caller may supply is `maxGasLimit = 15_000_000`, which scales to **1,500 tokens** per request.

**Root cause:** There is no per-source-IP, per-session, or per-identity bucket. All callers share the same two global token buckets. The `ContractCallRequest` model carries no caller identity that is enforced at the throttle layer, and the `RequestProperties` filter system (lines 44–48) is an operator-configured static rule set — not a dynamic per-caller isolation mechanism.

**Exploit arithmetic:**
- Global rate limit: 500 req/sec
- Gas tokens per max-gas request: 1,500
- Gas bucket capacity: 750,000 tokens
- Requests needed to drain gas bucket: 750,000 / 1,500 = **500** — exactly the global rate limit

A single attacker sending 500 requests/second at `gas=15_000_000` simultaneously exhausts both the `rateLimitBucket` (all 500 RPS slots consumed) and the `gasLimitBucket` (all 750,000 tokens consumed). Legitimate users receive `ThrottleException` on every attempt.

**Why `restore()` does not mitigate this:**
`restore()` (lines 59–64) returns unused gas tokens after a request completes. However: (a) the attacker can submit requests that actually consume all gas; (b) even if tokens are partially restored, the attacker continuously re-drains the bucket faster than it refills; (c) the throttle check occurs before execution, so the bucket is drained at request intake, not after.

**Why existing checks are insufficient:**
The `RequestProperties` filter/REJECT/THROTTLE system requires static operator configuration keyed on request fields (`DATA`, `FROM`, `TO`, `GAS`, etc.) — it cannot dynamically identify or isolate a flooding source IP. There is no IP-based or identity-based bucket anywhere in the throttle stack.

### Impact Explanation
All users of the `/api/v1/contracts/call` endpoint share the same global gas budget. A single attacker can hold the gas bucket at zero continuously, causing every other user's high-gas contract call simulation to fail with HTTP 429. This is a complete denial-of-service for the web3 contract call API with no authentication or privilege required. Severity: **High** (unauthenticated, full DoS of a public API endpoint).

### Likelihood Explanation
The attack requires only the ability to send ~500 HTTP POST requests per second to a public endpoint — trivially achievable from a single commodity machine or a small botnet. No credentials, tokens, or special knowledge are needed. The attack is fully repeatable and sustainable indefinitely. The default configuration makes the gas bucket exactly drainable at the global rate limit, leaving zero margin for legitimate traffic.

### Recommendation
1. **Per-IP gas and rate buckets**: Introduce a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `RemoteAddr`) so each source IP has its own gas and rate token buckets.
2. **Per-IP rate cap**: Enforce a per-IP request cap (e.g., 10–50 req/sec) well below the global limit so no single source can monopolize global capacity.
3. **Gas cap per request**: Enforce a hard maximum on `request.getGas()` at the throttle layer (in addition to the EVM-level `maxGasLimit`) to bound per-request token consumption.
4. **Fail-open restore on throttle**: When a request is rejected at the gas check, do not deduct tokens (currently `tryConsume` is non-transactional with the rate check — if rate passes but gas fails, the rate token is already consumed; consider atomic check-and-consume).

### Proof of Concept

```bash
# Exhaust the global gasLimitBucket and rateLimitBucket simultaneously.
# Requires: curl, GNU parallel or similar; target: http://mirror-node-web3:8545

PAYLOAD='{"gas":15000000,"to":"0x0000000000000000000000000000000000000001","data":"0x","block":"latest"}'

# Send 500 max-gas requests per second for 5 seconds
for i in $(seq 1 2500); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST http://mirror-node-web3:8545/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait

# Legitimate user attempt — will receive 429 "Gas per second rate limit exceeded."
curl -v -X POST http://mirror-node-web3:8545/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"gas":21000,"to":"0x0000000000000000000000000000000000000001","data":"0x","block":"latest"}'
```

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
