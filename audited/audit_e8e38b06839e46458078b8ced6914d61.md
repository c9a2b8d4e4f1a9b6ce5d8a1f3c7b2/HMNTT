### Title
Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode` Allows Unprivileged User to Freeze Contract Bytecode as Empty for Up to 1 Hour

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode` uses `unless = "#result == null"` to suppress caching, but the method returns `Optional<byte[]>`, not `byte[]`. `Optional.empty()` is a non-null object and is therefore cached. When a contract row exists in the DB with `runtime_bytecode = NULL` (a legitimate state produced by failed/reverted contract creation transactions where no sidecar bytecode is present), any unprivileged user who queries that contractId causes `Optional.empty()` to be stored in the cache for up to 1 hour. Subsequent queries — including after the importer later populates the bytecode — return the stale empty result, causing the EVM to treat the contract as having no code.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause — failed assumption:**
The `unless` SpEL expression `#result == null` evaluates `#result` as the `Optional<byte[]>` wrapper object. `Optional.empty()` is never `null`; it is a singleton non-null object. Therefore the condition is always `false` for an empty Optional, and Spring caches it. The developer assumed `unless = "#result == null"` would prevent caching of absent results, but it only prevents caching of a literal `null` return value.

**How `runtime_bytecode = NULL` enters the DB:**
`ContractCreateTransactionHandler.createContract()` (lines 122–153) creates a `Contract` object and only sets `runtimeBytecode` if a matching non-migration sidecar bytecode record is found: [2](#0-1) 

If no sidecar is present (e.g., failed/reverted transaction, missing sidecar file), `runtimeBytecode` stays `null` and the contract is still persisted via `entityListener.onContract(contract)`. This is an explicitly supported state — tests directly construct contracts with `runtimeBytecode(null)`: [3](#0-2) 

**Cache TTL:**
The contract cache is configured with `expireAfterAccess=1h`: [4](#0-3) 

**Downstream consumer of the poisoned cache:**
`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode` and uses `.orElse(null)`, returning `null` Bytecode to the EVM: [5](#0-4) 

The EVM then treats the contract as having no deployed code.

**Exploit flow:**
1. A contract creation transaction fails or its sidecar is absent → importer writes a contract row with `runtime_bytecode = NULL`.
2. Attacker (or any user) sends an `eth_call` / `eth_estimateGas` targeting that contractId.
3. `findRuntimeBytecode(contractId)` executes the native query, gets a row with NULL → Spring Data JPA returns `Optional.empty()`.
4. `Optional.empty()` passes the `unless = "#result == null"` guard (it is not null) and is stored in the Caffeine cache.
5. The importer later populates `runtime_bytecode` (e.g., via `SidecarContractMigration`).
6. For up to 1 hour after last access, all calls to `findRuntimeBytecode(contractId)` return the cached `Optional.empty()` without hitting the DB.
7. `ContractBytecodeReadableKVState` returns `null` → EVM sees the contract as codeless → all `eth_call` / `eth_estimateGas` requests to that contract fail or return incorrect results.

**Existing check shown insufficient:**
`unless = "#result == null"` — `Optional.empty()` is not null. The correct guard would be `unless = "#result == null || !#result.isPresent()"`.

### Impact Explanation
Any valid contract whose bytecode was temporarily absent at the time of first query becomes permanently invisible to the EVM layer for up to 1 hour. `eth_call` and `eth_estimateGas` requests to that contract will behave as if the contract has no code (STOP opcode / empty return), causing DeFi integrations, wallets, and tooling relying on the mirror node's JSON-RPC endpoint to receive incorrect results. The window is bounded by the 1-hour `expireAfterAccess` TTL but is refreshed on every access, meaning active polling extends the outage indefinitely.

### Likelihood Explanation
No privileges are required. Any user who can send a JSON-RPC request (`eth_call`, `eth_estimateGas`, `eth_getCode`) to the mirror node's web3 endpoint can trigger the cache poisoning. The precondition — a contract row with `null` runtime_bytecode — is a normal operational state that occurs whenever a contract creation transaction is processed before its sidecar file arrives, or when the transaction reverts but the importer still writes the entity. This is not a contrived edge case; it is explicitly handled in production code and tests.

### Recommendation
Fix the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

Alternatively, change the return type to `byte[]` (returning `null` for absent values) and keep `unless = "#result == null"`. Additionally, consider reducing the `expireAfterAccess=1h` TTL for the contract cache or switching to `expireAfterWrite` to bound the staleness window regardless of access patterns.

### Proof of Concept
1. Insert a contract row with `runtime_bytecode = NULL`:
   ```sql
   INSERT INTO contract (id) VALUES (12345);
   ```
2. Send an `eth_call` to the mirror node web3 endpoint targeting contract `0.0.12345`. Observe `Optional.empty()` is returned and cached.
3. Update the row with real bytecode:
   ```sql
   UPDATE contract SET runtime_bytecode = '\x6080604052' WHERE id = 12345;
   ```
4. Immediately send another `eth_call` to `0.0.12345`. Observe the response still reflects no code (cache hit on `Optional.empty()`).
5. Wait for the cache to expire (up to 1 hour with no intervening access) and repeat step 4 — the correct bytecode is now returned.
