### Title
Timestamp Nanosecond Overflow in `getBlocks()` Causes PostgreSQL Bigint Error via Unauthenticated Request

### Summary
`parseTimestampParam()` in `rest/utils.js` converts a user-supplied timestamp string to a raw nanosecond string by simple concatenation of the seconds and nanosecond parts, with no upper-bound check on the resulting value. The upstream validator `isValidTimestampParam()` only enforces digit-count limits (≤10 digits for seconds, ≤9 for nanoseconds), which permits a seconds value of `9999999999`. The resulting nanosecond string `9999999999999999999` (≈10¹⁹) exceeds PostgreSQL's `bigint` maximum of `9223372036854775807` (≈9.22×10¹⁸), causing the database to throw an out-of-range error when the value is bound to the `consensus_end` column comparison in `getBlocks()`.

### Finding Description

**Validation — `rest/utils.js` lines 151–154:**
```js
const isValidTimestampParam = (timestamp) => {
  return /^\d{1,10}$/.test(timestamp) || /^\d{1,10}\.\d{1,9}$/.test(timestamp);
};
```
The regex permits any 10-digit seconds value up to `9999999999` and any 9-digit nanosecond suffix up to `999999999`. No numeric range check is performed.

**Conversion — `rest/utils.js` lines 490–507:**
```js
const parseTimestampParam = (timestampParam) => {
  const tsSplit = timestampParam.split('.');
  const seconds = /^(\d)+$/.test(tsSplit[0]) ? tsSplit[0] : 0;
  let nanos = tsSplit.length === 2 && /^(\d)+$/.test(tsSplit[1]) ? tsSplit[1] : 0;
  nanos = `${nanos}000000000`.substring(0, 9);
  return `${seconds}${nanos}`;   // pure string concatenation, no overflow guard
};
```
For input `9999999999.999999999`, this returns the string `"9999999999999999999"`.

**Filter construction — `rest/controllers/blockController.js` lines 101–104:**
```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);
```
`buildAndValidateFilters` is called with no custom validator, so the default `filterValidityChecks` applies. The parsed nanosecond string is stored as `filter.value`.

**Where-clause assembly — `rest/controllers/baseController.js` lines 231–236:**
```js
getFilterWhereCondition = (key, filter) => {
  return { query: `${key} ${filter.operator}`, param: filter.value };
};
```
`filter.value` (`"9999999999999999999"`) is passed verbatim as a bound parameter.

**Query execution — `rest/service/recordFileService.js` lines 149–161:**
```js
async getBlocks(filters) {
  const {where, params} = buildWhereSqlStatement(filters.whereQuery);
  const query = RecordFileService.blocksQuery + `${where} order by ...`;
  const rows = await super.getRows(query, params);
```
The string `"9999999999999999999"` is sent to PostgreSQL as a parameter for the `consensus_end bigint` column. PostgreSQL attempts to cast it to `bigint`, fails with `ERROR: value "9999999999999999999" is out of range for type bigint`, and the request terminates with an unhandled 500 error.

**Root cause:** `isValidTimestampParam` enforces only digit-count constraints, not a numeric ceiling. The combined nanosecond value for any seconds input above `9223372036` overflows PostgreSQL `bigint`. The threshold is easily reachable: `9999999999` (10 digits, allowed by the regex) is the obvious maximum, but even `9223372037.000000000` is sufficient to trigger the overflow.

### Impact Explanation
Any unauthenticated caller can force a PostgreSQL error on every `GET /api/v1/blocks?timestamp=gte:9999999999.999999999` request. Smart contract clients (e.g., JSON-RPC relay, EVM tooling) that query block ranges via this endpoint receive a 500 error instead of block data, breaking block-range lookups and any dependent smart contract behavior that relies on accurate block metadata. The endpoint is publicly accessible with no authentication requirement.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single HTTP GET request. The payload is trivially constructed from the documented timestamp format. It is fully repeatable and can be automated. Any attacker aware of the API can trigger it on demand.

### Recommendation
Add an explicit upper-bound check on the combined nanosecond value inside `parseTimestampParam` (or in `isValidTimestampParam`) before the value is stored in the filter:

```js
const MAX_BIGINT_NS = 9223372036854775807n;

const parseTimestampParam = (timestampParam) => {
  // ... existing split/regex logic ...
  const combined = BigInt(seconds) * 1_000_000_000n + BigInt(nanos);
  if (combined > MAX_BIGINT_NS) return '';   // treat as invalid
  return `${seconds}${nanos}`;
};
```

Alternatively, enforce the ceiling in `isValidTimestampParam` by parsing the numeric value and comparing against `9223372036854775807`. The OpenAPI spec pattern `^\d{1,10}(\.\d{1,9})?$` should also be tightened to reflect the actual valid range.

### Proof of Concept
```
GET /api/v1/blocks?timestamp=gte:9999999999.999999999
```
1. `isValidTimestampParam("9999999999.999999999")` → `true` (10-digit seconds, 9-digit nanos — both within regex limits).
2. `parseTimestampParam("9999999999.999999999")` → `"9999999999999999999"`.
3. `buildAndValidateFilters` stores `{key: "timestamp", operator: ">=", value: "9999999999999999999"}`.
4. `getFilterWhereCondition` produces `{query: "consensus_end >=", param: "9999999999999999999"}`.
5. PostgreSQL receives: `SELECT … FROM record_file WHERE consensus_end >= $1` with `$1 = "9999999999999999999"`.
6. PostgreSQL throws: `ERROR: value "9999999999999999999" is out of range for type bigint`.

```javascript
const parseTimestampParam = (timestampParam) => {
  // Expect timestamp input as (a) just seconds,
  // (b) seconds.mmm (3-digit milliseconds),
  // or (c) seconds.nnnnnnnnn (9-digit nanoseconds)
  // Convert all of these formats to (seconds * 10^9 + nanoseconds) format,
  // after validating that all characters are digits
  if (!timestampParam) {
    return '';
  }
  const tsSplit = timestampParam.split('.');
  if (tsSplit.length <= 0 || tsSplit.length > 2) {
    return '';
  }
  const seconds = /^(\d)+$/.test(tsSplit[0]) ? tsSplit[0] : 0;
  let nanos = tsSplit.length === 2 && /^(\d)+$/.test(tsSplit[1]) ? tsSplit[1] : 0;
  nanos = `${nanos}000000000`.substring(0, 9);
  return `${seconds}${nanos}`;
};
```
