### Title
Missing Length Validation on Base32 Alias Allows Unbounded Byte Array Passed to DB Query

### Summary
The `getByAliasAndType()` method in `EntityServiceImpl.java` decodes a user-supplied Base32 alias string with no length constraint, producing an arbitrarily large `byte[]` that is passed directly to a native SQL query. The only input validation is a character-set regex (`@Pattern(regexp: "^[A-Z2-7]+$")`) that enforces valid Base32 alphabet characters but imposes no upper bound on string length. An unauthenticated attacker can repeatedly submit oversized alias strings to force large memory allocations and large parameter payloads sent to the database per request.

### Finding Description
**Code path:**

1. `AccountController.java` line 48 calls `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)` with the raw user-supplied string. [1](#0-0) 

2. `EntityServiceImpl.java` line 30 calls `decodeBase32(alias)` with no length check before passing the result to the repository. [2](#0-1) 

3. `GraphQlUtils.decodeBase32()` (line 83–85) is a one-liner with zero length validation — it blindly delegates to Apache Commons `Base32.decode()`. [3](#0-2) 

4. The decoded `byte[]` is passed to `EntityRepository.findByAlias()`, which executes a native query `select * from entity where alias = ?1 and deleted is not true`, transmitting the full byte array as a bound parameter to PostgreSQL. [4](#0-3) 

**Root cause:** The `AccountInput.alias` field in the GraphQL schema carries only a character-set constraint:

```
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
``` [5](#0-4) 

There is no `@Size` / `@Length` / max-length directive. Any string composed entirely of `[A-Z2-7]` characters passes validation regardless of length.

**Failed assumption:** The code assumes that a syntactically valid Base32 string is also a semantically bounded alias. Legitimate Hedera aliases are at most 33 bytes (ECDSA key) → ~56 Base32 characters. The code enforces no such bound.

### Impact Explanation
Each request with a large alias string causes:
- **JVM heap allocation** proportional to the decoded byte array (~5/8 of the input string length).
- **JDBC network I/O**: the full byte array is serialized and sent to PostgreSQL as a query parameter on every request.
- **PostgreSQL work**: receiving and comparing a large `bytea` parameter against the indexed `alias` column (the comparison is O(n) in parameter size for any potential match candidate).

With concurrent requests carrying, e.g., 1 MB Base32 strings (~625 KB decoded), the application server experiences sustained heap pressure and elevated DB connection I/O. Because the GraphQL endpoint is publicly accessible with no authentication, this is trivially repeatable. The absence of any visible rate limiting in the GraphQL layer compounds the risk.

### Likelihood Explanation
- **No authentication required**: the `account` GraphQL query is a public read endpoint.
- **No rate limiting visible** in the reviewed code path.
- **Trivial to automate**: a simple HTTP client loop sending large valid Base32 strings suffices.
- **Low attacker skill**: the exploit requires only knowledge of the GraphQL schema (publicly discoverable via introspection).

### Recommendation
1. Add a maximum-length constraint on the alias field in the schema:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]{1,56}$")
   ```
   56 characters covers the largest legitimate Hedera alias (33-byte ECDSA key encoded in Base32).

2. Add an explicit length guard in `decodeBase32()` or at the service layer before decoding:
   ```java
   if (base32.length() > 56) {
       throw new IllegalArgumentException("Alias exceeds maximum length");
   }
   return BASE32.decode(base32);
   ```

3. Configure HTTP request body size limits and GraphQL query depth/complexity limits at the framework level as defense-in-depth.

### Proof of Concept
```bash
# Generate a 100,000-character valid Base32 string (all 'A's are valid Base32)
ALIAS=$(python3 -c "print('A' * 100000)")

curl -X POST https://<mirror-node-host>/graphql/alpha \
  -H "Content-Type: application/json" \
  -d "{\"query\": \"{ account(input: {alias: \\\"$ALIAS\\\"}) { entityId { num } } }\"}"
```

Repeat in a loop with concurrent workers. Each request forces the server to allocate ~62,500 bytes, decode them, and transmit the full byte array to PostgreSQL as a query parameter. Scaling to hundreds of concurrent requests produces measurable heap and DB I/O pressure with no credentials required.
