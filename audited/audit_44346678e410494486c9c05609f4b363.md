### Title
Unauthenticated Entity ID Enumeration via Zero-Prefixed EVM Address Triggers Unbounded DB Scan in GraphQL `getByEvmAddressAndType()`

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl.java` routes any EVM address whose first 12 bytes are zero directly to a `findById()` database primary-key lookup. The GraphQL module has no rate limiting (unlike the `web3` module which uses bucket4j), so an unauthenticated attacker can send an unlimited stream of requests with sequentially incremented zero-prefixed EVM addresses, issuing one DB read per request, saturating the connection pool and degrading service for all users while also enumerating all valid entity IDs.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // first 12 bytes == 0
        return entityRepository.findById(buffer.getLong())         // PK lookup, attacker-controlled
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

The `findById()` call maps directly to `SELECT * FROM entity WHERE id = ?` (Spring Data `CrudRepository`). The type filter is applied in Java *after* the full row is fetched, so every request unconditionally hits the database.

**Root cause / failed assumption:** The code correctly identifies the "long-form" EVM address encoding but assumes callers are legitimate. There is no rate limiting, authentication, or per-IP throttle in the GraphQL module. The `web3` module has a full `ThrottleConfiguration` (bucket4j, `requestsPerSecond`, gas limits), but the `graphql` module has none — only query-complexity and query-depth limits that apply *per single request*, not across requests.

**Exploit flow:**
1. Attacker crafts GraphQL queries with `evmAddress: "0x000000000000000000000000000000000000000N"` (N = 1, 2, 3, …).
2. Each address passes the `buffer.getInt()==0 && buffer.getLong()==0` check, routing to `findById(N)`.
3. Each request issues one synchronous DB read; no caching layer is present.
4. With concurrent connections the attacker exhausts the HikariCP connection pool (monitored by `GraphQLHighDBConnections` alert at 75% utilization), blocking legitimate queries.

**Why existing checks are insufficient:**

| Control | What it limits | What it does NOT limit |
|---|---|---|
| `MaxQueryComplexityInstrumentation(200)` | Fields/resolvers per single query | Request rate |
| `MaxQueryDepthInstrumentation(10)` | Nesting depth per query | Request rate |
| Jackson/parser size limits | Byte size of one request | Request rate |
| No auth required | — | — |
| No IP throttle | — | — |

### Impact Explanation
An attacker can enumerate every valid entity ID in the system (information disclosure) and simultaneously saturate DB read capacity, causing elevated latency or timeouts for all legitimate GraphQL users. The HikariCP pool is finite; once exhausted, all queries queue or fail. This is a griefing/availability impact with no economic cost to the attacker.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The zero-prefix EVM address pattern is documented in the Hedera EVM address specification and is publicly known. A single machine with a modest HTTP client (e.g., `wrk`, `hey`, or a simple loop) can issue thousands of requests per second. The attack is trivially repeatable and requires no state.

### Recommendation
1. **Add rate limiting to the GraphQL module** mirroring the `web3` module's `ThrottleConfiguration` — apply a per-IP or global requests-per-second bucket using bucket4j at the HTTP filter layer.
2. **Add a DB-level index hint or result cache** (e.g., Spring `@Cacheable`) on `findById` lookups for the GraphQL service to reduce per-request DB cost.
3. **Move the type filter into the SQL query** (add `AND type = ?` to the native query) so the DB can short-circuit and avoid returning full rows for mismatched types.
4. **Require authentication or an API key** for the GraphQL endpoint, consistent with the access controls on other mirror-node services.

### Proof of Concept
```bash
# Sequential entity ID enumeration via zero-prefixed EVM addresses
# Each address: 0x + 24 zero hex chars (12 zero bytes) + 8-hex-char sequential ID

for i in $(seq 1 100000); do
  ID=$(printf '%016x' $i)
  ADDR="0x000000000000000000000000${ID}"
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { evmAddress: \\\"${ADDR}\\\" }) { id } }\"}" &
done
wait
```

Each iteration triggers `entityRepository.findById(i)` — a full DB primary-key read — with no server-side throttle to stop it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
