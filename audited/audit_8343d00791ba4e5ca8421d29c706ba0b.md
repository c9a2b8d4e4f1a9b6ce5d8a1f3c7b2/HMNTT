### Title
Connection Pool Exhaustion via Unbounded Concurrent Requests to `TopicRepository.findById()` During Database Degradation

### Summary
The `rest-java` service exposes `/api/v1/topics/{id}` with no application-level rate limiting or circuit breaker on the database call path. During database degradation, each request holds a HikariCP connection for up to the full `statementTimeout` (10 seconds). An unprivileged attacker flooding this endpoint during a degradation event can exhaust the connection pool before the Traefik ingress circuit breaker reactively trips, and the configured Traefik retry policy (3 attempts per request) amplifies the pool pressure by a factor of 3, extending the outage.

### Finding Description

**Code path:**

`TopicController.getTopic()` → `TopicServiceImpl.findById()` → `TopicRepository.findById()` (Spring Data `CrudRepository`) [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — three compounding gaps:**

1. **No application-level rate limiting in rest-java.** The `web3` module has a full `ThrottleConfiguration` with bucket4j, but rest-java has no equivalent. There is no `@RateLimiter`, no `ThrottleManager`, and no request-count guard anywhere in the `rest-java` module. [4](#0-3) 

2. **No application-level circuit breaker.** A search across all `rest-java/**` sources finds zero references to `CircuitBreaker`, `resilience4j`, or any equivalent. The repository call is a raw Spring Data proxy with no fallback. [5](#0-4) 

3. **Traefik retry amplifies pool pressure.** The Helm chart configures `attempts: 3` on the retry middleware. Each user-visible request that times out is retried 3 times by Traefik before the circuit breaker trips, tripling the number of concurrent HikariCP connections consumed per attacker request. [6](#0-5) 

**Why the Traefik circuit breaker is insufficient:**

The circuit breaker expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` only trips *after* 25% of responses are already 5xx. [7](#0-6) 

During the degradation window — slow queries that have not yet hit the 10-second `statementTimeout` — responses are not yet 5xx; they are simply slow. The circuit breaker does not trip on latency, only on error ratios. During this window, connections accumulate in the HikariCP pool. [8](#0-7) 

HikariCP is configured via `CommonConfiguration` with no explicit `maximumPoolSize` override for rest-java, defaulting to 10 connections. [9](#0-8) 

The `RestJavaHighDBConnections` Prometheus alert is `enabled: false` by default, so operators receive no automated warning. [10](#0-9) 

### Impact Explanation

When the HikariCP pool (default 10 connections) is exhausted, all subsequent requests to any endpoint in the `rest-java` service — not just `/api/v1/topics/{id}` — queue waiting for a connection. Once the queue fills, requests fail with 503/500. This converts a partial database degradation (slow queries) into a complete service outage for all users of the `rest-java` service. The Traefik retry policy means each attacker connection attempt consumes 3 pool slots, so only 3–4 concurrent attacker connections are needed to saturate the default pool. The outage persists until the database recovers AND the pool drains, which is longer than the underlying DB issue alone.

### Likelihood Explanation

The `/api/v1/topics/{id}` endpoint is publicly accessible with no authentication. Any unprivileged user can send concurrent HTTP GET requests. The attacker does not need to cause the database degradation — they only need to observe it (e.g., via elevated latency on public responses) and then flood the endpoint. The attack requires no special tooling: `ab`, `wrk`, or even a browser-based script sending parallel `fetch()` calls suffices. The GCP backend policy `maxRatePerEndpoint: 250` provides some protection but applies per-endpoint at the load balancer level and does not prevent pool exhaustion from a moderate number of concurrent slow-query-holding connections. [11](#0-10) 

### Recommendation

1. **Add application-level rate limiting to rest-java** mirroring the `web3` bucket4j `ThrottleConfiguration`, applied via a servlet filter or Spring MVC interceptor on all endpoints.
2. **Wrap the repository call with a Resilience4j `@CircuitBreaker` and `@Bulkhead`** in `TopicServiceImpl.findById()` to shed load and open the circuit when the DB is degraded, returning a fast 503 instead of holding a connection.
3. **Reduce the Traefik retry `attempts` from 3 to 1** (or 0) for endpoints that are idempotent reads but whose backend is already degraded — retries amplify pool pressure.
4. **Enable the `RestJavaHighDBConnections` Prometheus alert** (`enabled: true`) so operators are notified before pool exhaustion occurs.
5. **Explicitly configure `spring.datasource.hikari.maximum-pool-size`** for rest-java to a value informed by expected concurrency, and set `connection-timeout` to a low value (e.g., 2 seconds) so pool-exhausted requests fail fast rather than queuing.

### Proof of Concept

**Precondition:** Database is degraded (e.g., a long-running vacuum, replica lag, or network partition causing slow query execution — queries take 8–10 seconds before hitting `statementTimeout`).

**Steps:**

```bash
# 1. Confirm the endpoint is publicly reachable
curl https://<mirror-node>/api/v1/topics/0.0.1

# 2. During a DB degradation event, flood the endpoint with concurrent requests
# 15 concurrent requests × 3 Traefik retries = up to 45 simultaneous DB connection attempts
# Default HikariCP pool = 10 connections → pool exhausted in <1 second
ab -n 1000 -c 15 https://<mirror-node>/api/v1/topics/0.0.1

# 3. Observe that ALL rest-java endpoints (not just /topics) now return 500/503
curl https://<mirror-node>/api/v1/network/supply
# → 500 Internal Server Error (HikariCP connection timeout)

# 4. Observe that the outage persists for ~10 seconds after the DB recovers
# (connections held by in-flight queries must drain before pool is available)
```

**Expected result:** The `rest-java` service is completely unavailable to all users for the duration of the DB degradation plus the HikariCP drain period, whereas without the flood it would have degraded gracefully (slow responses, not total failure).

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
