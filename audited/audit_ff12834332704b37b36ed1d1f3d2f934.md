### Title
Per-Node In-Memory Rate Limit Bucket Allows N×500 RPS Bypass in Horizontally Scaled Deployments

### Summary
`rateLimitBucket()` in `ThrottleConfiguration.java` constructs a local, in-memory Bucket4j instance per JVM process. In a horizontally scaled deployment with N nodes, each node independently enforces its own 500 RPS cap with no cross-node coordination. An unprivileged attacker distributing requests across N nodes can sustain N×500 RPS total without any single node ever triggering its throttle, defeating the intended global rate limit.

### Finding Description
**Exact code location:**
- `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `rateLimitBucket()`, lines 24–32
- `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, line 38

**Root cause:**

```java
// ThrottleConfiguration.java lines 24-32
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default: 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build(); // LOCAL, in-memory only
}
```

`Bucket.builder().addLimit(limit).build()` creates a purely local, in-process Bucket4j bucket. There is no distributed backend (Redis, Hazelcast, etc.) configured. Contrast this with `gasLimitBucket()` which at least adds `SynchronizationStrategy.SYNCHRONIZED` (still local, but thread-safe within one JVM). The `rateLimitBucket` has no such annotation and no distributed state.

**Enforcement point:**

```java
// ThrottleManagerImpl.java line 38
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

`tryConsume(1)` only consults the local bucket. Node A has no knowledge of tokens consumed on Node B.

**Failed assumption:** The code assumes a single deployment node, or that an external load balancer will enforce a global cap. Neither assumption is enforced in code.

### Impact Explanation
With N horizontally scaled nodes (a common production configuration), the effective global rate limit becomes N×500 RPS. An attacker sending 500 RPS to each of N nodes saturates backend contract-call processing (EVM execution, database I/O) at N×500 RPS while every individual node reports normal operation and raises no throttle alarm. At sufficient N, this degrades or shuts down ≥30% of total network processing capacity without any brute-force detection triggering, matching the stated severity scope.

### Likelihood Explanation
- **Preconditions:** None. No authentication, no special privileges, no insider knowledge beyond the public JSON-RPC endpoint URLs of each node (discoverable via DNS, load balancer health endpoints, or public infrastructure listings).
- **Feasibility:** Standard HTTP load-testing tools (e.g., `wrk`, `vegeta`, `k6`) can target multiple endpoints simultaneously. Distributing 500 RPS per node is trivially achievable from a single attacker machine or a small botnet.
- **Repeatability:** The attack is stateless and continuously repeatable; there is no lockout or IP-ban mechanism visible in the throttle path.
- **Detection difficulty:** Each node logs normal operation; no single node's metrics surface the aggregate overload.

### Recommendation
Replace the local Bucket4j bucket with a **distributed** Bucket4j backend (e.g., `bucket4j-redis`, `bucket4j-hazelcast`) so all nodes share a single token pool. Alternatively, enforce a global rate limit at the ingress layer (API gateway / load balancer) before requests reach individual nodes. The fix must be applied to all three buckets (`rateLimitBucket`, `opcodeRateLimitBucket`, and `gasLimitBucket`) since all three are local.

Example for Redis-backed distributed bucket:
```java
// Use ProxyManager with a Redis/Lettuce or Redisson backend
BucketConfiguration config = BucketConfiguration.builder()
    .addLimit(Bandwidth.builder()
        .capacity(rateLimit)
        .refillGreedy(rateLimit, Duration.ofSeconds(1))
        .build())
    .build();
return proxyManager.builder().build(KEY, () -> config);
```

### Proof of Concept
**Preconditions:** N=3 mirror-node web3 instances behind a round-robin load balancer, each with default `requestsPerSecond=500`.

**Steps:**
1. Identify the individual node endpoints (e.g., `node1:8545`, `node2:8545`, `node3:8545`).
2. Launch three parallel load streams, each targeting one node at exactly 500 RPS:
   ```bash
   vegeta attack -rate=500 -duration=60s -targets=node1_targets.txt | vegeta report &
   vegeta attack -rate=500 -duration=60s -targets=node2_targets.txt | vegeta report &
   vegeta attack -rate=500 -duration=60s -targets=node3_targets.txt | vegeta report &
   ```
3. **Observed result:** All three streams receive HTTP 200 responses (no 429s); aggregate throughput is 1,500 RPS of EVM contract-call processing.
4. **Expected result (with correct global throttle):** Combined throughput across all nodes should be capped at 500 RPS total; excess requests should receive HTTP 429.
5. Backend CPU/DB load climbs to 3× the intended maximum, degrading or crashing dependent processing pipelines.
