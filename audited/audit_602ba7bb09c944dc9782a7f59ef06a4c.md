### Title
Unauthenticated Contract-Creation Code Path Triggered via Empty `to` Field Bypassing `hasTo()` Validation

### Summary
An unprivileged external user can submit a `POST /api/v1/contracts/call` request with `to = ""` (empty string) and `value = 0`, which passes all bean-validation checks. Inside `constructServiceParameters()`, the empty `to` causes `receiver = Address.ZERO`, which `TransactionExecutionService.execute()` interprets as a contract-creation transaction. When combined with `estimate = true`, the binary-gas-estimator re-executes the contract-creation EVM path ~10 additional times per request, while the gas throttle is consumed only once, creating a computation-to-throttle-token imbalance that can be used to exhaust server CPU.

### Finding Description

**Step 1 – Validation allows empty `to` when `value = 0`**

`ContractCallRequest.java` declares:
```java
@Hex(minLength = ADDRESS_LENGTH, maxLength = ADDRESS_LENGTH, allowEmpty = true)
private String to;
``` [1](#0-0) 

The cross-field guard `hasTo()` is:
```java
private boolean hasTo() {
    boolean isValidToField = value <= 0 || from == null || StringUtils.isNotEmpty(to);
    return BytecodeUtils.isValidInitBytecode(data) || isValidToField;
}
``` [2](#0-1) 

When `value = 0` (the default), `isValidToField` is unconditionally `true`, so `to = ""` or `to = null` passes validation. The controller test `estimateGas` explicitly confirms this with `@NullAndEmptySource` + `value = 0` expecting HTTP 200. [3](#0-2) 

**Step 2 – Empty `to` sets `receiver = Address.ZERO`**

```java
if (request.getTo() == null || request.getTo().isEmpty()) {
    receiver = Address.ZERO;
}
``` [4](#0-3) 

**Step 3 – `Address.ZERO` receiver unconditionally routes to contract-creation path**

```java
final var isContractCreate = params.getReceiver().isZero();
...
if (isContractCreate) {
    transactionBody = buildContractCreateTransactionBody(params, estimatedGas, maxLifetime);
} else {
    transactionBody = buildContractCallTransactionBody(params, estimatedGas);
}
``` [5](#0-4) 

**Step 4 – `estimate = true` multiplies EVM executions via binary search**

`ContractExecutionService.estimateGas()` first calls `callContract()` once, then runs `binaryGasEstimator.search()` which invokes `doProcessCall()` approximately `log2(maxGas / gasUsed)` additional times — up to ~10 iterations for a 15 M gas request. [6](#0-5) 

**Step 5 – Throttle is consumed only once per request**

`ThrottleManagerImpl.throttle()` deducts gas tokens once before `processCall()` is called:
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
``` [7](#0-6) 

The binary-search iterations inside `estimateGas()` are invisible to the throttle. Each iteration executes the heavier `buildContractCreateTransactionBody` path, but only one gas-bucket deduction occurs. The `restore()` path is only triggered for `InvalidParametersException`, not for normal execution. [8](#0-7) 

### Impact Explanation
Each crafted request with `to = ""`, `value = 0`, `estimate = true`, `gas = 15_000_000` causes the server to execute the contract-creation EVM path up to ~11 times while consuming only one unit of the gas-throttle budget. The default `gasPerSecond = 7_500_000_000` and `requestsPerSecond = 500` are calibrated for normal call workloads; the ~11× amplification factor for estimate-with-contract-create means the actual CPU work per throttle token is an order of magnitude higher than intended. At scale (many concurrent attackers or a single attacker at the 500 req/s rate limit), this can saturate CPU on mirror-node instances and degrade or halt contract-call processing for legitimate users.

### Likelihood Explanation
No authentication, API key, or privileged role is required. The endpoint is publicly reachable. The payload is trivial: a standard JSON body with `to` omitted or set to `""` and `estimate: true`. The attack is fully repeatable and scriptable. The only friction is the 500 req/s rate limit, which still allows sustained amplified load.

### Recommendation
1. **Reject empty `to` for non-deploy calls explicitly**: In `hasTo()`, tighten the condition so that `to` may only be absent when `BytecodeUtils.isValidInitBytecode(data)` is true — i.e., require valid init-bytecode to accompany a missing `to`, regardless of `value`.
2. **Throttle per EVM execution, not per HTTP request**: Move gas-bucket consumption inside `doProcessCall()` so that each binary-search iteration deducts tokens, or cap the number of binary-search iterations and account for them in the upfront deduction.
3. **Separate estimate and call gas budgets**: Apply a stricter per-second cap for `estimate = true` requests, since they inherently execute the EVM multiple times.

### Proof of Concept
```bash
# Trigger contract-creation path with binary-search amplification, no auth required
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{
    "to": "",
    "from": null,
    "value": 0,
    "gas": 15000000,
    "estimate": true,
    "data": "0x"
  }'
# Returns HTTP 200; server executes buildContractCreateTransactionBody ~11 times.
# Repeat at 500 req/s (rate limit) to sustain amplified CPU load.
```

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
```
