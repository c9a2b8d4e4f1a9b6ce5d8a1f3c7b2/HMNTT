### Title
Unauthenticated Thundering-Herd DoS via Unbounded Concurrent DB Lookups in `getAccountTokenAllowances`

### Summary
The `/accounts/:idOrAliasOrEvmAddress/allowances/tokens` endpoint is publicly accessible with no rate limiting or request concurrency control. Each request with an EVM address or alias value triggers a live database query through `EntityService.getEncodedId()`. During a database network partition, an attacker can flood this endpoint with distinct EVM address values, causing the connection pool to queue an unbounded number of pending DB queries that all time out simultaneously; when the partition heals, all queued connections attempt to reconnect at once, producing a thundering-herd reconnection storm that can overwhelm the database server.

### Finding Description

**Root cause and failed assumption:**

The design assumes that the database is always available and that request volume is bounded by upstream infrastructure (e.g., a reverse proxy with rate limiting). Neither assumption is enforced in the application layer. There is no:
- Per-IP or global rate limit on this endpoint
- In-process cache (memoization) for entity lookups
- Concurrency cap on simultaneous DB queries issued by this handler
- Circuit-breaker pattern to shed load when the DB is unreachable [5](#0-4) 

**Exploit flow:**

1. Attacker observes (or induces) a DB network partition.
2. Attacker sends a high-rate burst of HTTP GET requests to `/api/v1/accounts/<evmAddr>/allowances/tokens` using distinct, syntactically valid EVM addresses (e.g., `0xaabbccdd...` with varying bytes).
3. Each request passes input validation (valid 20-byte hex), enters `getEntityIdFromEvmAddress()`, and issues a `getRows()` call that blocks waiting for a DB connection.
4. The connection pool queues all pending queries. With no concurrency cap, the pool fills to its maximum size; additional requests queue at the application level.
5. All queued queries time out simultaneously when the partition duration exceeds the DB query timeout.
6. When the partition heals, the connection pool attempts to re-establish all connections at once — the classic thundering-herd reconnection storm — sending a spike of TCP SYN + authentication handshakes to the DB server that can exceed its `max_connections` limit and cause further cascading failures. [6](#0-5) 

### Impact Explanation

- **Availability**: The DB connection pool is exhausted for the duration of the partition. All other API endpoints sharing the same pool (the entire REST service) become unresponsive, not just the token allowance endpoint.
- **Post-partition amplification**: The reconnection storm can cause the DB server to reject connections or crash under load, extending the outage beyond the partition window itself.
- **No authentication required**: Any anonymous internet user can trigger this. The endpoint is part of the public mirror node REST API.
- **Severity**: High — full service DoS achievable by a single attacker with a modest HTTP flood tool. [7](#0-6) 

### Likelihood Explanation

- **Attacker capability**: Zero privileges required. Valid EVM addresses are trivially generated (any 40-character hex string). No API key, wallet, or on-chain state needed.
- **Feasibility**: A single machine with a standard HTTP benchmarking tool (e.g., `wrk`, `hey`, `ab`) can saturate a typical Node.js connection pool in seconds.
- **Repeatability**: The attack can be repeated every time a partition occurs (or can be combined with a separate network-layer attack to induce the partition).
- **Detection difficulty**: The requests are individually indistinguishable from legitimate traffic — each is a well-formed HTTP GET with a valid EVM address. [8](#0-7) 

### Recommendation

1. **Rate limiting at the application layer**: Apply a per-IP and global rate limit on all `/accounts/:id/allowances/*` routes before the async DB path is entered.
2. **In-process LRU cache for entity lookups**: Cache the result of `getEncodedId()` (keyed on the normalized input string) with a short TTL (e.g., 30 s). This eliminates redundant DB queries for the same address and prevents the pool from being flooded with identical lookups.
3. **Connection pool concurrency cap with fast-fail**: Configure the DB pool with a `max` size and a short `connectionTimeoutMillis`. Return HTTP 503 immediately when the pool queue depth exceeds a threshold rather than queuing indefinitely.
4. **Circuit breaker**: Wrap `getRows()` in a circuit breaker (e.g., `opossum`) that opens after a configurable number of consecutive DB timeouts, returning 503 to clients and preventing further connection attempts until the DB recovers — eliminating the thundering-herd reconnect on heal.
5. **Staggered reconnection**: Configure the pool client (e.g., `pg-pool`) with exponential backoff and jitter on reconnection attempts to spread the reconnect load after a partition heals. [3](#0-2)

    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }

  async getEncodedId(entityIdString, requireResult = true, paramName = filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS) {
    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }
    } catch (ex) {
      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
