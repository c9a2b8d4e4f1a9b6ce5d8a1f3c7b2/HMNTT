### Title
`BlockType.of(Long.MAX_VALUE)` Bypasses Identity Check in `findByBlockType()`, Causing `BlockNumberNotFoundException` for Valid "Latest" Block Requests

### Summary
`BlockType.LATEST` uses `Long.MAX_VALUE` as its internal sentinel number, but `BlockType.of("9223372036854775807")` (the decimal string of `Long.MAX_VALUE`) creates a **new** `BlockType` instance with the same number value rather than returning the `LATEST` constant. `RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect `LATEST`, so this new instance bypasses the check, falls through to `recordFileRepository.findByIndex(Long.MAX_VALUE)`, finds no matching block, and throws `BlockNumberNotFoundException` — causing unintended smart contract execution failure for any unprivileged caller.

### Finding Description

**Root cause — `BlockType.of()` decimal branch creates a new instance:**

In `BlockType.java` lines 59–65, when the input matches the decimal group `(\d{1,20})`, the code unconditionally calls `new BlockType(value, Long.parseLong(decimal, 10))`:

```java
final var decimal = matcher.group(GROUP_DECIMAL);
if (decimal != null) {
    try {
        return new BlockType(value, Long.parseLong(decimal, 10));  // always new instance
    } catch (NumberFormatException e) { ... }
}
```

`"9223372036854775807"` is 19 digits, matches the pattern, and `Long.parseLong` succeeds with `Long.MAX_VALUE`. The result is a **new** `BlockType` object — not `BlockType.LATEST`. [1](#0-0) 

The `LATEST` constant is defined as:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [2](#0-1) 

**Root cause — `findByBlockType()` uses `==` reference equality:**

`RecordFileServiceImpl.findByBlockType()` checks for `LATEST` using `==`:

```java
} else if (block == BlockType.LATEST) {
    return recordFileRepository.findLatest();
```

The new instance from `BlockType.of("9223372036854775807")` is a different object reference, so this branch is never taken. [3](#0-2) 

**Exploit flow:**

1. `block.isHash()` returns false (`number == -1L` is false for `Long.MAX_VALUE`). [4](#0-3) 

2. Execution falls through to `recordFileRepository.findByIndex(Long.MAX_VALUE)`, which queries `select r from RecordFile r where r.index = ?1` — no real block has index `9223372036854775807`, so `Optional.empty()` is returned. [5](#0-4) 

3. `ContractCallService.callContract()` wraps the supplier with `.orElseThrow(BlockNumberNotFoundException::new)`:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
``` [6](#0-5) 

**Why existing checks are insufficient:**

The `BlockTypeTest` confirms `BlockType.of(String.valueOf(Long.MAX_VALUE))` is accepted as valid input (line 34: `MAX_VALUE + "," + MAX_VALUE`) but only asserts the `number` field value — it does **not** assert `isSameAs(BlockType.LATEST)`. The tag-alias path (`"latest"`, `"pending"`, `"safe"`, `"finalized"`) correctly returns the `LATEST` constant, but the decimal numeric path has no such normalization. [7](#0-6) 

The same bypass applies to the hex input `"0x7fffffffffffffff"` (also `Long.MAX_VALUE`), which also creates a new instance via the hex-number branch. [8](#0-7) 

### Impact Explanation
Any `eth_call` or `eth_estimateGas` request specifying block `"9223372036854775807"` or `"0x7fffffffffffffff"` — values that semantically represent the latest block (since `Long.MAX_VALUE` is the internal sentinel) — will fail with `BlockNumberNotFoundException` ("Unknown block number") instead of executing against the latest state. This constitutes unintended smart contract execution failure. No funds are directly at risk, but it breaks the expected behavior of the JSON-RPC interface for these specific block parameter values and can be used to selectively deny service to any caller using these inputs.

### Likelihood Explanation
The attack requires zero privileges — any external user can submit a standard `POST /api/v1/contracts/call` JSON body with `"block": "9223372036854775807"`. It is trivially repeatable, requires no special knowledge beyond the Ethereum JSON-RPC spec, and is deterministic (always fails). The value `Long.MAX_VALUE` is a well-known Java constant that any developer or attacker familiar with the codebase could identify as the sentinel.

### Recommendation
Fix the identity check in `findByBlockType()` to use value-based comparison instead of reference equality:

```java
} else if (block == BlockType.LATEST || block.number() == Long.MAX_VALUE) {
    return recordFileRepository.findLatest();
```

Or, preferably, normalize in `BlockType.of()` — after parsing the decimal or hex number, if the result equals `Long.MAX_VALUE`, return `BlockType.LATEST` instead of a new instance:

```java
long parsed = Long.parseLong(decimal, 10);
return parsed == Long.MAX_VALUE ? LATEST : new BlockType(value, parsed);
```

Apply the same normalization to the hex-number branch. This ensures all representations of `Long.MAX_VALUE` resolve to the `LATEST` singleton and the `==` check works correctly.

### Proof of Concept

**Precondition:** A running mirror node web3 service with at least one record file in the database.

**Trigger:**
```bash
curl -X POST http://localhost:8545/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "0x0000000000000000000000000000000000000001",
    "data": "0x",
    "gas": 21000,
    "block": "9223372036854775807"
  }'
```

**Expected (correct) result:** Executes against the latest block, returns `{"result":"0x"}`.

**Actual result:** HTTP 400 with `{"_status":{"messages":[{"message":"Unknown block number"}]}}` — `BlockNumberNotFoundException` thrown.

**Alternate trigger (hex):**
```bash
"block": "0x7fffffffffffffff"
```
Same failure — the hex branch also creates a new instance with `Long.MAX_VALUE`.
