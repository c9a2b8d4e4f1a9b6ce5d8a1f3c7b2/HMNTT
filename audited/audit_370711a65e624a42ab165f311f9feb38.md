### Title
Unbounded Hash String Input Causes Resource Exhaustion in `findBlockByHash()`

### Summary
The `findBlockByHash()` function in `rosetta/app/persistence/block.go` accepts a hash string of arbitrary length with no upper-bound validation. Combined with the absence of an HTTP request body size limit (`http.MaxBytesReader`) anywhere in the middleware chain, an unauthenticated attacker can send megabyte-scale hash strings that are fully buffered in Go process memory and transmitted as a PostgreSQL query parameter, enabling memory exhaustion and database connection pool saturation.

### Finding Description
**Exact code path:**

`POST /block` → Rosetta SDK router → `blockAPIService.Block()` (`block_service.go:51`) → `BaseService.RetrieveBlock()` (`base_service.go:84-86`) → `blockRepo.FindByHash()` (`block.go:135-144`) → `findBlockByHash()` (`block.go:226-241`).

**Root cause:**

`FindByHash()` performs only an empty-string check:
```go
if hash == "" {
    return nil, hErrors.ErrInvalidArgument
}
``` [1](#0-0) 

No maximum length check exists. The hash is then passed directly to the parameterized query:
```go
db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb)
``` [2](#0-1) 

The HTTP server is constructed with no `http.MaxBytesReader` wrapping the request body anywhere in the middleware stack (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware`): [3](#0-2) 

The only time-based protection is `ReadTimeout` (default 5,000,000,000 ns = 5 s): [4](#0-3) 

At 100 Mbps, a 5-second window allows ~62 MB per request. The Rosetta SDK asserter (`rosettaAsserter.NewServer`) validates that the hash is non-empty but imposes no length ceiling.

**Why existing checks fail:**

- The empty-string guard (`hash == ""`) only rejects zero-length input.
- `ReadTimeout` bounds time, not bytes — a fast connection can deliver tens of MB within the window.
- The optional Traefik `inFlightReq` (limit 5) and `rateLimit` (10 req/s) middlewares are Helm-chart-level and absent in bare deployments. [5](#0-4) 

### Impact Explanation
Each request with a large hash string forces the Go runtime to allocate and hold the full string in memory for the duration of the request (JSON decode → service layer → DB driver). Concurrent requests (up to the DB pool maximum of 100 open connections) multiply this allocation. A sustained stream of requests each carrying a ~50 MB hash can exhaust available heap, trigger GC pressure, and cause the Rosetta node process to become unresponsive or OOM-killed — effectively taking the node offline without any authenticated access or brute-force credential guessing. [6](#0-5) 

### Likelihood Explanation
The `/block` endpoint is a standard, publicly documented Rosetta API endpoint requiring no authentication. Any unprivileged network-reachable client can POST a `BlockRequest` with an arbitrarily long `block_identifier.hash`. The attack is trivially scriptable with `curl` or any HTTP client, requires no special knowledge of the system, and is repeatable at the rate permitted by `ReadTimeout`. In deployments without the optional Traefik rate-limiting middleware, there is no application-level throttle.

### Recommendation
1. **Add `http.MaxBytesReader`** in the HTTP handler chain to cap request body size (e.g., 1 MB) before any JSON decoding occurs, in `rosetta/main.go` where the `http.Server` handler is assembled.
2. **Add hash length validation** in `FindByHash()` and `FindByIdentifier()` — a Rosetta block hash is a fixed-length hex string (96 hex chars for SHA-384); reject anything exceeding a small constant (e.g., 256 bytes).
3. **Make rate-limiting and in-flight request limits mandatory** at the application layer rather than relying solely on optional infrastructure middleware.

### Proof of Concept
```bash
# Generate a 10 MB hash string and send it to the /block endpoint
LARGE_HASH=$(python3 -c "print('a' * 10_000_000)")

curl -s -X POST http://<rosetta-host>:5700/block \
  -H "Content-Type: application/json" \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
    \"block_identifier\": {\"hash\": \"${LARGE_HASH}\"}
  }"
```
Repeat concurrently (e.g., 20 parallel processes) to amplify memory pressure. Monitor Go process RSS; it will grow proportionally with each in-flight request. Under sufficient concurrency the process will be OOM-killed or become unresponsive, taking the Rosetta node offline.
