### Title
Unauthenticated Parallel DB Query Amplification via `Promise.all` in `getOneAccount()` Enables Connection Pool Exhaustion DoS

### Summary
The `getOneAccount()` handler in `rest/accounts.js` unconditionally fires two simultaneous database queries per request via `Promise.all([entityPromise, transactionsPromise])` with no rate limiting on the REST Node.js service for this endpoint. Any unauthenticated external caller can flood `GET /api/v1/accounts/:id` to exhaust the finite DB connection pool at twice the rate of a single-query endpoint, degrading or denying service to all callers including those retrieving gossip transaction data.

### Finding Description
**Exact location:** `rest/accounts.js`, function `getOneAccount()`, lines 487–495. [1](#0-0) 

The handler builds and fires `entityPromise` (a `pool.queryQuietly` call for account/entity data) and `transactionsPromise` (a call to `transactions.doGetTransactions`, which itself issues at least two sequential DB queries — `getTransactionTimestamps` then `getTransactionsDetails`) simultaneously via `Promise.all`. The `includeTransactions` flag defaults to `true`: [2](#0-1) 

so both branches execute unless the caller explicitly passes `?transactions=false`.

**Root cause / failed assumption:** The code assumes that concurrent callers will be bounded by some upstream control. In reality, `server.js` applies no per-IP or global rate limiting to this route: [3](#0-2) 

The `authHandler` middleware only sets a custom response-size *limit* for authenticated users; it does not reject or throttle unauthenticated requests: [4](#0-3) 

The only throttle infrastructure found (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` Java service and is entirely absent from the REST Node.js service.

The DB pool is finite, configured via `config.db.pool.maxConnections` and `statement_timeout`: [5](#0-4) 

With `Promise.all`, each in-flight request holds **two** pool connections simultaneously. Under N concurrent attacker requests, 2N connections are consumed, halving the effective pool capacity compared to a sequential design.

### Impact Explanation
When the DB connection pool is saturated, all subsequent requests — including legitimate ones fetching gossip transaction data — queue or fail with connection-timeout errors. The `statement_timeout` only kills long-running individual queries; it does not prevent pool exhaustion caused by a high volume of short, concurrent queries. The `maxConnections` cap (e.g., `mirror_rest max_user_connections: 250` in the Helm chart) is a hard ceiling that an attacker can reach at half the request rate compared to a single-query endpoint. [6](#0-5) 

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. The endpoint is publicly reachable at `GET /api/v1/accounts/:id`. A single attacker with a modest botnet or even a single high-throughput HTTP client (e.g., `wrk`, `hey`) can sustain hundreds of concurrent requests. The attack is trivially repeatable and requires no special knowledge beyond the public API documentation.

### Recommendation
1. **Add rate limiting** to the REST Node.js service (e.g., `express-rate-limit` or an ingress/nginx rate-limit rule) scoped per IP for the `/api/v1/accounts/:id` endpoint.
2. **Short-circuit early:** validate the account exists (cheap indexed lookup) before launching the transactions query, converting the parallel pattern to a conditional sequential one.
3. **Default `transactions=false`** or require explicit opt-in, reducing the default query count per request.
4. **Set a concurrency cap** at the application layer (e.g., a semaphore on total in-flight DB queries) independent of the pool size.

### Proof of Concept
```bash
# Flood the endpoint with 200 concurrent unauthenticated requests (no credentials needed)
wrk -t 10 -c 200 -d 30s \
  "https://<mirror-node-host>/api/v1/accounts/0.0.98"

# Each request triggers Promise.all([entityPromise, transactionsPromise])
# consuming 2 DB connections simultaneously.
# With maxConnections=250, only ~125 concurrent requests can be served
# before the pool is exhausted and all subsequent requests time out.
# Observe: legitimate callers begin receiving 503/connection-timeout errors.
```
