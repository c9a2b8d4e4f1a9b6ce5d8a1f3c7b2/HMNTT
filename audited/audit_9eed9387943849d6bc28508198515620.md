### Title
Missing Entity-Type Enforcement in `getAccountTokenAllowances` Allows Contract Token Allowance Disclosure

### Summary
The `/api/v1/accounts/{id}/allowances/tokens` endpoint in `rest/controllers/tokenAllowanceController.js` resolves the path parameter via `EntityService.getEncodedId()`, which performs no entity-type check. Any unprivileged caller who supplies a numeric entity ID belonging to a contract (rather than an account) receives a `200 OK` response containing that contract's full token allowance records. The test fixture `contract-no-params.json` explicitly confirms this behavior is present and unguarded.

### Finding Description

**Exact code path:**

`getAccountTokenAllowances` (line 69, `rest/controllers/tokenAllowanceController.js`):
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
``` [1](#0-0) 

`EntityService.getEncodedId` (lines 118–137, `rest/service/entityService.js`) resolves a numeric string directly to an encoded integer with no entity-type lookup:
```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()          // ← returned immediately, no type check
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
``` [2](#0-1) 

The encoded ID is passed directly to `TokenAllowanceService.getAccountTokenAllowances(query)` which queries the `token_allowance` table by owner ID alone — no `entity.type = 'ACCOUNT'` filter is applied anywhere in the chain. [3](#0-2) 

**Root cause:** `getEncodedId` is a generic entity-ID resolver. It was designed to resolve any entity (account, contract, token, etc.) to its encoded integer. The caller (`getAccountTokenAllowances`) never validates that the resolved entity is of type `ACCOUNT`.

**Why existing checks fail:**

`EntityService.isValidAccount` exists but is **not called** in the token-allowance flow: [4](#0-3) 

It is only invoked in `accountController.js` and `tokenController.js`. Even if it were called, the underlying query (`entityExistenceQuery`) only checks existence, not entity type: [5](#0-4) 

**Proof from test fixture:** The spec `rest/__tests__/specs/accounts/{id}/allowances/tokens/contract-no-params.json` sets up a contract at `num: 1001`, seeds token allowances owned by that contract, and asserts `responseStatus: 200` with all allowances returned — confirming the unguarded path is exercised and expected to succeed: [6](#0-5) 

### Impact Explanation
An attacker can enumerate token allowances granted by any smart contract on the network by iterating numeric entity IDs. Token allowances reveal which spender addresses are authorized to transfer tokens on behalf of the contract owner, the approved amounts, and the token IDs involved. This is information that the `/accounts/` endpoint is not supposed to expose for contract entities. In DeFi contexts, this data can be used to front-run or craft targeted attacks against contracts with large outstanding allowances.

### Likelihood Explanation
No authentication is required. The attacker only needs to know (or brute-force) a contract's numeric entity ID, which is publicly discoverable via the `/api/v1/contracts` endpoint. The exploit is a single unauthenticated HTTP GET request, is fully repeatable, and requires no special tooling.

### Recommendation
In `getAccountTokenAllowances`, after resolving `accountId`, add an entity-type guard before proceeding. The simplest fix is to extend `EntityService.getEncodedId` with an optional `expectedType` parameter, or add a dedicated `getAccountEncodedId` method that queries `entity.type` and throws a `NotFoundError` (404) when the resolved entity is not of type `ACCOUNT`. The `entityExistenceQuery` already selects `entity.type`; the result just needs to be checked:

```js
// In entityService.js
static entityExistenceQuery = `select ${Entity.TYPE} from ${Entity.tableName} where ${Entity.ID} = $1`;

async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity) && entity.type === 'ACCOUNT';  // enforce type
}
```

Then call `isValidAccount` (or an equivalent type-checked lookup) inside `getAccountTokenAllowances` and throw `NotFoundError` if the entity is not an account.

### Proof of Concept
**Precondition:** A contract exists at entity ID `0.0.1001` with token allowances in the mirror node database (as set up in the existing test fixture).

**Steps:**
1. Send an unauthenticated HTTP GET request:
   ```
   GET /api/v1/accounts/1001/allowances/tokens
   ```
2. Observe HTTP `200 OK` response containing token allowances owned by contract `0.0.1001`:
   ```json
   {
     "allowances": [
       { "owner": "0.0.1001", "spender": "0.0.2001", "token_id": "0.0.10001", "amount": 23, ... },
       ...
     ]
   }
   ```
3. The same result is obtained via the contract's EVM address:
   ```
   GET /api/v1/accounts/0xebb9a1be370150759408cd7af48e9eda2b8ead57/allowances/tokens
   ```

No credentials, no special privileges, no rate-limit bypass required.

      throw ex;
