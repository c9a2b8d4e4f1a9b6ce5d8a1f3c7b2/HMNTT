### Title
Unauthenticated Unbounded gRPC Stream Subscription Enables Resource Exhaustion via Far-Future `endTime`

### Summary
Any unauthenticated external client can call `subscribeTopic` via gRPC with a `TopicMessageFilter` containing `endTime = Long.MAX_VALUE` (or any far-future nanosecond timestamp). Because there is no authentication, no global subscriber cap, no maximum `endTime` bound, and no rate limiting on the gRPC endpoint, an attacker can open an unbounded number of persistent streams that never terminate, exhausting server memory, scheduler threads, and database polling capacity.

### Finding Description

**Entry point** — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, `subscribeTopic()` (lines 43–53). No authentication interceptor exists in the production gRPC module; the only `ServerInterceptor` present is a test-only class (`grpc/src/test/java/…/interceptor/GrpcInterceptor.java`) that only sets endpoint context for table-usage tracking.

**Timestamp conversion** — `ConsensusController.convertTimestamp()` (lines 76–81) explicitly maps any protobuf timestamp with `seconds >= 9223372035` to `Long.MAX_VALUE`, handing a trivially reachable far-future value directly into the filter.

**No upper-bound validation on `endTime`** — `TopicMessageFilter.isValidEndTime()` (lines 43–46 of `TopicMessageFilter.java`) only asserts `endTime == null || endTime > startTime`. There is no cap. `Long.MAX_VALUE` passes this check.

**Stream never terminates** — In `TopicMessageServiceImpl.subscribeTopic()` (lines 72–73), the live flux is terminated by `takeUntilOther(pastEndTime(topicContext))`. `pastEndTime()` (lines 123–131) polls every `endTimeInterval` (default 30 s) checking `topicContext.isComplete()`. `isComplete()` (lines 203–215) evaluates `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(endTimeInterval).isBefore(Instant.now())` — this is `false` for centuries, so the stream never signals completion.

**No global subscriber limit** — `subscriberCount` (line 48) is a Micrometer gauge metric only; it is never checked against a maximum. `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14 of `NettyProperties.java`) limits calls *per TCP connection* but places no bound on the number of connections.

**No rate limiting on gRPC** — The `ThrottleConfiguration`/`ThrottleManagerImpl` rate-limiting infrastructure exists exclusively in the `web3` module for EVM calls; it is entirely absent from the `grpc` module.

**Per-subscriber DB polling** — Each subscriber triggers a safety-check poll after 1 second (`TopicMessageServiceImpl` lines 67–70) and the `SharedPollingTopicListener` polls the database on a shared schedule. Thousands of concurrent subscribers amplify DB load.

### Impact Explanation
An attacker with no credentials can open `N` TCP connections × 5 streams each = `5N` persistent gRPC streams, each holding a Reactor subscription, a `TopicContext` object, a scheduled safety-check task on `Schedulers.boundedElastic()`, and periodic `pastEndTime` polling every 30 seconds. At scale this exhausts JVM heap, bounded-elastic thread pool slots, and database connection pool entries, causing full service unavailability (DoS) for all legitimate subscribers. Because HCS topic subscriptions are the primary use case of the mirror node gRPC API, this directly degrades the availability of the consensus message streaming service.

### Likelihood Explanation
The exploit requires zero privileges, zero credentials, and only a standard gRPC client (e.g., `grpcurl`). The protobuf API is publicly documented (`docs/grpc/README.md`, line 16 shows a plain `grpcurl` invocation with no auth). The far-future timestamp path is explicitly handled in `convertTimestamp()`. An attacker can script thousands of connections in minutes. The attack is repeatable and requires no special knowledge beyond the public API schema.

### Recommendation
1. **Cap `endTime`**: Add a `@Max` or custom `@AssertTrue` constraint in `TopicMessageFilter` rejecting `endTime` values beyond a configurable maximum (e.g., `now + 24h`).
2. **Enforce a global subscriber limit**: Check `subscriberCount` against a configurable maximum in `subscribeTopic()` and reject new subscriptions when the limit is reached.
3. **Add per-IP connection rate limiting**: Introduce a gRPC `ServerInterceptor` that tracks and limits concurrent streams per remote IP.
4. **Add authentication/authorization**: Require API keys or mTLS for long-lived streaming subscriptions.
5. **Apply the existing throttle pattern**: Port the `bucket4j`-based rate limiter from the `web3` module to the `grpc` module.

### Proof of Concept
```bash
# Open 1000 persistent streams with endTime = Long.MAX_VALUE (seconds near max)
for i in $(seq 1 1000); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1},"consensusEndTime":{"seconds":9223372036}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Each stream holds open indefinitely; server memory/thread pool exhausts
# Monitor: hiero.mirror.grpc.subscribers gauge climbs without bound
# Result: legitimate subscribers receive errors or timeouts
```

Preconditions: gRPC port reachable (default 5600), any valid `topicNum` that exists in the mirror node database (or `checkTopicExists=false`).
Trigger: `consensusEndTime.seconds >= 9223372035` maps to `Long.MAX_VALUE` via `convertTimestamp()`; `isValidEndTime()` passes; stream never completes.
Result: Server resource exhaustion, denial of service to all subscribers.
