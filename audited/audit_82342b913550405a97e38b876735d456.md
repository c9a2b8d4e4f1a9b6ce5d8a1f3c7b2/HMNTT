### Title
`parseInt` Precision Loss on Oversized `Content-Length` Corrupts `requestSizeHistogram` in `metricsMiddleware`

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function reads the raw `content-length` header string and passes it directly to `parseInt`. When the header value exceeds `Number.MAX_SAFE_INTEGER` (e.g., `9999999999999999999999`), `parseInt` returns an imprecise IEEE-754 floating-point approximation rather than `NaN`, so the `|| 0` guard never fires and the corrupted value is recorded into `requestSizeHistogram`. Any unauthenticated external caller can trigger this repeatedly.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, line 209:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** JavaScript's `parseInt` converts its argument to a 64-bit IEEE-754 double. For strings whose numeric value exceeds `Number.MAX_SAFE_INTEGER` (9007199254740991), the conversion is lossy — the result is a rounded, imprecise float (e.g., `parseInt('9999999999999999999999', 10)` → `10000000000000000000000`). Because the result is a non-zero, non-`NaN` number, the `|| 0` fallback is never taken, and the inflated, imprecise value is passed directly to `requestSizeHistogram.record()`.

**Exploit flow:**
1. Attacker sends any HTTP request (e.g., `GET /api/v1/accounts`) with the header `Content-Length: 9999999999999999999999`.
2. Node.js's HTTP parser stores the raw string in `req.headers['content-length']`; the request reaches Express middleware normally.
3. On `res.on('finish')`, line 209 executes: `parseInt('9999999999999999999999', 10)` returns `10000000000000000000000`.
4. `10000000000000000000000 || 0` is truthy, so `10000000000000000000000` is recorded into `requestSizeHistogram`.
5. Repeated requests continuously inject arbitrarily large, imprecise values into the histogram, skewing all bucket counts, totals, and averages.

**Why existing checks fail:** The `|| 0` guard only catches `NaN` (complete parse failure) and `0`. It does not catch values that parse successfully but are imprecise due to floating-point overflow. There is no upper-bound validation, no `Number.isSafeInteger()` check, and no clamping before the `record()` call.

### Impact Explanation
The `api_request_size_bytes` Prometheus histogram is permanently corrupted for the lifetime of the process. Operators relying on this metric for capacity planning, anomaly detection, or SLO tracking receive false data. Because the OpenTelemetry SDK accumulates histogram state in memory, a sustained flood of such requests can also cause the running sum to overflow to `Infinity`, rendering the entire histogram useless. Severity is low-to-medium: no user data is exposed and no funds are at risk, but observability infrastructure is degraded, which can mask real incidents.

### Likelihood Explanation
The precondition is zero: no authentication, no special role, no API key. Any HTTP client can set an arbitrary `Content-Length` header. The attack is trivially scriptable (`curl -H 'Content-Length: 9999999999999999999999' http://mirror-node/api/v1/accounts`) and fully repeatable. Rate limiting (if present) would slow but not prevent the corruption, since even a single request per second is sufficient to skew long-term histogram aggregates.

### Recommendation
Replace the bare `parseInt` call with a safe integer parse that clamps or rejects out-of-range values:

```js
const rawLen = req.headers['content-length'];
const parsed = rawLen !== undefined ? Number(rawLen) : 0;
const requestSize = Number.isSafeInteger(parsed) && parsed >= 0 ? parsed : 0;
requestSizeHistogram.record(requestSize, labels);
```

Alternatively, use `BigInt` parsing with an explicit cap, or simply read the actual bytes received from the socket rather than trusting the client-supplied header.

### Proof of Concept
```bash
# Single request — injects imprecise value 10000000000000000000000 into histogram
curl -s -o /dev/null \
  -H 'Content-Length: 9999999999999999999999' \
  http://<mirror-node-host>/api/v1/accounts

# Verify corruption by scraping the metrics endpoint
curl -s http://<mirror-node-host>/api/metrics/ | grep api_request_size_bytes
# Expected: bucket counts and _sum will reflect the injected ~1e22 value
```

In a Node.js REPL, the precision loss can be confirmed directly:
```js
> parseInt('9999999999999999999999', 10)
10000000000000000000000          // not NaN → || 0 guard skipped
> Number.isSafeInteger(10000000000000000000000)
false
```
