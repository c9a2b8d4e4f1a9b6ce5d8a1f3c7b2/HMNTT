### Title
Unbounded Historical Timestamp Scan via Missing Lower Bound Enforcement in `getHookStorageChange`

### Summary
When a user supplies only an upper-bound timestamp filter (e.g., `timestamp=lte:X`) with no lower bound, `Bound.getAdjustedLowerRangeValue()` returns `0` (genesis), causing the repository query to scan every `hook_storage_change` row from timestamp `0` to `X` for the given `ownerId`/`hookId`. No authentication or lower-bound requirement is enforced, so any unauthenticated caller can repeatedly trigger arbitrarily wide historical scans that exhaust database I/O and CPU.

### Finding Description

**Code path:**

1. **Controller** — `HooksController.java` line 110: `timestamps` is `@Size(max = 2)` but no lower bound is required. A single `timestamp=lte:X` is accepted.

2. **`Bound.of()`** — `Bound.java` lines 169–182: builds a `Bound` with `lower = null`, `upper = lte:X`. The `primarySortField` argument is `false` (line 181), so the range-validity check at line 58 (`if (primarySortField && adjustedLower > adjustedUpper)`) is **skipped entirely**.

3. **`isHistorical()`** — `HookStorageRequest.java` lines 66–68: returns `!timestamp.isEmpty()`. Because `upper != null`, `isEmpty()` is `false`, so the request is routed to `getHookStorageChange`.

4. **`getAdjustedLowerRangeValue()`** — `Bound.java` lines 86–97:
   ```java
   if (this.lower == null) {
       return 0;          // ← genesis timestamp
   }
   ```
   With no lower bound, this unconditionally returns `0`.

5. **Repository query** — `HookStorageChangeRepository.java` line 30:
   ```sql
   and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
   ```
   Becomes `between 0 and X`, scanning every row ever written for that hook.

6. The `DISTINCT ON (key)` clause (line 16) forces the database to sort and de-duplicate **all** matching rows before the `LIMIT` from `Pageable` is applied, making the scan cost proportional to the full history depth, not the page size.

**Why existing checks are insufficient:**
- `@Size(max = 2)` on `timestamps` only caps the number of filter parameters, not their content.
- `@Max(MAX_LIMIT)` on `limit` caps rows *returned*, not rows *scanned*.
- No rate limiting, no authentication, and no minimum-lower-bound validation exist in this path.

### Impact Explanation
An attacker can force the database to perform a full sequential scan of `hook_storage_change` from genesis for any publicly known `ownerId`/`hookId` pair. Because `ownerId` and `hookId` are public blockchain identifiers, no secret knowledge is required. Repeated requests saturate DB I/O and CPU, degrading or denying service to all users of the mirror node REST API, including the gossip-adjacent transaction-status queries the service is designed to serve.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. The only precondition is knowing a valid `ownerId` and `hookId`, both of which are observable on-chain. A single attacker with a script issuing requests at a moderate rate (e.g., one per second) can sustain continuous full-table scans. The attack is trivially repeatable and requires no special tooling.

### Recommendation
Enforce a mandatory lower bound on the timestamp range for historical queries. Concretely:

- In `HooksController.hookStorageChangeRequest` (or in `Bound.of`), reject requests where `timestamp` is provided but contains only an upper bound — require at least one `gte`/`gt` parameter alongside any `lte`/`lt`.
- Alternatively, cap the maximum allowed timestamp window (e.g., reject ranges wider than a configurable maximum, such as 24 hours in nanoseconds).
- Add a validation step in `getHookStorageChange` before issuing the query:
  ```java
  if (timestampLowerBound == 0 && timestampUpperBound == Long.MAX_VALUE) {
      throw new IllegalArgumentException("A lower timestamp bound is required for historical queries");
  }
  ```
- Consider adding rate limiting at the API gateway level for historical scan endpoints.
