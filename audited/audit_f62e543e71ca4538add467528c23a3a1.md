### Title
Unauthenticated EVM Address Lookup Causes Unbounded DB Connection Pool Exhaustion via `getNftsByAccountId()`

### Summary
The `getNftsByAccountId()` handler in `rest/controllers/accountController.js` accepts any syntactically valid 40-hex-char EVM address from unauthenticated users and unconditionally issues a live database query via `EntityService.getEntityIdFromEvmAddress()`. There is no rate limiting, no caching of 404 (not-found) responses, and the finite DB connection pool is shared across all REST endpoints. An attacker flooding the endpoint with many distinct non-existent EVM addresses can saturate the pool, causing connection-timeout failures across all API endpoints on the affected node.

### Finding Description

**Exact code path:**

`GET /api/v1/accounts/<40-hex-evm-address>/nfts` → `AccountController.getNftsByAccountId()` → `EntityService.getEncodedId()` → `EntityService.getEntityIdFromEvmAddress()` → pool query → `NotFoundError` thrown.

**Step 1 – No pre-validation or rate limiting at the route level.** [1](#0-0) 

`req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]` is passed directly to `EntityService.getEncodedId()`. No IP-based throttle, no authentication requirement, and no middleware in the stack (`authHandler`, `requestHandler`, `requestNormalizer`, `responseCacheHandler`, `responseHandler`) implements rate limiting.

**Step 2 – `getEncodedId()` routes EVM addresses to a live DB query.** [2](#0-1) 

When `EntityId.isValidEntityId()` returns `true` for a 40-hex string and `entityId.evmAddress !== null`, execution falls into `getEntityIdFromEvmAddress()`.

**Step 3 – `getEntityIdFromEvmAddress()` always hits the database.** [3](#0-2) 

The query `select id from entity where deleted <> true and evm_address = $1` is executed unconditionally. For a non-existent address, zero rows are returned and `NotFoundError` is thrown. There is no in-process or Redis cache for this negative result.

**Step 4 – The response cache does NOT cache 404 errors.** [4](#0-3) 

`responseCacheUpdateHandler` only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 `NotFoundError` is never cached, so every distinct non-existent EVM address always triggers a fresh DB round-trip.

**Step 5 – The DB connection pool is finite and shared.** [5](#0-4) 

`max: config.db.pool.maxConnections` caps the pool. `connectionTimeoutMillis` causes waiting requests to fail with an error once the timeout is exceeded, propagating failures to all callers sharing the pool.

**Why existing checks are insufficient:**

- `statement_timeout` limits individual query duration but does not prevent many short queries from simultaneously holding all pool slots.
- `connectionTimeoutMillis` causes requests to fail after the timeout, but by then the pool is already exhausted and all concurrent API requests (not just this endpoint) are affected.
- The partial index `entity__evm_address on entity (evm_address) where evm_address is not null` makes each individual query fast, but fast queries still consume a pool connection for their duration, and at sufficient concurrency the pool saturates. [6](#0-5) 

### Impact Explanation

The shared `global.pool` is used by every REST service. Exhausting it via this endpoint degrades all API endpoints simultaneously. Operators of mirror nodes expose this API publicly; a sustained flood of requests with distinct non-existent EVM addresses (trivially generated) will cause connection-timeout errors across the node, matching the described "30%+ processing degradation" threshold for a node that serves multiple API consumers.

### Likelihood Explanation

No privileges are required. The endpoint is publicly accessible. Generating 40-hex strings is trivial. A single attacker with modest bandwidth can sustain thousands of requests per second. Because 404 responses are never cached, there is no natural dampening effect. The attack is repeatable and requires no special knowledge of the network state.

### Recommendation

1. **Add IP-based rate limiting** (e.g., `express-rate-limit`) on all `/accounts/:id*` routes, applied before the controller handler.
2. **Cache negative (404) EVM address lookups** in Redis with a short TTL (e.g., 5–30 seconds) so repeated lookups for the same non-existent address do not hit the DB.
3. **Add a `statement_timeout` guard at the application layer** (already present at pool level) and consider a per-request DB query budget.
4. **Validate EVM address format at the OpenAPI/middleware layer** before any DB interaction, rejecting malformed inputs early.

### Proof of Concept

```bash
# Generate 10,000 distinct non-existent EVM addresses and flood the endpoint
for i in $(seq 1 10000); do
  addr=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/${addr}/nfts" &
done
wait
```

**Preconditions:** Public network access to the mirror node REST API. No credentials needed.

**Trigger:** Each request with a unique non-existent 40-hex EVM address causes `EntityService.getEntityIdFromEvmAddress()` to execute `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1`, hold a pool connection, find no rows, and throw `NotFoundError`. The 404 response is never cached.

**Result:** At sufficient concurrency, all `maxConnections` pool slots are occupied. Subsequent requests across all API endpoints receive connection-timeout errors until the flood subsides.
