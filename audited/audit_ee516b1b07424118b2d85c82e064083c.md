### Title
Unbounded Concurrent Subscription DoS via Historical Poll Heap Exhaustion

### Summary
Any unauthenticated external client can open an unlimited number of concurrent `subscribeTopic` gRPC streams with `startTime=0` (epoch) and no `endTime` or `limit`. Each stream drives `PollingTopicMessageRetriever.poll()` to repeatedly load up to `maxPageSize` rows from the database into JVM heap via `Flux.fromStream()`. With no per-client or global subscription cap, an attacker opening hundreds of concurrent streams can exhaust JVM heap, causing an OOM crash that terminates gossip delivery for all legitimate subscribers.

### Finding Description

**Validation entry point — `ConsensusController.toFilter()`** [1](#0-0) 

The controller accepts any `ConsensusTopicQuery`. `startTime` defaults to `DomainUtils.now()` only when the field is absent; if the client sends `consensusStartTime = {seconds:0, nanos:0}`, it is converted and passed through as timestamp `0`.

**Only validation on `startTime` — `TopicMessageFilter.isValidStartTime()`** [2](#0-1) 

The constraint only requires `startTime <= now()`. Epoch-0 (year 1970) passes. There is no minimum recency check, no minimum `endTime` requirement, and no `limit` requirement.

**Per-poll memory load — `PollingTopicMessageRetriever.poll()`** [3](#0-2) 

When `filter.hasLimit()` is false, `limit = Integer.MAX_VALUE`, so `pageSize = Math.min(Integer.MAX_VALUE, maxPageSize)` = `maxPageSize`. For the throttled path (used by the public API), `maxPageSize = 1000`; for the unthrottled path, `maxPageSize = 5000`. [4](#0-3) 

`Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` materialises the JDBC result set into JVM heap. With a topic that has millions of historical messages, each poll call loads a full page of rows.

**Polling continues until all history is drained or timeout fires** [5](#0-4) 

For the throttled path, `isComplete()` returns `true` only when the last page was smaller than `maxPageSize`. With millions of historical rows, this never fires within the 60-second timeout window, so the subscription keeps polling every 2 seconds (≈30 polls × 1000 rows = 30 000 rows per subscription lifetime).

**No concurrent subscription cap** [6](#0-5) 

`subscriberCount` is a metrics gauge only — it is never checked against a maximum. There is no per-IP or global connection limit anywhere in the gRPC stack.

### Impact Explanation

With N concurrent attacker-controlled streams (e.g., N=500), each holding up to 1000 `TopicMessage` objects in heap simultaneously (≈1–2 KB each), the attacker can impose ≈500–1000 MB of sustained heap pressure. Repeated polls refresh this pressure every 2 seconds. On a typical JVM heap of 1–2 GB, this triggers GC thrashing and ultimately `OutOfMemoryError`, crashing the gRPC service. All legitimate subscribers — including those receiving live gossip messages — lose their connections. The Hedera Consensus Service gossip delivery guarantee is broken for the duration of the attack.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond the public protobuf definition, and no on-chain funds. A single attacker machine can open hundreds of gRPC HTTP/2 streams to the same endpoint. The `grpcurl` tool (referenced in the project's own README) is sufficient to script this. The attack is repeatable: after the service restarts, the attacker can immediately re-open streams. No existing network-layer defence (TLS termination, load balancer) prevents this without explicit connection-count or rate-limit configuration that is absent from the codebase.

### Recommendation

1. **Enforce a global and per-IP concurrent subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 200), and track per-source-IP counts via a gRPC `ServerInterceptor`.
2. **Require a minimum `startTime` recency** in `TopicMessageFilter.isValidStartTime()` (e.g., no older than 30 days) or enforce a mandatory `limit` when `startTime` is far in the past.
3. **Cap the total rows a single subscription may retrieve** across all polls (independent of the per-page limit) so that a limitless subscription cannot consume unbounded DB and heap resources.
4. **Use a streaming/cursor JDBC fetch size** in `findByFilter` to avoid materialising the full page into heap at once.
5. **Add gRPC-level rate limiting** (e.g., via an `io.grpc.ServerInterceptor` with a token-bucket per remote address) to bound the subscription creation rate.

### Proof of Concept

```bash
# Open 500 concurrent subscriptions with epoch-0 startTime, no endTime, no limit
# against a topic known to have millions of historical messages (e.g., topic 0.0.41110)
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":41110},
         "consensusStartTime":{"seconds":0,"nanos":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
# Each background stream drives poll() every 2 s, loading 1000 TopicMessage rows.
# Monitor mirror-node JVM heap: expect GC pressure → OOM within ~60 s.
# Legitimate subscribers will receive UNAVAILABLE errors as the JVM crashes.
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
