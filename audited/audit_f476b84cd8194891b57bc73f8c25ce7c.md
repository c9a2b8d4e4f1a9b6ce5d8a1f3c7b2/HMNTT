### Title
Unauthenticated EVM Address Miss-Flooding via Uncached, Unthrottled `findByEvmAddress()` in rest-java Raises DB CPU

### Summary
The `rest-java` module's `EntityRepository.findByEvmAddress()` is invoked on every request to publicly accessible account endpoints when the path parameter is a 20-byte EVM address. Unlike the `web3` module's equivalent, the `rest-java` version has no `@Cacheable` annotation and the `rest-java` module has no rate-limiting layer whatsoever. An unauthenticated attacker can flood these endpoints with random EVM addresses that match no entity, forcing a DB index scan on every request with no cache absorption, and sustain elevated DB CPU load.

### Finding Description

**Exact code path:**

`AllowancesController.getNftAllowances()` and `TokenAirdropsController.getOutstandingAirdrops()`/`getPendingAirdrops()` accept `@PathVariable EntityIdParameter id`. When the path segment matches the EVM address regex (`^(((\\d{1,5})\\.)?((\\d{1,5})\\.)?|0x)?([A-Fa-f0-9]{40})$`), it is parsed as `EntityIdEvmAddressParameter`. [1](#0-0) 

`EntityServiceImpl.lookup()` then dispatches to `entityRepository.findByEvmAddress(p.evmAddress())`: [2](#0-1) 

The repository method issues a raw SQL index scan with no caching: [3](#0-2) 

**Root cause — failed assumption:**

The `web3` module's `EntityRepository` applies `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)` to its EVM address lookup, and the `web3` module enforces a `ThrottleManager` (bucket4j, default 500 req/s) on all its endpoints. [4](#0-3) [5](#0-4) 

The `rest-java` module has **neither**. `RestJavaProperties` contains only response-header configuration — no rate-limit or throttle property exists: [6](#0-5) 

The `QueryProperties` for rest-java only configures `maxFileAttempts`: [7](#0-6) 

On a cache miss, the `web3` cache uses `unless = "#result == null"`, so misses also bypass the cache there — but the web3 rate limiter still caps throughput. The rest-java path has no such cap at all.

**Exploit flow:**

1. Attacker sends `GET /api/v1/accounts/0x<random_40_hex>/allowances/nfts` in a tight loop.
2. `EntityIdEvmAddressParameter.valueOfNullable()` accepts the address (valid format).
3. `EntityServiceImpl.lookup()` calls `entityRepository.findByEvmAddress(randomBytes)`.
4. PostgreSQL performs an index scan on `evm_address`; finds nothing; returns empty.
5. No result is cached. Next request with a different random address repeats the full index scan.
6. At high concurrency (e.g., 10 threads × 1000 req/s each), the DB receives 10,000 index-scan queries per second, all misses, with no throttle or cache to absorb them.

### Impact Explanation

Each miss-inducing request causes a PostgreSQL index scan on the `evm_address` column of the `entity` table. At production scale (millions of entities), even an index-only scan consumes measurable CPU per query. With no rate limiting and no negative-result caching, a sustained flood of unique random EVM addresses translates directly into proportional DB CPU consumption. The `statementTimeout` of 10 seconds does not limit request rate; it only caps individual query duration. The Hikari connection pool limits concurrency but not the rate at which connections are acquired and released. A single attacker machine with HTTP keep-alive and pipelining can saturate the pool and drive DB CPU above the 30% threshold relative to baseline.

### Likelihood Explanation

- **No authentication required**: all affected endpoints (`/api/v1/accounts/{id}/allowances/nfts`, `/api/v1/accounts/{id}/airdrops/outstanding`, `/api/v1/accounts/{id}/airdrops/pending`) are public GET endpoints.
- **No rate limiting in rest-java**: confirmed by code review of `RestJavaProperties`, `QueryProperties`, and all controller/service classes.
- **No caching for misses**: confirmed by absence of `@Cacheable` on `EntityRepository.findByEvmAddress()` in the rest-java module.
- **Trivially scriptable**: a single `curl` loop or `ab`/`wrk` invocation suffices; no special knowledge needed.
- **Repeatable indefinitely**: the attacker simply rotates through the 2^160 EVM address space.

### Recommendation

1. **Add application-level rate limiting to rest-java**: implement a bucket4j or Resilience4j `RateLimiter` filter analogous to `ThrottleManagerImpl` in the web3 module, applied globally to all rest-java endpoints.
2. **Cache negative results for EVM address lookups**: add a short-TTL (e.g., 30s) negative cache entry when `findByEvmAddress()` returns empty, so repeated misses for the same address do not hit the DB. Note that the web3 `@Cacheable` uses `unless = "#result == null"` which intentionally skips caching misses — this should be reconsidered for the rest-java path.
3. **Consider infrastructure-level rate limiting**: deploy a WAF or ingress rate-limit rule on the `/api/v1/accounts/{id}/*` path family as a defense-in-depth measure.

### Proof of Concept

```bash
# Generate random EVM addresses and flood the endpoint
python3 -c "
import os, subprocess, threading

def flood():
    while True:
        addr = '0x' + os.urandom(20).hex()
        subprocess.run([
            'curl', '-s', '-o', '/dev/null',
            f'http://<mirror-node-host>:8080/api/v1/accounts/{addr}/allowances/nfts'
        ])

threads = [threading.Thread(target=flood) for _ in range(50)]
for t in threads: t.start()
for t in threads: t.join()
"
```

Preconditions: network access to the rest-java service port (default 8080). No credentials required. Monitor PostgreSQL `pg_stat_activity` and CPU metrics; DB CPU will rise proportionally to request throughput with no self-limiting mechanism in the application.

     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,

    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())

                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

        }
