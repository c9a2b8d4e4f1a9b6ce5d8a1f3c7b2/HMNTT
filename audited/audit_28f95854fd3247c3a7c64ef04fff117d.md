### Title
Retry Loop Creates Fresh Transaction IDs on Each Attempt, Enabling Multiple On-Chain Transfers Per Tick via gRPC Error Injection

### Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a brand-new `TransferTransaction` (with a new unique transaction ID) on every loop iteration. If an attacker with a man-in-the-middle position on the gRPC channel injects a transient error after a transaction has already been accepted by the network, the function discards the response, creates a new transaction, and resubmits — up to `maxRetries+1` times per tick. Because each retry carries a distinct transaction ID, the Hedera network cannot deduplicate them, resulting in up to 11 independent on-chain fund transfers per scheduled tick (with default config).

### Finding Description

**Exact code path:** `pinger/transfer.go`, `submitWithRetry()`, lines 14–60.

The loop body (lines 29–33) unconditionally calls `hiero.NewTransferTransaction()` on every iteration:

```go
// pinger/transfer.go lines 29-33
cryptoTransfer := hiero.NewTransferTransaction().
    AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
    AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))

resp, err := cryptoTransfer.Execute(client)
```

`hiero.NewTransferTransaction()` generates a fresh transaction ID (operator account + current nanosecond timestamp) on every call. `Execute(client)` signs and submits that transaction to a Hedera consensus node over gRPC. The gRPC call is a two-phase operation: the node receives and processes the transaction, then sends back a response. If the attacker intercepts the response leg and replaces it with a transient gRPC error (e.g., `UNAVAILABLE`, `DEADLINE_EXCEEDED`), `Execute()` returns a non-nil error even though the transaction was already accepted on-chain.

The error path (lines 44–56) logs the failure and sleeps for a backoff period, then the loop continues to the next iteration — where a **new** `NewTransferTransaction()` is called, producing a new transaction ID. The Hedera network has no basis to deduplicate this second submission against the first; it processes it as an independent transfer.

**Failed assumption:** The code assumes that a failed `Execute()` call means the transaction was not submitted. This is false when the failure occurs on the response path after the node has already accepted the transaction.

**No deduplication guard exists.** The only early-exit checks are `ctx.Err()` (context cancellation, line 24) and a successful receipt (line 36–39). There is no idempotency key, no transaction ID reuse, and no check for a prior successful submission before retrying.

Default `maxRetries = 10` (from `pinger/config.go` line 58), so `attempts = 11`. Each of the 11 iterations can independently land on-chain.

### Impact Explanation

Each successful on-chain transfer debits `cfg.amountTinybar` (default 10,000 tinybar = 0.0001 HBAR) from the operator account and credits the destination. With 11 attempts per tick and a 1-second tick interval, an attacker sustaining the attack drains the operator account at up to 11× the intended rate. Over time this constitutes direct, repeated loss of funds from the operator account. Severity is **High** — it is a direct financial loss mechanism with no on-chain safeguard.

### Likelihood Explanation

The precondition is a man-in-the-middle position on the gRPC channel between the pinger pod and the Hedera consensus nodes. Realistic vectors:

- **Compromised container network / service mesh**: An attacker who has compromised another pod in the same Kubernetes namespace or cluster can intercept unencrypted or improperly verified gRPC traffic.
- **BGP hijacking / rogue DNS**: Redirecting the consensus node address to an attacker-controlled endpoint that accepts the transaction (forwarding it to the real node) and then returns a synthetic error.
- **Compromised load balancer or ingress**: Infrastructure-level MITM without requiring application credentials.

The attack is repeatable every tick (default every 1 second), is fully automated, and requires no knowledge of the operator private key. The attacker does not need any Hedera account or privileged application access.

### Recommendation

1. **Create the transaction once, outside the retry loop**, so all retry attempts reuse the same transaction ID. The Hedera network deduplicates submissions with identical transaction IDs within the validity window (default 3 minutes):

```go
cryptoTransfer := hiero.NewTransferTransaction().
    AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
    AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))

for i := 1; i <= attempts; i++ {
    resp, err := cryptoTransfer.Execute(client)
    ...
}
```

2. **On a successful `Execute()` but failed `GetReceipt()`**, query the network for the transaction record by ID before retrying, to confirm whether the transfer actually landed.

3. **Enforce TLS certificate pinning** on the gRPC channel to the consensus nodes to eliminate the MITM vector at the transport layer.

### Proof of Concept

**Preconditions:**
- Attacker controls a transparent proxy on the network path between the pinger and the Hedera consensus node (e.g., via ARP spoofing in the same subnet, or a rogue sidecar in the same Kubernetes pod network).

**Steps:**

1. Configure the proxy to forward all outbound gRPC `CryptoTransfer` requests to the real Hedera node transparently (so the transaction is accepted on-chain).
2. Intercept the gRPC response from the node and replace it with a `StatusCode::UNAVAILABLE` error before it reaches the pinger process.
3. The pinger's `Execute(client)` call returns a non-nil error. The retry loop logs `"attempt 1/11 failed"` and sleeps `baseBackoff` (default 2 s).
4. Repeat step 2 for each of the 11 attempts.
5. Observe on the Hedera mirror node (e.g., via `GET /api/v1/transactions?account.id=<operatorID>`) that 11 distinct `CryptoTransfer` transactions from the operator account appear on-chain within the same tick window, each debiting `amountTinybar`.
