### Title
Unauthenticated Maximum-Range Query Causes Expensive DB Scan in `getNftAllowances()`

### Summary
The `getNftAllowances()` endpoint in `AllowancesController.java` accepts publicly accessible requests with no authentication or rate limiting. An attacker can combine all maximum-cost parameters — `limit=100`, near-full-table-width `account.id` and `token.id` ranges, `owner=false`, and `order=desc` — and every such request passes all existing validation checks, causing the database to perform the widest possible range scan on the `nft_allowance` table on every call. This can be repeated indefinitely by any unprivileged user to exhaust mirror-node database resources.

### Finding Description

**Exact code path:**

`AllowancesController.java`, `getNftAllowances()`, lines 57–83:

```
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
@RequestParam(name = ACCOUNT_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] accountIds,
@RequestParam(name = TOKEN_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] tokenIds
```

`MAX_LIMIT = 100` (`Constants.java` line 35), `@Size(max = 2)` allows two range params per dimension.

**Root cause:** Validation only enforces parameter *format and count*, not the *computational cost* of the resulting query. The following request passes every check:

```
GET /api/v1/accounts/0.0.X/allowances/nfts
  ?limit=100
  &account.id=gte:0.0.1&account.id=lte:0.0.9223372036854775806
  &token.id=gt:0.0.1&token.id=lt:0.0.9223372036854775806
  &owner=false
  &order=desc
```

**Why all checks pass:**

In `NftAllowanceServiceImpl.checkOwnerSpenderParamValidity()` (lines 33–53):

- `ownerOrSpenderParams.isEmpty()` → false (both GTE and LTE present) — no early rejection.
- `verifyRangeId()` → passes: no NE operator, single occurrence of GTE and LTE each, no EQ mixed with range.
- `!ownerOrSpenderParams.hasLowerAndUpper()` → false → the token range validity check at line 42 is **skipped entirely**.
- `tokenParams.getCardinality(LT, LTE)` = 1 > 0 → requires `ownerOrSpenderParams.getCardinality(EQ, LTE)` > 0 → LTE is present → passes.
- `tokenParams.getCardinality(GT, GTE)` = 1 > 0 → requires `ownerOrSpenderParams.getCardinality(EQ, GTE)` > 0 → GTE is present → passes.

All Spring annotation constraints (`@Max(100)`, `@Size(max=2)`, `@Positive`) also pass. The request reaches `repository.findAll(request, id)` with a near-full-table range on both `account.id`/`owner` and `token.id`.

**`owner=false`** causes the query to filter on `NFT_ALLOWANCE.OWNER` (line 65) rather than `SPENDER`, which may have different index coverage, potentially making the scan more expensive.

**No rate limiting** is present anywhere in the controller or visible middleware.

### Impact Explanation

Every such request forces the database to perform a range scan across virtually the entire `nft_allowance` table (account IDs 1 through 9223372036854775806, token IDs 1 through 9223372036854775806) to collect up to 100 rows. On a production mirror node with millions of NFT allowance records, this is a full or near-full index scan per request. Sustained concurrent requests from one or more attackers can saturate database I/O, CPU, and connection pool resources, degrading or denying service to all legitimate users of the mirror node. There is no on-chain economic cost to the attacker.

### Likelihood Explanation

The endpoint is fully public — no authentication, no API key, no rate limit. The exploit requires only knowledge of the parameter syntax (publicly documented). It is trivially scriptable: a single `curl` loop or any HTTP load tool suffices. The attacker needs zero privileges and zero funds. The attack is repeatable indefinitely.

### Recommendation

1. **Rate limiting:** Apply per-IP (and optionally per-account-ID path parameter) rate limiting at the gateway or Spring filter layer for this endpoint.
2. **Range width cap:** Reject requests where the effective range width of `account.id` or `token.id` exceeds a configurable maximum (e.g., reject if `adjustedUpper - adjustedLower > MAX_RANGE_WIDTH`).
3. **Query timeout:** Enforce a short database query timeout for this endpoint so runaway scans are killed before exhausting resources.
4. **Index audit:** Ensure the `nft_allowance` table has a composite index covering `(owner, token_id)` and `(spender, token_id)` so range scans are bounded by the index rather than heap reads.
