### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables Full Table Scan via `DISTINCT ON` Query

### Summary
An unprivileged user can supply a single lower-bound timestamp parameter (e.g., `timestamp=gte:0`) to the historical hook storage endpoint. This makes `isHistorical()` return `true` while leaving `adjustUpperBound()` returning `Long.MAX_VALUE` and `getAdjustedLowerRangeValue()` returning `0`, effectively removing the timestamp filter from the native SQL query. Because the query uses `DISTINCT ON (key)` with an `ORDER BY`, PostgreSQL must scan every row for the given `owner_id`/`hook_id` before applying the `LIMIT`, making repeated requests a practical DB I/O exhaustion vector.

### Finding Description

**Code path:**

`HookServiceImpl.getHookStorageChange()` — [1](#0-0) 

**Step 1 — `isHistorical()` gate is bypassed by a single-sided bound.**

`isHistorical()` returns `true` whenever `!timestamp.isEmpty()`, and `Bound.isEmpty()` is `true` only when *both* `lower` and `upper` are `null`. [2](#0-1) [3](#0-2) 

Supplying only a lower-bound (`timestamp=gte:0`) sets `lower` but leaves `upper = null`, so `isEmpty()` is `false` and the historical path is entered.

**Step 2 — `adjustUpperBound()` returns `Long.MAX_VALUE` when `upper` is null.** [4](#0-3) 

**Step 3 — `getAdjustedLowerRangeValue()` returns `0` when `lower` is null, or `0` when `lower.value()` is `0` with `GTE`.** [5](#0-4) 

**Step 4 — The native SQL query receives `consensus_timestamp BETWEEN 0 AND 9223372036854775807`**, which is a no-op filter matching every row ever written for that `owner_id`/`hook_id`. [6](#0-5) 

**Step 5 — `DISTINCT ON (key) … ORDER BY key, consensus_timestamp DESC` forces PostgreSQL to materialise the full result set for the given `owner_id`/`hook_id` before the `LIMIT` clause is applied.** The `LIMIT` (default 25) does not prevent the full scan; it only limits the rows returned to the caller. [7](#0-6) 

**Why existing checks fail:**

- The `Bound` constructor validates `adjustedLower > adjustedUpper` only when `primarySortField = true`; `Bound.of()` always passes `false` for timestamp bounds, so no range-width check is ever performed. [8](#0-7) 
- `Bound.of()` only rewrites `EQ` to `LTE`; it imposes no maximum span constraint. [9](#0-8) 
- No controller-level validation was found that caps the timestamp range width.

### Impact Explanation
Every request with `timestamp=gte:0` (or any single lower-bound) triggers a full sequential (or index-range) scan of `hook_storage_change` for the targeted `owner_id`/`hook_id`. On a production node with millions of storage-change records, each such request can consume significant DB I/O and CPU (sort + dedup for `DISTINCT ON`). Sending a modest stream of concurrent requests (no brute-force required — each request is individually expensive) can push DB I/O utilisation well above 30% above baseline, satisfying the stated impact threshold. The attack requires no authentication and no knowledge beyond a valid `owner_id`/`hook_id` pair, both of which are public on-chain identifiers.

### Likelihood Explanation
The precondition is trivially met: any user who can reach the REST API endpoint can supply `timestamp=gte:0`. The attack is repeatable, stateless, and requires no special tooling — a simple HTTP client loop suffices. The attacker does not need to know internal state; they only need a valid `owner_id` and `hook_id`, which are enumerable from public ledger data.

### Recommendation
1. **Enforce a maximum timestamp range width.** In `Bound.of()` or in `getHookStorageChange()`, reject or cap requests where `adjustUpperBound() - getAdjustedLowerRangeValue()` exceeds a configurable maximum (e.g., 31 days in nanoseconds).
2. **Require both bounds.** If the timestamp parameter is present, require both a lower and an upper bound; reject single-sided open-ended ranges for the historical path.
3. **Add a default upper bound.** If `upper` is null when entering `getHookStorageChange()`, default it to `now()` (current consensus timestamp) rather than `Long.MAX_VALUE`.
4. **Rate-limit** the historical storage endpoint per source IP/entity.

### Proof of Concept

```
# Preconditions: valid ownerId and hookId are known (public on-chain data)
# No authentication required

GET /api/v1/hooks/{ownerId}/{hookId}/storage?timestamp=gte:0

# Result:
# timestampLowerBound = 0
# timestampUpperBound = Long.MAX_VALUE (9223372036854775807)
# SQL executed:
#   SELECT DISTINCT ON (key) ... FROM hook_storage_change
#   WHERE owner_id = <ownerId>
#     AND hook_id  = <hookId>
#     AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
#   ORDER BY key, consensus_timestamp DESC
#   LIMIT 25
#
# PostgreSQL must scan ALL rows for this owner/hookId to satisfy DISTINCT ON.
# Repeat in a loop (e.g., 50 concurrent clients) to exhaust DB I/O.
```

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
    }

    public static Bound of(TimestampParameter[] timestamp, String parameterName, Field<Long> field) {
        if (timestamp == null || timestamp.length == 0) {
            return Bound.EMPTY;
        }

        for (int i = 0; i < timestamp.length; ++i) {
            final var param = timestamp[i];
            if (param.operator() == RangeOperator.EQ) {
                timestamp[i] = new TimestampParameter(RangeOperator.LTE, param.value());
            }
        }

        return new Bound(timestamp, false, parameterName, field);
    }
