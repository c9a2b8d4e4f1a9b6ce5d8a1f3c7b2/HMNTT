### Title
Deleted Sender Entity Causes Silent Treasury Substitution in Opcode Replay, Producing Incorrect `msg.sender` Trace

### Summary
`OpcodeServiceImpl.getSenderAddress()` resolves the sender using the **current** entity state (`Optional.empty()`) rather than the historical state at the transaction's `consensusTimestamp`. When the original sender entity has since been deleted, `evmAddressFromId` returns `Address.ZERO`. Downstream in `TransactionExecutionService.getSenderAccountID()`, the zero-address + zero-value guard silently substitutes the treasury account as the EVM sender, causing the opcode replay to execute with the wrong `msg.sender` and producing a structurally misleading trace that any unprivileged caller can trigger.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` → `OpcodeServiceImpl.processOpcodeCall()` → `buildCallServiceParameters()` → `getSenderAddress(contractResult)`:

```java
// OpcodeServiceImpl.java line 177-180
private Address getSenderAddress(ContractResult contractResult) {
    final var address = commonEntityAccessor.evmAddressFromId(
            contractResult.getSenderId(), Optional.empty());   // ← always current state
    return address != null ? address : EMPTY_ADDRESS;
}
```

`evmAddressFromId` with `Optional.empty()` calls `findByIdAndDeletedIsFalse`:

```java
// CommonEntityAccessor.java line 91-95
.orElseGet(() -> entityRepository
        .findByIdAndDeletedIsFalse(entityId.getId())
        .orElse(null));
if (entity == null) {
    return Address.ZERO;          // deleted entity → Address.ZERO
}
```

The returned `Address.ZERO` is stored as `sender` in `ContractDebugParameters` (line 172). When the EVM replay is set up, `TransactionExecutionService.getSenderAccountID()` hits:

```java
// TransactionExecutionService.java line 241-242
if (params.getSender().isZero() && params.getValue() == 0L) {
    return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
}
```

The treasury account is silently substituted as the EVM sender. The replay proceeds with treasury as `msg.sender` in every `CALLER` opcode, diverging from the original execution.

**Root cause:** `getSenderAddress` ignores the `consensusTimestamp` that is available in the same `buildCallServiceParameters` scope (line 169 uses it for `block`, line 154 fetches `contractResult` by it), but never passes it to `evmAddressFromId`. The historical lookup path (`findActiveByIdAndTimestamp`) would correctly resolve the sender as it existed at transaction time.

**Failed assumption:** The code assumes that any entity referenced by a stored `ContractResult.senderId` will remain non-deleted indefinitely. Hedera accounts can be deleted after a transaction is finalized.

### Impact Explanation
The opcode trace returned to the caller contains `CALLER` opcodes reflecting the treasury address, not the original sender. Any contract logic gated on `msg.sender` (access control checks, ownership assertions, allowance logic) is replayed against the wrong identity, producing a trace that does not match what actually executed on-chain. Security auditors or automated tools consuming this endpoint to verify historical contract behavior receive structurally incorrect data. The `failed` field in the response may also differ from the original outcome if the contract's execution path branches on `msg.sender`.

### Likelihood Explanation
The precondition — a historical contract transaction whose sender account was subsequently deleted — is a normal Hedera lifecycle event (accounts can be deleted via `CryptoDelete`). No special privilege is required: `OpcodesController` has no authentication guard, only a gzip header check and a rate limiter. Any public caller who knows a transaction ID or hash meeting the precondition can trigger the incorrect replay repeatedly.

### Recommendation
Pass the historical `consensusTimestamp` to `getSenderAddress` so the lookup uses the entity's state at transaction time:

```java
// OpcodeServiceImpl.java — fix getSenderAddress signature
private Address getSenderAddress(ContractResult contractResult, Long consensusTimestamp) {
    final var address = commonEntityAccessor.evmAddressFromId(
            contractResult.getSenderId(), Optional.of(consensusTimestamp));
    return address != null ? address : EMPTY_ADDRESS;
}
```

Update the call site in `buildCallServiceParameters` (line 172) to pass `consensusTimestamp`. Apply the same fix to `getReceiverAddress` (line 203) which has the identical omission for the contract-id lookup path.

### Proof of Concept

1. Submit a Hedera contract-call transaction from account `0.0.X` (value = 0 tinybars). Record the transaction ID or Ethereum hash.
2. After the transaction is finalized and its `ContractResult` is mirrored, delete account `0.0.X` via `CryptoDelete`.
3. Wait for the mirror node to process the deletion (entity marked `deleted=true`).
4. As an unprivileged user, call:
   ```
   GET /api/v1/contracts/results/<txId>/opcodes
   Accept-Encoding: gzip
   ```
5. Observe that the returned opcode trace contains `CALLER` values equal to the treasury account address (e.g., `0x0000000000000000000000000000000000000062`) rather than the original sender's address.
6. If the replayed contract contains `require(msg.sender == originalSender)`, the trace will show a revert that never occurred on-chain, or vice-versa.
