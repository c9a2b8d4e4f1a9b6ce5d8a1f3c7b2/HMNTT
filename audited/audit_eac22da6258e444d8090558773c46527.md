### Title
Unbounded Netty Write-Buffer Growth via TCP Flow-Control Stall in `subscribeTopic()`

### Summary
`ConsensusController.subscribeTopic()` drives the outbound gRPC stream with a plain `subscribe(responseObserver::onNext, …)` call that requests `Long.MAX_VALUE` items and never consults `ServerCallStreamObserver.isReady()`. Because gRPC-Java's `onNext()` is non-blocking (it enqueues serialised frames into Netty's channel write buffer), the Reactor pipeline continues producing and handing off messages even after the HTTP/2 flow-control window is exhausted, causing Netty's write buffer to grow without an application-level bound. An unprivileged attacker who opens a subscription and then stops reading from the socket can therefore force the server to accumulate arbitrarily large amounts of heap-allocated byte buffers, degrading memory availability for all other subscribers.

### Finding Description

**Exact code path**

`ConsensusController.subscribeTopic()` (lines 44–48):
```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
    .flatMapMany(topicMessageService::subscribeTopic)
    .map(this::toResponse)
    .onErrorMap(ProtoUtil::toStatusRuntimeException)
    .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
``` [1](#0-0) 

The single-argument `subscribe(consumer)` overload internally issues an unbounded demand (`Long.MAX_VALUE`) to the upstream `Flux`. There is no call to `ServerCallStreamObserver.isReady()` and no `setOnReadyHandler()` registered to pause production when the transport is congested. [2](#0-1) 

The only backpressure operator in the pipeline lives in `SharedTopicListener.listen()`:
```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
.publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
``` [3](#0-2) 

`maxBufferSize` defaults to 16 384 and `prefetch` to 48. [4](#0-3) 

**Root cause and failed assumption**

The design assumes that `onBackpressureBuffer(16384, ERROR)` caps memory usage. That assumption is false. `onBackpressureBuffer` only buffers items when the *Reactor-level* downstream is slow. Because `responseObserver.onNext()` in gRPC-Java/Netty is **non-blocking** — it serialises the protobuf frame and enqueues it in Netty's `ChannelOutboundBuffer` without waiting for TCP acknowledgement — the Reactor pipeline always consumes items from the 16 384-slot buffer as fast as they are produced. The buffer therefore never fills up from Reactor's perspective. The actual accumulation happens in Netty's write buffer, which has no application-level size limit.

**Exploit flow**

1. Client opens a TCP connection and calls `subscribeTopic` for a high-volume topic (no credentials required).
2. Client stops calling `read()` on the socket (or sets `SO_RCVBUF` to zero and never drains it).
3. The TCP receive window on the client side fills; the server's TCP send buffer fills; HTTP/2 stream-level flow-control window exhausts.
4. Netty can no longer flush frames to the wire, but `responseObserver.onNext()` continues to succeed synchronously, enqueuing serialised `ConsensusTopicResponse` frames into `ChannelOutboundBuffer`.
5. The Reactor pipeline (driven by Redis pub/sub or DB polling) keeps emitting messages at the listener's polling rate (default every 500 ms, up to 5 000 messages per page).
6. Netty's write buffer grows until GC pressure causes latency spikes or an `OutOfMemoryError` terminates the JVM.

**Why existing checks fail**

| Check | Why it does not help |
|---|---|
| `onBackpressureBuffer(16384, ERROR)` | Reactor-level only; `onNext()` is non-blocking so the buffer is never the bottleneck |
| `BufferOverflowStrategy.ERROR` | Never triggered for the reason above |
| `maxConcurrentCallsPerConnection = 5` | Limits streams per TCP connection; attacker opens many connections |
| `setOnCancelHandler(disposable::dispose)` | Fires only on explicit client cancel, not on a stalled-but-open stream |
| No `isReady()` / `setOnReadyHandler()` | Absent entirely; gRPC's built-in flow-control signal is never consulted | [5](#0-4) 

### Impact Explanation
Each stalled subscription causes Netty to accumulate heap-allocated byte buffers proportional to `(message rate) × (message size) × (stall duration)`. For a topic receiving 1 000 messages/s at 1 KB each, a 60-second stall accumulates ~60 MB per stream. With multiple connections (each carrying 5 streams), an attacker can exhaust gigabytes of heap, causing GC pauses that degrade latency for all legitimate subscribers or triggering an OOM that kills the process. No authentication or special privilege is required.

### Likelihood Explanation
Any internet-reachable gRPC endpoint is exploitable. The attacker needs only a standard gRPC client library, a known topic ID, and the ability to open TCP connections without reading. The attack is trivially repeatable and scriptable. The only practical barrier is knowing a topic ID, which is public on-chain information.

### Recommendation
1. **Register a ready handler** on `ServerCallStreamObserver` and pause/resume the upstream `Flux` accordingly:
   ```java
   if (responseObserver instanceof ServerCallStreamObserver<ConsensusTopicResponse> scso) {
       scso.setOnCancelHandler(disposable::dispose);
       scso.setOnReadyHandler(() -> { /* resume upstream */ });
       scso.disableAutoInboundFlowControl();
   }
   ```
2. **Replace the plain `subscribe()` call** with a `BaseSubscriber` that requests items one-at-a-time (or in small batches) and only requests the next item after `isReady()` returns `true`.
3. **Set a Netty write-buffer high-watermark** via `NettyServerBuilder` and close streams that remain above the watermark for longer than a configurable timeout.
4. **Add a per-subscriber timeout** (e.g., `flux.timeout(Duration.ofSeconds(30))`) so stalled streams are reaped automatically.

### Proof of Concept
```python
import grpc, time
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery()
query.topicID.topicNum = <HIGH_VOLUME_TOPIC_ID>

# Open the stream but never iterate the response iterator
stream = stub.subscribeTopic(query)

# Stall: do not call next(stream) — TCP window fills, server buffers grow
time.sleep(300)  # hold for 5 minutes; monitor server heap via JMX/Prometheus
```
Observe `hiero.mirror.grpc.subscribers` gauge staying elevated and JVM heap (`jvm.memory.used`) growing monotonically for the duration of the stall. Repeat with N parallel connections to amplify heap pressure.
