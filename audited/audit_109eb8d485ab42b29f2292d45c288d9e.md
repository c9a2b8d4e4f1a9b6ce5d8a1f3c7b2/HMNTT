### Title
Unauthenticated Connection Pool Exhaustion via High-Concurrency GET /blocks Requests

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` performs an unconstrained database query for every incoming request with no per-IP rate limiting or in-flight request cap. The response cache has a default TTL of only 1 second, meaning the database connection pool is re-exposed to direct hits every second. An unauthenticated attacker sending thousands of concurrent identical requests can exhaust the finite pg-pool connection pool, causing all other REST API users to receive 500 errors or connection timeout failures.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:12` → `BlockController.getBlocks` → `rest/controllers/blockController.js:101-112`:

```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);  // DB hit, holds pool connection
  ...
};
```

Every call to `getBlocks()` acquires a connection from the shared global pool defined in `rest/dbpool.js:14`:

```js
max: config.db.pool.maxConnections,
```

The pool is a single shared resource across all concurrent requests.

**Cache mitigation is insufficient:**

`rest/middleware/responseCacheHandler.js:24` sets:
```js
const DEFAULT_REDIS_EXPIRY = 1;  // 1 second TTL
```

The cache key is `MD5(req.originalUrl)`. For the exact URL `GET /blocks?block.number=gte:1&block.number=lte:1`, the cache is populated after the first response — but expires after **1 second**. An attacker sustaining high concurrency re-exhausts the pool on every cache expiry cycle.

**No per-IP rate limiting on the REST API:**

The REST middleware stack (`rest/middleware/index.js`) contains no in-flight request cap or per-IP rate limiter for the `/blocks` route. The Traefik middleware chain in `charts/hedera-mirror-rest/templates/middleware.yaml` is only applied when `global.middleware: true` is set, which is an optional deployment configuration. The application itself has no built-in throttle for this endpoint.

**Root cause:** The combination of (a) no application-level rate limiting, (b) a 1-second cache TTL, and (c) a finite shared connection pool creates a repeating 1-second window in which unlimited concurrent requests each hold a pool connection.

### Impact Explanation

When the pool (`max: config.db.pool.maxConnections`) is exhausted, new requests block until `connectionTimeoutMillis` elapses, then fail. All legitimate REST API users — querying transactions, accounts, tokens, etc. — share the same pool and receive 500/503 errors for the duration of the attack. The pgbouncer `max_user_connections: 250` for `mirror_rest` (from `charts/hedera-mirror/values.yaml:372-373`) caps actual DB connections, but the Node.js pool queue fills up in memory and the `connectionTimeoutMillis` timeout causes cascading failures visible to all users. This constitutes a full REST API denial-of-service.

### Likelihood Explanation

No authentication, API key, or special capability is required. The attack requires only an HTTP client capable of high concurrency (e.g., `wrk`, `ab`, `hey`, or a simple async script). The query `block.number=gte:1&block.number=lte:1` is a valid, accepted parameter combination — it passes all input validation in `buildAndValidateFilters`. The attack is trivially repeatable and scriptable from a single machine or botnet. The 1-second cache window means the attacker must sustain ~1000+ RPS to keep the pool exhausted, which is achievable with commodity hardware.

### Recommendation

1. **Add per-IP in-flight request limiting** at the application layer (e.g., via `express-rate-limit` or a Traefik `inFlightReq` middleware enforced unconditionally), similar to the Rosetta deployment which enforces `inFlightReq: amount: 5` per IP.
2. **Increase the default cache TTL** beyond 1 second for stable, read-heavy endpoints like `/blocks`. A TTL of 5–10 seconds dramatically reduces the attack surface.
3. **Enforce Traefik middleware unconditionally** (set `global.middleware: true` as a required default, not opt-in) so the rate-limit and circuit-breaker chain is always active.
4. **Add a `statement_timeout`** at the query level (already present via pool config) and verify it is short enough (e.g., ≤500ms) to release connections quickly under load.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wg1/wrk) or equivalent
# No credentials needed

wrk -t 50 -c 1000 -d 60s \
  "http://<mirror-node-host>/api/v1/blocks?block.number=gte:1&block.number=lte:1"

# Simultaneously, from another terminal, observe legitimate requests failing:
for i in $(seq 1 100); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    "http://<mirror-node-host>/api/v1/transactions?limit=1"
done
# Expected: mix of 200 and 500/503 responses during attack window
```

The attack sustains pool exhaustion across repeated 1-second cache expiry cycles. Legitimate users on any REST endpoint sharing the same pool will observe failures for the duration.
