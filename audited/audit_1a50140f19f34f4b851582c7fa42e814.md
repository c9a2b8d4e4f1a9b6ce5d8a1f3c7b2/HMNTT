### Title
Integer Overflow in `poll()` Bypasses DB Query LIMIT via Negative `pageSize`

### Summary
In `PollingTopicMessageRetriever.poll()`, a user-supplied `limit` value greater than `Integer.MAX_VALUE` (valid in the `uint64` proto field) is narrowed to a negative `int` via an unchecked cast. `Math.min(negative_limit, positive_maxPageSize)` returns the negative value, which is then set as the `limit` on a new `TopicMessageFilter`. Because `hasLimit()` returns `false` for any value ‚Č§ 0, `TopicMessageRepositoryCustomImpl.findByFilter()` omits `setMaxResults` entirely, issuing an unbounded SQL query that can return every message stored for the target topic.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` reads `query.getLimit()` directly from the protobuf `uint64` field and stores it as a Java `long` in `TopicMessageFilter.limit` (no upper-bound constraint exists, only `@Min(0)`). [1](#0-0) 

The proto field is declared `uint64`, so any value from 0 to 2‚Ā∂‚Āī‚ąí1 is wire-legal. [2](#0-1) 

Inside `poll()`, the remaining limit is computed with a narrowing cast:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [3](#0-2) 

If `filter.getLimit()` is `2147483648L` (2¬≥¬Ļ), the cast `(int)(2147483648L)` yields `Integer.MIN_VALUE` (‚ąí2147483648). `maxPageSize` is a server-side config value (default 1000, minimum 32) ‚ÄĒ it is always positive. [4](#0-3) 

`Math.min(-2147483648, 1000)` = `‚ąí2147483648`. This negative `pageSize` is written into a new filter via `toBuilder().limit(pageSize)`. The `@Min(0)` annotation on `TopicMessageFilter.limit` is **not enforced** here because the builder is called directly in private code, not through a Spring AOP proxy. [5](#0-4) 

In `findByFilter`, `hasLimit()` checks `limit > 0`. A negative value fails this check, so `setMaxResults` is never called and the JPA query has no `LIMIT` clause: [6](#0-5) 

The entire result set for the topic (potentially millions of rows) is loaded into memory via `getResultList()`: [7](#0-6) 

**Why the outer `take()` does not save you:**
`TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` where `filter.getLimit()` is the original user-supplied `long` value (e.g., 2¬≥¬Ļ = 2,147,483,648). The DB query has already executed and loaded all rows into the JVM heap before the reactive `take` operator can discard them. [8](#0-7) 

### Impact Explanation
An attacker can force the mirror node to execute an unbounded `SELECT` against the `topic_message` table for any topic, loading the full result set into JVM heap memory. On a busy network with a high-volume topic, this causes heap exhaustion and JVM crash (OOM), taking down the gRPC service for all subscribers. Repeated requests amplify the effect. This is a denial-of-service with no authentication required. The "direct loss of funds" label in the question is inaccurate ‚ÄĒ the actual impact is service availability loss and potential data exfiltration of all topic messages.

### Likelihood Explanation
The gRPC endpoint is publicly accessible. The only precondition is knowing a valid `topicId` (trivially discoverable from the public ledger). Sending a single `ConsensusTopicQuery` with `limit = 2147483648` is sufficient to trigger the bug. No credentials, no special role, no prior state required. The attack is trivially repeatable and scriptable.

### Recommendation
Replace the unchecked narrowing cast with a safe clamp before the `Math.min`:

```java
// In poll(), replace:
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;

// With:
long remaining = filter.hasLimit()
        ? filter.getLimit() - context.getTotal().get()
        : (long) Integer.MAX_VALUE;
int limit = (int) Math.min(remaining, Integer.MAX_VALUE);  // safe clamp, never negative
```

Additionally, add an `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) constraint to `TopicMessageFilter.limit`, and enforce validation at the gRPC controller layer before the filter is accepted. [9](#0-8) 

### Proof of Concept

1. Identify any valid `topicId` on the network (e.g., from the REST API or public explorer).
2. Send a gRPC `subscribeTopic` request:
   ```
   ConsensusTopicQuery {
     topicID: { shardNum: 0, realmNum: 0, topicNum: <valid_topic> }
     limit: 2147483648   // 2^31, fits in uint64 wire format
   }
   ```
3. Observe in server logs/metrics: a single DB query with no LIMIT clause executes against `topic_message`, loading all rows for that topic.
4. On a topic with millions of messages, the JVM heap is exhausted and the gRPC service crashes with `OutOfMemoryError`.
5. Repeat to keep the service down.
