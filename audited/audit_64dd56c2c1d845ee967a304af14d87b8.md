### Title
Unauthenticated IP Spoofing via Trusted Proxy Headers Enables Audit Log Poisoning in `TracingMiddleware`

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` unconditionally trusts the `X-Real-IP` and `X-Forwarded-For` HTTP headers supplied by any client, with no validation that the request originates from a trusted proxy. Any unprivileged external attacker can set these headers to arbitrary values, causing every access log entry to record a fabricated source IP instead of the real `RemoteAddr`. This makes it impossible to reconstruct the true origin of transaction history lookups from audit logs.

### Finding Description
**Exact code path:**

`getClientIpAddress` (lines 63–74) resolves the client IP with strict header priority:

```go
// rosetta/app/middleware/trace.go, lines 63-74
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // X-Real-IP checked first
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // X-Forwarded-For second
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // real IP only as last resort
    }
    return ipAddress
}
```

`TracingMiddleware` (lines 43–61) then logs this attacker-controlled value at `log.Info` level for every non-internal endpoint:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
log.Info(message)
```

**Root cause:** There is no allowlist of trusted upstream proxy IPs, no comparison against `r.RemoteAddr`, and no format/range validation of the header value. The `r.RemoteAddr` (the only unforgeable network-layer IP) is only consulted when both headers are absent.

**Failed assumption:** The code assumes `X-Real-IP` / `X-Forwarded-For` are set exclusively by a trusted reverse proxy. In practice, the server is directly reachable by external clients (confirmed by `main.go` line 218–219 where `TracingMiddleware` wraps the router with no prior proxy-trust layer), so any client controls these headers.

**The test suite itself confirms the attack surface** — `trace_test.go` lines 52–58 explicitly verify that a client-supplied `X-Real-IP` or `X-Forwarded-For` header overrides the real remote address, treating this as correct behavior.

### Impact Explanation
Every `/block/transaction`, `/block`, `/account/balance`, and other Rosetta data-plane query is logged at `Info` level with the attacker-supplied IP. An attacker who cycles through thousands of spoofed IPs across repeated queries to the same transaction identifiers:

- Destroys the audit trail for those transaction lookups — operators cannot determine which real IP performed reconnaissance on specific transactions.
- Can inject log entries that attribute queries to arbitrary third-party IPs (e.g., known exchange or custodian addresses), framing them for suspicious access patterns.
- Can flood logs with high-volume noise (no rate limiting exists in the middleware stack) to bury legitimate entries, making SIEM/alerting systems ineffective.

The Rosetta API is a read-only query interface, so no on-chain state is altered; however, the integrity of the access audit log — the only mechanism for detecting who queried sensitive transaction history — is fully compromised.

### Likelihood Explanation
- **Zero privilege required:** Any HTTP client reachable to the Rosetta port can exploit this.
- **Trivially scriptable:** A single `curl -H "X-Real-IP: 1.2.3.4" http://<host>/block/transaction` suffices; rotating IPs requires only a loop.
- **No detection:** Because the spoofed IP is what gets logged, the attack is self-concealing.
- **Repeatable at scale:** No rate limiting, connection throttling, or request quotas exist in `TracingMiddleware` or the surrounding middleware chain (`main.go` lines 217–219).

### Recommendation
1. **Validate proxy source before trusting headers:** Only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` matches a configured allowlist of trusted proxy CIDR ranges. Reject or ignore these headers from all other sources.
2. **Always log `r.RemoteAddr` alongside the resolved IP:** Even when a proxy header is trusted, log both values so the network-layer origin is always preserved.
3. **Add rate limiting middleware** upstream of `TracingMiddleware` to bound log volume per real `RemoteAddr`.

Example fix for `getClientIpAddress`:
```go
func getClientIpAddress(r *http.Request, trustedProxies []*net.IPNet) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); ip != "" {
            return ip
        }
        if ip := r.Header.Get(xForwardedForHeader); ip != "" {
            return ip
        }
    }
    return remoteIP
}
```

### Proof of Concept
**Preconditions:** Network access to the Rosetta HTTP port (default 5700). No credentials needed.

**Steps:**
```bash
# Flood logs with 1000 different spoofed source IPs for the same transaction query
for i in $(seq 1 1000); do
  curl -s -X POST http://<rosetta-host>:5700/block/transaction \
    -H "Content-Type: application/json" \
    -H "X-Real-IP: 10.${i%256}.${i}.1" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1},
         "transaction_identifier":{"hash":"0x..."}}' &
done
wait
```

**Result:** The server log will contain 1000 `Info`-level entries for `/block/transaction`, each attributed to a different fabricated IP.
