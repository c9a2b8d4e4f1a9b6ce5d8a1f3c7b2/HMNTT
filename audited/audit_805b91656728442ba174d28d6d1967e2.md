### Title
Unbounded Concurrent gRPC Subscriptions with Eager Full-Page Heap Materialization Enabling JVM OOM DoS

### Summary
`TopicMessageRepositoryCustomImpl.findByFilter()` calls `getResultList()` which eagerly materializes the entire result set (up to 5,000 `TopicMessage` objects) into a `List<TopicMessage>` on the heap before returning a `Stream`. Because `maxConcurrentCallsPerConnection` is a per-connection limit (not a global one) and no global subscription cap or gRPC-layer rate limit exists, an unauthenticated attacker opening many connections can force concurrent heap allocations of up to 5,000 × ~6 KB = ~30 MB per active poll, exhausting JVM heap and causing an OOM-based denial of service.

### Finding Description

**Exact code path:**

`TopicMessageRepositoryCustomImpl.findByFilter()` — [1](#0-0)  — calls `typedQuery.getResultList().stream()`. The comment on that line explicitly acknowledges the design choice: `// getResultStream()'s cursor doesn't work with reactive streams`. This means the full JDBC result set is fetched and held as a `java.util.List<TopicMessage>` in heap before any item is emitted downstream.

`PollingTopicMessageRetriever.poll()` — [2](#0-1)  — wraps this with `Flux.fromStream(...)`. The list is materialized at the moment `poll()` is invoked on the `boundedElastic` scheduler thread and remains live in heap until the stream is fully consumed downstream (i.e., until the gRPC client reads all items).

**Page size configuration:**

- Throttled path: `maxPageSize = 1000` [3](#0-2) 
- Unthrottled path: `maxPageSize = 5000` [4](#0-3) 

**Root cause — failed assumption:**

The design assumes the `boundedElastic` scheduler's thread cap (`10 × availableProcessors`, e.g. 40 threads on a 4-core host) is a sufficient memory guard. It is not, because:

1. The `List<TopicMessage>` is allocated eagerly and held in heap for the entire duration of downstream consumption. A slow or non-reading gRPC client keeps the list alive indefinitely (up to the 60 s timeout).
2. `maxConcurrentCallsPerConnection = 5` — [5](#0-4)  — is enforced per TCP connection, not globally. An attacker opens many connections.
3. `subscriberCount` — [6](#0-5)  — is a Micrometer gauge for observability only; it is never checked against a maximum.
4. There is no rate limiter on gRPC subscription creation. The `ThrottleConfiguration` / `ThrottleManagerImpl` exists only in the `web3` module, not in `grpc`. [7](#0-6) 

**Exploit flow:**

1. Attacker opens `C` TCP connections to port 5600 (no authentication required).
2. On each connection, attacker opens 5 `subscribeTopic` streams (the per-connection maximum), each targeting a topic with ≥ 5,000 historical messages and no `limit` field set.
3. `TopicMessageServiceImpl.subscribeTopic()` calls `topicMessageRetriever.retrieve(filter, true)` — [8](#0-7)  — which schedules repeated `poll()` calls.
4. Each `poll()` invocation calls `findByFilter()` → `getResultList()`, allocating a `List<TopicMessage>` of up to 5,000 objects (~30 MB including message payloads of up to 6 KB each per the HCS proto spec).
5. The attacker's client reads responses at a near-zero rate, keeping each list live in heap.
6. With `C` connections × 5 subscriptions = `5C` concurrent subscriptions, and the `boundedElastic` scheduler running up to `10 × CPU` threads simultaneously, heap fills: e.g. 40 threads × 30 MB = 1.2 GB on a 4-core host. With a larger `C`, the task queue (100,000 capacity) ensures a continuous stream of new allocations as threads complete.
7. JVM heap exhausts → `OutOfMemoryError` → service crash.

### Impact Explanation

A successful attack crashes the gRPC mirror node service, making HCS topic subscriptions unavailable to all legitimate clients. Because the mirror node gRPC service is a critical read path for Hedera Consensus Service consumers (wallets, dApps, validators), this constitutes a high-impact non-network DoS. The `PollingTopicListener` path — [9](#0-8)  — also calls `findByFilter()` with the same `getResultList()` pattern, doubling the attack surface per subscription.

### Likelihood Explanation

The gRPC port (5600) is publicly exposed with no authentication and no global subscription rate limit. Any client with network access can open arbitrarily many TCP connections. The only per-connection guard (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening more connections. Topics with thousands of messages exist on mainnet (e.g., high-frequency HCS topics used by DeFi protocols). The attack requires no special knowledge beyond the public gRPC proto definition and is repeatable and automatable.

### Recommendation

1. **Enforce a global concurrent-subscription cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP / global gRPC subscription rate limiting**: Introduce a token-bucket interceptor (analogous to `ThrottleManagerImpl` in `web3`) on the gRPC server.
3. **Replace eager `getResultList()` with a true server-side cursor**: Use `setFetchSize()` + `getResultStream()` with a workaround for the reactive-streams incompatibility (e.g., wrap in a `Flux.create` with a blocking iterator on the `boundedElastic` scheduler), so only one page's worth of rows is in memory at a time rather than the full result list.
4. **Reduce default `unthrottled.maxPageSize`** from 5,000 to a lower value, or make it operator-configurable with a hard ceiling.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_ID = basic_types_pb2.TopicID(topicNum=<high_volume_topic>)
NUM_CONNECTIONS = 200  # 200 connections × 5 streams = 1000 concurrent subscriptions

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    streams = []
    for _ in range(5):  # maxConcurrentCallsPerConnection = 5
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=TOPIC_ID,
            consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
            # no limit set → retriever polls indefinitely
        )
        stream = stub.subscribeTopic(req)
        streams.append(stream)
    # Read at near-zero rate to keep List<TopicMessage> live in heap
    import time
    time.sleep(120)

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Expected result: gRPC service OOM-crashes within seconds to minutes
```
