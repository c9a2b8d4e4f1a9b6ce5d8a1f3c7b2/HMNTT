### Title
Global Gas Bucket DoS: Single Unprivileged Attacker Can Exhaust Shared gasLimitBucket and Deny Service to All Users

### Summary
The `gasLimitBucket` in `ThrottleManagerImpl` is a single application-scoped singleton with no per-user, per-IP, or per-source fairness partitioning. Any unauthenticated caller can send requests at the maximum allowed gas limit at the maximum allowed request rate, consuming the entire global gas token budget each second and causing all concurrent legitimate users to receive a `ThrottleException`. The global `rateLimitBucket` does not prevent this because it is also a shared singleton that the attacker simultaneously exhausts.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.gasLimitBucket()` (lines 34–45) creates a single Spring `@Bean` with a fixed capacity of `scaleGas(gasPerSecond)` tokens, shared across every incoming request with `SynchronizationStrategy.SYNCHRONIZED`:

```
@Bean(name = GAS_LIMIT_BUCKET)
Bucket gasLimitBucket() {
    long gasLimit = throttleProperties.getGasPerSecond();   // default 7_500_000_000 → scaled to 750_000 tokens
    ...
    return Bucket.builder()
            .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
            .addLimit(limit)
            .build();
}
```

`ThrottleManagerImpl.throttle()` (lines 37–48) deducts from this single bucket for every request:

```java
if (!rateLimitBucket.tryConsume(1)) {                                    // global 500 RPS
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // global gas
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Root cause:** Neither `gasLimitBucket` nor `rateLimitBucket` is keyed by any caller identity (IP, API key, `from` address, etc.). `ContractCallRequest` carries no IP field, `RequestFilter.FilterField` enumerates only `BLOCK, DATA, ESTIMATE, FROM, GAS, TO, VALUE` — no network-layer identity — and the `ThrottleManager` interface exposes no per-caller isolation primitive.

**Exploit flow:**

1. Attacker sends 500 concurrent HTTP POST requests per second to `POST /api/v1/contracts/call` with `"gas": 15000000` (the `maxGasLimit`).
2. Each request passes the RPS check (consuming 1 of 500 global RPS tokens) and then consumes `scaleGas(15_000_000) = 1_500` gas tokens from the shared bucket.
3. 500 requests × 1,500 tokens = 750,000 tokens = the entire per-second gas budget.
4. Every legitimate user's request fails at line 40–41 with HTTP 429 `"Gas per second rate limit exceeded."`.
5. The bucket refills every second; the attacker repeats indefinitely.

**Why existing checks fail:**

- `rateLimitBucket` (500 RPS global): The attacker consumes all 500 slots simultaneously, so legitimate users also fail the RPS check — this compounds the DoS rather than mitigating it.
- `maxGasLimit` (15M): Caps per-request gas but does not prevent a single caller from issuing the maximum number of max-gas requests.
- `RequestProperties` filters (`REJECT`/`THROTTLE`): Require operator pre-configuration of specific patterns (data, `from`, `to`, etc.) and cannot react dynamically to a novel attacker IP or address without manual intervention.
- `gasLimitRefundPercent` / `restore()`: Only returns gas after a completed call; the attacker's requests are accepted and gas is deducted before any refund occurs.

### Impact Explanation

A single unauthenticated attacker can render the `/api/v1/contracts/call` endpoint completely unavailable to all other users for as long as the attack is sustained. Every legitimate request receives HTTP 429. Because the mirror node's web3 endpoint is a public read API (no authentication required), this is a full availability denial with zero preconditions. The gas bucket refills every second, but the attacker can sustain the flood indefinitely with a single machine.

### Likelihood Explanation

The attack requires only the ability to send HTTP POST requests at ~500 req/s — trivially achievable from a single commodity server or a small botnet. No credentials, no special knowledge of the contract ecosystem, and no on-chain activity are needed. The endpoint is publicly reachable by design. The attack is repeatable, automatable, and leaves no persistent state that would allow the operator to block the attacker without external infrastructure (WAF, ingress-level IP rate limiting).

### Recommendation

1. **Per-IP gas rate limiting:** Replace the single `gasLimitBucket` with a `ConcurrentHashMap<String, Bucket>` keyed by client IP (extracted from `X-Forwarded-For` or `HttpServletRequest.getRemoteAddr()`), so each source IP has its own gas allowance.
2. **Per-IP RPS limiting:** Apply the same partitioning to `rateLimitBucket`.
3. **Add IP as a `FilterField`:** Extend `RequestFilter.FilterField` with an `IP` entry so operators can configure `REJECT` rules for specific abusive sources without code changes.
4. **Ingress-level protection:** Deploy a WAF or API gateway with per-IP rate limiting in front of the service as a defense-in-depth measure.
5. **Reduce `maxGasLimit` relative to `gasPerSecond`:** Ensure that `gasPerSecond / maxGasLimit > requestsPerSecond` so no single caller can exhaust the gas budget within the RPS limit.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or similar
# Sends 500 max-gas requests per second from a single machine

seq 500 | parallel -j500 \
  curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x","estimate":false}'
```

Expected result: the first batch of requests (up to the gas budget) returns 200; all subsequent requests within the same second return 429 with body `"Gas per second rate limit exceeded."`. Legitimate users issuing requests concurrently receive 429 for the duration of the attack.
