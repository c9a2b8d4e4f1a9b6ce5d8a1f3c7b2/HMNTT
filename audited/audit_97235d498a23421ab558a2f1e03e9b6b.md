### Title
gasLimitBucket Token Leak When Per-Filter THROTTLE Bucket Is Exhausted in `ThrottleManagerImpl`

### Summary
In `ThrottleManagerImpl.throttle()`, the global `gasLimitBucket` tokens are consumed before per-request filters are evaluated. When a `THROTTLE`-action filter's per-filter bucket is exhausted, `action()` throws a `ThrottleException` that propagates out of `throttle()` without ever restoring the already-consumed `gasLimitBucket` tokens. The controller's catch block only restores tokens for `InvalidParametersException`, leaving the global gas bucket permanently under-filled until natural refill, allowing an attacker to drain it faster than intended and cause legitimate callers to be rejected with "Gas per second rate limit exceeded."

### Finding Description

**Exact code path:**

`ContractController.call()` → `throttleManager.throttle(request)` → `ThrottleManagerImpl.throttle()` → `action()` → throws `ThrottleException`

**Step-by-step root cause:**

1. `ThrottleManagerImpl.throttle()` line 40 unconditionally consumes `gasLimitBucket` tokens before evaluating any per-request filters:
   ```java
   } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
       throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
   }
   ``` [1](#0-0) 

2. The loop then evaluates each configured filter and calls `action()` for matching ones (lines 44–48): [2](#0-1) 

3. Inside `action()`, the `THROTTLE` branch throws `ThrottleException` when the per-filter bucket is exhausted (lines 70–73):
   ```java
   case THROTTLE -> {
       if (!filter.getBucket().tryConsume(1)) {
           throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
       }
   }
   ``` [3](#0-2) 

4. This `ThrottleException` propagates back to `ContractController.call()`, which only catches `InvalidParametersException` and calls `restore()` there — `ThrottleException` is never caught, so `restore()` is never called:
   ```java
   } catch (InvalidParametersException e) {
       throttleManager.restore(request.getGas());
       throw e;
   }
   ``` [4](#0-3) 

5. `restore()` exists and is correct, but is simply never invoked on this code path: [5](#0-4) 

**Failed assumption:** The design assumes that a `ThrottleException` thrown from `action()` means no gas was consumed, but gas was already consumed at line 40 before `action()` is ever reached.

### Impact Explanation
Each request that matches a `THROTTLE` filter while that filter's per-filter bucket is exhausted silently drains `gasLimitBucket` by `scaleGas(request.getGas())` tokens without restoration. An attacker who can craft requests matching such a filter (e.g., targeting a known `to` address, `data` prefix, or `block` value that the operator has configured a THROTTLE rule for) can repeatedly submit high-gas requests to deplete the global gas bucket far faster than the legitimate refill rate. Once depleted, all subsequent callers — regardless of whether they match the filter — receive "Gas per second rate limit exceeded" until the bucket naturally refills (up to 1 second). Sustained at high request rate this becomes a continuous denial-of-service against the web3 API's gas rate limiter.

### Likelihood Explanation
The precondition is that the operator has at least one `THROTTLE`-action filter configured (a non-default but documented configuration option). An unprivileged external user needs no credentials — only knowledge of what request fields the filter matches (discoverable by probing or from public documentation/config leaks). The per-filter bucket capacity equals `rate` (0–100 tokens/second), so it exhausts trivially with a handful of requests. The attack is fully repeatable every second as the per-filter bucket refills, and each cycle leaks up to `maxGasLimit` tokens from the global bucket.

### Recommendation
Restore `gasLimitBucket` tokens whenever `throttle()` exits via any `ThrottleException` thrown after the gas tokens have been consumed. The cleanest fix is a try/finally inside `throttle()` itself:

```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    long scaled = throttleProperties.scaleGas(request.getGas());
    if (!gasLimitBucket.tryConsume(scaled)) {
        throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
    }
    try {
        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    } catch (ThrottleException e) {
        gasLimitBucket.addTokens(scaled); // restore on filter-level rejection
        throw e;
    }
}
```

Alternatively, add `ThrottleException` to the catch block in `ContractController.call()` and call `throttleManager.restore(request.getGas())` there, consistent with the existing `InvalidParametersException` handling.

### Proof of Concept

**Preconditions:**
- Operator has configured one `THROTTLE`-action filter, e.g. matching `block = latest`, with `rate = 1` (bucket capacity = 1 token/second).
- `gasPerSecond` is set to, say, `15_000_000`.

**Steps:**

1. Send Request A (gas=15_000_000, block=latest) → matches filter, filter bucket has 1 token → consumed → request proceeds normally. `gasLimitBucket` now has 0 tokens.
2. Wait for filter bucket to refill (1 second). `gasLimitBucket` also refills to 15_000_000.
3. Send Request B (gas=15_000_000, block=latest) → `gasLimitBucket` consumes 15_000_000 tokens (now 0). Filter matches, filter bucket has 1 token → consumed → request proceeds.
4. Send Request C (gas=15_000_000, block=latest) immediately → `gasLimitBucket` has 0 tokens → "Gas per second rate limit exceeded" (expected).
5. Wait 1 second. `gasLimitBucket` refills to 15_000_000. Filter bucket also refills to 1.
6. Send Request D (gas=15_000_000, block=latest) → `gasLimitBucket` consumes 15_000_000 (now 0). Filter matches, filter bucket has 1 token → consumed.
7. Send Request E (gas=15_000_000, block=latest) immediately → `gasLimitBucket` has 0 → rejected.
8. Send Request F (gas=15_000_000, block=latest) immediately → `gasLimitBucket` has 0 → rejected.

**Now trigger the leak:**

9. Wait 1 second. Both buckets refill.
10. Send Requests G1…G2 (gas=15_000_000, block=latest) in rapid succession:
    - G1: `gasLimitBucket` consumes 15_000_000 (→0). Filter bucket has 1 token → consumed. Proceeds normally.
    - G2: `gasLimitBucket` has 0 → "Gas per second rate limit exceeded." (normal throttle, no leak yet)
11. Wait 1 second. Filter bucket refills to 1. `gasLimitBucket` refills to 15_000_000.
12. Send G3 (gas=15_000_000, block=latest): `gasLimitBucket` consumes 15_000_000 (→0). Filter bucket has 1 → consumed. OK.
13. Send G4 (gas=15_000_000, block=latest): `gasLimitBucket` = 0 → rejected.

**Leak scenario (filter bucket already exhausted):**

14. Wait 1 second. `gasLimitBucket` refills to 15_000_000. Filter bucket refills to 1.
15. Send H1 (gas=15_000_000, block=latest): `gasLimitBucket` → 0. Filter bucket → 0. Proceeds.
16. Send H2 (gas=15_000_000, block=latest): `gasLimitBucket` = 0 → rejected immediately (no filter reached).
17. Wait 1 second. `gasLimitBucket` → 15_000_000. Filter bucket → 1.
18. Send H3 (gas=15_000_000, block=latest): `gasLimitBucket` → 0. Filter bucket → 0. Proceeds.
19. Send H4 (gas=1, block=latest): `gasLimitBucket` has ~14_999_999 tokens. Filter matches. Filter bucket = 0 → `ThrottleException` thrown. **`gasLimitBucket` tokens (1 scaled) are NOT restored.**
20. Repeat H4 rapidly with gas=15_000_000: each call consumes `scaleGas(15_000_000)` from `gasLimitBucket` and throws ThrottleException without restoring → bucket drains to 0 in a single second window, blocking all legitimate callers for the remainder of that second.

**Observed result:** Legitimate callers receive "Gas per second rate limit exceeded" even though they should have been allowed through, because the attacker drained the global gas bucket via unrestored tokens from filter-level rejections.
