### Title
Unauthenticated Alias-Based DB Query Flood via Uncached 404 Responses Exhausts Mirror Node Connection Pool

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` calls `EntityService.getEncodedId()` which, for any string matching the alias regex, unconditionally executes a live DB query against the `entity` table. Because 404 (not-found) responses are explicitly excluded from the response cache, every request with a syntactically valid but non-existent alias hits the database. With no application-level rate limiting on this endpoint, an unprivileged attacker can flood the DB connection pool, denying service to all mirror node consumers.

### Finding Description

**Code path:**

1. `rest/controllers/tokenAllowanceController.js:69` — `getAccountTokenAllowances` calls `EntityService.getEncodedId(req.params[...])` with no rate guard. [1](#0-0) 

2. `rest/service/entityService.js:125-126` — `getEncodedId` branches to `getAccountIdFromAlias` whenever `AccountAlias.isValid()` returns `true`. [2](#0-1) 

3. `rest/accountAlias.js:41-44` — `isValid` is a pure regex check (`/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`); it performs no DB lookup and accepts any base32-alphabet string. [3](#0-2) 

4. `rest/service/entityService.js:42-43` — `getAccountFromAlias` issues `SELECT id FROM entity WHERE … alias = $1` for every call. [4](#0-3) 

5. `rest/service/entityService.js:71-76` — When the entity is absent, `getAccountIdFromAlias` throws `NotFoundError` (HTTP 404). [5](#0-4) 

**Why the cache does not help:**

`responseCacheUpdateHandler` only stores responses when the status code is a success code: [6](#0-5) 

404 responses are never cached, so every unique (or even repeated) alias that is absent from the entity table causes a fresh DB round-trip.

**Why there is no application-level rate limiting:**

The middleware stack exported from `rest/middleware/index.js` contains: `authHandler`, `handleError`, `openApiValidator`, `requestLogger`, `requestQueryParser`, `responseCacheCheckHandler`, `responseCacheUpdateHandler`, `responseHandler`. There is no rate-limiting middleware. [7](#0-6) 

**Connection pool:**

The pool is bounded by `config.db.pool.maxConnections`. Saturating it blocks all other queries the mirror node needs to serve. [8](#0-7) 

### Impact Explanation

Saturating the DB connection pool denies service to the entire mirror node REST API: balance queries, transaction history, token queries, and all other endpoints stall waiting for a connection. Services (exchanges, wallets, dApps) that depend on the mirror node for account state or transaction data lose access. The mirror node itself does not participate in Hedera consensus, so on-chain transaction ordering is unaffected; however, any off-chain system relying on mirror node data for confirmation logic (e.g., checking finality before releasing funds) is effectively blinded.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public API spec. The alias format (`[A-Z2-7]+`) is trivially enumerable. A single machine sending concurrent HTTP requests with distinct alias strings (e.g., `AAAAAAA`, `AAAAAAB`, …) is sufficient. The attack is repeatable indefinitely and cheap to sustain because each request is small and the server does all the expensive work (DB query, connection acquisition).

### Recommendation

1. **Cache negative (404) alias lookups** in the response cache or in a short-lived in-process/Redis negative-result cache keyed on the decoded alias bytes.
2. **Add application-level rate limiting** (e.g., express-rate-limit or similar) scoped per IP on all `/accounts/:id/…` endpoints, independent of the optional Traefik infrastructure middleware.
3. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) around alias DB lookups so that a burst of alias requests cannot consume the entire connection pool.
4. Ensure the Traefik `rateLimit` middleware (present in rosetta values but not confirmed for the REST chart) is mandatorily enabled for the REST service in production.

### Proof of Concept

```bash
# Generate 10 000 syntactically valid but non-existent aliases and flood the endpoint
python3 -c "
import itertools, string, requests, concurrent.futures
chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'
aliases = (''.join(c) for c in itertools.product(chars, repeat=8))
def hit(alias):
    requests.get(f'http://<mirror-node>/api/v1/accounts/{alias}/allowances/tokens', timeout=5)
with concurrent.futures.ThreadPoolExecutor(max_workers=500) as ex:
    for alias in itertools.islice(aliases, 10000):
        ex.submit(hit, alias)
"
# Expected: DB connection pool exhausted; all mirror node API endpoints begin
# returning 503 / timing out for legitimate callers.
```
