### Title
Unauthenticated DB Resource Exhaustion via Unbounded Repeated Queries in `getAirdrops()`

### Summary
The `getAirdrops()` method in `TokenAirdropServiceImpl` executes a fresh, uncached database query on every invocation with no application-level rate limiting in the `rest-java` module. An unprivileged attacker can issue sustained single-threaded requests to `/api/v1/accounts/{id}/airdrops/outstanding` or `/api/v1/accounts/{id}/airdrops/pending` with continuously varying `token.id` bound values, causing each request to generate a unique SQL query that bypasses any ETag or response-level caching, directly increasing database CPU utilization.

### Finding Description

**Exact code path:**

`TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) calls `repository.findAll(request, id)` unconditionally with no caching annotation and no rate guard:

```java
// TokenAirdropServiceImpl.java:19-22
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}
```

`TokenAirdropRepositoryCustomImpl.findAll()` (lines 58–72) executes a raw jOOQ query to the `token_airdrop` table on every call — no `@Cacheable`, no result memoization:

```java
// TokenAirdropRepositoryCustomImpl.java:58-72
public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
    var condition = getBaseCondition(accountId, type.getBaseField())
            .and(getBoundConditions(bounds))
            .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));
    return dslContext.selectFrom(TOKEN_AIRDROP).where(condition)
            .orderBy(order).limit(request.getLimit()).fetchInto(TokenAirdrop.class);
}
```

The controller accepts up to two `token.id` range parameters (`@Size(max = 2)`), e.g. `token.id=gt:X`. By incrementing `X` on each request, every request produces a structurally distinct SQL predicate, defeating any query-plan cache reuse and ensuring a full index scan per request.

**Root cause:** The `rest-java` module has no application-level rate limiting. The only filters registered are `LoggingFilter` and `MetricsFilter`. `RestJavaConfiguration` registers only an ETag filter (`ShallowEtagHeaderFilter`), which only short-circuits identical responses — not varied-parameter requests. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) exists exclusively in the `web3` module and is not wired into `rest-java`.

### Impact Explanation
Each HTTP request forces a PostgreSQL index scan on `token_airdrop` filtered by `sender_account_id`/`receiver_account_id` + a varying `token_id` range. On a production node with millions of airdrop records, sustained single-threaded requests at even modest rates (e.g., 50–100 req/s) can measurably increase DB CPU. The Grafana alert rule `RestJavaHighCPU` triggers at 80% CPU, confirming the operational sensitivity. A 30%+ DB CPU increase above baseline is achievable without brute force — a single client loop suffices.

### Likelihood Explanation
The endpoint is fully public — no authentication, no API key, no IP-based throttle at the application layer. Any internet-accessible deployment is reachable. The attack requires only an HTTP client and knowledge of any valid account ID (which is publicly enumerable from the ledger). It is repeatable indefinitely and requires no special tooling.

### Recommendation
1. Add application-level rate limiting to `rest-java` analogous to the `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl`, enforcing a per-IP or global requests-per-second cap on `/api/v1/accounts/*/airdrops/*`.
2. Add a `@Cacheable` annotation (with a short TTL, e.g., 1–5 seconds) on `TokenAirdropRepositoryCustomImpl.findAll()` or `TokenAirdropServiceImpl.getAirdrops()` for identical request parameters to absorb repeated identical queries.
3. Ensure the GCP gateway `maxRatePerEndpoint` is enforced per-source-IP, not just as an aggregate endpoint cap.

### Proof of Concept
```bash
# Sustained single-threaded loop with varying token.id lower bound
# Replace ACCOUNT_ID with any valid sender account number
ACCOUNT_ID=1234567
i=1
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT_ID}/airdrops/outstanding?token.id=gt:${i}&limit=100" > /dev/null
  i=$((i + 1))
done
```

Each iteration sends a request with a unique `token.id=gt:N` predicate. Because `N` changes every request, the ETag filter never short-circuits, and `TokenAirdropRepositoryCustomImpl.findAll()` issues a fresh SQL query to the database on every iteration. Monitor `pg_stat_activity` or DB CPU metrics to observe the sustained increase.
