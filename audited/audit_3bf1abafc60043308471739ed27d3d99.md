### Title
Cache Key Pollution via Unnormalized `req.originalUrl` Enables Redis Eviction of Legitimate Entries

### Summary
The `cacheKeyGenerator` function hashes `req.originalUrl` directly without any normalization, meaning semantically identical requests with different parameter orderings or key casings produce distinct Redis cache keys. An unprivileged attacker can flood Redis with unique but valid-response cache entries, triggering `allkeys-lfu` eviction of frequently-accessed data and forcing legitimate users to experience cache misses and increased database load.

### Finding Description

**Exact code location**: `rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator`, lines 151ÔÇô153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

The comment at lines 147ÔÇô149 explicitly acknowledges the missing normalization:
> *"In the future, this will utilize Edwin's request normalizer (9113)."*

`req.originalUrl` is the raw URL string as received from the client. Express's `requestQueryParser` (registered at `server.js` line 60) normalizes `req.query` (lowercases keys, lowercases `order`/`result` values), but it **does not mutate `req.originalUrl`**. The `requestNormalizer.js` module (`normalizeRequestQueryParams`) exists but is **never imported or registered** in the middleware chain ÔÇö it appears only in test files.

**Exploit flow**:

1. Redis response cache is enabled (`config.cache.response.enabled && config.redis.enabled`, `server.js` line 54).
2. `responseCacheCheckHandler` is registered before all route handlers (`server.js` line 97). It calls `cacheKeyGenerator(req)` which hashes the raw `req.originalUrl`.
3. On a cache miss, the request proceeds to the route handler, returns a 200 response, and `responseCacheUpdateHandler` stores the result in Redis under that key (`responseCacheHandler.js` lines 95ÔÇô116).
4. An attacker sends the same semantically equivalent request with varied parameter orderings or key casings:
   - `/api/v1/accounts?limit=25&order=asc` Ôćĺ MD5 key A
   - `/api/v1/accounts?order=asc&limit=25` Ôćĺ MD5 key B (different)
   - `/api/v1/accounts?LIMIT=25&order=asc` Ôćĺ MD5 key C (different; `LIMIT` is lowercased in `req.query` but not in `req.originalUrl`)
5. Each variant is a cache miss, triggers a DB query, and writes a new Redis entry.
6. Redis is bounded at `maxMemory=250MB` with `maxMemoryPolicy=allkeys-lfu` (`cache.js` lines 48ÔÇô50, `docs/configuration.md` lines 594ÔÇô595). Flooding with low-frequency attacker keys causes LFU eviction of high-frequency legitimate entries.

**Why existing checks fail**:
- `maxRepeatedQueryParameters=100` (`docs/configuration.md` line 581) limits repetition of a single parameter but does not prevent ordering permutations.
- Parameter validation (`validateReq` in `utils.js` lines 455ÔÇô485) rejects invalid values but accepts all valid orderings ÔÇö these return 200 and are cached.
- `responseCacheUpdateHandler` only caches successful responses (`httpStatusCodes.isSuccess`, line 95), so 400 errors are not cached ÔÇö but valid permutations all return 200.

### Impact Explanation
Redis is capped at 250 MB with `allkeys-lfu` eviction. An attacker generating thousands of unique valid-URL variants (parameter ordering permutations across endpoints with multiple parameters like `/api/v1/transactions?timestamp=...&limit=...&order=...&result=...`) fills Redis with low-frequency entries. LFU eviction then removes the most-accessed legitimate entries. Legitimate users experience cache misses, forcing every request to hit the PostgreSQL database, increasing latency and DB load. This is a griefing/DoS-class issue with no economic damage to users but degraded service quality.

### Likelihood Explanation
No authentication or privilege is required. The attack is trivially scriptable: enumerate valid parameter orderings for any multi-parameter endpoint and send one request per permutation. For an endpoint with N sortable parameters, there are N! orderings. For `/api/v1/transactions` with ~6 parameters, that is 720 unique keys per logical query. Combined with the case-insensitivity gap in `req.originalUrl`, the attack surface is large. The attack is repeatable and persistent as long as the attacker continues sending requests.

### Recommendation
Normalize `req.originalUrl` (or a derived canonical URL) **before** computing the cache key in `cacheKeyGenerator`. Specifically:
1. Integrate `normalizeRequestQueryParams` from `rest/middleware/requestNormalizer.js` into the cache key generation path (the TODO at line 149 already tracks this as issue #9113).
2. Sort query parameters alphabetically (excluding order-sensitive params like `timestamp` and `block.number` which are already identified in `NON_SORTED_PARAMS`) before hashing.
3. Ensure parameter key casing is lowercased in the URL used for hashing, consistent with what `requestQueryParser` already does to `req.query`.

const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;

const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};
