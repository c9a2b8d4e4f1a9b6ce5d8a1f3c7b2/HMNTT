### Title
Unauthenticated `/tmp/alive` Heartbeat Allows Liveness-Probe Spoofing to Mask Compromised Pinger State

### Summary
The pinger's liveness probe relies exclusively on the modification time of `/tmp/alive`, a world-accessible file in a tmpfs/emptyDir mount, with no process-level binding, no cryptographic integrity check, and no content validation. Any entity that can write to the container's `/tmp` (e.g., a co-located sidecar sharing an emptyDir volume, or any process injected into the pod) can touch `/tmp/alive` at will, keeping the Kubernetes liveness probe green indefinitely — even while the pinger process is dead or actively making unauthorized `Execute(client)` calls in `submitWithRetry()`.

### Finding Description

**Heartbeat writer** — `pinger/main.go` lines 28–39:
```go
_ = os.WriteFile("/tmp/alive", []byte(time.Now().Format(time.RFC3339Nano)), 0644)
```
The file is written with mode `0644` into `/tmp`, which in every standard Kubernetes emptyDir or tmpfs mount is mode `1777` (sticky, world-writable). The pinger process (UID 1000) owns the file after first write, but because `/tmp` is sticky-world-writable, any other process running as any UID can **create a new file at the same path** (unlink + create) or, if the file already exists and is owned by UID 1000, can `touch`/overwrite it from a process running as the same UID.

**Liveness checker** — `pinger/cmd/healthcheck/main.go` lines 27–38:
```go
fi, err := os.Stat("/tmp/alive")
...
age := time.Since(fi.ModTime())
if age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)
```
The check is purely `ModTime < 2 minutes`. There is:
- No verification that the pinger PID is alive (no `/proc/<pid>/status` check, no pidfile, no socket).
- No cryptographic MAC or nonce in the file content.
- No check that the content is a valid, monotonically increasing timestamp.
- No check that the file was written by the expected process.

**Exploit path:**
1. Attacker gains write access to the pod's `/tmp` (e.g., via a shared `emptyDir` volume mounted at `/tmp` in a sidecar, or via a second container in the same pod that was compromised).
2. Attacker kills or hijacks the pinger process, or simply lets it run while injecting their own `Execute(client)` calls using the operator key already loaded in memory/env.
3. Attacker runs a trivial loop in the sidecar/injected process:
   ```sh
   while true; do date -Ins > /tmp/alive; sleep 30; done
   ```
4. Kubernetes calls `/healthcheck live` every N seconds; it always exits 0.
5. The container is never restarted; no alert fires; unauthorized transfers via `submitWithRetry()` → `cryptoTransfer.Execute(client)` continue indefinitely.

**Why existing checks fail:**
The only defense is the 2-minute staleness window. There is no defense against an attacker who can write the file themselves. The `FROM scratch` + `USER 1000:1000` Dockerfile hardens the attack surface but does not prevent a co-scheduled sidecar (running as UID 1000 or root) from sharing the same emptyDir `/tmp`.

### Impact Explanation
The operator private key (`HIERO_MIRROR_PINGER_OPERATOR_KEY`) is loaded at startup and held in the `hiero.Client` for the lifetime of the process. `submitWithRetry()` calls `cryptoTransfer.Execute(client)` which signs and submits real on-ledger HBAR transfers using that key. If the probe is spoofed, operators lose their only automated signal (container restart / CrashLoopBackOff) that the pinger is in an abnormal state. Unauthorized transfers drain the operator account without triggering any Kubernetes-level alert.

### Likelihood Explanation
The precondition — write access to the container's `/tmp` — is achievable without cluster-admin privileges in several realistic scenarios:
- A Kubernetes `emptyDir` volume mounted at `/tmp` and shared with a second container in the same pod (common pattern for log shipping sidecars).
- A compromised init-container that pre-seeds `/tmp/alive` before the pinger starts.
- Any future misconfiguration that mounts a writable hostPath or PVC at `/tmp`.

The `scratch`-based image prevents direct shell injection, but does not prevent shared-volume abuse. The attack is repeatable and requires no special kernel privileges.

### Recommendation
1. **Bind the heartbeat to the process**: Write the pinger's own PID into `/tmp/alive` and have the healthcheck verify that `/proc/<pid>/status` exists and the process is in a live state.
2. **Use a Unix domain socket or named pipe** owned exclusively by the pinger (mode `0600`) instead of a world-accessible file, so no other process can write to it.
3. **Add a HMAC/nonce**: Include a secret (generated at startup, stored only in memory) in the file content; the healthcheck binary receives the same secret via environment and validates it before accepting the heartbeat.
4. **Restrict `/tmp` permissions**: Mount `/tmp` as a dedicated emptyDir with `defaultMode: 0700` and ensure no other container in the pod shares it.

### Proof of Concept
```bash
# 1. Deploy pinger pod with a sidecar sharing /tmp via emptyDir
# 2. In the sidecar container (unprivileged, UID 1000):
while true; do
  printf '%s' "$(date --iso-8601=ns)" > /tmp/alive
  sleep 30
done &

# 3. Kill the pinger process inside its container (or let it crash):
kill 1   # inside pinger container

# 4. Observe: Kubernetes liveness probe keeps returning exit 0
kubectl exec <pod> -c healthcheck -- /healthcheck live
# => exits 0 (alive), container never restarted

# 5. Meanwhile, unauthorized Execute(client) calls continue or
#    the attacker replays signed transactions using the stolen operator key.
```
