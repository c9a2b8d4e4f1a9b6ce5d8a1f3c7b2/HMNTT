### Title
DB Connection Pool Exhaustion via Unauthenticated High-Frequency Requests to `getBlocks()`

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` issues a live database query on every invocation with no request-level deduplication or caching. The Redis-based response cache that could absorb repeated identical requests is **disabled by default** (`config.cache.response.enabled = false`), and the REST API has no rate-limiting middleware. With a default pool ceiling of only 10 connections, a single unprivileged attacker sending concurrent identical requests can saturate the pool and deny service to all other API consumers.

### Finding Description

**Code path:**

`rest/controllers/blockController.js` lines 101â€“112 â€” `getBlocks()` calls `RecordFileService.getBlocks(formattedFilters)` unconditionally on every request. [1](#0-0) 

`rest/service/recordFileService.js` lines 149â€“161 â€” `getBlocks()` builds and executes a raw SQL query via `super.getRows()`, which calls `pool.queryQuietly()` and acquires a connection from the shared pg pool. [2](#0-1) 

**Root cause â€” response cache disabled by default:**

`rest/server.js` line 54 gates the entire response-cache middleware on `config.cache.response.enabled && config.redis.enabled`. [3](#0-2) 

`docs/configuration.md` line 549 documents the default value of `hiero.mirror.rest.cache.response.enabled` as **`false`**. [4](#0-3) 

When the cache is disabled, `responseCacheCheckHandler` is never registered, so every request â€” including thousands of identical `GET /api/v1/blocks?limit=100` requests â€” falls through to the controller and hits the database. [5](#0-4) 

**Root cause â€” no rate limiting in the REST API:**

`rest/server.js` registers no rate-limiting middleware for any route. The throttle/bucket4j infrastructure found in the codebase belongs exclusively to the `web3` Java module and is not applied to the Node.js REST API. [6](#0-5) 

**Root cause â€” small default connection pool:**

`rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`. The documented default is **10 connections**. [7](#0-6) [8](#0-7) 

`connectionTimeoutMillis` defaults to 20 000 ms, meaning requests queue for up to 20 seconds before failing â€” long enough to keep the pool saturated under sustained load. [9](#0-8) 

### Impact Explanation

With 10 concurrent attacker requests in-flight, the entire pg pool is occupied. Every subsequent legitimate request â€” to `/api/v1/blocks`, `/api/v1/transactions`, `/api/v1/accounts`, or any other endpoint sharing the same pool â€” queues waiting for a free connection. Under sustained attack the queue grows unboundedly (Node.js event loop is non-blocking; new requests keep arriving), and all legitimate users receive connection-timeout errors after 20 seconds. This constitutes a complete, non-volumetric application-layer DoS of the mirror-node REST API with no network-level amplification required.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public API path. The attacker needs only a single machine capable of maintaining ~10â€“20 concurrent HTTP connections. The endpoint is publicly documented and returns a predictable, cacheable response, making it an ideal amplification target. The default-off cache means most production deployments that have not explicitly enabled Redis caching are vulnerable out of the box.

### Recommendation

1. **Enable the response cache by default** or document it as a required production setting. The infrastructure already exists (`responseCacheCheckHandler` / `responseCacheUpdateHandler`); the default for `hiero.mirror.rest.cache.response.enabled` should be `true` when Redis is available.
2. **Add rate-limiting middleware** to the Express application (e.g., `express-rate-limit`) applied globally before route handlers, limiting requests per IP per second.
3. **Increase the default pool size** or document a minimum recommended value for production deployments.
4. **Add a stampede/coalescing guard**: for cache-miss scenarios, deduplicate in-flight identical queries so that N concurrent identical requests result in only one DB query.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or similar
# Saturate the 10-connection pool with 20 concurrent identical requests

for i in $(seq 1 20); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks?limit=100" &
done
wait

# Legitimate request issued while pool is saturated:
time curl -v "http://<mirror-node-host>:5551/api/v1/blocks?limit=1"
# Expected: hangs for ~20 seconds then returns a 500/503 or connection-timeout error
# Confirms: DB pool exhausted, all API consumers denied service
```

Preconditions: `hiero.mirror.rest.cache.response.enabled = false` (the default), no external rate limiter (e.g., no Traefik/nginx in front of the service). No credentials required.
