### Title
Global Rate-Limit Bucket Monopolization via Single-Source Token Exhaustion (DoS)

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` is a single application-wide singleton `Bucket` with no per-IP or per-identity partitioning. Any unprivileged caller can continuously consume tokens at exactly the refill rate, permanently starving all other users. The `refillGreedy` strategy makes this trivially sustainable because tokens are replenished continuously, allowing an attacker to match the refill cadence without any coordination overhead.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `rateLimitBucket()`, lines 25–32:

```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default: 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

This `Bucket` is a Spring singleton injected into `ThrottleManagerImpl` and consumed via `rateLimitBucket.tryConsume(1)` for every incoming request (line 38 of `ThrottleManagerImpl.java`). There is no per-IP, per-session, or per-identity bucket — the entire `requestsPerSecond` budget (default 500 req/s) is shared globally.

**Root cause:** The failed assumption is that the global rate limit provides fair access across multiple callers. It does not. A single source can consume the entire token budget.

**Exploit flow:**

1. `refillGreedy(rateLimit, Duration.ofSeconds(1))` refills tokens continuously at a rate of `rateLimit / 1s` (i.e., 1 token per 2ms at the default of 500 req/s).
2. Attacker drains the full bucket at t=0 with `rateLimit` concurrent/rapid requests.
3. Attacker then issues one request every `1000ms / rateLimit` (≈2ms at default) to consume each token as it is minted.
4. Because `refillGreedy` distributes tokens uniformly over the interval, the attacker never needs to wait for a batch refill — they simply pace requests at the refill rate.
5. All other users receive HTTP 429 (`ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED)`) indefinitely.

**Why existing checks fail:**

- `RequestFilter` (lines 39–58 of `RequestFilter.java`) filters on request payload fields (`BLOCK`, `DATA`, `FROM`, `GAS`, `TO`, `VALUE`) — not on source IP. It cannot identify or isolate the attacking source.
- `RequestProperties.ActionType.THROTTLE` applies a secondary per-filter bucket, but this is opt-in configuration and still shares the same global primary `rateLimitBucket` gate.
- There is no IP-based allowlist, denylist, or per-source sub-bucket anywhere in the `web3/config/` or `web3/throttle/` packages.

### Impact Explanation
All legitimate users of the `/api/v1/contracts/call` and opcode endpoints are denied service for the duration of the attack. The mirror node's EVM simulation capability becomes completely unavailable. Since `eth_call` is used by wallets, dApps, and tooling to simulate transactions and read contract state before signing, sustained unavailability can cause users to make uninformed on-chain decisions (e.g., submitting transactions blind), which can result in wasted gas or incorrect contract interactions. The severity is **High** for availability; the "direct loss of funds" framing applies indirectly through degraded pre-transaction simulation.

### Likelihood Explanation
Preconditions: none. No authentication, no special privilege, no API key required. The attacker needs only a single HTTP client capable of sustaining ~500 req/s (trivially achievable from a single machine or a small botnet). The attack is fully repeatable, stateless, and requires no knowledge of the application beyond the public endpoint. The default `requestsPerSecond = 500` is low enough that a single commodity server can saturate it indefinitely.

### Recommendation
Replace the single global bucket with per-source-IP buckets using bucket4j's `ProxyManager` pattern (e.g., backed by Caffeine or Redis):

```java
// Per-IP bucket: each IP gets its own rateLimit tokens/sec
LoadingCache<String, Bucket> perIpBuckets = Caffeine.newBuilder()
    .expireAfterAccess(1, TimeUnit.MINUTES)
    .build(ip -> Bucket.builder().addLimit(limit).build());
```

Enforce the per-IP bucket in a servlet filter before `ThrottleManagerImpl`, keyed on `HttpServletRequest.getRemoteAddr()` (with X-Forwarded-For handling behind a trusted proxy). The global bucket can remain as a secondary server-protection ceiling.

### Proof of Concept

```bash
# Drain the global bucket and hold it (default: 500 req/s)
# Step 1: burst-drain at t=0
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000167","gas":50000}' &
done
wait

# Step 2: sustain at refill rate (~1 req per 2ms = 500/s)
while true; do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000167","gas":50000}'
  sleep 0.002
done
```
