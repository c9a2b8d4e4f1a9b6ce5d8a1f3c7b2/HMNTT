### Title
Connection Pool Exhaustion DoS via Concurrent GET /api/v1/schedules Requests

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues three `pool.queryQuietly()` calls per request — one sequential and two concurrent via `Promise.all()` — against a shared `node-postgres` pool capped at 10 connections by default. Because no rate-limiting middleware exists on the REST API, an unauthenticated attacker can send as few as 5 concurrent requests to hold all 10 pool connections during the `Promise.all()` phase, causing every subsequent request to block for up to `connectionTimeoutMillis` (20 s) before failing, making the service unavailable for the duration of the attack.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `getSchedules()`:
- Line 241 — first sequential query acquires and holds one pool connection for the full query duration.
- Lines 261–264 — `Promise.all([pool.queryQuietly(entityQuery, …), pool.queryQuietly(signatureQuery, …)])` acquires **two** pool connections simultaneously per in-flight request.

```
// rest/schedules.js line 241
const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

// rest/schedules.js lines 261-264
const [{rows: entities}, {rows: signatures}] = await Promise.all([
  pool.queryQuietly(entityQuery, entityIds),
  pool.queryQuietly(signatureQuery, entityIds),
]);
```

`rest/utils.js`, `queryQuietly` (lines 1518–1520): when `preQueryHint` is absent (always the case here), the implementation calls `this.query(query, params)` — the `node-postgres` `Pool.query()` method, which internally calls `pool.connect()`, holds the client for the full query round-trip, then releases it. The connection is therefore occupied for the entire query execution time.

`rest/dbpool.js` (lines 13–14):
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
max: config.db.pool.maxConnections,                         // default 10
```

**Root cause:** The pool has a hard ceiling of 10 connections. During the `Promise.all()` phase each concurrent request holds 2 connections. Five simultaneous requests in that phase consume all 10 connections. Any further request that calls `pool.query()` is queued by `node-postgres` and blocks until a connection is freed or `connectionTimeoutMillis` (20 s) elapses, at which point it throws a timeout error. The attacker simply re-sends the wave before the previous wave drains.

**Why existing checks fail:**
- `statement_timeout: 20 000 ms` — limits individual query duration but does not prevent pool exhaustion; it only caps how long each connection is held per wave.
- `connectionTimeoutMillis: 20 000 ms` — causes queued requests to error after 20 s, but this is the *symptom* of unavailability, not a mitigation.
- No rate-limiting, concurrency cap, or request-queue depth limit exists in the REST API middleware stack (`rest/server.js` lines 68–144 show only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — no throttle).

### Impact Explanation
All `GET /api/v1/schedules` requests (and any other endpoint that needs a pool connection) will time out with a `DbError` for the duration of the attack. Because Node.js is single-process and the global `pool` object is shared across all routes, pool exhaustion on this endpoint degrades the entire REST API. The `connectionTimeoutMillis` of 20 s means each attack wave produces a 20-second window of full unavailability, and the attacker can sustain this indefinitely with a trivial request rate.

### Likelihood Explanation
The endpoint requires no authentication (`GET /api/v1/schedules` is public). The attacker needs only an HTTP client capable of sending ~10 concurrent requests. No special knowledge, credentials, or network position is required. The attack is trivially scriptable (`curl`, `ab`, `wrk`, etc.) and repeatable at negligible cost.

### Recommendation
1. **Add a per-IP or global concurrency/rate limit** in the REST API middleware (e.g., `express-rate-limit` or a token-bucket middleware) before the route handlers, specifically capping concurrent in-flight requests to a fraction of `maxConnections`.
2. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer in transaction mode) in front of PostgreSQL so the application pool is not the bottleneck.
3. **Reduce queries per request** — merge the three queries in `getSchedules()` into a single SQL query (e.g., using CTEs or JOINs) so each request holds at most one connection at a time.
4. **Set a hard HTTP-level request timeout** upstream (e.g., in the load balancer/ingress) shorter than `connectionTimeoutMillis` to shed load before the pool queue grows unboundedly.

### Proof of Concept

```bash
# Send 10 concurrent requests (2× the pool-exhaustion threshold)
# No authentication required
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/schedules" &
done
wait

# Immediately send a probe request — it will block for ~20 s then return a 500/503
time curl -v "http://<mirror-node-host>:5551/api/v1/schedules"
# Expected: ~20 000 ms elapsed, HTTP 500 with DbError (connection timeout)
```
