### Title
Silent Order Override via Mixed-Case Duplicate `order` Query Parameters

### Summary
An unprivileged user can send duplicate `order` parameters with different letter cases (e.g., `?order=asc&Order=asc`) causing `requestQueryParser()` to merge them into an array. `validateReq()` accepts the array because each element is individually valid, but `parseLimitAndOrderParams()` silently ignores the array due to strict string equality checks and falls back to the default descending order, returning records in the opposite of the requested direction with incorrect pagination.

### Finding Description
**Code path:**

`requestQueryParser()` in `rest/middleware/requestHandler.js` (lines 38–69) uses `qs.parse()` (case-sensitive) then lowercases all keys and merges collisions via `merge()`: [4](#0-3) 

For input `?order=asc&Order=asc`, `qs.parse` sees two distinct keys (`order`, `Order`). After lowercasing, both map to `order`, and `merge()` produces `{order: ['asc', 'asc']}`.

`validateReq()` in `rest/utils.js` (lines 455–486) handles the array case by iterating each element: [2](#0-1) 

Each `'asc'` element passes `filterValidityChecks` at line 336–339: [5](#0-4) 

No error is thrown. Then `parseLimitAndOrderParams()` in `rest/utils.js` (lines 662–674) reads the value and applies strict equality: [3](#0-2) 

`['asc', 'asc'] === 'asc'` is `false`; `['asc', 'asc'] === 'desc'` is `false`. The condition is never entered, and `order` stays as `defaultOrder` (typically `'desc'`). The user requested ascending but receives descending.

**Root cause:** `parseLimitAndOrderParams` assumes `req.query['order']` is always a string, but `requestQueryParser` can produce an array when the same key appears with different cases. No guard converts or rejects the array before the strict equality check.

### Impact Explanation
Any API endpoint that uses `parseLimitAndOrderParams` (e.g., `/api/v1/accounts`, `/api/v1/transactions`) will silently return records in descending order when the user explicitly requested ascending. Pagination `next` links are also generated using the wrong order value: [6](#0-5) 

This means a client iterating pages with `?order=asc&Order=asc` will receive pages in descending order with pagination links that perpetuate the wrong direction — constituting incorrect record export behavior for mirror node consumers.

### Likelihood Explanation
No authentication or special privilege is required. The exploit requires only crafting a URL with two `order` parameters differing by case — trivially achievable by any HTTP client. It is fully repeatable and deterministic. The existing test suite even documents the array-producing behavior (line 47–50 of `requestHandler.test.js`) but no corresponding test validates that `parseLimitAndOrderParams` rejects or correctly handles this array input. [1](#0-0) 

### Recommendation
In `parseLimitAndOrderParams()`, normalize the `order` value before comparison — if it is an array, take the last element (consistent with how `getLimitParamValue` handles the `limit` array at line 547): [7](#0-6) 

```js
const rawValue = req.query[constants.filterKeys.ORDER];
const value = Array.isArray(rawValue) ? rawValue[rawValue.length - 1] : rawValue;
if (value === constants.orderFilterValues.ASC || value === constants.orderFilterValues.DESC) {
  order = value;
}
```

Alternatively, add a check in `validateReq` that rejects an array value for the `order` key specifically (since only one order direction is meaningful).

### Proof of Concept
```
GET /api/v1/accounts?order=asc&Order=asc HTTP/1.1
Host: <mirror-node-host>
```

**Step-by-step:**
1. Send the above request (no authentication needed).
2. `requestQueryParser` produces `req.query = {order: ['asc', 'asc']}`.
3. `validateReq` iterates the array; each `'asc'` passes — no 400 error.
4. `parseLimitAndOrderParams` evaluates `['asc','asc'] === 'asc'` → `false`; falls back to default `'desc'`.
5. SQL query executes with `ORDER BY id DESC`.
6. Response contains accounts in descending order despite the explicit `order=asc` request.
7. The `links.next` pagination URL also encodes `desc`, causing all subsequent pages to be in the wrong order.

```javascript
const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};
```
