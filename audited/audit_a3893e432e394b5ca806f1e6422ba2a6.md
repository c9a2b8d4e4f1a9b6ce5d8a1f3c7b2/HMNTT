### Title
O(n²) Sibling Chain Traversal in `parsePossiblyHookContractRelatedParent()` Enables Mirror Node CPU Exhaustion

### Summary
The `parsePossiblyHookContractRelatedParent()` method in `RecordItem.java` contains an unbounded while loop that traverses the `previous` linked chain for every child transaction being built. When n sibling child transactions all carry a `ContractFunctionResult` with `contractNum != 365`, the k-th sibling walks k−1 nodes, producing O(n²) total work across the sibling group. An unprivileged user can trigger this by submitting a single `ContractCall` that spawns many internal child calls to contracts whose IDs are never 365.

### Finding Description

**Exact location**: `common/src/main/java/org/hiero/mirror/common/domain/transaction/RecordItem.java`, inner class `RecordItemBuilder`, method `parsePossiblyHookContractRelatedParent()`, lines 350–375.

**How `previous` is wired**: Every record reader sets `previous` to the immediately preceding `RecordItem` in stream order. For example, `AbstractRecordFileItemReader.onRecordStreamItem()` (line 171) does `.previous(items.isEmpty() ? null : items.getLast())`, and `AbstractPreV5RecordFileReader.readBody()` (line 110) does `.previous(lastRecordItem)`. For a parent P followed by siblings C1…Cn the chain is: Cn.previous = C(n−1), …, C2.previous = C1, C1.previous = P.

**How `parent` is resolved**: `parseParent()` (lines 330–348) resolves the parent for any sibling Ck (k ≥ 2) via `previous.parent` (which equals P), so `parent = P` for every sibling.

**The loop** (lines 359–371):
```java
while (candidateRecord != null && candidateRecord != parent) {
    var candidateTxnRecordContractResult = candidateRecord.getContractResult();
    if (candidateTxnRecordContractResult == null) { break; }
    if (candidateTxnRecordContractResult.getContractID().getContractNum() == HOOK_CONTRACT_NUM) {
        return candidateRecord;
    }
    candidateRecord = candidateRecord.previous;
}
```

For sibling Ck the loop starts at C(k−1) and walks backward until it hits P (== `parent`). If every sibling has a non-null `ContractFunctionResult` with `contractNum != 365`, no early exit fires and the loop runs exactly k−1 iterations. Summing over all n siblings: 0+1+…+(n−1) = n(n−1)/2 = **O(n²)**.

**Entry conditions an attacker fully controls**:
1. `hasParentConsensusTimestamp()` — set by the consensus node for every internal child call; attacker triggers this by submitting a `ContractCall`.
2. `currentTxnRecordContractResult != null` — present on every child contract call record.
3. `contractNum != HOOK_CONTRACT_NUM (365)` — attacker deploys target contracts with any entity number other than 365; this is the default for all user-deployed contracts.
4. `previous != null` — always true for non-first items in a file.

**Why existing checks are insufficient**: The only early-exit conditions inside the loop are (a) `candidateRecord == parent` (terminates correctly but only after full traversal), (b) `candidateTxnRecordContractResult == null` (never fires if all siblings are contract calls), and (c) `contractNum == 365` (never fires if attacker avoids 365). There is no depth cap, no visited-set, and no maximum sibling count enforced at this layer.

### Impact Explanation
With Hedera's maximum gas per transaction (~15,000,000 gas) and a minimal CALL costing ~700–1,000 gas, an attacker can produce hundreds to low thousands of child transactions per parent. At n = 1,000 siblings the loop executes ~500,000 iterations per parent transaction. Submitting a sustained stream of such transactions (each costing the attacker gas but imposing quadratic work on the mirror node's importer thread) can drive the importer's CPU well above the 30% threshold. The importer runs `parse()` under a single `synchronized` lock, so CPU saturation directly stalls all record file ingestion.

### Likelihood Explanation
No special privileges are required. Any Hedera account holder can deploy a fan-out contract and submit `ContractCall` transactions. The attack is repeatable and can be automated. The cost to the attacker scales linearly with n while the cost to the mirror node scales quadratically, making the attack economically asymmetric. The attack is also silent — it produces no errors, only elevated CPU.

### Recommendation
Replace the linear backward walk with an O(1) lookup. The simplest fix is to record the first hook-contract sibling on the parent `RecordItem` itself (e.g., a `firstHookChild` field set when C1 is built), so subsequent siblings can read it in constant time. Alternatively, cap the loop at a small constant (e.g., 10 iterations) since a legitimate hook child will always be the immediately preceding sibling or very close to it. A depth guard such as:
```java
int maxLookback = 16;
while (candidateRecord != null && candidateRecord != parent && maxLookback-- > 0) { … }
```
would bound the worst case to O(n) total across all siblings.

### Proof of Concept
1. Deploy n contracts C_1…C_n on Hedera testnet, each with a trivial body (e.g., `STOP`). All will receive entity numbers ≠ 365.
2. Deploy a fan-out contract F whose constructor or function calls C_1 through C_n in sequence.
3. Submit a `ContractCall` to F. The consensus node emits one parent record and n child records, each with a `ContractFunctionResult` whose `contractID.contractNum` is the entity number of C_i (never 365).
4. The mirror node importer reads the record file. When building child record Ck, `parsePossiblyHookContractRelatedParent()` walks k−1 nodes backward. Total iterations: n(n−1)/2.
5. Repeat step 3 in a tight loop. Monitor the importer's CPU via JMX or `/actuator/metrics`. With n ≈ 500–1000 and a sustained submission rate, CPU rises by >30% compared to the baseline 24-hour average. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        private RecordItem parsePossiblyHookContractRelatedParent() {
            final var currentTxnRecordContractResult = getContractResult();

            if (transactionRecord.hasParentConsensusTimestamp()
                    && currentTxnRecordContractResult != null
                    && currentTxnRecordContractResult.getContractID().getContractNum() != HOOK_CONTRACT_NUM
                    && previous != null) {
                var candidateRecord = previous;

                while (candidateRecord != null && candidateRecord != parent) {
                    var candidateTxnRecordContractResult = candidateRecord.getContractResult();
                    if (candidateTxnRecordContractResult == null) {
                        break;
                    }

                    if (candidateTxnRecordContractResult.getContractID().getContractNum() == HOOK_CONTRACT_NUM) {
                        // we found the first hook child item, which has a ContractResult and is executed via a hook
                        return candidateRecord;
                    }

                    candidateRecord = candidateRecord.previous;
                }
            }

            return null;
        }
