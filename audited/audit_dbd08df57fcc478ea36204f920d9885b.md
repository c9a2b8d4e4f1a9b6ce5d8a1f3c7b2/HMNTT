### Title
Unauthenticated Connection Pool Exhaustion via Concurrent `findAll()` Requests on NFT Allowance Endpoint

### Summary
The `/api/v1/accounts/{id}/allowances/nfts` endpoint in the `rest-java` module is publicly accessible with no rate limiting and no per-query timeout. Each concurrent request holds a HikariCP database connection for the full duration of the JOOQ query execution. An unprivileged attacker sending a flood of concurrent requests can exhaust the connection pool, causing all subsequent legitimate requests to fail with connection timeout errors and effectively partitioning the service from the database.

### Finding Description

**Exact code path:**

`AllowancesController.java` (lines 57–83) exposes a public, unauthenticated GET endpoint: [1](#0-0) 

This calls `NftAllowanceService.getNftAllowances()`, which delegates to `NftAllowanceRepositoryCustomImpl.findAll()`: [2](#0-1) 

The JOOQ `dslContext.selectFrom(...).fetchInto(...)` call is synchronous and blocking. It holds a HikariCP connection from acquisition until the result set is fully fetched. No statement/query timeout is set on the JOOQ `DSLContext` in the `rest-java` module.

**Root cause — three compounding failures:**

1. **No rate limiting in `rest-java`**: The `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j-based) exist exclusively in the `web3` module. [3](#0-2) 
No equivalent exists anywhere under `rest-java/src/main/java/`. The `WebMvcConfiguration` for `rest-java` registers only argument resolvers and formatters — no throttle interceptor. [4](#0-3) 

2. **No query timeout**: No `spring.datasource.hikari` properties file exists under `rest-java/src/main/resources/` (only `banner.txt` is present). No `queryTimeout` or `statement_timeout` is configured for the JOOQ `DSLContext`. HikariCP defaults apply: `maximumPoolSize=10`, `connectionTimeout=30s`.

3. **`@Max(MAX_LIMIT)` is insufficient**: The limit cap (100 rows) only bounds result set size, not query execution time or connection hold duration. [5](#0-4) 

**Why existing checks fail:**

- `@Positive @Max(MAX_LIMIT)` on `limit` prevents large result sets but does not prevent many concurrent small queries.
- `@Size(max = 2)` on filter arrays limits filter complexity per request but not request concurrency.
- HikariCP's `connectionTimeout` (30 s default) means waiting threads eventually fail, but during those 30 seconds the pool remains fully occupied and the service is unavailable.

### Impact Explanation

With HikariCP's default pool of 10 connections, an attacker maintaining ≥10 concurrent in-flight requests saturates the pool. All subsequent requests — including legitimate ones — block waiting for a connection. After 30 seconds they receive a `SQLTimeoutException` / HTTP 500. The service is effectively partitioned from the database for the duration of the attack. Because the attack is stateless (no session, no auth token), it can be sustained indefinitely from a single machine or a small botnet. The `nft_allowance` table scan with `approved_for_all = true` and an `ORDER BY` clause can be slow on large datasets, increasing per-connection hold time and worsening the attack.

### Likelihood Explanation

- **Precondition**: None. The endpoint is public and unauthenticated.
- **Skill required**: Basic HTTP tooling (`ab`, `wrk`, `curl` in parallel). No exploit code needed.
- **Repeatability**: Continuous — the attacker simply keeps the flood running.
- **Detection difficulty**: Looks like organic traffic spikes; no anomalous payload.
- **Real-world feasibility**: High. Any motivated actor can sustain this from a single host with a modest number of concurrent connections.

### Recommendation

1. **Add rate limiting to `rest-java`**: Port the bucket4j `ThrottleConfiguration` pattern from `web3` into `rest-java`, or add a Spring MVC `HandlerInterceptor` / Servlet `Filter` that enforces per-IP or global request-per-second limits on `/api/v1/accounts/*/allowances/nfts`.

2. **Set a query timeout**: Configure `spring.datasource.hikari.connection-timeout` and add a JOOQ `ExecuteListener` or configure `spring.jooq.sql-dialect` with a statement timeout (e.g., PostgreSQL `SET statement_timeout = 5000`). Alternatively, configure `spring.datasource.hikari.maximum-pool-size` and `connection-timeout` explicitly.

3. **Increase pool size with a cap**: Tune `maximumPoolSize` to match expected concurrency, but pair it with rate limiting so the pool cannot be monopolized.

4. **Add async/reactive handling or a semaphore**: Limit the number of in-flight DB queries for this endpoint using a `Semaphore` or reactive back-pressure.

### Proof of Concept

```bash
# Requires: a valid accountId that exists in the mirror node (e.g., 0.0.1000)
# Tool: GNU parallel or wrk

# Step 1: Confirm endpoint is unauthenticated
curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1000/allowances/nfts" | head -c 200

# Step 2: Flood with concurrent requests (saturate the 10-connection pool)
seq 1 500 | parallel -j 200 \
  'curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node-host>/api/v1/accounts/0.0.{}.allowances/nfts?limit=100"'

# Expected result:
# - First ~10 requests: 200 OK (connections acquired)
# - Remaining requests during attack: 500 Internal Server Error
#   (HikariCP: Connection is not available, request timed out after 30000ms)
# - Legitimate users receive 500 for the duration of the attack
```

    public Collection<NftAllowance> findAll(NftAllowanceRequest request, EntityId accountId) {
        boolean byOwner = request.isOwner();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, byOwner).and(getBoundConditions(bounds));
        return dslContext
                .selectFrom(NFT_ALLOWANCE)
                .where(condition)
                .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
                .limit(request.getLimit())
                .fetchInto(NftAllowance.class);
    }

public class ThrottleConfiguration {

    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
