### Title
Unbounded gRPC Connection Count Enables Resource Exhaustion via Unlimited Concurrent Subscriptions

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection=5` on the `NettyServerBuilder`, but sets no limit on the total number of accepted connections. An unprivileged attacker can open an unbounded number of TCP connections, each saturating its 5-stream allowance with long-lived `subscribeTopic` server-streaming calls, causing unbounded accumulation of Reactor subscription state, prefetch buffers, and thread-pool tasks that drives the JVM into sustained GC pressure and degrades or denies service to all legitimate clients.

### Finding Description
**Exact code path:**

`GrpcConfiguration.java` lines 28–35 — the only `NettyServerBuilder` customization applied is:
```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```
`NettyProperties.java` line 14 confirms the single configurable field is `maxConcurrentCallsPerConnection = 5`. No call to `maxConnections()`, `maxConnectionAge()`, `maxConnectionIdle()`, or `maxInboundMessageSize()` is present anywhere in the configurer.

**Root cause / failed assumption:**
The design assumes that capping streams-per-connection to 5 bounds total server load. This assumption fails because there is no cap on the number of connections. Total concurrent streams = (number of connections) × 5, and the number of connections is unbounded.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication required — the gRPC service is public).
2. On each connection the attacker immediately opens 5 `subscribeTopic` streams with `consensusEndTime` unset (open-ended subscription), keeping each stream alive indefinitely.
3. The server accepts all N×5 streams. For each stream, the Reactor pipeline allocates subscription state and a prefetch queue of 48 messages (`ListenerProperties.prefetch = 48`, `ListenerProperties.java` line 34). At up to 6 KiB per HCS message, each stream can hold ~288 KiB in its prefetch buffer.
4. With N=2000 connections → 10,000 active streams → ~2.8 GB in prefetch buffers alone, plus Netty channel objects, HTTP/2 flow-control windows (default 1 MB per stream), and `applicationTaskExecutor` thread-pool tasks.
5. The JVM heap fills; the GC runs continuously trying to reclaim short-lived objects produced by the reactive pipelines, causing stop-the-world pauses. Existing legitimate clients experience timeouts and connection resets — a network-partition effect from the server's perspective.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection=5` limits streams per connection, not total connections.
- `listener.maxBufferSize=16384` caps the shared listener's internal ring buffer, not per-subscription memory.
- `retriever.timeout=60s` only applies when no messages are emitted; an active topic keeps streams alive indefinitely.
- No rate-limiting, IP-based connection throttling, or authentication is enforced at the gRPC layer in this codebase.

### Impact Explanation
A single attacker machine with sufficient file descriptors can open thousands of TCP connections and pin 5 streams each. The cumulative heap pressure from Reactor subscription objects, prefetch queues, and Netty channel state causes sustained GC storms. During GC pauses all gRPC threads are stopped, making the server appear partitioned to every connected client — including legitimate subscribers and internal monitoring. This is a complete denial-of-service against the gRPC API with no recovery until the attacker disconnects or the JVM OOM-kills the process.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required — the gRPC port is publicly accessible by design. The attack requires only a standard gRPC client library and the ability to open many TCP connections (trivially achievable from a single host or a small botnet). The `subscribeTopic` RPC is documented and its proto is public, so constructing valid requests is straightforward. The attack is repeatable and can be sustained indefinitely.

### Recommendation
Apply the following changes to `grpcServerConfigurer()` in `GrpcConfiguration.java` and expose them via `NettyProperties`:

1. **Add `serverBuilder.maxConnectionAge(duration, unit)`** — force periodic connection recycling to prevent indefinite accumulation.
2. **Add `serverBuilder.maxConnectionIdle(duration, unit)`** — evict connections that open no streams.
3. **Add `serverBuilder.maxInboundMessageSize(bytes)`** — cap inbound frame size (even though queries are small, this closes the default 4 MB window).
4. **Add a total-connection limit** — either via a Netty `ChannelOption` for `SO_BACKLOG` combined with a connection-counting `ChannelHandler`, or by fronting the service with an Envoy/ingress proxy configured with `max_connections` per downstream IP.
5. **Extend `NettyProperties`** to expose `maxConnectionAge`, `maxConnectionIdle`, and `maxInboundMessageSize` as configurable, validated fields alongside `maxConcurrentCallsPerConnection`.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor
# proto-generated stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

TARGET = "grpc.mainnet.mirrornode.hedera.com:443"
NUM_CONNECTIONS = 2000
STREAMS_PER_CONN = 5

def flood(i):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1)
        # no consensusEndTime → open-ended, stream lives forever
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        it = stub.subscribeTopic(query)
        streams.append(it)
    # keep streams alive
    for s in streams:
        try:
            for _ in s:
                pass
        except Exception:
            pass

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood, range(NUM_CONNECTIONS)))
# Result: server JVM heap exhausted, GC storms, all clients experience
# DEADLINE_EXCEEDED / connection resets within minutes.
