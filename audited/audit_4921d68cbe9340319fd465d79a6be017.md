### Title
Unbounded Exponential Backoff in `buildNetworkFromMirrorNodes()` Blocks Startup Indefinitely via HTTP 500 from Mirror REST Endpoint

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` retries up to `mirrorNodeClientMaxRetries+1` times (default: 11) with exponential backoff on any HTTP 5xx response. It is called from `newClient()` with `context.Background()` — not the signal-cancellable context — so no external cancellation is possible. A compromised or spoofed mirror REST endpoint returning HTTP 500 on every request causes the pinger to block for approximately 10+ minutes per startup attempt, and in a Kubernetes restart loop, this becomes a persistent denial-of-service against all transaction submissions.

### Finding Description

**Code path:**

`pinger/sdk_client.go` line 18 — `newClient()` calls `buildNetworkFromMirrorNodes(context.Background(), cfg)` only for `network=other`:

```go
case "other":
    netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
``` [1](#0-0) 

`pinger/mirror_node_client.go` lines 48–69 — the retry loop:

```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)  // default: 11
for attempt := 1; attempt <= attempts; attempt++ {
    network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
    if err == nil { return network, nil }
    lastErr = ...
    if !retry || attempt == attempts { break }
    backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
    select {
    case <-ctx.Done():   // ctx is context.Background() — NEVER fires
        return nil, ctx.Err()
    case <-time.After(backoff):
    }
}
``` [2](#0-1) 

`pinger/mirror_node_client.go` line 91 — HTTP 500 sets `retry=true`:

```go
retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
``` [3](#0-2) 

**Root cause:** Two compounding flaws:
1. `context.Background()` is passed instead of the signal-cancellable context from `main()`, so `ctx.Done()` never fires regardless of SIGINT/SIGTERM.
2. There is no cap on total elapsed time — only a cap on attempt count. With defaults (11 attempts, 500ms base, 10s HTTP timeout), total blocking = 11×10s (timeouts) + 500ms×(2⁰+2¹+…+2⁹) = 110s + 511.5s ≈ **621 seconds (~10.4 minutes)** per startup. [4](#0-3) [5](#0-4) 

**Why existing checks fail:** The `ctx.Done()` branch at line 65–66 is the only escape hatch, but `context.Background()` has no deadline and is never cancelled, making it dead code for this call site. The `attempt == attempts` guard only limits total attempts, not total time. [6](#0-5) 

### Impact Explanation

`newClient()` is called synchronously in `main()` before the ticker starts and before `/tmp/ready` is written:

```go
client, err := newClient(cfg)   // blocks here
if err != nil { log.Fatalf(...) }
// readiness file written only after this point
``` [7](#0-6) 

All downstream transaction submissions (`submitWithRetry`) are blocked for the full ~10 minutes. After exhausting retries, `log.Fatalf` kills the process. In Kubernetes (where this is deployed as a liveness/readiness probe service), the pod restarts and the cycle repeats indefinitely — a persistent startup-loop DoS. The `/tmp/ready` file is never created, so the readiness probe also fails continuously.

### Likelihood Explanation

The attack requires the attacker to control responses from the configured mirror REST endpoint (`HIERO_MIRROR_PINGER_REST`). This is realistic via:
- A compromised mirror node operator (the mirror node is a public, third-party-operated service in the Hiero ecosystem)
- DNS poisoning of the mirror REST hostname
- Network-level MITM between the pinger pod and the mirror node

This is scoped to `network=other` deployments. The attacker needs no credentials on the pinger host itself — only the ability to make the mirror REST URL return HTTP 500. The attack is trivially repeatable: every pod restart re-triggers the full blocking loop.

### Recommendation

1. **Pass the cancellable context**: In `sdk_client.go`, `newClient` should accept and forward the signal context from `main()`, not `context.Background()`.
2. **Add a total-time deadline**: Wrap the call with a `context.WithTimeout` (e.g., 60s) so the retry loop is bounded regardless of attempt count.
3. **Cap maximum backoff**: Add a `maxBackoff` clamp (e.g., 30s) to prevent the per-attempt sleep from growing unboundedly.
4. **Separate startup context from per-request context**: Use the signal context for the outer deadline and per-request timeouts for individual HTTP calls.

### Proof of Concept

1. Deploy pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host/`.
2. Configure `attacker-controlled-host` to return `HTTP 500` on all `GET /api/v1/network/nodes` requests.
3. Start the pinger. Observe:
   - `newClient()` enters the retry loop.
   - With defaults: attempt 1 sleeps 500ms, attempt 2 sleeps 1s, …, attempt 10 sleeps 256s. Total sleep ≈ 511.5s plus 110s of HTTP timeouts.
   - The process does not reach the ticker loop or write `/tmp/ready` for ~10 minutes.
   - After attempt 11, `log.Fatalf` kills the process.
4. In Kubernetes with `restartPolicy: Always`, the pod enters a `CrashLoopBackOff` with each restart re-triggering the full delay, permanently preventing transaction submissions.
