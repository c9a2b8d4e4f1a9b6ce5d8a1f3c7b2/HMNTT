### Title
Log Injection via Unsanitized `req.ip` (Attacker-Controlled `X-Forwarded-For`) in `responseCacheCheckHandler`

### Summary
`rest/server.js` sets `trust proxy` to `true` (boolean), causing Express to unconditionally accept the leftmost value of the attacker-supplied `X-Forwarded-For` header as `req.ip`. This value is interpolated verbatim into a `logger.info` call in `responseCacheCheckHandler`, and the custom logger writes the resulting string to stdout with no newline escaping or sanitization, enabling any unauthenticated external client to inject arbitrary forged log lines.

### Finding Description
**Code path:**

- `rest/server.js` line 58: `app.set('trust proxy', true)` — Express trusts every hop unconditionally; `req.ip` resolves to the leftmost value of `X-Forwarded-For`, which is fully attacker-controlled.
- `rest/middleware/responseCacheHandler.js` lines 84–86:
  ```js
  logger.info(
    `${req.ip} ${req.method} ${req.originalUrl} from cache (ttl: ${redisTtl}) in ${elapsed} ms: ${statusCode}`
  );
  ```
  `req.ip` is interpolated directly into the message string with no sanitization.
- `rest/logger.js` lines 69–70:
  ```js
  const text = `${time} ${levelName} ${requestId} ${msg}${stack}\n`;
  this.#write(text);
  ```
  The logger writes the raw string to `process.stdout` with no escaping of embedded newlines or control characters.

**Root cause / failed assumption:** The code assumes `req.ip` is a well-formed IP address. With `trust proxy: true`, Express instead uses whatever string the client places in `X-Forwarded-For`, including newlines, tabs, and arbitrary text. The logger has no sanitization layer.

**Exploit flow:**
1. Attacker sends a GET request to any cached endpoint with a crafted header:
   ```
   X-Forwarded-For: 1.2.3.4\n2024-01-01T00:00:00.000Z INFO Startup 127.0.0.1 GET /api/v1/accounts from cache (ttl: 999) in 0 ms: 200
   ```
2. The cache returns a hit; `responseCacheCheckHandler` executes.
3. `req.ip` evaluates to the attacker-supplied string (including the embedded `\n`).
4. The logger writes two lines to stdout: the real entry and the fully attacker-controlled forged entry.

**Why existing checks fail:** There are no checks. The logger (`rest/logger.js`) performs no escaping. The call site (`responseCacheHandler.js`) performs no validation. The `trust proxy: true` setting removes the only implicit protection (using the actual TCP peer address).

### Impact Explanation
An attacker can write arbitrary lines into the mirror node access logs that are indistinguishable from legitimate entries. This directly enables: (a) obscuring the true source IP of cache-poisoning or enumeration attacks by injecting fake "innocent" log lines, (b) injecting false audit trails (e.g., spoofed admin-looking requests), and (c) breaking log-based alerting or SIEM rules that parse structured log lines. Because the log is written to stdout and typically forwarded to centralized log aggregation, the impact extends beyond the local node.

### Likelihood Explanation
Exploitation requires zero privileges — any HTTP client reachable by the mirror node REST API can trigger it. The only precondition is that the targeted URL must have a cached response in Redis (which is the normal steady-state for popular endpoints). The attack is trivially repeatable and requires no special tooling beyond `curl`. The `trust proxy: true` setting is already in production configuration.

### Recommendation
1. **Sanitize `req.ip` before logging:** Strip or replace newlines, carriage returns, and other control characters before interpolation, e.g.:
   ```js
   const safeIp = (req.ip ?? '').replace(/[\r\n\t]/g, '_');
   logger.info(`${safeIp} ${req.method} ...`);
   ```
2. **Restrict `trust proxy`:** Change `app.set('trust proxy', true)` to a specific trusted CIDR or hop count (e.g., `app.set('trust proxy', 1)` or the actual load-balancer IP range) so `req.ip` is only derived from `X-Forwarded-For` when the request genuinely arrives through a trusted proxy.
3. **Sanitize at the logger level:** Add a sanitization step inside `Logger#log` in `rest/logger.js` to escape embedded newlines in `msg` before writing to stdout, providing defense-in-depth for all log call sites.

### Proof of Concept
```bash
# Assumes a cached endpoint exists, e.g. /api/v1/accounts
curl -s \
  -H $'X-Forwarded-For: 1.1.1.1\n2099-01-01T00:00:00.000Z INFO Startup 127.0.0.1 GET /api/v1/admin from cache (ttl: 9999) in 0 ms: 200' \
  http://<mirror-node-host>/api/v1/accounts
```
**Expected stdout output (two lines):**
```
2024-06-01T12:00:00.000Z INFO <reqId> 1.1.1.1
2099-01-01T00:00:00.000Z INFO Startup 127.0.0.1 GET /api/v1/admin from cache (ttl: 9999) in 0 ms: 200 GET /api/v1/accounts from cache (ttl: 30) in 2 ms: 200
```
The second line is the forged entry injected by the attacker, appearing as a legitimate historical log record.
