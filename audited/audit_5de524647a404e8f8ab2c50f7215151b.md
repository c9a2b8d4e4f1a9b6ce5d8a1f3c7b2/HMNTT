### Title
Missing Result Cache on `findByAlias` Enables Unauthenticated DB Exhaustion via Repeated GraphQL Alias Queries

### Summary
The GraphQL `EntityRepository.findByAlias` method issues a raw native SQL query on every invocation with no `@Cacheable` annotation, unlike the equivalent `web3` module which caches alias lookups. The GraphQL module has no application-level rate limiting, and the only concurrency guard (`inFlightReq: 5` per IP) does not throttle sequential request rates. Any unauthenticated user can drive unbounded repeated database queries by sending alias-lookup GraphQL requests in a tight loop.

### Finding Description

**Exact code path:**

`AccountController.account()` → `EntityService.getByAliasAndType()` → `EntityRepository.findByAlias()` [1](#0-0) 

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
```

No `@Cacheable` annotation is present. Compare with the web3 variant: [2](#0-1) 

which carries `@Cacheable(cacheNames = CACHE_NAME_ALIAS, cacheManager = CACHE_MANAGER_ENTITY, ...)`.

The service layer adds no caching either: [3](#0-2) 

**Root cause:** The graphql module's `EntityRepository` was never annotated with `@Cacheable`, so every call to `findByAlias` unconditionally executes a full table-scan-style native query against the `entity` table.

**Why the existing checks fail:**

1. `CachedPreparsedDocumentProvider` caches only the *parsed query document* (the query string AST), not query results: [4](#0-3) 

2. `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` limit query *structure*, not request *rate*: [5](#0-4) 

3. The only concurrency guard is `inFlightReq: amount: 5` per IP in the Traefik middleware: [6](#0-5) 
This limits *concurrent* requests, not the *rate* of sequential requests. An attacker sending requests one-after-another is not throttled.

4. There is zero application-level rate limiting in the graphql module (no `ThrottleConfiguration`, no `Bucket4j`, no Spring `@RateLimiter`).

### Impact Explanation
Each alias query executes a native SQL `SELECT * FROM entity WHERE alias = ?` against the database. With no result cache and no rate limit, an attacker can sustain thousands of identical DB queries per second from a single IP (sequential) or tens of thousands per second from multiple IPs (bypassing the per-IP concurrency cap). This degrades database connection pool availability and query throughput for all legitimate users of the mirror node, constituting a griefing/DoS against the read infrastructure. No authentication or special privilege is required.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication. The exploit requires only a valid alias value (obtainable from any public Hedera account) and a simple HTTP client loop. The attack is trivially repeatable, requires no special tooling, and is not prevented by any deployed control. A single attacker with one IP can sustain meaningful DB pressure; a distributed attacker trivially saturates the database.

### Recommendation
1. Add `@Cacheable` to `findByAlias` (and `findByEvmAddress`) in the graphql `EntityRepository`, mirroring the web3 module's `CACHE_NAME_ALIAS` / `CACHE_MANAGER_ENTITY` pattern.
2. Add application-level rate limiting to the GraphQL module (e.g., Bucket4j per-IP token bucket, analogous to `web3/ThrottleConfiguration`).
3. Optionally promote the Traefik `inFlightReq` to a true rate-limit middleware (e.g., Traefik `rateLimit`) rather than a concurrency cap.

### Proof of Concept
```bash
# Obtain any valid alias (e.g., from Hedera public mirror REST API)
ALIAS="ABCDEFGHIJKLMNOPQRSTUVWXYZ234567"  # base32-encoded alias

# Flood the GraphQL endpoint with sequential alias queries (no auth required)
while true; do
  curl -s -X POST https://<mirror-node-graphql>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { alias: \\\"$ALIAS\\\" }) { id } }\"}" \
    > /dev/null
done
```

Each iteration issues a fresh `SELECT * FROM entity WHERE alias = ? AND deleted IS NOT TRUE` to the database. No caching intercepts it. The loop is not rate-limited at the application layer. Database CPU and connection pool utilization rise linearly with request rate, degrading service for all users.
