### Title
LRU Token Cache Pollution via Unauthenticated Token Relationship Requests

### Summary
The `tokenCache` in `rest/service/tokenService.js` is a module-level singleton `quickLru` instance with a fixed `maxSize` and no per-request insertion limit or per-IP rate limiting. An unauthenticated attacker can repeatedly query the public `/accounts/:id/tokens` endpoint for accounts associated with many unique token IDs, causing the LRU cache to fill with attacker-driven entries and evicting legitimately cached tokens, degrading cache hit rates for all users.

### Finding Description
**Code location:** `rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161; invoked from `getTokenAccounts()` (line 104), which is called by `tokenController.js` `getTokenRelationships` (line 74).

**Root cause:** The `tokenCache` is initialized at module load with no `maxAge` and a fixed `maxSize`:

```js
// rest/service/tokenService.js lines 12-14
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,
});
```

Inside `getCachedTokens()`, every token ID returned from the DB for a request is unconditionally inserted into the cache:

```js
// lines 153-157
rows.forEach((row) => {
  const tokenId = row.token_id;
  const cachedToken = new CachedToken(row);
  tokenCache.set(tokenId, cachedToken);   // no insertion cap per request
  cachedTokens.set(tokenId, cachedToken);
});
```

There is no per-IP rate limiting enforced in the REST layer (no middleware found in `rest/**`), no cap on how many unique token IDs a single request may insert into the cache, and no `maxAge` to naturally expire entries.

**Exploit flow:**
1. Attacker identifies (or creates) accounts with associations to many unique token IDs — these are publicly queryable on-chain.
2. Attacker sends repeated unauthenticated GET requests to `/accounts/:id/tokens` for such accounts.
3. Each request causes `getCachedTokens()` to query the DB for uncached token IDs and insert all results into `tokenCache`.
4. Once `tokenCache` reaches `maxSize`, `quickLru` evicts the least-recently-used entries.
5. By cycling through many unique token IDs across requests, the attacker keeps inserting new entries, continuously evicting legitimately cached tokens used by other users.

**Why existing checks fail:** The `putTokenCache` method (lines 121–129) has a "no overwrite" guard (`if (tokenCache.has(tokenId)) return`), but `getCachedTokens` bypasses this guard and calls `tokenCache.set()` directly (line 156). There is no rate limiting, no per-request insertion quota, and no authentication requirement on the endpoint.

### Impact Explanation
Cache hit rate degrades for all users of the mirror node REST API. Every cache miss results in an additional DB query (`tokenCacheQuery`), increasing database load proportionally to the attack intensity. Under sustained attack, the token cache becomes effectively useless, causing every token relationship request to hit the database. This is a griefing/availability degradation with no data integrity impact and no direct economic loss to network participants.

### Likelihood Explanation
The attack requires no credentials, no special privileges, and no on-chain transactions if the attacker queries existing accounts with many token associations (common for DEX/protocol accounts). The endpoint is public, the attack is trivially scriptable, and it is repeatable indefinitely. The only practical barrier is network bandwidth and the attacker's willingness to sustain the requests.

### Recommendation
1. **Add `maxAge` to `tokenCache`** so entries expire naturally, limiting the window of cache pollution.
2. **Cap per-request cache insertions**: limit how many new token IDs a single `getCachedTokens()` call may insert (e.g., reject or skip if `uncachedTokenIds.length` exceeds a threshold).
3. **Enforce IP-based rate limiting** on the `/accounts/:id/tokens` endpoint at the middleware level.
4. **Fix the bypass in `getCachedTokens`**: replace the direct `tokenCache.set()` call (line 156) with a call to `putTokenCache()` so the "no overwrite if exists" guard is consistently applied (though this alone does not prevent the eviction attack).

### Proof of Concept
```bash
# Step 1: Find or create accounts with many token associations
# Step 2: Repeatedly query the endpoint with different high-token-count accounts
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$((i+1000))/tokens?limit=100" > /dev/null &
done
# Result: tokenCache fills with attacker-driven token IDs, evicting legitimate entries.

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
