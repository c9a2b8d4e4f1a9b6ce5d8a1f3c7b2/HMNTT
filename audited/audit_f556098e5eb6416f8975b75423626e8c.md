### Title
Unauthenticated Rapid Subscribe/Cancel Loop Causes Unbounded Log Flooding via gRPC `subscribeTopic`

### Summary
Any unauthenticated external client can call `subscribeTopic()` and immediately cancel the stream in a tight loop. Each cycle unconditionally emits at least two INFO-level log entries with no rate limiting applied to the gRPC subscription endpoint. The only server-side guard (`maxConcurrentCallsPerConnection = 5`) restricts concurrent calls per connection but does not bound the rate of sequential subscribe/cancel cycles or prevent an attacker from opening many parallel connections.

### Finding Description

**Code path:**

In `ConsensusController.subscribeTopic()`, a reactive subscription is created and a cancel handler is registered with no authentication or rate-limiting guard: [1](#0-0) 

When the client cancels, `disposable::dispose` is called, which terminates the reactive chain and triggers all `doFinally` hooks.

In `TopicMessageServiceImpl.subscribeTopic()`, line 60 emits an unconditional INFO log on every invocation: [2](#0-1) 

The `doFinally` hooks on lines 90–91 fire on every termination signal (including `CANCEL` from disposal): [3](#0-2) 

`TopicContext.finished()` emits a second INFO log on every termination: [4](#0-3) 

**Root cause:** There is no per-IP, per-connection, or global rate limit on the frequency of `subscribeTopic` invocations. The entity existence check uses a cache (`entityCacheSize = 50,000`), so after the first lookup the topic check is near-instant and does not act as a throttle. [5](#0-4) 

**Why the existing check fails:** The only server-side guard is `maxConcurrentCallsPerConnection = 5`: [6](#0-5) [7](#0-6) 

This limits *concurrent* calls per connection, not the *rate* of sequential calls. An attacker cycling subscribe→cancel→subscribe on a single connection stays within the limit. With multiple connections (no per-IP connection cap exists in the code), the effective rate multiplies linearly. The `web3` module has a `ThrottleConfiguration` with bucket4j rate limiting, but no equivalent exists in the `grpc` module. [8](#0-7) 

### Impact Explanation
Each subscribe/cancel cycle produces a minimum of two INFO log lines. At modest attack rates (e.g., 1,000 cycles/second across 10 connections), this generates ~20,000 log lines/second. Consequences include: log storage exhaustion, log aggregation pipeline saturation, and legitimate error/warning messages being buried or dropped. The `subscriberCount` gauge also oscillates rapidly, polluting metrics. No economic damage to network users occurs, but mirror node operator observability and storage are directly degraded.

### Likelihood Explanation
The attack requires zero privileges — the gRPC port (default 5600) is publicly accessible with no authentication. A standard gRPC client library in any language can implement the loop in under 20 lines of code. The attack is trivially repeatable, requires no special knowledge of the protocol beyond the public proto definition, and is not detectable until log volume spikes.

### Recommendation
1. **Add a per-IP or global subscription rate limiter** in the gRPC layer, analogous to the bucket4j `ThrottleConfiguration` already present in the `web3` module, applied as a `ServerInterceptor` before `subscribeTopic` is reached.
2. **Limit the total number of active subscriptions** by checking `subscriberCount` against a configurable maximum before allowing a new subscription to proceed.
3. **Downgrade or conditionally suppress** the `finished()` INFO log for very short-lived subscriptions (e.g., those that received 0 messages and lived under a threshold duration), or rate-limit log emission per subscriber ID / source IP.
4. **Add a per-connection sequential call rate limit** (not just a concurrency cap) at the Netty server level.

### Proof of Concept
```python
import grpc
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery()
query.topicID.topicNum = 1234  # any existing topic

for _ in range(10_000):
    # Open subscription
    stream = stub.subscribeTopic(query)
    # Immediately cancel — triggers setOnCancelHandler -> disposable::dispose
    stream.cancel()
    # Each iteration: 2 INFO log lines written to application log
```

Each iteration triggers:
- `log.info("Subscribing to topic: {}", filter)` in `TopicMessageServiceImpl` line 60
- `log.info("[{}] Topic {} {} with {} messages ...", ...)` in `TopicContext.finished()` line 229

At 10,000 iterations this produces ~20,000 INFO log entries with no server-side rejection.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
