### Title
IP Spoofing via X-Forwarded-For Due to Unconditional `trust proxy: true` in `handleError()` Log Attribution

### Summary
`rest/server.js` sets `app.set('trust proxy', true)`, which instructs Express to unconditionally trust all entries in the `X-Forwarded-For` header chain and derive `req.ip` from the leftmost (attacker-controlled) entry. The `handleError()` middleware logs `req.ip` verbatim in every error log line, so any unprivileged external client can inject an arbitrary IP address into the error logs, defeating attribution of repeated error-triggering requests (e.g., a transaction-gossip DoS).

### Finding Description
**Code locations:**

- `rest/server.js` line 58: `app.set('trust proxy', true);` [1](#0-0) 

- `rest/middleware/httpErrorHandler.js` lines 33–34 and 39: `req.ip` interpolated directly into `logger.warn(...)` and `logger.error(...)` [2](#0-1) 

**Root cause:** When Express's `trust proxy` is set to the boolean `true`, it trusts *every* IP in the `X-Forwarded-For` chain as a legitimate proxy hop. Express's `req.ip` resolution walks the chain from right to left, skipping all "trusted" entries, and since every entry is trusted, it lands on the leftmost value — which is fully attacker-controlled. The nginx reverse proxy (configured in `docker-compose.yml` line 205 with `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for`) does append the real remote address, but it does so *after* the attacker-supplied value, not instead of it. [3](#0-2) 

**Exploit flow:**
1. Attacker sends: `GET /api/v1/transactions?... HTTP/1.1` with header `X-Forwarded-For: 10.0.0.1` (any spoofed IP).
2. Nginx forwards the request, producing `X-Forwarded-For: 10.0.0.1, <real_attacker_ip>`.
3. Express, with `trust proxy: true`, treats both entries as trusted proxies and resolves `req.ip` to `10.0.0.1`.
4. The request triggers an error (e.g., `InvalidArgumentError`, `DbError`, or a 500); `handleError()` fires and logs `10.0.0.1 GET /api/v1/transactions ...` — the attacker's real IP never appears.

**Why existing checks are insufficient:** The nginx `$proxy_add_x_forwarded_for` directive appends the real IP but does not strip or replace a pre-existing `X-Forwarded-For` header supplied by the client. Express's `trust proxy: true` (boolean) is documented to trust all proxies without limit, unlike `trust proxy: 1` (numeric) which would only trust one hop and would correctly expose the real IP.

### Impact Explanation
An attacker conducting a high-volume error-triggering DoS (e.g., flooding malformed transaction queries to exhaust DB connections or trigger repeated 500s) can mask their real IP in every error log entry. Operators relying on these logs for incident response, rate-limit enforcement, or IP-based blocking will act on a false address. This directly undermines the ability to attribute and mitigate a transaction-gossip DoS attack.

### Likelihood Explanation
No authentication or special privilege is required. Any external HTTP client can set arbitrary request headers. The technique is trivially reproducible with `curl -H "X-Forwarded-For: <spoofed>" ...`. The misconfiguration (`trust proxy: true`) is a well-known Express pitfall and is actively exploited in the wild against misconfigured Node.js APIs.

### Recommendation
Replace the unconditional boolean with a numeric hop count or an explicit trusted-proxy subnet:

```js
// Trust exactly one proxy hop (the nginx sidecar)
app.set('trust proxy', 1);
// OR trust only the known proxy IP/subnet
app.set('trust proxy', '10.0.0.0/8');
```

With `trust proxy: 1`, Express will treat only the rightmost `X-Forwarded-For` entry (appended by nginx, reflecting the real TCP peer) as the client IP, making `req.ip` unspoofable by external clients.

### Proof of Concept
```bash
# Spoof IP as 1.2.3.4 while triggering an error (invalid param)
curl -s \
  -H "X-Forwarded-For: 1.2.3.4" \
  "http://<mirror-node-host>/api/v1/transactions?timestamp=INVALID"

# Observe server error log — req.ip will show 1.2.3.4, not the real attacker IP
# Expected log line (from handleError lines 33-34):
# WARN: 1.2.3.4 GET /api/v1/transactions?timestamp=INVALID in Xms: 400 InvalidArgumentError ...
```
