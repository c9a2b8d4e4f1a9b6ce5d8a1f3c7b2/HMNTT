### Title
Reference Equality Bypass on `BlockType.EARLIEST` Allows `findByIndex(0)` to Be Called Instead of `findEarliest()`, Causing `BlockNumberNotFoundException` in Partial-History Deployments

### Summary
`findByBlockType()` in `RecordFileServiceImpl` uses Java reference equality (`==`) to detect the `EARLIEST` sentinel. When a user submits a hex-zero block parameter such as `0x0000000000000000`, `BlockType.of()` constructs a **new** `BlockType` instance with `number=0` rather than returning the `EARLIEST` singleton, causing the identity check to fail. The call falls through to `recordFileRepository.findByIndex(0)`, which differs semantically from `findEarliest()` and can return an empty `Optional`, triggering `BlockNumberNotFoundException` instead of the expected genesis block.

### Finding Description

**Exact code path:**

`BlockType.java` defines the singleton: [1](#0-0) 

`BlockType.of("0x0000000000000000")` matches the `GROUP_HEX_NUM` regex group (`0x([0-9a-f]{1,16})`): [2](#0-1) 

This returns `new BlockType("0000000000000000", 0L)` — a **new heap object**, not the `EARLIEST` singleton.

In `findByBlockType()`, the guard uses `==` (reference equality): [3](#0-2) 

Because the new instance is not the same object reference as `EARLIEST`, the check at line 20 is false. `isHash()` is also false (number is 0, not `BLOCK_HASH_SENTINEL = -1`). The call falls through to `recordFileRepository.findByIndex(0)`.

**Semantic difference between the two repository methods:**

- `findEarliest()`: `SELECT * FROM record_file ORDER BY index ASC LIMIT 1` — always returns the lowest-indexed block present. [4](#0-3) 

- `findByIndex(0)`: `SELECT r FROM RecordFile r WHERE r.index = 0` — returns a block only if one with **exactly** index 0 exists. [5](#0-4) 

**Exception propagation:**

The caller wraps the result with `.orElseThrow(BlockNumberNotFoundException::new)`: [6](#0-5) 

When `findByIndex(0)` returns empty, `BlockNumberNotFoundException` is thrown and the API returns HTTP 400 with `"Unknown block number"`. [7](#0-6) 

**Why the existing check is insufficient:**

`BlockType` is a Java `record`. Records do not override `==`; it remains object-reference comparison. The only path that returns the `EARLIEST` singleton is the `"earliest"` string tag branch: [8](#0-7) 

Any numeric representation of block 0 (hex or decimal) bypasses this branch and creates a new instance, making the `==` guard in `findByBlockType()` permanently ineffective for those inputs.

### Impact Explanation

In a **partial-history** mirror node deployment (where the node began syncing after genesis), the earliest stored block has `index > 0`. In that configuration, `findByIndex(0)` returns empty and every `eth_call` or `eth_estimateGas` request specifying block `0x0` (or any zero-valued hex string) throws `BlockNumberNotFoundException` (HTTP 400), while the same request with the string `"earliest"` succeeds and returns the earliest available block. This is incorrect protocol behavior: per the Ethereum JSON-RPC spec, block number `0x0` and tag `earliest` are semantically equivalent. Even in full-history deployments the behavior is only accidentally correct because the genesis block happens to have `index=0`; the logic is structurally broken regardless.

### Likelihood Explanation

Any unprivileged external caller can trigger this with a standard JSON-RPC request. The input `0x0` or `0x0000000000000000` is a valid, commonly used block parameter. No authentication, special role, or knowledge of internal state is required. The condition is fully repeatable and deterministic.

### Recommendation

Replace the reference-equality (`==`) checks in `findByBlockType()` with value-based comparisons. The cleanest fix is to use `.equals()` or compare the `number` field directly:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block.equals(BlockType.EARLIEST) || (!block.isHash() && block.number() == 0)) {
        return recordFileRepository.findEarliest();
    } else if (block.equals(BlockType.LATEST) || block.number() == Long.MAX_VALUE) {
        return recordFileRepository.findLatest();
    } else if (block.isHash()) {
        return recordFileRepository.findByHash(block.name());
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Alternatively, normalize numeric-zero inputs inside `BlockType.of()` to return the `EARLIEST` singleton directly, mirroring how `blockTypeForTag("earliest")` works.

### Proof of Concept

**Precondition:** A Hiero mirror node where the earliest record file has `index > 0` (partial-history deployment), OR any deployment where the intent is to test the code path divergence.

**Steps:**

1. Send a valid `eth_call` with block tag `"earliest"`:
   ```json
   POST /api/v1/contracts/call
   {"block": "earliest", "to": "0x...", "data": "0x..."}
   ```
   **Result:** HTTP 200, returns result based on genesis/earliest block.

2. Send the same call with hex-zero block number `"0x0"`:
   ```json
   POST /api/v1/contracts/call
   {"block": "0x0", "to": "0x...", "data": "0x..."}
   ```
   **Result (partial-history):** HTTP 400, `{"_status":{"messages":[{"message":"Unknown block number"}]}}` — `BlockNumberNotFoundException` thrown because `findByIndex(0)` returns empty while `findEarliest()` would have returned the earliest available block.

3. Confirm the divergence by also testing `"0x0000000000000000"` (8-byte zero), `"0x00"`, and decimal `"0"` — all produce new `BlockType` instances that bypass the `== EARLIEST` guard.
