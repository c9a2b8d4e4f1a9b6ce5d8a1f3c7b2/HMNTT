### Title
Hardcoded `sslmode=disable` in `GetDsn()` Enables Plaintext MitM Injection on the RosettaŌĆōPostgreSQL Connection

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` unconditionally appends `sslmode=disable` to the PostgreSQL DSN, with no configuration field or code path to enable TLS. An attacker who can reach the same L2 network segment as the Rosetta appŌĆōdatabase link and perform ARP spoofing can intercept the unencrypted PostgreSQL wire protocol, inject crafted query responses, and cause the Rosetta API to serve false or stale blockchain state to all consumers.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
"host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
``` [1](#0-0) 

This string is the sole DSN source. `ConnectToDb()` in `rosetta/app/db/db.go` line 18 calls `dbConfig.GetDsn()` directly:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
``` [2](#0-1) 

The `Db` struct has no `Ssl`, `SslMode`, or `TlsCert` field: [3](#0-2) 

A codebase-wide search for `ssl`, `tls`, `TLS`, `SSL` in all Rosetta Go files returns exactly three hits ŌĆö all of them hardcoding `sslmode=disable`. There is no configuration key, environment variable, or YAML field that can override this.

**Root cause:** The developer assumed the database is always co-located on localhost or a trusted private link, so TLS was permanently disabled rather than made configurable.

**Exploit flow:**
1. Attacker gains a foothold on the same L2 segment (e.g., compromised container in the same pod network, rogue VM on the same VLAN, or physical LAN access).
2. Attacker runs ARP spoofing (`arpspoof`/`ettercap`) to become the MitM between the Rosetta process and the PostgreSQL host.
3. Because `sslmode=disable` is hardcoded, the PostgreSQL client never negotiates TLS; all traffic is plaintext PostgreSQL wire protocol.
4. Attacker intercepts a `SELECT` response for a balance or transaction query and replaces the row data with crafted values (e.g., zeroed balances, replayed old block hashes).
5. The Rosetta API returns the injected data to all callers without any integrity check.

### Impact Explanation
The Rosetta API is a blockchain data interface; consumers (exchanges, wallets, indexers) rely on it for account balances and transaction history. Injecting false responses causes those consumers to act on incorrect state ŌĆö e.g., showing wrong balances or missing transactions ŌĆö without any cryptographic signal that the data was tampered with. The impact is data-integrity corruption of the blockchain view served to all API users. No economic damage to the network itself occurs, but downstream systems making decisions on this data can be griefed.

### Likelihood Explanation
The precondition is same-L2-segment access, which is non-trivial for a purely external attacker but realistic in:
- Shared cloud tenancy / misconfigured VPC peering
- Kubernetes clusters where the Rosetta pod and DB pod share a flat pod network
- Any deployment where the DB is not on localhost

ARP spoofing is a well-understood, tool-assisted technique requiring no special privileges beyond network reachability. The attack is repeatable and silent (no application-layer error is raised).

### Recommendation
1. Add an `SslMode` (or `Ssl bool`) field to the `Db` config struct in `rosetta/app/config/types.go`.
2. Change `GetDsn()` to interpolate the configured value instead of the literal `disable`:
   ```go
   "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
   db.Host, db.Port, db.Username, db.Name, db.Password, db.SslMode,
   ```
3. Set the default in `application.yml` to `require` (or `verify-full` with a CA cert path).
4. Update `TestDbGetDsn` to assert the configurable value is used.

### Proof of Concept
