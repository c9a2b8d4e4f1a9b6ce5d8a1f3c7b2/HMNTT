### Title
Unauthenticated `X-Forwarded-For` Spoofing Bypasses IP Logging for `/construction/submit`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the raw `X-Forwarded-For` (or `X-Real-IP`) header value with no validation beyond `len(ipAddress) == 0`. Any unprivileged external client that can reach the service directly (or through a proxy that forwards the header) can supply an arbitrary non-empty string — including a fake IP, the string `unknown`, or Unicode whitespace such as a non-breaking space (U+00A0, two UTF-8 bytes, `len > 0`) — and that string will be written verbatim into every log line for the request. For `/construction/submit` this permanently destroys the IP audit trail for fraudulent transactions.

### Finding Description

**Exact location:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```
63: func getClientIpAddress(r *http.Request) string {
64:     ipAddress := r.Header.Get(xRealIpHeader)
65:
66:     if len(ipAddress) == 0 {
67:         ipAddress = r.Header.Get(xForwardedForHeader)
68:     }
69:
70:     if len(ipAddress) == 0 {
71:         ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
72:     }
73:
74:     return ipAddress
75: }
```

**Root cause:** The only guard is `len(ipAddress) == 0`. Go's `net/textproto` trims leading/trailing ASCII whitespace (0x09, 0x20) from header values, so a pure-ASCII-space or pure-tab value is reduced to `""` and falls through to `RemoteAddr`. However:

1. **Non-ASCII Unicode whitespace** (e.g., U+00A0 non-breaking space, encoded as `0xC2 0xA0` in UTF-8) is **not** trimmed by Go's `textproto` trimmer, which only strips bytes `0x09` and `0x20`. A header value of a single non-breaking space has `len() == 2`, passes both `len == 0` guards, and is returned as the "IP address."
2. **Any non-empty fake value** — `0.0.0.0`, `127.0.0.1`, `unknown`, a spoofed peer IP — also passes both guards with no format or ownership validation.

The returned value is interpolated directly into the log message at line 52–53:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```
and emitted at `log.Info` level for all non-internal paths (line 58), including `/construction/submit`.

No call to `net.ParseIP`, `strings.TrimSpace`, or any allowlist/denylist exists anywhere in the middleware.

### Impact Explanation

`/construction/submit` is the transaction-broadcast endpoint. Every submitted transaction is logged with the client IP as the primary forensic identifier. An attacker who spoofs this field causes:

- The logged "IP" to be an attacker-controlled string (fake IP, invisible Unicode whitespace, or `unknown`).
- Log-analysis pipelines that parse the first whitespace-delimited token as the source IP to either silently record the fake value or produce a parse error.
- Complete loss of IP-based correlation for fraudulent or double-spend transactions — the only post-hoc tracing mechanism available at the middleware layer.

Severity is high for a financial/blockchain relay service where audit trails are a compliance and incident-response requirement.

### Likelihood Explanation

**Precondition:** The attacker must be able to set `X-Forwarded-For` in a request that reaches the service. This is true when:
- The service is exposed directly (no reverse proxy), which is common in development/staging and possible in misconfigured production deployments.
- A reverse proxy is configured to *append* to (rather than *replace*) `X-Forwarded-For`, allowing the attacker's injected value to appear as the first entry.
- The attacker controls an intermediate hop that sets the header.

No authentication, no special privilege, and no prior knowledge of the system is required. The attack is trivially repeatable with a single `curl` command.

### Recommendation

Replace the `len == 0` guard with strict IP validation:

```go
import "strings"

func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        raw := r.Header.Get(header)
        // XFF may be a comma-separated list; take the first entry
        if idx := strings.IndexByte(raw, ','); idx != -1 {
            raw = raw[:idx]
        }
        raw = strings.TrimSpace(raw)
        if ip := net.ParseIP(raw); ip != nil {
            return ip.String()
        }
    }
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

`net.ParseIP` returns `nil` for any non-IP string (including whitespace, Unicode, fake tokens), ensuring only syntactically valid IP addresses are logged. Additionally, consider whether the deployment should strip/overwrite `X-Forwarded-For` at the ingress layer.

### Proof of Concept

**Whitespace (Unicode non-breaking space) variant — bypasses both `len == 0` checks:**
```bash
# U+00A0 encoded as UTF-8 bytes 0xC2 0xA0
curl -X POST http://<rosetta-host>/construction/submit \
  -H $'X-Forwarded-For: \xc2\xa0' \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":...,"signed_transaction":"..."}'
```
Expected log output: `  POST /construction/submit (200) in Xms` — the IP field is an invisible non-breaking space, breaking any whitespace-delimited log parser.

**Fake IP variant — trivially exploitable:**
```bash
curl -X POST http://<rosetta-host>/construction/submit \
  -H 'X-Forwarded-For: 1.2.3.4' \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":...,"signed_transaction":"..."}'
```
Expected log output: `1.2.3.4 POST /construction/submit (200) in Xms` — the attacker's real `RemoteAddr` is never recorded; the fake IP is logged instead.
