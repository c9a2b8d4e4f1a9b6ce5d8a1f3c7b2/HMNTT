### Title
Unrestricted `file.id` Parameter Allows Retrieval of Superseded Address Book Data in `getNetworkNodes()`

### Summary
The `/api/v1/network/nodes` endpoint accepts a user-supplied `file.id` query parameter with no whitelist validation on the actual value. An unprivileged external user can supply `file.id=101` (the legacy, superseded address book) instead of the current `file.id=102`, causing `getNetworkNodes()` to return historical node topology data. This allows any caller to obtain and present an outdated view of the Hashgraph network without any privilege or authentication.

### Finding Description
**Code path:**

In `NetworkController.getNodes()` ( [1](#0-0) ), the only validation applied to the `file.id` parameter is an operator check:

```java
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
```

This rejects range operators (e.g., `gt`, `lt`) but **does not restrict which file ID values are permitted**. Any numeric entity ID passes through.

The value is then forwarded to `NetworkServiceImpl.getAddressBookFileId()` ( [2](#0-1) ):

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
}
```

The default is `addressBookFile102`, but if the caller supplies any value, it is used verbatim — no check against a whitelist of `{101, 102}` or restriction to only `102`.

This `fileId` is passed directly into the SQL query in `NetworkNodeRepository.findNetworkNodes()` ( [3](#0-2) ):

```sql
with latest_address_book as (
    select start_consensus_timestamp, end_consensus_timestamp, file_id
    from address_book
    where file_id = :fileId
    order by start_consensus_timestamp desc
    limit 1
),
```

The query fetches the **latest** address book for whatever `file_id` is supplied. For `file_id=101`, this returns the most recent snapshot of the old (superseded) address book, which may contain stale node endpoints, decommissioned nodes, or different public keys.

**Root cause:** The failed assumption is that callers will only ever supply `file.id=102`. There is no server-side enforcement of this assumption.

### Impact Explanation
An attacker can craft a URL such as `GET /api/v1/network/nodes?file.id=101` and share it with SDK clients, wallet applications, or monitoring tools that consume the mirror node REST API. Those clients would receive a node list reflecting the old address book — potentially including nodes that are no longer active, have changed IP/gRPC endpoints, or have been replaced. Clients relying on this data to route transactions or verify network membership would be operating on a false picture of the current Hashgraph topology. The severity is moderate: the mirror node is read-only and the attacker cannot inject fabricated data, but they can surface real historical data in a misleading context.

### Likelihood Explanation
Exploitation requires zero privileges — the endpoint is unauthenticated and publicly accessible. The `file.id` parameter is documented in the API. Any external user who reads the API spec can attempt this. The attack is trivially repeatable and requires only a single HTTP GET request. The primary constraint on impact is whether downstream clients blindly trust the `file.id` value in the response or independently validate it.

### Recommendation
In `getAddressBookFileId()` (or in the controller before dispatch), enforce a whitelist of permitted file IDs. Only `addressBookFile102` (and optionally `addressBookFile101` if explicitly intended for historical queries) should be accepted:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long requestedId = request.getFileId().value();
    long allowedId = systemEntity.addressBookFile102().getId();
    if (requestedId != allowedId) {
        throw new IllegalArgumentException(
            "file.id must be " + allowedId);
    }
    return requestedId;
}
```

Alternatively, remove the `file.id` parameter from the public API entirely if querying non-current address books is not an intended use case.

### Proof of Concept
```
# Default (correct) — returns current address book (file 102)
GET /api/v1/network/nodes

# Exploit — returns superseded address book (file 101) with historical node data
GET /api/v1/network/nodes?file.id=101
```

No authentication, no special headers, no elevated privileges required. The response will contain node entries from the old address book, which may differ from the currently active network topology.
