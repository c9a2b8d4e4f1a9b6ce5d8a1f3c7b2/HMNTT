### Title
Unbounded `json_agg` on `transaction_signature` in `getScheduleById` Enables Memory Exhaustion DoS

### Summary
The `getScheduleByIdQuery` in `rest/schedules.js` performs an uncapped `json_agg` over all rows in `transaction_signature` matching a given `schedule_id`, with no `LIMIT` clause on the join or the aggregation. Because the `transaction_signature` table intentionally has no unique constraint (allowing the same key to sign multiple times), an attacker with any HBAR can submit arbitrarily many `ScheduleSign` transactions against a single schedule, causing the mirror node to aggregate and serialize an unbounded payload into Node.js heap memory, exhausting server resources.

### Finding Description
**Exact code path:**

`rest/schedules.js`, lines 39–64:

```js
const transactionSignatureJsonAgg = `
  json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp,
    'public_key_prefix', encode(ts.public_key_prefix, 'base64'),
    'signature', encode(ts.signature, 'base64'),
    'type', ts.type
  ) order by ts.consensus_timestamp)`;   // ← no LIMIT
const getScheduleByIdQuery = `
  ...
  ${transactionSignatureJsonAgg} as signatures
  from schedule s
  left join entity e on e.id = s.schedule_id
  left join transaction_signature ts on ts.entity_id = s.schedule_id   // ← unbounded join
  where s.schedule_id = $1
  group by s.schedule_id, e.id`;
``` [1](#0-0) 

**Root cause:** The `json_agg` aggregates every row in `transaction_signature` for the given `schedule_id` with no row cap. The failed assumption is that the number of signatures per schedule is bounded by the network protocol.

**Why the assumption fails:** The database schema design explicitly documents: *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key."* [2](#0-1) 

The `transaction_signature` table has no uniqueness constraint: [3](#0-2) 

The importer inserts one `TransactionSignature` row per `sigPair` entry in the `sigMap` of every `ScheduleCreate` or `ScheduleSign` transaction: [4](#0-3) 

**Exploit flow:**
1. Attacker creates a scheduled transaction on Hedera (one `ScheduleCreate`).
2. Attacker submits N `ScheduleSign` transactions, each containing M `sigPair` entries in the `sigMap`. Each submission inserts N×M rows into `transaction_signature` for that `schedule_id`.
3. Attacker calls `GET /api/v1/schedules/{scheduleId}`.
4. The mirror node executes `getScheduleByIdQuery`, which joins all N×M rows and passes them through `json_agg` with no limit.
5. The entire aggregated JSON array is materialized in PostgreSQL, transferred to Node.js, held in heap memory, and serialized via `JSONStringify` before being sent. [5](#0-4) 

**Why existing checks are insufficient:**

- The `response.limit.default` (25) and `response.limit.max` (100) configuration values apply only to paginated list endpoints. `getScheduleById` is a single-resource endpoint with no pagination and no limit applied to the signatures sub-array. [6](#0-5) 

- The `statementTimeout` of 20,000 ms limits query *execution time*, not the size of the result set materialized in memory. A query aggregating tens of thousands of rows can complete well within 20 seconds while producing a multi-hundred-MB payload.
- There is no HTTP response body size cap in the REST middleware (`responseHandler.js` calls `res.send()` unconditionally). [7](#0-6) 

### Impact Explanation
A single attacker-controlled schedule can be used to force the mirror node to allocate arbitrarily large amounts of heap memory per request. Because Node.js is single-threaded and heap allocation is synchronous during JSON serialization, a sufficiently large payload will stall the event loop and/or trigger an OOM crash, making the entire REST API unavailable. The attack targets a public, unauthenticated endpoint (`GET /api/v1/schedules/{id}`), so no credentials are required beyond HBAR to fund the setup transactions.

### Likelihood Explanation
Hedera transaction fees are fractions of a US cent. On testnet they are free. A `ScheduleSign` transaction can carry multiple `sigPair` entries per submission. An attacker can automate thousands of `ScheduleSign` submissions over the lifetime of a schedule (up to ~62 days) at negligible cost. The attack is repeatable: a new schedule can be created after each mitigation attempt. No privileged access to the mirror node or its database is required.

### Recommendation
1. **Add a `LIMIT` to the `transaction_signature` join** inside `getScheduleByIdQuery`, e.g. using a lateral subquery or a `FETCH FIRST N ROWS ONLY` on the aggregated sub-select, capped at a reasonable maximum (e.g. 1,000 signatures).
2. **Paginate signatures** in the API response (add `signatures.next` link) consistent with how other list sub-resources are handled.
3. **Add a unique constraint** (or a deduplication step in the importer) on `(entity_id, consensus_timestamp, public_key_prefix)` in `transaction_signature` to prevent unbounded duplicate accumulation.
4. **Add a response body size guard** in the REST middleware that aborts serialization above a configurable threshold.

### Proof of Concept
```
# 1. Create a scheduled CryptoTransfer on Hedera testnet (free)
#    Record the returned scheduleId, e.g. 0.0.12345

# 2. In a loop, submit ScheduleSign transactions each with a large sigMap
for i in $(seq 1 5000); do
  hedera-cli schedule sign --schedule-id 0.0.12345 \
    --sig-map "$(generate_large_sigmap 20)"   # 20 sigPairs per tx
done
# This inserts 5000 × 20 = 100,000 rows into transaction_signature for schedule 0.0.12345

# 3. Wait for mirror node ingestion (~seconds)

# 4. Trigger the DoS
curl https://<mirror-node>/api/v1/schedules/0.0.12345
# Server allocates ~100,000 JSON objects in heap; response is hundreds of MB;
# Node.js process OOMs or event loop stalls, dropping all concurrent requests.
```
