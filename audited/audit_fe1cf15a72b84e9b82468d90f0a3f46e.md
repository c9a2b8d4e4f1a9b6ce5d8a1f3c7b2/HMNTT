### Title
Unauthenticated Coordinated DB Query Flood via Unbounded Safety-Check Subscriptions

### Summary
`TopicMessageServiceImpl.subscribeTopic()` has no global subscriber count gate — `subscriberCount` is a metric only. An unauthenticated attacker can open arbitrarily many gRPC connections (each carrying up to 5 subscriptions via `maxConcurrentCallsPerConnection`), all with `startTime=0`. One second after the historical retriever drains (near-instant for a topic with no history), every subscription's safety-check fires simultaneously, each issuing an unthrottled `topicMessageRetriever.retrieve(gapFilter, false)` call that may execute up to 12 DB polls × 5,000 rows. During a network partition — when the live listener delivers nothing — this burst is guaranteed to fire for every open subscription at the same moment.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) builds three reactive streams per subscription:

```
historical  = topicMessageRetriever.retrieve(filter, true)          // throttled
live        = Flux.defer(() -> incomingMessages(topicContext))
safetyCheck = Mono.delay(1s)
              .filter(_ -> !topicContext.isComplete())               // always false when endTime==null
              .flatMapMany(_ -> missingMessages(topicContext, null)) // → retrieve(gapFilter, false)
              .subscribeOn(Schedulers.boundedElastic())
``` [1](#0-0) 

`missingMessages(topicContext, null)` (the safety-check branch, `current == null`) builds a `gapFilter` with `startTime = filter.getStartTime()` (i.e., 0) when no message has been received yet, then calls `retrieve(gapFilter, false)` — the **unthrottled** path. [2](#0-1) 

The unthrottled retriever runs up to `maxPolls=12` polls at `pollingFrequency=20ms` with `maxPageSize=5000` rows each. [3](#0-2) 

**Root cause — no subscriber gate:**

`subscriberCount` is wired only to a Micrometer `Gauge`; it is never checked before accepting a new subscription. [4](#0-3) 

**Connection limit is per-connection, not global:**

`maxConcurrentCallsPerConnection = 5` limits calls on a single HTTP/2 connection, but there is no cap on the number of connections. [5](#0-4) [6](#0-5) 

**`isComplete()` is always `false` for open-ended subscriptions:** [7](#0-6) 

**`startTime` validation only requires `startTime <= now()`**, so `startTime=0` (epoch) is accepted. [8](#0-7) 

**Exploit flow:**

1. Attacker opens *C* TCP connections to port 5600.
2. On each connection, opens 5 `subscribeTopic` streams with `consensusStartTime={seconds:0, nanos:0}`, no `consensusEndTime`, no `limit`.
3. For a topic with no (or few) historical messages, `retrieve(filter, true)` completes in milliseconds.
4. `Flux.merge(safetyCheck, live)` is subscribed; the 1-second `Mono.delay` starts for all *5C* subscriptions at approximately the same instant.
5. During a network partition the live Redis/poll listener delivers nothing; `topicContext.getLast()` remains `null`.
6. At T+1s all *5C* safety-checks fire simultaneously, each calling `retrieve(gapFilter=startTime:0, throttled:false)`.
7. Each unthrottled retrieve executes up to 12 DB queries × 5,000 rows = 60,000 rows scanned per subscription.
8. Total simultaneous DB queries: `5C × 12`. For *C=200* connections → **12,000 DB queries in ~240 ms**.

### Impact Explanation

The coordinated burst of unthrottled DB queries exhausts the database connection pool and CPU. The `TopicMessageRepositoryCustomImpl.findByFilter()` query scans from `consensusTimestamp >= 0` with no upper bound, touching the full `topic_message` table for every call. [9](#0-8) 

Result: database unavailability for all mirror-node services (REST API, other gRPC clients, importer), constituting a full denial-of-service. Severity: **High**.

### Likelihood Explanation

No authentication is required; the gRPC port (5600) is publicly exposed. Opening hundreds of HTTP/2 connections is trivial from a single host or a small botnet. The attack is deterministic and repeatable: the 1-second timer guarantees synchronised firing. The network-partition condition (Redis outage, importer lag, or deliberate disruption) is a realistic operational scenario that makes the safety-check fire reliably.

### Recommendation

1. **Add a global subscriber cap**: check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if the limit is exceeded.
2. **Add per-IP connection/subscription rate limiting** at the Netty or load-balancer layer.
3. **Stagger or debounce the safety-check**: replace the fixed 1-second `Mono.delay` with a jittered delay (e.g., `1s + random(0..2s)`) to prevent thundering-herd synchronisation.
4. **Throttle the safety-check retrieve**: pass `throttled=true` instead of `false` in the safety-check branch of `missingMessages()`, or enforce a minimum inter-query interval per subscription.
5. **Bound `startTime`**: reject subscriptions whose `startTime` is older than a configurable maximum lookback window.

### Proof of Concept

```python
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 12345          # any valid topic with few/no messages
CONNECTIONS = 200          # 200 connections × 5 streams = 1000 subscriptions

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = cs.ConsensusServiceStub(channel)
    query = cs_pb.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no endTime, no limit → isComplete() always False
    )
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    # Keep streams open; safety-check fires at T+1s after historical drains
    time.sleep(30)

threads = [threading.Thread(target=open_subscriptions, args=(i,))
           for i in range(CONNECTIONS)]
for t in threads: t.start()
# At T+1s: 1000 simultaneous unthrottled retrieve(startTime=0, false) calls hit the DB
for t in threads: t.join()
```
