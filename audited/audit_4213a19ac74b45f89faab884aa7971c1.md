### Title
Unprivileged UNION ALL Amplification via Dual-Range Filter on Token Allowance Endpoint

### Summary
An unauthenticated external user can craft a single HTTP request to `GET /api/v1/accounts/{id}/allowances/tokens` that forces `getQuery()` in `tokenAllowanceService.js` to execute all three subqueries joined by `UNION ALL`, causing the database to perform three independent index scans on `token_allowance` each bounded by `LIMIT $2` (up to 100 rows), then merge and sort up to 300 rows. This is approximately 3× the work of a baseline single-subquery request. No authentication, no brute force, and no rate limiting on the REST API path is required.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenAllowanceController.js` lines 22–59, `extractTokenMultiUnionQuery()` parses user-supplied `spender.id` and `token.id` query parameters into `bounds.primary` and `bounds.secondary`. It then calls `getLowerFilters(bounds)`, `getInnerFilters(bounds)`, and `getUpperFilters(bounds)` (inherited from `rest/controllers/baseController.js` lines 131–183) to populate `lower`, `inner`, and `upper` filter arrays.

In `rest/service/tokenAllowanceService.js` lines 63–74, `getQuery()` does:
```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

Each subquery independently contains `ORDER BY spender asc, token_id asc LIMIT $2`, where `$2` is the user-controlled limit (capped at 100 for unauthenticated users per `rest/utils.js` lines 544–553).

**Root cause:** `validateBounds()` in `baseController.js` lines 56–123 only rejects logically invalid combinations (range+equal, multiple equal, secondary without primary, etc.). It does not prevent the valid three-subquery case. When a user supplies both a lower bound (`gte`) and an upper bound (`lte`) on `spender.id` AND both a lower bound and upper bound on `token.id`, all three of `getLowerFilters`, `getInnerFilters`, and `getUpperFilters` return non-empty arrays, triggering the UNION ALL branch with all three subqueries.

**Trigger condition** (from `baseController.js` `getInnerFilters` lines 154–168 and `getUpperFilters` lines 176–183): `inner` is non-empty when `primary.hasBound() && secondary.hasBound()`; `upper` is non-empty when `primary.hasUpper() && secondary.hasUpper()`. Both conditions are satisfied by the crafted request.

**No rate limiting** exists on the REST API. The throttle configuration (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the web3/contract-call service, not to the Node.js REST API. The `authHandler` middleware (`rest/middleware/authHandler.js`) is optional and only sets a custom limit for authenticated users; unauthenticated requests proceed freely.

### Impact Explanation

Each crafted request causes the PostgreSQL engine to execute three independent index scans on `token_allowance` (filtered by `owner`, `amount > 0`, and spender/token range conditions), each returning up to 100 rows, followed by an outer merge-sort of up to 300 rows. A baseline request (no range filters) executes one scan returning up to 100 rows. The amplification factor is ~3×, well above the 30% threshold. An attacker targeting a busy node can sustain elevated DB CPU and I/O by continuously issuing these requests, degrading service for all users. The `token_allowance` table can be large on mainnet (millions of rows), making each scan non-trivial.

### Likelihood Explanation

The attack requires zero privileges — the endpoint is public. The crafted URL is trivial to construct and requires only knowledge of the API's documented query parameters (`spender.id`, `token.id`). The attack is fully repeatable and scriptable. No special account state is needed; any valid account ID (including one with zero allowances) will trigger the three-subquery path. The absence of REST-layer rate limiting means the attacker can sustain the load indefinitely.

### Recommendation

1. **Add a per-IP or global rate limit** to the REST API (e.g., via an Express rate-limiting middleware such as `express-rate-limit`) specifically for the token allowance endpoint.
2. **Count active subqueries and apply a cost multiplier** to the effective limit: if all three subqueries are active, reduce the per-subquery `LIMIT` so that total rows processed stays bounded (e.g., `Math.ceil(limit / activeSubQueryCount)`).
3. **Add a database statement timeout** for the REST API database role (analogous to `restJava.db.statementTimeout`) to cap worst-case query duration.
4. **Consider caching** responses for identical query parameter sets using the existing Redis cache layer.

### Proof of Concept

```

```

This maps to:
- `bounds.primary` = `{lower: gte:1, upper: lte:999999999}` → `primary.hasLower() && primary.hasUpper()` = true
- `bounds.secondary` = `{lower: gte:1, upper: lte:999999999}` → `secondary.hasLower() && secondary.hasUpper()` = true

    const bounds = {
      primary: new Bound(filterKeys.SPENDER_ID, 'spender'),
      secondary: new Bound(filterKeys.TOKEN_ID, 'token_id'),
    };
    let limit = defaultLimit;
    let order = orderFilterValues.ASC;

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.SPENDER_ID:
          bounds.primary.parse(filter);
          break;
        case filterKeys.TOKEN_ID:
          bounds.secondary.parse(filter);
          break;
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        default:
          break;
      }
    }

    this.validateBounds(bounds);

    return {
      bounds,
      lower: this.getLowerFilters(bounds),
      inner: this.getInnerFilters(bounds),
      upper: this.getUpperFilters(bounds),
      order,
      ownerAccountId,
      limit,
    };

  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
