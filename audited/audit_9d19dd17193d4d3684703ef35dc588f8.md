### Title
Unbounded DB Query Per Request via Opaque EVM Address in `getEncodedId()` Enables Unauthenticated Non-Network DoS

### Summary
`getEncodedId()` in `rest/service/entityService.js` unconditionally issues a live database query for every request containing an opaque EVM address (any 40-hex-char address whose first 12 bytes are non-zero). The in-process LRU cache only stores the parsed `EntityId` object, not the DB lookup result, and the response cache explicitly skips 404 responses. An unauthenticated attacker can exhaust the database connection pool by flooding any of the multiple public endpoints that call `getEncodedId()` with a stream of unique opaque EVM addresses.

### Finding Description

**Step 1 — Opaque address classification (`rest/entityId.js` lines 250–253):**

```js
let [prefix, num] = parseFromEvmAddress(numOrEvmAddress);
if (prefix !== longFormEvmAddressPrefix || num > maxNum) {
  return [shard, realm, null, numOrEvmAddress]; // Opaque EVM address
}
``` [1](#0-0) 

Any 40-hex-char string whose first 24 hex digits are not all-zero is classified as an opaque address. `num` is set to `null` and `evmAddress` is set to the raw hex string.

**Step 2 — Mandatory DB round-trip (`rest/service/entityService.js` lines 122–124):**

```js
return entityId.evmAddress === null
  ? entityId.getEncodedId()
  : await this.getEntityIdFromEvmAddress(entityId, requireResult);
``` [2](#0-1) 

Because `evmAddress` is non-null for every opaque address, the local encoding path is bypassed and `getEntityIdFromEvmAddress()` is always called.

**Step 3 — No caching of the DB result (`rest/service/entityService.js` lines 90–104):**

```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
    [Buffer.from(entityId.evmAddress, 'hex')]);
  ...
}
``` [3](#0-2) 

There is no memoization here. The `parseCached` LRU in `entityId.js` only caches the `EntityId` object (the parse result), not the DB lookup outcome. [4](#0-3) 

**Step 4 — Response cache skips 404s (`rest/middleware/responseCacheHandler.js` line 95):**

```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
``` [5](#0-4) 

Non-existent opaque addresses return HTTP 404, which is never stored in Redis. Repeated requests for the same non-existent address each trigger a fresh DB query.

**Step 5 — Multiple unauthenticated entry points:**

`getEncodedId()` is called directly from at least five public path-parameter handlers: [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9) 

No authentication or rate-limiting middleware is present in the middleware stack. [11](#0-10) 

### Impact Explanation
Each crafted request forces one `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query against the database. With a high-concurrency flood of unique opaque addresses the database connection pool is saturated, causing all other API requests (including legitimate ones) to queue and time out. Because the mirror node serves consensus data relied upon by wallets, explorers, and DeFi integrations, sustained unavailability constitutes a high-impact non-network DoS against the data availability layer of the network.

### Likelihood Explanation
The attack requires zero privileges, zero tokens, and zero on-chain interaction. Any HTTP client can generate an unlimited supply of valid opaque EVM addresses (e.g., increment the last byte of `ff000000000000000000000000000000000000XX`). The attack is trivially scriptable, repeatable, and amplifiable across multiple endpoints simultaneously. No existing control (parse cache, response cache, auth middleware) blocks or throttles it.

### Recommendation
1. **Cache DB lookup results** — after resolving an opaque EVM address to an entity ID (or confirming it does not exist), store the result in the existing `quickLru` cache keyed on the hex address. A short TTL (e.g., 5–30 s) is sufficient to collapse burst floods.
2. **Cache negative results** — explicitly cache `null` for addresses that resolve to no entity, so repeated 404-producing requests do not re-query the DB.
3. **Add request-rate limiting** — apply a per-IP rate limit on path-parameter endpoints at the middleware layer.
4. **Reject non-existent opaque addresses early** — consider a bloom filter or in-memory negative cache for recently queried addresses.

### Proof of Concept

```bash
# Generate 10 000 unique opaque EVM addresses and flood the accounts endpoint.
# Each request forces one DB query; no authentication required.

for i in $(seq 1 10000); do
  ADDR=$(printf 'ff%038x' $i)   # non-zero prefix → always opaque
  curl -s "http://<mirror-node>/api/v1/accounts/${ADDR}/nfts" &
done
wait
```

Preconditions: network access to the REST API port; no credentials needed.
Trigger: each `GET /api/v1/accounts/<opaque-evm-address>/nfts` call reaches `getEncodedId()` → `getEntityIdFromEvmAddress()` → one live DB query.
Result: DB connection pool exhaustion; legitimate requests begin timing out; mirror node becomes unavailable.
