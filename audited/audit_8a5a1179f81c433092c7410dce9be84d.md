### Title
Unauthenticated Subscription Churn DoS via Crafted Past-Window `ConsensusTopicQuery`

### Summary
An unprivileged external attacker can send a `ConsensusTopicQuery` with `consensusStartTime` set to a recent past timestamp and `consensusEndTime` set to `consensusStartTime + 1 nanosecond` (also in the past). Both bean-validation checks pass, a full subscription pipeline is assembled, two database queries are issued, and the stream immediately completes. By repeating this at high rate across many connections, the attacker exhausts the bounded-elastic scheduler thread pool and the database connection pool, denying service to legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` converts the protobuf timestamps without any minimum-window or recency check: [1](#0-0) 

The resulting `TopicMessageFilter` is validated by two `@AssertTrue` methods: [2](#0-1) 

`isValidEndTime()` only requires `endTime > startTime` (by any positive amount, including 1 ns). `isValidStartTime()` only requires `startTime <= now()`. A filter with `startTime = now - 1s` and `endTime = now - 1s + 1ns` satisfies both.

**Why the subscription immediately completes:**

`TopicContext.isComplete()` compares `filter.getEndTime()` against `startTime`, which is `DomainUtils.now()` captured at subscription creation time: [3](#0-2) 

Since `filter.getEndTime()` (a past timestamp) is less than `startTime` (current time), `isComplete()` returns `true` immediately. `pastEndTime()` then emits a completion signal on the first evaluation: [4](#0-3) 

`takeUntilOther(pastEndTime(topicContext))` terminates the main flux immediately.

**Resources consumed before termination:**

Before the stream terminates, `subscribeTopic()` unconditionally:
1. Issues a DB query via `entityRepository.findById()` (entity existence check)
2. Assembles and subscribes `topicMessageRetriever.retrieve(filter, true)` — which calls `topicMessageRepository.findByFilter()` on a `Schedulers.boundedElastic()` thread
3. Schedules a `safetyCheck` `Mono.delay(1s)` on `Schedulers.boundedElastic()` [5](#0-4) 

`PollingTopicMessageRetriever` uses its own dedicated `Schedulers.boundedElastic()` instance: [6](#0-5) 

**Missing rate-limiting:**

The only per-connection concurrency control is `maxConcurrentCallsPerConnection = 5`: [7](#0-6) [8](#0-7) 

There is no limit on the number of connections, no per-IP rate limit, no global subscription cap (only a metrics gauge), and no minimum time-window requirement. The `ThrottleConfiguration` with bucket4j rate limiting exists only in the `web3` module, not in the `grpc` module. [9](#0-8) 

### Impact Explanation

Each crafted request triggers at minimum two synchronous database queries and occupies a `boundedElastic` scheduler thread for the duration of the DB call. At high request rates (thousands/second across many connections), this exhausts the bounded-elastic thread pool, starves the database connection pool, and causes legitimate long-lived subscriptions to time out or fail. The `subscriberCount` gauge provides no enforcement — it is observability-only. The mirror node's gRPC service is the primary interface for HCS topic consumers; its unavailability prevents clients from receiving consensus messages.

### Likelihood Explanation

The attack requires zero authentication, zero on-chain cost, and no special knowledge beyond the public protobuf schema. A single attacker machine can open hundreds of TCP connections (each limited to 5 concurrent streams = 500+ concurrent in-flight subscriptions), cycling through them faster than the server can clean up. The pattern is trivially scriptable with any gRPC client library (e.g., `grpcurl` in a loop, or a custom Go/Java client). The exploit is repeatable and stateless — no session or token is needed between iterations.

### Recommendation

1. **Enforce a minimum time-window**: Add a validation constraint in `TopicMessageFilter.isValidEndTime()` requiring `endTime - startTime >= some_minimum` (e.g., 1 second), rejecting trivially short windows.
2. **Add per-IP or global subscription rate limiting** on the gRPC `subscribeTopic` endpoint, analogous to the bucket4j throttle in the `web3` module.
3. **Enforce a global concurrent-subscription cap**: Convert `subscriberCount` from a gauge to an enforced limit; reject new subscriptions when the cap is reached.
4. **Limit total inbound connections** via Netty's `maxConnectionAge` / `maxConnections` settings in `GrpcConfiguration`.

### Proof of Concept

```
# Set startTime = 10 seconds ago, endTime = startTime + 1 nanosecond (both in the past)
# Repeat in a tight loop from multiple clients

START_SECONDS=$(( $(date +%s) - 10 ))

while true; do
  grpcurl -plaintext \
    -d "{
      \"topicID\": {\"topicNum\": 1234},
      \"consensusStartTime\": {\"seconds\": ${START_SECONDS}, \"nanos\": 0},
      \"consensusEndTime\":   {\"seconds\": ${START_SECONDS}, \"nanos\": 1}
    }" \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
```

Each iteration passes both `isValidStartTime()` and `isValidEndTime()`, triggers a full subscription pipeline with DB queries, and immediately completes. Running this from N clients simultaneously saturates the `boundedElastic` scheduler and database connection pool, causing `DEADLINE_EXCEEDED` or `RESOURCE_EXHAUSTED` errors for legitimate subscribers.

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
```

```java
    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
```

```java
    public PollingTopicMessageRetriever(
            ObservationRegistry observationRegistry,
            RetrieverProperties retrieverProperties,
            TopicMessageRepository topicMessageRepository) {
        this.observationRegistry = observationRegistry;
        this.retrieverProperties = retrieverProperties;
        this.topicMessageRepository = topicMessageRepository;
        scheduler = Schedulers.boundedElastic();
    }

    @Override
    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
```
