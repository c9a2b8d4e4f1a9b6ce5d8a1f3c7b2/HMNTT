### Title
Cache Thrashing via LRU Eviction in `getCachedTokens()` Enables Unauthenticated DB Load Amplification

### Summary
The `tokenCache` in `rest/service/tokenService.js` is a `quickLru` instance with no `maxAge` (entries never expire, only evicted by LRU overflow). There is no rate limiting on the REST API for unauthenticated users. An attacker can continuously cycle requests through accounts holding disjoint sets of token IDs totaling more than `config.cache.token.maxSize` (default 100,000), causing the cache to continuously evict legitimate entries and force repeated database queries for every subsequent request touching those evicted token IDs.

### Finding Description

**Exact code location:** `rest/service/tokenService.js`, lines 12–14 (cache construction) and lines 136–161 (`getCachedTokens`).

**Cache construction — no TTL:** [1](#0-0) 

The `quickLru` instance is created with only `maxSize` and no `maxAge`. Entries are therefore immortal until evicted by LRU overflow. Once the cache reaches 100,000 entries, every new `tokenCache.set()` silently evicts the least-recently-used entry.

**Cache-miss path — unconditional DB query:** [2](#0-1) 

Any token ID not found in the cache triggers a `SELECT … WHERE token_id = ANY($1)` database round-trip. The result is then written back into the cache via `tokenCache.set()`, which itself may evict another entry.

**Attack trigger surface — `getTokenAccounts`:** [3](#0-2) 

`getCachedTokens` is called with token IDs derived from the `token_account` table for whatever account the caller queries. The caller controls which account is queried; the token IDs for that account are fixed in the DB but are fully attacker-observable (public blockchain).

**No rate limiting exists for unauthenticated callers.** The `authHandler` only sets a custom *response-size* limit for authenticated users; it does not throttle request frequency: [4](#0-3) 

The DB connection pool is capped at 10 connections by default: [5](#0-4) 

**Exploit flow:**

1. Attacker enumerates accounts on Hedera mainnet (public data) that collectively hold >100,000 distinct token IDs. Hedera mainnet has millions of tokens; no token creation cost is required.
2. Attacker sends rapid unauthenticated GET requests to `/api/v1/accounts/{id}/tokens` cycling through those accounts in a round-robin pattern.
3. Each request causes `getCachedTokens` to be called with up to 100 token IDs (the max response limit).
4. After ~1,000 requests the 100,000-entry cache is full. Subsequent requests for accounts whose token IDs were evicted cause cache misses and new DB queries.
5. By continuously cycling through >100,000 unique token IDs the attacker keeps the cache in a permanent thrashing state: every request for a "hot" account now hits the DB instead of the cache.

**Why existing checks are insufficient:**

- `putTokenCache` guards against overwriting existing entries but does not prevent LRU eviction caused by `getCachedTokens` writing new entries: [6](#0-5) 
- There is no `maxAge` to bound how long an entry stays in the cache independently of eviction pressure.
- There is no per-IP or global request-rate limit in the REST middleware stack.

### Impact Explanation

Every cache miss forces a synchronous DB query against the shared 10-connection pool. Under sustained thrashing, legitimate user requests that previously resolved entirely from cache now compete for DB connections, increasing query latency and connection-pool saturation. At high enough request rates this degrades response times for all users of the mirror node REST API. The DB `statementTimeout` of 20,000 ms means slow queries can hold connections for up to 20 seconds, compounding the effect. [7](#0-6) 

### Likelihood Explanation

The attack requires no credentials, no token creation, and no economic expenditure — only knowledge of public account IDs and their token holdings, which are freely queryable from the same API. The attacker needs a single HTTP client capable of sustaining a few hundred requests per second. Hedera mainnet already has millions of tokens distributed across many accounts, making the 100,000-entry threshold easily exceeded with existing on-chain data.

### Recommendation

1. **Add `maxAge`** to the `tokenCache` constructor so entries expire independently of eviction pressure, bounding the staleness window and naturally refreshing the working set.
2. **Add request-rate limiting** (e.g., via `express-rate-limit`) for unauthenticated callers at the middleware layer, applied before route handlers.
3. **Consider a two-tier cache** (small hot-set + larger cold-set with TTL) or a probabilistic admission policy to resist deliberate eviction attacks.

### Proof of Concept

```
# Step 1: collect >100,000 distinct token IDs held by different accounts
# (all public data, no credentials needed)
for ACCOUNT in $(cat accounts.txt); do
  curl -s "https://mirror-node/api/v1/accounts/$ACCOUNT/tokens?limit=100" \
    | jq -r '.tokens[].token_id' >> token_ids.txt
done

# Step 2: flood the API cycling through accounts, keeping the cache thrashing
while true; do
  for ACCOUNT in $(cat accounts.txt); do
    curl -s "https://mirror-node/api/v1/accounts/$ACCOUNT/tokens?limit=100" &
  done
  wait
done
```

After the working set exceeds 100,000 unique token IDs, every subsequent request for a previously-cached account will miss the cache and hit the database, observable as increased DB query latency and connection-pool utilization in the `/swagger/metrics/` endpoint.
