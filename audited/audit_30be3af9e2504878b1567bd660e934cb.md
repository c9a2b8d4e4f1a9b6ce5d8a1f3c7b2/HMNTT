### Title
Unauthenticated DB Saturation via `findStorageBatch()` Amplification Through Global 500 RPS Throttle

### Summary
The global 500 RPS throttle in `ThrottleManagerImpl` applies no per-user or per-IP limiting, allowing a single unauthenticated attacker to consume the entire request budget. Each request that misses the `contractStateCache` (TTL: 2 seconds) unconditionally calls `findStorageBatch()`, which issues an uncached `SELECT ... WHERE slot IN (N slots)` DB query. By targeting non-existent or freshly-expired slots, an attacker can sustain 500 `findStorageBatch()` DB queries per second with growing `IN` clause sizes, saturating the database and degrading service for all nodes.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.throttle()` enforces a single shared `rateLimitBucket` with capacity 500: [1](#0-0) 

This bucket is global — no per-IP or per-user partitioning exists. One attacker can consume all 500 tokens/second.

`findStorage()` checks `contractStateCache` (configured `expireAfterWrite=2s, maximumSize=25000`): [2](#0-1) 

On any cache miss, `findStorageBatch()` is called unconditionally. Inside `findStorageBatch()`, the new slot key is registered in `contractSlotsCache` (TTL: `expireAfterAccess=5m`, max 1500 slots per contract), and then a DB query is issued against **all accumulated slot keys** for that contract: [3](#0-2) 

`findStorageBatch()` carries **no `@Cacheable` annotation** — every invocation hits the database directly: [4](#0-3) 

**Root cause — two compounding flaws:**

1. **Non-existent slots are never written to `contractStateCache`** (only found slots are stored at line 109), so they produce a permanent cache miss on every subsequent call to `findStorage()` for that contract: [5](#0-4) 

2. **`contractStateCache` TTL is only 2 seconds** — even legitimately cached slots expire and re-trigger `findStorageBatch()` every 2 seconds: [6](#0-5) 

**Exploit flow:**

- **Phase 1 (slot poisoning):** Attacker sends ~1500 requests targeting unique non-existent slot keys for a single contract. Each request passes the global throttle, misses `contractStateCache`, calls `findStorageBatch()`, and registers the non-existent key in `contractSlotsCache` (5-minute TTL). These slots are never written to `contractStateCache` because they don't exist in the DB.

- **Phase 2 (sustained amplification):** Now any request for any slot of that contract calls `findStorageBatch()` with up to 1500 slots in the `IN` clause. The `contractStateCache` 2-second TTL means even real slots re-trigger the batch query constantly. At 500 RPS (the global ceiling), this produces 500 × `SELECT ... WHERE slot IN (1500 slots)` queries per second against the database.

- **No synchronization or coalescing** exists between concurrent `findStorageBatch()` calls for the same contract — all 500 concurrent requests issue independent DB queries simultaneously.

### Impact Explanation

500 uncached `SELECT ... WHERE slot IN (1500 slots)` queries per second against the `contract_state` table will exhaust the DB connection pool and CPU. The `db.statementTimeout` of 3000ms means each query holds a connection for up to 3 seconds, causing connection pool exhaustion and cascading timeouts across all web3 API nodes. This degrades or halts EVM contract call processing for all users, meeting the ≥30% node degradation threshold. The attack is sustained as long as the attacker maintains 500 RPS, which is trivially achievable from a single machine. [7](#0-6) 

### Likelihood Explanation

No authentication, API key, or account is required. The attacker only needs HTTP access to the `/api/v1/contracts/call` endpoint. Sending 500 HTTP requests/second is achievable with standard load-testing tools (e.g., `wrk`, `k6`, `hey`) from a single machine with a modest network connection. The attack is repeatable indefinitely because the `contractSlotsCache` TTL is 5 minutes and non-existent slots are never evicted by successful DB lookups. The attacker does not need to know valid slot keys — any 32-byte values work as non-existent slots. [8](#0-7) 

### Recommendation

1. **Add per-IP/per-client rate limiting** upstream (e.g., at the ingress/load-balancer layer or via a `HandlerInterceptor`) so no single source can consume the full 500 RPS budget.

2. **Cache negative results** (non-existent slots) in `contractStateCache` with a short TTL (e.g., 1–2 seconds) so repeated lookups for the same non-existent slot do not re-trigger `findStorageBatch()`. The current check `cachedValue != null && cachedValue != EMPTY_VALUE` at line 65 explicitly skips the early-return for `EMPTY_VALUE`, but `EMPTY_VALUE` is never stored in `contractStateCache` for missing slots — storing a sentinel there would close this gap. [9](#0-8) 

3. **Coalesce concurrent `findStorageBatch()` calls** for the same contract using a per-contract lock or a request-deduplication mechanism (e.g., Caffeine's `AsyncLoadingCache`) to prevent N concurrent threads from issuing N identical DB queries simultaneously.

4. **Increase `contractStateCache` TTL** beyond 2 seconds or make it configurable with a higher default to reduce the re-query frequency for legitimate slots. [6](#0-5) 

### Proof of Concept

```bash
# Step 1: Pre-populate contractSlotsCache with 1500 non-existent slots for contract 0x<TARGET>
# Each request uses a unique calldata that maps to a different storage slot (e.g., SLOAD of slot i)
for i in $(seq 1 1500); do
  SLOT=$(printf '%064x' $i)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x<TARGET>\",\"data\":\"0x<SLOAD_SLOT_$SLOT>\",\"gas\":50000}" &
done
wait

# Step 2: Sustain 500 RPS — each request now triggers findStorageBatch() with 1500 slots
# Use k6, wrk, or hey:
hey -n 100000 -c 500 -q 500 \
  -m POST \
  -H "Content-Type: application/json" \
  -d '{"to":"0x<TARGET>","data":"0x<ANY_SLOAD_CALLDATA>","gas":50000}' \
  http://<mirror-node>/api/v1/contracts/call

# Expected result:
# DB CPU spikes to 100%, connection pool exhausted,
# web3 API returns 503/timeout for all users within seconds.
# Monitor with: SELECT count(*), state FROM pg_stat_activity GROUP BY state;
```
