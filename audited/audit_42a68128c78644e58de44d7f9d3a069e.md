### Title
Unauthenticated Timestamp Filter Triggers Multi-Query DB Connection Amplification Leading to Connection Pool Exhaustion DoS

### Summary
Any unauthenticated user can send `GET /api/v1/accounts/:id?timestamp=lte:1` with a timestamp so old that no balance snapshot exists, forcing `accountBalanceQuery.forceUnionEntityHistory=true` and triggering a costly `entity UNION ALL entity_history` query concurrently with `doGetTransactions()`. Each such request consumes 3–4 database connections from the shared pool sequentially and concurrently, with no rate limiting in place. Flooding the endpoint with concurrent requests exhausts the connection pool and renders the entire REST API unavailable.

### Finding Description

**Code path in `rest/accounts.js`, function `getOneAccount()` (lines 396–495):**

**Step 1 — Sequential DB query 1 (balance snapshot lookup):**
At line 435, `await balances.getAccountBalanceTimestampRange(...)` issues a `pool.queryQuietly()` call in `rest/balances.js:189`. When the supplied timestamp predates all balance snapshots (e.g., `timestamp=lte:1`), `rows.length === 0` and the function returns `{}` (empty object, `upper === undefined`).

**Step 2 — `forceUnionEntityHistory` set:**
At line 463, the `else` branch sets `accountBalanceQuery.forceUnionEntityHistory = true`. This causes `getAccountQuery()` (lines 204–219) to build an `entityTable` as a full `UNION ALL` between `entity` and `entity_history`, a significantly more expensive query than a simple `entity` lookup.

**Step 3 — Two concurrent DB operations launched:**
- Line 487: `entityPromise = pool.queryQuietly(entityQuery, entityParams)` — the expensive UNION ALL query is started immediately (DB connection 2 acquired).
- Lines 491–493: `transactionsPromise = transactions.doGetTransactions(filters, req, timestampRange)` — this async function is called without `await`, so it starts concurrently. Inside `doGetTransactions` → `getTransactionTimestamps` → `pool.queryQuietly(query, params)` at `rest/transactions.js:483` (DB connection 3 acquired).
- Line 495: `await Promise.all([entityPromise, transactionsPromise])` — both run concurrently.

**Step 4 — Potential 4th DB query:**
After `getTransactionTimestamps` resolves, `doGetTransactions` calls `cache.get(payerAndTimestamps, loader, keyMapper)` at line 705. On a cache miss, `loader` calls `getTransactionsDetails` → `pool.queryQuietly(query, params)` at `rest/transactions.js:662` (DB connection 4).

**Root cause:** No per-IP rate limiting, no request concurrency cap, and no guard against the `forceUnionEntityHistory` path being triggered repeatedly. The pool configuration (`rest/dbpool.js:14`, `rest/config.js:139`) sets a finite `maxConnections`. The `connectionTimeoutMillis` and `statement_timeout` settings limit individual query duration but do not prevent pool exhaustion under concurrent load.

**Why existing checks fail:**
- `statement_timeout` limits how long a single query runs, not how many connections are held simultaneously.
- `connectionTimeoutMillis` causes new requests to fail fast once the pool is full, but does not prevent the pool from being filled.
- No middleware-level rate limiting was found in the REST layer (`grep` for `rateLimit`, `throttle`, `concurrency` returned no hits in `rest/` handler code).

### Impact Explanation
When the pool is exhausted, every subsequent request to any endpoint that issues a DB query will fail with a connection timeout error. This is a full denial-of-service of the mirror node REST API — not limited to the `/accounts` endpoint. The UNION ALL query against both `entity` and `entity_history` tables is more expensive than a normal entity lookup, amplifying the per-request DB load and reducing the number of concurrent attackers needed to exhaust the pool.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero prior account state. The trigger condition (`timestamp=lte:1`) is trivially constructed. The attack is repeatable and scriptable. A single attacker with a modest number of concurrent HTTP connections (e.g., matching or slightly exceeding `maxConnections / 3`) can sustain pool exhaustion indefinitely. The attack is also low-cost to the attacker since each request is a simple GET with no body.

### Recommendation
1. **Add rate limiting** at the HTTP layer (e.g., per-IP request rate limit using middleware such as `express-rate-limit`) before requests reach handler logic.
2. **Add a per-request DB connection budget**: consider using a semaphore or queue to cap the number of concurrent DB-heavy requests.
3. **Short-circuit the `forceUnionEntityHistory` path**: when no balance snapshot is found for a very old timestamp, return a 404 or empty result immediately rather than issuing the expensive UNION ALL + transactions queries.
4. **Increase observability**: add metrics/alerts on pool wait queue depth to detect exhaustion attacks early.

### Proof of Concept

```bash
# Trigger forceUnionEntityHistory=true by using a timestamp that predates all balance snapshots
# Run N concurrent requests where N > maxConnections / 3

for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.3?timestamp=lte:1" &
done
wait

# Expected result: after pool exhaustion, all subsequent API calls return
# 500 errors or connection timeout errors, including unrelated endpoints
curl "http://<mirror-node-host>/api/v1/transactions"
# -> 500 Internal Server Error (pool exhausted)
```

```javascript
const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
```

```javascript
const getTransactionsDetails = async (payerAndTimestamps, order) => {
  if (payerAndTimestamps.length === 0) {
    return {rows: []};
  }

  const payerAccountIds = new Set();
  const timestamps = [];
  payerAndTimestamps.forEach((row) => {
    timestamps.push(row.consensus_timestamp);
    payerAccountIds.add(row.payer_account_id);
  });

  const params = [];
  const payerAccountIdsCondition = getQueryWithEqualValues('payer_account_id', params, Array.from(payerAccountIds));
  const timestampsCondition = getQueryWithEqualValues('consensus_timestamp', params, timestamps);
  const outerPayerAccountIdsCondition = 't.' + payerAccountIdsCondition;
  const outerTimestampsCondition = 't.' + timestampsCondition;

  const query = `with c_list as (select consensus_timestamp,
                                        payer_account_id,
                                        ${cryptoTransferJsonAgg} as crypto_transfer_list
                                 from crypto_transfer
                                 where ${payerAccountIdsCondition}
                                   and ${timestampsCondition}
                                 group by consensus_timestamp, payer_account_id),
                      t_list as (select consensus_timestamp,
                                        payer_account_id,
                                        ${tokenTransferJsonAgg} as token_transfer_list
                                 from token_transfer
                                 where ${payerAccountIdsCondition}
                                   and ${timestampsCondition}
                                 group by consensus_timestamp, payer_account_id)

                        (select crypto_transfer_list
                         from c_list
                         where consensus_timestamp = t.consensus_timestamp
```

```javascript
const doGetTransactions = async (filters, req, timestampRange) => {
  const {
    limit,
    order,
    nextTimestamp,
    rows: payerAndTimestamps,
  } = await getTransactionTimestamps(filters, timestampRange);

  const loader = (keys) => getTransactionsDetails(keys, order).then((result) => formatTransactionRows(result.rows));

  const transactions = await cache.get(payerAndTimestamps, loader, keyMapper);

  const isEnd = transactions.length !== limit;
  const next = utils.getPaginationLink(
    req,
    isEnd && !nextTimestamp,
    {
      [constants.filterKeys.TIMESTAMP]: !isEnd
        ? transactions[transactions.length - 1]?.consensus_timestamp
        : nextTimestamp,
    },
    order
  );

  return {
    transactions,
    links: {next},
  };
};
```
