### Title
Unauthenticated, Rate-Limit-Free Account Staking Reward Enumeration via `/accounts/:id/rewards`

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` serves staking reward history — including exact HBAR amounts and timestamps — to any unauthenticated caller with no rate limiting enforced anywhere in the middleware stack. Because Hedera account IDs are sequential integers, an attacker can trivially iterate through the entire account space, identify which accounts have received staking rewards, and extract their cumulative reward amounts to build a target list for social engineering or phishing campaigns.

### Finding Description
**Code location:** `rest/controllers/accountController.js`, `listStakingRewardsByAccountId()`, lines 170–203.

The function resolves the account ID, checks existence via `EntityService.isValidAccount()`, then queries and returns full staking reward transfer history with no access control:

```js
// rest/controllers/accountController.js lines 170-186
listStakingRewardsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  const isValidAccount = await EntityService.isValidAccount(accountId);
  if (!isValidAccount) {
    throw new NotFoundError();   // 404 for non-existent accounts
  }
  // ... returns full reward history with amounts
};
```

**Middleware stack** (`rest/server.js` lines 67–98): `cors → express.json → compression → httpContext → requestLogger → authHandler → (optional metrics) → (optional cache) → routes`. There is no rate-limiting middleware at any layer.

**`authHandler` is not a gate** (`rest/middleware/authHandler.js` lines 15–19):
```js
const authHandler = async (req, res) => {
  const credentials = basicAuth(req);
  if (!credentials) {
    return;   // unauthenticated requests pass through unconditionally
  }
  ...
};
```
Authentication is entirely optional — requests without credentials are allowed through. No `express-rate-limit` or equivalent package appears anywhere in the REST service dependencies or middleware.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0.0.1/rewards` through `GET /api/v1/accounts/0.0.N/rewards` sequentially.
2. A `404` response means the account does not exist; a `200` response returns the full reward history including `amount` (in tinybars) and `timestamp` for every payout.
3. Attacker aggregates results to rank accounts by total staking rewards received, identifying high-balance stakers.
4. These accounts become targets for spear-phishing, SIM-swap, or social engineering attacks.

**Why the `isValidAccount` check is insufficient:** It is a correctness guard, not a security control. It distinguishes existing from non-existing accounts, which actually *aids* enumeration by providing a clean binary signal (200 vs 404).

### Impact Explanation
An attacker obtains a ranked list of Hedera accounts by staking reward income — a reliable proxy for account wealth — without any authentication or throttling. This financial intelligence directly enables targeted attacks against high-value holders. The data returned (`amount`, `timestamp` per reward event) is precise enough to estimate total staked balances. Severity: **Medium** (information disclosure enabling downstream social engineering; no direct fund theft from this endpoint alone).

### Likelihood Explanation
Preconditions: none. Any internet-connected client can perform this attack. Hedera account IDs are well-known sequential integers starting from low numbers, making full enumeration of the active account space feasible with a simple script in hours. The absence of rate limiting means there is no server-side friction. This is a low-skill, high-repeatability attack.

### Recommendation
1. **Add rate limiting** at the Express layer (e.g., `express-rate-limit`) applied globally or specifically to `/accounts/:id/rewards`, with a per-IP request budget (e.g., 60 req/min).
2. **Deploy an API gateway or reverse proxy** (e.g., nginx, Envoy, AWS API Gateway) with rate limiting and optionally IP-based throttling upstream of the Node.js process.
3. Consider whether staking reward amounts need to be returned in full detail to anonymous callers, or whether aggregated/redacted responses are sufficient for the public API use case.

### Proof of Concept
```bash
# Enumerate accounts 1 through 10000, collect those with rewards
for i in $(seq 1 10000); do
  RESP=$(curl -s -o /dev/null -w "%{http_code}" \
    "https://<mirror-node-host>/api/v1/accounts/0.0.$i/rewards")
  if [ "$RESP" = "200" ]; then
    # Fetch and store reward amounts
    curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.$i/rewards" \
      >> rewards_map.json
    echo "Account 0.0.$i has rewards"
  fi
done
# Sort rewards_map.json by amount to identify high-value targets
```
No authentication header is required. The loop runs without any server-side throttling, completing enumeration of tens of thousands of accounts in minutes. [1](#0-0) [2](#0-1) [3](#0-2)
