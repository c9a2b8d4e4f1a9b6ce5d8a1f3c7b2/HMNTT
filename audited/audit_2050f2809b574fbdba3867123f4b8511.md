### Title
Ghost NFT 206 Response via Token Deletion Timestamp Leakage in `getNftTransferHistoryRequest()`

### Summary
The timestamp-gathering query in `extractSqlFromNftTransferHistoryRequest()` unions the `entity` table for token-deletion events using only `id = $1 AND deleted is true`, with **no `serial_number` filter**. This means any valid deleted token will produce a non-empty timestamp result for *any* serial number — including ones that never existed. The subsequent `nftTransactionHistoryDetailsQuery` then synthesizes a `TOKENDELETION` nft_transfer entry for the attacker-supplied serial number, and the `NftService.getNft` null-check triggers a 206 Partial Content response, falsely implying the NFT existed and was partially recorded.

### Finding Description

**Step 1 — Timestamp query lacks serial_number guard for entity table**

In `extractSqlFromNftTransferHistoryRequest()`, the three-way UNION is:

```
nft          WHERE token_id = $1 AND serial_number = $2   ← serial-scoped
nft_history  WHERE token_id = $1 AND serial_number = $2   ← serial-scoped
entity       WHERE id = $1 AND deleted is true             ← NO serial_number filter
``` [1](#0-0) [2](#0-1) 

For a deleted token (e.g., `0.0.1500`), the `entity` branch returns the deletion timestamp for **every possible serial number**, including ones that were never minted.

**Step 2 — Early-exit check is bypassed**

```js
const {rows} = await pool.queryQuietly(timestampQuery, timestampParams);
if (rows.length === 0) {
  return;   // ← never reached; entity row is always present
}
``` [3](#0-2) 

**Step 3 — TOKENDELETION synthesizes a fake nft_transfer for the requested serial**

`nftTransactionHistoryDetailsQuery` special-cases transaction type 35:

```sql
when type = 35 then
    jsonb_build_array(jsonb_build_object(
        'is_approval', false,
        'receiver_account_id', null,
        'sender_account_id', null,
        'serial_number', $2::bigint,   ← attacker-controlled serial
        'token_id', $1::bigint))
``` [4](#0-3) 

The serial number injected here is whatever the attacker supplied — it is never validated against the `nft` or `nft_history` tables.

**Step 4 — 206 is set because `NftService.getNft` returns null**

```js
const nft = await NftService.getNft(tokenId, serialNumber);
if (nft?.createdTimestamp == null) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
}
``` [5](#0-4) 

`NftService.getNft` queries `SELECT * FROM nft WHERE token_id=$1 AND serial_number=$2`. [6](#0-5) 

For a never-minted serial, this returns `null`. In JavaScript `null?.createdTimestamp` evaluates to `undefined`, and `undefined == null` is `true`, so the 206 branch is taken unconditionally.

The test fixture `partial-data-missing-mint.json` confirms the 206 path is exercised and expected by the codebase when `created_timestamp` is null: [7](#0-6) 

### Impact Explanation

An unprivileged external user can query `GET /api/v1/tokens/{deletedTokenId}/nfts/{anySerial}/transactions` for any serial number (e.g., `999999`) that was never minted on a deleted token. The response is HTTP 206 with a `TOKENDELETION` transaction entry that includes `serial_number: 999999` and `token_id: <deleted token>`. Integrators and downstream systems that treat 206 as "partial history of a real NFT" will incorrectly conclude:

- The NFT with serial 999999 existed.
- It was deleted as part of a token deletion event.
- Ownership/provenance records for that serial are incomplete but real.

This can be exploited to fabricate apparent on-chain history for NFT serials that never existed, enabling fraud in systems that rely on the mirror node API for NFT provenance verification (e.g., marketplaces, custody systems, compliance tools).

### Likelihood Explanation

- **Precondition**: Any deleted NFT token on the network. Deleted tokens are common (token deletion is a standard operation).
- **Attacker capability**: Zero privileges required — this is a public read-only REST endpoint.
- **Repeatability**: Fully repeatable for any deleted token ID and any arbitrary serial number.
- **Detection difficulty**: The 206 status code is the *intended* signal for partial data, so monitoring systems are unlikely to flag it as anomalous.

### Recommendation

1. **Add a serial-number existence check before returning any data.** Before processing the timestamp rows, verify that the serial number has at least one record in `nft` or `nft_history`. If neither table has a row for `(token_id, serial_number)`, return HTTP 200 with an empty `transactions` array (or HTTP 404).

2. **Alternatively, exclude the entity-table branch when no NFT rows exist.** Only include the token-deletion timestamp in the UNION if `nft` or `nft_history` already has at least one row for the given `(token_id, serial_number)`. This preserves the intended "partial history" semantics for NFTs that genuinely existed but whose mint record is missing, while rejecting queries for serials that never existed.

3. **Tighten the 206 condition.** The check `nft?.createdTimestamp == null` fires for both "NFT exists but mint is missing" and "NFT never existed". These two cases should be distinguished: if `NftService.getNft` returns `null` (no row at all), return 404 or 200-empty rather than 206.

### Proof of Concept

**Preconditions:**
- Token `0.0.1500` exists in the mirror node database and has `deleted = true` in the `entity` table (e.g., `timestamp_range = [T_delete, )`).
- Serial number `99999` was **never minted** for token `0.0.1500` — no rows in `nft` or `nft_history` for `(token_id=1500, serial_number=99999)`.
- A `TOKENDELETION` transaction exists in the `transaction` table at timestamp `T_delete`.

**Request:**
```
GET /api/v1/tokens/0.0.1500/nfts/99999/transactions
```

**Expected (correct) response:** HTTP 200 `{"transactions": [], "links": {"next": null}}`

**Actual response:** HTTP 206
```json
{
  "transactions": [
    {
      "consensus_timestamp": "<T_delete>",
      "transaction_id": "...",
      "type": "TOKENDELETION",
      "is_approval": false,
      "receiver_account_id": null,
      "sender_account_id": null,
      "nonce": 0
    }
  ],
  "links": { "next": null }
}
```

The 206 status and the `TOKENDELETION` entry with `serial_number=99999` falsely imply the NFT existed and was destroyed, when it was never minted.
