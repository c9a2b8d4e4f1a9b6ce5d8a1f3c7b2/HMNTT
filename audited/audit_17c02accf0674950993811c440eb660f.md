### Title
DB Query Amplification via Unauthenticated Repeated SCHEDULECREATE Transaction ID Lookup

### Summary
`mayMissLongTermScheduledTransaction()` in `rest/transactions.js` returns `true` for any query against a successful, pending SCHEDULECREATE transaction ID when no `scheduled=false` filter is supplied. Because the REST API has no application-level rate limiting on the `/transactions/:transactionIdOrHash` endpoint, an unprivileged attacker who creates one long-expiry SCHEDULECREATE transaction can repeatedly query its ID to force two `pool.queryQuietly()` calls per request — the second with a significantly expanded timestamp window — doubling DB load indefinitely for the lifetime of the pending schedule.

### Finding Description

**Exact code path:**

`getTransactionsByIdOrHash()` (lines 923–938) calls `pool.queryQuietly()` once, then passes the result rows to `mayMissLongTermScheduledTransaction()`:

```
rest/transactions.js lines 930–938
``` [1](#0-0) 

`mayMissLongTermScheduledTransaction()` (lines 974–996) returns `true` when all three conditions hold:

1. `isTransactionHash === false` (query is by transaction ID, not hash)
2. `scheduled === undefined` (caller omitted the `scheduled` query parameter)
3. The first DB result contains a row with `type === 42` (SCHEDULECREATE) and `result` in `SUCCESS_PROTO_IDS`, **and** no row with `scheduled === true` [2](#0-1) 

When `true` is returned, the handler mutates `params[params.upperConsensusTimestampIndex]` to `lowerConsensusTimestamp + maxScheduledTransactionConsensusTimestampRangeNs` (up to 86 400 000 000 000 ns = 24 hours per the config test at line 217) and issues a second `pool.queryQuietly()` against that much wider range. [3](#0-2) 

**Root cause / failed assumption:** The design assumes that the "second query" path is rare and self-limiting (the scheduled transaction will eventually execute, making `scheduled === true` appear in results). It does not account for an attacker who deliberately keeps a long-expiry SCHEDULECREATE pending and floods the endpoint. There is no per-IP or per-transaction-ID rate limiting in the REST Node.js application. [4](#0-3) 

**Why the only guard is insufficient:** The sole escape hatch is `scheduled === false` in the query string (line 976). An attacker simply omits that parameter. The `SHORTER_CACHE_CONTROL_HEADER` (`max-age=5`) set by `getTransactionsByIdOrHashCacheControlHeader()` only helps if a caching reverse proxy is deployed; the application itself caches nothing and re-executes both queries on every cache miss. [5](#0-4) 

### Impact Explanation

Every HTTP request to `GET /api/v1/transactions/{schedulecreate-txid}` (without `?scheduled=false`) while the schedule is pending causes **two** database queries instead of one. The second query scans a timestamp range up to 24 hours wide — far larger than the normal `maxTransactionConsensusTimestampRangeNs` window — making it substantially heavier. Sustained flooding of this endpoint with even moderate concurrency can saturate the DB connection pool, increase query latency for all other API consumers, and degrade mirror-node availability. [6](#0-5) 

### Likelihood Explanation

Preconditions are minimal: the attacker needs one Hedera account (free to create on testnet, negligible cost on mainnet) and one SCHEDULECREATE transaction with a long expiry (Hedera allows up to 62 days). The SCHEDULECREATE fee is a few cents. After that, the attack is free and repeatable for the entire pending window. No authentication, no special privilege, and no on-chain interaction beyond the initial transaction is required. The REST API has no application-level rate limiting on this endpoint. [7](#0-6) 

### Recommendation

1. **Add application-level rate limiting** to the REST API's `/transactions/:transactionIdOrHash` handler (e.g., express-rate-limit keyed on IP or transaction ID).
2. **Cache the "second query not needed" result** per transaction ID for a short TTL (e.g., 5 s) so repeated requests for the same ID do not re-execute the expanded query.
3. **Short-circuit on repeated identical params**: before issuing the second query, check whether `params.upperConsensusTimestampIndex` already equals `lowerConsensusTimestamp + maxScheduledTransactionConsensusTimestampRangeNs` (i.e., the first query already used the full window).
4. Alternatively, require callers to explicitly pass `scheduled=true` to trigger the expanded-window lookup, removing the implicit second-query path entirely.

# Response contains type=SCHEDULECREATE, result=SUCCESS, scheduled=false → mayMissLongTermScheduledTransaction returns true

# Step 3 – Flood the endpoint (no scheduled=false filter)
while true; do
  curl -s "https://<mirror-node>/api/v1/transactions/0.0.12345-1700000000-000000000" -o /dev/null &
done
# Each request triggers pool.queryQuietly() twice; second query scans up to 24 h of data.
# DB connection pool exhaustion / latency spike observable within seconds at moderate concurrency.
```

const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
```
