### Title
Empty Schedule State Maps Cause `hasScheduleCapacity` to Always Return `true` in Contract Simulations, Producing Incorrect Schedule-Dependent Contract Behavior

### Summary
`MirrorNodeState.initKVStates()` registers `SCHEDULE_ID_BY_EQUALITY_STATE_ID`, `SCHEDULED_COUNTS_STATE_ID`, and `SCHEDULED_USAGES_STATE_ID` as permanently empty `MapReadableKVState` instances backed by empty `HashMap` objects, with no database-backed fallback. Any unprivileged user can trigger an `eth_call` or `eth_estimateGas` simulation against a contract that calls `hasScheduleCapacity` on the schedule system contract (`0x16b`), and the simulation will always return `true` regardless of actual network capacity. Contracts that gate scheduling logic on this check will always take the "has capacity" branch in simulation, diverging from real on-chain behavior when the network is at or near capacity.

### Finding Description
**Code location:** `web3/src/main/java/org/hiero/mirror/web3/state/MirrorNodeState.java`, `initKVStates()`, lines 159–167.

```java
final var defaultKvImplementations = Map.of(
        EVM_HOOK_STORAGE_STATE_ID, ContractService.NAME,
        EVM_HOOK_STATES_STATE_ID, ContractService.NAME,
        SCHEDULE_ID_BY_EQUALITY_STATE_ID, ScheduleService.NAME,   // always empty
        SCHEDULED_COUNTS_STATE_ID, ScheduleService.NAME,           // always empty
        SCHEDULED_USAGES_STATE_ID, ScheduleService.NAME);          // always empty
defaultKvImplementations.forEach(
        (stateId, serviceName) -> states.computeIfAbsent(serviceName, k -> new HashMap<>())
                .put(stateId, createMapReadableStateForId(serviceName, stateId)));
```

`createMapReadableStateForId` (line 174–176) creates a `MapReadableKVState` wrapping a plain `new HashMap<>()`. Unlike `SCHEDULES_BY_ID_STATE_ID`, which has a database-backed `ScheduleReadableKVState`, these three states have no database fallback and are permanently empty for the lifetime of the application.

**Root cause:** The schedule system contract's `hasScheduleCapacity` handler reads `SCHEDULED_COUNTS_STATE_ID` and `SCHEDULED_USAGES_STATE_ID` to determine whether a given second still has gas capacity for new scheduled calls. Because both maps are always empty, the handler sees zero existing counts and zero existing gas usage for every queried second, and therefore always concludes capacity is available. Similarly, `SCHEDULE_ID_BY_EQUALITY_STATE_ID` is used for duplicate schedule detection; because it is always empty, any `scheduleCreate` simulation that should return `IDENTICAL_SCHEDULE_ALREADY_CREATED` will instead succeed.

**Exploit flow:**
1. Attacker (or any user) deploys or targets a contract that calls `hasScheduleCapacity(expirySecond, gasLimit)` via `staticcall` to `0x16b`, then branches on the result (e.g., `HIP1215Contract.scheduleCallWithCapacityCheckAndDeleteExample`).
2. User submits an `eth_call` or `eth_estimateGas` request to the mirror node web3 endpoint targeting that contract function.
3. The EVM simulation reads `SCHEDULED_COUNTS_STATE_ID` and `SCHEDULED_USAGES_STATE_ID` — both empty — and returns `true` for `hasScheduleCapacity`.
4. The contract's `if (hasCapacity)` branch is taken; the simulation reports success and returns a schedule address.
5. On the real network, if the second is at capacity, the actual transaction reverts with a capacity error — the simulation result is incorrect.

### Impact Explanation
Any contract that uses `hasScheduleCapacity` as a guard before scheduling will always see `true` in mirror node simulations. This causes:
- **False positive simulation results**: `eth_call`/`eth_estimateGas` always reports the scheduling path succeeds, even when the real network would reject it due to capacity limits.
- **Incorrect gas estimates**: Gas estimation follows the "has capacity" branch, producing estimates that do not match the actual on-chain execution path (which may revert).
- **Duplicate schedule bypass in simulation**: `SCHEDULE_ID_BY_EQUALITY_STATE_ID` being empty means `scheduleCreate` simulations never detect existing identical schedules, so they always simulate as new creations rather than returning `IDENTICAL_SCHEDULE_ALREADY_CREATED`.

This is a simulation-layer divergence from real network state. No funds are directly at risk from the simulation itself, but developers and users relying on `eth_call`/`eth_estimateGas` to predict contract behavior will receive systematically incorrect results for all schedule-capacity-dependent logic.

### Likelihood Explanation
This is trivially triggerable by any unprivileged user with no special access. The `hasScheduleCapacity` function is a `view` function callable via `staticcall`, accessible to any contract. The `HIP1215Contract` pattern (`scheduleCallWithCapacityCheckAndDeleteExample`) is a documented and tested usage pattern. The empty state is a permanent structural condition — it is not a transient race condition. Any `eth_call` or `eth_estimateGas` against any deployed contract that uses `hasScheduleCapacity` will reproduce this behavior unconditionally.

### Recommendation
Implement a database-backed `ReadableKVState` for `SCHEDULED_COUNTS_STATE_ID` and `SCHEDULED_USAGES_STATE_ID` that queries the mirror node's persisted schedule data to compute actual per-second counts and gas usage, analogous to how `ScheduleReadableKVState` backs `SCHEDULES_BY_ID_STATE_ID`. Similarly, implement a database-backed state for `SCHEDULE_ID_BY_EQUALITY_STATE_ID` that looks up existing schedules by their equality key. Until these are implemented, the `hasScheduleCapacity` system contract call should either be explicitly unsupported (returning a defined error) or documented as always returning `true` in simulation context.

### Proof of Concept
1. Deploy `HIP1215Contract` (or any contract using `hasScheduleCapacity`) to a Hedera testnet with the mirror node web3 endpoint enabled.
2. Fill the schedule capacity for a specific `expirySecond` on the real network (submit enough `scheduleCall` transactions to exhaust the per-second gas limit).
3. Call `eth_call` on `hasScheduleCapacityExample(expiryShift)` targeting that same second via the mirror node web3 API.
4. **Observed**: The mirror node returns `true` (capacity available).
5. **Expected**: The mirror node should return `false` (capacity exhausted), matching real network state.
6. Additionally, call `eth_call` on `scheduleCallWithCapacityCheckAndDeleteExample(expiryShift)` for the same exhausted second — the simulation will succeed and return a schedule address, while the real transaction would revert with `"Failed to schedule. Has no capacity"`.
