### Title
Unbounded Historical Message Scan via Epoch `startTime=0` in `subscribeTopic()`

### Summary
An unprivileged gRPC subscriber can set `startTime=0` in a `TopicMessageFilter`, which passes all validation checks because `isValidStartTime()` only guards against future timestamps (`startTime <= DomainUtils.now()`), and `DomainUtils.now()` always returns a large positive nanosecond value making `0 <= now()` trivially true. Combined with `limit=0` (no limit enforced) and no `endTime`, the historical retriever is invoked to scan and stream the entire topic message history from Unix epoch with no bound, enabling resource exhaustion against the mirror node.

### Finding Description

**Code locations:**

- `grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java`, lines 25‚Äì51
- `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 59‚Äì92

**Root cause:**

`isValidStartTime()` is the only semantic guard on `startTime`:

```java
// TopicMessageFilter.java line 49-51
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```

`DomainUtils.now()` returns `convertToNanosMax(Instant.now())` ‚Äî the current wall-clock time in nanoseconds since Unix epoch, always a large positive number (~1.7 √ó 10¬π‚Å∏ for 2024). Therefore `0 <= DomainUtils.now()` is unconditionally `true`. The `@Min(0)` annotation explicitly permits 0 as the minimum value. No floor is applied to `startTime` before it reaches the retriever.

**Exploit flow:**

In `subscribeTopic()` (line 63):
```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```
The filter is forwarded verbatim. With `startTime=0`, the retriever queries all messages with `consensusTimestamp >= 0` ‚Äî i.e., the entire history of the topic.

The limit guard (lines 83‚Äì85):
```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
```
`hasLimit()` returns `limit > 0`, so setting `limit=0` (the `@Min(0)` minimum) disables the limit entirely. Similarly, omitting `endTime` disables the `takeWhile` guard (lines 79‚Äì81). The result is an unbounded full-history scan with no server-side cap.

**Why existing checks fail:**

| Check | Purpose | Fails because |
|---|---|---|
| `@Min(0)` on `startTime` | Rejects negative values | Explicitly allows 0 (epoch) |
| `isValidStartTime()` | Rejects future timestamps | `0 <= now()` is always true |
| `hasLimit()` guard | Caps result count | Disabled when `limit=0` |
| `endTime` guard | Caps by time window | Only applied when `endTime != null` |

### Impact Explanation
Any unauthenticated gRPC client can force the mirror node to execute a full database scan over all messages ever published to a topic, then stream every result over the network. On a high-traffic topic with millions of messages this exhausts database I/O, server CPU, heap memory (reactive backpressure buffers), and network bandwidth. Multiple concurrent such subscriptions compound the effect. This is a practical denial-of-service vector requiring zero privileges.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible by design (mirror nodes serve public data). The exploit requires only a standard gRPC client call with two field values set to their documented minimums (`startTime=0`, `limit=0`). No authentication, no special knowledge, and no race condition is required. It is trivially repeatable and scriptable.

### Recommendation
1. **Enforce a minimum `startTime` floor**: Reject or clamp `startTime` values that predate a reasonable operational epoch (e.g., Hedera mainnet genesis timestamp). Add a custom `@AssertTrue` validator or service-layer check:
   ```java
   // Reject startTime older than, e.g., 30 days or a configured minimum
   private static final long MIN_ALLOWED_START_TIME = /* genesis nanos */ ;
   return startTime >= MIN_ALLOWED_START_TIME && startTime <= DomainUtils.now();
   ```
2. **Require a non-zero limit or enforce a server-side maximum**: When `limit=0` (unlimited), apply a configurable server-side cap (e.g., `GrpcProperties.maxPageSize`) before passing the filter to the retriever.
3. **Enforce a maximum time window**: If `endTime` is null, cap the historical scan window to a configurable duration (e.g., 1 hour) relative to `startTime`.
4. **Rate-limit per subscriber**: Track and throttle the number of historical messages streamed per client IP/session.

### Proof of Concept
```java
// Using the Hedera Java SDK or any gRPC client targeting the mirror node gRPC port
TopicMessageFilter filter = TopicMessageFilter.builder()
    .topicId(EntityId.of(0, 0, <any_valid_topic_num>))
    .startTime(0L)   // Unix epoch ‚Äî passes @Min(0) and isValidStartTime()
    .limit(0L)       // hasLimit() returns false ‚Üí no .take() applied
    // endTime omitted ‚Üí no takeWhile applied
    .build();

// Call subscribeTopic(filter) via gRPC
// Result: server scans and streams ALL historical messages for the topic
//         from consensusTimestamp >= 0 with no bound
```

Preconditions: network access to the mirror node gRPC port (default 5600), knowledge of any valid topic ID (publicly discoverable via REST API). No credentials required.
