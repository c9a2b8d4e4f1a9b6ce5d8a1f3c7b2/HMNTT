### Title
Global Shared `gasLimitBucket` Exhaustion via Max-Gas Burst — Unauthenticated DoS

### Summary
The `gasLimitBucket` is a single process-wide token bucket with no per-user or per-IP partitioning. An unauthenticated attacker can send requests at the maximum rate permitted by the `rateLimitBucket` (500 req/s), each claiming the maximum allowed gas (15,000,000), consuming exactly 750,000 scaled tokens per second — which equals the bucket's full capacity and full refill rate. This keeps the bucket perpetually near-zero, causing every concurrent legitimate user to receive a `ThrottleException` ("Gas per second rate limit exceeded") for as long as the attack continues.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` before any per-user check:

```java
// ContractController.java lines 38-51
throttleManager.throttle(request);          // gas tokens consumed here
validateContractMaxGasLimit(request);       // only restores on InvalidParametersException
```

Inside `ThrottleManagerImpl.throttle()` (line 40):

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

`ThrottleConfiguration` creates the bucket as a single Spring singleton:

```java
// ThrottleConfiguration.java lines 34-45
@Bean(name = GAS_LIMIT_BUCKET)
Bucket gasLimitBucket() {
    long gasLimit = throttleProperties.getGasPerSecond();   // scaleGas(7_500_000_000) = 750_000
    final var limit = Bandwidth.builder()
            .capacity(gasLimit)
            .refillGreedy(gasLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder()
            .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
            .addLimit(limit)
            .build();
}
```

**Arithmetic:**
- `gasPerSecond` default = 7,500,000,000; `GAS_SCALE_FACTOR` = 10,000 → bucket capacity = **750,000 tokens**
- `scaleGas(15_000_000)` = **1,500 tokens per request**
- `requestsPerSecond` default = **500 req/s**
- Attacker throughput: 500 × 1,500 = **750,000 tokens/s = full bucket capacity = full refill rate**

**Root cause:** The bucket is global with no per-user isolation. The rate limiter (500 req/s) is the only guard, but 500 max-gas requests exactly saturates the gas bucket's refill rate, making the attack self-sustaining.

**Why restore doesn't save it:** `ContractCallService.restoreGasToBucket()` returns unused gas *after* execution completes. During the burst window — before any execution finishes — the bucket is already empty. The attacker can pipeline new requests faster than old ones complete, keeping the bucket perpetually drained. Even when gas is restored for failed calls (`gasLimitRefundPercent = 100`), the attacker immediately re-consumes it with the next wave of requests.

**Why `validateContractMaxGasLimit` doesn't help:** `maxGasLimit = 15_000_000` and the default `gas = 15_000_000L`. The check is `gas > maxGasLimit` (strict greater-than), so gas=15,000,000 passes validation and tokens are **never** restored by the controller for this value.

### Impact Explanation
All concurrent legitimate users receive HTTP 429 `ThrottleException("Gas per second rate limit exceeded.")` for the duration of the attack. The service is effectively unavailable for EVM call/estimate workloads. No authentication, funds, or special privileges are required. The attack is stateless and trivially scriptable.

### Likelihood Explanation
Any external actor with a basic HTTP client can execute this. The default gas value in `ContractCallRequest` is already 15,000,000, so no special crafting is needed — the attacker simply sends the default request body at maximum rate. The attack is repeatable, requires no on-chain state, and has zero cost to the attacker.

### Recommendation
1. **Per-source gas accounting**: Introduce per-IP (or per-authenticated-key) gas sub-buckets so one source cannot exhaust the global budget.
2. **Decouple rate limit from gas capacity**: Ensure `requestsPerSecond × scaleGas(maxGasLimit) < gasPerSecond` so the rate limiter alone cannot drain the gas bucket in one second. Currently they are exactly equal.
3. **Consume gas after validation**: Move `validateContractMaxGasLimit` before `throttleManager.throttle()`, or at minimum ensure tokens are reserved only after all cheap validations pass.
4. **Optimistic restore on fast-path failures**: For requests that fail before EVM execution begins, restore tokens synchronously rather than deferring to the service layer.

### Proof of Concept

```bash
# Drain the global gasLimitBucket in ~1 second using default gas value
# No authentication required

for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0x0000000000000000000000000000000000000001",
      "gas": 15000000
    }' &
done
wait

# All subsequent requests from any user now receive:
# HTTP 429: {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}

# Repeat every second to sustain the DoS indefinitely
```

**Verification:** After the burst, send a legitimate request with `gas=21000` (minimum). It will still fail with `GAS_PER_SECOND_LIMIT_EXCEEDED` because the bucket is empty and the attacker's next wave arrives before the 1-second refill completes.
