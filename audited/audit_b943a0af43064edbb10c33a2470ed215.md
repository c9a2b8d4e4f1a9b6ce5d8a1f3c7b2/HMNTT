### Title
Missing File ID Whitelist Validation in REST Java `getNodes()` Allows Silent Empty Node List Response

### Summary
The REST Java `NetworkController.getNodes()` endpoint accepts any arbitrary `file.id` value as long as the operator is `EQ`, but never validates that the value is one of the two permitted address book file IDs (`0.0.101` or `0.0.102`). Supplying a non-existent file ID causes the underlying SQL query to silently return zero rows, making the API report an empty network node list with HTTP 200. The gRPC path has this whitelist check; the REST Java path does not.

### Finding Description
**Controller** (`rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 153–158):
```java
final var fileId = request.getFileId();
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
final var networkNodeRows = networkService.getNetworkNodes(request);
```
The only guard is an operator check. No check that the value is `0.0.101` or `0.0.102`.

**Service** (`rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, lines 178–182):
```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
}
```
Returns the raw caller-supplied numeric ID with no whitelist check.

**Repository SQL** (`rest-java/src/main/java/org/hiero/mirror/restjava/repository/NetworkNodeRepository.java`, lines 26–31):
```sql
with latest_address_book as (
    select start_consensus_timestamp, end_consensus_timestamp, file_id
    from address_book
    where file_id = :fileId   -- attacker-controlled
    ...
)
```
If `:fileId` matches no row in `address_book`, the CTE is empty, the join with `address_book_entry` produces zero rows, and the endpoint returns `{"nodes": [], "links": {}}` with HTTP 200.

**Contrast**: The gRPC `NetworkServiceImpl.getNodes()` (`grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`, lines 56–59) explicitly rejects unknown file IDs:
```java
if (!getValidFileIds().contains(fileId)) {
    throw new IllegalArgumentException(INVALID_FILE_ID);
}
```
This guard is entirely absent from the REST Java code path.

### Impact Explanation
Any unauthenticated caller can issue `GET /api/v1/network/nodes?file.id=eq:0.0.9999` and receive a well-formed HTTP 200 response with an empty node list. Monitoring systems, SDKs, or downstream services that consume this endpoint and trust the response would conclude the network has no nodes. This silently corrupts the apparent network topology without triggering any error signal, satisfying the "reorganizing apparent network topology history" threat model described in the question.

### Likelihood Explanation
No authentication or special privilege is required. The parameter is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, line 954). Any external user who reads the API docs can craft the request. It is trivially repeatable and requires no tooling beyond `curl`.

### Recommendation
Add a whitelist check in `NetworkServiceImpl.getNetworkNodes()` (or in the controller before calling the service) mirroring the gRPC path:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    Set<Long> valid = Set.of(
        systemEntity.addressBookFile101().getId(),
        systemEntity.addressBookFile102().getId()
    );
    if (!valid.contains(id)) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

# Response: {"nodes": [...3+ nodes...], "links": {...}}

curl "https://<mirror-node>/api/v1/network/nodes?file.id=eq:0.0.9999"
