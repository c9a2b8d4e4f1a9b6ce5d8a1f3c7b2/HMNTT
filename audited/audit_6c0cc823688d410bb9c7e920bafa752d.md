### Title
Unauthenticated Connection Pool Exhaustion via `preQueryHint` in `getTopicMessages()`

### Summary
Any unauthenticated caller of `GET /api/v1/topics/:topicId/messages` with the default (or any non-`1`) limit triggers `pool.connect()` to acquire a **dedicated** PostgreSQL connection for the full query duration. Because the pool size is finite and no rate-limiting or authentication guards this path, a flood of concurrent requests exhausts the connection pool, denying database access to all REST API handlers.

### Finding Description

**Trigger condition** — `rest/topicmessage.js` lines 142–143: [1](#0-0) 

`queryHint` is set to `constants.zeroRandomPageCostQueryHint` (`'set local random_page_cost = 0'`) for every request where `limit !== 1`. The default limit is `defaultLimit` (never `1`), so every ordinary request enters this branch.

**Dedicated-connection branch** — `rest/utils.js` lines 1519–1527: [2](#0-1) 

When `preQueryHint` is truthy, `queryQuietly` calls `this.connect()` (a `pg.Pool` checkout), issues `begin; set local random_page_cost = 0`, runs the main query, then `commit`. The connection is held exclusively until all three round-trips complete.

**Release** only happens in `finally` — `rest/utils.js` lines 1540–1544: [3](#0-2) 

**Pool ceiling** — `rest/dbpool.js` lines 7–16: [4](#0-3) 

`max: config.db.pool.maxConnections` is the hard ceiling. Once all slots are occupied, every subsequent `pool.connect()` call blocks until `connectionTimeoutMillis` elapses, returning an error to the caller.

**No authentication or rate-limiting** is present in `getTopicMessages()`: [5](#0-4) 

Only `topicId` format is validated; any syntactically valid topic ID (e.g., `0.0.1`) is accepted.

### Impact Explanation
With a typical pool of 10–20 connections, an attacker sending `N ≥ maxConnections` concurrent requests to `/api/v1/topics/0.0.1/messages` saturates the pool. All other REST API endpoints that also call `pool.queryQuietly` (accounts, transactions, tokens, etc.) begin timing out or returning 500 errors. This constitutes a full REST API denial-of-service, degrading well beyond 30% of node processing capacity for the duration of the attack.

### Likelihood Explanation
The endpoint is public, requires no credentials, and the vulnerable path is the **default** behavior (no special parameters needed). The attack is trivially repeatable with any HTTP load tool (e.g., `ab`, `wrk`, `hey`). A single attacker with a modest internet connection can sustain pool exhaustion indefinitely by keeping `maxConnections` requests in-flight at all times.

### Recommendation
1. **Avoid dedicated connections for read-only hints.** Use `SET LOCAL` inside an explicit transaction only when strictly necessary, or apply the planner hint at the session level via `ALTER ROLE … SET random_page_cost = 0` for the mirror-node DB user, eliminating the need for per-query transactions entirely.
2. **Use `pool.query()` with a session-level GUC** or a `pg` named prepared statement with `options` instead of a full transaction.
3. **Add rate-limiting** (e.g., express-rate-limit) on public topic-message endpoints.
4. **Set a low `connectionTimeoutMillis`** and expose pool-exhaustion metrics so operators can detect and respond to the attack.

### Proof of Concept
```bash
# Requires: a running mirror-node REST API and any valid topic ID (e.g., 0.0.1)
# Send maxConnections+1 concurrent requests (default limit != 1 triggers preQueryHint)

hey -n 1000 -c 50 \
  "http://<mirror-node-host>/api/v1/topics/0.0.1/messages"

# Expected result:
# - First maxConnections requests each acquire a dedicated pg connection
# - Remaining requests receive connection-timeout errors or HTTP 500
# - All other API endpoints (/api/v1/accounts, /api/v1/transactions, etc.)
#   simultaneously begin failing due to pool exhaustion
```
