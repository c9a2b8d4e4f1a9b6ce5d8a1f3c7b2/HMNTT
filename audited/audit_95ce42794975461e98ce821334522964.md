### Title
Unprivileged Cache-Bypass Resource Exhaustion via Inline Alias Literals in GraphQL Account Query

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed exclusively on the raw query string. An unprivileged attacker can embed unique alias values as inline literals in each request, producing a distinct cache key per request and forcing a full parse-validate cycle every time. Combined with the unconditional `entityRepository.findByAlias()` DB call in `getByAliasAndType()` and the complete absence of rate limiting, this allows sustained CPU and DB I/O amplification with no authentication barrier.

### Finding Description

**Cache key is the raw query string only.** [1](#0-0) 

`cache.get(executionInput.getQuery(), ...)` uses the literal query text as the key. When an attacker embeds the alias as an inline literal instead of a variable:

```
# Inline literal — unique cache key per request
{ account(input: {alias: "AAAAAAAAAA"}) { id } }
{ account(input: {alias: "AAAAAAAAAB"}) { id } }
...

# Variable — same cache key every time (safe path)
query Q($a: String) { account(input: {alias: $a}) { id } }
```

Every inline-literal request is a cache miss → full parse + validate cycle runs.

**DB hit is unconditional and uncached.** [2](#0-1) [3](#0-2) 

`entityRepository.findByAlias()` executes a native SQL query on every invocation. The preparsed document cache covers only parse/validate, not execution; there is no second-level result cache. Every request — cache hit or miss — issues a `SELECT * FROM entity WHERE alias = ?` to the database.

**No rate limiting or authentication.** [4](#0-3) 

No `@PreAuthorize`, no Spring Security filter, no throttle/rate-limit bean exists anywhere in the graphql module. `LoggingFilter` only logs; it enforces nothing.

**Existing parser limits do not prevent the attack.** [5](#0-4) 

`maxCharacters(10000)` / `maxTokens(1000)` constrain query size but do not prevent sending thousands of syntactically valid, distinct queries. `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run post-parse, so they do not reduce the parse+validate overhead caused by cache misses.

**Cache size bound does not help.** [6](#0-5) 

`maximumSize=1000` means the attacker only needs to rotate through >1000 unique alias strings to keep the cache perpetually cold, or simply never reuse any alias value.

### Impact Explanation
Every request forces: (1) a full GraphQL parse+validate cycle (CPU), and (2) a synchronous native SQL query against the `entity` table (DB I/O). With no rate limiting and no authentication, a single attacker with a modest HTTP client can sustain thousands of requests per minute, each independently consuming CPU and issuing a DB read. This directly maps to the stated threat: >30% increase in node CPU and DB I/O relative to baseline, achievable without brute-force credential attacks.

### Likelihood Explanation
The attack requires zero privileges — the GraphQL endpoint is publicly reachable. The technique (inline literals vs. variables) is standard GraphQL knowledge. No special tooling is needed; `curl` or any HTTP library suffices. The attack is fully repeatable and can be automated trivially. The alias space (Base32-encoded bytes) is effectively unbounded, so the attacker never runs out of unique cache-busting values.

### Recommendation
1. **Rate-limit at the HTTP layer**: Apply per-IP (and optionally per-query-hash) rate limiting using a servlet filter or an API gateway before requests reach the GraphQL engine.
2. **Normalize cache keys**: Hash or canonicalize the query string (strip literals / use persisted queries) so that structurally identical queries with different inline values share a single cache entry.
3. **Add an execution-level result cache**: Cache `findByAlias()` results (e.g., Spring `@Cacheable`) with a short TTL so repeated alias lookups do not always hit the DB.
4. **Enforce query allow-listing / persisted queries**: Accept only pre-registered query hashes, eliminating arbitrary inline-literal variation entirely.

### Proof of Concept
```bash
# Generate and send 5000 requests, each with a unique inline alias value
for i in $(seq 1 5000); do
  ALIAS=$(python3 -c "import base64, os; print(base64.b32encode(os.urandom(20)).decode().rstrip('='))")
  curl -s -X POST http://<mirror-node-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {alias: \\\"${ALIAS}\\\"}) { id } }\"}" &
done
wait
```

Each request carries a structurally valid but unique query string. `CachedPreparsedDocumentProvider` finds no matching entry for any of them, triggering a full parse+validate cycle per request. Simultaneously, `EntityServiceImpl.getByAliasAndType()` → `entityRepository.findByAlias()` issues a native SQL query to the `entity` table for every request. Monitoring DB query rate and JVM CPU during the 5000-request burst will show a proportional spike in both metrics with no authentication or special access required.
