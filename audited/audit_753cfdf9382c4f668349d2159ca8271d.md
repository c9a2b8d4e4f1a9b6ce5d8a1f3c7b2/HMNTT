### Title
Unbounded Concurrent Subscriptions Exhaust Global `boundedElastic()` Scheduler in `PollingTopicListener`, Causing Polling Starvation

### Summary
`PollingTopicListener.listen()` schedules every subscriber's polling loop on the global `Schedulers.boundedElastic()` singleton, which has a hard thread cap of `10 × availableProcessors`. Because there is no global limit on the total number of concurrent `subscribeTopic` streams — only a per-connection cap of 5 — an unauthenticated attacker who opens many TCP connections can saturate the shared thread pool, starving legitimate subscribers of polling cycles and causing missing or delayed records exported to mirror nodes.

### Finding Description

**Exact code path:**

`PollingTopicListener` (line 31) stores the global Reactor scheduler as an instance field:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
``` [1](#0-0) 

Every call to `listen()` (lines 38–43) attaches two scheduling operations to that same global pool:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
``` [2](#0-1) 

`poll()` (line 61) issues a **synchronous JDBC stream** call:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [3](#0-2) 

Because `withScheduler(scheduler)` causes the repeat body to execute on a `boundedElastic` thread, each active subscription holds a thread for the full duration of every DB query (fired every 500 ms).

**Root cause — failed assumption:** The design assumes the number of concurrent subscriptions is small enough that the bounded thread pool is never saturated. No global subscription counter or admission gate enforces this assumption.

**Why the existing check is insufficient:** The only guard is `maxConcurrentCallsPerConnection = 5` (default): [4](#0-3) [5](#0-4) 

This is a **per-connection** limit, not a global one. An attacker opens `N` TCP connections and gets `5N` concurrent subscriptions with no further restriction.

**Shared pool amplification:** `Schedulers.boundedElastic()` is a JVM-wide singleton. The same pool is consumed by `PollingTopicMessageRetriever`, the safety-check path in `TopicMessageServiceImpl`, `SharedPollingTopicListener`, `SharedTopicListener`, and `NetworkServiceImpl` — all confirmed to call `Schedulers.boundedElastic()` in the same process. [6](#0-5) [7](#0-6) 

### Impact Explanation
When the `boundedElastic` thread pool is saturated, scheduled polling tasks queue behind attacker-owned tasks. Legitimate subscribers' 500 ms polling intervals stretch arbitrarily, causing:
- **Missing records**: messages written to the DB between two delayed polls are never delivered to subscribers whose polling is starved.
- **Delayed records**: the mirror node's live-streaming guarantee degrades; downstream consumers (e.g., SDK clients relying on HCS ordering) receive stale or out-of-order data.
- **Cascading failure**: the safety-check `missingMessages` path and `PollingTopicMessageRetriever` also use the same pool, so gap-recovery queries are equally starved, compounding message loss.

Severity: **High** — directly causes the "incorrect or missing records exported to mirror nodes" impact described in scope, with no data corruption required.

### Likelihood Explanation
- No authentication or API key is required to call `subscribeTopic`.
- gRPC over HTTP/2 allows many multiplexed connections from a single client process.
- The attacker needs only a valid `topicId` (any existing topic, publicly discoverable via the REST API) and a `startTime` in the past to keep each subscription alive indefinitely.
- The default `boundedElastic` cap is `10 × CPU cores` (e.g., 40 threads on a 4-core pod). Opening 10 connections × 5 streams = 50 subscriptions is sufficient to saturate a typical deployment.
- The attack is trivially repeatable and scriptable with any gRPC client library.

### Recommendation
1. **Enforce a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (the field already exists as a metric gauge at line 48 — promote it to an admission gate). [8](#0-7) 
2. **Isolate the polling scheduler**: replace `Schedulers.boundedElastic()` in `PollingTopicListener` with a dedicated, explicitly-sized `Schedulers.newBoundedElastic(...)` instance so attacker subscriptions cannot starve other subsystems. [1](#0-0) 
3. **Add per-IP or per-identity connection limits** at the Netty layer in addition to the existing per-connection call limit.
4. **Prefer `SHARED_POLL` or `REDIS` listener type** (the default is already `REDIS`) for production, as `POLL` mode creates one scheduler task per subscriber rather than sharing a single broadcast stream.

### Proof of Concept
```bash
# Requires grpcurl and a running mirror node with listener.type=POLL
# Step 1: discover any valid topic ID via REST
TOPIC=$(curl -s https://<mirror>/api/v1/topics?limit=1 | jq -r '.topics[0].topic_id')

# Step 2: open 50 concurrent indefinite subscriptions across 10 connections
for i in $(seq 1 10); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d "{\"topicID\":{\"topicNum\":${TOPIC##*.}},\"consensusStartTime\":{\"seconds\":0}}" \
      <mirror>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 3: observe legitimate subscriber latency
# A legitimate subscriber opened now will experience polling delays > 500ms
# because the boundedElastic thread pool (default ~40 threads) is saturated.
# Monitor via: curl http://<mirror>:8080/actuator/metrics/hiero.mirror.grpc.subscribers
```
