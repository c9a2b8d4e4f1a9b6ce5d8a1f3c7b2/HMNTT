### Title
Deleted Entity Bypass via Long-Zero EVM Address in GraphQL `getByEvmAddressAndType`

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes long-zero EVM addresses (first 12 bytes all zero) through `entityRepository.findById()`, which is the standard Spring Data `CrudRepository` method and carries **no** `deleted is not true` filter. All other lookup paths (`findByEvmAddress`, `findByAlias`) explicitly exclude soft-deleted entities. An unprivileged caller can craft a long-zero EVM address encoding any known entity ID and retrieve a soft-deleted contract or account through the GraphQL API.

### Finding Description

**Exact code path:**

`EntityServiceImpl.java` lines 34–41: [1](#0-0) 

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-3 (shard), bytes 4-11 (realm)
        return entityRepository.findById(buffer.getLong())         // bytes 12-19 → entity ID
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

`ByteBuffer` reads sequentially: 4 bytes (shard int), 8 bytes (realm long), 8 bytes (entity num long). When shard==0 and realm==0, the third read is passed directly to `findById`.

**Root cause — asymmetric deletion filtering:**

- `findByEvmAddress` query: `select * from entity where evm_address = ?1 and deleted is not true` [2](#0-1) 
- `findByAlias` query: `select * from entity where alias = ?1 and deleted is not true` [3](#0-2) 
- `findById` (inherited from `CrudRepository<Entity, Long>`): **no deletion filter whatsoever.** [4](#0-3) 

**Exploit flow:**

1. Attacker learns a deleted contract's entity num `N` (all Hedera entity IDs are public via transaction history).
2. For shard=0, realm=0, the encoded DB id equals `N` directly (per `EntityId.encode`: `if (shard == 0 && realm == 0) { return num; }`). [5](#0-4) 
3. Attacker constructs the 20-byte long-zero address: `0x000000000000000000000000` + big-endian 8-byte encoding of `N`.
4. Attacker submits a GraphQL query using this address as the `evmAddress` input.
5. `buffer.getInt()==0` and `buffer.getLong()==0` → branch taken → `findById(N)` executes.
6. The deleted entity is returned; the only remaining filter is `e.getType() == type`, which does not check deletion status.

**Why existing checks fail:**

The `.filter(e -> e.getType() == type)` guard only enforces entity type, not deletion state. The `decodeEvmAddress` helper performs only hex decoding with no structural validation. [6](#0-5) 

### Impact Explanation
Any soft-deleted contract or account (marked `deleted = true` in the `entity` table) can be fully resolved through the GraphQL mirror-node API by any unauthenticated caller. This exposes entity metadata (keys, memo, expiry, type, timestamps) that the deletion semantics are intended to hide. Because this is a read-only mirror-node API, no funds are directly at risk, placing this at Medium severity — information disclosure with unintended smart contract state visibility.

### Likelihood Explanation
Exploitation requires zero privileges and zero on-chain interaction. Entity IDs are permanently visible in public transaction records. The crafted address is a trivially computed 20-byte hex string. The attack is fully repeatable and automatable against any entity ID ever deleted on the network.

### Recommendation
Replace the bare `findById` call in the long-zero branch with a repository method that also enforces `deleted is not true`, consistent with the other two lookup paths:

```java
// Add to EntityRepository:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);
```

Then in `EntityServiceImpl`:
```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                           .filter(e -> e.getType() == type);
}
```

### Proof of Concept

**Precondition:** Contract with entity num `1234` exists and has been soft-deleted (`deleted = true` in DB).

**Step 1 — Compute the long-zero address:**
```
entity_num = 1234  →  hex = 00000000000004D2
long-zero address  = 0x000000000000000000000000 00000000000004D2
                   = 0x00000000000000000000000000000000000004D2
```

**Step 2 — Send GraphQL query (no authentication required):**
```graphql
query {
  contract(input: { evmAddress: "0x00000000000000000000000000000000000004D2" }) {
    entityId { shard realm num }
    deleted
    memo
    key
  }
}
```

**Step 3 — Observed result:**
The API returns the full entity record for the deleted contract, including `deleted: true` in the response — confirming the entity was retrieved despite being soft-deleted. The `findByEvmAddress` path would have returned `null` for the same entity.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
