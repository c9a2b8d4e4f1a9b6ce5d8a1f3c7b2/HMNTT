### Title
Missing Hash Canonicalization in `block.hash` Query Parameter Causes Silent Empty Results

### Summary
`requestQueryParser()` lowercases query parameter *keys* but only canonicalizes *values* for `order` and `result` — `block.hash` values are passed through in their original mixed case. `formatComparator()` strips the `0x` prefix but never lowercases the hash. `getRecordFileBlockDetailsFromHash()` then issues a case-sensitive PostgreSQL `LIKE` query against a lowercase-stored column, silently returning no records for a valid block hash supplied in mixed case.

### Finding Description

**Step 1 — `requestQueryParser()` does not canonicalize `block.hash` values**

`rest/middleware/requestHandler.js` lines 10–13 define the canonicalization map: [1](#0-0) 

Only `order` and `result` receive `lowerCaseQueryValue`. `block.hash` is absent. The loop at lines 57–66 lowercases the *key* but calls `canonicalizeValue(lowerKey, value)`, which returns the value unchanged for any key not in the map: [2](#0-1) 

A value like `0xAbCd1234...` survives intact.

**Step 2 — Validation accepts mixed-case hex**

`isValidEthHashOrHederaHash()` uses a case-insensitive character class `[0-9A-Fa-f]`: [3](#0-2) 

`filterValidityChecks()` for `BLOCK_HASH` calls this validator and passes: [4](#0-3) 

**Step 3 — `formatComparator()` strips `0x` but does not lowercase**

The `BLOCK_HASH` case in `formatComparator()` only calls `stripHexPrefix()`: [5](#0-4) 

After this, `AbCd1234...` (mixed case, no prefix) is the filter value forwarded to the service layer.

**Step 4 — `getRecordFileBlockDetailsFromHash()` issues a case-sensitive LIKE query without lowercasing**

The static query: [6](#0-5) 

The method passes the hash directly as a prefix pattern with no `.toLowerCase()`: [7](#0-6) 

**Contrast with `getByHashOrNumber()`**, which handles the path-parameter variant and *does* lowercase: [8](#0-7) 

**Step 5 — Caller treats a null result as "skip"**

In `contractController.js`, when `getRecordFileBlockDetailsFromHash()` returns `null` (no match), the handler returns `{skip: true}`, producing an empty response: [9](#0-8) 

### Impact Explanation
Any unprivileged caller querying `/api/v1/contracts/results?block.hash=0xAbCd...` with a valid but mixed-case hash receives an empty result set instead of the correct contract results. The block and its transactions exist in the database; they are simply not returned. This constitutes incorrect record export from the mirror node — valid on-chain data is silently withheld, breaking client applications that rely on case-insensitive hash lookup (consistent with Ethereum tooling conventions).

### Likelihood Explanation
No authentication or special privilege is required. Any HTTP client can trigger this by supplying uppercase or mixed-case hex digits in the `block.hash` query parameter. Ethereum tooling commonly produces checksummed (EIP-55 mixed-case) addresses and hashes. The bug is trivially repeatable and deterministic.

### Recommendation
Add `.toLowerCase()` inside `getRecordFileBlockDetailsFromHash()` before constructing the query parameter, mirroring the existing fix already present in `getByHashOrNumber()`:

```js
async getRecordFileBlockDetailsFromHash(hash) {
  const row = await super.getSingleRow(
    RecordFileService.recordFileBlockDetailsFromHashQuery,
    [`${hash.toLowerCase()}%`]   // ← add .toLowerCase()
  );
  return row === null ? null : new RecordFile(row);
}
```

Alternatively, add `'block.hash': lowerCaseQueryValue` to `queryCanonicalizationMap` in `requestHandler.js` so the value is normalized at the entry point, consistent with how `order` and `result` are handled.

### Proof of Concept

**Precondition:** A block exists whose hash starts with lowercase hex digits, e.g. `ff90773193892540084e62086df3953d2362061b16cf682fde5bb00830b232b2...`

**Trigger:**
```
GET /api/v1/contracts/results?block.hash=0xFF90773193892540084E62086DF3953D2362061B16CF682FDE5BB00830B232B2
```

**Expected:** Contract results for transactions in that block are returned (HTTP 200 with data).

**Actual:** `getRecordFileBlockDetailsFromHash()` executes `hash LIKE 'FF90773...%'` against a column storing `ff90773...` (lowercase). PostgreSQL `LIKE` is case-sensitive; no row matches; the method returns `null`; the controller returns `{skip: true}`; the API returns an empty result set.

**Verification of the asymmetry:** The identical hash supplied as a path parameter (`/api/v1/blocks/FF90773...`) succeeds because `getByHashOrNumber()` calls `hash.toLowerCase()` before querying — confirming the fix is known and applied inconsistently.
