### Title
Unauthenticated Repeated DB Miss via Crafted Long-Form EVM Address in GraphQL `getByEvmAddressAndType`

### Summary
Any unauthenticated user can craft a valid-format EVM address with bytes 0–11 all zero and arbitrary bytes 12–19, causing `getByEvmAddressAndType` to unconditionally execute a `SELECT * FROM entity WHERE id = ?` primary-key lookup against a non-existent row. Because the GraphQL module has no per-IP or per-endpoint rate limiting, an attacker can sustain a high-volume stream of such requests, each generating a DB miss, driving up database CPU without returning any data.

### Finding Description
**Code path:**
`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41
`graphql/src/main/resources/graphql/account.graphqls`, line 96

The schema-level `@Pattern` directive only validates that `evmAddress` is 40 hex characters (optionally `0x`-prefixed):
```
evmAddress: String @Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")
```
It places no restriction on the byte values themselves.

Inside `getByEvmAddressAndType`:
```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);   // hex → 20 bytes
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {     // bytes 0-3 == 0, bytes 4-11 == 0
    return entityRepository.findById(buffer.getLong())   // bytes 12-19 as Long → DB hit
           .filter(e -> e.getType() == type);
}
```
Any address of the form `000000000000000000000000XXXXXXXXXXXXXXXX` (bytes 0–11 all zero, bytes 12–19 arbitrary) passes the pattern check and enters the `findById` branch. If the resulting `Long` does not correspond to an existing entity, the DB executes a full primary-key index scan that returns empty — a DB miss — with no result cached and no request rejected.

**Root cause / failed assumption:** The code assumes the "long-form" EVM address branch (shard=0, realm=0) will only be reached for legitimate entity lookups. There is no guard preventing an attacker from supplying arbitrary values in bytes 12–19 to force repeated misses.

**No rate limiting on the GraphQL module:** `ThrottleConfiguration` and `ThrottleManagerImpl` live exclusively in the `web3` module. `GraphQlConfiguration` only installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which limit query structure but not request rate. There is no Bucket4j bucket, no Spring Security filter, and no connection-level throttle applied to `POST /graphql/alpha`.

**No caching of entity lookups:** The only cache in the GraphQL module (`CachedPreparsedDocumentProvider`) caches parsed query documents, not entity repository results. Every request causes a live DB round-trip.

### Impact Explanation
Each crafted request causes one `SELECT * FROM entity WHERE id = ?` execution. While a single primary-key miss is cheap, the absence of any rate limiting means an attacker can sustain thousands of requests per second from a single client or a small botnet. Sustained high-frequency misses increase DB CPU, connection pool pressure, and I/O wait. Because the attacker can vary bytes 12–19 across requests, no query-result cache (even if added) would absorb the load. The GraphQL service and its shared database are the affected components; other mirror-node services sharing the same DB instance (REST, web3) would experience collateral degradation.

### Likelihood Explanation
The exploit requires zero authentication, zero on-chain state, and zero knowledge beyond the public GraphQL schema. The crafted address passes all existing input validation. The attacker needs only an HTTP client capable of sending POST requests to `/graphql/alpha`. The attack is trivially scriptable, repeatable, and can be parallelized across multiple source IPs. No special tooling or protocol knowledge is required.

### Recommendation
1. **Add rate limiting to the GraphQL module.** Introduce a Bucket4j (or equivalent) filter bean scoped to the `graphql` Spring application context, mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`. Apply it as a servlet filter or Spring Security filter on `POST /graphql/alpha`.
2. **Validate the entity-num range before issuing the DB query.** After extracting `buffer.getLong()` (bytes 12–19), reject values that exceed the known maximum entity ID or are otherwise implausible (e.g., negative when interpreted as signed, or above a configurable ceiling), returning an empty `Optional` without touching the DB.
3. **Cache negative (miss) results.** Add a short-lived (e.g., 1–5 s) negative-result cache for `findById` lookups in the GraphQL service layer so that repeated misses for the same ID do not each generate a DB round-trip.

### Proof of Concept
```bash
# Craft an address: bytes 0-11 = 0x00, bytes 12-19 = non-existent entity num (e.g., 0xDEADBEEFCAFEBABE)
EVM_ADDR="000000000000000000000000DEADBEEFCAFEBABE"

# Single request — enters findById branch, returns null (DB miss)
curl -s -X POST http://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\":\"{ account(input: { evmAddress: \\\"$EVM_ADDR\\\" }) { id } }\"}"

# Flood — no rate limiting, each request hits the DB
for i in $(seq 1 10000); do
  # vary last 8 bytes to defeat any future caching
  SUFFIX=$(printf '%016X' $i)
  ADDR="000000000000000000000000${SUFFIX}"
  curl -s -X POST http://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { evmAddress: \\\"$ADDR\\\" }) { id } }\"}" &
done
wait
```
Each iteration passes `@Pattern` validation, enters the `buffer.getInt()==0 && buffer.getLong()==0` branch in `EntityServiceImpl.java:37`, and issues `entityRepository.findById(<num>)` against the database. With no rate limiting, all requests proceed concurrently, generating sustained DB load with zero returned data.
