### Title
Unprivileged Double DB Query Amplification via `?scheduled=true` in `getTransactionsByIdOrHash()`

### Summary
Any unauthenticated user can force two sequential `pool.queryQuietly()` calls per request in `getTransactionsByIdOrHash()` by supplying a syntactically valid but non-existent transaction ID with `?scheduled=true`. The second query also uses an expanded timestamp range (`maxScheduledTransactionConsensusTimestampRangeNs`), making it more expensive than the first. Under concurrent load this doubles effective DB query throughput requirements and can exhaust the connection pool.

### Finding Description

**Exact code path:** `rest/transactions.js`, `getTransactionsByIdOrHash()`, lines 930–938 and `mayMissLongTermScheduledTransaction()`, lines 974–996.

```js
// rest/transactions.js lines 930-938
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);   // ← second query
});
```

```js
// rest/transactions.js lines 991-993
} else if (scheduled && transactions.length === 0) {
  return true;   // ← triggers second query
}
```

**Root cause:** `mayMissLongTermScheduledTransaction` returns `true` — unconditionally triggering the second query — whenever `scheduled === true` (i.e., the caller passed `?scheduled=true`) AND the first query returns zero rows. There is no guard requiring the transaction to actually exist before attempting the second, wider-range query.

**Two reliable trigger paths for an unprivileged user:**

*Path A — `?scheduled=true` with any syntactically valid non-existent transaction ID:*
1. `extractSqlFromTransactionsByIdOrHashRequest` successfully parses any well-formed ID (e.g. `0.0.1-1234567890-000000000`) and builds a SQL query.
2. First `pool.queryQuietly` returns 0 rows (transaction does not exist).
3. `mayMissLongTermScheduledTransaction(false, true, [])` evaluates `scheduled && transactions.length === 0` → `true`.
4. Second `pool.queryQuietly` is issued with `upperConsensusTimestampIndex` expanded by `maxScheduledTransactionConsensusTimestampRangeNs` — a wider, more expensive scan.
5. Second query also returns 0 rows; `NotFoundError` is thrown.

*Path B — known SCHEDULECREATE transaction ID without `?scheduled`:*
1. First query returns a successful SCHEDULECREATE row (`type === 42`, `result` in `SUCCESS_PROTO_IDS`, `scheduled === false`).
2. `mayMissLongTermScheduledTransaction(false, undefined, [...])` sets `scheduleExists = true` and finds no `scheduled === true` row → returns `true`.
3. Second query is issued with the expanded range.

Path A requires zero knowledge of real network data and is trivially repeatable.

**Why existing checks are insufficient:**
- `acceptedSingleTransactionParameters` allows `SCHEDULED` without restriction.
- `buildAndValidateFilters` validates the value (`true`/`false`) but not whether the combination with a non-existent ID is safe.
- No rate limiting is applied at the handler or middleware level in the reviewed code.
- The `isTransactionHash` guard only short-circuits the second query for hash-based lookups, not ID-based ones. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Each crafted request consumes two sequential DB connection slots instead of one, and the second query scans a larger timestamp range. Under concurrent attack traffic this effectively halves the DB query throughput available to legitimate users. If the connection pool is sized for normal single-query-per-request load, sustained concurrent attack requests will queue and eventually exhaust the pool, causing all `/api/v1/transactions/:id` requests (and potentially other endpoints sharing the same pool) to time out or fail — matching the "total network shutdown" severity class for the REST API tier. [4](#0-3) 

### Likelihood Explanation
The attack requires no authentication, no special account, and no knowledge of real network state. A valid transaction ID format (`0.0.X-SECONDS-NANOS`) is publicly documented. The trigger is deterministic and 100% reproducible. A single attacker with a modest HTTP load tool (e.g., `wrk`, `ab`) sending concurrent `GET /api/v1/transactions/0.0.1-1-0?scheduled=true` requests is sufficient to exercise the path at scale.

### Recommendation
1. **Guard the second query on existence:** Only enter the second query path if the first query returned at least one row (i.e., the transaction is known to exist). A non-existent transaction cannot have a long-term scheduled counterpart.
2. **Short-circuit on zero rows before the `mayMissLongTermScheduledTransaction` check:**
   ```js
   const {rows} = await pool.queryQuietly(query, params).then((result) => {
     if (result.rows.length === 0 || !mayMissLongTermScheduledTransaction(...)) {
       return result;
     }
     ...
     return pool.queryQuietly(query, params);
   });
   ```
3. **Apply per-IP rate limiting** on `/api/v1/transactions/:id` to bound the amplification factor under attack.
4. **Reject `?scheduled=true` early** if the transaction ID does not resolve to any row in the first query, returning 404 without a second query.

### Proof of Concept
```bash
# Step 1: Confirm the double-query path triggers for a non-existent ID with ?scheduled=true
# (any syntactically valid transaction ID that does not exist in the DB)
curl -v "http://<mirror-node>/api/v1/transactions/0.0.1-1234567890-000000000?scheduled=true"
# Expected: 404 Not Found — but two DB queries were issued

# Step 2: Amplification attack (concurrent requests)
wrk -t 10 -c 200 -d 60s \
  "http://<mirror-node>/api/v1/transactions/0.0.1-1234567890-000000000?scheduled=true"

# Each request forces 2 sequential pool.queryQuietly() calls.

```

const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
