### Title
Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` Causes Persistent Bytecode Blackout

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"`, but the method returns `Optional<byte[]>`. When a contract is not yet indexed, the method returns `Optional.empty()` — which is not `null` — so the empty result is cached for up to 1 hour. An unprivileged attacker who queries a contract ID in the brief window before the importer indexes it can permanently poison that instance's cache, causing all subsequent `eth_getCode`/`eth_call` requests for that contract to return empty bytecode for the full cache TTL.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The `unless` SpEL expression `"#result == null"` is evaluated against the `Optional<byte[]>` return value. `Optional.empty()` is a non-null Java object, so the condition is `false` and the empty Optional is written into the Caffeine cache. The cache manager (`CACHE_MANAGER_CONTRACT`) is a `CaffeineCacheManager` — a JVM-local, in-process cache with a default TTL of `expireAfterAccess=1h` — not a distributed cache.

**Exploit flow:**
1. Attacker monitors the Hedera network (public) and observes a new `ContractCreate` transaction, learning the assigned contract entity ID before the mirror node importer processes it.
2. Attacker immediately sends an `eth_getCode` or `eth_call` request to the mirror node web3 API targeting that contract ID.
3. The importer has not yet written the `runtime_bytecode` row; the DB query returns no row, so `findRuntimeBytecode()` returns `Optional.empty()`.
4. Because `Optional.empty() != null`, the `unless` guard does not fire, and `Optional.empty()` is stored in the Caffeine cache under key `contractId`.
5. The importer subsequently writes the bytecode to the DB, but the cache is not invalidated (no `@CacheEvict` exists).
6. For the next ≤1 hour, every call to `findRuntimeBytecode(contractId)` on that instance returns the cached `Optional.empty()`.
7. `ContractBytecodeReadableKVState.readFromDataSource()` maps the empty Optional to `null` (line 42–46), so the EVM sees no code at that address — all contract calls revert or return empty bytecode.

**Why existing checks fail:** The `unless` condition only guards against a literal `null` return, not an empty `Optional`. There is no `@CacheEvict` or cache invalidation path. The Caffeine cache is per-instance, so in a multi-instance deployment, instances that were not poisoned serve correct results while poisoned instances serve empty bytecode — creating inconsistent record export across the cluster.

### Impact Explanation
Any valid deployed contract can be made to appear as a non-existent (zero-bytecode) account on a poisoned instance for up to 1 hour. This causes:
- `eth_getCode` to return `0x` for a live contract.
- `eth_call` and `eth_estimateGas` to fail or return incorrect results (EVM treats the address as an EOA).
- DApps and tooling relying on the mirror node's web3 API to malfunction for that contract.

Severity: **Medium** — targeted DoS on specific contracts; no data exfiltration or privilege escalation, but directly causes incorrect records to be exported/served by the mirror node.

### Likelihood Explanation
Hedera network transactions are public and near-real-time observable. The window between on-chain finality and importer indexing is typically a few seconds. An attacker with a script monitoring the Hedera mirror REST API or gRPC stream for new `ContractCreate` transactions can reliably race the importer. No credentials, tokens, or special access are required — only the ability to send HTTP requests to the public web3 endpoint. The attack is repeatable for every new contract deployment.

### Recommendation
Fix the `unless` condition to also exclude empty Optionals:
```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```
This prevents `Optional.empty()` from ever being written to the cache, so a subsequent request after indexing will hit the DB and return the correct bytecode.

### Proof of Concept
**Preconditions:** Multi-instance or single-instance mirror node web3 API; attacker has HTTP access to the endpoint.

**Steps:**
1. Monitor the Hedera network for a new `ContractCreate` transaction (e.g., via `GET /api/v1/transactions?type=CONTRACTCREATEINSTANCE`).
2. Extract the new contract entity ID (e.g., `0.0.12345`) from the transaction receipt before the mirror node importer processes it.
3. Send immediately:
   ```
   POST /api/v1/contracts/call
   {"data":"0x","to":"0x000000000000000000000000000000000000303d","block":"latest"}
   ```
   or an `eth_getCode` JSON-RPC call for that address.
4. Observe the response returns empty bytecode (`0x`).
5. Wait 30–60 seconds for the importer to index the contract.
6. Repeat the same request — the response still returns `0x` (cache hit on `Optional.empty()`).
7. Confirm via direct DB query that `runtime_bytecode` is now populated in the `contract` table.
