### Title
Unbounded Log Amplification via Unsanitized `X-Forwarded-For` Header and URL Path in `TracingMiddleware`

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` logs the raw, unsanitized `X-Forwarded-For` header value and the full `RequestURI()` path for every non-internal request via `log.Info`, with no length cap. An unprivileged attacker can craft requests with a near-1 MB `X-Forwarded-For` header and a long URL path, causing each request to produce a multi-kilobyte log entry. At a moderate sustained rate this drives disk write I/O well above 30% of baseline.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 43–61; `getClientIpAddress()`, lines 63–74.

```
// line 46
clientIpAddress := getClientIpAddress(request)   // reads X-Forwarded-For verbatim, no truncation
// line 47
path := request.URL.RequestURI()                  // reads full URI, no truncation

// lines 52-53
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)

// lines 55-59
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)   // fires for every non-internal endpoint
}
```

`getClientIpAddress()` (lines 64–68) reads `X-Real-IP` first, then `X-Forwarded-For`, and returns the raw string with zero length validation. Go's `net/http` server enforces `MaxHeaderBytes` (default `1 << 20` = 1 MB) but does **not** reject a request; it simply truncates the header map after that limit. A single header value can therefore be up to ~1 MB before the server rejects the connection.

**Root cause / failed assumption:** The code assumes the client IP and path fields are short, human-readable strings. No `maxLen` guard, no `strings.TruncateRune`, no sanitization is applied before the string is passed to `log.Info`.

**Why existing checks fail:**

- The Traefik `rateLimit` (10 req/s per `requestHost`) defined in `charts/hedera-mirror-rosetta/values.yaml` lines 157–160 is an **optional Helm-chart-level** control (`global.middleware: false` by default). It is absent when the server is accessed directly on port 5700, or when the chart is deployed without Traefik.
- Even with Traefik active, the limit is per `requestHost`; an attacker using multiple source hosts or IPs is not bounded.
- The `inFlightReq: amount: 5` limit (values.yaml line 153) only caps concurrency, not the per-request log payload size.
- There is no application-level rate limiter or log-message size cap anywhere in the Go code.

### Impact Explanation
Each crafted request can produce a log entry of ~1 MB (header) + URL length. At 10 req/s (within even the Traefik rate limit per host) that is ~10 MB/s of synchronous log writes. On a typical cloud VM with 50–150 MB/s disk write throughput, 10 MB/s of additional log I/O easily exceeds a 30% increase over a quiet baseline. Without Traefik, 100 req/s yields ~100 MB/s, saturating most disks entirely. This degrades all other disk-dependent operations (database WAL, importer writes) and can cause the node to fall behind chain sync.

### Likelihood Explanation
- **No authentication required.** Any external user can reach port 5700 (or the ingress).
- **Trivially reproducible** with `curl` or any HTTP client that supports custom headers.
- **Amplification ratio is high**: one small TCP request (~1 KB on the wire) produces a ~1 MB log write.
- The Traefik rate-limit mitigation is not guaranteed to be deployed and is bypassable with multiple source IPs.

### Recommendation
1. **Truncate logged fields** before formatting the message. Add a helper such as:
   ```go
   const maxLogFieldLen = 256
   func truncate(s string) string {
       if len(s) > maxLogFieldLen { return s[:maxLogFieldLen] + "…" }
       return s
   }
   ```
   Apply it to both `clientIpAddress` and `path` in the `fmt.Sprintf` call (lines 52–53).

2. **Validate/reject oversized headers at the application layer**: set `http.Server.MaxHeaderBytes` explicitly to a small value (e.g., 8 KB) in `main.go` line 220.

3. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate`) so the protection is not solely dependent on the infrastructure layer.

### Proof of Concept
```bash
# Build a ~900 KB X-Forwarded-For value
LARGE_IP=$(python3 -c "print('1.2.3.4, ' * 100000)")

# Send 100 requests per second to a non-internal endpoint
for i in $(seq 1 100); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: $LARGE_IP" \
    "http://<rosetta-host>:5700/network/list$(python3 -c "print('A'*4096)")" \
    -X POST -H "Content-Type: application/json" -d '{}' &
done
wait
```

Each request causes `TracingMiddleware` to call `log.Info` with a message exceeding 900 KB. Observe disk write I/O on the node (e.g., `iostat -x 1`) spike by >30% compared to the preceding 24-hour baseline. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```go
func TracingMiddleware(inner http.Handler) http.Handler {
	return http.HandlerFunc(func(responseWriter http.ResponseWriter, request *http.Request) {
		start := time.Now()
		clientIpAddress := getClientIpAddress(request)
		path := request.URL.RequestURI()
		tracingResponseWriter := newTracingResponseWriter(responseWriter)

		inner.ServeHTTP(tracingResponseWriter, request)

		message := fmt.Sprintf("%s %s %s (%d) in %s",
			clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))

		if internalPaths[path] {
			log.Debug(message)
		} else {
			log.Info(message)
		}
	})
}
```

```go
func getClientIpAddress(r *http.Request) string {
	ipAddress := r.Header.Get(xRealIpHeader)

	if len(ipAddress) == 0 {
		ipAddress = r.Header.Get(xForwardedForHeader)
	}

	if len(ipAddress) == 0 {
		ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
	}

	return ipAddress
```
