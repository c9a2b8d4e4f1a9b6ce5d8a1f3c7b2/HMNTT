### Title
Account Existence Enumeration via Differential HTTP Response on `/api/v1/accounts/{id}/rewards`

### Summary
The `/api/v1/accounts/{id}/rewards` endpoint is accessible to any unauthenticated user and returns a deterministic HTTP 404 for non-existent accounts versus HTTP 200 (with an empty rewards array) for existing accounts. This behavioral difference allows an attacker to enumerate which account IDs exist in the mirror node's entity table without any credentials or privileges.

### Finding Description
**Code path:**

In `rest/controllers/accountController.js`, `listStakingRewardsByAccountId` (lines 170–203):

```
171: const accountId = await EntityService.getEncodedId(req.params[...]);
172: const isValidAccount = await EntityService.isValidAccount(accountId);
173: if (!isValidAccount) {
174:   throw new NotFoundError();   // ← 404 for non-existent account
175: }
...
202: res.locals[responseDataLabel] = response;  // ← 200 for existing account
```

`EntityService.isValidAccount` (`rest/service/entityService.js`, lines 60–63) runs a bare existence query against the entity table:

```sql
SELECT type FROM entity WHERE id = $1
```

It returns `true` if a row exists, `false` otherwise. No deleted-flag filter is applied here (contrast with `entityFromAliasQuery` which filters `coalesce(deleted, false) <> true`).

When the account does **not** exist, `NotFoundError` is thrown before `res.locals[responseDataLabel]` is ever set. Express routes this to the error handler, producing a 404. The `responseHandler` middleware (`rest/middleware/responseHandler.js`, lines 26–31) would also throw `NotFoundError` if `responseData === undefined`, but in this flow the error is raised earlier in the controller.

When the account **does** exist, the controller sets `res.locals[responseDataLabel] = response` (line 202) with a `rewards` array (possibly empty) and `responseHandler` sends a 200.

**Observable difference:**
- `GET /api/v1/accounts/404/rewards` → `404 {"_status":{"messages":[{"message":"Not found"}]}}`
- `GET /api/v1/accounts/1001/rewards` → `200 {"rewards":[],"links":{"next":null}}` (account exists, no rewards)

This is confirmed by the integration test fixture `rest/__tests__/specs/accounts/{id}/rewards/not-found.json`.

### Impact Explanation
An attacker can iterate numeric account IDs (Hedera IDs are sequential integers) and build a complete map of all accounts tracked by the mirror node. While Hedera's ledger is public, the mirror node may track accounts not yet visible on public explorers, including newly created accounts, system accounts, or accounts on private/permissioned networks. For the "unpaid staking rewards" scope: once an attacker identifies which accounts exist, they can cross-reference the 200 responses (which include reward amounts and timestamps) to identify high-value accounts with large unclaimed staking rewards, enabling targeted social engineering or phishing attacks against those account holders.

### Likelihood Explanation
No authentication, API key, or rate-limit is enforced in the visible code path. The attack requires only HTTP GET requests with sequential integer IDs — trivially automatable with a single script. Hedera account IDs start at low integers and are monotonically increasing, so the full enumeration space is bounded and small. Any external user can perform this attack repeatably and at scale.

### Recommendation
1. **Normalize responses**: For numeric account IDs that do not exist in the entity table, return `200` with an empty `rewards` array (same as an existing account with no rewards) rather than `404`. This removes the oracle. The `isValidAccount` guard at lines 172–175 should be removed or replaced with a response that does not distinguish existence.
2. **Alternatively**, if 404 semantics are required for non-existent accounts, apply authentication/authorization middleware before the rewards endpoint so only privileged callers can observe the distinction.
3. **Rate limiting**: Add per-IP rate limiting on `/api/v1/accounts/*/rewards` to slow enumeration regardless of the above fix.

### Proof of Concept
```bash
# Account does NOT exist → 404 leaks non-existence
curl -s -o /dev/null -w "%{http_code}" \
  http://<mirror-node>/api/v1/accounts/999999/rewards
# Output: 404

# Account EXISTS but has no rewards → 200 leaks existence
curl -s http://<mirror-node>/api/v1/accounts/1001/rewards
# Output: {"rewards":[],"links":{"next":null}}

# Automated enumeration script
for id in $(seq 1 100000); do
  status=$(curl -s -o /dev/null -w "%{http_code}" \
    http://<mirror-node>/api/v1/accounts/${id}/rewards)
  [ "$status" = "200" ] && echo "EXISTS: 0.0.${id}"
done
```

The differential 404/200 response unambiguously identifies which account IDs are present in the mirror node entity table with zero privileges required.
