### Title
Post-Parse Throttle Ordering Enables Concurrent Memory Exhaustion via Maximum-Size JSON String Fields

### Summary
`JacksonConfiguration.jacksonCustomizer` sets `maxStringLength` and `maxDocumentLength` to `ethereumMaxCallDataSize() * 2 + 1024` (~263 KB with the 128 KiB default). Spring MVC fully deserializes the `@RequestBody` before `ContractController.call()` is entered, meaning `throttleManager.throttle(request)` executes only after the full string is already buffered in heap. An unprivileged attacker sending many concurrent requests with a `data` field of exactly `maxSize` characters can accumulate hundreds of megabytes to gigabytes of live heap objects before any throttle token is consumed, leading to GC thrashing or OOM.

### Finding Description

**Exact code path:**

`JacksonConfiguration.java` lines 22 and 29 set the per-request ceiling: [1](#0-0) 

`ContractController.java` line 38 shows `@RequestBody` deserialization happens before the method body runs, and line 40 is where the throttle is first consulted: [2](#0-1) 

**Root cause:** Spring MVC's `@RequestBody` argument resolver reads and deserializes the entire HTTP body into a `ContractCallRequest` object before the controller method is invoked. Jackson's `maxStringLength` check is a parse-time guard — it rejects strings *longer* than `maxSize`, but a string of exactly `maxSize` characters is fully buffered into a Java `String` (≈ 2 × `maxSize` bytes on the heap due to UTF-16 encoding) and then handed to the controller. Only at that point does `throttleManager.throttle(request)` run.

**Failed assumption:** The comment on line 17 says "Configure JSON parsing limits to reject malicious input," implying the limits are the primary defense. But the throttle — the only rate-control mechanism — is positioned *after* the memory has already been allocated.

**Concrete numbers:**
- `ethereumMaxCallDataSize()` default = 128 KiB (131 072 bytes), confirmed by test comment "any payload over 6 KiB and up to 128 KiB" [3](#0-2) 
- `maxSize = 131072 × 2 + 1024 = 263 168` bytes
- Default `requestsPerSecond = 500` [4](#0-3) 
- Default `requestTimeout = 10 s` [5](#0-4) 

At steady state: 500 req/s × 10 s timeout = up to **5 000 concurrent in-flight requests**, each holding ≈ 263 KB (raw) + ≈ 526 KB (UTF-16 `String`) ≈ **~2.5 GB of live heap** from string buffers alone, before any throttle token is checked.

No HTTP-level body-size limit (Tomcat `maxPostSize`, `spring.servlet.multipart.max-request-size`, or a servlet filter) is present in the codebase to reject oversized bodies before Jackson runs. [6](#0-5) 

### Impact Explanation
An attacker can drive the JVM heap to exhaustion, triggering continuous full GC cycles (CPU spike to 100 %, throughput collapse) or an `OutOfMemoryError` that kills the process. All legitimate users of `/api/v1/contracts/call` are denied service. No authentication or special privilege is required — the endpoint is public.

### Likelihood Explanation
The attack requires only an HTTP client capable of sending ~500 concurrent POST requests of ~263 KB each, which is trivially achievable with `curl`, `wrk`, or any load-testing tool from a single machine with a 1 Gbps uplink (500 × 263 KB = ~130 MB/s, well within commodity bandwidth). The attacker does not need a valid Ethereum address or contract — any hex string of the right length is accepted by the `data` field. The attack is repeatable and stateless.

### Recommendation

1. **Enforce a body-size limit before Jackson runs.** Add a Tomcat/servlet-level limit, e.g. in `application.yml`:
   ```yaml
   server:
     tomcat:
       max-http-form-post-size: 270000   # slightly above maxSize
   ```
   Or add a `OncePerRequestFilter` that checks `Content-Length` and rejects oversized requests before the body is read.

2. **Move the throttle check before body deserialization.** Implement a servlet filter that consumes a rate-limit token (from the existing `rateLimitBucket`) before Spring MVC reads the body. This ensures that even if a large body is sent, the connection is rejected at the network/filter layer.

3. **Reduce `maxStringLength` to the minimum needed.** The `data` field is hex-encoded calldata, so its maximum legitimate length is `ethereumMaxCallDataSize() * 2 + 2` (for the `0x` prefix). Setting `maxStringLength` to this tighter bound reduces per-request memory exposure.

### Proof of Concept

```bash
# Generate a payload with data field of exactly maxSize characters
# maxSize = 131072*2+1024 = 263168; subtract 2 for "0x" prefix = 263166 hex chars
PAYLOAD=$(python3 -c "
import json, sys
data = '0x' + 'aa' * 131583  # 263166 hex chars after 0x = 263168 total
print(json.dumps({'to': '0x0000000000000000000000000000000000000001', 'data': data, 'gas': 21000}))
")

# Send 500 concurrent requests (fills the rate-limit bucket in one burst)
# Repeat every second to sustain pressure
for i in $(seq 1 500); do
  curl -s -X POST http://<host>:8545/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "$PAYLOAD" &
done
wait
```

Each concurrent request causes Jackson to buffer ≈ 263 KB before the throttle is checked. With 500 concurrent requests in-flight and a 10-second processing window, heap pressure reaches ~1–2.5 GB from string buffers alone, causing GC thrashing or OOM on typical deployments.

@Configuration(proxyBeanMethods = false)
class JacksonConfiguration {

    // Configure JSON parsing limits to reject malicious input
    @Bean
    @SuppressWarnings("removal")
    Jackson2ObjectMapperBuilderCustomizer jacksonCustomizer(EvmProperties properties) {
        final var jumboConfig = properties.getVersionedConfiguration().getConfigData(JumboTransactionsConfig.class);
        final int maxSize = jumboConfig.ethereumMaxCallDataSize() * 2 + 1024;
        return builder -> {
            var streamReadConstraints = StreamReadConstraints.builder()
                    .maxDocumentLength(maxSize)
                    .maxNameLength(100)
                    .maxNestingDepth(10)
                    .maxNumberLength(19)
                    .maxStringLength(maxSize)
                    .maxTokenCount(100)
                    .build();
            var streamWriteConstraints =
                    StreamWriteConstraints.builder().maxNestingDepth(100).build();
            var factory = new MappingJsonFactory();
            factory.setStreamReadConstraints(streamReadConstraints);
            factory.setStreamWriteConstraints(streamWriteConstraints);
            builder.factory(factory);
        };
    }
