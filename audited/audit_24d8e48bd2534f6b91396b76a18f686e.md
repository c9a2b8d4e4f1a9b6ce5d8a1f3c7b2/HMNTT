### Title
Unbounded Inner Transaction Fetch and In-Memory Accumulation in `getTransactionsByIdOrHash` Enables Memory-Based DoS

### Summary
`getTransactionsByIdOrHash()` in `rest/transactions.js` fetches all inner transactions of an atomic batch transaction from the database with no application-level cap, spreads the result array directly into the existing `rows` array via `rows.push(...innerTransactionRows)`, and then passes the entire combined array to `formatTransactionRows()` for synchronous in-memory processing. An attacker who creates a large atomic batch transaction on the Hedera network (a one-time, fee-paying setup step) can then repeatedly trigger unbounded memory allocation in the Node.js process by querying the corresponding transaction ID — a completely unprivileged read-only operation.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactionsByIdOrHash()` (lines 923–972):

```js
// lines 944–947: collect ALL inner_transactions from all rows, no size cap
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();

// lines 949–958: second DB query for ALL inner transactions, no limit
if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);

  rows.push(...innerTransactionRows);   // ← spread of unbounded array
}

const transactions = await formatTransactionRows(rows);  // ← full in-memory processing
```

**Root cause:** There is no guard on `innerTransactions.length` before issuing the second query, no `LIMIT` clause in `getTransactionsByTransactionIdsSql` for this code path, and no cap on the combined `rows` array before it is passed to `formatTransactionRows`. The `inner_transactions` column is a raw `bigint[]` with no application-level size enforcement.

**Failed assumption:** The code assumes the number of inner transactions in a batch is small enough to be safely fetched and held in memory in a single request. The Hedera protocol (HIP-551) bounds batch size by the outer transaction's byte budget, not by an explicit inner-transaction count limit enforced at the mirror node REST layer.

**Exploit flow:**
1. Attacker submits an atomic batch transaction to the Hedera network containing the maximum allowed number of inner transactions (each with many crypto/token transfers to maximize per-row size).
2. The importer writes the batch parent row with a large `inner_transactions bigint[]` array and all inner transaction rows to the database.
3. Attacker (or any external user) sends repeated GET requests to `/api/v1/transactions/<batchTransactionId>`.
4. Each request triggers: initial DB query → extraction of all inner transaction keys → second DB query returning N full rows (each with JSON-aggregated transfer lists) → `rows.push(...innerTransactionRows)` → `formatTransactionRows(rows)` holding all N+1 rows in memory simultaneously.
5. Under concurrent load, multiple such requests multiply the memory pressure linearly.

**Why existing checks are insufficient:**

- The `!isTransactionHash` guard (line 949) only prevents the inner-transaction fetch for hash-based lookups; it does not limit the count for ID-based lookups.
- The `scheduled !== undefined` check (line 964) is unrelated to inner transaction count.
- `acceptedSingleTransactionParameters` (line 1014) only allows `nonce` and `scheduled` — no `limit` parameter exists for this endpoint.
- `getTransactionsByTransactionIdsSql` builds a query with one `OR` condition per inner transaction pair and no `LIMIT` clause, so the DB returns all matching rows. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

Each inner transaction row returned from the DB includes JSON-aggregated `crypto_transfer_list`, `token_transfer_list`, and `assessed_custom_fees` subqueries. A batch with N inner transactions causes the Node.js process to allocate memory proportional to N × (row size). Under concurrent requests targeting the same large batch transaction ID, memory consumption multiplies. Additionally, `rows.push(...innerTransactionRows)` uses the JavaScript spread operator, which passes all elements as individual function arguments; for very large arrays (approaching V8's ~65,536 argument limit), this throws `RangeError: Maximum call stack size exceeded`, crashing the request handler. The combined effect is process-level memory exhaustion or crash, causing denial of service for all API consumers — not just the attacker's requests. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

**Setup phase (one-time, requires fees):** Any Hedera account holder can submit an atomic batch transaction. The cost is proportional to the number of inner transactions. This is a realistic capability for a motivated attacker.

**Exploit phase (fully unprivileged, repeatable):** Once the batch transaction exists in the database, any external user — with no credentials, no HBAR, and no special access — can trigger the memory amplification by issuing GET requests to the public REST API. The endpoint requires no authentication. The exploit is trivially repeatable and scriptable. The attacker does not need to be the original submitter of the batch transaction; any large batch transaction created by any user (including legitimate ones) becomes a permanent amplification vector. [6](#0-5) [7](#0-6) 

### Recommendation

1. **Cap inner transaction fetch count:** Before issuing the second query, enforce a hard limit on `innerTransactions.length / 2` (number of inner transactions). Reject or truncate if it exceeds a configured maximum (e.g., 100).
2. **Replace spread with safe concatenation:** Replace `rows.push(...innerTransactionRows)` with `rows = rows.concat(innerTransactionRows)` or `Array.prototype.push.apply(rows, innerTransactionRows)` to avoid the call-stack argument limit.
3. **Add a `LIMIT` clause** in the inner transaction query path of `getTransactionsByTransactionIdsSql`.
4. **Paginate the response:** Apply the same `limit` semantics used by the `/transactions` list endpoint to the inner transaction expansion. [8](#0-7) 

### Proof of Concept

**Precondition setup (one-time):**
```
# Submit an atomic batch transaction to Hedera mainnet/testnet with
# the maximum number of inner transactions (each with many transfers).
# This populates the mirror node DB via the importer.
BATCH_TX_ID="0.0.1234-1700000000-000000000"
```

**Exploit (unprivileged, repeatable):**
```bash
# Trigger unbounded memory allocation on each request
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/transactions/$BATCH_TX_ID" &
done
wait
```

Each concurrent request causes the Node.js process to:
1. Query the DB for the batch parent row (returns `inner_transactions` array of length 2N).
2. Query the DB for all N inner transaction rows (no LIMIT).
3. Spread N rows into `rows` via `rows.push(...innerTransactionRows)`.
4. Process all N+1 rows through `formatTransactionRows()` in memory simultaneously.

Under sufficient concurrency, the Node.js heap exhausts available memory, causing OOM termination or severe GC pressure degrading all API responses. [1](#0-0)

```javascript
const getTransactionsByTransactionIdsSql = (transactionKeys, filters, timestampField) => {
  let payerAccountParams = [];
  let params = [];

  let minTimestamp = constants.MAX_LONG;
  let maxTimestamp = -1;
  let idConditions = [];

  if (transactionKeys.length % 2 !== 0) {
    throw new InvalidArgumentError('transaction keys must be in sequenced pairs of [payer,timestamp]');
  }

  for (let index = 0; index < transactionKeys.length; index += 2) {
    let paramIndex = params.length;
    const payer = transactionKeys[index];
    const timestamp = BigInt(transactionKeys[index + 1]);

    if (payer) {
      payerAccountParams.push(`$${++paramIndex}`);
      idConditions.push(`(${Transaction.PAYER_ACCOUNT_ID} = $${paramIndex} and 
                               ${timestampField} = $${++paramIndex})`);
      params.push(payer);
    } else if (timestampField === Transaction.VALID_START_NS) {
      throw new InvalidArgumentError('payer is required when timestamp is valid_start_ns');
    } else {
      idConditions.push(`${timestampField} = $${++paramIndex}`);
    }
    params.push(timestamp);

    if (timestamp < minTimestamp) {
      minTimestamp = timestamp;
    }
    if (timestamp > maxTimestamp) {
      maxTimestamp = timestamp;
    }
  }

  const maxConsensusTimestamp =
    timestampField === Transaction.CONSENSUS_TIMESTAMP
      ? transactionKeys[transactionKeys.length - 1]
      : maxTimestamp + maxTransactionConsensusTimestampRangeNs;
  const commonConditions = [
    `${Transaction.CONSENSUS_TIMESTAMP} >= $${params.length + 1}`,
    `${Transaction.CONSENSUS_TIMESTAMP} <= $${params.length + 2}`,
  ];
  params.lowerConsensusTimestampIndex = params.length;
  params.push(
    timestampField === Transaction.VALID_START_NS
      ? minTimestamp - config.query.maxValidStartTimestampDriftNs
      : minTimestamp
  );
  params.upperConsensusTimestampIndex = params.length;
  params.push(maxConsensusTimestamp);

  const {conditions: filterConditions, scheduled} = getTransactionByIdQueryParamConditions(filters, params);
  const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
  const subqueryConditions = payerAccountParams.length
    ? [...commonConditions, `${Transaction.PAYER_ACCOUNT_ID} in (${payerAccountParams.join(',')})`].join(' and ')
    : commonConditions.join(' and ');

  return {
    query: getTransactionQuery(mainConditions, subqueryConditions),
    params,
    scheduled,
  };
};
```
