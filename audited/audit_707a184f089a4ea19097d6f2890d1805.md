### Title
Unbounded Concurrent Subscriptions Exhaust `boundedElastic()` Thread Pool via Blocking DB Calls in `PollingTopicListener.poll()`

### Summary
`PollingTopicListener.poll()` wraps a fully synchronous, blocking JDBC call (`getResultList()`) inside `Flux.fromStream()` and schedules it on a shared `Schedulers.boundedElastic()` instance. Because each active subscription holds a scheduler thread for the entire duration of every DB query, an unauthenticated attacker opening a large number of concurrent `subscribeTopic` gRPC streams can saturate the thread pool, causing all subsequent poll tasks to queue indefinitely and halting live message delivery for every subscriber.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (no authentication, no rate limit) → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` → periodic `poll()`. [1](#0-0) 

`listen()` schedules every poll iteration on a single shared `boundedElastic()` scheduler: [2](#0-1) 

`poll()` calls `findByFilter()` and wraps the result in `Flux.fromStream()`: [3](#0-2) 

`findByFilter()` is declared `@Transactional(readOnly = true)` and its implementation calls `typedQuery.getResultList()` — a fully **synchronous, blocking** JDBC call that materialises the entire result set before returning: [4](#0-3) [5](#0-4) 

**Root cause:** `delaySubscription(interval, scheduler)` causes the subscription to `Flux.defer(() -> poll(context))` to execute on a `boundedElastic()` worker thread. Because `poll()` immediately calls the blocking `getResultList()`, that worker thread is held for the full duration of the DB query. There is no `.timeout()` on the poll `Flux`, no per-subscription concurrency cap, and no rate limiting at the gRPC layer.

**Why existing checks are insufficient:**

- `maxPageSize = 5000` limits the result-set size but does not bound query *execution time*. Under DB load induced by the attack itself, a `LIMIT 5000` query can take seconds.
- `Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads (e.g. 80 on an 8-core host). Once all threads are occupied, new poll tasks queue in the 100 000-task internal queue; they do not execute until a thread is freed, causing unbounded delivery delay.
- No authentication or per-IP connection limit exists on `subscribeTopic`. [6](#0-5) 

### Impact Explanation

Every active `POLL`-mode subscription periodically occupies a `boundedElastic()` thread for the duration of a blocking DB query. With enough concurrent subscriptions the thread pool is fully saturated. New poll tasks queue but never execute within the polling interval, so live topic messages are not delivered to any subscriber. Because the same scheduler is shared across all subscriptions for a single `PollingTopicListener` bean, the impact is global — all subscribers on the node are affected simultaneously. This constitutes a complete denial of live message delivery (network-level transaction confirmation visibility) for the duration of the attack.

### Likelihood Explanation

The `subscribeTopic` gRPC endpoint is unauthenticated and publicly reachable. An attacker needs only a gRPC client (e.g. `grpcurl`) and the ability to open many TCP connections. The attack is self-amplifying: each new subscription adds DB load, which slows existing queries, which holds threads longer, which makes it easier for the next batch of subscriptions to saturate the pool. The attack is repeatable and requires no special knowledge beyond the public protobuf definition. The only prerequisite is that the operator has configured `hiero.mirror.grpc.listener.type=POLL` (non-default but a documented option). [7](#0-6) 

### Recommendation

1. **Add a per-poll timeout**: apply `.timeout(listenerProperties.getInterval(), scheduler)` to the `Flux` returned by `poll()` so a slow query cannot hold a thread beyond one polling interval.
2. **Move blocking work off the scheduler**: wrap `findByFilter()` in `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())` with an explicit `.timeout()`, or switch to a non-blocking R2DBC driver.
3. **Limit concurrent subscriptions**: enforce a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
4. **Rate-limit at the gRPC layer**: use a server interceptor to cap connections per source IP.

### Proof of Concept

```bash
# 1. Ensure the mirror node is configured with:
#    hiero.mirror.grpc.listener.type=POLL

# 2. Open N concurrent subscriptions (no auth required):
for i in $(seq 1 300); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 3. Each subscription triggers a blocking getResultList() on a boundedElastic() thread
#    every 500 ms (default interval). With 300 concurrent subscriptions and DB under load,
#    all boundedElastic() threads become occupied.

# 4. Observe: legitimate subscribers stop receiving new topic messages.
#    Metrics: hiero.mirror.grpc.subscribers rises; poll tasks queue but do not execute.
#    New messages submitted to the network are not delivered via the gRPC stream.
```

```java
    private final Scheduler scheduler = Schedulers.boundedElastic();

    @Override
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
