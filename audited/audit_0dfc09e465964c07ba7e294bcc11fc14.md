### Title
Integer Overflow in `getInclusiveValue()` Allows Bypass of `hook.id=gt:` Filter, Returning All Hooks

### Summary
`NumberRangeParameter.getInclusiveValue()` converts a `GT` operator to an inclusive lower bound by computing `value + 1` with no overflow guard. When a user supplies `hook.id=gt:9223372036854775807` (`Long.MAX_VALUE`), the addition wraps to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` then resolves to `0L`, so the intended "return nothing" filter silently becomes "return all hooks," exposing every hook record owned by the target account.

### Finding Description

**Overflow site** — `NumberRangeParameter.getInclusiveValue()`:

```java
// NumberRangeParameter.java line 35-43
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        return value + 1;          // ← no overflow guard
    } else if (operator == RangeOperator.LT) {
        return value - 1;
    } else {
        return value;
    }
}
``` [1](#0-0) 

**Input validation** — `getNumberValue()` only rejects negative values; `Long.MAX_VALUE` (9223372036854775807) is positive and accepted:

```java
private static long getNumberValue(String number) {
    var value = Long.parseLong(number);
    if (value < 0) {                   // ← no upper-bound check
        throw new IllegalArgumentException("Invalid range value");
    }
    return value;
}
``` [2](#0-1) 

**Consumption site** — `HooksController.hooksRequest()`:

```java
// HooksController.java line 135-142

...

    lowerBound = Math.max(lowerBound, hookIdFilter.getInclusiveValue());
``` [3](#0-2) 

**Exploit flow:**

1. Attacker sends `GET /api/v1/accounts/0.0.123/hooks?hook.id=gt:9223372036854775807`
2. `getNumberValue("9223372036854775807")` → `Long.MAX_VALUE` (passes `< 0` check)
3. `getInclusiveValue()` → `Long.MAX_VALUE + 1` → **`Long.MIN_VALUE`** (signed overflow)
4. `Math.max(0L, Long.MIN_VALUE)` → `0L` (lowerBound unchanged)
5. `HooksRequest` built with `lowerBound=0L`, `upperBound=Long.MAX_VALUE`
6. `HookServiceImpl.getHooks()` calls `findByOwnerIdAndHookIdBetween(ownerId, 0L, Long.MAX_VALUE, page)` — returns **all hooks** [4](#0-3) 

**Secondary overflow site** — `LinkFactoryImpl.isEmptyRange()` has the identical `Long.parseLong(value.substring(3)) + 1` pattern and also fails to detect the empty range, so no early-exit guard fires either: [5](#0-4) 

### Impact Explanation
An unauthenticated (or any low-privilege) caller can retrieve the complete set of smart-contract hook records for any account by issuing a single crafted GET request. Hook records include `contract_id`, `extension_point`, `admin_key`, and lifecycle timestamps — data that should only be visible within the caller's intended filter window. Because hook data describes which contracts intercept account operations, leaking it aids targeted attacks against those contracts.

### Likelihood Explanation
No authentication or special privilege is required. The endpoint is publicly reachable (`GET /api/v1/accounts/{ownerId}/hooks`). The payload is a single query parameter with a well-known integer boundary value. Any attacker familiar with Java's `Long.MAX_VALUE` can reproduce this in seconds with `curl`. It is fully repeatable and leaves no distinguishing trace beyond a normal API request.

### Recommendation
Add an explicit overflow guard in `getInclusiveValue()`:

```java
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        if (value == Long.MAX_VALUE) {
            throw new IllegalArgumentException(
                "gt:" + value + " produces an empty range (no value exceeds Long.MAX_VALUE)");
        }
        return value + 1;
    } else if (operator == RangeOperator.LT) {
        if (value == Long.MIN_VALUE) {
            throw new IllegalArgumentException(
                "lt:" + value + " produces an empty range");
        }
        return value - 1;
    }
    return value;
}
```

Alternatively, add an upper-bound check in `getNumberValue()` to reject values that would overflow when incremented (e.g., reject `value >= Long.MAX_VALUE` for `GT` contexts). Apply the same fix to `LinkFactoryImpl.isEmptyRange()`.

### Proof of Concept

```bash
# Returns ALL hooks for account 0.0.123 instead of an empty list
curl -s "https://<mirror-node>/api/v1/accounts/0.0.123/hooks?hook.id=gt:9223372036854775807"

# Expected: {"hooks":[],"links":{}}
# Actual:   {"hooks":[...all hook records...],"links":{...}}
```

Step-by-step:
1. Identify any valid `ownerId` (e.g., from a public transaction).
2. Issue the request above with `hook.id=gt:9223372036854775807`.
3. Observe that the response contains all hooks for that owner rather than an empty list.
4. Confirm by comparing with `hook.id=gte:9223372036854775807` (which correctly returns only hooks with `hookId == Long.MAX_VALUE`, i.e., effectively nothing in practice).
