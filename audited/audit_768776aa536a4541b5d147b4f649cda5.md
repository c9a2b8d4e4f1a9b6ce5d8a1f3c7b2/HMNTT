### Title
`isValidAccount()` Accepts Non-Account Entity IDs, Returning Incorrect Token Relationship Data

### Summary
`getTokenRelationships()` in `rest/controllers/tokenController.js` uses `EntityService.isValidAccount()` to gate access, but `isValidAccount()` only checks entity existence — not entity type. An unprivileged attacker can supply the numeric ID of a TOPIC, TOKEN, FILE, or SCHEDULE entity and receive HTTP 200 with an empty token list instead of a 404, producing semantically incorrect data that smart contracts or downstream consumers may misinterpret.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–71: [1](#0-0) 

```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValidAccount = await EntityService.isValidAccount(accountId);
  if (!isValidAccount) {
    throw new NotFoundError();
  }
```

`rest/service/entityService.js`, `isValidAccount()`, lines 60–63: [2](#0-1) 

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
```

`entityExistenceQuery`, lines 28–30: [3](#0-2) 

```sql
select ${Entity.TYPE}
from ${Entity.tableName}
where ${Entity.ID} = $1
```

**Root cause:** The SQL query selects the `type` column but applies **no `WHERE type IN ('ACCOUNT', 'CONTRACT')` filter**. `isValidAccount()` only checks `!isNil(entity)` — it never inspects the returned `type` value. Any entity in the `entity` table (TOPIC, TOKEN, FILE, SCHEDULE) will satisfy the check.

**Contrast with correct patterns elsewhere:**
- `accounts.js` line 145 explicitly filters: `e.type in ('ACCOUNT', 'CONTRACT')` [4](#0-3) 
- GraphQL `AccountController.java` uses `getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` [5](#0-4) 

**Exploit flow:**
1. Attacker identifies any known TOPIC, TOKEN, FILE, or SCHEDULE entity ID (e.g., `0.0.1234` which is a TOPIC).
2. Sends `GET /api/v1/accounts/0.0.1234/tokens`.
3. `getEncodedId("0.0.1234")` succeeds — it's a valid entity ID format.
4. `isValidAccount(encodedId)` queries the entity table, finds the TOPIC row, returns `true`.
5. `TokenService.getTokenAccounts(query)` runs against `token_account` for that ID — returns empty.
6. API responds HTTP 200: `{"tokens": [], "links": {"next": null}}`.

The entity type enum confirms the exploitable types: [6](#0-5) 

### Impact Explanation
Smart contracts or off-chain services that call this mirror node API to determine whether an entity holds token associations will receive a valid HTTP 200 response with an empty token list for a non-account entity. This is semantically incorrect: the correct response is 404 (entity is not an account). A contract that interprets "empty token list" as "account exists but holds no tokens" will make incorrect state decisions — for example, incorrectly concluding a topic or schedule entity is a valid account with zero token balances. This matches the stated severity of "unintended smart contract behavior with no concrete funds at direct risk."

### Likelihood Explanation
No authentication or special privilege is required. Entity IDs for topics, tokens, files, and schedules are publicly enumerable via other mirror node REST endpoints (`/api/v1/topics`, `/api/v1/tokens`, etc.). The attack is trivially repeatable with a single HTTP GET request. Any developer or automated system integrating with the mirror node API is a potential victim.

### Recommendation
Fix `isValidAccount()` to check the returned entity type, not just existence. The `entityExistenceQuery` already selects `Entity.TYPE` — use it:

```js
// rest/service/entityService.js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity) && (entity.type === 'ACCOUNT' || entity.type === 'CONTRACT');
}
```

Alternatively, add a type filter directly to the SQL query:

```sql
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1
                                 and ${Entity.TYPE} in ('ACCOUNT', 'CONTRACT')`;
```

### Proof of Concept
**Precondition:** A TOPIC entity with ID `0.0.5678` exists in the mirror node database (verifiable via `GET /api/v1/topics/0.0.5678`).

**Steps:**
1. `curl -s "https://<mirror-node>/api/v1/accounts/0.0.5678/tokens"`
2. **Expected (correct):** HTTP 404 — entity is not an account.
3. **Actual (vulnerable):** HTTP 200 — `{"tokens":[],"links":{"next":null}}` — the API treats the TOPIC as a valid account with no token relationships.

Repeat with any FILE (`0.0.101`, `0.0.102`), SCHEDULE, or TOKEN entity ID to confirm the same behavior.
