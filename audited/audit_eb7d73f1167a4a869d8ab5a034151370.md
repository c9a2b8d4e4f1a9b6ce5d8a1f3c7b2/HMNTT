### Title
Unauthenticated Compression-Based CPU Exhaustion (DoS) via Unbounded Concurrent Gzip Requests to `/api/v1/transactions`

### Summary
The REST API applies `compression()` globally with no options (no concurrency cap, default zlib level 6) and has no application-level rate limiting or in-flight request limiting for the `/api/v1/transactions` endpoint. An unprivileged external attacker can flood the server with concurrent `Accept-Encoding: gzip` requests for maximum-size transaction responses, saturating Node.js's libuv thread pool and CPU, denying service to legitimate users.

### Finding Description
**Code location:** `rest/server.js` lines 76–79:
```js
if (config.response.compression) {
  logger.info('Response compression is enabled');
  app.use(compression());
}
```
`compression()` is invoked with zero options. The default configuration is:
- Compression level: `zlib.Z_DEFAULT_COMPRESSION` (level 6 — CPU-intensive)
- Threshold: 1024 bytes (any transaction list response exceeds this)
- No concurrency limit, no per-IP filter, no request-rate guard

`hiero.mirror.rest.response.compression` defaults to `true` (docs/configuration.md line 603), so this middleware is active in all default deployments.

The `/api/v1/transactions` route (`rest/server.js` line 132) accepts an unauthenticated `GET` with `limit` up to 100 (the default max, docs/configuration.md line 608). Each response for 100 transactions with full transfer/NFT/memo fields is easily 100–500 KB of JSON. The compression middleware compresses every such response synchronously within Node.js's async I/O pipeline using zlib, which offloads to libuv worker threads but still consumes all available CPU cores.

**No application-level rate limiting exists in the Node.js REST API.** The `ThrottleManagerImpl`/`ThrottleConfiguration` rate limiting is exclusively in the `web3` Java service. The Helm chart middleware for the REST service (`charts/hedera-mirror-rest/values.yaml` lines 134–139) only configures `circuitBreaker` and `retry` — there is no `inFlightReq` or `rateLimit` Traefik middleware, unlike the Rosetta chart which has both (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160).

**Exploit flow:**
1. Attacker opens N concurrent TCP connections (no authentication required).
2. Each sends: `GET /api/v1/transactions?limit=100 HTTP/1.1` with `Accept-Encoding: gzip`.
3. Each request hits the DB (up to 10 pool connections), retrieves 100 transactions, and the compression middleware compresses the large JSON payload at zlib level 6.
4. With many concurrent compressions, all libuv worker threads are saturated; the Node.js event loop stalls; new requests queue indefinitely.

### Impact Explanation
Node.js is single-process and single-event-loop. Saturating the libuv thread pool with concurrent zlib compressions causes the event loop to stall, making the entire REST API unresponsive. Legitimate transaction queries time out or receive no response. The DB pool limit (10 connections) does not prevent compression CPU exhaustion — the bottleneck is CPU, not DB. This is a complete availability loss for the `/api/v1/transactions` endpoint and all other REST endpoints sharing the same process.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The endpoint is publicly accessible. The attack requires only a standard HTTP client capable of sending concurrent requests with `Accept-Encoding: gzip`. The response size is bounded at 100 records but that is sufficient to produce large compressible payloads. The attack is trivially repeatable and automatable with tools like `ab`, `wrk`, or a simple script. The absence of `inFlightReq` or `rateLimit` in the REST Traefik middleware (unlike Rosetta) means there is no infrastructure-level backstop either.

### Recommendation
1. **Pass options to `compression()`** to limit CPU impact: set `level: zlib.Z_BEST_SPEED` (level 1) or `level: -1` with a higher `threshold` (e.g., 10 KB) to avoid compressing small responses.
2. **Add an in-flight request limit** to the Traefik middleware for the REST chart, mirroring the Rosetta chart's `inFlightReq` configuration (e.g., `amount: 50` per source IP).
3. **Add a per-IP rate limit** to the Traefik middleware for the REST chart (e.g., `rateLimit.average: 100` per second per source IP).
4. **Consider application-level rate limiting** (e.g., `express-rate-limit`) as a defense-in-depth measure independent of the ingress layer.

### Proof of Concept
```bash

  "https://<mirror-node-host>/api/v1/transactions?limit=100"

# Observe: response times for legitimate requests spike to timeouts;
# Node.js process CPU reaches 100%; health endpoint /health/liveness
