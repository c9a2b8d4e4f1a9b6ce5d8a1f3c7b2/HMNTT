### Title
Unauthenticated CPU Exhaustion via Unbounded `findUser()` Invocation in `authHandler` with No Rate Limiting

### Summary
The `authHandler` middleware in `rest/middleware/authHandler.js` invokes `findUser()` on every request that carries an `Authorization: Basic` header, performing a full linear scan of `config.users` using constant-time `tsscmp` comparisons with no rate limiting, IP-based throttling, or auth-result caching in the REST API layer. Any unprivileged external attacker can flood the server with crafted Basic Auth requests, saturating Node.js's single-threaded event loop with synchronous CPU work and degrading or halting transaction query processing.

### Finding Description

**Exact code path:**

`rest/middleware/authHandler.js` lines 10–13 — `findUser()` iterates the entire `config.users` array unconditionally:

```js
const findUser = (username, password) => {
  const users = config.users || [];
  return users.find((user) => tsscmp(user.username, username) && tsscmp(user.password, password)) || null;
};
``` [1](#0-0) 

`authHandler` is registered globally for all routes in `rest/server.js` line 86, before the response cache check (line 97):

```js
app.useExt(authHandler);          // line 86 — runs on every request
...
app.useExt(responseCacheCheckHandler); // line 97 — cache bypass never reached
``` [2](#0-1) 

**Root cause:** `findUser()` is called synchronously on every request bearing a `Basic` auth header. `tsscmp` is a constant-time comparator — it always runs to full completion regardless of match, making it more expensive than a short-circuit comparison. With N configured users, each malicious request costs 2N `tsscmp` calls minimum (username comparison for all N users, password comparison only on username match). No result is cached between requests.

**Failed assumption:** The code assumes only legitimate clients will send `Authorization: Basic` headers, and that the user list is small enough that a linear scan is acceptable. Neither assumption is enforced.

**Why existing checks fail:**

- A grep across all of `rest/**` for `rateLimit`, `rate-limit`, `express-rate`, `helmet`, or `slowDown` returns **zero matches** — there is no rate limiting in the REST API layer. [3](#0-2) 

- The throttling that exists (`ThrottleConfiguration`, `ThrottleManagerImpl`) is in the `web3` Java service only — it does not protect the Node.js REST API. [4](#0-3) 

- The response cache (`responseCacheCheckHandler`) is registered **after** `authHandler`, so cached responses do not bypass `findUser()`. [5](#0-4) 

### Impact Explanation

Node.js runs on a single-threaded event loop. Synchronous CPU work in middleware directly blocks processing of all other requests. An attacker flooding the server with Basic Auth requests forces repeated `tsscmp` iterations on every event-loop tick. At sufficient request volume, legitimate transaction queries queue behind auth processing, causing latency spikes and eventual timeout failures for all API consumers. Because `authHandler` is global and pre-cache, no route is exempt.

### Likelihood Explanation

The attack requires zero privileges — any internet-accessible client can send an HTTP request with an `Authorization: Basic <base64>` header. No authentication, account, or special capability is needed. The attack is trivially scriptable (`ab`, `wrk`, `curl` in a loop) and fully repeatable. The absence of any rate limiting or IP-based throttling in the REST layer means there is no server-side mechanism to slow or block the flood.

### Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) applied globally or specifically before `authHandler`, keyed by IP address, to cap requests per second per client.
2. **Cache auth results** — store a short-lived (e.g., 5–30 second) in-memory map of `hash(username+password) → user` so repeated requests from the same credentials skip the full scan.
3. **Fail fast on unknown usernames** — build a `Map<username, user>` at startup instead of a linear array scan; a single O(1) lookup replaces N `tsscmp` calls.
4. **Move `authHandler` after `responseCacheCheckHandler`** so cached responses bypass auth processing entirely.

### Proof of Concept

```bash
# Generate a random invalid Basic Auth credential
CREDS=$(echo -n "attacker:wrongpassword" | base64)

# Flood the server with 10,000 concurrent requests bearing invalid Basic Auth
ab -n 100000 -c 500 -H "Authorization: Basic ${CREDS}" \
   http://<mirror-node-rest-host>/api/v1/transactions

# Observe: legitimate requests begin timing out or returning 503
# as the Node.js event loop saturates with findUser() iterations
curl http://<mirror-node-rest-host>/api/v1/transactions
# → connection timeout or severely degraded response time
```

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
```
