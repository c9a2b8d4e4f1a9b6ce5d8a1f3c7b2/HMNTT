### Title
Cache-Timing Side-Channel in `topicExists()` Allows Enumeration of Recently-Accessed Topic IDs

### Summary
The `topicExists()` method in `TopicMessageServiceImpl` calls `entityRepository.findById()`, which is backed by a Caffeine in-process cache with a 24-hour write-expiry. Because the `@Cacheable` annotation's `unless = "#result == null"` condition never triggers (since `Optional<Entity>` is never `null`), both hits and misses are cached. An unauthenticated attacker can probe the publicly-accessible `subscribeTopic` gRPC endpoint with previously-unseen topic IDs and measure response latency to determine whether another user has recently subscribed to that topic, leaking which topics are actively monitored.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicExists()` (line 87) → `entityRepository.findById(topicId.getId())` (line 96). [1](#0-0) 

`EntityRepository.findById()` is annotated: [2](#0-1) 

The backing cache is a Caffeine instance configured with `expireAfterWrite=24h` and `maximumSize=50000`: [3](#0-2) 

**Root cause — failed assumption in `unless` condition:**

The `@Cacheable` condition `unless = "#result == null"` is intended to avoid caching absent results. However, `findById` returns `Optional<Entity>`, which is **never** `null` — it is either `Optional.empty()` or `Optional.of(entity)`. Therefore the condition never fires, and both "topic found" and "topic not found" results are cached on first access. After the first call for any topic ID, all subsequent calls within 24 hours are served from the in-memory Caffeine cache (nanosecond latency), while a first-ever call requires a PostgreSQL round-trip (millisecond latency). This creates a measurable timing oracle.

**Exploit flow:**

1. Attacker selects a topic ID `T` they have never queried before.
2. Attacker sends a `subscribeTopic` gRPC request for `T` and measures time-to-first-response (error or stream open).
3. **Fast response (~microseconds)** → cache hit → another client already called `findById(T)` within the last 24 hours → topic `T` is actively being subscribed to.
4. **Slow response (~milliseconds, DB round-trip)** → cache miss → no recent subscriber for `T`.
5. Attacker repeats across the topic ID space to build a map of "hot" topics.

**Why existing checks fail:**

- `unless = "#result == null"` does not prevent caching of `Optional.empty()`, so the timing oracle applies to both existing and non-existing topics.
- There is no authentication, rate-limiting, or request throttling on the `subscribeTopic` gRPC endpoint. [4](#0-3) 
- The gRPC README confirms the endpoint is publicly accessible with no credentials: [5](#0-4) 

### Impact Explanation

An unprivileged attacker can enumerate which HCS topic IDs have been recently subscribed to by legitimate clients within the past 24 hours. In the Hedera ecosystem, topics are application-specific (e.g., a topic per dApp, per token, per audit trail). Knowing which topics are "hot" leaks:
- Which applications or services are actively in use
- Subscription patterns that correlate with transaction submission activity (a topic receiving new subscribers typically means new `ConsensusSubmitMessage` transactions are expected)
- Potential targets for further DoS or subscription-flooding attacks

The default cache size is 50,000 entries, meaning up to 50,000 distinct topic IDs can be fingerprinted before eviction begins. [6](#0-5) 

### Likelihood Explanation

- **No privileges required**: any internet-accessible gRPC client can call `subscribeTopic`.
- **Timing gap is large**: Caffeine in-memory lookup is O(1) nanoseconds; a PostgreSQL query over a local network is typically 1–50 ms. This ~4-order-of-magnitude difference is reliably detectable even with network jitter using simple statistical averaging over a handful of repeated probes.
- **Repeatable and automatable**: a script can sweep topic ID ranges systematically.
- **Limitation**: the cache is per-JVM instance. In a horizontally-scaled deployment, the attacker must consistently hit the same replica (e.g., via sticky routing or by probing each replica independently).

### Recommendation

1. **Fix the `unless` condition** to correctly exclude empty Optionals:
   ```java
   @Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE,
              unless = "#result == null || !#result.isPresent()")
   ```
   This prevents caching of "not found" results, but does not by itself eliminate the timing oracle for cached hits vs. uncached misses.

2. **Add rate-limiting** on the `subscribeTopic` gRPC endpoint (e.g., via a gRPC interceptor or a token-bucket per source IP) to make bulk enumeration impractical.

3. **Reduce cache TTL** from 24 hours to a shorter window (e.g., 5–15 minutes) to limit the enumeration window.

4. **Consider constant-time response padding** or a uniform minimum response delay for the topic-existence check to eliminate the timing signal entirely.

### Proof of Concept

```bash
# Probe a topic ID never queried before (cold cache → slow)
time grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 99999}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

# Probe the same topic ID immediately after (warm cache → fast)
time grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 99999}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: <1ms (Caffeine cache hit)

# To detect another user's activity: probe a topic ID you have NEVER
# probed before. If the first probe is fast, the cache was already warm
# from another subscriber — that topic is actively being used.
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
