### Title
TOCTOU Race Condition in `RedisTopicListener`: Blind `remove()` in `unsubscribe()` Orphans Shared Flux and Creates Message-Loss Window

### Summary
`unsubscribe()` calls `topicMessages.remove(topic.getTopic())` unconditionally, without checking whether the value being removed is the same `Flux` that triggered the callback. Because `doOnCancel`/`doOnComplete` fire asynchronously on a scheduler thread while `getSharedListener()` can concurrently call `computeIfAbsent()` on another thread, the blind `remove()` can evict a freshly-inserted, live `Flux` from the map, leaving it orphaned and opening a gap during which all messages published to that Redis channel are silently dropped.

### Finding Description

**Exact code path**

`getSharedListener()` (line 61) inserts a new shared `Flux` F2 via `computeIfAbsent()`: [1](#0-0) 

`subscribe()` registers `doOnCancel`/`doOnComplete` hooks that call `unsubscribe()` (lines 74-75): [2](#0-1) 

`unsubscribe()` performs a blind, key-only `remove()` (line 83): [3](#0-2) 

**Root cause**

`ConcurrentHashMap.computeIfAbsent()` is atomic for a single check-and-insert, but there is no atomicity across the pair (`computeIfAbsent` → later `remove`). The `doOnCancel` callback fires asynchronously (on a Reactor scheduler thread) after `.share()` cancels its upstream, meaning the `remove()` can execute *after* a concurrent `computeIfAbsent()` has already placed a brand-new `Flux` F2 into the map. The single-argument `remove(key)` deletes whatever is currently mapped to that key — it does not verify the value is the stale F1.

**Exploit flow (race window)**

```
Thread A (attacker gRPC cancel):          Thread B (new legitimate subscriber):
  F1 is in map, refCount → 0
  .share() cancels upstream
  doOnCancel fires asynchronously
                                            computeIfAbsent() → F1 still present
                                            → returns F1 (or inserts F2 if F1 gone)
  unsubscribe() → remove("topic.X")
  ← removes F2 (the new live Flux!) ←
                                            Subscriber B holds reference to F2
                                            but F2 is no longer in the map
  [map is now empty for topic.X]
                                            Thread C: computeIfAbsent() → creates F3
                                            F2 (Subscriber B) and F3 (Subscriber C)
                                            are both active → resource leak
                                            AND gap window where no map entry exists
```

During the gap window (between `remove()` completing and the next `computeIfAbsent()` inserting F3), any messages published to the Redis channel `topic.X` are received by nobody and permanently lost.

**Why existing checks are insufficient**

`ConcurrentHashMap` only guarantees atomicity of individual operations. The retry logic on the new `Flux`: [4](#0-3) 

uses a backoff with `maxBackoff = interval × 4`. With the default `interval = 500 ms`: [5](#0-4) 

the new subscription can take up to **2 000 ms** to re-establish its Redis channel subscription. No guard prevents `unsubscribe()` from removing a freshly-inserted entry.

### Impact Explanation

Any messages published to the affected Redis channel during the gap window are permanently lost — they are fire-and-forget pub/sub messages with no replay. Subscribers that land on an orphaned `Flux` (F2 evicted from the map) will receive messages until they cancel, but new subscribers will create yet another parallel Redis subscription (F3, F4, …), causing unbounded resource leaks (open Redis channel subscriptions, threads, memory). The reconnection backoff of up to 2 000 ms means a sustained attack can keep the effective delivery gap open continuously, causing all live-stream consumers of a topic to miss consensus messages for the duration.

### Likelihood Explanation

The attacker requires only an unprivileged gRPC connection — no authentication, no special role. The standard `subscribeTopic` gRPC endpoint is publicly reachable. The attacker subscribes to a target topic, waits until they are the sole subscriber (easily verified by timing or by being the first subscriber on a low-traffic topic), then abruptly closes the TCP connection. This triggers `doOnCancel` and the race. The attack is trivially repeatable in a tight loop: subscribe → cancel → subscribe → cancel, keeping the map entry perpetually absent and preventing any stable shared subscription from forming. No exploit tooling beyond a standard gRPC client is needed.

### Recommendation

Replace the single-argument `remove(key)` with the two-argument `remove(key, value)` form, passing the specific `Flux` instance that registered the callback. This ensures only the stale entry is evicted:

```java
// In subscribe(), capture the Flux reference before returning:
Flux<TopicMessage> sharedFlux = container
        .flatMapMany(...)
        ...
        .share();

// Pass the reference into the hook:
// doOnCancel(() -> topicMessages.remove(topic.getTopic(), sharedFlux))

// unsubscribe becomes:
private void unsubscribe(Topic topic, Flux<TopicMessage> flux) {
    topicMessages.remove(topic.getTopic(), flux);
}
```

Because `ConcurrentHashMap.remove(key, value)` only removes the entry if the current mapped value is `==` to the provided value, a concurrently-inserted F2 will not be evicted by F1's cancellation callback.

### Proof of Concept

**Preconditions:** Mirror node running with `ListenerType.REDIS`; attacker has network access to the gRPC port; target topic has no other active subscribers (or attacker waits for a low-traffic window).

**Steps:**

1. Open gRPC stream S1 to `subscribeTopic` for `topicId=X`. This causes `computeIfAbsent` to insert F1 into `topicMessages`.
2. Immediately close S1 (TCP RST or clean close). `.share()` sees refCount → 0, cancels upstream, fires `doOnCancel` asynchronously.
3. Concurrently (tight loop), open stream S2 for the same `topicId=X`. Depending on scheduler timing, `computeIfAbsent` may return F1 (still in map) or insert F2.
4. `unsubscribe()` fires and calls `topicMessages.remove("topic.X")`, evicting whichever Flux is currently mapped — potentially F2.
5. Publish a consensus message to Redis channel `topic.X` during the gap. Observe that S2 (or any other active subscriber) does not receive it.
6. Repeat steps 1–5 in a loop. Observe via metrics (`hiero.mirror.grpc.subscribers`) that subscriber count oscillates while messages are continuously dropped, and that multiple parallel Redis channel subscriptions accumulate (resource leak visible in Redis `CLIENT LIST` or connection metrics).
