### Title
Unauthenticated Access to Metrics Endpoint When `authentication: false` Bypasses All Credential Checks

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` unconditionally returns `true` when `config.metrics.config.authentication` is `false`, skipping the `basicAuth` credential check entirely. Any unprivileged external user can then GET the metrics endpoint and receive the full Prometheus scrape output, including `api_request` counter label sets (`method`, `path`, `code`) that reveal API usage patterns across all routes.

### Finding Description
**Exact code path:**

In `rest/middleware/metricsHandler.js`, lines 137–144:

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ← credential check is never reached
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

When `authentication` is falsy, the function returns `true` before `basicAuth(req)` is ever called. The gate in `metricsMiddleware` (lines 156–165) then serves the full Prometheus payload to the caller:

```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) {          // always false → never blocks
    res.set('WWW-Authenticate', 'Basic realm="Metrics"');
    return res.status(401).send('Unauthorized');
  }
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));   // full dump
  });
}
```

The per-route `requestTotalCounter` (`api_request`) is recorded at line 207 with labels `{method, path, code}` (assembled at lines 190–192), and all four per-route instruments (`api_request`, `api_request_duration_milliseconds`, `api_request_size_bytes`, `api_response_size_bytes`) share those same labels. The serialized output therefore enumerates every distinct `(method, path, status_code)` tuple that has been observed since the process started.

**Root cause / failed assumption:** The design assumes that setting `authentication: false` is a safe, intentional operator choice. There is no secondary access control (network policy enforcement, IP allowlist, or separate listener) enforced in code, so the assumption that "only trusted scrapers reach this endpoint" is not guaranteed by the application itself.

### Impact Explanation
An attacker who can reach the metrics endpoint (default path `/swagger/metrics/`) with `authentication: false` receives:

- A complete enumeration of every API path that has received traffic (`path` label), including low-volume or internal-only routes.
- Per-path HTTP method distribution and response-code breakdown, revealing which endpoints return errors and at what rate.
- Request/response size histograms per route, which can fingerprint the type and volume of data being exchanged.
- In-flight request counts and CPU/memory gauges, enabling timing-based inference of backend load.

For a mirror-node deployment used by a custodian or exchange, the `path` label values (e.g., `/api/v1/transactions`, `/api/v1/accounts/{id}`, `/api/v1/schedules/{id}`) combined with request rates and error codes allow an attacker to infer which account IDs are queried most frequently, when activity spikes occur, and which scheduled-transaction or token endpoints are in active use — information useful for targeted social engineering or timing attacks.

Severity: **Medium** (information disclosure; no direct fund loss, but meaningful operational intelligence leak).

### Likelihood Explanation
- The configuration default is `authentication: true` (documented in `docs/configuration.md` line 569), so the vulnerable state requires an explicit operator change.
- However, operators commonly disable authentication for internal Prometheus scrapers in Kubernetes environments where they believe network segmentation is sufficient. If the metrics port is inadvertently exposed (misconfigured `Service`/`Ingress`, cloud load-balancer, or lateral movement within a cluster), the bypass requires zero credentials and a single HTTP GET — no exploit tooling needed.
- The attack is fully repeatable and leaves no application-level log entry distinguishable from a legitimate scrape.

### Recommendation
1. **Remove the unconditional `return true` shortcut.** When `authentication` is `false`, either still require the request to originate from a localhost/loopback address, or document that the endpoint must be protected at the network layer and add a startup warning log.
2. **Alternatively**, replace the boolean flag with an explicit `allowlist` of source CIDRs checked in `authenticate()`, so disabling password auth does not imply disabling all access control.
3. **Minimum fix** — add a log-level `warn` at startup when `authentication: false` so operators are alerted that the endpoint is open.
