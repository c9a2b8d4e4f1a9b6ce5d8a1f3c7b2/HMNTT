### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `/network/status` Requests Due to Disabled Rate-Limiting Middleware

### Summary
`NetworkStatus()` in `rosetta/app/services/network_service.go` issues two sequential DB queries per request (`RetrieveLatest` and `addressBookEntryRepo.Entries`), each acquiring a connection from a pool capped at 100 (`maxOpenConnections`). The Traefik middleware chain that enforces per-IP in-flight limits and rate limits is **disabled by default** (`global.middleware: false`), leaving no application-level concurrency control. An unprivileged attacker sending ≥100 concurrent requests can exhaust the pool, causing all subsequent requests to block until the 20-second statement timeout fires, effectively denying all clients access to network state.

### Finding Description

**Code path:**

`NetworkStatus()` [1](#0-0)  calls `RetrieveLatest(ctx)` [2](#0-1)  which calls `GetDbWithContext(ctx)` and executes `selectLatestWithIndex` via `db.Raw(...).First(rb)`. It then calls `addressBookEntryRepo.Entries(ctx)`, which also calls `GetDbWithContext` and executes its own query. Each call acquires a connection from the shared pool for the duration of query execution.

**Root cause — `GetDbWithContext` and pool behavior:**

`GetDbWithContext` does not open a transaction; it creates a context-scoped GORM handle. [3](#0-2)  When `statementTimeout > 0` (default: 20 s), a child context with a 20-second deadline is created. The underlying `database/sql` pool acquires a physical connection at query start and holds it until the query completes or the 20-second context deadline fires. Under DB load, queries slow down; connections are held longer; the pool drains faster.

**Pool configuration:**

`SetMaxOpenConns(100)` is the hard ceiling. [4](#0-3)  When all 100 slots are occupied, `database/sql` blocks the goroutine waiting for a free connection. There is no `WaitTimeout` configured, so goroutines block indefinitely (bounded only by the HTTP write timeout of 10 s or the 20-second statement-timeout context).

**Failed assumption — middleware disabled by default:**

The Helm chart defines a Traefik middleware chain with `inFlightReq: amount: 5` per source IP and `rateLimit: average: 10` per host. [5](#0-4)  However, the middleware template is gated on `global.middleware`, which defaults to `false`. [6](#0-5)  The template renders nothing when this flag is off: [7](#0-6)  meaning the ingress annotation referencing the middleware chain points to a non-existent CRD, and Traefik silently passes all traffic through without enforcement. There is no rate-limiting or concurrency-limiting code anywhere in the Go application itself.

**Exploit flow:**

1. Attacker opens 100+ persistent HTTP connections and sends concurrent POST `/network/status` requests.
2. Each request triggers two `GetDbWithContext` calls, each holding a pool connection for the query duration.
3. Under the resulting DB load, query latency rises; connections are held for seconds rather than milliseconds.
4. The 100-connection pool is exhausted; all new goroutines block on `db.Raw(...).First(...)`.
5. Legitimate clients' requests queue behind the attacker's flood; the HTTP write timeout (10 s) eventually returns errors to legitimate clients.
6. The attacker re-sends requests as old ones time out, maintaining pool exhaustion continuously.

### Impact Explanation

All clients are denied access to `/network/status`, which is the primary endpoint for querying current and genesis block identifiers and peer lists. Rosetta-based integrations (exchanges, wallets, block explorers) that poll this endpoint to track chain state receive errors or hang, constituting a network partition from the perspective of those clients. Severity is **High**: the endpoint is unauthenticated, the attack requires no special knowledge, and the impact is complete unavailability of network state for the duration of the attack.

### Likelihood Explanation

The attack requires only an HTTP client capable of maintaining ~100 concurrent connections — trivially achievable with `curl`, `wrk`, `hey`, or any load-testing tool. No credentials, tokens, or protocol knowledge are needed. The default deployment (`global.middleware: false`) ships without the only documented mitigation. The attack is repeatable and sustainable indefinitely at negligible cost to the attacker.

### Recommendation

1. **Enable the middleware by default**: Change `global.middleware` to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95. This activates the existing `inFlightReq: amount: 5` per-IP and `rateLimit: average: 10` per-host Traefik rules.
2. **Add application-level concurrency limiting**: Introduce a semaphore or `golang.org/x/time/rate` limiter inside `NetworkStatus()` (or as HTTP middleware in the router setup) so the Go process itself rejects excess requests before they reach the DB layer, independent of infrastructure configuration.
3. **Set a DB connection wait timeout**: Configure `db.SetConnMaxIdleTime` and use a short-deadline context when acquiring connections so that pool-exhaustion conditions fail fast (return 503) rather than blocking goroutines for up to 20 seconds.
4. **Reduce `statementTimeout`**: The default of 20 seconds is very long for a simple `SELECT` on an indexed table; reducing it to 2–3 seconds shrinks the window during which a connection is held under attack conditions.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wg/wrk) or equivalent
# Target: default Rosetta deployment with global.middleware=false

cat > /tmp/network_status.lua << 'EOF'
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
EOF

# Send 200 concurrent connections, sustaining for 60 seconds
wrk -t 20 -c 200 -d 60s \
    --script /tmp/network_status.lua \
    http://<ROSETTA_HOST>:5700/network/status

# Expected result:
# - Pool of 100 connections exhausted within first few seconds
# - Legitimate requests from a separate client begin returning errors or timing out
# - wrk reports high error rate; legitimate monitoring clients lose network state visibility
```

	if err != nil {

	}

		return nil, err

	if err != nil {
		return nil, err
	}

	return &rTypes.NetworkStatusResponse{
		CurrentBlockIdentifier: currentBlock.GetRosettaBlockIdentifier(),
		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}

func (br *blockRepository) RetrieveLatest(ctx context.Context) (*types.Block, *rTypes.Error) {
	if err := br.initGenesisRecordFile(ctx); err != nil {
		return nil, err
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	if rb.Index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	return rb.ToBlock(br.genesisBlock), nil
}

func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop
	}

	if ctx == nil {
		ctx = context.Background()
	}

	// #nosec G118
	childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
	return d.db.WithContext(childCtx), cancel
}
