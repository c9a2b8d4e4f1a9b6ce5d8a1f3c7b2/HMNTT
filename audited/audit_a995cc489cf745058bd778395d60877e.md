### Title
Uncached Alias-Based Entity Lookup Enables DB Resource Exhaustion via `getHooks()` Endpoint

### Summary
The `getHooks()` method in `HookServiceImpl` calls `entityService.lookup()` on every request, which for alias-based `ownerId` parameters directly executes a native SQL query against the `entity` table with no caching. The `rest-java` module has no rate limiting on the `/api/v1/accounts/{ownerId}/hooks` endpoint, allowing any unprivileged user to sustain a continuous stream of uncached DB queries by cycling through distinct valid aliases.

### Finding Description
**Exact code path:**

1. `HooksController.getHooks()` accepts `{ownerId}` as a path variable of type `EntityIdParameter`, which can be an `EntityIdAliasParameter` (alias format validated by `ALIAS_REGEX`). [1](#0-0) 

2. `HookServiceImpl.getHooks()` unconditionally calls `entityService.lookup(request.getOwnerId())` on line 35 before any other logic. [2](#0-1) 

3. `EntityServiceImpl.lookup()` dispatches to `entityRepository.findByAlias(p.alias())` for alias parameters — a raw native SQL call with no `@Cacheable` annotation. [3](#0-2) 

4. `EntityRepository.findByAlias()` in `rest-java` is a plain `@Query` with no caching whatsoever — every invocation hits the database. [4](#0-3) 

**Root cause:** The `rest-java` `EntityRepository` deliberately omits the `@Cacheable` annotation that other modules (e.g., `grpc`, `web3`) apply to equivalent lookups. There is no in-process or middleware-level rate limiting in the `rest-java` module — the `ThrottleConfiguration`/`ThrottleManager` infrastructure exists exclusively in the `web3` module. [5](#0-4) 

**Why existing checks fail:** The only input validation present is the `ALIAS_REGEX` pattern check in `EntityIdAliasParameter.valueOfNullable()`, which confirms the alias is syntactically valid — it does not prevent repeated or high-volume requests. There is no per-IP, per-endpoint, or global rate limit applied to `HooksController`. [6](#0-5) 

### Impact Explanation
Each request with an alias-based `ownerId` forces a full `SELECT id FROM entity WHERE alias = ? AND deleted <> true` query against the `entity` table. With no caching and no rate limiting, an attacker can drive a sustained, high-frequency query load against the database. Because each alias is distinct, no query result can be reused. This directly increases database CPU, I/O, and connection pool utilization. On a production node with a large `entity` table, sustained parallel requests from even a single client can measurably increase DB resource consumption well beyond 30% compared to baseline, degrading service for all users.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is publicly accessible, the alias format is documented and trivially constructable, and valid aliases can be enumerated from on-chain data. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) can issue hundreds of requests per second. The attack is repeatable, requires no authentication, and leaves no persistent side effects, making it low-risk for the attacker.

### Recommendation
1. **Add `@Cacheable`** to `EntityRepository.findByAlias()` and `findByEvmAddress()` in `rest-java`, mirroring the pattern used in the `grpc` and `web3` modules.
2. **Implement rate limiting** in the `rest-java` module for the `/api/v1/accounts/{ownerId}/hooks` endpoint, either via a servlet filter or Spring interceptor, analogous to the `ThrottleConfiguration`/`ThrottleManager` in `web3`.
3. Consider a short-lived (e.g., 30-second TTL) negative-result cache for alias lookups that return empty, to prevent amplification via non-existent aliases.

### Proof of Concept
**Preconditions:** Mirror node is running; at least one entity with a known alias exists on-chain (aliases are publicly visible).

**Steps:**
```bash
# 1. Obtain N distinct valid aliases from the network (public data)
# 2. Issue concurrent requests cycling through aliases:
wrk -t8 -c100 -d60s \
  -s <(echo 'local aliases = {...} -- list of valid base32 aliases
       local i = 0
       request = function()
         i = (i % #aliases) + 1
         return wrk.format("GET", "/api/v1/accounts/" .. aliases[i] .. "/hooks")
       end') \
  http://<mirror-node-host>

# Each request triggers:
# SELECT id FROM entity WHERE alias = ? AND deleted <> true
# with no cache, no rate limit, directly on the DB.
```

**Result:** Database query rate increases proportionally with request rate; no throttling or caching prevents the load from reaching the DB. Node resource consumption (DB CPU/IO) increases by >30% compared to the 24-hour baseline under sustained attack.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
