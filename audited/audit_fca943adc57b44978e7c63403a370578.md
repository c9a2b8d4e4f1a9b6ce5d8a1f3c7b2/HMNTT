### Title
Unauthenticated GraphQL `evmAddress` Lookup Enables Unbounded Database Query Flooding (Griefing DoS)

### Summary
The GraphQL endpoint exposes `account(input: {evmAddress: "..."})` to any unprivileged user with no per-request or per-IP rate limiting. Each call routes through `EntityServiceImpl.getByEvmAddressAndType()` to `EntityRepository.findByEvmAddress(byte[])`, issuing a live database query. Unlike the `web3` module which has `ThrottleConfiguration` with bucket-based rate limiting, the `graphql` module has no equivalent, allowing an attacker to flood the database with equality-scan queries at will.

### Finding Description
**Call chain:**

1. `AccountController.account()` accepts any `evmAddress` string with no authentication. [1](#0-0) 

2. `EntityServiceImpl.getByEvmAddressAndType()` decodes the hex string and, for any address whose first 12 bytes are not all-zero, calls `entityRepository.findByEvmAddress(evmAddressBytes)`. [2](#0-1) 

3. `findByEvmAddress` executes a native SQL query against the `entity` table on every call. [3](#0-2) 

**Existing mitigations are insufficient:**

The `GraphQlConfiguration` applies query complexity (max 200) and depth (max 10) limits, plus parser token/character caps. [4](#0-3) 

These controls limit the *size* of a single query, not the *rate* of queries. An attacker sends many small, fully-valid queries — each well within complexity/depth limits — at arbitrary frequency.

The `web3` module has a proper bucket-based rate limiter (`ThrottleConfiguration`, `ThrottleManagerImpl`) that enforces requests-per-second limits. [5](#0-4) 

No equivalent exists in the `graphql` module.

### Impact Explanation
An attacker can sustain a high-frequency stream of `account(input:{evmAddress:"<random 40-char hex>"})` queries. Each query hits the database with `SELECT * FROM entity WHERE evm_address = $1 AND deleted IS NOT TRUE`. Sustained flooding degrades database performance for all consumers of the mirror node (REST API, web3, importer), constituting a griefing DoS with no economic cost to the attacker. Severity is medium — no funds are at risk, but service availability for all users is impacted.

### Likelihood Explanation
Preconditions: none — the endpoint is public, no credentials required. The attacker needs only an HTTP client. The attack is trivially repeatable and scriptable. A single machine can sustain thousands of requests per second against a GraphQL endpoint with no server-side throttle.

### Recommendation
Add a rate-limiting filter to the `graphql` module analogous to `ThrottleConfiguration` in `web3`. Options include:

- A Bucket4j-based per-IP or global request-rate filter applied as a Spring `WebFilter` or `WebGraphQlInterceptor`.
- Configuring an ingress/load-balancer rate limit (the design doc for contract-log subscriptions already notes "We should implement rate limiting in the load balancer" — this should be enforced, not left as a note).
- At minimum, expose a configurable `requestsPerSecond` property in `graphql` application config mirroring `ThrottleProperties` in `web3`.

### Proof of Concept
```bash
# No authentication, no rate limit — flood with valid evmAddress queries
for i in $(seq 1 10000); do
  curl -s -X POST http://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"$(openssl rand -hex 20)\\\"}) { id } }\"}" &
done
```

Each request is a syntactically valid GraphQL query (complexity=1, depth=1), bypasses all existing parser/complexity guards, and issues a live `SELECT` against the `entity` table. No privileges, tokens, or special network position are required.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
