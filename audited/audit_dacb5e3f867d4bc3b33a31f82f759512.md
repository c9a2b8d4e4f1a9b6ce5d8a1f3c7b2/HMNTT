### Title
Log Injection via Unsanitized `req.ip` Derived from Attacker-Controlled `X-Forwarded-For` Header

### Summary
The Express REST server sets `trust proxy: true` unconditionally, making `req.ip` fully attacker-controlled via the `X-Forwarded-For` header. The Traefik ingress is explicitly configured with `--entryPoints.web.forwardedHeaders.insecure`, meaning it forwards client-supplied `X-Forwarded-For` values without any validation. The `responseHandler` middleware interpolates `req.ip` directly into a log string with no sanitization, enabling any unauthenticated external user to inject arbitrary content — including newlines and fake log lines — into the application log.

### Finding Description
**Code path:**

- `rest/server.js` line 58: `app.set('trust proxy', true);` — Express trusts all proxies unconditionally; `req.ip` is resolved from the leftmost value in `X-Forwarded-For`.
- `charts/hedera-mirror-common/values.yaml` lines 643–645: Traefik is configured with `--entryPoints.web.forwardedHeaders.insecure` and `--entryPoints.websecure.forwardedHeaders.insecure`, which disables all validation of `X-Forwarded-For` from clients.
- `rest/middleware/responseHandler.js` line 57: `logger.info(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${code}`);` — `req.ip` is interpolated into the log message with no escaping or format validation.
- `rest/logger.js` line 69: The logger writes plain-text lines in the format `${time} ${levelName} ${requestId} ${msg}\n` directly to stdout with no sanitization of the message content.

**Root cause:** The failed assumption is that `req.ip` will always contain a valid IP address string. With `trust proxy: true` and Traefik passing headers insecurely, `req.ip` is the raw, unvalidated first token from the client-supplied `X-Forwarded-For` header. No IP format validation or log-output sanitization exists anywhere in the chain.

**Exploit flow:**
1. Attacker sends a request with a crafted header: `X-Forwarded-For: 1.2.3.4\n2024-01-01T00:00:00.000Z INFO fakeRequestId INJECTED LOG ENTRY`
2. Traefik forwards the header as-is (insecure mode).
3. Express resolves `req.ip` to `1.2.3.4\n2024-01-01T00:00:00.000Z INFO fakeRequestId INJECTED LOG ENTRY`.
4. `responseHandler` calls `logger.info(...)` with this value embedded.
5. The logger writes the string verbatim to stdout, producing a spurious, fully attacker-controlled log line.

**Why existing checks fail:** There are no checks. `trust proxy: true` is a blanket setting with no IP allowlist. Traefik's `forwardedHeaders.insecure` flag explicitly removes header-source validation. The logger performs no escaping of newlines, carriage returns, or any other control characters in the message string.

### Impact Explanation
An unauthenticated attacker can forge arbitrary log entries indistinguishable from legitimate application logs. This enables: (1) log record forgery to cover tracks or frame other IPs; (2) corruption of log-based alerting or SIEM rules that parse the plain-text line format by pattern; (3) injection of content that exploits downstream log consumers (e.g., log shippers, parsers, or monitoring agents that process stdout). The impact is bounded to the integrity of the log stream — no direct code execution or data access — but it undermines the reliability of all observability derived from these logs.

### Likelihood Explanation
Exploitation requires zero privileges and zero authentication. Any HTTP client reachable by the public API endpoint can send a crafted `X-Forwarded-For` header. The Traefik insecure-forwarded-headers configuration is a deployment-level setting that is active in the provided Helm chart values. The attack is trivially repeatable and requires no special tooling beyond a standard HTTP client (`curl`). The only limiting factor is that the log line is only emitted when `code >= 400` or the random `sampleRate` condition is met, but the 400-path is trivially triggered by any malformed request.

### Recommendation
1. **Restrict `trust proxy`**: Replace `app.set('trust proxy', true)` with an explicit trusted proxy IP range or CIDR (e.g., `app.set('trust proxy', 'loopback, 10.0.0.0/8')`), so `req.ip` is only derived from `X-Forwarded-For` when the immediate connection comes from a known trusted proxy.
2. **Remove Traefik insecure forwarded headers**: Replace `--entryPoints.web.forwardedHeaders.insecure` with `--entryPoints.web.forwardedHeaders.trustedIPs=<pod-cidr>` to restrict which sources may set `X-Forwarded-For`.
3. **Sanitize before logging**: Strip or escape newlines (`\n`, `\r`) and other control characters from `req.ip` (and `req.originalUrl`) before interpolating into the log string, e.g.: `const safeIp = (req.ip ?? '').replace(/[\r\n]/g, '_');`

### Proof of Concept
```bash
# Inject a fake log line into the application log
curl -H "X-Forwarded-For: 1.2.3.4"$'\n'"2099-01-01T00:00:00.000Z INFO fakeId INJECTED: partition detected, node=evil" \
     http://<mirror-node-rest-api>/api/v1/nonexistent_path_to_trigger_400
```
The application stdout will contain two lines: the real 404 log entry and the fully attacker-controlled injected line, both appearing syntactically identical to legitimate log output.
