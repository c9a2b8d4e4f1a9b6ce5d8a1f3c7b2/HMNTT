### Title
Uncached EVM Address DB Lookup in `HookServiceImpl.getHooks()` Enables Resource Exhaustion via Repeated Unauthenticated Requests

### Summary
`HookServiceImpl.getHooks()` unconditionally calls `entityService.lookup(request.getOwnerId())`, which for `EntityIdEvmAddressParameter` inputs directly executes a raw SQL query against the database on every request with no caching. The `rest-java` module's `HooksController` has no rate-limiting or throttling mechanism, allowing any unauthenticated external caller to flood the endpoint with EVM address-based requests and drive repeated expensive DB lookups that cumulatively increase node resource consumption.

### Finding Description

**Exact code path:**

`HooksController.getHooks()` → `HookServiceImpl.getHooks()` line 35 → `EntityServiceImpl.lookup()` line 34 → `EntityRepository.findByEvmAddress()` line 17.

In `EntityServiceImpl.lookup()`, the EVM address branch directly calls the repository with no caching: [2](#0-1) 

The `rest-java` `EntityRepository.findByEvmAddress()` is a plain native SQL query with **no `@Cacheable` annotation**: [3](#0-2) 

**Root cause / failed assumption:** The design assumes either (a) a caching layer wraps EVM address lookups, or (b) a rate limiter protects the endpoint. Neither exists in the `rest-java` module. Contrast with the `web3` module's `EntityRepository`, which annotates its EVM address lookup with `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`, and the `importer` module's `EntityIdServiceImpl`, which wraps all lookups in `cacheLookup()`. The `rest-java` `EntityServiceImpl` has neither. [4](#0-3) 

The `HooksController` injects no `ThrottleManager` and applies no rate-limiting filter — unlike `web3`'s `ContractController` which calls `throttleManager.throttle(request)` before any processing: [5](#0-4) 

### Impact Explanation
Every request to `GET /api/v1/accounts/{evmAddress}/hooks` with an EVM address-format `ownerId` causes one full table scan / index lookup against the `entity` table (`WHERE evm_address = ? AND deleted <> true`). With no caching and no rate limiting, an attacker sending a sustained stream of requests (e.g., 500+ req/s from a single machine or distributed across IPs) forces proportional DB query load. This directly increases CPU, I/O, and connection pool consumption on the database node. A 30%+ increase over a 24-hour baseline is achievable without brute-force credential attacks — only HTTP GET requests to a public read endpoint are required.

### Likelihood Explanation
The endpoint is unauthenticated and publicly accessible. No special privileges, tokens, or accounts are required. The attacker only needs a valid 20-byte EVM address (even one that resolves to no entity — the DB query still executes before the `EntityNotFoundException` is thrown). The attack is trivially scriptable with `curl` or any HTTP load tool, is repeatable indefinitely, and can be distributed across multiple source IPs to avoid any upstream network-level rate limiting.

### Recommendation
1. **Add `@Cacheable`** to `rest-java`'s `EntityRepository.findByEvmAddress()` using a short-lived cache (e.g., 60s TTL), mirroring the pattern in `web3`'s `EntityRepository`.
2. **Add rate limiting** to `HooksController` (and other `rest-java` controllers) analogous to the `ThrottleManager` used in the `web3` module, enforcing a per-IP or global requests-per-second cap.
3. **Cache negative results** (entity not found) with a short TTL to prevent repeated DB hits for non-existent EVM addresses.

### Proof of Concept

**Preconditions:** Mirror node REST-Java service is running and publicly accessible. No authentication required.

**Steps:**
```bash
# Pick any 20-byte EVM address (valid or invalid)
EVM_ADDR="0xabcdef1234567890abcdef1234567890abcdef12"
BASE_URL="https://<mirror-node-host>"

# Flood the endpoint — each request triggers one uncached DB query
while true; do
  curl -s "$BASE_URL/api/v1/accounts/$EVM_ADDR/hooks" > /dev/null &
done
```

**Result:** Each concurrent request executes `SELECT id FROM entity WHERE evm_address = ? AND deleted <> true` against the database. With no caching and no throttle, DB CPU/IO rises linearly with request rate. Monitoring DB query throughput over 24 hours will show a sustained elevation exceeding 30% above the prior baseline.

final class EntityServiceImpl implements EntityService {

    private final EntityRepository entityRepository;

    @Override
    public Entity findById(EntityId id) {
        return entityRepository.findById(id.getId())
                .orElseThrow(() -> new EntityNotFoundException("Entity not found"));
    }

    @Override
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
