### Title
Unbounded Indefinite Subscription Accumulation via Missing gRPC Keepalive and No Server-Side Subscription Limit

### Summary
When a client subscribes with no `endTime`, `pastEndTime()` unconditionally returns `Flux.never()`, making the `takeUntilOther` termination mechanism permanently inert for that subscription. The `GrpcConfiguration` configures no gRPC keepalive on the `NettyServerBuilder`, so the server cannot detect silently-dead TCP connections (network partition). Combined with no server-side cap on total active subscriptions and no authentication requirement, an attacker can accumulate an unbounded number of zombie subscriptions that consume server memory and connection slots indefinitely.

### Finding Description

**Code path — `pastEndTime()` returns `Flux.never()`:**

`TopicMessageServiceImpl.java` lines 123–125:
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← termination signal is permanently suppressed
    }
```

This return value is used at line 73:
```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
```

When `endTime == null`, `takeUntilOther(Flux.never())` never fires. The `takeWhile` guard at line 80 is also skipped (`if (filter.getEndTime() != null)`), and `take(limit)` at line 84 is skipped when `limit == 0` (the default). The subscription has no application-level termination path.

**No gRPC keepalive configured:**

`GrpcConfiguration.java` lines 31–34:
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← no keepAliveTime(), keepAliveTimeout(), maxConnectionIdle(), maxConnectionAge()
};
```

Without `keepAliveTime` / `maxConnectionIdle` / `maxConnectionAge`, the Netty gRPC server relies entirely on the OS TCP stack to detect dead peers. A firewall-induced partition (packets silently dropped, TCP state kept alive on both ends) will never be detected. The subscription reactive pipeline — `TopicContext`, `AtomicLong` subscriber count, Redis listener subscription, Reactor scheduler resources — remains allocated forever.

**No total subscription limit:**

`TopicMessageServiceImpl.java` lines 48, 89–90:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

`subscriberCount` is exposed as a Micrometer gauge (metric only). There is no `if (subscriberCount.get() >= MAX) return error` guard anywhere in the call path.

**No authentication:**

`TopicMessageFilter` (`TopicMessageFilter.java`) has no credential or identity field. `topicExists()` only checks entity type, not caller identity.

**Root cause:** The design assumes the gRPC transport layer will signal stream cancellation when a client disappears. That assumption is violated during a network partition when no keepalive probes are sent, leaving the server with no mechanism to reclaim resources.

### Impact Explanation

Each zombie subscription holds:
- A `TopicContext` object with two `AtomicReference`/`AtomicLong` fields and a `Stopwatch`
- A live Redis pub/sub listener registration (or a polling scheduler task) via `topicListener.listen()`
- A Reactor `boundedElastic` scheduler task for the `safetyCheck` path
- An HTTP/2 stream slot counted against `maxConcurrentCallsPerConnection`

With `maxConcurrentCallsPerConnection = 5` (default), an attacker needs only `N/5` TCP connections to hold `N` subscriptions. There is no cap on the number of TCP connections. At scale, this exhausts JVM heap (OOM), the bounded-elastic thread pool, and Redis connection/subscription capacity, causing a full denial of service for all legitimate subscribers.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond the public HCS gRPC API (`ConsensusService/subscribeTopic`), and no `endTime` field in the `ConsensusTopicQuery` protobuf (omitting it is the default for live streaming clients). A single attacker machine can open thousands of TCP connections. The network-partition simulation is trivially achieved with `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` after the subscription is established, keeping the server-side TCP state alive while the client sends nothing. The attack is repeatable and requires no ongoing interaction after setup.

### Recommendation

1. **Configure gRPC server keepalive** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
   serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
   ```
   This ensures dead connections are detected and streams are cancelled, triggering `doFinally` cleanup.

2. **Enforce a server-side subscription cap** in `subscribeTopic()`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Flux.error(Status.RESOURCE_EXHAUSTED.asException());
   }
   ```
   Add `maxSubscribers` to `GrpcProperties` with a safe default (e.g., 5000).

3. **Add a maximum subscription duration** for no-endTime subscriptions via `timeout()` or `take(Duration)` on the live flux, configurable via `GrpcProperties`.

4. **Expose `maxConcurrentCallsPerConnection` and add `maxConnectionsPerIp`** at the load-balancer or Netty level to limit per-source amplification.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
import iptc  # or subprocess for iptables

TOPIC_ID = "0.0.12345"  # any valid topic
TARGET   = "mirror-node-grpc:5600"

def open_zombie(i):
    chan = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(chan)
    req  = consensus_pb2.ConsensusTopicQuery(
        topicID=...,   # set TOPIC_ID, no end_time field
        # limit omitted → 0
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass  # drain until partition
    except:
        pass

# Step 1: open N*5 subscriptions across N connections
threads = [threading.Thread(target=open_zombie, args=(i,)) for i in range(1000)]
for t in threads: t.start()

# Step 2: simulate network partition — server sees TCP alive, client sends nothing
# iptables -A OUTPUT -p tcp --dport 5600 -j DROP
import subprocess
subprocess.run(["iptables", "-A", "OUTPUT", "-p", "tcp", "--dport", "5600", "-j", "DROP"])

# Result: server holds 5000 zombie subscriptions indefinitely.
# subscriberCount metric climbs; JVM heap and Redis subscriptions are exhausted.
# Legitimate subscribers receive RESOURCE_EXHAUSTED or OOM errors.
