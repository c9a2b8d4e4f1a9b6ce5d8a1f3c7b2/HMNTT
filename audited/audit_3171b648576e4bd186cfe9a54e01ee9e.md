### Title
Unprivileged Treasury Account Substitution via Smart Contract Sender in `getSenderAccountID`

### Summary
In `TransactionExecutionService.getSenderAccountID()`, when a caller-supplied sender address resolves to an account whose `smartContract()` flag is `true`, the method silently returns the system treasury `AccountID` instead of rejecting the request. This treasury `AccountID` is then embedded as `transactionID.accountID` (the payer) in every transaction body submitted to the EVM executor, causing the simulation to run under treasury-account identity rather than the attacker-supplied address. Any unprivileged user can trigger this path by supplying the address of any deployed smart contract as the `from` field of an `eth_call` or `eth_estimateGas` request.

### Finding Description
**Exact code path:**

`ContractCallService.doProcessCall()` (line 115) → `TransactionExecutionService.execute()` (line 73) → `buildContractCallTransactionBody()` / `buildContractCreateTransactionBody()` (lines 185–199 / 173–183) → `defaultTransactionBodyBuilder()` (line 163) → `getSenderAccountID()` (line 239).

Inside `getSenderAccountID()`:

```java
// TransactionExecutionService.java lines 247–251
final var account = accountReadableKVState.get(accountIDNum);
if (account == null) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
} else if (account.smartContract()) {
    return EntityIdUtils.toAccountId(systemEntity.treasuryAccount()); // ← silent substitution
}
```

The returned treasury `AccountID` is placed directly into `transactionID.accountID` at line 167:

```java
.transactionID(TransactionID.newBuilder()
        .accountID(getSenderAccountID(params))   // treasury ID injected here
        .build())
```

**Root cause / failed assumption:** The code assumes that substituting the treasury account is a safe fallback when the sender is a smart contract. In reality, the Hedera consensus node would reject such a transaction outright with `PAYER_ACCOUNT_NOT_FOUND` (as the developer comment at lines 287–291 acknowledges for the null-account case). The smart-contract branch silently bypasses that rejection, so the mirror node executes the simulation with the treasury as the declared payer.

**Why existing checks fail:** The only guard for the smart-contract case is the `account.smartContract()` branch itself — it *enables* the substitution rather than blocking it. There is no downstream check that detects or rejects a treasury-substituted payer before the transaction body reaches the executor.

### Impact Explanation
The treasury account (`systemEntity.treasuryAccount()`) is a privileged system entity. By having it appear as `transactionID.accountID`, the EVM executor processes the simulated transaction as if the treasury is the payer. Hedera precompile handlers and internal authorization logic that inspect the payer account (e.g., solvency checks, allowance checks, token-operation authorization) will evaluate against the treasury's state rather than the attacker's account. This produces simulation results that diverge from what the consensus node would actually do (which is an immediate `PAYER_ACCOUNT_NOT_FOUND` failure). Concretely:

- A call that should fail pre-EVM (because a smart contract cannot be a payer) instead succeeds in simulation.
- Any precompile or EVM logic that grants elevated behavior to the treasury payer is reachable by an unprivileged caller.
- Developers relying on `eth_call` / `eth_estimateGas` to predict on-chain behavior receive misleading results, which can cause incorrect gas estimates, false-positive contract-interaction tests, or incorrect assumptions about access control.

Severity is Medium (griefing / incorrect simulation output; no direct fund loss on the mirror node itself).

### Likelihood Explanation
The precondition is trivially satisfied: the attacker only needs the EVM address of any deployed smart contract, which is publicly visible via the mirror node's own REST API or any block explorer. No special privileges, keys, or tokens are required. The attack is a single unauthenticated HTTP request to the `eth_call` or `eth_estimateGas` endpoint with `"from": "<contract_address>"`. It is repeatable at will and requires no on-chain transaction.

### Recommendation
Replace the silent treasury substitution with an explicit rejection consistent with consensus-node behavior:

```java
} else if (account.smartContract()) {
    // Consensus node returns PAYER_ACCOUNT_NOT_FOUND for smart-contract payers.
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
}
```

If the treasury-fallback behavior is intentional for some internal use-case (e.g., zero-sender default calls), it should be restricted exclusively to the already-guarded zero-address path at lines 241–243 and must not be reachable via an attacker-supplied non-zero smart-contract address.

### Proof of Concept
1. Identify any deployed smart contract address on the network, e.g. `0xABCD…` (obtainable from `/api/v1/contracts`).
2. Send an unauthenticated JSON-RPC request to the mirror node's web3 endpoint:
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_call",
     "params": [{
       "from": "0xABCD…",   // address of a known smart contract
       "to":   "0x<target_contract>",
       "data": "0x<calldata>"
     }, "latest"],
     "id": 1
   }
   ```
3. Observe that the call succeeds (returns a result) instead of returning `PAYER_ACCOUNT_NOT_FOUND`.
4. Confirm by checking `TransactionExecutionService.getSenderAccountID()`: the `account.smartContract()` branch is entered and the treasury `AccountID` is returned, causing the executor to run the simulation with the treasury as payer rather than rejecting the request.
