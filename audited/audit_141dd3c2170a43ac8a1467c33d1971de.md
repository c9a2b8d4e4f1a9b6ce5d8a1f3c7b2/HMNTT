### Title
Unbounded `json_agg` in `signatureQuery` Enables DB Amplification via `account.id` Filter

### Summary
In `getSchedules()` (`rest/schedules.js`), after the primary `schedulesQuery` returns up to `limit` (default max 100) schedule rows, a secondary `signatureQuery` is issued with an `IN (...)` clause over all returned schedule IDs. This `signatureQuery` uses `json_agg` with no `LIMIT` on the number of signatures per entity. Because the `transaction_signature` table has no unique constraint (by design), an attacker can repeatedly sign the same schedule to accumulate arbitrarily many rows per entity_id, then trigger expensive unbounded aggregation via a public API call.

### Finding Description

**Code path:**

`rest/schedules.js`, lines 234–264 (`getSchedules`):

1. `schedulesQuery` (line 240–241) filters by `creator_account_id = $1` and is bounded by `limit` (capped at `responseLimit.max`, default 100).
2. `entityIds` is built from the returned rows (line 250), up to 100 IDs.
3. `signatureQuery` (lines 255–259) is constructed as:

```sql
SELECT entity_id, json_agg(
  json_build_object('consensus_timestamp', ts.consensus_timestamp, ...)
  ORDER BY ts.consensus_timestamp
) AS signatures
FROM transaction_signature ts
WHERE entity_id IN ($1, ..., $N)
GROUP BY entity_id
ORDER BY entity_id ASC
```

**There is no `LIMIT` clause** on the number of signatures aggregated per entity_id. The `json_agg` will process every row in `transaction_signature` matching any of the up to 100 entity IDs.

**Root cause / failed assumption:** The code assumes the number of signatures per schedule is small. However, the design document (`docs/design/scheduled-transactions.md`, line 65) explicitly states: *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key."* This means `transaction_signature` can accumulate an unbounded number of rows per `entity_id`.

**Why existing checks are insufficient:**
- The `schedulesQuery` limit (max 100) only bounds the number of schedule IDs in the `IN` clause — it does not bound the number of `transaction_signature` rows fetched per schedule.
- `buildAndValidateFilters` / `getLimitParamValue` cap the `limit` parameter but have no effect on the inner `json_agg` aggregation.
- The index `transaction_signature (entity_id desc, consensus_timestamp desc)` speeds up the lookup but does not reduce the number of rows aggregated. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

An attacker who controls an account can:
- Create N schedules (creator_account_id = attacker account), each with many `transaction_signature` rows accumulated via repeated `ScheduleSign` calls.
- Issue `GET /api/v1/schedules?account.id=0.0.ATTACKER&limit=100`, causing the mirror node to execute a `json_agg` over potentially millions of rows in a single query.
- Repeat the request continuously to sustain elevated DB CPU/memory load.

The impact is degraded mirror node availability (slow or timed-out responses) for all users. No other user's on-chain assets are affected, consistent with the "griefing" classification. [4](#0-3) 

### Likelihood Explanation

- **No privilege required**: The `/api/v1/schedules` endpoint is public and unauthenticated.
- **Attacker cost**: Each `ScheduleSign` transaction costs a small HBAR fee, but fees are low enough that accumulating thousands of signatures per schedule is economically feasible for a motivated griever.
- **Repeatability**: The attack is stateless from the attacker's perspective — once signatures are accumulated on-chain, the same HTTP request can be replayed indefinitely.
- **No rate-limiting visible in code**: No per-IP or per-account request throttling is present in the reviewed path. [5](#0-4) 

### Recommendation

1. **Add a per-entity signature limit in the `signatureQuery`**: Use a subquery or `LIMIT` inside the `json_agg` window to cap signatures per schedule (e.g., most recent N signatures).
2. **Alternatively**, use a CTE with `ROW_NUMBER()` partitioned by `entity_id` to cap rows before aggregation:

```sql
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY entity_id ORDER BY consensus_timestamp DESC) AS rn
  FROM transaction_signature
  WHERE entity_id IN (...)
)
SELECT entity_id, json_agg(...) AS signatures
FROM ranked
WHERE rn <= 25
GROUP BY entity_id
ORDER BY entity_id ASC
```

3. **Document and enforce a maximum signatures-per-schedule** constant consistent with the existing `responseLimit` framework. [2](#0-1) [1](#0-0) 

### Proof of Concept

**Preconditions:**
- Attacker controls account `0.0.9999` on the network.
- Attacker creates 100 schedules with `creator_account_id = 0.0.9999`.
- For each schedule, attacker submits 10,000 `ScheduleSign` transactions (exploiting the lack of unique constraint on `transaction_signature`).

**Trigger:**
```
GET /api/v1/schedules?account.id=0.0.9999&limit=100
```

**What happens:**
1. `schedulesQuery` returns 100 rows (bounded by limit).
2. `signatureQuery` executes `json_agg` over `100 × 10,000 = 1,000,000` rows in `transaction_signature` in a single query, with no LIMIT.
3. The DB must sort and aggregate 1M rows, serialize them to JSON, and return the result.
4. Repeating this request at high frequency sustains DB overload.

**Result:** Mirror node REST API becomes slow or unresponsive for all users. [3](#0-2) [6](#0-5)

const getSchedules = async (req, res) => {
  // extract filters from query param
  const filters = utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters);

  // get sql filter query, params, order and limit from query filters
  const {filterQuery, params, order, limit} = extractSqlFromScheduleFilters(filters);
  const schedulesQuery = getSchedulesQuery(filterQuery, order, params.length);
  const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;
  }

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)
    .join(',');
  const entityQuery = `select ${entityFields} from entity where id in (${positions}) order by id ${order}`;
  const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),
    pool.queryQuietly(signatureQuery, entityIds),
  ]);
