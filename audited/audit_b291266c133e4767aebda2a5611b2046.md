### Title
Liveness Probe Decoupled from Transaction Submission Allows Silent Gap in Transaction History

### Summary
The pinger's liveness probe exclusively checks the freshness of `/tmp/alive`, which is updated by a goroutine that runs independently of whether transactions are actually being submitted. Because the heartbeat goroutine and the transaction submission loop are fully decoupled, any condition that silently stops transaction submission (network failure, API error, hung client) leaves the liveness probe permanently green while the pinger produces no transactions. A sidecar sharing the pod's `emptyDir` `/tmp` volume can additionally sustain this false-healthy state even after the pinger process itself dies.

### Finding Description

**Root cause — heartbeat goroutine is structurally decoupled from transaction submission**

`pinger/main.go` starts two independent control flows:

```
// goroutine 1 — heartbeat, runs every 15 s regardless of transaction health
go func() {
    t := time.NewTicker(15 * time.Second)
    for { select { case <-t.C:
        _ = os.WriteFile("/tmp/alive", ..., 0644)   // line 36
    }}
}()

// goroutine 2 — transaction loop
for { select { case <-ticker.C:
    if err := submitWithRetry(ctx, client, cfg); err != nil {
        log.Printf("transfer failed: %v", err)       // line 67 — only logged
    }
}}
``` [1](#0-0) [2](#0-1) 

A `submitWithRetry` failure only emits a log line; it does not stop the heartbeat goroutine, does not delete `/tmp/alive`, and does not set any flag that the healthcheck binary could read.

**Liveness check — only inspects file mtime, nothing else**

```go
fi, err := os.Stat("/tmp/alive")          // line 27
age := time.Since(fi.ModTime())
if age > 2*time.Minute {                  // line 35
    os.Exit(1)
}
os.Exit(0)
``` [3](#0-2) 

There is no check on transaction count, last-successful-submission timestamp, or any application-level health signal. A fresh file is the sole criterion.

**`/tmp` is an `emptyDir` shared within the pod**

```yaml
volumeMounts:
  - name: tmp
    mountPath: /tmp
volumes:
  - name: tmp
    emptyDir: {}
``` [4](#0-3) 

`podSecurityContext` forces every container in the pod (including injected sidecars) to run as UID 1000 / GID 1000: [5](#0-4) 

`/tmp/alive` is written with mode `0644` and owned by UID 1000. Any container in the pod running as UID 1000 — including a service-mesh proxy (Istio, Linkerd), a log-shipping sidecar, or any container injected by a mutating admission webhook — can call `os.WriteFile("/tmp/alive", ...)` or simply `touch /tmp/alive` to reset the mtime.

**Exploit flow (two paths)**

*Path A — no sidecar required (process still running, transactions silently failing):*
1. `submitWithRetry` begins returning errors (network partition, bad credentials, upstream API down).
2. The transaction loop logs errors and continues; the heartbeat goroutine is unaffected.
3. `/tmp/alive` is refreshed every 15 s; `age` never exceeds 2 minutes.
4. Liveness probe returns exit 0 indefinitely; Kubernetes never restarts the pod.
5. Transaction history has a silent gap of arbitrary duration.

*Path B — sidecar keeps probe green after pinger process death:*
1. The pinger process crashes or is OOM-killed.
2. The heartbeat goroutine stops; `/tmp/alive` begins aging.
3. A sidecar (UID 1000, sharing the `emptyDir`) runs `touch /tmp/alive` every ≤ 2 minutes.
4. Liveness probe continues to return exit 0; Kubernetes never restarts.
5. The gap in transaction history is invisible to the monitoring layer.

### Impact Explanation
The pinger's sole purpose is to submit transactions so the mirror node can be verified to be recording them. A silent failure of transaction submission — masked by a permanently-green liveness probe — creates an undetected gap in the transaction history. Operators and downstream consumers of the mirror node receive no alert; the gap can persist for hours or days. This directly satisfies the stated threat: "reorganizing transaction history without direct theft of funds" by creating a period of missing records that could be exploited to dispute or forge the canonical transaction sequence.

### Likelihood Explanation
Path A requires no attacker at all — it is a latent design defect triggered by any transient or persistent network/API failure. Path B requires a sidecar running as UID 1000 in the same pod. Service-mesh injection (Istio, Linkerd) is standard in many Kubernetes clusters; both inject sidecar proxies that inherit `podSecurityContext`. A cluster operator or anyone with `CREATE` rights on a `MutatingWebhookConfiguration` can inject such a sidecar without modifying the pinger deployment. The `emptyDir` volume is not protected by any access control beyond standard Unix permissions, and those permissions allow any UID-1000 process to write the file.

### Recommendation

1. **Tie the heartbeat to transaction success.** Only update `/tmp/alive` (or a separate `/tmp/last-tx-ok` file) immediately after a successful `submitWithRetry` call, not on a free-running ticker. The healthcheck should then check the age of the last-successful-submission timestamp, not a process-alive heartbeat.

2. **Separate liveness from transaction health.** Keep a process-alive heartbeat if desired, but add a second file (e.g., `/tmp/last-tx-ok`) written only on successful submission. The liveness probe should fail if `last-tx-ok` is older than `2 × cfg.interval` (or a configurable threshold).

3. **Restrict `/tmp` volume access.** Use a `subPath` mount or separate named volumes so the heartbeat file is not in the same directory accessible to sidecars. Alternatively, set explicit file permissions (`0600`) and ensure no sidecar shares the volume.

4. **Add a `failureThreshold`-aware circuit breaker.** If `submitWithRetry` fails N consecutive times, explicitly remove or corrupt `/tmp/alive` so the liveness probe triggers a restart.

### Proof of Concept

**Path A (no sidecar):**
```bash
# 1. Deploy pinger with a REST endpoint that always returns 500
#    (simulate upstream failure)
# 2. Observe logs: "transfer failed: ..." repeating every cfg.interval
# 3. Exec into the pod and verify /tmp/alive is still being refreshed:
kubectl exec -it <pinger-pod> -- stat /tmp/alive
# ModTime advances every ~15 s despite zero successful transactions
# 4. Liveness probe never fires; pod is never restarted
kubectl get events --field-selector reason=Unhealthy | grep pinger
# (empty — no liveness failures recorded)
```

**Path B (sidecar):**
```bash
# 1. Inject a sidecar into the pinger pod (e.g., via kubectl patch or
#    a MutatingWebhookConfiguration) that runs as UID 1000 and mounts
#    the same 'tmp' emptyDir volume at /tmp.
# 2. Kill the pinger process: kubectl exec <pinger-pod> -c pinger -- kill 1
# 3. From the sidecar, keep the file fresh:
kubectl exec <pinger-pod> -c sidecar -- \
  sh -c 'while true; do touch /tmp/alive; sleep 60; done'
# 4. Observe that the liveness probe never fails:
kubectl get events --field-selector reason=Unhealthy | grep pinger
# (empty — Kubernetes never restarts the dead pinger)
```
