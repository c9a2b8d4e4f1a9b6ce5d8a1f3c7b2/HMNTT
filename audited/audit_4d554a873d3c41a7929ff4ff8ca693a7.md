### Title
Unauthenticated Resource Exhaustion via Unrate-Limited `getScheduleById` with 1-Second Cache TTL on Active Schedules

### Summary
The `getScheduleById` handler in `rest/schedules.js` executes an expensive three-table join with `json_agg` aggregation on every request. For active (non-expired, non-executed) schedules, the response cache TTL collapses to 1 second (`DEFAULT_REDIS_EXPIRY`), meaning a sustained flood of concurrent unauthenticated requests continuously bypasses the cache and hammers the database. The `hedera-mirror-rest` Traefik middleware stack has no `inFlightReq` or `rateLimit` middleware, unlike other services in the same deployment, leaving no per-IP or global concurrency guard.

### Finding Description

**Exact code path:**

`rest/schedules.js` lines 46–64 define `getScheduleByIdQuery`:

```js
const getScheduleByIdQuery = `
  select ...
    ${transactionSignatureJsonAgg} as signatures
  from schedule s
  left join entity e on e.id = s.schedule_id
  left join transaction_signature ts on ts.entity_id = s.schedule_id
  where s.schedule_id = $1
  group by s.schedule_id, e.id`;
``` [1](#0-0) 

This query performs a three-table join and a `json_agg` ordered aggregate — non-trivial DB work per call.

`getScheduleById` (lines 115–128) fires this query on every request with no deduplication or in-handler guard:

```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
``` [2](#0-1) 

**Cache TTL collapse for active schedules:**

`getScheduleCacheControlHeader` returns an empty object `{}` for any schedule that has not yet executed or expired: [3](#0-2) 

`responseCacheHandler.js` line 24 sets `DEFAULT_REDIS_EXPIRY = 1` (one second), which is used when no `max-age` is present in the cache-control header: [4](#0-3) [5](#0-4) 

So for any active schedule, the Redis cache entry expires every second, and every new wave of concurrent requests within that second window all miss the cache simultaneously and each execute the full DB query.

**No rate limiting at the ingress layer:**

`charts/hedera-mirror-rest/values.yaml` middleware section contains only `circuitBreaker` and `retry` — no `inFlightReq`, no `rateLimit`: [6](#0-5) 

Compare to the Rosetta service, which explicitly configures both `inFlightReq` (5 concurrent per IP) and `rateLimit` (10 req/s): [7](#0-6) 

The REST Node.js application itself has zero rate-limiting middleware — confirmed by absence of any throttle/rateLimit code in `rest/**/*.js`.

**Retry amplification:** The REST middleware configures `retry: attempts: 10`, meaning each failed request is retried up to 10 times by Traefik, multiplying attacker-generated load by up to 10×. [8](#0-7) 

### Impact Explanation
An attacker can saturate the PostgreSQL database CPU by flooding `GET /api/v1/schedules/:scheduleId` for any known active schedule. The mirror node REST API becomes unresponsive, preventing clients (wallets, dApps, monitoring tools) from querying transaction and schedule status. The circuit breaker only trips after the error rate exceeds 25%, by which point the DB is already saturated. The retry middleware worsens the situation by amplifying each attacker request up to 10-fold at the Traefik layer.

### Likelihood Explanation
No authentication, API key, or account is required. Any external user knowing a valid active `scheduleId` (trivially discoverable via `GET /api/v1/schedules`) can execute this attack with standard tools (`ab`, `wrk`, `curl` in a loop). The attack is repeatable and sustainable indefinitely. The absence of `inFlightReq`/`rateLimit` in the REST chart — while present in other services — indicates this gap was not intentional hardening but an oversight.

### Recommendation
1. Add `inFlightReq` (per-IP concurrency cap) and `rateLimit` (per-IP request rate cap) to `charts/hedera-mirror-rest/values.yaml` middleware, mirroring the Rosetta configuration.
2. Increase the minimum Redis cache TTL for active schedules from 1 second to a meaningful value (e.g., 5–10 seconds) to absorb burst traffic.
3. Remove or reduce the `retry: attempts: 10` setting for the REST service, as it amplifies load under attack conditions.
4. Consider adding a DB-level statement timeout for this query class to bound worst-case DB CPU consumption per query.

### Proof of Concept
```bash
# Step 1: Discover a valid active schedule ID
SCHEDULE_ID=$(curl -s https://<mirror-node>/api/v1/schedules | jq -r '.schedules[0].schedule_id')

# Step 2: Flood the endpoint with concurrent requests (no auth required)
ab -n 10000 -c 500 "https://<mirror-node>/api/v1/schedules/${SCHEDULE_ID}"

# Expected result: DB CPU spikes to 100%, REST API response times degrade
# to timeouts, circuit breaker eventually trips (after damage is done),
# and legitimate clients receive 503 errors.
```
