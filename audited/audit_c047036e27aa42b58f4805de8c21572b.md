### Title
Unauthenticated Connection Pool Exhaustion via `account.publickey` Filter Forcing Dedicated DB Connection Acquisition

### Summary
Any unauthenticated caller can supply a syntactically valid 66-character ECDSA secp256k1 hex public key to `GET /api/v1/accounts?account.publickey=<key>`. The presence of this filter unconditionally activates `zeroRandomPageCostQueryHint`, which causes `queryQuietly` to acquire a **dedicated, exclusive connection** from the shared pg pool (rather than a pooled query). With the default pool ceiling of 10 connections and no rate limiting, an attacker sending ãË10 concurrent requests exhausts the pool, causing all subsequent API requests (across all endpoints) to fail with connection timeout errors for up to 20 seconds per wave.

Instead it calls `this.connect()` to acquire a **dedicated client**, opens a transaction (`BEGIN; SET LOCAL random_page_cost = 0`), runs the query, commits, and only then releases the client: [3](#0-2) 

The pool is configured with:
- `max: config.db.pool.maxConnections` ã default **10**
- `statement_timeout: config.db.pool.statementTimeout` ã default **20 000 ms**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` ã default **20 000 ms** [4](#0-3) 

**Root cause:** The `preQueryHint` branch in `queryQuietly` always acquires a dedicated connection for the full transaction lifecycle (connect ã BEGIN ã query ã COMMIT ã release). This is a fundamentally different resource model from a normal pool query. The failed assumption is that the pool's `max` ceiling (10) is sufficient to absorb concurrent public-key requests, but there is no mechanism to limit how many such requests can simultaneously hold dedicated connections.

**Exploit flow:**
1. Attacker sends 10 concurrent `GET /api/v1/accounts?account.publickey=<valid-66-char-hex>` requests (e.g., `0279be667ef9dcbbac55a06295ce870b07029bfcdb2dce28d959f2815b16f81798`).
2. Each request passes `isValidPublicKeyQuery` (66-char hex matches `publicKeyPattern`), sets `pubKeyQuery.query = 'public_key = ?'`, and triggers `preQueryHint`.
3. Each request calls `this.connect()`, acquiring one of the 10 pool slots.
4. All 10 slots are now held. The query runs (index seek on `entity__public_key_type`, returns empty), then COMMIT, then release ã but the attacker keeps sending new requests before slots are freed.
5. Any 11th request (from any endpoint) blocks waiting for a connection for up to 20 000 ms, then throws `Error: timeout exceeded when trying to connect`.
6. The attacker sustains this with a simple loop of ~10 concurrent HTTP clients.

### Impact Explanation

Connection pool exhaustion is service-wide: the pool is shared across all endpoints. Legitimate users querying transactions, balances, tokens, or any other endpoint are denied service for the duration of the attack. The default `statementTimeout` of 20 s means each wave of 10 requests can hold the pool for up to 20 s; the attacker needs only to re-send before the previous wave completes. With a single machine and 10 persistent HTTP connections, the mirror node REST API becomes completely unavailable. This directly affects the availability of a public blockchain infrastructure component.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public API documentation (the `account.publickey` parameter is documented in `openapi.yml`). A valid 66-character ECDSA secp256k1 public key is trivially generated or copied from any Ethereum wallet. The attack is repeatable, automatable, and requires no special tooling beyond `curl` or any HTTP client capable of concurrent requests. There is no IP-based rate limiting, no per-user throttle, and no circuit breaker in the REST layer. [5](#0-4) 

### Recommendation

1. **Decouple the hint from dedicated connection acquisition.** Use `SET LOCAL` inside a `BEGIN`/`COMMIT` block only when necessary, or apply the hint as a session-level parameter on pool initialization for the `mirror_rest` user rather than per-query.
2. **Add per-IP or global rate limiting** on the `/api/v1/accounts` endpoint (and all endpoints that trigger `preQueryHint`) using middleware such as `express-rate-limit`.
3. **Reduce the blast radius** by using a separate, smaller sub-pool for public-key queries, isolating them from the main pool used by all other endpoints.
4. **Lower `statementTimeout`** for public-key queries specifically (e.g., 2ã5 s) to reduce the window each dedicated connection is held.
5. **Consider a query result cache** (Redis response cache is present but disabled by default) to serve repeated identical public-key lookups without hitting the DB at all.

### Proof of Concept

```bash

# Send 15 concurrent requests (exceeds default pool of 10)

  curl -s "http://<mirror-node-host>:5551/api/v1/accounts?account.publickey=${KEY}" &
done
wait

# Immediately probe a different endpoint ã it will time out or return 500

# Expected: connection timeout / 500 Internal Error due to pool exhaustion
```
