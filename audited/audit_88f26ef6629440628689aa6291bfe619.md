### Title
Unauthenticated GET /contracts Endpoint Lacks Rate Limiting, Enabling DB Connection Pool Exhaustion via Concurrent Max-Limit Requests

### Summary
The `getContracts` handler in `rest/controllers/contractController.js` accepts unauthenticated GET requests with a `limit` parameter (capped at 100) and directly acquires a database connection from a pool whose default size is 10. No rate-limiting middleware exists in the REST Node.js server. An attacker sending more than 10 concurrent requests can hold all pool connections simultaneously, causing legitimate queries to queue until the 20-second connection timeout fires, effectively denying service with no economic cost.

### Finding Description
**Code path:**

`rest/routes/contractRoute.js:12` → `ContractController.getContracts` → `rest/controllers/contractController.js:745–752`

```js
getContracts = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedContractParameters);
  const {filterQuery, params, order, limit, limitQuery} = await extractSqlFromContractFilters(filters);
  const query = getContractsQuery(filterQuery, limitQuery, order);
  const {rows} = await pool.queryQuietly(query, params);   // ← acquires pool connection
  ...
};
```

`pool.queryQuietly` acquires a connection from the `pg` pool configured in `rest/dbpool.js:7–16`:

```js
const poolConfig = {
  ...
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
  max: config.db.pool.maxConnections,                         // default 10
  statement_timeout: config.db.pool.statementTimeout,         // default 20 000 ms
};
```

**Root cause – no rate limiting:** `rest/server.js` registers the following middleware chain: `urlencoded → json → cors → compression → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler`. There is no `express-rate-limit` or equivalent middleware. A `grep` for `rateLimit|rateLimiter|express-rate-limit` across all `rest/**/*.js` files returns only one hit in a test utility, confirming the absence of production rate limiting. The throttle mechanism (`ThrottleManagerImpl`, `ThrottleConfiguration`) lives exclusively in the `web3` Java module and does not apply to the Node.js REST API.

**Limit cap does not prevent the attack:** `buildAndValidateFilters` caps `limit` at `responseLimit.max` (default 100). Each request therefore executes a `SELECT … FROM entity LEFT JOIN contract … LIMIT 100` query. With the default pool of 10 connections, 10 concurrent requests saturate the pool. Any 11th request blocks in the `pg` queue until a connection is released or the 20-second `connectionTimeoutMillis` fires.

**Statement timeout provides only partial mitigation:** The 20-second `statement_timeout` kills long-running queries, but an attacker only needs to sustain ≥10 concurrent in-flight requests. Because each request holds its connection for the full query duration (up to 20 s), a continuous stream of requests keeps the pool exhausted indefinitely.

### Impact Explanation
Legitimate API consumers (wallets, explorers, dApps) querying `/api/v1/contracts` receive connection-timeout errors (HTTP 500 / pool exhaustion) for the duration of the attack. The REST API is a public read-only service; all other endpoints sharing the same pool (`/api/v1/accounts`, `/api/v1/transactions`, etc.) are equally affected. The impact is availability degradation (griefing) with no on-chain economic consequence to the attacker.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no on-chain activity. A single attacker with a standard HTTP client (e.g., `ab -c 15 -n 10000 'http://host/api/v1/contracts?limit=100'`) can reproduce it. The default pool size of 10 makes the threshold trivially low. The attack is repeatable and can be automated indefinitely.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST server (e.g., `express-rate-limit` with a per-IP sliding window) applied globally before route handlers in `rest/server.js`.
2. **Increase the default pool size** or document a minimum production value in `docs/configuration.md` (the Helm chart already configures pgbouncer with 250 connections for `mirror_rest`, but the Node.js pool default of 10 remains the bottleneck).
3. **Add a concurrency/inflight-request cap** at the application layer (e.g., `express-slow-down` or a semaphore) to bound the number of simultaneous DB-bound requests.
