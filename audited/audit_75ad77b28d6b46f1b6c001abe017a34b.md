### Title
Unauthenticated DoS via Unbounded Concurrent Requests to `/api/v1/network/stake` with No Caching or Rate Limiting

### Summary
The `GET /api/v1/network/stake` endpoint is publicly accessible with no authentication, no rate limiting, and no caching. Every request causes `NetworkServiceImpl.getLatestNetworkStake()` to directly invoke `NetworkStakeRepository.findLatest()`, which executes a native SQL query containing the subquery `select max(consensus_timestamp) from network_stake`. An unprivileged attacker flooding this endpoint with concurrent requests will exhaust the database connection pool and/or CPU, denying service to legitimate consumers.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` at [1](#0-0)  is a public `@GetMapping` with no access control annotation, no throttle guard, and no authentication requirement.

It delegates directly to `NetworkServiceImpl.getLatestNetworkStake()`: [2](#0-1) 

Which calls `NetworkStakeRepository.findLatest()` — a native SQL query with a full-table aggregation subquery: [3](#0-2) 

**Root cause — failed assumptions:**

1. **No caching**: A grep across all of `rest-java/src/main/java/**/*.java` for `@Cacheable`, `caffeine`, `redis`, `Cache` returns zero hits in the service or repository layer. There is no Spring Cache, no in-memory TTL cache, and no result memoization on `getLatestNetworkStake()`. [2](#0-1) 

2. **No rate limiting**: A grep across `rest-java/**` for `rate.limit`, `throttl`, `RateLimiter`, `bucket4j`, `resilience` returns no hits in the production source path. The `HIGH_VOLUME_THROTTLE` constant referenced in `NetworkController` applies only to the `POST /network/fees` fee-estimation endpoint, not to `GET /network/stake`. [4](#0-3) 

3. **Unbounded DB query per request**: The query `select * from network_stake where consensus_timestamp = (select max(consensus_timestamp) from network_stake)` is executed once per HTTP request with no deduplication or coalescing. [3](#0-2) 

### Impact Explanation

An attacker sending thousands of concurrent `GET /api/v1/network/stake` requests will:
- Saturate the JDBC/HikariCP connection pool (typically 10–50 connections by default), causing all subsequent requests — including legitimate ones — to queue or fail with connection timeout errors.
- Force repeated execution of the `max(consensus_timestamp)` aggregation. Even if an index exists on `consensus_timestamp`, each concurrent request still acquires a DB connection, executes a query, and holds that connection for the round-trip duration. Under high concurrency this exhausts the pool.
- If `consensus_timestamp` is not indexed on `network_stake`, each `max()` is a full sequential scan, multiplying CPU cost linearly with concurrency.

The result is complete unavailability of the `/api/v1/network/stake` endpoint and potential degradation of other endpoints sharing the same DB connection pool.

**Severity: High** — full DoS of a public read endpoint with zero preconditions.

### Likelihood Explanation

- **Preconditions**: None. No account, no token, no API key required.
- **Attacker capability**: Any script kiddie with `curl` or `ab` (Apache Bench) can execute this. A single machine with `ab -n 100000 -c 500 https://target/api/v1/network/stake` is sufficient.
- **Repeatability**: Fully repeatable and automatable. The attack can be sustained indefinitely.
- **Detection evasion**: Because the requests are individually valid and cheap-looking (simple GET, no body), WAF rules based on payload inspection will not block them.

### Recommendation

1. **Add response caching** on `getLatestNetworkStake()` with a short TTL (e.g., 15–30 seconds). The network stake data changes at most once per staking period (daily), so a short cache is safe and eliminates nearly all DB load:
   ```java
   @Cacheable(cacheNames = "networkStake")
   public NetworkStake getLatestNetworkStake() { ... }
   ```

2. **Add rate limiting** on the `/api/v1/network/stake` endpoint using a token-bucket or sliding-window limiter (e.g., Bucket4j, Spring Cloud Gateway rate filter, or an API gateway layer).

3. **Ensure an index exists** on `network_stake(consensus_timestamp)` so the `max()` subquery uses an index scan rather than a sequential scan.

4. **Set a DB connection pool maximum** and configure a short `connectionTimeout` so pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept

```bash
# Flood the endpoint with 1000 concurrent requests
ab -n 50000 -c 1000 http://<mirror-node-host>/api/v1/network/stake

# Observe: legitimate requests begin timing out or returning 503
# DB monitoring will show connection pool at 100% utilization
# and repeated sequential scans on network_stake
```

Expected result: after a few seconds of sustained load, legitimate requests to `/api/v1/network/stake` (and potentially other endpoints sharing the same pool) return connection timeout errors or HTTP 503, confirming denial of service.
