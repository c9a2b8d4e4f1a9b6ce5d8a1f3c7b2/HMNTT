### Title
Unauthenticated Historical Balance Requests Trigger Unbounded Dual DB Query Execution, Enabling DB CPU Exhaustion

### Summary
The Rosetta `/account/balance` endpoint accepts historical `block_identifier` values from any unauthenticated caller. Each such request unconditionally executes two sequential database queries inside `getLatestBalanceSnapshot()`: a CTE scan on `mirror_node_time_partitions` (`selectPreviousPartitionLowerBound`) followed by a correlated nested subquery on the large partitioned `account_balance` table (`latestBalanceBeforeConsensus`), plus a third query (`balanceChangeBetween`) on `crypto_transfer`. No application-level rate limiting is enforced by default, allowing an attacker to sustain a high request rate that drives DB CPU well above the 30% baseline threshold.

### Finding Description

**Exact code path:**

`rosetta/app/services/account_service.go` lines 41–78: `AccountBalance()` accepts `request.BlockIdentifier` directly from the HTTP body with no authentication or rate-gate check. [1](#0-0) 

It calls `RetrieveBalanceAtBlock()` in `rosetta/app/persistence/account.go` lines 161–223, which calls `getLatestBalanceSnapshot()` unconditionally for every request that includes a block identifier. [2](#0-1) 

Inside `getLatestBalanceSnapshot()` (lines 267–311), two sequential DB round-trips are always made:

1. **Query 1** — `selectPreviousPartitionLowerBound` (lines 76–86): a CTE that scans `mirror_node_time_partitions` ordered by `from_timestamp` to find the second-to-last partition boundary before the user-supplied timestamp. [3](#0-2) 

2. **Query 2** — `latestBalanceBeforeConsensus` (lines 30–50): a correlated nested subquery that first finds the latest treasury snapshot in `account_balance` within the partition range, then for each row looks up the target account's balance — both scans bounded only by the user-supplied `@timestamp` and `@lower_bound`. [4](#0-3) 

A third query (`balanceChangeBetween`) on `crypto_transfer` follows immediately. [5](#0-4) 

**Root cause / failed assumption:** The design assumes callers will be infrequent or that an external rate-limiter will be present. Neither assumption holds: the Traefik middleware is **disabled by default** (`middleware: false`), and even when enabled the `rateLimit` is keyed on `requestHost` (global, not per-source-IP), while `inFlightReq` is only 5 concurrent requests per IP — easily saturated by a small botnet or bypassed by rotating IPs. [6](#0-5) 

### Impact Explanation

Each `/account/balance` POST with a historical block identifier causes 3 DB queries, two of which (`latestBalanceBeforeConsensus` and `balanceChangeBetween`) scan large partitioned tables (`account_balance`, `crypto_transfer`) with range predicates that cannot be fully satisfied by a single partition index seek when the supplied timestamp is old or crafted to straddle partition boundaries. Sustained at even modest rates (e.g., 20–50 req/s from a handful of IPs), this can push DB CPU above the 30% 24-hour baseline, degrading query performance for all mirror-node consumers (REST API, web3, gRPC). No data is modified, but availability of the shared PostgreSQL instance is impacted.

### Likelihood Explanation

The attack requires zero privileges — only a valid network identifier and any known account address (e.g., `0.0.2`, the treasury). Historical block indices are publicly discoverable via `/network/status`. The payload is a standard Rosetta JSON POST. The middleware protection is opt-in and misconfigured by default (rate limit is per-host, not per-IP). A single attacker with a modest script can sustain the load; a distributed attacker trivially bypasses the per-IP in-flight cap of 5.

### Recommendation

1. **Enable per-source-IP rate limiting at the application layer** (not just optional Traefik middleware). Add a token-bucket or sliding-window limiter keyed on client IP inside the Go HTTP handler chain in `rosetta/main.go` before requests reach the DB layer.
2. **Change the Traefik `rateLimit` `sourceCriterion`** from `requestHost: true` to `ipStrategy` (per-source-IP) and set a realistic per-IP cap (e.g., 5 req/s).
3. **Enable the middleware by default** — change `middleware: false` to `middleware: true` in `charts/hedera-mirror-rosetta/values.yaml`.
4. **Merge the two queries in `getLatestBalanceSnapshot()`** into a single SQL statement to halve the per-request DB round-trips.
5. **Enforce a DB statement timeout** — ensure `statementTimeout` is always set to a non-zero value in the DB client configuration so runaway queries are killed.

### Proof of Concept

```bash
# 1. Discover a valid block index
BLOCK=$(curl -s -X POST https://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq '.current_block_identifier.index - 1000')

# 2. Flood /account/balance with historical block identifiers (no auth required)
for i in $(seq 1 500); do
  curl -s -X POST https://<rosetta-host>/account/balance \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\":{\"address\":\"0.0.2\"},
      \"block_identifier\":{\"index\":$((BLOCK - i))}
    }" &
done
wait

# Each request triggers selectPreviousPartitionLowerBound + latestBalanceBeforeConsensus
# + balanceChangeBetween = 3 DB queries. 500 concurrent requests = up to 1500 DB queries.
# Monitor DB CPU: should exceed 30% above 24h baseline within seconds.
```
