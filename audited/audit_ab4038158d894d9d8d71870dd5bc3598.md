### Title
Unauthenticated Log Flooding via Malformed Timestamps on `/topics/messages/:consensusTimestamp`

### Summary
The `/api/v1/topics/messages/:consensusTimestamp` endpoint accepts any string as a URL path segment, performs no rate limiting, and emits a `logger.warn` entry for every request that fails timestamp validation. An unprivileged attacker can flood this endpoint with malformed timestamps at arbitrary request rates, generating unbounded warn-log volume and drowning out legitimate operational signals.

### Finding Description
**Code path:**

`rest/server.js` line 129 registers the route with an unconstrained path parameter:
```
app.getExt(`${apiPrefix}/topics/messages/:consensusTimestamp`, topicmessage.getMessageByConsensusTimestamp);
``` [1](#0-0) 

`rest/topicmessage.js` lines 22–26 validate the parameter and throw on failure:
```js
const validateConsensusTimestampParam = (consensusTimestamp) => {
  if (!utils.isValidTimestampParam(consensusTimestamp)) {
    throw InvalidArgumentError.forParams(TopicMessage.CONSENSUS_TIMESTAMP);
  }
};
``` [2](#0-1) 

The thrown `InvalidArgumentError` is caught by `handleError` in `rest/middleware/httpErrorHandler.js`. Because `InvalidArgumentError` maps to HTTP 400 (a client error), `shouldReturnMessage` returns `true`, and the handler calls `logger.warn` — one warn-log line per request:
```js
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode} ${err.constructor.name} ${errorMessage}`
);
``` [3](#0-2) 

**Root cause / failed assumption:** The Node.js REST server has no rate-limiting middleware. A grep across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `express-rate` returns zero hits in production code. The only throttle logic in the repository lives in the separate `web3` Java service and does not apply here. [4](#0-3) 

### Impact Explanation
Every malformed request produces one `logger.warn` line containing the caller's IP, method, and full URL. At high request rates this:
- Fills warn-log storage, potentially triggering log-rotation or truncation that discards legitimate entries.
- Buries real warn-level signals (e.g., slow queries, auth failures) in noise, degrading threshold-based or pattern-based alerting.
- Increases I/O and CPU overhead on the logging subsystem proportional to request rate.

No database query is executed for invalid timestamps (validation fires before the DB call at line 63–65), so database load is not a factor, but the logging overhead is unbounded. [5](#0-4) 

### Likelihood Explanation
Preconditions: none. The endpoint is public, unauthenticated, and reachable over the internet. A single machine with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) can sustain thousands of requests per second. The attack is trivially repeatable, requires no special knowledge, and leaves no persistent state on the attacker's side.

### Recommendation
1. **Add rate limiting** to the Node.js REST server (e.g., `express-rate-limit` or an ingress-level policy) covering all `/api/v1/*` routes, with per-IP limits.
2. **Reduce log verbosity for repeated identical client errors**: deduplicate or sample warn logs for `InvalidArgumentError` (e.g., log once per IP per minute for the same error class).
3. **Enforce path-parameter format at the router level** using an Express regex route so that strings that cannot possibly be valid timestamps never reach the handler:
   ```js
   app.getExt(`${apiPrefix}/topics/messages/:consensusTimestamp(\\d+(\\.\\d+)?)`, ...);
   ```
   This rejects obviously malformed values before any JavaScript executes, producing a plain 404 with no warn log.

### Proof of Concept
```bash
# Send 10 000 requests with a malformed timestamp; observe warn log volume
wrk -t4 -c100 -d30s \
  "http://<mirror-node-host>/api/v1/topics/messages/AAAAAAAAAAAAAAAA"
```
Each request returns HTTP 400 and appends one line to the warn log:
```
<ip> GET /api/v1/topics/messages/AAAAAAAAAAAAAAAA in N ms: 400 InvalidArgumentError Invalid parameter: consensus_timestamp
```
Sustained for 30 seconds at ~5 000 req/s this produces ~150 000 warn-log lines, overwhelming any fixed-size log buffer or alerting window.
