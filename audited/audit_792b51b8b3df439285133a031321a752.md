### Title
Misplaced `sync.Once` in `initGenesisRecordFile` Allows Concurrent DB Query Amplification Leading to Connection Pool Exhaustion

### Summary
In `rosetta/app/persistence/block.go`, the `initGenesisRecordFile` function checks `br.genesisBlock.ConsensusStart != genesisConsensusStartUnset` outside of any synchronization primitive, then executes the expensive `selectGenesis` CTE query, and only afterward uses `sync.Once` to protect the assignment. This means N concurrent goroutines can all pass the unguarded check simultaneously and each independently execute the costly DB query, exhausting the connection pool before `sync.Once` ever fires. No authentication is required to trigger this path.

### Finding Description

**Exact code location:** `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263.

```
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    // ❶ Unguarded read — no mutex, no once.Do wrapping this check
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    db, cancel := br.dbClient.GetDbWithContext(ctx)   // ❷ acquires a pool connection
    defer cancel()

    var rb recordBlock
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {  // ❸ expensive CTE
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
    }

    br.once.Do(func() {          // ❹ sync.Once only protects the write, NOT steps ❶–❸
        br.genesisBlock = rb
    })
    ...
}
```

**Root cause:** `sync.Once` is placed around only the *assignment* (`br.genesisBlock = rb`), not around the entire check-query-set sequence. The guard at ❶ is a plain unsynchronized read. During the startup window while `ConsensusStart == -1`, every concurrent goroutine passes ❶, proceeds to ❷ (acquires a DB connection), and executes ❸ (the `selectGenesis` CTE, which joins `account_balance` and `record_file` with ordering and a correlated subquery). Only one goroutine's result is ever used; all others waste their connection and query execution.

**Failed assumption:** The developer assumed `sync.Once` would prevent redundant DB queries. It does not — `sync.Once` only serializes the callback passed to `Do`, not the code that precedes it.

**Exploit flow:**
1. Attacker identifies the node is in startup state (e.g., `/network/status` returns `ErrNodeIsStarting`).
2. Attacker floods any Rosetta endpoint that calls `initGenesisRecordFile` — `RetrieveGenesis`, `FindByHash`, `FindByIndex`, `FindByIdentifier`, `RetrieveLatest` — with concurrent HTTP requests (no auth required).
3. All goroutines simultaneously read `ConsensusStart == -1`, all acquire DB connections, all execute `selectGenesis`.
4. The DB connection pool (default `MaxOpenConnections: 100` per test config) is saturated; legitimate requests queue or fail.
5. Even after genesis is initialized, the attacker can restart the node or exploit the window on each restart. [1](#0-0) [2](#0-1) 

### Impact Explanation

Every concurrent request during the startup window independently executes the `selectGenesis` CTE — a multi-table join with ordering and a correlated subquery against `account_balance` and `record_file`. With a pool of 100 connections, 100 simultaneous requests saturate the pool entirely. Subsequent legitimate requests (from validators, monitoring, or other clients) receive connection-pool-timeout errors or are queued indefinitely. This constitutes a targeted denial-of-service against the Rosetta API layer and indirectly against the shared DB, potentially impacting other mirror-node services sharing the same DB. [3](#0-2) 

### Likelihood Explanation

- **No authentication required:** All Rosetta API endpoints are publicly accessible; the Traefik middleware (`inFlightReq: 5`, `rateLimit: 10`) is **disabled by default** (`global.middleware: false`).
- **Trivially repeatable:** The attacker only needs to know the node is starting (observable via error responses) and send concurrent HTTP POST requests to any block-related endpoint.
- **Distributed bypass:** Even if per-IP rate limiting is enabled, a modest botnet or multiple source IPs trivially bypasses the `inFlightReq: amount: 5` per-IP limit.
- **Repeatable on every restart:** The vulnerability resets on each node restart, giving the attacker a recurring exploitation window. [4](#0-3) [5](#0-4) 

### Recommendation

Wrap the **entire** initialization sequence inside `sync.Once.Do`, eliminating the unguarded check:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()

        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })

    // If once.Do failed, reset so next call retries
    if initErr != nil {
        br.once = sync.Once{} // or use a separate mutex+flag pattern for retryable init
        return initErr
    }

    if br.genesisBlock.ConsensusStart == genesisConsensusStartUnset {
        return hErrors.ErrNodeIsStarting
    }
    return nil
}
```

Note: because `sync.Once` cannot be retried on failure, a `sync.Mutex` with an explicit initialized flag is the more robust pattern when the initialization can fail transiently. [6](#0-5) 

### Proof of Concept

**Preconditions:**
- Rosetta node is starting up (genesis not yet loaded); observable when any block endpoint returns `ErrNodeIsStarting`.
- No authentication required.

**Steps:**
```bash
# 1. Confirm node is in startup state
curl -s -X POST http://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | grep -i "starting"

# 2. Flood RetrieveGenesis (maps to /block endpoint) with 200 concurrent requests
seq 200 | xargs -P200 -I{} curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":0}}'

# 3. Observe DB connection pool exhaustion:
#    - Subsequent legitimate requests return DB errors or time out
#    - DB server shows 100 active connections all running selectGenesis CTE
#    - Node logs show repeated "Fetched genesis record file" from N goroutines
```

**Expected result:** All 200 goroutines pass the unguarded `ConsensusStart == -1` check simultaneously, each acquires a DB connection and executes `selectGenesis`, saturating the pool and causing legitimate requests to fail. [7](#0-6)
