### Title
Unprotected Pre-Check Allows Concurrent `selectGenesis` Query Storms in `initGenesisRecordFile()`

### Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` uses `sync.Once` only to protect the **write** to `br.genesisBlock`, but the expensive `selectGenesis` SQL query executes **before** `once.Do` is called and is guarded only by an unsynchronized read of `br.genesisBlock.ConsensusStart`. Any number of concurrent goroutines can pass that unprotected check simultaneously and each independently issue the costly correlated-subquery against the database, exhausting the connection pool and DB CPU. No authentication is required to trigger this via the public Rosetta API.

### Finding Description

**Exact location:** `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263.

```

244:     if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {  // ← unprotected read, no mutex
245:         return nil
246:     }
247:
248:     db, cancel := br.dbClient.GetDbWithContext(ctx)
249:     defer cancel()
250:
251:     var rb recordBlock
252:     if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", ...)).  // ← expensive query runs here
253:         First(&rb).Error; err != nil {
254:         return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
255:     }
256:
257:     br.once.Do(func() {   // ← sync.Once only protects the WRITE, not the query above
258:         br.genesisBlock = rb
259:     })
```

**Root cause:** `sync.Once` is placed *after* the expensive `selectGenesis` query, not around it. The early-exit guard at line 244 reads `br.genesisBlock.ConsensusStart` with no mutex, no atomic load, and no `sync.Once` protection. Before the genesis block is initialized, every goroutine that reaches line 244 sees `ConsensusStart == genesisConsensusStartUnset (-1)` and proceeds to execute the full `selectGenesis` query independently.

**`selectGenesis` query cost** (lines 49–72): it is a CTE joined to `record_file` with a correlated subquery per row:
```sql
coalesce((
  select rf1.consensus_start-1
  from record_file rf1
  where rf1.index = rf.index + 1
), consensus_end) as consensus_end
```
On a large `record_file` table this is a non-trivial scan. N concurrent executions multiply that cost N-fold.

**Exploit flow:**
1. Attacker detects service startup (e.g., by observing HTTP 503 / `ErrNodeIsStarting` responses transitioning to success).
2. Attacker immediately floods any public Rosetta block endpoint (`/block`, `/block/transaction`, etc.) with a high-concurrency burst (e.g., 500 simultaneous requests).
3. All goroutines reach line 244 before any `once.Do` has fired; all see `ConsensusStart == -1`; all proceed past the guard.
4. All goroutines execute `selectGenesis` concurrently, consuming one DB connection each and hammering DB CPU.
5. DB connection pool is exhausted; subsequent legitimate requests queue or fail; DB CPU spikes to 100%.
6. `once.Do` eventually fires for the first goroutine to complete, but the remaining N-1 queries are already in flight and cannot be cancelled.

**Why existing checks are insufficient:**
- `sync.Once` at line 257 only serializes the *assignment* `br.genesisBlock = rb`. It does not prevent the query at lines 252–254 from running N times.
- The read at line 244 is a plain struct-field access with no atomic or mutex protection — it is a data race under Go's memory model when concurrent writers exist.
- There is no rate-limiting, request queuing, or connection-pool guard specific to this initialization path.

### Impact Explanation
An attacker can exhaust the database connection pool and saturate DB CPU on demand, rendering the entire Rosetta API unavailable. Because `initGenesisRecordFile()` is called by every public block-lookup method (`FindByHash`, `FindByIndex`, `FindByIdentifier`, `RetrieveGenesis`, `RetrieveLatest`), every API surface is affected. The attack is repeatable on every service restart. This is a non-network-based DoS that requires no credentials and no special knowledge beyond the public API surface.

### Likelihood Explanation
The Rosetta API is a public, unauthenticated HTTP service. Any external actor can send concurrent HTTP requests. The vulnerable window exists on every process startup and is widened by any DB slowness (slow `selectGenesis` = longer window). The attack requires only a standard HTTP load tool (e.g., `wrk`, `ab`, `hey`) and knowledge that the service just restarted — detectable from public health endpoints or error responses. No exploit code, no credentials, no chain of vulnerabilities required.

### Recommendation
Move the entire initialization — including the DB query — inside `once.Do`, so the expensive query is guaranteed to execute at most once regardless of concurrency:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
    })
    if initErr != nil {
        return initErr
    }
    if br.genesisBlock.ConsensusStart == genesisConsensusStartUnset {
        return hErrors.ErrNodeIsStarting
    }
    return nil
}
```

Note: if `once.Do` must not swallow transient errors (so retries are possible), replace `sync.Once` with a mutex-guarded double-checked locking pattern or use `golang.org/x/sync/singleflight` to coalesce concurrent callers into a single in-flight query.
