### Title
Unauthenticated DB Connection Pool Exhaustion via Uncached, Unthrottled `findById()` on Non-Existent Topic IDs

### Summary
`TopicServiceImpl.findById()` delegates directly to a plain Spring Data `CrudRepository.findById()` with no caching and no negative-result memoization. The rest-java module has no rate-limiting layer (unlike the web3 module which uses bucket4j). An unauthenticated attacker can flood `GET /api/v1/topics/{non-existent-id}` at high concurrency, issuing one uncached DB SELECT per request, exhausting the HikariCP connection pool and starving legitimate requests.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` → `TopicServiceImpl.findById()` → `TopicRepository.findById()` (Spring Data `CrudRepository`) → raw JDBC SELECT. [1](#0-0) 

`TopicRepository` is a bare `CrudRepository<Topic, Long>` with zero `@Cacheable` annotations: [2](#0-1) 

Compare with the grpc module's `EntityRepository`, which annotates `findById` with `@Cacheable` to absorb repeated lookups: [3](#0-2) 

The rest-java `EntityRepository` also has no `@Cacheable`: [4](#0-3) 

**No rate limiting in rest-java:** The web3 module has an explicit `ThrottleConfiguration` using bucket4j (500 req/s default): [5](#0-4) 

The rest-java module has no equivalent. Its `RestJavaProperties` contains only response-header configuration: [6](#0-5) 

The documented rest-java DB configuration exposes no `maxConnections` property (Spring Boot HikariCP default: 10), and a `statementTimeout` of 10 000 ms: [7](#0-6) 

**Root cause:** Every `GET /api/v1/topics/{id}` for a non-existent ID issues one uncached `SELECT` against the `topic` table. Because `topicService.findById()` is called first in the controller and throws `EntityNotFoundException` immediately on a miss, the subsequent `entityService` and `customFeeService` calls are skipped — but the one topic SELECT still consumes a pool connection for its full round-trip. With no rate limiting and no negative-result cache, an attacker can keep all pool connections occupied with identical, always-missing SELECTs. [8](#0-7) 

### Impact Explanation
With the default HikariCP pool of 10 connections and a `statementTimeout` of 10 s, an attacker holding 10 concurrent long-running or rapidly-repeated requests can prevent any other request from acquiring a DB connection. Legitimate clients querying topic state (e.g., to verify topic existence before submitting consensus messages) receive connection-timeout errors or queued delays. The REST Java API becomes unavailable for topic resolution for the duration of the attack. No authentication or special privilege is required.

### Likelihood Explanation
The endpoint is public, requires no credentials, and accepts any numeric topic ID. A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple script) can sustain hundreds of concurrent requests. The attack is trivially repeatable, requires no knowledge of the system beyond the public API path, and leaves no persistent state that would block re-execution.

### Recommendation
1. **Add negative-result caching** to `TopicRepository` (and `EntityRepository`, `CustomFeeRepository` in rest-java) using `@Cacheable(unless = "#result == null")` with a short TTL (e.g., 5–30 s), mirroring the pattern already used in the grpc module.
2. **Add rate limiting** to the rest-java module using the same bucket4j pattern already present in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, applied as a servlet filter or Spring MVC interceptor on all `/api/v1/**` routes.
3. **Tune the HikariCP pool size** (`spring.datasource.hikari.maximum-pool-size`) and expose it as a configurable property in `RestJavaProperties`, consistent with how the Node.js REST API exposes `hiero.mirror.rest.db.pool.maxConnections`.

### Proof of Concept
```bash
# Flood with 200 concurrent requests to a guaranteed non-existent topic ID
# Requires no authentication
ab -n 10000 -c 200 http://<mirror-node-host>/api/v1/topics/999999999999

# Simultaneously, observe legitimate requests timing out:
curl -v http://<mirror-node-host>/api/v1/topics/0.0.1
# Expected: connection pool exhaustion → 500 or timeout
```

Preconditions: network access to the REST Java API port (default 8080). No credentials needed. The topic ID `999999999999` need not exist — any non-existent ID suffices.

public class RestJavaProperties {

    @NotNull
    @Valid
    private ResponseConfig response = new ResponseConfig();

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }
