### Title
Case-Insensitive Validation Without Normalization Causes Incorrect SQL Filter and Wrong Cached Results for `result` Parameter

### Summary
The `result` query parameter is validated case-insensitively (accepting `success`, `SUCCESS`, `Success`, etc.) but the validated value is never normalized to lowercase before a strict equality comparison against `constants.transactionResultFilter.SUCCESS` (`'success'`). Any mixed-case or uppercase variant (e.g., `result=SUCCESS`) causes the comparison to evaluate `false`, silently treating the request as a FAIL filter and returning the wrong transaction set. Because the cache key is derived directly from the raw URL, this incorrect result is then stored and served from cache under a distinct key.

### Finding Description

**Step 1 — Validation accepts mixed-case without normalizing:**

In `rest/utils.js` line 342, the `RESULT` filter key is validated with `isValidValueIgnoreCase`, which accepts `'success'`, `'SUCCESS'`, `'Success'`, `'fail'`, `'FAIL'`, etc. The raw user-supplied string is stored as-is into the filter object — no `.toLowerCase()` normalization is applied. [1](#0-0) 

**Step 2 — Strict equality comparison downstream:**

In `rest/transactions.js` line 427, the stored `resultType` value is compared with strict equality (`===`) against `constants.transactionResultFilter.SUCCESS`, which is the lowercase string `'success'`. [2](#0-1) [3](#0-2) 

If the user submitted `result=SUCCESS`, then `resultType` is `'SUCCESS'`. The comparison `'SUCCESS' === 'success'` is `false`, so the operator becomes `'not in'` — the query returns **failed** transactions instead of successful ones. The user asked for successes and gets failures.

**Step 3 — Wrong result is cached under a distinct key:**

The cache key generator in `rest/middleware/responseCacheHandler.js` line 152 hashes `req.originalUrl` verbatim with no normalization. [4](#0-3) 

`GET /api/v1/transactions?result=success` and `GET /api/v1/transactions?result=SUCCESS` produce different MD5 hashes, so they are cached independently. The incorrect result set (failed transactions) is stored and served from cache for all subsequent requests using the uppercase variant.

**Why existing checks are insufficient:**

The test suite for `validateAndParseFilters` confirms `'SUCCESS'` and `'FAIL'` are accepted as valid: [5](#0-4) 

But the `extractSqlFromTransactionsRequest` tests only exercise lowercase `'success'` and `'fail'`: [6](#0-5) 

There is no test covering the uppercase path through SQL generation, so the mismatch is undetected.

### Impact Explanation
Any unauthenticated user querying `/api/v1/transactions?result=SUCCESS` (or any mixed-case variant) receives a response containing **failed** transactions instead of successful ones — the exact opposite of the requested filter. This incorrect response is then cached in Redis and served to all subsequent callers using the same URL variant for the duration of the TTL. This constitutes a data integrity violation: the API silently returns wrong data with no error, and the cache amplifies the incorrect result.

### Likelihood Explanation
No privileges are required. The exploit requires only a standard HTTP GET request with a mixed-case `result` parameter. It is trivially reproducible by any external user, automated client, or API explorer that uses uppercase parameter values (a common convention). The behavior is deterministic and repeatable on every request.

### Recommendation
Normalize the `result` filter value to lowercase immediately after validation, before it is stored in the filter object. The fix should be applied in the filter parsing/validation pipeline in `rest/utils.js` so that by the time any downstream code (e.g., `extractSqlFromTransactionsRequest`) receives the value, it is guaranteed to be lowercase. Alternatively, change the comparison in `rest/transactions.js` line 427 to use `.toLowerCase()`:

```js
const operator = resultType.toLowerCase() === constants.transactionResultFilter.SUCCESS ? 'in' : 'not in';
```

The same normalization should be applied consistently to `CREDIT_TYPE` and `ORDER` filters for defense in depth.

### Proof of Concept

1. Start the mirror node REST API.
2. Send: `GET /api/v1/transactions?result=SUCCESS`
3. Observe: the response contains transactions where `result != 22` (i.e., failed transactions), not successful ones.
4. Send: `GET /api/v1/transactions?result=success`
5. Observe: the response contains transactions where `result = 22` (successful transactions) — the correct set.
6. The two responses differ in content despite semantically identical intent, and each is independently cached under its own key.

Root cause confirmed at `rest/transactions.js` line 427: `resultType === constants.transactionResultFilter.SUCCESS` evaluates `'SUCCESS' === 'success'` → `false` → wrong SQL operator selected.
