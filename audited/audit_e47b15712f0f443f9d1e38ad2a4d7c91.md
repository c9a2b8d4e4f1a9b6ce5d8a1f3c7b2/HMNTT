### Title
Retriable: true on ErrBlockNotFound Enables Amplified DB Query DoS via Traefik Retry Compounding

### Summary
`ErrBlockNotFound` (code 101) is declared with `Retriable: true` in `rosetta/app/errors/errors.go`, which is semantically incorrect — a non-existent block will never appear on retry. Combined with the Traefik `retry` middleware configured with `attempts: 3`, every single client request for a non-existent block identifier triggers **4 raw database queries** (1 original + 3 Traefik retries) against the `record_file` table, with no application-level rate limiting or caching in the Rosetta service itself. An unprivileged attacker sending concurrent requests from multiple IPs can exhaust the DB connection pool and raise CPU/DB load well above 30%.

### Finding Description

**Exact code locations:**

`rosetta/app/errors/errors.go`, line 60:
```go
ErrBlockNotFound = newError(BlockNotFound, 101, true)
``` [1](#0-0) 

`newError` at line 114 sets `Retriable: retriable` directly on the `types.Error` struct, which is serialized into the Rosetta HTTP 500 response body. [2](#0-1) 

**DB query path — no caching for misses:**

`findBlockByIndex` in `rosetta/app/persistence/block.go` executes a raw SQL query on every call with no negative-result caching: [3](#0-2) 

`findBlockByHash` similarly executes a raw SQL query on every call: [4](#0-3) 

**No application-level rate limiting in the Rosetta Go service:**

`main.go` wraps the router only with `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting or request throttling at the application layer: [5](#0-4) 

**Traefik retry middleware multiplies DB load 4×:**

`charts/hedera-mirror-rosetta/values.yaml` configures Traefik with `retry: attempts: 3`, meaning every HTTP 500 response (which is what Rosetta errors produce) is retried 3 additional times by Traefik before the client sees the response: [6](#0-5) 

The ingress is enabled by default and references this middleware CRD: [7](#0-6) 

**Rate limit is per `requestHost`, not per source IP:**

The `rateLimit` middleware uses `sourceCriterion: requestHost: true`, meaning the 10 req/s limit is shared across all clients hitting the same hostname — not per attacker IP. Multiple source IPs are each independently subject to the `inFlightReq: amount: 5` per-IP limit, but that still allows 5 concurrent requests per IP × 4 DB queries each = 20 DB queries per IP per round-trip: [8](#0-7) 

**Root cause and failed assumption:**

The failed assumption is that `ErrBlockNotFound` is a transient condition worth retrying. It is not — a block with a fabricated index or random hash will never exist. Setting `Retriable: true` signals both the Traefik retry middleware (which acts on 5xx HTTP status) and compliant Rosetta SDK clients to retry, multiplying DB load with zero chance of success.

### Impact Explanation

Each attacker request for a non-existent block identifier causes 4 raw SQL queries against the `record_file` table (1 original + 3 Traefik retries). The DB connection pool is configured with `maxOpenConnections: 100` by default. With 10 concurrent source IPs each holding 5 in-flight requests, that is 10 × 5 × 4 = 200 concurrent DB queries, saturating the default pool. This directly raises DB CPU and connection utilization well above 30% compared to baseline, degrading or denying service to legitimate users. The `blockAPIService.Block` handler also calls `FindBetween` for transactions if the block lookup succeeds, but even the failed lookup path alone is sufficient for the attack. [9](#0-8) 

### Likelihood Explanation

No authentication or API key is required to call the `/block` endpoint — it is a public HTTP POST endpoint on port 5700. The attacker needs only an HTTP client and the ability to send JSON POST requests with arbitrary `block_identifier` values (e.g., `{"index": 999999999}`). The attack is trivially scriptable, requires no special knowledge of the chain state, and can be sustained indefinitely. Distributing across multiple IPs bypasses the per-IP `inFlightReq` limit. The Traefik retry amplification is automatic and requires no attacker-side retry logic.

### Recommendation

1. **Change `ErrBlockNotFound` to `Retriable: false`** in `rosetta/app/errors/errors.go` line 60. A missing block is a permanent condition for a given identifier and should never be retried.
2. **Add application-level rate limiting** in the Rosetta Go service (e.g., `golang.org/x/time/rate` per-IP token bucket) before the DB layer is reached, independent of infrastructure middleware.
3. **Change the Traefik `retry` middleware** to not retry on Rosetta application errors (HTTP 500 with a Rosetta error body), or reduce `attempts` to 0 for the `/block` path.
4. **Change the `rateLimit` `sourceCriterion`** from `requestHost: true` to `ipStrategy` to enforce per-source-IP rate limiting.

### Proof of Concept

```bash
# Send concurrent requests for a non-existent block index
# Each request triggers 4 DB queries (1 + 3 Traefik retries)
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/block \
    -H "Content-Type: application/json" \
    -d '{
      "network_identifier": {"blockchain": "Hedera", "network": "mainnet"},
      "block_identifier": {"index": 999999999}
    }' &
done
wait

# Expected response (repeated 4x per request at DB layer):
# {"code":101,"message":"Block not found","retriable":true}

# Monitor DB connections:
# SELECT count(*) FROM pg_stat_activity WHERE datname = 'mirror_node';
# Observe connection count spike to pool maximum (100)
```

Sending 50 concurrent requests from a single IP is blocked by `inFlightReq: amount: 5`, but 10 IPs sending 5 concurrent requests each (easily achieved with a botnet or cloud VMs) produces 200 simultaneous DB queries, saturating the default pool of 100 connections and causing queuing/timeout for legitimate traffic.

func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
	if index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)

func (br *blockRepository) findBlockByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	if rb.Index < br.genesisBlock.Index {
		log.Errorf("The block with hash %s is before the genesis block", hash)
		return nil, hErrors.ErrBlockNotFound
	}

	return rb.ToBlock(br.genesisBlock), nil
}
