### Title
Unbounded `FindBetween()` DB Amplification via `/block` Endpoint Allows Unauthenticated Resource Exhaustion

### Summary
The `/block` endpoint in the Rosetta service allows any unauthenticated caller to trigger `transactionRepository.FindBetween()`, which executes an unbounded number of batched SQL queries with two correlated subqueries per row against `crypto_transfer` and `staking_reward_transfer`. The `maxTransactionsInBlock` guard is applied only after all DB work completes, providing no protection against DB-level amplification. Repeated requests targeting a high-transaction block can drive DB CPU well above 30% of the 24-hour baseline.

### Finding Description

**Code path:**

`POST /block` → `blockAPIService.Block()` (`rosetta/app/services/block_service.go:56`) → `BaseService.FindBetween()` (`rosetta/app/services/base_service.go:54`) → `transactionRepository.FindBetween()` (`rosetta/app/persistence/transaction.go:112`)

**Root cause — two independent flaws compose:**

**Flaw 1 — `FindBetween()` has no total-row cap.** [1](#0-0) 

The loop runs until the DB returns fewer than `batchSize=2000` rows. For a block containing, say, 20 000 transactions, this issues 10 sequential SQL queries. Each query is:

```sql
SELECT t.*, 
  (SELECT json_agg(...) FROM crypto_transfer WHERE consensus_timestamp = t.consensus_timestamp ...) AS crypto_transfers,
  (SELECT json_agg(...) FROM staking_reward_transfer WHERE consensus_timestamp = t.consensus_timestamp) AS staking_reward_payouts
FROM transaction t
WHERE consensus_timestamp >= @start AND consensus_timestamp <= @end
ORDER BY consensus_timestamp
LIMIT 2000
``` [2](#0-1) 

Each batch of 2000 rows fires **2 correlated subqueries per row** = 4 000 additional index lookups per batch, 40 000 for the full block.

**Flaw 2 — `maxTransactionsInBlock` is a response-trimming guard, not a DB guard.** [3](#0-2) 

`FindBetween()` is called unconditionally at line 56 and returns the full result set. The slice truncation at line 61–66 happens entirely in Go memory after the DB has already done all the work. A block with 50 000 transactions and `maxTransactionsInBlock=100` still causes 25 full batched queries against the DB.

**Flaw 3 — No authentication, no rate limiting.**
The middleware stack is tracing → metrics → CORS only. [4](#0-3) 

No token, IP throttle, or concurrency limit is applied to `/block`.

### Impact Explanation

An attacker who identifies one or more historically dense blocks (e.g., during a network stress event) can issue concurrent `POST /block` requests targeting those blocks. Each request independently drives the DB through the full batched-query loop with correlated subqueries. Because the Rosetta DB connection pool is shared with other mirror-node consumers, sustained parallel requests can saturate DB CPU, increase query latency for all consumers, and cause the Rosetta service itself to time out on subsequent requests. The impact is a realistic, sustained >30% DB CPU increase achievable without brute force — a single attacker with modest bandwidth can maintain it indefinitely.

### Likelihood Explanation

The Rosetta API is a public, unauthenticated HTTP endpoint by design (Rosetta spec). Block indices are sequential integers; an attacker can trivially enumerate them to find dense blocks. No special knowledge, credentials, or exploit tooling is required — only a standard HTTP client. The attack is repeatable and stateless (each request is independent), making it trivially scriptable and resumable after interruption.

### Recommendation

1. **Apply a DB-level row cap inside `FindBetween()`**: add a configurable `maxRows` parameter and break the loop once the accumulated count exceeds it, returning an error or partial result before issuing further batches.
2. **Move the `maxTransactionsInBlock` guard before the DB call**: pass it into `FindBetween()` so the query loop stops as soon as enough transactions are collected.
3. **Replace correlated subqueries with JOINs or a single lateral join**: eliminates the per-row subquery fan-out.
4. **Add rate limiting middleware** on `/block` (e.g., token bucket per IP).
5. **Set a DB `statement_timeout`** in the connection config (`Db.StatementTimeout`) to bound the worst-case query duration. [5](#0-4) 

### Proof of Concept

```bash
# Step 1: Find a block with many transactions (enumerate by index)
curl -s -X POST http://<rosetta-host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index": <DENSE_BLOCK_INDEX>}}'

# Step 2: Flood with concurrent requests targeting that block
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index": <DENSE_BLOCK_INDEX>}}' &
done
wait
```

Each concurrent request independently executes the full `FindBetween()` batch loop with correlated subqueries. With 50 concurrent requests against a block containing 10 000 transactions, the DB receives 250 batched queries simultaneously, each with 4 000 correlated subquery executions, driving DB CPU above the 30% threshold without any authentication or brute-force enumeration.
