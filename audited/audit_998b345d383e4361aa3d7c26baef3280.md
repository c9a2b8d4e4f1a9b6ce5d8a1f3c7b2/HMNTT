### Title
Cache-Eviction Griefing via Unbounded Unique Query Rotation in `getDocumentAsync()`

### Summary
The `CachedPreparsedDocumentProvider` uses the raw query string as the cache key in a 1000-entry Caffeine cache with no query normalization and no per-client rate limiting. An unauthenticated attacker can continuously submit syntactically valid but unique queries (e.g., varying a single argument value or alias), rotating through more than 1000 distinct strings to keep evicting legitimate entries, forcing every legitimate query to be re-parsed and re-validated on every request.

### Finding Description
**Code location:**
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26, `getDocumentAsync()`
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15 (default: `maximumSize=1000`)
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, line 34 (`maxCharacters(10000)`)

**Root cause:**

```java
// CachedPreparsedDocumentProvider.java line 26
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

The cache key is `executionInput.getQuery()` — the raw, un-normalized query string. Two queries that are semantically identical but differ by a single character (different argument value, alias, or comment) produce distinct cache keys and occupy separate slots.

**Exploit flow:**

1. Attacker generates 1001+ syntactically valid, unique queries within the `maxCharacters(10000)` / `maxTokens(1000)` limits, e.g. by varying a numeric argument:
   ```
   { account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }
   { account(input:{entityId:{shard:0,realm:0,num:2}}){balance} }
   ...
   { account(input:{entityId:{shard:0,realm:0,num:1001}}){balance} }
   ```
2. Each request inserts a new entry into the 1000-slot cache, evicting the oldest entry (LRU/size-based eviction in Caffeine).
3. By cycling through ≥1001 unique queries in a tight loop, the attacker ensures the cache never retains any entry long enough for a legitimate repeated query to hit it.
4. Every legitimate user query misses the cache and triggers `parseAndValidateFunction.apply(executionInput)` — full parse + validation — on every single request.

**Why existing checks are insufficient:**

- `maxCharacters(10000)` limits query size but does not prevent uniqueness; an attacker needs only to vary one digit.
- `maxTokens(1000)` / `maxWhitespaceTokens(1000)` limit per-query token count, not the number of distinct queries submitted.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run *after* parsing, so parse overhead is already incurred before these checks fire.
- No per-IP rate limiting, no authentication, and no request throttling is present in the graphql module.

### Impact Explanation
All legitimate clients experience persistent cache misses, meaning every GraphQL request incurs full parse and validation overhead instead of a cache hit. Under sustained attack, this degrades throughput and increases CPU utilization on the mirror node. The impact is service degradation (griefing) with no direct economic harm to network users, consistent with the stated scope.

### Likelihood Explanation
The attack requires zero privileges — the GraphQL endpoint is publicly accessible with no authentication. Generating 1001 unique queries is trivial (a simple loop incrementing one integer). The attack is repeatable indefinitely and requires minimal bandwidth (small queries). Any script-kiddie with `curl` or a basic HTTP client can execute this.

### Recommendation
1. **Normalize the cache key** before caching: strip comments, normalize whitespace, and sort/canonicalize field selections so semantically equivalent queries share a single cache entry.
2. **Add per-IP (or per-client) rate limiting** at the HTTP layer (e.g., Spring's `HandlerInterceptor` or a reverse proxy) to bound the number of distinct queries any single client can inject per time window.
3. **Increase cache size** or use a size-weighted cache (weight by query length) to make eviction attacks more expensive.
4. **Consider query allow-listing** for production deployments, accepting only a known set of pre-registered query hashes.

### Proof of Concept
```bash
# Generate 1001 unique queries to saturate and rotate the 1000-entry cache
for i in $(seq 1 1001); do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input:{entityId:{shard:0,realm:0,num:$i}}){balance}}\"}" &
done
wait

# Now send a legitimate repeated query — it will always miss the cache
for j in $(seq 1 100); do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query": "{account(input:{entityId:{shard:0,realm:0,num:42}}){balance}}"}'
done
# Each of the 100 requests above triggers a full re-parse instead of a cache hit
```
