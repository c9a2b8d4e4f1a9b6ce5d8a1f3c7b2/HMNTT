### Title
Unbounded Timestamp Range Bypass Enables Resource Exhaustion via Full Outer Join on Transactions Endpoint

### Summary
The `getTransactions()` handler in `rest/transactions.js` explicitly disables timestamp range validation (`validateRange=false`) when calling `parseTimestampFilters`, and the `bindTimestampRange` feature is disabled by default. An unprivileged attacker can submit arbitrarily wide timestamp ranges combined with an `account.id` filter, triggering a full outer join across the `transaction`, `entity_transaction`, `crypto_transfer`, and `token_transfer` tables. While sub-query LIMIT clauses prevent literal full-table scans when indexes are used, the unbounded range forces each sub-query to scan the maximum index depth for the given account, and concurrent abuse can exhaust DB connection/CPU resources.

### Finding Description

**Code location:** `rest/transactions.js`, `getTransactions()`, line 674:

```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
```

The sixth argument is `validateRange = false`. In `rest/utils.js` lines 1657â€“1665, the range-size guard is gated on this flag:

```js
if (validateRange) {
  const {maxTimestampRange, maxTimestampRangeNs} = config.query;
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(...);
  }
}
```

Because `validateRange=false`, a closed range of `[0, 9999999999999999999]` passes without error.

**Second bypass â€” `bindTimestampRange`:** `rest/timestampRange.js` lines 19â€“22:

```js
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};   // returns the unbounded range unchanged
  }
```

`bindTimestampRange` is `false` by default (opt-in feature), so the range is never automatically constrained.

**Full outer join trigger:** `rest/transactions.js` lines 555â€“604. When `accountQuery` is non-empty (i.e., `account.id` filter is present), `getTransactionTimestampsQuery` builds:

```js
return `
    select coalesce(t.consensus_timestamp, ctl.consensus_timestamp, ttl.consensus_timestamp) ...
    from (${transactionOnlyQuery}) as t          -- transaction âˆª entity_transaction, LIMIT N
             full outer join (${cryptoTransferQuery}) as ctl   -- crypto_transfer, LIMIT N
                             on t.consensus_timestamp = ctl.consensus_timestamp
             full outer join (${tokenTransferQuery}) as ttl    -- token_transfer, LIMIT N
                             on coalesce(...) = ttl.consensus_timestamp
    order by consensus_timestamp ${order}
        ${limitQuery}`;
```

Each sub-query carries its own `LIMIT` (capped at `responseLimit.max`, default 100), but with a timestamp range of `[0, MAX]`, the database must traverse the full index depth for the given `entity_id`/`payer_account_id` to satisfy the ORDER BY + LIMIT. For high-volume accounts (e.g., treasury `0.0.2`), this index scan is deep and expensive. The `entity_transaction` union further doubles the work inside `transactionOnlyQuery`.

### Impact Explanation

Each request with a wide timestamp range and a common `account.id` forces four concurrent index scans (transaction, entity_transaction, crypto_transfer, token_transfer) plus a full outer join merge. Sending tens of concurrent such requests can saturate DB CPU and connection pool, causing latency spikes or outright denial of service for all API consumers. No authentication is required.

### Likelihood Explanation

The attack requires only a standard HTTP GET with two timestamp parameters and one `account.id` parameter â€” no credentials, no special knowledge. The endpoint is public. The parameters are documented in the OpenAPI spec. Any motivated attacker can script this in minutes and repeat it indefinitely.

### Recommendation

1. **Enable `validateRange` for the transactions endpoint.** Change the `parseTimestampFilters` call in `getTransactions()` to pass `validateRange=true` (last argument), so closed timestamp ranges are bounded by `maxTimestampRangeNs`.
2. **Enable `bindTimestampRange` by default**, or enforce it specifically for the transactions endpoint, so open/wide ranges are automatically clamped.
3. **Apply a per-IP or global rate limit** on the `/api/v1/transactions` endpoint at the API gateway layer.
4. **Add a DB statement timeout** (`statementTimeout` in pool config) to kill runaway queries before they exhaust resources.

### Proof of Concept

```bash
# Trigger full outer join with unbounded timestamp range (no auth required)
# Repeat concurrently (e.g., 50 parallel requests) to exhaust DB resources

for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/transactions?\
account.id=0.0.2&\
timestamp=gte:0&\
timestamp=lte:9999999999.999999999&\
limit=100&\
order=desc" &
done
wait
```

**Trigger:** The `account.id` parameter activates the full outer join path; the unbounded timestamp range forces deep index scans on all four tables simultaneously.
