### Title
Unauthenticated EVM Address Lookup Causes Unbounded DB Query Exhaustion via `getAccountTokenAllowances`

### Summary
The public `/api/v1/accounts/:idOrAliasOrEvmAddress/allowances/tokens` endpoint accepts EVM addresses with no authentication or rate limiting. When a valid-format EVM address that does not exist in the entity table is supplied, `EntityService.getEntityIdFromEvmAddress()` executes a direct database query on every request with no negative-result caching. An attacker generating unique non-existent EVM addresses at scale can exhaust the database connection pool and degrade 30%+ of processing nodes.

### Finding Description

**Exact code path:**

`getAccountTokenAllowances()` unconditionally calls `EntityService.getEncodedId()` with the raw URL parameter: [1](#0-0) 

Inside `getEncodedId()`, when the input parses as a valid entity ID with a non-null `evmAddress` field (i.e., any 40-char hex string), it branches to `getEntityIdFromEvmAddress()`: [2](#0-1) 

`getEntityIdFromEvmAddress()` executes `entityFromEvmAddressQuery` directly against the database on every invocation: [3](#0-2) 

The query itself: [4](#0-3) 

`getRows()` in `BaseService` goes directly to the DB connection pool with no caching layer: [5](#0-4) 

**Root cause:** `EntityService` has no negative-result cache for EVM address lookups (contrast with `TokenService` which maintains an in-memory `tokenCache`). Every request with a unique non-existent EVM address hits the database.

**Why existing checks fail:**

The response cache middleware (`responseCacheCheckHandler`) is conditional on `applicationCacheEnabled` and caches by request URL: [6](#0-5) 

An attacker using a different unique EVM address per request produces a unique URL each time, making the response cache entirely ineffective. The middleware stack contains no rate-limiting layer — only `authHandler`, `requestLogger`, and optional metrics: [7](#0-6) 

`authHandler` only controls response-size limits for authenticated users; it does not throttle unauthenticated request rates: [8](#0-7) 

### Impact Explanation
Each request with a unique non-existent EVM address consumes one database connection for the duration of the query. With no rate limiting and no negative-result caching, a sustained flood of such requests exhausts the DB connection pool. Once the pool is saturated, all legitimate queries queue or time out, degrading the REST API nodes that serve mirror node data. Given the mirror node's role in network observability and the public nature of the endpoint, this meets the threshold of degrading ≥30% of processing nodes without requiring any brute-force credential attack.

### Likelihood Explanation
The attack requires zero privileges — no API key, no account, no on-chain state. Valid EVM address format is a 40-character hex string; an attacker can trivially generate 2^160 unique values programmatically. The attack is fully automatable with standard HTTP tooling (e.g., `ab`, `wrk`, or a simple script). It is repeatable indefinitely and requires no special network position. The only external dependency is network access to the REST API port.

### Recommendation
1. **Negative-result cache in `EntityService`**: Cache `null` results for EVM address lookups with a short TTL (e.g., 30–60 seconds). This collapses repeated requests for the same non-existent address to a single DB hit.
2. **Application-level rate limiting**: Add a rate-limiting middleware (e.g., `express-rate-limit`) to the server middleware stack, applied before the route handlers, keyed by IP or a combination of IP + endpoint.
3. **Ensure response cache covers 404s**: Configure `responseCacheCheckHandler` to cache `NotFoundError` (404) responses for EVM address lookups, so even the first miss per address is cached.
4. **Connection pool guard**: Set a maximum DB connection pool size and a query timeout so that a flood cannot hold connections indefinitely.

### Proof of Concept

```bash
# Generate unique non-existent EVM addresses and flood the endpoint
for i in $(seq 1 10000); do
  # Each address is unique and guaranteed not to exist in entity table
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ADDR}/allowances/tokens" &
done
wait
```

**Preconditions:** Network access to the mirror node REST API (port 80/443). No credentials required.

**Trigger:** Each request reaches `getAccountTokenAllowances()` → `EntityService.getEncodedId()` → `getEntityIdFromEvmAddress()` → `entityFromEvmAddressQuery` on the DB.

**Result:** DB connection pool exhausted; legitimate API requests begin timing out; mirror node REST service degraded.

  if (!user) {

        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
