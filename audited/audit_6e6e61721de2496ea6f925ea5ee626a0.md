### Title
Cache Pollution via Whitespace-Varied Query Strings Bypasses Preparsed Document Cache (GraphQL DoS)

### Summary
`CachedPreparsedDocumentProvider` keys its Caffeine cache directly on the raw query string (`executionInput.getQuery()`), with no normalization. An unauthenticated attacker can send 1,000+ semantically identical queries that differ only in whitespace, generating unique cache keys that fill the 1,000-entry cache. Once full, Caffeine evicts existing entries, forcing every subsequent legitimate request to re-parse, re-validate, and re-analyze the query, degrading service throughput under sustained attack.

### Finding Description
**Exact code path:**

- `CachedPreparsedDocumentProvider.java` line 26: the cache lookup key is `executionInput.getQuery()` — the verbatim, un-normalized query string.
- `CacheProperties.java` line 15: the cache is configured as `expireAfterWrite=1h,maximumSize=1000,recordStats`.
- `GraphQlConfiguration.java` lines 33–34: `maxWhitespaceTokens(1000)` is set per-query, but this only rejects a single query that exceeds 1,000 whitespace tokens; it does not prevent an attacker from sending many queries each with a slightly different whitespace count.

**Root cause:** The cache key is the raw query string. Two queries that are semantically identical but differ by even a single space character (`" "`) are treated as entirely different cache entries. There is no query normalization (e.g., whitespace collapsing, AST-based keying) before the cache lookup.

**Exploit flow:**
1. Attacker sends query variant 0: `{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}` → cache miss → parsed → stored at key K0.
2. Attacker sends variant 1 with one extra leading space → cache miss → parsed → stored at key K1.
3. Repeat for variants 2–999 → cache is now full (1,000 entries).
4. Attacker sends variant 1000 → Caffeine evicts the least-recently-used entry (e.g., K0) to make room.
5. Any legitimate user sending the original query K0 now gets a cache miss and must re-parse.
6. Attacker continuously cycles through new whitespace variants, keeping the cache in a state of perpetual churn.

**Why existing checks are insufficient:**
- `maxWhitespaceTokens(1000)` (line 34) is a per-query parser guard; it does not limit the number of distinct queries an attacker can submit across requests.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (lines 43–44) execute *after* parsing and validation, so they do not prevent the parse-and-validate work from occurring on every cache miss.
- There is no rate limiting, per-IP throttling, or query normalization in the GraphQL layer itself.

### Impact Explanation
Every cache miss triggers a full parse, validation, and analysis cycle for the query. Under a sustained whitespace-cycling attack, the cache hit rate for legitimate queries approaches zero, forcing the GraphQL service to re-parse every request. This increases CPU consumption and response latency proportionally to request volume. The GraphQL service can be rendered effectively unresponsive without any authentication or special privileges. The impact is scoped to the GraphQL mirror node service (not the underlying Hiero network), but complete unavailability of this service endpoint is achievable.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public GraphQL endpoint, and no complex tooling — a simple loop sending HTTP POST requests with incrementally padded query strings suffices. The attack is repeatable and stateless. The 1,000-entry cache limit means only 1,000 requests are needed to saturate the cache initially, and a modest sustained rate (e.g., a few requests per second) is sufficient to maintain cache churn indefinitely. The `expireAfterWrite=1h` TTL means evicted legitimate entries do not self-heal quickly.

### Recommendation
1. **Normalize the cache key before lookup**: Strip or collapse all whitespace in the query string before using it as a cache key, or use the parsed AST's canonical string representation as the key.
2. **Alternatively, key on the AST hash**: Parse the query first (with limits enforced), then use a hash of the normalized AST as the cache key, so semantically equivalent queries always hit the same entry.
3. **Add per-IP or per-client rate limiting** at the GraphQL endpoint layer to bound the rate at which unique cache keys can be introduced.
4. **Consider increasing `maximumSize`** as a partial mitigation, though normalization is the correct fix.

### Proof of Concept
```bash
# Fill the cache with 1001 unique whitespace-varied keys
for i in $(seq 0 1000); do
  SPACES=$(printf '%*s' "$i" '')
  QUERY="{${SPACES}account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"
  curl -s -X POST http://<graphql-host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\"}" > /dev/null
done

# Now send the original (no-space) query — it was evicted; server must re-parse
curl -X POST http://<graphql-host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'

# Sustain the attack: keep cycling new whitespace variants to prevent cache stabilization
while true; do
  for i in $(seq 1001 2000); do
    SPACES=$(printf '%*s' "$i" '')
    QUERY="{${SPACES}account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"
    curl -s -X POST http://<graphql-host>:8083/graphql/alpha \
      -H 'Content-Type: application/json' \
      -d "{\"query\": \"$QUERY\"}" > /dev/null
  done
done
