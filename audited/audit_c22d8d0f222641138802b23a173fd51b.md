### Title
Unauthenticated gRPC Topic Subscription Exhausts Shared `boundedElastic()` Thread Pool, Causing DoS for Legitimate Subscribers

### Summary
The `PollingTopicMessageRetriever.retrieve()` method schedules all per-subscription database polling and timeout operations on a single shared `Schedulers.boundedElastic()` instance with no per-user, per-IP, or global subscription quota. Because the gRPC endpoint requires no authentication and imposes no connection-count limit, any external attacker can open an unbounded number of connections (each carrying up to 5 concurrent calls) to saturate the finite thread pool, blocking legitimate subscribers' polling tasks from executing.

### Finding Description
**Code path:**

- `PollingTopicMessageRetriever` constructor, line 41: `scheduler = Schedulers.boundedElastic();` — one shared scheduler for the entire service.
- `retrieve()`, lines 51–62: every subscription's `repeatWhen(...).withScheduler(scheduler)` and `.timeout(..., scheduler)` execute on this shared pool.
- `poll()`, line 78: `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` is a blocking JDBC call that holds the `boundedElastic()` thread for the full duration of the database query.
- `TopicMessageServiceImpl.subscribeTopic()`, line 70: the safety-check path also calls `.subscribeOn(Schedulers.boundedElastic())`, adding further pressure.
- `GrpcConfiguration`, lines 28–35: only `maxConcurrentCallsPerConnection = 5` is configured; no `maxConnections`, no `maxConnectionAge`, no global subscriber cap.
- `TopicMessageServiceImpl`, line 48: `subscriberCount` is a Micrometer gauge only — it is never checked to reject new subscriptions.
- The sole `GrpcInterceptor` (test package, but representative of production) performs no authentication or authorization.

**Root cause:** `Schedulers.boundedElastic()` defaults to `10 × CPU_cores` threads. Each active subscription's polling loop occupies one of those threads for the duration of a blocking JDBC call (up to `maxPageSize = 1000` rows per poll). With no connection limit and no subscription quota, an attacker can trivially exceed the thread cap, causing all subsequent polling tasks — including those of legitimate subscribers — to queue behind the attacker's tasks.

### Impact Explanation
When the thread pool is saturated:
1. Legitimate subscribers' `repeatWhen` callbacks queue behind attacker tasks; historical message delivery stalls.
2. The `.timeout(60s, scheduler)` check itself runs on the same pool; if the pool is full, timeout enforcement is delayed, keeping stalled subscriptions alive longer and worsening the backlog.
3. The safety-check `Mono.delay(1s).subscribeOn(Schedulers.boundedElastic())` in `TopicMessageServiceImpl` also queues, breaking gap-recovery for live subscribers.
4. Result: complete or near-complete denial of the HCS topic subscription service for all legitimate clients, with no self-recovery until the attacker disconnects.

Severity: **High** — full service unavailability for a public, unauthenticated API with no rate-limiting or connection cap.

### Likelihood Explanation
Preconditions: none. The gRPC port (default 5600) is publicly reachable, requires no credentials, and imposes no per-IP connection limit. A single attacker machine can open hundreds of TCP connections, each carrying 5 concurrent `subscribeTopic` calls with `startTime=0` (maximizing historical polling load). This is a trivial, repeatable, low-cost attack requiring only a gRPC client library.

### Recommendation
1. **Enforce a global and per-IP subscription cap** in `TopicMessageServiceImpl.subscribeTopic()` by checking `subscriberCount` (or a per-IP counter) against a configurable maximum before accepting a new subscription.
2. **Isolate per-subscription schedulers or use a bounded work queue** with fair scheduling (e.g., a separate `Schedulers.newBoundedElastic()` per tenant/IP, or move blocking DB calls to a dedicated executor with per-caller quotas).
3. **Add connection-level limits** in `GrpcConfiguration`: call `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnectionIdle(...)`, and consider a global `maxConnections` via a Netty `ChannelOption`.
4. **Add authentication/authorization** to the gRPC endpoint so that unauthenticated clients can be rate-limited or rejected at the transport layer.

### Proof of Concept
```
# Attacker script (pseudocode using grpc-java or grpcurl in a loop)
for i in 1..200:
    open new gRPC connection to mirror-node:5600
    for j in 1..5:
        call ConsensusService/subscribeTopic(
            topicID=<any valid topic>,
            consensusStartTime=0   # forces maximum historical polling load
        ) asynchronously, do not read responses

# Expected result:
# - boundedElastic thread pool (e.g., 40 threads on 4-core host) saturated
#   by 1000 concurrent blocking DB poll tasks
# - Legitimate subscriber calls to subscribeTopic stall; no messages delivered
# - Timeout enforcement delayed; stalled subscriptions persist
# - Service effectively unavailable until attacker disconnects
```
