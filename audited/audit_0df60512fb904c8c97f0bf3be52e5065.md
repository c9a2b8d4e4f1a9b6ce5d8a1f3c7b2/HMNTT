### Title
Unbounded gRPC Connection Exhaustion via Unauthenticated `subscribeTopic()` â€” No Global Connection or Subscription Limit

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent streaming subscriptions from any unauthenticated caller. The only server-side guard, `maxConcurrentCallsPerConnection = 5`, limits streams per TCP connection but imposes no cap on the number of TCP connections themselves. An attacker opening thousands of connections â€” each with 5 infinite-stream subscriptions â€” exhausts the server's thread pool, database connection pool, and heap, starving legitimate traffic.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43â€“53) immediately creates a reactive pipeline and subscribes it with no authentication, no IP check, and no rate gate:

```java
// ConsensusController.java lines 43-48
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 59â€“91) allocates a `TopicContext`, a historical retrieval `Flux`, a live listener `Flux`, and a safety-check `Flux` on `Schedulers.boundedElastic()` for every subscription. The `subscriberCount` field (line 48) is wired only to a Micrometer `Gauge` (lines 52â€“55) â€” it is **never checked** to reject new subscriptions.

**Server configuration:**

`GrpcConfiguration.java` (lines 31â€“34) configures the Netty server with exactly one constraint:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties.java` (line 14) sets this to `5`. No `maxConnections`, no `maxConnectionAge`, no `keepAliveTimeout`, no global concurrent-call ceiling is set. The `web3` module has a full Bucket4j rate-limiting stack (`ThrottleConfiguration`, `ThrottleManagerImpl`); the `grpc` module has none.

**Root cause:** The design assumes a single per-connection stream cap is sufficient, but HTTP/2 multiplexing means an attacker trivially bypasses it by opening more connections. There is no second line of defence.

### Impact Explanation

Each subscription allocates:
- A `TopicContext` object and associated `AtomicLong`/`AtomicReference` state
- A `Schedulers.boundedElastic()` thread for the safety-check `Mono.delay`
- One or more R2DBC database connections for historical retrieval and polling
- A listener registration in `RedisTopicListener` / `SharedPollingTopicListener`

With 2 000 attacker connections Ă— 5 streams = 10 000 concurrent infinite subscriptions (no `limit`, no `endTime`), the bounded-elastic scheduler queue overflows, the R2DBC connection pool is saturated, and heap pressure triggers GC storms. Legitimate `subscribeTopic` calls queue indefinitely or are rejected with `RESOURCE_EXHAUSTED`. The mirror node's gRPC surface becomes unavailable, blocking downstream clients from receiving HCS topic confirmations.

### Likelihood Explanation

No privileges, API keys, or special network position are required. The endpoint is publicly documented and reachable with `grpcurl` or any gRPC client library. Opening thousands of HTTP/2 connections from a single host (or a small botnet) is trivial with tools like `ghz` or a short Go/Python script. The attack is repeatable and can be sustained indefinitely because each subscription with no `endTime` and no `limit` never self-terminates. Detection is possible via the `hiero.mirror.grpc.subscribers` metric, but there is no automated enforcement action tied to it.

### Recommendation

1. **Global concurrent-subscription cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
2. **Per-IP connection rate limit**: Add a `ServerInterceptor` (alongside the existing `GrpcInterceptor`) that tracks connection/call rates per remote address using a token-bucket (Bucket4j is already a project dependency in `web3`).
3. **Netty connection ceiling**: In `GrpcConfiguration`, also call `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnectionIdle(...)`, and consider `serverBuilder.maxInboundConcurrentCallsPerConnection` together with a global `maxConnections` via a custom `ChannelzServerInterceptor` or Netty `ChannelHandler`.
4. **Require authentication**: Gate `subscribeTopic` behind at least a shared API token checked in the interceptor layer.

### Proof of Concept

```bash
# Install ghz: https://ghz.sh
# Fire 5000 concurrent infinite subscriptions from a single machine
ghz --insecure \
    --proto consensus_service.proto \
    --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    --data '{"topicID":{"topicNum":41110}}' \
    --concurrency 5000 \
    --connections 1000 \
    --duration 60s \
    localhost:5600
```

**Expected result:** After a few seconds the server's `hiero.mirror.grpc.subscribers` gauge climbs to the thread-pool/DB-pool ceiling; subsequent legitimate `subscribeTopic` calls from a separate client receive `RESOURCE_EXHAUSTED` or hang indefinitely. JVM heap and thread-count metrics confirm resource saturation.
