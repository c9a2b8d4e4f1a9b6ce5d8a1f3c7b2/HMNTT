### Title
Race Condition in RedisTopicListener Allows Stale Flux to Evict Active Shared Subscription from Map

### Summary
`RedisTopicListener.unsubscribe()` removes a topic entry from `topicMessages` by key only, without checking whether the stored value is the same `Flux` instance being cancelled. A race between `computeIfAbsent` returning a `Flux` reference and the caller actually subscribing to it allows an attacker who repeatedly subscribes and immediately cancels to trigger `unsubscribe()` at a moment when a newer `Flux` (F2) has already been stored in the map, causing F2 to be evicted while legitimate subscribers still hold it. Subsequent subscribers receive a third `Flux` (F3), fragmenting the shared subscription graph and creating windows during which messages are not delivered to affected subscribers.

### Finding Description

**Code locations:**

- `getSharedListener()` — [1](#0-0) 
- `subscribe()` / `doOnCancel` hook — [2](#0-1) 
- `unsubscribe()` — [3](#0-2) 

**Root cause:** `unsubscribe()` calls `topicMessages.remove(topic.getTopic())` — a key-only removal. It does not use the two-argument `ConcurrentHashMap.remove(key, value)` form, so it unconditionally removes whatever is currently stored under that key, even if it is a different, still-active `Flux` instance placed there after the cancelling `Flux` was already evicted.

**Exploit flow:**

```
T=0  Victim thread:   calls computeIfAbsent → receives reference to F1
T=1  Attacker thread: sole subscriber to F1 cancels
                      → .share() refcount → 0 → upstream cancel propagates
                      → doOnCancel fires → unsubscribe() removes F1 from map
T=2  New subscriber:  computeIfAbsent creates F2, stores F2 in map
T=3  Victim thread:   subscribes to F1 (stale reference)
                      → .share() refcount 0→1 → reconnects upstream to Redis
                      → F1 is NOT in the map; F2 is
T=4  Attacker thread: subscribes to topic → gets F2 from map, then cancels
                      → F2 refcount → 0 → doOnCancel fires
                      → unsubscribe() removes F2 from map
                      (F1 is still live with victim, but absent from map)
T=5  Any new subscriber: computeIfAbsent creates F3
     Victim is isolated on F1; new subscribers are on F3
     Messages published between T=1 and T=3 are missed by victim
```

**Why existing checks are insufficient:** `ConcurrentHashMap` makes individual operations atomic, but `computeIfAbsent` + deferred subscription is not a single atomic unit. The `doOnCancel` callback fires asynchronously on the Reactor scheduler, not under any lock, so the window between "last subscriber cancels" and "new subscriber calls computeIfAbsent" is real and exploitable under load. [4](#0-3) 

### Impact Explanation
Legitimate subscribers monitoring a topic for incoming fund-transfer messages can be silently isolated on a stale, map-orphaned `Flux`. They continue to receive a Redis stream, but it is a reconnected stream that missed messages published during the cancellation/reconnection gap. Additionally, the attacker's cycling creates repeated Redis `SUBSCRIBE`/`UNSUBSCRIBE` churn, causing resource exhaustion on the broker. Because the mirror node is the authoritative real-time delivery path for HCS topic messages (including payment confirmations), silent message loss is a high-severity availability and correctness issue for downstream financial applications.

### Likelihood Explanation
No authentication or rate-limiting on gRPC subscription calls is visible in this code path. Any unauthenticated client can open and immediately cancel a `subscribeToTopic` stream. The race window (between `computeIfAbsent` returning and the caller subscribing) is narrow but widens under load; automated tooling can fire hundreds of subscribe/cancel cycles per second, making reliable exploitation straightforward. The attack is stateless and repeatable.

### Recommendation
1. Replace the key-only removal with the atomic two-argument form:
   ```java
   private void unsubscribe(Topic topic, Flux<TopicMessage> flux) {
       topicMessages.remove(topic.getTopic(), flux);
   }
   ```
   Pass the specific `Flux` instance into `doOnCancel`/`doOnComplete` via a closure so only the owning `Flux` can evict itself.

2. Consider using `computeIfAbsent` with a `Sinks`-based approach where the sink is stored in the map and the `Flux` is derived from it, decoupling map lifecycle from subscriber refcount.

3. Apply rate-limiting or connection throttling at the gRPC layer to limit subscribe/cancel cycling from a single client.

### Proof of Concept
```
1. Open gRPC stream S1 to subscribeToTopic(topicId=X)  → F1 created, stored in map
2. Immediately cancel S1                                → F1 evicted from map
3. Open gRPC stream S2 to subscribeToTopic(topicId=X)  → F2 created, stored in map
   (S2 is the legitimate victim subscriber)
4. Open gRPC stream S3 to subscribeToTopic(topicId=X)  → returns F2 from map
5. Immediately cancel S3                                → F2 evicted from map
   (S2 is still subscribed to F2, but F2 is no longer in the map)
6. Publish a message to Redis channel topic.X
   → S2 receives it (F2 still connected), but any new subscriber gets F3
   → If step 5 races with S2's subscription setup, S2 misses the message
7. Repeat steps 1-6 in a tight loop to maximise collision probability
   and exhaust Redis connection resources.
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
