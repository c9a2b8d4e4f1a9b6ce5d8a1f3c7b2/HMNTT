### Title
Unauthenticated Brute-Force of Basic Auth Credentials Due to Missing Rate Limiting in `findUser()`

### Summary
The `findUser()` function in `rest/middleware/authHandler.js` performs credential lookup against `config.users` with no rate limiting, no failed-attempt counter, and no lockout mechanism. Any external attacker can submit an unlimited number of `Authorization: Basic ...` requests with varying credentials. A successful brute-force grants the attacker a higher per-request query `limit`, enabling bulk data harvesting of account balances and transaction histories that is otherwise throttled for unauthenticated callers.

### Finding Description
**Exact code path:**

`rest/middleware/authHandler.js`, lines 10‚Äì13 (`findUser`) and lines 15‚Äì36 (`authHandler`):

```js
const findUser = (username, password) => {
  const users = config.users || [];
  return users.find((user) => tsscmp(user.username, username) && tsscmp(user.password, password)) || null;
};
```

`tsscmp` is a timing-safe comparator ‚Äî it prevents timing-oracle attacks but provides zero protection against volumetric brute-force. Every failed attempt returns an immediate HTTP 401 with no delay, no counter increment, and no IP tracking.

`authHandler` is registered as a global Express middleware with no preceding rate-limiter:

```js
// rest/server.js line 86
app.useExt(authHandler);
```

A grep across all `rest/**/*.js` files for `rate.?limit|rateLimit|throttle|lockout|brute|attempt` returns zero hits in any auth-related file. The throttling that exists in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) is a separate Java service and does not apply to this Node.js REST API.

**Root cause:** The authentication middleware trusts that callers will not abuse the endpoint. There is no per-IP, per-username, or global failed-attempt budget enforced before or inside `findUser()`.

**Why existing checks fail:** `tsscmp` is the only defensive measure present. It eliminates timing side-channels but is entirely orthogonal to brute-force volume. A 401 response is returned in constant time, but it is still returned for every single attempt with no consequence to the attacker.

### Impact Explanation
A successful brute-force yields a session that sets `httpContext.set(userLimitLabel, user.limit)` (line 33), raising the effective query `limit` cap above the default maximum enforced for unauthenticated callers. The attacker can then issue paginated requests to `/api/v1/balances`, `/api/v1/accounts`, and `/api/v1/transactions` with a higher `limit` parameter, harvesting account balance data at a rate unavailable to anonymous users. This data can be used to identify high-value accounts for targeted attacks (phishing, SIM-swap, key-extraction). The mirror node is read-only, so funds cannot be moved directly through this API, but the intelligence gathered directly enables fund-theft targeting.

### Likelihood Explanation
Preconditions are minimal: network access to the REST API and knowledge that `config.users` is non-empty (inferable from a 401 response to any `Authorization: Basic` header). The username space is small (operator-configured, typically a handful of entries). Passwords are plain strings with no enforced complexity policy visible in `parseUsersConfig()` (`rest/config.js` lines 188‚Äì216). A dictionary or credential-stuffing attack against a small user list is trivially automated with `curl` or any HTTP fuzzer. No CAPTCHA, no IP ban, no exponential backoff, and no account lockout exist anywhere in the REST middleware stack.

### Recommendation
1. **Add a rate limiter before `authHandler`** using a library such as `express-rate-limit`, keyed on `req.ip`, targeting the authentication path specifically.
2. **Implement a failed-attempt counter** (in-memory or Redis-backed) per username/IP; lock out after N consecutive failures for a configurable window.
3. **Add artificial delay** on failed authentication (e.g., 1‚Äì2 s) to reduce throughput of automated attacks even without a full lockout.
4. **Enforce minimum password entropy** in `parseUsersConfig()` to raise the cost of offline dictionary attacks.

### Proof of Concept
```bash
# 1. Confirm a 401 is returned for wrong credentials (no lockout, no delay)
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Authorization: Basic $(echo -n "admin:password${i}" | base64)" \
    https://<mirror-node-host>/api/v1/transactions
done
# All 1000 requests return 401 immediately; no 429, no block.

# 2. On credential match, confirm elevated limit is granted
CREDS=$(echo -n "admin:correctpassword" | base64)
curl -s \
  -H "Authorization: Basic ${CREDS}" \
  "https://<mirror-node-host>/api/v1/balances?limit=500"

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};

const parseUsersConfig = () => {
  const users = getConfig().users || [];

  if (!Array.isArray(users)) {
    throw new InvalidConfigError('users configuration must be an array');
  }

  users.forEach((user, index) => {
    if (!user.username || typeof user.username !== 'string') {
      throw new InvalidConfigError(`users[${index}] must have a username string`);
    }
    if (!user.password || typeof user.password !== 'string') {
      throw new InvalidConfigError(`users[${index}] must have a password string`);
    }
    if (user.limit !== undefined) {
      const limit = parseInt(user.limit, 10);
      if (Number.isNaN(limit) || limit <= 0) {
        throw new InvalidConfigError(`users[${index}].limit must be a positive integer`);
      }
      user.limit = limit;
    }
  });

  const usernames = users.map((u) => u.username);
  const duplicates = usernames.filter((name, index) => usernames.indexOf(name) !== index);
  if (duplicates.length > 0) {
    throw new InvalidConfigError(`Duplicate usernames in users configuration: ${duplicates.join(', ')}`);
  }
};
