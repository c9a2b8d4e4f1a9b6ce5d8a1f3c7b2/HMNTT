### Title
Unbounded Storage Slot Accumulation in `captureStorage()` Causes Per-Request Memory Exhaustion via `storage=true` Opcode Replay

### Summary
When `storage=true` is passed to `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes`, the `captureStorage()` method in `AbstractOpcodeTracer` is invoked for **every single opcode** during EVM replay with no upper bound on the number of storage accesses collected. For a transaction that touched many storage slots across many opcodes, this produces O(opcodes × storage_slots) in-memory allocation per request. The rate limiter (1 req/sec globally) limits frequency but does not bound per-request memory, allowing a single crafted request to exhaust JVM heap on the node.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (line 53–68) accepts `storage=true` from any unauthenticated caller, calls `throttleManager.throttleOpcodeRequest()` (line 61), then delegates to `opcodeService.processOpcodeCall(request)` (line 64).

During EVM replay, `OpcodeActionTracer.tracePostExecution()` is called for every opcode:

```
// OpcodeActionTracer.java line 56
final var storage = captureStorage(frame, options, context);
```

`AbstractOpcodeTracer.captureStorage()` (lines 75–129) unconditionally iterates **all** accumulated storage accesses from `rootProxyWorldUpdater.getEvmFrameState().getTxStorageUsage(true).accesses()` and builds a new `TreeMap<String, String>` on every invocation:

```java
// lines 103–124
final var updates = rootProxyWorldUpdater
        .getEvmFrameState()
        .getTxStorageUsage(true)
        .accesses();
...
final var result = new TreeMap<String, String>();
for (final var storageAccesses : updates) {
    for (final var access : storageAccesses.accesses()) {
        final var key = hexCache.get(access.key(), Bytes::toHexString);
        ...
        result.put(key, value);
    }
}
return result;
```

Each returned `TreeMap` is attached to an `Opcode` object and appended to `OpcodeContext.opcodes` (an unbounded `ArrayList`). There is no cap on the number of opcodes, no cap on the number of storage entries per opcode, and no response-size limit anywhere in the pipeline.

**Root cause:** The failed assumption is that storage accesses per transaction are small enough to be safely serialized per-opcode. In reality, a contract that reads/writes N storage slots across T opcodes causes `captureStorage()` to allocate T separate `TreeMap` instances each containing up to N entries — O(T × N) total heap allocation for a single request.

**Why existing checks fail:**

- `validateAcceptEncodingHeader()` (line 75–86) requires `gzip`, which reduces **network** bandwidth but does not reduce **server-side heap** allocation — the full uncompressed object graph is built in memory before serialization.
- `throttleOpcodeRequest()` (line 61) uses a token bucket defaulting to `opcodeRequestsPerSecond = 1` (ThrottleProperties line 29). This limits to 1 request/second globally but places no bound on the memory consumed by that single request.
- `OpcodesProperties.enabled = false` by default (line 11) — but once an operator enables the feature, any unauthenticated caller can exploit it.

### Impact Explanation
A single request targeting a transaction with, e.g., 10,000 opcodes and 1,000 distinct storage accesses causes the server to allocate approximately 10,000 `TreeMap` instances each with 1,000 `String` entries. At ~200 bytes per entry, this is ~2 GB of heap for one request. Java `OutOfMemoryError` or severe GC pressure can crash or stall the web3 node process. At 1 request/second, an attacker can sustain this indefinitely. Because the web3 node is part of the mirror node infrastructure serving all downstream consumers, its unavailability constitutes a significant service disruption.

### Likelihood Explanation
The attacker requires no credentials — the endpoint has no authentication. The only precondition is that the operator has enabled the opcode tracer feature. The attacker needs only a valid transaction hash for a contract with many storage accesses (easily found by scanning historical on-chain data). The exploit is a single HTTP GET request with `Accept-Encoding: gzip` and `?storage=true`. It is repeatable at 1/second indefinitely.

### Recommendation
1. **Cap storage entries per opcode**: In `captureStorage()`, enforce a configurable maximum (e.g., 1,000 entries) and truncate or return an error if exceeded.
2. **Cap total opcodes**: In `OpcodeContext.addOpcodes()`, enforce a configurable maximum opcode count and abort replay if exceeded.
3. **Add a response-size circuit breaker**: Track estimated serialized size during replay and abort if it exceeds a configured threshold (e.g., 50 MB pre-compression).
4. **Tighten the rate limit**: Consider per-IP rate limiting in addition to the global bucket, and reduce the default to prevent sustained single-attacker abuse.

### Proof of Concept
1. Identify a historical transaction hash for a contract that performs many `SSTORE`/`SLOAD` operations across many storage slots (e.g., a contract that iterates and writes to a large mapping).
2. Send:
   ```
   GET /api/v1/contracts/results/<tx_hash>/opcodes?storage=true&stack=false&memory=false
   Accept-Encoding: gzip
   ```
3. Observe the web3 node JVM heap grow to OOM or GC thrash during the replay, as `captureStorage()` is called for every opcode and builds an unbounded `TreeMap` each time.
4. Repeat at 1-second intervals to sustain memory pressure.
