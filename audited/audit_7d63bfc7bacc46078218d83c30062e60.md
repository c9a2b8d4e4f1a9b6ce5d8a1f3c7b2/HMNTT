### Title
Data Race on `br.genesisBlock` in `blockRepository.initGenesisRecordFile` Enables Startup-Phase DoS

### Summary
`blockRepository.initGenesisRecordFile` contains a classic double-checked locking anti-pattern: the fast-path guard at line 244 reads `br.genesisBlock.ConsensusStart` without any synchronization, while the write `br.genesisBlock = rb` at line 258 is protected only by `sync.Once`. When multiple goroutines concurrently call `initGenesisRecordFile` before genesis is initialized, a goroutine writing the full `recordBlock` struct (including non-atomic string fields) races with another goroutine reading `ConsensusStart`, constituting a data race under the Go memory model. This can cause a panic that crashes the mirror node process.

### Finding Description

**Exact code path:**

`rosetta/app/persistence/block.go`, function `initGenesisRecordFile`, lines 243–263:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    // LINE 244: UNPROTECTED READ — no mutex, no atomic load
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }
    // ... DB query ...
    br.once.Do(func() {
        br.genesisBlock = rb   // LINE 258: WRITE of entire struct (5 fields)
    })
    log.Infof("...", br.genesisBlock.Index)  // LINE 261: UNPROTECTED READ
    return nil
}
```

`recordBlock` is a value-type struct with two `string` fields (`Hash`, `PrevHash`):

```go
type recordBlock struct {
    ConsensusStart int64
    ConsensusEnd   int64
    Hash           string   // 2-word value: pointer + length
    Index          int64
    PrevHash       string   // 2-word value: pointer + length
}
```

**Root cause:** The fast-path check at line 244 reads `br.genesisBlock.ConsensusStart` outside any synchronization primitive. `sync.Once` only serializes the write at line 258; it provides no happens-before guarantee for reads that occur *outside* `once.Do`. The Go memory model requires explicit synchronization for every concurrent access to shared memory — a `sync.Once`-protected write does not make unguarded reads of the same variable safe.

**Race scenario (two goroutines, G1 and G2):**

1. Mirror node starts; `br.genesisBlock.ConsensusStart == -1` (unset).
2. G1 and G2 both call `initGenesisRecordFile` concurrently (triggered by two simultaneous HTTP requests).
3. Both pass the check at line 244 (both read `-1`).
4. Both complete the DB query and reach `once.Do`.
5. G1 enters `once.Do` and begins writing `br.genesisBlock = rb` — a non-atomic multi-word struct assignment that writes `Hash` (pointer + length) and `PrevHash` (pointer + length) among other fields.
6. G2, blocked on `once.Do`, is released after G1's `once.Do` returns — but a *third* goroutine G3 (another concurrent request) can simultaneously be at line 244 reading `br.genesisBlock.ConsensusStart` while G1 is mid-write at line 258.
7. The concurrent read (G3, line 244) and write (G1, line 258) to the same memory location with no synchronization = **data race**.

**Why existing checks are insufficient:**

- `sync.Once` serializes the *write* but does not protect the *read* at line 244. The fast-path check is intentionally outside `once.Do` for performance, but this breaks the required synchronization contract.
- There is no `sync.RWMutex`, `sync/atomic` operation, or any other mechanism guarding the read at line 244 or line 261.
- The Go race detector (`-race`) will flag this as a data race on `br.genesisBlock`.

### Impact Explanation

The struct write `br.genesisBlock = rb` is not atomic. It involves writing string header values (pointer + length pairs for `Hash` and `PrevHash`). A concurrent reader at line 244 or line 261 can observe a torn write — e.g., a new pointer with the old length, or vice versa. In Go, accessing a string with a mismatched pointer/length pair causes a runtime panic (`runtime: out of range`), which crashes the mirror node process. Even without a panic, a torn read of `ConsensusStart` could cause the fast-path to return early with an uninitialized `genesisBlock`, causing all subsequent block lookups to use index 0 and `ConsensusStart = -1`, silently corrupting block responses for the lifetime of the process.

### Likelihood Explanation

The race window exists from process start until the first successful `once.Do` completes — typically a few hundred milliseconds to seconds depending on DB latency. Any unprivileged attacker who can send HTTP requests to the Rosetta API (which is a public-facing service per `main.go` line 231) can reliably trigger this by sending a burst of concurrent requests (e.g., 10–50 parallel `POST /block` requests) immediately after the mirror node starts. This is trivially scriptable with `curl --parallel` or any HTTP load tool. The attack is repeatable on every process restart.

### Recommendation

Replace the unprotected fast-path read with an atomic or mutex-guarded check. The idiomatic Go fix is to use `sync/atomic` for the sentinel field, or to move the entire check inside a `sync.RWMutex`:

```go
// Option 1: use sync/atomic for the sentinel
// Change ConsensusStart to be checked via atomic.LoadInt64
if atomic.LoadInt64(&br.genesisBlock.ConsensusStart) != genesisConsensusStartUnset {
    return nil
}
// ...
br.once.Do(func() {
    br.genesisBlock = rb
    atomic.StoreInt64(&br.genesisBlock.ConsensusStart, rb.ConsensusStart)
})

// Option 2 (cleaner): use sync.RWMutex to guard all accesses to br.genesisBlock
br.mu.RLock()
initialized := br.genesisBlock.ConsensusStart != genesisConsensusStartUnset
br.mu.RUnlock()
if initialized { return nil }
// ... DB query ...
br.mu.Lock()
if br.genesisBlock.ConsensusStart == genesisConsensusStartUnset {
    br.genesisBlock = rb
}
br.mu.Unlock()
```

Option 2 (full `sync.RWMutex`) is preferred because it also protects all other reads of `br.genesisBlock` scattered across `findBlockByIndex` (line 211), `findBlockByHash` (line 235), `RetrieveLatest` (line 203), and `RetrieveGenesis` (line 187), which are all currently unprotected reads that race with the write.
