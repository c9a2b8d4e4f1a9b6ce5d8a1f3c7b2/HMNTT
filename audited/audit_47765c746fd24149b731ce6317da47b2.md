### Title
eth_estimateGas Binary Search Bypasses Gas Throttle Bucket — Up to 21x EVM Computation Amplification

### Summary
The `gasLimitBucket` in `ThrottleConfiguration.gasLimitBucket()` is charged exactly once per HTTP request based on the user-supplied gas limit, regardless of whether the request is `eth_call` or `eth_estimateGas`. Because `eth_estimateGas` internally triggers one initial EVM execution plus up to 20 additional binary-search EVM executions (via `BinaryGasEstimator.search()`), an unprivileged attacker can obtain up to 21× more EVM computation per gas token consumed from the shared bucket compared to a plain `eth_call`. This constitutes a resource-amplification DoS vector requiring no credentials.

### Finding Description

**Throttle charge point — one shot per request:**

`ContractController.call()` calls `throttleManager.throttle(request)` exactly once before dispatching to `ContractExecutionService.processCall()`. [1](#0-0) 

Inside `ThrottleManagerImpl.throttle()`, the bucket is debited `scaleGas(request.getGas())` tokens — a single debit for the entire lifetime of the request. [2](#0-1) 

**eth_estimateGas path — multiple EVM executions:**

When `params.isEstimate() == true`, `processCall()` routes to `estimateGas()`. [3](#0-2) 

`estimateGas()` first calls `callContract(params, context)` (EVM execution #1, `estimate=false`), then immediately invokes `binaryGasEstimator.search(...)` passing `gas -> doProcessCall(params, gas, true)` as the iteration callback — up to 20 more EVM executions. [4](#0-3) 

`BinaryGasEstimator.search()` loops up to `maxGasEstimateRetriesCount` (default **20**) times, each time calling `doProcessCall` with `estimate=true`. [5](#0-4) 

**Why the binary-search iterations are free from the bucket's perspective:**

`doProcessCall()` only calls `restoreGasToBucket()` when `!estimate`. When `estimate=true` (all binary-search iterations), the `finally` block skips the restore entirely — and crucially, no *additional* debit is made either. [6](#0-5) 

The result: the bucket is charged once for `scaleGas(request.getGas())` tokens, but the EVM is invoked up to 21 times (1 initial + 20 binary-search iterations), each potentially executing a full, complex contract.

**Existing checks are insufficient:**

- `rateLimitBucket` caps at 500 requests/second globally, but does not prevent the per-request amplification.
- `gasLimitBucket` is a single shared bucket with no partition by call type.
- `maxGasEstimateRetriesCount = 20` is a hard cap on iterations but is the amplification factor itself.
- `scaleGas()` divides by 10,000, reducing token granularity but not the amplification ratio. [7](#0-6) [8](#0-7) 

### Impact Explanation
An attacker sending `eth_estimateGas` with `gas=15_000_000` (the maximum) consumes the same bucket tokens as a single `eth_call` at the same gas limit, but forces the server to execute the EVM up to 21 times. At the default rate limit of 500 req/s, this translates to up to **10,500 EVM executions per second** while the bucket only accounts for 500. For computationally expensive contracts (precompile-heavy, storage-intensive), this can saturate CPU and memory, causing degraded service or complete unavailability of the mirror node's web3 API for all users. No funds are at risk (mirror node is read-only for `eth_call`/`eth_estimateGas`), but availability is directly threatened.

### Likelihood Explanation
The exploit requires no authentication, no on-chain assets, and no special knowledge beyond standard JSON-RPC. Any public-facing mirror node deployment is reachable. The attacker needs only an HTTP client and a contract address (or can deploy a gas-burning contract). The attack is trivially repeatable and scriptable. The amplification factor (up to 21×) is deterministic and reliable.

### Recommendation
1. **Charge the bucket per EVM execution, not per HTTP request.** Debit `scaleGas(mid)` from the `gasLimitBucket` inside `BinaryGasEstimator.search()` before each iteration, and restore unused gas after each iteration, mirroring the `!estimate` path in `doProcessCall`.
2. **Alternatively, partition the bucket by call type.** Introduce a separate, lower-capacity `gasLimitBucket` for `eth_estimateGas` requests that accounts for the expected amplification factor (e.g., capacity = `gasPerSecond / maxGasEstimateRetriesCount`).
3. **Reduce `maxGasEstimateRetriesCount`** as a short-term mitigation to lower the amplification ceiling.
4. **Add per-IP rate limiting** as a defense-in-depth layer to limit the blast radius from a single source.
