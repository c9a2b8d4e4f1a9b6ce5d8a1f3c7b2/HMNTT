### Title
Cache Miss on Non-Existent Block Indices Enables Unauthenticated DB Saturation via `findByIndex()`

### Summary
`RecordFileRepository.findByIndex()` is annotated with `@Cacheable(unless = "#result == null")`. Spring's cache abstraction unwraps `Optional` return values before evaluating SpEL expressions, so `Optional.empty()` (returned for any non-existent block index) is treated as `null`, causing the `unless` condition to be `true` and preventing the result from being cached. An unauthenticated attacker can flood the endpoint with requests for non-existent block numbers, each of which bypasses the cache and issues a live database query. The only protection is a single global (not per-IP) rate limiter of 500 req/s, which the attacker can fully consume, starving legitimate users and saturating the DB connection pool.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` (line 28) unconditionally delegates numeric block lookups to `recordFileRepository.findByIndex(block.number())` with no upper-bound validation: [1](#0-0) 

`RecordFileRepository.findByIndex()` is declared as: [2](#0-1) 

**Root cause — Spring Optional unwrapping in `unless`:**

Spring's `CacheAspectSupport` calls `unwrapReturnValue()` on the method's return value before evaluating `unless`. `ObjectUtils.unwrapOptional(Optional.empty())` returns `null`. Therefore `#result == null` evaluates to `true` for every non-existent block index, and the empty result is never stored in the `CACHE_MANAGER_RECORD_FILE_INDEX` Caffeine cache (configured with `maximumSize(10000)`, `expireAfterWrite(10 min)`): [3](#0-2) 

**Exploit flow:**

1. Attacker sends `eth_call` / `eth_getBlockByNumber` requests with distinct non-existent block numbers (e.g., `9999999000`, `9999999001`, …).
2. Each request reaches `findByBlockType()` → `findByIndex(N)`.
3. Cache lookup misses (key `N` was never stored because the previous miss was not cached).
4. JPA executes `SELECT r FROM RecordFile r WHERE r.index = N` against the database.
5. DB returns empty result → `Optional.empty()` → Spring unwraps to `null` → `unless` fires → not cached.
6. Repeat indefinitely; every unique `N` is a guaranteed DB round-trip.

**Why existing checks fail:**

The only rate-limiting mechanism is a **global** (not per-IP) token-bucket with a default of **500 requests/second**: [4](#0-3) 

A single attacker can consume all 500 tokens/second. The throttle is applied to `ContractCallRequest` objects at the contract-call layer: [5](#0-4) 

There is no per-IP rate limiting, no authentication requirement, and no validation that the requested block number is within a plausible range before the DB query is issued.

### Impact Explanation

At 500 req/s (the full global budget), the attacker drives 500 uncached `SELECT … WHERE index = ?` queries per second against the `record_file` table. This:
- Exhausts the DB connection pool, increasing query latency for all legitimate requests.
- Consumes the entire global rate-limit budget, causing legitimate `eth_call` requests to receive `429 Too Many Requests` responses.
- Because block resolution is on the critical path for every historical contract call, legitimate callers experience delays proportional to DB queue depth — easily exceeding 5× the normal per-block processing time under sustained attack.

### Likelihood Explanation

No privileges, API keys, or special network access are required. The attack requires only the ability to send HTTP requests to the public JSON-RPC endpoint. The attacker needs no knowledge of the system beyond the standard Ethereum JSON-RPC API (`eth_call` with a `blockNumber` parameter). The attack is trivially scriptable, repeatable, and can be sustained indefinitely from a single host within the global rate limit, or amplified across multiple IPs to exceed it.

### Recommendation

1. **Cache negative results explicitly.** Change `unless = "#result == null"` to `unless = "false"` (or remove `unless` entirely) on `findByIndex()`, so that `Optional.empty()` is stored in the cache under the queried index key. This eliminates repeated DB hits for the same non-existent index.
2. **Add per-IP rate limiting** at the ingress layer (reverse proxy or Spring `HandlerInterceptor`) so a single source cannot consume the global budget.
3. **Validate block number range** in `findByBlockType()` before issuing the DB query: reject any block number greater than the current latest block index (obtainable from the already-cached `findLatest()` result).

### Proof of Concept

# Send 500 requests/second with distinct non-existent block numbers
# Each request bypasses the cache and issues a live DB query

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{
      \"data\": \"0x\",
      \"to\": \"0x0000000000000000000000000000000000000001\",
      \"block\": \"$((9000000000 + i))\"
    }" &
done
wait

# Repeat in a loop to sustain the attack.
