### Title
Unauthenticated Database CPU/IO Exhaustion via Uncached SQL Query in `/network/status`

### Summary
Every call to `NetworkStatus()` unconditionally executes the `latestNodeServiceEndpoints` SQL query — a multi-join, `string_agg`-aggregated query — against the database with no application-level caching or rate limiting. An unprivileged external attacker can flood `POST /network/status` with concurrent requests, causing each to independently execute this expensive query, exhausting database CPU and I/O and rendering the service unavailable.

### Finding Description
**Code path:**

`rosetta/app/services/network_service.go`, `NetworkStatus()`, line 77:
```go
peers, err := n.addressBookEntryRepo.Entries(ctx)
``` [1](#0-0) 

This calls `Entries()` in `rosetta/app/persistence/address_book_entry.go` (lines 57–83), which executes `latestNodeServiceEndpoints` — a raw SQL query with a correlated subquery (`MAX(start_consensus_timestamp)`), three JOINs (`address_book_entry`, `node`, `address_book_service_endpoint`), `string_agg` with `ORDER BY`, and `GROUP BY` — on **every single invocation**, with no caching: [2](#0-1) [3](#0-2) 

**Root cause:** The `addressBookEntryRepository` struct holds no cache. The `Entries()` method always issues up to two full SQL queries per call (once for file 101, once for file 102 if 101 returns empty). There is no memoization, TTL cache, or in-process deduplication anywhere in the rosetta Go service.

**Application-level middleware:** `main.go` wraps the router only with `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting: [4](#0-3) 

**Existing checks and why they fail:**

The Traefik-based rate limiting defined in `charts/hedera-mirror-rosetta/values.yaml` (lines 157–160) is gated on `global.middleware`, which defaults to `false`: [5](#0-4) [6](#0-5) 

Even when enabled, `rateLimit.sourceCriterion: requestHost: true` limits by hostname (not IP), trivially bypassed by rotating source IPs or using multiple clients. The `inFlightReq: amount: 5` per-IP limit only caps concurrency per single IP, not aggregate load from distributed sources.

The `statementTimeout` in `db/client.go` (line 36) provides a per-query deadline but does not prevent a flood of concurrent queries from saturating the DB connection pool and CPU simultaneously: [7](#0-6) 

### Impact Explanation
An attacker sustaining a high-concurrency flood of `POST /network/status` requests forces the PostgreSQL database to execute the multi-join `string_agg` query in parallel for every request. This exhausts database CPU, I/O bandwidth, and connection pool slots, causing query latency to spike across all services sharing the database (importer, REST API, gRPC). The Rosetta service becomes unresponsive, and dependent blockchain infrastructure (exchanges, validators, wallets relying on the Rosetta API) loses access to network state — a non-network DoS with potential systemic impact on the Hedera/Hiero ecosystem.

### Likelihood Explanation
The endpoint requires no authentication, no API key, and no session. A single attacker with a modest botnet (or even a single high-throughput client using HTTP/1.1 pipelining or HTTP/2 multiplexing) can sustain thousands of concurrent requests. The Traefik middleware is off by default, and even when on, the per-host rate limit is trivially bypassed. This is fully repeatable and requires no special knowledge beyond the public Rosetta API specification.

### Recommendation
1. **Add an in-process result cache** in `addressBookEntryRepository.Entries()` (e.g., a TTL-based cache with a 30–60 second expiry), so the expensive SQL query is executed at most once per interval regardless of request volume.
2. **Enable and harden the Traefik middleware** by default, switching `rateLimit.sourceCriterion` to `ipStrategy` (per-IP) rather than `requestHost`.
3. **Add a global concurrency semaphore** or connection pool limit at the application layer to cap simultaneous in-flight DB queries for this endpoint.
4. **Set a conservative `statementTimeout`** in the DB client config to bound the worst-case duration of each query.

### Proof of Concept
```bash
# Obtain a valid network_identifier first (e.g., from /network/list)
NETWORK_ID='{"blockchain":"Hiero","network":"mainnet"}'

# Flood /network/status with 200 concurrent goroutines (adjust URL as needed)
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/network/status \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\": $NETWORK_ID, \"metadata\": {}}" &
done
wait
```
Each concurrent request independently triggers the `latestNodeServiceEndpoints` query. Monitor PostgreSQL `pg_stat_activity` and CPU utilization — both will spike to saturation. The Rosetta service will begin returning timeouts or errors for all endpoints as the DB connection pool is exhausted.

const latestNodeServiceEndpoints = `select
                                    abe.node_id,
                                    coalesce(n.account_id, abe.node_account_id) as node_account_id,
                                    string_agg(ip_address_v4 || ':' || port::text, ','

func (aber *addressBookEntryRepository) Entries(ctx context.Context) (*types.AddressBookEntries, *rTypes.Error) {
	db, cancel := aber.dbClient.GetDbWithContext(ctx)
	defer cancel()

	nodes := make([]nodeServiceEndpoint, 0)
	// address book file 101 has service endpoints for nodes, resort to file 102 if 101 doesn't exist
	for _, fileId := range []int64{aber.addressBook101.EncodedId, aber.addressBook102.EncodedId} {
		if err := db.Raw(
			latestNodeServiceEndpoints,
			sql.Named("file_id", fileId),
		).Scan(&nodes).Error; err != nil {
			log.Error("Failed to get latest node service endpoints", err)
			return nil, errors.ErrDatabaseError
		}

		if len(nodes) != 0 {
			break
		}
	}
