### Title
Unbounded INFO Log Flooding via Rapid gRPC Subscription Open/Cancel Cycling

### Summary
`TopicContext.finished()` unconditionally emits an INFO-level log entry for every terminal signal, including `SignalType.CANCEL`. Because there is no per-client rate limit on subscription creation in the gRPC service layer, an unprivileged attacker can open and immediately cancel subscriptions in a tight loop, generating two INFO log entries per cycle (one at subscription open, one at cancel) with no throttling or back-pressure.

### Finding Description
**Exact code path:**

- `TopicMessageServiceImpl.subscribeTopic()` — line 60: `log.info("Subscribing to topic: {}", filter)` fires unconditionally on every call.
- Line 91: `.doFinally(topicContext::finished)` attaches `finished()` to every terminal signal.
- `TopicContext.finished()` lines 227–236: calls `log.info(...)` unconditionally regardless of whether `signalType` is `ON_COMPLETE`, `ON_ERROR`, or `CANCEL`.

```java
// line 60
log.info("Subscribing to topic: {}", filter);

// line 91
.doFinally(topicContext::finished)

// lines 227-236
void finished(SignalType signalType) {
    log.info(
            "[{}] Topic {} {} with {} messages in {} ({}/s)",
            filter.getSubscriberId(),
            signalType,
            topicId,
            count,
            stopwatch,
            rate());
}
```

**Root cause:** The `finished()` method does not branch on `signalType`; it logs at INFO for cancellations identically to normal completions. Combined with the absence of any per-client or per-IP subscription rate limit in the gRPC service, every open+cancel cycle produces exactly 2 INFO log lines.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties` line 14, applied in `GrpcConfiguration` line 33) limits *concurrent* calls per TCP connection, not the *rate* of new connections or the rate of subscription cycling across connections.
- The `ThrottleConfiguration` (bucket4j) exists only in the `web3` module; it is absent from the `grpc` module entirely.
- The GCP backend policy `maxRatePerEndpoint: 250` in the Helm chart is an optional infrastructure layer that is not enforced at the application level and may not be deployed.
- The `topicExists` DB check (line 87) is async/non-blocking and does not prevent the attacker from cancelling immediately after subscribing, which still triggers `doFinally`.

### Impact Explanation
Each open+cancel cycle writes 2 INFO log lines to the application log. At scale (e.g., 10 connections × 5 concurrent calls × rapid cycling), an attacker can generate thousands of log lines per second. This can:
- Exhaust log storage quotas on the node or in a centralized log aggregator (e.g., Elasticsearch, Loki, Stackdriver).
- Degrade log aggregation pipeline performance, causing legitimate operational log entries (errors, warnings) to be delayed, dropped, or buried.
- Increase infrastructure costs for log ingestion and retention.

Severity is Medium (griefing, no direct economic damage to network users, but operational impact on the mirror node operator).

### Likelihood Explanation
The gRPC port (default 5600) is publicly exposed. No authentication is required to call `subscribeTopic`. The attack requires only a standard gRPC client (e.g., `grpcurl`, any HCS SDK) and a valid (or even non-existent, if `checkTopicExists=false`) topic ID. The cycle can be scripted trivially and repeated indefinitely. The `maxConcurrentCallsPerConnection = 5` limit is easily bypassed by opening multiple TCP connections.

### Recommendation
1. **Conditional log level for cancellations:** In `TopicContext.finished()`, branch on `signalType` and log cancellations at `DEBUG` or `WARN` instead of `INFO`:
   ```java
   void finished(SignalType signalType) {
       if (signalType == SignalType.CANCEL) {
           log.debug("[{}] Topic {} {} with {} messages in {} ({}/s)", ...);
       } else {
           log.info("[{}] Topic {} {} with {} messages in {} ({}/s)", ...);
       }
   }
   ```
2. **Per-IP or per-client subscription rate limiting** in the gRPC service layer (e.g., using bucket4j, similar to the `web3` module's `ThrottleConfiguration`).
3. **Connection rate limiting** at the Netty level or load balancer to cap new connections per source IP.

### Proof of Concept
```bash
# Using grpcurl, loop rapidly opening and cancelling subscriptions
while true; do
  # Each invocation subscribes and is immediately killed (SIGINT = cancel)
  timeout 0.05 grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    &>/dev/null &
done
```

Each iteration triggers:
1. `log.info("Subscribing to topic: {}", filter)` at line 60.
2. `finished(SignalType.CANCEL)` → `log.info(...)` at lines 227–236.

Running 10 parallel connections × 5 concurrent calls each, cycling at ~20 Hz, produces ~2,000 INFO log lines/second with no application-level rejection. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
