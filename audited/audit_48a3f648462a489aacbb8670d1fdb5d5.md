### Title
Unbounded `X-Real-IP` Header Causes Per-Request ~1 MB Heap Allocation in `TracingMiddleware`, Enabling GC-Pressure CPU Exhaustion

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` reads the `X-Real-IP` header and the full request URI with no length cap, then passes both directly into `fmt.Sprintf` to build a log message. Because Go's default `MaxHeaderBytes` allows headers up to 1 MB, an unauthenticated attacker can craft requests that force a ~1 MB heap allocation per request. Sustained at a moderate rate this creates enough short-lived garbage to trigger Go's GC far more frequently than normal, measurably increasing CPU consumption.

### Finding Description
**Exact code path**

`getClientIpAddress` (lines 63–74) reads `X-Real-IP` with a single empty-string guard and no length limit:
```go
ipAddress := r.Header.Get(xRealIpHeader)
if len(ipAddress) == 0 { ... }   // only guards empty, not oversized
```
`TracingMiddleware` (lines 47, 52–53) then concatenates the unbounded IP string with the unbounded request URI into a new heap string:
```go
path := request.URL.RequestURI()          // no length cap
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

**Root cause / failed assumption**
The code assumes `X-Real-IP` will contain a short IP address string. Go's `net/http` server enforces `MaxHeaderBytes` (default 1,048,576 bytes, never overridden in `main.go` lines 220–227) but does **not** limit individual header values. A single `X-Real-IP` value can therefore be close to 1 MB. The request line is capped at ~4 KB by Go's HTTP/1.1 parser, so the combined `fmt.Sprintf` output can reach ~1 MB per request.

**Why existing checks are insufficient**
- `len(ipAddress) == 0` rejects only the empty string; any non-empty value, however large, passes through.
- The Traefik rate-limit middleware (`rateLimit: average: 10`) is **disabled by default** (`global.middleware: false`, `values.yaml` line 95). Even when enabled it is keyed on `requestHost` (destination), not source IP, so a single attacker is not individually throttled.
- There is no `MaxHeaderBytes` override in the `http.Server` struct, so the 1 MB default applies.

### Impact Explanation
Each malicious request allocates roughly 1 MB of short-lived string data on the Go heap. At 100 requests/second that is ~100 MB/s of garbage, forcing the Go GC to run orders of magnitude more often than for normal ~100-byte log messages (ratio ≈ 10,000×). Benchmark data for Go GC shows that sustained high-allocation rates can push GC CPU overhead from <1 % to 30–50 % of total CPU. On a lightly loaded node this can breach the 30 % threshold stated in the threat model, degrading response latency for all legitimate users and potentially triggering the Traefik circuit-breaker (`NetworkErrorRatio() > 0.25`), causing a partial outage.

### Likelihood Explanation
- **No authentication** is required; any internet-reachable client can send arbitrary headers.
- The attack requires only standard HTTP tooling (`curl`, `wrk`, `ab`) with a custom header.
- The Traefik protective middleware is off by default, so most deployments are unprotected.
- The attacker does not need to sustain a high packet rate; even 50–100 req/s with 900 KB `X-Real-IP` values is sufficient.
- The attack is repeatable and stateless; no session or token is needed.

### Recommendation
1. **Truncate `clientIpAddress` before logging**: cap it to a safe maximum (e.g., 45 characters for IPv6):
   ```go
   const maxIPLen = 45
   if len(ipAddress) > maxIPLen {
       ipAddress = ipAddress[:maxIPLen]
   }
   ```
2. **Truncate `path` before logging** (e.g., 512 characters).
3. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g., 8 KB) since legitimate Rosetta requests carry only standard headers.
4. **Enable the Traefik middleware by default** and key `rateLimit` on source IP (`ipStrategy`) rather than `requestHost`.

### Proof of Concept
```bash
# Generate a ~900 KB X-Real-IP value
FAKE_IP=$(python3 -c "print('A' * 921600)")

# Send 100 req/s for 60 seconds
wrk -t4 -c100 -d60s \
    -H "X-Real-IP: ${FAKE_IP}" \
    -H "Content-Type: application/json" \
    --script <(echo 'wrk.method="POST"; wrk.body="{}"') \
    http://<rosetta-host>:5700/network/list

# Observe Go runtime metrics: GC pause frequency and CPU usage
# Expected: GC runs increase from ~1/s to >100/s; CPU rises >30%
```
