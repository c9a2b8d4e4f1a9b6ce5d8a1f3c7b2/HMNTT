### Title
Unbounded Memory Consumption via Unlimited Log and State Change Fetching in `getContractResultsByTimestamp`

### Summary
The `GET /contracts/:contractId/results/:consensusTimestamp` endpoint calls `getContractLogsByTimestamps()` and `getContractStateChangesByTimestamps()` with no `LIMIT` clause, causing all matching rows for a given timestamp to be fetched and materialized in memory. Any unauthenticated user who knows a valid contract ID and consensus timestamp for a high-activity transaction can trigger this, repeatedly exhausting Node.js heap memory across mirror node instances.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 18 routes `GET /:contractId/results/:consensusTimestamp` to `getContractResultsByTimestamp`.

`rest/controllers/contractController.js` lines 1002–1042: `getContractResultsByTimestamp` calls `getDetailedContractResults` at line 1017.

`rest/controllers/contractController.js` lines 1205–1220: `getDetailedContractResults` issues a `Promise.all` that includes:
- `ContractService.getContractLogsByTimestamps(contractDetails.consensusTimestamp, contractDetails.contractIds)` (line 1213)
- `ContractService.getContractStateChangesByTimestamps(contractDetails.consensusTimestamp, contractId, contractDetails.contractIds)` (lines 1214–1218)

`rest/service/contractService.js` lines 395–414: `getContractLogsByTimestamps` builds a SQL query with only a `WHERE` and `ORDER BY` clause — **no `LIMIT`**:
```js
const query = [ContractService.contractLogsWithEvmAddressQuery, whereClause, orderClause].join('\n');
const rows = await super.getRows(query, params);
return rows.map((row) => new ContractLog(row));
```

`rest/service/contractService.js` lines 416–450: `getContractStateChangesByTimestamps` similarly builds a query with only `WHERE` and `ORDER BY` — **no `LIMIT`**:
```js
const query = [ContractService.contractStateChangesQuery, whereClause, orderClause].join('\n');
const rows = await super.getRows(query, params);
```

**Root cause:** The `limit` parameter enforced on paginated list endpoints is never passed to these two internal fetch functions. The assumption that a single transaction's logs and state changes are bounded in practice is false — a complex DeFi or batch-processing contract can emit thousands of EVM events and write thousands of storage slots in one transaction.

**Why existing checks fail:**
- The REST API has no per-IP or per-endpoint rate limiting (the `ThrottleConfiguration`/`ThrottleManagerImpl` found in `web3/` applies only to the web3 JSON-RPC module, not the REST API).
- The `authHandler` middleware (`rest/middleware/authHandler.js`) only sets a custom `limit` for paginated list responses; it has no effect on this endpoint.
- The `requestQueryParser` and `requestHandler` middleware impose no cap on response size or row count for this code path.

### Impact Explanation
Each request to a high-activity timestamp causes the Node.js process to allocate heap memory proportional to the number of logs × state changes for that transaction. With thousands of rows, a single request can consume hundreds of MB. Concurrent requests targeting the same or different high-activity timestamps can exhaust available heap, triggering OOM crashes or severe GC pressure across mirror node instances. If 30%+ of nodes are targeted simultaneously (trivially achievable with a small botnet or even a single fast client), network-wide processing capacity is degraded.

### Likelihood Explanation
- No authentication or privilege is required.
- The contractId and consensusTimestamp are public on-chain data, discoverable via the same REST API (`/api/v1/contracts/results`).
- High-activity contracts (DEX routers, NFT mints, airdrop contracts) are well-known and their busy transactions are easily identified.
- The attack is repeatable with a simple loop; no cryptographic material or special tooling is needed.

### Recommendation
1. Add a hard `LIMIT` clause to both `getContractLogsByTimestamps` and `getContractStateChangesByTimestamps` (e.g., cap at `config.response.limit.max` or a dedicated constant).
2. If the full set is legitimately needed, paginate the sub-queries and return a `206 Partial Content` or a `links.next` cursor.
3. Add per-IP rate limiting middleware (e.g., `express-rate-limit`) to the REST API, specifically for the `/:contractId/results/:consensusTimestamp` route.

### Proof of Concept

**Precondition:** Identify a contract transaction with many logs. This is trivially done via:
```
GET /api/v1/contracts/results?limit=25&order=desc
```
Pick any `contract_id` and `timestamp` from a known high-activity contract (e.g., a DEX router).

**Trigger (no auth required):**
```bash
# Single request — fetches ALL logs and state changes for that timestamp into memory
curl "https://<mirror-node>/api/v1/contracts/0.0.1234/results/1700000000.123456789"

# Amplified attack — concurrent requests
for i in $(seq 1 100); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234/results/1700000000.123456789" &
done
wait
```

**Result:** Node.js heap grows unboundedly per request; with sufficient concurrency the process OOMs or becomes unresponsive, degrading or shutting down the mirror node.
