### Title
Log Injection via Unsanitized `X-Real-IP` Header in `TracingMiddleware`

### Summary
`getClientIpAddress()` reads the `X-Real-IP` header with no content validation beyond an empty-string check, and `TracingMiddleware` embeds the raw value directly into a `fmt.Sprintf`-constructed log message. Because the application explicitly configures logrus `TextFormatter` for logfmt output (targeting Loki), an attacker who controls the `X-Real-IP` header can inject arbitrary logfmt key=value tokens into the `msg` field, causing naive or misconfigured log parsers to interpret fabricated fields (e.g., `level=error`, `method=DELETE`, `path=/block/1`) as real log entries, corrupting the apparent request/transaction history.

### Finding Description
**Code path:**

- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74: reads `X-Real-IP` with only an empty-length guard — no character filtering, no IP format validation.
- `rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 52–53: constructs the log message as `fmt.Sprintf("%s %s %s (%d) in %s", clientIpAddress, request.Method, path, ...)` — the attacker-controlled string is the first token.
- `rosetta/main.go`, lines 43–52: logrus is configured with `log.TextFormatter{DisableColors: true, FullTimestamp: true}` and the comment explicitly states *"Use logfmt for easy parsing by Loki"*.

**Root cause:** The failed assumption is that `X-Real-IP` will always contain a bare IP address. No RFC 5737 / IP-format validation is applied.

**Exploit flow:**
1. Attacker sends any valid HTTP request to the Rosetta API with header:
   `X-Real-IP: 1.2.3.4 level=error method=DELETE path=/block/99`
2. `getClientIpAddress()` returns the full string `1.2.3.4 level=error method=DELETE path=/block/99`.
3. `fmt.Sprintf` produces: `1.2.3.4 level=error method=DELETE path=/block/99 GET /network/list (200) in 3ms`.
4. Logrus TextFormatter wraps this in `msg="..."` because it contains spaces, yielding:
   ```
   time="..." level=info msg="1.2.3.4 level=error method=DELETE path=/block/99 GET /network/list (200) in 3ms"
   ```
5. Any log consumer that tokenizes on whitespace without respecting logfmt quoting (common in Splunk pipelines, grep-based alerting, ELK Grok patterns, or Loki label-extraction rules) will parse `level=error`, `method=DELETE`, and `path=/block/99` as top-level fields, fabricating a high-severity DELETE event against block 99.

**Why existing checks fail:** `trace_test.go` tests only that a well-formed IP like `10.0.0.100` appears in the output; there is no test or runtime guard against header values containing spaces, `=`, or other logfmt-significant characters.

### Impact Explanation
An unprivileged external attacker can silently inject arbitrary logfmt fields into every access-log line they trigger. In a Loki/Grafana or SIEM environment, this allows: (a) spoofing `level=error` or `level=fatal` to trigger false alerts; (b) injecting fake `path=/block/N` or `path=/construction/submit` entries to fabricate an apparent transaction or block-reorganization history; (c) drowning real anomaly signals in noise. Because the Rosetta API is a blockchain data-access layer, falsified block/transaction log entries directly mislead operators about the node's apparent state. No authentication is required to set HTTP headers.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request — no credentials, no special network position. The attack is trivially repeatable and scriptable. The logfmt injection is silent (no error, no rejected request) and leaves no distinguishing marker in the fabricated log fields. Any internet-facing or load-balancer-fronted deployment is exposed.

### Recommendation
Sanitize the IP string returned by `getClientIpAddress()` before embedding it in the log message. The minimal fix is to validate that the value is a well-formed IP address (or CIDR/comma-list for `X-Forwarded-For`) using `net.ParseIP` / `net.ParseCIDR`, and fall back to `r.RemoteAddr` if validation fails:

```go
func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        if raw := r.Header.Get(header); raw != "" {
            // take only the first token and validate it
            candidate := strings.SplitN(raw, ",", 2)[0]
            candidate = strings.TrimSpace(candidate)
            if ip := net.ParseIP(candidate); ip != nil {
                return ip.String()
            }
        }
    }
    ip, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ip
}
```

Additionally, add a test case in `trace_test.go` that supplies a header value containing spaces and logfmt tokens, asserting the injected tokens do not appear as separate fields in the log output.

### Proof of Concept
```bash
# Send a request with an injected X-Real-IP value
curl -s http://<rosetta-host>:<port>/network/list \
  -X POST \
  -H 'Content-Type: application/json' \
  -H 'X-Real-IP: 1.2.3.4 level=error method=DELETE path=/block/99' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# Observe the resulting log line (stdout / Loki):
# time="..." level=info msg="1.2.3.4 level=error method=DELETE path=/block/99 POST /network/list (200) in Xms"
#
# A naive logfmt tokenizer will extract:
#   level  = error          ← fabricated severity
#   method = DELETE         ← fabricated HTTP method
#   path   = /block/99      ← fabricated block endpoint
# making it appear a privileged DELETE /block/99 occurred at error severity.
```
