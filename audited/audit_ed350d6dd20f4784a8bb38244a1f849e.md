### Title
Unsynchronized Genesis Initialization Allows Unbounded Concurrent DB Query Amplification

### Summary
`initGenesisRecordFile` in `rosetta/app/persistence/block.go` performs an unsynchronized check of `br.genesisBlock.ConsensusStart` before executing the expensive `selectGenesis` query. The `sync.Once` guard is misplaced — it only protects the *assignment* of `br.genesisBlock`, not the DB query itself. Any number of concurrent goroutines can simultaneously pass the check and each independently acquire a connection pool slot and execute the full CTE query. When genesis data is unavailable (node syncing), `once.Do` is never reached, making this condition permanent and every request a full DB query.

### Finding Description

**Exact code path:** `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263.

```
line 244: if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset { // unsynchronized read
line 248: db, cancel := br.dbClient.GetDbWithContext(ctx)                   // connection acquired
line 252: db.Raw(selectGenesis, ...).First(&rb)                             // expensive CTE query
line 254: return handleDatabaseError(err, hErrors.ErrNodeIsStarting)        // once.Do never reached on error
line 257: br.once.Do(func() { br.genesisBlock = rb })                       // only guards the write
```

**Root cause:** The `sync.Once` is placed *after* the DB query, so it only serializes the write to `br.genesisBlock`. The check at line 244 and the query at line 252 are completely unsynchronized. N goroutines can simultaneously observe `ConsensusStart == -1`, all call `GetDbWithContext`, and all execute `selectGenesis` concurrently.

**Two exploit paths:**

1. **Startup race window:** During server startup, before any request completes genesis initialization, an attacker floods block endpoints. All concurrent goroutines pass line 244 simultaneously and each executes the full `selectGenesis` CTE.

2. **Persistent amplification (more severe):** When genesis data is not yet in the DB (node still syncing), the query returns `gorm.ErrRecordNotFound`, `handleDatabaseError` returns `ErrNodeIsStarting` at line 254, and `once.Do` is **never called**. `br.genesisBlock.ConsensusStart` remains `-1` permanently. Every subsequent request to any of the five public endpoints (`FindByHash`, `FindByIndex`, `FindByIdentifier`, `RetrieveGenesis`, `RetrieveLatest`) unconditionally executes the full `selectGenesis` query with no deduplication, no caching, and no rate limiting.

**`selectGenesis` query cost:** The query is a CTE joining `account_balance` (a large, partitioned table) with `record_file`, ordered and limited — non-trivial cost per execution.

**Connection pool:** `GetDbWithContext` draws from a bounded pool configured via `Pool.MaxOpenConnections`. Each concurrent call holds a slot for the duration of the query. [1](#0-0) [2](#0-1) 

### Impact Explanation
An unprivileged attacker sending concurrent HTTP requests to any block endpoint during node sync (which can last hours or days) causes every request to independently execute the `selectGenesis` CTE. This directly increases DB CPU and I/O proportional to request concurrency. With a modest number of concurrent connections (e.g., 20–50), the DB query load increases by well over 30% compared to normal operation. If `MaxOpenConnections` is reached, legitimate requests queue or fail, causing denial of service for all Rosetta API consumers. [3](#0-2) 

### Likelihood Explanation
The Rosetta API is a public HTTP service with no authentication required on block query endpoints. The persistent failure scenario (node syncing) is a normal operational state that can last hours. The attacker needs only an HTTP client and knowledge of any block endpoint URL. The attack is trivially repeatable and requires no special privileges, credentials, or protocol knowledge. [4](#0-3) 

### Recommendation
Move the entire check-query-assign sequence inside `sync.Once`, so the DB query executes at most once regardless of concurrency:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: if genesis initialization can fail and retry is needed, `sync.Once` must be reset (use a mutex + bool flag instead) so that a later request can retry after the DB becomes available, but still serializing concurrent attempts. [5](#0-4) 

### Proof of Concept
1. Deploy the Rosetta service against a mirror node that is still syncing (genesis data not yet in `account_balance`).
2. Send 50 concurrent HTTP GET requests to `/block` (or any block endpoint):
   ```bash
   for i in $(seq 1 50); do
     curl -s -X POST http://<rosetta-host>/block \
       -H 'Content-Type: application/json' \
       -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":0}}' &
   done
   wait
   ```
3. Observe in the DB server metrics (pg_stat_activity, CPU, I/O) that 50 concurrent `selectGenesis` CTE queries execute simultaneously.
4. Repeat continuously. Because `once.Do` is never reached (query fails), every subsequent batch of requests triggers the same amplification with no bound.
5. Confirm connection pool slots are exhausted when concurrency exceeds `MaxOpenConnections`, causing legitimate requests to fail. [6](#0-5)
