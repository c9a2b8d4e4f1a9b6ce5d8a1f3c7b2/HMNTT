### Title
Unauthenticated `GET /blocks` Endpoint Lacks Application-Level Rate Limiting, Enabling DoS via Request Flood

### Summary
The `GET /blocks` route in the Node.js REST API has no application-level rate limiting or per-IP throttling. Any unauthenticated external user can send an unbounded volume of requests, exhausting the PostgreSQL connection pool and saturating the Node.js event loop, causing legitimate requests to be dropped or time out. The only throttling present in the codebase applies exclusively to the separate `web3` Java service, not to this REST API.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` registers the route with no middleware guards: [1](#0-0) 

`rest/server.js` defines the full middleware stack applied globally. Inspecting every `app.use` / `app.useExt` call in order: [2](#0-1) 

The stack is: `urlencoded → json → cors → compression → httpContext → requestLogger → authHandler → metricsHandler (optional) → responseCacheCheckHandler (optional) → routes`. **No rate-limiting middleware appears anywhere in this chain.**

`rest/middleware/authHandler.js` is the only access-control middleware, but it is entirely opt-in: if no `Authorization` header is sent, the handler returns immediately without blocking the request: [3](#0-2) 

`rest/controllers/blockController.js` `getBlocks()` issues a live database query on every invocation with no guard: [4](#0-3) 

A grep across all `rest/**/*.js` files for `rateLimit`, `throttle`, `express-rate`, or `helmet` returns zero matches in production code. The throttling infrastructure that does exist (`ThrottleManagerImpl`, `ThrottleConfiguration`, bucket4j) lives entirely in the `web3` Java module and is never applied to the Node.js REST service: [5](#0-4) 

The Traefik `rateLimit` middleware visible in the Helm values is scoped to the Rosetta service, not the REST API: [6](#0-5) 

**Root cause:** The design assumes an external reverse proxy (e.g., Traefik, nginx) will enforce rate limits before traffic reaches the Node.js process. No such enforcement is implemented at the application layer, and no equivalent Helm chart configuration for the REST API was found. The assumption fails for any deployment that exposes the Node.js port directly, or where the proxy is misconfigured.

### Impact Explanation
Each `GET /blocks` request triggers a synchronous PostgreSQL query through the shared connection pool. A flood of concurrent requests will: (1) exhaust the pg connection pool, causing all subsequent queries—including those from legitimate users—to queue or fail; (2) accumulate pending async callbacks, increasing Node.js event loop lag; (3) cause health-check endpoints (`/health/readiness`, `/health/liveness`) to time out, triggering Kubernetes pod restarts and creating a cascading availability failure. The response cache (`responseCacheCheckHandler`) is conditional on both `config.cache.response.enabled` and `config.redis.enabled` being true, so it provides no protection in default or Redis-less deployments. [7](#0-6) 

### Likelihood Explanation
The precondition is zero: no account, API key, or network privilege is required. The endpoint is a standard REST GET, discoverable from the public OpenAPI spec served by the same process. A single attacker with a modest botnet or even a single machine running async HTTP clients (e.g., `wrk`, `hey`, `ab`) can sustain thousands of requests per second. The attack is repeatable, stateless, and requires no knowledge of internal state.

### Recommendation
1. **Application-layer rate limiting:** Add `express-rate-limit` (or equivalent) as a global middleware in `rest/server.js` before the route definitions, keyed on `req.ip` (already trusted via `app.set('trust proxy', true)`).
2. **Per-IP connection limiting:** Enforce a maximum number of in-flight requests per source IP.
3. **DB query timeout:** Set a statement timeout on the PostgreSQL connection pool so runaway queries do not hold connections indefinitely.
4. **Infrastructure hardening:** Document and enforce that the REST API port must not be exposed without a rate-limiting reverse proxy, and add a corresponding Helm middleware configuration analogous to the one already present for the Rosetta service.
