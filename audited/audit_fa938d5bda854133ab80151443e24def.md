### Title
Unbounded Inbound Message Size on `getNodes()` Enables Memory Exhaustion via Padded `AddressBookQuery` Unknown Fields

### Summary
The `getNodes()` handler in `NetworkController.java` accepts `AddressBookQuery` messages from any unauthenticated caller with no explicit inbound message size cap configured in the application. The `GrpcConfiguration` bean only sets `executor` and `maxConcurrentCallsPerConnection` on the `NettyServerBuilder`, omitting `maxInboundMessageSize`. An attacker can pad `AddressBookQuery` with unknown protobuf fields up to the gRPC-Java framework default of 4 MB per message, and by opening many connections (no connection-count limit exists) each carrying 5 concurrent calls, force the server to allocate tens of gigabytes of heap for deserialization, exhausting memory.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java` lines 28–34 configure the Netty server builder:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← no serverBuilder.maxInboundMessageSize(...) call
};
``` [1](#0-0) 

`NettyProperties.java` exposes only `maxConcurrentCallsPerConnection`; there is no `maxInboundMessageSize` property: [2](#0-1) 

No `application.yml` exists under `grpc/src/main/resources/` (only `banner.txt`), so no Spring-level override sets this value either.

The `AddressBookQuery` proto defines only two tiny fields (`file_id` and `limit`), making a legitimate message ~20 bytes: [3](#0-2) 

**Root cause:** The application relies entirely on the gRPC-Java framework default of 4 MB (`GrpcUtil.DEFAULT_MAX_MESSAGE_SIZE = 4 * 1024 * 1024`). No application-level constraint tightens this to the ~20-byte legitimate size of `AddressBookQuery`. Proto3 preserves unknown fields in `UnknownFieldSet` during deserialization, so an attacker can craft a message with arbitrary high-numbered unknown fields (wire type 2, length-delimited) that fills the 4 MB window.

**Exploit flow:**
1. Attacker opens *N* TCP connections to port 5600 (no connection-count limit enforced).
2. On each connection, sends 5 concurrent `getNodes` RPCs (the per-connection cap).
3. Each RPC carries a `AddressBookQuery` padded to ~4 MB with unknown protobuf fields.
4. gRPC-Java's `MessageDeframer` passes the full 4 MB byte array to protobuf; protobuf allocates heap for the message and stores unknown fields in `UnknownFieldSet`.
5. Total heap pressure: *N* × 5 × 4 MB. With *N* = 500 connections, that is ~10 GB of concurrent allocation.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits per-connection parallelism but places no cap on the number of connections or on message size. [4](#0-3) 
- No IP-based rate limiting, no authentication, and no connection-count limit are present anywhere in the gRPC configuration.
- The 4 MB framework default is 200,000× larger than the legitimate `AddressBookQuery` payload, providing no meaningful protection.

### Impact Explanation
Each mirror node JVM can be driven into `OutOfMemoryError` by a single attacker with enough open connections. Because mirror nodes are typically deployed identically (same configuration, same JVM heap), the same script targets all replicas simultaneously. A coordinated flood across all mirror node instances would degrade or crash the address-book query service across the entire deployment, satisfying the ≥30% node-impact threshold. The service is publicly accessible with no authentication, making this a zero-credential attack.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. Crafting a padded protobuf message requires only a standard protobuf library (one line of code to append a large `bytes` unknown field). Opening hundreds of TCP connections is trivial from a single host or a small botnet. The attack is fully repeatable and requires no prior knowledge of the system beyond the public gRPC service address and the published proto schema.

### Recommendation
1. **Set an explicit, tight `maxInboundMessageSize`** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.maxInboundMessageSize(1024); // 1 KB is ample for AddressBookQuery
   ``` [5](#0-4) 
2. **Expose the limit as a configurable property** in `NettyProperties` so it can be tuned per environment without code changes. [2](#0-1) 
3. **Add a connection-count limit** via `NettyServerBuilder.maxConnectionsTotal()` or an upstream proxy rule.
4. **Add IP-based rate limiting** at the nginx proxy layer (already present in `docker-compose.yml`) using `limit_req_zone` for the gRPC location block.
