### Title
Unbounded Per-Subscriber DB Query Amplification in `PollingTopicListener.listen()` (POLL Mode)

### Summary
When the gRPC listener is configured with `type: POLL`, each call to `PollingTopicListener.listen()` creates a fully independent polling loop that issues a dedicated database query every `interval` (default 500 ms). Because there is no per-IP connection limit, no global subscription cap, and no authentication requirement, an unauthenticated attacker can open arbitrarily many TCP connections (each carrying up to `maxConcurrentCallsPerConnection=5` streams) and flood the database with O(N) identical queries per interval, causing a thundering-herd denial-of-service.

### Finding Description

**Code path — `PollingTopicListener.listen()` lines 34–49:**

```
grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java
``` [1](#0-0) 

Every invocation allocates a fresh `PollingContext` and wires up an independent `Flux.defer(() -> poll(context))` that fires `topicMessageRepository.findByFilter(newFilter)` on every tick. There is no shared state, no deduplication, and no fanout — N active subscriptions produce exactly N concurrent DB queries per interval. [2](#0-1) 

**Root cause — no per-subscriber or per-IP guard:**

The only server-side throttle is `maxConcurrentCallsPerConnection = 5`, which is a *per-TCP-connection* limit enforced by Netty. [3](#0-2) [4](#0-3) 

There is no limit on the *number of TCP connections* from a single IP, no global subscription count ceiling, and no authentication on the `subscribeTopic` RPC. [5](#0-4) 

**Contrast with safe alternatives:** `SharedPollingTopicListener` uses `.share()` so all subscribers fan out from a single DB poll regardless of subscriber count. [6](#0-5) 

`PollingTopicListener` has no equivalent sharing mechanism.

**Exploit flow:**

1. Attacker opens `C` TCP connections to port 5600 (no auth required).
2. On each connection, attacker opens 5 `subscribeTopic` streams to the same `topicId` (the per-connection cap).
3. Each stream triggers an independent `PollingTopicListener.listen()` call → independent `poll()` → independent `findByFilter()` SQL query every 500 ms.
4. Total DB queries per second = `C × 5 × 2` (2 polls/s at default 500 ms interval).
5. With `C = 200` connections: **2 000 identical DB queries/second** from a single attacker IP.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` caps streams per connection, not total connections.
- `TopicMessageServiceImpl.subscriberCount` is a metric gauge only — it is never checked against a limit. [7](#0-6) 

- No IP-based throttle, no token bucket, no circuit breaker exists anywhere in the call path.

### Impact Explanation

Under `POLL` mode, the database receives O(N_connections × 5) identical `SELECT … FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` queries every 500 ms. Each query acquires a connection from the HikariCP pool. Pool exhaustion (monitored at 75% utilization in the Helm alerts) causes all other gRPC and REST API calls to queue or fail, resulting in a full denial of service for legitimate users. The attack is amplified because all queries are identical and hit the same index range, maximising lock/buffer contention. [8](#0-7) 

### Likelihood Explanation

- **Precondition:** operator has set `hiero.mirror.grpc.listener.type: POLL` (a documented, supported option for Redis-free deployments).
- **Attacker capability:** zero — no credentials, no special knowledge beyond the public gRPC endpoint and topic ID (both publicly documented).
- **Repeatability:** fully scriptable with `grpcurl` or any gRPC client in a loop; reconnection after server-side teardown is trivial.
- **Detection difficulty:** high — each individual connection looks legitimate; the attack is purely volumetric at the subscription layer.

### Recommendation

1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
2. **Add per-IP connection/stream rate limiting** at the Netty layer (e.g., via a `ServerInterceptor` that tracks remote address → active stream count).
3. **Prefer `SHARED_POLL` or `REDIS`** in production and document that `POLL` mode is unsafe for public-facing deployments.
4. **Introduce a `maxConcurrentConnections` property** in `NettyProperties` and apply it via `NettyServerBuilder.maxConnectionAge` / a custom connection limiter. [9](#0-8) 

### Proof of Concept

```bash
# Open 200 parallel subscribeTopic streams to the same topic (40 connections × 5 streams each)
# Requires: grpcurl, bash, a running mirror-node with type: POLL

TOPIC_NUM=41110
HOST=localhost:5600

for i in $(seq 1 40); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d "{\"topicID\": {\"topicNum\": $TOPIC_NUM}, \"limit\": 0}" \
      $HOST com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Each background process holds an open stream.
# The mirror-node DB will now receive 200 × 2 = 400 findByFilter queries/second.
# Monitor with: watch -n1 "psql -c \"SELECT count(*) FROM pg_stat_activity WHERE query LIKE '%topic_message%'\""
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
