### Title
Unbounded Timestamp Upper Limit in Hook Storage Historical Query Enables Full Table Scan DoS

### Summary
An unprivileged user can supply `timestamp=gte:0` (a lower-bound-only timestamp filter) to the `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint. Because `Bound.adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is present, the resulting SQL query scans every historical record in `hook_storage_change` with `consensus_timestamp BETWEEN 0 AND 9223372036854775807`. This can be repeated by any unauthenticated caller to exhaust database CPU and I/O resources.

### Finding Description

**Step 1 — Controller accepts lower-bound-only timestamp:**

In `HooksController.java`, the `timestamps` parameter has only a `@Size(max = 2)` constraint — no requirement for an upper bound: [1](#0-0) 

`hookStorageChangeRequest()` passes the raw timestamps directly to `Bound.of()`: [2](#0-1) 

**Step 2 — `Bound.of()` creates a Bound with only a lower value:**

`Bound.of()` iterates the parameters and sets `lower` for `gte:0`, leaving `upper = null`: [3](#0-2) 

**Step 3 — `isHistorical()` returns `true`, routing to the historical query path:**

Because `lower != null`, `isEmpty()` is `false`, so `isHistorical()` returns `true`: [4](#0-3) 

**Step 4 — `adjustUpperBound()` returns `Long.MAX_VALUE`:**

With `upper == null`, the method unconditionally returns `Long.MAX_VALUE`: [5](#0-4) 

**Step 5 — `getHookStorageChange()` passes `Long.MAX_VALUE` to the repository:** [6](#0-5) 

**Step 6 — Repository executes an unbounded `BETWEEN 0 AND 9223372036854775807` scan:**

The native query uses `DISTINCT ON (key)` with `consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound`. With bounds `0` and `Long.MAX_VALUE`, PostgreSQL must scan every partition of `hook_storage_change` for the given `owner_id`/`hook_id` to resolve `DISTINCT ON`: [7](#0-6) 

The table is range-partitioned on `consensus_timestamp`, so this query crosses all existing partitions: [8](#0-7) 

**Why existing checks fail:**
- `@Size(max = 2)` only limits the number of timestamp parameters, not their content or the absence of an upper bound.
- `@Max(MAX_LIMIT)` on `limit` caps result rows returned to the caller, but does **not** limit the number of rows the database must scan to compute `DISTINCT ON (key)`.
- No authentication or rate limiting is present on the endpoint.

### Impact Explanation
Each request with `timestamp=gte:0` forces a full cross-partition scan of `hook_storage_change` for the targeted `owner_id`/`hook_id`. Because `DISTINCT ON (key)` requires evaluating all matching rows before applying `LIMIT`, the database work is proportional to the total number of historical storage change records — not the page size. Repeated requests from a single unauthenticated attacker can drive sustained high CPU and I/O on the database node, degrading service for all users. This directly satisfies the "30% resource increase without brute force" threshold since a single well-timed request against a large dataset can spike DB load significantly.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The exploit requires only a standard HTTP GET request with a single query parameter. The attacker needs only a valid `ownerId`/`hookId` pair (discoverable via the `GET /api/v1/accounts/{ownerId}/hooks` endpoint). The attack is trivially repeatable and scriptable.

### Recommendation
1. **Require an upper timestamp bound** when a lower bound is supplied for historical queries. In `hookStorageChangeRequest()`, validate that if `timestamp` has a lower bound, an upper bound must also be present, or default the upper bound to `now()` (current consensus timestamp).
2. **Cap the maximum timestamp range** — e.g., reject requests where `upper - lower > MAX_ALLOWED_RANGE`.
3. **Add a default upper bound** in `Bound.adjustUpperBound()` for the historical query path: instead of returning `Long.MAX_VALUE`, return the current system time in nanoseconds when `upper == null`.
4. **Add rate limiting** at the API gateway or Spring layer for this endpoint.

### Proof of Concept

```
# Step 1: Discover a valid hookId for a known account
GET /api/v1/accounts/0.0.1234/hooks

# Step 2: Trigger unbounded historical scan (no auth required)
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100

# Result:
# - Bound.of() sets lower=GTE:0, upper=null
# - isHistorical() = true
# - adjustUpperBound() = Long.MAX_VALUE (9223372036854775807)
# - getAdjustedLowerRangeValue() = 0
# - SQL: SELECT DISTINCT ON (key) ... FROM hook_storage_change
#        WHERE owner_id=? AND hook_id=?
#          AND key BETWEEN '\x00...' AND '\xff...'
#          AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
# - Full cross-partition scan executed on every request
# Repeat in a loop to sustain elevated DB resource consumption
```
