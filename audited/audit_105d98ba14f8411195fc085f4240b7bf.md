### Title
Unauthenticated Thundering Herd via Uncached, Unthrottled `GET /api/v1/topics/{id}` Exhausts Database Connection Pool

### Summary
`TopicRepository` carries no `@Cacheable` annotation and the `rest-java` service has no rate-limiting filter for the `/api/v1/topics/{id}` endpoint. Every concurrent HTTP request causes three synchronous database round-trips (topic, entity, custom-fee), allowing any unprivileged client to saturate the HikariCP connection pool and degrade or deny database access for all other operations.

### Finding Description

**Code path:**

`TopicController.getTopic()` (line 32-36) calls three uncached repository lookups per request:

```
topicService.findById(id.id())      → TopicServiceImpl → topicRepository.findById(id.getId())
entityService.findById(id.id())     → EntityServiceImpl → entityRepository.findById(id.getId())
customFeeService.findById(id.id())  → CustomFeeServiceImpl → customFeeRepository.findById(...)
```

`TopicRepository` is a bare `CrudRepository<Topic, Long>` with no `@Cacheable` or any other caching decoration:

```java
public interface TopicRepository extends CrudRepository<Topic, Long> {}
```

`TopicServiceImpl.findById()` (line 19-21) passes directly through to the repository with no intermediate cache:

```java
public Topic findById(EntityId id) {
    return topicRepository.findById(id.getId())
            .orElseThrow(() -> new EntityNotFoundException("Topic not found"));
}
```

**Existing mitigations reviewed and found insufficient:**

- `LoggingFilter` — logs requests only; no rate limiting.
- `ShallowEtagHeaderFilter` — adds ETag support; no rate limiting.
- `RestJavaConfiguration` — registers ETag filter and protobuf converter; no rate limiting.
- `WebMvcConfiguration` — registers argument resolvers; no rate limiting.
- `Cache-Control: public, max-age=5` response header (confirmed in test line 94) — a client/CDN hint only; the application itself performs a full DB round-trip on every request regardless.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` with Bucket4j rate limiting exists **only in the `web3` module** for contract-call endpoints; it is entirely absent from `rest-java`.

**Root cause:** The failed assumption is that a short `Cache-Control` TTL on the HTTP response is sufficient to protect the database. It is not — it only helps downstream caches (CDN, browser). The application layer has no request coalescing, no server-side cache, and no per-IP or global rate limit on this endpoint.

### Impact Explanation

Each call to `GET /api/v1/topics/{id}` issues three `SELECT` statements against the database. With N concurrent requests, N×3 database connections are consumed simultaneously from the HikariCP pool. Once the pool is exhausted, all other database-dependent operations (importer writes, other REST endpoints, health checks) queue or time out. Because the primary-key lookup is fast, the attacker can sustain a high request rate with modest resources, keeping the pool continuously saturated. This constitutes a griefing-class denial-of-service with no economic cost to the attacker and no authentication requirement.

### Likelihood Explanation

The endpoint is publicly documented in the OpenAPI spec (`/api/v1/topics/{topicId}`), requires no credentials, and accepts a simple numeric path parameter. A single machine with a standard HTTP benchmarking tool (e.g., `wrk`, `ab`, `hey`) is sufficient to trigger the condition. The attack is repeatable indefinitely and requires no special knowledge beyond a valid topic ID, which is itself publicly discoverable.

### Recommendation

1. **Server-side caching:** Add `@Cacheable` to `TopicRepository.findById()` (and the corresponding entity/custom-fee repositories) using a short TTL (e.g., 5 s, matching the existing `Cache-Control` value) backed by Caffeine, consistent with the pattern already used in the `grpc` module's `EntityRepository`.
2. **Rate limiting:** Introduce a Bucket4j-based rate-limiting filter in `rest-java` (mirroring `web3`'s `ThrottleConfiguration`) applied globally or per-IP to all `/api/v1/**` endpoints.
3. **Connection pool sizing + statement timeout:** Ensure HikariCP `maximumPoolSize` and a `statement_timeout` are explicitly configured so a flood cannot hold connections indefinitely.

### Proof of Concept

```bash
# Precondition: obtain any valid topic ID (e.g., 0.0.1234) from the public network
TOPIC_ID="0.0.1234"
BASE_URL="https://<mirror-node-host>"

# Trigger: send 500 concurrent requests for the same topic ID
hey -n 10000 -c 500 "${BASE_URL}/api/v1/topics/${TOPIC_ID}"

# Result: HikariCP pool saturated; concurrent requests to other endpoints
