### Title
Thread Pool Exhaustion via Synchronous Blocking JDBC in `getByAliasAndType()` During Network Partition

### Summary
`EntityServiceImpl.getByAliasAndType()` issues a synchronous blocking JDBC call (`entityRepository.findByAlias()`) with no application-level socket timeout. The only timeout defense is a PostgreSQL server-side `statement_timeout` of 10 seconds, which is undeliverable to the JDBC client during a network partition because the TCP socket is broken. An unauthenticated attacker who floods the publicly accessible GraphQL endpoint with alias-based queries during such a partition will exhaust the servlet thread pool, rendering the entire service unresponsive.

### Finding Description

**Exact code path:**

`AccountController.account()` → `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)` → `EntityServiceImpl.getByAliasAndType()` line 30 → `entityRepository.findByAlias(decodeBase32(alias))` — a synchronous Spring Data JPA call executing the native query:

```sql
select * from entity where alias = ?1 and deleted is not true
``` [1](#0-0) [2](#0-1) 

**Root cause — failed assumption:**

The design assumes that the PostgreSQL-level `statement_timeout = 10000` (set via `ALTER USER mirror_graphql SET statement_timeout TO '10000'`) will bound the blocking duration of every query. [3](#0-2) 

This assumption fails during a network partition. When the TCP path between the GraphQL pod and the PostgreSQL server is severed:
- PostgreSQL fires `statement_timeout` server-side and cancels the query after 10 s.
- The cancellation notification is a TCP packet that never arrives at the JDBC client.
- The JDBC driver (HikariCP + PostgreSQL JDBC) continues blocking on `socket.read()` indefinitely — until the OS TCP keepalive fires (Linux default: ~2 hours) or the connection is forcibly closed.
- No application-level `socketTimeout` or `loginTimeout` is configured on the JDBC URL or HikariCP pool for the GraphQL module. [4](#0-3) 

**Exploit flow:**

1. Network partition between GraphQL pods and PostgreSQL occurs (infrastructure event, not attacker-controlled).
2. Attacker sends a stream of concurrent unauthenticated POST requests to `/graphql/alpha`:
   ```json
   {"query":"{account(input:{alias:\"AAAAAAAAAAAAAAAA\"}){balance}}"}
   ```
3. Each request reaches `getByAliasAndType()`, acquires a HikariCP connection, and blocks on `findByAlias()`.
4. Once all HikariCP connections are held (default pool size 10), subsequent requests block waiting for a connection (up to HikariCP `connectionTimeout`, default 30 s), holding their servlet threads.
5. Spring Boot Tomcat's default thread pool (200 threads) is exhausted.
6. All subsequent requests queue and eventually time out at the HTTP layer; the service is fully unresponsive.

**Why existing checks are insufficient:**

| Defense | Why it fails |
|---|---|
| `statement_timeout = 10000` at PostgreSQL | Server-side only; TCP packet never delivered during partition |
| `MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)` | Fires before execution; does not bound blocking I/O duration |
| JSON/GraphQL parser limits | Fires before execution; irrelevant to thread hold time |
| No rate limiting | The GraphQL module has no `ThrottleConfiguration` or equivalent (unlike `web3`) |
| No authentication | Endpoint is publicly accessible | [5](#0-4) 

### Impact Explanation

Complete denial of service for the GraphQL API. All in-flight and queued requests fail once the 200-thread Tomcat pool is saturated. The service cannot recover until the network partition resolves and blocked threads are released. Because the GraphQL service is documented as publicly accessible with no authentication, any internet-reachable deployment is exposed. The same `findByEvmAddress()` path in `getByEvmAddressAndType()` is identically affected.

### Likelihood Explanation

**Precondition:** A network partition between the GraphQL pod and PostgreSQL must exist. This is an infrastructure event (not directly attacker-controlled), but it occurs naturally during rolling restarts, cloud provider incidents, misconfigured network policies, or deliberate network-layer interference by a privileged network attacker. The attacker does not need to cause the partition — they only need to detect it (e.g., by observing increased latency on alias queries) and then flood the endpoint. No credentials are required. The attack is trivially scriptable with `curl` or any HTTP load tool.

### Recommendation

1. **Add a JDBC socket timeout** on the HikariCP data source for the GraphQL module: set `socketTimeout` (in seconds) on the JDBC URL or via `spring.datasource.hikari.connection-init-sql` / `dataSourceProperties`. A value of 15 s (slightly above `statement_timeout`) ensures the JDBC client unblocks even when the TCP path is dead.
2. **Add application-level rate limiting** to the GraphQL service, mirroring the `ThrottleConfiguration` / bucket4j pattern used in the `web3` module.
3. **Configure HikariCP `connectionTimeout`** explicitly and keep it short (e.g., 5 s) so threads waiting for a pool connection fail fast rather than holding for 30 s.
4. **Consider async/reactive execution** for the `findByAlias` path using Spring Data R2DBC or wrapping the blocking call in a bounded `@Async` executor with a `Future.get(timeout)` guard.

### Proof of Concept

```bash
# Step 1: Confirm the partition scenario (or simulate with iptables on the DB host)
# iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# Step 2: Flood the GraphQL alias endpoint from an unprivileged client
for i in $(seq 1 250); do
  curl -s -o /dev/null \
    -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{alias:\"AAAAAAAAAAAAAAAA\"}){balance}}"}' &
done
wait

# Step 3: Verify service unresponsiveness
curl -v -X POST http://<graphql-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'
# Expected: connection timeout / no response — thread pool exhausted
```
