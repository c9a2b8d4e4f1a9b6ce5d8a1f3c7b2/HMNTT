### Title
Unauthenticated EVM Address Lookup Causes Unbounded DB Query Flood (Non-Network DoS)

### Summary
`getEntityIdFromEvmAddress()` in `rest/service/entityService.js` executes a database query on every invocation with no result caching at the service layer. The REST API has no HTTP-level rate limiting for unauthenticated users. An attacker can flood any endpoint that resolves EVM addresses (accounts, tokens, contracts, etc.) with unique valid addresses, exhausting the database connection pool and degrading service availability.

### Finding Description

**Exact code path:**

`rest/service/entityService.js` line 91:
```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
``` [1](#0-0) 

`BaseService.getRows()` directly calls the DB pool with no caching:
```js
async getRows(query, params) {
  return (await this.pool().queryQuietly(query, params)).rows;
}
``` [2](#0-1) 

**Root cause — failed assumption:** The `parseCached` LRU cache in `rest/entityId.js` (lines 301–333) only caches the *parsed* `EntityId` object (string → shard/realm/num/evmAddress struct). It does **not** cache the DB lookup result. [3](#0-2) 

In `getEncodedId()`, after `EntityId.parseString()` (which is cached), `getEntityIdFromEvmAddress()` is called unconditionally — even for repeated identical addresses, a fresh DB query fires every time: [4](#0-3) 

**No HTTP rate limiting:** A grep across `rest/**` for `rateLimit`, `rateLimiter`, and `express-rate` returns only a single test-file hit. The `authHandler` middleware only sets custom response *limits* for authenticated users; it does not throttle or reject unauthenticated requests: [5](#0-4) 

**Many exposed endpoints:** `getEncodedId()` is called from `accounts.js`, `tokens.js`, `transactions.js`, `topicmessage.js`, `schedules.js`, `balances.js`, and multiple controllers — all reachable without authentication.

**`evmAddress` stored without `0x`:** `parseFromString()` strips the `0x` prefix and stores the raw 40-char hex string as `evmAddress`. `Buffer.from(entityId.evmAddress, 'hex')` therefore always succeeds for any valid 40-char hex input, including `0x`-prefixed inputs accepted by `evmAddressRegex`: [6](#0-5) 

### Impact Explanation
Each unique EVM address bypasses the EntityId parse cache and triggers a `SELECT id FROM entity WHERE evm_address = $1` query. At high request rates, this exhausts the PostgreSQL connection pool shared across all mirror node services, causing query timeouts and HTTP 500 errors for all legitimate users. Because the mirror node is read-only infrastructure for the Hedera network, degrading it prevents dApps, wallets, and explorers from resolving account/contract state — a non-network DoS against a critical piece of network infrastructure.

### Likelihood Explanation
No privileges, API keys, or special network position are required. Any internet-accessible mirror node deployment is vulnerable. Generating valid 40-character hex strings is trivial. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) sending requests to `/api/v1/accounts/0x<random>` at a few hundred RPS can saturate a typical connection pool (default pool sizes are usually 10–20 connections). The attack is repeatable and stateless.

### Recommendation
1. **Cache DB results in `getEntityIdFromEvmAddress()`**: Add a bounded TTL cache (e.g., `quick-lru`) keyed on the hex EVM address string, storing the resolved entity ID (or a sentinel for "not found"). A short TTL (e.g., 30–60 s) is sufficient.
2. **Add HTTP-level rate limiting**: Integrate `express-rate-limit` (or equivalent) as middleware in `rest/server.js`, applying a per-IP request rate cap to all routes, especially those that resolve entity IDs.
3. **Extend `parseCached` scope**: Consider caching the full `getEncodedId()` result (not just the EntityId parse) for EVM address inputs, so repeated lookups of the same address never reach the DB.

### Proof of Concept
```bash
# Generate 10,000 unique random EVM addresses and flood the accounts endpoint
python3 -c "
import os, requests, threading
def flood(i):
    addr = '0x' + os.urandom(20).hex()
    requests.get(f'http://<mirror-node>/api/v1/accounts/{addr}', timeout=5)
threads = [threading.Thread(target=flood, args=(i,)) for i in range(10000)]
[t.start() for t in threads]
[t.join() for t in threads]
"
# Expected result: DB connection pool exhausted; legitimate requests begin timing out
# Observable: HTTP 500 / 503 responses; DB logs show connection wait queue growing
```

const parseCached = (id, allowEvmAddress, evmAddressType, error) => {
  const key = `${id}_${allowEvmAddress}_${evmAddressType}`;
  const value = cache.get(key);

  if (evmAddress === null && (num > maxNum || realm > maxRealm || shard > maxShard)) {
