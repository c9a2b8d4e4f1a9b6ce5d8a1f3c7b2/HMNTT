### Title
Unbounded Throttled Subscription Polling Exhausts DB Connection Pool via Unauthenticated gRPC Flood

### Summary
The `PollingTopicMessageRetriever.retrieve()` method, when invoked in throttled mode, sets `numRepeats = Long.MAX_VALUE` with a fixed 2-second polling interval and issues a live DB query on every tick. Because the gRPC server enforces only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) with no per-IP connection cap, no global subscription ceiling, and no authentication requirement, an unprivileged attacker can open an arbitrary number of TCP connections — each carrying 5 concurrent subscriptions against a topic with a large message backlog — generating a sustained, unbounded flood of DB queries that saturates the gRPC module's connection pool and degrades or halts the mirror node's ability to serve and ingest transaction data.

### Finding Description

**Exact code path:**

In `PollingTopicMessageRetriever.java`, `retrieve()` (lines 45–63), when `throttled = true`, the `PollingContext` constructor (lines 94–108) unconditionally sets:

```java
numRepeats = Long.MAX_VALUE;                          // line 99
frequency = retrieverProperties.getPollingFrequency(); // line 100 — default 2s
```

The `repeatWhen` operator (lines 52–55) re-invokes `poll()` every 2 seconds as long as `!context.isComplete()`. Each `poll()` call (lines 65–79) executes:

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter)); // line 78
```

— a live database query on every tick.

**Root cause — `isComplete()` for throttled mode (lines 121–128):**

```java
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
```

`pageSize` is reset to 0 at the start of each poll (line 73) and incremented per message received. If the topic has a backlog where each page fills the full `maxPageSize` (default 1000), `isComplete()` returns `false` indefinitely, keeping `numRepeats = Long.MAX_VALUE` active and issuing a DB query every 2 seconds per subscription, forever.

**Why existing checks fail:**

1. **`maxConcurrentCallsPerConnection = 5`** (`NettyProperties.java` line 14, applied in `GrpcConfiguration.java` line 33): This is a *per-connection* limit only. There is no cap on the number of inbound TCP connections from a single IP. An attacker opens N connections × 5 calls = 5N concurrent subscriptions.

2. **`subscriberCount`** (`TopicMessageServiceImpl.java` lines 48, 89–90): This is a Micrometer gauge for observability only — it has zero enforcement logic. No maximum is checked or rejected.

3. **`timeout = 60s`** (`RetrieverProperties.java` line 28): This is the idle-emission timeout (time between messages), not a hard session wall-clock timeout. A topic with a continuous backlog keeps emitting, so the timeout never fires.

4. **`retryWhen(Retry.backoff(Long.MAX_VALUE, ...))`** (line 58): Retries DB errors indefinitely, meaning transient pool exhaustion causes retries that compound the load rather than shedding it.

5. **No authentication, no per-IP rate limiting**: The gRPC module has no `ThrottleConfiguration` (that exists only in the `web3` module). The `subscribeTopic` RPC is open to any unauthenticated caller.

### Impact Explanation

With N connections (no limit enforced), an attacker sustains `5N` concurrent DB queries every 2 seconds against the gRPC module's HikariCP connection pool. Once the pool is saturated, all subsequent queries queue or time out. This degrades or halts the mirror node's gRPC service (topic subscriptions fail for all legitimate users). Because the gRPC module and the importer share the same PostgreSQL instance, exhausting PostgreSQL's `max_connections` at the server level can also starve the importer's write path, preventing new transaction records from being persisted and served. The mirror node becomes unable to confirm or relay new transaction data to downstream consumers.

### Likelihood Explanation

The attack requires no credentials, no special tooling beyond a standard gRPC client (e.g., `grpcurl`, any HCS SDK), and no privileged network position. The only meaningful precondition is a topic with a message backlog large enough that each poll returns a full page (≥1000 messages). On a production network such topics are common (high-volume HCS topics). An attacker can also control the topic themselves by pre-populating it. The attack is trivially repeatable and scriptable, and the `retryWhen(Long.MAX_VALUE)` ensures subscriptions self-heal through transient errors, making the flood self-sustaining.

### Recommendation

1. **Enforce a global (or per-IP) subscription limit**: Add an atomic counter with a configurable ceiling in `TopicMessageServiceImpl.subscribeTopic()` and reject new subscriptions with `RESOURCE_EXHAUSTED` when the limit is reached.
2. **Add per-IP connection rate limiting at the Netty layer**: Use `NettyServerBuilder.maxConnectionsTotal()` or a gRPC `ServerInterceptor` that tracks and limits connections per remote address.
3. **Add a hard wall-clock session timeout**: Supplement the idle `timeout` with an absolute maximum subscription duration so long-lived throttled pollers are eventually evicted regardless of message flow.
4. **Apply bucket4j-style rate limiting to the gRPC service**: Mirror the `ThrottleConfiguration` pattern from the `web3` module to cap the total DB query rate from the retriever across all active subscriptions.
5. **Require authentication or API keys** for `subscribeTopic` to attribute and throttle per-caller.

### Proof of Concept

```bash
# Pre-condition: topic 0.0.12345 has >= 1000 messages in its backlog

# Open 200 connections, each with 5 concurrent subscriptions = 1000 active pollers
# Each polls the DB every 2s => ~500 DB queries/second sustained

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: gRPC DB connection pool saturates; legitimate subscribers receive errors;
# mirror node importer latency increases as PostgreSQL max_connections is approached.
```
