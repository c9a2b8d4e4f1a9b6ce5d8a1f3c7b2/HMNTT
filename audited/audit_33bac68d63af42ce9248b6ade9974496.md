### Title
Identity-Check Bypass via `0x7fffffffffffffff` Block Parameter Causes `BlockNumberNotFoundException` in `findByBlockType()`

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `LATEST` sentinel, but `BlockType.of("0x7fffffffffffffff")` constructs a new `BlockType` record instance whose `number` field equals `Long.MAX_VALUE` — the same value stored in `BlockType.LATEST`. Because `==` compares object identity, not value, the new instance bypasses the `LATEST` branch and falls through to `findByIndex(Long.MAX_VALUE)`, which returns empty and causes a `BlockNumberNotFoundException`. Any unauthenticated caller of `eth_call` (or any JSON-RPC endpoint accepting a block parameter) can trigger this with a single crafted request.

### Finding Description
`BlockType` is a Java `record` declared as `public record BlockType(String name, long number)`. The singleton `BlockType.LATEST = new BlockType("latest", Long.MAX_VALUE)` is allocated once at class-load time.

When a user submits block parameter `"0x7fffffffffffffff"`, `BlockType.of()` reaches the `GROUP_HEX_NUM` branch:
```java
return new BlockType(hexNum, Long.parseLong(hexNum, 16));
// → new BlockType("7fffffffffffffff", 9223372036854775807L)
```
This is a distinct heap object. In `findByBlockType()`:
```java
if (block == BlockType.EARLIEST)   // false — different reference
else if (block == BlockType.LATEST) // false — different reference, same number
else if (block.isHash())            // false — number != -1
return recordFileRepository.findByIndex(block.number()); // findByIndex(Long.MAX_VALUE)
```
`findByIndex` runs `SELECT … WHERE r.index = 9223372036854775807`, which matches no row and returns `Optional.empty()`. The caller then throws `BlockNumberNotFoundException`.

The root cause is the use of `==` instead of value-based comparison (`.equals()` or numeric comparison) for sentinel detection on a `record` type that can be independently instantiated with identical field values.

### Impact Explanation
Any JSON-RPC consumer (e.g., `eth_call`, `eth_estimateGas`, `eth_getBalance`) that accepts a block parameter and internally calls `findByBlockType` will return an error response instead of executing against the latest block. This breaks smart contract simulation/execution for any caller that passes the numeric equivalent of `latest` as a hex block number. While no funds are directly drained, it constitutes a reliable, zero-cost denial-of-service against the "latest block" execution path, and may cause integrations that rely on `0x7fffffffffffffff` as a sentinel for "latest" to silently fail.

### Likelihood Explanation
No authentication or special privilege is required. The input `0x7fffffffffffffff` is a valid hex block number accepted by the regex and parsed without error. Any external user who knows (or discovers by fuzzing) that `Long.MAX_VALUE` is the internal sentinel for `LATEST` can reproduce this deterministically on every request. The Ethereum JSON-RPC specification does not prohibit large hex block numbers, so standard clients and fuzzers will naturally generate this value.

### Recommendation
Replace reference equality with value-based comparison in `findByBlockType()`. Since `BlockType` is a record, use `.equals()`:
```java
if (block.equals(BlockType.EARLIEST)) { … }
else if (block.equals(BlockType.LATEST)) { … }
```
Alternatively, add an explicit numeric guard before the `findByIndex` fallthrough:
```java
if (block.number() == Long.MAX_VALUE) {
    return recordFileRepository.findLatest();
}
```
Additionally, consider validating in `BlockType.of()` that a hex/decimal numeric input does not equal `Long.MAX_VALUE` (the reserved sentinel), rejecting it with `IllegalArgumentException` to prevent ambiguity at the parsing layer.
