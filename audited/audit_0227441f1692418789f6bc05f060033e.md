### Title
Unauthenticated Connection Pool Exhaustion via `GET /topics/messages/:consensusTimestamp`

### Summary
The `getMessageByConsensusTimestamp()` handler in `rest/topicmessage.js` issues two sequential `pool.queryQuietly()` calls per request against a shared `pg` pool capped at 10 connections by default. Because the REST API has no rate-limiting or concurrency-control middleware, an unprivileged attacker can flood this endpoint with concurrent requests, saturate the pool, and force all subsequent requests from any user to queue until `connectionTimeoutMillis` (20 s) expires, effectively denying database access to the entire application.

### Finding Description

**Exact code path:**

`rest/topicmessage.js` lines 60–81 — `getMessageByConsensusTimestamp`:
```js
const getMessageByConsensusTimestamp = async (req, res) => {
  utils.validateReq(req);
  const consensusTimestampParam = req.params.consensusTimestamp;
  validateConsensusTimestampParam(consensusTimestampParam);          // format-only check
  const consensusTimestamp = utils.parseTimestampParam(consensusTimestampParam);

  let query = `select consensus_timestamp, entity_id
    from transaction
    where consensus_timestamp = $1 and result = 22 and type = 27`;
  const {rows} = await pool.queryQuietly(query, consensusTimestamp); // DB call 1
  if (rows.length !== 1) { throw new NotFoundError(); }

  query = `select * from topic_message
    where topic_id = $1 and consensus_timestamp = $2`;
  const params = [rows[0].entity_id, rows[0].consensus_timestamp];
  res.locals[...] = await getMessage(query, params);                 // DB call 2
};
```

**Pool configuration** (`rest/dbpool.js` lines 13–15, defaults from `docs/configuration.md` line 556):
- `max` (maxConnections) = **10**
- `connectionTimeoutMillis` = **20 000 ms**
- `statement_timeout` = **20 000 ms**

**Root cause — failed assumption:** The design assumes that ambient infrastructure (reverse proxy, load balancer) will enforce per-client request rates before they reach the Node.js process. No such control exists in the application layer itself. `rest/server.js` (lines 1–168) registers no rate-limiting middleware for the REST API; the throttling found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate Web3 Java service.

**Why existing checks are insufficient:**
- `validateConsensusTimestampParam` only verifies the timestamp format (`utils.isValidTimestampParam`). Any syntactically valid timestamp (e.g., `1234567890.000000001`) passes.
- `pool.queryQuietly` with no `preQueryHint` calls `this.query()`, which acquires a pool connection for the full query duration. With `statement_timeout = 20 s`, each connection can be held for up to 20 seconds per query.
- The `pg` Pool has no built-in queue depth limit; excess requests queue indefinitely until `connectionTimeoutMillis` expires.

### Impact Explanation
With 10 pool connections and a 20 s statement timeout, an attacker maintaining ≥ 10 concurrent in-flight requests keeps the entire pool saturated. Every other request (from any endpoint, any user) queues for up to 20 s and then receives a `DbError` (connection timeout). This is a complete application-layer database partition: no endpoint that touches the database can succeed while the attack is sustained. Severity is **High** — full denial of service to all users with no authentication required.

### Likelihood Explanation
The attack requires only an HTTP client capable of issuing concurrent GET requests with valid timestamp strings. No account, token, or special knowledge is needed. The endpoint is publicly documented. A single attacker with a modest script (e.g., `ab`, `wrk`, or a simple async loop) can sustain the condition indefinitely at negligible cost. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add rate limiting at the application layer** — integrate `express-rate-limit` (or equivalent) in `rest/server.js` before route handlers, keyed on client IP, to cap requests per second per client.
2. **Increase `maxConnections`** or introduce a per-route concurrency semaphore so a single endpoint cannot monopolize the entire pool.
3. **Set a pool queue timeout / max queue depth** via `pg`'s `connectionTimeoutMillis` tuned to a lower value, and add a queue-depth cap to shed load early with a `503` rather than silently queuing.
4. **Consider infrastructure-level controls** (nginx `limit_req`, cloud WAF) as a defense-in-depth layer, but do not rely on them as the sole mitigation.

### Proof of Concept
```bash
# Requires: a valid timestamp format (does not need to exist in DB;
# the first query returns 0 rows quickly, but the connection is still
# acquired and held for that round-trip. Use a timestamp known to exist
# for maximum hold time.)

TIMESTAMP="1234567890.000000001"
URL="http://<mirror-node-host>/api/v1/topics/messages/${TIMESTAMP}"

# Open 15 concurrent connections (exceeds default pool of 10)
for i in $(seq 1 15); do
  curl -s "$URL" &
done
wait

# While the above runs, in a separate terminal:
curl -v "http://<mirror-node-host>/api/v1/transactions"
# Expected: hangs for up to 20 s then returns a 500 DbError,
# demonstrating that the pool is exhausted and legitimate requests are blocked.
```
