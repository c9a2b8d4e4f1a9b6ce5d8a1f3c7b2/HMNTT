### Title
Unauthenticated Unbounded DB Query via `/contracts/results/logs` Without Timestamp Filters

### Summary
The `getContractLogs()` handler at `/api/v1/contracts/results/logs` calls `extractContractLogsMultiUnionQuery()` without a `contractId`, unconditionally entering the `optimizeTimestampFilters` → `bindTimestampRange` path for every request that lacks a `transaction_hash`. When `bindTimestampRange` is disabled (the default), requests with no timestamp parameters produce SQL queries against `contract_log` with zero timestamp bounds, enabling any unprivileged user to trigger arbitrarily expensive full-table scans at will.

### Finding Description

**Code path:**

`getContractLogs` (line 820–848, `rest/controllers/contractController.js`) calls:
```js
const query = await this.extractContractLogsMultiUnionQuery(filters);  // no contractId
``` [1](#0-0) 

Inside `extractContractLogsMultiUnionQuery` (line 678), when `contractId === undefined` and no `transactionHash` is present, the optimization branch is unconditionally taken:
```js
} else if (contractId === undefined) {
  const {filters: timestampFilters, next} = await optimizeTimestampFilters(bounds.primary.getAllFilters(), order);
``` [2](#0-1) 

`optimizeTimestampFilters` (line 384–405) calls `parseTimestampFilters` with `filterRequired=false` and `allowOpenRange=true`, so an empty filter list returns `{range: null, eqValues: [], neValues: []}` without error. Since `eqValues.length === 0`, it calls `bindTimestampRange(null, order)`: [3](#0-2) 

`bindTimestampRange` (line 19–38, `rest/timestampRange.js`) immediately returns `{range}` (i.e., `{range: null}`) when `queryConfig.bindTimestampRange` is `false` (the default): [4](#0-3) 

With `optimizedRange === null`, neither `optimizedRange?.begin` nor `optimizedRange?.end` is truthy, so no timestamp filters are added. The resulting SQL query against `contract_log` carries **no timestamp WHERE clause**, causing a full-table scan.

**Why existing checks fail:**

`checkTimestampsForTopics` (line 281–305) only enforces timestamp presence when topic filters exist. A bare request with no topics and no timestamp passes through silently: [5](#0-4) 

There is no other guard that requires a timestamp or contractId for the `/contracts/results/logs` endpoint. [6](#0-5) 

### Impact Explanation
Every unauthenticated HTTP request to `GET /api/v1/contracts/results/logs` (with no `timestamp`, `topic*`, or `transaction_hash` parameters) executes an unbounded `SELECT` against the `contract_log` table. On a production network with millions of contract log entries, each such request forces a sequential scan. Flooding this endpoint with concurrent requests can saturate DB I/O and CPU, degrading or halting query processing for all mirror-node consumers — including consensus-critical monitoring tooling — without any credentials or special knowledge.

### Likelihood Explanation
The endpoint is public, documented in the OpenAPI spec, and requires zero authentication. The exploit requires a single HTTP GET with no parameters. It is trivially scriptable and repeatable. Any attacker with network access to the mirror-node REST API can execute it.

### Recommendation
1. **Require at least one timestamp filter** (or a `transaction_hash`) for the `/contracts/results/logs` endpoint when no `contractId` is present, rejecting requests that provide neither.
2. **Enable `bindTimestampRange` by default** in production configuration so that even timestamp-less requests are bounded to `maxTransactionsTimestampRangeNs`.
3. Add a pre-flight check in `getContractLogs` analogous to `checkTimestampsForTopics` that rejects requests with no timestamp and no transaction hash before entering `extractContractLogsMultiUnionQuery`.

### Proof of Concept
```
# No authentication, no parameters — triggers unbounded contract_log scan
curl -s "https://<mirror-node-host>/api/v1/contracts/results/logs"

# Flood with concurrent requests to exhaust DB resources
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results/logs" &
done
wait
```
Each request enters the `contractId === undefined` branch, calls `optimizeTimestampFilters([])`, receives `{range: null}` from `bindTimestampRange` (when disabled), and issues a `SELECT … FROM contract_log` with no timestamp predicate.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
};
