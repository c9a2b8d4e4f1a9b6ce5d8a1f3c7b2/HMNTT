### Title
Unauthenticated Timestamp Parameter Triggers Expensive UNION ALL on entity_stake_history, Enabling DB Connection Pool Exhaustion

### Summary
Any unauthenticated caller of `GET /api/v1/accounts/:id?timestamp=<value>` causes `getOneAccount()` to pass `isHistorical=true` to `getEntityStakeQuery()`, which replaces a simple `entity_stake` table reference with a correlated UNION ALL subquery across both `entity_stake` and `entity_stake_history`. There is no application-level rate limiting on this endpoint. Flooding with concurrent requests exhausts the bounded DB connection pool, degrading or halting the mirror node's ability to serve requests and ingest new network transactions.

### Finding Description
**Code path:**

`rest/accounts.js` line 396 → `getOneAccount()` collects `timestampFilters` (lines 401–412). At line 484, `getAccountQuery()` is called with `isHistorical = timestampFilters.length > 0`. Inside `getAccountQuery()` (line 295–303), this is forwarded to `getEntityBalanceQuery()`, which at line 231 calls:

```js
${getEntityStakeQuery(entityAccountQuery.query, isHistorical)}
```

`getEntityStakeQuery()` (lines 18–32):

```js
const getEntityStakeQuery = (filter, isHistorical = false) => {
  if (isHistorical) {
    return `(
      select * from (
          select * from entity_stake as e where ${filter}
                union all
          select * from entity_stake_history as e where ${filter}
      )
      as asd
      order by asd.timestamp_range desc limit 1
    )`;
  }
  return 'entity_stake';
};
```

**Root cause:** `timestamp` is explicitly listed in `acceptedSingleAccountParameters` (line 526), so it is a valid, accepted query parameter for any caller. The non-historical path returns the bare table name `entity_stake` (a simple join target). The historical path wraps both `entity_stake` and `entity_stake_history` in a correlated UNION ALL subquery with `ORDER BY … DESC LIMIT 1`. As `entity_stake_history` grows over time (it accumulates one row per staking period per entity), this subquery becomes progressively more expensive even with the index on `(id, lower(timestamp_range))`.

**Failed assumption:** The design assumes the historical path is only marginally more expensive because of the `id`-filtered index lookup. In practice, the GiST `timestamp_range` index operations, the UNION ALL materialization, and the sort step all add measurable per-query cost that compounds under concurrent load.

**No rate limiting:** `rest/dbpool.js` shows the pool is bounded by `config.db.pool.maxConnections` and `statement_timeout`, but there is no application-layer rate limiting or per-IP throttling visible in the REST layer. Once all pool connections are occupied by in-flight historical queries, new requests (including the importer's write path) queue or fail.

### Impact Explanation
The mirror node's REST API becomes unresponsive for all users. More critically, the shared PostgreSQL connection pool is also used by the importer process to write newly ingested transactions. Pool exhaustion or severe I/O contention delays or stalls transaction ingestion, meaning the mirror node falls behind the live network and can no longer confirm recent transactions to downstream consumers. This is a denial-of-service against the mirror node's core function, not the main Hedera consensus network itself.

### Likelihood Explanation
The exploit requires zero authentication, zero special knowledge beyond the public API documentation, and only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`). The `timestamp` parameter is documented as a valid filter. The attack is trivially repeatable and can be sustained indefinitely. The growing size of `entity_stake_history` over the network's lifetime makes the attack progressively cheaper for the attacker and more expensive for the server.

### Recommendation
1. **Rate-limit the `/api/v1/accounts/:id` endpoint** at the API gateway or middleware layer (e.g., per-IP, per-second limits).
2. **Add a DB-level query timeout** specifically for the historical UNION ALL path, or enforce a shorter `statement_timeout` for read-only API queries.
3. **Separate the DB connection pool** used by the REST API from the one used by the importer, so API exhaustion cannot starve the write path.
4. **Consider caching** the result of the historical entity_stake lookup (keyed by account ID + timestamp range) with a short TTL, since historical data is immutable.
5. **Require at least one non-timestamp filter** (e.g., a valid account ID that exists) before executing the historical path, to fail fast before hitting the DB.

### Proof of Concept
```bash
# Trigger the isHistorical=true branch with any valid account ID and any timestamp
# Flood with concurrent requests to exhaust the DB connection pool

wrk -t 16 -c 200 -d 60s \
  "https://<mirror-node-host>/api/v1/accounts/0.0.2?timestamp=1000000000"

# Observable result:
# - API latency spikes, then requests time out
# - DB pool maxConnections reached (visible in pool metrics)
# - Importer lag increases as write queries queue behind read queries
# - Mirror node falls behind live network head
```
