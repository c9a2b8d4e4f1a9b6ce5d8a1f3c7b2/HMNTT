### Title
Unbounded Request Body Allocation in `MempoolTransaction()` Enables Memory Exhaustion DoS

### Summary
The `MempoolTransaction()` handler in `rosetta/app/services/mempool_service.go` immediately discards its parsed `MempoolTransactionRequest` and returns `ErrNotImplemented`, but the rosetta-sdk-go controller layer has already fully read and JSON-decoded the entire request body into heap memory before the handler is invoked. Because neither the application code nor the `http.Server` configuration applies any `http.MaxBytesReader` or equivalent body-size cap, an unauthenticated attacker can send arbitrarily large POST bodies to `/mempool/transaction`, forcing the server to allocate large amounts of heap memory per request with zero useful work performed, potentially exhausting memory and disrupting legitimate block-history responses.

### Finding Description

**Exact code path:**

`rosetta/app/services/mempool_service.go`, lines 30–35:
```go
func (m *mempoolAPIService) MempoolTransaction(
    _ context.Context,
    _ *types.MempoolTransactionRequest,   // parameter is discarded
) (*types.MempoolTransactionResponse, *types.Error) {
    return nil, errors.ErrNotImplemented
}
```

The `_` blank identifier on the request parameter confirms the parsed body is never used.

**HTTP server configuration** (`rosetta/main.go`, lines 220–227):
```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // 5 s default
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```

No `http.MaxBytesReader` is applied anywhere in the rosetta codebase (confirmed by exhaustive grep across `rosetta/**/*.go`). The `Http` config struct (`rosetta/app/config/types.go`, lines 64–69) exposes only time-based limits, not size-based ones.

**Root cause:** The rosetta-sdk-go `MempoolAPIController` (external library, invoked via `server.NewMempoolAPIController` at `rosetta/main.go` line 90) calls `json.NewDecoder(r.Body).Decode(&body)` on the raw `http.Request.Body` without wrapping it in `http.MaxBytesReader`. This allocates heap memory proportional to the body size before the service handler is ever called. The handler then immediately discards the allocation. The only constraint is the time-based `ReadTimeout` (default 5 s), which at typical network speeds still permits tens to hundreds of megabytes per connection.

**Failed assumption:** The design assumes that returning `ErrNotImplemented` immediately is sufficient to avoid resource waste. It is not — the SDK controller has already consumed and allocated memory for the full body before the service function executes.

### Impact Explanation

Each request to `/mempool/transaction` with a large body forces heap allocation proportional to the body size (e.g., a 50 MB `TransactionIdentifier.Hash` string allocates ~50 MB). With multiple concurrent connections (the Go HTTP server imposes no connection limit by default), an attacker can sustain high memory pressure. Go's GC will eventually reclaim the memory, but under sustained attack the allocation rate can outpace collection, leading to OOM conditions. This disrupts all co-located services including block-history (`/block`, `/block/transaction`) responses, which share the same process heap.

Severity: **Medium–High** (availability impact, no authentication required, no funds at risk).

### Likelihood Explanation

The endpoint is publicly reachable (no authentication, no API key). The attack requires only the ability to send HTTP POST requests with large bodies — trivially achievable with `curl`, `wrk`, or any HTTP client. The Traefik `inFlightReq: amount: 5` middleware in the Kubernetes Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 152–156) limits concurrent requests to 5 per IP in K8s deployments, but:
- This middleware is optional (`{{ if and .Values.global.middleware .Values.middleware }}`).
- It is absent in direct/docker-compose deployments.
- 5 concurrent × ~50 MB/request = 250 MB of wasted allocation per round-trip cycle, repeatable continuously.

The attack is fully repeatable and requires no privileges.

### Recommendation

1. **Apply `http.MaxBytesReader` in the SDK controller or via middleware.** Add a middleware that wraps `r.Body` with `http.MaxBytesReader(w, r.Body, maxBodyBytes)` (e.g., 1 MB) before the router dispatches the request. This causes the JSON decoder to receive an `io.EOF`-equivalent error if the body exceeds the limit, and the SDK returns a 400 before any large allocation occurs.

2. **Alternatively**, add a global middleware in `rosetta/main.go` before `corsMiddleware`:
```go
func limitBody(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, 1<<20) // 1 MB
        next.ServeHTTP(w, r)
    })
}
```

3. Enforce Traefik `inFlightReq` and `rateLimit` middleware in all deployment configurations, not just Kubernetes.

### Proof of Concept

```bash
# Generate a 50 MB payload with a large hash field
python3 -c "
import json, sys
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
  'transaction_identifier': {'hash': 'A' * 50_000_000}
}
sys.stdout.write(json.dumps(payload))
" > large_body.json

# Send 5 concurrent requests (matching Traefik inFlightReq limit)
for i in $(seq 1 5); do
  curl -s -X POST http://<rosetta-host>:5700/mempool/transaction \
    -H 'Content-Type: application/json' \
    --data-binary @large_body.json &
done
wait

# Repeat in a loop to sustain memory pressure
# Monitor rosetta process RSS: watch -n1 'ps -o rss= -p $(pgrep rosetta)'
```

Each iteration forces the server to allocate ~50 MB per concurrent connection. Sustained over seconds, this drives RSS well above normal operating levels and can trigger OOM, causing block-history endpoints to fail. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
