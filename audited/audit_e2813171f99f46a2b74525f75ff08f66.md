### Title
Unauthenticated DB Connection Pool Exhaustion via EVM Address Lookup in `getOneAccount()`

### Summary
`GET /api/v1/accounts/<evm_address>` triggers at minimum three sequential/parallel DB queries per request with no rate limiting for unauthenticated callers. The default REST API connection pool is capped at 10 connections. A small number of concurrent requests (≥4) is sufficient to exhaust the pool, causing all subsequent REST API requests to queue and time out, effectively taking the REST API offline.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount()`, lines 396–512:

1. **DB query 1 (sequential)** — Line 399: `await EntityService.getEncodedId(req.params[...])` → `entityService.js` line 124: `await this.getEntityIdFromEvmAddress(entityId, requireResult)` → line 91: `await this.getRows(EntityService.entityFromEvmAddressQuery, [...])`. This issues a `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` and holds a pool connection until it resolves. **The result is never cached** — `getEntityIdFromEvmAddress()` has no memoization layer.

2. **DB query 2 (parallel)** — Line 487: `const entityPromise = pool.queryQuietly(entityQuery, entityParams)` — the full account entity query.

3. **DB query 3 (parallel)** — Lines 491–493: `transactions.doGetTransactions(filters, req, timestampRange)` → `transactions.js` line 483: `await pool.queryQuietly(query, params)` inside `getTransactionTimestamps()`.

4. **DB query 4 (conditional, parallel)** — `transactions.js` line 662: `pool.queryQuietly(query, params)` inside `getTransactionsDetails()` when timestamps are found.

Queries 2–4 are launched concurrently via `Promise.all([entityPromise, transactionsPromise])` at line 495.

**Root cause:** No rate limiting exists for unauthenticated callers on the REST API. `server.js` lines 86–102 show the middleware stack: `authHandler` only sets a custom response-size limit for credentialed users — it does not throttle or reject anonymous requests. The Traefik rate-limit middleware present in `charts/hedera-mirror-rosetta/values.yaml` is absent from the REST API chart. The `entityId` LRU cache in `entityId.js` (lines 301–304) caches only the parsed `EntityId` struct, not the DB lookup result, so every request with an EVM address path parameter hits the database.

**Pool size:** `docs/configuration.md` line 556 and `rest/dbpool.js` line 14 confirm `maxConnections` defaults to **10**.

**Math:** Each request holds 1 connection sequentially (EVM lookup) then 2–3 connections in parallel. Four concurrent requests occupy 8–12 connections, saturating or exceeding the pool. The `statementTimeout` default is 20 000 ms, meaning each connection is held for up to 20 seconds per request.

### Impact Explanation

With the pool exhausted, every other REST API endpoint (`/api/v1/transactions`, `/api/v1/balances`, etc.) blocks waiting for a free connection and eventually returns a 503/timeout. The REST API — a primary public interface of the mirror node — becomes completely unavailable for the duration of the attack. Because the REST API is a stateless Node.js process with a single shared pool, a single attacker targeting one pod takes that pod offline entirely. In a multi-pod deployment, the same attack replicated across pods (still requiring only a handful of requests per pod) takes down the entire REST API tier.

### Likelihood Explanation

No authentication, no API key, no CAPTCHA, and no IP-based rate limit is required. Any public internet client can issue four concurrent HTTP GET requests. Valid EVM addresses are publicly observable on-chain. The attack is trivially scriptable, repeatable indefinitely, and requires no special tooling. The low pool size (10) means the threshold for exhaustion is extremely low.

### Recommendation

1. **Cache the EVM-address → entity-id mapping** in `EntityService.getEntityIdFromEvmAddress()` using the existing `entityId` LRU cache (already configured with `maxAge: 1800s`, `maxSize: 100000`). This eliminates the first DB query for repeated addresses.
2. **Add rate limiting** to the REST API middleware stack (e.g., `express-rate-limit`) for unauthenticated callers, mirroring the Traefik `rateLimit` middleware already applied to the Rosetta API.
3. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer) in front of the REST API's DB user, consistent with the production Helm chart which already configures `max_user_connections: 250` for `mirror_rest` via PgBouncer — ensure this is enforced in all deployment profiles.
4. **Add a concurrency limiter** (e.g., `express-slow-down` or a semaphore) to cap in-flight DB-bound requests per pod.

### Proof of Concept

```bash
# Obtain any valid EVM address from the public mirror node (e.g., mainnet)
EVM_ADDR="0xac384c53f03855fa1b3616052f8ba32c6c2a2fec"
TARGET="https://<mirror-node-host>/api/v1/accounts/${EVM_ADDR}"

# Fire 12 concurrent requests (exceeds pool of 10 by 3x)
for i in $(seq 1 12); do
  curl -s -o /dev/null -w "%{http_code}\n" "$TARGET" &
done
wait

# Immediately probe a different endpoint — expect 503 or connection timeout
curl -v "https://<mirror-node-host>/api/v1/transactions?limit=1"
```

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
