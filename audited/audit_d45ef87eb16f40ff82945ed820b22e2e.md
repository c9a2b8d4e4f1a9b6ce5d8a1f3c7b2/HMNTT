### Title
Unauthenticated EVM Address Lookup Causes Unbounded DB Queries with No Rate Limiting, Enabling DoS Against Block Ingestion

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` calls `EntityService.getEncodedId()` for every request, which for opaque EVM addresses (0x + 40 hex chars where the first 12 bytes are not all zeros) unconditionally executes an uncached PostgreSQL query against the `entity` table. With no rate limiting on unauthenticated requests and no negative-result caching, an attacker can flood the shared database with index-miss queries at arbitrary frequency, starving the importer's DB connections and delaying block ingestion.

### Finding Description

**Exact code path:**

`GET /api/v1/accounts/{idOrAliasOrEvmAddress}/tokens` → `TokenController.getTokenRelationships()` (tokenController.js:66) → `EntityService.getEncodedId()` (entityService.js:118) → `EntityService.getEntityIdFromEvmAddress()` (entityService.js:90) → DB query.

`EntityId.parseString()` is called, which routes through `parseFromString()`: [2](#0-1) 

If the first 24 hex chars of the address are **not** the `longFormEvmAddressPrefix` (`'00'.repeat(12)`), or the numeric suffix exceeds `maxNum`, the address is classified as an opaque EVM address and `evmAddress` is set to the hex string (not null). This causes `getEntityIdFromEvmAddress()` to be called: [3](#0-2) 

The query executed is: [4](#0-3) 

**Root cause — no caching of negative results:** The `parseCached` cache in `entityId.js` only caches the parsed `EntityId` struct (shard/realm/num/evmAddress fields), not the DB lookup result: [5](#0-4) 

Every request — even with the same EVM address — re-executes the DB query. An attacker using unique addresses per request bypasses even this struct cache entirely.

**Root cause — no rate limiting:** The REST server applies no per-IP or global request-rate throttle for unauthenticated users. The `authHandler` middleware only sets a custom response-size limit for authenticated users; it does not throttle request frequency: [6](#0-5) [7](#0-6) 

**DB index exists but does not prevent the attack:** A partial index `entity__evm_address on entity (evm_address) where evm_address is not null` exists: [8](#0-7) 

Because the attacker's addresses are not in the table, every query is an index miss. Each miss is individually cheap, but at high concurrency (thousands/second) the aggregate DB CPU, I/O, and connection-pool pressure is substantial. The importer shares this PostgreSQL instance and its ingestion queries compete for the same resources.

### Impact Explanation

The mirror node's importer and REST API share one PostgreSQL database. Flooding the REST API with opaque EVM address lookups that return no rows generates a sustained stream of DB round-trips. At sufficient request rates this exhausts the DB connection pool, increases query latency for the importer's write path, and causes the mirror node to fall behind the Hedera network's block production rate. The specific 500% block-time threshold is achievable on under-provisioned deployments or with a botnet-scale flood; on well-provisioned deployments the impact is degraded ingestion latency rather than a complete stall. The endpoint is publicly accessible with no authentication required.

### Likelihood Explanation

No privileges are required. The attack requires only HTTP access to the public REST API. An attacker can generate an unlimited supply of unique opaque EVM addresses (vary any of the first 12 bytes away from zero). Standard HTTP flood tooling (e.g., `wrk`, `ab`, `hey`) is sufficient. The attack is repeatable indefinitely and is not self-limiting because each 404 response is returned quickly, allowing the attacker to sustain a high request rate with minimal bandwidth.

### Recommendation

1. **Add negative-result caching in `getEntityIdFromEvmAddress()`**: Cache `null` results for EVM addresses not found in the entity table, with a short TTL (e.g., 30 seconds). This converts repeated identical-address floods into in-process cache hits.
2. **Add per-IP rate limiting**: Introduce a rate-limiting middleware (e.g., `express-rate-limit`) before route handlers, applied to unauthenticated requests, to cap requests per IP per second.
3. **Add a global DB query concurrency limit**: Use a semaphore or connection-pool cap on the REST API side to prevent it from monopolizing DB connections needed by the importer.
4. **Separate DB read replicas**: Route REST API reads to a read replica so importer write throughput is not affected by API load.

### Proof of Concept

```bash
# Generate unique opaque EVM addresses (first byte = 0xde, rest random)
# and flood the endpoint with no authentication required

for i in $(seq 1 10000); do
  ADDR=$(printf 'de%038x' $RANDOM$RANDOM$RANDOM)
  curl -s "http://<mirror-node>/api/v1/accounts/0x${ADDR}/tokens" &
done
wait
```

Each request causes `EntityService.getEntityIdFromEvmAddress()` to execute:
```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = '\xde...'
```
against the shared PostgreSQL instance. At sustained high concurrency, monitor importer lag via `SELECT * FROM record_file ORDER BY consensus_end DESC LIMIT 1` — the gap between `consensus_end` and wall-clock time will grow, indicating block ingestion delay.
