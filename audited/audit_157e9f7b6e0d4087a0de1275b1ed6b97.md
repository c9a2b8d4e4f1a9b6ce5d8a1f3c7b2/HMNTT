### Title
Missing Timestamp Range Inversion Validation in `getHookStorageChange()` Allows Forced Empty-Result Queries

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` passes user-supplied timestamp bounds directly to a native SQL `BETWEEN` clause without validating that `timestampLowerBound <= timestampUpperBound`. The guard that would catch an inverted range exists in `Bound`'s constructor but is unconditionally bypassed because the timestamp `Bound` is always constructed with `primarySortField = false`. Any unauthenticated caller can supply an inverted timestamp range (e.g., `gte:2000000000&lte:1000000000`) to force the query to return zero rows, and then present that empty result as proof that no hook storage history exists for a given key in that period.

### Finding Description

**Exact code path:**

`HooksController.hookStorageChangeRequest()` builds the timestamp `Bound` at line 186:

```java
final var bound = Bound.of(timestamps, TIMESTAMP, HookStorageChange.HOOK_STORAGE_CHANGE.CONSENSUS_TIMESTAMP);
``` [1](#0-0) 

`Bound.of()` always passes `false` for `primarySortField`:

```java
return new Bound(timestamp, false, parameterName, field);
``` [2](#0-1) 

Inside the `Bound` constructor, the only inversion guard is:

```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
``` [3](#0-2) 

Because `primarySortField` is always `false` for timestamp bounds, this check is **never reached**. The inverted values flow unchanged into `getHookStorageChange()`:

```java
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
final long timestampUpperBound = timestamp.adjustUpperBound();
``` [4](#0-3) 

No inversion check exists anywhere between the controller and the repository call. The values are passed directly to the native SQL query:

```sql
and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [5](#0-4) 

In PostgreSQL, `BETWEEN x AND y` is equivalent to `col >= x AND col <= y`. When `x > y`, the condition is always false and zero rows are returned — regardless of what data actually exists in `hook_storage_change`.

**Root cause:** The failed assumption is that `Bound.of()` (used for non-primary sort fields) does not need to validate range inversion. This is incorrect when the resulting bounds are used in a `BETWEEN` clause that silently returns empty results on inversion.

### Impact Explanation

An attacker can craft a request such as:

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:9999999999&timestamp=lte:1000000000
```

The API returns `{"storage": []}` — an empty result — even if thousands of hook storage changes exist in the database for that key. The attacker can present this API response as authoritative evidence that no Hashgraph history exists for a given hook storage slot during any chosen time window. This directly enables false assertions about the immutability and completeness of hook execution history, undermining the integrity guarantees of the mirror node's historical record API. The impact is confined to the read path (no data is modified), but the ability to selectively suppress visible history is a meaningful integrity violation for any consumer relying on this endpoint for audit or compliance purposes.

### Likelihood Explanation

The exploit requires zero privileges — the endpoint is a public `GET` request with no authentication. The attack is trivially reproducible by any external user who can reach the API. It requires only knowledge of the timestamp query parameter syntax (publicly documented in the OpenAPI spec). It is fully repeatable and deterministic. [6](#0-5) 

### Recommendation

Add an explicit inversion check in `getHookStorageChange()` before issuing the query, mirroring the pattern already used in `NetworkServiceImpl`:

```java
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
final long timestampUpperBound = timestamp.adjustUpperBound();
if (timestampLowerBound > timestampUpperBound) {
    throw new IllegalArgumentException("Invalid range provided for timestamp");
}
``` [7](#0-6) 

Alternatively, change `Bound.of()` to pass `true` for `primarySortField` so the check in the constructor fires universally, or add a dedicated `verifyNonInverted()` method on `Bound` and call it from the controller before building the request. [8](#0-7) 

### Proof of Concept

**Preconditions:** The mirror node REST-Java service is running and has at least one `hook_storage_change` row for a known `ownerId`/`hookId`/`key`.

**Step 1 — Confirm data exists (legitimate query):**
```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:1000000000&timestamp=lte:9999999999
```
Expected: returns one or more storage entries.

**Step 2 — Trigger the vulnerability (inverted range):**
```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:9999999999&timestamp=lte:1000000000
```
Expected (with bug): returns `{"storage": []}` — zero entries — despite the data existing.

**Step 3 — Verify root cause:** The SQL executed is:
```sql
consensus_timestamp BETWEEN 9999999999 AND 1000000000
```
PostgreSQL evaluates this as `col >= 9999999999 AND col <= 1000000000`, which is always false, producing zero rows. [9](#0-8)
