### Title
Global Shared `opcodeRateLimitBucket` Allows Any Unauthenticated User to DoS All Opcode Tracing Requests

### Summary
The `opcodeRateLimitBucket` is a single application-wide token bucket with a default capacity of 1 token per second, shared across all callers with no per-IP or per-user isolation. Any unauthenticated user can consume the sole token with a single HTTP GET request, causing every other concurrent user's opcode request to receive HTTP 429 for the remainder of that second. Because the attacker can repeat this at 1 req/s indefinitely, they can sustain a near-total denial of the opcode tracing endpoint.

### Finding Description
**Code path:**

- `ThrottleProperties.java` line 29: `private long opcodeRequestsPerSecond = 1;` — default capacity is 1.
- `ThrottleConfiguration.java` lines 47–55: a single Spring bean `opcodeRateLimitBucket` is created with `capacity = opcodeRequestsPerSecond`, shared by the entire JVM process.
- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` — one global check, no caller identity involved.
- `OpcodesController.java` lines 59–65: every GET to `/{transactionIdOrHash}/opcodes` calls `throttleManager.throttleOpcodeRequest()` before any processing.

**Root cause:** The bucket is a process-wide singleton. `tryConsume(1)` succeeds for the first caller each second and fails for every subsequent caller, regardless of who they are. There is no per-IP, per-session, or per-authenticated-user bucket.

**Failed assumption:** The design assumes the endpoint will only be used by a small number of trusted operators. In practice, when `hiero.mirror.web3.opcode.tracer.enabled=true`, the endpoint is publicly reachable with no authentication gate before the throttle check.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/<any_tx_hash>/opcodes` with `Accept-Encoding: gzip` once per second.
2. Each request consumes the single token in `opcodeRateLimitBucket`.
3. All legitimate users' requests within that second hit `tryConsume(1) == false` and receive HTTP 429.
4. The bucket refills after 1 second; attacker immediately fires again.

**Why existing checks are insufficient:** The only guard is the global `tryConsume(1)` call. There is no IP-based rate limiting, no authentication requirement, and no per-user sub-bucket. The `Accept-Encoding: gzip` header check (`OpcodesController.java` lines 75–86) is trivially satisfied by any HTTP client.

### Impact Explanation
When the opcode tracer is enabled, a single attacker with no credentials can sustain a continuous denial-of-service against the entire opcode tracing endpoint. All legitimate users — developers debugging contract execution, auditors replaying transactions — are blocked for as long as the attacker maintains the 1 req/s flood. The impact is scoped to the opcode endpoint (read-only EVM replay); it does not affect fund transfers or the main contract-call endpoint. Severity is **Medium**: availability impact on a sensitive debugging feature, no confidentiality or integrity loss.

### Likelihood Explanation
The opcode tracer is **disabled by default** (`OpcodesProperties.java` line 11: `enabled = false`), which significantly reduces the attack surface in default deployments. However, any operator who enables it exposes the endpoint to this attack with zero preconditions: no account, no token, no special knowledge beyond the public API path. The attack is trivially scriptable with `curl` and repeatable indefinitely. Likelihood is **Medium** conditional on the feature being enabled.

### Recommendation
1. **Per-IP rate limiting:** Replace (or supplement) the global bucket with a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `HttpServletRequest.getRemoteAddr()`), so each IP gets its own 1 req/s allowance.
2. **Require authentication:** Gate the opcode endpoint behind an authentication/authorization filter so only known operators can reach `throttleOpcodeRequest()`.
3. **Increase global capacity with per-user sub-limits:** If per-IP bucketing is not feasible, raise `opcodeRequestsPerSecond` to a value that accommodates N concurrent legitimate users and add a per-request IP check at the controller layer.

### Proof of Concept
```bash
# Terminal 1 – attacker (fires once per second, consuming the sole token)
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/0xABCD.../opcodes"
  sleep 0.9
done

# Terminal 2 – legitimate user (receives 429 on every attempt)
curl -v -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/0xDEAD.../opcodes"
# Expected: HTTP/1.1 429 Too Many Requests
#           {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

Precondition: `hiero.mirror.web3.opcode.tracer.enabled=true` in the deployment configuration. No credentials required.
