### Title
Unauthenticated Metrics Endpoint DoS via Unbounded `exporter.collect()` Invocation

### Summary
When `config.metrics.config.authentication` is `false`, the `metricsMiddleware` function in `rest/middleware/metricsHandler.js` allows any unprivileged caller to trigger `exporter.collect()` and `serializer.serialize()` on every request with no rate limiting or concurrency guard. Because Node.js is single-threaded, a flood of concurrent requests to the metrics path saturates the event loop, exhausting CPU and memory and causing the process to stop servicing all other connections.

### Finding Description
**Code path:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware`, lines 156–164.

```
// line 137-144
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ← unconditional bypass, no identity check at all
  }
  ...
};

// line 156-164
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) { ... }          // passes immediately when auth=false
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));   // synchronous, CPU-bound
  });
}
```

**Root cause:** Two independent failures compose:

1. `authenticate()` returns `true` unconditionally when `authentication` is `false` — no IP allowlist, no token, no identity of any kind is required.
2. There is no rate limiter, concurrency cap, or request queue anywhere in the middleware stack for this path. The grep across `rest/**/*.js` for `rateLimit`, `throttle`, `semaphore`, and `queue` returns zero matches.

**Exploit flow:**

- `exporter.collect()` fires all registered observable callbacks synchronously inside the Promise microtask (CPU usage delta, four `process.memoryUsage()` reads, etc.) before resolving.
- `serializer.serialize(resourceMetrics)` is a fully synchronous, CPU-bound loop over every registered instrument and its bucket boundaries.
- With N concurrent requests in flight, N copies of this work are queued in the microtask/Promise chain. Because the Node.js event loop is single-threaded, each serialization turn blocks I/O processing for all other sockets.

**Why existing checks are insufficient:**

- The `authHandler` middleware (`rest/middleware/authHandler.js`) handles `config.users` API-key auth for data endpoints; it has no effect on the metrics path.
- The `authenticate()` function inside `metricsHandler.js` is the sole gate and is explicitly designed to be a no-op when `authentication=false`.
- No infrastructure-level guard is enforced by the application code itself. [1](#0-0) [2](#0-1) 

### Impact Explanation
The metrics endpoint shares the same Node.js process and event loop as the entire REST API. Saturating it with `exporter.collect()` + `serializer.serialize()` calls stalls the event loop, causing:

- All in-flight API requests to time out (TCP connections dropped from the client's perspective).
- Health-check endpoints (`/health/readiness`, `/health/liveness`) to become unresponsive, potentially triggering a Kubernetes pod restart loop.
- Complete loss of service availability for all consumers of the mirror-node REST API until the flood stops or the process is restarted.

Severity: **High** — full process-level DoS with no authentication required.

### Likelihood Explanation
- The precondition (`authentication: false`) is a documented, operator-selectable option. Operators commonly disable authentication on internal or scrape-only deployments to simplify Prometheus configuration.
- The attack requires zero privileges, zero credentials, and zero knowledge beyond the publicly documented default metrics path (`/swagger/metrics/`).
- It is trivially repeatable with any HTTP load tool (`wrk`, `ab`, `hey`, `curl` in a loop). No exploit code needs to be written.

### Recommendation
Apply at least two independent mitigations:

1. **Rate-limit the metrics path** — add a per-IP rate limiter (e.g., `express-rate-limit`) scoped exclusively to `metricsPath` before the `exporter.collect()` call. A limit of ~10 req/s per IP is more than sufficient for any legitimate Prometheus scraper.

2. **Concurrency cap** — use a simple in-flight counter or a Promise-based semaphore to reject (HTTP 429) requests when a `collect()` is already in progress, since there is no value in running two simultaneous collections.

3. **Restrict network exposure** — when `authentication` is `false`, document and enforce that the metrics port/path must be firewalled to the scrape network only; do not expose it on the public-facing port.

### Proof of Concept
```bash
# Precondition: config.metrics.config.authentication = false (non-default but valid)
# Default metrics path: /swagger/metrics/

# Step 1 – verify unauthenticated access works
curl -s http://<mirror-node-host>:5551/swagger/metrics/   # returns 200 + Prometheus text

# Step 2 – flood with concurrent requests (no credentials needed)
wrk -t 8 -c 200 -d 30s http://<mirror-node-host>:5551/swagger/metrics/

# Step 3 – in a second terminal, observe API becoming unresponsive
curl -s http://<mirror-node-host>:5551/api/v1/accounts   # hangs / connection refused

# Step 4 – observe health checks failing
curl -s http://<mirror-node-host>:5551/health/liveness    # hangs / 503
```

During the flood, Node.js CPU will pin at ~100% on a single core and the event loop lag will grow unboundedly, causing all non-metrics connections to time out.
