### Title
EVM Address Existence Oracle via `from=` Filter in `getContractResultsById()`

### Summary
Any unauthenticated caller can supply a valid EVM address as the `from=` query parameter to `GET /api/v1/contracts/:contractId/results`. The server performs a live database lookup and, when the address is absent, throws a `NotFoundError` that is mapped to an HTTP 404 response. When the address exists, the request proceeds and returns HTTP 200. This creates a direct, unauthenticated oracle for EVM address existence in the Hedera entity table — stronger than a timing side-channel because the response code itself is the signal.

### Finding Description

**Exact code path:**

1. Route: `rest/routes/contractRoute.js:14` — `router.getExt('/:contractId/results', ContractController.getContractResultsById)` — no authentication middleware.

2. Handler: `rest/controllers/contractController.js:856-892` — `getContractResultsById` calls `extractContractResultsByIdQuery(filters, contractId)` at line 871.

3. Filter processing: `contractController.js:441-444` — inside `extractContractResultsByIdQuery`, for `filterKeys.FROM`:
   ```js
   if (EntityId.isValidEvmAddress(filter.value)) {
     filter.value = await EntityService.getEncodedId(filter.value);
   }
   ```
   `getEncodedId` is called with the default `requireResult = true`.

4. `rest/service/entityService.js:118-137` — `getEncodedId` detects an EVM address and delegates to `getEntityIdFromEvmAddress(entityId, requireResult=true)`.

5. `entityService.js:90-104` — `getEntityIdFromEvmAddress` executes:
   ```sql
   SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
   ```
   If zero rows are returned **and** `requireResult` is `true`, it throws `new NotFoundError()` (line 94).

6. `NotFoundError` is not caught inside `getEncodedId` (line 132-133 re-throws non-`InvalidArgumentError` exceptions) and propagates all the way to the Express error handler.

7. `rest/middleware/httpErrorHandler.js:17-18` — `handleError` maps `NotFoundError` → HTTP 404.

**Root cause:** `getEncodedId` is called with `requireResult = true` (the default). There is no caller-side guard that passes `requireResult = false` for the `from=` filter case. A missing EVM address therefore aborts the entire request with a 404 instead of silently returning zero results.

**Failed assumption:** The code assumes that a `from=` EVM address that is not in the entity table is an error condition. In reality, it is a valid filter that should simply match no rows.

**Exploit flow:**
- Attacker sends: `GET /api/v1/contracts/0.0.1/results?from=<candidate_evm_address>`
- Address absent → HTTP 404 `{"_status":{"messages":[{"message":"Not found"}]}}`
- Address present → HTTP 200 (empty or populated results list)
- Repeat for any set of candidate addresses.

**Why existing checks are insufficient:**
- `EntityId.isValidEvmAddress` only validates the 20-byte hex format; it does not prevent the lookup.
- No authentication or authorization is required on the route.
- No rate-limiting middleware is visible in `contractRoute.js`.
- The `requireResult` flag exists precisely to make the lookup optional, but it is never set to `false` at the call site in `contractController.js:444`.

### Impact Explanation
An attacker can enumerate which EVM addresses are registered as live entities (accounts, contracts, tokens) in the Hedera mirror node's entity table without any credentials. Beyond existence disclosure, the 200-path also executes a second database query (`getContractResultsByIdAndFilters`), so response latency for existing addresses is measurably higher — providing a corroborating timing signal. On a public blockchain the existence of an address is not a secret per se, but this oracle allows bulk, automated, unauthenticated enumeration of the internal entity registry, which can be used for targeted reconnaissance (e.g., identifying which EVM addresses map to active Hedera accounts before crafting further attacks).

### Likelihood Explanation
The endpoint is public, requires no authentication, and accepts arbitrary EVM addresses. The oracle is binary and deterministic (not probabilistic), so a single HTTP request per address is sufficient. Any attacker with network access can automate enumeration at the rate the server accepts connections. No special privileges, tokens, or insider knowledge are required.

### Recommendation
At `contractController.js:443-444`, pass `requireResult = false` to `getEncodedId` so that a missing EVM address returns `null` instead of throwing:

```js
case filterKeys.FROM:
  if (EntityId.isValidEvmAddress(filter.value)) {
    const encodedId = await EntityService.getEncodedId(filter.value, false); // requireResult=false
    if (encodedId === null) {
      // Address not found — no results can match; short-circuit with empty response
      return {skip: true};
    }
    filter.value = encodedId;
  }
  ...
```

This eliminates the 404/200 divergence. Additionally, consider adding rate-limiting middleware to the `/contracts/:contractId/results` route to mitigate bulk enumeration regardless of this fix.

### Proof of Concept

```bash
# Step 1 – address known NOT to exist in entity table → 404
curl -s -o /dev/null -w "%{http_code}" \
  "https://<mirror-node>/api/v1/contracts/0.0.1/results?from=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"
# → 404

# Step 2 – address known to exist (e.g., a deployed contract's EVM address) → 200
curl -s -o /dev/null -w "%{http_code}" \
  "https://<mirror-node>/api/v1/contracts/0.0.1/results?from=0x<known_evm_address>"
# → 200

# Step 3 – automate enumeration
for addr in $(cat candidate_addresses.txt); do
  code=$(curl -s -o /dev/null -w "%{http_code}" \
    "https://<mirror-node>/api/v1/contracts/0.0.1/results?from=${addr}")
  echo "$addr -> $code"
done
# 200 lines reveal registered EVM addresses; 404 lines confirm absence
```
