### Title
UNION ALL Subquery Overlap Produces Duplicate Allowance Rows via Crafted `spender.id` Range

### Summary
When a caller supplies a degenerate closed range for `spender.id` (e.g., `gte:X` and `lte:X` with the same value) together with a closed range for `token.id`, the `getLowerFilters` and `getUpperFilters` helpers both emit `spender = X` as the primary equality predicate. Because `TokenAllowanceService.getQuery()` joins the resulting subqueries with `UNION ALL` (no deduplication), every row that satisfies both predicates is returned twice, presenting incorrect allowance data to the caller.

### Finding Description

**Exact code path**

`getAccountTokenAllowances` (controller, line 68–81) → `extractTokenMultiUnionQuery` (line 22–60) → `validateBounds` (line 49) → `getLowerFilters` / `getInnerFilters` / `getUpperFilters` (baseController.js lines 131–183) → `TokenAllowanceService.getQuery` (tokenAllowanceService.js lines 54–78) → `UNION ALL` at line 74.

**Root cause**

`getLowerFilters` (baseController.js line 141) rewrites `primary.lower` to an equality operator and emits `spender = lower_value`:

```js
filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
```

`getUpperFilters` (baseController.js line 182) independently rewrites `primary.upper` to an equality operator and emits `spender = upper_value`:

```js
return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
```

When `lower_value === upper_value` (i.e., the user supplies `spender.id=gte:5&spender.id=lte:5`), both subqueries carry the identical predicate `spender = 5`. Any database row that also satisfies both the `token.id >= lower` and `token.id <= upper` predicates is matched by **both** the lower and the upper subquery. `UNION ALL` at line 74 concatenates without deduplication, so those rows appear twice.

**Why validation does not catch it**

`validateBoundsRange` (baseController.js line 117–123) only rejects the combination of a range **and** an equality on the same key:

```js
if (bounds[key].hasBound() && bounds[key].hasEqual()) { throw ... }
```

A range where `gte_value === lte_value` has no `equal` field set, so the check passes. `validateLowerBounds` and `validateUpperBounds` (lines 83–109) only verify that the operator tier (gte vs gt, lte vs lt) is compatible with the secondary bound direction; they do not compare the numeric values of the lower and upper bounds. No validation anywhere detects or rejects `gte:X lte:X` with the same `X`.

**Resulting SQL (concrete example)**

Request: `GET /api/v1/accounts/1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200`

```sql
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $3 and token_id >= $4   -- lower: spender=5, token_id>=100
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender > $5 and spender < $6     -- inner: spender>5 AND spender<5 → ∅
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $7 and token_id <= $8   -- upper: spender=5, token_id<=200
   order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
-- params: [1, limit, 5, 100, 5, 5, 5, 200]
```

Every row with `spender=5` and `100 ≤ token_id ≤ 200` satisfies both the lower and the upper subquery and is returned twice.

### Impact Explanation

The REST API returns a JSON `allowances` array containing duplicate entries for the affected `(spender, token_id)` pairs. Any client that iterates the list, sums amounts, or makes authorization decisions based on the presence of an allowance will process phantom duplicate records. Pagination is also corrupted: the outer `LIMIT` is consumed by duplicates, so the `next` link may skip real records. The data integrity of the read path is broken for the targeted account without any write access required.

### Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The trigger requires only two query parameters with the same numeric value (`spender.id=gte:X&spender.id=lte:X`) plus a `token.id` range — a trivially crafted URL. No account ownership, signing key, or elevated privilege is needed. The condition is deterministic and 100% reproducible for any account that has at least one token allowance for the targeted spender.

### Recommendation

1. **Add a value-level range check in `validateBoundsRange`** (baseController.js): after confirming both `lower` and `upper` are set, assert `lower.value < upper.value` (strict). If they are equal, either reject the request with `InvalidArgumentError` or internally collapse the range to a single equality filter, routing everything through the single-subquery path.

2. **Use `UNION` instead of `UNION ALL`** in `TokenAllowanceService.getQuery` (line 74) as a defence-in-depth measure, so that even if overlapping predicates are generated the database deduplicates the result set.

### Proof of Concept

**Preconditions:** Account `0.0.1` has at least one token allowance for spender `0.0.5` with a token whose numeric ID is between 100 and 200 (e.g., token `0.0.150`).

**Trigger:**
```
GET /api/v1/accounts/0.0.1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200
```

**Expected (correct) result:** one entry for `(spender=5, token_id=150)`.

**Actual result:** two identical entries for `(spender=5, token_id=150)` in the `allowances` array, because the row is matched by both the lower subquery (`spender=5 AND token_id>=100`) and the upper subquery (`spender=5 AND token_id<=200`), and `UNION ALL` does not deduplicate.

**Verification:** Repeat with `spender.id=gte:5&spender.id=lt:6` (strict upper bound, different value) — the inner query becomes `spender>5 AND spender<6` and the upper subquery is suppressed (no `secondary.upper`), so no duplicate is produced, confirming the overlap only occurs when `lower_value === upper_value`.
