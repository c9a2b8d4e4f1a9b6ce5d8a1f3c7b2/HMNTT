### Title
Unauthenticated Full-Table-Scan DoS via Unbounded Repeated `name` Filter Queries in `getTokensRequest()`

### Summary
In `rest/tokens.js`, `getTokensRequest()` constructs a PostgreSQL `ILIKE '%value%'` query for the `name` filter. The leading wildcard prevents any B-tree index from being used, forcing a full sequential scan of the token table on every request. Because `nextLink` is unconditionally set to `null` when `hasNameParam` is true, there is no cursor-based pagination — every name-filtered request re-scans the entire table from scratch. The REST API layer has no rate-limiting middleware, so any unauthenticated external user can flood this endpoint with concurrent name queries, exhausting database CPU and I/O.

### Finding Description

**Code path:**

`rest/tokens.js` — `extractSqlFromTokenRequest()`, line 177:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
```
The `%` prefix on the ILIKE pattern makes the predicate non-sargable: PostgreSQL cannot use a B-tree index on `t.name` and must perform a sequential scan of the entire token table. The `LIMIT` clause (line 210) only caps the number of rows returned — the database engine still scans the full table to find the first `limit` matching rows, especially when the search term matches few or no rows.

`rest/tokens.js` — `getTokensRequest()`, lines 405–406:
```js
const nextLink = hasNameParam
  ? null
```
When `name` is present, `nextLink` is always `null`. This is not merely a UI omission — it means there is no cursor anchor (`token.id` lower bound) injected into the query. Every name-filtered request therefore starts from `token_id = 0` and scans the full table, with no way to resume from a checkpoint.

`rest/server.js`, lines 68–144: The middleware stack for the REST API is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. There is no rate-limiting middleware applied to `/api/v1/tokens` or any other REST route. The throttle configuration found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the Java-based web3 module (contract calls), not to the Node.js REST API.

**Validation bypass:** The only guard on the `name` parameter is:
```js
// rest/tokens.js line 334
ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
```
This enforces a 3–100 byte length and `eq` operator only. It does not prevent high-frequency repeated requests, does not require authentication, and does not prevent the leading-wildcard scan pattern.

**Exploit flow:**
1. Attacker (no credentials required) sends a high volume of concurrent requests: `GET /api/v1/tokens?name=abc`
2. Each request passes validation (3 bytes, `eq` operator).
3. `extractSqlFromTokenRequest` builds: `WHERE t.name ILIKE '%abc%' ORDER BY token_id ASC LIMIT 25`
4. PostgreSQL performs a full sequential scan of the token table for each request.
5. Because `nextLink` is `null`, the attacker cannot be redirected to a cursor-based path; each request is a fresh full scan.
6. With N concurrent requests, N full sequential scans run simultaneously, saturating DB I/O and CPU.

### Impact Explanation
A sustained flood of name-filter requests causes repeated full sequential scans of the token table. On a production network with millions of token records, each scan is expensive. Concurrent scans degrade query latency for all other API consumers (transactions, accounts, balances), potentially causing timeouts and cascading failures across the mirror node REST API. This is a denial-of-service against the database tier, achievable without any credentials or special knowledge.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The attack requires only a standard HTTP client and knowledge of the public API (documented in the OpenAPI spec at `/api/v1/openapi.yml`). The attacker needs no special privileges, no tokens, and no prior state. The attack is trivially repeatable and automatable (e.g., a simple loop or load-testing tool). The 3-byte minimum name length is not a meaningful barrier.

### Recommendation
1. **Eliminate the leading wildcard**: Change the ILIKE pattern from `'%' + value + '%'` to `value + '%'` (prefix match only), which allows PostgreSQL to use a B-tree or `text_pattern_ops` index on `t.name`.
2. **Add a functional index**: Create `CREATE INDEX ON token (lower(name) text_pattern_ops)` to support case-insensitive prefix searches efficiently.
3. **Add rate limiting to the REST API**: Apply a per-IP rate-limiting middleware (e.g., `express-rate-limit`) to the Express app in `rest/server.js`, with a stricter limit specifically for name-filtered token queries.
4. **Re-enable cursor pagination for name searches**: Instead of setting `nextLink = null`, allow cursor-based pagination even with the name filter by including a `token.id` lower bound in the query, preventing repeated full-table scans.
5. **Add a query timeout**: Set a PostgreSQL `statement_timeout` for REST API connections to bound the maximum duration of any single scan.

  curl -s "https://<mirror-node-host>/api/v1/tokens?name=abc" &

# Each request triggers:
# SELECT ... FROM token t JOIN entity e ON e.id = t.token_id

# PostgreSQL executes a full sequential scan of the token table for each of the 50 concurrent requests.
