### Title
Missing File ID Whitelist Validation in REST Java `getNetworkNodes()` Allows Arbitrary Address Book Queries

### Summary
The REST Java `NetworkServiceImpl.getNetworkNodes()` accepts any caller-supplied `file.id` value and passes it directly to the database query without validating it against the canonical set `{101, 102}`. The gRPC counterpart explicitly enforces this whitelist, but the REST Java path does not, allowing any unauthenticated user to query address book entries for arbitrary file IDs stored in the database.

### Finding Description

**Controller** (`rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 153–157):

```java
final var fileId = request.getFileId();
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
```

The only check is that the operator is `EQ` — the actual numeric value of `file.id` is never validated.

**Service** (`rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, lines 178–182):

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
}
```

The user-supplied value is returned verbatim and passed directly to `networkNodeRepository.findNetworkNodes(fileId, ...)`.

**Repository** (`rest-java/src/main/java/org/hiero/mirror/restjava/repository/NetworkNodeRepository.java`, lines 26–31):

```sql
with latest_address_book as (
    select start_consensus_timestamp, end_consensus_timestamp, file_id
    from address_book
    where file_id = :fileId
    order by start_consensus_timestamp desc
    limit 1
)
```

The `file_id` filter is the raw attacker-controlled value.

**Contrast with the gRPC service** (`grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`, lines 50–59):

```java
private final Set<EntityId> validFileIds =
        Set.of(systemEntity.addressBookFile101(), systemEntity.addressBookFile102());

if (!getValidFileIds().contains(fileId)) {
    throw new IllegalArgumentException(INVALID_FILE_ID);
}
```

The gRPC service enforces the whitelist; the REST Java service does not. This is the root cause: a missing guard that exists in the parallel implementation.

### Impact Explanation
An attacker can query the `address_book` table for any `file_id` present in the database. The `address_book` table stores historical snapshots keyed by `file_id`. Supplying a non-canonical ID (e.g., `file.id=101` when the canonical REST default is `102`, or any other numeric ID) returns the latest address book snapshot for that file, which may differ from the authoritative current topology. Downstream consumers (wallets, SDKs, monitoring tools) that trust the `/api/v1/network/nodes` endpoint could be fed a stale or non-canonical node list, enabling topology confusion, node impersonation setup, or denial-of-service by directing traffic to decommissioned nodes.

### Likelihood Explanation
No authentication or privilege is required. The `file.id` query parameter is publicly documented and trivially supplied. Any HTTP client can reproduce this with a single GET request. The attack is repeatable, stateless, and requires no prior knowledge beyond the API endpoint.

### Recommendation
Add a whitelist check in `NetworkServiceImpl.getAddressBookFileId()` (or in the controller before calling the service) mirroring the gRPC service:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    long valid101 = systemEntity.addressBookFile101().getId();
    long valid102 = systemEntity.addressBookFile102().getId();
    if (id != valid101 && id != valid102) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

### Proof of Concept

```
