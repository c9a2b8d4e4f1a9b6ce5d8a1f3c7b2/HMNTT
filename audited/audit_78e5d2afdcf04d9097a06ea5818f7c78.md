### Title
Unprivileged Historical Block Requests Allow Metric Tag Backdating via Insufficient Timestamp Filter in `updateMetrics()`

### Summary
The `getBlock()` method in `ContractCallService` only filters out **future** timestamps to prevent metric tag cardinality explosion, but applies no equivalent guard against **past** timestamps. Any unprivileged caller can submit a standard `eth_call` with a historical block number, causing `invocationCounter` to be incremented under a `YearMonth` tag derived from that block's consensus timestamp — making it appear that EVM invocations occurred in an arbitrary past month.

### Finding Description
**Code path:**
- `ContractCallService.java`, `updateMetrics()`, lines 163–170
- `ContractCallService.java`, `getBlock()`, lines 177–184
- `ContractCallContext.java`, `getTimestamp()`, lines 111–116

**Root cause:**

In `getBlock()`:
```java
private String getBlock() {
    return ContractCallContext.get()
            .getTimestamp()
            .filter(t -> t <= Utils.getCurrentTimestamp()) // only blocks future timestamps
            .map(t -> YearMonth.from(Instant.ofEpochSecond(0L, t).atZone(UTC)).toString())
            .orElse(BlockType.LATEST.toString());
}
```

The filter `t <= Utils.getCurrentTimestamp()` only removes timestamps that are in the **future**. Any timestamp from a historical block (i.e., `t < now`) passes through and is formatted as a `YearMonth` string (e.g., `"2020-03"`).

In `ContractCallContext.getTimestamp()`:
```java
public Optional<Long> getTimestamp() {
    if (useHistorical()) {
        return getTimestampOrDefaultFromRecordFile();
    }
    return Optional.empty();
}
```

`useHistorical()` returns `true` whenever `params.getBlock() != BlockType.LATEST` — i.e., whenever the caller specifies any explicit block number. The timestamp returned is the `consensusEnd` of the record file for that block.

**Exploit flow:**
1. Attacker sends `eth_call` (or `eth_estimateGas`) with `block = 0x1` (or any early block number).
2. `recordFileService.findByBlockType(params.getBlock())` resolves the record file for block 1, whose `consensusEnd` is a nanosecond timestamp from, e.g., September 2019.
3. `getTimestamp()` returns that historical nanosecond value.
4. The filter `t <= Utils.getCurrentTimestamp()` passes (past timestamp ≤ now).
5. The timestamp is mapped to `"2019-09"`.
6. `invocationCounter.withTags(...TAG_BLOCK, "2019-09").increment()` fires.
7. Metrics now record an EVM invocation attributed to September 2019.

**Why existing checks fail:**
The only guard is the future-timestamp filter. There is no floor check, no restriction on how far in the past a block can be, and no normalization of historical block tags to a fixed sentinel value. The comment in the code ("Filter future timestamps to reduce cardinality") confirms the developer only considered the future direction.

### Impact Explanation
- **Metrics integrity**: Operational dashboards and alerting systems that track `hiero.mirror.web3.evm.invocation` by `block` (month) will show fabricated historical invocation counts. Months that had zero real invocations can be made to show arbitrarily large counts.
- **High cardinality DoS on metrics registry**: Each unique `YearMonth` tag value creates a new `Counter` instance in the `MeterProvider`. An attacker cycling through blocks spanning many months (e.g., 2019-01 through present = ~80+ distinct tag values) inflates the metrics registry, potentially causing memory pressure or metric scrape failures.
- **Audit/billing distortion**: If invocation metrics feed into capacity planning, SLA reporting, or usage billing, backdated counts corrupt those downstream processes.
- Severity: **Medium** (no fund loss, but observable system integrity is compromised and DoS on metrics is feasible).

### Likelihood Explanation
- **No privileges required**: `eth_call` with a block parameter is a standard, unauthenticated JSON-RPC operation.
- **Trivially repeatable**: A script cycling through historical block numbers can generate thousands of backdated metric increments per minute.
- **No rate-limit specific to block parameter**: The throttle manager operates on gas, not on the block parameter value, so this bypass is orthogonal to existing throttling.

### Recommendation
Replace the one-sided future filter with a two-sided guard that normalizes any historical block query to a fixed tag value (e.g., `"historical"`) rather than exposing the raw `YearMonth`:

```java
private String getBlock() {
    return ContractCallContext.get()
            .getTimestamp()
            .filter(t -> t <= Utils.getCurrentTimestamp())
            // Normalize all historical timestamps to a single tag to prevent backdating
            .map(t -> "historical")
            .orElse(BlockType.LATEST.toString());
}
```

Alternatively, if month-level granularity is genuinely needed for current-month invocations only, restrict the tag to the current `YearMonth` or `"historical"`:

```java
.map(t -> {
    YearMonth ym = YearMonth.from(Instant.ofEpochSecond(0L, t).atZone(UTC));
    YearMonth now = YearMonth.now(UTC);
    return ym.equals(now) ? ym.toString() : "historical";
})
```

This eliminates both the backdating attack and the cardinality explosion.
