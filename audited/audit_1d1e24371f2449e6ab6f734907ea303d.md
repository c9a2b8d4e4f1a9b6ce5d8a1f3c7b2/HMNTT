### Title
Range Filter Bypass via Unfiltered `keys` Passed to SQL in `getHookStorageChange()`

### Summary
In `HookServiceImpl.getHookStorageChange()`, the Java-side range filter (`getKeysInRange()`) is computed and used only as an early-exit guard, but the actual SQL query receives the original unfiltered `keys` collection. An unprivileged user can craft a request containing at least one in-range key (to pass the guard) alongside arbitrary out-of-range keys, causing the database to return `hook_storage_change` rows for keys entirely outside the declared bounds.

### Finding Description
**Exact code path:**

`HookStorageRequest.getKeysInRange()` ( [1](#0-0) ) filters the caller-supplied `keys` collection using `Arrays.compareUnsigned()`, returning only keys that fall within `[keyLowerBound, keyUpperBound]`.

Back in `getHookStorageChange()` ( [2](#0-1) ), `keysInRange` is used **only** for the early-exit guard:

```java
if (keysInRange.isEmpty() && requestHasKeys) {
    return new HookStorageResult(ownerId, List.of());
}
``` [3](#0-2) 

When the guard is not triggered, the code passes the **original, unfiltered `keys`** to the repository:

```java
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys,   // ← 'keys', NOT 'keysInRange'
        timestampLowerBound, timestampUpperBound, page);
``` [4](#0-3) 

The SQL query performs a plain `key in (:keys)` equality match with no range enforcement: [5](#0-4) 

**Root cause:** `keysInRange` is computed but never substituted for `keys` in the branch that executes the query. The failed assumption is that the early-exit guard is sufficient to enforce the range contract; it is not — it only rejects the case where *all* keys are out of range.

**Contrast with the non-historical path**, which correctly passes `keysInRange` to the repository: [6](#0-5) 

### Impact Explanation
An attacker can read `hook_storage_change` rows for arbitrary keys belonging to any `(ownerId, hookId)` pair, bypassing the key-range access control that the API is supposed to enforce. Because `hook_storage_change` records EVM hook storage writes (including financial state such as token balances and allowances written by hooks), this constitutes unauthorized disclosure of sensitive on-chain financial data. In a system where storage-change history is used to reconstruct account state or audit fund movements, leaking out-of-range entries can expose confidential financial positions of other users.

### Likelihood Explanation
No privileges are required. The attacker only needs to know (or enumerate) one valid key within the declared range for a target `(ownerId, hookId)` and can then append any number of out-of-range keys to the same request. The API is publicly accessible, the exploit is deterministic and repeatable, and requires no special tooling beyond a standard HTTP client.

### Recommendation
Replace `keys` with `keysInRange` in the `findByKeyInAndTimestampBetween` call:

```java
// Before (vulnerable)
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, ...);

// After (fixed)
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keysInRange, ...);
``` [4](#0-3) 

This mirrors the correct pattern already used in the non-historical `getHookStorage()` path.

### Proof of Concept
**Preconditions:**
- The `hook_storage_change` table contains rows for `(ownerId=A, hookId=B)` with keys `K_in` (within range `[LB, UB]`) and `K_secret` (outside range `[LB, UB]`).

**Steps:**
1. Send a historical storage request (with a `timestamp` parameter so `isHistorical()` returns `true`) for `(ownerId=A, hookId=B)` with:
   - `keyLowerBound = LB`
   - `keyUpperBound = UB`
   - `keys = [K_in, K_secret]`
2. `getKeysInRange()` returns `[K_in]` — non-empty, so the early-exit guard at line 93 is **not** triggered.
3. The code reaches line 104 and calls `findByKeyInAndTimestampBetween(..., keys=[K_in, K_secret], ...)`.
4. The SQL `key in (:keys)` matches both `K_in` and `K_secret`.
5. The response contains the storage entry for `K_secret`, which is outside the declared range and should not have been returned.

    public List<byte[]> getKeysInRange() {
        if (keys.isEmpty()) {
            return List.of();

        return keys.stream()
                .filter(key -> Arrays.compareUnsigned(key, keyLowerBound) >= 0
                        && Arrays.compareUnsigned(key, keyUpperBound) <= 0)

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
```
