### Title
Missing Application-Level Rate Limiting and Disabled-by-Default Circuit Breaker on `getHooks()` Enables Unauthenticated Request Flood

### Summary
`HookServiceImpl.getHooks()` performs unbounded-rate database queries with no application-level throttle or circuit breaker. The only circuit-breaker protection is defined in the Traefik Helm chart but is **disabled by default** (`global.middleware: false`), meaning a default deployment has zero automatic load-shedding. An unauthenticated attacker can flood the public endpoint to exhaust database connections and cause mirror node service unavailability.

### Finding Description

**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, lines 32–53; `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, lines 80–102.

Every call to `GET /api/v1/accounts/{id}/hooks` reaches `HookServiceImpl.getHooks()` which unconditionally issues one of two database queries:

```java
// HookServiceImpl.java:40
return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
// or line 51
return hookRepository.findByOwnerIdAndHookIdIn(id.getId(), idsInRange, page);
```

No rate-limiting, no circuit-breaker annotation (`@RateLimiter`, `@CircuitBreaker`, `@Bulkhead` from Resilience4j), and no bucket4j throttle (contrast with `web3`'s `ThrottleManagerImpl`) exist anywhere in the `rest-java` service layer.

The only protection is the Traefik middleware defined in `charts/hedera-mirror-rest-java/values.yaml` lines 150–155:

```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 3
      initialInterval: 100ms
```

However, the middleware template at `charts/hedera-mirror-rest-java/templates/middleware.yaml` line 3 gates deployment on:

```
{{ if and .Values.global.middleware .Values.middleware -}}
```

And `global.middleware` defaults to `false` (`charts/hedera-mirror-rest-java/values.yaml` line 98):

```yaml
global:
  middleware: false
```

This means **the circuit breaker is never deployed in a default installation**. The `retry: attempts: 3` middleware is also absent, but even if it were present, retries amplify load rather than shed it.

The controller's only guards are parameter-size constraints (`@Max(MAX_LIMIT)` on `limit`, `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `hookId`), which bound result-set size per request but impose no constraint on request rate.

### Impact Explanation
An attacker can sustain a high-rate flood of `GET /api/v1/accounts/{id}/hooks` requests. Each request consumes a HikariCP connection for the duration of the query. The Prometheus alert `RestJavaHighDBConnections` fires at 75% pool utilization but is monitoring-only — it does not shed load. Once the connection pool is exhausted, all mirror node REST-Java endpoints (not just hooks) begin queuing or rejecting requests, causing full service unavailability of the mirror node read API. This does not directly halt Hedera consensus-layer transaction confirmation (the mirror node is read-only), but it prevents clients from querying transaction status, block data, and network state, effectively making the mirror node unusable.

### Likelihood Explanation
The endpoint requires no authentication. Any external user can send HTTP GET requests. The attack requires only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`). The default `hpa.enabled: false` means no horizontal scaling occurs automatically. The default `podDisruptionBudget.enabled: false` provides no resilience floor. This is trivially repeatable from a single host or botnet.

### Recommendation
1. **Application-level rate limiting:** Add a bucket4j or Resilience4j `@RateLimiter` / `@Bulkhead` to `HooksController.getHooks()` or at the service layer, mirroring the pattern in `web3`'s `ThrottleManagerImpl`.
2. **Enable middleware by default:** Change `global.middleware` default to `true` in `charts/hedera-mirror-rest-java/values.yaml`, or document it as a required production setting with a deployment-time check.
3. **Enable HPA by default:** Set `hpa.enabled: true` so the service can scale under load.
4. **Add `inFlightReq` middleware:** As done in `hedera-mirror-rosetta/values.yaml` (lines 152–155), add an `inFlightReq` limit per source IP to the rest-java middleware chain.

### Proof of Concept
```bash
# Precondition: default Helm deployment with global.middleware=false (the default)

```java
    public Collection<Hook> getHooks(HooksRequest request) {
        final var sort = Sort.by(request.getOrder(), HOOK_ID);
        final var page = PageRequest.of(0, request.getLimit(), sort);
        final var id = entityService.lookup(request.getOwnerId());
        final long lowerBound = request.getLowerBound();
        final long upperBound = request.getUpperBound();

        if (request.getHookIds().isEmpty()) {
            return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
        } else {
            // Both equal and range filters are present.
            final var idsInRange = request.getHookIds().stream()
                    .filter(hookId -> hookId >= lowerBound && hookId <= upperBound)
                    .toList();

            if (idsInRange.isEmpty()) {
                return List.of();
            }

            return hookRepository.findByOwnerIdAndHookIdIn(id.getId(), idsInRange, page);
        }
    }
```
