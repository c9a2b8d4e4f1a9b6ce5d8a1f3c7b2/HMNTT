### Title
Unauthenticated GraphQL EVM Address Lookup Enables Unbounded DB Query Flooding via Crafted Addresses

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` routes to `entityRepository.findByEvmAddress()` whenever the first 4 bytes of the decoded address are non-zero OR the next 8 bytes are non-zero. An unauthenticated attacker can craft valid hex addresses (passing the schema `@Pattern` check) that always bypass the fast `findById` path and force a secondary-index DB query per request. The GraphQL module has no rate limiting, enabling unbounded DB query flooding.

### Finding Description

**Code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34â€“41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-3 AND bytes 4-11 must all be zero
        return entityRepository.findById(buffer.getLong())...;     // fast PK lookup
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)...;  // secondary-index query
}
```

The fast path (`findById`) is only taken when bytes 0â€“11 of the 20-byte address are all zero. Any address where bytes 4â€“11 are non-zero (while bytes 0â€“3 are zero) fails the `getLong() == 0` check and falls through to `findByEvmAddress`.

**Schema validation** (`graphql/src/main/resources/graphql/account.graphqls`, line 96):
```
evmAddress: String @Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")
```
This only validates hex format and length â€” it places no restriction on byte content. Any 40-char hex string passes.

**The DB query triggered** (`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 16â€“17):
```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```
This executes a full parameterized SQL query against the `entity` table on every call.

**No rate limiting exists in the GraphQL module.** `GraphQlConfiguration` (`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`) only configures query complexity (max 200), depth (max 10), and JSON/parser size limits â€” no per-IP or global request rate throttle. The `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiting exists only in the `web3` module, not `graphql`.

**Root cause:** The combination of (1) a publicly accessible, unauthenticated GraphQL endpoint, (2) no request rate limiting in the graphql module, and (3) a routing condition that can be deterministically bypassed by crafting valid hex addresses, allows an attacker to generate an unbounded stream of secondary-index DB queries.

### Impact Explanation

Every crafted request causes one `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` query. For non-existent addresses the query returns empty but still consumes DB CPU for the index scan. At high request rates (easily achievable from a single machine against an unthrottled HTTP endpoint), this can saturate DB CPU well above the 30% baseline threshold. The `entity` table is central to the mirror node and shared across components; elevated CPU on it degrades all consumers.

### Likelihood Explanation

No privileges, accounts, tokens, or special network access are required. The GraphQL endpoint is publicly reachable (port 8083 by default, documented in `docs/graphql/README.md`). The exploit requires only an HTTP client and knowledge of the 40-char hex format. The crafted address pattern (`0x00000000FF000000000000000000000000000000`) is trivially generated. The attack is fully repeatable and automatable with standard tools (curl, wrk, etc.).

### Recommendation

1. **Add rate limiting to the GraphQL module.** Implement a per-IP or global token-bucket throttle (similar to `ThrottleConfiguration` in the `web3` module) applied at the HTTP filter level before GraphQL processing.
2. **Add a schema-level or service-level constraint** rejecting addresses that are structurally invalid for the Hedera network (e.g., all-zero addresses, or addresses known to never exist).
3. **Consider caching negative results** from `findByEvmAddress` with a short TTL so repeated queries for the same non-existent address do not hit the DB each time.
