### Title
Cache-Control: public, max-age=5 Applied to 404 EntityNotFoundException Responses Enables Proxy Caching of Negative Topic Lookups

### Summary
The `@ModelAttribute responseHeaders()` method in `GenericControllerAdvice` unconditionally sets `Cache-Control: public, max-age=5` directly on the `HttpServletResponse` before the controller method executes. When `TopicService.findById()` throws `EntityNotFoundException` and the `notFound()` exception handler returns a `ResponseEntity` with `null` headers, the previously written `Cache-Control: public, max-age=5` header is not overridden and is included in the 404 response. RFC 7234-compliant intermediate proxies may cache this 404 response, causing all users behind the proxy to receive a stale "not found" error for up to 5 seconds even after the topic becomes available in the mirror node.

### Finding Description

**Code path:**

`TopicController.getTopic()` → `TopicService.findById()` throws `jakarta.persistence.EntityNotFoundException` → `GenericControllerAdvice.notFound()` handles it.

**Root cause — step 1:** The `@ModelAttribute` method runs before the controller method and writes directly to the servlet response:

```java
// GenericControllerAdvice.java lines 79–84
@ModelAttribute
private void responseHeaders(HttpServletRequest request, HttpServletResponse response) {
    var requestMapping = String.valueOf(request.getAttribute(HandlerMapping.BEST_MATCHING_PATTERN_ATTRIBUTE));
    var responseHeaders = properties.getResponse().getHeaders().getHeadersForPath(requestMapping);
    responseHeaders.forEach(response::setHeader);   // sets Cache-Control: public, max-age=5 on the raw HttpServletResponse
}
```

For the `/api/v1/topics/{id}` path this writes `Cache-Control: public, max-age=5` to the underlying `HttpServletResponse` object. This is confirmed by the test assertion at `TopicControllerTest.java` line 94.

**Root cause — step 2:** The `notFound` handler passes `null` as the `HttpHeaders` argument:

```java
// GenericControllerAdvice.java lines 115–118
@ExceptionHandler
private ResponseEntity<Object> notFound(final EntityNotFoundException e, final WebRequest request) {
    return handleExceptionInternal(e, null, null, NOT_FOUND, request);
}
```

`handleExceptionInternal` (lines 139–157) constructs `new ResponseEntity<>(errorResponse, headers, statusCode)` where `headers` is `null`. Spring MVC's `HttpEntityMethodProcessor` merges the `ResponseEntity` headers on top of the already-committed servlet response headers — it does not clear them. The `Cache-Control: public, max-age=5` header set by `@ModelAttribute` therefore survives and is emitted with the 404 body.

**Why existing checks are insufficient:** There is no `Cache-Control: no-store` or `Cache-Control: private` override applied in any error-handling path. The `responseCacheUpdateHandler` in the Node.js REST module explicitly skips non-2xx responses (`httpStatusCodes.isSuccess(res.statusCode)`), but that is a different module; the rest-java Spring service has no equivalent guard.

**Exploit flow:**
1. Attacker monitors the Hedera network (public ledger) and observes a `TopicCreate` transaction for topic `0.0.99999`.
2. Before the mirror node ingests the transaction (ingestion lag is typically 1–10 seconds), the attacker sends `GET /api/v1/topics/0.0.99999` through a shared reverse proxy or CDN.
3. The mirror node returns HTTP 404 with `Cache-Control: public, max-age=5`.
4. The proxy stores the 404 response per RFC 7234 §3 (explicit freshness on a cacheable status code).
5. The mirror node ingests the topic. Legitimate users behind the same proxy now receive the cached 404 for up to 5 seconds, making the topic appear non-existent.

### Impact Explanation
All users sharing the intermediate proxy are denied access to a valid topic for up to 5 seconds per poisoning event. Because the attacker can repeat the attack immediately after each cache expiry (every 5 seconds), the disruption can be sustained indefinitely at negligible cost. No authentication or special privilege is required. The impact is griefing / denial-of-information with no direct economic loss, consistent with the Medium severity classification.

### Likelihood Explanation
The preconditions are trivially met: the Hedera ledger is public, topic creation events are observable in real time, and mirror node ingestion lag is a documented and expected property of the system. Any unauthenticated HTTP client can trigger the attack. Proxies and CDNs that honour `Cache-Control: public` for 4xx responses (e.g., Nginx, Varnish, Cloudflare in default configurations) will cache the response automatically.

### Recommendation
Override the `Cache-Control` header specifically for error responses. The cleanest fix is to add `Cache-Control: no-store` inside `handleExceptionInternal` before constructing the `ResponseEntity`:

```java
@Override
protected ResponseEntity<Object> handleExceptionInternal(
        Exception ex, Object body, HttpHeaders headers, HttpStatusCode statusCode, WebRequest request) {
    HttpHeaders responseHeaders = headers != null ? new HttpHeaders(headers) : new HttpHeaders();
    if (!statusCode.is2xxSuccessful()) {
        responseHeaders.setCacheControl(CacheControl.noStore());
    }
    // ... existing body-building logic ...
    return new ResponseEntity<>(errorResponse, responseHeaders, statusCode);
}
```

Alternatively, set `Cache-Control: no-store` in the `notFound`, `badRequest`, `queryTimeout`, and `defaultExceptionHandler` methods individually, or configure a servlet filter that rewrites the `Cache-Control` header for all non-2xx responses before they leave the server.

    protected ResponseEntity<Object> handleExceptionInternal(
            Exception ex, @Nullable Object body, HttpHeaders headers, HttpStatusCode statusCode, WebRequest request) {

        Error errorResponse =
                switch (ex) {
                    case BindException e -> bindExceptionResponse(e);
                    case Errors errors -> errorResponse(errors.getAllErrors());
                    case MethodValidationResult errors -> errorResponse(errors.getAllErrors());
                    default -> {
                        var message =
                                statusCode instanceof HttpStatus hs ? hs.getReasonPhrase() : statusCode.toString();
                        var detail = body instanceof ProblemDetail pb ? pb.getDetail() : ex.getMessage();
                        var nonSensitiveDetail = !statusCode.is5xxServerError() ? detail : StringUtils.EMPTY;
                        yield errorResponse(message, nonSensitiveDetail);
                    }
                };

        request.setAttribute(WebUtils.ERROR_EXCEPTION_ATTRIBUTE, ex, SCOPE_REQUEST);
        return new ResponseEntity<>(errorResponse, headers, statusCode);
