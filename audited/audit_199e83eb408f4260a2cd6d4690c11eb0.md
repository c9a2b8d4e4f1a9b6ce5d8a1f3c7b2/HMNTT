### Title
Unsanitized `X-Forwarded-For` Header Logged as Client IP, Producing Non-Parseable IP Records in Mirror Node Exports

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` blindly returns the raw value of the `X-Forwarded-For` header without parsing or validating it. An unprivileged external user can supply a comma-separated list of IPs (e.g., `1.2.3.4, 5.6.7.8`), causing the entire unsanitized string to be logged as the client IP address. This produces structurally invalid IP records in the tracing log, corrupting any downstream mirror node export or audit pipeline that expects a single valid IP.

### Finding Description
**Exact code path:**
`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 — raw, unsanitized
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }

    return ipAddress  // returned verbatim, no IP validation
}
```

`r.Header.Get()` returns the full raw header value. The `X-Forwarded-For` header is a de-facto standard that carries a comma-separated list of IPs appended by each proxy in the chain (e.g., `1.2.3.4, 5.6.7.8`). The code performs **only an emptiness check** (`len(ipAddress) == 0`) and returns the entire string verbatim.

This value is then interpolated directly into the log message at line 52–53:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

**Root cause / failed assumption:** The code assumes `X-Forwarded-For` always contains a single IP. This assumption is wrong by design — the header is multi-valued by the HTTP standard, and any attacker (or legitimate proxy chain) can produce a comma-separated value.

**Why existing checks fail:** The sole guard is `len(ipAddress) == 0`. It does not:
- Split on commas to extract the first (client-originating) IP
- Validate the result with `net.ParseIP()`
- Sanitize or reject non-IP characters

The test suite (`trace_test.go`, line 56–58) only tests a single clean IP in `X-Forwarded-For`, never a comma-separated list, so this path is untested and unguarded.

### Impact Explanation
Every request with a multi-value `X-Forwarded-For` header causes a structurally invalid string (e.g., `1.2.3.4, 5.6.7.8`) to be written into the tracing log as the client IP field. Any mirror node export pipeline, log aggregator, or SIEM that parses this field as a valid IP address will either fail to parse the record, silently drop it, or store a corrupt entry — directly matching the "incorrect or missing records exported to mirror nodes" scope. Additionally, because the string is user-controlled and unsanitized, an attacker can inject arbitrary content (including log-format metacharacters or newlines) into the log stream, enabling log injection attacks.

### Likelihood Explanation
No authentication or special privilege is required. Any HTTP client reachable by the Rosetta API endpoint can set arbitrary request headers. The `X-Forwarded-For` header is not stripped or normalized by the application itself. The attack is trivially repeatable with a single `curl` command and requires no knowledge of the system beyond the public API surface.

### Recommendation
Parse the `X-Forwarded-For` value and extract only the **first** IP (the original client), then validate it with `net.ParseIP()` before use:

```go
if len(ipAddress) == 0 {
    raw := r.Header.Get(xForwardedForHeader)
    if len(raw) > 0 {
        // X-Forwarded-For may be "client, proxy1, proxy2"
        first := strings.SplitN(raw, ",", 2)[0]
        ipAddress = strings.TrimSpace(first)
        if net.ParseIP(ipAddress) == nil {
            ipAddress = "" // reject non-IP values
        }
    }
}
```

Apply the same treatment to `X-Real-IP`. Fall back to `r.RemoteAddr` (already parsed via `net.SplitHostPort`) when the header value is absent or invalid.

### Proof of Concept
```bash
# Send a request with a comma-separated X-Forwarded-For header
curl -H "X-Forwarded-For: 1.2.3.4, 5.6.7.8" \
     http://<rosetta-host>/network/list

# Resulting log entry (observed):
# time="..." level=info msg="1.2.3.4, 5.6.7.8 GET /network/list (200) in 3ms"
#                            ^^^^^^^^^^^^^^^^^ — entire string, not a valid IP

# With log injection via newline:
curl -H $'X-Forwarded-For: 1.2.3.4\ntime="2024-01-01" level=info msg="INJECTED"' \
     http://<rosetta-host>/network/list
# Injects a fake log line into the log stream
