### Title
Gas Throttle Bypass via Out-of-Gas Exhaustion: `contracts.throttle.throttleByGas=false` + Full Gas Restoration on Exceptional Halt

### Summary
`buildTransactionProperties()` hardcodes `contracts.throttle.throttleByGas=false` and `executor.disableThrottles=true`, removing all consensus-node-library-level gas throttling. The mirror node's own gas bucket throttle is then fully neutralized for out-of-gas transactions because `restoreGasToBucket()` unconditionally returns 100% of the gas limit to the bucket whenever `gasUsed == gasLimit` (exceptional halt). An unprivileged attacker can submit a continuous stream of max-gas (15,000,000) transactions containing a gas-exhausting loop, forcing the node to perform maximum EVM computation on every request while the gas bucket is always fully replenished, leaving only the 500 req/s rate limit as the sole defense.

### Finding Description

**Code locations:**

`EvmProperties.java` `buildTransactionProperties()` lines 159–162 hardcodes:
```
contracts.throttle.throttleByOpsDuration = false
contracts.throttle.throttleByGas         = false   ← disables consensus-node gas throttle
executor.disableThrottles                = true    ← disables all executor throttles
``` [1](#0-0) 

`maxGasLimit` defaults to 15,000,000 with no lower bound enforced on user input beyond `@Min(21_000)`: [2](#0-1) 

`ContractController.call()` calls `throttleManager.throttle(request)` (consuming gas tokens) then immediately proceeds to EVM execution with no further gas-based guard: [3](#0-2) 

`ThrottleManagerImpl.throttle()` consumes `scaleGas(gas)` tokens from the bucket before execution. `scaleGas(15_000_000) = 15_000_000 / 10_000 = 1_500` tokens. The bucket capacity is `scaleGas(7_500_000_000) = 750_000` tokens: [4](#0-3) [5](#0-4) 

**The critical flaw — `restoreGasToBucket()` in `ContractCallService.java` lines 140–151:**

```java
final var gasLimitToRestoreBaseline = (long)(gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
// gasLimitRefundPercent defaults to 100 → gasLimitToRestoreBaseline = gasLimit = 15,000,000

if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);  // ← restores 100% of gas limit
``` [6](#0-5) 

When a gas-exhausting loop causes an **exceptional halt** (out-of-gas), the EVM sets `gasUsed = gasLimit`. The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` evaluates to `true`, and the full `gasLimitToRestoreBaseline` (= 15,000,000 with default 100% refund) is returned to the bucket. Net gas bucket consumption per request: **zero tokens**.

The `restore()` call in `ThrottleManagerImpl` simply calls `gasLimitBucket.addTokens(tokens)` with no cap check beyond bucket4j's internal maximum: [7](#0-6) 

**Root cause:** The failed assumption is that out-of-gas transactions represent "wasted" work that should be refunded to the throttle bucket. In reality, an out-of-gas transaction is the *most* expensive case — the EVM ran the full gas budget — so restoring 100% of the gas limit eliminates the gas throttle's protective effect entirely for this class of transactions.

### Impact Explanation

With `contracts.throttle.throttleByGas=false` and `executor.disableThrottles=true`, the consensus node library performs no gas-based gating. The mirror node's own gas bucket is the only gas-level defense, but it is fully bypassed for out-of-gas transactions. The attacker forces the node to execute up to 500 × 15,000,000 = **7.5 billion gas units of EVM computation per second** (bounded only by the 500 req/s rate limit), causing CPU exhaustion, thread pool saturation, and denial of service to legitimate users. No funds, no privileged account, and no special setup are required.

### Likelihood Explanation

Any anonymous HTTP client can reach `POST /api/v1/contracts/call`. The attack requires only a single crafted request body with `gas=15000000` and bytecode for an infinite loop (e.g., `JUMPDEST JUMP` — two opcodes). It is trivially repeatable at 500 req/s from a single machine, and can be amplified across multiple source IPs to saturate even a rate-limited deployment. No authentication, no on-chain funds, and no prior knowledge of the system beyond the public API spec are needed.

### Recommendation

1. **Do not restore gas for out-of-gas (exceptional halt) transactions.** The condition on line 144 of `ContractCallService.java` should treat `gasUsed == gasLimit` as a fully consumed slot, not a refundable one. Remove or invert the restoration for this case:
   ```java
   if (result == null) {
       // null result: restore baseline
       throttleManager.restore(gasLimitToRestoreBaseline);
   } else if (!result.isSuccessful() && gasLimit == result.gasUsed()) {
       // Exceptional halt (out-of-gas): do NOT restore — full gas was consumed
   } else {
       final var gasRemaining = gasLimit - result.gasUsed();
       throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
   }
   ```
2. **Lower `gasLimitRefundPercent` default** from 100 to a value that reflects realistic unused-gas refunds (e.g., 20–50%).
3. **Re-evaluate `contracts.throttle.throttleByGas=false`**: if the consensus node library's gas throttle can be enabled without breaking simulation semantics, it provides defense-in-depth.
4. **Add per-IP rate limiting** at the ingress layer to limit amplification.

### Proof of Concept

**Preconditions:** Mirror node running with default configuration (`maxGasLimit=15_000_000`, `gasLimitRefundPercent=100`, `requestsPerSecond=500`).

**Step 1 — Deploy or target any contract address (even a non-existent one triggers EVM execution with the supplied bytecode via `data`).**

**Step 2 — Craft a gas-exhausting payload.** The minimal infinite-loop bytecode is `0x5b600056` (`JUMPDEST PUSH1 0x00 JUMP`). This loops until all gas is consumed.

**Step 3 — Send the request:**
```bash
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{
    "data": "0x5b600056",
    "gas": 15000000
  }'
```

**Step 4 — Observe:** The response is a revert/out-of-gas error. The gas bucket is fully restored (net 0 tokens consumed).

**Step 5 — Repeat at 500 req/s** (the rate limit). Each request forces 15,000,000 gas of EVM computation. The gas throttle never depletes. Node CPU climbs to saturation; legitimate requests time out.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
