### Title
Cache Stampede via Unprotected `contractState` TTL Expiry Enabling DB Connection Pool Exhaustion

### Summary
The `contractState` Caffeine cache is configured with `expireAfterWrite=2s,maximumSize=25000` but uses Spring's `@Cacheable` abstraction without any thundering-herd / single-flight protection. When the TTL expires, up to 500 concurrent requests (the full burst capacity of the rate-limit token bucket) simultaneously miss the cache and each independently issue a database query, exhausting the HikariCP connection pool and causing service-wide degradation.

### Finding Description

**Cache configuration** — `CacheProperties.java` line 28 sets `contractState = "expireAfterWrite=2s,maximumSize=25000,recordStats"`. The bean is wired in `EvmConfiguration.java` lines 83–89 via `CaffeineCacheManager.setCacheSpecification()`, which creates a plain Caffeine cache, not a `LoadingCache`. Spring's `@Cacheable` proxy is the only consumer.

**Cache access without stampede protection** — `ContractStateServiceImpl.findStorage()` (lines 58–70) calls `contractStateCache.get(key, byte[].class)`. On a miss it falls through to `findStorageBatch()` (lines 85–122), which unconditionally calls `contractStateRepository.findStorageBatch()` — a raw JPA query with no synchronization, no lock, and no single-flight guard. Multiple threads that simultaneously observe a cache miss all execute the DB query independently.

The secondary path, `ContractStateRepository.findStorage()` (lines 19–21), is annotated `@Cacheable` via Spring AOP. Spring's `@Cacheable` does **not** implement single-flight loading: concurrent threads that all see an expired/absent entry all invoke the underlying method before any of them can repopulate the cache.

**Throttle does not prevent the stampede** — `ThrottleConfiguration.java` lines 24–32 creates a token bucket with `capacity(500).refillGreedy(500, Duration.ofSeconds(1))`. This means an attacker can legally burst **500 requests simultaneously** in a single instant. The gas bucket (`gasPerSecond = 7_500_000_000`, scaled by 10 000 = 750 000 tokens; max gas per request = 15 000 000 → 1 500 scaled tokens) also permits ~500 max-gas requests per burst. Neither bucket prevents concurrent cache misses.

**Exploit flow:**
1. Attacker sends ≤500 `eth_call` requests that each read distinct `(contractId, slot)` pairs, populating the `contractState` cache.
2. Attacker waits exactly 2 seconds for `expireAfterWrite` to expire all entries.
3. Attacker immediately fires another burst of 500 requests for the same keys. All 500 are admitted by the token bucket (full capacity available after the 2-second wait).
4. All 500 threads call `contractStateCache.get()`, all observe a miss (entries expired), and all enter `findStorageBatch()` concurrently.
5. Each thread issues a `findStorageBatch` query with up to 1 500 slots in an `IN (...)` clause (bounded by `slotsPerContract = maximumSize=1500`).
6. 500 concurrent expensive queries hit the database. HikariCP's default pool (10 connections; pgbouncer `max_user_connections: 250` in production) is saturated; queries queue, the 3-second `statementTimeout` fires, and HTTP 500 / timeout responses are returned to all callers.
7. The attacker repeats every 2 seconds.

### Impact Explanation
Exhausting the HikariCP connection pool blocks **all** database-dependent operations in the web3 service, not just contract-state reads. Every in-flight request that needs a DB connection will time out or fail. This constitutes a non-network-based application-layer DoS. The 3-second `statementTimeout` means each stampede wave produces a 3-second outage window, repeatable indefinitely at 2-second intervals. The `mirror_web3` pgbouncer user is capped at 250 server connections; 500 concurrent queries from a single pod will saturate that limit and spill into connection-wait queues, amplifying latency for all users.

### Likelihood Explanation
No authentication or special privilege is required — any caller of the public `eth_call` / `eth_estimateGas` JSON-RPC endpoints can trigger this. The attacker needs only: (a) knowledge of any 500 distinct contract storage slots (trivially obtained by replaying on-chain transactions or reading public block explorers), and (b) the ability to send 500 HTTP requests in a tight burst, which is achievable from a single machine. The 2-second TTL makes the attack trivially repeatable and automatable. The token bucket refills greedily, so the full 500-token burst is available again after each 1-second refill cycle, meaning the attacker does not even need to wait the full 2 seconds between waves.

### Recommendation
1. **Replace `expireAfterWrite` with Caffeine's `refreshAfterWrite` + `LoadingCache`**: Caffeine's `LoadingCache` with `refreshAfterWrite` performs async background refresh and returns the stale value to concurrent callers during reload, eliminating the stampede entirely.
2. **Alternatively, use Caffeine's built-in single-flight loading**: Replace `CaffeineCacheManager.setCacheSpecification()` with a `Caffeine.newBuilder().build(loader)` `LoadingCache` so that only one thread executes the DB query per key while others wait for the result.
3. **Reduce burst capacity relative to TTL**: The token bucket capacity equals the per-second rate (500). Reducing burst capacity (e.g., `capacity(50).refillGreedy(500, Duration.ofSeconds(1))`) limits the number of simultaneous cache misses per TTL window.
4. **Add a DB-level semaphore or connection-pool guard**: Limit the number of concurrent `findStorageBatch` calls (e.g., via a `Semaphore` in `ContractStateServiceImpl`) to a value well below the HikariCP pool size.

        if (!cacheProperties.isEnableBatchContractSlotCaching()) {

        }

        final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

        if (cachedValue != null && cachedValue != EMPTY_VALUE) {
            return Optional.of(cachedValue);

        return findStorageBatch(contractId, key);

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
