### Title
Silent HTTP 200 Empty Response for Multiple `timestamp=eq` Values in `/api/v1/transactions`

### Summary
An unprivileged external user can supply two distinct `timestamp=eq:X` query parameters to the `/api/v1/transactions` endpoint, causing `getTransactionTimestamps()` to silently return `{rows: []}` and the API to respond with HTTP 200 and an empty transaction list. This is indistinguishable from a legitimate "no transactions found" result, masking data availability and potentially deceiving automated monitoring or audit systems.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactions` calls `parseTimestampFilters` with `strictCheckOverride=false` and `validateRange=false`:

```js
// rest/transactions.js line 674
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [1](#0-0) 

In `rest/utils.js`, `parseTimestampFilters` computes `forceStrictChecks = false || config.strictTimestampParam`. Even when `config.strictTimestampParam` is `true` (the documented default), the strict-check block only guards against:
- `neValues.size > 0` (when `!allowNe`)
- `lowerBoundFilterCount > 1`
- `upperBoundFilterCount > 1`
- `eqValues.size > 0` combined with range operators

There is **no check for `eqValues.size > 1`** in isolation: [2](#0-1) 

Two distinct `timestamp=eq:A` and `timestamp=eq:B` values are collected into a `Set`, deduplicated only if identical, and returned as `eqValues: [A, B]` with no error thrown: [3](#0-2) [4](#0-3) 

Then in `getTransactionTimestamps`:

```js
// rest/transactions.js line 452
if (timestampRange.eqValues.length > 1 || timestampRange.range?.isEmpty()) {
  return {rows: []};
}
``` [5](#0-4) 

This silently short-circuits with `{rows: []}`. `doGetTransactions` then returns `{transactions: [], links: {next: null}}` with HTTP 200: [6](#0-5) 

**Root cause:** `parseTimestampFilters` never rejects `eqValues.size > 1` even under strict mode, and `getTransactionTimestamps` silently discards the query instead of returning a 400 error.

### Impact Explanation
Any caller (monitoring system, audit tool, client application) that queries `GET /api/v1/transactions?timestamp=eq:T1&timestamp=eq:T2` receives HTTP 200 `{"transactions":[],"links":{"next":null}}`. This response is byte-for-byte identical to a legitimate "no transactions exist" response. Automated systems relying on this API for transaction completeness checks, reconciliation, or alerting cannot distinguish between "no data" and "query was silently dropped." This can mask data availability issues or be used to suppress expected transaction results in audit trails.

### Likelihood Explanation
Exploitation requires zero authentication, zero privileges, and zero special knowledge beyond the public API. The attacker simply repeats the `timestamp` query parameter with two different values. It is trivially reproducible with a single HTTP request and works consistently regardless of server state. Any external user of the public mirror node REST API can trigger this.

### Recommendation
In `getTransactionTimestamps`, replace the silent early return with an explicit `InvalidArgumentError` (HTTP 400):

```js
if (timestampRange.eqValues.length > 1) {
  throw new InvalidArgumentError('Only one timestamp eq value is supported');
}
if (timestampRange.range?.isEmpty()) {
  return {rows: []};
}
```

Alternatively, add a `eqValues.size > 1` check inside the `forceStrictChecks` block in `parseTimestampFilters` in `rest/utils.js` so the error is raised uniformly before reaching the handler. [2](#0-1) 

### Proof of Concept

```
GET /api/v1/transactions?timestamp=eq:1638921702000000000&timestamp=eq:1638921703000000000
```

**Expected (correct) behavior:** HTTP 400 with an error message about invalid parameters.

**Actual behavior:** HTTP 200 with body:
```json
{
  "transactions": [],
  "links": { "next": null }
}
```

This is identical to the response when no transactions exist in the queried range, making it impossible for the caller to detect that the query was silently discarded.
