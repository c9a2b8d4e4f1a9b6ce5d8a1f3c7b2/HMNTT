### Title
Race Window in `RedisTopicListener.subscribe()` Causes Permanent Message Loss via `.share()` Refcount Drop

### Summary
In `RedisTopicListener.subscribe()`, the `.share()` operator cancels the upstream Redis subscription when the last downstream subscriber cancels. The `doOnCancel` callback immediately removes the topic entry from `topicMessages`, creating a race window during which any Redis messages published before the next `computeIfAbsent` call are permanently lost. Any unauthenticated gRPC client can trigger this by being (or becoming) the sole subscriber to a topic and then cancelling their stream.

### Finding Description

**Exact code path:**

`RedisTopicListener.java`, `subscribe()` lines 68–80 and `getSharedListener()` lines 59–62:

```
getSharedListener()  →  computeIfAbsent(key, k -> subscribe(topic))   [line 61]
subscribe()          →  ...doOnCancel(() -> unsubscribe(topic))        [line 74]
                         .retryWhen(...)
                         .share()                                       [line 79]
unsubscribe()        →  topicMessages.remove(topic.getTopic())         [line 83]
```

**Root cause:**

`Flux.share()` is equivalent to `publish().refCount()`. When the downstream subscriber count drops to zero, `refCount` cancels the upstream. That cancellation propagates back through `retryWhen` to `doOnCancel`, which calls `unsubscribe(topic)` and removes the entry from `topicMessages`. Between that removal and the next `computeIfAbsent` call (triggered only when a new subscriber arrives), there is no active Redis subscription. Redis pub/sub is fire-and-forget — messages published during this gap are never buffered and are permanently lost.

**Failed assumption:**

The code assumes that `computeIfAbsent` will always produce a live entry before any relevant Redis messages arrive. This is false: the entry is eagerly removed on cancel, but re-creation is lazy (only on next subscriber), leaving an unbounded gap.

**Why existing checks fail:**

- `ConcurrentHashMap` prevents concurrent map corruption but does not close the TOCTOU window between `remove()` and the next `computeIfAbsent`.
- The one-shot safety check in `TopicMessageServiceImpl` (line 67–70) fires once after 1 second for the *next* subscriber's context, but only if that subscriber exists. Messages lost during the gap before any new subscriber arrives are never recovered.
- `retryWhen` on line 78 retries on *errors*, not on cancellation, so it provides no protection here.

**Exploit flow:**

1. Attacker opens a gRPC `subscribeTopic` stream to a low-traffic topic (no auth required — `ConsensusController.subscribeTopic` has no authentication gate).
2. Attacker waits until they are the sole subscriber (or opens the only subscription to a rarely-subscribed topic).
3. Attacker cancels the gRPC stream. `ConsensusController` line 51 calls `disposable.dispose()`, which cancels the Flux chain.
4. `.share()` sees refcount → 0, cancels upstream → `doOnCancel` fires → `topicMessages.remove(topic)`.
5. Redis publishes one or more topic messages during the gap.
6. A legitimate subscriber arrives; `computeIfAbsent` creates a new subscription, but the messages from step 5 are gone from Redis and are not delivered.

### Impact Explanation

Topic message subscribers receive an incomplete stream — sequence numbers are silently skipped. For HCS (Hedera Consensus Service) consumers relying on the mirror node for ordered, complete message history, this constitutes data integrity loss. The mirror node's core contract is to faithfully replay all consensus messages; missing records break downstream applications that depend on sequence continuity (e.g., token bridges, audit logs, smart-contract event consumers).

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. Any external client can subscribe and cancel at will. For topics with zero or one concurrent subscriber — common for application-specific or low-traffic topics — the attacker trivially becomes the last subscriber. The gap duration is bounded only by network/scheduler latency (microseconds to milliseconds), which is sufficient for Redis to deliver messages. The attack is repeatable: the attacker can re-subscribe and cancel in a loop to extend the gap indefinitely.

### Recommendation

Replace the eager `doOnCancel`/`doOnComplete` removal with a delayed or reference-counted cleanup:

1. **Use `refCount` with a grace period**: Replace `.share()` with `.publish().refCount(1, Duration.ofSeconds(N))`. This keeps the upstream subscription alive for N seconds after the last subscriber cancels, eliminating the gap for transient subscriber churn.
2. **Remove the eager `unsubscribe` side-effect**: Do not call `topicMessages.remove()` inside `doOnCancel`/`doOnComplete`. Instead, let the shared Flux manage its own lifecycle and only remove the map entry when the upstream terminates with an error or natural completion, not on downstream cancellation.
3. **Persistent subscription**: Keep the Redis subscription alive as long as the topic is known to be active, independent of downstream subscriber count.

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
