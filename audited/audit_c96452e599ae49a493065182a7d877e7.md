### Title
Unparameterized IN Clause in `getInClauseSubQuery()` Enables Query-Plan Cache Pollution

### Summary
`getInClauseSubQuery()` in `rest/service/tokenService.js` directly interpolates validated numeric token IDs into the SQL string instead of using PostgreSQL bind parameters. Because every unique combination of token IDs produces a structurally distinct query string, PostgreSQL cannot reuse a cached query plan across requests. Any unauthenticated caller of the `/accounts/:id/tokens` endpoint can repeatedly submit requests with different sets of valid `token.id=eq:X` values, forcing repeated query parse-and-plan cycles and polluting the DB's plan cache.

### Finding Description
**Exact location:** `rest/service/tokenService.js`, `getInClauseSubQuery()`, line 85.

```js
// rest/service/tokenService.js  lines 78-87
getInClauseSubQuery(inConditions, params) {
  const tokenIdInParams = [];
  inConditions.forEach((condition) => {
    tokenIdInParams.push(condition.value);
  });

  if (!isEmpty(tokenIdInParams)) {
    return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${tokenIdInParams})`;
    //                                                                    ^^^^^^^^^^^^^^^^
    //  values are string-interpolated, NOT pushed into `params`
  }
}
```

The `params` array is passed in but never used for the IN-clause values. The result is a query like:

```sql
... and ta.token_id in (1000,2000,3000)
```

**Call chain:**
1. `GET /api/v1/accounts/:id/tokens?token.id=eq:A&token.id=eq:B&…` → `tokenController.getTokenRelationships` [1](#0-0) 
2. `utils.buildAndValidateFilters` validates each value as a numeric entity ID and encodes it. [2](#0-1) 
3. `extractTokensRelationshipQuery` pushes every `eq` filter into `inConditions`. [3](#0-2) 
4. `getTokenRelationshipsQuery` calls `getInClauseSubQuery(inConditions, params)`. [4](#0-3) 
5. `getInClauseSubQuery` interpolates the values directly into the SQL string. [5](#0-4) 

**Why existing checks are insufficient:**
`buildFilters` enforces `maxRepeatedQueryParameters` per key, rejecting arrays that exceed the configured cap. [6](#0-5)  This limits the *width* of a single IN clause but does not prevent the unparameterized interpolation itself. Each request with a different set of token IDs (even within the cap) produces a unique SQL string, so PostgreSQL must parse and plan it from scratch every time. The `params` array that is passed to `getInClauseSubQuery` and could have been used for bind parameters is simply ignored. [7](#0-6) 

### Impact Explanation
- PostgreSQL's shared plan cache is polluted with a large number of single-use query plans, one per unique token-ID combination.
- Each request forces a full query parse + plan cycle instead of a cheap plan-cache hit.
- Under sustained concurrent load this increases DB CPU and memory pressure with no benefit to legitimate users.
- No SQL injection is possible because values are pre-validated as numeric entity IDs before interpolation.

### Likelihood Explanation
The endpoint `/api/v1/accounts/:id/tokens` requires no authentication. Any external actor can craft HTTP requests with up to `maxRepeatedQueryParameters` distinct `token.id=eq:X` values (all valid token IDs), vary the set across requests, and repeat at will. No special knowledge or privilege is required. The attack is trivially scriptable and repeatable.

### Recommendation
Push the IN-clause values into the existing `params` array and use a `= any($N)` parameterized form (consistent with how the same pattern is handled in `tokens.js`):

```js
// rest/tokens.js lines 204-205 — the correct pattern already used elsewhere
if (inClause.length > 0) {
  conditions.push(`${sqlQueryColumns.TOKEN_ID} = any($${params.push(inClause)})`);
}
```

Apply the same fix in `getInClauseSubQuery`:

```js
getInClauseSubQuery(inConditions, params) {
  const tokenIdInParams = inConditions.map((c) => c.value);
  if (tokenIdInParams.length > 0) {
    return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} = any($${params.push(tokenIdInParams)})`;
  }
}
```

This produces a single bind parameter containing the array, allowing PostgreSQL to cache and reuse the query plan regardless of which token IDs are supplied.

### Proof of Concept
```bash
# Assume account 0.0.98 exists; supply maxRepeatedQueryParameters valid token IDs
BASE="http://localhost:5551/api/v1/accounts/0.0.98/tokens"
PARAMS="token.id=eq:1000&token.id=eq:1001&token.id=eq:1002&token.id=eq:1003"

# Each request with a different combination forces a new query parse+plan cycle
for i in $(seq 1 1000); do
  curl -s "${BASE}?token.id=eq:$((RANDOM % 9000 + 1000))&token.id=eq:$((RANDOM % 9000 + 1000))&token.id=eq:$((RANDOM % 9000 + 1000))" &
done
wait

# Observe pg_stat_statements: high calls count with near-zero shared_blks_hit
# and elevated total_plan_time for the token_account IN-clause queries
psql -c "SELECT query, calls, total_plan_time FROM pg_stat_statements WHERE query LIKE '%token_id in%' ORDER BY calls DESC LIMIT 10;"
```
