### Title
`ConstructionParse` Returns Unverified `AccountIdentifierSigners` Derived Solely from Transaction Body Fields

### Summary
`ConstructionParse()` in `rosetta/app/services/construction_service.go` populates `AccountIdentifierSigners` from accounts extracted out of the raw transaction body (payer `TransactionID.AccountID` and negative-amount transfer accounts) without ever verifying that the signatures in the transaction's `sigMap` actually correspond to those accounts. Any unauthenticated caller can craft a transaction naming an arbitrary victim account as payer/sender, submit it with `Signed=true`, and receive that victim account back as a verified signer — with zero cryptographic proof.

### Finding Description

**Exact code path:**

`ConstructionParse()` (lines 183–214) decodes the hex transaction, then delegates to `c.transactionHandler.Parse()` to obtain `accounts`, and unconditionally promotes them to `AccountIdentifierSigners` when `request.Signed == true`:

```go
// construction_service.go:198-211
operations, accounts, err := c.transactionHandler.Parse(ctx, transaction)
...
signers := make([]*rTypes.AccountIdentifier, 0, len(accounts))
if request.Signed {
    for _, account := range accounts {
        signers = append(signers, account.ToRosetta())
    }
}
return &rTypes.ConstructionParseResponse{
    AccountIdentifierSigners: signers, ...
}, nil
``` [1](#0-0) 

**How `accounts` is derived — `CryptoTransfer`:**

`cryptoTransferTransactionConstructor.Parse()` reads `GetHbarTransfers()` from the transaction body and classifies any account with a negative amount as a signer. No `sigMap` inspection occurs:

```go
// crypto_transfer_transaction_constructor.go:68-87
hbarTransferMap := transferTransaction.GetHbarTransfers()
...
for _, operation := range operations {
    if operation.Amount.GetValue() < 0 {
        senderMap[operation.AccountId.String()] = operation.AccountId
    }
}
return operations, senderMap.toSenders(), nil
``` [2](#0-1) 

**How `accounts` is derived — `CryptoCreate`:**

`cryptoCreateTransactionConstructor.Parse()` reads the payer directly from `GetTransactionID().AccountID` in the transaction body:

```go
// crypto_create_transaction_constructor.go:79
payer, err := types.NewAccountIdFromSdkAccountId(*cryptoCreateTransaction.GetTransactionID().AccountID)
...
return types.OperationSlice{operation}, []types.AccountId{payer}, nil
``` [3](#0-2) 

**Root cause:** The `Signed` flag in the request is used only as a gate to decide whether to emit signers — it never triggers any cryptographic check. The `sigMap` embedded in the serialised transaction is completely ignored by the parse path.

**Contrast with `ConstructionCombine`**, which *does* verify signatures before attaching them:

```go
// construction_service.go:82-84
if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {
    return nil, errors.ErrInvalidSignatureVerification
}
``` [4](#0-3) 

That check exists only in `ConstructionCombine`, not in `ConstructionParse`.

### Impact Explanation
Any Rosetta client (exchange, block explorer, reconciliation tool) that calls `/construction/parse` with `signed=true` and trusts `account_identifier_signers` to confirm "who authorised this transaction" can be fed a completely fabricated signer list. An attacker can make it appear that an arbitrary account — including high-value or well-known accounts — signed a transaction they never touched. Downstream systems that use this response for audit trails, compliance checks, or pre-broadcast signer verification will record falsified attribution. The Rosetta specification explicitly states that `account_identifier_signers` for a signed transaction must reflect accounts that have actually signed; this implementation violates that contract.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable (standard Rosetta HTTP API). Crafting a valid-looking `TransferTransaction` or `AccountCreateTransaction` protobuf with an arbitrary `TransactionID.AccountID` requires only the Hiero SDK or raw protobuf construction — no private keys, no privileged access, no on-chain interaction. The attack is trivially repeatable and requires no special timing or race conditions.

### Recommendation
Inside `ConstructionParse`, when `request.Signed == true`, extract the `sigMap` from the deserialised transaction and verify that each claimed signer account has a corresponding valid Ed25519 signature over the frozen transaction body bytes — mirroring the logic already present in `ConstructionCombine`. Only accounts whose signatures pass `ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, sig)` should be included in `AccountIdentifierSigners`. Accounts present in the transaction body but lacking a valid signature must be excluded.

### Proof of Concept
1. Using the Hiero Go SDK, construct a `TransferTransaction` that debits `0.0.VICTIM` by 1 tinybar and credits any other account by 1 tinybar. Set `TransactionID` with `AccountID = 0.0.VICTIM`. Do **not** sign with VICTIM's key; sign (or leave unsigned) with any other key.
2. Serialise to bytes and hex-encode: `hex.EncodeToString(txBytes)`.
3. POST to the mirror node's Rosetta endpoint:
   ```json
   POST /construction/parse
   {
     "network_identifier": {"blockchain":"Hiero","network":"mainnet"},
     "signed": true,
     "transaction": "<hex from step 2>"
   }
   ```
4. Observe the response:
   ```json
   {
     "account_identifier_signers": [{"address": "0.0.VICTIM"}],
     ...
   }
   ```
   VICTIM's account is returned as a confirmed signer despite never having signed the transaction.
