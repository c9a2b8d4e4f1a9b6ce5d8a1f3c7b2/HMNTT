### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent `GET /blocks/0` Requests

### Summary
The REST API's `/blocks/:hashOrNumber` endpoint accepts block number `0` as a valid input (via `isPositiveLong` with `allowZero=true`) and issues an uncached database query for every request with no application-level rate limiting. The default database connection pool is capped at 10 connections. An unprivileged attacker flooding this endpoint with concurrent requests can exhaust the pool, causing all REST API endpoints that require database access to queue indefinitely or time out, resulting in a service-wide outage.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:13` registers `GET /:hashOrNumber` → `BlockController.getByHashOrNumber` [1](#0-0) 

`rest/controllers/blockController.js:28-35` — `validateHashOrNumber` calls `utils.isPositiveLong(hashOrNumber, true)`, which explicitly allows `0` as a valid block number. [2](#0-1) 

`rest/controllers/blockController.js:114-124` — `getByHashOrNumber` calls `RecordFileService.getByHashOrNumber(hash, number)` with no caching, deduplication, or concurrency guard. [3](#0-2) 

`rest/service/recordFileService.js:164-179` — `getByHashOrNumber` builds and executes `SELECT ... FROM record_file WHERE index = $1` directly against the database for every call. [4](#0-3) 

`rest/dbpool.js:14` — The pool's `max` is set from `config.db.pool.maxConnections`, which defaults to **10** per the documented configuration. [5](#0-4) 

**Root cause:** The REST API (Node.js) has no application-level rate limiting. The middleware stack in `rest/server.js` includes only `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, and optional metrics/cache handlers — none of which throttle request rate or concurrent DB usage per IP. [6](#0-5) 

The optional Traefik middleware is gated behind `{{ if and .Values.global.middleware .Values.middleware }}` and is not enabled by default. [7](#0-6) 

**Exploit flow:**
1. Attacker sends N >> 10 concurrent `GET /api/v1/blocks/0` requests.
2. Each passes `validateHashOrNumber` (zero is allowed) and reaches `RecordFileService.getByHashOrNumber`.
3. Each call acquires a connection from the 10-connection pool via `super.getSingleRow`.
4. Pool is exhausted; subsequent requests queue internally, each holding a pending Promise for up to `connectionTimeoutMillis` (default 20 000 ms).
5. The queue grows unboundedly under sustained flood, consuming Node.js heap memory.
6. All other REST API endpoints (accounts, transactions, contracts, etc.) that also need DB connections are starved — they either time out after 20 s or fail with pool errors.

**Why existing checks fail:**
- `authHandler` only sets a custom response `limit` for authenticated users; it does not reject or throttle unauthenticated requests. [8](#0-7) 
- `statement_timeout` (20 s) and `connectionTimeoutMillis` (20 s) bound individual query duration but do not prevent pool exhaustion under sustained concurrency. [9](#0-8) 
- The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` Java module and is entirely absent from the Node.js REST API. [10](#0-9) 

### Impact Explanation
All REST API endpoints share the same 10-connection pool. Exhausting it with requests to `/blocks/0` denies database access to every other endpoint simultaneously — accounts, transactions, contracts, tokens, etc. — constituting a full REST API outage. The default pool size of 10 is extremely small relative to the number of concurrent HTTP connections Node.js can accept, making the disparity trivially exploitable. Severity: **High** (unauthenticated, service-wide DoS).

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. Any internet-accessible deployment is reachable. The attack requires only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`) targeting a single stable URL. The endpoint is stable and predictable (`/api/v1/blocks/0`). The attack is repeatable and can be sustained indefinitely. Likelihood: **High**.

### Recommendation
1. **Add application-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit` or `rate-limiter-flexible`) applied globally or per-IP before any route handler.
2. **Increase the default pool size** or add a per-request concurrency semaphore so that a single endpoint cannot monopolize all connections.
3. **Enable response caching** (`cache.response.enabled`) for the blocks-by-number endpoint so repeated identical lookups (e.g., block 0) are served from Redis without hitting the DB.
4. **Enable and configure the Traefik middleware** in production Helm values to enforce rate limits at the ingress layer.
5. Consider **rejecting requests that would return 404** (e.g., block 0 on a live network) at the cache/CDN layer.

  async getByHashOrNumber(hash, number) {
    let whereStatement = '';
    const params = [];
    if (hash) {
      hash = hash.toLowerCase();
      whereStatement += `${RecordFile.HASH} like $1`;
      params.push(hash + '%');
    } else {
      whereStatement += `${RecordFile.INDEX} = $1`;
      params.push(number);
    }

    const query = `${RecordFileService.blocksQuery} where ${whereStatement}`;
    const row = await super.getSingleRow(query, params);
    return row ? new RecordFile(row) : null;
  }
