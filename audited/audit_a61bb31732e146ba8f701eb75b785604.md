### Title
Unbounded `X-Forwarded-For` Header Logged Per Request in `TracingMiddleware` Enables Log Storage Exhaustion

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the raw `X-Forwarded-For` header value with no length validation or truncation and returns it directly. `TracingMiddleware` embeds this value verbatim into a log line written on every non-internal request. Because Go's `net/http` server defaults to a 1 MB header limit and no application-level rate limiting exists, an unauthenticated attacker can flood the server with requests carrying ~1 MB `X-Forwarded-For` values, writing ~1 MB to the log per request and exhausting log storage.

### Finding Description
**Code path:**
- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74: reads `r.Header.Get(xForwardedForHeader)` with no length check and returns the raw string.
- `TracingMiddleware`, lines 46–58: calls `getClientIpAddress(request)`, then unconditionally formats it into `message` via `fmt.Sprintf` and calls `log.Info(message)` for every non-internal path.

**Root cause:** The function makes the implicit assumption that the header contains a short, valid IP address string. No maximum length is enforced before the value is stored and logged.

**Failed assumption:** The `X-Forwarded-For` header is fully attacker-controlled. Any unauthenticated HTTP client can set it to an arbitrary byte sequence up to Go's default `MaxHeaderBytes` limit (1,048,576 bytes), since `main.go` constructs the `http.Server` without setting `MaxHeaderBytes`:

```go
// rosetta/main.go lines 220-227
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
    // MaxHeaderBytes not set → defaults to 1 MB
}
```

**Why existing checks fail:**
- The Traefik `rateLimit` and `inFlightReq` middlewares (defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166) are **optional** Kubernetes infrastructure gated on `global.middleware: true`. They are not present in the application binary and provide no protection in bare-metal, Docker, or misconfigured Kubernetes deployments.
- There is no application-level rate limiting, header size cap, or IP validation anywhere in the Rosetta Go codebase.

### Impact Explanation
Each crafted request causes up to ~1 MB of attacker-controlled data to be written to the operator's log sink (stdout/file/syslog). At a sustained rate of 100 requests/second (trivially achievable from a single machine), this generates ~100 MB/s of log data. Log storage exhaustion degrades or eliminates observability for all operators sharing the same log infrastructure, making it impossible to detect other incidents. The Rosetta API itself remains functional, so the impact is purely observability/operational — consistent with the "griefing, no economic damage" severity classification.

### Likelihood Explanation
The Rosetta API is publicly accessible (port 5700, no authentication). The attack requires only the ability to send HTTP requests — no credentials, no special network position, no prior knowledge of the system. The exploit is trivially scriptable with `curl` or any HTTP library. It is repeatable indefinitely and can be parallelized across multiple source IPs to bypass any IP-based rate limiting that may be present at the infrastructure layer.

### Recommendation
1. **Truncate the header value before logging.** In `getClientIpAddress()`, cap the returned string to a safe maximum (e.g., 45 characters for the longest valid IPv6 address):
   ```go
   const maxIpLength = 45
   if len(ipAddress) > maxIpLength {
       ipAddress = ipAddress[:maxIpLength]
   }
   ```
2. **Validate the value is a valid IP address** using `net.ParseIP()` and fall back to `r.RemoteAddr` if it is not.
3. **Set `MaxHeaderBytes`** on the `http.Server` to a small value appropriate for the Rosetta API (e.g., 8 KB) to limit the total header attack surface.
4. **Do not rely solely on optional infrastructure-level controls** for input sanitization that must hold at the application layer.
