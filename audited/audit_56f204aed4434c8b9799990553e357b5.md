### Title
Unbounded Per-Subscriber Memory Accumulation via gRPC Topic Subscriptions Enables Heap Exhaustion

### Summary
`SharedTopicListener.listen()` allocates two per-subscriber in-memory buffers — an `onBackpressureBuffer` of up to 16,384 messages and a `publishOn` prefetch queue of 48 messages — with no global cap on the total number of concurrent subscribers. Because `maxConcurrentCallsPerConnection=5` only limits calls per TCP connection and there is no limit on the number of connections, an unprivileged attacker can open enough connections to aggregate these per-subscriber buffers and exhaust JVM heap, crashing the node.

### Finding Description

**Exact code path:**

`SharedTopicListener.listen()` ( [1](#0-0) ) applies two sequential per-subscriber buffers to every caller:

```java
return getSharedListener(filter)
    .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
    .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
```

- `maxBufferSize` defaults to **16,384** messages per subscriber. [2](#0-1) 
- `prefetch` defaults to **48** messages per subscriber. [3](#0-2) 

Both buffers are allocated on the JVM heap per subscriber, not shared. The Redis subscription itself is shared per topic via `.share()` [4](#0-3) , but each downstream subscriber gets its own independent buffer chain.

**Root cause — failed assumption:**

The only connection-level guard is `maxConcurrentCallsPerConnection=5` [5](#0-4) , applied in `GrpcConfiguration` [6](#0-5) . This limits calls *per connection*, not globally. There is no limit on the number of TCP connections, no per-IP connection cap, and no global subscriber count ceiling. `TopicMessageServiceImpl` tracks `subscriberCount` as a Micrometer gauge [7](#0-6)  but never enforces a maximum — it is purely observational.

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (no global connection limit).
2. On each connection, attacker opens 5 `subscribeTopic` streams to a high-traffic topic (e.g., a topic receiving hundreds of messages/second).
3. Attacker intentionally stalls consumption (does not read from the gRPC stream, or reads very slowly).
4. For each stalled subscriber, Reactor fills the `onBackpressureBuffer(16384)` queue. At ~500 bytes per `TopicMessage`, this is ~8 MB per subscriber before the `ERROR` strategy terminates it.
5. The attacker immediately reconnects terminated subscriptions, keeping `N × 5` subscriptions in the filling state simultaneously.
6. Aggregate heap usage: `N × 5 × 8 MB`. With the default Helm resource limit of 2,048 MiB [8](#0-7) , only ~51 connections (255 subscriptions) are needed to exhaust heap.

**Why existing checks fail:**

- `onBackpressureBuffer(16384, ERROR)` terminates individual subscriptions when full, but does not prevent the attacker from immediately re-subscribing, keeping the aggregate buffer count high.
- `maxConcurrentCallsPerConnection=5` is bypassed by opening many connections; gRPC/HTTP2 connections are cheap and there is no server-side global connection limit configured.
- No authentication or authorization is required to call `subscribeTopic`. [9](#0-8) 

### Impact Explanation

JVM heap exhaustion causes an `OutOfMemoryError`, crashing the gRPC mirror node process. Because mirror nodes are stateless replicas, an attacker targeting multiple nodes simultaneously can take down 30%+ of the fleet, degrading or eliminating the gRPC subscription service for all legitimate users. The Helm chart's `GrpcHighMemory` alert fires only after 5 minutes above 80% memory [10](#0-9) , giving the attacker a wide window before operators can respond.

### Likelihood Explanation

No authentication is required. The attack requires only a standard gRPC client library and the ability to open ~50 TCP connections — trivially achievable from a single machine or a small botnet. The exploit is repeatable: after a node restarts, the attacker can immediately re-execute. The attack is amplified on high-traffic topics (e.g., the HCS ping topic used by the monitor), which are publicly known.

### Recommendation

1. **Enforce a global subscriber limit**: In `TopicMessageServiceImpl.subscribeTopic()`, reject new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., 1,000), returning `RESOURCE_EXHAUSTED`.
2. **Add a global connection limit**: Configure `maxConnections` on the Netty server builder in `GrpcConfiguration` in addition to `maxConcurrentCallsPerConnection`.
3. **Reduce `maxBufferSize`**: Lower the default from 16,384 to a smaller value (e.g., 256–512) to reduce per-subscriber heap footprint, or make it configurable with a lower default.
4. **Add per-IP rate limiting**: Limit the number of concurrent subscriptions per source IP using a filter or interceptor.
5. **Consider `DROP_LATEST` or `DROP_OLDEST`** instead of `ERROR` for `onBackpressureBuffer` to avoid the reconnect-loop amplification, though this alone does not fix the root cause.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2, timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234  # high-traffic topic
NUM_CONNECTIONS = 60
CALLS_PER_CONNECTION = 5

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        # no endTime = infinite subscription
    )
    streams = []
    for _ in range(CALLS_PER_CONNECTION):
        stream = stub.subscribeTopic(query)
        streams.append(stream)
        # Do NOT consume: stall the stream to fill onBackpressureBuffer
    # Hold connections open indefinitely
    threading.Event().wait()

threads = [threading.Thread(target=open_subscriptions, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.daemon = True
    t.start()

# 60 connections × 5 subscriptions = 300 stalled subscribers
# Each fills onBackpressureBuffer(16384) × ~500 bytes ≈ 8 MB
# Total: ~2.4 GB → OOM on a 2 GB heap node
print("Attack running: 300 stalled subscriptions open")
threading.Event().wait()
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
