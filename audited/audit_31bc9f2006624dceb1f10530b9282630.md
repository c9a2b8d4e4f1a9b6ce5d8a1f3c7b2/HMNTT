### Title
Missing Application-Level Rate Limiting on GraphQL Endpoint Enables Connection Pool Exhaustion DoS

### Summary
The GraphQL module exposes the `/graphql` endpoint with no application-level rate limiting, unlike the `web3` module which has a full `ThrottleConfiguration` using bucket4j. An unprivileged attacker can flood `findByAlias` (or any other resolver) with concurrent requests, exhausting the HikariCP connection pool and the servlet thread pool. During a network partition this is amplified because each request blocks for the full HikariCP `connectionTimeout` (default 30 s) before failing, keeping threads and pool slots occupied far longer than normal.

### Finding Description

**Exact code path:**

`findByAlias` issues a native SQL query with no caching or throttle guard: [1](#0-0) 

The only application-level defences in the GraphQL module are query-complexity/depth limits and JSON/parser size caps: [2](#0-1) 

These limits constrain the *shape* of a single query but impose no bound on the *number* of requests per second. There is no `ThrottleConfiguration`, no bucket4j bean, and no servlet filter performing rate limiting anywhere in the `graphql/` source tree — confirmed by the absence of any match for throttle/rate-limit patterns in that module.

By contrast, the `web3` module has a full per-second token-bucket implementation: [3](#0-2) 

**Infrastructure mitigations are conditional and off by default:**

The Traefik `inFlightReq` middleware (amount: 5) is only rendered when `global.middleware: true`: [4](#0-3) 

The default value is `false`: [5](#0-4) 

The GCP backend policy `maxRatePerEndpoint: 250` is similarly gated on `gateway.gcp.enabled`: [6](#0-5) 

Deployments that do not use GCP Gateway or Traefik (e.g., bare-metal, non-GCP cloud, local) have no rate limiting at any layer.

**Root cause:** The GraphQL module was never given the same throttle infrastructure as `web3`. The failed assumption is that infrastructure-level controls (load balancer, gateway) will always be present and sufficient — they are optional and disabled by default.

### Impact Explanation

HikariCP's default pool size is 10 connections. Spring MVC's default thread pool is bounded (typically 200 threads). An attacker sending thousands of concurrent `findByAlias` requests will:

1. Saturate all pool connections immediately.
2. Force subsequent requests to queue inside HikariCP for up to 30 s (default `connectionTimeout`).
3. Simultaneously exhaust the servlet thread pool, making the service unresponsive to all traffic including health checks.

During a network partition the DB is unreachable, so every connection attempt blocks for the full timeout. The backlog of queued requests continues to consume threads even after the partition resolves, delaying recovery. The Prometheus alert for high DB connection utilisation only fires after 5 minutes: [7](#0-6) 

meaning the service can be kept in a degraded state continuously with no automatic remediation.

**Severity: High** — complete denial of service achievable by any unauthenticated client with network access to the endpoint.

### Likelihood Explanation

- **Precondition:** Network access to the `/graphql` endpoint. No credentials required.
- **Skill required:** Trivial — a single `curl` loop or any HTTP benchmarking tool (e.g., `wrk`, `hey`) suffices.
- **Repeatability:** Continuous; the attacker simply keeps the flood running.
- **Detection:** Slow — the alert threshold is 75 % pool utilisation sustained for 5 minutes, giving the attacker a large window.

### Recommendation

1. **Add application-level rate limiting to the GraphQL module** mirroring the `web3` pattern: introduce a `ThrottleConfiguration` bean using bucket4j and a servlet filter or `WebGraphQlInterceptor` that consumes a token per request and returns HTTP 429 when the bucket is empty.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) or document it as a required production setting.
3. **Reduce HikariCP `connectionTimeout`** from the 30 s default to a value like 2–5 s so that requests fail fast during a partition rather than holding threads.
4. **Set an explicit `maximum-pool-size`** in the GraphQL application properties so the pool size is a known, tuned value rather than the HikariCP default of 10.

### Proof of Concept

```bash
# No credentials needed. Replace HOST with the GraphQL service address.
# Send 2000 concurrent findByAlias queries:
wrk -t 20 -c 2000 -d 60s \
  -s <(cat <<'EOF'
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"query":"{ account(input:{alias:\"AABBCCDD\"}) { balance } }"}'
EOF
) http://HOST/graphql/alpha

# Observe: after a few seconds all threads are occupied.
# Health endpoint becomes unresponsive:
curl -v http://HOST/actuator/health/readiness
# Expected: connection hangs or times out → liveness probe fails → pod restart loop.
```
