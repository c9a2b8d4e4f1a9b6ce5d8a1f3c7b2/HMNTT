### Title
EXTCODEHASH-Driven DB Query Amplification via Undersized Contract Bytecode Cache

### Summary
An unprivileged attacker can submit a single `eth_call` containing EVM bytecode that loops over `EXTCODEHASH` against thousands of distinct contract addresses. Each distinct address not present in the application-level Caffeine cache triggers a synchronous database query through `findRuntimeBytecode()`. The contract cache is capped at 1,000 entries while a single 15M-gas request can issue ~5,769 cold `EXTCODEHASH` calls, creating a DB query amplification factor that the gas throttle does not bound.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → EVM execution → `EXTCODEHASH` opcode → `ContractBytecodeReadableKVState.readFromDataSource()` → `ContractRepository.findRuntimeBytecode()`

**Root cause — cache size vs. gas budget mismatch:**

`findRuntimeBytecode()` is annotated:

```java
// ContractRepository.java line 16-18
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

The backing Caffeine cache is configured with `maximumSize=1000`:

```java
// CacheProperties.java line 22
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
``` [2](#0-1) 

Every `EXTCODEHASH` opcode execution resolves through `ContractBytecodeReadableKVState.readFromDataSource()`:

```java
// ContractBytecodeReadableKVState.java lines 39-47
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        .map(Bytes::wrap)
        .map(Bytecode::new)
        .orElse(null);
``` [3](#0-2) 

**Gas budget math:**

- `maxGasLimit = 15_000_000` per request
- Cold `EXTCODEHASH` costs 2,600 gas (EIP-2929)
- Maximum `EXTCODEHASH` calls per request: `15,000,000 / 2,600 ≈ 5,769`
- Cache holds only 1,000 entries → 4,769 cache misses → 4,769 DB queries per request [4](#0-3) 

**Why `unless = "#result == null"` does not help:**

The return type is `Optional<byte[]>`. `Optional.empty()` is not `null`, so non-existent contract lookups are cached. However, the cache evicts entries once it exceeds 1,000 entries (Caffeine LRU). An attacker rotating through fresh sets of distinct IDs on each request keeps the cache perpetually cold for the majority of lookups.

### Impact Explanation

At the default rate limit of 500 requests/second and 4,769 cache-miss DB queries per request, the attacker can drive approximately **2.38 million DB queries per second** from a single client. Each query executes `SELECT runtime_bytecode FROM contract WHERE id = ?` against the PostgreSQL backend. This saturates DB connection pools, increases CPU and I/O on the database host, and degrades response times for all other mirror-node consumers (REST API, gRPC, importer). The 30% resource increase threshold is easily exceeded because normal operation does not involve millions of bytecode lookups per second. [5](#0-4) 

### Likelihood Explanation

- **No privileges required**: `POST /api/v1/contracts/call` is a public, unauthenticated endpoint.
- **No on-chain state needed**: The attacker targets non-existent contract IDs; the DB returns empty rows, which are not cached when the cache is full.
- **Trivially repeatable**: The attack bytecode is a simple loop; the attacker only needs to increment the target address on each request to defeat the cache.
- **Gas throttle does not prevent it**: The gas throttle (`gasPerSecond = 7.5B`) limits total gas consumed per second but does not limit the number of DB queries per gas unit. EXTCODEHASH is cheap in gas but expensive in DB I/O. [6](#0-5) [7](#0-6) 

### Recommendation

1. **Increase cache size**: Raise `hiero.mirror.web3.cache.contract` `maximumSize` to at least 10,000–50,000 to absorb realistic attack patterns. The current value of 1,000 is far below the number of distinct IDs reachable in a single max-gas request. [8](#0-7) 

2. **Add a per-request DB query cap**: Track the number of `findRuntimeBytecode()` calls within a single EVM execution context (`ContractCallContext`) and abort execution with `OUT_OF_GAS` or a custom error once a configurable limit (e.g., 200) is reached. [3](#0-2) 

3. **Cache negative results explicitly**: The `unless = "#result == null"` condition correctly caches `Optional.empty()`, but Caffeine's LRU eviction removes these entries under pressure. Consider a dedicated negative-result cache with a smaller TTL and larger capacity to prevent repeated DB hits for non-existent IDs. [9](#0-8) 

4. **Rate-limit by DB query count, not only gas**: Introduce a secondary throttle bucket that counts external state reads (bytecode, storage, entity) per second across all requests, independent of gas.

### Proof of Concept

**Preconditions:** Public access to `POST /api/v1/contracts/call`. No account, no authentication.

**Attack bytecode (Solidity pseudocode):**
```solidity
// Deploy this contract, then call scan()
contract Attacker {
    function scan() external view {
        for (uint256 i = 1; i <= 5000; i++) {
            address target = address(uint160(0x10000000 + i + block.number * 5000));
            assembly { pop(extcodehash(target)) }
        }
    }
}
```

**Steps:**
1. Encode the `scan()` call selector and submit:
   ```
   POST /api/v1/contracts/call
   {
     "to": "<attacker_contract_address>",
     "data": "<scan() calldata>",
     "gas": 15000000
   }
   ```
2. Repeat at 500 requests/second (within the rate limit) using a different `block.number` offset each time to rotate through fresh address ranges.
3. Each request causes ~4,769 cache-miss DB queries (`SELECT runtime_bytecode FROM contract WHERE id = ?`).
4. Monitor PostgreSQL `pg_stat_activity` or `pg_stat_user_tables` — query rate and I/O will spike by well over 30% compared to the preceding 24-hour baseline.
