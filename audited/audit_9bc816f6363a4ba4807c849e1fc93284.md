### Title
Mirror REST Endpoint Poisoning via All-Zero Ports Causes Non-Retryable Pinger Shutdown

### Summary
When `network=other`, `newClient()` calls `buildNetworkFromMirrorNodes()`, which fetches node topology from an operator-configured HTTP mirror REST URL. An attacker who can control or intercept that HTTP response can return a `nodesEnvelope` where every `service_endpoints` entry has `port: 0`. The `port == 0` guard in `fetchMirrorNodeNetwork()` silently skips every endpoint, the empty-network check returns a **non-retryable** error (`retry=false`), and `newClient()` propagates that error — permanently halting the pinger without ever submitting a transaction.

### Finding Description

**Code path:**

`pinger/sdk_client.go` `newClient()` (line 18) calls `buildNetworkFromMirrorNodes()` only when `cfg.network == "other"`. [1](#0-0) 

Inside `fetchMirrorNodeNetwork()`, each service endpoint is filtered: [2](#0-1) 

If every endpoint has `port: 0` (or empty host), all are silently skipped. The subsequent guard: [3](#0-2) 

returns `(nil, false, error)` — the `false` (retry=false) is critical.

Back in `buildNetworkFromMirrorNodes()`, the retry loop immediately breaks on `!retry`: [4](#0-3) 

So even with `mirrorNodeClientMaxRetries=10`, **a single crafted response bypasses all retries** and propagates a fatal error to `newClient()`.

**Root cause:** The code unconditionally trusts the mirror REST response. There is no minimum-node-count enforcement, no TLS certificate pinning, and no signature verification. The HTTP client is constructed with no TLS configuration: [5](#0-4) 

The example URL in config comments is `http://mirror-rest:5551` (plain HTTP), making interception trivial on any shared network path.

### Impact Explanation

The pinger's sole purpose is to submit periodic transactions and confirm network liveness. If `newClient()` returns an error, the pinger process exits or never enters its transaction loop, meaning **zero transactions are submitted**. Network confirmation monitoring is completely halted for as long as the attacker sustains the poisoned response. This is a full Denial-of-Service against the monitoring layer, which could mask a real network outage or allow one to go undetected.

### Likelihood Explanation

**Preconditions:**
- `HIERO_MIRROR_PINGER_NETWORK=other` must be set (required for any custom/private network deployment).
- The attacker must be able to serve a crafted HTTP response at the configured mirror REST URL.

**Realistic attacker paths (no privilege required):**
1. **Plain HTTP MITM:** If the mirror REST URL uses `http://` (as shown in the default example), any on-path attacker (same LAN, rogue router, ARP spoofing, DNS hijacking) can intercept and replace the response.
2. **Compromised or malicious mirror node:** The mirror node operator (not necessarily the pinger operator) can return any JSON payload.
3. **DNS hijacking:** If the mirror REST URL uses a hostname, DNS poisoning redirects requests to an attacker-controlled server.

The attack is repeatable: the pinger re-calls `buildNetworkFromMirrorNodes()` on restart, so the attacker only needs to sustain the poisoned DNS/route until the pinger gives up.

### Recommendation

1. **Enforce HTTPS with certificate validation** for all mirror REST URLs; reject plain `http://` schemes at config load time.
2. **Change the empty-network error to retryable** (`retry=true`) so transient or injected bad responses do not permanently halt the pinger:
   - In `fetchMirrorNodeNetwork()`, return `(nil, true, fmt.Errorf(...))` when `len(network) == 0`.
3. **Enforce a minimum node count** (e.g., ≥ 1 node with a valid port) and log a warning when nodes are skipped.
4. **Validate port range** (1–65535) explicitly rather than only rejecting `0`.
5. Consider **cross-validating** the returned node list against a locally pinned seed list for `network=other`.

### Proof of Concept

**Setup:** Pinger configured with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.

**Step 1 — Attacker serves crafted response** at `http://mirror-rest:5551/api/v1/network/nodes`:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"domain_name": "node.example.com", "ip_address_v4": "", "port": 0},
        {"domain_name": "", "ip_address_v4": "1.2.3.4", "port": 0}
      ]
    }
  ],
  "links": {"next": null}
}
```

**Step 2 — Trace through code:**
- Both endpoints hit `ep.Port == 0` → `continue` (line 118 of `mirror_node_client.go`).
- `len(network) == 0` → `return nil, false, fmt.Errorf("no usable service_endpoints...")` (line 128).
- `retry=false` → `buildNetworkFromMirrorNodes()` breaks immediately, no retries attempted.
- `newClient()` receives `err != nil` → returns `nil, err`.

**Step 3 — Result:** Pinger fails to initialize, logs a fatal error, and submits no transactions. Network confirmation monitoring is halted for the duration of the attack.
