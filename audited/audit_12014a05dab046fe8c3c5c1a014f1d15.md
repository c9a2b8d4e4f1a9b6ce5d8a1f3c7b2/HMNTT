### Title
Contract Filter Bypass via Unresolved Recipient Entity Enables Trace Log Flooding

### Summary
In `MirrorOperationActionTracer.tracePostExecution()`, the contract allowlist filter (`contractFilterCheck`) is guarded by a short-circuit `&&` that requires `recipientNum.isPresent()`. When `commonEntityAccessor.get()` returns `Optional.empty()` for an unknown recipient address, the filter is never evaluated and `log.info()` fires unconditionally for every EVM opcode in the call. Any unprivileged user can exploit this by targeting non-existent addresses to bypass the operator-configured contract filter and flood trace logs.

### Finding Description

**Exact code path:**

`MirrorOperationActionTracer.java` lines 59–68: [1](#0-0) 

```java
final var recipientAddress = frame.getRecipientAddress();
final var recipientNum = recipientAddress != null
        ? commonEntityAccessor.get(recipientAddress, Optional.empty())
        : Optional.empty();

if (recipientNum.isPresent()
        && traceProperties.contractFilterCheck(
                CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
    return;
}
log.info(...);
```

`TraceProperties.java` lines 33–35: [2](#0-1) 

```java
public boolean contractFilterCheck(String contract) {
    return !getContract().isEmpty() && !getContract().contains(contract);
}
```

**Root cause:** The guard `recipientNum.isPresent() && contractFilterCheck(...)` uses short-circuit evaluation. When `commonEntityAccessor.get()` returns `Optional.empty()` (address not in the entity store), `isPresent()` is `false`, the `&&` short-circuits, `contractFilterCheck` is never called, and execution falls through to `log.info()`. The failed assumption is that every recipient address will resolve to a known entity — but the EVM allows calls to arbitrary addresses that may not exist in the mirror node's entity store.

**Exploit flow:**
1. Operator enables tracing (`hiero.mirror.web3.evm.trace.enabled=true`) and configures a non-empty contract allowlist (e.g., only trace contract `0xABCD...`).
2. Attacker submits a `eth_call` or contract call targeting any address not present in the entity store (e.g., a freshly generated random 20-byte address).
3. `commonEntityAccessor.get(recipientAddress, Optional.empty())` returns `Optional.empty()`.
4. The `if (recipientNum.isPresent() && ...)` condition is `false`; no early return.
5. `log.info(...)` fires for **every EVM opcode** executed in the call frame, regardless of the configured contract filter.

**Why existing checks are insufficient:**

- The `isEnabled()` check at line 51 only gates the entire feature; it does not protect against filter bypass when tracing is on. [3](#0-2) 
- The `stateFilterCheck` at line 55 only filters by EVM frame state, not by contract address. [4](#0-3) 
- The test suite (`contractFilterMismatch`) only tests the case where the entity **is** found; there is no test for the `Optional.empty()` path with a non-empty contract filter, confirming the bypass is untested. [5](#0-4) 

### Impact Explanation
When tracing is enabled with a non-empty contract filter, an attacker can generate unbounded `INFO`-level log entries by repeatedly calling non-existent addresses. Each call produces one log line per EVM opcode executed. This can exhaust disk space, overwhelm log aggregation pipelines (e.g., Elasticsearch, Splunk), degrade observability for legitimate operators, and potentially cause the mirror node process to crash or become unresponsive due to I/O saturation. The contract filter — the operator's primary tool for scoping trace verbosity — is rendered completely ineffective. [6](#0-5) 

### Likelihood Explanation
The precondition is that an operator has enabled tracing (`enabled=false` by default). However, tracing is a legitimate operational feature; any deployment that enables it with a contract filter is immediately vulnerable. The attacker requires no credentials, no special role, and no on-chain funds beyond gas for the call. The attack is trivially repeatable via scripted `eth_call` requests (which consume no gas on mirror nodes since they simulate calls). The non-existent address space is effectively infinite, making blocklisting impractical. [7](#0-6) 

### Recommendation
Change the guard logic so that when the contract filter is non-empty and the entity cannot be resolved, logging is **suppressed** (treat unknown recipients as not matching the allowlist):

```java
// Proposed fix
if (!traceProperties.getContract().isEmpty()) {
    if (recipientNum.isEmpty()) {
        return; // unknown address: not in allowlist, suppress
    }
    if (traceProperties.contractFilterCheck(
            CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
        return;
    }
}
```

Alternatively, restructure `contractFilterCheck` to accept `Optional<Entity>` directly and treat `Optional.empty()` as a non-match when the filter set is non-empty. [8](#0-7) 

### Proof of Concept
**Preconditions:**
- Mirror node deployed with `hiero.mirror.web3.evm.trace.enabled=true`
- `hiero.mirror.web3.evm.trace.contract` set to one or more specific contract addresses (e.g., `0x000000000000000000000000000000000000abcd`)

**Steps:**
1. Generate a random 20-byte address that does not correspond to any entity in the mirror node's database, e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`.
2. Submit repeated `eth_call` requests targeting that address:
   ```bash
   for i in $(seq 1 10000); do
     curl -s -X POST http://mirror-node:8545 \
       -H 'Content-Type: application/json' \
       -d '{"jsonrpc":"2.0","method":"eth_call","params":[{"to":"0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef","data":"0x"},"latest"],"id":1}'
   done
   ```
3. Observe that `log.info(...)` fires for every opcode in every call despite the contract filter not including `0xdeadbeef...`.
4. Monitor disk usage and log volume — entries accumulate without bound, bypassing the configured allowlist entirely. [1](#0-0)
