### Title
Unbounded Concurrent Subscription DoS via `PollingTopicMessageRetriever.retrieve()` Scheduler Exhaustion

### Summary
`PollingTopicMessageRetriever.retrieve()` creates an unbounded `PollingContext` and schedules a repeating polling task on a shared `BoundedElasticScheduler` for every subscription, with no global cap on concurrent subscriptions. Because `maxConcurrentCallsPerConnection` only limits streams per TCP connection and no total-connection limit is configured, an unauthenticated attacker can open arbitrarily many connections and saturate the scheduler, starving legitimate subscribers of fund-transfer confirmations.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (no authentication check) calls `TopicMessageService.subscribeTopic()`, which calls `topicMessageRetriever.retrieve(filter, true)` at line 63 of `TopicMessageServiceImpl.java`. This lands in `PollingTopicMessageRetriever.retrieve()`:

```java
// PollingTopicMessageRetriever.java lines 45-63
public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
    if (!retrieverProperties.isEnabled()) { return Flux.empty(); }

    PollingContext context = new PollingContext(filter, throttled);   // new context per call, no limit
    return Flux.defer(() -> poll(context))
            .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                    .jitter(0.1)
                    .withFixedDelay(context.getFrequency())
                    .withScheduler(scheduler))                        // shared BoundedElasticScheduler
            ...
            .timeout(retrieverProperties.getTimeout(), scheduler)
```

The `scheduler` field is a single `BoundedElasticScheduler` instance created at bean construction (line 41): `scheduler = Schedulers.boundedElastic();`. Every active subscription in the historical-retrieval phase holds a live repeating task on this scheduler. For the `throttled=true` path, `numRepeats = Long.MAX_VALUE` (line 99) and `frequency = 2s` (line 100), so the task persists until `isComplete()` returns true (i.e., until the last DB page is smaller than `maxPageSize=1000`).

**Root cause / failed assumption:**

`TopicMessageServiceImpl` tracks `subscriberCount` (line 48) as a metrics gauge only — it is never checked against a maximum before accepting a new subscription (lines 87-91). The only server-side throttle is `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, applied in GrpcConfiguration line 33), which is a *per-connection* limit. `GrpcConfiguration` sets no `maxConnections` or `maxConnectionAge` on the `NettyServerBuilder` (lines 27-35), so an attacker can open an unlimited number of TCP connections.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| `maxConcurrentCallsPerConnection = 5` | Per-connection only; attacker opens N connections → N×5 streams |
| `subscriberCount` AtomicLong | Metrics gauge, never enforced as a cap |
| `retrieverProperties.timeout = 60s` | Idle timeout; resets on each emitted message; attacker keeps connections alive |
| `checkTopicExists = true` | Only validates topic existence, not subscriber count |

**Exploit flow:**

1. Attacker opens C TCP connections to port 5600 (no auth required).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams for the same `topicId` with a historical `startTime` far in the past (ensuring the historical retrieval phase is long-lived).
3. Each stream triggers `retrieve(filter, true)` → new `PollingContext` + repeating task on the shared `BoundedElasticScheduler`.
4. `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 78) is a blocking call executed on the `BoundedElasticScheduler` thread. With C×5 concurrent subscriptions all polling every 2 seconds, the scheduler's thread pool (default: `10 × CPU_cores`) is exhausted.
5. New tasks from legitimate subscribers are queued; once the queue (default 100,000) fills, tasks are rejected with `RejectedExecutionException`, breaking delivery for all subscribers.

### Impact Explanation

`subscribeTopic` is the mechanism by which clients receive fund-transfer confirmations from HCS topics. Exhausting the `BoundedElasticScheduler` prevents the historical-retrieval phase from completing for legitimate subscribers, meaning they never transition to the live-message phase and never receive confirmations. This is a complete denial of service for the confirmation delivery path, directly matching the "Critical: Direct loss of funds" classification — a user waiting for a transfer confirmation that never arrives may re-submit the transaction, causing double-spend or missed deadlines.

### Likelihood Explanation

No authentication, no IP rate limiting in the application layer, and no connection count limit make this trivially exploitable from a single machine. Opening thousands of HTTP/2 connections is a standard load-testing technique. The attacker needs only a valid `topicId` (all topic IDs are public on-chain) and a network path to port 5600. The attack is repeatable and persistent as long as the attacker maintains connections.

### Recommendation

1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP or per-connection subscription limit** using a `ConcurrentHashMap<InetAddress, AtomicInteger>` or a token-bucket rate limiter at the gRPC interceptor layer.
3. **Set `maxConnections`** on the `NettyServerBuilder` in `GrpcConfiguration` (e.g., `serverBuilder.maxConnectionAge(...)` and a total connection cap).
4. **Use a dedicated, bounded scheduler** for `PollingTopicMessageRetriever` with an explicit thread cap and rejection policy, rather than the shared `Schedulers.boundedElastic()`.
5. **Add authentication/authorization** to `subscribeTopic` to prevent anonymous abuse.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234          # any valid topic
NUM_CONNECTIONS = 500
STREAMS_PER_CONN = 5      # maxConcurrentCallsPerConnection default

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        # set consensusStartTime to epoch 0 to maximize historical retrieval phase
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Non-blocking: just open the stream, don't consume
        streams.append(stub.subscribeTopic(query))
    # Hold connections open
    import time; time.sleep(300)

threads = [threading.Thread(target=flood_connection, args=(i,))
           for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()

        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))

                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))

                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
