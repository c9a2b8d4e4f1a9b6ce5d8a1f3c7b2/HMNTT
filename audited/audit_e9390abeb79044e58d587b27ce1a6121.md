### Title
Unauthenticated High-Frequency Requests to `/accounts/:id/rewards` Exhaust DB Connection Pool via Uncached `isValidAccount()` Query

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` unconditionally calls `EntityService.isValidAccount()` on every request, which issues a live SQL query against the database with no caching. The REST service has no rate-limiting middleware, and account IDs are trivially enumerable sequential integers. An unprivileged attacker can flood this endpoint to saturate the finite DB connection pool, degrading availability for all legitimate users.

### Finding Description

**Exact code path:**

`listStakingRewardsByAccountId` at [1](#0-0)  calls `EntityService.isValidAccount(accountId)` on every single request with no short-circuit, no cache, and no guard.

`EntityService.isValidAccount` at [2](#0-1)  executes `entityExistenceQuery` — `SELECT type FROM entity WHERE id = $1` — directly against the database via `super.getSingleRow()` on every invocation. There is no memoization, no in-process cache, and no Redis cache wrapping this call.

The static query is defined at [3](#0-2) 

The DB connection pool is initialized at [4](#0-3)  with `max: config.db.pool.maxConnections` — a finite, configurable integer. The Helm chart caps `mirror_rest` at `max_user_connections: 250`. [5](#0-4) 

**Root cause — failed assumption:** The code assumes that either (a) the call rate will be low enough that per-request DB queries are acceptable, or (b) an upstream rate limiter or cache will absorb excess load. Neither assumption holds.

Reviewing `rest/server.js` in full: [6](#0-5)  — the middleware stack is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, optional `responseCacheCheckHandler`. There is **no rate-limiting middleware** (no `express-rate-limit`, no token bucket, no IP throttle) applied to the REST service for this endpoint.

The optional Redis response cache at [7](#0-6)  is only active when `config.cache.response.enabled && config.redis.enabled` — both are off by default and not guaranteed in all deployments. Even when enabled, cache misses (distinct account IDs) bypass it entirely and still hit the DB.

The throttle configuration found in the codebase (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) belongs exclusively to the **web3** Java service, not the Node.js REST service. [8](#0-7) 

**Exploit flow:**
1. Attacker enumerates valid account IDs (public, sequential integers: `0.0.1`, `0.0.2`, …).
2. For a numeric ID like `0.0.1001`, `EntityService.getEncodedId()` performs only local parsing — no DB query. [9](#0-8) 
3. Control immediately reaches `isValidAccount()`, which fires a DB query.
4. If the account is valid, `StakingRewardTransferService.getRewards()` fires a second DB query. [10](#0-9) 
5. With N concurrent requests, N pool slots are held simultaneously. When N ≥ `maxConnections`, legitimate requests queue and eventually time out.

### Impact Explanation
DB connection pool exhaustion causes all REST API endpoints sharing the same pool to fail or queue indefinitely. Legitimate users receive connection timeout errors or degraded response times across the entire REST service — not just the rewards endpoint. The impact is availability degradation (griefing) with no economic damage to on-chain state.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero special knowledge. Account IDs are public sequential integers visible in any block explorer. The attack is trivially scriptable with `curl` or any HTTP load tool (e.g., `ab`, `wrk`, `hey`). It is repeatable indefinitely. The only practical barrier is network bandwidth, which is low given the small request size.

### Recommendation
1. **Add per-IP rate limiting** to the REST service using `express-rate-limit` or equivalent, applied globally or specifically to the `/accounts/:id/rewards` route.
2. **Cache `isValidAccount()` results** in-process (e.g., with a short-TTL LRU cache like `lru-cache`) so repeated lookups for the same account ID do not each consume a DB connection.
3. **Ensure the Redis response cache is enabled** in production deployments, which would serve repeated identical requests without hitting the DB at all.
4. **Consider reordering** the handler logic so that filter validation (`buildAndValidateFilters`) occurs before the DB existence check, allowing cheap input rejection before any DB I/O.

### Proof of Concept
```bash
# Step 1: Identify valid account IDs (publicly known, e.g., 0.0.800 is the staking reward account)
# Step 2: Flood the endpoint with concurrent requests using known valid IDs

# Using 'hey' HTTP load tool (no auth required):
hey -n 100000 -c 300 \
  "https://<mirror-node-rest>/api/v1/accounts/0.0.800/rewards"

# Or cycle through sequential IDs to maximize distinct DB lookups:
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node-rest>/api/v1/accounts/0.0.$i/rewards" &
done
wait

# Expected result:
# - Each request triggers SELECT type FROM entity WHERE id = $1
# - With 300 concurrent requests and maxConnections=250, pool is saturated
# - Legitimate API calls to /transactions, /accounts, /tokens etc. begin timing out
# - REST service returns 500 or hangs for legitimate users
```

public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
