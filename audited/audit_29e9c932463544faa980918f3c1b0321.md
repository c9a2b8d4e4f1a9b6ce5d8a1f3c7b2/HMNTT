### Title
Unauthenticated DB Load Amplification via Uncached `findByAlias()` in GraphQL Module

### Summary
The GraphQL module's `EntityRepository.findByAlias()` lacks the `@Cacheable` annotation present on the equivalent web3 module method, causing every alias lookup to issue a direct database query. Combined with the complete absence of application-level rate limiting in the GraphQL module (unlike the web3 module's `ThrottleManagerImpl`), an unprivileged attacker can drive unbounded sequential or distributed-concurrent alias queries that exhaust the HikariCP connection pool and degrade query availability for all users.

### Finding Description
**Exact code path:**

`AccountController.account()` (line 48) â†’ `EntityServiceImpl.getByAliasAndType()` (line 30) â†’ `EntityRepository.findByAlias()` (line 13-14) â†’ raw SQL every call.

```
// graphql EntityRepository â€” no @Cacheable
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
```

The web3 equivalent explicitly caches:
```
// web3 EntityRepository â€” @Cacheable present
@Cacheable(cacheNames = CACHE_NAME_ALIAS, cacheManager = CACHE_MANAGER_ENTITY,
           key = "@spelHelper.hashCode(#alias)", unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

**Root cause:** The graphql module's `EntityRepository` was never given a `@Cacheable` annotation, and no application-level throttle bean (equivalent to web3's `ThrottleConfiguration` / `ThrottleManagerImpl`) exists in the graphql module. The only controls are:
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` â€” limit query structure, not request frequency.
- Helm chart Traefik middleware `inFlightReq: amount: 5` per source IP â€” limits concurrent in-flight requests per single IP only; does not limit sequential request rate; bypassable with multiple IPs; deployment-specific and not enforced at the application layer.
- `statementTimeout: 10000` â€” caps individual query duration but does not prevent many simultaneous queries from queuing.

**Failed assumption:** The design assumes infrastructure-level controls (5 concurrent requests per IP) are sufficient to prevent DB load amplification. They are not: sequential requests from one IP, or concurrent requests from multiple IPs, bypass this entirely.

### Impact Explanation
Each attacker-controlled alias lookup issues a `SELECT * FROM entity WHERE alias = ? AND deleted IS NOT TRUE` against the shared PostgreSQL instance. With no caching and no application-level rate limit, an attacker can saturate the HikariCP connection pool (default pool size applies to all graphql module connections). When the pool is exhausted, legitimate Hashgraph history queries â€” including those from other users querying account state, balances, and entity data â€” queue or time out. The Helm chart's own `GraphQLHighDBConnections` alert fires at 75% pool utilization, confirming the operators themselves recognize this as a critical resource boundary. Availability of Hashgraph history query responses is directly degraded.

### Likelihood Explanation
No authentication or privilege is required. The endpoint `POST /graphql/alpha` is publicly accessible. A single attacker with a modest number of IP addresses (cloud VMs, proxies) can trivially bypass the per-IP `inFlightReq: 5` control. Generating unique alias byte sequences to prevent any hypothetical future caching is trivial. The attack is fully scriptable with `curl` or any HTTP client in a loop, as shown in the project's own smoke-testing documentation.

### Recommendation
1. Add `@Cacheable` to `findByAlias()` and `findByEvmAddress()` in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, mirroring the web3 module's `CACHE_NAME_ALIAS` / `CACHE_NAME_EVM_ADDRESS` pattern.
2. Introduce an application-level rate-limiting bean in the graphql module (equivalent to web3's `ThrottleConfiguration` / `ThrottleManagerImpl`) that enforces a per-second request cap independent of infrastructure deployment.
3. Do not rely solely on the Helm chart `inFlightReq` middleware as the only request-frequency control; it is per-IP, concurrent-only, and deployment-specific.

### Proof of Concept
```bash
# Requires no credentials. Run from multiple IPs to bypass inFlightReq=5 per-IP limit.
# Each request generates a unique alias to defeat any future caching.
for i in $(seq 1 10000); do
  ALIAS=$(head -c 20 /dev/urandom | base32 | tr -d '=')
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {alias: \\\"$ALIAS\\\"}){balance}}\"}" &
done
wait
```
Each iteration triggers `EntityRepository.findByAlias()` â†’ direct DB query with no cache hit. Sustained execution across multiple source IPs exhausts the HikariCP pool, causing `hikaricp_connections_pending` to spike and legitimate queries to time out or return errors.
