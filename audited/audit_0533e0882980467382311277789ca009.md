### Title
Unbounded Unthrottled Historical Query via Safety-Check with `last == null` and User-Controlled `startTime`

### Summary
In `TopicMessageServiceImpl.missingMessages()`, when the safety check fires with `current == null` and `last == null` (no messages received yet), the gap-recovery query is built using the subscriber's own `filter.getStartTime()` with no upper bound (`endTime`) and no row cap (`limit`), and is issued with `throttled = false`. An unprivileged subscriber who sets `startTime = 0` can reliably trigger this path after one second, causing the server to execute a full, unthrottled historical scan from epoch start against the database.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.java`, `missingMessages()`, lines 138–149:

```java
private Flux<TopicMessage> missingMessages(TopicContext topicContext, @Nullable TopicMessage current) {
    final var last = topicContext.getLast();

    // Safety check triggered
    if (current == null) {
        long startTime = last != null
                ? last.getConsensusTimestamp() + 1
                : topicContext.getFilter().getStartTime();   // ← attacker-controlled
        var gapFilter =
                topicContext.getFilter().toBuilder().startTime(startTime).build();
        // ↑ inherits original filter's endTime (null) and limit (0) — no upper bound
        return topicMessageRetriever.retrieve(gapFilter, false);  // ← throttled=false
    }
```

The safety check is scheduled unconditionally at subscription time:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))   // current=null
        .subscribeOn(Schedulers.boundedElastic());
```

**Root cause and failed assumption:**

The code assumes that `filter.getStartTime()` is a "recent" timestamp because `TopicMessageFilter.isValidStartTime()` only enforces `startTime <= DomainUtils.now()`. The value `0` (nanoseconds since epoch, i.e., 1970-01-01) satisfies this check. When `last == null` (no messages delivered yet — empty topic, importer down, or historical query still in flight), the `gapFilter` is constructed with:
- `startTime = 0` (attacker-supplied)
- `endTime = null` (inherited — no upper bound)
- `limit = 0` (inherited — no row cap)

and passed to `retrieve(gapFilter, false)` where `false` disables throttling between polling pages.

**Why existing checks fail:**

1. `isValidStartTime()` — only checks `startTime <= now()`, not a minimum age.
2. The outer `flux.take(filter.getLimit())` — only applied when `filter.hasLimit()` is true (`limit > 0`); a subscriber with no limit bypasses it entirely.
3. The outer `flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime())` — only applied when `filter.getEndTime() != null`; a subscriber with no end time bypasses it.
4. `topicContext.isComplete()` — returns `false` when `endTime == null`, so the safety check always fires.
5. `throttled = false` — the normal gap-fill path (lines 164–177) also passes `false`, but that path always sets a bounded `endTime` and a computed `limit`. The safety-check path sets neither.

### Impact Explanation

Each subscription with `startTime = 0`, no `endTime`, and no `limit` causes the server to issue a full table scan of `topic_message` from epoch start, unthrottled, one second after the connection is opened. With a large topic history this is a multi-million-row query per connection. Multiple concurrent connections multiply the load linearly. The database connection pool and heap can be exhausted, causing denial of service for all subscribers on the node. Because the `safetyCheck` flux runs on `Schedulers.boundedElastic()`, the bounded elastic thread pool is also a target for exhaustion.

### Likelihood Explanation

The preconditions require no authentication, no special role, and no knowledge of internal state. Any gRPC client can set `startTime = 0` in a `ConsensusTopicQuery`. The trigger is deterministic: wait one second. The condition `last == null` is guaranteed when subscribing to a topic that has no messages yet, or when the importer is temporarily unavailable. The attack is trivially repeatable and scriptable with any HCS SDK or raw gRPC client.

### Recommendation

1. **Bound the safety-check gap query by wall-clock time:** when building `gapFilter` in the `current == null` branch, always set `endTime = DompcUtils.now()` so the query cannot scan beyond the present moment.
2. **Apply a maximum row cap:** set a configurable `maxSafetyCheckLimit` in `RetrieverProperties` and apply it to `gapFilter` when the original filter has no limit.
3. **Enforce a minimum `startTime` age:** add a validation constraint (e.g., no more than N days in the past) or clamp `startTime` server-side before building any gap filter.
4. **Pass `throttled = true`** for the safety-check retrieval path, consistent with the initial historical retrieval at line 63.
