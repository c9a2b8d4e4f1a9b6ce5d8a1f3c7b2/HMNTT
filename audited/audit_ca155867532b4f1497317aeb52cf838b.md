### Title
Unbounded Infinite-Poll Subscriptions Allow Resource Exhaustion via `RepeatSpec.times(Long.MAX_VALUE)`

### Summary
`PollingTopicListener.listen()` creates a polling loop using `RepeatSpec.times(Long.MAX_VALUE)` with no enforced subscription duration, no per-client subscription count limit, and no rate limiting on the gRPC endpoint. An unprivileged attacker can open many connections, each carrying multiple indefinite subscriptions, continuously draining the `boundedElastic` scheduler thread pool and the HikariCP database connection pool, denying service to legitimate subscribers.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–43:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← effectively infinite
                .jitter(0.1)
                .withFixedDelay(interval)              // default 500 ms
                .withScheduler(scheduler))             // boundedElastic
```

Every subscription schedules a recurring task on `Schedulers.boundedElastic()` that fires every 500 ms (default `listenerProperties.getInterval()`). Each firing calls `topicMessageRepository.findByFilter(newFilter)` (line 61), which acquires a JDBC connection from the HikariCP pool for the duration of the streaming query.

**Why subscriptions never terminate:**

`ConsensusController.toFilter()` (lines 55–73) maps the client-supplied `ConsensusTopicQuery` directly to a `TopicMessageFilter`. If the client sends `limit=0` and omits `consensusEndTime`:
- `filter.hasLimit()` → `false` → `TopicMessageServiceImpl` never calls `.take()` (line 83–85)
- `filter.getEndTime()` → `null` → `pastEndTime()` returns `Flux.never()` (lines 123–126), so no time-based termination fires

The `RepeatSpec.times(Long.MAX_VALUE)` loop therefore runs for ~9.2 × 10¹⁸ iterations — practically forever.

**Existing checks and why they are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14; `GrpcConfiguration`, line 33): limits calls *per TCP connection*, not globally. An attacker opens N connections → 5N simultaneous subscriptions.
- No IP-level or identity-level rate limiting exists on the gRPC service (the `ThrottleConfiguration`/`ThrottleManagerImpl` is scoped only to the `web3` module).
- No global maximum subscriber count is enforced (the `subscriberCount` gauge in `TopicMessageServiceImpl` is metrics-only, never checked against a ceiling).

### Impact Explanation

Each attacker-controlled subscription:
1. Permanently occupies one slot in `Schedulers.boundedElastic()` (default cap: 10 × CPU cores, with a bounded task queue).
2. Acquires a HikariCP JDBC connection every 500 ms; under load, connections queue and eventually time out for legitimate callers.

With enough connections (easily scripted), the attacker saturates both the scheduler and the DB pool. Legitimate `subscribeTopic` calls either block waiting for a DB connection or are rejected with a pool-timeout error. The mirror node's gRPC service becomes unavailable to real users with no economic cost to the attacker.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open multiple TCP connections — no special privileges, tokens, or on-chain transactions. The attack is trivially scriptable, repeatable, and leaves no persistent state on the attacker's side. The default `maxConcurrentCallsPerConnection = 5` means only ~20–40 TCP connections are needed to saturate a typical deployment.

### Recommendation

1. **Enforce a maximum subscription duration**: Add a configurable `maxSubscriptionDuration` to `ListenerProperties` and apply `.timeout(maxSubscriptionDuration)` in `listen()`, forcing clients to re-subscribe periodically.
2. **Enforce a global concurrent-subscriber ceiling**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleManagerImpl` used in the web3 module.
4. **Require a non-zero `limit` or a finite `endTime`** in `ConsensusController.toFilter()`, or at minimum document and enforce a server-side cap on open-ended subscriptions.

### Proof of Concept

```bash
# Open 10 connections, each with 5 indefinite subscriptions (50 total)
for i in $(seq 1 10); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe HikariCP pool exhaustion in metrics:
# hikaricp_connections_active / hikaricp_connections_max → 1.0
# Legitimate subscribeTopic calls begin returning UNAVAILABLE or timing out
```

Preconditions: network access to port 5600; listener type configured as `POLL`; no external load balancer enforcing connection limits.
Trigger: each background process holds an open gRPC stream; `PollingTopicListener` schedules a DB poll every 500 ms per stream.
Result: DB connection pool and `boundedElastic` scheduler saturated; new subscriptions from legitimate users are denied.
