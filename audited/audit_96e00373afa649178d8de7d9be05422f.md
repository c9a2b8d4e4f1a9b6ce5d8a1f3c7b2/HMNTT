### Title
Unbounded Metadata Reflection in `ConstructionPreprocess` Enables Memory and Bandwidth Exhaustion

### Summary
`ConstructionPreprocess()` directly assigns the caller-supplied `request.Metadata` map to the `options` map with no size, key-count, or value-length validation, then echoes the entire map back in `ConstructionPreprocessResponse.Options`. Because no HTTP body-size limit is enforced anywhere in the rosetta server stack, an unauthenticated attacker can send arbitrarily large metadata payloads that are fully reflected in the response, wasting server memory, CPU (JSON serialization), and network bandwidth on every request.

### Finding Description

**Exact code path:**

`rosetta/app/services/construction_service.go`, `ConstructionPreprocess()`, lines 304–320:

```go
options := make(map[string]any)
if len(request.Metadata) != 0 {
    options = request.Metadata          // ← entire attacker-controlled map, no size check
}

options[optionKeyOperationType] = operations[0].Type
...
return &rTypes.ConstructionPreprocessResponse{
    Options:            options,         // ← reflected verbatim to caller
    RequiredPublicKeys: requiredPublicKeys,
}, nil
```

**Root cause:** The code assumes `request.Metadata` is a small, well-formed map. There is no guard on the number of entries, the length of individual keys or values, or the total serialized size of the map.

**Why existing checks fail:**

1. No `http.MaxBytesReader` or equivalent body-size limit exists anywhere in the rosetta Go server. A grep for `MaxBytesReader`, `LimitReader`, and `ReadLimit` across all `rosetta/**/*.go` files returns zero matches.
2. The HTTP server in `main.go` (lines 220–227) sets only `IdleTimeout`, `ReadHeaderTimeout`, `ReadTimeout`, and `WriteTimeout` — none of which bound the request body size.
3. The `rosetta-sdk-go` asserter wrapping the endpoint does not validate the size or key count of the `Metadata` field in `ConstructionPreprocessRequest`.
4. The only validation performed before the metadata is used is `len(request.Metadata) != 0` — a truthiness check, not a size bound.

**Exploit flow:**

1. Attacker crafts a valid `ConstructionPreprocessRequest` with a minimal but valid `Operations` array (two `cryptoTransfer` operations summing to zero, satisfying `getOperationSlice` and `Preprocess`).
2. The `Metadata` field is populated with, e.g., 100,000 key-value pairs each with 1 KB values (~100 MB of JSON).
3. The server deserializes the body into `request.Metadata` (Go's `encoding/json` has no default limit).
4. `options = request.Metadata` copies the map reference.
5. The server serializes `options` back into the JSON response body and sends it to the attacker.
6. Steps 1–5 are repeated concurrently from multiple connections.

### Impact Explanation

Each request forces the server to:
- Allocate heap memory proportional to the metadata payload size.
- Serialize the entire map to JSON for the response.
- Transmit the response over the network.

With no rate limiting or body-size cap, a single attacker with a modest connection can sustain enough concurrent requests to exhaust server heap memory or saturate the write bandwidth, degrading or denying service to legitimate users. The impact is griefing / resource exhaustion with no economic cost to the attacker.

### Likelihood Explanation

The endpoint (`/construction/preprocess`) is publicly reachable in both online and offline modes (see `main.go` lines 111–119 and 152). No authentication is required. The attack requires only the ability to send HTTP POST requests with a crafted JSON body — trivially achievable by any external actor. The attack is repeatable and automatable.

### Recommendation

1. **Enforce an HTTP body-size limit** at the server level using `http.MaxBytesReader` in a middleware wrapper, e.g., 1–4 MB, applied before JSON decoding.
2. **Whitelist known metadata keys** in `ConstructionPreprocess`: instead of `options = request.Metadata`, copy only the keys the service actually understands (`memo`, and any other documented keys), discarding all others.
3. **Bound individual value lengths** for any accepted metadata key.
