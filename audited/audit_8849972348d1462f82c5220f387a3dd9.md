### Title
Accept-Encoding Validation Bypass via `gzip;q=0` Allows Uncompressed Large Response Delivery

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController` uses a naive `contains("gzip")` substring check that passes for values like `gzip;q=0`, which per RFC 7231 explicitly signals that gzip is *not acceptable*. An unprivileged attacker can craft this header to pass the guard while causing the HTTP layer (Spring embedded server or upstream proxy) to serve the opcode trace response uncompressed, potentially exhausting bandwidth and degrading service availability.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76:

```java
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

**Root cause:** The check performs only a substring match. It does not parse the HTTP quality factor (`q` parameter). RFC 7231 §5.3.4 defines `q=0` as "not acceptable." The value `gzip;q=0` contains the substring `"gzip"`, so the guard passes — but the value semantically means "do not use gzip."

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip;q=0`.
2. `validateAcceptEncodingHeader("gzip;q=0")` evaluates `"gzip;q=0".toLowerCase().contains("gzip")` → `true` → no exception thrown.
3. `throttleManager.throttleOpcodeRequest()` is called but only rate-limits; it does not enforce compression.
4. The opcode trace (potentially megabytes of stack/memory/storage data) is generated and returned.
5. The embedded Tomcat or any RFC-compliant upstream proxy/CDN reads `q=0`, correctly refrains from compressing, and delivers the full uncompressed payload.

**Why existing checks fail:**
- `throttleManager.throttleOpcodeRequest()` limits request frequency but does not prevent each allowed request from producing a massive uncompressed response.
- No `server.compression.*` configuration was found in the repository, meaning compression is not enforced server-side independently of the `Accept-Encoding` header — the guard is the sole mechanism.

### Impact Explanation
Each bypassed request can force the server to transmit an uncompressed opcode trace. Opcode traces with `stack=true`, `memory=true`, `storage=true` for complex contracts can be tens to hundreds of megabytes. Even at the throttle rate, repeated requests from multiple clients (or distributed sources) can saturate outbound bandwidth, cause response timeouts for legitimate users, and degrade the availability of the mirror node's web3 API. The opcode trace data itself is not semantically corrupted, but delivery reliability is compromised under load.

### Likelihood Explanation
No authentication or privilege is required — the endpoint is public once `hiero.mirror.web3.opcode.tracer.enabled=true`. The bypass requires only crafting a single HTTP header, which any HTTP client supports. The attack is trivially repeatable and scriptable. The only friction is the throttle, which limits but does not eliminate the impact.

### Recommendation
Replace the substring check with proper RFC 7231 quality-value parsing. Specifically, reject the request if `gzip` is absent **or** if its `q` value is `0`:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (!isGzipAccepted(acceptEncodingHeader)) {
        throw HttpClientErrorException.create(...);
    }
}

private boolean isGzipAccepted(String header) {
    if (header == null) return false;
    for (String token : header.split(",")) {
        String[] parts = token.trim().split(";");
        if (parts[0].trim().equalsIgnoreCase("gzip")) {
            for (int i = 1; i < parts.length; i++) {
                String[] qv = parts[i].trim().split("=");
                if (qv.length == 2 && qv[0].trim().equalsIgnoreCase("q")) {
                    try { return Double.parseDouble(qv[1].trim()) > 0; }
                    catch (NumberFormatException e) { return false; }
                }
            }
            return true; // gzip present, no q param → q defaults to 1.0
        }
    }
    return false;
}
```

Alternatively, use Spring's `MimeTypeUtils` or `MediaType` parsing utilities which handle quality factors natively.

### Proof of Concept
```bash
# Assuming the endpoint is enabled and accessible
curl -v \
  -H "Accept-Encoding: gzip;q=0" \
  "http://<mirror-node-host>/api/v1/contracts/results/<txHash>/opcodes?stack=true&memory=true&storage=true"

# Expected (vulnerable): HTTP 200 with full uncompressed body (no Content-Encoding: gzip header in response)
# Expected (patched):    HTTP 406 Not Acceptable
```

Confirm bypass by checking the response lacks `Content-Encoding: gzip` while returning a large body, versus a correctly rejected request when sending `Accept-Encoding: identity` (no gzip substring).
