### Title
Unbounded gRPC Connection Multiplexing Enables DB Pool Exhaustion and CPU Amplification

### Summary
The `grpcServerConfigurer()` function in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` (default: 5) but sets no limit on the total number of accepted connections. An unauthenticated attacker can open an arbitrary number of HTTP/2 connections, each carrying exactly 5 concurrent `subscribeTopic` streaming calls, multiplying total concurrent database-polling subscriptions without any single connection exceeding the per-connection cap. This exhausts the finite Hikari connection pool and drives CPU above 30% of baseline.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // ← NO serverBuilder.maxConnections(...) call
};
```

`NettyProperties.java`, line 14: `private int maxConcurrentCallsPerConnection = 5;`

**Root cause:** `NettyServerBuilder.maxConcurrentCallsPerConnection(5)` is the only resource guard applied. Netty's default for total accepted connections is `Integer.MAX_VALUE`. No `maxConnections`, no per-IP connection limit, no authentication, and no rate-limiting interceptor is registered anywhere in the gRPC server configuration.

**Exploit flow:**

Each `subscribeTopic` streaming RPC triggers `TopicMessageServiceImpl.subscribeTopic()` → `PollingTopicMessageRetriever.retrieve()` → `topicMessageRepository.findByFilter()` (via `TopicMessageRepositoryCustomImpl`, using `EntityManager`/JPA), which acquires a Hikari connection on every poll cycle (default: every 2 s throttled, 20 ms unthrottled). The `transactionOperationsReadOnly` `TransactionTemplate` (read-only JPA transaction) wraps these reads, holding a pool connection for the duration of each query.

With N attacker-controlled connections × 5 calls each = **5N concurrent polling subscriptions**. The Hikari pool defaults to 10 connections (`spring.datasource.hikari` has no explicit `maximumPoolSize` override in the grpc module; `CommonConfiguration` defers entirely to external config). At N ≥ 3 connections (15 concurrent subscriptions), the pool is saturated; threads queue waiting for connections, JVM thread-pool pressure rises, and CPU climbs due to context switching and connection-wait overhead.

**Why the existing check fails:** `maxConcurrentCallsPerConnection=5` is a *per-connection* ceiling. It is precisely designed to be bypassed by opening more connections — the attacker stays at exactly 5 calls/connection to remain compliant while scaling horizontally.

### Impact Explanation

- **Database connection pool saturation**: With Hikari's default pool of 10, ≥3 attacker connections (15 subscriptions) fully occupy the pool, blocking legitimate queries from the importer, REST API, and other gRPC clients.
- **CPU amplification**: Each subscription polls the DB on a `boundedElastic` scheduler thread. Hundreds of concurrent subscriptions generate continuous thread scheduling, query parsing, and result-set processing, measurably increasing CPU well above 30% compared to idle baseline.
- **Service degradation**: Legitimate `subscribeTopic` and `getNodes` callers receive `RESOURCE_EXHAUSTED` or timeout errors while the attack is sustained.
- **Persistence**: `subscribeTopic` with no `limit` or `consensusEndTime` streams indefinitely; the attacker holds connections open at zero marginal cost.

### Likelihood Explanation

- **No privileges required**: The gRPC port (5600) is publicly exposed with no authentication.
- **Trivial tooling**: Standard `grpcurl` or any gRPC client library supports HTTP/2 multiplexing; opening 20–50 connections with 5 streams each requires a single script.
- **Undetectable per-connection**: Each individual connection is within policy (≤5 calls), so per-connection monitoring does not flag the attack.
- **Repeatable**: The attacker can sustain the attack indefinitely by keeping streams open with no `limit` field set.

### Recommendation

1. **Add a total connection limit** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionsTotal(nettyProperties.getMaxConnections()); // e.g., default 100
   ```
   Add `maxConnections` to `NettyProperties` with a safe default (e.g., 100–200).

2. **Add per-IP connection limiting** via a Netty `ChannelHandler` or a gRPC `ServerInterceptor` that tracks and rejects connections beyond a per-source-IP threshold.

3. **Set an explicit Hikari pool size** in the grpc module's `application.yml` (`spring.datasource.hikari.maximum-pool-size`) sized to match the expected concurrent subscription load, and add a connection-acquisition timeout to fail fast rather than queue indefinitely.

4. **Add a global subscriber cap** in `TopicMessageServiceImpl` (the `subscriberCount` `AtomicLong` is already tracked — enforce a ceiling and reject new subscriptions when exceeded).

### Proof of Concept

```bash
# Open 20 connections, each with 5 concurrent indefinite subscribeTopic streams (100 total)
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Observe: hikaricp_connections_active reaches maximum, CPU rises >30%,
# legitimate clients receive RESOURCE_EXHAUSTED or connection timeouts.
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
