### Title
Missing Length Constraint on Base32 Alias Input Enables DoS via Memory Exhaustion and Degraded DB Query Performance

### Summary
The `getByAliasAndType()` method in `EntityServiceImpl` passes a user-supplied alias string through `decodeBase32()` with no length validation before sending the resulting byte array to `entityRepository.findByAlias()`. The GraphQL schema only enforces a character-set pattern (`^[A-Z2-7]+$`) on the alias field but imposes no maximum length, allowing any unauthenticated caller to submit an arbitrarily large valid Base32 string. This causes unbounded JVM heap allocation and forces the database to process an oversized parameter in its equality comparison, degrading entity resolution for all concurrent users.

### Finding Description
**Code path:**
1. `AccountController.account()` receives `AccountInput.alias` and calls `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)`.
2. `EntityServiceImpl.getByAliasAndType()` (line 30) calls `decodeBase32(alias)` with no prior length check.
3. `GraphQlUtils.decodeBase32()` (line 84) delegates directly to `BASE32.decode(base32)` — Apache Commons Codec's `Base32.decode()` allocates a `byte[]` proportional to the input length with no cap.
4. The resulting byte array is passed as parameter `?1` to the native query `select * from entity where alias = ?1 and deleted is not true`.

**Root cause / failed assumption:** The schema directive `@Pattern(regexp: "^[A-Z2-7]+$")` (account.graphqls line 90) validates only the character set, not the string length. There is no `@Size` or equivalent constraint anywhere in the input validation chain. The implementation assumes the alias will be a short, protocol-bounded value (Hedera aliases are at most ~50 bytes), but nothing enforces this.

**Why existing checks fail:**
- `@Pattern` passes for any string composed entirely of `[A-Z2-7]` characters, regardless of length.
- `validateOneOf()` in `AccountController` only checks that exactly one field is non-null.
- `decodeBase32()` has no guard.
- The repository query is a plain equality check with no server-side length rejection.

### Impact Explanation
An attacker sending repeated requests with a multi-megabyte Base32 alias string (all valid `[A-Z2-7]` characters) causes:
1. **JVM heap exhaustion**: Each request allocates a `byte[]` of ~5/8 the input length. A 10 MB alias string allocates ~6.25 MB per request; concurrent requests can exhaust heap and trigger GC storms or OOM.
2. **Database parameter overhead**: PostgreSQL must receive, buffer, and hash/compare the oversized bytea parameter for every request. Even with a B-tree or hash index on `alias`, computing the hash or traversing the index with a large value is O(n) in parameter size, increasing per-query CPU and I/O cost and degrading entity resolution latency for all users.
3. **Cascading effect on smart contract resolution**: Any component relying on alias-based entity lookup (e.g., contract account resolution) is affected during the degradation window.

Severity: Medium — no direct fund loss, but availability of entity resolution is impaired for all users with no authentication required.

### Likelihood Explanation
- **No authentication required**: The GraphQL `account` query is a public read endpoint.
- **Trivially repeatable**: A single HTTP client can flood the endpoint with large alias strings in a loop.
- **Low attacker sophistication**: Only knowledge of the GraphQL schema and valid Base32 characters is needed.
- **No rate limiting visible in the reviewed code**: No throttling or request-size cap is applied at the service layer.

### Recommendation
1. Add a `@Size(max = N)` constraint to the `alias` field in `AccountInput` (GraphQL schema or the corresponding Java DTO), where `N` reflects the maximum legitimate alias length (Hedera public-key aliases are ≤ 50 bytes → ≤ 80 Base32 characters).
2. Add an explicit length guard in `decodeBase32()` or at the start of `getByAliasAndType()`:
   ```java
   if (alias.length() > MAX_ALIAS_LENGTH) {
       throw new IllegalArgumentException("Alias exceeds maximum allowed length");
   }
   ```
3. Configure a global GraphQL request size limit (e.g., Spring Boot `spring.graphql.websocket.connection-init-timeout` and HTTP max body size) as a defense-in-depth measure.

### Proof of Concept
```bash
# Generate a 5 MB valid Base32 alias string (all 'A' characters are valid Base32)
ALIAS=$(python3 -c "print('A' * 8000000)")

# Send to the GraphQL endpoint (no authentication needed)
curl -X POST https://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"
```
Repeat this request concurrently (e.g., 20 parallel connections). Each request allocates ~5 MB on the JVM heap and sends a ~5 MB bytea parameter to PostgreSQL. Observe increased GC activity, heap pressure, and elevated query latency for legitimate alias lookups.
