### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent GET /api/v1/blocks Requests

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` performs no rate limiting, enforces no timestamp range bounds, and issues a blocking database query per request against a default pool of only 10 connections with a 20-second statement timeout. An unprivileged attacker sending ≥10 concurrent requests with open-ended timestamp filters can exhaust the pool, causing all subsequent block-time queries to queue for up to 20 seconds and return stale data, making the apparent block delay visible through the mirror node API exceed 500% of the 24-hour average.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:12` → `BlockController.getBlocks` → `rest/controllers/blockController.js:101-112` → `RecordFileService.getBlocks` → `rest/service/recordFileService.js:149-162` → `super.getRows(query, params)` → pool connection.

`getBlocks` calls `utils.buildAndValidateFilters(req.query, acceptedBlockParameters)` with the default `filterValidityChecks` validator. Unlike other endpoints (e.g., contract results), it does **not** call `parseTimestampFilters` with `validateRange=true`, so the `maxTimestampRange` (7-day default) is **never enforced** for the blocks endpoint. The resulting SQL is:

```sql
SELECT count, hash, name, ... FROM record_file
WHERE consensus_end >= $1 AND consensus_end <= $2
ORDER BY consensus_end DESC
LIMIT 100
```

With `timestamp=gte:0&timestamp=lte:9999999999.999999999` this performs a full table scan on `record_file`.

**Pool configuration** (`rest/dbpool.js:14`, `docs/configuration.md:556-557`):
- `maxConnections` = **10** (default)
- `statementTimeout` = **20 000 ms** (default)
- `connectionTimeout` = **20 000 ms** (default)

**No rate limiting** exists on the REST blocks endpoint at the application layer. The Traefik `inFlightReq`/`rateLimit` middleware shown in `charts/hedera-mirror-rosetta/values.yaml:149-166` is specific to the Rosetta service and is not applied to the REST API by default.

**Root cause:** The failed assumption is that the 10-connection pool and 20-second statement timeout together provide sufficient protection. They do not: 10 concurrent long-running queries hold all connections for up to 20 seconds each, and the `maxRepeatedQueryParameters=100` guard only limits parameter count per request, not concurrent request count.

### Impact Explanation
Any monitoring system (block explorer, alerting pipeline, consensus health check) that polls `GET /api/v1/blocks` to compute the 24-hour average block time will receive queued responses delayed by up to 20 seconds per request. With the pool fully saturated, the mirror node reports no new blocks for the duration of the attack. A monitoring tool computing "time since last reported block" will observe an apparent block delay that trivially exceeds 500% of the ~3-second Hedera average block time. This is a Denial-of-Service against the mirror node's block-reporting capability, not against consensus itself, but it directly affects any downstream system relying on this API for liveness detection.

### Likelihood Explanation
No authentication, no API key, no IP-based rate limit, and no per-endpoint concurrency cap are required. Any internet-accessible deployment is reachable. Sending 10 concurrent HTTP requests is trivially achievable with `curl --parallel`, `ab`, or any scripting language. The attack is repeatable indefinitely as long as the attacker maintains the concurrent request stream.

### Recommendation
1. **Enforce timestamp range for blocks**: Call `parseTimestampFilters` with `validateRange=true` inside `extractSqlFromBlockFilters`, or add an explicit check that rejects open-ended timestamp ranges (no lower or upper bound) for the blocks endpoint.
2. **Add per-IP concurrency/rate limiting**: Apply Traefik `inFlightReq` and `rateLimit` middleware to the REST API service, mirroring the Rosetta configuration.
3. **Increase pool size or add a request queue limit**: Raise `hiero.mirror.rest.db.pool.maxConnections` and add an HTTP-level concurrency cap (e.g., Express `express-rate-limit` or a reverse-proxy rule) so pool exhaustion cannot be triggered by a single client.
4. **Reduce `statementTimeout`** for the blocks query specifically, or add a query-level timeout shorter than the pool connection timeout.

### Proof of Concept

```bash
# Exhaust the 10-connection pool with open-ended timestamp scans
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>/api/v1/blocks?timestamp=gte:0&timestamp=lte:9999999999.999999999&limit=100" &
done
wait

# Now issue a legitimate block-time query and measure response latency
time curl -s "http://<mirror-node>/api/v1/blocks?limit=1"
# Expected: response delayed 10-20 seconds (pool queue timeout)
# Apparent block delay = response_delay / avg_block_time > 500%
```

const buildAndValidateFilters = (
  query,
  acceptedParameters,
  filterValidator = filterValidityChecks,
  filterDependencyChecker = filterDependencyCheck
) => {
  const {badParams, filters} = buildFilters(query);
  const {invalidParams, unknownParams} = validateAndParseFilters(filters, filterValidator, acceptedParameters);
  badParams.push(...invalidParams);
  badParams.push(...unknownParams);
  if (badParams.length > 0) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }

  if (filterDependencyChecker) {
    filterDependencyChecker(query);
  }

  return filters;
};
