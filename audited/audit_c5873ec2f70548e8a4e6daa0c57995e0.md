### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Ethereum Hash Lookups in `getContractResultsByTransactionIdOrHash`

### Summary
The `getContractResultsByTransactionIdOrHash` handler in `rest/controllers/contractController.js` issues 7 sequential/parallel DB queries per request with no rate limiting or concurrency cap. The REST API's default connection pool size is 10. Because `getDetailedContractResults` fires 5 simultaneous queries via `Promise.all`, just 2 concurrent requests from an unprivileged attacker holding valid Ethereum transaction hashes can exhaust the entire pool for up to 20 seconds, rendering the node unable to serve any other request.

### Finding Description

**Exact code path:**

`getContractResultsByTransactionIdOrHash` (lines 1120–1203) executes the following sequential/parallel DB calls for every request carrying a valid 32-byte Ethereum hash:

1. **Query 1** – `ContractService.getContractTransactionDetailsByHash(...)` (line 1134) — acquires 1 connection, releases it.
2. **Query 2** – `ContractService.getInvolvedContractsByTimestampAndContractId(...)` (line 1156) — acquires 1 connection, releases it.
3. **Queries 3–7** – `this.getDetailedContractResults(...)` (line 1166) calls `Promise.all` (lines 1205–1219) with **5 simultaneous queries**:
   - `ContractService.getContractResultsByTimestamps`
   - `TransactionService.getEthTransactionByTimestampAndPayerId`
   - `RecordFileService.getRecordFileBlockDetailsFromTimestamp`
   - `ContractService.getContractLogsByTimestamps`
   - `ContractService.getContractStateChangesByTimestamps` [1](#0-0) 

**Pool configuration:**

`rest/dbpool.js` initialises the `pg` pool with `max: config.db.pool.maxConnections` and `statement_timeout: config.db.pool.statementTimeout`. [2](#0-1) 

The documented defaults are `maxConnections = 10` and `statementTimeout = 20000 ms`. [3](#0-2) 

**No rate limiting on the REST API:**

`rest/server.js` applies only CORS, compression, logging, auth, and metrics middleware — no per-IP rate limiter or concurrency cap exists for this endpoint. [4](#0-3) 

The route is registered publicly with no authentication guard: [5](#0-4) 

**Root cause:** The `Promise.all` in `getDetailedContractResults` holds 5 pool connections simultaneously for the full duration of the slowest of those 5 queries (up to 20 s). With a pool of 10, only **2 concurrent requests** in the `Promise.all` phase are needed to saturate the pool. All subsequent requests queue on `connectionTimeoutMillis` (20 s) and then fail — this is the DoS condition itself.

### Impact Explanation

While the pool is exhausted, every REST API endpoint on that node fails with a connection-timeout error. Because the attack requires only 2 concurrent HTTP requests per node and Ethereum transaction hashes are publicly observable on-chain, an attacker can trivially target multiple REST API replicas simultaneously. Sustaining the attack requires sending a new pair of requests every ≤20 seconds (the statement timeout window). This can render 30 %+ of mirror-node REST API instances unresponsive to all traffic without any brute-force or privileged access.

### Likelihood Explanation

- **Precondition**: Any valid Ethereum transaction hash — freely available from block explorers or the mirror node's own `/contracts/results` listing endpoint.
- **Skill required**: None beyond basic HTTP tooling (`curl`, `ab`, `wrk`).
- **Repeatability**: Fully repeatable; the pool recovers after ≤20 s but the attacker simply re-fires.
- **Detection evasion**: Requests are syntactically valid and indistinguishable from legitimate traffic; no anomaly threshold is configured.

### Recommendation

1. **Add a per-IP rate limiter** (e.g., `express-rate-limit`) specifically for `/contracts/results/:transactionIdOrHash` before the handler is reached.
2. **Reduce `statementTimeout`** for this endpoint's queries to a value well below 20 s (e.g., 5 s) to shorten the exhaustion window.
3. **Increase `maxConnections`** or deploy a connection-pooling proxy (PgBouncer in transaction mode) in front of the REST API nodes to absorb burst demand.
4. **Add a global in-flight request concurrency cap** (e.g., via `express-slow-down` or a semaphore) so that no single endpoint can monopolise the pool.

### Proof of Concept

```bash
# Step 1: Obtain any valid Ethereum transaction hash from the public mirror node
HASH=$(curl -s "https://<mirror-node>/api/v1/contracts/results?limit=1" \
  | jq -r '.results[0].hash')

# Step 2: Fire 10 concurrent requests (2× the pool-exhaustion threshold)
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/contracts/results/${HASH}" &
done
wait

# Step 3: Immediately probe any other endpoint — it will time out or return 503
curl -v "https://<mirror-node>/api/v1/transactions?limit=1"
# Expected: connection timeout / pool exhausted error after ~20 s
```

Repeat Step 2 every 15 seconds to maintain continuous pool exhaustion.
