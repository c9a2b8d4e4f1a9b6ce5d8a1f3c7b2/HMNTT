### Title
Unbounded Concurrent gRPC Subscription Streams Enable Connection-Exhaustion DoS

### Summary
`PollingTopicMessageRetriever.retrieve()` imposes no per-user, per-IP, or global subscription count limit. The only server-side guard — `maxConcurrentCallsPerConnection` (default 5) — limits streams per TCP connection but places no cap on the number of TCP connections a single client may open. An unprivileged attacker can open an arbitrary number of connections, each carrying 5 polling streams, exhausting the database connection pool and the bounded-elastic scheduler queue and denying service to legitimate subscribers.

### Finding Description
**Exact code path:**

`PollingTopicMessageRetriever.retrieve()` (lines 45–63) creates a `PollingContext` per call and schedules a repeating DB-poll `Flux` on a shared `Schedulers.boundedElastic()` instance with no guard on how many such `Flux` chains may be active simultaneously.

```
retrieve()  →  Flux.defer(() -> poll(context))
               .repeatWhen(...)          // polls DB every 2 s (throttled) or 20 ms (unthrottled)
               .retryWhen(Retry.backoff(Long.MAX_VALUE, ...))   // retries forever
               .timeout(retrieverProperties.getTimeout(), ...)  // 60 s inactivity only
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) increments `subscriberCount` as a Micrometer gauge but enforces no ceiling — it is purely observational.

`GrpcConfiguration.java` (line 33) configures only:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```
No `maxConnections`, no IP-based throttle, no bucket4j rate limiter (contrast with `web3/ThrottleConfiguration.java` which does use bucket4j). A grep across all `grpc/**/*.java` for `maxConnections`, `rateLimiter`, `bucket4j`, or `Bucket` returns zero matches.

**Root cause / failed assumption:** The design assumes that `maxConcurrentCallsPerConnection=5` is a meaningful global bound. It is not — it is a per-connection limit, and the number of connections is unbounded.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no server-side connection limit).
2. On each connection, opens 5 `subscribeTopic` streams with `startTime=0` and no `limit` (infinite subscription).
3. Each stream invokes `retrieve(filter, true)` → `poll()` → `topicMessageRepository.findByFilter()` every 2 seconds.
4. With N=500 connections: 2 500 concurrent streams → 1 250 DB queries/second against a HikariCP pool of ~10–20 connections.
5. DB pool is exhausted; all subsequent queries block or fail; legitimate subscribers receive errors or never receive messages.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection=5`: only per-connection; attacker multiplies by opening more connections.
- `retrieverProperties.getTimeout()=60s`: inactivity timeout only; a live topic (or attacker-controlled topic with injected messages) keeps streams alive indefinitely.
- `subscriberCount` gauge: metric only, no enforcement.
- `Retry.backoff(Long.MAX_VALUE, ...)`: retries forever on error, meaning even streams that fail due to DB exhaustion immediately re-queue.

### Impact Explanation
Complete denial of service for the `subscribeTopic` gRPC endpoint. The database connection pool is exhausted, causing all new and existing subscriptions to fail. Because `Retry.backoff(Long.MAX_VALUE, ...)` is applied inside `retrieve()`, each attacker stream automatically retries on DB errors, sustaining the pressure. Legitimate clients — including production HCS consumers relying on gossip delivery — cannot receive topic messages for the duration of the attack.

### Likelihood Explanation
No authentication or API key is required to call `subscribeTopic`. The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to open many TCP connections, which is trivially achievable from a single host or a small botnet. The attack is repeatable and self-sustaining due to the infinite retry loop. The public `grpcurl` example in `docs/grpc/README.md` (line 16) demonstrates that the endpoint is intentionally publicly accessible.

### Recommendation
1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge()` / a custom `ServerInterceptor` that tracks connections per remote address.
3. **Add a per-IP stream rate limit** using a `ServerInterceptor` with bucket4j (already a dependency in the `web3` module), rejecting new `subscribeTopic` calls from IPs that exceed a threshold.
4. **Remove `Retry.backoff(Long.MAX_VALUE, ...)` or bound it** in `retrieve()` (line 58) to prevent exhausted streams from immediately re-queuing.
5. **Set `maxConnectionAge`** on the Netty server to force periodic connection recycling and limit long-lived connection accumulation.

### Proof of Concept
```bash
# Open 200 connections, each with 5 concurrent streams = 1000 total polling streams
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Observe: DB connection pool saturated (HikariCP active connections = max)
# Legitimate subscriber attempt:
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Result: hangs or returns UNAVAILABLE / deadline exceeded
```
