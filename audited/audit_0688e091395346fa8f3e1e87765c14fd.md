### Title
Silent Empty Response on `timestamp=gt:MAX_LONG` in `/api/v1/balances` Due to BigInt Overflow in `getOptimizedTimestampRange`

### Summary
When an unprivileged user sends `timestamp=gt:9223372036.854775807` (which equals `MAX_LONG` nanoseconds), the `gt`-to-`gte` conversion at line 253 of `rest/balances.js` computes `value + 1n = MAX_LONG + 1n`, making `lowerBound > upperBound`. The function returns `{}` at line 261, causing the API to silently return HTTP 200 with an empty balances payload instead of a 400 error. The question's stated trigger (`MAX_LONG-1`) is slightly off — the actual trigger is `MAX_LONG` itself.

### Finding Description

**Code path:**

`rest/balances.js`, `getOptimizedTimestampRange()`, lines 229–276:

```js
// line 232
let upperBound = constants.MAX_LONG;          // 9223372036854775807n

// lines 251-254
} else if (query.includes(utils.opsMap.gt)) {
  const gtValue = value + 1n;                 // ← overflow point
  lowerBound = utils.bigIntMax(lowerBound, gtValue);
}

// lines 260-262
if (lowerBound > upperBound) {
  return {};                                  // ← silent empty return
}
``` [1](#0-0) 

When `value = MAX_LONG = 9223372036854775807n`, `gtValue = MAX_LONG + 1n = 9223372036854775808n`. JavaScript `BigInt` does not overflow — it produces a value larger than `MAX_LONG`. Since `lowerBound (MAX_LONG+1) > upperBound (MAX_LONG)`, the function returns `{}`.

**Propagation:**

`getAccountBalanceTimestampRange` receives `{}` (lowerBound is `undefined`), returns `{}` at line 169–171. [2](#0-1) 

`getTsQuery` receives `{}` (lower is `undefined`), returns `{}` at line 279–281. [3](#0-2) 

`getBalances` hits `if (!tsQueryResult.query) { return; }` at line 115–117, leaving `res.locals` as the pre-set empty payload. [4](#0-3) 

**Root cause:** No upper-bound guard on the timestamp value before performing `value + 1n`. The assumption that any accepted timestamp value is strictly less than `MAX_LONG` is violated.

**Why existing checks fail:**

The OpenAPI schema pattern for `timestampQueryParam` is `^((eq|gt|gte|lt|lte|ne):)?\d{1,10}(\.\d{1,9})?$`. [5](#0-4) 

This allows seconds up to 10 digits (max `9999999999`) and nanoseconds up to 9 digits. The value `9223372036.854775807` (= `MAX_LONG` ns) passes this regex. No additional semantic check against `MAX_LONG` exists in `parseTimestampParam` or `filterValidityChecks` for the balances flow. [6](#0-5) 

### Impact Explanation
Any unprivileged caller receives HTTP 200 with `{timestamp: null, balances: [], links: {next: null}}` instead of HTTP 400. A client or monitoring system interpreting this as "no balances exist" rather than "invalid input" is silently misled. There is no data breach or privilege escalation; severity is **Low**.

### Likelihood Explanation
The trigger requires knowing `MAX_LONG` in nanoseconds (`9223372036.854775807`), which is public knowledge. No authentication is required. The request is trivially repeatable and requires no special tooling.

### Recommendation
Add an explicit guard in `getOptimizedTimestampRange` (or in the timestamp validation layer) that rejects any timestamp value ≥ `MAX_LONG` with a 400 `InvalidArgumentError` before performing the `+ 1n` / `- 1n` arithmetic. For example:

```js
} else if (query.includes(utils.opsMap.gt)) {
  if (value >= constants.MAX_LONG) {
    throw new InvalidArgumentError(`Invalid timestamp value: ${value}`);
  }
  const gtValue = value + 1n;
  lowerBound = utils.bigIntMax(lowerBound, gtValue);
}
```

Similarly guard the `lt` branch (`value - 1n` when `value === 0n`).

const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }
```

const parseTimestampParam = (timestampParam) => {
  // Expect timestamp input as (a) just seconds,
  // (b) seconds.mmm (3-digit milliseconds),
  // or (c) seconds.nnnnnnnnn (9-digit nanoseconds)
  // Convert all of these formats to (seconds * 10^9 + nanoseconds) format,
  // after validating that all characters are digits
  if (!timestampParam) {
    return '';
  }
  const tsSplit = timestampParam.split('.');
  if (tsSplit.length <= 0 || tsSplit.length > 2) {
    return '';
  }
  const seconds = /^(\d)+$/.test(tsSplit[0]) ? tsSplit[0] : 0;
  let nanos = tsSplit.length === 2 && /^(\d)+$/.test(tsSplit[1]) ? tsSplit[1] : 0;
  nanos = `${nanos}000000000`.substring(0, 9);
  return `${seconds}${nanos}`;
};
```
