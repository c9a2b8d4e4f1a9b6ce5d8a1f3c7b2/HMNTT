### Title
Log Source IP Spoofing via Unvalidated `X-Forwarded-For` Header in `responseHandler`

### Summary
`rest/server.js` configures Express with `app.set('trust proxy', true)`, which causes `req.ip` to unconditionally reflect the leftmost value of any client-supplied `X-Forwarded-For` header. On line 57 of `responseHandler.js`, this attacker-controlled value is written directly into access logs, allowing any unprivileged external user to forge the source IP recorded during incident response.

### Finding Description
**Root cause chain:**

1. `rest/server.js` line 58 sets `app.set('trust proxy', true)`. [1](#0-0) 

   With `trust proxy` set to the boolean `true`, Express trusts **all** hops in `X-Forwarded-For` without restriction and resolves `req.ip` to the **leftmost** (i.e., client-supplied) entry in that header — regardless of whether the request actually passed through a trusted proxy.

2. `responseHandler.js` line 57 logs `req.ip` verbatim: [2](#0-1) 

   No sanitisation, allowlist, or validation of the header value is performed before it reaches the log sink.

**Exploit flow:**
- Attacker sends: `GET /api/v1/transactions HTTP/1.1` with header `X-Forwarded-For: 192.168.1.1` (or any IP they wish to impersonate).
- Express resolves `req.ip` → `"192.168.1.1"`.
- `responseHandler` emits: `INFO 192.168.1.1 GET /api/v1/transactions in 12 ms: 200`.
- The attacker's real TCP source IP is never recorded.

**Why existing checks are insufficient:**
There are no checks. The `trust proxy: true` flag is a blanket trust grant with no IP-range restriction (Express supports `trust proxy` being set to a subnet or hop count to limit trust scope, but neither is used here). [3](#0-2) 

### Impact Explanation
Security analysts responding to an incident (e.g., abuse, scraping, or an attack originating from this API) will investigate a forged IP address. The real attacker remains unidentified in logs. This directly undermines forensic integrity and SIEM alerting that relies on source-IP correlation. Severity is **Medium** — it does not enable data exfiltration or privilege escalation on its own, but it actively degrades the effectiveness of every other detective control.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and a single HTTP request. Any client on the internet can send an arbitrary `X-Forwarded-For` header. The technique is well-known, trivially scriptable, and repeatable at will. Likelihood is **High**.

### Recommendation
Replace the blanket `trust proxy: true` with a specific trusted-proxy subnet or hop count matching the actual infrastructure (e.g., the load-balancer CIDR):

```js
// Instead of:
app.set('trust proxy', true);

// Use the actual upstream proxy address(es):
app.set('trust proxy', '10.0.0.0/8');   // or a hop count, e.g. 1
```

This causes Express to only honour `X-Forwarded-For` when the direct TCP peer is within the trusted range, so external clients cannot inject arbitrary IPs. If the deployment topology is unknown, a hop count of `1` is a safe conservative default that trusts only the immediately adjacent proxy.

### Proof of Concept
```bash
# Forge source IP as 203.0.113.99 (a public IP the attacker does not own)
curl -H "X-Forwarded-For: 203.0.113.99" \
     https://<mirror-node-host>/api/v1/transactions

# Resulting log line (actual attacker IP absent):
# INFO  203.0.113.99 GET /api/v1/transactions in 8 ms: 200
```
The real TCP source IP of the attacker does not appear anywhere in the log entry.
