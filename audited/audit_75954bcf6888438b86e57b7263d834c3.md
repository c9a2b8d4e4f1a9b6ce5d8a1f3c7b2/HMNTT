### Title
Missing `lower(timestamp_range) <= blockTimestamp` Guard on `entity` Branch Causes Current State to Shadow Historical State in `findActiveByEvmAddressAndTimestamp()`

### Summary
The `entity` branch of the UNION ALL in `findActiveByEvmAddressAndTimestamp()` omits the `lower(timestamp_range) <= ?2` predicate that the sibling method `findActiveByIdAndTimestamp()` correctly applies. Because the `entity_cte` filters only on the immutable `created_timestamp`, an entity whose state was last updated *after* `blockTimestamp` still passes the CTE filter and is emitted by the entity branch with its future-dated `timestamp_range = [T_update, ∞)`. The final `ORDER BY timestamp_range DESC LIMIT 1` then ranks that open-ended range above the correct `entity_history` row, returning the wrong (future) state for any historical EVM query.

### Finding Description

**Exact code path**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, `findActiveByEvmAddressAndTimestamp()`, lines 59–85.

The entity branch (lines 68–72) is:
```sql
select *
from entity e
where e.deleted is not true
  and e.id = (select id from entity_cte)
```
No `lower(e.timestamp_range) <= ?2` predicate exists.

The CTE (lines 60–66) filters only on `created_timestamp <= ?2` — an immutable field that never changes after entity creation.

The `entity_history` branch (lines 75–81) correctly applies `lower(eh.timestamp_range) <= ?2`.

The reference method `findActiveByIdAndTimestamp()` (lines 136–155) applies `lower(timestamp_range) <= ?2` on its entity branch (line 140), making the asymmetry explicit.

**Root cause**

`created_timestamp` is set once at entity creation and never updated. `lower(timestamp_range)` in the `entity` table advances to the timestamp of each subsequent update. After any update at `T_update > T_created`, the entity row carries `timestamp_range = [T_update, ∞)` while `created_timestamp` remains `T_created`. The CTE's `created_timestamp <= blockTimestamp` check passes whenever `blockTimestamp >= T_created`, even when `blockTimestamp < T_update`. The entity branch then emits the post-update row unconditionally.

**Exploit flow**

1. Entity E with `evm_address = A` is created at `T1` → entity table: `timestamp_range = [T1, ∞)`, `created_timestamp = T1`.
2. E is updated at `T2 > T1` → entity_history: `timestamp_range = [T1, T2)`, entity table: `timestamp_range = [T2, ∞)`, `created_timestamp = T1` (unchanged).
3. Attacker issues a historical EVM call (e.g., `eth_call` with `blockNumber` mapping to `T_mid` where `T1 ≤ T_mid < T2`).
4. `findActiveByEvmAddressAndTimestamp(A, T_mid)` executes:
   - CTE: `created_timestamp (T1) <= T_mid` → entity ID resolved ✓
   - Branch 1: entity row `[T2, ∞)` returned — no upper-bound check ✗
   - Branch 2: entity_history row `[T1, T2)` returned — `lower(T1) <= T_mid` ✓
   - `ORDER BY timestamp_range DESC LIMIT 1`: `[T2, ∞)` sorts above `[T1, T2)` → entity row wins.
5. Caller receives the post-`T2` state instead of the correct `T_mid` state.

**Why existing checks fail**

- `entity_cte` guards only creation time, not the validity window of the current state.
- `deleted is not true` is orthogonal to timestamp correctness.
- No application-layer re-validation of the returned row's `timestamp_range` against `blockTimestamp` exists in `CommonEntityAccessor` (lines 66–84).

### Impact Explanation
Any historical `eth_call`, `eth_getBalance`, `eth_getCode`, or contract-state read routed through `CommonEntityAccessor.getEntityByEvmAddressAndTimestamp()` / `getEntityByEvmAddressTimestamp()` (lines 66–84 of `CommonEntityAccessor.java`) will silently return the entity's *current* attributes (balance, bytecode, key, memo, etc.) for a past block. This corrupts the historical view of the ledger: balances appear inflated or deflated, contracts appear to have code they did not yet have, and deleted-then-recreated accounts may appear with wrong state. Because the mirror node is the authoritative source for historical EVM replay, this undermines the integrity guarantee of historical queries.

### Likelihood Explanation
No privilege is required. Any caller that can issue a standard JSON-RPC call with a historical block number (a universally supported feature of Ethereum-compatible APIs) can trigger the path. The precondition — an entity that has been updated at least once — is satisfied by virtually every active account or contract on the network. The bug is deterministic and repeatable for any entity whose `lower(timestamp_range)` in the entity table exceeds the queried `blockTimestamp`.

### Recommendation
Add `lower(e.timestamp_range) <= ?2` to the entity branch of `findActiveByEvmAddressAndTimestamp()` (and the identical branch in `findActiveByEvmAddressOrAliasAndTimestamp()`), mirroring the guard already present in `findActiveByIdAndTimestamp()`:

```sql
(
    select *
    from entity e
    where e.deleted is not true
      and lower(e.timestamp_range) <= ?2   -- ADD THIS
      and e.id = (select id from entity_cte)
)
```

Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp()` (lines 95–121), which has the identical structural defect.
