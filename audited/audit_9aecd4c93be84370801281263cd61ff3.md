### Title
Unauthenticated Alias-Based Historical Balance Queries Trigger Expensive Unindexed UNION ALL Scans on `entity_history`, Enabling DB CPU Exhaustion

### Summary

The `/account/balance` Rosetta endpoint accepts alias-based account identifiers with historical block identifiers from any unauthenticated caller. Each such request causes `getCryptoEntity()` to execute a `UNION ALL` query across both `entity` and `entity_history` tables using a GiST `timestamp_range @>` containment check, with no composite index on `(alias, timestamp_range)` in `entity_history`. With no rate limiting enabled by default and no statement timeout configured by default, an attacker can flood the endpoint with concurrent requests to drive database CPU well above the 30% baseline threshold.

### Finding Description

**Exact code path:**

`AccountAPIService.AccountBalance()` in `rosetta/app/services/account_service.go` (lines 41–79) accepts any `AccountBalanceRequest`. When `request.BlockIdentifier != nil`, it calls `a.RetrieveBlock(ctx, request.BlockIdentifier)` to resolve a historical `ConsensusEndNanos` timestamp, then passes it to `a.accountRepo.RetrieveBalanceAtBlock()`. [1](#0-0) 

`RetrieveBalanceAtBlock()` immediately calls `ar.getCryptoEntity(ctx, accountId, consensusEnd)`. [2](#0-1) 

`getCryptoEntity()` (lines 225–265) detects `accountId.HasAlias() == true` and selects the `selectCryptoEntityByAlias` query:

```sql
select id, deleted, key, timestamp_range
from entity
where alias = @alias and timestamp_range @> @consensus_end
union all
select id, deleted, key, timestamp_range
from entity_history
where alias = @alias and timestamp_range @> @consensus_end
order by timestamp_range desc
``` [3](#0-2) [4](#0-3) 

**Root cause — missing composite index on `entity_history(alias, timestamp_range)`:**

`entity_history` is created with `LIKE entity` (no `INCLUDING INDEXES`), so it inherits column definitions but not the indexes from `entity`. The only indexes created are a GiST index on `timestamp_range` alone and a B-tree on `(id, lower(timestamp_range))`. [5](#0-4) [6](#0-5) 

There is no index on `alias` in `entity_history`. The query `WHERE alias = @alias AND timestamp_range @> @consensus_end` therefore requires either a full sequential scan of `entity_history` or a GiST index scan on `timestamp_range` followed by a recheck on `alias` — both are expensive at production scale (millions of history rows). Unlike `selectCryptoEntityById`, this query also has **no `LIMIT` clause**, so all matching rows are returned. [3](#0-2) [7](#0-6) 

**No authentication or rate limiting by default:**

`main.go` wires only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no auth, no rate limiting, no per-IP throttle at the application layer. [8](#0-7) 

The Helm chart defines a Traefik middleware chain (rate limit: 10 req/s, in-flight: 5) but it is **only deployed when `global.middleware: true`**, which defaults to `false`. [9](#0-8) [10](#0-9) 

**No statement timeout by default:**

`GetDbWithContext()` only applies a timeout when `statementTimeout > 0`. The default value is `0`, meaning queries run unbounded. [11](#0-10) 

### Impact Explanation

On a production Hedera mirror node, `entity_history` accumulates one row per entity state change across the entire chain history — potentially hundreds of millions of rows. A single alias-based historical balance query forces a full scan of this table. Sending tens of concurrent such requests (trivially achievable with a single script) can saturate PostgreSQL CPU, degrading all mirror node services (REST API, web3 API, importer) that share the same database. This meets the stated threshold of >30% CPU increase over the 24-hour baseline. Severity: **High** (availability impact, no authentication required, no default mitigation).

### Likelihood Explanation

The attack requires zero privileges — the Rosetta API is a public HTTP endpoint. The attacker needs only to know any valid alias (obtainable from the public Hedera network or mirror REST API) or can use arbitrary byte strings (the query will still execute the full scan and return empty). The attack is trivially scriptable, repeatable, and requires no special tooling beyond `curl` or any HTTP client. Default deployments without the Traefik middleware enabled are fully exposed.

### Recommendation

1. **Add a composite GiST/B-tree index** on `entity_history(alias, timestamp_range)` (or at minimum a B-tree on `alias` with a separate GiST on `timestamp_range`) to make the alias + containment filter efficient.
2. **Add `LIMIT 1`** to `selectCryptoEntityByAlias` (consistent with `selectCryptoEntityById`) since only the most recent owner of the alias is needed.
3. **Enable `statementTimeout`** in the default configuration to bound runaway queries.
4. **Enable the Traefik middleware by default** (`global.middleware: true`) or implement application-level rate limiting in the Go server.

### Proof of Concept

```bash
# Step 1: Obtain any valid alias from the mirror REST API (or use arbitrary bytes)
ALIAS="0x3a2103..."  # any hex-encoded alias

# Step 2: Obtain a historical block index (any old block)
BLOCK_INDEX=1000000

# Step 3: Flood /account/balance with concurrent alias + historical block requests
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/account/balance \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"account_identifier\": {\"address\": \"${ALIAS}\"},
      \"block_identifier\": {\"index\": ${BLOCK_INDEX}}
    }" &
done
wait
# Each request triggers a UNION ALL scan of entity_history with no alias index.
# 50 concurrent scans on a large entity_history table will spike DB CPU.
```
