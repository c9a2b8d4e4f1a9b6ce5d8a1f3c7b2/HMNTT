### Title
Cache Key Injection via Whitespace Variations in `CachedPreparsedDocumentProvider.getDocumentAsync()`

### Summary
`CachedPreparsedDocumentProvider.getDocumentAsync()` uses the raw, un-normalized query string as the Caffeine cache key. An unprivileged attacker can send semantically identical GraphQL queries with minor whitespace variations (extra spaces, tabs, newlines) that each produce a distinct cache key, filling the bounded cache with attacker-controlled entries and evicting legitimate cached documents, forcing all users to incur repeated parse-and-validate overhead.

### Finding Description
**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26, inside `getDocumentAsync()`:

```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

`executionInput.getQuery()` returns the raw query string exactly as submitted by the client. No normalization (whitespace stripping, canonicalization, or hashing after normalization) is applied before it is used as the `AsyncCache<String, PreparsedDocumentEntry>` key.

The cache is configured in `CacheProperties` with `maximumSize=1000` and `expireAfterWrite=1h`. An attacker submitting 1,000 whitespace-variant forms of the same query (e.g., inserting one extra space, newline, or tab at different positions) fills the entire cache. Caffeine's LRU/size-based eviction then removes legitimate entries, causing cache misses for all real users for up to one hour per attack cycle.

The `GraphQlConfiguration` static block sets `maxWhitespaceTokens(1000)` and `maxCharacters(10000)` on the parser, but these are per-query limits on token counts and character length — they do not prevent whitespace-variant strings from being treated as distinct cache keys. There is no rate limiting on the GraphQL endpoint (the `ThrottleConfiguration`/`ThrottleManagerImpl` applies only to the web3 contract-call path).

### Impact Explanation
The preparsed document cache exists specifically to avoid the CPU cost of parsing and validating GraphQL queries on every request. With the cache polluted to its 1,000-entry maximum, every legitimate query misses the cache and triggers a full parse + validation cycle (including `MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` traversal). This degrades response latency and increases CPU load for all users of the service. The attack is self-renewing: after 1 hour the attacker can repeat it. No economic damage to the network occurs, but service quality is measurably degraded — consistent with the stated "griefing" scope.

### Likelihood Explanation
No authentication or special privilege is required to submit GraphQL queries to the mirror node. The attack is trivially scriptable: generate 1,000 queries by inserting one additional space at each character position of a valid query. The attacker needs no knowledge of internal state. The attack is repeatable every hour (cache TTL) and is not detectable by existing instrumentation as anything other than a high cache-miss rate.

### Recommendation
Normalize the query string before using it as a cache key. The simplest approach is to strip and collapse all whitespace sequences to a single space before keying:

```java
String normalizedKey = executionInput.getQuery().strip().replaceAll("\\s+", " ");
return cache.get(normalizedKey, key -> parseAndValidateFunction.apply(executionInput));
```

A more robust alternative is to parse the query into an AST and serialize it canonically (graphql-java's `AstPrinter.printAst()` produces a canonical form), or to SHA-256 hash the normalized string to bound key size. Additionally, consider adding per-IP or global rate limiting on the GraphQL endpoint analogous to the web3 `ThrottleConfiguration`.

### Proof of Concept
1. Identify a valid query, e.g.: `{ account(input:{entityId:{shard:0,realm:0,num:1}}) { balance } }`
2. Generate 1,000 variants by inserting 1–1,000 extra spaces at a fixed position (e.g., after `{`).
3. Send all 1,000 variants to the GraphQL endpoint in rapid succession (no auth required).
4. Now send the canonical query repeatedly and observe that it never hits the cache (parse-and-validate is invoked on every request), confirming cache eviction.
5. Measure increased response latency for all concurrent legitimate users.
