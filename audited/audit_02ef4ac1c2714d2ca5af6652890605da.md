### Title
Unbounded Subscription Fan-Out Enables DB Query Amplification via One-Shot Safety Check

### Summary
`TopicMessageServiceImpl.subscribeTopic()` schedules a one-shot `Schedulers.boundedElastic()` DB query for every new subscription via `Flux.merge(safetyCheck, live)`. There is no global subscriber cap, no per-IP connection limit, and no authentication on `ConsensusController.subscribeTopic()`. An unprivileged attacker can open arbitrarily many gRPC connections (each carrying up to 5 concurrent calls per the default `maxConcurrentCallsPerConnection`) and cause a synchronized burst of `topicMessageRetriever.retrieve()` DB queries ~1 second after connection establishment.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any unauthenticated gRPC call and delegates directly to `topicMessageService.subscribeTopic()` with no rate-limiting or subscriber-count guard. [1](#0-0) 

Inside `TopicMessageServiceImpl.subscribeTopic()` (lines 67–70), a `Mono.delay(1s)` safety check is constructed for **every** subscription:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [2](#0-1) 

`isComplete()` returns `false` whenever `filter.getEndTime() == null` (lines 204–205), which is the default for live subscriptions — so the `filter` predicate never suppresses the check. [3](#0-2) 

When the safety check fires (`current == null`), `missingMessages()` unconditionally calls `topicMessageRetriever.retrieve(gapFilter, false)` — a full DB query (lines 142–149): [4](#0-3) 

**Root cause / failed assumption:** The design assumes the safety check is a rare, low-frequency event. It fires once per subscription, but there is no bound on the number of concurrent subscriptions. The `subscriberCount` AtomicLong (lines 89–90) is a metric gauge only — it is never checked to reject new subscriptions. [5](#0-4) 

**Why `Schedulers.boundedElastic()` is insufficient:** The bounded elastic scheduler limits thread concurrency (default `10 × CPU cores`) but has a task queue of 100,000 entries. All N queued safety-check tasks will eventually execute and issue DB queries; the scheduler only serializes them slightly, it does not drop or reject them.

**Why `maxConcurrentCallsPerConnection = 5` is insufficient:** This limits calls *per TCP connection* to 5, but there is no limit on the number of TCP connections an attacker may open. [6](#0-5) [7](#0-6) 

### Impact Explanation
An attacker opening C connections × 5 calls each = 5C simultaneous subscriptions causes 5C concurrent `retrieve()` DB queries to fire ~1 second later. At C = 200 connections (trivially achievable), that is 1,000 simultaneous DB queries. Because the queries are unbounded in scope (no `endTime`, no `limit` unless the attacker sets one), each query can be a full table scan of `topic_message`. This can saturate the DB connection pool, spike CPU/IO on the database, and degrade or deny service to legitimate subscribers. The attacker can repeat the burst by cycling connections.

### Likelihood Explanation
No authentication, no API key, no IP-based rate limiting, and no subscriber cap are present in the code. Any internet-reachable deployment is exploitable with a simple gRPC client loop. The attack requires only the ability to open TCP connections to port 5600 and send a valid `ConsensusTopicQuery` protobuf — a capability any unprivileged user has. The attack is repeatable and scriptable.

### Recommendation
1. **Add a global subscriber cap**: Check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if a configurable maximum is exceeded.
2. **Add per-IP / per-connection subscription limits** at the gRPC interceptor layer.
3. **Limit safety-check query scope**: Always enforce a short `endTime` window and a `maxPageSize` cap on the `gapFilter` passed to `retrieve()` in the `current == null` branch of `missingMessages()`.
4. **Consider making the safety check conditional on actual gap evidence** rather than firing unconditionally for every live subscription.
5. **Expose `maxConnections`** as a configurable Netty property alongside `maxConcurrentCallsPerConnection`.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hedera.mirror.api.proto import consensus_topic_query_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 300  # 300 connections × 5 calls = 1500 subscriptions

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = 1  # any valid topic
    # No endTime set → isComplete() always returns false → safety check always fires
    threads = []
    for _ in range(5):  # maxConcurrentCallsPerConnection default
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    # After ~1 second, 5 DB queries fire from this connection

pool = [threading.Thread(target=open_subscriptions, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in pool: t.start()
# Result: ~1500 concurrent topicMessageRetriever.retrieve() DB queries fire simultaneously
```
