### Title
Cache Collision via 32-bit `Arrays.hashCode()` Key in `CACHE_NAME_EVM_ADDRESS` Allows Wrong Entity Resolution for EVM Address Lookups

### Summary
`findByEvmAddressAndDeletedIsFalse()` uses `@spelHelper.hashCode(#alias)` — which delegates to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole cache key for the `CACHE_NAME_EVM_ADDRESS` Caffeine cache. Because the key space is only 2^32 while EVM addresses are 20 bytes (2^160 possible inputs), collisions are trivially computable (~2^16 iterations by birthday paradox). Two distinct EVM addresses sharing the same 32-bit hash will map to the same cache slot, causing a lookup for one address to return the cached entity of the other.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:
```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

`Arrays.hashCode(byte[])` computes:
```
result = 1
for each byte b: result = 31 * result + b   (all mod 2^32)
```
This produces a signed 32-bit `int` — only ~4.3 billion distinct values for a 20-byte (160-bit) input space.

**Root cause:** The cache key is a lossy 32-bit hash of a 20-byte address. No secondary equality check on the actual byte content is performed after a cache hit. The `unless = "#result == null"` guard only prevents caching of empty results; it does not prevent a non-null result for address `A` from being served for a colliding address `B`.

**Exploit flow:**

1. Attacker identifies a target entity `T` with known EVM address `addr_T` (e.g., a high-value contract or privileged account).
2. Attacker computes `h_T = Arrays.hashCode(addr_T)`.
3. Attacker iterates CREATE2 salts (controlling `salt` and `bytecode`) to find an address `addr_A` such that `Arrays.hashCode(addr_A) == h_T`. By birthday paradox this requires ~2^16 ≈ 65,536 iterations — trivial.
4. Attacker deploys a contract at `addr_A` (registering it as a real entity in the mirror node).
5. Attacker triggers a `eth_call` or similar RPC that causes `findByEvmAddressAndDeletedIsFalse(addr_A)` to execute. Entity_A is fetched from DB and stored in cache under key `h_T`.
6. Any subsequent call that resolves `addr_T` via `findByEvmAddressAndDeletedIsFalse(addr_T)` hits the cache with key `h_T` and receives `entity_A` instead of `entity_T`.
7. `CommonEntityAccessor.getEntityByEvmAddressAndTimestamp()` (line 66–71) returns the wrong entity with no post-cache address verification.

**Why existing checks fail:**
- `unless = "#result == null"` only skips caching null results; it does not validate that the cached entity's EVM address matches the queried address.
- There is no secondary key component (e.g., the actual byte array) in the cache key expression.
- The Caffeine cache itself performs exact key equality on the integer key — it cannot distinguish two different addresses that hash to the same integer.

### Impact Explanation
Any EVM address lookup that goes through `findByEvmAddressAndDeletedIsFalse` can be poisoned to return a different entity. This affects: entity type resolution (account vs. contract), balance and nonce reads, contract code existence checks, and any downstream logic in `CommonEntityAccessor` that relies on the returned `Entity` object. An attacker can make the mirror node report that address `T` is a different entity (e.g., a non-existent or attacker-controlled contract), disrupting `eth_call` results, `eth_getBalance`, and contract interaction simulations. This constitutes reorganization of which entity is seen as active for a given address without requiring privileged access.

### Likelihood Explanation
The attack requires no special privileges — only the ability to submit transactions (deploy a contract via CREATE2) and trigger RPC calls. Finding a 32-bit hash collision against a target address requires ~65,536 hash computations on average, achievable in milliseconds on commodity hardware. The attacker fully controls the CREATE2 salt and bytecode, making the search space entirely under their control. The attack is repeatable: after cache eviction (TTL expiry), the attacker can re-poison the cache. No cryptographic hardness assumption protects `Arrays.hashCode`.

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the full byte-array content as the key, so the cache performs exact-match keying:

```java
key = "T(java.util.Arrays).toString(#alias)"
// or wrap in a dedicated value type with proper equals/hashCode
key = "new org.hiero.mirror.web3.cache.ByteArrayKey(#alias)"
```

A `ByteArrayKey` wrapper that holds the `byte[]` and implements `equals` via `Arrays.equals` and `hashCode` via `Arrays.hashCode` (used only for bucket selection, with `equals` as the collision guard) is the standard fix. This ensures two different byte arrays never resolve to the same cache entry even if their 32-bit hashes collide.
