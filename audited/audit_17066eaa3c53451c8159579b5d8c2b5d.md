### Title
Missing Rate Limiting on `GET /api/v1/network/registered-nodes` Enables DoS via Thread Pool Exhaustion

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module is publicly accessible with no authentication and no rate limiting. An unprivileged attacker can flood this endpoint with thousands of requests per second, each triggering a database query, saturating the Spring Boot thread pool and rendering the mirror node unable to serve legitimate topology discovery requests to registered nodes.

### Finding Description

**Exact code path:**

`NetworkController.getRegisteredNodes()` at [1](#0-0)  handles `GET /api/v1/network/registered-nodes` with no authentication annotation, no rate-limit guard, and no throttle check. Each request unconditionally invokes `networkService.getRegisteredNodes(request)`, which calls the repository method `findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()`: [2](#0-1) 

This executes a native SQL query against the `registered_node` table on every request.

**Root cause — failed assumption:**

The only filters registered for the `rest-java` module are `LoggingFilter` (logs requests) and `MetricsFilter` (records byte counts). [3](#0-2)  The `RestJavaConfiguration` registers only a `ShallowEtagHeaderFilter` — no rate-limiting filter exists. [4](#0-3) 

The `WebMvcConfiguration` adds only argument resolvers and formatters — no interceptor-based throttling. [5](#0-4) 

The Bucket4j/rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module for contract-call endpoints and is **not wired into the `rest-java` module at all**. [6](#0-5) 

The only input constraint is `@Max(MAX_LIMIT)` on the `limit` parameter of `RegisteredNodesRequest`, which bounds result size but does nothing to limit request frequency. [7](#0-6) 

**Exploit flow:**

1. Attacker sends a high-frequency flood of `GET /api/v1/network/registered-nodes` requests (no credentials required).
2. Each request passes through `LoggingFilter` → `MetricsFilter` → `NetworkController.getRegisteredNodes()` → `NetworkServiceImpl` → `RegisteredNodeRepository` (native SQL query).
3. The Tomcat/Spring Boot thread pool (default: 200 threads) becomes saturated holding threads blocked on DB I/O.
4. Legitimate registered nodes polling for topology updates receive connection timeouts or HTTP 503 responses.
5. Nodes that cannot retrieve updated topology may fail to discover new peers, causing them to operate on stale network views.

### Impact Explanation

The `registered-nodes` endpoint is the authoritative source for network topology discovery. If the mirror node stops responding to these queries, registered nodes cannot receive topology updates, potentially isolating a significant fraction of the network from peer discovery. The impact aligns with the stated scope: denial of topology-update service to ≥30% of registered nodes without any brute-force cryptographic attack, purely through HTTP flood. The mirror node's single shared thread pool means this endpoint competes with all other endpoints, causing collateral degradation of the entire mirror node service.

### Likelihood Explanation

This requires zero privileges — no API key, no authentication token, no special network position. Any internet-connected host can execute this attack using standard HTTP tooling (`wrk`, `ab`, `hey`). The attack is trivially repeatable, cheap to sustain, and requires no prior knowledge beyond the public API path. The absence of any server-side rate limiting makes this a low-skill, high-impact attack.

### Recommendation

1. **Immediate:** Add a Bucket4j-based rate-limiting filter to the `rest-java` module, mirroring the pattern already used in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, and apply it to `/api/v1/network/**` endpoints.
2. **Per-IP limiting:** Implement per-source-IP token buckets (e.g., via a `HandlerInterceptor` or `OncePerRequestFilter`) so a single attacker cannot consume the global budget.
3. **Infrastructure layer:** Deploy an ingress-level rate limiter (e.g., nginx `limit_req`, AWS WAF, or Kubernetes ingress annotations) as a defense-in-depth measure independent of application code.
4. **Thread pool isolation:** Consider configuring a dedicated, bounded thread pool for the registered-nodes endpoint to prevent cross-endpoint saturation.

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
