### Title
Unbounded Response Body Read in `fetchMirrorNodeNetwork` Enables Memory Exhaustion via Malicious/MITMed Mirror Node

### Summary
`fetchMirrorNodeNetwork` in `pinger/mirror_node_client.go` passes `resp.Body` directly to `json.NewDecoder(...).Decode(&payload)` with no `io.LimitReader` cap. An attacker who controls or can MITM the configured mirror node REST endpoint can stream an arbitrarily large JSON body, causing the pinger process to allocate memory proportional to the response size within the HTTP client's timeout window. On a high-bandwidth internal network this can reach gigabytes, starving the pinger of memory needed to submit transactions.

### Finding Description
**Exact location:** `pinger/mirror_node_client.go`, lines 95–98.

```go
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

`resp.Body` is the raw HTTP response stream with no size cap applied before decoding. Go's `encoding/json` decoder accumulates all decoded array elements into `payload.Nodes []nodeEntry` in heap memory. The larger the `nodes` array in the response, the more memory is allocated.

**Root cause / failed assumption:** The code assumes the mirror node REST endpoint is trusted and will return a reasonably sized response. There is no `io.LimitReader` guard and no `Content-Length` validation before decoding begins.

**Existing check — HTTP client timeout (10 s default):**
`pinger/config.go` line 90–95 sets `mirrorNodeClientTimeout` (default `10s`) and `buildNetworkFromMirrorNodes` constructs `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}`. In Go, `http.Client.Timeout` covers the entire request lifecycle including body reads, so it does bound *time* — but not *bytes*. On a 1 Gbps internal link (common for co-located services), 10 seconds allows ~1.25 GB of data to be streamed and decoded into heap memory before the timeout fires. On a 10 Gbps link the figure is ~12.5 GB.

**Retry amplification:** `buildNetworkFromMirrorNodes` retries up to `mirrorNodeClientMaxRetries + 1` times (default 11 attempts). Each attempt re-issues the request and re-decodes the body, so the attacker can trigger repeated large allocations with exponential backoff delays between them.

### Impact Explanation
The pinger process is responsible for submitting periodic Hedera transactions to verify network liveness. If the OS is forced into heavy swap due to gigabyte-scale heap allocations, the pinger's transaction submission latency increases sharply, reducing throughput. Because `buildNetworkFromMirrorNodes` is called at startup (via `newClient`) and on retry, a sustained attack keeps the pinger in a degraded or OOM-killed state, preventing it from submitting transactions and thus degrading observable network processing throughput by more than 30% relative to baseline.

### Likelihood Explanation
**Precondition:** The attacker must control or MITM the mirror node REST endpoint configured in `HIERO_MIRROR_PINGER_REST` (for `network=other`) or the SDK's built-in mirror node URL. In a private/enterprise deployment the mirror node is often on the same internal network as the pinger, making ARP/DNS spoofing or a compromised mirror node a realistic vector. No authentication or TLS certificate pinning is enforced by the pinger's `http.Client`. The attack is repeatable on every pinger restart or retry cycle.

### Recommendation
Wrap `resp.Body` with `io.LimitReader` before passing it to the JSON decoder. Choose a limit that comfortably exceeds any legitimate response (e.g., 10 MB for a paginated node list):

```go
import "io"

const maxMirrorNodeResponseBytes = 10 * 1024 * 1024 // 10 MB

var payload nodesEnvelope
limited := io.LimitReader(resp.Body, maxMirrorNodeResponseBytes)
if err := json.NewDecoder(limited).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

Additionally, validate `Content-Length` before reading if present, and consider enforcing TLS with certificate verification for the mirror node endpoint.

### Proof of Concept
1. Stand up a malicious HTTP server that responds to `GET /api/v1/network/nodes` with HTTP 200 and streams an infinite (or very large) JSON body:
   ```json
   {"nodes": [{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"x","ip_address_v4":"","port":50211}]}, ... repeated millions of times ...]}
   ```
2. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<malicious-server>`.
3. Start the pinger. Observe via `top`/`/proc/<pid>/status` that `VmRSS` grows rapidly until the 10-second timeout fires or the process is OOM-killed.
4. With `mirrorNodeClientMaxRetries=10` (default), the pinger retries 11 times, each time re-allocating the large buffer, keeping memory pressure sustained and preventing successful transaction submission. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
