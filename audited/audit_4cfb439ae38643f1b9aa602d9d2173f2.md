### Title
Missing Rate Limiting in Rosetta Middleware Chain Enables Unbounded /construction/submit Flooding

### Summary
The `MetricsMiddleware` in `rosetta/app/middleware/metrics.go` is a pure observability wrapper with no rate limiting or IP-based blocking. The entire rosetta middleware chain — CORS → Tracing → Metrics → Router — contains no throttling layer, allowing any unprivileged external user to flood `/construction/submit` with unlimited requests per second. Each request is faithfully instrumented by `requestInflightGauge` and forwarded to `ConstructionSubmit`, which attempts SDK-level transaction execution with no per-client quota.

### Finding Description

**Middleware chain** (`rosetta/main.go`, lines 217–219):
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
``` [1](#0-0) 

No rate-limiting middleware exists anywhere in this chain. The only middleware layers are observational (metrics, tracing) or permissive (CORS).

**MetricsMiddleware** (`rosetta/app/middleware/metrics.go`, lines 76–83) wraps every incoming request with Prometheus instrumentation — including `requestInflightGauge` (`hiero_mirror_rosetta_request_inflight`) — and unconditionally passes it to the next handler: [2](#0-1) 

**ConstructionSubmit** (`rosetta/app/services/construction_service.go`, lines 332–363) performs no per-IP or per-client quota check. After an `IsOnline()` guard and transaction unmarshalling, it calls `hiero.TransactionExecute` directly: [3](#0-2) 

A grep across all rosetta Go sources for `rate`, `ratelimit`, `throttle` returns zero hits inside the rosetta package — the only throttle implementation in the repository is in the unrelated `web3` Java module (`ThrottleManagerImpl`), which is not wired into the rosetta service.

**Exploit flow:**
1. Attacker crafts a syntactically valid but semantically tampered hex-encoded transaction (e.g., replayed transaction ID, wrong node account, or garbage payload that passes `hex.DecodeString` but fails at the Hiero network level).
2. Attacker sends thousands of POST requests/second to `/construction/submit` with no authentication.
3. Each request traverses CORS → Tracing → Metrics (increments `hiero_mirror_rosetta_request_inflight`) → `ConstructionSubmit`.
4. For malformed payloads: fails at `unmarshallTransactionFromHexString`, but only after consuming a goroutine, CPU, and a metrics counter increment/decrement cycle.
5. For structurally valid payloads: `hiero.TransactionExecute` is called, consuming SDK client resources and sending traffic to Hiero network nodes.
6. No blocking, no 429, no IP ban occurs at any point.

### Impact Explanation
- **Resource exhaustion**: Goroutine pool, CPU, and memory on the rosetta process are consumed proportionally to request rate with no upper bound enforced by the application.
- **Hiero network flooding**: Valid-but-tampered transactions reach `hiero.TransactionExecute` and are forwarded to configured network nodes, amplifying the attack surface beyond the mirror node itself.
- **Metrics pollution**: `hiero_mirror_rosetta_request_inflight` is artificially inflated, degrading the reliability of operational alerting based on this gauge.
- Severity: **High** — availability impact on the rosetta service and downstream Hiero nodes; no privileges required.

### Likelihood Explanation
The attack requires zero authentication, zero on-chain funds, and only a basic HTTP client. The attacker needs only a valid hex-encoded transaction skeleton (obtainable from public Rosetta API documentation or by calling `/construction/payloads` once). The attack is trivially repeatable and scriptable, and the absence of any server-side counter makes it undetectable until resource exhaustion manifests.

### Recommendation
1. **Add a rate-limiting middleware** in `rosetta/main.go` between `CorsMiddleware` and `TracingMiddleware`, using a token-bucket or leaky-bucket algorithm keyed on client IP (e.g., `golang.org/x/time/rate` with a per-IP `sync.Map` of limiters, or a library such as `github.com/ulule/limiter`).
2. **Apply stricter limits specifically to `/construction/submit`** (e.g., 5–10 req/s per IP) compared to read-only endpoints.
3. **Set `http.Server` concurrency limits** (`MaxHeaderBytes`, connection limits via `nethttp.Server` or a reverse proxy like nginx/envoy) as a defense-in-depth layer.
4. **Deploy behind a reverse proxy** (nginx, Envoy, or a cloud load balancer) with connection-rate and request-rate limits enforced at the network edge before traffic reaches the Go process.

### Proof of Concept
```bash
# Requires: a valid hex transaction skeleton (obtainable from /construction/payloads)
SIGNED_TX="0x<valid_hex_encoded_transaction>"
ENDPOINT="http://<rosetta-host>:5700/construction/submit"

# Flood with 1000 concurrent requests using GNU parallel or wrk
seq 1 1000 | parallel -j 1000 \
  curl -s -X POST "$ENDPOINT" \
  -H "Content-Type: application/json" \
  -d "{\"network_identifier\":{\"blockchain\":\"Hiero\",\"network\":\"testnet\"},\"signed_transaction\":\"$SIGNED_TX\"}"

# Observe: no 429 responses, all requests processed
# Observe Prometheus metric climbing without bound:
curl http://<rosetta-host>:5700/metrics | grep hiero_mirror_rosetta_request_inflight
```

func (c *constructionAPIService) ConstructionSubmit(
	_ context.Context,
	request *rTypes.ConstructionSubmitRequest,
) (*rTypes.TransactionIdentifierResponse, *rTypes.Error) {
	if !c.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
	if rErr != nil {
		return nil, rErr
	}

	hashBytes, err := hiero.TransactionGetTransactionHash(transaction)
	if err != nil {
		return nil, errors.ErrTransactionHashFailed
	}

	hash := tools.SafeAddHexPrefix(hex.EncodeToString(hashBytes))
	transactionId, _ := hiero.TransactionGetTransactionID(transaction)
	log.Infof("Submitting transaction %s (hash %s) to node %s", transactionId,
		hash, transaction.GetNodeAccountIDs()[0])

	_, err = hiero.TransactionExecute(transaction, c.sdkClient)
	if err != nil {
		log.Errorf("Failed to execute transaction %s (hash %s): %s", transactionId, hash, err)
		return nil, errors.AddErrorDetails(
			errors.ErrTransactionSubmissionFailed,
			"reason",
			fmt.Sprintf("%s", err),
		)
	}
```
