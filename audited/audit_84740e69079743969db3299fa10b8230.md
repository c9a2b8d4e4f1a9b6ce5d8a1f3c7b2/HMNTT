### Title
Unbounded Database Query Amplification via Unique Opaque EVM Address Flooding in `getEntityIdFromEvmAddress()`

### Summary
`getEntityIdFromEvmAddress()` in `rest/service/entityService.js` issues a direct database query on every invocation with no service-layer result caching. Because the response cache only stores successful (2xx) responses, 404 results from non-existent EVM addresses are never cached. An unauthenticated attacker flooding the API with unique opaque EVM addresses forces one uncached database query per request, enabling a non-network DoS against the database tier.

### Finding Description

**Exact code path:**

`getEncodedId()` (lines 118–137) branches on `entityId.evmAddress === null`:

```js
// rest/service/entityService.js lines 122-124
return entityId.evmAddress === null
  ? entityId.getEncodedId()
  : await this.getEntityIdFromEvmAddress(entityId, requireResult);
```

When the input is a 40-hex-char address that does not match the long-zero prefix (i.e., an opaque EVM address), `parseFromString()` in `rest/entityId.js` (lines 252–253) sets `evmAddress` to the hex string and `num` to `null`, routing every such request into `getEntityIdFromEvmAddress()`.

`getEntityIdFromEvmAddress()` (lines 90–104) performs a raw database query with zero caching:

```js
// rest/service/entityService.js lines 91
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
```

`getRows()` in `rest/service/baseService.js` (line 56) calls directly into the PostgreSQL pool:

```js
return (await this.pool().queryQuietly(query, params)).rows;
```

**Why the response cache does not help:**

`responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` (line 95) only stores responses when the status code is a success:

```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```

A non-existent EVM address causes `getEntityIdFromEvmAddress()` to throw `NotFoundError` (line 94), which resolves to a 404. 404 is not a success code, so it is never written to Redis. The cache key is also URL-derived (MD5 of `req.originalUrl`), so every unique EVM address is a guaranteed cache miss regardless.

**No rate limiting in the REST layer:**

`rest/server.js` registers no rate-limiting middleware. The throttle configuration found (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate web3/JSON-RPC module, not to the Node.js REST API.

**Affected endpoints** (all call `getEncodedId` or `getEntityIdFromEvmAddress` directly):
- `GET /api/v1/accounts/:idOrAliasOrEvmAddress`
- `GET /api/v1/contracts/:contractId` and sub-routes
- `GET /api/v1/tokens/:tokenId`
- `GET /api/v1/topics/:topicId/messages`
- `GET /api/v1/transactions/:transactionIdOrHash`
- `GET /api/v1/balances`

### Impact Explanation
Each request with a unique opaque EVM address consumes one PostgreSQL connection and executes a full indexed scan on the `entity` table's `evm_address` column. Sustained flooding exhausts the database connection pool, causing query queuing, latency spikes, and eventual service unavailability for all API consumers. Because the REST API is the primary read interface for the mirror node, this degrades the entire data-access layer. The impact is a non-network DoS affecting availability of the mirror node REST API.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. Any external user can craft valid 40-hex-char opaque EVM addresses (e.g., incrementing a counter: `0xdeadbeef000000000000000000000000000000XX`). The attack is trivially scriptable with `curl` or any HTTP client in a loop. The attacker needs only a network connection to the public API endpoint. The attack is repeatable and stateless — no session or prior interaction is needed.

### Recommendation
1. **Add a service-layer negative-result cache**: Cache `null` results from `getEntityIdFromEvmAddress()` with a short TTL (e.g., 5–30 seconds) using an in-process LRU cache (e.g., `lru-cache`) or Redis. This collapses repeated queries for the same non-existent address.
2. **Cache 404 responses at the response layer**: Extend `responseCacheUpdateHandler` to also cache 404 responses with a short TTL, so repeated requests for the same unknown address are served from Redis without hitting the database.
3. **Add rate limiting to the REST API**: Integrate `express-rate-limit` (or equivalent) as middleware in `rest/server.js` to cap requests per IP per second, particularly for parameterized lookup endpoints.
4. **Validate EVM address format early**: Reject addresses that are syntactically valid but structurally implausible (e.g., all-zero or obviously synthetic) before issuing a DB query.

### Proof of Concept

```bash
# Generate and flood with unique opaque EVM addresses (none exist in DB)
for i in $(seq 1 10000); do
  ADDR=$(printf "deadbeef%032x" $i)
  curl -s "https://<mirror-node>/api/v1/accounts/0x${ADDR}" &
done
wait
```

Each request:
1. Passes `EntityId.isValidEntityId()` — 40 hex chars is a valid EVM address format.
2. `parseFromString()` detects non-long-zero prefix → sets `evmAddress = deadbeef...`, `num = null`.
3. `getEncodedId()` routes to `getEntityIdFromEvmAddress()`.
4. A `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query is issued to PostgreSQL.
5. No row found → `NotFoundError` → HTTP 404.
6. 404 is not cached → next unique address repeats from step 1.

Sustained at modest concurrency (e.g., 100 parallel connections), this saturates the PostgreSQL connection pool and degrades or denies service to all legitimate API users.
