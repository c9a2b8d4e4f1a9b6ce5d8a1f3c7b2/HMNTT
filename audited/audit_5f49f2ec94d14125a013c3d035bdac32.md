### Title
Unauthenticated gRPC `getNodes()` Endpoint Susceptible to Resource Exhaustion via Unbounded Concurrent Connections

### Summary
The `getNodes()` endpoint in `NetworkController` accepts unauthenticated requests with no global rate limiting, no per-IP throttle, and no cap on the total number of concurrent connections. The only defense — `maxConcurrentCallsPerConnection = 5` — is a per-connection limit trivially bypassed by opening multiple connections. Each call drives blocking database work through `Schedulers.boundedElastic()`, and a flood of concurrent calls exhausts the DB connection pool and the bounded elastic thread pool, rendering the gRPC service unable to serve any client.

### Finding Description
**Code path:**

`NetworkController.getNodes()` ( [1](#0-0) ) accepts any `AddressBookQuery` with no authentication check and no rate-limiting guard. It immediately delegates to `NetworkServiceImpl.getNodes()`.

`NetworkServiceImpl.getNodes()` ( [2](#0-1) ) performs three synchronous blocking DB calls on the calling thread (lines 61–65), then constructs a `Flux` that pages through the address book using `RepeatSpec` with `Long.MAX_VALUE` repeats and `withScheduler(Schedulers.boundedElastic())` (line 72). Each page iteration calls `transactionOperations.execute()` (line 80) — a blocking DB operation — on a `boundedElastic()` worker thread. [3](#0-2) 

The only server-level protection is `maxConcurrentCallsPerConnection = 5` set in `GrpcConfiguration` ( [4](#0-3) ) sourced from `NettyProperties` ( [5](#0-4) ). This is a **per-connection** limit, not a global one. No `maxConnections`, no global concurrent-call cap, and no per-IP rate limiter is configured anywhere in the gRPC module. [6](#0-5) 

The sole `GrpcInterceptor` in the codebase only sets an endpoint-context label for table-usage tracking and passes the call through unconditionally — no throttling logic. [7](#0-6) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` constitutes a sufficient global guard. It does not — an attacker controls how many connections they open.

**Exploit flow:**
1. Attacker opens *N* HTTP/2 connections to port 5600.
2. On each connection, sends 5 concurrent `getNodes(limit=0, file_id=0.0.102)` RPCs.
3. Total concurrent active calls = *N × 5* with no server-side ceiling.
4. Each call executes blocking DB queries via `transactionOperations.execute()` on `boundedElastic()` threads (default cap: `10 × availableProcessors`, e.g. 40 on a 4-core host).
5. DB connection pool (typically 10–20 connections) is exhausted; `boundedElastic()` threads block waiting for a DB connection.
6. Thread pool fills up; new tasks are queued (up to 100 000) then rejected with `RejectedExecutionException`, propagating errors to all subscribers.
7. Legitimate clients receive errors or hang indefinitely.

### Impact Explanation
When the DB connection pool and `boundedElastic()` thread pool are exhausted, every subsequent `getNodes()` call — including those from legitimate SDK clients bootstrapping node lists — fails or hangs. Clients that cannot discover network nodes cannot submit transactions, effectively causing a network-wide transaction submission outage from the mirror node's perspective. The `pageDelay` of 250 ms per page ( [8](#0-7) ) means each call holds resources for the full duration of address book traversal, amplifying the impact of each connection.

### Likelihood Explanation
No authentication is required. The endpoint is publicly documented and reachable on port 5600. Opening hundreds of HTTP/2 connections is trivial with standard tooling (`grpcurl`, `ghz`, custom scripts). The attack is repeatable, requires no special knowledge beyond the public proto definition ( [9](#0-8) ), and can be sustained indefinitely. The optional GCP `maxRatePerEndpoint: 250` infrastructure hint ( [10](#0-9) ) is not enforced at the application level and is noted as requiring additional HPA changes to take effect.

### Recommendation
1. **Global concurrent-call cap:** Add `serverBuilder.maxConcurrentCallsPerConnection(...)` alongside a `NettyServerBuilder.maxConnectionAge(...)` and configure a global call limit via a `ServerInterceptor` that tracks in-flight calls with an `AtomicInteger` and rejects with `RESOURCE_EXHAUSTED` when exceeded.
2. **Per-IP rate limiting:** Add a `ServerInterceptor` (similar to the web3 `ThrottleConfiguration` bucket4j pattern) that enforces a per-source-IP RPS limit on `getNodes`.
3. **Enforce a hard `limit` ceiling:** In `NetworkServiceImpl.getNodes()`, cap `filter.getLimit()` to a configurable maximum (e.g., 1000) so `limit=0` cannot trigger `Long.MAX_VALUE` repeats.
4. **Non-blocking DB access:** Move the blocking `transactionOperations.execute()` calls in `page()` off `boundedElastic()` onto a dedicated, bounded DB scheduler with a connection-pool-aligned thread count to prevent cascading exhaustion.

### Proof of Concept
```bash
# Install ghz: https://ghz.sh
# Open 200 connections, 5 streams each = 1000 concurrent getNodes calls
ghz --insecure \
    --proto network_service.proto \
    --call com.hedera.mirror.api.proto.NetworkService.getNodes \
    --data '{"file_id":{"fileNum":102},"limit":0}' \
    --connections 200 \
    --concurrency 1000 \
    --duration 60s \
    localhost:5600

# Observe: legitimate clients begin receiving UNAVAILABLE / RESOURCE_EXHAUSTED
grpcurl -plaintext \
    -d '{"file_id":{"fileNum":102},"limit":0}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes
# Expected result during attack: hangs or returns error — node discovery fails
```

    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }
