### Title
Unbounded Request Body Consumption in Rosetta `/mempool/transaction` Endpoint Due to Missing `http.MaxBytesReader`

### Summary
The `mempoolAPIService` in `rosetta/app/services/mempool_service.go` registers a `/mempool/transaction` endpoint whose HTTP handler (provided by the upstream `rosetta-sdk-go` `MempoolAPIController`) reads and JSON-decodes the full request body into memory before invoking `MempoolTransaction()`. No `http.MaxBytesReader` or equivalent byte-level cap is applied anywhere in the middleware chain. While a `ReadTimeout` of 5 seconds (default) provides partial mitigation, it is a time-based limit, not a byte-based limit, allowing an attacker on a fast link to push tens-to-hundreds of megabytes per connection and exhaust process memory across multiple concurrent connections.

### Finding Description

**Exact code path:**

`rosetta/app/services/mempool_service.go` lines 30–35 — `MempoolTransaction()` immediately returns `ErrNotImplemented`, but the upstream `rosetta-sdk-go` `MempoolAPIController` (registered at line 90 of `rosetta/main.go`) performs `json.NewDecoder(r.Body).Decode(&mempoolTransactionRequest)` before ever calling the service method. This decode step reads the raw body stream into memory.

`rosetta/main.go` lines 217–227 — the middleware chain is:
```
MetricsMiddleware → TracingMiddleware → CorsMiddleware → http.Server
```
None of these layers wrap `r.Body` with `http.MaxBytesReader`. The `http.Server` struct sets `ReadTimeout` (default 5 s per docs) but sets no `MaxBytes` field.

**Root cause:** Go's `net/http` package provides `http.MaxBytesReader` specifically to cap the number of bytes read from a request body. It is never called anywhere in the rosetta codebase (confirmed: `grep -r "MaxBytesReader"` returns zero matches under `rosetta/`). The only protection is `ReadTimeout`, which is time-based.

**Why `ReadTimeout` is insufficient:** `ReadTimeout` closes the connection after N nanoseconds of reading, but during that window the JSON decoder actively buffers data in the Go heap. On a 1 Gbps link, 5 seconds allows ~625 MB per connection. With the Traefik `inFlightReq: amount: 5` middleware being an optional Kubernetes-only deployment artifact (not enforced at the application layer), a direct deployment accepts unlimited concurrent connections, each consuming up to `bandwidth × ReadTimeout` bytes of heap.

### Impact Explanation
An unprivileged attacker with no credentials sends repeated large POST bodies to `POST /mempool/transaction`. Each request causes the `rosetta-sdk-go` controller to allocate heap memory proportional to the body size during JSON decoding. With multiple concurrent connections, aggregate heap pressure can trigger OOM-kill of the rosetta process. Because the rosetta mirror node is a stateless API layer, killing it on ≥30% of deployed instances degrades or halts block/transaction query availability for the network's data consumers. The endpoint is publicly reachable (exposed via ingress at `/rosetta/mempool` per `charts/hedera-mirror-rosetta/values.yaml` line 131).

### Likelihood Explanation
No authentication is required. The endpoint accepts any POST body. The attack is trivially scriptable: open N concurrent connections and stream large JSON-padded bodies. The `ReadTimeout` raises the required bandwidth per connection but does not prevent the attack — it merely bounds each individual request's contribution. A botnet or a single host with a fast uplink can sustain the attack indefinitely. The endpoint being permanently `ErrNotImplemented` means operators are unlikely to monitor it for abuse.

### Recommendation
Wrap `r.Body` with `http.MaxBytesReader` in a custom middleware applied before the router, limiting request bodies to a small fixed cap (e.g., 1 MB, consistent with the largest legitimate Rosetta request):

```go
func bodyLimitMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, 1<<20) // 1 MB
        next.ServeHTTP(w, r)
    })
}
```

Insert this before `MetricsMiddleware` in `rosetta/main.go`. Additionally, enforce `inFlightReq` limits at the application layer rather than relying solely on optional Traefik configuration.

### Proof of Concept

```bash
# Generate a ~200 MB body (well within 5s on a fast link)
python3 -c "
import json, sys
payload = {'network_identifier': {'blockchain':'Hiero','network':'mainnet'},
           'transaction_identifier': {'hash': 'A'*200_000_000}}
sys.stdout.write(json.dumps(payload))
" > big_body.json

# Fire 10 concurrent requests directly at the rosetta port
for i in $(seq 1 10); do
  curl -s -X POST http://<rosetta-host>:5700/mempool/transaction \
       -H 'Content-Type: application/json' \
       --data-binary @big_body.json &
done
wait
# Observe rosetta process RSS grow; repeat until OOM-kill
```
