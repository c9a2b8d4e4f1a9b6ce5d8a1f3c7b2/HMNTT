### Title
Unbounded `node.id` Array Parameter Enables DoS via Unconstrained JDBC Array Serialization

### Summary
`NetworkNodeRequest.nodeIds` has no `@Size` constraint, unlike the analogous `RegisteredNodesRequest.registeredNodeIds` which is capped at `@Size(max = 2)`. An unauthenticated attacker can submit an arbitrarily large number of `node.id=eq:X` query parameters, which flow without any count check through the service layer into a `Long[]` that is serialized into a native JDBC call, consuming heap memory proportional to the number of supplied values and enabling repeated denial-of-service against the REST service.

### Finding Description

**Exact code path:**

`NetworkNodeRequest.nodeIds` is declared as a plain `List<NumberRangeParameter>` with only `@RestJavaQueryParam` — no `@Size` annotation: [1](#0-0) 

Compare with `RegisteredNodesRequest.registeredNodeIds`, which correctly applies `@Size(max = 2)`: [2](#0-1) 

`RequestParameterArgumentResolver.validateAndAddParameter()` detects `List` as a multi-value field and passes all supplied values directly to the binder with no count guard: [3](#0-2) 

`binder.validate()` is called, but because `nodeIds` carries no `@Size` constraint, JSR-303 validation passes regardless of list size: [4](#0-3) 

In `NetworkServiceImpl.getNetworkNodes()`, every `EQ` entry is added to a `HashSet<Long>` and then converted to a `Long[]` with no size check: [5](#0-4) 

That `Long[]` is passed directly to the native-query repository method: [6](#0-5) 

where Hibernate/JDBC must serialize the entire array into the `= any(:nodeIds)` bind parameter before the query is sent to PostgreSQL: [7](#0-6) 

**Root cause:** The `nodeIds` field in `NetworkNodeRequest` is missing the `@Size` upper-bound constraint that is present on every other multi-value parameter in the codebase. The failed assumption is that HTTP-layer URL-length limits are a sufficient guard; they are not a reliable application-level control and vary by deployment configuration.

### Impact Explanation
Each `node.id=eq:X` value contributes a `Long` (8 bytes) to the `HashSet`, a `Long` to the `Long[]`, and a serialized 8-byte element in the JDBC array type sent over the wire. With Tomcat's default `maxParameterCount` of 10 000, an attacker can force allocation of multiple large in-memory structures per request. Concurrent flood requests multiply this effect, leading to heap exhaustion (`OutOfMemoryError`) or sustained GC pressure that degrades or crashes the REST service. Because the `/api/v1/network/nodes` endpoint is unauthenticated and publicly reachable, this directly threatens the availability of the mirror-node REST API, which downstream clients use to discover consensus-node addresses — a prerequisite for submitting transactions to the network.

### Likelihood Explanation
No authentication or rate-limiting is required. The endpoint is a standard HTTP GET, trivially scriptable. The attacker needs only a URL with many repeated `node.id=eq:N` parameters, which any HTTP client can produce. The attack is repeatable and stateless, making it suitable for sustained automated flooding. The missing constraint is a single-line oversight compared to sibling DTOs, so it is unlikely to be caught by code review without a direct comparison.

### Recommendation
Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds`, consistent with the pattern already used in `RegisteredNodesRequest`:

```java
// NetworkNodeRequest.java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 2)   // add this — mirrors RegisteredNodesRequest pattern
private List<NumberRangeParameter> nodeIds = List.of();
```

A maximum of 2 is appropriate for range semantics (one lower bound + one upper bound, or two equality values). If multiple equality values must be supported, set a small explicit cap (e.g., `max = 25`, matching `MAX_LIMIT`) and document it. Additionally, enforce `server.tomcat.max-parameter-count` at the deployment level as a defence-in-depth measure.

### Proof of Concept

```bash
# Generate a request with 500 node.id equality parameters (well within default URL limits)
PARAMS=$(python3 -c "print('&'.join(f'node.id=eq:{i}' for i in range(500)))")
curl -s "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS" -o /dev/null

# Repeat concurrently to amplify heap pressure
for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS" -o /dev/null &
done
wait
```

Each request forces the service to allocate a `HashSet<Long>` of 500 entries, convert it to a `Long[500]`, and serialize it into a JDBC array — all before the query even reaches PostgreSQL. Fifty concurrent such requests create 50 simultaneous large allocations, driving heap pressure and GC overhead. Scaling the parameter count toward Tomcat's `maxParameterCount` (default 10 000) amplifies the per-request allocation proportionally.
