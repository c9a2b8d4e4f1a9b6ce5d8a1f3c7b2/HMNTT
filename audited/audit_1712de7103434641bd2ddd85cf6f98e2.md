### Title
Unauthenticated gRPC Reflection API Flooding via Absent Connection-Rate Quota in `grpcServerConfigurer()`

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection=5` on the Netty server, with no connection-rate limit, no total-connection cap, no per-IP throttle, and no application-level request quota for the gRPC module. The gRPC reflection service (`grpc.reflection.v1.ServerReflection` and `grpc.reflection.v1alpha.ServerReflection`) is publicly exposed without authentication. An unprivileged attacker can open an unbounded number of connections and issue sequential reflection calls at full line rate, saturating the shared `applicationTaskExecutor` thread pool and driving CPU consumption well above baseline.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:
```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `grpcServerConfigurer()` applies exactly one guard: `maxConcurrentCallsPerConnection=5`. This limits *simultaneous in-flight calls on a single connection* but imposes no constraint on:
- The number of connections from a single IP or in total (`maxConnections` / `maxConnectionAge` / `maxConnectionIdle` are never set)
- The sequential call rate on any connection
- The aggregate request rate across all connections

There is no application-level rate-limiting in the gRPC module at all. The `web3` module has a full bucket4j `ThrottleConfiguration` (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`), but nothing equivalent exists under `grpc/`. The sole `GrpcInterceptor` (test path `grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`) only sets `EndpointContext` and passes the call through unconditionally.

**Reflection API exposure:** The Helm chart (`charts/hedera-mirror-grpc/values.yaml`, lines 93–95 and 140–141) explicitly routes both `grpc.reflection.v1alpha.ServerReflection` and `grpc.reflection.v1.ServerReflection` through the ingress and GCP gateway. Spring gRPC auto-registers `ProtoReflectionService` when it is on the classpath. No authentication or authorization is applied to this service.

**Exploit flow:**
1. Attacker opens *N* TCP/HTTP2 connections to port 5600 (no connection limit enforced).
2. On each connection, attacker issues up to 5 concurrent `ServerReflection/ServerReflectionInfo` streaming calls (the per-connection limit).
3. Each call sends a `list_services` or `file_containing_symbol` request in a tight loop.
4. Each request dispatches to the shared `applicationTaskExecutor`, performs a lookup in the cached reflection index, serializes a protobuf `ServerReflectionResponse`, and writes it to the network.
5. With *N* connections × 5 concurrent calls × high sequential rate, the executor queue fills, thread context-switching overhead grows, and CPU consumption rises proportionally.

**Why the existing check is insufficient:** `maxConcurrentCallsPerConnection=5` only throttles parallelism within one connection. An attacker with a single host can trivially open hundreds of connections (HTTP/2 connection establishment is cheap), multiplying effective concurrency to 5×N with no server-side enforcement.

### Impact Explanation
The shared `applicationTaskExecutor` services all gRPC calls including `ConsensusService/subscribeTopic` and `NetworkService/getNodes`. Flooding it with reflection calls degrades or starves legitimate subscriber traffic. At sufficient connection count (e.g., 200 connections × 5 concurrent calls = 1000 concurrent reflection dispatches), CPU and thread-pool saturation is achievable on a standard JVM deployment, meeting the 30%+ resource-increase threshold. The reflection responses also expose the full service/method/type schema to the attacker, aiding further targeted exploitation.

### Likelihood Explanation
No privileges, credentials, or special tooling are required. `grpcurl` (a standard CLI tool) supports scripted reflection queries. A single attacker machine with a modest number of threads can open hundreds of HTTP/2 connections. The attack is repeatable, stateless, and leaves no persistent side-effects on the server, making detection and attribution harder. The infrastructure-level `maxRatePerEndpoint: 250` in the GCP backend policy (`charts/hedera-mirror-grpc/values.yaml`, line 69) is optional, deployment-specific, and not present in all environments (e.g., bare-metal or non-GCP deployments).

### Recommendation
1. **Add connection-level limits in `grpcServerConfigurer()`:**
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(5), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionIdle(Duration.ofSeconds(30), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionAgeGrace(Duration.ofSeconds(5), TimeUnit.MILLISECONDS);
   ```
2. **Add a gRPC-module rate-limiting interceptor** (analogous to `ThrottleManagerImpl` in `web3`) using bucket4j or a `ServerInterceptor` that enforces a global or per-IP request-per-second cap.
3. **Disable or restrict the reflection service** in production via `spring.grpc.server.reflection.enabled=false`, or gate it behind an IP allowlist at the ingress level unconditionally (not just for GCP deployments).
4. **Expose `maxConcurrentCallsPerConnection` alongside new properties** (`maxConnectionAge`, `maxConnectionIdle`) in `NettyProperties` so operators can tune them without code changes.

### Proof of Concept
```bash
# Open 100 parallel connections, each issuing reflection list_services in a loop
for i in $(seq 1 100); do
  (
    while true; do
      grpcurl -plaintext <host>:5600 grpc.reflection.v1.ServerReflection/ServerReflectionInfo \
        <<< '{"list_services": ""}' 2>/dev/null
    done
  ) &
done

# Monitor server CPU (expect sustained increase vs. 24h baseline)
# Each connection dispatches up to 5 concurrent calls to applicationTaskExecutor
# 100 connections × 5 concurrent = 500 simultaneous reflection dispatches
# Sequential call rate per connection is unbounded
```
Baseline CPU is measured from Prometheus metric `process_cpu_usage{application="grpc"}` over the preceding 24 hours. After running the above for several minutes, the same metric will show a sustained increase exceeding 30% of the prior baseline on a standard single-replica deployment.
