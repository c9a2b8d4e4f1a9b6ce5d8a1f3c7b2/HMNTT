### Title
Unauthenticated Amplified DB Memory Exhaustion via Correlated `json_agg` Subquery in `/api/v1/balances` with Timestamp Filter

### Summary
An unprivileged external user can send concurrent requests to `/api/v1/balances?timestamp=lte:<ts>&limit=100`, each of which causes PostgreSQL to execute a correlated `json_agg` subquery (`getTokenBalanceSubQuery`) for every one of up to 100 returned accounts, scanning up to 50 `token_balance` rows per account (5,000 rows per request). The REST API has no per-IP rate limiting or in-flight request cap, so an attacker can sustain enough concurrent requests to exhaust the database connection pool and working memory, degrading service for 30%+ of nodes.

### Finding Description

**Code path:**

`getBalances()` (line 83) parses a `timestamp` query parameter. When present, it calls `getTsQuery()` → `getAccountBalanceTimestampRange()` to resolve a valid snapshot range, then calls `getBalancesQuery()`: [1](#0-0) 

`getBalancesQuery()` (line 199) embeds `getTokenBalanceSubQuery()` as a correlated scalar subquery in the `SELECT` list: [2](#0-1) 

`getTokenBalanceSubQuery()` (line 294) generates a correlated subquery that, for each outer row (`ab.account_id`), performs a `DISTINCT ON (token_id)` scan of `token_balance` within the timestamp range, then aggregates with `json_agg`, hard-limited to `tokenBalanceLimit.multipleAccounts`: [3](#0-2) 

The default value of `tokenBalanceLimit.multipleAccounts` is **50** and `response.limit.max` is **100**: [4](#0-3) [5](#0-4) 

**Root cause:** The outer query returns up to 100 accounts; for each, PostgreSQL executes the correlated subquery scanning up to 50 `token_balance` rows with `json_agg`. This is 5,000 row-level operations per request, all within a single query. The `token_balance` table is a historical snapshot table and can be very large. The `DISTINCT ON (token_id) … ORDER BY token_id, consensus_timestamp DESC` within a monthly timestamp range forces a sort/dedup pass per account.

**Why existing checks fail:**

1. `statement_timeout` exists at the pool level (`rest/dbpool.js` line 15) but only kills individual long-running queries after the fact — it does not prevent many concurrent queries from simultaneously consuming DB working memory and connection slots. [6](#0-5) 

2. The REST API Helm chart middleware contains **only** a `circuitBreaker` (triggers on error rate) and `retry` — **no `rateLimit` and no `inFlightReq` cap**, unlike the Rosetta chart which has both: [7](#0-6) 

Compare with Rosetta which has `inFlightReq: amount: 5` and `rateLimit: average: 10`: [8](#0-7) 

3. No application-level rate limiting exists in the Node.js REST service for the `/balances` endpoint.

### Impact Explanation
Each request with `timestamp` filter and `limit=100` causes PostgreSQL to perform up to 5,000 `token_balance` row reads with per-account `json_agg` aggregation. With the default `db.pool.maxConnections` (typically 10–20 for Node.js pools), an attacker sending 20–50 concurrent requests can saturate all DB connections and force PostgreSQL to allocate large amounts of working memory for sort/aggregate operations simultaneously. This degrades query latency for all other API consumers and can cause connection timeouts, effectively taking down 30%+ of mirror node REST processing capacity without any brute-force volume requirement.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge beyond the public API spec, and a single valid historical timestamp (easily obtained from any prior `/api/v1/balances` response). The request is indistinguishable from legitimate use. Any attacker with a script sending ~20–50 concurrent HTTP requests can trigger the condition. The absence of rate limiting or in-flight caps on the REST API makes this trivially repeatable and sustainable.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` Traefik middleware** to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta chart's configuration.
2. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) for the `/balances` endpoint.
3. **Consider rewriting the correlated subquery** as a single `LEFT JOIN LATERAL` or a CTE to reduce per-account query overhead.
4. **Set a conservative `work_mem` limit** for the REST DB role to cap per-sort memory allocation.

### Proof of Concept
```bash
# 1. Obtain a valid historical timestamp from any prior response
TS=$(curl -s "https://<mirror-node>/api/v1/balances?limit=1" | jq -r '.timestamp')

# 2. Send 50 concurrent requests with timestamp filter and max limit
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/balances?timestamp=lte:${TS}&limit=100" &
done
wait

# Expected result: DB connection pool exhaustion, increased latency or 500 errors
# for concurrent legitimate users; PostgreSQL shows high work_mem usage in pg_stat_activity
```

```javascript
const getBalancesQuery = async (accountQuery, balanceQuery, accountIdsQuery, limitQuery, order, tsQueryResult) => {
  const tokenBalanceSubQuery = getTokenBalanceSubQuery(order, tsQueryResult.query);
  const whereClause = `
      where ${[tsQueryResult.query, accountQuery, accountIdsQuery, balanceQuery].filter(Boolean).join(' and ')}`;
  const {lower, upper} = tsQueryResult.timestampRange;
  // The first upper is for the consensus_timestamp in the select fields, also double the lower and the upper since
  // they are used twice, in the token balance subquery and in the where clause of the main query
  const tsParams = [upper, lower, upper, lower, upper];
  const sqlQuery = `
      select distinct on (account_id)
        ab.account_id,
        ab.balance,
        ?::bigint as consensus_timestamp,
        (${tokenBalanceSubQuery}) as token_balances
      from account_balance ab
      ${whereClause}
      order by ab.account_id ${order}, ab.consensus_timestamp desc
      ${limitQuery}`;
  return [sqlQuery, tsParams];
```

**File:** rest/balances.js (L294-306)
```javascript
const getTokenBalanceSubQuery = (order, consensusTsQuery) => {
  consensusTsQuery = consensusTsQuery.replaceAll('ab.', 'tb.');
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select distinct on (token_id) token_id, balance
      from token_balance tb
      where tb.account_id = ab.account_id
        and ${consensusTsQuery}
      order by token_id ${order}, consensus_timestamp desc
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
```
