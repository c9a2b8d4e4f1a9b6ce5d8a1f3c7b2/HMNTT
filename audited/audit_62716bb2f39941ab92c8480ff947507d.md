### Title
Unauthenticated DoS via Wide Spender Range Filter Flooding on `/accounts/:id/nfts` Endpoint

### Summary
The `getNftsByAccountId()` handler in `rest/controllers/accountController.js` accepts `spender.id=gte:X&spender.id=lte:Y` range filters from any unauthenticated caller with no per-IP rate limiting in the REST Node.js server. A wide range (e.g., `gte:0&lte:9999999999`) forces the database to scan all NFT rows for the target `account_id` filtered by spender, ordered by `token_id, serial_number` — columns that are not aligned with the spender filter — before returning the capped 100 rows. Flooding this endpoint exhausts the DB connection pool and I/O bandwidth.

### Finding Description

**Route registration** — no authentication required:
`rest/routes/accountRoute.js` line 15:
```js
router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
```

**Handler** (`rest/controllers/accountController.js` lines 90–103) calls `utils.buildAndValidateFilters`, then `this.extractNftMultiUnionQuery`, then `NftService.getNfts(query)` — all synchronous validation, no rate gate.

**Validation** (`accountController.js` lines 18–32) only rejects: more than one `lte/lt` operator, more than one `gte/gt` operator, and `ne`. A single `gte:0` + `lte:9999999999` pair passes cleanly.

**SQL generated** (`rest/service/nftService.js` lines 55–76, 84–126):
```sql
SELECT account_id, nft.created_timestamp, ...
FROM nft
LEFT JOIN entity e ON e.id = nft.token_id
WHERE account_id = $1
  AND spender >= $3
  AND spender <= $4
ORDER BY token_id DESC, serial_number DESC
LIMIT $2   -- capped at 100
```
The ORDER BY is on `(token_id, serial_number)` while the filter is on `spender`. Without a composite index on `(account_id, spender)`, PostgreSQL must scan every NFT row for the given `account_id`, apply the spender predicate, then sort and truncate. For high-NFT accounts this is a full-account sequential scan per request.

**No rate limiting** — `rest/server.js` (lines 67–144) registers: cors, compression, httpContext, requestLogger, authHandler, metricsHandler, responseCacheCheckHandler. There is no `express-rate-limit` or equivalent per-IP throttle. The throttle found in `web3/src/main/java/…/ThrottleConfiguration.java` applies only to the separate web3 Java service.

**Limit cap** (`rest/utils.js` lines 544–553) caps `limit` at `responseLimit.max` (default **100**), which bounds result size but does not bound query scan cost.

**Statement timeout** exists (`rest/dbpool.js` line 15: `statement_timeout: config.db.pool.statementTimeout`) and provides partial mitigation, but concurrent in-flight queries still hold DB connections from the pool (`max: config.db.pool.maxConnections`) and consume I/O until the timeout fires.

### Impact Explanation
An attacker flooding this endpoint with wide-range spender queries forces repeated expensive sequential scans on the `nft` table. This exhausts the PostgreSQL connection pool, saturates disk I/O, and degrades or denies service to all REST API consumers. The REST API is the primary read interface for mirror node data; its unavailability prevents clients from querying account balances, transactions, and NFT state. The Hedera consensus network itself is unaffected, but the mirror node REST layer becomes unavailable.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The endpoint is publicly documented in `rest/api/v1/openapi.yml` lines 104–166. The attack requires only a standard HTTP client and a valid (or even invalid — the account lookup happens first but any existing account works) account ID. The attack is trivially scriptable, repeatable, and can be launched from a single host or distributed across IPs since there is no per-IP throttle.

### Recommendation
1. **Add per-IP rate limiting** to the REST Node.js server (e.g., `express-rate-limit`) applied globally before route handlers.
2. **Add a composite index** on `nft(account_id, spender)` or `nft(account_id, spender, token_id, serial_number)` so the spender range filter is index-supported.
3. **Enforce a concurrency limit** on DB pool usage per request source.
4. **Restrict spender range width**: reject or warn when `lte - gte` exceeds a configurable threshold in `validateFilters` (`accountController.js` lines 18–32).

### Proof of Concept
```bash
# Flood with wide spender range at max limit — no credentials needed
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/nfts?spender.id=gte:0&spender.id=lte:9999999999&limit=100" &
done
wait
# Observe: DB connection pool exhausted, subsequent REST API requests time out or return 503
