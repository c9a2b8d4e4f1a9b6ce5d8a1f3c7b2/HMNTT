### Title
Connection Pool Exhaustion DoS via Unauthenticated `GET /api/v1/network/registered-nodes` Flooding

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module requires no authentication and has no application-level rate limiting. All endpoints in the `rest-java` service share a single HikariCP connection pool. An unprivileged attacker can flood this endpoint with concurrent requests, exhausting the shared pool and starving other endpoints (`/api/v1/network/nodes`, `/api/v1/network/stake`, etc.) of database connections, causing them to return errors.

### Finding Description

**Code path:**

`NetworkController.getRegisteredNodes()` (lines 173–187) calls `NetworkServiceImpl.getRegisteredNodes()` (lines 139–152), which calls `RegisteredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` (lines 21–22). Each invocation acquires a HikariCP connection from the shared pool for the duration of the query.

The shared pool is constructed in `CommonConfiguration.dataSource()` (lines 66–95) with no `maximumPoolSize` override in `rest-java/src/main/resources/` (the directory contains only `banner.txt`). HikariCP's default `maximumPoolSize` is **10**. All endpoints in the `rest-java` service — `/api/v1/network/registered-nodes`, `/api/v1/network/nodes`, `/api/v1/network/stake`, `/api/v1/network/supply`, `/api/v1/network/fees`, etc. — draw from this same pool.

**Root cause — failed assumption:**

The `web3` module has a `ThrottleConfiguration` / `ThrottleManagerImpl` with a bucket4j rate limiter (default 500 req/s globally). The `rest-java` module has **no equivalent**. The only filters registered in `RestJavaConfiguration` are `ShallowEtagHeaderFilter` (ETag caching), `MetricsFilter`, and `LoggingFilter` — none of which reject or throttle requests. There is no per-IP, per-endpoint, or global rate limiter in the `rest-java` application code.

The GCP gateway `maxRatePerEndpoint: 250` is an optional, deployment-specific configuration (`gateway.gcp.enabled: true` defaults to `true` in the chart but is not enforced at the application layer and is absent in non-GCP deployments). The Traefik middleware circuit breaker (`NetworkErrorRatio() > 0.10`) only activates after errors have already occurred.

**Exploit flow:**

1. Attacker sends ≥10 concurrent `GET /api/v1/network/registered-nodes` requests (no auth, no token, no precondition).
2. Each request acquires one HikariCP connection and holds it for the duration of the native SQL query against `registered_node`.
3. With the pool at capacity (10 connections), any subsequent request to any endpoint that needs a DB connection blocks in HikariCP's wait queue.
4. HikariCP's default `connectionTimeout` is 30 seconds. Requests queued beyond that throw `SQLTimeoutException`, surfacing as 500 errors to callers of `/api/v1/network/nodes`, `/api/v1/network/stake`, etc.
5. Attacker sustains the flood; legitimate traffic to all `rest-java` endpoints is denied.

### Impact Explanation

All `rest-java` endpoints share the same HikariCP pool. Pool exhaustion causes a cascading failure: endpoints that are unrelated to `registered-nodes` begin returning 500-class errors indistinguishable from a database outage. The Prometheus alert `RestJavaHighDBConnections` fires only after 75% utilization sustained for 5 minutes — well after the DoS is effective. HPA is disabled by default (`hpa.enabled: false`), so horizontal scaling does not automatically mitigate the attack.

### Likelihood Explanation

No privileges, credentials, API keys, or special knowledge are required. The endpoint is publicly documented in the OpenAPI spec (`/api/v1/network/registered-nodes`). A single attacker with a modest number of concurrent HTTP connections (≥10) can trigger the condition. The attack is trivially repeatable and scriptable with any HTTP load tool (`curl`, `wrk`, `ab`). The absence of application-level rate limiting in `rest-java` (unlike `web3`) means there is no in-process defense.

### Recommendation

1. **Add application-level rate limiting to `rest-java`**: Introduce a bucket4j or Resilience4j `RateLimiter` filter analogous to `ThrottleConfiguration` in `web3`, applied globally or per-endpoint in `RestJavaConfiguration`.
2. **Increase and tune the HikariCP pool size**: Set an explicit `spring.datasource.hikari.maximum-pool-size` appropriate for the expected concurrency, and configure a short `connectionTimeout` (e.g., 5 seconds) so pool exhaustion fails fast rather than queuing for 30 seconds.
3. **Add a query-level statement timeout**: Configure `spring.datasource.hikari.connection-init-sql` or a Postgres `statement_timeout` for the `mirror_node` role to bound how long any single query can hold a connection.
4. **Enforce rate limiting at the infrastructure layer unconditionally**: Make the GCP/Traefik rate limit mandatory rather than opt-in, and add per-IP limits.

### Proof of Concept

```bash
# Exhaust the 10-connection HikariCP pool with concurrent requests
# (no authentication required)
for i in $(seq 1 20); do
  curl -s "https://<mirror-node-host>/api/v1/network/registered-nodes" &
done
wait

# Simultaneously, observe that /api/v1/network/stake returns 500
curl -v "https://<mirror-node-host>/api/v1/network/stake"
# Expected: HTTP 500 or connection timeout due to pool exhaustion
```

Sustained with a tool like `wrk -t10 -c50 -d60s https://<host>/api/v1/network/registered-nodes`, all `rest-java` endpoints will begin returning errors within seconds.
