### Title
GraphQL Alias-Based Query Amplification Allows Unauthenticated DB Connection Pool Exhaustion

### Summary
The GraphQL `account` query uses a default flat complexity calculator that assigns equal weight (1) to both trivial scalar fields and DB-backed resolver fields. An unauthenticated attacker can craft a single request with up to ~62 aliased `account` fields (bounded by the 1000-token parser limit, well under the 200-complexity ceiling), each triggering a separate `EntityService` database query, amplifying DB load ~62× per HTTP request with no application-level rate limiting on the GraphQL endpoint.

### Finding Description

**Exact code path:**

`GraphQlConfiguration.java` configures complexity instrumentation with no custom field cost weighting: [1](#0-0) 

`new MaxQueryComplexityInstrumentation(200)` uses graphql-java's default `FieldComplexityCalculator`, which assigns a flat cost of **1 per field** regardless of whether the field is a trivial property or a DB-backed resolver. No custom calculator is registered anywhere in the codebase (confirmed: zero matches for `FieldComplexityCalculator`).

`AccountController.account()` is the resolver for the `account` query field: [2](#0-1) 

Every invocation calls `validateOneOf()` and then exactly one `EntityService` method (`getByIdAndType`, `getByAliasAndType`, or `getByEvmAddressAndType`), each of which issues a native SQL query via `EntityRepository`: [3](#0-2) [4](#0-3) 

**Root cause:** The complexity budget of 200 is consumed at 2 units per aliased `account { id }` pair (1 for `account` + 1 for `id`), allowing up to 100 aliases before hitting the complexity ceiling. The binding constraint is actually the GraphQL token limit of 1000: [5](#0-4) 

Each alias like `a1: account(input: {alias: "ABCDEFGHIJKLMNOPQ"}) { id }` consumes approximately 16 GraphQL tokens. With `maxTokens(1000)`, an attacker can pack ~62 aliased `account` fields into one request while staying under both the token limit and the complexity limit (62 × 2 = 124 < 200).

**Failed assumption:** The complexity limit was intended to prevent resource exhaustion, but the flat default calculator does not model the actual cost difference between a DB-backed root resolver (`account`) and a trivial scalar field (`id`). The schema only exposes one query type: [6](#0-5) 

**No application-level rate limiting exists for the GraphQL module.** The throttle infrastructure (`ThrottleManagerImpl`, `RequestFilter`) lives exclusively in the `web3` module. The `inFlightReq: amount: 5` middleware is an optional Helm/Traefik infrastructure concern, not enforced in application code: [7](#0-6) 

### Impact Explanation
Each crafted request with 62 aliased `account` fields issues 62 synchronous database queries against the `entity` table. An attacker sending concurrent such requests (e.g., 10 concurrent connections × 62 DB queries = 620 simultaneous DB queries) can exhaust the JDBC connection pool, causing legitimate requests to queue or fail. The `statementTimeout: 10000` (10 s) means each query holds a connection for up to 10 seconds, maximizing pool starvation duration. This is a non-network DoS affecting the availability of the mirror node GraphQL service without any credentials.

### Likelihood Explanation
The endpoint `/graphql/alpha` requires no authentication. The exploit requires only a standard HTTP POST with a crafted JSON body — no special tooling, no privileged access, no prior knowledge beyond the public GraphQL schema (discoverable via introspection). The attack is trivially repeatable and scriptable. The GraphQL alias technique is well-documented and commonly used in GraphQL DoS research.

### Recommendation
1. **Register a custom `FieldComplexityCalculator`** that assigns a significantly higher cost (e.g., 50–100) to the `account` root resolver field, reducing the maximum aliases per request to 2–4 within the existing complexity budget of 200.
2. **Add application-level rate limiting** to the GraphQL module (e.g., a Spring `HandlerInterceptor` or `WebGraphQlInterceptor` that tracks requests per IP/time window), independent of infrastructure middleware.
3. **Disable GraphQL introspection** in production to prevent schema discovery.

### Proof of Concept

```bash
# Craft a query with 62 aliased account fields (under 1000 tokens, complexity ~124)
python3 -c "
aliases = '\n'.join(
  f'  a{i}: account(input: {{alias: \"ABCDEFGHIJKLMNOPQABCDEFGHIJKLMNOPQ\"}}) {{ id }}'
  for i in range(62)
)
query = '{' + aliases + '}'
import json
print(json.dumps({'query': query}))
" > payload.json

# Send to unauthenticated GraphQL endpoint
for i in $(seq 1 20); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d @payload.json &
done
wait
# Result: 20 concurrent requests × 62 DB queries = 1240 simultaneous entity table queries,
# exhausting the DB connection pool and degrading/denying service to legitimate users.
```

    public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
        return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
        return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
