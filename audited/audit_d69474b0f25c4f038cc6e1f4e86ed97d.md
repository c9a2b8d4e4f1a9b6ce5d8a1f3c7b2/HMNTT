### Title
Unbounded Exponential Backoff in `buildNetworkFromMirrorNodes` Allows Startup DoS via Malicious Mirror REST Endpoint

### Summary
When `network=other`, pinger startup calls `buildNetworkFromMirrorNodes()` which retries `fetchMirrorNodeNetwork()` with exponential backoff. Any HTTP 5xx response (including 503) sets `retry=true`, and with default configuration (10 retries, 500ms base backoff), the total accumulated backoff reaches ~511 seconds (~8.5 minutes) plus up to 110 seconds of per-request HTTP timeouts, blocking pinger startup entirely. An attacker who can control or intercept the configured mirror REST endpoint can exploit this to prevent the pinger from ever becoming ready.

### Finding Description

**Code location:** `pinger/mirror_node_client.go`

In `fetchMirrorNodeNetwork()` (lines 90–92):
```go
if resp.StatusCode < 200 || resp.StatusCode >= 300 {
    retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
    return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
}
```
HTTP 503 causes `retry=true` to be returned.

In `buildNetworkFromMirrorNodes()` (lines 48, 52–68):
```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)  // default: 11
for attempt := 1; attempt <= attempts; attempt++ {
    network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
    if err == nil { return network, nil }
    lastErr = ...
    if !retry || attempt == attempts { break }
    backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
    select {
    case <-ctx.Done(): return nil, ctx.Err()
    case <-time.After(backoff):
    }
}
```

With defaults (`mirrorNodeClientMaxRetries=10`, `mirrorNodeClientBaseBackoff=500ms`), the 10 inter-attempt backoffs are:
`500ms + 1s + 2s + 4s + 8s + 16s + 32s + 64s + 128s + 256s = 511.5s`

Plus 11 × 10s HTTP timeouts (`mirrorNodeClientTimeout=10s`) = 110s.

**Total blocking time: ~621 seconds (~10.4 minutes).**

`newClient()` in `sdk_client.go` (line 18) calls `buildNetworkFromMirrorNodes()` synchronously, and `main()` in `main.go` (line 41) calls `newClient()` before writing `/tmp/ready` (line 47). The readiness probe file is never written during this entire window.

**Root cause:** No cap on total retry duration; the exponential backoff formula `baseBackoff * 2^(attempt-1)` grows to 256 seconds on the 10th retry with no maximum total-wait guard.

**Why existing checks fail:** The `ctx.Done()` branch (line 65) only fires on SIGTERM/SIGINT — it provides no automatic timeout. There is no `context.WithTimeout` wrapping the entire retry loop.

### Impact Explanation

- Pinger startup is blocked for up to ~10 minutes per restart cycle.
- `/tmp/ready` is never written, so Kubernetes readiness probes fail indefinitely, preventing the pod from receiving traffic or being considered healthy.
- No HBAR transfers are submitted during this window, defeating the monitoring purpose of the pinger.
- If the pod is restarted by a liveness probe (which touches `/tmp/alive` only after `newClient` returns — the goroutine starts before `newClient` but the liveness file is only written every 15s after startup), the attack can be repeated on every restart, causing a permanent DoS loop.

Severity: **Medium** — griefing/availability impact on the pinger service with no direct economic damage to the broader network.

### Likelihood Explanation

**Precondition:** Attacker must be able to respond to HTTP requests at the configured `mirrorRest` URL. Realistic vectors:
1. The configured URL uses plain HTTP (e.g., `http://mirror-rest:5551` as shown in config defaults) — susceptible to ARP spoofing or DNS poisoning within the same network segment or Kubernetes namespace.
2. A compromised pod in the same Kubernetes namespace can intercept service traffic.
3. The mirror node operator themselves (in a third-party deployment) can return 503s.
4. BGP hijacking for internet-facing endpoints.

No cryptographic credentials or special privileges on the pinger host are required — only the ability to respond at the mirror REST URL.

### Recommendation

1. **Cap total retry duration** by wrapping the retry loop with a `context.WithTimeout`:
   ```go
   retryCtx, cancel := context.WithTimeout(ctx, cfg.mirrorNodeClientMaxTotalWait)
   defer cancel()
   // use retryCtx in the select and fetchMirrorNodeNetwork calls
   ```
2. **Add a configurable `mirrorNodeClientMaxTotalWait`** (e.g., default 30s) so operators can tune it.
3. **Cap the maximum single backoff** (e.g., `min(backoff, 30s)`) to prevent the exponential growth from dominating.
4. **Use HTTPS** for the mirror REST endpoint to prevent MITM-based injection of 503 responses.

### Proof of Concept

1. Deploy pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host:5551`.
2. On `attacker-controlled-host`, serve HTTP 503 for all requests to `/api/v1/network/nodes`.
3. Observe pinger logs: 11 attempts with backoffs of 500ms, 1s, 2s, 4s, 8s, 16s, 32s, 64s, 128s, 256s between them.
4. Confirm `/tmp/ready` is never created for ~10 minutes; Kubernetes readiness probe fails throughout.
5. With default settings, `log.Fatalf("client error: ...")` fires after ~621 seconds, and if the pod restarts, the cycle repeats indefinitely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
