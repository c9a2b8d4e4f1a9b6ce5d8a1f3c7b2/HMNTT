### Title
Connection Pool Exhaustion via Unauthenticated Concurrent Requests to `/topics/:topicId` Triggering Dedicated Client Acquisition

### Summary
Every call to `getTopicMessages()` with `limit != 1` (which is the default and any `limit=<maxAllowed>` value) unconditionally triggers the `zeroRandomPageCostQueryHint` path in `queryQuietly()`, which acquires a **dedicated PostgreSQL client** from the shared pool and holds it across three sequential async operations (`begin`, main query, `commit`). Because the REST service has no rate limiting, concurrency control, or per-IP throttling, an unprivileged attacker sending a moderate number of concurrent requests can exhaust the shared pool, causing all other API endpoints that depend on the same pool to queue or fail.

### Finding Description

**Exact code path:**

In `rest/topicmessage.js`, `getTopicMessages()` at line 142:
```js
const queryHint = limit !== 1 ? constants.zeroRandomPageCostQueryHint : undefined;
``` [1](#0-0) 

`limit` defaults to `defaultLimit` (line 174 of the same file) and is overridden by the user-supplied `limit` parameter. Since `limit != 1` is true for every default or `limit=<maxAllowed>` request, `queryHint` is always set to `constants.zeroRandomPageCostQueryHint` (`'set local random_page_cost = 0'`). [2](#0-1) 

This non-null `queryHint` is passed to `getMessages()` → `pool.queryQuietly()` in `rest/utils.js`. Inside `queryQuietly()`, when `preQueryHint` is truthy, the code takes the dedicated-client branch:

```js
client = await this.connect();           // acquires dedicated client from pool
client.on('error', clientErrorCallback);
await client.query(`begin; ${preQueryHint}`); // holds client
result = await client.query(query, params);   // holds client
await client.query('commit');                 // holds client
``` [3](#0-2) 

The client is only released in the `finally` block after all three sequential async operations complete: [4](#0-3) 

**Root cause:** The pool is shared across all REST API endpoints. A dedicated client is held for the full round-trip of three sequential DB calls per request. There is no concurrency limit, no per-IP rate limit, and no request queue depth cap in the REST service middleware.

**Confirmed absence of rate limiting:** A search across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, `throttle`, `concurrency`, and `semaphore` returns zero matches. The only throttling in the codebase is in the `web3` Java service (`ThrottleManagerImpl`), which is entirely separate. [5](#0-4) 

**Pool configuration:** The pool `maxConnections` is a configurable integer validated to be `> 0`. The node-postgres default is 10. The Helm chart shows `mirror_rest` is capped at `max_user_connections: 250` at the pgbouncer layer, but the Node.js application pool is a separate, smaller limit. [6](#0-5) 

### Impact Explanation

When the pool is exhausted, every subsequent call to `pool.queryQuietly()` — across **all** REST endpoints (accounts, transactions, tokens, contracts, etc.) — blocks waiting for a connection. Once the `connectionTimeout` is exceeded, those requests fail with a `DbError`. Since all endpoints share one pool, exhausting it with topic-message requests causes 100% of concurrent requests to queue or fail, far exceeding the 30% threshold. The mirror node's REST API becomes effectively unavailable for the duration of the attack.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no brute-force volume. An attacker needs only to send `P` concurrent HTTP requests (where `P` equals the pool `maxConnections`) to `/api/v1/topics/<any_valid_topicId>?limit=<maxAllowed>&order=desc`. Each request holds a pool connection for the full DB round-trip. With a default pool of 10 connections and slow queries (large result sets at `limit=<maxAllowed>`), even a single attacker with a basic HTTP client (e.g., `curl` in parallel or a simple script) can sustain the exhaustion indefinitely by re-issuing requests as soon as prior ones complete. The attack is repeatable, stateless, and requires no persistence.

### Recommendation

1. **Add a concurrency semaphore** in `queryQuietly()` for the dedicated-client path, capping the number of simultaneous `this.connect()` calls to a fraction of `maxConnections` (e.g., 50%).
2. **Add per-IP or global rate limiting middleware** to the REST Express application (e.g., `express-rate-limit`) specifically for endpoints that trigger the `zeroRandomPageCostQueryHint` path.
3. **Set a `connectionTimeout`** low enough to fail fast and return a `503` rather than queuing indefinitely, so legitimate requests are not silently delayed.
4. **Consider using `SET LOCAL` inside a single query** rather than a dedicated client + explicit transaction, or use `pool.query('SET random_page_cost = 0')` as a session-level hint without holding a dedicated client.

### Proof of Concept

**Preconditions:** Mirror node REST API accessible; at least one valid `topicId` with messages exists; pool `maxConnections` = N (default 10).

**Steps:**
```bash
# Replace <host>, <topicId>, and <maxLimit> with actual values
# Send N+5 concurrent requests (N = pool maxConnections)
for i in $(seq 1 15); do
  curl -s "http://<host>/api/v1/topics/<topicId>?limit=100&order=desc" &
done
wait

# Immediately after (or concurrently), send a request to a different endpoint:
curl -v "http://<host>/api/v1/accounts"
# Expected result: request hangs until connectionTimeout, then returns 500 DbError
# or queues for the full duration of the topic-message queries
```

**Result:** While the 15 topic-message requests hold pool connections, the accounts request (and all other concurrent requests) cannot acquire a connection and either queue indefinitely or fail with a `DbError` once `connectionTimeout` is exceeded, demonstrating full REST API unavailability from a moderate, unauthenticated concurrent load.

function parseDbPoolConfig() {

  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
