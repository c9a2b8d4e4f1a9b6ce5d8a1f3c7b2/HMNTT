### Title
Unauthenticated HTTP Mirror Node Response Enables gRPC Endpoint Injection via MITM

### Summary
`buildNetworkFromMirrorNodes` fetches the Hiero network topology over a plain HTTP connection with no TLS enforcement and no cryptographic integrity check on the response. Any network-level attacker who can intercept or spoof the HTTP response can inject arbitrary `service_endpoints`, causing `hiero.ClientForNetwork(netmap)` to direct all subsequent gRPC transactions to attacker-controlled infrastructure.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 79–124.

```
// line 79-84: plain HTTP, no TLS pinning, no HMAC, no signature
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
...
resp, err := httpClient.Do(req)

// line 95-98: response body decoded directly, no integrity check
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }

// line 102-124: attacker-supplied host:port pairs inserted verbatim
for _, n := range payload.Nodes {
    for _, ep := range n.ServiceEndpoints {
        host := strings.TrimSpace(ep.DomainName)
        ...
        addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
        network[addr] = nodeAccountId   // ← attacker controls addr
    }
}
```

`newClient()` in `sdk_client.go` (lines 18–22) then passes this map directly to `hiero.ClientForNetwork(netmap)` with no further validation.

**Root cause / failed assumption:** The code assumes the HTTP response from `cfg.mirrorRest` is authentic. The default example URL (`http://mirror-rest:5551`, `config.go` line 37) uses plaintext HTTP. There is no TLS requirement, no certificate pinning, no HMAC, no signature over the node list, and no allowlist of permitted gRPC addresses. The only validation performed is that `NodeAccountID` parses as a valid account ID and that `host`/`port` are non-empty — both trivially satisfied by attacker-supplied values.

**Exploit flow:**
1. Attacker positions themselves to intercept HTTP traffic between the pinger pod and the mirror REST endpoint (DNS poisoning, ARP spoofing on the same subnet, BGP hijack, or compromised in-cluster DNS in a Kubernetes deployment without strict NetworkPolicy).
2. Pinger starts with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
3. `buildNetworkFromMirrorNodes` issues `GET http://mirror-rest:5551/api/v1/network/nodes`.
4. Attacker intercepts and returns a crafted JSON body:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"attacker.example.com","port":50211}]}],"links":{}}
   ```
5. `fetchMirrorNodeNetwork` decodes this without complaint and returns `{"attacker.example.com:50211": AccountID{0,0,3}}`.
6. `hiero.ClientForNetwork(netmap)` configures the SDK to send all gRPC calls to `attacker.example.com:50211`.
7. All subsequent `CryptoTransfer` transactions (signed with the operator key) are submitted to the attacker's gRPC server.

**Existing checks reviewed and shown insufficient:**
- HTTP status code check (lines 90–93): only rejects non-2xx; does not authenticate the responder.
- `NodeAccountID` parse check (lines 103–109): validates format, not origin.
- `host == ""` / `ep.Port == 0` guard (lines 118–119): trivially bypassed by supplying any non-empty host and non-zero port.
- Retry/backoff logic: irrelevant to integrity.

### Impact Explanation
The pinger's operator private key signs every `CryptoTransfer` transaction and submits it to whatever gRPC endpoints are in the network map. With attacker-controlled endpoints:
- **Transaction interception:** The attacker receives fully-formed, signed transactions including amounts, account IDs, and memo fields.
- **Denial of service:** The attacker can silently drop all transactions, breaking the liveness/health-check purpose of the pinger.
- **Replay attacks:** Captured signed transactions can be replayed against the real network within their validity window.
- **Operator key exposure risk:** If the attacker's gRPC server responds with crafted SDK-level errors that trigger key-material logging paths, key leakage is possible depending on SDK internals.

Severity: **High** — complete redirection of all network traffic for the pinger service, with signed transaction capture.

### Likelihood Explanation
- The default and documented `mirrorRest` value uses `http://` (plaintext), making TLS interception unnecessary.
- In Kubernetes (the primary deployment target, per `pinger/Dockerfile`), in-cluster DNS is a well-known MITM surface; without `NetworkPolicy`, any pod in the namespace can spoof DNS responses.
- The attack is repeatable: `newClient` is called once at startup; a single poisoned response permanently misconfigures the client for the lifetime of the process.
- No special application-level credentials are required — only network-layer access.

### Recommendation
1. **Enforce HTTPS:** Validate that `cfg.mirrorRest` uses the `https` scheme and reject `http` URLs at config load time (`config.go`, `loadConfig()`).
2. **TLS certificate verification:** Do not disable `InsecureSkipVerify`; ensure the default `http.Client` TLS config is used (it already is, but document this as a requirement).
3. **Response signing / allowlist:** If the mirror node supports response signing (e.g., via a known public key), verify the signature before consuming the node list. Alternatively, maintain a configurable allowlist of permitted gRPC address prefixes/CIDRs.
4. **Immutable bootstrap:** For `network=other`, consider accepting the node map as a static configuration value rather than fetching it dynamically over HTTP at startup.
