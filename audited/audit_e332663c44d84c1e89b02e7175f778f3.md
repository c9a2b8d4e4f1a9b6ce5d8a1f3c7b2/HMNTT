### Title
Missing `deleted is not true` Filter in `entity_history` Sub-query of `findActiveByEvmAddressOrAliasAndTimestamp` Causes Soft-Deleted Entities to Be Returned as Active in Historical Contract Calls

### Summary
The `entity_history` sub-query inside `findActiveByEvmAddressOrAliasAndTimestamp()` omits the `deleted is not true` predicate that is present in the sibling `entity` sub-query and in the equivalent ID-based method `findActiveByIdAndTimestamp()`. As a result, any unprivileged caller who submits a historical contract call using the alias or `evm_address` of a soft-deleted entity with a `blockTimestamp` at or after the deletion timestamp will receive the deleted entity record as if it were active, producing incorrect state in mirror node responses.

### Finding Description

**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 95–121, method `findActiveByEvmAddressOrAliasAndTimestamp(byte[] alias, long blockTimestamp)`.

**Root cause — the missing predicate:**

The `entity_history` branch of the UNION (lines 110–116) has no `deleted is not true` guard:

```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2          -- only timestamp filter
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

The `entity` branch immediately above it (lines 103–108) correctly carries `e.deleted is not true`. The ID-based peer method `findActiveByIdAndTimestamp()` (lines 144–151) applies `deleted is not true` to **both** branches. The alias/evm_address variant is the only historical lookup that omits it from `entity_history`.

**Failed assumption:** The query assumes that the `ORDER BY timestamp_range DESC LIMIT 1` final sort will always prefer a non-deleted record. It does not — when the entity is currently deleted, the `entity` branch returns zero rows (correctly filtered), leaving only the `entity_history` branch, whose most recent record at or after the deletion timestamp is the deletion record itself (`deleted = true`). That record wins the final `LIMIT 1` and is returned.

**Exploit flow:**

1. Entity A (alias `X`, `evm_address` `E`) is created at consensus timestamp `T1`. The `entity` table holds the live record.
2. Entity A is deleted at `T2 > T1`. The importer writes a row to `entity_history` with `deleted = true` and `timestamp_range = [T2, T3)`, and updates the `entity` table row to `deleted = true`.
3. Attacker (no privileges) issues a historical `eth_call` (or any RPC that resolves an account by alias/evm_address at a block) with `alias = X` and `blockTimestamp = T2` (or any value ≥ T2).
4. Query execution:
   - `entity_cte`: finds entity A from `entity` (no `deleted` filter in CTE) → returns entity ID.
   - `entity` branch: `deleted is not true` filters out the deleted row → **zero rows**.
   - `entity_history` branch: `lower(timestamp_range) = T2 ≤ T2` matches the deletion record (`deleted = true`) → **one row returned, no filter stops it**.
   - `UNION ALL` + `ORDER BY timestamp_range DESC LIMIT 1` → the deletion record is the only candidate → **returned**.
5. `CommonEntityAccessor.get(Bytes alias, Optional<Long> timestamp)` (line 48) passes the result straight through with no `deleted` check. Callers receive a non-empty `Optional<Entity>` containing a deleted entity and treat it as active.

**Why existing checks are insufficient:**

The existing test coverage for the deleted-history case (`findHistoricalEntityByEvmAddressOrAliasAndTimestampRangeAndDeletedTrueCallWithAlias`, lines 367–373, and the evm_address variant, lines 376–382) only asserts emptiness at `entityHistory.getTimestampLower() - 1` — i.e., **before** the deletion record's lower bound. There is no test for `getTimestampLower()` or `getTimestampLower() + 1`, which are the timestamps that actually trigger the bug. The tests pass today precisely because they avoid the vulnerable timestamp range.

### Impact Explanation

Any historical query resolved by alias or `evm_address` at a timestamp ≥ the entity's deletion time returns the deleted entity as active. Downstream consumers of the mirror node (indexers, wallets, dApps, other mirror nodes) receive incorrect historical state: a deleted contract or account appears to exist, potentially with stale balance, key, or type fields. This directly violates the "Incorrect or missing records exported to mirror nodes" integrity guarantee that is explicitly in scope. The same class of bug also exists in `findActiveByEvmAddressAndTimestamp` (lines 73–81), widening the affected surface.

### Likelihood Explanation

Preconditions are trivially met by any unprivileged external user:
- The alias and `evm_address` of deleted entities are permanently visible on-chain and in mirror node REST APIs.
- Historical `eth_call` (specifying a block number) is a standard, unauthenticated RPC endpoint.
- The attacker only needs to know one alias/evm_address of any previously deleted entity and supply a `blockTimestamp` ≥ its deletion time.

No keys, credentials, or internal access are required. The condition is deterministic and repeatable for every deleted entity in the network's history.

### Recommendation

Add `and eh.deleted is not true` to the `entity_history` sub-query in `findActiveByEvmAddressOrAliasAndTimestamp()`, mirroring the pattern already used in `findActiveByIdAndTimestamp()`:

```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    and eh.deleted is not true          -- add this line
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

Apply the same fix to `findActiveByEvmAddressAndTimestamp()` (lines 73–81) for consistency. Add integration tests asserting that `findActiveByEvmAddressOrAliasAndTimestamp` returns empty when queried at `entityHistory.getTimestampLower()` and `getTimestampLower() + 1` for a deleted history record, analogous to the existing `findHistoricalEntityByIdAndTimestampRangeAndDeletedTrueCall` test for the ID-based method.

### Proof of Concept

**Database state setup:**

```sql
-- Entity A created at T1=1000, alias=X, evm_address=E
INSERT INTO entity (id, alias, evm_address, created_timestamp, deleted, timestamp_range)
VALUES (42, X, E, 1000, false, '[1000,2000)');

-- Entity A deleted at T2=2000; deletion record written to entity_history
INSERT INTO entity_history (id, alias, evm_address, created_timestamp, deleted, timestamp_range)
VALUES (42, X, E, 1000, true, '[2000,3000)');

-- entity table updated to reflect deletion
UPDATE entity SET deleted = true, timestamp_range = '[3000,)' WHERE id = 42;
```

**Trigger (unprivileged RPC call):**

```
eth_call { to: <contract>, data: <calldata resolving alias X> } at blockNumber corresponding to blockTimestamp=2000
```

This routes through `CommonEntityAccessor.get(Bytes alias, Optional.of(2000L))` → `entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(X, 2000)`.

**Expected result:** `Optional.empty()` (entity is deleted).

**Actual result:** `Optional.of(Entity{id=42, deleted=true, ...})` — the deletion record from `entity_history` is returned because the `deleted is not true` predicate is absent from that branch.

**Equivalent JUnit reproduction (would fail against current code):**

```java
@Test
void findHistoricalEntityByAliasAtDeletionTimestampReturnsEmpty() {
    final var entityHistory = persistEntityHistoryWithDeleted(); // deleted=true
    // Query AT the deletion record's lower bound — currently returns the deleted record
    assertThat(entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(
                    entityHistory.getAlias(), entityHistory.getTimestampLower()))
            .isEmpty(); // FAILS: returns non-empty Optional with deleted=true entity
}
