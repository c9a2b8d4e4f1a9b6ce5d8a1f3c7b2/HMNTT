### Title
Unbounded DB Query Amplification via Uncached, Unthrottled `getAirdrops()` in rest-java

### Summary
The `getAirdrops()` path in the rest-java module executes a full database query on every request with no response caching and no rate limiting. Unlike the Node.js REST service (which has Redis-based response caching) and the web3 module (which has bucket4j rate limiting), the rest-java airdrop endpoints have neither protection. An unprivileged attacker can flood the endpoint with identical requests targeting a high-volume account, causing the database to re-execute the same expensive query on every call.

### Finding Description
**Code path:**

- `TokenAirdropsController` (`/api/v1/accounts/{id}/airdrops/outstanding` and `/pending`) calls `service.getAirdrops(request)` at line 108 with no caching layer above it.
- `TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) calls `repository.findAll(request, id)` directly on every invocation — no `@Cacheable`, no memoization.
- `TokenAirdropRepositoryCustomImpl.findAll()` (lines 58–72) executes a raw jOOQ `SELECT … WHERE … ORDER BY … LIMIT` against the `token_airdrop` table on every call.

**Root cause — failed assumptions:**

1. `RestJavaConfiguration` registers only a `ShallowEtagHeaderFilter` (lines 42–46). This filter computes an ETag *after* the full response is generated — the DB query is still executed on every request regardless of whether the client sends a matching `If-None-Match` header.
2. The rest-java module has zero rate-limiting for airdrop endpoints. The bucket4j throttle (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists exclusively in the `web3` module for contract-call endpoints. A grep across all `rest-java/src/main/java/**/*.java` for `RateLimit`, `Throttle`, `@Cacheable`, or `@Cache` returns no hits for the airdrop path.
3. The Node.js `rest` service has Redis-backed `responseCacheCheckHandler` / `responseCacheUpdateHandler` middleware, but no equivalent exists in rest-java.

**Exploit flow:**

An attacker picks a known high-volume account (e.g., an exchange or airdrop distributor with thousands of pending airdrops). They send a continuous stream of identical GET requests:
```
GET /api/v1/accounts/0.0.12345/airdrops/outstanding?limit=100&order=asc
```
Each request bypasses the ETag filter (or the attacker simply omits `If-None-Match`), hits `TokenAirdropServiceImpl.getAirdrops()`, and causes `TokenAirdropRepositoryCustomImpl.findAll()` to execute a full filtered, sorted, limited scan of the `token_airdrop` table. No deduplication, no short-circuit, no throttle.

### Impact Explanation
Every request results in a DB round-trip with a multi-condition WHERE clause, ORDER BY on three columns, and a LIMIT scan. For accounts with large airdrop sets, this is a non-trivial query. At sustained high request rates (easily achievable from a single machine), DB CPU and I/O increase proportionally with no server-side bound. This can degrade query latency for all other mirror-node consumers sharing the same database, meeting the ">30% DB CPU increase" threshold for a sufficiently active account and a sufficiently high request rate. No authentication or special privilege is required.

### Likelihood Explanation
The endpoint is public and unauthenticated. The attacker needs only to know (or enumerate) a high-volume account ID, which is publicly observable on-chain. The attack is trivially scriptable with `curl` or any HTTP load tool. It is repeatable indefinitely because there is no per-IP or per-account rate limit on this path.

### Recommendation
1. **Add rate limiting** to the rest-java airdrop endpoints using the same bucket4j pattern already present in the web3 module — either globally or per-IP/per-account.
2. **Add response caching** (e.g., Spring's `@Cacheable` with Caffeine or Redis) on `TokenAirdropServiceImpl.getAirdrops()` keyed on the full request parameters, with a short TTL (e.g., 5–10 seconds) to bound DB amplification while keeping data reasonably fresh.
3. **Replace `ShallowEtagHeaderFilter`** with a true server-side response cache (analogous to the Node.js `responseCacheCheckHandler`) so that cache hits short-circuit before the DB is queried.

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }
