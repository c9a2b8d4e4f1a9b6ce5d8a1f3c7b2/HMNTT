### Title
Hardcoded `sslmode=disable` in `GetDsn()` Prevents TLS Enforcement on Rosetta→PostgreSQL Connection

### Summary
`GetDsn()` in `rosetta/app/config/types.go` unconditionally appends `sslmode=disable` to the PostgreSQL DSN with no configuration field or override path. Unlike every other service in the same codebase (`restJava`, `web3`) which expose a configurable `sslMode` property, the Rosetta `Db` struct has no such field, making it impossible for operators to enforce TLS. Any attacker with network position between the Rosetta pod and the PostgreSQL backend can intercept all database traffic in plaintext.

### Finding Description
**Exact location:** `rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
"host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
```
The `Db` struct (lines 39–47) has no `SslMode` field. The DSN is consumed directly by `gorm.Open` in `rosetta/app/db/db.go` line 18, and also by the health check in `rosetta/app/middleware/health.go` line 42. There is no environment variable, YAML key, or code path that can override `sslmode=disable`.

**Contrast with other services:** `docs/configuration.md` lines 629 and 701 document `hiero.mirror.restJava.db.sslMode` and `hiero.mirror.web3.db.sslMode` as configurable properties accepting `DISABLE`, `ALLOW`, `PREFER`, `REQUIRE`, `VERIFY_CA`, or `VERIFY_FULL`. The Rosetta section (lines 652–675) lists no equivalent property — the omission is structural, not accidental.

**Why checks fail:** The Kubernetes NetworkPolicy in `charts/hedera-mirror/templates/networkpolicy.yaml` (lines 3–23) is gated on `if .Values.networkPolicy.enabled` and is therefore optional. Even when enabled, it restricts which pods can reach PostgreSQL but does not encrypt traffic. A compromised pod within the same namespace, a CNI-level ARP/routing attack, or a misconfigured cloud VPC still exposes plaintext wire traffic. The NetworkPolicy is a network-access control, not a substitute for transport encryption.

### Impact Explanation
An attacker with network position reads all SQL queries and responses in plaintext: this includes the full mirror node blockchain dataset (transactions, balances, account data) and the database credentials transmitted in the DSN itself (password field visible in process environment and logs). Because `sslmode=disable` is hardcoded, operators deploying in environments that require TLS (PCI, SOC 2, regulated cloud) have no application-level remediation path — they cannot fix this without patching the source.

### Likelihood Explanation
The precondition is network position between the Rosetta container and PostgreSQL, not OS-level privilege. In Kubernetes this is achievable via: (1) a compromised container in the same namespace with a CNI that does not enforce pod-to-pod encryption, (2) ARP spoofing on a flat overlay network, (3) a cloud provider misconfiguration exposing the pod CIDR. The all-in-one Docker image (`docs/rosetta/README.md` lines 76–81) runs both services in the same container on localhost, making MITM impossible in that specific topology — but the Kubernetes Helm chart deployment (`charts/hedera-mirror-rosetta/templates/deployment.yaml`) connects to an external DB host, making the attack surface real. The `stackgres` config (`charts/hedera-mirror/values.yaml` line 337) already notes `sslmode=disable` as a workaround for a known issue, indicating SSL is broadly not enforced in this stack.

### Recommendation
1. Add a `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go`.
2. Use it in `GetDsn()`: replace the hardcoded `sslmode=disable` with `sslmode=%s` populated from `db.SslMode`.
3. Set the default to `"disable"` for backward compatibility, but document the property `hiero.mirror.rosetta.db.sslMode` in `docs/configuration.md` alongside the equivalent `restJava` and `web3` entries.
4. For production deployments, recommend `sslmode=verify-full` with a CA certificate.

### Proof of Concept
1. Deploy the Rosetta Helm chart pointing to an external PostgreSQL host (non-localhost topology).
2. On a node in the same Kubernetes cluster, run a pod with `tcpdump` or `ettercap` targeting the pod IP of the Rosetta deployment and the PostgreSQL service IP.
3. Observe that all PostgreSQL wire protocol traffic (including the startup message containing the username, and subsequent query/response frames) is transmitted in plaintext — no TLS handshake occurs.
4. Attempt to set `HIERO_MIRROR_ROSETTA_DB_SSLMODE=require` via environment variable — confirm it has no effect because `GetDsn()` ignores all SSL configuration and always emits `sslmode=disable`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
