### Title
Unbounded Pagination Enumeration via `limit=1` in `getBalances()` Enables Sustained DB Query Amplification

### Summary
The `getBalances()` function in `rest/balances.js` accepts `limit=1` with no minimum enforcement and generates a `links.next` cursor after every single-row response. Because the REST API layer has no per-IP or per-endpoint rate limiting, an unauthenticated attacker can follow the cursor chain indefinitely, forcing one full DB query per account in the system and sustaining a high-frequency query stream that degrades the shared database connection pool for all other users.

**Root cause:** `getLimitParamValue` enforces only an upper bound; no lower bound exists. Combined with the absence of any rate-limiting middleware on the REST API, the pagination cursor mechanism becomes an amplification vector.

### Impact Explanation

On a network with *N* accounts, an attacker can force exactly *N* sequential DB queries by following the cursor chain with `limit=1`. Each query hits the `entity` (or `account_balance`) table with a correlated subquery for token balances. The PostgreSQL connection pool is shared across all API handlers; a sustained stream of queries from one attacker reduces available connections and increases query latency for every other concurrent user. Because the endpoint is public and unauthenticated, multiple attackers or multiple concurrent cursor-following loops multiply the effect linearly. The scope classification ("griefing with no economic damage") matches: no funds are at risk, but service quality degrades measurably.

### Likelihood Explanation

No special privileges, API keys, or on-chain assets are required. The attack is fully scriptable with a simple loop that reads `response.links.next` and issues the next HTTP GET. It is repeatable indefinitely and restartable from any account ID. The cursor values are deterministic and predictable (`account.id=gt:0.0.<n>`), so an attacker does not even need to follow the server-provided link â€” they can generate the sequence independently. Likelihood is high for any motivated griefing actor.

### Recommendation

1. **Enforce a minimum `limit` value** (e.g., `limit >= 1` is fine, but pair it with a per-IP request rate cap). In `getLimitParamValue`, add `ret = parsed < 1 ? 1 : ret` and document the floor.
2. **Add rate-limiting middleware to the REST API** (e.g., `express-rate-limit` or an equivalent) applied globally or specifically to high-fan-out endpoints like `/api/v1/balances`. A reasonable default is 60â€“120 requests/minute per IP.
3. **Consider a minimum effective limit** (e.g., floor at 10 or 25) so that the number of round-trips required to enumerate all accounts is bounded by `N / min_limit` rather than `N`.
4. **Apply the existing Traefik `inFlightReq` + `rateLimit` middleware** (already configured for Rosetta) to the REST API ingress as well.

### Proof of Concept

```bash
NEXT="/api/v1/balances?limit=1&order=asc&account.id=gt:0.0.1"
BASE="https://<mirror-node-host>"

while [ "$NEXT" != "null" ]; do
  RESPONSE=$(curl -s "${BASE}${NEXT}")
  echo "$RESPONSE" | jq '.balances[0].account'
  NEXT=$(echo "$RESPONSE" | jq -r '.links.next')
done
```

Each iteration of the loop triggers one DB query. On a network with 50 million accounts this loop issues 50 million queries with zero authentication and no server-side throttle to stop it. Running multiple parallel instances of this loop compounds the load linearly.
