### Title
Unsynchronized Concurrent `findStorageBatch()` Invocations Enable DB Query Amplification via Shared Slot-Key Cache

### Summary
`ContractStateServiceImpl.findStorageBatch()` contains no synchronization between adding a slot key to the shared per-contract Caffeine cache and reading all accumulated keys to issue a batch DB query. When N concurrent unauthenticated `eth_call` requests target the same contract with distinct slot keys, each thread observes the growing shared key set and independently issues a batch query containing all accumulated keys, producing up to N×M redundant DB key lookups (where M is the `slotsPerContract` cache size, default 1500) instead of N single-key lookups. For non-existent slot keys the `contractStateCache` is never populated, making the amplification perpetual rather than bounded by the 2-second TTL.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, `findStorageBatch()`, lines 85–122.

```
85:  private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
86:      final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
87:          contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
88:      final var wrappedKey = ByteBuffer.wrap(key);
90:      contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);          // (A) atomic insert
91:      final var cachedSlotKeys = contractSlotsCache.getNativeCache()
                                       .asMap().keySet();                  // (B) live snapshot
93:      final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
96:      for (var slot : cachedSlotKeys) { cachedSlots.add(...); }         // (C) iterate all keys
103:     final var contractSlotValues =
             contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots); // (D) DB query
```

**Root cause:** Steps A–D are not atomic. Between A and D, any other thread can execute its own A, enlarging the shared `cachedSlotKeys` set that both threads then read at B. There is no lock, `synchronized` block, or compare-and-swap guarding the read-then-query sequence.

**Exploit flow:**

1. Attacker sends 500 concurrent `eth_call` requests (within the 500 req/s global rate limit), each targeting the same `contractId` with a distinct, non-existent slot key.
2. All 500 threads enter `findStorageBatch()` simultaneously (cache miss in `contractStateCache` for all).
3. Each thread executes `putIfAbsent(keyN, EMPTY_VALUE)`. Because Caffeine's `asMap()` is a live `ConcurrentHashMap` view, by the time any thread reaches line 91, the map already contains keys inserted by other threads.
4. Each of the 500 threads reads a `cachedSlotKeys` set containing up to 500 entries and issues `findStorageBatch(contractId, [up to 500 keys])`.
5. After the `slotsPerContract` cache is pre-warmed to its maximum of 1500 keys (requires only 1500 sequential requests spread over 5 minutes, well within rate limits), each concurrent request issues a 1500-key batch query.
6. Because the slot keys do not exist in the DB, `contractStateCache` is never populated (line 109 only caches found values), so the miss path is taken on every subsequent request — the amplification is perpetual.

**Why existing checks fail:**

- `contractStateCache` (2s TTL, line 63–67): Only caches slots that exist in the DB. Non-existent keys are never stored, so the cache miss path is always taken.
- The `cachedValue != EMPTY_VALUE` guard (line 65) uses reference equality against the static sentinel; `EMPTY_VALUE` is never written to `contractStateCache`, so this guard never fires for non-existent slots.
- Rate limiter (`requestsPerSecond=500`, `ThrottleProperties` line 35): Operates at the HTTP request level and limits N, but does not prevent the N×M DB-query amplification within that N.
- Gas limit: Constrains EVM execution depth, not the number of DB rows scanned per batch query.

### Impact Explanation

With the default `slotsPerContract` cache size of 1500 (`CacheProperties` line 39) and the default rate limit of 500 req/s (`ThrottleProperties` line 35), an attacker can sustain 500 × 1500 = **750,000 DB key lookups per second** against the `contract_state` table, compared to the expected 500. This is a **1500× amplification** of DB read load, far exceeding the 30% threshold. Sustained load at this level can exhaust the DB connection pool, trigger the 3000ms statement timeout (`db.statementTimeout`), and degrade or deny service for all users of the mirror node web3 endpoint.

### Likelihood Explanation

The attack requires no authentication, no on-chain funds, and no special knowledge beyond a valid contract address. Any HTTP client capable of sending concurrent POST requests to `/api/v1/contracts/call` (the public `eth_call` endpoint) can execute it. The pre-warming phase (1500 sequential requests over 5 minutes) is invisible to rate-limit defenses. The amplification phase is repeatable indefinitely because non-existent slot keys never populate `contractStateCache`. The attack is fully scriptable and requires no privileged access.

### Recommendation

1. **Deduplicate in-flight batch queries per contract:** Use a `ConcurrentHashMap<EntityId, CompletableFuture<...>>` to ensure only one `findStorageBatch()` DB call is in flight per contract at a time; other threads wait on the same future (request coalescing / "thundering herd" prevention).
2. **Cache negative results:** After a batch query returns, store a sentinel (e.g., `EMPTY_VALUE`) in `contractStateCache` for every key in `cachedSlots` that was not found in the DB. This prevents the perpetual miss path for non-existent keys.
3. **Bound the batch size:** Cap `cachedSlots` to a configurable maximum (e.g., 100) before issuing the DB query, preventing a single request from scanning the entire accumulated key set.
4. **Synchronize the read-then-query window:** Wrap lines 91–103 in a per-contract lock (e.g., `contractId`-keyed `Striped<Lock>`) so that only one thread at a time reads the key set and issues the batch query for a given contract.

### Proof of Concept

```python
import asyncio, aiohttp, json

TARGET = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract_address>"

# Phase 1: pre-warm slotsPerContract cache with 1500 non-existent slot keys
async def prewarm(session):
    for i in range(1500):
        slot = hex(i).zfill(66)  # non-existent storage slot
        payload = {"block": "latest", "to": CONTRACT,
                   "data": "0x" + "00" * 32}  # any SLOAD-triggering calldata
        await session.post(TARGET, json=payload)
        await asyncio.sleep(0.2)  # 5 req/s, well within 500 req/s limit

# Phase 2: send 500 concurrent requests to trigger N×M amplification
async def amplify(session):
    tasks = []
    for i in range(500):
        slot = hex(i + 10000).zfill(66)
        payload = {"block": "latest", "to": CONTRACT, "data": "0x" + "00" * 32}
        tasks.append(session.post(TARGET, json=payload))
    await asyncio.gather(*tasks)

async def main():
    async with aiohttp.ClientSession() as session:
        print("Pre-warming slot key cache...")
        await prewarm(session)
        print("Launching amplification burst...")
        while True:
            await amplify(session)
            await asyncio.sleep(2)  # repeat every contractStateCache TTL window

asyncio.run(main())
```

Expected result: DB monitoring shows `contract_state` index scans spiking to ~750,000 key lookups/second (500 concurrent queries × 1500 keys each) while only 500 HTTP requests/second are issued, confirming the amplification. Query latency increases and connection pool saturation follows within seconds of sustained attack.
