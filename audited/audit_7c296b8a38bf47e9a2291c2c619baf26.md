### Title
Unbounded JVM Heap Exhaustion via EVM Memory Capture in Opcode Tracer with `memory=true`

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint, when called with `memory=true`, replays a historical EVM transaction and captures the full EVM memory frame at every single opcode step with no size or count bound. A transaction that used `EXTCODECOPY` to load large external contract bytecode (up to 24 KB per EIP-170) into EVM memory causes every subsequent opcode step to snapshot that entire memory region as hex strings, accumulating unbounded `ArrayList<String>` entries in the JVM heap. A single such request can exhaust JVM heap and crash the web3 service. No authentication is required.

### Finding Description

**Endpoint entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

The only guards are:
1. A `gzip` `Accept-Encoding` header check (trivially satisfied).
2. `throttleManager.throttleOpcodeRequest()` — a token-bucket rate limiter defaulting to **1 request/second globally** (not per-IP, not per-user). [2](#0-1) [3](#0-2) 

**Memory capture — root cause** — `AbstractOpcodeTracer.captureMemory()`: [4](#0-3) 

At every opcode step, `frame.memoryWordSize()` is read with no upper bound, and a new `ArrayList<String>` of `size` 66-character hex strings is allocated. There is no cap on `size`.

**Per-step invocation** — `OpcodeActionTracer.tracePostExecution()`: [5](#0-4) 

**Unbounded accumulation** — `OpcodeContext.addOpcodes()`: [6](#0-5) 

The `opcodes` `ArrayList` has no size limit. Every opcode step appends a new `Opcode` object containing the full memory snapshot.

**Exploit flow:**
1. Attacker identifies (or waits for) any on-chain transaction where a contract called `EXTCODECOPY` to copy a large external contract's bytecode (up to 24 KB) into EVM memory, followed by many subsequent opcodes.
2. Attacker sends: `GET /api/v1/contracts/results/<txHash>/opcodes?memory=true` with `Accept-Encoding: gzip`.
3. During EVM replay, after `EXTCODECOPY` expands EVM memory to ~24 KB (768 × 32-byte words), every subsequent opcode step calls `captureMemory()`, allocating ~50 KB of `String` objects per step.
4. For a transaction with 10,000 post-`EXTCODECOPY` opcodes (well within gas limits of complex contracts), the JVM accumulates ~500 MB of live heap objects before the response is serialized.
5. With multiple concurrent requests (the throttle is global, not per-IP), or with a single sufficiently complex transaction, the JVM throws `OutOfMemoryError`, crashing the web3 service process.

### Impact Explanation
A JVM `OutOfMemoryError` in the web3 service crashes or severely degrades the mirror node's web3 API, which serves EVM-compatible JSON-RPC and REST contract calls. This constitutes a denial-of-service against the mirror node's processing capacity. The mirror node is infrastructure used by dApps and network participants; its unavailability disrupts contract query and execution services. The severity matches the stated scope: shutdown of ≥30% of network processing nodes without brute force, since a single well-chosen transaction replay can OOM the JVM.

### Likelihood Explanation
- No authentication or account required — any anonymous HTTP client can call the endpoint.
- The `memory=true` parameter is user-controlled and explicitly documented.
- Transactions using `EXTCODECOPY` on large contracts exist on Hedera mainnet and testnet (e.g., proxy patterns, bytecode inspection contracts).
- The global throttle of 1 req/sec does not prevent a single request from exhausting heap; it only slows repeated attempts.
- The attack is repeatable: after a service restart, the same transaction hash can be replayed again.
- No special tooling is needed — a single `curl` command suffices.

### Recommendation
1. **Cap EVM memory capture size**: In `captureMemory()`, enforce a maximum number of words (e.g., 1024 words = 32 KB) and return a truncated or error result if exceeded.
2. **Cap total opcodes list size**: In `OpcodeContext.addOpcodes()`, enforce a maximum list size (e.g., 50,000 entries) and stop tracing or return an error when exceeded.
3. **Per-IP/per-user rate limiting**: The current global `opcodeRequestsPerSecond=1` bucket does not isolate attacker traffic from legitimate traffic. Add per-source-IP throttling.
4. **Request timeout**: Enforce a hard wall-clock timeout on the EVM replay execution to bound both CPU and memory usage per request.
5. **JVM heap guard**: Consider running the opcode replay in a bounded memory context or a separate worker with a memory limit.

### Proof of Concept
```bash
# Step 1: Identify a transaction hash on Hedera testnet/mainnet
# where a contract called EXTCODECOPY on a large contract (e.g., a proxy or factory).
# Example: any transaction calling EquivalenceContract.copyCode() on a large contract.

TX_HASH="0x<hash_of_extcodecopy_transaction>"

# Step 2: Send the opcode replay request with memory=true
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true&stack=false&storage=false"

# Result: The JVM allocates unbounded heap for memory snapshots at each opcode step.
# With a sufficiently complex transaction, the web3 service throws OutOfMemoryError
# and becomes unavailable (HTTP 503 or process crash).

# Step 3 (amplification): Send the same request repeatedly at 1 req/sec
# from multiple source IPs to bypass the global throttle bucket.
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true" &
  sleep 1
done
```

```java
    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
```
