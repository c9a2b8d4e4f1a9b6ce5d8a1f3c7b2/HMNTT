### Title
Unauthenticated Request Flood on GET /blocks Causes Node.js Event Loop Saturation and Database Pool Exhaustion (DoS)

### Summary
The `GET /blocks` route in the Node.js REST API has no per-IP rate limiting, no authentication requirement, and no request throttling. The `authHandler` middleware explicitly allows all unauthenticated requests to pass through. An unprivileged external attacker can flood this endpoint to saturate the single-threaded Node.js event loop and exhaust the PostgreSQL connection pool, denying service to all legitimate users.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` registers the route with no guards: [1](#0-0) 

`rest/server.js` assembles the middleware stack. The only candidate for access control is `authHandler`, applied globally at line 86, followed by the block routes at line 112: [2](#0-1) 

`rest/middleware/authHandler.js` is the critical failure point. When no `Authorization` header is present, `basicAuth(req)` returns `null`, and the handler immediately `return`s — the request is passed to the next middleware with zero restriction: [3](#0-2) 

Authentication is purely opt-in and only grants authenticated users a custom response `limit`. It is not a gate that blocks unauthenticated traffic.

A `grep_search` across all `rest/**/*.js` for `rateLimit`, `throttle`, `express-rate-limit`, and `helmet` returned a single match only in a test utility file — confirming **zero production rate-limiting middleware** exists in the REST API layer.

The response cache (`responseCacheCheckHandler`) is the only potential mitigant, but it is explicitly conditional: [4](#0-3) 

It requires both `config.cache.response.enabled` AND `config.redis.enabled` to be true. When Redis is unavailable or disabled (the common default), every request reaches the database. Even when enabled, cache misses (e.g., requests with unique query parameters like `?limit=1&block.number=gt:X`) bypass the cache entirely.

The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` Java module and has no effect on the Node.js REST API: [5](#0-4) 

The Traefik `rateLimit` middleware in the Helm chart applies only to the Rosetta service, not the REST API: [6](#0-5) 

### Impact Explanation
Node.js is single-threaded. A sustained flood of concurrent `GET /blocks` requests will:
1. Fill the event loop queue, causing all other routes (transactions, accounts, topics) to stop responding.
2. Exhaust the PostgreSQL connection pool (`db.pool.maxConnections`), causing DB queries to queue or fail with timeouts.
3. Effectively partition the mirror node from the network — no block data, transaction data, or health checks will be served to legitimate consumers. Severity: **High** (full REST API DoS achievable by any anonymous attacker).

### Likelihood Explanation
Preconditions: none. The attacker needs only network access to the mirror node's REST port. The attack is trivially scriptable (`ab`, `wrk`, `hey`, or a simple loop of `curl` calls). It is repeatable, requires no credentials, no special knowledge, and no prior access. Any public-facing mirror node deployment is exposed.

### Recommendation
1. **Add per-IP rate limiting** using `express-rate-limit` (or equivalent) as a global middleware in `rest/server.js`, applied before route registration, with a sensible default (e.g., 100 req/min per IP).
2. **Apply the limit at the unauthenticated tier**: the `authHandler` already has the concept of a per-user `limit`; extend this to enforce a stricter default limit for unauthenticated requests via the same mechanism.
3. **Enforce connection-level concurrency limits** (e.g., `express-slow-down` or `toobusy-js`) to shed load when the event loop lag exceeds a threshold.
4. **Ensure Traefik/ingress-level rate limiting** is configured for the REST API service (not just Rosetta) in the Helm chart.

### Proof of Concept
```bash
# No credentials, no special headers required.
# Run from any machine with network access to the mirror node REST port.

# Using 'hey' (https://github.com/rakyll/hey):
hey -n 100000 -c 500 http://<mirror-node-host>:<port>/api/v1/blocks

# Or with wrk:
wrk -t12 -c400 -d60s http://<mirror-node-host>:<port>/api/v1/blocks

# Expected result:
# - Legitimate requests to /api/v1/transactions, /api/v1/accounts, etc.
#   begin timing out or returning 503/ECONNRESET within seconds.
# - Node.js process CPU pegged at 100%.
# - PostgreSQL connection pool exhausted; DB logs show max_connections reached.
# - /health/liveness and /health/readiness stop responding.
```

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
