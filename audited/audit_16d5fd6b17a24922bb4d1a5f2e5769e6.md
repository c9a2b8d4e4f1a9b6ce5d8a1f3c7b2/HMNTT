### Title
CPU Exhaustion via Crafted `accept-encoding` Header in `negotiate()` on Every Cache Hit

### Summary
The `negotiate()` function in `rest/middleware/responseCacheHandler.js` unconditionally instantiates `new Negotiator(req)` and invokes `negotiator.encoding([GZIP_ENCODING])` on every cache hit, parsing the raw `accept-encoding` request header each time. Because the cache key is derived solely from the URL (not the `accept-encoding` header), an unauthenticated attacker can send concurrent requests to any cached endpoint with a crafted header containing hundreds of encoding tokens, forcing repeated O(n log n) parsing work on Node.js's single-threaded event loop with no rate limiting in place.

### Finding Description

**Exact code path:**

`negotiate()` is called on every cache hit in `responseCacheCheckHandler`: [1](#0-0) 

Inside `negotiate()`: [2](#0-1) 

`new Negotiator(req)` is constructed fresh on every invocation, and `negotiator.encoding([GZIP_ENCODING])` triggers full parsing and quality-sorting of the raw `accept-encoding` header value.

**Root cause — failed assumption:** The cache key is computed only from `req.originalUrl`: [3](#0-2) 

The code comment explicitly acknowledges that `accept-encoding` is intentionally excluded from the cache key. This means every request to a cached URL — regardless of its `accept-encoding` value — hits the cache and unconditionally executes `negotiate()`. There is no guard on the size or token count of the `accept-encoding` header before passing it to `Negotiator`.

**No rate limiting on the REST API:** The Helm chart for `hedera-mirror-rest` defines only `circuitBreaker` and `retry` middleware — no `inFlightReq` or `rateLimit`: [4](#0-3) 

Compare with Rosetta, which has both `inFlightReq` and `rateLimit`. The REST API has neither. There is also no application-level rate limiting in the Node.js middleware chain: [5](#0-4) 

**Exploit flow:**
1. Attacker identifies any popular endpoint whose response is cached in Redis (e.g., `/api/v1/accounts`).
2. Attacker sends a flood of concurrent GET requests to that URL with a crafted header: `accept-encoding: e1;q=0.999, e2;q=0.998, e3;q=0.997, ...` (hundreds of tokens, staying within Node.js's 16 KB header limit — approximately 1,000–1,100 tokens fit).
3. Each request: cache key matches → `responseCacheCheckHandler` finds a hit → `negotiate()` is called → `new Negotiator(req)` parses and quality-sorts ~1,000 tokens → O(n log n) CPU work per request.
4. Node.js's single-threaded event loop is saturated; legitimate requests queue up and time out.

**Why existing checks fail:**
- The `circuitBreaker` triggers only on `NetworkErrorRatio > 0.25` or `ResponseCodeRatio(5xx) > 0.25`. CPU exhaustion returns HTTP 200 responses (from cache), so neither condition fires.
- The `retry` middleware (10 attempts) amplifies the attack by re-issuing timed-out requests.
- Node.js's default `--max-http-header-size` (16 KB) limits token count but does not prevent the attack — ~1,000 tokens is sufficient.
- No application-level header validation exists in `requestHandler.js` or any other REST middleware.

### Impact Explanation
The Hedera Mirror Node REST API is a public-facing, unauthenticated service. CPU exhaustion on the Node.js process (single-threaded event loop) blocks all concurrent request processing, causing denial of service for all users of the REST API. Because the attack targets the cache fast-path (the path designed to avoid database load), it is effective even when the backend is healthy. The `retry: attempts: 10` Traefik middleware amplifies each attacker request tenfold.

### Likelihood Explanation
No authentication or special privileges are required. Any external user can send HTTP requests with arbitrary headers. The attack is trivially scriptable (e.g., with `curl` or `wrk` with a custom header). The targeted endpoint must be cached, which is the normal steady-state for popular endpoints. The attack is repeatable and sustainable as long as the cache remains warm.

### Recommendation
1. **Limit `accept-encoding` header complexity before negotiation:** In `negotiate()` or a preceding middleware, reject or truncate `accept-encoding` headers exceeding a reasonable token count (e.g., 10) or byte length (e.g., 100 bytes).
2. **Add rate limiting to the REST API:** Add `inFlightReq` and `rateLimit` Traefik middleware entries to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta configuration.
3. **Cache the negotiation result:** Since the only supported encoding is `gzip`, the negotiation result is binary. Cache whether the client accepts gzip (a simple `req.get('accept-encoding')?.includes('gzip')` check) instead of invoking the full `Negotiator` on every cache hit.
4. **Consider a simpler check:** Replace `new Negotiator(req)` with a direct string check: `req.get('accept-encoding')?.split(',').some(e => e.trim().split(';')[0].trim() === 'gzip')`, which is O(n) but avoids the full quality-sort and is bounded by the same header size limit.

### Proof of Concept

```bash
# Step 1: Warm up the cache
curl -s "https://<mirror-node-host>/api/v1/accounts?limit=1" > /dev/null

# Step 2: Generate a crafted accept-encoding header with ~900 tokens
HEADER=$(python3 -c "print(','.join(f'enc{i};q=0.{999-i:03d}' for i in range(900)))")

# Step 3: Flood with crafted requests (use parallel tool like wrk or ab)
# Using curl in a loop for demonstration:
for i in $(seq 1 500); do
  curl -s -o /dev/null \
    -H "accept-encoding: $HEADER" \
    "https://<mirror-node-host>/api/v1/accounts?limit=1" &
done
wait

# Expected result: Node.js CPU spikes to 100%; legitimate requests time out.
# The server continues returning HTTP 200 (from cache), so circuit breaker does NOT trigger.
```

const negotiate = (cachedResponse, req, res) => {
  res.setHeader(VARY_HEADER, 'accept-encoding');

  if (cachedResponse.compressed) {
    const negotiator = new Negotiator(req);
    if (negotiator.encoding([GZIP_ENCODING]) === GZIP_ENCODING) {
      res.setHeader(CONTENT_ENCODING_HEADER, GZIP_ENCODING);
      res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getLength());
      return true;
    }
  }

  res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getUncompressedLength());
  return false;
};
