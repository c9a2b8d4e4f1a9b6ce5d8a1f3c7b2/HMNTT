### Title
Unauthenticated EVM Address Lookup DoS via Uncached, Unthrottled Database Queries in `EntityServiceImpl.lookup()`

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint in the `rest-java` module accepts EVM address format path parameters and resolves them via a direct, uncached, unthrottled database query on every request. An unprivileged attacker can flood this endpoint with unique non-existent EVM addresses, causing unbounded `SELECT` queries against the `entity` table that saturate the database connection pool and degrade or deny service to legitimate users.

### Finding Description

**Exact code path:**

`AllowancesController.getNftAllowances()` receives the `{id}` path variable as an `EntityIdParameter`. When the value matches the EVM address regex (40 hex chars, optionally prefixed with `0x` or `shard.realm.`), it is parsed as `EntityIdEvmAddressParameter`. [1](#0-0) 

The controller delegates to `NftAllowanceServiceImpl.getNftAllowances()`, which calls `entityService.lookup(request.getAccountId())`. [2](#0-1) 

Inside `EntityServiceImpl.lookup()`, the `EntityIdEvmAddressParameter` branch unconditionally calls `entityRepository.findByEvmAddress()` with no caching layer: [3](#0-2) 

`EntityRepository.findByEvmAddress()` executes a raw SQL query against the `entity` table on every invocation — no `@Cacheable`, no negative-result caching: [4](#0-3) 

**Root cause:** The `rest-java` `EntityServiceImpl` has no caching for EVM address lookups, unlike the `importer`'s `EntityIdServiceImpl` which wraps identical lookups in `cacheLookup()` backed by a Spring `Cache`. [5](#0-4) 

**Why existing checks fail:**

- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module and is not wired into `rest-java`. [6](#0-5) 
- The `rest-java` configuration files (`RestJavaConfiguration`, `WebMvcConfiguration`) contain no rate-limiting filter or interceptor for this endpoint. [7](#0-6) 
- The `EntityIdNumParameter` branch returns an `EntityId` directly from the parsed number with zero DB interaction, so only EVM address and alias formats are affected. [8](#0-7) 

### Impact Explanation
Each request with a unique non-existent EVM address issues one `SELECT id FROM entity WHERE evm_address = ? AND deleted <> true` query that returns no rows. Under high concurrency (e.g., thousands of requests/second from a botnet or a single high-throughput client), the database connection pool is exhausted and the `entity` table index is hammered with cache-miss reads. This blocks all other queries that depend on the same database — including transaction ingestion queries from the importer — constituting a full service degradation or denial of service for the mirror node.

### Likelihood Explanation
No authentication, API key, or prior account creation is required. Any internet-accessible deployment is reachable. EVM addresses are trivially generated (random 40-hex-char strings). A single attacker with a modest HTTP load-generation tool (e.g., `wrk`, `k6`, `hey`) can sustain thousands of unique-address requests per second. The attack is repeatable indefinitely and requires no special knowledge of the target network state.

### Recommendation
1. **Add negative-result caching** to `EntityServiceImpl.lookup()` for the `EntityIdEvmAddressParameter` branch, using a short TTL (e.g., 5–30 seconds) so that repeated lookups for the same non-existent address are served from memory.
2. **Add per-IP or global rate limiting** to the `rest-java` module for all `/api/v1/accounts/{id}/allowances/*` endpoints, mirroring the `ThrottleManagerImpl` pattern already present in `web3`.
3. **Consider a bloom filter or in-memory negative cache** for EVM addresses known to not exist, to short-circuit DB queries entirely for addresses that have never been seen.

### Proof of Concept

```bash
# Generate unique random EVM addresses and flood the endpoint
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/0x${ADDR}/allowances/nfts" &
done
wait
```

**Preconditions:** Public network access to the mirror node's rest-java port (default 8084). No credentials required.

**Trigger:** Each request with a unique `0x<40-hex-chars>` path parameter that does not correspond to any entity in the database causes `EntityServiceImpl.lookup()` → `entityRepository.findByEvmAddress()` → one uncached SQL query.

**Result:** Database connection pool saturation; legitimate queries (including importer writes and other API reads) time out or fail. The mirror node stops reflecting new network state, effectively causing a network-visibility blackout for all consumers of the mirror node API.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
