### Title
Unbounded Historical Retrieval Loop via `isComplete()` Full-Page Heuristic in Throttled Mode

### Summary
In `PollingTopicMessageRetriever`, the `isComplete()` method uses a "full-page" heuristic to decide whether to keep polling: if the last DB page returned exactly `maxPageSize` messages, it assumes more data exists and repeats. In throttled mode `numRepeats` is set to `Long.MAX_VALUE` with no absolute poll cap, and the `.timeout()` guard only fires on element-emission silence — not on continuous message flow. Any unprivileged subscriber pointing at a topic with a large historical backlog can hold the retriever in an infinite DB-polling loop, consuming a DB connection and a `boundedElastic` scheduler thread for the lifetime of the gRPC stream.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`

Constructor sets `numRepeats = Long.MAX_VALUE` for throttled mode: [1](#0-0) 

The repeat predicate is `!context.isComplete()` with that unbounded repeat count: [2](#0-1) 

`isComplete()` returns `false` whenever the last page was full: [3](#0-2) 

`pageSize` is reset to 0 at the start of every poll and incremented per message in `onNext`: [4](#0-3) [5](#0-4) 

The only termination guard is a `.timeout()`: [6](#0-5) 

**Root cause and failed assumption:**

The design assumes that a full page is a transient condition — eventually the topic's historical backlog will be exhausted and a sub-full page will be returned, causing `isComplete()` to return `true`. This assumption fails for any topic whose historical message count is a multiple of `maxPageSize` (1000) or whose backlog is continuously growing faster than the 2-second poll interval can drain it.

**Why the timeout is insufficient:**

Reactor's `.timeout(Duration)` fires only when *no element is emitted* within the window (default 60 s). When the retriever is continuously returning 1000 messages per poll, elements are emitted constantly and the timeout never triggers. The timeout is documented as "How long to wait between emission of messages before returning an error": [7](#0-6) 

**Why `maxConcurrentCallsPerConnection` is insufficient:**

The Netty limit of 5 concurrent calls per connection applies per TCP connection: [8](#0-7) 

An attacker opening multiple TCP connections multiplies the allowed concurrent subscriptions linearly.

**Exploit flow:**

1. Attacker identifies a topic with a large historical backlog (e.g., a mainnet HCS topic active for months with millions of messages).
2. Attacker opens N TCP connections to port 5600 and issues 5 `subscribeTopic` calls per connection, each with `startTime=0`, no `limit`, no `endTime`.
3. `TopicMessageServiceImpl.subscribeTopic` calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true): [9](#0-8) 
4. Each subscription enters the `repeatWhen` loop. Every 2 seconds it issues a DB query returning 1000 rows. `pageSize` = 1000 = `maxPageSize`, so `isComplete()` = `false`, loop repeats.
5. With N=20 connections × 5 calls = 100 concurrent subscriptions, the server issues 100 DB queries every 2 seconds indefinitely, holding 100 `boundedElastic` threads and 100 DB connections for the duration.

### Impact Explanation

Each stuck subscription holds one `boundedElastic` scheduler thread and one DB connection for its entire lifetime. The `boundedElastic` pool defaults to `max(10, 4 × CPU_count)` threads; exhausting it stalls all other reactive work on the server. DB connection pool exhaustion causes new queries (including from legitimate subscribers and the live listener) to queue or fail. This is a resource-exhaustion griefing DoS with no economic cost to the attacker — they only need a gRPC client and knowledge of a high-volume topic ID, both of which are publicly available on mainnet/testnet.

### Likelihood Explanation

The precondition — a topic with a historical backlog large enough that each 2-second poll returns exactly 1000 messages — is trivially satisfied on any production network. Mainnet HCS topics used by DeFi protocols or NFT platforms routinely have millions of messages. The attack requires no credentials, no special permissions, and no on-chain transactions. A single attacker with a commodity machine and a gRPC client library can open dozens of connections. The attack is repeatable and persistent as long as the attacker keeps the gRPC streams open.

### Recommendation

1. **Add a maximum poll count for throttled mode.** Introduce a `maxPolls` configuration for throttled mode analogous to `unthrottled.maxPolls`. After that many polls, terminate the historical retrieval and hand off to the live listener regardless of page fullness.

2. **Add a wall-clock deadline.** Track the subscription start time in `PollingContext` and add a check in `isComplete()` that returns `true` after a configurable maximum retrieval duration (e.g., 5 minutes), independent of message flow.

3. **Bound the `boundedElastic` scheduler.** Configure an explicit thread cap on the scheduler used by the retriever so that exhaustion of retriever threads cannot starve other server components.

4. **Rate-limit subscriptions per source IP.** Apply a per-IP concurrent-subscription limit at the gRPC interceptor layer before the retriever is invoked.
