### Title
Unbounded `X-Forwarded-For` / `X-Real-IP` Header Logged Without Length Validation Enables Log-Flooding Disk Exhaustion DoS

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Forwarded-For` (and `X-Real-IP`) header value with no length cap and returns it verbatim. `TracingMiddleware` then embeds that value into a `log.Info` / `log.Debug` line for **every** request. An unauthenticated attacker can send a continuous stream of requests carrying a near-maximum-size header (up to Go's default 1 MB `MaxHeaderBytes`), causing the container-runtime log sink to grow without bound until disk space is exhausted and the node crashes.

### Finding Description

**Exact code path**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // no length check
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // no length check
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress
}
```

The only guard (`len(ipAddress) == 0`) checks for an *empty* value, not an *oversized* one.

`TracingMiddleware`, lines 52–59, then writes the raw value into every log line:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path,
    tracingResponseWriter.statusCode, time.Since(start))
// ...
log.Info(message)   // or log.Debug for internal paths
```

**Server configuration** (`rosetta/main.go`, lines 220–227): the `http.Server` struct sets `ReadHeaderTimeout`, `ReadTimeout`, etc., but **never sets `MaxHeaderBytes`**, so Go's default of `1 << 20` (1,048,576 bytes) applies. A single `X-Forwarded-For` value can therefore be up to ~1 MB.

**Log sink** (`rosetta/main.go`, line 54): `log.SetOutput(os.Stdout)`. In Docker/Kubernetes the container runtime captures stdout to JSON log files on the host disk (e.g., `/var/lib/docker/containers/<id>/<id>-json.log`). These files grow without bound unless an explicit `--log-opt max-size` / `max-file` policy is in place, which is not enforced by the application.

**Root cause**: the code assumes the client-supplied IP header is a short, well-formed string. There is no truncation, no validation against a maximum length, and no rate-limiting in the middleware chain.

### Impact Explanation

Each HTTP request with a ~1 MB `X-Forwarded-For` header produces a ~1 MB log line. At a modest 1,000 req/s the log sink grows at ~1 GB/s. Once the host disk is full:
- The container runtime can no longer write logs, causing the process to block or crash.
- The OS may be unable to write other critical files (WAL, temp files), potentially corrupting the database or causing the entire node to become unresponsive.
- Recovery requires manual operator intervention (disk cleanup, restart), meaning the node is unavailable for an extended period.

This directly impacts the availability of the Rosetta API endpoint, which is a critical infrastructure component for exchanges and wallets interacting with the network.

### Likelihood Explanation

- **No authentication required**: the Rosetta API is a public HTTP endpoint; any network-reachable client can send arbitrary headers.
- **Trivially scriptable**: a single `curl` loop or any HTTP load-generator suffices.
- **Repeatable**: the attacker can sustain the attack indefinitely from a single machine or a small botnet.
- **No existing application-level rate limiting** is present in the middleware chain.

### Recommendation

1. **Truncate the header value** in `getClientIpAddress()` before returning it:
   ```go
   const maxIPLength = 45 // max IPv6 length
   if len(ipAddress) > maxIPLength {
       ipAddress = ipAddress[:maxIPLength]
   }
   ```
2. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g., 8 KB) in `main.go`:
   ```go
   httpServer := &http.Server{
       MaxHeaderBytes: 8 << 10, // 8 KB
       ...
   }
   ```
3. **Enforce log rotation** at the deployment level (`--log-opt max-size=100m --log-opt max-file=3` for Docker, or a `logrotate` policy).
4. Consider adding a **rate-limiting middleware** before `TracingMiddleware`.

### Proof of Concept

```bash
# Generate a ~900 KB X-Forwarded-For value
PAYLOAD=$(python3 -c "print('1.2.3.4, ' * 100000)")

# Flood the node (adjust host/port as needed)
while true; do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: $PAYLOAD" \
    http://<rosetta-node>:8082/network/list \
    -X POST -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'
done
```

Each iteration writes a ~900 KB log line to stdout. The container-runtime log file grows at the rate of requests × ~900 KB per request. Disk exhaustion and node crash follow within minutes on a typical deployment. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
