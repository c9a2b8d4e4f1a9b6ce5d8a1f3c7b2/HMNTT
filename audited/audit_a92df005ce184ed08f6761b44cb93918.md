### Title
Unauthenticated DoS via Uncached, Unthrottled `GET /api/v1/network/stake` Causing Database CPU Exhaustion

### Summary
`NetworkServiceImpl.getLatestNetworkStake()` in the `rest-java` module calls `networkStakeRepository.findLatest()` on every request with no caching and no rate limiting. Each call executes a correlated subquery (`SELECT * FROM network_stake WHERE consensus_timestamp = (SELECT MAX(consensus_timestamp) FROM network_stake)`) directly against the database. An unauthenticated attacker flooding this endpoint can exhaust database CPU, denying stake data to all consumers including staking transaction construction.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (line 127-130) → `NetworkServiceImpl.getLatestNetworkStake()` (lines 52-56) → `NetworkStakeRepository.findLatest()` (lines 12-19).

```
// NetworkServiceImpl.java lines 52-56
public NetworkStake getLatestNetworkStake() {
    return networkStakeRepository
            .findLatest()
            .orElseThrow(() -> new EntityNotFoundException("No network stake data found"));
}
```

```
// NetworkStakeRepository.java lines 12-19
@Query(value = """
    select *
    from network_stake
    where consensus_timestamp = (
        select max(consensus_timestamp) from network_stake
    )
    """, nativeQuery = true)
Optional<NetworkStake> findLatest();
```

**Root cause — two absent controls:**

1. **No caching**: The `rest-java` `NetworkStakeRepository` has zero `@Cacheable` annotations. Contrast with the `grpc` module's `NodeStakeRepository` which explicitly annotates `findAllStakeByConsensusTimestamp` with `@Cacheable(cacheManager = NODE_STAKE_CACHE, cacheNames = CACHE_NAME)`. The rest-java service layer also has no `@Cacheable` anywhere (confirmed: grep for `Cacheable` in `rest-java/src/main/java/**` returns no matches).

2. **No rate limiting**: The `ThrottleManager`/`ThrottleManagerImpl`/`ThrottleConfiguration` stack exists exclusively in the `web3` module and is wired only to `ContractController`. The `rest-java` `NetworkController` has no throttle dependency, no bucket4j guard, and no per-IP limiting. The `rest-java/src/main/resources/` directory contains only `banner.txt` — no application configuration defining connection pool caps or statement timeouts for this module.

**Why checks fail:** The only throttle infrastructure in the codebase is scoped to `hiero.mirror.web3.throttle.*` properties and the `web3` Spring context. The `rest-java` Spring context is entirely separate and inherits none of it.

### Impact Explanation
Every concurrent `GET /api/v1/network/stake` request fires a correlated aggregate subquery against `network_stake`. Under high concurrency this saturates database CPU, causing query latency to spike across all tables sharing the same PostgreSQL instance (the mirror node uses a shared DB). Stake data becomes unavailable, breaking any client or SDK that calls this endpoint to construct staking transactions (e.g., `CryptoCreate`/`CryptoUpdate` with staking fields). The `network_stake` table is updated only once per staking period (~24 h), so the result is effectively static between updates — making the absence of caching purely a defensive gap with no functional justification.

### Likelihood Explanation
The endpoint is a public, unauthenticated `GET` with no parameters, making it trivially scriptable. A single attacker with a modest botnet or even a single high-concurrency HTTP client (e.g., `wrk`, `hey`, `k6` — the repo itself ships a k6 load test at `tools/k6/src/rest-java/test/networkStake.js`) can sustain thousands of RPS. No authentication, no API key, no CAPTCHA, and no application-layer rate limit stands in the way.

### Recommendation
1. **Add `@Cacheable`** to `NetworkStakeRepository.findLatest()` in the `rest-java` module with a TTL aligned to the staking period (e.g., 5–60 minutes), mirroring the pattern already used in the `grpc` module's `NodeStakeRepository`.
2. **Add application-layer rate limiting** to the `rest-java` `NetworkController` (or a servlet filter/Spring interceptor), analogous to the bucket4j `ThrottleManager` in the `web3` module.
3. **Set a PostgreSQL `statement_timeout`** for the `rest-java` DB role to bound the worst-case query duration.
