### Title
Unbounded Memory Accumulation in EVM Opcode Replay Causes JVM OOM via `memory=true` Parameter

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` allows any unauthenticated caller to replay a historical transaction with `memory=true`. During replay, `AbstractOpcodeTracer.captureMemory()` snapshots the entire EVM memory frame at every single opcode step with no size cap, and all snapshots are accumulated in an unbounded `ArrayList` inside `OpcodeContext`. For a transaction that executed a memory-intensive loop, this causes the JVM to allocate gigabytes of heap in a single request, crashing the mirror node process.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) accepts `memory=true` from any caller with no authentication. The only guards are a gzip header check and a global rate-limit bucket: [1](#0-0) 

The rate limit is configured at `opcodeRequestsPerSecond = 1` (one request per second globally, not per-IP): [2](#0-1) 

`ThrottleManagerImpl.throttleOpcodeRequest()` only checks the global bucket — a single request is always allowed through: [3](#0-2) 

`OpcodeServiceImpl.processOpcodeCall()` creates an `OpcodeContext` with initial capacity `gas / 3` (up to 5,000,000 for the 15M gas limit) and passes it to the EVM replay: [4](#0-3) 

`OpcodeActionTracer.tracePostExecution()` is called after **every single opcode** and invokes `captureMemory()`: [5](#0-4) 

`AbstractOpcodeTracer.captureMemory()` reads the **entire EVM memory** at each opcode step with no size cap, converting every 32-byte word into a 66-character hex string and adding it to a new `ArrayList`: [6](#0-5) 

All resulting `Opcode` objects (each carrying the full memory snapshot) are appended to an unbounded `ArrayList` in `OpcodeContext`: [7](#0-6) 

**Root cause:** There is no cap on (a) the EVM memory size captured per opcode step, (b) the number of opcode snapshots accumulated, or (c) the total heap consumed per request.

**Memory math:** With `maxGasLimit = 15,000,000`: [8](#0-7) 

A contract that allocates ~10,000 EVM memory words (320 KB raw, ~225K gas for expansion) and then loops on already-allocated memory can execute ~869,000 loop iterations on the remaining gas. Each iteration triggers `captureMemory()`, producing a snapshot of 10,000 × 66-byte strings (~660 KB of Java strings per opcode). Total accumulated heap: ~869,000 × 660 KB ≈ **574 GB** — the JVM OOMs long before completion.

### Impact Explanation

A single HTTP request causes an unrecoverable `OutOfMemoryError` in the mirror node JVM, crashing the process. This takes down all mirror node services (REST API, gRPC, web3 API) for all users. Recovery requires a process restart. Because the rate limit is global (not per-IP) and allows 1 req/sec, a single attacker can continuously crash the node at 1-second intervals, constituting a sustained denial of service. This matches the stated scope: shutdown of ≥30% of network processing nodes without brute force.

### Likelihood Explanation

**Preconditions:**
1. The opcode tracer feature must be enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`). This is `false` by default but is the intended production configuration for debug/trace nodes.
2. A memory-intensive transaction must exist on-chain. Any contract using dynamic arrays, sorting, or data-processing loops qualifies. The attacker does not need to deploy the contract — they only need to find an existing transaction hash.
3. No credentials are required.

The attacker needs only a valid transaction hash (publicly visible on-chain) and the ability to send one HTTP GET request with `Accept-Encoding: gzip`. This is trivially achievable by any external party.

### Recommendation

1. **Add a hard cap on memory snapshots per opcode:** In `captureMemory()`, return `Collections.emptyList()` (or a truncated list) if `frame.memoryWordSize()` exceeds a configurable threshold (e.g., 1,024 words / 32 KB).
2. **Add a hard cap on total opcode count:** In `OpcodeContext.addOpcodes()`, throw or silently drop entries once the list exceeds a configurable limit (e.g., 100,000 opcodes).
3. **Add per-request memory budget tracking:** Track cumulative bytes allocated during a single replay and abort with HTTP 413 or 503 if the budget is exceeded.
4. **Add per-IP rate limiting** in addition to the global bucket to prevent a single attacker from monopolizing the 1 req/sec slot.
5. **Consider running EVM replay in a separate JVM/process** with a hard memory limit so an OOM does not kill the main mirror node process.

### Proof of Concept

**Step 1 — Deploy a memory-intensive contract on the target network:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;
contract MemoryBomb {
    function bomb() external pure returns (uint256) {
        uint256 result;
        // Allocate ~10,000 words (320 KB) of memory, then loop
        assembly {
            // Expand memory to word 10000
            mstore(mul(10000, 32), 0)
            // Loop ~800,000 times on already-allocated memory
            let i := 0
            loop:
                mstore(mod(i, 10000), i)
                i := add(i, 1)
                jumpi(loop, lt(i, 800000))
            result := mload(0)
        }
        return result;
    }
}
```

**Step 2 — Execute the transaction and record its hash** (standard Ethereum/Hedera transaction submission).

**Step 3 — Trigger the exploit:**
```bash
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/<TX_HASH>/opcodes?memory=true&stack=true"
```

**Expected result:** The mirror node JVM throws `java.lang.OutOfMemoryError: Java heap space` during the EVM replay inside `captureMemory()`, crashing the process. All mirror node endpoints become unavailable until the process is restarted.

```java
    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
```
