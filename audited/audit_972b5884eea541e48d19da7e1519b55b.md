### Title
Unbounded Subscription Accumulation via Near-Future `endTime` Causes Resource Exhaustion DoS

### Summary
An unprivileged attacker can open an unlimited number of gRPC topic subscriptions by setting `endTime = startTime + 1` nanosecond. Due to the `endTimeInterval` grace period in `isComplete()`, each subscription remains alive for at least 30 seconds regardless of how quickly the `endTime` passes. With no total connection limit enforced, an attacker can accumulate thousands of concurrent subscriptions, each driving a DB-polling loop and consuming bounded-elastic scheduler threads.

### Finding Description

**Validation bypass — `TopicMessageFilter`**

`isValidEndTime()` only requires `endTime > startTime` (strictly greater by even 1 nanosecond): [1](#0-0) 

`isValidStartTime()` only requires `startTime <= now`, so an attacker sets `startTime = now`, `endTime = now + 1`. Both validations pass.

**`pastEndTime()` — keeps subscription alive with 30-second polling loop** [2](#0-1) 

This flux repeats every `endTimeInterval` (default 30 s) until `isComplete()` returns `true`. It is wired as the termination signal for the entire subscription pipeline via `takeUntilOther`.

**`isComplete()` — grace period prevents early termination** [3](#0-2) 

Even when `endTime = now + 1 ns`, the second branch only fires when `endTime + endTimeInterval < Instant.now()`, i.e., ~30 seconds after `endTime`. The first branch (`filter.getEndTime() < startTime`) does not fire because `endTime > startTime` was enforced by validation. Result: every subscription with a future `endTime` lives for at least one full `endTimeInterval` (30 s).

**`PollingTopicListener` — per-subscription DB polling loop on bounded-elastic scheduler** [4](#0-3) 

Each call to `listen()` creates an independent `RepeatSpec` polling loop scheduled on a shared `Schedulers.boundedElastic()`. Every active subscription issues DB queries at the configured interval for its entire 30-second lifetime.

**`maxConcurrentCallsPerConnection = 5` does not bound total subscriptions** [5](#0-4) [6](#0-5) 

This limit is per-connection only. There is no total connection cap, no per-IP rate limit, and no maximum subscriber count enforcement anywhere in the code. The `subscriberCount` gauge is metrics-only.

**Exploit flow**:
1. Attacker opens N TCP connections to the gRPC port (no auth required).
2. On each connection, opens 5 subscriptions (`maxConcurrentCallsPerConnection`) with `startTime = now`, `endTime = now + 1`.
3. Each subscription passes validation, enters the pipeline, and stays alive for ~30 s.
4. Attacker repeats every few seconds; after 30 s, N×5×(30/interval) subscriptions are concurrently active.
5. Each subscription drives a DB polling loop and holds Reactor pipeline memory.

### Impact Explanation

With 200 connections × 5 calls = 1,000 concurrent subscriptions, each polling the DB every few seconds, the attacker can saturate the DB connection pool and exhaust the `boundedElastic` thread pool (capped at `10 × CPU cores` by default in Reactor). This causes legitimate subscribers to receive no messages, DB queries to queue or time out, and the gRPC service to become unresponsive — a full denial of service. No messages need to exist on the topic; the attack works on any valid topic ID.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the gRPC proto schema, and is trivially scriptable with any gRPC client library. The 30-second subscription lifetime means a sustained attack requires only modest connection throughput. The exploit is repeatable indefinitely and is not self-limiting.

### Recommendation

1. **Enforce a minimum `endTime` window**: Reject subscriptions where `endTime - startTime` is below a configurable threshold (e.g., 1 second), preventing trivially short windows that exploit the grace period.
2. **Enforce a global (or per-IP) concurrent subscription cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Reduce or eliminate the `endTimeInterval` grace period for already-expired subscriptions**: In `isComplete()`, if `endTime` is already in the past at subscription creation time (`filter.getEndTime() < DomainUtils.now()`), return `true` immediately without waiting for the grace period.
4. **Add per-IP connection rate limiting** at the Netty layer to prevent connection flooding.

### Proof of Concept

```python
import grpc
import time
# Using the Hedera/Hiero mirror node proto
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hedera.mirror.api.proto import mirror_network_service_pb2

NOW_NS = time.time_ns()

def make_filter():
    return consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,           # any valid topic ID
        consensusStartTime=...,# startTime = NOW_NS (nanoseconds)
        consensusEndTime=...,  # endTime   = NOW_NS + 1
    )

channels = []
streams  = []
for _ in range(200):                        # 200 connections
    ch = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    for _ in range(5):                      # 5 calls per connection
        stream = stub.subscribeTopic(make_filter())
        streams.append(stream)
    channels.append(ch)
    time.sleep(0.1)                         # open 10 connections/sec

# After ~30 s: 1,000 concurrent subscriptions each polling the DB
# Monitor: DB connection pool exhaustion, grpc subscriber gauge spike
time.sleep(60)
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
