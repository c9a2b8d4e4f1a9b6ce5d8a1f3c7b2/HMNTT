### Title
Unbounded O(N) `constructTransaction()` Processing Per `/block` Request with No Caching or Rate Limiting

### Summary
The `FindBetween()` function in `rosetta/app/persistence/transaction.go` fetches all transactions in a block's timestamp range across unlimited batches, then calls `constructTransaction()` once per unique transaction hash — each call performing two `json.Unmarshal` operations and multiple map-building passes. Because there is no result caching and no rate limiting on the `/block` endpoint, any unprivileged external user can repeatedly trigger this full O(N) CPU workload against any block with a large transaction count, without submitting a single transaction themselves.

### Finding Description

**Code path:**

In `FindBetween()` ( [1](#0-0) ), the loop fetches batches of `batchSize = 2000` rows with no upper bound on total rows accumulated:

```go
for start <= end {
    ...
    transactions = append(transactions, transactionsBatch...)
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

All rows are held in memory, then grouped by hash ( [2](#0-1) ), and `constructTransaction()` is called once per unique hash ( [3](#0-2) ).

Each `constructTransaction()` call performs:
- Two `json.Unmarshal` calls (lines 223, 228) [4](#0-3) 
- `categorizeHbarTransfers()` which builds multiple maps via `aggregateHbarTransfers()` and `getFeeHbarTransfers()` [5](#0-4) 

**Why `maxTransactionsInBlock` does not help:**

In `block_service.go`, the truncation check at lines 61–67 runs *after* `FindBetween()` returns — all `constructTransaction()` calls have already completed. [6](#0-5)  The limit only reduces the response payload; it does not bound the CPU work.

**No caching of `FindBetween()` results:**

`BaseService.FindBetween()` delegates directly to `transactionRepo.FindBetween()` with no memoization. [7](#0-6)  The only cache present is an LRU for entity aliases, not for block transaction results. [8](#0-7) 

**No rate limiting:**

The middleware stack is metrics + tracing + CORS only. [9](#0-8)  No per-IP or per-endpoint rate limiting exists anywhere in the rosetta middleware directory. 

### Impact Explanation
An attacker repeatedly POSTing to `/block` for a block with N unique transaction hashes forces the mirror node to re-execute O(N) `constructTransaction()` calls — each with 2 JSON unmarshals and 3–4 map iterations — on every request. On a busy Hedera mainnet block (hundreds to thousands of transactions), this compounds into significant CPU and memory pressure. Because there is no caching, the work is never amortized across requests. This can realistically push CPU consumption well above the 30% threshold relative to baseline, degrading service for all users.

### Likelihood Explanation
The Rosetta `/block` endpoint requires no authentication. The attacker does not need to submit any transactions — they only need to identify a naturally busy block (trivially done by querying recent block indices) and issue repeated POST requests. The attack is cheap, repeatable, and requires no special privileges or network access beyond HTTP connectivity to the Rosetta port.

### Recommendation
1. **Cache `FindBetween()` results** keyed by `(consensusStart, consensusEnd)` with a short TTL (e.g., 30–60 s), since historical blocks are immutable once finalized.
2. **Enforce a hard cap on total transactions fetched** inside `FindBetween()` (not just on the response), aborting or returning an error if the accumulated row count exceeds `maxTransactionsInBlock` before calling `constructTransaction()`.
3. **Add per-IP or global rate limiting** on the `/block` endpoint at the middleware layer.
4. **Pre-compute or lazily cache** the JSON-unmarshaled transfer slices at ingestion time rather than re-parsing on every API call.

### Proof of Concept
1. Identify a Hedera block with a large transaction count (e.g., query `/block` with `block_identifier: {index: <recent_busy_block>}` and note the `other_transactions` list length).
2. In a loop, repeatedly POST to the Rosetta `/block` endpoint:
   ```
   POST /block
   {"network_identifier": {...}, "block_identifier": {"index": <target_block>}}
   ```
3. Observe mirror node CPU via `top`/`htop` or Prometheus metrics (`hiero_mirror_rosetta_request_duration`). Each request triggers a full re-fetch and O(N) `constructTransaction()` processing cycle.
4. With a block containing ~2000 transactions and 50 concurrent requests/second, CPU usage on the mirror node will spike measurably above baseline with no attacker cost beyond HTTP requests.
