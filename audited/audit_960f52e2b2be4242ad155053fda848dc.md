### Title
Unbounded Timestamp Range in `getHookStorageChange` Enables Unauthenticated DoS via Full-Table Scan on `hook_storage_change`

### Summary
An unauthenticated user can supply a timestamp range spanning the entire ledger history (e.g., `timestamp=gte:0&timestamp=lte:9999999999`) to the `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint. This causes `HookServiceImpl.getHookStorageChange` to invoke `hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, forcing PostgreSQL to perform a full sequential scan of `hook_storage_change` before applying `DISTINCT ON (key)`. During a network partition where a replica is already degraded, concurrent such requests hold long-running connections, exhaust the connection pool, and cascade timeouts to all other users.

### Finding Description

**Code path:**

`HooksController.getHookStorage` (line 104ŌĆō130) accepts unauthenticated requests with optional `timestamp` parameters bounded only by `@Size(max = 2)` ŌĆö no span validation. [1](#0-0) 

`Bound.of()` is called with `primarySortField = false`, so the only range check (`adjustedLower > adjustedUpper`) is **skipped**. [2](#0-1) 

When no lower timestamp is provided, `getAdjustedLowerRangeValue()` returns `0`; when no upper is provided, `adjustUpperBound()` returns `Long.MAX_VALUE`. [3](#0-2) 

`HookServiceImpl.getHookStorageChange` passes these values directly to the repository with no span cap: [4](#0-3) 

The native query uses `SELECT DISTINCT ON (key) ŌĆ” WHERE ŌĆ” consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound`. PostgreSQL must evaluate **all rows** matching the WHERE clause before it can compute `DISTINCT ON (key)` and apply the `LIMIT`. The `Pageable` limit of 100 does not prevent the full scan ŌĆö it only limits the returned rows after the scan completes. [5](#0-4) 

**Failed assumption:** The design assumes `LIMIT` (via `Pageable`) bounds query cost. It does not for `DISTINCT ON` aggregations over unbounded ranges.

### Impact Explanation
Any unauthenticated caller can issue repeated requests that pin database connections on long-running scans. On a degraded replica during a network partition, these queries do not time out (no `statement_timeout` is configured in the rest-java service), hold connections indefinitely, exhaust the JDBC connection pool, and cause all subsequent requests to the replica to fail with connection-pool-exhaustion errors ŌĆö a full read-path denial of service.

### Likelihood Explanation
No authentication is required. The endpoint is publicly accessible. The attack requires only two HTTP parameters (`timestamp=gte:0&timestamp=lte:9999999999`). It is trivially scriptable and repeatable. A single attacker with a modest number of concurrent connections can trigger the condition. The network-partition precondition amplifies impact but is not required ŌĆö even on a healthy replica, enough concurrent wide-range requests will degrade performance.

### Recommendation
1. **Enforce a maximum timestamp span** in `HookServiceImpl.getHookStorageChange` or in `Bound.of()` when `primarySortField = false` for timestamp bounds ŌĆö reject or cap requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 7 days in nanoseconds).
2. **Set a PostgreSQL `statement_timeout`** on the read-replica connection pool used by rest-java (e.g., `spring.datasource.hikari.connection-init-sql=SET statement_timeout=5000`).
3. **Add per-IP or per-account rate limiting** to the `/hooks/{hookId}/storage` endpoint in rest-java (analogous to the throttle already present in web3).
4. **Require a timestamp upper bound** when a lower bound is provided, preventing open-ended historical scans.

            upperBound--;
