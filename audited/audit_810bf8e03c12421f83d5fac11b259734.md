### Title
Unbounded Prometheus Label Cardinality via Arbitrary HTTP Methods in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` passes `r.Method` verbatim as the `method` label to all four `prometheus.*Vec` instruments, including `requestInflightGauge`. Because Go's `net/http` server accepts any RFC 7230-valid token as an HTTP method and no normalization or allowlist exists in the middleware chain, an unauthenticated attacker can flood the server with requests carrying unique method strings, creating an unbounded number of in-memory Prometheus time-series and exhausting process memory.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83 — `MetricsMiddleware` constructs a `weaveworks/common/middleware.Instrument` and calls `.Wrap(next)`. The `Instrument.Wrap` implementation increments `InflightRequests.WithLabelValues(r.Method, route)` at the start of every request, before `next.ServeHTTP` is called. The same raw `r.Method` value is used for `requestBytesHistogram`, `requestDurationHistogram`, and `responseBytesHistogram` as well.

**Root cause:** The `method` label value is taken directly from `r.Method` with no allowlist, normalization, or truncation. The `route` label is bounded (gorilla/mux returns a fixed template string or empty string for unmatched paths), but the `method` dimension is fully attacker-controlled.

**Middleware chain (main.go lines 217–219):**
```
corsMiddleware → tracingMiddleware → metricsMiddleware → router
```
`MetricsMiddleware` sits outside the router, so it instruments every request — including those with non-standard methods — before any routing or business-logic rejection occurs.

**Failed assumption:** The code assumes the `method` label cardinality is bounded by the small set of standard HTTP verbs. Go's `net/http` server does not enforce this; it accepts any token-valid method string and delivers it to the handler unchanged.

### Impact Explanation
Each unique `(method, route)` pair causes the Prometheus client library to allocate a new `Gauge` (and histogram bucket set for the other three vectors) in the process heap. With ~20 known routes and an unbounded method dimension, an attacker can create millions of distinct label combinations. The Prometheus `GaugeVec` and `HistogramVec` internal maps grow without bound, eventually exhausting heap memory and crashing the Rosetta process (OOM kill). This is a complete denial-of-service of the Rosetta API node with no economic damage to on-chain state.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no knowledge of the application beyond the fact that it listens on an HTTP port. A trivial script sending `METHOD_<i> /network/list HTTP/1.1` in a loop is sufficient. The attack is repeatable, cheap (each request is a single TCP connection), and fully automatable. There is no rate limiter, no connection-level method filter, and no Prometheus cardinality cap configured.

### Recommendation
1. **Normalize the method label** before passing it to the metric vectors. Replace `r.Method` with a helper that returns the value only if it is in the standard set (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `HEAD`, `OPTIONS`, `CONNECT`, `TRACE`) and returns a fixed string (e.g., `"unknown"`) otherwise.
2. **Add a method allowlist at the HTTP layer** (e.g., via a thin middleware that rejects requests with non-standard methods with `405 Method Not Allowed`) placed *before* `MetricsMiddleware` in the chain.
3. Consider enabling Prometheus cardinality limits (e.g., `prometheus.NewGoCollector` with `MaxProcsVersion`) or using a `prometheus.WrapRegistererWithPrefix` that enforces a label value allowlist.

### Proof of Concept
```bash
# Send 10 000 requests each with a unique HTTP method to a known Rosetta path.
# Each request creates a new (method, route) label pair in all four metric vectors.
for i in $(seq 1 10000); do
  printf "METHOD${i} /network/list HTTP/1.1\r\nHost: target\r\nContent-Type: application/json\r\nContent-Length: 2\r\n\r\n{}" \
    | nc -q1 <rosetta-host> <rosetta-port> &
done
wait

# Observe memory growth:
curl http://<rosetta-host>:<rosetta-port>/metrics | grep hiero_mirror_rosetta_request_inflight
# Thousands of distinct {method="METHOD_N", route="..."} lines will appear.
# Repeat until OOM.
```
