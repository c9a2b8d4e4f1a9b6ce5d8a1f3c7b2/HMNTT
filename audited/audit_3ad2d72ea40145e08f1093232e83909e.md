### Title
Unrestricted `file.id` Parameter Allows Unprivileged Access to Internal Gossip Address Book (File 101)

### Summary
The `getAddressBookFileId()` method in `NetworkServiceImpl` accepts any user-supplied entity ID for the `file.id` query parameter with no allowlist validation. An unprivileged external user can supply `file.id=101` to retrieve the internal gossip address book (file 101), which contains node-to-node gossip endpoints not intended for public client exposure. The only default is `addressBookFile102`, but no enforcement prevents querying file 101 or any other arbitrary file ID.

### Finding Description
**Code path:**

`NetworkServiceImpl.java` lines 178–182 — `getAddressBookFileId()`: [1](#0-0) 

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()   // ← raw user value, no allowlist
            : systemEntity.addressBookFile102().getId();
}
```

The returned `fileId` is passed directly into the SQL query in `NetworkNodeRepository`: [2](#0-1) 

```sql
where file_id = :fileId   -- user-controlled
```

**Root cause:** `getAddressBookFileId()` performs no allowlist check. It blindly trusts `request.getFileId().value()`.

**Parameter parsing:** `EntityIdRangeParameter.valueOf()` only validates that the entity ID is syntactically valid (non-negative longs in shard.realm.num format): [3](#0-2) 

There is no semantic check against an allowlist of permitted file IDs.

**Known address books:** `SystemEntity` defines both `addressBookFile101` (entity 101, gossip/internal) and `addressBookFile102` (entity 102, client-facing): [4](#0-3) 

File 101 is the internal node-to-node gossip address book. The API is designed to default to 102 for public use, but nothing prevents a caller from explicitly requesting 101.

**`NetworkNodeRequest`:** The `fileId` field has `required = false` and no `@Valid` constraint restricting its value: [5](#0-4) 

### Impact Explanation
File 101 contains the gossip address book with internal IP addresses and ports used for node-to-node consensus communication. Exposing these endpoints to arbitrary external users enables:
- Targeted DoS or connection-flooding attacks against gossip ports
- Network topology mapping (identifying all consensus nodes and their internal addresses)
- Reconnaissance for further attacks against the gossip layer

This is an information disclosure vulnerability with medium-to-high severity because the gossip endpoints are operationally sensitive infrastructure details.

### Likelihood Explanation
Exploitation requires zero privileges — any HTTP client can issue a GET request to the network nodes endpoint with `?file.id=101`. It is trivially repeatable, requires no authentication, no special tooling, and no prior knowledge beyond knowing that Hedera uses file 101 for the gossip address book (publicly documented in the Hedera protocol). Likelihood is **high**.

### Recommendation
Add an explicit allowlist check in `getAddressBookFileId()`. Only permit the two known public address book file IDs:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long requestedId = request.getFileId().value();
    long allowed101 = systemEntity.addressBookFile101().getId();
    long allowed102 = systemEntity.addressBookFile102().getId();
    if (requestedId != allowed101 && requestedId != allowed102) {
        throw new IllegalArgumentException(
            "Invalid file.id: only address book files 101 and 102 are permitted");
    }
    return requestedId;
}
```

If file 101 should never be publicly accessible, restrict the allowlist to only `addressBookFile102`.

### Proof of Concept

**Precondition:** Mirror node REST-Java API is publicly accessible. No authentication required.

**Steps:**

1. Send a request to the network nodes endpoint with `file.id=101`:
   ```
   GET /api/v1/network/nodes?file.id=101
   ```
2. `getAddressBookFileId()` returns the entity ID for file 101 (no validation occurs).
3. `NetworkNodeRepository.findNetworkNodes()` executes the SQL with `file_id = <id of 101>`.
4. The response contains all `address_book_entry` rows for the gossip address book, including internal IP addresses, ports, and node certificates stored in `address_book_service_endpoint` for file 101.

**Expected (vulnerable) result:** HTTP 200 with JSON listing internal gossip endpoints for all consensus nodes.

**Expected (fixed) result:** HTTP 400 `Invalid file.id: only address book files 101 and 102 are permitted` (or restrict to 102 only).

    public static EntityIdRangeParameter valueOf(String entityIdRangeParam) {
        if (StringUtils.isBlank(entityIdRangeParam)) {
            return EMPTY;
        }

        var splitVal = entityIdRangeParam.split(":");
        return switch (splitVal.length) {
            case 1 -> new EntityIdRangeParameter(RangeOperator.EQ, getEntityId(splitVal[0]));
            case 2 -> new EntityIdRangeParameter(RangeOperator.of(splitVal[0]), getEntityId(splitVal[1]));
            default -> throw new IllegalArgumentException("Invalid range operator. Should have format 'operator:ID'");
        };
    }

    private static EntityId getEntityId(String entityId) {
        List<Long> parts = Splitter.on('.')
                .splitToStream(Objects.requireNonNullElse(entityId, ""))
                .map(Long::valueOf)
                .filter(n -> n >= 0)
                .toList();

        if (parts.size() != StringUtils.countMatches(entityId, ".") + 1) {
            throw new IllegalArgumentException("Invalid entity ID");
        }

        var properties = CommonProperties.getInstance();
        return switch (parts.size()) {
            case 1 -> EntityId.of(properties.getShard(), properties.getRealm(), parts.get(0));
            case 2 -> EntityId.of(properties.getShard(), parts.get(0), parts.get(1));
            case 3 -> EntityId.of(parts.get(0), parts.get(1), parts.get(2));
            default -> throw new IllegalArgumentException("Invalid entity ID");
        };
    }
