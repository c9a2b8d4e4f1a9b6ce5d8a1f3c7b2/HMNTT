### Title
Unauthenticated Request Flood via `getByAliasAndType()` Exhausts Database Connection Pool (GraphQL DoS)

### Summary
The GraphQL module exposes `getByAliasAndType()` publicly with no per-request rate limiting or throttling. Every call issues a native SQL query against the database. An unprivileged attacker can flood the endpoint with thousands of concurrent, syntactically valid requests carrying distinct Base32 alias values, exhausting the HikariCP connection pool and denying service to all legitimate users.

### Finding Description
**Code path:**

- `AccountController.java` line 47–48: the `account` query handler calls `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)` with no throttle guard.
- `EntityServiceImpl.java` line 29–31: `getByAliasAndType()` calls `entityRepository.findByAlias(decodeBase32(alias))` unconditionally.
- `EntityRepository.java` line 13–14: `findByAlias()` executes a native SQL query (`select * from entity where alias = ?1 and deleted is not true`) that acquires a JDBC connection from the pool for every call.

**Root cause:** The `graphql` module contains zero rate-limiting logic. The only defenses configured in `GraphQlConfiguration.java` (lines 43–48) are:
- `MaxQueryComplexityInstrumentation(200)` — limits structural complexity of a single query.
- `MaxQueryDepthInstrumentation(10)` — limits nesting depth.
- Parser token/character limits (line 34).

A minimal alias lookup query such as `{ account(input: {alias: "AAAA"}) { id } }` has complexity = 1 and depth = 1, trivially passing all instrumentation checks. The `CachedPreparsedDocumentProvider` caches only the parsed query AST, not results; every distinct alias value still triggers a live DB round-trip.

The `web3` module does have a `ThrottleManagerImpl` with bucket4j rate limiting, but that code is entirely absent from the `graphql` module — there is no equivalent `ThrottleConfiguration`, `ThrottleManager`, or filter in the graphql package.

**Exploit flow:**
1. Attacker sends N concurrent HTTP POST requests to `/graphql`, each with a unique valid Base32 alias string.
2. Each request passes all parser and instrumentation checks (complexity 1, depth 1).
3. Each request acquires a JDBC connection from HikariCP (Spring Boot default pool size: 10).
4. With N >> pool size, all connections are held; subsequent legitimate requests block waiting for a connection, then time out.
5. The service returns errors or hangs for all users until the attacker stops.

### Impact Explanation
Complete denial of service for the GraphQL endpoint. Because the database connection pool is shared, exhaustion affects all query types (`getByIdAndType`, `getByEvmAddressAndType`, etc.), not just alias lookups. The default HikariCP pool of 10 connections means even a modest flood (dozens of concurrent requests) is sufficient to saturate it. Recovery requires the attacker to stop or the pool timeout to expire.

### Likelihood Explanation
No authentication or API key is required to reach the GraphQL endpoint. The attacker needs only an HTTP client capable of concurrent requests (e.g., `curl`, `ab`, `wrk`, any scripting language). Valid Base32 strings are trivially generated. The attack is repeatable indefinitely and requires no special knowledge of the system beyond the public GraphQL schema. This is a low-skill, high-repeatability attack.

### Recommendation
1. **Add a rate-limiting filter to the `graphql` module** mirroring the `web3` module's `ThrottleConfiguration` / `ThrottleManagerImpl` pattern using bucket4j, enforcing a per-IP or global requests-per-second cap.
2. **Configure an explicit HikariCP pool size** and a short `connectionTimeout` in the graphql application configuration so pool exhaustion fails fast rather than queuing indefinitely.
3. **Consider query result caching** (not just AST caching) for alias lookups, since alias→entity mappings are stable.
4. **Deploy an API gateway or WAF** in front of the GraphQL endpoint to enforce connection-rate limits at the infrastructure level as a defense-in-depth measure.

### Proof of Concept
```bash
# Generate 500 concurrent requests with distinct alias values
for i in $(seq 1 500); do
  curl -s -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {alias: \\\"$(python3 -c "import base64, os; print(base64.b32encode(os.urandom(20)).decode())")\\\") { id } }\"}" &
done
wait
# Legitimate request now times out or receives a connection pool error
curl -X POST http://<host>/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: {alias: \"AAAAAAAAAAAAAAAA\"}) { id } }"}'
```
