### Title
Unauthenticated DoS via Epoch-0 startTime Forcing Unconditional Safety-Check Gap-Recovery Scan

### Summary
Any unauthenticated gRPC client can call `subscribeTopic` with `consensusStartTime` set to epoch 0 and omit `consensusEndTime`. Because `TopicMessageFilter.isValidStartTime()` only checks `startTime <= now()`, epoch 0 passes validation. With no `endTime`, `TopicContext.isComplete()` unconditionally returns `false`, so the 1-second safety-check `Mono.delay` always fires and triggers an **unthrottled** `topicMessageRetriever.retrieve(gapFilter, false)` that scans from epoch 0, compounding the already-expensive throttled historical scan. Repeating this across many concurrent connections amplifies database load without bound.

### Finding Description

**Code path and root cause:**

1. `ConsensusController.toFilter()` converts the protobuf timestamp to nanoseconds with no lower-bound guard: [1](#0-0) 

   For `seconds=0, nanos=0` this produces `startTime = 0`.

2. `TopicMessageFilter` validation accepts 0 because `@Min(0)` allows it and `isValidStartTime()` only checks `startTime <= DomainUtils.now()`: [2](#0-1) 

3. `TopicMessageServiceImpl.subscribeTopic()` immediately issues a throttled historical retrieval from epoch 0 (full table scan): [3](#0-2) 

4. The safety-check fires after 1 second, gated only by `!topicContext.isComplete()`: [4](#0-3) 

5. `TopicContext.isComplete()` unconditionally returns `false` when `endTime` is null: [5](#0-4) 

   Therefore the safety-check **always** fires for any live (no-endTime) subscription.

6. `missingMessages(topicContext, null)` (safety-check path) builds a `gapFilter` starting from epoch 0 (if no message has been processed yet) and calls `retrieve(gapFilter, false)` — **unthrottled**: [6](#0-5) 

7. In `PollingTopicMessageRetriever`, the unthrottled path uses `unthrottled.getMaxPolls()` repeats and `isComplete()` only terminates on a limit hit (no limit set → polls `maxPolls` times with `maxPageSize` rows each): [7](#0-6) [8](#0-7) 

**Failed assumption:** The safety check was designed to fire only when a genuine gap exists between historical and live streams. The code assumes `isComplete()` will eventually return `true` to suppress it, but with no `endTime` this never happens — the check fires unconditionally for every live subscriber.

**No authentication gate exists** in `ConsensusController.subscribeTopic()`: [9](#0-8) 

### Impact Explanation
Each malicious connection causes two overlapping full-history DB scans: one throttled (historical Flux from epoch 0) and one unthrottled (safety-check gap-recovery from epoch 0 or last processed position). With N concurrent connections the database receives 2N concurrent scans of the entire `topic_message` table from the beginning of time. This can exhaust DB connection pools, saturate I/O, and degrade or deny service to legitimate subscribers. The unthrottled path is particularly dangerous because it bypasses the page-size throttling designed to protect the database.

### Likelihood Explanation
The attack requires zero privileges — only the ability to open a gRPC connection to the mirror node's public port. The request is a standard `subscribeTopic` call with `consensusStartTime.seconds = 0`. It is trivially scriptable (e.g., with `grpcurl` or any gRPC client library) and repeatable at will. A single attacker with modest bandwidth can open hundreds of concurrent subscriptions.

### Recommendation
1. **Enforce a minimum startTime floor**: Reject or clamp `startTime` values older than a configurable maximum lookback window (e.g., 30 days) in `ConsensusController.toFilter()` or `TopicMessageFilter.isValidStartTime()`.
2. **Fix the safety-check guard**: The safety check should only fire when the historical retrieval is genuinely complete and a gap is suspected — not unconditionally. Gate it on `topicContext.getLast() != null` or a flag set when the historical Flux completes.
3. **Rate-limit concurrent subscriptions per client IP/identity** at the gRPC server layer (e.g., via `NettyProperties` or an interceptor).
4. **Require authentication** for `subscribeTopic` calls, or at minimum enforce per-client connection limits.

### Proof of Concept
```bash
# Open 50 concurrent subscriptions from epoch 0 with no end time
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{
      "topicID": {"topicNum": 1},
      "consensusStartTime": {"seconds": 0, "nanos": 0}
    }' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

Each connection immediately triggers a throttled full-history scan. After 1 second, each connection's safety-check fires (since `isComplete()` is always `false` with no `endTime`) and issues an additional unthrottled scan from epoch 0. The database receives 100 concurrent full-table scans, exhausting connection pool capacity and degrading service for all users.
