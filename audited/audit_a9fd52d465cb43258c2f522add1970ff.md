### Title
Unbounded `limit` Field Enables Indefinite Stream Retention and Resource Exhaustion via `flux.take(Long.MAX_VALUE)`

### Summary
`TopicMessageFilter` enforces only `@Min(0)` on the `limit` field with no upper bound, allowing any unprivileged caller to submit `limit=Long.MAX_VALUE`. Because `hasLimit()` returns `true` for any value `> 0`, the service calls `flux.take(Long.MAX_VALUE)`, which in Reactor is practically equivalent to no termination condition. Combined with the absence of any enforced subscriber cap, an attacker can hold arbitrarily many server-side streams open indefinitely.

### Finding Description

**Root cause — no `@Max` on `limit`:** [1](#0-0) 

Only `@Min(0)` is present. `Long.MAX_VALUE` satisfies this constraint and passes Bean Validation.

**`hasLimit()` returns `true` for `Long.MAX_VALUE`:** [2](#0-1) 

Any value `> 0` is treated as "has a limit," including `Long.MAX_VALUE`.

**`flux.take(Long.MAX_VALUE)` is called:** [3](#0-2) 

Reactor's `take(Long.MAX_VALUE)` requests `Long.MAX_VALUE` items from upstream — it never terminates by count in any realistic scenario.

**No `endTime` → stream never completes:** [4](#0-3) 

When `endTime` is `null`, `pastEndTime()` returns `Flux.never()`, so the `takeUntilOther` never fires and `isComplete()` always returns `false`.

**Subscriber count is a metric only, never enforced:** [5](#0-4) 

`subscriberCount` is registered as a Micrometer gauge but there is no guard that rejects new subscriptions when a threshold is reached.

**`GrpcProperties` has no max-subscriber or connection-timeout setting:** [6](#0-5) 

### Impact Explanation
Each open subscription holds a Reactor scheduler thread slot, a database polling connection (via `TopicMessageRetriever`), and associated heap objects for `TopicContext`. An attacker opening hundreds of concurrent streams with `limit=Long.MAX_VALUE` and no `endTime` exhausts thread pools and DB connection pools, degrading or denying service to legitimate subscribers. This is a medium-severity DoS: no funds are directly at risk, but the mirror node's gRPC layer becomes unavailable.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated by design (public mirror node). No special privileges, tokens, or prior state are required. The attack is trivially scriptable: open N concurrent gRPC streams with the crafted filter. It is fully repeatable and requires only a network connection.

### Recommendation
1. Add `@Max(Long.MAX_VALUE - 1)` or a domain-appropriate cap (e.g., `@Max(10_000)`) to the `limit` field in `TopicMessageFilter`.
2. Enforce a configurable maximum concurrent subscriber count in `TopicMessageServiceImpl` — reject new subscriptions when the count exceeds the threshold rather than merely recording the metric.
3. Add a configurable maximum stream duration for subscriptions without `endTime`, terminating streams that exceed it.

### Proof of Concept
```
# Using grpcurl against a running mirror-node gRPC endpoint:
for i in $(seq 1 500); do
  grpcurl -d '{
    "topicID": {"topicNum": 1},
    "limit": 9223372036854775807
  }' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# 500 concurrent streams, each held open indefinitely.
# Server thread pool and DB connection pool exhausted.
```
