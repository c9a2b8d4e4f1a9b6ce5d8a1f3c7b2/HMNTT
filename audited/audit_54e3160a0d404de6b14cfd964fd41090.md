### Title
Unbounded Opcode Collection in `OpcodeContext.addOpcodes()` Enables Heap Exhaustion via EVM Replay

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint is publicly accessible with no authentication. When replaying a high-gas transaction, `OpcodeContext.addOpcodes()` appends every traced EVM operation into an unbounded `ArrayList` with no cap on list size. A transaction consuming the Hedera protocol maximum of 15,000,000 gas can produce millions of `Opcode` objects, exhausting JVM heap and crashing the mirror node.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (no auth check) calls `opcodeService.processOpcodeCall(request)`. [1](#0-0) 

Inside `OpcodeServiceImpl.processOpcodeCall()`, the `OpcodeContext` is constructed with an initial ArrayList capacity of `gas / 3`: [2](#0-1) 

For a 15,000,000-gas transaction, `new ArrayList<>(5_000_000)` is allocated immediately — ~40 MB just for the backing array — before a single opcode is traced. [3](#0-2) 

During EVM replay, `OpcodeActionTracer.tracePostExecution()` calls `addOpcodes()` for every single executed EVM operation: [4](#0-3) 

`addOpcodes()` performs an unconditional `opcodes.add(opcode)` with no size check, no cap, and no circuit-breaker: [5](#0-4) 

**Root cause:** The `ArrayList` in `OpcodeContext` is unbounded. The only natural bound is the gas limit of the replayed transaction, which on Hedera is up to 15,000,000. The cheapest EVM opcode (`JUMPDEST`) costs 1 gas, so a tight loop contract can produce up to 15,000,000 traced `Opcode` objects per request. Each `Opcode` object carries `pc`, `op`, `gas`, `gasCost`, `depth`, `stack` (List), `memory` (List), `storage` (Map), and `reason` fields — conservatively 200–500 bytes each with `stack=true` (the default). That is 3–7.5 GB of heap per request.

**Why existing checks fail:**

- `throttleManager.throttleOpcodeRequest()` rate-limits the number of concurrent/sequential requests but does not limit the memory consumed per individual request. [6](#0-5) 
- The `Accept-Encoding: gzip` requirement only ensures the HTTP response is compressed over the wire; it does not prevent the in-process ArrayList from growing to gigabytes before serialization. [7](#0-6) 
- `OpcodesProperties.enabled` is a feature flag; when the feature is on, there is no per-request memory guard. [8](#0-7) 

### Impact Explanation
A single request replaying a 15M-gas tight-loop transaction can allocate 3–7.5 GB of heap, triggering `OutOfMemoryError` and crashing the JVM process. Because the mirror node is a shared infrastructure component serving all downstream clients (wallets, dApps, explorers), its crash constitutes shutdown of significant network processing capacity. Multiple concurrent requests (even if throttled to a low rate) compound the effect because each request holds its full opcode list in memory simultaneously until the response is serialized and GC'd.

### Likelihood Explanation
The attacker requires no credentials. They only need the hash of any previously recorded high-gas contract transaction — publicly visible on any Hedera block explorer. The attacker does not need to submit or pay for the transaction themselves; they simply replay an existing one. The exploit is repeatable: after a crash and restart, the same transaction hash can be used again. The only cost to the attacker is knowing one valid high-gas transaction hash, which is trivially discoverable.

### Recommendation
1. **Hard cap on opcode count:** In `addOpcodes()`, enforce a configurable maximum (e.g., 1,000,000 opcodes). Throw a controlled exception or truncate and return a partial result when the cap is reached.
2. **Cap the initial ArrayList capacity:** Replace `(int) params.getGas() / 3` with `Math.min((int) params.getGas() / 3, MAX_OPCODES)` to prevent the upfront large allocation.
3. **Per-request memory budget:** Track estimated memory usage during tracing and abort the replay if it exceeds a configured threshold.
4. **Streaming/pagination:** Instead of accumulating all opcodes in memory before responding, stream them to the HTTP response incrementally.

### Proof of Concept

1. Identify or deploy a Solidity contract with a tight loop consuming close to 15,000,000 gas (e.g., a `for` loop of `JUMPDEST`-heavy no-ops).
2. Submit the transaction to the Hedera network and record its transaction hash `TX_HASH`.
3. Wait for the mirror node to index the transaction.
4. Send the following request from any unauthenticated client:
   ```
   GET /api/v1/contracts/results/TX_HASH/opcodes?stack=true&memory=true
   Accept-Encoding: gzip
   ```
5. Observe the mirror node JVM heap grow to several gigabytes during EVM replay as `OpcodeContext.addOpcodes()` is called millions of times with no bound check.
6. The JVM throws `OutOfMemoryError`, crashing the mirror node process.
7. Repeat from step 4 after restart to confirm repeatability with zero additional cost.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
