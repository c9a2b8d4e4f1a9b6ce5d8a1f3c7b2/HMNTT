### Title
Username Enumeration via `&&` Short-Circuit Timing Side-Channel in `authenticate()`

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` uses JavaScript's short-circuit `&&` operator to chain two `tsscmp` calls, meaning the password comparison is entirely skipped when the username comparison fails. This creates a measurable timing difference between "wrong username" and "right username, wrong password," allowing an unprivileged external attacker to enumerate valid usernames and subsequently brute-force the password to gain access to the Prometheus metrics endpoint.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `authenticate()`, line 143.

```js
// line 143
return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
```

**Root cause:** `tsscmp` is a constant-time string comparison library intended to prevent timing attacks. However, the `&&` short-circuit operator undermines this protection at the composition level:

- **Wrong username path:** `tsscmp(credentials.name, username)` returns `false` → JS short-circuits → `tsscmp(credentials.pass, password)` is **never invoked** → total wall-clock time ≈ cost of 1 `tsscmp` call.
- **Correct username, wrong password path:** `tsscmp(credentials.name, username)` returns `true` → `tsscmp(credentials.pass, password)` **is invoked** → total wall-clock time ≈ cost of 2 `tsscmp` calls.

This produces a statistically distinguishable timing difference between the two rejection paths. An attacker who can send many requests and measure response latency can determine which submitted username matches the configured one.

The path check at line 156 (`normalizedPath === metricsPath`) uses a non-timing-safe `===`, but `metricsPath` is not a secret credential, so that comparison does not independently leak sensitive material. The `WWW-Authenticate: Basic realm="Metrics"` header (line 158) is only a response hint and carries no incoming-request check. The sole exploitable timing oracle is the `&&` short-circuit on line 143.

### Impact Explanation
Once the attacker enumerates the valid username, the search space for brute-forcing the password collapses to a single dimension. Successful authentication grants full read access to the Prometheus metrics endpoint, which records per-route counters, duration histograms, and response-size histograms with labels including `path`. Routes such as `/api/v1/accounts/{id}/rewards` (confirmed present in `rest/routes/accountRoute.js`) appear as labeled metric series, exposing query frequency and timing patterns for staking reward lookups — information that could reveal which accounts are being monitored and at what cadence.

**Severity: Medium.** The metrics endpoint is internal/operational, but the data it exposes (query patterns, in-flight counts, error rates per route) constitutes sensitive operational intelligence.

### Likelihood Explanation
- **Preconditions:** `config.metrics.config.authentication` must be `true` (non-default hardening); the metrics endpoint must be reachable by the attacker (network-accessible, not firewalled).
- **Attacker capability:** No privileges required. Only the ability to send HTTP requests and measure response latency.
- **Feasibility:** Network jitter makes single-sample timing attacks unreliable, but statistical aggregation over hundreds to thousands of requests (a standard technique) reliably separates the two timing distributions. The difference is one full `tsscmp` invocation, which is a non-trivial and consistent delta. This is well within the capability of an attacker on the same network segment or with a stable low-latency path.
- **Repeatability:** Fully repeatable; no rate-limiting or lockout mechanism is present in the `authenticate()` function.

### Recommendation
Evaluate both comparisons unconditionally before combining their results, so the total execution time is always the cost of two `tsscmp` calls regardless of whether the username matches:

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  if (!credentials) return false;
  // Evaluate BOTH comparisons before combining — eliminates short-circuit timing leak
  const nameMatch = tsscmp(credentials.name, username);
  const passMatch = tsscmp(credentials.pass, password);
  return nameMatch && passMatch;
};
```

Additionally, consider adding a request rate-limit or IP-based throttle on the metrics path to raise the cost of statistical timing attacks.

### Proof of Concept
```python
import requests, time, statistics, base64

TARGET = "http://<host>/swagger/metrics/"
CANDIDATE_USERNAMES = ["admin", "metrics", "monitor", "prometheus", "user"]
FIXED_WRONG_PASS = "wrongpassword"
SAMPLES = 300

timings = {}
for uname in CANDIDATE_USERNAMES:
    cred = base64.b64encode(f"{uname}:{FIXED_WRONG_PASS}".encode()).decode()
    times = []
    for _ in range(SAMPLES):
        t0 = time.perf_counter()
        requests.get(TARGET, headers={"Authorization": f"Basic {cred}"})
        times.append(time.perf_counter() - t0)
    timings[uname] = statistics.median(times)

# The username with the highest median latency had its password tsscmp also invoked
# → that is the valid configured username
print(sorted(timings.items(), key=lambda x: -x[1]))
```

The username producing the highest median response latency is the configured one. Repeat the same procedure against that username with candidate passwords to complete credential recovery. [1](#0-0) [2](#0-1)
