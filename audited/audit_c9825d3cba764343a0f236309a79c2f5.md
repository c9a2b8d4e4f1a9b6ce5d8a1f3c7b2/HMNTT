### Title
Flat Rate-Limit Token Consumed Regardless of `memory`/`storage` Flag Cost Multiplier in `getContractOpcodes()`

### Summary
The `getContractOpcodes()` endpoint accepts `memory=true` and `storage=true` query parameters that cause full EVM memory and storage snapshots to be captured at every single opcode step during transaction replay. The throttle mechanism unconditionally consumes exactly one token per request regardless of these flags, meaning an unprivileged attacker can trigger orders-of-magnitude more server-side computation per rate-limit token by enabling both flags on a historically complex transaction.

### Finding Description

**Code path:**

**Root cause and failed assumption:** The rate-limit design assumes each opcode request has roughly uniform cost. This assumption is false: enabling `memory=true` and `storage=true` multiplies per-request CPU and heap allocation by a factor proportional to `(number_of_opcodes × memory_size_at_each_step)`. A complex DeFi transaction with thousands of opcodes and growing memory can produce gigabytes of snapshot data per request, all for 1 rate-limit token.

### Impact Explanation
An attacker can exhaust server CPU and heap memory by repeatedly replaying a single historically complex transaction (e.g., a large DeFi swap or a contract deployment with many storage writes) with `memory=true&storage=true`. Each request is throttled as if it were a cheap stack-only trace. This can cause sustained high CPU load, JVM GC pressure, and potential OOM crashes, effectively denying service to all other users of the web3 module. No funds are at risk, but availability of the mirror node's web3 API is severely impacted.

### Likelihood Explanation
The endpoint requires no authentication or privileged role — any external user can call it. The only precondition is that the opcode tracer feature is enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`). The attacker only needs to identify one historically complex transaction (publicly visible on-chain) and issue repeated requests within the allowed rate. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Differentiate token cost by flags:** In `throttleOpcodeRequest()`, accept the `OpcodeRequest` (or at minimum the `memory` and `storage` booleans) and consume proportionally more tokens when these flags are enabled (e.g., `memory=true` costs N tokens, `storage=true` costs M tokens).
2. **Cap memory/storage snapshot size:** In `captureMemory()`, enforce a maximum `memoryWordSize` before snapshotting; reject or truncate if exceeded.
3. **Require authentication/authorization** for the `memory=true` and `storage=true` flags, restricting them to trusted callers.
4. **Add a per-request timeout** on the EVM replay execution to bound worst-case server time regardless of transaction complexity.

### Proof of Concept

**Preconditions:**
- Mirror node deployed with `hiero.mirror.web3.opcode.tracer.enabled=true`
- A known complex historical transaction hash (e.g., a large DeFi contract call with thousands of opcodes)

**Steps:**
```bash
# Identify a complex transaction hash from the mirror node's contract results API
TX_HASH="0x<complex_transaction_hash>"

# Issue repeated requests with memory+storage enabled, each consuming only 1 rate-limit token
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true&storage=true" \
    -o /dev/null &
done
```

Each concurrent request triggers a full EVM replay with full memory and storage snapshots at every opcode step. The server's CPU and heap are consumed at a rate far exceeding what the single-token rate limit was designed to handle, leading to degraded or unavailable service for all other API consumers.

        if (!options.isMemory()) {

        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;

            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }
