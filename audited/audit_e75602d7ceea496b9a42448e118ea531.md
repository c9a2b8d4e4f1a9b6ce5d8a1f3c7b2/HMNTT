### Title
Unbounded Concurrent `getNodes(limit=0)` Requests Exhaust `Schedulers.boundedElastic()` Pool, Causing gRPC Service DoS

### Summary
The `getNodes()` method in `NetworkServiceImpl` uses `Schedulers.boundedElastic()` to schedule repeat-paging delays with no global rate limit or concurrent-call cap across connections. An unauthenticated attacker opening many gRPC connections and sending `limit=0` requests can saturate the bounded elastic thread pool, preventing new database queries from being scheduled and effectively partitioning the service from its database.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`, lines 68–76:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())
                .withScheduler(Schedulers.boundedElastic()))   // ← shared global pool
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)  // ← limit=0 → Long.MAX_VALUE
        .doOnNext(context::onNext)
```

**Root cause:**

1. When `limit=0` is sent, `.take(Long.MAX_VALUE)` applies no subscriber-side cap.
2. `RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)` repeats until `context.completed()` is set, which only happens inside `page()` when `nodes.size() < pageSize` (line 96–98). For a typical address book of ~30 nodes with `pageSize=10`, this is 3–4 iterations, each separated by `pageDelay` (default 250 ms).
3. Each active subscription holds a `boundedElastic` worker thread for the duration of each 250 ms inter-page sleep. The `page()` call itself is a blocking `transactionOperations.execute()` (line 80), which also runs on a `boundedElastic` thread.
4. `Schedulers.boundedElastic()` is a **global, shared** singleton with a default cap of `10 × availableProcessors` threads (e.g., 40 threads on a 4-core host) and a task queue of 100,000.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java`, line 14; `GrpcConfiguration.java`, line 33) limits calls **per TCP connection only**. An attacker opens N connections × 5 calls = 5N concurrent streams.
- There is no global gRPC call limit, no IP-based rate limiter, and no authentication requirement for `getNodes()`. The `GrpcInterceptor` (test path `grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`) only sets endpoint context; no production rate-limiting interceptor exists for this service.
- The `ThrottleManagerImpl` with bucket4j rate limiting exists only in the `web3` module, not in `grpc`.
- The GCP backend policy `maxRatePerEndpoint: 250` (`charts/hedera-mirror-grpc/values.yaml`, line 69) is a per-endpoint RPS cap at the load balancer, not a concurrent-stream cap, and does not apply in non-GCP or direct-access deployments.

### Impact Explanation

Once `boundedElastic` is saturated, all new reactive chains that need a `boundedElastic` thread (including the blocking DB calls inside `page()`) queue up or are rejected with `reactor.core.scheduler.Schedulers$BoundedElasticScheduler` overflow errors. This prevents the gRPC service from executing any further database queries — effectively a network partition between the service and its PostgreSQL backend. All in-flight and new `getNodes()` and `subscribeTopic()` calls stall or error. The service becomes unavailable to all legitimate clients.

### Likelihood Explanation

No authentication is required. The attacker needs only a gRPC client (e.g., `grpcurl`) and the ability to open multiple TCP connections. On a 4-core host with 40 `boundedElastic` threads and a 250 ms page delay, saturating the pool requires approximately `40 / (pageDelay_s × pages_per_request)` ≈ 40 / (0.25 × 4) = 40 concurrent streams — achievable with just 8 TCP connections at 5 streams each. The attack is trivially repeatable and requires no special knowledge beyond the public proto definition.

### Recommendation

1. **Add a global concurrent-stream limit** in `GrpcConfiguration` using `serverBuilder.maxConcurrentCallsPerConnection` combined with a global semaphore or `maxConnectionAge`/`maxConnectionIdle` settings.
2. **Enforce a hard cap on `limit=0` behavior**: treat `limit=0` as a configurable maximum (e.g., `addressBookProperties.getMaxLimit()`) rather than `Long.MAX_VALUE`.
3. **Offload blocking DB calls explicitly**: wrap `transactionOperations.execute()` in `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())` with a bounded per-request timeout, and use a **dedicated** (not the global shared) `boundedElastic` scheduler for address book paging to isolate blast radius.
4. **Add a gRPC-layer rate limiter** (e.g., a `ServerInterceptor` using bucket4j) analogous to `ThrottleManagerImpl` in the `web3` module, applied to `NetworkService/getNodes`.

```java
    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
```
