### Title
Unbounded `inner_transactions` Array Causes Amplified SQL Query DoS in `getTransactionsByIdOrHash`

### Summary
In `rest/transactions.js`, `getTransactionsByIdOrHash` collects all `inner_transactions` from the initial query result and passes the entire array directly to `getTransactionsByTransactionIdsSql` with no upper-bound check. `getTransactionsByTransactionIdsSql` generates one SQL `OR` condition per inner-transaction pair, so a batch transaction with many inner transactions produces an arbitrarily large SQL query. Because the batch transaction is stored on-chain and the REST endpoint is public, any user who can pay Hedera network fees can plant the trigger once and then replay the expensive query indefinitely at zero marginal cost.

### Finding Description

**Exact code path:**

`rest/transactions.js`, `getTransactionsByIdOrHash` (lines 944–958):

```js
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();                                          // no length cap

if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,          // ← passed verbatim, no limit
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);
  ...
}
```

`getTransactionsByTransactionIdsSql` (lines 840–863) iterates every pair and appends one `OR` clause per pair:

```js
for (let index = 0; index < transactionKeys.length; index += 2) {
  idConditions.push(`(payer_account_id = $N and valid_start_ns = $M)`);
  ...
}
// mainConditions = "... AND (cond1 OR cond2 OR ... OR condN) AND ..."
```

**Root cause:** The code assumes `inner_transactions` will always be small. The DB column is `bigint[] default null` with no array-length constraint. No application-level cap is applied before building the SQL string.

**Exploit flow:**
1. Attacker submits an ATOMIC_BATCH transaction to the Hedera network containing the maximum allowed number of inner transactions (protocol limit is bounded by the 6 KB transaction-size cap, yielding ~30–60 inner transactions per batch).
2. The importer stores the batch row with `inner_transactions = [payer1, ts1, payer2, ts2, …, payerN, tsN]` (2 × N longs).
3. Attacker repeatedly calls `GET /api/v1/transactions/<batchTxId>` — a fully public, unauthenticated endpoint.
4. Each call triggers `getTransactionsByTransactionIdsSql` with 2 × N keys, generating a SQL query with N `OR` conditions plus N bind parameters.
5. The DB must evaluate the full OR chain on every request; with concurrent requests the DB connection pool and query planner are stressed.

**Why existing checks are insufficient:**
- The only guard is `innerTransactions.length > 0` (line 949) — a lower-bound check, not an upper-bound check.
- `acceptedSingleTransactionParameters` (line 1014) only restricts `nonce` and `scheduled` query params; it does not limit the inner-transaction fan-out.
- There is no rate-limiting or query-complexity cap in the middleware visible in this path.

### Impact Explanation
An attacker can cause repeated, abnormally large SQL queries against the mirror-node database using a single on-chain batch transaction as a permanent trigger. Each REST request is cheap (HTTP GET, no auth), while the DB cost scales with the number of inner transactions. Under concurrent load this can exhaust the DB connection pool, cause statement timeouts, or degrade query performance for all other users — a targeted, repeatable DoS against the mirror-node read path.

### Likelihood Explanation
Submitting a batch transaction to Hedera requires paying a small HBAR fee (fractions of a cent at typical prices), which is a negligible barrier. Once the batch is on-chain it is permanent and immutable; the attacker never needs to pay again. The REST endpoint is public and unauthenticated. The attack is trivially scriptable (e.g., `while true; do curl …; done`) and repeatable indefinitely.

### Recommendation
Add an explicit upper-bound check on `innerTransactions.length` before calling `getTransactionsByTransactionIdsSql`, and enforce the same limit inside that function:

```js
const MAX_INNER_TRANSACTIONS = 50; // tune to protocol max
if (!isTransactionHash && innerTransactions.length > 0) {
  if (innerTransactions.length > MAX_INNER_TRANSACTIONS * 2) {
    throw new InvalidArgumentError('inner_transactions exceeds maximum allowed size');
  }
  ...
}
```

Additionally, consider adding a `LIMIT` clause or pagination to the inner-transaction sub-query, and apply rate-limiting at the API gateway level for the `/transactions/:id` endpoint.
