### Title
Missing `lower(timestamp_range) <= blockTimestamp` Guard in Entity Branch of `findActiveByEvmAddressAndTimestamp` Causes Current State to Override Correct Historical State

### Summary
The entity branch of the `UNION ALL` in `findActiveByEvmAddressAndTimestamp` omits the `lower(timestamp_range) <= ?2` predicate that is present in the `entity_history` branch and in the analogous `findActiveByIdAndTimestamp` query. Because the final `ORDER BY timestamp_range DESC LIMIT 1` always prefers the row with the highest `timestamp_range` lower bound, the current-state row from `entity` (whose `timestamp_range` starts at the entity's last-modification time, which may be far in the future relative to `blockTimestamp`) will always win over the correct historical row from `entity_history`. Any unprivileged caller who supplies a historical `block` value to the `/api/v1/contracts/call` endpoint can trigger this path and receive the wrong entity state.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`) and the identical pattern at lines 95–121 (`findActiveByEvmAddressOrAliasAndTimestamp`).

**Entity branch (lines 67–72) — no timestamp guard:**
```sql
(
    select *
    from entity e
    where e.deleted is not true          -- ← only guard
    and e.id = (select id from entity_cte)
)
```

**History branch (lines 74–81) — correctly guarded:**
```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2   -- ← timestamp guard present
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

**Contrast with the correctly-written `findActiveByIdAndTimestamp` (lines 136–142):**
```sql
select *
from entity
where id = ?1 and lower(timestamp_range) <= ?2   -- ← guard present
and deleted is not true
```

**Root cause:** The `entity_cte` only checks `created_timestamp <= ?2`, confirming the entity existed at the queried block. It does not constrain the entity's *current* `timestamp_range` lower bound. The entity row in the `entity` table always carries `timestamp_range = [T_last_modified, ∞)`. If `T_last_modified > blockTimestamp`, that row should be excluded, but it is not.

**Exploit flow:**

| Step | Detail |
|---|---|
| Precondition | Entity with `evm_address = A` was created at `T=100` and later modified at `T=200`. `entity` table holds `timestamp_range=[200,∞)`. `entity_history` holds `timestamp_range=[100,200)`. |
| Trigger | Attacker calls `POST /api/v1/contracts/call` with `"block": "<block_at_T=150>"`. The service resolves this to `blockTimestamp=150` and calls `findActiveByEvmAddressAndTimestamp(A, 150)`. |
| Query execution | `entity_cte` resolves the ID (created_timestamp=100 ≤ 150 ✓). Entity branch returns the row with `timestamp_range=[200,∞)` (no guard). History branch returns the row with `timestamp_range=[100,200)`. `ORDER BY timestamp_range DESC LIMIT 1` picks `[200,∞)`. |
| Result | The **current** (post-T=200) state is returned instead of the correct T=150 historical state. |

**Why existing checks are insufficient:**
- `entity_cte`'s `created_timestamp <= ?2` only proves the entity existed; it does not bound the entity's current modification time.
- The `deleted is not true` guard is orthogonal.
- The test suite does not cover the production-normal scenario (entity has higher timestamp than entity_history). The test `findHistoricalEntityByEvmAddressAndTimestampRangeEqualToBlockTimestampAndDeletedIsFalse` (lines 157–168) constructs the data in reverse order (`persistEntityHistoryWithId` after `persistEntity`), giving entity_history the higher timestamp, which accidentally makes the test pass while leaving the real bug undetected.

### Impact Explanation
Historical `eth_call` simulations that resolve an account or contract by EVM address will receive the entity's **current** state (balances, keys, admin settings, freeze/KYC flags, etc.) rather than the state valid at the requested block. Any off-chain system, audit tool, or dApp that relies on historical EVM simulation accuracy (e.g., replaying past governance votes, verifying historical token ownership, auditing past access-control state) will receive silently incorrect data. The same flaw exists in `findActiveByEvmAddressOrAliasAndTimestamp`, widening the affected surface to alias-based lookups as well.

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint is publicly documented and requires no authentication. The `block` field accepts any historical block number. Any entity that has ever been modified (key rotation, balance change, freeze/unfreeze, etc.) is a valid target. The attacker needs only to know a valid EVM address and any block number prior to the entity's last modification — both are trivially discoverable from public mirror-node REST APIs. The attack is fully repeatable and requires no special privileges.

### Recommendation
Add `lower(e.timestamp_range) <= ?2` to the entity branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
(
    select *
    from entity e
    where e.deleted is not true
    and lower(e.timestamp_range) <= ?2      -- add this line
    and e.id = (select id from entity_cte)
)
```

Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp`. Add a regression test that:
1. Persists an entity at `T1` (lower timestamp, goes to `entity_history`).
2. Persists the same entity ID at `T2 > T1` (current state in `entity`).
3. Queries with `blockTimestamp = T1` and asserts the `entity_history` row is returned, not the `entity` row.
