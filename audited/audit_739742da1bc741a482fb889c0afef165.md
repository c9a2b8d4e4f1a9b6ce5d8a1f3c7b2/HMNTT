### Title
Unbounded Concurrent Topic Subscriptions Enable Resource Exhaustion via Fan-Out DoS

### Summary
`ConsensusController.subscribeTopic()` accepts subscription requests from any unauthenticated caller with no enforcement of a maximum concurrent subscriber count per topic or globally. The only subscriber tracking (`subscriberCount`) is a metrics gauge that is never checked before accepting a new subscription. An attacker opening thousands of connections and subscribing to the same topic causes the shared `topicListener` to fan out every incoming message to all subscribers simultaneously, exhausting memory and CPU.

### Finding Description

**Entry point — no admission control:**

`ConsensusController.subscribeTopic()` (lines 43–53) immediately delegates to `topicMessageService::subscribeTopic` with zero checks on how many subscriptions already exist. [1](#0-0) 

**Subscriber counter is metric-only — never enforced:**

`TopicMessageServiceImpl` maintains `subscriberCount` (line 48) and increments/decrements it around each subscription lifecycle (lines 89–90), but this value is only wired to a Micrometer `Gauge` (lines 52–55). There is no guard of the form `if (subscriberCount.get() >= maxSubscribers) reject()` anywhere in the class. [2](#0-1) [3](#0-2) 

**`GrpcProperties` has no max-subscriber field:**

`GrpcProperties` exposes only `checkTopicExists`, `endTimeInterval`, `entityCacheSize`, and `netty`. There is no `maxSubscribers`, `maxSubscribersPerTopic`, or equivalent property. [4](#0-3) 

**The only relevant limit is per-connection, not global:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` caps concurrent RPC calls on a single TCP/HTTP2 connection. An attacker trivially bypasses this by opening multiple connections — there is no limit on the number of connections. [5](#0-4) 

**Fan-out is unbounded via shared listener:**

`SharedTopicListener.listen()` attaches each subscriber to the shared `Flux` returned by `getSharedListener()`. Each subscriber gets its own `onBackpressureBuffer(maxBufferSize)` (default 16 384 slots). Every message published to the topic is delivered to every active subscriber. [6](#0-5) 

`RedisTopicListener.getSharedListener()` uses `.share()` so one Redis channel subscription fans out to all downstream subscribers. [7](#0-6) [8](#0-7) 

### Impact Explanation

With N connections × 5 subscriptions each = 5N active subscribers on one topic:

- **Memory**: each subscriber holds a `onBackpressureBuffer` of up to 16 384 `TopicMessage` objects. At 5 000 connections that is 25 000 buffers × 16 384 slots, easily exhausting JVM heap (default limit 2 048 Mi per the Helm chart).
- **CPU**: every incoming message triggers 5N reactive pipeline executions on `Schedulers.boundedElastic()`, saturating the thread pool.
- **Cascading failure**: OOM or thread-pool exhaustion causes the gRPC pod to crash or become unresponsive, severing all legitimate subscribers — a network partition from the perspective of HCS consumers.

Severity: **Critical** — complete denial of service of the mirror node gRPC API with no authentication required. [9](#0-8) 

### Likelihood Explanation

- No authentication or API key is required to call `subscribeTopic`.
- Opening thousands of gRPC connections is trivial with standard tooling (`ghz`, `grpcurl`, custom scripts).
- The `maxConcurrentCallsPerConnection = 5` limit is the only server-side guard and is trivially bypassed by using more connections.
- The attack is repeatable and can be sustained indefinitely since subscriptions with no `endTime` and no `limit` never self-terminate.
- A single attacker with a modest number of machines (or even one machine with many file descriptors) can execute this.

### Recommendation

1. **Add a configurable global maximum subscriber count** to `GrpcProperties` (e.g., `maxSubscribers`) and enforce it at the start of `TopicMessageServiceImpl.subscribeTopic()` before constructing the `TopicContext`.
2. **Add a per-topic subscriber cap** tracked in a `ConcurrentHashMap<EntityId, AtomicLong>` and checked/incremented atomically before subscribing.
3. **Add a per-IP connection rate limit** at the Netty/load-balancer layer to bound the number of simultaneous connections from a single source.
4. **Require authentication** (e.g., API key or mTLS) for long-lived streaming RPCs.

The design pattern already exists in the codebase: the GraphQL `ContractLogTopicListener` design document specifies a `canSubscribe()` check against `maxActiveSubscriptions` before allowing a new subscription — the same pattern should be applied here. [10](#0-9) 

### Proof of Concept

```bash
# Step 1: Install ghz (gRPC benchmarking tool) or use a custom script
# Step 2: Open 2000 concurrent streaming subscriptions to the same topic
# (each connection gets 5 streams = 10,000 total subscribers)

for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}}' \
    -H 'content-type: application/grpc' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 3: Observe JVM heap exhaustion via:
#   curl http://<mirror-node-host>:8080/actuator/metrics/jvm.memory.used
# Step 4: Legitimate subscribers begin receiving errors or no responses
# Step 5: Pod OOMKilled or gRPC port stops accepting connections
```

Each background process holds an open streaming RPC. With no `endTime` and no `limit` in the request, each subscription is infinite. The `subscriberCount` metric will climb without bound, and each new topic message fans out to all active subscribers simultaneously.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
