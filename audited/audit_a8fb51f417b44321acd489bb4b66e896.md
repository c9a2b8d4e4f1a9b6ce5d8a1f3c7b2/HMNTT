### Title
Unbounded gRPC Topic Subscription Duration Enables Resource Exhaustion DoS

### Summary
`subscribeTopic()` in `ConsensusController` accepts `ConsensusTopicQuery` requests with no server-side enforcement of a maximum subscription duration. When a client omits `consensusEndTime` and `limit`, the resulting reactive pipeline is configured to run forever (`Flux.never()`). An unauthenticated attacker opening many connections — each saturating the per-connection call limit — can permanently hold server threads, DB polling slots, and Redis listener capacity, denying service to legitimate users.

### Finding Description
**Exact code path:**

In `ConsensusController.java` lines 43–53, `subscribeTopic()` converts the raw protobuf request to a `TopicMessageFilter` via `toFilter()` and immediately delegates to `topicMessageService::subscribeTopic` with no duration cap applied:

```java
// ConsensusController.java:44-48
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`toFilter()` (lines 55–73) only copies `limit`, `startTime`, and `endTime` from the client request. `TopicMessageFilter` has no `maxDuration` field and no server-enforced cap on how far into the future `endTime` may be set (or whether it is set at all).

In `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92), the termination signal for the live stream is `pastEndTime(topicContext)`. That method (lines 123–131) explicitly returns `Flux.never()` when `filter.getEndTime() == null`:

```java
// TopicMessageServiceImpl.java:124-125
if (topicContext.getFilter().getEndTime() == null) {
    return Flux.never();
}
```

This means the merged live stream `Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext))` never receives a termination signal from the server side. The subscription runs until the client disconnects.

**Root cause:** The server unconditionally trusts the client to supply a finite `endTime` or `limit`. No server-side maximum subscription lifetime is enforced anywhere in `GrpcProperties`, `GrpcConfiguration`, or `TopicMessageFilter`.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java:14`, applied in `GrpcConfiguration.java:33`) is a *per-connection* limit. An attacker opens N independent TCP connections, each carrying 5 indefinite subscriptions, for a total of 5N concurrent subscriptions. No global connection count limit or `maxConnectionAge` is configured in `GrpcConfiguration`.
- The nginx `grpc_read_timeout 600s` (`docker-compose.yml:226`) resets to zero on every message received. For any active topic with regular traffic, this timeout never fires.
- `subscriberCount` in `TopicMessageServiceImpl` (lines 48, 89–90) is a Micrometer gauge for observability only — it enforces nothing.
- No authentication or authorization is required to call `subscribeTopic`.

### Impact Explanation
Each indefinite subscription holds: a Reactor subscription context, periodic DB polling via `TopicMessageRetriever` (every 2 s by default), a Redis listener slot (default listener type `REDIS`), and a bounded-elastic scheduler thread for the safety-check path. At scale, these resources are finite. Exhausting the DB connection pool (configured at the pgbouncer layer) or the bounded-elastic thread pool causes all new subscriptions — including those from legitimate clients — to queue or fail. Because the mirror node's gRPC service is the only path for HCS topic consumers to receive real-time consensus confirmations, a sustained resource exhaustion attack constitutes a denial of service against transaction confirmation visibility for the entire network of downstream consumers.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the public HCS gRPC API, and no expensive computation. The protobuf schema for `ConsensusTopicQuery` is publicly documented. A single attacker with a modest number of IP addresses (to avoid any upstream network-layer rate limiting) can open hundreds of connections and thousands of subscriptions. The attack is trivially repeatable and scriptable with any gRPC client library. The only friction is the per-connection call limit of 5, which merely requires proportionally more connections.

### Recommendation
1. **Enforce a server-side maximum subscription duration.** Add a `maxSubscriptionDuration` field (e.g., `Duration`, default `PT1H`) to `GrpcProperties`. In `ConsensusController.toFilter()` or `TopicMessageServiceImpl.subscribeTopic()`, cap the effective `endTime` to `now() + maxSubscriptionDuration` regardless of what the client supplies. Clients requiring longer subscriptions must reconnect.
2. **Set a global connection age limit.** In `GrpcConfiguration.grpcServerConfigurer()`, call `serverBuilder.maxConnectionAge(duration, unit)` and `serverBuilder.maxConnectionAgeGrace(duration, unit)` to periodically recycle connections and release held resources.
3. **Add a global concurrent-subscription cap.** Use `subscriberCount` (already tracked) to reject new subscriptions above a configurable threshold, returning `RESOURCE_EXHAUSTED`.
4. **Add per-IP connection rate limiting** at the nginx/gateway layer for the `subscribeTopic` endpoint.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_NUM = 1234  # any valid topic

def open_subscriptions(conn_id):
    creds = grpc.ssl_channel_credentials()
    channel = grpc.secure_channel(TARGET, creds)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = TOPIC_NUM
    # No consensusEndTime, no limit — subscription runs forever
    for _ in range(5):  # saturate maxConcurrentCallsPerConnection
        threading.Thread(
            target=lambda: list(stub.subscribeTopic(query)),
            daemon=True
        ).start()

# Open hundreds of connections from multiple IPs/processes
for i in range(200):
    threading.Thread(target=open_subscriptions, args=(i,), daemon=True).start()

# Server-side subscriberCount metric will climb to 1000+
# DB polling threads and Redis listener slots exhausted
# New legitimate subscriptions receive UNAVAILABLE or hang indefinitely
import time; time.sleep(86400)

public class TopicMessageFilter {

    private static final SecureRandom RANDOM = new SecureRandom();

    private Long endTime;

    @Min(0)
    private long limit;

    @Min(0)
    @NotNull
    @Builder.Default
    private long startTime = DomainUtils.now();

    @Builder.Default
    private String subscriberId = RandomStringUtils.random(8, 0, 0, true, true, null, RANDOM);

    @NotNull
    private EntityId topicId;

    public boolean hasLimit() {
        return limit > 0;
    }

    @AssertTrue(message = "End time must be after start time")
    public boolean isValidEndTime() {
        return endTime == null || endTime > startTime;
    }
