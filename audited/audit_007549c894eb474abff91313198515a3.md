### Title
Cache-Bypass DoS via Unique Contract IDs Exhausts DB Connection Pool

### Summary
The contract routes in `rest/server.js` accept arbitrary `contractId` path parameters, and the response cache key is derived solely from the full request URL. An unauthenticated attacker can flood the API with requests using unique contract IDs, each generating a distinct cache key that always misses the cache, forcing a live DB query per request. With no rate limiting in place, this exhausts the finite connection pool and starves legitimate transaction-confirmation queries.

### Finding Description

**Cache key generation** — `rest/middleware/responseCacheHandler.js` lines 151–153: [1](#0-0) 

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

Every unique URL produces a unique cache key. A request to `/api/v1/contracts/0.0.1` and `/api/v1/contracts/0.0.2` are two separate keys — both will be cache misses on first access.

**Contract routes registered without any rate-limiting guard** — `rest/server.js` line 109: [2](#0-1) 

The middleware stack in `server.js` is: `requestLogger → authHandler → (optional metricsHandler) → (optional responseCacheCheckHandler) → route handlers`. There is no rate-limiting middleware anywhere in `rest/middleware/`:


**Auth is entirely optional for unauthenticated callers** — `rest/middleware/authHandler.js` lines 16–20: [3](#0-2) 

If no `Authorization` header is sent, `authHandler` returns immediately without blocking the request. No credentials are required to reach the contract routes.

**DB pool is finite and shared** — `rest/dbpool.js` lines 7–16: [4](#0-3) 

`max: config.db.pool.maxConnections` caps the total concurrent DB connections. Every cache-miss contract request acquires one connection for the duration of its query. When the pool is saturated, subsequent requests queue and wait up to `connectionTimeoutMillis` before failing — blocking all other queries, including those serving transaction-status lookups.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/0.0.N` in a tight loop, incrementing `N` each time (or randomizing it).
2. Each request misses the cache (unique URL → unique MD5 key, never populated before).
3. Each cache miss proceeds to the route handler, which issues a DB query and holds a pool connection.
4. With enough concurrent requests (≥ `maxConnections`), the pool is fully occupied.
5. Legitimate requests — including `/api/v1/transactions/:id` queries used by clients to confirm transaction finality — queue behind the flood and time out.

### Impact Explanation
The DB connection pool is the single shared resource for all REST API queries. Full pool exhaustion causes all endpoints — not just contracts — to become unresponsive. Clients relying on the mirror node to confirm transaction status receive timeouts or errors, effectively making transaction confirmation unavailable for the duration of the attack. The `statement_timeout` limits individual query duration but does not prevent pool saturation when the request rate exceeds the pool drain rate.

### Likelihood Explanation
No privileges, tokens, or special knowledge are required. The contract ID namespace (`0.0.1` through `0.0.9999999999`) is effectively unbounded, so the attacker never runs out of unique cache-busting IDs. The attack is trivially scriptable with `curl`, `ab`, or any HTTP load tool. It is repeatable indefinitely and requires only network access to the public REST port.

### Recommendation
1. **Add rate limiting middleware** (e.g., `express-rate-limit`) applied globally before route handlers, keyed on client IP, with a burst cap low enough to prevent pool saturation.
2. **Normalize cache keys** to strip or canonicalize unknown/non-existent entity IDs before cache lookup, so repeated misses for non-existent contracts do not each generate a unique key.
3. **Reserve a pool partition** for internal/health queries so that even under pool pressure, transaction-status queries can still be served.
4. **Set a hard concurrency limit per IP** at the reverse-proxy/ingress layer (e.g., nginx `limit_conn`) as a defense-in-depth measure independent of the application layer.

### Proof of Concept
```bash
# Flood with unique contract IDs — no credentials needed
for i in $(seq 1 10000); do
  curl -s "http://<mirror-node-host>/api/v1/contracts/0.0.$i" &
done
wait

# Observe: legitimate transaction queries now time out
curl -v "http://<mirror-node-host>/api/v1/transactions/0.0.1234-1234567890-000000000"
# Expected result: connection timeout or pool exhaustion error
```
