### Title
Cache-Eviction DoS via Block Index Cycling Causes Unbounded DB I/O on `record_file` Table

### Summary
The `CACHE_MANAGER_RECORD_FILE_INDEX` Caffeine LRU cache is capped at 10,000 entries. An unauthenticated attacker can cycle POST `/api/v1/contracts/call` requests through more than 10,000 distinct valid block indices, continuously evicting cached entries and forcing a database query against the `record_file` table for every request. The global rate limiter (500 req/s, not per-IP) is the only guard and is itself exploitable as a single-attacker resource.

### Finding Description

**Code path:**

1. `POST /api/v1/contracts/call` → `ContractController.call()` → `throttleManager.throttle(request)` (global, 500 req/s) → `contractExecutionService.processCall(params)`.
2. Inside `ContractCallService.callContract()`, a per-request memoized supplier is set: `recordFileService.findByBlockType(params.getBlock())`.
3. `RecordFileServiceImpl.findByBlockType()` calls `recordFileRepository.findByIndex(block.number())` for any numeric block.
4. `findByIndex` is Spring-`@Cacheable` with `CACHE_MANAGER_RECORD_FILE_INDEX` (Caffeine, `maximumSize=10000`, `expireAfterWrite=10m`). The cache key is the block index `long`.

**Root cause:**

Hedera mainnet has millions of record files (block indices). The cache holds only 10,000 entries. Once an attacker has sent requests for 10,001 distinct valid block indices, the LRU policy evicts the oldest entry. Cycling through a working set of >10,000 valid indices (e.g., indices 1–20,000 in round-robin) guarantees a 100% cache miss rate, so every request issues `SELECT r FROM RecordFile r WHERE r.index = ?1` against the database.

**Why the `unless = "#result == null"` condition does not help:**

For valid existing indices, results are non-null and are cached — but only until evicted. The eviction is attacker-controlled by the size of the cycling window.

**Why the rate limiter is insufficient:**

The `rateLimitBucket` is a single global token bucket (500 tokens/s). It is not per-IP, per-user, or per-session. A single attacker consuming all 500 req/s simultaneously starves legitimate users and drives 500 uncached DB queries/second to `record_file`.

### Impact Explanation

At 500 req/s with 100% cache miss rate, the database receives 500 `SELECT … WHERE index = ?` queries per second against `record_file` continuously. On a production node where baseline DB I/O is moderate (e.g., a few hundred queries/second across all tables), this sustained load easily exceeds a 30% increase in DB I/O. The attack is self-sustaining: the attacker needs only to keep the cycling window larger than 10,000 and stay within the 500 req/s global budget. Because the throttle is global, the attacker simultaneously degrades service for all other users.

### Likelihood Explanation

- **No authentication required.** Any external caller can POST to `/api/v1/contracts/call`.
- **Block indices are sequential and publicly known** from the REST API (`/api/v1/blocks`).
- **Minimal gas required.** With `gas=21000` (minimum), `scaleGas(21000) = Math.floorDiv(21000, 10000) = 2` tokens consumed from the gas bucket (capacity 750,000 scaled tokens/s), so the gas throttle is not the binding constraint.
- **Single machine sufficient.** 500 HTTP requests/second is trivially achievable from one host.
- **Repeatable indefinitely.** No cooldown, no IP ban, no per-client quota.

### Recommendation

1. **Add per-IP (or per-client) rate limiting** in addition to the global bucket, so a single source cannot consume the entire 500 req/s budget.
2. **Increase `maximumSize`** of `CACHE_MANAGER_RECORD_FILE_INDEX` to cover the realistic working set of historical block queries, or use a time-based eviction strategy that better matches access patterns.
3. **Cache null results** (remove `unless = "#result == null"`) with a short TTL to prevent repeated DB hits for non-existent block numbers (a related but distinct amplification vector).
4. **Validate block number range** at the controller layer: reject block numbers outside `[earliest_index, latest_index]` before reaching the service layer, short-circuiting the DB query entirely.
