### Title
N+1 Database Query DoS via Multiple CREATE2 EVM Address `contract_id` Filters in `getContracts()`

### Summary
The `getContracts()` handler passes all `contract_id` query filters through `extractSqlFromContractFilters()`, which sequentially `await`s `ContractService.computeContractIdFromString()` for each filter value. When a filter value is a CREATE2 EVM address, this triggers a separate database lookup per value via `getContractIdByEvmAddress()`. An unauthenticated attacker can supply up to 100 `contract_id` filters (the default `maxRepeatedQueryParameters` limit) each containing a distinct CREATE2 EVM address, causing 100 sequential database queries per single HTTP request and enabling connection pool exhaustion.

**Root cause:** The `for...of` loop with sequential `await` means N `contract_id` filters with CREATE2 EVM addresses produce exactly N blocking database round-trips before the handler can respond. There is no batching, no parallelism, and no caching.

**Why the existing check fails:** The `maxRepeatedQueryParameters` guard (default: **100**) is enforced in `requestHandler.js` via `qs`'s `arrayLimit` and in `buildFilters`: [5](#0-4) 

This check is intended to prevent URL abuse, but it actually defines the upper bound of the attack: it permits exactly 100 repeated `contract_id` values, meaning 100 DB queries per request is the allowed maximum. The check does not distinguish between numeric entity IDs (no DB lookup) and CREATE2 EVM addresses (one DB lookup each).

### Impact Explanation
The default DB connection pool size is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). [6](#0-5) 

A single request with 100 CREATE2 EVM address filters holds a DB connection for the duration of 100 sequential queries. With just 10–20 concurrent such requests, the entire connection pool is saturated. All other API consumers (including legitimate users) receive connection timeout errors (default `connectionTimeout = 20000ms`). The `statementTimeout` of 20 seconds per query means a single malicious request can occupy a connection for up to 2000 seconds in the worst case. This is a complete availability denial for the REST API with minimal attacker resources.

### Likelihood Explanation
- **No authentication required.** `GET /api/v1/contracts` is a fully public endpoint.
- **No rate limiting** is present in the codebase at the application layer.
- **Trivially repeatable.** The attacker needs only a valid 20-byte hex string per filter; these do not need to correspond to real contracts (a `NotFoundError` is thrown per address, but only after the DB query completes).
- **Scriptable.** A single `curl` or HTTP client loop suffices. The attacker can sustain the attack with a handful of concurrent connections.

### Recommendation
1. **Batch the EVM address lookups:** Collect all CREATE2 EVM address filter values first, then resolve them in a single SQL query using `WHERE evm_address = ANY($1)` before entering the filter-building loop.
2. **Separate resolution from loop:** Move all `computeContractIdFromString` calls out of the sequential `for` loop and use `Promise.all()` to parallelize them, reducing wall-clock time even before batching is implemented.
3. **Lower `maxRepeatedQueryParameters` for `contract_id`:** Apply a tighter per-key limit (e.g., 10) specifically for parameters that trigger DB lookups, separate from the global limit.
4. **Cache EVM address → entity ID resolutions:** A short-lived in-process or Redis cache keyed on the EVM address hex would eliminate repeated DB hits for the same address across requests.

### Proof of Concept

```bash
# Generate 100 distinct fake CREATE2 EVM addresses and send in one request
python3 -c "
import os, urllib.parse
params = '&'.join(
  'contract_id=' + os.urandom(20).hex()
  for _ in range(100)
)
print('GET /api/v1/contracts?' + params)
" | xargs -I{} curl -s "http://<mirror-node-host>:5551/{}"

# Repeat with ~10 concurrent processes to exhaust the DB pool:
for i in $(seq 1 10); do
  python3 -c "
import os, requests
params = {'contract_id': [os.urandom(20).hex() for _ in range(100)]}
r = requests.get('http://<mirror-node-host>:5551/api/v1/contracts', params=params)
print(r.status_code)
" &
done
wait
```

Each concurrent request triggers 100 sequential `SELECT` queries against the `entity` table on the `evm_address` column. With 10 concurrent attackers, all 10 DB pool connections are occupied, and legitimate requests begin timing out with `503` or connection pool exhaustion errors.
