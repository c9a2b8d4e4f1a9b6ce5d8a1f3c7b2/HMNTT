### Title
Unbounded Cache Flooding via Unauthenticated GraphQL Query Submission Leading to Heap Exhaustion and DoS

### Summary
The `CachedPreparsedDocumentProvider.getDocumentAsync()` caches parsed GraphQL documents keyed by the raw query string, with no per-IP or per-user rate limiting at the application layer. An unauthenticated attacker can submit 1000 distinct syntactically valid query strings (each up to the 10,000-character parser limit) to fill all `maximumSize=1000` cache slots with large parsed AST objects, consuming hundreds of megabytes of heap and causing sustained GC pressure or OOM on a pod with a 2048Mi memory limit.

### Finding Description

**Exact code locations:**

`CachedPreparsedDocumentProvider.java`, line 26: [1](#0-0) 

The cache key is `executionInput.getQuery()` — the raw query string — with no normalization, no per-client quota, and no size check on the key before insertion.

`CacheProperties.java`, line 15: [2](#0-1) 

`maximumSize=1000` with `expireAfterWrite=1h` means up to 1000 entries live in heap for one hour.

**Parser limits** are set in `GraphQlConfiguration.java`, lines 33–38: [3](#0-2) 

`maxCharacters(10000)` and `maxTokens(1000)` bound individual queries, but do **not** prevent 1000 distinct queries from each being cached. Queries that exceed parser limits throw `ParseCancelledException`, which causes Caffeine to discard that entry — so only parseable queries are cached. This means the attacker must stay within limits, but 10,000-character, 1000-token queries are still large.

**No rate limiting exists in the GraphQL module.** A search across all Java files under `graphql/src/main/java/` finds:


Only `LoggingFilter.java` (pure logging), `GraphQlConfiguration.java` (parser limits), `MetricsConfiguration.java`, and `CustomExceptionResolver.java` exist. There are no `*Security*`, `*Throttle*`, or `*RateLimit*` classes. The `LoggingFilter` passes all requests through unconditionally: [4](#0-3) 

The web3 module has `ThrottleManagerImpl` with bucket4j rate limiting, but **no equivalent exists for the GraphQL module**: [5](#0-4) 

**Root cause:** The cache is globally shared, keyed by raw query string, with no per-client admission control. The `maximumSize=1000` cap bounds entry count but not the total heap consumed by those entries.

**Memory math:**
- 1000 cache keys × 10,000 chars × 2 bytes (Java UTF-16) = **~20 MB** for keys alone
- Each `PreparsedDocumentEntry` holds a parsed `Document` AST. With up to 1000 tokens per query, each AST is a tree of Java objects. Conservative estimate: 200–800 bytes per token × 1000 tokens × 1000 entries = **200 MB–800 MB** for AST values
- Pod memory limit is 2048Mi with requests starting at 128Mi: [6](#0-5) 

Combined with JVM overhead and Spring Boot baseline (~300–500 MB), the cache alone can push the pod to or past its 2048Mi limit.

### Impact Explanation

An attacker filling all 1000 cache slots with maximum-size parsed queries can consume 200–800 MB of heap in the GraphQL JVM process. Combined with normal application memory usage, this can trigger:
1. Sustained full GC cycles, causing request latency spikes for all users
2. `OutOfMemoryError` and pod crash/restart
3. During the 1-hour `expireAfterWrite` window, the memory is not reclaimed even if the attacker stops

The Caffeine `maximumSize=1000` is the only in-process bound. It prevents unbounded growth but does not prevent the bounded-but-large footprint from being harmful. Caffeine's Window TinyLFU eviction is also approximate and can temporarily exceed `maximumSize` under high insertion rates.

### Likelihood Explanation

- **No authentication required**: The GraphQL endpoint (`/graphql/alpha`) is publicly accessible with no credentials
- **No rate limiting**: No application-level throttle; the Helm chart for graphql does not configure Traefik `rateLimit` middleware (unlike the rosetta chart which does)
- **Trivially scriptable**: A simple loop sending 1000 POST requests with distinct padded query strings fills the cache in seconds
- **Repeatable**: After 1 hour (expiry), the attacker can repeat to maintain pressure
- **Low attacker sophistication**: No exploit code needed, just HTTP POST requests

### Recommendation

1. **Add per-IP rate limiting** at the application layer (e.g., bucket4j as used in the web3 module) or at the ingress/Traefik layer for the GraphQL Helm chart, mirroring the rosetta chart's `rateLimit` middleware
2. **Bound cache key size**: Before inserting into the cache, reject or truncate query strings above a practical threshold (e.g., 1000 chars for typical legitimate queries)
3. **Reduce `maximumSize`**: Lower from 1000 to a value that reflects realistic distinct query patterns (e.g., 100–200), reducing worst-case heap consumption by 5–10×
4. **Reduce `expireAfterWrite`**: Shorten from 1h to reduce the window during which flooded entries occupy heap
5. **Consider per-client cache quotas** or use a cache that supports per-key size weighting (`Caffeine.weigher()`) to cap total byte consumption rather than entry count

### Proof of Concept

```bash
#!/bin/bash
# Fill all 1000 cache slots with distinct large queries
for i in $(seq 1 1000); do
  # Craft a syntactically valid query padded to ~9900 chars with a unique alias
  PADDING=$(python3 -c "print('#' * 9800)")
  QUERY="{ account${i}: account(input: {entityId: {shard: 0, realm: 0, num: ${i}}}) { balance } }"
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"${QUERY}\"}" &
done
wait
echo "Cache flooded with 1000 distinct entries"
# Monitor heap via /actuator/metrics/jvm.memory.used
curl http://<graphql-host>/actuator/metrics/jvm.memory.used
```

Each request uses a unique alias (`account1`, `account2`, …, `account1000`), producing 1000 distinct cache keys. All 1000 slots fill within seconds. The heap usage for the cache entries persists for 1 hour. Repeating the loop continuously causes constant cache churn, parse CPU load, and GC pressure.
