### Title
Unauthenticated Cache-Miss DB Flooding via `/accounts/{id}/tokens` with No Rate Limiting

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` issues a database query for every batch of token IDs not present in the in-memory `quickLru` cache. Because there is no rate limiting at the endpoint, service, or middleware level, an unauthenticated attacker can sustain a high request rate against `/accounts/{id}/tokens`, repeatedly bypassing the cache and forcing DB queries on every request. The cache itself can be kept cold by rotating across accounts whose associated tokens are not yet cached or by exhausting the bounded LRU cache size.

### Finding Description
**Exact code path:**

`rest/controllers/tokenController.js` → `getTokenRelationships` (line 66) → `TokenService.getTokenAccounts(query)` (`rest/service/tokenService.js`, line 97) → `this.getCachedTokens(tokenIds)` (line 104).

Inside `getCachedTokens` (lines 136–161):
- Each token ID from the `token_account` result set is checked against the `quickLru` `tokenCache` (line 140).
- Any ID not in the cache is pushed to `uncachedTokenIds` (line 144).
- If any uncached IDs exist, a DB query is unconditionally issued (line 152): `select decimals, freeze_status, kyc_status, token_id from token where token_id = any ($1)`.

**Root cause / failed assumption:**
The design assumes the LRU cache will absorb repeated lookups. However, the cache has a finite `maxSize` (`config.cache.token.maxSize`). An attacker who rotates across many accounts (each associated with distinct token IDs) can keep the cache cold or cause LRU eviction, ensuring every request triggers the secondary DB query. No rate limiting exists anywhere in the REST service for this endpoint — confirmed by a full search of `rest/**/*.js` returning zero matches for `rateLimit`, `rate_limit`, `express-rate`, or `limiter`.

**Why existing checks fail:**
- The `quickLru` cache is a pure in-process LRU with no TTL-based protection against deliberate eviction.
- The `limit` parameter bounds the result set per request (default 25), but does not prevent high request concurrency.
- No authentication or authorization is required to call `/accounts/{id}/tokens`.
- No connection-level or application-level rate limiter is present.

### Impact Explanation
Each HTTP request to `/accounts/{id}/tokens` with cache-miss token IDs causes two DB round-trips: one for `token_account` and one for the `token` table via `getCachedTokens`. At high concurrency (e.g., hundreds of requests/second from a single client or botnet), this doubles the effective DB query rate. Sustained flooding can exhaust the DB connection pool (`db.pool.maxConnections`), increase query latency for all users, and potentially cause service unavailability. No privilege is required.

### Likelihood Explanation
The endpoint is public and unauthenticated. The attack requires only an HTTP client capable of high concurrency. Keeping the cache cold is straightforward: use accounts with many distinct token associations, or rotate across many account IDs. The `quickLru` cache is process-local and non-persistent, so a restart or cache eviction resets all protection. This is trivially repeatable and automatable.

### Recommendation
1. **Add rate limiting middleware** (e.g., `express-rate-limit`) at the application router level, keyed by IP or API key, before token-related routes.
2. **Bound the LRU cache size appropriately** and consider adding a per-token-ID negative cache entry (with short TTL) for IDs that return no DB rows, to prevent repeated DB hits for non-existent tokens.
3. **Add a `statementTimeout`** (already configurable in `db.pool.statementTimeout`) and ensure it is set aggressively enough to shed load under flooding.
4. **Consider requiring authentication** for high-frequency endpoints or enforcing per-IP concurrency limits at the ingress/proxy layer.

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
