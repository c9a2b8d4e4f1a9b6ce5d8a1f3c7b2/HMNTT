### Title
Unauthenticated Request Flood on `/api/v1/accounts/:id/rewards` Exhausts DB Connection Pool (REST API DoS)

### Summary
The `listStakingRewardsByAccountId` handler in `rest/controllers/accountController.js` issues at least two synchronous database queries per request (`EntityService.isValidAccount` + `StakingRewardTransferService.getRewards`) with no rate limiting applied to unauthenticated callers. The REST API's default database connection pool is capped at 10 connections, and the Redis response cache is disabled by default. A flood of requests from any unprivileged external user will exhaust the pool, stall all REST API queries, and impose heavy read load on the shared PostgreSQL instance that the importer also uses for gossip-transaction ingestion.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 16 registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/controllers/accountController.js` lines 170–185 show the handler unconditionally issuing two DB round-trips per request: [2](#0-1) 

`rest/service/entityService.js` lines 60–63 show `isValidAccount` always executing `SELECT type FROM entity WHERE id = $1`: [3](#0-2) 

`rest/service/stakingRewardTransferService.js` lines 18–22 show `getRewards` always executing a second query against `staking_reward_transfer`: [4](#0-3) 

**Root cause — no rate limiting for unauthenticated users:**

`rest/middleware/authHandler.js` only sets a custom response limit for authenticated users; it does not reject or throttle unauthenticated requests at all: [5](#0-4) 

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is never wired into the REST API server: [6](#0-5) 

`rest/server.js` registers no rate-limiting middleware before the accounts routes: [7](#0-6) 

**Root cause — tiny default connection pool:**

`rest/dbpool.js` creates the pool with `max: config.db.pool.maxConnections`: [8](#0-7) 

The documented default is **10 connections**: [9](#0-8) 

**Root cause — response cache disabled by default and bypassable:**

`hiero.mirror.rest.cache.response.enabled` defaults to `false`: [10](#0-9) 

Even when enabled, the cache key is the MD5 of `req.originalUrl`: [11](#0-10) 

An attacker trivially bypasses it by varying the `timestamp` query parameter (e.g., `?timestamp=gte:0`, `?timestamp=gte:1`, …), each producing a distinct cache key while still hitting the DB.

### Impact Explanation

With 10 pooled connections and 2 DB queries per request, only 5 concurrent in-flight requests are needed to hold all connections. Additional requests queue until `connectionTimeoutMillis` (default 20 000 ms) expires, returning 503 errors to all REST API consumers. Beyond pool exhaustion, the flood of `SELECT` queries against the `entity` and `staking_reward_transfer` tables imposes I/O and CPU load on the shared PostgreSQL instance. Because the importer (gossip-transaction processor) connects to the same PostgreSQL server, elevated DB server load can increase query latency for the importer, indirectly delaying gossip-transaction ingestion. The REST API becomes fully unavailable for the duration of the attack.

### Likelihood Explanation

No authentication, API key, CAPTCHA, or network-level credential is required. Any internet-accessible mirror node deployment is reachable. The attack requires only a standard HTTP client and a valid (or even invalid — the `isValidAccount` query still fires after `getEncodedId` resolves) account identifier. The attack is trivially scriptable, repeatable, and can be sustained indefinitely. Varying the `timestamp` parameter defeats the optional Redis cache. The attacker does not need to know any secret or hold any on-chain asset.

### Recommendation

1. **Add per-IP rate limiting** in the REST API middleware stack (e.g., `express-rate-limit` or an ingress-level policy) before the accounts router, targeting the `/rewards` path specifically given its two-query cost.
2. **Enable the Redis response cache** by default and ensure the cache key normalizer is applied so that semantically equivalent requests (same account, same effective time range) share a cache entry.
3. **Increase the default DB pool size** or add a request-concurrency cap (e.g., `express-async-semaphore`) so that pool exhaustion cannot be triggered by a small number of concurrent requests.
4. **Short-circuit on numeric account IDs**: when the path parameter is a plain numeric ID, `getEncodedId` already returns without a DB query; `isValidAccount` could be merged into the `getRewards` query (return 404 if zero rows and account does not exist) to reduce the per-request DB cost from 2 queries to 1.

### Proof of Concept

```bash
# Flood with varying timestamps to defeat the Redis cache
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/rewards?timestamp=gte:$i" &
done
wait
```

**Preconditions:** No credentials required. Mirror node must be network-accessible.

**Trigger:** Each request causes `EntityService.isValidAccount` (1 DB query) + `StakingRewardTransferService.getRewards` (1 DB query) to execute. With 500 concurrent requests and a pool of 10, 490 requests queue immediately.

**Result:** REST API returns HTTP 503 / connection-timeout errors for all endpoints. PostgreSQL CPU/IO rises, increasing latency for the importer's gossip-transaction writes.
