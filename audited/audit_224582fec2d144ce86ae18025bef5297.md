### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to `/api/v1/network/registered-nodes`

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module has no application-level rate limiting. Any unauthenticated external user can flood this endpoint with concurrent requests, exhausting the HikariCP database connection pool shared by the service, causing all subsequent requests (including legitimate node discovery queries) to time out or fail. The only rate-limiting control present is an optional infrastructure-level GCP gateway policy that is not enforced at the application layer.

### Finding Description

**Exact code path:**

`NetworkController.getRegisteredNodes()` (lines 173–186) receives the unauthenticated GET request and delegates to `NetworkServiceImpl.getRegisteredNodes()` (lines 139–151), which calls `RegisteredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` (lines 21–22). Each invocation acquires a HikariCP database connection for the duration of the query. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — no application-level rate limiting in rest-java:**

The `web3` module has a full `ThrottleManager`/`ThrottleConfiguration` (bucket4j) that enforces per-second request limits. The `rest-java` module has no equivalent. A search of all filters and interceptors in `rest-java/src/main/java/` reveals only `MetricsFilter` and `LoggingFilter` — neither enforces rate limits. [4](#0-3) 

**Existing check — GCP gateway `maxRatePerEndpoint: 250` — is insufficient:**

The Helm chart sets `maxRatePerEndpoint: 250` under `gateway.gcp.backendPolicy`, but this is gated on `gateway.gcp.enabled: true` and is an infrastructure-level control. It is not enforced at the application layer, can be absent in non-GCP or non-gateway deployments, and does not protect against distributed attacks from many source IPs that individually stay under the per-endpoint threshold. [5](#0-4) 

**Connection pool:**

The `rest-java` module has no explicit HikariCP configuration in its resources (only `banner.txt` exists under `rest-java/src/main/resources/`), meaning it relies on Spring Boot's default pool size of **10 connections**. With thousands of concurrent requests each holding a connection for the duration of the DB query, the pool is trivially exhausted.

### Impact Explanation
Once the HikariCP pool is exhausted, all threads waiting for a connection will block until the configured `connectionTimeout` (default 30 s) elapses and throw an exception. This renders the `/api/v1/network/registered-nodes` endpoint completely unresponsive for the duration of the attack. Registered nodes (block nodes, mirror nodes, RPC relays) that rely on this endpoint for peer discovery cannot obtain up-to-date endpoint information, satisfying the ≥30% network processing node disruption threshold described in the scope.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no brute force. A single attacker with a modest botnet or even a single machine using async HTTP clients (e.g., `wrk`, `ab`, `hey`) can sustain thousands of concurrent connections. The endpoint is publicly routable, documented in the OpenAPI spec, and requires only a valid HTTP GET. The attack is repeatable and can be sustained indefinitely. [6](#0-5) 

### Recommendation
1. **Add application-level rate limiting to `rest-java`**: Implement a bucket4j-based `ThrottleFilter` (mirroring the `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl`) applied to all endpoints in `rest-java`, including `/api/v1/network/registered-nodes`. Configure a global requests-per-second limit and optionally a per-IP limit.
2. **Explicitly configure HikariCP pool size**: Set `spring.datasource.hikari.maximum-pool-size` to a value appropriate for the expected concurrency, and set `connection-timeout` to a low value (e.g., 5 s) to fail fast rather than queue indefinitely.
3. **Do not rely solely on infrastructure controls**: The GCP gateway rate limit should be a secondary defense, not the primary one.

### Proof of Concept
```bash
# Install: apt install apache2-utils
# Send 5000 concurrent requests with 500 concurrency level
ab -n 5000 -c 500 \
  'https://<mirror-node-host>/api/v1/network/registered-nodes'

# Observe: after pool exhaustion (~10 in-flight DB queries),
# all subsequent requests return 500 / connection timeout errors.
# Legitimate registered nodes querying the endpoint receive no response.

# Alternatively, using hey:
hey -n 10000 -c 1000 \
  'https://<mirror-node-host>/api/v1/network/registered-nodes'
```

```java
    @Override
    public Collection<RegisteredNode> getRegisteredNodes(RegisteredNodesRequest request) {
        final var sort = Sort.by(request.getOrder(), REGISTERED_NODE.REGISTERED_NODE_ID.getName());

                lowerBound, upperBound, nodeTypeId, page);
```

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
