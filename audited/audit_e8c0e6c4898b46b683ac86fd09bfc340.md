### Title
Missing Rate Limiting on `getHookStorage()` Enables Database Query Saturation DoS

### Summary
The `getHookStorage()` endpoint in `HooksController` accepts up to 100 `key=eq:N` parameters per request and passes them directly to a database `IN`-clause query. The `rest-java` module contains no rate-limiting infrastructure whatsoever, unlike the `web3` module which uses `bucket4j`-backed `ThrottleManagerImpl`. An unprivileged attacker can flood the endpoint with concurrent maximum-parameter requests, saturating the database connection pool and denying service to legitimate users.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, `getHookStorage()`, lines 104–130:

```java
@GetMapping("/{hookId}/storage")
ResponseEntity<HooksStorageResponse> getHookStorage(
        ...
        @RequestParam(name = KEY, required = false, defaultValue = "") @Size(max = MAX_REPEATED_QUERY_PARAMETERS)
                List<SlotRangeParameter> keys,
        ...
```

`MAX_REPEATED_QUERY_PARAMETERS = 100` (defined in `Constants.java` line 36). This annotation is an *input-size validation*, not a rate limit — it permits up to 100 `key=eq:N` parameters per request.

When all 100 keys use the `EQ` operator, `hookStorageChangeRequest()` (lines 158–198) populates `keyFilters` with all 100 byte-array values. `HookServiceImpl.getHookStorage()` (lines 56–82) then issues:

```java
hookStorageRepository.findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse(
        ownerId.getId(), request.getHookId(), keysInRange, page);
```

This is a `SELECT … WHERE key IN (100 values)` query per request.

**Root cause — no rate limiting in `rest-java`:**

The `rest-java/src/main/java/org/hiero/mirror/restjava/config/` directory contains only: `JacksonConfiguration`, `LoggingFilter`, `MetricsConfiguration`, `MetricsFilter`, `NetworkProperties`, `RestJavaConfiguration`, `RuntimeHintsConfiguration`, `WebMvcConfiguration`. None implement rate limiting. A `grep` for `RateLimiter|throttle|bucket4j|RateLimit` across all `rest-java/**` source files returns zero hits in production code. The `ThrottleConfiguration`/`ThrottleManagerImpl`/`bucket4j` stack exists exclusively in the `web3` module.

### Impact Explanation
Each concurrent request with 100 `key=eq:N` parameters issues one expensive parameterized DB query. With N concurrent connections (trivially achievable with `curl --parallel` or any HTTP load tool), the attacker exhausts the JDBC connection pool, causing all subsequent legitimate queries — including unrelated endpoints — to queue or fail. Because the endpoint is unauthenticated (no `@PreAuthorize` or security filter visible in the controller or `rest-java` config), no credential is required. Severity: **High** (availability impact, no authentication barrier).

### Likelihood Explanation
Any internet-accessible deployment is trivially exploitable. The attacker needs only an HTTP client capable of concurrent requests. The maximum query cost per request is deterministic and predictable (100-item IN clause), making it easy to tune the attack for maximum DB saturation. No special knowledge of the system is required beyond the public API path `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage`.

### Recommendation
1. Add a `bucket4j`-backed rate-limiting filter to the `rest-java` module mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` and `ThrottleManagerImpl.java`, applied globally or specifically to the hooks endpoints.
2. Enforce a per-IP or per-client-token request rate (e.g., configurable `requestsPerSecond` property analogous to `hiero.mirror.web3.throttle.requestsPerSecond`).
3. Consider reducing `MAX_REPEATED_QUERY_PARAMETERS` for the `key` parameter specifically, or adding a separate, lower cap for EQ-operator keys to bound the IN-clause size.
4. Add a DB-level `statement_timeout` for the `rest-java` datasource to bound worst-case query duration.

### Proof of Concept
```bash
# Generate a request with 100 key=eq:N parameters
PARAMS=$(python3 -c "
import urllib.parse
keys = '&'.join(f'key=eq:{hex(i)[2:].zfill(64)}' for i in range(100))
print(keys)
")

# Send 200 concurrent requests
for i in $(seq 1 200); do
  curl -s "http://<host>/api/v1/accounts/0.0.1234/hooks/1/storage?${PARAMS}" &
done
wait
```

Each of the 200 concurrent requests triggers a `SELECT … WHERE key IN (100 values)` query. With no rate limiting in `rest-java`, all 200 queries hit the database simultaneously, exhausting the connection pool and causing HTTP 500 / timeout responses for all other users.
