### Title
Unbounded Prometheus Label Cardinality via Unmatched Route Paths in MetricsMiddleware Causes OOM DoS

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` instruments all four Prometheus metrics (`requestBytesHistogram`, `requestDurationHistogram`, `requestInflightGauge`, `responseBytesHistogram`) with a `route` label whose value is derived from the weaveworks `Instrument` middleware's `RouteMatcher`. When an incoming request does not match any registered gorilla/mux route, the weaveworks `Instrument` middleware falls back to using the raw `r.RequestURI` as the `route` label value. Because `r.RequestURI` is fully attacker-controlled and unbounded, an unprivileged external user can create an unbounded number of unique Prometheus time series by sending requests to unique paths, exhausting heap memory and triggering an OOM kill of the rosetta process.

### Finding Description
**Exact code location:** `rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83.

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,   // labeled by {"method", "route"}
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

**Root cause:** The `github.com/weaveworks/common/middleware` `Instrument.Wrap` implementation resolves the `route` label by calling `RouteMatcher.Match(r, routeMatch)` (gorilla/mux). When the request path does not match any registered route, `Match` returns `false` and the weaveworks middleware falls back to `r.RequestURI` as the route label. Since `r.RequestURI` is the raw, unvalidated, attacker-supplied URL path, every unique path the attacker sends creates a new label combination in the Prometheus registry.

**Failed assumption:** The code assumes that the `RouteMatcher` will always return a bounded set of route templates. It does not account for the fallback behavior of the weaveworks library for unmatched routes, and there is no application-level normalization, capping, or rejection of unknown route labels.

**Exploit flow:**
1. Attacker sends HTTP requests to the rosetta port with unique, never-before-seen paths: `GET /aaaa`, `GET /aaab`, `GET /aaac`, …
2. For each request, gorilla/mux `Match` returns `false` (no registered route).
3. The weaveworks `Instrument` middleware uses `r.RequestURI` (e.g., `/aaaa`) as the `route` label.
4. The Prometheus client library allocates a new `*histogram` (with 5 buckets + sum + count = 7 counters) for each unique `{method, route}` pair across all four metric vectors.
5. After N unique paths, the process holds N × 4 × 7 counter objects in heap memory, growing without bound.
6. The Go runtime OOM-kills the rosetta process.

**Why existing checks are insufficient:**
- The Traefik ingress middleware (`rateLimit: average: 10`, `inFlightReq: amount: 5`) is optional infrastructure-level configuration that is only active when `global.middleware: true` is set in the Helm chart. It is not enforced at the application layer and is absent in direct-access or non-Kubernetes deployments.
- There is no application-level rate limiting, path length cap, or label cardinality guard in the Go code.
- The Prometheus client library has no built-in cardinality limit; it allocates indefinitely.

### Impact Explanation
An attacker can crash the rosetta process (OOM kill) without any authentication, causing complete denial of service for all Rosetta API consumers. Since the rosetta node is a critical infrastructure component for blockchain interaction (account queries, block queries, construction), its unavailability directly impacts network participants. The memory growth is permanent across the process lifetime — there is no eviction — so even a slow, low-rate attack accumulates unbounded state.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no complex payload — only the ability to send HTTP GET/POST requests to the rosetta port with unique path strings. The attack is trivially scriptable (`for i in $(seq 1 100000); do curl http://rosetta:8080/x$i & done`) and repeatable. The Traefik rate limit of 10 req/s per host still allows 864,000 unique label registrations per day, each permanently consuming memory. The attack is realistic against any deployment where the rosetta port is reachable, including internal network attackers or cases where the Traefik middleware is not deployed.

### Recommendation
1. **Normalize unmatched routes to a fixed label:** Wrap the `RouteMatcher` to return a fixed string (e.g., `"unmatched"`) when no route matches, preventing raw URL paths from becoming label values.
2. **Use `prometheus.MustCurryWith` or a label allowlist:** Pre-register only the known finite set of route label values and drop or aggregate any observation that falls outside this set.
3. **Apply `MaxLabelValueLength` or a cardinality-limiting registry wrapper** (e.g., `github.com/prometheus/client_golang/prometheus` `WrapRegistererWithPrefix` combined with a custom collector that caps cardinality).
4. **Enforce application-level rate limiting** independent of the Traefik layer, so it applies in all deployment configurations.

### Proof of Concept
```bash
# Send 50,000 requests with unique paths to the rosetta port
# No authentication required
for i in $(seq 1 50000); do
  curl -s -o /dev/null "http://<rosetta-host>:8080/unique-path-$i"
done

# Monitor rosetta process memory growth
watch -n1 'ps aux | grep rosetta'

# After sufficient unique paths, the process is OOM-killed
# Verify: rosetta becomes unresponsive to legitimate API calls
curl http://<rosetta-host>:8080/network/list
# Connection refused or timeout
```

Each unique path `/unique-path-$i` registers a new `{method="GET", route="/unique-path-$i"}` label set across all four Prometheus metric vectors, permanently consuming heap memory until the process is killed by the OOM killer.
