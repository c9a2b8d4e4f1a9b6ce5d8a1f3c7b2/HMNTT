### Title
Unauthenticated DoS via DB Connection Pool Exhaustion on Unrate-Limited Airdrop Endpoints

### Summary
The `/api/v1/accounts/{id}/airdrops/outstanding` and `/api/v1/accounts/{id}/airdrops/pending` endpoints in the `rest-java` module perform synchronous, blocking DB queries on every request with no rate limiting. An unprivileged attacker can flood these endpoints with concurrent requests, exhausting the HikariCP connection pool and causing mirror node REST-Java instances to queue indefinitely and become unresponsive.

### Finding Description

**Exact code path:**

`TokenAirdropsController` exposes two unauthenticated GET endpoints with no rate-limiting guard: [1](#0-0) 

Each request flows to `TokenAirdropRepositoryCustomImpl.findAll()`, which executes a synchronous, blocking DB query via `dslContext.selectFrom(TOKEN_AIRDROP)`: [2](#0-1) 

**Root cause — rate limiting exists only in `web3`, not `rest-java`:**

The only throttle infrastructure in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) is scoped exclusively to the `web3` module: [3](#0-2) 

The `rest-java` module has no equivalent. Its filter chain contains only `MetricsFilter` and `LoggingFilter`: [4](#0-3) 

The `rest-java` module has no `application.yml` (only `banner.txt` in resources), so HikariCP runs at Spring Boot defaults: `maximumPoolSize=10`, `connectionTimeout=30s`. The `CommonConfiguration` wires HikariCP from `spring.datasource.hikari` properties with no rest-java-specific override: [5](#0-4) 

**Why existing input checks are insufficient:**

- `@Max(MAX_LIMIT)` caps result rows at 100 — limits data returned per query, not concurrent query count
- `@Positive` and `@Size(max = 2)` validate parameter format only
- No authentication is required; any numeric account ID is accepted [6](#0-5) 

### Impact Explanation

With a default pool of 10 connections, 10 concurrent slow queries (e.g., against accounts with many pending airdrops) saturate the pool. All subsequent requests block waiting for a connection for up to 30 seconds. Tomcat's default 200-thread executor fills up, causing the JVM to queue or reject new requests entirely. Because each mirror node instance shares one pool, a sustained flood renders the entire `rest-java` service on each targeted instance unresponsive — meeting the ≥30% mirror node disruption threshold with a modest number of attacking threads.

### Likelihood Explanation

Preconditions are minimal: no credentials, no special tokens, no privileged access. Any valid account ID (e.g., `0.0.1`) suffices. The attack is trivially repeatable with standard HTTP load tools (`wrk`, `ab`, `hey`) from a single machine. The endpoints are publicly documented in the OpenAPI spec: [7](#0-6) 

### Recommendation

1. **Add rate limiting to `rest-java`**: Port the `bucket4j`-based `ThrottleConfiguration`/`ThrottleManagerImpl` pattern from `web3` into `rest-java`, applying it as a servlet filter or Spring MVC interceptor on all airdrop endpoints.
2. **Set explicit HikariCP bounds**: Configure `maximumPoolSize`, `connectionTimeout`, and `maximumLifetime` explicitly in rest-java's configuration rather than relying on defaults.
3. **Add a DB query timeout**: Set `statement_timeout` at the JDBC/jOOQ level so runaway queries are killed server-side, freeing pool connections faster.
4. **Consider per-IP throttling**: A global bucket limits total RPS but a per-IP bucket prevents a single attacker from consuming the entire budget.

### Proof of Concept

```bash
# Flood the outstanding endpoint with 500 concurrent connections
wrk -t 50 -c 500 -d 60s \
  "http://<mirror-node-host>/api/v1/accounts/0.0.1/airdrops/outstanding"

# Simultaneously monitor pool exhaustion (expect hikaricp_connections_pending to spike)
# and observe HTTP 500 / connection timeout errors after pool saturation
```

Expected result: after ~10 concurrent in-flight requests, all subsequent requests queue for 30 seconds then fail. The mirror node REST-Java process becomes unresponsive to all endpoints sharing the same HikariCP pool.

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
