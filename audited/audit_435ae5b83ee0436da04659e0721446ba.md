### Title
Unauthenticated Repeated Hook Storage Queries with Arbitrary hookId Enable Sustained Database Resource Exhaustion

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts any `hookId >= 0` (including `Long.MAX_VALUE`) with no upper-bound constraint, no existence pre-check, and no rate limiting in the `rest-java` module. Each request unconditionally issues a database query against the `hook_storage` table's primary-key index. An unprivileged attacker can flood this endpoint with arbitrary non-existent hookId values, generating sustained database index-scan load with no throttle mechanism to stop them.

### Finding Description

**Code path:**

`HooksController.java` line 107 — the `hookId` path variable is annotated only with `@Min(0)`, placing no upper bound on the accepted value: [1](#0-0) 

This value flows directly into `HookServiceImpl.getHookStorage()` at lines 65–69, which calls the repository without any existence check: [2](#0-1) 

The repository method `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` is a Spring Data JPA derived query that translates directly to:
```sql
SELECT * FROM hook_storage
WHERE owner_id = ? AND hook_id = ? AND key BETWEEN ? AND ? AND deleted = false
``` [3](#0-2) 

The `hook_storage` table's primary key is `(owner_id, hook_id, key)`: [4](#0-3) 

For a non-existent `hookId`, the database performs a B-tree index seek to `(owner_id, Long.MAX_VALUE)`, finds no rows, and returns. This is repeated for every request. Additionally, `entityService.lookup(request.getOwnerId())` is called first, adding a second DB query per request.

**Root cause — failed assumption:** The code assumes callers will only supply valid, existing hookIds. There is no guard: no `@Max` on the path variable, no pre-check that `(ownerId, hookId)` exists in the `hook` table before querying `hook_storage`, and no rate limiting on this endpoint in the `rest-java` module.

**Rate limiting gap:** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives entirely in the `web3` module and applies only to contract-call endpoints. The `rest-java` module has only a `MetricsFilter` and `LoggingFilter` — no request-rate enforcement for the hooks endpoints: [5](#0-4) 

### Impact Explanation
Each attacker request causes two database round-trips (entity lookup + hook_storage index scan). With no rate limiting, a single attacker with a standard HTTP client can sustain thousands of requests per second. Across many concurrent connections, this generates continuous database CPU consumption from repeated index traversals, connection pool pressure, and thread exhaustion in the application server — all returning empty results at zero cost to the attacker. This directly maps to the stated scope of increasing node resource consumption by ≥30% without brute-forcing credentials.

### Likelihood Explanation
The endpoint requires no authentication and no special privileges. Any external user who knows a valid `ownerId` (account IDs are public on the Hedera network) can immediately begin the attack. The attack is trivially scriptable with `curl` or any HTTP load tool. The absence of any rate-limiting layer in `rest-java` for this endpoint means there is no automated defense.

### Recommendation
1. **Add `@Max` to hookId**: Constrain `hookId` to a realistic maximum (e.g., the maximum valid Hedera entity number) using `@Max` alongside `@Min(0)` at line 107 of `HooksController.java`.
2. **Add existence pre-check**: Before querying `hook_storage`, verify that `(ownerId, hookId)` exists in the `hook` table and return 404 immediately if not. This short-circuits the storage query for non-existent hooks.
3. **Add rate limiting to rest-java hooks endpoints**: Implement a per-IP or global request-rate limiter (e.g., using Bucket4j, already a dependency in the `web3` module) for the `/api/v1/accounts/{ownerId}/hooks/**` routes.
4. **Add a database statement timeout**: Ensure a `statement_timeout` is configured for the `rest-java` database connection pool to bound the worst-case query duration.

### Proof of Concept
```bash
# Precondition: obtain any valid ownerId (e.g., 0.0.1234) from the public network

# Single request — succeeds, returns empty storage, causes 2 DB queries
curl "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/9223372036854775807/storage"
# HTTP 200, {"storage": [], ...}

# Sustained attack — no rate limiting stops this
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/$((RANDOM * RANDOM))/storage" &
done
# Generates sustained DB index-scan load with no throttle response
```

Each iteration causes the database to seek `(owner_id=1234, hook_id=<random large value>)` in the `hook_storage` primary-key index, find nothing, and return — with no server-side defense preventing repetition.
