### Title
Unconditional Secondary DB Query Per Page in `createStakingRewardTransferList()` Enables Amplified DB Load by Unprivileged Users

### Summary
`formatTransactionRows()` unconditionally calls `createStakingRewardTransferList()` for every page of transaction results. When any transaction in the page contains a transfer from the staking reward account (0.0.800), a second database query is issued against the `staking_reward_transfer` table. Because staking reward transfers are a normal, persistent network condition on mainnet, an unprivileged user can reliably trigger this secondary query on every single request to `/api/v1/transactions`, increasing DB query rate by ~33–50% per request and compressing the effective throughput of the REST API's small connection pool under concurrent load.

### Finding Description

**Code path:**

`rest/transactions.js`, lines 191–257:

```
formatTransactionRows(rows)                          // line 191
  └─ createStakingRewardTransferList(rows)           // line 192 / 253
       └─ getStakingRewardTimestamps(transactions)   // line 254 — in-memory filter
       └─ getStakingRewardTransferList(timestamps)   // line 255 — issues DB query
```

`getStakingRewardTimestamps()` (lines 234–243) scans each row's `crypto_transfer_list` for any entry whose `entity_id` equals the encoded ID of the staking reward account (0.0.800). If any match is found, those consensus timestamps are collected.

`getStakingRewardTransferList()` (lines 265–282) then issues a single SQL query:

```sql
SELECT consensus_timestamp, json_agg(...) AS staking_reward_transfers
FROM staking_reward_transfer
WHERE consensus_timestamp IN ($1, $2, ..., $N)
GROUP BY consensus_timestamp
```

**Root cause / failed assumption:**

The design assumes staking reward transfers are rare or exceptional. In practice, on a live network, staking rewards are paid on virtually every block. Any page of transactions that includes even one HBAR transfer involving account 0.0.800 triggers the secondary query. The only guard is:

```js
if (stakingRewardTimestamps.length === 0) {
  return [];
}
```

This guard is bypassed whenever any transaction in the page has a staking reward transfer — which is the normal, steady-state condition on mainnet.

**Exploit flow:**

1. Attacker (no credentials required) sends repeated GET requests to `/api/v1/transactions` (or with varying `timestamp=lt:X` pagination parameters to avoid cache hits).
2. Each request fetches a page of transactions. On mainnet, the page will contain transactions with staking reward transfers.
3. `formatTransactionRows` calls `createStakingRewardTransferList`, which issues the secondary DB query.
4. Each request now consumes two sequential DB connections from the pool (one for `getTransactionsDetails`, one for `getStakingRewardTransferList`) instead of one.
5. Under concurrent load, the pool of 10 connections (default `maxConnections: 10`, `docs/configuration.md` line 556) is exhausted at half the normal concurrency level.

**Why existing checks are insufficient:**

- The `stakingRewardTimestamps.length === 0` check only skips the query when zero transactions have staking rewards — not the normal case.
- The Redis response cache (`cache.get` at line 705) caches by `payerAndTimestamps` key. An attacker using different `timestamp=lt:X` values (pagination) generates distinct cache keys, bypassing the cache entirely.
- There is no rate limiting on the `/api/v1/transactions` endpoint for unauthenticated users beyond the default response limit.

### Impact Explanation

The REST API DB pool defaults to **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). Each request to `/api/v1/transactions` that triggers the secondary query holds a connection for the duration of both the main query and the staking reward query (sequential awaits). Under concurrent load, the effective maximum concurrency is reduced. At 10 concurrent requests, all pool connections are occupied; additional requests queue or time out (`connectionTimeoutMillis` default 20 000 ms). This degrades API availability for all users, including legitimate ones querying transaction status — which is the primary use case of the mirror node REST API.

### Likelihood Explanation

- No authentication or special privileges are required.
- The trigger condition (staking reward transfers in a page) is the normal steady-state on mainnet — no special data crafting is needed.
- The attacker only needs to send concurrent HTTP GET requests to a public endpoint.
- Pagination via `timestamp=lt:X` parameters generates unique cache keys, defeating the response cache.
- The attack is trivially repeatable and scriptable (e.g., `ab`, `wrk`, or a simple loop with `curl`).

### Recommendation

1. **Batch the staking reward lookup into the main query**: Join or LEFT JOIN `staking_reward_transfer` directly in `getTransactionsDetails()` using a CTE, eliminating the secondary round-trip entirely.
2. **Alternatively, cache the staking reward lookup result** separately with a short TTL keyed on the set of timestamps, so repeated pages with overlapping timestamps reuse the result.
3. **Increase the default connection pool size** or add per-IP rate limiting on the `/api/v1/transactions` endpoint to reduce the blast radius.
4. **Add a fast-path check** in `getStakingRewardTimestamps` that short-circuits if the `crypto_transfer_list` column is not populated in the result set at all (e.g., when `transactionType` filters exclude HBAR transfers).

### Proof of Concept

```bash
# Precondition: mainnet or testnet mirror node with staking rewards in DB
# No credentials required

# Step 1: Get the latest timestamp to start pagination
LATEST=$(curl -s "https://<mirror-node>/api/v1/transactions?limit=1&order=desc" \
  | jq -r '.transactions[0].consensus_timestamp')

# Step 2: Send N concurrent requests with distinct pagination timestamps
# Each request triggers the secondary staking reward DB query
for i in $(seq 1 50); do
  TS=$(echo "$LATEST - 0.0000000$i" | bc)
  curl -s "https://<mirror-node>/api/v1/transactions?limit=25&timestamp=lt:$TS" \
    -o /dev/null &
done
wait

# Result: DB connection pool saturated; subsequent requests queue or return
# 503/timeout errors. Monitor with:
# SELECT count(*), state FROM pg_stat_activity WHERE usename='mirror_api' GROUP BY state;
```

Each concurrent request issues 2–3 sequential DB queries instead of 1–2, compressing effective pool throughput. With a 10-connection pool and 50 concurrent requests, all connections are held and new requests time out after 20 seconds.
