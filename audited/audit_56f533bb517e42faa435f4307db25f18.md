### Title
Unauthenticated Cache Poisoning of Staking Rewards Endpoint via Empty-Array Response Caching

### Summary
`responseCacheUpdateHandler()` in `rest/middleware/responseCacheHandler.js` caches any non-empty-string response body with a 2xx status code, including a semantically empty `{"rewards":[],"links":{"next":null}}` JSON response. Because the cache key is derived solely from the request URL with no user identity component, any unauthenticated external user can poison the shared Redis cache for `/api/v1/accounts/<id>/rewards` by issuing a GET request before a staking reward is recorded, causing the empty response to be served to all subsequent callers — including the account owner — for up to 60 seconds.

### Finding Description

**Code path:**

`responseCacheUpdateHandler()` at `rest/middleware/responseCacheHandler.js` line 95:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```

`responseBody` is set by `responseHandler.js` line 50:
```js
res.locals[responseBodyLabel] = contentType === APPLICATION_JSON ? JSONStringify(responseData) : responseData;
```

When the staking rewards table has no entries for an account, `listStakingRewardsByAccountId()` (`rest/controllers/accountController.js` line 186-202) sets `res.locals[responseDataLabel] = { rewards: [], links: { next: null } }`. After JSON serialization this becomes the string `'{"rewards":[],"links":{"next":null}}'`.

**Root cause — failed assumption:**

The guard `responseBody &&` is a JavaScript truthy check. It correctly rejects an empty string `''` (the "Do not cache empty body" test at `rest/__tests__/middleware/responseCacheHandler.test.js` line 146-155 sets `responseBodyLabel = ''`). However, `'{"rewards":[],"links":{"next":null}}'` is a 36-character non-empty string and is fully truthy. There is no semantic check on whether the payload contains meaningful data.

**Cache key — no user identity:**

`cacheKeyGenerator()` at `rest/middleware/responseCacheHandler.js` line 151-153:
```js
return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
```
The key is purely URL-based. Any request to the same URL from any IP produces the same Redis key, so a poisoned entry is served to every caller including the legitimate account owner.

**TTL:**

The rewards endpoint sets `cache-control: public, max-age=60` (`rest/__tests__/specs/accounts/{id}/rewards/responseHeaders.json`), so each poisoned entry persists for 60 seconds.

**Exploit flow:**

1. Attacker sends `GET /api/v1/accounts/0.0.12345/rewards` at a moment when no staking reward has yet been ingested for that account (e.g., just before the daily reward distribution is processed by the mirror node).
2. The controller returns HTTP 200 with `{"rewards":[],"links":{"next":null}}`.
3. `responseCacheUpdateHandler` evaluates: `responseBody` = truthy string, `responseCacheKey` = set, `httpStatusCodes.isSuccess(200)` = true, TTL = 60 → stores the empty response in Redis.
4. The account owner queries the same URL within the 60-second window and receives the cached empty response from `responseCacheCheckHandler` without hitting the database.
5. The attacker can repeat step 1 continuously to keep the cache poisoned across multiple reward distribution cycles.

**Existing checks reviewed and shown insufficient:**

| Check | What it guards | Why it fails here |
|---|---|---|
| `responseBody &&` | Empty string `''` | `'{"rewards":[],...}'` is truthy |
| `httpStatusCodes.isSuccess()` | Non-2xx errors | 200 is a success code |
| `ttl > 0` | Zero max-age | TTL is 60 s |
| No auth on endpoint | N/A | Endpoint is fully public |

The integration test at `rest/__tests__/integration/responseCacheHandler.test.js` lines 174-193 explicitly confirms this behavior — it calls `warmUpCache()` with an empty staking rewards table and asserts the empty-array response is cached and served.

### Impact Explanation

Any unauthenticated user can cause the mirror node REST API to serve a stale `rewards: []` response for any account for up to 60 seconds per poisoning cycle. If the attacker continuously re-poisons the cache (one request per minute), the account owner's rewards view is permanently suppressed at the API layer. While the underlying on-chain rewards are not destroyed, applications, wallets, or dashboards that rely on this endpoint to display or act on reward data will show incorrect zero-balance information. This constitutes a targeted denial-of-information attack against specific accounts and undermines the integrity of the mirror node as a reliable data source.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no exploit tooling — only the ability to send an HTTP GET request. Hedera staking rewards are distributed on a predictable daily schedule, making the timing window easy to target. The attacker needs only to issue the request before the mirror node ingests the reward record for the target account. The attack is repeatable indefinitely with a single HTTP client and imposes no cost on the attacker.

### Recommendation

1. **Content-aware cache suppression:** In `responseCacheUpdateHandler`, parse the response body and skip caching if the primary data array (e.g., `rewards`) is empty, or add a flag set by the controller to suppress caching of empty-result responses.
2. **User-scoped or short-lived cache for mutable per-account data:** Either incorporate a user/session identifier into the cache key for account-specific endpoints, or reduce the TTL for the rewards endpoint to a value that limits the poisoning window.
3. **Cache invalidation on ingestion:** Invalidate the Redis key for `/api/v1/accounts/<id>/rewards` when a new staking reward transfer is ingested for that account.
4. **Minimum-content guard:** Add a test case that asserts `{"rewards":[],...}` is NOT cached (analogous to the existing "Do not cache empty body" test at `rest/__tests__/middleware/responseCacheHandler.test.js` line 146).

### Proof of Concept

const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
