### Title
Negative Cache Miss via `unless = "#result == null"` on Optional Return Type Enables Unbounded DB Enumeration in `findByEvmAddressAndDeletedIsFalse()`

### Summary
`findByEvmAddressAndDeletedIsFalse()` uses `unless = "#result == null"` on a method returning `Optional<Entity>`. Spring Cache unwraps `Optional` before evaluating `unless`, so `Optional.empty()` (non-existent entity) is treated as `null` and is never cached. Every request for a non-existent EVM address bypasses the cache and hits the database directly. The global rate limiter (default 500 req/s) is the only mitigation, but it is not per-IP and is shared across all callers.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:

```java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

**Root cause:** Spring Cache has special `Optional` handling (since Spring 4.3). When evaluating the `unless` SpEL expression, `#result` is the *unwrapped* value inside the `Optional`, not the `Optional` wrapper itself. For `Optional.empty()` (entity not found), the unwrapped value is `null`, so `#result == null` evaluates to `true` and the result is **never stored in cache**. This means every lookup for a non-existent EVM address unconditionally executes a database query.

**Call path to trigger:**
`ContractController.call()` → `ContractExecutionService` → `CommonEntityAccessor.getEntityByEvmAddressAndTimestamp()` (line 70) → `entityRepository.findByEvmAddressAndDeletedIsFalse(addressBytes)` — all reachable with a standard unauthenticated `POST /api/v1/contracts/call` with a crafted `to` field.

**Why existing checks are insufficient:**
- The global rate limiter (`rateLimitBucket`, default 500 req/s) is a token-bucket shared across all callers — it is not per-IP or per-user. A single attacker consuming the full 500 req/s budget causes 500 DB queries/second for non-existent addresses.
- The gas throttle (`gasLimitBucket`) can be minimized by sending requests with `gas: 21000` (minimum), which `scaleGas()` rounds to 0 tokens consumed (`≤ GAS_SCALE_FACTOR`), effectively bypassing the gas bucket entirely.
- The entity cache (`expireAfterWrite=1s, maximumSize=10000`) provides no protection for negative lookups. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every request for a non-existent EVM address causes a full DB query against the `entity` table. At the default 500 req/s rate limit, an attacker can sustain 500 DB queries/second indefinitely. Because the gas bucket is bypassable with `gas=21000`, the effective throttle is only the request-rate bucket. Additionally, the attacker receives a deterministic signal (entity found vs. not found) for each queried address, enabling systematic enumeration of which EVM addresses exist in the Hashgraph ledger — an information-disclosure oracle against the entity table. ### Likelihood Explanation
No authentication or special privilege is required. The endpoint is publicly accessible. The exploit requires only standard HTTP tooling (curl, wrk, etc.) and knowledge of the JSON-RPC `eth_call` interface. The gas bypass (`gas ≤ 21000`) is trivially discoverable. A single attacker can sustain the attack at the full rate-limit budget, and multiple coordinated attackers can exceed it if the rate limiter is not per-IP. [6](#0-5) [7](#0-6) 

### Recommendation
1. **Cache negative results:** Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` — but this is semantically wrong for Optional. The correct fix is to use `unless = "!#result.isPresent()"` only if short-lived negative caching is acceptable, or remove the `unless` condition entirely and rely on the TTL (`expireAfterWrite=1s`) to expire stale negative entries.
2. **Per-IP rate limiting:** Enforce per-source-IP request throttling at the ingress/load-balancer layer or within `ThrottleManagerImpl` to prevent a single client from consuming the entire global budget.
3. **Gas floor enforcement:** Ensure `scaleGas()` does not return 0 for the minimum valid gas value, so the gas bucket always consumes tokens. [1](#0-0) [2](#0-1)
