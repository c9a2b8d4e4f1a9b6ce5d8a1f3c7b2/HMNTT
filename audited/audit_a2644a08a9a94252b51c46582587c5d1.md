### Title
Unauthenticated DB Amplification via Uncached Aggregation Query on `/api/v1/network/supply`

### Summary
The `getSupply()` method in `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java` executes a full `SUM`/`MAX` aggregation join across the entire `entity` table on every invocation, with no `@Cacheable` annotation and no rate limiting anywhere in the rest-java module. Any unprivileged external user can flood `GET /api/v1/network/supply` with concurrent requests, causing each to independently execute this expensive query, multiplying database CPU and I/O load by the number of concurrent connections and raising node resource consumption well above 30%.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) → `NetworkServiceImpl.getSupply()` (lines 59–89) → `EntityRepository.getSupply()` (lines 19–29).

The repository query:
```sql
select cast(coalesce(sum(e.balance), 0) as bigint) as unreleased_supply,
       cast(coalesce(max(e.balance_timestamp), 0) as bigint) as consensus_timestamp
from entity e
join unnest(
        cast(string_to_array(:lowerBounds, ',') as bigint[]),
        cast(string_to_array(:upperBounds, ',') as bigint[])
     ) as ranges(min_val, max_val)
  on e.id between ranges.min_val and ranges.max_val
``` [1](#0-0) 

This is a full-table aggregation with a multi-range join. On mainnet the `entity` table contains millions of rows. There is no `@Cacheable` annotation on this method. [2](#0-1) 

`NetworkServiceImpl.getSupply()` calls it directly with no service-layer caching: [3](#0-2) 

**Contrast with web3:** The `web3` `EntityRepository` applies `@Caching`/`@Cacheable` on every lookup method, and the `web3` module has a full `ThrottleManager` (bucket4j, 500 req/s default) enforced at the controller level. [4](#0-3) [5](#0-4) 

The rest-java config directory contains only `JacksonConfiguration`, `LoggingFilter`, `MetricsFilter`, `MetricsConfiguration`, `NetworkProperties`, `RestJavaConfiguration`, `WebMvcConfiguration`, and `RuntimeHintsConfiguration` — none implement rate limiting. [6](#0-5) 

The endpoint is public and unauthenticated: [7](#0-6) 

**Root cause:** The failed assumption is that a public, read-only endpoint with a fixed query shape is cheap enough to serve without caching or rate limiting. In reality the query is an unbounded aggregation over a multi-million-row table, making it a high-cost operation that scales linearly with concurrent callers.

### Impact Explanation

With HikariCP's default pool of 10 connections, up to 10 copies of the aggregation query run simultaneously; additional requests queue on application threads consuming heap. On a production mirror node with millions of entities, each query performs a sequential scan and aggregation across the 7 configured account ranges (accounts 2–750), which is I/O and CPU intensive. Ten simultaneous such queries can saturate the database's CPU and I/O, raising resource consumption by 30%+ compared to baseline. The rest of the mirror node's workload (importer, other REST endpoints) shares the same PostgreSQL instance, so the impact is node-wide. [8](#0-7) 

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and a single publicly documented endpoint. A trivial script sending 50–100 concurrent HTTP GET requests to `/api/v1/network/supply` is sufficient. The attack is repeatable indefinitely; the only natural brake is the 10-second `statementTimeout` [9](#0-8) 
which limits individual query duration but does not prevent the next wave of requests from immediately re-queuing. No CAPTCHA, API key, or IP-based throttle exists in the rest-java module.

### Recommendation

1. **Add `@Cacheable`** on `EntityRepository.getSupply()` with a short TTL (e.g., 15–30 seconds), consistent with how the `web3` module caches entity lookups.
2. **Add rate limiting** to the rest-java module equivalent to the `web3` `ThrottleConfiguration`/`ThrottleManagerImpl` (bucket4j), applied as a servlet filter or interceptor on `/api/v1/network/*`.
3. **Alternatively**, cache the result at the service layer in `NetworkServiceImpl.getSupply()` using Spring's `@Cacheable` with a Caffeine or Redis-backed cache manager, so repeated identical calls within the TTL window return the cached value without hitting the database.

### Proof of Concept

```bash
# Send 50 concurrent requests to the supply endpoint
seq 50 | xargs -P50 -I{} curl -s -o /dev/null \
  "https://<mirror-node-host>/api/v1/network/supply"
```

**Preconditions:** None. No authentication or special headers required.

**Trigger:** Each of the 50 concurrent requests independently calls `EntityRepository.getSupply()`, executing the full aggregation join on the `entity` table.

**Observable result:** Database CPU and I/O spike proportionally to the number of concurrent connections. With a pool of 10, 10 queries run simultaneously; the remaining 40 threads block on pool acquisition, consuming JVM heap. Repeating the burst continuously sustains elevated DB load. Monitor with `pg_stat_activity` to confirm multiple simultaneous executions of the aggregation query, and with `pg_stat_bgwriter` / OS `iostat` to confirm I/O amplification exceeding 30% above baseline.     public NetworkSupply getSupply(Bound timestamp) {
        final NetworkSupply networkSupply;

        final var bounds = networkProperties.getUnreleasedSupplyRangeBounds();
        final var lowerBounds = bounds.lowerBounds();
        final var upperBounds = bounds.upperBounds();

        if (timestamp.isEmpty()) {
            networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
        } else {
            var minTimestamp = timestamp.getAdjustedLowerRangeValue();
            final var maxTimestamp = timestamp.adjustUpperBound();

            // Validate timestamp range
            if (minTimestamp > maxTimestamp) {
                throw new IllegalArgumentException("Invalid range provided for timestamp");
            }

            final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
            minTimestamp = Math.max(minTimestamp, optimalLowerBound);

            networkSupply =
                    accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp);
        }

        if (networkSupply.consensusTimestamp() == 0L) {
            throw new EntityNotFoundException("Network supply not found");
        }

        return networkSupply;
    }
