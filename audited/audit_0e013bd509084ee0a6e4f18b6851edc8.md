### Title
Cache Key Pollution via Unsorted `timestamp` Parameter Permutations

### Summary
`normalizeRequestQueryParams()` intentionally skips sorting `timestamp` query parameters (they are in `NON_SORTED_PARAMS` but not `COLLAPSABLE_PARAMS`), preserving their original client-supplied order. The Redis cache key generator hashes `req.originalUrl` directly — the raw, un-normalized URL — so every distinct ordering of N `timestamp` values produces a unique cache key. An unauthenticated attacker can submit up to 100 `timestamp` parameters (the configured `maxRepeatedQueryParameters` limit) in all N! orderings, flooding Redis with distinct entries for semantically equivalent requests and evicting legitimate cached responses.

### Finding Description

**Code path 1 — normalizer skips sorting for `timestamp`:**

`rest/middleware/requestNormalizer.js`, `getNormalizedArrayValue()`, lines 70–76:
```js
if (!NON_SORTED_PARAMS.includes(name)) {
    valueArray.sort();                          // skipped for timestamp
} else if (COLLAPSABLE_PARAMS.includes(name)) {
    valueArray = valueArray.slice(valueArray.length - 1); // also skipped for timestamp
}
return valueArray.join('&' + name + '=');       // order preserved as-sent
```
`timestamp` is in `NON_SORTED_PARAMS` (line 23) but absent from `COLLAPSABLE_PARAMS` (line 11), so neither branch executes. The values are emitted in the exact order the client supplied them.

**Code path 2 — cache key uses raw `originalUrl`, not normalized URL:**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, lines 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The comment at line 149 explicitly acknowledges the normalizer is **not yet wired in**: *"In the future, this will utilize Edwin's request normalizer (9113)."*

**Root cause:** The normalizer produces order-sensitive output for `timestamp`, and the cache key is derived from the raw URL rather than the normalized form. These two facts combine to make every permutation of `timestamp` values a distinct Redis key.

**Failed assumption:** The design assumes the normalizer will eventually be used as the cache key source (issue 9113), but until that integration is complete, the raw URL is used, making the normalizer's ordering behavior irrelevant to cache deduplication.

### Impact Explanation
Redis is configured with a default `maxMemory` of 250 MB and `allkeys-lfu` eviction. An attacker who continuously submits requests with shuffled `timestamp` arrays fills Redis with junk entries. Under LFU eviction, infrequently-accessed legitimate entries are evicted first, degrading cache hit rates for real users and forcing every subsequent legitimate request to hit the PostgreSQL database. With `maxRepeatedQueryParameters = 100`, an attacker can theoretically generate 100! unique keys from a single logical query, though even N=5–10 (120–3,628,800 permutations) is sufficient to saturate a 250 MB cache. The result is sustained elevated database load and degraded API response times — a griefing-class denial-of-service with no economic cost to the attacker.

### Likelihood Explanation
No authentication or rate limiting is required. Any HTTP client can issue GET requests to any timestamp-accepting endpoint (e.g., `/api/v1/transactions`, `/api/v1/contracts/results/logs`). The attack is trivially scriptable: generate all permutations of a small set of timestamp values and fire them in a loop. The `maxRepeatedQueryParameters` guard (100) prevents a single request from exceeding 100 `timestamp` values but does not limit the number of distinct requests. The attack is repeatable indefinitely and requires no special knowledge of the system.

### Recommendation
1. **Immediate fix:** In `cacheKeyGenerator()` (`rest/middleware/responseCacheHandler.js` line 152), replace `req.originalUrl` with the output of `normalizeRequestQueryParams()` as the hash input. This is the integration already planned under issue 9113.
2. **Complementary fix:** In `getNormalizedArrayValue()` (`rest/middleware/requestNormalizer.js` lines 70–76), sort `timestamp` values that carry the same operator prefix (e.g., all `gte:` values sorted among themselves) so that semantically identical multi-timestamp queries always produce the same normalized string. The existing comment documents that order matters for mixed-operator queries (e.g., `lt:X` before bare `Y`), so a full sort is not safe — but a stable canonical form (e.g., group by operator, sort within group) would eliminate most permutation variants.
3. **Rate limiting:** Apply per-IP rate limiting on the REST API to reduce the throughput of cache-flooding attacks regardless of the above fixes.

### Proof of Concept
```bash
# Endpoint that accepts multiple timestamp params
BASE="http://localhost:5551/api/v1/transactions"

# 3 timestamp values → 6 permutations → 6 distinct Redis keys
T1="1639010141.000000001"
T2="1639010141.000000002"
T3="1639010141.000000003"

for perm in \
  "timestamp=${T1}&timestamp=${T2}&timestamp=${T3}" \
  "timestamp=${T1}&timestamp=${T3}&timestamp=${T2}" \
  "timestamp=${T2}&timestamp=${T1}&timestamp=${T3}" \
  "timestamp=${T2}&timestamp=${T3}&timestamp=${T1}" \
  "timestamp=${T3}&timestamp=${T1}&timestamp=${T2}" \
  "timestamp=${T3}&timestamp=${T2}&timestamp=${T1}"; do
    curl -s "${BASE}?${perm}" -o /dev/null
done

# Each request lands in Redis under a different MD5(originalUrl) key.
# Scale to N=10 for 3,628,800 unique keys; N=20 for ~2.4×10^18.
# Observe Redis memory grow and legitimate cache entries evicted.
```
