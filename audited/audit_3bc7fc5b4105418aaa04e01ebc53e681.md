### Title
Unauthenticated DB Query Amplification (3x) via GET /api/v1/topics/{id} with No Rate Limiting

### Summary
`TopicController.getTopic()` unconditionally issues three sequential database queries per request — against `topic`, `entity`, and `custom_fee` tables — with no caching and no rate limiting in the `rest-java` module. Any unauthenticated caller supplying a valid-format topic ID that exists in the database can multiply their effective database load by a factor of 3, enabling sustained griefing of the shared PostgreSQL instance.

### Finding Description
**Exact code path:**

`TopicController.java` lines 33–35:
```java
var topic     = topicService.findById(id.id());    // → topicRepository.findById(id.getId())
var entity    = entityService.findById(id.id());   // → entityRepository.findById(id.getId())
var customFee = customFeeService.findById(id.id());// → customFeeRepository.findById(id.getId())
``` [1](#0-0) 

Each service implementation delegates directly to a plain Spring `CrudRepository` with no `@Cacheable` or connection-pool batching: [2](#0-1) [3](#0-2) [4](#0-3) 

The repositories are bare `CrudRepository` interfaces with no caching annotations: [5](#0-4) [6](#0-5) 

**Root cause / failed assumption:** The design assumes that the three queries are cheap enough individually that their aggregate cost per request is acceptable. This assumption fails under adversarial load because there is no per-IP or global rate limiter on this endpoint in the `rest-java` module.

**Why existing checks are insufficient:** The only filters registered in `rest-java` are `LoggingFilter` and `MetricsFilter` — neither enforces any request rate limit: [7](#0-6) [8](#0-7) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter exists only in the `web3` module and is not applied to `rest-java` endpoints: [9](#0-8) 

**Input validation** (`EntityIdNumParameter`) only enforces format (regex), not existence, so any syntactically valid ID passes through to the DB layer: [10](#0-9) 

### Impact Explanation
For every single HTTP GET request to `/api/v1/topics/{id}` with a valid existing topic ID, the shared PostgreSQL database receives exactly 3 independent round-trip queries. An attacker sending *N* requests/second generates *3N* DB queries/second with no server-side throttle. Because topic IDs are public ledger data, valid IDs are trivially enumerable. The impact is degraded database performance for all other mirror-node consumers (REST, gRPC, importer) sharing the same DB cluster — a griefing/availability concern with no economic damage to network participants.

### Likelihood Explanation
Preconditions are minimal: no account, no authentication, no special knowledge beyond a single valid topic ID (e.g., `0.0.1`). The attack is fully repeatable with a simple HTTP client loop (`curl`, `ab`, `wrk`). The absence of any rate-limiting layer in `rest-java` means the attacker's request rate is bounded only by their own network bandwidth and the server's TCP accept queue.

### Recommendation
1. **Add a rate-limiting filter to `rest-java`** analogous to `ThrottleConfiguration` in `web3`, enforcing a per-IP and/or global requests-per-second cap on all `/api/v1/**` endpoints.
2. **Add response caching** (`Cache-Control: public, max-age=5` is already set in the test assertions, but no server-side cache exists): apply `@Cacheable` or a short-lived in-process cache (e.g., Caffeine) on `TopicServiceImpl.findById`, `EntityServiceImpl.findById`, and `CustomFeeServiceImpl.findById` to collapse repeated identical queries.
3. **Consider parallelising the three queries** (e.g., `CompletableFuture`) so that even without caching, the wall-clock DB time per request is reduced.

### Proof of Concept
```bash
# 1. Identify any valid topic ID from the public ledger (e.g., 0.0.1)
TOPIC_ID="0.0.1"
BASE_URL="https://<mirror-node-host>"

# 2. Flood the endpoint — each request triggers 3 DB queries
ab -n 10000 -c 100 "${BASE_URL}/api/v1/topics/${TOPIC_ID}"

# 3. Observe on the DB side:
#    SELECT count(*), query FROM pg_stat_activity
#    WHERE query LIKE '%topic%' OR query LIKE '%entity%' OR query LIKE '%custom_fee%'
#    GROUP BY query;
# Expected: ~3x the HTTP request count in active/recent DB queries,
# with no 429 responses returned to the attacker.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
