### Title
Unguarded `ne` Operator on `schedule.id` Filter Enables Full Table Scan via Unauthenticated Request

### Summary
The `filterValidityChecks` function in `rest/utils.js` validates the `schedule.id` parameter by checking only the value's format (valid entity ID), without restricting the operator. Because `isValidOperatorQuery` accepts `ne` as a valid operator, an unauthenticated user can supply `schedule.id=ne:0.0.1`, which passes all validation and causes `extractSqlFromScheduleFilters` to emit `WHERE s.schedule_id != $1` — a predicate that cannot use the primary-key index and forces a sequential scan of the entire `schedule` table.

### Finding Description

**Operator validation gap — `rest/utils.js` lines 347–349:**
```js
case constants.filterKeys.SCHEDULE_ID:
  ret = EntityId.isValidEntityId(val);   // only value is checked; operator is ignored
  break;
``` [1](#0-0) 

Compare with `BLOCK_NUMBER` and `SLOT`, which explicitly exclude `ne` via `basicOperators`:
```js
case constants.filterKeys.BLOCK_NUMBER:
  ret = (isPositiveLong(val, true) || isHexPositiveInt(val, true)) && includes(basicOperators, op);
  break;
case constants.filterKeys.SLOT:
  ret = isValidSlot(val) && includes(basicOperators, op);
  break;
``` [2](#0-1) [3](#0-2) 

`basicOperators` is explicitly defined to exclude `ne`:
```js
const basicOperators = Object.values(constants.queryParamOperators).filter(
  (o) => o !== constants.queryParamOperators.ne
);
``` [4](#0-3) 

`isValidOperatorQuery` accepts `ne` as a syntactically valid operator:
```js
const isValidOperatorQuery = (query) => {
  return /^(gte?|lte?|eq|ne)$/.test(query);
};
``` [5](#0-4) 

After `buildAndValidateFilters` calls `formatComparator`, the string `ne` is mapped to the SQL `!=` operator. `extractSqlFromScheduleFilters` then directly interpolates it:
```js
whereQuery += `${filterColumnMap[filter.key]}${filter.operator}$${paramCount}`;
``` [6](#0-5) 

Resulting SQL:
```sql
select ... from schedule s
where s.schedule_id != $1
order by s.schedule_id asc
limit $2
```

The `schedule_id` column is the primary key; a `!=` predicate cannot use the B-tree index for a range scan and forces PostgreSQL to perform a sequential scan of the entire table before applying the `LIMIT`.

`getSchedules` calls `buildAndValidateFilters` with no custom validator, so the default `filterValidityChecks` (which does not restrict the operator for `SCHEDULE_ID`) is used:
```js
const filters = utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters);
``` [7](#0-6) 

### Impact Explanation
Every invocation with `schedule.id=ne:<any_valid_id>` causes the database to read every row in the `schedule` table. On a production mirror node with millions of schedule records, this saturates I/O and CPU on the database host, degrading response times for all concurrent users. No financial loss occurs, but availability and latency are directly harmed — consistent with the "griefing / medium" classification.

### Likelihood Explanation
The endpoint is public and requires no authentication. The exploit is a single HTTP GET request with a trivially crafted query parameter. It is trivially repeatable in a loop (e.g., with `curl` or any HTTP client), and no special knowledge beyond the public API documentation is required. The OpenAPI spec even documents `ne` as an example operator for `schedule.id`: [8](#0-7) 

### Recommendation
In `filterValidityChecks` inside `rest/utils.js`, restrict the `SCHEDULE_ID` case to `basicOperators`, mirroring the pattern already used for `BLOCK_NUMBER` and `SLOT`:

```js
case constants.filterKeys.SCHEDULE_ID:
  ret = EntityId.isValidEntityId(val) && includes(basicOperators, op);
  break;
```

Apply the same fix to `ACCOUNT_ID`, `TOKEN_ID`, `SPENDER_ID`, and any other entity-ID filter that currently omits the operator check, to prevent the same class of issue across all list endpoints.

### Proof of Concept
```bash
# Trigger a full sequential scan on the schedule table
curl -s "https://<mirror-node-host>/api/v1/schedules?schedule.id=ne:0.0.1&limit=25"

# Repeat in a loop to amplify DB load
for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/schedules?schedule.id=ne:0.0.$i" &
done
wait
```

Expected result: each request generates `WHERE s.schedule_id != $1` in PostgreSQL, causing a sequential scan. Under concurrent load, DB CPU and I/O spike, and latency for all other API consumers increases measurably.
