### Title
Unauthenticated Log Flooding and Database Connection Exhaustion via Repeated QueryTimeoutException Triggering

### Summary
The `queryTimeout()` handler in `GenericControllerAdvice` unconditionally calls `log.error("Query timed out: {}", e.getMessage())` on every `QueryTimeoutException` with no rate limiting or authentication requirement on the rest-java API. Because the rest-java module has no request throttling (unlike the web3 module which uses bucket4j), any unauthenticated external user can send a high volume of slow queries, each holding a database connection for up to the configured statement timeout (default 10,000 ms), simultaneously flooding error logs and exhausting the database connection pool.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/GenericControllerAdvice.java`, lines 120–124:
```java
@ExceptionHandler
private ResponseEntity<Object> queryTimeout(final QueryTimeoutException e, final WebRequest request) {
    log.error("Query timed out: {}", e.getMessage());
    return handleExceptionInternal(e, null, null, SERVICE_UNAVAILABLE, request);
}
```

**Root cause:** There is no rate limiting, authentication, or per-IP throttling applied to the rest-java API. The `RestJavaProperties` class contains only response header configuration — no throttling properties exist. The only filters registered are `LoggingFilter` and `MetricsFilter`; neither enforces rate limits. By contrast, the `web3` module has a full bucket4j-based `ThrottleManagerImpl` with `requestsPerSecond` and `gasPerSecond` limits — this protection is entirely absent from rest-java.

**Database timeout configuration:** Per `docs/configuration.md` line 629, `hiero.mirror.restJava.db.statementTimeout` defaults to 10,000 ms. This means each slow query holds a database connection for up to 10 seconds before triggering the exception. The connection pool has a bounded `maxConnections`.

**Exploit flow:**
1. Attacker identifies a rest-java endpoint that performs a database query (e.g., `/api/v1/tokens`, `/api/v1/accounts`, `/api/v1/transactions` with wide timestamp ranges or complex filters).
2. Attacker sends N concurrent requests crafted to be slow enough to approach or exceed the 10-second statement timeout.
3. Each request holds a DB connection for ~10 seconds, then throws `QueryTimeoutException`.
4. `queryTimeout()` is invoked for each, calling `log.error(...)` — no deduplication, no rate limiting on the log call.
5. Attacker repeats continuously.

**Why checks fail:** The `statementTimeout` is a per-query safeguard, not a concurrency safeguard. It does not prevent N simultaneous connections from being held. CORS is configured as `allowedOrigins("*")` for all `/api/**` paths, confirming the API is fully public.

### Impact Explanation
Two concrete impacts:
1. **Error log flooding:** Each timeout writes an ERROR-level log entry. At high request rates, this fills log storage, drowns out legitimate error signals, and increases I/O load on the logging subsystem — degrading observability for operators.
2. **Database connection pool exhaustion:** With a bounded connection pool and each slow query holding a connection for up to 10 seconds, a sustained flood of concurrent slow queries can exhaust all available connections, causing legitimate requests to fail with connection timeout errors (`connectionTimeout` default: 20,000 ms per the rest module docs). This constitutes a denial-of-service against legitimate API users.

Severity matches the stated scope: griefing with operational impact (log noise, availability degradation) but no direct economic damage to network participants.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public API surface, and no sophisticated tooling — a simple script using `curl` or any HTTP client in a loop suffices. The absence of any rate limiting in the rest-java module (confirmed by code review of all filters and configuration) means there is no server-side defense. The attack is fully repeatable and can be automated. Any internet-accessible deployment is vulnerable.

### Recommendation
1. **Add request rate limiting to rest-java:** Implement a bucket4j-based throttle filter analogous to `ThrottleManagerImpl` in the web3 module, applied globally to all `/api/**` endpoints.
2. **Rate-limit the error log call:** Use a rate-limited logger (e.g., SLF4J's `LoggingEventBuilder` with a sampler, or a Guava `RateLimiter` guard) inside `queryTimeout()` to suppress duplicate log entries within a time window.
3. **Limit concurrent slow queries per IP:** Apply connection-level or servlet-level concurrency limits (e.g., Tomcat's `maxConnections`, or a `SemaphoreSlim` per remote IP).
4. **Reduce `statementTimeout`** to the minimum acceptable value to reduce the window each connection is held.

### Proof of Concept
```bash
# Send 50 concurrent requests with a wide timestamp range to trigger slow queries
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gte:0.000000000&timestamp=lte:9999999999.999999999&limit=100" &
done
wait

# Observe: server logs fill with repeated ERROR entries:
# ERROR Query timed out: ERROR: canceling statement due to statement timeout
# Observe: subsequent legitimate requests receive 503 Service Unavailable
# due to connection pool exhaustion
```
