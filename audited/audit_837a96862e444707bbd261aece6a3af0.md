### Title
Deleted Entity Bypass via Long-Form EVM Address in `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when a "long-form" EVM address is supplied (first 12 bytes all zero, last 8 bytes encoding an entity ID), the code falls into a branch that calls `entityRepository.findById()` — a Spring Data `CrudRepository` method that carries no deletion filter. The sibling branch calls `entityRepository.findByEvmAddress()`, which explicitly appends `and deleted is not true`. An unauthenticated attacker can craft a long-form EVM address encoding the numeric ID of any deleted account and retrieve its full record through the public GraphQL endpoint.

### Finding Description

**Exact code path:**

`EntityServiceImpl.java` lines 34–41: [1](#0-0) 

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // shard=0, realm=0
        return entityRepository.findById(buffer.getLong())         // ← NO deletion filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // ← has deletion filter
                           .filter(e -> e.getType() == type);
}
```

`EntityRepository.java` lines 16–17 (the guarded path): [2](#0-1) 

```sql
select * from entity where evm_address = ?1 and deleted is not true
```

`findById()` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT … WHERE id = ?` with no `deleted` predicate.

**Root cause:** The two branches of the `if` statement apply inconsistent deletion semantics. The `findByEvmAddress` branch is correctly guarded; the `findById` branch is not.

**Exploit flow:**

1. Attacker learns (or enumerates) the numeric entity ID of a deleted account — entity IDs are sequential integers, trivially guessable.
2. Attacker constructs a 20-byte long-form EVM address: bytes 0–3 = `0x00000000`, bytes 4–11 = `0x0000000000000000`, bytes 12–19 = big-endian encoding of the target entity ID.
3. Attacker submits a GraphQL query to the public `/graphql` endpoint:
   ```graphql
   { account(input: { evmAddress: "0x000000000000000000000000<entity_id_hex>" }) {
       id balance alias evmAddress
   }}
   ```
4. `getByEvmAddressAndType` detects the long-form pattern, calls `findById(<entity_id>)`, and returns the deleted entity.
5. The only downstream check is `.filter(e -> e.getType() == type)` — it checks entity type, not deletion status.

The `AccountController` exposes this path to any caller with no authentication requirement: [3](#0-2) 

### Impact Explanation
Any deleted account record — including its balance history, alias, EVM address, and associated metadata — can be read by an unauthenticated external user. This violates the intended invariant that deleted entities are invisible to API consumers, and may expose sensitive account state that operators expected to be suppressed after deletion.

### Likelihood Explanation
No privileges, credentials, or special network access are required. The GraphQL endpoint is public. Entity IDs are sequential integers starting from 1, so brute-force enumeration of deleted IDs is trivial. The exploit is fully repeatable and requires only a single HTTP request per target ID.

### Recommendation
Apply the same deletion guard in the long-form branch. Replace:

```java
return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
```

with a dedicated repository query that includes the deletion predicate, for example:

```java
// In EntityRepository:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

```java
// In EntityServiceImpl:
return entityRepository.findByIdAndNotDeleted(buffer.getLong()).filter(e -> e.getType() == type);
```

Alternatively, add a post-fetch guard: `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` immediately after `findById()`, consistent with how `findByAlias` and `findByEvmAddress` handle deletion at the SQL level.

### Proof of Concept

**Precondition:** Entity with ID `1001` exists in the database with `deleted = true` and `type = ACCOUNT`.

**Step 1 — Construct the long-form EVM address:**
```
Bytes 0-3  (shard):  00 00 00 00
Bytes 4-11 (realm):  00 00 00 00 00 00 00 00
Bytes 12-19 (id):    00 00 00 00 00 00 03 E9   ← 1001 in big-endian
Full hex: 0x000000000000000000000000000003E9
```

**Step 2 — Send GraphQL query:**
```graphql
{
  account(input: { evmAddress: "0x000000000000000000000000000003E9" }) {
    balance
    alias
    evmAddress
  }
}
```

**Expected (correct) result:** `null` — entity is deleted.

**Actual result:** The deleted account record is returned in full, because `findById(1001)` has no `deleted is not true` predicate. [4](#0-3)
