### Title
Unbounded Entity CTE Full-Table Scan Tripled Per Request via Public Contract Logs API

### Summary
The `contractLogsExtendedQuery` static query in `rest/service/contractService.js` defines a CTE that unconditionally selects all rows from the `entity` table with no WHERE clause. When `getContractLogsQuery()` receives non-empty `lower`, `inner`, and `upper` filter sets simultaneously, it emits three independent UNION sub-queries each containing this full-table CTE, causing the database to scan the entire entity table up to three times per single unauthenticated HTTP request. Any unprivileged external user can trigger this path by supplying a valid combination of `timestamp` and `index` range filters to the public REST endpoints.

### Finding Description

**Exact code location:**

`rest/service/contractService.js` lines 134–144 — the CTE has no predicate:

```sql
with entity as (
  select evm_address, id
  from entity          -- no WHERE clause; full table scan
)
select ...
from contract_log cl
left join entity e on id = contract_id
``` [1](#0-0) 

`getContractLogsQuery()` lines 335–364 — when all three of `lower`, `inner`, `upper` are non-empty, each is independently wrapped in `buildSelectQuery(ContractService.contractLogsExtendedQuery, ...)` and joined with `UNION`:

```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) =>
    super.buildSelectQuery(
      ContractService.contractLogsExtendedQuery, ...
    )
  );
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion\n'), ...].join('\n');
``` [2](#0-1) 

The test suite confirms the exact SQL emitted — three separate `WITH entity AS (select evm_address, id from entity)` blocks, one per UNION branch: [3](#0-2) 

**Root cause:** The CTE is defined without any filter on the entity table. Because each UNION branch is a self-contained query with its own `WITH` clause, PostgreSQL evaluates (and potentially materializes) the CTE independently for each branch. The failed assumption is that the planner will always push the `contract_id` join predicate into the CTE; in practice, PostgreSQL treats CTEs as optimization fences (guaranteed in PG ≤ 11; planner-dependent in PG 12+ unless `NOT MATERIALIZED` is specified), so the full entity table is read up to three times per request.

**Trigger path:**

`extractContractLogsMultiUnionQuery` (controller, lines 582–698) populates `lower`, `inner`, and `upper` via `getContnerLogsLowerFilters`, `getInnerFilters`, and `getUpperFilters`. All three are non-empty when the user supplies both a lower-bound and an upper-bound on `timestamp` **and** at least one bound on `index`: [4](#0-3) 

Both public endpoints route directly to this logic with no authentication: [5](#0-4) 

### Impact Explanation
The `entity` table in a production Hedera mirror node accumulates every account, contract, token, and topic ever created — potentially tens of millions of rows. Each CTE materialization reads all `(evm_address, id)` pairs. Three materializations per request, multiplied across concurrent requests from a single or distributed attacker, can drive PostgreSQL's `work_mem` and shared-buffer usage well above normal operating levels. Because the endpoint returns paginated results (default limit applies), the attacker receives a cheap, bounded response while the database absorbs the full scan cost. This directly maps to the stated threat: increasing DB memory/CPU consumption by ≥30% without brute-force volume, since a small number of carefully crafted requests each impose 3× the normal entity-table I/O.

### Likelihood Explanation
The exploit requires zero privileges, zero accounts, and zero knowledge beyond the public OpenAPI specification, which documents the `timestamp` and `index` query parameters explicitly. [6](#0-5) 

The throttle infrastructure found in the repository (`ThrottleManagerImpl`, `ThrottleConfiguration`) applies only to the `web3` Java service (contract-call simulation), not to the Node.js REST service contract-logs endpoints. Infrastructure-level rate limiting (Traefik) limits request count per source IP but does not account for per-request DB cost, so a single request already imposes the amplified scan. The attack is trivially repeatable from multiple IPs or via a single slow-drip of requests that stay under per-IP rate limits.

### Recommendation

1. **Add `NOT MATERIALIZED` hint** (PostgreSQL 12+) to the CTE so the planner can inline and push predicates:
   ```sql
   with entity as NOT MATERIALIZED (
     select evm_address, id from entity
   )
   ```
2. **Eliminate the CTE entirely** and replace with a direct `LEFT JOIN entity e ON e.id = cl.contract_id` in the base query, allowing the planner to use the primary-key index on `entity.id` for a nested-loop join rather than a full scan.
3. **Apply a cost-based statement timeout** at the PostgreSQL session level for the REST API database role to cap runaway queries.
4. **Add per-endpoint rate limiting** in the Node.js REST middleware that accounts for query complexity (e.g., presence of both timestamp and index range filters), not just request count.

### Proof of Concept

Send a single unauthenticated GET request to either public endpoint with parameters that force all three UNION branches:

```
GET /api/v1/contracts/results/logs?timestamp=gte:1000000000000000000&timestamp=lte:9999999999999999999&index=gt:0&index=lt:9999
```

Or against a specific contract:

```
GET /api/v1/contracts/0.0.1234/results/logs?timestamp=gte:1000000000000000000&timestamp=lte:9999999999999999999&index=gt:0&index=lt:9999
```

**What happens in the database:**
```sql
(with entity as (select evm_address, id from entity)   -- full scan #1
 select ... from contract_log cl left join entity e on id = contract_id
 where cl.index > $4 and cl.consensus_timestamp = $5 ...)
union
(with entity as (select evm_address, id from entity)   -- full scan #2
 select ... from contract_log cl left join entity e on id = contract_id
 where cl.consensus_timestamp > $6 and cl.consensus_timestamp < $7 ...)
union
(with entity as (select evm_address, id from entity)   -- full scan #3
 select ... from contract_log cl left join entity e on id = contract_id
 where cl.index < $8 and cl.consensus_timestamp = $9 ...)
order by consensus_timestamp desc, index desc limit $3
```

Monitor `pg_stat_activity` and `pg_stat_bgwriter` / `pg_stat_user_tables` (`seq_scan`, `seq_tup_read` on the `entity` table) before and after the request to observe the three sequential scans. Repeat the request at a moderate rate (e.g., 5–10 req/s from multiple IPs) to observe sustained memory pressure growth in `pg_stat_statements` (`shared_blks_hit`, `temp_blks_written`).
