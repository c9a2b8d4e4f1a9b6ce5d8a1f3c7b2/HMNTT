### Title
Global Shared Rate-Limit Bucket Allows Single Unprivileged Client to Monopolize Full Request Budget

### Summary
The `rateLimitBucket` bean in `ThrottleConfiguration.java` is a single JVM-wide token bucket with no per-IP or per-user partitioning. Any unauthenticated caller who sustains 500 requests/second (the default `requestsPerSecond`) continuously drains all tokens as fast as `refillGreedy` replenishes them, leaving zero tokens for every other concurrent user and causing them to receive HTTP 429 for the entire duration of the attack.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) creates one singleton `Bucket` with `capacity(rateLimit)` and `refillGreedy(rateLimit, Duration.ofSeconds(1))`. [1](#0-0) 

`ThrottleManagerImpl.throttle()` (line 38) calls `rateLimitBucket.tryConsume(1)` with no caller identity check — the same singleton bucket is decremented for every request from every source IP. [2](#0-1) 

**Root cause / failed assumption:**
The design assumes the 500-token budget will be shared fairly across many concurrent callers. That assumption fails because there is no per-source enforcement: a single caller who issues requests at the full refill rate consumes every token the moment `refillGreedy` makes it available.

`refillGreedy` replenishes tokens continuously (1 token per 2 ms at the 500 req/s default). An attacker who maintains a steady 500 req/s pipeline drains each token within microseconds of its creation, leaving the bucket perpetually at 0 for all other callers. [3](#0-2) 

**Why existing checks are insufficient:**
There is no per-IP bucket, no authentication gate, and no connection-level quota anywhere in the web3 config layer. [4](#0-3) 

The Kubernetes `maxRatePerEndpoint: 250` in the Helm chart applies per backend pod endpoint, not per client IP, and is disabled by default (`gateway.gcp.enabled: false`). [5](#0-4) 

### Impact Explanation
Every legitimate user of the `/api/v1/contracts/call` endpoint receives HTTP 429 for the entire duration of the attack. Because the bucket is JVM-wide and the attacker can sustain the load indefinitely at zero cost (no authentication, no fee), this constitutes a complete, persistent denial-of-service of the Web3 API for all other users. Severity is Medium (griefing / availability impact, no on-chain economic damage).

### Likelihood Explanation
The precondition is zero: no account, no API key, no special network position. Sending 500 HTTP POST requests per second is trivially achievable with a single commodity machine and a tool such as `wrk`, `hey`, or a simple async HTTP client. The attack is repeatable indefinitely and requires no timing precision because `refillGreedy` makes tokens available continuously — the attacker simply keeps a saturated connection pool open.

### Recommendation
1. **Per-IP rate limiting (primary fix):** Replace the single global bucket with a `CaffeineCache`-backed or `Bucket4j`-`ProxyManager`-based per-source-IP bucket so each client is limited to a fraction of the global budget.
2. **Reduce burst capacity:** Use `refillIntervally` instead of `refillGreedy` so tokens are added in discrete intervals rather than continuously, preventing continuous draining.
3. **Enforce at the ingress layer:** Enable the GCP Gateway `maxRatePerEndpoint` with `sessionAffinity: CLIENT_IP` so rate limiting is applied before requests reach the JVM. [6](#0-5) 

### Proof of Concept
```
# Precondition: no credentials required
# Tool: wrk (or any HTTP load generator)

# Step 1 – drain the bucket continuously
wrk -t10 -c100 -d60s \
    -s post.lua \
    http://<web3-host>/api/v1/contracts/call

# post.lua sends a minimal valid JSON body

# Step 2 – from a second machine, observe all requests return 429
curl -X POST http://<web3-host>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"data":"0x","to":"0x0000000000000000000000000000000000000001"}'
# Expected: HTTP 429 "Requests per second rate limit exceeded"

# Step 3 – stop wrk; legitimate requests immediately succeed again,
#           confirming the bucket was fully monopolized by the attacker.
``` [7](#0-6)

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
