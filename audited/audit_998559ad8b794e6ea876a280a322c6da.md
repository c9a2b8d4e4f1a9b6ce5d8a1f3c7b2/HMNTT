### Title
Integer Overflow in `poll()` Bypasses `maxPageSize` Cap, Enabling Unbounded Database Queries

### Summary
`PollingTopicMessageRetriever.poll()` casts the user-supplied `long` limit to `int` without an upper-bound check. Any `limit > Integer.MAX_VALUE` (e.g., `2147483648L`) overflows to a negative `int`, which then passes through `Math.min()` as-is, causing `newFilter.hasLimit()` to return `false` and the repository to issue a completely unbounded SQL query against the database. An unprivileged gRPC caller can trigger this with a single subscription request.

### Finding Description

**Root cause — unsafe narrowing cast with no upper-bound validation:**

`TopicMessageFilter.limit` is a `long` with only `@Min(0)` — no `@Max` constraint: [1](#0-0) 

In `PollingTopicMessageRetriever.poll()`, the remaining limit is computed with a narrowing cast: [2](#0-1) 

When `filter.getLimit()` is `2147483648L` (= `Integer.MAX_VALUE + 1`) and `context.getTotal()` is `0`:
- `(int)(2147483648L - 0L)` → `Integer.MIN_VALUE` (`-2147483648`) due to Java narrowing-cast overflow
- `Math.min(-2147483648, 1000)` → `-2147483648` (negative wins; `maxPageSize` is completely bypassed)
- `newFilter` is built with `limit = -2147483648` [3](#0-2) 

**Repository layer — no limit applied when `hasLimit()` is false:**

`hasLimit()` returns `limit > 0`, so `-2147483648` evaluates to `false`: [4](#0-3) 

`TopicMessageRepositoryCustomImpl.findByFilter()` only calls `setMaxResults()` when `hasLimit()` is true: [5](#0-4) 

With `hasLimit() = false`, `setMaxResults()` is never called, and the JPA query returns **every row** matching the topic ID and start time — no SQL `LIMIT` clause is emitted.

**Outer `take()` does not save you:**

`TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` using the original user-supplied value: [6](#0-5) 

With `limit = 2147483648L`, this allows up to ~2.1 billion messages to stream through — effectively unlimited for any real topic. The unbounded DB query has already executed and loaded all rows into memory before `take()` can act.

**`isComplete()` does not terminate early:**

`isComplete()` checks `filter.getLimit() == total.get()`, which requires exactly `2147483648L` messages to have been seen — never true in practice: [7](#0-6) 

**Note on `Integer.MAX_VALUE` exactly:** The question's stated value of `2147483647` does NOT overflow — `(int)(2147483647L)` = `2147483647`, and `Math.min(2147483647, 1000)` = `1000`. The actual trigger is any value ≥ `2147483648L`.

### Impact Explanation
A single gRPC `subscribeTopic` call with `limit = 2147483648` causes the server to execute an unbounded `SELECT *` against the `topic_message` table for the target topic, loading all matching rows into JVM heap. On a busy topic with millions of messages this causes OOM / GC pressure on the mirror node, constituting a denial-of-service. The attack is repeatable and requires no authentication.

### Likelihood Explanation
The gRPC `ConsensusTopicQuery.limit` field is `uint64`; any client library can trivially set it to `2147483648`. No privilege, API key, or special knowledge is required. The attack is a single RPC call and is fully repeatable.

### Recommendation
1. Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) to `TopicMessageFilter.limit` to reject oversized values at the validation layer.
2. Replace the unsafe cast in `poll()` with a safe clamp:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. Apply the same fix to the identical pattern in `PollingTopicListener.poll()` (line 55). [8](#0-7) 

### Proof of Concept
**Preconditions:** A topic with at least one message exists. No authentication required.

**Steps:**
1. Connect to the mirror node gRPC endpoint (default port 5600).
2. Send a `subscribeTopic` RPC with:
   ```
   topicID = <any valid topic>
   consensusStartTime = 0
   limit = 2147483648   // Integer.MAX_VALUE + 1
   ```
3. **Trigger:** `poll()` computes `(int)(2147483648L - 0L)` = `-2147483648`; `Math.min(-2147483648, 1000)` = `-2147483648`; `newFilter.limit = -2147483648`; `hasLimit()` = `false`.
4. **Result:** `TopicMessageRepositoryCustomImpl.findByFilter()` issues `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with **no LIMIT clause**, returning all rows for that topic. On a large topic this exhausts server heap and causes OOM / service degradation.
