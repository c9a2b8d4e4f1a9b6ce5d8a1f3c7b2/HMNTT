### Title
Unauthenticated HTTP Node-List Fetch in `buildNetworkFromMirrorNodes` Enables MITM-Driven Network Routing Takeover

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the Hiero network node list over a plain, unauthenticated HTTP connection with no scheme enforcement, no TLS requirement, and no response integrity verification. An attacker positioned on the network path between the pinger and the mirror REST endpoint can intercept the HTTP response and substitute a crafted node list containing only targeted consensus nodes. The Hiero SDK client is then initialized exclusively against those nodes, concentrating all transaction traffic on them.

### Finding Description

**Exact code path:**

`pinger/config.go` line 37 shows the `mirrorRest` field is populated from the `-mirror-rest` flag or `HIERO_MIRROR_PINGER_REST` env var, with the documented example being `http://mirror-rest:5551` — an explicit HTTP URL. The only validation at line 133 is a non-empty check; no scheme (`https://`) is enforced.

`pinger/mirror_node_client.go` lines 36–44: `buildNetworkFromMirrorNodes` takes `cfg.mirrorRest` verbatim, trims it, and appends the API path — no scheme check, no HTTPS upgrade.

```go
base := strings.TrimRight(strings.TrimSpace(cfg.mirrorRest), "/")
// ... appends /api/v1/network/nodes
```

Line 46: a plain `http.Client` is constructed with no TLS configuration, no certificate pinning, no `Transport` override:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

Lines 79–98 (`fetchMirrorNodeNetwork`): the HTTP GET is issued and the response body is decoded directly into `nodesEnvelope` with no signature check, no HMAC, no comparison against a known-good baseline.

Lines 100–124: every `service_endpoint` from the attacker-controlled response is inserted into the `network` map, which is returned to `pinger/sdk_client.go` line 22:
```go
client = hiero.ClientForNetwork(netmap)
```

The SDK client is now configured exclusively with the attacker-supplied node addresses. `main.go` line 63 then submits transactions on every ticker tick against this poisoned client.

**Root cause:** The code assumes the mirror REST endpoint is trusted and reachable only over a secure channel. Neither assumption is enforced in code.

**Why existing checks fail:** The only guard (`config.go` line 133) verifies the field is non-empty. There is no `strings.HasPrefix(cfg.mirrorRest, "https://")` check, no TLS-enforcing `http.Transport`, and no cryptographic verification of the returned payload.

### Impact Explanation

An attacker who successfully injects a crafted node list can direct 100% of the pinger's transaction volume at an arbitrary subset of consensus nodes. If the injected list covers ≥30% of the network's processing nodes (e.g., listing only 3–4 specific nodes on a 10-node network), those nodes receive concentrated gRPC load from every `submitWithRetry` tick. Sustained flooding can exhaust node resources and cause them to stop processing transactions, meeting the stated impact threshold of shutting down ≥30% of network processing nodes without brute-force key compromise.

### Likelihood Explanation

**Preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` is set (required to invoke `buildNetworkFromMirrorNodes`).
2. `HIERO_MIRROR_PINGER_REST` is an HTTP URL (the documented default example is HTTP).
3. The attacker can intercept traffic between the pinger pod and the mirror REST service.

**Feasibility:** In a Kubernetes deployment, the attacker needs a foothold in the cluster (e.g., a compromised pod in the same namespace) to perform DNS spoofing or ARP poisoning of the internal service name `mirror-rest`. If the mirror REST URL resolves to a public endpoint over HTTP, any on-path network attacker (ISP, BGP hijacker, cloud provider network) can intercept without any cluster access. The attack is repeatable: the node list is fetched once at startup, so a single successful interception permanently poisons the client for the lifetime of the process.

### Recommendation

1. **Enforce HTTPS in `loadConfig()`** (`pinger/config.go`): reject any `mirrorRest` value that does not begin with `https://`.
2. **Pin TLS or verify the server certificate** in `fetchMirrorNodeNetwork` by configuring `http.Transport` with a specific CA or certificate fingerprint.
3. **Validate the returned node list** against a known-good set (e.g., cross-reference with a hardcoded or separately-fetched signed address book) before passing it to `hiero.ClientForNetwork`.
4. Update the documented default example from `http://mirror-rest:5551` to `https://mirror-rest:443` to prevent operators from inadvertently deploying with HTTP.

### Proof of Concept

1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. On the network path (e.g., via a compromised pod performing DNS spoofing of `mirror-rest`), intercept the GET request to `http://mirror-rest:5551/api/v1/network/nodes`.
3. Return a crafted JSON response:
   ```json
   {
     "nodes": [
       {
         "node_account_id": "0.0.3",
         "service_endpoints": [
           {"domain_name": "target-node-1.example.com", "port": 50211}
         ]
       },
       {
         "node_account_id": "0.0.4",
         "service_endpoints": [
           {"domain_name": "target-node-2.example.com", "port": 50211}
         ]
       }
     ],
     "links": {"next": null}
   }
   ```
4. `buildNetworkFromMirrorNodes` returns this map; `hiero.ClientForNetwork(netmap)` is called with only the two targeted nodes.
5. Every subsequent `submitWithRetry` tick sends gRPC transactions exclusively to `target-node-1` and `target-node-2`, concentrating load and contributing to their shutdown.
