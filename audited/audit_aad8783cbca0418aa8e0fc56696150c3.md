### Title
`isValidAccount()` Excludes `entity_history`, Causing False 404 for Deleted Accounts on Token and Reward Endpoints

### Summary
`EntityService.isValidAccount()` queries only the `entity` table to determine account validity. When an account has been deleted and its record removed from `entity` (but historical records remain in `entity_history`), the function returns `false`, causing the `/accounts/{id}/tokens` and `/accounts/{id}/rewards` endpoints to throw a `NotFoundError` (HTTP 404) for a legitimately historical account. Any unprivileged user can trigger this by supplying such an account ID.

### Finding Description
**Code location:**
- `rest/service/entityService.js`, lines 28–30 (`entityExistenceQuery`) and lines 60–63 (`isValidAccount`)
- `rest/controllers/tokenController.js`, lines 68–71 (`getTokenRelationships`)
- `rest/controllers/accountController.js`, lines 172–175 (`listStakingRewardsByAccountId`)

**Root cause:**

`entityExistenceQuery` is defined as:
```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}   // resolves to 'entity'
                               where ${Entity.ID} = $1`;
```
`Entity.tableName = 'entity'` (line 9 of `rest/model/entity.js`). The query never touches `entity_history` (defined as `Entity.historyTableName = 'entity_history'` on line 7).

`isValidAccount()` returns `false` whenever the account is absent from the live `entity` table:
```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);   // false if account only exists in entity_history
}
```

Both controllers gate their responses on this check:
```js
// tokenController.js:68-71
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }

// accountController.js:172-175
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }
```

**Exploit flow:**
1. Account `0.0.X` is created, accumulates token relationships and staking rewards, then is deleted/permanently removed from the `entity` table. Its historical state rows remain in `entity_history`.
2. An unprivileged user sends `GET /accounts/0.0.X/tokens` or `GET /accounts/0.0.X/rewards`.
3. `EntityService.getEncodedId()` resolves the numeric ID successfully (no DB lookup needed for a plain `shard.realm.num` string).
4. `isValidAccount(accountId)` queries only `entity`, finds no row, returns `false`.
5. The controller throws `NotFoundError` → HTTP 404, even though the account has valid historical token/reward records.

**Why existing checks are insufficient:**
The `entityExistenceQuery` has no `UNION` with `entity_history` and no fallback. The `entityFromAliasQuery` and `entityFromEvmAddressQuery` also only query `entity`, so the omission is consistent but uniformly wrong for deleted accounts.

### Impact Explanation
Any account that has been permanently deleted from the `entity` table but retains rows in `entity_history` becomes inaccessible via the `/tokens` and `/rewards` REST endpoints. Legitimate users (e.g., account holders, auditors, integrators) receive a misleading 404 instead of the correct historical data. This constitutes incorrect data export from the mirror node — the stated scope of the finding — and can break downstream applications that rely on these endpoints for historical account data.

### Likelihood Explanation
The precondition (account deleted from `entity` but present in `entity_history`) is a normal lifecycle event on Hedera: accounts can be deleted via `CryptoDelete`. No special privileges are required; any user who knows or discovers a deleted account ID can trigger the 404. The trigger is a plain unauthenticated HTTP GET request. The condition is repeatable for every such account.

### Recommendation
Extend `entityExistenceQuery` (or add a fallback) to also check `entity_history`:
```sql
SELECT type FROM entity WHERE id = $1
UNION ALL
SELECT type FROM entity_history WHERE id = $1
LIMIT 1
```
Alternatively, restructure `isValidAccount()` to perform a two-step check: query `entity` first, and if no row is found, query `entity_history` before returning `false`. This mirrors the pattern already used in the Java `web3` layer (`findActiveByIdAndTimestamp` uses a `UNION ALL` across both tables).

### Proof of Concept
1. Insert an account into `entity` (id = 5000), add token relationships and staking reward rows referencing it.
2. Delete the account: move/remove its row from `entity` (simulating `CryptoDelete` importer behavior), leaving its history in `entity_history`.
3. Send `GET /api/v1/accounts/0.0.5000/tokens` — observe HTTP 404.
4. Send `GET /api/v1/accounts/0.0.5000/rewards` — observe HTTP 404.
5. Directly query `SELECT * FROM entity_history WHERE id = <encoded_5000>` — confirm rows exist.
6. Patch `entityExistenceQuery` to include `UNION ALL` with `entity_history`, repeat steps 3–4 — observe HTTP 200 with correct data. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
