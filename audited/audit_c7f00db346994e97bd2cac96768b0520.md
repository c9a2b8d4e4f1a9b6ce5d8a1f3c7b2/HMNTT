### Title
Unauthenticated Thread-Exhaustion via Long-Running EVM Calls Bounded Only by Global Gas Throttle

### Summary
The `call()` endpoint in `ContractController` is a synchronous blocking handler with no per-IP rate limiting and no concurrent-request cap. The global gas/request throttle is consumed atomically at the start of each request, before any EVM execution begins. An unprivileged attacker can pass the throttle check and then hold a server thread for up to the full `requestTimeout` (10 s default) by crafting EVM calls that trigger many database reads, because the timeout is only enforced at Hibernate SQL-statement boundaries, not continuously. Sending requests at the allowed rate (500 req/s) with each holding a thread for ~10 s can accumulate thousands of blocked threads, starving legitimate users.

### Finding Description

**Code path:**

`ContractController.call()` (lines 38–51) calls `throttleManager.throttle(request)` first, then immediately delegates to the blocking `contractExecutionService.processCall(params)`. [1](#0-0) 

`ThrottleManagerImpl.throttle()` (lines 37–48) deducts tokens from the global `rateLimitBucket` (500 req/s) and `gasLimitBucket` (7.5 B gas/s scaled by 10 000) and returns immediately. There is no mechanism to account for how long the request will actually hold a thread. [2](#0-1) 

The only timeout enforcement is in `HibernateConfiguration.statementInspector()` (lines 31–47), which checks elapsed time only immediately before each Hibernate SQL statement executes. If the EVM execution path between two consecutive DB reads is long (pure computation, or a single very slow query), the thread is blocked for that entire interval with no interruption. [3](#0-2) 

`ContractCallContext.startTime` is set once at context creation and never reset; the inspector compares `System.currentTimeMillis() - startTime` against the 10 s threshold. [4](#0-3) 

`Web3Properties.requestTimeout` defaults to 10 seconds with no upper bound enforced by the throttle. [5](#0-4) 

`ContractCallRequest.gas` has a minimum of 21 000 and a default of 15 000 000; no authentication is required. [6](#0-5) 

**Root cause / failed assumption:** The throttle assumes that gas consumed ≈ wall-clock time consumed. This holds on average but not per-request. A single request that passes the throttle can hold a thread for the entire 10 s window. Because the throttle is global and not per-IP, a single attacker can saturate the allowed request rate and simultaneously accumulate a large pool of in-flight, thread-blocking executions.

### Impact Explanation

With the default limits (500 req/s, 10 s timeout), an attacker sustaining the maximum request rate for 10 s accumulates up to 5 000 concurrently blocked threads. Spring Boot's embedded Tomcat defaults to 200 worker threads; even a fraction of this load exhausts the thread pool. Legitimate requests queue behind the backlog and eventually receive 503 responses or connection-refused errors. No funds are at risk, but the mirror-node web3 API becomes unavailable for the duration of the attack, which is repeatable indefinitely at zero cost to the attacker.

### Likelihood Explanation

No account, API key, or on-chain asset is required. The attacker needs only network access and knowledge of any deployed contract that performs storage reads (SLOAD). Such contracts are publicly visible on-chain. The attack is trivially scriptable with standard HTTP tooling (e.g., `wrk`, `ab`, or a simple loop with `curl`). Because the throttle is global, a single attacker machine can saturate it without needing a botnet.

### Recommendation

1. **Add a concurrent-request semaphore** in `ContractController.call()` (or as a servlet filter) that caps the number of simultaneously executing EVM calls (e.g., 50–100). Reject with HTTP 429 when the cap is reached.
2. **Add per-source-IP rate limiting** (e.g., via a Bucket4j key derived from `HttpServletRequest.getRemoteAddr()`) so a single client cannot consume the entire global budget.
3. **Use async/non-blocking execution** (e.g., Spring WebFlux or a bounded `ExecutorService`) so long-running EVM calls do not hold HTTP worker threads.
4. **Enforce a hard wall-clock deadline** using `Thread.interrupt()` or `Future.get(timeout)` rather than relying solely on the SQL-boundary inspector, which cannot interrupt pure-computation or single-long-query phases.

### Proof of Concept

```bash
# 1. Identify any contract on the network with a storage-heavy read function.
#    Example: a contract whose getter iterates over a large mapping.
CONTRACT="0x000000000000000000000000000000000000XXXX"
DATA="0x<selector_of_storage_heavy_function>"

# 2. Flood the endpoint at the allowed rate from a single machine.
#    Each request uses max gas to maximise execution time.
for i in $(seq 1 5000); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"$CONTRACT\",\"data\":\"$DATA\",\"gas\":15000000,\"block\":\"latest\"}" &
done
wait

# 3. Observe: after ~10 s the server thread pool is exhausted.
#    Legitimate requests receive 503 Service Unavailable or time out.
curl -v http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x...","data":"0x...","gas":21000,"block":"latest"}'
# Expected: connection refused or 503
```

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
    }
