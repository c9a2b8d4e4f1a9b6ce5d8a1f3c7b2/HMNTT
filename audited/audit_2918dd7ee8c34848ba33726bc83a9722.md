### Title
Unauthenticated Repeated Expensive DB Query via Unbounded NFT Allowance Range Scan (DoS/Griefing)

### Summary
The `getNftAllowances()` endpoint in `AllowancesController` accepts fully open-ended range parameters (`account.id=gte:0.0.1`, `token.id=gt:0.0.1`) combined with `owner=false`, `order=desc`, and `limit=100` from any unauthenticated caller. The `rest-java` module has no rate-limiting mechanism, allowing an attacker to repeatedly trigger the most expensive query variant — a broad-range scan filtered by `SPENDER` but sorted by `OWNER DESC, TOKEN_ID DESC` — degrading database performance for all users.

### Finding Description

**Code path:**

`AllowancesController.java` line 57–83 accepts the parameters with only bean-validation constraints (`@Positive @Max(MAX_LIMIT)`, `@Size(max=2)`). No authentication, no rate limiting. [1](#0-0) 

`NftAllowanceServiceImpl.checkOwnerSpenderParamValidity()` validates parameter *correctness* (no NE, single occurrence, GTE present when GT token.id used), but does **not** reject or throttle expensive open-ended range combinations. [2](#0-1) 

The combination `account.id=gte:0.0.1` + `token.id=gt:0.0.1` passes all service-layer checks: `ownerOrSpenderIds` is non-empty, no NE operator, single occurrence, and the GTE on `account.id` satisfies the GT-token.id co-presence rule (lines 50–53). [3](#0-2) 

`NftAllowanceRepositoryCustomImpl.findAll()` then executes:
```sql
SELECT * FROM nft_allowance
WHERE spender = {path_id} AND approved_for_all = true
  AND owner >= 1 AND token_id > 1
ORDER BY owner DESC, token_id DESC
LIMIT 100
```
The equality filter is on `SPENDER` but the sort is on `OWNER DESC, TOKEN_ID DESC`. The DB cannot satisfy both the filter and the sort order from a single composite index starting with `OWNER`, forcing it to scan all rows for the given spender, apply the broad range filters, sort the full result set, and then truncate to 100. [4](#0-3) 

**No rate limiting exists in `rest-java`:** The `ThrottleConfiguration`, `ThrottleManagerImpl`, and `ThrottleProperties` classes are exclusively in the `web3` module. The `rest-java` configuration (`RestJavaConfiguration`, `WebMvcConfiguration`) registers only an ETag filter and argument resolvers — no request throttle. [5](#0-4) [6](#0-5) 

`MAX_LIMIT = 100` is the only numeric guard, and it is the maximum the attacker is already requesting. [7](#0-6) 

### Impact Explanation
An attacker targeting a high-activity account (e.g., a popular spender with thousands of NFT allowances) can flood the endpoint with the described request. Each request forces a full per-spender scan + sort on the DB. With no rate limiting, a single attacker machine can sustain hundreds of such requests per second, saturating DB CPU/IO and causing latency spikes or timeouts for all other API consumers. This is a medium-severity griefing DoS with no economic cost to the attacker.

### Likelihood Explanation
No account, API key, or token is required. The parameters are all valid and pass every existing check. The exploit is trivially scriptable with `curl` or any HTTP client. Any motivated actor can execute it from a single machine or a small botnet.

### Recommendation
1. **Add rate limiting to `rest-java`**: Implement a per-IP (or global) token-bucket throttle equivalent to the one in `web3/ThrottleConfiguration` for all `/api/v1/` endpoints.
2. **Reject or cost-limit open-ended range queries**: In `checkOwnerSpenderParamValidity`, reject or warn when both `ownerOrSpenderIds` and `tokenIds` have only a lower bound (no upper bound), as this produces the most expensive scan.
3. **Require an upper bound when a lower bound is provided for both parameters**: Enforce that if `account.id=gte:X` is given without `account.id=lte:Y`, `token.id` must also be bounded above, limiting the scan range.

### Proof of Concept
```bash
# Replace 0.0.XXXXX with any account known to be a spender of many NFT allowances
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.XXXXX/allowances/nfts\
?limit=100&owner=false&order=desc&account.id=gte:0.0.1&token.id=gt:0.0.1" \
    -o /dev/null &
done
```
Each iteration triggers the broad-range, descending-order, cross-column sort query with no server-side throttle. Sustained at scale, this degrades DB performance for all concurrent users of the mirror node API.

    private void checkOwnerSpenderParamValidity(Bound ownerOrSpenderParams, Bound tokenParams) {
        if (ownerOrSpenderParams.isEmpty() && !tokenParams.isEmpty()) {
            throw new IllegalArgumentException("token.id parameter must have account.id present");
        }

        verifyRangeId(ownerOrSpenderParams);
        verifyRangeId(tokenParams);

        if (!ownerOrSpenderParams.hasLowerAndUpper()
                && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
            throw new IllegalArgumentException("Invalid range provided for %s".formatted(Constants.TOKEN_ID));
        }

        if (tokenParams.getCardinality(RangeOperator.LT, RangeOperator.LTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.LTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an lte or eq account.id parameter");
        }
        if (tokenParams.getCardinality(RangeOperator.GT, RangeOperator.GTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.GTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an gte or eq account.id parameter");
