### Title
Unauthenticated Sequential Dual-DB-Query DoS via `block_number` Filter on `/contracts/:contractId/results`

### Summary
The `getContractResultsById` handler unconditionally performs two sequential database round-trips when the `block_number` query parameter is supplied: one to resolve block timestamps via `RecordFileService.getRecordFileBlockDetailsFromIndex()` and a second to fetch the timestamp-bounded contract results. No authentication, rate limiting, or result caching exists on this code path, allowing any unauthenticated attacker to flood the endpoint with concurrent requests and exhaust the shared database connection pool, denying service to all users.

**Root cause and failed assumption:**
The design assumes that the overhead of two sequential DB queries per request is acceptable and that the connection pool is large enough to absorb concurrent load. Neither assumption holds under adversarial conditions. The two queries are strictly sequential (the second cannot begin until the first completes and its result is used to build the timestamp range), meaning each request holds a pool connection for the combined latency of both queries. The `contractResultsFilterValidityChecks` function (lines 273–279) only enforces that `block_number` uses the `eq` operator — it performs no rate limiting, no caching, and no concurrency control:
```js
const contractResultsFilterValidityChecks = (param, op, val) => {
  const ret = utils.filterValidityChecks(param, op, val);
  if (ret && param === filterKeys.BLOCK_NUMBER) {
    return op === queryParamOperators.eq;  // only operator check, nothing else
  }
  return ret;
};
```

**Why existing checks are insufficient:**
- Input validation only rejects non-`eq` operators on `block_number`; it does not throttle request rate.
- No caching layer exists for `RecordFileService.getRecordFileBlockDetailsFromIndex()`, so every request re-queries the DB even for the same block number.
- No authentication is required; the route is fully public.
- No concurrency limit or queue depth guard is applied at the handler level.

### Impact Explanation
Each concurrent attacker request occupies a DB connection for the duration of two sequential queries. With a typical Node.js connection pool of 10–20 connections, as few as 10–20 concurrent attacker requests can saturate the pool. All subsequent legitimate requests — including unrelated API endpoints sharing the same pool — will queue or fail with connection-pool-exhausted errors, causing a full REST API outage. Because the mirror node REST API is the primary data-access layer for Hedera ecosystem tooling (wallets, explorers, dApps), this constitutes a high-severity non-network DoS against infrastructure with significant market-cap exposure.

### Likelihood Explanation
The attack requires zero privileges, zero tokens, and zero on-chain activity. Any HTTP client capable of sending concurrent GET requests suffices. The `block_number` value can be any valid integer (e.g., `1`); the attacker does not need to know a valid contract ID — an invalid one simply causes `computeContractIdFromString` to throw before the DB calls, so the attacker uses any known valid contract ID (trivially obtained from the public mirror node). The attack is fully repeatable and scriptable, and the cost to the attacker is negligible (outbound HTTP bandwidth only).

### Recommendation
1. **Cache block-index lookups:** Memoize `RecordFileService.getRecordFileBlockDetailsFromIndex()` results (block index → timestamp range) with a short TTL (e.g., 5 s). Block data is immutable once finalized, so cache hit rates will be very high.
2. **Merge into a single query:** Rewrite the query builder to perform a single SQL query with a subquery or JOIN against the `record_file` table on `index = $N`, eliminating the first round-trip entirely.
3. **Apply per-IP rate limiting** on all `/contracts/` endpoints at the reverse-proxy or middleware layer.
4. **Isolate connection pools** per endpoint class so that a flood on one endpoint cannot starve others.

      req,
      response.results.length !== limit,
      {
        [filterKeys.TIMESTAMP]: lastContractResultTimestamp,

    );
