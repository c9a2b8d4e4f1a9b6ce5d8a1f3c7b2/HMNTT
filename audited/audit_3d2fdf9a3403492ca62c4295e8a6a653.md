### Title
Hardcoded `sslmode=disable` in `GetDsn()` Enables Undetectable MITM Injection of Transaction History

### Summary
`GetDsn()` in `rosetta/app/config/types.go` hardcodes `sslmode=disable` with no configuration field to override it, unlike the importer component which exposes a configurable `SslMode`. In any deployment where the Rosetta pod and the PostgreSQL database run on separate hosts (standard Kubernetes production topology), all DB wire traffic is transmitted in plaintext. An attacker with cluster-network access — requiring no application-level credentials — can intercept and inject arbitrary rows into query responses. Because the Rosetta persistence layer trusts the `transaction_hash` field returned by the DB without re-deriving it from transaction content, injected or reordered rows are returned to all Rosetta clients with no cryptographic detection.

### Finding Description

**Exact code location — hardcoded, non-configurable `sslmode=disable`:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
"host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
```
The `Db` struct (lines 39–47) has no `SslMode` field at all. Compare with the importer's `DBProperties` (`importer/src/main/java/org/hiero/mirror/importer/db/DBProperties.java`, line 50), which exposes `private SslMode sslMode = SslMode.DISABLE` as a configurable property. The Rosetta `Db` struct provides no equivalent — the disable is permanent in code.

**How the DSN is consumed:**

`rosetta/app/db/db.go`, line 18:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), &gorm.Config{...})
```
This is the sole connection path; there is no TLS upgrade after the fact.

**No cryptographic verification of returned data:**

`rosetta/app/persistence/transaction.go`, lines 32–59 (`selectTransactionsInTimestampRange`): the query selects `t.transaction_hash as hash` directly from the DB. Lines 77–79 (`getHashString`) hex-encode the raw bytes returned by the DB and use them verbatim as the Rosetta `TransactionIdentifier.Hash` (line 217). There is no re-computation of the hash from the transaction body fields. An injected row with a crafted `transaction_hash` and modified `crypto_transfers` amounts passes through `constructTransaction()` (lines 210–269) without any integrity check.

**Exploit flow:**

1. Attacker gains network-level access to the cluster (compromised pod, cloud-provider network tap, ARP poisoning on a flat cluster network — no DB credentials or application privileges required).
2. Attacker intercepts the unencrypted PostgreSQL wire protocol stream between the Rosetta pod and the DB.
3. Attacker injects or modifies rows in the response to `selectTransactionsInTimestampRangeOrdered` — e.g., altering `amount` fields in `crypto_transfers` JSON, swapping `transaction_hash` bytes, or inserting phantom transactions.
4. `FindBetween()` (lines 112–171) assembles the tampered rows into `types.Transaction` objects and returns them to all Rosetta API clients.
5. No layer in the call path detects the tampering.

### Impact Explanation

Every Rosetta client (exchanges, block explorers, reconciliation tools) receives a falsified transaction history. Transfer amounts, account IDs, and transaction hashes can all be silently altered. Because the Rosetta API is specifically used by exchanges for balance reconciliation, injected rows can cause exchanges to credit or debit incorrect amounts — a direct financial impact — without any on-chain evidence of the discrepancy. The `transaction_hash` field is the sole identifier used by Rosetta clients to deduplicate and reference transactions; spoofing it breaks all downstream integrity checks that rely on it.

### Likelihood Explanation

In the standard Kubernetes deployment (Helm chart at `charts/hedera-mirror-rosetta`), the Rosetta pod and the PostgreSQL pod are separate workloads communicating over the cluster network. The default DB host is `127.0.0.1` only in the all-in-one Docker image; in production Helm deployments the host is a separate service endpoint. Any attacker who has compromised a single unprivileged pod in the same namespace — or who can perform ARP/BGP-level interception on the cluster network — can execute this attack. Cloud-provider insider threat or a misconfigured network policy (the Helm chart does not enforce a `NetworkPolicy` restricting DB traffic) further lowers the bar. The attack is repeatable on every query.

### Recommendation

1. Add a `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a default of `require` (not `disable`).
2. Use the field in `GetDsn()`:
   ```go
   "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
   db.Host, db.Port, db.Username, db.Name, db.Password, db.SslMode,
   ```
3. Set the default to at minimum `require` (server certificate not verified) and document `verify-full` for production.
4. Add a Kubernetes `NetworkPolicy` to the Helm chart restricting PostgreSQL port access to only the Rosetta pod.
5. Consider adding application-level integrity checks: re-derive or verify the `transaction_hash` against the returned fields before trusting it.
