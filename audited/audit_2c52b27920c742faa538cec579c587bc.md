### Title
DB Connection Pool Exhaustion via Unauthenticated Concurrent Requests to `/topics/:topicId/messages`

### Summary
The `getTopicMessages()` handler unconditionally activates the `zeroRandomPageCostQueryHint` path in `queryQuietly()` for any request where `limit != 1` (the default limit is 25). This path acquires a **dedicated** pg pool client via `this.connect()` and holds it for the entire transaction duration. With a default pool size of 10 and no rate limiting on the REST Node.js API, an unauthenticated attacker sending 10 concurrent requests exhausts all pool connections, causing every subsequent DB query across all endpoints to queue or fail, rendering the mirror node unable to confirm transactions.

### Finding Description

**Code path:**

In `rest/topicmessage.js` line 142, `getTopicMessages()` sets:
```js
const queryHint = limit !== 1 ? constants.zeroRandomPageCostQueryHint : undefined;
```
Since the default `limit` is 25 (and the max is 100), virtually every real-world request triggers `queryHint != undefined`. This is passed to `getMessages()` → `pool.queryQuietly(query, params, preQueryHint)`.

In `rest/utils.js` lines 1518–1527, `queryQuietly()` branches on `preQueryHint`:
```js
if (!preQueryHint) {
  result = await this.query(query, params);   // pool manages connection internally
} else {
  client = await this.connect();              // acquires a DEDICATED client
  client.on('error', clientErrorCallback);
  await client.query(`begin; ${preQueryHint}`);
  result = await client.query(query, params);
  await client.query('commit');
}
```
The dedicated client is held until the `finally` block at line 1541–1544 calls `client.release()`. This means the connection is held for the full round-trip: `BEGIN` + hint + main query + `COMMIT`.

In `rest/dbpool.js` line 14, the pool is initialized with:
```js
max: config.db.pool.maxConnections,  // default: 10
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default: 20000ms
```

**Root cause:** The `preQueryHint` code path in `queryQuietly()` holds a dedicated pool connection for the entire transaction, rather than using the pool's auto-managed `pool.query()`. With a pool of only 10 connections and no rate limiting on the REST API, 10 concurrent requests saturate the pool. All subsequent calls to `pool.query()` or `pool.connect()` from any endpoint queue up waiting for a free connection, and after `connectionTimeoutMillis` (20 s) they throw a `DbError`.

**Why existing checks fail:**
- `statement_timeout = 20000ms` limits individual query duration but does not prevent an attacker from continuously cycling new requests to keep the pool saturated.
- `connectionTimeoutMillis = 20000ms` causes queued requests to fail after 20 s, but does not prevent the attacker from maintaining saturation.
- `server.js` has **no rate-limiting middleware** (no `express-rate-limit`, no per-IP throttle, no concurrency cap). The throttle configuration found in the codebase (`ThrottleConfiguration.java`) belongs exclusively to the `web3` Java service, not the REST Node.js API.
- The endpoint requires no authentication.

### Impact Explanation
When the pool is exhausted, every DB-dependent endpoint (transactions, accounts, balances, etc.) fails to acquire a connection. The mirror node cannot answer transaction-confirmation queries, effectively making it unable to confirm new transactions for the duration of the attack. Because Node.js is single-threaded and the pool queue grows unboundedly, the attacker can sustain the outage with a modest, continuous stream of requests.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no sophisticated tooling — only the ability to send HTTP GET requests. The default pool size of 10 makes the threshold trivially low. Any attacker with a script or tool like `ab`, `wrk`, or `curl --parallel` can reproduce this. The endpoint is publicly documented and accessible on mainnet/testnet mirror nodes.

### Recommendation
1. **Add rate limiting to the REST Node.js API** — apply `express-rate-limit` (or equivalent) globally and/or per-route, especially on `/topics/:topicId/messages`.
2. **Increase the default pool size** or document that operators must tune `hiero.mirror.rest.db.pool.maxConnections` relative to expected concurrency.
3. **Avoid holding dedicated pool clients for the hint path** — instead, consider using `SET LOCAL` within a single `pool.query()` call wrapped in a transaction, or use PostgreSQL session-level settings that don't require a dedicated client.
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) in front of the `queryQuietly` hint path to cap the number of simultaneous dedicated-client acquisitions.

### Proof of Concept

```bash
# Send 15 concurrent requests to exhaust the default pool of 10
# Replace <MIRROR_NODE_HOST> and <TOPIC_ID> with real values
for i in $(seq 1 15); do
  curl -s "http://<MIRROR_NODE_HOST>/api/v1/topics/<TOPIC_ID>/messages?limit=25" &
done
wait

# While the above is running, observe that all other endpoints fail:
curl -v "http://<MIRROR_NODE_HOST>/api/v1/transactions"
# Expected: connection timeout error or DbError after ~20 seconds
# Actual mirror node log: "error event emitted on pg pool" / DbError thrown
```

Reproducible steps:
1. Start the mirror node REST API with default configuration (`maxConnections=10`).
2. Identify any valid `topicId` (or use `0.0.1` — the endpoint does not require the topic to exist to trigger the DB path).
3. Fire 10+ concurrent GET requests to `/api/v1/topics/<topicId>/messages` (no `limit=1` parameter).
4. Simultaneously query `/api/v1/transactions` or any other DB-backed endpoint.
5. Observe that the second set of requests hangs for up to 20 seconds then returns a 500 `DbError`, confirming pool exhaustion.
