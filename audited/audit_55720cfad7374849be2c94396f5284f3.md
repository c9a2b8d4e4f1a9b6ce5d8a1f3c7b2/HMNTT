### Title
Unauthenticated HTTP Flood DoS on OpenAPI Spec Endpoint Bypassing All Middleware

### Summary
The `serveSpec` handler in `rest/middleware/openapiHandler.js` serves the full uncompressed OpenAPI YAML file to any caller with no rate limiting, no authentication, and no response caching. Because `serveSwaggerDocs(app)` is called at line 62 of `rest/server.js` before any middleware is registered, the swagger spec routes are inserted into Express's pipeline ahead of compression, Redis response caching, and any future rate-limiting middleware, meaning every flood request receives the full raw YAML payload and the server has no application-level mechanism to shed load.

### Finding Description

**Exact code path:**

`rest/server.js` line 62 calls `serveSwaggerDocs(app)` before any `app.use()` middleware: [1](#0-0) 

Inside `serveSwaggerDocs`, two routes are registered: [2](#0-1) 

Both routes point to `serveSpec`: [3](#0-2) 

`getV1OpenApiFile()` reads the YAML from disk once and stores it in the module-level `v1OpenApiFile` variable; every subsequent call returns the cached string immediately: [4](#0-3) 

**Root cause / failed assumption:**

Because `serveSwaggerDocs(app)` is invoked at `server.js:62` — before `app.use(compression())` (line 78), before `app.useExt(responseCacheCheckHandler)` (line 97), and before any rate-limiting middleware — Express executes the `serveSpec` route handler and calls `res.send()` (ending the response) before any of those middleware functions ever run for this path. The assumption that globally-registered middleware protects all routes fails here because route registration order determines execution order in Express. [5](#0-4) 

**Why existing checks are insufficient:**

- `authHandler` (line 86) is registered after the swagger routes and never runs for them.
- `responseCacheCheckHandler` (line 97) is registered after the swagger routes; even if it ran, `serveSpec` calls `res.send()` synchronously, so the cache middleware is never reached.
- `compression()` (line 78) is registered after the swagger routes, so responses are sent uncompressed, maximising per-response byte cost.
- No `express-rate-limit` or equivalent middleware exists anywhere in the REST server middleware stack. [6](#0-5) 

### Impact Explanation

An attacker opens as many concurrent TCP connections as the OS allows and continuously issues `GET /api/v1/<swaggerUIPath>/<specFileName>.yml`. Each request:
1. Hits no rate limiter.
2. Bypasses Redis caching (no cached short-circuit).
3. Receives the full uncompressed YAML body (compression middleware is skipped).

The server must maintain a socket write buffer per connection for the duration of each transfer. With a sufficiently large spec file and high concurrency, the aggregate of open socket buffers exhausts the Node.js process heap and/or the kernel's socket buffer pool, causing legitimate API requests to be dropped or the process to OOM-crash. Because the file content itself is in-memory (no disk I/O per request), the bottleneck is outbound bandwidth and socket-buffer memory, not CPU — making the attack cheap for the attacker and expensive for the server.

### Likelihood Explanation

No special privileges, credentials, or knowledge beyond the public API URL are required. The endpoint is documented and discoverable via the Swagger UI path. A single attacker with a modest connection pool (e.g., `wrk`, `ab`, or a simple script) can sustain thousands of requests per second. The attack is trivially repeatable and requires no state.

### Recommendation

1. **Move `serveSwaggerDocs(app)` after all middleware registrations** in `rest/server.js` so that compression, caching, and any rate-limiting middleware apply to swagger routes.
2. **Add a per-IP rate limiter** (e.g., `express-rate-limit`) specifically for the swagger/spec paths, or apply a global rate limiter before all routes.
3. **Set a `Cache-Control: public, max-age=<N>` header** in `serveSpec` so that reverse proxies and CDNs can absorb repeated requests without hitting the origin.
4. **Enable compression** for the spec response (moving the route registration after `app.use(compression())` achieves this automatically).

### Proof of Concept

```bash
# Prerequisites: know the public base URL and default swagger path
# Default paths from config: swaggerUIPath = "docs", specFileName = "openapi"

TARGET="https://<mirror-node-host>/api/v1/docs/openapi.yml"

# Flood with 500 concurrent connections, unlimited requests
wrk -t 10 -c 500 -d 60s "$TARGET"

# Observable effect:
# - Server memory climbs as socket buffers accumulate
# - Legitimate /api/v1/transactions requests begin timing out
# - No 429 or 401 responses are returned for the spec endpoint
# - Process may OOM or become unresponsive under sustained load
```
