### Title
No Rate Limiting on GraphQL Endpoint Enables Sustained CPU Exhaustion via Maximum-Complexity Queries

### Summary
The `graphQlCustomizer()` method in `GraphQlConfiguration.java` configures a `MaxQueryComplexityInstrumentation(200)` limit but installs **no rate limiting** on the GraphQL endpoint. An unprivileged attacker can continuously submit queries at exactly complexity 200 — each triggering multiple database resolver chains — without any throttling mechanism stopping them. The `CachedPreparsedDocumentProvider` further amplifies this by eliminating parsing/validation overhead on repeated identical queries, leaving only resolver execution (DB I/O + CPU) per request.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `graphQlCustomizer()`, lines 42–49:

```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);

return b -> b.configureGraphQl(
        graphQL -> graphQL.instrumentation(instrumentation).preparsedDocumentProvider(provider));
```

No rate-limiting bean, filter, or interceptor exists anywhere in the `graphql/` module (confirmed: grep for `throttle`, `rateLimit`, `rate-limit`, `requestsPerSecond` across all GraphQL Java sources returns zero results). The `web3/` module has `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j, but this is a completely separate service.

**Root cause:** The design assumes `MaxQueryComplexityInstrumentation` is sufficient to bound server load. It bounds work *per query* but places no bound on *query rate*. An attacker can issue maximum-complexity queries at line rate.

**Exploit flow:**

1. The schema (`account.graphqls`, `common.graphqls`) exposes nested `Account` fields: `autoRenewAccount: Account`, `stakedAccount: Account`, `obtainer: Accountable` — each of which is itself an `Account` with the same nested fields, resolvable up to depth 10 (`MaxQueryDepthInstrumentation(10)`).
2. An attacker crafts a query that fills complexity budget 200 using these nested resolver chains, e.g., deeply nesting `autoRenewAccount { autoRenewAccount { stakedAccount { ... } } }` combined with all scalar fields. Each nested `Account` field triggers a separate `EntityRepository` lookup (via `EntityServiceImpl.getByIdAndType()` → `entityRepository.findById()`).
3. The attacker sends this query repeatedly. On the **first** request, `CachedPreparsedDocumentProvider` parses and validates it (including the complexity check). On **every subsequent** request with the same query string, the cached `PreparsedDocumentEntry` is returned immediately — skipping parsing, validation, and the complexity instrumentation overhead — and execution goes straight to resolver chains and DB queries.
4. Because there is no rate limit, the attacker can sustain this at whatever rate the network and connection pool allow.

### Impact Explanation

Each maximum-complexity request drives multiple synchronous JPA/JDBC calls through `EntityRepository` (backed by a HikariCP connection pool). Sustained high-rate requests saturate DB connections and keep JVM threads busy, raising CPU above baseline. The `GraphQLHighCPU` alert in `charts/hedera-mirror-graphql/values.yaml` only fires after CPU exceeds 80% for 5 minutes — well above the 30% threshold the attacker needs to sustain. There is no circuit breaker or back-pressure mechanism in the GraphQL module to shed load before that alert fires.

### Likelihood Explanation

Preconditions: none. The `/graphql/alpha` endpoint is unauthenticated and publicly reachable (confirmed by Postman collection). The attacker needs only a single valid query string (trivially discoverable via GraphQL introspection or the public schema files). The attack is fully repeatable, requires no special tooling beyond `curl` or `ab`, and the caching behavior means the attacker's per-request cost is lower than the server's per-request cost.

### Recommendation

1. **Add request-rate limiting to the GraphQL module** using the same bucket4j pattern already used in `web3/` (`ThrottleConfiguration`, `ThrottleManagerImpl`). Apply it as a `WebGraphQlInterceptor` or a servlet filter on `/graphql/**`, keyed per source IP.
2. **Assign non-uniform field complexity weights** to `MaxQueryComplexityInstrumentation` — nested `Account` resolver fields (`autoRenewAccount`, `stakedAccount`, `obtainer`) should carry a higher weight (e.g., 10–20) than scalar fields, so the effective number of DB-triggering fields per request is much lower than 200.
3. **Add a query timeout** at the datasource/JPA level to bound the wall-clock cost of any single request.
4. Consider **disabling GraphQL introspection** in production to slow schema discovery.
