### Title
EVM Address `ownerId` Causes Extra DB Lookup Per Request, Enabling Unauthenticated DB Load Amplification

### Summary
The `getHookStorage()` endpoint at `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage` accepts EVM-address-format `ownerId` values (e.g., `0x<40 hex chars>`) from any unauthenticated caller. When such a value is supplied, `EntityServiceImpl.lookup()` executes an additional `findByEvmAddress()` database query before the main storage query, whereas a numeric `ownerId` incurs zero extra DB queries in `lookup()`. An attacker can exploit this asymmetry to approximately double per-request DB load with no privileges required.

### Finding Description
**Code path:**

1. `HooksController.java` line 105–130: `getHookStorage()` accepts `@PathVariable EntityIdParameter ownerId`. No authentication or rate-limiting is applied.
2. `EntityIdParameter.valueOf()` (`EntityIdParameter.java` lines 17–19) parses `0x<40 hex chars>` as `EntityIdEvmAddressParameter` — a fully valid, accepted input type.
3. The `ownerId` is stored in `HookStorageRequest` and passed to `HookServiceImpl.getHookStorage()`.
4. `HookServiceImpl.java` line 61: `entityService.lookup(request.getOwnerId())` is called unconditionally.
5. `EntityServiceImpl.java` lines 31–35: the `switch` dispatches on the runtime type:
   - `EntityIdNumParameter` → `Optional.of(p.id())` — **zero DB queries**
   - `EntityIdEvmAddressParameter` → `entityRepository.findByEvmAddress(p.evmAddress())` — **one extra DB query**
6. After `lookup()` resolves, the main storage repository query executes regardless.

**Root cause:** The design assumes all `EntityIdParameter` subtypes are equivalent in cost, but `EntityIdEvmAddressParameter` incurs a full DB lookup while `EntityIdNumParameter` does not. There is no caching, deduplication, or restriction on which format external callers may use.

**Why existing checks fail:** The `@PathVariable` binding and `EntityIdParameter.valueOf()` parsing intentionally accept EVM addresses as valid input. No authentication guard, rate limiter, or format restriction is present on this endpoint.

### Impact Explanation
Every request using an EVM-address `ownerId` triggers two DB queries (EVM lookup + storage query) versus one for a numeric `ownerId`. An attacker sending a sustained flood of EVM-address requests can increase DB query throughput by ~100% relative to equivalent numeric requests — well above the 30% threshold. This can degrade database performance for all users of the mirror node without any privileged access.

### Likelihood Explanation
The exploit requires zero authentication, zero special knowledge beyond the public API path, and is trivially repeatable with a simple HTTP client. The EVM address format is publicly documented and accepted by design. Any external actor can automate this at scale.

### Recommendation
- In `EntityServiceImpl.lookup()`, add a short-lived cache (e.g., Caffeine/Guava) keyed on EVM address bytes to avoid repeated DB hits for the same address.
- Alternatively, apply rate limiting per IP/client on the `/api/v1/accounts/{ownerId}/hooks` endpoint family.
- Consider restricting `ownerId` path variables to numeric (`shard.realm.num`) format only on endpoints where EVM address resolution is not functionally required.

### Proof of Concept
```
# Step 1: Send repeated requests with EVM-address ownerId
for i in $(seq 1 1000); do
  curl -s "http://<mirror-node>/api/v1/accounts/0xaabbccddee112233445566778899aabbccddeeff/hooks/1/storage" &
done

# Each request triggers:
#   1. entityRepository.findByEvmAddress(...)   ← extra DB query
#   2. hookStorageRepository.findBy...()        ← main DB query

# Compare with numeric ownerId (only 1 DB query per request):
curl "http://<mirror-node>/api/v1/accounts/0.0.12345/hooks/1/storage"
```

The EVM-address variant doubles DB query count per request with no privileges, no brute force, and no special preconditions.
