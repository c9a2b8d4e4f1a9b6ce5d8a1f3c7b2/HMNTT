### Title
Connection Pool Exhaustion DoS via Concurrent `GET /accounts/:id` Requests in `getOneAccount()`

### Summary
`getOneAccount()` in `rest/accounts.js` unconditionally fires an entity DB query and a transactions DB query concurrently via `Promise.all`, consuming 2 pool connections simultaneously per request. With the default pool size of 10 connections and no rate limiting on the Node.js REST API, an unauthenticated attacker sending as few as 5 concurrent requests can exhaust the pool, causing all subsequent requests to queue and time out after 20 seconds, rendering the REST API unresponsive.

### Finding Description
**Code path:** `rest/accounts.js`, `getOneAccount()`, lines 487–495.

```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);          // line 487 — acquires 1 connection
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)             // line 492 — acquires 1 connection via getTransactionTimestamps()
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]); // line 495
```

`doGetTransactions()` (`rest/transactions.js:695`) calls `getTransactionTimestamps()` which immediately issues `pool.queryQuietly()` (line 483). Because `entityPromise` and `transactionsPromise` are both started before `await Promise.all`, both DB queries are in-flight simultaneously, each holding a pool connection for the duration of their SQL execution.

**Root cause / failed assumption:** The design assumes the pool is large enough to absorb concurrent in-flight queries across all simultaneous HTTP requests. The default `maxConnections` is **10** (documented at `docs/configuration.md:556`). With 2 connections consumed per request at peak, only 5 concurrent HTTP requests are needed to saturate the pool.

**Why timestamp filters amplify the attack:** When `timestampFilters.length > 0`, the entity query becomes a `UNION ALL` of `entity` and `entity_history` (lines 210–219 of `accounts.js`), and `getAccountBalanceTimestampRange` adds a prior `await` DB call (line 438). This makes each query more expensive, holding connections longer and widening the exhaustion window.

**No rate limiting:** The Express middleware stack (`rest/server.js:67–98`) registers only `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and `responseCacheCheckHandler`. There is no per-IP or global request-rate-limiting middleware for the Node.js REST API. The `maxRepeatedQueryParameters` config (default 100) only limits repeated query params within a single request, not concurrent HTTP connections.

**Pool behavior under exhaustion:** `node-postgres` queues connection requests when the pool is full. `connectionTimeoutMillis` defaults to 20,000 ms (`rest/dbpool.js:13`). Queued requests wait up to 20 seconds before throwing a timeout error, during which the REST API is effectively unresponsive to all users.

### Impact Explanation
An attacker can make the mirror node REST API completely unresponsive to all clients by sustaining ~5–10 concurrent requests to `GET /api/v1/accounts/:id`. All legitimate API consumers (wallets, explorers, dApps) relying on the mirror node for account state, transaction history, and balance queries will receive errors or 20-second timeouts. The impact is a full denial-of-service of the mirror node REST API. Note: this does not affect Hedera consensus node transaction processing directly, but it does prevent any client from querying confirmed transaction state via the mirror node.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and no special knowledge beyond the public API endpoint. A single attacker with a basic HTTP client (e.g., `curl`, `ab`, `wrk`) can sustain the attack indefinitely. The attack is trivially repeatable and scriptable. The default pool size of 10 makes the threshold extremely low (5 concurrent requests). Production deployments may increase `maxConnections`, but without rate limiting, a proportionally larger flood achieves the same result.

### Recommendation
1. **Add per-IP and global rate limiting middleware** to the Express REST API (e.g., `express-rate-limit`) before route handlers, specifically targeting expensive endpoints like `/accounts/:id`.
2. **Increase default `maxConnections`** from 10 to a value appropriate for expected concurrency (e.g., 50–100), and document the relationship between pool size and concurrent request capacity.
3. **Serialize the two queries** within `getOneAccount()` rather than running them concurrently, or use a single connection/transaction for both, reducing peak connection consumption to 1 per request.
4. **Add a concurrency limiter** (e.g., `p-limit`) around `getOneAccount()` to cap the number of simultaneously executing handler invocations.
5. **Deploy an API gateway or reverse proxy** (e.g., Traefik with `inFlightReq` middleware, as already done for the Rosetta API at `charts/hedera-mirror-rosetta/values.yaml:152–155`) in front of the REST API.

### Proof of Concept
```bash
# Exhaust the default 10-connection pool with 6 concurrent requests
# Each request uses 2 connections simultaneously via Promise.all
for i in $(seq 1 6); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/0.0.1000?timestamp=gte:0&timestamp=lte:9999999999" &
done
wait

# All subsequent requests will queue and time out after 20 seconds:

# Expected: 20-second hang followed by connection timeout error
```
