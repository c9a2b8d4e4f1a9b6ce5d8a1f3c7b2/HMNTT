### Title
Unauthenticated Unbounded Pagination on `/api/v1/blocks` Enables DB Connection Exhaustion and Importer Starvation

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` applies no application-level rate limiting and requires no authentication. Any external client can issue `GET /api/v1/blocks?block.number=gte:0` and follow the `next` link produced by `generateNextLink()` indefinitely, issuing one DB query per page. With many concurrent clients doing this simultaneously, the finite REST API connection pool is saturated, starving the importer of database connections and halting ingestion of new blocks from the live network.

### Finding Description

**Exact code path:**

`rest/controllers/blockController.js`, `getBlocks()` (lines 101–112) and `generateNextLink()` (lines 90–99):

```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);   // DB query, no rate gate
  res.locals[responseDataLabel] = {
    blocks: blocks.map((model) => new BlockViewModel(model)),
    links: { next: this.generateNextLink(req, blocks, formattedFilters) },
  };
};

generateNextLink = (req, blocks, filters) => {
  return blocks.length
    ? utils.getPaginationLink(req, blocks.length !== filters.limit,
        {[filterKeys.BLOCK_NUMBER]: last(blocks).index}, filters.order)
    : null;
};
```

`generateNextLink` returns a non-null link whenever `blocks.length === filters.limit`, i.e., as long as there are more pages. Starting from `block.number=gte:0` with the default `asc` order, every page of a chain with millions of blocks produces a valid next link.

**Root cause and failed assumption:**

The design assumes that infrastructure-level throttling (Traefik middleware) is always present. However, the Traefik middleware is only rendered when `global.middleware` is `true` in the Helm values — it is opt-in, not default. The application itself has zero rate-limiting logic for the `/api/v1/blocks` path. The `authHandler.js` only sets a custom per-user limit for *authenticated* users; unauthenticated requests receive the full `defaultLimit` with no per-IP or global request-rate cap.

The `extractLimitFromFilters` function (lines 57–61) also contains a logic error: when `limit.value > maxLimit` it falls back to `defaultLimit` rather than clamping to `maxLimit`, but this does not reduce the attack surface.

**DB connection sharing:**

`rest/dbpool.js` (lines 7–16) creates a `pg.Pool` with `max: config.db.pool.maxConnections`. The REST API uses the `mirror_rest` database user; the importer uses `mirror_importer`. Both share the same PostgreSQL instance. In the production Helm chart (`charts/hedera-mirror/values.yaml`, lines 371–373), `mirror_rest` is allocated `max_user_connections: 250` at the pgbouncer layer, while PostgreSQL itself is configured with `max_connections: 200` (line 210). The importer uses `pool_mode: session` (line 368), meaning it holds connections for the full duration of each write batch. If the REST pool saturates the available PostgreSQL connections, the importer's connection attempts queue or fail, halting block ingestion.

### Impact Explanation

When the importer cannot acquire a database connection it cannot persist new record files / blocks. The mirror node's view of the chain freezes at the last successfully written block. Any downstream consumer (ETH JSON-RPC, Rosetta, gRPC) serves stale data. On a mainnet mirror node processing ~1 block/second, even a 60-second stall produces a visible gap. Sustained attack causes indefinite partition from the live network without any on-chain action or privileged access.

### Likelihood Explanation

The attack requires:
- No credentials
- No special knowledge beyond the public OpenAPI spec (`/api/v1/blocks` is documented)
- A script that follows the `links.next` URL in each response

A single attacker with ~250 concurrent HTTP clients (trivially achievable with `async`/`await` loops or `ab`/`wrk`) can saturate the REST connection pool. The attack is stateless, repeatable, and survives server restarts. The absence of application-level rate limiting means no 429 responses are ever returned to the attacker.

### Recommendation

1. **Application-level rate limiting**: Add a per-IP token-bucket middleware (e.g., `express-rate-limit`) to the REST API, applied before route handlers, covering all list endpoints including `/api/v1/blocks`.
2. **Connection pool isolation**: Give the importer a reserved connection quota at the pgbouncer level (`min_pool_size` / `reserve_pool_size`) so REST traffic cannot starve it even if the REST pool is fully saturated.
3. **Pagination depth limit**: Introduce a maximum cursor depth or require a tighter upper bound on `block.number` range so a single client cannot traverse the entire chain without re-anchoring.
4. **Make Traefik middleware non-optional**: The `inFlightReq` and `rateLimit` middleware entries should be enabled by default in the REST chart values rather than gated on `global.middleware`.

### Proof of Concept

```bash
# Step 1: fetch first page (no auth required)
NEXT=$(curl -s "https://<mirror-node>/api/v1/blocks?block.number=gte:0&order=asc" \
  | jq -r '.links.next')

# Step 2: spawn N concurrent paginators
for i in $(seq 1 250); do
  (while [ -n "$NEXT" ]; do
     NEXT=$(curl -s "https://<mirror-node>${NEXT}" | jq -r '.links.next')
   done) &
done
wait

# Step 3: observe importer lag
# Monitor: SELECT index FROM record_file ORDER BY index DESC LIMIT 1;
# The returned index will stop advancing while the attack runs.
```

Each of the 250 concurrent goroutines holds an open DB query for the duration of its page fetch. With `max_user_connections: 250` for `mirror_rest` and PostgreSQL `max_connections: 200`, the importer's connection attempts will be rejected or queued indefinitely, halting block writes.
