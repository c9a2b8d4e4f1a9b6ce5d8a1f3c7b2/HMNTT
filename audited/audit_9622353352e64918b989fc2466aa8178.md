### Title
Connection Pool Exhaustion via Unbounded Socket Blocking on `findByAlias` During Network Partition

### Summary
`EntityRepository.findByAlias` issues a native SQL query with no JDBC-level socket timeout configured for the GraphQL module's HikariCP datasource. During a network partition, JDBC driver threads block indefinitely on the socket read, holding pool connections. An unprivileged attacker sending requests at a rate exceeding the pool size during a partition can exhaust all connections, rendering the service unable to serve any database-backed request until the partition resolves.

### Finding Description
**Exact code location:** `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–14:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
```

No `@QueryHints`, `@Lock`, or any application-level timeout annotation is present on this method.

**Root cause and failed assumption:** The deployment sets a PostgreSQL-level `statement_timeout = 10000` for the `mirror_graphql` user (`charts/hedera-mirror/templates/secret-passwords.yaml`, line 122). The failed assumption is that this server-side timeout protects against hung connections. It does not: `statement_timeout` is enforced by the PostgreSQL backend process, which sends a cancellation message over the TCP connection. During a network partition, that TCP connection is broken; the cancellation message never reaches the JDBC driver, and the driver thread remains blocked indefinitely on `socket.read()`.

The GraphQL module's datasource is constructed in `common/src/main/java/org/hiero/mirror/common/CommonConfiguration.java` (lines 66–95) from `spring.datasource.hikari` properties. No `socketTimeout` JDBC URL parameter is configured anywhere in the GraphQL module's resources or configuration files (confirmed by exhaustive search of `graphql/**`). The `socketTimeout` present in `CommonProperties.DatabaseStartupProperties` (line 96) is used only by `DatabaseWaiter` during startup validation, not for runtime query connections.

**Exploit flow:**
1. A network partition occurs between the GraphQL service and PostgreSQL (attacker does not need to cause this; natural partitions occur in cloud environments).
2. Attacker sends concurrent GraphQL queries that resolve to `findByAlias` (e.g., account lookup by alias — a public, unauthenticated endpoint).
3. Each request acquires a HikariCP connection and blocks indefinitely on `socket.read()` because no JDBC `socketTimeout` is set.
4. Once all pool connections are held by blocked threads, HikariCP's `connectionTimeout` (default 30 seconds) causes every subsequent request to fail with `SQLTransientConnectionException`.
5. The service is completely unable to serve any database-backed GraphQL query for the entire duration of the partition.

### Impact Explanation
The GraphQL service becomes fully unavailable for all users (not just the attacker) for the duration of the network partition. All HikariCP connections are consumed by threads blocked on dead sockets. The service cannot self-recover until either the partition resolves (OS TCP keepalive detects the dead connection, typically after hours with default kernel settings) or the process is restarted. This is a complete denial of service of the GraphQL API.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required. An attacker needs only to: (a) observe or wait for a network partition event (common in cloud/Kubernetes environments during node failures, rolling restarts, or AZ outages), and (b) send concurrent requests to any alias-based account lookup. The number of concurrent requests needed equals the HikariCP pool size (typically 10 by default). This is trivially achievable with standard HTTP tooling (`curl`, `ab`, `wrk`). The attack is repeatable on every partition event.

### Recommendation
1. **Add JDBC `socketTimeout`** to the GraphQL module's datasource JDBC URL (e.g., `?socketTimeout=15` for 15 seconds). This causes the PostgreSQL JDBC driver to raise a `SocketTimeoutException` if no data is received within the timeout, releasing the connection back to the pool.
2. **Configure HikariCP `keepaliveTime`** to detect dead connections proactively and evict them before the pool is exhausted.
3. **Add `@QueryHints({@QueryHint(name = "jakarta.persistence.query.timeout", value = "10000")})` on `findByAlias`** as a defense-in-depth measure (though this alone does not protect against network partitions).
4. **Configure OS-level TCP keepalive** (`tcp_keepalive_time`, `tcp_keepalive_intvl`, `tcp_keepalive_probes`) at the pod/node level to detect dead connections within seconds rather than hours.

### Proof of Concept
```bash
# 1. Simulate network partition (on the DB host or via iptables on the app pod):
iptables -I OUTPUT -p tcp --dport 5432 -j DROP

# 2. Flood the GraphQL endpoint with alias-based queries (no auth required):
for i in $(seq 1 20); do
  curl -s -X POST http://<graphql-host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{alias:\"AAEC\"}){balance}}"}' &
done
wait

# 3. Observe: first ~10 requests block indefinitely (pool exhausted).
#    Subsequent requests fail with connection pool timeout after 30s.
#    Service is unavailable for all users until partition resolves.

# 4. Restore network (partition ends):
iptables -D OUTPUT -p tcp --dport 5432 -j DROP
# Blocked threads eventually unblock; service recovers.
