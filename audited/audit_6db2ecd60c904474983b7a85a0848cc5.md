### Title
Unbounded Cross-Parameter Topic IN Clause Allows DB Resource Exhaustion via `getContractLogsById()`

### Summary
The `getContractLogsById()` handler enforces `maxRepeatedQueryParameters` (default: 100) **per individual query key**, not as a total across all four topic parameters. An unauthenticated attacker can supply 100 values each for `topic0`, `topic1`, `topic2`, and `topic3`, causing `updateQueryFiltersWithInValues()` to build four separate IN clauses totalling 400 bind parameters. Because the query is structured as a UNION of up to three sub-queries, each sub-query repeats all four IN conditions, amplifying the load on the database.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 16 routes `GET /:contractId/results/logs` to `ContractController.getContractLogsById`. [1](#0-0) 

`getContractLogsById()` calls `extractContractIdAndFiltersFromValidatedRequest(req, acceptedContractLogsByIdParameters)`, which calls `utils.buildAndValidateFilters`. [2](#0-1) 

`buildFilters()` in `utils.js` checks `isRepeatedQueryParameterValidLength(values)` — defined as `values.length <= config.query.maxRepeatedQueryParameters` — **per key only**: [3](#0-2) [4](#0-3) 

The default value of `maxRepeatedQueryParameters` is **100**: [5](#0-4) 

In `extractContractLogsMultiUnionQuery()`, each topic's `eq` values are accumulated into per-topic `inValues` arrays with no cross-topic total cap: [6](#0-5) 

`updateQueryFiltersWithInValues()` then emits one `cl.topicN in ($x, ..., $y)` condition per topic: [7](#0-6) [8](#0-7) 

**Root cause:** The per-key limit of 100 is applied independently to each of `topic0`, `topic1`, `topic2`, `topic3`. There is no aggregate cap. An attacker can legally supply 4 × 100 = 400 IN-clause values in a single request. The UNION query structure (lower / inner / upper sub-queries) then repeats all four IN conditions in each branch, multiplying the parameter evaluation work.

**Why existing checks fail:**

The `qs` parser's `arrayLimit` / `throwOnLimitExceeded` guard in `requestHandler.js` also operates per-key: [9](#0-8) 

`checkTimestampsForTopics()` requires a timestamp range but does not limit the number of topic values: [10](#0-9) 

The `acceptedContractLogsByIdParameters` set does not impose any per-topic count restriction: [11](#0-10) 

### Impact Explanation
Each crafted request forces PostgreSQL to evaluate up to 400 IN-clause bind parameters across four columns, repeated in up to three UNION branches. With a database connection pool capped at 10 connections and a statement timeout of 20 seconds, an attacker sending ~10 concurrent such requests can saturate the pool for up to 20 seconds per wave, causing legitimate API requests to queue or fail. The `contract_log` table can be large; even with the required timestamp range (max 7 days), scanning with 400 topic filters is significantly more expensive than a normal query. [12](#0-11) 

### Likelihood Explanation
No authentication is required. The endpoint is publicly accessible. The exploit requires only a crafted HTTP GET request with repeated query parameters — trivially scriptable. The attacker can repeat the attack in a loop. The 20-second statement timeout provides some protection but does not prevent connection pool exhaustion during the timeout window.

### Recommendation
1. Add a **total topic-value cap** across all four topic keys combined (e.g., max 25 total topic `eq` values per request) inside `extractContractLogsMultiUnionQuery()`, throwing `InvalidArgumentError` if exceeded.
2. Alternatively, reduce `maxRepeatedQueryParameters` specifically for topic parameters by introducing per-key limits in `acceptedContractLogsByIdParameters` rather than relying on the global setting.
3. Consider adding rate limiting at the API gateway or middleware layer for this endpoint.

### Proof of Concept

```
GET /api/v1/contracts/0.0.1234/results/logs?timestamp=gte:1700000000&timestamp=lte:1700604800\
  &topic0=0xAAAA...01&topic0=0xAAAA...02&...&topic0=0xAAAA...64\   # 100 values
  &topic1=0xBBBB...01&topic1=0xBBBB...02&...&topic1=0xBBBB...64\   # 100 values
  &topic2=0xCCCC...01&topic2=0xCCCC...02&...&topic2=0xCCCC...64\   # 100 values
  &topic3=0xDDDD...01&topic3=0xDDDD...02&...&topic3=0xDDDD...64    # 100 values
```

Each request passes all per-key validation checks (≤100 per key), generates a SQL UNION query with four `IN (…100 values…)` conditions in each branch, and ties up a DB connection for the duration of the statement timeout. Sending 10 such requests concurrently exhausts the connection pool.
