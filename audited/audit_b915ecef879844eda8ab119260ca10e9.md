### Title
Historical Hook Storage Key Range Filter Bypass via Raw Key Collection in `getHookStorageChange()`

### Summary
In `HookServiceImpl.getHookStorageChange()`, when a historical (timestamped) query includes explicit keys, the code correctly computes `keysInRange` (keys filtered to `[keyLowerBound, keyUpperBound]`) but then passes the raw, unfiltered `keys` collection to `HookStorageChangeRepository.findByKeyInAndTimestampBetween()`. The SQL query for that method contains no key-range predicate, only `key in (:keys)`. An unprivileged caller who includes at least one in-range key (to bypass the empty-guard) alongside out-of-range keys will receive historical storage change records for keys outside the declared range.

### Finding Description

**Root cause — wrong variable passed at line 104:**

In `HookServiceImpl.java` `getHookStorageChange()`:

```
89:  final var keys        = request.getKeys();          // raw user input
90:  final boolean requestHasKeys = !keys.isEmpty();
91:  final var keysInRange = request.getKeysInRange();   // filtered to [lower,upper]

93:  if (keysInRange.isEmpty() && requestHasKeys) {      // guard: ALL keys out-of-range → empty
94:      return new HookStorageResult(ownerId, List.of());
95:  }
...
103: if (requestHasKeys) {
104:     changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
105:             ownerId.getId(), hookId, keys, ...);     // ← BUG: `keys`, not `keysInRange`
``` [1](#0-0) 

The guard on line 93 only fires when **every** supplied key is outside the range. As soon as one key is in-range, `keysInRange` is non-empty, the guard is skipped, and the raw `keys` list (containing out-of-range entries) is forwarded to the repository.

**SQL query has no key-range predicate:**

```sql
where owner_id = :ownerId
  and hook_id  = :hookId
  and key in (:keys)                          -- no >= / <= bounds
  and consensus_timestamp between ...
``` [2](#0-1) 

**Contrast with the non-historical path**, which correctly uses `keysInRange`:

```
72: final var keysInRange = request.getKeysInRange();
78: hookStorageRepository.findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse(
79:         ownerId.getId(), request.getHookId(), keysInRange, page);
``` [3](#0-2) 

**`getKeysInRange()` correctly filters:**

```java
return keys.stream()
    .filter(key -> Arrays.compareUnsigned(key, keyLowerBound) >= 0
                && Arrays.compareUnsigned(key, keyUpperBound) <= 0)
    .toList();
``` [4](#0-3) 

The filtering logic exists and works; it is simply not used on the historical code path.

### Impact Explanation
A caller can retrieve historical `hook_storage_change` rows for arbitrary storage keys belonging to a given `(ownerId, hookId)` pair, as long as they know at least one legitimate in-range key. This means a smart contract hook can be made to observe storage values at a past timestamp for slots it was never supposed to read, violating the access-control invariant of the key-range parameter. Depending on what is stored (e.g., sensitive state, balances, secrets written to hook storage), this constitutes an unauthorized information-disclosure at the protocol layer. No funds need to be directly at risk for this to be a medium-severity protocol bug — the hook's view of historical state is corrupted/expanded beyond its declared scope.

### Likelihood Explanation
Any unauthenticated HTTP client that can call the hook-storage historical endpoint can trigger this. The attacker needs only:
1. A valid `(ownerId, hookId)` — publicly enumerable from the mirror node.
2. One known in-range key (to pass the guard).
3. Arbitrary additional keys to leak.

No special privileges, no on-chain transactions, and no rate-limiting specific to this path are required. The exploit is deterministic and repeatable.

### Recommendation
Replace `keys` with `keysInRange` at line 104 of `HookServiceImpl.java`:

```java
// Before (buggy):
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);

// After (fixed):
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keysInRange, timestampLowerBound, timestampUpperBound, page);
```

This mirrors the already-correct non-historical branch and ensures only keys within `[keyLowerBound, keyUpperBound]` are ever forwarded to the database.

### Proof of Concept

**Preconditions:**
- Mirror node REST-Java service is running.
- `ownerId=1`, `hookId=42` exist with historical storage changes at keys `0x0A` (in-range) and `0xFF` (out-of-range, e.g., a sensitive slot).

**Steps:**

1. Issue a historical hook-storage query with a range `[0x00, 0x0F]` and an explicit key list containing both `0x0A` (in-range) and `0xFF` (out-of-range):

```
GET /api/v1/accounts/1/hooks/42/storage
    ?key=0x0A&key=0xFF
    &key.gte=0x00&key.lte=0x0F
    &timestamp=lte:1700000000000000000
```

2. `getKeysInRange()` returns `[0x0A]` (non-empty → guard at line 93 is skipped).
3. `findByKeyInAndTimestampBetween` is called with `keys = [0x0A, 0xFF]`.
4. The SQL returns rows for both `0x0A` and `0xFF`.

**Expected (correct) result:** Only the row for `0x0A` is returned.
**Actual (buggy) result:** Rows for both `0x0A` and `0xFF` are returned, leaking the historical value of the out-of-range key `0xFF`.
