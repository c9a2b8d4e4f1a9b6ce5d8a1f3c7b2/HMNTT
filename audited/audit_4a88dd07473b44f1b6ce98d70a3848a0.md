### Title
Unbounded Throttled Polling Loop via `isComplete()` Bypass on High-Volume Topics

### Summary
In `PollingTopicMessageRetriever`, when `throttled=true` (always set server-side for historical retrieval), `isComplete()` only terminates the polling loop when the last page returned fewer results than `maxPageSize`. Any unprivileged user can subscribe to a topic with a continuously growing message backlog — ensuring every poll returns exactly `maxPageSize` results — causing `isComplete()` to perpetually return `false` with `numRepeats = Long.MAX_VALUE`, keeping the loop alive indefinitely. The only guard (a 60-second idle timeout) is neutralized because messages keep flowing, and there are no per-user subscription limits.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` always calls the retriever with `throttled=true`: [1](#0-0) 

Inside `PollingContext`, when `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE`: [2](#0-1) 

The repeat predicate is `!context.isComplete()`: [3](#0-2) 

`isComplete()` for the throttled path: [4](#0-3) 

**Root cause:** The termination condition `pageSize.get() < retrieverProperties.getMaxPageSize()` assumes that a full page means "more data exists." On a topic that continuously receives messages at a rate that keeps each DB query returning exactly `maxPageSize` (default 1000) rows, this assumption is permanently false. Since `numRepeats = Long.MAX_VALUE` and no limit is required from the client, the loop never exits.

**Why the timeout fails:** The timeout is a Reactor idle timeout — it fires only if no item is emitted within the window: [5](#0-4) 

With 1000 messages emitted every 2 seconds (default `pollingFrequency`), the 60-second idle timer resets continuously and never fires.

**Exploit flow:**
1. Attacker creates or identifies a topic with a large and/or continuously growing message backlog (e.g., a topic receiving >1000 messages per 2-second poll interval).
2. Attacker opens a gRPC `subscribeTopic` call with `startTime=0` and no `limit` field set.
3. Server calls `retrieve(filter, true)` — `throttled=true` is hardcoded.
4. Each poll returns exactly `maxPageSize=1000` rows → `isComplete()` returns `false` → loop repeats.
5. Attacker opens N concurrent connections (each allowed up to 5 streams per `maxConcurrentCallsPerConnection`), multiplying the number of live polling loops.
6. Each loop holds a live DB query stream via `topicMessageRepository.findByFilter()`, exhausting the DB connection pool.

### Impact Explanation
Each indefinitely-running subscription holds a database connection and a `boundedElastic` scheduler thread. With enough concurrent subscriptions, the DB connection pool is exhausted, causing all subsequent queries (including those from legitimate users and internal components) to queue or fail. This results in full unavailability of the gRPC mirror node service. The `maxConcurrentCallsPerConnection=5` limit is per-connection and does not bound total subscriptions across multiple attacker-controlled connections or IPs. [6](#0-5) 

### Likelihood Explanation
The precondition — a high-volume topic — is trivially achievable on a public network: the attacker submits many `ConsensusSubmitMessage` transactions to a self-created topic, or targets an existing high-traffic topic. No privileged access is required; the gRPC `subscribeTopic` API is public. The attack is repeatable and can be automated. The `throttled=true` flag is set unconditionally by the server, so the attacker does not need to control it. [1](#0-0) 

### Recommendation
1. **Add a maximum wall-clock duration per retrieval session** (distinct from the idle timeout) so that no single throttled retrieval can run longer than a configurable bound (e.g., 5 minutes), regardless of message flow.
2. **Add a per-IP or per-subscriber concurrent subscription limit** at the gRPC interceptor layer.
3. **Change `isComplete()` to also check elapsed time** against a hard cap, or introduce a `maxTotalPolls` for the throttled path analogous to `unthrottled.maxPolls`.
4. **Require or enforce a `limit`** on subscriptions that use the throttled historical retriever, or cap the maximum number of messages returnable per session.

        private PollingContext(TopicMessageFilter filter, boolean throttled) {
            this.filter = filter;
            this.throttled = throttled;

            if (throttled) {
                numRepeats = Long.MAX_VALUE;
                frequency = retrieverProperties.getPollingFrequency();
                maxPageSize = retrieverProperties.getMaxPageSize();
            } else {
                RetrieverProperties.UnthrottledProperties unthrottled = retrieverProperties.getUnthrottled();
                numRepeats = unthrottled.getMaxPolls();
                frequency = unthrottled.getPollingFrequency();
                maxPageSize = unthrottled.getMaxPageSize();
            }
        }
