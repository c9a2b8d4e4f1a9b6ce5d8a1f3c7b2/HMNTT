### Title
Unauthenticated `/health/readiness` Endpoint Allows Goroutine and Connection Exhaustion via Recursive HTTP Amplification

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is exposed without authentication or application-level rate limiting. Each request spawns concurrent goroutines (one per registered check, with a 10-second timeout) and the `checkNetworkStatus` check recursively issues two HTTP calls back into the same server process, creating a compounding resource amplification path accessible to any unprivileged external caller.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `Routes()` (lines 63–78) registers `/health/readiness` bound to `c.readinessHealth.HandlerFunc` with no authentication wrapper, no rate-limit middleware, and no concurrency guard.

`NewHealthController` (lines 37–54) registers two checks on `readinessHealth`, each with `Timeout: time.Second * 10`:
- `postgresql` — opens a DB connection per invocation
- `network` — calls `checkNetworkStatus` (lines 80–101), which issues `NetworkList` then `NetworkStatus` back to `http://localhost:<port>` — i.e., back into the same running Rosetta process

The `hellofresh/health-go/v4` library runs all registered checks **concurrently in goroutines** on every `HandlerFunc` invocation. With a 10-second timeout, goroutines from concurrent in-flight requests accumulate rather than completing immediately.

**Why existing checks fail:**

The Traefik ingress middleware (`inFlightReq: amount: 5`, `rateLimit: average: 10`) is applied only to the paths listed in `charts/hedera-mirror-rosetta/values.yaml` lines 125–133:
```
/rosetta/account, /rosetta/block, /rosetta/call, /rosetta/construction,
/rosetta/events, /rosetta/mempool, /rosetta/network, /rosetta/search
```
`/health/liveness` and `/health/readiness` are **absent** from this list. The health endpoints are served directly by the Go `net/http` server (lines 220–227 of `rosetta/main.go`) with no rate-limiting middleware in the chain (`MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`). No application-level concurrency cap exists.

**Amplification chain per request to `/health/readiness`:**
1. health-go spawns 2 goroutines (postgresql + network), each held up to 10 s
2. The `network` goroutine calls `rosettaClient.NetworkAPI.NetworkList` → HTTP POST to `localhost:<port>/network/list` on the same server
3. Then calls `rosettaClient.NetworkAPI.NetworkStatus` → HTTP POST to `localhost:<port>/network/status`
4. Those two internal requests are handled by the same Go HTTP server, consuming additional goroutines, CPU, and DB connections

At N concurrent external requests: `N × 2` health goroutines + `N × 2` internal HTTP goroutines + `N` DB connections, all held for up to 10 seconds.

### Impact Explanation
A sustained flood of GET requests to `/health/readiness` causes goroutine accumulation (each held 10 s), recursive internal HTTP load, and DB connection exhaustion. This degrades or denies service to legitimate Rosetta API consumers. The `/health/liveness` endpoint has no registered checks and returns immediately — it does not contribute to goroutine accumulation, making the readiness endpoint the sole amplification vector. The 30% resource increase threshold is achievable with modest request rates (tens of requests/second) given the 10-second goroutine lifetime and recursive internal HTTP calls.

### Likelihood Explanation
No credentials, tokens, or special network position are required. Any host that can reach the Rosetta service port can exploit this. In deployments where the pod port is directly reachable (NodePort, LoadBalancer, or direct pod access within a cluster), this is trivially repeatable with standard HTTP tooling (`curl`, `ab`, `wrk`). The Traefik rate-limit does not protect these paths.

### Recommendation
1. **Add application-level rate limiting** to health endpoints in the Go server, e.g., using `golang.org/x/time/rate` or a middleware that caps concurrent in-flight health requests.
2. **Add the health paths to the Traefik ingress** middleware chain in `charts/hedera-mirror-rosetta/values.yaml` so `inFlightReq` and `rateLimit` apply.
3. **Cache health check results** with a short TTL (e.g., 5 s) in the health-go configuration so repeated requests do not each spawn fresh goroutines and DB/HTTP connections.
4. **Reduce the check timeout** from 10 s to a value consistent with expected probe intervals (e.g., 2–3 s) to limit goroutine lifetime.

### Proof of Concept

watch -n1 'ps aux | grep rosetta'

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
}
```
