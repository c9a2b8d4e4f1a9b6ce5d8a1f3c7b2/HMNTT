### Title
Cache Stampede in `getFirstTransactionTimestamp()` Allows Unauthenticated DB Query Amplification

### Summary
`getFirstTransactionTimestamp()` in `rest/timestampRange.js` uses a module-level variable as a cache guard, but the guard check (`if (timestamp === undefined)`) is not protected against concurrent async execution. Because Node.js yields the event loop at every `await`, multiple simultaneous unauthenticated requests with no timestamp filter can all pass the guard before any one of them completes the DB query and sets the cached value, causing N concurrent `ORDER BY consensus_timestamp LIMIT 1` queries against the `transaction` table. No authentication or rate-limiting prevents this.

### Finding Description
**Code location:** `rest/timestampRange.js`, lines 46–71 (`getFirstTransactionTimestamp`) and line 25 (`bindTimestampRange`).

`getFirstTransactionTimestamp` is an IIFE-returned closure that caches the result in a closed-over `timestamp` variable:

```js
// rest/timestampRange.js lines 46-64
const getFirstTransactionTimestamp = (() => {
  let timestamp;
  const func = async () => {
    if (timestamp === undefined) {          // ← guard
      const {rows} = await pool.queryQuietly(`select consensus_timestamp
                                              from transaction
                                              order by consensus_timestamp
                                              limit 1`);  // ← yields event loop
      ...
      timestamp = rows[0].consensus_timestamp;  // ← set only after await resolves
    }
    return BigInt(timestamp);
  };
  return func;
})();
```

`bindTimestampRange` calls this unconditionally when no lower-bound timestamp is supplied by the caller:

```js
// rest/timestampRange.js line 25
const boundRange = Range(range?.begin ?? (await getFirstTransactionTimestamp()), ...);
```

`getTransactionTimestamps` in `rest/transactions.js` (lines 464–466) calls `bindTimestampRange` for every request that has no `eqValues` timestamp:

```js
// rest/transactions.js lines 464-466
if (timestampRange.eqValues.length === 0) {
  const {range, next} = await bindTimestampRange(timestampRange.range, order);
```

**Root cause:** The guard `if (timestamp === undefined)` is evaluated synchronously, but the assignment `timestamp = rows[0].consensus_timestamp` only happens after the `await pool.queryQuietly(...)` resolves. Between those two points the event loop is free to service other requests. Every concurrent request that arrives during that window also sees `timestamp === undefined` and issues its own identical DB query. There is no promise deduplication, no mutex, and no in-flight marker.

**Exploit flow:**
1. Process starts (or `func.reset()` is called in test env); `timestamp` is `undefined`.
2. Attacker sends N concurrent `GET /api/v1/transactions` requests with no `timestamp` query parameter.
3. Each request reaches `getTransactionTimestamps` → `bindTimestampRange` → `getFirstTransactionTimestamp`.
4. All N calls evaluate `timestamp === undefined` as `true` before any DB response arrives.
5. All N calls issue `SELECT consensus_timestamp FROM transaction ORDER BY consensus_timestamp LIMIT 1` simultaneously.
6. DB receives N identical sequential-scan (or index-scan) queries at once.

### Impact Explanation
Each query performs a full ordered scan of the `transaction` table (even with a primary-key index, N simultaneous scans compete for I/O and connection pool slots). Under sufficient concurrency this exhausts the DB connection pool, increases query latency for all other API consumers, and delays the visibility of recently ingested transactions in subsequent paginated responses. The mirror node's REST API is publicly accessible with no authentication requirement, making this trivially reachable. After the first response returns and `timestamp` is set, the cache works correctly — but the window is re-opened on every process restart or test reset.

### Likelihood Explanation
Any unauthenticated external user can trigger this. A single HTTP client sending 50–100 concurrent requests (trivial with `curl --parallel` or any async HTTP library) is sufficient to reproduce the stampede. No special knowledge, credentials, or protocol access is required. The attack is repeatable after any process restart.

### Recommendation
Replace the value-based cache with a **promise-based cache** so that all concurrent callers await the same in-flight DB query rather than each issuing their own:

```js
const getFirstTransactionTimestamp = (() => {
  let timestampPromise;          // cache the Promise, not the value

  return async () => {
    if (!timestampPromise) {
      timestampPromise = pool.queryQuietly(`select consensus_timestamp
                                           from transaction
                                           order by consensus_timestamp
                                           limit 1`)
        .then(({rows}) => rows.length === 1 ? BigInt(rows[0].consensus_timestamp) : 0n)
        .catch((err) => { timestampPromise = undefined; throw err; }); // reset on error
    }
    return timestampPromise;
  };
})();
```

This ensures exactly one DB query is issued regardless of how many concurrent callers arrive before the first response.

### Proof of Concept
```bash
# Send 100 concurrent requests with no timestamp filter
# (requires bindTimestampRange=true in config, which is the production default)
seq 100 | xargs -P100 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/transactions" -o /dev/null

# Observe in DB logs: 100 simultaneous executions of:
# SELECT consensus_timestamp FROM transaction ORDER BY consensus_timestamp LIMIT 1
```

On a freshly started process (or after any restart), all 100 requests arrive before the first DB response, each independently issuing the full-table ordered query.
