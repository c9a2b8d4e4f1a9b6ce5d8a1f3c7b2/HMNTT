### Title
Pre-Skip Database Query Exhaustion via Crafted Block Filter in `getContractResultsById`

### Summary
In `getContractResultsById`, multiple database queries are unconditionally executed before the `skip=true` guard is evaluated. An unprivileged attacker can craft a request combining a create2 EVM address as `contractId`, an EVM address as the `from` filter, and a non-existent `block_number`, forcing up to three wasted DB round-trips per request before the function returns early. Repeated at scale with no rate-limiting visible in the controller, this constitutes a reliable resource-exhaustion vector against the backing PostgreSQL instance.

### Finding Description

**Exact code path** — `rest/controllers/contractController.js`, function `getContractResultsById` (lines 856–892):

```
856  getContractResultsById = async (req, res) => {
...
862    const contractId = await ContractService.computeContractIdFromString(contractIdParam);  // ← DB query #1
...
871    const {conditions, params, order, limit, skip} = await this.extractContractResultsByIdQuery(filters, contractId);
872    if (skip) {
873      return;   // ← early return, but queries already fired
874    }
```

Inside `extractContractResultsByIdQuery` (lines 415–542), before the skip is returned:

- **DB query #2** — line 444: `filter.value = await EntityService.getEncodedId(filter.value)` — triggered when the `from` query parameter is an EVM address.
- **DB query #3** — lines 489/491: `RecordFileService.getRecordFileBlockDetailsFromIndex(blockFilter.value)` / `getRecordFileBlockDetailsFromHash(blockFilter.value)` — triggered whenever `block_number` or `block_hash` is present.
- **skip=true returned** — line 500: `return {skip: true}` — when the block lookup returns null (non-existent block).

**Root cause**: The ordering assumption is wrong. `computeContractIdFromString` (which issues a `SELECT` against the `entity` table for create2 addresses) and the `from`-filter EVM resolution are both executed eagerly, before the block-filter lookup that determines whether the entire request should be skipped. There is no short-circuit or lazy evaluation.

**Why existing checks are insufficient**: `validateContractIdParam` (line 330–338) only validates format, not existence. `buildAndValidateFilters` validates parameter syntax but cannot know whether a block number exists. Nothing prevents the three DB calls from all firing before `skip` is tested.

### Impact Explanation

Each crafted request forces 2–3 `SELECT` statements against PostgreSQL (entity table, record_file table) that produce no useful output. Because the endpoint is public and unauthenticated, an attacker can sustain a high request rate. The DB connection pool and query executor are the bottleneck; saturating them degrades or denies service to legitimate users of all mirror-node REST endpoints that share the same pool. This matches the stated severity scope: degradation of ≥30% of network processing nodes without brute force.

### Likelihood Explanation

- No authentication or privilege required — the `/contracts/:contractId/results` endpoint is fully public.
- The trigger is deterministic: any 40-hex-char string as `contractId` (valid create2 EVM address format) + any EVM address as `from` + any integer as `block_number` that does not exist in `record_file.index` reliably produces the three-query-then-skip path.
- The attack is stateless and trivially parallelisable (e.g., `ab`, `wrk`, or a simple script).
- No CAPTCHA, API key, or rate-limit middleware is present in the controller.

### Recommendation

1. **Reorder operations**: Move `computeContractIdFromString` (line 862) to after `extractContractResultsByIdQuery` returns, and only call it when `skip` is false.
2. **Lazy EVM resolution for `from` filter**: Defer `EntityService.getEncodedId` (line 444) until after the block-filter validity check inside `extractContractResultsByIdQuery`, or restructure the loop so block-filter resolution runs first.
3. **Rate limiting**: Apply per-IP rate limiting at the router/middleware level for all `/contracts` endpoints.
4. **Block existence pre-check short-circuit**: If a `block_number`/`block_hash` filter is present, resolve it first (before any other DB work) and return `{skip: true}` immediately on miss.

### Proof of Concept

```bash
# Craft: valid create2 EVM address as contractId,
#        EVM address as `from` filter,
#        non-existent block_number (e.g., 999999999)
# Result: 3 DB queries fired, then skip=true, empty 200 response

curl "http://<mirror-node>/api/v1/contracts/0x000000000000000000000000000000000000dead/results\
?from=0xabcdef1234567890abcdef1234567890abcdef12\
&block.number=999999999"

# Flood:
for i in $(seq 1 10000); do
  curl -s "http://<mirror-node>/api/v1/contracts/0x000000000000000000000000000000000000dead/results\
?from=0xabcdef1234567890abcdef1234567890abcdef12\
&block.number=$((RANDOM + 900000000))" &
done
```

Each iteration triggers:
1. `ContractService.computeContractIdFromString` → `SELECT` on `entity` table [1](#0-0) 
2. `EntityService.getEncodedId` → `SELECT` on `entity` table for `from` EVM address [2](#0-1) 
3. `RecordFileService.getRecordFileBlockDetailsFromIndex` → `SELECT` on `record_file` table [3](#0-2) 
4. `return {skip: true}` fires, all work discarded [4](#0-3) 
5. Handler returns at line 873 with an empty response [5](#0-4)
