### Title
Unauthenticated Resource Exhaustion via Unbounded Concurrent Requests to `/accounts/:id/rewards` Endpoint

### Summary
The `listStakingRewardsByAccountId` handler in `rest/controllers/accountController.js` performs two database queries per request (account entity validation + staking reward transfer lookup with an IN clause of up to `maxRepeatedQueryParameters` timestamp values) with no per-IP or global rate limiting in the REST Node.js service. An unprivileged attacker can sustain high DB load by issuing many concurrent, fully valid requests across different account IDs, each maximizing the allowed timestamp parameters and limit, without triggering any brute-force detection.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 16 registers the route with no rate-limiting middleware: [1](#0-0) 

`listStakingRewardsByAccountId` in `rest/controllers/accountController.js` (lines 170–203) performs:
1. `EntityService.getEncodedId(...)` — DB lookup to resolve the account
2. `EntityService.isValidAccount(accountId)` — second DB query to validate the account
3. `StakingRewardTransferService.getRewards(...)` — third DB query with an IN clause built from all provided `timestamp=eq:X` values [2](#0-1) 

`extractStakingRewardsQuery` aggregates all `timestamp=eq:X` filters into a single `IN (...)` clause via `updateConditionsAndParamsWithInValues` / `updateQueryFiltersWithInValues`: [3](#0-2) 

The resulting SQL is:
```sql
SELECT ... FROM staking_reward_transfer srt
WHERE srt.account_id = $1
  AND srt.consensus_timestamp IN ($3, $4, ..., $N)
ORDER BY srt.consensus_timestamp DESC LIMIT $2
``` [4](#0-3) 

**Root cause — failed assumption:** The only guard on repeated parameters is `maxRepeatedQueryParameters` (enforced in `buildFilters` and the `qs` parser's `arrayLimit`), which caps the IN clause size per request but does not limit the *rate* of requests: [5](#0-4) [6](#0-5) 

The throttle/rate-limit infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` Java service and is scoped to `ContractCallRequest` objects — it does not apply to the REST Node.js service at all: [7](#0-6) 

The `authHandler` middleware only sets a custom response limit for authenticated users; it does not rate-limit unauthenticated requests: [8](#0-7) 

### Impact Explanation
Each request to `/api/v1/accounts/{id}/rewards` with `maxRepeatedQueryParameters` (default ~25) `timestamp=eq:X` values and `limit=100` causes at minimum 2–3 sequential DB queries. With no rate limiting, an attacker sending hundreds of concurrent requests per second across different valid account IDs (which are sequential integers, trivially enumerable) can saturate the PostgreSQL connection pool and drive CPU/IO on the DB node well above 30% above baseline. The `staking_reward_transfer` table scan with a large IN clause is not a point-lookup — it requires index range scans per timestamp value. This is a sustained, unauthenticated denial-of-service against the database layer.

### Likelihood Explanation
Preconditions are minimal: no account, no credentials, no special knowledge. Valid account IDs are sequential integers starting from low values and are publicly discoverable via the same API. The attack is fully automatable with standard HTTP tooling (e.g., `wrk`, `ab`, `curl` in parallel). Because every request is syntactically and semantically valid, no anomaly/brute-force detection (which typically looks for repeated failures or identical payloads) triggers. The attack is repeatable indefinitely.

### Recommendation
1. **Add per-IP rate limiting** to the REST Node.js service (e.g., `express-rate-limit`) applied globally or specifically to expensive endpoints like `/rewards`.
2. **Cap the number of `timestamp=eq` values** specifically for the rewards endpoint below the global `maxRepeatedQueryParameters`, or require at least one range operator instead of allowing unbounded IN lists.
3. **Add a DB-level statement timeout** specific to this query class so runaway queries are killed before they exhaust pool connections.
4. **Consider caching** the `isValidAccount` result (e.g., short-lived in-process cache) to eliminate the redundant entity lookup DB round-trip per request.

  async getRewards(order, limit, conditions, initParams) {
    const {query, params} = this.getRewardsQuery(order, limit, conditions, initParams);
    const rows = await super.getRows(query, params);
    return rows.map((srt) => new StakingRewardTransfer(srt));
  }

  getRewardsQuery(order, limit, conditions, params) {
    const query = [
      StakingRewardTransferService.listStakingRewardsByAccountIdQuery,
      conditions.length > 0 ? `and ${conditions.join(' and ')}` : '', // "and" since we already have "where account_id = $1" at the end of the above line
      super.getOrderByQuery(
        OrderSpec.from(StakingRewardTransfer.getFullName(StakingRewardTransfer.CONSENSUS_TIMESTAMP), order)
      ),
      super.getLimitQuery(2), // limit is specified in $2 (not necessarily a limit *of* 2)
    ].join('\n');

    return {query, params};
  }

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
