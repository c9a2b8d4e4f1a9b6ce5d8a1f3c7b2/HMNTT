### Title
Unauthenticated GraphQL Introspection + Near-Maximum Complexity Queries Enable Amplified Database Load Without Rate Limiting

### Summary
The GraphQL endpoint at `/graphql/alpha` requires no authentication, has introspection enabled by default (Spring Boot default, no disabling configuration found), and enforces a complexity ceiling of 200 but imposes no application-level rate limiting. The `Account` type contains self-referential fields (`autoRenewAccount`, `stakedAccount`, `obtainer`) that, when nested up to the enforced depth limit of 10, allow any unauthenticated caller to craft queries that approach complexity 200 while triggering multiple chained database lookups per request, enabling sustained amplified server-side work at no cost to the attacker.

### Finding Description
**Code locations:**

- Complexity/depth limits: `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` lines 43–44 — `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are the only execution guards.
- Self-referential schema fields: `graphql/src/main/resources/graphql/account.graphqls` lines 9 (`autoRenewAccount: Account`), 55 (`stakedAccount: Account`), 66 (`obtainer: Accountable`) — each resolved field triggers an independent database lookup.
- No authentication or application-level rate limiting exists anywhere in `graphql/src/main/java/org/hiero/mirror/graphql/`.
- Introspection: `spring-boot-starter-graphql` enables introspection by default; no `spring.graphql.schema.introspection.enabled=false` property or programmatic disabling was found in any configuration file under `graphql/src/main/resources/`.

**Root cause and failed assumption:**

The design assumes that a complexity ceiling of 200 and a depth ceiling of 10 are sufficient to bound per-request server cost. This assumption fails because:

1. graphql-java's default complexity calculator assigns **1 per field**, so a query selecting 199 fields is fully accepted and executed.
2. The `Account` type is self-referential through three fields. A query nesting `autoRenewAccount` → `stakedAccount` → `autoRenewAccount` … to depth 10 causes the resolver to issue a new database query at each level.
3. No per-IP or per-session request rate limit exists at the application layer. The Helm chart's `inFlightReq.amount: 5` (Traefik middleware) is infrastructure-optional and absent in non-Helm deployments.
4. The `CachedPreparsedDocumentProvider` (Caffeine, 1 h TTL) caches parsed/validated documents, reducing parse overhead for repeated identical queries — but execution (and thus DB I/O) still occurs on every request.

**Exploit flow:**

1. Attacker sends a standard GraphQL introspection query to `/graphql/alpha` with no credentials, receiving the full schema including `Account`, `AccountInput`, `EntityIdInput`, `HbarUnit`, and all field definitions.
2. Attacker constructs a query that nests `autoRenewAccount` and `stakedAccount` to depth 9–10, selecting all scalar fields at each level, keeping total field count at 199 (just under the complexity limit).
3. Each such request causes the resolver to issue up to 10 sequential or parallel database queries (one per nested Account resolution).
4. Attacker sends this query in a tight loop from multiple source IPs, bypassing the per-IP infrastructure limit.

### Impact Explanation
Each accepted near-limit query causes disproportionate database load relative to the HTTP request cost. With no application-level rate limiting, an attacker can sustain a high request rate that exhausts database connection pool capacity or CPU, degrading or denying service to legitimate users. The `db.statementTimeout: 10000` (10 s) bounds individual query duration but does not limit the number of concurrent queries triggered per GraphQL request.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The introspection endpoint self-documents the attack surface. The exploit is trivially scriptable with standard HTTP tooling (curl, Python requests). The schema is small and stable, so the attacker needs only one introspection call to permanently know the optimal query shape. Repeatability is high.

### Recommendation
1. **Disable introspection in production**: Set `spring.graphql.schema.introspection.enabled=false` in the production `application.yml`.
2. **Apply field-weight complexity**: Replace the default `MaxQueryComplexityInstrumentation` with a custom `FieldComplexityCalculator` that assigns higher weights to self-referential fields (`autoRenewAccount`, `stakedAccount`, `obtainer`) to reflect their actual DB cost.
3. **Add application-level rate limiting**: Implement a `WebGraphQlInterceptor` or servlet filter that enforces per-IP request rate limits independent of infrastructure deployment.
4. **Limit resolver fan-out**: Use DataLoader batching for nested Account fields to collapse N chained DB queries into a single batched query per nesting level.

### Proof of Concept

```bash
# Step 1: Enumerate schema via introspection (no credentials)
curl -s -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ __schema { types { name fields { name } } } }"}'

# Step 2: Craft near-maximum-complexity nested query (complexity ~190, depth 9)
curl -s -X POST http://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "{
      account(input: {entityId: {shard: 0, realm: 0, num: 2}}) {
        alias memo deleted declineReward
        autoRenewAccount {
          alias memo deleted declineReward
          autoRenewAccount {
            alias memo deleted declineReward
            autoRenewAccount {
              alias memo deleted declineReward
              stakedAccount {
                alias memo deleted declineReward
                autoRenewAccount {
                  alias memo deleted declineReward
                  stakedAccount {
                    alias memo deleted declineReward
                    autoRenewAccount {
                      alias memo deleted declineReward
                      stakedAccount {
                        alias memo deleted declineReward
                      }
                    }
                  }
                }
              }
            }
          }
        }
      }
    }"
  }'

# Step 3: Loop at high rate from multiple IPs
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '<above query>' &
done
```
