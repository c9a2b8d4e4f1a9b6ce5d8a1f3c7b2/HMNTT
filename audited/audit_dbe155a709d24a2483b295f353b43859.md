### Title
Unauthenticated DB Hammering via Non-Existent 0x-Prefixed Transaction Hash Lookups (No Negative-Result Caching)

### Summary
Any unauthenticated user can submit an unlimited stream of syntactically valid but non-existent `0x`-prefixed hex transaction hashes to `GET /api/v1/transactions/:transactionIdOrHash`. Each request causes `extractSqlFromTransactionsByIdOrHashRequest()` to strip the prefix and fire a live database query via `getTransactionHash()`. Because the response cache layer explicitly excludes both hash-based lookups (no cache-control header override) and error responses (404s are never stored), every request hits the database with zero protection.

### Finding Description

**Code path:**

1. `getTransactionsByIdOrHash()` at [1](#0-0)  calls `extractSqlFromTransactionsByIdOrHashRequest()` with the raw URL parameter.

2. Inside `extractSqlFromTransactionsByIdOrHashRequest()`, `isValidTransactionHash()` is called first. The regex in [2](#0-1)  accepts `(0x)?[\dA-Fa-f]{96}` — any 98-character string starting with `0x` followed by 96 hex digits passes validation with no existence check.

3. When the hash length equals `Transaction.HEX_HASH_WITH_PREFIX_SIZE` (= 98, defined at [3](#0-2) ), the prefix is stripped and `getTransactionHash()` is called unconditionally: [4](#0-3) 

4. `getTransactionHash()` issues a direct database query on every call with no internal caching: [5](#0-4) 

5. When the DB returns zero rows, a `NotFoundError` (HTTP 404) is thrown: [6](#0-5) 

**Why the response cache does not help:**

- `getTransactionsByIdOrHashCacheControlHeader()` explicitly returns an empty object `{}` (no cache-control header) whenever `isTransactionHash` is `true`: [7](#0-6)  This means no `max-age` is set for hash-based responses.

- `responseCacheUpdateHandler` only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true: [8](#0-7)  A 404 is never a success, so negative results are never cached regardless of the cache-control header.

- There is no rate limiting in the REST layer: [9](#0-8) 

**Root cause:** The design assumes hash lookups are rare or that the response cache will absorb repeated identical requests. Neither assumption holds for non-existent hashes: the cache-control path explicitly opts out of caching for hash lookups, and the cache update path never stores 4xx responses.

### Impact Explanation
An attacker can generate an unbounded number of syntactically valid but non-existent 48-byte hashes (e.g., `0x` + 96 random hex chars) and flood the endpoint. Each request causes at minimum one call to the `get_transaction_info_by_hash` PostgreSQL function. Under sustained load this exhausts DB connection pool capacity and degrades or denies service for all users of the mirror node REST API. No funds are at risk, but availability of the public API is directly impacted.

### Likelihood Explanation
The precondition is zero: no account, no authentication, no special knowledge. The attacker only needs to know the public endpoint URL and the hash format (documented in the OpenAPI spec). The attack is trivially scriptable with `curl` or any HTTP load tool. Each request is independent and stateless, making it trivially parallelizable. The attack is repeatable indefinitely.

### Recommendation
1. **Cache negative results**: Store 404 responses for hash lookups in Redis with a short TTL (e.g., 5–30 seconds). Modify `responseCacheUpdateHandler` to also cache 404 responses, or add a dedicated negative-result cache inside `getTransactionHash()`.
2. **Add rate limiting**: Apply per-IP rate limiting (e.g., via `express-rate-limit`) on the `/transactions/:transactionIdOrHash` route, especially for hash-format inputs.
3. **Set a cache-control header for hash lookups**: Remove the early-return in `getTransactionsByIdOrHashCacheControlHeader()` for `isTransactionHash === true` so that successful hash responses are also cached and served from Redis on repeat.

### Proof of Concept

```bash
# Generate a valid-format but non-existent 0x-prefixed hex hash
HASH="0x$(python3 -c "import secrets; print(secrets.token_hex(48))")"

# Single request — causes one DB query, returns 404
curl -i "https://<mirror-node-host>/api/v1/transactions/$HASH"

# Flood — each request hits the DB; no caching absorbs any of them
for i in $(seq 1 10000); do
  HASH="0x$(python3 -c "import secrets; print(secrets.token_hex(48))")"
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/transactions/$HASH" &
done
wait
```

Each iteration passes `isValidTransactionHash()`, strips the `0x` prefix, calls `getTransactionHash()` → DB, gets 0 rows, throws `NotFoundError`, and the 404 is never stored in Redis. The next request for the same or any other non-existent hash repeats the full DB round-trip.

const getTransactionHash = async (hash, {order = orderFilterValues.ASC, timestampFilters = []} = {}) => {
  const normalized = normalizeTransactionHash(hash);
  const params = [normalized];

  const timestampConditions = [];
  for (const filter of timestampFilters) {
    timestampConditions.push(`${TransactionHash.CONSENSUS_TIMESTAMP} ${filter.operator} $${params.push(filter.value)}`);
  }

  const query = `${mainQuery}
    ${timestampConditions.length !== 0 ? `where ${timestampConditions.join(' and ')}` : ''}
    ${orderClause} ${order}
    ${limitClause}`;

  const {rows} = await pool.queryQuietly(query, params);
  return normalized !== hash ? rows.filter((row) => row.hash.equals(hash)) : rows;
