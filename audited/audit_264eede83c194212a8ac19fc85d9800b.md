### Title
Race Condition in `RedisTopicListener.getSharedListener` Allows Unprivileged User to Cause Message Loss for Subsequent Subscribers

### Summary
`RedisTopicListener.getSharedListener` uses `computeIfAbsent` to maintain a per-topic shared `Flux`, but the `doOnCancel` callback calls `unsubscribe()` which unconditionally removes the topic from the `topicMessages` map. Any unauthenticated client can subscribe to a topic and immediately cancel, triggering removal of the shared subscription. Messages published to Redis in the window between removal and the next subscriber's `computeIfAbsent` call are not received by the new Redis subscription, causing a gap in live message delivery.

### Finding Description

**Exact code path:**

`RedisTopicListener.java` lines 59–85:

```java
// Line 59-62
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}

// Line 68-80
private Flux<TopicMessage> subscribe(Topic topic) {
    return container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> unsubscribe(topic))   // ← fires when last subscriber cancels
            .doOnComplete(() -> unsubscribe(topic))
            ...
            .retryWhen(...)
            .share();                               // ← cancels upstream when refcount → 0
}

// Line 82-85
private void unsubscribe(Topic topic) {
    topicMessages.remove(topic.getTopic());         // ← unconditional removal
    log.info("Unsubscribing from {}", topic);
}
```

**Root cause:** The `.share()` operator (backed by `publish().refCount()`) cancels the upstream Redis subscription when the last downstream subscriber cancels. The `doOnCancel` hook then calls `unsubscribe()`, which removes the topic key from `topicMessages`. There is no guard preventing this removal when a new subscriber is about to arrive. `ConcurrentHashMap.computeIfAbsent` is atomic per-call, but it does not protect the cross-call sequence of: *cancel fires → map entry removed → new computeIfAbsent creates fresh subscription*.

**Exploit flow:**

1. Attacker (no credentials required — `ConsensusController.subscribeTopic` has zero authentication, line 43–53) opens a gRPC stream to `subscribeTopic` for topic T.
2. Attacker is the sole subscriber (or waits until they are the last one).
3. Attacker immediately closes the gRPC connection. `ServerCallStreamObserver.setOnCancelHandler(disposable::dispose)` (line 51) fires `dispose()`, propagating cancel up the reactive chain.
4. `.share()` sees refcount drop to 0, cancels the upstream Redis subscription, and `doOnCancel` fires `unsubscribe(topic)` → `topicMessages.remove("topic.<N>")`.
5. Redis publishes messages for topic T. No listener exists; messages are silently dropped at the Redis layer.
6. Legitimate subscriber B calls `getSharedListener` → `computeIfAbsent` creates a brand-new Redis subscription. Subscriber B never receives the messages from step 5.

**Why existing checks are insufficient:**

- `ConcurrentHashMap` only makes individual `computeIfAbsent` calls atomic; it does not prevent the TOCTOU window between the async `doOnCancel` callback and the next `computeIfAbsent`.
- The safety check in `TopicMessageServiceImpl` (lines 66–70) fires a database poll 1 second after subscription. Because the parser persists to the database *before* publishing to Redis, messages from step 5 are in the database and the safety check will eventually recover them — but only after a 1-second blind window. If the attacker repeats the subscribe/cancel cycle faster than the safety check fires, subscriber B is continuously delayed.
- The safety check fires only once per subscription context; it does not continuously guard against repeated disruption.

### Impact Explanation

Subscriber B experiences up to a 1-second gap in live message delivery per attacker-triggered cycle. While the safety check recovers messages from the database, the live Redis stream is silently broken and must be re-established. In high-frequency HCS topic scenarios (e.g., DeFi settlement, audit trails), a 1-second blind window per disruption cycle is a meaningful data-integrity regression. An attacker who loops the subscribe/cancel pattern can keep the shared Redis subscription in a perpetual torn-down state, forcing all legitimate subscribers to rely solely on the 1-second delayed database fallback rather than the real-time Redis path.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint requires no authentication or API key (confirmed in `ConsensusController`, lines 43–53, and the public `grpcurl` example in `docs/grpc/README.md`). Any internet-reachable mirror node is exploitable. The attacker needs only a gRPC client (e.g., `grpcurl`) and knowledge of a topic ID. The attack is trivially repeatable in a tight loop with negligible cost. The only precondition — being the last subscriber — is easily satisfied for low-traffic topics or by timing the cancel to coincide with other subscribers' natural disconnects.

### Recommendation

Replace the unconditional `topicMessages.remove` with a conditional check that only removes the entry if the stored value is still the same `Flux` instance that is being torn down (compare-and-remove). Additionally, consider using `computeIfPresent` with a reference-equality guard, or restructure so that `unsubscribe` is only called after verifying no new subscriber has already re-registered the topic:

```java
private void unsubscribe(Topic topic, Flux<TopicMessage> flux) {
    // Only remove if the map still holds *this* flux instance
    topicMessages.remove(topic.getTopic(), flux);
    log.info("Unsubscribing from {}", topic);
}
```

Pass the `Flux` reference into `doOnCancel`/`doOnComplete` via a local variable captured at `subscribe()` time. This eliminates the TOCTOU window: if a new subscriber has already replaced the map entry via `computeIfAbsent`, the `remove(key, value)` call is a no-op.

### Proof of Concept

```bash
# Terminal 1 – legitimate subscriber (victim)
grpcurl -plaintext -d '{"topicID": {"topicNum": 1000}}' \
  localhost:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# Terminal 2 – attacker loop (no credentials needed)
while true; do
  # Subscribe and immediately cancel
  grpcurl -plaintext -d '{"topicID": {"topicNum": 1000}}' \
    localhost:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  PID=$!
  sleep 0.05   # 50ms – enough to become last subscriber if victim is slow
  kill $PID
done

# Observe: victim's stream receives messages only via the 1-second safety-check
# database fallback, not via the real-time Redis path. Messages published during
# the torn-down window are delayed by ~1s or missed if the loop outruns the check.
```
