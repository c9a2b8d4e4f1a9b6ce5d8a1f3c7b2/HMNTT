### Title
Unauthenticated Per-Request DB Query in `getTokenRelationships()` Enables DoS via DB Connection Pool Exhaustion

### Summary
The `getTokenRelationships()` handler unconditionally calls `EntityService.isValidAccount()` for every inbound request, which issues a raw database query with no caching. Because 404 responses (for non-existent accounts) are explicitly excluded from the Redis response cache, an unprivileged attacker can flood the endpoint with requests for non-existent account IDs, driving one DB query per request with no throttle, and exhaust the database connection pool.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 67–71: [1](#0-0) 

Every request unconditionally calls `EntityService.isValidAccount(accountId)`.

`rest/service/entityService.js`, `isValidAccount()`, lines 60–63: [2](#0-1) 

This issues `entityExistenceQuery` — a raw `SELECT type FROM entity WHERE id = $1` — directly against the DB with no in-memory or Redis caching of the result. [3](#0-2) 

**Why the response cache does not protect this path:**

`rest/middleware/responseCacheHandler.js`, `responseCacheUpdateHandler()`, line 95: [4](#0-3) 

The cache update only fires when `httpStatusCodes.isSuccess(res.statusCode)` is true. A request for a non-existent account ID throws `NotFoundError` → HTTP 404, which is not a success code, so the response is **never stored in Redis**. Every subsequent request for the same non-existent account ID hits the DB again.

**No rate limiting exists:** No rate-limiting middleware was found anywhere in the REST service middleware stack.


**Root cause:** The failed assumption is that the DB can absorb an unbounded number of concurrent existence-check queries from unauthenticated callers. There is no per-IP throttle, no caching of negative results, and no authentication gate.

### Impact Explanation

An attacker who exhausts the DB connection pool causes all DB-backed endpoints to queue or fail, making the mirror node REST API unavailable to legitimate users. This is a denial-of-service against service availability with no economic cost to the attacker (no tokens, no fees, no on-chain transactions required).

### Likelihood Explanation

Hedera account IDs are sequential integers in the form `0.0.N`. An attacker can trivially generate a list of IDs beyond the current maximum (guaranteed non-existent) and issue them in parallel. No credentials, API keys, or privileged access are required. The attack is repeatable and automatable with a simple HTTP client loop.

### Recommendation

1. **Cache negative results:** Add a short-lived (e.g., 5–30 s) in-memory or Redis cache for `isValidAccount()` results, keyed by `accountId`, so repeated queries for the same ID do not hit the DB.
2. **Rate-limit the endpoint:** Apply per-IP (or global) rate limiting middleware to `/accounts/:id/tokens` and similar unauthenticated endpoints.
3. **Cache 404 responses:** Extend `responseCacheUpdateHandler` to also cache 404 responses for a short TTL, so repeated requests for the same non-existent account are served from Redis without a DB round-trip.

### Proof of Concept

```bash
# Generate requests for non-existent account IDs in parallel
# (IDs well above current network maximum are guaranteed non-existent)
seq 9000000 9010000 | xargs -P 200 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.{}/tokens" -o /dev/null

# Each request triggers:
#   1. EntityService.getEncodedId()  -> parses "0.0.{}" locally (no DB)
#   2. EntityService.isValidAccount() -> SELECT type FROM entity WHERE id = {} (DB hit)
#   3. NotFoundError thrown -> 404 response NOT cached in Redis
#   4. Next request for same ID repeats the DB query

# With 200 concurrent workers and IDs cycling through 10,000 values,
# the DB connection pool is saturated; legitimate queries begin timing out.
```
