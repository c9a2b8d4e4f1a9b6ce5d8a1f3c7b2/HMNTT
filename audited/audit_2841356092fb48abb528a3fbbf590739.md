### Title
Unbounded EVM Memory Snapshot Accumulation in Opcode Tracer Causes JVM Heap Exhaustion (DoS)

### Summary
The `getContractOpcodes()` endpoint accepts `memory=true` from any unauthenticated caller and replays the historical transaction through the EVM. During replay, `captureMemory()` in `AbstractOpcodeTracer` snapshots the **entire EVM memory** at every single opcode step with no size cap, and each snapshot is appended to an unbounded `ArrayList<Opcode>` in `OpcodeContext`. A transaction that expanded EVM memory to its gas-bounded maximum and executed many cheap opcodes (e.g., `JUMPDEST`) causes the JVM to accumulate gigabytes of heap-allocated hex strings, crashing the mirror node process.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) accepts `memory=true` from any caller with no authentication. The only guards are a gzip header check and a rate throttle: [1](#0-0) 

The throttle only limits request *rate*, not per-request heap consumption: [2](#0-1) 

`OpcodeServiceImpl.processOpcodeCall()` pre-allocates the opcode list with capacity `gas / 3` — for a 15 M-gas transaction this is ~5 M entries — and then runs the full EVM replay: [3](#0-2) 

`OpcodeContext` stores every captured opcode in an unbounded `ArrayList` with no maximum size: [4](#0-3) 

**Root cause — `captureMemory()` in `AbstractOpcodeTracer`:** [5](#0-4) 

`frame.memoryWordSize()` is read with no upper-bound check. `frame.readMutableMemory(0L, (long) wordSize * size)` reads the entire EVM memory into a byte array. A new `ArrayList<String>` of `size` entries is allocated, and each 32-byte word is hex-encoded into a 66-character `String`. This snapshot is stored inside an `Opcode` object that is appended to `OpcodeContext.opcodes`. The process repeats for **every opcode step** in the replay.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| `Accept-Encoding: gzip` required | Compresses the HTTP *response*; does nothing to limit server-side heap during replay |
| `throttleOpcodeRequest()` | Rate-limits the number of requests per time window; does not cap memory consumed per request |
| EVM gas limit | Bounds EVM execution cost, not the Java heap consumed by snapshot accumulation |
| `OpcodesProperties.isEnabled` | Feature flag; once enabled, no memory guard exists |

### Impact Explanation

With Hedera's maximum gas limit (~15 M gas), an attacker can craft a transaction that:
1. Expands EVM memory to ~88 KB (quadratic expansion cost consumes ~2 M gas).
2. Executes ~13 M `JUMPDEST` opcodes (1 gas each) in a tight loop.

Each of the 13 M opcode steps captures an 88 KB memory snapshot → ~88 KB × 2 (hex encoding overhead) ≈ 176 KB per `Opcode` object. Total heap: **13 M × 176 KB ≈ 2.3 TB** — far beyond any realistic JVM heap. Even a fraction of this (e.g., 100 K opcodes × 176 KB = ~17 GB) exceeds typical JVM heap sizes and triggers `OutOfMemoryError`, crashing the mirror node process. Because the endpoint is unauthenticated and the throttle only limits rate, a single well-crafted request is sufficient.

### Likelihood Explanation

- **No privilege required**: any external user can call the endpoint.
- **Precondition is trivially met**: the attacker only needs a historical transaction hash for a contract that used significant EVM memory. Such transactions already exist on mainnet (e.g., contracts doing large ABI encoding, array operations, or keccak over large buffers).
- **Repeatability**: the throttle allows repeated attempts; a single successful request is enough to crash the JVM.
- **Detectability**: the attack looks like a normal API call with a valid transaction hash and `memory=true`.

### Recommendation

1. **Cap `captureMemory()` output size**: add a configurable maximum word count (e.g., 1024 words = 32 KB) and truncate or return an error if exceeded.
2. **Cap total opcode count**: add a configurable limit on `OpcodeContext.opcodes` size; reject or truncate once the limit is reached.
3. **Cap initial ArrayList pre-allocation**: `(int) params.getGas() / 3` can pre-allocate millions of entries; clamp this to a safe maximum (e.g., 100 K).
4. **Add per-request memory accounting**: track estimated heap usage during replay and abort with HTTP 413/507 if a threshold is exceeded.
5. **Require authentication/authorization** for the opcodes endpoint, or restrict it to internal/operator use only.

### Proof of Concept

**Preconditions:**
- Mirror node running with `hiero.mirror.web3.opcode.tracer.enabled=true`.
- A historical transaction hash for any contract that allocated ≥ 32 KB of EVM memory (e.g., a contract doing `for(uint i=0;i<1000;i++) assembly { mstore(mul(i,32), i) }`).

**Steps:**

```bash
# 1. Identify a transaction hash for a memory-heavy contract call (or deploy one)
TX_HASH="0x<hash_of_memory_heavy_tx>"

# 2. Send a single request with memory=true (no authentication needed)
curl -H "Accept-Encoding: gzip" \
     "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true" \
     --output /dev/null

# 3. Observe: mirror node JVM throws OutOfMemoryError and the process crashes.
#    With a sufficiently memory-heavy transaction, a single request is sufficient.
#    With the throttle in place, a low-rate stream of such requests achieves the same result.
```

**Expected result:** JVM heap exhaustion (`java.lang.OutOfMemoryError: Java heap space`) during `captureMemory()` accumulation inside `AbstractOpcodeTracer`, crashing the mirror node.

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
