### Title
`statementInspector()` Statement-Boundary-Only Timeout Fails to Bound Single Long-Running Historical Queries, Enabling DB Connection and Thread Exhaustion

### Summary
The `statementInspector()` in `HibernateConfiguration.java` enforces the `requestTimeout` (10 seconds) only at SQL statement boundaries — before each new statement executes — and cannot interrupt a query already in flight. A single slow historical-block DB query can therefore occupy a DB connection and a server thread for up to the full DB-level `statement_timeout` (10 seconds, matching `requestTimeout`). Because the endpoint requires no authentication and the rate limit is global (not per-IP), an unprivileged attacker can sustain a stream of such requests to exhaust the Hikari connection pool and inflate thread utilization well above 30%.

### Finding Description

**Exact code path**

`statementInspector()` — `web3/src/main/java/org/hiero/mirror/web3/config/HibernateConfiguration.java`, lines 31–47:

```java
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis(); // 10 000 ms
    return sql -> {
        if (!ContractCallContext.isInitialized()) { return sql; }
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {
            throw new QueryTimeoutException(...);
        }
        return sql;   // ← fires only BEFORE the next statement, never mid-query
    };
}
```

`requestTimeout` default — `web3/src/main/java/org/hiero/mirror/web3/Web3Properties.java`, line 20:

```java
private Duration requestTimeout = Duration.ofSeconds(10L);
```

`ContractCallContext.startTime` — `web3/src/main/java/org/hiero/mirror/web3/common/ContractCallContext.java`, line 31:

```java
private final long startTime = System.currentTimeMillis();
```

**Root cause**

Hibernate's `StatementInspector.inspect(sql)` is a pre-execution hook. It is called once before JDBC sends the SQL to the database. Once the database begins executing the query, the inspector is never called again until the application issues the *next* statement. A single slow historical query (e.g., `findHistoricalAccountBalanceUpToTimestamp`) can therefore run uninterrupted for the entire duration of the DB-level `statement_timeout`.

**Failed assumption**

The design assumes that the application-level `requestTimeout` (10 s) will be enforced by the `statementInspector()`. This holds only when a request issues multiple short statements; for a single long-running statement the inspector provides zero intra-query protection.

**DB-level backstop is equally long**

The Helm chart sets the PostgreSQL-level timeout for the `mirror_web3` role to **10 000 ms** — identical to `requestTimeout`:

```
alter user mirror_web3 set statement_timeout to '10000';
```
(`charts/hedera-mirror/templates/secret-passwords.yaml`, line 127)

The documented application-level property `hiero.mirror.web3.db.statementTimeout = 3000` (docs line 702) is not wired to any Java datasource configuration in the web3 module (no `DbProperties.java` exists there, and no Hikari `connectionInitSql` was found). Additionally, PgBouncer is configured to silently ignore `statement_timeout` startup parameters:

```yaml
ignore_startup_parameters: extra_float_digits,options,statement_timeout
```
(`charts/hedera-mirror/values.yaml`, line 430)

The net result: the effective per-query ceiling is 10 seconds, not 3 seconds.

**Exploit flow**

1. Attacker sends `POST /api/v1/contracts/call` with `{"block": "0x1", "to": "<valid contract>", "gas": 21000, ...}` — no credentials required (`ContractController.java`, line 38).
2. `ContractCallContext.run()` creates a new context; `startTime` is recorded.
3. The EVM resolves historical state, issuing one or more DB queries (e.g., `findHistoricalAccountBalanceUpToTimestamp`).
4. If the first query is slow, `statementInspector()` is not called again until the query returns.
5. The query runs for up to 10 seconds before PostgreSQL kills it via `statement_timeout`.
6. During those 10 seconds the Hikari connection and the Tomcat/virtual thread are fully occupied.
7. Attacker repeats at the global rate limit (500 req/s, not per-IP). With a Hikari pool of ~50 connections, 50 concurrent slow requests saturate the pool in under 0.1 s; subsequent legitimate requests queue or fail.

### Impact Explanation

Each crafted request can hold one DB connection and one server thread for up to 10 seconds. The Hikari pool for the web3 service is small (tens of connections). Saturating it blocks all concurrent contract calls, including `LATEST`-block calls, causing cascading 503 / timeout responses. Thread-pool exhaustion follows if the server uses platform threads. Resource consumption (DB connections, threads, CPU for query planning) increases proportionally to the number of concurrent slow requests, easily exceeding 30% above baseline with a modest sustained attack rate.

### Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The rate limit (500 req/s) is global and not per-source-IP, so a single attacker can consume the entire budget. Historical block calls are a documented, supported feature — no special knowledge is required. The attacker only needs to supply a valid contract address and a low block number; no brute-force enumeration is needed. The attack is repeatable and scriptable with a single HTTP client.

### Recommendation

1. **Enforce a per-statement DB timeout shorter than `requestTimeout`**: Apply `SET statement_timeout = 3000` via Hikari's `connectionInitSql` (or a `DataSourceConnectionCustomizer`) so each individual query is killed after 3 seconds, not 10. Ensure this is not overridden by PgBouncer by setting it at the session level inside the connection rather than as a startup parameter.
2. **Use JDBC query timeouts**: Set `spring.jpa.properties.jakarta.persistence.query.timeout` (or Hibernate's `javax.persistence.query.timeout`) to 3000 ms so Hibernate cancels the JDBC statement after 3 seconds regardless of statement boundaries.
3. **Add per-IP rate limiting**: The current global bucket does not prevent a single client from monopolizing the quota.
4. **Align documented and actual timeouts**: The documented `hiero.mirror.web3.db.statementTimeout = 3000` should be wired to the actual datasource; currently it has no effect.

### Proof of Concept

```bash
# Send 60 concurrent historical contract calls (no auth required)
for i in $(seq 1 60); do
  curl -s -X POST http://<web3-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"block":"0x1","to":"0x0000000000000000000000000000000000000167","gas":21000,"data":"0x"}' &
done
wait
# Observe: legitimate /api/v1/contracts/call requests now time out or return 503
# DB connection pool (hikaricp_connections_active) saturates to max
```

Each background request holds a DB connection for up to 10 seconds while the historical query runs. With 60 concurrent requests and a pool of ~50 connections, the pool is exhausted immediately, and all subsequent requests stall until connections are released.
