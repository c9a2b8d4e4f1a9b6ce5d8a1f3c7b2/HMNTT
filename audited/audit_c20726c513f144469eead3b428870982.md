### Title
Unbounded Retry Backoff in `PollingTopicMessageRetriever` Enables Thread Pool Exhaustion During DB Partition

### Summary
`PollingTopicMessageRetriever.retrieve()` uses `retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` with no `maxBackoff` cap, while each retry drives a blocking JDBC call (`getResultList()`) that holds a `Schedulers.boundedElastic()` thread for up to the DB statement timeout. Any unprivileged user who subscribes during a DB partition triggers this loop; with enough concurrent subscriptions the global `boundedElastic` thread pool is exhausted, and because the `timeout` operator is scheduled on that same pool, the self-termination mechanism is also starved.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java` [1](#0-0) 

```
Flux.defer(() -> poll(context))
    .repeatWhen(... .withScheduler(scheduler))   // line 52-55
    .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))  // line 58 — NO maxBackoff
    .timeout(retrieverProperties.getTimeout(), scheduler)             // line 59 — same scheduler
```

**Root cause 1 – missing `maxBackoff`:**
`Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` has no `.maxBackoff(...)` call. Reactor's default `maxBackoff` is `Duration.ofMillis(Long.MAX_VALUE)` — effectively unlimited exponential growth (1 s → 2 s → 4 s → 8 s → 16 s → 32 s → …). The sibling class `SharedPollingTopicListener` correctly caps this: [2](#0-1) 

`PollingTopicMessageRetriever` has no equivalent cap. [3](#0-2) 

**Root cause 2 – blocking JDBC on `boundedElastic` thread:**
`poll()` calls `topicMessageRepository.findByFilter()`, which resolves to `typedQuery.getResultList()` — a fully blocking JPA/JDBC call: [4](#0-3) 

There is no `.subscribeOn(scheduler)` in `retrieve()`. The `repeatWhen(...withScheduler(scheduler))` causes each repeated `poll()` to execute on a `boundedElastic` thread. During a DB partition that thread is held for the full DB statement timeout (configured at 10 000 ms). [5](#0-4) 

**Root cause 3 – `timeout` uses the same exhausted scheduler:**
The `timeout` operator that is supposed to terminate stale subscriptions is scheduled on the same `boundedElastic` instance: [6](#0-5) 

If all threads are occupied by blocking `poll()` calls, the timeout task is queued behind them. Under sustained load the queue fills and the timeout task is never executed, so subscriptions never self-terminate.

**Exploit flow:**
1. DB partition begins (network-level, not attacker-controlled, but attacker exploits the window).
2. Attacker opens N gRPC connections (no authentication required), each with up to 5 concurrent `subscribeTopic` calls (`maxConcurrentCallsPerConnection = 5`). [7](#0-6) 
3. Each subscription calls `retrieve()` → `poll()` → `getResultList()` blocks a `boundedElastic` thread for ≤10 s.
4. After the statement timeout, `retryWhen` schedules the next attempt with exponential backoff (no cap). The thread is released during the backoff delay, then re-acquired for the next `poll()`.
5. With `10 × CPU_cores` threads available (e.g., 40 on a 4-core host), 8 connections × 5 subscriptions = 40 simultaneous blocking calls saturate the pool.
6. New `boundedElastic` tasks (including the `timeout` scheduled tasks for existing subscriptions) queue up. Once the 100 000-task queue fills, tasks are rejected with `RejectedExecutionException`, preventing any subscription from terminating.
7. The service cannot process new legitimate requests that depend on `boundedElastic`.

### Impact Explanation

Thread pool exhaustion of the global `Schedulers.boundedElastic()` scheduler affects every component in the gRPC service that uses it — including `SharedTopicListener.publishOn(Schedulers.boundedElastic())` and the `timeout` self-healing mechanism. The service becomes unable to serve new subscribers or recover existing ones, constituting a full denial-of-service for the gRPC API for the duration of the DB partition plus the time needed to drain the thread pool.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The attacker needs only a DB partition to be in progress (which can be a transient infrastructure event, not attacker-caused) and the ability to open multiple TCP connections to port 5600. Both conditions are realistic in production. The attack is repeatable: each new partition window is a new opportunity. The thread count threshold (40 on a 4-core host) is low enough to be reached with a modest number of connections.

### Recommendation

1. **Add `maxBackoff` to the `retryWhen` call**, consistent with `SharedPollingTopicListener`:
   ```java
   .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))
       .maxBackoff(Duration.ofSeconds(4)))
   ```
2. **Offload blocking JDBC to a dedicated scheduler** (not the global `boundedElastic`) so thread exhaustion is isolated:
   ```java
   Scheduler blockingScheduler = Schedulers.newBoundedElastic(
       10, 1000, "retriever-blocking");
   ```
3. **Schedule the `timeout` on a separate scheduler** so it cannot be starved by the same pool the blocking calls use.
4. **Add connection-level rate limiting** (e.g., max connections per IP) at the Netty layer to limit the number of concurrent subscriptions an attacker can open.

### Proof of Concept

```
Precondition: DB network partition is active (e.g., iptables DROP on port 5432).

1. Open 8 gRPC connections to mirror-node:5600.
2. On each connection, issue 5 concurrent subscribeTopic(ConsensusTopicQuery{topicID: <any valid topic>}) RPCs.
   → 40 concurrent retrieve() calls are active.
3. Each call enters poll() → getResultList() blocks for 10 s (statementTimeout).
4. After 10 s, QueryTimeoutException propagates to retryWhen; backoff delay starts (1 s).
5. After 1 s, next poll() attempt: another 40 threads blocked for 10 s.
6. Repeat. Within ~30 s the boundedElastic thread pool (40 threads) is continuously saturated.
7. Attempt a new subscribeTopic from a different client:
   → The repeatWhen scheduler task is queued; the subscription hangs indefinitely.
8. Observe via /actuator/metrics that executor.pool.size for boundedElastic is at max
   and executor.queued.tasks is growing toward 100 000.
9. Once the queue fills, RejectedExecutionException is thrown for all new scheduler tasks,
   including the timeout tasks for existing subscriptions — subscriptions never terminate.
```
