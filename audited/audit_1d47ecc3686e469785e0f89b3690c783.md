### Title
Unbounded `data` Field Logged in Full via `log.info("{}", request)` Enables Log-Based Disk Exhaustion

### Summary
In `ThrottleManagerImpl.action()`, when a `RequestProperties` filter with `ActionType.LOG` matches an incoming `ContractCallRequest`, the entire request object is written to the log via `log.info("{}", request)` with no size truncation. The `data` field of `ContractCallRequest` accepts up to 1,048,576 hex characters (~1 MB string) by default. An unprivileged attacker who can craft requests matching the configured LOG filter can repeatedly force ~1 MB log entries, exhausting disk space and degrading or crashing the node.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.java`, `action()`, line 68: [1](#0-0) 

```java
private void action(RequestProperties filter, ContractCallRequest request) {
    switch (filter.getAction()) {
        case LOG -> log.info("{}", request);   // ← full request, no truncation
```

`log.info("{}", request)` calls Lombok-generated `ContractCallRequest.toString()`, which includes every field — including `data`.

**Root cause — no upper bound enforced at log time:**

`ContractCallRequest.data` carries `@Hex` with no explicit `maxLength`, so the default from `Hex.java` applies: [2](#0-1) 

```java
long maxLength() default 1048576L;   // 1,048,576 hex chars ≈ 1 MB string
``` [3](#0-2) 

```java
@Hex
private String data;   // no explicit maxLength → inherits 1,048,576
```

Bean Validation (`@Valid` on the controller parameter) rejects data longer than 1,048,576 hex chars, but passes everything up to that limit straight through to `action()`.

**Filter matching — attacker-controlled:**

`RequestFilter.test()` evaluates whether the request's `DATA` field (or any other field) contains or equals a configured expression: [4](#0-3) 

If the operator configures a DATA CONTAINS filter with a short, common hex pattern (e.g., `"0x"` or `"dead"`), any attacker payload that embeds that pattern triggers the LOG action. The attacker controls the full 1 MB payload and can trivially satisfy any CONTAINS expression.

**Rate limiting is insufficient:**

`ThrottleManagerImpl.throttle()` enforces a per-second request cap and a gas cap: [5](#0-4) 

These caps limit *request count*, not *bytes written to disk per request*. At a modest 100 req/s rate limit, each carrying a 1 MB `data` string, the logger emits ≥100 MB/s of log data — enough to fill a typical disk partition in minutes.

### Impact Explanation

Sustained log flooding at ~100 MB/s exhausts the filesystem hosting the log files. Once the disk is full, the JVM and/or the OS can no longer write logs, which typically causes the application to crash or become unresponsive. On a shared node, other services on the same filesystem are also affected. This maps to the stated severity: shutdown of ≥30% of network processing nodes without brute-force actions, because a single attacker stream can take down any node whose LOG filter is configured to match a trivial pattern.

### Likelihood Explanation

- **No privileges required.** The `/api/v1/contracts/call` endpoint is public.
- **Precondition:** An operator must have configured at least one `RequestProperties` entry with `ActionType.LOG` and a DATA filter whose expression the attacker can match. This is a realistic operational configuration (operators use LOG filters for debugging/auditing).
- **Repeatability:** The attack is fully automated and repeatable; a single script sending max-size hex payloads at the rate limit is sufficient.
- **Detection evasion:** Each individual request is valid and passes all Bean Validation checks, so it generates no error-level signals until the disk is full.

### Recommendation

1. **Truncate before logging.** Replace the bare `log.info("{}", request)` with a version that caps the `data` field:
   ```java
   case LOG -> {
       String truncatedData = request.getData() != null
           ? request.getData().substring(0, Math.min(request.getData().length(), 256)) + "…"
           : null;
       log.info("ContractCallRequest[data={}, gas={}, to={}, from={}]",
           truncatedData, request.getGas(), request.getTo(), request.getFrom());
   }
   ```
2. **Lower the `@Hex` default `maxLength`** on `ContractCallRequest.data` to match the OpenAPI spec's stated limit of 262,146 characters, or enforce it explicitly:
   ```java
   @Hex(maxLength = 262146)
   private String data;
   ```
3. **Add a log-rate guard** (e.g., a separate token bucket) inside the `LOG` branch so that even if many requests match, the logger is called at most N times per second.
4. **Configure a Spring Boot request-size limit** (`spring.servlet.multipart.max-request-size`, or Tomcat's `maxPostSize`) as a defence-in-depth measure.

### Proof of Concept

```bash
# 1. Generate a 1 MB hex payload that contains a common pattern ("dead")
PAYLOAD="0xdead$(python3 -c "print('ab' * 524286)")"

# 2. Send at the rate limit (adjust --rate to match server config)
for i in $(seq 1 200); do
  curl -s -X POST http://<node>:8080/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x00000000000000000000000000000000000004e4\",
         \"gas\":15000000,
         \"data\":\"$PAYLOAD\"}" &
done
wait

# 3. Observe: each request that passes the rate limiter and matches the
#    DATA CONTAINS "dead" filter writes ~1 MB to the application log.
#    Monitor disk usage: watch -n1 df -h /var/log
```

Each successful request causes `log.info("{}", request)` to serialize and write the full ~1 MB `data` string. At the default rate limit the log partition fills within minutes.
