### Title
Unbounded Hex Alias Allocation in `AccountBalance()` Enables Memory and CPU Exhaustion

### Summary
`NewAccountIdFromString()` performs `hex.DecodeString()` on the caller-supplied `AccountIdentifier.Address` with no length check, allocating a byte slice proportional to the input. The server applies no HTTP body size limit, so an unauthenticated attacker can send arbitrarily large hex strings on every request, causing sustained memory allocation pressure and CPU load from `proto.Unmarshal` on the resulting byte slice before the error path is reached.

### Finding Description

**Exact code path:**

`AccountBalance()` at [1](#0-0)  unconditionally calls `types.NewAccountIdFromString(request.AccountIdentifier.Address, ...)`.

Inside `NewAccountIdFromString()`, the only pre-decode check is a `0x` prefix test: [2](#0-1) 

Line 195 calls `hex.DecodeString(tools.SafeRemoveHexPrefix(address))` with **no length bound**. For a 10 MB hex string this allocates a 5 MB byte slice. The slice is then passed to `NewAccountIdFromAlias()` → `NewPublicKeyFromAlias()`: [3](#0-2) 

`proto.Unmarshal(alias, &key)` at line 66 performs an O(n) parse of the full byte slice before returning an error. Only after both the allocation and the protobuf parse does the error propagate back to `AccountBalance()`, which returns `errors.ErrInvalidAccount`.

**No HTTP body size limit is configured.** `main.go` sets only timeout values on the `http.Server`; no `http.MaxBytesReader` or equivalent is applied anywhere in the middleware chain: [4](#0-3) 

The middleware stack (MetricsMiddleware → TracingMiddleware → CorsMiddleware) adds no body-size enforcement: [5](#0-4) 

### Impact Explanation
Each malicious request causes: (1) a heap allocation of `len(address)/2` bytes, (2) an O(n) CPU pass through `proto.Unmarshal`. Under a sustained flood of requests carrying multi-megabyte hex strings, heap pressure forces more frequent GC cycles and CPU utilization rises proportionally. Because the Rosetta endpoint is unauthenticated and the error path is cheap to trigger at scale, an attacker can realistically push memory and CPU consumption well above the 30% threshold without brute-force volume — a moderate request rate with large payloads suffices.

### Likelihood Explanation
The `/account/balance` endpoint is publicly reachable with no authentication. Crafting a valid hex string (even-length sequence of `[0-9a-f]` characters prefixed with `0x`) requires zero cryptographic knowledge. The attack is trivially scriptable, repeatable, and requires only a standard HTTP client. No special privileges or network position are needed.

### Recommendation
1. **Add a length check in `NewAccountIdFromString()`** before calling `hex.DecodeString`. Valid aliases are at most 35 bytes (70 hex chars); reject anything longer:
   ```go
   stripped := tools.SafeRemoveHexPrefix(address)
   if len(stripped) > 70 { // max valid alias hex length
       return zero, errors.Errorf("alias too long")
   }
   alias, err := hex.DecodeString(stripped)
   ```
2. **Apply `http.MaxBytesReader`** in a middleware wrapper (e.g., 64 KB) to cap the entire request body before JSON decoding begins.

### Proof of Concept
```bash
# Generate a 5 MB hex payload (10 MB hex string)
PAYLOAD=$(python3 -c "print('0x' + 'aa' * 5_000_000)")

# Send to /account/balance in a tight loop
while true; do
  curl -s -X POST http://<rosetta-host>:8082/account/balance \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\": {\"address\": \"$PAYLOAD\"}
    }" &
done
```
Each request causes `hex.DecodeString` to allocate ~5 MB and `proto.Unmarshal` to scan it before returning an error. At 20 concurrent requests/second this produces ~100 MB/s of short-lived heap churn and sustained `proto.Unmarshal` CPU load, measurably increasing node resource consumption above the 30% threshold.
