### Title
Unauthenticated Unbounded Concurrent Requests to `getNft()` via `SELECT *` with No REST-Layer Rate Limiting Enables DB I/O Exhaustion

### Summary
`NftService.getNft()` executes `SELECT * FROM nft` for every call to the public `/api/v1/tokens/{tokenId}/nfts/{serialNumber}` endpoint, fetching all columns including the `metadata` field. The REST API has no rate-limiting middleware, allowing any unauthenticated attacker to flood the shared database with concurrent point-lookup queries across arbitrary `tokenId`/`serialNumber` pairs, causing sustained I/O pressure on the database instance shared by all mirror-node processing components.

### Finding Description

**Exact code location:**

`rest/service/nftService.js`, line 20 — static query definition:
```js
static nftByIdQuery = `select * from nft where ${Nft.TOKEN_ID} = $1 and ${Nft.SERIAL_NUMBER} = $2`;
``` [1](#0-0) 

`getNft()` at lines 38–41 executes this query unconditionally for every request:
```js
async getNft(tokenId, serialNumber) {
  const {rows} = await pool.queryQuietly(NftService.nftByIdQuery, [tokenId, serialNumber]);
  return isEmpty(rows) ? null : new Nft(rows[0]);
}
``` [2](#0-1) 

**Root cause — two compounding issues:**

1. `SELECT *` fetches every column in the `nft` table, including `metadata` (a `bytea` column), `timestamp_range`, and all other fields, even though only a subset is needed. The `Nft` model maps exactly these columns: [3](#0-2) 

2. A full audit of `rest/middleware/` reveals **no rate-limiting middleware** anywhere in the REST API stack. The middleware directory contains only: `authHandler.js`, `httpErrorHandler.js`, `metricsHandler.js`, `openapiHandler.js`, `requestHandler.js`, `requestNormalizer.js`, `responseCacheHandler.js`, `responseHandler.js`.


The throttle/rate-limit infrastructure that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` Java module and does **not** protect the Node.js REST API: [4](#0-3) 

**Exploit flow:**

The public OpenAPI spec confirms the endpoint requires no authentication: [5](#0-4) 

An attacker issues thousands of concurrent `GET /api/v1/tokens/{tokenId}/nfts/{serialNumber}` requests with distinct `(tokenId, serialNumber)` pairs. Each request causes `pool.queryQuietly` to issue a `SELECT *` to the shared PostgreSQL instance. Because there is no per-IP, per-user, or global request cap in the REST layer, the database connection pool is saturated and disk/network I/O is driven high for all consumers of that shared DB.

**Why existing checks are insufficient:**

- `requestHandler.js` only parses and canonicalizes query strings; it applies no rate limit. [6](#0-5) 
- `responseCacheHandler.js` may cache responses for repeated identical requests, but distinct `(tokenId, serialNumber)` pairs bypass the cache entirely, meaning every unique pair hits the DB.
- Input validation (`buildAndValidateFilters`) only validates parameter types/formats; it does not throttle request volume.

### Impact Explanation

The PostgreSQL database is shared infrastructure for all mirror-node processing components. Sustained high-volume `SELECT *` queries — each pulling `account_id`, `created_timestamp`, `delegating_spender`, `deleted`, `metadata`, `serial_number`, `spender`, `timestamp_range`, `token_id` — increase buffer pool churn, I/O bandwidth, and connection slot consumption. If the attacker drives DB I/O to saturation, ingestion pipelines and other REST/gRPC services that depend on the same DB instance degrade or stall, consistent with a ≥30% processing-node impact scenario.

### Likelihood Explanation

No authentication, no rate limiting, and a fully public endpoint make this trivially reachable by any external actor. The attacker needs only an HTTP client and a list of valid or guessed `(tokenId, serialNumber)` pairs (token IDs and serial numbers are sequential integers, enumerable from the public `/api/v1/tokens` and `/api/v1/tokens/{tokenId}/nfts` endpoints). The attack is repeatable and requires no special privileges or credentials.

### Recommendation

1. **Add rate-limiting middleware to the REST API** (e.g., `express-rate-limit`) applied globally and/or per-route, enforcing a per-IP and global request-per-second cap before queries reach the DB.
2. **Replace `SELECT *` with explicit column projection** in `nftByIdQuery` to match the columns actually consumed by the `Nft` constructor, reducing per-row data transfer:
   ```js
   static nftByIdQuery = `select account_id, created_timestamp, delegating_spender,
     deleted, metadata, serial_number, spender, timestamp_range, token_id
     from nft where token_id = $1 and serial_number = $2`;
   ```
3. **Enforce DB connection pool limits** with a hard cap on concurrent connections from the REST service, so pool exhaustion does not cascade to other consumers.
4. **Add a response cache** (or extend the existing `responseCacheHandler`) for the single-NFT lookup endpoint, since NFT state changes infrequently.

### Proof of Concept

```bash
# Enumerate valid token IDs from the public API
curl https://<mirror-node>/api/v1/tokens?limit=100 | jq '.[].token_id'

# Flood getNft with concurrent requests across distinct (tokenId, serialNumber) pairs
seq 1 10000 | xargs -P 500 -I{} \
  curl -s "https://<mirror-node>/api/v1/tokens/0.0.1234/nfts/{}" -o /dev/null

# Observe DB I/O metrics spike; other API endpoints and ingestion pipelines degrade
```

Each request bypasses any cache (distinct serial numbers), hits `SELECT * FROM nft WHERE token_id=$1 AND serial_number=$2` directly, and with 500 parallel workers sustains hundreds of concurrent DB queries with no server-side throttle to stop them.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
