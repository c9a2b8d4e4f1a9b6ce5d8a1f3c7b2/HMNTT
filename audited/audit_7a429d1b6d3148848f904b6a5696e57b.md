### Title
Unauthenticated Steady-Rate Flooding via Permanently Unimplemented `id` Input Path in `account` Query

### Summary
The `id` field in `AccountInput` accepts values matching `^[a-zA-Z0-9+/=]+$` via `@Pattern` validation, but the `AccountController.account()` method unconditionally throws `IllegalStateException("Not implemented")` for every such request. Because the GraphQL module has no rate limiting (unlike the `web3` module), any unauthenticated external user can send a sustained stream of these queries, each of which passes schema validation, enters the execution phase, and triggers exception-handling overhead — with no server-side throttle to stop them.

### Finding Description

**Exact code path:**

`account.graphqls` line 99 defines the `id` field with a permissive pattern:
```
id: ID @Pattern(regexp: "^[a-zA-Z0-9+/=]+$")
``` [1](#0-0) 

`AccountController.java` line 57 is reached whenever `id` is the sole non-null field (all other branches are skipped):
```java
throw new IllegalStateException("Not implemented");
``` [2](#0-1) 

`CustomExceptionResolver` catches `IllegalStateException` and converts it to a `GraphQLError`, meaning the server returns HTTP 200 with an error body — no crash, but full execution overhead is consumed on every request: [3](#0-2) 

**Preparsed document cache — partial mitigation only:**

`CachedPreparsedDocumentProvider` caches parsed+validated documents keyed by the raw query string. For a fixed query string, parsing and validation are skipped after the first hit. However, the **execution phase** (controller invocation, `validateOneOf`, exception throw, exception resolver) runs on every request regardless of cache state: [4](#0-3) 

An attacker can also vary the literal `id` value (e.g., `"A"`, `"B"`, `"AA"`, …) to generate distinct cache keys, polluting the 1000-entry Caffeine cache and forcing repeated parse+validate cycles for each unique string: [5](#0-4) 

**No rate limiting in the GraphQL module:**

`GraphQlConfiguration` configures only query complexity (max 200) and depth (max 10) limits. There is no per-IP or global request-rate throttle: [6](#0-5) 

The `ThrottleConfiguration`/`ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module and is not applied to the GraphQL endpoint. [7](#0-6) 

**Root cause and failed assumption:**

The `id` field was added to the schema as a forward-looking placeholder but was never implemented. The code assumes that `id`-only requests will not reach production, but the schema exposes the field publicly with no authentication gate, no rate limit, and a permissive regex that any client can satisfy.

### Impact Explanation

Every request with a valid `id` value (e.g., `"A"`) forces: HTTP deserialization → cache lookup → GraphQL execution dispatch → controller method → `validateOneOf` → `IllegalStateException` → `CustomExceptionResolver` → response serialization. This is CPU and thread-pool work on every single request. At a sustained rate (e.g., hundreds of requests/second from a single client or a small botnet), this accumulates into measurable thread-pool saturation and increased error-handling overhead. The `GraphQLDataFetcherErrors` Prometheus alert fires at >5% error rate, meaning even moderate flooding is operationally visible. The cache-pollution variant (varied `id` literals) additionally forces repeated regex compilation and parse/validate cycles, amplifying CPU cost. [8](#0-7) 

### Likelihood Explanation

Preconditions: none. No authentication, no API key, no prior knowledge beyond the public GraphQL schema. The exploit is a single `curl` command repeated in a loop. The `id` field and its `@Pattern` are visible via GraphQL introspection. The test suite itself documents the exact triggering value (`"azAZ0123456789+/="`), confirming the path is reachable and always errors: [9](#0-8) 

### Recommendation

1. **Remove the `id` field from `AccountInput`** until it is implemented, or add a schema directive/interceptor that rejects it before execution.
2. **Add rate limiting to the GraphQL module** (e.g., using the same Bucket4j infrastructure already present in `web3`) to cap requests per IP per second.
3. If the `id` field must remain, implement it or replace the `throw new IllegalStateException` with a proper `throw new IllegalArgumentException` (which semantically signals a client error, not a server bug) and document it as unsupported.

### Proof of Concept

```bash
# Single triggering request (always returns "Not implemented" error)
curl -s -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: {id: \"azAZ0123456789+/=\"}) { id } }"}'

# Sustained flood (no auth required, no rate limit)
while true; do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: {id: \"azAZ0123456789+/=\"}) { id } }"}' &
done

# Cache-pollution variant: vary the id literal to force repeated parse+validate
for i in $(seq 1 1000); do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {id: \\\"$(head -c 8 /dev/urandom | base64 | tr -d '\\n')\\\") { id } }\"}" &
done
```

Each request passes `@Pattern` validation, enters the execution phase, and unconditionally throws `IllegalStateException`. With no rate limiting in the GraphQL module, this can be sustained indefinitely by an unauthenticated attacker.
