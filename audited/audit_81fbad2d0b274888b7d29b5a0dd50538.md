### Title
Unbounded Request Body Deserialization in `NetworkStatus()` Enables Memory Exhaustion via Oversized JSON Payloads

### Summary
The `/network/status` endpoint accepts `POST` requests with no application-level body size limit. The `rosetta-sdk-go` controller fully deserializes the request body into a `NetworkRequest` (including an unbounded `Metadata map[string]interface{}` field) before `NetworkStatus()` is ever called. An unauthenticated attacker can send arbitrarily large JSON payloads, forcing the Go JSON decoder to allocate memory proportional to the payload size, causing memory pressure or OOM on the server.

### Finding Description
**Exact code path:**

- `rosetta/app/services/network_service.go`, lines 59â€“88: `NetworkStatus()` receives `*rTypes.NetworkRequest` but the parameter is discarded (`_`). The deserialization has already occurred in the upstream `rosetta-sdk-go` `NetworkAPIController` before this function is reached.
- `rosetta/main.go`, lines 220â€“227: The `http.Server` is configured with only time-based timeouts (`ReadTimeout: 5s`, `ReadHeaderTimeout: 3s`). No `http.MaxBytesReader` is applied anywhere in the middleware chain.
- `rosetta/app/middleware/metrics.go`, lines 76â€“83: The only middleware wrapping the router is metrics instrumentation and tracing â€” neither enforces a body size cap.

**Root cause:** The application relies solely on `ReadTimeout` (5 seconds) as a guard. This is time-based, not size-based. On a high-bandwidth connection (e.g., 1 Gbps within a datacenter or cloud environment), an attacker can transmit hundreds of megabytes within the 5-second window. The `rTypes.NetworkRequest.Metadata` field is typed as `map[string]interface{}`, which the Go `encoding/json` decoder will fully materialize into heap-allocated objects for any valid JSON object, regardless of depth or size.

**Why existing checks fail:**
- `ReadTimeout = 5s`: Time-based only. Does not bound bytes read. A 1 Gbps link delivers ~625 MB in 5 s.
- Traefik `inFlightReq: 5` and `rateLimit: 10/s`: These are Kubernetes/Traefik deployment-specific (`charts/hedera-mirror-rosetta/values.yaml`, lines 152â€“161). They are absent in docker-compose and bare-metal deployments. Even when present, 5 concurrent 100 MB payloads = 500 MB of simultaneous allocations.
- No `http.MaxBytesReader`, no `Content-Length` check, no body size middleware exists anywhere in the rosetta Go codebase (confirmed: zero matches for `MaxBytesReader`, `LimitReader`, `maxBodySize` in `rosetta/**/*.go`).

### Impact Explanation
An attacker can force the rosetta server process to allocate arbitrarily large heap memory per request. With multiple concurrent connections (no per-IP concurrency limit at the application layer), this can exhaust available RAM, trigger Go's GC to thrash, or cause an OOM kill of the process. The result is denial of service for all legitimate users of the Rosetta API. No authentication, API key, or special privilege is required. The impact is classified as griefing/DoS with no on-chain economic damage, consistent with the medium severity label.

### Likelihood Explanation
Any external user with network access to port 5700 (or the proxied rosetta endpoint) can exploit this. The attack requires only a single HTTP client capable of sending a large POST body. It is trivially scriptable, repeatable, and requires no prior knowledge of the system beyond the public Rosetta API specification. In non-Kubernetes deployments (docker-compose, bare metal), there are zero rate-limiting or concurrency controls at any layer.

### Recommendation
Apply `http.MaxBytesReader` in a middleware wrapper before the router processes any request body. A reasonable limit for Rosetta endpoints (which carry small structured payloads) is 1â€“4 MB:

```go
func BodySizeLimitMiddleware(maxBytes int64) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
            next.ServeHTTP(w, r)
        })
    }
}
```

Insert this into the middleware chain in `rosetta/main.go` before `metricsMiddleware`, and return HTTP 413 on overflow. This is a size-based limit independent of connection speed, unlike `ReadTimeout`.

### Proof of Concept
```bash
# Generate a ~50 MB JSON payload with a large metadata field
python3 -c "
import json, sys
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
  'metadata': {'x': 'A' * 50_000_000}
}
sys.stdout.write(json.dumps(payload))
" > big_payload.json

# Send to the NetworkStatus endpoint (no auth required)
curl -s -o /dev/null -w "%{http_code} %{size_upload}\n" \
  -X POST http://<rosetta-host>:5700/network/status \
  -H "Content-Type: application/json" \
  --data-binary @big_payload.json

# Repeat concurrently to amplify memory pressure
for i in $(seq 1 10); do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/network/status \
    -H "Content-Type: application/json" --data-binary @big_payload.json &
done
wait
```

Each request forces the Go JSON decoder to allocate ~50 MB on the heap before `NetworkStatus()` is called. Ten concurrent requests = ~500 MB of simultaneous allocations, achievable well within the 5-second `ReadTimeout` on any reasonable network link.
