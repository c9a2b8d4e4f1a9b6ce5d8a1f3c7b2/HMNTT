### Title
`requestTimeout` Not Enforced for EVM Precompile Execution — CPU Exhaustion DoS via Unauthenticated Precompile Calls

### Summary
The `requestTimeout` in `Web3Properties` is enforced exclusively through a Hibernate `StatementInspector` that fires only at database query boundaries. Native EVM precompile calls (ECRecover, ModExp, altBN128Pairing, BLAKE2F, etc.) are pure CPU operations that never issue SQL statements, so the timeout is never checked during their execution. An unauthenticated attacker can send requests at the full `requestsPerSecond` rate (default 500/s), each targeting a computationally expensive precompile at maximum gas, holding server threads indefinitely and saturating CPU capacity.

### Finding Description
**Timeout enforcement — DB-only:**
`HibernateConfiguration.statementInspector()` reads `web3Properties.getRequestTimeout()` (default 10 s) and checks elapsed time only when Hibernate is about to execute a SQL statement:

```java
// HibernateConfiguration.java:31-47
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis();
    return sql -> {
        if (!ContractCallContext.isInitialized()) { return sql; }
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {
            throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
        }
        return sql;
    };
}
```

For a direct call to a native precompile address, the EVM dispatches execution entirely within the Besu precompile implementation — no Hibernate/DB calls are made. The `StatementInspector` is never invoked, so `requestTimeout` is never checked.

**Execution path for precompile calls (no DB touch):**
`ContractController.call()` → `ThrottleManagerImpl.throttle()` (pre-execution check only) → `ContractExecutionService.processCall()` → `ContractCallService.doProcessCall()` → `transactionExecutionService.execute()` → Besu EVM dispatches to precompile → returns result. Zero SQL statements issued; zero timeout checks.

**Throttle checks are pre-execution and do not bound wall-clock time:**
`ThrottleManagerImpl.throttle()` checks `rateLimitBucket` (500 req/s) and `gasLimitBucket` (scaled gas/s) before execution begins. These are token-bucket counters that limit throughput but place no upper bound on how long a single request's EVM execution runs. The `gasLimitRefundPercent` (default 100%) restores gas tokens after execution, but only after the request completes — it does not interrupt a running request.

**Supported precompile targets confirmed in codebase:**
`ContractCallNativePrecompileTest` confirms direct calls to `Address.ECREC` (0x01), `Address.SHA256` (0x02), `Address.RIPEMD160` (0x03), `Address.MODEXP` (0x05), `Address.ALTBN128_ADD` (0x06), `Address.ALTBN128_MUL` (0x07), `Address.ALTBN128_PAIRING` (0x08), `Address.BLAKE2B_F_COMPRESSION` (0x09) all succeed without authentication.

**No authentication on the endpoint:**
`ContractController` at `/api/v1/contracts/call` accepts any unauthenticated POST request.

### Impact Explanation
An attacker targeting `0x08` (altBN128Pairing) with `gas=15_000_000` can request ≈440 pairing operations per call (gas cost = 45,000 + 34,000×k). Each pairing is a multi-millisecond elliptic curve bilinear map. At 500 req/s, hundreds of concurrent threads are held in CPU-bound precompile computation. The server's Tomcat thread pool (default 200 threads) is exhausted, and legitimate requests receive 503 or queue indefinitely. The `requestTimeout` — the intended safety net — never fires because no SQL is executed. This is a complete denial of service achievable by a single unauthenticated client.

### Likelihood Explanation
No credentials, API keys, or special network access are required. The attack requires only knowledge of EVM precompile addresses (publicly documented in the Ethereum Yellow Paper) and the ability to send HTTP POST requests. The exploit is trivially scriptable with `curl` or any HTTP client. The default `requestsPerSecond=500` provides a generous attack budget. The vulnerability is repeatable and persistent as long as the attacker maintains the request rate.

### Recommendation
1. **Enforce `requestTimeout` at the EVM execution layer**, not only at the DB layer. Wrap `transactionExecutionService.execute()` in a `Future` with a timeout (e.g., `ExecutorService.submit(...).get(requestTimeout, MILLISECONDS)`) and interrupt/cancel the thread on expiry.
2. **Add a per-request wall-clock deadline check inside the EVM frame loop** (e.g., via a custom Besu `OperationTracer` that checks `ContractCallContext.getStartTime()` on each opcode and throws if elapsed ≥ timeout).
3. **Apply tighter per-address rate limits** for known expensive precompile addresses (0x05, 0x07, 0x08, 0x09) using the existing `throttle.request[].filters` mechanism with `ActionType.THROTTLE` or `REJECT`.
4. **Reduce `maxGasLimit`** for direct precompile calls, or apply a separate gas cap for precompile-targeted requests.

### Proof of Concept
```bash
# Target altBN128Pairing (0x08) with maximum gas — no authentication needed
# Craft input: 6 field elements × 2 pairs = 384 bytes of valid G1/G2 points
PAYLOAD='{
  "to": "0x0000000000000000000000000000000000000008",
  "gas": "0xE4E1C0",
  "data": "0x<valid_pairing_input_bytes>"
}'

# Send at requestsPerSecond rate (500/s) from a single client
for i in $(seq 1 500); do
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait
# Repeat every second — server threads saturate, legitimate requests time out
```

**Verification**: Monitor server thread pool utilization and response times for legitimate requests. Observe that no `QueryTimeoutException` is thrown for precompile-targeted requests regardless of execution duration, confirming the timeout bypass.     protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }
