### Title
Silent Empty Response for Dissociated Token Relationships Enables Association-State Ambiguity

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` always queries `token_account` with a hardcoded `associated = true` predicate. When an unprivileged caller supplies `?token.id=eq:X` for a token that the account has dissociated from (row exists with `associated=false`), the SQL returns zero rows and the API responds HTTP 200 with `{"tokens": [], "links": {"next": null}}` — identical to the response for a token the account was never associated with. No error, no flag, and no distinction is surfaced.

### Finding Description

**Exact code path:**

`rest/service/tokenService.js` lines 20–29 define the base query:
```sql
SELECT ... FROM token_account ta
WHERE ta.account_id = $1
AND ta.associated = true
``` [1](#0-0) 

When `token.id=eq:X` is supplied, `extractTokensRelationshipQuery()` pushes it into `inConditions`: [2](#0-1) 

`getInClauseSubQuery()` appends `AND ta.token_id IN (X)` to the already-`associated=true`-filtered base: [3](#0-2) 

The controller then blindly returns whatever the service returns, with no check for whether zero results means "never associated" vs. "dissociated": [4](#0-3) 

**Root cause:** The `associated = true` predicate is unconditional and baked into the static base query string. A `TokenDissociate` transaction sets `associated=false` on the `token_account` row (confirmed in `TokenDissociateTransactionHandler`): [5](#0-4) 

The row is retained in the DB with `associated=false`, but the REST API never exposes it. The failed assumption is that an empty result unambiguously means "no relationship ever existed."

**Exploit flow:**
1. Account A associates with token T → `token_account` row: `(account_id=A, token_id=T, associated=true)`
2. Account A dissociates from token T → row updated to `associated=false`
3. Any caller (no privileges required) queries `GET /accounts/A/tokens?token.id=eq:T`
4. SQL: `WHERE account_id=A AND associated=true AND token_id IN (T)` → 0 rows
5. Response: HTTP 200 `{"tokens": [], "links": {"next": null}}`
6. Indistinguishable from a token T that account A was never associated with

**Why existing checks are insufficient:**

The only guard in `getTokenRelationships()` is `isValidAccount()`, which only verifies the account entity exists: [6](#0-5) 

There is no check for whether a `token_account` row exists with `associated=false` for the queried token, and no differentiated response (e.g., HTTP 404 or a `"associated": false` field) is produced.

The test suite confirms the expected SQL always includes `associated = true` with no alternative path for dissociated state: [7](#0-6) 

### Impact Explanation
Mirror nodes are authoritative data sources for downstream systems — compliance tools, DeFi protocols, and audit pipelines — that query `/accounts/{id}/tokens?token.id=eq:X` to verify whether an account holds or has held a token relationship. The API returns HTTP 200 with an empty token list for both "never associated" and "previously associated, now dissociated" states. Any system that interprets an empty list as "no relationship ever existed" will produce incorrect audit records. An account holder can dissociate from a token and then present the mirror node API response as evidence they were never associated, undermining association-history integrity.

### Likelihood Explanation
The trigger requires zero privileges — any HTTP client can issue the request. The precondition (an account that has dissociated from a token) is a routine, common Hedera operation. The behavior is deterministic and repeatable. Any actor who wants to obscure a prior token association can do so trivially and verify the result immediately.

### Recommendation
1. When `token.id=eq:X` is supplied and the result set is empty, perform a secondary query to check whether a `token_account` row exists for `(account_id, token_id)` with `associated=false`. If found, return the record with an explicit `"associated": false` field rather than an empty list, or return HTTP 404 with a descriptive error distinguishing "no relationship" from "dissociated."
2. Alternatively, remove the hardcoded `associated = true` from the base query and instead include `associated` as a selectable/filterable field, defaulting the filter to `true` but allowing the dissociated state to be surfaced when an exact `token.id=eq:X` filter is present.
3. Add a test case asserting that querying a dissociated token relationship returns a distinguishable response from a never-associated one.

### Proof of Concept
```
# Step 1: Associate account 0.0.1001 with token 0.0.5000 on Hedera mainnet/testnet
# (standard TokenAssociate transaction)

# Step 2: Dissociate account 0.0.1001 from token 0.0.5000
# (standard TokenDissociate transaction)
# DB state: token_account row (account_id=1001, token_id=5000, associated=false) exists

# Step 3: Query the mirror node REST API as any unprivileged user
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/tokens?token.id=eq:0.0.5000"

# Actual response (HTTP 200):
# {"tokens": [], "links": {"next": null}}

# Step 4: Query for a token the account was NEVER associated with
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/tokens?token.id=eq:0.0.9999"

# Actual response (HTTP 200):
# {"tokens": [], "links": {"next": null}}

# Both responses are byte-for-byte identical.
# No consumer can distinguish dissociated from never-associated.
```
