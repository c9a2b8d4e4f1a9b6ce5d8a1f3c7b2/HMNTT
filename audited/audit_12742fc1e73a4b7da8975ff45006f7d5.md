### Title
Global Rate Limit Exhaustion — Single Client Can Starve All Others on `ContractController.call()`

### Summary
The `rateLimitBucket` in `ThrottleManagerImpl.throttle()` is a single, process-wide token bucket shared across all clients with no per-IP partitioning. A single unprivileged caller can continuously consume up to 499 of the 500 available tokens per second, reducing all other clients to at most 1 request per second on the `/api/v1/contracts/call` endpoint. No existing check prevents this.

### Finding Description

**Exact code path:**

`ContractController.call()` (line 40) delegates unconditionally to `throttleManager.throttle(request)`. [1](#0-0) 

Inside `ThrottleManagerImpl.throttle()`, the only per-request check is `rateLimitBucket.tryConsume(1)` — one token deducted per call, no caller identity involved. [2](#0-1) 

The bucket itself is a single global bean constructed from `throttleProperties.getRequestsPerSecond()` (default **500**), with a greedy refill of 500 tokens/second and no per-client partitioning. [3](#0-2) [4](#0-3) 

**Why existing checks fail:**

The `RequestFilter` / `RequestProperties` system (the only other throttle mechanism) can match on `BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, and `VALUE` — there is no `IP` or `SOURCE` field. [5](#0-4) 

An attacker can trivially rotate the `from`, `to`, `data`, and `gas` fields per request, defeating any content-based filter. Even if a static filter were configured, it would not address the global bucket exhaustion.

**Root cause:** The design assumes the global 500 req/s limit is shared fairly across many clients. There is no enforcement mechanism to prevent a single client from claiming an arbitrarily large fraction of that budget.

### Impact Explanation

Any unprivileged user who sends ≥499 POST requests per second to `/api/v1/contracts/call` will consume 99.8% of the global token budget. All other clients are collectively left with ≤1 token/second, causing their requests to receive HTTP 429 (`ThrottleException` → `TOO_MANY_REQUESTS`). This is a complete denial-of-service against the `eth_call` / `eth_estimateGas` surface for all other users while the attacker is active. Because the bucket refills greedily every second, the attack is self-sustaining with no warm-up. [6](#0-5) 

### Likelihood Explanation

Preconditions: none beyond HTTP access to the endpoint. No authentication, no API key, no privileged role is required. The attacker needs only a single machine capable of ~500 HTTP POST requests per second, which is trivially achievable with tools like `wrk`, `hey`, or a simple async HTTP client. The attack is repeatable indefinitely and requires no knowledge of internal state.

### Recommendation

1. **Per-IP rate limiting**: Introduce a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`) so each client has its own token bucket, e.g. 50 req/s per IP.
2. **Add `IP` to `RequestFilter.FilterField`**: Pass the client IP through `ContractCallRequest` (or a separate context object) so operators can configure per-source throttle/reject rules.
3. **Alternatively**, enforce per-IP limits at the ingress layer (nginx `limit_req`, API gateway, or a Spring `HandlerInterceptor`) before requests reach `ThrottleManagerImpl`.

### Proof of Concept

```bash
# Terminal 1 — attacker: flood 499 req/s from a single IP
wrk -t8 -c100 -d60s -s post.lua http://<host>/api/v1/contracts/call
# post.lua sets Content-Type: application/json and a minimal valid body

# Terminal 2 — victim: send one legitimate request per second
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":50000}';
  sleep 1;
done
# Expected: victim receives HTTP 429 on nearly every attempt while attacker is running
```

The victim's requests will be rejected with `429 Too Many Requests` / `"Requests per second rate limit exceeded"` because the global bucket is exhausted by the attacker before the victim's token can be claimed. [7](#0-6)
