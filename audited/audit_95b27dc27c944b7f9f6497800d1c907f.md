### Title
Cache Key Normalization Bypass via Whitespace Variation Enables Unbounded Parse/Validate CPU Exhaustion in GraphQL Service

### Summary
`CachedPreparsedDocumentProvider.getDocumentAsync()` uses the raw, unnormalized query string as the Caffeine cache key. An unauthenticated attacker can send semantically identical queries with minor whitespace variations, each producing a distinct cache key and triggering a full parse-and-validate cycle on every request. No application-level rate limiting exists on the GraphQL endpoint, making this a straightforward CPU exhaustion path.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

`executionInput.getQuery()` is the verbatim query string supplied by the client. The Caffeine cache performs an exact-string lookup, so `{ account { id } }` and `{  account { id } }` are two distinct keys.

**Root cause:** No query normalization (whitespace stripping, AST-based canonical form) is applied before the cache lookup. The failed assumption is that clients will send byte-for-byte identical queries on repeated calls.

**Exploit flow:**
1. Attacker sends `POST /graphql/alpha` with query `{ account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }`.
2. Cache miss → full lex → parse → schema validation → complexity check → depth check.
3. Attacker resends with one extra space: `{  account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }`.
4. Different cache key → another full parse+validate cycle.
5. Attacker loops, varying whitespace position across the up-to-1000 whitespace-token budget, generating effectively unlimited unique cache keys.

**Why existing checks are insufficient:**

- `maxWhitespaceTokens(1000)` (line 34 of `GraphQlConfiguration.java`) limits whitespace *per query* but does not prevent an attacker from generating thousands of distinct queries, each within the limit. [2](#0-1) 
- `maximumSize=1000` on the Caffeine cache means the attacker can also evict all legitimate cached queries, forcing legitimate users to re-parse too. [3](#0-2) 
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run *after* parsing, providing no protection against parse-phase CPU cost. [4](#0-3) 
- The throttle/rate-limiting infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists exclusively in the `web3` module for contract calls; **no equivalent exists for the GraphQL endpoint**. [5](#0-4) 

### Impact Explanation
Every cache-miss request forces the JVM to lex, parse, and validate a GraphQL document against the full schema. Under sustained attack, CPU is consumed entirely by parse/validate work, starving legitimate request processing and causing service degradation or outright unavailability of the GraphQL API. The cache's `maximumSize=1000` also means the attacker can poison the cache with 1000 junk entries, evicting all legitimate cached queries and forcing legitimate users to also re-parse.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable (`POST /graphql/alpha`). No credentials, tokens, or special network access are required. The attack requires only an HTTP client capable of sending POST requests in a loop with trivially varied query strings. It is fully automatable, repeatable, and requires no prior knowledge of the system beyond the public GraphQL endpoint URL.

### Recommendation
1. **Normalize the cache key** before lookup: strip all insignificant whitespace, or serialize the parsed AST to a canonical string and use that as the key. This ensures semantically equivalent queries share a single cache entry.
2. **Add application-level rate limiting** to the GraphQL endpoint, mirroring the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern already used in the `web3` module.
3. Consider **per-IP connection throttling** at the Spring Security or servlet filter layer as a defense-in-depth measure.

### Proof of Concept
```bash
# Baseline – first request, cache miss (expected)
curl -s -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}){balance} }"}'

# Attack loop – each request adds one extra leading space, bypassing cache
for i in $(seq 1 2000); do
  SPACES=$(printf '%*s' "$i" '')
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"${SPACES}{ account(input:{entityId:{shard:0,realm:0,num:2}}){balance} }\"}" &
done
wait
# Observe CPU spike to ~100% on the GraphQL pod; legitimate requests begin timing out.
# Cache recordStats will show near-zero hit rate.
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {

        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
