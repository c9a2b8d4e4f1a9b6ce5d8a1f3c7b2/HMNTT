### Title
Unauthenticated Content-Length Header Spoofing Corrupts `api_request_size_bytes` Histogram

### Summary
`metricsMiddleware()` in `rest/middleware/metricsHandler.js` records the raw `content-length` request header value directly into the `api_request_size_bytes` histogram without verifying it against actual bytes received. Any unauthenticated external client can send a GET request with `Content-Length: 2147483647` and zero body bytes, causing the histogram to record a 2 GB observation per request. Repeated at scale, this permanently corrupts the metric, making dashboards and alerting rules based on `api_request_size_bytes` unreliable.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback within `metricsMiddleware()`:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** The code trusts the client-supplied `Content-Length` header as a proxy for actual request body size. `parseInt(..., 10) || 0` only guards against non-numeric strings (returning 0 for `NaN`); it does not cap the value, validate it against actual bytes read from the socket, or reject implausible values.

**Contrast with response size:** Response size is measured correctly by intercepting `res.write` and `res.end` to count real bytes (lines 176–183). No equivalent real-byte measurement exists for the request side.

**Why body-parser limits don't help:** `express.json()` and `express.urlencoded()` are registered before the metrics middleware (server.js lines 68–73 vs. line 91). However, for GET/HEAD requests — which are the primary API surface — neither body parser inspects or rejects based on `Content-Length` because they only activate on requests with a parseable body (`Content-Type: application/json` or `application/x-www-form-urlencoded`). The metrics middleware therefore runs unconditionally for all GET requests, and the `finish` listener fires and records the spoofed header value.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/accounts HTTP/1.1` with header `Content-Length: 2147483647` and no body.
2. Body parsers pass through (GET, no body).
3. `metricsMiddleware` registers the `finish` listener and calls `next()`.
4. Route handler responds normally (200 OK).
5. `finish` fires; `parseInt('2147483647', 10)` = 2147483647 is recorded into `api_request_size_bytes`.

### Impact Explanation
The `api_request_size_bytes` histogram accumulates falsely inflated observations. Prometheus-based dashboards (e.g., the Grafana panels already present in the repo) and any alerting rules derived from this metric will show the server as receiving gigabyte-sized requests. Operators may make incorrect capacity, scaling, or security decisions. The metric cannot self-correct without a process restart (histogram state is in-memory). This is a pure griefing/observability-corruption impact with no direct economic damage, consistent with the Medium scope.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no knowledge beyond the public API. Any HTTP client (curl, Python requests, etc.) can set arbitrary headers. The attack is trivially repeatable in a loop. The API endpoints are publicly accessible by design.

### Recommendation
Replace the header-trust approach with actual byte counting on the request side, mirroring how response size is already measured:

```js
// In metricsMiddleware, before next():
let requestSize = 0;
req.on('data', (chunk) => { requestSize += chunk.length; });
req.on('end', () => { /* requestSize is now accurate */ });
// Then in res.on('finish', ...) use requestSize instead of the header
```

Alternatively, if header-based measurement is kept for performance reasons, clamp the value to a sane maximum (e.g., the configured body-parser limit, 100 KB by default) and add a validation check:

```js
const MAX_EXPECTED = 100 * 1024; // match express.json() default
const rawLen = parseInt(req.headers['content-length'] ?? '0', 10) || 0;
requestSizeHistogram.record(Math.min(rawLen, MAX_EXPECTED), labels);
```

### Proof of Concept
```bash
# Single request inflating the histogram by ~2 GB
curl -s -X GET https://<mirror-node-host>/api/v1/accounts \
  -H "Content-Length: 2147483647" \
  --max-time 5

# Verify corruption by scraping metrics
curl -s https://<mirror-node-host>/api/v1/metrics/ | \
  grep api_request_size_bytes
# Expected: bucket counts and sum reflect 2147483647-byte observations
# despite zero bytes actually transmitted

# Loop to amplify corruption
for i in $(seq 1 1000); do
  curl -s -X GET https://<mirror-node-host>/api/v1/accounts \
    -H "Content-Length: 2147483647" --max-time 2 &
done
wait
