### Title
Unbounded `string_agg` File Fetch in `getContractById` Enables Memory Exhaustion via Large Contract Bytecode File

### Summary
`getContractById()` unconditionally fetches the full content of any user-created file referenced by a contract's `file_id` using a `string_agg` SQL query with no size cap. Because the `file_id` is resolved from the database record of any publicly queryable contract, an unprivileged attacker who has previously deployed a contract backed by a maximally large file can repeatedly trigger this path to exhaust Node.js heap memory on mirror-node REST processing instances.

### Finding Description
**Exact code path:**

`rest/controllers/contractController.js`, `getContractById()`, lines 731–732:
```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
``` [1](#0-0) 

`isValidUserFileId` in `rest/utils.js` line 445–447 only checks that the numeric entity ID is greater than 1000 (i.e., not a system file). It performs no check on the size of the data stored under that file ID:
```js
const isValidUserFileId = (val) => {
  return !isNil(val) && val !== '' && EntityId.parse(val).num > 1000;
};
``` [2](#0-1) 

`FileDataService.getFileData()` in `rest/service/fileDataService.js` lines 70–75 executes `getFileDataQuery` and returns the entire result: [3](#0-2) 

`getFileDataQuery` (lines 43–62) uses an unbounded `string_agg` that concatenates every `file_data` chunk for the given `entity_id` into a single in-memory buffer with no `LIMIT` or size guard:
```sql
select string_agg(file_data, '' order by consensus_timestamp) data
from file_data
where entity_id = $1 ...
``` [4](#0-3) 

**Root cause / failed assumption:** The code assumes contract bytecode files are small. There is no size check before calling `getFileData()`, no `LIMIT` in the SQL, and no cap on the `string_agg` result. The entire concatenated file content is materialized in PostgreSQL, transferred over the wire, and held as a JavaScript `Buffer` in the Node.js heap for the lifetime of the request.

**Exploit flow:**
1. Attacker submits a `FileCreate` + N × `FileAppend` transactions on the Hedera network, building a file as large as the network permits (Hedera allows files up to ~1 MB per chunk; many appends can produce multi-MB files stored in `file_data`).
2. Attacker submits a `ContractCreate` referencing that file's ID. The mirror node importer records the contract row with `file_id` pointing to the large file.
3. Attacker (or any colluding party) sends concurrent `GET /api/v1/contracts/{contractId}` requests to the mirror-node REST API. No authentication is required.
4. Each request triggers `getFileData()`, causing PostgreSQL to `string_agg` all chunks into one large bytea, which is then loaded into the Node.js process heap.
5. With enough concurrent requests, the Node.js heap is exhausted, crashing or severely degrading the REST processing node.

### Impact Explanation
Each request to `GET /api/v1/contracts/{contractId}` for a contract backed by a large file allocates an unbounded heap buffer in the Node.js process. Concurrent requests multiply this allocation. A sufficiently large file (e.g., several MB) combined with tens of concurrent requests can exhaust the default Node.js heap (typically 1.5–4 GB), causing OOM crashes or severe GC pressure. Because the mirror-node REST tier is stateless and horizontally scaled, an attacker targeting multiple instances simultaneously can degrade or crash ≥30% of processing nodes without brute-force volume — a single well-crafted contract ID is sufficient to trigger the condition on each node.

### Likelihood Explanation
The precondition (deploying a contract backed by a large file) requires only HBAR to pay transaction fees — no privileged access, no special role. HBAR is freely purchasable. The trigger is a standard unauthenticated public REST endpoint. The attack is repeatable indefinitely: the contract and its file persist on-chain, so the attacker can re-trigger the fetch at any time. No exploit tooling beyond `curl` or a simple HTTP client is needed.

### Recommendation
1. **Add a size guard before fetching**: Query `pg_column_size` or `sum(length(file_data))` for the `file_id` before calling `getFileData()`; reject or truncate if it exceeds a configured maximum (e.g., 512 KB).
2. **Cap the SQL result**: Add a PostgreSQL `string_agg` size limit or use a `LIMIT` on the number of chunks fetched.
3. **Return a placeholder for oversized bytecode**: If the file exceeds the threshold, return `null` or a sentinel value in the `bytecode` field rather than the full content.
4. **Rate-limit the `/contracts/{id}` endpoint** per IP/client to reduce the blast radius of repeated large fetches.

### Proof of Concept
```
# Step 1 – Create a large file on Hedera testnet (requires HBAR)
# FileCreate with 1 KB initial content, then FileAppend ~1000 times → ~1 MB file
# Record the resulting file_id, e.g. 0.0.99999

# Step 2 – Create a contract referencing that file
# ContractCreate with file_id=0.0.99999
# Record the resulting contract_id, e.g. 0.0.100000

# Step 3 – Wait for mirror node to ingest the records

# Step 4 – Trigger concurrent large fetches (no auth required)
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/0.0.100000" &
done
wait

# Result: each request causes the REST node to load the full ~1 MB file_data
# string_agg result into Node.js heap; 50 concurrent requests = ~50 MB heap
# allocation per node; scaling up concurrency exhausts available heap.
```
