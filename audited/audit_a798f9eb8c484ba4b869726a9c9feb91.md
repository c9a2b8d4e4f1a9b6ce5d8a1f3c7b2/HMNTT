### Title
ETH_ESTIMATE_GAS Binary Search Computation Amplification via Throttle Bypass at Minimum Gas

### Summary
An unprivileged user can POST to `/api/v1/contracts/call` with `estimate=true` and `gas=21000` (the minimum allowed value). The gas-based throttle consumes only 2 scaled tokens for this gas value — effectively bypassing the gas bucket — while the binary search in `BinaryGasEstimator` performs up to ~14 additional EVM executions per request. This results in approximately 15x more CPU work per request compared to an equivalent `ETH_CALL`, with no additional throttle cost.

### Finding Description

**Code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → `estimateGas()` → `BinaryGasEstimator.search()`

**Step 1 — Throttle consumes tokens once, based on declared gas:**

`ThrottleManagerImpl.throttle()` calls `gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))`. [1](#0-0) 

`scaleGas()` in `ThrottleProperties` divides by `GAS_SCALE_FACTOR = 10_000`:
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [2](#0-1) 

For `gas=21000`: `scaleGas(21000) = Math.floorDiv(21000, 10000) = 2` tokens. The gas bucket capacity is `scaleGas(7_500_000_000) = 750_000` tokens, so 2 tokens is negligible — the gas throttle is effectively bypassed.

**Step 2 — ETH_ESTIMATE_GAS triggers binary search with multiple EVM executions:**

When `estimate=true`, `ContractExecutionService.estimateGas()` first makes one full EVM call, then — if `gasUsedByInitialCall < params.getGas()` — invokes `BinaryGasEstimator.search()` with `lo=gasUsedByInitialCall` and `hi=21000`. [3](#0-2) 

The binary search loop runs while `lo + 1 < hi && iterationsMade < maxGasEstimateRetriesCount` (default 20), calling `doProcessCall` (a full EVM execution) each iteration. [4](#0-3) 

With `hi=21000` and `lo≈0`, the range yields `log2(21000) ≈ 14` iterations before `lo+1 >= hi` terminates the loop. Total EVM executions per request: **1 initial + up to 14 binary search = ~15**.

**Step 3 — Gas tokens are never restored during estimate iterations:**

`ContractCallService.doProcessCall()` only calls `restoreGasToBucket()` when `!estimate`. During all binary search iterations, `estimate=true`, so no gas tokens are ever returned to the bucket. [5](#0-4) 

**Step 4 — Rate limit is the only real protection, and it is equal for both call types:**

The rate limit bucket consumes 1 token per request regardless of `estimate`. With `requestsPerSecond=500` (default), an attacker can send 500 req/s of `ETH_ESTIMATE_GAS` with `gas=21000`, each triggering ~15 EVM executions, for a total of ~7,500 EVM executions/second — vs 500 for `ETH_CALL` at the same gas. [1](#0-0) 

**Root cause:** The throttle model accounts for gas cost once per HTTP request, but `ETH_ESTIMATE_GAS` multiplies the actual EVM execution count by up to `maxGasEstimateRetriesCount + 1`. The gas bucket is not charged for each binary search iteration.

### Impact Explanation

An attacker can sustain ~15x more EVM execution work per second compared to `ETH_CALL` at the same gas value, while consuming the same rate-limit tokens. At `gas=21000`, each individual EVM execution is lightweight, but the amplification is structural and scales with the number of concurrent attackers. Under sustained load from multiple sources, this can exhaust CPU on the web3 service, degrading or denying service to legitimate users. The `gasLimitRefundPercent=100` default means even failed estimate calls restore gas tokens, making the gas bucket irrelevant for this attack vector. [6](#0-5) 

### Likelihood Explanation

No authentication, API key, or special privilege is required. The endpoint is publicly accessible. The attack requires only standard HTTP POST requests with `estimate=true` and `gas=21000`. It is trivially scriptable, repeatable, and can be parallelized across multiple source IPs to bypass per-IP rate limiting if present. The minimum gas constraint (`@Min(21_000)`) is enforced by the `ContractCallRequest` model, confirming `gas=21000` is a valid, accepted input. [7](#0-6) 

### Recommendation

1. **Charge the gas bucket per binary search iteration**, not just once at request entry. In `BinaryGasEstimator.search()`, call `throttleManager.throttle` (or a dedicated gas-deduction method) for each iteration's gas consumption.
2. **Apply a separate, lower rate limit for `ETH_ESTIMATE_GAS` requests** using the existing `RequestFilter`/`RequestProperties` mechanism with `FilterField.ESTIMATE` and `ActionType.THROTTLE`.
3. **Scale the gas token cost for estimate requests** by a multiplier reflecting the expected number of binary search iterations (e.g., multiply by `log2(maxGasLimit / minGas)` ≈ 10) at throttle time in `ThrottleManagerImpl.throttle()`.
4. **Cap `maxGasEstimateRetriesCount`** more aggressively for low-gas requests, since the binary search converges faster with a small range.

### Proof of Concept

```bash
# Attacker sends estimate requests at minimum gas in a tight loop
# No authentication required

TARGET="https://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD='{"to":"0x0000000000000000000000000000000000000001","estimate":true,"gas":21000,"data":"0x"}'

# Single request triggers ~15 EVM executions, consuming only 2 gas-bucket tokens
curl -s -X POST "$TARGET" \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD"

# Flood at rate limit (500 req/s) to maximize amplification
# Each second: 500 requests × ~15 EVM executions = ~7,500 EVM executions
# vs ETH_CALL: 500 requests × 1 EVM execution = 500 EVM executions
for i in $(seq 1 500); do
  curl -s -X POST "$TARGET" \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait
```

**Expected result:** Server CPU usage is ~15x higher than an equivalent `ETH_CALL` flood at the same rate, while the gas throttle bucket is barely touched (2 tokens per request vs. 750,000 capacity).
