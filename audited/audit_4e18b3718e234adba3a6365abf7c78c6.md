### Title
Unauthenticated Multi-Query DoS via `GET /contracts/results?block.hash=` in `getContractResults()`

### Summary
An unprivileged external user can send `GET /api/v1/contracts/results?block.hash=<valid_hash>` to trigger a sequential and parallel chain of up to 5 database queries per request with no rate limiting enforced. Because the `authHandler` middleware does not block unauthenticated callers and no rate-limiting middleware exists anywhere in the stack, an attacker can flood this endpoint to exhaust the PostgreSQL connection pool and deny service to all other users.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 19 registers the route with no authentication guard:
```js
router.getExt('/results', ContractController.getContractResults);
```

`getContractResults()` (`rest/controllers/contractController.js`, lines 1050–1112) calls `extractContractResultsByIdQuery(filters)` (lines 415–542) with no `contractId`, then executes:

**DB Query 1** — inside `extractContractResultsByIdQuery`, when `blockFilter.key === BLOCK_HASH`:
```js
blockData = await RecordFileService.getRecordFileBlockDetailsFromHash(blockFilter.value);
// SQL: SELECT … FROM record_file WHERE hash LIKE $1 LIMIT 1
```
(lines 491–492, `rest/service/recordFileService.js` lines 58–62, 143–147)

**DB Query 2** — because `contractId === undefined`, `optimizeTimestampFilters()` is always called (line 505), which calls `bindTimestampRange()` (`rest/timestampRange.js` lines 19–38), which calls `getFirstTransactionTimestamp()` (lines 46–71) — a DB query on the `transaction` table (cached only after the very first call; every cold-start or reset re-issues it).

**DB Query 3** — back in `getContractResults()` line 1072:
```js
const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
```

**DB Queries 4 & 5** — lines 1083–1086, fired in parallel if rows are returned:
```js
const [ethereumTransactionMap, recordFileMap] = await Promise.all([
  ContractService.getEthereumTransactionsByPayerAndTimestampArray(payers, timestamps),
  RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps),
]);
```

**Root cause / failed assumption:** The design assumes that callers are either rate-limited externally (e.g., by a reverse proxy) or that the query fan-out is acceptable. Neither assumption is enforced in code. The `authHandler` (`rest/middleware/authHandler.js` lines 15–36) only sets a custom response-size limit for authenticated users; it does not reject or throttle unauthenticated requests. A `grep` across all `rest/**/*.js` files finds zero references to any rate-limiting library (`express-rate-limit`, `throttle`, etc.).

### Impact Explanation
Each unauthenticated request with a valid `block.hash` value consumes 4–5 PostgreSQL connections/queries. The PostgreSQL connection pool is finite (`rest/dbpool.js`). A modest flood of concurrent requests (e.g., 50–100 req/s from a single host) will saturate the pool, causing all subsequent queries — including those from legitimate users hitting any other endpoint — to queue or fail. This constitutes a non-network-based DoS: the attacker does not need to saturate bandwidth; they only need to keep the DB busy. The affected endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 640–665), making it trivially discoverable.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The valid `block.hash` format is documented in the OpenAPI spec (lines 5792–5826) and any real block hash from the public ledger works. The attack is fully repeatable, scriptable with a single `curl` loop or any HTTP benchmarking tool, and requires no prior account creation or authentication.

### Recommendation
1. **Add per-IP rate limiting** at the application layer (e.g., `express-rate-limit`) applied before route handlers in `rest/server.js`, targeting at minimum the `/contracts/results` endpoint.
2. **Short-circuit early on missing block data**: the current code already returns `{skip: true}` when `blockData` is null (line 500), but an attacker can supply a hash that *does* exist, so this guard is insufficient.
3. **Require at least one bounded timestamp filter** when no `block.hash` or `block.number` is supplied, to prevent unbounded scans; extend this to cap the number of DB round-trips per request.
4. **Cache block-hash-to-timestamp lookups** (similar to how `getFirstTransactionTimestamp` is cached) to reduce repeated identical DB hits.

### Proof of Concept

**Preconditions:** Network access to the mirror node REST API. One valid block hash (obtainable from `GET /api/v1/blocks`).

**Steps:**
```bash
# Step 1: obtain a valid block hash
HASH=$(curl -s https://<mirror-node>/api/v1/blocks?limit=1 | jq -r '.blocks[0].hash')

# Step 2: flood the endpoint (no credentials needed)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?block.hash=${HASH}" &
done
wait
```

**Result:** Each of the 500 concurrent requests triggers up to 5 DB queries. The PostgreSQL connection pool is exhausted; subsequent requests to any endpoint return connection-pool timeout errors or HTTP 500 responses, achieving denial of service without any authentication.
