### Title
Missing Null-Byte Sanitization on `ContractResult.errorMessage` Causes COPY-Based Batch Insert Failure (DoS)

### Summary
The `errorMessage` field of `ContractResult` is populated directly from the protobuf `ContractFunctionResult.getErrorMessage()` without calling `DomainUtils.sanitize()`, which is the project's own utility for stripping null bytes (0x0000) that PostgreSQL rejects in text columns. When the `BatchInserter` streams this unsanitized string via `COPY … FROM STDIN WITH CSV`, PostgreSQL throws an encoding error, the entire batch fails, and all contract result writes are halted until the importer recovers or restarts.

### Finding Description

**Exact code path:**

`ContractResultServiceImpl.java`, `processContractResult()`, line 251: [1](#0-0) 

```java
contractResult.setErrorMessage(functionResult.getErrorMessage());
```

No call to `DomainUtils.sanitize()` is made here.

**Root cause — the sanitize utility exists but is not applied:**

`DomainUtils.sanitize()` is explicitly documented and implemented to strip null bytes before database insertion: [2](#0-1) 

It is applied to other user-controlled string fields (entity memo in `AbstractEntity`, token name/symbol/memo in `AbstractToken`), but the `errorMessage` field on `ContractResult` is omitted. [3](#0-2) 

**How the batch insert fails:**

`BatchInserter.persistItems()` serializes the entire collection to CSV and streams it directly into a PostgreSQL `COPY … FROM STDIN WITH CSV` command: [4](#0-3) 

PostgreSQL rejects any text value containing the null byte (0x00) with `ERROR: invalid byte sequence for encoding "UTF8": 0x00`. The exception propagates up and is re-thrown as `ParserException`: [5](#0-4) 

**Why existing checks are insufficient:**

There is no validation, length cap, or character-set check on `errorMessage` anywhere in the ingestion pipeline. The `ContractResult` domain object has no custom setter for `errorMessage` (unlike `setBloom`, which does sanitize). The `CsvMapper` configured in `BatchInserter` registers no string-sanitizing serializer; it only handles `byte[]`, `EntityId`, `List`, and `Range` types. [6](#0-5) 

### Impact Explanation

A single poisoned contract transaction halts the entire `contract_result` batch write for the affected block/record-file batch. Because `BatchInserter.persist()` processes all items in one COPY stream, one bad row aborts the whole batch. Downstream consumers (REST API, mirror node queries) lose visibility into all contract results from that batch onward until the importer is manually remediated or the offending record is skipped. This is a non-network-based DoS against the mirror node's contract result persistence layer.

### Likelihood Explanation

Any account on the Hedera/Hiero network can submit a `ContractCall` or `ContractCreate` transaction. When the EVM reverts, the consensus node populates `ContractFunctionResult.errorMessage` from the revert reason. Java's protobuf runtime does not enforce UTF-8 validity on `string` fields; a consensus node that copies raw EVM revert-reason bytes (which are ABI-encoded and contain 0x00 padding bytes) into this field will produce a protobuf string containing null bytes. The mirror node then ingests this verbatim. The attack is repeatable: the attacker simply resubmits the same contract call each time the importer recovers.

### Recommendation

Apply `DomainUtils.sanitize()` to `errorMessage` at the point of assignment in `ContractResultServiceImpl.processContractResult()`:

```java
contractResult.setErrorMessage(DomainUtils.sanitize(functionResult.getErrorMessage()));
```

Alternatively, add a custom setter on `ContractResult` analogous to `setBloom()` that calls `DomainUtils.sanitize()` internally, ensuring all callers (including `ContractResultMigration.java` line 98 and `AbstractBlockTransactionTransformer`) are covered automatically. [7](#0-6) 

### Proof of Concept

1. Deploy a Solidity contract on a Hedera testnet that executes `revert` with a payload containing null bytes (e.g., via inline assembly: `revert(ptr, len)` where the memory at `ptr` contains `0x00`).
2. Submit a `ContractCall` transaction targeting that contract from any unprivileged account.
3. The consensus node records the transaction; `ContractFunctionResult.errorMessage` is populated with a string containing 0x00.
4. The mirror node importer reads the record file and calls `ContractResultServiceImpl.processContractResult()`.
5. Line 251 sets `contractResult.errorMessage` to the unsanitized string.
6. `BatchInserter.persistItems()` serializes the batch to CSV and issues `COPY contract_result(…) FROM STDIN WITH CSV`.
7. PostgreSQL returns `ERROR: invalid byte sequence for encoding "UTF8": 0x00`.
8. `BatchInserter.persist()` catches the exception and throws `ParserException`.
9. All contract results in the batch are lost; the importer stalls or skips the batch depending on error-handling configuration.
10. Repeat step 2 to sustain the DoS.
