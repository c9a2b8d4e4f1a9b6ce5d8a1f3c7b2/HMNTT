### Title
Missing Application-Level Rate Limiting on Hook Storage Key Range Query Endpoint Enables DB Connection Pool Exhaustion

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint in the `rest-java` module has no application-level rate limiting, unlike the `web3` module which uses `ThrottleManagerImpl` with bucket4j. An unauthenticated attacker can flood this endpoint with concurrent full-range key scan requests, each holding a HikariCP database connection for the duration of the query, exhausting the connection pool and denying service to all legitimate users.

### Finding Description
**Code path:**

- `HooksController.getHookStorage()` at [1](#0-0)  accepts unauthenticated GET requests with no rate-limiting annotation or filter.
- When no `key` parameter is supplied, `lowerBound` defaults to `MIN_KEY_BYTES` (32×`0x00`) and `upperBound` to `MAX_KEY_BYTES` (32×`0xFF`) [2](#0-1) , producing a full-table range scan.
- `HookServiceImpl.getHookStorage()` directly calls the repository with those bounds [3](#0-2) .
- `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` issues a `BETWEEN` query holding a DB connection until completion [4](#0-3) .

**Root cause:** The `web3` module has a full rate-limiting stack (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) [5](#0-4) , but the `rest-java` module has no equivalent. No `@RateLimiter`, no `ThrottleManager`, no per-IP filter exists anywhere in `rest-java` for this endpoint. The only application-side constraint is `MAX_LIMIT = 100` rows returned [6](#0-5) , which limits result size but does not limit concurrent connection usage.

### Impact Explanation
Each concurrent request to the storage endpoint acquires a HikariCP connection for the full duration of the range query. The Prometheus alert rule `hikaricp_connections_active > 75%` [7](#0-6)  confirms the connection pool is a recognized finite resource. Once the pool is exhausted, all subsequent requests — including those to unrelated endpoints served by the same `rest-java` instance — queue or fail with timeout errors, constituting a full API denial-of-service. The Traefik `retry: attempts: 3` middleware [8](#0-7)  further amplifies load by automatically retrying failed requests.

### Likelihood Explanation
No authentication is required. The endpoint is publicly routable via the ingress path `/api/v1/accounts/.../hooks/.../storage` [9](#0-8) . The optional GCP gateway `maxRatePerEndpoint: 250` [10](#0-9)  is a global endpoint cap (not per-IP), is only active for GCP deployments, and 250 req/s is sufficient to saturate a default HikariCP pool (typically 10 connections). The attack is trivially repeatable with any HTTP load tool and requires no special knowledge beyond the public API documentation.

### Recommendation
1. Add application-level rate limiting to `rest-java` mirroring the `web3` `ThrottleConfiguration`/`ThrottleManagerImpl` pattern, applied as a servlet filter or Spring interceptor on the `HooksController`.
2. Enforce per-IP rate limiting at the ingress/gateway layer (not just a global endpoint cap).
3. Set a query timeout on the `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse` repository call so long-running range scans release connections promptly.
4. Consider requiring at least one non-trivial key bound (i.e., disallow full `0x00…00` to `0xFF…FF` range scans without authentication or explicit opt-in).

### Proof of Concept
```bash
# Send 500 concurrent full-range scan requests with no key filter
# Each holds a DB connection for the query duration
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage" &
done
wait

# Observe: subsequent legitimate requests return 503 or timeout
curl -v "https://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage"
# Expected: connection timeout or HTTP 503 due to pool exhaustion
```

        hookStorageResponse.setLinks(links);

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
