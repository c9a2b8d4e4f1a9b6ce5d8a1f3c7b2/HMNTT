### Title
Unbounded Per-Subscriber DB Polling in `PollingTopicListener.listen()` Enables Unauthenticated Resource Exhaustion DoS

### Summary
Each call to `PollingTopicListener.listen()` unconditionally creates a new, independent `PollingContext` and Reactor `Flux` pipeline that issues a live database query every 500 ms with no shared throttling or global subscription cap. Because the only server-side guard (`maxConcurrentCallsPerConnection = 5`) is scoped per-connection and there is no limit on the number of connections, an unprivileged attacker can open arbitrarily many gRPC connections and saturate the database connection pool, causing legitimate subscribers to be denied service.

### Finding Description

**Exact code path**

`PollingTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 34–49:

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);          // new context per call
    Duration interval = listenerProperties.getInterval();        // default 500 ms

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)         // infinite loop
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))
            ...
}
```

`poll()` (lines 51–62) calls `topicMessageRepository.findByFilter(newFilter)`, which in `TopicMessageRepositoryCustomImpl.findByFilter()` (lines 33–61) executes a full JPA criteria query, holding a JDBC connection for the duration of each poll cycle.

**Root cause**

There is no global subscription counter enforcement, no per-IP rate limit, and no cap on the total number of active `listen()` pipelines. The `subscriberCount` field in `TopicMessageServiceImpl` (line 48) is wired only to a Micrometer `Gauge` for observability — it is never checked against a maximum and never used to reject new subscriptions (lines 89–90). The sole structural guard is `NettyProperties.maxConcurrentCallsPerConnection = 5` (default), which is applied per-connection by `GrpcConfiguration` (line 33) but places no bound on the number of connections.

**Why existing checks fail**

| Check | Scope | Enforcement |
|---|---|---|
| `maxConcurrentCallsPerConnection = 5` | Per TCP connection | Attacker opens N connections → 5N subscriptions |
| `subscriberCount` AtomicLong | Global counter | Metrics only, never enforced |
| `maxPageSize = 5000` | Per query result | Caps rows returned, not query frequency |
| No IP/auth gate | — | gRPC endpoint is unauthenticated |

The `ThrottleManagerImpl` / `ThrottleConfiguration` rate-limiting infrastructure exists only in the `web3` module and is entirely absent from the `grpc` module.

### Impact Explanation
With the `POLL` listener type active, each attacker-controlled subscription issues one DB query per 500 ms interval. With N open connections × 5 calls each, the attacker generates 10N queries/second. The default HikariCP pool (typically 10 connections) is exhausted at roughly 2 attacker connections. Beyond that threshold, legitimate subscriber queries queue indefinitely or time out, producing a complete denial of service for the HCS topic subscription API — a core mirror-node function used by wallets, explorers, and dApps on the Hiero/Hedera network.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or any HTTP/2 client). No credentials, tokens, or privileged network position are needed. The `subscribeTopic` RPC is publicly documented and reachable on the standard mirror-node gRPC port. The attack is trivially scriptable, repeatable, and requires no knowledge of internal state. The only practical friction is that the default listener type is `REDIS`; however, operators who configure `POLL` (a documented, supported option) are fully exposed.

### Recommendation
1. **Enforce a global subscription cap**: Add an enforced maximum to `TopicMessageServiceImpl.subscribeTopic()` — reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., `grpcProperties.getMaxSubscribers()`).
2. **Add per-IP / per-connection subscription rate limiting**: Introduce a `ServerInterceptor` in the gRPC module (analogous to `ThrottleManagerImpl` in `web3`) that tracks and limits subscription attempts per remote address.
3. **Limit total gRPC connections**: Configure `NettyServerBuilder.maxConnectionAge` / `maxConnections` in `GrpcConfiguration` to bound the total number of simultaneous connections.
4. **Consider `SHARED_POLL` or `REDIS` as the only production-safe modes**: `SharedPollingTopicListener` uses a single shared `Flux` with `.share()`, meaning all subscribers share one polling loop regardless of subscriber count — the correct design for multi-subscriber scenarios.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
