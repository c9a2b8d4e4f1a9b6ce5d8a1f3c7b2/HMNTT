### Title
Deleted Entity Disclosure via Long-Zero EVM Address Bypassing `deleted is not true` Filter

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes long-zero EVM addresses (bytes 0–11 all zero, bytes 12–19 = numeric entity ID) to `entityRepository.findById()`, which carries no deletion filter. The `findByEvmAddress()` path explicitly enforces `deleted is not true`, but the `findById()` fallback does not, allowing any unprivileged caller to retrieve a deleted entity's full record by supplying a crafted long-zero address.

### Finding Description
**Exact code path:**

`EntityServiceImpl.java` lines 34–41:
```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-11 == 0
        return entityRepository.findById(buffer.getLong())         // bytes 12-19 = entity ID
                               .filter(e -> e.getType() == type);  // NO deleted check
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // has "deleted is not true"
                           .filter(e -> e.getType() == type);
}
```

`EntityRepository.java` line 16–17:
```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

**Root cause:** The long-zero branch calls Spring Data's generic `CrudRepository.findById(Long)`, which issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate. The `Entity` class carries no Hibernate `@Where` clause to inject that filter automatically. The only guard is the `.filter(e -> e.getType() == type)` chain, which checks entity type but not deletion status.

**Failed assumption:** The developer assumed long-zero addresses always correspond to live system accounts (0.0.X), but any numeric entity ID—including IDs of deleted accounts, contracts, or tokens—can be encoded in this format.

### Impact Explanation
An attacker can retrieve the full `Entity` row (memo, keys, auto-renew settings, balance snapshot, `deleted = true` flag, etc.) for any entity that has been deleted from the ledger. Through the GraphQL API this entity object is the root resolver for related data (e.g., transaction history queries keyed on entity ID). Exposing deleted entities violates the ledger's soft-delete semantics and leaks historical account/contract metadata to unauthenticated callers. Severity is **Medium**: no funds can be moved, but confidential entity metadata and associated history become readable after deletion.

### Likelihood Explanation
Exploitation requires zero privileges—only knowledge of a target entity's numeric ID, which is publicly derivable from any prior transaction referencing that entity. The long-zero address format is documented Hedera behavior (EVM alias for `0.0.<num>`). The attack is trivially repeatable via a single GraphQL query and requires no special tooling.

### Recommendation
Replace the bare `findById` call with a query that enforces the same deletion filter:

```java
// Option A: add a filtered method to EntityRepository
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);

// Option B: post-filter in the service (weaker, but immediate)
return entityRepository.findById(buffer.getLong())
    .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
    .filter(e -> e.getType() == type);
```
Option A is preferred because it keeps the deletion invariant at the repository layer, consistent with `findByEvmAddress` and `findByAlias`.

### Proof of Concept
1. Identify a deleted entity with numeric ID `N` (e.g., `N = 12345`).
2. Construct a 20-byte long-zero EVM address: `0x000000000000000000000000000000000000303900` (hex of `N` in the last 8 bytes, zero-padded).
3. Send a GraphQL query:
   ```graphql
   query {
     account(input: { evmAddress: "0x0000000000000000000000000000000000003039" }) {
       id
       memo
       deleted
       transactions { ... }
     }
   }
   ```
4. Observe that the deleted entity is returned with `deleted: true` and its full metadata/history, whereas the same entity queried by its actual `evm_address` column value returns null (filtered by `deleted is not true`).
