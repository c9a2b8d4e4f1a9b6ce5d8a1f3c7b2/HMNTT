### Title
Unbounded Global Subscription Count Exhausts Shared `boundedElastic` Scheduler in `PollingTopicMessageRetriever.retrieve()`

### Summary
`PollingTopicMessageRetriever.retrieve()` contains no global concurrent-subscription cap. Every subscription schedules repeat-delay and timeout tasks on a single shared `Schedulers.boundedElastic()` instance. Because the only per-connection limit (`maxConcurrentCallsPerConnection=5`) is not a global cap, an unprivileged attacker opening many TCP connections can saturate the scheduler's task queue (default 100,000 tasks), causing `RejectedExecutionException` for legitimate subscribers and degrading service for all users.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructor (line 41) creates one shared scheduler: [1](#0-0) 

`retrieve()` (lines 45–63) attaches every subscription's repeat-delay and timeout to that same scheduler with no guard: [2](#0-1) 

When `throttled=true` (the path taken by `TopicMessageServiceImpl.subscribeTopic()` for historical retrieval), `numRepeats = Long.MAX_VALUE`, so the subscription polls indefinitely: [3](#0-2) 

`TopicMessageServiceImpl.subscribeTopic()` calls `retrieve(filter, true)` unconditionally and tracks `subscriberCount` only as a metrics gauge — there is no enforcement check: [4](#0-3) [5](#0-4) 

**Root cause:** The only connection-level guard is `maxConcurrentCallsPerConnection=5`, which is a *per-connection* Netty limit: [6](#0-5) [7](#0-6) 

There is no `maxConnections` configured in `GrpcConfiguration`, so an attacker can open an arbitrary number of TCP connections. With `C` connections, the attacker can hold `5C` concurrent subscriptions. Each subscription enqueues periodic delay tasks and a timeout task on the single shared `boundedElastic` scheduler. Reactor's default `boundedElastic` has `10 × CPU_cores` threads and a task queue of 100,000 tasks. When the queue is exhausted, new tasks throw `RejectedExecutionException`, which propagates as errors to legitimate subscribers.

### Impact Explanation

When the shared scheduler's task queue fills, legitimate subscriptions fail with `RejectedExecutionException` on their next repeat-delay or timeout scheduling attempt. This causes service-wide degradation of the gRPC topic subscription API for all users. The impact is denial-of-service at the scheduler level — no financial loss, but complete unavailability of the `subscribeTopic` endpoint for honest clients. Severity is Medium (griefing/DoS with no economic damage).

### Likelihood Explanation

The gRPC port (5600) is publicly accessible with no authentication required to call `subscribeTopic`. An attacker needs only a standard gRPC client and the ability to open many TCP connections (trivially achievable from a single host or a small botnet). The attack is repeatable: after connections are dropped (e.g., by a timeout), the attacker simply reconnects. No special knowledge, credentials, or on-chain resources are required.

### Recommendation

1. **Add a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum before proceeding, returning `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a global connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge()` / `maxConnections()` or an equivalent load-balancer rule.
3. **Use a dedicated, size-bounded scheduler** per retriever (or a shared one with an explicit thread/queue cap) rather than the default unbounded-queue `boundedElastic`.
4. **Add a per-IP connection rate limit** at the ingress/load-balancer layer.

### Proof of Concept

```
# Precondition: gRPC port 5600 reachable, at least one valid topic exists (e.g., 0.0.1234)

# Step 1: Open 200 TCP connections, each with 5 concurrent subscribeTopic streams
#         (total: 1000 subscriptions, each throttled=true → numRepeats=Long.MAX_VALUE)
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1234},"consensusStartTime":{"seconds":0}}' \
    -H "grpc-timeout: 300S" \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  # open 5 streams per connection by repeating in same connection context
done

# Step 2: Observe that after ~100K scheduler tasks are queued,
#         new legitimate subscribeTopic calls fail with:
#         "INTERNAL: reactor.core.Exceptions$ReactorRejectedExecutionException:
#          Scheduler unavailable"
```
