### Title
Unauthenticated DB Connection Pool Exhaustion via Uncached `/api/v1/network/stake` Endpoint

### Summary
`NetworkServiceImpl.getLatestNetworkStake()` calls `networkStakeRepository.findLatest()` on every request with no caching, no rate limiting, and no authentication. The underlying query is a correlated subquery that runs two full-table scans on `network_stake` per request. A sustained flood of unauthenticated GET requests to `/api/v1/network/stake` can exhaust the shared HikariCP connection pool, starving all other service operations of DB connections and causing the mirror node service to become unresponsive.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (line 127-130) → `NetworkServiceImpl.getLatestNetworkStake()` (lines 52-56) → `NetworkStakeRepository.findLatest()` (lines 12-19).

Every HTTP request to `GET /api/v1/network/stake` executes this native SQL query unconditionally:
```sql
select * from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
```
This is a correlated subquery requiring two passes over `network_stake` per call.

**Root cause — failed assumption:** The code assumes request volume will be low enough that per-request DB queries are acceptable. No `@Cacheable` annotation exists anywhere in `rest-java/src/main/java/` (confirmed: grep for `Cacheable|CacheEvict|CachePut|EnableCaching` returns zero matches in the rest-java service layer). No rate limiter (bucket4j, resilience4j, or similar) is present in the rest-java module. The endpoint requires no authentication.

**Contrast with grpc module:** The grpc module explicitly caches equivalent stake data via `@Cacheable(cacheManager = NODE_STAKE_CACHE, cacheNames = CACHE_NAME)` on `NodeStakeRepository.findAllStakeByConsensusTimestamp()` with a dedicated `CaffeineCacheManager`. The rest-java module has `spring-boot-starter-cache` as a dependency but applies it nowhere for this path.

**Exploit flow:**
1. Attacker sends concurrent `GET /api/v1/network/stake` requests (no auth required, no parameters needed).
2. Each request acquires a HikariCP connection and executes the correlated subquery.
3. With default HikariCP pool size (~10 connections), ~10 concurrent requests saturate the pool.
4. Subsequent requests (including importer writes, other REST endpoints) queue waiting for a connection.
5. HikariCP connection timeout fires → requests fail with 500 errors across the entire service.

### Impact Explanation
DB connection pool exhaustion is a shared resource: all service operations (importer ingestion, other REST endpoints) draw from the same pool. Saturating it via this single unauthenticated endpoint degrades or halts the entire mirror node service instance. In a horizontally scaled deployment, each instance is independently vulnerable; an attacker targeting multiple instances simultaneously can take down a significant fraction of the serving fleet. The `network_stake` table grows over time (one row per staking period, daily), making the `max()` subquery progressively more expensive.

### Likelihood Explanation
Preconditions: none. The endpoint is public, requires no API key, no session, no parameters. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can sustain hundreds of concurrent requests from a single machine. The attack is trivially repeatable and requires no special knowledge beyond the public OpenAPI spec, which documents `GET /api/v1/network/stake` with no security requirements.

### Recommendation
1. **Add result caching** on `NetworkStakeRepository.findLatest()` or in `NetworkServiceImpl.getLatestNetworkStake()` using `@Cacheable` with a short TTL (e.g., 60 seconds, matching the staking period update frequency). Mirror the pattern already used in the grpc module's `NodeStakeRepository`.
2. **Add per-IP rate limiting** at the servlet filter or API gateway level for all unauthenticated endpoints.
3. **Tune HikariCP** to set a short `connectionTimeout` and `maximumPoolSize` appropriate for the expected load, and consider a separate read-only pool for public endpoints.

### Proof of Concept
```bash
# Flood the endpoint with 50 concurrent connections, 10000 total requests
# No authentication, no parameters required
wrk -t50 -c50 -d60s http://<mirror-node-host>/api/v1/network/stake

# Observe: after pool exhaustion (~10 concurrent DB queries held),
# all subsequent requests to ANY endpoint return 500 (HikariCP timeout)
# Monitor with: SELECT count(*) FROM pg_stat_activity WHERE state = 'active';
```

Expected result: DB active connection count pegged at pool maximum; other endpoints (`/api/v1/network/supply`, `/api/v1/accounts/*`, etc.) begin returning 500 errors due to connection acquisition timeout.
