### Title
Unprivileged User Can Trigger Unthrottled DB Polling via Safety-Check Gap-Recovery Path

### Summary
Any unauthenticated gRPC subscriber can force the mirror node into the unthrottled polling code path by subscribing to any valid topic with no `endTime`. After one second, the safety-check in `TopicMessageServiceImpl` unconditionally invokes `topicMessageRetriever.retrieve(gapFilter, false)`, spawning a `PollingContext` with `numRepeats=maxPolls` (default 12), `frequency=20ms`, and `maxPageSize=5000`. With no rate limiting on concurrent subscriptions, an attacker can sustain amplified DB load by repeatedly opening connections.

### Finding Description

**Exact code path:**

In `TopicMessageServiceImpl.subscribeTopic()` (lines 67–70), a safety-check flux is unconditionally scheduled for every subscription:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

`topicContext.isComplete()` (lines 203–215) returns `false` whenever `filter.getEndTime() == null`:

```java
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;
    }
    ...
}
```

So for any subscription without an `endTime` (the common case for live subscriptions), the safety check always fires. It calls `missingMessages(topicContext, null)` (line 69), which at line 149 unconditionally passes `throttled=false`:

```java
return topicMessageRetriever.retrieve(gapFilter, false);
```

In `PollingTopicMessageRetriever`, `PollingContext(filter, false)` (lines 102–107) sets:

```java
numRepeats = unthrottled.getMaxPolls();       // default: 12
frequency = unthrottled.getPollingFrequency(); // default: 20ms
maxPageSize = unthrottled.getMaxPageSize();    // default: 5000
```

The unthrottled `isComplete()` (lines 126–128) only terminates early on `limitHit`. If the user's filter has no limit (`hasLimit()` returns false), `isComplete()` always returns `false`, so all 12 polls execute at 20ms intervals with page size 5000.

**Root cause:** The safety-check path has no guard — it fires for every subscription with no `endTime`, and the `missingMessages(ctx, null)` branch hardcodes `throttled=false` with no access control, rate limiting, or per-user quota.

**Why existing checks fail:** `topicExists()` (lines 94–106) only validates that the entity exists and is of type `TOPIC`. There is no per-user subscription limit, no cap on concurrent unthrottled retrievers, and no check before entering the `current == null` branch of `missingMessages`.

### Impact Explanation

Each subscription triggers one unthrottled burst: 12 DB queries at 20ms intervals, each fetching up to 5,000 rows. With N concurrent subscriptions (no enforced limit), an attacker generates N simultaneous unthrottled retrievers. This exhausts the DB connection pool, starves legitimate queries (including fee-schedule topic ingestion reads), and degrades mirror node availability. The `subscriberCount` gauge (lines 52–55) is purely observational and imposes no limit.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication beyond topic existence. The attacker needs only a valid topic ID (trivially obtained from any Hedera explorer) and the ability to open many concurrent gRPC streams. No special privileges, credentials, or insider knowledge are required. The attack is repeatable: after each 240ms burst completes, the attacker reconnects and repeats.

### Recommendation

1. **Rate-limit unthrottled retrievals per client/IP**: Track concurrent unthrottled retriever instances and reject or queue excess requests.
2. **Guard the safety-check trigger**: Before calling `missingMessages(topicContext, null)`, verify that the subscription has been active long enough and that no unthrottled retriever is already running for this context.
3. **Add a per-subscriber cap on concurrent subscriptions**: Enforce a maximum number of active subscriptions per remote address in `subscribeTopic`.
4. **Make the safety-check conditional on actual gap evidence**: Only trigger gap recovery if `topicContext.getLast()` is non-null and a real sequence gap is detected, rather than firing unconditionally after 1 second.

### Proof of Concept

1. Obtain any valid topic ID (e.g., from a public Hedera explorer).
2. Open N concurrent gRPC streams calling `subscribeTopic` with:
   - `topicId` = valid topic
   - `startTime` = current time
   - `endTime` = not set (omitted)
   - `limit` = not set (omitted)
3. Wait 1 second. Each stream's safety-check fires, calling `missingMessages(ctx, null)` → `retrieve(gapFilter, false)`.
4. Each spawned `PollingContext` executes 12 polls at 20ms with page size 5000 against the DB.
5. Observe DB connection pool saturation and increased query latency for all other mirror node operations.
