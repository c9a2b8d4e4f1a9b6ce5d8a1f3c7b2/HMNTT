### Title
Unauthenticated Connection Pool Exhaustion via Concurrent Uncached Token Lookups in `getCachedTokens()`

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` performs no concurrency control between the in-memory cache check and the subsequent database query. Because the REST Node.js service has no rate limiting or request-concurrency throttle, an unauthenticated attacker can send 10+ simultaneous requests referencing accounts with uncached token IDs, causing each request to independently acquire and hold one of the pool's 10 default connections for the full `statementTimeout` duration (20 s), exhausting the pool and causing all subsequent DB-dependent requests — including transaction confirmation queries — to fail with connection-timeout errors.

### Finding Description

**Exact code path:**

`rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161: [1](#0-0) 

The function checks the `quickLru` in-memory cache (line 140), collects uncached IDs (line 144), then unconditionally fires a DB query (line 152) with no mutex, promise-coalescing, or single-flight guard. There is no mechanism to detect that another concurrent request is already fetching the same uncached tokens.

`baseService.js` `getRows()` (line 55–57) delegates directly to the global `pool.queryQuietly()`: [2](#0-1) 

The pool is configured in `rest/dbpool.js` with `max: config.db.pool.maxConnections`: [3](#0-2) 

The documented default for `maxConnections` is **10**, `statementTimeout` is **20 000 ms**, and `connectionTimeout` is **20 000 ms**: [4](#0-3) 

**Root cause / failed assumption:** The design assumes the in-memory LRU cache will absorb repeated lookups, but it provides no protection against the first wave of concurrent requests that all arrive before any of them has populated the cache. There is no promise-coalescing ("single-flight") pattern and no per-request or per-IP concurrency limit in the REST Node.js service.

**Exploit flow:**
1. Attacker identifies (or enumerates) 10+ account IDs that each hold token associations not yet in the LRU cache (cold start, or after cache eviction due to the 100 000-entry LRU limit).
2. Attacker fires 10+ concurrent HTTP GET requests to `/api/v1/accounts/{id}/tokens`.
3. Each request enters `getTokenAccounts()` → `getCachedTokens()`. All find the same tokens uncached and each independently calls `super.getRows(tokenCacheQuery, [uncachedTokenIds])`.
4. Each `getRows()` call acquires one connection from the pool. With 10 concurrent requests the entire pool of 10 connections is held simultaneously for up to 20 s (the `statementTimeout`).
5. Any subsequent request that needs a DB connection — including transaction confirmation queries via `queryQuietly` in `rest/transactions.js` — blocks waiting for a free connection. After `connectionTimeout` (20 s) those requests throw, returning 500 errors to callers.
6. The attacker repeats the burst every ~20 s to maintain continuous pool exhaustion.

**Why existing checks fail:**
- The `statementTimeout` (20 s) only bounds how long a single query runs; it does not prevent 10 queries from running simultaneously.
- The `connectionTimeout` (20 s) only determines how long a queued request waits before failing; it does not prevent the queue from forming.
- A grep across `rest/**/*.js` for `rateLimit`, `rateLimiter`, `throttle`, `semaphore`, `concurrency`, and `queue` returns **zero matches** — there is no request-rate or concurrency-limiting middleware in the REST Node.js service.
- The throttling found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate Web3 Java service, not to the REST Node.js service.

### Impact Explanation
Pool exhaustion causes every DB-dependent endpoint in the REST mirror node to return errors for the duration of the attack. This includes `/api/v1/transactions`, which clients and wallets use to confirm transaction finality. Sustained bursts prevent the mirror node from serving transaction confirmation data, effectively making the mirror node unavailable. While the underlying Hedera network continues to process transactions, no external observer relying on this mirror node can confirm them. Severity is **High** (availability impact on a critical public infrastructure component, no authentication required).

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond knowing the mirror node's public URL, and no sophisticated tooling — a simple script sending 10 concurrent HTTP requests suffices. The endpoint `/api/v1/accounts/{id}/tokens` is publicly documented and unauthenticated. The cold-cache condition is trivially achieved at service startup or by referencing accounts whose tokens have been evicted from the 100 000-entry LRU. The attack is repeatable indefinitely at ~20-second intervals. Likelihood is **High**.

### Recommendation
1. **Promise coalescing (single-flight):** Maintain a `Map<tokenId, Promise>` of in-flight DB fetches. Before issuing a new `getRows` call, check whether a promise for the same set of IDs is already pending and await it instead of issuing a duplicate query.
2. **Concurrency limiter on DB calls:** Use a semaphore (e.g., `async-sema` or `p-limit`) to cap the number of simultaneous `getRows` calls from `getCachedTokens` to a value well below `maxConnections`.
3. **Increase pool size or add pgBouncer transaction pooling** for the REST Node.js service to reduce the blast radius.
4. **Add per-IP or global request-rate limiting middleware** (e.g., `express-rate-limit`) to the REST Node.js application to bound the number of concurrent requests that can reach the DB layer.

### Proof of Concept

```bash

# Identify 10 accounts each associated with many tokens not yet cached

  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$((1000 + i))/tokens" &
done
wait

# Immediately after, attempt a transaction confirmation query:
curl -v "https://<mirror-node>/api/v1/transactions/0.0.1234@1234567890.000000000"
# Expected result during pool exhaustion: HTTP 500 or connection timeout
# (pool of 10 connections held by the 12 concurrent token-cache queries)
```

Repeat the burst every 20 seconds to maintain continuous pool exhaustion and sustained denial of service.
