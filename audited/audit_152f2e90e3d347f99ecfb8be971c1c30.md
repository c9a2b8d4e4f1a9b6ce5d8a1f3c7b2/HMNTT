### Title
Unauthenticated Endpoint `/api/v1/network/stake` Lacks Caching and Rate Limiting, Enabling Database Exhaustion via Repeated `findLatest()` Queries

### Summary
The public REST endpoint `GET /api/v1/network/stake` invokes `NetworkServiceImpl.getLatestNetworkStake()` on every request, which unconditionally executes a native SQL query containing a correlated subquery against the `network_stake` table. There is no caching, no rate limiting, and no authentication on this endpoint. An unprivileged attacker can flood this endpoint to force repeated database queries, exhausting the DB connection pool and degrading or denying service to all consumers of the mirror node API.

### Finding Description
**Code path:**

- `NetworkController.getNetworkStake()` — [1](#0-0)  — is a public `@GetMapping("/stake")` with no authentication annotation.
- It calls `networkService.getLatestNetworkStake()`, implemented in `NetworkServiceImpl` — [2](#0-1)  — which directly delegates to `networkStakeRepository.findLatest()` with no caching layer.
- `NetworkStakeRepository.findLatest()` executes a native SQL query on every invocation: [3](#0-2)  — this query contains a correlated subquery (`select max(consensus_timestamp) from network_stake`) that runs on every call.

**Root cause — failed assumptions:**
1. No `@Cacheable` annotation exists anywhere in the rest-java service layer (confirmed: zero cache configuration in `rest-java/src/main/java/`).
2. No rate limiter is applied to this endpoint. The only rate-limiting code found in the module is scoped to `FeeEstimationService`, not to the `/stake` route.
3. The filters present (`MetricsFilter`, `LoggingFilter`) — [4](#0-3)  — perform no throttling.

### Impact Explanation
Every HTTP request to `GET /api/v1/network/stake` causes two database operations: a `MAX()` aggregate scan and a row fetch on `network_stake`. At high request rates (e.g., thousands of requests/second from a single or distributed client), this exhausts the JDBC connection pool, increases DB I/O and CPU, and causes query queuing/timeouts. This degrades or denies the mirror node's ability to serve staking data and other concurrent API requests that share the same DB connection pool. The mirror node is a read-only service; it does not directly participate in consensus, so the claim of impacting "transaction validators for consensus" is overstated — the realistic impact is **mirror node API unavailability**.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and only a standard HTTP client. The endpoint is publicly reachable on any deployed mirror node. The attack is trivially repeatable and automatable (e.g., `ab`, `wrk`, or a simple loop). No special knowledge of the system is required beyond knowing the public API path `/api/v1/network/stake`, which is documented in the mirror node OpenAPI spec.

### Recommendation
1. **Add response caching** at the service layer using `@Cacheable` with a short TTL (e.g., 30–60 seconds), since staking data changes only once per staking period (~24 hours). This eliminates the per-request DB hit entirely for the common case.
2. **Add rate limiting** on the `/api/v1/network/stake` endpoint (e.g., via a servlet filter, Spring Cloud Gateway, or an API gateway in front of the service) to cap requests per IP per second.
3. **Optimize the query** — if caching is not immediately feasible, at minimum ensure `consensus_timestamp` is indexed so the `MAX()` subquery uses an index-only scan rather than a sequential scan.

### Proof of Concept
```bash
# Flood the endpoint with concurrent requests (no auth required)
wrk -t 10 -c 200 -d 60s http://<mirror-node-host>/api/v1/network/stake

# Or with curl in a loop
while true; do
  curl -s http://<mirror-node-host>/api/v1/network/stake -o /dev/null &
done
```

**Expected result:** DB connection pool exhaustion; subsequent requests to `/api/v1/network/stake` (and other endpoints sharing the pool) begin returning 500 errors or timing out. DB server shows sustained high I/O and CPU from repeated `SELECT max(consensus_timestamp) FROM network_stake` executions.
