### Title
Unauthenticated Redis Injection Enables Falsified Transaction Response Cache Poisoning

### Summary
`responseCacheCheckHandler()` in `rest/middleware/responseCacheHandler.js` unconditionally trusts data retrieved from Redis, using `Object.assign(new CachedApiResponse(), redisValue)` with no integrity verification. The reference docker-compose deployment exposes Redis on `0.0.0.0:6379` with no authentication, allowing any network-adjacent attacker to write arbitrary cache entries. Once injected, the falsified response is served verbatim to every user querying that endpoint until the TTL expires.

### Finding Description
**Exact code path:**

`rest/middleware/responseCacheHandler.js`, lines 50–78:
```js
const {ttl: redisTtl, value: redisValue} = cachedTtlAndValue;  // line 50
const cachedResponse = Object.assign(new CachedApiResponse(), redisValue); // line 51
// ...
res.set(headers);       // line 75 — attacker-controlled headers
res.status(statusCode); // line 76 — attacker-controlled status code
res.send(body);         // line 78 — attacker-controlled body
```

`redisValue` is the result of `JSONParse(rawValue)` from `cache.js` line 79, where `rawValue` is the raw string stored in Redis. There is no HMAC, signature, schema validation, or any other integrity check between Redis storage and response delivery.

**Root cause:** The code assumes Redis is a trusted internal store. This assumption fails when Redis is reachable without authentication.

**Failed assumption confirmed by deployment config:**

`docker-compose.yml` lines 95–98 and 113:
```yaml
redis:
  image: redis:6-alpine
  ports:
    - 6379:6379   # bound to 0.0.0.0, no requirepass
...
HIERO_MIRROR_REST_REDIS_URI: redis://redis:6379  # no password
```

The documented default Redis URI (`docs/configuration.md` line 602) is also `redis://127.0.0.1:6379` — no password. The Helm chart generates a random password, but the docker-compose reference deployment does not.

**Exploit flow:**
1. Attacker identifies the MD5-based cache key format: `md5(req.originalUrl) + "-v1"` (line 152–153).
2. Attacker computes the key for a target endpoint, e.g., `/api/v1/transactions/0.0.1234@1234567890.000000000`: `md5("/api/v1/transactions/0.0.1234@1234567890.000000000") + "-v1"`.
3. Attacker connects to Redis on port 6379 (no auth required in docker-compose) and runs:
   ```
   SETEX <computed_key> 3600 '{"statusCode":200,"headers":{"content-type":"application/json; charset=utf-8","etag":"\"fakeetag\""},"body":"{\"transactions\":[{\"transaction_id\":\"0.0.1234@1234567890.000000000\",\"result\":\"SUCCESS\",\"transfers\":[]}]}","compressed":false}'
   ```
4. All subsequent requests to that endpoint hit the cache check handler, which calls `Object.assign(new CachedApiResponse(), redisValue)` and serves the attacker's fabricated body, headers, and status code directly to users.

### Impact Explanation
Any transaction query endpoint can be poisoned to return fabricated data: false `SUCCESS` results for failed transactions, suppressed transfers, altered balances, or entirely invented transaction records. Because the cache hit path short-circuits the database entirely (the handler returns after `res.send(body)` without calling `next()`), the real database is never consulted during the TTL window. All users querying the poisoned key receive the falsified response. This directly undermines the integrity of the mirror node as a source of truth for Hedera network state, enabling financial fraud scenarios (e.g., falsely confirming a payment was received).

### Likelihood Explanation
The docker-compose deployment — the primary reference deployment for self-hosted operators — binds Redis to `0.0.0.0:6379` with no password. Any host reachable on that port is exploitable with zero credentials. The response cache is disabled by default (`cache.response.enabled: false`), which limits exposure to deployments that have explicitly enabled it. However, operators enabling the cache for performance (a documented and intended use case) are fully exposed. The attack is repeatable, scriptable, and requires no special tooling beyond a Redis client.

### Recommendation
1. **Integrity verification:** Compute an HMAC (e.g., HMAC-SHA256 with a server-side secret) over the serialized cache value at write time (`responseCacheUpdateHandler`) and verify it at read time (`responseCacheCheckHandler`) before trusting any field from Redis. Reject and delete entries that fail verification.
2. **Schema validation:** After `Object.assign`, validate that `statusCode` is a known safe integer, `headers` contains only expected keys with safe values, and `body` is a string/Buffer before passing to `res.set`/`res.status`/`res.send`.
3. **Enforce Redis authentication in all deployments:** The docker-compose file should require a password for Redis and pass it in `HIERO_MIRROR_REST_REDIS_URI`. Remove the unauthenticated `6379:6379` port binding or restrict it to `127.0.0.1:6379`.
4. **Network isolation:** Redis should never be bound to a public interface; restrict to the internal Docker network only.

### Proof of Concept
**Preconditions:** docker-compose deployment running with `cache.response.enabled: true`; Redis port 6379 reachable from attacker's host.

```bash
