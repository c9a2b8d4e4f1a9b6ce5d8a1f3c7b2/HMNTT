### Title
Unbounded Concurrent Parse/Validate Amplification via Caffeine AsyncCache in GraphQL Module

### Summary
The `CachedPreparsedDocumentProvider.getDocumentAsync()` uses Caffeine's `AsyncCache.get()` which, for each distinct uncached query key, immediately submits a `parseAndValidateFunction.apply()` task to `ForkJoinPool.commonPool()` without any concurrency bound. The GraphQL module contains no application-level rate limiting, so an unprivileged attacker sending a high volume of concurrent requests with distinct query strings can saturate the JVM's common thread pool with simultaneous parse+validate operations, causing CPU and thread resource consumption to spike well above 30%.

### Finding Description
**Exact code location:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

**Root cause:**
Caffeine's `AsyncCache.get(K key, Function<K, V> mappingFunction)` wraps the mapping function in `CompletableFuture.supplyAsync()` using the default executor (`ForkJoinPool.commonPool()`). For each distinct query string that is absent from the cache, a new async task is immediately submitted to the common pool — there is no concurrency cap, no semaphore, and no back-pressure mechanism. The cache (`maximumSize=1000`, `expireAfterWrite=1h`) only helps for repeated identical queries; distinct queries always miss.

**Failed assumption:**
The design assumes the cache will absorb repeated load. It provides zero protection against a flood of distinct (but syntactically valid) queries, each of which triggers a full parse+validate cycle on the thread pool.

**Exploit flow:**
1. Attacker generates a large set of syntactically distinct but valid GraphQL queries (e.g., varying field aliases, argument values, or operation names — all within the parser limits).
2. Attacker sends these concurrently to the `/graphql/alpha` endpoint.
3. Each request calls `getDocumentAsync()`, which calls `cache.get(query, ...)`.
4. Each distinct query misses the cache and submits `parseAndValidateFunction.apply()` to `ForkJoinPool.commonPool()`.
5. N concurrent parse+validate tasks run simultaneously, saturating the common pool and driving CPU usage up.

**Why existing checks fail:**
- **Parser limits** (`maxCharacters=10000`, `maxTokens=1000`, `maxRuleDepth=100`): These bound the cost *per query* but impose no limit on *concurrency*. An attacker can send thousands of small, valid queries simultaneously.
- **`MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)`**: These run *after* parsing, so they do not prevent parse work from being submitted.
- **Infrastructure controls** (GCP `maxRatePerEndpoint: 250`, Traefik `inFlightReq: amount: 5`): These are opt-in and disabled by default (`global.middleware: false`; GCP gateway requires explicit enablement). They are not present in the application itself.
- **No application-level rate limiting** exists in the `graphql` module — unlike the `web3` module which has `ThrottleManagerImpl` with `Bucket4j`.

### Impact Explanation
An attacker can drive CPU utilization of the GraphQL node to saturation, degrading or denying service to legitimate users. Because `ForkJoinPool.commonPool()` is shared across the entire JVM, saturation also affects other async operations. The 30%+ resource consumption threshold is easily achievable: even a modest flood of ~100–500 concurrent requests with distinct queries will submit that many simultaneous parse tasks. The cache's `maximumSize=1000` means the attacker can cycle through 1000+ distinct queries before any eviction helps.

### Likelihood Explanation
No authentication or API key is required to reach the GraphQL endpoint. Generating syntactically distinct valid queries is trivial (e.g., incrementing a numeric argument: `{ account(entityId: "0.0.1") { ... } }`, `{ account(entityId: "0.0.2") { ... } }`). The attack is repeatable, scriptable, and requires no special knowledge beyond the public GraphQL schema. A single attacker with a standard HTTP client can execute this.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module** — mirror the `Bucket4j`-based `ThrottleManagerImpl` pattern from the `web3` module, enforcing a per-IP or global requests-per-second cap before `getDocumentAsync()` is reached.
2. **Bound async concurrency in the cache**: Configure Caffeine with a custom executor that has a bounded thread pool (e.g., `Caffeine.newBuilder().executor(boundedExecutor).buildAsync()`) so that at most N parse tasks run concurrently.
3. **Enable the Traefik `inFlightReq` middleware by default** (`global.middleware: true`) to enforce the existing `amount: 5` in-flight request limit per IP at the infrastructure layer.
4. **Consider a parse-only semaphore** (e.g., `Semaphore`) inside `getDocumentAsync()` to cap simultaneous `parseAndValidateFunction.apply()` invocations.

### Proof of Concept
```python
import asyncio, aiohttp, random, string

URL = "http://<graphql-node>/graphql/alpha"

def make_distinct_query(i):
    # Each query is syntactically valid but distinct (different alias)
    alias = ''.join(random.choices(string.ascii_lowercase, k=6))
    return {"query": f'{{ {alias}: account(entityId: "0.0.{i}") {{ balance }} }}'}

async def send(session, i):
    async with session.post(URL, json=make_distinct_query(i)) as r:
        return await r.text()

async def main():
    async with aiohttp.ClientSession() as session:
        # Send 500 concurrent requests with distinct uncached queries
        tasks = [send(session, i) for i in range(500)]
        await asyncio.gather(*tasks)

asyncio.run(main())
```
**Expected result:** CPU usage on the target node spikes significantly (30%+) as 500 simultaneous `parseAndValidateFunction.apply()` tasks are submitted to `ForkJoinPool.commonPool()`. Monitor with `process_cpu_usage{application="graphql"}` (Prometheus) or `top`/`jstack` on the JVM process.
