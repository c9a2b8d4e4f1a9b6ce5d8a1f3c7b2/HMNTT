### Title
Missing Entity Type Validation in `isValidAccount()` Allows Contract Addresses to Pass Account Check in `getTokenRelationships()`

### Summary
`EntityService.isValidAccount()` fetches `Entity.TYPE` from the database but never validates it, only checking entity existence. This allows any contract entity's EVM address to pass the account validation gate in `getTokenRelationships()`, causing the `/accounts/{id}/tokens` endpoint to return token relationships for contract entities. Any unprivileged user can trigger this with a publicly known contract EVM address.

### Finding Description

**Exact code path:**

`rest/service/entityService.js` lines 28–30 define the query:
```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [1](#0-0) 

`isValidAccount()` at lines 60–63 executes this query but only checks `!isNil(entity)` — the fetched `type` column is **never inspected**:
```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [2](#0-1) 

`getTokenRelationships()` at lines 66–71 relies entirely on this check before proceeding:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) {
  throw new NotFoundError();
}
``` [3](#0-2) 

**Root cause:** The `entityExistenceQuery` was designed to fetch `Entity.TYPE` (as noted by the comment "use a small column in existence check to reduce return payload size"), but `isValidAccount()` discards the type value entirely. The function name implies account-type validation, but the implementation only validates existence.

**Exploit flow:**
1. Attacker identifies a deployed contract with a known EVM address (e.g., `0xcef2a2c6c23ab8f2506163b1af55830f35c483ca`) — this is public information on any Hedera network.
2. Attacker calls `GET /api/v1/accounts/0xcef2a2c6c23ab8f2506163b1af55830f35c483ca/tokens`.
3. `getEncodedId()` resolves the EVM address via `getEntityIdFromEvmAddress()`, which queries the entity table by `evm_address` with no type filter, returning the contract's encoded entity ID. [4](#0-3) 
4. `isValidAccount()` finds the contract entity exists → returns `true`.
5. `TokenService.getTokenAccounts()` queries `token_account` using the contract's entity ID as `account_id` — contracts can hold token associations in Hedera, so this returns real data. [5](#0-4) 
6. The API returns a `200 OK` with token relationship data for a contract entity under the `/accounts/` namespace.

**Why existing checks fail:** The `entityFromEvmAddressQuery` used in `getEntityIdFromEvmAddress()` has no `type` filter:
```sql
select ${Entity.ID}
from ${Entity.tableName}
where ${Entity.DELETED} <> true
  and ${Entity.EVM_ADDRESS} = $1
``` [4](#0-3) 
And `isValidAccount()` never checks the returned `type` field. There is no downstream type guard before the `token_account` query executes.

### Impact Explanation
The `/accounts/{id}/tokens` endpoint is documented and consumed as an account-scoped API. Returning token relationships for contract entities violates this contract. Smart contracts or off-chain systems that call this mirror node API to determine account token balances may receive data for a contract entity, leading to incorrect balance reads, incorrect authorization decisions, or incorrect state assumptions in dependent smart contract logic. The severity is bounded because the mirror node is read-only and no funds are directly manipulated, but the data integrity violation can propagate into on-chain decisions via smart contracts that consume mirror node data.

### Likelihood Explanation
Exploitation requires zero privileges — the REST API is public. The attacker only needs a contract EVM address, which is trivially discoverable from block explorers or the `/contracts` endpoint. The attack is fully repeatable and deterministic. Any Hedera network with deployed contracts that hold token associations is affected.

### Recommendation
In `isValidAccount()`, validate the returned entity type against the expected account types. The `entityExistenceQuery` already fetches `Entity.TYPE`; the fix is to use it:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  if (isNil(entity)) return false;
  // Only ACCOUNT and CRYPTO_ALLOWANCE entity types are valid accounts
  return entity.type === 'ACCOUNT';
}
```

Alternatively, add a `type` filter directly to `entityExistenceQuery`:
```sql
select ${Entity.TYPE}
from ${Entity.tableName}
where ${Entity.ID} = $1
  and ${Entity.TYPE} = 'ACCOUNT'
```

The same fix should be applied to `accountController.js` which uses the identical `isValidAccount()` pattern for `listStakingRewardsByAccountId`. [6](#0-5) 

### Proof of Concept
**Precondition:** A contract exists in the Hedera network with EVM address `0xCEF2A2C6C23AB8F2506163B1AF55830F35C483CA` and has at least one token association in the `token_account` table.

**Steps:**
```bash
# Step 1: Confirm the address belongs to a contract (not an account)
curl https://mainnet-public.mirrornode.hedera.com/api/v1/contracts/0xCEF2A2C6C23AB8F2506163B1AF55830F35C483CA
# Returns contract entity data

# Step 2: Query the accounts/tokens endpoint with the contract EVM address
curl https://mainnet-public.mirrornode.hedera.com/api/v1/accounts/0xCEF2A2C6C23AB8F2506163B1AF55830F35C483CA/tokens

# Expected (correct) result: 404 Not Found
# Actual result: 200 OK with token relationship data for the contract entity
```

**Expected vs Actual:**
- Expected: `404 Not Found` (contract is not an account)
- Actual: `200 OK` with `{"tokens": [...], "links": {...}}` containing the contract's token associations
