### Title
Unauthenticated Request Flood Exhausts Tomcat Thread Pool via Unthrottled `/api/v1/accounts/{id}/airdrops/pending` Endpoint

### Summary
The `rest-java` module exposes `/api/v1/accounts/{id}/airdrops/pending` with no per-IP or global rate limiting. Any unauthenticated external caller can flood this endpoint with concurrent requests, saturating the embedded Tomcat thread pool (default 200 threads) and causing all other `rest-java` endpoints — including those used by clients to poll transaction confirmation status — to queue indefinitely and time out.

### Finding Description
**Code path:**

- `TokenAirdropsController.getPendingAirdrops()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, lines 77–86 — accepts unauthenticated GET requests with no throttle guard.
- Delegates to `TokenAirdropServiceImpl.getAirdrops()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/TokenAirdropServiceImpl.java`, lines 19–22 — which calls `entityService.lookup()` then `repository.findAll()`, both of which hit the database.
- `TokenAirdropRepositoryCustomImpl.findAll()` — lines 58–72 — executes a full `SELECT … WHERE … LIMIT ?` against the `token_airdrop` table on every call.

**Root cause — no rate limiting in `rest-java`:**

The only servlet filters registered in `rest-java` are `LoggingFilter` (logs only) and `MetricsFilter` (records bytes only). `RestJavaConfiguration` registers only an ETag filter, a Protobuf converter, and a jOOQ customizer. `WebMvcConfiguration` registers only argument resolvers and formatters. There is no `application.yml` in `rest-java/src/main/resources/` that configures Tomcat thread limits or connection limits. No bucket4j bean, no `@RateLimiter`, no servlet filter performing IP-based throttling exists anywhere in the `rest-java` source tree.

By contrast, the `web3` module has an explicit `ThrottleConfiguration` (bucket4j `rateLimitBucket`, `gasLimitBucket`, `opcodeRateLimitBucket`) and `ThrottleManagerImpl` that enforces `requestsPerSecond = 500` globally. The `rest-java` module has no equivalent.

**Why the `limit` parameter cap is insufficient:**
The `@Max(MAX_LIMIT)` annotation on the `limit` parameter caps results at 100 rows per response, but does nothing to limit the *rate* of incoming requests. Each request still occupies a Tomcat thread for the duration of the DB round-trip.

### Impact Explanation
Spring Boot's embedded Tomcat defaults to 200 worker threads. An attacker sending 200+ concurrent, long-running requests to `/api/v1/accounts/{id}/airdrops/pending` (e.g., with a large result set or a slow DB) saturates the thread pool. All subsequent requests to any endpoint served by the same `rest-java` process — including endpoints clients use to poll transaction confirmation (e.g., `/api/v1/transactions/{id}`) — are queued by Tomcat's accept queue and eventually time out. This constitutes a full denial-of-service of the `rest-java` service, preventing clients from querying transaction status via the mirror node REST Java API.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no rate limiting are required. Any attacker with internet access and a basic HTTP client (e.g., `curl`, `ab`, `wrk`) can reproduce this. The attack is repeatable indefinitely, requires no special knowledge of the system, and is trivially automatable. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, lines 241–266).

### Recommendation
1. Add a global rate-limiting servlet filter to `rest-java` using bucket4j (mirroring the `web3` `ThrottleConfiguration`), enforcing a maximum requests-per-second across all endpoints.
2. Add per-IP rate limiting (keyed on `request.getRemoteAddr()` or `X-Forwarded-For`) using a `ConcurrentHashMap<String, Bucket>` of per-IP buckets.
3. Configure Tomcat connection and thread limits explicitly in `application.yml` (`server.tomcat.max-threads`, `server.tomcat.accept-count`, `server.tomcat.max-connections`).
4. Consider deploying an ingress-level rate limiter (e.g., Traefik `rateLimit` middleware) in the Helm chart, in addition to application-level controls.

### Proof of Concept
```bash
# Flood the endpoint with 300 concurrent requests (exceeds default Tomcat max-threads=200)
# Replace ACCOUNT_ID with any valid account ID present in the mirror node DB
ACCOUNT_ID="0.0.1234"
BASE_URL="https://<mirror-node-rest-java-host>/api/v1/accounts/${ACCOUNT_ID}/airdrops/pending?limit=100"

for i in $(seq 1 300); do
  curl -s "$BASE_URL" &
done
wait

# While the above is running, observe that legitimate requests time out:
curl -v "https://<mirror-node-rest-java-host>/api/v1/transactions/0.0.1234-1234567890-000000000"
# Expected: connection hangs or returns 503/timeout
```

**Verification:** Monitor `tomcat_threads_busy_threads` in Prometheus/Grafana (already instrumented per `charts/hedera-mirror-rest-java/values.yaml` dashboards). Under attack, busy threads will saturate at `tomcat_threads_config_max_threads` and all other endpoint latencies will spike to timeout values.     public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
```java
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }
```
