### Title
Unauthenticated Mirror REST Response Causes Non-Retryable Process Crash (DoS)

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` unconditionally trusts the HTTP response from the configured mirror REST endpoint. An attacker who can influence that response (e.g., via MITM on a plaintext HTTP connection, DNS poisoning, or a compromised mirror node) can serve a syntactically valid JSON payload where every `nodeEntry.NodeAccountID` is an empty string. All entries are silently skipped, the resulting `network` map is empty, and the function returns `retry=false` with an error — causing `log.Fatalf` in `main.go` to permanently terminate the pinger process.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 102–129:

```go
for _, n := range payload.Nodes {
    if n.NodeAccountID == "" {   // line 103 — silently skips entry
        continue
    }
    ...
}

if len(network) == 0 {           // line 127
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
    //              ^^^^^ retry=false — no retry will be attempted
}
``` [1](#0-0) 

**Root cause:** The function returns `retry=false` when the network map is empty. This is treated identically to a parse error — a permanent, non-retryable failure — even though the cause is a fully attacker-controllable field in the response body.

**Propagation to crash:**

In `buildNetworkFromMirrorNodes` (lines 59–61), `retry=false` causes the retry loop to break immediately:

```go
if !retry || attempt == attempts {
    break
}
``` [2](#0-1) 

`newClient()` in `sdk_client.go` propagates the error:

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil {
    return nil, err
}
``` [3](#0-2) 

`main.go` calls `log.Fatalf`, terminating the process:

```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)
}
``` [4](#0-3) 

**No TLS enforcement:** The `http.Client` is constructed with no custom TLS configuration and no certificate pinning. If `cfg.mirrorRest` is an `http://` URL (common in internal Kubernetes deployments), the connection is plaintext and trivially interceptable. [5](#0-4) 

### Impact Explanation

A single crafted response permanently kills the pinger process. The pinger's purpose is to continuously submit transactions to verify network liveness; killing it silently disables that monitoring. Kubernetes liveness/readiness probes (`/tmp/alive`, `/tmp/ready`) are never updated after startup, so the pod may be restarted — but if the attacker sustains the malicious response, every restart immediately crashes again, creating a persistent DoS loop against the monitoring infrastructure.

### Likelihood Explanation

- **Precondition:** `network=other` must be configured (required for custom/private networks). The mirror REST URL must be reachable over HTTP (no TLS), or the attacker must control DNS for the mirror hostname.
- **Attacker capability:** A network-adjacent attacker (same cluster, same L2 segment, or controlling a DNS resolver) with no application-level credentials can perform this attack. No authentication is required against the mirror REST API.
- **Repeatability:** The attack is fully repeatable and automated. A single HTTP response is sufficient to crash the process; sustaining the response keeps it crashed across restarts.

### Recommendation

1. **Treat empty-network as retryable:** Change `return nil, false, fmt.Errorf(...)` at line 128 to `return nil, true, fmt.Errorf(...)` so transient or injected bad responses are retried rather than causing immediate fatal failure.
2. **Enforce HTTPS with certificate validation:** Reject `cfg.mirrorRest` URLs that do not use `https://`, or configure the `http.Client` with a pinned CA/certificate.
3. **Require a minimum node count:** Reject responses with fewer than a configurable minimum number of valid nodes (e.g., 1), and treat that as a retryable error.
4. **Do not use `log.Fatalf` for recoverable network errors:** Degrade gracefully (log + wait + retry) rather than crashing the process on a bad mirror response.

### Proof of Concept

1. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`.
2. Serve the following JSON from `GET /api/v1/network/nodes`:
   ```json
   {"nodes": [{"node_account_id": "", "service_endpoints": []}], "links": {"next": null}}
   ```
3. Start the pinger. Observe `log.Fatalf("client error: no usable service_endpoints found from ...")` and immediate process exit.
4. For a MITM variant: intercept the HTTP response from the legitimate mirror REST endpoint and replace the body with the above payload. The pinger crashes on every restart as long as the MITM is active.
