### Title
Unauthenticated LRU Cache Thrashing via `/block` Endpoint Causes Amplified DB Query Load

### Summary
The `Block()` function in `rosetta/app/services/block_service.go` is publicly accessible with no application-level authentication or rate limiting. An unprivileged attacker can flood the endpoint with requests for distinct valid blocks whose operations reference account IDs not present in the shared `entityCache` LRU, forcing `updateOperationAccountAlias()` to issue a `GetAccountAlias()` DB query for every cache-missing account ID on every request, degrading service responsiveness for all users.

### Finding Description
**Code path:**

In `rosetta/app/services/block_service.go` lines 47–74, `Block()` calls `updateOperationAccountAlias()` (lines 104–129) unconditionally for every request. Inside that function, for each operation in each transaction, the code checks the shared `entityCache` LRU:

```go
// block_service.go:115-122
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
    ...
    s.entityCache.Set(result.GetId(), result)
    cached = result
}
```

`GetAccountAlias()` in `rosetta/app/persistence/account.go` line 117 executes a synchronous DB query:
```go
db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity)
// selectCryptoEntityWithAliasById = "select alias, id from entity where id = @id"
```

**Root cause and failed assumption:**

The `entityCache` is a single shared LRU instance (initialized at lines 34–37 with `lru.WithCapacity(entityCacheConfig.MaxSize)`, default `524288` per `docs/configuration.md` line 654). The design assumes the cache warms up under normal traffic and stays warm. An adversary can deliberately defeat this by cycling through blocks whose operations reference more than 524288 distinct account IDs, or simply by continuously requesting cold blocks before the cache warms.

**Why existing checks are insufficient:**

The only rate-limiting controls are in the optional Traefik middleware defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 (`rateLimit: average: 10` per requestHost, `inFlightReq: amount: 5` per IP). These are:
1. Disabled by default at the global level (`global.middleware: false`, line 95)
2. Traefik-specific — not present in bare-metal or non-Traefik deployments
3. Bypassable via distributed multi-IP attack (the `inFlightReq` per-IP limit does not help)

There is zero application-level rate limiting or authentication in the Rosetta Go server itself (`rosetta/main.go` lines 58–119 shows no rate-limiting middleware registered).

### Impact Explanation
Each `/block` request for a block with K distinct uncached account IDs triggers K synchronous DB queries against the `entity` table. The DB connection pool is capped at `maxOpenConnections: 100` (`docs/configuration.md` line 661). A sustained flood of requests for blocks with many distinct account IDs saturates the pool, causing all other Rosetta API calls (account balance lookups, construction, network info) to queue or time out. This is a service-wide degradation affecting all legitimate users, with no economic cost to the attacker.

### Likelihood Explanation
Hedera mainnet has tens of millions of accounts. An attacker can trivially enumerate valid block indices from the public chain and identify high-transaction blocks. No credentials, tokens, or special access are required — only the ability to send HTTP POST requests to `/block`. The attack is repeatable, scriptable, and can be amplified by using multiple source IPs to bypass the optional per-IP in-flight limit. The barrier to entry is extremely low.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP (or global) request rate limiter directly in the Go HTTP server, independent of any infrastructure middleware, so it applies in all deployment configurations.
2. **Per-request DB query cap**: Limit the number of `GetAccountAlias()` calls per single `Block()` invocation (e.g., batch the lookups or cap at a maximum).
3. **Negative caching**: Cache accounts that have no alias so repeated requests for the same account do not re-query the DB.
4. **Do not rely solely on infrastructure controls**: The Traefik middleware is optional and deployment-dependent; the application must defend itself.

### Proof of Concept
```
# 1. Identify N blocks on mainnet with many distinct account IDs in their transactions
#    (e.g., blocks with high-volume DeFi activity)

# 2. Flood the endpoint concurrently from multiple IPs:
for block_index in $(seq 1 100000); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$block_index}}" &
done

# 3. Observe: DB connection pool saturates (maxOpenConnections=100),
#    GetAccountAlias() queries pile up, legitimate /account/balance
#    and /construction/* requests begin timing out or returning errors.
```

Each concurrent request for a cold block with T transactions × O operations each triggers up to T×O DB queries. With 100 concurrent requests each touching 50 distinct accounts, that is 5,000 simultaneous `SELECT alias, id FROM entity WHERE id = ?` queries against a pool of 100 connections.
