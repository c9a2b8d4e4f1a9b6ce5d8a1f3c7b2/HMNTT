### Title
Unauthenticated DB Connection Pool Exhaustion via `GET /api/v1/accounts/<evm_address>?timestamp=gte:X&timestamp=lte:Y`

### Summary
The `getOneAccount()` handler in `rest/accounts.js` triggers up to four sequential and parallel database operations per request when an EVM address path parameter is combined with timestamp range filters. The REST API has no rate-limiting middleware, and the default connection pool size is 10. An attacker sending a small number of concurrent unauthenticated requests can exhaust the pool, causing all subsequent REST API requests to queue and time out for up to 20 seconds per cycle.

### Finding Description

**Exact code path:**

**Step 1 — EVM address DB lookup (sequential, awaited):**
`getOneAccount()` calls `EntityService.getEncodedId()`, which detects an EVM address and calls `getEntityIdFromEvmAddress()`. This issues a synchronous `await` DB query before any other work proceeds. [1](#0-0) [2](#0-1) 

**Step 2 — Balance snapshot timestamp range lookup (sequential, awaited):**
When `timestampFilters.length > 0`, `getAccountBalanceTimestampRange()` is `await`-ed before the main queries are launched. This issues a second DB query against `account_balance`. [3](#0-2) [4](#0-3) 

**Steps 3 & 4 — Entity query and transaction query (parallel via `Promise.all`):**
After steps 1 and 2 complete, `entityPromise` (the main UNION ALL entity query) and `transactionsPromise` (`doGetTransactions()`) are launched concurrently. `doGetTransactions()` itself issues at least one more DB query (`getTransactionTimestamps()`), and a second one (`getTransactionsDetails()`) if results are found. [5](#0-4) [6](#0-5) 

**Root cause — no rate limiting on the REST API:**
The REST API middleware stack contains `authHandler`, `httpErrorHandler`, `openapiHandler`, `requestHandler`, `responseCacheHandler`, and `responseHandler`. There is no rate-limiting or concurrency-limiting middleware. [7](#0-6) 

The throttle configuration that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) applies exclusively to the **web3** service (`/api/v1/contracts/call`), not to the REST API accounts endpoint. [8](#0-7) 

**Root cause — small default pool:**
The REST API DB pool defaults to `maxConnections = 10` with a `statementTimeout = 20000` ms (20 s) and `connectionTimeoutMillis = 20000` ms. [9](#0-8) 

The documentation confirms the default:

> `hiero.mirror.rest.db.pool.maxConnections` — Default: **10** [10](#0-9) 

**Why existing checks fail:**
- `statementTimeout` (20 s) limits individual query duration but does not prevent pool exhaustion; it only bounds how long each connection is held.
- `connectionTimeoutMillis` (20 s) means queued requests will fail after 20 s, but during that window the service is fully degraded.
- The Redis response cache is **disabled by default** (`hiero.mirror.rest.cache.response.enabled: false`), so no caching shields the DB.
- The entity ID cache (`maxAge: 1800`, `maxSize: 100000`) is not applied inside `getEntityIdFromEvmAddress()`, which calls `this.getRows()` directly without a cache lookup.

### Impact Explanation

With a pool of 10 connections and up to 3 simultaneous connections consumed per in-flight request (1 during `getAccountBalanceTimestampRange` + 2 during `Promise.all`), approximately **4–5 concurrent attacker requests** exhaust the pool. All legitimate REST API requests then queue and time out. Because the REST API is the primary data-access interface for mirror node consumers (wallets, explorers, dApps), its unavailability constitutes a significant service disruption. In a multi-replica deployment, each replica has its own pool of 10, so the attacker must sustain ~5 concurrent requests per replica — still trivially achievable with a single machine.

### Likelihood Explanation

- **No authentication required.** Any public internet user can send the request.
- **No brute force required.** A single valid EVM address (obtainable from any public block explorer or by querying `/api/v1/accounts`) is sufficient.
- **Low request volume.** Only ~5 concurrent HTTP requests per REST API replica are needed to exhaust the pool.
- **Repeatable.** After the 20-second statement timeout releases connections, the attacker simply re-sends the same requests.
- **Scriptable in seconds** with `curl`, `ab`, `wrk`, or any HTTP load tool.

### Recommendation

1. **Add per-IP (or global) rate limiting** to the REST API, specifically for the `GET /api/v1/accounts/:id` route. An Express middleware such as `express-rate-limit` or a reverse-proxy-level limit (nginx, Envoy) should be applied.
2. **Increase the default pool size** or document that operators must tune `hiero.mirror.rest.db.pool.maxConnections` relative to expected concurrency.
3. **Cache the EVM-address-to-entity-ID resolution** inside `getEntityIdFromEvmAddress()` using the existing entity ID cache infrastructure, eliminating DB op #1 for repeated addresses.
4. **Add a concurrency semaphore** at the application level to cap the number of in-flight DB-heavy requests (e.g., using `p-limit` or a similar library).
5. **Enable the Redis response cache** for this endpoint to serve repeated identical queries without hitting the DB.

### Proof of Concept

```bash
# Obtain any valid EVM address from the public mirror node
EVM="0xabcdef1234567890abcdef1234567890abcdef12"
BASE="https://<mirror-node-host>/api/v1/accounts"

# Send 6 concurrent requests — enough to exhaust the default pool of 10
for i in $(seq 1 6); do
  curl -s "${BASE}/${EVM}?timestamp=gte:1600000000&timestamp=lte:1700000000" &
done
wait

# Immediately probe with a legitimate request — expect timeout or connection error
curl -v "${BASE}/0.0.1"
```

**Expected result:** The final `curl` either hangs for ~20 seconds (connection timeout) or returns a 500/503 error, demonstrating that the DB connection pool is exhausted and the REST API is unresponsive to legitimate traffic.

const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
