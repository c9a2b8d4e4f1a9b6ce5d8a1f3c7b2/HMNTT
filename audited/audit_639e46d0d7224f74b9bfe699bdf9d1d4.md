### Title
Unvalidated `ne` Operator on Timestamp Accepted by `getContractResultsById`, Enabling Targeted Result Exclusion

### Summary
`filterValidityChecks` in `rest/utils.js` validates only the value format for `TIMESTAMP` parameters, not the operator, allowing `ne` to pass silently. In `extractContractResultsByIdQuery`, when a `contractId` is present, `optimizeTimestampFilters` is bypassed entirely and the raw `ne` filter is translated directly into a SQL `!=` condition. An unprivileged external user can therefore submit `timestamp=ne:T` to `GET /api/v1/contracts/{id}/results` and cause the endpoint to exclude any specific consensus timestamp from its results.

### Finding Description

**Code path:**

`rest/routes/contractRoute.js` line 14 → `getContractResultsById` → `extractContractIdAndFiltersFromValidatedRequest` → `utils.buildAndValidateFilters(..., contractResultsFilterValidityChecks)` → `contractResultsFilterValidityChecks` (line 273-279) → `utils.filterValidityChecks`.

In `rest/utils.js` lines 362-364:
```js
case constants.filterKeys.TIMESTAMP:
  ret = isValidTimestampParam(val);
  break;
```
The operator is **never checked** for `TIMESTAMP`. Compare to `SLOT` (line 357): `ret = isValidSlot(val) && includes(basicOperators, op)` which explicitly excludes `ne`. The `TIMESTAMP` case has no such guard.

`validateOpAndValue` (line 274-276) only checks `isValidOperatorQuery(op)`, which accepts all operators including `ne`.

So `timestamp=ne:1639010141000000000` passes validation and produces filter `{key: 'timestamp', operator: '!=', value: '1639010141000000000'}`.

In `extractContractResultsByIdQuery` (`rest/controllers/contractController.js` lines 504-505):
```js
const {filters: optimizedTimestampFilters, next} =
  contractId === undefined ? await optimizeTimestampFilters(timestampFilters, order) : {filters: timestampFilters};
```
Since `contractId` is defined for `getContractResultsById`, `optimizeTimestampFilters` (which calls `parseTimestampFilters` with `allowNe=true` and could at least process the filter) is **skipped**. The raw `ne` filter goes directly to `updateConditionsAndParamsWithInValues` (lines 510-518).

In `rest/controllers/baseController.js` lines 20-26:
```js
if (filter.operator === utils.opsMap.eq) {
  invalues.push(filter.value);
} else {
  existingParams.push(filter.value);
  existingConditions.push(`${fullName}${filter.operator}$${position}`);
}
```
This emits: `cr.consensus_timestamp != $2` directly into the SQL WHERE clause.

**Root cause:** `filterValidityChecks` for `TIMESTAMP` omits operator validation (unlike `SLOT`, `BLOCK_NUMBER`, `TRANSACTION_INDEX`). No secondary guard exists in `extractContractResultsByIdQuery` for the `ne` operator on timestamp (unlike `extractContractStateByIdQuery` which explicitly throws).

**Failed assumption:** The code assumes that only valid operators for each parameter type will reach the SQL-building layer. This holds for most parameters but not for `TIMESTAMP`.

### Impact Explanation
Any caller of `GET /api/v1/contracts/{id}/results?timestamp=ne:T` receives results with the row at consensus timestamp `T` excluded. If an off-chain fund-release bridge or automated settlement system uses this endpoint to confirm a specific contract execution occurred (e.g., checking that results exist for a known transfer timestamp), an attacker who knows timestamp `T` can craft a URL that makes the endpoint return an empty set or omit the relevant result. A system that interprets an empty response as "transfer not confirmed" would withhold fund release, effectively freezing funds. The mirror node API is read-only so on-chain state is unaffected, but off-chain systems that rely on it for confirmation are vulnerable.

### Likelihood Explanation
The attacker requires no credentials — only knowledge of the target contract ID and the consensus timestamp of the transfer (both publicly observable on-chain or via other mirror node endpoints). The attack is trivially repeatable: the attacker simply appends `?timestamp=ne:T` to any query. The realistic risk depends on whether fund-release systems use this endpoint with user-influenced parameters; systems that construct query URLs from user input or that use this listing endpoint rather than the specific-timestamp endpoint (`/contracts/{id}/results/{timestamp}`) are directly exploitable.

### Recommendation
1. In `filterValidityChecks` (`rest/utils.js`), add an operator check for `TIMESTAMP` analogous to `SLOT`:
   ```js
   case constants.filterKeys.TIMESTAMP:
     ret = isValidTimestampParam(val) && includes(basicOperators, op);
     break;
   ```
2. Alternatively, add an explicit `ne` rejection in `extractContractResultsByIdQuery` for timestamp filters (mirroring the guard in `extractContractStateByIdQuery` lines 910-912).
3. Add a test spec for `GET /api/v1/contracts/{id}/results?timestamp=ne:...` asserting HTTP 400, analogous to the existing `contracts/{id}/results/logs/invalid-params.json` test.

### Proof of Concept

**Precondition:** Contract `0.0.1234` has a successful fund-transfer execution at consensus timestamp `1639010141.000000000`.

**Step 1 — Confirm transfer exists (normal query):**
```
GET /api/v1/contracts/0.0.1234/results?timestamp=eq:1639010141.000000000
→ 200 OK, results: [{timestamp: "1639010141.000000000", ...}]
```

**Step 2 — Exploit (attacker-crafted query):**
```
GET /api/v1/contracts/0.0.1234/results?timestamp=ne:1639010141.000000000
→ 200 OK, results: []   (or results excluding the transfer timestamp)
```

**Step 3 — SQL generated:**
```sql
SELECT cr.amount, cr.consensus_timestamp, ...
FROM contract_result cr
LEFT JOIN entity e ON e.id = cr.contract_id
WHERE cr.contract_id = $1
  AND cr.consensus_timestamp != $2
  AND cr.transaction_nonce = 0
ORDER BY cr.consensus_timestamp DESC
LIMIT 25
-- $1 = <encoded contract id>, $2 = 1639010141000000000
```
