### Title
Soft-Deleted Topic Entities Exposed via `GET /api/v1/topics/{id}` Due to Missing `deleted` Flag Check in `EntityServiceImpl.findById()`

### Summary
`EntityServiceImpl.findById()` uses the standard Spring Data `CrudRepository.findById()` which performs a plain primary-key lookup with no filter on the `deleted` column. When a topic's entity record has `deleted=true` (soft-deleted on the Hedera network), any unauthenticated user can still retrieve its full metadata via `GET /api/v1/topics/{id}`, receiving a well-formed `Topic` response that gives no indication the topic was deleted.

### Finding Description
**Code path:**

`TopicController.getTopic()` (line 34) calls `entityService.findById(id.id())`: [1](#0-0) 

`EntityServiceImpl.findById()` delegates directly to `entityRepository.findById(id.getId())`, which is the inherited Spring Data `CrudRepository.findById()` â€” a plain `SELECT * FROM entity WHERE id = ?` with no predicate on `deleted`: [2](#0-1) 

**Root cause â€” inconsistent deleted-flag enforcement:**
`EntityRepository` explicitly adds `and deleted <> true` in `findByAlias` and `findByEvmAddress`, proving the developers intended deleted entities to be invisible. However, the inherited `findById` path has no such guard: [3](#0-2) 

**Mapper produces a clean response with no deletion signal:**
`TopicMapper.map()` maps `entity.autoRenewAccountId`, `entity.createdTimestamp`, `entity.id`, and `entity.timestampRange` â€” but never maps `entity.deleted` â€” so the returned `Topic` JSON contains no indication the topic was deleted: [4](#0-3) 

`TopicServiceImpl.findById()` also uses a plain `CrudRepository.findById()` on the `topic` table, which has no deleted check either: [5](#0-4) 

**Exploit flow:**
1. A topic is created on Hedera â†’ entity row inserted with `deleted=false`.
2. The topic is deleted on Hedera â†’ mirror node sets `deleted=true` on the entity row; the `topic` table row remains.
3. Attacker sends `GET /api/v1/topics/{id}` with no credentials.
4. `topicService.findById()` finds the `topic` row (no deleted check) â†’ succeeds.
5. `entityService.findById()` finds the entity row (no deleted check) â†’ succeeds.
6. `topicMapper.map()` produces a full `Topic` JSON with `adminKey`, `submitKey`, `memo`, `autoRenewAccount`, timestamps, custom fees, etc.
7. Response is `200 OK` with complete topic metadata â€” no `deleted` field, no error.

### Impact Explanation
Any unauthenticated user can retrieve the full metadata of any soft-deleted topic, including sensitive fields such as `adminKey`, `submitKey`, `memo`, and `autoRenewAccount`. This violates the expected access-control invariant that deleted resources return `404 Not Found`. The impact is information disclosure of deleted topic configuration data at medium severity.

### Likelihood Explanation
Exploitation requires zero privileges and zero special tooling â€” a single HTTP GET to a known or guessed topic ID. Topic IDs are sequential integers on Hedera, making enumeration trivial. Any external user who knows (or can guess) a deleted topic's numeric ID can exploit this repeatedly and reliably.

### Recommendation
Add a `deleted` check in `EntityServiceImpl.findById()`, consistent with the pattern already used in `findByAlias` and `findByEvmAddress`:

```java
@Override
public Entity findById(EntityId id) {
    return entityRepository.findById(id.getId())
            .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
            .orElseThrow(() -> new EntityNotFoundException("Entity not found"));
}
```

Alternatively, add a custom query to `EntityRepository`:
```java
@Query("select e from Entity e where e.id = ?1 and (e.deleted is null or e.deleted = false)")
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Also apply the same fix to `TopicServiceImpl.findById()` if the `topic` table carries its own `deleted` column.

### Proof of Concept
**Preconditions:**
- A topic with numeric ID `12345` exists in the mirror node database.
- Its `entity` row has `deleted = true` (topic was deleted on Hedera).
- Its `topic` table row still exists (mirror node retains history).

**Trigger:**
```
GET /api/v1/topics/0.0.12345
```
No authentication header required.

**Expected result:** `404 Not Found`

**Actual result:** `200 OK` with full topic JSON:
```json
{
  "topic_id": "0.0.12345",
  "admin_key": { ... },
  "submit_key": { ... },
  "memo": "sensitive memo",
  "auto_renew_account": "0.0.999",
  "created_timestamp": "...",
  "timestamp": { ... }
}
```
The response contains no `deleted` field and no indication the topic no longer exists on the network.
