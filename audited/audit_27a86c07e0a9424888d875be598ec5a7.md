### Title
Unauthenticated Full-Table Scan via Unindexed `e.balance` Filter Enables DB Connection Pool Exhaustion (DoS)

### Summary
Any unauthenticated user can supply `account.balance=<operator>:<value>` to `GET /accounts`, which routes through `getAccountQuery()` → `getEntityBalanceQuery()` and injects a WHERE predicate on `e.balance` — a column with no database index. Because the REST API's connection pool defaults to 10 connections and carries a 20-second statement timeout, an attacker maintaining ≥10 concurrent requests with a selective balance filter (e.g., `eq:<rare_value>`) forces full sequential scans of the `entity` table, saturating all pool connections and blocking every other API consumer for the duration.

### Finding Description

**Code path (confirmed):**

`getAccounts()` at [1](#0-0)  unconditionally parses the `account.balance` query parameter and maps it to `e.balance`:

```js
const balanceQuery = toQueryObject(utils.parseBalanceQueryParam(req.query, 'e.balance'));
```

This `balanceQuery` is passed as the `entityBalanceQuery` argument to `getAccountQuery()` at [2](#0-1)  with `accountBalanceQuery = undefined` (defaults to `{query:'', params:[]}`).

Inside `getAccountQuery()`, because `includeBalance` is `true` by default, execution reaches `getEntityBalanceQuery()`. [3](#0-2) 

Inside `getEntityBalanceQuery()`, since `accountBalanceQuery.query` is empty, `needHistoricalBalanceInfo` is `false`, so `entityTable = 'entity'` (the live table, not a union). The `entityBalanceQuery.query` is injected directly into the WHERE clause: [4](#0-3) 

The resulting SQL is:
```sql
SELECT ... FROM entity AS e
LEFT JOIN entity_stake es ON es.id = e.id
WHERE e.type IN ('ACCOUNT','CONTRACT')
  AND NOT(e.num = 2 AND e.memo = '...')
  AND e.balance = $1          -- injected from user input
ORDER BY e.id ASC
LIMIT 25
```

**Root cause — no index on `e.balance`:**

The `balance` column was added to the `entity` table by migration `V1.65.0__entity_balance.sql` with no accompanying index: [5](#0-4) 

No subsequent migration creates an index on `entity.balance`. A search across all `*.sql` migration files returns zero hits for any index on that column.

When the attacker supplies a selective filter (e.g., `account.balance=eq:999999999`), PostgreSQL cannot satisfy both `WHERE balance = X` and `ORDER BY id LIMIT 25` using the primary-key index alone. With no index on `balance`, the planner performs a sequential scan of the entire `entity` table (potentially millions of rows on mainnet) to locate matching rows.

**Why `gte:0` specifically:** `gte:0` matches all accounts, so PostgreSQL will use the primary-key index and stop at 25 rows — this particular value is fast. The effective attack uses equality or tight range filters on rare balance values (e.g., `eq:1`, `lte:100`) that force a full scan while still satisfying the LIMIT only after reading most of the table.

**No rate limiting on the REST API:**

The rate-limiting infrastructure found in the codebase (`ThrottleManagerImpl`, `RequestFilter`) belongs exclusively to the `web3` module. The Node.js REST API has no HTTP-level rate limiting middleware. [6](#0-5) 

**Connection pool is small:**

The default pool is 10 connections with a 20-second statement timeout: [7](#0-6) 

The pool is configured in `dbpool.js` directly from these values: [8](#0-7) 

### Impact Explanation

With 10 pool connections and a 20-second statement timeout, an attacker maintaining 10 concurrent requests ties up every available connection for up to 20 seconds per wave. All other REST API consumers (legitimate users, monitoring, health checks) receive connection-timeout errors (`connectionTimeoutMillis = 20000`). The attack is repeatable indefinitely with no authentication, no API key, and no account registration. On a production mainnet mirror node with millions of entity rows, each sequential scan is CPU- and I/O-intensive, compounding the impact on the database host.

### Likelihood Explanation

The preconditions are zero: no account, no token, no special header. A single `curl` loop or a trivial script sending 10–20 concurrent HTTP GET requests is sufficient. The endpoint is publicly documented in the OpenAPI spec. [9](#0-8)  The attack is repeatable, stateless, and requires no knowledge of internal state.

### Recommendation

1. **Add a partial index** on `entity(balance)` where `type IN ('ACCOUNT','CONTRACT')`, or a composite index on `(type, balance, id)` to allow index-only scans for balance-filtered, id-ordered queries.
2. **Add HTTP-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit`) keyed on IP, applied before query execution.
3. **Reject or cap** requests where `account.balance` is the only filter and no `account.id` anchor is provided, or require a minimum selectivity (e.g., disallow open-ended `gte:0`).
4. **Increase `maxConnections`** or deploy a PgBouncer pool in front of the REST API to decouple HTTP concurrency from DB connection exhaustion.
5. **Reduce `statementTimeout`** for the `mirror_api` role to a value well below 20 seconds for list endpoints.

### Proof of Concept

```bash
# Trigger full entity-table scan with a selective balance filter
# Run 15 concurrent requests to exceed the 10-connection pool

for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/accounts?account.balance=eq:1&limit=100" &
done
wait

# Expected result: legitimate requests begin returning 503 / connection timeout errors
# within the 20-second statement_timeout window.
# Repeat the loop to maintain continuous saturation.
```
