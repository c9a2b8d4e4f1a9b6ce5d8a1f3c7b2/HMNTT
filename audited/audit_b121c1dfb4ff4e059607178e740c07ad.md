### Title
Unbounded Scheduler Exhaustion via Unlimited Throttled `subscribeTopic` Connections

### Summary
`PollingTopicMessageRetriever.retrieve()` unconditionally sets `numRepeats = Long.MAX_VALUE` for every throttled subscription and schedules all repeat polls on a single shared `Schedulers.boundedElastic()` instance. Because there is no global cap on concurrent subscriptions and the only per-connection limit (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening many connections, an unprivileged attacker can saturate the shared scheduler's task queue, causing `RejectedExecutionException` for legitimate subscribers and preventing topic messages from being delivered.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructs one `boundedElastic` scheduler at bean-creation time and shares it across every subscription: [1](#0-0) 

Every call to `retrieve(filter, true)` hard-codes `numRepeats = Long.MAX_VALUE` and binds the repeat loop to that shared scheduler: [2](#0-1) [3](#0-2) 

`TopicMessageServiceImpl.subscribeTopic()` calls `retrieve(filter, true)` for every incoming subscription with no admission control: [4](#0-3) 

The `subscriberCount` field is a **metric gauge only** — it enforces nothing: [5](#0-4) 

**Root cause:** The only connection-level guard is `maxConcurrentCallsPerConnection = 5`, which is per-TCP-connection and imposes no global ceiling: [6](#0-5) 

**Why the timeout does not help:** The 60-second `timeout` is an *inactivity* timeout (time between emitted items). On a topic with active messages it never fires; even when it does fire, the attacker immediately reconnects. Additionally, `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 58 retries errors indefinitely, adding further scheduler pressure: [7](#0-6) 

`Schedulers.boundedElastic()` is Reactor's **shared singleton** scheduler (default: 10 × CPU threads, 100 000-task queue). The same instance is also used by `SharedTopicListener.listen()` and the safety-check path in `TopicMessageServiceImpl`, so exhaustion affects the entire reactive pipeline: [8](#0-7) [9](#0-8) 

### Impact Explanation
When the scheduler queue is full, Reactor throws `RejectedExecutionException` for any new scheduled task. This means legitimate subscribers' polling tasks are never executed, so they receive no topic messages. Because the same `boundedElastic` scheduler backs the safety-check and live-listener paths, the entire gRPC topic-message delivery pipeline stalls. The result is a complete denial-of-service for all `subscribeTopic` clients — including monitoring infrastructure and any application relying on HCS message delivery.

### Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. An attacker needs only a network path to port 5600. Opening 200 TCP connections × 5 calls each = 1 000 concurrent throttled subscriptions, each enqueuing a task every 2 seconds. On a 4-core host the default queue cap (~40 000 tasks) is reachable within seconds. The attack is trivially scriptable with any gRPC client library and is fully repeatable; the attacker simply reconnects after any timeout.

### Recommendation
1. **Enforce a global concurrent-subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 500).
2. **Use a per-subscription dedicated scheduler or a rate-limited work queue** instead of the shared `Schedulers.boundedElastic()` singleton, so one subscriber class cannot starve others.
3. **Add per-IP or per-client connection rate limiting** at the Netty/gRPC interceptor layer (e.g., via a `ServerInterceptor` with a token-bucket per remote address).
4. **Set a hard wall-clock subscription lifetime** (not just an inactivity timeout) so long-lived attacker subscriptions are forcibly terminated.
5. Consider setting `maxConcurrentCallsPerConnection` to a lower value and adding a global `maxConnections` limit in the Netty server configuration.

### Proof of Concept
```python
# Requires: grpcio, hedera-mirror-node proto stubs
import grpc, threading, time
from mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-host:5600"
CONNECTIONS = 200   # 200 TCP connections
CALLS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topic_id=...,           # any valid topic ID
        consensus_start_time=Timestamp(seconds=0),
    )
    streams = []
    for _ in range(CALLS_PER_CONN):
        # Each call triggers retrieve(filter, throttled=true) with numRepeats=Long.MAX_VALUE
        s = stub.subscribeTopic(req)
        streams.append(s)
    # Hold streams open indefinitely
    time.sleep(3600)

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
# After ~seconds: boundedElastic queue saturated;
# legitimate subscribers receive RejectedExecutionException / no messages
```

**Expected result:** Legitimate `subscribeTopic` clients stop receiving messages; server logs show `RejectedExecutionException` from the `boundedElastic` scheduler. The `hiero.mirror.grpc.subscribers` gauge climbs to 1 000+ with no enforced ceiling.
