### Title
`getEncodedId()` Skips DB Existence Check for Numeric Entity IDs, Silently Returning Encoded IDs for Non-Existent Entities

### Summary
`EntityService.getEncodedId()` applies a DB existence check only for EVM address and alias inputs, but for numeric encoded entity IDs it performs a pure local computation and returns the encoded ID without any DB lookup — even when `requireResult=true`. An unprivileged user can submit any syntactically valid numeric entity ID (e.g., `"9999999999"`) that does not exist in the database, receive a valid-looking encoded ID back, and cause all downstream DB queries using that ID to silently return empty results rather than a proper `NotFoundError`.

### Finding Description
**Exact code path:**

In `rest/service/entityService.js` lines 118–137, `getEncodedId()` branches on `EntityId.isValidEntityId()`:

```javascript
// entityService.js lines 120-124
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()                                    // ← pure local computation, NO DB check
    : await this.getEntityIdFromEvmAddress(entityId, requireResult); // ← DB check
}
```

In `rest/entityId.js` line 134, `isValidEntityId()` accepts numeric strings via `encodedEntityIdRegex`:

```javascript
// entityId.js line 134
if ((typeof entityId === 'string' && entityIdRegex.test(entityId)) || encodedEntityIdRegex.test(entityId)) {
  return true;
}
```

where `encodedEntityIdRegex = /^-?\d{1,19}$/` — any integer string up to 19 digits passes.

**Root cause:** The failed assumption is that `requireResult=true` uniformly enforces DB existence. It does for aliases (`getAccountIdFromAlias` throws `NotFoundError`) and EVM addresses (`getEntityIdFromEvmAddress` throws `NotFoundError`), but for numeric IDs the branch at line 123 calls `entityId.getEncodedId()` — a pure bit-manipulation function in `entityId.js` lines 67–82 — and returns immediately without touching the DB.

**Exploit flow:**
1. Attacker submits `"9999999999"` (a syntactically valid encoded entity ID matching `encodedEntityIdRegex`).
2. `isValidEntityId("9999999999")` → `true`.
3. `parseString` → `parseCached` → `parseFromEncodedId` decodes it to shard/realm/num; `evmAddress = null`.
4. `entityId.getEncodedId()` returns the numeric value directly.
5. Caller receives a valid-looking `BigInt`/`Number` encoded ID.
6. Caller uses this ID in a DB query (e.g., `entityExistenceQuery`, contract lookup, token lookup).
7. DB returns zero rows; caller either returns an empty response or a misleading 404 — not the `NotFoundError` that `requireResult=true` was supposed to guarantee.

**Why existing checks fail:** The `requireResult` parameter is documented to throw `NotFoundError` "if the entity is not present," but the code only honors this contract for the EVM address and alias branches. The numeric branch has no corresponding guard.

### Impact Explanation
Any API endpoint that calls `getEncodedId(input, true)` expecting a guaranteed-existent entity ID will silently proceed with a phantom ID. Downstream DB queries return empty result sets. Callers that assume a returned ID implies existence (e.g., skipping their own existence checks) will produce inconsistent or misleading API responses. The `requireResult=true` contract is broken for the numeric input class, creating a "found ID but no data" state that can confuse callers and clients. Severity is low-to-medium: no data is leaked or corrupted, but the API contract is violated and the behavior is exploitable by any unauthenticated user with knowledge of the ID format.

### Likelihood Explanation
Exploitation requires zero privileges and zero special knowledge beyond the public API. The `encodedEntityIdRegex` accepts any integer string up to 19 digits, so an attacker can trivially enumerate or guess IDs that do not exist. The attack is repeatable at will and requires only a single HTTP request per attempt.

### Recommendation
When `requireResult=true` and the input is a numeric entity ID (i.e., `entityId.evmAddress === null` after parsing), add a DB existence check before returning, consistent with the alias and EVM address branches:

```javascript
// entityService.js, inside getEncodedId(), numeric branch
const encodedId = entityId.getEncodedId();
if (requireResult && !(await this.isValidAccount(encodedId))) {
  throw new NotFoundError();
}
return encodedId;
```

Alternatively, document explicitly that `requireResult` has no effect for numeric IDs and update all callers to perform their own existence checks.

### Proof of Concept
**Precondition:** Entity with encoded ID `9999999999` does not exist in the DB.

**Trigger:**
```
GET /api/v1/accounts/9999999999
```
or any endpoint that internally calls:
```javascript
await EntityService.getEncodedId("9999999999", true);
```

**Result:**
- `getEncodedId()` returns `9999999999` without throwing `NotFoundError`.
- Downstream DB query (e.g., `entityExistenceQuery` with `$1 = 9999999999`) returns zero rows.
- API returns an empty or misleading response instead of a proper 404 with a "not found" error body.
