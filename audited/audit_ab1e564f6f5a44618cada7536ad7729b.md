### Title
Unauthenticated Full-Table-Scan DoS via `/balances` Without Timestamp Filter

### Summary
When `GET /api/v1/balances` is called without a `timestamp` parameter, the code builds a query whose `entity_balance` CTE unconditionally materializes every `ACCOUNT`/`CONTRACT` row in the `entity` table before the outer `LIMIT` is applied. Because the REST API has no rate limiting and only 10 pooled DB connections each capped at a 20-second `statement_timeout`, a small number of concurrent unauthenticated requests is sufficient to exhaust the connection pool and deny service to all other API consumers. The correlated `getTokenAccountBalanceSubQuery()` executes only `LIMIT` times (25–100), not once per account; the true amplifier is the unbounded CTE full-table scan.

### Finding Description

**Root cause:** The `entity_balance` CTE carries no caller-supplied filter. PostgreSQL treats a plain CTE as an optimization fence (materialized by default in PG ≤ 11; still frequently materialized in PG 12+ when the planner cannot prove inlining is safe). This forces a sequential scan of the entire `entity` table — potentially millions of rows — before the outer `ORDER BY … LIMIT` is evaluated. The correlated subquery then runs only `LIMIT` times (default 25, max 100), but the full-table scan and sort dominate execution time.

**Failed assumption:** The developers assumed the outer `LIMIT` clause would bound the work done by the query. It bounds the correlated subquery invocations, but it does not prevent the CTE from being fully materialized first.

**No rate limiting on the REST API.** The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` (EVM/contract-call) service. A `grep` across `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns only one test-utility hit — confirming zero rate limiting on the Node.js REST layer.

### Impact Explanation

With `maxConnections = 10` and `statement_timeout = 20 000 ms`, an attacker who sends 10 concurrent `GET /api/v1/balances` requests (no timestamp, no credentials) saturates the entire connection pool for up to 20 seconds per wave. Every other REST endpoint that requires a DB connection (`/transactions`, `/accounts`, `/topics`, etc.) will either queue indefinitely or fail with a connection-timeout error during that window. Repeating the wave every ~20 seconds produces a near-continuous denial of service for the entire mirror-node REST API — matching the "Critical: Network not being able to confirm new transactions (total network shutdown)" severity label, because downstream consumers (wallets, explorers, relay operators) lose the ability to query transaction status.

### Likelihood Explanation

- **No authentication required** — the endpoint is fully public per the OpenAPI spec.
- **No rate limiting** — confirmed absent in the REST layer.
- **Trivially reproducible** — a single `curl` loop or any HTTP load tool suffices.
- **Low attacker sophistication** — no special knowledge of the codebase is needed; the attack is a plain HTTP flood against one endpoint.
- **Repeatable indefinitely** — the `statement_timeout` terminates each query, freeing the connection, but the attacker simply re-issues requests.

### Recommendation

1. **Add a `LIMIT` guard to the CTE** or rewrite the no-timestamp branch to push the account-id ordering directly onto the `entity` table with an index scan, avoiding full materialization:
   ```sql
   SELECT e.id AS account_id, e.balance, e.balance_timestamp AS consensus_timestamp,
          (<tokenSubQuery>) AS token_balances
   FROM entity e
   WHERE e.type IN ('ACCOUNT', 'CONTRACT')
   <whereClause>
   ORDER BY e.id <order>
   <limitQuery>
   ```
   This lets PostgreSQL use the primary-key index and stops after `LIMIT` rows without scanning the whole table.

2. **Add per-IP or global rate limiting** to the Node.js REST layer (e.g., `express-rate-limit`) for expensive endpoints, mirroring the throttle already present in the web3 service.

3. **Reduce `statement_timeout`** for the REST DB user from 20 s to a value consistent with expected query latency (e.g., 5 s), limiting the window during which each connection is held.

4. **Increase `maxConnections`** or use a separate, smaller pool for the `/balances` endpoint to prevent it from starving other endpoints.

### Proof of Concept

```bash
# Send 10 concurrent timestamp-free balance requests (no auth needed)
for i in $(seq 1 10); do
  curl -s "https://<mirror-node-host>/api/v1/balances" &
done
wait

# Immediately probe another endpoint — expect connection-timeout or 503
curl -v "https://<mirror-node-host>/api/v1/transactions?limit=1"
```

With a large `entity` table (millions of accounts), each of the 10 background requests triggers a full sequential scan, holds a DB connection for up to 20 seconds, and the final `transactions` probe fails to acquire a connection from the exhausted pool. Repeating the loop every 15–20 seconds sustains the outage.
