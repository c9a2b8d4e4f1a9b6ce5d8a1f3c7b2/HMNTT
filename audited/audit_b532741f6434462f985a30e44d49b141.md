### Title
Global Rate-Limit Bucket Allows Single Unprivileged User to Monopolize Full Request Capacity via Burst Consumption

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` is a single application-wide singleton with a capacity of 500 tokens and no per-source-IP or per-user partitioning. Because the bucket initializes full and `refillGreedy` continuously accumulates up to the full 500-token capacity after any ~1-second quiet period, a single unauthenticated attacker can drain all 500 tokens in one burst, causing every concurrent legitimate user to receive HTTP 429 for the remainder of that refill window. This is repeatable every second indefinitely.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` lines 24–32 — `rateLimitBucket()` creates a Spring `@Bean` (singleton scope) with `capacity(rateLimit)` and `refillGreedy(rateLimit, Duration.ofSeconds(1))` where `rateLimit` defaults to 500 (`ThrottleProperties.requestsPerSecond = 500`). [1](#0-0) 

`ThrottleManagerImpl.java` line 38 — every inbound request calls `rateLimitBucket.tryConsume(1)` against this single shared bucket instance, with no per-IP, per-connection, or per-identity keying. [2](#0-1) 

**Root cause and failed assumption:**
The design assumes the 500 RPS budget will be distributed across many concurrent users. The failed assumption is that no single client can consume a disproportionate share. Because the bucket is global and starts full (bucket4j initializes with full capacity), and because `refillGreedy` continuously accumulates tokens up to the 500-token ceiling, any single client that fires 500 concurrent requests will atomically drain the entire budget. There is no per-source rate limit, no connection-level limit, and no IP-based partitioning anywhere in the throttle stack. [3](#0-2) 

The `RequestFilter` system only matches on request-body fields (`BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE`) — it has no concept of source IP and cannot be used to implement per-client limiting. [4](#0-3) 

Notably, `rateLimitBucket` does **not** use `SynchronizationStrategy.SYNCHRONIZED` (unlike `gasLimitBucket`), meaning concurrent token consumption races are resolved optimistically, which actually makes the burst drain faster under high concurrency. [5](#0-4) 

### Impact Explanation
An attacker with a single machine and a standard HTTP client can fire 500 concurrent POST requests to any web3 endpoint (e.g., `/api/v1/contracts/call`). All 500 tokens are consumed in milliseconds. Every other user hitting the service during that window receives HTTP 429 `"Requests per second rate limit exceeded"`. The attack is repeatable every ~1 second (the refill period), creating a sustained, near-total denial of service for all legitimate users. The web3 API is a public-facing JSON-RPC-compatible endpoint with no authentication requirement, making the blast radius the entire user population of the node. [6](#0-5) 

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The attacker only needs network access to the web3 endpoint. The exploit requires nothing more than a standard HTTP client capable of sending 500 concurrent requests (e.g., `ab`, `wrk`, `curl` with `xargs`, or a trivial script). The attack is fully repeatable every second, is stateless, and leaves no persistent side effects that would impede re-execution. Any motivated party — including competitors, censors, or vandals — can execute this with minimal effort.

### Recommendation
1. **Add per-IP rate limiting** at the application layer using a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), so each source IP gets its own token bucket rather than sharing the global one.
2. **Alternatively, enforce per-IP limits at the infrastructure layer** (e.g., Traefik `inFlightReq` with `ipStrategy`, nginx `limit_req_zone`, or a WAF/API gateway) before requests reach the Spring application.
3. **Reduce burst capacity** by using `refillIntervally` instead of `refillGreedy`, which prevents accumulation of tokens during idle periods and limits each interval to exactly `rateLimit/N` tokens, reducing the maximum burst size.
4. **Add connection-level limiting** (max concurrent connections per IP) to prevent a single client from holding many in-flight requests simultaneously.

### Proof of Concept
```bash
# Prerequisites: web3 endpoint reachable at TARGET, standard HTTP client (Apache Bench)
TARGET="http://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD='{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'

# Step 1: Wait ~1 second for bucket to fully refill (or exploit at startup)
sleep 1

# Step 2: Fire 500 concurrent requests from a single source IP
ab -n 500 -c 500 -p /tmp/payload.json -T application/json "$TARGET"
# All 500 requests succeed (HTTP 200) — bucket is now empty

# Step 3: Immediately after, a legitimate user's request is denied
curl -s -o /dev/null -w "%{http_code}" -X POST "$TARGET" \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD"
# Returns: 429 (Too Many Requests) — "Requests per second rate limit exceeded"

# Step 4: Repeat step 2 every second to maintain sustained DoS
for i in $(seq 1 60); do
  ab -n 500 -c 500 -p /tmp/payload.json -T application/json "$TARGET" &
  sleep 1
done
```

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()

                .build();
