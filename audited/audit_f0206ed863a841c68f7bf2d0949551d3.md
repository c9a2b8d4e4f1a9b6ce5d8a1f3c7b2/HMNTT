### Title
Missing Account Existence Validation in `getAccountCryptoAllowances` Enables Unauthenticated DB Query Flooding

### Summary
`getAccountCryptoAllowances()` resolves numeric account IDs via `EntityService.getEncodedId()` without verifying the account exists in the database before issuing a `SELECT * FROM crypto_allowance WHERE owner = $1 AND amount > 0` query. An unauthenticated attacker can flood the endpoint with arbitrary non-existent numeric account IDs, generating a sustained stream of DB queries that consume connection pool slots. The default pool size is 10 connections, and no application-level rate limiting exists on the Node.js REST API.

### Finding Description
**Exact code path:**

In `rest/controllers/cryptoAllowanceController.js` lines 76–80:
```js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  // no isValidAccount check here
  const {conditions, params, order, limit} = this.extractCryptoAllowancesQuery(filters, accountId);
  const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(conditions, params, order, limit);
```

In `rest/service/entityService.js` lines 120–123, `getEncodedId()` for a numeric ID (e.g. `0.0.9999999`) takes the fast path:
```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()   // ← returns encoded ID with NO DB lookup
```

No existence check is performed. The encoded ID is passed directly to `CryptoAllowanceService.getAccountCryptoAllowances()`, which executes `SELECT * FROM crypto_allowance WHERE owner = $1 AND amount > 0` against the database for every request, regardless of whether the account exists.

**Failed assumption:** The code assumes that resolving a syntactically valid numeric account ID is sufficient before querying `crypto_allowance`. It is not — the account may not exist.

**Why existing checks are insufficient:** `EntityService.isValidAccount()` exists and is used by the comparable `listStakingRewardsByAccountId` handler (`accountController.js` lines 172–175), which explicitly calls it and throws `NotFoundError` for non-existent accounts. `getAccountCryptoAllowances` omits this check entirely, creating an inconsistency. The `getEncodedId()` function itself only performs a DB lookup for EVM addresses and aliases, not for numeric IDs.

### Impact Explanation
The REST API database pool defaults to **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, `docs/configuration.md` line 556) with a `statementTimeout` of 20 seconds. An attacker flooding the endpoint with distinct non-existent numeric account IDs generates one `crypto_allowance` query per request. While each individual query is fast (indexed lookup returning empty), at sufficient request volume the pool can be saturated, causing legitimate requests to queue or time out with connection timeout errors (`connectionTimeout = 20000ms`). This degrades or denies service to the mirror node REST API. **Note:** this does not affect the Hedera network's consensus or transaction confirmation — the mirror node is a read-only observer. The actual impact is REST API availability degradation, not network shutdown.

### Likelihood Explanation
Exploitation requires no credentials, no special knowledge, and no prior setup. Any HTTP client can issue `GET /api/v1/accounts/0.0.<random_large_number>/allowances/crypto` in a loop. The endpoint is publicly documented in the OpenAPI spec. The absence of application-level rate limiting on the Node.js REST API (unlike the web3/Rosetta services which have Traefik middleware rate limits) means the only throttle is the DB pool itself. This is trivially repeatable and automatable.

### Recommendation
Add an account existence check before querying `crypto_allowance`, consistent with the pattern already used in `listStakingRewardsByAccountId`:

```js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValid = await EntityService.isValidAccount(accountId);
  if (!isValid) {
    throw new NotFoundError();
  }
  // ... rest of handler
```

Additionally, consider deploying Traefik rate-limiting middleware for the REST API (as already done for Rosetta) and increasing the default pool size for production deployments.

### Proof of Concept

for i in $(seq 1000000 1001000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.$i/allowances/crypto" &

wait
# Observe: all return HTTP 200 with {"allowances":[],"links":{"next":null}}
# instead of 404; DB pool connections are consumed for each request.
