### Title
Unbounded `keys` Collection Passed to SQL `IN` Clause Enables Database Denial of Service

### Summary
In `HookServiceImpl.getHookStorageChange()`, when a user supplies a non-empty `timestamp` bound and a large `keys` collection, the full unfiltered `keys` collection is forwarded directly to `hookStorageChangeRepository.findByKeyInAndTimestampBetween()`. This generates an arbitrarily large SQL `IN` clause with no server-side size cap, allowing any unprivileged caller to exhaust database resources.

### Finding Description

``` [1](#0-0) 

The repository query at line 54 of `HookStorageChangeRepository.java`:

```sql
and key in (:keys)
``` [2](#0-1) 

expands `:keys` into one SQL bind parameter per element. There is no `@Size` or any other constraint on the `keys` field in `HookStorageRequest`: [3](#0-2) 

**Root cause / failed assumption:** The guard on line 93 only short-circuits when *every* supplied key falls outside the key range. As soon as a single key is in range, `keysInRange` is non-empty, the guard is bypassed, and the *entire* original `keys` collection (not the filtered `keysInRange`) is forwarded to the SQL query. The `limit` / `Pageable` parameter only caps the *result* rows, not the size of the `IN` list.

### Impact Explanation

A single HTTP request carrying tens of thousands of key values causes the database to parse and evaluate a massive `IN` clause against the `hook_storage_change` table. Repeated requests can saturate database CPU, memory, and connection pools, making the mirror-node REST API unavailable to all users (denial of service). No data is exfiltrated, but availability is fully compromised.

### Likelihood Explanation

The endpoint is reachable by any unprivileged API consumer. The attacker needs only to:
1. Know the API schema (public documentation).
2. Craft a single request with a valid `timestamp` filter and a large `keys` array.

No credentials, no special role, and no prior knowledge of stored data are required. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Cap input size at the DTO/controller layer**: add a `@Size(max = N)` (e.g., `max = 1000`) constraint on `HookStorageRequest.keys`.
2. **Pass `keysInRange` instead of `keys` to the repository**: replace `keys` with `keysInRange` on line 104–105 of `HookServiceImpl.java` so the already-filtered, smaller list is used.
3. **Enforce the cap in `getKeysInRange()`**: if `keys.size()` exceeds the allowed maximum, throw an `IllegalArgumentException` before filtering.

### Proof of Concept

```
GET /api/v1/hooks/{ownerId}/storage?timestamp=gte:0&key=0x00&key=0x01&key=0x02...
    (repeat key= parameter 50,000 times with at least one value inside the key range)
```

1. `timestamp=gte:0` makes `isHistorical()` return `true` → enters `getHookStorageChange()`.
2. At least one key is within the default key range → `keysInRange` is non-empty → guard on line 93 is bypassed.
3. All 50,000 keys are passed to `findByKeyInAndTimestampBetween()`.
4. The database receives `WHERE key IN (val_1, val_2, ..., val_50000) AND consensus_timestamp BETWEEN 0 AND 9223372036854775807`.
5. Repeated requests exhaust database resources, causing service-wide denial of service.
