### Title
Unauthenticated GraphQL `findByEvmAddress` Endpoint Lacks Rate Limiting and Caching, Enabling Thread-Pool Exhaustion DoS During DB Unavailability

### Summary
The GraphQL service exposes `account(input: {evmAddress: ...})` publicly with no authentication, no rate limiting, and no caching on `EntityRepository.findByEvmAddress`. During a network partition (or DB slowdown), each concurrent request blocks a Tomcat thread waiting for a HikariCP connection until timeout, allowing an unprivileged attacker sending a modest flood of concurrent requests to exhaust the thread pool and render the service completely unresponsive.

### Finding Description

**Exact code path:**

`POST /graphql/alpha` → `AccountController.account()` (line 51-54) → `EntityServiceImpl.getByEvmAddressAndType()` → `EntityRepository.findByEvmAddress(byte[])` (line 16-17) → synchronous blocking JDBC query. [1](#0-0) [2](#0-1) 

**Root cause — three absent controls, all confirmed:**

1. **No rate limiting in the graphql module.** The `ThrottleConfiguration`/`ThrottleManagerImpl` (bucket4j) exist exclusively in the `web3` module. A grep for `rate.limit|throttle|RateLimiter|bucket4j` across all `graphql/**` files returns zero matches. [3](#0-2) 

2. **No caching on `findByEvmAddress` in the graphql module.** The graphql `EntityRepository` carries no `@Cacheable` annotation, so every request unconditionally hits the database. (Contrast: the `web3` module's `EntityRepository` annotates the equivalent method with `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)`.) [4](#0-3) 

3. **No authentication or security filter.** No `SecurityConfiguration` exists in the graphql module. The `graphql/src/main/resources/` directory contains only `banner.txt`. The endpoint is fully public. [5](#0-4) 

**Why existing checks are insufficient:**

The `GraphQlConfiguration` enforces query complexity ≤ 200, depth ≤ 10, and JSON/token size limits. These controls prevent a single malicious *query* from being expensive, but they do not bound the *number of concurrent requests*. A simple, schema-valid `{ account(input: { evmAddress: "0x..." }) { id } }` passes all instrumentation checks and proceeds directly to the blocking DB call. [6](#0-5) 

**Blocking mechanics during a network partition:**

The graphql service uses HikariCP (configured via `CommonConfiguration`) with Spring Boot defaults: pool size 10, `connectionTimeout` 30 s. During a network partition, active DB connections hang until TCP keepalive or socket timeout fires (minutes). New requests queue in HikariCP waiting for a free connection (up to 30 s each). Tomcat's default thread pool (200 threads) is consumed by threads blocked in `HikariDataSource.getConnection()`. Once all 200 threads are occupied, the HTTP acceptor stops accepting new connections and the service is fully unresponsive. [7](#0-6) 

### Impact Explanation

Complete denial of service for the GraphQL API. All legitimate account lookups fail. Because the GraphQL service is the only public query surface for account data by EVM address, this affects all downstream integrators and users. The attack requires no credentials, no special knowledge beyond the public schema (documented in `postman.json` and `account.graphqls`), and is repeatable indefinitely. [8](#0-7) 

### Likelihood Explanation

Preconditions: none beyond internet access. The attacker does not need privileged keys, internal network access, or leaked secrets. The attack is amplified by any real or attacker-induced DB slowdown (e.g., a separate volumetric attack on the DB tier). Even without a partition, sustaining ~200 concurrent requests with slow DB responses achieves the same thread exhaustion. The attack is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool.

### Recommendation

1. **Add per-IP rate limiting to the GraphQL endpoint** using bucket4j (already a project dependency in `web3`) or a servlet filter, mirroring the pattern in `ThrottleConfiguration`.
2. **Add `@Cacheable` to `EntityRepository.findByEvmAddress`** in the graphql module, matching the caching already present in the `web3` module's `EntityRepository`.
3. **Set an explicit HikariCP `connectionTimeout`** (e.g., 5 s) and **`maximumPoolSize`** in the graphql application configuration so that DB unavailability causes fast-fail HTTP 503 responses rather than thread blocking.
4. **Configure Tomcat's `max-connections` and `accept-count`** to bound the queue depth and return 503 early under overload.

### Proof of Concept

```bash
# During a network partition (or with DB artificially slowed):
# Send 500 concurrent requests with a valid 20-byte EVM address
seq 1 500 | xargs -P 500 -I{} curl -s -X POST http://<graphql-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: { evmAddress: \"0xabcdef1234567890abcdef1234567890abcdef12\" }) { id } }"}'

# Expected result:
# - First ~10 requests acquire DB connections and block (partition scenario)
# - Remaining requests queue in HikariCP (30 s timeout each)
# - Tomcat threads fill up; new legitimate requests receive connection refused or timeout
# - Service remains unresponsive until DB connectivity is restored AND all queued threads drain
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
