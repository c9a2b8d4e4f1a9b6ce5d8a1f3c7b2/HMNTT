### Title
Cache Key Fragmentation via Whitespace Variations in CachedPreparsedDocumentProvider

### Summary
`CachedPreparsedDocumentProvider.getDocumentAsync()` uses the raw query string (`executionInput.getQuery()`) as the Caffeine cache key with no normalization. An unprivileged attacker can send semantically identical GraphQL queries with minor whitespace variations to generate unique cache keys, filling the 1000-entry cache with useless entries and forcing all legitimate users to incur repeated parse-and-validate overhead on every request.

### Finding Description
**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26, inside `getDocumentAsync()`:

```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

The cache key is the verbatim query string returned by `executionInput.getQuery()`. No normalization (whitespace collapsing, AST-based fingerprinting, or hashing after stripping insignificant tokens) is applied before the lookup.

The cache is configured with `maximumSize=1000,expireAfterWrite=1h` (default in `CacheProperties.java`, line 15). Caffeine's Window TinyLFU eviction is frequency-based: entries that are accessed rarely are evicted first. An attacker who continuously submits novel whitespace-variant queries will keep inserting new low-frequency entries, displacing legitimate high-frequency entries.

**Why existing checks are insufficient:**
- `maxWhitespaceTokens(1000)` (set in `GraphQlConfiguration.java`, line 34) limits the number of whitespace tokens *within a single query*, but does not prevent an attacker from sending many distinct queries each with slightly different whitespace placement.
- `maxCharacters(10000)` and `maxTokens(1000)` bound the cost of a single parse but do not prevent cache flooding.
- There is no per-IP rate limiting, query deduplication, or normalization layer visible in the codebase before the cache is consulted.

**Exploit flow:**
1. Attacker identifies any valid GraphQL query (e.g., from the public GraphiQL interface at `/graphiql`).
2. Attacker generates 1000+ variants: `{account(...){balance}}`, `{ account(...){ balance }}`, `{\naccount(...)\n{\nbalance\n}\n}`, etc. Each is syntactically and semantically identical but differs in whitespace.
3. Each variant is a distinct string → distinct cache key → distinct cache entry.
4. After 1000 submissions the cache is full of attacker-controlled entries. Caffeine evicts the least-frequently-used entries (legitimate user queries) to make room.
5. Legitimate users' queries are no longer cached; every request triggers `parseAndValidateFunction.apply(executionInput)` — full parse + schema validation — on the hot path.
6. Attacker repeats continuously to maintain cache saturation.

### Impact Explanation
Cache hit rate for legitimate users degrades toward zero. Every GraphQL request must re-parse and re-validate the query document, increasing CPU consumption and latency for all users of the service. The `maximumSize=1000` bound limits heap growth but does not prevent the denial-of-cache-service: 1000 cached AST entries (each holding a full parsed document) also represent non-trivial heap pressure. The attack degrades service quality for all users without requiring any credentials.

### Likelihood Explanation
The attack requires no authentication, no special knowledge beyond a single valid query, and is trivially automatable with a script that appends one extra space or newline per request. The GraphQL endpoint is publicly accessible. The attack is repeatable and persistent as long as the attacker keeps sending new variants (the 1-hour TTL means entries do not self-evict quickly). Any external user can execute this.

### Recommendation
Normalize the query string before using it as a cache key. Options in increasing robustness:

1. **Whitespace normalization**: Strip/collapse all insignificant whitespace before the `cache.get(...)` call:
   ```java
   String normalizedKey = executionInput.getQuery().replaceAll("\\s+", " ").trim();
   return cache.get(normalizedKey, key -> parseAndValidateFunction.apply(executionInput));
   ```
2. **AST-based fingerprinting**: Parse the query once to an AST and use a canonical printed form or a hash of the AST as the key (more robust, handles comment stripping too).
3. **Rate limiting**: Add per-IP or per-client request rate limiting at the HTTP layer to bound the rate at which new cache entries can be inserted.

### Proof of Concept
```bash
# Send 1000 whitespace-variant queries to fill the cache
for i in $(seq 1 1000); do
  SPACES=$(printf '%*s' "$i" '')
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{${SPACES}account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}\"}" \
    > /dev/null
done

# Now send the canonical query — it will miss the cache and be re-parsed every time
for i in $(seq 1 100); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query": "{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}' \
    > /dev/null
done
# Observe via /actuator/metrics or cache recordStats that hit rate is near 0%
