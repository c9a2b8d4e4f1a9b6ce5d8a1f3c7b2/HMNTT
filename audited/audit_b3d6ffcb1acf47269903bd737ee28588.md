### Title
Unauthenticated Connection Pool Exhaustion via Parallel DB Queries in `getContractResultsByTimestamp`

### Summary
The `GET /api/v1/contracts/:contractId/results/:consensusTimestamp` endpoint is publicly accessible with no authentication or rate limiting. Each successful request sequentially fires 1 DB query then 5 parallel DB queries via `Promise.all`, consuming up to 6 connections simultaneously from a default pool of 10. An attacker sending as few as 2 concurrent requests with valid contract/timestamp pairs can exhaust the entire connection pool, denying service to all other API consumers.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 18 registers the route with no auth middleware: [1](#0-0) 

`getContractResultsByTimestamp` in `rest/controllers/contractController.js` (lines 1002–1042) executes:
1. **Query 1 (sequential):** `ContractService.getInvolvedContractsByTimestampAndContractId(timestamp, contractId)` — 1 DB connection
2. If `contractDetails` is non-null, calls `getDetailedContractResults` which fires **5 parallel queries** via `Promise.all`: [2](#0-1) 

The 5 parallel queries are:
- `ContractService.getContractResultsByTimestamps`
- `TransactionService.getEthTransactionByTimestampAndPayerId`
- `RecordFileService.getRecordFileBlockDetailsFromTimestamp`
- `ContractService.getContractLogsByTimestamps`
- `ContractService.getContractStateChangesByTimestamps`

3. Optionally a 6th query: `FileDataService.getLatestFileDataContents` if `ethTransaction.callDataId` is valid. [3](#0-2) 

**Root cause:** The default connection pool is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections` default = 10): [4](#0-3) [5](#0-4) 

With 5 connections held simultaneously per request, only **2 concurrent requests** are needed to exhaust the pool (2 × 5 = 10). The `connectionTimeoutMillis` is 20,000 ms and `statement_timeout` is 20,000 ms, meaning each connection is held for up to 20 seconds per request, creating a sustained exhaustion window. [6](#0-5) 

**Failed assumption:** The design assumes the pool is large enough relative to concurrent load, but the default of 10 is trivially exhausted by this endpoint's fan-out pattern.

### Impact Explanation

When the pool is exhausted, every other API endpoint that requires a DB connection (transactions, accounts, tokens, etc.) will queue waiting for a connection and fail after `connectionTimeoutMillis` (20 s). This is a complete non-network DoS of the REST API service — all endpoints become unavailable, not just the contract endpoint. The `statementTimeout` provides no protection because it limits query *duration*, not the number of concurrent connections held. [7](#0-6) 

### Likelihood Explanation

No privileges are required. The Hedera network is public; valid `contractId`/`consensusTimestamp` pairs are trivially enumerable from public block explorers or the mirror node's own `/api/v1/contracts` and `/api/v1/transactions` endpoints. An attacker needs only a script sending ~10–20 concurrent HTTP GET requests to a single well-known endpoint. No authentication, no API key, no special knowledge beyond a valid contract address and one of its transaction timestamps. The attack is repeatable indefinitely and requires minimal resources.

### Recommendation

1. **Increase default pool size** or document a minimum safe value relative to expected concurrency. The default of 10 is dangerously low for an endpoint that holds 5 connections simultaneously.
2. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or an in-flight request counter) specifically for this endpoint, similar to the `inFlightReq` middleware used in the Rosetta Helm chart. [8](#0-7) 
3. **Serialize the DB queries** instead of using `Promise.all` — or batch them into a single query — to reduce peak connection consumption from 5 to 1 per request.
4. **Add a per-IP or global rate limit** to the REST API server middleware stack (currently absent from `rest/server.js`). [9](#0-8) 

### Proof of Concept

```bash
# Step 1: Find a valid contractId and consensusTimestamp
CONTRACT_ID="0.0.1234"  # any deployed contract
TIMESTAMP="1234567890.000000000"  # from /api/v1/contracts/$CONTRACT_ID/results

# Step 2: Send 15 concurrent requests (well above pool exhaustion threshold)
for i in $(seq 1 15); do
  curl -s "https://<mirror-node>/api/v1/contracts/$CONTRACT_ID/results/$TIMESTAMP" &
done
wait

# Step 3: Observe that concurrent requests to OTHER endpoints now timeout or return 503
curl -s "https://<mirror-node>/api/v1/transactions"
# Expected: connection timeout / pool exhaustion error after ~20s
```

With pool size 10 and 5 connections consumed per in-flight request, 3 concurrent requests to this endpoint hold all 10 connections, causing all other API calls to queue and eventually time out after 20 seconds.
