### Title
Unauthenticated Connection Pool Exhaustion via Concurrent /block and /block/transaction Requests

### Summary
The Rosetta API exposes DB-backed endpoints (`/block`, `/block/transaction`) with no application-level rate limiting or concurrency control. Each request acquires a database connection and holds it for up to `Db.StatementTimeout` (default 20 seconds). An unprivileged attacker sending 100 concurrent requests can exhaust `Pool.MaxOpenConnections` (default 100), causing all subsequent legitimate requests to queue indefinitely and rendering the Rosetta API unavailable.

### Finding Description

**Code locations:**

- `rosetta/app/config/types.go` lines 77–81: `Pool.MaxOpenConnections` is a plain integer field with no enforced upper bound.
- `rosetta/app/db/db.go` line 33: `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)` — the pool hard cap is set to the configured value (default 100).
- `rosetta/app/db/client.go` lines 35–37: every DB-backed call creates a child context with `time.Duration(d.statementTimeout)*time.Second` (default 20s). The connection is held until the query completes or the timeout fires.
- `rosetta/main.go` lines 217–219: the middleware stack is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. None of these implement rate limiting, concurrency limiting, or a semaphore over DB access.
- `rosetta/app/persistence/transaction.go` lines 120–147: `FindBetween` acquires one connection via `GetDbWithContext` and then loops in batches of 2000 rows, holding that connection for the entire loop duration — potentially the full 20-second timeout window.
- `rosetta/app/services/block_service.go` lines 51–58: a single `/block` request calls both `RetrieveBlock` (one DB query) and `FindBetween` (potentially many batched DB queries), each acquiring a connection.

**Root cause:** The application assumes an upstream proxy enforces concurrency limits. There is no in-process semaphore, `http.Server.MaxHeaderBytes`, or connection-pool-aware request throttle. Go's `database/sql` blocks callers waiting for a free connection when the pool is exhausted, so all goroutines pile up indefinitely.

**Exploit flow:**
1. Attacker identifies the Rosetta API port (default 5700) — no authentication is required.
2. Attacker sends 100 concurrent POST requests to `/block` (or `/block/transaction`) with valid-looking but slow-to-resolve block identifiers (e.g., a very early block index that triggers the genesis init path and the full `FindBetween` loop).
3. Each goroutine calls `GetDbWithContext`, acquires one of the 100 pool connections, and holds it for up to 20 seconds.
4. The pool is fully exhausted. All subsequent legitimate requests block in `db.DB().Acquire()` with no timeout on the wait side.
5. The Rosetta API is effectively unavailable for the full 20-second window; the attacker re-floods before connections are released, sustaining the outage indefinitely.

**Why existing checks fail:**

The Helm chart (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160) defines a Traefik `inFlightReq: amount: 5` per source IP and `rateLimit: average: 10` per request host. These are insufficient because:
- They are optional Kubernetes deployment configuration, not enforced at the application layer. A bare deployment without Traefik has zero protection.
- The per-IP limit of 5 is bypassable: 20 source IPs × 5 concurrent requests = 100 connections, exhausting the pool while staying within per-IP limits.
- The `rateLimit: average: 10` is a token-bucket average, not a hard concurrency cap, and applies per `requestHost` (the server hostname), not per client.

### Impact Explanation
Complete denial of service of the Rosetta API's database-dependent endpoints (`/block`, `/block/transaction`, `/account`, `/network/status`). All 100 pool connections are held, causing every new request to block indefinitely. Since the Rosetta API is a mirror-node component used by exchanges and indexers for block/transaction data, sustained unavailability directly impacts downstream consumers. The severity matches the stated scope: ≥30% of node-related processing queues indefinitely without any brute-force requirement.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the API port, and only 100 concurrent HTTP connections — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The 20-second hold window means the attacker needs to sustain only ~5 requests/second to keep the pool saturated. The bypass of the per-IP Traefik limit requires only a small botnet or a cloud provider with multiple egress IPs. This is highly repeatable and automatable.

### Recommendation
1. **Application-level concurrency semaphore:** Add a middleware that limits total in-flight DB-bound requests (e.g., a buffered channel semaphore of size `Pool.MaxOpenConnections * 0.8`), returning HTTP 503 immediately when the limit is reached rather than blocking.
2. **Pool wait timeout:** Use `sql.DB.SetConnMaxIdleTime` and configure a `db.WaitTimeout` so callers waiting for a pool connection receive an error after a short deadline (e.g., 2–5 seconds) instead of blocking indefinitely.
3. **Per-IP rate limiting in-process:** Add a middleware (e.g., `golang.org/x/time/rate`) keyed by client IP, independent of Traefik, so protection is enforced regardless of deployment topology.
4. **Reduce default `StatementTimeout`:** Lower the default from 20s to ≤5s to shrink the hold window.
5. **Enforce Traefik middleware as non-optional** in production Helm values, and document that bare deployments without it are insecure.

### Proof of Concept

```bash
# Exhaust the 100-connection pool with 110 concurrent /block requests
# No authentication required; replace HOST:PORT with the Rosetta API address

for i in $(seq 1 110); do
  curl -s -X POST http://HOST:5700/block \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1}}' &
done
wait

# While the above runs, in a second terminal, observe that legitimate requests hang:
time curl -s -X POST http://HOST:5700/block \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":2}}'
# Expected: request hangs for ~20 seconds (StatementTimeout) before receiving a response,
# demonstrating pool exhaustion and indefinite queuing.
