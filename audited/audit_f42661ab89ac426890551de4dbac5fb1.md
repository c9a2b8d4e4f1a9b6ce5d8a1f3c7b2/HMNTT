### Title
Unbounded Concurrent Unthrottled DB Polling via Gap Recovery and Safety Check Mechanism

### Summary
Any unprivileged gRPC client can open an unlimited number of concurrent `subscribeTopic()` sessions. Each session unconditionally triggers a safety-check after 1 second that invokes `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path — which polls the database up to 12 times at 20 ms intervals with a page size of 5000 rows. Because there is no cap on concurrent subscriptions and no rate limiting, an attacker can sustain aggregate DB query rates that exhaust database CPU.

### Finding Description

**Code path 1 — Safety check always fires:**

In `TopicMessageServiceImpl.subscribeTopic()`, a safety check is unconditionally scheduled for every subscription:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [1](#0-0) 

`topicContext.isComplete()` returns `false` whenever `endTime` is `null` (the default for open-ended subscriptions): [2](#0-1) 

So for any subscription without an `endTime`, the safety check **always** fires after 1 second and calls `missingMessages(topicContext, null)`.

**Code path 2 — Safety check calls unthrottled retrieve:**

`missingMessages()` with `current == null` calls:
```java
return topicMessageRetriever.retrieve(gapFilter, false);
``` [3](#0-2) 

**Code path 3 — Unthrottled PollingContext parameters:**

`PollingContext(filter, false)` sets:
- `frequency = 20ms`
- `maxPageSize = 5000`
- `numRepeats = 12` [4](#0-3) 

**Code path 4 — `isComplete()` never terminates early for unthrottled path without a limit:**

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;  // unthrottled: only stops at limit
}
``` [5](#0-4) 

When the safety-check-derived `gapFilter` has no limit (the original subscription had no limit), `limitHit` is always `false`, so the unthrottled retriever always exhausts all 12 polls — even when the DB returns zero rows.

**Code path 5 — No subscription rate limiting:**

`subscriberCount` is a Micrometer gauge only — it is never checked as a gate: [6](#0-5) 

There is no per-IP, per-client, or global cap on concurrent subscriptions anywhere in the service layer.

**Exploit arithmetic:**
- 1 session → 12 queries over ~240 ms → ~50 DB queries/second burst
- 50 concurrent sessions → ~2,500 DB queries/second, each fetching up to 5,000 rows
- Attacker sustains load by continuously opening new sessions (old ones complete after ~1.24 s)

### Impact Explanation
Each unthrottled poll issues a `SELECT` with `LIMIT 5000` against the `topic_message` table. At scale, this saturates DB CPU and I/O, causing query latency to spike for all other mirror-node consumers (REST API, other gRPC subscribers, importer). A sustained attack can render the mirror node unresponsive, preventing clients from confirming transaction finality — a total service outage for the mirror node tier.

### Likelihood Explanation
The attacker requires only:
1. Network access to the gRPC port (publicly exposed in production deployments).
2. A valid `topicId` (trivially discoverable from the REST API, or irrelevant if `checkTopicExists=false`).
3. A gRPC client capable of opening many concurrent streams (standard tooling: `grpcurl`, `ghz`, any gRPC library).

No credentials, keys, or privileged access are needed. The attack is fully repeatable and automatable.

### Recommendation
1. **Enforce a global and per-IP concurrent subscription cap** checked in `subscribeTopic()` before proceeding, returning `RESOURCE_EXHAUSTED` when exceeded.
2. **Fix the unthrottled `isComplete()` logic** to also terminate when the last page returned fewer rows than `maxPageSize` (mirror the throttled branch), preventing wasted polls on empty result sets.
3. **Make the safety check conditional**: only fire it when there is evidence of a gap (e.g., the live listener has received at least one message), not unconditionally for every open-ended subscription.
4. **Apply backpressure / rate limiting at the gRPC server level** (e.g., Netty connection limits, token-bucket per client IP).

### Proof of Concept

```bash
# Open 50 concurrent unlimited subscriptions to any valid topic
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

After ~1 second, 50 safety checks fire simultaneously. Each triggers an unthrottled `PollingTopicMessageRetriever` that issues 12 × `SELECT … LIMIT 5000` queries at 20 ms intervals. Monitor DB CPU with `pg_stat_activity` or equivalent — query rate will spike to ~2,500 queries/second with up to 5,000 rows per query, exhausting DB CPU and causing latency degradation across all mirror node services.
