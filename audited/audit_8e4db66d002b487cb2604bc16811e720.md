### Title
Unbounded DB Row Fetch in `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` Wastes Resources on Every Opcode Replay Request

### Summary
The Spring Data JPA method `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` in `TransactionRepository` carries no `Top1`/`First1` modifier and therefore generates SQL with no `LIMIT` clause, fetching every transaction row matching `(payer_account_id, valid_start_ns)` from the database. The caller in `OpcodeServiceImpl.buildCallServiceParameters` immediately discards all but the first element via `transactionList.getFirst()`, so every extra row represents pure waste: unnecessary DB I/O, JDBC result-set deserialization, and JVM heap allocation of fully-hydrated `Transaction` objects (including `byte[]` fields such as `transactionBytes`, `transactionRecordBytes`, `memo`, `batchKey`, and JSON columns `nftTransfer`/`itemizedTransfer`). An unauthenticated attacker who can reference a transaction ID whose `(payer_account_id, valid_start_ns)` key maps to many rows (e.g., a large atomic-batch parent with many child transactions) can amplify per-request resource consumption arbitrarily within the global rate limit.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/TransactionRepository.java`, lines 14–15:
```java
List<Transaction> findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);
```
No `Top1`/`First1` prefix → Spring Data JPA emits `SELECT … FROM transaction WHERE payer_account_id = ? AND valid_start_ns = ? ORDER BY consensus_timestamp ASC` with **no LIMIT**.

`web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, lines 100–107:
```java
final var transactionList =
        transactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
                payerAccountId, validStartNs);
if (transactionList.isEmpty()) { throw …; }
final var parentTransaction = transactionList.getFirst();   // only row 0 is ever used
```

**Root cause:** The repository interface returns `List<Transaction>` with no pagination hint. Spring Data JPA translates the method name literally; without `Top1`/`First1` or a `Pageable` parameter, the generated query has no `LIMIT`. All N matching rows are fetched, deserialized into `Transaction` POJOs (each carrying multiple `byte[]` and JSON fields), and placed on the JVM heap, then N−1 of them are immediately eligible for GC.

**Why the throttle is insufficient:** `OpcodesController` calls `throttleManager.throttleOpcodeRequest()` before invoking the service. The `opcodeRateLimitBucket` is a **global** token bucket (default capacity = 1 token/second, configurable via `hiero.mirror.web3.throttle.opcodeRequestsPerSecond`). It is not per-IP. A single attacker consumes the entire global budget. More importantly, the throttle limits *request rate*, not *per-request row count*. One request against a transaction ID with N child rows still fetches all N rows regardless of the throttle value.

**How many rows can share a `(payer_account_id, valid_start_ns)` key?** The `Transaction` entity's `innerTransactions` field stores repeated `(payer_account_id, valid_start_ns)` pairs for atomic-batch children. The test `TransactionRepositoryTest.findByPayerAccountIdAndValidStartNsSuccessful` explicitly inserts **10 rows** with the same key and asserts all 10 are returned. In production, a large atomic-batch transaction can have many more children, all sharing the parent's `valid_start_ns`.

### Impact Explanation

Each extra row fetched is a fully-hydrated `Transaction` object containing `byte[]` fields (`transactionBytes`, `transactionRecordBytes`, `batchKey`, `memo`, `transactionHash`) and JSON-deserialized `List<NftTransfer>` / `List<ItemizedTransfer>`. For a batch with N children, the DB must scan and transmit N rows over the JDBC connection, and the JVM must allocate N objects. With the default throttle of 1 req/sec and N=100 children, the node performs 100× the necessary DB work per second on this query path. If the throttle is raised (it is operator-configurable), the multiplier scales linearly. This constitutes increased DB bandwidth and JVM heap pressure that can push overall node resource consumption above the 30% threshold, particularly on nodes with modest baseline load.

### Likelihood Explanation

The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint requires no authentication — any external user can call it. The only preconditions are: (1) the `opcode.tracer.enabled` feature flag is `true`, (2) the `Accept-Encoding: gzip` header is present, and (3) the attacker knows or can enumerate a transaction ID whose `(payer_account_id, valid_start_ns)` maps to many rows. Condition (3) is trivially satisfied on a public network by observing atomic-batch transactions. The attack is repeatable at the configured rate limit and requires no credentials, no on-chain funds, and no special tooling beyond a standard HTTP client.

### Recommendation

Change the repository method to return only the first row by adding the `First` (or `Top1`) keyword:

```java
Optional<Transaction> findFirstByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);
```

Spring Data JPA will then emit `… LIMIT 1`, eliminating the over-fetch entirely. Update the caller in `OpcodeServiceImpl` to use `Optional`:

```java
final var parentTransaction =
    transactionRepository.findFirstByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
            payerAccountId, validStartNs)
        .orElseThrow(() -> new EntityNotFoundException("Transaction not found: " + transactionId));
```

### Proof of Concept

1. On a network with the opcode tracer enabled, identify or submit an atomic-batch transaction whose parent has `payerAccountId = 0.0.X` and `validStartNs = T`, with ≥50 child transactions sharing the same `(payer_account_id, valid_start_ns)`.
2. Send repeated requests (up to the configured `opcodeRequestsPerSecond` rate):
   ```
   GET /api/v1/contracts/results/0.0.X-<T_seconds>-<T_nanos>/opcodes
   Accept-Encoding: gzip
   ```
3. Observe via DB slow-query logs or `pg_stat_activity` that each request executes `SELECT … FROM transaction WHERE payer_account_id = X AND valid_start_ns = T ORDER BY consensus_timestamp ASC` returning all 50+ rows.
4. Monitor node CPU, heap, and DB bandwidth metrics over a 24-hour baseline vs. attack window; the per-request row amplification factor (N vs. 1) will manifest as elevated DB read throughput and JVM GC pressure proportional to N.
