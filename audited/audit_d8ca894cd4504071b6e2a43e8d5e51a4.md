### Title
Unbounded DB Connection Hold During Network Partition via Missing Application-Level Timeout in `GetDbWithContext()`

### Summary
When `statementTimeout <= 0` (the Go zero-value default for the `StatementTimeout int` field), `GetDbWithContext()` creates no `context.WithTimeout` and returns a `noop` cancel function. During a network partition between the Rosetta service and PostgreSQL, every in-flight DB query blocks indefinitely at the TCP layer, holding its connection from the pool. An unprivileged external user who sends a flood of concurrent requests during such a partition can exhaust the connection pool, causing all subsequent `accountRepository` calls to block or fail.

### Finding Description

**Exact code path:**

`rosetta/app/db/client.go`, lines 22–28:
```go
func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
    if d.statementTimeout <= 0 {
        db := d.db
        if ctx != nil {
            db = db.WithContext(ctx)
        }
        return db, noop   // ← no deadline, noop cancel
    }
    ...
    childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
    return d.db.WithContext(childCtx), cancel
}
```

Every `accountRepository` method (`GetAccountAlias`, `GetAccountId`, `getCryptoEntity`, `getLatestBalanceSnapshot`, `getBalanceChange`) calls `ar.dbClient.GetDbWithContext(ctx)` and `defer cancel()`. When `statementTimeout <= 0`, `cancel` is `noop` and the returned `*gorm.DB` carries only the HTTP request context.

**Root cause and failed assumption:**

The `StatementTimeout` field in `config.Db` is a plain `int` with no default value:
```go
// rosetta/app/config/types.go line 45
StatementTimeout int `yaml:"statementTimeout"`
```
Its Go zero value is `0`, so unless the operator explicitly sets it, `statementTimeout <= 0` is always true and the timeout branch is never reached.

The HTTP server's `Http` struct also has no default timeouts:
```go
// rosetta/app/config/types.go lines 64-69
type Http struct {
    IdleTimeout       time.Duration
    ReadTimeout       time.Duration
    ReadHeaderTimeout time.Duration
    WriteTimeout      time.Duration
}
```
With `WriteTimeout == 0`, Go's `net/http` never closes the server-side connection on its own, so the HTTP request context (`r.Context()`) is never cancelled by the server. The only way the context is cancelled is if the client disconnects — which an attacker deliberately avoids.

**During a network partition**, the `lib/pq` or `pgx` driver's TCP `connect()` / `send()` call blocks at the OS level. Default Linux TCP keepalive fires after ~2 hours. The goroutine and its leased DB connection are held for the entire duration.

**Why the DB-level `statement_timeout` does not help:**
The Helm chart sets `ALTER USER rosetta SET statement_timeout TO '10000'` at the PostgreSQL level. This timeout only fires once the query reaches the PostgreSQL backend and begins executing. During a network partition the TCP handshake or data send never completes; the query never reaches the backend, so PostgreSQL's `statement_timeout` is never triggered.

### Impact Explanation

Each concurrent request that hits a DB-touching `accountRepository` method consumes one connection from the pool (`SetMaxOpenConns` in `rosetta/app/db/db.go` line 33). PostgreSQL's default `max_connections` is 100. Once all connections are held by hung goroutines, every new request that needs a DB connection blocks indefinitely (or until the pool's `connectionTimeout`, which is also not set by default). This is a complete denial of service for all Rosetta API endpoints that touch account data, with no self-recovery until the network partition resolves or the process is restarted.

Severity: **High** — full service unavailability for the Rosetta API, no authentication required, no special privileges needed.

### Likelihood Explanation

The Rosetta API is a public-facing HTTP service. Any unauthenticated client can send requests. The attacker's required capability is:
1. Send ~100 concurrent HTTP requests to any account-querying endpoint (e.g., `/account/balance`, `/block`) and keep the TCP connections open (trivial with `curl --no-buffer`, `wrk`, or any async HTTP client).
2. The network partition can be a naturally occurring infrastructure event, a BGP-level disruption, or a firewall rule change — the attacker does not need to cause it themselves; they only need to exploit it opportunistically.

The default-zero `statementTimeout` means this condition exists in every deployment that does not explicitly set the value. The attack is repeatable and requires no credentials.

### Recommendation

1. **Set a non-zero default for `StatementTimeout`** in `rosetta/app/config/types.go`:
   ```go
   StatementTimeout int `yaml:"statementTimeout"` // default: 10 (seconds)
   ```
   and initialize it in the config loader or struct literal to `10`.

2. **Set HTTP server `WriteTimeout`** to a value (e.g., 30s) so that the request context is cancelled if the server cannot write a response in time, which propagates cancellation to the in-flight DB query.

3. **Set a pool `connectionTimeout`** via `sql.DB` so that acquiring a connection from an exhausted pool fails fast rather than blocking indefinitely.

4. **Set `ConnectTimeout` in the DSN** (`rosetta/app/config/types.go`, `GetDsn()`) so that the TCP connect phase itself is bounded:
   ```go
   "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable connect_timeout=5"
   ```
