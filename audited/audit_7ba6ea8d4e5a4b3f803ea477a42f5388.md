### Title
`incrementByteArray()` Overflow on MAX_KEY Wraps `lowerBound` to Zero, Causing Full Key-Space Scan Instead of Empty Result

### Summary
When an unauthenticated user supplies `key=gt:0xffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff` to the hook storage endpoint, `SlotRangeParameter.getInclusiveValue()` calls `incrementByteArray()` on a 32-byte all-`0xFF` value. Apache Tuweni's `MutableBytes32.increment()` wraps around to `0x00...00` (all zeros). The downstream guard in `hookStorageChangeRequest()` uses a strict `> 0` comparison against `MIN_KEY_BYTES` (also `0x00...00`), which evaluates to false, so the wrapped value is silently discarded and `lowerBound` stays at `MIN_KEY_BYTES`. The resulting query scans the entire key space (`key >= 0x00...00 AND key <= 0xFF...FF`) instead of returning an empty result.

### Finding Description

**Step 1 — Input parsing in `SlotRangeParameter.valueOf()`** [1](#0-0) 

Input `"gt:0xffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff"` matches the pattern; `operator = GT`, `hex` is already 64 `f` chars. The call becomes:
```
new SlotRangeParameter(GT.toInclusive()=GTE, getInclusiveValue(GT, "ffff...ffff"))
```

**Step 2 — Overflow in `getInclusiveValue()`** [2](#0-1) 

`bytes` = 32 × `0xFF`. Because `operator == GT`, `incrementByteArray(bytes)` is called: [3](#0-2) 

`MutableBytes32.wrap(bytes).increment()` performs unsigned 256-bit increment with wrap-around (standard Tuweni/EVM behavior). `0xFF...FF + 1` wraps to `0x00...00`. The returned `SlotRangeParameter` is `(GTE, 0x00...00)`.

**Step 3 — Guard check fails in `hookStorageChangeRequest()`** [4](#0-3) 

`MIN_KEY_BYTES` is a 32-byte all-`0x00` array. [5](#0-4) 

The guard is:
```java
} else if (Arrays.compareUnsigned(value, lowerBound) > 0) {
    lowerBound = value;
}
```
`value` = `0x00...00`, `lowerBound` = `MIN_KEY_BYTES` = `0x00...00`.  
`Arrays.compareUnsigned(0x00...00, 0x00...00)` = `0`, which is **not** `> 0`.  
The wrapped value is silently dropped. `lowerBound` stays `0x00...00`.

**Step 4 — Full key-space query issued** [6](#0-5) 

The repository receives `keyLowerBound = 0x00...00` and `keyUpperBound = 0xFF...FF`, executing:
```sql
WHERE key >= '\x0000...00' AND key <= '\xffff...ff'
```
— a full key-space scan for that `(owner_id, hook_id)` pair.

### Impact Explanation

**Information disclosure**: The attacker receives all storage entries for the targeted hook instead of an empty response. The semantic contract of `gt:MAX_KEY` (return nothing) is violated.

**Denial of service**: The database must execute a `DISTINCT ON (key)` scan over every row for the targeted hook. An attacker can flood the endpoint with this single-parameter request, forcing repeated full-table scans with no rate limiting visible in the controller, degrading database performance for all users.

Severity: **High** — unauthenticated, trivially repeatable, causes both data leakage and resource exhaustion.

### Likelihood Explanation

The endpoint is a public, unauthenticated REST GET endpoint. No special privileges, tokens, or knowledge of internal state are required. The exploit is a single HTTP request with a fixed, known payload. It is trivially repeatable and scriptable for sustained DoS. Any external user who discovers the API can trigger it.

### Recommendation

1. **Detect overflow before use**: In `getInclusiveValue()`, after calling `incrementByteArray()`, check whether the result is all-zero (i.e., wrapped). If so, throw `IllegalArgumentException("Key value out of range for gt operator")` — the same way the numeric parameters reject out-of-range values.

```java
private static byte[] getInclusiveValue(RangeOperator operator, String hexValue) throws DecoderException {
    byte[] bytes = Hex.decodeHex(hexValue);
    if (operator == RangeOperator.GT) {
        byte[] incremented = incrementByteArray(bytes);
        // Detect wrap-around: all-0xFF incremented to all-0x00
        if (Arrays.equals(incremented, new byte[incremented.length])) {
            throw new IllegalArgumentException("Key value 0x" + hexValue + " has no successor; gt operator is unsatisfiable");
        }
        return incremented;
    } else if (operator == RangeOperator.LT) {
        byte[] decremented = decrementByteArray(bytes);
        // Detect wrap-around: all-0x00 decremented to all-0xFF
        if (isAllFF(decremented)) {
            throw new IllegalArgumentException("Key value 0x" + hexValue + " has no predecessor; lt operator is unsatisfiable");
        }
        return decremented;
    }
    return bytes;
}
```

2. **Apply the same fix symmetrically** for `lt:0x00...00` (decrement wraps to `0xFF...FF`), which has the analogous problem on the upper-bound path.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1001/hooks/1/storage?key=gt:0xffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff
```

**Expected**: HTTP 200 with empty `storage` array (no key can be greater than MAX_KEY).

**Actual**: HTTP 200 returning all storage entries for hook `1` owned by account `0.0.1001`, with the database executing a full key-space scan (`key >= 0x00...00 AND key <= 0xFF...FF`).

Repeat the request in a tight loop to cause sustained full-table scans and degrade database performance (DoS).
