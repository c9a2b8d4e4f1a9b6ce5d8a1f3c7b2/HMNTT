### Title
Persistent Double DB Query Triggered by Attacker-Controlled Successful SCHEDULECREATE Transaction

### Summary
`mayMissLongTermScheduledTransaction()` in `rest/transactions.js` returns `true` indefinitely for any successful SCHEDULECREATE (type=42) transaction that has no corresponding scheduled execution in the result set, with no time-based expiry check. An attacker who submits one such transaction to the network can then repeatedly query the mirror node REST API, causing every request to execute two `pool.queryQuietly()` calls — the second with a significantly expanded timestamp range — doubling (or worse) the DB load per request, permanently and at will.

### Finding Description

**Root cause:** `mayMissLongTermScheduledTransaction()` contains no time-based guard. It returns `true` for any successful SCHEDULECREATE that has no `scheduled=true` sibling in the result set, regardless of how old the transaction is or whether the schedule has already expired/been deleted. The only time-based check in the file is inside `getTransactionsByIdOrHashCacheControlHeader()` (line 906–908), which is a separate function controlling cache headers — it has no effect on whether the second DB query fires. [3](#0-2) 

**Exploit flow:**
1. Attacker creates a Hedera account and submits a SCHEDULECREATE transaction crafted to never execute (e.g., schedule expires before the required signatures are gathered, or the attacker immediately deletes the schedule after creation).
2. The SCHEDULECREATE lands on-chain with `type=42`, `result=22` (SUCCESS), and no corresponding `scheduled=true` execution row.
3. Attacker queries `GET /api/v1/transactions/{payerAccount}-{validStart}` (no `?scheduled=` param) in a loop.
4. Each request: first `pool.queryQuietly()` returns the SCHEDULECREATE row → `mayMissLongTermScheduledTransaction()` evaluates `scheduleExists=true`, no `scheduled` row found → returns `true` → second `pool.queryQuietly()` fires with `upperConsensusTimestamp = lowerConsensusTimestamp + maxScheduledTransactionConsensusTimestampRangeNs`.

**Why existing checks fail:**
- `isTransactionHash` guard (line 976): bypassed — attacker uses a transaction ID, not a hash.
- `scheduled === false` guard (line 976): bypassed — attacker omits the `?scheduled=` query parameter, leaving `scheduled` as `undefined`.
- `transaction.scheduled` loop check (line 985): bypassed — the scheduled execution never occurred, so no `scheduled=true` row exists.
- No rate-limiting or per-transaction-ID deduplication is applied before the second query fires. [4](#0-3) 

### Impact Explanation
Every HTTP request to the affected endpoint for the attacker's transaction ID issues two database queries instead of one. The second query uses `maxScheduledTransactionConsensusTimestampRangeNs` as its upper bound — a value intended to cover long-term scheduled transactions, likely orders of magnitude larger than `maxTransactionConsensusTimestampRangeNs` — making it a heavier scan. With multiple crafted transactions and a high request rate, the attacker can multiply DB load, degrading mirror node availability for all users. The condition persists indefinitely (not just while the schedule is "pending"), so a single on-chain transaction enables unlimited future amplification.

### Likelihood Explanation
The precondition is minimal: one Hedera account and one successful SCHEDULECREATE transaction fee (fractions of a cent on mainnet, free on testnet). No privileged access, no insider knowledge, and no ongoing on-chain cost is required after the initial setup. The attack is fully repeatable and scriptable. Any actor motivated to degrade mirror node availability can execute this.

### Recommendation
Add a time-based expiry check inside `mayMissLongTermScheduledTransaction()` mirroring the one already present in `getTransactionsByIdOrHashCacheControlHeader()`. Specifically, pass the `consensus_timestamp` of the SCHEDULECREATE transaction and compare it against `nowInNs() - maxScheduledTransactionConsensusTimestampRangeNs`; if the schedule is older than the maximum possible execution window, return `false` immediately. Additionally, consider caching the result of the first query per transaction ID for a short TTL so that repeated requests for the same ID do not re-issue the second query on every hit.

### Proof of Concept
1. On Hedera testnet, create account A.
2. Submit a SCHEDULECREATE from account A that schedules a transfer requiring a second signature that will never be provided (so the schedule expires without executing).
3. Record the transaction ID: `0.0.A-{validStartSec}-{validStartNanos}`.
4. Confirm the transaction is indexed in the mirror node with `type=42`, `result=22`, and no `scheduled=true` sibling row.
5. In a loop, send:
   ```
   GET /api/v1/transactions/0.0.A-{validStartSec}-{validStartNanos}
   ```
   (no `?scheduled=` parameter)
6. Observe in mirror node DB metrics (e.g., `pg_stat_activity`, slow query logs) that each HTTP request generates two DB queries — the second with a timestamp upper bound of `lowerConsensusTimestamp + maxScheduledTransactionConsensusTimestampRangeNs`.
7. Scale the loop to confirm proportional DB load increase with no additional on-chain cost.

const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
