### Title
Unauthenticated Thread Pool Exhaustion via Unbounded Concurrent Requests to `GET /api/v1/network/stake`

### Summary
The `GET /api/v1/network/stake` endpoint in the `rest-java` module has no rate limiting, no response caching, and no authentication requirement. Every request synchronously executes `NetworkStakeRepository.findLatest()`, which issues a native SQL query against the database, blocking a Tomcat worker thread for the full duration. An unprivileged attacker sending a large number of concurrent requests can exhaust the server's HTTP thread pool and database connection pool, causing denial of service for all other endpoints.

- `NetworkStakeRepository.findLatest()` â€” `rest-java/src/main/java/org/hiero/mirror/restjava/repository/NetworkStakeRepository.java`, lines 12â€“19: a `nativeQuery = true` correlated subquery (`SELECT * FROM network_stake WHERE consensus_timestamp = (SELECT MAX(consensus_timestamp) FROM network_stake)`) executed on every invocation.

**Root cause â€” failed assumptions:**

1. **No rate limiting in `rest-java`**: The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` infrastructure exists exclusively in the `web3` module (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`). There is no equivalent in `rest-java`. A search for `@Cacheable`, rate-limit filters, or bucket4j usage in `rest-java` returns zero results.
2. **No caching**: `NetworkServiceImpl` carries no `@Cacheable` annotation and no Spring Cache configuration for `getLatestNetworkStake()`. Every HTTP request triggers a live database round-trip.
3. **Synchronous, blocking execution**: Spring Boot's default Tomcat NIO connector assigns one worker thread per active request. The thread is held until the response is fully written. Under concurrent load, threads accumulate waiting for DB responses.

**Exploit flow:**

An attacker opens hundreds of concurrent TCP connections and issues `GET /api/v1/network/stake` on each. Each connection causes Tomcat to allocate a worker thread that blocks on `findLatest()`. Spring Boot's default Tomcat thread pool is 200 threads (`server.tomcat.threads.max = 200`). Once all 200 threads are occupied, incoming requests queue in the accept backlog and eventually time out. Because the DB connection pool is also finite, concurrent queries additionally exhaust JDBC connections, causing further latency or errors for all other endpoints sharing the same pool.

**Why existing checks are insufficient:**

- The `web3` throttle (`ThrottleManagerImpl.throttle()`) is wired only to `ContractCallRequest` objects and is not present in the `rest-java` application context.
- No Kubernetes-level ingress rate limiting is configured in the `hedera-mirror-rest-java` Helm chart (`charts/hedera-mirror-rest-java/values.yaml`) for this specific endpoint.
- No `server.tomcat.threads.max`, `server.tomcat.accept-count`, or `spring.datasource.hikari.maximum-pool-size` overrides are present in any `rest-java` configuration file.

### Impact Explanation
A successful attack renders the entire `rest-java` service unresponsive â€” not just the `/network/stake` endpoint â€” because all Tomcat worker threads are consumed. All other endpoints (`/network/nodes`, `/network/fees`, `/network/supply`, etc.) stop responding. The impact is a full availability outage of the Java REST API tier with no economic damage to the Hedera network itself, consistent with the "griefing" severity classification.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no protocol-level access. A single attacker with a modest number of concurrent HTTP connections (e.g., 200â€“500) from a single host is sufficient. The endpoint is publicly documented in `rest/api/v1/openapi.yml` (line 990). The attack is trivially repeatable and scriptable with standard tools (`ab`, `wrk`, `hey`, `curl --parallel`).

### Recommendation
1. **Add response caching**: Annotate `NetworkServiceImpl.getLatestNetworkStake()` with `@Cacheable` (e.g., `expireAfterWrite=60s`). Network stake data changes at most once per day (staking period), so aggressive caching is safe and eliminates per-request DB load entirely.
2. **Add rate limiting to `rest-java`**: Port the bucket4j-based `ThrottleConfiguration` pattern from `web3` into `rest-java`, or add a servlet filter / Spring `HandlerInterceptor` that enforces a per-IP or global RPS cap on this endpoint.
3. **Configure a DB statement timeout**: Set `spring.datasource.hikari.connection-timeout` and a PostgreSQL `statement_timeout` for the `rest-java` DB user to bound the maximum time a thread can be held waiting for a query.
4. **Configure Tomcat connection limits**: Set `server.tomcat.accept-count` and `server.tomcat.max-connections` to prevent unbounded queuing.

### Proof of Concept
```bash
# Exhaust the thread pool with 250 concurrent requests, repeated in a loop
while true; do
  for i in $(seq 1 250); do
    curl -s -o /dev/null "https://<mirror-node-host>/api/v1/network/stake" &
  done
  wait
done
```
After a few seconds, requests to any other `rest-java` endpoint (e.g., `/api/v1/network/nodes`) will begin timing out or returning 503, confirming thread pool exhaustion.
