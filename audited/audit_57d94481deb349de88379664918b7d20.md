### Title
`parseInt` Truncation of Scientific-Notation `Content-Length` Silently Corrupts `requestSizeHistogram` Metrics

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function reads the raw `content-length` header string and passes it to `parseInt(..., 10)` before recording it in `requestSizeHistogram`. Because `parseInt` stops parsing at the first non-digit character, a value like `'1.5e10'` is parsed as `1` rather than `15000000000`. The existing `|| 0` guard only catches `NaN` (fully non-numeric strings), so the corrupted value `1` passes through silently and is recorded as the request size.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback within `metricsMiddleware`.

```js
// line 209
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** `parseInt('1.5e10', 10)` returns `1` — it parses the leading `'1'` and stops at `'.'` because `'.'` is not a valid base-10 digit. The result `1` is truthy, so the `|| 0` fallback is never triggered. The histogram records `1` byte for a request that claimed a body of 15 GB.

**Exploit flow:**
1. Attacker crafts an HTTP request with `Content-Length: 1.5e10` (or any `<digit>.<rest>` pattern).
2. Node.js/Express passes the raw header string through to `req.headers['content-length']`.
3. `parseInt('1.5e10', 10)` → `1`; `1 || 0` → `1`.
4. `requestSizeHistogram.record(1, labels)` is called — a near-zero value is recorded.
5. Prometheus/OpenTelemetry metrics now show a 1-byte request instead of the claimed multi-gigabyte one.

**Why existing checks fail:** The `|| 0` guard is designed to handle `NaN` (e.g., `parseInt('abc', 10)` → `NaN` → `0`). It does not handle the case where `parseInt` returns a small but truthy integer due to early truncation at a non-digit character like `'.'` or `'e'`. [1](#0-0) 

### Impact Explanation
The impact is confined to observability/monitoring: `api_request_size_bytes` histogram data is silently corrupted. Operators relying on this metric for capacity planning, anomaly detection, or SLA monitoring will see misleading data. No user funds, authentication, or business logic are affected. Severity is low — pure griefing/metric poisoning with no economic damage.

### Likelihood Explanation
Any unauthenticated external user can send an HTTP request with an arbitrary `Content-Length` header value. No special privileges, credentials, or network position are required. The attack is trivially repeatable with a single `curl` command. Node.js's HTTP parser (llhttp) does not strictly reject non-integer `Content-Length` values in all versions, so the header reaches Express middleware intact.

### Recommendation
Replace `parseInt` with `Number()` and add an explicit integer/range check, or use a stricter parse:

```js
const rawLen = req.headers['content-length'];
const contentLength = rawLen !== undefined ? Math.max(0, Math.trunc(Number(rawLen))) : 0;
requestSizeHistogram.record(Number.isFinite(contentLength) ? contentLength : 0, labels);
```

`Number('1.5e10')` correctly yields `15000000000`; `Math.trunc` drops any fractional part; `Number.isFinite` rejects `NaN`/`Infinity`. Alternatively, validate that the header matches `/^\d+$/` before parsing.

### Proof of Concept
```bash
# Send a request with a scientific-notation Content-Length
curl -s -X POST http://<mirror-rest-host>/api/v1/transactions \
  -H "Content-Length: 1.5e10" \
  -H "Content-Type: application/json" \
  -d '{}'

# Then scrape metrics and observe:
curl http://<mirror-rest-host>/api/v1/metrics/ | grep api_request_size_bytes
# Expected (honest): bucket counts reflect ~15 GB
# Actual (corrupted): bucket counts reflect 1 byte
```

`parseInt('1.5e10', 10)` returning `1` can be verified directly in any Node.js REPL:
```js
> parseInt('1.5e10', 10)
1          // stops at '.'
> parseInt('1.5e10', 10) || 0
1          // || 0 does NOT fire — 1 is truthy
```
