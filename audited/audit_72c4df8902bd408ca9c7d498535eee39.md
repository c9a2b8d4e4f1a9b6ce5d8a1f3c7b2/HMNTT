### Title
Unauthenticated, Unthrottled `getNodes()` gRPC Endpoint Enables Database Connection Pool Exhaustion DoS

### Summary
`NetworkController.getNodes()` accepts `AddressBookQuery` requests from any unauthenticated caller with no rate limiting at the gRPC layer. Each request drives multiple unbounded database queries in `NetworkServiceImpl.getNodes()` and repeated `transactionOperations.execute()` calls in `page()`, each of which acquires a HikariCP connection. A flood of concurrent requests from many connections can exhaust the finite connection pool, starving legitimate smart contract clients of address book data.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any `AddressBookQuery` with no authentication check, no IP-based throttle, and no per-client rate limit. The only per-connection limit is `maxConcurrentCallsPerConnection = 5` (default), which limits calls *per connection* but places no cap on the number of connections an attacker may open. [1](#0-0) 

The call immediately delegates to `NetworkServiceImpl.getNodes()`, which performs three uncached database queries before the paging loop begins:
- `addressBookRepository.findLatestTimestamp()` — no `@Cacheable`
- `nodeStakeRepository.findLatestTimestamp()` — no `@Cacheable`
- `nodeStakeRepository.findAllStakeByConsensusTimestamp()` — no `@Cacheable`

Each of these runs on the gRPC executor thread and acquires a HikariCP connection independently. [2](#0-1) 

The paging loop then calls `page()` repeatedly. Each invocation of `page()` calls `transactionOperations.execute()`, which acquires a HikariCP connection for the duration of the transaction, even when the inner `findByConsensusTimestampAndNodeId` result is served from the Caffeine cache (the transaction is opened before the cache proxy is consulted). [3](#0-2) 

The `withScheduler(Schedulers.boundedElastic())` applies only to the *delay* between repeat pages, not to the first `page()` call, which runs on the gRPC executor thread. [4](#0-3) 

**Root cause:** No rate limiting, authentication, or connection-count cap exists at the gRPC controller layer for `getNodes()`. The throttling infrastructure (Bucket4j `ThrottleManagerImpl`) is present only in the `web3` module and is never applied to the gRPC `NetworkService`. [5](#0-4) 

The gRPC server is configured with `maxConcurrentCallsPerConnection` (default 5), which is per-connection only and does not bound total concurrent calls across many connections. [6](#0-5) 

The HikariCP pool is configured via `spring.datasource.hikari` with no explicit `maximumPoolSize` override found in the gRPC application config, defaulting to HikariCP's default of 10 connections. [7](#0-6) 

The existing Prometheus alert fires only after 75% pool utilization is sustained for 5 minutes — well after a DoS is already in effect. [8](#0-7) 

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` is per-connection; an attacker opens N connections for N×5 concurrent calls.
- `Schedulers.boundedElastic()` bounds only the inter-page delay scheduling, not the initial three DB queries or the first `page()` call per request.
- `@Cacheable` on `findByConsensusTimestampAndNodeId` reduces SQL execution time but does not prevent the transaction (and thus the HikariCP connection acquisition) from occurring.
- The optional GCP backend policy `maxRatePerEndpoint: 250` is infrastructure-level and not enforced in all deployments. [9](#0-8) 

### Impact Explanation
When the HikariCP pool is exhausted, all gRPC calls requiring a DB connection — including `getNodes()` for legitimate callers — queue and eventually time out with `SQLTransientConnectionException`. Smart contract clients (e.g., Hedera SDK instances bootstrapping their node list) that rely on `getNodes()` to discover network topology will fail to obtain address book data, breaking their ability to submit transactions. This is a Layer 1 network availability impact: the mirror node's address book service becomes unavailable, indirectly disrupting smart contract client connectivity to the Hedera network. Severity is Medium consistent with the scope classification (no direct fund loss, but unintended smart contract behavior via broken node discovery).

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the publicly documented gRPC endpoint (`grpcurl -plaintext -d '{"file_id": {"fileNum": 102}, "limit": 0}' host:5600 com.hedera.mirror.api.proto.NetworkService/getNodes`), and no on-chain funds. A single attacker with a modest number of TCP connections (e.g., 20 connections × 5 concurrent calls = 100 concurrent requests) is sufficient to saturate a default 10-connection HikariCP pool. The attack is trivially repeatable and scriptable. Deployments without the GCP gateway rate limit are fully exposed. [10](#0-9) 

### Recommendation
1. **Add a server-side rate limiter interceptor** in the gRPC module (analogous to `ThrottleManagerImpl` in `web3`) as a `@GlobalServerInterceptor` that enforces a per-IP or global requests-per-second cap on `NetworkService/getNodes`.
2. **Increase HikariCP pool size** and set an explicit `maximumPoolSize` in the gRPC application configuration to match expected concurrency.
3. **Cache the three pre-loop queries** (`findLatestTimestamp`, `findAllStakeByConsensusTimestamp`) with a short TTL (e.g., matching `cacheExpiry = 2s`) so repeated concurrent requests do not each acquire a DB connection for these lookups.
4. **Enforce the GCP `maxRatePerEndpoint`** in all production deployments and add a non-GCP fallback (e.g., Netty connection limit via `maxConnectionAge` or a token-bucket interceptor).

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
    }

    @Bean

    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
