### Title
Unauthenticated Three-Way FULL OUTER JOIN DoS via `account.id`-Only Filter in `/api/v1/transactions`

### Summary
Any unauthenticated external user can send repeated requests to `/api/v1/transactions?account.id=<id>` (no `credit_type`, no timestamp bounds) to force `getTransactionTimestampsQuery()` into its most expensive execution path: a three-way FULL OUTER JOIN across `transaction`, `crypto_transfer`, and `token_transfer`. Because the REST API has no per-IP rate limiting and `bindTimestampRange` defaults to `false`, each request issues unbounded sub-queries against all three tables. With only 10 DB connections and a 20-second statement timeout, a small number of concurrent attackers can saturate the connection pool and degrade service for all legitimate users.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactionTimestampsQuery()` at lines 555–604:

```js
if (creditDebitQuery || accountQuery) {
  // ...build cryptoTransferQuery and tokenTransferQuery...
  if (creditDebitQuery) {
    // two-way FULL OUTER JOIN (ctl + ttl only)
    return `...full outer join...`;
  }
  // account-only path: THREE-WAY FULL OUTER JOIN
  return `
    select coalesce(t.consensus_timestamp, ctl.consensus_timestamp, ttl.consensus_timestamp) ...
    from (${transactionOnlyQuery}) as t
      full outer join (${cryptoTransferQuery}) as ctl on ...
      full outer join (${tokenTransferQuery}) as ttl on ...
    order by consensus_timestamp ${order} ${limitQuery}`;
}
``` [1](#0-0) 

**Root cause — three compounding failures:**

1. **No timestamp bounding by default.** `hiero.mirror.rest.query.bindTimestampRange` defaults to `false`. [2](#0-1) 
When `false`, `bindTimestampRange()` returns the range unchanged with no enforcement. [3](#0-2) 

2. **Timestamp range validation is disabled for this endpoint.** `getTransactions()` calls `parseTimestampFilters` with `validateRange=false` (6th argument) and `allowOpenRange=true` (4th argument), so no timestamp filter at all is required and no `maxTimestampRange` check is applied. [4](#0-3) 

3. **No rate limiting on the REST API.** The throttle/rate-limit infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) applies only to the web3 contract-call module, not to the REST `/api/v1/transactions` endpoint. The `authHandler` middleware only sets a custom response limit for authenticated users; unauthenticated requests proceed freely. [5](#0-4) [6](#0-5) 

**Exploit flow:**

With `account.id` set and `creditDebitQuery` empty, the branch at line 555 is entered but the inner `if (creditDebitQuery)` at line 580 is false, so execution falls through to the three-way FULL OUTER JOIN at lines 595–604. Each of the three sub-queries (`transactionOnlyQuery`, `cryptoTransferQuery`, `tokenTransferQuery`) scans its respective table filtered only by account ID with no timestamp constraint, ordered and limited. The outer FULL OUTER JOIN then merges and re-sorts up to 3×limit rows. For high-volume accounts (e.g., exchange accounts, system accounts), each sub-query must traverse a large portion of the account's index entries before the `LIMIT` can be applied.

### Impact Explanation

The DB connection pool is capped at 10 connections (`hiero.mirror.rest.db.pool.maxConnections: 10`) with a 20-second statement timeout (`statementTimeout: 20000`). [7](#0-6) 

An attacker sending 10 concurrent requests targeting a high-volume `account.id` (e.g., a known exchange account with millions of transfers) can hold all 10 DB connections busy for up to 20 seconds each. By immediately re-issuing requests after each timeout, the attacker maintains continuous saturation. Legitimate users receive connection-pool exhaustion errors or severely degraded response times. No economic cost is required — only HTTP requests. This is a realistic, repeatable griefing attack causing service degradation for all users of the mirror node REST API.

### Likelihood Explanation

- **Precondition:** None. No authentication, no API key, no on-chain action required.
- **Knowledge required:** Public knowledge that `/api/v1/transactions?account.id=` is a valid endpoint. High-volume account IDs are publicly discoverable via the same API.
- **Repeatability:** Fully automated with a simple HTTP client loop. A single attacker with modest bandwidth can sustain the attack indefinitely.
- **Detection difficulty:** Requests are syntactically valid and indistinguishable from legitimate queries at the HTTP layer.

### Recommendation

1. **Enable `bindTimestampRange: true` by default** or enforce it unconditionally for the `account.id` filter path, bounding queries to `maxTransactionsTimestampRangeNs` (default 60 days).
2. **Add per-IP or global rate limiting** to the REST API (e.g., via an Express rate-limit middleware or an upstream API gateway), separate from the web3 throttle.
3. **Require at least one timestamp bound** when `account.id` is provided and no `credit_type` is given, to prevent unbounded full-history scans.
4. **Increase `maxConnections`** or add a query queue with a shorter per-request timeout for the account-filter path.

### Proof of Concept

```bash
# Identify a high-volume account (e.g., a known exchange)
ACCOUNT="0.0.98"  # treasury/fee account with millions of transfers

# Saturate the 10-connection pool with concurrent unbounded three-way FULL OUTER JOIN queries
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/transactions?account.id=${ACCOUNT}&limit=100" &
done
wait

# Repeat in a loop to maintain saturation
while true; do
  for i in $(seq 1 10); do
    curl -s "https://<mirror-node>/api/v1/transactions?account.id=${ACCOUNT}&limit=100" &
  done
  wait
done
```

Legitimate requests to any REST endpoint will time out or receive errors while the pool is saturated. No credentials, tokens, or on-chain transactions are required.
