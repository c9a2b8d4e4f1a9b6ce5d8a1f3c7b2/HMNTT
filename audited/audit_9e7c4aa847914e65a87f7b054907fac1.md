### Title
Default Gas of 15,000,000 Enables Maximum Gas-Throttle Bucket Drain via Minimal-Payload Requests

### Summary
`ContractCallRequest` defaults `gas` to `15_000_000` (the configured maximum). `ThrottleManagerImpl.throttle()` consumes gas tokens from `gasLimitBucket` **before** EVM execution, and restoration only occurs **after** execution completes. An unprivileged attacker sending concurrent requests without a `gas` field each consume the maximum scaled token count (1,500 tokens), allowing 500 simultaneous requests to drain the entire 750,000-token bucket, causing legitimate users to receive `Gas per second rate limit exceeded` errors during the execution window.

### Finding Description

**Code path and root cause:**

`ContractCallRequest.java` line 37 sets the default:
```java
@Min(21_000)
private long gas = 15_000_000L;
```

`ContractController.java` lines 40–41 consume throttle tokens **before** validation or execution:
```java
throttleManager.throttle(request);          // tokens consumed here
validateContractMaxGasLimit(request);       // only throws if gas > 15M (not >=)
```

`ThrottleManagerImpl.java` line 40 consumes scaled gas upfront:
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
```

`ThrottleProperties.java` lines 42–47 scale gas by dividing by 10,000:
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);  // 15_000_000 / 10_000 = 1_500 tokens
}
```

`ThrottleConfiguration.java` lines 35–44 set bucket capacity:
```java
long gasLimit = throttleProperties.getGasPerSecond();  // scaleGas(7_500_000_000) = 750_000 tokens
```

Restoration only occurs in `ContractCallService.java` lines 127–135 inside the `finally` block of `doProcessCall`, **after** EVM execution completes:
```java
finally {
    if (!estimate) {
        restoreGasToBucket(result, params.getGas());
    }
}
```

**Exploit flow:**

1. Attacker sends 500 concurrent POST `/api/v1/contracts/call` requests with body `{"to":"0x..."}` — no `gas` field, so `gas` defaults to `15_000_000`.
2. Each request passes `rateLimitBucket.tryConsume(1)` (capacity = 500).
3. Each request calls `gasLimitBucket.tryConsume(1_500)`. After 500 requests: `500 × 1_500 = 750_000` tokens consumed = bucket fully drained.
4. During EVM execution of those 500 requests (which takes non-zero wall-clock time), any new request from a legitimate user hits `gasLimitBucket.tryConsume(...)` → returns `false` → throws `ThrottleException("Gas per second rate limit exceeded.")`.
5. After execution completes, `restoreGasToBucket` restores `min(gasLimit − gasUsed, gasLimitToRestoreBaseline)`. For a minimal call using 21,000 gas: restores `scaleGas(14_979_000)` = 1,497 tokens. Net permanent drain per request = 3 tokens — but the **temporary** drain during execution is the attack window.

**Why existing checks are insufficient:**

- `validateContractMaxGasLimit` (line 93) checks `gas > maxGasLimit` (strict greater-than). Since default gas equals `maxGasLimit` (both 15,000,000), this check never triggers and no restore occurs via the `InvalidParametersException` path.
- The `rateLimitBucket` (500 RPS) limits the attacker to 500 requests/second — but 500 requests at default gas is exactly enough to drain the 750,000-token gas bucket (500 × 1,500 = 750,000).
- The `gasLimitRefundPercent = 100` restore mitigates **permanent** drain but does not prevent the **transient** drain window during concurrent execution.

### Impact Explanation

During the execution window of 500 concurrent attacker requests, the `gasLimitBucket` is fully exhausted. Any legitimate user request arriving in that window receives HTTP 429 with `Gas per second rate limit exceeded`. The impact is a transient denial-of-service on the gas throttle path. No funds are at risk and no on-chain state is modified, consistent with the "griefing with no economic damage" classification. The amplification factor is significant: a request with default gas consumes 1,500 tokens vs. 2 tokens for a minimum-gas (21,000) request — a 750× difference in throttle impact per request.

### Likelihood Explanation

No authentication, API key, or privileged access is required. The endpoint is publicly reachable. The attack requires only an HTTP client capable of sending 500 concurrent POST requests with a minimal JSON body. Many standard HTTP clients and load-testing tools (e.g., `wrk`, `ab`, `curl` with parallel jobs) can trivially achieve this. The attack is repeatable every second as the bucket refills, making sustained griefing feasible with minimal infrastructure.

### Recommendation

1. **Do not default `gas` to the maximum.** Change the default in `ContractCallRequest` to a lower value (e.g., `21_000`) or require the field explicitly. If a high default is needed for compatibility, document it clearly and consider a separate lower default for unauthenticated callers.
2. **Validate gas before throttling.** Move `validateContractMaxGasLimit` before `throttleManager.throttle(request)` so that out-of-range gas values are rejected without consuming any bucket tokens.
3. **Throttle based on actual gas used, not gas limit.** Consume only a base token on entry and settle the remainder post-execution, eliminating the transient drain window.
4. **Per-IP or per-client rate limiting** at the ingress layer (e.g., reverse proxy) to prevent a single attacker from consuming the full 500 RPS allowance.

### Proof of Concept

```bash
# Send 500 concurrent requests with no gas field (defaults to 15,000,000)
# Replace <HOST> and <CONTRACT_ADDR> with actual values
seq 500 | xargs -P500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST https://<HOST>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"<CONTRACT_ADDR>"}'

# Immediately after, send a legitimate request:
curl -X POST https://<HOST>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"<CONTRACT_ADDR>","gas":21000}'
# Expected: HTTP 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

The 500 attacker requests each consume `scaleGas(15_000_000)` = 1,500 tokens, totalling 750,000 tokens = full `gasLimitBucket` capacity. The legitimate follow-up request is rejected during the execution window of the attacker's requests.
