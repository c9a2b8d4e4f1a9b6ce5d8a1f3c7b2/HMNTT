### Title
Gas Bucket Fully Bypassed via 100% Refund on Failing Requests, Enabling Sustained High-Calldata DoS at Rate-Limit Ceiling

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` consumes exactly 1 token per request regardless of calldata size or computational weight. The `gasLimitBucket` — intended as the computational throttle — is fully refunded (100% by default via `gasLimitRefundPercent`) whenever a transaction fails with out-of-gas, which is trivially forced by sending minimum gas (21,000) with maximum calldata (up to 512 KB). The result is that an unprivileged attacker can sustain 500 requests/second, each carrying 512 KB of calldata, with zero net gas-bucket cost, triggering expensive DB queries, EVM state loading, and calldata parsing on every request.

### Finding Description

**Code path:**

`ContractController.call()` → `throttleManager.throttle(request)` → `contractExecutionService.processCall(params)` → `restoreGasToBucket()`

**Step 1 — `rateLimitBucket` is request-count-only:** [1](#0-0) 

`rateLimitBucket.tryConsume(1)` always deducts exactly 1 token regardless of the `data` field size. The `data` field is validated only as hex with a default `maxLength` of 1,048,576 hex characters (512 KB of calldata): [2](#0-1) [3](#0-2) 

**Step 2 — `gasLimitBucket` is bypassed via full refund on failure:**

`gasLimitRefundPercent` defaults to 100: [4](#0-3) 

`restoreGasToBucket` restores 100% of `gasLimit` whenever the transaction fails with `gasUsed == gasLimit` (i.e., out-of-gas) or when `result == null`: [5](#0-4) 

**Step 3 — Minimum gas forces out-of-gas with large calldata:**

`ContractCallRequest.gas` has `@Min(21_000)`. With `gas = 21,000` and 512 KB of non-zero calldata, the EVM immediately exhausts gas (calldata costs 16 gas/non-zero byte × 524,288 bytes ≈ 8.4 M gas >> 21,000). The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` is satisfied, so `scaleGas(21000) = 2` tokens are restored — net gas-bucket consumption is **0**: [6](#0-5) [7](#0-6) 

**Failed assumption:** The design assumes the gas bucket will throttle computationally expensive requests. This assumption breaks entirely for failing transactions because the refund policy returns all consumed tokens, leaving only the flat request-count bucket as protection.

### Impact Explanation

At 500 req/s (the default `requestsPerSecond` limit), each request carrying 512 KB of calldata produces:
- **256 MB/s** of calldata to parse and hex-validate
- Per-request DB queries to load contract bytecode, account state, and EVM context (these occur before gas is consumed)
- EVM initialization and calldata ingestion before the out-of-gas halt

This is sustained indefinitely with zero gas-bucket cost. A single attacker thread at the rate limit can increase node CPU, memory, and DB I/O by well over 30% compared to a baseline of normal-sized requests, satisfying the stated impact threshold. No authentication, API key, or privileged access is required.

### Likelihood Explanation

The exploit requires only an HTTP client and knowledge of the public `/api/v1/contracts/call` endpoint. The minimum gas value (21,000) and maximum calldata size are both documented in the OpenAPI spec. The attack is trivially scriptable, repeatable, and requires no on-chain state or tokens. Any external party can execute it.

### Recommendation

1. **Decouple calldata size from the flat rate limit:** Charge additional `rateLimitBucket` tokens proportional to `data` length (e.g., 1 token per N bytes of calldata), or add a separate calldata-size bucket.
2. **Do not fully refund the gas bucket on failure:** Reduce `gasLimitRefundPercent` to a value significantly below 100 (e.g., 10–25%) so that failed requests still consume a meaningful fraction of their declared gas from the bucket.
3. **Add a hard calldata size cap independent of gas:** Enforce a maximum `data` length (e.g., 6 KB for non-jumbo calls) at the controller layer before throttle tokens are consumed, consistent with the documented 6 KB smart contract call size in the design doc.

### Proof of Concept

```bash
# Craft a 512 KB hex payload (all non-zero bytes to maximize gas cost)
PAYLOAD="0x$(python3 -c "print('ff' * 524288)")"

# Send at the rate limit ceiling (500 req/s) with minimum gas
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{
      \"to\": \"0x0000000000000000000000000000000000000001\",
      \"gas\": 21000,
      \"data\": \"$PAYLOAD\"
    }" &
done
wait
```

Each request:
1. Passes `rateLimitBucket.tryConsume(1)` — 1 of 500 tokens consumed
2. Passes `gasLimitBucket.tryConsume(2)` — 2 tokens consumed for `scaleGas(21000)`
3. Triggers DB lookup for contract/account state and EVM initialization
4. EVM halts immediately with out-of-gas (512 KB calldata >> 21,000 gas)
5. `restoreGasToBucket` restores 2 tokens — **net gas-bucket cost: 0**
6. Repeat every second indefinitely [8](#0-7) [9](#0-8) [10](#0-9) [11](#0-10)

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
