### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent NFT Transaction History Requests

### Summary
`getNftTransferHistoryRequest()` in `rest/tokens.js` issues two to three sequential database queries per request against a finite connection pool, with no application-level rate limiting. An unauthenticated attacker flooding `GET /api/v1/tokens/:tokenId/nfts/:serialNumber/transactions` with concurrent requests can exhaust the pool, causing all REST API queries to fail with connection timeout errors and rendering the mirror node unable to serve any transaction data.

### Finding Description

**Exact code path:** `rest/tokens.js`, `getNftTransferHistoryRequest()`, lines 997–1056.

Each invocation performs:

1. **Query 1** (line 1022): `pool.queryQuietly(timestampQuery, timestampParams)` — fetches timestamps from `nft`, `nft_history`, and `entity` tables.
2. **Query 2** (line 1032): `pool.queryQuietly(query, params)` — fetches full transaction details for those timestamps.
3. **Query 3** (line 1047, conditional): `NftService.getNft(tokenId, serialNumber)` — checks NFT existence when results are non-empty. [1](#0-0) 

The DB pool is initialized in `rest/dbpool.js` with a hard cap of `config.db.pool.maxConnections` connections and a `statement_timeout` per query: [2](#0-1) 

**Root cause:** The function holds a pool connection for each of the 2–3 sequential queries. Under high concurrency, all pool slots are occupied simultaneously. The `statement_timeout` limits individual query duration but does not prevent pool saturation when hundreds of requests are in-flight concurrently — each request occupies a slot for the full sequential chain before releasing it.

**Why existing checks fail:**

- `statement_timeout` limits per-query execution time but does not cap the number of concurrent connections consumed.
- `connectionTimeoutMillis` causes new requests to fail fast once the pool is exhausted, but this means legitimate users receive errors — the pool is still exhausted.
- There is **no application-level rate limiting** anywhere in the REST JavaScript layer: [3](#0-2) 

- The Traefik middleware template for the REST service is **conditionally applied** (`{{ if and .Values.global.middleware .Values.middleware }}`), meaning it is optional and may not be deployed: [4](#0-3) 

No `inFlightReq` or `rateLimit` middleware is defined in the REST chart values (unlike the Rosetta chart which explicitly configures both).

### Impact Explanation
When the pool is exhausted, every subsequent REST API query — including queries for new blocks, transactions, and account data — fails with a connection timeout. The mirror node's REST API becomes completely unresponsive to all clients. While the importer continues ingesting data independently, no consumer can retrieve newly reflected network transactions via the API. This constitutes a full REST-layer denial of service with no authentication barrier.

### Likelihood Explanation
The endpoint requires zero authentication or privileges. Any external actor with network access can issue thousands of concurrent HTTP GET requests using standard tools (`ab`, `wrk`, `hey`, `curl` in parallel). The target NFT serial number can be any high-activity token visible on-chain. The attack is trivially repeatable and requires no special knowledge beyond the public API specification documented in `rest/api/v1/openapi.yml`: [5](#0-4) 

### Recommendation
1. **Add application-level concurrency limiting** in the REST server (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) independent of infrastructure middleware.
2. **Enforce Traefik `inFlightReq` and `rateLimit` middleware** as non-optional defaults in `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta chart pattern.
3. **Merge the two sequential queries** into a single SQL query (CTE joining timestamps with transaction details) to halve connection hold time per request.
4. **Set a per-request DB query budget** (e.g., abort the handler if the first query exceeds a threshold) to bound worst-case connection occupancy.

### Proof of Concept

```bash
# Identify a high-activity NFT token and serial number from public chain data
TOKEN_ID="0.0.1234567"
SERIAL="1"
MIRROR_URL="https://<mirror-node-host>"

# Flood with 500 concurrent requests (repeat to sustain pressure)
seq 500 | xargs -P 500 -I{} curl -s \
  "${MIRROR_URL}/api/v1/tokens/${TOKEN_ID}/nfts/${SERIAL}/transactions" \
  -o /dev/null -w "%{http_code}\n" &

# Observe: legitimate requests begin returning 500/503 (connection pool timeout)
curl -v "${MIRROR_URL}/api/v1/tokens/${TOKEN_ID}/nfts/${SERIAL}/transactions"
# Expected: connection timeout error or HTTP 500 from pool exhaustion
```

Sustained for the duration of the pool's `connectionTimeoutMillis`, all REST API endpoints become unavailable to legitimate users.
