### Title
Uncached EVM Address DB Lookup on Every Request Enables DB Throughput Griefing

### Summary
Any unauthenticated user can supply a valid 40-hex-character opaque EVM address as the `idOrAliasOrEvmAddress` path parameter to `GET /accounts/:idOrAliasOrEvmAddress/nfts`. Because `EntityService.getEntityIdFromEvmAddress()` performs a live database query with no caching on every invocation, and `BaseService.getRows()` has no memoization layer, an attacker can trivially force one DB round-trip per request at arbitrary request rates, degrading shared database throughput for all users.

### Finding Description

**Exact code path:**

1. `rest/controllers/accountController.js` line 91 — `getNftsByAccountId()` passes the raw path parameter directly to `EntityService.getEncodedId()` with no pre-filtering:
   ```js
   const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
   ```

2. `rest/service/entityService.js` lines 118–137 — `getEncodedId()` branches on the input type. For a 40-hex-char string that is **not** a long-form num-alias (i.e., whose first 12 bytes are not all zeros), `entityId.evmAddress` is non-null and the code unconditionally calls `getEntityIdFromEvmAddress()`:
   ```js
   return entityId.evmAddress === null
     ? entityId.getEncodedId()                                    // pure math, no DB
     : await this.getEntityIdFromEvmAddress(entityId, requireResult); // DB lookup
   ```

3. `rest/service/entityService.js` lines 90–104 — `getEntityIdFromEvmAddress()` issues a live SQL query on every call:
   ```js
   const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
   ```

4. `rest/service/baseService.js` lines 55–57 — `getRows()` has **no cache**; it goes straight to the connection pool:
   ```js
   async getRows(query, params) {
     return (await this.pool().queryQuietly(query, params)).rows;
   }
   ```

**Root cause:** The branch that resolves an opaque EVM address to an entity ID always performs a DB lookup, and neither `EntityService` nor `BaseService` implements any result cache (e.g., LRU, TTL, or memoization). The regex `evmAddressRegex = /^(0x)?[A-Fa-f0-9]{40}$/` (entityId.js line 40) correctly validates the format, but a valid 40-char hex string that is not a num-alias always reaches the DB.

**Failed assumption:** The code assumes that EVM address resolution is an infrequent or low-volume operation. There is no rate limiting, no per-IP throttling, and no caching at the service layer to absorb repeated identical or varied lookups.

### Impact Explanation
Every request with a valid opaque EVM address consumes one DB connection slot and executes one `SELECT` against the `entity` table. An attacker sending a high volume of such requests (using the same or rotating EVM addresses) forces proportional DB load. Because the DB is shared across all REST API handlers, sustained abuse degrades query latency and throughput for all other users of the mirror node API — including legitimate account, token, and transaction queries. The impact is availability degradation (griefing) with no direct economic loss to network participants.

### Likelihood Explanation
The endpoint requires no authentication. Any external user can craft a valid 40-hex-char EVM address (e.g., `0x000000000000000000000000000000000000dead`) and issue it as a path parameter. The attack is trivially scriptable with `curl` or any HTTP client in a loop. Rotating through different valid EVM addresses prevents any hypothetical future per-key cache from helping. The attack is repeatable indefinitely.

### Recommendation
1. **Add a result cache in `EntityService.getEntityIdFromEvmAddress()`** — use an LRU or TTL cache (e.g., via `lru-cache` or a Redis layer) keyed on the hex EVM address. Cache both positive hits (entity ID) and negative hits (not found) with a short TTL (e.g., 60 seconds) to absorb repeated lookups without stale-data risk.
2. **Apply rate limiting at the HTTP layer** — add per-IP or global rate limiting middleware (e.g., `express-rate-limit`) specifically for path-parameter-driven endpoints that trigger DB lookups.
3. **Short-circuit on known-invalid addresses** — if the EVM address is provably not registered (e.g., a well-known zero address), return early without a DB round-trip.

### Proof of Concept
```bash
# Any valid 40-hex-char opaque EVM address triggers a DB lookup on every request
EVM="0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Flood the endpoint — each request forces one DB query with no caching
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${EVM}/nfts" &
done
wait

# Observe: DB query rate on `entity` table spikes proportionally to request rate.
# No authentication, no rate limiting, no cache prevents this.
```

Each iteration hits `getEntityIdFromEvmAddress()` → `BaseService.getRows()` → `pool().queryQuietly()` → live DB `SELECT` against the `entity` table, with no caching layer absorbing the load.
