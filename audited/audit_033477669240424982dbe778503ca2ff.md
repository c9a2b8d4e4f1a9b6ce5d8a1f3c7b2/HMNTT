### Title
Cache Pollution via Unnormalized `evmAddress` Literal Variants in `CachedPreparsedDocumentProvider`

### Summary
`CachedPreparsedDocumentProvider` keys its Caffeine cache on the raw GraphQL query string. Because the `evmAddress` field accepts both `0x`-prefixed and unprefixed hex strings (and mixed case), an attacker who inlines the address as a query literal—rather than a variable—can produce multiple distinct cache keys for semantically identical queries. This pollutes the bounded 1,000-entry cache, evicting legitimate entries and forcing repeated re-parsing.

### Finding Description

**Cache key is the raw query string, with no normalization:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The key is `executionInput.getQuery()` — the verbatim query text sent by the client.

**Schema permits both `0x`-prefixed and bare hex, and mixed case:**

`graphql/src/main/resources/graphql/account.graphqls`, line 96:
```graphql
evmAddress: String @Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")
```

**`decodeEvmAddress` strips the prefix at execution time, making both forms semantically identical:**

`graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, line 93:
```java
evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
return Hex.decodeHex(evmAddress);
```

**Cache is bounded at 1,000 entries with a 1-hour TTL:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
```java
private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```

**Exploit flow:**

The `@Pattern` directive is wired via `ValidationSchemaWiring` as a data-fetcher interceptor, meaning it runs during *execution*, not during the `parseAndValidateFunction` that populates the cache. Therefore, any syntactically valid GraphQL document—regardless of the evmAddress value—produces a cache entry.

An attacker inlines the address as a literal (not a variable):
```
{ account(input: { evmAddress: "0xAAAA...0001" }) { id } }   → cache slot A
{ account(input: { evmAddress: "aaaa...0001"   }) { id } }   → cache slot B  (same address, different string)
{ account(input: { evmAddress: "AAAA...0001"   }) { id } }   → cache slot C
{ account(input: { evmAddress: "0xaaaa...0001" }) { id } }   → cache slot D
```
Each of the 40 hex characters `a–f` can be upper or lower case, yielding up to 2²⁴ ≈ 16 million syntactically distinct strings per address. The attacker only needs ~1,000 unique strings to saturate the cache.

### Impact Explanation
Once the 1,000-entry cache is saturated with attacker-controlled entries, every legitimate query misses the cache and must be re-parsed and re-validated on every request for the full 1-hour TTL window. This increases CPU load and latency for all users. While Caffeine's `maximumSize` prevents unbounded memory growth, the cache's protective purpose (amortizing parse cost) is completely defeated. The attack is repeatable: the attacker can continuously rotate new address variants to keep the cache polluted.

### Likelihood Explanation
No authentication or privilege is required. The GraphQL endpoint is publicly accessible (`/graphql/alpha`). The `inFlightReq` middleware caps *concurrent* requests at 5 per IP but does not rate-limit sequential requests, so a single attacker IP can fill the 1,000-entry cache with sequential requests. The attack is trivially scriptable.

### Recommendation
1. **Normalize the cache key**: Before caching, strip the `0x` prefix and lowercase the evmAddress literal in the query string, or derive the cache key from a canonical form of the parsed document rather than the raw string.
2. **Prefer variables over literals**: Document and enforce (via linting or server-side guidance) that clients should use GraphQL variables for dynamic values; variable values are not part of the query string and therefore do not affect the cache key.
3. **Add per-IP request rate limiting** at the HTTP layer (not just concurrency limiting) to slow cache-flooding attempts.
4. **Reduce cache TTL or add per-IP cache-entry quotas** to limit the blast radius of a pollution attack.

### Proof of Concept
```bash
BASE="http://localhost:8083/graphql/alpha"
ADDR="aabbccddeeff00112233445566778899aabbccdd"

# Fill the cache with 1000 unique query strings for the same address
for i in $(seq 1 500); do
  # variant 1: no prefix, lowercase
  curl -s -X POST $BASE -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { evmAddress: \\\"${ADDR}\\\" }) { id } }\"}" &

  # variant 2: 0x prefix, uppercase
  UPPER=$(echo $ADDR | tr 'a-f' 'A-F')
  curl -s -X POST $BASE -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { evmAddress: \\\"0x${UPPER}\\\" }) { id } }\"}" &
done
wait

# Now send a legitimate query — it will miss the cache and be re-parsed every time
curl -X POST $BASE -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: { entityId: {shard: 0, realm: 0, num: 2} }) { balance } }"}'
```
The legitimate query at the end will always miss the cache (evicted by attacker entries) and incur full parse+validate cost on every call for up to 1 hour.
