### Title
Missing Timestamp Range Validation in `extractStakingRewardsQuery()` Enables Full Table Scan DoS

### Summary
`extractStakingRewardsQuery()` in `rest/controllers/accountController.js` processes timestamp filters by directly building SQL conditions without invoking `parseTimestampFilters()`, which is the only code path that enforces the `maxTimestampRange` config guard (default 7 days). An unauthenticated attacker can supply `timestamp=gte:0&timestamp=lte:9999999999.999999999` — both values pass per-value format validation — causing a near-full table scan on `staking_reward_transfer` for every request. Flooding with concurrent requests exhausts the database connection pool (default max 10 connections), denying service to all other API consumers.

### Finding Description

**Exact code path:**

`listStakingRewardsByAccountId` (line 170) calls `buildAndValidateFilters` with the default `filterValidityChecks` validator: [1](#0-0) 

`filterValidityChecks` for `TIMESTAMP` only calls `isValidTimestampParam(val)`: [2](#0-1) 

`isValidTimestampParam` accepts any value matching `^\d{1,10}$` or `^\d{1,10}\.\d{1,9}$`: [3](#0-2) 

This means `gte:0` (valid) and `lte:9999999999.999999999` (valid — 10-digit seconds, 9-digit nanoseconds) both pass. The parsed nanosecond values span the entire possible timestamp space (0 → 9,999,999,999,999,999,999 ns).

`extractStakingRewardsQuery` then builds conditions directly via `updateConditionsAndParamsWithInValues` with no range check: [4](#0-3) 

**The range guard that is bypassed:** `parseTimestampFilters()` contains the only enforcement of `maxTimestampRange` (default 7 days = 604,800,000,000,000 ns): [5](#0-4) 

`extractStakingRewardsQuery` never calls `parseTimestampFilters`. The contract-logs endpoint does call it (via `checkTimestampsForTopics`), but the rewards endpoint has no equivalent guard.

**Resulting SQL** (from `stakingRewardTransferService.js`): [6](#0-5) 

The table schema has a composite primary key `(consensus_timestamp, account_id)`: [7](#0-6) 

`account_id` is the **trailing** column of the only index. With `ORDER BY consensus_timestamp DESC LIMIT 100`, PostgreSQL scans the primary key index in reverse, checking each row for `account_id = $1`. For an account with zero or few staking rewards, this degenerates into a full table scan before returning.

### Impact Explanation

Each crafted request holds a database connection for up to the full `statementTimeout` (default 20,000 ms). The connection pool is capped at 10 connections by default. An attacker sending ~10 concurrent requests with the maximum timestamp range against an account with no staking rewards saturates the pool, causing all other REST API queries to queue or fail. This is a complete REST API denial of service — matching the "network not able to confirm new transactions" severity label because mirror node availability is required for clients to submit and track transactions. [8](#0-7) 

### Likelihood Explanation

- **No authentication required.** Any public internet client can reach `/:idOrAliasOrEvmAddress/rewards`.
- **Account validity check is trivially bypassed.** `isValidAccount` only checks the `entity` table; any real account with no staking history (e.g., a freshly created account) satisfies it.
- **Payload is a standard HTTP GET.** No special tooling needed — `curl` or any HTTP load generator suffices.
- **Repeatable and automatable.** The attacker can sustain the attack indefinitely with a simple loop.

### Recommendation

Apply the same `parseTimestampFilters` range validation used by other endpoints. In `extractStakingRewardsQuery`, after collecting timestamp filters, call:

```js
parseTimestampFilters(timestampFilters, false, false, true, false, true);
```

with `validateRange = true` (the last argument). This enforces `maxTimestampRange` (default 7 days) and rejects requests whose `gte`/`lte` span exceeds it, consistent with the contract-logs and other guarded endpoints.

Alternatively, add a dedicated `filterDependencyChecker` passed to `buildAndValidateFilters` for the rewards endpoint that calls `parseTimestampFilters` on any collected timestamp filters.

### Proof of Concept

```bash
# Step 1: Find any valid account with no staking rewards (e.g., account 0.0.12345)
# Step 2: Flood with maximum-range timestamp requests

for i in $(seq 1 15); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.12345/rewards?timestamp=gte:0&timestamp=lte:9999999999.999999999" &
done
wait
```

Each request generates:
```sql
SELECT srt.account_id, srt.amount, srt.consensus_timestamp
FROM staking_reward_transfer srt
WHERE srt.account_id = <id>
  AND srt.consensus_timestamp >= 0
  AND srt.consensus_timestamp <= 9999999999999999999
ORDER BY srt.consensus_timestamp DESC
LIMIT 25
```

With 10+ concurrent requests, the DB connection pool (max 10) is exhausted. Subsequent legitimate API calls receive connection timeout errors until the attack stops or the 20-second statement timeout expires per connection.
