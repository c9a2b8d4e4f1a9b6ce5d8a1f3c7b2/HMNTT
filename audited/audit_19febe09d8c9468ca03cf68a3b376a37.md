### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent EVM Address Lookups in `getAccountCryptoAllowances`

### Summary
The `getAccountCryptoAllowances()` handler unconditionally calls `EntityService.getEntityIdFromEvmAddress()` for every EVM-address-style path parameter, which always executes a live database query with no result caching. Because the REST API's connection pool defaults to only 10 connections and no application-level rate limiting exists, an unprivileged attacker sending concurrent requests with distinct valid EVM addresses can exhaust the pool and deny service to all API consumers.

### Finding Description

**Code path:**

`rest/controllers/cryptoAllowanceController.js:77` calls `EntityService.getEncodedId()` for every inbound request: [1](#0-0) 

`rest/service/entityService.js:118-124` — when the path parameter is an EVM address, `getEncodedId()` always delegates to `getEntityIdFromEvmAddress()`: [2](#0-1) 

`rest/service/entityService.js:90-91` — `getEntityIdFromEvmAddress()` unconditionally executes `entityFromEvmAddressQuery` against the database on every call, with no caching of the result: [3](#0-2) 

**Failed assumption — the `entityId` cache does NOT protect this path.** The `quickLru` cache in `rest/entityId.js` caches only the *parsing* of the EVM address string into an `EntityId` object (key = `${id}_${allowEvmAddress}_${evmAddressType}`): [4](#0-3) 

It does not cache the database lookup result. `getEntityIdFromEvmAddress()` is called after parsing and always issues a fresh query regardless of whether the same address was seen before.

**Pool size:** The default REST API connection pool is capped at **10 connections**: [5](#0-4) [6](#0-5) 

**No application-level rate limiting:** The REST API middleware performs no throttling: [7](#0-6) 

The throttling found in the codebase applies only to the separate `web3` Java service, not to this Node.js REST API.

### Impact Explanation

With a pool of 10 connections and a `connectionTimeoutMillis` of 20 000 ms, an attacker sending ≥10 concurrent requests — each using a distinct, syntactically valid EVM address — holds all pool connections simultaneously. Every subsequent legitimate request blocks for up to 20 seconds waiting for a free connection, then either times out or receives an error. This constitutes a complete, sustained denial of service for all API consumers from a single unauthenticated client. The `statement_timeout` of 20 000 ms means each attacker connection can be held for up to 20 seconds per wave, making the attack easy to sustain with a simple loop.

### Likelihood Explanation

The endpoint is public and unauthenticated. The attack requires only a standard HTTP client capable of issuing concurrent requests. EVM addresses need only be syntactically valid 20-byte hex strings — they do not need to resolve to real entities (a `NotFoundError` is thrown after the query completes, but the connection is already consumed during query execution). The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public API specification.

### Recommendation

1. **Cache DB lookup results:** Introduce a bounded, TTL-based in-process cache (e.g., `quick-lru`) inside `getEntityIdFromEvmAddress()` keyed on the hex EVM address, mirroring the existing `entityId` parse cache. The configuration keys `hiero.mirror.rest.cache.entityId.maxAge` and `hiero.mirror.rest.cache.entityId.maxSize` already exist and could govern this cache.
2. **Increase the default pool size** or document a minimum recommended value appropriate for expected concurrency.
3. **Add application-level rate limiting** to the Node.js REST API (e.g., via an Express middleware such as `express-rate-limit`) to bound per-IP or global request rates independently of any optional infrastructure-layer Traefik configuration.

### Proof of Concept

```bash
# Generate 50 syntactically valid but distinct EVM addresses and fire concurrent requests
for i in $(seq 1 50); do
  addr=$(printf "0x%040x" $i)
  curl -s "https://<mirror-node>/api/v1/accounts/${addr}/allowances/crypto" &
done
wait
```

**Preconditions:** No credentials required; the mirror node REST API must be reachable.

**Trigger:** All 50 requests arrive concurrently. Each acquires a DB connection to execute `entityFromEvmAddressQuery`. With a pool of 10, 40 requests queue immediately.

**Result:** The DB connection pool is exhausted. All other API endpoints sharing the same pool (`/api/v1/accounts`, `/api/v1/transactions`, etc.) become unresponsive for the duration of the attack wave (up to 20 seconds per wave, indefinitely repeatable).
