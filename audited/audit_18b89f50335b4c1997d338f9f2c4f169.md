### Title
Unbounded `route` Label Cardinality Explosion via Unmatched URL Paths in Rosetta Metrics Middleware

### Summary
The `MetricsMiddleware` in `rosetta/app/middleware/metrics.go` uses `github.com/weaveworks/common/middleware.Instrument` with a gorilla/mux `RouteMatcher`. When an incoming request does not match any registered route, the weaveworks `Instrument` falls back to using the raw `r.URL.Path` as the `route` label value across all four registered metrics. An unprivileged attacker can send requests to an unbounded number of unique URL paths, creating a new Prometheus time-series label combination per unique path in all four metrics, causing unbounded memory growth in `prometheus.DefaultRegisterer`.

### Finding Description
**Exact code location:** `rosetta/app/middleware/metrics.go`, lines 46–52 (`init()`) and lines 76–83 (`MetricsMiddleware`).

All four metrics are declared with a `"route"` label:
- `requestBytesHistogram` — labels: `["method", "route"]`
- `requestDurationHistogram` — labels: `["method", "route", "status_code", "ws"]`
- `requestInflightGauge` — labels: `["method", "route"]`
- `responseBytesHistogram` — labels: `["method", "route"]` [1](#0-0) 

`MetricsMiddleware` wraps the router with `middleware.Instrument{RouteMatcher: next.(middleware.RouteMatcher)}`: [2](#0-1) 

The weaveworks `Instrument.Wrap` implementation calls `RouteMatcher.Match(r, routeMatch)` (gorilla/mux). When the request path does not match any registered Rosetta route, `Match` returns `false` and the weaveworks middleware falls back to `r.URL.Path` as the `route` label value. This is the standard behavior of `github.com/weaveworks/common/middleware` at the pinned commit `dd9e68f319d5`.

The middleware is applied before the router in the server's handler chain: [3](#0-2) 

This means every request — including those to non-existent paths — passes through the metrics middleware and records a new label combination using the raw URL path. There is no normalization, truncation, or allowlist applied to the `route` label value before it is passed to `With(labels).Observe(...)` on the `HistogramVec`/`GaugeVec`.

**Root cause:** The `route` label is populated with an unbounded, attacker-controlled string (`r.URL.Path`) for all unmatched requests. The failed assumption is that the `RouteMatcher` will always return a finite set of known route templates; it does not — it silently falls back to the raw path.

**Why existing checks fail:** There are no checks. No rate limiting, no path length cap, no label value allowlist, and no authentication on the Rosetta HTTP port. The `prometheus.DefaultRegisterer` stores every unique label combination as a separate in-memory time series indefinitely.

### Impact Explanation
Each unique URL path sent by the attacker creates approximately 23+ new in-memory Prometheus time series (7 per histogram × 3 histograms + 1 gauge = 22, plus internal bucket tracking). Sending 50,000 unique paths creates ~1.1 million time series. The Go process heap grows proportionally with no upper bound and no eviction. This directly causes memory exhaustion of the Rosetta node process, leading to OOM kills or severe GC pressure, degrading or halting blockchain API availability. The impact satisfies the ≥30% resource consumption increase threshold with a modest number of requests.

### Likelihood Explanation
The Rosetta API is a public-facing HTTP service with no authentication required on standard endpoints. Any network-reachable client can send HTTP requests. The attack requires no credentials, no special protocol knowledge, and no brute force — a simple loop generating unique path strings (e.g., `/x/1`, `/x/2`, ...) is sufficient. The attack is repeatable, automatable, and produces permanent memory growth (label combinations are never evicted from the Prometheus registry). A single attacker with a basic script can sustain the attack indefinitely.

### Recommendation
1. **Normalize the route label to a fixed allowlist:** Before passing the route value to the metrics, check whether it matches a known route template. If not, use a fixed sentinel value such as `"unmatched"` or `"unknown"`.
2. **Use a custom `RouteMatcher` wrapper** that intercepts the fallback case and substitutes a constant string instead of `r.URL.Path`.
3. **Alternatively**, pre-initialize all label combinations at startup using `prometheus.MustCurryWith` or `GetMetricWith` for each known route, and reject (or bucket) any label value not in the pre-initialized set.
4. **Add rate limiting** (e.g., via a token bucket middleware) before the metrics middleware to bound the rate at which new label combinations can be created.

### Proof of Concept
```bash
# Send 10,000 requests to unique paths (no auth required)
for i in $(seq 1 10000); do
  curl -s -o /dev/null "http://<rosetta-host>:<port>/exploit/path/$i" &
done
wait

# Observe memory growth via /metrics endpoint
curl http://<rosetta-host>:<port>/metrics | grep -c 'hiero_mirror_rosetta_request_bytes'
# Expected: ~10,000+ unique route label combinations per metric
# Baseline: ~20 (one per registered Rosetta route)
# Memory increase: proportional to unique label count × series overhead (~500 bytes each)
```

Each iteration of the loop creates a new `route="/exploit/path/N"` label combination across all four metrics in `prometheus.DefaultRegisterer`, permanently consuming heap memory with no eviction path.
