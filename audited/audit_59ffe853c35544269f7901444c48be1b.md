### Title
Connection Pool Exhaustion via Hanging DB Queries During Network Partition on `/accounts/:idOrAliasOrEvmAddress/allowances/crypto`

### Summary
The `getAccountCryptoAllowances` handler unconditionally calls `EntityService.getEncodedId()` before any other processing. When the path parameter is an EVM address or account alias, this issues a live database query. During a network partition, node-postgres connections block indefinitely at the TCP layer because the only configured timeout (`statement_timeout`) is server-side and cannot be delivered back to the client. With a default pool of 10 connections and no rate limiting on the REST API, an unprivileged attacker can exhaust the entire pool with 10 concurrent requests, rendering the service unavailable to all other users.

### Finding Description

**Exact code path:**

`rest/controllers/cryptoAllowanceController.js` line 77 — the very first `await` in the handler:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
``` [1](#0-0) 

`rest/service/entityService.js` lines 91 and 43 — the two DB-hitting branches:
- EVM address → `this.getRows(EntityService.entityFromEvmAddressQuery, ...)` [2](#0-1) 
- Alias → `super.getRows(EntityService.entityFromAliasQuery, ...)` [3](#0-2) 

Note: plain numeric IDs (`0.0.X`) return `entityId.getEncodedId()` without a DB call (line 123), so only EVM-address and alias-format parameters are exploitable. [4](#0-3) 

**Pool configuration** (`rest/dbpool.js`):
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20 000 ms
``` [5](#0-4) 

Documented defaults confirm `maxConnections = 10` and `statementTimeout = 20000`: [6](#0-5) 

**Root cause — missing client-side query timeout:**

`statement_timeout` is a PostgreSQL server-side setting applied via `SET statement_timeout = X` at connection time. During a network partition the server fires the timeout and attempts to send an error frame, but that frame never reaches the client. The node-postgres client has no independent client-side `query_timeout` configured, so the `pool.query()` / `client.query()` promise never rejects. The connection slot remains checked out from the pool indefinitely (until OS-level TCP keepalive fires, which defaults to ~2 hours on Linux).

`connectionTimeoutMillis` (20 s) is the time a caller waits for a *free slot* in the pool, not a query execution deadline. Once all 10 slots are occupied by hanging queries, every subsequent request waits 20 s and then receives a pool-timeout error — the service is effectively down for all users.

### Impact Explanation

- **Availability (High):** With only 10 pool connections (default), 10 concurrent requests with EVM-address or alias parameters during a network partition exhaust the pool. All subsequent requests to *any* endpoint that touches the database fail with a connection-timeout error for the duration of the partition (potentially hours without manual intervention).
- **No authentication required:** The endpoint is public; any unauthenticated user can send the 10 requests.
- **Blast radius:** The pool is shared across all REST API endpoints, so the DoS is not limited to the allowances endpoint.

### Likelihood Explanation

- **Trivial to trigger:** An attacker needs only 10 concurrent HTTP requests using EVM-address-format path parameters (e.g., `0x` + 40 hex chars). No credentials, tokens, or special knowledge are required.
- **Network partition is not required to be attacker-controlled:** Any transient DB connectivity issue (cloud provider blip, failover, misconfigured firewall rule) creates the same window. An attacker who can observe or induce even brief DB unreachability can reliably trigger pool exhaustion.
- **Repeatable:** After the partition resolves and connections drain, the attack can be repeated immediately.

### Recommendation

1. **Add a client-side query timeout** in the node-postgres pool config:
   ```js
   query_timeout: config.db.pool.statementTimeout,  // client-side, fires even during partition
   ```
   This causes node-postgres to reject the promise and release the connection back to the pool after the configured interval, regardless of network state.

2. **Add per-IP or global concurrency / rate limiting** to the REST API (the existing throttle infrastructure is only wired to the `web3` Java module, not the Node.js REST service).

3. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer is already present in the Helm chart for other users) in front of the REST API's pool to absorb burst load.

4. **Validate the path parameter format early** and reject non-numeric formats that would require a DB lookup before any pool connection is acquired, where the use-case allows it.

### Proof of Concept

**Preconditions:**
- Mirror Node REST API is running with default config (`maxConnections=10`, no rate limiting).
- A transient or induced DB network partition is in effect (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the REST API host).

**Steps:**
```bash
# 1. Induce network partition to DB (or wait for one)
sudo iptables -I OUTPUT -p tcp --dport 5432 -j DROP

# 2. Send 10 concurrent requests using EVM-address format (triggers DB lookup)
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/accounts/0x000000000000000000000000000000000000abcd/allowances/crypto" &
done
wait

# 3. Immediately send a legitimate request (any endpoint requiring DB)
curl -v "http://<mirror-node>/api/v1/accounts/0.0.1234/allowances/crypto"
# Expected result: hangs for 20 s then returns:
# {"_status":{"messages":[{"message":"Unable to connect to database"}]}}
# (or equivalent pool-timeout error)

# 4. Restore network — pool drains only after TCP keepalive (~2 h) or process restart
sudo iptables -D OUTPUT -p tcp --dport 5432 -j DROP
```
