### Title
Unauthenticated Unbounded Enumeration of NFT Allowances via Missing Rate Limiting on `findAll()` in `rest-java`

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint in the `rest-java` module has no rate limiting of any kind. Any unauthenticated external user can issue unlimited requests per second with arbitrary `accountId` path values, causing both full enumeration of all `approved_for_all` NFT allowance relationships on the network and database exhaustion leading to denial of service for legitimate users.

### Finding Description
**Exact code path:**

`AllowancesController.getNftAllowances()` (lines 57–83) receives the request and delegates to `NftAllowanceServiceImpl.getNftAllowances()`, which calls `NftAllowanceRepositoryCustomImpl.findAll()` (lines 37–47). The repository executes a direct jOOQ query against the `nft_allowance` table filtered only by `owner`/`spender` and `approved_for_all = true`, with a `LIMIT` clause.

**Root cause:**

The `rest-java` module has no rate limiting infrastructure. The throttle system (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`, bucket4j `Bucket` beans) exists exclusively in the `web3` module and is never applied to `rest-java`. The only servlet filters registered in `rest-java` are `ShallowEtagHeaderFilter` (ETag caching), `LoggingFilter` (request logging), and `MetricsFilter` (byte metrics) — none enforce any request rate limit.

**Why existing checks fail:**

- `NftAllowanceServiceImpl.checkOwnerSpenderParamValidity()` (lines 33–53) only validates parameter combinations (e.g., requiring `account.id` when `token.id` is present) — it does not throttle.
- `@Max(MAX_LIMIT)` on the `limit` parameter caps result set size per request but does not limit request frequency.
- No `@RateLimiter`, no `Filter` implementing throttle logic, no Spring Security layer, and no API gateway configuration is present in the `rest-java` codebase.

**Exploit flow:**

An attacker iterates over all valid Hedera account IDs (shard.realm.num format, publicly enumerable) and issues `GET /api/v1/accounts/{id}/allowances/nfts` for each, with `limit=100` (or `MAX_LIMIT`). Each request triggers a full database query. With no throttle, thousands of requests per second are possible from a single IP or distributed across IPs.

### Impact Explanation
- **Data exfiltration:** The full graph of `approved_for_all` NFT allowances (owner → spender → token relationships) is exposed to any unauthenticated caller at arbitrary speed. This reveals sensitive financial delegation relationships across the entire network.
- **Denial of service:** Sustained high-frequency requests saturate the PostgreSQL connection pool and query executor shared by all `rest-java` endpoints, degrading or denying service to legitimate users. The `db.statementTimeout` only limits individual query duration, not concurrent query volume.
- **Severity:** High — unauthenticated, no preconditions, fully repeatable, affects both confidentiality and availability.

### Likelihood Explanation
Hedera account IDs are sequential integers in a known range, making enumeration trivial. No authentication, API key, or CAPTCHA is required. A single attacker with a modest botnet or even a single high-bandwidth host can sustain thousands of requests per second indefinitely. The endpoint is publicly documented in the OpenAPI spec, lowering the bar further.

### Recommendation
1. Add a rate-limiting filter to the `rest-java` module analogous to `ThrottleManagerImpl` in `web3`, using bucket4j or Spring's `RateLimiter` abstraction, applied globally via a `OncePerRequestFilter` registered in `RestJavaConfiguration`.
2. Apply per-IP rate limiting (e.g., 100 req/s per IP) in addition to a global cap.
3. Consider adding an API key or token requirement for high-frequency access patterns.
4. Set a database connection pool limit and a maximum concurrent query cap to bound DoS impact in the interim.

### Proof of Concept
```bash
# Enumerate NFT allowances for all accounts 1–100000 at full speed
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.$i/allowances/nfts?limit=100" &
done
wait
```
No authentication header is required. Each request executes a live database query. Running this from multiple hosts simultaneously will exhaust the DB connection pool and return 500 errors to legitimate users while leaking the full `approved_for_all` NFT allowance graph.
