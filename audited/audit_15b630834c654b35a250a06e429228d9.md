### Title
Unbounded Timestamp Range in `getHookStorage()` Enables Unauthenticated Full-Table Scan DoS

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts two timestamp parameters with no validation on the range width. An unauthenticated attacker can supply `timestamp=gte:0&timestamp=lte:9000000000`, causing `getHookStorageChange()` to execute a `DISTINCT ON (key)` SQL query spanning the entire `hook_storage_change` table for the targeted hook, maximizing database I/O. No authentication, rate limiting, or timestamp-range-width guard exists to prevent this.

### Finding Description

**Code path:**

1. `HooksController.getHookStorage()` — [1](#0-0)  accepts `TimestampParameter[] timestamps` with only `@Size(max = 2)` (count limit, not range width) and no authentication annotation.

2. `Bound.of()` is called with `primarySortField = false` — [2](#0-1)  — meaning the only range sanity check (`adjustedLower > adjustedUpper`) is **skipped** for timestamps.

3. When no lower bound is given, `getAdjustedLowerRangeValue()` returns `0`; when no upper bound is given, `adjustUpperBound()` returns `Long.MAX_VALUE` — [3](#0-2)  — so supplying `gte:0` and `lte:<max>` is equivalent to the widest possible range.

4. `HookServiceImpl.getHookStorageChange()` passes these bounds directly to the repository — [4](#0-3)  — with no additional clamping.

5. The repository executes: — [5](#0-4) 
   ```sql
   SELECT DISTINCT ON (key) ...
   FROM hook_storage_change
   WHERE owner_id = :ownerId
     AND hook_id = :hookId
     AND key >= :keyLowerBound AND key <= :keyUpperBound
     AND consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound
   ```
   `DISTINCT ON (key)` requires the database to find the latest row per key across **all matching rows** before the `LIMIT` clause can be applied. With `consensus_timestamp BETWEEN 0 AND 9000000000000000000`, every historical row for the hook is a candidate, forcing a full index/table scan regardless of the `LIMIT 100` on the result.

**Failed assumption:** The `@Max(MAX_LIMIT)` guard on `limit` — [6](#0-5)  — caps returned rows but does **not** reduce the number of rows the database must read to satisfy `DISTINCT ON`. The `MAX_LIMIT = 100` constant — [7](#0-6)  — only bounds the output, not the I/O cost.

### Impact Explanation
Each crafted request forces a full historical scan of `hook_storage_change` for the targeted `(owner_id, hook_id)` pair. A hook with a large change history (common in EVM-style storage) will cause significant disk I/O and CPU on the database per request. Because the endpoint is unauthenticated and the cost is asymmetric (cheap HTTP request, expensive DB query), an attacker can sustain elevated database load with modest bandwidth, degrading service for all users (griefing/DoS). No economic damage to network participants occurs, consistent with the medium scope.

### Likelihood Explanation
The endpoint is publicly reachable with no credentials. Valid `ownerId` values are enumerable from other public mirror-node APIs. `hookId` values are sequential integers starting at 0. The attack requires only two HTTP query parameters and is trivially scriptable. It is repeatable at will with no per-request cost to the attacker.

### Recommendation
1. **Enforce a maximum timestamp range width**: In `Bound.of()` or `hookStorageChangeRequest()`, reject requests where `adjustedUpper - adjustedLower` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Require at least one narrow bound**: Reject requests where both `gte:0` and `lte:<large>` are supplied simultaneously without a tighter key filter.
3. **Add rate limiting** on the `/hooks/{hookId}/storage` endpoint, keyed by IP or caller identity.
4. **Rewrite the query** to use a lateral join or recursive CTE with an index on `(owner_id, hook_id, key, consensus_timestamp DESC)` so `LIMIT` is applied per-key before aggregation, reducing I/O even for wide ranges.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public static Bound of(TimestampParameter[] timestamp, String parameterName, Field<Long> field) {
        if (timestamp == null || timestamp.length == 0) {
            return Bound.EMPTY;
        }

        for (int i = 0; i < timestamp.length; ++i) {
            final var param = timestamp[i];
            if (param.operator() == RangeOperator.EQ) {
                timestamp[i] = new TimestampParameter(RangeOperator.LTE, param.value());
            }
        }

        return new Bound(timestamp, false, parameterName, field);
    }
