### Title
Unauthenticated Information Disclosure via Monitor Subscriber REST API Exposing Internal gRPC Status Codes and Exception Class Names

### Summary
The monitor module's `SubscriberController` exposes `/api/v1/subscriber/{name}/{id}` with no authentication requirement. Any external user can retrieve a `Scenario` object whose serialized `errors` map contains internal gRPC status code names (e.g., `UNKNOWN`, `UNAVAILABLE`, `DEADLINE_EXCEEDED`) and Java exception class simple names (e.g., `TimeoutException`, `StatusRuntimeException`), leaking internal error-handling behavior of the mirror node's gRPC subscription layer.

### Finding Description

**Endpoint with no access control:**

`SubscriberController` is annotated only with `@RestController` and `@RequestMapping("/api/v1/subscriber")`. No Spring Security configuration exists in the monitor module (confirmed: no `SecurityConfig`, no `@EnableWebSecurity`, no `application.yml` with security settings). The `subscription()` handler at line 54–59 is fully open: [1](#0-0) 

**Error map populated with internal details — two paths:**

*Path 1 — `AbstractScenario.onError()`:* stores the root cause's Java class simple name directly: [2](#0-1) 

*Path 2 — `GrpcSubscription.onError()`:* overrides the above and stores the gRPC `Status.Code` name (e.g., `UNKNOWN`, `UNAVAILABLE`, `RESOURCE_EXHAUSTED`): [3](#0-2) 

**`getErrors()` returns this map verbatim in the JSON response:** [4](#0-3) 

The documentation confirms the `errors` field is part of the public JSON response body: [5](#0-4) 

### Impact Explanation
An attacker learns which gRPC status codes the mirror node's subscription layer is returning (e.g., `UNAVAILABLE` signals backend DB issues; `RESOURCE_EXHAUSTED` signals overload; `UNKNOWN` signals unhandled exceptions). For REST subscriptions, Java exception class names such as `TimeoutException` or `WebClientResponseException` are exposed. This information aids in fingerprinting the node's internal state, timing attacks, and targeted denial-of-service or exploit attempts against known failure modes. The monitor is intended as an internal operational tool, not a public API, making this exposure unintentional.

### Likelihood Explanation
The monitor REST API is documented and its OpenAPI spec is served at `/api/v1/docs/openapi`. An attacker first calls `GET /api/v1/subscriber` (no credentials needed) to enumerate all scenario names and IDs, then queries individual scenarios. This requires zero privileges, zero authentication, and is trivially repeatable. The only precondition is network reachability to the monitor port.

### Recommendation
1. Add Spring Security to the monitor module and require authentication (e.g., HTTP Basic or token) for all `/api/v1/subscriber/**` endpoints.
2. As a defense-in-depth measure, sanitize the `errors` map before serialization — replace raw exception class names and gRPC status code strings with opaque numeric codes or generic categories before returning them in the REST response.
3. Ensure the monitor port is not exposed to untrusted networks via firewall/network policy regardless of application-level fixes.

    public <T extends ScenarioProperties> Mono<Scenario<T, Object>> subscription(

    }
