### Title
Unbounded O(N × T) DB Amplification via Correlated `getTokenBalanceSubQuery()` in Timestamp-Filtered `/api/v1/balances`

### Summary
When a timestamp filter is supplied to `/api/v1/balances`, `getBalancesQuery()` embeds `getTokenBalanceSubQuery()` as a correlated subquery that executes once per account row returned by the outer query. The `DISTINCT ON (token_id) ... ORDER BY token_id, consensus_timestamp DESC` pattern inside the subquery requires PostgreSQL to read and sort **all** `token_balance` rows for each account within the timestamp range before the `LIMIT 50` can be applied, producing O(N × T × S) total DB work (N accounts × T tokens × S historical snapshots). No rate limiting exists in the REST layer, so any unauthenticated caller can repeat this at will.

### Finding Description

**Outer query** — `rest/balances.js:207-216`:
```js
select distinct on (account_id)
  ab.account_id, ab.balance,
  ?::bigint as consensus_timestamp,
  (${tokenBalanceSubQuery}) as token_balances   // ← correlated subquery
from account_balance ab
${whereClause}
order by ab.account_id ${order}, ab.consensus_timestamp desc
${limitQuery}                                   // ← up to 100 rows
``` [1](#0-0) 

**Correlated subquery** — `rest/balances.js:294-306`:
```js
const getTokenBalanceSubQuery = (order, consensusTsQuery) => {
  consensusTsQuery = consensusTsQuery.replaceAll('ab.', 'tb.');
  return `
    select json_agg(...)
    from (
      select distinct on (token_id) token_id, balance
      from token_balance tb
      where tb.account_id = ab.account_id
        and ${consensusTsQuery}           // ← timestamp range filter
      order by token_id ${order}, consensus_timestamp desc
      limit ${tokenBalanceLimit.multipleAccounts}   // ← 50
    ) as account_token_balance`;
};
``` [2](#0-1) 

**Root cause:** `DISTINCT ON (token_id) ... ORDER BY token_id, consensus_timestamp DESC` forces PostgreSQL to materialize and sort **all** rows matching `account_id = X AND consensus_timestamp IN [lower, upper]` before the `LIMIT 50` can be applied. The `LIMIT 50` bounds the *output* rows, not the *scan work*. This subquery is then executed once per account in the outer result set (up to `responseLimit.max` = 100). [3](#0-2) 

**No rate limiting in the REST layer:** A grep across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, `express-rate`, and `throttle` returns zero matches. The throttle infrastructure found in the codebase lives exclusively in `web3/` (Java), not in the Node.js REST service.


**Timestamp range optimization** (`getOptimizedTimestampRange`) narrows the window to at most two monthly partitions, but does not bound the number of rows per account within that window. [4](#0-3) 

### Impact Explanation

Each request with `timestamp=<recent>&limit=100` causes the DB to:
1. Scan up to 100 accounts from `account_balance`.
2. For each account, scan **all** `token_balance` rows in the 2-month window, sort them, apply `DISTINCT ON`, then truncate to 50.

Hedera balance snapshots are recorded roughly every 15 minutes → ~5,760 snapshots/token/2 months. An account with 50 tokens yields ~288,000 rows scanned per account. At 100 accounts per request: **~28.8 million `token_balance` rows read per single request**. Repeated concurrent requests from one or many IPs degrade DB I/O for all users with no economic cost to the attacker.

### Likelihood Explanation

- Zero authentication or API key required.
- No per-IP or global rate limiting in the REST service.
- The endpoint is publicly documented and trivially reachable.
- A single attacker with a script loop or a small botnet can sustain the load indefinitely.
- The worst-case parameters (`limit=100`, a timestamp that resolves to a valid snapshot) are easy to discover by reading the OpenAPI spec. [5](#0-4) 

### Recommendation

1. **Break the correlated subquery amplification:** Rewrite `getTokenBalanceSubQuery` as a single lateral join or a CTE that fetches token balances for all returned accounts in one pass, rather than one subquery execution per account.
2. **Add a covering index** on `token_balance (account_id, token_id, consensus_timestamp DESC)` so that `DISTINCT ON (token_id) ORDER BY token_id, consensus_timestamp DESC` can be resolved via an index-only scan that stops after 50 rows, eliminating the full sort.
3. **Implement rate limiting** in the Node.js REST layer (e.g., `express-rate-limit`) keyed on IP address, applied before DB queries are issued.
4. **Reduce `tokenBalanceLimit.multipleAccounts`** from 50 to a lower value (e.g., 10) for the multi-account endpoint, reserving higher limits for single-account lookups.

### Proof of Concept

```bash
# Step 1: find a valid recent balance snapshot timestamp
SNAP=$(curl -s "https://<mirror-node>/api/v1/balances?limit=1" \
  | jq -r '.timestamp')

# Step 2: fire the amplified request (100 accounts, each with many token balances)
curl "https://<mirror-node>/api/v1/balances?timestamp=${SNAP}&limit=100"

# Step 3: repeat in a tight loop / parallel to saturate DB I/O
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/balances?timestamp=${SNAP}&limit=100" &
done
wait
```

Observe: DB CPU and I/O spike proportionally; response latency for all other API consumers increases. No credentials, tokens, or special network access are required.

**File:** rest/balances.js (L229-276)
```javascript
const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
};
```
