### Title
Global Rate Limit Exhaustion via Zero-Gas Requests Bypassing Gas Throttle

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` is a global singleton shared across all clients with no per-IP partitioning. An unprivileged attacker can send requests with `gas ≤ 10,000`, which causes `scaleGas()` to return `0`, making `gasLimitBucket.tryConsume(0)` always succeed — bypassing the gas-based throttle entirely. The attacker is then constrained only by the flat global rate limit (default 500 req/s), which they can fully exhaust with trivial requests, denying service to all legitimate users.

### Finding Description

**Code path:**

`ThrottleConfiguration.java` lines 24–32 — `rateLimitBucket()` is a Spring `@Bean` (singleton), meaning it is a single shared bucket across all callers with no per-IP or per-client isolation. [1](#0-0) 

`ThrottleManagerImpl.java` line 38 — `rateLimitBucket.tryConsume(1)` always deducts exactly 1 token regardless of request payload, gas, or data size. [2](#0-1) 

`ThrottleProperties.java` lines 42–47 — `scaleGas()` returns `0L` for any `gas ≤ GAS_SCALE_FACTOR` (10,000). Bucket4j's `tryConsume(0)` always returns `true`, so the gas bucket is never debited for such requests. [3](#0-2) 

**Root cause:** The failed assumption is that the gas-based secondary throttle would act as a meaningful backstop against low-cost flooding. It does not: `scaleGas(gas=0)` → `0` → `tryConsume(0)` → always `true`. The only effective guard is the flat global rate limit, which is shared across all users and exhaustible by a single attacker.

**Exploit flow:**
1. Attacker crafts a minimal `POST /api/v1/contracts/call` with `gas=0` (or any value ≤ 10,000).
2. `rateLimitBucket.tryConsume(1)` succeeds, consuming 1 of the 500 global tokens.
3. `gasLimitBucket.tryConsume(scaleGas(0))` = `tryConsume(0)` → always `true`; gas bucket untouched.
4. Attacker repeats at 500 req/s, exhausting the entire global rate limit.
5. All subsequent requests from any legitimate user hit the `ThrottleException("Requests per second rate limit exceeded")` and receive HTTP 429. [4](#0-3) 

**Why existing checks fail:**
- The gas bucket check (`gasLimitBucket.tryConsume(...)`) is the only complexity-aware guard, but it is neutralized for `gas ≤ 10,000`.
- There is no per-IP, per-session, or per-client rate limiting anywhere in the throttle stack.
- The `requestsPerSecond` default of 500 is a global ceiling, not a per-caller ceiling. [5](#0-4) 

### Impact Explanation
A single attacker with no credentials can render the `/api/v1/contracts/call` endpoint unavailable for all other users for as long as they sustain the flood. The global rate limit (500 req/s) is fully exhaustible by one client sending minimal HTTP POST requests. Legitimate users receive HTTP 429 with no recourse. This is a complete denial-of-service at the application layer with zero economic cost to the attacker.

### Likelihood Explanation
The attack requires no authentication, no special knowledge, and no on-chain resources. Any HTTP client capable of sustaining 500 req/s (trivially achievable with tools like `wrk`, `hey`, or `ab`) can execute this. The attack is repeatable indefinitely and is not detectable at the throttle layer since each individual request is technically "allowed" until the bucket empties. Likelihood is high.

### Recommendation
1. **Per-IP rate limiting:** Partition the `rateLimitBucket` by client IP (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on `X-Forwarded-For` / `RemoteAddr`), so one client cannot exhaust the global quota.
2. **Enforce a minimum gas floor:** Reject or assign a minimum token cost (e.g., treat `gas=0` as `gas=21,000`) so `scaleGas` never returns `0` for real requests, ensuring the gas bucket always provides a meaningful secondary guard.
3. **Minimum token consumption:** Change `rateLimitBucket.tryConsume(1)` to consume tokens proportional to a normalized complexity metric (e.g., `max(1, scaleGas(request.getGas()))`) so trivial requests are not cheaper than complex ones from the rate-limit perspective. [6](#0-5) 

### Proof of Concept

```bash
# Exhaust the global rate limit with zero-gas requests
# Default limit: 500 req/s; run from a single machine

ENDPOINT="http://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD='{"gas":0,"to":"0x0000000000000000000000000000000000000001","data":"0x"}'

# Send 600 requests rapidly (exceeds 500/s global limit)
for i in $(seq 1 600); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "$PAYLOAD" &
done
wait

# Expected: first ~500 return 200, remaining return 429
# Legitimate users concurrently will receive 429 Too Many Requests
```

**Verification:** With `gas=0`, `ThrottleProperties.scaleGas(0)` returns `0L` [7](#0-6) 
so `gasLimitBucket.tryConsume(0)` always returns `true`, and only the global `rateLimitBucket` (500 tokens/s, shared) is consumed — one token per attacker request, starving all other clients. [4](#0-3)
