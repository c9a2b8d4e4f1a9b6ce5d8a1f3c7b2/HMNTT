### Title
Global Rate Limit Without Per-IP Concurrency Control Enables Thread Pool Exhaustion DoS

### Summary
The web3 API enforces a single global token-bucket rate limit (500 RPS) with no per-IP or per-client concurrency cap. Because `ContractController` runs on a blocking Tomcat thread-per-request model and each request may hold its thread for up to the full `requestTimeout` (10 seconds), a single unauthenticated attacker can monopolize all 500 RPS slots and exhaust the Tomcat thread pool in under one second, rendering the service unavailable to all other users.

### Finding Description

**Code path:**

`ContractController.java` line 38–51 is a standard blocking Spring MVC handler. A thread is allocated by Tomcat before the throttle check runs. [1](#0-0) 

The throttle is a single shared global bucket — there is no per-IP, per-session, or per-client bucket anywhere in the codebase. [2](#0-1) [3](#0-2) 

The default `requestsPerSecond` is 500 and `requestTimeout` is 10 seconds. [4](#0-3) [5](#0-4) 

**Root cause — timeout enforcement is lazy (SQL-boundary only):**

The timeout is not enforced by an interrupt or a deadline on the thread. It is checked only when the next Hibernate SQL statement is about to execute, via a `StatementInspector`. [6](#0-5) 

Between SQL calls, the EVM computation runs uninterrupted. A crafted contract call that performs heavy EVM computation with infrequent DB access can hold its thread for the full 10-second window (or beyond) before the inspector fires.

**Exploit flow:**

1. Attacker opens many concurrent HTTP connections (no authentication required).
2. Each connection sends `POST /api/v1/contracts/call` with `gas = 15,000,000` (max allowed) and calldata targeting a computationally expensive code path.
3. The global `rateLimitBucket.tryConsume(1)` passes for up to 500 requests/second — all from the same attacker IP.
4. Each admitted request occupies a Tomcat worker thread for up to 10 seconds.
5. Tomcat's default thread pool (200 threads) is exhausted after ~200 requests (≈0.4 seconds at 500 RPS).
6. Subsequent legitimate requests are queued, then rejected with connection errors.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| Global RPS bucket (500/sec) | Shared across all clients; one attacker consumes 100% of capacity |
| Gas-per-second bucket | Also global; same monopolization problem |
| `requestTimeout = 10s` | Only fires at SQL boundaries; does not interrupt the thread |
| DB `statementTimeout = 3000ms` | Limits individual queries, not total request duration or thread hold time |

### Impact Explanation

Thread pool exhaustion causes complete service unavailability for all users. The web3 API is the Ethereum-compatible read layer used by DeFi bots and wallets to simulate contract calls (`eth_call`, `eth_estimateGas`) before submitting real transactions to the Hedera network. During an outage, arbitrage bots cannot check prices, liquidation bots cannot check collateral ratios, and users cannot estimate gas. Time-sensitive DeFi operations that depend on this API miss their execution windows, resulting in direct financial loss from missed arbitrage and failed liquidations.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no on-chain funds. Any attacker with a standard HTTP client and a single IP address can execute it. The 500 RPS global limit is generous enough that a single attacker can saturate it with a trivial script. The attack is repeatable indefinitely and is difficult to distinguish from legitimate high-volume usage without per-IP tracking. Likelihood is **high**.

### Recommendation

1. **Add per-IP rate limiting** as a first-class throttle dimension, either in the application (e.g., a `ConcurrentHashMap<String, Bucket>` keyed on `X-Forwarded-For` / remote address) or at the ingress/load-balancer layer.
2. **Limit per-IP concurrency** (in-flight requests), not just admission rate.
3. **Enforce the request timeout as a hard deadline** using a `Future`/`CompletableFuture` with `get(timeout)` or Spring MVC's async request handling with `AsyncContext.setTimeout`, so the thread is released even if EVM computation is between SQL calls.
4. **Consider virtual threads** (already used in the importer's actuator server) to reduce the cost of thread-per-request blocking, though this does not eliminate the DoS risk without the above controls.

### Proof of Concept

```bash
# Flood with max-gas contract calls from a single unauthenticated client
# Requires: curl, GNU parallel or xargs

TARGET="http://<web3-host>/api/v1/contracts/call"
PAYLOAD='{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x<expensive-calldata>","estimate":false}'

# Send 500 concurrent requests, each designed to run near the 10s timeout
seq 1 500 | parallel -j500 \
  curl -s -X POST "$TARGET" \
       -H "Content-Type: application/json" \
       -d "$PAYLOAD" &

# Legitimate user request — will time out or be refused
curl -v -X POST "$TARGET" \
     -H "Content-Type: application/json" \
     -d '{"to":"0x0000000000000000000000000000000000000167","gas":21000,"data":"0x","estimate":false}'
# Expected: connection refused or HTTP 503 after thread pool exhaustion
```

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
