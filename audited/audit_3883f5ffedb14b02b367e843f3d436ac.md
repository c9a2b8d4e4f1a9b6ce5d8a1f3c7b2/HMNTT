### Title
Unauthenticated GraphQL Complexity Analysis DoS via Cache-Bypassing Alias Flooding

### Summary
The `MaxQueryComplexityInstrumentation` in the GraphQL module runs on every execution request regardless of the `CachedPreparsedDocumentProvider` cache, and the cache is keyed on the raw query string. An unprivileged attacker can trivially bypass the cache by varying field aliases across requests, forcing full parse+validate+complexity analysis work on every request with no DB access and no application-level rate limiting in the GraphQL module.

### Finding Description

**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 43–48: `MaxQueryComplexityInstrumentation(200)` is registered as an execution-phase instrumentation. It runs on **every** `GraphQL.execute()` call, not during the parse/validate phase.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26: The cache key is `executionInput.getQuery()` — the raw query string. The `parseAndValidateFunction` (parse + static validation) is cached, but the complexity instrumentation is **not** part of this cached path.
- `graphql/src/main/resources/graphql/query.graphqls`, line 5: `account(input: AccountInput!)` is the only query root, publicly accessible with no authentication.

**Root cause:** The `PreparsedDocumentProvider` only caches the parse+validate step. `MaxQueryComplexityInstrumentation` is an execution-phase instrumentation that traverses the full query AST on every call. The cache key is the literal query string, so any syntactic variation (e.g., changing one alias name) produces a cache miss, forcing full parse+validate+complexity traversal for every unique query string.

**Exploit flow:**
1. Attacker crafts a query with ~199 fields using aliases (just under or just over complexity=200). With the default complexity of 1 per field, ~200 aliased scalar selections suffice.
2. Each request uses a slightly different alias (e.g., `f1` → `f2` → `f3`...) so every request is a cache miss.
3. For each request: GraphQL parses the query (CPU), validates it (CPU), then `MaxQueryComplexityInstrumentation` traverses the full AST (CPU) and rejects it — zero DB access.
4. No application-level rate limiting exists in the `graphql` module (confirmed: no `RateLimiter`, throttle, or `@PreAuthorize` in `graphql/**/*.java`).

**Why existing checks fail:**
- Parser limits (`maxTokens=1000`, `maxCharacters=10000`) bound per-query work but do not limit request rate.
- `MaxQueryComplexityInstrumentation(200)` rejects the query but only after performing the full traversal.
- The `CachedPreparsedDocumentProvider` (max 1000 entries, 1h TTL) is bypassed by alias variation — the attacker can cycle through unlimited unique strings.
- Infrastructure-level Traefik `inFlightReq: amount: 5` limits concurrent in-flight requests per IP but does not limit request rate (requests per second), so a single client can sustain a high-throughput stream of sequential requests.

### Impact Explanation
An unauthenticated attacker can saturate CPU on the GraphQL service by sending a continuous stream of unique high-complexity queries. Each request forces parse+validate+complexity analysis work with no DB access, meaning the attack is cheap for the attacker and expensive for the server. This can degrade or deny service to legitimate users. The GraphQL module has its own JVM process with CPU limits (`2` cores per the Helm chart), making it susceptible to CPU saturation.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only knowledge of the public GraphQL schema (which is introspectable by default). The alias variation technique is trivial to implement in a loop. The endpoint is publicly exposed at `/graphql/alpha`. Any attacker with network access can execute this attack repeatedly and indefinitely.

### Recommendation
1. **Add application-level rate limiting** to the GraphQL module (e.g., using Bucket4j as already used in the `web3` module — `ThrottleConfiguration`/`ThrottleManagerImpl`) to limit requests per second per IP.
2. **Disable GraphQL introspection** in production to reduce schema discoverability.
3. **Consider caching complexity results**: Use a separate bounded cache keyed on query string to skip re-analysis of already-seen (and already-rejected) queries.
4. **Normalize queries before caching**: Strip aliases or canonicalize the query AST before using it as a cache key, so alias-varied duplicates hit the same cache entry.

### Proof of Concept

```bash
# Send 1000 unique high-complexity queries, each bypassing the cache via alias variation
for i in $(seq 1 1000); do
  FIELDS=$(python3 -c "print(' '.join(f'f{i}_{j}: id' for j in range(1,200)))")
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {entityId: {num: 1}}) { $FIELDS } }\"}" &
done
wait
```

Each iteration uses a unique alias prefix (`f{i}_*`), guaranteeing a cache miss. The server performs full parse+validate+complexity analysis for each request before rejecting it.
