### Title
Unbounded Re-execution of Expensive `selectGenesis` CTE on Every Request When Genesis Record Is Unavailable

### Summary
`initGenesisRecordFile()` in `rosetta/app/persistence/block.go` uses `br.genesisBlock.ConsensusStart != genesisConsensusStartUnset` as its initialization guard, but this guard only passes after a *successful* DB query. When the DB returns `ErrRecordNotFound` (genesis not yet available), the function returns early without ever setting `genesisBlock` or consuming `sync.Once`, leaving the guard permanently open. Every subsequent request re-executes the expensive multi-table `selectGenesis` CTE with no backoff, negative caching, or application-level rate limiting, allowing any unauthenticated external user to drive sustained DB resource consumption by flooding the API during the genesis unavailability window.

### Finding Description

**Exact code path** — `rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {  // guard: only true after success
        return nil
    }
    // ... acquire DB handle ...
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)  // returns WITHOUT touching once or genesisBlock
    }
    br.once.Do(func() {          // only reached on success
        br.genesisBlock = rb
    })
    ...
}
``` [1](#0-0) 

**Root cause — failed assumption:** The design assumes `sync.Once` prevents repeated expensive queries, but `once.Do` is placed *after* the error check and is never called on failure. The guard `ConsensusStart != genesisConsensusStartUnset` evaluates to `false` (i.e., does not short-circuit) for every request until a successful query occurs, because `genesisBlock.ConsensusStart` remains at its initial value of `-1`. [2](#0-1) [3](#0-2) 

**The expensive query** — `selectGenesis` is a multi-table CTE that scans `account_balance` (ordered by `consensus_timestamp`) and joins it with `record_file` (with a correlated subquery for `consensus_end`): [4](#0-3) 

Every public endpoint (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) calls `initGenesisRecordFile()` before doing anything else: [5](#0-4) 

**Why existing checks fail:**

1. The `sync.Once` guard is structurally bypassed on every failure path — it is never consumed.
2. The Traefik middleware (`rateLimit`, `inFlightReq`) is conditionally deployed only when `Values.global.middleware` is enabled; it is not enforced at the application layer. [6](#0-5) 
3. The Traefik `retry` middleware (`attempts: 3, initialInterval: 100ms`) is configured by default, meaning each inbound request can trigger **3 CTE executions** before the client receives a response. [7](#0-6) 
4. The Go HTTP server itself has no rate limiting, connection throttling, or backoff for repeated genesis failures. [8](#0-7) 

### Impact Explanation

During any period when the genesis record is unavailable (node startup, DB migration, empty `account_balance` table, or a transient DB issue), an attacker sending a sustained flood of requests to any block-related Rosetta endpoint causes the DB to execute the `selectGenesis` CTE on every request. With the Traefik retry middleware amplifying each request 3×, even a modest 10 req/s from a single client produces 30 CTE executions per second against potentially large `account_balance` and `record_file` tables. This can increase DB CPU and I/O well beyond 30% of baseline, degrading or denying service to legitimate users and the mirror node's own importer processes. No authentication or special privilege is required.

### Likelihood Explanation

The genesis unavailability window is predictable and observable: it occurs on every node startup and is signaled by the `ErrNodeIsStarting` HTTP response. An attacker can detect this condition by polling any block endpoint and begin flooding immediately. The Rosetta API is a public, unauthenticated HTTP interface. The attack requires only an HTTP client capable of sending concurrent requests — no cryptographic material, no account, no prior knowledge of the network state.

### Recommendation

1. **Add negative caching with a short TTL**: After a failed genesis lookup, record the failure time and skip re-querying for a configurable interval (e.g., 1–5 seconds), returning `ErrNodeIsStarting` immediately without hitting the DB.
2. **Move `once.Do` to wrap the entire query attempt**, not just the assignment, so that at most one goroutine executes the CTE at a time (use a mutex or a `singleflight.Group` instead of `sync.Once` to allow retry after failure with backoff).
3. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate`) independent of the optional Traefik middleware, so the DB is protected even in bare deployments.
4. **Decouple genesis initialization from the request path**: initialize genesis eagerly at startup in a background goroutine with exponential backoff, and return `ErrNodeIsStarting` to all requests until initialization succeeds, without executing the CTE per-request.

### Proof of Concept

**Preconditions:**
- Rosetta node is starting up, or `account_balance` table is temporarily empty / not yet populated.
- Attacker can reach the Rosetta HTTP port (default 5700) or the ingress.

**Steps:**

```bash
# 1. Confirm genesis is unavailable (node returns ErrNodeIsStarting / 500)
curl -s -X POST http://<rosetta-host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"block_identifier":{"index":0}}'
# Expected: 500 with ErrNodeIsStarting

# 2. Flood the endpoint concurrently during the unavailability window
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"block_identifier":{"index":0}}' &
done
wait

# 3. Observe DB: each request executes the selectGenesis CTE
# With Traefik retry (3 attempts), 200 requests → up to 600 CTE executions
# Monitor DB CPU/IO: pg_stat_activity, pg_stat_statements, or system metrics
```

**Result:** DB CPU and I/O spike proportionally to request rate. The node remains unable to serve legitimate requests, and the genesis initialization is never completed faster — the attack sustains itself as long as the attacker keeps sending requests and the genesis data remains unavailable.

func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
	if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
		return nil
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var rb recordBlock
	if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
		First(&rb).Error; err != nil {
		return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
	}

	br.once.Do(func() {
		br.genesisBlock = rb
	})

	log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
	return nil
}
