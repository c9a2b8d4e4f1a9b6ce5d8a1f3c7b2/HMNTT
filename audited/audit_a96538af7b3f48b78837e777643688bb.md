### Title
Unauthenticated Repeated DB Query DoS via Uncached Crypto Allowance Endpoint

### Summary
The `getAccountCryptoAllowances()` function in `rest/service/cryptoAllowanceService.js` executes a direct database query on every request with no application-level caching within the service. The Redis-based response cache is **disabled by default** (`hiero.mirror.rest.cache.response.enabled = false`), and the REST API layer contains no per-IP or per-endpoint rate limiting. An unauthenticated attacker can flood the endpoint to exhaust the DB connection pool (default: 10 connections), causing denial of service of the mirror node REST API.

### Finding Description
**Code path:**

`rest/service/cryptoAllowanceService.js` lines 13–16: `getAccountCryptoAllowances()` calls `getAccountAllowancesQuery()` which builds and immediately executes `select * from crypto_allowance where owner = $1 and amount > 0 order by spender desc limit $2` via `super.getRows(query, params)` → `pool.queryQuietly()` on every invocation. [1](#0-0) 

**Cache disabled by default:**

`rest/server.js` line 54 gates the entire response cache on `config.cache.response.enabled && config.redis.enabled`. Per documentation, `hiero.mirror.rest.cache.response.enabled` defaults to `false`. [2](#0-1) [3](#0-2) 

When the cache is disabled, `responseCacheCheckHandler` is never registered, so every request reaches the DB. [4](#0-3) 

**No rate limiting in the REST API layer:**

The throttling code (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) exists only in the `web3` Java module and applies only to `ContractCallRequest` objects. There is no equivalent middleware in the Node.js REST API. [5](#0-4) 

**DB pool is small:**

Default `maxConnections = 10`, `statementTimeout = 20000ms`. With 10 concurrent slow queries, the pool is fully saturated. [6](#0-5) 

### Impact Explanation
An attacker can exhaust the 10-connection DB pool by sending concurrent requests to `/api/v1/accounts/{id}/allowances/crypto`. Once the pool is saturated, all other REST API endpoints that require DB access (transactions, balances, etc.) will queue or time out, causing a full REST API denial of service. Note: the Hedera consensus network itself is unaffected — the mirror node is a read-only indexer and does not participate in transaction confirmation. The impact is DoS of the mirror node REST API, not network-level transaction confirmation failure.

### Likelihood Explanation
Exploitation requires no credentials, no special knowledge, and no tooling beyond a basic HTTP client. The endpoint is publicly accessible. A single attacker with modest bandwidth can sustain 10+ concurrent requests indefinitely. The default deployment (cache disabled, no rate limiting) makes this trivially repeatable.

### Recommendation
1. **Enable the response cache by default** or at minimum document it as a required production setting. Set `hiero.mirror.rest.cache.response.enabled = true` with Redis in all deployment configurations.
2. **Add per-IP rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) for all endpoints, particularly allowance endpoints.
3. **Add an infrastructure-level rate limit** via Traefik ingress for the REST API (similar to the existing Rosetta configuration). [7](#0-6) 
4. Consider adding a query result cache at the service layer for allowance queries with a short TTL (e.g., 5–10 seconds).

### Proof of Concept
```bash
# Identify a large account with many spenders (publicly observable on-chain)
ACCOUNT="0.0.98"  # treasury or high-activity account

# Saturate the DB pool with 15 concurrent requests (exceeds default pool of 10)
for i in $(seq 1 15); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/allowances/crypto" &
done
wait

# Subsequent requests to any DB-backed endpoint will queue/timeout
curl -v "https://<mirror-node>/api/v1/transactions"
# Expected: delayed response or 503 once pool is exhausted
```
