### Title
Unvalidated `X-Forwarded-For` Header Allows Arbitrary String Injection into Client IP Logs in `TracingMiddleware`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the `X-Forwarded-For` (and `X-Real-IP`) header value verbatim without any IP format validation. An unprivileged external attacker can supply any string — including `attacker:8080`, a fake IP, or a newline-containing payload — which is then logged by `TracingMiddleware` as the authoritative client IP. This breaks any downstream IP-based security monitoring, geolocation, or threat intelligence correlation applied to Rosetta API logs, including for `/construction/submit` (transaction submission) requests.

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74; consumed at line 46 and logged at line 52–53.

```
63: func getClientIpAddress(r *http.Request) string {
64:     ipAddress := r.Header.Get(xRealIpHeader)        // raw, no validation
66:     if len(ipAddress) == 0 {
67:         ipAddress = r.Header.Get(xForwardedForHeader) // raw, no validation
68:     }
70:     if len(ipAddress) == 0 {
71:         ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback parses
72:     }
74:     return ipAddress
75: }
```

**Root cause:** `net.SplitHostPort` (which strips the port and validates structure) is called **only** on `r.RemoteAddr` — the TCP-layer address that the OS provides and the attacker cannot forge. For the two header paths (lines 64 and 67), the raw string is returned with no call to `net.ParseIP`, `net.SplitHostPort`, or any length/character check beyond `len == 0`.

**Failed assumption:** The code assumes that if `X-Forwarded-For` or `X-Real-IP` is non-empty, it contains a valid IP address. HTTP headers are fully attacker-controlled.

**Exploit flow:**
1. Attacker sends `POST /construction/submit` with `X-Forwarded-For: attacker:8080`.
2. `getClientIpAddress` returns the string `"attacker:8080"` (line 67, no further processing).
3. `TracingMiddleware` formats and emits: `attacker:8080 POST /construction/submit (200) in Xms` at `log.Info` level (line 52–58).
4. The real TCP `RemoteAddr` is never consulted; the attacker's actual IP is never logged.

**Why existing checks are insufficient:** The sole guard is `len(ipAddress) == 0` (lines 66, 70). This rejects only an absent header; it does not reject `"attacker:8080"`, `"0.0.0.0"`, `"127.0.0.1"` (localhost spoofing), multi-value lists like `"1.2.3.4, 5.6.7.8"`, or newline-injected strings. The test suite (`trace_test.go` lines 52–58) only tests a well-formed IP in `X-Forwarded-For`, so the malformed case is untested and undetected.

### Impact Explanation
Any SIEM, log aggregator, or threat-intelligence pipeline that parses Rosetta access logs to extract client IPs for `/construction/submit` will receive attacker-controlled, non-IP strings. Concretely:
- **IP geolocation fails** — `"attacker:8080"` is not a parseable address.
- **Threat intelligence lookups fail or are silently skipped** — tools that call IP reputation APIs will error or no-op on non-IP tokens.
- **The attacker's real IP is never recorded** — the TCP `RemoteAddr` fallback is bypassed entirely whenever the header is present.
- **Log injection is possible** — values containing `\n` can inject synthetic log lines (e.g., fake successful submissions from a spoofed IP), further corrupting audit trails.

Severity: **Medium** (security monitoring bypass; no direct compromise of transaction integrity, but audit/forensic capability for tampered-transaction detection is undermined).

### Likelihood Explanation
- **No authentication or privilege required** — the Rosetta Construction API is designed to be called by external clients.
- **Trivially reproducible** — any HTTP client (`curl`, Python `requests`) can set arbitrary headers.
- **No rate limiting or WAF assumed** — the middleware itself applies no such controls.
- **Repeatable at will** — every request with the header set produces a poisoned log entry.

### Recommendation
Replace the raw header read with validated extraction:

```go
import "net"

func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        val := r.Header.Get(header)
        if val == "" {
            continue
        }
        // X-Forwarded-For may be a comma-separated list; take the first entry
        ip := strings.TrimSpace(strings.SplitN(val, ",", 2)[0])
        if parsed := net.ParseIP(ip); parsed != nil {
            return parsed.String()
        }
    }
    // Fallback: TCP remote address (already validated by net.SplitHostPort)
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

Key changes:
1. Call `net.ParseIP` on the extracted value; discard and fall through if it is not a valid IP.
2. Handle comma-separated `X-Forwarded-For` lists by taking only the first (leftmost) entry.
3. Only trust headers from known upstream proxies if the deployment topology allows it (configure a trusted-proxy allowlist).

### Proof of Concept

```bash
# Send a /construction/submit request with a poisoned X-Forwarded-For header
curl -s -X POST http://<rosetta-host>:<port>/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Forwarded-For: attacker:8080" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"},"signed_transaction":"..."}'

# Observe the Rosetta log output — the client IP field will be "attacker:8080":
# time="..." level=info msg="attacker:8080 POST /construction/submit (200) in 3ms"
```

The real client IP is never recorded. Repeating with `X-Forwarded-For: 127.0.0.1` causes the request to appear to originate from localhost, bypassing any IP-allowlist checks applied at the log-analysis layer. [1](#0-0) [2](#0-1)
