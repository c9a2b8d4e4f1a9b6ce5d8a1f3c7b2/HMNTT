### Title
Unbounded gRPC Connection Count Exhausts Shared `boundedElastic()` Thread Pool in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` schedules every subscriber's DB poll on the JVM-wide `Schedulers.boundedElastic()` singleton. Because the gRPC server enforces only a per-connection call cap (`maxConcurrentCallsPerConnection = 5`) with no limit on the number of connections, and because no authentication or IP-level rate limiting exists in the gRPC layer, an unauthenticated attacker can open an unbounded number of TCP connections and saturate the shared thread pool, causing all legitimate subscribers' polls to queue and stall.

### Finding Description
**Exact code path:**

`PollingTopicListener.java` line 31 stores a reference to the JVM-wide singleton:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
``` [1](#0-0) 

`listen()` (lines 34–48) uses this scheduler for both the initial delay and every repeat cycle:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))   // ← every repeat fires on boundedElastic()
``` [2](#0-1) 

`poll()` (lines 51–62) calls `topicMessageRepository.findByFilter()` which returns a `Stream` and is consumed synchronously via `Flux.fromStream()`. Because `RepeatSpec.withScheduler(scheduler)` causes the re-subscription to happen on a `boundedElastic()` thread, the blocking JDBC call holds that thread for the full query duration (up to `statementTimeout = 10 000 ms`). [3](#0-2) 

**The only server-side guard** is `maxConcurrentCallsPerConnection = 5` (default), which limits calls *per TCP connection* but places no cap on the number of connections: [4](#0-3) [5](#0-4) 

No `ServerInterceptor` for authentication or authorization exists in the gRPC module, and no IP-level rate limiting or connection throttle is configured for the gRPC server. The `ThrottleConfiguration` (Bucket4j) present in the `web3` module is entirely absent here.

**Root cause / failed assumption:** The design assumes the number of concurrent `listen()` subscriptions is bounded. It is not — `maxConcurrentCallsPerConnection` only limits multiplexed streams per HTTP/2 connection; an attacker simply opens more connections.

### Impact Explanation
`Schedulers.boundedElastic()` caps its thread pool at `10 × availableProcessors()` (e.g., 40 threads on a 4-core JVM). Each active subscription holds one of those threads for the duration of every DB poll. With enough attacker-controlled subscriptions polling concurrently, the pool is fully occupied, and legitimate subscribers' repeat tasks are queued behind attacker tasks. Poll intervals balloon from the configured 500 ms to seconds or minutes, causing message delivery latency or apparent stream stalls for all real users sharing the same JVM instance. No economic damage occurs, but the mirror node's topic subscription service is effectively degraded for all users.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`, any gRPC stub) and the ability to open many TCP connections — no credentials, no special network position, no prior knowledge beyond the public API. The `subscribeToTopic` RPC is publicly documented. Opening 20 connections × 5 streams = 100 concurrent subscriptions is trivial and already exceeds the default thread cap on a 4-core host. The attack is repeatable and stateless; the attacker need not maintain any session state beyond keeping the connections open.

### Recommendation
1. **Add a global connection limit** to the Netty server builder (e.g., `serverBuilder.maxConnectionAge(...)` and a custom `ServerTransportFilter` that counts active connections and rejects beyond a threshold).
2. **Decouple polling from `boundedElastic()`** for the delay/repeat scheduling: use `Schedulers.parallel()` (non-blocking timer) for the `withScheduler` argument and only switch to `boundedElastic()` for the actual blocking DB call via `.subscribeOn(Schedulers.boundedElastic())` scoped to `poll()` alone.
3. **Enforce a per-IP or global subscription cap** via a `ServerInterceptor` that tracks active streaming calls and returns `RESOURCE_EXHAUSTED` once a threshold is reached.
4. **Apply the same Bucket4j rate-limiting** already present in the `web3` module to the gRPC layer.

### Proof of Concept
```bash
# Prerequisites: grpcurl installed, mirror node running on localhost:5600 with POLL listener type

# Open 20 connections, each with 5 concurrent subscribeToTopic streams = 100 subscriptions
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
      -H 'content-type: application/grpc' \
      localhost:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeToTopic &
  done
done

# On a 4-core host (boundedElastic cap = 40 threads), 100 concurrent polling subscriptions
# will saturate the thread pool. Legitimate subscriber poll latency will grow from 500ms
# to several seconds, observable via the GrpcHighLatency Prometheus alert:
# hiero_mirror_grpc_consensus_latency_seconds > 15s
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
