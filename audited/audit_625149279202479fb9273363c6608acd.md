### Title
Unbounded Concurrent `fallbackRetry()` Loop Exhausts DB Connection Pool via Unauthenticated `/network/exchangerate` Flooding

### Summary
The `fallbackRetry()` function in `rest/service/fileDataService.js` executes up to 10 sequential database queries per request when the `ExchangeRate` constructor fails on returned data. With no rate limiting, no concurrency guard, and a finite DB connection pool, an unprivileged attacker can flood the unauthenticated `/network/exchangerate` endpoint to exhaust all available connections, causing a sustained DoS across all endpoints sharing the pool.

### Finding Description

**Exact code path:**

`rest/service/fileDataService.js`, lines 93–118 (`fallbackRetry`), called from `getExchangeRate` at line 89–91, which is invoked by the `/network/exchangerate` route handler.

```js
// lines 93-118
fallbackRetry = async (fileEntityId, filterQueries, resultConstructor) => {
  let attempts = 0;
  while (++attempts <= 10) {                          // up to 10 iterations
    const row = await this.getLatestFileDataContents(fileEntityId, filters); // DB query each iteration
    try {
      return row === null ? null : new resultConstructor(row);
    } catch (error) {
      // only catches ExchangeRate constructor errors — DB errors propagate out
      filters.whereQuery = [...whereQuery, { query: ..., param: row.first_consensus_timestamp }];
    }
  }
  return null;
};
```

**Root cause:** The `try/catch` block (lines 100–114) wraps only `new resultConstructor(row)`, not `getLatestFileDataContents()`. When the `ExchangeRate` constructor throws (e.g., due to malformed/partial file data — a realistic condition during a degraded DB state or partial network partition), the loop retries up to 10 times, each time issuing a new DB query via `getSingleRow` → `getRows` → `pool.queryQuietly` (baseService.js lines 55–66), each acquiring a connection from the shared global pool.

**Why existing checks fail:**

- `rest/dbpool.js` lines 7–16: The pool is bounded by `config.db.pool.maxConnections` with a `connectionTimeoutMillis`, but no per-endpoint or per-function concurrency limit exists.
- Grep across all `.js` files for `rateLimit`, `throttle`, `concurrency`, `semaphore` returns zero hits in production code — no middleware rate-limiting is applied to this endpoint.
- The `statement_timeout` (dbpool.js line 15) limits individual query duration but does not prevent 10 queries from being issued sequentially per request.
- No circuit breaker or in-flight request counter guards `fallbackRetry()`.

### Impact Explanation

With a pool of N connections (default is typically 10–20 in pg-pool), an attacker sending N/10 concurrent requests that each trigger the full 10-iteration retry loop saturates the pool. All other endpoints (`/api/v1/transactions`, `/api/v1/accounts`, etc.) share the same `global.pool` (baseService.js line 97: `return pool`), so pool exhaustion causes a complete service outage. New requests queue until `connectionTimeoutMillis` expires, returning errors to all users. The condition persists as long as the attacker maintains the flood and the `ExchangeRate` constructor continues to fail.

### Likelihood Explanation

The endpoint requires no authentication. The `ExchangeRate` constructor failure condition (malformed/partial file data) is reachable during any degraded DB state, not only a full partition. An attacker needs only a moderate botnet or even a single machine with async HTTP tooling (e.g., `ab`, `wrk`, `hey`) to sustain hundreds of concurrent requests. The attack is repeatable and requires no special knowledge beyond the public API path.

### Recommendation

1. **Wrap `getLatestFileDataContents` in the try/catch** so DB errors abort the loop immediately rather than propagating silently.
2. **Add a per-endpoint concurrency limiter** (e.g., `p-limit` or a semaphore) capping simultaneous in-flight `fallbackRetry` executions.
3. **Apply HTTP-layer rate limiting** (e.g., `express-rate-limit`) to `/network/exchangerate` and other unauthenticated endpoints.
4. **Cap the retry loop** at a lower bound (e.g., 3) and add exponential backoff to reduce burst DB load.
5. **Isolate the file-data query pool** from the main query pool so exhaustion in one does not affect the other.

### Proof of Concept

**Precondition:** DB is in a state where `file_data` rows exist but are malformed enough to cause `new ExchangeRate(row)` to throw (e.g., truncated protobuf bytes in the `file_data` column — achievable by an insider or by exploiting a separate write path, or naturally during a partial network partition where only some chunks are written).

**Trigger:**
```bash
# Send 50 concurrent requests to the unauthenticated endpoint
hey -n 5000 -c 50 https://<mirror-node-host>/api/v1/network/exchangerate
```

**Result:**
- Each of the 50 concurrent requests enters `fallbackRetry()` and issues up to 10 DB queries.
- Up to 500 DB connection acquisitions are attempted simultaneously against a pool of (default) 10–20 connections.
- Pool is exhausted; all subsequent requests to any endpoint receive connection timeout errors.
