### Title
Unbounded Indefinite DB Polling via Topic Subscription with No Empty-Result Termination in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` schedules `poll()` to repeat `Long.MAX_VALUE` times with no termination condition when `findByFilter()` returns empty results. When `checkTopicExists` is `false` (a documented, supported configuration), any unprivileged user can subscribe to an arbitrary non-existent `topicId` and cause the server to issue a DB query every 500 ms indefinitely. Even with the default `checkTopicExists=true`, the same effect is achievable by subscribing to any valid topic that has no messages, since the existence check only validates the entity table, not message presence. There is no global subscription limit, allowing an attacker to open many connections and multiply the DB load.

### Finding Description

**Code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()` lines 38–48:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← essentially infinite
                .jitter(0.1)
                .withFixedDelay(interval)              // ← default 500 ms
                .withScheduler(scheduler))
```

`poll()` lines 51–62:

```java
private Flux<TopicMessage> poll(PollingContext context) {
    ...
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter)); // ← DB query every interval
}
```

There is **no check on whether the returned stream is empty**. When `findByFilter()` returns zero rows, `poll()` returns `Flux.empty()`, the `repeatWhen` operator fires again after `interval`, and the cycle repeats forever.

**The `topicExists()` guard** in `TopicMessageServiceImpl` lines 94–106:

```java
return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
        .switchIfEmpty(
                grpcProperties.isCheckTopicExists()
                        ? Mono.error(new EntityNotFoundException(topicId))
                        : Mono.just(Entity.builder().memo("").type(EntityType.TOPIC).build()))
```

- When `checkTopicExists=false` (documented option, `hiero.mirror.grpc.checkTopicExists`): the guard is bypassed entirely; any `topicId` proceeds to the polling loop.
- When `checkTopicExists=true` (default): the guard only checks the `entity` table for existence and type. A topic that exists but has zero messages passes this check and still triggers indefinite polling.

**No global subscription limit:** `GrpcConfiguration.java` line 33 sets only `maxConcurrentCallsPerConnection=5` (default). There is no cap on total connections or total active subscriptions server-wide.

### Impact Explanation

Each active subscription with no matching messages issues one DB query per `interval` (default 500 ms) indefinitely, consuming a DB connection, a scheduler thread from `Schedulers.boundedElastic()`, and CPU for query planning. An attacker opening `N` connections × 5 calls each = `5N` concurrent subscriptions, each generating 2 DB queries/second. At modest scale (e.g., 200 connections = 1,000 subscriptions), this produces 2,000 DB queries/second of pure waste, starving legitimate subscribers and potentially exhausting the DB connection pool. This is a non-network-based DoS against the gRPC service and its backing database.

### Likelihood Explanation

- **`checkTopicExists=false` path**: Zero preconditions. Any unauthenticated gRPC client sends a `subscribeTopic` request with an arbitrary `topicId`. No valid topic knowledge required. Fully scriptable.
- **`checkTopicExists=true` path (default)**: Attacker needs one valid topic ID (trivially obtained from the public Hedera ledger or the mirror node's REST API). Subscribing to it with no messages (or a future `startTime` far ahead) triggers the same indefinite polling. Also fully scriptable.
- Repeatability: unlimited. The attacker simply keeps connections open; the server does all the work.

### Recommendation

1. **Add empty-result termination in `poll()`**: Track consecutive empty polls and terminate (or exponentially back off) after a configurable threshold, e.g.:
   ```java
   if (results.isEmpty() && ++emptyPollCount > maxEmptyPolls) {
       return Flux.error(new NoSuchElementException("Topic has no messages"));
   }
   ```
2. **Add a global active-subscription limit** in `TopicMessageServiceImpl`: reject new subscriptions when `subscriberCount` exceeds a configured maximum.
3. **Enforce per-IP or per-client connection/subscription rate limits** at the gRPC interceptor layer.
4. **Require `endTime`** or a maximum subscription duration for open-ended subscriptions to bound their lifetime.

### Proof of Concept

**With `checkTopicExists=false`:**

```bash
# Set hiero.mirror.grpc.checkTopicExists=false in application.yml

# Open 200 parallel gRPC connections, 5 subscriptions each = 1000 active subscriptions
# Each subscribes to a random non-existent topicId with no endTime/limit
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":999999999},"consensusStartTime":{"seconds":0}}' \
    localhost:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Observe DB: ~2000 queries/second of SELECT on topic_message table, all returning 0 rows
# DB connection pool exhausted; legitimate subscribers time out
```

**With `checkTopicExists=true` (default), using a valid topic with no messages:**

```bash
# Obtain any valid topic ID from REST API (e.g., 0.0.12345)
# Subscribe with startTime far in the future so findByFilter always returns empty
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":9999999999}}' \
    localhost:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Same result: indefinite empty-result polling, DB resource exhaustion
```
