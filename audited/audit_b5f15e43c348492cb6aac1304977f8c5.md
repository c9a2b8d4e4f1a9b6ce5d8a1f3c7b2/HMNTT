### Title
Unrestricted `file.id` Parameter Allows Arbitrary Address Book Access in `getNetworkNodes()`

### Summary
The `/api/v1/network/nodes` endpoint accepts a user-supplied `file.id` query parameter that is passed directly to the database query without validating that the value is restricted to the canonical address book file IDs (101 or 102). Any unauthenticated caller can supply an arbitrary numeric file ID and retrieve address book entries from non-standard address books stored in the database. The only server-side check validates the query operator type (must be `EQ`), not the value itself.

### Finding Description
**Code path:**

In `NetworkController.java` lines 154–157, the only validation on `fileId` is:
```java
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
```
This rejects range operators but places **no restriction on the actual value**.

The request is then forwarded to `NetworkServiceImpl.getNetworkNodes()`. At lines 178–182, `getAddressBookFileId()` reads:
```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
}
```
When `fileId` is present, the caller-supplied value is returned verbatim with no allowlist check against `{101, 102}`.

This value is then passed directly to `NetworkNodeRepository.findNetworkNodes()`, whose native SQL query (lines 26–29) uses it as:
```sql
where file_id = :fileId
```
Any `file_id` present in the `address_book` table is therefore reachable.

**Root cause:** The assumption that users will only supply file IDs 101 or 102 is never enforced in code. The controller validates operator type only; the service performs no value-range check before forwarding to the repository.

### Impact Explanation
An unprivileged, unauthenticated user can enumerate and retrieve address book entries for any `file_id` stored in the `address_book` table — including historical snapshots, non-production configurations, or any address book written under a non-standard file ID. The exposed data includes node account IDs, IP endpoints, ports, public keys, certificate hashes, and service endpoints. While this data is not secret in the same way as private keys, it violates the intended access boundary of the API (only files 101/102 should be queryable), and historical or non-standard address books may contain stale or sensitive network topology information that operators did not intend to expose publicly.

### Likelihood Explanation
The exploit requires no authentication, no special tooling, and no prior knowledge beyond the public API documentation. A single HTTP GET request with a numeric `file.id` value is sufficient. An attacker can trivially enumerate all valid file IDs by iterating integers and observing non-empty responses. The endpoint is publicly accessible and the parameter is documented as optional, making this highly repeatable and low-effort.

### Recommendation
Add an allowlist check in `getAddressBookFileId()` (or in the controller before dispatch) that restricts accepted values to the canonical address book file IDs:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    long file101 = systemEntity.addressBookFile101().getId();
    long file102 = systemEntity.addressBookFile102().getId();
    if (id != file101 && id != file102) {
        throw new IllegalArgumentException(
            "file.id must be " + file101 + " or " + file102);
    }
    return id;
}
```

Alternatively, enforce this constraint at the controller layer before calling the service, consistent with the existing operator-type check.

### Proof of Concept
**Precondition:** The mirror node database contains at least one `address_book` row with a `file_id` other than 101 or 102 (e.g., from historical ingestion or a non-standard address book update).

**Steps:**
1. Send a request with a non-standard file ID:
   ```
   GET /api/v1/network/nodes?file.id=999
   ```
2. The controller passes the request to `NetworkServiceImpl.getNetworkNodes()`.
3. `getAddressBookFileId()` returns `999` without any validation.
4. The repository executes:
   ```sql
   ... from address_book where file_id = 999 ...
   ```
5. If rows exist for `file_id = 999`, the response contains full node configuration data for that non-standard address book — data the API was never intended to expose.

**Expected (correct) behavior:** A `400 Bad Request` or `404 Not Found` response for any `file.id` value outside `{101, 102}`.
**Actual behavior:** Data from the requested address book is returned with HTTP 200. [1](#0-0) [2](#0-1) [3](#0-2)
