### Title
Global Rate-Limit Bucket Exhaustion via Greedy Burst — Unauthenticated DoS on Shared `rateLimitBucket`

### Summary
The `rateLimitBucket` in `ThrottleConfiguration.java` is a single JVM-wide singleton with no per-IP or per-user partitioning. Using `refillGreedy`, the full token budget (`requestsPerSecond`, default 500) is available as a burst at the start of every 1-second window. Any unauthenticated caller can fire exactly 500 concurrent requests at window-open, drain all tokens, and force every other user to receive HTTP 429 for the remainder of that window — repeatable every second indefinitely.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 24–32:
```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond();   // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))       // full burst at window start
            .build();
    return Bucket.builder().addLimit(limit).build();              // single global bean
}
```

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {                         // shared across ALL callers
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    ...
}
```

**Root cause:** `refillGreedy(N, 1s)` with `capacity(N)` makes all N tokens available simultaneously at the start of each refill period (fixed-window burst semantics). The bucket is a Spring singleton — one instance shared by every HTTP thread, with no caller identity tracked. `tryConsume(1)` is called without any per-source accounting.

**Failed assumption:** The design assumes traffic is distributed across many callers so no single caller can exhaust the budget. This assumption is false: there is no enforcement mechanism preventing one caller from consuming all N tokens.

**Exploit flow:**
1. Attacker opens 500 parallel HTTP connections (or uses a single async client).
2. At the moment the bucket refills (every ~1 s), attacker fires all 500 requests simultaneously.
3. All 500 `tryConsume(1)` calls succeed; bucket reaches 0.
4. Every subsequent request from any other user hits `tryConsume(1) == false` → `ThrottleException` → HTTP 429.
5. Attacker waits ~1 s and repeats, maintaining a continuous blackout for all other users.

### Impact Explanation
All legitimate users of the web3 JSON-RPC endpoint (`/api/v1/contracts/call`, gas estimation, opcode tracing) are denied service for up to 1 second per cycle. DeFi front-ends and wallets that rely on `eth_call` / `eth_estimateGas` to simulate or price transactions before submission are blocked. While the mirror node does not itself submit Hedera transactions, blocking contract-call simulation prevents users from constructing or validating time-sensitive transactions (e.g., DEX swaps with slippage deadlines, token approvals). The attack is zero-cost, requires no credentials, and is trivially repeatable.

### Likelihood Explanation
Any unprivileged external user with a standard HTTP client can execute this. No authentication, API key, or special network position is required. The default limit of 500 RPS is easily saturated by a single machine with concurrent HTTP/1.1 or HTTP/2 connections. The attack is deterministic (not probabilistic), repeatable every second, and leaves no persistent state to clean up. It is a well-known pattern against fixed-window token buckets.

### Recommendation
1. **Switch to `refillIntervally`** (smooth/token-bucket semantics) instead of `refillGreedy` to prevent full-burst consumption at window boundaries:
   ```java
   .refillIntervally(rateLimit, Duration.ofSeconds(1))
   ```
2. **Add per-source rate limiting** keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`) so one caller cannot exhaust the global budget.
3. **Enforce a per-IP sub-limit** (e.g., `max(requestsPerSecond / expectedConcurrentClients)`) in addition to the global limit.
4. **Consider infrastructure-level rate limiting** (API gateway, reverse proxy) as a defense-in-depth layer before requests reach the JVM.
