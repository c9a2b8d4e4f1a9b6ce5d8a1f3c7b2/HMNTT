### Title
Unauthenticated DB Connection Pool Exhaustion via Dual-Query Path in `GET /tokens/:tokenId/balances`

### Summary
When an unauthenticated caller supplies both `account.publickey` and `timestamp` query parameters to `GET /tokens/:tokenId/balances`, `extractSqlFromTokenBalancesRequest()` issues two sequential database queries per request: one inside `balances.getAccountBalanceTimestampRange()` and one for the main `token_balance` query with an `entity` JOIN. With a default pool ceiling of 10 connections and no application-level rate limiting on the REST API, an attacker maintaining ≥10 concurrent requests can saturate the pool and render the entire REST API unresponsive.

### Finding Description

**Code path — `rest/tokens.js`, `extractSqlFromTokenBalancesRequest()`, lines 589–690:** [1](#0-0) 

When `filterKeys.TIMESTAMP` is present, the value is pushed into `tsConditions`/`tsParams`. At line 638–640 the function branches into the timestamp path and immediately `await`s a separate database call: [2](#0-1) 

`balances.getAccountBalanceTimestampRange()` issues its own `pool.queryQuietly` call against the `account_balance` table to resolve a lower/upper timestamp range — **DB query #1**, holding a pool connection for its full execution duration.

After that call returns, the function builds the main query. When `account.publickey` is also present, a JOIN against the `entity` table is appended: [3](#0-2) 

The caller `getTokenBalances()` then executes this second query via `pool.queryQuietly` — **DB query #2**: [4](#0-3) 

**Pool ceiling:** The default `maxConnections` is **10**: [5](#0-4) 

**Statement timeout:** Up to **20 000 ms** per query: [6](#0-5) 

**No application-level rate limiting:** The REST API middleware only enforces `maxRepeatedQueryParameters` (default 100); there is no per-IP or global request-rate limit: [7](#0-6) 

**Both parameters are accepted without authentication:** [8](#0-7) 

**Root cause / failed assumption:** The design assumes that the `statement_timeout` and infrastructure-level proxies (pgBouncer, Traefik) will absorb burst traffic. However, the REST API itself has no rate-limiting layer, and the `timestamp` code path doubles the number of sequential DB round-trips per request. With only 10 pool slots and queries that can each hold a slot for up to 20 seconds, 10 concurrent crafted requests are sufficient to saturate the pool.

### Impact Explanation

Once the 10-connection pool is exhausted, every subsequent request to **any** REST API endpoint blocks waiting for a free connection. New requests queue until `connectionTimeoutMillis` (default 20 000 ms) elapses, after which they fail with a pool-timeout error. This constitutes a full REST API denial-of-service, preventing clients and network participants from querying balances, transactions, or any other data — matching the "total network shutdown" severity label for the mirror-node REST layer.

### Likelihood Explanation

The attack requires zero privileges, zero accounts, and only a valid hex public key (any 64-character hex string passes the filter). The attacker needs to sustain only 10 concurrent HTTP connections, achievable with a single `curl`/`ab`/`wrk` invocation from a laptop. The attack is trivially repeatable and scriptable, and because each request legitimately exercises a documented query parameter combination, it is indistinguishable from normal traffic at the HTTP layer.

### Recommendation

1. **Add per-endpoint rate limiting** in the REST API middleware (e.g., express-rate-limit keyed on IP) before the handler is reached.
2. **Increase `maxConnections`** or deploy a pgBouncer pool in front of the REST API user (`mirror_rest`) with a hard cap on concurrent server-side connections.
3. **Merge the two queries into one**: the timestamp range resolution can be inlined as a subquery or CTE inside the main `token_balance` query, eliminating the extra round-trip and halving pool pressure per request.
4. **Add a short `connectionTimeoutMillis`** (e.g., 2 000 ms) so queued requests fail fast rather than holding Node.js event-loop slots.

### Proof of Concept

```bash
# Replace TOKEN_ID with any valid token (e.g., 0.0.1)
# Replace KEY with any 64-char hex string
TOKEN="0.0.1"
KEY="3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be"
TS="1234567890000000000"
URL="http://<mirror-node-host>:5551/api/v1/tokens/${TOKEN}/balances?account.publickey=${KEY}&timestamp=eq:${TS}"

# Fire 20 concurrent requests; with pool=10 the API stalls for all other callers
for i in $(seq 1 20); do
  curl -s "$URL" &
done
wait
```

Expected result: the 11th–20th requests (and all other API requests during this window) block for up to 20 seconds or return a 500/503, demonstrating pool exhaustion.
