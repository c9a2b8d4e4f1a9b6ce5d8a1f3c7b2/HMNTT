### Title
Unbounded Query String Logged Per-Request in `TracingMiddleware` Enables Log-Disk Exhaustion DoS

### Summary
`TracingMiddleware` captures `request.URL.RequestURI()` — which includes the full query string — and writes it to the log on every non-internal request with no length truncation. Because the Rosetta HTTP server imposes no rate limiting and no per-request URL-length cap beyond Go's default `textproto` line limit (~4 KB), an unauthenticated attacker can flood the log sink at a sustained rate sufficient to exhaust available disk space and crash the node.

### Finding Description
**Exact location:** `rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47 and 52–58.

```go
// line 47
path := request.URL.RequestURI()          // includes ?query=...

// lines 52–58
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)

if internalPaths[path] {                  // path includes query string,
    log.Debug(message)                    // so /health/liveness?x=... is NOT
} else {                                  // matched → logged at Info, not Debug
    log.Info(message)
}
```

**Root cause:** `url.URL.RequestURI()` returns the raw path **plus** the literal query string (e.g. `/network/list?a=AAAA…AAAA`). The result is interpolated directly into the log message with no maximum-length guard. The `internalPaths` map is keyed on the bare path strings (`/health/liveness`, etc.); a request to `/health/liveness?<junk>` does **not** match, so it is logged at `Info` level rather than `Debug`, making it impossible to suppress via log-level tuning.

**Server configuration** (`rosetta/main.go`, lines 220–227): the `http.Server` is constructed with only timeout fields; `MaxHeaderBytes` is left at the Go default (1 MB for headers), and **no rate-limiting middleware is present**. Go's `net/textproto` enforces a 4 096-byte hard limit on the request line, so each URL is capped at ≈ 4 080 bytes — but that is still ≈ 4 KB written to disk per request.

**Exploit flow:**
1. Attacker crafts POST requests to any valid Rosetta path (e.g. `/network/list`) with a query string padded to ≈ 4 080 bytes: `POST /network/list?a=AAAA…(×4000) HTTP/1.1`.
2. Go's HTTP layer accepts the request (within the 4 096-byte line limit) and passes it to the middleware chain.
3. `TracingMiddleware` logs the full URI at `Info` level — one ≈ 4 KB line per request.
4. With no rate limiting, a single attacker machine can sustain tens of thousands of requests per second, producing tens of MB/s of log output.
5. When the log partition fills, the process can no longer write logs; depending on the OS/logger configuration this can cause the process to stall or crash, taking the Rosetta node offline.

**Why existing checks are insufficient:**
- Go's 4 096-byte request-line limit bounds the *per-request* log size but does not limit *request rate*.
- The nginx reverse proxy (`docker-compose.yml`, line 235) is optional and not present in direct-access deployments (port 5700 is exposed directly).
- No `http.MaxBytesHandler`, no token-bucket middleware, and no log-line truncation exist anywhere in the middleware chain.

### Impact Explanation
A sustained flood at 10 000 req/s × 4 KB/log-line = ~40 MB/s of log data. A 100 GB log partition is exhausted in roughly 42 minutes. Once disk is full, the Rosetta node stops functioning, denying exchange integrations access to the Hedera network. Because Rosetta is used by exchanges to validate and submit transactions, this constitutes a non-network-based DoS against a high-value financial integration layer.

### Likelihood Explanation
No authentication or API key is required to reach the Rosetta endpoint. A single host with a 1 Gbps uplink can sustain the necessary request rate. The attack is trivially scriptable (`ab`, `wrk`, or a simple loop with `curl`), fully repeatable, and leaves no persistent footprint beyond log entries that will themselves be lost when the disk fills.

### Recommendation
1. **Truncate the logged URI** before writing: cap `path` to a safe maximum (e.g. 256 characters) in `TracingMiddleware`.
2. **Add rate-limiting middleware** (e.g. `golang.org/x/time/rate` token bucket) wrapping the entire handler chain in `main.go` before `TracingMiddleware`.
3. **Set `MaxHeaderBytes`** on the `http.Server` to a small value (e.g. 8 KB) to reduce the per-request log payload.
4. **Configure log rotation with size caps** (`logrotate` or equivalent) as a defence-in-depth measure.

### Proof of Concept
```bash
# Generate a ~4000-byte query string
PADDING=$(python3 -c "print('a'*4000)")

# Flood the Rosetta node (no auth required)
wrk -t4 -c200 -d60s \
  "http://<rosetta-host>:5700/network/list?$PADDING"

# Observe log growth
watch -n1 'du -sh /var/log/rosetta/ 2>/dev/null || df -h /'
```
Each request produces a log line of the form:
```
time="..." level=info msg="<ip> POST /network/list?aaa…aaa (404) in 1.2ms"
```
At sustained load the log partition fills and the Rosetta process becomes unresponsive.
