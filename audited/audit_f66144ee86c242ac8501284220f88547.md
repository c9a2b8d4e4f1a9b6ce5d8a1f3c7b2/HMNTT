### Title
Resource Exhaustion DoS via Unbatched Historical SLOAD Queries Consuming Full `requestTimeout` Window

### Summary
An unprivileged user can submit `eth_call` requests targeting a historical block against a contract with many storage slots. For historical calls, each EVM SLOAD opcode triggers an individual, uncached database query with no batching. The `requestTimeout` enforcement in `HibernateConfiguration.StatementInspector` only fires **before** each SQL statement — not during — allowing a single request to execute thousands of DB queries and hold a server thread for the full 10-second timeout window. Flooding the service with such requests exhausts the thread pool, causing 503 errors for all legitimate users.

### Finding Description

**Code path:**

`ContractStorageReadableKVState.readFromDataSource()` branches on whether a timestamp is present: [1](#0-0) 

When a historical block is requested (`timestamp` is present), it calls `findStorageByBlockTimestamp()`. In `ContractStateServiceImpl`: [2](#0-1) 

This directly delegates to the repository with **no caching and no batching** — unlike the `findStorage()` path which uses `findStorageBatch()` with a Caffeine cache. The repository executes one individual SQL query per SLOAD: [3](#0-2) 

**Timeout enforcement flaw:**

`ContractCallContext.startTime` is set at context creation: [4](#0-3) 

The `StatementInspector` in `HibernateConfiguration` checks elapsed time **before** each SQL statement: [5](#0-4) 

The check fires only at the boundary between statements. Once a statement begins executing, it runs to completion (bounded by the DB-level `statementTimeout` of 3000 ms). The actual maximum request duration is therefore `requestTimeout (10 s) + db.statementTimeout (3 s) = 13 seconds`, not 10 seconds. More critically, within the 10-second window, thousands of individual queries can execute unimpeded.

**Gas math:**

- `maxGasLimit = 15,000,000` (from `EvmProperties`)
- Cold SLOAD costs 2,100 gas (EIP-2929)
- Maximum cold SLOADs per request: `15,000,000 / 2,100 ≈ 7,142`
- Each SLOAD on a historical call = 1 uncached DB query
- If each query takes ~1.4 ms average: `7,142 × 1.4 ms ≈ 10 seconds` — exactly hitting the timeout [6](#0-5) 

### Impact Explanation

Each malicious request holds a server thread for the full `requestTimeout` (10 s). With the default `requestsPerSecond = 500` throttle, an attacker can sustain 500 requests/second, each occupying a thread for 10 seconds — potentially 5,000 concurrent threads. A typical JVM thread pool is far smaller. Legitimate users receive HTTP 503 `Service Unavailable` responses. The attack targets the entire web3 API service, not just one contract. No authentication, no gas cost (it is a simulation), and no on-chain action is required.

### Likelihood Explanation

- **No privilege required**: `POST /api/v1/contracts/call` is a public endpoint.
- **No on-chain cost**: `eth_call` is a read-only simulation; the attacker pays nothing.
- **Trivially repeatable**: The attacker only needs to know any contract address with many storage slots (or deploy one cheaply on testnet/mainnet).
- **Historical block parameter**: Any non-`latest` block value (e.g., `"block": "earliest"`) activates the unbatched path.
- **Amplification**: The `gasPerSecond` throttle (1.5 B gas/s) allows `1,500,000,000 / 15,000,000 = 100` max-gas requests per second, each consuming 10 seconds of thread time.

### Recommendation

1. **Apply caching/batching to historical SLOAD queries**: Extend the batch-caching logic in `ContractStateServiceImpl.findStorageBatch()` to also cover `findStorageByBlockTimestamp`, keyed on `(contractId, slot, blockTimestamp)`. The `contractState` cache already has a 2-second TTL which is appropriate.

2. **Enforce timeout during in-progress statements**: Set the JDBC `queryTimeout` (or PostgreSQL `statement_timeout`) to `requestTimeout` value so that individual long-running queries are also interrupted, not just the pre-statement check.

3. **Reduce `maxGasLimit` for historical calls**: Apply a lower gas cap specifically for historical block simulations, since they are more expensive per gas unit due to the unbatched query pattern.

4. **Rate-limit by `(from, to, block)` tuple**: Add a per-caller or per-contract rate limit in `ThrottleManager` to prevent a single caller from monopolizing the thread pool.

### Proof of Concept

1. Deploy a Solidity contract that reads N distinct storage slots in a loop (e.g., `SlotContract` with 600+ slots, as referenced in the k6 load test at `tools/k6/src/web3/test/contractCallEstimateReadStorage.js`).

2. Send repeated `eth_call` requests with a historical block:
```json
POST /api/v1/contracts/call
{
  "to": "<contract_address_with_many_slots>",
  "data": "<selector_for_heavy_read_function>",
  "gas": 15000000,
  "block": "earliest",
  "estimate": false
}
```

3. Each request triggers ~7,142 individual calls to `findStorageByBlockTimestamp` (one per cold SLOAD), consuming the full 10-second `requestTimeout` window.

4. Send 50+ such requests concurrently. Observe that the server thread pool is saturated and legitimate `eth_call` requests begin returning HTTP 503.

5. The `HibernateConfiguration.StatementInspector` will throw `QueryTimeoutException` only after the 10-second window, confirming the thread was held for the full duration.
