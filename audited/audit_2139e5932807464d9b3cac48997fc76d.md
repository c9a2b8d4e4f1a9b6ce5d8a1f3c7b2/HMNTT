### Title
Historical Block Parameter Enables Slow-Query DoS via Insufficient Pre-Statement-Only `requestTimeout` Enforcement

### Summary
The `requestTimeout` guard in `HibernateConfiguration.StatementInspector` is implemented as a **pre-statement check only**: it measures elapsed wall-clock time before each new SQL statement is dispatched, but never interrupts a statement already in flight. A historical contract-call simulation against a very old block triggers multiple sequential UNION queries across `*_history` tables; on a production node with years of accumulated history each such query can approach the PostgreSQL-level `statementTimeout` (3 s). Because the pre-statement check cannot stop a running query, the actual maximum request duration is `requestTimeout + statementTimeout` (≈ 13 s), not 10 s, and the global rate-limit (500 req/s, no per-IP cap) allows an unprivileged attacker to sustain thousands of concurrent long-running DB connections, exhausting the connection pool and making historical simulation unavailable to all users.

### Finding Description

**Code path**

`ContractCallContext.run()` creates a fresh context and records `startTime = System.currentTimeMillis()`.

```
ContractCallContext.java:31
private final long startTime = System.currentTimeMillis();
```

`HibernateConfiguration.statementInspector()` registers a Hibernate `StatementInspector` that fires **before** every SQL statement:

```
HibernateConfiguration.java:31-46
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis();   // default 10 000 ms
    return sql -> {
        if (!ContractCallContext.isInitialized()) return sql;
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {
            throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
        }
        return sql;
    };
}
```

The check fires only **before** a statement is handed to JDBC. Once a statement is executing, no cancellation occurs until the PostgreSQL-level `statementTimeout` (default 3 000 ms) fires.

**Block parameter acceptance**

`BlockType.of()` accepts any non-negative `long` value with no lower-bound restriction:

```
BlockType.java:37-55  (extractNumericBlock)
long blockNumber = Long.parseLong(cleanedValue, radix);
return new BlockType(value, blockNumber);
```

`EARLIEST` maps to block 0; any numeric block that exists in `record_file` is valid. `RecordFileServiceImpl.findByBlockType` resolves it via `recordFileRepository.findByIndex(block.number())`.

**Historical state loading**

For a historical block, every EVM state read (entity, token, NFT, contract slot, custom fee, etc.) issues a UNION query against both the live table and its `*_history` counterpart filtered by `blockTimestamp`. Example from `TokenRepository`:

```
TokenRepository.java:44-60
(select * from token where token_id = ?1 and lower(timestamp_range) <= ?2)
union all
(select * from token_history where token_id = ?1 and lower(timestamp_range) <= ?2
 order by lower(timestamp_range) desc limit 1)
order by timestamp_range desc limit 1
```

On a mainnet node with years of history, these scans can be slow. A complex contract call (e.g., one that reads multiple tokens, NFTs, allowances, and contract slots) issues 10–20 such queries per request.

**Why the timeout check fails**

| t (ms) | Event |
|--------|-------|
| 0 | Request starts; `startTime` set |
| 0 | Pre-check: elapsed=0 < 10 000 → statement 1 dispatched |
| ~3 000 | `statementTimeout` cancels statement 1 (or it completes slowly) |
| ~3 000 | Pre-check: elapsed=3 000 < 10 000 → statement 2 dispatched |
| ~6 000 | Statement 2 completes |
| ~6 000 | Pre-check: elapsed=6 000 < 10 000 → statement 3 dispatched |
| ~9 000 | Statement 3 completes |
| ~9 000 | Pre-check: elapsed=9 000 < 10 000 → statement 4 dispatched |
| ~12 000 | Statement 4 completes (elapsed now 12 000 ms) |
| ~12 000 | Pre-check: elapsed=12 000 ≥ 10 000 → `QueryTimeoutException` thrown |

The request has already consumed ~12 s of wall-clock time and held a DB connection throughout. The `requestTimeout` of 10 s is never actually enforced as an upper bound.

**Rate-limiting is insufficient**

`ThrottleManagerImpl` enforces a global `requestsPerSecond` (default 500) and `gasPerSecond` bucket, but there is no per-IP or per-user limit. An attacker sustaining 500 req/s × 12 s/request = 6 000 concurrent requests, each holding a DB connection, will exhaust any realistic connection pool (typically 10–100 connections).

### Impact Explanation

An unprivileged attacker can render the `/api/v1/contracts/call` endpoint's historical-block path completely unavailable. DB connection pool exhaustion causes all subsequent historical (and potentially non-historical) contract-call requests to queue indefinitely or fail with connection-acquisition timeouts. The `QueryTimeoutException` returned to the attacker is a 500-class error that does not deter repeated submission. Severity: **High** (availability impact, no authentication required, repeatable).

### Likelihood Explanation

The attack requires only HTTP access to the public endpoint and knowledge of any valid early block number (trivially obtained from `/api/v1/blocks`). No wallet, token, or privileged credential is needed. The attacker can automate the flood with a simple script. On mainnet or testnet nodes with multi-year history the slow-query condition is reliably reproducible. Likelihood: **High**.

### Recommendation

1. **Enforce a hard wall-clock deadline at the request level**, not just before each statement. Use a `CompletableFuture`/virtual-thread with `Future.get(requestTimeout, MILLISECONDS)` wrapping the entire `ContractCallService.callContract` invocation, and cancel the underlying JDBC connection on timeout via `Statement.cancel()` or connection close.
2. **Add per-IP (or per-client) rate limiting** for historical-block requests, separate from the global gas/RPS buckets.
3. **Restrict the minimum allowed historical block age** (e.g., reject blocks older than N days) or require an explicit opt-in header for deep-history queries.
4. **Lower `db.statementTimeout`** for the web3 DB role to reduce the per-statement contribution to total request duration.
5. Consider **caching** resolved historical state for recently-queried `(blockNumber, contractAddress)` pairs to reduce repeated full-table scans.

### Proof of Concept

```bash
# 1. Find the earliest available block
EARLIEST=$(curl -s https://<mirror-node>/api/v1/blocks?order=asc&limit=1 \
  | jq '.blocks[0].number')

# 2. Craft a complex historical call (e.g., call a token precompile at genesis block)
PAYLOAD=$(cat <<EOF
{
  "block": "$EARLIEST",
  "to": "0x0000000000000000000000000000000000000167",
  "data": "0x<complex_precompile_calldata>",
  "gas": 15000000
}
EOF
)

# 3. Flood with concurrent requests (no auth required)
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "$PAYLOAD" &
done
wait

# Expected: DB connection pool exhausted; subsequent legitimate requests
# return 500 or hang indefinitely.
```
