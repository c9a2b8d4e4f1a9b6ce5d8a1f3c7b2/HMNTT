### Title
Deleted Account Data Exposed via Long-Zero EVM Address Lookup Bypassing `deleted` Filter

### Summary
The `account(input: {evmAddress: ...})` GraphQL query contains a branching lookup in `EntityServiceImpl.getByEvmAddressAndType()` that, when the supplied address is in Hedera's "long-zero" format (`0x000000000000000000000000000000000000XXXX`), falls through to `CrudRepository.findById()` — which carries no `deleted is not true` guard. Every other lookup path (`findByEvmAddress`, `findByAlias`) explicitly filters deleted entities at the SQL level. An unprivileged caller can therefore retrieve full account data (keys, memo, etc.) for any deleted account by encoding its numeric ID as a long-zero EVM address.

### Finding Description
**Code path:**

`AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → branch on long-zero format → `entityRepository.findById()`

In `EntityServiceImpl.java` lines 37-38:
```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
}
```
`findById` is the standard Spring Data `CrudRepository` method; it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

The two custom repository queries both carry the guard:
- `findByEvmAddress`: `select * from entity where evm_address = ?1 and deleted is not true`
- `findByAlias`: `select * from entity where alias = ?1 and deleted is not true`

The only filter applied after `findById` is `e.getType() == type`, which checks entity type but not deletion status.

**Root cause:** The long-zero branch was written to resolve Hedera account-num aliases to their internal entity row, but the author omitted the `deleted` check that all other branches enforce at the SQL level.

**Failed assumption:** The developer assumed `findById` would be safe because deleted accounts are "gone," but the `deleted` column is a soft-delete flag — rows remain in the table.

### Impact Explanation
Any deleted account's full record — including its admin key, memo, auto-renew settings, and balance snapshot — is readable by any unauthenticated caller who knows or can enumerate the account's numeric ID. Because Hedera account numbers are sequential and public, enumeration is trivial. This constitutes unintended information disclosure of data that the protocol explicitly marks as deleted/inaccessible. No funds are directly at risk, consistent with the Medium scope, but key material exposure could facilitate targeted attacks on accounts that were deleted but whose keys were reused.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required. The long-zero EVM address format is documented Hedera behavior, so any developer familiar with the ecosystem knows it. Account numbers are sequential and publicly observable on-chain. An attacker needs only to: know a deleted account's number, encode it as `0x` + 24 zero hex chars + 16-hex-char num, and issue a single POST request. Fully repeatable with zero privilege.

### Recommendation
Add a `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` predicate to the `findById` branch, mirroring the SQL-level guard on the other paths:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
            .filter(e -> e.getType() == type)
            .filter(e -> !Boolean.TRUE.equals(e.getDeleted()));
}
```

Alternatively, add a custom repository method `findByIdAndDeletedIsFalse(Long id)` so the guard is enforced at the SQL level, consistent with `findByEvmAddress` and `findByAlias`.

### Proof of Concept
1. Identify a deleted account with numeric ID, e.g., `1234` (num).
2. Encode as long-zero EVM address: `0x` + `000000000000000000000000` (12 zero bytes = 24 hex chars) + `00000000000004D2` (8-byte big-endian hex of 1234).
3. Send the following GraphQL request (no authentication needed):
```graphql
{
  account(input: { evmAddress: "0x000000000000000000000000000000000000000004D2" }) {
    deleted
    key
    memo
    entityId { shard realm num }
  }
}
```
4. The response returns the deleted account's full record with `deleted: true`, proving the filter is bypassed. [1](#0-0) [2](#0-1) [3](#0-2)
