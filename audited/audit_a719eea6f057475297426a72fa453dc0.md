### Title
Indefinite Historical Retrieval Loop via Continuous Message Injection Starves Live Message Delivery in Throttled Mode

### Summary
In throttled mode, `PollingTopicMessageRetriever.isComplete()` uses a "last page was full" heuristic to decide whether historical retrieval is done. An unprivileged attacker who continuously submits messages to a public topic can keep every DB poll returning exactly `maxPageSize` records, causing `isComplete()` to return `false` indefinitely. Because `TopicMessageServiceImpl` concatenates the historical flux before the live flux, live messages are never delivered to any subscriber targeting that topic for as long as the attack persists.

### Finding Description

**Exact code path and root cause**

`isComplete()` in throttled mode:

```java
// PollingTopicMessageRetriever.java lines 121-129
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();

    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }

    return limitHit;
}
``` [1](#0-0) 

The only way `isComplete()` returns `true` in throttled mode (when no limit is set) is when `pageSize.get() < maxPageSize` — i.e., the last DB page was not full. `pageSize` is reset to 0 at the start of each poll and incremented by `onNext()` for every record returned. [2](#0-1) [3](#0-2) 

In throttled mode the repeat loop is unbounded (`numRepeats = Long.MAX_VALUE`):

```java
// lines 98-101
if (throttled) {
    numRepeats = Long.MAX_VALUE;
    frequency = retrieverProperties.getPollingFrequency();   // default 2 s
    maxPageSize = retrieverProperties.getMaxPageSize();       // default 1000
}
``` [4](#0-3) 

The repeat predicate is `!context.isComplete()`:

```java
// lines 51-55
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                ...
``` [5](#0-4) 

**Why the 60-second timeout does not save the situation**

Reactor's `.timeout(Duration)` is an *idle* timeout — it fires only if no item is emitted within the window. Because the DB is continuously returning `maxPageSize` records, items are emitted on every poll (every 2 s by default), so the 60-second idle timer never expires. The retriever loops forever. [6](#0-5) [7](#0-6) 

**Live message starvation**

`TopicMessageServiceImpl.subscribeTopic()` always invokes the retriever in throttled mode and concatenates the historical flux *before* the live flux:

```java
// lines 63-73
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));
...
Flux<TopicMessage> flux = historical
        .concatWith(Flux.merge(safetyCheck, live)...);
``` [8](#0-7) 

`concatWith` does not subscribe to `live` until `historical` completes. If `historical` never completes, no live messages are ever delivered to the subscriber.

### Impact Explanation

- **Live message starvation**: Any subscriber to a targeted public topic receives historical messages indefinitely but never transitions to live delivery. Real-time consumers (e.g., applications watching for new HCS messages) are silently broken.
- **Server resource exhaustion**: Each affected subscription holds a `boundedElastic` scheduler thread and an open DB connection for the duration of the attack. Multiple concurrent subscriptions to the same topic multiply the resource drain, potentially exhausting the thread pool and DB connection pool for all users of the mirror node.
- **Severity**: High — complete denial of live message delivery for targeted topics, with secondary resource exhaustion.

### Likelihood Explanation

- On Hedera, any account can submit messages to a topic that has no `submitKey` (the common case for public topics). No special privilege is required.
- Default `maxPageSize = 1000`, `pollingFrequency = 2 s` → the retriever consumes ~500 records/second. The attacker must sustain >500 new messages/second ahead of the retriever's cursor to keep every page full.
- On mainnet this rate carries non-trivial transaction fees, reducing casual exploitation. On testnet or a private/permissioned network the cost barrier is negligible.
- The attack is fully repeatable and can be sustained indefinitely with a scripted client. A single attacker can target all current and future subscribers to a topic simultaneously.

### Recommendation

1. **Add a wall-clock deadline to the historical retrieval phase.** Track elapsed time in `PollingContext` and have `isComplete()` return `true` once a configurable maximum duration (e.g., `retrieverProperties.getTimeout()`) has elapsed, regardless of page fullness. This bounds the historical phase independently of message volume.

2. **Decouple historical completion from page fullness when no limit is set.** Consider switching to a timestamp-based completion criterion: record the subscription start timestamp and declare historical retrieval complete once the retriever's cursor has advanced past that timestamp, rather than relying solely on the page-size heuristic.

3. **Cap `numRepeats` in throttled mode** to a finite value (similar to the unthrottled `maxPolls` cap) so the loop cannot run indefinitely even if `isComplete()` never returns `true`.

### Proof of Concept

**Preconditions**
- A public Hedera topic (no `submitKey`) exists, e.g., `0.0.12345`.
- A victim subscriber connects: `subscribeTopic(filter{topicId=0.0.12345, startTime=0})`.
- Default config: `maxPageSize=1000`, `pollingFrequency=2s`, `timeout=60s`.

**Steps**

1. Attacker opens a scripted HCS client and begins submitting messages to topic `0.0.12345` at >500 messages/second (e.g., 600 msg/s).

2. The mirror node importer writes these messages to the DB. Each 2-second poll by `PollingTopicMessageRetriever` fetches exactly 1000 records (600 msg/s × 2 s poll interval = 1200 new records available; page is always full).

3. After each poll, `pageSize.get() == 1000 == maxPageSize`, so `isComplete()` returns `false`. The `repeatWhen` predicate re-schedules the next poll.

4. Because items are emitted every 2 seconds, the 60-second idle timeout never fires.

5. The `historical` flux never completes. `concatWith(live)` never subscribes to the live flux. The victim subscriber receives a continuous stream of historical messages but **zero live messages** for the entire duration of the attack.

6. Each additional victim subscriber to the same topic adds another indefinitely-running DB polling loop, multiplying resource consumption linearly.

**Verification**: Stop the attacker's message injection. Within one polling cycle (≤2 s), the next DB page returns fewer than 1000 records, `isComplete()` returns `true`, the historical flux completes, and live messages begin flowing — confirming the attack is the sole cause of the starvation.
