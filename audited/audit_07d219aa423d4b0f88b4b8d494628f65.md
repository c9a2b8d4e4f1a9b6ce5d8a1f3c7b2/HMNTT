### Title
Stale Contract Bytecode Cache (1h TTL) Causes Incorrect Gas Estimates After Contract Deletion/Self-Destruct

### Summary
The `CACHE_MANAGER_CONTRACT` cache is configured with `expireAfterAccess=1h` and has no invalidation mechanism tied to contract deletion or self-destruct events. `ContractBytecodeReadableKVState.readFromDataSource()` fetches bytecode via `ContractRepository.findRuntimeBytecode()` without checking entity deletion status, so a deleted contract's bytecode remains cached and is used for gas estimation for up to one hour. Any unprivileged Hedera user who can deploy and self-destruct a contract can trigger this condition.

### Finding Description

**Exact code locations:**

`CacheProperties.java` line 22 — the contract cache TTL: [1](#0-0) 

`ContractRepository.java` lines 16–18 — `@Cacheable` with `unless="#result==null"`, no eviction annotation: [2](#0-1) 

`ContractBytecodeReadableKVState.java` lines 39–47 — `readFromDataSource` calls `findRuntimeBytecode` with **no entity-deletion check**: [3](#0-2) 

For long-zero (standard Hedera `0.0.N`) addresses the code path is: [4](#0-3) 

**Root cause:** `findRuntimeBytecode` is cached for 1 hour after last access. There is no `@CacheEvict`, no listener, and no deletion-status guard anywhere in the bytecode lookup path. The entity cache (`expireAfterWrite=1s`) correctly reflects deletion within one second, but the bytecode cache is completely independent and has no invalidation hook.

**Proven by the existing integration test** (`ContractRepositoryTest`, lines 25–29): after `contractRepository.deleteAll()`, `findRuntimeBytecode(contract1.getId())` still returns the cached bytecode: [5](#0-4) 

**Exploit flow:**
1. Attacker deploys a contract with expensive logic (high gas cost) on the Hedera network.
2. Attacker calls `eth_estimateGas` (or any `eth_call`) on the mirror node → `findRuntimeBytecode` populates the cache with the bytecode; TTL resets to 1 hour on every access.
3. Attacker self-destructs the contract on the consensus network.
4. Within the 1-hour window, any caller (including the attacker) calls `eth_estimateGas` targeting that address.
5. Entity cache (1s TTL) correctly shows the entity as deleted/absent. However, `ContractBytecodeReadableKVState.readFromDataSource()` bypasses the entity check for long-zero addresses and calls `findRuntimeBytecode` directly, which returns the stale cached bytecode.
6. The EVM simulation executes against the stale bytecode and returns a gas estimate as if the contract still exists — producing an incorrect fee estimate.

**Why existing checks are insufficient:** The `unless="#result==null"` guard only prevents caching a null result (i.e., a contract that never existed). It does nothing for a contract that existed at cache-population time but was subsequently deleted. There is no `@CacheEvict` on any deletion path, no Spring `ApplicationEvent` listener, and no TTL short enough to bound the staleness window for bytecode.

### Impact Explanation
Gas estimates returned by `eth_estimateGas` for a self-destructed or deleted contract address will reflect the old contract's execution cost rather than the correct value (zero code / revert). Wallets and dApps that rely on these estimates to set transaction gas limits will submit transactions with incorrect gas parameters. Because the contract no longer exists on the consensus network, those transactions will fail, wasting the submitted gas fee. The window of incorrect estimates lasts up to 1 hour (or longer if the cache entry is kept alive by repeated accesses via `expireAfterAccess`). This constitutes fee estimation outside design parameters for any caller of the affected address during that window.

### Likelihood Explanation
The preconditions are minimal: any Hedera account holder (no special privilege required) can deploy a contract, trigger a cache population via a mirror node `eth_estimateGas` call, and then self-destruct the contract. The self-destruct path is exercised in the existing test suite (`SelfDestructOperationTest`, `EquivalenceFeature`), confirming it is a supported and reachable operation. The 1-hour `expireAfterAccess` window is long enough to be practically exploitable and can be extended indefinitely by periodically querying the mirror node to reset the TTL before it expires.

### Recommendation
1. **Add a `@CacheEvict` on deletion paths.** Any code path that marks a contract entity as deleted (importer's `ContractDeleteTransactionHandler`, self-destruct handling) should evict the corresponding entry from `CACHE_MANAGER_CONTRACT`.
2. **Guard bytecode lookup with entity status.** In `ContractBytecodeReadableKVState.readFromDataSource()`, check entity deletion status (via `CommonEntityAccessor` or `EntityRepository`) before returning cached bytecode, and return `null` if the entity is deleted.
3. **Reduce the TTL or switch to `expireAfterWrite`.** A 1-hour `expireAfterAccess` TTL is too long for mutable state. `expireAfterWrite` with a shorter duration (e.g., matching the entity cache's 1s) would bound the staleness window.

### Proof of Concept
```
# 1. Deploy a contract with expensive logic on Hedera testnet.
#    Record its EVM address: 0x<CONTRACT_ADDR>

# 2. Populate the mirror node bytecode cache:
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -d '{"to":"0x<CONTRACT_ADDR>","estimate":true,"data":"<expensive_selector>"}'
# → Returns a high gas estimate (e.g., 500000). Cache is now populated.

# 3. Self-destruct the contract on the consensus network via SDK or HAPI.

# 4. Wait > 1 second (entity cache expires), then immediately re-query:
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -d '{"to":"0x<CONTRACT_ADDR>","estimate":true,"data":"<expensive_selector>"}'
# → Still returns ~500000 gas (stale cached bytecode).
# Expected correct behavior: error or 0 gas (no code at address).

# 5. Repeat step 4 every <1 hour to keep the cache entry alive indefinitely.
```
