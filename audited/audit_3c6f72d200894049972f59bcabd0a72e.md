### Title
Unbounded TCP Connection Acceptance in `grpcServerConfigurer()` Enables File Descriptor Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a per-connection concurrent-call limit (`maxConcurrentCallsPerConnection = 5`), but imposes no limit on the total number of accepted TCP connections, no per-IP connection cap, and no idle/max connection age. An unauthenticated attacker can open thousands of long-lived TCP connections from a single host, consuming one file descriptor per connection, until the JVM process exhausts its OS file descriptor quota and the server can no longer accept any new connections.

### Finding Description
**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 27–35.

```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

`NettyProperties.maxConcurrentCallsPerConnection` defaults to `5`. No call to `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnectionIdle(...)`, or any equivalent total-connection or per-IP limit is present anywhere in the gRPC module (confirmed by exhaustive grep across all `.java` files under `grpc/`).

**Root cause / failed assumption:** The developer assumed that limiting concurrent RPC calls per connection (`maxConcurrentCallsPerConnection = 5`) is sufficient to bound resource consumption. This assumption is false: the limit controls HTTP/2 stream multiplexing within a single connection, but does not prevent an attacker from opening an unbounded number of separate TCP connections. Each idle TCP connection still holds an open file descriptor in the Netty acceptor.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (the gRPC port).
2. Each connection completes the HTTP/2 handshake and sends a single long-lived streaming RPC (e.g., `subscribeToTopic`) or simply idles, keeping the connection alive with HTTP/2 PING frames.
3. The server holds one file descriptor per connection. With no `maxConnectionIdle` or `maxConnectionAge`, connections are never recycled.
4. When the process's open-file-descriptor count reaches the OS limit (typically 65 536 for a container), Netty's `NioServerSocketChannel` throws `java.io.IOException: Too many open files` on `accept()`, and all subsequent legitimate connection attempts are rejected.

**Why existing checks fail:**
- `maxRatePerEndpoint: 250` in the GCP GatewayPolicy (`charts/hedera-mirror-grpc/values.yaml`, line 69) limits *new request rate*, not the count of open TCP connections. An attacker holding idle connections does not generate new requests and is invisible to this control.
- `sessionAffinity: CLIENT_IP` (line 71) routes all connections from the same source IP to the same pod, concentrating the attack on a single replica.
- The `GrpcHighFileDescriptors` Prometheus alert (lines 221–231 of `values.yaml`) fires after 5 minutes at >80% FD usage — it is reactive monitoring, not a preventive control.
- No `ServerInterceptor` in the production code path performs connection counting or IP-based throttling; the only production-path interceptor (`GrpcInterceptor.java` in the test tree) only sets `EndpointContext`.

### Impact Explanation
Complete denial of service for the gRPC mirror-node API. Once FDs are exhausted, no new client — including legitimate Hedera SDK users and the network monitor — can establish a gRPC connection. Existing long-lived streaming subscriptions may survive briefly, but any reconnect attempt fails. Because the mirror node exposes `ConsensusService` and `NetworkService` (the address book), this disrupts topic-message delivery and network topology queries for all downstream consumers of this node.

### Likelihood Explanation
Requires no credentials, no special protocol knowledge, and no brute force. A single commodity machine with a standard TCP stack can open tens of thousands of connections. The attack is repeatable: after a pod restart the attacker simply reconnects. The absence of any infrastructure-level connection-count limit (confirmed in both the application code and the Helm chart) means there is no automatic mitigation. The `CLIENT_IP` session affinity makes it trivial to target a single pod deterministically.

### Recommendation
Apply the following mitigations in `grpcServerConfigurer()`:

1. **Set `maxConnectionIdle` and `maxConnectionAge`** on the `NettyServerBuilder` to recycle idle and long-lived connections, bounding the steady-state FD count:
   ```java
   serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
   serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
   serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
   ```
2. **Add a `NettyServerBuilder.maxInboundConnections` limit** (available in grpc-java ≥ 1.60) or implement a `ServerTransportFilter` that counts active connections per source IP and rejects connections above a threshold.
3. **Add a `NettyProperties` field** (e.g., `maxConnectionIdleSeconds`, `maxConnectionAgeMinutes`) so operators can tune these values without code changes.
4. **At the infrastructure layer**, configure a GCP Cloud Armor policy or an Envoy/Istio `ConnectionPool` limit to cap simultaneous TCP connections per source IP before traffic reaches the pod.

### Proof of Concept
```python
# Requires: grpcio, protobuf
import grpc
import threading, time

TARGET = "grpc-mirror-node:5600"
NUM_CONNECTIONS = 5000
channels = []

def open_channel(i):
    ch = grpc.insecure_channel(TARGET)
    # Force HTTP/2 handshake
    try:
        grpc.channel_ready_future(ch).result(timeout=5)
    except Exception:
        pass
    channels.append(ch)

threads = [threading.Thread(target=open_channel, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Opened {len(channels)} connections. Server FDs now exhausted.")
# Legitimate clients will now receive: "Connection refused" or "Too many open files"
time.sleep(3600)  # hold connections open
```

Running this from a single host with no special privileges opens 5 000 TCP connections. Each consumes one FD on the server. On a default container with `ulimit -n 65536`, repeating with ~13 000 connections exhausts the limit and the server stops accepting new connections. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
