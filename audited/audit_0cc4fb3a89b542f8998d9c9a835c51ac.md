### Title
Unbounded Subscription Lifetime in `PollingTopicListener.listen()` Enables Resource Exhaustion by Unprivileged Users

### Summary
`PollingTopicListener.listen()` constructs a Flux chain with `RepeatSpec.times(Long.MAX_VALUE)` and no subscription-level idle or wall-clock timeout operator. Any unauthenticated client that omits both `endTime` and `limit` in a `subscribeTopic` gRPC call will hold an active polling subscription indefinitely, issuing a database query every 500 ms (default interval) for the lifetime of the connection. An attacker opening many such connections can exhaust the shared `boundedElastic` scheduler queue, the database connection pool, and gRPC server stream resources with no server-side cleanup.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` — [1](#0-0) 

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← effectively infinite
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
        // no .timeout(...) operator anywhere in this chain
        .doOnNext(context::onNext)
        .doOnSubscribe(s -> log.info(...));
```

**Root cause:** `Long.MAX_VALUE` repeats with no `.timeout()`, `.take(Duration)`, or idle-timeout operator. The Flux never self-terminates unless the subscriber cancels or an `endTime`/`limit` is provided by the caller.

**Upstream service — no forced timeout either:**

`TopicMessageServiceImpl.subscribeTopic()` only terminates the live listener stream when `endTime` is set: [2](#0-1) 

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← subscription lives forever when endTime is absent
    }
    ...
}
```

When `endTime == null`, `pastEndTime()` returns `Flux.never()`, so `.takeUntilOther(pastEndTime(...))` never fires. [3](#0-2) 

**Contrast with the retriever**, which correctly applies a timeout: [4](#0-3) 

`.timeout(retrieverProperties.getTimeout(), scheduler)` exists in `PollingTopicMessageRetriever` but is entirely absent from `PollingTopicListener`.

**Controller — cancel handler only fires on client disconnect:** [5](#0-4) 

There is no server-initiated timeout; cleanup only happens if the client sends a cancel or drops the TCP connection.

### Impact Explanation

Each open subscription with no messages being produced:
- Schedules a `topicMessageRepository.findByFilter()` DB query every 500 ms on the shared `Schedulers.boundedElastic()` pool.
- Holds a live gRPC server stream and a Reactor subscription chain in memory.
- Contributes to the shared DB connection pool load.

With `maxConcurrentCallsPerConnection = 5` (per connection, not global), an attacker opens `N` connections × 5 subscriptions each. At `N = 200` connections, 1,000 subscriptions fire 2,000 DB queries per second continuously. This can saturate the DB connection pool, starve legitimate subscribers, and degrade or crash the gRPC service — all with zero economic cost to the attacker.

### Likelihood Explanation

- **No authentication required** — the gRPC `subscribeTopic` endpoint is publicly accessible.
- **Trivial to script** — a standard gRPC client (e.g., `grpcurl`, Hedera SDK) can open hundreds of persistent subscriptions in seconds.
- **Repeatable** — the attacker simply reconnects if the nginx proxy's `grpc_read_timeout 600s` closes connections; the 600 s timeout is deployment-specific, not enforced in application code, and resets on any message receipt. [6](#0-5) 
- **Listener type dependency** — this path is active when `hiero.mirror.grpc.listener.type=POLL`; the default is `REDIS`, but POLL mode is a documented and supported configuration. [7](#0-6) 

### Recommendation

1. **Add an idle/wall-clock timeout to `PollingTopicListener.listen()`:**
   ```java
   .timeout(listenerProperties.getMaxSubscriptionDuration(),
            Mono.error(new TimeoutException("Subscription idle timeout")),
            scheduler)
   ```
   Mirror the pattern already used in `PollingTopicMessageRetriever.retrieve()`.

2. **Add a configurable `maxSubscriptionDuration` property** to `ListenerProperties` (e.g., default 10–30 minutes) so operators can tune it.

3. **Add a global concurrent-subscription cap** (not just per-connection) enforced in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge — reject new subscriptions above a configured threshold.

4. **Do not rely solely on the nginx `grpc_read_timeout`** as the only backstop; it is absent in direct-to-port deployments and is not a substitute for application-level resource management.

### Proof of Concept

```bash
# Open 5 persistent subscriptions per connection, 100 connections = 500 open subscriptions
# Each polls the DB every 500ms → 1000 DB queries/sec with zero messages produced

for i in $(seq 1 100); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# No endTime, no limit → subscriptions never self-terminate server-side
# Monitor DB connection pool saturation and scheduler thread queue depth
wait
```

Preconditions: `hiero.mirror.grpc.listener.type=POLL`, topic 0.0.1 exists, direct access to port 5600 (bypassing nginx).
Trigger: subscriptions with no `endTime`/`limit` remain open indefinitely.
Result: DB query storm, scheduler exhaustion, degraded service for legitimate subscribers.
