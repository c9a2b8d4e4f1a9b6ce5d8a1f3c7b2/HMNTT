### Title
Unauthenticated Inflight Gauge Inflation via Unbounded Concurrent `/construction/submit` Requests

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` unconditionally increments the `hiero_mirror_rosetta_request_inflight` gauge for every incoming HTTP request with no application-level rate limiting or authentication. Because the rosetta Go server applies no per-client throttling, any unauthenticated external caller can flood `/construction/submit` with thousands of concurrent requests, artificially inflating the inflight gauge and creating monitoring noise under which a real malicious transaction submission is obscured from operators relying on that gauge.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go` lines 76–83 — `MetricsMiddleware` wraps the entire router with `middleware.Instrument`, which calls `requestInflightGauge.With(labels).Inc()` on every request arrival and `.Dec()` on completion, with no gating:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,   // incremented unconditionally
        ...
    }.Wrap(next)
}
```

`rosetta/main.go` lines 217–219 — the full middleware chain assembled at startup is:

```
MetricsMiddleware → TracingMiddleware → CorsMiddleware → router
```

There is no authentication middleware, no per-IP connection limiter, and no rate limiter anywhere in this chain.

**Root cause / failed assumption:**

The design assumes that an external infrastructure layer (Traefik) will enforce connection limits before requests reach the Go process. However, `charts/hedera-mirror-rosetta/values.yaml` line 95 shows `global.middleware: false` as the default, meaning the Traefik `inFlightReq` (5 per IP) and `rateLimit` (10/s per host) middleware CRDs are **not created by default**. Even when enabled, the `inFlightReq` criterion uses `ipStrategy.depth: 1` (single IP), trivially bypassed with multiple source IPs or by accessing the rosetta port directly (bypassing Traefik entirely).

**Exploit flow:**

1. Attacker opens thousands of concurrent TCP connections to the rosetta port and sends POST requests to `/construction/submit` with a valid-length but slow-to-process body (or keeps connections open via slow-read). Each connection causes `MetricsMiddleware` to call `.Inc()` on the gauge immediately upon request arrival.
2. `ConstructionSubmit` (construction_service.go lines 332–368) calls `hiero.TransactionExecute`, which involves network I/O to the Hiero consensus node — this keeps each goroutine (and thus the gauge increment) alive for the duration of the network round-trip.
3. While the gauge reads artificially high (e.g., 5,000 inflight), the attacker submits one real malicious transaction in the same flood. Operators monitoring `hiero_mirror_rosetta_request_inflight` see a spike consistent with a DDoS event and focus on availability, not on the content of individual submissions.
4. The malicious transaction is logged at INFO level but is buried in thousands of identical log lines from the flood.

**Why existing checks fail:**

- Traefik middleware: disabled by default (`global.middleware: false`); opt-in only; bypassable via direct port access or multiple source IPs.
- No application-level authentication on `/construction/submit`: any caller can submit.
- No Go-level `http.Server` connection limit beyond OS TCP backlog.
- The `MetricsMiddleware` itself has no awareness of per-client request counts.

### Impact Explanation

An operator using `hiero_mirror_rosetta_request_inflight` as a health/anomaly signal (as the provided Prometheus rules suggest) will see a gauge value orders of magnitude above baseline. This creates two concrete harms: (1) **monitoring obfuscation** — the real malicious submission is indistinguishable from the flood in both the gauge and the log stream; (2) **resource exhaustion** — each inflight goroutine holds a connection to the Hiero consensus node, potentially exhausting the SDK client's connection pool and degrading or blocking legitimate transaction processing. Severity is **High** for deployments without Traefik middleware enabled (the default).

### Likelihood Explanation

Preconditions are minimal: no credentials, no special network position, no prior knowledge beyond the public Rosetta API spec. The attack is fully scriptable with standard HTTP tooling (e.g., `wrk`, `hey`, `curl` in parallel). The default Helm chart ships with the protection disabled. Any operator who deployed without explicitly setting `global.middleware: true` and configuring Traefik is exposed. The attack is repeatable and stateless.

### Recommendation

1. **Application-level rate limiting**: Add a per-IP connection/rate limiter directly in the Go server (e.g., `golang.org/x/time/rate` or `github.com/didip/tollbooth`) applied before `MetricsMiddleware` in the chain in `rosetta/main.go`.
2. **Enable Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95, so `inFlightReq` and `rateLimit` are active out of the box.
3. **Tighten `inFlightReq` criterion**: Use `sourceCriterion.requestHeaderName` (e.g., a forwarded client ID) in addition to IP depth to resist multi-IP bypass.
4. **Separate gauge alerting from anomaly detection**: Do not rely solely on the inflight gauge for security alerting; correlate with per-transaction log entries and the `hiero_mirror_rosetta_request_duration` histogram.

```go
func (c *constructionAPIService) ConstructionSubmit(
	_ context.Context,
	request *rTypes.ConstructionSubmitRequest,
) (*rTypes.TransactionIdentifierResponse, *rTypes.Error) {
	if !c.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
	if rErr != nil {
		return nil, rErr
	}

	hashBytes, err := hiero.TransactionGetTransactionHash(transaction)
	if err != nil {
		return nil, errors.ErrTransactionHashFailed
	}

	hash := tools.SafeAddHexPrefix(hex.EncodeToString(hashBytes))
	transactionId, _ := hiero.TransactionGetTransactionID(transaction)
	log.Infof("Submitting transaction %s (hash %s) to node %s", transactionId,
		hash, transaction.GetNodeAccountIDs()[0])

	_, err = hiero.TransactionExecute(transaction, c.sdkClient)
	if err != nil {
		log.Errorf("Failed to execute transaction %s (hash %s): %s", transactionId, hash, err)
		return nil, errors.AddErrorDetails(
			errors.ErrTransactionSubmissionFailed,
			"reason",
			fmt.Sprintf("%s", err),
		)
	}

	return &rTypes.TransactionIdentifierResponse{
		TransactionIdentifier: &rTypes.TransactionIdentifier{Hash: hash},
	}, nil
}
```
