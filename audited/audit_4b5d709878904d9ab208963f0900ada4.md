### Title
Unbounded OpenTelemetry Histogram Label Cardinality via Unmatched-Route Path Injection

### Summary
In `rest/middleware/metricsHandler.js`, the `toOpenApiPath` function falls back to the raw `req.path` value when no Express route matches a request (`req.route === null`). This raw, attacker-controlled path is then used as the `path` label in `durationHistogram.record()`, `requestTotalCounter.add()`, `requestSizeHistogram.record()`, and `responseSizeHistogram.record()`. Because the OpenTelemetry SDK accumulates a distinct in-memory data point per unique label set with no cardinality cap, an unprivileged attacker can exhaust Node.js heap memory by flooding the server with requests to unique paths, causing OOM crashes and halting metric collection.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, `toOpenApiPath`, lines 13–31:
```js
const toOpenApiPath = (req, res) => {
  let path = res.locals[requestPathLabel];   // undefined for unmatched routes

  if (!path) {
    if (!req.route) {
      path = req.path;                        // ← raw attacker-controlled value
    } else {
      path = (req.baseUrl ?? '') + req.route?.path;
    }
  }
  path = path.replace(/:([^/]+)/g, '{$1}');
  ...
  return path;
};
```

`res.locals[requestPathLabel]` is only set by route handlers that explicitly write it. For any request that does not match a registered route (resulting in a 404), `req.route` is `null` and `res.locals[requestPathLabel]` is `undefined`, so the function returns the verbatim `req.path`.

`rest/middleware/metricsHandler.js`, `res.on('finish', ...)`, lines 185–211:
```js
const path = toOpenApiPath(req, res);   // raw path for 404s
const code = String(res.statusCode);
const method = req.method;
const labels = {method, path, code};    // unique per unique path

durationHistogram.record(duration, labels);       // line 208
requestTotalCounter.add(1, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
```

Every unique `{method, path, code}` triple causes the `@opentelemetry/sdk-metrics` `MeterProvider` to allocate a new accumulation entry. The SDK imposes no default cardinality limit. There is no rate-limiting middleware anywhere in the REST server (`rest/server.js` lines 68–144 show only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, and `metricsHandler` — confirmed by grep returning zero rate-limit hits in `rest/**/*.js`).

**Root cause / failed assumption:** The code assumes `req.path` is always a small, bounded set of known API paths. This assumption fails for unmatched routes, where `req.path` is fully attacker-controlled and unbounded.

### Impact Explanation
Each unique path string creates a permanent in-memory accumulation record inside the OTel `MeterProvider` for the lifetime of the process. With ~100 000 unique paths the Node.js heap grows by hundreds of MB; with millions of paths the process is OOM-killed. A crashed REST API process stops serving block and transaction queries entirely. Even before OOM, the `exporter.collect()` call (line 161) must iterate every accumulated label set on each `/metrics` scrape, causing CPU spikes and scrape timeouts that blind the monitoring stack to block-time anomalies — directly matching the described impact of delaying block metric visibility.

### Likelihood Explanation
No authentication, API key, or rate-limiting is required to reach the metrics instrumentation path. Any internet-accessible deployment is reachable. A single attacker with a modest script (e.g., `curl` in a loop with a counter in the URL) can generate millions of unique label sets within minutes. The attack is stateless, trivially parallelisable, and leaves no persistent side-effects that would alert defenders before memory pressure becomes critical.

### Recommendation
1. **Normalize unknown paths to a sentinel**: In `toOpenApiPath`, when `req.route` is `null`, return a fixed string such as `"unknown"` or `apiPrefix + "/unknown"` instead of `req.path`. This collapses all unmatched-route requests into a single label value.
2. **Enforce a cardinality cap**: Configure the OTel `MeterProvider` with a `ViewRegistry` that applies a cardinality limit (e.g., `cardinalityLimit: 100`) to the per-route histograms and counters.
3. **Add application-level rate limiting**: Integrate `express-rate-limit` (or equivalent) before the metrics middleware in `rest/server.js` to bound the request rate per IP.

### Proof of Concept
```bash
# Flood the server with unique 404 paths — no credentials needed
for i in $(seq 1 500000); do
  curl -s "http://<mirror-node-rest>:5551/api/v1/nonexistent_path_${i}" &
done
wait

# Observe heap growth
curl http://<mirror-node-rest>:5551/swagger/metrics/ | grep nodejs_process_memory_heap_used_bytes
# Value climbs unboundedly with each batch; eventually the process is OOM-killed
```

Each request to `/api/v1/nonexistent_path_N` matches no Express route → `req.route === null` → `toOpenApiPath` returns `/api/v1/nonexistent_path_N` → `durationHistogram.record(duration, {method:'GET', path:'/api/v1/nonexistent_path_N', code:'404'})` allocates a new OTel accumulation record → heap grows without bound.
