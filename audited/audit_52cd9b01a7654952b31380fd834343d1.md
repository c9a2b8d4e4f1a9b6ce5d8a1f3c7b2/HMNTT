### Title
Unauthenticated Multi-Connection gRPC Stream Flood Exhausts DB Pool and Delays Address Book Delivery

### Summary
The `getNodes()` handler in `NetworkController` is publicly accessible with no authentication, no per-IP connection limit, and no global rate limiting. The only protection, `maxConcurrentCallsPerConnection = 5`, is scoped per TCP connection, not per IP. An attacker can open an unbounded number of TCP connections (each multiplexing up to 5 concurrent HTTP/2 streams), flooding the server with concurrent `getNodes()` calls that each hold a database connection and execute paged queries, exhausting the DB connection pool and starving legitimate clients of address book delivery.

### Finding Description

**Code path:**

`NetworkController.getNodes()` at [1](#0-0)  immediately subscribes a Reactor pipeline that calls `networkService::getNodes` with no authentication check, no per-caller rate limit, and no concurrency guard beyond what Netty enforces per connection.

`GrpcConfiguration` applies the only server-side concurrency control: [2](#0-1) 

The default value of that limit is: [3](#0-2) 

This is a **per-TCP-connection** limit enforced at the HTTP/2 stream layer. There is no limit on the number of TCP connections from a single IP, no per-IP stream quota, and no global rate limiter in the gRPC module (contrast with the web3 module's `ThrottleConfiguration` / `bucket4j` setup, which is entirely absent here).

The only registered `ServerInterceptor` is: [4](#0-3)  — it only sets an endpoint context label; it performs no rate limiting or authentication.

Each `getNodes()` call in `NetworkServiceImpl.getNodes()` executes multiple synchronous DB queries (address book timestamp lookup, node stake lookup, then paged address book entry queries with a 250 ms inter-page delay): [5](#0-4) 

Each page holds a DB transaction for its duration: [6](#0-5) 

**Root cause:** `maxConcurrentCallsPerConnection` limits HTTP/2 streams per TCP connection, but there is no limit on the number of TCP connections from a single source IP. An attacker opens N connections × 5 streams = N×5 concurrent `getNodes()` calls, each consuming a DB connection for the full paged-query duration.

**Failed assumption:** The design assumes `maxConcurrentCallsPerConnection` is a sufficient DoS guard. It is not — it is trivially bypassed by opening additional TCP connections, which gRPC-Netty accepts without bound.

### Impact Explanation

Each concurrent `getNodes()` call holds a DB connection for the duration of all paged queries (pageSize=10, pageDelay=250ms per page). With a bounded DB connection pool, a flood of concurrent calls exhausts the pool. Legitimate clients' `getNodes()` calls queue behind the attacker's calls, delaying address book delivery well beyond normal response time. The `pageDelay` of 250ms per page means each call is long-lived, amplifying pool exhaustion. No authentication or privilege is required.

### Likelihood Explanation

Any unprivileged external user can execute this attack with a standard gRPC client library (e.g., `grpcurl` in a loop, or a simple Go/Java client opening N channels). The endpoint is documented as publicly accessible. No credentials, tokens, or special network position are required. The attack is repeatable and scriptable.

### Recommendation

1. Add a per-IP TCP connection limit at the Netty level (`serverBuilder.maxConnectionsPerIp(...)` or via an ingress/load-balancer policy).
2. Add a global and/or per-IP rate limiter (e.g., bucket4j, as already used in the web3 module) as a `ServerInterceptor` applied to `getNodes()`.
3. Consider adding a server-wide `maxConcurrentCalls` (not just per-connection) via a custom interceptor that tracks in-flight `getNodes()` calls globally.
4. Enforce a short deadline/timeout on `getNodes()` streams so stale attacker streams are reaped quickly.

### Proof of Concept

```bash
# Open 200 TCP connections, each sending 5 concurrent getNodes() streams
# (1000 concurrent DB-holding calls total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <target>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes &
  done
done
wait
```

With the default DB connection pool size, legitimate `getNodes()` calls will queue or fail with pool-exhaustion errors, delaying address book delivery for all other clients.

            ServerCall<ReqT, RespT> call, Metadata headers, ServerCallHandler<ReqT, RespT> next) {

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
