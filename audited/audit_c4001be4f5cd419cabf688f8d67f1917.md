### Title
Missing Contract ID Filter Guard in `getContractLogsByTimestamps()` Enables Timestamp-Only Full Scan via Public API

### Summary
When `contractDetails.contractIds` is an empty array (sourced from a NULL `contract_ids` DB column), `getContractLogsByTimestamps()` silently omits the `contract_id` filter from its WHERE clause, issuing a query that scans all contract logs matching only the timestamp. Any unauthenticated user who can identify a `contract_transaction` row with a NULL `contract_ids` column can repeatedly trigger this unfiltered query via the public REST API, degrading database performance.

### Finding Description

**Exact code path:**

`rest/service/contractService.js`, `getContractLogsByTimestamps()`, lines 395–414:

```js
async getContractLogsByTimestamps(timestamps, involvedContractIds = []) {
    ...
    const conditions = [`${ContractLog.CONSENSUS_TIMESTAMP} ${timestampsOpAndValue}`];
    if (involvedContractIds.length) {                                          // line 405
      conditions.push(`${ContractLog.CONTRACT_ID} in (${involvedContractIds.join(',')})`);
    }
    // if involvedContractIds is [], only timestamp filter is applied
    const whereClause = `where ${conditions.join(' and ')}`;
```

**Root cause:** The guard `if (involvedContractIds.length)` at line 405 is falsy for an empty array, so the `contract_id IN (...)` predicate is never appended. The resulting SQL becomes:

```sql
WHERE consensus_timestamp = $1
-- no contract_id filter
```

**How `involvedContractIds` becomes `[]`:**

`rest/model/contractTransaction.js`, line 16:
```js
this.contractIds = contractTransaction[ContractTransaction.CONTRACT_IDS] || [];
```
If the `contract_ids` column in the `contract_transaction` table is NULL for a row, the model produces `contractIds = []`. The `|| []` fallback is explicit, confirming NULL is a known possible DB state.

**Call chain from public API to vulnerable function:**

1. `GET /api/v1/contracts/{contractId}/results/{timestamp}` → `getContractResultsByTimestamp` (contractController.js:1002)
2. → `getInvolvedContractsByTimestampAndContractId(timestamp, contractId)` (line 1011) — fetches `contractDetails`
3. → `getDetailedContractResults(contractDetails, contractId)` (line 1017)
4. → `ContractService.getContractLogsByTimestamps(contractDetails.consensusTimestamp, contractDetails.contractIds)` (contractController.js:1213)
5. → `involvedContractIds = []` → no contract_id filter → full timestamp scan

**Why existing checks are insufficient:**

The only guard before `getDetailedContractResults` is:
```js
if (!contractDetails) { throw new NotFoundError(); }   // contractController.js:1012
```
This only blocks non-existent transactions. It does not check whether `contractDetails.contractIds` is empty, so a valid transaction with a NULL `contract_ids` column passes through unchecked.

### Impact Explanation
The unfiltered query `WHERE consensus_timestamp = $1` on the `contract_log` table forces a full scan of all contract log rows for the given timestamp(s), bypassing the intended `contract_id` scoping. On a busy network, a single timestamp can correspond to many contract log rows across many contracts. An attacker can repeatedly issue the same request to amplify DB load, causing query latency spikes and degrading service for all users. No data is leaked or modified; the impact is purely availability/performance (griefing).

### Likelihood Explanation
The API endpoint is public and requires no authentication or privileges. The only precondition is discovering a `contract_transaction` row where `contract_ids` is NULL — a state the codebase explicitly anticipates (the `|| []` fallback). Such rows can arise from failed transactions, ingestion edge cases, or schema migrations. An attacker can enumerate valid `contractId`/`timestamp` pairs from the public ledger and probe until they find one that triggers the unfiltered path. The attack is trivially repeatable with a simple HTTP client.

### Recommendation
Add an explicit guard in `getContractLogsByTimestamps()` that refuses to execute when `involvedContractIds` is empty, rather than silently degrading to a timestamp-only scan:

```js
async getContractLogsByTimestamps(timestamps, involvedContractIds = []) {
  if (!involvedContractIds.length) {
    return [];   // or throw an internal error — never issue an unscoped query
  }
  ...
}
```

Alternatively, enforce the same guard in `getDetailedContractResults` before calling the function:

```js
// contractController.js:1213
if (!contractDetails.contractIds?.length) {
  throw new NotFoundError();
}
```

Apply the same pattern to the parallel `getContractResultsByTimestamps()` and `getContractStateChangesByTimestamps()` calls, which share the identical `if (involvedContractIds.length)` pattern.

### Proof of Concept

**Precondition:** Identify a `contract_transaction` row where `contract_ids IS NULL`. This can be found by querying the mirror node DB directly (if accessible) or by probing the API for transactions that return results but exhibit anomalously slow response times.

**Steps:**
1. Find a valid `contractId` (e.g., `0.0.12345`) and `timestamp` (e.g., `1234567890.000000001`) that exist in `contract_transaction` with a NULL `contract_ids` column.
2. Issue repeated requests:
   ```
   GET /api/v1/contracts/0.0.12345/results/1234567890.000000001
   ```
3. Each request causes `getContractLogsByTimestamps` to execute:
   ```sql
   SELECT ... FROM contract_log cl
   LEFT JOIN entity e ON e.id = cl.contract_id
   WHERE cl.consensus_timestamp = 1234567890000000001
   ORDER BY cl.consensus_timestamp, cl.index
   ```
   with no `contract_id` filter — scanning all contract logs at that timestamp.
4. Repeat at high frequency to saturate DB query capacity.

**Expected result:** DB query time increases proportionally to the number of contract log rows at the targeted timestamp; service degradation is observable for concurrent users.
