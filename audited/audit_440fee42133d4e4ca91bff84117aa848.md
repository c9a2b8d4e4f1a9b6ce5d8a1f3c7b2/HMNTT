### Title
Cache Key Bypass via Unnormalized Query Parameter Ordering in `cacheKeyGenerator`

### Summary
`cacheKeyGenerator` in `rest/middleware/responseCacheHandler.js` derives the Redis cache key by MD5-hashing `req.originalUrl` verbatim, with no query-parameter normalization. Because parameter order is part of the hash input, semantically identical requests with different parameter orderings (e.g., `?limit=25&order=asc` vs `?order=asc&limit=25`) produce distinct cache keys, causing every such variant to bypass the cache and trigger a fresh database read. The code comment at line 149 explicitly acknowledges this gap, referencing issue 9113, and the existing `requestNormalizer` is never wired into the cache path.

### Finding Description
**Exact code location:** `rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator`, lines 151–153.

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

`req.originalUrl` is the raw URL string as received from the client. Its query string is never sorted or normalized before hashing.

**Root cause / failed assumption:** The developer assumed (and documented as a known gap) that the cache key would eventually be derived from a normalized URL. Until that is done, the key is order-sensitive. The `normalizeRequestQueryParams` function in `rest/middleware/requestNormalizer.js` (lines 35–59) already sorts sortable parameters and injects defaults, but a `grep` across the entire repository confirms it is only referenced in its own test file (`rest/__tests__/middleware/requestNormalizer.test.js`) — it is never imported or called in `server.js` or `responseCacheHandler.js`.

**Middleware pipeline (server.js lines 94–98):**
```
requestLogger → authHandler → [metricsHandler] → responseCacheCheckHandler → <route handlers>
```
`responseCacheCheckHandler` is invoked before any route handler, and no normalization step precedes it. The `requestQueryParser` (line 60) only lowercases keys and canonicalizes a few values; it does not sort parameters or rewrite `req.originalUrl`.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts?limit=25&order=asc` → cache miss → DB read → response stored under key `K1 = md5("/api/v1/accounts?limit=25&order=asc")-v1`.
2. Attacker immediately sends `GET /api/v1/accounts?order=asc&limit=25` → `cacheKeyGenerator` produces `K2 = md5("/api/v1/accounts?order=asc&limit=25")-v1` ≠ K1 → cache miss → second DB read → response stored under K2.
3. If the DB state changed between steps 1 and 2 (a new account was indexed), the two responses differ for what is semantically the same query.
4. For an endpoint with N sortable parameters, N! orderings each produce a unique cache key, multiplying DB load and Redis memory consumption linearly.

**Why existing checks are insufficient:**
- `requestQueryParser` (`rest/middleware/requestHandler.js` lines 38–69) lowercases keys and merges duplicates but does not sort parameters and does not rewrite `req.originalUrl`.
- `normalizeRequestQueryParams` sorts parameters but is never called in the request pipeline.
- There is no deduplication or canonicalization step between the client and `cacheKeyGenerator`.

### Impact Explanation
In a Hedera mirror-node context, the cache exists to serve consistent, point-in-time snapshots of ledger state. Cache bypass via parameter reordering means:

1. **Inconsistent record views:** Two clients querying identical data with different parameter orderings may receive responses reflecting different ledger states if any indexing occurred between the two DB reads. For financial/audit use cases this is a correctness violation.
2. **Cache pollution / Redis memory exhaustion:** For an endpoint with 5 sortable parameters, 120 orderings each occupy a separate Redis slot. An attacker can systematically enumerate orderings to fill Redis, evicting legitimate entries and degrading cache hit rates for all users.
3. **Amplified DB load (DoS):** Each unique ordering forces a DB read. A single attacker script cycling through orderings can multiply effective query load by N! with no rate-limiting specific to this pattern.

Severity: **Medium–High**. The cache is explicitly described as a performance and consistency layer; bypassing it degrades both.

### Likelihood Explanation
- **No authentication required.** All mirror-node REST endpoints (`/api/v1/accounts`, `/api/v1/transactions`, etc.) are publicly accessible GET endpoints.
- **Trivial to exploit.** HTTP clients (curl, Python requests, browser) allow arbitrary query-string ordering. No special tooling is needed.
- **Repeatable and scriptable.** A loop over parameter permutations is a one-liner. The attacker does not need to know internal state.
- **Self-documenting gap.** The comment in the source code referencing issue 9113 tells any reader exactly where the weakness is.

### Recommendation
Wire `normalizeRequestQueryParams` into the cache key derivation path. The simplest fix is to call it inside `cacheKeyGenerator` before hashing:

```js
import {normalizeRequestQueryParams} from './requestNormalizer.js';

const cacheKeyGenerator = (req) => {
  const normalizedUrl = normalizeRequestQueryParams(
    req.openapi?.openApiRoute,
    req.path,
    req.query
  );
  return crypto.createHash('md5').update(normalizedUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

This resolves issue 9113 and ensures that `?limit=25&order=asc` and `?order=asc&limit=25` map to the same cache key. Additionally, consider adding a Redis key-count or memory-usage guard to limit cache pollution from unknown/unnormalized paths.

### Proof of Concept
```bash
# Step 1 – prime the cache with ordering A
curl -s "https://<mirror-node>/api/v1/accounts?limit=25&order=asc" > resp1.json

# Step 2 – request the same data with ordering B (different cache key)
curl -s "https://<mirror-node>/api/v1/accounts?order=asc&limit=25" > resp2.json

# Step 3 – if ledger state changed between the two requests, diff will be non-empty
diff resp1.json resp2.json

# Step 4 – amplify DB load by cycling all permutations (5 params → 120 keys)
python3 - <<'EOF'
import itertools, requests
params = [("limit","25"),("order","asc"),("balance","true"),("account.id","gt:0"),("result","success")]
for perm in itertools.permutations(params):
    qs = "&".join(f"{k}={v}" for k,v in perm)
    r = requests.get(f"https://<mirror-node>/api/v1/accounts?{qs}")
    print(r.headers.get("x-cache","MISS"), qs)
EOF
# Every permutation returns a cache MISS and triggers a fresh DB read.
```
