### Title
Global Rate Limit Exhaustion via Unauthenticated Requests Enabling DoS for All Users

### Summary
The `rateLimitBucket` in `ThrottleManagerImpl` is a single, process-wide token bucket with no per-IP or per-user partitioning. Any unauthenticated caller can consume all available tokens at the configured `requestsPerSecond` rate (default 500/s), causing every subsequent request from any user to receive `ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED)` until the bucket refills one second later. This is continuously repeatable, making the denial of service permanent for as long as the attacker sustains the flood.

### Finding Description
**Code path:**

- `ThrottleConfiguration.java` lines 24–32: a single Spring `@Bean` named `rateLimitBucket` is created with capacity = `requestsPerSecond` (default 500) and a greedy refill of the same amount per second. One instance is shared across all threads and all callers.
- `ThrottleManagerImpl.java` lines 37–39: every incoming `ContractCallRequest` calls `rateLimitBucket.tryConsume(1)`. If the bucket is empty, `ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED)` is thrown immediately, with no fallback, queuing, or per-source check.

**Root cause:** The design assumes the aggregate request rate across *all* clients will stay below `requestsPerSecond`. There is no per-IP, per-account (`from` field), or per-session bucket. The `from` field in `ContractCallRequest` is only used for optional `RequestProperties` filter matching (lines 44–48), never for rate-limit partitioning.

**Exploit flow:**
1. Attacker opens N concurrent HTTP connections (no authentication required).
2. Attacker sends `POST /api/v1/contracts/call` at ≥500 req/s (trivially achievable with a single machine and tools like `wrk` or `hey`).
3. All 500 tokens are consumed within the first millisecond of each second.
4. Every request from every other user for the remaining ~999 ms of that second hits the empty bucket and receives HTTP 429 / `ThrottleException`.
5. Attacker repeats continuously; the bucket refills every second but is immediately re-exhausted.

**Why existing checks are insufficient:**
- The `RequestProperties` filter/THROTTLE/REJECT mechanism (lines 44–48) is opt-in, requires operator configuration, and still operates on the same shared global bucket.
- The `gasLimitBucket` check (line 40) is a secondary guard that is never reached when the rate bucket is empty.
- There is no IP-level or identity-level isolation anywhere in `ThrottleManagerImpl` or `ThrottleConfiguration`.

### Impact Explanation
Any user relying on the web3 JSON-RPC endpoint (e.g., `eth_call`, `eth_estimateGas`) to simulate or validate contract interactions is completely blocked for the duration of the attack. Because the mirror node's web3 layer is the standard EVM-compatible interface for dApps and wallets on Hedera, this effectively makes the service unavailable to all legitimate users simultaneously. Severity is **High** for availability; the attack requires zero privileges and zero cost beyond network bandwidth.

### Likelihood Explanation
The attack requires no credentials, no special knowledge of the target, and no on-chain resources. A single commodity server can sustain 500+ HTTP requests per second indefinitely. The endpoint is publicly reachable by design. The attack is trivially scriptable, repeatable every second, and leaves no persistent state to clean up. Likelihood is **High**.

### Recommendation
Replace the single global bucket with per-source rate limiting:
1. Extract the caller's IP address (or `X-Forwarded-For` header behind a proxy) at the HTTP layer and maintain a per-IP `Bucket` (e.g., via a `ConcurrentHashMap<String, Bucket>` with expiry, or a distributed cache).
2. Alternatively, enforce per-IP limits at the ingress layer (API gateway / reverse proxy) *before* requests reach `ThrottleManagerImpl`.
3. If a global cap is still desired, layer it on top of per-source limits rather than as the sole guard.
4. Consider adding a per-`from`-address sub-limit so that even spoofed or rotated IPs cannot collectively exhaust the global budget faster than the per-source limit allows.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg1/wrk) or equivalent load tool
# Target: mirror-node web3 endpoint

cat > payload.json <<'EOF'
{"from":"0x0000000000000000000000000000000000000001",
 "to":"0x0000000000000000000000000000000000000002",
 "gas":21000,"data":"0x"}
EOF

# Attacker terminal: flood at >500 req/s
wrk -t8 -c200 -d60s \
    -s <(echo 'wrk.method="POST"; wrk.body=io.open("payload.json"):read("*a"); wrk.headers["Content-Type"]="application/json"') \
    http://<mirror-node-host>/api/v1/contracts/call

# Victim terminal (concurrent): observe all requests rejected
curl -s -X POST http://<mirror-node-host>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"from":"0xlegitimateuser...","to":"0x...","gas":21000,"data":"0x"}'
# Response: HTTP 429 {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
