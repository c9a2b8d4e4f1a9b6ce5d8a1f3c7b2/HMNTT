### Title
Liveness Probe Decoupled from Transaction Submission — Indefinite Silent Failure Window

### Summary
The heartbeat goroutine in `pinger/main.go` runs entirely independently of the transaction submission loop. When `submitWithRetry` fails, errors are only logged; the heartbeat continues touching `/tmp/alive` every 15 seconds. The liveness check in `pinger/cmd/healthcheck/main.go` only verifies file freshness, never transaction health, so the pod is never restarted and the gap in Hashgraph history is **indefinite**, not merely 2 minutes.

### Finding Description

**Heartbeat goroutine** — `pinger/main.go` lines 28–39:
```go
go func() {
    t := time.NewTicker(15 * time.Second)
    defer t.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-t.C:
            _ = os.WriteFile("/tmp/alive", []byte(...), 0644)
        }
    }
}()
```
This goroutine has no awareness of transaction outcomes. It writes `/tmp/alive` unconditionally every 15 s as long as the process is alive.

**Transaction loop** — `pinger/main.go` lines 57–70:
```go
case <-ticker.C:
    if err := submitWithRetry(ctx, client, cfg); err != nil {
        if errors.Is(err, context.Canceled) { return }
        log.Printf("transfer failed: %v", err)   // ← only logged, no health signal
    }
```
Persistent `submitWithRetry` failures are silently absorbed. There is no mechanism to stop the heartbeat, delete `/tmp/alive`, or otherwise signal degraded health.

**Liveness check** — `pinger/cmd/healthcheck/main.go` lines 27–38:
```go
fi, err := os.Stat("/tmp/alive")
...
age := time.Since(fi.ModTime())
if age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)
```
The only criterion is file age. As long as the heartbeat goroutine is alive (which it will be, because a goroutine panic in Go crashes the whole process — the real failure mode is the transaction loop returning errors, not panicking), the file stays fresh and the probe always returns 0.

**Root cause**: The failed assumption is that a live heartbeat implies live transaction submission. These are two independent code paths with no coupling.

### Impact Explanation
Every tick of `cfg.interval` (default 1 s) that fails to submit a transaction is a missing entry in Hashgraph history. Because the liveness probe never fires, Kubernetes never restarts the pod. The gap accumulates for as long as the underlying failure condition persists — hours or days — with no automated remediation. This directly undermines the integrity guarantee the pinger exists to provide: a continuous, verifiable record of activity on the ledger.

### Likelihood Explanation
No privileged access to the pinger process is required. Any condition that causes `submitWithRetry` to exhaust its retries (network partition to the Hashgraph node, operator account drained of hbar, SDK-level TLS failure, DNS poisoning of the node endpoint) triggers the silent failure. These conditions can arise naturally or be induced by a network-adjacent attacker. The default `maxRetries=10` with exponential backoff capped at 30 s means a sustained outage of ~5 minutes per tick exhausts retries; after that every tick fails and is only logged.

### Recommendation
Couple the liveness signal to transaction health. Two concrete options:

1. **Delete or not-touch `/tmp/alive` on consecutive failures**: In the transaction loop, track a consecutive-failure counter; after N failures, remove `/tmp/alive` so the 2-minute window expires and the probe fires.
2. **Write a separate health file only on success**: Replace the time-based heartbeat with a file written (or `os.Chtimes`-updated) only when `submitWithRetry` returns `nil`. The 2-minute staleness window then becomes a meaningful SLO boundary.

Either approach ensures the liveness probe reflects actual transaction submission health rather than mere process liveness.

### Proof of Concept

1. Deploy the pinger with a valid config pointing to a Hashgraph testnet node.
2. Confirm `/tmp/alive` is being updated and the liveness probe returns 0.
3. Block outbound traffic from the pod to the Hashgraph node port (e.g., `iptables -A OUTPUT -p tcp --dport 50211 -j DROP` from within the pod's network namespace, or via a network policy — no root on the pinger process required if a network policy is applied externally).
4. Observe `transfer failed` log lines every `cfg.interval`; `submitWithRetry` exhausts retries each tick.
5. Run `healthcheck live` every 10 s: it returns 0 indefinitely because the heartbeat goroutine is unaffected.
6. Confirm via the mirror node REST API that no new transactions from the operator account appear after step 3 — the gap in Hashgraph history grows without bound while the pod remains "healthy." [1](#0-0) [2](#0-1) [3](#0-2)
