### Title
Unrestricted Combination of `spender.id` IN-List and Range Filters Enables Maximum-Cost DB Queries in `getAccountCryptoAllowances`

### Summary
`CryptoAllowanceController.extractCryptoAllowancesQuery` accepts simultaneous `spender.id` equality (IN-list) and range (`gte`/`lte`) filters without any mutual-exclusion check, unlike `TokenAllowanceController` which calls `validateBoundsRange`. An unauthenticated attacker can craft a single request with up to 98 equality values plus a full range span plus `order=desc` and `limit=100`, forcing the database to evaluate a compound `spender IN (...98 values...) AND spender >= X AND spender <= Y` predicate with a reverse-ordered index scan at maximum result size on every request.

### Finding Description

**Code path:**

`rest/controllers/cryptoAllowanceController.js`, `extractCryptoAllowancesQuery`, lines 26–68.

In the filter loop (lines 33–57), every `spender.id` filter is routed through `updateConditionsAndParamsWithInValues`:
- `eq` operators accumulate into `spenderInValues[]`
- Range operators (`gt`/`gte`/`lt`/`lte`) are pushed directly into `conditions` and `params`

After the loop (line 59), `updateQueryFiltersWithInValues` appends `spender IN ($4, $5, ..., $101)` to `conditions`. There is **no call to `validateBoundsRange` or `validateBounds`**, so both a range condition and an IN-list condition coexist in the final WHERE clause.

The generated SQL for the worst-case request is:
```sql
SELECT * FROM crypto_allowance
WHERE owner = $1
  AND spender >= $2          -- range lower bound
  AND spender <= $3          -- range upper bound
  AND spender IN ($4 ... $101)  -- 98 equality values
  AND amount > 0
ORDER BY spender DESC
LIMIT $102
```

**Root cause / failed assumption:** `buildAndValidateFilters` (via `buildFilters`, `rest/utils.js` line 1241) only enforces that the total count of repeated `spender.id` values does not exceed `config.query.maxRepeatedQueryParameters` (default: 100). It does **not** prevent the semantically invalid combination of range + equality filters. `TokenAllowanceController` correctly calls `this.validateBounds(bounds)` (line 49 of `tokenAllowanceController.js`) which would throw `InvalidArgumentError("Can't support both range and equal for ...")`, but `CryptoAllowanceController` has no equivalent guard.

### Impact Explanation

Every such request forces the database to:
1. Perform a reverse-ordered index scan on `(owner, spender)` (DESC)
2. Evaluate a 98-element IN predicate on every candidate row
3. Apply the range predicate simultaneously
4. Return up to 100 rows

While the 20 s statement timeout (`db.pool.statementTimeout`) and `LIMIT 100` bound the worst case per query, an attacker sending these requests in a tight loop from multiple clients saturates the DB connection pool (default: 10 connections) and inflates per-query CPU/memory cost, degrading response times for all other users. No authentication is required.

### Likelihood Explanation

The endpoint `/api/v1/accounts/:id/allowances/crypto` is public. The attack requires only a valid account ID (trivially obtained from any block explorer or prior API call) and a crafted query string. It is fully scriptable, repeatable at will, and requires zero on-chain funds or privileged access. The `maxRepeatedQueryParameters = 100` limit is the only throttle, and the attack fits exactly within it (98 eq + 1 gte + 1 lte = 100 total).

### Recommendation

Add a mutual-exclusion check in `extractCryptoAllowancesQuery` mirroring `TokenAllowanceController.validateBounds`. Specifically, after the filter loop, detect when both `spenderInValues.length > 0` and any range condition exists in `conditions`, and throw `InvalidArgumentError("Can't support both range and equal for spender.id")`. Alternatively, refactor `CryptoAllowanceController` to use the `Bound`-based pattern already used by `TokenAllowanceController`, which enforces this invariant automatically via `validateBoundsRange`.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1000/allowances/crypto?\
  spender.id=eq:1&spender.id=eq:2&...&spender.id=eq:98\
  &spender.id=gte:1\
  &spender.id=lte:9999999999\
  &order=desc\
  &limit=100
```

Steps:
1. Obtain any valid account ID (e.g., `0.0.1000`) from the public API.
2. Construct the URL above with 98 `spender.id=eq:N` parameters (N = 1..98), one `spender.id=gte:1`, one `spender.id=lte:9999999999`, `order=desc`, `limit=100`. Total `spender.id` occurrences = 100, exactly at the allowed maximum.
3. Send the request — no authentication header required.
4. Observe the server accepts it (no 400 error) and issues the compound SQL query described above.
5. Repeat in a loop from multiple clients to exhaust the DB connection pool (max 10 connections by default) and degrade service for legitimate users.
