### Title
Unauthenticated Timestamp-Triggered DB Query Amplification via `getTsQuery()` / `getAccountBalanceTimestampRange()` with No Application-Level Rate Limiting

### Summary
Any unauthenticated external user can supply a `timestamp` query parameter to `GET /api/v1/balances`, which unconditionally causes `getTsQuery()` to invoke `getAccountBalanceTimestampRange()`, executing a live SQL query against the `account_balance` table filtered on the treasury account. Because the REST API layer has zero application-level rate limiting, an attacker can flood the endpoint with concurrent requests carrying distinct timestamp values, generating a sustained stream of index-range scans that compete with the importer's write path for DB resources.

### Finding Description

**Exact code path:**

`getBalances()` (line 113–114) calls `getTsQuery(tsQuery, tsParams)` whenever the request contains any `timestamp` filter:

```js
if (tsQuery) {
  const tsQueryResult = await getTsQuery(tsQuery, tsParams);
``` [1](#0-0) 

`getTsQuery()` (lines 278–292) unconditionally delegates to `getAccountBalanceTimestampRange()`:

```js
const getTsQuery = async (tsQuery, tsParams) => {
  const {lower, upper} = await getAccountBalanceTimestampRange(tsQuery, tsParams);
``` [2](#0-1) 

`getAccountBalanceTimestampRange()` (lines 167–197) builds and executes a live DB query against `account_balance` with the treasury account hardcoded as `$1`:

```js
let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
...
const {rows} = await pool.queryQuietly(query, params);
``` [3](#0-2) 

`getOptimizedTimestampRange()` (lines 270–273) does clamp the range to at most two monthly partitions, which is a partial mitigation:

```js
const nowInNs = utils.nowInNs();
const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);
``` [4](#0-3) 

**Root cause / failed assumption:**

`server.js` registers the balances route with no rate-limiting middleware:

```js
app.getExt(`${apiPrefix}/balances`, balances.getBalances);
``` [5](#0-4) 

The entire middleware stack (`cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional metrics, optional cache) contains no per-IP or global request-rate throttle for this endpoint. [6](#0-5) 

The only query-level guard is `maxRepeatedQueryParameters` in the query parser, which limits repeated parameters *within a single request*, not the request rate itself. [7](#0-6) 

The throttle infrastructure that exists (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` Java module and is entirely absent from the Node.js REST service.

**Exploit flow:**

1. Attacker sends `N` concurrent HTTP GET requests to `/api/v1/balances?timestamp=lte:<T_i>` where each `T_i` is a distinct nanosecond timestamp near `now`.
2. Each request passes `validateReq`, parses a valid `tsQuery`, and enters the `if (tsQuery)` branch.
3. Each request independently calls `getAccountBalanceTimestampRange()`, which fires a DB query: `SELECT consensus_timestamp FROM account_balance WHERE account_id = <treasury> AND consensus_timestamp >= <lower> AND consensus_timestamp <= <upper> ORDER BY consensus_timestamp DESC LIMIT 1`.
4. Because the `account_balance` table is partitioned by month and the treasury account is present in every snapshot, each query performs an index range scan across up to two monthly partitions.
5. With `N` concurrent requests, `N` concurrent DB connections are consumed from the pool, and `N` concurrent index scans run simultaneously, competing with the importer's write transactions for I/O and lock resources.

### Impact Explanation

The mirror node's PostgreSQL database is shared between the REST API read path and the importer write path. Sustained high-concurrency index scans on `account_balance` consume connection pool slots, buffer pool pages, and I/O bandwidth. This degrades the importer's ability to commit new balance snapshots and transaction records, causing the mirror node to fall behind the network. Downstream consumers (wallets, explorers, dApps) relying on the mirror node for transaction confirmation status lose visibility into new transactions. The `account_balance` table is the most frequently queried table in this path, making it the highest-value target for this amplification.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only a standard HTTP client. The endpoint is publicly documented in the OpenAPI spec. [8](#0-7) 

Varying the timestamp value per request is trivial (e.g., `timestamp=lte:<unix_ns - i>`). The attack is fully scriptable, repeatable, and can be distributed across multiple source IPs to bypass any infrastructure-level IP-based rate limiting. No special knowledge of the system internals is required beyond reading the public API documentation.

### Recommendation

1. **Add application-level rate limiting** to the Node.js REST server for the `/api/v1/balances` endpoint (and all timestamp-filtered endpoints). Use a middleware such as `express-rate-limit` with per-IP and global limits before the route handler.
2. **Add a DB-query result cache** (Redis is already optionally wired in `server.js`) specifically for `getAccountBalanceTimestampRange()` keyed on the computed `(lowerBound, upperBound)` tuple, so repeated or near-identical timestamp queries are served from cache rather than hitting the DB.
3. **Enforce a minimum timestamp granularity** (e.g., round user-supplied timestamps to the nearest 15-minute snapshot boundary) to collapse many distinct attacker-supplied values into a small set of cache-hit keys.
4. **Set a hard concurrency limit** on the DB connection pool for read queries, separate from the write path used by the importer.

### Proof of Concept

```bash
# Send 500 concurrent requests with distinct timestamps near now
NOW_NS=$(date +%s%N)
for i in $(seq 1 500); do
  TS=$((NOW_NS - i * 1000000))  # each 1ms apart, all within current month
  curl -s "https://<mirror-node-host>/api/v1/balances?timestamp=lte:$(python3 -c "print($TS / 1e9)")" &
done
wait
```

Each of the 500 concurrent requests independently triggers a `SELECT` on `account_balance` with `account_id = <treasury>` and a range within the current two monthly partitions. Monitor PostgreSQL `pg_stat_activity` to observe 500 simultaneous active queries and the corresponding spike in `blks_read`, `blk_read_time`, and importer write latency.
