### Title
Blocking Kubernetes API Calls on Netty I/O Thread via Unauthenticated Health Endpoint

### Summary
`ReleaseHealthIndicator.createHelmReleaseHealth()` wraps two synchronous, blocking Kubernetes API calls in `Mono.fromCallable()` and inside `flatMap()` without a `subscribeOn(Schedulers.boundedElastic())` directive. The `.cache()` operator's `Schedulers.newSingle(...)` parameter only manages the TTL timer — it does not offload the blocking work to a separate thread. Because the monitor is a Spring WebFlux/Netty application, the subscribing thread is a Netty I/O event-loop thread; blocking it stalls all requests multiplexed onto that thread. The `/actuator/health/cluster` path is exposed via an ingress enabled by default, making this reachable by any unauthenticated external user.

### Finding Description

**Exact code locations:**

`monitor/src/main/java/org/hiero/mirror/monitor/health/ReleaseHealthIndicator.java`

```
Line 74:  Mono.fromCallable(this::getHelmRelease)          // blocking, no subscribeOn
Line 75:  .cacheInvalidateIf(v -> false)
Line 78:  .flatMap(this::getHelmReleaseReadyStatus)         // blocking call inside flatMap, no publishOn
Line 82:  .cache(properties.getCacheExpiry(),
               Schedulers.newSingle("helmrelease-health-cache"));  // scheduler = TTL timer only
```

`getHelmRelease()` (lines 85–89):
```java
client.pods().withName(hostname).get()   // synchronous blocking HTTP to k8s API server
```

`getHelmReleaseReadyStatus()` (lines 93–95):
```java
client.genericKubernetesResources(RESOURCE_DEFINITION_CONTEXT)
      .withName(release)
      .get();   // synchronous blocking HTTP to k8s API server
```

**Root cause — two distinct blocking calls, neither offloaded:**

1. `Mono.fromCallable(this::getHelmRelease)` at line 74 has no `subscribeOn()`. In Reactor, without `subscribeOn()`, the `Callable` executes on whatever thread calls `subscribe()` — in WebFlux that is a Netty NIO event-loop thread.

2. `getHelmReleaseReadyStatus()` is called inside `flatMap` at line 78. Although `flatMap` subscribes to the inner `Mono`, the blocking Kubernetes API call at lines 93–95 executes *synchronously before* the `Mono<Health>` is even returned, so it also runs on the subscribing (Netty I/O) thread.

**Why the existing `.cache()` does not fix this:**

The call at line 82:
```java
.cache(properties.getCacheExpiry(), Schedulers.newSingle("helmrelease-health-cache"))
```
The `Scheduler` argument to `cache(Duration, Scheduler)` is used exclusively to schedule the TTL expiry timer. It has no effect on which thread the upstream `Callable` executes on. Every time the 30-second (minimum) TTL expires, the very next subscriber re-triggers the full upstream chain — including both blocking Kubernetes API calls — on the Netty I/O thread that handles that HTTP request.

**Why `.cacheInvalidateIf(v -> false)` does not fix this:**

This inner cache at line 75 permanently caches the pod label lookup result after the first call. It eliminates the `getHelmRelease()` blocking call after startup. However, `getHelmReleaseReadyStatus()` is downstream of this inner cache and is re-executed on every outer cache miss (every ≥30 s), still on the Netty I/O thread.

**Exploit flow:**

The monitor Helm chart enables an ingress by default exposing `/actuator/health/cluster`:
```yaml
# charts/hedera-mirror-monitor/values.yaml line 91-96
ingress:
  enabled: true
  hosts:
    - host: ""
      paths: ["/actuator/health/cluster"]
```

`health()` at line 63–66 calls `getHealth()` which returns the cached `Mono<Health>`. When the cache is cold or expired, subscribing to this `Mono` triggers the blocking Kubernetes API call on the Netty I/O thread.

### Impact Explanation
Netty uses a small, fixed pool of I/O event-loop threads (typically `2 × CPU cores`). Blocking even one of these threads for the duration of a slow or unresponsive Kubernetes API call (which can take seconds under load or network partition) stalls *all* HTTP connections multiplexed onto that thread. An attacker who times requests to arrive just after each 30-second cache expiry can repeatedly trigger this condition, progressively exhausting the Netty thread pool and causing application-wide request processing stalls — a targeted, low-rate DoS. The monitor's liveness probe (`/actuator/health/liveness`) also runs on the same Netty server; if I/O threads are blocked, liveness checks time out and Kubernetes restarts the pod, amplifying the disruption.

### Likelihood Explanation
- The ingress is **enabled by default** in the Helm chart with no authentication configured.
- The cache TTL has a **minimum of 30 seconds** (`@DurationMin(seconds = 30)`), giving a predictable, repeatable trigger window.
- No credentials, tokens, or cluster access are required — a single unauthenticated HTTP GET to `/actuator/health/cluster` at the right moment is sufficient.
- The attacker does not need to know the cache expiry value precisely; sending a request every ~30 seconds guarantees a hit on every cache miss.
- The `@ConditionalOnCloudPlatform(CloudPlatform.KUBERNETES)` and `enabled` flag (default `false`) mean the vulnerability is active only when `hiero.mirror.monitor.health.release.enabled=true` is set, which is the intended production Kubernetes configuration.

### Recommendation
Offload both blocking calls to a non-I/O scheduler by adding `subscribeOn(Schedulers.boundedElastic())` immediately after `Mono.fromCallable(...)`, and wrap the blocking portion of `getHelmReleaseReadyStatus()` in its own `Mono.fromCallable()` with the same scheduler:

```java
private Mono<Health> createHelmReleaseHealth() {
    return Mono.fromCallable(this::getHelmRelease)
            .subscribeOn(Schedulers.boundedElastic())   // ADD THIS
            .cacheInvalidateIf(v -> false)
            .doOnError(e -> log.error("Unable to get helm release", e))
            .onErrorComplete()
            .flatMap(release ->
                Mono.fromCallable(() -> getHelmReleaseReadyStatusBlocking(release))
                    .subscribeOn(Schedulers.boundedElastic())  // ADD THIS
                    .flatMap(Function.identity()))
            ...
}
```

Alternatively, restructure `getHelmReleaseReadyStatus` to return a `Mono` that defers the blocking call inside `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())`.

### Proof of Concept

**Preconditions:**
1. Monitor deployed on Kubernetes with `hiero.mirror.monitor.health.release.enabled=true`.
2. Ingress enabled (default) exposing `/actuator/health/cluster`.
3. Kubernetes API server reachable from the pod (standard cluster setup).

**Steps:**
```bash
# 1. Send an initial request to prime the cache
curl -s https://<monitor-ingress-host>/actuator/health/cluster

# 2. Wait for cache expiry (default 30 s)
sleep 31

# 3. Send request timed to hit the cold cache — this triggers the blocking k8s API call
#    on a Netty I/O thread. Simultaneously flood with concurrent requests to the same
#    or other endpoints to observe stalls.
curl -s https://<monitor-ingress-host>/actuator/health/cluster &
for i in $(seq 1 50); do
  curl -s https://<monitor-ingress-host>/actuator/health/liveness &
done
wait

# 4. Observe: liveness/readiness probe requests time out or return 503 while the
#    Netty I/O thread is blocked on the Kubernetes API call.
#    Repeat every 31 s to sustain the condition.
```

**Expected result:** Concurrent requests stall or time out during the window when the Netty I/O thread is blocked on `client.pods().withName(hostname).get()` or `client.genericKubernetesResources(...).get()`. Kubernetes liveness probe failures trigger pod restarts, amplifying the DoS.
