### Title
Non-Idempotent Retry Loop Enables Duplicate Transfers via Malicious Consensus Node Injection

### Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a brand-new `TransferTransaction` on every retry iteration, giving each attempt a distinct auto-generated `TransactionID`. A malicious consensus node injected through the unauthenticated HTTP response of `fetchMirrorNodeNetwork()` can accept and forward a transaction to the real network (causing it to commit), then return a fabricated error on the `GetReceipt()` call, triggering a retry that submits a second, independently-committed transaction. Because Hedera deduplicates only by `TransactionID`, and each retry carries a fresh ID, both transfers land permanently in the ledger.

### Finding Description

**Injection surface — `pinger/mirror_node_client.go` lines 74–131:**

`fetchMirrorNodeNetwork()` issues a plain `http.Client` GET with no TLS certificate pinning, no response signature, and no integrity check on the returned JSON. The default `mirrorRest` URL is `http://mirror-rest:5551` (plain HTTP). Any party who can MITM that HTTP connection, or who operates the mirror-node REST service itself, can return an arbitrary `nodes` array, injecting any gRPC endpoint as a "legitimate" consensus node. The resulting `map[string]hiero.AccountID` is handed directly to `hiero.ClientForNetwork()` and becomes the sole node pool for all subsequent SDK calls.

**Non-idempotent retry — `pinger/transfer.go` lines 23–57:**

```go
for i := 1; i <= attempts; i++ {          // attempts = maxRetries+1 (default 11)
    cryptoTransfer := hiero.NewTransferTransaction(). // ← NEW object, NEW TransactionID each iteration
        AddHbarTransfer(...).
        AddHbarTransfer(...)

    resp, err := cryptoTransfer.Execute(client)   // line 33
    if err == nil {
        receipt, rerr := resp.GetReceipt(client)  // line 35
        if rerr == nil { return nil }             // only exit on receipt success
        err = rerr
    }
    // fall through → retry with a completely new TX
}
```

`hiero.NewTransferTransaction()` produces a fresh object whose `TransactionID` is auto-generated (operator account + current wall-clock time) at `Execute()` time. There is no frozen transaction reused across iterations, no explicit `SetTransactionID()` call, and no idempotency key. Hedera's network-level deduplication window is keyed on `TransactionID`; distinct IDs receive no deduplication protection.

**Exploit flow:**

1. Malicious node receives the gRPC `CryptoTransfer` from the SDK.
2. It forwards the transaction bytes to a real Hedera node → transaction reaches consensus and is committed.
3. It returns a well-formed `TransactionResponse` with `OK` precheck to the SDK → `Execute()` returns `nil` error.
4. When `resp.GetReceipt(client)` is called, the malicious node (still the only node in the client's pool) returns a gRPC error (e.g., `DEADLINE_EXCEEDED`, `UNKNOWN`, or a receipt with a non-SUCCESS status) → `rerr != nil`.
5. `submitWithRetry()` falls through to the next iteration, constructs a **new** `TransferTransaction` with a **new** `TransactionID`, and calls `Execute()` again.
6. The second transaction is also committed. Both appear as distinct transfer records in Hashgraph history.
7. With `maxRetries=10` (default), up to 11 transfers can be committed per single pinger tick.

**Why existing checks are insufficient:**

- `ctx.Err()` check (line 24) only guards against cancellation, not duplicate submission.
- The backoff (lines 47–56) delays retries but does not prevent them.
- There is no call to query whether the previous `resp.TransactionID` was actually committed before constructing a new transaction.
- `buildNetworkFromMirrorNodes()` has no TLS enforcement, no certificate pinning, and no HMAC/signature verification on the node list.

### Impact Explanation

Each successful exploit iteration produces a permanent, consensus-finalized HBAR transfer that cannot be reversed. With the default `maxRetries=10` and `interval=1s`, an attacker sustaining the attack drains up to 11× the intended amount per second from the operator account. The duplicate records also corrupt the mirror node's transfer history, undermining the integrity of any downstream analytics or reconciliation that relies on the pinger as a ground-truth liveness signal. Severity is **High**: direct financial loss and ledger history tampering with no recovery path.

### Likelihood Explanation

The attack is scoped to deployments using `network=other` (custom/private Hiero networks), which is the exact deployment mode that relies on `fetchMirrorNodeNetwork()`. In Kubernetes environments (the pinger's intended runtime, evidenced by the `/tmp/ready` and `/tmp/alive` probe files), a compromised sidecar, a rogue DNS entry, or a misconfigured service mesh can trivially MITM the plain-HTTP mirror REST call. The attacker does not need any cryptographic material; they only need network-layer access to the mirror REST endpoint. The attack is fully repeatable for as long as the injected node remains in the client's pool (which is set once at startup and never refreshed).

### Recommendation

1. **Fix the retry loop to be idempotent**: Freeze the transaction and fix its `TransactionID` before the loop, then reuse the same frozen transaction bytes on every retry. Hedera will deduplicate subsequent submissions of the same ID within the valid-start window.

```go
cryptoTransfer := hiero.NewTransferTransaction().
    AddHbarTransfer(...).
    AddHbarTransfer(...)
frozen, err := cryptoTransfer.FreezeWith(client)  // fixes TransactionID
if err != nil { return err }

for i := 1; i <= attempts; i++ {
    resp, err := frozen.Execute(client)   // same TX bytes, same ID every time
    ...
}
```

2. **Before retrying after a GetReceipt failure**, query the network for the previous `resp.TransactionID` receipt explicitly. If it resolves to `SUCCESS`, return immediately without submitting a new transaction.

3. **Enforce TLS for the mirror REST endpoint**: Reject `http://` URLs or require a configurable CA bundle; add certificate pinning for known deployments.

4. **Periodically re-fetch and validate the node list** against a trusted out-of-band source rather than trusting a single unauthenticated HTTP response at startup.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`
- OR attacker can MITM the HTTP connection to the legitimate mirror REST endpoint (same L2 segment, rogue DNS, etc.)

**Steps:**

1. Stand up a malicious gRPC server implementing the Hedera node API. On `CryptoTransfer`:
   - Forward the transaction bytes to a real Hedera node.
   - Return `TransactionResponse{NodeTransactionPrecheckCode: OK}` to the pinger.
   - On the subsequent `TransactionGetReceipt` query for that transaction ID, return `DEADLINE_EXCEEDED` or drop the connection.

2. Serve a mirror REST `/api/v1/network/nodes` response pointing `service_endpoints` at the malicious gRPC server.

3. Start the pinger. On the first tick, `submitWithRetry()` will:
   - Call `Execute()` → malicious node forwards TX1 to real network (committed), returns OK.
   - Call `GetReceipt()` → malicious node returns error.
   - Retry: call `Execute()` with TX2 (new ID) → malicious node forwards TX2 to real network (committed), returns OK.
   - Repeat up to `maxRetries+1` times.

4. Query the real Hedera network for the operator account's transaction history. Observe multiple distinct `CryptoTransfer` records with different transaction IDs, all within the same pinger tick interval, each debiting `amountTinybar` from the operator account.
