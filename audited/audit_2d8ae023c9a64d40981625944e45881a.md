### Title
UNION ALL Double-Counting of Token Allowance Records via Degenerate `spender.id gte X lte X` Range with `token.id` Bounds

### Summary
When an unprivileged user supplies `spender.id=gte:N&spender.id=lte:N` (identical lower and upper bound values) together with `token.id=gte:T1&token.id=lte:T2`, the `extractTokenMultiUnionQuery` / `getLowerFilters` / `getUpperFilters` split produces two non-empty subqueries that both match `spender = N`, joined with `UNION ALL`. Every token allowance record in the intersection `spender = N AND T1 <= token_id <= T2` is returned twice, causing the mirror node to export duplicate/inflated allowance data to consumers.

### Finding Description

**Code path:**
- `rest/routes/accountRoute.js` line 18 → `TokenAllowanceController.getAccountTokenAllowances`
- `rest/controllers/tokenAllowanceController.js` lines 22–60 (`extractTokenMultiUnionQuery`)
- `rest/controllers/baseController.js` lines 131–183 (`getLowerFilters`, `getInnerFilters`, `getUpperFilters`)
- `rest/service/tokenAllowanceService.js` lines 54–78 (`getQuery`, joined with `union all` at line 74)

**Root cause — missing value-equality check in validation:**

`validateBoundsRange` (baseController.js lines 117–123) only rejects the combination of a range operator *and* an `eq` operator on the same field. It does **not** reject `gte X` paired with `lte X` when X is the same value. `validateLowerBounds` (lines 83–92) only checks that the lower-bound operator is `gte` (not `gt`); it never compares the actual bound values. `validateUpperBounds` (lines 100–109) is symmetric. No other guard exists.

**Filter-split logic with degenerate input `spender.id gte 5, lte 5, token.id gte 100, lte 200`:**

`getLowerFilters` (line 138–141): because `primary.hasLower() && secondary.hasLower()` is true, it emits:
```
[{spender, op: eq, val: 5}, {token_id, op: gte, val: 100}]
```

`getInnerFilters` (lines 160–167): emits:
```
[{spender, op: gt, val: 5}, {spender, op: lt, val: 5}]   ← vacuous, always false
```

`getUpperFilters` (lines 176–182): because both `primary.hasUpper()` and `secondary.hasUpper()` are true, emits:
```
[{spender, op: eq, val: 5}, {token_id, op: lte, val: 200}]
```

`getQuery` (line 74) joins all three non-empty subqueries with `union all`:
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=5 AND token_id>=100 ...)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>5 AND spender<5 ...)   -- always empty
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=5 AND token_id<=200 ...)
```

Every row satisfying `spender=5 AND 100 <= token_id <= 200` is returned by **both** the first and third subqueries. `UNION ALL` does not deduplicate, so those rows appear twice in the final result set.

### Impact Explanation
Mirror node consumers (wallets, exchanges, DeFi protocols) that read token allowance data via `/accounts/:id/allowances/tokens` receive duplicate records. A downstream system that sums or counts allowances will see inflated figures. In financial or authorization contexts this constitutes incorrect state exported from the mirror node, violating data integrity guarantees. Severity is medium: no funds are directly moved, but exported state is provably wrong and repeatable on demand.

### Likelihood Explanation
Any unauthenticated HTTP client can craft the request. No account ownership, special role, or prior knowledge beyond a valid account ID is required. The trigger is a single GET request with four query parameters. It is trivially repeatable and scriptable, and the degenerate range is not an edge case that would be caught by typical input fuzzing — it is a semantically valid but logically degenerate filter combination that the validation layer was not designed to reject.

### Recommendation
Add a value-level check in `validateBoundsRange` (or a dedicated `validateBoundValues` step called from `validateBounds`) that rejects the case where `primary.lower.value === primary.upper.value` when both are present. Equivalently, after computing `lower`, `inner`, and `upper` filter arrays, detect that the inner subquery is vacuous (its conditions are contradictory) and collapse the lower+upper pair into a single subquery with explicit `AND token_id >= T1 AND token_id <= T2` rather than emitting two overlapping `spender = N` subqueries joined by `UNION ALL`.

### Proof of Concept

**Precondition:** Account `0.0.1234` exists and has at least one token allowance for spender `0.0.5` with a token ID between 100 and 200.

**Request:**
```
GET /api/v1/accounts/0.0.1234/allowances/tokens?spender.id=gte:0.0.5&spender.id=lte:0.0.5&token.id=gte:0.0.100&token.id=lte:0.0.200
```

**Observed SQL (reconstructed from `getQuery`):**
```sql
(SELECT * FROM token_allowance
   WHERE owner = $1 AND amount > 0 AND spender = $3 AND token_id >= $4
   ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance
   WHERE owner = $1 AND amount > 0 AND spender > $5 AND spender < $6
   ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance
   WHERE owner = $1 AND amount > 0 AND spender = $7 AND token_id <= $8
   ORDER BY spender ASC, token_id ASC LIMIT $2)
ORDER BY spender ASC, token_id ASC LIMIT $2
-- params: [1234, 25, 5, 100, 5, 5, 5, 200]
```

**Result:** Every allowance row for `spender=5, 100 <= token_id <= 200` appears twice in the JSON response `allowances` array. The inner subquery (`spender > 5 AND spender < 5`) is always empty and contributes nothing, confirming the overlap is entirely between the lower and upper subqueries.
