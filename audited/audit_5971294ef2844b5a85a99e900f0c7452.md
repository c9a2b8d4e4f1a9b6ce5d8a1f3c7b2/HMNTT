### Title
Unbounded Batch DB Query Amplification via Cached Slot Key Accumulation in `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` issues a single `SELECT ... WHERE slot IN (...)` query containing every slot key ever accumulated in the per-contract Caffeine cache, not just the one being requested. An unprivileged caller can pre-fill that per-contract cache with up to 1,500 non-existent slot keys (the configured `slotsPerContract` maximum), then continuously trigger a 1,500-element `IN` clause on the `contract_state` table with every subsequent `eth_call`, because non-existent slots are never written to `contractStateCache` and therefore bypass the early-return guard on every call.

### Finding Description

**Code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` first. If the value is absent or equals the `EMPTY_VALUE` sentinel it falls through to `findStorageBatch()` (lines 85–122).

Inside `findStorageBatch()`: [1](#0-0) 

Line 90 unconditionally inserts the requested key into the per-contract Caffeine cache. Line 91 then snapshots **all** keys currently in that cache. Line 103 passes the entire snapshot to the repository: [2](#0-1) 

This produces `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (?, ?, … up to 1500 elements)`.

**Root cause — non-existent slots are never written to `contractStateCache`:**

After the batch query returns, only rows that actually exist in the DB are stored in `contractStateCache` (lines 106–113). A slot key that has no matching row is never cached. Therefore the guard at lines 63–67: [3](#0-2) 

always evaluates to `false` for non-existent slots, and `findStorageBatch()` is invoked on **every single call** for those keys.

**Cache size ceiling:**

The per-contract slot cache is created via `cacheManagerSlotsPerContract`, configured with: [4](#0-3) 

`maximumSize=1500` — so each contract accumulates at most 1,500 slot keys, producing an IN clause of up to 1,500 elements on every cache-miss call.

**Exploit flow:**

1. Attacker sends 1,500 `eth_call` requests to contract `C`, each specifying a distinct, non-existent 32-byte slot key. Each call reaches `findStorageBatch()`, adds the key to the per-contract cache, and issues a growing IN-clause query (1-element, 2-element, … 1,500-element). After 1,500 calls the cache is saturated.
2. From this point forward, **every** `eth_call` to contract `C` for any non-existent slot triggers a 1,500-element IN-clause query. Because non-existent slots are never written to `contractStateCache`, the early-return guard never fires and `findStorageBatch()` is called on 100% of such requests.
3. The attacker repeats step 2 at the maximum allowed rate (500 req/s by default), sustaining continuous 1,500-element batch queries against the DB.

### Impact Explanation

Each 1,500-element `IN` clause forces the PostgreSQL planner to evaluate a large parameter list against the `contract_state` table. On a production table with millions of rows this materially increases per-query execution time and I/O. Because the throttle is gas-based and request-rate-based — not DB-query-complexity-based — the attacker can sustain this at 500 req/s with minimal gas (21,000 gas per call consumes only 2 tokens from a 750,000-token/s bucket), making the DB the bottleneck rather than the application tier. Sustained amplification of this kind can push DB CPU and I/O above the 30% degradation threshold for legitimate traffic.

### Likelihood Explanation

No authentication or special privilege is required — `eth_call` is a public, unauthenticated endpoint. The attacker needs only to know a valid contract address (publicly discoverable on-chain) and to generate 1,500 arbitrary 32-byte slot keys. The cache-poisoning phase requires only 1,500 HTTP requests, well within the 500 req/s rate limit (achievable in 3 seconds). The attack is fully repeatable and stateless from the attacker's perspective.

### Recommendation

1. **Cap the IN-clause size**: Before calling `contractStateRepository.findStorageBatch()`, truncate `cachedSlots` to a configurable maximum (e.g., 100–200 entries) so a single call can never produce an arbitrarily large IN clause.
2. **Cache non-existent slots**: After `findStorageBatch()` returns, explicitly write `EMPTY_VALUE` into `contractStateCache` for every requested key that was absent from the result set. This allows the early-return guard to fire on subsequent calls for the same non-existent slot, eliminating the infinite re-query loop.
3. **Scope the batch to uncached keys only**: Before building `cachedSlots`, filter out keys whose values are already present in `contractStateCache`, so the IN clause only contains keys that genuinely need a DB round-trip.

for i in $(seq 1 1500); do
  SLOT=$(printf '%064x' $i)   # 32-byte non-existent slot key
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"<contract_address>\",
         \"data\":\"0x<sload_selector>$(printf '%064x' $i)\",
         \"gas\":21000}"
done

# Phase 2: sustain 1500-element IN-clause queries at max rate
while true; do
  SLOT=$(printf '%064x' $RANDOM)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"<contract_address>\",
         \"data\":\"0x<sload_selector>$SLOT\",
         \"gas\":21000}" &
done
# Each background request triggers findStorageBatch() with 1500 keys.
# Monitor DB: SELECT query, mean_exec_time FROM pg_stat_statements
#             WHERE query LIKE '%contract_state%IN%';
# Observe mean_exec_time rising >30% vs baseline single-key queries.
