### Title
Unauthenticated `/metrics` Endpoint Leaks `status_code` Labels, Enabling Degradation-Timed Attacks

### Summary
The `/metrics` endpoint is registered with no authentication and is publicly reachable by any external user. The `requestDurationHistogram` records every HTTP response status code — including `503` — as a Prometheus label value. An attacker can continuously poll `/metrics` to detect when the server enters a degraded state (503s accumulating) and time further attacks during that window.

### Finding Description
**Code path:**

- `rosetta/app/middleware/metrics.go` lines 28–32: `requestDurationHistogram` is declared with labels `["method", "route", "status_code", "ws"]`. Every response status code, including `503`, is recorded as a label value and emitted at `/metrics`.
- `rosetta/app/middleware/metrics.go` lines 64–73: The `/metrics` route is registered via `promhttp.Handler().ServeHTTP` with zero authentication or access control.
- `rosetta/app/middleware/trace.go` line 19: `/metrics` is listed in `internalPaths`, which only suppresses verbose logging — it applies **no access restriction**.
- `rosetta/main.go` lines 217–219: `MetricsMiddleware` wraps the entire router, so all routes — including `/health/readiness` — have their status codes recorded in `requestDurationHistogram`.
- `rosetta/app/middleware/health.go` lines 37–53: The readiness endpoint performs PostgreSQL and network checks and returns HTTP `503` when either fails.

**Root cause:** The failed assumption is that `/metrics` is an internal-only path. `internalPaths` only controls log verbosity; it does not gate network access. `promhttp.Handler()` is served directly to the public listener with no middleware enforcing authentication or IP allowlisting.

**Exploit flow:**
1. Attacker continuously polls `GET /metrics` (no credentials required).
2. Attacker watches the counter `hiero_mirror_rosetta_request_duration_bucket{status_code="503", ...}` increment.
3. When 503s appear (DB down, network degraded, or server overloaded), the attacker knows the server is in a degraded state.
4. Attacker times resource-exhaustion requests, replay attempts, or other attacks during this window when the server's capacity to validate and reject requests is reduced.

**Why existing checks fail:** The `internalPaths` map in `trace.go` is purely a logging filter. No middleware, no IP allowlist, no token check, and no network policy is enforced at the application layer on the `/metrics` route.

### Impact Explanation
An attacker gains real-time visibility into server health state without any credentials. Knowing when the server is returning 503s (DB unavailable, network degraded) allows precise timing of follow-on attacks — e.g., flooding expensive endpoints when the DB connection pool is already saturated, or submitting malformed transactions when validation throughput is reduced. This converts a passive information disclosure into an active attack amplifier.

### Likelihood Explanation
The `/metrics` endpoint requires no authentication, no API key, and no network-level restriction at the application layer. Any internet-reachable deployment is trivially exploitable with a single `curl` command. Polling at 15-second intervals (Prometheus default scrape interval) is indistinguishable from normal scraper traffic and generates negligible load. The attacker needs zero privileges and zero prior knowledge beyond the server's IP and port.

### Recommendation
1. **Restrict `/metrics` at the application layer**: Add an authentication middleware (e.g., Bearer token, mTLS) specifically for the `/metrics` route before `promhttp.Handler()` is invoked.
2. **Alternatively, serve metrics on a separate internal-only port**: Bind the Prometheus handler to a second `http.Server` listening on `127.0.0.1` or a private network interface only, separate from the public Rosetta API port.
3. **Remove `status_code` as a high-cardinality label if not needed internally**, or at minimum ensure the metrics port is not publicly reachable.

### Proof of Concept
```bash
# Step 1: Confirm /metrics is publicly accessible (no auth required)
curl -s http://<rosetta-host>:<port>/metrics | grep request_duration

# Step 2: Poll for 503 accumulation (indicates server degradation)
watch -n 15 'curl -s http://<rosetta-host>:<port>/metrics \
  | grep hiero_mirror_rosetta_request_duration \
  | grep status_code=\"503\"'

# Step 3: When 503 count starts incrementing, the server is degraded.
# Launch resource-exhausting requests during this window:
for i in $(seq 1 500); do
  curl -s -X POST http://<rosetta-host>:<port>/block \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hiero","network":"mainnet"},"block_identifier":{"index":1}}' &
done
```
