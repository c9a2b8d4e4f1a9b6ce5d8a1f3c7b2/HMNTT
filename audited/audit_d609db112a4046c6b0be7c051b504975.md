### Title
Post-Load Memory Accumulation in `FindBetween()` Allows Griefing via High-Transaction Block Queries

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` accumulates all transactions for a block into a single in-memory slice across unlimited batches of 2000 before returning, with no total-count cap at the persistence layer. The only guard — `maxTransactionsInBlock` — is applied in `block_service.go` **after** the full slice is already resident in memory. Any unprivileged caller can repeatedly POST to the public `/block` endpoint with the index of a high-transaction block, forcing the server to allocate unbounded heap per request and degrading service for all concurrent users.

### Finding Description
**Code path:**

`rosetta/app/persistence/transaction.go`, `FindBetween()`, lines 123–147:

```go
transactions := make([]*transaction, 0)          // line 123 – no capacity cap
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error  // batchSize = 2000
    ...
    transactions = append(transactions, transactionsBatch...)           // line 140 – unbounded accumulation
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

The loop continues fetching 2000-row batches and appending them to `transactions` until the DB returns a partial batch. There is no ceiling on the total number of rows accumulated.

**Post-load cap (insufficient):**

`rosetta/app/services/block_service.go`, `Block()`, lines 56–66:

```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
    return nil, err
}
// cap applied AFTER full load
if len(block.Transactions) > s.maxTransactionsInBlock {
    ...
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
```

`maxTransactionsInBlock` (configured via `hiero.mirror.rosetta.response.maxTransactionsInBlock`, used in `rosetta/app/config/types.go` line 84) only trims the **response**; it does not prevent `FindBetween()` from loading every transaction into the Go heap first.

**Root cause:** The persistence layer has no total-row budget. The service layer's cap is a response-shaping concern, not a memory-protection concern.

**Attacker input surface:** The `/block` endpoint accepts a `BlockIdentifier` containing an integer `index` or a hash string — both are unauthenticated. The server resolves the block's `ConsensusStartNanos`/`ConsensusEndNanos` from the database and passes them directly to `FindBetween()`. The attacker does not need to supply timestamps; they only need to know (or enumerate) the index of a block that contains a large number of transactions.

### Impact Explanation
On a live Hedera mainnet mirror node, individual record files (blocks) can contain tens of thousands of transactions during high-throughput periods. Each concurrent `/block` request for such a block causes the Go runtime to allocate a `[]*transaction` slice holding all rows, plus the `sameHashMap` and `hashes` structures built from it (lines 149–158), before any trimming occurs. Under concurrent load from multiple attackers (or a single attacker with parallel connections), heap pressure grows multiplicatively, triggering GC storms and eventually OOM-kills or severe latency spikes that affect all users of the Rosetta API. The impact is service degradation (griefing) with no economic damage to network participants, consistent with the Medium severity classification.

### Likelihood Explanation
The `/block` endpoint is public and requires no credentials. An attacker needs only to identify a block index with a high transaction count — trivially done by querying `/network/status` for the current block index and then binary-searching or iterating to find dense blocks, or by consulting public Hedera explorers. The attack is fully repeatable and scriptable with standard HTTP tooling (e.g., `curl` or `wrk`). No special network position, cryptographic material, or on-chain funds are required.

### Recommendation
Apply the transaction count limit **inside `FindBetween()`** at the persistence layer, before accumulation completes:

1. Pass `maxTransactionsInBlock` (or a configurable `maxRows` parameter) into `FindBetween()` and break the batch loop as soon as `len(transactions) >= maxRows`, returning the partial result along with a flag or error indicating truncation.
2. Alternatively, push the limit into the SQL query itself by replacing the per-batch `LIMIT batchSize` with a global `LIMIT maxRows` across the entire range, eliminating the loop entirely for the common case.
3. Retain the existing post-load cap in `block_service.go` as a secondary defence, but it must not be the only guard.

### Proof of Concept

**Preconditions:** A running Rosetta mirror node connected to mainnet (or testnet) with at least one block containing > `maxTransactionsInBlock` transactions.

**Steps:**

1. Identify a high-transaction block index (e.g., via public explorer or by iterating `/block` responses):
   ```
   POST /block
   {"network_identifier": {...}, "block_identifier": {"index": <HIGH_TX_BLOCK_INDEX>}}
   ```

2. Confirm the response contains `other_transactions` (indicating the block exceeds `maxTransactionsInBlock`), proving all transactions were loaded.

3. Send concurrent requests to exhaust heap:
   ```bash
   for i in $(seq 1 200); do
     curl -s -X POST http://<rosetta-host>:5700/block \
       -H 'Content-Type: application/json' \
       -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":<HIGH_TX_BLOCK_INDEX>}}' &
   done
   wait
   ```

4. **Expected result:** Go heap grows proportionally to `(concurrent_requests × transactions_in_block × per_transaction_struct_size)`. With 200 concurrent requests and a 50,000-transaction block, this materialises hundreds of MB of heap before GC can reclaim it, causing measurable latency degradation or OOM for all concurrent users.
