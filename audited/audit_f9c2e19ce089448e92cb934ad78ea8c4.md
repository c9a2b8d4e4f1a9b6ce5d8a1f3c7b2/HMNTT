### Title
Silent Sentinel-Value Collision in `getIntMetadataValue` Allows `valid_start_nanos: "0"` to Trigger Random Transaction ID Generation

### Summary
`getIntMetadataValue()` accepts the string `"0"` as a valid `valid_start_nanos` value because its boundary check is `metadataValue < 0` rather than `<= 0`. The value `0` is then passed to `transactionSetTransactionId()`, which treats `validStartNanos == 0` as a sentinel meaning "generate a random ID now." Any unprivileged caller of `/construction/payloads` can trigger this by supplying `valid_start_nanos: "0"`, causing the transaction to be assigned an unpredictable ID that the mirror node records under a value the caller did not specify.

### Finding Description

**Code path:**

`getIntMetadataValue()` — `rosetta/app/services/construction_service.go` lines 540–554:
```go
if metadataValue, err = tools.ToInt64(value); err != nil || metadataValue < 0 {
    return metadataValue, errors.ErrInvalidArgument
}
```
The guard is `< 0`. The value `"0"` parses to `0`, which is not `< 0`, so the function returns `(0, nil)` — no error.

`getTransactionTimestampProperty()` — lines 424–427:
```go
validStartNanos, err = c.getIntMetadataValue(metadata, metadataKeyValidStartNanos)
if err != nil {
    log.Errorf(...)
}
return
```
No post-call check for `validStartNanos <= 0` exists here. Compare with the `valid_until_nanos` branch at lines 407–412, which **does** explicitly check `if validStartNanos <= 0` and returns `ErrInvalidArgument`. That guard is entirely absent for the direct `valid_start_nanos` path.

`transactionSetTransactionId()` — lines 730–733:
```go
if validStartNanos == 0 {
    transactionId = hiero.TransactionIDGenerate(payer)   // random current-time ID
} else {
    transactionId = hiero.NewTransactionIDWithValidStart(payer, time.Unix(0, validStartNanos))
}
```
`0` is the sentinel that means "I have no caller-supplied value; generate one." Because `getIntMetadataValue` lets `0` through silently, the sentinel is reachable by any caller.

**Root cause:** The boundary check `< 0` in `getIntMetadataValue` and the sentinel `== 0` in `transactionSetTransactionId` share the value `0`, but there is no guard between them in the `valid_start_nanos` code path to detect the collision.

### Impact Explanation
A caller who supplies `valid_start_nanos: "0"` receives an unsigned transaction whose transaction ID was generated at server time, not at the epoch they specified. The mirror node indexes and returns this transaction under the server-generated `payer@<server_nanos>` identifier. Any downstream system that pre-computed or stored the expected transaction ID (`payer@0.000000000`) will fail to locate the transaction. In a Rosetta construction flow where the caller is expected to track the transaction ID for status polling, this silent substitution breaks the caller's ability to correlate the submitted transaction with mirror-node records without inspecting the raw bytes of the response.

### Likelihood Explanation
No authentication or privilege is required. The `/construction/payloads` endpoint is reachable by any Rosetta API client. The trigger is a single field set to the string `"0"`. It is deterministically reproducible on every invocation.

### Recommendation
In `getTransactionTimestampProperty()`, add an explicit positivity check after retrieving `validStartNanos`, mirroring the guard already present in the `valid_until_nanos` branch:

```go
validStartNanos, err = c.getIntMetadataValue(metadata, metadataKeyValidStartNanos)
if err != nil || validStartNanos <= 0 {
    log.Errorf("Invalid valid start nanos %s", metadata[metadataKeyValidStartNanos])
    err = errors.ErrInvalidArgument
    return
}
```

Alternatively, change the boundary check in `getIntMetadataValue` to `<= 0` for the `valid_start_nanos` key, or document and enforce that callers must omit the key (not set it to `"0"`) to request auto-generation.

### Proof of Concept

```
POST /construction/payloads
{
  "network_identifier": { ... },
  "operations": [ ... ],
  "metadata": {
    "node_account_id": "0.0.3",
    "valid_duration": "120",
    "valid_start_nanos": "0"   // <-- triggers the bug
  }
}
```

1. `getIntMetadataValue(metadata, "valid_start_nanos")` parses `"0"` → `0`; check `0 < 0` is false → returns `(0, nil)`.
2. `getTransactionTimestampProperty` returns `validStartNanos = 0` with no error.
3. `transactionSetTransactionId(payer, 0)` executes `hiero.TransactionIDGenerate(payer)`, assigning a server-time-based transaction ID.
4. The response `unsigned_transaction` encodes a transaction ID of `payer@<server_nanos>`, not `payer@0.000000000`.
5. The mirror node records the transaction under `payer@<server_nanos>`; any lookup by the caller using the expected `payer@0.000000000` returns no result.
