### Title
Thundering Herd via Uncached, Unthrottled `GET /api/v1/network/supply` Causing Database Aggregation Saturation

### Summary
The `GET /api/v1/network/supply` endpoint in the `rest-java` module invokes a heavy native SQL aggregation query on every request with no caching, no request coalescing, and no rate limiting. An unprivileged attacker sending a burst of concurrent requests after a cache miss or service restart will cause all threads to simultaneously execute the expensive `SUM`/`JOIN`/`unnest` aggregation against the shared mirror node database, potentially saturating the DB connection pool and degrading the importer's ability to process records.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (no rate-limit annotation, no throttle) calls `NetworkService.getSupply()`: [1](#0-0) 

`NetworkServiceImpl.getSupply()` has **no `@Cacheable` annotation** and no single-flight/request-coalescing guard. On the no-timestamp path it directly calls the repository: [2](#0-1) 

`EntityRepository.getSupply()` executes a native SQL aggregation — `SUM(balance)` + `MAX(balance_timestamp)` over the entire `entity` table joined with an `unnest`-ed range array (7 account ranges, potentially millions of rows): [3](#0-2) 

**Root cause — failed assumptions:**

1. **No caching**: A search across all `rest-java` Java sources finds zero `@Cacheable`/`@CachePut` annotations on `NetworkServiceImpl` or `EntityRepository.getSupply()`. The only caching infrastructure in `rest-java` is response-header configuration in `RestJavaProperties`. [4](#0-3) 

2. **No rate limiting**: The `ThrottleConfiguration` and `ThrottleManagerImpl` (bucket4j) exist exclusively in the `web3` module. The `rest-java` config directory contains only `MetricsFilter` and `LoggingFilter` — no rate-limit filter. [5](#0-4) 

3. **No request coalescing**: Each concurrent HTTP thread independently calls `entityRepository.getSupply()`, issuing an identical aggregation query.

**Exploit flow:**

- Attacker sends N concurrent `GET /api/v1/network/supply` requests (no auth required, no timestamp param needed).
- All N threads pass through `NetworkController` → `NetworkServiceImpl.getSupply()` → `EntityRepository.getSupply()`.
- All N threads simultaneously submit the same heavy `SUM`/`JOIN`/`unnest` query to PostgreSQL.
- The HikariCP connection pool queues excess threads, but the active connections each hold a long-running aggregation, blocking the pool for the importer and other consumers.

### Impact Explanation

The `entity` table is the core mirror node table; a full-scan aggregation across 7 account ranges (accounts 2–750 by default) on a production network with millions of entities is a multi-second query. Saturating the shared PostgreSQL connection pool degrades the mirror node importer's ability to write new record files, directly impacting record processing throughput. Sustained attack can cause the importer to fall behind consensus, effectively degrading ≥30% of mirror node processing capacity. The endpoint is unauthenticated and publicly documented in the OpenAPI spec. [6](#0-5) 

### Likelihood Explanation

No attacker capability beyond HTTP access is required. The endpoint is public, documented, and accepts no authentication. A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple script) can issue hundreds of simultaneous requests. The attack is trivially repeatable and can be automated. After each cache-miss window (e.g., service restart), the window re-opens. The fixed query parameters (derived from `NetworkProperties`) mean every request is identical, maximizing DB-level contention. [7](#0-6) 

### Recommendation

1. **Add `@Cacheable`** on `NetworkServiceImpl.getSupply()` (for the no-timestamp path) with a short TTL (e.g., 5–15 seconds), using Caffeine. This collapses all concurrent cache-miss requests into a single DB call.
2. **Add request coalescing** (e.g., Guava `LoadingCache` with `refreshAfterWrite`) so only one thread computes the value while others wait.
3. **Add rate limiting** in the `rest-java` module (bucket4j or similar) for the `/api/v1/network/supply` path, mirroring the pattern already used in `web3`. [8](#0-7) 
4. **Tune HikariCP** to reserve a connection pool partition for the importer, isolating it from REST query saturation.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or wrk
# Step 1: Ensure service is freshly started (cold cache)
# Step 2: Send 200 concurrent requests
seq 200 | parallel -j200 \
  curl -s -o /dev/null -w "%{http_code} %{time_total}\n" \
  "https://<mirror-node-host>/api/v1/network/supply"

# Expected: response times spike to several seconds; DB CPU pegs at 100%;
# importer lag (observable via /api/v1/blocks latency) increases measurably.
# Repeat immediately after first wave completes to sustain pressure.
```
