### Title
EVM Address Resolution Causes Uncached Extra DB Query Per Request, Enabling 2x Database Load Amplification

### Summary
In `getAccountCryptoAllowances()`, when the path parameter is supplied as an EVM address instead of a numeric entity ID, `EntityService.getEncodedId()` unconditionally issues a live database query to resolve the address before the main allowances query. No caching exists for this resolution in the REST service layer. Any unauthenticated caller can exploit this to double the database query count per request compared to numeric ID callers.

### Finding Description
**Code path:**

`rest/controllers/cryptoAllowanceController.js` line 77 calls:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

Inside `rest/service/entityService.js` lines 118–137, `getEncodedId()` branches on the parsed form of the input:

```js
// Line 122-124
return entityId.evmAddress === null
  ? entityId.getEncodedId()                                    // pure computation — zero DB queries
  : await this.getEntityIdFromEvmAddress(entityId, requireResult); // DB query #1
```

`getEntityIdFromEvmAddress()` (lines 90–91) executes:
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
  [Buffer.from(entityId.evmAddress, 'hex')]);
```

where `entityFromEvmAddressQuery` (lines 22–25) is:
```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```

This is a direct, uncached database call. After it returns, `CryptoAllowanceService.getAccountCryptoAllowances()` (line 80) issues **DB query #2** regardless of input format.

**Root cause:** The `EntityService` in the REST module has no caching layer (unlike the web3 `EntityRepository` which uses `@Cacheable`). The branch at line 122 only skips the DB call for numeric IDs; EVM address inputs always hit the database. No rate limiting exists in the REST module's production code path.

**Why existing checks fail:** The only rate-limiting reference in the REST module is in `rest/__tests__/integrationUtils.js` (test scaffolding only). There is no middleware or application-level throttle protecting this endpoint from repeated EVM-address-format requests.

### Impact Explanation
Every request using an EVM address format generates 2 database queries; every request using a numeric ID generates 1. An attacker sending N concurrent requests with EVM addresses imposes 2N queries on the database, versus N for numeric IDs. At sustained high request rates this doubles database CPU, connection pool consumption, and I/O pressure. Because the entity table is queried without a result cache, even repeated lookups of the same EVM address re-execute the query each time. The endpoint is public and requires no authentication.

### Likelihood Explanation
The exploit requires zero privileges — the endpoint is unauthenticated and publicly accessible. The EVM address format is a documented, valid input format for the `:idOrAliasOrEvmAddress` path parameter. An attacker needs only to send HTTP GET requests to `/api/v1/accounts/<evm_address>/allowances/crypto` at high concurrency. No special tooling beyond a standard HTTP load generator (e.g., `wrk`, `ab`, `hey`) is required. The attack is trivially repeatable and stateless.

### Recommendation
1. **Add an in-process cache** for EVM address → entity ID resolution in the REST `EntityService`, keyed on the hex EVM address, with a short TTL (e.g., 60 seconds). This mirrors the `@Cacheable` pattern already used in the web3 `EntityRepository`.
2. **Add rate limiting middleware** to the REST API server (e.g., via `express-rate-limit`) scoped to per-IP or per-path to bound the maximum query rate from any single source.
3. Optionally, accept a long-form EVM address that encodes shard/realm/num (the "long-zero" prefix pattern already handled in the graphql `EntityServiceImpl`) so that numeric-equivalent EVM addresses can be resolved without a DB round-trip.

### Proof of Concept
**Precondition:** Mirror node REST API is running and accessible. At least one account with a known EVM address exists (e.g., `0x000000000000000000000000000000000000abcd`).

**Steps:**

1. Identify a valid account EVM address, e.g. `0x71eaa748d5252be68c1185588beca495459fdba4`.
2. Send high-concurrency requests using EVM address format:
   ```
   GET /api/v1/accounts/0x71eaa748d5252be68c1185588beca495459fdba4/allowances/crypto
   ```
3. Observe in database monitoring (e.g., `pg_stat_activity`) that each request generates **2 queries**: one against `entity` (EVM address lookup) and one against `crypto_allowance`.
4. Send the same volume of requests using numeric ID format:
   ```
   GET /api/v1/accounts/0.0.1002/allowances/crypto
   ```
5. Observe that each request generates only **1 query** against `crypto_allowance`.
6. At N=1000 concurrent requests/sec, EVM address format produces 2000 DB queries/sec vs. 1000 for numeric ID — confirming the 2x amplification with no authentication required.
