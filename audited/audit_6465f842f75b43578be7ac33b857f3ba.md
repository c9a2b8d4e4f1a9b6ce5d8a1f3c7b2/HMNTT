### Title
Missing Block Filter Dependency Check Allows `transaction.index` to Return Cross-Block Contract Results

### Summary
The `/api/v1/contracts/results` (and `/api/v1/contracts/:contractId/results`) endpoints accept `filterKeys.TRANSACTION_INDEX` as a standalone query parameter with no enforcement that `filterKeys.BLOCK_NUMBER` or `filterKeys.BLOCK_HASH` must also be present. Because `transaction_index` is only meaningful within a single block, omitting a block constraint causes the mirror node to return contract results matching that index position across every block in the database — semantically incorrect records returned to any unauthenticated caller.

### Finding Description

**Accepted parameter set** — `acceptedContractResultsParameters` at `rest/controllers/contractController.js` lines 1336–1346 includes all three keys together with no ordering or dependency constraint:

```js
const acceptedContractResultsParameters = new Set([
  filterKeys.FROM,
  filterKeys.BLOCK_HASH,
  filterKeys.BLOCK_NUMBER,
  ...
  filterKeys.TRANSACTION_INDEX,   // line 1345
]);
``` [1](#0-0) 

**Filter processing** — Inside `extractContractResultsByIdQuery` (lines 415–542), the `TRANSACTION_INDEX` case (lines 468–477) unconditionally appends a SQL condition for `transaction_index`:

```js
case filterKeys.TRANSACTION_INDEX:
  this.updateConditionsAndParamsWithInValues(
    filter,
    transactionIndexInValues,
    params,
    conditions,
    transactionIndexFullName,
    conditions.length + 1
  );
  break;
``` [2](#0-1) 

**Block filter guard** — The block-to-timestamp translation only executes when `blockFilter` is truthy (lines 486–502). If neither `BLOCK_NUMBER` nor `BLOCK_HASH` is supplied, `blockFilter` remains `undefined`, no timestamp range is added, and the query runs against the entire `contract_result` table:

```js
if (blockFilter) {          // only entered when block filter present
  ...
  timestampFilters.push(
    {key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: blockData.consensusStart},
    {key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: blockData.consensusEnd}
  );
}
``` [3](#0-2) 

**No dependency check exists anywhere** — there is no validation in `contractResultsFilterValidityChecks`, `utils.buildAndValidateFilters`, or any middleware that rejects a `TRANSACTION_INDEX` filter when no block filter is present. [4](#0-3) 

The resulting SQL WHERE clause becomes simply:
```sql
WHERE cr.transaction_index = <user_value>
  AND cr.transaction_nonce = 0
```
with no block/timestamp boundary, causing the database to scan and return every contract result whose position within its respective block equals the supplied index.

### Impact Explanation
`transaction_index` is a per-block ordinal (position 0 = first transaction in a block, position 1 = second, etc.). Returning all rows matching a given index across all blocks is factually incorrect data: a caller querying `transaction.index=0` receives the first transaction from every block ever recorded, not the first transaction of any specific block. This constitutes **incorrect record disclosure** — the mirror node exports data that misrepresents the blockchain state. Depending on the number of blocks, the response can also be very large, enabling a lightweight amplification/resource-exhaustion request from an unauthenticated user.

### Likelihood Explanation
No authentication is required to call `/api/v1/contracts/results`. The parameter name `transaction.index` is documented in the OpenAPI spec. Any external user who reads the API docs can craft the request in seconds. The condition is trivially reproducible and repeatable with a single HTTP GET.

### Recommendation
Add an explicit dependency check before building the query. In `extractContractResultsByIdQuery` (or in `contractResultsFilterValidityChecks`), verify that whenever `filterKeys.TRANSACTION_INDEX` is present in the filter list, at least one of `filterKeys.BLOCK_NUMBER` or `filterKeys.BLOCK_HASH` is also present. If the dependency is not satisfied, throw an `InvalidArgumentError` with a message such as:
> `transaction.index requires block.number or block.hash to be specified`

This mirrors the existing pattern used for topic filters requiring a timestamp range (`checkTimestampsForTopics`, lines 281–306). [5](#0-4) 

### Proof of Concept

```
GET /api/v1/contracts/results?transaction.index=0
```

**Preconditions:** Mirror node is running and has ingested multiple blocks. No credentials needed.

**Expected (correct) behavior:** HTTP 400 — `transaction.index` requires a block filter.

**Actual behavior:** HTTP 200 — returns all contract results where `transaction_index = 0` across every block in the database, i.e., the first contract transaction from every block ever recorded.

To confirm cross-block leakage, compare the `block_number` field across items in the response — they will span many different blocks, demonstrating that the result set is not scoped to any single block.
