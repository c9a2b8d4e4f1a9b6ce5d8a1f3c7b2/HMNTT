### Title
Unauthenticated High-Frequency EVM Address Lookups Exhaust DB Connection Pool via `getContractIdByEvmAddress()`

### Summary
`getContractIdByEvmAddress()` issues one synchronous parameterized DB query per call with no application-layer rate limiting, no per-address caching, and a finite connection pool. An unauthenticated attacker flooding contract endpoints with distinct EVM addresses can saturate the pool, causing legitimate requests to queue and time out, degrading service for all users.

**Root cause ã failed assumption:** The design assumes that infrastructure-level rate limiting (load balancer, API gateway) will be present. At the application layer there is no rate limiting middleware, no per-address result cache at the service layer, and no concurrency guard on this code path.

**DB pool is finite and directly consumed:** `dbpool.js` creates a `pg.Pool` with `max: config.db.pool.maxConnections`: [6](#0-5) 

**No rate limiting middleware exists:** `server.js` registers only `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, and optional `responseCacheCheckHandler` ã no rate-limiting middleware at any layer: [7](#0-6) 

**Response cache does not help:** The cache is gated on both `config.cache.response.enabled` AND `config.redis.enabled` (line 54). Even when enabled, each unique EVM address URL is a distinct cache key ã every address in the flood is a cache miss, so every request reaches the DB. [8](#0-7) 

**`authHandler` does not rate-limit:** It only adjusts the response result-set `limit` for authenticated users; it does not throttle request frequency for unauthenticated callers: [9](#0-8) 

### Impact Explanation
When the pool is saturated, `pg` queues new connection requests until `connectionTimeoutMillis` expires, at which point they throw. All contract endpoints that resolve an EVM address path parameter share this pool. Sustained flooding causes 503/timeout responses for legitimate users across all contract API routes. No authentication, API key, or on-chain asset is required to mount the attack.

### Likelihood Explanation
Any internet-accessible deployment is reachable. The attacker needs only an HTTP client capable of sending concurrent GET requests to `/api/v1/contracts/{40-hex-char-address}/results` (or `/state`, `/logs`, etc.) with addresses that differ by one byte per request to defeat any hypothetical future caching. The attack is stateless, requires no credentials, and is trivially scriptable. It is repeatable indefinitely.

### Recommendation
1. **Add application-layer rate limiting** (e.g., `express-rate-limit`) keyed on IP address, applied before contract route handlers, as a defense-in-depth measure independent of infrastructure.
2. **Cache EVM-address-to-entity-ID resolutions** in a short-lived in-process LRU cache (e.g., 30 s TTL). A resolved address is immutable; caching it eliminates the DB round-trip for repeated or near-duplicate lookups and drastically reduces pool pressure.
3. **Set a conservative `maxConnections`** and ensure `connectionTimeoutMillis` is low enough to fail fast rather than queue indefinitely, limiting blast radius.
4. **Enforce rate limiting at the ingress/load-balancer layer** as the primary control, with the application-layer limit as a secondary backstop.

### Proof of Concept
```bash
# Send 200 concurrent GET requests, each with a unique EVM address
# (addresses differ by last byte: 00..FF then wrap)
for i in $(seq 0 199); do
  addr=$(printf "aabbccddeeff00112233445566778899aabbcc%02x" $((i % 256)))
  curl -s "https://<mirror-node-host>/api/v1/contracts/0x${addr}/results" &
done
wait
# Observe: legitimate requests begin returning 503 or connection-timeout errors
# as the pg pool (default maxConnections) is saturated
```

  async getContractIdByEvmAddress(evmAddressFilter) {
    const create2EvmAddress = evmAddressFilter.create2_evm_address;
    const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
      Buffer.from(create2EvmAddress, 'hex'),
    ]);
    if (rows.length === 0) {
      throw new NotFoundError(`No contract with the given evm address 0x${create2EvmAddress} has been found.`);
    }
    // since evm_address is not a unique index, it is important to make this check.
    if (rows.length > 1) {
      throw new Error(`More than one contract with the evm address 0x${create2EvmAddress} have been found.`);
    }

    return rows[0].id;
  }
