### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses Query Limit, Enabling Unbounded Database Reads

### Summary
In `PollingTopicListener.poll()`, the remaining limit is computed as `(int)(filter.getLimit() - context.getCount().get())`. A user-supplied `limit` value of `2147483648L` (just above `Integer.MAX_VALUE`) causes this cast to produce `Integer.MIN_VALUE`. Because `TopicMessageFilter.hasLimit()` returns `false` for negative values, `TopicMessageRepositoryCustomImpl.findByFilter()` never calls `setMaxResults()`, executing an unbounded SQL query on every poll cycle.

### Finding Description
**Exact code path:**

`PollingTopicListener.java` lines 54–57:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
``` [4](#0-3) 

With `filter.getLimit() = 2147483648L` and `context.getCount().get() = 0`:
- `(int)(2147483648L) = Integer.MIN_VALUE = -2147483648`
- `Math.min(Integer.MIN_VALUE, 5000) = Integer.MIN_VALUE`
- `newFilter` is built with `limit = Integer.MIN_VALUE`

`TopicMessageRepositoryCustomImpl.java` lines 51–53:
```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [3](#0-2) 

`hasLimit()` checks `limit > 0`; since `Integer.MIN_VALUE < 0`, the condition is false and `setMaxResults()` is never called. The JPA query executes with no row limit, returning every `TopicMessage` row matching the topic and start time.

**Root cause:** No upper-bound validation on `TopicMessageFilter.limit` (only `@Min(0)` exists), combined with a narrowing `(int)` cast that silently wraps large positive longs to negative ints, which then defeats the `hasLimit()` guard downstream. [5](#0-4) 

**Why checks fail:**
- `@Min(0)` on `limit` permits any positive `long`, including values > `Integer.MAX_VALUE`.
- The internal `filter.toBuilder().limit(pageSize).build()` call in `poll()` does not go through Spring's `@Valid` AOP interception, so the `@Min(0)` constraint is never re-evaluated on the derived filter.
- `hasLimit()` returning `false` for the overflowed negative value causes `findByFilter` to silently omit the limit rather than reject or clamp it.
- The service-level `flux.take(filter.getLimit())` uses the original `2147483648L`, allowing ~2 billion messages through. [6](#0-5) 

### Impact Explanation
Every poll cycle (default every 500 ms) issues a full table scan against `topic_message` for the targeted topic with no row limit. For a high-volume topic (e.g., a fee-schedule topic with years of history), this can:
1. **Exhaust database I/O and CPU** — repeated unbounded scans degrade service for all users (DoS).
2. **Expose complete message history** — all fee-update records (or any topic's full history) are streamed to the attacker, up to 2 billion messages per subscription.
3. **Exhaust JVM heap** — `getResultList()` materializes the entire result set in memory before streaming.

Severity: **High** (DoS + information disclosure, no authentication required beyond a valid gRPC connection).

### Likelihood Explanation
- Any unauthenticated gRPC client can call `subscribeTopic` with an arbitrary `limit` value.
- The trigger value (`2147483648L`) is trivially derivable from knowledge of Java integer overflow.
- The attack is repeatable: each new subscription with the crafted limit re-triggers the unbounded query on every poll interval.
- Requires `listenerProperties.type = POLL` (non-default; default is `REDIS`), but this is a documented deployment option and the `PollingTopicListener` is a first-class supported component.

### Recommendation
1. **Add `@Max(Integer.MAX_VALUE)` to `TopicMessageFilter.limit`** to reject values that would overflow the `(int)` cast.
2. **Replace the unsafe cast** with a bounds-checked clamp:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
3. **Add a defensive guard in `findByFilter`**: if `filter.getLimit() < 0`, either throw `IllegalArgumentException` or treat it as no-limit with a logged warning.
4. Apply the same fix to the identical pattern in `PollingTopicMessageRetriever.poll()` at line 68–70. [7](#0-6) 

### Proof of Concept
**Preconditions:**
- Mirror node deployed with `hiero.mirror.grpc.listener.type=POLL`
- A known topic ID (e.g., fee-schedule topic `0.0.111`)

**Steps:**
1. Construct a gRPC `ConsensusTopicQuery` protobuf with:
   - `topic_id = { shardNum: 0, realmNum: 0, topicNum: 111 }`
   - `limit = 2147483648` (i.e., `Integer.MAX_VALUE + 1`)
   - `consensus_start_time = { seconds: 0, nanos: 0 }`
2. Send to `ConsensusService/subscribeTopic`.
3. Validation passes (`2147483648 >= 0`).
4. `PollingTopicListener.poll()` computes `(int)(2147483648L - 0) = Integer.MIN_VALUE`.
5. `Math.min(Integer.MIN_VALUE, 5000) = Integer.MIN_VALUE`.
6. `findByFilter` receives a filter with `limit = Integer.MIN_VALUE`; `hasLimit()` returns `false`; no `setMaxResults()` call.
7. Every 500 ms, the database executes `SELECT * FROM topic_message WHERE topic_id = 111 AND consensus_timestamp >= 0 ORDER BY consensus_timestamp` with no LIMIT clause.
8. Observer receives the full fee-update history streamed back, and the database is subjected to repeated full scans until the subscription is cancelled.
