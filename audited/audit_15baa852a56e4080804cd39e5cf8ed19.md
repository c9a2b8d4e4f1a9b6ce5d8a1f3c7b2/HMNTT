### Title
Unbounded Opcode List Serialization Causes Server-Side Resource Exhaustion (DoS)

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` returns the complete, unpaginated list of all EVM opcodes executed during a transaction replay in a single JSON response. There is no cap on the number of opcodes that can be accumulated in `OpcodeContext`, meaning a single request referencing a maximally gas-intensive historical transaction can force the server to build, hold in memory, and serialize millions of opcode entries. The gzip requirement reduces network transfer size but does not prevent server-side memory and CPU exhaustion during list construction and JSON serialization.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (line 52–68) calls `opcodeService.processOpcodeCall(request)` with no size constraint. [1](#0-0) 

`OpcodeServiceImpl.processOpcodeCall()` (line 67) initializes the opcode list with `gas/3` as an initial ArrayList capacity hint — not a hard limit. [2](#0-1) 

`OpcodeContext.addOpcodes()` (line 54–56) appends every traced opcode unconditionally with no size check. [3](#0-2) 

`OpcodeActionTracer.tracePostExecution()` calls `addOpcodes()` for every single EVM opcode executed during replay. [4](#0-3) 

**Root cause:** The failed assumption is that the gas-based throttle and gzip requirement together bound the server-side resource cost per request. They do not. The gas throttle applies only to the regular `ContractCallService` path; `throttleOpcodeRequest()` only checks a request-count bucket (default: 1 req/sec globally), not gas consumed. [5](#0-4) 

**Exploit flow:**
1. Attacker identifies or deploys a historical transaction that consumed close to the Hedera maximum gas limit (15,000,000 gas). A simple 3-gas opcode (e.g., `PUSH1`) means up to ~5,000,000 opcodes per transaction.
2. Attacker sends: `GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true&storage=true` with `Accept-Encoding: gzip`.
3. Server replays the transaction, calling `addOpcodes()` millions of times, building a multi-million-entry `ArrayList<Opcode>` in heap memory.
4. Spring's Jackson serializer then iterates the entire list to produce a JSON array — consuming additional CPU and memory proportional to list size × per-opcode payload (each opcode includes stack words, memory chunks, and storage maps as hex strings).
5. Server heap is exhausted or GC thrashes, causing OOM or severe latency degradation for all concurrent users.

### Impact Explanation

A single well-crafted request can exhaust JVM heap memory and saturate CPU on the mirror node web3 service, causing a complete denial of service for all users of the endpoint. Because the opcode list is materialized entirely in memory before serialization begins, there is no streaming escape valve. With `memory=true` and `storage=true`, each opcode entry can carry hundreds of bytes of hex-encoded data, multiplying the memory footprint further. The impact is server-wide, not scoped to the attacker's connection.

### Likelihood Explanation

No privileged access is required. Any user who can submit or observe a historical Hedera contract transaction can trigger this. The attacker only needs to know a transaction hash for a gas-heavy contract call — such transactions are publicly visible on the ledger. The global rate limit of 1 opcode request/second does not prevent a single in-flight request from consuming all available heap; it only prevents a second request from being admitted while the first is processing. The attack is trivially repeatable and requires no special tooling beyond a standard HTTP client that sets `Accept-Encoding: gzip`. [6](#0-5) 

### Recommendation

1. **Enforce a hard cap on opcodes collected.** In `OpcodeContext.addOpcodes()`, check `opcodes.size()` against a configurable maximum (e.g., 10,000) and throw a descriptive exception or truncate with a warning when exceeded.
2. **Add pagination to the endpoint.** Introduce `limit` and `offset` (or cursor) query parameters so clients retrieve opcodes in bounded pages, consistent with other paginated endpoints in the REST API (e.g., `/actions`).
3. **Apply gas-based throttling to opcode requests.** Reuse the existing `gasLimitBucket` for opcode requests so that high-gas transactions consume proportionally more throttle tokens, preventing a single max-gas replay from monopolizing the server.
4. **Stream the JSON response.** Use Jackson's streaming API or `StreamingResponseBody` to avoid materializing the entire opcode list in heap before writing begins.

### Proof of Concept

```bash
# 1. Find a high-gas-consumption historical transaction hash on the network.
#    (Publicly visible on Hedera mirror node REST API or hashscan.io)
TX_HASH="0x<high_gas_tx_hash>"

# 2. Send the opcode trace request with all data fields enabled.
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-web3-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
  --output /dev/null

# 3. Observe: server JVM heap climbs to OOM or response takes tens of seconds,
#    blocking the single global opcode rate-limit slot and degrading service
#    for all other users during that window.
```

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
