### Title
Unauthenticated Sustained Range-Scan DoS via Unbounded `schedule_id` Range Filters in `getSchedules()`

### Summary
The `getSchedules()` handler in `rest/schedules.js` accepts multiple `schedule_id` comparator filters (e.g., `gte:X` and `lte:Y`) with no range-width validation, no per-IP rate limiting, and no query cost controls. Each request triggers three database queries — including an expensive `json_agg`/`GROUP BY` aggregation on `transaction_signature` — and the pagination link preserves the upper bound, enabling any unauthenticated client to sustain continuous full-limit DB scans indefinitely.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `extractSqlFromScheduleFilters()` (lines 154–204) iterates all filters and appends each `schedule_id` comparator directly to the SQL `WHERE` clause with no range-width check:

```js
whereQuery += `${filterColumnMap[filter.key]}${filter.operator}$${paramCount}`;
``` [1](#0-0) 

This produces queries like:
```sql
SELECT ... FROM schedule s
WHERE s.schedule_id >= $1 AND s.schedule_id <= $2
ORDER BY s.schedule_id ASC
LIMIT $3
```

Unlike timestamp filters — which are validated against `maxTimestampRangeNs` (7-day default) via `parseTimestampFilters` — there is **no equivalent `maxScheduleIdRange` check** for `schedule_id`. The only range guard in the codebase (`isEmptyRange` in `utils.js` lines 901–946) only prevents generating a next link when `upper < lower`; it does not reject wide initial ranges. [2](#0-1) 

**Three DB queries per request:**

`getSchedules()` (lines 234–282) fires three queries per call:
1. Main schedule range scan
2. `SELECT ... FROM entity WHERE id IN (...)` for up to `limit` IDs
3. `SELECT entity_id, json_agg(...) FROM transaction_signature ts WHERE entity_id IN (...) GROUP BY entity_id` — an aggregation query

Queries 2 and 3 run in parallel via `Promise.all`, but query 3 is expensive due to `json_agg` + `GROUP BY` over potentially many signature rows per schedule. [3](#0-2) 

**Pagination preserves the upper bound:**

`getPaginationLink` is called with `schedulesResponse.schedules.length !== limit` as `isEnd`. When the range returns exactly `limit` rows, `isEnd = false` and a next link is generated. `getNextParamQueries` replaces the `gte:` bound with `gt:lastScheduleId` but **retains the `lte:Y` upper bound**, enabling automated traversal: [4](#0-3) [5](#0-4) 

**No rate limiting on the REST API:**

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module for contract calls. There is no per-IP or per-endpoint rate limiting applied to `GET /api/v1/schedules`. [6](#0-5) 

**`maxRepeatedQueryParameters` does not mitigate this:**

The default is 100 repeated params per key. A range scan requires only 2 (`gte:X` and `lte:Y`), well within the limit. [7](#0-6) 

**Max limit cap does not mitigate this:**

`getLimitParamValue` caps at `responseLimit.max` (default 100). This limits rows per page but does not prevent sustained sequential requests. [8](#0-7) 

### Impact Explanation

An attacker can sustain continuous full-limit DB queries against the mirror node's PostgreSQL database with no authentication and no rate limiting. Each request generates three queries, including an expensive `json_agg`/`GROUP BY` aggregation. With a wide `schedule_id` range (e.g., `gte:1&lte:9999999999`) and `limit=100`, the attacker can page through millions of schedules automatically. Concurrent requests from multiple clients (or a single client with parallel connections) can saturate the DB connection pool (default `maxConnections: 10`) and exhaust statement timeouts (default 20 seconds), degrading or blocking all other REST API endpoints that share the same database. [9](#0-8) 

### Likelihood Explanation

No privileges, credentials, or special knowledge are required. The attack is fully automatable: send the initial request, parse the `links.next` field from the JSON response, and immediately follow it in a loop. The `schedule_id` parameter is documented in the OpenAPI spec as accepting `gte`/`lte` operators, so the attack surface is publicly documented. A single attacker with a script can sustain this indefinitely. [10](#0-9) 

### Recommendation

1. **Add a `maxScheduleIdRange` validation** in `extractSqlFromScheduleFilters`: when both a lower (`gte`/`gt`) and upper (`lte`/`lt`) `schedule_id` bound are present, reject requests where the range exceeds a configured maximum (analogous to `maxTimestampRangeNs` for timestamp filters).
2. **Add per-IP rate limiting** to the REST API (e.g., via an Express middleware such as `express-rate-limit`) for expensive list endpoints.
3. **Increase the DB connection pool** or add a query queue to prevent pool exhaustion under load.
4. **Consider cursor-based pagination** that does not allow arbitrary range re-specification on each page follow.

### Proof of Concept

```bash
# Step 1: Initial wide-range request (unauthenticated)
curl -s "http://<mirror-node>/api/v1/schedules?schedule_id=gte:1&schedule_id=lte:9999999999&limit=100" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data['links']['next'])
"
# Output: /api/v1/schedules?schedule_id=lte:9999999999&schedule_id=gt:0.0.100&limit=100

# Step 2: Automated loop — follow next links indefinitely
BASE="http://<mirror-node>"
NEXT="/api/v1/schedules?schedule_id=gte:1&schedule_id=lte:9999999999&limit=100"
while [ -n "$NEXT" ]; do
  NEXT=$(curl -s "${BASE}${NEXT}" | python3 -c "
import sys,json; d=json.load(sys.stdin); print(d['links']['next'] or '')
")
done
# Each iteration fires 3 DB queries (including json_agg aggregation).
# Run multiple parallel instances to exhaust the DB connection pool (default: 10 connections).
```

const isEmptyRange = (key, value) => {
  const values = Array.isArray(value) ? value : [value];
  let lower = null;
  let upper = null;

  for (const v of values) {
    if (!gtLtPattern.test(v)) {
      continue;
    }

    const filter = buildComparatorFilter(key, v);
    formatComparator(filter);

    // formatComparator doesn't handle CONTRACT_ID and SLOT
    if (key === constants.filterKeys.CONTRACT_ID) {
      filter.value = EntityId.parse(filter.value).getEncodedId();
    } else if (key === constants.filterKeys.SLOT) {
      filter.value = addHexPrefix(filter.value);
    }

    if (filter.value == null) {
      continue;
    }

    let parsed = BigInt(filter.value);
    switch (filter.operator) {
      case opsMap.gt:
        parsed += 1n;
      case opsMap.gte:
        lower = lower === null ? parsed : bigIntMax(lower, parsed);
        break;
      case opsMap.lt:
        parsed -= 1n;
      case opsMap.lte:
        upper = upper === null ? parsed : bigIntMin(upper, parsed);
        break;
      default:
        break;
    }
  }

  if (lower === null || upper === null) {
    return false;
  }

  return upper < lower;
