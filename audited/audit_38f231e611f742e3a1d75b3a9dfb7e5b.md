### Title
Unauthenticated DB Connection Pool Exhaustion via Timestamp + TransactionType Filter Combination in `getOneAccount()`

### Summary
The `getOneAccount()` handler in `rest/accounts.js` accepts `timestamp` and `transactiontype` query parameters from any unauthenticated caller and passes them directly to `transactions.doGetTransactions()`, which executes a multi-table UNION/FULL-OUTER-JOIN query across `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction`. With no rate limiting middleware and a default DB connection pool of only 10 connections (each held for up to 20 seconds per `statementTimeout`), a small number of concurrent requests is sufficient to exhaust the pool and render the mirror node REST API unavailable.

**Root cause â€” failed assumptions:**

1. No rate-limiting middleware exists in `rest/server.js` (lines 67â€“144); the search for `rateLimit`/`throttle` in `rest/**` returns only OpenAPI spec entries, not actual middleware.
2. The DB pool defaults to **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, docs line 556) with a **20-second statement timeout** (docs line 557).
3. Each `getOneAccount` request with timestamp+transactiontype fires **3+ concurrent DB queries** (balance snapshot range lookup, entity query, transaction timestamps query, then transaction details query).
4. `maxRepeatedQueryParameters` (checked in `buildFilters`, `utils.js` lines 1241â€“1248) only limits repeated params within a single request; it does not throttle concurrent requests.

### Impact Explanation

With 10 pool connections and a 20-second statement timeout, as few as **4â€“5 concurrent requests** each holding 2â€“3 connections can exhaust the pool. Subsequent requests queue or fail with connection timeout errors, making the mirror node REST API completely unresponsive. Note: the mirror node is a read-only replica and does not participate in Hedera network consensus â€” the actual transaction confirmation layer is unaffected. The real impact is **full REST API unavailability** (HTTP 500/timeout for all endpoints sharing the same pool), which disrupts wallets, explorers, and dApps relying on the mirror node.

### Likelihood Explanation

The endpoint is fully public (`/api/v1/accounts/{id}?timestamp=gte:0&transactiontype=CRYPTOTRANSFER`), requires no credentials, and is trivially discoverable from the OpenAPI spec. The attack requires only a basic HTTP client capable of sending concurrent requests. The small default pool size (10) makes exhaustion achievable with minimal tooling (e.g., `ab -c 10 -n 1000` or a simple async script). The 20-second statement timeout means each "slot" is occupied long enough to sustain the exhaustion continuously.

### Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) in `rest/server.js` before route handlers, scoped per IP and per endpoint.
2. **Reduce `statementTimeout`** for read-only mirror API queries to 5â€“8 seconds to free connections faster under load.
3. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer is already referenced in `charts/hedera-mirror/values.yaml` lines 358â€“376 for production, but the default Node.js pool is 10).
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) in `getOneAccount()` to cap simultaneous in-flight DB queries.
5. Consider making the `transactions` sub-query opt-in (require `transactions=true` explicitly) rather than opt-out, to reduce default query complexity.

  if (result === null) {
    return {rows: []};
  }

    timestampQuery,

    limitQuery,
    creditDebitQuery,
    transactionTypeQuery,
