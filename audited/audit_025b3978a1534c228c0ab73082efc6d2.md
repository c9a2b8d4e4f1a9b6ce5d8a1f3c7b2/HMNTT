### Title
Cache Pollution via Unauthenticated Token-Type Queries Filling `tokenType` Cache with `Optional.empty()` Entries

### Summary
The `tokenType` Caffeine cache (`maximumSize=100000, expireAfterAccess=24h`) is populated by `TokenRepository.findTypeByTokenId()`, which uses `unless = "#result == null"`. Because the method returns `Optional<TokenTypeEnum>`, a miss for a non-existent token yields `Optional.empty()` — a non-null object — which is therefore cached. Any unauthenticated caller can drive 100,000 distinct non-existent token-ID queries through the public `/api/v1/contracts/call` endpoint, filling the cache with useless entries, evicting legitimate cached results, and sustaining elevated DB query rates for 24 hours per fill cycle.

### Finding Description

**Exact code path:**

`EvmConfiguration.java` lines 129–135 — `cacheManagerTokenType()` creates the Caffeine cache with the spec from `CacheProperties.getTokenType()`: [1](#0-0) 

`CacheProperties.java` line 51 — default spec is `expireAfterAccess=24h,maximumSize=100000,recordStats`: [2](#0-1) 

`TokenRepository.java` lines 23–29 — `findTypeByTokenId()` is annotated `@Cacheable(..., unless = "#result == null")` and returns `Optional<TokenTypeEnum>`: [3](#0-2) 

**Root cause:** Spring's `unless` SpEL expression evaluates `#result` against the actual return value. When a token does not exist, the JPA query returns no rows and Spring Data wraps that as `Optional.empty()`. `Optional.empty()` is a non-null Java object, so `#result == null` is `false` and the empty Optional is stored in the cache. There is no secondary guard such as `unless = "#result == null || !#result.isPresent()"`.

**Throttle is global, not per-IP:** `ThrottleManagerImpl` consumes from a single shared `rateLimitBucket` (default 500 req/s): [4](#0-3) [5](#0-4) 

A single attacker can consume the entire global budget. At 500 req/s, 100,000 distinct token IDs are submitted in ~200 seconds.

**Exploit flow:**
1. Attacker crafts `POST /api/v1/contracts/call` requests encoding the `getTokenType(address)` HTS precompile selector with 100,000 distinct addresses that map to non-existent token IDs.
2. Each request reaches `findTypeByTokenId(tokenId)` → cache miss → DB `SELECT t.type FROM token t WHERE token_id = ?` → empty result → `Optional.empty()` stored in cache.
3. After 100,000 unique IDs the cache is at `maximumSize`. Caffeine begins evicting existing entries (LRU/W-TinyLFU) to make room.
4. Legitimate queries for real tokens whose entries were evicted now miss the cache and hit the DB.
5. The attacker rotates through a second batch of 100,000 IDs to maintain cache pressure indefinitely; each new ID causes a DB lookup and an eviction.
6. The 24-hour `expireAfterAccess` window means the poisoned entries persist as long as the attacker keeps touching them.

### Impact Explanation

- **Elevated DB query rate:** Every legitimate token-type lookup that was evicted now requires a DB round-trip. Under normal operation the cache absorbs these reads; under attack the cache is full of useless `Optional.empty()` entries and the DB absorbs the full legitimate read load.
- **Memory footprint:** 100,000 cache entries held for up to 24 hours. While each entry is small, the sustained allocation prevents GC reclamation and keeps the JVM heap inflated.
- **Global rate-limit starvation:** Because the throttle bucket is shared, the attacker's 500 req/s consumption leaves zero capacity for legitimate users, causing HTTP 429 responses for the duration of the attack.
- **Severity:** Medium-High. The attack is bounded by the global throttle but that throttle is itself the denial-of-service vector; the cache pollution compounds the impact by degrading DB performance after the initial fill.

### Likelihood Explanation

- No authentication is required for `POST /api/v1/contracts/call`.
- The attacker needs only an HTTP client and a list of 100,000 arbitrary 20-byte addresses; no on-chain assets or privileged credentials are needed.
- The attack is repeatable every 24 hours (or continuously with rotating IDs).
- The global (non-per-IP) throttle means a single host can saturate the entire service capacity.

### Recommendation

1. **Fix the `unless` condition** to also exclude empty Optionals:
   ```java
   unless = "#result == null || (#result instanceof T(java.util.Optional) && !#result.isPresent())"
   ```
   or more simply, change the return type to a nullable `TokenTypeEnum` and keep `unless = "#result == null"`.

2. **Add per-IP rate limiting** in addition to the global bucket so a single source cannot consume the entire request budget.

3. **Reduce `maximumSize` and `expireAfterAccess`** for the `tokenType` cache to limit the blast radius (e.g., `maximumSize=10000,expireAfterAccess=1h`).

4. **Validate token addresses** before invoking the repository: reject or short-circuit requests for addresses that cannot correspond to a valid Hedera entity ID range.

### Proof of Concept

```bash
# Generate 100,000 distinct non-existent token addresses and send getTokenType calls
# getTokenType selector: 0x93272baf
for i in $(seq 1 100000); do
  TOKEN_ADDR=$(printf "0x%040x" $((9000000 + i)))  # IDs above any real token
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{
      \"data\": \"0x93272baf000000000000000000000000${TOKEN_ADDR:2}\",
      \"to\": \"0x0000000000000000000000000000000000000167\",
      \"gas\": 300000,
      \"estimate\": false,
      \"block\": \"latest\"
    }" &
done
# After ~200 seconds at 500 req/s the tokenType cache is full of Optional.empty() entries.
# Subsequent legitimate getTokenType calls for real tokens miss the cache and hit the DB.
```
