### Title
Unauthenticated DB Connection Pool Exhaustion via Non-Existent Topic ID Flooding

### Summary
The `getTopicMessages()` handler in `rest/topicmessage.js` performs no rate limiting and always executes a database query for any format-valid `topicId`, including non-existent ones. When the default response limit is used (limit ≠ 1), `queryQuietly` explicitly checks out a dedicated connection from the finite pool. An unprivileged attacker can flood the endpoint with concurrent requests using a valid-format but non-existent `topicId`, exhausting the DB connection pool and denying service to legitimate users.

### Finding Description

**Code path:**

`rest/topicmessage.js`, `getTopicMessages()`, lines 116–160:

```js
const getTopicMessages = async (req, res) => {
  const topicIdStr = req.params.topicId;
  validateGetTopicMessagesParams(topicIdStr);   // format-only check
  ...
  const {query, params, order, limit} = await extractSqlFromTopicMessagesRequest(topicId, filters);
  const queryHint = limit !== 1 ? constants.zeroRandomPageCostQueryHint : undefined;
  const messages = await getMessages(query, params, queryHint);  // always hits DB
``` [1](#0-0) 

`validateGetTopicMessagesParams()` (lines 49–53) only checks that the topicId matches the entity ID format — it does **not** verify existence: [2](#0-1) 

When `topicMessageLookup` is disabled (the default, confirmed in config tests at line 226: `topicMessageLookup: false`), `extractSqlFromTopicMessagesRequest` skips the early-exit path and always builds and executes the main `SELECT * FROM topic_message WHERE topic_id = $1 ... LIMIT $N` query, even for non-existent topics. [3](#0-2) 

Because `limit !== 1` for the default limit of 25, `queryHint = zeroRandomPageCostQueryHint` is set. In `queryQuietly` (lines 1518–1526 of `rest/utils.js`), a non-null `preQueryHint` causes an **explicit dedicated connection checkout** via `client = await this.connect()`: [4](#0-3) 

The pool is finite, configured via `max: config.db.pool.maxConnections` in `rest/dbpool.js`: [5](#0-4) 

**No rate limiting middleware exists.** The full middleware stack in `rest/server.js` (lines 67–98) contains: `urlencoded`, `json`, `cors`, optional `compression`, `httpContext`, `requestLogger`, `authHandler` (authentication only, not throttling), optional `metricsHandler`, optional `responseCacheCheckHandler`. There is no `express-rate-limit` or equivalent: [6](#0-5) 

**Root cause:** Format-only input validation + unconditional DB query execution + explicit per-request connection checkout + zero rate limiting.

### Impact Explanation

An attacker sending N concurrent requests where N ≥ `maxConnections` (default pool size) will hold all pool slots simultaneously. New legitimate requests will block waiting for `connectionTimeoutMillis` and then fail with a connection timeout error, effectively making the `/topics/:topicId/messages` endpoint (and all other endpoints sharing the same pool) unavailable. This is a full availability denial for the REST API with no authentication or special privilege required.

### Likelihood Explanation

The attack requires only an HTTP client capable of sending concurrent requests — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The topicId `0.0.999999999` is a valid format and requires zero knowledge of the system. The attack is repeatable indefinitely, stateless, and requires no prior account or credential. Any internet-accessible deployment is exposed.

### Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) globally in `rest/server.js` before route handlers, with per-IP limits.
2. **Add a topic existence pre-check** before executing the main query (a lightweight `SELECT 1 FROM entity WHERE id = $1 AND type = 'TOPIC'`), so non-existent topics return 404 without consuming a dedicated pool connection.
3. **Increase pool size** and/or use a connection queue with a bounded wait to limit blast radius.
4. Consider enabling `topicMessageLookup` (which provides an early-exit path when the lookup table returns no rows), though this still consumes a connection for the lookup query itself.

### Proof of Concept

```bash
# Send 20 concurrent requests with a valid-format but non-existent topicId
# Adjust -c to match or exceed maxConnections
wrk -t4 -c20 -d30s \
  "http://<mirror-node-host>/api/v1/topics/0.0.999999999/messages"

# In a separate terminal, observe legitimate requests timing out:
curl -v "http://<mirror-node-host>/api/v1/topics/0.0.1/messages"
# Expected: connection timeout or 503 after connectionTimeoutMillis
```

Preconditions: unauthenticated access to the REST API, no WAF/rate-limiter in front of the service.
Trigger: concurrent flood of GET requests to `/api/v1/topics/0.0.999999999/messages`.
Result: DB pool exhausted → legitimate requests fail with timeout errors.

  const filters = utils.buildAndValidateFilters(req.query, acceptedTopicsParameters);
