### Title
Silent Message Exclusion via `Long.MAX_VALUE` startTime Injection in `ConsensusController.convertTimestamp()`

### Summary
Any unauthenticated gRPC client can send a `ConsensusTopicQuery` with `consensusStartTime.seconds = 9223372035` (or higher), causing `convertTimestamp()` to silently substitute `Long.MAX_VALUE` as the `startTime` filter. Because the `isValidStartTime()` Bean Validation constraint on `TopicMessageFilter` is never enforced (the `subscribeTopic` method parameter lacks `@Valid`), the filter is accepted and both the historical retriever and the live listener return zero messages with no error signal to the caller.

### Finding Description

**Exact code path:**

`ConsensusController.java` lines 76–81 — `convertTimestamp()`:
```java
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {   // ← attacker triggers this branch
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
```
`Long.MAX_VALUE` = `9223372036854775807` ns ≈ year 2262.

`ConsensusController.java` lines 62–65 — `toFilter()` passes the value directly into the filter:
```java
if (query.hasConsensusStartTime()) {
    long startTime = convertTimestamp(query.getConsensusStartTime());
    filter.startTime(startTime);          // startTime = Long.MAX_VALUE
}
```

**Root cause — validation guard is dead code:**

`TopicMessageFilter.java` lines 48–51 defines:
```java
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();   // Long.MAX_VALUE > now() → false
}
```
`DomainUtils.now()` returns current epoch-nanoseconds (~`1.7 × 10¹⁸`), far below `Long.MAX_VALUE` (`9.2 × 10¹⁸`), so this constraint would reject the value — **but it is never triggered**. `TopicMessageServiceImpl` carries `@Validated` at the class level, yet the `subscribeTopic(TopicMessageFilter filter)` method signature has no `@Valid` or `@Validated` on the `filter` parameter. Spring's method-level validation proxy only intercepts parameters that are explicitly annotated; without the annotation the constraint is silently skipped.

**Downstream effect — zero records, no error:**

1. `TopicMessageRepositoryCustomImpl.findByFilter()` line 40 builds:
   `WHERE consensus_timestamp >= 9223372036854775807` — no stored message ever satisfies this; historical retrieval returns empty.

2. `CompositeTopicListener.filterMessage()` line 63:
   `message.getConsensusTimestamp() >= filter.getStartTime()` — always `false` for real messages; live stream returns nothing.

3. `TopicMessageServiceImpl.subscribeTopic()` emits an empty `Flux` that completes (or hangs indefinitely if no `endTime` is set) with no error propagated to the caller.

### Impact Explanation

A subscriber receives zero topic messages with no indication of why. For mirror-node operators or downstream consumers relying on this gRPC stream to replicate HCS data, the result is a silent data gap: the subscription appears healthy (no error, no disconnect) but exports nothing. If the subscription is used for automated data synchronisation, the gap goes undetected until an out-of-band audit. Severity is **Medium**: data availability is broken for the affected subscription; other subscribers are unaffected.

### Likelihood Explanation

The gRPC endpoint is unauthenticated and publicly reachable. The trigger value (`seconds = 9223372035`) is a single integer field in a standard protobuf message — trivially crafted with any gRPC client. No special privileges, tokens, or knowledge of internal state are required. The condition is deterministic and 100% reproducible.

### Recommendation

1. **Enforce the existing validation**: Add `@Valid` to the `filter` parameter in `TopicMessageService.subscribeTopic()` and `TopicMessageServiceImpl.subscribeTopic()`:
   ```java
   Flux<TopicMessage> subscribeTopic(@Valid TopicMessageFilter filter);
   ```
   This activates the already-written `isValidStartTime()` guard and returns a proper validation error to the caller instead of silently returning zero records.

2. **Reject or clamp in `convertTimestamp()`**: Rather than silently substituting `Long.MAX_VALUE`, throw an `IllegalArgumentException` (or return a sentinel that `toFilter()` rejects) when the supplied timestamp exceeds a reasonable future bound.

3. **Add an integration test** that sends `consensusStartTime.seconds = 9223372035` and asserts a non-2xx / non-OK gRPC status is returned rather than an empty stream.

### Proof of Concept

```python
# grpcurl / Python pseudocode
import grpc
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from google.protobuf.timestamp_pb2 import Timestamp

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=...,                          # any valid topic
    consensusStartTime=Timestamp(seconds=9223372035, nanos=0),  # triggers Long.MAX_VALUE path
)

responses = list(stub.subscribeTopic(query))
assert len(responses) == 0   # zero messages returned, no error raised
```

**Expected (correct) behaviour:** gRPC `INVALID_ARGUMENT` status.  
**Actual behaviour:** empty stream, `OK` status, zero messages exported.
