### Title
LRU Token Cache Eviction via Unauthenticated Bulk Cache-Miss Requests to `/accounts/{id}/tokens`

### Summary
Any unauthenticated user can repeatedly request `/accounts/{id}/tokens` for accounts whose associated token IDs are not in the LRU cache. Each such request causes `getCachedTokens()` to issue a bulk `tokenCacheQuery` for all uncached token IDs in the page and then unconditionally populate the LRU cache, evicting previously cached entries. By cycling through accounts with different token associations, an attacker can continuously pollute the cache and degrade hit rates for concurrent legitimate users.

### Finding Description

**Code path:**

1. `rest/controllers/tokenController.js` `getTokenRelationships()` (lines 66–92) — no authentication, no rate limiting. Any caller can reach this handler. [1](#0-0) 

2. `getTokenRelationships()` calls `TokenService.getTokenAccounts(query)`, which collects **all** token IDs from the DB result page into a `Set` and passes them to `getCachedTokens()`. [2](#0-1) 

3. `getCachedTokens()` separates cached from uncached IDs, then for every uncached ID issues a **single bulk query** (`tokenCacheQuery`: `where token_id = any ($1)`) and unconditionally calls `tokenCache.set(tokenId, cachedToken)` for every result row. [3](#0-2) 

4. The LRU cache is bounded by `config.cache.token.maxSize`. Each `tokenCache.set()` on a full cache evicts the least-recently-used entry. [4](#0-3) 

**Root cause / failed assumption:** The design assumes that across normal traffic the cache will warm up organically and most token IDs in a page will already be cached. There is no guard preventing an attacker from deliberately supplying pages of entirely uncached token IDs, and `getCachedTokens()` unconditionally writes every fetched token into the LRU cache (unlike `putTokenCache`, which skips existing keys). This means an attacker controls which tokens occupy the cache.

**Why existing checks are insufficient:**
- `putTokenCache` (called from `getTokensRequest`) does check `tokenCache.has()` before writing, but `getCachedTokens()` does **not** — it always calls `tokenCache.set()` (line 156), resetting LRU order and triggering eviction. [5](#0-4) 
- The `limit` parameter is validated but only bounded by the configured max (typically 25–100), not reduced further for cache-safety. [6](#0-5) 
- No rate limiting or authentication is enforced on the endpoint. [7](#0-6) 

### Impact Explanation
An attacker can continuously evict legitimately cached token entries, forcing every subsequent legitimate request for those tokens to incur a DB round-trip via `tokenCacheQuery`. At scale this increases DB load and response latency for all concurrent users. The cache provides no benefit when it is continuously polluted. Severity is medium/griefing: no funds are at risk, but service quality degrades proportionally to attack intensity.

### Likelihood Explanation
The attack requires zero privileges and zero on-chain cost: the attacker only needs to identify existing accounts with many token associations (trivially discoverable via the mirror node's own `/accounts` and `/tokens` endpoints). The attack is fully repeatable with simple HTTP tooling (e.g., `curl` in a loop). No special knowledge of internals is required.

### Recommendation
1. **Guard `getCachedTokens()` writes**: Before calling `tokenCache.set()`, check `tokenCache.has(tokenId)` (matching the pattern in `putTokenCache`) to avoid resetting LRU order for already-cached entries and to prevent overwrite-driven eviction.
2. **Rate-limit the endpoint**: Apply per-IP or per-account-ID rate limiting on `/accounts/{id}/tokens`.
3. **Cap the effective cache-miss batch size**: If `uncachedTokenIds.length` exceeds a threshold, split into smaller DB fetches or return partial cache data, limiting per-request cache churn.
4. **Separate hot/cold cache segments**: Use a two-tier cache (small hot LRU + larger cold store) so bulk cold misses cannot evict hot entries.

### Proof of Concept
```bash
# Step 1: Identify accounts with many token associations (no auth needed)
curl "https://<mirror-node>/api/v1/accounts?limit=25&order=desc"

# Step 2: For each such account, repeatedly request its token list with max limit
# Cycle through N different accounts to keep injecting new uncached token IDs
for account_id in 0.0.1001 0.0.1002 0.0.1003 ...; do
  curl "https://<mirror-node>/api/v1/accounts/${account_id}/tokens?limit=100&order=asc"
done

# Each iteration:
# - getTokenRelationships() -> getTokenAccounts() -> getCachedTokens()
# - All token IDs for that account are uncached (or evicted by prior iterations)
# - tokenCacheQuery fires with up to 100 token IDs
# - tokenCache.set() is called for each result, evicting LRU entries
# - Legitimate concurrent users querying those evicted tokens now get cache misses
```

```javascript
  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
```

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
