### Title
Unauthenticated Connection Pool Exhaustion DoS in `getTransactions()` via Unique Timestamp Range Flooding

### Summary
The `getTransactions()` handler in `rest/transactions.js` performs at least one unconditional `pool.queryQuietly()` call per request (the timestamp-range query), with no rate limiting on the endpoint. Because the default DB pool is capped at 10 connections and the per-request timestamp-range query is never cached at the response level, an unauthenticated attacker sending concurrent requests with unique timestamp ranges can saturate the pool, causing all subsequent requests to queue and time out, resulting in a complete denial-of-service of the mirror node REST API.

### Finding Description

**Code path:**

`rest/transactions.js` `getTransactions()` (line 671) → `doGetTransactions()` (line 695) → `getTransactionTimestamps()` (line 701) → `pool.queryQuietly(query, params)` (line 483). A second `pool.queryQuietly()` is issued at line 662 (`getTransactionsDetails`) for any cache-missing transaction details.

`rest/dbpool.js` configures the pool with `max: config.db.pool.maxConnections` (line 14), defaulting to **10** connections (`docs/configuration.md` line 556).

`rest/utils.js` `queryQuietly` (line 1481): when no `preQueryHint` is supplied (the case for both calls above), it falls through to `this.query(query, params)` (line 1520), which in `node-postgres` checks out a connection for the full duration of the query before releasing it.

**Root cause — two failed assumptions:**

1. The timestamp-range query in `getTransactionTimestamps` is **never cached**. The Redis transaction-detail cache (`cache.get()` at line 705) is keyed per `consensus_timestamp` of individual transactions, not per the incoming timestamp-range filter. The full-response Redis cache (`responseCacheHandler.js`) is **disabled by default** (`cache.response.enabled: false`, docs line 549). Every unique `?timestamp=gt:X&timestamp=lt:Y` combination always hits the database.

2. There is **no rate limiting** on the REST API. `server.js` registers `getTransactions` at line 132 with no throttle middleware. The only throttling in the codebase is in the Java `web3/` module (`ThrottleConfiguration.java`), which is a separate service. The `rest/` middleware stack (`server.js` lines 68–98) contains: CORS, compression, httpContext, requestLogger, authHandler (optional Basic Auth for response-limit overrides only), optional metrics, optional response-cache check — no request-rate limiter.

**Exploit flow:**

An attacker sends N concurrent `GET /api/v1/transactions?timestamp=gt:T_i&timestamp=lt:T_i+1` requests where each `T_i` is a distinct nanosecond-precision timestamp. Each request:
1. Passes filter validation (valid timestamp syntax).
2. Calls `getTransactionTimestamps` → `pool.queryQuietly()` → acquires one of the 10 pool connections for up to `statementTimeout` (20 000 ms default).
3. On cache miss, calls `getTransactionsDetails` → second `pool.queryQuietly()`.

With N ≥ 10 concurrent slow queries, the pool is saturated. New requests queue inside `node-postgres`. The queue drains only when a connection is released (up to 20 s per query). The attacker sustains saturation by sending ≥ 1 new request every 2 s (10 connections ÷ 20 s timeout), which is trivially achievable from a single machine.

**Why existing checks are insufficient:**

- `statementTimeout = 20 000 ms` (line 557): limits individual query duration but does not prevent the attacker from immediately re-filling released slots.
- `connectionTimeoutMillis = 20 000 ms` (line 555): causes queued legitimate requests to fail with a timeout error rather than queue indefinitely, but this means legitimate users receive errors, not that the attack is mitigated.
- `authHandler` (line 86 of `server.js`): entirely optional; unauthenticated requests proceed normally.
- `maxRepeatedQueryParameters = 100` (docs line 581): limits repeated params per request, not concurrent requests.

### Impact Explanation

The mirror node REST API becomes unavailable to all legitimate consumers. Queries for transaction history, account balances, and all other endpoints sharing the same pool return `503`/timeout errors for the duration of the attack. The mirror node is a read-only service and does not participate in Hedera consensus; transaction *confirmation* on the network itself is unaffected. However, the REST API is the primary interface for wallets, explorers, and dApps to verify transaction status, so sustained unavailability constitutes a significant service-level impact. Severity: **High** (DoS of public API, no authentication required, trivially repeatable).

### Likelihood Explanation

No special privileges, accounts, or credentials are required. The attack requires only HTTP access to the public endpoint and the ability to open ~10–20 concurrent TCP connections. Any commodity machine or small botnet can sustain the required request rate. The attack is repeatable indefinitely and leaves no persistent state to clean up. The default pool size of 10 makes the threshold extremely low.

### Recommendation

1. **Add a per-IP (or global) rate limiter** to the Express middleware stack before route handlers, e.g., `express-rate-limit` or `express-slow-down`, targeting the `/api/v1/transactions` route.
2. **Increase `maxConnections`** for production deployments and document a minimum safe value relative to expected concurrency.
3. **Cache the timestamp-range query result** (first `pool.queryQuietly` in `getTransactionTimestamps`) in Redis with a short TTL (e.g., 1–2 s), keyed on the normalized filter parameters, so repeated or near-identical range queries are served from cache.
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) capping the number of simultaneous in-flight DB queries from the REST layer.
5. Consider enabling the full-response Redis cache (`cache.response.enabled: true`) as a first line of defence for repeated identical requests.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent unique-range requests
for i in $(seq 1 50); do
  START=$((1600000000000000000 + i * 1000000000))
  END=$((START + 500000000))
  curl -s "http://<mirror-node>/api/v1/transactions?timestamp=gt:${START}&timestamp=lt:${END}" &
done
wait

# Legitimate request now times out or returns 503
curl -v "http://<mirror-node>/api/v1/transactions"
```

Expected result: the final `curl` either hangs for ~20 s (connection timeout) or receives a 503/error response while the pool is saturated. Repeat the flood loop every 15 s to maintain saturation indefinitely.
