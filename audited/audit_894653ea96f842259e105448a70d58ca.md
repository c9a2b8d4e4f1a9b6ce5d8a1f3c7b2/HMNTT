### Title
Unbounded HTTP Connection Hold in `getTransactions()` Due to Missing Client-Side Query Timeout and pgbouncer Stripping `statement_timeout`

### Summary
`getTransactions()` in `rest/transactions.js` issues two sequential `pool.queryQuietly()` calls with no client-side timeout guard. The only server-side protection (`statement_timeout`) is explicitly stripped by the production pgbouncer configuration before it reaches PostgreSQL. During a network partition between Node.js and PostgreSQL, the TCP socket stalls indefinitely, holding every in-flight HTTP connection open with no bound, exhausting the Node.js event loop and causing all subsequent requests to queue without limit. No privilege is required to trigger this condition.

### Finding Description

**Code path:**

`getTransactions()` (line 671) calls `doGetTransactions()` (line 676), which sequentially `await`s:
1. `getTransactionTimestamps()` → `pool.queryQuietly(query, params)` (timestamp-range query)
2. `getTransactionsDetails()` → `pool.queryQuietly(query, params)` (line 662, large CTE with `crypto_transfer` and `token_transfer` joins)

`queryQuietly` (in `rest/utils.js:1481`) calls `this.query(query, params)` (line 1520) — a raw `node-postgres` query with no `query_timeout` option and no `AbortController`/`Promise.race` wrapper.

**Root cause — three compounding failures:**

1. **No client-side `query_timeout`.**
   `rest/dbpool.js:7-16` configures the pool with only `statement_timeout: config.db.pool.statementTimeout`. In `node-postgres`, `statement_timeout` is a *startup parameter* sent to PostgreSQL at connection time — it is not a client-side timer. The `query_timeout` option (which arms a `setTimeout` inside the `pg` client itself) is absent entirely.

2. **pgbouncer explicitly strips `statement_timeout`.**
   `charts/hedera-mirror/values.yaml:361`:
   ```
   ignore_startup_parameters: extra_float_digits,options,statement_timeout
   ```
   pgbouncer discards the `statement_timeout` startup parameter before forwarding the connection to PostgreSQL. The pool-level `statement_timeout` in `dbpool.js:15` therefore never reaches the database in the production deployment.

3. **No HTTP-level request timeout.**
   `rest/server.js:149-166` starts the HTTP server with `app.listen()` and no `server.setTimeout()` call. No timeout middleware is registered in the Express pipeline between `requestLogger` and `handleError`.

**Exploit flow:**

During a network partition between the Node.js process and PostgreSQL, the TCP socket for an existing pool connection stalls: no bytes flow in either direction. PostgreSQL's server-side `statement_timeout` (even the role-level default of 20 000 ms set in the Helm SQL) cannot fire because the error notification cannot traverse the partitioned link. The `pg` client's `this.query()` call is awaiting a response that will never arrive. Because `queryQuietly` has no `query_timeout` and no `AbortController`, the `await` in `doGetTransactions` never resolves. The Express handler never calls `next()`, so the HTTP response is never sent and the TCP connection is held open indefinitely.

### Impact Explanation

Each stalled request occupies:
- One Node.js async context (microtask queue entry)
- One HTTP keep-alive socket (file descriptor)
- One pg pool slot (up to `maxConnections`, default 10)

With the default pool size of 10 connections, 10 concurrent requests during a partition saturate the pool. All subsequent requests queue at the pool level (`connectionTimeoutMillis` = 20 000 ms default), then at the OS TCP accept queue. The Node.js event loop is not blocked (it is async), but every new request that arrives while the pool is saturated will itself hang waiting for a pool slot, compounding the exhaustion. The service becomes completely unresponsive to all endpoints (not just `/transactions`) because the shared pool is exhausted. This is a full denial-of-service with no authentication required.

### Likelihood Explanation

Network partitions between application pods and database pods are a realistic operational event (Kubernetes network policy changes, CNI bugs, cloud provider transient failures, rolling restarts). An attacker who can influence network routing (e.g., cloud tenant with access to shared infrastructure, or an insider) can deliberately induce a partial partition. Even without adversarial intent, a natural partition combined with any sustained request traffic (including legitimate users) triggers the condition. The exploit requires only the ability to send HTTP GET requests to `/api/v1/transactions` — no credentials, no special headers.

### Recommendation

Apply defense in depth at three layers:

1. **Client-side query timeout in `rest/dbpool.js`:**
   Add `query_timeout: <ms>` to `poolConfig` (e.g., matching `statementTimeout`). This arms a `setTimeout` inside `node-postgres` that rejects the query promise from the client side, independent of database reachability.

2. **HTTP request timeout in `rest/server.js`:**
   After `app.listen()`, call `server.setTimeout(<ms>)` (e.g., 30 000 ms) to close stalled HTTP connections at the transport layer.

3. **Do not ignore `statement_timeout` in pgbouncer:**
   Remove `statement_timeout` from `ignore_startup_parameters` in `charts/hedera-mirror/values.yaml:361`, or ensure the role-level default is set and pgbouncer is in session pool mode for `mirror_rest` so per-connection settings are honoured.

### Proof of Concept

**Preconditions:** Access to the mirror-node REST API (no credentials needed). Ability to induce a network partition between the Node.js pod and PostgreSQL (e.g., a Kubernetes `NetworkPolicy` drop rule, or simply a firewall rule that drops — not rejects — TCP packets on port 5432).

**Steps:**

```bash
# 1. Establish baseline — confirm /transactions responds normally
curl -s "http://<mirror-node>/api/v1/transactions?limit=1" | jq .

# 2. Induce a silent TCP drop partition (example: iptables on the DB host)
#    iptables -I INPUT -p tcp --dport 5432 -j DROP
#    (existing connections stall; no RST is sent)

# 3. Send 10 concurrent requests (saturates the default pool of 10)
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/transactions" &
done
wait

# 4. Send one more request — observe it hangs indefinitely (pool exhausted)
curl -v --max-time 120 "http://<mirror-node>/api/v1/transactions"
# Expected: no response for the full 120 s; server is unresponsive

# 5. Confirm all other endpoints are also unresponsive (shared pool)
curl -v --max-time 10 "http://<mirror-node>/api/v1/accounts"
# Expected: hangs / times out
```

The server remains unresponsive until the OS-level TCP keepalive timer fires (default ~2 hours on Linux) or the partition is lifted.
