### Title
Unauthenticated EVM Address Lookup Causes Unconditional DB Query with No Negative-Result Caching in `getTokenRelationships()`

### Summary
Any unauthenticated caller can supply a syntactically valid but non-existent EVM address to `GET /api/v1/accounts/{evmAddress}/tokens`, causing `EntityService.getEntityIdFromEvmAddress()` to unconditionally issue a database query on every request. Because there is no negative-result cache and no rate limiting in the REST service, an attacker can sustain arbitrarily high DB query rates using the 2¹⁶⁰ available valid EVM address strings, degrading database performance for all users.

### Finding Description

**Exact code path:**

`getTokenRelationships()` calls `EntityService.getEncodedId()` with `requireResult = true` (default): [1](#0-0) 

Inside `getEncodedId()`, when the input passes `EntityId.isValidEntityId()` and resolves to an opaque EVM address (`entityId.evmAddress !== null`), it calls `getEntityIdFromEvmAddress(entityId, requireResult)`: [2](#0-1) 

`getEntityIdFromEvmAddress()` issues the DB query **unconditionally** before any result check: [3](#0-2) 

The SQL executed is: [4](#0-3) 

**Root cause — two missing controls:**

1. **No negative-result caching for DB lookups.** `entityId.js` contains a `quickLru` cache, but it only caches the *parsed `EntityId` object* (shard/realm/num/evmAddress fields), not the DB lookup result: [5](#0-4) 
The DB query in `getEntityIdFromEvmAddress()` is re-issued on every request, even for the same address.

2. **No rate limiting in the REST service.** `server.js` registers no rate-limiting middleware: [6](#0-5) 
The optional Redis response cache (`responseCacheCheckHandler`) only activates when both `config.cache.response.enabled` and `config.redis.enabled` are true: [7](#0-6) 
Even when enabled, it caches the 404 response keyed by the specific EVM address string — an attacker using a fresh address each request bypasses it entirely.

**Why the EVM address space makes this unbounded:** `isValidEvmAddress` accepts any 40-hex-character string: [8](#0-7) 
There are 2¹⁶⁰ ≈ 10⁴⁸ valid addresses, so an attacker trivially generates unique non-existent addresses indefinitely.

### Impact Explanation
Each crafted request forces a full indexed scan of the `entity` table on `evm_address`. Under sustained flood (e.g., thousands of requests/second from a botnet or a single high-throughput client), the database connection pool saturates and query latency rises for all legitimate users. This is a griefing/DoS at the DB layer with no economic cost to the attacker and no privileged access required.

### Likelihood Explanation
The endpoint is public, unauthenticated, and reachable from the internet. Generating valid 40-hex-char strings requires zero knowledge of the system. A single attacker with a modest HTTP client can sustain thousands of requests per second. No exploit tooling is needed beyond `curl` in a loop with randomized addresses.

### Recommendation
1. **Cache negative DB results**: In `getEntityIdFromEvmAddress()`, store a sentinel (e.g., `null`) in an in-process LRU cache keyed by the hex EVM address bytes, with a short TTL (e.g., 30 s). Return the cached miss immediately without hitting the DB.
2. **Add rate limiting**: Apply a per-IP or global rate limiter (e.g., `express-rate-limit`) in `server.js` before route handlers, specifically for the `/accounts/:id/tokens` path.
3. **Ensure Redis response cache is on by default** in production deployments, so repeated identical-address requests are short-circuited before reaching the DB.

### Proof of Concept
```bash
# Generate and fire 10,000 requests with unique random EVM addresses
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/${ADDR}/tokens" &
done
wait
```
Each request triggers one `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query. With no rate limiting and no negative cache, all 10,000 queries reach the database. Observe DB CPU and connection pool exhaustion in monitoring.
