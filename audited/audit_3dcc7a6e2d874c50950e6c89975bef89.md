### Title
`BlockType` Reference-Equality Bypass via Decimal `Long.MAX_VALUE` Block Parameter Causes `BlockNumberNotFoundException`

### Summary
`BlockType.of("9223372036854775807")` (Long.MAX_VALUE as a decimal string) creates a **new** `BlockType` instance with `number=Long.MAX_VALUE`, distinct from the `BlockType.LATEST` singleton. `RecordFileServiceImpl.findByBlockType()` uses `==` (reference equality) to detect `LATEST`, so this new instance bypasses the check, falls through to `recordFileRepository.findByIndex(Long.MAX_VALUE)`, returns `Optional.empty()`, and causes `BlockNumberNotFoundException` — making any `eth_call` with that block parameter fail unconditionally.

### Finding Description

**Code path:**

`BlockType.java` line 30 defines the singleton:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [1](#0-0) 

`BlockType.of()` handles the decimal branch (lines 59–65):
```java
final var decimal = matcher.group(GROUP_DECIMAL);
if (decimal != null) {
    return new BlockType(value, Long.parseLong(decimal, 10));  // NEW object, not LATEST
}
``` [2](#0-1) 

The regex `(\d{1,20})` matches "9223372036854775807" (19 digits), `Long.parseLong` succeeds without overflow, and a **new** `BlockType("9223372036854775807", Long.MAX_VALUE)` is returned — not the `LATEST` singleton.

`findByBlockType()` then uses `==` identity checks:
```java
} else if (block == BlockType.LATEST) {   // FALSE — different reference
    return recordFileRepository.findLatest();
}
return recordFileRepository.findByIndex(block.number()); // findByIndex(Long.MAX_VALUE) → empty
``` [3](#0-2) 

`ContractCallService.callContract()` wraps the result with `orElseThrow`:
```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
``` [4](#0-3) 

`Optional.empty().orElseThrow(...)` throws `BlockNumberNotFoundException`, which the controller maps to HTTP 400 "Unknown block number". [5](#0-4) 

**Root cause:** The `blockTypeForTag` switch correctly returns the `LATEST` singleton for string tags (`"latest"`, `"pending"`, `"safe"`, `"finalized"`), but the decimal branch unconditionally allocates a new object, even when the parsed value equals `Long.MAX_VALUE`. The `==` guard in `findByBlockType` then silently misroutes it. [6](#0-5) 

### Impact Explanation
Every `eth_call` (or `eth_estimateGas`) submitted with `block: "9223372036854775807"` or `block: "0x7fffffffffffffff"` fails with HTTP 400 instead of executing against the latest block. Any client or DApp that encodes "latest" as its numeric sentinel value (Long.MAX_VALUE decimal) will receive a permanent, deterministic error. No funds are directly at risk, but smart contract execution is silently denied for a valid-looking input, matching the Medium scope definition.

### Likelihood Explanation
No privileges are required. The attack surface is the public JSON-RPC `eth_call` endpoint. The input is a valid decimal integer that passes all input validation (regex match + `Long.parseLong` succeeds). It is trivially repeatable and requires no special knowledge beyond knowing `Long.MAX_VALUE = 9223372036854775807`. The same issue applies to the hex form `0x7fffffffffffffff`. [7](#0-6) 

### Recommendation
In `BlockType.of()`, after parsing the decimal (or hex) value, check whether the result equals `Long.MAX_VALUE` and return the `LATEST` singleton:
```java
final var num = Long.parseLong(decimal, 10);
if (num == Long.MAX_VALUE) return LATEST;
return new BlockType(value, num);
```
Alternatively, replace the `==` identity checks in `findByBlockType()` with value-based equality using `block.equals(BlockType.LATEST)` (Java records generate `equals()` based on component values), or add an explicit `block.number() == Long.MAX_VALUE && !block.isHash()` guard for the LATEST path.

### Proof of Concept
```bash
# Against a running mirror-node web3 instance:
curl -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0",
    "method":"eth_call",
    "params":[{"to":"0x<valid_contract>","data":"0x<selector>"},"9223372036854775807"],
    "id":1
  }'

# Expected (correct): 200 OK with contract result
# Actual (buggy):     400 Bad Request {"detail":"Unknown block number"}

# Same result with hex form:
# "0x7fffffffffffffff"
```
