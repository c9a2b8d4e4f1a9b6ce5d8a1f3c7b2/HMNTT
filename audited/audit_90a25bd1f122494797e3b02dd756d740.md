### Title
Cache Stampede (Thundering Herd) in `getCachedTokens` Allows Unprivileged DB Resource Exhaustion

### Summary
`TokenService.getCachedTokens()` in `rest/service/tokenService.js` contains no promise-coalescing or mutex guard around the cache-miss → DB-query → cache-populate sequence. Because Node.js yields the event loop at the `await super.getRows(...)` call, N concurrent requests that all arrive before the cache is warm will each independently issue a full DB query for the same token IDs, multiplying DB load by N with zero benefit.

### Finding Description
**Exact code path:** `rest/controllers/tokenController.js` → `getTokenRelationships()` (line 74) → `TokenService.getTokenAccounts()` (`rest/service/tokenService.js`, lines 96–115) → `getCachedTokens()` (lines 136–161).

**Root cause:** The cache check (lines 139–146) and the DB query (line 152) are not atomic. The `quickLru` cache at line 12–14 is a plain synchronous in-memory map with no concept of "pending" entries. The guard at line 148 (`if (uncachedTokenIds.length === 0) return cachedTokens`) only short-circuits when the cache is already warm — it provides zero protection during the initial population window.

**Exploit flow:**
1. The `tokenCache` is cold (e.g., after startup, or after the LRU evicts entries).
2. N concurrent requests arrive for the same account/token IDs.
3. All N coroutines execute lines 139–146 synchronously and each builds a non-empty `uncachedTokenIds` list.
4. Each coroutine reaches `await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])` at line 152 and suspends, yielding the event loop.
5. The remaining N−1 coroutines each repeat steps 3–4 before any of them has returned and written to `tokenCache`.
6. All N DB queries execute in parallel against the same token IDs.
7. All N coroutines then write identical data into `tokenCache` (lines 153–158) — redundant work.

**Why existing checks fail:** The only guard is `uncachedTokenIds.length === 0` (line 148), which is evaluated synchronously before any `await`. It cannot see that another coroutine is already in-flight fetching the same IDs. There is no per-token-ID in-flight promise stored in the cache, no mutex, and no deduplication of concurrent fetches.

### Impact Explanation
An attacker can force O(N) simultaneous DB queries for the same rows by sending N concurrent HTTP GET requests to `/api/v1/accounts/{id}/tokens`. Each query consumes a DB connection from the pool and CPU on the DB server. With a large enough N, this can saturate the connection pool, starve legitimate requests of DB connections, and degrade overall API latency. The impact is resource exhaustion / griefing; no user funds or data integrity are affected, consistent with the "no economic damage" scope.

### Likelihood Explanation
Any unauthenticated user can reach the `/api/v1/accounts/{id}/tokens` endpoint. No API key, authentication, or per-IP rate limit is enforced at the REST service layer for this endpoint (the throttling found in the codebase applies only to the separate `web3` Java service for contract calls). The attack requires only a standard HTTP client capable of issuing concurrent requests (e.g., `curl`, `ab`, `wrk`). It is trivially repeatable: the attacker can re-trigger the stampede whenever the LRU evicts the target entries, or simply target a large enough set of token IDs to exceed `config.cache.token.maxSize`.

### Recommendation
Implement **promise coalescing** (also called "single-flight"): store the in-flight `Promise` itself in the cache keyed by token ID. Any concurrent caller that finds a `Promise` (rather than a resolved `CachedToken`) awaits the same promise instead of issuing a new DB query. Example pattern:

```js
// Replace the plain quickLru with a map that can hold Promises
const tokenId = ...; 
if (!inFlightCache.has(tokenId)) {
  inFlightCache.set(tokenId, fetchFromDb(tokenId).then(result => {
    inFlightCache.set(tokenId, result); // replace promise with value
    return result;
  }));
}
return inFlightCache.get(tokenId); // may be a Promise or a value
```

Alternatively, batch all uncached IDs into a single shared promise keyed by a sorted tuple of IDs, or use a library such as `p-memoize` with `cacheKey` set to the sorted token-ID list.

Additionally, consider adding per-IP rate limiting at the REST API layer for this endpoint.
