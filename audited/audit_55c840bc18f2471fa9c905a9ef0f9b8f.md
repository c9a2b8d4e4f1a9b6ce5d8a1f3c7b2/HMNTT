### Title
Gas Throttle Bypass via Unthrottled Binary Search Iterations in `estimate=true` Path of `doProcessCall()`

### Summary
When `estimate=true`, the `finally` block in `doProcessCall()` unconditionally skips both `restoreGasToBucket()` and `updateMetrics()`. The binary search in `estimateGas()` invokes `doProcessCall(..., true)` up to `maxGasEstimateRetriesCount` (default: 20) times per request, none of which consume or restore gas tokens from the throttle bucket. A single throttle-passing request with `gas=15,000,000` therefore triggers up to 21 full EVM executions while the throttle accounts for only one, allowing any unprivileged caller to multiply their effective EVM compute budget by up to 21× per request.

### Finding Description

**Exact code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)`, consuming `scaleGas(request.getGas())` tokens from the gas bucket — a one-time deduction for the entire request. [1](#0-0) 

`ContractExecutionService.processCall()` then branches on `params.isEstimate()` and calls `estimateGas()`. [2](#0-1) 

`estimateGas()` first calls `callContract(params, context)`, which internally calls `doProcessCall(params, params.getGas(), false)` — the only invocation where `estimate=false`, so `restoreGasToBucket()` and `updateMetrics()` fire normally. [3](#0-2) 

Then `binaryGasEstimator.search()` is called, running up to `maxGasEstimateRetriesCount = 20` iterations, each invoking `doProcessCall(params, gas, true)`. [4](#0-3) 

**Root cause — the guarded `finally` block:**

```java
// ContractCallService.java lines 127-135
finally {
    if (!estimate) {                          // ← binary-search calls pass estimate=true
        restoreGasToBucket(result, params.getGas());
        if (result != null) {
            updateMetrics(params, result.gasUsed(), 1, status);
        }
    }
}
``` [5](#0-4) 

When `estimate=true`, neither `restoreGasToBucket()` nor per-iteration `updateMetrics()` is called. The gas bucket is never debited for these executions, and the throttle has no visibility into them.

**Failed assumption:** The design assumes the single upfront token deduction in `throttleManager.throttle()` covers the entire cost of an estimate request. It does not — it covers exactly one EVM execution, while the binary search performs up to 20 more.

### Impact Explanation

With default settings (`maxGasLimit = 15,000,000`, `maxGasEstimateRetriesCount = 20`, `gasPerSecond = 7,500,000,000`, `requestsPerSecond = 500`):

- The gas bucket allows ~500 requests/second at max gas (500 × 1,500 scaled tokens = 750,000 tokens/second capacity).
- Each estimate request actually executes up to **21 EVM runs** at up to 15 M gas each = **315 M gas of real EVM work per request**.
- At 500 RPS, actual EVM throughput reaches **~157.5 B gas/second** while the throttle believes it is processing **7.5 B gas/second** — a **21× bypass**.

This enables a sustained CPU exhaustion / denial-of-service attack against the mirror node's EVM execution engine by any unauthenticated caller. The monitoring gap (per-iteration gas not recorded) also means operators cannot detect the overload through normal metrics. [6](#0-5) 

### Likelihood Explanation

- **No privileges required.** The `/api/v1/contracts/call` endpoint is public; any HTTP client can set `"estimate": true`.
- **Trivially repeatable.** A single script sending POST requests at the 500 RPS rate limit with `gas: 15000000` and `estimate: true` is sufficient.
- **No special contract needed.** Any deployed contract (or even a zero-address call) that succeeds on the initial probe will trigger the full binary search loop.
- The attacker does not need to observe any output — the damage is purely computational load on the server.

### Recommendation

1. **Account for all binary-search iterations in the throttle.** Before entering `binaryGasEstimator.search()`, pre-deduct tokens proportional to `maxGasEstimateRetriesCount × gas`, or deduct one token per iteration inside `BinaryGasEstimator.safeCall()` via a callback.
2. **Alternatively, call `restoreGasToBucket()` per binary-search iteration** (with `estimate=true` treated the same as `estimate=false` for throttle purposes), restoring only the unused portion after each EVM run.
3. **Cap the effective multiplier** by reducing `maxGasEstimateRetriesCount` or enforcing a per-request wall-clock timeout that also gates the binary search loop.
4. Ensure `updateMetrics()` records gas consumed by each binary-search iteration so operators can observe the true load.

### Proof of Concept

```bash
# Precondition: mirror node running with default config, any deployed contract at <CONTRACT_ADDR>

# Single request triggers 1 throttled + up to 20 unthrottled EVM executions
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "<CONTRACT_ADDR>",
    "data": "<CALLDATA>",
    "gas": 15000000,
    "estimate": true
  }'

# Flood at the RPS limit to exhaust CPU
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"<CONTRACT_ADDR>","data":"<CALLDATA>","gas":15000000,"estimate":true}' &
done
wait

# Observable result:
# - Server CPU spikes to saturation
# - Gas throttle bucket shows ~7.5B gas/s consumed (normal)
# - Actual EVM work is ~157.5B gas/s (21x the throttled amount)
# - Legitimate eth_call requests begin timing out or receiving throttle errors
```

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
