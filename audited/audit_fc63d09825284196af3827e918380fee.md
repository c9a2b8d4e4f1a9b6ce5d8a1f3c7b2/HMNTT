### Title
Unauthenticated Pagination Loop Enables Sustained DB Load via `getNftsByAccountId` with No Application-Level Rate Limiting

### Summary
The `getNftsByAccountId` handler in `rest/controllers/accountController.js` accepts unlimited repeated requests from any unauthenticated caller. The endpoint enforces a per-response row cap of 100 but applies no per-IP rate limiting, no concurrency limiting, and no cost-based throttling at the application layer. An attacker can drive a tight pagination loop against an account with many NFTs, issuing continuous DB queries that generate sustained load and degrade service for other users.

### Finding Description

**Code location:** `rest/controllers/accountController.js`, `getNftsByAccountId`, lines 90–103.

```
getNftsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);
  const filters = utils.buildAndValidateFilters(req.query, acceptedNftAccountParameters);
  const query = this.extractNftMultiUnionQuery(filters, accountId);
  const nonFungibleTokens = await NftService.getNfts(query);   // DB hit every call
  ...
  links: { next: this.getPaginationLink(...) }
};
``` [1](#0-0) 

**Query construction:** `NftService.getQuery()` builds a `UNION ALL` of up to three subqueries (lower/inner/upper bounds), each with `LIMIT $2` and an `ORDER BY token_id, serial_number`. Every HTTP request triggers a fresh execution of this compound query against the `nft` table joined with `entity`. [2](#0-1) 

**Limit enforcement:** `getLimitParamValue` caps the `limit` parameter at `responseLimit.max` (default 100). This bounds rows *per response* but does not bound the *request rate*. [3](#0-2) 

**No application-level rate limiting in the REST Node.js service:** A `grep` across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, `express-rate`, and `throttle` returns zero matches. The `ThrottleManagerImpl` / `ThrottleConfiguration` classes exist only in the `web3` Java service, not in the REST Node.js service.

**Helm chart confirms the gap:** The `hedera-mirror-rest` Helm chart middleware section contains only `circuitBreaker` and `retry`. It is missing the `inFlightReq` (per-IP concurrent connection cap) and `rateLimit` (per-host request rate cap) entries that are present in the `hedera-mirror-rosetta` chart. [4](#0-3) [5](#0-4) 

**Root cause / failed assumption:** The design assumes that capping rows per response (limit ≤ 100) is sufficient to bound DB cost. It is not: cost is `O(pages × query_cost)`, and pages are unbounded for any caller.

### Impact Explanation

Each paginated request issues a compound SQL query (up to 3 subqueries UNION ALL'd) against the `nft` table. For an account holding tens of thousands of NFTs, a single full traversal requires hundreds of sequential requests. An attacker running this loop from multiple IPs or with concurrent connections can saturate the DB connection pool, increase query latency for all users, and potentially trigger the circuit breaker — causing 503 responses for legitimate traffic. No funds are at risk, but service availability degrades. This matches the stated "griefing / no economic damage" severity.

### Likelihood Explanation

- **No privilege required:** Any internet user can call the endpoint.
- **Target discovery is trivial:** NFT-heavy accounts are visible on public explorers.
- **Automation is trivial:** The `next` link in each response is machine-readable; a simple script can loop indefinitely.
- **No existing deterrent at the application layer:** confirmed by code search and Helm chart review.
- **Repeatability:** The attack is stateless and can be restarted at any time.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST Node.js service (e.g., `express-rate-limit` or Traefik `rateLimit` middleware), consistent with what is already deployed for the Rosetta service.
2. **Add per-IP concurrency limiting** via Traefik `inFlightReq` in `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta chart configuration.
3. **Optionally add a server-side pagination cost budget** (e.g., reject or throttle callers who exceed N paginated requests per minute for the same account ID).

### Proof of Concept

```bash
# 1. Identify an account with many NFTs (public data)
ACCOUNT="0.0.XXXXX"
BASE="https://<mirror-node>/api/v1"

# 2. Seed the loop
NEXT="${BASE}/accounts/${ACCOUNT}/nfts?order=asc&limit=100"

# 3. Tight pagination loop — no authentication required
while [ "$NEXT" != "null" ]; do
  RESP=$(curl -s "$NEXT")
  NEXT=$(echo "$RESP" | jq -r '.links.next // "null"')
  [ "$NEXT" != "null" ] && NEXT="${BASE}${NEXT}"
done

# 4. Run multiple instances of the above loop concurrently from the same or
#    different IPs. Each iteration issues a fresh UNION ALL DB query.
#    Observe rising DB CPU / query latency for other API consumers.
```

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit, spenderIdInFilters, spenderIdFilters} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${Nft.ACCOUNT_ID} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...NftService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) =>
        this.getSubQuery(
          filters,
          params,
          accountIdCondition,
          limitClause,
          orderClause,
          spenderIdInFilters,
          spenderIdFilters
        )
      );

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery(
        [],
        params,
        accountIdCondition,
        limitClause,
        orderClause,
        spenderIdInFilters,
        spenderIdFilters
      );
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
