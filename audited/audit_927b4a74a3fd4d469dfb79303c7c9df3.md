### Title
Unauthenticated `from` Field Enables Bypass of `FilterField.FROM`-Based REJECT/THROTTLE Rules

### Summary
The `from` field in `ContractCallRequest` is a plain, user-supplied string with no authentication or identity verification. The throttle system's `RequestFilter` uses `FilterField.FROM` to extract this field directly from the request and match it against configured expressions. Because any caller can supply any arbitrary hex address as `from`, any `RequestProperties` rule configured with `action = REJECT` (or `THROTTLE`) and a `FilterField.FROM` filter is trivially bypassable by substituting a different address.

### Finding Description
**Code path:**

- `ContractCallRequest.java:34` — `from` is a plain `@Hex`-validated string; the only constraint is format (40-char hex). No authentication, no signature, no session binding.
- `RequestFilter.java:43` — `FilterField.FROM` extracts the value via `ContractCallRequest::getFrom`, returning whatever the caller supplied.
- `RequestFilter.java:31-35` — `test()` applies the configured `FilterType` predicate against this raw, caller-controlled string.
- `ThrottleManagerImpl.java:44-48` — `throttle()` iterates `requestFilter.test(request)` and calls `action()` only when the filter matches.
- `RequestProperties.java:66-75` — `action()` throws `ThrottleException("Invalid request")` for `REJECT`, or rate-limits for `THROTTLE`.

**Root cause:** The system assumes `from` identifies the caller, but it is entirely attacker-controlled. There is no mechanism (JWT, API key, IP binding, cryptographic signature) that ties the `from` value to the actual HTTP client. A filter rule on `FilterField.FROM` is therefore not an access-control boundary — it is a string-matching hint that the caller can freely manipulate.

**Exploit flow:**
1. Operator configures a `RequestProperties` with `action = REJECT`, `FilterField.FROM`, `FilterType.EQUALS`, `expression = "0x000…dead"` to block a known abusive address.
2. Attacker sends a `ContractCallRequest` with `from = "0x000…beef"` (any other valid 40-char hex address).
3. `RequestFilter.test()` evaluates `"0x000…beef".equalsIgnoreCase("0x000…dead")` → `false`.
4. `requestFilter.test(request)` returns `false`; `action()` is never called.
5. The request passes all throttle checks and is executed.

**Why existing checks are insufficient:**
- `@Hex(minLength = ADDRESS_LENGTH, maxLength = ADDRESS_LENGTH)` on `from` only validates format, not identity.
- `hasFrom()` / `hasTo()` `@AssertTrue` validators only enforce presence when `value > 0`, not authenticity.
- The global `rateLimitBucket` and `gasLimitBucket` in `ThrottleConfiguration.rateLimitBucket()` are shared counters — they do not enforce per-identity limits and are unaffected by the `from` field at all.

### Impact Explanation
Any operator-configured address-based REJECT or THROTTLE rule is rendered completely ineffective. An attacker can execute an unlimited number of contract-call simulations that the operator intended to block, consuming server-side EVM simulation resources (CPU, memory, gas budget from `gasLimitBucket`). If the blocked address was meant to prevent access to sensitive contract state reads or to enforce business-logic restrictions, those restrictions are bypassed. Severity is **Medium–High** depending on deployment: in a public-facing mirror node, this enables targeted DoS of the gas/rate buckets and circumvention of all address-based access policies.

### Likelihood Explanation
Exploitation requires zero privileges, zero special tooling, and zero knowledge beyond the public API schema. Any caller who discovers (or guesses) that their address is blocked can immediately retry with a different `from` value. The attack is repeatable, stateless, and undetectable without additional logging correlation. Likelihood is **High**.

### Recommendation
1. **Do not use `FilterField.FROM` as a security control** unless the `from` value is cryptographically bound to the caller (e.g., verified via signed request or authenticated session). Document this limitation explicitly.
2. If per-identity rate limiting is required, key it on a verifiable attribute such as the authenticated API key, client TLS certificate, or source IP address — not the user-supplied `from` field.
3. If `from`-based filtering must be retained for non-security purposes (e.g., logging), add a warning in configuration validation that `FilterField.FROM` filters with `action = REJECT` or `THROTTLE` provide no security guarantee.
4. Consider adding a `FilterField.IP` or `FilterField.API_KEY` backed by server-side extraction from the HTTP request context, which cannot be spoofed by the caller.

### Proof of Concept
**Setup:** Configure `application.yml`:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    filters:
      - field: FROM
        type: EQUALS
        expression: "0x00000000000000000000000000000000000004e2"
```

**Step 1 — Confirm block works for the exact address:**
```bash
curl -X POST http://localhost:8545/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"from":"0x00000000000000000000000000000000000004e2","to":"0x...","data":"0x..."}'
# Expected: 429 / ThrottleException "Invalid request"
```

**Step 2 — Bypass by spoofing `from`:**
```bash
curl -X POST http://localhost:8545/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"from":"0x00000000000000000000000000000000000004e3","to":"0x...","data":"0x..."}'
# Result: 200 OK — REJECT rule is bypassed; request executes normally
```

The attacker can use any valid 40-character hex address other than the blocked one. No authentication prevents this substitution.
