### Title
Unauthenticated gRPC `getNodes()` Lacks Rate Limiting, Enabling Database Exhaustion via Thundering Herd

### Summary
`NetworkController.getNodes()` is a publicly accessible, unauthenticated gRPC endpoint with no rate limiting, no per-IP connection cap, and no request deduplication. Each invocation unconditionally executes multiple uncached database queries in `NetworkServiceImpl.getNodes()` before reaching any cached layer. An unprivileged attacker opening many connections and flooding concurrent requests can exhaust the database connection pool, starving legitimate network nodes of address book data.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) directly delegates every incoming request to `networkService::getNodes` with no throttle check, no deduplication, and no authentication:

```java
// NetworkController.java:33-38
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(networkService::getNodes)
        .map(this::toNodeAddress)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`NetworkServiceImpl.getNodes()` (lines 55–77) executes **three uncached database calls** on every invocation before any cached layer is reached:

```java
// NetworkServiceImpl.java:61-66
long addressBookTimestamp = addressBookRepository
        .findLatestTimestamp(fileId.getId())          // uncached DB hit
        .orElseThrow(...);
long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp()  // uncached DB hit
        .orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp); // uncached DB hit
```

Only the paginated `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` call is `@Cacheable` (via `ADDRESS_BOOK_ENTRY_CACHE`). The three upstream queries are not cached anywhere in the grpc module.

**Root cause / failed assumption:** The design assumes that the `maxConcurrentCallsPerConnection = 5` Netty setting (configured in `GrpcConfiguration.java:33`) is sufficient protection. This setting only limits concurrent streams *per connection*; it imposes no cap on the total number of connections or on the aggregate rate of requests across all connections. There is no bucket4j throttle, no gRPC interceptor rate limiter, and no IP-based connection limit applied to this endpoint. The web3 throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) is entirely absent from the grpc module.

### Impact Explanation

Each `getNodes()` call issues at minimum three synchronous database queries that bypass all caching. Under a flood of concurrent connections (each allowed 5 concurrent calls), the database connection pool is rapidly saturated. Once the pool is exhausted, legitimate Hedera network nodes that rely on this endpoint to bootstrap their peer lists receive errors or timeouts, causing address book data to become unavailable. This directly disrupts network partition recovery and node discovery — a critical infrastructure function.

**Severity: High.** The endpoint serves critical network topology data. Unavailability of address book information can prevent nodes from reconnecting after a network partition.

### Likelihood Explanation

The endpoint is publicly documented and reachable without credentials (`grpcurl -plaintext -d '{"file_id": {"fileNum": 102}}' ... NetworkService/getNodes`). No authentication, API key, or IP allowlist is required. A single attacker with a modest number of TCP connections (e.g., 200 connections × 5 concurrent calls = 1,000 concurrent DB-hitting requests) can sustain the attack indefinitely. The attack is trivially repeatable and requires no special knowledge beyond the public proto definition.

### Recommendation

1. **Add a gRPC server interceptor rate limiter** (e.g., bucket4j or Resilience4j) scoped to the `getNodes` method, enforcing a global requests-per-second cap and optionally a per-source-IP cap.
2. **Cache the uncached repository calls** (`findLatestTimestamp` for both `AddressBookRepository` and `NodeStakeRepository`, and `findAllStakeByConsensusTimestamp`) with a short TTL (e.g., matching the existing `cacheExpiry = 2s` in `AddressBookProperties`).
3. **Add a global maximum connection count** to the Netty server builder in `GrpcConfiguration` (e.g., `serverBuilder.maxConnectionAge(...)` and a connection-count guard).
4. **Implement request coalescing / single-flight** so that N identical in-flight `getNodes` requests share a single database round-trip.

### Proof of Concept

```bash
# Open 200 parallel connections, each sending getNodes in a tight loop
for i in $(seq 1 200); do
  (while true; do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes \
      > /dev/null 2>&1
  done) &
done
wait
```

```java
    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }
```
