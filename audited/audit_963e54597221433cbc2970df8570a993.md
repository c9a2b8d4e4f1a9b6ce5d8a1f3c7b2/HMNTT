### Title
Uncached Historical Storage Queries Enable DB Saturation via Global Rate Limit Exhaustion

### Summary
`findStorageByBlockTimestamp()` in `ContractStateServiceImpl` performs a direct, uncached `ORDER BY consensus_timestamp DESC LIMIT 1` query against `contract_state_change` for every invocation. Unlike `findStorage()`, which carries a `@Cacheable` annotation, the historical path has no caching at any layer. The global (non-per-IP) rate limiter can be fully consumed by a single attacker, causing every allowed request to generate a fresh DB query, degrading response times for all users.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageByBlockTimestamp()` (lines 72–76) passes through directly to the repository with no cache check:

```java
// ContractStateServiceImpl.java lines 72-76
public Optional<byte[]> findStorageByBlockTimestamp(
        final EntityId entityId, final byte[] slotKeyByteArray, final long blockTimestamp) {
    return contractStateRepository.findStorageByBlockTimestamp(entityId.getId(), slotKeyByteArray, blockTimestamp);
}
```

The repository method has no `@Cacheable` annotation (contrast with `findStorage` at line 20–21 which does):

```java
// ContractStateRepository.java lines 44-54
@Query(value = """
        select coalesce(value_written, value_read) as value
        from contract_state_change
        where contract_id = ?1
        and slot = ?2
        and consensus_timestamp <= ?3
        order by consensus_timestamp desc
        limit 1
        """, nativeQuery = true)
Optional<byte[]> findStorageByBlockTimestamp(long id, byte[] slot, long blockTimestamp);
```

**Root cause:** The `findStorageByBlockTimestamp` method in `ContractStateRepository` carries no `@Cacheable` annotation, and `ContractStateServiceImpl` adds no manual cache logic either. Every call unconditionally executes the range-scan query.

**Rate limiter is global, not per-IP:** `ThrottleManagerImpl.throttle()` consumes from a single shared `rateLimitBucket` (default 500 req/s per `ThrottleProperties.requestsPerSecond`). There is no per-source-IP bucket. A single attacker can legally consume all 500 slots/second.

**Exploit flow:**
1. Attacker identifies the `/api/v1/contracts/call` endpoint with a historical block parameter (e.g., `block: "0x1A2B3C"`).
2. Attacker sends requests at the global rate limit (500/s) with a different `blockTimestamp` value each time (e.g., incrementing by 1 nanosecond).
3. Each request passes the global throttle check (consuming one token), reaches `ContractStorageReadableKVState.readFromDataSource()`, which calls `contractStateService.findStorageByBlockTimestamp()`.
4. Because `blockTimestamp` varies, no cache entry can ever be reused even if one existed.
5. Every request executes the `ORDER BY consensus_timestamp DESC LIMIT 1` range scan against `contract_state_change`.
6. Legitimate users receive throttle rejections (HTTP 429) or experience degraded DB-backed response times.

**Why existing checks fail:**
- The global rate limiter prevents unbounded requests but does not prevent a single attacker from consuming the entire budget.
- The gas-per-second bucket (`gasLimitBucket`) is only relevant for EVM execution cost, not for read-only historical storage lookups which may have low declared gas.
- There is no per-IP, per-contract, or per-slot rate limiting.
- There is no cache at the service or repository layer for the historical path.

### Impact Explanation

A single unprivileged attacker can monopolize the entire 500 req/s global budget, causing all legitimate users to receive HTTP 429 responses. Simultaneously, each attacker request generates a DB range scan on `contract_state_change` (which can be a large, append-only table), increasing DB CPU and I/O load. This is a griefing-class DoS: no funds are at risk, but service availability for all users is degraded for as long as the attacker sustains the flood. Severity is Medium per the stated scope.

### Likelihood Explanation

No special privileges, tokens, or on-chain assets are required. Any user with HTTP access to the public JSON-RPC endpoint can trigger this by sending `eth_call` (or equivalent REST call) with a historical block number and varying slot keys or block values. The attack is trivially scriptable with a simple loop and requires no knowledge of contract internals. It is repeatable indefinitely.

### Recommendation

1. **Add `@Cacheable` to `findStorageByBlockTimestamp`** in `ContractStateRepository` using a dedicated cache manager with a bounded TTL and size, keyed on `(contractId, slot, blockTimestamp)`. Because historical data is immutable (past state never changes), cache entries can have a long TTL.
2. **Add per-IP rate limiting** at the servlet filter or API gateway layer so a single source cannot exhaust the global budget.
3. **Alternatively**, add a manual cache check in `ContractStateServiceImpl.findStorageByBlockTimestamp()` mirroring the pattern used in `findStorage()`.
