### Title
`incrementByteArray()` Overflow on Max-Value GT Parameter Exposes Full Hook Storage

### Summary
When a user supplies `key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF` to the hook storage endpoint, `MutableBytes32.increment()` silently wraps the all-`0xFF` byte array to all-`0x00`. The resulting wrapped value fails the "more restrictive than current lower bound" comparison in `hookStorageChangeRequest()`, so the lower bound is never updated from `MIN_KEY_BYTES`, and the repository query returns every storage entry for the targeted hook.

### Finding Description

**Step 1 — Parameter parsing** (`SlotRangeParameter.java:44-45`):

```
operator = GT, hex = "FFFF...FFFF" (64 chars)
getInclusiveValue(GT, hex) → incrementByteArray([0xFF×32])
```

`BytesUtil.incrementByteArray` delegates to Tuweni:

```java
// BytesUtil.java:13-14
return MutableBytes32.wrap(bytes).increment().toArrayUnsafe();
```

`MutableBytes32.increment()` treats the 32-byte array as a big-endian integer. `0xFF…FF + 1` overflows and wraps to `0x00…00`. No exception is thrown; the method returns silently.

**Step 2 — Operator normalisation** (`SlotRangeParameter.java:38`):

```java
return new SlotRangeParameter(operator.toInclusive(), getInclusiveValue(operator, hex));
// GT → GTE, value = [0x00×32]
```

**Step 3 — Lower-bound selection** (`HooksController.java:167,176-178`):

```java
var lowerBound = MIN_KEY_BYTES;          // [0x00×32]
...
} else if (Arrays.compareUnsigned(value, lowerBound) > 0) {
    lowerBound = value;
}
```

`value` is `[0x00×32]`; `lowerBound` is also `[0x00×32]`. `compareUnsigned` returns `0`, which is **not** `> 0`, so the branch is skipped. `lowerBound` stays at `MIN_KEY_BYTES`.

**Step 4 — Repository call** (`HookServiceImpl.java:66-67`):

```java
hookStorageRepository.findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(
    ownerId.getId(), request.getHookId(),
    request.getKeyLowerBound(),   // [0x00×32]
    request.getKeyUpperBound(),   // [0xFF×32]
    page);
```

This is a full-table scan over the hook's storage — every non-deleted slot is returned.

**Why existing checks are insufficient:**
The guard at line 176 is designed to keep only the *most restrictive* lower bound. It correctly handles normal GT inputs (e.g., `gt:0x01` → `0x02 > 0x00` → update). It does not handle the overflow edge case where the incremented value wraps below the current lower bound.

### Impact Explanation
An unauthenticated caller can enumerate the complete storage of any smart contract hook by issuing a single (or paginated) GET request. Storage slots may contain sensitive application state (balances, access-control mappings, cryptographic material). The endpoint is publicly reachable with no authentication requirement, making this a direct information-disclosure vulnerability against any deployed hook.

### Likelihood Explanation
The exploit requires zero privileges and zero on-chain interaction. The attacker needs only to know a valid `ownerId` and `hookId` (both are enumerable via the `/hooks` endpoint). The payload is a single well-formed query parameter. It is trivially repeatable and automatable.

### Recommendation
Add an overflow guard in `getInclusiveValue` before returning the incremented value:

```java
private static byte[] getInclusiveValue(RangeOperator operator, String hexValue) throws DecoderException {
    byte[] bytes = Hex.decodeHex(hexValue);
    if (operator == RangeOperator.GT) {
        // All-0xFF is the maximum representable key; GT that is the empty set.
        if (Arrays.equals(bytes, MAX_KEY_BYTES)) {
            throw new IllegalArgumentException(
                "No key can be greater than the maximum slot value 0xFF...FF");
        }
        return incrementByteArray(bytes);
    } else if (operator == RangeOperator.LT) {
        if (Arrays.equals(bytes, MIN_KEY_BYTES)) {
            throw new IllegalArgumentException(
                "No key can be less than the minimum slot value 0x00...00");
        }
        return decrementByteArray(bytes);
    }
    return bytes;
}
```

Alternatively, after computing the inclusive value, verify `Arrays.compareUnsigned(result, original) > 0` (for GT) and reject if not.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF&limit=100
```

1. `SlotRangeParameter.valueOf` parses operator=`GT`, value=`0xFF×32`.
2. `incrementByteArray([0xFF×32])` → `[0x00×32]` (Tuweni overflow wrap).
3. `SlotRangeParameter` is constructed as `(GTE, [0x00×32])`.
4. In `hookStorageChangeRequest`, `Arrays.compareUnsigned([0x00×32], [0x00×32]) == 0`, guard not triggered.
5. `keyLowerBound = [0x00×32]`, `keyUpperBound = [0xFF×32]`.
6. `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse` returns all storage entries for the hook.
