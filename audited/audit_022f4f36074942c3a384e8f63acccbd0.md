### Title
Unauthenticated EVM Address Lookup Bypasses LRU Cache, Exhausts DB Connection Pool via `getEncodedId()`

### Summary
`getEncodedId()` in `rest/service/entityService.js` unconditionally issues a live database query for every opaque EVM address input, regardless of whether the `entityId` LRU cache in `rest/entityId.js` has already seen that address. Because the LRU cache stores only the parsed `EntityId` object (not the DB lookup result), a cache hit for an opaque EVM address still triggers `getEntityIdFromEvmAddress()` and a `getRows()` call. With the default DB connection pool capped at 10 connections and no rate limiting on the REST API, an unauthenticated attacker can exhaust the pool with as few as 10 concurrent requests, causing denial of service for all entity-dependent operations.

### Finding Description

**Root cause / failed assumption:** The LRU cache is assumed to short-circuit repeated work, but it only caches the *parsing* result. For opaque EVM addresses (`evmAddress !== null`), the caller in `getEncodedId()` always proceeds to `getEntityIdFromEvmAddress()` regardless of cache state. The cache never stores the DB lookup result, so every request — cached or not — issues a `SELECT` against the `entity` table. [3](#0-2) 

`getEntityIdFromEvmAddress` calls `this.getRows()` which calls `this.pool().queryQuietly()`, consuming a connection from the shared pool: [4](#0-3) 

The pool is configured with a default of **10 connections** and a **20-second statement timeout**: [5](#0-4) [6](#0-5) 

**No rate limiting exists on the REST API.** The middleware stack in `rest/server.js` registers only `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, and `handleError` — no request-rate or concurrency limiter: [7](#0-6) 

The throttling found (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies exclusively to the **web3 Java API**, not the Node.js REST API.

**`authHandler` does not block unauthenticated requests** — it simply skips setting a custom limit and calls `next()`: [8](#0-7) 

### Impact Explanation
With 10 default pool connections and a 20-second statement timeout, an attacker needs only 10 concurrent in-flight requests to hold every connection. All other REST API operations that require a DB connection (transactions, balances, account lookups, etc.) will queue or fail with a connection-timeout error for the duration of the attack. The attack is self-sustaining: as long as the attacker keeps sending new requests, the pool stays saturated. This is a complete denial of service for the REST API's database-backed functionality.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero knowledge of the system beyond the public API. Any valid 40-hex-character EVM address that is not a long-form num alias (i.e., whose first 12 bytes are not all zeros, or whose num exceeds `maxNum`) qualifies as an opaque EVM address and triggers a DB query. Generating thousands of such addresses is trivial. The attack is repeatable, scriptable, and requires only an HTTP client. The low default pool size (10) makes the threshold for impact extremely low.

### Recommendation
1. **Cache DB lookup results for EVM addresses.** Introduce a second LRU cache (or extend the existing one) that maps `evmAddress → encodedId` so repeated lookups for the same address do not hit the database.
2. **Add rate limiting to the REST API.** Apply a per-IP or global request-rate limiter (e.g., `express-rate-limit`) in `rest/server.js` before route handlers, analogous to the bucket4j throttle used in the web3 API.
3. **Increase the default pool size** or add a concurrency semaphore so that EVM address lookups cannot monopolize all connections.
4. **Reduce `statementTimeout`** for entity-existence queries to limit how long a single query can hold a connection.

### Proof of Concept
```bash
# Generate 50 unique opaque EVM addresses (first byte non-zero ensures they are not num aliases)
# and fire them concurrently at any endpoint that resolves an account/contract by EVM address.
# Example: GET /api/v1/accounts/{evmAddress}

for i in $(seq 1 50); do
  ADDR=$(printf "AA%038X" $i)   # prefix AA ensures non-zero first byte → opaque EVM address
  curl -s "http://<mirror-node>/api/v1/accounts/0x${ADDR}" &
done
wait
```

Each of the 50 requests passes `EntityId.isValidEntityId()`, resolves to an opaque EVM address (`evmAddress !== null`), bypasses any LRU cache benefit, and calls `getEntityIdFromEvmAddress()` → `getRows()` → DB query. With 10 pool connections and a 20-second timeout, the pool is exhausted within the first 10 concurrent requests. Subsequent legitimate requests receive connection-timeout errors until the attack stops.
