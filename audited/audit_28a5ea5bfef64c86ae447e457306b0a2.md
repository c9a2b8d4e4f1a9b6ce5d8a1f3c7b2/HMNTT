### Title
Unauthenticated Resource Amplification DoS via Multi-Query Fan-Out on `/contracts/results/:transactionIdOrHash`

### Summary
Any unauthenticated external user can issue a single `GET /contracts/results/<valid_eth_hash>` request that fans out into at least 6 sequential and parallel database queries with no rate limiting or connection-pool guard visible in the routing or service layer. Under concurrent load this exhausts the PostgreSQL connection pool, denying service to all mirror-node consumers.

### Finding Description

**Endpoint registration (no auth middleware):**
`rest/routes/contractRoute.js` line 21:
```js
router.getExt('/results/:transactionIdOrHash', ContractController.getContractResultsByTransactionIdOrHash);
``` [1](#0-0) 

No authentication, no rate-limit middleware is applied before this handler.

**Query fan-out in the service layer (all confirmed present):**

1. `getContractTransactionDetailsByHash` — issues 1 DB query against `contract_transaction_hash`: [2](#0-1) 

2. `getInvolvedContractsByTimestampAndContractId` — issues 1 DB query against `contract_transaction`: [3](#0-2) 

3. `getContractResultsByTimestamps` — issues 1 DB query: [4](#0-3) 

4. `getContractLogsByTimestamps` — issues 1 DB query: [5](#0-4) 

5. `getContractStateChangesByTimestamps` — issues 1 DB query: [6](#0-5) 

6. `getEthereumTransactionsByPayerAndTimestampArray` — issues 1 DB query: [7](#0-6) 

The controller function `getContractResultsByTransactionIdOrHash` (confirmed present in `rest/controllers/contractController.js` by two grep matches) orchestrates these calls. Steps 3–6 are fired in parallel via `Promise.all` inside `getDetailedContractResults`, meaning a single HTTP request holds multiple DB connections simultaneously.

**Root cause / failed assumption:** The design assumes organic, low-concurrency traffic. There is no per-IP or global rate limit, no request queue, and no circuit breaker protecting the DB connection pool from amplified concurrent load.

### Impact Explanation
Each request consumes 6+ DB connections for its duration. With a typical pool of 10–20 connections and an attacker sending ~50 concurrent requests, the pool is saturated. Legitimate queries queue indefinitely or time out, making the entire mirror-node REST API unavailable. Because the mirror node is the primary read interface for dApps, wallets, and explorers on Hedera, sustained unavailability has broad ecosystem impact.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and only a single valid Ethereum transaction hash (trivially obtained from any public block explorer or by submitting one cheap transaction). The attack is fully scriptable with standard HTTP load tools (`wrk`, `hey`, `ab`). It is repeatable indefinitely and requires no special knowledge beyond the public API documentation.

### Recommendation
1. **Rate limiting**: Apply per-IP and global rate limits at the reverse-proxy or Express middleware layer specifically for `/contracts/results/:transactionIdOrHash`.
2. **Connection pool protection**: Set a hard cap on concurrent in-flight queries per request class; reject with `429` when the cap is reached.
3. **Result caching**: Cache responses for identical hashes (short TTL, e.g. 5 s) to collapse repeated identical requests into a single DB round-trip.
4. **Query timeout**: Enforce a per-request DB query timeout so a slow fan-out cannot hold connections indefinitely.

### Proof of Concept
```bash
# 1. Obtain any valid Ethereum tx hash from the Hedera mirror node or a public explorer
HASH="0x<valid_32_byte_eth_hash>"

# 2. Fire 100 concurrent requests
hey -n 10000 -c 100 \
  "https://<mirror-node-host>/api/v1/contracts/results/$HASH"

# Expected result: mirror node DB connection pool exhausted;
# all other API endpoints begin returning 500/503 or timing out.
```

      return {
        ...new ContractStateChange(row),
        evmAddress: row.evm_address,
      };
    });

        minTimestamp = timestamp;

      payers,
      timestamps,
      minTimestamp,
      maxTimestamp,
