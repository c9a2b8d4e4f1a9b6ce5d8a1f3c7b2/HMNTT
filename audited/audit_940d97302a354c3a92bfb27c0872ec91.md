### Title
Unauthenticated DB Query Storm via Uncached `FileDataService.getFileData()` in `getContractById()`

### Summary
`getContractById()` unconditionally invokes `FileDataService.getFileData()` — which executes a complex correlated SQL query against the `file_data` table with no service-layer caching — for every GET `/contracts/:contractId` request where the contract has a user-created `file_id`. Because the REST API has no rate limiting on this endpoint and the response cache is an optional opt-in requiring Redis, an unprivileged attacker can flood this endpoint to generate a sustained DB query storm that degrades database performance for all users.

### Finding Description

**Exact code path:**

`rest/controllers/contractController.js`, `getContractById()`, lines 731–732:
```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
``` [1](#0-0) 

`utils.isValidUserFileId()` returns `true` for any non-null `file_id` with `num > 1000` — i.e., any user-created file, which is the normal case for deployed contracts: [2](#0-1) 

`FileDataService.getFileData()` in `rest/service/fileDataService.js` lines 71–76 executes `getFileDataQuery` directly against the DB on every invocation with **no caching**: [3](#0-2) 

`getFileDataQuery` (lines 44–63) is a correlated subquery that scans the `file_data` table twice per request — once in the inner subquery to find the anchor timestamp, and once in the outer query to aggregate all file chunks: [4](#0-3) 

**Root cause:** The REST-layer `FileDataService` (JavaScript, `rest/service/fileDataService.js`) has no in-memory or distributed cache. The Java importer's `FileDataServiceImpl.java` has a Spring cache, but it is a completely separate service in a different process and is irrelevant to the REST API path. [5](#0-4) 

**Why existing checks fail:**

1. **No rate limiting on GET `/contracts/:contractId`**: `rest/server.js` registers no rate-limiting middleware for the contracts route. The `ThrottleManagerImpl` in the `web3` module only applies to `ContractCallRequest` (POST `/api/v1/contracts/call`), not to REST GET endpoints: [6](#0-5) [7](#0-6) 

2. **Response cache is optional and gated on two config flags**: `applicationCacheEnabled` requires both `config.cache.response.enabled` AND `config.redis.enabled`. When Redis is not deployed (common in self-hosted or development environments), the cache is entirely absent: [6](#0-5) 

3. **Even when the response cache is enabled**, it only prevents re-execution for the *same URL* after the first response is stored. Concurrent requests arriving before the first response is cached all bypass it and hit the DB simultaneously (cache stampede). Additionally, varying the `timestamp` query parameter produces distinct cache keys, trivially bypassing the cache.

### Impact Explanation

Each request to GET `/contracts/:contractId` for a contract with a `file_id` causes two sequential scans of the `file_data` table (inner + outer correlated subquery). Under high concurrency, this saturates the PostgreSQL connection pool and increases I/O on the `file_data` table, degrading query latency for all other API consumers (exchange rate lookups, address book reads, and other contract bytecode fetches) that also depend on `file_data`. The impact is service degradation (griefing) with no economic cost to the attacker.

### Likelihood Explanation

The attack requires zero privileges — the endpoint is publicly accessible with no authentication. The attacker only needs to know one valid contract ID with a `file_id` (trivially discoverable via GET `/api/v1/contracts`). The attack is fully repeatable and can be automated with a simple HTTP flood tool. The `timestamp` query parameter can be varied to defeat any response cache that may be present.

### Recommendation

1. **Add service-layer caching in `FileDataService.getFileData()`**: Cache results keyed on `(fileId, timestamp)` using an in-process LRU cache (e.g., `quick-lru`, already used elsewhere in the codebase) with a short TTL. This eliminates redundant DB queries for the same `(fileId, created_timestamp)` pair.
2. **Add rate limiting to the REST API**: Apply a per-IP rate limiter (e.g., `express-rate-limit`) to the contracts router, or at the ingress/reverse-proxy layer, to bound the request rate from any single source.
3. **Ensure the response cache is enabled in production**: Document and enforce that `config.cache.response.enabled` + `config.redis.enabled` are set in production deployments.

### Proof of Concept

**Preconditions:**
- Mirror node REST API is running without Redis response cache enabled (`config.redis.enabled = false` or default).
- At least one contract exists with a user-created `file_id` (num > 1000), e.g., `0.0.8001` with `file_id = 0.0.5001`.

**Steps:**
```bash
# 1. Discover a valid contract with a file_id
curl http://mirror-node/api/v1/contracts?limit=1
# Response includes: "file_id": "0.0.5001", "contract_id": "0.0.8001"

# 2. Flood the endpoint (no auth required)
# Each request triggers FileDataService.getFileData() -> correlated SQL query on file_data
for i in $(seq 1 10000); do
  curl -s "http://mirror-node/api/v1/contracts/0.0.8001" &
done
wait

# 3. Observe: DB connection pool exhaustion, increased query latency on file_data table,
#    degraded response times for all other API endpoints that query file_data.
```

**Variant (bypasses response cache if enabled):**
```bash
# Vary the timestamp parameter to produce distinct cache keys
for i in $(seq 1 10000); do
  curl -s "http://mirror-node/api/v1/contracts/0.0.8001?timestamp=lte:$((1000000000 + i))" &
done
```
