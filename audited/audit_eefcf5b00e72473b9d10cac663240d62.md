### Title
Future Timestamp Injection Causes Denial-of-Service on Network Supply Endpoint via EntityNotFoundException

### Summary
The `getSupply(Bound timestamp)` method in `NetworkServiceImpl` accepts a user-controlled timestamp with no validation against the current consensus time. An unprivileged caller can supply a far-future timestamp, causing `getFirstDayOfMonth(maxTimestamp, -1)` to compute an `optimalLowerBound` that is still in the future, resulting in zero rows returned from `accountBalanceRepository.getSupplyHistory()`, a `consensusTimestamp` of `0`, and an `EntityNotFoundException` being thrown ‚Äî making supply data unavailable.

### Finding Description

**Exact code path:**

`NetworkServiceImpl.java`, `getSupply`, lines 59‚Äì89:

```java
// line 70
final var maxTimestamp = timestamp.adjustUpperBound();

// line 73 ‚Äî only check: lower > upper, NOT future validation
if (minTimestamp > maxTimestamp) {
    throw new IllegalArgumentException("Invalid range provided for timestamp");
}

// line 77 ‚Äî future maxTimestamp ‚Üí future optimalLowerBound
final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
minTimestamp = Math.max(minTimestamp, optimalLowerBound);

// line 80-81 ‚Äî query with both bounds in the future
networkSupply = accountBalanceRepository.getSupplyHistory(
    lowerBounds, upperBounds, minTimestamp, maxTimestamp);

// line 84-86 ‚Äî no rows ‚Üí consensusTimestamp == 0 ‚Üí exception
if (networkSupply.consensusTimestamp() == 0L) {
    throw new EntityNotFoundException("Network supply not found");
}
```

**Root cause:** `Bound.adjustUpperBound()` (lines 63‚Äì74 of `Bound.java`) returns the raw user-supplied value with no upper ceiling against current time. The only guard in `getSupply` (line 73) checks `minTimestamp > maxTimestamp`, which is a range-consistency check, not a future-timestamp check.

**SQL behavior:** `AccountBalanceRepository.getSupplyHistory` (lines 12‚Äì28 of `AccountBalanceRepository.java`) uses `WHERE ab.consensus_timestamp BETWEEN :lowerTimestamp AND :upperTimestamp`. When both bounds are in the future, no rows exist, the CTE is empty, and the outer `SELECT` returns `(coalesce(sum(balance),0)=0, coalesce(max(consensus_timestamp),0)=0)`. This causes `consensusTimestamp() == 0L` ‚Üí `EntityNotFoundException`.

**Why existing checks fail:** The sole validation at line 73 only ensures the range is internally consistent (lower ‚â§ upper). It does not compare either bound against the current wall clock or the latest known consensus timestamp in the database.

### Impact Explanation

Any unauthenticated or unprivileged HTTP caller can make the `/api/v1/network/supply?timestamp=lte:9999999999.999999999` request (or equivalent far-future nanosecond timestamp). The endpoint will consistently throw `EntityNotFoundException` for that request. If downstream systems (e.g., transaction integrity checks) call this endpoint and treat the exception as a hard failure, supply data becomes unavailable for those checks. This is a targeted, repeatable denial-of-service on the supply endpoint requiring zero privileges.

### Likelihood Explanation

The exploit requires only a single HTTP GET request with a crafted `timestamp` query parameter ‚Äî no authentication, no special network position, no rate-limit bypass needed. The timestamp parameter is a standard, documented API parameter. Any external user who reads the API documentation can reproduce this. It is trivially repeatable and automatable.

### Recommendation

Add a validation step in `getSupply` that rejects any `maxTimestamp` exceeding the current consensus time (or a reasonable upper bound). For example, after computing `maxTimestamp`, query the latest known consensus timestamp from the database and throw `IllegalArgumentException` if `maxTimestamp` exceeds it. Alternatively, cap `maxTimestamp` to `System.currentTimeMillis() * 1_000_000L` (converted to nanoseconds) before proceeding. The fix should be applied before line 77 where `getFirstDayOfMonth` is called.

### Proof of Concept

1. Start the mirror node REST-Java service normally.
2. Send the following HTTP request (no authentication required):
   ```
   GET /api/v1/network/supply?timestamp=lte:9999999999.999999999
   ```
3. The service computes `maxTimestamp = 9999999999999999999` (nanoseconds, far future).
4. `getFirstDayOfMonth(maxTimestamp, -1)` returns the first day of the previous month relative to year ~2262, still in the future.
5. `minTimestamp` is set to this future `optimalLowerBound`.
6. `getSupplyHistory(...)` is called with both bounds in the future ‚Üí returns `(0, 0)`.
7. `networkSupply.consensusTimestamp() == 0L` ‚Üí `EntityNotFoundException("Network supply not found")` is thrown.
8. The API returns a 404/500 error. Repeat indefinitely with zero privileges.
