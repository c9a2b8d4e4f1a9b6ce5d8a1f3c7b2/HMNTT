### Title
Unauthenticated Unfiltered `/api/v1/balances` Requests Enable Resource Exhaustion DoS via CTE Full-Scan and No Rate Limiting

### Summary
The `getBalances()` function in `rest/balances.js` allows any unauthenticated user to issue requests with zero filter parameters. When no `timestamp` filter is present, the code falls into the `else` branch and constructs a CTE query against the `entity` table with no outer `WHERE` clause. The REST API has no application-level rate limiting, so an attacker can flood the endpoint with concurrent unfiltered requests, exhausting the database connection pool and CPU.

### Finding Description

**Exact code path:**

In `rest/balances.js`, `getBalances()` at lines 133–149:

```js
} else {
    const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
    const whereClause = conditions && `where ${conditions}`;   // line 136 — empty string when no filters
    ...
    sqlQuery = `
      with entity_balance as (
        select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
        from entity
        where type in ('ACCOUNT', 'CONTRACT')          -- only filter inside CTE
      )
      select ab.*, (${tokenBalanceSubQuery}) as token_balances
      from entity_balance ab
      ${whereClause}                                   -- empty — no outer WHERE
      order by ab.account_id ${order}
      ${limitQuery}`;                                  // LIMIT applied here
}
``` [1](#0-0) 

**Root cause — two compounding issues:**

1. **Empty `whereClause` is structurally permitted.** `conditions` is an empty string (falsy) when `accountQuery`, `pubKeyQuery`, and `balanceQuery` are all absent, so `whereClause` evaluates to `""`. The CTE itself only carries `WHERE type IN ('ACCOUNT', 'CONTRACT')`, which is a full-table-type scan of `entity`. [2](#0-1) 

2. **No application-level rate limiting exists in the REST API.** `server.js` registers cors, compression, httpContext, requestLogger, authHandler, metricsHandler, and cache middleware — but zero rate-limiting middleware. [3](#0-2) 

   The throttling found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` Java module and does not protect the Node.js REST API. [4](#0-3) 

**Why LIMIT does not fully mitigate this:**

`parseLimitAndOrderParams` always injects a `LIMIT` (default 25, max 100). [5](#0-4) 

- **PostgreSQL ≥ 12 (non-materialized CTE):** The planner can inline the CTE and use an index on `entity.id` to satisfy `ORDER BY account_id LIMIT 25` efficiently. However, each request still consumes a DB connection from the pool and executes a query. With no rate limiting, thousands of concurrent requests exhaust the pool.
- **PostgreSQL < 12 (materialized CTE):** CTEs are optimization fences — the entire `entity WHERE type IN ('ACCOUNT', 'CONTRACT')` result set is materialized in memory before `LIMIT` is applied. This is a genuine full table scan on every unfiltered request, regardless of LIMIT.

Additionally, each returned row triggers the correlated `token_account` subquery (`getTokenAccountBalanceSubQuery`), multiplying per-request DB work. [6](#0-5) 

**Existing checks reviewed and shown insufficient:**

- `utils.validateReq` only rejects *unknown* query parameters; it does not require any filter to be present. [7](#0-6) 
- `acceptedBalancesParameters` defines which parameters are *allowed*, not which are *required*. [8](#0-7) 
- The `authHandler` middleware supports optional authentication for elevated limits; unauthenticated requests are fully served. [9](#0-8) 

### Impact Explanation

An attacker sending a high volume of concurrent `GET /api/v1/balances` requests (no parameters) will:
1. Exhaust the PostgreSQL connection pool (`db.pool.maxConnections`), causing all subsequent queries — including those from legitimate users — to queue or fail.
2. On PostgreSQL < 12, force repeated full scans of the `entity` table, spiking CPU and I/O.
3. Trigger N correlated `token_account` subqueries per response (up to `tokenBalanceLimit.multipleAccounts` per row × 25–100 rows), amplifying DB load.

This constitutes a complete availability denial for all REST API consumers with no authentication or special capability required.

### Likelihood Explanation

The attack requires only an HTTP client and knowledge of the public API endpoint (documented in the OpenAPI spec). No credentials, tokens, or special network position are needed. The endpoint is publicly accessible by design. The attack is trivially scriptable (`ab`, `wrk`, `hey`, etc.) and fully repeatable. Infrastructure-level rate limiting (e.g., Traefik) may exist in some deployments but is not enforced by the application itself and is not guaranteed.

### Recommendation

1. **Require at least one meaningful filter** (e.g., `account.id` or `account.balance`) when no `timestamp` is provided, or return a 400 error for fully unfiltered requests.
2. **Add application-level rate limiting** to the Node.js REST server (e.g., `express-rate-limit`) keyed by IP, applied before route handlers.
3. **Add `MATERIALIZED` or `NOT MATERIALIZED` hint** explicitly to the CTE to control PostgreSQL behavior across versions, and ensure an index exists on `entity.type` or `(type, id)`.
4. **Set a DB statement timeout** (`db.pool.statementTimeout`) to bound the maximum duration of any single query.

### Proof of Concept

```bash
# Send 500 concurrent unfiltered requests to the balances endpoint
# No authentication, no filters — hits the empty-whereClause else-branch every time
wrk -t10 -c500 -d30s "http://<mirror-node-host>/api/v1/balances"

# Equivalent with curl in a loop
for i in $(seq 1 1000); do
  curl -s "http://<mirror-node-host>/api/v1/balances" &
done
wait
```

**Expected result:** DB connection pool exhaustion; subsequent requests return 500 errors or hang; PostgreSQL CPU spikes to 100% (especially on PostgreSQL < 12 due to CTE materialization).

const parseLimitAndOrderParams = (req, defaultOrder = constants.orderFilterValues.DESC) => {
  // Parse the limit parameter
  const limitQuery = `${constants.filterKeys.LIMIT} ? `;
  const limitValue = getLimitParamValue(req.query[constants.filterKeys.LIMIT]);

  // Parse the order parameters (default: descending)
  let order = defaultOrder;
  const value = req.query[constants.filterKeys.ORDER];
  if (value === constants.orderFilterValues.ASC || value === constants.orderFilterValues.DESC) {
    order = value;
  }

  return buildPgSqlObject(limitQuery, [limitValue], order, limitValue);
};
