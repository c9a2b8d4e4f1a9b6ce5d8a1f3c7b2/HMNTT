### Title
Cache Flooding via Unbounded Error-Entry Caching in PreparsedDocumentProvider Allows Cache Eviction DoS

### Summary
The `CachedPreparsedDocumentProvider` caches every query result — including parse/validation error entries — keyed by raw query string, with no rate limiting on the GraphQL endpoint. An unauthenticated attacker can flood the 1000-slot Caffeine cache with unique invalid queries, continuously evicting legitimate query entries and forcing re-parsing on every legitimate request for as long as the attack is sustained.

### Finding Description
**Exact code path:**

`CacheProperties.java` line 15 configures the cache as `expireAfterWrite=1h,maximumSize=1000,recordStats`. [1](#0-0) 

`CachedPreparsedDocumentProvider.java` line 26 caches the result of `parseAndValidateFunction.apply(executionInput)` unconditionally — whether the result is a valid document or a `PreparsedDocumentEntry` containing only errors. [2](#0-1) 

**Root cause:** There is no check on whether the cached `PreparsedDocumentEntry` contains errors before storing it. In graphql-java, `PreparsedDocumentEntry` wraps both valid documents and validation/parse error results. The implementation caches both indiscriminately. Combined with the absence of any rate limiting on the GraphQL endpoint (unlike the web3 module which has `requestsPerSecond=500`), any unauthenticated caller can submit an unlimited number of unique invalid queries.

**Exploit flow:**
1. Attacker generates 1000+ syntactically unique but invalid GraphQL queries (e.g., `{a1}`, `{a2}`, ..., `{a1000}` referencing non-existent fields, or malformed syntax variants). Each is unique enough to be a distinct cache key.
2. Each query is submitted to the GraphQL endpoint. The parser limits (`maxCharacters=10000`, `maxTokens=1000`) are satisfied by short queries; the queries fail validation but are still cached. [3](#0-2) 
3. Once 1000 slots are filled, Caffeine's W-TinyLFU eviction begins. The attacker continuously sends new unique invalid queries to maintain cache pressure, causing legitimate query entries to be evicted.
4. Every legitimate query that is evicted must be fully re-parsed and re-validated on the next request, increasing CPU load proportional to the attack rate.

**Why existing checks are insufficient:**
- Parser limits bound the *size* of each query but do not limit the *number* of unique queries submitted.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` apply at execution time, after the cache lookup — they do not prevent cache flooding. [4](#0-3) 
- There is no per-IP or global request rate limiter in the graphql module.

### Impact Explanation
The impact is performance degradation: legitimate queries are forced to re-parse on every request while the attack is active. Re-parsing is CPU-bound; under sustained attack with high query volume, this can degrade response latency and throughput. The service remains functional but operates without the benefit of the preparsed document cache. This is a non-network-based DoS affecting availability/performance of the GraphQL service.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of the schema beyond the ability to craft syntactically unique queries. The attacker needs only HTTP access to the GraphQL endpoint. The attack is trivially scriptable (a loop generating unique field names or random tokens within parser limits) and repeatable. Caffeine's W-TinyLFU does provide some resistance to one-time floods by protecting frequently-accessed entries, but sustained high-rate submission of new unique queries overcomes this protection.

### Recommendation
1. **Do not cache error entries:** In `CachedPreparsedDocumentProvider.getDocumentAsync()`, check whether the resulting `PreparsedDocumentEntry` has errors before caching it. Only cache entries with `!entry.hasErrors()`.
2. **Add rate limiting to the GraphQL endpoint:** Apply a per-IP or global request rate limiter (similar to web3's `requestsPerSecond` throttle) to bound the rate at which unique queries can be submitted.
3. **Optionally reduce cache TTL or size** for error entries if caching them is desired for deduplication purposes, using a separate smaller cache with a short TTL (e.g., `expireAfterWrite=10s,maximumSize=100`).

### Proof of Concept
```bash
# Generate and send 1100 unique invalid queries to fill and overflow the 1000-slot cache
for i in $(seq 1 1100); do
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ nonExistentField_${i} }\"}" &
done
wait

# Now send a legitimate query - it will not be in the cache and must be re-parsed
curl -X POST http://<graphql-host>/graphql \
  -H 'Content-Type: application/json' \
  -d '{"query": "{ account(input: {entityId: {shard: 0, realm: 0, num: 1}}) { alias } }"}'
# Repeat the flood loop to maintain cache pressure and prevent re-caching of legitimate queries
```
