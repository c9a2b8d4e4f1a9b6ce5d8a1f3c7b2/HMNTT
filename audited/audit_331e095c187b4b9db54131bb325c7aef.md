### Title
Unprivileged Repeated Queries Against SCHEDULECREATE Transaction IDs Force Double DB Query Per Request in `getTransactionsByIdOrHash`

### Summary
Any unauthenticated user who knows a successful SCHEDULECREATE transaction ID can query `/api/v1/transactions/{transactionId}` without the `scheduled=false` filter to deterministically trigger two sequential database queries per request instead of one. The `mayMissLongTermScheduledTransaction()` function returns `true` whenever the first query returns a successful SCHEDULECREATE without a corresponding scheduled-execution row, causing `getTransactionsByIdOrHash()` to unconditionally issue a second `pool.queryQuietly()` call with an extended timestamp range. Because SCHEDULECREATE transaction IDs are publicly visible on-chain and the response cache TTL for this case is only 5 seconds, an attacker can sustain a 2× DB query amplification indefinitely with no credentials.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactionsByIdOrHash` (lines 923–972):

```js
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);   // ← second DB query
});
``` [1](#0-0) 

`mayMissLongTermScheduledTransaction` (lines 974–996) returns `true` under the following conditions: [2](#0-1) 

1. `isTransactionHash` is `false` (query is by transaction ID, not hash).
2. `scheduled` is **not** `false` (i.e., the caller omitted the `scheduled` query param or passed `scheduled=true`).
3. The first query result contains a successful SCHEDULECREATE (`type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(result)`) **and** no row with `scheduled === true`.

**Root cause / failed assumption:**

The design assumes that callers who do not need the long-range re-query will pass `scheduled=false`. However, `scheduled=false` is entirely optional — it is one of only two accepted single-transaction parameters: [3](#0-2) 

There is no server-side enforcement that forces clients to include `scheduled=false` when they are not interested in the scheduled execution. Any omission of this filter on a SCHEDULECREATE transaction ID causes the double-query path to fire unconditionally.

**Why the cache does not mitigate this:**

`getTransactionsByIdOrHashCacheControlHeader` sets a short 5-second cache TTL (`SHORTER_CACHE_CONTROL_HEADER = {'cache-control': 'public, max-age=5'}`) precisely for the case where a successful SCHEDULECREATE exists but no scheduled execution has been seen yet: [4](#0-3) [5](#0-4) 

This means the cache expires every 5 seconds, and each cache miss re-triggers both DB queries.

### Impact Explanation

Every request that hits the double-query path consumes two DB connections and two full query executions instead of one. An attacker with a list of N known SCHEDULECREATE transaction IDs can sustain 2N DB queries per polling cycle (one cycle every 5 seconds per ID, or faster with multiple IDs or concurrent connections). At scale this degrades overall DB throughput for all users of the mirror node REST API, increasing query latency and potentially exhausting the DB connection pool. No economic damage to network participants occurs, but service quality degrades for all consumers of the public API.

### Likelihood Explanation

- **No privileges required**: the endpoint is public and unauthenticated.
- **Transaction IDs are public**: SCHEDULECREATE transaction IDs are visible in block explorers and the mirror node's own `/api/v1/transactions` list endpoint.
- **Trigger is deterministic**: as long as the scheduled execution has not yet appeared in the first query's timestamp window, every request without `scheduled=false` fires the second query. This window can be very long for long-expiry schedules.
- **No application-level rate limiting** is present in the REST API middleware for this endpoint (rate limiting found in the codebase applies to the web3/Rosetta services, not the REST transactions endpoint). [6](#0-5) 

### Recommendation

1. **Require `scheduled=false` implicitly when the caller does not supply a `scheduled` param and the first result already contains a successful SCHEDULECREATE**: only issue the second query when there is genuine ambiguity (e.g., the SCHEDULECREATE is recent enough that the scheduled execution could still be pending).
2. **Add a time-based guard inside `mayMissLongTermScheduledTransaction`**: only return `true` if `nowInNs() - transaction.consensus_timestamp < maxScheduledTransactionConsensusTimestampRangeNs`, mirroring the logic already present in `getTransactionsByIdOrHashCacheControlHeader`. This prevents the second query for schedules that have already expired.
3. **Apply per-IP or per-key request rate limiting** to the `/api/v1/transactions/:transactionIdOrHash` endpoint at the application layer, not just at the infrastructure layer.

### Proof of Concept

**Preconditions:**
- Identify any successful SCHEDULECREATE transaction on the network whose scheduled execution either has not yet occurred or occurred outside the default `maxTransactionConsensusTimestampRangeNs` window. These are trivially discoverable via `GET /api/v1/transactions?type=SCHEDULECREATE&result=success`.

**Trigger:**
```
# Repeat at will (every 5 seconds to bypass cache, or faster with multiple IDs)
GET /api/v1/transactions/0.0.12345-1700000000-000000000
# No scheduled=false parameter
```

**Observed result:**
- First `pool.queryQuietly` executes with the standard timestamp range.
- `mayMissLongTermScheduledTransaction` evaluates to `true` (successful SCHEDULECREATE in results, no scheduled row).
- Second `pool.queryQuietly` executes with `maxScheduledTransactionConsensusTimestampRangeNs` upper bound.
- Two DB queries consumed per HTTP request instead of one.
- With 100 concurrent clients each cycling through 10 known SCHEDULECREATE IDs every 5 seconds: 100 × 10 × 2 = 2,000 DB queries per 5-second window vs. the expected 1,000, sustained indefinitely with no credentials.

const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
```
