### Title
Unbounded `endTime` in `TopicMessageFilter` Bypasses `pastEndTime` Termination, Enabling Indefinite gRPC Stream Exhaustion

### Summary
An unprivileged user can submit a `TopicMessageFilter` with `endTime = Long.MAX_VALUE`, which passes the only validation check (`endTime > startTime`). The `TopicContext.isComplete()` method evaluates this as a timestamp in year ~2262 and never returns `true`, causing `pastEndTime()` to poll a scheduler thread every 30 seconds indefinitely without ever emitting a termination signal. Combined with the absence of a global subscription cap, an attacker opening many connections can exhaust gRPC stream slots, scheduler threads, and memory.

### Finding Description

**Validation failure** — `TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`:

```java
// TopicMessageFilter.java, lines 43-46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

`Long.MAX_VALUE` (9,223,372,036,854,775,807 ns from epoch ≈ year 2262) trivially satisfies this. No upper-bound constraint exists.

**`isComplete()` never returns `true`** — `TopicContext.isComplete()` at lines 203–214:

```java
boolean isComplete() {
    if (filter.getEndTime() == null) { return false; }
    if (filter.getEndTime() < startTime) { return true; }
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());
}
```

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to year 2262. Adding `endTimeInterval` (default 30 s) still yields year 2262. `.isBefore(Instant.now())` is permanently `false`.

**`pastEndTime()` never emits** — lines 123–131:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // <-- null path: no scheduler overhead
    }
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```

With `endTime = Long.MAX_VALUE`, the non-null branch is taken. The returned Flux repeats `Flux.empty()` every 30 s as long as `!isComplete()` — which is always `true`. It never emits an item, so `takeUntilOther(pastEndTime(...))` at line 73 never fires. The `takeWhile(t -> t.getConsensusTimestamp() < Long.MAX_VALUE)` at line 80 also never terminates since no real message carries that timestamp.

**No global subscription cap** — `subscriberCount` (line 48) is a metric gauge only; no enforcement gate exists. `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14) limits per-connection concurrency but not the total number of connections.

### Impact Explanation

Each subscription with `endTime = Long.MAX_VALUE` and no `limit`:
- Holds a gRPC server stream open indefinitely.
- Schedules a recurring task every 30 s on the Reactor scheduler to poll `isComplete()`.
- Consumes memory for the `TopicContext`, Reactor operator chain, and associated state.

An attacker opening N connections × 5 streams each creates 5N permanently open subscriptions. At sufficient scale this exhausts the bounded-elastic scheduler thread pool, gRPC stream slots, and heap, degrading or denying service to legitimate subscribers. The mirror node's gRPC service is the primary interface for HCS topic subscriptions, so degradation here directly impacts network observability.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (default 5600) is publicly accessible. The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open multiple TCP connections. The filter value `Long.MAX_VALUE` is a single integer field. The attack is trivially scriptable and repeatable from a single machine or botnet.

### Recommendation

1. **Enforce a maximum `endTime` offset** in `TopicMessageFilter.isValidEndTime()`, e.g., reject any `endTime` more than a configurable window (e.g., 30 days) beyond `DomainUtils.now()`.
2. **Enforce a global subscription limit** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return an error when exceeded.
3. **Add per-IP connection rate limiting** at the Netty/gRPC layer to prevent connection flooding.
4. Optionally, treat `endTime = null` and `endTime > now + maxWindow` identically (both as live subscriptions) to eliminate the scheduler overhead difference.

### Proof of Concept

```
# Prerequisites: grpcurl installed, mirror node gRPC at localhost:5600,
# a valid topic ID (e.g., 0.0.12345)

# Open 5 permanent subscriptions per connection, repeat across many connections:
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{
    "filter": {
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
      "consensusStartTime": {"seconds": 0},
      "consensusEndTime":   {"seconds": 9223372036, "nanos": 854775807},
      "limit": 0
    }
  }' localhost:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
```
