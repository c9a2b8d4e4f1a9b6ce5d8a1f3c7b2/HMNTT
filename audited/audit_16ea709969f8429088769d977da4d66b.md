### Title
Unauthenticated GraphQL Endpoint Lacks Rate Limiting, Enabling DB Connection Pool Exhaustion via `getByIdAndType()`

### Summary
The GraphQL module's `EntityServiceImpl.getByIdAndType()` issues an uncached `findById()` DB query for every request, and the GraphQL endpoint has no rate limiting. An unprivileged attacker can flood the endpoint with concurrent requests using sequential `EntityId` values, exhausting the HikariCP connection pool and denying service to legitimate users. During a network partition (where DB connections are held until `statementTimeout` expires), the attack requires far fewer concurrent requests to achieve full pool exhaustion.

### Finding Description

**Code path:**

`AccountController.account()` → `EntityService.getByIdAndType()` → `EntityRepository.findById()` [1](#0-0) 

```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
}
```

The graphql `EntityRepository` extends `CrudRepository<Entity, Long>` with **no `@Cacheable` annotation**: [2](#0-1) 

Every call to `getByIdAndType()` unconditionally issues a DB query. Compare with the `grpc` module's `EntityRepository`, which applies `@Cacheable`: [3](#0-2) 

**No rate limiting on the GraphQL endpoint.** `GraphQlConfiguration` only configures query complexity/depth limits and JSON parsing limits — there is no RPS throttle: [4](#0-3) 

The `ThrottleManagerImpl` (bucket4j-based RPS throttle) exists only in the `web3` module: [5](#0-4) 

It is not wired into the GraphQL module at all. The GraphQL README confirms no rate-limiting properties exist for this module: [6](#0-5) 

**Root cause:** The graphql `EntityServiceImpl` assumes each `findById()` call is cheap (cached or infrequent), but there is no caching and no request rate control. The HikariCP pool (Spring Boot default: 10 connections) is the only resource bound.

**Exploit flow:**

1. Attacker sends N concurrent POST requests to `POST /graphql/alpha` (public, no auth):
   ```json
   {"query": "{account(input: {entityId: {shard: 0, realm: 0, num: X}}) { balance }}"}
   ```
   with X = 1, 2, 3, … (sequential, so no cache hit is possible even if caching were added naively).
2. Each request acquires a HikariCP connection and issues `SELECT * FROM entity WHERE id = X`.
3. During a network partition, the DB is unreachable; each connection is held until `statementTimeout` (default 10 seconds per the README).
4. With 10 concurrent requests, all pool connections are held for up to 10 seconds each.
5. All subsequent legitimate requests block waiting for a pool connection, then time out.

### Impact Explanation

Full denial of service for the GraphQL API. Legitimate users receive no responses while the pool is exhausted. The attack is amplified during a network partition because each held connection occupies a pool slot for the full `statementTimeout` duration (10 seconds), meaning only ~1 request/second per pool connection is needed to sustain exhaustion. The GraphQL API is documented as publicly accessible with no authentication. [7](#0-6) 

### Likelihood Explanation

**Preconditions:** None. The endpoint is public, unauthenticated, and reachable over HTTP. No special knowledge or credentials are required. Sequential `EntityId` values (shard=0, realm=0, num=N) are trivially enumerable. A single attacker with a modest HTTP client (e.g., `curl` in a loop or any load-testing tool) can sustain the attack. The attack is repeatable and requires no state.

### Recommendation

1. **Add rate limiting to the GraphQL module.** Port the existing `ThrottleManagerImpl`/bucket4j pattern from the `web3` module, or add a Spring `HandlerInterceptor`/servlet filter that enforces a per-IP or global RPS limit on `/graphql/alpha`.
2. **Add caching to `EntityRepository.findById()`** in the graphql module, mirroring the `@Cacheable` pattern used in the `grpc` module's `EntityRepository`.
3. **Set a DB statement timeout** at the connection pool level (via `spring.datasource.hikari.connection-timeout` and `connectionInitSql` setting `statement_timeout`) to bound how long a partition can hold a pool connection.
4. **Increase pool size** or configure a short `connectionTimeout` so blocked requests fail fast rather than queuing indefinitely.

### Proof of Concept

```bash
# Exhaust the connection pool with 15 concurrent requests (exceeds default pool of 10)
# During a network partition, each holds a connection for statementTimeout (10s)
for i in $(seq 1 15); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance }}\"}" &
done

# Legitimate request now blocks/times out:
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance }}"}'
# Expected: hangs or returns connection pool timeout error
```
