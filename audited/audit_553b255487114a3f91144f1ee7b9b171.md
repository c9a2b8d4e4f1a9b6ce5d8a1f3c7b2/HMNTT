### Title
Unauthenticated Global Rate Limit Bypass via Minimal-Output EVM Re-Execution in `getContractOpcodes`

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint performs a full EVM re-execution for every request regardless of the `stack`, `memory`, and `storage` flag values. The only throttle protecting this endpoint is a single global bucket (default: 1 req/sec), not per-IP or per-user. An unauthenticated attacker can continuously saturate this global quota by sending requests with `stack=false&memory=false&storage=false`, triggering maximum server-side CPU cost while receiving a minimal response, effectively monopolizing the resource and denying service to all legitimate users.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) accepts `stack`, `memory`, `storage` as optional boolean query parameters with no authentication check:

```java
@GetMapping(value = "/{transactionIdOrHash}/opcodes")
OpcodesResponse getContractOpcodes(
        @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
        @RequestParam(required = false, defaultValue = "true") boolean stack,
        @RequestParam(required = false, defaultValue = "false") boolean memory,
        @RequestParam(required = false, defaultValue = "false") boolean storage,
        @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
    if (properties.isEnabled()) {
        validateAcceptEncodingHeader(acceptEncoding);
        throttleManager.throttleOpcodeRequest();          // <-- global bucket only
        final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
        return opcodeService.processOpcodeCall(request);  // <-- always re-executes EVM
    }
``` [1](#0-0) 

`ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–56) consumes from a **single global** `opcodeRateLimitBucket` — there is no per-IP, per-session, or per-user dimension:

```java
@Override
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
``` [2](#0-1) 

The bucket is configured globally with a default of `opcodeRequestsPerSecond = 1`: [3](#0-2) [4](#0-3) 

Inside `OpcodeServiceImpl.processOpcodeCall()`, the flags are passed to `OpcodeContext` but `contractDebugService.processOpcodeCall(params, opcodeContext)` is **unconditionally invoked** — the EVM re-executes the full transaction regardless of flag values. The flags only gate what data is collected into the response: [5](#0-4) 

**Root cause:** The failed assumption is that the `stack/memory/storage` flags reduce server-side computational cost. They do not — they only reduce response payload size. The EVM re-execution (the expensive operation) always runs. Combined with a global (not per-source) rate limiter and zero authentication, a single attacker can monopolize the entire allowed request quota.

### Impact Explanation
A single unauthenticated attacker sending requests at the maximum allowed rate (1/sec by default) with `stack=false&memory=false&storage=false` and `Accept-Encoding: gzip`:
- Triggers full EVM re-execution of the most gas-intensive historical transactions on every request
- Receives a minimal response (no stack/memory/storage data), minimizing their own bandwidth cost
- Consumes the entire global `opcodeRateLimitBucket`, causing HTTP 429 for all other legitimate users
- Sustains maximum server CPU load indefinitely with minimal client-side cost

This is a denial-of-service against the opcode tracing feature and a resource-abuse primitive (free EVM computation oracle).

### Likelihood Explanation
Preconditions are trivially met: no account, no API key, no special privilege is required. The attacker only needs to know a valid historical transaction hash (publicly available on-chain) and send a standard HTTP GET with `Accept-Encoding: gzip`. The attack is fully automatable, repeatable, and requires no special tooling. The global (not per-IP) throttle means a single attacker from a single IP can fully block all other users.

### Recommendation
1. **Per-IP / per-client rate limiting**: Replace or augment the global `opcodeRateLimitBucket` with a per-source-IP bucket (e.g., using Bucket4j's distributed or local per-key approach).
2. **Authentication gate**: Require an API key or authenticated session for this endpoint, consistent with its high computational cost.
3. **Gas-based throttle parity**: Apply the same gas-scaled throttle used by `ContractController` (which deducts from `gasLimitBucket` proportional to the transaction's gas limit) so that re-executing high-gas transactions costs proportionally more quota.
4. **Flag-aware cost reduction**: If `stack=false&memory=false&storage=false`, consider whether a cached result or a lighter execution path can be used to avoid full tracer overhead.

### Proof of Concept
```bash
# Obtain any valid historical contract transaction hash from the network
TX_HASH="0xabc123..."

# Continuously trigger full EVM re-execution with minimal response
while true; do
  curl -s -o /dev/null \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=false&memory=false&storage=false"
  # Repeat at 1/sec to saturate the global opcodeRateLimitBucket
  sleep 1
done

# Simultaneously, a legitimate user's request returns HTTP 429:
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes"
# Expected: HTTP 429 Too Many Requests
```

    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
