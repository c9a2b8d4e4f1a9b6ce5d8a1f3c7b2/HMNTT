### Title
Unauthenticated DB Query Amplification via Pending SCHEDULECREATE Transaction ID Lookup

### Summary
In `rest/transactions.js`, `getTransactionsByIdOrHash()` unconditionally issues a second, wider-range `pool.queryQuietly()` call whenever `mayMissLongTermScheduledTransaction()` returns `true`. This condition is deterministically and persistently `true` for any successful SCHEDULECREATE transaction that has not yet executed its scheduled inner transaction. Because the endpoint has no per-IP or per-resource rate limiting, any unauthenticated user can flood requests against a known pending SCHEDULECREATE transaction ID, doubling DB query load per request with the second query scanning a significantly larger timestamp range.

### Finding Description
**Exact code path:**

`rest/transactions.js`, `getTransactionsByIdOrHash()`, lines 930–938:
```js
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);  // second, wider-range query
});
```

`mayMissLongTermScheduledTransaction()`, lines 974–996, returns `true` when all of:
- `isTransactionHash === false` (query by transaction ID, not hash)
- `scheduled === undefined` (no `?scheduled=` query param — the default for any plain request)
- The first query result contains a successful SCHEDULECREATE (`type === 42`, `result` in `SUCCESS_PROTO_IDS`)
- The first query result does **not** contain a row with `scheduled === true` (the inner transaction has not yet executed)

This state is **persistent** for the entire lifetime of a long-term scheduled transaction — potentially days or weeks. Every request during this window unconditionally triggers the second query with `maxScheduledTransactionConsensusTimestampRangeNs` as the upper bound, which is a much wider timestamp scan than the first query.

**Root cause / failed assumption:** The design assumes the second query path is rare and short-lived. It fails to account for long-term scheduled transactions and the absence of any request-rate control on the endpoint.

**Why existing checks fail:**
- `server.js` (lines 67–98) shows no rate-limiting middleware applied to the transactions route — only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, and optional metrics/cache handlers.
- The optional response cache (`responseCacheCheckHandler`/`responseCacheUpdateHandler`) is gated on `config.cache.response.enabled && config.redis.enabled` (line 54 of `server.js`) and is disabled by default.
- Even when the cache is active, `getTransactionsByIdOrHashCacheControlHeader()` (lines 895–916) sets only `SHORTER_CACHE_CONTROL_HEADER = {'cache-control': 'public, max-age=5'}` — a 5-second TTL — for pending SCHEDULECREATE responses, meaning the cache provides negligible protection under a sustained flood.
- `acceptedSingleTransactionParameters` (line 1014) only accepts `nonce` and `scheduled` filters; there is no mechanism to throttle or deduplicate identical repeated lookups.

### Impact Explanation
Each request against a pending SCHEDULECREATE transaction ID causes two DB queries instead of one. The second query uses `maxScheduledTransactionConsensusTimestampRangeNs` as its upper timestamp bound — a range far larger than the first query's `maxTransactionConsensusTimestampRangeNs` — making it materially more expensive. Under a sustained flood from a single unauthenticated client, the DB connection pool becomes saturated with wide-range scans, degrading query throughput for all users and delaying transaction visibility across the mirror node. The amplification factor is 2× per request at minimum, with the second query's cost potentially much higher due to the expanded timestamp range.

### Likelihood Explanation
Preconditions are trivially met: SCHEDULECREATE transaction IDs are publicly visible on the Hedera network (via the mirror node's own `/api/v1/transactions` list endpoint or any block explorer). No authentication, API key, or privileged access is required. The attacker needs only a valid pending SCHEDULECREATE transaction ID and the ability to send HTTP GET requests. The attack is fully repeatable for the entire duration the scheduled transaction remains pending. A single attacker with modest bandwidth can sustain thousands of requests per second against the endpoint.

### Recommendation
1. **Add per-IP rate limiting** to the `/api/v1/transactions/:transactionIdOrHash` route (e.g., via `express-rate-limit` or an ingress-level rate limiter such as Traefik's `rateLimit` middleware, already used for other services in the repo).
2. **Cache the "pending SCHEDULECREATE" response at the application layer** with a short but non-trivial TTL (e.g., 30 seconds) keyed on the transaction ID, so repeated identical requests are served from cache without hitting the DB at all.
3. **Bound the second query** by also checking elapsed time since the SCHEDULECREATE consensus timestamp: if the elapsed time is less than a minimum threshold (e.g., one consensus round), skip the second query entirely.
4. **Consider making the response cache mandatory** (not opt-in) for the single-transaction lookup endpoint, or at minimum document the DoS risk of deploying without it.

### Proof of Concept
1. Identify any pending SCHEDULECREATE transaction on the Hedera network (e.g., via `GET /api/v1/transactions?transactiontype=SCHEDULECREATE&result=success&order=desc&limit=1`). Note the `transaction_id` field, e.g., `0.0.1234-1700000000-000000000`.
2. Confirm the scheduled inner transaction has not yet executed (no `scheduled: true` entry in the response).
3. Run the following flood loop (no authentication required):
   ```bash
   while true; do
     curl -s "https://<mirror-node>/api/v1/transactions/0.0.1234-1700000000-000000000" -o /dev/null &
   done
   ```
4. Observe via DB metrics (pg_stat_activity, slow query logs) that each request generates two queries — the second with a timestamp upper bound of `valid_start_ns - maxValidStartTimestampDriftNs + maxScheduledTransactionConsensusTimestampRangeNs` — and that DB CPU/IO climbs proportionally to request rate.

```javascript
const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
```
