### Title
Unauthenticated DoS via `BufferUnderflowException` in `getByEvmAddressAndType()` with Under-Length EVM Address

### Summary
`decodeEvmAddress()` performs no length validation on the decoded byte array, allowing any caller to supply a 19-byte (38 hex-char) EVM address. When the first 12 bytes are zero, the two zero-checks on line 37 both pass, and the subsequent `buffer.getLong()` on line 38 attempts to read 8 bytes from a buffer with only 7 remaining, throwing an uncaught `BufferUnderflowException` that crashes the request. No authentication or privilege is required.

### Finding Description
**Code path:**

- `GraphQlUtils.decodeEvmAddress()` — `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 87–98: strips the `0x` prefix and calls `Hex.decodeHex()`. There is no check that the resulting byte array is exactly 20 bytes.
- `EntityServiceImpl.getByEvmAddressAndType()` — `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41: wraps the raw byte array in a `ByteBuffer` and reads `getInt()` (4 bytes) then `getLong()` (8 bytes) as a guard condition, then reads a second `getLong()` (8 bytes) on line 38 to obtain the entity ID.

**Byte-by-byte trace for a 19-byte input:**

| Operation | Bytes consumed | Buffer position | Remaining |
|---|---|---|---|
| `ByteBuffer.wrap(19 bytes)` | — | 0 | 19 |
| `buffer.getInt()` (line 37) | 4 | 4 | 15 |
| `buffer.getLong()` (line 37) | 8 | 12 | 7 |
| `buffer.getLong()` (line 38) | needs 8 | — | **only 7 → `BufferUnderflowException`** |

The condition on line 37 uses short-circuit `&&`, so the second `getLong()` on line 37 is only evaluated when `getInt() == 0`. The third `getLong()` on line 38 is only reached when **both** prior reads return zero. Therefore the attacker must supply a 19-byte address whose first 12 bytes are all `0x00`.

**Why existing checks fail:**

- `AccountController` applies `@Valid` to `AccountInput` (line 33), but there is no `@Size` or pattern constraint on the `evmAddress` field that enforces exactly 40 hex characters.
- `decodeEvmAddress()` only catches `DecoderException` (odd-length or non-hex input) and re-throws as `IllegalArgumentException`; it never checks `result.length == 20`.
- The GraphQL endpoint `/graphql/alpha` is publicly reachable with no authentication.

### Impact Explanation
Every `account` query that supplies an `evmAddress` value of exactly 38 hex characters with the first 24 characters being `0` will throw `BufferUnderflowException`. Spring GraphQL propagates this as an unhandled runtime exception, returning an error response to the caller and logging a stack trace server-side. The attack is repeatable at will, constitutes a targeted request-level DoS against the account lookup path, and produces noisy error logs that can obscure legitimate operational alerts.

### Likelihood Explanation
The GraphQL schema and the `evmAddress` input field are publicly documented and exercised by the Postman collection shipped in the repository. An attacker needs only to craft a single GraphQL query with a crafted string — no credentials, no tokens, no prior knowledge of internal state. The payload is trivially constructable and the crash is 100% reproducible on every matching request.

### Recommendation
Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// In GraphQlUtils.decodeEvmAddress(), after Hex.decodeHex():
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException(
        "evmAddress must be exactly 20 bytes, got " + decoded.length);
}
return decoded;
```

Alternatively, add a Bean Validation constraint on the `evmAddress` field in `AccountInput`:

```java
@Pattern(regexp = "^(0x)?[0-9a-fA-F]{40}$")
private String evmAddress;
```

Either fix prevents the under-length buffer from ever reaching the `ByteBuffer` reads.

### Proof of Concept
Send the following GraphQL request to `/graphql/alpha` (no authentication required):

```graphql
{
  account(input: {
    evmAddress: "0x000000000000000000000000aabbccddeeff00"
  }) {
    entityId { shard realm num }
  }
}
```

- The hex string is 38 chars after the `0x` prefix → 19 decoded bytes.
- Bytes 0–3 are `00 00 00 00` → `getInt() == 0` ✓
- Bytes 4–11 are `00 00 00 00 00 00 00 00` → `getLong() == 0` ✓
- `buffer.getLong()` on line 38 attempts to read bytes 12–19 but only bytes 12–18 exist → `java.nio.BufferUnderflowException` is thrown, the request crashes with a 200-envelope GraphQL error response, and a stack trace is written to the server log. [1](#0-0) [2](#0-1) [3](#0-2)

        if (evmAddress == null) {
