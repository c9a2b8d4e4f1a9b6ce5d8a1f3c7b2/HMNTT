### Title
Safety-Check Path in `missingMessages()` Allows Unauthenticated Users to Force Expensive Historical Scans via `startTime=0`

### Summary
Any unauthenticated gRPC client can subscribe to a valid topic with `consensusStartTime` set to epoch zero (`{seconds:0, nanos:0}`). After exactly one second, the unconditional safety-check timer fires and — because no messages have been received yet (`last == null`) — calls `topicMessageRetriever.retrieve()` with `startTime = 0` and `throttled = false`. This triggers the unthrottled polling retriever to issue up to 12 sequential queries, each fetching up to 5,000 rows, scanning all historical messages for that topic from the beginning of time. With many concurrent subscriptions, this degrades database performance for all other workloads.

### Finding Description

**Exact code path:**

1. **Input accepted without lower-bound guard** — `ConsensusController.toFilter()` passes the client-supplied `consensusStartTime` directly to `TopicMessageFilter.startTime`. The only validation on `startTime` is `@Min(0)` and `startTime <= DomainUtils.now()`, both of which `0` satisfies. [1](#0-0) [2](#0-1) [3](#0-2) 

2. **Safety check fires unconditionally after 1 s** — `subscribeTopic()` schedules a `Mono.delay(1s)` that calls `missingMessages(topicContext, null)` for every subscription that has not yet completed. For a topic with no recent messages, `topicContext.isComplete()` returns `false` (no `endTime` set), so the check always fires. [4](#0-3) [5](#0-4) 

3. **`missingMessages(ctx, null)` uses original `startTime` when `last == null`** — Because no messages have been delivered, `topicContext.getLast()` is `null`, so the branch falls through to `topicContext.getFilter().getStartTime()`, which is `0`. [6](#0-5) 

4. **Unthrottled retriever issues up to 12 × 5,000-row queries** — `retrieve(gapFilter, false)` uses `throttled=false`, which sets `numRepeats = maxPolls = 12` and `maxPageSize = 5000`. With `startTime = 0`, each poll executes `WHERE topic_id = X AND consensus_timestamp >= <last+1>` starting from epoch zero, scanning the entire message history for that topic. [7](#0-6) [8](#0-7) 

**Root cause:** The failed assumption is that the safety-check path is only reached when the importer is temporarily down (i.e., messages exist but are delayed). The code does not account for an attacker deliberately subscribing with `startTime = 0` to a quiet topic, which makes `last == null` a permanent condition for the lifetime of the subscription, guaranteeing the expensive fallback query fires every time the safety check triggers.

**Why existing checks are insufficient:**
- `@Min(0)` explicitly permits `startTime = 0`.
- `isValidStartTime()` only checks `startTime <= now()`, which `0` trivially satisfies.
- There is no authentication, no per-client rate limit, and no cap on concurrent subscriptions.
- The `isComplete()` guard only returns `true` when an `endTime` is set and has elapsed; without `endTime`, it always returns `false`. [5](#0-4) 

### Impact Explanation
Each subscription with `startTime = 0` to a high-volume topic causes up to 60,000 rows to be read from the `topic_message` table (12 polls × 5,000 rows) within the first two seconds of connection. Because the query is `WHERE topic_id = X AND consensus_timestamp >= 0`, it performs a full index range scan over the entire history of that topic. With N concurrent connections, the database receives N × 12 queries in rapid succession. This competes directly with the importer's write path and with other subscribers' read queries, degrading overall transaction-processing throughput. The `boundedElastic` scheduler provides some back-pressure but does not prevent the queries from being issued.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond a valid topic ID (which is public on-chain), and no sophisticated tooling — a standard gRPC client suffices. The trigger is deterministic (fires after exactly 1 second) and repeatable (a new connection resets the timer). An attacker can open hundreds of connections from a single host. The attack is therefore highly feasible and easily automated.

### Recommendation
1. **Enforce a minimum `startTime` floor at the service layer** — reject or clamp `startTime` values older than a configurable horizon (e.g., 24 hours before `now()`), or at minimum require `startTime > 0`.
2. **Guard the safety-check branch** — before calling `retrieve()` with `throttled=false`, verify that `startTime` is not suspiciously old (e.g., `startTime < now() - maxHistoricalWindow`). If it is, skip the safety check or use the throttled retriever.
3. **Add per-client connection/rate limiting** — limit the number of concurrent subscriptions per source IP at the gRPC interceptor layer.
4. **Separate the safety-check `startTime` from the user-supplied `startTime`** — record `DomainUtils.now()` at subscription time and use that as the floor for the safety-check query when `last == null`, rather than the user-supplied value.

### Proof of Concept
```
# Prerequisites: a valid topic ID (e.g., 0.0.12345) with no recent messages.
# Tool: grpcurl or any HCS gRPC client.

# Step 1 – open a subscription with consensusStartTime = epoch zero
grpcurl -plaintext \
  -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0,"nanos":0}}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# Step 2 – repeat N times to amplify load
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":0,"nanos":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# After ~1 second, each connection triggers missingMessages(ctx, null) with startTime=0.
# The unthrottled retriever issues up to 12 queries × 5000 rows per connection.
# 200 connections → up to 2,400 concurrent DB queries scanning topic history from epoch 0.
# Observable: DB CPU/IO spikes; importer write latency increases; other subscribers stall.
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
