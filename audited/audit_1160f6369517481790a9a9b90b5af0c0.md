### Title
`scaleGas()` Floor-Division Truncation Allows Systematic Throttle Undercount, Enabling Up to ~100% Gas Amplification Per Token

### Summary
The `scaleGas()` method in `ThrottleProperties` uses `Math.floorDiv(gas, 10_000)`, which silently discards up to 9,999 gas units per request. An unprivileged attacker can exploit this by always submitting gas values just below a 10,000-unit boundary (e.g., 19,999 instead of 10,001), consuming the same number of throttle tokens while providing the EVM up to ~100% more gas budget than the throttle accounts for. No authentication or special privileges are required.

### Finding Description

**Exact code location:**

`ThrottleProperties.java`, lines 42–47:
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

`ThrottleManagerImpl.java`, line 40:
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
```

**Root cause:** `Math.floorDiv` truncates the remainder. Every gas value in the half-open interval `(N×10,000, (N+1)×10,000)` maps to exactly `N` tokens, regardless of whether the actual gas is `N×10,000 + 1` or `N×10,000 + 9,999`. The throttle's core assumption — that token cost is proportional to computational load — is violated by up to 9,999 gas units per request.

**Exploit flow:**

1. Attacker identifies the truncation: `scaleGas(10_001) == 1` and `scaleGas(19_999) == 1`.
2. Attacker always submits `gas = N×10,000 − 1` (e.g., 19,999; 29,999; 99,999).
3. The bucket deducts `N−1` tokens instead of the `N` tokens that would be deducted for `gas = N×10,000`.
4. The EVM receives a gas budget of `N×10,000 − 1`, executing up to that limit.

**Why existing checks fail:**

- **RPS limit (500/s):** Limits request count but does not correct the per-request gas undercount. With gas=19,999 and 500 RPS, the throttle counts 500 tokens/s but the EVM processes up to 9,999,500 gas/s — nearly 2× what 500 tokens represent (5,000,000 gas).
- **`maxGasLimit` (15,000,000):** Caps the absolute gas ceiling but does not fix the truncation; at gas=14,999,999 the undercount is 9,999 units per request.
- **`gasLimitRefundPercent` restore path:** `restore()` also calls `scaleGas()` (line 60 of `ThrottleManagerImpl`), so the refund is equally truncated — the bucket never recovers the "free" gas fraction either.

### Impact Explanation

The gas throttle is the primary mechanism protecting processing nodes from computational overload. The truncation allows every request to carry up to 9,999 "free" gas units that the throttle never accounts for. In the worst-case band (gas ∈ [10,001, 19,999]):

- Throttle cost: 1 token
- Actual EVM budget: up to 19,999 gas (~100% over the 10,000-gas baseline the token represents)

At 500 RPS (the default rate limit), an attacker sustains ~10M gas/s of EVM load while the throttle records only ~5M gas/s consumed — a sustained 2× overload relative to what the throttle believes it is permitting. Across multiple coordinated clients (each within the per-IP RPS limit), the aggregate undercount scales linearly, potentially driving processing nodes to resource exhaustion without triggering the gas throttle.

### Likelihood Explanation

- **No privileges required:** Any caller of `POST /api/v1/contracts/call` can exploit this.
- **Trivially repeatable:** The attacker simply hard-codes `gas = 19999` (or any `N×10,000 − 1`) in every request.
- **No detection signal:** The throttle never fires; requests appear normal in all metrics.
- **Amplification is deterministic:** The attacker can calculate the optimal gas value analytically.

### Recommendation

Replace floor division with ceiling division (or rounding) so that the token cost always meets or exceeds the actual gas:

```java
public long scaleGas(long gas) {
    if (gas <= 0) return 0L;
    // Ceiling division: ensures partial 10,000-gas blocks cost at least 1 token
    return (gas + GAS_SCALE_FACTOR - 1) / GAS_SCALE_FACTOR;
}
```

This ensures `scaleGas(10_001) == 2` and `scaleGas(19_999) == 2`, eliminating the free-gas window. The special-case `gas <= GAS_SCALE_FACTOR → 0` should also be removed or changed to `return 1L` so that even small requests consume at least one token.
