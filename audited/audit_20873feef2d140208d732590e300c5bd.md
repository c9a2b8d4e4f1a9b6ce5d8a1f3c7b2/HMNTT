### Title
Last-Subscriber Cancel Griefing via `doOnCancel` Before `.share()` in `RedisTopicListener`

### Summary
In `RedisTopicListener.subscribe()`, the `doOnCancel(() -> unsubscribe(topic))` side-effect is chained on the upstream Flux before `.share()`. When an attacker is the last (or only) subscriber and cancels, Reactor's `.share()` (i.e., `publish().refCount()`) propagates a cancel upstream, firing `doOnCancel`, which calls `unsubscribe(topic)` and removes the entry from `topicMessages`. Any subsequent legitimate subscriber is forced to re-establish a brand-new Redis channel subscription, creating a message gap during re-subscription. The attack requires no credentials and is trivially repeatable.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–80:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    return container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> unsubscribe(topic))   // line 74 — fires when share() cancels upstream
            .doOnComplete(() -> unsubscribe(topic))
            ...
            .retryWhen(...)
            .share();                               // line 79 — cancels upstream when last sub leaves
}

private void unsubscribe(Topic topic) {
    topicMessages.remove(topic.getTopic());         // line 83 — evicts the shared Flux
}
```

**Root cause:** `.share()` is `publish().refCount()`. When the subscriber count drops to zero, `refCount` cancels the upstream subscription. That cancel signal propagates through `.retryWhen()` and triggers `doOnCancel`, which calls `unsubscribe(topic)` → `topicMessages.remove(...)`. The next call to `getSharedListener` via `computeIfAbsent` (line 61) finds no entry and creates a fresh Redis subscription, introducing a gap.

**Failed assumption:** The design assumes that `doOnCancel` only fires during legitimate teardown (i.e., when the service is shutting down or the topic is truly idle). In reality, any client that becomes the last subscriber and cancels triggers the same path.

**Exploit flow:**
1. Attacker opens a gRPC `subscribeTopic` stream for topic T (no auth required — `ConsensusController.subscribeTopic` has no authentication gate).
2. If no other subscriber is active, the attacker's subscription is the only one; `.share()` has exactly one downstream subscriber.
3. Attacker immediately cancels the gRPC stream → `ServerCallStreamObserver.setOnCancelHandler(disposable::dispose)` (line 51 of `ConsensusController`) disposes the subscription → `.share()` sees zero subscribers → cancels upstream → `doOnCancel` fires → `topicMessages.remove("topic.<id>")`.
4. A legitimate subscriber arriving now hits `computeIfAbsent`, gets a cache miss, and calls `subscribe(topic)` again, re-establishing a new Redis `SUBSCRIBE` command.
5. Messages published to the Redis channel between step 3 and the new subscription becoming active are permanently lost to that subscriber.
6. Attacker repeats steps 1–5 in a tight loop to keep the gap open continuously.

**Why existing checks are insufficient:**
- `ConcurrentHashMap.computeIfAbsent` (line 61) is only a creation guard; it does not prevent concurrent removal by `unsubscribe`.
- There is no authentication, rate limiting, or connection throttling on the gRPC endpoint (confirmed by `ConsensusController` and the plaintext `grpcurl` example in the docs).
- There is no grace period or subscriber-count check before evicting the entry.

### Impact Explanation
Every new subscriber to the targeted topic after an attacker-triggered eviction experiences a message gap whose duration equals the Redis re-subscription round-trip. On a high-frequency HCS topic this can mean missed consensus messages. The attacker can sustain the disruption indefinitely by looping subscribe/cancel, effectively preventing any new subscriber from receiving a continuous stream. Existing subscribers (those who subscribed before the eviction) are unaffected because they hold a direct reference to the already-cancelled shared Flux — but they will also stop receiving messages once the upstream Redis subscription is cancelled.

### Likelihood Explanation
The gRPC endpoint is unauthenticated and reachable over plaintext (port 5600 by default). The attack requires only a standard gRPC client (e.g., `grpcurl`), a single RPC call, and an immediate cancel. It is trivially scriptable, requires no special knowledge beyond the topic ID (which is public on-chain), and is repeatable at network speed.

### Recommendation
1. **Add a grace-period / delayed eviction**: Instead of removing the entry immediately on cancel, schedule removal after a short delay (e.g., 5 s) and abort the removal if a new subscriber arrives in the meantime.
2. **Track subscriber count explicitly**: Wrap the shared Flux with an `AtomicInteger` reference counter; only call `unsubscribe` when the count reaches zero *and* remains zero after the grace period.
3. **Keep the Redis subscription alive while the entry is in `topicMessages`**: Move the `doOnCancel`/`doOnComplete` cleanup out of the upstream chain and into a separate lifecycle manager that is not triggered by individual subscriber cancellations.
4. **Rate-limit or authenticate gRPC connections** to raise the cost of the looped subscribe/cancel attack.

### Proof of Concept

```bash
# Requires grpcurl and access to the mirror node gRPC port (default 5600)
# Target topic: 0.0.41110

while true; do
  # Subscribe and immediately cancel (timeout after 100ms)
  timeout 0.1 grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    2>/dev/null || true
  # Each iteration: attacker is last subscriber → cancel → topicMessages evicted
  # Any legitimate subscriber arriving between iterations gets a new Redis sub → gap
done
```

**Observable result:** A legitimate subscriber running concurrently will log repeated "Creating shared subscription to topic.41110" entries (one per attacker iteration) and will miss messages published during each re-subscription window.
