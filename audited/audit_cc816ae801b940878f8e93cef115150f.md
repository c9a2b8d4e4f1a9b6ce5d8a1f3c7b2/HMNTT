### Title
Unauthenticated DB Connection Pool Exhaustion via Opaque EVM Address Flooding in `getEntityIdFromEvmAddress()`

### Summary
The `getEntityIdFromEvmAddress()` function in `rest/service/entityService.js` unconditionally issues a database query for every opaque EVM address lookup, with no result-level caching. An unprivileged attacker can flood the REST API with requests using valid `shard.realm.evmAddress` inputs (where shard/realm match system config and the address is opaque), causing a DB query per request, exhausting the default 10-connection pool and denying service to all legitimate users.

### Finding Description

**Code path:**

In `getEncodedId()` (`rest/service/entityService.js` lines 118–137), when the input is a valid entity ID string, it is parsed and then branched:

```js
return entityId.evmAddress === null
  ? entityId.getEncodedId()
  : await this.getEntityIdFromEvmAddress(entityId, requireResult);
``` [1](#0-0) 

The `evmAddress` field is set to non-null (opaque) when `parseFromString()` in `rest/entityId.js` determines the address is not a long-form account-num alias:

```js
if (prefix !== longFormEvmAddressPrefix || num > maxNum) {
  return [shard, realm, null, numOrEvmAddress]; // Opaque EVM address → DB lookup
}
``` [2](#0-1) 

The shard/realm check at line 246 only rejects mismatched shard/realm — addresses with matching shard/realm pass through: [3](#0-2) 

`getEntityIdFromEvmAddress()` then always fires a DB query:

```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
``` [4](#0-3) 

**Why the cache is insufficient:**

`parseCached()` in `entityId.js` uses a `quickLru` cache keyed on the input string, but it only caches the parsed `EntityId` object (whether the address is opaque or not). It does NOT cache the DB lookup result. [5](#0-4) 

So even for a repeated opaque address: the cache returns the EntityId with `evmAddress !== null`, and `getEntityIdFromEvmAddress()` is still called, hitting the DB every time. For unique addresses, even the parsing cache is bypassed.

**DB pool:**

The default pool is capped at 10 connections: [6](#0-5) 

**No rate limiting on the REST API:**

The REST API Helm chart middleware only configures `circuitBreaker` and `retry` — no per-IP rate limiting or in-flight request cap: [7](#0-6) 

Compare to the Rosetta chart which explicitly adds `inFlightReq` and `rateLimit`: [8](#0-7) 

### Impact Explanation
With a default pool of 10 DB connections and a `connectionTimeout` of 20 seconds, an attacker sending concurrent requests with unique opaque EVM addresses (e.g., `0.0.<random-40-hex-chars-with-non-zero-prefix>`) saturates all connections. Legitimate queries queue and eventually time out. The `circuitBreaker` only triggers on error ratios above 25%, which the attacker can stay below by pacing requests just enough to hold connections open. This constitutes a complete non-network DoS of the REST API service for all users.

### Likelihood Explanation
No authentication is required. The attacker only needs HTTP access to any public endpoint that resolves entity IDs (e.g., `/api/v1/accounts/{id}`, `/api/v1/contracts/{id}`). Generating valid opaque EVM addresses is trivial — any 40-hex-char string where the first 24 chars are not all zeros qualifies. The attack is fully repeatable and scriptable with standard HTTP tooling.

### Recommendation
1. **Add a result-level cache** in `getEntityIdFromEvmAddress()` (e.g., a bounded LRU cache keyed on the hex EVM address) so repeated lookups for the same address do not hit the DB.
2. **Add per-IP rate limiting** to the REST API Traefik middleware (matching the Rosetta configuration pattern with `inFlightReq` and `rateLimit`).
3. **Add an in-flight request cap** per source IP to bound concurrent DB queries from a single client.
4. Consider increasing `maxConnections` or using a query queue with a hard cap on queued EVM-address lookups.

### Proof of Concept

```bash
# System config: shard=0, realm=0 (default)
# Generate opaque EVM addresses: first byte non-zero ensures non-parsable prefix

for i in $(seq 1 200); do
  # deadXX... prefix is non-zero → opaque → forces DB lookup
  ADDR=$(printf "dead%036x" $i)
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.${ADDR}" &
done
wait
# Result: DB connection pool (default 10) saturated; legitimate requests time out
# with "connection timeout" after 20s; service effectively unavailable
```

Each request hits `getEntityIdFromEvmAddress()` → `entityFromEvmAddressQuery` → DB. With 200 concurrent requests and only 10 DB connections, the pool is exhausted and all other API queries are blocked.
