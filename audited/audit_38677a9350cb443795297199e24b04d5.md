### Title
Unguarded `DateTimeParseException` in `GraphQlTimestamp.parseLiteral()` Combined with Absent Rate Limiting Enables Unauthenticated DoS on the GraphQL Endpoint

### Summary
`GraphQlTimestamp.parseLiteral()` calls `Instant.parse()` directly with no try/catch, so any malformed ISO 8601 string causes an unchecked `DateTimeParseException` to propagate through the graphql-java framework. The graphql module has no rate limiting whatsoever (unlike the `web3` module which uses bucket4j). An unauthenticated attacker can flood the endpoint with structurally valid GraphQL queries carrying malformed Timestamp literals, causing each request to fail fast at coercion time with no database cost, sustaining a very high request rate that exhausts the HTTP thread pool and starves legitimate queries.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/scalar/GraphQlTimestamp.java`, lines 33–34:
```java
if (input instanceof StringValue str) {
    return Instant.parse(str.getValue());   // throws DateTimeParseException — unchecked, not CoercingParseLiteralException
}
``` [1](#0-0) 

Same defect in `parseValue()` at line 45:
```java
return Instant.parse(string);  // no try/catch
``` [2](#0-1) 

**Root cause:** The `Coercing` contract requires invalid input to be signalled via `CoercingParseLiteralException`. Instead, the raw `DateTimeParseException` (unchecked) propagates. graphql-java catches it internally and converts it to a GraphQL error, so the server does not crash — but every such request still consumes a thread, allocates a full stack-trace object, and completes without touching the database, meaning the attacker can sustain a far higher request rate than legitimate queries that do hit the DB.

**Why existing checks are insufficient:**

| Protection | Present? | Covers this? |
|---|---|---|
| GraphQL parser limits (10 000 chars, 1 000 tokens) | Yes | No — query structure is valid; only the scalar value is malformed |
| `MaxQueryComplexityInstrumentation(200)` / `MaxQueryDepthInstrumentation(10)` | Yes | No — coercion happens at execution, after complexity checks |
| `CachedPreparsedDocumentProvider` | Yes | No — caches parsed AST; scalar coercion is per-execution |
| bucket4j rate limiting | Yes — **web3 module only** | **No** — graphql module has zero rate limiting |
| `CustomExceptionResolver` | Yes | No — only handles `IllegalStateException`, `IllegalArgumentException`, `MirrorNodeException` | [3](#0-2) [4](#0-3) [5](#0-4) 

### Impact Explanation
An attacker can degrade or deny service to the GraphQL endpoint of the mirror node. Requests with malformed Timestamp values fail before any database I/O, so the attacker's cost per request is minimal while each request still occupies a Servlet/Reactor thread and allocates a stack-trace object. Under sufficient concurrency the HTTP thread pool is exhausted and legitimate queries queue indefinitely or are rejected. Note: the mirror node is a read-only query service; this does not affect Hedera network consensus or transaction confirmation. The impact is availability loss of the mirror node GraphQL API.

### Likelihood Explanation
No authentication is required. The attack requires only an HTTP client and knowledge of any GraphQL field that accepts a `Timestamp` scalar (publicly documented in the schema). The query structure is valid so it bypasses all parser-level defences. The absence of any rate limiting in the graphql module means the attack is trivially repeatable from a single IP or a small botnet. Reproducing the condition requires no special privileges or insider knowledge.

### Recommendation
1. **Wrap `Instant.parse()` in both `parseLiteral()` and `parseValue()`** to catch `DateTimeParseException` and rethrow as the correct `CoercingParseLiteralException` / `CoercingParseValueException`:
   ```java
   try {
       return Instant.parse(str.getValue());
   } catch (DateTimeParseException e) {
       throw new CoercingParseLiteralException("Invalid Timestamp: " + e.getMessage(), e);
   }
   ```
2. **Add rate limiting to the graphql module** using the same bucket4j pattern already present in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`. A `WebGraphQlInterceptor` or a Servlet filter is the appropriate insertion point.
3. Consider adding a `@Pattern` or `@Size` validation directive on Timestamp input fields in the GraphQL schema to reject obviously malformed values before coercion.

### Proof of Concept
```bash
# Send 500 concurrent requests with a malformed Timestamp literal
# (no auth required; query structure is valid, only the scalar value is bad)
seq 500 | xargs -P 500 -I{} curl -s -X POST http://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(id: \"0.0.2\") { timestamp { from } } }","variables":{"ts":"NOT-A-VALID-DATE"}}'
```
Or inline as a literal in a query that accepts a Timestamp filter:
```graphql
{ accounts(filter: { timestamp: { from: "AAAA-BB-CCTHH:MM:SSZ" } }) { id } }
```
Each request throws `DateTimeParseException` at coercion time, returns HTTP 200 with a GraphQL error body, and frees the thread — but at sufficient concurrency the thread pool is saturated and legitimate requests time out.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
