### Title
Unauthenticated HTTP Node-Discovery Allows MITM Injection of Rogue gRPC Endpoints into SDK Client

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` issues a plaintext HTTP GET to `cfg.mirrorRest` with no TLS enforcement, no certificate verification, and no integrity check on the returned JSON. A network-adjacent attacker who can intercept or spoof that response can inject arbitrary gRPC node endpoints, causing the Hiero SDK client to route all subsequent fee-bearing `CryptoTransfer` transactions to attacker-controlled nodes that can impose fees outside design parameters.

### Finding Description
**Code path:**

- `pinger/config.go` line 37: `mirrorRest` is operator-supplied and the documented example is `http://mirror-rest:5551` — no scheme validation enforces HTTPS.
- `pinger/mirror_node_client.go` line 46: `httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}` — a vanilla `http.Client` with no TLS configuration, no certificate pinning, no `http.Transport` with `TLSClientConfig`.
- Lines 79–98: `fetchMirrorNodeNetwork()` issues `http.NewRequestWithContext(ctx, http.MethodGet, url, nil)` over the wire in plaintext, then `json.NewDecoder(resp.Body).Decode(&payload)` — the decoded `nodesEnvelope` is accepted verbatim.
- Lines 100–124: The only validation applied to each returned `nodeEntry` is: non-empty `NodeAccountID` string (line 103), parseable `AccountID` format (line 107), non-empty host string (line 114–116), and non-zero port (line 118). None of these checks prevent an attacker from supplying a syntactically valid but rogue endpoint.
- `pinger/sdk_client.go` lines 18–22: The resulting `map[string]hiero.AccountID` is passed directly to `hiero.ClientForNetwork(netmap)` with no further validation.
- `pinger/transfer.go` lines 29–33: Every periodic tick calls `cryptoTransfer.Execute(client)` against this client, with no explicit `SetMaxTransactionFee()` cap.

**Root cause:** The code assumes the mirror REST endpoint is trusted and reachable only over a secure channel. There is no enforcement of that assumption in code — any `http://` URL is accepted, and the HTTP response carries no signature or other integrity proof.

**Exploit flow:**
1. Attacker gains network-adjacent position (ARP poisoning, DNS spoofing, rogue DHCP, or compromised network device on the same segment as the pinger pod).
2. Pinger starts with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
3. On startup, `buildNetworkFromMirrorNodes()` issues `GET http://mirror-rest:5551/api/v1/network/nodes` in plaintext.
4. Attacker intercepts and returns a crafted HTTP 200 response:
```json
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{"ip_address_v4": "10.0.0.99", "port": 50211}]
  }],
  "links": {"next": null}
}
```
5. `fetchMirrorNodeNetwork()` passes all validation checks (non-empty account ID, valid format, non-empty IP, non-zero port) and returns `{"10.0.0.99:50211": AccountID{0,0,3}}`.
6. `hiero.ClientForNetwork(netmap)` initializes the SDK client pointing exclusively at `10.0.0.99:50211` (attacker's gRPC server).
7. Every subsequent `CryptoTransfer` in `transfer.go` is sent to the attacker's node.

### Impact Explanation
The attacker-controlled gRPC node receives signed `CryptoTransfer` transactions. Because no `SetMaxTransactionFee()` is set in `transfer.go`, the SDK applies its default maximum fee. The rogue node can charge up to that maximum on every transaction tick (default interval: 1 second per `config.go` line 56). Beyond fee manipulation, the attacker can silently drop transactions, return false receipts, or harvest transaction metadata including the operator account ID and destination. This constitutes a direct financial loss to the operator and a breach of transaction integrity.

### Likelihood Explanation
The `network=other` mode is explicitly designed for internal/private deployments (Kubernetes, Docker Compose, on-premise), where the pinger and mirror-rest service share a flat internal network. The documented example URL (`http://mirror-rest:5551`) uses plaintext HTTP. In such environments, ARP poisoning, DNS spoofing, or a compromised sidecar/init-container are realistic and well-documented attack techniques requiring no privileged OS access. The attack is repeatable: the client is built once at startup, so a single successful interception persists for the entire lifetime of the pinger process.

### Recommendation
1. **Enforce HTTPS at configuration load time** (`config.go`): reject any `mirrorRest` URL whose scheme is not `https`.
2. **Pin or verify the TLS certificate**: configure `http.Transport{TLSClientConfig: &tls.Config{...}}` with a known CA or certificate fingerprint.
3. **Validate returned endpoints against an allowlist**: compare returned IP addresses or domain names against a configured set of trusted node addresses before building the network map.
4. **Set an explicit max transaction fee** in `transfer.go` via `SetMaxTransactionFee()` to bound financial exposure even if node routing is compromised.

### Proof of Concept
```bash
# 1. Start a rogue HTTP server on the attacker machine (10.0.0.99)
python3 -c "
import http.server, json
class H(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        body = json.dumps({'nodes':[{'node_account_id':'0.0.3',
            'service_endpoints':[{'ip_address_v4':'10.0.0.99','port':50211}]}],
            'links':{'next':None}}).encode()
        self.send_response(200)
        self.send_header('Content-Type','application/json')
        self.end_headers()
        self.wfile.write(body)
http.server.HTTPServer(('0.0.0.0',5551),H).serve_forever()
