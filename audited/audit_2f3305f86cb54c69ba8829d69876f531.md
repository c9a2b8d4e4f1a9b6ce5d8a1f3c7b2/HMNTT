### Title
Unauthenticated Concurrent `/block` Requests Cause Redundant Unbounded DB Batching via Uncached `FindBetween()`

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` performs no result caching and independently executes a full multi-batch database loop for every invocation. The `/block` endpoint calls this function on every request with no deduplication of concurrent identical requests. The only rate-limiting defense (Traefik middleware) is disabled by default via `global.middleware: false` in the Helm chart, leaving the database fully exposed to concurrent amplification from any unprivileged caller.

### Finding Description
**Code path:**

`rosetta/app/services/block_service.go` line 56 calls `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` on every `/block` request with no caching layer above it. [1](#0-0) 

`FindBetween()` in `rosetta/app/persistence/transaction.go` lines 124–147 executes a `for start <= end` loop, issuing a new `LIMIT 2000` database query per iteration until all transactions in the block's timestamp range are fetched. For a block containing N transactions, this requires `ceil(N/2000)` sequential round-trips to the database per call. [2](#0-1) 

There is no in-process cache, no singleflight/coalescing, and no result memoization anywhere in `FindBetween()` or its callers. The `blockAPIService` only caches account alias lookups (the `entityCache` LRU), not block transaction results. [3](#0-2) 

**Root cause / failed assumption:** The design assumes that callers will not issue concurrent identical requests, or that an upstream layer will enforce concurrency limits. Neither assumption holds.

**Why existing checks fail:**

The Traefik middleware chain (which includes `inFlightReq: amount: 5` per IP and `rateLimit: average: 10`) is gated on `{{ if and .Values.global.middleware .Values.middleware }}`. The default value of `global.middleware` is `false`, so the entire middleware chain — including rate limiting and in-flight request limiting — is **not deployed by default**. [4](#0-3) [5](#0-4) 

Even when the middleware is enabled, the `rateLimit` uses `sourceCriterion: requestHost: true`, which applies a single shared bucket of 10 req/s across **all clients** to the same host — not per source IP. An attacker can consume the entire shared quota. The `inFlightReq` limit of 5 is per IP but only applies when the middleware is active. [6](#0-5) 

### Impact Explanation
An attacker targeting a block with a large transaction count (e.g., 10,000 transactions) forces `ceil(10000/2000) = 5` sequential DB queries per request. Sending 50 concurrent requests for the same block causes 250 independent DB queries simultaneously — all fetching identical data — with no deduplication. Each query involves correlated subqueries against `crypto_transfer` and `staking_reward_transfer` tables joined per `consensus_timestamp`. [7](#0-6) 

This directly increases database CPU, I/O, and connection pool consumption proportionally to the number of concurrent requests, easily exceeding the 30% threshold with modest concurrency. Legitimate node operations (block syncing, network queries) are starved of DB connections.

### Likelihood Explanation
The `/block` endpoint is a standard Rosetta API endpoint, publicly documented, and requires no authentication. The Rosetta protocol is designed for external consumption by exchanges and indexers. Any unprivileged external user can issue POST requests to `/block` with a valid block identifier. The attack requires no credentials, no special knowledge beyond the block index, and is trivially scriptable. Default deployments (with `global.middleware: false`) have zero server-side throttling.

### Recommendation
1. **Application-level caching**: Add a short-lived (e.g., 5–30 second) in-process cache keyed on `(consensusStart, consensusEnd)` in `FindBetween()` or in `blockAPIService.Block()`. Use a singleflight/`sync.Map` pattern to coalesce concurrent identical requests into a single DB execution.
2. **Enable middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`.
3. **Fix rate limit source criterion**: Change `sourceCriterion` from `requestHost: true` to `ipStrategy` (per-source-IP) so the rate limit applies per attacker, not globally.
4. **Add per-IP in-flight limiting at the application layer** as a defense-in-depth measure independent of Traefik.

### Proof of Concept
```bash
# Identify a large block (high transaction count)
BLOCK_INDEX=<large_block_index>

# Send 50 concurrent /block requests for the same block
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/block \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},\"block_identifier\":{\"index\":$BLOCK_INDEX}}" &
done
wait

# Observe: database CPU and active query count spike by N× (N = concurrent requests)
# Each request independently executes ceil(transactions/2000) DB queries
# With 50 concurrent requests and 10,000 transactions: 250 simultaneous DB queries
# Monitor via: SELECT count(*) FROM pg_stat_activity WHERE state = 'active';
```
