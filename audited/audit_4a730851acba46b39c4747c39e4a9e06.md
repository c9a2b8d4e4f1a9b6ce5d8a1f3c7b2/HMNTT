### Title
Unauthenticated Alias-Based DB Connection Pool Exhaustion via `getNftsByAccountId()`

### Summary
Any unauthenticated caller can supply a syntactically valid but non-existent account alias as the path parameter to `GET /api/v1/accounts/:idOrAliasOrEvmAddress/nfts`. Each such request unconditionally executes a full database query against the `entity` table with no result caching and no application-level rate limiting. Flooding this endpoint with random valid-format aliases exhausts the database connection pool, degrading or denying service to all legitimate users.

### Finding Description

**Exact code path:**

`getNftsByAccountId()` at [1](#0-0)  immediately calls `EntityService.getEncodedId()` with the raw path parameter before any other guard.

Inside `getEncodedId()`, the branch at [2](#0-1)  checks `AccountAlias.isValid(entityIdString)` — a pure regex test — and, on success, calls `this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult)` with `requireResult = true` (the default).

`AccountAlias.isValid()` is defined as: [3](#0-2) 

It only tests the string against `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` — a purely syntactic check. Any uppercase base32 string of sufficient length passes.

`getAccountIdFromAlias()` calls `getAccountFromAlias()`, which unconditionally fires a database query: [4](#0-3) 

When the alias is not found, `isEmpty(rows)` is true, `null` is returned, and `getAccountIdFromAlias()` throws `NotFoundError` at: [5](#0-4) 

**No caching for negative results:** The Node.js `EntityService` has no in-process or Redis-backed cache for alias lookups. Every request — hit or miss — goes directly to the database. The Redis `Cache` class in `rest/cache.js` is used for response-level caching, not for alias resolution.

**No rate limiting on the REST API:** The Helm chart middleware for the REST service contains only `circuitBreaker` and `retry` (10 attempts): [6](#0-5) 

Compare this to the Rosetta chart, which explicitly configures both `inFlightReq` and `rateLimit` per-IP. The REST API has neither. The `retry: attempts: 10` setting actually amplifies the attack — each attacker request can trigger up to 10 upstream DB queries.

### Impact Explanation

Each crafted request holds a database connection slot for the duration of the `SELECT` against the `entity` table. Under sustained flood (e.g., hundreds of concurrent requests), the connection pool is exhausted. Legitimate requests — including numeric account ID lookups and EVM address lookups — begin queuing or failing. The circuit breaker fires only after a 25% network error ratio or 25% 5xx ratio is sustained, meaning significant damage occurs before any automatic mitigation activates.

### Likelihood Explanation

No authentication, API key, or account registration is required. The alias format (`[A-Z2-7]+`) is trivially generated — a simple loop producing random 8-character uppercase base32 strings suffices. The attack is stateless, parallelizable across many source IPs, and requires no prior knowledge of the system. The absence of per-IP rate limiting or in-flight request caps makes sustained exploitation straightforward.

### Recommendation

1. **Add negative-result caching** in `getAccountFromAlias()`: cache a sentinel value (e.g., `null` with a short TTL) for aliases that return no rows, so repeated lookups for the same non-existent alias skip the DB.
2. **Add per-IP rate limiting** to the REST Helm chart middleware (matching the Rosetta pattern with `inFlightReq` and `rateLimit`).
3. **Add an in-flight request cap** at the ingress layer for the `/accounts/:id/nfts` route family.
4. **Enforce a minimum alias length** consistent with real Hedera aliases (32+ base32 characters) to reduce the valid input space.

### Proof of Concept

```bash
# Generate and flood with random valid-format aliases (no auth required)
for i in $(seq 1 500); do
  ALIAS=$(cat /dev/urandom | tr -dc 'A-Z2-7' | head -c 32)
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/nfts" &
done
wait
```

Each request passes `AccountAlias.isValid()`, triggers `SELECT id FROM entity WHERE alias = $1`, finds nothing, and throws `NotFoundError` — consuming a DB connection slot for the full query duration. With 500 concurrent requests and a retry factor of 10, up to 5,000 DB queries are in flight simultaneously, exhausting the connection pool and causing 5xx errors for all other API consumers.
