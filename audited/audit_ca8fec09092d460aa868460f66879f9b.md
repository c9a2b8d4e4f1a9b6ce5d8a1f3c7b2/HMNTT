### Title
Unbounded Prometheus Label Cardinality via Arbitrary Unregistered Paths in `toOpenApiPath()`

### Summary
When an HTTP request matches no registered Express route, `req.route` is `null` and `res.locals[requestPathLabel]` is unset, causing `toOpenApiPath()` to use `req.path` verbatim as the `path` label on all four per-route OpenTelemetry instruments. Because there is no cardinality cap, rate limiting, or path normalization for unregistered paths, an unprivileged attacker can create an unbounded number of unique label combinations, exhausting the Node.js heap and degrading service for all users.

### Finding Description
**Exact code path:**

In `rest/middleware/metricsHandler.js`, `toOpenApiPath()` (lines 13–31):

```js
const toOpenApiPath = (req, res) => {
  let path = res.locals[requestPathLabel];   // only set for registered routes

  if (!path) {
    if (!req.route) {
      path = req.path;                       // verbatim attacker-controlled value
    } else {
      path = (req.baseUrl ?? '') + req.route?.path;
    }
  }
  ...
};
``` [1](#0-0) 

`res.locals[requestPathLabel]` is only populated by `recordRequestPath` in `rest/routes/index.js` for routes that actually match a registered handler:

```js
const recordRequestPath = async (req, res) => {
  const path = req.route?.path;
  if (path && !path.startsWith(apiPrefix) && !res.locals[requestPathLabel]) {
    res.locals[requestPathLabel] = `${req.baseUrl}${req.route.path}`.replace(/\/+$/g, '');
  }
};
``` [2](#0-1) 

For any unregistered path, `req.route` is `null` and `res.locals[requestPathLabel]` is never set, so the verbatim `req.path` flows directly into the `labels` object and is recorded on all four instruments:

```js
const path = toOpenApiPath(req, res);
const labels = {method, path, code};

requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
``` [3](#0-2) 

**Root cause:** The failed assumption is that only a bounded set of paths will ever reach the `!req.route` branch. In reality, any HTTP client can request arbitrary paths, and each unique path string becomes a distinct label value stored permanently in the OpenTelemetry SDK's in-memory accumulator.

**Why existing checks fail:** A search for rate limiting (`rateLimit`, `throttle`) in `rest/**/*.js` finds no middleware applied to the metrics instrumentation path. There is no cardinality cap (`cardinalityLimit`, `maxCardinality`) anywhere in the REST middleware stack. The `metricsHandler` is registered before route handlers and fires for every request, including 404s. [4](#0-3) 

### Impact Explanation
Each unique `{method, path, code}` triple creates a new time series entry in the OpenTelemetry SDK's in-memory map for each of the four instruments (`requestTotalCounter`, `durationHistogram`, `requestSizeHistogram`, `responseSizeHistogram`). With N unique paths, 4×N entries accumulate and are never evicted. At scale (e.g., 100 000 unique paths), heap usage grows into the hundreds of megabytes, triggering GC pressure, latency spikes, and ultimately OOM crashes — degrading or denying service for all legitimate users of the mirror node REST API.

### Likelihood Explanation
The attack requires zero authentication, zero privileges, and zero knowledge of the application beyond its public hostname. A single attacker with a modest script (e.g., `for i in $(seq 1 100000); do curl -s http://target/api/v1/junk$i & done`) can trigger the condition. The attack is repeatable, cheap, and leaves no persistent state on the attacker's side. The metrics middleware is enabled by default when `config.metrics.enabled` is true, which is the standard production configuration.

### Recommendation
1. **Normalize unknown paths to a fixed sentinel label** — in the `!req.route` branch, replace `path = req.path` with `path = 'unknown'` (or `'/api/v1/unknown'`). This collapses all unregistered paths into a single label value.
2. **Add a hard cardinality cap** — configure the OpenTelemetry SDK's `ViewOptions` with a `cardinalityLimit` (available in `@opentelemetry/sdk-metrics` ≥ 1.9) so that even if new label values appear, the SDK drops them after the limit is reached rather than growing without bound.
3. **Apply rate limiting** before the metrics middleware for all non-metrics endpoints to reduce the rate at which new label combinations can be injected.

### Proof of Concept
```bash
# Send 50 000 requests to unique unregistered paths (no auth required)
for i in $(seq 1 50000); do
  curl -s "http://<mirror-node-host>/api/v1/nonexistent-path-$i" > /dev/null &
done
wait

# Observe heap growth via the metrics endpoint itself
curl http://<mirror-node-host>/swagger/metrics/ | grep nodejs_process_memory_heap_used_bytes
```

After the loop, `nodejs_process_memory_heap_used_bytes` will show a significant and sustained increase. Repeating the loop with fresh path suffixes will continue to grow heap usage until the process is OOM-killed or becomes unresponsive.
