### Title
Unauthenticated Repeated Calls to `GET /api/v1/network/supply` Trigger Uncached Expensive DB Aggregate Query on Every Request

### Summary
The `getSupply()` method in `NetworkServiceImpl` unconditionally calls `entityRepository.getSupply()` when no timestamp is provided, executing a full-table aggregate SQL query against the `entity` table on every invocation. No caching is applied to this repository method, and the `rest-java` module has no rate limiting or throttling for this endpoint. Any unauthenticated external user can flood this endpoint to repeatedly trigger expensive DB queries, degrading performance for all users.

### Finding Description

**Code path:**

`NetworkController.getSupply()` (line 132–150) accepts an unauthenticated `GET /api/v1/network/supply` request with no `timestamp` parameter. It constructs an empty `Bound` and calls `networkService.getSupply(bound)`. [1](#0-0) 

`NetworkServiceImpl.getSupply()` (line 66–67) checks `timestamp.isEmpty()` and, when true, directly calls `entityRepository.getSupply(lowerBounds, upperBounds)` with no caching layer. [2](#0-1) 

`EntityRepository.getSupply()` is a plain `@Query` native SQL method with **no `@Cacheable` annotation**. It executes a full aggregate query (`SUM(balance)`, `MAX(balance_timestamp)`) over the entire `entity` table joined with an `unnest` range array on every call. [3](#0-2) 

**Why existing checks fail:**

The only servlet filters in `rest-java` are `LoggingFilter` (logs requests) and `MetricsFilter` (records byte metrics). Neither enforces rate limiting. [4](#0-3) [5](#0-4) 

The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to `rest-java` endpoints. [6](#0-5) 

No `@Cacheable` annotation exists anywhere on `EntityRepository.getSupply()` in the `rest-java` module, unlike other repositories (e.g., `grpc`'s `EntityRepository.findById()` which is explicitly cached). [7](#0-6) 

### Impact Explanation
Each call to `GET /api/v1/network/supply` (no timestamp) executes a full aggregate SQL query (`SUM` + `MAX`) over the `entity` table, which can contain millions of rows, joined with an `unnest` range array. Sustained flooding of this endpoint by a single attacker saturates DB connection pool and CPU, causing query latency to spike for all concurrent users of the mirror node. This is a medium-severity griefing issue: no funds are at risk, but service availability and response times for all users degrade proportionally to attack volume.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication, no API key, and no rate limit. An attacker needs only a single HTTP client and a loop. The attack is trivially scriptable (`while true; do curl https://mirror-node/api/v1/network/supply; done`), requires zero privileges, and is repeatable indefinitely. The absence of any throttle in `rest-java` means there is no server-side defense.

### Recommendation
Apply one or more of the following:

1. **Cache the result**: Annotate `EntityRepository.getSupply()` with `@Cacheable` using a short TTL (e.g., 10–30 seconds), since the current supply changes infrequently. This is the highest-impact fix.
2. **Add rate limiting to `rest-java`**: Introduce a `bucket4j`-based rate-limiting filter (similar to `web3`'s `ThrottleConfiguration`) applied globally or specifically to `/api/v1/network/supply`.
3. **Cache at the service layer**: Cache the result in `NetworkServiceImpl.getSupply()` using Spring's `@Cacheable` on the service method itself, keyed on the empty-bound case.

### Proof of Concept
```bash
# No authentication required. Run from any machine with network access.
# Each request triggers a full aggregate SQL query on the entity table.
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/network/supply" -o /dev/null &
done
wait
# Observe: DB CPU spikes, query latency increases for all concurrent users.
```
