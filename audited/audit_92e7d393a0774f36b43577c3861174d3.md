### Title
Unbounded `BlockIdentifier.Index` in `AccountBalance` Causes Uncached DB Query Per Request, Enabling Resource Exhaustion

### Summary
The `AccountBalance()` endpoint accepts a caller-supplied `BlockIdentifier.Index` of any non-negative 64-bit integer with no upper-bound validation and no result caching. Every such value — including arbitrarily large, non-existent indices — is forwarded directly to a live PostgreSQL query. Because the Rosetta Go application contains no built-in rate limiting, an unprivileged attacker can flood the endpoint with distinct large indices, forcing a DB round-trip per request and driving database CPU and connection-pool consumption well above the 30 % threshold.

### Finding Description

**Exact code path:**

1. `rosetta/app/services/account_service.go`, `AccountBalance()`, lines 52–53:
   ```go
   if request.BlockIdentifier != nil {
       block, rErr = a.RetrieveBlock(ctx, request.BlockIdentifier)
   ```
   No validation of `request.BlockIdentifier.Index` before delegation.

2. `rosetta/app/services/base_service.go`, `RetrieveBlock()`, lines 82–83:
   ```go
   } else if bIdentifier.Hash == nil && bIdentifier.Index != nil {
       return b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)
   ```
   Index is passed through without any range check.

3. `rosetta/app/persistence/block.go`, `FindByIndex()`, lines 171–181:
   ```go
   func (br *blockRepository) FindByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
       if index < 0 {
           return nil, hErrors.ErrInvalidArgument
       }
       ...
       return br.findBlockByIndex(ctx, index)
   }
   ```
   Only a negativity check exists; any non-negative value proceeds.

4. `rosetta/app/persistence/block.go`, `findBlockByIndex()`, lines 210–224:
   ```go
   func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
       if index < br.genesisBlock.Index {
           return nil, hErrors.ErrBlockNotFound
       }
       db, cancel := br.dbClient.GetDbWithContext(ctx)
       defer cancel()
       rb := &recordBlock{}
       if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
           return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
       }
       ...
   }
   ```
   The SQL query `selectRecordBlockByIndex` (lines 75–85) is executed unconditionally for every index ≥ genesis index, including all large non-existent values.

**Root cause / failed assumption:** The code assumes that callers will supply plausible block indices. There is no upper-bound guard (e.g., `index > latestKnownIndex → ErrBlockNotFound` without a DB round-trip), no result cache for block lookups, and — critically — no rate limiting inside the Rosetta Go application itself. The middleware directory (`rosetta/app/middleware/`) contains only `health.go`, `metrics.go`, and `trace.go`; none implement throttling. The Traefik-based rate limit (`average: 10`, `inFlightReq: 5`) in `charts/hedera-mirror-rosetta/values.yaml` is an optional Kubernetes deployment artifact, absent in bare or non-Kubernetes deployments.

### Impact Explanation
Each malicious request causes one PostgreSQL query against the `record_file` table (with a correlated subquery in the SELECT list). The database connection pool is capped at 100 open connections (`hiero.mirror.rosetta.db.pool.maxOpenConnections = 100`). An attacker sending requests at a rate that saturates the pool starves legitimate queries across all Rosetta endpoints. Even below pool saturation, sustained high-frequency queries increase DB CPU utilisation. Because each request uses a distinct large index, no query-result cache (none exists for block lookups) can absorb the load. The impact is service degradation or denial of service for all users of the node.

### Likelihood Explanation
The `/account/balance` endpoint is unauthenticated and publicly reachable. The attack requires only an HTTP client capable of sending POST requests with a JSON body — no credentials, no special protocol knowledge. Distinct large indices are trivially generated (e.g., sequential integers starting at `math.MaxInt64 - N`). Without Traefik in front of the service, there is no throttle whatsoever. Even with Traefik, the per-host rate limit of 10 req/s is per-host (not per-IP globally), and an attacker using multiple source IPs or hosts bypasses it entirely. The attack is repeatable and automatable.

### Recommendation
1. **Short-circuit before the DB query**: In `findBlockByIndex`, compare the requested index against the cached latest block index (already available via `RetrieveLatest`) and return `ErrBlockNotFound` immediately if `index > latestIndex`, avoiding any DB round-trip.
2. **Add application-level rate limiting**: Implement a token-bucket or leaky-bucket middleware in the Rosetta Go server (analogous to the `ThrottleManagerImpl` used in the Web3 module) so the limit is enforced regardless of deployment topology.
3. **Cache block-by-index results**: Introduce a short-lived (e.g., 1–2 s) in-process LRU cache for `FindByIndex` results, similar to the entity cache already configured (`hiero.mirror.rosetta.cache.entity.maxSize`), so repeated lookups for the same index do not hit the DB.

### Proof of Concept

```bash
# Requires: curl, bash, a running Rosetta node at $HOST:5700
HOST="localhost"
ACCOUNT="0.0.98"   # any valid account

flood() {
  for i in $(seq 1 500); do
    curl -s -o /dev/null -X POST "http://$HOST:5700/account/balance" \
      -H "Content-Type: application/json" \
      -d "{
        \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"testnet\"},
        \"account_identifier\": {\"address\":\"$ACCOUNT\"},
        \"block_identifier\": {\"index\": $((9000000000000 + i))}
      }" &
  done
  wait
}

# Baseline: record DB CPU/connection count for 60 s

flood
```
