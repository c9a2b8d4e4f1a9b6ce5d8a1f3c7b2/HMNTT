### Title
Unauthenticated GraphQL `account` Query with Zero EVM Address Causes Unbounded Repeated DB Lookups (No Caching, No Rate Limiting)

### Summary
The `account(input: AccountInput!)` GraphQL query accepts `evmAddress: "0x0000000000000000000000000000000000000000"` as valid input per the `@Pattern` directive. Each such request unconditionally issues a live database query (`SELECT * FROM entity WHERE id = 0`) with no result caching and no rate limiting in the GraphQL module, allowing any unauthenticated caller to exhaust the database connection pool through tight-loop repetition.

### Finding Description

**Validation passes for the zero address:**

`evmAddress` is validated by `@Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")` in `account.graphqls` line 96. The zero address `0x0000000000000000000000000000000000000000` is 40 hex characters and satisfies this regex. [1](#0-0) 

**Code path in `EntityServiceImpl.getByEvmAddressAndType`:**

For the zero address, all 20 decoded bytes are `0x00`. The `ByteBuffer` check at lines 37–38 reads the first 4 bytes as `int` (= 0) and the next 8 bytes as `long` (= 0), so the condition is true and the code falls into `entityRepository.findById(buffer.getLong())`. The final `buffer.getLong()` reads the last 8 bytes, also 0, so this becomes `findById(0L)` — a live DB lookup for entity ID 0, which does not exist. [2](#0-1) 

**No caching on the GraphQL `EntityRepository`:**

The GraphQL module's `EntityRepository` has no `@Cacheable` annotations anywhere. Every call to `findById`, `findByAlias`, or `findByEvmAddress` goes directly to the database. [3](#0-2) 

The only cache in the GraphQL module is `CachedPreparsedDocumentProvider`, which caches the parsed query AST (document structure), not query results. It does not prevent the DB from being hit on every execution. [4](#0-3) 

**No rate limiting in the GraphQL module:**

`GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which are structural query shape limits, not request-rate limits. [5](#0-4) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module and is not wired into the `graphql` module at all. [6](#0-5) 

**Why the null-result cache bypass is structural:**

Even if `@Cacheable(unless = "#result == null")` were added (as done in the `web3` `EntityRepository`), the zero address would still bypass it because `findById(0L)` always returns `Optional.empty()` — entity ID 0 never exists. The `unless` condition would prevent caching the empty result, so every request would still hit the DB. [7](#0-6) 

### Impact Explanation

An attacker sending a tight loop of `account(input: { evmAddress: "0x0000000000000000000000000000000000000000" })` queries will generate one `SELECT * FROM entity WHERE id = 0` database query per request, with no application-level throttle or cache to absorb the load. This exhausts the HikariCP connection pool (the `GraphQLHighDBConnections` alert fires at 75% utilization), causing legitimate queries to queue and time out. The result is service degradation or denial of service for all GraphQL users. No economic damage to the Hedera network itself occurs, placing this in the griefing / medium-severity category. [8](#0-7) 

### Likelihood Explanation

The attack requires zero privileges — the GraphQL endpoint at `/graphql/alpha` is publicly accessible with no authentication. The payload is a single, static HTTP POST body that never changes. The `PreparsedDocumentProvider` cache means the attacker pays no parsing overhead after the first request, making the loop maximally efficient. Any attacker with basic HTTP tooling (`curl`, `wrk`, `ab`) can execute this.

### Recommendation

Apply at least two independent mitigations:

1. **Add application-level rate limiting to the GraphQL module.** Port the `bucket4j`-based `ThrottleManagerImpl` pattern from the `web3` module, or add a Spring `HandlerInterceptor` / `WebGraphQlInterceptor` that enforces a per-IP or global requests-per-second cap before query execution.

2. **Cache null/empty results for known-stable non-existent inputs.** Add `@Cacheable` to `EntityRepository.findById` in the GraphQL module with a short TTL (e.g., 30 s) that also caches empty `Optional` results (i.e., remove the `unless = "#result == null"` guard for this specific lookup, or use a sentinel value). This prevents the zero-address and similar always-null inputs from generating repeated DB round-trips.

3. **Reject the zero address explicitly.** Add a custom constraint or pre-check in `getByEvmAddressAndType` that returns `Optional.empty()` immediately (without a DB call) when the decoded address is all zeros, since entity ID 0 is guaranteed not to exist.

### Proof of Concept

```bash
# No authentication required. Run from any machine with network access.
while true; do
  curl -s -X POST http://<graphql-host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: { evmAddress: \"0x0000000000000000000000000000000000000000\" }) { id } }"}' \
    -o /dev/null &
done
```

Each iteration:
1. Passes `@Pattern` validation (40 hex zeros, valid).
2. Enters `getByEvmAddressAndType` → `buffer.getInt()==0 && buffer.getLong()==0` → `findById(0L)`.
3. Issues `SELECT * FROM entity WHERE id = 0` to the database.
4. Returns `Optional.empty()` — not cached.
5. No rate limiter fires.

Observed effect: HikariCP active connection count climbs to pool maximum; subsequent legitimate queries receive connection-timeout errors.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
