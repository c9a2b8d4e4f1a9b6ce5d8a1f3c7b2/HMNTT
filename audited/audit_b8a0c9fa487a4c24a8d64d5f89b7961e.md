### Title
Unauthenticated EVM Address Path Parameter Forces Extra DB Query Per Request, Enabling Connection Pool Exhaustion DoS

### Summary
Any unauthenticated user can supply an opaque EVM address as the `idOrAliasOrEvmAddress` path parameter to `/api/v1/accounts/{evmAddress}/allowances/tokens`, causing `EntityService.getEncodedId()` to unconditionally execute a database lookup via `entityFromEvmAddressQuery` before the main `token_allowance` query. With no rate limiting and no caching of the EVM-address-to-entity-ID DB lookup, sustained concurrent requests double the per-request DB query count, exhausting the shared connection pool and degrading or blocking both the REST API and the underlying PostgreSQL instance shared with the importer.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenAllowanceController.js`, line 69, `getAccountTokenAllowances` unconditionally calls:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
``` [1](#0-0) 

In `rest/service/entityService.js`, `getEncodedId()` (lines 118–137) checks `EntityId.isValidEntityId()` first. A 40-hex-char EVM address passes this check. `EntityId.parseString()` is then called; for an opaque EVM address (non-num-alias), it returns an `EntityId` with `evmAddress !== null`, which triggers the branch:
```js
: await this.getEntityIdFromEvmAddress(entityId, requireResult);
``` [2](#0-1) 

`getEntityIdFromEvmAddress()` (lines 90–104) executes a live DB query every time:
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
``` [3](#0-2) 

The `entityFromEvmAddressQuery` is:
```sql
select id from entity where deleted <> true and evm_address = $1
``` [4](#0-3) 

After this lookup resolves, the main `TokenAllowanceService.getAccountTokenAllowances(query)` executes a second DB query. This results in **2 DB queries per request** when an EVM address is used, vs. 1 for a numeric ID.

**Root cause / failed assumption:** The code assumes EVM address resolution is cheap or infrequent. There is no caching of the DB lookup result. The `quickLru` cache in `entityId.js` (lines 301–304) only caches the *parsed* `EntityId` object (avoiding re-parsing), but the `evmAddress` field is still non-null after cache hit, so `getEntityIdFromEvmAddress` is still called and hits the DB on every request. [5](#0-4) 

**Connection pool:** The REST API uses a single shared `pg` pool with `max: config.db.pool.maxConnections` connections. [6](#0-5) 

There is no rate limiting anywhere in the REST service (confirmed by search returning zero matches for `rateLimit`, `throttle`, `requestLimit`).

### Impact Explanation

Under sustained concurrent EVM-address requests, all pool connections become occupied waiting for `entityFromEvmAddressQuery` and the subsequent allowance query. New requests queue behind `connectionTimeoutMillis`, causing REST API timeouts and effective DoS. More critically, the REST API pool and the importer both connect to the same PostgreSQL instance. PostgreSQL enforces a global `max_connections` ceiling. If the REST API pool is sized large enough (or if many REST instances run), the aggregate open connections can crowd out the importer's connection pool, preventing it from writing new transaction records. Even short of that ceiling, the increased I/O and lock contention on the `entity` table degrades importer write throughput.

### Likelihood Explanation

The attack requires zero authentication, zero privileges, and zero knowledge of valid accounts — any syntactically valid 40-hex-char string (e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`) is accepted as a valid EVM address by `isValidEntityId` and routed to the DB lookup path. The endpoint is public. A single attacker with a modest HTTP load tool (e.g., `wrk`, `ab`, `hey`) can sustain thousands of concurrent requests. The attack is trivially repeatable and requires no special tooling.

### Recommendation

1. **Cache the EVM-address → entity-ID DB lookup**: Add an in-process LRU cache (similar to the existing `entityId.js` parse cache) keyed on the hex EVM address, storing the resolved entity ID with a short TTL. This eliminates the redundant DB round-trip for repeated addresses.
2. **Add rate limiting**: Introduce per-IP or global request rate limiting middleware on the REST API (e.g., using `express-rate-limit`) to bound the maximum concurrent DB-hitting requests.
3. **Reserve importer DB connections**: Configure PostgreSQL `pg_hba.conf` or use a connection pooler (PgBouncer) with separate pools and connection limits for the REST API and importer roles, ensuring the importer always has reserved capacity.
4. **Validate EVM address existence before queuing**: Consider returning a 404 immediately for EVM addresses that resolve to no entity (already done via `NotFoundError`), but ensure this short-circuits before consuming a pool connection for the allowance query.

### Proof of Concept

```bash
# Generate a valid but non-existent opaque EVM address
EVM_ADDR="0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Flood the endpoint with concurrent requests (requires 'hey' or 'wrk')
hey -n 10000 -c 200 \
  "http://<mirror-node-host>/api/v1/accounts/${EVM_ADDR}/allowances/tokens"
```

**Expected result:**
- Each of the 200 concurrent requests acquires a DB connection for `entityFromEvmAddressQuery`, then a second for the allowance query.
- Pool connections are exhausted; subsequent requests receive connection timeout errors.
- DB monitoring shows doubled query rate on the `entity` table.
- If PostgreSQL `max_connections` is reached, the importer's write queries begin failing with "sorry, too many clients already."

**Verification (single request):**
```bash
curl "http://<mirror-node-host>/api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/allowances/tokens"
# Returns 404 (entity not found) — but the DB lookup was still executed
# Enable pg_stat_activity logging to confirm two queries fired per request
```
