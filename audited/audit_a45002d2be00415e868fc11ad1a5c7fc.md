### Title
Token ID Range Inversion Bypass via Dual-Bound Account Parameter Silently Returns Zero Records

### Summary
In `NftAllowanceServiceImpl.getNftAllowances()`, the token ID range validity check is guarded by `!ownerOrSpenderParams.hasLowerAndUpper()`. When a caller supplies both a lower and upper bound on `account.id` (making `hasLowerAndUpper()` return `true`), the guard condition is false and the invalid token range check is entirely skipped. Because `tokenIds` is constructed with `primarySortField=false`, the `Bound` constructor also skips its own range validation. The impossible SQL predicate reaches the database and returns zero records with no error.

### Finding Description

**Root cause — `Bound` constructor skips validation for non-primary fields:** [1](#0-0) 

The range check `adjustedLower > adjustedUpper` only fires when `primarySortField == true`. `tokenIds` is always constructed with `primarySortField=false`, so an inverted range (GTE value > LTE value) passes the constructor without error.

**Root cause — service-level check is gated on the wrong condition:** [2](#0-1) 

The only service-level guard that would catch an inverted token range is wrapped in `!ownerOrSpenderParams.hasLowerAndUpper()`. If the caller supplies both a GTE and LTE on `account.id`, `hasLowerAndUpper()` returns `true`, the negation is `false`, and the block is never entered.

**`verifyRangeId()` does not catch inverted ranges:** [3](#0-2) 

`verifyUnsupported`, `verifySingleOccurrence`, and `verifyEqualOrRange` only check operator type/count, not whether lower > upper.

**Exploit flow:**

1. Caller sends: `account.id=gte:100&account.id=lte:200&token.id=gte:9999&token.id=lte:1`
2. `ownerOrSpenderIds` Bound: lower=GTE:100, upper=LTE:200 → `hasLowerAndUpper()` = `true`
3. `tokenIds` Bound: lower=GTE:9999, upper=LTE:1, `primarySortField=false` → constructor skips range check
4. `checkOwnerSpenderParamValidity`: condition `!ownerOrSpenderParams.hasLowerAndUpper()` = `false` → token range check skipped
5. `verifyRangeId(tokenIds)` passes (no operator-type violations)
6. Repository executes: `WHERE owner = X AND token_id >= 9999 AND token_id <= 1` → 0 rows, no error [4](#0-3) 

### Impact Explanation
Any unprivileged caller can silently suppress all NFT allowance records for a given account by crafting an inverted token ID range paired with a dual-bound account ID range. The API returns HTTP 200 with an empty list instead of a 400 error, making the missing data invisible to consumers. This directly matches the "Incorrect or missing records exported to mirror nodes" classification.

### Likelihood Explanation
No authentication or special privilege is required. The parameters are standard query-string inputs on a public REST endpoint. The condition is trivially reproducible by any user who reads the API documentation and supplies two `account.id` operators alongside an inverted `token.id` range. It is fully repeatable and deterministic.

### Recommendation
Remove the `!ownerOrSpenderParams.hasLowerAndUpper()` guard from the token range check, or move the inversion check into `verifyRangeId()` so it applies unconditionally:

```java
// In verifyRangeId or checkOwnerSpenderParamValidity — always validate token range
if (tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
    throw new IllegalArgumentException("Invalid range provided for token.id");
}
```

Alternatively, pass `primarySortField=true` when constructing the `tokenIds` `Bound` so the constructor itself rejects the inverted range before the service layer is reached.

### Proof of Concept

```
GET /api/v1/accounts/{accountId}/allowances/nfts
    ?account.id=gte:100
    &account.id=lte:200
    &token.id=gte:9999
    &token.id=lte:1
```

**Expected:** HTTP 400 — invalid token.id range  
**Actual:** HTTP 200 — empty list `{ "allowances": [], "links": { "next": null } }`

The inverted token range bypasses all validation and the database predicate `token_id >= 9999 AND token_id <= 1` matches no rows, silently omitting all real allowance records for the queried accounts.
