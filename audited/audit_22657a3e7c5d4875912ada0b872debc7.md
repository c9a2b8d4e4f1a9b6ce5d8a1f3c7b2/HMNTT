### Title
Missing Global Rate Limit on `/graphql/alpha` Enables Circuit-Breaker DoS via Multi-Source Flood

### Summary
The `/graphql/alpha` endpoint exposes the `account(input: AccountInput!)` query with no authentication and no global rate-limit middleware. The Traefik circuit breaker fires at `NetworkErrorRatio() > 0.10`, a very low threshold. An unprivileged attacker using multiple source IPs can bypass the per-IP `inFlightReq` cap, overwhelm the backend until it drops TCP connections, push `NetworkErrorRatio()` above 10%, and open the circuit breaker — denying service to all users.

### Finding Description

**Exact code locations:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 — the sole public query, requiring no authentication: [1](#0-0) 

`charts/hedera-mirror-graphql/values.yaml` lines 135–145 — the full middleware chain for the graphql ingress: [2](#0-1) 

`charts/hedera-mirror-graphql/values.yaml` line 137 — the circuit-breaker expression with the 0.10 `NetworkErrorRatio` threshold: [3](#0-2) 

**Root cause — missing `rateLimit` middleware:**

The graphql middleware chain contains only `circuitBreaker`, `inFlightReq` (per-IP, depth 1), and `retry`. Compare with the rosetta chart, which adds a global `rateLimit: average: 10` entry: [4](#0-3) 

No equivalent `rateLimit` block exists anywhere in the graphql chart or in the application itself — confirmed by zero matches for `rateLimit`/`rate_limit` under `graphql/`.

**Exploit flow:**

1. `inFlightReq` limits each source IP to 5 concurrent requests (`ipStrategy.depth: 1`). An attacker using N distinct IPs (e.g., a small botnet or N×5 concurrent connections from N cloud egress IPs) bypasses this cap entirely.
2. Each `account(input: {entityId: {num: N}})` call hits `AccountController.account()` → `EntityService.getByIdAndType()` → a synchronous JPA/JDBC query against the `entity` table: [5](#0-4) 
3. With the DB connection pool exhausted, the Spring backend begins refusing new TCP connections — these are counted by Traefik as **network errors** (not HTTP 5xx), directly feeding `NetworkErrorRatio()`.
4. The `retry` middleware (`attempts: 3`) re-issues each dropped connection up to 3 additional times, quadrupling the effective load and accelerating pool exhaustion: [6](#0-5) 
5. Once `NetworkErrorRatio()` crosses 0.10 (10 of every 100 requests fail at the network layer), Traefik opens the circuit and returns 503 to **all** clients for the duration of the open state.

**Why application-level guards do not help:**

`GraphQlConfiguration` enforces query complexity ≤ 200, depth ≤ 10, and parser token limits — these reject malformed queries but do not throttle the rate of well-formed, cheap `account` lookups: [7](#0-6) 

There is no Spring-level rate limiter, no Bucket4j throttle (that exists only in `web3`), and no authentication gate on `/graphql/alpha`.

### Impact Explanation

When the circuit opens, Traefik returns 503 to every request routed through the graphql ingress — including legitimate users. The circuit remains open for Traefik's configured recovery window (default 10 s, then half-open probe). A sustained flood keeps it open indefinitely, constituting a complete denial of service for the GraphQL API. Because the circuit breaker is a shared, global resource (not per-IP), a small number of attacking IPs can affect the entire user population.

### Likelihood Explanation

No credentials, API keys, or special network position are required. The attack requires only the ability to open concurrent HTTP connections from multiple IPs — achievable with a handful of cloud VMs or a modest botnet. The threshold of 10% network errors is low enough that even moderate backend saturation (not full exhaustion) can trigger it. The `retry` amplification makes the attack self-reinforcing once the backend begins to slow. This is repeatable and automatable with standard HTTP load tools (e.g., `wrk`, `hey`, `ab`).

### Recommendation

1. **Add a global `rateLimit` middleware** to the graphql middleware chain, mirroring the rosetta configuration:
   ```yaml
   - rateLimit:
       average: 100        # tune to expected legitimate traffic
       burst: 20
       sourceCriterion:
         requestHost: true
   ```
2. **Raise or tune the `NetworkErrorRatio` threshold** — 0.10 is unusually low; rosetta and rest use 0.25.
3. **Add a global `inFlightReq` cap** (without `sourceCriterion`) in addition to the per-IP cap to bound total backend concurrency regardless of source diversity.
4. **Consider application-level throttling** (e.g., Bucket4j, as used in `web3`) keyed on client IP for the GraphQL service.

### Proof of Concept

```bash
# Requires: wrk or parallel curl; N distinct egress IPs (e.g., N cloud VMs)
# Each VM runs:
wrk -t 10 -c 50 -d 60s \
  -s <(echo '
    wrk.method = "POST"
    wrk.headers["Content-Type"] = "application/json"
    wrk.body = "{\"query\":\"{account(input:{entityId:{num:1}}){balance}}\"}"
  ') \
  https://<target>/graphql/alpha

# With N=5 VMs × 50 connections = 250 concurrent requests,
# well above the per-IP inFlightReq cap of 5.
# Monitor Traefik metrics: once NetworkErrorRatio() > 0.10,
# all /graphql/alpha requests return HTTP 503.
```
