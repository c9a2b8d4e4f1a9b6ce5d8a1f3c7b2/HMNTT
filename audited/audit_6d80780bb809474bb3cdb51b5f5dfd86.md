### Title
Global Per-Filter Bucket Monopolization via Single-Token Exhaustion (rate=1 THROTTLE Filter)

### Summary
The `RequestProperties` per-filter `Bucket` is a single shared instance across all callers, created with `capacity(rate)` and `refillGreedy(rate, Duration.ofSeconds(1))`. When an operator configures a `THROTTLE`-action filter with `rate=1`, the bucket holds exactly 1 token. Any unprivileged caller whose request matches the filter consumes that sole token, leaving the bucket empty for every other user until the 1-second refill cycle completes. By sending one matching request per second, an attacker can continuously monopolize the entire filter capacity and deny all other users from passing that filter indefinitely.

### Finding Description
**Code path:**

- `RequestProperties.java` lines 63–68 — bucket is created once (lazy) with `capacity(rate)` and `refillGreedy(rate, Duration.ofSeconds(1))`. With `rate=1` this yields a bucket of exactly 1 token that refills 1 token/second.
- `RequestProperties.test()` lines 46–60 — for `ActionType.THROTTLE` the probabilistic sampling guard (`action != ActionType.THROTTLE && RandomUtils...`) is **skipped**, so every request that matches the filter predicates proceeds to `action()`.
- `ThrottleManagerImpl.throttle()` lines 44–48 — iterates all configured `RequestProperties` and calls `action()` for each matching filter.
- `ThrottleManagerImpl.action()` lines 70–73 — for `THROTTLE`, calls `filter.getBucket().tryConsume(1)`; if the bucket is empty, throws `ThrottleException`.

**Root cause:** The `Bucket` object is a single shared (non-partitioned) instance stored in the `RequestProperties` bean. There is no per-IP, per-session, or per-user partitioning. The failed assumption is that a global token budget of 1/second is sufficient to protect the endpoint; in reality, the first caller each second claims the only available token, leaving zero tokens for every other caller until the next refill.

**Exploit flow:**
1. Attacker identifies (or guesses) a request pattern that matches a configured `THROTTLE` filter with `rate=1`.
2. Attacker sends one matching request → `test()` returns `true` → `action()` calls `tryConsume(1)` → succeeds, bucket now has 0 tokens.
3. Every subsequent request from any user within that 1-second window → `tryConsume(1)` returns `false` → `ThrottleException("Requests per second rate limit exceeded")`.
4. Attacker repeats step 2 at the start of each new second → continuous monopolization.

**Why existing checks are insufficient:**
- The global `rateLimitBucket` (default 500 req/s) only prevents the attacker from being rate-limited at the outer layer; it does not prevent them from consuming the per-filter token.
- The `limit` counter (`counter.getAndIncrement() >= limit`) defaults to `Long.MAX_VALUE`, providing no practical bound.
- There is no IP-based or identity-based partitioning anywhere in the throttle stack.

### Impact Explanation
Any user whose requests match the targeted filter is denied service for the duration of the attack. Because the filter can match on arbitrary fields (`DATA`, `TO`, `FROM`, `GAS`, etc.), an attacker who knows or can infer the filter expression can selectively block an entire class of contract-call requests (e.g., calls to a specific contract address). The impact is a targeted, sustained denial-of-service against all users of that request class, with no collateral effect on unmatched requests.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send HTTP requests to the public JSON-RPC endpoint. The attacker needs to know (or brute-force) a request pattern that matches the filter, which is feasible given that filter fields are standard EVM call parameters. Sustaining the attack costs one HTTP request per second. No special tooling, credentials, or economic stake is required.

### Recommendation
1. **Per-caller partitioning:** Replace the single shared `Bucket` with a `ProxyManager`-backed bucket keyed on caller IP (or API key), so each caller has its own token budget.
2. **Minimum capacity floor:** Enforce `rate >= 2` (or a higher minimum) in the `@Min` validation on `RequestProperties.rate` to prevent single-token buckets that are trivially exhausted.
3. **Separate capacity from refill rate:** Allow operators to configure `capacity` independently from `rate` (e.g., burst capacity > 1 even when sustained rate = 1) so a single request does not exhaust the entire burst allowance.
4. **Audit filter expressions:** Treat filter configurations with `rate=1` as high-risk and document the global-bucket behavior explicitly.

### Proof of Concept
```
# Assume a THROTTLE filter matching requests where `to` equals 0xTargetContract, rate=1

# Step 1 – consume the sole token (attacker's request succeeds)
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xTargetContract","data":"0x","gas":50000}'
# → HTTP 200

# Step 2 – immediately send a second request (same second)
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xTargetContract","data":"0x","gas":50000}'
# → HTTP 429 / ThrottleException: "Requests per second rate limit exceeded"

# Step 3 – legitimate user also denied within the same second
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xTargetContract","data":"0x","gas":50000}'
# → HTTP 429 / ThrottleException (victim)

# Step 4 – attacker loops at 1 req/s to maintain monopoly indefinitely
while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0xTargetContract","data":"0x","gas":50000}'
  sleep 1
done
# All legitimate users matching the filter are denied for the duration of the loop.
```
