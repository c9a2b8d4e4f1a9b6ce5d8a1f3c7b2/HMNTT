### Title
Unauthenticated Alias-Based DB Connection Pool Exhaustion Leading to REST API Denial of Service

### Summary
Any unauthenticated external user can send concurrent `GET /api/v1/accounts/<alias>` requests with valid base32 alias strings, each of which forces `EntityService.getEncodedId()` to issue an uncached DB query against the `entity` table. With the default pool capped at 10 connections and no rate limiting on the REST API, a modest flood of concurrent alias requests exhausts the pool, causing all subsequent REST API requests to queue or time out. The claimed impact of "total network shutdown / inability to confirm transactions" is **incorrect** — the mirror node is a read-only service and does not participate in Hedera consensus — but a full REST API denial of service is real and achievable.

### Finding Description
**Exact code path:**

`rest/accounts.js:399` — `getOneAccount()` calls `EntityService.getEncodedId(req.params[...])` with no authentication or rate-limiting guard before it. [1](#0-0) 

`rest/service/entityService.js:125-126` — `getEncodedId()` branches on `AccountAlias.isValid(entityIdString)` and, when true, calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias])`, issuing a live DB query every time. [2](#0-1) [3](#0-2) 

**Root cause — three compounding failures:**

1. **No result caching for alias lookups.** The `entityId` LRU cache in `rest/entityId.js` caches parsed `EntityId` objects (shard.realm.num), not alias→encoded-ID resolution results. Every alias-based request hits the DB. [4](#0-3) 

2. **Tiny default connection pool.** `hiero.mirror.rest.db.pool.maxConnections` defaults to **10**. [5](#0-4) [6](#0-5) 

3. **No rate limiting on the REST API.** A grep across all `rest/**/*.js` files finds zero rate-limiting middleware. The throttle logic (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` Java module and does not apply here. [7](#0-6) 

**Why existing checks fail:**
- `AccountAlias.isValid()` accepts any string matching `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` — trivially satisfied with random base32 strings. [8](#0-7) 
- The `statementTimeout` (default 20 s) only limits individual query duration, not concurrent connection count.
- pgbouncer's `max_user_connections: 250` for `mirror_rest` is the upstream ceiling; the application-side pool of 10 is the binding constraint. [9](#0-8) 

### Impact Explanation
When the 10-connection pool is saturated, `pool.queryQuietly()` calls queue until `connectionTimeoutMillis` (default 20 s) elapses, at which point they throw. Every REST API endpoint that touches the DB — accounts, transactions, tokens, balances — becomes unavailable. The mirror node REST API is a read-only service; it does **not** confirm Hedera transactions. The correct severity is **REST API denial of service**, not "total network shutdown." Downstream consumers (wallets, explorers, dApps) that depend on the mirror REST API lose service for the duration of the attack.

### Likelihood Explanation
Preconditions: none — the endpoint is publicly accessible with no authentication. A single attacker with a standard HTTP client can open 10+ concurrent connections, each sending a valid alias string (e.g., `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`). Because each request holds a pool connection for the duration of the alias query plus the subsequent account/transaction queries (two `Promise.all` branches), the pool saturates quickly. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Cache alias→ID resolution results** in `EntityService.getAccountIdFromAlias()` using the existing `entityId` cache (already configured with `maxAge: 1800 s`, `maxSize: 100000`).
2. **Add rate limiting** to the REST API (e.g., `express-rate-limit`) scoped at minimum to the `/accounts/:id` endpoint.
3. **Increase the default pool size** or add a per-request concurrency semaphore so alias lookups cannot monopolize all connections.
4. **Validate alias format strictly before querying** — reject syntactically valid but semantically nonsensical aliases (e.g., minimum length) to raise attacker cost.

### Proof of Concept
```bash
# Generate a valid base32 alias string (passes AccountAlias.isValid())
ALIAS="AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"

# Flood with 50 concurrent alias-based requests
for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/${ALIAS}" &
done
wait

# Subsequent legitimate requests will time out or return 500
curl -v "http://<mirror-node-host>:5551/api/v1/accounts/0.0.98"
# Expected: connection timeout or error after pool exhaustion
```
Each concurrent request acquires a DB connection for the alias lookup (`entityFromAliasQuery`) and holds it while the subsequent `Promise.all([entityPromise, transactionsPromise])` runs. With 10 connections and 50 concurrent requests, the pool is exhausted within milliseconds. [10](#0-9)
