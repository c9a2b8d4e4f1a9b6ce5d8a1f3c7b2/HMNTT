### Title
Global Request Counter Exhausted by Non-Matching Requests, Permanently Disabling Filter Rules

### Summary
In `RequestProperties.test()`, the global `counter.getAndIncrement()` is called unconditionally before any filter matching occurs. Any request that reaches `test()` — even one that matches no filter — increments the counter toward `limit`. Once `counter >= limit`, the filter rule permanently returns `false` for all users, disabling the configured action (LOG, REJECT, or THROTTLE) forever with no reset path.

### Finding Description

**Exact code path:**

`RequestProperties.java`, `test()`, line 46: [1](#0-0) 

```java
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0 || counter.getAndIncrement() >= limit) {  // ← counter incremented HERE
        return false;
    }
    // filter matching happens AFTER
    for (var filter : filters) {
        if (filter.test(contractCallRequest)) {
            return true;
        }
    }
    return filters.isEmpty();
}
```

**Root cause:** `counter.getAndIncrement()` executes unconditionally on every call to `test()`, regardless of whether the request matches any `RequestFilter`. The counter is an `AtomicLong` with no reset mechanism anywhere in the class. [2](#0-1) 

**Failed assumption:** The design assumes the counter only advances when a request matches the filter. In reality it advances for every evaluated request, including non-matching ones.

**Caller context:** `ThrottleManagerImpl.throttle()` iterates over all configured `RequestProperties` and calls `requestFilter.test(request)` for every inbound request that passes the global bucket checks. [3](#0-2) 

### Impact Explanation

When an operator configures a finite `limit` on a `RequestProperties` rule (e.g., a REJECT rule blocking a known-malicious `to` address), an attacker can permanently disable that rule by exhausting the counter with requests that do not match the filter. After exhaustion, `counter.getAndIncrement() >= limit` is always true, `test()` always returns `false`, and the action (REJECT/THROTTLE/LOG) never fires again for any user. The `counter` field has no reset path in the codebase. [4](#0-3) 

Severity matches the stated scope: griefing / availability impact on operator-configured filter rules, no direct economic damage to network users.

### Likelihood Explanation

- **Precondition:** Operator must have configured at least one `RequestProperties` entry with a finite `limit` (not the default `Long.MAX_VALUE`). This is the intended use of the `limit` field.
- **Attacker capability:** Zero privileges required — any HTTP client can send `eth_call` / `eth_estimateGas` requests.
- **Rate constraint:** The global `rateLimitBucket` defaults to 500 req/s. [5](#0-4) 
  With `limit = 10 000`, exhaustion takes ~20 seconds of sustained traffic. With `limit = 1 000 000`, it takes ~33 minutes. Both are trivially achievable.
- **Repeatability:** Once exhausted the rule is permanently disabled until the service restarts; the attacker need only do this once.

### Recommendation

Move `counter.getAndIncrement()` to after the filter-matching logic, so the counter only advances when the request actually matches:

```java
@Override
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0) {
        return false;
    }

    if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
        return false;
    }

    boolean matched = filters.isEmpty();
    for (var filter : filters) {
        if (filter.test(contractCallRequest)) {
            matched = true;
            break;
        }
    }

    if (!matched) {
        return false;
    }

    // Only increment after confirmed match
    return counter.getAndIncrement() < limit;
}
```

This ensures non-matching requests do not consume the limit.

### Proof of Concept

**Setup:** Configure one `RequestProperties` rule:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    limit: 1000
    filters:
      - field: TO
        type: EQUALS
        expression: "0xdeadbeef"
```

**Steps:**
1. Send 1000 POST requests to the JSON-RPC endpoint with `to` set to any address **other than** `0xdeadbeef` (e.g., `0x1234`). Each request passes the global rate bucket and reaches `RequestProperties.test()`.
2. Each call executes `counter.getAndIncrement()` at line 46, advancing the counter from 0 to 999.
3. On the 1000th request, `counter.getAndIncrement()` returns 999 (≥ limit 1000 is false yet), but on the 1001st it returns 1000 ≥ 1000 → `test()` returns `false` permanently.
4. Now send a request with `to = 0xdeadbeef`. Expected: REJECT. Actual: `test()` returns `false` immediately at line 46, `action()` is never called, the request is **not rejected**.

The REJECT rule is permanently bypassed for all subsequent requests until service restart.
