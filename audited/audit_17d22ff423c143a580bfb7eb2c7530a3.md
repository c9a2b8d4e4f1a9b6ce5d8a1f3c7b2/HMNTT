### Title
Unbounded `endTime` with No Subscription Limit Enables Reactive Pipeline Resource Exhaustion via gRPC Topic Subscriptions

### Summary
`TopicMessageFilter` enforces no upper bound on `endTime`, and `TopicMessageServiceImpl.subscribeTopic()` enforces no global limit on concurrent subscriptions. An unprivileged attacker can open arbitrarily many gRPC connections (each carrying up to 5 calls per the per-connection cap), each subscription holding a live reactive pipeline open until `endTime + 30 s` elapses, draining the shared `boundedElastic` scheduler, DB connection pool, and heap.

### Finding Description

**Validation gap — no upper bound on `endTime`**

`TopicMessageFilter.isValidEndTime()` only requires `endTime > startTime`: [1](#0-0) 

`isValidStartTime()` only requires `startTime <= DomainUtils.now()`: [2](#0-1) 

There is no constraint preventing `endTime = now + 1 year` (or any arbitrarily large future value).

**`isComplete()` keeps the pipeline alive until `endTime + 30 s`** [3](#0-2) 

The pipeline is only terminated by `pastEndTime()` once `isComplete()` returns `true`, which polls every `endTimeInterval` (30 s): [4](#0-3) 

**Per-subscription resource consumption**

Each `subscribeTopic()` call:
1. Schedules a safety-check `Mono.delay` on `Schedulers.boundedElastic()`: [5](#0-4) 
2. Drives a `PollingTopicListener` that polls the DB every 500 ms on its own `boundedElastic` scheduler: [6](#0-5) 

**No global subscription limit**

`subscriberCount` is a Micrometer gauge only — it is never checked to reject new subscriptions: [7](#0-6) 

**Per-connection cap is trivially bypassed**

`maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection, not globally: [8](#0-7) 

An attacker opens `N` connections × 5 calls each = `5N` concurrent subscriptions with no server-side rejection.

**No gRPC-layer rate limiting exists** — the `ThrottleConfiguration`/`ThrottleManagerImpl` is scoped exclusively to the `web3` module and is never applied to gRPC topic subscriptions.



### Impact Explanation

Each open subscription consumes:
- One or more `boundedElastic` worker slots (safety check + polling scheduler)
- Periodic DB queries every 500 ms per subscription (DB connection pool exhaustion)
- Heap for the reactive pipeline, `TopicContext`, and buffered messages

With `endTime` set far in the future (e.g., `now + 1 hour`), each subscription stays alive for `1 hour + 30 s`. Flooding with thousands of subscriptions exhausts the `boundedElastic` thread pool (default cap: `10 × CPU cores`), starves legitimate subscribers of scheduler threads, and saturates the DB connection pool — resulting in a full denial of service for all gRPC topic consumers.

### Likelihood Explanation

The attack requires only:
- Network access to the gRPC port (publicly exposed in typical deployments)
- Knowledge of any valid `topicId` (publicly enumerable via the REST API)
- A standard gRPC client library

No credentials, tokens, or on-chain funds are needed. The attack is trivially scriptable, repeatable, and can be sustained indefinitely by cycling connections.

### Recommendation

1. **Enforce a maximum `endTime` horizon** in `TopicMessageFilter` — e.g., reject any `endTime > DomainUtils.now() + configurable_max_future_window` (e.g., 5 minutes).
2. **Add a global concurrent-subscription cap** — check `subscriberCount` against a configurable maximum in `subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
3. **Add per-IP or per-client connection rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleConfiguration` in the `web3` module.
4. **Enforce a maximum `endTime` gap from `startTime`** to bound the lifetime of any single subscription pipeline.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_network_service_pb2

# Any valid topic ID discoverable via REST API
TOPIC_ID = {"shardNum": 0, "realmNum": 0, "topicNum": 1234}
# endTime = now + 1 hour (nanoseconds), well beyond now + 30s
import time
END_TIME = int(time.time_ns()) + 3_600_000_000_000

def flood(conn_id):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for _ in range(5):  # maxConcurrentCallsPerConnection = 5
        req = mirror_network_service_pb2.ConsensusTopicQuery(
            topicID=TOPIC_ID,
            consensusStartTime={"seconds": 0},
            consensusEndTime={"seconds": END_TIME // 1_000_000_000,
                              "nanos": END_TIME % 1_000_000_000},
        )
        # Non-blocking: fire and hold open
        threading.Thread(target=lambda: list(stub.subscribeTopic(req))).start()

# Open 1000 connections × 5 subscriptions = 5000 concurrent pipelines
for i in range(1000):
    threading.Thread(target=flood, args=(i,)).start()

# Each pipeline holds a boundedElastic thread + 500ms DB poll for 1 hour
# Server boundedElastic pool (10×cores) exhausted within seconds
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
