### Title
Unbounded gRPC Connection + Redis Subscription Exhaustion via Unauthenticated Multi-Connection Topic Subscription

### Summary
`RedisTopicListener` maintains an unbounded `ConcurrentHashMap<String, Flux<TopicMessage>>` keyed by topic ID, with one Redis subscription per distinct topic. Because `GrpcConfiguration` sets only `maxConcurrentCallsPerConnection=5` with no `maxInboundConnections` limit and no per-IP rate limiting, an unauthenticated attacker can open an arbitrary number of connections, subscribe to many distinct valid topic IDs, and exhaust Redis connections, heap (via per-subscriber `onBackpressureBuffer` allocations), and `boundedElastic` threads — causing stop-the-world GC pressure or OOM that takes down the node.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java` lines 31–34 — only `maxConcurrentCallsPerConnection` is configured; `maxInboundConnections` is never called: [1](#0-0) 

`NettyProperties.java` line 14 — default is 5 calls per connection, no connection count cap: [2](#0-1) 

`RedisTopicListener.java` line 33 — `topicMessages` is an unbounded `ConcurrentHashMap` with no size limit: [3](#0-2) 

`RedisTopicListener.java` line 61 — `computeIfAbsent` creates a new map entry and a new Redis subscription for every distinct topic ID: [4](#0-3) 

`RedisTopicListener.java` lines 68–79 — `subscribe()` opens a `ReactiveRedisMessageListenerContainer` channel per topic, with infinite retry, cached and shared: [5](#0-4) 

`SharedTopicListener.java` line 24 — each individual gRPC subscriber gets its own `onBackpressureBuffer` of up to `maxBufferSize` (default 16 384) messages: [6](#0-5) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection=5` bounds total server load. It does not — it is a per-connection limit. With no `maxInboundConnections`, no per-IP connection throttle, and no authentication, an attacker multiplies load linearly with connection count.

**Exploit flow:**

1. Attacker enumerates valid topic IDs (publicly visible on-chain; `checkTopicExists=true` only rejects non-existent IDs, not rate-limits valid ones).
2. Attacker opens `C` TCP connections to port 5600 (no limit enforced).
3. On each connection, attacker issues 5 `subscribeTopic` calls to 5 distinct valid topic IDs.
4. `getSharedListener` calls `computeIfAbsent` for each new topic key → `subscribe(topic)` is called → a new `ReactiveRedisMessageListenerContainer` channel is opened to Redis.
5. After `C` connections: `5C` active gRPC streams, up to `5C` distinct Redis pub/sub channels, and `5C` independent `onBackpressureBuffer` allocations each holding up to 16 384 `TopicMessage` objects.
6. With active topics, buffers fill: heap consumed ≈ `5C × 16384 × ~200 bytes/msg`. At C=1000 connections this is ~16 GB. Redis connection pool is also exhausted.
7. JVM enters continuous GC; stop-the-world pauses cause the node to stop processing. With no connection limit, the attacker can sustain this indefinitely.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection=5` is per-connection, trivially bypassed by opening more connections.
- `checkTopicExists=true` only validates topic existence — it does not rate-limit or cap subscriptions per IP/client.
- `BufferOverflowStrategy.ERROR` terminates individual subscribers on overflow but the attacker simply reconnects (infinite retry on the client side).
- `unsubscribe` only fires when all `.share()` subscribers disconnect; the attacker keeps connections open.
- No authentication, no IP-based throttle, no total connection cap anywhere in the gRPC stack. [7](#0-6) 

### Impact Explanation
Each mirror node gRPC pod can be driven into heap exhaustion or Redis connection pool exhaustion by a single attacker with enough open TCP connections. Because the service is stateless and horizontally scaled, the attacker must repeat the attack per pod, but since connections are free and unauthenticated this is trivially scriptable. A sustained attack exhausts heap causing repeated full-GC pauses (effectively halting message processing) or OOM kills, meeting the ≥30% node shutdown threshold if the attacker targets enough replicas simultaneously.

### Likelihood Explanation
No special privileges are required — the gRPC port (5600) is publicly exposed. Topic IDs are publicly enumerable from the mirror node REST API. Opening thousands of TCP connections is a standard capability for any attacker with a modest botnet or even a single machine with high file-descriptor limits. The attack is repeatable and self-sustaining because the attacker only needs to keep connections open, not send traffic.

### Recommendation

1. **Set `maxInboundConnections`** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.maxConnectionAge(30, TimeUnit.SECONDS);
   serverBuilder.maxInboundConnections(/* e.g. */ 1000);
   ```
2. **Add per-IP connection rate limiting** via a gRPC `ServerInterceptor` or an upstream load-balancer policy (the GCP `maxRatePerEndpoint=250` in `values.yaml` applies to requests, not persistent streaming connections).
3. **Cap the `topicMessages` map size** (e.g., use a bounded `LinkedHashMap` with LRU eviction or a Caffeine cache with a maximum entry count) to bound Redis subscription count.
4. **Require authentication/authorization** for `subscribeTopic` calls so anonymous bulk subscription is not possible.
5. **Limit subscriptions per authenticated identity or IP** at the service layer in `TopicMessageServiceImpl`.

### Proof of Concept

```python
import grpc
import threading
# proto stubs for com.hedera.mirror.api.proto.ConsensusService

TOPIC_IDS = list(range(1, 1001))  # 1000 valid topic IDs from REST API
connections = []

def open_connection(topic_batch):
    ch = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(ch)
    for tid in topic_batch:
        req = ConsensusTopicQuery(topicID=TopicID(topicNum=tid), consensusStartTime=...)
        # fire and forget — keep iterator alive
        threading.Thread(target=lambda s=stub.subscribeTopic(req): list(s), daemon=True).start()
    connections.append(ch)  # keep connection open

# Open 200 connections × 5 topics each = 1000 Redis subscriptions, 1000 subscriber buffers
for i in range(0, 1000, 5):
    threading.Thread(target=open_connection, args=(TOPIC_IDS[i:i+5],)).start()

# Monitor target JVM: watch for GC pauses, heap near max, Redis connection exhaustion
input("Attack running. Press Enter to stop.")
```

Expected result: target pod's heap climbs to maximum, full-GC pauses appear in logs, Redis connection pool reports exhaustion, gRPC message delivery latency spikes to seconds or the pod OOM-kills.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
