### Title
Pagination-Driven Sustained 3-Subquery UNION ALL Database Load Amplification via Unauthenticated Token Allowance Endpoint

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` generates pagination `next` links that preserve both `spender.id` and `token.id` range bounds. Following these links on every page unconditionally triggers the 3-subquery UNION ALL execution path in `rest/service/tokenAllowanceService.js`. Because the REST API has no per-IP rate limiting, any unauthenticated user can sustain this amplified database load indefinitely by looping over server-provided pagination links.

### Finding Description

**Exact code path:**

In `rest/service/tokenAllowanceService.js` lines 63–75, `getQuery()` builds the SQL:

```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(...));

if (subQueries.length >= 2) {
  sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
}
``` [1](#0-0) 

When all three of `lower`, `inner`, and `upper` are non-empty, this emits three independent `SELECT * FROM token_allowance … LIMIT $2` subqueries joined by `UNION ALL`, plus an outer `ORDER BY … LIMIT $2`. Each subquery independently scans the table up to the page limit.

**How pagination next links lock in the 3-subquery path:**

`getPaginationLink()` in `rest/controllers/baseController.js` lines 195–223 preserves the original `spender.id` lower and upper range bounds in every next link, and appends the cursor as an additional `token.id` bound: [2](#0-1) 

The test fixture confirms this. An initial request:
```
GET /api/v1/accounts/1001/allowances/tokens?spender.id=gte:2001&token.id=gt:10001&spender.id=lte:3102&token.id=lt:10006&limit=2
```
returns next link:
```
?spender.id=lte:3102&spender.id=gte:0.0.2001&token.id=lt:10006&token.id=gt:0.0.10006&limit=2
``` [3](#0-2) 

This next link carries `spender.id=gte:X`, `spender.id=lte:Y`, `token.id=gt:Z`, `token.id=lt:W`. In `extractTokenMultiUnionQuery()` / `getLowerFilters()` / `getInnerFilters()` / `getUpperFilters()`:

- `lower` → `[spender = X, token_id > Z]` (non-empty)
- `inner` → `[spender > X, spender < Y]` (non-empty)
- `upper` → `[spender = Y, token_id < W]` (non-empty) [4](#0-3) 

All three arrays are non-empty on every subsequent page, so every page request unconditionally executes the 3-subquery UNION ALL path.

**Root cause and failed assumption:**

The design assumes that the 3-subquery UNION ALL path is only triggered when the user explicitly supplies a closed spender+token range. The failed assumption is that following a server-generated pagination link is equivalent to a "normal" single-subquery request. In reality, the server's own next-link generation permanently encodes the range bounds, guaranteeing the expensive path on every subsequent page.

**No rate limiting on the REST API:**

`rest/server.js` applies `authHandler`, `requestLogger`, `metricsHandler`, and optional response caching — no per-IP or per-endpoint rate limiting middleware exists for unauthenticated callers. [5](#0-4) 

The only protection is `statementTimeout` in the DB pool config, which limits individual query duration but does not limit request frequency. [6](#0-5) 

### Impact Explanation

Each page request in a paginated traversal issues 3 independent table scans on `token_allowance` (each up to `limit` rows) plus an outer merge sort, compared to 1 scan for a simple query. With `limit=1`, the attacker maximises the number of pages and thus the number of 3-subquery executions per unit time. Across multiple concurrent connections targeting accounts with large allowance sets, this multiplies DB CPU and I/O consumption. Because the REST API is public and unauthenticated, there is no credential barrier. Sustained load exceeding 30% above baseline is achievable without any brute-force pattern — the attacker simply follows server-provided links.

### Likelihood Explanation

No special privileges, credentials, or knowledge of internal state are required. The attacker only needs to:
1. Know any valid account ID (publicly discoverable via the accounts endpoint).
2. Issue one initial request with a wide spender+token range.
3. Mechanically follow the `links.next` URL returned in each response.

This is trivially scriptable with a single `while next_link` loop. The attack is repeatable, parallelisable across multiple accounts, and indistinguishable from legitimate pagination at the HTTP layer.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST API (e.g., `express-rate-limit`) applied before route handlers, with a limit appropriate for unauthenticated callers.
2. **Limit the maximum page size for range queries**: when both primary and secondary range bounds are present (triggering the 3-subquery path), cap `limit` lower than the default maximum to reduce per-request DB cost.
3. **Consider query cost accounting**: track in-flight DB query count per source IP and reject or queue requests that exceed a threshold.
4. **Add a `statementTimeout` guard at the application layer** (not just DB pool) so that slow UNION ALL queries are cancelled before they accumulate.

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }

  getPaginationLink(req, rows, bounds, limit, order) {
    const primaryBound = bounds.primary;
    const secondaryBound = bounds.secondary;
    const isEnd = rows.length < limit;

    if (!bounds.next && (isEnd || (primaryBound.hasEqual() && secondaryBound.hasEqual()))) {
      // fetched all matching rows or the query is for a specific combination
      return null;
    }

    const lastRow = last(rows);
    const lastValues = {};
    if (primaryBound.hasBound() || primaryBound.isEmpty()) {
      // the primary param has bound or no primary param at all
      // primary param should be exclusive when the secondary operator is eq
      lastValues[primaryBound.filterKey] = {
        inclusive: !secondaryBound.hasEqual(),
        primary: true,
        value: !isEnd ? lastRow[primaryBound.viewModelKey] : bounds.next,
      };
    }

    if (secondaryBound.hasBound() || secondaryBound.isEmpty()) {
      // the secondary param has bound or no secondary param at all
      lastValues[secondaryBound.filterKey] = {value: lastRow[secondaryBound.viewModelKey]};
    }

    return utils.getPaginationLink(req, false, lastValues, order);
  }
