### Title
Gas Throttle Bypass via Full Refund on Pre-Check Failures (result == null)

### Summary
In `ContractCallService.restoreGasToBucket()`, when `result == null` (indicating the call failed before reaching the EVM), the full `gasLimitToRestoreBaseline` (defaulting to 100% of the submitted gas limit) is unconditionally restored to the gas bucket. An unprivileged attacker can craft requests that deterministically trigger pre-check failures, receiving a complete gas refund on every call and effectively nullifying the gas-per-second throttle. Only the weaker `requestsPerSecond` bucket (default 500/s) remains as a constraint, allowing sustained high-throughput request flooding that elevates node CPU.

### Finding Description

**Code path:**

`ContractCallService.doProcessCall()` (lines 109–138) initializes `result = null`. It calls `transactionExecutionService.execute()`. Inside `TransactionExecutionService.handleFailedResult()` (lines 132–143), when the transaction record has no `contractCallResult` / `contractCreateResult` (i.e., the call never reached the EVM), a `MirrorEvmTransactionException` is thrown **without** an `EvmTransactionResult` payload:

```java
// TransactionExecutionService.java:140-143
if (result == null) {
    // No result - the call did not reach the EVM and probably failed at pre-checks.
    throw new MirrorEvmTransactionException(status, StringUtils.EMPTY, StringUtils.EMPTY);
}
```

Back in `doProcessCall`, the `MirrorEvmTransactionException` catch block assigns `result = e.getResult()`, which is `null` because no result was attached to the exception. The `finally` block then calls `restoreGasToBucket(null, params.getGas())`.

`restoreGasToBucket()` (lines 140–152):

```java
final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);  // ← always fires when result==null
}
```

With the default `gasLimitRefundPercent = 100`, `gasLimitToRestoreBaseline == gasLimit`. `ThrottleManagerImpl.restore()` (lines 59–64) calls `gasLimitBucket.addTokens(tokens)`, fully replenishing the bucket.

**Root cause:** The design assumes pre-check failures are rare/exceptional and grants them a full gas refund to avoid penalizing legitimate callers for infrastructure errors. This assumption fails when an attacker deliberately engineers pre-check failures.

**Triggerable pre-check conditions (no privilege required):**
- Supplying a `from` address that does not exist in state → `getSenderAccountID()` throws `MirrorEvmTransactionException(PAYER_ACCOUNT_NOT_FOUND)` with no result attached.
- Supplying any address that causes the executor pipeline to fail before writing a `contractCallResult` / `contractCreateResult` to the transaction record (e.g., `INVALID_ACCOUNT_ID`, as confirmed by `testExecuteContractCallFailureOnPreChecks`).

**Exploit flow:**
1. Attacker sends `eth_call` with `gas = 15_000_000` (max) and `from = <non-existent address>`.
2. `throttleManager.throttle()` consumes `15_000_000 / 10_000 = 1_500` gas tokens from `gasLimitBucket`.
3. `transactionExecutionService.execute()` throws `MirrorEvmTransactionException` with `result == null`.
4. `restoreGasToBucket(null, 15_000_000)` restores 1,500 tokens — net gas bucket consumption: **0**.
5. Attacker repeats immediately; the gas throttle never depletes.
6. Only the `rateLimitBucket` (default 500 req/s) limits throughput.

### Impact Explanation
The gas-per-second throttle (default 7.5 B gas/s) is the primary defense against expensive EVM workloads. By keeping `result == null` on every request, an attacker renders it completely inoperative. The attacker sustains 500 req/s of requests that each invoke the full transaction executor pipeline (transaction body construction, pre-check evaluation, account state lookups) without any gas-bucket back-pressure. This sustained load — with no throttle relief — can elevate node CPU by 30%+ compared to the preceding 24-hour baseline, satisfying the threat model's criterion without brute force.

### Likelihood Explanation
The attack requires zero privileges: any caller of the public `/api/v1/contracts/call` endpoint can supply a non-existent `from` address. The condition is 100% reproducible and deterministic. A single attacker with a scripted HTTP client can sustain the maximum rate indefinitely. No special knowledge of the contract state is needed.

### Recommendation
1. **Do not restore gas for pre-check failures.** When `result == null`, the call failed before any EVM work was done, so no gas was "unused" — the correct refund is 0, not `gasLimitToRestoreBaseline`. Change the condition:
   ```java
   if (result != null && !result.isSuccessful() && gasLimit == result.gasUsed()) {
       throttleManager.restore(gasLimitToRestoreBaseline);
   } else if (result != null) {
       final var gasRemaining = gasLimit - result.gasUsed();
       throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
   }
   // result == null → no restore; gas is consumed as a penalty for the failed pre-check
   ```
2. **Apply a separate, lower per-IP or per-sender rate limit** at the API gateway layer to limit the blast radius of the `requestsPerSecond` bucket.
3. **Consider reducing `gasLimitRefundPercent`** from 100 to a lower value (e.g., 25–50) to reduce the refund magnitude even for legitimate failure cases.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
