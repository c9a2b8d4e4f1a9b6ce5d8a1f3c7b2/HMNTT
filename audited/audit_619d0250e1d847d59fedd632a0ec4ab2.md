### Title
Mirror Node REST Response Poisoning Causes Permanent DoS of HBAR Transfers via Unvalidated Dead Endpoints

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` accepts any syntactically valid endpoint (non-empty host, non-zero port) from the mirror node REST response without validating reachability or address legitimacy. Because `newClient()` in `sdk_client.go` builds the Hiero SDK client exactly once from this response and never refreshes it, an attacker who can control the mirror node REST response can permanently poison the network map with unreachable addresses, causing every `Execute(client)` call in `transfer.go` to time out for the lifetime of the process.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–128:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← ONLY syntactic check
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
// ...
if len(network) == 0 {               // ← passes if ANY syntactically valid entry exists
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
return network, false, nil
``` [1](#0-0) 

The only guards are:
- `host == ""` — a string like `"0.0.0.0"` passes.
- `ep.Port == 0` — port `50211` passes.
- `len(network) == 0` — passes as long as at least one syntactically valid entry exists.

There is **no reachability probe, no IP-range exclusion, no TLS/authentication on the HTTP fetch, and no cross-validation against known-good addresses**.

The poisoned map is then passed directly to `hiero.ClientForNetwork(netmap)`:

```go
// sdk_client.go line 18-22
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil {
    return nil, err
}
client = hiero.ClientForNetwork(netmap)
``` [2](#0-1) 

This client is created **once** in `main.go` and reused for every ticker tick:

```go
client, err := newClient(cfg)   // line 41 — one-time init, never refreshed
``` [3](#0-2) 

Every subsequent transfer attempt calls `cryptoTransfer.Execute(client)` against the poisoned client:

```go
resp, err := cryptoTransfer.Execute(client)
``` [4](#0-3) 

**Root cause / failed assumption:** The code assumes the mirror node REST API is a trusted source of valid, reachable gRPC endpoints. No defense-in-depth exists at the parsing layer to reject bogus addresses.

### Impact Explanation

All HBAR transfers from the operator account via the pinger service fail permanently (until the process is restarted and the mirror node returns valid data). The operator's funds are not frozen on-chain, but the pinger — which is the sole mechanism for automated HBAR transfers in this service — is rendered completely non-functional. In a monitoring/liveness context this also means the heartbeat file `/tmp/alive` continues to be written (the goroutine is independent), so the liveness probe does not detect the failure, masking the outage from Kubernetes. [5](#0-4) 

### Likelihood Explanation

**Precondition:** The attacker must be able to influence the HTTP response from the mirror node REST endpoint. Realistic vectors:

1. **Plain HTTP (default example is `http://mirror-rest:5551`)** — no TLS, so any on-path attacker (ARP spoofing, rogue container on the same Docker/K8s network, compromised sidecar) can inject a crafted response.
2. **DNS poisoning** — redirect the mirror node hostname to an attacker-controlled server.
3. **Mirror node compromise** — if the mirror node itself is compromised, the attacker controls the REST response directly. [6](#0-5) 

In containerized deployments with shared overlay networks (common in Kubernetes), vector 1 is achievable by any workload on the same node or namespace without elevated privileges. The attack is repeatable: every time the pinger restarts (e.g., after a crash or rolling update), it re-fetches the mirror node response and can be re-poisoned.

### Recommendation

1. **Validate endpoint addresses before accepting them:** Reject loopback (`127.0.0.0/8`), unspecified (`0.0.0.0`), link-local, and RFC-1918 addresses unless explicitly configured for private networks.
2. **Probe reachability before building the client:** Attempt a TCP dial (with a short timeout) to each endpoint; exclude endpoints that fail.
3. **Enforce TLS on the mirror node REST connection** and verify the server certificate against a pinned CA or the system trust store.
4. **Periodically refresh the network map** rather than building the client once at startup, so a poisoned response does not cause permanent failure.
5. **Require a minimum number of distinct reachable nodes** (e.g., ≥ 3) before accepting the network map, making it harder to poison all entries.

### Proof of Concept

**Preconditions:** Attacker can intercept or respond to the HTTP GET to `http://<mirror-rest>/api/v1/network/nodes` (e.g., via ARP spoofing on the same Docker network, or by running a rogue HTTP server at the configured `HIERO_MIRROR_PINGER_REST` address).

**Steps:**

1. Stand up a malicious HTTP server that responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "0.0.0.0", "port": 50211 }
      ]
    },
    {
      "node_account_id": "0.0.4",
      "service_endpoints": [
        { "ip_address_v4": "0.0.0.0", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

2. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-server>`.

3. Start the pinger. `fetchMirrorNodeNetwork()` returns `{"0.0.0.0:50211": AccountID{0,0,3}, ...}` — `len(network) == 2 > 0`, so no error is raised.

4. `hiero.ClientForNetwork(netmap)` is called with the poisoned map. The client is set as the operator client.

5. Every `cryptoTransfer.Execute(client)` call attempts to connect to `0.0.0.0:50211`, times out, and logs a failure. All retries exhaust. No HBAR is ever transferred.

6. The process continues running (heartbeat goroutine is unaffected), so the liveness probe reports healthy while all transfers silently fail.
