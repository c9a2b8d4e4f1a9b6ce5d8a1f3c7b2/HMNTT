### Title
Missing `deleted` Filter in `getByIdAndType()` Allows Retrieval of Deleted Entities via Numeric EntityId

### Summary
`EntityServiceImpl.getByIdAndType()` uses Spring Data JPA's unfiltered `CrudRepository.findById()`, which has no `deleted is not true` predicate. In contrast, both `findByAlias` and `findByEvmAddress` in the same `EntityRepository` explicitly exclude deleted entities. Any unprivileged user can query the public GraphQL endpoint with a numeric `entityId` and receive a full entity record for a network-deleted entity, while the same entity is correctly suppressed under alias or EVM address lookups.

### Finding Description
**Exact code path:**

`AccountController.account()` dispatches to `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` when the caller supplies an `entityId` input. [1](#0-0) 

`EntityServiceImpl.getByIdAndType()` resolves the entity with:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
``` [2](#0-1) 

`findById()` is the inherited `CrudRepository` method — it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

The two custom query methods in the same repository both carry the guard:
```sql
select * from entity where alias = ?1 and deleted is not true
select * from entity where evm_address = ?1 and deleted is not true
``` [3](#0-2) 

**Root cause:** The developer added `deleted is not true` to the two custom query methods but relied on the inherited `findById()` for the ID path, omitting the same guard. No downstream check in `AccountController` or the mapper filters on `entity.getDeleted()` before returning the result.

**Why existing checks fail:** The only filter applied after `findById()` is a type check (`.filter(e -> e.getType() == type)`), which does not inspect the `deleted` flag. The test suite has no test case for `getByIdAndType` with a deleted entity, confirming the gap. [4](#0-3) 

### Impact Explanation
The GraphQL mirror node API is a public, unauthenticated read endpoint. A deleted Hedera entity (account, contract, token, etc.) is supposed to be absent from active-entity queries; the design intent is explicit in the `findByAlias`/`findByEvmAddress` filters. Returning a deleted entity via the ID path means:

- Downstream applications that query by numeric ID and check only for a non-null response (rather than inspecting `deleted: true`) will incorrectly treat the entity as live.
- The inconsistency between lookup methods creates a split-brain view: the same deleted entity is invisible by alias/EVM address but visible by ID, violating the mirror node's correctness guarantee of accurately reflecting consensus-node state.
- Severity: **Medium** — no private data is leaked (all fields are public blockchain data), but protocol-level data integrity is violated and dependent integrations can be misled.

### Likelihood Explanation
Exploitation requires zero privileges and zero special knowledge beyond knowing (or enumerating) a numeric entity ID, which is sequential and publicly observable on-chain. The GraphQL endpoint is publicly accessible. Any developer or automated tool querying `account(input: {entityId: {num: X}})` for a deleted account will trigger the behavior deterministically and repeatably.

### Recommendation
Replace the bare `findById()` call in `getByIdAndType()` with a custom repository method that includes the `deleted is not true` predicate, consistent with the other two lookup methods:

```java
// In EntityRepository:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);

// In EntityServiceImpl.getByIdAndType():
return entityRepository.findByIdAndNotDeleted(entityId.getId()).filter(e -> e.getType() == type);
```

Add a corresponding integration test that persists an entity with `deleted(true)` and asserts that `getByIdAndType` returns empty, mirroring the existing tests for `findByAlias` and `findByEvmAddress`. [2](#0-1) 

### Proof of Concept
**Preconditions:**
1. A Hedera entity (e.g., account `0.0.12345`) exists in the mirror node database with `deleted = true`.
2. The GraphQL endpoint is reachable at `/graphql/alpha`.

**Trigger:**
```graphql
query {
  account(input: { entityId: { shard: 0, realm: 0, num: 12345 } }) {
    entityId { shard realm num }
    deleted
    balance
    key
  }
}
```

**Result:** The API returns the full account record with `"deleted": true` — the entity is exposed. Repeating the same lookup via alias or EVM address returns `null`, demonstrating the inconsistency.

**Contrast (alias path — correctly suppressed):**
```graphql
query {
  account(input: { alias: "<base32-alias-of-same-entity>" }) {
    entityId { shard realm num }
  }
}
```
Returns `null` for the same deleted entity, confirming the asymmetry.
