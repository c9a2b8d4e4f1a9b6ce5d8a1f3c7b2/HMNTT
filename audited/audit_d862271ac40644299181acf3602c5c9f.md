### Title
Unauthenticated DB Exhaustion via Uncached `FindByIndex()` in `AccountBalance()` with No Default Rate Limiting

### Summary
Any unauthenticated external user can POST to `/account/balance` with a `BlockIdentifier.Index` set to the current latest block index, forcing `RetrieveBlock()` to call `FindByIndex()` on every request. `FindByIndex()` executes a correlated SQL subquery against the database with zero caching. The Traefik rate-limiting middleware is defined in the Helm chart but is **disabled by default** (`global.middleware: false`), leaving the application fully exposed to DB exhaustion from concurrent unauthenticated requests.

### Finding Description

**Code path:**

`AccountBalance()` (`account_service.go:52-56`) checks whether `request.BlockIdentifier != nil`. If so, it calls `a.RetrieveBlock(ctx, request.BlockIdentifier)`. [1](#0-0) 

`RetrieveBlock()` (`base_service.go:82-83`) routes to `b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)` when only `Index` is set (no `Hash`). [2](#0-1) 

`FindByIndex()` (`block.go:171-181`) calls `findBlockByIndex()` unconditionally. [3](#0-2) 

`findBlockByIndex()` (`block.go:210-224`) executes a raw SQL query with a **correlated subquery** (`selectRecordBlockByIndex`) on every invocation — there is no in-process cache, no result memoization, and no deduplication of concurrent identical requests. [4](#0-3) 

The SQL itself (`block.go:75-85`) includes a correlated subquery to compute `consensus_end`, making it more expensive than the simple `ORDER BY index DESC LIMIT 1` used by `RetrieveLatest()`. [5](#0-4) 

**Root cause:** The `blockRepository` struct holds no cache field. Every call to `FindByIndex()` unconditionally opens a DB connection and executes the correlated query. The failed assumption is that infrastructure-level rate limiting will always be present; it is not — `global.middleware: false` is the default. [6](#0-5) 

**Why existing checks fail:**

The only middleware applied in `main.go` is `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which throttle requests. [7](#0-6) 

The Traefik middleware chain (`inFlightReq: amount: 5` per IP, `rateLimit: average: 10` per host) is defined but gated behind `global.middleware` which defaults to `false`, meaning it is **not deployed** in a standard installation. [8](#0-7) 

### Impact Explanation
All rosetta service replicas share a single PostgreSQL database. Flooding `/account/balance` with concurrent requests each carrying a valid `BlockIdentifier.Index` saturates the DB connection pool and query executor. Because the latest block always exists (no early-exit for "not found"), every request completes the full correlated subquery plus the subsequent `RetrieveBalanceAtBlock()` query. This can render the rosetta service and its backing DB unavailable across all replicas simultaneously, meeting the ≥30% node-processing disruption threshold for mirror node operators.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero prior knowledge beyond the current block index (trivially obtained from `/network/status`). It is fully scriptable with standard HTTP tooling (`curl`, `wrk`, `hey`). The default deployment disables the only protective middleware. Repeatability is unlimited — the attacker simply keeps sending POST requests.

### Recommendation
1. **Enable rate limiting by default**: Change `global.middleware` default to `true` in `values.yaml`, or apply `inFlightReq` and `rateLimit` unconditionally at the ingress level.
2. **Add application-level caching**: Cache `FindByIndex()` results in `blockRepository` using a short-lived TTL (e.g., 1–2 seconds) keyed on block index, similar to how `initGenesisRecordFile` uses `sync.Once` for the genesis block.
3. **Add DB connection pool limits**: Configure GORM/pgx pool `MaxOpenConns` and `MaxIdleConns` to bound the blast radius.
4. **Consider request coalescing**: Use `singleflight` to deduplicate concurrent identical `FindByIndex()` calls for the same index.

### Proof of Concept

**Step 1** — Discover the latest block index:
```bash
curl -s -X POST http://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq '.current_block_identifier.index'
# Returns e.g. 72345678
```

**Step 2** — Flood `/account/balance` with that index (no credentials needed):
```bash
hey -n 10000 -c 200 -m POST \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier":{"blockchain":"Hedera","network":"mainnet"},
    "account_identifier":{"address":"0.0.98"},
    "block_identifier":{"index":72345678}
  }' \
  http://<rosetta-host>/account/balance
```

**Expected result:** DB connection pool exhaustion; rosetta pods begin returning 500 errors or timing out; liveness/readiness probes fail; pods restart or become unavailable across the deployment.

```go
func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
	if index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	return rb.ToBlock(br.genesisBlock), nil
}
```
