### Title
Concurrent `findStorageBatch()` Invocations Cause Quadratic DB Query Amplification via Unsynchronized Shared Slot Key Cache

### Summary
`ContractStateServiceImpl.findStorageBatch()` adds the requesting slot key to a shared per-contract Caffeine cache and then immediately reads the entire key set to issue a batch DB query. With no synchronization between key insertion and key-set snapshot, N concurrent callers for the same contract each observe a growing key set and issue overlapping batch queries, producing O(N²) total DB slot lookups. Any unprivileged user can trigger this via concurrent `eth_call` or `eth_getStorageAt` requests.

### Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`
**Function:** `findStorageBatch()`, lines 85–122

**Code path:**
- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — atomically inserts the current key into the shared per-contract cache.
- Line 91: `contractSlotsCache.getNativeCache().asMap().keySet()` — reads the live key set, which may already contain keys inserted by other concurrent threads.
- Lines 93–101: Iterates over all cached keys and builds `cachedSlots`.
- Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a DB query for all accumulated keys.

**Root cause:** There is no lock or atomic compare-and-query operation spanning steps 1–3. The Caffeine cache's `asMap()` is a live, concurrent view. Thread T1 inserts key1 and reads {key1}; Thread T2 inserts key2 and reads {key1, key2}; Thread T3 inserts key3 and reads {key1, key2, key3}. Each thread issues a DB query for the full accumulated set. With N threads, total DB slot lookups = 1+2+…+N = N(N+1)/2.

**Why existing checks fail:**
- `putIfAbsent` (line 90) is atomic but only prevents duplicate key storage — it does not prevent concurrent threads from reading each other's newly inserted keys before their own query completes.
- The `isKeyEvictedFromCache` check (lines 94–100) only handles the case where the requesting key was evicted; it does not prevent over-querying other keys.
- The `contractStateCache` hit check (lines 63–67) returns early only if the value is already cached — for new/unseen keys it always falls through to `findStorageBatch()`.
- No rate limiting or concurrency control exists in the service layer. [4](#0-3) 

### Impact Explanation
With 20 concurrent requests for different slot keys of the same contract, the DB receives 20 queries with sizes 1, 2, …, 20 = 210 total slot lookups instead of 20 — a 10.5× amplification. At 50 concurrent requests: 1,275 lookups vs. 50 expected (25.5×). This directly translates to increased DB CPU, I/O, and connection pool pressure. The `contract_state` table is queried with `slot IN (...)` clauses that grow with each concurrent request, and the query is not cached at the repository level for the batch variant (only `findStorage` single-key has `@Cacheable`). [5](#0-4) 

### Likelihood Explanation
The web3 JSON-RPC endpoints (`eth_call`, `eth_getStorageAt`, `eth_estimateGas`) are publicly accessible with no authentication. An attacker needs only to send concurrent HTTP requests targeting a single contract address with different storage slot indices. This requires no special knowledge, no credentials, and no on-chain interaction. Standard HTTP load tools (curl, wrk, ab) suffice. The attack is repeatable and can be sustained indefinitely. The per-contract Caffeine cache size bounds the maximum batch size per query, but does not prevent the amplification pattern from repeating as keys cycle through the cache.

### Recommendation
1. **Snapshot the key set before inserting the new key**, so each invocation only queries keys that were already accumulated before this request arrived — preventing the current request from inflating the batch for itself.
2. **Alternatively, serialize batch queries per contract** using a per-contract lock or `computeIfAbsent`-style atomic batch coalescing (e.g., a `ConcurrentHashMap<EntityId, CompletableFuture>` that lets concurrent callers join an in-flight batch rather than issuing their own).
3. **Cap the batch size** explicitly before issuing the DB query, regardless of how many keys are in the cache.
4. **Add rate limiting** at the API gateway or service layer per source IP or per contract ID.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
