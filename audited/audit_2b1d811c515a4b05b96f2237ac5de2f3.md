### Title
Unfiltered HTTP Method Label in Metrics Middleware Allows Unprivileged Metrics Pollution on Transaction History Routes

### Summary
The `metricsMiddleware()` function in `rest/middleware/metricsHandler.js` records `req.method` directly into Prometheus labels without any allowlisting or normalization. Because the middleware is registered globally before route handlers, any unprivileged user can send arbitrary HTTP methods (POST, DELETE, PATCH, or even custom tokens like `FOOBAR`) to read-only transaction history endpoints, causing those methods to be permanently recorded as label values in `requestTotalCounter` and `durationHistogram`. This pollutes observability data, can trigger false security alerts, and can cause unbounded Prometheus label cardinality.

### Finding Description
**Code location:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware()`, lines 185–211.

**Root cause:** At line 191, `const method = req.method;` captures the raw HTTP method from the request object with no validation, normalization, or allowlisting. This value is immediately used as a Prometheus label at lines 207–210:

```js
const method = req.method;          // line 191 — no filtering
const labels = {method, path, code};
requestTotalCounter.add(1, labels); // line 207
durationHistogram.record(duration, labels); // line 208
```

**Middleware ordering:** In `rest/server.js` lines 89–92, `metricsHandler()` is registered via `app.useExt()` — a global middleware that runs for every request before any route handler. Transaction routes are registered later (lines 132–133) using `app.getExt()`, which only matches GET. A non-GET request to `/api/v1/transactions` will receive a 404 from Express, but the `res.on('finish')` callback in `metricsMiddleware` fires regardless, recording whatever method was sent.

**Failed assumption:** The middleware assumes only well-known HTTP methods (GET, POST, etc.) will appear as label values. Express does not restrict `req.method` to a known set — it passes through any token from the HTTP request line verbatim (uppercased). There is no method allowlist anywhere in the middleware chain prior to metric recording.

**Exploit flow:**
1. Attacker sends `POST /api/v1/transactions HTTP/1.1` (no auth required).
2. `metricsMiddleware` runs, registers `res.on('finish')`.
3. Express finds no POST route, returns 404.
4. `res.on('finish')` fires; `{method:"POST", path:"/api/v1/transactions", code:"404"}` is recorded in `requestTotalCounter` and `durationHistogram`.
5. Repeat with DELETE, PATCH, OPTIONS, HEAD, or arbitrary tokens like `FOOBAR`.

### Impact Explanation
- **Metrics integrity:** Dashboards and alerts built on `api_request_total{method=~"GET|POST"}` will show unexpected method distributions for transaction history routes, misleading operators.
- **False security alerts:** SIEM/alerting rules that fire on non-GET methods against read-only endpoints will generate noise or be desensitized.
- **Prometheus cardinality attack:** Because Express passes through arbitrary HTTP method tokens, an attacker can generate unbounded unique `{method, path, code}` label combinations. Prometheus stores one time series per unique label set; high cardinality degrades query performance and memory usage.
- No authentication or special privilege is required.

### Likelihood Explanation
Trivially exploitable: any HTTP client can send arbitrary methods. No credentials, tokens, or network position are required beyond basic connectivity to the REST API port. The attack is repeatable at high frequency with a simple loop. Standard HTTP fuzzing tools (e.g., `ffuf`, `curl`) perform this automatically.

### Recommendation
Add an HTTP method allowlist in `metricsMiddleware` before recording labels. Normalize or replace unknown methods with a sentinel value:

```js
const ALLOWED_METHODS = new Set(['GET','POST','PUT','PATCH','DELETE','HEAD','OPTIONS']);
const method = ALLOWED_METHODS.has(req.method) ? req.method : 'UNKNOWN';
const labels = {method, path, code};
```

This caps label cardinality to a fixed set and prevents metrics pollution from arbitrary method strings.

### Proof of Concept
```bash
# Record baseline GET metric
curl -s http://<host>/api/v1/transactions > /dev/null

# Inject unusual methods into metrics
for METHOD in POST DELETE PATCH PUT OPTIONS FOOBAR INJECT; do
  curl -s -X $METHOD http://<host>/api/v1/transactions > /dev/null
done

# Observe polluted metrics — non-GET methods now appear for transaction routes
curl -s http://<host>/swagger/metrics/ | grep 'api_request_total.*transactions'
# Expected output includes lines like:
# api_request_total{method="POST",path="/api/v1/transactions",code="404"} 1
# api_request_total{method="DELETE",path="/api/v1/transactions",code="404"} 1
