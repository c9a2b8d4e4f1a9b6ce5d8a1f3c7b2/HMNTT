### Title
Unbounded O(N×M) Heap Exhaustion via SLOAD-Heavy EVM Replay with `storage=true` in `getContractOpcodes()`

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint, accessible to any unauthenticated user, replays a historical transaction through the EVM. When `storage=true` is passed, `AbstractOpcodeTracer.captureStorage()` is invoked on **every single opcode execution** and re-serializes the entire cumulative set of storage accesses into a new `TreeMap<String, String>` that is stored in an unbounded `ArrayList`. For a transaction containing a SLOAD-heavy loop of N storage reads across M total opcodes, this produces O(N×M) string objects in JVM heap with no cap, no timeout, and no per-request resource limit.

### Finding Description

**Entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

No authentication is required. The only guards are:
1. A gzip `Accept-Encoding` header check (trivially satisfied by any HTTP client).
2. `throttleManager.throttleOpcodeRequest()` — rate-limits the number of requests, but places **no bound on per-request resource consumption**.

**Per-opcode storage capture** — `OpcodeActionTracer.tracePostExecution()` calls `captureStorage()` after **every** opcode: [2](#0-1) 

**Root cause** — `AbstractOpcodeTracer.captureStorage()` fetches the full cumulative list of all storage accesses accumulated so far in the transaction and builds a brand-new `TreeMap` from it on every call: [3](#0-2) 

Each resulting `Opcode` object (containing this map) is appended to an unbounded `ArrayList`: [4](#0-3) 

The initial capacity hint is `gas / 3`, but the list has no maximum size: [5](#0-4) 

**Memory amplification math**: A contract loop executing N SLOAD opcodes across M total opcodes produces up to N×M `TreeMap` entries. Each entry is two 66-character hex strings (~132 bytes). Example: 1 000 SLOADs, 10 000 total opcodes → ~10 000 000 map entries → ~1.3 GB of live heap objects, all retained until the response is serialized.

**Database pressure**: Each SLOAD during replay triggers a `ContractStateRepository.findStorageByBlockTimestamp()` query: [6](#0-5) 

Thousands of SLOADs in a single replay exhaust the JDBC connection pool for the duration of the request.

### Impact Explanation
A single HTTP request targeting a historical transaction with a SLOAD-heavy loop can:
- Exhaust JVM heap, triggering `OutOfMemoryError` and crashing the web3 service.
- Hold JDBC connections for the full replay duration, starving other requests of database access.
- Because the endpoint is unauthenticated and the throttle is per-request-rate (not per-request-cost), a small number of concurrent requests (or even one) is sufficient to bring down the node. This aligns with the stated scope of ≥30% network processing node shutdown.

### Likelihood Explanation
- No account, API key, or privilege is required.
- Any historical transaction hash is sufficient as input; an attacker can deploy and execute a SLOAD-heavy contract on mainnet, then immediately replay it via this endpoint.
- The exploit is deterministic and repeatable. A single attacker with one HTTP client can trigger it.
- The gzip header requirement is a one-line addition to any `curl` or script (`-H "Accept-Encoding: gzip"`).

### Recommendation
1. **Cap the number of opcodes collected**: enforce a hard limit (e.g., 100 000) in `OpcodeContext.addOpcodes()` and return a 413/507 error when exceeded.
2. **Cap storage entries per opcode**: in `captureStorage()`, stop adding entries once a configurable maximum (e.g., 1 000) is reached.
3. **Cap total storage accesses per replay**: abort the EVM replay if cumulative SLOAD count exceeds a threshold.
4. **Add a request-level execution timeout**: wrap `contractDebugService.processOpcodeCall()` in a bounded `CompletableFuture` or use a `TimeLimiter`.
5. **Require authentication or an API key** for this endpoint, given its documented high resource cost.

### Proof of Concept
```
# 1. Deploy a Solidity contract with a SLOAD-heavy loop on the target network:
#    for (uint i = 0; i < 2000; i++) { assembly { sload(i) } }
#
# 2. Execute the transaction and record its hash (TX_HASH).
#
# 3. Send the replay request with storage=true:
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/<TX_HASH>/opcodes?storage=true"
#
# Result: the server allocates O(N*M) TreeMap entries (N=SLOAD count, M=total opcodes),
# retains them all in heap until JSON serialization completes, and issues thousands of
# DB queries. With N=2000 SLOADs and M~20000 opcodes, heap pressure exceeds 2 GB.
# Concurrent requests cause OOM / connection pool exhaustion.
```

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
