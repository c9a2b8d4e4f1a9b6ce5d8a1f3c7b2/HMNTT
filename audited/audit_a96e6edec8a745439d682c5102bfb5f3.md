### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent `timestamp=eq:X` Requests to `/api/v1/balances`

### Summary
The `/api/v1/balances` endpoint in `rest/balances.js` performs a synchronous DESC-ordered database query against the `account_balance` table on every request that includes a timestamp filter, with no rate limiting applied at the application layer. An unprivileged attacker can flood the endpoint with many concurrent `timestamp=eq:X` requests, each holding a connection from the finite pool, exhausting it and rendering the REST API unresponsive to all callers including health checks and the importer's dependent services.

### Finding Description

**Code path:**

`getBalances()` (`rest/balances.js:83`) → `getTsQuery()` (`rest/balances.js:278`) → `getAccountBalanceTimestampRange()` (`rest/balances.js:167`) → `pool.queryQuietly(query, params)` (`rest/balances.js:189`) [1](#0-0) 

When `tsQuery` is non-empty (any `timestamp=` filter), `getTsQuery` is awaited synchronously before any response is sent. Inside `getAccountBalanceTimestampRange`, the following query is issued unconditionally against the `account_balance` table: [2](#0-1) 

This query scans the `account_balance` partition for the treasury account with a DESC sort and `LIMIT 1`. While `getOptimizedTimestampRange` narrows the scan to at most two monthly partitions, it does not eliminate the query — it always executes a real DB round-trip for any valid timestamp value. [3](#0-2) 

**Root cause — no rate limiting:**

The middleware stack registered in `server.js` is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. [4](#0-3) 

There is no `express-rate-limit`, `helmet`, `slowDown`, or any per-IP/global concurrency cap applied to the `/api/v1/balances` route. The `authHandler` only sets a custom response `limit` for authenticated users; it does not throttle unauthenticated requests. [5](#0-4) 

**Connection pool:**

The pool is bounded by `config.db.pool.maxConnections` and `connectionTimeoutMillis`. [6](#0-5) 

Each in-flight request with a timestamp filter holds one pool connection for the duration of the `getAccountBalanceTimestampRange` query. Once all `maxConnections` slots are occupied, subsequent requests queue waiting for `connectionTimeoutMillis` ms, then fail — making the entire REST API unresponsive.

### Impact Explanation

When the pool is exhausted, every endpoint sharing the global `pool` object fails, including `/health/readiness` which queries the DB: [7](#0-6) 

A failed readiness check causes Kubernetes to mark the pod as not ready and stop routing traffic to it. If all REST API replicas are simultaneously pool-exhausted, the public API goes dark. The importer is a separate Java service with its own DB pool, so it is not directly blocked; however, any downstream service or operator tooling that depends on the REST API to confirm transaction state is denied service. The severity is high for REST API availability.

### Likelihood Explanation

No authentication, no API key, no CAPTCHA, and no application-layer rate limit is required. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple async script) can open hundreds of concurrent connections from a single IP. Because each request is a valid HTTP GET with a syntactically legal `timestamp=eq:X` parameter, it bypasses all input validation. The attack is trivially repeatable and requires no special knowledge of the system beyond the public OpenAPI spec.

### Recommendation

1. **Add a per-IP rate limiter** (e.g., `express-rate-limit`) as Express middleware before the balances route, limiting requests per second per IP.
2. **Add a global concurrency cap** on in-flight DB queries (e.g., a semaphore or `pg-pool` `max` tuned with a short `connectionTimeoutMillis`) so that pool exhaustion causes fast 503 responses rather than indefinite queuing.
3. **Cache the result of `getAccountBalanceTimestampRange`** for a short TTL (e.g., 1–5 seconds) keyed on the normalized timestamp bounds, since balance snapshots are written on a fixed schedule (every ~15 minutes). This collapses many distinct `timestamp=eq:X` values that resolve to the same snapshot into a single DB query.
4. **Deploy an ingress-level rate limit** (e.g., Traefik `rateLimit` middleware, as already done for the Rosetta service) for the REST API ingress.

### Proof of Concept

```bash
# Send 500 concurrent requests each with a distinct timestamp value
# No authentication required
seq 1 500 | xargs -P 500 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/balances?timestamp=eq:169000000{}" \
  -o /dev/null -w "%{http_code}\n" &

# In a separate terminal, observe pool exhaustion:
curl -v "https://<mirror-node-host>/api/v1/balances?timestamp=eq:1690000001"
# Expected: connection timeout or 503 once pool is exhausted

# Health check also fails:
curl -v "https://<mirror-node-host>/health/readiness"
# Expected: 503 / timeout
```

Each of the 500 concurrent requests enters `getAccountBalanceTimestampRange` and blocks on `pool.queryQuietly`. Once `maxConnections` slots are filled, all subsequent requests — including health checks — are denied DB access, making the pod appear unhealthy and causing it to be removed from the load balancer rotation.
