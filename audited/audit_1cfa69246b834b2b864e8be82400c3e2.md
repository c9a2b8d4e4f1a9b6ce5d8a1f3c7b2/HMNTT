### Title
Unauthenticated DB Connection Pool Exhaustion via Unique Opaque EVM Address Flooding in `getEncodedId()`

### Summary
The `/api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/crypto` endpoint accepts any valid EVM address as a path parameter with no authentication or rate limiting. When an opaque EVM address is supplied, `EntityService.getEntityIdFromEvmAddress()` unconditionally issues a live database query on every request. The `EntityId` LRU cache only caches the parsed `EntityId` object — not the DB lookup result — so flooding with unique EVM addresses bypasses the cache entirely and saturates the DB connection pool, denying service to all mirror node API consumers.

### Finding Description

**Entry point** — `rest/controllers/cryptoAllowanceController.js:77`: [1](#0-0) 

The path parameter is passed directly to `EntityService.getEncodedId()` with no prior rate limiting or authentication check.

**Branch that triggers the DB query** — `rest/service/entityService.js:122-124`: [2](#0-1) 

When `entityId.evmAddress !== null` (i.e., the address is "opaque" — first 12 bytes are not all zeros or `num > maxNum`), execution falls into `getEntityIdFromEvmAddress()`.

**The unconditional DB query** — `rest/service/entityService.js:90-104`: [3](#0-2) 

This executes `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` on every call. There is no result-level cache here.

**Why the LRU cache does not help** — `rest/entityId.js:301-333`: [4](#0-3) 

`parseCached` caches the `EntityId` *object* (the parse result). For an opaque EVM address this object has `evmAddress` set to the hex string and `num = null`. When `getEncodedId()` receives this cached object it still evaluates `entityId.evmAddress === null` as `false` and re-issues the DB query. The cache eliminates the regex/parse overhead but **not the database round-trip**.

**Opaque address classification** — `rest/entityId.js:250-253`: [5](#0-4) 

Any 40-hex-char string whose first 24 hex digits are not `000000000000000000000000` (or whose trailing num exceeds `maxNum`) is classified as opaque and routed to the DB lookup. An attacker trivially generates such addresses (e.g., random bytes with a non-zero byte in the first 12).

**Finite connection pool** — `rest/dbpool.js:14`: [6](#0-5) 

`max: config.db.pool.maxConnections` is the hard ceiling. No application-level rate limiting exists on this endpoint in the Node.js REST service code.

### Impact Explanation

When the DB connection pool is exhausted, every subsequent request — including legitimate account, transaction, and allowance queries — blocks waiting for a connection until `connectionTimeoutMillis` expires and returns an error. This renders the mirror node REST API completely unavailable. Note: the Hedera consensus network itself confirms transactions independently of the mirror node, so the impact is a full mirror node REST API denial-of-service rather than a consensus-layer shutdown. For operators and integrators who depend on the mirror node for account state, allowance data, and transaction history, this is a critical availability failure.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no privileged access. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) can generate thousands of unique 40-hex-char opaque EVM addresses per second. Each request is syntactically valid, passes all input validation, and forces a DB query. The attack is trivially repeatable and automatable.

### Recommendation

1. **Cache DB lookup results**: In `EntityService`, wrap `getEntityIdFromEvmAddress()` with a short-lived in-process cache (e.g., a `quickLru` keyed on the hex EVM address) so repeated lookups — including cache-miss results — do not hit the DB.
2. **Rate-limit the endpoint**: Apply per-IP or global request-rate limiting middleware (e.g., `express-rate-limit`) to all `/accounts/:id/allowances/*` routes before the async handler is entered.
3. **Validate EVM address existence cheaply**: Consider returning a fast 404 from a bloom filter or a small in-memory set of known-valid EVM addresses before issuing a DB query.
4. **Set a DB query concurrency limit**: Use a semaphore or queue in `BaseService.getRows()` to cap the number of simultaneous DB queries from EVM address resolution independently of the pool size.

### Proof of Concept

```bash
# Generate N unique opaque EVM addresses and flood the endpoint concurrently
python3 - <<'EOF'
import subprocess, secrets, threading

def attack(i):
    # First byte 0xAA ensures non-zero prefix -> opaque address
    addr = "aa" + secrets.token_hex(19)  # 40 hex chars, opaque
    subprocess.run([
        "curl", "-s", "-o", "/dev/null",
        f"http://<mirror-node-host>/api/v1/accounts/0x{addr}/allowances/crypto"
    ])

threads = [threading.Thread(target=attack, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
EOF
```

**Preconditions**: Network access to the mirror node REST API (port 80/443). No authentication.

**Trigger**: 500+ concurrent requests each with a distinct opaque EVM address path parameter.

**Result**: DB connection pool (`max` connections) is fully occupied by `SELECT id FROM entity WHERE evm_address = $1` queries. Legitimate requests receive connection timeout errors. Mirror node REST API is unavailable for the duration of the flood.
