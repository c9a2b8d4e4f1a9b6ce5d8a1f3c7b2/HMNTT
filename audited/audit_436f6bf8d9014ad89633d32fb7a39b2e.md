### Title
Unauthenticated Cache-Miss Flood Exhausts DB Connection Pool in `doGetTransactions`, Causing REST API Denial of Service

### Summary
The `/api/v1/transactions` endpoint in `rest/transactions.js` has no rate limiting and requires no authentication. An attacker can flood it with concurrent requests using unique `timestamp` query parameters, causing every request to miss the Redis cache and independently invoke `getTransactionsDetails()`, which issues a heavy multi-table CTE query against the database. With a default pool size of only 10 connections, as few as 10 concurrent cache-miss requests saturate the pool, blocking all subsequent REST API database operations for up to 20 seconds per query.

### Finding Description

**Exact code path:**

`doGetTransactions()` at `rest/transactions.js:695-723` executes two sequential DB-touching operations per request:

1. `getTransactionTimestamps(filters, timestampRange)` — always issues a DB query, no caching. [1](#0-0) 

2. `cache.get(payerAndTimestamps, loader, keyMapper)` — checks Redis for each key produced by `keyMapper`. The `keyMapper` maps entries to `transaction:{consensus_timestamp}` keys. [2](#0-1) 

**Cache miss path in `cache.get()`:**

When Redis is not ready (`!this.ready`), `loader(keys)` is called directly with no caching at all. When Redis is ready but keys are absent, `loader(missingKeys)` is called once per request for all missing keys. [3](#0-2) 

**`loader` invokes `getTransactionsDetails()`**, which executes a heavy CTE query joining `crypto_transfer`, `token_transfer`, and `transaction` tables via `pool.queryQuietly()`. [4](#0-3) 

**Root cause — failed assumption:** The design assumes the Redis cache will absorb repeated requests for the same timestamps. However, an attacker using unique `timestamp=lt:X` values per request produces unique `payerAndTimestamps` rows → unique cache keys → guaranteed cache misses on every request. Each concurrent request independently holds a DB connection for the full query duration.

**DB pool configuration:**

The pool is initialized with `max: config.db.pool.maxConnections`, defaulting to **10 connections**, with `connectionTimeoutMillis` of 20,000 ms and `statement_timeout` of 20,000 ms. [5](#0-4) [6](#0-5) 

**No rate limiting exists** on the `/transactions` route in `server.js`. The middleware stack applies only logging, authentication (optional), metrics, and response caching — no per-IP or global request throttle. [7](#0-6) 

### Impact Explanation

With 10+ concurrent requests using distinct timestamp parameters, all 10 default DB connections are held simultaneously for up to 20 seconds each (the `statementTimeout`). Any further REST API request requiring a DB connection — across all endpoints — queues for up to 20 seconds (`connectionTimeoutMillis`) before failing. This renders the entire REST API unavailable for the duration of the attack. The importer's HikariCP pool is separate, so consensus-layer transaction ingestion continues, but no client can query transaction data via the REST API. In production deployments where PostgreSQL `max_connections` is shared across services (200 in default config), sustained flooding can also starve the importer's connection pool, blocking transaction record writes.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and is trivially reproducible with any HTTP load tool (`ab`, `wrk`, `curl` in parallel). The attacker only needs to vary the `timestamp` query parameter (e.g., `timestamp=lt:$(date +%s%N)`) to guarantee cache misses. The attack is repeatable indefinitely and requires no knowledge of the system beyond the public API documentation. The default pool size of 10 makes the threshold extremely low.

### Recommendation

1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) applied globally or specifically to `/api/v1/transactions`, limiting requests per IP per second.
2. **Add a response-level cache** (the existing `responseCacheCheckHandler` is disabled by default — `cache.response.enabled: false`). Enabling it would serve identical URLs from cache before hitting the DB at all. [8](#0-7) 
3. **Increase `maxConnections`** from the default of 10 to a value appropriate for expected concurrency, and configure a shorter `connectionTimeoutMillis` to fail fast rather than queue.
4. **Add a concurrency limiter** (e.g., `express-slow-down` or a semaphore) to cap simultaneous in-flight DB queries from the REST API.
5. Consider adding a **Traefik/ingress-level rate limit** for the REST service, similar to what is already configured for the Rosetta service. [9](#0-8) 

### Proof of Concept

```bash
# Send 20 concurrent requests with unique timestamps to guarantee cache misses
for i in $(seq 1 20); do
  TS=$(($(date +%s%N) - i))
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=lt:${TS}&limit=100" &
done
wait

# Observe: subsequent legitimate requests hang for ~20s then fail with 503/timeout
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: connection timeout or 503 due to exhausted DB pool
```

The attack is sustained by repeating the loop continuously. With `statement_timeout=20000ms` and `maxConnections=10`, only 10 concurrent requests are needed to fully saturate the pool.
