### Title
Unbounded Timestamp Scan via `eq`→`lte` Conversion in `getContractStateById` Enables Unauthenticated DoS

### Summary
In `rest/controllers/contractController.js`, `extractContractStateByIdQuery()` silently converts a user-supplied `timestamp=eq:X` operator to `lte` before building the SQL condition, with no upper-bound validation on `X`. When `X` is set to a far-future value (e.g., `9999999999`), the resulting `DISTINCT ON (slot)` query against `contract_state_change` must process every historical row for that contract across all partitions before applying `LIMIT`, causing a full per-contract index scan that degrades database performance for all concurrent users.

### Finding Description

**Code path:**

`getContractStateById` (line 964) calls `extractContractStateByIdQuery` (line 970), which processes timestamp filters at lines 909–918:

```js
// contractController.js lines 914-917
if (utils.opsMap.eq === filter.operator) {
  filter.operator = utils.opsMap.lte;
}
conditions.push(this.getFilterWhereCondition(ContractStateChange.CONSENSUS_TIMESTAMP, filter));
```

The mutated filter (now `lte`) is passed to `ContractService.getContractStateByIdAndFilters` (line 971), which — because `timestamp === true` — executes `contractStateTimestampQuery` (lines 111–122 of `contractService.js`):

```js
static contractStateTimestampQuery = `
  with ${ContractService.entityCTE}
  select DISTINCT on (${ContractStateChange.SLOT})
        ${ContractStateChange.CONTRACT_ID},
        ${ContractStateChange.SLOT},
        ...
  from ${ContractStateChange.tableName} ${ContractStateChange.tableAlias}
  ...
`;
```

assembled with `ORDER BY slot, consensus_timestamp DESC LIMIT $N` (lines 258–263 of `contractService.js`).

**Root cause — missing validation:**

Every other timestamp-bounded endpoint calls `utils.parseTimestampFilters()` (lines 1583–1681 of `utils.js`), which enforces `config.query.maxTimestampRangeNs`. `extractContractStateByIdQuery` never calls this function. The only guard present is a rejection of the `ne` operator (line 910–912); there is no upper-bound cap on the supplied value.

**Exploit flow:**

1. Attacker sends: `GET /api/v1/contracts/0.0.5001/state?timestamp=eq:9999999999`
2. `extractContractStateByIdQuery` converts `eq` → `lte`, producing condition `csc.consensus_timestamp <= 9999999999000000000`.
3. Because `contract_state_change` is range-partitioned on `consensus_timestamp` (V2.0.0 schema, line 181), PostgreSQL must scan **all** partitions for that `contract_id`.
4. `DISTINCT ON (slot) ORDER BY slot, consensus_timestamp DESC` forces the planner to materialise and sort every matching row before it can emit the first result; `LIMIT` is applied only after this work is complete.
5. The index `contract_state_change__id_slot_timestamp (contract_id, slot, consensus_timestamp)` (V2.0.3 migration, line 68) is used, but it must traverse the entire index range for that `contract_id` across all partitions.
6. For a heavily-used contract (many distinct slots × many historical writes), this is a multi-million-row scan per request.
7. The attacker repeats the request in a tight loop; no rate-limiting or query-timeout is visible in the controller path.

### Impact Explanation

The attack degrades shared database resources (CPU, I/O, connection pool) for all users of the mirror-node REST API. Because the scan is bounded only by the number of historical state changes for the targeted contract, a popular contract (e.g., a high-volume DeFi contract) amplifies the cost. No data is modified or leaked; the impact is availability (griefing / denial-of-service) with no direct economic damage to any on-chain participant, consistent with the Medium severity scope.

### Likelihood Explanation

The endpoint requires no authentication. The exploit requires only knowledge of a contract ID with a large number of historical state changes — trivially discoverable from public blockchain data. The request is a standard HTTP GET with a single query parameter. It is repeatable at will, and a single attacker with a modest request rate can sustain elevated database load. No special tooling is needed beyond `curl`.

### Recommendation

1. **Enforce an upper-bound cap on the timestamp parameter** inside `extractContractStateByIdQuery`: call `utils.parseTimestampFilters()` (with `allowOpenRange: true` and `validateRange: true`) on the collected timestamp filters before building conditions, so `maxTimestampRangeNs` is respected.
2. **Reject or cap a bare `eq` / `lte` timestamp that exceeds `now()`**: a timestamp in the far future carries no semantic value for historical state queries.
3. **Add a statement-level query timeout** (e.g., PostgreSQL `statement_timeout`) for this endpoint class to bound worst-case execution time regardless of input.

### Proof of Concept

```bash
# Target a contract known to have many state changes
CONTRACT="0.0.5001"
MIRROR="https://<mirror-node-host>"

# Single request — forces full historical scan
curl -s "$MIRROR/api/v1/contracts/$CONTRACT/state?timestamp=eq:9999999999"

# Sustained attack — repeat in a loop
while true; do
  curl -s "$MIRROR/api/v1/contracts/$CONTRACT/state?timestamp=eq:9999999999" &
done
```

**Expected observable effect:** database CPU and I/O spike; response latency for all other API endpoints increases; connection pool exhaustion under sustained load.
