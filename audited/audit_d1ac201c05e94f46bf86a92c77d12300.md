### Title
Cache Poisoning via Incorrect `unless` Condition Allows Permanent Empty-Bytecode Caching for Any Contract ID

### Summary
`findRuntimeBytecode()` in `ContractRepository` uses `unless = "#result == null"` but the method returns `Optional<byte[]>`, which is **never** `null` — Spring Data JPA always returns a non-null `Optional`. This means `Optional.empty()` (returned for non-existent contracts) is unconditionally cached, allowing any unprivileged caller to poison the bytecode cache for arbitrary contract IDs. Since the cache uses `expireAfterAccess=1h` with a `maximumSize=1000`, an attacker can fill all 1000 slots with empty entries and keep them alive indefinitely by periodically re-querying, causing the mirror node's web3 API to report no bytecode for any targeted contract.

### Finding Description
**Exact location:**
- File: `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18
- Consumer: `web3/src/main/java/org/hiero/mirror/web3/state/keyvalue/ContractBytecodeReadableKVState.java`, lines 42–46

**Root cause:**

```java
// ContractRepository.java line 16
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

Spring's `@Cacheable` evaluates `#result` against the actual return value. Since the return type is `Optional<byte[]>`, `#result` is always a non-null `Optional` object — either `Optional.empty()` or `Optional.of(bytes)`. The condition `#result == null` is therefore **always false**, so the `unless` guard never fires. Both present and absent contract lookups are unconditionally stored in the cache.

**Cache configuration** (`CacheProperties.java` line 22):
```
expireAfterAccess=1h, maximumSize=1000
```

**Downstream effect** (`ContractBytecodeReadableKVState.java` lines 42–46):
```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        .map(Bytes::wrap)
        .map(Bytecode::new)
        .orElse(null);   // Optional.empty() → returns null → EVM sees no code
```

When the cache serves a poisoned `Optional.empty()` for a real contract, `readFromDataSource` returns `null`, and the EVM treats the contract as having no deployed bytecode.

**Exploit flow:**
1. Attacker sends web3 API calls (e.g., `eth_getCode`, `eth_call`) targeting contract IDs that do not yet exist (or IDs of contracts about to be deployed — predictable on Hedera since entity IDs are sequential).
2. Each call reaches `findRuntimeBytecode(contractId)` → DB returns no row → method returns `Optional.empty()` → `unless` check passes (it is not `null`) → `Optional.empty()` is stored in cache under that `contractId` key.
3. Attacker repeats for up to 1000 distinct IDs, filling the entire cache with empty entries.
4. Attacker re-queries each poisoned ID every ~55 minutes to reset the `expireAfterAccess` timer, keeping all 1000 slots occupied indefinitely.
5. When a legitimate contract is deployed at any of those IDs, subsequent `eth_call`/`eth_getCode` requests served from the mirror node return "no code" for that contract, causing all EVM simulations against it to fail.

**Why existing checks are insufficient:**
The only guard is `unless = "#result == null"`. Because `Optional` is never `null`, this check provides zero protection against caching empty results. There is no secondary validation in `ContractBytecodeReadableKVState` — it blindly calls `orElse(null)` and propagates the cached empty result to the EVM.

### Impact Explanation
Any contract whose ID was pre-poisoned in the cache will appear code-less to the mirror node's web3 API for up to 1 hour per access cycle (indefinitely with attacker refresh). All `eth_call`, `eth_estimateGas`, and contract simulation requests against those contracts will fail or return incorrect results. With a cache size of 1000, an attacker can simultaneously poison all slots, effectively disabling bytecode resolution for every contract that is not already cached — forcing cache misses and elevated DB load for all other contracts, and silently breaking EVM execution for targeted ones.

### Likelihood Explanation
No authentication or special privilege is required. The mirror node's web3 API is publicly accessible with a default rate limit of 500 RPS. Filling 1000 cache slots requires only 1000 HTTP requests — achievable in under 2 seconds at the default rate limit. Hedera entity IDs are sequential and publicly observable, making it straightforward to predict upcoming contract IDs. The attack is repeatable and requires no on-chain action.

### Recommendation
Change the `unless` condition to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || #result.isEmpty()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This ensures only successful (non-empty) bytecode lookups are cached, which is the intended behavior. Additionally, consider switching the cache policy from `expireAfterAccess` to `expireAfterWrite` to bound the maximum staleness of any cached entry regardless of access patterns.

### Proof of Concept
**Preconditions:** Public access to the mirror node's web3 JSON-RPC endpoint.

**Steps:**

1. Identify 1000 sequential contract IDs that do not yet exist (e.g., IDs above the current max entity number, observable from the mirror node REST API).

2. For each ID, send an `eth_getCode` request:
```bash
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["0x000000000000000000000000000000000000XXXX","latest"],"id":1}'
```
Each call returns `0x` (no code) and caches `Optional.empty()` for that contract ID.

3. Wait for one of those contract IDs to be deployed on-chain (or deploy one yourself).

4. Re-query the same contract ID via `eth_getCode` or `eth_call`:
```bash
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["<deployed-contract-address>","latest"],"id":1}'
```

**Result:** The mirror node returns `0x` (no bytecode) even though the contract is deployed, because `Optional.empty()` is served from cache. Any `eth_call` to that contract will fail as if calling an EOA.

5. Repeat step 2 every ~55 minutes to keep all 1000 cache slots poisoned indefinitely.
