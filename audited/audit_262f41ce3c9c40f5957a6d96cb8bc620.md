### Title
Unauthenticated Opcode Replay of Failed Transactions Leaks Full EVM Execution Trace, Enabling Targeted Fund-Draining Attacks

### Summary
The `/api/v1/contracts/results/{transactionId}/opcodes` endpoint is publicly accessible with no authentication. `OpcodeServiceImpl.buildCallServiceParameters()` calls `TransactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc()` and proceeds to replay the first returned transaction without checking its `result` field, meaning failed fund-transfer transactions (result != SUCCESS) are replayed identically to successful ones. The full EVM execution trace — including stack values, memory contents, and storage slot data at every opcode step — is returned to any caller, exposing balance thresholds, access-control storage values, and the exact revert conditions needed to craft a successful attack.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (no `@PreAuthorize` or any auth annotation): [1](#0-0) 

The only guards are a feature-flag check, a gzip header check, and a rate-limit throttle — no authentication or authorization: [2](#0-1) 

`OpcodeServiceImpl.buildCallServiceParameters()` calls the repository and takes the first result with **no check on `transaction.getResult()`**: [3](#0-2) 

`TransactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc()` returns all transactions (including failed ones) ordered by consensus timestamp ascending: [4](#0-3) 

The replay engine in `TransactionExecutionService.handleFailedResult()` explicitly handles the opcode-trace case by **returning** a failed result instead of throwing, so the full opcode list is captured and returned even for reverted/failed transactions: [5](#0-4) 

**Root cause:** The failed assumption is that the opcode-replay endpoint is a privileged debugging tool. In reality it is an unauthenticated public endpoint, and `buildCallServiceParameters()` never inspects `parentTransaction.getResult()` before triggering the replay. The repository query intentionally returns all transactions for a given payer+validStartNs (including failed ones), and the replay engine is explicitly designed to return traces for failed transactions in opcode-trace mode.

### Impact Explanation

An attacker who replays a failed fund-transfer transaction with `?stack=true&memory=true&storage=true` receives:
- **Storage slot values** at every opcode step — reveals balance thresholds, access-control mappings, owner addresses, and any other state that caused the revert.
- **Stack values** — reveals intermediate computation results (e.g., the exact value compared against a threshold).
- **Memory contents** — reveals ABI-encoded function arguments and return data.
- **Revert reason** — the exact `reason` field on the REVERT opcode pinpoints the failing `require`/`revert` condition.

With this data an attacker can reconstruct the exact preconditions for a successful call, enabling targeted fund-draining (e.g., learning the minimum balance required to pass a guard, or the exact storage slot that controls a withdrawal allowance).

Severity: **High** — direct path from public information disclosure to crafting a successful fund-draining transaction.

### Likelihood Explanation

- Hedera transaction IDs (`payerAccountId-validStartSeconds-validStartNanos`) are fully public on-chain and via the mirror-node REST API.
- The attacker needs zero credentials; only a known transaction ID and an HTTP client with `Accept-Encoding: gzip`.
- Failed contract transactions are common (insufficient balance, access-control reverts, slippage checks) and are trivially enumerable via the public `/api/v1/transactions` endpoint filtered by `result=fail`.
- The throttle is the only friction; it can be bypassed by distributing requests or waiting between calls.
- Repeatability: the replay is deterministic and can be called for any historical failed transaction indefinitely.

### Recommendation

1. **Add a result check in `OpcodeServiceImpl.buildCallServiceParameters()`** before proceeding with the replay. After line 107 (`final var parentTransaction = transactionList.getFirst()`), verify that `parentTransaction.getResult()` equals the SUCCESS proto ID. If the transaction failed, throw an appropriate exception (e.g., `InvalidInputException` or a dedicated error) rather than replaying it.

2. **Require authentication/authorization** on the opcodes endpoint. The endpoint is explicitly a debugging tool (the Javadoc says so); it should require an operator-level API key or equivalent credential, not be open to the public internet.

3. **Scope the repository query** to only return transactions with `result = SUCCESS` by adding a `findByPayerAccountIdAndValidStartNsAndResultOrderByConsensusTimestampAsc()` variant that filters at the database level.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service is running with `opcodes.enabled=true`.
- A failed contract-call or crypto-transfer transaction exists on-chain (trivially found via `GET /api/v1/transactions?result=fail&type=CONTRACTCALL`).

**Steps:**

```
# 1. Find a failed transaction ID (public endpoint, no auth)
curl "https://<mirror-node>/api/v1/transactions?result=fail&type=CONTRACTCALL&limit=1"
# → returns e.g. transaction_id: "0.0.1234-1700000000-123456789"

# 2. Replay the failed transaction with full trace (no auth required)
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/0.0.1234-1700000000-123456789/opcodes?stack=true&memory=true&storage=true" \
  | gunzip

# 3. Response contains full opcode trace including storage slot values,
#    stack values at every step, memory contents, and the exact revert reason.
#    Example storage entry revealing a balance threshold:
#    "storage": {"0x0000...0001": "0x0000...03e8"}  ← threshold = 1000
#    Example revert reason: "0x08c379a0...496e73756666696369656e742062616c616e6365"
#    → decoded: "Insufficient balance"

# 4. Attacker now knows the exact threshold and crafts a successful call
#    meeting all the conditions revealed by the trace.
```
