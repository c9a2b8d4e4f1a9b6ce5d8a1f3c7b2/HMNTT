### Title
Unbounded Concurrent Throttled Subscriptions Exhaust Shared `boundedElastic()` Scheduler, Starving Legitimate Topic Message Delivery

### Summary
`PollingTopicMessageRetriever.retrieve()` allocates a single application-wide `Schedulers.boundedElastic()` instance shared across every subscription. When `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE` and each subscription indefinitely re-executes a blocking JDBC poll on that shared scheduler. Because the only server-side guard is `maxConcurrentCallsPerConnection = 5` (per-connection, not global), an unprivileged attacker can open an unbounded number of TCP connections and flood the shared scheduler, starving legitimate subscribers of their periodic polls and preventing topic message gossip delivery.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructor (line 41) calls `Schedulers.boundedElastic()`, which returns the **default JVM-wide shared** `BoundedElasticScheduler` instance — the same instance used by `TopicMessageServiceImpl.subscribeTopic()` (line 70, `safetyCheck.subscribeOn(Schedulers.boundedElastic())`).

```
PollingTopicMessageRetriever.java:41  scheduler = Schedulers.boundedElastic();
PollingTopicMessageRetriever.java:94-107  throttled=true → numRepeats = Long.MAX_VALUE
PollingTopicMessageRetriever.java:52-55  .repeatWhen(...withScheduler(scheduler))
PollingTopicMessageRetriever.java:65-79  poll() → Flux.fromStream(topicMessageRepository.findByFilter(...))  [blocking JDBC]
PollingTopicMessageRetriever.java:58    .retryWhen(Retry.backoff(Long.MAX_VALUE, ...))  [infinite retry]
GrpcConfiguration.java:33              serverBuilder.maxConcurrentCallsPerConnection(5)  [only guard]
```

**Root cause:** `RepeatSpec.withFixedDelay(frequency).withScheduler(scheduler)` re-subscribes to `Flux.defer(() -> poll(context))` on the scheduler thread after each delay. `poll()` calls `topicMessageRepository.findByFilter()` — a blocking `Stream`-returning JDBC call — which holds the `boundedElastic()` worker thread for the full duration of the database round-trip. With `numRepeats = Long.MAX_VALUE` and `pollingFrequency = 2s` (default), each throttled subscription permanently and periodically occupies a scheduler thread.

**Failed assumption:** The design assumes `maxConcurrentCallsPerConnection = 5` bounds total scheduler load. It does not — it only limits calls per TCP connection. There is no global concurrent-subscription cap, no per-IP connection limit, and no total-connection limit configured in `GrpcConfiguration.java`.

**Exploit flow:**
1. Attacker opens `C` TCP connections to port 5600 (no server-side connection limit).
2. On each connection, attacker opens 5 `subscribeTopic` streams (per-connection limit is 5).
3. Each stream triggers `topicMessageRetriever.retrieve(filter, true)` → `throttled=true` → `numRepeats = Long.MAX_VALUE`.
4. Every 2 seconds, each subscription schedules a blocking DB query on the shared `boundedElastic()` scheduler.
5. With `C × 5` subscriptions, the scheduler's thread pool (`10 × CPU_cores`, e.g. 40 threads on 4-core) is saturated.
6. Legitimate subscribers' poll tasks queue behind attacker tasks; with enough connections the 100,000-task queue fills and tasks are rejected with `RejectedExecutionException`, which the `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` on line 58 catches and retries — further amplifying scheduler pressure.

### Impact Explanation
Legitimate subscribers receive no topic messages (or severely delayed messages) because their periodic `poll()` tasks cannot be scheduled. This directly breaks the gossip/delivery guarantee for `ConsensusTopicResponse` streams. The `timeout = 60s` (line 28, `RetrieverProperties`) does not protect against this: it fires only when no message is emitted for 60 consecutive seconds, and the attacker can subscribe to any active topic to keep their own subscriptions alive while still saturating the scheduler. Even if attacker subscriptions time out, the attacker immediately reconnects, maintaining continuous saturation. Severity: **High** — complete denial of topic message delivery to all legitimate subscribers.

### Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. The gRPC port (5600) is publicly exposed. Opening hundreds of TCP connections from a single host is trivial with any gRPC client library (e.g., `grpc-java`, `grpcurl`). The attack is fully repeatable and automatable with ~50 lines of client code. No special network position or privileged account is needed.

### Recommendation
1. **Add a global concurrent-subscription cap** in `GrpcConfiguration.java` via `NettyServerBuilder.maxConcurrentCallsPerConnection` combined with `maxConnectionAge` / `maxConnectionIdle`, or track active subscription count in `TopicMessageServiceImpl.subscriberCount` and reject new subscriptions above a threshold.
2. **Isolate the retriever scheduler**: Replace `Schedulers.boundedElastic()` in `PollingTopicMessageRetriever` with a **dedicated, bounded** `Schedulers.newBoundedElastic(threadCap, queueSize, "retriever-poll")` so attacker subscriptions cannot starve the shared JVM-wide scheduler used by other components.
3. **Add per-IP or per-client connection rate limiting** at the Netty/ingress layer (e.g., `maxConnectionsPerIp` via a Netty `ChannelHandler` or an external proxy).
4. **Enforce a maximum subscription lifetime** independent of message activity (e.g., absolute `maxAge` timeout) so attacker subscriptions cannot persist indefinitely by subscribing to active topics.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools
import grpc, threading, time
# Assume proto stubs generated from hedera consensus service proto
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import timestamp_pb2, consensus_topic_id_pb2

TARGET = "mirror-node-grpc-host:5600"
CONNECTIONS = 200   # 200 connections × 5 streams = 1000 concurrent throttled subscriptions
STREAMS_PER_CONN = 5

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=consensus_topic_id_pb2.ConsensusTopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0)
        # no endTime, no limit → throttled=true, numRepeats=Long.MAX_VALUE
    )
    streams = [stub.subscribeTopic(query) for _ in range(STREAMS_PER_CONN)]
    for s in streams:
        threading.Thread(target=lambda st=s: list(st), daemon=True).start()
    time.sleep(3600)  # hold connections open

threads = [threading.Thread(target=flood_connection, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
# Result: 1000 concurrent throttled subscriptions saturate the shared boundedElastic()
# scheduler; legitimate subscriber polls queue indefinitely → no message delivery.
```

**Verification:** Monitor `hiero.mirror.grpc.subscribers` gauge (jumps to 1000+) and observe that legitimate subscriber streams stop receiving `ConsensusTopicResponse` messages within seconds of the attack starting.
