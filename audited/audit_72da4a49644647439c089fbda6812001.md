### Title
Hardcoded `sslmode=disable` in Rosetta `GetDsn()` Exposes PostgreSQL Credentials to Network Sniffing

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` permanently hardcodes `sslmode=disable` in the PostgreSQL DSN with no override mechanism, unlike every other component in the project. Because the `Db` struct contains no `SslMode` field, operators cannot enable TLS regardless of environment. Combined with Kubernetes network policies being disabled by default, any workload co-located in the cluster can intercept the plaintext PostgreSQL authentication handshake and extract the database username and password.

### Finding Description
**Exact location:** `rosetta/app/config/types.go`, lines 49–58, function `GetDsn()`:

```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        ...
    )
}
``` [1](#0-0) 

This DSN is consumed unconditionally by `ConnectToDb()` in `rosetta/app/db/db.go` line 18:

```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
``` [2](#0-1) 

**Root cause — no configurable SSL field:** The `Db` struct (lines 39–47) has no `SslMode` field, making it impossible to override via YAML or environment variable. This is in direct contrast to the importer's `DBProperties.java`, which exposes a configurable `SslMode` field defaulting to `DISABLE` but allowing operators to set `require` or `verify-full`: [3](#0-2) [4](#0-3) 

**Network policy disabled by default:** The Helm chart sets `networkPolicy.enabled: false`, meaning no Kubernetes NetworkPolicy restricts which pods can reach the PostgreSQL port: [5](#0-4) 

Even when enabled, the policy only restricts ingress to port 5432 from pods within the same namespace — it does not enforce TLS on the wire: [6](#0-5) 

**Exploit flow:**
1. Attacker deploys or compromises any pod in the same Kubernetes cluster (e.g., via a vulnerable public-facing service, supply-chain compromise, or misconfigured RBAC).
2. With `networkPolicy.enabled: false` (the default), the attacker pod can reach the PostgreSQL service endpoint.
3. The attacker performs ARP spoofing or uses a CNI-level tap (e.g., on a shared node) to intercept TCP traffic on port 5432 between the Rosetta pod and the database.
4. Because `sslmode=disable` is hardcoded, the PostgreSQL client never negotiates TLS. The authentication handshake — including the username and, depending on `pg_hba.conf` auth method, the password or a crackable MD5 hash — is transmitted in cleartext.
5. The attacker extracts the `mirror_rosetta` credentials and gains direct read access to the `mirror_node` database.

### Impact Explanation
PostgreSQL with `sslmode=disable` sends the username in cleartext in every connection. If `pg_hba.conf` uses `password` or `md5` authentication, the password or its MD5 hash is also transmitted in cleartext or as a trivially crackable challenge-response. An attacker who captures even a single authentication handshake obtains valid database credentials. The `mirror_rosetta` user has read access to the full mirror node database, exposing all ledger data. Because the `sslmode` is hardcoded and the struct has no override field, this cannot be mitigated by configuration alone — a code change is required. [7](#0-6) 

### Likelihood Explanation
The precondition is network adjacency to the Rosetta-to-PostgreSQL path. In a default Kubernetes deployment (network policy disabled, flat pod network), any pod in the cluster satisfies this. A single compromised workload — a vulnerable sidecar, a misconfigured ingress controller, or a supply-chain-compromised image — is sufficient. Cloud-provider shared-node scenarios (noisy-neighbor attacks) and insider threats also apply. The attack is passive (sniffing only), repeatable on every new connection, and requires no special kernel privileges when performed via ARP spoofing at the application layer. [5](#0-4) 

### Recommendation
1. Add a `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a default of `"require"` or `"verify-full"`.
2. Use that field in `GetDsn()` instead of the hardcoded literal `sslmode=disable`.
3. Document the field in `docs/configuration.md` under the Rosetta API section, consistent with how `hiero.mirror.importer.db.sslMode` is documented.
4. Enable `networkPolicy.enabled: true` in production Helm values as a defense-in-depth measure.
5. Set `pg_hba.conf` to require `scram-sha-256` and enforce `hostssl`-only entries for the `mirror_rosetta` user. [8](#0-7) [9](#0-8) 

### Proof of Concept
```bash
# 1. Deploy an attacker pod in the same namespace (no special privileges needed)
kubectl run attacker --image=nicolaka/netshoot -n mirror -- sleep 3600

# 2. Identify the Rosetta pod's node and the DB service ClusterIP
DB_IP=$(kubectl get svc mirror-postgresql -n mirror -o jsonpath='{.spec.clusterIP}')

# 3. From the attacker pod, use tcpdump or tshark on port 5432
kubectl exec -it attacker -n mirror -- \
  tcpdump -i eth0 -w /tmp/pg.pcap "host $DB_IP and port 5432"

# 4. Wait for the Rosetta pod to restart or open a new DB connection
# (connection pool reconnects periodically per MaxLifetime config)

# 5. Inspect the capture — the PostgreSQL startup message contains the username
# in plaintext; with md5 auth the password hash is also present
kubectl exec -it attacker -n mirror -- \
  tshark -r /tmp/pg.pcap -Y pgsql -T fields \
    -e pgsql.parameter_name -e pgsql.parameter_value

# Expected output includes:
# user    mirror_rosetta
# (plus md5 challenge-response or cleartext password depending on pg_hba.conf)
``` [1](#0-0) [2](#0-1)
