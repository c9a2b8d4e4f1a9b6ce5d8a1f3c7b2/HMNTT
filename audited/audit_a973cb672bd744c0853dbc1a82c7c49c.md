### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses SQL LIMIT, Enabling Unbounded DB Queries via Large `limit` Value

### Summary
An unprivileged gRPC client can submit a `ConsensusTopicQuery` with `limit` set to any value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`). The narrowing cast `(int)(filter.getLimit() - context.getCount().get())` in `PollingTopicListener.poll()` overflows to a negative integer. Because `TopicMessageFilter.hasLimit()` returns `false` for negative values, the SQL query is issued with no `LIMIT` clause, causing every poll cycle to fetch all matching records from the database — a resource-exhaustion DoS.

### Finding Description

**Entry point — no upper-bound check on `limit`:**

`ConsensusController.toFilter()` copies the raw protobuf `uint64` field directly into the filter with no cap:
```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```
`TopicMessageFilter` only enforces `@Min(0)` on `limit` — there is no `@Max` constraint — so any value in `[1, Long.MAX_VALUE]` passes Spring validation.

**Propagation through `incomingMessages`:**

`TopicMessageServiceImpl.incomingMessages()` (line 115-118) computes the residual limit as a `long` and passes it straight to the listener:
```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter)...
```
With `filter.getLimit() = Long.MAX_VALUE` and `count = 0`, `newFilter.getLimit() = Long.MAX_VALUE`.

**Overflow in `poll()`:**

`PollingTopicListener.poll()` (lines 54-57) performs an unchecked narrowing cast:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())   // ← overflow
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```
`(int)(Long.MAX_VALUE - 0L)` = `(int)(0x7FFFFFFFFFFFFFFF)` = `0xFFFFFFFF` = **`-1`** (signed).  
`Math.min(-1, 5000)` = **`-1`**.

**SQL LIMIT silently dropped:**

`TopicMessageRepositoryCustomImpl.findByFilter()` (lines 51-53) guards `setMaxResults` with `hasLimit()`:
```java
if (filter.hasLimit()) {          // hasLimit() = (-1 > 0) = false
    typedQuery.setMaxResults((int) filter.getLimit());
}
```
Because `hasLimit()` returns `false` for any negative value, `setMaxResults` is never called. The JPA query runs with **no SQL LIMIT**, returning every matching row in the topic.

**Subscription never terminates:**

Back in `TopicMessageServiceImpl.subscribeTopic()` (line 83-85):
```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());   // take(Long.MAX_VALUE) ≈ infinite
}
```
`take(Long.MAX_VALUE)` is effectively unbounded, so the subscription never completes on its own.

### Impact Explanation

Every poll interval (default 500 ms) the mirror node issues a full table-scan-equivalent query against the `topic_message` table with no row limit. On a busy topic with millions of messages this saturates database I/O, exhausts JPA/JDBC connection pool threads, and degrades or denies service for all other subscribers. The attacker's subscription also never self-terminates, holding a bounded-elastic scheduler thread and a DB connection for the lifetime of the connection. Multiple concurrent such subscriptions compound the effect. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. The protobuf `limit` field is a `uint64`; any gRPC client library can set it to `Long.MAX_VALUE` (or `Integer.MAX_VALUE + 1`) in a single line. The attack is trivially repeatable, requires no special knowledge of the system, and leaves no obvious error in logs (the query simply returns all rows). The only prerequisite is that the deployment uses `ListenerType.POLL`; the default is `REDIS`, but `POLL` is a documented and supported mode.

### Recommendation

1. **Add `@Max` to `TopicMessageFilter.limit`** — cap it at a safe value (e.g., `Integer.MAX_VALUE` or a domain-specific maximum):
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)
   private long limit;
   ```
2. **Replace the unsafe narrowing cast** in `PollingTopicListener.poll()` with a saturating conversion:
   ```java
   int limit = filter.hasLimit()
       ? (int) Math.min(filter.getLimit() - context.getCount().get(), Integer.MAX_VALUE)
       : Integer.MAX_VALUE;
   ```
3. **Enforce the same cap in `ConsensusController.toFilter()`** before building the filter, so the constraint is applied at the protocol boundary regardless of future internal changes.

### Proof of Concept

```java
// Any unauthenticated gRPC client:
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0), no @Max exists
    .build();

// Send to mirror node gRPC endpoint (default :5600).
// Result per poll cycle (every 500 ms):
//   (int)(Long.MAX_VALUE - 0) = -1
//   Math.min(-1, 5000)        = -1
//   hasLimit(-1)              = false  → no SQL LIMIT
//   SELECT * FROM topic_message WHERE topic_id=1 AND consensus_timestamp >= 0
//   → returns ALL rows, unbounded, on every poll tick
//   → DB connection pool exhausted; other subscribers starved
```
