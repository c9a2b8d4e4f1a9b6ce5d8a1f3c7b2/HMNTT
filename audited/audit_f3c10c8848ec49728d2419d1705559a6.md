### Title
3× DB Work Amplification via Three-Subquery UNION ALL Path in `getQuery()` — Unauthenticated DoS

### Summary
In `rest/service/tokenAllowanceService.js`, `getQuery()` constructs a three-branch `UNION ALL` query where each subquery independently receives `LIMIT $2` (the user-supplied limit), and the outer wrapper query also applies `LIMIT $2`. An unprivileged user can craft a single valid HTTP request that causes the database to sort and scan up to `3 × max_limit` (300) rows before the final limit of 100 is applied. Repeated concurrent requests amplify database CPU and I/O load by a factor of 3 with no rate-limiting guard.

### Finding Description

**Exact code location:** `rest/service/tokenAllowanceService.js`, `getQuery()`, lines 54–78.

```js
// line 58 — limitClause is "limit $2" where $2 = user limit (max 100)
const limitClause = super.getLimitQuery(2);

// lines 63–65 — every non-empty subquery gets the same limitClause
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));

// line 74 — outer query also gets the same limitClause
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

`getLimitQuery(2)` in `rest/service/baseService.js` line 34–36 simply returns the string `"limit $2"`. Because `$2` is the single shared parameter slot for the user-supplied limit, every subquery and the outer wrapper all reference the same value.

**Generated SQL (confirmed by test at lines 215–232 of `rest/__tests__/service/tokenAllowanceService.test.js`):**

```sql
(select * from token_allowance where owner=$1 and amount>0 and spender=$3 and token_id>=$4
 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender>$5 and spender<$6
 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender=$7 and token_id<=$8
 order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc
limit $2
```

With `$2 = 100` (max), the DB materialises up to 300 rows, sorts them, then discards 200.

**How all three subqueries are triggered:** `getUpperFilters()` (`baseController.js` lines 176–183) returns a non-empty array only when both `primary.hasUpper()` AND `secondary.hasUpper()`. `getInnerFilters()` (lines 154–168) returns non-empty when both primary and secondary have any bound. `getLowerFilters()` (lines 138–141) returns non-empty when `primary.hasLower() && secondary.hasLower()`. All three conditions are satisfied simultaneously by:

```
spender.id=gte:X  → primary.lower (gte operator passes validateLowerBounds)
spender.id=lte:Y  → primary.upper (lte operator passes validateUpperBounds)
token.id=gte:A    → secondary.lower
token.id=lte:B    → secondary.upper
```

**Root cause:** `limitClause` is built once as `"limit $2"` and reused verbatim in every subquery and the outer query. There is no per-subquery limit reduction or a higher intermediate limit that is later narrowed.

**Existing checks are insufficient:** `validateBounds()` (`baseController.js` lines 56–61) only enforces logical correctness of operator combinations (no `ne`, no simultaneous range+equal). It does not detect or prevent the amplification. `getLimitParamValue()` (`utils.js` lines 544–553) caps the user value at `responseLimit.max` (100 per config test line 325) but does nothing about the 3× multiplier.

### Impact Explanation

Each request with the four-filter combination causes the PostgreSQL engine to execute three independent index scans on `token_allowance`, each materialising up to 100 rows, then merge-sort 300 rows and discard 200. With concurrent unauthenticated requests this triples effective DB read/sort load. On a busy node with a large `token_allowance` table, sustained parallel requests can saturate the DB connection pool (default pool size is small) or exhaust CPU allocated to the mirror-node DB, degrading or denying service to all API consumers. The endpoint is publicly accessible with no authentication requirement.

### Likelihood Explanation

The exploit requires only a valid account ID (publicly enumerable from `/api/v1/accounts`) and four standard query parameters that are explicitly documented as supported. No credentials, tokens, or special privileges are needed. The request is indistinguishable from legitimate pagination use. An attacker can script hundreds of concurrent requests trivially. The amplification is deterministic and repeatable on every request.

### Recommendation

1. **Use a multiplied intermediate limit for subqueries.** When N subqueries are unioned, pass `N × limit` as the per-subquery limit and `limit` for the outer query, or use a separate parameter slot:
   ```js
   const subLimitClause = `limit ${subQueries.length * limit}`; // literal, not parameterised
   // use subLimitClause inside each subquery, limitClause only on the outer query
   ```
2. **Alternatively**, push the `ORDER BY … LIMIT` only to the outer query and remove per-subquery limits entirely, relying on the outer limit to bound total work (acceptable when subquery result sets are already bounded by tight indexed predicates).
3. **Add rate limiting** on the `/api/v1/accounts/{id}/allowances/tokens` endpoint for unauthenticated callers.

### Proof of Concept

**Precondition:** Any account ID with token allowances spanning multiple spenders (e.g., `0.0.1000`).

**Trigger request (no authentication):**
```
GET /api/v1/accounts/0.0.1000/allowances/tokens?spender.id=gte:0.0.1&spender.id=lte:0.0.9999999&token.id=gte:0.0.1&token.id=lte:0.0.9999999&limit=100
```

**What happens in the DB:**
```sql
-- Subquery 1 (lower): scans spender=1, token_id>=1 → up to 100 rows
-- Subquery 2 (inner): scans spender>1 AND spender<9999999 → up to 100 rows
-- Subquery 3 (upper): scans spender=9999999, token_id<=9999999 → up to 100 rows
-- Outer: sort 300 rows, return 100
```

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }
