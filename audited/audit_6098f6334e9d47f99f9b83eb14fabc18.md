### Title
Unbounded Repeated DB Query Amplification via Unauthenticated `/block` Endpoint with Index-Only PartialBlockIdentifier

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` performs no response-level caching. Any unauthenticated caller can repeatedly POST to `/block` with a `PartialBlockIdentifier` containing only an `Index` (e.g., index 1), causing the server to re-execute full database queries for block lookup, all transactions in that block's time range (paginated but unbounded), and per-operation account alias resolution on every single request. The only partial mitigation — an LRU entity cache for aliases — does not protect the block or transaction query paths.

### Finding Description

**Exact code path:**

`Block()` at `rosetta/app/services/block_service.go` lines 47–74:

- **Line 51**: `s.RetrieveBlock(ctx, request.BlockIdentifier)` → dispatches to `base_service.go` line 83: `b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)` when only `Index` is set and `Hash` is nil. This executes `selectRecordBlockByIndex` SQL against the DB on every call — no caching exists at this layer.

- **Line 56**: `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` → `transaction.go` lines 112–171: a paginated loop (`for start <= end`) that issues repeated `selectTransactionsInTimestampRangeOrdered` queries until all transactions in the block's time range are fetched. The full set is always fetched from DB regardless of `maxTransactionsInBlock`.

- **Lines 61–67**: `maxTransactionsInBlock` truncates the slice *after* `FindBetween` returns the full result set. It does not prevent the DB from scanning all transactions.

- **Line 69**: `s.updateOperationAccountAlias(ctx, block.Transactions...)` → `block_service.go` lines 104–130: for each operation in each (capped) transaction, checks `entityCache`; on a cache miss calls `s.accountRepo.GetAccountAlias()` — another DB query. The LRU cache (`entityCache`) only covers aliases, not blocks or transactions.

**Root cause:** The failed assumption is that callers will not repeatedly request the same historical block. There is no memoization or response cache for the `/block` endpoint at the application level. The `entityCache` is scoped only to account alias resolution and provides no protection for the block-lookup or transaction-fetch paths.

**Why existing checks fail:**

- `entityCache` (LRU, `block_service.go` lines 22, 34–37): caches `AccountId` aliases only; block and transaction queries are uncached.
- `maxTransactionsInBlock` (lines 61–67): truncates the response slice but does not short-circuit `FindBetween`; the full DB scan still occurs.
- Traefik rate-limiting middleware (`charts/hedera-mirror-rosetta/values.yaml` lines 149–166: `rateLimit: average: 10`, `inFlightReq: amount: 5`): gated behind `global.middleware: false` (line 95) — **disabled by default**. No application-level rate limiting exists.

### Impact Explanation
An attacker can sustain elevated DB load by flooding the endpoint with requests for a single early block index. Each request forces: one `selectRecordBlockByIndex` query, one or more `selectTransactionsInTimestampRangeOrdered` paginated queries (proportional to transaction count), and up to `maxTransactionsInBlock × ops_per_tx` alias queries on cold cache. For a block with thousands of transactions this is a significant per-request DB amplification. Sustained at even modest request rates this degrades query latency for all legitimate users sharing the same DB connection pool.

### Likelihood Explanation
The Rosetta API is a public, unauthenticated HTTP API by design (Coinbase Rosetta spec). No API key, session, or privilege is required. The attacker needs only network access and knowledge of the endpoint (publicly documented). The attack is trivially scriptable (`curl` in a loop), repeatable indefinitely, and requires no special tooling. Default deployments ship with `global.middleware: false`, meaning the only defined rate-limiting layer (Traefik) is inactive out of the box.

### Recommendation
1. **Add a response-level cache** for `Block()` results keyed on `(index, hash)` or `(consensusStart, consensusEnd)` with a short TTL (e.g., 5–30 s). Historical blocks are immutable once finalized, making them safe to cache aggressively.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) or add application-level rate limiting (e.g., `golang.org/x/time/rate` per-IP token bucket) so the protection is not opt-in.
3. **Short-circuit `FindBetween`** to respect `maxTransactionsInBlock` at the query level (add a DB-side `LIMIT`) rather than fetching all rows and truncating in memory.

### Proof of Concept

```bash
# No authentication required. Repeat at high concurrency.
while true; do
  curl -s -X POST http://<rosetta-host>/block \
    -H 'Content-Type: application/json' \
    -d '{
      "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
      "block_identifier": {"index": 1}
    }' > /dev/null &
done
```

Each iteration triggers:
1. `FindByIndex(ctx, 1)` → `selectRecordBlockByIndex` DB query
2. `FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` → N paginated `selectTransactionsInTimestampRangeOrdered` DB queries
3. `updateOperationAccountAlias(...)` → up to `maxTransactionsInBlock × ops_per_tx` `GetAccountAlias` DB queries (cold cache on first request; partially warm thereafter)

With no rate limiting active (default config), the DB connection pool saturates under sustained parallel requests, degrading response times for all users.
