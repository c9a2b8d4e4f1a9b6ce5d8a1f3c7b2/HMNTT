### Title
Unbounded Concurrent Subscriptions Enable Database Exhaustion via gRPC `subscribeTopic`

### Summary
The `TopicMessageRetriever.retrieve()` interface is a cold publisher that spawns an independent database polling loop per subscription. No per-IP, per-client, or global subscription limit exists at the gRPC layer. The only guard — `maxConcurrentCallsPerConnection = 5` — is scoped to a single TCP connection and is trivially bypassed by opening multiple connections, allowing any unauthenticated client to multiply database load without bound.

### Finding Description
**Entry point** — `ConsensusController.subscribeTopic()` (`grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, lines 43–53): accepts any unauthenticated `ConsensusTopicQuery`, builds a `TopicMessageFilter`, and delegates to `TopicMessageServiceImpl.subscribeTopic()` with no caller identity check, no IP-based rate limit, and no subscription count gate.

**Service layer** — `TopicMessageServiceImpl.subscribeTopic()` (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 59–92): calls `topicMessageRetriever.retrieve(filter, true)` unconditionally. The `subscriberCount` `AtomicLong` (line 48) is wired only to a Micrometer gauge for observability; it is never checked against any ceiling before accepting a new subscription.

**Retriever** — `PollingTopicMessageRetriever.retrieve()` (`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 44–63): each call creates an independent `PollingContext` and a `Flux.defer(() -> poll(context))` that repeatedly calls `topicMessageRepository.findByFilter()` — every 2 s in throttled mode (page size 1 000) or every 20 ms in unthrottled mode (page size 5 000). There is no shared resource or semaphore limiting how many of these loops run concurrently.

**Only existing guard** — `GrpcConfiguration` (`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, line 33) sets `maxConcurrentCallsPerConnection = 5` (default, `NettyProperties` line 14). This is a Netty-level limit scoped to a **single TCP connection**. An attacker opens *M* TCP connections, each carrying 5 streams, yielding *M × 5* independent polling loops and *M × 5* concurrent database queries every polling interval.

**Failed assumption**: the design assumes a single connection per client and that 5 concurrent calls is a meaningful cap. Neither holds for an adversarial client.

### Impact Explanation
Each active subscription issues periodic `SELECT` queries against the `topic_message` table. With *N* concurrent subscriptions targeting a high-volume topic with `startTime = 0` (full history), the database receives *N* queries every 2 seconds, each potentially returning up to 1 000 rows. At a few hundred subscriptions the database connection pool is saturated, query latency spikes for all users, and the mirror node becomes unavailable — a full denial-of-service. No funds are directly at risk, but availability of the HCS mirror service is completely compromised.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge, and no on-chain resources. `grpcurl` or any gRPC client library suffices. Opening 200 TCP connections with 5 streams each (1 000 subscriptions) is achievable from a single machine with standard tooling. The attack is repeatable and can be sustained indefinitely because subscriptions with no `endTime` and no `limit` never self-terminate (`Flux.never()` path in `pastEndTime`, line 125).

### Recommendation
1. **Global subscription cap**: check `subscriberCount` against a configurable maximum before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when exceeded.
2. **Per-source-IP (or per-identity) subscription limit**: track active subscriptions keyed by the remote peer address obtained from the gRPC `Context`/interceptor, and reject new subscriptions that exceed a per-IP ceiling.
3. **gRPC server-side interceptor for rate limiting**: add a `ServerInterceptor` that applies token-bucket rate limiting (analogous to the existing `ThrottleConfiguration` in the web3 module) to `subscribeTopic` calls.
4. **Require `endTime` or `limit`**: reject open-ended subscriptions (`endTime = null` AND `limit = 0`) from unauthenticated callers, or enforce a maximum subscription lifetime.

### Proof of Concept
```bash
# Open 200 parallel TCP connections, each with 5 concurrent subscribeTopic streams
# targeting a high-volume topic from the beginning of time (startTime omitted → defaults to now,
# but setting consensusStartTime to epoch forces full-history retrieval)

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID":{"topicNum":41110},"consensusStartTime":{"seconds":0},"limit":0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Result: 1000 independent PollingTopicMessageRetriever loops each issuing
# SELECT queries against topic_message every 2 seconds → DB saturation
```
