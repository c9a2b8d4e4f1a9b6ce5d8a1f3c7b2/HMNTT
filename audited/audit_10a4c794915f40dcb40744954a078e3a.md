### Title
Missing `Vary: Accept-Encoding` Header in 304 Responses Enables Downstream Cache Poisoning

### Summary
In `responseCacheCheckHandler()`, when a client sends a request with a matching `if-none-match` ETag, the code enters the `clientCached` branch which only deletes `content-type` but never calls `negotiate()`. Since `negotiate()` is the sole place that sets `Vary: accept-encoding` on the response, the resulting 304 omits `Vary` entirely. This violates RFC 7234 §4.3.4, which mandates that 304 responses MUST include `Vary` if it would have been sent in a 200, enabling a downstream cache to strip `Vary` from its stored entry and subsequently serve a gzip-encoded body to clients that do not support gzip.

### Finding Description

**Exact code path:**

In `rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler()`:

```
lines 63-73:
  if (isHead || clientCached) {
    if (clientCached) {
      delete headers[contentTypeHeader];   // ← only action taken
    } else {
      negotiate(cachedResponse, req, res); // ← HEAD path calls negotiate
    }
  } else {
    const useCompressed = negotiate(cachedResponse, req, res); // ← normal GET calls negotiate
    ...
  }
```

`negotiate()` (lines 121-135) is the **only** function that calls `res.setHeader(VARY_HEADER, 'accept-encoding')` (line 122). It is never called when `clientCached === true`.

The `headers` object (lines 58-61) is built from `cachedResponse.headers`, which has `vary` explicitly stripped before storage in `responseCacheUpdateHandler` (line 112: `delete headers[VARY_HEADER]`). Therefore `res.set(headers)` at line 75 also does not include `vary`.

**Result:** A 304 response is emitted with `etag` and `cache-control` but without `vary: accept-encoding`.

This is confirmed as the current behavior by the integration test assertion at line 211:
```js
expect(response.headers).not.toHaveProperty('vary');
```

**RFC violation:** RFC 7234 §4.3.4 states:
> "The server generating a 304 response MUST generate any of the following header fields that would have been sent in a 200 (OK) response to the same request: Cache-Control, Content-Location, Date, ETag, Expires, and **Vary**."

**Downstream cache update rule (RFC 7234 §4.3.4):**
> "If a stored response is selected for update, the cache MUST use the header fields provided in the 304 response to replace the corresponding header fields in the stored response."

A compliant cache receiving a 304 without `Vary` will remove `Vary` from its stored entry. The stored entry then has no `Vary: Accept-Encoding`, so the cache treats the response as encoding-agnostic and serves it to all subsequent clients regardless of their `Accept-Encoding`.

### Impact Explanation

If a CDN or reverse proxy sits in front of the mirror node (a standard deployment pattern):

1. The proxy has a cached 200 response with `Vary: Accept-Encoding` and a gzip-encoded body.
2. An attacker sends `GET /api/v1/... + if-none-match: <valid-etag> + Accept-Encoding: gzip` through the proxy.
3. The origin returns 304 without `Vary`.
4. The proxy updates its stored entry, removing `Vary: Accept-Encoding`.
5. The proxy now serves the gzip-encoded body to all subsequent clients, including those that sent no `Accept-Encoding` or `Accept-Encoding: identity`.
6. Clients without gzip support receive garbled binary content instead of JSON, causing application-level failures for all users of that cached resource.

Severity: **Medium**. No data is leaked, but availability of API responses is degraded for all clients served by the poisoned cache entry, and the attacker can repeat the attack on any cached endpoint.

### Likelihood Explanation

Preconditions are minimal and require no privileges:
- The attacker only needs a valid ETag, obtainable by making any normal GET request to the same endpoint.
- The attack is a single HTTP request with two standard headers (`if-none-match`, `Accept-Encoding`).
- Any deployment with a CDN, load-balancer cache, or reverse proxy (nginx, Varnish, Cloudflare, etc.) is affected.
- The attack is repeatable: the attacker can re-poison the cache entry after it expires.

### Recommendation

In the `clientCached` branch, call `negotiate()` (or at minimum unconditionally emit `Vary: accept-encoding`) before sending the 304 response:

```js
if (isHead || clientCached) {
  if (clientCached) {
    delete headers[contentTypeHeader];
    res.setHeader(VARY_HEADER, 'accept-encoding'); // ← add this
  } else {
    negotiate(cachedResponse, req, res);
  }
}
```

Alternatively, refactor so `negotiate()` is always called and its side-effect of setting `Vary` is separated from the body/encoding logic. The integration test at line 211 (`not.toHaveProperty('vary')`) must also be updated to assert `vary: 'accept-encoding'` is present in 304 responses.

### Proof of Concept

```bash
# Step 1: Warm up the cache and capture the ETag
ETAG=$(curl -si "https://<mirror-node>/api/v1/accounts/0.0.12345/rewards" \
  | grep -i '^etag:' | awk '{print $2}' | tr -d '\r')

# Step 2: Send the poisoning request through the downstream cache
# (replace <proxy> with the CDN/proxy address)
curl -si "https://<proxy>/api/v1/accounts/0.0.12345/rewards" \
  -H "if-none-match: $ETAG" \
  -H "Accept-Encoding: gzip"
# → HTTP/1.1 304 Not Modified
# → cache-control: public, max-age=...
# → etag: <etag>
# → (no vary header)

# Step 3: Verify the proxy now serves gzip to a client that did not request it
curl -si "https://<proxy>/api/v1/accounts/0.0.12345/rewards" \
  -H "Accept-Encoding: identity"
# → HTTP/1.1 200 OK
# → content-encoding: gzip   ← wrong; client receives garbled binary
