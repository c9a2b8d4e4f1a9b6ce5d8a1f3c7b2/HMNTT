### Title
Compounded DB Load via Simultaneous `account.id` + `publickey` Filters in `GET /api/v1/tokens`

### Summary
An unprivileged external user can combine `account.id` and `publickey` query parameters in a single `GET /api/v1/tokens` request. This simultaneously triggers the `tokenAccountCte` path (an unbounded `SELECT *` CTE over `token_account`) and appends an `e.public_key` condition on the `entity` table, producing a more expensive compound query than either filter alone. No application-level rate limiting exists for this REST endpoint, enabling repeated abuse to amplify database load.

### Finding Description

**Code path — `getTokensRequest()`, `rest/tokens.js` lines 360–422:** [1](#0-0) 

When `account.id` is present, the function unconditionally prepends `tokenAccountCte` and appends `tokenAccountJoinQuery` to the SQL being built: [2](#0-1) 

`tokenAccountCte` is defined as an **unbounded** `SELECT *` with no `LIMIT`: [3](#0-2) 

The `entityIdJoinQuery` is always appended afterward: [4](#0-3) 

**Code path — `extractSqlFromTokenRequest()`, lines 164–214:**

`filterColumnMap` maps `publickey` → `e.public_key`: [5](#0-4) 

When `publickey` is present in the filters, line 201 appends `e.public_key = $N` as a WHERE condition: [6](#0-5) 

**Resulting combined SQL (confirmed by test at lines 322–348):** [7](#0-6) 

The CTE materializes **all** `token_account` rows for the given `account_id` (no LIMIT inside the CTE). In PostgreSQL, if the planner chooses to materialize the CTE (an optimization fence in PG ≤ 11, or when the planner deems it beneficial in PG ≥ 12), the full set of token associations is scanned before the outer `LIMIT` is applied. The `e.public_key` filter then requires evaluating `public_key` for every entity matched by the join.

**Root cause:** There is no guard preventing the simultaneous use of `account.id` and `publickey`. The `validateTokenQueryFilter` only validates individual parameter formats, not their combination: [8](#0-7) 

The only combination-exclusion logic in `getTokensRequest` is for `name` + `token.id` and `name` + `account.id`; no such check exists for `account.id` + `publickey`: [9](#0-8) 

### Impact Explanation
Each crafted request forces the database to: (1) scan and materialize all token associations for the target account via the unbounded CTE, (2) join to `token`, (3) join to `entity` on `e.id = t.token_id`, and (4) evaluate `e.public_key` for every matched row. For accounts with large numbers of token associations (e.g., treasury accounts), this is significantly more expensive than either filter alone. Flooding the endpoint with concurrent requests amplifies database CPU and I/O load, potentially degrading or denying service to all users of the mirror node.

### Likelihood Explanation
The endpoint is public and requires no authentication. The parameters are documented and accepted by `acceptedTokenParameters`. An attacker needs only a valid entity ID (trivially obtained from the network) and any 64-character hex string as the public key value. The attack is trivially scriptable and repeatable with standard HTTP tooling.

### Recommendation
1. **Disallow the combination** of `account.id` and `publickey` in `getTokensRequest()`, mirroring the existing guard for `name` + `account.id`, since the semantic intersection (tokens associated with account X whose admin key matches public key Y) is rarely useful and always expensive.
2. **Add a LIMIT inside `tokenAccountCte`** equal to the outer query's limit, so the CTE cannot materialize more rows than will be returned.
3. **Add application-level rate limiting** to the REST API for the `/api/v1/tokens` endpoint, analogous to the `ThrottleManagerImpl` used by the web3 API.

### Proof of Concept
```
# Step 1: Identify an account with many token associations (e.g., a known treasury)
# Step 2: Send the compound request
GET /api/v1/tokens?account.id=0.0.98&publickey=3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be

# Step 3: Flood concurrently
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/tokens?account.id=0.0.98&publickey=3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be" &
done
wait
```
Each request executes the compound CTE + `public_key` filter query. With 200 concurrent requests, the database processes 200 simultaneous unbounded `token_account` scans joined to `entity` with a non-selective `public_key` predicate, compounding CPU and I/O load.

const validateTokenQueryFilter = (param, op, val) => {
  let ret = false;

  if (op === undefined || val === undefined) {
    return ret;
  }

  // Validate operator
  if (!utils.isValidOperatorQuery(op)) {
    return ret;
  }

  // Validate the value
  switch (param) {
    case filterKeys.ACCOUNT_ID:
      ret = EntityId.isValidEntityId(val);
      break;
    case filterKeys.ENTITY_PUBLICKEY:
      // Acceptable forms: exactly 64 characters or +12 bytes (DER encoded)
      ret = utils.isValidPublicKeyQuery(val);
      break;
    case filterKeys.LIMIT:
      ret = utils.isPositiveLong(val);
      break;
    case filterKeys.NAME:
      ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
      break;
    case filterKeys.ORDER:
      // Acceptable words: asc or desc
      ret = utils.isValidValueIgnoreCase(val, Object.values(orderFilterValues));
      break;
    case filterKeys.SERIAL_NUMBER:
      ret = utils.isPositiveLong(val);
      break;
    case filterKeys.TOKEN_ID:
      ret = EntityId.isValidEntityId(val, true, EvmAddressType.NUM_ALIAS);
      break;
    case filterKeys.TOKEN_TYPE:
      ret = utils.isValidValueIgnoreCase(val, Object.values(tokenTypeFilter));
      break;
    case filterKeys.TIMESTAMP:
      ret = utils.isValidTimestampParam(val);
      break;
    default:
      // Every token parameter should be included here. Otherwise, it will not be accepted.
      break;
  }

  return ret;
};
