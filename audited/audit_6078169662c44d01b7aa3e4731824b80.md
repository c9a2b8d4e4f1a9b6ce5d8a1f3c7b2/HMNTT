### Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Dynamic Per-Contract Cache Creation in `findStorageBatch()`

### Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` triggers a call to `cacheManagerSlotsPerContract.getCache(contractId.toString())`, which causes Spring's `CaffeineCacheManager` to create a new `CaffeineCache` instance and store it permanently in its internal unbounded `ConcurrentHashMap`. The outer `contractSlots` Caffeine cache is bounded at `maximumSize=3000`, but its eviction does not remove entries from `cacheManagerSlotsPerContract`'s internal map, causing unbounded heap growth. An unauthenticated attacker can exploit this by issuing requests for arbitrarily many distinct contract addresses.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()`, lines 86–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [1](#0-0) 

**Root cause:**

`cacheManagerSlotsPerContract` is a `CaffeineCacheManager` bean configured with no fixed cache names (`setCacheNames` is never called): [2](#0-1) 

Because no fixed names are set, `CaffeineCacheManager.dynamic` is `true`. Every call to `getCache(name)` with a previously unseen name creates a new `CaffeineCache` and inserts it into the manager's internal `ConcurrentHashMap<String, Cache>` (`cacheMap`). This internal map has **no size bound and is never evicted**.

The outer `contractSlots` Caffeine cache is bounded at `maximumSize=3000`: [3](#0-2) 

When the outer cache evicts an entry (e.g., the oldest `contractId`), it removes the `contractId → CaffeineCache` mapping from its own Caffeine table. However, it does **not** call any cleanup on the evicted `CaffeineCache` value, and it does **not** remove the corresponding entry from `cacheManagerSlotsPerContract.cacheMap`. The `CaffeineCache` object remains strongly referenced by `cacheManagerSlotsPerContract` and is never garbage collected.

**Exploit flow:**

1. Attacker sends `eth_call` (or equivalent) requests targeting N distinct contract addresses, where N ≫ 3000.
2. Each request reaches `findStorage()` → `findStorageBatch()`.
3. For each new `contractId`, the loader `cacheManagerSlotsPerContract.getCache(contractId.toString())` is invoked, creating a new `CaffeineCache` (with its own Caffeine internal data structures sized for `maximumSize=1500`) and storing it in `cacheManagerSlotsPerContract.cacheMap`.
4. After 3001 requests: outer `contractSlots` cache holds 3000 entries; `cacheManagerSlotsPerContract.cacheMap` holds 3001 entries.
5. After N requests: outer cache stays at 3000; `cacheManagerSlotsPerContract.cacheMap` holds N entries — unbounded.

**Why existing checks fail:**

- `contractSlots` `maximumSize=3000` bounds only the outer Caffeine cache, not `cacheManagerSlotsPerContract.cacheMap`.
- `slotsPerContract` `maximumSize=1500` bounds the number of slot keys *within* each per-contract cache, not the number of per-contract cache instances. [4](#0-3) 

There is no eviction listener on the outer `contractSlots` cache that would call `cacheManagerSlotsPerContract` to remove the stale cache, and no periodic cleanup of `cacheManagerSlotsPerContract.cacheMap`.

### Impact Explanation

Each `CaffeineCache` instance backed by a Caffeine cache with `maximumSize=1500` allocates internal data structures (segment arrays, node arrays, reference queues, scheduler threads) on the order of tens to hundreds of KB per instance. With tens of thousands of distinct contractIds targeted, `cacheManagerSlotsPerContract.cacheMap` accumulates gigabytes of heap. This causes:

- Sustained heap pressure forcing frequent full GC cycles, degrading throughput for all users.
- Potential `OutOfMemoryError` crashing the JVM process entirely.
- The effect is **permanent** within a JVM lifetime — the map never shrinks.

This satisfies the "≥30% resource increase" threshold without brute force: a few thousand targeted requests are sufficient to fill the outer cache and begin accumulating leaked `CaffeineCache` objects.

### Likelihood Explanation

The Hedera mirror node's web3 module exposes standard Ethereum JSON-RPC endpoints (`eth_call`, `eth_estimateGas`) that are publicly accessible without authentication. Contract IDs on Hedera are sequential integers, making enumeration trivial. The attacker needs only to issue `eth_call` requests to thousands of distinct contract addresses — a script sending HTTP POST requests in a loop. The attack is:

- **No privilege required** — public RPC endpoint.
- **Low cost** — each request is a single HTTP call; no on-chain transaction or gas needed.
- **Persistent** — leaked `CaffeineCache` objects accumulate for the lifetime of the JVM.
- **Repeatable** — after a node restart, the attack can be re-executed immediately.

### Recommendation

1. **Add an eviction listener** to the outer `contractSlots` Caffeine cache that removes the corresponding entry from `cacheManagerSlotsPerContract` when a `contractId` is evicted:
   ```java
   Caffeine.newBuilder()
       .maximumSize(3000)
       .removalListener((contractId, cache, cause) ->
           // invalidate/remove from cacheManagerSlotsPerContract
       )
       ...
   ```

2. **Alternatively, eliminate the nested cache manager pattern entirely.** Use a single two-level key `(contractId, slotKey)` in one bounded Caffeine cache instead of a `CacheManager` that dynamically creates per-contract caches.

3. **Or replace `CaffeineCacheManager` for `slotsPerContract`** with a direct `com.github.benmanes.caffeine.cache.Cache<String, CaffeineCache>` with a bounded `maximumSize`, so eviction is handled natively without the unbounded `ConcurrentHashMap`.

### Proof of Concept

```python
import requests, json

RPC_URL = "http://<mirror-node-host>:8545"

def eth_call(contract_address):
    payload = {
        "jsonrpc": "2.0",
        "method": "eth_call",
        "params": [{"to": contract_address, "data": "0x"}, "latest"],
        "id": 1
    }
    requests.post(RPC_URL, json=payload, timeout=5)

# Hedera contract IDs map to EVM addresses 0x000...0<contractId>
# Target 10,000 distinct contract IDs
for contract_num in range(1, 10001):
    addr = f"0x{contract_num:040x}"
    eth_call(addr)
    # Each call leaks one CaffeineCache into cacheManagerSlotsPerContract.cacheMap
    # After 3001 calls, outer contractSlots cache is full and evicting,
    # but cacheManagerSlotsPerContract.cacheMap keeps growing

# Monitor JVM heap: heap usage grows linearly with contract_num
# jcmd <pid> VM.native_memory or jmap -heap <pid>
```

After running this script, heap profiling (e.g., via `jmap -histo`) will show an ever-increasing count of `com.github.benmanes.caffeine.cache.BoundedLocalCache` instances corresponding to the leaked per-contract caches, with no corresponding decrease as the outer cache evicts entries.
