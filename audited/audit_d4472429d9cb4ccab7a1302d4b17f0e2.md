### Title
Plaintext gRPC Server with TLS Disabled by Default Enables Network-Level Interception of Mirror Node Responses

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only concurrency and executor settings â€” no TLS context is ever applied. Combined with the Helm chart default of `ingress.tls.enabled: false`, the gRPC server runs entirely in plaintext in the default deployment. A network-positioned attacker can intercept and tamper with gRPC responses including address book entries, fee estimates, and topic messages.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28â€“35:

```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

No call to `serverBuilder.useTransportSecurity(sslContext)` or any equivalent is present. `NettyProperties` (`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`) contains only `maxConcurrentCallsPerConnection` â€” there is no TLS field, no certificate path, no key path. `GrpcProperties` similarly has no TLS configuration.

**Default Helm chart (`charts/hedera-mirror-grpc/values.yaml`, lines 142â€“144):**

```yaml
tls:
  enabled: false
  secretName: ""
```

Ingress-level TLS is also disabled by default. The GrpcRoute template (`charts/hedera-mirror-grpc/templates/grpcroute.yaml`) routes directly to port 5600 with no TLS backend annotation.

**Docker Compose (`docker-compose.yml`, lines 200â€“229):** The nginx proxy listens on plain port 8080 and forwards to `grpc://grpc_host` (plaintext). No TLS termination exists at any layer.

**Official documentation (`docs/grpc/README.md`, lines 16, 26, 37 and `charts/marketplace/gcp/README.md`, line 240):** All example invocations use `grpcurl -plaintext`, confirming plaintext is the expected and documented mode.

**Root cause:** The `grpcServerConfigurer()` customizer is the sole place where `NettyServerBuilder` is configured for this service. Because it never calls `useTransportSecurity()`, the Netty gRPC server unconditionally binds in plaintext mode. There is no fallback, no property to enable TLS at the application layer, and the infrastructure layer defaults to TLS-off.

**Why existing checks fail:** The architecture assumes TLS termination at the Traefik ingress layer, but the default value for `ingress.tls.enabled` is `false`. Even when Traefik TLS is enabled, traffic between the ingress pod and the gRPC pod traverses the cluster network in plaintext â€” any attacker with intra-cluster network access (compromised pod, CNI misconfiguration, node-level access) can intercept this leg.

### Impact Explanation

The mirror node gRPC API exposes three services over this plaintext channel:

- **`NetworkService/getNodes`** â€” returns address book entries (consensus node IPs, ports, TLS certificate hashes). A MITM can replace these with attacker-controlled node endpoints, redirecting clients to submit transactions to malicious consensus nodes.
- **`NetworkService/getFeeEstimate`** â€” returns fee schedules. Tampered responses can cause clients to systematically overpay or construct invalid transactions.
- **`ConsensusService/subscribeTopic`** â€” streams HCS topic messages. Tampered responses deliver false consensus data to subscribers, undermining the integrity guarantee that is the core value proposition of the mirror node.

The address book manipulation vector is the highest-severity consequence: clients that use `getNodes` to discover consensus nodes and then submit signed transactions will be directed to attacker infrastructure.

### Likelihood Explanation

**Preconditions required:**
- Network-level access between client and server (for external MITM) OR intra-cluster network access (for internal MITM on the ingressâ†’pod leg).
- No special credentials or authentication needed â€” the gRPC API is unauthenticated and publicly exposed by design.

**Feasibility:**
- In the default deployment (TLS disabled at ingress, plaintext at application), any on-path attacker (ISP, cloud provider network, ARP poisoning on shared network, compromised Kubernetes node) can perform this attack with standard tools (`mitmproxy`, `grpc-mitm`).
- The GCP Marketplace README explicitly documents `grpcurl -plaintext` against a public load balancer IP, confirming external plaintext exposure is the documented default.
- Intra-cluster MITM is achievable by any compromised pod in the same namespace or with sufficient RBAC to manipulate network policies.

Repeatability is high â€” the attack is passive and leaves no application-layer trace.

### Recommendation

1. **Application layer (primary fix):** Add TLS configuration to `NettyProperties` (certificate path, private key path) and configure it in `grpcServerConfigurer()`:
   ```java
   if (nettyProperties.getTlsCertPath() != null) {
       serverBuilder.useTransportSecurity(
           new File(nettyProperties.getTlsCertPath()),
           new File(nettyProperties.getTlsKeyPath()));
   }
   ```
2. **Helm chart:** Change `ingress.tls.enabled` default to `true` and require a `secretName`.
3. **Intra-cluster:** Enable mutual TLS (mTLS) via a service mesh (e.g., Istio, Linkerd) to encrypt the ingressâ†’pod leg.
4. **Documentation:** Remove `-plaintext` from all public-facing examples; document TLS as the required mode.

### Proof of Concept

**Prerequisites:** Network path to the gRPC server (default port 5600) or to the ingress (default port 443 with TLS disabled â†’ effectively port 80/8080).

**Steps:**

1. Confirm plaintext operation:
   ```bash
   grpcurl -plaintext <GRPC_IP>:5600 list
   # Returns service list without TLS â€” confirms plaintext
   ```

2. Set up a MITM proxy (e.g., using `mitmproxy` with gRPC support or `grpc-mitm`):
   ```bash
   # Redirect traffic via ARP poisoning or DNS manipulation
   # Intercept on attacker machine at port 5600
   mitmproxy --mode transparent --tcp-hosts '.*' -p 5600
   ```

3. Trigger a victim client to call `getNodes`:
   ```bash
   grpcurl -plaintext <ATTACKER_IP>:5600 \
     -d '{"file_id": {"fileNum": 102}}' \
     com.hedera.mirror.api.proto.NetworkService/getNodes
   ```

4. In the MITM proxy, intercept the response and replace node IP addresses with attacker-controlled endpoints.

5. The victim client receives tampered address book entries and subsequently submits transactions to attacker-controlled consensus nodes.

**Result:** Attacker observes all gRPC response payloads in cleartext and can modify any field in transit with no cryptographic protection preventing detection or rejection by the client.
