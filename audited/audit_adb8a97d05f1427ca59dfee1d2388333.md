### Title
Unbounded DB Query Amplification via EXTCODESIZE Loop in eth_call — Single Unauthenticated Request Triggers Thousands of `findRuntimeBytecode()` Queries

### Summary
An unprivileged user can submit a single `eth_call` containing EVM bytecode that loops `EXTCODESIZE` over thousands of unique contract addresses. Each unique address bypasses both the per-request in-memory cache and the cross-request Caffeine cache, causing `ContractBytecodeReadableKVState.readFromDataSource()` to invoke `ContractRepository.findRuntimeBytecode()` — and thus a live SQL query — for every unique address. With a 15 M gas limit and `EXTCODESIZE` costing 700 gas, a single HTTP request can generate up to ~21,000 database queries, far exceeding the 30 % resource-increase threshold.

### Finding Description

**Exact code path:**

1. `ContractController.call()` → `contractExecutionService.processCall()` → EVM execution [1](#0-0) 

2. Each `EXTCODESIZE` opcode executed by the EVM calls `ReadableKVStateBase.get()`: [2](#0-1) 

   The per-request cache (`getReadCache()`) is a `ConcurrentHashMap` stored in `ContractCallContext` — it is **scoped to a single request** and is empty at the start of every `eth_call`. For each unique `ContractID` key not yet seen in this request, `readFromDataSource()` is called. [3](#0-2) 

3. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode()`: [4](#0-3) 

4. `findRuntimeBytecode()` is backed by a Caffeine cache with `maximumSize=1000`: [5](#0-4) [6](#0-5) 

   For any address not already in this 1,000-entry cache, the annotated method executes the SQL: `SELECT runtime_bytecode FROM contract WHERE id = :contractId`.

**Root cause:** There is no limit on the number of distinct DB queries a single `eth_call` execution can generate. The gas limit bounds EVM opcode count, not DB round-trips. The two-layer cache (per-request `ConcurrentHashMap` + cross-request Caffeine) only deduplicates repeated lookups for the **same** address; it provides no protection when the attacker uses a large set of **distinct** addresses.

**Why `unless = "#result == null"` does not help:** The return type is `Optional<byte[]>`. `Optional.empty()` is not `null`, so the `unless` condition evaluates to `false` and the empty result is cached — but only up to the 1,000-entry limit. Once the cache is full, Caffeine evicts entries and subsequent lookups for evicted addresses hit the DB again.

### Impact Explanation

With `maxGasLimit = 15,000,000` and `EXTCODESIZE` costing 700 gas (EIP-1884), a single request can issue up to ~21,000 `EXTCODESIZE` calls against unique addresses, generating up to ~21,000 SQL queries. The gas throttle (`gasPerSecond = 7,500,000,000`) does not prevent this: 15 M gas is only 0.2 % of the per-second gas budget, so the throttle does not reject the request. The 10-second request timeout (`requestTimeout = 10000`) is the only practical upper bound on execution, but 21,000 simple indexed lookups easily fit within it. This constitutes a severe DB-layer amplification attack (1 HTTP request → up to 21,000 DB queries) that can exhaust database connection pools and degrade service for all users. [7](#0-6) [8](#0-7) 

### Likelihood Explanation

No authentication, no special account, and no on-chain state is required. The attacker only needs access to the public `POST /api/v1/contracts/call` endpoint. The attack is trivially repeatable: the attacker can send one request per second (limited by the gas bucket refill), each generating thousands of DB queries. Sequential long-zero addresses (e.g., `0x0000…0001` through `0x0000…5265`) are valid inputs and are not filtered. The attack is stateless and requires no prior contract deployment. [9](#0-8) 

### Recommendation

1. **Add a per-request DB query counter** in `ContractCallContext` and enforce a hard cap (e.g., 500 unique bytecode lookups per `eth_call`). Reject or short-circuit execution when the cap is exceeded.
2. **Increase the Caffeine cache size** for `contract` from 1,000 to a value that covers realistic address-space usage (e.g., 50,000), reducing cold-miss amplification.
3. **Cache negative results explicitly**: Ensure `Optional.empty()` results are cached with a short TTL (e.g., 30 s) so repeated lookups for non-existent addresses do not hit the DB. Verify Spring Cache's handling of `Optional` return types with the `unless` condition.
4. **Apply a per-request gas-to-DB-query ratio limit**: If gas consumed per DB query falls below a threshold, treat the request as abusive and terminate it.

### Proof of Concept

**Preconditions:** Public access to `POST /api/v1/contracts/call`. No account or contract needed.

**Attacker-crafted EVM bytecode (Solidity pseudocode):**
```solidity
// Inline assembly loop: EXTCODESIZE on 10,000 sequential addresses
for (uint160 i = 1; i <= 10000; i++) {
    assembly { pop(extcodesize(i)) }
}
```

**Compiled to init bytecode and submitted:**
```json
POST /api/v1/contracts/call
{
  "data": "<init_bytecode_of_loop_contract>",
  "gas": 15000000
}
```

**Trigger:** The EVM executes ~10,000 `EXTCODESIZE` opcodes against unique addresses `0x0000…0001` through `0x0000…2710`. Each address is absent from the 1,000-entry Caffeine cache (or evicts earlier entries once the cache is full), causing `findRuntimeBytecode()` to execute a SQL query for each one.

**Result:** A single HTTP request generates ~10,000 SQL queries (`SELECT runtime_bytecode FROM contract WHERE id = ?`), measurably increasing database CPU and I/O. Repeating this request once per second (within the gas throttle budget) sustains the elevated load indefinitely. [5](#0-4) [2](#0-1) [4](#0-3) [6](#0-5)
