### Title
RPS Token Not Restored on Gas-Check Failure Enables Sustained DoS via Rate-Limit Starvation

### Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is consumed from `rateLimitBucket` before the gas-limit check runs. When the gas check fails, a `ThrottleException` is thrown but the already-consumed RPS token is never restored — `restore()` only adds tokens back to `gasLimitBucket`, and `ContractController` only calls `restore()` on `InvalidParametersException`, not `ThrottleException`. An unprivileged attacker can exploit this to exhaust the gas bucket and then continuously drain the shared RPS bucket, starving all legitimate callers.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.java`, `throttle()`, lines 38–42: [1](#0-0) 

```java
if (!rateLimitBucket.tryConsume(1)) {           // (A) RPS token consumed here
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(...)) {   // (B) gas check fails
    throw new ThrottleException(...);            // (C) RPS token from (A) is NEVER restored
}
```

`restore()` only touches `gasLimitBucket`: [2](#0-1) 

`ContractController.call()` only catches `InvalidParametersException` and calls `restore()` — `ThrottleException` propagates uncaught with no restoration: [3](#0-2) 

**Root cause:** The two-bucket check is not atomic and has no rollback. The failed assumption is that a gas-check failure is a "free" rejection — in reality it silently burns an RPS slot.

**Exploit flow:**

1. Attacker sends requests with `gas = 15,000,000`. `scaleGas(15_000_000) = 1,500` tokens per request. [4](#0-3) 

2. Gas bucket capacity = `scaleGas(7_500_000_000) = 750,000` tokens. At 500 RPS (the default limit), 500 × 1,500 = 750,000 tokens — the gas bucket is fully exhausted in exactly one second. [5](#0-4) 

3. Gas bucket refills at 750,000 tokens/second. Attacker sustains 500 req/s × 1,500 tokens = 750,000 tokens/second consumed — exactly matching the refill rate, keeping the gas bucket perpetually empty.

4. Every one of those 500 requests/second passes the RPS check (consuming 1 RPS token), fails the gas check, and the RPS token is discarded. The shared `rateLimitBucket` (capacity 500) is fully drained by the attacker alone. [6](#0-5) 

5. Legitimate users receive `429 Too Many Requests` on the RPS check — they never even reach the gas check.

**Why existing checks are insufficient:**

- `rateLimitBucket` is a single global shared bucket with no per-IP partitioning.
- `restore()` has no path to return RPS tokens.
- The `catch (InvalidParametersException)` block in the controller is the only restoration site, and `ThrottleException` is a different exception hierarchy — it is never caught there. [7](#0-6) 

### Impact Explanation
The `rateLimitBucket` is a process-wide singleton shared across all callers. A single unauthenticated attacker can monopolize all 500 RPS slots per second indefinitely, reducing legitimate throughput to zero on any mirror-node instance running this code. Because mirror-node deployments typically run as horizontally scaled replicas behind a load balancer, and each replica has its own independent in-process bucket (no distributed coordination), the attack must be replicated per instance — but it is trivially parallelizable. The result is complete API unavailability for `POST /api/v1/contracts/call` on targeted nodes.

### Likelihood Explanation
- **No authentication required.** Any HTTP client can reach the endpoint.
- **No special knowledge required.** The attacker only needs to know the public API path and set a high `gas` field.
- **Sustainable with a single connection.** 500 req/s from one host is achievable with standard HTTP tooling (e.g., `wrk`, `hey`, `ab`).
- **Repeatable indefinitely.** The attack loop is self-sustaining once the gas bucket is exhausted.

### Recommendation
1. **Restore the RPS token on gas-check failure.** Add `rateLimitBucket.addTokens(1)` before throwing `ThrottleException` when the gas check fails, or restructure the check to be atomic (check both buckets before consuming either).
2. **Alternatively, check gas availability before consuming the RPS token** using `rateLimitBucket.getAvailableTokens()` / `gasLimitBucket.getAvailableTokens()` as a pre-flight, then consume both atomically.
3. **Add per-source-IP rate limiting** as a defense-in-depth layer so a single client cannot monopolize the global bucket.
4. **Extend the `catch` block in `ContractController`** (or add a `finally` block) to restore the RPS token whenever `throttle()` throws for a gas-limit reason.

### Proof of Concept
```bash
# Step 1: Exhaust the gas bucket (runs for ~1 second)
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000}' &
done
wait

# Step 2: Sustain the attack — keep draining both buckets
while true; do
  for i in $(seq 1 500); do
    curl -s -o /dev/null -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000}' &
  done
  wait
done

# Verification: from a separate client, observe all responses are 429
curl -v -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21001}'
# Expected: HTTP 429 "Requests per second rate limit exceeded"
```
