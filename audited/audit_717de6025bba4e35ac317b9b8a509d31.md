### Title
Unauthenticated Unbounded Subscription Flood in `retrieve()` Enables DB Exhaustion and Denial of Service to Legitimate Topic Subscribers

### Summary
`PollingTopicMessageRetriever.retrieve()` accepts any `TopicMessageFilter` from any caller with zero authentication or authorization checks. Because there is no per-IP connection limit and no global subscriber cap, an unprivileged attacker can open an unbounded number of gRPC connections and flood the server with concurrent subscriptions to any topicId, including high-traffic system topics. Each subscription drives repeated DB polling queries, exhausting the database connection pool and starving legitimate subscribers of timely topic message delivery.

### Finding Description

**Entry point — no auth at the controller:**
`ConsensusController.subscribeTopic()` (lines 43–53) converts the raw protobuf request directly into a `TopicMessageFilter` and calls `topicMessageService.subscribeTopic()` with no authentication interceptor, no token check, and no caller identity validation. [1](#0-0) 

The production interceptor directory (`grpc/src/main/java/org/hiero/mirror/grpc/interceptor/`) is **empty** — the only `ServerInterceptor` in the codebase lives under the test tree and only sets an endpoint-context tag for metrics; it performs no auth. [2](#0-1) 

**Service layer — only existence check, no access control:**
`TopicMessageServiceImpl.subscribeTopic()` calls `topicExists()`, which only verifies the entity is present in the DB and has type `TOPIC`. There is no check on who is allowed to subscribe, no per-caller quota, and no subscriber cap — `subscriberCount` is a Micrometer gauge only, never enforced as a ceiling. [3](#0-2) 

**`retrieve()` — direct DB polling with no caller checks:**
`retrieve()` immediately constructs a `PollingContext` and begins issuing `topicMessageRepository.findByFilter()` calls on a `boundedElastic` scheduler. For the throttled path (used for historical retrieval), it polls every 2 s with pages up to 1 000 rows; for the unthrottled path it polls every 20 ms with pages up to 5 000 rows. Neither path checks caller identity or enforces a per-caller or per-topic subscription limit. [4](#0-3) [5](#0-4) 

**The only connection-level mitigation is trivially bypassed:**
`GrpcConfiguration` sets `maxConcurrentCallsPerConnection = 5` (default). This limits calls *per TCP connection* but imposes no cap on the number of connections from a single IP or in total. An attacker opens N connections and immediately has 5N concurrent subscriptions. [6](#0-5) [7](#0-6) 

**`TopicMessageFilter` validation — structural only:**
The filter validates non-null `topicId`, non-negative `limit`, and time-range sanity. It enforces no access policy. [8](#0-7) 

### Impact Explanation

Each attacker-controlled subscription issues a DB query every 2 seconds (throttled) or every 20 ms (unthrottled phase). With hundreds of concurrent subscriptions targeting a high-traffic topic, the DB connection pool is saturated with `SELECT` queries for that topic. Legitimate subscribers' queries queue behind the attacker's, causing message delivery latency to spike or connections to time out. Because the mirror node's gRPC service is the canonical delivery path for HCS topic messages (gossip output), this constitutes a denial-of-service against legitimate subscribers of those topics. The `timeout = 60 s` applies only when *no messages arrive*; on a high-traffic topic it never fires, so attacker subscriptions persist indefinitely. [9](#0-8) 

### Likelihood Explanation

The gRPC port (5600) is publicly documented and reachable without credentials. The attack requires only a standard gRPC client (e.g., `grpcurl`) and knowledge of any valid `topicId` — both are trivially obtained. No exploit code, no special privileges, and no prior account are needed. The attack is repeatable and automatable. The only operational barrier is network-level firewall rules, which are not enforced by the application itself. [10](#0-9) 

### Recommendation

1. **Add a global subscriber cap**: Enforce a configurable maximum on `subscriberCount` in `TopicMessageServiceImpl`; reject new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Add per-IP connection rate limiting**: Implement a gRPC `ServerInterceptor` (production, not test) that tracks active subscriptions per remote IP and rejects excess connections.
3. **Add per-topic subscriber limits**: Reject subscriptions to a topic when the number of active subscribers for that topic exceeds a configurable threshold.
4. **Introduce authentication/authorization**: Require a bearer token or mTLS for `subscribeTopic` calls, especially for system/high-traffic topics.
5. **Reduce unthrottled polling aggressiveness**: The 20 ms / 5 000-row unthrottled path amplifies the attack; gate it behind authentication or remove it for unauthenticated callers.

### Proof of Concept

```bash
# Step 1: Identify a high-traffic topic (e.g., topicNum 41110 from docs)
# Step 2: Open 200 parallel subscriptions with no limit (runs indefinitely on active topic)
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 3: Observe DB connection pool exhaustion via metrics or logs
# Step 4: Legitimate subscriber now experiences delayed or dropped messages
```

No credentials, no special tooling, and no prior account are required. Each background process holds an open streaming RPC that drives continuous `findByFilter` DB queries, saturating the pool.

    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }

public class TopicMessageFilter {

    private static final SecureRandom RANDOM = new SecureRandom();

    private Long endTime;

    @Min(0)
    private long limit;

    @Min(0)
    @NotNull
    @Builder.Default
    private long startTime = DomainUtils.now();

    @Builder.Default
    private String subscriberId = RandomStringUtils.random(8, 0, 0, true, true, null, RANDOM);

    @NotNull
    private EntityId topicId;

    public boolean hasLimit() {
        return limit > 0;
    }

    @AssertTrue(message = "End time must be after start time")
    public boolean isValidEndTime() {
        return endTime == null || endTime > startTime;
    }

    @AssertTrue(message = "Start time must be before the current time")
    public boolean isValidStartTime() {
        return startTime <= DomainUtils.now();
    }
