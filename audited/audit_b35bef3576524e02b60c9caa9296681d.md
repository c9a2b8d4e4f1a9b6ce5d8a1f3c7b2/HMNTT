### Title
Unbounded `spenderIdInFilters` Replication Across Three-Subquery UNION ALL Path Enables 3x Database Work Amplification

### Summary
In `rest/service/nftService.js`, the `getQuery()` function passes the full `spenderIdInFilters` list (populated from all `spender.id=eq:X` query parameters) into every subquery generated by the three-branch UNION ALL path. When an unprivileged user supplies token.id and serial number range bounds that activate all three subqueries (`lower`, `inner`, `upper`), the database executes three independent full-scan queries each containing the same large `spender IN (...)` clause, followed by an outer merge-sort and limit. This structurally multiplies per-request database work by approximately 3× with no additional guard beyond the general `maxRepeatedQueryParameters` cap.

### Finding Description

**Exact code path:**

`rest/service/nftService.js`, `getQuery()`, lines 93–105: [1](#0-0) 

The three filter arrays `[lower, inner, upper]` are each passed to `getSubQuery()` along with the **same** `spenderIdInFilters` reference: [2](#0-1) 

Inside `getSubQuery()`, every entry in `spenderIdInFilters` is appended to `params` and rendered into a `spender IN ($n, ...)` clause: [3](#0-2) 

When all three subqueries are non-empty, the outer UNION ALL wrapper is assembled at line 122: [4](#0-3) 

**Root cause:** `spenderIdInFilters` is replicated verbatim into every subquery. There is no per-subquery deduplication, no cap on how many `eq` spender filters may be combined with the three-subquery path, and no check that the combined query cost is bounded.

**Triggering the three-subquery path:** In `accountController.js`, `extractNftMultiUnionQuery()` populates `lower`, `inner`, and `upper` when the user provides both a lower and upper bound on `token.id` together with serial number bounds: [5](#0-4) 

`validateFilters()` only rejects multiple range operators (`gt`/`gte`/`lt`/`lte`) for `spender.id`; it places **no limit** on the count of `eq` operators that populate `spenderIdInFilters`: [6](#0-5) 

**Only guard — `maxRepeatedQueryParameters`:** `buildFilters()` in `utils.js` rejects a single query-parameter key whose value array exceeds `config.query.maxRepeatedQueryParameters`: [7](#0-6) 

This cap is a global configuration value (default is not 4; the test that uses 4 sets it manually). It is not specific to the NFT endpoint and does not account for the 3× amplification introduced by the UNION ALL path.

### Impact Explanation

Each of the three subqueries performs an independent sequential scan of the `nft` table filtered by `account_id` and the full `spender IN (...)` list, then sorts and limits. The outer query re-sorts and limits the merged result. Total database work scales as:

```
3 × (scan cost with IN evaluation over N spender IDs) + outer sort(3 × limit rows)
```

versus a single-subquery request:

```
1 × (scan cost with IN evaluation over N spender IDs)
```

This is a structural 3× amplification achievable with a single HTTP request, well above the 30% threshold. Because the `nft` table can be large and the `LEFT JOIN entity` is included in every subquery, the absolute cost per request can be significant. The attack is repeatable at will with no authentication.

### Likelihood Explanation

The endpoint `/api/v1/accounts/{id}/nfts` is public and unauthenticated. The required parameter combination (`token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` plus multiple `spender.id=eq:Z`) is documented in the OpenAPI spec: [8](#0-7) 

No special knowledge, credentials, or brute-force is required. A single crafted request is sufficient. The attack can be sustained at the rate limit of the HTTP layer (if any), and each request independently causes 3× the normal database load.

### Recommendation

1. **Cap `spenderIdInFilters` size independently** of `maxRepeatedQueryParameters`, with a small limit (e.g., ≤ 10) enforced in `validateFilters()` or `extractNftMultiUnionQuery()`.
2. **Detect the three-subquery path** and reduce the allowed `spenderIdInFilters` count further when `lower`, `inner`, and `upper` are all non-empty.
3. **Consider collapsing** the `spenderIdInFilters` into the outer query rather than replicating it into each subquery, so the IN evaluation occurs once on the merged result set.
4. Apply a per-endpoint rate limit on the `/accounts/{id}/nfts` route.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1001/nfts?token.id=gte:1&token.id=lte:9999999&serialnumber=gte:1&serialnumber=lte:9999999&spender.id=100&spender.id=200&spender.id=300&...&spender.id=<maxRepeatedQueryParameters>&limit=100
```

**Steps:**
1. Send the above request to a live mirror node with a large `nft` table.
2. Observe that the generated SQL contains three `UNION ALL` branches, each with `spender IN ($5, ..., $N)`.
3. Measure database CPU/IO for this request vs. a baseline request without the range bounds (single subquery path).
4. Repeat the request in a loop; observe cumulative database resource consumption exceeding 30% above the 24-hour baseline within a short window.
