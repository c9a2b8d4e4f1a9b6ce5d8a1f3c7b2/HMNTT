### Title
Unauthenticated `X-Real-IP` Header Spoofing Enables Unbounded Log Inflation in `TracingMiddleware`

### Summary
`TracingMiddleware()` unconditionally trusts the `X-Real-IP` (and `X-Forwarded-For`) HTTP header supplied by any client, embedding its raw value into every log line at `Info` level with no validation of source, format, or length. An unprivileged attacker can flood the service with requests carrying unique, arbitrarily long fake IP strings, generating an unbounded stream of distinct log entries that defeat IP-based log deduplication and inflate log storage costs linearly with attack volume.

### Finding Description
**Code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()` (lines 63–74) and `TracingMiddleware()` (lines 43–61).

```
getClientIpAddress():
  line 64: ipAddress := r.Header.Get(xRealIpHeader)       // blindly read from client
  line 67: ipAddress = r.Header.Get(xForwardedForHeader)  // fallback, also client-controlled
  line 71: ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only used if both headers absent

TracingMiddleware():
  line 52-53: message := fmt.Sprintf("%s %s %s (%d) in %s",
                  clientIpAddress, ...)                    // attacker string embedded verbatim
  line 58: log.Info(message)                              // written to log unconditionally
```

**Root cause:** The function performs no check that the request arrived from a trusted reverse proxy before accepting `X-Real-IP`/`X-Forwarded-For` as authoritative. Any external client can set these headers to any arbitrary string (not even a valid IP address). There is no length cap, no format validation, and no allowlist of trusted upstream addresses.

**Exploit flow:**
1. Attacker sends HTTP POST requests directly to the Rosetta endpoint (or through any path that reaches the Go process).
2. Each request carries a unique `X-Real-IP` value, e.g., a random 39-character string or a valid-looking but unique IP.
3. `getClientIpAddress()` returns the attacker-supplied string verbatim.
4. `TracingMiddleware()` formats and emits a `log.Info` line containing that unique string.
5. Repeated at scale: every request produces a distinct log line with a distinct "IP", defeating any deduplication or aggregation that groups by source IP.

**Why existing checks are insufficient:**

The Traefik middleware chain in `charts/hedera-mirror-rosetta/values.yaml` (lines 157–160) defines a rate limit of `average: 10` keyed on `requestHost: true` — i.e., per hostname, not per IP. A single attacker host is capped at ~10 req/s, but:
- `global.middleware` defaults to `false` (line 95), so the entire Traefik chain is **opt-in and may not be active**.
- Even when active, 10 req/s × 60 s = 600 unique log lines/minute, 864,000/day — still significant log inflation.
- The `inFlightReq` limit (line 152–156) uses `ipStrategy.depth: 1` against `X-Forwarded-For` at the Traefik layer, not the application layer, and does not prevent log writes for requests that do pass through.
- No application-level guard exists in the Go code itself.

### Impact Explanation
Every request causes one `log.Info` write containing the attacker-controlled string. With unique values per request, log aggregation pipelines (Loki, Elasticsearch, Splunk, etc.) that deduplicate or index by IP field store every entry separately. At sustained 10 req/s (the Traefik cap, when enabled), log volume grows by ~864,000 entries/day from a single attacker host. With multiple attacker hosts or without the Traefik middleware, the rate is unbounded. Additionally, since the header value is not length-validated, each log line can be made arbitrarily large (HTTP header size limits permitting, typically 8 KB), multiplying storage impact further. Disk exhaustion on the logging backend is a realistic outcome in unmonitored deployments.

### Likelihood Explanation
No authentication or special privilege is required. Any internet-accessible Rosetta endpoint is vulnerable. Setting an HTTP header is trivial with `curl`, `ab`, `wrk`, or any scripting language. The attack is repeatable, stateless, and requires no knowledge of the application beyond the public API surface. The only friction is the optional Traefik rate limit, which is disabled by default.

### Recommendation
1. **Validate proxy trust before accepting forwarded headers:** Only read `X-Real-IP`/`X-Forwarded-For` if `r.RemoteAddr` matches a configured allowlist of trusted proxy CIDRs. Otherwise fall back to `r.RemoteAddr`.
2. **Validate the extracted value is a well-formed IP address:** Use `net.ParseIP()` on the extracted string; discard and fall back if it returns `nil`.
3. **Cap header value length** before processing to prevent oversized log lines.
4. **Enable the Traefik middleware chain by default** (`global.middleware: true`) so rate limiting is active in all deployments.

### Proof of Concept
```bash
# Send 1000 requests each with a unique fake X-Real-IP, observe 1000 distinct Info log lines
for i in $(seq 1 1000); do
  curl -s -o /dev/null \
    -H "X-Real-IP: 10.$(( RANDOM % 256 )).$(( RANDOM % 256 )).$i" \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
    http://<rosetta-host>/network/status &
done
wait

# Each log line will contain a distinct IP string, e.g.:
# INFO 10.142.87.1 POST /network/status (200) in 12ms
# INFO 10.93.201.2 POST /network/status (200) in 11ms
# ...
# Log deduplication by IP field produces 1000 separate buckets instead of 1.
```
