### Title
Unauthenticated Hash-Lookup DoS via Unbounded Block Endpoint Requests with No Rate Limiting or 404 Caching

### Summary
The `GET /api/v1/blocks/:hashOrNumber` endpoint accepts any syntactically valid 32-byte or 48-byte hex hash from any unauthenticated caller, immediately issues a `LIKE`-prefixed database query against the `record_file` table, and returns a 404 when no row is found. Because the REST API applies no rate limiting to this endpoint and the response cache explicitly skips non-2xx responses, an attacker can sustain an arbitrarily high rate of unique-hash requests that each hit the database, exhausting connection pool capacity and degrading query throughput for all users.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:13` → `BlockController.getByHashOrNumber` (`rest/controllers/blockController.js:114-124`) → `validateHashOrNumber` (`rest/controllers/blockController.js:28-38`) → `RecordFileService.getByHashOrNumber` (`rest/service/recordFileService.js:164-179`)

**Root cause – three independent missing controls:**

1. **No rate limiting on the REST API.** `rest/server.js` registers no rate-limiting middleware for any route. The only throttling in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belongs to the separate `web3` Java service and guards only `POST /contracts/call`. The Node.js REST server has no equivalent.

2. **`validateHashOrNumber` is format-only.** It calls `utils.isValidEthHashOrHederaHash()` which tests the regex `/^(0x)?([0-9A-Fa-f]{64}|[0-9A-Fa-f]{96})$/` and returns immediately on a match. No existence pre-check, no authentication requirement, no per-IP accounting.

3. **404 responses are never cached.** `responseCacheUpdateHandler` stores a response only when `httpStatusCodes.isSuccess(res.statusCode)` is true (`rest/middleware/responseCacheHandler.js:95`). Every miss for a unique hash therefore bypasses the Redis cache entirely and reaches the database.

**Database query issued on every miss:**
```sql
SELECT ... FROM record_file WHERE hash LIKE '<attacker_hash>%'
```
The `record_file__hash` index (created with `collate "C"`) allows PostgreSQL to use an index scan for prefix-LIKE queries, so each individual query is cheap. However, with no rate cap, an attacker can issue thousands of such queries per second, saturating the connection pool and the index's buffer-pool footprint, starving legitimate queries of both connections and cached index pages.

### Impact Explanation

Every concurrent attacker request consumes one database connection from the shared pool. At sustained high rates, the pool is exhausted and legitimate API calls (account lookups, transaction queries, etc.) queue or time out. Because each hash is unique, the index pages for the non-existent prefix ranges are repeatedly fetched and evicted, increasing I/O pressure. The attack requires no credentials, no special knowledge, and no on-chain activity. All users of the mirror-node REST API are affected for the duration of the attack.

### Likelihood Explanation

The exploit requires only an HTTP client and the ability to generate random 64-character hex strings. No authentication, no tokens, no privileged access. The endpoint is publicly documented in `openapi.yml`. A single attacker with a modest network connection can sustain thousands of requests per second. The attack is trivially scriptable and repeatable.

### Recommendation

1. **Add rate limiting to the REST API.** Apply a per-IP (or global) rate limiter (e.g., `express-rate-limit`) in `rest/server.js` before the route handlers, targeting at minimum the `/:hashOrNumber` path.
2. **Cache 404 responses for hash lookups.** Extend `responseCacheUpdateHandler` to cache 404 responses for the block-by-hash path with a short TTL (e.g., 5 s). Repeated requests for the same non-existent hash will then be served from Redis.
3. **Consider a Bloom filter or existence pre-check** before issuing the SQL query, so syntactically valid but non-existent hashes are rejected in-process.

### Proof of Concept

```bash
# Generate and fire 10 000 requests with unique random valid-format hashes
for i in $(seq 1 10000); do
  HASH=$(openssl rand -hex 32)   # 64 hex chars = valid 32-byte eth hash
  curl -s -o /dev/null "https://<mirror-node>/api/v1/blocks/${HASH}" &
done
wait
```

Each request passes `isValidEthHashOrHederaHash()`, reaches `RecordFileService.getByHashOrNumber()`, executes `WHERE hash LIKE '<hash>%'` against the database, finds nothing, and returns HTTP 404 — which is never cached. Running this loop continuously from one or more hosts exhausts the database connection pool and degrades response times for all concurrent users.
