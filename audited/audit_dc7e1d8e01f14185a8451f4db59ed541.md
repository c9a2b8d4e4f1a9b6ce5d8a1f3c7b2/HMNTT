### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent GET /tokens/:tokenId/nfts Requests

### Summary
`getNftTokensRequest()` in `rest/tokens.js` issues a synchronous DB query per request with no rate limiting or concurrency control on the REST API layer. The default connection pool is capped at 10 connections with a 20-second statement timeout. An unauthenticated attacker sending 10+ concurrent requests can exhaust the pool, causing all subsequent DB-dependent queries to block for up to 20 seconds per wave, effectively denying service to all other API consumers.

### Finding Description
**Code path:**

`rest/tokens.js`, `getNftTokensRequest()`, lines 827–858:
```js
const getNftTokensRequest = async (req, res) => {
  const tokenId = getAndValidateTokenIdRequestPathParam(req);
  const filters = utils.buildAndValidateFilters(req.query, acceptedNftsParameters, validateTokenQueryFilter);
  const {query, params, limit, order} = extractSqlFromNftTokensRequest(tokenId, nftSelectQuery, filters);
  const {rows} = await pool.queryQuietly(query, params);   // acquires a pool connection
  ...
};
```

`extractSqlFromNftTokensRequest()` (lines 754–785) builds:
```sql
SELECT ... FROM nft
LEFT JOIN entity e ON e.id = nft.token_id
WHERE nft.token_id = $1 [AND nft.account_id > $2]
ORDER BY nft.serial_number DESC
LIMIT $N
```

**Root cause — three compounding weaknesses:**

1. **No rate limiting on the REST API.** The Express server (`rest/server.js`) has no per-IP or global request throttle for this endpoint. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` Java service, not in the Node.js REST service.

2. **Tiny default connection pool.** `rest/dbpool.js` sets `max: config.db.pool.maxConnections`, whose documented default is **10** (`hiero.mirror.rest.db.pool.maxConnections = 10`). Every concurrent `await pool.queryQuietly(...)` call holds one connection until the query completes.

3. **Long statement timeout.** The default `statementTimeout` is **20 000 ms** (`hiero.mirror.rest.db.pool.statementTimeout = 20000`). A query can hold a connection for up to 20 seconds before being killed.

**Why the LIMIT clause does not fully mitigate this:**
- The attacker can combine `account.id` filters (accepted via `nftFilterColumnMap`) with a large-collection token, forcing PostgreSQL to scan the `nft` table on a non-leading index column, degrading query time.
- Even without slow queries, sending a continuous stream of requests keeps all 10 slots occupied: as soon as one slot frees, the next queued request claims it.
- `connectionTimeoutMillis` is also 20 000 ms, so waiting requests block the response for 20 s before returning an error — amplifying perceived unavailability.

### Impact Explanation
With the pool exhausted, every other API endpoint that calls `pool.queryQuietly` (transactions, accounts, balances, etc.) queues behind the attacker's requests. New legitimate requests wait up to 20 seconds for a connection and then fail. This constitutes a full read-path denial of service for the mirror node REST API. Because the attacker can continuously replenish the 10 slots, the outage is sustained for as long as the attack continues. Severity: **High** (availability impact, no authentication required, trivially repeatable).

### Likelihood Explanation
The endpoint is public and unauthenticated. Sending 10–20 concurrent HTTP requests is trivial from a single machine or a small botnet. No special knowledge of the chain state is required — any valid token ID with a non-trivial NFT collection suffices. The attack is repeatable indefinitely with no cost to the attacker.

### Recommendation
1. **Add a concurrency/rate-limit middleware** to the Node.js REST server (e.g., `express-rate-limit` or a semaphore on DB-bound routes) to cap concurrent in-flight DB queries per IP and globally.
2. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer) in front of PostgreSQL so pool exhaustion does not cascade across all endpoints.
3. **Reduce `statementTimeout`** for this endpoint (e.g., 5 000 ms) to shorten the window each connection can be held.
4. **Add a `connectionTimeoutMillis` fast-fail** (e.g., 2 000 ms) so queued requests fail quickly rather than blocking for 20 seconds.

### Proof of Concept
```bash
# Requires: a token ID with a large NFT collection (e.g., 0.0.1234567)
# Send 20 concurrent requests; 10 exhaust the pool, 10 queue and block

for i in $(seq 1 20); do
  curl -s "http://<mirror-node>:5551/api/v1/tokens/0.0.1234567/nfts?limit=100&account.id=gt:0.0.1" &
done
wait

# Simultaneously, verify other endpoints are blocked:
time curl "http://<mirror-node>:5551/api/v1/transactions?limit=1"
# Expected: ~20 second delay or connection timeout error
```
