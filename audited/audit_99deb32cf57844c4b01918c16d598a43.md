### Title
Unbounded Subscription Duration in `subscribeTopic()` Enables Resource Exhaustion via Permanent Open Streams

### Summary
The `subscribeTopic()` method in `TopicMessageServiceImpl.java` imposes no server-side maximum subscription duration. When a client omits `endTime`, the internal `pastEndTime()` helper explicitly returns `Flux.never()`, creating a stream that never terminates. Combined with the absence of a global connection or subscriber count cap, an unprivileged attacker can open many persistent connections to exhaust server-side resources (threads, memory, database connections), denying service to legitimate observers.

### Finding Description
**Exact code path:**

In `subscribeTopic()` (lines 72–73), the live flux is bounded only by `pastEndTime(topicContext)`:

```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)));
```

`pastEndTime()` (lines 123–131) explicitly returns `Flux.never()` when `endTime` is null:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← stream never terminates
    }
    ...
}
```

`TopicMessageFilter` (lines 22–46) declares `endTime` as a nullable `Long` with no server-enforced upper bound, and `limit` defaults to `0` (no limit). The `isValidEndTime()` and `isValidStartTime()` validators only check relative ordering, not maximum duration.

The secondary guards are insufficient:
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14; applied in `GrpcConfiguration`, line 33) limits calls **per TCP connection**, not the total number of connections or global subscribers.
- The nginx `grpc_read_timeout 600s` (docker-compose.yml, line 226) resets every time a message is received; subscribing to any topic with periodic activity keeps the timeout from firing indefinitely. Direct access to port 5600 bypasses nginx entirely.
- `subscriberCount` (line 48) is a Micrometer gauge for observability only — it enforces nothing.

### Impact Explanation
Each permanent subscription holds a Reactor `Subscription`, a `TopicContext` object, a listener registration (Redis/poll), and indirectly a database polling thread. An attacker opening `N` TCP connections × 5 calls each = `5N` permanent subscriptions. With enough connections, the bounded-elastic scheduler thread pool, JVM heap, and database connection pool are exhausted. Legitimate transaction observers receive `RESOURCE_EXHAUSTED` or connection-refused errors, constituting a full denial of service for the gRPC mirror node API. Severity is **High** (availability impact, no authentication required).

### Likelihood Explanation
No privileges, accounts, or special knowledge are required — only the ability to open TCP connections to port 5600 (or via the nginx proxy to a topic with any message activity). The attack is trivially scriptable with any gRPC client library. It is repeatable and persistent: connections survive server-side until the process is restarted or the attacker disconnects. The only practical friction is network-level firewall rules, which are not enforced by the application itself.

### Recommendation
1. **Enforce a maximum subscription duration server-side**: Add a `maxSubscriptionDuration` property to `GrpcProperties` (e.g., default 1 hour). In `subscribeTopic()`, apply `.timeout(maxDuration)` or synthesize a non-null `endTime = startTime + maxDuration` when the client omits `endTime`.
2. **Enforce a global concurrent-subscriber cap**: Check `subscriberCount` against a configurable maximum at the start of `subscribeTopic()` and return `RESOURCE_EXHAUSTED` when exceeded.
3. **Add per-IP connection rate limiting** at the gRPC/nginx layer to prevent a single source from opening unbounded connections.
4. **Validate `endTime` has a maximum ceiling** in `TopicMessageFilter.isValidEndTime()`.
