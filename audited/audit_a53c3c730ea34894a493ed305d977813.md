### Title
Unescaped ILIKE Wildcard in `filterKeys.NAME` Causes Full Table Scan via Unauthenticated Token Name Query

### Summary
The `name` query parameter in the token listing endpoint is validated only for byte length (3–100 bytes) and operator type, but never sanitized for PostgreSQL ILIKE wildcard characters (`%`, `_`). An unauthenticated attacker can supply a value composed entirely of `%` characters (e.g., `%%%`), which passes all validation, bypasses the GIN trigram index, and forces a full sequential scan of the `token` table on every request.

### Finding Description

**Exact code path:**

Validation in `rest/tokens.js` lines 333–335:
```js
case filterKeys.NAME:
  ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
  break;
``` [1](#0-0) 

`isByteRange` in `rest/utils.js` lines 131–134 checks only byte length — no character content filtering:
```js
const isByteRange = (str, minSize, maxSize) => {
  const length = Buffer.from(str).length;
  return length >= minSize && length <= maxSize;
};
``` [2](#0-1) 

Query construction in `rest/tokens.js` line 177 — user value is concatenated with `%` on both sides with no escaping:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
``` [3](#0-2) 

**Root cause:** The assumption is that a 3-byte minimum enforces trigram availability for the GIN index. This assumption fails when the 3 bytes are all wildcard characters. The GIN trigram index (`token__name on token using gin (name gin_trgm_ops)`) requires at least 3 consecutive non-wildcard characters to produce a usable trigram set. [4](#0-3) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/tokens?name=%%%`
2. Validation: `op = 'eq'` ✓, `isByteRange('%%%', 3, 100)` = 3 bytes ✓ — passes.
3. Constructed ILIKE pattern: `'%' + '%%%' + '%'` = `'%%%%%'`
4. Executed SQL: `... WHERE t.name ILIKE '%%%%%' ORDER BY t.token_id ASC LIMIT 25`
5. PostgreSQL cannot extract any trigrams from `%%%%%` → GIN index skipped → full sequential scan of `token` table.
6. `LIMIT 25` bounds the result set but does **not** prevent the full scan needed to find matching rows.

**Why existing checks fail:**
- `isByteRange` is purely a byte-length check; `%` and `_` are single-byte ASCII characters that satisfy the 3-byte minimum. [2](#0-1) 
- No wildcard escaping (e.g., `replace(val, '%', '\%')`) is applied before interpolation. [3](#0-2) 
- `zeroRandomPageCostQueryHint` is not applied to the tokens query (only used in `accounts.js` and `topicmessage.js`), so there is no compensating hint to force index use. [5](#0-4) 
- No rate limiting is present in the REST layer for this endpoint.

### Impact Explanation
Each request with `name=%%%` (or `name=___%`, `name=_%_`, etc.) forces a full sequential scan of the `token` table. On a production mirror node with millions of token records, this is a CPU- and I/O-intensive operation. Concurrent requests from a single attacker or a small botnet can saturate DB connection pools and I/O bandwidth, causing query timeouts for all other API consumers and extending any existing degraded-replica window. This is a denial-of-service against the database tier with no authentication required.

### Likelihood Explanation
The endpoint is public and unauthenticated. The exploit requires a single HTTP GET request with a trivially crafted parameter. No special knowledge, credentials, or tooling is needed. The attack is trivially repeatable in a loop. Any external user who reads the OpenAPI spec (which documents the `name` parameter) can discover and exploit this. [6](#0-5) 

### Recommendation
1. **Escape ILIKE wildcards** before interpolation in `extractSqlFromTokenRequest`:
   ```js
   const escaped = filter.value.replace(/\\/g, '\\\\').replace(/%/g, '\\%').replace(/_/g, '\\_');
   conditions.push(`t.name ILIKE $${params.push('%' + escaped + '%')} ESCAPE '\\'`);
   ```
2. **Strengthen validation** in `validateTokenQueryFilter` to reject values that consist entirely of wildcard characters, or use a regex that requires at least 3 consecutive non-wildcard, non-space characters (ensuring a usable trigram exists).
3. **Apply rate limiting** to the `/api/v1/tokens` endpoint, particularly when the `name` parameter is present.
4. Consider enforcing a minimum of 3 consecutive literal (non-wildcard) characters at the validation layer, matching the trigram index requirement.

### Proof of Concept
```bash
# Single request — forces full sequential scan, matches all tokens
curl "https://<mirror-node-host>/api/v1/tokens?name=%%%"

# DoS loop — saturates DB with concurrent full-table-scan queries
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/tokens?name=%%%" &
done
wait
```
Expected result: each request executes `t.name ILIKE '%%%%%'` with no GIN index usage (verifiable via `EXPLAIN ANALYZE`), performing a sequential scan of the entire `token` table. Under load, DB CPU and I/O spike, and response times for all other queries degrade.
