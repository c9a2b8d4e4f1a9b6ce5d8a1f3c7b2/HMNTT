### Title
Unbounded `DISTINCT ON` Scan via `account.id` + Credit/Debit Filter Exhausts DB Connection Pool

### Summary
`getTransferDistinctTimestampsQuery()` in `rest/transactions.js` generates a `DISTINCT ON (consensus_timestamp)` query over `crypto_transfer` with a `LIMIT` clause. When a credit/debit filter (`type=credit`/`type=debit`) is combined with an `account.id` filter, the `amount` predicate is not covered by the `(entity_id, consensus_timestamp)` index, forcing PostgreSQL to scan every row for that account in the timestamp range before the `LIMIT` can be satisfied. Because the `/transactions` endpoint calls `parseTimestampFilters` with `validateRange=false` and `bindTimestampRange` is disabled by default, an attacker can supply an unbounded timestamp range, causing the scan to cover the account's entire transfer history. With a DB connection pool of only 10 connections and a 20-second `statement_timeout`, 10 concurrent such requests exhaust the pool and block all other API consumers for up to 20 seconds per wave, far exceeding 500% of Hedera's ~2–3 second block time.

### Finding Description

**Exact code path:**

`getTransferDistinctTimestampsQuery()` at `rest/transactions.js` lines 325–356 emits:

```sql
SELECT DISTINCT ON (ctl.consensus_timestamp)
       ctl.consensus_timestamp, ctl.payer_account_id
FROM   crypto_transfer AS ctl
WHERE  ctl.entity_id = $1
  AND  ctl.consensus_timestamp >= $2   -- or unbounded
  AND  ctl.amount > 0                  -- credit/debit filter, NOT in index
ORDER BY ctl.consensus_timestamp DESC
LIMIT $3
``` [1](#0-0) 

The index that exists is:

```sql
CREATE INDEX crypto_transfer__entity_id_consensus_timestamp
    ON crypto_transfer (entity_id, consensus_timestamp)
    WHERE entity_id NOT IN (98, 800);
``` [2](#0-1) 

The `amount` column is absent from this index. PostgreSQL must perform an index scan on `(entity_id, consensus_timestamp)` and then evaluate `amount > 0` for every row, scanning until `LIMIT` matching rows are found. If the account has millions of rows but only a small fraction are credits (or debits), the engine scans the entire account history.

**Root cause — no timestamp range enforcement:**

`getTransactions` calls `parseTimestampFilters` with `validateRange=false`:

```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [3](#0-2) 

The `validateRange=false` argument skips the `maxTimestampRange` (7-day) guard entirely: [4](#0-3) 

`bindTimestampRange` is **disabled by default** (`bindTimestampRange: false`), so `bindTimestampRange()` returns the range unchanged with no upper-bound capping: [5](#0-4) 

An attacker supplying only `timestamp=gte:0` (or no timestamp at all) gets an open-ended range covering the entire table.

**Connection pool and timeout:**

The REST DB pool has a maximum of 10 connections and a 20-second `statement_timeout`: [6](#0-5) 

There is no per-IP or per-endpoint rate limiting on the REST API (the throttling code found is exclusively in the `web3` module for contract calls).

### Impact Explanation

Ten concurrent requests targeting a high-volume account with a minority credit/debit ratio hold all 10 pool connections for up to 20 seconds each. Every other API consumer receives connection-pool-exhaustion errors for the full 20-second window. The attacker can immediately re-issue the next wave, sustaining the outage indefinitely. Hedera's consensus block time is approximately 2–3 seconds; a 20-second stall is ≥ 667% of that baseline, satisfying the stated threshold. The mirror node's transaction-visibility service is completely unavailable to legitimate users during each wave.

### Likelihood Explanation

No privilege is required — the `/api/v1/transactions` endpoint is fully public. The attacker needs only to identify a high-volume account (trivially discoverable via the same API) and issue 10 concurrent HTTP requests with `account.id=<target>&type=credit` (or `debit`) and no timestamp upper bound. The attack is repeatable with a simple script and requires no special tooling.

### Recommendation

1. **Enforce a timestamp range for account-filtered queries.** When `accountQuery` is non-empty, require both a lower and upper timestamp bound, or apply `maxTimestampRange` (7d) unconditionally by passing `validateRange=true` in the `parseTimestampFilters` call at line 674.
2. **Add `amount` to the partial index** (or create a separate partial index on `(entity_id, consensus_timestamp, amount)`) so the credit/debit filter is index-covered and the scan terminates at `LIMIT` rows.
3. **Implement per-IP or per-endpoint rate limiting** on the REST API, analogous to the `web3` throttle, to prevent pool exhaustion from concurrent slow queries.
4. **Increase the connection pool size** or add a queue with a short wait timeout so pool exhaustion returns a 503 quickly rather than blocking indefinitely.

### Proof of Concept

```bash
# Identify a high-volume account (e.g., 0.0.98 has no entity_id index at all)
# Send 10 concurrent requests with an open timestamp range and credit filter
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/transactions?account.id=0.0.98&type=credit&timestamp=gte:0&limit=25" &
done
wait
# All 10 pool connections are now held for up to 20 s.
# Any concurrent legitimate request receives a connection-pool error or hangs.
```

For accounts with the index (non-98/800), substitute a known high-volume account where credits are rare (e.g., a pure-payer account that almost never receives funds), supply `type=credit`, and omit the upper timestamp bound. The scan traverses the entire account history before the `LIMIT` is satisfied.
