### Title
Unbounded Sub-Cache Accumulation in `cacheManagerSlotsPerContract` Leads to Memory Exhaustion DoS

### Summary
In `ContractStateServiceImpl.findStorageBatch`, every unique `contractId` causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to register a new named `CaffeineCache` in the `CaffeineCacheManager`'s internal `ConcurrentHashMap`. This internal registry is never evicted. The outer `contractSlotsCache` (bounded at 3000 entries) evicts entries over time, but the sub-caches remain permanently referenced in `cacheManagerSlotsPerContract`, causing unbounded heap growth. An unprivileged attacker querying millions of distinct contract addresses will exhaust JVM heap memory and crash the service.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorageBatch`, lines 85–87:
```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-layer cache design with a one-sided eviction boundary:**

1. The outer `contractSlotsCache` is backed by `CACHE_MANAGER_CONTRACT_SLOTS`, configured with `"expireAfterAccess=5m,maximumSize=3000,recordStats"` (`CacheProperties.java` line 25). It maps `EntityId → CaffeineCache` and is bounded at 3000 entries.

2. The `cacheManagerSlotsPerContract` bean (`EvmConfiguration.java` lines 107–112) is a `CaffeineCacheManager` created **without** a `setCacheNames()` call, placing it in *dynamic mode*. In dynamic mode, Spring's `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` for any previously unseen name and stores it in its internal `ConcurrentHashMap<String, Cache> cacheMap`. **This map has no eviction mechanism.**

3. When the outer `contractSlotsCache` evicts an entry (size > 3000 or TTL expiry), the `CaffeineCache` object is removed from the outer cache. However, `cacheManagerSlotsPerContract` still holds a strong reference to it in its `cacheMap` under the key `contractId.toString()`. The sub-cache is never garbage-collected.

4. Each new unique `contractId` permanently adds one `CaffeineCache` instance (backed by a Caffeine cache with `maximumSize=1500`, `expireAfterAccess=5m`) to `cacheManagerSlotsPerContract.cacheMap`. Over time, this map grows without bound.

**Why existing checks fail:**

- The outer cache's `maximumSize=3000` only limits the outer mapping; it does not remove entries from `cacheManagerSlotsPerContract.cacheMap`.
- The global rate limiter (`ThrottleManagerImpl`, `ThrottleConfiguration.java`) enforces a per-second request cap globally, not per-IP. An attacker can sustain the allowed rate indefinitely across multiple IPs or over time, continuously adding new sub-caches.
- `enableBatchContractSlotCaching` defaults to `true` (`CacheProperties.java` line 30), so the vulnerable path is active by default.
- There is no validation that `contractId` corresponds to an existing contract before the sub-cache is created; the cache is allocated before the DB query returns.

### Impact Explanation
Each `CaffeineCache` wrapping a Caffeine instance with `maximumSize=1500` allocates internal data structures (timer wheel, node arrays, stats counters) even when empty — conservatively 2–10 KB per instance. At a sustained rate of N requests/second with distinct contractIds, `cacheManagerSlotsPerContract.cacheMap` grows at N entries/second. With a default rate limit of hundreds of requests/second, an attacker can accumulate millions of sub-caches within hours, exhausting JVM heap and triggering `OutOfMemoryError`, crashing the web3 service. This prevents all contract call processing, including gossip-related transaction propagation.

### Likelihood Explanation
The attack requires no authentication, no privileged access, and no valid on-chain contracts — any 20-byte address value accepted by the JSON-RPC layer suffices as a `contractId`. The attacker only needs to send `eth_call` or `eth_getStorageAt` requests with distinct `to` addresses at the permitted rate. This is trivially scriptable. The attack is persistent: sub-caches accumulate across the full uptime of the service, not just within a single rate-limit window.

### Recommendation
1. **Bound the sub-cache registry**: Call `cacheManagerSlotsPerContract.setCacheNames(Set.of(...))` with a fixed set of names, or replace the dynamic per-contractId sub-cache design with a single shared cache keyed by `(contractId, slot)`.
2. **Alternatively, remove the dynamic `CaffeineCacheManager` indirection**: Store slot keys directly in the outer `contractSlotsCache` using a composite key `(contractId, slotKey)` with a single bounded Caffeine cache, eliminating the per-contractId sub-cache entirely.
3. **If the two-level design is retained**: Implement a `RemovalListener` on the outer `contractSlotsCache` that calls `cacheManagerSlotsPerContract.getCache(contractId.toString()).invalidateAll()` and removes the cache from the manager's registry on eviction.
4. **Short-term mitigation**: Set `hiero.mirror.web3.cache.enableBatchContractSlotCaching=false` to disable the vulnerable code path until a fix is deployed.

### Proof of Concept
```python
import requests, threading, itertools

URL = "http://<mirror-node-web3>:8545"

def make_address(n):
    return "0x" + hex(n)[2:].zfill(40)

def send_request(addr):
    payload = {
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [addr, "0x0", "latest"], "id": 1
    }
    requests.post(URL, json=payload, timeout=5)

# Send requests with millions of distinct contract addresses
for i in itertools.count(1):
    # Each unique address creates a new sub-cache in cacheManagerSlotsPerContract
    t = threading.Thread(target=send_request, args=(make_address(i),))
    t.start()
    # Observe JVM heap growth via JMX/metrics; OOM crash expected after
    # cacheManagerSlotsPerContract.cacheMap reaches millions of entries
```

Monitor heap via `/actuator/metrics` or JMX. `cacheManagerSlotsPerContract.cacheMap.size()` grows monotonically. JVM heap exhaustion and `OutOfMemoryError` follow.
