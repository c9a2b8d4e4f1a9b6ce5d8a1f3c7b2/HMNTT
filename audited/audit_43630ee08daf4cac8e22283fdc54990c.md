### Title
DB Connection Pool Exhaustion via Unauthenticated Concurrent Requests Due to WriteTimeout/StatementTimeout Gap and Absent Rate Limiting

### Summary
The Rosetta API's `GetDbWithContext()` creates a 20-second context timeout per DB operation, but Go's `net/http` `WriteTimeout` (10 seconds) does **not** cancel the request context when it fires — it only sets a write deadline on the TCP connection. This means each in-flight request holds a DB connection for up to 20 seconds regardless of the HTTP timeout. With `maxOpenConnections=100`, no application-level rate limiting, and the optional Traefik middleware disabled by default, an unauthenticated attacker can saturate the entire connection pool with concurrent requests, forcing all subsequent requests to block in goroutines and driving memory consumption well above 30%.

### Finding Description

**Code path:**

`rosetta/app/db/client.go`, `GetDbWithContext()`, lines 22–38:
```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
return d.db.WithContext(childCtx), cancel
```
`statementTimeout` defaults to **20 seconds** (`docs/configuration.md` line 662). The `ctx` argument is the HTTP request context passed down from every persistence-layer call (e.g., `block.go` line 195, `account.go` lines 113, 146, 229, 272, 317).

**Root cause — WriteTimeout does not cancel the request context:**

`main.go` lines 220–227 configures the HTTP server with `WriteTimeout: 10s` and `ReadTimeout: 5s`. In Go's `net/http`, `WriteTimeout` is implemented by calling `conn.rwc.SetWriteDeadline(t)` on the underlying TCP connection. This sets a deadline for *writes* but does **not** call `cancelCtx()` on the request context. The handler goroutine — and therefore the GORM query holding the DB connection — continues running until either:
- the 20-second child context timeout fires, or
- the query completes naturally.

The connection is only released after the handler returns, which happens at ~20 s, not ~10 s.

**No application-level rate limiting:**

The middleware stack in `main.go` lines 217–219 is:
```
MetricsMiddleware → TracingMiddleware → CorsMiddleware → router
```
`trace.go` and `metrics.go` contain zero rate-limiting or concurrency-limiting logic. The Traefik `inFlightReq: 5` / `rateLimit: 10/s` middleware defined in `charts/hedera-mirror-rosetta/values.yaml` lines 152–160 is gated by `global.middleware: false` (line 95) and is therefore **not applied** in the default deployment.

**Pool configuration:**

`db.go` lines 31–33 applies:
- `MaxOpenConnections = 100` (default, `docs/configuration.md` line 660)
- `MaxIdleConnections = 20`
- `MaxLifetime = 30 min`

When all 100 connections are in use, `database/sql` blocks the calling goroutine until a connection is freed or the context times out (up to 20 s). Each blocked goroutine consumes memory (Go goroutine stacks start at 8 KB and grow).

**Multi-query amplification:**

`RetrieveBalanceAtBlock` in `account.go` issues **four** sequential `GetDbWithContext` calls per request (`getCryptoEntity`, `selectPreviousPartitionLowerBound`, `latestBalanceBeforeConsensus`, `balanceChangeBetween`). Each call acquires and holds a connection for the duration of its query, extending the total per-request connection-hold time.

### Impact Explanation

With 100 concurrent attacker requests each holding a connection for up to 20 seconds, the pool is fully saturated. All legitimate requests block in goroutines waiting for a free connection. At 8 KB base stack per goroutine, 10,000 queued goroutines consume ~80 MB of stack memory alone, plus GORM/SQL overhead per blocked call. On a node with 200 MB memory limit (the default in `charts/hedera-mirror-rosetta/values.yaml` line 251), this easily exceeds 30% additional memory and can trigger OOM kills. The Rosetta node becomes unavailable to all legitimate callers (exchanges, validators, indexers) for the duration of the attack.

### Likelihood Explanation

The Rosetta API is publicly exposed via Ingress (values.yaml lines 119–134) with no authentication requirement — it is a read-only blockchain data API by design. The attacker needs only an HTTP client capable of sending concurrent POST requests to any DB-backed endpoint (`/block`, `/account/balance`, `/network/status`). No credentials, tokens, or special knowledge are required. The attack is trivially repeatable and scriptable with tools like `wrk`, `hey`, or `curl -Z`. The absence of rate limiting in the default deployment makes this a low-effort, high-impact attack.

### Recommendation

1. **Close the WriteTimeout/StatementTimeout gap**: Set `statementTimeout ≤ writeTimeout` (e.g., both at 10 s), or propagate cancellation explicitly by wrapping the HTTP handler to cancel the request context when `WriteTimeout` fires (e.g., using a middleware that calls `cancel()` after the write deadline).
2. **Enable application-level concurrency limiting**: Add a semaphore-based middleware in `main.go` (before the router) that limits total in-flight DB-bound requests, independent of the Traefik layer.
3. **Enable the Traefik middleware by default**: Change `global.middleware: false` to `true` in `values.yaml`, or make `inFlightReq` and `rateLimit` opt-out rather than opt-in.
4. **Reduce `maxOpenConnections`**: Lower the default from 100 to a value consistent with expected legitimate concurrency (e.g., 20–30), reducing the blast radius.
5. **Add a pool-wait timeout**: Use `db.SetConnMaxIdleTime()` and ensure the context timeout on pool-wait is shorter than `statementTimeout`.

### Proof of Concept

```bash
# Send 120 concurrent POST requests to /block/transaction (DB-heavy endpoint)
# Each request will acquire a DB connection and hold it for up to 20 s
# After ~100 requests, the pool is saturated; subsequent requests block in goroutines

seq 120 | xargs -P 120 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<rosetta-host>:5700/block \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}'

# Observe: first ~100 requests return 200/500 after ~20 s
# Requests 101+ block until a connection is freed (up to 20 s wait)
# Monitor goroutine count and memory:
curl http://<rosetta-host>:5700/metrics | grep rosetta_request_inflight
# Memory increase >30% observable via container metrics or /proc/self/status
```
