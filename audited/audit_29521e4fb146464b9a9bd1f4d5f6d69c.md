### Title
Unauthenticated Global Gas-Bucket Exhaustion via Concurrent Max-Gas Requests (Shared Throttle DoS)

### Summary
The `gasLimitBucket` and `rateLimitBucket` in `ThrottleManagerImpl` are singleton Spring beans shared across all callers with no per-IP or per-user partitioning. A single unauthenticated attacker can send 500 concurrent POST requests to `/api/v1/contracts/call` each carrying `gas=15000000` (the default maximum), atomically draining both shared buckets and causing every legitimate user to receive HTTP 429 for the entire refill window (up to 1 second), repeatable indefinitely.

### Finding Description

**Exact code path:**

`ContractController.call()` (lines 40–41) calls `throttleManager.throttle(request)` **before** `validateContractMaxGasLimit(request)`: [2](#0-1) 

Inside `ThrottleManagerImpl.throttle()`, both buckets are global singletons — no caller identity is considered: [3](#0-2) 

The `gasLimitBucket` is a single `SYNCHRONIZED` bucket with capacity `scaleGas(gasPerSecond)`: [4](#0-3) 

`scaleGas` divides by `GAS_SCALE_FACTOR = 10_000`: [5](#0-4) 

Default `gasPerSecond = 7_500_000_000L` → bucket capacity = **750,000 tokens**. Default `requestsPerSecond = 500` → rate bucket capacity = **500 tokens**.

`ContractCallRequest.gas` defaults to `15_000_000` with only a `@Min(21_000)` constraint and no `@Max`: [1](#0-0) 

Each request with `gas=15_000_000` consumes `scaleGas(15_000_000) = 1500` tokens from the gas bucket. 500 such requests consume `500 × 1500 = 750,000` tokens — exactly the full bucket capacity — while simultaneously exhausting the 500-token rate bucket.

The `restore()` path only fires on `InvalidParametersException` (gas > maxGasLimit). A request with `gas = 15_000_000` (equal to `maxGasLimit = 15_000_000`) passes validation, so **no restore occurs** until after full EVM execution completes: [6](#0-5) [7](#0-6) 

**Root cause:** The throttle is a global resource with no per-caller isolation. A single client can consume the entire shared budget in one burst.

### Impact Explanation

All legitimate users of the `/api/v1/contracts/call` endpoint receive HTTP 429 for up to 1 second per attack wave. Since the attack is trivially repeatable every second (the bucket refills greedily every 1 second), an attacker can maintain a near-continuous denial of service against the entire user population. The `gasLimitBucket` and `rateLimitBucket` — intended as protection mechanisms — become the attack surface itself.

### Likelihood Explanation

No authentication, API key, or IP-based rate limiting is required. Any anonymous HTTP client can trigger this. The attacker needs only a standard HTTP client capable of sending 500 concurrent requests — achievable with `curl`, `wrk`, `ab`, or a trivial script. The attack is repeatable every second with no cooldown penalty for the attacker (rejected requests after bucket exhaustion do not consume tokens). This is a low-skill, low-cost, high-impact attack.

### Recommendation

1. **Add per-IP (or per-API-key) rate limiting** at the application layer (e.g., a `Bucket` keyed by `X-Forwarded-For` / `RemoteAddr`) or at the ingress/gateway layer (the web3 Helm chart lacks the per-IP middleware present in the rosetta chart).
2. **Add `@Max` validation on `ContractCallRequest.gas`** so that the field-level constraint rejects oversized values before throttle tokens are consumed, eliminating the ordering dependency between `throttle()` and `validateContractMaxGasLimit()`.
3. **Move `validateContractMaxGasLimit` before `throttleManager.throttle`** so that invalid gas values never touch the shared bucket.

### Proof of Concept

```bash
# Send 500 concurrent requests with maximum gas, exhausting both shared buckets
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}' &
done
wait

# Immediately after, a legitimate user's request is denied:
curl -v -X POST http://<host>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000,"data":"0x"}'
# Expected: HTTP 429 "Gas per second rate limit exceeded." or "Requests per second rate limit exceeded"
# Repeat every ~1 second to maintain continuous DoS
```

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }
