### Title
Cache Negative-Result Bypass via Spring Optional Unwrapping Enables Unauthenticated DB Connection Exhaustion

### Summary
`findByIdAndDeletedIsFalse(Long entityId)` in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` uses `unless = "#result == null"` on a method returning `Optional<Entity>`. Spring's `CacheAspectSupport` unwraps the `Optional` before evaluating the `unless` SpEL expression, so for a non-existent entity the unwrapped value is `null`, `#result == null` evaluates to `true`, and the empty result is **never stored in the cache**. Every subsequent request for the same non-existent entity ID therefore hits the database. Because the global rate-limit bucket is not per-IP and no authentication is required, an unauthenticated attacker can sustain a flood of unique non-existent entity ID lookups that saturates the DB connection pool.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 20–30:

```java
@Caching(cacheable = {
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_ENTITY,
               unless = "#result == null"),          // ← line 22
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_SYSTEM_ACCOUNT,
               condition = "#entityId < 1000 && ...",
               unless = "#result == null")
})
Optional<Entity> findByIdAndDeletedIsFalse(Long entityId);
```

**Root cause – Spring Optional unwrapping in `CacheAspectSupport`:**
Spring's caching infrastructure (in `CacheAspectSupport.execute()`) calls `unwrapReturnValue(returnValue)` to obtain the value that will actually be stored:

```
returnValue  = invokeOperation(invoker);          // Optional<Entity>
cacheValue   = unwrapReturnValue(returnValue);    // Entity | null
cachePutRequest.apply(cacheValue);                // unless evaluated with cacheValue
```

`canPutToCache(cacheValue)` evaluates the `unless` SpEL with `#result = cacheValue`. When the entity does not exist, `cacheValue` is `null`, so `#result == null` → `true` → the result is **not written to the cache**. The `Optional.empty()` wrapper is never null, but the unwrapped value is, which is what Spring evaluates.

**Call chain from public endpoint to the vulnerable method:**
```
POST /api/v1/contracts/call
  → ContractController.call()
  → ContractExecutionService.processCall()
  → CommonEntityAccessor.get(EntityId, Optional.empty())   [line 63]
  → entityRepository.findByIdAndDeletedIsFalse(entityId.getId())
```

`CommonEntityAccessor` at line 63 calls `findByIdAndDeletedIsFalse` for every "latest-block" call where the `to` address is a Hedera long-zero address (`0x000...0{entityNum}`).

**Cache configuration** (`CacheProperties.java` line 19, `EvmConfiguration.java` lines 99–104):
```
expireAfterWrite=1s, maximumSize=10000
```
Even for existing entities the TTL is only 1 second, but for non-existent entities there is **zero caching** at all.

**Rate-limiting** (`ThrottleManagerImpl.java` lines 37–42, `ThrottleProperties.java` line 35):
The `rateLimitBucket` is a single **global** token bucket (default 500 req/s), not per-IP. A single attacker can consume the entire budget.

**DB statement timeout** (`docs/configuration.md` line 702): 3 000 ms. With a typical HikariCP pool of 10–20 connections and 500 req/s each holding a connection for up to 3 s, the pool is exhausted in under a second.

### Impact Explanation
Every request for a non-existent entity ID issues a live `SELECT` against the `entity` table. An attacker rotating through a large space of non-existent IDs (e.g., `Long.MAX_VALUE - n`) can sustain 500 DB queries/second indefinitely. With a 3 000 ms statement timeout and a pool of ~10–20 connections, the pool saturates, causing all legitimate contract-call requests to queue and time out. This degrades or halts the web3 mirror-node service without requiring any credentials or privileged access.

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint is publicly reachable with no authentication. The attack requires only a standard HTTP client sending `POST` requests with a valid 40-hex-char `to` address encoding a non-existent entity number. No special knowledge of the system is needed; the attacker simply increments a counter in the address field. The global (non-per-IP) rate limiter means a single host can monopolize the entire allowed request budget.

### Recommendation
1. **Cache negative results explicitly.** Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` — but note this still won't cache because `#result` is the Optional. The correct fix is to use `unless = "T(java.util.Optional).empty().equals(#result)"` or, better, switch to `Optional.isPresent()` awareness:
   ```java
   unless = "#result != null && !#result.isPresent()"
   ```
   This correctly evaluates against the `Optional` object (not the unwrapped value) and caches empty results.
2. **Add per-IP rate limiting** in addition to the global bucket, so a single client cannot exhaust the global budget.
3. **Validate entity ID range** before issuing the DB query: reject IDs greater than `findMaxId()` (already available on the repository) at the service layer.

### Proof of Concept
```bash
# Rotate through non-existent entity IDs encoded as long-zero EVM addresses
# Entity num 0x7FFFFFFF00000001 ... 0x7FFFFFFF0000FFFF (all non-existent)
for i in $(seq 1 500); do
  ENTITY_HEX=$(printf '%040x' $((0x7FFFFFFF0000 + i)))
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x${ENTITY_HEX}\",\"data\":\"0x\",\"gas\":21000}" &
done
wait
# Each request misses the cache (empty Optional not cached due to Spring unwrapping),
# issues a live SELECT against the entity table, and holds a DB connection for up to 3 s.
# With 500 concurrent requests and a pool of ~10-20 connections, the pool is exhausted.
```
