### Title
Spurious Pagination Next Link in `getTokenRelationships` Due to Off-by-One Pagination Check

### Summary
`getTokenRelationships()` in `rest/controllers/tokenController.js` uses `tokens.length === query.limit` to decide whether to emit a `next` pagination link. Because the SQL query fetches exactly `limit` rows (not `limit + 1`), when an account has exactly N token relationships and a caller supplies `limit=N`, the equality check is true and a next link is generated. Following that link returns zero records, causing mirror node consumers to observe a spurious empty page and an incorrect terminal state.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenController.js` lines 78–84: [1](#0-0) 

The condition `tokens.length === query.limit` is used as a proxy for "there may be more data." However, `TokenService.getTokenAccounts()` issues a SQL query that fetches exactly `limit` rows via `limit $2`: [2](#0-1) 

The SQL template in `TokenService.tokenRelationshipsQuery` passes `limit` directly as `$2` with no `+1` sentinel: [3](#0-2) 

**Root cause:** The standard "has next page" pattern requires fetching `limit + 1` rows and checking `rows.length > limit`. This code fetches exactly `limit` rows and checks `rows.length === limit`. When the total number of token relationships for an account equals exactly `limit`, the DB returns all N rows, the equality holds, and a next link is unconditionally emitted via `utils.getPaginationLink(req, false, ...)` — the `false` argument means "not at end, emit link": [4](#0-3) 

**Failed assumption:** The code assumes that receiving exactly `limit` rows implies there are more rows beyond the cursor. This is false when the total count equals `limit`.

**Exploit flow:**
1. Attacker enumerates the token relationship count for a target account (binary search with varying `limit` values, or direct observation).
2. Attacker (or any consumer) issues `GET /api/v1/accounts/{id}/tokens?limit=N` where N equals the exact count of associated tokens.
3. The response contains all N tokens **and** a non-null `next` link (e.g., `?limit=N&token.id=gt:<last_token_id>`).
4. A mirror node consumer following the `next` link receives `{"tokens": [], "links": {"next": null}}`.
5. The consumer has now observed a pagination sequence that terminates with an empty page, which is indistinguishable from a mid-stream data gap or deletion event.

### Impact Explanation

Mirror node consumers that faithfully follow pagination links will receive an empty terminal page. Consumers that treat an empty page as an error, a data gap, or a sign of record deletion will export an incorrect view of the account's token relationships. Automated exporters may log false errors, retry indefinitely, or record a stale/incomplete snapshot. The `next` link in the response is a protocol-level signal; emitting it spuriously violates the contract that following it will yield at least one record.

### Likelihood Explanation

No privileges are required. The `limit` parameter is accepted from any unauthenticated caller and is only capped at the configured maximum (default 100 for unauthenticated users): [5](#0-4) 

An attacker can trivially determine the exact token relationship count for any account by issuing requests with increasing `limit` values and observing when the `next` link disappears (or appears). The trigger condition is deterministic and repeatable. Any account with ≤ 100 associated tokens is trivially exploitable by an unauthenticated caller.

### Recommendation

Fetch `limit + 1` rows from the database and truncate the result to `limit` before returning. Use `tokens.length > limit` (after slicing) as the "has next page" signal, consistent with the correct pattern used elsewhere (e.g., `rest/accounts.js` line 380 uses `ret.accounts.length !== limitAndOrderQuery.limit` against a query that fetches `limit + 1`). Specifically:

1. In `getTokenRelationshipsQuery`, pass `limit + 1` as the SQL `LIMIT` parameter.
2. In `getTokenRelationships`, check `tokens.length > query.limit` and slice `tokens` to `query.limit` before responding.

### Proof of Concept

**Precondition:** Account `0.0.1001` has exactly 3 associated token relationships.

```
# Step 1 – request with limit=3 (equals total count)
GET /api/v1/accounts/0.0.1001/tokens?limit=3

Response:
{
  "tokens": [ {token_id: "0.0.1135"}, {token_id: "0.0.2500"}, {token_id: "0.0.45678"} ],
  "links": { "next": "/api/v1/accounts/0.0.1001/tokens?limit=3&token.id=gt:0.0.45678" }
}
# tokens.length (3) === query.limit (3) → next link emitted

# Step 2 – follow the next link
GET /api/v1/accounts/0.0.1001/tokens?limit=3&token.id=gt:0.0.45678

Response:
{
  "tokens": [],
  "links": { "next": null }
}
# Empty page returned — mirror node consumer observes spurious empty terminal page
```

The test fixture in `rest/__tests__/specs/accounts/{id}/tokens/token-id-pagination.json` contains exactly this account with 3 token relationships, making the scenario directly reproducible in the existing test environment: [6](#0-5)
