### Title
Operator Private Key Exposed in World-Readable `/proc/<pid>/cmdline` via `-operator-key` CLI Flag

### Summary
`loadConfig()` in `pinger/config.go` registers the operator private key as a standard Go `flag.StringVar` under `-operator-key`. When the binary is launched with this flag, the key is embedded in the process's argv array, which Linux exposes verbatim in `/proc/<pid>/cmdline` with world-readable permissions. Any unprivileged local user can trivially extract the key without any special privileges.

### Finding Description
**Exact code location:** `pinger/config.go`, `loadConfig()`, line 40:
```go
flag.StringVar(&cfg.operatorKey, "operator-key", envOr("HIERO_MIRROR_PINGER_OPERATOR_KEY", "302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137"), "operator private key string")
``` [1](#0-0) 

**Root cause:** Go's `flag` package stores the value directly from `os.Args`. The kernel keeps the full argv of every running process in `/proc/<pid>/cmdline`, null-byte-delimited, with permissions `0444` (world-readable). There is no mechanism in this code to scrub, mask, or warn against passing the key on the command line.

**Exploit flow:**
1. Operator launches: `./pinger -operator-key 302e020100...privatekey...`
2. Attacker (any local user, no privileges needed) enumerates pinger's PID: `pgrep pinger` or `ps aux | grep pinger`
3. Attacker reads: `cat /proc/<pid>/cmdline | tr '\0' '\n'`
4. The full private key string appears in plaintext output.

**Why existing checks fail:** The only validation in `loadConfig()` is a non-empty check on `cfg.operatorKey` (lines 105–107). There is no check for whether the key was supplied via CLI vs. env var, no argv scrubbing, and no redaction. The `ps aux` output also leaks the key to any user who can run `ps`. [2](#0-1) 

**Secondary issue:** A hardcoded default private key (`302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137`) is baked into the source as the fallback value on line 40. Any deployment that omits both the flag and the env var silently uses this well-known key. [1](#0-0) 

**Note on env var path:** `/proc/<pid>/environ` is mode `0400` on Linux — readable only by the process owner or root — so the env var `HIERO_MIRROR_PINGER_OPERATOR_KEY` is **not** readable by an unprivileged third party. The CLI flag path is the exploitable vector.

### Impact Explanation
The `operatorKey` is a Hiero/Hedera ED25519 private key used to sign and submit cryptocurrency transfer transactions on behalf of the operator account. [3](#0-2) 

An attacker who obtains this key can: sign arbitrary transactions from the operator account, drain the account's HBAR balance, and impersonate the operator on the Hiero network. The impact is full compromise of the operator account's funds and signing authority.

### Likelihood Explanation
Precondition is only that the attacker has a shell on the same host (e.g., a co-tenant in a shared Kubernetes node, a compromised sidecar container with host PID namespace access, or any other local user account). No privilege escalation is required. The `/proc/<pid>/cmdline` read is a single, instantaneous, non-destructive operation. The attack is repeatable as long as the process is running. In containerized deployments where `hostPID: true` is set or where the container shares the host PID namespace, the attack surface extends to any container on the node.

### Recommendation
1. **Remove the CLI flag for the private key entirely.** Secrets must not be passed via argv. Accept the key only through the environment variable (`HIERO_MIRROR_PINGER_OPERATOR_KEY`) or a secrets-manager file path.
2. **Remove the hardcoded default private key** on line 40. Replace the default with `""` and require explicit configuration; the existing empty-check at line 105 will enforce this.
3. If a file-based secret is used, read the key from the file path and immediately zero the path string from memory after use.
4. Add a startup warning (or hard error) if the key is detected in `os.Args` at runtime.

### Proof of Concept
```bash
# Terminal 1 — launch pinger with key on CLI (as any user)
./pinger -operator-key 302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137 &
PID=$!

# Terminal 2 — extract key as a different unprivileged user
cat /proc/$PID/cmdline | tr '\0' '\n' | grep -A1 operator-key
# Output:
# -operator-key
# 302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137

# Alternative: visible in ps output to all users
ps aux | grep pinger
# shows full argv including the private key
```
