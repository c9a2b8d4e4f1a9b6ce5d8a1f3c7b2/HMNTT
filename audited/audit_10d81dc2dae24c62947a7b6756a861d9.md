### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Requests to `/api/v1/contracts/results/:transactionIdOrHash` and `/api/v1/contracts/results/:transactionIdOrHash/actions`

### Summary
The REST API Node.js service has no per-IP rate limiting or concurrency controls on the `/api/v1/contracts/results/:transactionIdOrHash` and `/api/v1/contracts/results/:transactionIdOrHash/actions` endpoints. Both endpoints call `getTransactionDetailsFromTransactionId()` without any guard, and the `/results/:transactionIdOrHash` handler additionally fires five parallel DB queries via `Promise.all` inside `getDetailedContractResults()`. With a default pool ceiling of 10 connections, as few as 2 concurrent requests to the `/results` endpoint can exhaust the entire pool, causing all DB-dependent REST API endpoints to queue and time out.

### Finding Description

**Code path — `/results/:transactionIdOrHash` (non-hash branch):**

`rest/controllers/contractController.js` line 1142:
```js
const transactions = await TransactionService.getTransactionDetailsFromTransactionId(transactionId, nonce);
```
Then line 1156:
```js
const contractDetails = await ContractService.getInvolvedContractsByTimestampAndContractId(...)
```
Then lines 1165–1166 via `getDetailedContractResults()` (lines 1205–1219):
```js
return Promise.all([
  ContractService.getContractResultsByTimestamps(...),
  TransactionService.getEthTransactionByTimestampAndPayerId(...),
  RecordFileService.getRecordFileBlockDetailsFromTimestamp(...),
  ContractService.getContractLogsByTimestamps(...),
  ContractService.getContractStateChangesByTimestamps(...),
]);
```
`Promise.all` launches all five queries simultaneously, each checking out a separate connection from the shared pool.

**Code path — `/results/:transactionIdOrHash/actions` (non-hash branch):**

`rest/controllers/contractController.js` line 1252:
```js
tx = await TransactionService.getTransactionDetailsFromTransactionId(transactionId);
```
Then line 1263:
```js
const rows = await ContractService.getContractActionsByConsensusTimestamp(...)
```
Two sequential connections per request.

**Pool configuration:**

`rest/dbpool.js` line 14:
```js
max: config.db.pool.maxConnections,
```
Default value from `docs/configuration.md` line 556: `maxConnections = 10`.

**No rate limiting in the REST module:**

A search of `rest/**/*.js` for any rate-limiting or throttle middleware returns no results. The `ThrottleManagerImpl` / `ThrottleConfiguration` classes exist only in the `web3` (Spring Boot Java) module and are not applied to the Node.js REST service. The route file `rest/routes/contractRoute.js` registers both endpoints with no middleware beyond `extendExpress`.

**Root cause:** The system assumes the pool of 10 connections is sufficient because it expects low concurrency. There is no mechanism to reject or queue requests before they acquire pool connections.

**Failed assumption:** Each in-flight request to `/results/:transactionIdOrHash` holds up to 5 simultaneous pool connections during the `Promise.all` phase. Two concurrent requests therefore consume all 10 connections. Any additional request — including legitimate ones — blocks until `connectionTimeoutMillis` (default 20 000 ms) elapses and then fails.

### Impact Explanation

When the pool is exhausted, every endpoint that touches the database (transactions, accounts, tokens, topics, etc.) stalls for up to 20 seconds per request and then returns a 500 error. This constitutes a full REST API denial-of-service. Because the mirror node REST API is the primary read interface for dApps and wallets querying Hedera network state, sustained pool exhaustion prevents clients from confirming transaction finality, reading balances, or verifying contract results — effectively a network-observable outage for all consumers of the mirror node.

### Likelihood Explanation

No authentication, API key, or IP-based rate limit is required. Any external user with a valid (or even non-existent) transaction ID can trigger the attack. A single attacker with a basic script sending 2–3 concurrent HTTP requests per second to `/api/v1/contracts/results/<txId>` is sufficient to keep the pool permanently exhausted. The attack is trivially repeatable, requires no special knowledge beyond the public OpenAPI spec, and is invisible to application-layer auth checks.

### Recommendation

1. **Add a concurrency/rate-limit middleware** to the Node.js REST service (e.g., `express-rate-limit` or a semaphore) specifically for the `/results/:transactionIdOrHash` and `/results/:transactionIdOrHash/actions` routes, limiting concurrent in-flight requests per IP.
2. **Increase the default pool size** or document a minimum safe value relative to expected concurrency.
3. **Refactor `getDetailedContractResults()`** to use sequential queries or a bounded concurrency helper instead of unbounded `Promise.all`, so a single request cannot hold more than one or two pool connections at a time.
4. **Add a global in-flight request counter** that rejects new requests with HTTP 429 when the pool utilization exceeds a configurable threshold, before a connection checkout is even attempted.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or xargs
# Replace <MIRROR_HOST> and <TX_ID> with real values (TX_ID need not resolve to a real tx;
# the pool is exhausted before the 404 is returned).

TX_ID="0.0.1234-1700000000-000000000"
BASE="https://<MIRROR_HOST>/api/v1/contracts/results/${TX_ID}"

# Step 1: exhaust pool with /results requests (5 connections each via Promise.all)
for i in $(seq 1 10); do
  curl -s "${BASE}" &
done

# Step 2: simultaneously hammer /actions (2 connections each)
for i in $(seq 1 10); do
  curl -s "${BASE}/actions" &
done

wait

# Expected result: all requests either hang for ~20 s then return 500,
# or return connection-timeout errors, while legitimate API traffic is blocked.
```
