### Title
Unbounded Batch Query Amplification via Per-Contract Slot Key Cache Poisoning in `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally issues a DB `IN` query containing every slot key accumulated in the per-contract Caffeine cache (up to `maximumSize=1500` per contract) on every cache miss, regardless of how many keys are actually needed. An unprivileged caller can pre-fill this cache with up to 1500 non-existent slot keys, after which every single `findStorage()` invocation for that contract permanently triggers a 1500-key `IN` clause query against the database, because non-existent slots are never written back to `contractStateCache` and therefore never short-circuit the batch path.

### Finding Description

**Code location**: `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, `findStorageBatch()`, lines 85–122.

**Cache architecture** (from `CacheProperties.java`):
- `contractSlots` (outer cache, field `this.contractSlotsCache`): `expireAfterAccess=5m, maximumSize=3000` — maps `contractId → per-contract CaffeineCache`
- `slotsPerContract` (per-contract cache, local `contractSlotsCache`): `expireAfterAccess=5m, maximumSize=1500` — maps `ByteBuffer(slotKey) → EMPTY_VALUE`
- `contractState`: `expireAfterWrite=2s, maximumSize=25000` — maps `(contractId, slotKey) → value`

**Root cause — three compounding flaws**:

1. **Unconditional full-set batch query** (lines 90–103): Every call to `findStorageBatch()` adds the requested key via `putIfAbsent`, then immediately snapshots the *entire* key set of the per-contract cache and passes it to `findStorageBatch(contractId, cachedSlots)`. There is no filtering of keys whose values are already known or whose slots are non-existent.

2. **Non-existent slots never populate `contractStateCache`** (lines 106–113): Only slots returned by the DB query are written to `contractStateCache`. A slot that does not exist in `contract_state` is never cached, so the `contractStateCache.get(...)` check in `findStorage()` (line 63) always returns `null` for it, unconditionally routing every future call back through `findStorageBatch()`.

3. **Per-contract cache retains poisoned keys for 5 minutes** (`expireAfterAccess=5m`): Once the attacker fills the per-contract cache with 1500 non-existent keys, those keys persist for 5 minutes after last access. Because the attacker's own repeated calls keep refreshing the access time, the poisoned state can be maintained indefinitely.

**Exploit flow**:
- Step 1 (poison): Attacker deploys a contract on Hedera with a function that reads N distinct storage slots from calldata (e.g., via `SLOAD` in a loop). They call `POST /api/v1/contracts/call` (eth_call) with calldata encoding 1500 distinct non-existent slot indices. The EVM calls `findStorage()` 1500 times, each miss calls `findStorageBatch()`, accumulating all 1500 keys in the per-contract cache.
- Step 2 (sustain): Attacker periodically re-calls the same function to refresh `expireAfterAccess` on the per-contract cache entries, keeping the 1500 poisoned keys alive.
- Step 3 (amplify): Any subsequent `eth_call` that reads even a single slot of that contract misses `contractStateCache` (2s TTL) and triggers `findStorageBatch()` with all 1500 keys in the SQL `IN` clause. This applies to legitimate users querying the same contract.

**Why existing checks fail**:
- The `contractStateCache` 2-second TTL means legitimate traffic continuously re-enters `findStorageBatch()` after expiry, each time hitting the full 1500-key query.
- The `ThrottleManagerImpl` rate-limits at the HTTP request level (default 500 req/s), not at the internal `findStorage()` call level. A single `eth_call` can trigger dozens of `findStorage()` calls, each amplified to 1500 DB keys.
- The `slotsPerContract` `maximumSize=1500` cap bounds the amplification but does not prevent it.

### Impact Explanation
Every `findStorage()` cache miss for a poisoned contract issues a `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (?, ?, ..., ?)` with up to 1500 bind parameters instead of 1. This is a 1500× amplification of DB read load per cache miss. Because `contractStateCache` expires every 2 seconds, legitimate users querying the same contract are continuously subjected to these oversized queries. Under sustained attack across multiple contracts (up to 3000 contracts in the outer cache), the aggregate DB load can degrade query performance for all users of the mirror node.

### Likelihood Explanation
The attack requires no credentials, no privileged access, and no on-chain funds beyond the one-time cost of deploying a loop-SLOAD contract on Hedera mainnet (a few HBAR). The exploit is fully repeatable, scriptable, and can be sustained indefinitely by periodically refreshing the per-contract cache. The public `POST /api/v1/contracts/call` endpoint is the only entry point needed.

### Recommendation
1. **Do not batch-query keys whose values are already absent**: After a batch query returns, write a sentinel (e.g., `EMPTY_VALUE`) into `contractStateCache` for every key that was queried but not returned, so future calls for non-existent slots short-circuit at line 63 and never reach `findStorageBatch()`.
2. **Limit batch size per invocation**: Cap `cachedSlots` to a configurable maximum (e.g., 100) per `findStorageBatch()` call, prioritizing recently-accessed keys.
3. **Evict per-contract cache keys on confirmed non-existence**: Remove a slot key from the per-contract cache when the batch query confirms it does not exist in the DB, preventing indefinite accumulation of ghost keys.
