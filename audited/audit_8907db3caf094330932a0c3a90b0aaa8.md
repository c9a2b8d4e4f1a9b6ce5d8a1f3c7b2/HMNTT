### Title
Unbounded `token.id` Range Filter Enables Large Index Scans in `getTokenRelationships()`

### Summary
The `/api/v1/accounts/{id}/tokens` endpoint accepts `token.id=gte:X&token.id=lte:Y` range filters with no validation on the width of the range. Because the SQL query applies `LIMIT` only to rows returned — not to rows scanned — an unauthenticated attacker can force the database to perform a full index scan over an arbitrarily large portion of the `token_account` index for any valid account, with no rate-limiting or range-width guard in place.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()` (line 66): [1](#0-0) 

calls `extractTokensRelationshipQuery()` (line 22), which pushes `gte`/`lte` `TOKEN_ID` filters directly into `conditions` with no range-width check: [2](#0-1) 

Those conditions are assembled verbatim into SQL in `TokenService.getTokenRelationshipsQuery()`: [3](#0-2) 

The base query already fixes `account_id = $1 AND associated = true`, and the range conditions are appended as `ta.token_id >= $3 AND ta.token_id <= $4`. The `LIMIT $2` clause caps **returned** rows but does not cap **scanned** rows: [4](#0-3) 

**Root cause:** `buildAndValidateFilters` is called with no custom validator for this endpoint (the default `filterValidityChecks` is used), and neither `filterValidityChecks` nor `formatComparator` enforces any maximum span between a lower and upper `token.id` bound: [5](#0-4) 

There is no equivalent of `maxTimestampRange` for token ID ranges. The `maxRepeatedQueryParameters` guard only limits how many times the same parameter appears, not the numeric distance between bounds: [6](#0-5) 

**Why the `LIMIT` is insufficient:** With a B-tree index on `(account_id, token_id)`, the database seeks to `(account_id, X)` and scans forward. If the account has zero or very few `associated = true` rows in `[X, Y]`, the engine must traverse every index entry in that range before returning an empty (or small) result set. The `LIMIT` only terminates the scan once enough qualifying rows are found; it provides no protection when qualifying rows are sparse or absent.

### Impact Explanation

An attacker can issue repeated requests such as:
```
GET /api/v1/accounts/0.0.1001/tokens?token.id=gte:0.0.1&token.id=lte:0.0.9999999999
```
Each request forces a full index range scan on `token_account` for the target account. Concurrent requests from multiple source IPs amplify the I/O and CPU load on the database, degrading response times for all users of the mirror node. No economic damage to on-chain users occurs, but the mirror node REST API becomes unavailable or severely degraded — consistent with the "griefing" classification.

### Likelihood Explanation

No authentication or API key is required. The endpoint is publicly documented in the OpenAPI spec: [7](#0-6) 

The attack requires only knowledge of one valid account ID (trivially obtained from any other public endpoint) and the ability to send HTTP GET requests. It is fully repeatable and scriptable with commodity tools.

### Recommendation

1. **Enforce a maximum token ID range width** in `extractTokensRelationshipQuery()`: if both a lower (`gte`/`gt`) and upper (`lte`/`lt`) `token.id` bound are present, reject the request when `upper_value - lower_value` exceeds a configured maximum (analogous to `maxTimestampRange`).
2. **Add a config key** (e.g., `query.maxTokenIdRange`) and validate it in `parseQueryConfig()` alongside the existing duration-based range guards.
3. As a secondary defence, apply per-IP rate limiting at the ingress layer for this endpoint.

### Proof of Concept

**Preconditions:** Any valid account ID (e.g., `0.0.1001`) that exists in the mirror node database.

**Steps:**
```bash
# Single large-range request
curl "https://<mirror-node>/api/v1/accounts/0.0.1001/tokens?token.id=gte:0.0.1&token.id=lte:0.0.9999999999"

# Flood with concurrent requests to amplify DB load
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/tokens?token.id=gte:0.0.1&token.id=lte:0.0.9999999999" &
done
wait
```

**Expected result:** Each request triggers a full index range scan on `token_account` for `account_id = 1001` across the entire token ID space. Under concurrent load, database CPU and I/O spike, causing latency degradation or timeouts for all mirror node API consumers.
