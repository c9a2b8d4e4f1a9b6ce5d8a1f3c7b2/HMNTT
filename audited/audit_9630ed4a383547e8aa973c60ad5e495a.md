### Title
Unauthenticated BLOCK_NUMBER Filter Triggers Uncached DB Lookup Per Request, Exhausting the REST Service Connection Pool

### Summary
Every request to `/api/v1/contracts/results?block.number=<N>` or `/api/v1/contracts/<id>/results?block.number=<N>` unconditionally invokes `RecordFileService.getRecordFileBlockDetailsFromIndex()`, which issues a live database query with no caching. The REST service's connection pool defaults to only 10 connections. An unprivileged attacker flooding these endpoints with concurrent requests can exhaust the pool, causing all subsequent legitimate database operations across the REST service to queue or time out.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `extractContractResultsByIdQuery()` (lines 486–492):

```js
if (blockFilter) {
  let blockData;
  if (blockFilter.key === filterKeys.BLOCK_NUMBER) {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromIndex(blockFilter.value);
  } else {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromHash(blockFilter.value);
  }
``` [1](#0-0) 

This is called on every invocation of `getContractResults` (line 1067) and `getContractResultsById` (line 871), both of which accept `BLOCK_NUMBER` as a filter per `acceptedContractResultsParameters` (lines 1336–1346). [2](#0-1) 

`RecordFileService.getRecordFileBlockDetailsFromIndex()` in `rest/service/recordFileService.js` (lines 131–135) executes a direct database query every time — there is **no caching**:

```js
async getRecordFileBlockDetailsFromIndex(index) {
  const row = await super.getSingleRow(RecordFileService.recordFileBlockDetailsFromIndexQuery, [index]);
  return row === null ? null : new RecordFile(row);
}
``` [3](#0-2) 

The query executed is:

```sql
select consensus_start, consensus_end, hash, index
from record_file
where index = $1
limit 1
``` [4](#0-3) 

**Root cause:** The REST Node.js service has no caching layer for this lookup (contrast: the web3 Java `RecordFileRepository.findByIndex()` uses `@Cacheable` with a Caffeine cache). Every request with a `block.number` filter consumes a database connection for the duration of the query.

**Failed assumption:** The design assumes either (a) the query is cheap enough to not matter, or (b) infrastructure-level rate limiting exists. Neither is guaranteed: the default pool is only 10 connections, and no application-level rate limiting is present in the REST service.

### Impact Explanation

The REST service connection pool defaults to **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`): [5](#0-4) 

The pool is initialized in `rest/dbpool.js` with `max: config.db.pool.maxConnections`: [6](#0-5) 

With 10+ concurrent requests each holding a connection for the `getRecordFileBlockDetailsFromIndex` query, the pool is exhausted. All other REST API operations (account lookups, transaction queries, etc.) that also require a DB connection will queue or fail with connection timeout errors (`connectionTimeoutMillis` default: 20 seconds). This constitutes a denial of service against the entire REST API, not just the contracts endpoint — matching the stated severity of ≥30% network processing node disruption.

### Likelihood Explanation

- No authentication or API key is required for either endpoint.
- The attack requires only a standard HTTP client capable of sending concurrent requests.
- The attacker can use any valid or invalid block number; even non-existent block numbers trigger the DB query (the `null` result is only checked after the query returns).
- The attack is trivially repeatable and scriptable (e.g., `ab -n 1000 -c 50 'https://host/api/v1/contracts/results?block.number=1'`).
- The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the **web3 Java service**, not the REST Node.js service. [7](#0-6) 

### Recommendation

1. **Add an in-process cache** for `getRecordFileBlockDetailsFromIndex()` in the REST service (e.g., an LRU map keyed by block index), mirroring the Caffeine cache used in the web3 Java `RecordFileRepository`.
2. **Add application-level rate limiting** to the REST Node.js service for the `/contracts/results` family of endpoints, similar to the `bucket4j`-based throttle in the web3 service.
3. **Increase the default pool size** or document that operators must tune it for production workloads.
4. As a short-term mitigation, validate that the block number is within a plausible range (e.g., ≤ current max block index) before issuing the DB query, to reduce the attack surface from arbitrary large integers.

### Proof of Concept

```bash
# Send 50 concurrent requests with a block.number filter (no auth required)
# Each request triggers an uncached DB query, exhausting the 10-connection pool

for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results?block.number=$i" &
done
wait

# Immediately after, legitimate requests to any REST endpoint will time out
# or return 500 errors due to connection pool exhaustion:
curl -v "https://<mirror-node-host>/api/v1/accounts/0.0.1"
# Expected: connection timeout or pool exhaustion error
```

The attack is amplified by using different block numbers per request (preventing any future caching from helping) and by keeping connections alive with slow HTTP clients.
