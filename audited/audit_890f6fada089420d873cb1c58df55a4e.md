### Title
Gas Throttle Bypass via Deliberate OOG Failure in `restoreGasToBucket()`

### Summary
In `ContractCallService.restoreGasToBucket()`, when a transaction fails with `gasUsed == gasLimit` (e.g., an Out-of-Gas exception), the method restores `gasLimitRefundPercent` (default: 100%) of the gas limit back to the throttle bucket. With the default configuration, this results in a net gas cost of zero to the bucket for any OOG-failing transaction. An unprivileged attacker can deliberately craft OOG transactions at the maximum gas limit, bypassing the gas throttle entirely and submitting far more computationally expensive requests than the throttle is designed to allow.

### Finding Description

**Code location:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractCallService.java`, `restoreGasToBucket()`, lines 140–152; `doProcessCall()`, lines 109–138.

**Root cause:** The `restoreGasToBucket()` method applies a special refund path for transactions where `!result.isSuccessful() && gasLimit == result.gasUsed()`:

```java
// ContractCallService.java:143-145
final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);
```

With `gasLimitRefundPercent = 100` (the default in `ThrottleProperties.java:22`), `gasLimitToRestoreBaseline == gasLimit`. The throttle consumed `gasLimit` tokens on entry (`ThrottleManagerImpl.java:40`), and restores exactly `gasLimit` tokens on exit. **Net gas cost to the bucket = 0.**

**Exploit flow:**
1. On every request, `ThrottleManagerImpl.throttle()` consumes `scaleGas(gasLimit)` tokens from the gas bucket.
2. The attacker submits a `eth_call`/`eth_estimateGas` request targeting a contract that loops until OOG (or any call that exhausts all provided gas). This is trivially achievable with any infinite-loop contract or by providing a gas value just below what the contract needs.
3. The EVM runs until it exhausts all gas: `gasUsed == gasLimit`.
4. `doProcessCall()` catches the `MirrorEvmTransactionException`, sets `result = e.getResult()`, and in the `finally` block calls `restoreGasToBucket(result, params.getGas())`.
5. The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` is true, so `gasLimitToRestoreBaseline` (= `gasLimit` at 100%) is restored.
6. The gas bucket is back to where it started. The attacker can immediately repeat.

The `result == null` branch (line 144) also triggers a full restore when pre-EVM validation throws `IllegalStateException`/`IllegalArgumentException` (line 120–121), though that path involves no EVM computation.

**Why existing checks are insufficient:**
- The RPS throttle (`rateLimitBucket`, 500 req/s) still applies, but it is a separate, independent bucket. It does not account for per-request computational cost.
- The gas throttle (`gasLimitBucket`) is the mechanism designed to cap total EVM computation per second. The refund logic completely neutralizes it for OOG failures.
- There is no authentication, IP rate-limiting, or request signing requirement on the `/api/v1/contracts/call` endpoint.

### Impact Explanation

The gas throttle is the primary defense against computationally expensive request floods. With `gasLimitRefundPercent = 100` (default), an attacker can submit 500 req/s (the RPS limit) of 15M-gas OOG transactions. Each request causes the EVM to execute 15M gas worth of opcodes. The gas throttle, configured to allow e.g. 1.5B gas/s (per documentation), would normally cap this at ~100 req/s at max gas. The bypass allows 5× more EVM computation than intended, enabling a sustained CPU exhaustion / DoS attack on the mirror node with no special privileges.

### Likelihood Explanation

Any unauthenticated HTTP client can reach the contract call endpoint. Crafting an OOG transaction requires only deploying or targeting any contract with a loop (or simply providing insufficient gas for a known contract). The attack is trivially scriptable, repeatable at 500 req/s, and requires no on-chain funds or privileged access. The default `gasLimitRefundPercent = 100` means the bypass is active in all default deployments.

### Recommendation

1. **Charge actual gas used for OOG failures.** When `gasUsed == gasLimit` due to OOG, the EVM did perform real computation. The bucket should not be fully refunded. Instead, treat OOG the same as a revert: restore only `min(gasLimit - gasUsed, gasLimitRefundPercent% * gasLimit)`. Since `gasUsed == gasLimit` in OOG, `gasRemaining = 0`, so the restore would be 0 — correctly charging the full gas cost.
2. **Distinguish OOG from other exceptional halts.** If leniency for non-OOG exceptional halts is desired, inspect the specific failure reason from `EvmTransactionResult` and only apply the refund for non-OOG halt types.
3. **Lower the default `gasLimitRefundPercent`.** A value of 100 means zero net cost for any OOG failure. A lower value (e.g., 25–50%) reduces the bypass window even if the OOG distinction is not implemented.
4. **Add per-IP or per-sender rate limiting** as a defense-in-depth measure independent of the gas bucket.

### Proof of Concept

**Preconditions:** Access to the mirror node's `/api/v1/contracts/call` endpoint. A deployed contract with an infinite loop (or any contract address — just provide gas below what the call needs).

**Steps:**

```bash
# 1. Craft a call that will OOG: target any contract, provide max gas
#    The contract runs until gas exhaustion -> gasUsed == gasLimit

for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "data": "0x<selector-of-infinite-loop-or-expensive-function>",
      "to": "0x<contract-address>",
      "gas": 15000000,
      "estimate": false
    }' &
done
wait

# 2. Observe: all 500 requests are processed (RPS limit hit, not gas limit)
# 3. Repeat every second: gas bucket is never depleted because each OOG
#    failure restores 100% of 15,000,000 gas back to the bucket.
# 4. The mirror node EVM executes 500 * 15M = 7.5B gas/s of computation,
#    while the gas throttle intended to cap this at a lower value.
```

**Expected result without the bug:** After the first ~100 requests (at 1.5B gas/s limit), subsequent requests should be rejected with "Gas per second rate limit exceeded."

**Actual result:** All 500 req/s succeed because the gas bucket is fully restored after each OOG failure, and only the RPS bucket is depleted.
