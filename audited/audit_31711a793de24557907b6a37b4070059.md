### Title
Cache Pollution DoS via Unbounded User-Controlled Timestamp Keys in `cacheManagerRecordFileTimestamp`

### Summary
The `cacheManagerRecordFileTimestamp` cache in `EvmConfiguration.java` uses the raw user-supplied `timestamp` parameter as its cache key with a hard cap of 10,000 entries. Because no per-IP or per-user rate limiting exists — only a shared global bucket — an unauthenticated attacker can flood the cache with distinct timestamp values, evicting legitimate entries and forcing repeated database queries for all other users.

### Finding Description
**Code location:**
- Cache definition: `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 166–176
- Cache consumer: `web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 38–46

The cache is configured as:

```java
// EvmConfiguration.java:168-171
final var caffeine = Caffeine.newBuilder()
        .expireAfterAccess(10, TimeUnit.MINUTES)
        .maximumSize(10000)
        .recordStats();
``` [1](#0-0) 

It is populated by `RecordFileRepository.findByTimestamp(long timestamp)`, where the cache key is the raw `timestamp` argument:

```java
// RecordFileRepository.java:38-46
@Caching(
        cacheable = @Cacheable(
                cacheNames = CACHE_NAME,
                cacheManager = CACHE_MANAGER_RECORD_FILE_TIMESTAMP,
                unless = "#result == null"),
        put = @CachePut(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX))
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
``` [2](#0-1) 

**Root cause — two compounding failures:**

1. **`unless = "#result == null"` does not prevent caching of misses.** The method returns `Optional<RecordFile>`. An `Optional.empty()` is not `null`, so the condition `#result == null` is always `false` for any return value, including empty Optionals. Every call — whether the timestamp exists in the DB or not — produces a cached entry. An attacker does not need valid timestamps.

2. **Rate limiting is global, not per-IP/per-user.** The only throttle is a shared `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` in `ThrottleManagerImpl`. There is no per-source-IP or per-user isolation. [3](#0-2) [4](#0-3) 

**Exploit flow:**

1. Attacker sends POST `/api/v1/contracts/call` requests with the `block` field set to distinct historical block numbers (or any numeric value that resolves to a distinct timestamp lookup).
2. Each request passes the global rate limiter (shared with all users) and triggers `RecordFileRepository.findByTimestamp(timestamp)`.
3. Each distinct timestamp value occupies one cache slot. After 10,000 distinct values, Caffeine's LRU eviction begins, expelling legitimate entries.
4. The attacker cycles through a rolling window of >10,000 distinct timestamps, keeping the cache permanently polluted.
5. Every legitimate user request now misses the cache and hits the database.

The `OpcodeServiceImpl` also calls `recordFileService.findByTimestamp(consensusTimestamp)` directly, providing a second entry point (though its rate limit is 1 req/s by default). [5](#0-4) 

### Impact Explanation
Cache misses on `findByTimestamp` cause the DB query `select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1` to execute on every request. Under sustained cache pollution, all users experience this DB hit on every historical block lookup, increasing database load proportionally to request volume. At the default 500 req/s global limit, the attacker can fill the entire 10,000-entry cache in ~20 seconds and maintain the polluted state indefinitely. The attacker simultaneously consumes the shared rate-limit budget, degrading throughput for all legitimate users. Severity: **Medium** (availability degradation, no data exfiltration).

### Likelihood Explanation
No authentication, API key, or IP-based restriction is required. Any internet-accessible deployment is reachable. The attack requires only standard HTTP POST requests with varying numeric `block` parameters. It is trivially scriptable, repeatable, and can be sustained indefinitely from a single host. The global rate limiter does not prevent a single attacker from monopolizing the entire request budget.

### Recommendation
1. **Fix the `unless` condition** to also exclude empty Optionals: `unless = "#result == null || !#result.isPresent()"`. This prevents cache pollution from non-existent timestamps.
2. **Add per-IP rate limiting** (e.g., via a `ConcurrentHashMap<String, Bucket>` keyed on remote IP) in `ThrottleManagerImpl` or a servlet filter, so a single source cannot consume the entire global budget.
3. **Validate timestamp ranges** before the cache lookup — reject timestamps outside the known chain range (earliest to latest block) to reduce the effective key space.
4. **Consider a TTL-based eviction** (`expireAfterWrite`) in addition to `expireAfterAccess` to bound the lifetime of attacker-inserted entries even if the attacker keeps accessing them.

### Proof of Concept
```bash
# Fill the cache with 10,001 distinct timestamps (all cache misses, all cached as Optional.empty())
for i in $(seq 1 10001); do
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"0x0000000000000000000000000000000000000001\",
         \"data\":\"0x\",
         \"block\":\"$i\"}" &
done
wait

# Now legitimate users querying real block timestamps will miss the cache and hit the DB on every request.
# Verify by observing DB query rate increase or cache hit-rate metrics (recordStats() is enabled).
```
