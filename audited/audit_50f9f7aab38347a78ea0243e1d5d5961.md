### Title
Unbounded Query Parameter Passthrough in `addParamMapToQueryParams()` Enables Per-Response Memory and CPU Amplification

### Summary
`createNextLink()` in `LinkFactoryImpl` calls `request.getParameterMap()` and passes the entire raw servlet parameter map — including all unknown/arbitrary parameters — to `addParamMapToQueryParams()` without any size bound. Because the `rest-java` request processing layer (`RequestParameterArgumentResolver`) silently ignores unknown parameters rather than rejecting them, an unauthenticated attacker can inject arbitrarily many query parameters that survive into the pagination link-building phase, causing unbounded iteration and allocation on every paginated response.

### Finding Description

**Code path:**

`LinkFactoryImpl.createNextLink()` at [1](#0-0)  retrieves the raw servlet parameter map with no size check and passes it directly to `addParamMapToQueryParams()`.

`addParamMapToQueryParams()` at [2](#0-1)  iterates over every entry in `paramsMap` and unconditionally adds each key/value pair into a freshly allocated `LinkedMultiValueMap<String, String>`. There is no guard on the number of entries.

**Root cause — unknown parameters are not rejected upstream:**

`RequestParameterArgumentResolver.resolveArgument()` at [3](#0-2)  only iterates over the DTO's declared `@RestJavaQueryParam` fields. Parameters that do not match any declared field are silently ignored — no 400 error is raised for unknown parameters. This is the opposite of the Node.js `rest` module, which explicitly rejects unknown parameters via `validateReq()`.

**Failed assumption:** The code assumes `request.getParameterMap()` will only contain a small, bounded set of known parameters. In reality, Tomcat's default `maxParameterCount` is 10,000, and no lower limit is configured anywhere in the `rest-java` resources. [4](#0-3) 

**Per-field `@Size` constraints** (e.g., `@Size(max = 2)` on `registeredNodeIds`) only limit repetitions of a single *known* parameter, not the total number of distinct unknown parameters. [5](#0-4) 

### Impact Explanation

On every paginated response (i.e., whenever `items.size() == pageable.getPageSize()`), the server:
1. Allocates a new `LinkedMultiValueMap` and copies all N attacker-supplied parameters into it.
2. Iterates over all N entries in `addParamMapToQueryParams()`.
3. Builds a URI string containing all N parameters and writes it into the `Link` response header.

With N approaching Tomcat's default limit of 10,000, this creates significant per-response heap pressure and CPU overhead. The generated `Link` header URL can also exceed downstream HTTP header size limits, causing proxy or client failures. The attacker can sustain this by continuously following the returned next-page link (which itself carries all injected parameters forward).

### Likelihood Explanation

The endpoint is public and unauthenticated. The attack requires only a standard HTTP GET request with many query string parameters — no special privileges, tokens, or protocol knowledge. It is trivially scriptable and repeatable. Any paginated endpoint in `rest-java` (e.g., NFT allowances, registered nodes, hooks) is affected.

### Recommendation

1. **Reject unknown parameters early** in `RequestParameterArgumentResolver` (or a dedicated filter/interceptor): compare `request.getParameterMap().keySet()` against the set of declared `@RestJavaQueryParam` names and return HTTP 400 for any unrecognized key, matching the behavior of the Node.js `rest` module.
2. **Cap total parameter count** via `server.tomcat.max-parameter-count` in application configuration (e.g., set to 20–50).
3. As a defense-in-depth measure, add a size check at the top of `addParamMapToQueryParams()` and throw an `IllegalArgumentException` if `paramsMap.size()` exceeds a small constant (e.g., 20).

### Proof of Concept

```bash
# Generate a request with 500 unknown query parameters to a paginated endpoint
PARAMS=$(python3 -c "print('&'.join(f'x{i}=v{i}' for i in range(500)))")

# Send to any paginated rest-java endpoint (adjust host/path as needed)
curl -v "http://localhost:8084/api/v1/accounts/0.0.1001/allowances/nfts?limit=1&${PARAMS}"

# Observe:
# 1. Request succeeds (200 OK) — unknown params are not rejected
# 2. Link response header contains all 500 injected parameters
# 3. Repeat by following the next link — all 500 params are preserved
# 4. Each response triggers a full 500-entry LinkedMultiValueMap allocation
```

    private void addParamMapToQueryParams(
            Map<String, String[]> paramsMap,
            Map<String, String> paginationParamsMap,
            Direction order,
            LinkedMultiValueMap<String, String> queryParams) {
        for (var entry : paramsMap.entrySet()) {
            var key = entry.getKey();
            if (!paginationParamsMap.containsKey(key)) {
                for (var value : entry.getValue()) {
                    queryParams.add(entry.getKey(), value);
                }
            } else {
                addQueryParamToLink(entry, order, queryParams);
            }
        }
    }
