### Title
Unauthenticated Unbounded Subscription Flood in `subscribeTopic()` Enables gRPC Server Exhaustion

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent streaming subscriptions from any unauthenticated caller with no per-IP, per-client, or global rate limiting. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits streams per HTTP/2 connection, not the number of connections, so an attacker opening thousands of TCP connections multiplies that limit arbitrarily. Each subscription allocates a Reactor pipeline, a DB retriever poll chain, and a listener slot, exhausting the thread pool and database connection pool and starving legitimate traffic.

### Finding Description
**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) immediately subscribes a new Reactor pipeline for every inbound call with no guard:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java:43-53
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            .map(this::toResponse)
            .onErrorMap(ProtoUtil::toStatusRuntimeException)
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    ...
}
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) allocates a historical retriever flux, a live listener flux, and a safety-check flux per subscription, incrementing `subscriberCount` with no ceiling:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java:59-91
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

`subscriberCount` is a metric gauge only — it enforces nothing.

**The only server-side constraint** is set in `GrpcConfiguration` (line 33):

```java
// grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java:28-34
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

with default value:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java:14
private int maxConcurrentCallsPerConnection = 5;
```

This is an HTTP/2 `MAX_CONCURRENT_STREAMS` setting scoped to a single TCP connection. It does not limit the number of TCP connections an attacker may open.

**No rate limiting exists in the gRPC module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` stack lives exclusively in the `web3` module and is never wired into the `grpc` module. The sole `ServerInterceptor` registered in the gRPC module (test path `GrpcInterceptor`) only sets an `EndpointContext` tag and calls `next.startCall()` unconditionally — no throttle, no auth.

**Root cause:** The design assumes network-layer controls (load balancer, firewall) will enforce connection limits, but no such enforcement is present in the application itself, and the question scope is the application code.

### Impact Explanation
Each subscription consumes:
- A bounded-elastic scheduler thread (Reactor `subscribeOn(Schedulers.boundedElastic())` in the safety-check path)
- A DB connection from the R2DBC pool (historical retriever + gap-fill queries)
- A Redis/poll listener slot (live message path)

Flooding with N connections × 5 streams = 5N concurrent subscriptions exhausts the thread pool and DB connection pool. Once the pool is saturated, legitimate `subscribeTopic` calls queue indefinitely or fail, and the shared DB pool starvation also degrades the importer's ability to write new consensus data, indirectly blocking transaction confirmation visibility. Severity: **Critical** — complete denial of the gRPC API surface with spillover to the data pipeline.

### Likelihood Explanation
Preconditions: none. The endpoint is plaintext by default (port 5600, no TLS required, no authentication). A single attacker machine with a standard gRPC client library (e.g., `grpcurl` in a loop, or a Go/Java client opening parallel channels) can open thousands of HTTP/2 connections in seconds. The attack is stateless, requires no prior knowledge beyond the public protobuf schema, and is trivially repeatable. The `limit: 0` field (unlimited messages) keeps each subscription alive indefinitely, preventing natural cleanup.

### Recommendation
1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and throw a `StatusRuntimeException(RESOURCE_EXHAUSTED)` before allocating any pipeline resources.
2. **Add a per-IP concurrent-subscription limit** via a `ServerInterceptor` that tracks active calls per remote address and rejects excess calls with `RESOURCE_EXHAUSTED`.
3. **Add a new-subscription rate limiter** (e.g., Bucket4j token bucket, mirroring the `web3` `ThrottleConfiguration`) in a `GlobalServerInterceptor` applied to the gRPC module, limiting new `subscribeTopic` calls per second globally and/or per source IP.
4. **Set `maxConnectionAge`** on the `NettyServerBuilder` to recycle long-lived idle connections and prevent indefinite resource hold.
5. Add `maxConcurrentCallsPerConnection` to `NettyProperties` alongside a new `maxConnections` field and wire both into `GrpcConfiguration`.

### Proof of Concept
```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Open 1000 parallel indefinite subscriptions from a single host:

for i in $(seq 1 1000); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Observe: legitimate subscribeTopic calls now hang or fail with UNAVAILABLE.
# Server metrics will show DB connection pool exhausted and thread pool saturated.
# subscriberCount gauge will climb to 1000+ with no server-side rejection.
```
