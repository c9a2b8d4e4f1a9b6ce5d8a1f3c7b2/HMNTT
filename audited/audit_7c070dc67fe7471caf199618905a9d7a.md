### Title
Connection Pool Exhaustion via Unbounded Concurrent `Promise.all()` DB Queries in `getContractResults()` with No Client-Side Query Timeout

### Summary
`getContractResults()` in `rest/controllers/contractController.js` executes a two-phase database pattern with no rate limiting and no client-side query timeout. Each request holds up to two simultaneous pool connections during `Promise.all()`. With a default pool ceiling of 10 connections and only a server-side `statement_timeout`, a network partition silences the PostgreSQL cancellation signal, leaving all in-flight Promises permanently hung and every pool slot permanently occupied, causing a complete service outage for all users.

### Finding Description

**Phase 1 — sequential query (1 connection):**
`rest/controllers/contractController.js` lines 1072:
```js
const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
```

**Phase 2 — parallel queries (2 connections simultaneously):**
`rest/controllers/contractController.js` lines 1083–1086:
```js
const [ethereumTransactionMap, recordFileMap] = await Promise.all([
  ContractService.getEthereumTransactionsByPayerAndTimestampArray(payers, timestamps),
  RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps),
]);
```

**Pool configuration** (`rest/dbpool.js` lines 7–16):
```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // 20 000 ms — new-connection wait only
  max: config.db.pool.maxConnections,                         // default: 10
  statement_timeout: config.db.pool.statementTimeout,         // default: 20 000 ms — PostgreSQL server-side
};
```

There is **no `query_timeout`** (node-postgres client-side per-query deadline) anywhere in the pool config. `statement_timeout` is a PostgreSQL session variable; the server enforces it and sends a cancellation notice back over the TCP connection. During a network partition that connection is silent — the cancellation never arrives at the client, so the awaited Promise never settles.

`connectionTimeoutMillis` governs only the time spent *waiting to acquire* a free slot from the pool. It does not abort a query that is already executing on a checked-out connection.

No rate-limiting middleware exists for the REST API path (confirmed: `grep` for `rateLimit|throttle` in `rest/**/*.js` returns only a test-utility hit, not production middleware).

**Exploit flow:**
1. Attacker sends ≥ 5 concurrent `GET /api/v1/contracts/results` requests (no authentication, no rate limit).
2. Each request passes Phase 1 and enters Phase 2, checking out 2 pool connections for `Promise.all()`.
3. 5 concurrent requests × 2 connections = 10 connections — pool is fully saturated.
4. A network partition (BGP disruption, physical cut, or attacker-controlled network-layer interference) silences the TCP path to PostgreSQL.
5. All 10 in-flight `Promise.all()` calls hang indefinitely; no client-side timeout fires.
6. Every subsequent request blocks waiting for a free connection; after `connectionTimeoutMillis` (20 s) each new request fails with a pool-timeout error.
7. The service is completely unavailable to all users until the partition resolves or the process is restarted.

### Impact Explanation
The default pool size of 10 connections means only 5 simultaneous requests in the `Promise.all()` phase are needed to saturate the pool. Because Node.js is single-threaded and the event loop is not blocked (the Promises are simply never resolved), the process remains alive but every incoming request immediately queues and then times out. This is a **complete, sustained service outage** — not degraded performance — for the entire user base, triggered by an unprivileged attacker with no credentials.

### Likelihood Explanation
The preconditions are realistic:
- **No authentication or rate limiting** on `GET /api/v1/contracts/results` means any external actor can issue unlimited concurrent requests.
- **Network partitions** occur naturally (cloud provider incidents, BGP misconfigurations) and can also be induced by a sufficiently resourced attacker (network-layer DDoS targeting the DB subnet, ARP poisoning in shared environments).
- The pool ceiling of 10 is very low; 5 concurrent requests is a trivial load for any automated tool.
- The attack is **repeatable**: once the partition clears, the attacker can immediately re-flood to re-exhaust the pool.

### Recommendation
1. **Add a client-side `query_timeout`** to the pool config in `rest/dbpool.js`:
   ```js
   query_timeout: config.db.pool.statementTimeout, // enforced by node-postgres, independent of network
   ```
   This fires client-side regardless of whether the server's cancellation notice is deliverable.
2. **Add per-IP or global concurrency/rate limiting** middleware on the REST API (e.g., `express-rate-limit` or an ingress-level policy) to cap simultaneous in-flight requests per client.
3. **Increase the pool size** or introduce a per-request connection-hold deadline so that a saturated pool sheds load rather than queuing indefinitely.
4. **Set TCP keepalive** on pool connections (`keepAlive: true`, `keepAliveInitialDelayMillis`) so the OS detects a dead network path and closes the socket, allowing node-postgres to reject the pending query.
