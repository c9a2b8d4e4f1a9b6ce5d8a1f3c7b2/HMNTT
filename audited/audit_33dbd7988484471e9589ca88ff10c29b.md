### Title
Unauthenticated Full-Table Scan DoS via Unfiltered GET /accounts Exhausting the DB Connection Pool

### Summary
The `getAccounts()` handler in `rest/accounts.js` accepts requests with no query filters and generates a full-table scan against the `entity` table (joined with `token_account` via CTE). The REST Node.js server has no rate-limiting middleware, and the default database connection pool is only 10 connections with a 20-second statement timeout. A flood of concurrent unfiltered requests from any unauthenticated user can exhaust the pool, blocking all other REST API queries including transaction lookups.

### Finding Description

**Code path:**

`getAccounts()` at `rest/accounts.js:335-388` calls `utils.parseLimitAndOrderParams` (capped at `responseLimit.max = 100`) and `utils.parseAccountIdQueryParam`, `parseBalanceQueryParam`, `parsePublicKeyQueryParam`. When no filters are supplied, all three return empty query fragments. [1](#0-0) 

This flows into `getEntityBalanceQuery` at `rest/accounts.js:133-236`, which builds:

```sql
with latest_token_balance as (
  select account_id, balance, token_id from token_account where associated is true
)
select [entityFields], [tokenBalance subquery], e.balance, e.balance_timestamp
from entity as e
left join entity_stake as es on es.id = e.id
where e.type in ('ACCOUNT', 'CONTRACT')
  and NOT(e.num = 2 and e.memo = 'Mirror node created synthetic treasury account')
order by e.id ASC
limit 100
``` [2](#0-1) 

On mainnet the `entity` table contains millions of rows. PostgreSQL must scan the entire table to satisfy the `ORDER BY e.id LIMIT 100` with no selective index predicate, and the `latest_token_balance` CTE scans all of `token_account`.

**No rate limiting in the REST API:**

`rest/server.js` registers only: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler` (custom response-size limits only), optional `metricsHandler`, optional `responseCacheCheckHandler`, then route handlers. [3](#0-2) 

There is no per-IP or global request-rate-limiting middleware. The `ThrottleManager`/`ThrottleConfiguration` classes exist only in the `web3` Java module and do not apply here.

**DB pool constraints:**

Default `hiero.mirror.rest.db.pool.maxConnections = 10`, `statementTimeout = 20000 ms`. [4](#0-3) 

Each unfiltered `getAccounts` query holds a connection for the full duration of the table scan (up to 20 s). With 10 concurrent requests, the pool is fully occupied. All subsequent queries — including transaction lookups — queue behind them.

**Limit cap is insufficient:**

`getLimitParamValue` caps at `responseLimit.max` (default 100), which only limits rows returned, not the scan cost. [5](#0-4) 

### Impact Explanation

With the pool of 10 connections saturated for up to 20 seconds per wave of requests, every other REST API endpoint (`/transactions`, `/balances`, etc.) queues indefinitely or times out. On a mainnet-scale deployment (millions of entities), each unfiltered query is a multi-second full sequential scan plus a full `token_account` CTE scan. This constitutes a complete REST API denial-of-service achievable by a single attacker with a modest HTTP client. Transaction query latency increases by many multiples of the normal response time (well beyond 500% of the ~3-second Hedera consensus round).

### Likelihood Explanation

No authentication, no rate limiting, no mandatory filter parameter, and no query complexity guard are required. Any attacker with network access can issue 10–20 concurrent `GET /api/v1/accounts` requests in a loop using `curl`, `ab`, or any scripting language. The attack is trivially repeatable and requires no special knowledge of the system.

### Recommendation

1. **Add a mandatory filter or require at least one selective filter** (e.g., `account.id` range) for the unfiltered list endpoint, or add a minimum `account.id` lower-bound cursor requirement.
2. **Add application-level rate limiting** (e.g., `express-rate-limit`) in `rest/server.js` before route handlers, scoped per source IP.
3. **Increase `maxConnections`** and/or add a **concurrency limiter** (e.g., `express-slow-down` or a semaphore) specifically for expensive list endpoints.
4. **Reduce `statementTimeout`** for the REST API pool to a lower value (e.g., 5 s) to bound the blast radius of each slow query.
5. **Add a PostgreSQL partial index** on `entity(id) WHERE type IN ('ACCOUNT','CONTRACT')` to make the ordered scan cheaper.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent unfiltered scans
for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/accounts" &
done
wait

# Simultaneously measure transaction query latency (should spike to 20s+)
time curl "https://<mirror-node-host>/api/v1/transactions?limit=1"
```

Preconditions: network access to the REST API, no credentials needed.
Trigger: 10+ concurrent `GET /api/v1/accounts` with no query parameters.
Result: transaction and all other REST queries stall until the pool drains (up to 20 s per wave), effectively rendering the API unavailable for the duration of the attack.
