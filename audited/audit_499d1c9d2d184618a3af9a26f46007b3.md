### Title
Connection Pool Exhaustion via Unauthenticated High-Concurrency Requests to `/accounts/:idOrAliasOrEvmAddress/nfts`

### Summary
The Node.js REST service exposes the `/:idOrAliasOrEvmAddress/nfts` route with no per-IP or global rate limiting middleware. The underlying PostgreSQL connection pool defaults to only 10 connections with a 20-second statement timeout. An unprivileged attacker sending as few as 10 concurrent requests can saturate the entire pool, causing all other users' requests to queue for up to 20 seconds or time out, degrading service availability for the entire network.

### Finding Description

**Route registration** — `rest/routes/accountRoute.js` line 15:
```js
router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
```
No rate-limiting middleware is applied at the route or router level. [1](#0-0) 

**Handler** — `rest/controllers/accountController.js` lines 90–103: each request calls `EntityService.getEncodedId()` (a DB lookup) and then `NftService.getNfts(query)` (a potentially multi-union SQL query), both of which acquire connections from the shared global pool. [2](#0-1) 

**Pool configuration** — `rest/dbpool.js` lines 7–16: the pool is bounded by `config.db.pool.maxConnections` (default **10**) and `connectionTimeoutMillis` (default **20,000 ms**). [3](#0-2) 

**Default values confirmed** in documentation:
- `hiero.mirror.rest.db.pool.maxConnections` = **10**
- `hiero.mirror.rest.db.pool.connectionTimeout` = **20000 ms**
- `hiero.mirror.rest.db.pool.statementTimeout` = **20000 ms** [4](#0-3) 

**Middleware stack** — `rest/server.js` lines 68–144: the full middleware chain is `httpContext → requestLogger → authHandler → metricsHandler (optional) → responseCacheCheckHandler (optional) → routes`. There is no rate-limiting, concurrency-limiting, or per-IP throttling middleware anywhere in the chain for the Node.js REST service. [5](#0-4) 

The `authHandler` only adjusts per-user response `limit` values; it does not throttle request rates. The throttle/rate-limit code (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the separate `web3` Java service and is not applied here. [6](#0-5) 

**Root cause**: The failed assumption is that infrastructure-level controls (pgbouncer, Kubernetes ingress) will always be present. In the default code path, there is no application-level guard preventing a single unauthenticated client from holding all 10 pool connections simultaneously for up to 20 seconds each.

### Impact Explanation
With a pool of 10 connections and a 20-second statement timeout, an attacker holding all 10 connections forces every other concurrent request to queue in the `pg` pool's internal wait queue for up to 20 seconds before either being served or receiving a connection-timeout error. This degrades response times for all users of the REST API to near-unusable levels for the duration of the attack. The attack requires no credentials, no special knowledge, and no economic cost. Severity is **Medium** (griefing / availability degradation, no economic loss to network participants).

### Likelihood Explanation
Any external user can execute this with a trivial script (e.g., 10 parallel `curl` or `fetch` calls). Valid account IDs are publicly enumerable from the same API. The attack is repeatable indefinitely, requires no authentication, and is not detectable until response times degrade. The attacker does not need to maintain a persistent connection — they simply need to keep 10 slow queries in flight.

### Recommendation
1. **Add a global concurrency/rate-limit middleware** to the Node.js REST service (e.g., `express-rate-limit` or a token-bucket per IP) applied before route handlers in `rest/server.js`.
2. **Increase the default pool size** (`hiero.mirror.rest.db.pool.maxConnections`) to a value that reflects realistic concurrency, or document a minimum recommended value.
3. **Reduce `statementTimeout`** for the `/nfts` endpoint specifically, or add a per-route query timeout shorter than 20 seconds to limit how long a single request can hold a connection.
4. **Add an in-flight request limit** per IP at the application layer (similar to the Traefik `inFlightReq` middleware already used for the Rosetta service).

### Proof of Concept
```bash
# Saturate the 10-connection pool with concurrent requests
# Replace <HOST> with the mirror node REST API host
for i in $(seq 1 15); do
  curl -s "http://<HOST>/api/v1/accounts/0.0.$((RANDOM % 1000000 + 1))/nfts?limit=100" &
done
wait

# Simultaneously, measure response time for a legitimate user:
time curl "http://<HOST>/api/v1/accounts/0.0.98/nfts"
# Expected: response time approaches or exceeds connectionTimeoutMillis (20 seconds)
# or returns a 503/timeout error while the pool is exhausted
```

    const filters = utils.buildAndValidateFilters(req.query, acceptedNftAccountParameters);

    res.locals[responseDataLabel] = {
      nfts,
      links: {
        next: this.getPaginationLink(req, nfts, query.bounds, query.limit, query.order),
      },
    };
  };
