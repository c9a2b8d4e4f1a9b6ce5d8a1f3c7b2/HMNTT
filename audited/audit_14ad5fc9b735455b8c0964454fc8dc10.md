### Title
Hardcoded `sslmode=disable` in Rosetta DB DSN Enables Plaintext MITM Interception of Staking Reward Transfer Data

### Summary
`GetDsn()` in `rosetta/app/config/types.go` unconditionally appends `sslmode=disable` to the PostgreSQL DSN, and the `Db` struct contains no SSL/TLS configuration field, making it impossible to enable encryption even via config override. An attacker with L2 network adjacency between the Rosetta service and its PostgreSQL backend can ARP-spoof the connection, intercept the plaintext PostgreSQL wire protocol, and modify query responses for `staking_reward_transfer` rows before they reach the Rosetta API layer, causing it to serve falsified staking reward data to clients.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        ...
    )
}
```
The `Db` struct (lines 39–47) defines `Host`, `Name`, `Password`, `Pool`, `Port`, `StatementTimeout`, and `Username` — no `Ssl` or `SslMode` field exists. There is no mechanism to override `sslmode=disable` via YAML config or environment variable.

`rosetta/app/db/db.go`, `ConnectToDb()`, line 18:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), &gorm.Config{...})
```
This unconditionally opens a plaintext connection.

`rosetta/app/persistence/transaction.go` and `rosetta/app/persistence/domain/staking_reward_transfer.go` confirm that Rosetta queries the `staking_reward_transfer` table as part of its transaction persistence layer, and these query results are returned directly to API consumers.

**Root cause:** The DSN format string hardcodes `sslmode=disable` with no configurable override. The failed assumption is that the DB is always co-located on localhost; the `Host` field is fully configurable and in any deployment where the DB is on a separate host, all wire traffic is plaintext.

**Exploit flow:**
1. Attacker gains L2 adjacency (e.g., shared cloud subnet, compromised switch, or container network) between the Rosetta pod and the PostgreSQL server.
2. Attacker performs ARP cache poisoning to become the MITM for the TCP stream on port 5432.
3. Since `sslmode=disable` is hardcoded, the PostgreSQL client never negotiates TLS; all traffic is the plaintext PostgreSQL wire protocol (v3).
4. Attacker uses a PostgreSQL wire-protocol-aware proxy (e.g., a custom tool or `pg_mitm`-style interceptor) to parse `DataRow` messages in query responses for `staking_reward_transfer` queries.
5. Attacker rewrites the `account_id` or `amount` fields in the `DataRow` packets before forwarding to the Rosetta process.
6. Rosetta deserializes the tampered rows and returns falsified staking reward balances/transfers to API clients.

**Why existing checks fail:** There are no checks. The `Db` struct has no SSL field, `GetDsn()` has no conditional logic, and `ConnectToDb()` passes the DSN directly to GORM without any post-connection TLS verification. The test in `types_test.go` line 24 explicitly asserts `sslmode=disable` is present, confirming this is intentional and tested behavior with no override path.

### Impact Explanation
Any client querying the Rosetta API for staking reward transfer data (account balances, transaction history involving staking rewards) can receive attacker-controlled values. This can cause clients to believe rewards were paid to a different account or in a different amount than actually recorded in the database. In financial/accounting contexts this constitutes data integrity compromise of the mirror node's read API. Severity: **High** — confidentiality (credentials in DSN are also exposed in plaintext) and integrity (query response tampering) are both violated.

### Likelihood Explanation
Preconditions are realistic in cloud-native deployments: Kubernetes pods on the same node share a virtual L2 network; cloud VMs in the same VPC subnet are L2-adjacent; a compromised sidecar or co-tenant can ARP-spoof. The attacker requires no privileges on the Rosetta host or DB server — only network adjacency. PostgreSQL wire protocol v3 is well-documented and parseable with open-source tooling. The attack is repeatable on every query. The only limiting factor is that if `Host` is `127.0.0.1` (loopback), network MITM is not possible; however, the field is fully configurable and remote DB hosts are standard in production.

### Recommendation
1. Add an `SslMode` field to the `Db` struct with a secure default (e.g., `"require"` or `"verify-full"`):
```go
type Db struct {
    Host             string
    Name             string
    Password         string
    Pool             Pool
    Port             uint16
    SslMode          string `yaml:"sslMode"`
    StatementTimeout int    `yaml:"statementTimeout"`
    Username         string
}
```
2. Update `GetDsn()` to use the configurable value:
```go
func (db Db) GetDsn() string {
    sslMode := db.SslMode
    if sslMode == "" {
        sslMode = "require"
    }
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
        db.Host, db.Port, db.Username, db.Name, db.Password, sslMode,
    )
}
```
3. Update `application.yml` default to `sslMode: require`.
4. For production, use `verify-full` with a CA certificate to prevent certificate spoofing as well.
