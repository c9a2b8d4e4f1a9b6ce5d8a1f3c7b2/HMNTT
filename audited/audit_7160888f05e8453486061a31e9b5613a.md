### Title
Unauthenticated Unbounded DB Query Amplification via Uncached, Unthrottled `/api/v1/network/stake` Endpoint

### Summary
The `GET /api/v1/network/stake` endpoint in the `rest-java` module has no rate limiting and no response caching. Every HTTP request — including pipelined requests over a single HTTP/1.1 keep-alive connection — unconditionally invokes `NetworkStakeRepository.findLatest()`, which executes a correlated subquery (`SELECT max(consensus_timestamp)` + outer `SELECT *`) against the `network_stake` table. An unprivileged external user can sustain a high request rate from a single TCP connection, driving repeated DB queries with no server-side guard.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 126–130) calls `networkService.getLatestNetworkStake()` on every request with no guard: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) directly delegates to the repository with no `@Cacheable` or any in-memory cache: [2](#0-1) 

`NetworkStakeRepository.findLatest()` (lines 12–19) executes a correlated subquery on every call: [3](#0-2) 

**Root cause — failed assumption:** The design assumes an upstream proxy or infrastructure layer enforces per-IP or global request rate limits before requests reach the application. The application itself has no such enforcement for this endpoint.

**Why existing checks are insufficient:**

The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) exists exclusively in the `web3` module: [4](#0-3) 

The `rest-java` module's configuration (`WebMvcConfiguration`, `RestJavaConfiguration`) registers only an ETag filter and argument resolvers — no rate-limiting filter, no per-IP connection limit, no request throttle: [5](#0-4) 

The REST Java API configuration table confirms there is no `requestsPerSecond` or throttle property defined for `hiero.mirror.restJava.*`: [6](#0-5) 

**HTTP/1.1 keep-alive amplification:** Spring Boot's embedded Tomcat enables HTTP/1.1 persistent connections by default. An attacker can pipeline many `GET /api/v1/network/stake` requests over a single TCP connection. Because there are no per-IP connection-count limits in `rest-java`, there is nothing to bypass — the attacker simply sends requests at the maximum rate the connection allows, each one triggering a fresh DB query.

### Impact Explanation
Each request causes two sequential DB operations against `network_stake` (a subquery for `max(consensus_timestamp)` and an outer row fetch). With no caching and no throttle, a single attacker sustaining ~500–1000 req/s (easily achievable over HTTP/1.1 keep-alive with pipelining) can saturate DB connection pool threads and CPU. The `RestJavaHighDBConnections` alert fires at 75% pool utilization, confirming DB connections are a recognized bottleneck: [7](#0-6) 

A 30%+ increase in DB query load is realistic with a modest request rate from a single unprivileged client.

### Likelihood Explanation
No credentials, tokens, or special network access are required. Any internet-accessible deployment is reachable. The attack requires only a standard HTTP client capable of keep-alive pipelining (e.g., `curl`, `wrk`, `ab`). It is repeatable, automatable, and requires no knowledge of the system beyond the public API documentation.

### Recommendation
1. **Add response caching** to `NetworkServiceImpl.getLatestNetworkStake()` using `@Cacheable` with a short TTL (e.g., 30s–60s), since `network_stake` data changes only once per staking period (~24h). This eliminates the DB query for the vast majority of requests.
2. **Add application-level rate limiting** to the `rest-java` module, mirroring the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern from the `web3` module, applied as a servlet filter on `/api/v1/*`.
3. **Configure Tomcat keep-alive limits** (`server.tomcat.keep-alive-timeout`, `server.tomcat.max-keep-alive-requests`) to bound the number of requests per persistent connection.

### Proof of Concept
```bash
# Single connection, 1000 pipelined requests using wrk (HTTP/1.1 keep-alive by default)
wrk -t1 -c1 -d30s --latency \
  http://<mirror-node-host>/api/v1/network/stake

# Observe DB query rate spike in metrics:
# spring_data_repository_invocations_seconds_count{repository="NetworkStakeRepository"}
# should increase proportionally with request rate, confirming no caching.
```

Each request in the pipeline triggers one `findLatest()` call → one correlated subquery pair against `network_stake`, with no server-side limit stopping the attacker.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
