### Title
Deleted Account Data Disclosed via Long-Form EVM Address Lookup in GraphQL `account` Query

### Summary
The `account(input: {evmAddress: ...})` GraphQL query is publicly accessible with no authentication. When a "long-form" EVM address is supplied (first 12 bytes are zero, encoding a Hedera numeric entity ID), the code path falls through to `findById()`, which does **not** filter out deleted entities. This is inconsistent with the `findByEvmAddress()` path, which explicitly excludes deleted accounts via `deleted is not true`. Any unauthenticated user can retrieve full account details for deleted accounts by constructing the appropriate long-form EVM address.

### Finding Description

**Code path:**

`AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → `EntityRepository.findById()` (no deleted filter)

In `EntityServiceImpl.getByEvmAddressAndType()`:

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
}
return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
``` [1](#0-0) 

When the first 4 bytes (int) and next 8 bytes (long) of the decoded EVM address are all zero, the code calls `entityRepository.findById(buffer.getLong())`. This is the standard Spring Data `CrudRepository.findById`, which issues a plain `SELECT * FROM entity WHERE id = ?` with **no deleted filter**.

By contrast, the `findByEvmAddress` path uses:

```java
select * from entity where evm_address = ?1 and deleted is not true
``` [2](#0-1) 

The `findByAlias` path also correctly excludes deleted accounts:

```java
select * from entity where alias = ?1 and deleted is not true
``` [3](#0-2) 

The only post-fetch filter applied in the long-form path is `e.getType() == type`, which does not check `deleted`. [4](#0-3) 

The GraphQL endpoint has no authentication or authorization controls — no Spring Security config exists in the graphql module, and the `AccountController` carries no `@PreAuthorize` or equivalent annotation. [5](#0-4) 

The `AccountInput` schema accepts `evmAddress` matching `^(0x)?[a-fA-F0-9]{40}$`, which permits long-form addresses like `0x0000000000000000000000000000000000004D2`. [6](#0-5) 

### Impact Explanation

A deleted account's full record — including `key`, `memo`, `alias`, `balance`, `autoRenewAccount`, `receiverSigRequired`, and all other fields exposed by the `Account` type — is returned to any unauthenticated caller. The `deleted: Boolean` field in the response will show `true`, confirming the account is deleted, but the data is still fully disclosed. This violates the invariant enforced by every other lookup path (`findByEvmAddress`, `findByAlias`) that deleted accounts must not be returned. Sensitive metadata (admin keys, memos containing private notes, staking configuration) of deleted accounts is exposed. [7](#0-6) 

### Likelihood Explanation

Preconditions are minimal: the attacker needs only a known numeric account ID (trivially obtained by iterating small integers or from public ledger history) and network access to the GraphQL endpoint (`/graphql/alpha`). No credentials, tokens, or special privileges are required. The long-form EVM address encoding is a documented Hedera convention, so constructing the payload requires no reverse engineering. The attack is fully repeatable and automatable. [1](#0-0) 

### Recommendation

Add a `deleted` filter to the `findById` branch in `getByEvmAddressAndType`, consistent with the other two lookup paths. The simplest fix is to add `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` after the type filter, or to introduce a dedicated repository query:

```java
// In EntityRepository:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then in `EntityServiceImpl.getByEvmAddressAndType()`, replace `entityRepository.findById(buffer.getLong())` with `entityRepository.findByIdAndNotDeleted(buffer.getLong())`. Apply the same fix to `getByIdAndType()` if deleted accounts should also be excluded from numeric-ID lookups for consistency. [4](#0-3) 

### Proof of Concept

1. Identify a deleted account's numeric ID from ledger history (e.g., account ID `1234` = `0x4D2`).

2. Construct the long-form EVM address (20 bytes, first 12 bytes zero, last 8 bytes = entity ID in big-endian):
   - Account `1234` → `0x0000000000000000000000000000000000004D2`

3. Send the following unauthenticated POST request:

```bash
curl -X POST http://<mirror-node-host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "{ account(input: {evmAddress: \"0x0000000000000000000000000000000000004D2\"}) { deleted key memo alias balance receiverSigRequired } }"
  }'
```

4. Observe that the response returns the full account record with `"deleted": true` — data that is suppressed on all other lookup paths.

5. Repeat for any numeric account ID to enumerate deleted accounts and their sensitive metadata.
