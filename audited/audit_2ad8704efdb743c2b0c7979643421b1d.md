### Title
Unauthenticated AccountBalance Endpoint Triggers Multiple Expensive DB Queries Per Request With No Rate Limiting or Caching

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts requests from any unauthenticated caller and, for each request, triggers 4 distinct database queries against large tables (`entity`, `entity_history`, `account_balance`, `crypto_transfer`). There is no application-level rate limiting, no result caching, and the Traefik middleware that provides the only rate-limiting layer is disabled by default (`global.middleware: false`). An attacker can flood the endpoint with requests using distinct account-ID/block-index pairs, generating a high volume of unique, uncacheable DB queries that measurably increases node resource consumption.

### Finding Description
**Code path:**

`AccountBalance()` at `rosetta/app/services/account_service.go:41-79` calls `a.accountRepo.RetrieveBalanceAtBlock()` at line 61 with no throttle, cache check, or concurrency guard.

`RetrieveBalanceAtBlock()` at `rosetta/app/persistence/account.go:161-223` unconditionally executes **4 sequential DB queries** per call:

1. `getCryptoEntity()` (line 167) — UNION of `entity` + `entity_history` filtered by `timestamp_range @> @consensus_end`.
2. `selectPreviousPartitionLowerBound` inside `getLatestBalanceSnapshot()` (line 276) — scans `mirror_node_time_partitions`.
3. `latestBalanceBeforeConsensus` (line 289) — correlated subquery on `account_balance` anchored to the treasury account's snapshot timestamp.
4. `balanceChangeBetween` inside `getBalanceChange()` (line 322) — `SUM(amount) FROM crypto_transfer WHERE consensus_timestamp > @start AND consensus_timestamp <= @end AND entity_id = @account_id`.

**Root cause / failed assumption:** The design assumes that callers are well-behaved Rosetta clients making infrequent balance queries. There is no authentication, no per-caller quota, no in-process semaphore, and no result cache. Because each request can specify an arbitrary `BlockIdentifier` (index or hash), every distinct `(accountId, blockIndex)` pair produces a unique set of query parameters that bypasses any DB-level query-plan cache and forces full range scans on `crypto_transfer` and `account_balance`.

**Why existing checks fail:**

The only rate-limiting mechanism is the Traefik middleware defined in `charts/hedera-mirror-rosetta/values.yaml:149-166`:
```yaml
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
- rateLimit:
    average: 10
    sourceCriterion:
      requestHost: true
```
This middleware is gated on `global.middleware` which defaults to `false` at line 95:
```yaml
global:
  middleware: false
```
So in a default deployment the middleware chain is never instantiated. Even when enabled, the `inFlightReq` limit is per source IP (trivially bypassed with multiple IPs or a botnet), and `rateLimit: average: 10` per host still allows 10 req/s × 4 queries = 40 DB queries/s from a single host, which compounds across multiple attackers.

### Impact Explanation
Each `AccountBalance` request with a unique `(accountId, blockIndex)` pair forces:
- A UNION range-scan on `entity`/`entity_history`
- A partition-boundary lookup
- A correlated subquery on `account_balance`
- A `SUM` aggregate scan on `crypto_transfer` over an arbitrary time range

On a production mirror node where `crypto_transfer` contains hundreds of millions of rows, the `balanceChangeBetween` query alone can consume significant CPU and I/O. Sending ~50–100 concurrent requests with distinct parameters is sufficient to saturate DB connection pools and push CPU/IO well above the 30% threshold. The attack degrades service for all legitimate users and operators of the mirror node.

### Likelihood Explanation
The attack requires zero privileges — the Rosetta `/account/balance` endpoint is a public, unauthenticated API by design. Valid account IDs in `shard.realm.num` format are publicly enumerable from the mirror node's own REST API. Valid block indices are sequential integers starting from 0. An attacker needs only a script that iterates account IDs and block indices and fires HTTP POST requests to `/account/balance`. This is trivially repeatable and requires no special knowledge or access.

### Recommendation
1. **Enable the Traefik middleware by default** — change `global.middleware: false` to `global.middleware: true` in `charts/hedera-mirror-rosetta/values.yaml:95`.
2. **Add application-level concurrency limiting** — introduce a semaphore or token-bucket rate limiter in `AccountBalance()` before calling `RetrieveBalanceAtBlock()`, independent of infrastructure configuration.
3. **Cache balance results** — add a short-lived (e.g., 5–10 second) in-process LRU cache keyed on `(accountId, consensusEndNanos)` in `accountRepository` to deduplicate repeated queries for the same account-block pair.
4. **Merge DB queries** — consolidate the 4 sequential queries in `RetrieveBalanceAtBlock()` into a single CTE-based query to reduce per-request DB round-trips.

### Proof of Concept
```bash
# Enumerate valid account IDs from the public REST API
# Then flood /account/balance with distinct (account, block) pairs

for i in $(seq 1 1000); do
  curl -s -X POST http://<rosetta-host>/account/balance \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\": {\"address\":\"0.0.$i\"},
      \"block_identifier\": {\"index\": $i}
    }" &
done
wait
```

Each iteration sends a request with a unique `(accountId=0.0.$i, blockIndex=$i)` pair. With 1000 concurrent requests, `RetrieveBalanceAtBlock()` is invoked 1000 times in parallel, each executing 4 DB queries against `entity`, `account_balance`, and `crypto_transfer`, with no throttle or cache to absorb the load.
