### Title
Publicly Hardcoded Default Operator Private Key Enables Full Account Compromise on Any Default-Configuration Deployment

### Summary
The pinger service hardcodes a private key (`302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137`) as the default value for `HIERO_MIRROR_PINGER_OPERATOR_KEY` in a public source repository. Any deployment that does not explicitly override this default exposes its operator account to full compromise by any internet user who reads the source code. Because the key is publicly visible — not merely leaked — no privileged access is required to obtain it.

### Finding Description
**Exact code location:**
- `pinger/config.go`, line 40: default value for `operatorKey` flag
- `pinger/sdk_client.go`, lines 40–45: key parsed and set as the signing identity for all submitted transactions

**Root cause:**
`loadConfig()` in `config.go` registers the operator private key flag with a hardcoded default:

```go
flag.StringVar(&cfg.operatorKey, "operator-key",
    envOr("HIERO_MIRROR_PINGER_OPERATOR_KEY",
        "302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137"),
    "operator private key string")
``` [1](#0-0) 

This default is used verbatim if neither the environment variable nor the CLI flag is set. In `newClient()`, the key is parsed and installed as the operator signing key with no check that it differs from the well-known default:

```go
opKey, err := hiero.PrivateKeyFromString(cfg.operatorKey)
...
client.SetOperator(opID, opKey)
``` [2](#0-1) 

**Exploit flow:**
1. Attacker reads the public repository and extracts the private key from `config.go` line 40 — zero privilege required.
2. Attacker identifies a pinger deployment running with default configuration (operator account `0.0.2` paired with the default key, or any account whose key was set to this value during provisioning).
3. Using the known private key, the attacker independently constructs and signs `CryptoTransfer` transactions as the operator account, submitting them directly to the Hedera network nodes — bypassing the pinger service entirely.
4. The attacker can drain the operator account balance or submit arbitrary transfers that appear to originate from the legitimate operator.

**Why existing checks fail:**
The only validation in `loadConfig()` is a non-empty check:
```go
if cfg.operatorKey == "" {
    return cfg, fmt.Errorf("missing operator key ...")
}
``` [3](#0-2) 

This check passes trivially for the default value. There is no check that the key differs from the known default, no warning, and no enforcement that the key was explicitly supplied by the operator.

### Impact Explanation
Any deployment using the default configuration has its operator account fully controlled by anyone who reads the source code. The attacker can:
- Drain all HBAR from the operator account via signed `CryptoTransfer` transactions.
- Submit fraudulent transfers that appear to originate from the legitimate pinger operator, corrupting the integrity of the monitoring signal.
- On mainnet, this results in direct financial loss and integrity failure of the health-check infrastructure.

Severity: **Critical** — full account key compromise, direct asset theft, transaction forgery, no privilege required.

### Likelihood Explanation
The exploit requires only reading a public GitHub repository. No network access to the deployment, no credentials, no social engineering. The precondition (deployment using the default key) is realistic: developers commonly use defaults for staging environments and accidentally promote them to production, or provision the operator account using the default key during initial setup without rotating it. The key is stable across repository history, so it can be extracted from any historical commit as well.

### Recommendation
1. **Remove the hardcoded default entirely.** The `operatorKey` flag must have no default value; `loadConfig()` must return an error if it is not explicitly supplied.
2. **Add a known-bad-key check.** On startup, compare the loaded key against the well-known default and refuse to start if they match, printing a clear error.
3. **Document that the key must be injected via a secret manager** (e.g., Kubernetes Secret, Vault) and never stored in source or image layers.
4. **Rotate** the operator account key on any deployment that may have used the default.

### Proof of Concept
```
# Step 1: Extract the known private key from public source
PRIVATE_KEY="302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137"

# Step 2: Identify the operator account (default: 0.0.2, or observe pinger logs)
OPERATOR_ACCOUNT="0.0.2"

# Step 3: Using the Hiero Go SDK or any Hedera SDK, construct and sign a transfer
# as the operator — no interaction with the pinger service required:
client := hiero.ClientForName("testnet")
opKey, _ := hiero.PrivateKeyFromString(PRIVATE_KEY)
opID, _ := hiero.AccountIDFromString(OPERATOR_ACCOUNT)
client.SetOperator(opID, opKey)

// Drain to attacker-controlled account
hiero.NewTransferTransaction().
    AddHbarTransfer(opID, hiero.HbarFromTinybar(-<balance>)).
    AddHbarTransfer(<attacker_account>, hiero.HbarFromTinybar(<balance>)).
    Execute(client)

# Result: Transfer succeeds; operator account drained; transaction signed with
# the legitimate operator key, indistinguishable from pinger-originated traffic.
```
