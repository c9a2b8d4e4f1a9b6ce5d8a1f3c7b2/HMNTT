### Title
Unbounded Request Body Buffering in `estimateFees()` Enables Heap Exhaustion via Large Protobuf Payloads

### Summary
The `POST /api/v1/network/fees` endpoint in `NetworkController.estimateFees()` accepts a raw `byte[]` body with no size constraint. Spring's message converter fully buffers the entire request body into heap memory before the controller method is invoked, and `Bytes.wrap(body)` then wraps that allocation. An unauthenticated attacker can send arbitrarily large payloads in concurrent requests to cause sustained heap pressure, frequent GC cycles, and measurable resource consumption increases without any brute-force volume.

### Finding Description

**Exact code location**: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 110–124.

```java
@PostMapping(
        consumes = {"application/protobuf", "application/x-protobuf"},
        value = "/fees")
FeeEstimateResponse estimateFees(
        @RequestBody @NotNull byte[] body,          // ← no @Size or length constraint
        ...
) {
    final var transaction = Transaction.PROTOBUF.parse(Bytes.wrap(body));  // ← full heap allocation
    ...
}
```

**Root cause**: The `@RequestBody byte[] body` parameter carries no `@Size(max = …)` annotation and no upstream filter enforces a content-length ceiling. Spring's `ByteArrayHttpMessageConverter` reads the entire request body into a `byte[]` on the heap before the controller method is entered. `Bytes.wrap(body)` then creates a zero-copy view of that already-allocated array, and `Transaction.PROTOBUF.parse(…)` traverses it.

**Why existing checks are insufficient**:

- `JacksonConfiguration` (`rest-java/src/main/java/org/hiero/mirror/restjava/config/JacksonConfiguration.java`, lines 20–27) sets `maxDocumentLength=1000`, `maxStringLength=1000`, etc., but these constraints apply exclusively to the Jackson JSON parser. They have no effect on the protobuf binary path.
- `MetricsFilter` (`rest-java/src/main/java/org/hiero/mirror/restjava/config/MetricsFilter.java`, lines 64–67) reads the `Content-Length` header only to record a metric; it does not reject or truncate oversized requests.
- No `server.tomcat.max-http-form-post-size` or `server.tomcat.max-swallow-size` property is set anywhere in `rest-java/src/main/resources/` (which contains only `banner.txt`). Spring Boot's default `max-http-form-post-size` (2 MB) applies only to `application/x-www-form-urlencoded` and `multipart/form-data`, not to raw binary `application/protobuf` bodies.
- The `@Min(0) @Max(10000)` validation on `highVolumeThrottle` constrains only that integer parameter, not the body.
- No rate-limiting or authentication is present on this endpoint.

### Impact Explanation
Each oversized request allocates a contiguous heap region equal to the payload size. With payloads of, for example, 50–200 MB sent in a handful of concurrent connections, the JVM heap fills rapidly, triggering major GC pauses. This directly raises CPU utilization (GC threads), reduces throughput for all other endpoints sharing the same JVM, and can cause stop-the-world pauses. A 30%+ resource consumption increase is achievable with a small number of concurrent requests (not brute-force volume), making this a medium-severity denial-of-service condition against the mirror node's REST Java service.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The attack requires only a standard HTTP client capable of sending a POST with `Content-Type: application/protobuf` and a large body. It is trivially repeatable, requires no special knowledge of the protocol (the payload need not be a valid protobuf transaction — the body is buffered before parsing), and can be sustained indefinitely. A single attacker with a modest upload bandwidth can maintain the pressure.

### Recommendation
1. **Enforce a body size limit at the server layer**: Set `server.tomcat.max-swallow-size` and add a `ContentSizeLimitFilter` (or equivalent `OncePerRequestFilter`) that rejects requests exceeding a defined threshold (e.g., 64 KB, which is well above any legitimate HAPI transaction size) before the body is read.
2. **Add a `@Size` or custom constraint on the `body` parameter**: Annotate `@RequestBody @NotNull @Size(max = 65536) byte[] body` and ensure Spring's validation is triggered before message conversion (requires a custom `HandlerMethodArgumentResolver` or pre-read filter).
3. **Configure Spring Boot properties** in `application.yml`:
   ```yaml
   server:
     tomcat:
       max-swallow-size: 65536
   ```
4. **Add rate limiting** on the POST endpoint to bound the number of concurrent large-body requests per source IP.

### Proof of Concept
```bash
# Generate a 100 MB payload (content need not be valid protobuf)
dd if=/dev/urandom bs=1M count=100 of=/tmp/large_payload.bin

# Send 10 concurrent requests to the endpoint
for i in $(seq 1 10); do
  curl -s -o /dev/null \
    -X POST \
    -H "Content-Type: application/protobuf" \
    --data-binary @/tmp/large_payload.bin \
    http://<mirror-node-host>:8080/api/v1/network/fees &
done
wait
```

Each request causes the JVM to allocate a 100 MB `byte[]` on the heap. Ten concurrent requests attempt to hold 1 GB simultaneously, triggering major GC cycles.
