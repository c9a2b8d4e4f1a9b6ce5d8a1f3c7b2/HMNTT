### Title
Unbounded Query in `getContractResultsByTimestamps()` Allows Resource Exhaustion via Crafted Transaction

### Summary
`getContractResultsByTimestamps()` in `rest/service/contractService.js` constructs a SQL query with no `LIMIT` clause, unlike every other paginated query in the same service. Any unprivileged user who can submit a Hedera transaction (permissionless) can craft one that generates a large number of `contract_result` rows sharing the same `consensus_timestamp`, then repeatedly query the public REST API to force the mirror node to fetch and deserialize all rows into memory with no bound.

### Finding Description

**Exact code location — `rest/service/contractService.js` lines 269–297:**

```js
async getContractResultsByTimestamps(timestamps, involvedContractIds = []) {
  // ... builds whereClause ...
  const query = [
    `with ${ContractService.entityCTE} `,
    ContractService.contractResultsWithEvmAddressQuery,
    ContractService.joinContractResultWithEvmAddress,
    whereClause,          // ← only a WHERE clause
  ].join('\n');           // ← NO LIMIT appended

  const rows = await super.getRows(query, params);
  return rows.map((row) => {          // ← every row materialised in memory
    return { ...new ContractResult(row), evmAddress: row.evm_address };
  });
}

**Call chain to public endpoints:**

`getDetailedContractResults()` calls `getContractResultsByTimestamps()` directly: [3](#0-2) 

`getDetailedContractResults()` is invoked by two unauthenticated public handlers:
- `getContractResultsByTimestamp` — `GET /api/v1/contracts/:contractId/results/:consensusTimestamp`
- `getContractResultsByTransactionIdOrHash` — `GET /api/v1/contracts/results/:transactionIdOrHash` [4](#0-3) [5](#0-4) 

**Root cause:** The developer assumed a single `consensus_timestamp` maps to a small, bounded set of `contract_result` rows. No `LIMIT` is appended to the query, and no cap is applied to the returned slice before object mapping.

**Why the `involvedContractIds` filter is insufficient:** The filter `CONTRACT_ID in (...)` narrows by which contracts are involved but does not cap the *number of rows returned per contract*. A contract that is called many times within one transaction (e.g., via a factory pattern or recursive dispatcher) can produce many rows sharing the same `consensus_timestamp` and the same `contract_id`. [6](#0-5) 

### Impact Explanation
Each `contract_result` row contains multiple binary blob columns (`call_result`, `bloom`, `function_parameters`, `failed_initcode`, `transaction_hash`). Fetching thousands of such rows in a single unbounded query causes:
- **DB I/O spike**: full sequential scan of matching rows with no early stop.
- **Application heap pressure**: every row is spread into a new `ContractResult` object plus spread-copy (`{...new ContractResult(row), ...}`).
- **Amplification**: the same timestamp can be queried concurrently by multiple attacker-controlled clients, multiplying the effect.

A sustained attack targeting a single crafted timestamp can push DB I/O and Node.js heap well above the 30% threshold stated in the question.

### Likelihood Explanation
Hedera is a permissionless network; anyone can submit transactions for a small fee. The Hedera gas cap per transaction is 15,000,000 gas. A minimal internal contract call costs ~700–2,000 gas, meaning a single transaction can legitimately produce thousands of `contract_result` rows. Once the transaction is on-chain, the attacker's timestamp is permanent and can be queried indefinitely at zero additional on-chain cost. No privileged access, API key, or special role is required to call either triggering endpoint.

### Recommendation
Add a hard `LIMIT` to the query built in `getContractResultsByTimestamps()`, consistent with how every other query in the service is bounded:

```js
const query = [
  `with ${ContractService.entityCTE} `,
  ContractService.contractResultsWithEvmAddressQuery,
  ContractService.joinContractResultWithEvmAddress,
  whereClause,
  super.getLimitQuery(params.push(maxContractResultsPerTimestamp)),
].join('\n');
```

Where `maxContractResultsPerTimestamp` is a configurable constant (e.g., `defaultLimit` or a dedicated cap). Additionally, consider rate-limiting the two triggering endpoints per source IP.

### Proof of Concept
1. Deploy a Hedera smart contract that, in a single transaction, dispatches calls to N child contracts (or loops calling a factory), generating N `contract_result` rows all sharing one `consensus_timestamp`. With 15M gas budget, N can reach the thousands.
2. Record the `transactionHash` of that transaction.
3. Send repeated concurrent requests to the mirror node:
   ```
   GET /api/v1/contracts/results/<transactionHash>
   ```
4. Each request triggers `getContractResultsByTimestamps()` with no LIMIT, causing the DB to return all N rows and the application to allocate N `ContractResult` objects.
5. Monitor mirror node CPU, heap, and DB I/O — resource consumption rises proportionally to N and the request rate, with no server-side bound to stop it.
