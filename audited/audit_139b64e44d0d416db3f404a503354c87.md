### Title
EntityId LRU Cache Pollution via Repeated `account.id` Query Parameters in `getSchedules()`

### Summary
An unprivileged external user can send repeated requests to `GET /api/v1/schedules` with up to `maxRepeatedQueryParameters` (default 100) distinct valid `account.id` values per request. Each distinct value is unconditionally written into the shared process-level `quick-lru` EntityId cache. By sending ~1,000 such requests with non-overlapping account IDs, an attacker can fill the entire 100,000-entry cache with attacker-controlled entries, evicting all legitimate cached entity IDs and forcing cache misses for every other endpoint that relies on the same cache.

### Finding Description

**Code path:**

1. `rest/schedules.js:236` — `getSchedules()` calls `utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters)` with `account.id` in the accepted set.

2. `rest/utils.js:1234–1259` — `buildFilters()` checks `isRepeatedQueryParameterValidLength(values)`. Arrays of length ≤ 100 pass; each value is pushed into `filters` as a comparator object. No cache interaction yet.

3. `rest/utils.js:1350–1354` — `validateAndParseFilters()` calls `validateFilters()` then, **only if all params are valid**, calls `formatFilters(filters)`.

4. `rest/utils.js:1365–1367` — Inside `formatComparator()`, the `ACCOUNT_ID` case unconditionally calls:
   ```js
   comparator.value = EntityId.parseString(comparator.value).getEncodedId();
   ```

5. `rest/entityId.js:314–332` — `parseCached()` computes a cache key `${id}_${allowEvmAddress}_${evmAddressType}`, checks the LRU, and on a miss calls `cache.set(key, entityId)` — writing into the shared `quick-lru` instance.

6. `rest/entityId.js:301–304` — The cache is a single process-level `quick-lru` with `maxSize: 100000` (default).

**Root cause:** The `maxRepeatedQueryParameters` guard (line 1241–1248) only limits the number of values *per request* to 100. It does not prevent an attacker from issuing many requests, each inserting up to 100 new distinct entries into the shared LRU cache. There is no rate limiting, no per-IP throttling, and no cap on total cache writes from external input.

**Why existing checks fail:** The check at line 1241 rejects arrays longer than 100 but silently allows exactly 100 distinct values through. Each of those 100 values, if syntactically valid, is written to the cache at line 331. Across 1,000 requests the attacker inserts 100,000 entries — exactly the cache's `maxSize` — evicting all prior legitimate entries.

### Impact Explanation

The EntityId LRU cache is shared across all REST endpoints (accounts, tokens, contracts, transactions, etc.) that call `EntityId.parse()` or `EntityId.parseString()`. Flooding it with attacker-controlled entries forces every subsequent legitimate request to incur a cache miss and re-parse entity IDs from scratch. While re-parsing is CPU-only (no database hit), sustained flooding keeps the cache permanently cold for legitimate traffic, increasing per-request CPU cost and latency across the entire REST API surface. The impact is a griefing/degradation attack with no economic damage to any individual user but measurable service degradation under load.

### Likelihood Explanation

No authentication or API key is required. The `/api/v1/schedules` endpoint is publicly accessible. The attacker needs only an HTTP client and knowledge of the valid entity ID format (`0.0.N`). With `maxRepeatedQueryParameters = 100` and `maxSize = 100000`, exactly 1,000 HTTP requests suffice to saturate the cache. This is trivially scriptable and repeatable in a continuous loop to maintain cache pollution indefinitely. No special network position or privileged access is needed.

### Recommendation

1. **Bound cache writes from external input:** Before calling `formatFilters` / `parseCached`, check whether the incoming `account.id` values are already in the cache; skip `cache.set` if the cache is near capacity or if the key originates from an untrusted query parameter that has not been seen in the DB.
2. **Separate caches by trust level:** Use a smaller, separate LRU for query-parameter entity IDs (e.g., `maxSize: 1000`) distinct from the cache used for internally resolved entity IDs.
3. **Rate-limit per IP:** Apply a request-rate limit on the `/api/v1/schedules` endpoint (and all endpoints accepting repeated entity ID parameters) to bound the rate of cache pollution.
4. **Limit total distinct `account.id` values per time window per IP** at the middleware layer, independent of the per-request `maxRepeatedQueryParameters` check.

### Proof of Concept

```bash
# Fill the 100,000-entry LRU cache with ~1,000 requests of 100 distinct account IDs each.
# Adjust BASE and STEP to generate non-overlapping account numbers across requests.

BASE=1
for i in $(seq 1 1000); do
  PARAMS=""
  for j in $(seq 0 99); do
    NUM=$((BASE + i * 100 + j))
    PARAMS="${PARAMS}&account.id=0.0.${NUM}"
  done
  curl -s "http://<mirror-node-host>:5551/api/v1/schedules?${PARAMS}" > /dev/null
done

# After this loop, the EntityId LRU cache contains 100,000 attacker-inserted entries.
# All legitimate cached entity IDs have been evicted.
# Subsequent requests from real users to /api/v1/accounts, /api/v1/tokens, etc.
# will incur cache misses for every EntityId.parse() call until the cache naturally
# repopulates — which the attacker can prevent by continuing the loop.
```
