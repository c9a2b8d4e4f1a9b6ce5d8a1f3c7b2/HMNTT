### Title
Uncached Alias Lookup in `EntityServiceImpl.lookup()` Enables Database Query Flood via Hook Endpoints

### Summary
The `rest-java` `EntityServiceImpl.lookup()` performs a direct, uncached database query for every `EntityIdAliasParameter` or `EntityIdEvmAddressParameter` input. The `HooksController` endpoints (`/api/v1/accounts/{ownerId}/hooks` and `/{hookId}/storage`) accept alias-format `ownerId` path parameters with no rate limiting in the `rest-java` module. An unprivileged attacker can flood these endpoints with syntactically valid but non-existent aliases, generating a sustained stream of unbounded database queries that degrades the shared PostgreSQL instance and impairs the importer's transaction ingestion throughput.

### Finding Description

**Exact code path:**

`HooksController` (lines 81–102, 104–130) accepts `@PathVariable EntityIdParameter ownerId` and passes it directly to `HookServiceImpl.getHooks()` / `getHookStorage()`. [1](#0-0) 

`HookServiceImpl` calls `entityService.lookup(request.getOwnerId())` at lines 35, 61, and 86. [2](#0-1) 

The `entityService` is the `rest-java` `EntityServiceImpl`, whose `lookup()` is:

```java
case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
``` [3](#0-2) 

This issues a raw SQL query on every call:

```sql
select id from entity where alias = ?1 and deleted <> true
``` [4](#0-3) 

**Root cause — failed assumption:** The importer's `EntityIdServiceImpl` has a `cacheLookup()` wrapper that caches alias→EntityId resolutions in a Spring `Cache`. [5](#0-4) 

The `rest-java` `EntityServiceImpl` is a completely separate bean with **no cache at all**. There is no `@Cacheable`, no `Cache` field, and no negative-result caching. A miss (non-existent alias) returns `Optional.empty()` and throws `EntityNotFoundException` — but only after the DB round-trip completes.

**Alias format:** `EntityIdAliasParameter` accepts any base32 string of 40–70 uppercase characters (`[A-Z2-7]`), giving an astronomically large space of syntactically valid, unique aliases. [6](#0-5) 

**No rate limiting on hooks endpoints:** The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module and is wired to `ContractCallRequest` processing. The `rest-java` `HooksController` has no equivalent protection. [7](#0-6) 

### Impact Explanation

The REST API and the importer share the same PostgreSQL database. A sustained flood of alias-lookup queries against the `entity` table (which is large and indexed on `alias`) consumes DB connection pool slots, CPU, and I/O bandwidth. This degrades write throughput for the importer, which must continuously `INSERT`/`UPDATE` rows as it ingests transactions from the Hedera network. Under sufficient load, the importer falls behind consensus, causing the mirror node to report stale or missing transaction data. In a worst-case scenario with an undersized DB, connection exhaustion can halt ingestion entirely.

### Likelihood Explanation

No authentication or API key is required to call `/api/v1/accounts/{ownerId}/hooks`. The alias format is publicly documented and trivially generated (e.g., any 40-character base32 string such as `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`). A single attacker with a modest HTTP client can issue thousands of requests per second, each with a unique alias, from a single IP or a small botnet. There is no server-side deduplication, no negative-result cache, and no per-IP throttle in the `rest-java` module to absorb or reject the flood.

### Recommendation

1. **Add negative-result caching** in `rest-java` `EntityServiceImpl.lookup()`: cache `Optional.empty()` results for alias/EVM-address misses with a short TTL (e.g., 30 s) so repeated lookups for the same non-existent alias do not hit the DB.
2. **Add rate limiting** to the `HooksController` endpoints, mirroring the `web3` `ThrottleManagerImpl` pattern (bucket4j token bucket per IP or globally).
3. **Validate alias existence early** and return `404` before executing further repository queries, keeping the fast-path cheap.
4. Consider a **DB connection pool limit** specifically for the REST API tier, separate from the importer pool, to prevent REST-API query floods from starving importer writes.

### Proof of Concept

```bash
# Generate unique, syntactically valid, non-existent aliases and flood the endpoint
for i in $(seq 1 10000); do
  ALIAS=$(python3 -c "
import base64, os
raw = os.urandom(25)
print(base64.b32encode(raw).decode().rstrip('='))
")
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.${ALIAS}/hooks" &
done
wait
```

Each request passes `EntityIdAliasParameter` validation (40-char base32), reaches `EntityServiceImpl.lookup()`, and executes `SELECT id FROM entity WHERE alias = ? AND deleted <> true` against the live database. With 10 000 concurrent requests, the DB connection pool saturates and the importer's write latency spikes measurably. Monitor with `SELECT count(*) FROM pg_stat_activity WHERE state = 'active'` to observe connection exhaustion.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }

@NullMarked
@RequestMapping(value = "/api/v1/accounts/{ownerId}/hooks", produces = APPLICATION_JSON)
@RequiredArgsConstructor
@RestController
final class HooksController {

    private static final int KEY_BYTE_LENGTH = 32;
    private static final byte[] MIN_KEY_BYTES = new byte[KEY_BYTE_LENGTH]; // A 32-byte array of 0x00
    private static final byte[] MAX_KEY_BYTES;

    private static final Function<Hook, Map<String, String>> HOOK_EXTRACTOR =
            hook -> ImmutableSortedMap.of(HOOK_ID, hook.getHookId().toString());

    private static final Function<HookStorage, Map<String, String>> HOOK_STORAGE_EXTRACTOR =
            hook -> ImmutableSortedMap.of(KEY, hook.getKey());

    static {
        MAX_KEY_BYTES = new byte[KEY_BYTE_LENGTH];
        Arrays.fill(MAX_KEY_BYTES, (byte) 0xFF); // A 32-byte array of 0xFF
    }

    private final HookService hookService;
    private final HookMapper hookMapper;
    private final HookStorageMapper hookStorageMapper;
    private final LinkFactory linkFactory;

    @GetMapping
    ResponseEntity<HooksResponse> getHooks(
            @PathVariable EntityIdParameter ownerId,
            @RequestParam(defaultValue = "", name = HOOK_ID, required = false)
                    @Size(max = MAX_REPEATED_QUERY_PARAMETERS)
                    NumberRangeParameter[] hookId,
            @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
            @RequestParam(defaultValue = "desc") Sort.Direction order) {

        final var hooksRequest = hooksRequest(ownerId, hookId, limit, order);
        final var hooksServiceResponse = hookService.getHooks(hooksRequest);
        final var hooks = hookMapper.map(hooksServiceResponse);

        final var sort = Sort.by(order, HOOK_ID);
        final var pageable = PageRequest.of(0, limit, sort);
        final var links = linkFactory.create(hooks, pageable, HOOK_EXTRACTOR);

        final var response = new HooksResponse();
        response.setHooks(hooks);
        response.setLinks(links);

        return ResponseEntity.ok(response);
    }
