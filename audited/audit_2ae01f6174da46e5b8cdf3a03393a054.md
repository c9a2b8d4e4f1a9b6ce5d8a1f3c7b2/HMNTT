### Title
Silent Exception Swallowing in `BinaryGasEstimator.safeCall()` Causes Gas Estimate Inflation Without Error Surfacing

### Summary
`BinaryGasEstimator.safeCall()` catches all `Exception` subtypes and returns `null`, which the binary search loop unconditionally treats as "insufficient gas" by advancing `lo = mid`. An unprivileged user can deploy or target a contract whose execution throws a non-OOG exception (e.g., a `MirrorEvmTransactionException` revert) at intermediate gas values during the search, causing every iteration to silently fail and the estimator to converge on `hi` (the caller-supplied gas ceiling, up to 15 M gas) while returning that inflated value to the caller with no error.

### Finding Description

**Code locations:**

`BinaryGasEstimator.java` lines 67–74 — `safeCall` swallows every `Exception`: [1](#0-0) 

`BinaryGasEstimator.java` lines 45–57 — `null` result is treated identically to "gas too low": [2](#0-1) 

`ContractCallService.java` lines 109–138 — `doProcessCall` with `estimate=true` still re-throws `MirrorEvmTransactionException` (and wraps `IllegalStateException`/`IllegalArgumentException`), all of which propagate out of the lambda and into `safeCall`: [3](#0-2) 

`ContractExecutionService.java` lines 91–95 — the lambda passed to `search` is `gas -> doProcessCall(params, gas, true)`: [4](#0-3) 

**Root cause:** The failed assumption is that any exception thrown during a binary-search probe means "gas was insufficient." In reality, `doProcessCall` throws `MirrorEvmTransactionException` for contract reverts, precompile failures, and other non-gas-related errors. `safeCall` cannot distinguish between an OOG exception and a revert exception; it maps both to `null`, which the loop treats as `err = true` → `lo = mid`.

**Exploit flow:**
1. Attacker deploys a contract `C` whose function `f()` is designed to:
   - Succeed (return normally) when called with the full gas ceiling (e.g., 15 M gas) — so the initial `callContract` probe in `estimateGas()` passes.
   - Revert (throw a `MirrorEvmTransactionException`) at every intermediate gas value the binary search probes.
2. Attacker (or victim) calls `eth_estimateGas` with `to=C`, `data=<selector of f>`, `gas=15000000`.
3. `estimateGas()` runs the initial probe at 15 M gas → succeeds, `gasUsedByInitialCall = X`.
4. Binary search starts: `lo = X`, `hi = 15_000_000`.
5. Every `safeCall(mid, ...)` invocation triggers a revert → exception → `null` → `err = true` → `lo = mid`.
6. After `maxGasEstimateRetriesCount` iterations, `lo` converges toward `hi`; the loop exits and returns `hi = 15_000_000`.
7. The caller receives `0xE4E1C0` (15 M) as the gas estimate with HTTP 200 and no error indication.

### Impact Explanation
The caller receives a massively inflated gas estimate (up to the configured `maxGasLimit`, default 15 M) with no error. On Hedera, gas limits directly determine transaction fees; a user who trusts this estimate and submits a transaction with a 15 M gas limit will pay the maximum possible fee for what may be a cheap operation. Because the response is a normal `200 OK` with a hex-encoded integer, client libraries and wallets have no signal that the estimate is unreliable. The attacker can use this to drain funds from users who interact with a malicious contract via a dApp that calls `eth_estimateGas` before submission.

### Likelihood Explanation
No special privileges are required. Any account holding enough HBAR to deploy a contract can execute this attack. The contract logic needed (succeed at max gas, revert at lower gas using `gasleft()`) is trivial to write. The attack is fully repeatable and deterministic. Client-side tooling (ethers.js, web3.js, MetaMask) unconditionally trusts `eth_estimateGas` responses, so users have no practical way to detect the inflation.

### Recommendation
1. **Distinguish exception types in `safeCall`:** Only treat OOG-related exceptions as "insufficient gas." Re-throw (or propagate as a failed result) any exception that indicates a non-gas-related revert, so the binary search can terminate early with a proper error.
2. **Propagate revert errors to the caller:** If `doProcessCall` throws a `MirrorEvmTransactionException` during the binary search, surface it to the caller rather than silently absorbing it.
3. **Cap the returned estimate:** If the search returns `hi` unchanged (i.e., no iteration ever succeeded), treat this as an estimation failure and return an error rather than the raw ceiling value.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract GasManipulator {
    // Succeeds only when called with near-maximum gas; reverts otherwise.
    function f() external view returns (uint256) {
        // gasleft() is high on the initial probe (full gas ceiling).
        // During binary search, mid values are much lower → revert.
        require(gasleft() > 14_000_000, "insufficient gas for operation");
        return 42;
    }
}
```

**Steps:**
1. Deploy `GasManipulator` on the target network.
2. Send `POST /api/v1/contracts/call` with:
   ```json
   {
     "to": "<deployed address>",
     "data": "<selector of f()>",
     "gas": 15000000,
     "estimate": true
   }
   ```
3. Observe the response: `{"result": "0xe4e1c0"}` (15,000,000) — no error, HTTP 200.
4. The actual gas needed for `f()` to succeed is ~21,000 + intrinsic costs; the estimate is ~714× inflated.

    protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }
