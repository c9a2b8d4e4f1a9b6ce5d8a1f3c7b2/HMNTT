### Title
Unbounded gRPC Topic Subscription Admission — Resource Exhaustion via Unauthenticated Subscriber Flood

### Summary
`TopicMessageServiceImpl.subscribeTopic()` maintains a `subscriberCount` `AtomicLong` solely as a Micrometer metrics gauge with no admission-control gate. Any unauthenticated caller can open an arbitrary number of subscriptions across multiple connections, exhausting the `boundedElastic` scheduler (one task per subscription for the `safetyCheck` Flux), the DB connection pool, and heap, causing severe latency degradation for all legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` accepts any `ConsensusTopicQuery` with no authentication or rate-limit check and immediately delegates to `topicMessageService.subscribeTopic()`: [1](#0-0) 

Inside `TopicMessageServiceImpl.subscribeTopic()`, `subscriberCount` is incremented unconditionally on every new subscription via `doOnSubscribe` and decremented on termination via `doFinally`. It is registered only as a Micrometer `Gauge` — it is **never read back to gate admission**: [2](#0-1) [3](#0-2) 

Every accepted subscription unconditionally constructs a `safetyCheck` Flux that schedules a task on `Schedulers.boundedElastic()`: [4](#0-3) 

`GrpcProperties` has no `maxSubscribers` or `maxActiveSubscriptions` field: [5](#0-4) 

**Root cause:** The `subscriberCount` field exists purely for observability. No code path reads it to reject new subscriptions. The design document for the GraphQL `ContractLogTopicListener` explicitly specifies a `canSubscribe()` check against `maxActiveSubscriptions`, but this pattern was never applied to the gRPC service: [6](#0-5) 

**Only existing mitigation — `maxConcurrentCallsPerConnection = 5`:** [7](#0-6) [8](#0-7) 

This limits concurrent calls **per TCP connection**, not globally. An attacker opens `C` connections and gets `5 × C` concurrent subscriptions with no upper bound on `C`.

**Exploit flow:**
1. Attacker opens `N` TCP connections to port 5600 (no auth required).
2. On each connection, sends 5 concurrent `subscribeTopic` RPCs targeting a high-traffic topic with no `endTime` and no `limit` (so subscriptions never self-terminate).
3. Each subscription: allocates a `TopicContext`, schedules a `Mono.delay(1s)` task on `boundedElastic`, registers a live listener on the Redis/polling topic listener, and holds a DB connection for historical retrieval.
4. With `N = 200` connections → 1,000 concurrent subscriptions. The `boundedElastic` scheduler (default cap: `10 × CPU cores`, e.g. 40 threads on a 4-core pod) queues thousands of tasks. The DB HikariCP pool (typically 10–20 connections) is saturated. Heap fills with `TopicContext` and filter objects.
5. Legitimate subscribers experience stalled or dropped message delivery as scheduler threads are starved and DB queries time out.

### Impact Explanation

The gRPC mirror node is the sole real-time HCS (Hedera Consensus Service) message delivery path. Saturating its `boundedElastic` scheduler and DB pool causes message delivery latency to spike for all subscribers on the affected pod. In a single-replica or low-replica deployment this is a complete denial of service for HCS consumers. The `subscriberCount` metric will alarm (via the `GrpcNoSubscribers` alert going the wrong direction) but there is no automatic remediation — the attacker's connections remain open indefinitely. [9](#0-8) 

### Likelihood Explanation

No credentials, API keys, or Hedera account are required. The gRPC port (5600) is publicly exposed. A single attacker machine can open hundreds of HTTP/2 connections. The attack is trivially scriptable with any gRPC client library (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). It is repeatable and persistent as long as the attacker holds connections open. The only infrastructure-level partial mitigation is the GCP backend policy `maxRatePerEndpoint: 250`, which limits new-request rate but not concurrent long-lived streaming calls already established. [10](#0-9) 

### Recommendation

1. **Add a global `maxActiveSubscriptions` property to `GrpcProperties`** (analogous to the GraphQL design's `maxActiveSubscriptions: 10`).
2. **Gate admission in `TopicMessageServiceImpl.subscribeTopic()`**: before constructing `TopicContext`, check `subscriberCount.get() >= grpcProperties.getMaxActiveSubscriptions()` and return `Mono.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED))` if exceeded.
3. **Add per-IP or per-connection subscription tracking** at the `ServerInterceptor` level to prevent a single source from consuming the entire quota.
4. **Increase `maxConcurrentCallsPerConnection`** awareness: document that this alone is insufficient as a DoS control since connection count is unbounded.
5. Consider moving the `safetyCheck` subscription off `Schedulers.boundedElastic()` to a dedicated, bounded scheduler to isolate its resource consumption.

### Proof of Concept

```bash
# Requires grpcurl and a valid topic ID (e.g., 0.0.1234) on the target network.
# Open 200 connections × 5 streams = 1000 concurrent subscriptions

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 1234}, "consensusStartTime": {"seconds": 0}}' \
      <GRPC_HOST>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Monitor server-side:
# kubectl exec -it <grpc-pod> -- curl -s localhost:8081/actuator/metrics/hiero.mirror.grpc.subscribers
# Watch DB pool saturation:
# kubectl exec -it <grpc-pod> -- curl -s localhost:8081/actuator/metrics/hikaricp.connections.active
```

Expected result: `hiero.mirror.grpc.subscribers` climbs to 1000+, HikariCP active connections saturate, `boundedElastic` task queue fills, and legitimate subscriber message latency degrades significantly or stalls entirely.
