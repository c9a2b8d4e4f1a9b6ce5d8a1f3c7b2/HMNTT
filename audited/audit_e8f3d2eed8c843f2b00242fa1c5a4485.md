### Title
Unauthenticated Timestamp-Range Flood Exhausts DB Connection Pool During Partition, Causing Persistent 503 DoS

### Summary
The REST API's Node.js connection pool defaults to 10 connections with a 20-second statement timeout. During a DB network partition, any in-flight query holds its connection for the full timeout duration. Because there is no per-IP rate limiting on the REST API, an unprivileged attacker sending as few as 10 concurrent requests with valid `timestamp=gte:X&timestamp=lte:Y` filters can exhaust the entire pool. Every subsequent legitimate request then waits for `connectionTimeoutMillis` (20 s), fails with a connection-timeout error, is wrapped in `DbError`, and is returned to the client as HTTP 503 via the error handler.

### Finding Description

**Code path — pool configuration:**
`rest/dbpool.js` lines 13–15 set `connectionTimeoutMillis: config.db.pool.connectionTimeout` (default 20 000 ms), `max: config.db.pool.maxConnections` (default **10**), and `statement_timeout: config.db.pool.statementTimeout` (default 20 000 ms). The pool is a plain `pg.Pool` with no queue-depth cap beyond `connectionTimeoutMillis`.

**Code path — query execution and error wrapping:**
`rest/utils.js` lines 1518–1539 (`queryQuietly`) issue `this.query(query, params)` and catch **all** thrown errors — including `Error: timeout expired` from `pg` when `connectionTimeoutMillis` elapses — and re-throw them as `new DbError(err.message)`.

**Code path — 503 mapping:**
`rest/middleware/httpErrorHandler.js` lines 21–22 map every `DbError` instance to `httpStatusCodes.SERVICE_UNAVAILABLE` (503), defined in `rest/constants.js` line 156.

**Code path — timestamp filter validation:**
`rest/utils.js` lines 1583–1681 (`parseTimestampFilters`) enforce `maxTimestampRange` (default **7 days**) only when `validateRange = true`. For the standard transaction/contract endpoints this check fires and rejects over-range requests with 400. However, the check does **not** prevent valid 7-day-range queries from being slow or from holding connections during a partition. `rest/controllers/contractController.js` line 387 calls `parseTimestampFilters(timestampFilters, false, true, true, false, false)` — the last argument `validateRange = false` — meaning contract-results endpoints accept **unbounded open ranges** that bypass even the 7-day cap.

**Root cause and failed assumption:**
The design assumes DB queries complete quickly enough that 10 connections are sufficient. During a partition, that assumption fails: every query blocks for up to 20 s. There is no per-IP concurrency limit, no queue-depth limit, and no circuit-breaker on the REST API layer. The `maxRepeatedQueryParameters` guard (default 100) limits how many times a single parameter can repeat in one URL but does not limit concurrent requests from the same IP.

**Exploit flow:**
1. Attacker observes or induces a DB partition (e.g., network-level disruption between the REST pod and PostgreSQL).
2. Attacker sends 10 concurrent HTTP GET requests to any timestamp-filtered endpoint, e.g. `/api/v1/transactions?timestamp=gte:1&timestamp=lte:604800000000000` (valid 7-day range).
3. Each request acquires one of the 10 pool connections; the DB is unreachable so each query blocks for 20 s.
4. All 10 connections are occupied. Any new request calls `pool.query()`, waits 20 s for `connectionTimeoutMillis`, receives a timeout error, which `queryQuietly` wraps as `DbError`.
5. `handleError` maps `DbError` → 503. Every legitimate user receives 503 for the entire 20-second window.
6. Attacker re-sends 10 requests every ~20 s to maintain the outage indefinitely.

### Impact Explanation
All legitimate API consumers receive `503 Service Unavailable` for the duration of the attack. Because the pool has only 10 slots and the timeout is 20 s, the attacker needs to sustain only 0.5 requests/second (10 requests per 20-second window) to keep the pool fully exhausted. This is a complete denial of service of the REST API, amplified by the partition: without the partition, queries complete in milliseconds and connections are released before the pool saturates.

### Likelihood Explanation
No authentication, API key, or special network position is required. The attack requires only HTTP access to the public REST API and knowledge that a partition is occurring (observable via elevated latency or 503 responses). The sustained request rate needed (10 req/20 s) is trivially achievable from a single host. The absence of any per-IP rate limiting or connection-queue cap on the REST API makes this repeatable and reliable.

### Recommendation
1. **Add per-IP concurrency limiting** at the Express middleware layer (e.g., `express-rate-limit` or an ingress-level policy) to cap concurrent in-flight DB requests per source IP.
2. **Increase the default pool size** or add a **queue-depth limit** so that pool exhaustion causes fast-fail (immediate 503) rather than a 20-second wait that holds the event loop.
3. **Reduce `connectionTimeoutMillis`** to a value much shorter than `statementTimeout` so that pool-exhaustion errors surface quickly rather than after 20 s.
4. **Add a circuit breaker** that stops accepting new DB-bound requests when the pool utilization exceeds a threshold (e.g., 80%), returning 503 immediately rather than queuing.
5. For the contract-results path, enforce `validateRange = true` (or at minimum enforce an upper bound on open ranges) to prevent unbounded timestamp scans.

### Proof of Concept
```bash
# During a DB partition, run from any unauthenticated client:
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gte:1&timestamp=lte=604800000000000" &
done
wait

# All 10 pool connections are now held for up to 20 s.
# Any concurrent legitimate request returns:
# {"_status":{"messages":[{"message":"Service unavailable"}]}}  HTTP/1.1 503

# Repeat every 19 s to maintain the outage:
while true; do
  for i in $(seq 1 10); do
    curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gte:1&timestamp=lte=604800000000000" &
  done
  sleep 19
done
```
